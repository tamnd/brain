## 10.2 Partial vs Total Functions

A **total function** gives an output for every input in its domain.

For functions on natural numbers, a total function has the form:

$$
f : \mathbb{N} \to \mathbb{N}
$$

This means that for every natural number $n$, the value $f(n)$ is defined.

Examples:

$$
f(n)=n+1
$$

$$
g(n)=2n
$$

$$
h(n)=n^2
$$

Each of these returns a value for every $n \in \mathbb{N}$.

A **partial function** may be undefined for some inputs.

We write:

$$
f : \mathbb{N} \rightharpoonup \mathbb{N}
$$

The arrow $\rightharpoonup$ indicates that the function may not return a value on every input.

For example, define:

$$
f(n)=
\begin{cases}
0 & \text{if } n \text{ is even},\
\text{undefined} & \text{if } n \text{ is odd}.
\end{cases}
$$

Then:

$$
f(4)=0
$$

but:

$$
f(5)
$$

has no value.

### Defined and Undefined Values

Computability theory uses special notation for termination.

If $f(n)$ is defined, we write:

$$
f(n)\downarrow
$$

This means the computation halts and returns a value.

If $f(n)$ is undefined, we write:

$$
f(n)\uparrow
$$

This means the computation does not halt.

So:

| Notation         | Meaning             |
| ---------------- | ------------------- |
| $f(n)\downarrow$ | $f(n)$ is defined   |
| $f(n)\uparrow$   | $f(n)$ is undefined |

For multiple inputs, we write similarly:

$$
f(x_1,\dots,x_k)\downarrow
$$

or:

$$
f(x_1,\dots,x_k)\uparrow
$$

### Why Partial Functions Appear

Partial functions appear naturally because programs may fail to halt.

Consider a program that searches for a number satisfying some condition:

```text
y := 0
while condition(x, y) is false:
    y := y + 1
return y
```

If such a $y$ exists, the program eventually returns it. If no such $y$ exists, the search continues forever.

This kind of computation defines a partial function:

$$
f(x)=\mu y,[\text{condition}(x,y)]
$$

If the least suitable $y$ exists, then $f(x)$ is defined. Otherwise, $f(x)$ is undefined.

### Total Computable Functions

A function is **total computable** if there is an algorithm that halts on every input and returns the correct value.

Examples include:

$$
f(n)=n+1
$$

$$
f(n)=n!
$$

$$
f(x,y)=x+y
$$

$$
f(x,y)=x \cdot y
$$

These can be computed by finite procedures that always terminate.

Total computable functions match the ordinary idea of algorithms that always give an answer.

### Partial Computable Functions

A function is **partial computable** if there is an algorithm that computes it whenever it is defined.

This means:

* if $f(n)$ is defined, the algorithm eventually halts and outputs $f(n)$
* if $f(n)$ is undefined, the algorithm may run forever

The algorithm does not need to detect undefinedness. In general, it cannot.

Partial computability is therefore more flexible than total computability.

Every total computable function is partial computable. But some partial computable functions are not total.

### Example: Division with Remainder Condition

Define:

$$
f(x,y)=
\begin{cases}
z & \text{if } y \cdot z = x \text{ for some } z \in \mathbb{N},\
\text{undefined} & \text{otherwise}.
\end{cases}
$$

This function returns the exact quotient when $y$ divides $x$.

For example:

$$
f(12,3)=4
$$

because:

$$
3 \cdot 4 = 12
$$

But:

$$
f(12,5)
$$

is undefined, because no natural number $z$ satisfies:

$$
5z=12
$$

A program can compute this by searching:

```text
z := 0
while y * z != x:
    z := z + 1
return z
```

If $y$ divides $x$, the search halts. If not, it runs forever.

### Domains of Partial Functions

The **domain** of a partial function is the set of inputs where it is defined:

$$
\operatorname{dom}(f)={n \in \mathbb{N} : f(n)\downarrow}
$$

For a total function:

$$
\operatorname{dom}(f)=\mathbb{N}
$$

For a partial function, the domain may be a proper subset:

$$
\operatorname{dom}(f)\subsetneq \mathbb{N}
$$

The domain is important because it describes exactly where the computation halts.

### Partiality and Decision Problems

A decision problem asks for a yes/no answer.

A set $A \subseteq \mathbb{N}$ is decidable if its characteristic function is total computable:

$$
\chi_A(n)=
\begin{cases}
1 & \text{if } n \in A,\
0 & \text{if } n \notin A.
\end{cases}
$$

The algorithm must halt for both yes and no cases.

A weaker notion is recognizability. A set $A$ is recognizable if there is a partial computable procedure that halts exactly on members of $A$.

That is:

$$
n \in A \quad \Longrightarrow \quad \text{the program halts}
$$

$$
n \notin A \quad \Longrightarrow \quad \text{the program may run forever}
$$

So decidability requires answers for both sides. Recognizability only requires confirmation of membership.

### Comparison

| Notion                      | Halts on yes? | Halts on no? | Output             |
| --------------------------- | ------------: | -----------: | ------------------ |
| decidable                   |           yes |          yes | yes/no             |
| recognizable                |           yes | not required | yes or no halt     |
| partial computable function |  when defined | not required | value when defined |

This distinction becomes essential in undecidability results. Many problems can be recognized but cannot be decided.
