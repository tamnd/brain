# 1. What CPython Is

CPython is the primary implementation of the Python programming language. It is the implementation most people use when they install Python from python.org, package managers, Linux distributions, Homebrew, pyenv, Docker images, or most cloud runtimes.

CPython has two roles.

First, it implements the Python language. It defines how Python source code is parsed, compiled, executed, and connected to the standard library.

Second, it acts as the practical reference point for the Python ecosystem. Other implementations may follow the language specification, but CPython is the implementation against which most packages, tools, debuggers, profilers, and C extensions are tested.

The CPython source tree contains the interpreter, compiler, object system, runtime, C API, standard library, test suite, documentation, and build machinery. The official developer guide describes CPython as a changing code base with internal documentation maintained in the source tree. ([Python Developer's Guide][1])

## 1.1 Python the Language vs CPython the Implementation

Python is the language.

CPython is one implementation of that language.

This distinction matters because the Python language describes visible behavior: syntax, names, expressions, statements, objects, modules, exceptions, iteration, async behavior, imports, and so on.

CPython describes one concrete way to make that behavior run on real machines.

A Python program such as this:

```python
def add(a, b):
    return a + b

print(add(2, 3))
```

is language-level Python. The language tells us that `def` creates a function object, `return` returns a value, `+` invokes addition semantics, and `print` writes a textual representation.

CPython implements those operations through lower-level mechanisms:

```text
source text
    ↓
tokens
    ↓
parser
    ↓
abstract syntax tree
    ↓
symbol table
    ↓
code object
    ↓
bytecode
    ↓
evaluation loop
    ↓
object operations
    ↓
C functions and machine code
```

The programmer usually sees only the top of this pipeline. CPython developers work across the whole pipeline.

## 1.2 CPython Is Both a Compiler and an Interpreter

CPython is often called an interpreter. That is correct, but incomplete.

CPython first compiles Python source code into bytecode. It then interprets that bytecode.

The compilation stage does not produce native machine code in the usual ahead-of-time compiler sense. It produces `code` objects containing bytecode instructions, constants, names, variable layout information, exception tables, line number information, and metadata needed by the runtime.

A file like this:

```python
x = 10
y = 20
print(x + y)
```

is converted into bytecode roughly shaped like:

```text
load constant 10
store name x
load constant 20
store name y
load name print
load name x
load name y
binary add
call
```

The actual bytecode changes across Python versions, but the model remains: CPython lowers source code into a compact instruction stream and executes that stream in a virtual machine.

You can inspect bytecode with `dis`:

```python
import dis

def add(a, b):
    return a + b

dis.dis(add)
```

This gives a Python-level view into CPython’s execution model.

## 1.3 The CPython Virtual Machine

The CPython virtual machine is a stack-based bytecode interpreter.

Most bytecode instructions operate on an evaluation stack. Instructions push values, pop values, call functions, create containers, branch, raise exceptions, and return results.

For example, evaluating:

```python
a + b
```

requires CPython to load `a`, load `b`, and apply binary addition. Conceptually:

```text
LOAD_FAST a      push local variable a
LOAD_FAST b      push local variable b
BINARY_OP +      pop two values, add them, push result
```

This stack machine exists inside the frame currently being executed. A frame stores the execution state of a function call: local variables, the value stack, instruction pointer, code object, globals, builtins, and exception state.

At runtime, CPython repeatedly performs this pattern:

```text
fetch next bytecode instruction
decode instruction and operand
dispatch to implementation
operate on Python objects
update frame state
continue or return
```

The core interpreter loop is one of the most important parts of CPython. It connects bytecode to object behavior.

## 1.4 CPython Is Written Mostly in C and Python

CPython is implemented mostly in C, with a large amount of Python code in the standard library.

The C code implements the core runtime:

```text
object representation
reference counting
garbage collector
interpreter loop
compiler
parser support
built-in types
C API
threading primitives
import machinery support
memory allocators
```

The Python code implements much of the standard library:

```text
collections
pathlib
asyncio
typing
unittest
importlib portions
dataclasses
logging
email
venv
many utility modules
```

Some modules have both Python and C implementations. The C version is usually used for performance, while the Python version may exist for clarity, portability, tests, or fallback behavior.

This mixed design matters. CPython is not just a C program that runs Python files. It is a runtime where C-level and Python-level components cooperate.

## 1.5 CPython Defines the Object Model in Memory

Every value in Python is an object.

In CPython, every object starts with a C-level object header. At minimum, that header stores a reference count and a pointer to the object’s type.

Conceptually:

```c
typedef struct {
    Py_ssize_t ob_refcnt;
    PyTypeObject *ob_type;
} PyObject;
```

Variable-sized objects use a larger header:

```c
typedef struct {
    PyObject ob_base;
    Py_ssize_t ob_size;
} PyVarObject;
```

This means that a Python integer, string, list, tuple, function, class, module, exception, frame, and generator all have a type pointer. The type object describes what operations the object supports.

For example:

```python
x = 42
```

creates or refers to a Python integer object. CPython does not store `x` as a raw CPU integer in normal Python code. It stores a reference to a `PyLongObject`.

That distinction explains many Python behaviors:

```python
x = 42
y = x
```

Here, `x` and `y` refer to the same object. Assignment binds names to objects. It does not copy the object by default.

## 1.6 Types Are Objects Too

In CPython, types are also objects.

The expression:

```python
type(42)
```

returns the object representing `int`.

The expression:

```python
type(int)
```

returns `type`.

This is not just a language trick. CPython represents types using `PyTypeObject`, a large C structure that stores type metadata and function pointers called slots.

A type object answers questions such as:

```text
How large is each instance?
How should instances be deallocated?
How is attribute lookup performed?
How is the object called?
How does addition work?
How does indexing work?
How does iteration work?
How is the object represented as text?
```

For a list, the type object points to list-specific implementations.

For a dict, it points to dict-specific implementations.

For a user-defined class, CPython builds a type object dynamically.

This is why Python’s data model maps cleanly onto CPython internals. Methods such as `__len__`, `__getitem__`, `__call__`, `__iter__`, and `__add__` correspond to internal operation slots.

## 1.7 Names Are References

Python variables are names bound to objects.

CPython implements this through references.

In Python:

```python
a = [1, 2, 3]
b = a
b.append(4)
print(a)
```

The output is:

```text
[1, 2, 3, 4]
```

Both `a` and `b` refer to the same list object.

At the CPython level, the assignment `b = a` increments the reference count of the list object. It does not duplicate the list.

This model affects performance and correctness. Passing a list to a function passes a reference to the object:

```python
def push(xs):
    xs.append(4)

items = [1, 2, 3]
push(items)
```

The function receives another reference to the same list.

## 1.8 Reference Counting Is the Main Memory Management Mechanism

CPython mainly uses reference counting to reclaim objects.

Each object records how many strong references point to it. When the count reaches zero, CPython deallocates the object immediately. The Python documentation describes the garbage collector as a supplement to reference counting, mainly for detecting unreachable reference cycles. ([Python documentation][2])

Conceptually:

```text
create object         refcount = 1
bind another name     refcount += 1
delete a name         refcount -= 1
refcount reaches 0    destroy object
```

Example:

```python
x = []
y = x
del x
del y
```

The list survives after `del x` because `y` still refers to it. After `del y`, the reference count can reach zero, so CPython can free the list.

Reference counting gives CPython predictable object finalization in many cases. For example, a file object may close promptly when its last reference disappears. But reference counting alone cannot handle cycles:

```python
a = []
b = []
a.append(b)
b.append(a)
```

Here, `a` and `b` reference each other. Even if no outside name points to them, their reference counts may remain nonzero. CPython’s cyclic garbage collector exists to find and clean up such cycles. Container types that can participate in cycles need explicit GC support. ([Python documentation][3])

## 1.9 CPython Has a Cyclic Garbage Collector

The cyclic garbage collector handles groups of objects that reference each other but are unreachable from live program roots.

It mainly tracks container objects: lists, dicts, sets, tuples containing references, functions, classes, frames, and other objects that can point to other objects.

Atomic objects such as many integers and strings usually do not need cyclic GC tracking because they do not contain references to other Python objects.

The collector works alongside reference counting:

```text
reference counting handles most objects immediately
cyclic GC handles unreachable reference cycles
```

This hybrid design shapes CPython behavior. Memory is often released quickly, but some cyclic structures are collected later.

The `gc` module exposes part of this machinery:

```python
import gc

gc.collect()
print(gc.get_count())
print(gc.isenabled())
```

This module is useful when studying object lifetime, leaks, finalizers, and cycle behavior.

## 1.10 CPython Uses a Global Interpreter Lock

Traditional CPython uses a Global Interpreter Lock, usually called the GIL.

The GIL allows only one thread at a time to execute Python bytecode within one interpreter. This simplifies memory management and protects internal interpreter state, especially reference count updates.

This does not mean CPython cannot use threads. It means CPU-bound Python bytecode does not normally run in parallel across multiple OS threads in the same interpreter.

Threads can still be useful for:

```text
I/O-bound programs
network clients
file operations
concurrent waiting
C extensions that release the GIL
```

For CPU-bound parallelism, Python programs often use:

```text
multiprocessing
native extensions
vectorized libraries
subinterpreters in newer designs
free-threaded builds in modern CPython work
```

The GIL is an implementation detail with language-level consequences. Code that appears concurrent at the Python level may serialize during bytecode execution.

## 1.11 CPython Has a Public C API

CPython exposes a C API for extension modules and embedding.

This is one of CPython’s most important ecosystem features. Libraries such as NumPy, pandas, lxml, cryptography, Pillow, and many database drivers depend on native extension mechanisms.

A minimal extension function might receive `PyObject *` values, parse arguments, call C code, and return a new `PyObject *`.

At this boundary, reference ownership is critical. CPython distinguishes borrowed references, new references, and stolen references. Incorrect ownership handling can cause memory leaks, use-after-free bugs, crashes, or corrupted interpreter state.

Example concepts:

```text
Py_INCREF     take another strong reference
Py_DECREF     release a strong reference
Py_XDECREF    release if pointer may be NULL
PyObject      base object pointer
PyTypeObject  type object structure
PyModuleDef   extension module definition
```

The C API makes CPython unusually extensible, but it also ties much of the ecosystem to CPython’s object representation and runtime behavior.

## 1.12 CPython Is Not the Same as Python Semantics

Some CPython behaviors are implementation details.

For example, immediate object destruction through reference counting is CPython behavior. Python code should usually avoid depending on exact destruction timing unless the behavior is explicitly guaranteed by the language or by a context manager.

This is portable:

```python
with open("data.txt") as f:
    data = f.read()
```

This is CPython-specific in timing:

```python
f = open("data.txt")
data = f.read()
f = None
```

In CPython, assigning `None` may close the file immediately if it removes the last reference. In another Python implementation, cleanup may happen later.

A good CPython internals reader must constantly separate three layers:

| Layer                   | Example                                  | Stability                          |
| ----------------------- | ---------------------------------------- | ---------------------------------- |
| Language semantics      | `for` loops call iteration protocol      | Very stable                        |
| CPython behavior        | Objects use reference counts             | Stable but implementation-specific |
| CPython internal layout | Exact struct fields and bytecode opcodes | Can change between versions        |

## 1.13 CPython and Other Python Implementations

CPython is not the only Python implementation.

Common alternatives include:

| Implementation | Runtime base | Main distinction                                                 |
| -------------- | ------------ | ---------------------------------------------------------------- |
| CPython        | C            | Reference implementation and strongest C extension compatibility |
| PyPy           | RPython      | JIT-oriented implementation                                      |
| Jython         | JVM          | Python on the Java platform                                      |
| IronPython     | .NET         | Python on the .NET runtime                                       |
| MicroPython    | C            | Python subset for microcontrollers and constrained devices       |

Most Python users run CPython. Most Python packages are tested first on CPython. Most native extension libraries target CPython’s C API.

This makes CPython central even when studying Python as a language.

## 1.14 What CPython Executes

CPython does not execute source text directly.

It executes code objects.

A code object contains compiled bytecode and metadata:

```text
bytecode instruction stream
constants
names
local variable names
free variables
cell variables
argument counts
stack size
filename
function name
line number table
exception table
flags
```

You can inspect a function’s code object:

```python
def f(x):
    y = x + 1
    return y

code = f.__code__

print(code.co_name)
print(code.co_varnames)
print(code.co_consts)
print(code.co_names)
```

A function object wraps a code object together with runtime context:

```text
code object
globals dictionary
defaults
keyword defaults
closure cells
annotations
qualname
module
```

This distinction matters. The same code object can be reused with different function objects, globals, defaults, or closures.

## 1.15 CPython’s Main Execution Units

A running CPython process contains several important execution units.

| Unit         | Meaning                                                                          |
| ------------ | -------------------------------------------------------------------------------- |
| Process      | The operating system process running CPython                                     |
| Runtime      | Global CPython runtime state                                                     |
| Interpreter  | An interpreter state with modules, builtins, import state, and execution context |
| Thread state | Per-thread execution state                                                       |
| Frame        | One active function/module/class execution                                       |
| Code object  | Compiled bytecode and metadata                                                   |
| Object       | Runtime value manipulated by bytecode                                            |

A simple function call creates or reuses a frame. That frame executes a code object. The frame belongs to a thread state. The thread state belongs to an interpreter. The interpreter belongs to the runtime.

Conceptually:

```text
CPython process
    runtime
        interpreter
            thread state
                frame
                    code object
                    value stack
                    locals
```

This hierarchy appears throughout the CPython C code.

## 1.16 CPython’s Standard Library Is Part of the Implementation

The standard library is not separate from CPython in practice.

Modules such as `sys`, `types`, `gc`, `inspect`, `dis`, `importlib`, and `ctypes` expose or interact with runtime internals.

Examples:

```python
import sys
print(sys.getrefcount(object()))

import dis
dis.dis(lambda x: x + 1)

import gc
print(gc.get_objects())

import inspect
print(inspect.currentframe())
```

Some modules are pure Python. Some are implemented in C. Some use both.

The boundary between interpreter and standard library is intentionally practical. CPython puts low-level operations in C when necessary and higher-level orchestration in Python when possible.

## 1.17 CPython Has a Large Test Suite

CPython includes a large regression test suite.

The test suite serves several purposes:

```text
protect language behavior
protect standard library behavior
catch platform-specific bugs
test C API behavior
test compiler and parser behavior
test memory management behavior
test edge cases from historical bugs
```

For internals work, the test suite is not optional. CPython changes can silently affect parsing, bytecode, object lifetime, imports, exceptions, threading, async behavior, or extension compatibility.

A normal development loop looks like:

```text
edit source
build CPython
run targeted tests
run broader tests
check reference leaks if relevant
inspect performance if relevant
```

## 1.18 Why CPython Internals Matter

CPython internals matter for several kinds of work.

For Python developers, internals explain performance behavior:

```text
why attribute access has cost
why function calls matter
why small object allocation is fast
why dicts are central
why some loops are slower than C loops
why local variables are faster than globals
```

For extension authors, internals are required knowledge:

```text
reference ownership
type definitions
buffer protocol
error handling
thread state
GIL rules
module initialization
ABI compatibility
```

For tooling authors, internals explain how to build debuggers, profilers, tracers, linters, coverage tools, bytecode tools, and import hooks.

For CPython contributors, internals are the working material: object layout, evaluation loop, compiler passes, memory allocators, platform abstractions, tests, and documentation.

## 1.19 A Mental Model for the Rest of the Book

Use this model while reading CPython source code:

```text
Python code is compiled into code objects.
Code objects contain bytecode.
Frames execute code objects.
The evaluation loop executes bytecode instructions.
Bytecode instructions manipulate PyObject pointers.
Objects point to type objects.
Type objects define behavior through slots.
Memory is managed mainly through reference counts.
The cyclic GC cleans up unreachable cycles.
The C API exposes much of this machinery to extensions.
```

That model is enough to orient almost every CPython subsystem.

## 1.20 Chapter Summary

CPython is the main implementation of Python. It compiles Python source code to bytecode, executes that bytecode in a stack-based virtual machine, represents all runtime values as objects, manages memory mainly through reference counting, supplements that with cyclic garbage collection, and exposes a powerful C API.

The rest of the book treats CPython as a concrete system: a compiler, virtual machine, object runtime, memory manager, extension platform, and standard library implementation.
