# 7.25 Practical Guidelines

This section consolidates operational advice for choosing, implementing, and validating binary search and ordered data structures in real systems. The emphasis is on decision criteria, invariant discipline, and performance trade-offs.

## Choosing the Right Approach

Start from the query and update profile.

| Scenario | Recommended approach |
| --- | --- |
| Static array, equality or boundary queries | Binary search on array |
| Static array, range sums | Prefix sums |
| Static array, range min or max | Sparse table |
| Dynamic updates, prefix aggregates | Fenwick tree |
| Dynamic updates, general range queries | Segment tree |
| Dynamic ordered set with predecessor or successor | Balanced tree |
| Repeated extraction of minimum or maximum | Heap |
| Large sparse coordinates | Coordinate compression + indexed structure |
| Optimization over a parameter | Binary search on answer |

The structure should match the dominant operation.

## Defining the Predicate

For search on answer, write the predicate before writing code.

```
```
P(x) answers: is x sufficient?
```
```

Check:

```
```
if x works, does every larger x also work?
```
```

or the reverse for maximum problems.

Avoid predicates that compute the full optimal solution. The predicate should check feasibility only.

## Bounds Selection

Choose bounds that are correct, not merely large.

For minimum search:

```
```
lo may fail  
hi must succeed
```
```

For maximum search:

```
```
lo must succeed  
hi may fail
```
```

When no natural upper bound is known, use doubling to find one, then binary search.

## Interval Discipline

Pick one convention and use it consistently.

Half-open:

```
```
[lo, hi)
```
```

Closed:

```
```
[lo, hi]
```
```

Half-open intervals simplify reasoning for first-true searches and array indexing.

State the invariant in terms of the chosen convention and verify that every update preserves it.

## Midpoint Calculation

Use:

```
```
mid = lo + (hi - lo) // 2
```
```

For last-true:

```
```
mid = lo + (hi - lo + 1) // 2
```
```

This avoids overflow in languages with fixed-width integers and ensures progress.

## Handling Duplicates

Do not rely on equality-based binary search.

Use:

```
```
lower_bound for first occurrence  
upper_bound for last occurrence
```
```

Then derive:

```
```
count = upper_bound - lower_bound
```
```

This removes ambiguity.

## Floating-Point Considerations

Use fixed iteration count when possible:

```
```
60 to 80 iterations for double precision
```
```

or a tolerance:

```
```
while hi - lo > eps
```
```

Avoid equality comparisons on floating-point values.

When squaring or multiplying large values, guard against overflow by rewriting comparisons.

## Performance Considerations

Binary search has low constant factors but poor cache locality due to non-sequential access. For very small arrays, a linear scan may be faster.

Balanced trees incur pointer chasing and reduced locality. Prefer arrays or Fenwick trees when possible.

Segment trees are more flexible but use more memory than Fenwick trees.

Sparse tables trade memory for constant-time queries.

## Implementation Checklist

Before finalizing code:

```
```
1. State the invariant  
2. Verify initialization  
3. Verify each update shrinks the interval  
4. Confirm termination condition  
5. Confirm return value matches specification
```
```

## Testing Strategy

Use layered testing.

1. Edge cases:

```
```
empty input  
single element  
all equal values  
strictly increasing  
strictly decreasing
```
```

2. Property checks:

```
```
boundary correctness  
monotonicity  
range coverage
```
```

3. Exhaustive testing on small inputs.
4. Random stress testing against a reference implementation.

## Debugging Method

When a bug appears:

1. print `lo`, `hi`, and `mid` each iteration
2. check whether the invariant still holds
3. test minimal failing input
4. rewrite using a known template

Most errors come from violating the invariant or mixing interval conventions.

## Common Mistakes

Choosing bounds without proof.

Using non-monotone predicates.

Forgetting to handle empty input.

Returning the wrong endpoint.

Using `lo = mid` instead of `lo = mid + 1`.

Mixing inclusive and exclusive intervals.

Ignoring overflow or precision issues.

## Integration with Other Techniques

Binary search often combines with:

- greedy checks
- prefix sums
- graph traversal
- dynamic programming
- coordinate compression

The search isolates the parameter. The auxiliary technique evaluates feasibility.

## Practical Heuristics

If the problem asks for a minimum or maximum value, attempt a feasibility predicate.

If the predicate is monotone, apply binary search.

If the input size is large but the value range is larger, prefer searching on the value range.

If updates are frequent, avoid static structures.

If memory is tight, prefer Fenwick trees over segment trees.

## Takeaway

Binary search and ordered structures are simple in code but strict in logic. Correctness depends on invariants, monotonicity, and consistent interval handling. Performance depends on matching the data structure to the operation pattern. Once these principles are applied systematically, implementations become reliable and predictable.
