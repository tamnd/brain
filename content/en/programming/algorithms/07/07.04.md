# 7.4 Monotone Predicates

A monotone predicate is a yes-or-no test whose answer changes in only one direction over an ordered domain. Binary search depends on this property. Without monotonicity, the algorithm has no reason to discard half of the search space.

For a first-true search, the predicate has this shape:

```text id="z2b8rr"
False False False False True True True
```

For a last-true search, the predicate has this shape:

```text id="hm4ikg"
True True True True False False False
```

Both cases have a single boundary. Binary search finds that boundary.

## Problem

Given a problem that asks for a minimum or maximum value, decide whether binary search on the answer is valid.

To use binary search, you must identify:

1. an ordered search space
2. a predicate over that search space
3. a monotonicity argument
4. valid initial bounds

The hardest part is usually the monotonicity argument. It must be a proof, not a guess.

## Solution

Use this test:

```text id="kmt49u"
If candidate x works, does every larger candidate also work?
```

If yes, the predicate is false-to-true, and you can search for the first true value.

Use this dual test:

```text id="6tylsn"
If candidate x works, does every smaller candidate also work?
```

If yes, the predicate is true-to-false, and you can search for the last true value.

## Example: Minimum Capacity

Suppose we must choose the smallest machine capacity that can finish a fixed workload before a deadline.

Define:

```text id="bybwzz"
P(c) = capacity c can finish the workload on time
```

If capacity `c` works, then any larger capacity also works, because the stronger machine can follow the same schedule or a better one. Therefore:

```text id="dbay07"
P(c) = False False False True True True
```

This is a first-true search.

```python id="s4duhj"
def first_true(lo: int, hi: int, pred) -> int:
    while lo < hi:
        mid = lo + (hi - lo) // 2

        if pred(mid):
            hi = mid
        else:
            lo = mid + 1

    return lo
```

## Example: Maximum Distance

Suppose we must place `k` objects so that the minimum distance between any two objects is as large as possible.

Define:

```text id="3yqrxj"
P(d) = it is possible to place all objects with distance at least d
```

If distance `d` works, then any smaller distance also works, because relaxing a constraint cannot destroy a valid placement. Therefore:

```text id="h27z5e"
P(d) = True True True False False False
```

This is a last-true search.

```python id="vl9c7s"
def last_true(lo: int, hi: int, pred) -> int:
    while lo < hi:
        mid = lo + (hi - lo + 1) // 2

        if pred(mid):
            lo = mid
        else:
            hi = mid - 1

    return lo
```

The midpoint is biased upward so that the interval shrinks when `lo + 1 == hi`.

## Monotonicity Proof Pattern

A monotonicity proof usually follows this form:

```text id="074ke2"
Assume candidate x is feasible.
Take any candidate y that is easier than x.
Show that the same construction also works for y.
Therefore feasibility is monotone.
```

For a first-true search, larger candidates are easier. A larger capacity, larger time limit, larger budget, or larger number of operations usually relaxes the constraint.

For a last-true search, smaller candidates are easier. A smaller required distance, smaller required score, or smaller lower bound usually relaxes the constraint.

The word "usually" matters. You still need to check the actual problem statement.

## Counterexample: Non-Monotone Predicate

Consider this predicate over integers:

```python id="xag6w6"
def pred(i: int) -> bool:
    return i % 2 == 0
```

The truth pattern is:

```text id="ibd83t"
True False True False True False
```

There is no single boundary. Binary search cannot be used.

Another common non-monotone predicate is:

```text id="e6fuxq"
P(k) = there exists a solution using exactly k operations
```

The word `exactly` often breaks monotonicity. A solution may exist for `k = 3`, fail for `k = 4`, and exist again for `k = 5`.

A monotone version may be:

```text id="1t713h"
P(k) = there exists a solution using at most k operations
```

If a solution exists using at most `k` operations, then it also exists using at most any larger number of operations.

## Designing the Predicate

A good predicate is simple, deterministic, and cheaper than the full optimization problem.

For a minimum feasible value, write:

```text id="57j3as"
P(x) = can satisfy the constraints with resource x
```

For a maximum feasible value, write:

```text id="44j46i"
P(x) = can satisfy the constraints while requiring at least x
```

The predicate should not compute the optimal answer internally. If it does, binary search adds unnecessary work.

Bad predicate:

```python id="k1p681"
def feasible(x: int) -> bool:
    best = compute_exact_optimum()
    return best <= x
```

This defeats the purpose because `compute_exact_optimum` already solves the problem.

Good predicate:

```python id="f5db6z"
def feasible(x: int) -> bool:
    return can_build_one_valid_solution_under_limit(x)
```

This checks one candidate value.

## Bounds

A binary search starts with a valid interval. The bounds must match the monotone direction.

For first-true search:

```text id="8h0h2m"
lo may fail
hi must work
```

For last-true search:

```text id="2k03xp"
lo must work
hi may fail
```

When the search is written over a closed interval `[lo, hi]`, both endpoints are candidates. When it is written over a half-open interval `[lo, hi)`, the right endpoint is excluded. Use one convention consistently.

A common safe pattern for first-true integer search is:

```python id="wtrscz"
def first_true(lo: int, hi: int, pred) -> int:
    """
    Returns the first x in [lo, hi] such that pred(x) is true.
    Requires pred(hi) == True.
    """
    while lo < hi:
        mid = lo + (hi - lo) // 2

        if pred(mid):
            hi = mid
        else:
            lo = mid + 1

    return lo
```

A common safe pattern for last-true integer search is:

```python id="hcvcuj"
def last_true(lo: int, hi: int, pred) -> int:
    """
    Returns the last x in [lo, hi] such that pred(x) is true.
    Requires pred(lo) == True.
    """
    while lo < hi:
        mid = lo + (hi - lo + 1) // 2

        if pred(mid):
            lo = mid
        else:
            hi = mid - 1

    return lo
```

## Finding an Upper Bound by Expansion

Sometimes an obvious upper bound is unavailable. If the predicate is false-to-true, you can find one by repeated doubling.

```python id="5n5nb4"
def find_first_true(pred) -> int:
    hi = 1

    while not pred(hi):
        hi *= 2

    lo = 0
    return first_true(lo, hi, pred)
```

This works only when a true value is guaranteed to exist. If no true value exists, the expansion loop does not terminate unless you add a limit.

A bounded version is safer in production code:

```python id="0olsfi"
def find_first_true_bounded(pred, limit: int) -> int | None:
    hi = 1

    while hi < limit and not pred(hi):
        hi *= 2

    hi = min(hi, limit)

    if not pred(hi):
        return None

    return first_true(0, hi, pred)
```

## Floating-Point Predicates

For real-valued domains, binary search cannot stop by waiting for `lo == hi`. The interval may shrink forever without becoming exactly zero because floating-point numbers are finite approximations.

Use either a fixed number of iterations:

```python id="j4bxb8"
def binary_search_real(lo: float, hi: float, pred, steps: int = 80) -> float:
    for _ in range(steps):
        mid = lo + (hi - lo) / 2.0

        if pred(mid):
            hi = mid
        else:
            lo = mid

    return hi
```

or use an explicit tolerance:

```python id="eerwbj"
def binary_search_real_eps(lo: float, hi: float, pred, eps: float = 1e-9) -> float:
    while hi - lo > eps:
        mid = lo + (hi - lo) / 2.0

        if pred(mid):
            hi = mid
        else:
            lo = mid

    return hi
```

Use fixed iterations when the numeric scale is uncertain. Use a tolerance when the result has a clear required precision.

## Common Pitfalls

Do not assume every optimization problem admits binary search. The feasible set must be contiguous in the search order.

Do not use a predicate involving `exactly k` unless you have separately proven monotonicity.

Do not change the predicate during search. The predicate must refer to the same problem for every candidate.

Do not hide non-determinism inside the predicate. If the predicate can return different results for the same input, binary search may discard the wrong half.

Do not use floating-point equality as a stopping condition.

Do not choose bounds without proving that they contain the answer.

## Testing Checklist

Test the predicate separately before testing the binary search.

```python id="jbnr5y"
assert pred(known_failing_value) is False
assert pred(known_working_value) is True
```

Test the boundary.

```python id="1nt1lb"
ans = first_true(lo, hi, pred)
assert pred(ans) is True
assert ans == lo or pred(ans - 1) is False
```

For last-true search:

```python id="dm1ylb"
ans = last_true(lo, hi, pred)
assert pred(ans) is True
assert ans == hi or pred(ans + 1) is False
```

Test the smallest interval.

```python id="n3ik7s"
assert first_true(0, 0, lambda x: True) == 0
assert last_true(0, 0, lambda x: True) == 0
```

Test adjacent boundaries.

```python id="rw3lvu"
assert first_true(0, 1, lambda x: x >= 1) == 1
assert last_true(0, 1, lambda x: x <= 0) == 0
```

## Takeaway

Binary search is justified by monotonicity, not by sorted arrays alone. Before writing code, state the predicate and prove that its true values form one contiguous region. Once that proof is clear, the implementation reduces to a boundary search.
