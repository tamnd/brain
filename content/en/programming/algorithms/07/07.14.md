# 7.14 Boundary Bugs

Boundary bugs are the most common failures in binary search and ordered data algorithms. The logic may be correct at a high level, but a single wrong comparison, interval convention, or midpoint update can produce an infinite loop, skip the answer, or read outside the array.

The cure is to choose one interval convention, write the invariant in terms of that convention, and make every update shrink the interval.

## Problem

Given a binary search implementation, determine whether it:

1. keeps the answer inside the search interval
2. terminates
3. handles empty and one-element inputs
4. returns the correct boundary
5. avoids out-of-bounds access

## A Safe Half-Open Template

For first-true search over indices `[0, n)`, use:

```python id="u5finw"
def first_true(n: int, pred) -> int:
    lo = 0
    hi = n

    while lo < hi:
        mid = lo + (hi - lo) // 2

        if pred(mid):
            hi = mid
        else:
            lo = mid + 1

    return lo
```

This returns the first index where `pred(i)` is true, or `n` if no such index exists.

The invariant is:

```text id="0zjjl1"
all i < lo are false
all i >= hi are true
```

At the end, `lo == hi`, and that index is the boundary.

## Bug: Using `hi = mid - 1` in a Half-Open Search

Incorrect:

```python id="oyxne2"
def lower_bound_bug(a: list[int], x: int) -> int:
    lo = 0
    hi = len(a)

    while lo < hi:
        mid = lo + (hi - lo) // 2

        if a[mid] >= x:
            hi = mid - 1
        else:
            lo = mid + 1

    return lo
```

The right endpoint `hi` is excluded. If `mid` is true, `mid` is still a possible answer. Setting `hi = mid - 1` discards it.

Counterexample:

```python id="2h7flw"
assert lower_bound_bug([10], 10) != 0
```

Correct:

```python id="z17kw3"
if a[mid] >= x:
    hi = mid
```

## Bug: Using `lo = mid`

Incorrect:

```python id="bog5ln"
def first_true_bug(n: int, pred) -> int:
    lo = 0
    hi = n

    while lo < hi:
        mid = lo + (hi - lo) // 2

        if pred(mid):
            hi = mid
        else:
            lo = mid

    return lo
```

When `lo + 1 == hi`, `mid == lo`. If `pred(mid)` is false, assigning `lo = mid` does not change the interval. The loop can run forever.

Correct:

```python id="ujwddz"
lo = mid + 1
```

because `mid` has been proven false.

## Bug: Returning the Wrong Endpoint

In the half-open first-true template, return `lo` or `hi`; they are equal at termination. Returning `lo - 1` changes the meaning from first true to last false.

Incorrect for lower bound:

```python id="g2m3fq"
return lo - 1
```

Correct:

```python id="lydpo8"
return lo
```

If you need the last value less than `x`, derive it explicitly:

```python id="ms43di"
last_lt = lower_bound(a, x) - 1
```

This may be `-1`, which correctly means no such element exists.

## Bug: Mixing Closed and Half-Open Intervals

Closed interval style uses `[lo, hi]`, where both endpoints are candidates.

Half-open style uses `[lo, hi)`, where `hi` is not a candidate.

Incorrect mixture:

```python id="um6hks"
lo = 0
hi = len(a)

while lo <= hi:
    mid = lo + (hi - lo) // 2
    ...
```

Here `hi = len(a)` is outside the array, but `while lo <= hi` allows `mid` to become `len(a)`, causing out-of-bounds access.

Closed interval initialization should be:

```python id="t31gtj"
lo = 0
hi = len(a) - 1
```

Half-open loop condition should be:

```python id="4sm5jo"
while lo < hi:
```

## Bug: Not Handling Empty Arrays

Closed interval code must handle empty arrays carefully.

```python id="u5wpyo"
def binary_search(a: list[int], x: int) -> int:
    lo = 0
    hi = len(a) - 1

    while lo <= hi:
        mid = lo + (hi - lo) // 2

        if a[mid] == x:
            return mid
        elif a[mid] < x:
            lo = mid + 1
        else:
            hi = mid - 1

    return -1
```

This code handles the empty array because `hi = -1`, so the loop does not run.

But in languages with unsigned integers, `len(a) - 1` may underflow. Prefer signed indices or half-open templates when possible.

## Bug: Incorrect Midpoint Bias in Last-True Search

For last-true search, this code can loop forever:

```python id="xjeg86"
def last_true_bug(lo: int, hi: int, pred) -> int:
    while lo < hi:
        mid = lo + (hi - lo) // 2

        if pred(mid):
            lo = mid
        else:
            hi = mid - 1

    return lo
```

When `lo + 1 == hi`, `mid == lo`. If `pred(mid)` is true, assigning `lo = mid` does not shrink the interval.

Correct:

```python id="h3qijg"
def last_true(lo: int, hi: int, pred) -> int:
    while lo < hi:
        mid = lo + (hi - lo + 1) // 2

        if pred(mid):
            lo = mid
        else:
            hi = mid - 1

    return lo
```

The upward midpoint bias makes `mid == hi` when two candidates remain.

## Bug: Off-by-One in Range Counting

Suppose `a` is sorted and we want the number of elements in `[L, R]`.

Incorrect:

```python id="m7vkel"
count = lower_bound(a, R) - lower_bound(a, L)
```

This counts elements in `[L, R)`, not `[L, R]`.

Correct:

```python id="tr6lon"
count = upper_bound(a, R) - lower_bound(a, L)
```

For half-open value range `[L, R)`, use:

```python id="phkceg"
count = lower_bound(a, R) - lower_bound(a, L)
```

The interval convention for values must be stated just as clearly as the interval convention for indices.

## Bug: Wrong Duplicate Handling

Equality-based binary search may return any occurrence of a duplicate, not necessarily the first or last.

```python id="r1gmpy"
def binary_search_any(a: list[int], x: int) -> int:
    lo = 0
    hi = len(a) - 1

    while lo <= hi:
        mid = lo + (hi - lo) // 2

        if a[mid] == x:
            return mid
        elif a[mid] < x:
            lo = mid + 1
        else:
            hi = mid - 1

    return -1
```

For:

```text id="q1h34e"
a = [2, 2, 2, 2]
```

this may return index `1` or `2`, depending on midpoint behavior.

Use lower bound for first occurrence:

```python id="25dmko"
def first_occurrence(a: list[int], x: int) -> int:
    i = lower_bound(a, x)
    if i < len(a) and a[i] == x:
        return i
    return -1
```

Use upper bound for last occurrence:

```python id="picvqr"
def last_occurrence(a: list[int], x: int) -> int:
    i = upper_bound(a, x) - 1
    if i >= 0 and a[i] == x:
        return i
    return -1
```

## Bug: Invalid Bounds in Search on Answer

In a first-true search, the upper bound must be known to work.

Incorrect:

```python id="ntrv0u"
lo = 0
hi = 10**9
```

This is only correct if you can prove that `pred(10**9)` is true.

Safer:

```python id="ubx7lt"
hi = 1
while not pred(hi):
    hi *= 2
```

This finds a working upper bound when one exists. In production code, add a maximum limit so the loop cannot run forever.

## Bug: Non-Monotone Predicate

Binary search over a predicate like this is invalid:

```python id="fsim3z"
def pred(i: int) -> bool:
    return i % 2 == 0
```

The truth pattern is:

```text id="js21a8"
True False True False True
```

There is no boundary. Binary search may return a value, but the value has no correctness guarantee.

Before coding, check that the predicate has one of these forms:

```text id="89cjfc"
False False False True True True
```

or:

```text id="h70zyl"
True True True False False False
```

## Bug: Floating-Point Termination

Incorrect:

```python id="6wek1h"
while lo < hi:
    mid = (lo + hi) / 2.0
```

For floating-point values, `mid` can eventually equal `lo` or `hi` due to rounding. The loop may stop making progress.

Use a fixed number of iterations:

```python id="bsy94o"
for _ in range(80):
    mid = lo + (hi - lo) / 2.0
```

or a tolerance:

```python id="uopunj"
while hi - lo > eps:
    mid = lo + (hi - lo) / 2.0
```

## Debugging Method

When a binary search fails, write down these five items:

```text id="d34htn"
1. What does lo mean?
2. What does hi mean?
3. Is hi included or excluded?
4. Which side remains possible after pred(mid)?
5. Does every update shrink the interval?
```

Then test the smallest arrays:

```python id="z2n1c0"
[]
[1]
[1, 2]
[1, 1]
```

Most boundary bugs appear in these cases.

## Testing Checklist

Lower bound:

```python id="yyf9lk"
assert lower_bound([], 5) == 0
assert lower_bound([5], 5) == 0
assert lower_bound([5], 6) == 1
assert lower_bound([1, 3], 2) == 1
```

Upper bound:

```python id="q3tv6v"
assert upper_bound([], 5) == 0
assert upper_bound([5], 5) == 1
assert upper_bound([5], 4) == 0
assert upper_bound([1, 1, 1], 1) == 3
```

Range count:

```python id="j22axa"
a = [1, 2, 2, 2, 3, 5]
assert upper_bound(a, 2) - lower_bound(a, 2) == 3
assert upper_bound(a, 4) - lower_bound(a, 4) == 0
```

Last true:

```python id="bs36j1"
assert last_true(0, 10, lambda x: x <= 7) == 7
assert last_true(0, 0, lambda x: True) == 0
```

## Takeaway

Boundary bugs disappear when the interval convention and invariant are explicit. For first-true search, use a half-open interval and return the boundary. For last-true search, bias the midpoint upward. For floating-point search, stop by tolerance or fixed iterations. The algorithm is short, but the invariant is the real implementation.
