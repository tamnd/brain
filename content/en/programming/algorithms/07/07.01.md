7.23 Heaps and Priority Queues

Heaps maintain a partial order that allows fast access to the smallest or largest element. Unlike binary search trees, heaps do not support full ordered traversal, but they provide efficient extraction and insertion of extremal elements.

A binary heap is the most common implementation. It is stored as an array and represents a complete binary tree.

Structure

For a min-heap:

parent <= children

For a max-heap:

parent >= children

The tree is complete, so it can be stored in an array.

For index i:

left child  = 2*i + 1
right child = 2*i + 2
parent      = (i - 1) // 2
Problem

Support operations:

insert a value

extract minimum (or maximum)

peek at the minimum (or maximum)

All operations should be efficient.

Push Operation

Insert a value at the end of the array, then restore the heap property by moving it upward.

Python
def push(heap: list[int], x: int) -> None:
    heap.append(x)
    i = len(heap) - 1

    while i > 0:
        p = (i - 1) // 2

        if heap[p] <= heap[i]:
            break

        heap[p], heap[i] = heap[i], heap[p]
        i = p

This process is called sift-up.

Pop Operation

Remove the root, replace it with the last element, then restore the heap property by moving it downward.

Python
def pop(heap: list[int]) -> int:
    if not heap:
        raise IndexError("pop from empty heap")

    root = heap[0]
    last = heap.pop()

    if heap:
        heap[0] = last
        sift_down(heap, 0)

    return root
Python
def sift_down(heap: list[int], i: int) -> None:
    n = len(heap)

    while True:
        left = 2 * i + 1
        right = 2 * i + 2
        smallest = i

        if left < n and heap[left] < heap[smallest]:
            smallest = left

        if right < n and heap[right] < heap[smallest]:
            smallest = right

        if smallest == i:
            break

        heap[i], heap[smallest] = heap[smallest], heap[i]
        i = smallest

This process is called sift-down.

Peek
Python
def peek(heap: list[int]) -> int:
    if not heap:
        raise IndexError("peek from empty heap")
    return heap[0]
Build Heap

Convert an arbitrary array into a heap in linear time.

Python
def heapify(a: list[int]) -> None:
    n = len(a)

    for i in range(n // 2 - 1, -1, -1):
        sift_down(a, i)

This works because leaves are already valid heaps, and internal nodes are fixed bottom-up.

Complexity
push:    O(log n)
pop:     O(log n)
peek:    O(1)
heapify: O(n)

The logarithmic cost comes from the height of the tree.

Comparison with Binary Search

Heaps and binary search solve different problems.

Operation	Binary Search	Heap
find element	O(log n)	O(n)
min / max	O(1) if known	O(1)
insert	O(n)	O(log n)
delete min / max	O(n)	O(log n)
sorted iteration	O(n)	O(n log n)

Heaps are optimized for repeated extraction of extremal elements.

Applications
Priority Queue

Process tasks in order of priority.

Python
heap = []
push(heap, (priority, task))
priority, task = pop(heap)
Dijkstra Algorithm

Use a min-heap to extract the node with the smallest distance.

Top K Elements

Find the k largest elements.

Python
import heapq

def top_k(nums, k):
    heap = []

    for x in nums:
        heapq.heappush(heap, x)
        if len(heap) > k:
            heapq.heappop(heap)

    return heap
Merge K Sorted Lists

Push the first element of each list, then repeatedly extract the smallest.

Heaps vs Balanced Trees
Feature	Heap	Balanced Tree
min / max	O(1)	O(log n)
insert	O(log n)	O(log n)
delete arbitrary	O(n)	O(log n)
order queries	not supported	supported

Heaps are specialized. Trees are general.

Common Pitfalls

Do not assume a heap is sorted. Only the root is guaranteed to be the smallest or largest.

Do not modify heap elements without restoring heap property.

Do not use heap for binary search. The internal structure is not ordered globally.

Do not forget that heapify runs in linear time, not O(n log n).

Testing Checklist
Python
heap = []
for x in [5, 3, 8, 1, 2]:
    push(heap, x)

assert peek(heap) == 1
assert pop(heap) == 1
assert pop(heap) == 2
assert pop(heap) == 3
assert pop(heap) == 5
assert pop(heap) == 8

Heapify:

Python
a = [5, 3, 8, 1, 2]
heapify(a)

assert pop(a) == 1
assert pop(a) == 2
Takeaway

Heaps maintain a partial order that supports efficient access to extremal elements. They complement binary search and trees by solving a different class of problems: repeated selection of minimum or maximum values.

peek:    O(1)
heapify: O(n)

The logarithmic cost comes from the height of the tree.

Comparison with Binary Search

Heaps and binary search solve different problems.

Operation	Binary Search	Heap
find element	O(log n)	O(n)
min / max	O(1) if known	O(1)
insert	O(n)	O(log n)
delete min / max	O(n)	O(log n)
sorted iteration	O(n)	O(n log n)

Heaps are optimized for repeated extraction of extremal elements.

Applications
Priority Queue

Process tasks in order of priority.

Python
heap = []

priority, task = pop(heap)
Dijkstra Algorithm

Use a min-heap to extract the node with the smallest distance.

Top K Elements

Find the k largest elements.

Python
import heapq

def top_k(nums, k):
    heap = []

    for x in nums:
        heapq.heappush(heap, x)
        if len(heap) > k:
            heapq.heappop(heap)

    return heap
Merge K Sorted Lists

Push the first element of each list, then repeatedly extract the smallest.

Heaps vs Balanced Trees
Feature	Heap	Balanced Tree
min / max	O(1)	O(log n)
insert	O(log n)	O(log n)
delete arbitrary	O(n)	O(log n)
order queries	not supported	supported

Heaps are specialized. Trees are general.

Common Pitfalls

Do not assume a heap is sorted. Only the root is guaranteed to be the smallest or largest.

Do not modify heap elements without restoring heap property.

Do not use heap for binary search. The internal structure is not ordered globally.

Do not forget that heapify runs in linear time, not O(n log n).

7.24 Range Query Structures

Range query structures answer questions about intervals of an array. Instead of searching for a single value, we ask for an aggregate over a range, such as a sum, minimum, maximum, or count.

The common pattern is to preprocess data so that each query avoids scanning the full interval.

Problem

Given an array a, support queries such as:

sum of a[L..R]
minimum of a[L..R]
maximum of a[L..R]
number of updates in a[L..R]

The right data structure depends on whether the array is static or dynamic.

Prefix Sums

For static range sum queries, prefix sums are simplest.

Python
def build_prefix(a: list[int]) -> list[int]:
    prefix = [0] * (len(a) + 1)

    for i, x in enumerate(a):
        prefix[i + 1] = prefix[i] + x

    return prefix

Query sum over inclusive range [L, R]:

Python
def range_sum(prefix: list[int], L: int, R: int) -> int:
    return prefix[R + 1] - prefix[L]

The prefix array uses a half-open convention:

prefix[i] = sum of a[0..i)

This makes the query formula uniform.

Fenwick Tree

A Fenwick tree supports point updates and prefix sum queries in logarithmic time.

Python
class Fenwick:
    def __init__(self, n: int):
        self.bit = [0] * (n + 1)

    def add(self, i: int, delta: int) -> None:
        i += 1
        while i < len(self.bit):
            self.bit[i] += delta
            i += i & -i

    def prefix_sum(self, i: int) -> int:
        total = 0
        i += 1
        while i > 0:
            total += self.bit[i]
            i -= i & -i
        return total

    def range_sum(self, L: int, R: int) -> int:
        if R < L:
            return 0
        return self.prefix_sum(R) - (self.prefix_sum(L - 1) if L > 0 else 0)

Fenwick trees are compact and fast for invertible prefix aggregates such as sum.

Segment Tree

A segment tree supports more general range queries. It can handle sums, minimums, maximums, greatest common divisors, and many other associative operations.

Python
class SegmentTree:
    def __init__(self, a: list[int]):
        self.n = len(a)
        self.tree = [0] * (2 * self.n)

        for i in range(self.n):
            self.tree[self.n + i] = a[i]

        for i in range(self.n - 1, 0, -1):
            self.tree[i] = self.tree[2 * i] + self.tree[2 * i + 1]

    def update(self, i: int, value: int) -> None:
        i += self.n
        self.tree[i] = value

        i //= 2
        while i >= 1:
            self.tree[i] = self.tree[2 * i] + self.tree[2 * i + 1]
            i //= 2

    def query(self, L: int, R: int) -> int:
        # query half-open interval [L, R)
        L += self.n
        R += self.n

        result = 0

        while L < R:
            if L % 2 == 1:
                result += self.tree[L]
                L += 1

            if R % 2 == 1:
                R -= 1
                result += self.tree[R]

            L //= 2
            R //= 2

        return result

This version answers range sum queries over half-open intervals [L, R).

Static Range Minimum Query

If the array never changes and only range minimum queries are needed, a sparse table gives O(1) queries after preprocessing.

Python
import math

class SparseTableMin:
    def __init__(self, a: list[int]):
        self.n = len(a)
        self.log = [0] * (self.n + 1)

        for i in range(2, self.n + 1):
            self.log[i] = self.log[i // 2] + 1

        k = self.log[self.n] + 1
        self.table = [a[:]]

        j = 1
        while (1 << j) <= self.n:
            prev = self.table[j - 1]
            length = 1 << j
            half = length >> 1
            row = []

            for i in range(self.n - length + 1):
                row.append(min(prev[i], prev[i + half]))

            self.table.append(row)
            j += 1

    def query(self, L: int, R: int) -> int:
        # inclusive range [L, R]
        length = R - L + 1
        j = self.log[length]
        return min(self.table[j][L], self.table[j][R - (1 << j) + 1])

This works because minimum is idempotent:

min(x, x) = x

The query combines two overlapping blocks that cover the requested range.

Choosing the Structure
Structure	Updates	Query	Best for
Prefix sum	none	O(1)	static sums
Fenwick tree	O(log n)	O(log n)	dynamic sums
Segment tree	O(log n)	O(log n)	dynamic associative queries
Sparse table	none	O(1)	static min/max/gcd
Complexity

space: O(n)

space:  O(n)

Segment tree:

build:  O(n)
update: O(log n)
query:  O(log n)
space:  O(n)

Sparse table:

build: O(n log n)
query: O(1)
space: O(n log n)
Common Pitfalls

Do not mix inclusive and half-open intervals. Prefix sums and iterative segment trees are usually cleaner with half-open ranges.

Do not use a sparse table for updates. It is designed for static data.

Do not use a Fenwick tree for non-invertible range queries unless the operation fits the Fenwick model.

Do not forget the identity element. Segment trees need the correct neutral value, such as 0 for sum and positive infinity for minimum.

Do not ignore coordinate compression when array indices come from large sparse values.

7.25 Practical Guidelines

This section consolidates operational advice for choosing, implementing, and validating binary search and ordered data structures in real systems. The emphasis is on decision criteria, invariant discipline, and performance trade-offs.

Choosing the Right Approach

Start from the query and update profile.

Scenario	Recommended approach
Static array, equality or boundary queries	Binary search on array
Static array, range sums	Prefix sums
Static array, range min or max	Sparse table
Dynamic updates, prefix aggregates	Fenwick tree
Dynamic updates, general range queries	Segment tree
Dynamic ordered set with predecessor or successor	Balanced tree
Repeated extraction of minimum or maximum	Heap
Large sparse coordinates	Coordinate compression + indexed structure
Optimization over a parameter	Binary search on answer

The structure should match the dominant operation.

Defining the Predicate

For search on answer, write the predicate before writing code.

P(x) answers: is x sufficient?

Check:

if x works, does every larger x also work?

or the reverse for maximum problems.

Avoid predicates that compute the full optimal solution. The predicate should check feasibility only.

Bounds Selection

Choose bounds that are correct, not merely large.

For minimum search:

lo may fail
hi must succeed

For maximum search:

lo must succeed
hi may fail

When no natural upper bound is known, use doubling to find one, then binary search.

Interval Discipline

Pick one convention and use it consistently.

Half-open:

[lo, hi)

Closed:

[lo, hi]

Half-open intervals simplify reasoning for first-true searches and array indexing.

State the invariant in terms of the chosen convention and verify that every update preserves it.

Midpoint Calculation

Use:

mid = lo + (hi - lo) // 2

For last-true:

mid = lo + (hi - lo + 1) // 2

This avoids overflow in languages with fixed-width integers and ensures progress.

Handling Duplicates

Do not rely on equality-based binary search.

Use:

lower_bound for first occurrence
upper_bound for last occurrence

Then derive:

count = upper_bound - lower_bound

This removes ambiguity.

Floating-Point Considerations

Use fixed iteration count when possible:

60 to 80 iterations for double precision

or a tolerance:

while hi - lo > eps

Avoid equality comparisons on floating-point values.

When squaring or multiplying large values, guard against overflow by rewriting comparisons.

Performance Considerations

Binary search has low constant factors but poor cache locality due to non-sequential access. For very small arrays, a linear scan may be faster.

Balanced trees incur pointer chasing and reduced locality. Prefer arrays or Fenwick trees when possible.

Segment trees are more flexible but use more memory than Fenwick trees.

Sparse tables trade memory for constant-time queries.

Implementation Checklist

Before finalizing code:

1. State the invariant
2. Verify initialization
3. Verify each update shrinks the interval
4. Confirm termination condition
5. Confirm return value matches specification
Testing Strategy

Use layered testing.

Edge cases:

empty input
single element
all equal values
strictly increasing
strictly decreasing

Property checks:

boundary correctness
monotonicity
range coverage

Exhaustive testing on small inputs.

Random stress testing against a reference implementation.

Debugging Method

When a bug appears:

print lo, hi, and mid each iteration

check whether the invariant still holds

test minimal failing input

rewrite using a known template

Most errors come from violating the invariant or mixing interval conventions.

Common Mistakes

Choosing bounds without proof.

Using non-monotone predicates.

Forgetting to handle empty input.

Returning the wrong endpoint.

Using lo = mid instead of lo = mid + 1.

Mixing inclusive and exclusive intervals.

Ignoring overflow or precision issues.

Integration with Other Techniques

Binary search often combines with:

greedy checks

prefix sums

graph traversal

dynamic programming

coordinate compression

The search isolates the parameter. The auxiliary technique evaluates feasibility.

Practical Heuristics

If the problem asks for a minimum or maximum value, attempt a feasibility predicate.

If the predicate is monotone, apply binary search.

If the input size is large but the value range is larger, prefer searching on the value range.

If updates are frequent, avoid static structures.

If memory is tight, prefer Fenwick trees over segment trees.

Takeaway

Binary search and ordered structures are simple in code but strict in logic. Correctness depends on invariants, monotonicity, and consistent interval handling. Performance depends on matching the data structure to the operation pattern. Once these principles are applied systematically, implementations become reliable and predictable.
