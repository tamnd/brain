# 7.12 Coordinate Compression

Coordinate compression replaces large, sparse, or inconvenient values with small integer indices while preserving order. It is useful when an algorithm needs array indexing, Fenwick trees, segment trees, dynamic programming tables, or compact key ranges, but the original values are too large to use directly.

The central requirement is order preservation. If one original value is smaller than another, its compressed index must also be smaller.

## Problem

Given values such as:

```text
[1000000000, -50, 200, 1000000000]
```

build a mapping to small indices:

```text
-50          -> 0
200          -> 1
1000000000   -> 2
```

Then transform the original list into:

```text
[2, 0, 1, 2]
```

The compressed values are easier to store and query, while comparisons by order remain valid.

## Solution

Sort the unique values, then use their positions as compressed coordinates.

```python
def compress(values: list[int]) -> tuple[list[int], dict[int, int], list[int]]:
    unique = sorted(set(values))
    index = {value: i for i, value in enumerate(unique)}
    compressed = [index[value] for value in values]
    return compressed, index, unique
```

Example:

```python
values = [1000000000, -50, 200, 1000000000]

compressed, index, unique = compress(values)

assert compressed == [2, 0, 1, 2]
assert index[-50] == 0
assert index[200] == 1
assert index[1000000000] == 2
assert unique == [-50, 200, 1000000000]
```

The `unique` array is also the inverse map. If `c` is a compressed coordinate, then `unique[c]` gives the original value.

## Why It Works

The sorted unique array defines the rank of each value.

```text
unique[i] is the i-th smallest distinct value
```

Therefore:

```text
x < y  implies  index[x] < index[y]
x = y  implies  index[x] = index[y]
```

This preserves exactly the ordering needed by binary search, range queries, Fenwick trees, and segment trees.

## Using Binary Search Instead of a Dictionary

If values are queried after compression, and the queried value may not exist in the original set, use binary search on the `unique` array.

```python
from bisect import bisect_left, bisect_right

def compressed_lower_bound(unique: list[int], x: int) -> int:
    return bisect_left(unique, x)

def compressed_upper_bound(unique: list[int], x: int) -> int:
    return bisect_right(unique, x)
```

For example:

```python
unique = [-50, 200, 1000000000]

assert compressed_lower_bound(unique, 150) == 1
assert compressed_upper_bound(unique, 200) == 2
```

This is useful for range queries over original coordinates.

## Range Query Example

Suppose we count how many inserted values fall in `[L, R]`.

```python
from bisect import bisect_left, bisect_right

def count_in_range(values: list[int], L: int, R: int) -> int:
    unique = sorted(values)
    left = bisect_left(unique, L)
    right = bisect_right(unique, R)
    return right - left
```

Here the sorted array itself is enough. If updates are needed, compressed coordinates allow a Fenwick tree or segment tree.

## Fenwick Tree Example

A Fenwick tree requires compact integer indices. Compression lets us use it with arbitrary values.

```python
class Fenwick:
    def __init__(self, n: int):
        self.bit = [0] * (n + 1)

    def add(self, i: int, delta: int) -> None:
        i += 1
        while i < len(self.bit):
            self.bit[i] += delta
            i += i & -i

    def sum_prefix(self, i: int) -> int:
        total = 0
        i += 1
        while i > 0:
            total += self.bit[i]
            i -= i & -i
        return total

    def sum_range(self, l: int, r: int) -> int:
        if r < l:
            return 0
        return self.sum_prefix(r) - (self.sum_prefix(l - 1) if l > 0 else 0)
```

Count values in an original coordinate range:

```python
from bisect import bisect_left, bisect_right

values = [1000000000, -50, 200, 1000000000]
compressed, index, unique = compress(values)

fw = Fenwick(len(unique))

for value in values:
    fw.add(index[value], 1)

def count_original_range(L: int, R: int) -> int:
    l = bisect_left(unique, L)
    r = bisect_right(unique, R) - 1
    return fw.sum_range(l, r)

assert count_original_range(-100, 300) == 2
assert count_original_range(1000000000, 1000000000) == 2
```

The original coordinates may be huge, but the Fenwick tree only has one slot per distinct value.

## Compression for Intervals

For interval problems, compressing only endpoints may be insufficient if the algorithm needs to represent gaps between endpoints.

Consider intervals:

```text
[10, 20]
[30, 40]
```

Compressing endpoints gives:

```text
10 -> 0
20 -> 1
30 -> 2
40 -> 3
```

This preserves endpoint order, but it does not explicitly represent the open gap between `20` and `30`.

For algorithms that need to mark covered points over integer coordinates, include neighboring coordinates when needed:

```python
def compress_interval_points(intervals: list[tuple[int, int]]) -> list[int]:
    points = set()

    for l, r in intervals:
        points.add(l)
        points.add(r)
        points.add(l - 1)
        points.add(r + 1)

    return sorted(points)
```

Use this carefully. Adding neighbors is meaningful for integer coordinate domains, but not for real-valued coordinates.

## Compression vs Normalization

Coordinate compression preserves order, but not distance.

Original values:

```text
10, 20, 1000
```

Compressed values:

```text
0, 1, 2
```

The gaps `10` and `980` both become distance `1` between adjacent compressed indices. This is correct for order queries, but wrong for algorithms that need actual lengths.

For example, when computing covered length of intervals, store original coordinates in `unique` and use differences:

```python
length = unique[i + 1] - unique[i]
```

Do not use compressed index difference as physical length.

## Complexity

For `n` input values and `m` distinct values:

```text
building unique values: O(n log n)
building map:           O(m)
compressing values:     O(n)
space:                  O(m)
```

Lookup by dictionary is `O(1)` average case.

Lookup by binary search is:

```text
O(log m)
```

## Common Pitfalls

Do not compress values when actual numeric distances are required unless you also keep the original coordinate array.

Do not assume compressed values are dense by original distance. They are dense only by rank.

Do not forget duplicates. Equal original values must map to the same compressed index.

Do not use a dictionary lookup for a value that may be absent. Use binary search on the sorted unique array.

Do not compress interval endpoints blindly when the algorithm needs gaps, lengths, or point coverage semantics.

## Testing Checklist

Basic compression:

```python
values = [1000000000, -50, 200, 1000000000]
compressed, index, unique = compress(values)

assert compressed == [2, 0, 1, 2]
assert unique == [-50, 200, 1000000000]
```

Order preservation:

```python
for x in unique:
    for y in unique:
        assert (x < y) == (index[x] < index[y])
```

Range query:

```python
assert count_original_range(-100, 300) == 2
assert count_original_range(201, 999999999) == 0
```

Absent value lookup:

```python
assert compressed_lower_bound(unique, 150) == 1
assert compressed_lower_bound(unique, 1000000001) == 3
```

## Takeaway

Coordinate compression converts sparse ordered values into dense array indices. It preserves comparison order, not distance. Use it when the algorithm needs compact indexing, and keep the sorted original coordinate array whenever range boundaries, absent queries, or physical lengths matter.
