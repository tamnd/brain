# 7.20 Summary

Binary search is best understood as boundary detection over an ordered domain. Arrays are only one instance. The essential ingredients are an ordered search space, a monotone predicate, and a loop that preserves an invariant while shrinking the interval.

## Core Model

Define a predicate over an ordered domain:

```text
P(x)
```

Require that `P` changes value once:

```text
False False False True True True
```

or:

```text
True True True False False False
```

The goal is to locate the boundary where the change occurs.

## Canonical Forms

First true (minimum feasible):

```text
find smallest x such that P(x) is true
```

Last true (maximum feasible):

```text
find largest x such that P(x) is true
```

Lower and upper bounds are special cases over arrays:

```text
lower_bound: first i with a[i] >= x
upper_bound: first i with a[i] > x
```

## Invariant

For first-true search over a half-open interval:

```text
all i < lo: P(i) is false
all i >= hi: P(i) is true
```

Initialization must satisfy the invariant. Each update must preserve it. Termination occurs when `lo == hi`, which is the boundary.

## Patterns

A small number of patterns covers most use cases:

- boundary search in arrays  
- search on answer  
- parametric search  
- floating-point search  
- tree-based ordered queries  
- coordinate compression with indexed structures  

Each pattern differs in domain and predicate cost, not in logic.

## Reductions

Many problems reduce to binary search once expressed correctly:

- optimization becomes feasibility  
- counting becomes difference of bounds  
- equality search becomes boundary search  
- geometric and scheduling problems become threshold checks  

Recognizing the reduction is the main step.

## Complexity

Binary search reduces the search space exponentially:

```text
O(log R)
```

where `R` is the size of the search domain.

With a predicate cost `T_pred`:

```text
O(T_pred * log R)
```

Balanced trees maintain:

```text
O(log n)
```

per operation.

## Implementation Discipline

Correct implementations share these properties:

- one interval convention, used consistently  
- explicit invariant  
- updates that always shrink the interval  
- careful midpoint computation  
- correct return value based on the invariant  

Avoid ad-hoc variations.

## Failure Modes

Common sources of error:

- mixing interval conventions  
- incorrect midpoint updates  
- off-by-one errors in bounds  
- non-monotone predicates  
- invalid initial bounds  
- incorrect handling of duplicates  
- floating-point termination issues  

These errors often appear only on small or extreme inputs.

## Testing Strategy

Reliable testing combines:

- edge cases such as empty and single-element inputs  
- property-based checks of boundary conditions  
- exhaustive testing on small inputs  
- stress testing with random data  
- invariant assertions during execution  

Testing should target the structure of the algorithm, not only specific examples.

## Mental Checklist

Before applying binary search:

```text
1. What is the ordered domain?
2. What is the predicate?
3. Is the predicate monotone?
4. What are valid bounds?
5. Which boundary do I need?
```

If these questions are answered precisely, the implementation follows directly.

## Final Perspective

Binary search is a general method for locating transitions in ordered systems. Arrays, trees, numeric domains, and optimization problems all fit this model. Once the boundary viewpoint is internalized, a wide range of problems reduce to a small set of reliable patterns.
