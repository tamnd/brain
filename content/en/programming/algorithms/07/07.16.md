# 7.16 Complexity Analysis

Binary search and ordered data structures derive their efficiency from systematic reduction of the search space. This section formalizes the cost of these reductions and clarifies how predicate cost, data structure height, and parameter ranges influence overall complexity.

## Binary Search on Discrete Domains

Consider a search space of size `n`. Each iteration reduces the interval size by about half.

After `k` iterations:

```text
remaining size ≤ n / 2^k
```

Termination occurs when the remaining size becomes `1`. Solve:

```text
n / 2^k ≤ 1
```

which gives:

```text
k ≥ log2(n)
```

Therefore:

```text
time = O(log n)
```

This bound holds regardless of the actual data, provided the predicate is monotone and constant-time.

## Binary Search with Non-Constant Predicate

If the predicate costs `T_pred`, the total cost becomes:

```text
O(T_pred * log n)
```

This is common in search-on-answer and parametric search.

### Example

For shipping capacity:

- predicate scans `n` items → `T_pred = O(n)`  
- binary search over range `R`  

Total:

```text
O(n log R)
```

where `R = hi - lo`.

## Search Space Size vs Input Size

Binary search complexity depends on the size of the search space, not directly on input size.

Examples:

| Problem                     | Search space size | Complexity         |
|----------------------------|------------------|--------------------|
| lower_bound on array       | n                | O(log n)           |
| capacity search            | sum(weights)     | O(log sum)         |
| time scheduling            | max_time         | O(log max_time)    |
| real-valued search         | (hi - lo)/eps    | O(log(1/eps))      |

When `R` is large (e.g. up to `10^18`), binary search still uses about 60 iterations.

## Floating-Point Search

For tolerance `eps` and initial interval length `L`, the number of iterations satisfies:

```text
k ≈ log2(L / eps)
```

Example:

```text
L = 10^6, eps = 10^-6
k ≈ log2(10^12) ≈ 40
```

In practice, 60 to 80 iterations are sufficient for double precision.

## Binary Search Trees

For a balanced tree with `n` nodes:

```text
height = O(log n)
```

Each operation follows a path from root to leaf.

```text
search: O(log n)
insert: O(log n)
delete: O(log n)
```

### AVL Trees

```text
height ≤ 1.44 log2(n)
```

### Red Black Trees

```text
height ≤ 2 log2(n)
```

The constant factor affects performance but not asymptotic complexity.

## Order Statistics Trees

Augmenting nodes with subtree sizes does not change height.

```text
select: O(log n)
rank:   O(log n)
```

The additional work per node is constant.

## Range Queries

Using lower and upper bounds:

```text
search cost: O(log n)
scan cost:   O(k)
total:       O(log n + k)
```

where `k` is the number of results.

## Fenwick Tree with Compression

Let `m` be the number of distinct values after compression.

```text
update: O(log m)
query:  O(log m)
```

Since `m ≤ n`, this remains efficient.

## Segment Tree

For array size `n`:

```text
build:  O(n)
update: O(log n)
query:  O(log n)
```

Memory:

```text
O(n)
```

## Trade-offs

| Structure       | Query        | Update       | Notes                    |
|-----------------|--------------|--------------|--------------------------|
| Sorted array    | O(log n)     | O(n)         | cache-friendly           |
| Binary search   | O(log n)     | N/A          | static data              |
| Balanced tree   | O(log n)     | O(log n)     | dynamic data             |
| Fenwick tree    | O(log n)     | O(log n)     | prefix sums              |
| Segment tree    | O(log n)     | O(log n)     | flexible range queries   |

Choice depends on update frequency and query type.

## Amortized Considerations

Some operations have amortized guarantees.

Example: doubling search for unknown upper bound

```text
1, 2, 4, 8, ..., 2^k
```

Total work:

```text
1 + 2 + 4 + ... + 2^k = O(2^k)
```

Once a valid bound is found, binary search adds `O(k)` steps.

Overall complexity remains logarithmic in the final bound.

## Lower Bounds

Comparison-based search has a lower bound:

```text
Ω(log n)
```

This follows from decision tree arguments. Any algorithm that distinguishes among `n` ordered possibilities must make at least `log2(n)` comparisons in the worst case.

Binary search is optimal under this model.

## Cache and Practical Performance

Binary search has poor cache locality compared to linear scan for small arrays.

For very small `n`, a linear scan may be faster despite worse asymptotic complexity.

Balanced trees have pointer-based structure, which reduces cache efficiency compared to arrays.

Fenwick and segment trees use contiguous memory and often perform better in practice.

## Common Pitfalls

Confusing `log n` with `log R`. These may differ significantly.

Ignoring predicate cost when analyzing search-on-answer.

Assuming floating-point search runs in `O(log n)`; it depends on precision, not input size.

Overlooking constant factors in tree structures.

Assuming compressed coordinate size equals original value magnitude.

## Testing Complexity

Empirically verify complexity by scaling input size.

```python
import time

def measure(f, *args):
    start = time.time()
    f(*args)
    return time.time() - start
```

Run with increasing sizes and observe growth rate.

## Takeaway

Binary search reduces search space exponentially, yielding logarithmic time. When combined with a non-constant predicate, total complexity becomes `O(T_pred log R)`. Balanced trees maintain logarithmic height, ensuring efficient dynamic operations. Understanding the relationship between search space, predicate cost, and structure height is essential for accurate performance analysis.
