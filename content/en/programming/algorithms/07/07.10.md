# 7.10 Balanced Trees

Balanced trees enforce structural constraints so that the height of the tree remains logarithmic in the number of nodes. This guarantees that all search, insertion, and deletion operations run in `O(log n)` time.

Without balancing, a binary search tree can degrade into a linear chain:

```text
1 -> 2 -> 3 -> 4 -> 5
```

Balanced trees prevent this degeneration.

## Problem

Maintain a dynamic ordered set of keys with:

- search  
- insertion  
- deletion  
- order queries  

All operations must run in logarithmic time regardless of input order.

## Core Idea

A binary search tree maintains ordering:

```text
left subtree < node < right subtree
```

Balanced trees add constraints that limit height:

```text
height = O(log n)
```

The constraints differ by tree type, but all aim to keep subtrees roughly balanced.

## AVL Trees

AVL trees maintain strict height balance:

```text
balance factor = height(left) - height(right) ∈ {-1, 0, 1}
```

After insertion or deletion, the tree may violate this condition. Rotations restore balance.

### Rotations

A rotation is a local restructuring that preserves in-order traversal.

Right rotation:

```text
    y              x
   / \            / \\n  x   T3   ->    T1  y
 / \                / \\nT1  T2             T2  T3
```

Left rotation:

```text
  x                y
 / \              / \\nT1  y    ->      x  T3
   / \          / \\n  T2  T3       T1  T2
```

Double rotations handle more complex imbalance patterns.

### Insertion

1. Insert as in a normal BST  
2. Update heights  
3. Check balance factor  
4. Apply rotation if needed  

```python
def insert(node, key):
    if not node:
        return Node(key)

    if key < node.key:
        node.left = insert(node.left, key)
    else:
        node.right = insert(node.right, key)

    update_height(node)

    balance = height(node.left) - height(node.right)

    # left heavy
    if balance > 1:
        if key < node.left.key:
            return rotate_right(node)
        else:
            node.left = rotate_left(node.left)
            return rotate_right(node)

    # right heavy
    if balance < -1:
        if key > node.right.key:
            return rotate_left(node)
        else:
            node.right = rotate_right(node.right)
            return rotate_left(node)

    return node
```

### Properties

```text
height ≤ 1.44 log2(n)
```

AVL trees provide faster lookups due to stricter balance but require more rotations.

## Red Black Trees

Red black trees relax balance conditions and use coloring rules.

Each node is either red or black, and the following properties hold:

```text
1. root is black
2. no two red nodes are adjacent
3. every path from root to leaf has the same number of black nodes
```

These constraints ensure logarithmic height without strict balancing.

### Rotations and Recoloring

Insertion and deletion may violate color rules. Fixing involves:

- recoloring nodes  
- performing rotations  

The operations are more complex than AVL but require fewer rotations on average.

### Properties

```text
height ≤ 2 log2(n)
```

Red black trees trade slightly worse balance for cheaper updates.

## Comparison

| Feature        | AVL Tree        | Red Black Tree   |
|----------------|----------------|------------------|
| Balance strict | Yes            | No               |
| Height bound   | ~1.44 log n    | ~2 log n         |
| Lookup speed   | Faster         | Slightly slower  |
| Insert cost    | Higher         | Lower            |
| Rotations      | More frequent  | Less frequent    |

AVL trees are preferable when lookups dominate. Red black trees are preferable when updates are frequent.

## Maintaining Invariants

After every update:

1. Check structural invariant (BST property)  
2. Check balance invariant  
3. Apply local fixes (rotations or recoloring)  
4. Propagate changes upward  

The key insight is that violations are local. Fixing a small region restores global correctness.

## Augmentation

Balanced trees support augmentation with additional fields.

Example: subtree size

```text
size(node) = 1 + size(left) + size(right)
```

Supports:

- k-th smallest element  
- rank queries  

Example: range sums

```text
sum(node) = node.value + sum(left) + sum(right)
```

Supports efficient interval aggregation.

Augmented values must be updated during rotations.

## Complexity

For both AVL and red black trees:

```text
search:   O(log n)
insert:   O(log n)
delete:   O(log n)
```

Space:

```text
O(n)
```

Rotations take constant time.

## Common Pitfalls

Forgetting to update height or color after rotation breaks invariants.

Applying the wrong rotation pattern leads to incorrect structure.

Not handling duplicate keys consistently causes ambiguity.

Failing to rebalance after deletion is a common source of bugs.

Augmented data must be recomputed after every structural change.

## Testing Checklist

Insert increasing sequence:

```python
# should remain balanced
for i in range(1, 1000):
    tree.insert(i)
```

Search correctness:

```python
assert tree.find(500) is not None
assert tree.find(1001) is None
```

Deletion:

```python
tree.delete(500)
assert tree.find(500) is None
```

Height bound:

```python
assert height(tree.root) <= C * log2(n)
```

for a small constant `C`.

## Takeaway

Balanced trees maintain logarithmic height by enforcing structural invariants. Rotations are the core mechanism that restores balance locally. Once height is controlled, all ordered operations inherit logarithmic complexity.
