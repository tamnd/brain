# 8.6 Binary Lifting

## Problem

You need to answer repeated ancestor queries on a tree, such as finding the k-th ancestor of a node, jumping upward quickly, or supporting operations like LCA efficiently.

## Solution

Precompute a jump table where each node stores its ancestor at distances that are powers of two.

```python id="1g0vmp"
LOG = 20
parent = [[-1] * n for _ in range(LOG)]
depth = [0] * n
```

First compute the immediate parent and depth using DFS.

```python id="b0g0qz"
def dfs(u, p):
    parent[0][u] = p
    for v in g[u]:
        if v == p:
            continue
        depth[v] = depth[u] + 1
        dfs(v, u)

dfs(0, -1)
```

Build higher jumps.

```python id="9e3b93"
for k in range(1, LOG):
    for v in range(n):
        if parent[k-1][v] == -1:
            parent[k][v] = -1
        else:
            parent[k][v] = parent[k-1][ parent[k-1][v] ]
```

Now `parent[k][v]` is the node reached after moving `2^k` steps upward from `v`.

## Query: k-th Ancestor

Jump upward using the binary representation of `k`.

```python id="4iqq2z"
def kth_ancestor(v, k):
    for i in range(LOG):
        if k & (1 << i):
            v = parent[i][v]
            if v == -1:
                break
    return v
```

## Example

If `k = 13`, then in binary `13 = 8 + 4 + 1`. The algorithm jumps:

- `2^3` steps  
- `2^2` steps  
- `2^0` steps  

This reduces a linear climb to a logarithmic sequence of jumps.

## Discussion

Binary lifting is a precomputation technique that trades memory for speed. It turns repeated upward traversal into a small number of table lookups.

The structure is similar to exponentiation by squaring. Instead of moving one edge at a time, the algorithm composes jumps of size `2^k`. This is efficient because any integer can be decomposed into powers of two.

Binary lifting is not limited to ancestors. The same idea applies to function iteration, jumping in linked lists, and doubling techniques in dynamic programming.

## Correctness

We prove that `parent[k][v]` is the `2^k`-th ancestor of `v`.

Base case: for `k = 0`, `parent[0][v]` is defined as the immediate parent, which is the `2^0 = 1`-st ancestor.

Inductive step: assume `parent[k-1][v]` is the `2^(k-1)`-th ancestor. Then

- `parent[k][v] = parent[k-1][ parent[k-1][v] ]`

This means we first move `2^(k-1)` steps up to reach `parent[k-1][v]`, then another `2^(k-1)` steps from there. The total is `2^k` steps. Therefore the table is correct.

For queries, the algorithm decomposes `k` into powers of two. Each jump moves exactly the required number of steps, so the total movement equals `k`.

## Complexity

Preprocessing takes `O(n log n)` time and space.

Each query runs in `O(log n)` time.

## Common Pitfalls

Choosing `LOG` too small leads to incorrect answers for deep trees. Use `LOG >= ceil(log2(n))`.

Failing to stop when the node becomes `-1` can cause invalid indexing.

Building the table before initializing `parent[0]` correctly leads to propagation of errors.

Confusing depth differences with number of jumps can cause incorrect lifting in LCA implementations.

## Extensions

You can store additional information in the lifting table, not just parents. For example:

- Maximum edge weight on the path upward  
- Sum of values along the path  
- Minimum value along the path  

```python id="p9i1w6"
maxEdge = [[0] * n for _ in range(LOG)]
```

Update rules follow the same doubling pattern:

```python id="t4kq8p"
maxEdge[k][v] = max(maxEdge[k-1][v], maxEdge[k-1][ parent[k-1][v] ])
```

This allows answering path queries in `O(log n)` time.

## Recipe Summary

Precompute ancestors at powers of two. Use binary decomposition of steps to jump upward efficiently. Extend the table with additional data when path queries are needed.
