# 8.13 Lazy Propagation

## Problem

You need to apply updates to entire ranges repeatedly and still answer queries efficiently. Applying updates element by element leads to `O(n)` per operation, which is too slow.

Typical operations:
- Add value to all elements in a range  
- Assign value to all elements in a range  
- Query sum, min, or max over a range  

## Solution

Use lazy propagation on a segment tree. Delay updates and apply them only when necessary.

Each node stores:
- The aggregate value for its interval  
- A lazy value representing a pending update  

## Range Add + Range Sum

```python id="q8h2kd"
class LazySegmentTree:
    def __init__(self, arr):
        self.n = len(arr)
        self.tree = [0] * (4 * self.n)
        self.lazy = [0] * (4 * self.n)
        self._build(arr, 1, 0, self.n - 1)

    def _build(self, arr, node, l, r):
        if l == r:
            self.tree[node] = arr[l]
            return
        mid = (l + r) // 2
        self._build(arr, node*2, l, mid)
        self._build(arr, node*2+1, mid+1, r)
        self.tree[node] = self.tree[node*2] + self.tree[node*2+1]
```

### Apply Update to Node

```python id="v6f8qt"
    def _apply(self, node, l, r, val):
        self.tree[node] += val * (r - l + 1)
        self.lazy[node] += val
```

### Push Lazy Value

```python id="z3g8kx"
    def _push(self, node, l, r):
        if self.lazy[node] == 0 or l == r:
            return

        mid = (l + r) // 2
        val = self.lazy[node]

        self._apply(node*2, l, mid, val)
        self._apply(node*2+1, mid+1, r, val)

        self.lazy[node] = 0
```

### Range Update

```python id="t0fq3c"
    def range_add(self, ql, qr, val):
        self._range_add(1, 0, self.n - 1, ql, qr, val)

    def _range_add(self, node, l, r, ql, qr, val):
        if qr < l or r < ql:
            return

        if ql <= l and r <= qr:
            self._apply(node, l, r, val)
            return

        self._push(node, l, r)

        mid = (l + r) // 2
        self._range_add(node*2, l, mid, ql, qr, val)
        self._range_add(node*2+1, mid+1, r, ql, qr, val)

        self.tree[node] = self.tree[node*2] + self.tree[node*2+1]
```

### Range Query

```python id="c5k7av"
    def range_sum(self, ql, qr):
        return self._range_sum(1, 0, self.n - 1, ql, qr)

    def _range_sum(self, node, l, r, ql, qr):
        if qr < l or r < ql:
            return 0

        if ql <= l and r <= qr:
            return self.tree[node]

        self._push(node, l, r)

        mid = (l + r) // 2
        return (
            self._range_sum(node*2, l, mid, ql, qr) +
            self._range_sum(node*2+1, mid+1, r, ql, qr)
        )
```

## Discussion

Lazy propagation stores updates at internal nodes instead of pushing them immediately to children. This avoids redundant work.

The key invariant:

```text
tree[node] reflects all updates applied to its interval
lazy[node] stores updates not yet propagated to children
```

When a query or update needs to access children, `_push` distributes the pending updates.

## Why It Works

Without lazy propagation:

- Updating range `[l, r]` requires visiting all affected leaves  
- This leads to `O(n)` worst-case time  

With lazy propagation:

- If an update fully covers a node interval, update the node once  
- Defer updates to children until needed  

This reduces complexity to logarithmic.

## Range Assignment Variant

Assignment is more complex than addition because it overwrites previous values.

You need:
- A separate lazy array for assignment  
- A flag indicating whether assignment is pending  

```python id="g8x2rt"
self.lazy_set = [None] * (4 * n)
```

Apply assignment:

```python id="r7k1nt"
def _apply_set(self, node, l, r, val):
    self.tree[node] = val * (r - l + 1)
    self.lazy_set[node] = val
    self.lazy[node] = 0
```

Push must handle assignment before addition.

## Correctness

For range add:

When `_apply` is called, the node correctly reflects the update over its interval. The lazy value records the deferred update for children.

When `_push` is called, children receive the pending update, ensuring their values remain consistent.

All queries either:
- Use stored values directly when fully covered  
- Push pending updates before descending  

Thus every query observes the correct state.

## Complexity

| Operation        | Time     |
|-----------------|----------|
| Build           | O(n)     |
| Range update    | O(log n) |
| Range query     | O(log n) |

Each operation visits at most `O(log n)` nodes.

## Common Pitfalls

Forgetting to call `_push` before descending leads to incorrect results.

Mixing assignment and addition lazily without proper priority handling causes errors.

Incorrect interval boundaries create off-by-one bugs.

Not resetting lazy values after pushing causes duplicate updates.

Using wrong identity values breaks queries.

## Recipe Summary

Use lazy propagation when range updates are frequent. Store deferred updates at nodes, push them only when needed, and maintain interval correctness at all times.
