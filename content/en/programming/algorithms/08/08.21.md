# 8.21 Complexity Analysis

## Problem

Tree algorithms often appear linear, but hidden factors such as recursion depth, repeated traversals, or auxiliary data structures can increase cost. You need a clear method to reason about time and space.

## Core Principle

A tree with `n` nodes has exactly `n - 1` edges. Most algorithms traverse nodes and edges a constant number of times, leading to linear complexity.

## DFS and BFS

Each edge is visited at most twice.

| Algorithm | Time | Space |
|---|---:|---:|
| DFS | O(n) | O(h) recursion |
| BFS | O(n) | O(n) queue |

`h` is tree height. Worst case `h = n` for a chain.

## Subtree DP

Each node aggregates values from children once.

```text
Total work = sum over nodes of (work per node)
           = O(n)
```

Even nested loops over children remain linear because total edges are `n - 1`.

## Binary Lifting

Precomputation builds a table of size `n log n`.

| Phase | Time | Space |
|---|---:|---:|
| Preprocess | O(n log n) | O(n log n) |
| Query | O(log n) | O(1) |

## Euler Tour + Range Structures

Flattening takes linear time.

Range queries depend on structure:

| Structure | Query | Update |
|---|---:|---:|
| Prefix sum | O(1) | O(n) |
| Fenwick tree | O(log n) | O(log n) |
| Segment tree | O(log n) | O(log n) |

Total complexity combines DFS + data structure cost.

## Heavy-Light Decomposition

Each path is split into at most `O(log n)` segments.

| Operation | Time |
|---|---:|
| Preprocess | O(n) |
| Query | O(log^2 n) |
| Optimized query | O(log n) |

The logarithmic factor comes from the number of chains crossed.

## Centroid Decomposition

Each level splits the tree roughly in half.

```text
T(n) = T(n/2) + O(n)
```

Solution:

```text
T(n) = O(n log n)
```

Each node participates in `O(log n)` levels.

## Persistent Structures

Each update creates `O(log n)` new nodes.

| Operation | Time | Space per update |
|---|---:|---:|
| Update | O(log n) | O(log n) |
| Query | O(log n) | O(1) |

Total memory grows with number of versions.

## Rerooting DP

Two DFS passes:

```text
Total work = 2 * O(n) = O(n)
```

Each edge is processed a constant number of times.

## Recursion Depth

Depth depends on tree shape:

| Tree type | Height |
|---|---:|
| Balanced | O(log n) |
| Chain | O(n) |

Recursive DFS may fail on deep trees. Iterative versions avoid this.

## Amortized Analysis

Some algorithms perform variable work per node but remain linear overall.

Example: merging child results

```text
Total merges = total edges = n - 1
```

Even if one node has many children, total work across all nodes is bounded.

## Correctness of Linear Bounds

A common mistake is to assume nested loops imply quadratic time.

Example:

```python
for u in nodes:
    for v in children[u]:
        process(u, v)
```

This loop runs once per edge, not per pair of nodes.

## Memory Analysis

Tree structures typically use:

- Adjacency list: `O(n)`  
- Auxiliary arrays: `O(n)`  
- Segment tree: `O(n)`  
- Binary lifting: `O(n log n)`  

Be careful with recursion stack and persistent structures.

## Common Pitfalls

Confusing worst-case height with average height.

Ignoring recursion limits in languages with small stacks.

Overlooking hidden logarithmic factors from data structures.

Double counting edges in undirected traversal.

Assuming operations inside DFS are constant when they depend on subtree size.

## Recipe Summary

Count how many times each node and edge is processed. Use structural properties of trees to bound total work. Combine traversal cost with data structure cost to obtain final complexity.
