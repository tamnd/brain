# 8.1 Tree Representation

## Problem

You need to store a tree so that algorithms can traverse it, query parent child relationships, compute subtree values, and avoid revisiting nodes. The representation should match the kind of tree you are solving on: rooted, unrooted, binary, weighted, or implicit.

## Solution

Use an adjacency list for the general case. It works for both rooted and unrooted trees, supports linear traversal, and has predictable memory use.

```python
n = 5
edges = [
    (0, 1),
    (0, 2),
    (1, 3),
    (1, 4),
]

g = [[] for _ in range(n)]

for a, b in edges:
    g[a].append(b)
    g[b].append(a)
```

For an unrooted tree, every edge is stored twice. The pair `(a, b)` appears as `b` in `g[a]` and `a` in `g[b]`. This makes traversal simple, but it also means that a DFS must remember the parent node.

```python
def dfs(u, parent):
    for v in g[u]:
        if v == parent:
            continue
        dfs(v, u)

dfs(0, -1)
```

If the tree is already rooted, store children directly.

```python
children = [
    [1, 2],   # children of 0
    [3, 4],   # children of 1
    [],       # children of 2
    [],       # children of 3
    [],       # children of 4
]
```

A rooted representation does not need a parent check when moving downward.

```python
def dfs(u):
    for v in children[u]:
        dfs(v)

dfs(0)
```

For a binary tree, use explicit left and right child fields.

```python
class Node:
    def __init__(self, value, left=None, right=None):
        self.value = value
        self.left = left
        self.right = right
```

This representation is natural when each node has at most two children and the position of each child matters.

## Discussion

The representation is part of the algorithm. A poor representation can turn a simple linear traversal into a complicated implementation with many special cases. A good representation makes the invariant visible.

An adjacency list represents a tree as a graph. It says only which nodes are connected. It does not say which node is the root, which node is the parent, or which nodes form a subtree. Those facts are created by traversal.

In an unrooted tree, the edge between `u` and `v` has no direction. When DFS enters `v` from `u`, the algorithm temporarily treats `u` as the parent of `v`. This rooted view exists for the duration of the traversal.

```python
def build_parent_and_order(root):
    parent = [-1] * n
    order = []

    stack = [root]
    parent[root] = root

    while stack:
        u = stack.pop()
        order.append(u)

        for v in g[u]:
            if v == parent[u]:
                continue
            parent[v] = u
            stack.append(v)

    parent[root] = -1
    return parent, order
```

The `parent` array gives each node a predecessor in the rooted tree. The `order` array records a traversal order. If you process `order` from left to right, parents appear before children. If you process it from right to left, children appear before parents, which is useful for subtree aggregation.

```python
parent, order = build_parent_and_order(0)

subtree_size = [1] * n

for u in reversed(order):
    for v in g[u]:
        if v == parent[u]:
            continue
        subtree_size[u] += subtree_size[v]
```

This pattern is common. First, root the tree. Then compute values in reverse traversal order.

## Weighted Trees

For weighted trees, store pairs `(neighbor, weight)`.

```python
n = 4
edges = [
    (0, 1, 7),
    (1, 2, 3),
    (1, 3, 5),
]

g = [[] for _ in range(n)]

for a, b, w in edges:
    g[a].append((b, w))
    g[b].append((a, w))
```

Traversal must carry both the neighbor and the edge weight.

```python
dist = [0] * n

def dfs(u, parent):
    for v, w in g[u]:
        if v == parent:
            continue
        dist[v] = dist[u] + w
        dfs(v, u)

dfs(0, -1)
```

The invariant is simple: after entering node `u`, `dist[u]` equals the total weight of the unique path from the root to `u`.

## Parent Array Representation

Some trees are given by a parent array. In this format, `parent[u]` stores the parent of node `u`, and the root has parent `-1`.

```python
parent = [-1, 0, 0, 1, 1]
n = len(parent)

children = [[] for _ in range(n)]
root = -1

for u, p in enumerate(parent):
    if p == -1:
        root = u
    else:
        children[p].append(u)
```

This representation is compact and useful for rooted trees. It is common in file systems, parse trees, organizational charts, and heap-like structures. When downward traversal is needed, convert the parent array into a children list once.

## Edge List Representation

An edge list is often the input format, but it is rarely the best working representation for traversal.

```python
edges = [
    (0, 1),
    (0, 2),
    (1, 3),
    (1, 4),
]
```

The edge list is good for reading input, sorting edges, or passing the tree to algorithms that process edges globally. For DFS, BFS, subtree queries, and path algorithms, convert it to an adjacency list.

## Binary Tree Representation

Binary trees differ from general rooted trees because the left and right child positions carry meaning. Swapping them may change the structure.

```python
root = Node(
    10,
    left=Node(5),
    right=Node(20, left=Node(15), right=Node(30)),
)
```

A traversal must explicitly check each child.

```python
def inorder(node):
    if node is None:
        return

    inorder(node.left)
    print(node.value)
    inorder(node.right)
```

This representation is appropriate for binary search trees, expression trees, heaps represented by pointers, and parsing tasks.

## Array Representation of Complete Binary Trees

A complete binary tree can be stored in an array. For a node at index `i`, the children are located at `2*i + 1` and `2*i + 2`.

```python
heap = [4, 10, 7, 20, 15]

def left(i):
    return 2 * i + 1

def right(i):
    return 2 * i + 2

def parent(i):
    return (i - 1) // 2
```

This representation avoids pointers and has excellent locality. It is used for heaps and segment trees. It works best when the tree shape is regular.

## Correctness Notes

For an undirected tree with `n` nodes, there are exactly `n - 1` edges. The adjacency list stores `2(n - 1)` neighbor entries. DFS with a parent check visits each node once because every non-root node is first reached through its unique parent edge.

The parent check is enough for a tree because there is only one simple path between any two nodes. In a general graph, the parent check alone is not enough; you also need a visited array. This distinction matters. Tree algorithms often look like graph algorithms, but they rely on stronger structural facts.

## Complexity

For an adjacency list representation, storage is `O(n)` because a tree has `n - 1` edges. Building the list from an edge list takes `O(n)` time. DFS and BFS also take `O(n)` time because each edge is examined a constant number of times.

For a rooted children list, storage is also `O(n)`. Each non-root node appears in exactly one child list. Downward traversal visits each node once.

For a complete binary tree stored in an array, storage is `O(n)`, and parent child navigation takes `O(1)` time.

## Common Pitfalls

Using a parent check in a general graph can cause repeated visits or infinite recursion. Use a visited array unless the input is guaranteed to be a tree.

Forgetting to add both directions in an undirected adjacency list makes traversal incomplete.

Using `0` as a parent sentinel can break code when node `0` is a valid node. Prefer `-1` or set `parent[root] = root` temporarily during construction.

Assuming recursive DFS is always safe can fail on deep trees. A path-shaped tree with many nodes can overflow the call stack in languages with limited recursion depth.

Mixing one-based and zero-based node labels causes off-by-one errors. Normalize labels as soon as input is read.

## Recipe Summary

Use an adjacency list for most tree algorithms. Root the tree during traversal by tracking the parent. Use children lists when the tree is already rooted. Use explicit left and right fields for binary trees. Use array layout for complete binary trees and heaps. The representation should expose the relationships required by the algorithm and avoid storing facts that can be derived cheaply during traversal.
