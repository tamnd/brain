# 8.14 Persistent Trees

## Problem

You need to preserve older versions of a tree-like data structure after updates. A normal segment tree overwrites nodes in place, so previous states are lost. Persistent trees solve this by sharing unchanged nodes between versions and copying only the nodes affected by an update.

Typical uses include:

- Versioned range queries
- Offline query processing
- Undoable data structures
- K-th order statistic queries
- Historical snapshots of an array

## Solution

Use path copying. On each update, create new nodes only along the path from the root to the changed leaf. All other nodes are shared with the previous version.

For a persistent segment tree supporting point updates and range sums:

```python
class Node:
    def __init__(self, left=None, right=None, value=0):
        self.left = left
        self.right = right
        self.value = value
```

Build the initial tree:

```python
def build(arr, l, r):
    if l == r:
        return Node(value=arr[l])

    mid = (l + r) // 2
    left = build(arr, l, mid)
    right = build(arr, mid + 1, r)

    return Node(left, right, left.value + right.value)
```

Update one index by returning a new root:

```python
def update(node, l, r, index, value):
    if l == r:
        return Node(value=value)

    mid = (l + r) // 2

    if index <= mid:
        new_left = update(node.left, l, mid, index, value)
        return Node(new_left, node.right, new_left.value + node.right.value)
    else:
        new_right = update(node.right, mid + 1, r, index, value)
        return Node(node.left, new_right, node.left.value + new_right.value)
```

Query works the same as in a normal segment tree:

```python
def query(node, l, r, ql, qr):
    if qr < l or r < ql:
        return 0

    if ql <= l and r <= qr:
        return node.value

    mid = (l + r) // 2

    return (
        query(node.left, l, mid, ql, qr)
        + query(node.right, mid + 1, r, ql, qr)
    )
```

Maintain one root per version:

```python
arr = [5, 2, 7, 3, 6]
n = len(arr)

roots = []
roots.append(build(arr, 0, n - 1))

roots.append(update(roots[0], 0, n - 1, 2, 10))
roots.append(update(roots[1], 0, n - 1, 4, 1))

print(query(roots[0], 0, n - 1, 0, 4))  # 23
print(query(roots[1], 0, n - 1, 0, 4))  # 26
print(query(roots[2], 0, n - 1, 0, 4))  # 21
```

Each root represents a complete logical array, but the physical memory is shared.

## Discussion

A persistent tree separates logical versions from physical storage. After an update, the new version reuses most of the old structure. Only the nodes on the update path change, because only those intervals have aggregates affected by the modified element.

For an array of length `n`, a point update touches one leaf and one node at each level above it. Therefore each update creates `O(log n)` new nodes. The remaining `O(n)` structure is shared.

The main invariant is:

```text
Each root points to an immutable segment tree representing one version of the array.
```

Once a node is created, it should not be modified. Mutation breaks persistence because old versions may still point to that node.

## Example: Versioned Prefix Counts

Persistent segment trees are often used to answer frequency queries over prefixes.

Suppose you process an array from left to right. Each version stores counts of values seen so far.

```python
MAXV = 10

zero = build([0] * (MAXV + 1), 0, MAXV)
roots = [zero]

data = [3, 1, 4, 1, 5]

for x in data:
    old_count = query(roots[-1], 0, MAXV, x, x)
    roots.append(update(roots[-1], 0, MAXV, x, old_count + 1))
```

Now `roots[i]` represents the multiset of the first `i` values.

To count how many values in `data[l:r+1]` equal `x`:

```python
def count_in_range(l, r, x):
    right_count = query(roots[r + 1], 0, MAXV, x, x)
    left_count = query(roots[l], 0, MAXV, x, x)
    return right_count - left_count
```

This works because the count in a subarray is the difference between two prefix versions.

## K-th Smallest in a Range

With coordinate compression, persistent segment trees can answer k-th smallest queries in subarrays.

Each version stores value frequencies for a prefix. To query range `[l, r]`, compare roots `roots[r + 1]` and `roots[l]`.

```python
def kth(left_root, right_root, l, r, k):
    if l == r:
        return l

    mid = (l + r) // 2

    left_count = right_root.left.value - left_root.left.value

    if k <= left_count:
        return kth(left_root.left, right_root.left, l, mid, k)
    else:
        return kth(left_root.right, right_root.right, mid + 1, r, k - left_count)
```

The two roots define the frequency distribution inside the queried range. At each level, the algorithm decides whether the k-th element lies in the left half or the right half.

## Correctness

The persistent update is correct by induction on the segment interval.

For a leaf interval, the update creates a new leaf with the new value. This represents the updated array at that index.

For an internal interval, only one child interval contains the updated index. The algorithm recursively creates a new version of that child and reuses the unchanged child from the old tree. The new node stores the merge of the two children, so its aggregate is correct.

Old versions remain correct because no existing node is modified. Each old root still points to exactly the nodes that represented its version before the update.

For prefix frequency versions, each version differs from the previous one by adding one occurrence of the next value. Therefore the difference between two prefix versions gives exactly the frequencies in the intervening subarray.

## Complexity

For an array of length `n`:

| Operation | Time | New Space |
|---|---:|---:|
| Build initial tree | O(n) | O(n) |
| Point update | O(log n) | O(log n) |
| Range query | O(log n) | O(1) |
| K-th query | O(log n) | O(1) |

For `m` updates, total memory is `O(n + m log n)`.

## Common Pitfalls

Mutating existing nodes destroys persistence. Treat all nodes as immutable after creation.

Forgetting to store each new root loses access to that version.

Using persistent segment trees without coordinate compression can waste memory when values are large.

Range updates with persistence are more complex than point updates. Combining lazy propagation and persistence requires careful copying of lazy state.

Python object overhead can be high. For large inputs, array-based node pools are often faster.

## Recipe Summary

Use persistent trees when old versions must remain queryable. Copy only the update path, reuse unchanged nodes, and store one root per version. This gives versioned queries with logarithmic update cost and controlled memory growth.
