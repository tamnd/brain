# 8.17 Suffix Trees Overview

## Problem

You need to process substring queries efficiently on a string, such as:

- Check if a substring exists  
- Count occurrences of a substring  
- Find longest repeated substring  
- Find longest common substring  

A naive approach scans the string repeatedly, leading to `O(n^2)` or worse.

## Solution

Use a suffix tree. It is a compressed trie of all suffixes of a string. Each path from root represents a substring.

Instead of storing each character as a separate edge, edges are labeled with substrings, which compresses chains of nodes.

## Structure

For a string `S`, insert all suffixes:

```text
S[i...n-1] for all i
```

A suffix tree stores these efficiently by merging common prefixes.

Each edge is labeled with a range `[l, r]` into the original string.

Each leaf corresponds to one suffix.

## Example

For string:

```text
banana
```

Suffixes:

```text
banana
anana
nana
ana
na
a
```

The suffix tree merges repeated prefixes like `"ana"`.

## Operations

### Substring Search

To check if pattern `P` exists:

- Start at root  
- Follow edges matching characters of `P`  
- If fully matched, substring exists  

Time: `O(|P|)`

### Count Occurrences

Each occurrence corresponds to a leaf under the matching node.

Store subtree sizes to answer quickly.

### Longest Repeated Substring

Find the deepest internal node (node with ≥ 2 children). The path to that node is the answer.

### Longest Common Substring

Build a generalized suffix tree for two strings and find the deepest node containing suffixes from both.

## Construction

Suffix trees can be built in linear time using Ukkonen’s algorithm.

Key ideas:
- Incrementally add suffixes  
- Maintain active points  
- Use suffix links to skip redundant work  

Implementation is complex, but the result is `O(n)` construction.

## Discussion

Suffix trees provide optimal asymptotic performance for many string problems, but they are heavy:

- Complex to implement  
- Large constant factors  
- High memory usage  

Because of this, suffix arrays are often preferred in practice.

## Comparison

| Structure      | Build Time | Query Time | Memory |
|----------------|----------:|-----------:|-------:|
| Trie           | O(n^2)    | O(m)       | High   |
| Suffix Tree    | O(n)      | O(m)       | Very high |
| Suffix Array   | O(n log n) or O(n) | O(m log n) | Low |

## Correctness

Each suffix is inserted into the tree, and common prefixes are shared. Therefore every substring of the original string corresponds to some path in the tree.

If a pattern matches along edges from the root, it must appear in the string. Conversely, every occurrence of a substring corresponds to a path in the tree.

## Complexity

Let `n` be string length, `m` pattern length.

| Operation | Time |
|----------|------|
| Build    | O(n) |
| Search   | O(m) |
| Count    | O(m) |

Space is `O(n)` but with large constants due to nodes and pointers.

## Common Pitfalls

Implementing Ukkonen incorrectly is common. Off-by-one errors and incorrect suffix links break the structure.

Using explicit substrings instead of index ranges increases memory to `O(n^2)`.

Not handling edge splitting properly leads to incorrect trees.

Ignoring memory constraints can cause performance issues.

## When to Use

Use suffix trees when:
- You need optimal substring queries  
- String length is moderate  
- You need complex substring analytics  

Otherwise prefer suffix arrays or rolling hashes.

## Recipe Summary

Suffix trees compress all suffixes into a single structure, enabling linear-time substring queries. They are powerful but complex, so use them when performance requirements justify the cost.
