# 8.5 Lowest Common Ancestor

## Problem

Given a rooted tree and two nodes `u` and `v`, you need the lowest node that is an ancestor of both. Queries should be fast after preprocessing.

## Solution

Precompute binary lifting tables. Each node stores its ancestors at powers of two distances.

```python
LOG = 20

parent = [[-1] * n for _ in range(LOG)]
depth = [0] * n
```

Run a DFS to fill the immediate parent and depths.

```python
def dfs(u, p):
    parent[0][u] = p
    for v in g[u]:
        if v == p:
            continue
        depth[v] = depth[u] + 1
        dfs(v, u)

dfs(0, -1)
```

Build the lifting table.

```python
for k in range(1, LOG):
    for v in range(n):
        if parent[k-1][v] == -1:
            parent[k][v] = -1
        else:
            parent[k][v] = parent[k-1][ parent[k-1][v] ]
```

## Query

Lift the deeper node up to match depths, then lift both together.

```python
def lca(u, v):
    if depth[u] < depth[v]:
        u, v = v, u

    # lift u up
    diff = depth[u] - depth[v]
    for k in range(LOG):
        if diff & (1 << k):
            u = parent[k][u]

    if u == v:
        return u

    # lift both
    for k in reversed(range(LOG)):
        if parent[k][u] != parent[k][v]:
            u = parent[k][u]
            v = parent[k][v]

    return parent[0][u]
```

## Discussion

Binary lifting converts the tree into a jump table. Each entry `parent[k][v]` means the node reached by moving `2^k` steps upward from `v`. This reduces ancestor queries from linear time to logarithmic time.

The key idea is that any upward movement can be decomposed into powers of two. Instead of moving one step at a time, the algorithm jumps in large steps, then refines.

Depth alignment is necessary because ancestors are defined relative to levels. Once both nodes are at the same depth, moving them upward in sync preserves the invariant that the LCA remains above them.

## Alternative: Euler Tour + RMQ

Flatten the tree using an Euler tour and reduce LCA to a range minimum query over depths.

Outline:

1. Perform DFS and record visit order.
2. Record depth at each visit.
3. For each node, record first occurrence in the tour.
4. LCA becomes RMQ over the interval between first occurrences.

This gives `O(1)` query time with preprocessing `O(n log n)` or `O(n)` depending on RMQ method.

## Correctness

For binary lifting:

After depth alignment, both nodes lie on paths that eventually meet at their LCA. The second phase finds the highest level where their ancestors differ. Once no larger jump keeps them different, their parents must be equal and equal to the LCA.

Formally, we maintain that after lifting, `u` and `v` are below the LCA and in different subtrees of it. The loop preserves this condition while moving them upward. When the loop ends, both nodes are direct children of the LCA.

## Complexity

Preprocessing takes `O(n log n)` time and space.

Each query runs in `O(log n)` time.

## Common Pitfalls

Using too small `LOG` leads to incorrect results for deep trees. Ensure `LOG >= ceil(log2(n))`.

Failing to initialize parents of the root properly can propagate invalid values.

Mixing zero-based and one-based indexing in parent tables causes subtle bugs.

Forgetting to swap nodes so that `u` is deeper leads to incorrect lifting.

## Recipe Summary

Use binary lifting when you need many LCA queries with static trees. Precompute ancestors, align depths, then lift both nodes upward until their parents match.
