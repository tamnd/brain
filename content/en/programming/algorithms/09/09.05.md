# 9.5 Directed vs Undirected Graphs

## Problem

You must decide whether edges carry direction. This choice determines reachability, traversal behavior, and which algorithms are valid. Treating a directed problem as undirected, or the reverse, produces incorrect results even if the implementation is otherwise correct.

## Solution

Classify each relationship:

- If the relation is symmetric, use an undirected graph.
- If the relation has direction or causality, use a directed graph.

Formally:

- Undirected edge: $\{u, v\} \in E$ allows movement both ways.
- Directed edge: $(u, v) \in E$ allows movement from $u$ to $v$ only.

## Semantics

In an undirected graph, adjacency is mutual:

```text
u is adjacent to v  ⇔  v is adjacent to u
```

In a directed graph, adjacency is asymmetric:

```text
u -> v  does not imply  v -> u
```

This affects reachability. In a directed graph, there are two notions:

- Out-reachability: paths that follow edge direction.
- In-reachability: paths that go against edge direction.

Strong connectivity requires both directions between every pair of vertices. Weak connectivity ignores direction and treats edges as undirected.

## Construction

Adjacency list construction differs only in how edges are inserted.

Directed:

```python
adj[u].append(v)
```

Undirected:

```python
adj[u].append(v)
adj[v].append(u)
```

Adjacency matrix construction follows the same rule:

```python
# directed
matrix[u][v] = True

# undirected
matrix[u][v] = True
matrix[v][u] = True
```

This difference is small in code but fundamental in meaning.

## Traversal Behavior

Breadth first search and depth first search behave differently depending on direction.

Given edges:

```text
0 -> 1 -> 2
```

Starting from `2`:

- Directed BFS or DFS cannot reach `0`.
- Undirected BFS or DFS can reach `0` if edges are treated as bidirectional.

This changes connectivity, distances, and component structure.

## Modeling Examples

Road networks:

- Two way streets: undirected graph.
- One way streets: directed graph.

Dependencies:

```text
compile -> link -> package
```

This is directed. Reversing edges changes the meaning.

Web links:

```text
A links to B
```

This is directed. A link from A to B does not imply a link from B to A.

Friendship vs following:

- Friendship is symmetric, so undirected.
- Following is asymmetric, so directed.

## Degree

In undirected graphs, each vertex has a single degree:

```text
deg(u) = number of adjacent vertices
```

In directed graphs, each vertex has:

- Out-degree: number of outgoing edges.
- In-degree: number of incoming edges.

```text
outdeg(u) = |adj[u]|
indeg(u)  = count of edges (v, u)
```

Some algorithms depend on these values. Topological sort, for example, uses in-degrees.

## Edge Count

In an undirected graph without duplicates or self loops, the maximum number of edges is:

$$
\frac{n(n - 1)}{2}
$$

In a directed graph without self loops, the maximum is:

$$
n(n - 1)
$$

This affects density and representation choice. A graph that is dense in the directed sense may still be sparse in the undirected sense.

## Conversion

Sometimes a problem requires converting between directed and undirected forms.

To ignore direction:

```python
undirected_adj[u].append(v)
undirected_adj[v].append(u)
```

To reverse a directed graph:

```python
rev[v].append(u)
```

This builds the transpose graph. It is used in strongly connected component algorithms.

## Correctness Considerations

A directed graph encodes constraints. Adding reverse edges removes those constraints.

For example, in a dependency graph:

```text
A -> B
```

means "A must happen before B". If you add `B -> A`, you introduce a cycle, making the schedule impossible.

Conversely, treating an undirected graph as directed by storing only one direction removes valid paths.

```text
u - v
```

stored as only `u -> v` prevents traversal from `v` to `u`.

Algorithms assume one model or the other:

| Algorithm | Requires |
|---|---|
| BFS, DFS | Either, but interpretation differs |
| Topological sort | Directed acyclic graph |
| Strongly connected components | Directed graph |
| Connected components | Undirected graph |
| Minimum spanning tree | Undirected graph |
| Dijkstra | Either, but direction matters |

Using the wrong model invalidates the assumptions of the algorithm.

## Common Pitfalls

A frequent error is reading undirected input but inserting only one direction. This silently converts the graph into a directed graph.

Another error is duplicating edges in a directed graph:

```python
adj[u].append(v)
adj[v].append(u)  # unintended
```

This turns the graph into an undirected graph and can hide bugs in logic.

A third error is mixing models within the same program. For example, building an undirected adjacency list but computing in-degrees as if the graph were directed.

A fourth error is forgetting that reversing a directed graph changes reachability but preserves strongly connected components. Some algorithms rely on this property.

## Takeaway

Direction is a structural property of the graph, not a minor detail. It determines reachability, connectivity, and which algorithms apply. Decide direction from the problem statement, encode it consistently in the representation, and verify that every edge insertion matches that decision.
