# 9.24 Modeling Problems as Graphs

## Problem

Many problems do not look like graphs at first. They involve tasks, states, constraints, grids, strings, or transitions. The difficulty is not the algorithm but the modeling step: deciding what the vertices and edges represent.

A correct model makes the solution straightforward. A wrong model makes even simple algorithms fail.

## Solution

Translate the problem into:

- Vertices: entities or states
- Edges: valid transitions or relationships
- Edge attributes: weights, costs, capacities, or labels

Then apply a standard graph pattern.

## Step 1: Identify Entities

Ask:

```
```
What are the objects?
```
```

These usually become vertices.

Examples:

- Cities in a map
- Tasks in a schedule
- Cells in a grid
- States in a puzzle
- Strings or words

## Step 2: Identify Transitions

Ask:

```
```
When can we move from one object to another?
```
```

Each valid move becomes an edge.

Examples:

- Road between cities
- Dependency between tasks
- Adjacent grid cells
- Valid move in a game

## Step 3: Decide Direction

Ask:

```
```
Is the relationship symmetric?
```
```

- If yes, use an undirected edge
- If no, use a directed edge

Examples:

- Friendship: undirected
- Following: directed
- Prerequisite: directed

## Step 4: Assign Weights

Ask:

```
```
Does each transition have a cost or value?
```
```

Examples:

- Distance → weight
- Time → weight
- Capacity → edge attribute
- Probability → transform or store directly

If all transitions have equal cost, treat the graph as unweighted.

## Step 5: Choose Representation

Choose based on operations:

- Traversal → adjacency list
- Edge scanning → edge list
- All-pairs → matrix

## Example 1: Grid Shortest Path

Problem:

```
```
Find shortest path from top-left to bottom-right in a grid
```
```

Model:

- Vertex: each cell `(r, c)`
- Edge: move to adjacent cell
- Weight: 1

Implicit adjacency:

```
Python



```
for dr, dc in [(1,0), (-1,0), (0,1), (0,-1)]:  
    nr = r + dr  
    nc = c + dc
```
```

Algorithm:

- BFS

## Example 2: Course Scheduling

Problem:

```
```
Determine order of courses with prerequisites
```
```

Model:

- Vertex: course
- Edge: prerequisite relation

```
```
u -> v means u must be done before v
```
```

Algorithm:

- Topological sort

## Example 3: Word Ladder

Problem:

```
```
Transform one word to another by changing one letter
```
```

Model:

- Vertex: word
- Edge: words differ by one letter

Algorithm:

- BFS

Optimization:

- Use pattern mapping instead of building all edges explicitly

## Example 4: Puzzle State Graph

Problem:

```
```
Solve a puzzle with moves
```
```

Model:

- Vertex: configuration
- Edge: valid move

Algorithm:

- BFS for shortest solution
- DFS for exploration

State generation replaces adjacency list.

## Modeling Checklist

Before coding, verify:

| Question | Check |
| --- | --- |
| Vertices | Do they represent all states or objects? |
| Edges | Are all valid transitions included? |
| Direction | Correct orientation? |
| Weights | Needed or uniform? |
| Constraints | Encoded in graph or separately? |

## Common Pitfalls

A common mistake is modeling states incorrectly. Missing states leads to incomplete search.

Another mistake is building too many edges explicitly. Some graphs are implicit and should generate neighbors on the fly.

A third mistake is ignoring direction in constraints. Many scheduling problems require directed edges.

A fourth mistake is overcomplicating the model. Simpler graphs often lead to simpler algorithms.

## Takeaway

Modeling is the central step in graph problems. Once vertices and edges are defined correctly, the algorithm is usually standard. Focus on representing the problem faithfully before choosing the algorithm.
