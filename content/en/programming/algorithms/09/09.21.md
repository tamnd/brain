# 9.21 Complexity Analysis

## Problem

You need to predict how a graph algorithm scales with input size and structure. Graph inputs vary widely in density, representation, and access patterns. A correct algorithm can still fail in practice if its complexity is mismatched to the graph.

## Solution

Analyze time and space in terms of:

- `n`: number of vertices  
- `m`: number of edges (stored edges, not theoretical maximum)  

Then account for the representation.

## Core Bounds

For adjacency lists:

| Operation | Cost |
|---|---:|
| Visit all vertices | $O(n)$ |
| Scan all edges | $O(m)$ |
| Full traversal (DFS, BFS) | $O(n + m)$ |

For adjacency matrices:

| Operation | Cost |
|---|---:|
| Visit all vertices | $O(n)$ |
| Scan all edges | $O(n^2)$ |
| Edge lookup | $O(1)$ |

For edge lists:

| Operation | Cost |
|---|---:|
| Scan all edges | $O(m)$ |
| Neighbor query | $O(m)$ |

These bounds assume constant time access to basic data structures.

## Traversal Algorithms

DFS and BFS have the same asymptotic complexity under adjacency lists:

$$
O(n + m)
$$

Each vertex is processed once. Each stored edge is examined a constant number of times.

For adjacency matrices, traversal becomes:

$$
O(n^2)
$$

because neighbor iteration scans all possible vertices.

## Degree-Based Cost

Many operations depend on vertex degree.

Neighbor iteration cost:

$$
O(\deg(u))
$$

Total cost over all vertices:

$$
\sum_{u \in V} \deg(u) = 2m \quad \text{(undirected)}
$$

$$
\sum_{u \in V} \operatorname{outdeg}(u) = m \quad \text{(directed)}
$$

This is why adjacency list traversal is linear in `m`.

## Sparse vs Dense Graphs

A graph is sparse when:

$$
m \ll n^2
$$

A graph is dense when:

$$
m \approx n^2
$$

Representation choice affects complexity:

| Graph type | Preferred representation |
|---|---|
| Sparse | Adjacency list |
| Dense | Matrix or list |

Example:

- `n = 10^5`, `m = 10^5`: sparse  
- `n = 10^4`, `m = 5 × 10^7`: dense  

An adjacency matrix for the first case is infeasible. An adjacency list for the second case may not save much memory.

## Algorithm-Specific Complexity

| Algorithm | Time |
|---|---:|
| DFS | $O(n + m)$ |
| BFS | $O(n + m)$ |
| Connected components | $O(n + m)$ |
| Topological sort | $O(n + m)$ |
| Cycle detection | $O(n + m)$ |
| SCC (Kosaraju, Tarjan) | $O(n + m)$ |
| Bipartite check | $O(n + m)$ |
| Bridges, articulation points | $O(n + m)$ |
| Dijkstra (binary heap) | $O((n + m)\log n)$ |
| Bellman-Ford | $O(nm)$ |
| Floyd-Warshall | $O(n^3)$ |
| Kruskal | $O(m \log m)$ |
| Prim (heap) | $O(m \log n)$ |

These assume adjacency list representations unless otherwise noted.

## Memory Complexity

| Representation | Space |
|---|---:|
| Adjacency list | $O(n + m)$ |
| Adjacency matrix | $O(n^2)$ |
| Edge list | $O(m)$ |
| CSR | $O(n + m)$ |

Extra arrays:

- visited, distance, parent: $O(n)$  
- edge metadata: $O(m)$  

Memory can dominate runtime for large graphs.

## Constant Factors

Asymptotic complexity hides constant factors.

Examples:

- Hash sets increase constant cost per operation.
- Python lists are dynamic arrays with amortized behavior.
- CSR improves cache locality.
- Matrix operations may benefit from contiguous memory.

For large inputs, these constants affect real performance.

## Directed vs Undirected Cost

In adjacency lists:

- Undirected graph stores each edge twice
- Directed graph stores each edge once

Traversal still runs in $O(n + m)$, but:

```text
undirected scan cost ≈ 2m
directed scan cost ≈ m
```

This matters for memory and constant factors.

## Worst Case vs Typical Case

Worst-case bounds assume adversarial inputs.

Typical cases may be faster:

- BFS often terminates early when target is found.
- Dijkstra may not explore all vertices.
- Sparse graphs reduce scanning work.

However, algorithm design should respect worst-case guarantees unless probabilistic assumptions are valid.

## Amortized Analysis

Some operations are amortized:

- Dynamic array append: $O(1)$ amortized  
- Union-Find with path compression: nearly constant  

Example:

Union-Find operations:

$$
O(\alpha(n))
$$

where $\alpha(n)$ is the inverse Ackermann function, which grows extremely slowly.

## Practical Estimation

To estimate runtime:

1. Identify dominant loop
2. Count how many times it runs
3. Multiply by cost per iteration

Example BFS:

```python id="m2cczt"
while queue:
    u = queue.popleft()
    for v in adj[u]:
        ...
```

Outer loop runs `n` times  
Inner loop runs once per edge  

Total:

$$
O(n + m)
$$

## Common Pitfalls

A common mistake is analyzing adjacency list algorithms as $O(n^2)$. This ignores that only existing edges are scanned.

Another mistake is ignoring representation. BFS is $O(n + m)$ with lists, but $O(n^2)$ with matrices.

A third mistake is forgetting hidden costs:

```python id="8fjlwm"
if v in adj[u]:
```

This is $O(\deg(u))$ unless using a set.

A fourth mistake is assuming small inputs. Algorithms like Floyd-Warshall become impractical quickly.

## Takeaway

Graph complexity depends on both the algorithm and the representation. Use `O(n + m)` as the baseline for traversal on adjacency lists. Check whether your graph is sparse or dense, and ensure the chosen representation matches the dominant operation.
