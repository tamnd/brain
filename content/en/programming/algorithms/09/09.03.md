# 9.3 Adjacency Matrices

## Problem

You need a graph representation that answers edge-existence queries quickly. An adjacency list is efficient for scanning neighbors, but testing whether a particular edge exists may require scanning the whole neighbor list of one vertex. An adjacency matrix makes that lookup direct.

## Solution

For a graph with $n$ vertices numbered from `0` to `n - 1`, store a two-dimensional table `matrix`.

For an unweighted graph:

```text
matrix[u][v] = true if there is an edge from u to v
matrix[u][v] = false otherwise
```

For a weighted graph:

```text
matrix[u][v] = weight of the edge from u to v
matrix[u][v] = INF if no edge exists
```

For an undirected graph, the matrix is symmetric:

$$
matrix[u][v] = matrix[v][u]
$$

For a directed graph, symmetry is not required.

## Basic Example

Consider this directed graph:

```text
0 -> 1
0 -> 2
2 -> 1
```

The adjacency matrix is:

```text
    0      1      2
0:  false  true   true
1:  false  false  false
2:  false  true   false
```

The entry at row `0`, column `2` is `true`, so there is an edge from `0` to `2`. The entry at row `2`, column `0` is `false`, so there is no edge from `2` to `0`.

For an undirected graph:

```text
0 - 1
1 - 2
```

the matrix is:

```text
    0      1      2
0:  false  true   false
1:  true   false  true
2:  false  true   false
```

The entries mirror across the diagonal.

## Implementation

For an unweighted directed graph:

```python
def build_directed_matrix(n, edges):
    matrix = [[False for _ in range(n)] for _ in range(n)]

    for u, v in edges:
        matrix[u][v] = True

    return matrix
```

For an unweighted undirected graph:

```python
def build_undirected_matrix(n, edges):
    matrix = [[False for _ in range(n)] for _ in range(n)]

    for u, v in edges:
        matrix[u][v] = True
        matrix[v][u] = True

    return matrix
```

The lookup operation is constant time:

```python
if matrix[u][v]:
    print("edge exists")
```

This is the main reason to use an adjacency matrix.

## Weighted Matrices

For weighted graphs, use a sentinel value for missing edges.

```python
def build_weighted_matrix(n, edges):
    INF = 10**18
    matrix = [[INF for _ in range(n)] for _ in range(n)]

    for i in range(n):
        matrix[i][i] = 0

    for u, v, w in edges:
        matrix[u][v] = w

    return matrix
```

The diagonal is often set to `0` because the distance from a vertex to itself is zero. This convention is especially useful for shortest path algorithms such as Floyd-Warshall.

For an undirected weighted graph, store both directions:

```python
matrix[u][v] = w
matrix[v][u] = w
```

If repeated edges are allowed, decide whether to keep the latest edge, the smallest weight, the largest capacity, or all parallel edges in a different structure.

For shortest path problems, the usual rule is:

```python
matrix[u][v] = min(matrix[u][v], w)
```

## Traversal Pattern

An adjacency matrix can still be used for traversal, but neighbor iteration costs more.

```python
for v in range(n):
    if matrix[u][v]:
        process_edge(u, v)
```

This scans every possible vertex, not only actual neighbors. If a vertex has only three outgoing edges but the graph has one million vertices, the loop still checks one million entries.

For a full traversal:

```python
for u in range(n):
    for v in range(n):
        if matrix[u][v]:
            process_edge(u, v)
```

This costs $O(n^2)$ even if the graph has very few edges.

## Complexity

For a graph with $n$ vertices and $m$ edges:

| Operation | Adjacency matrix cost |
|---|---:|
| Test whether `u -> v` exists | $O(1)$ |
| Add an edge | $O(1)$ |
| Remove an edge | $O(1)$ |
| Iterate neighbors of `u` | $O(n)$ |
| Iterate all edges | $O(n^2)$ |
| Memory usage | $O(n^2)$ |

The memory cost is the main limitation. A matrix for $100,000$ vertices has $10^{10}$ entries. Even if each entry used only one byte, that would require about ten gigabytes before accounting for object overhead. In high-level languages, the actual memory cost may be much larger.

## When to Use It

Use an adjacency matrix when the graph is dense or when edge lookup dominates the algorithm.

A dense graph has many possible edges. In a directed graph with $n$ vertices, the maximum number of edges is roughly $n^2$. In an undirected simple graph, the maximum number of edges is roughly:

$$
\frac{n(n-1)}{2}
$$

If $m$ is close to $n^2$, then an adjacency list no longer saves much memory, and a matrix may be simpler.

Adjacency matrices are also natural for algorithms that already use all pairs of vertices, such as Floyd-Warshall. That algorithm updates a distance table indexed by two vertices, so a matrix representation matches the computation directly.

## Boolean Matrix vs Numeric Matrix

An unweighted graph can use a Boolean matrix. This is clear and safe:

```python
matrix = [[False for _ in range(n)] for _ in range(n)]
```

A weighted graph should avoid using `0` to mean "no edge" unless zero weight edges are impossible by problem statement. Otherwise, the representation becomes ambiguous.

Bad representation when zero weight edges may exist:

```text
matrix[u][v] = 0
```

This could mean either "no edge" or "edge of weight zero."

Prefer an explicit sentinel:

```python
INF = 10**18
matrix[u][v] = INF
```

or use `None`:

```python
matrix[u][v] = None
```

Use `INF` when arithmetic on distances is expected. Use `None` when the graph stores general edge objects or when arithmetic is not the main operation.

## Correctness Considerations

The matrix must preserve direction. For a directed graph, writing both entries creates a different graph.

```python
# Directed edge only
matrix[u][v] = True
```

For an undirected graph, failing to mirror the edge breaks symmetry.

```python
# Undirected edge
matrix[u][v] = True
matrix[v][u] = True
```

Self loops live on the diagonal:

```python
matrix[u][u] = True
```

Some algorithms assume no self loops. In that case, reject them while reading input or ignore diagonal entries explicitly.

For weighted graphs, repeated edges require a policy. Suppose the input contains:

```text
0 -> 1 with weight 10
0 -> 1 with weight 3
```

For shortest path algorithms, the correct stored value is usually `3`. For capacity problems, the values may need to be summed. The representation alone cannot decide this; the problem contract must decide it.

## Common Pitfalls

A frequent Python bug is creating rows by multiplication:

```python
matrix = [[False] * n] * n
```

This creates multiple references to the same row. Updating one row updates every row.

Use a list comprehension instead:

```python
matrix = [[False for _ in range(n)] for _ in range(n)]
```

Another mistake is using an adjacency matrix for a large sparse graph. The code may be simple, but the memory cost can dominate immediately.

A third mistake is treating a weighted matrix like a Boolean matrix:

```python
if matrix[u][v]:
    ...
```

This fails when a valid edge has weight `0`, and it also treats `INF` as true in Python. Test explicitly:

```python
if matrix[u][v] != INF:
    ...
```

A fourth mistake is forgetting that matrix traversal has cost $O(n^2)$. This matters when the number of actual edges $m$ is much smaller than $n^2$.

## Takeaway

An adjacency matrix gives constant time edge lookup at the cost of quadratic memory and quadratic edge scanning. It is appropriate for dense graphs, small graphs, and all-pairs algorithms. For sparse graphs and traversal-heavy algorithms, an adjacency list is usually the better default.
