# 9.2 Adjacency Lists

## Problem

You need a graph representation that supports efficient neighbor iteration. Most graph algorithms, including depth first search, breadth first search, connected components, topological sort, and Dijkstra's algorithm, repeatedly ask the same question: "Which vertices can I visit next from this vertex?"

An adjacency list answers this question directly.

## Solution

Represent the graph by storing, for each vertex, the list of vertices adjacent to it.

For an unweighted directed graph:

```text
adj[u] = list of vertices v such that there is an edge u -> v
```

For an unweighted undirected graph:

```text
adj[u] = list of vertices connected to u
```

In an undirected graph, each edge is stored twice: once from $u$ to $v$, and once from $v$ to $u$.

```text
addEdge(u, v):
  adj[u].append(v)
  adj[v].append(u)
```

For a directed graph, store only the forward edge.

```text
addEdge(u, v):
  adj[u].append(v)
```

## Basic Example

Consider the undirected graph:

```text
0 - 1 - 2
    |
    3
```

Its adjacency list is:

```text
0: [1]
1: [0, 2, 3]
2: [1]
3: [1]
```

The list for vertex `1` contains all vertices reachable from `1` in one step.

A directed version:

```text
0 -> 1 -> 2
     |
     v
     3
```

has the adjacency list:

```text
0: [1]
1: [2, 3]
2: []
3: []
```

Here, `2` and `3` have empty lists because no outgoing edges are shown from them.

## Implementation

For vertices numbered from `0` to `n - 1`, use an array of lists.

```python
def build_directed_graph(n, edges):
    adj = [[] for _ in range(n)]

    for u, v in edges:
        adj[u].append(v)

    return adj
```

For an undirected graph:

```python
def build_undirected_graph(n, edges):
    adj = [[] for _ in range(n)]

    for u, v in edges:
        adj[u].append(v)
        adj[v].append(u)

    return adj
```

This representation is compact for sparse graphs. If the graph has $n$ vertices and $m$ edges, the memory cost is:

```text
Directed graph:    O(n + m)
Undirected graph:  O(n + 2m), usually written O(n + m)
```

The constant factor differs, but the asymptotic bound is the same.

## Weighted Edges

For weighted graphs, store both the neighbor and the weight.

```python
def build_weighted_directed_graph(n, edges):
    adj = [[] for _ in range(n)]

    for u, v, w in edges:
        adj[u].append((v, w))

    return adj
```

For example:

```text
edges = [
  (0, 1, 7),
  (0, 2, 3),
  (2, 1, 2),
]
```

produces:

```text
0: [(1, 7), (2, 3)]
1: []
2: [(1, 2)]
```

The meaning is that vertex `0` has an edge to `1` with weight `7`, and an edge to `2` with weight `3`.

## Traversal Pattern

Most adjacency list algorithms have this shape:

```python
for v in adj[u]:
    process_edge(u, v)
```

For weighted graphs:

```python
for v, w in adj[u]:
    process_edge(u, v, w)
```

This makes adjacency lists especially good for traversal. The algorithm pays only for the edges it actually scans.

For a full traversal over all vertices and all outgoing edges:

```python
for u in range(n):
    for v in adj[u]:
        process_edge(u, v)
```

The total running time is $O(n + m)$ for a directed graph, because every vertex is visited once by the outer loop and every stored edge is scanned once by the inner loop.

## Correctness Considerations

The adjacency list must match the graph model.

For a directed graph, storing a reverse edge changes reachability.

```python
# Correct for directed graph
adj[u].append(v)

# Wrong unless the graph is undirected
adj[v].append(u)
```

For an undirected graph, forgetting the reverse edge makes the graph behave like a directed graph.

```python
# Correct for undirected graph
adj[u].append(v)
adj[v].append(u)
```

If the graph is weighted, do not discard the weight unless the algorithm genuinely ignores it. Breadth first search can ignore weights only when every edge has the same cost. Dijkstra's algorithm and shortest path computations need the weights.

## Duplicate Edges

An adjacency list can represent repeated edges naturally.

```text
0: [1, 1, 1]
```

This means there are three stored edges from `0` to `1`. That may be correct for a multigraph, but it may be wrong for a simple graph.

Duplicate edges can affect:

| Algorithm or property | Effect of duplicates |
|---|---|
| Degree counting | Degree may be too large |
| Path counting | Counts may be inflated |
| Flow algorithms | Parallel capacities may need merging |
| Traversal | Extra scans increase runtime |
| Simple graph logic | Assumptions may break |

For simple graphs, either reject duplicates while building the graph or normalize the lists after construction.

```python
def deduplicate_adjacency(adj):
    return [list(sorted(set(neighbors))) for neighbors in adj]
```

This changes neighbor order, so use it only when order does not matter.

## Vertex Labels

External vertex names are often strings or tuples:

```text
"NYC"
"Boston"
"Chicago"
```

Most algorithms are simpler with integer IDs. Use a mapping layer.

```python
def compress_labels(edges):
    id_of = {}
    labels = []

    def get_id(x):
        if x not in id_of:
            id_of[x] = len(labels)
            labels.append(x)
        return id_of[x]

    compressed = []
    for a, b in edges:
        u = get_id(a)
        v = get_id(b)
        compressed.append((u, v))

    return compressed, labels, id_of
```

Then build the adjacency list using the integer IDs. Keep `labels[id]` if you need to convert results back to external names.

## Complexity

For a graph with $n$ vertices and $m$ edges:

| Operation | Adjacency list cost |
|---|---:|
| Iterate neighbors of `u` | $O(\deg(u))$ |
| Iterate all edges | $O(n + m)$ |
| Add an edge | $O(1)$ amortized |
| Test whether `u -> v` exists | $O(\deg(u))$ |
| Store sparse graph | $O(n + m)$ |

The main weakness is edge lookup. If you often ask whether a particular edge exists, an adjacency matrix or a set-backed adjacency list may be better.

```python
adj = [set() for _ in range(n)]
adj[u].add(v)

if v in adj[u]:
    ...
```

This gives expected $O(1)$ lookup, but it uses more memory and gives less predictable iteration order.

## Common Pitfalls

A common mistake is allocating the outer list incorrectly:

```python
adj = [[]] * n
```

This creates `n` references to the same inner list. Appending to one vertex appends to all vertices.

Use this instead:

```python
adj = [[] for _ in range(n)]
```

Another mistake is mixing directed and undirected construction in the same helper. Keep separate functions unless the direction flag is explicit.

A third mistake is using one based vertex IDs directly in a zero based array. Convert input carefully:

```python
u -= 1
v -= 1
```

A fourth mistake is assuming neighbor order is irrelevant. Some algorithms return different valid answers depending on neighbor order, especially DFS, topological sort, and path reconstruction. For deterministic tests, sort each adjacency list after construction.

## Takeaway

An adjacency list stores exactly what traversal algorithms need: the outgoing neighbors of each vertex. It is the default representation for sparse graphs and for algorithms that scan edges. The representation is simple, but correctness depends on matching the graph model: directed edges must stay directed, undirected edges must be inserted both ways, and weights or duplicate edges must be handled according to the problem contract.
