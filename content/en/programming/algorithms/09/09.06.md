# 9.6 Weighted Graphs

## Problem

You need to model a graph where edges carry a numeric value. The value may represent distance, time, cost, capacity, probability, penalty, or reward. A plain graph only records whether two vertices are connected. A weighted graph records how expensive or valuable each connection is.

## Solution

Store each edge with an extra value called its weight.

For a directed weighted graph:

$$
(u, v, w)
$$

means that there is an edge from $u$ to $v$ with weight $w$.

For an undirected weighted graph:

$$
\{u, v\}, w
$$

means that $u$ and $v$ are connected by an edge with weight $w$ in both directions.

In code, a weighted adjacency list usually stores pairs inside each neighbor list:

```python
adj[u].append((v, w))
```

For an undirected graph:

```python
adj[u].append((v, w))
adj[v].append((u, w))
```

## Basic Example

Suppose the graph represents travel times between cities.

```text
0 --7-- 1
0 --3-- 2
2 --2-- 1
```

The weighted edge list is:

```text
(0, 1, 7)
(0, 2, 3)
(2, 1, 2)
```

The weighted adjacency list is:

```text
0: [(1, 7), (2, 3)]
1: [(0, 7), (2, 2)]
2: [(0, 3), (1, 2)]
```

The edge from `0` to `1` exists, and its cost is `7`.

## Implementation

For a directed weighted graph:

```python
def build_weighted_directed_graph(n, edges):
    adj = [[] for _ in range(n)]

    for u, v, w in edges:
        adj[u].append((v, w))

    return adj
```

For an undirected weighted graph:

```python
def build_weighted_undirected_graph(n, edges):
    adj = [[] for _ in range(n)]

    for u, v, w in edges:
        adj[u].append((v, w))
        adj[v].append((u, w))

    return adj
```

A traversal over weighted edges has this form:

```python
for v, w in adj[u]:
    process_edge(u, v, w)
```

The neighbor `v` tells where the edge goes. The weight `w` tells the cost or value of using that edge.

## Weight Meaning

The algorithm depends on what the weight means.

| Weight meaning | Typical algorithm |
|---|---|
| Distance or time | Shortest path |
| Cost | Minimum spanning tree or shortest path |
| Capacity | Flow algorithm |
| Profit or reward | Optimization with maximization |
| Probability | Often transform using logarithms |
| Penalty | Dynamic programming or shortest path |

Do not treat all weights as interchangeable. A shortest path algorithm minimizes total weight. A flow algorithm constrains how much can pass through an edge. A maximum probability path may multiply probabilities rather than add them, unless transformed.

## Path Cost

For many graph problems, the cost of a path is the sum of edge weights.

If a path is:

$$
v_0 \to v_1 \to v_2 \to \cdots \to v_k
$$

then its total cost is:

$$
w(v_0, v_1) + w(v_1, v_2) + \cdots + w(v_{k-1}, v_k)
$$

This additive model is used by Dijkstra, Bellman-Ford, Floyd-Warshall, and many dynamic programming formulations on graphs.

## Nonnegative Weights

Many efficient shortest path algorithms require nonnegative weights.

Dijkstra's algorithm assumes:

$$
w(u, v) \ge 0
$$

for every edge. If a graph has negative edges, Dijkstra's greedy choice can become invalid.

Use Bellman-Ford when negative edges may exist and you need single-source shortest paths. Use Floyd-Warshall when the graph is small enough and all-pairs distances are needed.

## Zero Weights

Zero weight edges are valid in many problems.

```text
u -> v with weight 0
```

means that moving from `u` to `v` has no cost. This is different from having no edge.

For this reason, avoid using `0` as a sentinel for missing edges in a weighted adjacency matrix. Use `INF` or `None`.

```python
INF = 10**18
matrix = [[INF for _ in range(n)] for _ in range(n)]
```

Then a zero weight edge can be stored safely:

```python
matrix[u][v] = 0
```

## Negative Weights

Negative weights are also valid in some models.

```text
u -> v with weight -5
```

This may represent a discount, gain, or offset. Negative weights require special care because cycles can reduce path cost indefinitely.

A negative cycle is a cycle whose total weight is negative. If a path can reach such a cycle and then leave it, there may be no finite shortest path.

```text
A -> B weight 2
B -> C weight -5
C -> A weight 1
```

The cycle has total weight:

$$
2 + (-5) + 1 = -2
$$

Going around the cycle repeatedly makes the path cost smaller each time.

## Repeated Weighted Edges

Repeated edges require a policy.

```text
0 -> 1 weight 10
0 -> 1 weight 3
```

For shortest path problems, keep the smaller weight:

```python
best[u][v] = min(best[u][v], w)
```

For flow problems, repeated capacities may need to be summed:

```python
capacity[u][v] += c
```

For multigraph problems, keep both edges:

```python
adj[u].append((v, w1))
adj[u].append((v, w2))
```

The correct policy comes from the problem definition.

## Correctness Considerations

The graph representation must preserve the weight attached to each edge. Dropping the weight changes the problem from weighted reachability to unweighted reachability.

For example, in this graph:

```text
0 -> 1 weight 100
0 -> 2 weight 1
2 -> 1 weight 1
```

The fewest-edge path from `0` to `1` is:

```text
0 -> 1
```

Its cost is `100`.

The minimum-cost path is:

```text
0 -> 2 -> 1
```

Its cost is `2`.

Breadth first search finds the fewest-edge path, not the cheapest weighted path.

## Common Pitfalls

A common mistake is using BFS on a weighted graph. BFS is correct for shortest paths only when every edge has the same weight.

Another mistake is using Dijkstra on graphs with negative edges. The algorithm may return a distance that appears plausible but is wrong.

A third mistake is using `if matrix[u][v]` to test edge existence in a weighted matrix. This fails for zero weight edges and treats large sentinel values as true.

A fourth mistake is forgetting to mirror weights in an undirected graph. Both directions must store the same weight unless the model is actually directed.

## Takeaway

A weighted graph extends the graph model by attaching values to edges. The meaning of those values determines the correct algorithm. Preserve weights in the representation, distinguish zero weight from missing edge, and check algorithm assumptions before solving the problem.
