# 9.20 Testing Graph Code

## Problem

You need confidence that a graph algorithm handles direction, disconnected components, self loops, duplicate edges, isolated vertices, and boundary cases. Graph code often fails on small modeling mistakes rather than on the main algorithmic idea.

Testing should verify both the algorithm and the graph representation contract.

## Solution

Build a small suite of graph fixtures. Each fixture should isolate one structural feature.

```python
def make_path_graph(n):
    adj = [[] for _ in range(n)]

    for u in range(n - 1):
        v = u + 1
        adj[u].append(v)
        adj[v].append(u)

    return adj
```

```python
def make_cycle_graph(n):
    adj = make_path_graph(n)

    if n > 1:
        adj[0].append(n - 1)
        adj[n - 1].append(0)

    return adj
```

```python
def make_directed_chain(n):
    adj = [[] for _ in range(n)]

    for u in range(n - 1):
        adj[u].append(u + 1)

    return adj
```

These helpers make tests readable. A test should describe the graph shape, not hide it behind a large input blob.

## Core Fixtures

Use these cases repeatedly:

| Case | Purpose |
|---|---|
| Empty graph | Checks boundary behavior |
| One vertex | Checks singleton logic |
| Isolated vertices | Checks outer traversal loops |
| Path graph | Checks simple reachability |
| Cycle graph | Checks cycle logic |
| Complete graph | Checks dense behavior |
| Disconnected graph | Checks component handling |
| Directed chain | Checks direction |
| Directed cycle | Checks directed cycle detection |
| Self loop | Checks loop policy |
| Duplicate edges | Checks multigraph policy |

A good graph test suite is small but structural. It should catch representation errors before large random tests are used.

## Testing Connectivity

For connected components, test isolated vertices and disconnected pieces.

```python
def test_connected_components():
    adj = [
        [1],
        [0, 2],
        [1],
        [4],
        [3],
        [],
    ]

    comp, count = connected_components(adj)

    assert count == 3
    assert comp[0] == comp[1] == comp[2]
    assert comp[3] == comp[4]
    assert comp[5] != comp[0]
    assert comp[5] != comp[3]
```

This verifies that vertex `5` is not ignored.

## Testing Direction

Direction bugs are common. A directed graph should not accidentally behave like an undirected graph.

```python
def test_directed_reachability():
    adj = [
        [1],
        [2],
        [],
    ]

    dist, _ = bfs(2, adj)

    assert dist[0] == -1
    assert dist[1] == -1
    assert dist[2] == 0
```

If this test fails, the graph was probably built with reverse edges.

## Testing Cycle Detection

Use separate tests for directed and undirected cycle detection.

```python
def test_undirected_cycle():
    adj = make_cycle_graph(4)
    assert has_cycle_undirected(adj)
```

```python
def test_directed_cycle():
    adj = [
        [1],
        [2],
        [0],
    ]

    assert has_cycle_directed(adj)
```

Also test acyclic cases.

```python
def test_directed_acyclic():
    adj = [
        [1, 2],
        [3],
        [3],
        [],
    ]

    assert not has_cycle_directed(adj)
```

## Testing Topological Sort

Topological sort may return multiple valid answers, so do not assert one exact order unless the graph forces it. Instead, verify the property.

```python
def is_valid_topological_order(order, adj):
    if order is None:
        return False

    pos = {u: i for i, u in enumerate(order)}

    for u in range(len(adj)):
        for v in adj[u]:
            if pos[u] >= pos[v]:
                return False

    return True
```

```python
def test_topological_sort():
    adj = [
        [2],
        [2],
        [3],
        [],
    ]

    order = topological_sort(adj)
    assert is_valid_topological_order(order, adj)
```

For a cyclic graph:

```python
def test_topological_sort_cycle():
    adj = [
        [1],
        [2],
        [0],
    ]

    assert topological_sort(adj) is None
```

## Testing BFS Distances

BFS has a clear oracle on small graphs.

```python
def test_bfs_distances():
    adj = [
        [1, 2],
        [0, 3],
        [0],
        [1],
    ]

    dist, parent = bfs(0, adj)

    assert dist == [0, 1, 1, 2]
    assert parent[0] == -1
```

For parent arrays, avoid over-specifying when multiple shortest paths exist. Check that the parent relation is valid.

```python
def check_bfs_parent_tree(start, dist, parent, adj):
    for v in range(len(adj)):
        if v == start:
            assert parent[v] == -1
            continue

        if dist[v] == -1:
            assert parent[v] == -1
            continue

        p = parent[v]
        assert p in adj[v] or v in adj[p]
        assert dist[p] + 1 == dist[v]
```

## Property Tests

Some graph properties have simple invariants.

For an undirected graph:

$$
\sum_{v \in V} \deg(v) = 2|E|
$$

Test it directly.

```python
def test_handshaking(edges, n):
    adj = build_undirected_graph(n, edges)

    degree_sum = sum(len(neighbors) for neighbors in adj)

    assert degree_sum == 2 * len(edges)
```

For a directed graph:

$$
\sum_v indeg(v) = \sum_v outdeg(v) = |E|
$$

```python
def test_directed_degree_sums(edges, n):
    adj = build_directed_graph(n, edges)

    indeg = in_degrees(adj)
    outdeg = out_degrees(adj)

    assert sum(indeg) == len(edges)
    assert sum(outdeg) == len(edges)
```

These tests catch many construction bugs.

## Randomized Cross-Checks

For small graphs, compare two independent implementations.

Example: compare BFS reachability against transitive closure from repeated DFS.

```python
def reachable_by_dfs(start, adj):
    seen = [False] * len(adj)

    def visit(u):
        seen[u] = True
        for v in adj[u]:
            if not seen[v]:
                visit(v)

    visit(start)
    return seen
```

```python
def test_bfs_against_dfs(adj, start):
    dist, _ = bfs(start, adj)
    seen = reachable_by_dfs(start, adj)

    for v in range(len(adj)):
        assert (dist[v] != -1) == seen[v]
```

This kind of test is useful because the two algorithms fail in different ways.

## Common Pitfalls

A common mistake is testing only connected graphs. Many traversal bugs appear only when the graph has multiple components.

Another mistake is asserting exact traversal order. DFS and BFS order may depend on neighbor ordering. Test structural properties unless the order is part of the contract.

A third mistake is forgetting graph direction in the oracle. A path that exists in the undirected version may not exist in the directed version.

A fourth mistake is omitting empty and singleton graphs. Boundary cases often reveal bad initialization.

## Takeaway

Testing graph code works best when tests are structural. Use small graphs with known properties, test direction explicitly, verify invariants, and avoid over-specifying arbitrary traversal order. Randomized tests are useful after the deterministic fixtures are in place.
