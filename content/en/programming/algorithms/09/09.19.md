# 9.19 Graph Representation Tradeoffs

## Problem

You need to choose a graph representation before implementing an algorithm. The wrong representation can turn a clean linear algorithm into a slow one, or make a large graph impossible to store. The right representation depends on the operations the algorithm performs most often.

## Solution

Choose the representation from the access pattern.

| Representation | Best for | Weakness |
|---|---|---|
| Adjacency list | Neighbor iteration on sparse graphs | Slow edge lookup |
| Adjacency matrix | Constant time edge lookup | Quadratic memory |
| Edge list | Scanning, sorting, and input parsing | Slow neighbor lookup |
| Set-backed adjacency list | Neighbor iteration plus fast edge lookup | Higher memory cost |
| Compressed sparse row | Large static graphs | Harder updates |

Most graph algorithms in this book use adjacency lists because they scan neighbors. Edge lists are common at input boundaries and in edge-centric algorithms. Matrices are best for dense graphs and all-pairs algorithms.

## Adjacency List

An adjacency list stores outgoing neighbors for each vertex.

```text
0: [1, 4]
1: [2]
2: []
3: [0]
4: [2, 3]
```

Use it when the algorithm repeatedly does this:

```python
for v in adj[u]:
    process(u, v)
```

This is the natural form for DFS, BFS, connected components, topological sort, Dijkstra, and many dynamic programming algorithms on graphs.

For a graph with $n$ vertices and $m$ edges, memory is:

$$
O(n + m)
$$

Neighbor iteration over vertex `u` costs:

$$
O(\deg(u))
$$

Edge lookup costs:

$$
O(\deg(u))
$$

unless the neighbor list is sorted or backed by a set.

## Adjacency Matrix

An adjacency matrix stores one entry for every possible pair of vertices.

```text
matrix[u][v] = true if edge u -> v exists
```

Use it when the algorithm frequently asks:

```python
if matrix[u][v]:
    ...
```

The lookup cost is:

$$
O(1)
$$

The memory cost is:

$$
O(n^2)
$$

This is acceptable for small or dense graphs. It is usually a poor choice for large sparse graphs.

Adjacency matrices also fit algorithms that update all pairs of vertices, such as Floyd-Warshall.

## Edge List

An edge list stores edges directly.

```text
[(0, 1), (1, 2), (4, 3)]
```

For weighted graphs:

```text
[(0, 1, 7), (1, 2, 4), (4, 3, 9)]
```

Use it when the algorithm scans or sorts edges:

```python
for u, v, w in edges:
    process(u, v, w)
```

This representation is natural for Kruskal's algorithm, Bellman-Ford, and graph input parsing.

It is weak for traversal because finding the neighbors of a vertex requires scanning the entire edge list.

## Set-Backed Adjacency Lists

A set-backed adjacency list stores neighbors in sets instead of lists.

```python
adj = [set() for _ in range(n)]
adj[u].add(v)
```

This gives expected constant time edge lookup:

```python
if v in adj[u]:
    ...
```

It also supports neighbor iteration.

The tradeoff is memory. Hash sets store more metadata than arrays or lists. Iteration order may also be nondeterministic unless the language preserves insertion order or you sort before output.

Use this representation when both operations matter:

- Iterate over all neighbors.
- Frequently test whether a specific edge exists.

## Compressed Sparse Row

Compressed sparse row, often called CSR, stores all adjacency lists in two arrays:

```text
offsets[u] to offsets[u + 1] gives the range of neighbors of u
neighbors stores all neighbor ids contiguously
```

Example:

```text
offsets  = [0, 2, 3, 3, 4]
neighbors = [1, 4, 2, 0]
```

This means:

```text
0: neighbors[0:2] = [1, 4]
1: neighbors[2:3] = [2]
2: neighbors[3:3] = []
3: neighbors[3:4] = [0]
```

CSR is useful for large static graphs because it is compact and cache-friendly. It is poor for frequent edge insertion or deletion because updating the middle of the neighbor array is expensive.

## Choosing by Density

A sparse graph has far fewer edges than the maximum possible number of edges. A dense graph has a large fraction of possible edges.

For a directed simple graph without self loops, the maximum number of edges is:

$$
n(n - 1)
$$

For an undirected simple graph, the maximum is:

$$
\frac{n(n - 1)}{2}
$$

When $m$ is close to $n^2$, an adjacency matrix may be reasonable. When $m$ is closer to $n$, an adjacency list is usually much better.

## Choosing by Algorithm

| Algorithm | Typical representation |
|---|---|
| DFS | Adjacency list |
| BFS | Adjacency list |
| Connected components | Adjacency list |
| Topological sort | Adjacency list |
| Dijkstra | Weighted adjacency list |
| Bellman-Ford | Edge list |
| Floyd-Warshall | Matrix |
| Kruskal | Weighted edge list |
| Prim | Weighted adjacency list or matrix |
| SCC algorithms | Adjacency list |
| Flow algorithms | Residual adjacency list |

The representation should match the inner loop of the algorithm. If the inner loop scans neighbors, use adjacency lists. If it scans all edges, use edge lists. If it updates all vertex pairs, use a matrix.

## Correctness Considerations

Representation choices can affect correctness when they silently change the graph model.

For an undirected graph, an adjacency list usually stores each edge twice. An edge list usually stores each edge once. Mixing these conventions can double-count edges.

For a weighted graph, converting from an edge list to an adjacency list must preserve weights.

```python
adj[u].append((v, w))
```

For a directed graph, adding reverse edges changes reachability.

```python
adj[u].append(v)      # directed
adj[v].append(u)      # only for undirected
```

For a simple graph, sets remove duplicates. For a multigraph, sets are wrong because parallel edges are meaningful.

## Performance Considerations

Memory layout matters on large graphs. A list of lists is convenient, but each inner list may be allocated separately. CSR stores neighbors contiguously and can be faster for full scans.

Hash sets give faster lookup but poorer locality. Matrices give direct lookup but often waste memory on absent edges.

For small graphs, clarity usually matters more than micro-optimization. For large graphs, representation can dominate runtime and memory.

## Common Pitfalls

A common mistake is using an adjacency matrix because it feels simple, then running out of memory on sparse input.

Another mistake is using an edge list for BFS or DFS and scanning the whole edge list for every vertex.

A third mistake is using a set-backed adjacency list for a multigraph, losing repeated edges.

A fourth mistake is forgetting isolated vertices when building from an edge list. Keep `n` separately; the edge list alone may not mention every vertex.

## Takeaway

Graph representation is an algorithmic choice. Adjacency lists serve traversal. Matrices serve edge lookup and all-pairs algorithms. Edge lists serve scanning and sorting. CSR serves large static graphs. Match the representation to the algorithm's inner loop and preserve the graph model exactly.
