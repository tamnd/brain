# 9.4 Edge Lists

## Problem

You need a graph representation that stores edges directly, without organizing them by source vertex or by a two-dimensional matrix. This is useful when the main operation is scanning, sorting, filtering, or transforming edges rather than repeatedly asking for the neighbors of one vertex.

An edge list is the simplest graph representation. It records the graph as a sequence of edges.

## Solution

Store each edge as a pair for an unweighted graph:

```text id="5ri5e8"
(u, v)
```

Store each edge as a triple for a weighted graph:

```text id="dxawq4"
(u, v, w)
```

For a directed graph, `(u, v)` means an edge from `u` to `v`.

For an undirected graph, `(u, v)` means an edge connecting `u` and `v`. Unlike an adjacency list, an undirected edge is usually stored once in an edge list.

```text id="pjycff"
edges = [
  (0, 1),
  (1, 2),
  (1, 3),
]
```

This representation says what edges exist, but it does not make neighbor lookup fast.

## Basic Example

Consider the undirected graph:

```text id="o78gtj"
0 - 1 - 2
    |
    3
```

An edge list representation is:

```text id="2gydxr"
[(0, 1), (1, 2), (1, 3)]
```

A directed version:

```text id="dp9z68"
0 -> 1
1 -> 2
1 -> 3
```

uses the same textual list, but the interpretation changes:

```text id="8g38uu"
[(0, 1), (1, 2), (1, 3)]
```

Now each pair is ordered. The edge `(1, 2)` does not imply the edge `(2, 1)`.

## Implementation

For an unweighted graph, the representation can be a plain list of pairs.

```python id="9skxuq"
edges = [
    (0, 1),
    (1, 2),
    (1, 3),
]
```

For a weighted graph, use triples.

```python id="g47wvk"
edges = [
    (0, 1, 4),
    (1, 2, 7),
    (1, 3, 2),
]
```

To scan all edges:

```python id="sq7b11"
for u, v in edges:
    process_edge(u, v)
```

For weighted edges:

```python id="prie95"
for u, v, w in edges:
    process_edge(u, v, w)
```

This gives direct access to every edge exactly once, which is useful for algorithms that are edge-centric.

## Building Other Representations

An edge list is often used as the input format, then converted into another representation.

To build an adjacency list for a directed graph:

```python id="idlg20"
def edge_list_to_directed_adj(n, edges):
    adj = [[] for _ in range(n)]

    for u, v in edges:
        adj[u].append(v)

    return adj
```

For an undirected graph:

```python id="ac7wzh"
def edge_list_to_undirected_adj(n, edges):
    adj = [[] for _ in range(n)]

    for u, v in edges:
        adj[u].append(v)
        adj[v].append(u)

    return adj
```

To build an adjacency matrix:

```python id="q2d3ta"
def edge_list_to_matrix(n, edges):
    matrix = [[False for _ in range(n)] for _ in range(n)]

    for u, v in edges:
        matrix[u][v] = True

    return matrix
```

This conversion is common because input files frequently list edges one per line, while algorithms often prefer adjacency lists or matrices.

## Weighted Edge Lists

Weighted edge lists are especially useful when the algorithm needs to sort edges by weight. Kruskal's algorithm for minimum spanning trees is the standard example.

```python id="quhcds"
def sort_edges_by_weight(edges):
    return sorted(edges, key=lambda edge: edge[2])
```

Given:

```text id="7wryq8"
(0, 1, 10)
(0, 2, 3)
(1, 2, 5)
```

sorting by weight produces:

```text id="v1mpx9"
(0, 2, 3)
(1, 2, 5)
(0, 1, 10)
```

This representation keeps the weight close to the endpoints, which makes scanning and sorting straightforward.

## Neighbor Queries

An edge list is inefficient for neighbor queries. To find all outgoing neighbors of `u`, you must scan the whole list.

```python id="jyikmb"
def neighbors_from_edges(u, edges):
    result = []

    for a, b in edges:
        if a == u:
            result.append(b)

    return result
```

This costs $O(m)$ per query, where $m$ is the number of edges. If an algorithm asks for neighbors repeatedly, this becomes expensive.

For example, depth first search and breadth first search usually need all neighbors of many vertices. Running them directly on an edge list can turn a linear traversal into a much slower computation. Convert to an adjacency list first.

## Complexity

For a graph with $n$ vertices and $m$ edges:

| Operation | Edge list cost |
|---|---:|
| Store all edges | $O(m)$ |
| Add an edge | $O(1)$ amortized |
| Iterate all edges | $O(m)$ |
| Test whether `u -> v` exists | $O(m)$ |
| Find all neighbors of `u` | $O(m)$ |
| Sort edges | $O(m \log m)$ |
| Convert to adjacency list | $O(n + m)$ |
| Convert to adjacency matrix | $O(n^2 + m)$ |

The strength of an edge list is simple edge iteration. The weakness is vertex-local access.

## Correctness Considerations

The meaning of an edge list depends on the graph model.

For a directed graph, each pair is ordered:

```text id="zsexok"
(u, v) means u -> v
```

For an undirected graph, each pair is unordered:

```text id="4wr1ri"
(u, v) means u - v
```

Do not store both `(u, v)` and `(v, u)` in an undirected edge list unless the algorithm expects duplicated directed entries. Kruskal's algorithm, for example, normally expects each undirected edge once. Storing both directions would make the algorithm scan duplicate candidates.

Repeated edges require a policy. In a multigraph, repeated edges are meaningful. In a simple graph, they are usually input noise.

```text id="8x0bj1"
(0, 1)
(0, 1)
(0, 1)
```

This may mean three parallel edges, or it may mean one edge repeated three times by mistake. The algorithm cannot infer the intent.

Self loops also need explicit handling.

```text id="j11j9w"
(2, 2)
```

Some algorithms ignore self loops. Some reject them. Some allow them. The representation can store them, but the graph contract must decide whether they are legal.

## Common Pitfalls

A common mistake is using an edge list for traversal-heavy algorithms without conversion. BFS and DFS want neighbor iteration, so an adjacency list is usually the correct representation.

Another mistake is duplicating undirected edges and then using an algorithm that expects each edge once. This can affect counts, sorting behavior, and performance.

A third mistake is losing edge attributes during conversion. If edges have weights, capacities, labels, or IDs, the adjacency representation must preserve them.

```python id="xff5p5"
adj[u].append((v, w))
```

A fourth mistake is assuming that an edge list contains all vertices. Isolated vertices may have no incident edges and therefore never appear in the list. Keep `n` separately.

```text id="mw9062"
n = 4
edges = [(0, 1), (1, 2)]
```

Vertex `3` exists even though it appears in no edge.

## Takeaway

An edge list is the most direct way to store a graph as data. It is best for scanning, sorting, filtering, and input parsing. It is poor for repeated neighbor lookup, so traversal algorithms usually convert it into an adjacency list. Keep the graph contract explicit: direction, weights, duplicate edges, self loops, and isolated vertices all affect correctness.
