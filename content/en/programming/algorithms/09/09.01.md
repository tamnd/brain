# 9.1 Graph Models

## Problem

You need a precise way to describe a graph before choosing an algorithm or implementation. The same real problem can produce several different graph models, and each model changes what the algorithm is allowed to assume.

## Solution

Start by identifying three things:

1. What objects become vertices.
2. What relationships become edges.
3. What extra data belongs on vertices or edges.

A graph is usually written as:

$$
G = (V, E)
$$

where $V$ is the set of vertices and $E$ is the set of edges.

For an undirected graph, each edge connects two vertices without orientation:

$$
\{u, v\} \in E
$$

For a directed graph, each edge has a source and a target:

$$
(u, v) \in E
$$

This distinction matters. In an undirected graph, an edge from $u$ to $v$ also permits movement from $v$ to $u$. In a directed graph, $(u, v)$ says only that $v$ is reachable from $u$ in one step.

## Basic Model

A simple graph has no self loops and no repeated edges. That means an edge never connects a vertex to itself, and the same pair of vertices appears at most once.

```text
Vertices:
  A, B, C

Edges:
  A - B
  B - C
```

This graph is undirected. It says that $A$ is adjacent to $B$, and $B$ is adjacent to $C$.

A directed version uses arrows:

```text
Vertices:
  A, B, C

Edges:
  A -> B
  B -> C
```

Now $A$ can reach $C$ by following arrows, but $C$ cannot reach $A$ unless more edges are added.

## Recipe

When modeling a problem as a graph, use this checklist.

| Question | Typical choice |
|---|---|
| Are relationships symmetric? | Use an undirected graph |
| Do relationships have direction? | Use a directed graph |
| Can two vertices have more than one edge between them? | Use a multigraph |
| Can an edge start and end at the same vertex? | Allow self loops |
| Does each edge have a cost, distance, or capacity? | Use a weighted graph |
| Does the graph change over time? | Use a dynamic graph |
| Are vertices arranged in layers or states? | Use a state graph |

## Example: Roads

A road map can be modeled in more than one way.

If every road is two way, use an undirected graph:

$$
\{u, v\} \in E
$$

If some roads are one way, use a directed graph:

$$
(u, v) \in E
$$

If roads have lengths, use a weighted graph:

$$
w(u, v) = \text{length of the road from } u \text{ to } v
$$

The choice affects the algorithm. Breadth first search works for fewest number of roads. Dijkstra's algorithm works for shortest total distance when all edge weights are nonnegative.

## Example: Dependencies

A dependency graph is directed. If task $A$ must finish before task $B$, use the edge:

$$
A \to B
$$

This means $A$ points to the task that depends on it. A valid execution order is a topological ordering of this graph. If the graph contains a directed cycle, then the dependency system is inconsistent.

```text
parse input -> validate data -> write output
```

This graph has a clear direction. Reversing the arrows would describe a different convention, and the implementation must be consistent with whichever convention is chosen.

## Example: State Graphs

Many search problems are easier to express as state graphs. A vertex is not a physical object. It is a possible state of computation.

For example, in a grid problem, a state may include a position:

$$
(r, c)
$$

If the problem includes keys, doors, or remaining resources, the state must include those too:

$$
(r, c, keys, fuel)
$$

Two states are connected by an edge when one legal move transforms the first state into the second.

This model is common in puzzle solving, path finding, and dynamic programming over configurations. The graph may be too large to build explicitly, so neighbors are generated on demand.

## Implementation

A graph model usually becomes one of three concrete representations.

An edge list stores edges directly:

```text
[(0, 1), (1, 2), (2, 3)]
```

An adjacency list stores neighbors for each vertex:

```text
0: [1]
1: [0, 2]
2: [1, 3]
3: [2]
```

An adjacency matrix stores a table of edge existence:

```text
    0 1 2 3
0:  0 1 0 0
1:  1 0 1 0
2:  0 1 0 1
3:  0 0 1 0
```

Use an adjacency list for most sparse graphs. Use an adjacency matrix when constant time edge lookup matters and the number of vertices is small enough. Use an edge list when sorting or scanning edges is the main operation, as in Kruskal's algorithm.

## Correctness Considerations

The model must preserve the property the algorithm needs.

If an algorithm assumes an undirected graph, every edge must be available in both directions in the representation.

```text
add u to adj[v]
add v to adj[u]
```

If an algorithm assumes a directed graph, adding the reverse edge changes the problem.

If an algorithm assumes nonnegative weights, then negative edges invalidate Dijkstra's algorithm.

If an algorithm assumes a simple graph, repeated edges can change degree counts, path counts, and flow capacities.

These are modeling errors, not implementation details. A correct implementation of the wrong model still gives the wrong answer.

## Common Pitfalls

A common error is to confuse vertices with labels. Labels may be strings, coordinates, or objects. The algorithm usually wants compact integer IDs. Keep a mapping layer between external labels and internal IDs.

Another common error is to ignore direction. A dependency edge, a hyperlink, and a one way road all have orientation. Treating them as undirected can create false paths.

A third error is to omit part of the state. In a grid with keys, the cell alone is not enough. Reaching the same cell with different keys can lead to different future moves, so those must be different vertices in the state graph.

A final error is to choose a representation before understanding the operation. If the main operation is neighbor iteration, an adjacency list is natural. If the main operation is edge lookup, an adjacency matrix may be better. If the main operation is edge sorting, an edge list is usually simplest.

## Takeaway

A graph model is the contract between the problem and the algorithm. Vertices encode objects or states. Edges encode relationships or transitions. Direction, weights, repeated edges, and self loops are part of the model, and changing any of them can change the answer.
