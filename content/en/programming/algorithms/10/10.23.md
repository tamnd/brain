# 10.23 Complexity Analysis

10.23 Complexity Analysis
Problem

You need to determine how the running time and memory usage of shortest path algorithms scale with input size. The goal is not only to compute asymptotic bounds, but to understand what actually dominates cost in practice.

Input Parameters

Let:

n = |V| number of vertices

m = |E| number of edges

All complexity expressions are stated in terms of n and m.

Core Cost Components

Shortest path algorithms typically consist of:

vertex processing

edge relaxation

data structure operations

The total time is a combination of these costs.

BFS Complexity

For unweighted graphs:

Time: O(n + m)
Space: O(n)

Each vertex is processed once, and each edge is examined a constant number of times.

Dijkstra Complexity

Depends on priority queue choice.

Structure	Time
Binary heap	O((n + m) log n)
Fibonacci heap	O(n log n + m)
Array (dense graph)	O(n^2)

The dominant cost is priority queue operations.

Bellman-Ford Complexity
Time: O(n * m)
Space: O(n)

Each pass scans all edges. In worst case, n - 1 passes are required.

Floyd-Warshall Complexity
Time: O(n^3)
Space: O(n^2)

Three nested loops over vertices.

DAG Shortest Paths
Time: O(n + m)
Space: O(n)

Topological sort plus one relaxation per edge.

0-1 BFS Complexity
Time: O(n + m)
Space: O(n)

Each edge causes at most one effective relaxation.

A* Complexity

Worst case:

O((n + m) log n)

In practice, depends heavily on heuristic quality. A strong heuristic reduces explored nodes.

Johnson Algorithm Complexity
Time: O(nm log n)

Bellman-Ford: O(nm)

Dijkstra from each vertex: O(n m log n)

Efficient for sparse graphs.

Summary Table
Algorithm	Time	Space	Constraints
BFS	O(n + m)	O(n)	unit weights
Dijkstra	O(m log n)	O(n)	non-negative weights
Bellman-Ford	O(nm)	O(n)	handles negative weights
Floyd-Warshall	O(n^3)	O(n^2)	all-pairs
DAG	O(n + m)	O(n)	acyclic graph
0-1 BFS	O(n + m)	O(n)	weights in {0,1}
A*	heuristic-dependent	O(n)	target search
Lower Bounds

Any algorithm must at least inspect the input graph:

Ω(n + m)

For all-pairs shortest paths, output size alone is:

Ω(n^2)

so no algorithm can be faster than quadratic in output size.

Practical Considerations
Sparse vs Dense

sparse graphs: use adjacency lists and Dijkstra

dense graphs: array-based methods or Floyd-Warshall

Constant Factors

Priority queues introduce logarithmic overhead. For dense graphs, simple arrays may outperform heaps.

Memory Access

Matrix-based algorithms benefit from contiguous memory. List-based algorithms involve pointer chasing.

Early Termination

Single-target queries can stop early in Dijkstra or A*.

Amortized Behavior

Some algorithms perform better in practice than worst-case bounds suggest.

SPFA often faster than Bellman-Ford

A* often explores far fewer nodes

0-1 BFS avoids heap overhead

Trade-offs
Goal	Preferred Method
fastest single-source	Dijkstra
handle negative weights	Bellman-Ford
all-pairs dense	Floyd-Warshall
all-pairs sparse	Johnson
minimal memory	BFS / Dijkstra
Common Pitfalls

Do not ignore m. Many analyses mistakenly assume m = n.

Do not use worst-case bounds blindly. Practical performance may differ.

Do not assume asymptotic dominance always determines runtime. Constant factors matter.

Do not forget preprocessing costs when using advanced methods.

Recipe Pattern

Identify n and m. Choose an algorithm whose dominant term matches graph structure. Analyze cost in terms of edge relaxations and data structure operations. Adjust for practical constraints such as memory and constant factors.
