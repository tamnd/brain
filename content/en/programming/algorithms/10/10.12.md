# 10.12 Bidirectional Search

## Problem

You need the shortest path between one source vertex `s` and one target vertex `t`. A normal BFS or Dijkstra search starts at `s` and expands outward until it reaches `t`. On large graphs, this can visit many vertices that are irrelevant to the final path.

Bidirectional search reduces the search space by searching from both ends at once.

## Solution

For an unweighted graph, run BFS forward from `s` and backward from `t`. Stop when the two frontiers meet.

```text id="gyj36n"
bidirectional_bfs(G, s, t):
    if s == t:
        return [s]

    front_s = queue containing s
    front_t = queue containing t

    dist_s[s] = 0
    dist_t[t] = 0

    parent_s[s] = none
    parent_t[t] = none

    while front_s is not empty and front_t is not empty:
        expand the smaller frontier

        if a vertex appears in both searches:
            reconstruct the path

    return no path
```

The forward search records parents from `s`. The backward search records parents from `t`. When the searches meet at a vertex `m`, the final path is:

```text id="nsko20"
s -> ... -> m -> ... -> t
```

## Discussion

Ordinary BFS expands a ball around the source. If the shortest path has length `d` and the average branching factor is `b`, then the search may inspect roughly:

```text id="xdtgag"
b^d
```

vertices.

Bidirectional BFS expands two smaller balls of radius about `d / 2`. The rough search size becomes:

```text id="qf2ehk"
2 * b^(d / 2)
```

This can be much smaller when `b` and `d` are large.

## Python Implementation

```python id="568dbp"
from collections import deque

def bidirectional_bfs(graph, source, target):
    if source == target:
        return [source]

    q_s = deque([source])
    q_t = deque([target])

    parent_s = {source: None}
    parent_t = {target: None}

    visited_s = {source}
    visited_t = {target}

    while q_s and q_t:
        meet = expand_frontier(graph, q_s, visited_s, visited_t, parent_s)
        if meet is not None:
            return build_bidirectional_path(parent_s, parent_t, meet)

        meet = expand_frontier(graph, q_t, visited_t, visited_s, parent_t)
        if meet is not None:
            return build_bidirectional_path(parent_s, parent_t, meet)

    return None

def expand_frontier(graph, queue, own_seen, other_seen, parent):
    for _ in range(len(queue)):
        u = queue.popleft()

        for v in graph.get(u, []):
            if v in own_seen:
                continue

            own_seen.add(v)
            parent[v] = u

            if v in other_seen:
                return v

            queue.append(v)

    return None

def build_bidirectional_path(parent_s, parent_t, meet):
    left = []
    cur = meet

    while cur is not None:
        left.append(cur)
        cur = parent_s[cur]

    left.reverse()

    right = []
    cur = parent_t[meet]

    while cur is not None:
        right.append(cur)
        cur = parent_t[cur]

    return left + right
```

This implementation assumes an undirected graph. For a directed graph, the backward search must use the reverse graph.

## Correctness

Bidirectional BFS is correct because both searches expand in increasing distance from their respective endpoints.

Let `dist_s[x]` be the distance from `s` to `x`, and let `dist_t[x]` be the distance from `x` to `t`. When a vertex `m` is found by both searches, it defines a path of length:

```text id="0x4t8v"
dist_s[m] + dist_t[m]
```

Since BFS expands by layers, the first valid meeting after completing the relevant frontier layers gives a shortest path. A careful implementation expands full layers rather than single vertices, so it does not stop prematurely on a non-minimal crossing.

## Complexity

If the branching factor is `b` and the shortest path length is `d`, ordinary BFS may visit `O(b^d)` vertices. Bidirectional BFS often visits about `O(b^(d/2))` vertices from each side.

| Method | Approximate visited vertices |
|---|---:|
| BFS | `O(b^d)` |
| Bidirectional BFS | `O(b^(d/2))` |

Worst-case complexity remains `O(|V| + |E|)`, but the practical improvement can be large.

## Common Pitfalls

Do not use the same adjacency direction for backward search in a directed graph. You need reverse edges.

Do not stop in the middle of a layer unless the implementation preserves the shortest meeting condition.

Do not use bidirectional search for all-pairs shortest paths. It solves one source-target query.

Do not assume it helps when the graph has small diameter or very uneven branching.

## Recipe Pattern

Use bidirectional search for one source-target shortest path in a large unweighted graph. Search forward from the source and backward from the target, expand by layers, and reconstruct the path at the meeting point.
