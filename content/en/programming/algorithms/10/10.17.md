# 10.17 Dynamic Shortest Paths

## Problem

The graph changes over time. Edges may be added, removed, or have their weights updated. You need to maintain shortest path information without recomputing from scratch after every update.

Dynamic shortest paths split into two main categories:

- incremental: only insertions or weight decreases  
- decremental: only deletions or weight increases  
- fully dynamic: both types of updates  

## Core Challenge

Shortest paths depend globally on the graph. A local change can invalidate many distances. The goal is to limit recomputation to only affected regions.

## Incremental Case

When edges are added or weights decrease, shortest paths can only improve.

### Strategy

Start from the affected vertex and propagate improvements.

```text id="x7a1c3"
if dist[u] + w < dist[v]:
    dist[v] = dist[u] + w
    push v into queue
```

Then run a relaxation process similar to Dijkstra or BFS depending on weights.

### Example Pattern

```python id="u3l7bk"
import heapq

def decrease_update(graph, dist, parent, start):
    pq = [(dist[start], start)]

    while pq:
        d, u = heapq.heappop(pq)

        if d > dist[u]:
            continue

        for v, w in graph.get(u, []):
            nd = d + w
            if nd < dist[v]:
                dist[v] = nd
                parent[v] = u
                heapq.heappush(pq, (nd, v))
```

This recomputes only the affected region.

## Decremental Case

When edges are removed or weights increase, shortest paths may become invalid.

### Strategy

Detect affected vertices and recompute their distances.

One approach:

1. Identify vertices whose shortest path used the removed edge  
2. Reset their distances  
3. Recompute using a local shortest path algorithm  

This is harder than the incremental case because paths can worsen in many ways.

## Fully Dynamic Case

When both insertions and deletions occur, no simple local method always works. General solutions rely on advanced data structures.

Approaches include:

- recomputation from scratch when changes accumulate  
- maintaining auxiliary structures such as dynamic trees  
- approximation algorithms for faster updates  

## Reoptimization via Reuse

A practical method is to reuse previous results as a starting point.

Instead of reinitializing:

```text id="2z8g9w"
dist[v] = infinity
```

keep old distances and repair them.

This works well when updates are small relative to graph size.

## Special Cases

### DAG

For DAGs, recompute using topological order. Updates propagate forward only.

### Trees

Shortest paths in trees can be updated efficiently using tree-specific structures.

### Small Weight Changes

If weights change slightly, incremental relaxation often converges quickly.

## Complexity

Dynamic algorithms are often measured per update.

| Type | Typical cost |
|---|---|
| Incremental | proportional to affected region |
| Decremental | may approach full recomputation |
| Fully dynamic | complex, often high overhead |

Worst-case complexity can still be as large as recomputing from scratch.

## Practical Strategy

Use dynamic updates when:

- graph changes are small  
- queries are frequent  
- recomputation cost is high  

Otherwise, recompute from scratch with a fast static algorithm.

## Common Pitfalls

Do not assume updates are local. A small change can affect many vertices.

Do not reuse distances blindly after deletions. Paths may no longer exist.

Do not mix incremental logic with negative weights without care.

Do not overengineer. Fully dynamic algorithms are complex and often unnecessary.

## Recipe Pattern

Maintain current distances and parent pointers. When an update occurs, identify affected vertices and run localized relaxation. Choose between incremental repair and full recomputation based on update size and frequency.
