# 10.11 A* Search

## Problem

You need the shortest path from a source `s` to a target `t` in a weighted graph with non-negative edge weights. The graph may be large, and exploring all vertices, as Dijkstra does, is too expensive.

You have additional information: a heuristic function `h(v)` that estimates the remaining distance from any vertex `v` to the target `t`.

## Solution

Use A* search. It modifies Dijkstra by prioritizing vertices using:

```text id="c1n4yz"
f(v) = g(v) + h(v)
```

where:

- `g(v)` is the current best-known distance from `s` to `v`  
- `h(v)` is the heuristic estimate from `v` to `t`  

```text id="m7a2dp"
a_star(G, s, t, h):
    for each vertex v:
        g[v] = infinity
        parent[v] = none

    g[s] = 0
    pq = empty min-priority queue
    push (f = h[s], s)

    while pq is not empty:
        (f, u) = extract-min

        if u == t:
            return g, parent

        for each edge (u, v, w):
            tentative = g[u] + w

            if tentative < g[v]:
                g[v] = tentative
                parent[v] = u
                push (g[v] + h[v], v)

    return no path
```

The algorithm stops as soon as the target is extracted from the queue.

## Heuristic Requirements

The heuristic must satisfy:

### Admissibility

```text id="o9h8c1"
h(v) <= true distance from v to t
```

It never overestimates. This guarantees that A* finds an optimal path.

### Consistency (Monotonicity)

```text id="3m2d8r"
h(u) <= w(u, v) + h(v)
```

for every edge `(u, v)`.

Consistency implies admissibility and ensures that once a vertex is processed, its distance is final. With a consistent heuristic, A* behaves like Dijkstra with modified edge weights.

## Discussion

A* is best understood as Dijkstra with a bias toward the target. The priority is no longer just the cost so far, but the cost so far plus an estimate of the remaining cost.

If the heuristic is:

- `h(v) = 0`, A* reduces to Dijkstra  
- exact distance to target, A* explores only the optimal path  

In practice, a good heuristic dramatically reduces the search space.

## Correctness

Assume the heuristic is admissible.

Let `P` be an optimal path from `s` to `t`. A* always expands nodes in order of increasing `f(v) = g(v) + h(v)`.

For any node `v` on the optimal path:

```text id="2h7vqb"
f(v) = g(v) + h(v) <= g(v) + true_distance(v, t)
```

Since the remaining part of the optimal path has length `true_distance(v, t)`, we get:

```text id="9sk6bz"
f(v) <= total optimal path cost
```

Thus nodes on the optimal path always have priority no greater than the optimal solution. Therefore, A* cannot skip the optimal path.

When the target `t` is extracted from the queue, its `g[t]` value equals the shortest path distance.

## Python Implementation

```python id="2k8d3q"
import heapq

def a_star(graph, source, target, heuristic):
    g = {v: float("inf") for v in graph}
    parent = {v: None for v in graph}

    g[source] = 0
    pq = [(heuristic(source), source)]

    while pq:
        f, u = heapq.heappop(pq)

        if u == target:
            return g, parent

        for v, w in graph.get(u, []):
            tentative = g[u] + w

            if tentative < g[v]:
                g[v] = tentative
                parent[v] = u
                heapq.heappush(pq, (g[v] + heuristic(v), v))

    return None, None
```

## Path Reconstruction

```python id="9mj3q2"
def reconstruct(parent, s, t):
    path = []
    cur = t

    while cur is not None:
        path.append(cur)
        if cur == s:
            path.reverse()
            return path
        cur = parent[cur]

    return None
```

## Example Heuristics

For grid graphs:

- Manhattan distance for 4-direction movement  
- Euclidean distance for continuous space  

For road networks:

- Straight-line geographic distance  

For puzzles:

- Problem-specific admissible estimates  

## Complexity

Worst-case complexity matches Dijkstra:

```text id="2mf4ye"
O((|V| + |E|) log |V|)
```

In practice, a good heuristic reduces the number of explored vertices dramatically.

## Common Pitfalls

Do not use a heuristic that overestimates. This breaks optimality.

Do not assume A* is always faster. With a weak heuristic, it behaves like Dijkstra.

Do not stop early unless the heuristic is admissible. Otherwise the first path to the target may not be optimal.

Do not forget that inconsistent heuristics may require reprocessing vertices.

## When to Use

Use A* when:

- You need a path from one source to one target  
- Edge weights are non-negative  
- A meaningful heuristic is available  

Avoid A* when:

- No good heuristic exists  
- All-pairs distances are needed  
- Graph is very small  

## Recipe Pattern

Use Dijkstra’s structure, replace priority with `g + h`, and stop when the target is reached. The quality of the heuristic determines efficiency.
