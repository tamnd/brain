# 10.16 Sparse vs Dense Graph Strategies

## Problem

Shortest path performance depends heavily on graph structure. Two graphs with the same number of vertices can differ by orders of magnitude in edge count. Choosing the wrong algorithm for the density leads to avoidable cost.

You need a systematic way to select algorithms based on graph density, query type, and constraints.

## Density Model

Let:

- `n = |V|` number of vertices  
- `m = |E|` number of edges  

Define density:

```text
sparse: m = O(n)
dense:  m = Θ(n^2)
```

Real graphs often fall in between, but these extremes guide design.

## Core Tradeoffs

| Factor | Sparse Graph | Dense Graph |
|---|---|---|
| Edge count | small | large |
| Adjacency representation | lists | matrix often viable |
| Traversal cost | cheap | expensive |
| Priority queue overhead | dominant | less dominant |
| Dynamic programming over matrix | wasteful | efficient |

## Single-Source Shortest Paths

### Unweighted

| Graph type | Method | Time |
|---|---|---:|
| Sparse | BFS | `O(n + m)` |
| Dense | BFS (matrix) | `O(n^2)` |

### Non-negative Weights

| Graph type | Method | Time |
|---|---|---:|
| Sparse | Dijkstra with heap | `O(m log n)` |
| Dense | Dijkstra with array | `O(n^2)` |

For dense graphs, using a simple array instead of a heap avoids logarithmic overhead.

### Negative Weights

| Graph type | Method | Time |
|---|---|---:|
| Any | Bellman-Ford | `O(nm)` |
| DAG | Topological | `O(n + m)` |

Bellman-Ford scales poorly with density because `nm` becomes `n^3`.

## All-Pairs Shortest Paths

| Graph type | Method | Time |
|---|---|---:|
| Sparse | Johnson | `O(n m log n)` |
| Dense | Floyd-Warshall | `O(n^3)` |

Johnson reduces repeated work by using Dijkstra, which is efficient on sparse graphs. Floyd-Warshall uses matrix operations that match dense structure.

## Memory Considerations

| Representation | Space | Use case |
|---|---:|---|
| Adjacency list | `O(n + m)` | sparse graphs |
| Adjacency matrix | `O(n^2)` | dense graphs |
| Edge list | `O(m)` | Bellman-Ford, streaming |

Dense graphs justify matrix storage because `m` is already close to `n^2`.

## Practical Selection Rules

Use BFS when weights are uniform.

Use Dijkstra with a heap when:

- weights are non-negative  
- graph is sparse  

Use Dijkstra with an array when:

- graph is dense  
- `n` is moderate  

Use Bellman-Ford when:

- negative weights exist  
- graph is not a DAG  

Use DAG shortest paths when:

- graph is acyclic  

Use Floyd-Warshall when:

- all-pairs distances are required  
- graph is dense or small  

Use Johnson when:

- all-pairs distances are required  
- graph is sparse  
- negative edges exist  

## Cost Comparison Example

Assume:

```text
n = 10^5
m = 3 * 10^5   (sparse)
```

| Algorithm | Estimated cost |
|---|---:|
| Dijkstra | `~ m log n` ≈ `3e5 * 17` |
| Bellman-Ford | `~ n m` ≈ `3e10` |

The difference is several orders of magnitude.

Now assume:

```text
n = 2000
m ≈ n^2 ≈ 4e6   (dense)
```

| Algorithm | Estimated cost |
|---|---:|
| Dijkstra repeated | `~ n * m log n` ≈ large |
| Floyd-Warshall | `~ n^3` ≈ `8e9` operations |

In dense settings, Floyd-Warshall becomes competitive.

## Cache and Constant Factors

Dense algorithms often benefit from:

- contiguous memory access  
- predictable iteration  
- fewer pointer dereferences  

Sparse algorithms benefit from:

- avoiding unnecessary edge scans  
- focusing only on reachable structure  

These constant factors matter in real systems.

## Hybrid Strategies

Some graphs exhibit mixed structure. Common techniques:

- use adjacency lists but detect dense subgraphs  
- switch algorithm based on subproblem size  
- compress graph using contraction or clustering  
- precompute landmarks for heuristic search  

## Common Pitfalls

Do not use Floyd-Warshall on large sparse graphs. Memory and time both explode.

Do not use Bellman-Ford on dense graphs unless necessary.

Do not assume heap-based Dijkstra is always best. For dense graphs, array-based selection can be faster.

Do not ignore memory layout. Poor locality can dominate runtime.

## Recipe Pattern

Classify the graph by size and density. Match the algorithm to structural properties and query type. Prefer linear or near-linear methods for sparse graphs and matrix-based methods for dense graphs.
