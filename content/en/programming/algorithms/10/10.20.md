# 10.20 Path Queries and Distance Oracles

## Problem

You need to answer many shortest path or distance queries on a graph. Running a full shortest path algorithm per query is too slow.

Goal: preprocess the graph so that queries are fast, while keeping preprocessing time and memory reasonable.

## Model

Given a graph `G = (V, E)`:

- preprocess once  
- answer queries of the form `dist(u, v)` or `path(u, v)`  

Trade-offs:

- preprocessing time  
- query time  
- memory usage  
- exact vs approximate answers  

## Exact Distance Oracles

### All-Pairs Table

Store all distances:

```text id="c7w2xp"
dist[u][v] for all u, v
```

| Property | Value |
|---|---|
| Preprocessing | `O(n^3)` or repeated shortest paths |
| Query | `O(1)` |
| Space | `O(n^2)` |

Works only for small graphs.

### Tree-Based Oracles

For trees, preprocess with LCA and depth:

```text id="k3m8q1"
dist(u, v) = depth[u] + depth[v] - 2 * depth[lca(u, v)]
```

Query time is constant or logarithmic.

### Sparse Graph Labeling

Assign each vertex a label containing distances to selected hubs.

Query:

```text id="b5n9jd"
dist(u, v) = min over hubs h of dist(u, h) + dist(h, v)
```

This is called a hub labeling scheme.

## Approximate Distance Oracles

Exact answers may be too expensive. Approximation reduces cost.

### Stretch Factor

An oracle has stretch `α` if:

```text id="f2p7kr"
true_distance <= returned_distance <= α * true_distance
```

### Thorup-Zwick Oracle

- preprocessing: `O(n^(1+1/k))`  
- query: `O(k)`  
- stretch: `2k - 1`  

Used for theoretical guarantees.

### Landmark-Based Methods

Select landmarks `L`. Precompute distances:

```text id="n7r1vx"
dist(v, l) for all v and l in L
```

Estimate:

```text id="u6z2wm"
dist(u, v) ≈ max over l in L of |dist(u, l) - dist(v, l)|
```

Works well in practice for large graphs.

## Path Retrieval

Distance oracles often return only distances. To recover paths:

- store parent pointers for selected sources  
- reconstruct via hubs  
- store shortcut edges  

Trade-off: more memory vs faster path extraction.

## Indexing Structures

### Hub Labeling

Each vertex stores a small label:

```text id="x4d8pq"
L(v) = set of hubs with distances
```

Query:

```text id="t9m3wr"
intersect L(u) and L(v)
```

Pick hub minimizing sum of distances.

### Contraction Hierarchies

Preprocess graph into hierarchical structure. Queries use upward and downward edges.

### Transit Node Routing

Identify important nodes (highways). Precompute distances between them.

Query:

1. go from `u` to nearby transit nodes  
2. jump between transit nodes  
3. go from transit nodes to `v`  

## Complexity

| Method | Preprocessing | Query | Space |
|---|---|---|---|
| All-pairs | `O(n^3)` | `O(1)` | `O(n^2)` |
| Hub labeling | high | `O(1)` | high |
| Landmark | `O(n|L|)` | `O(|L|)` | `O(n|L|)` |
| Thorup-Zwick | `O(n^(1+1/k))` | `O(k)` | `O(n^(1+1/k))` |

## Practical Guidance

Use full tables only for small graphs.

Use hub labeling or contraction hierarchies for large static graphs with many queries.

Use landmarks for large dynamic graphs with moderate accuracy needs.

Use tree formulas when the graph is a tree or tree-like.

## Common Pitfalls

Do not over-preprocess if queries are few.

Do not ignore memory constraints. Many oracles use large storage.

Do not expect approximate methods to give exact answers.

Do not forget update costs. Most oracles assume static graphs.

## Recipe Pattern

Precompute auxiliary structures that encode distances or shortcuts. Answer queries by combining precomputed pieces instead of recomputing paths. Balance preprocessing cost against query frequency and accuracy requirements.
