# 10.19 Dynamic Shortest Paths on Road Networks

## Problem

You maintain shortest paths on a large road network where edge weights represent travel time. Weights change due to traffic, incidents, or temporary closures. Queries are frequent and latency matters.

A full recomputation after every update is often too slow. The structure of road graphs can be exploited to reduce work.

## Graph Structure

Road networks have properties that differ from arbitrary graphs:

- sparse: `m ≈ O(n)`  
- near-planar: low average degree  
- metric-like weights: obey triangle inequality in practice  
- hierarchical: highways connect regions, local roads fill details  

These properties support specialized acceleration.

## Baseline Strategy

Maintain a shortest path tree from a source. When weights change, repair only the affected region.

### Decrease Updates (faster)

When an edge weight decreases, distances can only improve.

```text id="a8s1pc"
if dist[u] + w_new < dist[v]:
    update dist[v]
    propagate forward
```

Use a localized Dijkstra from affected nodes.

### Increase Updates (harder)

When an edge weight increases or is removed, shortest paths that used that edge may become invalid.

Strategy:

1. identify vertices whose parent edge changed  
2. invalidate their distances  
3. recompute from a boundary  

This often requires partial recomputation of a subtree.

## Practical Repair Pattern

```python id="x0yqk3"
import heapq

def repair(graph, dist, parent, start_nodes):
    pq = []

    for u in start_nodes:
        heapq.heappush(pq, (dist[u], u))

    while pq:
        d, u = heapq.heappop(pq)

        if d > dist[u]:
            continue

        for v, w in graph.get(u, []):
            nd = d + w
            if nd < dist[v]:
                dist[v] = nd
                parent[v] = u
                heapq.heappush(pq, (nd, v))
```

`start_nodes` are vertices affected by the update.

## Heuristic Acceleration

### A* with Landmarks

Precompute distances from a small set of landmark vertices. Use them to build a heuristic:

```text id="x2w3zp"
h(v) = max over landmarks L of |dist(L, t) - dist(L, v)|
```

This improves A* performance significantly on road graphs.

### Contraction Hierarchies

Preprocess the graph by contracting vertices and adding shortcut edges. Queries then run in a reduced search space.

Key idea:

- assign an importance order to vertices  
- remove low-importance vertices  
- add shortcuts to preserve shortest paths  

Query time becomes very fast.

### Multi-Level Graphs

Partition graph into regions. Precompute distances between boundary nodes. Queries jump across regions.

## Time-Dependent Weights

In realistic systems, edge weights depend on time.

```text id="g4m2lt"
w = w(edge, departure_time)
```

Shortest path must account for time evolution.

Approach:

- treat time as a dimension  
- use label-setting methods with time states  
- ensure FIFO property (no overtaking)  

## Incremental vs Rebuild

| Scenario | Strategy |
|---|---|
| few updates, many queries | incremental repair |
| many updates | periodic full recomputation |
| heavy traffic changes | time-dependent model |

## Complexity

| Method | Query time | Update time |
|---|---|---|
| Dijkstra | `O(m log n)` | none |
| A* with heuristic | much faster in practice | none |
| Incremental repair | localized | proportional to affected region |
| Contraction hierarchies | very fast | expensive preprocessing |

## Common Pitfalls

Do not assume updates are local. A highway closure can affect large regions.

Do not reuse parent pointers blindly after increases.

Do not ignore time-dependence if travel times vary significantly.

Do not overuse complex preprocessing if updates are frequent.

## Recipe Pattern

Exploit road graph structure. Use A* with strong heuristics for queries. Apply incremental repair for small updates. Use hierarchical preprocessing when the graph is mostly static.
