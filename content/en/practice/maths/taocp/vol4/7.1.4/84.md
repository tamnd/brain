## Solution

Let $\mathcal{M}$ denote the memo cache used by the recursive apply procedure underlying equation (55), and let $\mathcal{U}$ denote the unique table that enforces canonical representation of BDD nodes via hash-consing of triples $(V, LO, HI)$.

The effectiveness of $\mathcal{M}$ depends on the recurrence structure of calls of the form $\mathrm{Apply}(f,g)$ for subfunctions $f$ and $g$ encountered during BDD construction. Each entry in $\mathcal{M}$ corresponds to a pair $(f,g)$ that has already been evaluated. Increasing the size of $\mathcal{M}$ reduces recomputation of identical subproblems, but each entry consumes memory and increases lookup overhead. A useful balance is obtained when the number of stored entries is proportional to the number of distinct reachable pairs of nodes in the current BDD operation sequence. Since this number is bounded above by $B(f),B(g)$ for operands $f$ and $g$, a practical choice is to allocate $\mathcal{M}$ with capacity proportional to the current working set of reachable nodes in the unique table, typically a fixed fraction of $|\mathcal{U}|$. When memory is constrained, $\mathcal{M}$ should be smaller than $\mathcal{U}$, since missed cache entries incur recomputation but do not threaten correctness, whereas unique-table overflow destroys canonicity.

The memo cache should be implemented as a hash table with replacement, for example random replacement or least-recently used eviction, since exact LRU ordering is expensive relative to the cost of BDD recursion. The hash function should depend only on identifiers of $f$ and $g$, not on structural properties of the subgraphs, since structural equality is already enforced by $\mathcal{U}$. The load factor should remain bounded away from $1$, since collision chains directly affect the cost of each application of equation (55). A stable regime is obtained when resizing occurs whenever the number of stored entries exceeds a fixed constant multiple of the table size.

The unique table $\mathcal{U}$ has a stricter role, since it enforces the identity of nodes. Each insertion requires locating an existing triple $(V, LO, HI)$ or allocating a new node if none exists. The performance of this structure depends on maintaining a controlled load factor in the hash buckets. Let $N$ be the current number of live BDD nodes. The table size should be chosen proportional to $N$, with a constant factor ensuring that expected bucket occupancy remains bounded. Doubling strategies for resizing preserve amortized constant-time insertion, since each node is rehashed only when the table grows. Since BDD construction tends to increase $N$ monotonically until garbage collection, resizing should be triggered when the load factor exceeds a fixed threshold, for example when the average chain length begins to grow noticeably in measured operation cost.

Garbage collection interacts directly with both $\mathcal{U}$ and the memo cache. Since nodes carry reference counts in the sense of the preceding exercises, a node becomes unreachable precisely when its reference count drops to $0$. However, reference count updates alone do not reclaim chains of mutually dependent temporary nodes efficiently when many transient BDD operations are performed. A practical strategy is to combine reference counting with periodic global collection.

A suitable scheduling method is to trigger garbage collection when either of two conditions holds. The first condition is that the number of allocated nodes exceeds a fixed multiple of the number of nodes currently reachable from designated roots. The second condition is that allocation failures occur in the free list of reusable nodes. When either condition is met, a full traversal from all root nodes is performed, marking reachable nodes and rebuilding the free list from unmarked nodes. After this reconstruction, both the unique table and memo cache entries referencing freed nodes must be invalidated, since their keys may point to reclaimed addresses.

Between full collections, lightweight maintenance can be performed incrementally by exploiting reference count updates: whenever a node’s reference count becomes zero, it is placed on a deletion queue, and its outgoing edges are recursively decremented. This incremental phase reduces the cost of full collections and spreads deallocation work over time.

A stable overall policy is obtained by coupling the sizes of $\mathcal{M}$ and $\mathcal{U}$ to the number of live nodes and triggering garbage collection when either the free list falls below a fixed fraction of the total capacity or when the cost of memo misses exceeds a monitored threshold. This ensures that both lookup structures remain balanced and that memory usage tracks the actual size of the BDD rather than the transient history of its construction.
