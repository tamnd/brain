**Exercise 82.** &#9654; [*25*] [25] When the nodes of a BDD base have been endowed with REF fields, explain
how those fields should be adjusted within (55) and within Algorithm U.

**Solve time:** 4m

Let `REF(v)` denote the number of **stored references** to a node `v` in the BDD base. A stored reference is a pointer that persists in one of the following structures:

- a `LO` or `HI` field of a BDD node,
- a memoization table entry used in (55),
- the unique table entry that stores the canonical representative of a node.

Transient pointers produced during evaluation, including return values from procedures, do not affect `REF`.

The invariant is:

$$
REF(v)=\text{number of persistent data-structure fields currently containing a pointer to } v.
$$

## Adjustment of `REF` within (55)

Procedure (55) performs recursive construction with memoization and pointer updates. All `REF` adjustments arise only from **changes to persistent storage**.

### 1. Overwriting `LO` and `HI` fields

Whenever a node field is updated, reference counts are adjusted by a swap discipline.

Suppose `LO(p)` is changed from `q` to `r`.

- If `q \neq \text{nil}`:

$$
REF(q)\leftarrow REF(q)-1
$$

- If `r \neq \text{nil}`:

$$
REF(r)\leftarrow REF(r)+1
$$

The same rule applies to `HI(p)`.

Thus every persistent pointer replacement is implemented as a decrement of the old target and an increment of the new target.

### 2. Memoization table updates

Let a memo table entry change from `q` to `r`.

- If the entry previously contained `q \neq \text{nil}`:

$$
REF(q)\leftarrow REF(q)-1
$$

- If the new entry is `r \neq \text{nil}`:

$$
REF(r)\leftarrow REF(r)+1
$$

Memo tables are treated identically to node fields because they store persistent references.

### 3. Calls to Algorithm U

When (55) receives a pointer from Algorithm U, **no `REF` change occurs at receipt time**.

The pointer becomes a stored reference only if it is later written into a `LO`, `HI`, or memo table entry.

## Adjustment of `REF` within Algorithm U

Algorithm U performs unique-table lookup and node creation. It is the **only place where new BDD nodes are introduced**, hence it must establish correct reference counts for all persistent structures created at that moment.

### 1. Unique-table lookup (hit case)

If a node with signature `(i,u,v)` already exists, U returns the existing node `w`.

No `REF` fields are changed, since no new persistent reference is created.

### 2. Node creation (miss case)

If no such node exists, U creates a new node `w` with

$$
V(w)=i,\quad LO(w)=u,\quad HI(w)=v.
$$

At the moment of creation, several **stored references are simultaneously established**.

#### (a) References from the new node to its children

The assignments `LO(w)=u` and `HI(w)=v` immediately create persistent references (when non-nil). Therefore:

$$
REF(u)\leftarrow REF(u)+1,\quad REF(v)\leftarrow REF(v)+1.
$$

These increments are essential because `w` permanently stores pointers to `u` and `v`.

#### (b) Reference from the unique table

Insertion of `w` into the unique table makes the table hold a persistent pointer to `w`. Hence:

$$
REF(w)\leftarrow REF(w)+1.
$$

This accounts for the fact that the unique table is part of the permanent BDD base structure.

#### (c) Return value

The returned pointer to `w` is not itself a stored reference. No `REF` change is associated with returning.

### 3. Resulting initial state

After creation, a new node `w` satisfies:

$$
REF(w)=1 + (\text{number of incoming LO/HI or memo references})
$$

At creation time itself, this reduces to:

$$
REF(w)=1,
$$

coming solely from the unique-table entry, while its children already reflect the new incoming edges from `w`.

## Why these updates are necessary

The key invariant is that every persistent pointer must be counted exactly once.

When U creates `w`, three classes of persistent structure are updated:

1. The unique table stores `w`  → contributes to `REF(w)`.
2. `LO(w)` and `HI(w)` store pointers to `u` and `v` → contribute to `REF(u)` and `REF(v)`.
3. No transient computation affects reference counts.

If child increments were omitted, the BDD would lose accounting for the permanent edges introduced by `w`, violating the reference-counting invariant immediately upon node creation.

## Final summary of correct rules

1. **Algorithm U**

- On lookup hit: no `REF` changes.
- On node creation:

$$
REF(u)\leftarrow REF(u)+1,\quad REF(v)\leftarrow REF(v)+1
$$

(for non-nil children),

$$
REF(w)\leftarrow REF(w)+1
$$

for insertion into the unique table.
2. **Procedure (55)**

- On every overwrite of a `LO`, `HI`, or memo table entry:

decrement the old target (if any) and increment the new target (if any).
- No `REF` updates occur for transient values or returned pointers.

These rules ensure that `REF` counts exactly the number of persistent references in the BDD base at all times.
