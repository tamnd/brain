**Exercise 69.** [*21*] [21] Algorithm S ought to have one or more precautionary instructions such as
“if NTOP > TBOT, terminate the algorithm unsuccessfully,” in case it runs out of room.
Where are the best places to insert them?

**Verified:** yes  
**Solve time:** 2m09s  

---

We restart from the actual structure of Algorithm S’s storage discipline in TAOCP.

Algorithm S maintains a single node table with two moving boundaries:

- new nodes are allocated only from one end of the table (the “top” end in Knuth’s notation),
- free space is the interval between the advancing allocation pointer and the fixed opposite boundary.

Thus there is only one effective resource invariant: the number of unused table entries is proportional to the gap between the two boundaries. No node creation ever occurs except by consuming one previously unused table cell.

## Correct overflow condition

A failure occurs exactly when there is no free cell available for allocation. In TAOCP terms, this is when the two boundaries meet.

Hence the correct exhaustion condition is

$$
\mathrm{NTOP} \ge \mathrm{TBOT}.
$$

The equality case must be included because when $\mathrm{NTOP} = \mathrm{TBOT}$, the table has exactly zero free positions remaining, so any further allocation is impossible.

## Where overflow can occur

We now identify where the variables $\mathrm{NTOP}$ and $\mathrm{TBOT}$ change.

In Algorithm S, these variables are modified only when a new node is physically created. All other operations, including:

- traversal of BDD structure,
- Boolean tests on node labels,
- memoization lookups,

do not change either boundary pointer.

Therefore, overflow can occur only at instants when a node is about to be allocated.

More precisely, every allocation has the structure:

1. check or assume a free cell exists,
2. assign a new node into that cell,
3. update the allocation boundary.

Thus the only dangerous point is immediately before step 2.

## Correct placement of precautionary instructions

A precautionary instruction of the form

$$
\text{if } \mathrm{NTOP} \ge \mathrm{TBOT}, \text{ terminate unsuccessfully}
$$

must be placed at every point in Algorithm S where a new node is about to be created.

This includes every distinct allocation context in the algorithm, namely:

- creation of a new BDD node during APPLY-like combination steps,
- creation of any node introduced by reduction or caching mechanisms if such nodes are explicitly stored in the same table.

The correct principle is structural:

> Every instruction that writes a fresh triple $(v,\mathrm{LO},\mathrm{HI})$ into the node table must be immediately preceded by the overflow test.

No other locations require checks, because no other instruction can reduce available storage.

## Why no additional global checks are needed

There is no separate initialization hazard beyond normal allocation.

At entry to Algorithm S, the table boundaries are initialized consistently with the invariant that all free space lies between $\mathrm{NTOP}$ and $\mathrm{TBOT}$. No nodes are assumed to exist before the first allocation step unless explicitly created through the same allocation mechanism.

Therefore, if the first allocation occurs at the beginning of the algorithm, it is already protected by the same local check required at all later allocations. A separate global check would duplicate logic without adding safety.

## Sufficiency of local checks

To justify completeness, observe:

- $\mathrm{NTOP}$ and $\mathrm{TBOT}$ are modified only during node allocation,
- every allocation is preceded by the test $\mathrm{NTOP} \ge \mathrm{TBOT}$,
- therefore no modification of either pointer can occur after exhaustion,
- hence no out-of-bounds write into the node table is possible.

This shows that the local insertion points are both necessary and sufficient.

## Final conclusion

The correct precautionary instructions are placed immediately before every node allocation in Algorithm S, and they test the condition

$$
\mathrm{NTOP} \ge \mathrm{TBOT}.
$$

No additional global or traversal-time checks are required, since only allocation steps can exhaust the node table.
