**Exercise 164.** [*17*] [17] When Algorithm M finds a solution $x_0 x_1 \ldots x_{t-1}$ in step M2, some of the nodes $x_j$ might represent the fact that some primary item will appear in no further options. Explain how to handle this "null" case, by modifying answer 13.

The previous solution correctly described failure detection in DLX, but it did not address the actual representational issue in Algorithm M: the tuple $x_0 x_1 \ldots x_{t-1}$ is meant to encode, for each primary item position, either a chosen option or the fact that no explicit choice is needed because the item is already forced.

The exercise’s “null case” is precisely this second situation, not backtracking.

## 1. Where the previous reasoning went wrong

### Error 1: treating the situation as failure

The condition “a primary item has no remaining options” was interpreted as a dead end. This is incorrect in the context of Algorithm M as used in answer (13).

In Algorithm M, a primary item may have no available rows **because it is already satisfied by earlier choices**, not because the instance is infeasible. Thus, emptiness of the column is not inherently a failure condition in the representation step; it may simply indicate that no additional decision is required for that item.

### Error 2: missing modification of (13)

Answer (13) defines how the solution record is constructed from choices of options. The previous solution did not modify this mechanism; it only added a runtime check. That does not change the representation of $x_j$, which is what the exercise asks for.

### Error 3: conflating search state with solution encoding

The exercise concerns the structure of the output sequence $x_0, \dots, x_{t-1}$, not whether the search continues. The solution must therefore specify how the “null” situation is recorded inside this sequence.

## 2. Correct interpretation of the null case

At the moment Algorithm M constructs the solution tuple, each position $x_j$ corresponds to processing a primary item $j$-th in the chosen ordering.

Two situations may occur:

1. A concrete option $p$ is selected that covers the current primary item.
2. The primary item is already covered by earlier choices, so no new option is needed.

In case (2), the set of available options for that primary item is effectively empty, but this is not a contradiction; it indicates that the item is already satisfied.

This is the “null case.”

## 3. Modification of answer (13)

We modify (13) by extending the codomain of the solution entries.

Introduce a distinguished symbol

$$
\bot
$$

meaning “no option is chosen because the primary item is already satisfied.”

### Modified rule for constructing $x_j$

When Algorithm M processes primary item $i_j$ at stage $j$:

- If $i_j$ is already covered by previously chosen options, then set

$$
x_j := \bot,
$$

and perform no covering operation for this stage.
- Otherwise, choose an option $p$ containing $i_j$, and set

$$
x_j := p,
$$

followed by the standard update procedure of (13) (covering rows and columns associated with $p$).

## 4. Why this correctly resolves the null case

This modification ensures that:

- Every primary item position $j$ still has a well-defined entry $x_j$.
- The algorithm does not force artificial option selection when none is needed.
- The representation remains faithful to Algorithm M’s invariant: each primary constraint is either explicitly satisfied by a chosen option or already satisfied implicitly.

Crucially, no failure or backtracking is triggered by the null case, because emptiness of options is interpreted structurally, not as inconsistency.

## 5. Consistency with DLX structure

No change to the underlying dancing-links structure is required:

- Nodes still represent only real options.
- Columns becoming empty during search still signal infeasibility of _further covering steps_, but the null case here occurs at the _encoding stage_, after satisfaction has already been achieved.
- The symbol $\bot$ lives only in the output sequence $x_j$, not in the matrix representation.

## Conclusion

The correct modification of answer (13) is to extend the solution encoding so that:

$$
x_j \in \{\text{options } p\} \cup \{\bot\},
$$

where $x_j = \bot$ exactly when the corresponding primary item is already satisfied at the time it would be processed.

This handles the null case purely at the representation level, without introducing artificial nodes or conflating it with search failure.
