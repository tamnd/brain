**Exercise 254.** &#9654; [*28*] [28] The signature $\sigma$ computed by Algorithm Z in step Z2 is supposed to characterize the current subproblem completely. It contains one bit for each primary item, indicating whether or not that item still needs to be covered.

a) Explain why one bit isn't sufficient for secondary items with colors.

b) Suggest a good way to implement the computation of $\sigma$.

c) Algorithm C uses the operations hide$'$ and unhide$'$ in (39)–(37), in order to avoid unnecessary accesses to memory in nodes for secondary items. Explain why Algorithm Z does not want to use those optimizations. *Hint:* Algorithm Z needs to know whether the option list for a secondary item is empty.

d) When the list for item $i$ is purified, its options of the wrong color are removed from other lists. But they remain on list $i$, in order to be unpurified later. How then can Algorithm Z know when list $i$ is no longer relevant to the current subproblem?

### (a) Why one bit is insufficient for secondary items with colors

A secondary item with colors is not a binary constraint. Its behavior depends on _which colors still have at least one surviving compatible option_ under the current partial solution.

After purification, options incompatible with the current color context are removed from various item lists. For a secondary item $i$, different colors may lose feasibility at different times, and multiple colors can remain simultaneously viable.

Thus the state of $i$ is not simply “present or absent”, but rather a subset of admissible colors:

$$
C_i \subseteq \{\text{all colors of item } i\}.
$$

Two subproblems may both have item $i$ present, yet differ in which colors remain feasible. Therefore a single bit cannot represent the necessary information; at minimum, one needs a representation that distinguishes all surviving color classes (for example, one bit per color, or a dynamic list of active colors).

### (b) A good way to implement computation of $\sigma$

The signature $\sigma$ is defined only over **primary items**, and it encodes whether each primary item is currently uncovered.

A correct implementation is to maintain $\sigma$ _incrementally_ as the search progresses:

- Initially, all primary items are uncovered, so $\sigma_j = 1$ for every primary item $j$.
- When Algorithm Z covers a primary item $j$, set $\sigma_j \leftarrow 0$.
- When backtracking (uncovering $j$), restore $\sigma_j \leftarrow 1$.

Equivalently, one may compute $\sigma$ from the current header structure by traversing the primary item list, but this is not intended as a full recomputation at each step. The essential invariant is:

$$
\sigma_j = 1 \iff \text{primary item } j \text{ is currently in the header list (uncovered)}.
$$

Thus $\sigma$ is maintained in $O(1)$ per cover/uncover operation and always reflects the exact current primary-item subproblem.

### (c) Why Algorithm Z does not use hide$'$ and unhide$'$

Operations hide$'$ and unhide$'$ in Algorithm C optimize access by suppressing traversal into nodes belonging to secondary items.

Algorithm Z cannot use this optimization because it must still maintain _accurate detection of emptiness of secondary item option lists_.

Even when secondary items are not part of branching decisions, Algorithm Z must be able to determine whether a secondary column has become empty after purification, since this signals failure of the current partial solution.

If hide$'$ were used, nodes belonging to secondary items could be bypassed in traversal, but their presence or absence would no longer be reliably detectable during structural updates. In particular, Algorithm Z would lose the ability to maintain correct bookkeeping about whether any options remain associated with a secondary item.

Therefore, Algorithm Z keeps explicit access to secondary-item nodes so that list updates and emptiness conditions remain observable and maintainable.

### (d) How Algorithm Z knows when a purified secondary list is no longer relevant

The key correction is that Algorithm Z does **not** determine relevance by inspecting global row consistency, nor by scanning for “fully intact options.” Instead, relevance is tracked through **local structural bookkeeping on columns and nodes**.

Although purified options may remain physically present in the column list of a secondary item $i$, Algorithm Z maintains an additional invariant:

- Each option node has a status indicating whether it is still _active_ in the current subproblem.
- When an option is invalidated by purification (i.e., removed from any of its incident item lists), it is marked inactive and excluded from all effective column counts.

Each column $i$, including secondary ones, maintains a live count (or equivalent pointer invariant) of active nodes in its list:

$$
\text{active\_size}(i) = \#\{\text{nodes in column } i \text{ that are still active}\}.
$$

Then:

- Column $i$ is **relevant** to the current subproblem iff $\text{active\_size}(i) > 0$.
- Column $i$ is **no longer relevant** iff $\text{active\_size}(i) = 0$.

Crucially, this avoids any need to reconstruct global row integrity. The algorithm never tests whether a node still appears consistently across all its incident columns; instead, it maintains activity information locally during updates.

When purification removes an option from any other column, the corresponding node is marked inactive, and the active counters of all columns it participates in are updated accordingly. Thus the moment the last active option involving item $i$ disappears, this is immediately reflected in $\text{active\_size}(i)$, without scanning rows or performing global checks.

This is the mechanism by which Algorithm Z detects irrelevance: it is a direct consequence of maintained column-level accounting, not a derived property of row consistency.

### Summary

- **(a)** One bit is insufficient because secondary items must track a _set of surviving colors_, not just presence.
- **(b)** $\sigma$ is an incrementally maintained bit vector over primary items, updated during cover/uncover operations.
- **(c)** hide$'$ is incompatible with Algorithm Z because it would destroy reliable detection of emptiness in secondary item structures.
- **(d)** irrelevance of secondary items is determined by maintained column-level activity counts of options, not by scanning or reconstructing global row consistency.
