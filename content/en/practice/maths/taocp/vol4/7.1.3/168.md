**Exercise 168.** &#9654; [*23*] [23] To keep an image finite, we might insist that a 3 × 3 cellular automaton
treats a M ×N bitmap as a torus, wrapping around seamlessly between top and bottom
and between left and right. The task of simulating its actions efficiently with bitwise
operations is somewhat tricky: We want to minimize references to memory, yet each
new pixel value depends on old values that lie on all sides. Furthermore the shifting of
bits between neighboring words tends to be awkward, taxing the capacity of a register.
Show that such difficulties can be surmounted by maintaining an array of n-bit
words Ajk for 0 ≤j ≤M and 0 ≤k ≤N ′ = ⌈N/(n−2)⌉. If j ̸= M and k ̸= 0, word Ajk
should contain the pixels of row j and columns (k −1)(n −2) through k(n −2) + 1,
inclusive; the other words AMk and Aj0 provide auxiliary buffer space. (Notice that
some bits of the raster appear twice.)

The error in the previous solution is structural: it conflates real data rows with wraparound buffer rows. A correct construction must separate storage from auxiliary copies so that every real bitmap cell is represented exactly once, while buffers are derived and never part of the logical data.

## 1. Bitmap and indexing

Let the toroidal bitmap be $x_{ji}$ for

$$
1 \le j \le M,\quad 1 \le i \le N,
$$

with indices taken modulo $M$ and $N$.

Fix word size $n$ and set

$$
N' = \left\lceil \frac{N}{n-2} \right\rceil.
$$

We store data in $n$-bit words $A_{jk}$ where:

$$
0 \le j \le M+1,\quad 0 \le k \le N'.
$$

Rows $j=1,\dots,M$ are real data. Rows $j=0$ and $j=M+1$ are buffers.

Column $k=1,\dots,N'$ are real blocks. Column $k=0$ is a buffer column.

## 2. Block structure (horizontal layout)

For $1 \le j \le M$ and $1 \le k \le N'$, word $A_{jk}$ contains $n$ bits:

- bit $1$: left halo
- bits $2,\dots,n-1$: $n-2$ payload pixels
- bit $n$: right halo

Define a cyclic column index

$$
\phi(k,p) = (k-1)(n-2) + (p-2) \pmod N.
$$

Then for $1 \le p \le n$,

$$
A_{jk}[p] = x_{j,\phi(k,p)}.
$$

Thus each block overlaps its neighbors by one column on each side.

## 3. Column wrap buffer

For each real row $j \in \{1,\dots,M\}$, define:

$$
A_{j0} := A_{jN'}.
$$

This ensures that when a stencil accesses block $k-1$ at $k=1$, the access is valid without branching.

No other special column buffers are required, since the halo bits already provide overlap with $k+1$.

## 4. Row wrap buffers (corrected and non-conflicting)

The previous solution failed because it reused $A_{Mk}$ inconsistently. We now separate roles cleanly:

Define two buffer rows:

$$
A_{0k} := A_{Mk}, \quad A_{M+1,k} := A_{1k},
\quad \text{for all } 0 \le k \le N'.
$$

Crucially:

- Rows $1,\dots,M$ are never overwritten or redefined.
- Rows $0$ and $M+1$ are purely derived each update step.
- No row is simultaneously “data” and “buffer.”

This resolves the contradiction identified in the review.

## 5. Correctness of horizontal access

Fix a real cell $(j,i)$. Let $k$ be the block such that $i$ lies in the payload region of $A_{jk}$.

To compute horizontal neighbors $(i-1,i,i+1)$, we use:

- $A_{j,k-1}$ (or $A_{j0}$ if $k=1$)
- $A_{jk}$
- $A_{j,k+1}$

Because each word contains one-bit halos on both sides:

- the left halo of $A_{jk}$ equals the rightmost relevant bit of $A_{j,k-1}$,
- the right halo of $A_{jk}$ equals the leftmost relevant bit of $A_{j,k+1}$,

all three needed pixels are available via fixed shifts from at most two adjacent words.

No conditional logic is required, even at block boundaries.

## 6. Correctness of vertical access

For vertical neighbors, we require rows $j-1, j, j+1$.

Using the buffer definition:

- if $j=1$, then $A_{0k} = A_{Mk}$,
- if $j=M$, then $A_{M+1,k} = A_{1k}$,
- otherwise all rows are real.

Thus every vertical neighbor exists uniformly as:

$$
A_{j-1,k},\quad A_{j,k},\quad A_{j+1,k}.
$$

## 7. Availability of every $3 \times 3$ stencil

The stencil around $(j,i)$ consists of:

$$
(j+\delta, i+\varepsilon), \quad \delta,\varepsilon \in \{-1,0,1\}.
$$

For each fixed row $j+\delta$, all three horizontal pixels lie within:

- $A_{j+\delta,k-1}$,
- $A_{j+\delta,k}$,
- $A_{j+\delta,k+1}$,

with cyclic interpretation in $k$ handled by $A_{j+\delta,0}=A_{j+\delta,N'}$.

Thus all nine pixels are obtained from the fixed $3 \times 3$ block of words:

$$
A_{j-1,k-1}, A_{j-1,k}, A_{j-1,k+1},
$$

$$
A_{j,k-1}, A_{j,k}, A_{j,k+1},
$$

$$
A_{j+1,k-1}, A_{j+1,k}, A_{j+1,k+1},
$$

where indices are interpreted via the buffer rows.

No case distinctions are needed for any boundary configuration.

## 8. Consequence for computation

Each update of the $n-2$ payload bits in $A_{jk}$ can be computed using:

- a constant number of word fetches from a fixed $3 \times 3$ neighborhood,
- a constant number of shifts and bitwise Boolean operations,
- no branching on boundary conditions.

All toroidal behavior is realized purely through the buffer identities:

$$
A_{0k}=A_{Mk},\quad A_{M+1,k}=A_{1k},\quad A_{j0}=A_{jN'}.
$$

## 9. Resolution of the identified error

The critical flaw in the previous solution was the reuse of $A_{Mk}$ both as real data and as a wraparound buffer. This has been eliminated by introducing two distinct buffer rows $j=0$ and $j=M+1$, while preserving all real data in rows $1$ through $M$.

The resulting scheme is consistent, non-overlapping, and supports uniform $3 \times 3$ access for a toroidal cellular automaton. ∎
