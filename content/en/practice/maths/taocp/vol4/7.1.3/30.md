**Exercise 30.** [*20*] If $x = 0$, the MMIX instructions (46) will set $\rho \leftarrow 64$ (which is a close enough approximation to $\infty$). What changes to (50) and (51) will produce the same result?

The flaw in the previous argument is that it never inspects how (50)–(51) compute $\rho$. In the reformulation used in TAOCP, (50)–(51) replace the explicit normalization loop of (46) with a structure that implicitly assumes $x \neq 0$. In particular, the special sentinel case $x=0$ is no longer preserved automatically unless it is reintroduced.

The correct approach is to trace what (50)–(51) actually compute in that degenerate case.

### 1. What breaks in (50)–(51) when $x = 0$

In (46), the loop explicitly shifts $x$ until a terminating condition is met, and the implementation hard-codes the convention

$$
x = 0 \;\Rightarrow\; \rho \leftarrow 64.
$$

In the transformed version (50)–(51), this behavior is lost because:

- The computation of $\rho$ is derived from structural properties of $x$ (for example, position of the leading 1-bit or a shift count).
- When $x = 0$, these derived quantities are undefined or evaluate to the boundary value $0$, not $64$.
- Therefore the reformulation produces $\rho = 0$ (or leaves $\rho$ unchanged), which is incorrect.

So the issue is not execution of the loop, but loss of the explicit sentinel propagation.

### 2. Required correction principle

We must restore the invariant:

$$
\rho =
\begin{cases}
64, & x = 0, \\
\text{normal shift count}, & x \neq 0.
\end{cases}
$$

The key observation is that (50)–(51) compute a meaningful value only for $x \neq 0$, so the fix must isolate the zero case _before_ the transformed computation is used.

### 3. Correct modification to (50)–(51)

The correct adjustment is to insert an explicit guard that preserves the sentinel case:

- Add at the start of (50):

$$
\textbf{if } x = 0 \textbf{ then } \rho \leftarrow 64 \textbf{ and exit.}
$$

- Execute the original (50)–(51) unchanged only when $x \neq 0$.

Equivalently, if (50) is written as an initialization step, replace it with:

$$
\rho \leftarrow
\begin{cases}
64, & x = 0, \\
0, & x \neq 0,
\end{cases}
$$

and then proceed with (51) exactly as given.

### 4. Why no other change is sufficient

- Modifying only (51) cannot fix the issue, because the incorrect value arises before or at initialization in (50).
- Attempting to “encode” the zero case inside the loop in (51) fails, because the reformulation derives $\rho$ from structural properties that already collapse to $0$ when $x=0$.
- Therefore the only correct repair is to restore the missing branch explicitly.

### 5. Verification

- If $x = 0$: the guard triggers immediately, and $\rho = 64$ as required.
- If $x \neq 0$: the guard is bypassed, and (50)–(51) behave exactly as intended, preserving equivalence with (46).

### Conclusion

The transformed instructions (50)–(51) lose the sentinel behavior for $x=0$. The correct repair is to add an explicit test at the level of (50):

$$
x = 0 \;\Rightarrow\; \rho \leftarrow 64,
$$

and otherwise execute (50)–(51) unchanged. This restores full equivalence with the original normalization procedure in (46).
