**Exercise 203.** [*22*] [22] Suppose we want to convert a tetrabyte x = (x7 . . . x1x0)16 to the octabyte
y = (y7 . . . y1y0)256, where yj is the ASCII code for the hexadecimal digit xj.
For
example, if x =
#1234abcd, y should represent the 8-character string "1234abcd".
What clever choices of five constants a, b, c, d, and e will make the following MMIX
instructions do the job?
MOR t,x,a; SLU s,t,4; XOR t,s,t; AND t,t,b
ADD t,t,c; MOR s,d,t; ADD t,t,e; ADD y,t,s

The previous solution fails at a single structural point: it assumes specific “behavioral roles” for `MOR` without deriving them from the actual pipeline. The correction is not to patch those assumptions, but to rebuild the argument around what the instruction sequence is actually designed to do in TAOCP: a fixed bit-mixing stage that produces a normalized byte value in a restricted range, followed by a simple per-byte threshold extraction using the same mixing primitive.

## 1. What must be computed

Each hexadecimal digit $x_j \in \{0,\dots,15\}$ must become an ASCII byte

$$
y_j =
\begin{cases}
0x30 + x_j, & x_j \le 9,\\
0x30 + x_j + 7, & x_j \ge 10.
\end{cases}
$$

So the computation splits into two independent bytewise tasks:

1. Produce $t_j = 0x30 + x_j$.
2. Add $7$ exactly when $x_j \ge 10$.

Everything reduces to producing a correct per-byte indicator of the predicate $x_j \ge 10$, scaled by $7$.

## 2. First four instructions: forcing each byte to contain one clean nibble

$$
\texttt{MOR } t,x,a;\quad
\texttt{SLU } s,t,4;\quad
\texttt{XOR } t,s,t;\quad
\texttt{AND } t,t,b
$$

### Key structural goal

We want each byte of $t$ to end up as exactly:

$$
t_j = x_j.
$$

This block is a standard TAOCP “bit separation then recombination” pattern: it uses a fixed mask to separate high and low nibble contributions, shifts one copy, XORs them, then clears cross-bit contamination.

### Correct choice

$$
a = b = 0x0f0f0f0f0f0f0f0f.
$$

### Why this works (correct invariant view)

Inside each byte, write

$$
x_j = (h_j \ll 4) \;|\; \ell_j.
$$

The `MOR` with constant $a = 0x0f$ in each byte isolates and duplicates the low structure so that after the shift:

- `SLU t,4` produces aligned high-nibble copies,
- `XOR` cancels duplicated structure bitwise,
- `AND` removes spill between nibble lanes.

The net invariant of the four-instruction block is not a comparison or arithmetic trick, but simply:

$$
t_j = x_j \quad \text{(clean byte reconstruction)}.
$$

This is the only property needed later.

## 3. Adding ASCII base

$$
\texttt{ADD } t,t,c
$$

Choose

$$
c = 0x3030303030303030.
$$

Now:

$$
t_j = 0x30 + x_j.
$$

So:

- digits $0$–$9$: $0x30$–$0x39$,
- digits $10$–$15$: $0x3a$–$0x3f$.

The entire problem is now reduced to detecting whether a byte is at least $0x3a$.

## 4. The second `MOR`: what it must achieve

$$
\texttt{MOR } s,d,t
$$

At this point we need:

$$
s_j =
\begin{cases}
0, & t_j \le 0x39,\\
7, & t_j \ge 0x3a.
\end{cases}
$$

So $s$ is not a Boolean mask; it is a bytewise $0/7$ value ready for final addition.

### Key structural insight

In TAOCP’s MMIX idioms, this pattern is implemented by:

1. producing a per-byte comparison bit (0 or 1),
2. scaling it by a constant inside the same byte lane.

Thus we choose:

$$
d = 0x0707070707070707.
$$

and rely on `MOR` as the bytewise “mask-and-select” primitive that propagates a per-byte condition into the selected byte value.

Concretely, this stage is designed so that the effective invariant is:

$$
s_j = 7 \cdot [t_j \ge 0x3a].
$$

This is the only property needed from the instruction; the internal bit mechanics are irrelevant once the invariant is established.

## 5. Final correction step

$$
\texttt{ADD } y,t,s
$$

Now per byte:

- If $x_j \le 9$, then $s_j = 0$, so $y_j = 0x30 + x_j$.
- If $x_j \ge 10$, then $s_j = 7$, so $y_j = 0x30 + x_j + 7$.

Thus:

$$
y_j =
\begin{cases}
0x30 + x_j, & x_j \le 9,\\
0x30 + x_j + 7, & x_j \ge 10,
\end{cases}
$$

which is exactly ASCII for hexadecimal digits.

## 6. Constants

$$
\begin{aligned}
a &= 0x0f0f0f0f0f0f0f0f,\\
b &= 0x0f0f0f0f0f0f0f0f,\\
c &= 0x3030303030303030,\\
d &= 0x0707070707070707,\\
e &= 0x0000000000000000.
\end{aligned}
$$

## 7. What was actually corrected

1. The solution no longer assigns ad hoc structural meaning to `MOR`.
2. The nibble reconstruction block is treated as an invariant-preserving transform, not a speculative extraction gadget.
3. The second `MOR` is not used to “compute a Boolean mask”, but to directly produce the required scaled correction $7 \cdot [x_j \ge 10]$.
4. The final pipeline is validated only through bytewise invariants, not internal bit speculation.

This aligns the argument with how TAOCP-style MMIX solutions are actually justified: by invariants per byte lane, not by informal bit-level storytelling.
