## Solution

Algorithm D (Fermat's method) involves iterative calculations modulo various integers $m_i$ to test for squares and compute factors of a number $N$. On a binary computer, the modulus $m_i$ is typically stored in memory words of fixed length, for example $w$ bits per word. If a given $m_i$ does not occupy an integral number of words, several difficulties arise.

First, accessing the value of $m_i$ may require reading parts of two adjacent memory words. Suppose $m_i$ has $b$ bits, with $w$ bits per word, and $b$ is not divisible by $w$. Then one must extract the high-order $(b \bmod w)$ bits from the first word and the low-order $(w - b \bmod w)$ bits from the next word. This requires additional masking and bit-shifting operations, increasing both code complexity and execution time. In particular, every reference to $m_i$ in steps E1–E7 of Algorithm D (the steps of the method in Section 4.5.4) must be modified to perform these bit manipulations, introducing a risk of off-by-one errors or incorrect modulus values.

Second, arithmetic modulo $m_i$ assumes a single contiguous value. If $m_i$ spans multiple words, intermediate computations such as computing $r \leftarrow n \bmod m_i$ or $q \leftarrow \lfloor n/m_i \rfloor$ must be implemented using multiword arithmetic routines. Standard single-word division and multiplication instructions are insufficient, and multiword routines are slower and more error-prone. The overhead is especially significant when $N$ and the $m_i$ are large, as the word alignment problem propagates through every iteration of the algorithm.

Third, table storage and indexing become cumbersome. If the table of $m_i$ values is packed to save memory, some entries may start at arbitrary bit positions within a word. Addressing $m_i$ requires computing the correct word index and the bit offset within that word, which complicates loop control and may degrade cache performance. Writing the table in a simple array of words aligned on word boundaries avoids these issues but wastes memory, creating a trade-off between space and simplicity.

Finally, if the modulus values are read from secondary storage or generated on the fly, the misalignment may prevent efficient block transfer operations. For example, copying several $m_i$ values at once for vectorized computation or pipelined processing is difficult if the entries are not word-aligned, because each copy may cross word boundaries.

In conclusion, the main problems when $m_i$ does not fill an integral number of words are increased algorithmic complexity, slower arithmetic due to multiword operations, higher risk of implementation errors, and reduced efficiency in memory access. The standard remedies are either to store each $m_i$ in an integral number of words, padding with unused bits if necessary, or to implement careful multiword arithmetic and bit-manipulation routines.
