**Exercise 11.** [*M25*] *[M25]* Assuming that floating point arithmetic on numbers of type **double** is properly rounded in the sense of Section 4.2.2 (hence exact when the values are suitably restricted), convert the C routines *run_array* and *run_start* to similar programs that deliver double-precision random fractions in the range $[0\,.\,.\,1)$, instead of 30-bit integers.

**Verified:** yes  
**Solve time:** 2m40s

## Solution

Let $S = 2^{-30} = 1/m$, where $m = 2^{30} = \text{MM}$. The original generator produces integers $X_j \in {0,1,\dots,m-1}$ satisfying

$X_j = (X_{j-100} - X_{j-37}) \bmod m.$

To obtain random fractions in $[0,1)$, define

$R_j = \frac{X_j}{m} = S X_j.$

All arithmetic in the recurrence remains exact in the integer domain, while conversion to double occurs only at output time.

The implementation is therefore split into an integer core (identical to the original generator) and a double-precision interface.

The integer state array $ran_x$ and the modulus arithmetic are unchanged.

### Conversion of `ran_array`

The original procedure fills an integer array and updates the state. In the modified version, the computation is done in integers and only the output array is rescaled.

Let `mod_diff(x,y) = ((x)-(y)) & (m-1)` as before.

We introduce a local integer buffer $x[\cdot]$ to hold the working sequence.

```
#define KK 100
#define LL 37
#define MM (1L<<30)
#define mod_diff(x,y) (((x)-(y))&(MM-1))
#define SCALE (1.0 / (double)MM)

long ran_x[KK];

void ran_array(double aa[], int n)
{
    register int i, j;
    long x[KK + KK - 1];

    for (j = 0; j < KK; j++)
        x[j] = ran_x[j];

    for (j = KK; j < n; j++)
        x[j] = mod_diff(x[j-KK], x[j-LL]);

    for (i = 0; i < LL; i++, j++)
        ran_x[i] = mod_diff(x[j-KK], x[j-LL]);

    for (; i < KK; i++, j++)
        ran_x[i] = mod_diff(x[j-KK], ran_x[i-LL]);

    for (j = 0; j < n; j++)
        aa[j] = (double)x[j] * SCALE;
}
```

The recurrence and state update are identical to the original integer version, so the produced sequence $X_j$ is unchanged. The only modification is the final mapping $x \mapsto x/m$, which places all outputs in $[0,1)$.

### Conversion of `ran_start`

The initialization routine constructs the integer state $ran_x$ exactly as before. The warm-up step must use the integer version of `ran_array`, so a local integer driver is introduced that performs no scaling.

```
#define TT 70
#define is_odd(x) ((x)&1)

static void ran_array_int(long aa[], int n);

void ran_start(long seed)
{
    register int t, j;
    long x[KK + KK - 1];
    register long ss = (seed + 2) & (MM - 2);

    for (j = 0; j < KK; j++) {
        x[j] = ss;
        ss <<= 1;
        if (ss >= MM) ss -= MM - 2;
    }

    x[1]++;

    for (ss = seed & (MM - 1), t = TT - 1; t; ) {
        for (j = KK - 1; j > 0; j--)
            x[j+j] = x[j], x[j+j-1] = 0;

        for (j = KK + KK - 2; j >= KK; j--)
            x[j-(KK-LL)] = mod_diff(x[j-(KK-LL)], x[j]),
            x[j-KK] = mod_diff(x[j-KK], x[j]);

        if (is_odd(ss)) {
            for (j = KK; j > 0; j--)
                x[j] = x[j-1];
            x[0] = x[KK];
            x[LL] = mod_diff(x[LL], x[KK]);
        }

        if (ss) ss >>= 1; else t--;
    }

    for (j = 0; j < LL; j++)
        ran_x[j+KK-LL] = x[j];
    for (; j < KK; j++)
        ran_x[j-LL] = x[j];

    for (j = 0; j < 10; j++)
        ran_array_int(x, KK + KK - 1);
}
```

### Integer warm-up routine used internally

This routine is identical to the original `ran_array`, except that it does not produce double output.

```
static void ran_array_int(long aa[], int n)
{
    register int i, j;

    for (j = KK; j < n; j++)
        aa[j] = mod_diff(aa[j-KK], aa[j-LL]);

    for (i = 0; i < LL; i++, j++)
        ran_x[i] = mod_diff(aa[j-KK], aa[j-LL]);

    for (; i < KK; i++, j++)
        ran_x[i] = mod_diff(aa[j-KK], ran_x[i-LL]);
}
```

This separation ensures that all recurrence relations remain in exact integer arithmetic, while floating-point rounding occurs only in the final scaling step $X_j / 2^{30}$. The assumption of correctly rounded double arithmetic guarantees that each conversion produces the intended dyadic fraction in $[0,1)$ without introducing bias beyond representation limits.

This completes the construction. ∎
