**Exercise 8.** [*M15*] *[M15]* Suggest a good way to test whether an implementation of linear congruential generators is working properly.

**Verified:** yes  
**Solve time:** 6m24s

The original proposal correctly identified a sensible three-layer testing strategy, but it relied on an incorrect and oversimplified description of LCG period behavior. The corrected solution below fixes the theory and restates the testing method in a mathematically sound way.

## Solution

A linear congruential generator is defined by the recurrence

$$
X_{n+1} \equiv aX_n + c \pmod m, \quad n \ge 0,
$$

with modulus $m$, multiplier $a$, increment $c$, and seed $X_0$. Testing an implementation should verify that it correctly executes this recurrence and that it behaves consistently with the known arithmetic structure of such sequences.

A good testing strategy has three complementary parts: deterministic verification, structural (cycle) verification, and statistical validation. The second part must be based on correct number-theoretic facts about LCG periods.

### 1. Deterministic arithmetic verification

The first requirement is to ensure that the implementation performs the recurrence correctly, including modular reduction and state update.

This is best done by selecting a small modulus $m'$ and parameters $a', c'$ for which the entire sequence can be computed independently by a trusted method (hand calculation or a separate reference implementation using arbitrary-precision arithmetic). One then computes

$$
X_0', X_1', \dots, X_{k}'
$$

directly from the recurrence and compares term by term with the output of the implementation.

Any discrepancy must be treated as a definite implementation error in modular arithmetic, overflow handling, or state transition logic. This step validates correctness of the recurrence itself but does not address long-term behavior.

### 2. Cycle structure and period verification

Every LCG on a finite state space must eventually become periodic. That is, there exist integers $\mu \ge 0$ and $\lambda \ge 1$ such that

$$
X_{n+\lambda} = X_n \quad \text{for all } n \ge \mu.
$$

Here $\mu$ is the transient length and $\lambda$ is the cycle length.

A correct test does not assume a particular period in advance. Instead, it proceeds in two steps:

First, detect $(\mu, \lambda)$ using a cycle-finding method such as Floyd’s tortoise-and-hare algorithm or Brent’s algorithm. This checks that the implementation indeed produces a deterministic finite-state sequence and that the state update is consistent over long runs.

Second, compare the observed structure with the theoretical expectations derived from number theory when parameters are known.

For the mixed LCG ($c \ne 0$), the generator has full period $m$ for every seed if and only if the Hull–Dobell conditions hold:

1. $\gcd(c, m) = 1$,
2. every prime divisor of $m$ divides $a - 1$,
3. if $4 \mid m$, then $4 \mid (a - 1)$.

Only under these conditions is it correct to expect that the period equals $m$. If these conditions are not assumed or verified, one should not compare the observed period against $m$, since shorter cycles may be perfectly valid for the given parameters.

For the multiplicative case ($c = 0$ with $X_n \ne 0$), the sequence evolves in the multiplicative group modulo $m$. In general, the period of a state is the multiplicative order of $a$ modulo $m$, which always divides $\varphi(m)$ when $\gcd(a,m)=1$. Only in the special case where $m$ is prime does this order divide $m-1$, since the nonzero residues form a cyclic group of size $m-1$.

Thus, the correct interpretation is not that a specific period should be expected, but that the observed cycle length must be consistent with these algebraic constraints and with any known theoretical guarantees for the chosen parameters.

This distinction is essential: the purpose of the test is to detect implementation errors (such as incorrect modulus arithmetic or state corruption), not to assume maximal-period behavior unless it is mathematically guaranteed by the parameters.

### 3. Statistical validation

After arithmetic and structural correctness are established, one may test whether the output has acceptable distributional properties for its intended use.

Define normalized values

$$
U_n = \frac{X_n}{m}.
$$

These should approximate independent samples from the uniform distribution on $[0,1)$, though independence is never guaranteed by structure alone.

Standard empirical tests include histogram-based uniformity checks (such as a chi-squared test), tests for serial correlation, and examination of low-dimensional lattice structure via tuples

$$
(U_n, U_{n+1}, \dots, U_{n+t-1}).
$$

These tests do not prove correctness of the implementation, but they are useful for detecting gross failures such as poor parameter choice, arithmetic bias, or unintended correlations introduced by implementation errors.

A particularly strong validation method is to compare results against a trusted reference implementation or against published test vectors for the same parameter set, when available.

### Summary

A reliable test for an LCG implementation should not assume a desired period but should instead verify three independent properties.

First, it must confirm that the recurrence $X_{n+1} \equiv aX_n + c \pmod m$ is implemented correctly using deterministic small-scale checks. Second, it must detect the actual cycle structure and interpret it using correct number-theoretic facts, including the Hull–Dobell theorem for full-period behavior and multiplicative order theory in the case $c = 0$. Third, it should apply statistical tests only as a secondary diagnostic for distributional quality, not as a correctness proof.

Together, these steps provide a mathematically sound and practically effective method for validating an LCG implementation.
