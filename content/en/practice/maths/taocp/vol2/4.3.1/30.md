**Exercise 30.** &#9654; [*22*] [22] If memory space is limited, it may be desirable to use the same storage locations for both input and output during the performance of some of the algorithms in this section. Is it possible to have the quotient $q_0, \ldots, q_n$ stored in the same respective locations as $u_0, \ldots, u_{n-1}$ or $v_0, \ldots, v_{n-1}$ during Algorithm A or S? Is it possible to have the quotient $q_0, \ldots, q_m$ occupy the same locations as $u_0, \ldots, u_{m+n}$ in Algorithm D? Is there any permissible overlap of memory locations between input and output in Algorithm M?

**Verified:** yes  
**Solve time:** 3m49s  

---

## Solution

In Algorithm A, each digit $w_j$ is computed from the expression $u_j + v_j + k$, where $k$ is the carry from the previous position. The value $u_j$ and $v_j$ are used only in step A2 for index $j$, and after step A3 increases $j$, neither $u_j$ nor $v_j$ is ever accessed again. The carry $k$ is propagated forward independently of the storage locations of $u_j$ and $v_j$.

This locality implies that one of the input arrays may be overwritten as the computation proceeds from $j=0$ to $j=n-1$. If, for example, the result digits $w_j$ are stored in the same locations as $u_j$, then at the moment $u_j$ is read in step A2 it still contains the correct input value, and immediately afterward it may be replaced by $w_j$ without affecting any later computation. The same argument applies if $v_j$ is used as the output array instead. However, simultaneous use of both $u$ and $v$ as output storage is unnecessary rather than impossible; a single input array suffices to host the output. Thus Algorithm A permits in-place storage of the sum in the locations of either operand, but not a requirement of multiple overlapping outputs beyond one array.

The same reasoning applies to Algorithm S. In step S2, the value $u_j - v_j + k$ is computed using only the current digits $u_j$, $v_j$, and the incoming borrow indicator $k$. After $w_j$ is stored, neither $u_j$ nor $v_j$ is needed again. The borrow value is carried independently of the digit arrays. Therefore, as in Algorithm A, either operand array may be overwritten by the output array $w$, since each entry is consumed exactly once in left-to-right order. No future step refers to earlier digit positions of either operand.

In Algorithm D, the situation is different because the quotient digits are produced in a dependency structure tied to progressively refined prefixes of the dividend. However, the algorithm explicitly places the quotient digits in the high-order positions of the dividend array: at step D8 (and corresponding steps), each quotient digit $q_j$ is stored into a position of the form $u_{j+n}$. This shows that the quotient array $q_0,\ldots,q_m$ is intended to occupy storage originally associated with the high-index portion of $u_0,\ldots,u_{m+n}$. The correctness of this reuse follows from the fact that once the algorithm has advanced past stage $j$, the values in positions below $u_{j+n}$ are no longer needed in their original form except as modified by the subtraction steps already completed. Hence the quotient may be safely written into the upper part of the dividend storage, so that $q_j$ replaces $u_{j+n}$ without loss of required information.

No similar overlap is possible with the divisor $v_0,\ldots,v_{n-1}$. Each quotient digit computation in Algorithm D depends repeatedly on the full current divisor, in particular during trial quotient estimation and correction steps. Since these values must remain unchanged throughout all iterations, any overwrite of $v$ would destroy necessary data for subsequent quotient digit computations. Therefore the divisor storage cannot be reused for the quotient.

In Algorithm M, each product digit $w_{i+j}$ is formed through repeated accumulation of terms of the form $u_i \times v_j + w_{i+j} + k$. Here every $u_i$ is used $n$ times, once for each $v_j$, and every $v_j$ is used $m$ times, once for each $u_i$. Moreover, these uses are interleaved across all stages of the algorithm, so no digit of $u$ or $v$ becomes permanently irrelevant after a single step. If either input array were overwritten during computation, later partial products would require values that have already been destroyed. This dependency structure prevents any safe in-place replacement of either operand array by the product array.

The output array $w_0,\ldots,w_{m+n-1}$ is also larger than either input array. Even ignoring dependencies, there is no disjoint region within either $u$ or $v$ large enough to store all product digits without overlap. Consequently, Algorithm M does not permit overlap between input and output storage.

This completes the proof. ∎
