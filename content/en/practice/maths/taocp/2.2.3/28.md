## Setup

Let $G$ be a finite directed graph whose nodes correspond to all possible positions in a two-person game, as described in the exercise. Each node $x \in G$ has zero or more successors, where a successor $y$ of $x$ satisfies that there exists a legal move from $x$ to $y$. We define the _predecessor_ relationship as the converse: $x$ is a predecessor of $y$ if $y$ is a successor of $x$.

Certain nodes are terminal positions with no successors; these are labeled as _won_ or _lost_ according to the rules of the game. Let $W_0 \subseteq G$ denote the set of positions initially classified as _won_, and $L_0 \subseteq G$ denote positions initially classified as _lost_. All other positions are initially unclassified.

The goal is to propagate these classifications throughout the graph to determine the outcome of every node. Specifically, we must assign to each position $x$ a value in ${\text{won},\text{lost}}$ according to the following rules:

1. A position $x$ is _lost_ if every successor of $x$ is _won_.
2. A position $x$ is _won_ if it has at least one successor that is _lost_.

We assume that the positions are stored in memory using linked allocation. Let `FIRST` be the link to the first node of a list representing all positions yet to be classified, and let `AVAIL` point to the pool of available nodes, as defined in Section 2.2.3. Each node contains fields `INFO` for storing the classification and `LINK` for linking nodes together.

We must design an algorithm to assign a classification to every position in $G$, using only linked list operations (`X \Leftarrow \text{AVAIL}`, `AVAIL \Leftarrow X`) and iterative updates to the nodes' `INFO` fields.

## Solution

We proceed by backward induction on the game graph, starting from terminal nodes and moving to predecessors. The algorithm iteratively marks nodes as _won_ or _lost_ according to the definitions above, using a linked list of nodes whose outcome is not yet determined.

1. **Initialization**: Place all terminal nodes in two linked lists, `W` for won nodes and `L` for lost nodes, using the `AVAIL` mechanism to allocate nodes. For each terminal node $x \in W_0$, set `INFO(x) ← won`; for $x \in L_0$, set `INFO(x) ← lost`. Link these nodes together via their `LINK` fields to form the lists `W` and `L`. Set `UNDECIDED` to the list of all other nodes.
2. **Propagation loop**: While `UNDECIDED ≠ Λ`, do the following:

a. Remove a node `X` from `UNDECIDED` (`X \Leftarrow UNDECIDED`).

b. Inspect all successors of `X`. Let `S` denote the list of successors. If there exists `Y ∈ S` with `INFO(Y) = lost`, then set `INFO(X) ← won`. Otherwise, if all successors have `INFO(Y) = won`, then set `INFO(X) ← lost`.

c. If `INFO(X)` has been determined, append `X` to the appropriate list (`W` or `L`) via its `LINK` field and remove it from `UNDECIDED` by setting `LINK(X) ← Λ` if necessary.

d. If `INFO(X)` remains undetermined, append `X` back to the end of `UNDECIDED` to be processed in a future iteration.
3. **Termination**: The loop terminates when `UNDECIDED = Λ`. At this point, every node has `INFO` set to either _won_ or _lost_. The propagation ensures that the classification rules are satisfied: every _won_ node has at least one _lost_ successor, and every _lost_ node has only _won_ successors.

**Linked list operations used**:

- `X \Leftarrow UNDECIDED` to remove a node from the list.
- `UNDECIDED \Leftarrow X` to return a node to the list if its classification is still unknown.
- `LINK(X) ← W` or `LINK(X) ← L` to append a node to the appropriate classified list.
- `X \Leftarrow AVAIL` and `AVAIL \Leftarrow X` to manage auxiliary nodes if needed.

**Correctness argument**:

Each iteration either moves a node from `UNDECIDED` to `W` or `L` or postpones its classification by placing it at the end of `UNDECIDED`. Since the graph is finite and each move strictly propagates the classification backward, the process must terminate. Every node is examined after all its successors have been classified. Therefore, the algorithm correctly assigns `INFO(x) = won` if $x$ has a successor with `INFO = lost`, and `INFO(x) = lost` if all successors are `won`.

**Memory efficiency**:

The algorithm uses a single `UNDECIDED` list and auxiliary `W` and `L` lists. All node allocations and deallocations are handled via `AVAIL`, in accordance with the linked allocation procedures (4) and (5) in Section 2.2.3. No additional sequential arrays are required, so the algorithm is suitable for large game graphs where sequential allocation would be impractical.

## Verification

Let $x$ be an arbitrary node:

1. If $x$ is terminal and has no successors, then it is placed in `W` or `L` according to its initial classification. The `INFO` field is correctly set.
2. Suppose all successors of $x$ have been classified. If any successor $y$ has `INFO(y) = lost`, then the propagation rule sets `INFO(x) ← won`. Otherwise, all successors have `INFO(y) = won`, so `INFO(x) ← lost`. No node is left in `UNDECIDED` indefinitely because each iteration moves nodes whose successors are fully classified, guaranteeing eventual classification of all nodes.
3. Cycles are handled correctly: if a set of nodes forms a cycle, each node will be deferred until at least one successor outside the cycle is classified. By finiteness of the graph, some node in the cycle eventually acquires a classification, propagating through the cycle until all nodes are classified.

Thus, every node satisfies the defining conditions for _won_ or _lost_, and no node is misclassified. The linked list manipulations follow Section 2.2.3 conventions, using `AVAIL` as needed, preserving the integrity of the lists.

This completes the proof.

## Notes

1. The algorithm can be expressed entirely in MIXAL using the linked stack operations (8) and (9), treating `UNDECIDED` as a stack. Nodes can be pushed back onto `UNDECIDED` if their classification is not yet determined.
2. An optimization is to maintain, for each node, a counter of unclassified successors. When this counter reaches zero, the node's classification can be decided immediately, reducing the number of passes over the list.
3. The approach generalizes to games with more than two outcomes (draw, tie, etc.) by extending the `INFO` field to represent multiple classifications and updating propagation rules accordingly.
