**Exercise 5.** [**] [24] Show that if the search for max(Ki,...,Kj) in step S2 is carried out by examining keys in left-to-right order Ki, Ki+1, ..., Kj, instead of going from right to left as in Program S, it is often possible to reduce the number of comparisons needed on the next iteration of step S2. Write a MIX program based on this observation.

The earlier solution fails at a single decisive point: it treats “comparison history” as if it implies a structural elimination of part of the array. That is not valid in a selection procedure, and it is the reason both the correctness argument and the MIX program break down.

The correct fix is to restart from first principles and separate two facts that were incorrectly merged:

1. A left-to-right scan can record extra information (maximum, second maximum, and where the maximum last changed).
2. That information does not permit skipping any part of the remaining array in the next iteration of $S2$.

Once that distinction is respected, the only legitimate optimization is more modest: it reduces redundant comparisons during initialization of the next scan, not elimination of a prefix.

## 1. Where the previous argument fails

### Critical error: “prefix can be skipped”

The claim that the segment $K_1,\dots,K_{m-1}$ can be omitted in later iterations is false.

Even if an element has been compared against earlier maxima, that does not imply it cannot become the maximum of the reduced set $K_1,\dots,K_{j-1}$. The key logical gap is:

$$
\text{“has lost earlier comparisons”} \;\not\Rightarrow\; \text{“cannot win later selection among fewer elements”}.
$$

Selection depends only on relative order within the current set, not on history of comparisons.

So any correct solution must retain:

$$
S2 \text{ always scans all of } K_1,\dots,K_{j-1}.
$$

No prefix skipping is allowed.

## 2. What left-to-right scanning _actually_ gives

A left-to-right scan of $K_i,\dots,K_j$ while maintaining:

- $rA$: current maximum
- $rB$: second largest value
- $m$: position where $rA$ was last updated

produces one valid piece of structural information:

At the end of the scan, every element after position $m$ was directly compared against the final maximum $rA$ at the moment $rA$ became dominant.

However, this still does **not** certify any element as irrelevant in future iterations. It only records how the maximum was discovered.

So the only safe reuse is:

- better initialization of the next maximum search
- occasional avoidance of one redundant comparison in the first steps of the next $S2$

not any change in the scan range.

## 3. Correct way to exploit the information

In Program S, each iteration of $S2$ finds the maximum of a shrinking prefix:

$$
K_1,\dots,K_j \to K_1,\dots,K_{j-1}.
$$

Normally, $S2$ initializes the candidate maximum from one element and then performs comparisons for all remaining elements.

With left-to-right scanning, we additionally know:

- the largest value $rA$
- the second largest value $rB$

This allows a small improvement in the next iteration:

### Key observation

When the next $S2$ begins, we can initialize the running maximum and second maximum in a way that avoids one redundant comparison that would otherwise occur in establishing initial order between the first two elements of the scan.

This is the only universally valid gain:

- no skipped elements
- no reduced scan interval
- only reduced initialization comparisons

## 4. Correct revised structure of $S2$

We assume the standard framework of Program S.

### Standard behavior (baseline)

Each $S2$:

1. sets initial maximum from one element
2. compares every remaining element once

So each pass costs exactly:

$$
(j-1), (j-2), \dots
$$

## 5. Improved $S2$ using left-to-right scan information

We maintain across iterations:

- $rA$: maximum of previous pass
- $rB$: second maximum of previous pass

### Correct idea

At the start of the next pass:

- We already know two strong candidates.
- We can initialize the scan so that the first comparison between two fresh elements is avoided.

### Correct MIX-level structure (conceptual)

Let the current range be $K_1,\dots,K_j$.

#### Initialization

Instead of blindly starting from the last element pair, we initialize:

- $rA \leftarrow rA$ from previous pass
- $rB \leftarrow rB$
- start scanning from $K_{j-1}$

We ensure the first comparison is between a new element and an already ordered pair $(rA,rB)$, not between two unstructured elements.

### Correct $S2$ loop behavior

```
S2:  i ← j-1

     rA, rB already known

     while i ≥ 1 do
         if Ki > rA then
             rB ← rA
             rA ← Ki
         else if Ki > rB then
             rB ← Ki
         fi
         i ← i - 1
```

This is exactly Program S style, but with one difference:

- the first comparison of each iteration is cheaper in practice because $rA, rB$ are pre-structured from a left-to-right scan

There is no change to the index range.

## 6. What improvement is actually achieved

### What is saved

A small number of comparisons is saved due to:

- better initial ordering of candidate maxima
- fewer comparisons needed to establish initial $rA,rB$ relationships in each pass

### What is NOT saved

- no prefix is skipped
- no iteration range is reduced
- worst-case comparison count remains unchanged

## 7. Correct MIX program (structure-level correction)

A faithful MIX version must reflect:

- full scan in each $S2$
- maintained $rA, rB$
- no “jumping” index $m$

### Correct modification principle

Only initialization changes.

Pseudo-MIX form:

```
        S2:
        LDA rA
        STA MAX

        LDA rB
        STA SEC

        ENT1 j-1        i ← j-1

LOOP    LDX Ki

        CMPX MAX
        JLE  NOTMAX
        STX SEC
        LDA MAX
        STA SEC
        STX MAX
        JMP NEXT

NOTMAX  CMPX SEC
        JLE  NEXT
        STX SEC

NEXT    DEC1 1
        J1P LOOP
```

The key correction is conceptual:

- no altered lower bound
- no use of $m$
- only improved initialization state from previous pass

## 8. Final conclusion

The previous solution failed because it attempted to convert “knowledge about past comparisons” into “removal of elements from future searches.” That is not valid in selection algorithms.

The correct interpretation is weaker but sound:

- Left-to-right scanning preserves extra ordering information (especially $rA$ and $rB$).
- That information can reduce a small number of redundant comparisons at the start of subsequent $S2$ executions.
- It does not change the range of the scan and does not permit skipping any portion of the array.

Thus, the improvement exists, but it is strictly a constant-factor reduction in comparisons, not a structural reduction of the search interval.
