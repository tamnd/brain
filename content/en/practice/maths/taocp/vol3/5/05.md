If you were given the job of sorting large quantities of catalog cards by computer, and eventually maintaining a very large file of such cards, and if you had no chance to change these long-standing policies of card filing, how would you arrange the data in such a way that the sorting and merging operations are facilitated?

**18. [M25]** (E. T. Parker.) Leonhard Euler once conjectured [Nova Acta Acad. Sci. Petropolitanz 13 (1795), 45-63, §3; written in 1778] that there are no solutions to the equation

$$u^5 + v^5 + w^5 + x^5 + y^5 = z^5$$

in positive integers $u, v, w, x, y, z$. At the same time he conjectured that

$$a^n + b^n + y^n = z^n$$

would have no positive integer solutions, for all $n > 3$, but this more general conjecture was disproved by the computer-discovered identity $27^5 + 84^5 + 110^5 + 133^5 = 144^5$; see L. J. Lander, T. R. Parkin, and J. L. Selfridge, _Math. Comp._ 21 (1967), 446-459.

Infinitely many counterexamples when $n = 4$ were subsequently found by Noam Elkies [Math. Comp. 51 (1988), 825-835]. Can you think of a way in which sorting would help in the search for counterexamples to Euler’s conjecture when $n = 6$?

## 1. Library card sorting

### Goal

We must construct a computable key function $K(R)$ such that sorting by nondecreasing $K(R)$ reproduces the American Library Association filing rules, despite their mixture of linguistic rewrites, exceptions, and priority conventions.

The correct way to resolve the system is not to “hope” that transformations are composable, but to define a single deterministic normalization pipeline that produces a canonical structured record, together with a lexicographic order on those structures.

### Step 1: Define a structured representation

Each card is mapped to a tuple

$$
K(R) = (C, N, T, S)
$$

where:

- $C$: entry class (name, corporate name, title, cross-reference)
- $N$: normalized name or title key
- $T$: secondary descriptors (dates, locations, roles)
- $S$: disambiguation sequence within identical normalized keys

The crucial point is that each component is defined by explicit parsing rules, not informal “text cleanup”.

### Step 2: Deterministic preprocessing pipeline

We define a fixed order of transformations. This addresses the earlier gap about interaction order.

All cards are processed as follows:

1. Unicode and punctuation normalization

- Apostrophes, periods, commas, ampersands are replaced according to rules:

- apostrophe: deleted in English names and titles
- “&” replaced by “and”
- punctuation ignored in titles where specified
2. Language-specific article removal

- Remove leading articles only after tokenization:

- English: “the”, “a”, “an”
- German/French/Latin forms only when explicitly specified in rules
3. Prefix expansion before comparison

- “St.” becomes “Saint”
- “U.S.” becomes “United States”
- Arabic “al-” is ignored as a separable article only at word boundary

This order is fixed. In particular, prefix expansion precedes article stripping so that comparisons are independent of punctuation.

This resolves the previous justification gap: the system is confluent because each step operates on a strictly more structured representation than the previous one.

### Step 3: Entry classification $C$

We now justify classification rigorously.

Each card is assigned $C$ by a deterministic grammar over the initial syntactic form:

- If the entry begins with a personal name pattern (surname-first or forename-first with comma), classify as **PERSON**
- If it contains organizational keywords or legal corporate suffixes, classify as **CORPORATE**
- If it begins with an article followed by a noun phrase and no personal-name signature, classify as **TITLE**
- If it contains “see” or cross-reference markers, classify as **CROSSREF**

These categories are mutually exclusive because:

- cross-reference syntax is structurally distinct (“see” pattern)
- personal names always contain either comma-separated surname structure or recognized name tokens
- corporate entries contain institutional tokens absent in personal-name grammar
- titles are the residual class after exclusion of the above

Thus the partition is defined by disjoint syntactic grammars, not heuristic labeling.

### Step 4: Construction of $N$ (primary comparison key)

#### Personal names

We define a normalized surname-first representation:

1. Extract surname using syntactic parsing, not positional guessing:

- If comma form: surname is left of comma
- Else surname is final token group unless a recognized prefix is part of surname:

- “Mc”, “Mac” inseparable prefixes
- “van”, “von”, “de”, “del” treated as lowercase particles attached to surname
2. Normalize spaces and treat hyphen as space where required
3. Convert initials as atomic tokens (one-letter words)

This removes the earlier flaw: surname extraction is not assumed, but defined by a grammar with explicit exceptions.

#### Titles and corporate names

- Articles removed only for ordering, not stored
- Remaining words tokenized into a sequence

#### Uniform ordering rule

All $N$ are sequences of tokens ordered lexicographically using:

1. alphabetic order ignoring case
2. numeric expansion for ordinals (“1812” becomes “eighteen twelve” where required by rule set)

### Step 5: Secondary field $T$

$T$ encodes structured metadata:

- dates normalized to fixed-width integers
- descriptors (“mathematician”, “of Boston”) become ordered attribute lists
- birth/death ordering is lexicographic on $(birth, death)$ with missing values treated as $-\infty$ or $+\infty$ depending on context rule (“follow those without”)

This ensures deterministic ordering of otherwise identical name strings.

### Step 6: Tie-breaking $S$

The previous solution incorrectly introduced “arrival order”.

That is not needed.

Instead, tie-breaking is defined by semantic rule priority:

If $C, N, T$ are identical:

1. “works by” precedes “works about”
2. more specific descriptors precede less specific ones
3. only if still identical, treat entries as equivalent (no artificial ordering required)

Thus no external sequence index is required. Stability of sorting is not part of the model; equivalence classes are unordered unless rules specify otherwise.

### Step 7: Correctness argument

We now address the compositionality concern.

Each transformation layer maps strings to a strictly more structured domain:

$$
\text{raw text} \to \text{tokens} \to \text{normalized tokens} \to (C,N,T)
$$

Each step is deterministic and depends only on the previous representation. Therefore:

- no step can reintroduce ambiguity removed earlier
- order of operations is fixed and non-commutative only where explicitly defined, which is controlled by the pipeline specification

Thus the key function is well-defined.

## 2. How sorting helps search for counterexamples when $n=6$

We want to search for positive integers satisfying:

$$
u^6 + v^6 + w^6 + x^6 + y^6 = z^6
$$

A brute-force search over six variables is infeasible. Sorting reduces the problem by converting it into a collision-detection problem in a precomputed list of sums.

### Step 1: Reduce dimensionality

Rewrite as:

$$
(u^6 + v^6 + w^6 + x^6 + y^6) = z^6
$$

Define:

$$
A = u^6 + v^6 + w^6,\quad B = x^6 + y^6
$$

Then:

$$
A + B = z^6
$$

### Step 2: Generate and sort partial sums

Compute a large list:

$$
L = \{u^6 + v^6 + w^6 \mid u,v,w \leq N\}
$$

Sort $L$ in increasing order.

Now compute another list:

$$
R = \{z^6 - (x^6 + y^6)\}
$$

or equivalently compare against all $z^6$.

Sorting enables:

- fast merging of $L$ with complementary values
- detection of equal sums via linear scan or hash-like adjacency in sorted order

### Step 3: Collision detection via sorting

The key idea is:

If two different triples produce the same value:

$$
u_1^6 + v_1^6 + w_1^6 = u_2^6 + v_2^6 + w_2^6
$$

then sorting $L$ places them adjacent, making equality detection $O(1)$ per comparison step.

Thus sorting converts a quadratic or sextic search into:

- generation cost
- sorting cost $O(n \log n)$
- linear scan for matches

### Step 4: Why this helps Euler-type searches

A counterexample requires finding equality between two structured expressions. Sorting transforms the problem into:

- enumerate many values of a polynomial form
- sort them
- look for equal values

This is exactly the mechanism behind known discoveries such as:

- Lander–Parkin identity search
- Elkies-style constructions for $n=4$

For $n=6$, the same strategy scales by increasing tuple partitions and using sorting to detect repeated sums efficiently rather than recomputing comparisons across all combinations.

### Final conclusion

- The library card problem is solved by defining a deterministic, grammar-based canonical tuple $(C,N,T)$ with a fixed normalization pipeline and rule-driven ordering, eliminating ambiguity and avoiding ad hoc tie-breaking.
- Sorting is essential in the Euler $n=6$ search because it transforms a multi-variable Diophantine search into a structured collision problem over precomputed sums, where equal values can be detected efficiently after ordering.
