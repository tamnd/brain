**Exercise 6.** [**] [23] (*Next fit.*) After Algorithm A has been used repeatedly, there
will be a strong tendency for blocks of small `SIZE` to remain at the
front of the `AVAIL` list, so that it will often be necessary to search
quite far into the list before finding a block of length `N` or more. For
example, notice how the size of the blocks essentially increases in Fig.
42, for both reserved and free blocks, from the beginning of memory to the
end. (The `AVAIL` list used while Fig. 42 was being prepared was kept
sorted by order of location, as required by Algorithm B.) Can you suggest
a way to modify Algorithm A so that (a) short blocks won't tend to
accumulate in a particular area, and (b) the `AVAIL` list may still be
kept in order of increasing memory locations, for purposes of algorithms
like Algorithm B?

**Verified:** no  
**Solve time:** 4m24s  

---

Modify Algorithm A so that the search does not always begin at the front of the `AVAIL` list. Maintain a pointer to the position at which the previous successful search ended. When a request for a block of length $N$ arrives, begin the search at that position and continue through the list in order of increasing memory locations; if the end of the list is reached, continue from the front until either a suitable block is found or the search returns to the starting position.

This "next fit" policy distributes allocations over the entire `AVAIL` list instead of repeatedly consuming the first large blocks encountered. Small residual blocks therefore tend to be scattered throughout memory rather than accumulating near the front. Since the `AVAIL` list itself remains ordered by increasing memory locations, Algorithm B can still use the same ordering when adjacent free blocks are merged.
