**Exercise 9.** [**] [26] In what ways could a reservation algorithm be designed to use the
best-fit method, without searching through the whole `AVAIL` list? (Try
to think of ways that cut down the necessary search as much as possible.)

**Verified:** yes  
**Solve time:** 3m37s  

---

A reservation algorithm can avoid scanning the entire `AVAIL` list by storing free blocks in a structure that makes the successor of a given size efficiently accessible, so that the smallest block of size at least $N$ can be located without examining all blocks.

The simplest correct modification is to maintain `AVAIL` in nondecreasing order of `SIZE` rather than in order of memory location. The invariant is that if $P_1, P_2, \ldots$ are the blocks in `AVAIL`, then

$SIZE(P_1) \le SIZE(P_2) \le \cdots.$

Given this ordering, a reservation request for size $N$ proceeds by scanning from the beginning of the list until the first block $P$ satisfying $SIZE(P) \ge N$ is encountered. This block must be the best-fit block, since every earlier block has strictly smaller size and therefore cannot satisfy the constraint, while every later block has size at least $SIZE(P)$ and therefore is not a better fit.

The correctness depends only on the ordering invariant. Under this invariant, no comparison with blocks after the first qualifying block is necessary, so the scan stops immediately upon finding a feasible block. This avoids searching through the entire `AVAIL` list in the sense that only a prefix of blocks with size less than $N$ is examined.

A second way to reduce search is to replace the single list by multiple free lists indexed by block size. One can maintain an array of lists $L_s$, where $L_s$ contains all free blocks of size $s$. A request for size $N$ then begins by inspecting $L_N$, then $L_{N+1}$, $L_{N+2}$, and so on until a nonempty list is found. The first nonempty list encountered provides a block of minimum possible size at least $N$, hence it is a best-fit choice. This reduces search to only those size classes that are actually traversed, and no scanning within unrelated sizes is required if each $L_s$ is maintained as a linked list.

A third method uses a balanced search structure keyed by `SIZE`, such as a binary search tree in which each node represents a distinct block size and stores a linked list of blocks of that size. A best-fit request for $N$ is processed by searching the tree for the smallest key $s \ge N$. This search follows a single root-to-leaf path determined by comparisons with $N$. Once the node with minimal qualifying size is found, any block from its associated list is used. The search does not examine all nodes in `AVAIL`, only those on the search path determined by the tree structure.

Each method preserves correctness because the selected block is guaranteed to be of minimum size among all blocks with size at least $N$, while reducing search by organizing `AVAIL` so that irrelevant blocks are never inspected. ∎
