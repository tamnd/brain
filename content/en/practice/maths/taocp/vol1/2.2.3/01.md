[Section 2.2.3: Linked Allocation](../)

**Exercise 1.** [*10*] Operation (9) for popping up a stack mentions the possibility of `UNDERFLOW`; why doesn't operation (8), pushing down a stack, mention the possibility of `OVERFLOW`?

**Verified:** yes  
**Solve time:** 1m09s  

---

Operation (8) uses the allocation primitive $P \Leftarrow \text{AVAIL}$, which is defined in (6). That definition already includes the test for $\text{AVAIL} = \Lambda$ and triggers `OVERFLOW` in that case before any node is removed from the available space list. Hence the possibility of running out of storage is handled at the level of the primitive operation, so no additional check is written in (8).

Operation (9) cannot rely on a similar hidden mechanism, since returning a node to the available list via `AVAIL \Leftarrow P` is always possible and never fails, while removing a node from the stack depends on the run-time condition $T = \Lambda$. This condition is local to the stack structure and is not mediated by a separate allocation routine, so it must be tested explicitly and leads to `UNDERFLOW` when it holds. This completes the distinction. ∎
