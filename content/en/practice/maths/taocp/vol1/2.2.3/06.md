**Exercise 6.** [*21*] In operation (14) we set `LINK(P) ← Λ`, while the very next insertion at the rear of the queue will change the value of this same link field. Show how the setting of `LINK(P)` in (14) could be avoided if we make a change to the testing of "`F = Λ`" in (17).

The flaw in the previous solution is that it still implicitly relies on evaluating $\text{LINK}(P)$ in the case where $P$ is the last node. The correct repair is to ensure that $\text{LINK}(P)$ is never accessed when $P$ is the final element, so that its value need not be defined at all. This is achieved by changing the emptiness test in (17) so that the special case of a one-element queue is detected before any link field is used.

## Corrected solution

We consider the linked queue representation with pointers $F$ (front) and $R$ (rear). In the original algorithm, insertion at the rear (14) sets $\text{LINK}(P) \leftarrow \Lambda$ to guarantee that the last node always has a null link, so that deletion (17), which uses $\text{LINK}(P)$, is safe.

We show that this assignment can be eliminated if we modify the test for $F = \Lambda$ in (17) so that deletion does not require the link field of the last node.

### Key idea

We prevent any access to $\text{LINK}(P)$ when $P$ is the last node by detecting the one-element case using the rear pointer $R$.

Thus we replace the deletion step (17) by a version that distinguishes:

- the case $P = R$ (the queue has exactly one element),
- the case $P \neq R$ (the queue has at least two elements).

### Modified deletion (replacement for (17))

Assume deletion removes the front element $P \leftarrow F$.

1. Set $P \leftarrow F$.
2. If $P = R$, then the queue has only one element. Set

$$
F \leftarrow \Lambda,\quad R \leftarrow \Lambda.
$$
3. Otherwise (so $P \neq R$), set

$$
F \leftarrow \text{LINK}(P).
$$
4. (Optional consistency check) If $F = \Lambda$, then also set $R \leftarrow \Lambda$. This follows automatically from step 2 and the structure of the queue.

### Why $\text{LINK}(P) \leftarrow \Lambda$ is unnecessary

We examine when $\text{LINK}(P)$ is used.

- If $P \neq R$, then $P$ is not the last node. Its successor exists, and $\text{LINK}(P)$ is defined and points to the next node. No special initialization is required.
- If $P = R$, then we are deleting the only node in the queue. In this case, the algorithm explicitly avoids evaluating $\text{LINK}(P)$ and directly sets $F \leftarrow \Lambda$.

Hence $\text{LINK}(P)$ is never read when $P$ is the last node.

### Effect on the emptiness test

The original test in (17) uses $F = \Lambda$ after updating $F$. In the modified version, emptiness is still characterized by

$$
F = \Lambda \iff R = \Lambda.
$$

This invariant is preserved because:

- In the one-element case we explicitly set both $F$ and $R$ to $\Lambda$.
- In all other cases, $F$ becomes a valid node via $\text{LINK}(P)$, and $R$ remains unchanged.

Thus the condition $F = \Lambda$ remains correct and sufficient.

### Corrected invariant

We no longer require:

$$
\text{LINK}(\text{last node}) = \Lambda.
$$

Instead we maintain only:

- The nodes reachable from $F$ via repeated application of $\text{LINK}$ form a linear list.
- $R$ points to the last node.
- If $F = \Lambda$, then $R = \Lambda$, and vice versa.
- The link field of the last node is irrelevant and may contain an arbitrary value.

### Conclusion

By modifying deletion (17) to test $P = R$ before accessing $\text{LINK}(P)$, we ensure that the link field of the last node is never referenced. Therefore the assignment

$$
\text{LINK}(P) \leftarrow \Lambda
$$

in insertion (14) is unnecessary.
