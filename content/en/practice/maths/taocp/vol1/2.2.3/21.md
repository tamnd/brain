**Exercise 21.** [*21*] Would Algorithm `T` still perform a valid topological sort if one of the relations "$j \prec k$" were repeated several times in the input? What if the input contained a relation of the form "$j \prec j$"?

## Solution

Let Algorithm $T$ process an input consisting of relations of the form $j \prec k$. In the algorithm, each such relation contributes to the value of $\text{COUNT}[k]$, and later, when a node $j$ is removed from the queue, the algorithm decreases $\text{COUNT}[k]$ once for every occurrence of $j \prec k$ in the processed structure.

### Repeated relations $j \prec k$

Assume the input contains the same relation $j \prec k$ repeated $m \ge 1$ times.

Each occurrence of $j \prec k$ increases $\text{COUNT}[k]$ by $1$, so the initial value of $\text{COUNT}[k]$ reflects the multiplicity of incoming relations.

During execution, when node $j$ is removed from the queue, Algorithm $T$ scans the list of successors of $j$ and processes each stored arc $j \prec k$ exactly once. If the input contained $m$ copies of $j \prec k$, then $k$ appears $m$ times in the adjacency structure of $j$, and therefore $\text{COUNT}[k]$ is decreased exactly $m$ times.

The invariant that $\text{COUNT}[k]$ equals the number of predecessors of $k$ not yet removed from the queue remains valid, since both initialization and decrementing account for multiplicity in the same way.

Whenever $\text{COUNT}[k]$ reaches $0$, node $k$ is enqueued exactly once, since the algorithm inserts $k$ into the queue only at the moment the counter first becomes zero. Additional duplicate edges cannot cause further enqueue operations, because subsequent decrements would make $\text{COUNT}[k]$ negative or keep it nonzero, but no second zero-crossing occurs.

Thus repeated relations do not change the order in which nodes are output, except that they may increase the number of counter updates and list traversals.

The produced linear order is still a valid topological sort of the underlying directed graph, since every genuine precedence constraint $j \prec k$ is satisfied whenever it occurs, and multiplicity of identical constraints imposes no additional ordering requirement beyond a single constraint.

### Self-referential relation $j \prec j$

Assume the input contains a relation of the form $j \prec j$.

This contributes $1$ to $\text{COUNT}[j]$ during initialization. In the course of execution, node $j$ can only be removed from the queue after $\text{COUNT}[j]$ becomes $0$. However, the only event that decreases $\text{COUNT}[j]$ is the removal of predecessors of $j$ from the queue. The relation $j \prec j$ can only be processed when $j$ itself is removed, since it appears in the successor list of $j$. This creates a circular dependency: $\text{COUNT}[j] > 0$ prevents $j$ from being output, but processing of $j$ is required to reduce $\text{COUNT}[j]$.

Therefore $\text{COUNT}[j]$ can never reach $0$, so $j$ is never inserted into the queue. Algorithm $T$ terminates with at least one node unoutput, and the resulting output is not a topological ordering of the full input relation set.

The presence of $j \prec j$ therefore constitutes a cycle of length $1$, and Algorithm $T$ correctly fails to produce a complete topological sort.

## Conclusion

Repeated relations $j \prec k$ do not affect correctness of Algorithm $T$; they only introduce redundant counting and processing while preserving all required precedence constraints.

A relation of the form $j \prec j$ prevents completion of the algorithm, since it creates a self-cycle that keeps $\text{COUNT}[j]$ positive indefinitely, so no valid topological ordering of all vertices can be produced.

This completes the proof. ∎
