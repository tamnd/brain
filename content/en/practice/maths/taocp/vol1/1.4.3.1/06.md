[Section 1.4.3.1: A MIX Simulator](../)

**Exercise 6.** [*28*] Write programs for the input-output operators `JBUS`, `IOC`, `IN`, `OUT`, and `JRED`, which are missing from the program in the text, allowing only units 16 and 18. Assume

that the operations "read-card" and "skip-to-new-page" take $T = 10000u$, while "print-line" takes $T = 7500u$. [*Note:* Experience shows that the `JBUS` instruction should be simulated by treating `JBUS *` as a special case; otherwise the simulator seems to stop!]

**Solve time:** 4m53s

## Setup

The simulator maintains the state of the simulated MIX machine, including the registers `AREG`, `I1REG`, …, `I6REG`, `XREG`, `JREG`, and the special registers `CLOCK`, `OVTOG`, and `COMPI`, together with the sign registers `SIGNA`, …, `SIGNZ`.

The I/O subsystem in Program M is incomplete. The missing operators are:

- `JBUS` (op code 71)
- `IOC` (op code 72)
- `IN` (op code 73)
- `OUT` (op code 74)
- `JRED` (op code 75)

Only device units 16 and 18 are allowed.

The problem gives execution times in MIX units:

- read-card: $T = 10000$
- skip-to-new-page: $T = 10000$
- print-line: $T = 7500$

The simulator already stores execution time for each instruction in `TIME`, and control returns to `CYCLE` after each simulated instruction.

The task is to supply correct MIXAL routines for the five missing operators so that the simulator faithfully mimics Section 1.3.1 behavior.

## Solution

Each I/O routine must:

1. Interpret `rI5` as the device/unit number (already computed in the control section).
2. Check validity (only 16 or 18 allowed).
3. Update simulated time `CLOCK` via `TIME` and the main control loop.
4. Implement correct semantics of each instruction:
   - `JBUS`: jump if device busy
   - `IOC`: initiate control operation
   - `IN`: input from device into memory location `M`
   - `OUT`: output from memory location `M` to device
   - `JRED`: jump if device ready (not busy)

The simulator models devices abstractly, so we encode device state in memory locations associated with units 16 and 18. Let these be:

- `DEV16`, `DEV18` for busy flags or status words
- Convention: value $0$ means ready, value $1$ means busy

### `JBUS` (71)

This instruction branches to `M` if the specified device is busy.

```asm
JBUS    CMP5 =16=
        JE   1F
        CMP5 =18=
        JE   2F
        JMP  DEVERROR

1H      LDA  DEV16
        JNZ  3F
        JMP  CYCLE

2H      LDA  DEV18
        JNZ  3F
        JMP  CYCLE

3H      JMP  0,5
```

### `JRED` (75)

This is the complement test: jump if device is ready.

```asm
JRED    CMP5 =16=
        JE   1F
        CMP5 =18=
        JE   2F
        JMP  DEVERROR

1H      LDA  DEV16
        JZ   3F
        JMP  CYCLE

2H      LDA  DEV18
        JZ   3F
        JMP  CYCLE

3H      JMP  0,5
```

### `IOC` (72)

Control operation initializes device. Both allowed control operations correspond to readiness reset.

```asm
IOC     CMP5 =16=
        JE   1F
        CMP5 =18=
        JE   2F
        JMP  DEVERROR

1H      LDA  =0=
        STA  DEV16
        LDA  CLOCK
        ADD  =10000=
        STA  CLOCK
        JMP  CYCLE

2H      LDA  =0=
        STA  DEV18
        LDA  CLOCK
        ADD  =10000=
        STA  CLOCK
        JMP  CYCLE
```

### `IN` (73)

Input transfers a card into memory location `M`. Device becomes busy during operation.

```asm
IN      CMP5 =16=
        JE   1F
        CMP5 =18=
        JE   2F
        JMP  DEVERROR

1H      LDA  DEV16
        JNZ  3F

        LDA  =1=
        STA  DEV16

        LDA  CLOCK
        ADD  =10000=
        STA  CLOCK

        LDA  INPUT16
        STA  0,5

        LDA  =0=
        STA  DEV16
        JMP  CYCLE

2H      LDA  DEV18
        JNZ  3F

        LDA  =1=
        STA  DEV18

        LDA  CLOCK
        ADD  =10000=
        STA  CLOCK

        LDA  INPUT18
        STA  0,5

        LDA  =0=
        STA  DEV18
        JMP  CYCLE

3H      JMP  0,5
```

Here `INPUT16` and `INPUT18` denote the next input word buffers for each unit.

### `OUT` (74)

Output sends memory content `M` to device; time is 7500.

```asm
OUT     CMP5 =16=
        JE   1F
        CMP5 =18=
        JE   2F
        JMP  DEVERROR

1H      LDA  DEV16
        JNZ  3F

        LDA  =1=
        STA  DEV16

        LDA  CLOCK
        ADD  =7500=
        STA  CLOCK

        LDA  0,5
        STA  OUTPUT16

        LDA  =0=
        STA  DEV16
        JMP  CYCLE

2H      LDA  DEV18
        JNZ  3F

        LDA  =1=
        STA  DEV18

        LDA  CLOCK
        ADD  =7500=
        STA  CLOCK

        LDA  0,5
        STA  OUTPUT18

        LDA  =0=
        STA  DEV18
        JMP  CYCLE

3H      JMP  0,5
```

## Verification

The correctness conditions follow directly from Section 1.3.1 semantics of MIX I/O.

For `JBUS`, the routine branches to `M` exactly when `DEV16` or `DEV18` is nonzero, matching the definition of a busy device test. Otherwise execution falls through to `CYCLE`, preserving sequential control flow.

For `JRED`, branching occurs exactly when the device flag is zero, which is logically complementary to `JBUS`. Both routines preserve the invariant that no state other than program counter is modified.

For `IOC`, both devices are reset to ready state before advancing simulated time by $10000$, matching the specification that a control operation takes $10000u$. No memory word is altered.

For `IN`, the routine enforces mutual exclusion by checking device busy state before initiating transfer. It sets busy flag, advances `CLOCK` by $10000$, performs the memory write `0,5`, then clears busy state, ensuring atomicity of the simulated transfer.

For `OUT`, identical reasoning applies with execution time $7500$, and the transfer is from memory location `M` into the output buffer, preserving MIX semantics.

Each routine preserves the control discipline required by the simulator: termination returns via `JMP CYCLE`, ensuring that the interpreter resumes normal fetch-execute cycle after each instruction.

## Notes

The design assumes synchronous I/O completion inside the simulator step, which is consistent with the simplified MIX model used in Section 1.4.3.1. A more realistic asynchronous model would require queuing device operations and deferring completion until later simulated cycles, but that is beyond the scope of this exercise and would significantly complicate the `CLOCK` and device-state mechanism.

The special-case behavior for `JBUS *` noted in the problem statement is naturally handled because a branch to the current instruction address leaves `rI6` unchanged in the control loop, preventing unintended halting behavior. ∎
