**Exercise 1.** [*14*] Study all the uses of the `FCHECK` subroutine in the simulator program. Can you suggest a better way to organize the code? (See step 3 in the discussion at the end of Section `1.4.1`.)

Let the field specification byte be denoted by $x = \mathrm{INST}(4:4)$. The subroutine `FCHECK` is invoked in all contexts where a field specification is used, and its effect is always the same: it interprets $x$ as a pair $(L,R)$ satisfying $0 \le L \le R \le 5$, and signals an error if the specification is not valid. The important structural observation is that this interpretation is purely functional on $x$ and does not depend on any other machine state.

The earlier solution attempted to justify a particular arithmetic decomposition of $x$, but that is unnecessary and potentially misleading. The simulator already defines the meaning of a field specification through `FCHECK` itself, so the correct starting point is to treat it as an abstract decoding function

$$
(L,R) \leftarrow \mathrm{decode}(x),
$$

where `decode` is exactly the specification implemented by `FCHECK`, including its validity checks.

In the MIX machine model used by the simulator, a field specification occupies one byte and encodes one of the admissible pairs $(L,R)$ with $0 \le L \le R \le 5$. There are 36 such pairs, and the encoding is fixed by the machine definition. The key point is not the explicit radix representation, but that the mapping between $x$ and $(L,R)$ is total on valid inputs and is checked uniformly by `FCHECK`.

Now consider all uses of `FCHECK`. In the simulator, every operand access that involves a field specification follows the same pattern: the current instruction in `INST` contains $x = \mathrm{INST}(4:4)$, then `FCHECK` is called to produce $(L,R)$, and only then is the result used either to fetch a value from memory (`GETV`) or to extract a register field (`GETAV`). Since the instruction register does not change during the execution of a single cycle, the value $x$ is constant throughout all these calls.

This implies that the computation performed by `FCHECK` is repeated unnecessarily whenever multiple operands are evaluated from the same instruction. The redundancy is not in the decoding logic itself, but in its placement inside low-level operand routines.

A better organization is therefore to move the application of `FCHECK` out of `GETV` and `GETAV` and into the instruction execution cycle at the point where `INST` is first established. In the main cycle routine `CYCLE`, immediately after fetching the instruction into `INST`, the simulator already performs opcode decoding and address extraction. At this same point, the field specification should also be decoded once:

$$
x := \mathrm{INST}(4:4), \quad (L,R) := \mathrm{decode}(x),
$$

and the resulting values should be stored in dedicated parts of the instruction state.

The essential requirement is that these values be preserved for the remainder of the cycle without recomputation. This can be done either by extending the instruction representation with two additional temporary fields holding $L$ and $R$, or by storing them in designated control registers reserved for the current instruction context. What matters is that `GETV` and `GETAV` no longer perform any decoding; they only consume the already validated pair $(L,R)$.

With this change, `GETV` simplifies to applying the field $(L,R)$ to a memory word, and `GETAV` similarly applies it to a register. The validity check that was previously embedded in `FCHECK` is still performed exactly once per instruction, preserving correctness, but eliminating repeated evaluation.

This restructuring preserves semantics because every call to `FCHECK` in the original program is a pure function of the current instruction’s field byte and has no side effects beyond signaling invalid input. Since the instruction does not change during a cycle, all repeated calls would necessarily produce the same result. Hoisting the computation to the cycle boundary therefore does not change observable behavior, provided the validity check is retained.

The improved organization also clarifies the logical separation of concerns in the simulator. Instruction decoding, including field specification interpretation, becomes part of the central control phase, while operand routines are reduced to pure data access operations parameterized by predecoded fields. This matches the intended structure of a two-level interpretation model: decode once, execute many micro-operations on the decoded representation.

Thus the recommended redesign is to treat `FCHECK` as part of instruction decoding rather than as a repeatedly invoked operand-level subroutine, and to eliminate all calls to it from `GETV` and `GETAV` by storing its result in the instruction context established in `CYCLE`.
