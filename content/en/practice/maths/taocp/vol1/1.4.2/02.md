**Exercise 2.** [*20*] The program in the text starts up the `OUT` coroutine first. What would happen if `IN` were the first to be executed, that is, if line 60 were changed from `JMP OUT1` to `JMP IN1`?

**Verified:** yes  
**Solve time:** 2m26s  

---

The proposed failure analysis is incorrect because it assumes a missing or premature dependency in the initialization of `INX`. In fact, the coroutine linkage is self-synchronizing: each link variable is initialized before it is ever used.

## Correct analysis

We recall the structure:

- `OUTX = JMP OUT1`, `INX = JMP IN1` initially.
- In coroutine `OUT`:

$$
\texttt{OUT: STJ INX}
$$

meaning: save the return address into `INX`.
- In coroutine `IN`:

$$
\texttt{IN: STJ OUTX}
$$

meaning: save the return address into `OUTX`.

Each coroutine terminates (or suspends) by jumping through the other coroutine’s saved transfer address.

## Case 1: Standard order (OUT starts first)

When execution begins at `OUT1`, the register `J` contains the return address into the initialization sequence. The first action is:

$$
\texttt{STJ INX}
$$

so `INX` is initialized **before IN ever attempts to return to OUT**. Then `OUT` transfers control to `IN`, and the system alternates correctly.

## Case 2: Modified order (IN starts first)

Now suppose line 60 is changed so execution begins at `IN1`.

### Step 1: First execution of `IN1`

`IN` executes:

$$
\texttt{STJ OUTX}
$$

This stores the current return address (the continuation of the startup sequence) into `OUTX`. This is not harmful; it simply makes the first return from `OUT` go back into the correct place in `IN`.

Then `IN` transfers control to `OUT1` via:

$$
\texttt{JMP OUTX}
$$

At this moment, `OUTX` has already been correctly initialized.

### Step 2: First execution of `OUT1`

Now `OUT` runs and immediately executes:

$$
\texttt{STJ INX}
$$

This is the crucial point: **this is the first time `INX` is written**, and it occurs before any attempt by `OUT` to return to `IN`.

So `INX` is properly initialized exactly when it becomes relevant.

### Step 3: Return from `OUT` to `IN`

When `OUT` suspends, it transfers control using:

$$
\texttt{JMP INX}
$$

At this moment, `INX` contains the correct return address (set during this first execution of `OUT1`). Therefore control returns correctly to `IN`.

## Structural conclusion

Nothing in the coroutine mechanism depends on which coroutine is started first. What matters is only that:

- `STJ OUTX` is executed before `IN` first returns to `IN`
- `STJ INX` is executed before `OUT` first returns to `OUT`

Both conditions are satisfied in either starting order because:

- whichever coroutine starts first initializes the other coroutine’s link variable before that variable is used for the first time.

Thus the linkage equations are not order-dependent in a way that breaks correctness.

## Final conclusion

Changing line 60 so that execution begins at `IN1` does **not** break the coroutine system.

The effect is only to reverse the initial direction of control transfer:

- Instead of starting with an `OUT → IN` exchange,
- the program begins with an `IN → OUT → IN` exchange,

after which the same alternating coroutine behavior proceeds normally.

$$
\boxed{\text{The program still functions correctly; only the initial ordering of the first exchange changes.}}
$$
