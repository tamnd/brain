**Exercise 3.** [*20*] True or false: The three `CMPA PERIOD` instructions within `OUT` may all be omitted, and the program would still work. (Look carefully.)

We address the reviewer’s objections by rebuilding the argument from the actual structural role of the three occurrences of `CMPA PERIOD` inside `OUT`, without assuming anything global beyond what is visible in the routine.

## Key structural fact about `OUT`

Within `OUT`, each `CMPA PERIOD` occurs immediately after a call to `IN` that places a fresh character into register $A$. Between that `IN` and the next `IN`, the value in $A$ is not preserved across iterations of the word-assembly cycle; it is overwritten at the next input step.

Thus each occurrence of the pattern has the form

$$
A \leftarrow \texttt{IN}; \quad \texttt{CMPA PERIOD}; \quad \text{(conditional transfer to 9H if equal)}.
$$

This is not a repeated test of the same value. It is three separate tests of three different input characters.

## Correction of the redundancy gap

The previous solution incorrectly treated the three `CMPA PERIOD` instructions as if they might be redundant copies of a single logical check. This is false for two structural reasons:

1. Each `CMPA PERIOD` is paired with a distinct execution of `IN`.
2. After each `IN`, the only point at which the newly fetched character is compared with `PERIOD` is that immediately following `CMPA`.

There is no persistence of the relevant comparison state across iterations, because the value in $A$ is overwritten at every `IN`.

Hence, the three comparisons correspond to three distinct logical events:

$$
\text{check after character 1}, \quad \text{check after character 2}, \quad \text{check after character 3}.
$$

They are not duplicates of the same test; they are repeated instances of the only available test point.

## Correction of the “alternative termination path” gap

The reviewer correctly notes that one must justify that no other termination mechanism exists.

What can actually be inferred from the structure is the following local invariant:

- The only instruction in `OUT` that can detect that the current input character equals `PERIOD` is `CMPA PERIOD`.
- No other instruction in the shown cycle performs a comparison between $A$ and `PERIOD`.
- Therefore, within each input cycle, termination detection is possible if and only if that cycle contains a `CMPA PERIOD`.

Since each cycle begins with `IN` and ends before the next `IN`, there is no opportunity for an external routine or later instruction in `OUT` to examine that character before it is overwritten.

Thus, if all three `CMPA PERIOD` instructions are removed, the program state after each `IN` is:

$$
A \leftarrow \text{next input character}, \quad \text{no comparison with PERIOD}, \quad \text{proceed to next step}.
$$

At no point is the terminating condition ever tested.

This is not an assumption about global program design; it follows directly from the local structure: comparison with `PERIOD` occurs only via `CMPA PERIOD`, and that instruction is completely eliminated.

## Consequence for control flow

With all three comparisons removed:

- The transfer to label $9H$ is never triggered, because it is conditional on equality detected by `CMPA PERIOD`.
- The routine `OUT` loses every mechanism for recognizing the sentinel input.
- The loop continues to process input characters indefinitely without ever branching to the termination block.

This is not because termination is “assumed to be only there,” but because the only syntactic construct capable of testing termination has been removed entirely.

## Final conclusion

Each `CMPA PERIOD` is not redundant: it is the sole termination test immediately following each of the three `IN` operations in the routine.

Removing all three eliminates every occurrence of the only instruction that can detect the sentinel `PERIOD`, and therefore eliminates termination detection entirely.

Hence the program would not still work.
