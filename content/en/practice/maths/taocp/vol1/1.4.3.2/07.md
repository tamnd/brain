[Section 1.4.3.2: Trace Routines](../)

**Exercise 7.** [*25*] Discuss how to write an efficient *jump trace routine*, which emits much less output than a normal trace. Instead of displaying the register contents, a jump trace simply records the jumps that occur. It outputs a sequence of pairs $(x_1, y_1)$, $(x_2, y_2)$, ... , meaning that the program jumped from location $x_1$ to $y_1$, then (after performing the instructions in locations $y_1, y_1 + 1, ... , x_2$) it jumped from $x_2$ to $y_2$, etc. [From this information it is possible for a subsequent routine to reconstruct the flow of the program and to deduce how frequently each instruction was performed.]

**Solve time:** 2m13s

The correct construction is event-based: one output is produced exactly when a jump instruction is executed, and the output records that instruction’s location and its destination.

### Correct jump trace routine

Let the trace routine observe execution at each step. Let $x$ denote the current instruction location, given by the program counter (in MIX notation, $x = \text{PREG}$) before the instruction is executed. Let the instruction word at location $x$ be available in the trace system register (as in the standard MIX trace routine), and let its effective address be computable by the same mechanism used for normal execution.

The jump trace outputs a sequence of pairs $(x_k, y_k)$, where $x_k$ is the location of a jump instruction and $y_k$ is its transfer address.

### Trace behavior

At each instruction execution step:

1. The current instruction location is

$$
x \leftarrow \text{PREG}.
$$

1. The instruction at $x$ is decoded as in the standard trace routine.
2. If the instruction is **not a jump**, then no output is produced, and execution proceeds normally.
3. If the instruction is a **jump instruction**, then:

- Compute its effective address $y$, using the same effective-address computation already present in the MIX execution/trace routine (the same mechanism that determines the target of the jump during execution).
- Output the pair

$$
(x, y).
$$
- Execute the jump in the usual way, so that the program counter is updated to $y$.

### Correctness of the recorded data

Each emitted pair corresponds exactly to one executed jump instruction. The first component is the actual location $x$ of that instruction, since the program counter is sampled before execution changes it. The second component is the effective target $y$, which is precisely the location to which control is transferred when the jump is executed.

Thus the output sequence lists all control transfers in execution order. Between two successive jumps, execution proceeds through non-jump instructions exactly as in the underlying program, so the reconstruction property stated in the problem follows directly.

### Termination case

If the program halts without executing a jump, no additional output is produced. The trace is therefore finite and consists only of those jump events that actually occur during execution. This is consistent with the specification, since the reconstruction of instruction frequencies and control flow does not require a final artificial pair at halt.

### Efficiency

The routine performs constant additional work per instruction: a jump-class test and, only when a jump occurs, one effective-address computation and one output operation. Since output is generated only on control transfers rather than at every instruction, the total volume of trace output is reduced substantially compared to a full register trace.

This yields a correct and efficient jump trace routine that records exactly the sequence of executed jumps in the required $(x_k, y_k)$ form. ∎
