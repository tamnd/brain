**Exercise 18.** [*30*] A proposed extension of MIX allows its computations to be interrupted, as explained below. Your task in this exercise is to modify Algorithms and Programs A, R, and B of the text so that they use these interrupt facilities instead of the `JRED` instructions.

The new MIX features include an additional 3999 memory cells, locations $-3999$ through $-0001$. The machine has two internal "states," *normal state* and *control state*. In normal state, locations $-3999$ through $-0001$ are not admissible memory locations and the MIX computer behaves as usual. When an "interrupt" occurs, due to conditions explained later, locations $-0009$ through $-0001$ are set equal to the contents of MIX's registers: rA in $-0009$; rI1 through rI6 in $-0008$ through $-0003$; rX in $-0002$; and rJ, the overflow toggle, the comparison indicator, and the location of the next instruction are stored in $-0001$ as

```text
+ | next inst. | OV, CI | rJ ;
```

the machine enters control state, at a location depending on the type of interrupt.

Location $-0010$ acts as a "clock": Every $1000u$ of time, the number appearing in this location is decreased by one, and if the result is zero an interrupt to location $-0011$ occurs.

The new MIX instruction `INT` $(C = 5, F = 9)$ works as follows: (a) In normal state, an interrupt occurs to location $-0012$. (Thus a programmer may force an interrupt, to communicate with a control routine; the address of `INT` has no effect, although the control routine may use it for information to distinguish between types of interrupt.) (b) In control state, all MIX registers are loaded from locations $-0009$ to $-0001$, the computer goes into normal state, and it resumes execution. The execution time for `INT` is $2u$ in each case.

An `IN`, `OUT`, or `IOC` instruction given in *control* state will cause an interrupt to occur as soon as the I/O operation is completed. The interrupt goes to location $-(0020 + \text{unit number})$.

No interrupts occur while in control state; any interrupt conditions are "saved" until after the next `INT` operation, and interrupt will occur after one instruction of the normal state program has been performed.

### Solution

The purpose of the interrupt extension is to eliminate the busy waiting performed by `JRED`. The important observation is that `JRED` is used in Programs A, R, and B only to delay execution until a previously requested I/O operation has completed. Under the new architecture, completion of an I/O operation is reported by an interrupt to location

$$
-(20+u),
$$

where $u$ is the unit number. Hence the control flow must be reorganized so that every point that was formerly reached after a successful `JRED` is instead entered from the corresponding interrupt vector.

A second point must also be observed. Interrupt handlers execute in **control state**. Arithmetic, transfer, and memory-reference instructions behave normally in control state, but `IN`, `OUT`, and `IOC` have modified semantics: they may be issued normally, and their completion generates another interrupt; however, no interrupt is taken until the handler executes `INT`, because interrupts are inhibited during control state. Therefore it is legitimate for a handler to initiate the next transfer, provided that the program does not expect its completion before returning to normal state. This is exactly the usage required in Programs A, R, and B.

Finally, although the interrupt mechanism permits a handler to alter the saved register images in locations $-0009$ through $-0001$, none of the modifications below requires this. The handlers leave those locations unchanged, so `INT` restores precisely the interrupted computation.

## Algorithm A

Algorithm A performs one transfer at a time.

Originally the sequence was

```
JRED U
IN (or OUT)
...
```

The `JRED` merely waited until the device became idle before starting the transfer.

With interrupts, the algorithm becomes:

1. Whenever a transfer is required, issue the corresponding `IN` or `OUT` instruction immediately.
2. Continue the independent computation.
3. When the transfer finishes, control enters location

$$
-(20+U).
$$
4. The interrupt routine performs the bookkeeping that originally followed completion of the transfer and returns by executing `INT`.

The only initialization change is that the first transfer must be started explicitly, because there is no initial `JRED` to trigger it.

### Program A

Every occurrence of

```
JRED U
IN ...
```

is replaced by

```
IN ...
```

and every occurrence of

```
JRED U
OUT ...
```

is replaced by

```
OUT ...
```

The interrupt vector for unit $U$ contains the continuation that was formerly executed after the transfer had completed. Schematically,

```
-(20+U)

    completion bookkeeping
    INT
```

The handler does **not** execute code that assumes normal-state execution except for instructions whose meanings are unchanged in control state. If another transfer is to be started immediately, its `IN` or `OUT` instruction may be issued here, since completion of that transfer merely queues another interrupt until after the return to normal state.

## Algorithm R

Algorithm R overlaps computation with input.

Originally the sequence was

```
JRED input
process newly read record
IN next record
```

where `JRED` waited until the previous read had completed.

The interrupt version is organized as follows.

1. Before entering the main processing loop, start the first input operation.
2. Continue processing the current record.
3. When the input completes, an interrupt occurs at

$$
-(20+\text{input unit}).
$$
4. The interrupt routine:

- makes the newly read buffer available;
- updates the buffer pointers exactly as the original program did after the successful `JRED`;
- immediately issues the `IN` instruction for the following record;
- executes `INT`.

The new `IN` is executed in control state. This is valid because its only effect is to begin another asynchronous transfer. Its completion interrupt is postponed until after the handler returns, exactly as required by the interrupt specification.

### Program R

Delete every `JRED` instruction.

Insert an initialization step that starts the first input transfer before entering the processing loop.

Install the interrupt handler

```
-(20+input unit)

    make completed buffer current
    update pointers
    IN next buffer
    INT
```

The processing loop now operates without polling. Every completed input automatically prepares the next record and starts another read.

## Algorithm B

Algorithm B overlaps both input and output.

Originally there were two waiting points,

```
JRED input
```

and

```
JRED output
```

Each merely synchronized the program with completion of a previously initiated transfer.

The interrupt version uses one handler for each device.

### Input interrupt

When the input transfer completes:

1. make the completed input buffer available;
2. advance the input-buffer pointers;
3. if another input is required, issue its `IN`;
4. execute `INT`.

### Output interrupt

When the output transfer completes:

1. release the completed output buffer;
2. advance the output queue;
3. if another completed buffer is waiting, issue its `OUT`;
4. execute `INT`.

As in Algorithm R, the `IN` and `OUT` instructions are executed in control state. Their completion interrupts are deferred until after the return to normal state, so no nested interrupt occurs.

### Program B

Remove every `JRED`.

Before entering the principal computation, explicitly start the initial input operation (and any initial output operation required by the original initialization).

Install two interrupt handlers.

For the input unit,

```
-(20+input unit)

    make input buffer available
    update input pointers
    IN next buffer        (if required)
    INT
```

For the output unit,

```
-(20+output unit)

    release completed output buffer
    update output queue
    OUT next completed buffer   (if any)
    INT
```

The main computation no longer waits for either device. It proceeds until interrupted by completion of an input or output transfer.

## Correctness

We prove that the modified programs preserve the behavior of the originals.

Each `JRED` in the original programs waits for completion of one previously issued I/O operation. Under the extended machine, completion of that same operation generates exactly one interrupt to location

$$
-(20+u),
$$

where $u$ is the device number. Thus every successful `JRED` has a unique corresponding interrupt event.

The original continuations after `JRED` consist of bookkeeping, pointer updates, and in Algorithms R and B the initiation of the next asynchronous transfer. These operations are valid in control state. In particular, issuing `IN`, `OUT`, or `IOC` from control state is explicitly permitted by the machine specification. The only difference is that completion of the newly started transfer causes a later interrupt rather than immediate nested execution. Since interrupts are inhibited until `INT` returns to normal state, this postponement cannot change the logical ordering required by the algorithms.

Initialization is preserved by explicitly issuing the first transfer before entering the main computation. This replaces the original pattern in which execution first reached a `JRED` waiting for completion of an already initiated operation.

The handlers never modify locations $-0009$ through $-0001$. Consequently, `INT` restores exactly the register contents, overflow toggle, comparison indicator, return address, and next instruction saved when the interrupt occurred. The interrupted computation therefore resumes exactly where it left off.

Hence:

1. every transfer completion that formerly caused one successful `JRED` now causes one interrupt;
2. each interrupt executes the same bookkeeping and scheduling actions that formerly followed the successful `JRED`;
3. every newly started transfer is initiated at the same logical point as in the original programs;
4. the interrupted computation resumes with identical machine state after `INT`.

Therefore Algorithms A, R, and B perform exactly the same sequence of computations and I/O operations as the original versions, while eliminating all busy waiting by replacing polling with interrupt-driven synchronization.
