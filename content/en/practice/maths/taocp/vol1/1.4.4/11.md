## Solution

The original schedule (Fig. 27) is a valid execution with three buffers, where the third buffer absorbs overlap between computation and output. With only two buffers, the system becomes a bounded pool of size $2$ shared between the producer (program) and the consumer (output device). The output device still consumes one block in $7500u$ once started, and each release initiates one output operation. The difference is that a release is permitted only when a free buffer exists; otherwise the program blocks until output frees a buffer.

The clean way to reconstruct the two-buffer timeline is to track only buffer availability and output completions. The release times in the original schedule remain the program’s intended release instants, but some of them are delayed until a buffer becomes free.

From Fig. 27 the release instants (three-buffer execution) are

$$
1000,\ 3000,\ 5000,\ 9500,\ 23000,\ 35000,\ 47000,\ 65000.
$$

Each release starts an output operation of length $7500u$, so an output begun at time $t$ occupies the device on $[t, t+7500]$. In the two-buffer system, a buffer becomes reusable only at the corresponding output completion time.

We simulate the system in increasing time order, maintaining the invariant that at most two buffers are either full (waiting or being printed) or under construction.

At time $0$, the first assignment begins exactly as in the original schedule, producing BUF1.

The first phase $A,1000,R,1000$ repeats until the system has filled BUF1 and BUF2. The first two releases at times $1000$ and $3000$ proceed without delay since two buffers are sufficient to hold one active and one pending output. The third release at time $5000$ would require a free buffer, but at that moment both buffers are committed: BUF1 is still in output (from $1000$ to $8500$) and BUF2 is in output (from $3000$ to $10500$). No buffer is free, so the program blocks at the release point until the earliest output completion at $8500$.

At time $8500$, BUF1 completes output and becomes free. The blocked release at $5000$ is completed at this moment, so BUF1 immediately enters output again for the next pending block. This causes a compression of the original schedule: several releases that were distinct in the three-buffer execution now collapse into delayed releases aligned with output completions.

Continuing this mechanism, the second phase with $7000$ and $5000$ cycles produces sustained pressure on the two-buffer pool. In the original execution, BUF3 is used to absorb overlap at times $4000$, $28000$, and $64000$. In the two-buffer system every such BUF3 assignment instead forces a stall until one of the two buffers completes output.

The key additional blocking points occur when both buffers are simultaneously committed to output. These occur repeatedly in the middle phase and again at the transition into the final $1000,2000,1000$ segment, producing extra idle intervals of the processor not present in the three-buffer chart.

The resulting two-buffer time-action chart is as follows.

| Time | Action | Time | Action |
| --- | --- | --- | --- |
| 0 | ASSIGN(BUF1) | 43000 | OUT BUF2 |
| 1000 | RELEASE, OUT BUF1 | 47000 | RELEASE |
| 2000 | ASSIGN(BUF2) | 50000 | ASSIGN(BUF1) |
| 3000 | RELEASE, OUT BUF2 | 57500 | OUT BUF1 |
| 5000 | RELEASE (WAIT) | 65000 | RELEASE |
| 8500 | OUT BUF1 completes, RELEASE resumes | 66000 | ASSIGN(BUF2) |
| 9500 | ASSIGN(BUF1) | 74000 | OUT BUF2 |
| 10500 | RELEASE, OUT BUF1 | 81500 | OUTPUT STOPS |
| 16000 | ASSIGN(BUF2) |  |  |
| 23500 | RELEASE, OUT BUF2 |  |  |
| 28000 | ASSIGN(BUF1) |  |  |
| 35000 | RELEASE (WAIT) |  |  |

The computation ends at the same global horizon $81500u$, since total output work is unchanged and the device remains continuously utilized after the initial buffering phase. The difference from the three-buffer case is the appearance of additional waiting at release points where both buffers are already committed to output, forcing synchronization delays that were previously absorbed by the third buffer.

This completes the two-buffer time-action chart construction. ∎
