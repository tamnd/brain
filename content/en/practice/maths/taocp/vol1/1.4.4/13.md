**Exercise 13.** [*20*] What instructions should be included at the end of the COMPUTE coroutine in the text, if the buffering algorithms are being applied to output, to ensure that all information has been output from the buffers?

The key point in Knuth’s buffered coroutine design is that termination is expressed purely through the **buffer–handoff protocol**, not through any external flag or global state. Therefore, the end of `COMPUTE` must ensure only two things: the last data are placed into a buffer in the correct format, and `CONTROL` receives a final buffer that it can recognize as the end of the stream.

## Correct termination of `COMPUTE`

At the end of the `COMPUTE` coroutine, the following instructions are required.

### 1. Flush the partially filled buffer

If the current output buffer contains $k>0$ words, then before termination it must be completed in the same format expected by `CONTROL`.

This means:

- The $k$ valid words already in the buffer remain unchanged.
- The remaining positions are filled according to the buffering convention used in the text (typically irrelevant values beyond the indicated length, since only the length matters).
- The buffer descriptor is set so that `CONTROL` recognizes that exactly $k$ words are valid in this buffer.

Then this buffer is handed to `CONTROL` exactly as in the normal full-buffer case.

No special termination logic is used here, only the standard buffer transfer step.

### 2. Transmit a final empty buffer as end-of-data signal

After the last real buffer has been transmitted, `COMPUTE` must perform one additional buffer transfer to indicate termination.

In Knuth’s coroutine structure, this is done by sending a buffer whose valid length is zero:

$$
k = 0
$$

That is, `COMPUTE` invokes the same buffer-passing mechanism used during normal execution, but with a buffer descriptor indicating that no data words are present.

`CONTROL` interprets this empty buffer as the end-of-input condition and stops requesting further buffers.

No separate flag or global variable is involved.

### 3. Final transfer of control

After sending the empty buffer, `COMPUTE` executes its final coroutine transfer to `CONTROL` and does not resume.

In MIX coroutine form, this is simply the final:

- transfer to `CONTROL`
- without any expectation of return

This ensures that all pending buffers have been handed off and that execution continues permanently in `CONTROL`, which completes the draining of any remaining device activity.

## Why this is sufficient

- Every partially filled buffer is explicitly flushed and handed off under the normal protocol.
- The zero-length buffer provides an unambiguous structural end marker within the same communication mechanism.
- No auxiliary termination variables are required, consistent with Knuth’s design principle that coroutine communication is entirely data-structure based.
- The final transfer ensures `COMPUTE` does not re-enter the scheduling cycle.

## Final instruction sequence in `COMPUTE`

1. If the current buffer is partially filled, complete it and pass it to `CONTROL` in the standard way.
2. Invoke one additional buffer transfer with $k = 0$ (empty buffer) to signal end of data.
3. Transfer control permanently to `CONTROL` and terminate `COMPUTE`.
