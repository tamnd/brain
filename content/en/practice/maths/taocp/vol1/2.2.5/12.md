[Section 2.2.5: Doubly Linked Lists](../)

**Exercise 12.** [*22*] Why is it a good idea to use doubly linked lists instead of singly linked or sequential lists in the simulation program of this section?

**Verified:** yes  
**Solve time:** 55s  

---

## Solution

The simulation program maintains two principal dynamic structures, `QUEUE[IN]` and `ELEVATOR`, in which individual users are inserted and later removed according to events generated by the coroutines. In particular, step `U4` requires deletion of a user from `QUEUE[IN]` when the user abandons the system, and step `U5` requires insertion into the elevator list followed later by deletion at step `U6`.

In step `U4`, the algorithm acts on a specific user node `X` already identified in `QUEUE[IN]`. The operation required is deletion of `NODE(X)` from the middle of a doubly linked list. Formula (4) performs this using only local link adjustments:

$$
\text{RLINK}(\text{LLINK}(X)) \leftarrow \text{RLINK}(X),\qquad
\text{LLINK}(\text{RLINK}(X)) \leftarrow \text{LLINK}(X),\qquad
\text{AVAIL} \Leftarrow X.
$$

Each assignment changes exactly one pointer field, and no search is required because both neighbors of `X` are directly accessible through `LLINK(X)` and `RLINK(X)`.

In a singly linked representation, deletion of `NODE(X)` cannot be performed from `X` alone, since the pointer to the predecessor `P` satisfying `RLINK(P)=X` is not stored in `X`. Determining `P` requires traversal of the list starting from its head and following `RLINK` pointers until reaching the node whose right link equals `X`. This traversal is repeated for each deletion that occurs at step `U4`. Since each user may abandon the system independently of position in the queue, such deletions occur at arbitrary locations, and each one forces a full or partial scan of `QUEUE[IN]`.

Insertion in step `U5` and insertion of new users in step `U3` also require local pointer updates. Formula (5) inserts a node adjacent to `NODE(X)` by changing five links:

$$
P \Leftarrow \text{AVAIL},\qquad
\text{LLINK}(P) \leftarrow X,\qquad
\text{RLINK}(P) \leftarrow \text{RLINK}(X),\qquad
\text{LLINK}(\text{RLINK}(X)) \leftarrow P,\qquad
\text{RLINK}(X) \leftarrow P.
$$

These updates depend only on `X` and its immediate neighbors, so insertion is performed without traversal. A singly linked list cannot perform the symmetric insertion to the left of a node without first locating its predecessor, producing the same traversal problem as in deletion.

A sequential representation of `QUEUE[IN]` would require shifting elements to fill the gap created by deletion and to make space for insertion. Each such shift modifies a sequence of consecutive memory locations, and the number of moved entries depends on the position of the affected element within the queue. Since users may enter and leave at arbitrary positions in the queue, repeated shifting introduces non-local work that depends on queue length.

The simulation repeatedly performs operations of the form “insert at rear,” “delete arbitrary waiting user,” and “move user between states.” In each case, the program already manipulates explicit node references produced by event scheduling. A doubly linked list preserves direct access to both neighbors of any node, so that deletion and adjacent insertion are performed entirely through local pointer updates as in (4) and (5), without search or shifting.
