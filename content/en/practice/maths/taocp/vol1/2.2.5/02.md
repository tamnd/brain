[Section 2.2.5: Doubly Linked Lists](../)

**Exercise 2.** [*22*] Explain why a list that is singly linked cannot allow efficient operation as a general deque; the deletion of items can be done efficiently at only one end of a singly linked list.

**Verified:** yes  
**Solve time:** 2m01s  

---

A deque requires efficient insertion and deletion at both ends. Consider a singly linked list in which each node contains a pointer only to its successor.

Insertion at the front is efficient. A new node can be created and its next pointer set to the current head, then the head pointer updated. This takes constant time.

Deletion at the front is also efficient. The head pointer can be advanced to the second node in constant time.

The difficulty arises at the other end of the list. To delete the last element, one must locate the predecessor of the last node, since there is no backward pointer. In a singly linked list this predecessor cannot be accessed directly. The list must be traversed from the head until the node whose next pointer points to the last node is found. This traversal takes linear time in the length of the list.

Thus deletion at the rear is not efficient. It requires $\Theta(n)$ time in a list of $n$ elements.

Insertion at the rear can be made efficient only if an additional pointer to the last node is maintained. However, even with such a pointer, deletion at the rear still requires finding the predecessor, which again requires traversal from the head.

Therefore a singly linked list supports efficient operations at only one end: the front. A general deque requires efficient insertion and deletion at both ends, so a singly linked list cannot implement a deque efficiently without additional structure such as a doubly linked list.
