**Exercise 7.** [*20*] Why is self-modifying code now frowned on?

**Verified:** yes  
**Solve time:** 1m43s

## Solution

Self-modifying code is discouraged because modern computer architectures and software systems separate the treatment of instructions and data, and this separation is essential for correctness, performance, and security.

In machines like MIX, instructions reside in writable memory and may be altered as part of subroutine linkage or parameter handling. Modern processors instead use instruction pipelines and instruction caches that assume code is stable during execution. When a program modifies an instruction in memory, the corresponding cached instruction may remain in the instruction cache, so the processor can execute a stale version of the code unless explicit cache invalidation occurs. This introduces architecture-dependent behavior and undermines predictable execution.

Operating systems also enforce memory protection policies in which executable pages are marked read-only for writing and writable pages are non-executable. This write-execute separation prevents self-modifying code from functioning in normal user programs because modifying code requires changing page permissions, which is costly and tightly controlled. The restriction supports process isolation and prevents accidental corruption of executable code.

Self-modifying code also interferes with program analysis. Compilers, optimizers, and verification tools rely on the property that instructions are fixed so that control flow and data flow can be analyzed statically. When instructions change during execution, the effective program cannot be determined from the text alone, which prevents reliable optimization and complicates debugging and reasoning about correctness.

For these reasons, modern systems replace self-modification with explicit mechanisms such as registers, stacks, and link registers for return addresses, and parameter passing conventions that keep program text invariant. This yields more portable, efficient, and secure execution models than instruction modification in memory.
