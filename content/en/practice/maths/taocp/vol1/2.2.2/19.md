**Exercise 19.** [*16*] (*0-origin indexing.*) Experienced programmers learn that it is generally wise to denote the elements of a linear list by $X[0], X[1], \ldots, X[n-1]$, instead of using the more traditional notation $X[1], X[2], \ldots, X[n]$. Then, for example, the base address $L_0$ in (1) points to the smallest cell of the array.

Revise the insertion and deletion methods (2a), (3a), (6a), and (7a) for stacks and queues so that they conform to this convention. In other words, change them so that the list elements will appear in the array $X[0], X[1], \ldots, X[M-1]$, instead of $X[1], X[2], \ldots, X[M]$.

The stack part of the proposed solution is already correct under 0-origin indexing. The only substantive error lies in the queue model, where the roles of $F$ and $R$ are inconsistently defined. The correction requires restarting the queue representation so that it matches a consistent circular-array invariant.

# Corrected Solution

We use a linear array

$$
X[0], X[1], \ldots, X[M-1]
$$

for both stacks and queues.

All indexing is 0-origin.

# Stack (unchanged in substance)

Let $T$ denote the number of elements in the stack. The stack occupies

$$
X[0], X[1], \ldots, X[T-1]
$$

when $T > 0$. The empty stack has $T = 0$, and $0 \le T \le M$.

## (2a) Insertion into stack

$$
X \Leftarrow Y:
\begin{cases}
\text{if } T = M, \text{ then OVERFLOW;} \\
X[T] \leftarrow Y; \\
T \leftarrow T + 1.
\end{cases}
$$

## (3a) Deletion from stack

$$
Y \Leftarrow X:
\begin{cases}
\text{if } T = 0, \text{ then UNDERFLOW;} \\
T \leftarrow T - 1; \\
Y \leftarrow X[T].
\end{cases}
$$

# Queue (correct circular-array formulation)

The error in the previous solution is the interpretation of $F$ and $R$. To obtain a consistent TAOCP-style circular queue, we use the standard invariant:

- $F$ points to the position **just before the first element**
- $R$ points to the position of the **last element**
- the queue elements are:

$$
X[(F+1)\bmod M], X[(F+2)\bmod M], \ldots, X[R]
$$

(wrapping modulo $M$)

This convention ensures:

- empty queue: $F = R$
- all indices remain in $0, \ldots, M-1$
- exactly one representation for each queue state

Thus the queue is empty iff:

$$
F = R
$$

and full iff advancing $R$ would meet $F$:

$$
(R + 1) \bmod M = F
$$

# (6a) Insertion into queue

$$
X \Leftarrow Y:
\begin{cases}
\text{if } (R + 1)\bmod M = F, \text{ then OVERFLOW;} \\
R \leftarrow (R + 1)\bmod M; \\
X[R] \leftarrow Y.
\end{cases}
$$

Explanation: $R$ is advanced first to create a new last position, then the element is stored.

# (7a) Deletion from queue

$$
Y \Leftarrow X:
\begin{cases}
\text{if } F = R, \text{ then UNDERFLOW;} \\
F \leftarrow (F + 1)\bmod M; \\
Y \leftarrow X[F].
\end{cases}
$$

Explanation: advancing $F$ moves it from “before the first element” to the first element itself, which is then removed.

# Consistency check

Under this corrected invariant:

- empty: no elements between $F$ and $R$
- first element is $X[(F+1)\bmod M]$
- last element is $X[R]$
- insertion and deletion preserve all invariants
- overflow is detected before modifying $R$

# Conclusion

- Stack revision: correct as given.
- Queue revision: corrected by restoring a consistent TAOCP circular-array invariant where $F$ precedes the first element and $R$ is the last element.

This removes the contradiction between empty-state handling and pointer interpretation, and restores full consistency with Knuth’s circular queue method.
