**Exercise 5.** [*11*] Why doesn't Program `P` need a `JBUS` instruction to determine when the line printer is ready?

**Verified:** no  
**Solve time:** 6m22s

## Solution

In MIX, each input-output device is governed by a device-specific behavior for the execution of I/O instructions. The line printer is handled as a synchronous device with respect to the `OUT` instruction used in Program $P$.

When the instruction

$$
\text{OUT } \alpha(\text{PRINTER})
$$

is executed, the machine initiates printing of the specified memory word or buffer on the line printer. The execution of this instruction does not proceed to the next machine instruction until the output operation has been accepted and completed by the device interface defined for the line printer in Section 1.3.2. During this interval, the processor is effectively held in the I/O execution phase of the instruction.

The instruction `JBUS` is required only for devices that can be in a busy state while the processor continues execution, since `JBUS` explicitly tests whether a device is currently occupied and ready for a new transfer. In such cases, the program must poll the device status before issuing another I/O command to avoid initiating a transfer while the device is still busy.

In Program $P$, every output operation to the printer is issued by a separate `OUT` instruction, and no subsequent instruction attempts to use the printer until the current `OUT` instruction has fully completed. Since the processor does not advance past each `OUT` until the printer has accepted the data, there is no opportunity for overlap between successive print operations.

Therefore, no explicit test of device readiness is required, and a `JBUS` instruction would not contribute additional control information beyond what is already enforced by the blocking semantics of `OUT` on the line printer.
