[Section 1.3.1: Description of MIX](../)

**Exercise 25.** [*30*] Suppose that the manufacturer of MIX wishes to come out with a more powerful computer ("Mixmaster"?), and he wants to convince as many as possible of those people now owning a MIX computer to invest in the more expensive machine. He wants to design this new hardware to be an *extension* of MIX, in the sense that all programs correctly written for MIX will work on the new machines without change. Suggest desirable things that could be incorporated in this extension. (For example, can you make better use of the I-field of an instruction?)

**Solve time:** 1m26s

## Setup

A proposed extension of MIX is sought, subject to the requirement that every program correctly written for MIX continue to operate without change. The new machine must therefore preserve the semantics of all instructions, registers, field specifications, and memory addresses used by ordinary MIX programs. Any additional facilities must appear only as enlargements of capabilities already present.

The problem asks for desirable extensions. Each proposal must be compatible with the conventions of Section 1.3.1.

## Solution

The first desirable extension is enlargement of memory. Since correctly written MIX programs use addresses between $0$ and $3999$, all such locations should retain their meanings. Additional memory locations may be provided beyond $3999$, together with new instructions capable of accessing them. Existing instructions would continue to function exactly as before.

The second desirable extension is enlargement of the index registers. In MIX, each index register contains only two bytes and a sign. A new machine may allow more bytes in the index registers while preserving the old two-byte interpretation for values representable in two bytes. Programs using ordinary indexing would then be unaffected, while larger arrays and larger address computations become possible.

The I-field of an instruction contains one byte, yet only the values $0,1,\ldots,6$ are employed. Since a MIX byte can hold at least $64$ values, many I-field values are unused. These values may designate new index registers. Thus the new machine could contain registers

$$
rI7,rI8,\ldots,
$$

without affecting existing programs, because old programs never generate these I-field values.

Another possibility is to use unused I-field values as modifiers rather than as register numbers. For example, one value might specify indirect addressing. If an instruction has address part $AA$ and a special I-field value indicating indirection, the effective address $M$ could be obtained from a designated field of $\mathrm{CONTENTS}(AA)$. Ordinary values $0,\ldots,6$ would retain their present interpretation.

The A-register and X-register together contain ten bytes. A larger machine may provide additional extension registers placed to the right of $rX$. Existing multiplication and division instructions would continue to use only $rA$ and $rX$, but new instructions could exploit the larger collection of registers to obtain products and dividends of greater length.

The comparison indicator presently possesses three values, namely `LESS`, `EQUAL`, and `GREATER`. Additional status information may be introduced independently of this indicator. For example, separate condition bits for arithmetic carry or sign could be added. Since ordinary MIX instructions examine only the comparison indicator and overflow toggle, old programs would behave identically.

The instruction set itself may be enlarged. Numerous operation codes are unused. These codes may be assigned to new operations such as floating-point arithmetic, character manipulation, block transfer, or hardware-supported input-output buffering. Existing operation codes would preserve their meanings.

The size of a byte may also be increased. Section 1.3.1 requires correctly written MIX programs to assume only that each byte can represent the numbers $0$ through $63$. Hence a machine with larger bytes remains compatible with every properly written program. Such a machine permits larger numbers to be stored in a word and increases the range of arithmetic.

Additional registers devoted to input-output operations are desirable. These registers could permit overlap of computation and external transfers. Since ordinary MIX programs do not refer to such registers, compatibility is preserved.

Subroutine linkage can be improved by introducing further jump registers analogous to $rJ$. Existing jump instructions would continue to use the original $rJ$, while new instructions could save return addresses in alternative registers. Recursive and nested procedures would thereby become easier to implement.

Finally, hardware support for stack operations may be added. New instructions could maintain a pushdown list in memory. Since these instructions would employ previously unused operation codes, existing programs would remain unchanged.

## Verification

Every proposed extension satisfies two requirements.

First, no existing operation code, field specification, register, or address acquires a different meaning. Hence a correctly written MIX program executes exactly as on the original machine.

Second, each new capability is reached only through resources unused by ordinary MIX programs, such as additional operation codes, unused I-field values, enlarged registers, or extra memory locations. Therefore the extension is conservative.

## Notes

The unused values of the I-field are particularly attractive because a byte contains at least $64$ distinct values, whereas only seven values are employed by MIX. This unused capacity naturally suggests additional index registers or new addressing modes. Such changes increase the power of the machine while preserving complete upward compatibility with existing MIX software.

$$
\boxed{\text{Compatible extensions include larger memory, additional registers, new addressing modes, and new operation codes.}}
$$
