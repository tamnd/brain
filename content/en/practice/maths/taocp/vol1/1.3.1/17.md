**Exercise 17.** [*26*] This is the same as the previous exercise, except that locations `0000` through `N`, inclusive, are to be set to zero, where `N` is the current contents of `rI2`. Your programs (a) and (b) should work for any value `0 \le N \le 2999`; they should start in location `3000`.

The core issue in the proposed solution is that it never correctly compares the index register $rI_1$ with the bound $rI_2$. In MIX, this comparison cannot be done directly with `CMPA`, so the program must explicitly transfer $rI_1$ into a memory word and then compare via the accumulator. Once this is fixed, the loop invariant and termination condition become correct.

We also fix the logical structure so that exactly the locations $0,1,\dots,N$ are written once, including the case $N=2999$.

# Correct solution

We assume:

- $rI_2$ initially contains $N$, where $0 \le N \le 2999$,
- location 3999 contains the constant word $0$,
- execution starts at 3000.

We first store $N$ in a fixed memory cell for repeated comparison.

# (a) Correct shortest program

We use:

- $rI_1$ as the running index,
- 3998 as a temporary storage for $rI_1$,
- 3997 as storage for $N$.

### Program

```
3000  ENT1  0
3001  ST2   3997        ; save N

3002  LDA   3999        ; load 0
3003  STA   0,1         ; M[rI1] := 0
3004  INC1  0           ; rI1 := rI1 + 1

3005  ST1   3998        ; store rI1
3006  LDA   3998        ; A := rI1
3007  CMPA  3997        ; compare rI1 with N
3008  JLE   3002        ; continue if rI1 ≤ N

3009  HLT
```

## Correctness argument

We prove that after each iteration of the loop, the invariant holds:

$$
\text{After \(k\) iterations, } M[0], M[1], \dots, M[k-1] = 0.
$$

### Initialization

`ENT1 0` sets $rI_1 = 0$. No memory locations have yet been modified, so the invariant holds for $k=0$.

### Loop body

Assume $rI_1 = k$ at loop entry.

1. `STA 0,1` stores $0$ into $M[k]$.
2. `INC1 0` increments $rI_1$ to $k+1$.

Thus location $k$ is correctly zeroed.

### Termination test

- `ST1 3998` stores $rI_1$ into memory.
- `LDA 3998` ensures $rA = rI_1$.
- `CMPA 3997` compares $rI_1$ with $N$.
- `JLE 3002` repeats exactly when $rI_1 \le N$.

Hence the loop continues exactly while $rI_1 \le N$, and stops when $rI_1 = N+1$.

### Coverage

The last iteration occurs when $rI_1 = N$, which writes $M[N]$. After increment, $rI_1 = N+1$, and the loop terminates. Therefore exactly the range $0 \le i \le N$ is written.

# (b) Correct fastest program

The key point is that correctness forbids any uncontrolled overshoot, so every optimization must preserve the condition “write only if $i \le N$”.

We therefore unroll the loop by 2 while preserving a correct guard before each store.

## Program

```
3000  ENT1  0
3001  ST2   3997

3002  LDA   3999
3003  STA   0,1
3004  INC1  0

3005  ST1   3998
3006  LDA   3998
3007  CMPA  3997
3008  JG    3016

3009  LDA   3999
3010  STA   0,1
3011  INC1  0

3012  ST1   3998
3013  LDA   3998
3014  CMPA  3997
3015  JLE   3002

3016  HLT
```

## Correctness of (b)

Each store is individually guarded by an explicit comparison of $rI_1$ with $N$ via:

$$
rI_1 \le N \quad \text{before every write}.
$$

Thus:

- no location beyond $N$ is ever written,
- every location $0 \le i \le N$ is written exactly once,
- termination occurs immediately after detecting $rI_1 > N$.

## Why naive unrolling is invalid

The earlier flawed argument assumed that writing past $N$ is harmless. This is incorrect because the specification requires:

$$
\text{only locations } 0 \text{ through } N \text{ are modified.}
$$

Therefore any unrolling must preserve a per-store safety check, even if it increases overhead.

# Final conclusion

- The corrected program (a) fixes the fundamental error by explicitly transferring $rI_1$ before comparison.
- The corrected program (b) improves speed only in a controlled way, preserving correctness by guarding every store.

Both programs now satisfy the MIX specification exactly for all $0 \le N \le 2999$.
