**Exercise 14.** [*26*] Suppose you are allowed to use only stacks as data structures. How can you implement a queue efficiently with two stacks?

The simplest and most natural way to keep a linear list inside a computer is to put the list items in consecutive locations, one node after the other. Then we will have

$$
\operatorname{LOC}(X[j+1]) = \operatorname{LOC}(X[j]) + c,
$$

where $c$ is the number of words per node. (Usually $c = 1$. When $c > 1$, it is sometimes more convenient to split a single list into $c$ "parallel" lists, so that the $k$th word of node $X[j]$ is stored a fixed distance from the location of the first word of $X[j]$, depending on $k$. We will continually assume, however, that adjacent groups of $c$ words form a single node.) In general,

$$
\operatorname{LOC}(X[j]) = L_0 + cj,
\tag{1}
$$

where $L_0$ is a constant called the *base address*, the location of an artificially assumed node $X[0]$.

This technique for representing a linear list is so obvious and well-known that there seems to be no need to dwell on it at any length. But we will be seeing many other "more sophisticated" methods of representation later on in this chapter, and it is a good idea to examine the simple case first to see just how far we can go with it. It is important to understand the limitations as well as the power of the use of sequential allocation.

Sequential allocation is quite convenient for dealing with a *stack*. We simply have a variable `T` called the *stack pointer*. When the stack is empty, we let `T = 0`. To place a new element `Y` on top of the stack, we set

$$
T \leftarrow T + 1;\qquad X[T] \leftarrow Y.
\tag{2}
$$

And when the stack is not empty, we can set `Y` equal to the top node and delete that node by reversing the actions of (2):

$$
Y \leftarrow X[T];\qquad T \leftarrow T - 1.
\tag{3}
$$

(Inside a computer it is usually most efficient to maintain the value $cT$ instead of $T$, because of (1). Such modifications are easily made, so we will continue our discussion as though $c = 1$.)

The representation of a *queue* or a more general *deque* is a little trickier. An obvious solution is to keep two pointers, say `F` and `R` (for the front and rear of the queue), with `F = R = 0` when the queue is empty. Then inserting an element at the rear of the queue would be

$$
R \leftarrow R + 1;\qquad X[R] \leftarrow Y.
\tag{4}
$$

Removing the front node (`F` points just below the front) would be

$$
F \leftarrow F + 1;\qquad Y \leftarrow X[F];\qquad \text{if } F = R,\text{ then set } F \leftarrow R \leftarrow 0.
\tag{5}
$$

But note what can happen: If `R` always stays ahead of `F` (so that there is always at least one node in the queue) the table entries used are `X[1], X[2], ..., X[1000], ...`, *ad infinitum*, and this is terribly wasteful of storage space. ## Setup

A queue is a linear list in which insertions occur at the rear and deletions occur at the front, using Knuth’s notation (4) and (5). A stack is a linear list in which both insertion and deletion occur at the top, using notation (2) and (3).

The problem is to implement a queue using only stacks, while keeping the operations efficient. Efficiency is interpreted in the sense that each queue operation should take constant amortized time.

We are allowed two stacks, denoted $S_1$ and $S_2$, each supporting push and pop operations:

$$
S \Leftarrow x
$$

and

$$
x \Leftarrow S.
$$

We must implement the queue operations:

- insert $x$ at rear,
- delete and return the front element.

## Solution

The representation uses two stacks $S_1$ and $S_2$.

Stack $S_1$ stores elements in order of arrival, with the newest element on top. Stack $S_2$ stores elements in reversed order, with the oldest element (in queue order) on top whenever a deletion is performed.

Insertion into the queue is performed by pushing directly onto $S_1$:

$$
S_1 \Leftarrow x.
$$

Deletion from the queue requires accessing the oldest element. If $S_2$ is nonempty, its top element is the oldest remaining queue element, so it is removed directly:

$$
x \Leftarrow S_2.
$$

If $S_2$ is empty, all elements currently in $S_1$ are transferred into $S_2$ by repeated transfers. If $S_1$ contains elements $a_1, a_2, \dots, a_k$ in stack order (with $a_k$ at the top), then performing repeated operations

$$
y \Leftarrow S_1,\quad S_2 \Leftarrow y
$$

moves them into $S_2$ in reverse order, so that $a_1$ becomes the top of $S_2$. After this transfer, the deletion proceeds by

$$
x \Leftarrow S_2.
$$

Thus the full deletion procedure is:

If $S_2$ is nonempty, remove its top element. Otherwise, while $S_1$ is nonempty, repeatedly pop from $S_1$ and push into $S_2$, then pop from $S_2$.

## Verification

Correctness of insertion follows directly from the definition of a queue: elements must be inserted at the rear, and every new element is placed into $S_1$ in order of arrival. No reordering occurs during insertion, so the arrival order is preserved in $S_1$.

Correctness of deletion follows from the transfer invariant. Suppose $S_2$ is empty and $S_1$ contains elements in arrival order $a_1, a_2, \dots, a_k$, where $a_k$ is most recently inserted. The transfer applies the operation $y \Leftarrow S_1$ followed by $S_2 \Leftarrow y$ exactly $k$ times, producing $S_2$ with contents $a_1, a_2, \dots, a_k$ in stack order with $a_1$ at the top. The element $a_1$ is exactly the oldest element in the queue, so removing the top of $S_2$ returns the correct queue element.

After transfer, subsequent deletions from $S_2$ continue to remove elements in increasing order of arrival until $S_2$ becomes empty again. At that point, any remaining elements are still in $S_1$, and the same reversal procedure reestablishes correct order. Therefore every deletion returns the oldest element among all elements currently stored in $S_1$ and $S_2$ combined.

## Efficiency

Each element is pushed into $S_1$ once, moved from $S_1$ to $S_2$ at most once, and popped from $S_2$ once. Each individual stack operation is constant time, so each element participates in a bounded number of constant-time operations. Therefore any sequence of $m$ queue operations requires total time proportional to $m$.
