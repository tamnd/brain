**Exercise 10.** [**] [26] Can you think of a better way to organize a personnel table so that searches as described in the previous exercise would be more efficient? (The answer to this exercise is *not* merely "yes" or "no.")

We reorganize the personnel table so that every attribute list is an **inverted list sorted by a single fixed total order on persons**, for example by a unique person index $1,2,\dots,n$. This is the key structural change: all lists become comparable by the same ordering, so intersections can be done by coordinated linear traversal.

### Data structure

For each attribute value $a$ (for example `FEMALE`, `BLUE`, `BLOND`, `A21`, etc.), store the set of persons having that attribute as a linked list sorted by increasing person index.

Thus every list is monotone in the same global order.

For a range condition such as “age $21$ through $23$”, we do not treat it as a single unordered set. Instead we define it as the union

$$
A = A_{21} \cup A_{22} \cup A_{23},
$$

and we represent $A$ as a single ordered stream obtained by merging the three sorted lists.

This merge is done lazily: we maintain one pointer into each of $A_{21}, A_{22}, A_{23}$, and repeatedly output the smallest current element among them, advancing only that pointer. This produces a single sorted list $A$ without backtracking.

### Query processing as a k-way intersection

A query such as

“blue-eyed blonde women aged 21–23”

becomes an intersection of sorted lists:

$$
BLUE \cap BLOND \cap FEMALE \cap A.
$$

Let the lists be $L_1, L_2, \dots, L_k$, all sorted by the same person index.

We maintain one pointer $p_i$ into each list $L_i$, initially at the first element.

At each step:

1. Let $x_i$ be the current element pointed to by $p_i$ in list $L_i$, if any.
2. Let $m = \min_i x_i$ and $M = \max_i x_i$.
3. If all $x_i = M$, then the same person appears in every list, so we output $M$ and advance all pointers.
4. Otherwise, advance every pointer $p_i$ such that $x_i < M$.

Because all lists are sorted, no pointer ever moves backward.

### Correctness argument

The algorithm maintains the invariant that each pointer $p_i$ always refers to the smallest unexamined element of $L_i$ that could still participate in a future match. Any element smaller than the current maximum $M$ cannot appear in the intersection, because it is missing from at least one list at the current position, and no earlier elements will reappear.

Thus, every output element is in all lists, and every common element must eventually become the current element of all pointers simultaneously.

### Efficiency

Each pointer only moves forward and never backtracks. Therefore each list is traversed at most once. The total work is

$$
O\left(\sum_{i=1}^k |L_i|\right)
$$

for the intersection phase, plus the cost of forming the merged list $A_{21} \cup A_{22} \cup A_{23}$, which is also linear in the sizes of the three age lists.

Hence the cost of a multi-attribute query depends primarily on the total size of the participating lists, not on repeated rescanning.

### Conclusion

The improved organization is to store each attribute value as a globally ordered inverted list and evaluate queries by synchronized pointer traversal, with unions handled by on-the-fly merging. This removes repeated scans entirely and reduces multi-attribute search to a single linear pass over the relevant lists.
