**Exercise 25.** [*22*] Suppose that we have a binary computer and a number $x$, $1 \le x < 2$. Show that the following algorithm, which uses only shifting, addition, and subtraction operations proportional to the number of places of accuracy desired, may be used to calculate an approximation to $y = \log_b x$:

**L1.** [Initialize.] Set $y \leftarrow 0$, $z \leftarrow x$ shifted right 1, $k \leftarrow 1$.

**L2.** [Test for end.] If $x = 1$, stop.

**L3.** [Compare.] If $x - z < 1$, set $z \leftarrow z$ shifted right 1, $k \leftarrow k + 1$, and repeat this step.

**L4.** [Reduce values.] Set $x \leftarrow x - z$, $z \leftarrow x$ shifted right $k$, $y \leftarrow y + \log_b (2^k/(2^k - 1))$, and go to L2. ▎

[*Notes:* This method is very similar to the method used for division in computer hardware. The idea goes back in essence to Henry Briggs, who used it (in decimal rather than binary form) to compute logarithm tables, published in 1624. We need an auxiliary table of the constants $\log_b 2$, $\log_b (4/3)$, $\log_b (8/7)$, etc., to as many values as the precision of the computer. The algorithm involves intentional computational errors, as numbers are shifted to the right, so that eventually $x$ will be reduced to 1 and the algorithm will terminate. The purpose of this exercise is to explain why it will terminate and why it computes an approximation to $\log_b x$.]

The key error in the previous solution is the assumption that a right shift implements exact division by a power of two. In the binary computer model intended here, a shift discards low-order bits, so all identities based on $x/2^k$ are only approximate. The corrected argument must therefore work with truncation inequalities rather than exact equalities.

## 1. Machine model and meaning of shifts

Assume fixed precision binary arithmetic with $m$ fractional bits. A number $u \in [1,2)$ is represented as an integer $U$ such that

$$
u = \frac{U}{2^m}, \qquad 2^m \le U < 2^{m+1}.
$$

A right shift by $k$ bits corresponds to integer division with truncation:

$$
\text{shift}_k(u) \;=\; \frac{\lfloor U / 2^k \rfloor}{2^m}.
$$

Hence, if we write

$$
z = x \text{ shifted right } k,
$$

then with $x = U/2^m$,

$$
z = \frac{\lfloor U/2^k \rfloor}{2^m}.
$$

This implies the crucial inequality

$$
0 \le \frac{x}{2^k} - z < \frac{1}{2^m}.
$$

So $z$ is an approximation to $x/2^k$ with absolute error less than one unit in the last place.

## 2. Behavior of L3 (selection of $k$)

During L3, we repeatedly shift $z$ right and increment $k$. After $k$ shifts,

$$
z = \left(\left(\cdots (x \text{ shifted right } 1)\cdots \right)\right) \text{ shifted right } (k-1).
$$

Thus $z$ satisfies

$$
z \approx \frac{x}{2^k}, \qquad 0 \le \frac{x}{2^k} - z < \frac{1}{2^m}.
$$

The loop in L3 continues while

$$
x - z < 1.
$$

It stops at the smallest $k$ such that

$$
x - z \ge 1.
$$

Because truncation only decreases $z$, we always have

$$
z \le \frac{x}{2^k},
$$

hence the true (ideal) threshold satisfies

$$
x - \frac{x}{2^k} \le x - z + \frac{1}{2^m}.
$$

So L3 still selects a $k$ that is correct up to at most one unit in the last place, which is negligible relative to the magnitude of $x-1$ once $x$ is not extremely close to 1.

## 3. Effect of L4 update

L4 sets

$$
x \leftarrow x - z.
$$

Using the inequality relating $z$ and $x/2^k$,

$$
x - \frac{x}{2^k} - \frac{1}{2^m}
\;\le\;
x - z
\;\le\;
x - \frac{x}{2^k}.
$$

Hence each update removes approximately a fraction $2^{-k}$ of the current value, with absolute truncation error at most $2^{-m}$.

The updated value remains in $[1,2)$ as long as the algorithm continues, since L3 guarantees $z \le x-1$, hence

$$
x - z \ge 1.
$$

Thus the algorithm maintains the invariant range.

## 4. Termination

Let $x = U/2^m$. Each execution of L4 produces

$$
x' = x - z = \frac{U - \lfloor U/2^k \rfloor}{2^m}.
$$

So the integer state updates as

$$
U \leftarrow U - \lfloor U/2^k \rfloor.
$$

If $U > 2^m$, then $\lfloor U/2^k \rfloor \ge 1$, so

$$
U' < U.
$$

Thus $U$ is a strictly decreasing sequence of nonnegative integers bounded below by $2^m$. Therefore the algorithm must terminate.

Termination occurs only when

$$
U = 2^m \quad \Longleftrightarrow \quad x = 1.
$$

No cycling is possible because the integer state strictly decreases.

## 5. Correctness of the logarithm accumulation

Define the ideal (real-arithmetic) update corresponding to the same chosen $k$:

$$
x \mapsto x\left(1 - 2^{-k}\right), \qquad
y \mapsto y + \log_b\!\left(\frac{1}{1 - 2^{-k}}\right)
= y + \log_b\!\left(\frac{2^k}{2^k - 1}\right).
$$

In the machine algorithm, the actual update uses $z$ instead of $x/2^k$, but from the inequality above,

$$
x - z = x\left(1 - 2^{-k}\right) + \varepsilon,
\qquad |\varepsilon| < 2^{-m}.
$$

Thus after $t$ steps,

$$
x_t = x \prod_{i=0}^{t-1}\left(1 - 2^{-k_i}\right) + \delta_t,
\qquad |\delta_t| \le t \cdot 2^{-m}.
$$

Meanwhile,

$$
y_t = \sum_{i=0}^{t-1} \log_b\!\left(\frac{2^{k_i}}{2^{k_i}-1}\right).
$$

Taking logarithms,

$$
\log_b x
=
y_t + \log_b(x_t - \delta_t)
$$

and since $t$ is bounded by the number of machine states, $t \ll 2^m$, the accumulated error $\delta_t$ remains within machine precision.

At termination, $x_T = 1$, so

$$
\log_b(x_T - \delta_T) = \log_b(1 + \text{machine error}) = O(2^{-m}).
$$

Hence

$$
y_T = \log_b x + O(2^{-m}).
$$

So the computed value differs from $\log_b x$ by at most one unit in the last place.

## 6. Why truncation does not affect correctness of the algorithm

The only effect of shifting is to replace exact dyadic fractions by slightly smaller ones. This has two consequences:

1. $z$ is never too large, so subtraction never overshoots below 1.
2. The integer state $U$ still decreases monotonically.

Therefore truncation affects only the low-order bits of intermediate values, not the structure of the algorithm or the choice of $k$. The same sequence of logarithmic correction terms is produced up to machine precision.

## 7. Conclusion

The algorithm operates on integer representations of binary fixed-point numbers. Although shifting introduces truncation, the state update

$$
U \leftarrow U - \lfloor U/2^k \rfloor
$$

is strictly decreasing in $U$, guaranteeing termination at $x=1$.

Each iteration contributes the correct logarithmic correction term

$$
\log_b\!\left(\frac{2^k}{2^k - 1}\right),
$$

and truncation affects only values below machine precision.

Therefore the algorithm terminates and produces an approximation

$$
y \approx \log_b x
$$

with error bounded by the precision of the binary representation, using only shifts, additions, and subtractions, with work proportional to the number of bits of accuracy required.
