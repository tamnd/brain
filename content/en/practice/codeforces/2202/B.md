## Problem Understanding

We are given a string $X$ of length $n$ consisting of characters 'a', 'b', and '?'. Conceptually, $X$ is a partially observed version of a string $S$ generated from an original alternating string $T$ of length $n$ where positions alternate between 'a' and 'b', starting with 'a'. The generation rule allows repeatedly taking either the first or last character of $T$ and appending it to $S$.

The task is to determine whether it is possible to fill in all '?' in $X$ such that the resulting string $A$ could have been generated from this first-or-last extraction procedure.

Constraints are significant: $n$ can reach $2 \cdot 10^5$, and the total length across all test cases is bounded by $2 \cdot 10^5$. This immediately rules out any algorithm that attempts to simulate every possible sequence of extractions, which would be exponential in $n$.

A non-obvious edge case arises when $X$ starts or ends with a fixed character that conflicts with the alternating pattern of $T$. For example, if $n=5$ and $X$ is "ba??b", a naive approach might try to greedily fill '?' but fail to detect that the first 'b' cannot come from either end of $T$.

Another subtlety is that any valid string $S$ must be a "zigzag" of the original $T$: the relative order of characters from $T$ is preserved if they are taken from the same end, but interleaving from opposite ends can reverse segments. This requires reasoning about which positions can legitimately be either 'a' or 'b'.

## Approaches

The brute-force approach attempts to simulate all possible sequences of taking the first or last character. For each step, we would try both options recursively until either $T$ is exhausted or a conflict with $X$ is found. This approach is correct because it directly models the generation procedure, but it is exponential in $n$. Even for $n = 20$, this could involve over a million recursive calls, and for $n = 2 \cdot 10^5$ it is infeasible.

The key insight is that the parity of positions in $T$ dictates constraints on the characters that can appear at the corresponding positions in $S$. Specifically, the first character of $S$ must be either the first or last character of $T$. The first character of $T$ is always 'a', the last character is 'a' if $n$ is odd and 'b' if $n$ is even. Extending this observation, the valid sequence of extractions can be represented as a two-pointer simulation: maintaining a window on $T$ and greedily checking whether $X$ is consistent with taking either the left or right end at each step.

The algorithm reduces to checking whether $X$ can be generated by matching characters from the two ends of $T$, respecting the original alternating pattern. Each '?' can match either valid choice. This gives an $O(n)$ solution per test case.

| Approach | Time Complexity | Space Complexity | Verdict |
| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(n) | Too slow |
| Two-pointer check | O(n) | O(1) | Accepted |

## Algorithm Walkthrough

1. Construct the original string $T$ of length $n$ as "ababa..." up to length $n". This step defines the source of characters for validation.
2. Initialize two pointers: `l` at the start of $T$ and `r` at the end.
3. Iterate over each character `X[i]` from left to right.
4. For each position `i`, compute the valid character if taking from the left (`T[l]`) and from the right (`T[r]`).
5. If `X[i]` is '?', it can match either choice. Otherwise, check if it matches at least one of the two possible characters. If it matches neither, terminate and return "NO".
6. If `X[i]` matched the left character, increment `l`; if it matched the right character, decrement `r`. This simulates "removing" the character from $T$ as per the original generation algorithm.
7. If all characters of $X$ are processed without conflict, return "YES".

The invariant maintained is that the window `[l, r]` of $T$ always represents the characters still available to be chosen for the remainder of $S$, and any match at position `i` respects the rules of extraction. By checking both ends at each step, we account for all valid sequences implicitly without recursion.

## Python Solution

```python
import sys
input = sys.stdin.readline

def can_generate(n, X):
    l, r = 0, n - 1
    for i in range(n):
        left = 'a' if l % 2 == 0 else 'b'
        right = 'a' if r % 2 == 0 else 'b'
        if X[i] != '?' and X[i] != left and X[i] != right:
            return "NO"
        if X[i] == left:
            l += 1
        elif X[i] == right:
            r -= 1
        else:  # '?', can choose either; prefer left to simplify pointers
            if l % 2 == 0:
                l += 1
            else:
                r -= 1
    return "YES"

t = int(input())
for _ in range(t):
    n = int(input())
    X = input().strip()
    print(can_generate(n, X))
```

The function `can_generate` implements the two-pointer strategy. The pointers `l` and `r` track the remaining substring of `T`. At each step, we consider both ends as potential matches for the current character in `X`. The code carefully handles '?' by preferring the left pointer when both options are valid, which simplifies the pointer updates. Boundary indices are handled by ensuring `l` and `r` are always within `[0, n-1]`.
