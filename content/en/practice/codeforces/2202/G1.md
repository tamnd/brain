## Solution
## Problem Understanding

We are asked to maintain an initially all-white $n \times n$ grid and process a sequence of queries that each turn a single white cell black. After each query, we need to check whether the current grid remains "monotone." Monotonicity is defined by a subtle condition: we cannot find two rows $i < j$ and two columns $k < l$ where the four corresponding cells form a pattern of alternating colors - specifically, where $C[i,k] = C[j,l]$, $C[i,l] = C[j,k]$, and the colors in the same column differ. Intuitively, this is saying that no 2x2 submatrix can form a "checkerboard" pattern.

We are guaranteed that every query changes a previously white cell to black and that the sum of $n$ across all test cases is at most 25,000, and the total number of queries does not exceed 200,000. Since $n$ can be as large as 25,000, any approach that inspects the full grid after each query is immediately too slow. A naive $O(n^2)$ per query would result in roughly $10^9$ operations per test case, which is not feasible in practice.

A subtle edge case arises when updates occur near each other along consecutive rows and columns. For instance, in a 3x3 grid, if we first fill the diagonal $(1,1)$, $(2,2)$, and then $(1,2)$, the resulting pattern immediately violates monotonicity because it allows a 2x2 checkerboard with rows 1 and 2, columns 1 and 2. A careless approach that only looks at newly added cells or scans rows individually could miss such violations.

## Approaches

The brute-force method is straightforward. After each query, one could iterate over all pairs of rows $i < j$ and all pairs of columns $k < l$, checking whether the four cells form the forbidden checkerboard pattern. This is correct because it directly implements the definition of monotonicity, but the complexity is $O(n^2)$ for each query and $O(n^2 q)$ overall. With $n$ up to 25,000 and $q$ up to 200,000, this would easily require $10^{14}$ operations and is hopelessly slow.

The key insight to optimize comes from the structure of the forbidden pattern. A checkerboard can only form between consecutive or nearby cells where one row has a black cell strictly to the left of a black cell in another row. In other words, we can track for each row the rightmost black cell that has been painted so far. Then, a violation occurs if some row has a black cell in a column left of the rightmost black in the previous row. This reduces the problem to tracking a single "max black column so far" value for each row and comparing it with adjacent rows.

Once we realize that we only need to track these rightmost-black positions, checking monotonicity reduces to a simple update and a local comparison between consecutive rows. Each query becomes $O(1)$ to update and $O(1)$ to check for violation. The overall complexity becomes $O(n + q)$ per test case, which easily fits within the problem constraints.

| Approach | Time Complexity | Space Complexity | Verdict |
| --- | --- | --- | --- |
| Brute Force | O(n²·q) | O(n²) | Too slow |
| Rightmost Black Tracking | O(n + q) | O(n) | Accepted |

## Algorithm Walkthrough

1. For each test case, initialize an array `rightmost_black` of size $n$, storing for each row the index of the rightmost black cell. Initially, all values are zero because all cells are white.
2. Maintain a global flag `monotone` which starts as `True`. This flag will be updated as soon as a violation occurs.
3. For each query `(r, c)`, update `rightmost_black[r]` to `c` if `c` is greater than the current value. This records the new rightmost black cell for row `r`.
4. Check only the row `r` and its immediate neighbors (row `r-1` and row `r+1` if they exist). If any neighbor has a rightmost black column that is to the right of `r`’s rightmost black column, a checkerboard violation occurs. In that case, mark `monotone` as `False`.
5. Output "YES" if `monotone` is still `True`, otherwise "NO". Once `monotone` becomes `False`, it cannot revert to `True` because cells are only turned black.

Why it works: This approach works because the monotone condition is equivalent to forbidding any 2x2 checkerboard. The checkerboard can only appear when a row has a black cell that is left of another row's black cell further down. By tracking rightmost black cells per row and comparing with adjacent rows, we capture all possible positions where a violation can appear. No violations can occur elsewhere, so checking only the updated row and its neighbors suffices.

## Python Solution

```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    for _ in range(t):
        n, q = map(int, input().split())
        rightmost_black = [0] * (n + 2)  # 1-indexed, add dummy 0 and n+1
        monotone = True
        output = []
        for _ in range(q):
            r, c = map(int, input().split())
            if c > rightmost_black[r]:
                rightmost_black[r] = c
            # Check for violation with previous row
            if r > 1 and rightmost_black[r] < rightmost_black[r-1]:
                monotone = False
            # Check for violation with next row
            if r < n and rightmost_black[r] > rightmost_black[r+1]:
                monotone = False
            output.append("YES" if monotone else "NO")
        print("\n".join(output))

if __name__ == "__main__":
    solve()
```

The solution initializes an array one element larger than $n$ to avoid boundary checks. Each query updates the rightmost black cell in constant time and immediately checks for a violation with the neighboring rows. The comparison `rightmost_black[r] < rightmost_black[r-1]` captures the checkerboard that would arise if a lower row has a black cell to the left of an upper row, and the symmetric check `rightmost_black[r] > rightmost_black[r+1]` captures the reverse. Once `monotone` is set to `False`, it is never reset because black cells never revert.
