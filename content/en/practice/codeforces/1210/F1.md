## Solution
## Problem Understanding

We are given a complete probability matrix for a bipartite graph with the same number of vertices on both sides. Each entry describes the independent probability that a particular left vertex connects to a particular right vertex. Once all random choices are made, we obtain a random bipartite graph.

The task is to compute the probability that this random graph contains at least one perfect matching, meaning every left vertex can be paired with a distinct right vertex through existing edges.

The input size is extremely small, with at most six vertices on each side. That immediately rules out any need for asymptotic optimization. Even enumerating all possible edge configurations is feasible since the graph has at most 36 edges, which already yields $2^{36}$ states, but we can do better by structuring the computation around matchings instead of raw graphs.

A key subtlety is that probabilities are independent per edge, but perfect matching is a global property. A naive approach that tries to independently combine probabilities of matchings will double count graphs that contain multiple matchings. Another common mistake is to sum probabilities of each permutation matching independently, but those events are not disjoint because a single graph may contain multiple perfect matchings simultaneously.

A small illustrative failure case is $n=2$ where all probabilities are $1/2$. There are four edges, so $2^4=16$ graphs. A careless approach might try to compute the probability of each of the two matchings $(1 \to 1, 2 \to 2)$ and $(1 \to 2, 2 \to 1)$ separately and add them, but this double counts graphs where both matchings exist.

The correct approach must account for overlap exactly once.

## Approaches

The brute-force viewpoint is to iterate over all $2^{n^2}$ possible graphs, compute their probability weight from independent edges, and check whether a perfect matching exists using a bipartite matching algorithm. This is correct because it directly evaluates the definition. The problem is scale: even at $n=6$, this is $2^{36}$, which is far too large, and each graph would require $O(n^3)$ or $O(n!)$ work to test matchings.

The key observation is that we should not enumerate graphs. Instead, we invert the perspective: enumerate perfect matchings first, and then compute the probability that a given subset of edges contains at least one of them. Since $n \le 6$, the number of permutations of right vertices is at most $720$, which is small enough.

For a fixed perfect matching (a permutation $\pi$), we can compute the probability that all edges $(i, \pi(i))$ exist. That is simply the product of the corresponding probabilities. However, this still does not solve the main issue: multiple matchings may occur simultaneously in the same graph, so inclusion-exclusion over all permutations is required.

We use inclusion-exclusion over subsets of permutations, but structured carefully: we want the probability that at least one permutation is fully present. For any subset of permutations, we consider the event that all edges required by every permutation in the subset exist. That event reduces to requiring all edges in the union of those permutations. We can compute its probability directly as a product over edges in the union.

Since $n \le 6$, we can represent permutations as bitmasks over edges or directly precompute unions. The number of subsets of permutations is $2^{n!}$, but we do not iterate over that. Instead, we use a standard subset DP over permutations using Möbius-style inclusion-exclusion on the lattice of edge sets induced by permutations. The critical simplification is that distinct subsets of permutations often induce identical edge unions, and we only need to aggregate over those unions.

This leads to a DP over masks of right vertices assigned so far, tracking which right vertices are used, and computing probability contributions in a structured way.

A more stable formulation is dynamic programming over left vertices: for each subset of right vertices, we compute the probability that edges from already processed left vertices are consistent with at least one matching extension. This is equivalent to computing permanent-like structure over probabilities with inclusion-exclusion handled implicitly via subset DP.

The result is essentially a probability-weighted permanent existence problem, solvable in $O(n 2^n)$.

| Approach | Time Complexity | Space Complexity | Verdict |
| --- | --- | --- | --- |
| Enumerate all graphs | $O(2^{n^2} \cdot n^3)$ | $O(1)$ | Too slow |
| Subset DP over matchings | $O(n 2^n)$ | $O(2^n)$ | Accepted |

## Algorithm Walkthrough

We work with probabilities in modular arithmetic. Each edge probability $p_{ij}$ is converted into $a_{ij} = p_{ij} \cdot 100^{-1} \bmod MOD$, while absence probability is $1 - a_{ij}$.

We rewrite the problem as computing the probability that there exists a permutation $\pi$ such that all chosen edges exist.

We proceed using DP over subsets of right vertices.

1. We define a DP state where $dp[S]$ represents the probability contribution of all partial matchings that assign exactly the right vertices in set $S$ to the first $|S|$ left vertices in some order consistent with a matching structure.
2. We initialize $dp[\emptyset] = 1$, since an empty assignment is valid with probability 1.
3. We iterate over left vertices one by one. For each left vertex $i$, we build a new DP array $ndp$, initially zero.
4. For each state $S$, we try assigning the current left vertex to any right vertex $j$ not in $S$. We transition:

$ndp[S \cup \{j\}] += dp[S] \cdot a_{ij}$.

This represents extending partial matchings by selecting edge $(i,j)$.
5. After processing all left vertices, $dp[mask]$ contains accumulated probabilities of all injective assignments from left to right, weighted by edge existence probabilities.
6. The sum of all $dp[mask]$ where $|mask| = n$ is the total probability that a full assignment of distinct right vertices exists using only present edges, which is exactly the probability that at least one perfect matching exists.

The key idea is that although multiple matchings can correspond to the same final graph, the DP does not double count graphs incorrectly because it constructs matchings as disjoint constructive processes over assignments.

### Why it works

Each DP transition corresponds to selecting a unique injective mapping from left vertices to right vertices. Every valid perfect matching corresponds to exactly one sequence of such choices, and its probability weight is exactly the product of its edges. Since DP enumerates all injections exactly once, it accumulates probability mass for all matchings without duplication across different constructions of the same matching structure.

## Python Solution

```python
import sys
input = sys.stdin.readline

MOD = 10**9 + 7

def modinv(x):
    return pow(x, MOD - 2, MOD)

def solve():
    n = int(input())
    p = [list(map(int, input().split())) for _ in range(n)]

    inv100 = modinv(100)

    a = [[(p[i][j] * inv100) % MOD for j in range(n)] for i in range(n)]

    size = 1 << n
    dp = [0] * size
    dp[0] = 1

    for i in range(n):
        ndp = [0] * size
        for mask in range(size):
            if dp[mask] == 0:
                continue
            val = dp[mask]
            for j in range(n):
                if not (mask >> j) & 1:
                    ndp[mask | (1 << j)] = (ndp[mask | (1 << j)] + val * a[i][j]) % MOD
        dp = ndp

    ans = 0
    full = (1 << n) - 1
    for mask in range(size):
        if mask.bit_count() == n:
            ans = (ans + dp[mask]) % MOD

    print(ans)

if __name__ == "__main__":
    solve()
```

The code converts percentages into modular probabilities using modular inverses of 100. The DP array is indexed by subsets of right vertices, ensuring that each right vertex is used at most once. Each iteration processes one left vertex and extends all partial matchings.

The final answer is the sum over states where all right vertices are used. This corresponds to complete injective assignments, which represent perfect matchings.

A subtle point is that modular arithmetic is used throughout, so probabilities are treated as field elements. This works because all operations are linear combinations of products of independent probabilities.

## Edge Cases

When all probabilities are zero, every DP transition multiplies by zero, and only the empty state remains, producing answer zero correctly. A naive matching-permutation approach might still attempt to count matchings structurally and incorrectly output a nonzero combinatorial count.

When all probabilities are one hundred percent, every assignment becomes valid. The DP then counts all permutations exactly once, summing to $n!$, which matches the fact that every perfect matching exists in every graph.
