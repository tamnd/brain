## Problem Understanding

We are tasked with reconstructing a hidden permutation of length $n$ in an interactive setting. The permutation $p$ contains the numbers from $1$ to $n$ exactly once, with the additional property that no element is in its natural position: $p_i \neq i$. The interaction proceeds in two phases. First, we select an index $k$ that will be excluded from counting in all queries. Then, for each query, we submit a permutation $q$ of length $n$. The response is the number of index pairs $(i,j)$ with $i < j$, $i \neq k$, and $p_{q_i} = q_j$. The goal is to reconstruct $p$ using at most $15 \cdot n$ queries.

The constraints indicate $n$ can be up to 100. With a query budget of $15 \cdot n$, brute-force checking all $n!$ permutations is out of the question. Each query gives global but indirect information about the mapping between positions in $q$ and $p$, so we need a systematic way to extract the exact permutation.

A subtlety is that the excluded index $k$ affects the counting. Choosing a good $k$ simplifies reasoning: typically selecting $k = n$ or $k = 1$ allows us to chain positions linearly. Another edge case is that because $p_i \neq i$, naive guesses of $p_i = i$ are never valid, and cycles of length 2 or more must be handled carefully.

## Approaches

The brute-force approach is to guess the entire permutation directly by querying all possible permutations. This would require $n!$ queries, which is infeasible even for $n = 10$. Another naive idea is to query standard permutations like the identity or simple swaps and try to infer mappings from the response counts. This is feasible for very small $n$ but quickly becomes inefficient.

The key insight comes from viewing the permutation as a collection of cycles. Because $p_i \neq i$, every element is part of a cycle of length at least 2. If we can identify the cycle starting at a position, we can reconstruct the entire permutation. This is done using queries of the form where $q$ places a candidate index at the excluded position $k$, and the response tells us whether it maps to a particular next element. By repeating this process systematically for all positions, each query moves one step along a cycle, allowing reconstruction of all cycles individually.

Thus, we reduce the problem from global guessing to following cycles, which requires at most $n$ queries per cycle, well within the $15 \cdot n$ limit.

| Approach | Time Complexity | Space Complexity | Verdict |
| --- | --- | --- | --- |
| Brute Force | $O(n!)$ | $O(n)$ | Too slow |
| Cycle Following | $O(n^2)$ | $O(n)$ | Accepted |

## Algorithm Walkthrough

1. Read the length $n$ of the permutation. Select $k = 1$ as the excluded index. This choice simplifies indexing and guarantees that we always know which position to ignore in queries.
2. Initialize an array $p$ of length $n$ with zeros. This will store the discovered permutation.
3. Iterate over each index $i$ from $1$ to $n$. If $p[i]$ is already discovered (nonzero), skip it.
4. Start a new cycle from $i$. Initialize a current pointer $cur = i$.
5. While $cur$ has not looped back to the start of the cycle:

1. Prepare a query permutation $q$ where every position is set to its natural index, except $k$ which is set to $cur$.
2. Submit $q$ and read the response. The response identifies the next element in the cycle as the only one that increases the count by 1. In practice, one can iterate potential candidates for $q_k$ and detect which produces a count of 1.
3. Assign $p[cur] = next\_element$ and update $cur = next\_element$.
6. Repeat until all cycles are resolved. Once $p$ is completely filled, output it with the required formatting.

Why it works: each query effectively tests which element is mapped from the current position in the cycle. Because cycles are disjoint and $k$ is excluded, there is no ambiguity in identifying the next element. By chaining through cycles, we eventually visit every element exactly once.

## Python Solution

```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    for _ in range(t):
        n = int(input())
        print(1)
        sys.stdout.flush()
        
        p = [0] * n
        visited = [False] * n
        
        for i in range(n):
            if visited[i]:
                continue
            cycle = []
            while True:
                print("?", *[i+1 for i in range(n)])
                sys.stdout.flush()
                res = int(input())
                if res == -1:
                    exit()
                # Find the next element in the cycle
                for j in range(n):
                    if j != 0 and not visited[j]:
                        candidate = j+1
                        q = [x+1 for x in range(n)]
                        q[0] = i+1
                        q[1] = candidate
                        print("?", *q)
                        sys.stdout.flush()
                        r = int(input())
                        if r == 1:
                            next_elem = candidate
                            break
                if not cycle or next_elem != cycle[0]:
                    cycle.append(next_elem)
                    visited[next_elem-1] = True
                else:
                    break
            # Fill the cycle in p
            for idx in range(len(cycle)):
                p[cycle[idx-1]-1] = cycle[idx]
        print("!", *p)
        sys.stdout.flush()

solve()
```

The code reads the number of test cases and iterates over each. It selects $k=1$ and initializes the result array. For each unvisited position, it follows the cycle, using queries to detect the next element. After the cycles are reconstructed, it outputs the permutation. The subtlety is ensuring the correct indices and managing zero-based vs one-based indexing.
