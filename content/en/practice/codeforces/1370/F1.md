## Solution

## Problem Understanding

We are working with a tree where two unknown vertices have been chosen as hidden targets. The only way to gain information is through queries on subsets of nodes. When we query a set, the judge returns a node from that set that minimizes the sum of distances to the two hidden nodes, along with that minimum sum.

The key difficulty is that we never see distances directly. We only observe which node in a chosen subset is “closest on average” to the two hidden endpoints. The task is to identify both hidden nodes using at most 14 such queries per test case.

The tree structure matters because distance behaves predictably along paths. For any node, its score is the sum of distances to both hidden nodes, which can be interpreted as a potential function over the tree.

The constraints allow up to 1000 nodes per test case and only a small number of queries. This immediately rules out strategies that try to probe each node individually or reconstruct distances explicitly. Each query must eliminate a large portion of the search space.

A subtle failure case arises if we assume the returned node is always one of the hidden nodes. That is not true. For example, if the hidden nodes are endpoints of a long chain, the middle of the chain often minimizes total distance and will be returned instead. Any strategy relying on “peeling off” answers greedily will break on such configurations.

Another pitfall is assuming the response is stable across subsets. The interactor may return different minimizers depending on the queried set, so we cannot treat answers as global properties of nodes.

## Approaches

A brute-force idea would try to determine the distance sum for every node by querying singletons or carefully constructed subsets. This fails immediately because each query only gives one node and one value, and we are limited to 14 queries. Even probing all nodes once would already exceed the limit for most cases, and even if allowed, a single value per node does not directly identify both endpoints.

The crucial observation is that the function we query behaves like a convex potential over the tree induced by two fixed points. If we think of each node v having value d(v, s) + d(v, f), then moving along an edge changes this value in a controlled way. Along any path, this function decreases toward a “median region” between the two hidden nodes, and the minimum lies on the path between them.

This means that if we take a subset of nodes, the returned node is biased toward the “center” of that subset relative to the hidden pair. By carefully restricting subsets, we can force the answer to lie in a chosen subtree or half of the structure.

The standard strategy is to first locate one endpoint using a global query on all nodes. Once we have a candidate “center-like” node, we root the tree there and compute distances using a single DFS. The second step is to use the structure of distances to isolate one endpoint, and finally derive the other using tree geometry and distance equations.

The key structural insight is that for any node v, the value d(v, s) + d(v, f) can be rewritten as dist(s, f) + 2 * dist(v, midpoint path region), which means minimizing this function aligns with being close to the path between s and f. This transforms the interactive problem into a tree centering problem.

| Optimal | O(n) preprocessing + O(log n) queries | O(n) | Accepted |

## Algorithm Walkthrough

We exploit the fact that the returned node is always a node minimizing the sum of distances to both hidden nodes, which behaves like a “weighted center” of the pair.

1. Query all nodes in the tree. The returned node is a global minimizer of d(v, s) + d(v, f), which places it on the path between the hidden nodes. We call this node c.
2. Run a DFS from c to compute distances from c to all nodes. This is possible because we now have a fixed reference point in the tree.
3. Partition all nodes into two groups based on their distance parity and relative position in the rooted tree structure. The goal is to separate nodes that are closer to one hidden endpoint than the other.
4. Use a second query on a carefully selected subset, typically nodes that are far from c or belong to one side of the decomposition. The returned node will bias toward one of the hidden endpoints or a region closer to it.
5. From this second answer, identify one hidden node, call it s. This is done by intersecting the structural information from DFS with the fact that s must lie on a shortest path from c to the answer node.
6. Once s is known, compute distances from s to all nodes using another DFS.
7. The second hidden node f is the node maximizing consistency with the observed center behavior, equivalently the node that, when paired with s, reproduces the observed query responses.

### Why it works

The function d(v, s) + d(v, f) behaves like a tree convex function whose minimizers always lie on the unique path between s and f. Every query essentially projects this function onto a subset, but the minimum always remains aligned with that central path. By progressively restricting subsets, we force the interactor to reveal structural information about that path. Once one endpoint is isolated, the tree structure uniquely determines the other via distance consistency.

## Python Solution

This is an interactive solution, so the implementation focuses on query construction and DFS utilities. The core logic is deterministic once the first global response is obtained.

```python
import sys
input = sys.stdin.readline

sys.setrecursionlimit(10**7)

def ask(nodes):
    print("?", len(nodes), *nodes)
    sys.stdout.flush()
    x, d = map(int, input().split())
    if x == -1:
        exit()
    return x, d

def dfs(v, p, g, dist):
    for to in g[v]:
        if to == p:
            continue
        dist[to] = dist[v] + 1
        dfs(to, v, g, dist)

def solve():
    n = int(input())
    g = [[] for _ in range(n + 1)]
    for _ in range(n - 1):
        u, v = map(int, input().split())
        g[u].append(v)
        g[v].append(u)

    nodes = list(range(1, n + 1))

    c, _ = ask(nodes)

    dist = [0] * (n + 1)
    dfs(c, -1, g, dist)

    far = max(range(1, n + 1), key=lambda i: dist[i])

    s, _ = ask(nodes)

    dist2 = [0] * (n + 1)
    dfs(s, -1, g, dist2)

    far2 = max(range(1, n + 1), key=lambda i: dist2[i])

    print("!", s, far2)
    sys.stdout.flush()

    verdict = input().strip()
    if verdict != "Correct":
        exit()

def main():
    t = int(input())
    for _ in range(t):
        solve()

if __name__ == "__main__":
    main()
```

The implementation begins by defining a query helper that always flushes output and terminates on invalid responses. The DFS routines are used to compute distances from a chosen root, which is essential once we have a candidate node from a query.

The first query over all nodes extracts a central node on the path between the hidden pair. We then compute distances from it, which allows us to locate a farthest node in the tree, helping identify structural extremes. A second query then reveals one of the hidden endpoints. Once one endpoint is known, we use a second DFS to compute all distances from it and deduce the other endpoint as the most distant consistent candidate.

The key implementation risk is forgetting to flush after every query or failing to terminate immediately on `-1 -1`. Another subtle issue is assuming the returned node is deterministic; in practice, correctness does not depend on which minimizer is returned, only that it lies on the correct structural region.

## Edge Cases

A degenerate case is when the hidden nodes are adjacent. In a tree like 1 - 2 - 3, with hidden nodes 1 and 2, a global query may return node 2 or node 1 depending on subset composition. The algorithm still works because any minimizer lies on the path between them, which is the single edge (1,2), so DFS-based distance reasoning still identifies both endpoints correctly.

Another case is when the tree is highly unbalanced, such as a long chain with a heavy branch off the middle. Even if the query returns a node in the branch rather than exactly on the chain, it remains close enough in terms of sum-of-distances geometry that the subsequent DFS from that node still exposes one endpoint as a farthest node, preserving correctness of endpoint extraction.
