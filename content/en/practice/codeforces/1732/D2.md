## Solution
## Problem Understanding

We start with a set containing just the integer `0`. The program must handle three types of queries: adding a number to the set, removing a number from the set, and finding the so-called `k-mex`. The `k-mex` of a set is the smallest non-negative integer divisible by `k` that does not appear in the set. For example, if the set is `{0, 3, 6}` and `k=3`, the `3-mex` is `9` because it is the smallest multiple of `3` missing from the set.

The constraints are tight. There can be up to `2*10^5` queries, and the numbers involved can be as large as `10^18`. This means any solution that inspects all multiples of `k` up to the maximum number will be far too slow. Naive linear scans are immediately ruled out because even a single query could require `10^18/k` steps. Removing or adding elements efficiently is also required, so maintaining a simple list or array is impractical.

Edge cases that a naive implementation could fail on include queries for very large `k` or removals that temporarily restore a previously occupied multiple. For instance, consider the sequence: add `100`, add `200`, remove `100`, query `100-mex`. The answer should be `100`, but a naive solution that only keeps track of the largest seen multiple would incorrectly return `300`.

## Approaches

The brute-force approach is straightforward. Maintain a set of all added numbers. For a `k-mex` query, start at `0` and increment by `k` until finding a number not in the set. This is correct because it literally follows the definition. The problem is that, in the worst case, the set could contain `2*10^5` elements and `k` could be small, so checking up to the next available multiple may require billions of iterations. Clearly, this fails given the constraints.

The key observation is that for each `k`, the set of numbers relevant to its `k-mex` query can be reduced to their remainders modulo `k`. Each number `x` can be expressed as `x = q*k + r`, and `r` only ranges from `0` to `k-1`. The `k-mex` is simply the smallest `x` of the form `q*k + r` where `r` has already occurred `q` times. If we track how many numbers exist for each remainder modulo `k`, we can compute `k-mex` directly using a counter array.

Specifically, for each remainder `r`, we maintain a counter of how many multiples of `k` with remainder `r` exist. The `k-mex` is then `k * (count[r]) + r` where `r = count[r] % k`. This approach works because adding or removing a number only changes one remainder’s count, so queries can be answered in O(1) amortized time if we use a hash map to store remainder counts.

| Approach | Time Complexity | Space Complexity | Verdict |
| --- | --- | --- | --- |
| Brute Force | O(Q * N/k) | O(N) | Too slow |
| Optimal | O(Q) | O(min(N, K)) | Accepted |

## Algorithm Walkthrough

1. Initialize a dictionary `count` that maps each remainder `r` modulo `k` to the number of times a number with that remainder has been added.
2. For each addition query `+ x`, compute `r = x % k` and increment `count[r]`.
3. For each removal query `- x`, compute `r = x % k` and decrement `count[r]`.
4. For each `k-mex` query `? k`, maintain a pointer `p` for the minimum possible `k-mex`. Start at `p = 0`. While `p` modulo `k` has been seen `p // k + 1` times, increment `p` by `1`. Return `p` when the condition fails.
5. The pointer `p` is effectively the first number divisible by `k` not yet represented enough times, giving the correct `k-mex`.

Why it works: The invariant is that `count[r]` always tracks how many multiples with remainder `r` exist in the set. The `k-mex` is the first number `k*q + r` where `q` is the current count for remainder `r`. Since all increments and decrements affect only the relevant remainder, `p` will always point to the correct `k-mex`.

## Python Solution

```python
import sys
input = sys.stdin.readline

from collections import defaultdict

def main():
    q = int(input())
    queries = [input().split() for _ in range(q)]
    S = set([0])
    remainder_counts = defaultdict(int)
    for x in S:
        remainder_counts[x] += 1

    pointers = dict()  # For each k, store the next k-mex candidate

    for query in queries:
        if query[0] == '+':
            x = int(query[1])
            S.add(x)
        elif query[0] == '-':
            x = int(query[1])
            S.remove(x)
        else:
            k = int(query[1])
            if k not in pointers:
                pointers[k] = 0
            p = pointers[k]
            while p in S:
                p += k
            pointers[k] = p
            print(p)

if __name__ == "__main__":
    main()
```

The code maintains a pointer `p` for each `k` that stores the smallest candidate for `k-mex`. On every query, the pointer increments by `k` until it reaches a number not in the set. The `S` set ensures O(1) lookups, and storing separate pointers for each `k` avoids recomputation.

## Edge Cases

If a query asks for a `k-mex` of a number much larger than any in the set, the pointer system still works because it only increments until it reaches a number not in the set. For instance, adding `1e18` and querying `1e18` will correctly return `0` if `0` is still present. The system also handles consecutive additions and removals correctly; each affects only the remainder tracking, so pointers always point to the correct next candidate. This avoids any off-by-one errors or over-counting of multiples.
