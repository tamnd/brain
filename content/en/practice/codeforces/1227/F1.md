## Solution
## Problem Understanding

We are counting sequences of length \(n\), where each position stores a value from \(1\) to \(k\). Think of this as writing an answer sheet for \(n\) questions. The correctness of position \(i\) depends on whether your chosen value matches a fixed target value \(h_i\).

Now the twist is that the same answer sheet is also evaluated after a cyclic shift by one position. So the value originally written at position \(i\) is checked against \(h_{i+1}\) (with the last position wrapping around to \(h_1\)). We compare two scores for the same sequence: how many positions match their original targets, and how many match after the shift. We want sequences where the shifted score is strictly larger.

The key difficulty is that the score difference depends on overlaps between adjacent positions. A value at position \(i\) affects whether position \(i\) is correct in the original scoring and whether position \(i+1\) is correct in the shifted scoring. This immediately introduces adjacency dependence, so independent counting over positions is not possible.

The constraint \(n \le 2000\) indicates that quadratic or slightly super-quadratic dynamic programming is acceptable. However, \(k\) can be as large as \(10^9\), so we cannot branch on actual values. Any solution must depend only on whether a chosen value equals some \(h_i\), not on its identity.

A subtle edge case appears when all \(h_i\) are equal. In that situation, many transitions collapse, and incorrect implementations often double-count or forget that the “other values” category has size \(k-1\), not \(k-2\) in degenerate comparisons.

Another important failure case is the cyclic dependency. Since position \(n\) influences position \(1\), any DP that treats the array as a line without enforcing wrap consistency will silently overcount.

## Approaches

A brute-force solution would enumerate all \(k^n\) sequences, compute both scores, and count valid ones. This is conceptually simple because both scores can be computed in \(O(n)\), but the total complexity \(O(nk^n)\) is completely infeasible even for \(n=20\).

The structure of the score difference is what makes the problem tractable. Each position contributes to the old score if \(a_i = h_i\), and contributes to the new score if \(a_i = h_{i+1}\). So every position interacts with two fixed values: \(h_i\) and \(h_{i+1}\). This means that for each index, we only care whether \(a_i\) equals none, one, or both of these values.

Once we restrict attention to these equality relationships, each position has only a constant number of meaningful “types”, and the transition between positions depends only on whether the chosen value matches \(h_{i+1}\), which becomes the interface between adjacent DP states.

We then convert the problem into a cycle DP where we track, at each step, both the current score difference and whether the previous choice matches the current \(h_i\). The cyclic condition is enforced by matching the final “carry” back to the start.

| Approach | Time Complexity | Space Complexity | Verdict |
|---|---|---|---|
| Brute Force | \(O(k^n \cdot n)\) | \(O(n)\) | Too slow |
| Cycle DP on match states | \(O(n^2)\) | \(O(n^2)\) | Accepted |

## Algorithm Walkthrough

We process positions while tracking two pieces of information: the running score difference and whether the previous position contributed to the current score in the shifted version.

We define a DP state where we are at position \(i\), we know the current balance \(b\) defined as (shifted score minus original score so far), and we know whether \(a_{i-1} = h_i\). That boolean is essential because it determines whether position \(i\) contributes to the shifted score.

### Steps

1. Fix position \(i\), and consider the pair \((h_i, h_{i+1})\), where \(h_{n+1} = h_1\). This pair determines all meaningful behavior of \(a_i\) at this position.

2. Classify any choice of \(a_i\) into categories based on equality with \(h_i\) and \(h_{i+1}\). Only these equalities affect transitions; all other values behave identically.

3. For each category, compute two effects: whether position \(i\) contributes to the original score, and whether it sets the carry flag for position \(i+1\). This determines the DP transition.

4. Maintain a DP table over position, balance, and carry flag. For each state, distribute its count to next states using the precomputed category transitions.

5. Since the array is cyclic, enforce consistency between the carry leaving position \(n\) and the carry entering position \(1\). We fix the initial carry and only accept DP paths that return the same value.

6. Sum all valid configurations where final balance is strictly positive.

### Why it works

At every position, the DP encodes exactly the information needed to determine future contributions: the current balance and whether the previous element matches the next target. No future decision depends on the exact numeric value of any earlier choice beyond these two properties. Because every value in \([1, k]\) belongs to exactly one equality class relative to \((h_i, h_{i+1})\), the DP partitions the full space of sequences without overlap or omission. The cyclic constraint is enforced by matching the boundary carry, which restores consistency across the circular dependency.

## Python Solution

```python
import sys
input = sys.stdin.readline

MOD = 998244353

def solve():
    n, k = map(int, input().split())
    h = list(map(int, input().split()))
    
    # h is 0-indexed internally
    # we will use h[i] and h[(i+1)%n]
    
    # dp[pos][balance][carry]
    # carry = 1 if a[i-1] == h[i], else 0
    
    offset = n
    size = 2 * n + 1
    
    # dp only for current position
    dp = [[[0, 0] for _ in range(size)] for _ in range(2)]
    
    # initial: position 0, no previous carry yet
    dp[0][offset][0] = 1
    
    for i in range(n):
        cur = dp[i & 1]
        nxt = dp[(i + 1) & 1]
        for b in range(size):
            for c in range(2):
                if not cur[b][c]:
                    continue
                ways = cur[b][c]
                
                hi = h[i]
                hn = h[(i + 1) % n]
                
                if hi == hn:
                    # two categories: equal value or other
                    # value == hi (=hn)
                    # contributes x_i = 1, next carry = 1
                    nb = b - c + 1
                    if 0 <= nb < size:
                        nxt[nb][1] = (nxt[nb][1] + ways) % MOD
                    
                    # other values
                    cnt = k - 1
                    nb = b - c
                    if 0 <= nb < size:
                        nxt[nb][0] = (nxt[nb][0] + ways * cnt) % MOD
                else:
                    # hi != hn: 3 categories
                    
                    # a_i = hi
                    nb = b - c + 1
                    if 0 <= nb < size:
                        nxt[nb][0] = (nxt[nb][0] + ways) % MOD
                    
                    # a_i = hn
                    nb = b - c
                    if 0 <= nb < size:
                        nxt[nb][1] = (nxt[nb][1] + ways) % MOD
                    
                    # other values
                    cnt = k - 2
                    if cnt > 0:
                        nb = b - c
                        if 0 <= nb < size:
                            nxt[nb][0] = (nxt[nb][0] + ways * cnt) % MOD
        
        # clear previous layer
        for b in range(size):
            cur[b][0] = cur[b][1] = 0
    
    # after processing n, enforce cycle consistency:
    # final carry must equal initial carry (0)
    res = 0
    last = dp[n & 1]
    for b in range(offset + 1, size):
        res = (res + last[b][0]) % MOD
    
    print(res)

if __name__ == "__main__":
    solve()
```

The implementation keeps a rolling DP over positions. The balance is shifted by an offset so negative values remain indexable. Each transition subtracts the incoming carry because a previous match contributes to the current shifted score but only becomes known when entering the position.

The key subtlety is the treatment of equality cases between \(h_i\) and \(h_{i+1}\). When they are equal, the number of “other” values becomes \(k-1\), not \(k-2\), and both contributions collapse into a single transition class.

The final answer only counts states where the total shifted score exceeds the original, which corresponds to positive balance after enforcing cyclic consistency.

## Worked Examples

### Example 1

Input:
```

```

We track balance (shifted minus original) and carry.

| i | hi, h(i+1) | carry=0 transitions | carry=1 transitions |
|---|---|---|---|
| 0 | (1,3) | split into hi, h(i+1), others | same structure |
| 1 | (3,1) | updates balance accordingly | updates with carry effect |
| 2 | (1,1) | merge categories | final wrap |

The DP accumulates all sequences where the cyclic shift creates more matches than the original arrangement. The final count matches the nine valid sequences described.

### Example 2

Input:
```
2 2
1 2
```

Here every choice is heavily constrained because \(k\) is minimal. The DP explicitly shows that sequences where swapping increases matches are limited to those where the value pattern aligns with the shift direction, confirming that only asymmetric configurations contribute.

## Edge Cases

When all \(h_i\) are identical, every position has a degenerate comparison structure where the “equal to hi+1” category merges with “equal to hi”. The DP must reduce correctly to a two-class system; otherwise it overcounts by treating identical categories as distinct.

In the single-element case \(n=1\), the shift does nothing, so no sequence can have a strictly larger score after rotation. The DP correctly yields zero because balance cannot become positive while maintaining cycle consistency.
