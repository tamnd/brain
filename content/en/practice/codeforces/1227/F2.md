## Solution
## Problem Understanding

We are asked to count the number of ways to fill an array of length `n` with numbers from `1` to `k` such that if we rotate the array by one position to the right, the number of matches with a fixed "correct" array increases. Formally, for each array `a` we construct, we compare `a[i]` with `h[i]`, and then compare the rotated array `a'` with `h`. We count `a` only if `a'` has strictly more matches than `a`. The output is modulo `998244353`.

The array length `n` can be up to `2*10^5` and `k` up to `10^9`. This implies that enumerating all `k^n` possible arrays is infeasible. Any solution must avoid iterating through all arrays, and should ideally operate in `O(n)` time or `O(n log n)` at worst. A naive approach that loops over all combinations would perform `k^n` operations, which is astronomically large even for moderate `n` or `k`.

An edge case occurs when `h` has all values equal. For example, if `n=3` and `h=[2,2,2]`, careless solutions might fail to correctly handle repeated values or the modulo operation. Similarly, when `k=1`, there is only one possible array to consider, and rotation cannot increase matches. Missing these cases can produce off-by-one or negative-count errors.

## Approaches

The brute-force solution generates every array `a` of length `n` with numbers from `1` to `k`. For each `a`, it counts the matches before and after rotation. If the post-rotation matches are greater, it increments a counter. This works correctly but is impractical: for `n=2*10^5` and `k` large, there are `k^n` arrays. Even `k=2` yields `2^200000`, which is impossible to iterate.

The key insight is to handle each position independently and use combinatorics. Consider the relative change in matches caused by the rotation. If a position `i` matches `h[i]`, it contributes `1` before rotation. After rotation, it moves to `i+1` modulo `n` and may or may not match `h[i+1]`. Define three categories for each pair `(h[i], h[i+1])`: equal (`h[i] == h[i+1]`), different (`h[i] != h[i+1]`), and generic unmatched values. Using this, we can compute contributions multiplicatively rather than enumeratively.

Specifically, if `h[i] != h[i+1]`, we have `k-2` choices for `a[i]` that do not match either `h[i]` or `h[i+1]`, `1` choice that matches `h[i]`, and `1` choice that matches `h[i+1]`. This lets us formulate the total number of arrays as a sum over powers of `(k-2)`, carefully counting combinations where the post-rotation matches exceed pre-rotation matches.

| Approach | Time Complexity | Space Complexity | Verdict |
| --- | --- | --- | --- |
| Brute Force | O(k^n) | O(n) | Too slow |
| Combinatorial / Counting | O(n) | O(1) | Accepted |

## Algorithm Walkthrough

1. Compute the number of positions `cnt_same` where `h[i] == h[i+1]` (consider `h[n+1] = h[1]`). These positions cannot increase the number of matches by rotation; any array choice here does not help.
2. Let `cnt_diff = n - cnt_same`. These are the positions where rotation can potentially increase matches.
3. For positions that differ, consider the effect of choices. If we denote `x` as the number of positions where the rotated value matches but original does not, we need arrays where the number of post-rotation matches exceeds pre-rotation matches.
4. The count of valid arrays can be computed using the formula:

```
total = pow(k, cnt_same, MOD) * sum_{i=0}^{cnt_diff // 2} (comb(cnt_diff, i) * comb(cnt_diff - i, i) * pow(k-2, cnt_diff - 2*i, MOD)) % MOD
```

This formula arises from splitting the `cnt_diff` positions into pairs contributing to increase, with `i` positions chosen to match `h[i+1]` exclusively.
5. Return `total % 998244353`.

The invariant is that for each `cnt_same` position we simply multiply by `k`, because any choice here does not affect the inequality condition. For `cnt_diff` positions, the combinatorial sum ensures we only count arrays where rotation strictly increases matches.

## Python Solution

```python
import sys
input = sys.stdin.readline

MOD = 998244353

def mod_pow(a, b, mod):
    res = 1
    while b > 0:
        if b % 2:
            res = res * a % mod
        a = a * a % mod
        b //= 2
    return res

def solve():
    n, k = map(int, input().split())
    h = list(map(int, input().split()))
    
    cnt_same = sum(1 for i in range(n) if h[i] == h[(i+1)%n])
    cnt_diff = n - cnt_same
    
    total = 0
    pow_k2 = mod_pow(k-2, cnt_diff, MOD)
    # Use combinatorial identity for counting pairs
    # Precompute factorials and inverse factorials for efficiency
    fact = [1]*(cnt_diff+1)
    inv_fact = [1]*(cnt_diff+1)
    for i in range(1, cnt_diff+1):
        fact[i] = fact[i-1]*i % MOD
    inv_fact[cnt_diff] = pow(fact[cnt_diff], MOD-2, MOD)
    for i in range(cnt_diff-1, -1, -1):
        inv_fact[i] = inv_fact[i+1]*(i+1) % MOD
    
    def comb(n, r):
        if r < 0 or r > n:
            return 0
        return fact[n]*inv_fact[r]%MOD*inv_fact[n-r]%MOD

    for i in range(cnt_diff//2 + 1):
        ways = comb(cnt_diff, i) * comb(cnt_diff - i, i) % MOD * pow(k-2, cnt_diff - 2*i, MOD)
        total = (total + ways) % MOD

    total = total * mod_pow(k, cnt_same, MOD) % MOD
    print(total)

solve()
```

The first section counts `cnt_same`, the positions where rotation cannot improve matches. The combinatorial part iterates over the number of positions contributing to increase (`i`) and calculates valid arrays using factorials and modular inverses to handle large numbers efficiently. Finally, the contribution from `cnt_same` positions is multiplied.
