## Solution
## Problem Understanding

We are given an array of integers, and we are allowed to repeatedly remove pairs of adjacent elements under a very specific condition: the first element of the pair must be equal to its 1-based index. The task is to find the maximum number of such removals for a given array. Each test case provides a new array, and we must output the maximum number of operations per case.

The constraints are small: each array has at most 100 elements, and the total sum across all test cases is also at most 100. This allows us to consider algorithms that examine all or most elements repeatedly without worrying about efficiency at scale. We need to be careful with edge cases where the array is small or where no index satisfies the removal condition, such as arrays `[1]` or `[3,3,3]`. A naive approach that removes elements in the first valid position it finds might fail to maximize the number of removals, because a later element could have allowed more removals if we had chosen differently.

An example clarifies the subtleties: consider the array `[1, 5, 3, 2, 4]`. If we greedily remove the first element that satisfies the condition, `1`, we remove `[1,5]` and are left with `[3,2,4]`. The next operation removes `[3,2]`, leaving `[4]`, for a total of 2 operations. But if we had removed `[3,2]` first (not allowed here because index 3 is not 3 in the original array), we could not have performed as many operations. So the key is that we must simulate removals carefully in order, choosing the first valid index in the current array at each step.

## Approaches

The brute-force approach iterates over the array repeatedly, scanning for the first index where the value equals its 1-based index and then removing it along with the next element. After each removal, we start scanning from the beginning again. This is correct because it directly implements the rules of the operation. For an array of size `n`, each operation potentially scans up to `n` elements, and up to `n/2` removals may happen, resulting in `O(n^2)` operations per test case. With `n <= 100`, `O(n^2)` is acceptable.

The key insight to simplify and implement efficiently is that we do not need to backtrack or search for a better sequence. The operation is strictly local: removing `a[i]` and `a[i+1]` does not create new possibilities that a naive left-to-right scan would miss. In other words, always choosing the first valid index in the current array produces an optimal sequence. This is because after removing a valid pair, any new valid pair will necessarily be further to the left in the remaining array, which will eventually be processed. So we can implement a straightforward greedy approach: scan left to right, remove whenever possible, and restart scanning from the current index.

| Approach | Time Complexity | Space Complexity | Verdict |
| --- | --- | --- | --- |
| Brute Force | O(n^2) | O(n) | Accepted |
| Greedy Left-to-Right | O(n^2) | O(n) | Accepted |

Both approaches converge to the same solution under these constraints. The greedy approach is simpler to code.

## Algorithm Walkthrough

1. Read the number of test cases `t`. For each test case, read `n` and the array `a`.
2. Initialize a counter `ops = 0` to track the number of operations.
3. Start scanning the array from the first element. Use an index `i = 0`.
4. While `i < len(a) - 1` (so that a next element exists), check if `a[i] == i + 1`.
5. If the condition is true, increment `ops` by 1, remove elements `a[i]` and `a[i+1]`, and restart scanning from `i = 0`. Resetting ensures that we always process the new array from the left, preserving the greedy correctness.
6. If the condition is false, increment `i` to examine the next index.
7. Repeat steps 4-6 until no further operations are possible.
8. Output `ops` for the current test case.

Why it works: at each step, the algorithm removes the leftmost valid pair, which is guaranteed not to block future operations. Because the problem requires maximizing the number of operations and each removal strictly decreases the array size, there is no need for backtracking. The invariant is that after each operation, all possible future operations can still be discovered by scanning from the left.

## Python Solution

```python
import sys
input = sys.stdin.readline

t = int(input())
for _ in range(t):
    n = int(input())
    a = list(map(int, input().split()))
    ops = 0
    i = 0
    while i < len(a) - 1:
        if a[i] == i + 1:
            ops += 1
            a.pop(i)
            a.pop(i)
            i = 0
        else:
            i += 1
    print(ops)
```

The code directly follows the algorithm walkthrough. We use `a.pop(i)` twice to remove two consecutive elements starting at index `i`. After each removal, we reset the index `i = 0` because the array has shifted and earlier elements may now satisfy the operation condition. The loop condition ensures we do not access out-of-bounds elements.
