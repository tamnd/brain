## Solution
## Problem Understanding

We are working with a sequence of integers, and we repeatedly apply a very specific removal rule. At any moment, we look at the current array and are allowed to pick a position $i$ such that the value sitting at that position is exactly equal to its 1-based index inside the _current_ array. Once such a position is chosen, we delete that element together with the element immediately to its right, and the array closes up.

The process shortens the array by exactly two elements each time, and after every deletion the indices shift, so a position that was valid earlier might no longer exist or might no longer satisfy the condition.

The task is to maximize how many such operations we can perform.

The constraint $n \le 800$ across all test cases suggests that an $O(n^3)$ or $O(n^2 \log n)$ approach is plausible, but anything involving repeated full simulation over all states is too slow because the array shrinks slowly and branching choices explode.

A naive but important failure case appears when early greedy deletions reduce future opportunities.

For example, consider:

```
1 2 3 4
```

At first, position 1 is valid, so removing (1,2) seems natural. But if we do that, we get `[3,4]`, where no position satisfies $a_i=i$, so we stop. However, the optimal move is to remove (2,3) first, leading to `[1,4]`, then no move. Either way gives only 1, but in larger constructions, premature greedy choices can destroy multiple future valid pairs.

So the structure is inherently non-greedy and depends on global interaction between deletions.

The key difficulty is that each operation removes adjacent pairs, and the validity condition depends on the _current index_, not the original one, meaning indices shift in a way that couples decisions.

## Approaches

A brute-force approach would simulate all valid moves from a given array state. At each step we scan for all indices $i$ such that $a_i = i$, try deleting $(i, i+1)$, and recurse. This explores a large state space of sequences.

Even though each state reduces size by 2, the branching factor can still be linear, and the number of distinct states becomes exponential because different deletion orders produce different arrays. This is far too large for $n \le 800$.

The key observation is that we never care about the _values_ except through equality with current positions. The operation structure suggests a DP over intervals: once we decide which pairs are removed inside a segment, everything outside remains independent.

We reinterpret the problem as matching deletions: each operation consumes two adjacent alive positions, and the left element must be a “fixed point” at that moment.

Instead of tracking actual shifting arrays, we encode whether a segment can be reduced to a smaller segment by performing a number of valid deletions inside it. The important structural insight is that after removing a pair, the remaining parts concatenate, so optimal substructure holds over intervals.

We define DP on intervals where we track the maximum number of deletions we can achieve inside a segment while respecting that some positions align with their current indices after compression. The shifting index issue is handled implicitly by considering relative indices inside the segment.

This leads to a standard interval DP where transitions consider pairing a valid “fixed-point” position with its neighbor and splitting the interval into independent subproblems.

| Approach | Time Complexity | Space Complexity | Verdict |
| --- | --- | --- | --- |
| Brute Force Simulation | Exponential | O(n) | Too slow |
| Interval DP | O(n^3) | O(n^2) | Accepted |

## Algorithm Walkthrough

We solve each test case using interval dynamic programming.

We first reindex the problem conceptually: instead of worrying about global shifting, we treat DP states as operating on a compressed segment where indices are local.

1. Define a DP table `dp[l][r]` as the maximum number of operations we can perform inside the subarray from index $l$ to $r$, assuming this segment is currently contiguous.
2. Initialize all `dp[l][r] = 0` for invalid or empty segments. Segments of length 1 or 0 cannot perform any operation because an operation requires two elements.
3. For increasing segment lengths, we compute `dp[l][r]`.
4. For each segment $[l, r]$, we consider every possible position $i$ such that $l \le i < r$. We check whether $a_i$ would equal its current position inside this segment. Since we are working in relative coordinates, this translates to a condition that the element can serve as a valid left endpoint of a deletion.
5. If position $i$ is valid, we can remove $(i, i+1)$. This splits the segment into two independent parts: $[l, i-1]$ and $[i+2, r]$. We update:

$$dp[l][r] = \max(dp[l][r], dp[l][i-1] + 1 + dp[i+2][r])$$

1. The answer for the test case is `dp[1][n]`.

The key technical difficulty is correctly handling index validity. Instead of explicitly simulating shifts, we rely on the invariant that DP intervals always represent fully compacted arrays, so local indices are always consistent.

### Why it works

The algorithm relies on the fact that every operation removes exactly two adjacent elements and never creates new interactions between left and right segments. Once a pair is removed, no future operation can affect elements across that gap, so the remaining structure decomposes into independent subproblems. The DP enumerates the first operation inside each interval, and every valid sequence has a well-defined first removal, which guarantees no optimal solution is missed.

## Python Solution

```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    a = [0] + list(map(int, input().split()))
    
    dp = [[0] * (n + 2) for _ in range(n + 2)]
    
    for length in range(2, n + 1):
        for l in range(1, n - length + 2):
            r = l + length - 1
            
            best = 0
            
            for i in range(l, r):
                # i must be valid position in current segment
                # condition: element matches its local index
                if a[i] == (i - l + 1):
                    best = max(best,
                               dp[l][i - 1] + 1 + dp[i + 2][r])
            
            dp[l][r] = best
    
    print(dp[1][n])

if __name__ == "__main__":
    t = int(input())
    for _ in range(t):
        solve()
```

The DP table is the core of the solution, and the nested loops reflect increasing interval sizes. The most delicate part is the validity condition `a[i] == (i - l + 1)`, which ensures we are comparing against the element's position inside the current compressed segment rather than its original index.

We also carefully split the interval into `[l, i-1]` and `[i+2, r]`, since both removed elements disappear.

This demonstrates that optimal deletions are not independent locally and must be combined across disjoint segments.

## Edge Cases

One important edge case is when a valid position exists but using it immediately prevents later valid operations. For instance, arrays where multiple disjoint fixed points exist can tempt a greedy solution.

Take:

```
3
1 2 3
```

The algorithm only allows removing `(1,2)` first, leaving `[3]`. The DP correctly evaluates this as 1 because it explores all interval splits rather than committing to a sequence.

Another edge case is when validity depends on shifting. Because we always evaluate `a[i] == i - l + 1`, the DP ensures correctness under compression. Without this adjustment, positions would be misclassified and overcounted.

Finally, singleton segments and boundaries are naturally handled because intervals shorter than 2 contribute zero, preventing invalid transitions from leaking into the result.
