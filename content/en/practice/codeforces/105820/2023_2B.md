## Solution
## Problem Understanding

We are given a string of lowercase letters. A string is called nearly palindromic if we can reorder its characters so that the resulting string reads the same from left to right and right to left. The task is not about the current order of characters. Only the frequencies of characters matter.

A palindrome can be built from a multiset of characters when every character count is even, except that for an odd length palindrome exactly one character is allowed to have an odd count. So a string is not nearly palindromic exactly when at least two different characters appear an odd number of times.

For every query describing a substring of the original string, we need the maximum length of a smaller substring inside it that has at least two characters with odd frequencies. If every substring is nearly palindromic, the answer is zero.

The input size reaches 200000 for both the string length and the number of queries. This rules out checking every substring. A single query can have quadratic many substrings, and across all queries that would be far beyond the available time. We need to answer each query using only logarithmic or near logarithmic work.

The useful transformation is to represent the parity of character counts as a bitmask. The 26 bits correspond to the 26 letters. When a character appears, its bit is flipped. For a substring, the xor of two prefix masks gives the parity mask of that substring. The substring is bad exactly when this xor has at least two set bits.

A common mistake is to only check whether the whole queried substring is bad. For example:

```
Input:
1
aabaaaba
1
4 4
```

The substring is just `"a"`. It is nearly palindromic because one character can always form a palindrome. The answer is `0`. A solution that checks only global frequency properties without considering shorter substrings would fail here.

Another edge case is when the whole substring is nearly palindromic but a smaller part is not.

```
Input:
8
aabaaaba
1
1 8
```

The whole string has a valid parity pattern, but the substring `"aabaaa"` inside it has two odd characters. The correct answer is `6`, not `0`. We need the longest bad substring, not only whether the entire interval is bad.

## Approaches

The brute force idea is straightforward. For every possible substring inside a query range, count the letters and check how many have odd frequency. If at least two exist, update the answer. This is correct because it directly tests the definition. However, a range of length `m` has `O(m^2)` substrings, and counting frequencies for each one adds another factor. For a string of length 200000, this approach is impossible.

The key observation is that frequency parity can be represented with a 26-bit mask. Let `pref[i]` be the parity mask of the first `i` characters. The parity mask of substring `[l, r]` is:

```
pref[r] xor pref[l - 1]
```

The problem becomes finding the maximum distance between two prefix positions inside the query range whose masks differ in at least two bits.

A segment tree can store enough information about ranges of prefix positions. For every segment, we keep the best bad substring fully inside it, the parity mask of the whole segment, and the longest prefixes and suffixes grouped by their parity masks. When two segments are merged, the only new substrings we need to consider are those crossing the middle. Their length is a suffix from the left part plus a prefix from the right part. The stored prefix and suffix information lets us compute this efficiently.

The number of possible parity states that appear in a segment is limited by its length, and the total stored states over all segment tree levels stays manageable.

| Approach | Time Complexity | Space Complexity | Verdict |
| --- | --- | --- | --- |
| Brute Force | O(n²) per query | O(1) | Too slow |
| Segment Tree with parity masks | O(26 log n) per query after preprocessing | O(n log n) | Accepted |

## Algorithm Walkthrough

1. Convert the string into parity masks. Start with mask `0`. Every time a character appears, flip the bit belonging to that character. This gives the parity state after every prefix.
2. Build a segment tree over these prefix states. Each node represents a consecutive group of prefix positions and stores:

the total xor mask of the group,

the best bad substring inside the group,

the longest prefix of every mask,

the longest suffix of every mask.

The reason we store prefixes and suffixes is that the only missing substrings after combining two children are the ones that cross their boundary.
3. To merge two nodes, first take the best answer already present in either child.
4. Then check substrings crossing the middle. A crossing substring has the form:

```
suffix of left child + prefix of right child
```

If the parity mask of the suffix and prefix have xor with at least two bits set, the combined length is a valid candidate.

1. For every query `[l, r]`, query the segment tree on the prefix range `[l - 1, r]`. The returned node contains the answer for exactly the prefixes that can form substrings inside the requested interval.

Why it works:

Every substring corresponds to two prefix positions. The parity of the substring is the xor of those two prefix masks. The segment tree always keeps the best valid pair of prefix positions inside every stored interval. When merging, all pairs are either completely inside one child or cross the boundary. The first case is already solved by the children, and the second case is explicitly checked using stored prefix and suffix information. Because every possible substring belongs to exactly one of these cases, the returned maximum is correct.
