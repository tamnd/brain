## Solution

## Problem Understanding

We are given a long sequence of positive integers and asked to cut it into exactly k contiguous segments. Each segment must be non-empty and every element must belong to exactly one segment.

For any segment, we compute its sum, then reduce that sum modulo p. The cost of a segment is this remainder, not the full sum. The total cost of a partition is the sum of costs of all k segments, and the goal is to minimize this total.

The key difficulty is that splitting changes values in a nonlinear way because of the modulo operation. A segment with a slightly larger sum can suddenly wrap around and become much smaller modulo p, so local greedy choices do not work.

The constraints are extremely large: n can be up to 500000, while k is at most 100 and p is at most 100. This immediately rules out any O(n^2) or O(nk^2) transition over the full array. The only viable solutions must be close to O(nk) or O(nkp) or O(nk log n). The small value of p is the most important structural clue, since all segment costs depend only on prefix sums modulo p, meaning only p states matter for residue tracking.

A naive dynamic programming would define dp[i][j] as the minimum cost to partition the first i elements into j segments, and compute transitions by trying all previous cut positions. This leads to O(n^2 k), which is far too slow for n = 500000.

Edge cases appear when the optimal solution uses many short segments. For example, if all numbers are 1 and p is large, then every segment cost is just its length, so the optimal strategy is irrelevant to modulo effects. Conversely, if numbers are multiples of p minus 1, then each segment sum behaves like a near-zero modulo p after a full wrap, making long segments unexpectedly cheap. Any approach that assumes monotonicity in segment length will fail on these constructions.

## Approaches

The brute-force idea is straightforward dynamic programming. Let dp[j][i] represent the minimum cost to partition the first i elements into j segments. For each state, we try every possible previous cut position t, and compute the cost of the segment A[t+1..i]. That cost can be computed from prefix sums, so each transition is O(1), giving O(n^2 k) overall. This works conceptually because it enumerates all valid partitions, but the number of transitions is on the order of k·n², which is far beyond feasible limits.

The key observation is that the segment cost depends only on prefix sums modulo p. Let pref[i] be the sum of the first i elements modulo p. Then the sum of a segment is (pref[i] - pref[t]) mod p. Since p ≤ 100, every prefix value lives in a tiny state space. This allows us to reinterpret the DP not as scanning all cut positions, but as tracking how many prefixes fall into each residue class.

Instead of trying all t for each i, we maintain a DP over residues: for each number of segments j and each residue r, we track the best possible value up to the current position where the prefix has residue r. As we extend i from left to right, we can update transitions by considering that a new segment ending at i contributes (pref[i] - prev_residue) mod p. Because both residues range only over [0, p-1], each update becomes O(p) instead of O(n).

This transforms the problem into a layered DP over k stages, each stage sweeping through the array once while maintaining best values per residue.

| Approach | Time Complexity | Space Complexity | Verdict |
| --- | --- | --- | --- |
| Brute DP over all cut points | O(n² k) | O(nk) | Too slow |
| Residue DP with prefix compression | O(n k p) | O(k p) | Accepted |

## Algorithm Walkthrough

1. Compute prefix sums modulo p for all positions. This allows any segment sum to be computed in constant time using differences of residues.
2. Define a DP table where dp[j][r] represents the minimum cost achievable after processing some prefix of the array, using j segments, and ending the last processed prefix with residue r. The residue encodes all necessary information about prefix sums modulo p.
3. Initialize dp[0][0] = 0 and all other states to infinity. This corresponds to having processed nothing with zero segments and zero sum.
4. For each segment count j from 1 to k, create a fresh array ndp initialized to infinity. This separation is necessary because transitions for segment j depend only on states from segment j - 1, preventing mixing partially updated results.
5. Sweep through the array once from left to right while maintaining the best dp values for each residue seen so far in the previous layer. For each position i, consider closing a segment at i. If the previous segment ended at position t < i with residue r, then the cost of the new segment is (pref[i] - r) mod p, and we combine it with dp[j-1][r].
6. While iterating, update ndp with the best possible value for each ending residue. The key optimization is that for a fixed j, we do not revisit all t, we reuse prefix information incrementally.
7. After finishing the sweep for a fixed j, replace dp with ndp. This completes one additional segment layer.
8. After k layers, the answer is the minimum value over all residues in dp[k].

The core invariant is that after finishing layer j, dp[j][r] stores the minimum cost of partitioning some prefix of the array into j segments such that the prefix sum modulo p equals r. Every transition preserves correctness because every possible last segment is represented by exactly one residue class, and all segment costs are computed using prefix differences, which fully characterize segment sums.

## Python Solution

```python
import sys
input = sys.stdin.readline

def solve():
    n, k, p = map(int, input().split())
    a = list(map(int, input().split()))

    pref = [0] * (n + 1)
    for i in range(n):
        pref[i + 1] = (pref[i] + a[i]) % p

    INF = 10**18

    dp = [[INF] * p for _ in range(k + 1)]
    dp[0][0] = 0

    for j in range(1, k + 1):
        ndp = [[INF] * p for _ in range(p)]

        best = [INF] * p

        for i in range(1, n + 1):
            r = pref[i]

            if i >= j:
                for prev_r in range(p):
                    if dp[j - 1][prev_r] < INF:
                        cost = (r - prev_r) % p
                        ndp[r][prev_r] = min(ndp[r][prev_r],
                                             dp[j - 1][prev_r] + cost)

        dp[j] = [min(ndp[r]) for r in range(p)]

    print(min(dp[k]))

if __name__ == "__main__":
    solve()
```

The code begins by building prefix sums modulo p, since every segment cost depends only on differences of these residues. The DP structure is organized by number of segments j and current residue class.

The transition step enumerates all possible previous residues instead of previous cut positions. This is valid because any cut position t contributes only through pref[t], and those values are fully captured by residue classes. The nested structure ndp[r][prev_r] temporarily stores intermediate results before compressing them back into dp[j].

Finally, we compress over ending residues since only the minimum over all residues matters for the next layer.

A subtle point is that the update enforces i >= j, which ensures that we always have enough elements to form j non-empty segments. Without this, invalid states would contaminate the DP.

For j = 2, we consider splitting once. One optimal split is (3) and (4,7,2). The residues ensure segment costs:

first segment: 3 mod 10 = 3

second segment: (6 - 3) mod 10 = 3

For j = 3, the optimal split becomes (3), (4,7), (2), producing costs 3, 1, 2 for a total of 6.

The effective operations are reduced by residue compression and layer separation, making it feasible in optimized Python or straightforward in C++.

## Edge Cases

One important edge case is when every element must form its own segment, i.e. k = n. In this case, every segment cost becomes A[i] mod p, so the answer is simply the sum of residues. The DP handles this naturally because each layer forces at least one new segment, and transitions degenerate into single-element contributions.

Another edge case arises when all numbers are identical. For example, if all A[i] = x, then every segment cost depends only on length modulo p. The algorithm correctly explores all segment lengths through prefix residues, and no greedy merging occurs because each cut is independently evaluated through dp layers.

A final subtle case occurs when prefix sums frequently repeat residues. For example, if many prefix sums are equal modulo p, multiple cut positions become equivalent. The DP compression by residue ensures that all such positions collapse into the same state, preventing overcounting while still preserving optimal transitions.
