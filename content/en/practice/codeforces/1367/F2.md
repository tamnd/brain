## Problem Understanding

We are given an array of integers and can repeatedly move any element to the start or the end of the array. The goal is to sort the array in non-decreasing order using as few of these moves as possible. Each move is counted individually, whether moving an element to the front or to the back. The input consists of multiple test cases, and the total number of array elements across all test cases is bounded by 200,000. This constraint indicates that any algorithm with worse than linearithmic complexity per test case will likely be too slow, since $O(n^2)$ operations would reach $4 \times 10^{10}$ steps in the worst case.

The tricky part comes from repeated elements. A naive approach might assume that simply moving the largest element to the end or the smallest element to the front in sequence would eventually sort the array, but equal values can break this assumption. For instance, given the array $[0, 1, 0, 0]$, simply moving zeros individually without considering their relative positions might result in an unnecessary move count. The correct minimum number of operations for this array is one, by moving the last zero to the front.

Other non-obvious cases include already sorted arrays, arrays sorted in reverse order, or arrays with multiple equal blocks. Handling these efficiently requires recognizing contiguous non-decreasing subsequences rather than attempting to move each misplaced element independently.

## Approaches

The brute-force approach would consider all sequences of moves and select the one with the minimal total. This works in principle because every array can eventually be sorted with at most $n$ moves, but enumerating all possible move sequences is exponentially expensive. For $n = 2 \cdot 10^5$, this is completely infeasible.

A more structured approach is to find the longest prefix of the array that can remain in place and the longest suffix that can remain in place. The key insight is that elements forming a contiguous non-decreasing subsequence from left to right (or right to left) do not need to be moved. If we can identify the largest middle block that is already sorted with respect to the target sorted array, we only need to move elements outside this block. This observation reduces the problem to identifying the minimal number of elements outside the longest prefix-suffix subsequence that matches the sorted array.

To implement this efficiently, we can iterate from the left and from the right over the sorted target array, checking which positions of the original array already align in increasing order. The number of moves is then determined by subtracting the length of this maximal aligned segment from the total array length. Handling repeated elements correctly requires using pointers that skip over duplicates, ensuring that equal values do not trigger unnecessary moves.
