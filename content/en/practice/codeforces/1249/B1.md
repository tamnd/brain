## Solution
## Problem Understanding

Each child initially holds a unique book, and each day every child passes the book they currently hold to another fixed child. The passing rule is given by a permutation, so every child has exactly one destination and every destination is used exactly once.

If we follow a single book starting from its owner, its location changes deterministically every day. Because the mapping is a permutation, repeated application of this mapping eventually brings the book back to its starting position, forming a closed cycle of children.

The task is to compute, for each starting child, how many days it takes until the book originally owned by that child returns to them for the first time. This is equivalent to finding the length of the cycle in the permutation that contains that child.

The constraints are small, with both the number of queries and the size of each permutation up to 200. This already suggests that an O(n²) or even O(n) per query solution is sufficient. Anything cubic or worse is unnecessary but still might pass due to small limits. The real goal is to exploit the structure of permutations rather than simulate day by day for each starting point independently.

A naive simulation that tracks each book independently for up to n steps per query is already borderline but still acceptable in this setting. However, a careless implementation might repeatedly simulate the same cycles for different starting nodes, duplicating work and making the solution unnecessarily slow or error-prone.

A subtle edge case appears when a node is a self-loop, meaning p[i] = i. In that case the answer is 1 immediately. Another case is a full cycle of length n, where every node shares the same answer. Any approach that assumes different starting points behave independently without sharing computed structure will waste work or risk inconsistencies.

## Approaches

A direct simulation approach would treat each child separately. Starting from a node i, we repeatedly apply the permutation and count steps until we return to i. Since the permutation length is at most n, each simulation takes O(n) time, and repeating this for all nodes leads to O(n²) per query. With n ≤ 200 and q ≤ 200, this is still feasible.

The inefficiency is that the same cycle is traversed repeatedly for every node inside it. If a cycle has length k, we recompute its structure k times, once per node.

The key observation is that the permutation decomposes the graph into disjoint cycles. Every node belongs to exactly one cycle, and every node in that cycle shares the same return time equal to the cycle length. Once we identify a cycle and its size, we can assign the answer to all nodes in it in one pass. This reduces the problem to a graph traversal where each node is visited exactly once.

| --- | --- | --- | --- |
| Brute Force Simulation | O(n²) per query | O(n) | Accepted |
| Cycle Decomposition | O(n) per query | O(n) | Accepted |

## Algorithm Walkthrough

We model the permutation as a directed graph where each index i points to p[i]. Because this is a permutation, every node has exactly one outgoing edge and one incoming edge, forming disjoint cycles.

### Steps

1. Mark all nodes as unvisited. This ensures each node is processed exactly once and belongs to exactly one cycle traversal.
2. Iterate through each node from 1 to n. If the node is already visited, skip it since it belongs to a previously processed cycle.
3. When an unvisited node is found, start walking along the permutation starting from that node, recording all nodes encountered until we return to the starting node. This walk is guaranteed to stay inside a single cycle because every node has exactly one outgoing edge.
4. Count how many nodes were visited in this traversal. This count is the cycle length.
5. Assign this cycle length as the answer for every node visited during the traversal.
6. Continue until all nodes are processed.

### Why it works

Each connected component of a permutation graph is a directed cycle. Following the mapping from any starting point never branches, so the traversal must eventually repeat a node, and because every node has in-degree and out-degree exactly one, the repetition closes a cycle without leaving it. The first time a node is revisited is when the cycle completes, so the number of steps taken equals the cycle length. Since all nodes in a cycle share the same traversal before returning to themselves, they all receive the same answer.

## Python Solution

```python
import sys
input = sys.stdin.readline

def solve():
    q = int(input())
    out = []

    for _ in range(q):
        n = int(input())
        p = [0] + list(map(int, input().split()))

        vis = [False] * (n + 1)
        ans = [0] * (n + 1)

        for i in range(1, n + 1):
            if vis[i]:
                continue

            cycle = []
            cur = i

            while not vis[cur]:
                vis[cur] = True
                cycle.append(cur)
                cur = p[cur]

            length = len(cycle)
            for v in cycle:
                ans[v] = length

        out.append(" ".join(str(ans[i]) for i in range(1, n + 1)))

    print("\n".join(out))

if __name__ == "__main__":
    solve()
```

The solution builds the permutation with 1-based indexing to match the problem statement directly, which avoids repeated index shifts. The visited array ensures each node is processed exactly once.

The inner loop follows the permutation until a visited node is reached. In a permutation, this condition coincides with completing a full cycle, so the collected nodes represent exactly one cycle. The cycle length is then assigned uniformly to all its members.

A common mistake is to restart traversal from each node without marking visited nodes globally, which leads to repeated recomputation of the same cycle. Another mistake is attempting to simulate day-by-day movement for each node independently, which is unnecessary and duplicates work.

## Edge Cases

A self-loop where p[i] = i is handled naturally because the traversal immediately returns to the starting node after visiting only itself, producing a cycle length of 1.

A permutation consisting of one large cycle is also handled correctly since the traversal starting from any node visits all nodes exactly once before returning, and all nodes receive the same length.

A mixture of cycles of different sizes is handled because each unvisited node triggers a new traversal, and visited marking ensures no cycle is processed twice.
