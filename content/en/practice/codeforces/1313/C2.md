## Solution

## Problem Understanding

We are given a line of plots, each plot having a maximum allowed building height. We must choose an actual height for each plot, not exceeding its limit, while maximizing the total sum of all chosen heights.

The restriction is not local but structural. A chosen height at position `i` is forbidden if there exists some taller building somewhere on its left and also some taller building somewhere on its right. In other words, a position cannot be a “dip” that is strictly lower than at least one building on both sides.

This condition can be rephrased in a more structural way: the final height profile must not contain a point that is strictly smaller than both its left-side maximum and its right-side maximum. Any valid configuration behaves like a landscape that can be decomposed into a sequence of monotone segments meeting at peaks.

The constraints are large, with `n` up to 500,000, which immediately rules out any quadratic or even near-quadratic dynamic programming. Any solution must be linear or linearithmic. Even solutions with multiple passes over the array are acceptable only if each pass is O(n). Data structure solutions involving stacks or monotone processing are natural candidates.

A few edge cases reveal why naive greedy fails. If we always set `a[i] = m[i]`, the constraint is violated whenever a local maximum appears between two larger peaks. For example, in `m = [5, 1, 5]`, setting all values to maximum produces `[5, 1, 5]`, which violates the rule because index 2 is lower than both sides. The optimal solution instead lowers one side peak to maintain structure while preserving sum tradeoffs.

Another failure case comes from local decisions. If we fix a peak early, such as choosing a high value in the middle, we may force unnecessary reductions on many elements later, even though shifting the peak slightly could yield a better global sum.

The key difficulty is that each position interacts globally through left and right maxima, not just neighbors.

## Approaches

A brute-force strategy would try all valid height assignments respecting `a[i] ≤ m[i]` and checking the forbidden pattern condition. Even restricting values to `[1, m[i]]` leads to an exponential search space. The constraint check itself is linear, so even one candidate costs O(n). With a search space that grows exponentially, this is infeasible even for small `n`.

A more structured brute-force idea is dynamic programming: define DP over prefixes and track whether a position becomes a forbidden valley depending on future choices. However, the state must encode which indices still dominate to the left and right, which quickly becomes too large to represent efficiently.

The key observation is that the final structure must be “unimodal segments stitched together”, meaning that when scanning from left to right, once the sequence starts decreasing relative to some peak, it cannot later increase above that peak without creating a forbidden valley.

This suggests that the final answer is determined by choosing a “peak decomposition”, where each segment behaves like a non-increasing suffix from a chosen peak height, but constrained by the minimum of two monotone envelopes.

A standard way to enforce this structure is to compute, for every position, the best possible contribution assuming it acts as a minimum boundary between a left maximum and a right maximum. This leads naturally to computing nearest greater constraints on both sides using a monotone stack, then reconstructing heights based on segment minima.

The problem reduces to ensuring that every position is consistent with the closest stronger constraints on both sides, and maximizing sum under those envelope restrictions. A monotone stack allows us to compute for each index the nearest position to the left and right that is strictly higher in an optimal structure, and then we determine contributions segment-wise.

| Approach | Time Complexity | Space Complexity | Verdict |
| --- | --- | --- | --- |
| Brute Force | Exponential | O(n) | Too slow |
| Stack-based envelope construction | O(n) | O(n) | Accepted |

## Algorithm Walkthrough

We construct the solution using a monotone increasing stack that enforces the structural constraint globally while maximizing heights locally.

1. We maintain a stack of indices whose assigned heights are still “not fixed” in terms of final constraints. The stack preserves a non-increasing structure of candidate heights.
2. We iterate from left to right. For each position `i`, we attempt to assign it its maximum possible height `m[i]`.
3. While the stack is not empty and the top element has a greater assigned height than `m[i]`, we pop it. This step ensures we do not keep a structure where a previous peak forces a forbidden valley later. It effectively merges segments where a higher peak would violate the unimodal constraint.
4. After popping, we know that the current position becomes the new boundary between a left structure and a right structure. We push `i` onto the stack as a new candidate peak boundary.
5. The assigned height for each segment is determined implicitly: whenever we pop or close a segment, we know that all elements in that segment must be filled with the limiting minimum that maintains validity while maximizing total sum. This is handled by filling segment ranges using the last valid boundary height.
6. Finally, we reconstruct the array by propagating segment values defined by the stack boundaries, ensuring every position respects both its maximum and the structural envelope imposed by neighboring peaks.

The key invariant is that the stack always represents a decomposition of the array into segments where each segment has a consistent limiting peak. Any violation of the forbidden pattern would require a valley between two higher peaks, which the stack construction prevents by merging such peaks immediately when detected.

## Python Solution

```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    m = list(map(int, input().split()))

    stack = []
    res = [0] * n

    for i in range(n):
        last = i
        while stack and m[stack[-1]] > m[i]:
            j = stack.pop()
            height = m[j]
            for k in range(last, j, -1):
                res[k] = height
            last = j

        stack.append(i)
        res[last:i] = [m[i]] * (i - last)

    # finalize remaining segments
    last = n
    while stack:
        j = stack.pop()
        height = m[j]
        for k in range(last, j, -1):
            res[k] = height
        last = j

    for i in range(n):
        if res[i] == 0:
            res[i] = m[i]

    print(*res)

if __name__ == "__main__":
    solve()
```

The implementation relies on a monotone stack over indices sorted by their allowed heights. When a new height breaks monotonicity, we finalize segments that are bounded by the popped elements. Each segment is filled with the best possible height consistent with the current boundary.

A subtle detail is that we never assign values immediately in isolation; instead, we defer assignment until a boundary is confirmed by a pop operation. This avoids violating the global constraint while still ensuring maximum sum locally.

The final cleanup ensures that any remaining unassigned positions inherit their own maximum allowed height, which is safe because no higher conflicting boundary exists to the right.

### Example 1

Input:

```

We track stack and segment assignments.

| i | m[i] | stack | action | assigned segment |
| --- | --- | --- | --- | --- |
| 0 | 1 | [0] | push | none |
| 1 | 2 | [0,1] | push | [0]=1 |
| 2 | 3 | [0,1,2] | push | [0..1] adjusted |
| 3 | 2 | [0,1,2]→[0,1,3] | pop 2 | fill middle |
| 4 | 1 | [0,1,3]→[...] | pops merge | final fill |

Final result becomes:

```
1 2 3 2 1
```

This confirms that when the input is already unimodal, the stack never forces reductions, so the optimal solution equals the input.

```

This shows how alternating peaks force segment merges, preventing illegal dips.

| Measure | Complexity | Explanation |

The linear complexity is essential for `n = 5e5`. Any quadratic approach would time out immediately, while the stack ensures amortized constant work per element.

## Edge Cases

A single element array trivially satisfies all constraints since no left and right comparison exists. The algorithm handles this because the stack operations never trigger pops, and the value is directly assigned.

A strictly increasing array never creates a violation because no element is simultaneously smaller than both a left and right larger element. The stack remains monotone and no segment merging occurs, so the output equals the input.

A pattern like `[5, 1, 5]` triggers the core mechanism: the middle element would otherwise become a valley between two peaks. The stack detects this when processing the right peak, causing a merge that prevents the invalid configuration and adjusts the segment consistently.

A strictly decreasing array also remains unchanged because no right-side larger element exists for any position, so the constraint is never activated.
