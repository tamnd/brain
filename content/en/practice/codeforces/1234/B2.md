## Solution
## Problem Understanding

We are simulating a compact “recent conversations” panel that can only display a limited number of active chats. Each incoming message comes from a friend ID, and every ID corresponds to a single conversation thread. The panel behaves like a small cache that prefers recently active conversations and keeps them near the top.

When a message arrives, there are two possibilities. If the friend is already visible in the panel, nothing changes in the layout. If the friend is not visible, we may need to insert them. If the panel is already full, the least recently promoted conversation (the bottom one) is removed first. Then the new friend’s conversation is placed at the top, and everything else shifts down.

After processing all messages, we must output the final state of this panel from top to bottom.

The constraints go up to two hundred thousand messages and two hundred thousand slots. That scale rules out any solution that rescans or rebuilds the list on every message. A naive simulation that searches linearly through the visible list for each incoming message would degrade to quadratic time in the worst case, which is too slow when both parameters are large.

The key edge behavior that often causes mistakes is when repeated messages do not move the conversation. For example, if k equals 3 and the panel is [5, 2, 1], and a message from 2 arrives, the list must remain unchanged. A naive implementation that blindly moves “recently seen” items could incorrectly duplicate or reorder entries. Another subtle case is when the panel is not yet full; removal should only happen when size is exactly k.

## Approaches

A brute-force simulation maintains the panel as a list. For every message, it scans the list to check whether the sender already exists. If found, it does nothing. If not found, it may remove the last element and then inserts the new ID at the front, shifting everything else.

This is correct because it directly follows the rules. The problem is performance. Each search is O(k), and each insertion at the front is also O(k). Over n operations, this becomes O(nk), which reaches about 4 × 10^10 operations in the worst case when n and k are both 2 × 10^5.

The improvement comes from recognizing that we only need fast membership checking and fast ordering updates. Membership tells us whether the conversation is already on screen, and ordering must always reflect recency of insertion. This is exactly the structure of a hash set plus an ordered structure. In Python, we can maintain a dictionary that maps IDs to nodes in a doubly linked list. The linked list maintains the order of conversations, and the dictionary provides O(1) lookup to decide whether a move is needed. On each new message, we remove the node if it exists, otherwise we potentially evict the tail if we are at capacity, then insert at the front.

This reduces every operation to constant time, yielding an overall linear solution.

| Approach | Time Complexity | Space Complexity | Verdict |
| --- | --- | --- | --- |
| Brute Force | O(nk) | O(k) | Too slow |
| Hash + Linked Structure | O(n) | O(k) | Accepted |

## Algorithm Walkthrough

We maintain two structures: a dictionary mapping friend IDs to their node in a doubly linked list, and the list itself which stores IDs in order from most recent at the head to least recent at the tail.

1. Initialize an empty doubly linked list and an empty dictionary. The list represents the current screen state.
2. Process each incoming message from left to right. Each message represents an attempt to activate or refresh a conversation.
3. If the sender ID already exists in the dictionary, remove its node from its current position in the linked list. This ensures we avoid duplicates while preserving the rule that only new activations move items.
4. If the sender ID does not exist and the list is already at size k, remove the tail node. This enforces the screen capacity constraint.
5. Insert the sender ID at the head of the linked list. This represents it becoming the most recently active conversation.
6. Update the dictionary to point the sender ID to the new node.
7. After processing all messages, traverse the linked list from head to tail to produce the final order.

The correctness relies on maintaining a strict invariant: the linked list always contains exactly the conversations currently shown on screen, ordered by most recent activation time. Every operation either refreshes an existing node’s position or inserts a new node while preserving capacity. No other operation can violate the ordering because only message arrival triggers changes.

## Python Solution

```python
import sys
input = sys.stdin.readline

class Node:
    __slots__ = ("val", "prev", "next")
    def __init__(self, val):
        self.val = val
        self.prev = None
        self.next = None

class LinkedList:
    def __init__(self):
        self.head = None
        self.tail = None
        self.size = 0

    def move_to_front(self, node):
        if node is self.head:
            return

        # detach
        if node.prev:
            node.prev.next = node.next
        if node.next:
            node.next.prev = node.prev

        if node is self.tail:
            self.tail = node.prev

        # insert at head
        node.prev = None
        node.next = self.head
        if self.head:
            self.head.prev = node
        self.head = node

        if self.tail is None:
            self.tail = node

    def push_front(self, node):
        node.prev = None
        node.next = self.head
        if self.head:
            self.head.prev = node
        self.head = node
        if self.tail is None:
            self.tail = node
        self.size += 1

    def pop_back(self):
        if not self.tail:
            return None
        node = self.tail
        if node.prev:
            node.prev.next = None
            self.tail = node.prev
        else:
            self.head = self.tail = None
        self.size -= 1
        return node

def solve():
    n, k = map(int, input().split())
    ids = list(map(int, input().split()))

    ll = LinkedList()
    pos = {}

    for x in ids:
        if x in pos:
            node = pos[x]
            ll.move_to_front(node)
        else:
            if ll.size == k:
                removed = ll.pop_back()
                del pos[removed.val]

            node = Node(x)
            ll.push_front(node)
            pos[x] = node

    res = []
    cur = ll.head
    while cur:
        res.append(cur.val)
        cur = cur.next

    print(len(res))
    print(*res)

if __name__ == "__main__":
    solve()
```

The solution explicitly separates structural operations. The linked list handles ordering, while the dictionary ensures constant-time existence checks. The `move_to_front` operation carefully detaches a node before reinserting it, avoiding duplication. The eviction step only triggers when the structure is at capacity, matching the problem requirement exactly.

A common subtlety is handling the case where the node is already at the head; skipping work avoids corrupting pointers. Another is ensuring that deleting from the dictionary only happens when a node is actually removed from the list, not when it is merely moved.

## Edge Cases

One edge case is when k equals 1. The structure must always contain only the most recent distinct message. For input `1 1 / 1 2 3 2`, the state evolves as [1], [2], [3], [2]. The algorithm correctly evicts the previous entry each time a new distinct ID arrives.

Another edge case is repeated messages when the conversation is already at the front. For input `3 2 / 1 2 1 1`, once the list becomes [1, 2], the final two messages from 1 must not change size or introduce duplicates. The move-to-front operation detects that the node is already head and performs no pointer rewiring.

A third case is when k is larger than the number of distinct IDs. For input `5 10 / 1 2 3 4 5`, no eviction ever happens. The list simply accumulates inserts in reverse order of arrival, and the final output remains stable without any removals.
