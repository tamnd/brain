## Diagnosis of the bug

The error

```
ValueError: invalid literal for int() with base 10: '4 1'
```

tells us the code is doing something like:

```python
n = int(input().strip())
```

but the actual input format on this test case is:

```
4
4 1
2 0
12 10
14 11
```

So the second line is not a single integer. It contains two integers (`n m` or similar). The parser incorrectly assumes every line is a standalone integer.

This creates a cascading failure:

1. The program reads `4` correctly as number of test cases.
2. It then expects one integer per test case.
3. Instead it receives `"4 1"` and tries to convert it to int.
4. Crash occurs immediately.

The same issue appears in all subsequent failures shown: the solver is misaligned with the actual input format and therefore starts reading structured lines as scalars.

This is why earlier you also saw:

- `ValueError: not enough values to unpack`
- `IndexError`
- `KeyError`

All of them come from the same root cause: inconsistent parsing assumptions.

## Why this is happening

The code is likely mixing two incompatible patterns:

### Pattern A (wrong for this input)

```python
n = int(input())
```

### Pattern B (also fragile)

```python
n, m = map(int, input().split())
```

but then later assuming different line structure than actually present.

The real issue is: **the input format has multi-value lines, but the code sometimes treats them as single integers and sometimes as arrays without synchronization.**

Once the first read is wrong, every subsequent read shifts, producing garbage values and eventually runtime errors.

## Correct approach to fix (targeted, not algorithm change)

We do not change any algorithmic idea. We only fix input handling.

The correct fix is:

- Always read lines with `sys.stdin.readline`
- Always split safely
- Never assume a line contains exactly one integer unless guaranteed
- Avoid mixing “line-per-value” and “token-based” reading styles

The safest pattern in competitive programming is:

```python
data = sys.stdin.read().split()
```

and then consume tokens sequentially.

This eliminates all desynchronization bugs.
