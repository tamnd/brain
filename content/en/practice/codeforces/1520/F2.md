## Problem Understanding

We are given a hidden array of size `n` containing only zeros and ones. The task is to answer `t` queries, each asking for the position of the `k`-th zero from the left. After we identify a zero, it flips to one for the remaining queries. The only interaction we are allowed is asking for the sum of a subarray, which tells us how many ones exist in that range. Each query to the system counts toward a strict total limit of 60,000 across all test cases.

The key challenge is that `n` can be as large as 200,000, and `t` can be up to 10,000. This immediately rules out any solution that scans the array linearly for each query. A naive approach that checks each element one by one would perform up to 200,000 operations per query, leading to billions of operations in total, far beyond the time limit.

Edge cases appear when zeros are clustered near the start or end, or when multiple queries ask for the same zero positions after some flips. For example, if the array is `[0, 0, 1, 0, 0]` and we are asked for the 4th zero, a careless linear scan could miscount because one zero may have flipped in a previous query. Similarly, querying a range incorrectly could return the wrong count of ones, producing an off-by-one error in locating the `k`-th zero.

## Approaches

A brute-force method would simulate the array explicitly. For each query, we could walk from the start of the array, counting zeros until reaching the `k`-th one. After finding it, we would mark it as one. This works correctly but is prohibitively slow: for each query, a linear scan costs O(n), so in the worst case of `t` queries, this becomes O(n * t), which is up to 2 × 10^9 operations-far too high.

The insight that leads to an efficient solution is that we can treat this problem as a dynamic search over a sequence of zeros and ones where only the sum of segments is available. Because we can query subarray sums, we can determine the number of zeros in any range as `range_length - sum_of_ones`. This transforms the problem into a classic binary search: for each query, we can repeatedly split a range into halves, querying sums until we isolate the position of the `k`-th zero. Each query requires only O(log n) sum queries rather than a full scan. To handle updates after finding a zero, we can store a dynamic data structure (a segment tree or a Fenwick tree) to keep track of how many ones have been turned on locally, so we do not count flipped zeros in subsequent queries.

This strategy reduces the worst-case query count per search from O(n) to O(log n), which is feasible under the limit of 60,000 total sum queries.

## Algorithm Walkthrough

1. **Initialize a Fenwick Tree** to track which positions have been flipped from zero to one. This allows us to query efficiently how many zeros have been converted in any prefix.
2. **For each query `k`**, set the search boundaries `l = 1` and `r = n`. Our goal is to locate the `k`-th zero by repeatedly halving the search interval.
3. **Perform binary search**. At each midpoint `m = (l + r) // 2`, query the sum of ones from the original array in the range `[1, m]` using the interactive query. Subtract the count of flipped zeros in `[1, m]` stored in the Fenwick tree to determine the number of zeros in that prefix.
4. If the number of zeros in `[1, m]` is less than `k`, move the left boundary to `m + 1`. Otherwise, move the right boundary to `m`. Repeat until `l == r`, which will be the position of the `k`-th zero.
5. **Output `l`** as the answer, flush, and update the Fenwick tree to reflect that this position has now been flipped from zero to one.
6. **Repeat** for the next query `k` using the updated Fenwick tree.

**Why it works**: Each binary search preserves the invariant that the `k`-th zero lies within `[l, r]`. The Fenwick tree correctly adjusts for prior flips, so the count of zeros is always accurate. Because the search space halves each time, the binary search isolates the correct position in O(log n) queries. Flipping positions ensures we never double-count a zero.
