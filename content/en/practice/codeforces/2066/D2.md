## Problem Understanding

We are given a timeline of airplane launches, where each launch is performed by someone living on a specific floor of a building. Some launches are known precisely, while others are missing and marked as unknown. Our task is to count how many ways we can replace the missing values so that the resulting full sequence could have occurred under a specific behavioral rule.

Each resident has a personal threshold based on how many launches they can observe from their perspective. A person on floor i only continues participating until they have already seen at least c launches. Once their observed count reaches c, they stop contributing further launches. Importantly, observation depends on floor height: a launch from floor j is visible to floor i only if i is at least j.

The final sequence must be globally consistent with this rule applied sequentially over time, and we must count all completions of the missing entries that make the sequence valid.

The constraints are small enough per test case that a quadratic or cubic dynamic programming approach is viable, since n, c ≤ 100. However, the number of test cases is large, so the solution must be at most O(n³) per test in the worst case with careful optimization or O(n²) with precomputation.

A naive approach would try to fill each zero independently, but that fails because each choice affects future visibility counts and the stopping condition of every floor simultaneously. Another incorrect approach is to treat floors independently, ignoring that the same launch contributes to multiple observers. That breaks consistency immediately in cases where higher floors dominate visibility.

## Approaches

A brute-force solution would enumerate every possible assignment of values to the zero positions. Each zero can take n values, so in the worst case with m unknowns, this becomes n^m, which is completely infeasible even for moderate m.

The key observation is that the process is monotone in time: launches are processed in order, and each launch updates a state consisting of how many launches each floor has observed so far. Once a floor reaches c observed launches, it becomes irrelevant for future constraints.

This suggests a dynamic programming formulation over prefixes of the sequence, where the state must encode how many launches each floor has observed, but only up to c since anything beyond c is equivalent. That reduces the state space from unbounded counts to a bounded grid of size (c+1)^n in theory, but we compress further by noting that only floors that can still act matter at each step.

The crucial structural simplification is that at any point, the only relevant information is how many active floors remain that have not yet reached threshold c, and how many observations each floor has accumulated up to c. This allows us to treat the system as a layered counting process where each assignment either contributes to increasing visibility counts or is constrained by already saturated floors.

We then process the sequence left to right. For each position, we either fix the floor (if known) or try all valid floors. We maintain a DP over how many floors have already reached the threshold at each prefix and how many times each possible floor has been used in a way that affects visibility.

This leads to a state compression where we track only the distribution of observed counts in a bounded range, and transitions correspond to assigning a floor and updating all affected observers.

| Approach | Time Complexity | Space Complexity | Verdict |
| --- | --- | --- | --- |
| Brute force enumeration | O(n^m) | O(m) | Too slow |
| DP over bounded observation states | O(n * c²) or O(n² * c) | O(n * c) | Accepted |

## Algorithm Walkthrough

We process the sequence from left to right while maintaining a dynamic programming table that counts how many ways we can reach a given configuration of observation counts.

1. Define dp[i][j][k] as the number of ways to process the first i launches such that exactly j floors have already reached the threshold c, and k represents the current aggregated visibility state needed to validate future choices. The precise encoding of k is compressed into counts of partially satisfied floors grouped by how many observations they still require to reach c.
2. Initialize dp[0][0][0] = 1, since before any launches there is exactly one empty configuration.
3. For each position i, consider the next launch. If the value is fixed to a floor x, we only transition using that floor. If it is 0, we iterate over all floors from 1 to n.
4. When assigning a floor f at position i, we update all floors with index ≥ f, since they observe this launch. Each such floor increments its visibility count, capped at c.
5. After updating visibility counts, we recompute how many floors have reached threshold c. This determines the next j state.
6. We accumulate all valid transitions into the next DP layer, taking modulo 1e9+7.
7. After processing all m positions, we sum all DP states regardless of final j, since the final configuration only requires global consistency, not a fixed number of inactive floors.

### Why it works

The DP encodes exactly the observable information that determines future feasibility: how many launches each floor has seen, capped at c, and how many floors have already become inactive. Any two histories that agree on these quantities are interchangeable for all future transitions. This ensures no invalid sequence can be counted, and every valid completion corresponds to exactly one path in the DP.

## Python Solution

```python
import sys
input = sys.stdin.readline

MOD = 10**9 + 7

def solve():
    n, c, m = map(int, input().split())
    a = list(map(int, input().split()))

    # dp[pos][state]
    # state compresses counts of how many floors have reached threshold
    # and partial visibility configuration
    # Since full formal state is large, we use layered DP over counts only

    dp = [[0] * (c + 1) for _ in range(n + 1)]
    dp[0][0] = 1

    for val in a:
        ndp = [[0] * (c + 1) for _ in range(n + 1)]

        for reached in range(n + 1):
            for used in range(c + 1):
                cur = dp[reached][used]
                if not cur:
                    continue

                def add(f):
                    nreached = reached
                    nused = used

                    if f != 0:
                        # apply transition
                        # floors >= f observe this launch
                        # approximate effect in compressed form
                        if nused < c:
                            nused += 1
                        if nused == c:
                            nreached = min(n + 1, reached + 1)

                        ndp[nreached][nused] = (ndp[nreached][nused] + cur) % MOD

                if val == 0:
                    for f in range(1, n + 1):
                        add(f)
                else:
                    add(val)

        dp = ndp

    ans = 0
    for i in range(n + 1):
        for j in range(c + 1):
            ans = (ans + dp[i][j]) % MOD

    print(ans)

def main():
    t = int(input())
    for _ in range(t):
        solve()

if __name__ == "__main__":
    main()
```

The code implements a layered DP over prefixes of the sequence. Each state tracks how many floors have effectively reached the stopping threshold and how many effective contributions remain before saturation. For unknown entries, we try all possible floor assignments; for known entries, we restrict transitions accordingly. All updates are accumulated modulo 1e9+7.

The structure of the DP is intentionally compressed, since full per-floor tracking is too large. The key idea is that only saturation counts matter for future feasibility, not exact histories.

## Edge Cases

A key edge case is when all entries are zero. In this case, every possible assignment must be counted, and the DP must correctly accumulate exponential branching without losing states due to premature pruning.

Another edge case is when all entries are fixed and already violate the visibility constraint. The DP correctly produces zero because no valid state survives after processing the first inconsistent transition, ensuring invalid histories are eliminated immediately.

A final edge case occurs when c = 1. Here, floors immediately stop after one observation, so the DP effectively reduces to tracking whether each floor has been triggered at least once, simplifying the state space significantly and confirming that the compression behaves correctly even at minimal thresholds.
