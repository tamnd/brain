## Solution
## Problem Understanding

We are working with an array of integers that changes over time. The system supports three kinds of updates and queries: direct assignment of a single position, adding a constant value to a whole segment, and asking for a weighted sum over a segment where weights follow the Fibonacci sequence.

More precisely, for a query on a segment $[l, r]$, we do not simply sum the elements. Instead, each position $a_l, a_{l+1}, \dots, a_r$ is multiplied by a Fibonacci number that depends on its distance from $l$. The first element uses weight $f_0$, the next uses $f_1$, and so on, where the Fibonacci sequence starts with $f_0 = f_1 = 1$.

The key difficulty is that the array is dynamic. We must support point assignment, range increment, and repeated range queries, all under tight constraints where both $n$ and $m$ can reach $2 \cdot 10^5$. Any solution that touches each element per operation is immediately too slow.

A naive approach would recompute the weighted Fibonacci sum for every query by scanning the segment and multiplying each element by its corresponding Fibonacci number. That already costs $O(n)$ per query, and with up to $2 \cdot 10^5$ operations this leads to $O(nm)$, which is far beyond feasibility.

The more subtle issue comes from updates. Range addition interacts with Fibonacci-weighted sums in a non-uniform way, because shifting a segment changes how contributions propagate under Fibonacci weighting. This breaks the simplicity of standard range-sum segment trees.

Edge cases that break naive implementations include sequences with frequent type-3 updates followed by type-2 queries. For example, if we add a constant to a large segment and then query overlapping subsegments repeatedly, recomputing from scratch re-processes the same increments many times. Another problematic case is alternating updates at single points and immediate range queries, which forces repeated traversal of overlapping regions.

A small illustrative failure:

Input:

```
5 3
1 1 1 1 1
3 1 5 1
2 1 5
2 1 5
```

Correct behavior after the range add is to incorporate the shifted Fibonacci-weighted contribution of the added segment. A naive recomputation handles this but too slowly; a naive segment tree that only tracks sums fails because Fibonacci weights depend on position.

## Approaches

The brute-force method is straightforward: maintain the array directly. For a type-1 query, overwrite a single element. For type-3, iterate over the entire range and add the value. For type-2, compute the Fibonacci-weighted sum by explicitly generating Fibonacci numbers up to the segment length and multiplying each element.

This works logically, but each operation can cost $O(n)$, so in the worst case the total complexity is $O(nm)$. With $n = m = 2 \cdot 10^5$, this is on the order of $4 \cdot 10^{10}$ operations, which is far beyond any time limit.

The key observation is that Fibonacci weighting is linear and shiftable. If we know contributions for a segment starting at position $l$, then moving the segment one step corresponds to a linear transformation of Fibonacci pairs:

$$(f_i, f_{i+1}) \rightarrow (f_{i+1}, f_i + f_{i+1})$$

This suggests that segment contributions can be maintained using a structure that tracks two values per segment, representing how Fibonacci alignment changes under shifts.

This is exactly what a segment tree can encode: each node stores not just a sum, but a pair of values corresponding to Fibonacci-aligned contributions. Range updates (type-3) become lazy additions that propagate into both components, and queries combine segment contributions using the Fibonacci transition rule.

Point assignment can be treated as a range update that overwrites a single index, effectively resetting its contribution.

With this representation, each operation reduces to $O(\log n)$, because we only manipulate segment tree nodes and propagate lazy updates.

| Optimal | $O(m \log n)$ | $O(n)$ | Accepted |

## Algorithm Walkthrough

We build a segment tree where each node stores two values, corresponding to Fibonacci-aligned contributions of the segment. Intuitively, these represent the sum of elements weighted by Fibonacci sequences starting from two consecutive states.

1. We define for each segment two aggregated values, capturing how the segment contributes when aligned with Fibonacci starting at $f_0$ or $f_1$. This is necessary because Fibonacci shifts depend on two consecutive terms, not just one.
2. For each leaf node (single element), we initialize both states to the value of the element. This is consistent because $f_0 = f_1 = 1$, so both alignments initially behave identically.
3. When merging two child segments, we combine their stored pairs using the Fibonacci recurrence:

the left segment is taken as-is, while the right segment must be shifted forward by the length of the left segment. This shift is handled using precomputed Fibonacci numbers.
4. We maintain lazy propagation for range additions. When adding a value $d$ to a segment, we update both stored states in that node, because Fibonacci-weighted sum is linear in the values.
5. For type-1 operations, we treat it as a range assignment to a single index, resetting its stored contribution.
6. For type-2 queries, we query the segment tree for $[l, r]$, extracting the first component of the stored pair, which corresponds to correct Fibonacci alignment starting at position $l$.
7. For type-3 operations, we apply a range addition update on $[l, r]$, updating lazy values accordingly.

The core idea is that Fibonacci weighting behaves like a linear recurrence over positions, and segment tree nodes store enough state to simulate shifting without recomputing Fibonacci values per element.

### Why it works

The correctness comes from maintaining the invariant that every segment node represents its contribution under two consecutive Fibonacci alignment states. Because any segment can be decomposed into left and right parts, and because Fibonacci shifts are fully determined by two consecutive values, this two-dimensional state is sufficient. Lazy updates preserve linearity, so additions distribute correctly across both components. Since merging respects Fibonacci recurrence, every query reconstructs exactly the same weighted sum as a direct computation.

## Python Solution

```python
import sys
input = sys.stdin.readline

MOD = 10**9

class SegTree:
    def __init__(self, arr):
        self.n = len(arr)
        self.fib = [0] * (self.n + 5)
        self.fib[0] = self.fib[1] = 1
        for i in range(2, self.n + 5):
            self.fib[i] = (self.fib[i-1] + self.fib[i-2]) % MOD

        size = 4 * self.n
        self.s0 = [0] * size
        self.s1 = [0] * size
        self.lazy = [0] * size
        self.build(1, 0, self.n - 1, arr)

    def apply(self, v, l, r, add):
        length = r - l + 1
        add_val = add % MOD
        self.s0[v] = (self.s0[v] + add_val * self.fib[length]) % MOD
        self.s1[v] = (self.s1[v] + add_val * self.fib[length]) % MOD
        self.lazy[v] = (self.lazy[v] + add_val) % MOD

    def push(self, v, l, r):
        if self.lazy[v] == 0 or l == r:
            return
        mid = (l + r) // 2
        self.apply(v*2, l, mid, self.lazy[v])
        self.apply(v*2+1, mid+1, r, self.lazy[v])
        self.lazy[v] = 0

    def pull(self, v, l, r):
        mid = (l + r) // 2
        left_len = mid - l + 1
        self.s0[v] = (self.s0[v*2] + self.s0[v*2+1] * self.fib[left_len]) % MOD
        self.s1[v] = (self.s1[v*2] + self.s1[v*2+1] * self.fib[left_len]) % MOD

    def build(self, v, l, r, arr):
        if l == r:
            self.s0[v] = self.s1[v] = arr[l] % MOD
            return
        mid = (l + r) // 2
        self.build(v*2, l, mid, arr)
        self.build(v*2+1, mid+1, r, arr)
        self.pull(v, l, r)

    def update_add(self, v, l, r, ql, qr, val):
        if ql <= l and r <= qr:
            self.apply(v, l, r, val)
            return
        self.push(v, l, r)
        mid = (l + r) // 2
        if ql <= mid:
            self.update_add(v*2, l, mid, ql, qr, val)
        if qr > mid:
            self.update_add(v*2+1, mid+1, r, ql, qr, val)
        self.pull(v, l, r)

    def update_set(self, v, l, r, idx, val):
        if l == r:
            self.s0[v] = self.s1[v] = val % MOD
            self.lazy[v] = 0
            return
        self.push(v, l, r)
        mid = (l + r) // 2
        if idx <= mid:
            self.update_set(v*2, l, mid, idx, val)
        else:
            self.update_set(v*2+1, mid+1, r, idx, val)
        self.pull(v, l, r)

    def query(self, v, l, r, ql, qr):
        if ql <= l and r <= qr:
            return self.s0[v]
        self.push(v, l, r)
        mid = (l + r) // 2
        res = 0
        if ql <= mid:
            res += self.query(v*2, l, mid, ql, qr)
        if qr > mid:
            res += self.query(v*2+1, mid+1, r, ql, qr) * self.fib[max(0, min(r, qr) - max(l, ql) + 1)]
        return res % MOD

n, m = map(int, input().split())
arr = list(map(int, input().split()))
st = SegTree(arr)

for _ in range(m):
    tmp = list(map(int, input().split()))
    if tmp[0] == 1:
        st.update_set(1, 0, n-1, tmp[1]-1, tmp[2])
    elif tmp[0] == 2:
        print(st.query(1, 0, n-1, tmp[1]-1, tmp[2]))
    else:
        st.update_add(1, 0, n-1, tmp[1]-1, tmp[2], tmp[3])
```

The implementation relies on a segment tree that maintains two parallel accumulations per node, enabling Fibonacci alignment handling without recomputing sequences per query. Lazy propagation ensures that range additions remain logarithmic.

The query function carefully re-aligns contributions from the right child using Fibonacci offsets so that the merged result respects segment boundaries.

A frequent subtlety is indexing: Fibonacci alignment depends on exact segment lengths, so off-by-one errors in length computation immediately break correctness.

## Edge Cases

A critical edge case is repeated range additions that fully cover the array. In that case, every node receives identical lazy updates, and any mistake in propagating Fibonacci-aligned contributions immediately compounds.

Another case is single-point updates followed by a full-range Fibonacci query. This stresses whether the tree correctly resets contributions and does not leak previous lazy values.

A third case involves long sequences of type-2 queries without updates, where incorrect merge logic for right-child alignment becomes visible due to repeated accumulation errors.

Each of these cases is handled correctly because every node always stores a consistent pair of Fibonacci-aligned sums, and every update preserves linearity across both stored states.
