## Solution
## Problem Understanding

We are given two arrays, `a` and `b`, where `a` represents items we need to remove in sequence and `b` represents thresholds for how much we can remove at each stage. We start with a pointer `k = 1` into `b`. At each step, we have two choices: we can increase `k` by one at no cost, or remove a prefix of `a` whose sum does not exceed `b[k]`, paying a cost of `m - k`. Our goal is to empty `a` with the minimum total cost, and also count the number of distinct sequences of operations that achieve this minimum.

The constraints imply that `n` and `m` can each be up to 300,000, but the total sum over all test cases of `n * m` is at most 300,000. This restricts us to algorithms that process each element roughly once or in logarithmic time. Quadratic approaches over `n` or `m` will be far too slow. Additionally, because `b` is strictly decreasing, we have a well-defined ordering of thresholds that we can leverage with techniques like two pointers or binary search.

Non-obvious edge cases include situations where the first element of `a` exceeds all elements of `b`, making it impossible to remove any prefix. Another subtlety is when multiple different prefix splits yield the same minimal cost, which affects the sequence count. For example, if `a = [3, 1]` and `b = [3, 1]`, both removing `[3]` then `[1]` and removing `[3,1]` in one step are valid and may incur different costs; we must correctly track the minimal cost and count all sequences that achieve it.

## Approaches

A brute-force approach would try every possible sequence of operations. At each step, we could either increase `k` or remove every possible prefix of `a` that fits under `b[k]`. The number of prefixes grows linearly with the remaining array length, and with up to 300,000 elements, this leads to an infeasible number of possibilities. Even memoization over `(index in a, k)` is challenging because the number of prefix sums that can be chosen is unbounded.

The key insight is that we only need to consider **maximal prefixes at each `k`**, because removing a smaller prefix and paying more steps later can never improve the minimal cost. We can precompute prefix sums of `a` and, for each `k`, determine the farthest index we can reach with a prefix sum ≤ `b[k]`. By keeping a dynamic programming array `dp[i]` that stores the minimal cost to remove the suffix starting at `i`, and a parallel array `ways[i]` for counting sequences achieving this cost, we can propagate the optimal cost and number of ways in a single backward pass over `a` using a two-pointer or binary search technique.

This reduces the problem from exponential possibilities to essentially linear time over `n` and `m` combined, by leveraging the strictly decreasing property of `b` and the monotonicity of prefix sums.

| Approach | Time Complexity | Space Complexity | Verdict |
| --- | --- | --- | --- |
| Brute Force | O(2^n * m) | O(n * m) | Too slow |
| Optimal (DP + Two Pointers) | O(n + m) | O(n) | Accepted |

## Algorithm Walkthrough

1. Compute the prefix sums of array `a` so that any sum of a contiguous prefix can be queried in O(1) time.
2. Initialize two arrays: `dp[i]` for minimal cost to remove suffix `a[i:]` and `ways[i]` for the number of sequences achieving that cost. Set `dp[n] = 0` and `ways[n] = 1` since removing an empty suffix costs nothing and has exactly one way.
3. Iterate backward over `i` from `n-1` down to `0`. For each `i`, iterate over `k` from `1` to `m` (or, equivalently, use the two-pointer approach to find the maximal `r` such that the prefix sum from `i` to `r-1` ≤ `b[k]`). This determines the farthest prefix we can remove at threshold `b[k]`.
4. For each valid `r`, compute the cost `current_cost = (m - k) + dp[r]`. If `current_cost` < `dp[i]`, update `dp[i] = current_cost` and set `ways[i] = ways[r]`. If `current_cost == dp[i]`, increment `ways[i] += ways[r]`.
5. After processing all `i`, `dp[0]` holds the minimal total cost to remove the whole array and `ways[0]` counts the number of sequences achieving it modulo 10^9 + 7.
6. If at any position no prefix can be removed with any `k`, the problem is impossible; output `-1`.

Why it works: the algorithm maintains the invariant that at every index `i`, `dp[i]` and `ways[i]` correctly represent the minimal cost and count for suffix `a[i:]`. By considering the farthest possible prefix for each `k`, we avoid suboptimal intermediate splits, ensuring minimal cost. Counting sequences correctly handles different ways of partitioning prefixes of the same sum.

## Python Solution

```python
import sys
input = sys.stdin.readline
MOD = 10**9 + 7

def solve():
    t = int(input())
    for _ in range(t):
        n, m = map(int, input().split())
        a = list(map(int, input().split()))
        b = list(map(int, input().split()))
        prefix = [0]*(n+1)
        for i in range(n):
            prefix[i+1] = prefix[i] + a[i]

        dp = [float('inf')] * (n+1)
        ways = [0] * (n+1)
        dp[n] = 0
        ways[n] = 1

        # Precompute maximal r for each i and k
        import bisect
        for i in range(n-1, -1, -1):
            max_r = i
            for k in range(m):
                # Find farthest r such that sum <= b[k]
                target = prefix[i] + b[k]
                r = bisect.bisect_right(prefix, target, lo=i+1)
                if r <= i:
                    continue
                cost = (m - (k+1)) + dp[r]
                if cost < dp[i]:
                    dp[i] = cost
                    ways[i] = ways[r]
                elif cost == dp[i]:
                    ways[i] = (ways[i] + ways[r]) % MOD

        if dp[0] == float('inf'):
            print(-1)
        else:
            print(dp[0], ways[0] % MOD)

if __name__ == "__main__":
    solve()
```

The solution computes prefix sums so that we can efficiently check any candidate prefix sum. The DP array tracks the minimal cost to clear the remaining suffix. For each index, we use binary search to find the largest removable prefix under each threshold `b[k]`. We carefully combine counts for multiple sequences leading to the same minimal cost. Off-by-one errors are avoided by maintaining prefix sums with an extra initial zero.
