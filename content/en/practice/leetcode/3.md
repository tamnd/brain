## Solution

## Problem Understanding

The problem gives us a string `s` and asks for the length of the longest substring that contains no repeated characters.

A substring is a contiguous section of the string. This detail is extremely important. Characters must appear next to each other in the original string. For example, in `"pwwkew"`, the string `"wke"` is a valid substring because those characters appear consecutively. However, `"pwke"` is not a substring because it skips characters, making it a subsequence instead.

The input is a single string that may contain letters, digits, symbols, and spaces. The output is a single integer representing the maximum length of any substring that contains only unique characters.

The constraint `0 <= s.length <= 5 * 10^4` tells us that the solution must be efficient. A quadratic solution may become too slow for large inputs. Since the string can contain up to 50,000 characters, we should aim for a linear time solution if possible.

There are several important edge cases to consider early:

- The string may be empty. In that case, the answer should be `0`.
- The string may contain only one character. The answer should then be `1`.
- All characters may be identical, such as `"bbbbb"`. The longest valid substring would contain only one character.
- Repeated characters may appear far apart, which can confuse naive implementations if they do not properly track the current valid substring.
- The string may contain spaces or symbols, not just letters, so the solution must work for any valid character.

The key challenge is efficiently tracking the current substring without repeatedly recomputing or scanning previously processed characters.

## Approaches

### Brute Force Approach

A straightforward solution is to generate every possible substring and check whether each substring contains duplicate characters.

For a string of length `n`, there are `O(n^2)` possible substrings. For each substring, we can use a set to determine whether all characters are unique. Checking uniqueness takes up to `O(n)` time in the worst case.

This produces a total time complexity of `O(n^3)` if substrings are explicitly constructed and checked character by character. With some optimizations, it can be reduced to `O(n^2)`, but even that is still too slow for strings of length 50,000.

The brute force approach is correct because it examines every possible substring and keeps track of the maximum valid length. However, it performs a large amount of repeated work. When moving from one substring to the next, most characters overlap with the previous substring, yet the algorithm recomputes uniqueness from scratch.

### Optimal Sliding Window Approach

The important observation is that when we already have a valid substring with no duplicates, we do not need to restart from scratch when extending it.

Instead, we can maintain a sliding window representing the current substring without repeating characters. As we move through the string:

- We expand the window by adding characters from the right.
- If a duplicate appears, we shrink the window from the left until the duplicate is removed.
- At every step, the window remains valid.

A hash map is ideal for this problem because it allows us to quickly determine whether a character has already appeared in the current window and where it last appeared.

This approach avoids redundant work because each character is processed at most twice, once when entering the window and once when leaving it.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) to O(n³) | O(n) | Checks many overlapping substrings repeatedly |
| Optimal Sliding Window | O(n) | O(min(n, charset)) | Maintains a valid window while scanning once |

## Algorithm Walkthrough

1. Initialize a hash map called `last_seen` to store the most recent index of each character.
2. Initialize two variables:

- `left` represents the start of the current window.
- `max_length` stores the length of the longest valid substring found so far.
3. Iterate through the string using an index `right`. This index represents the end of the current window.
4. For each character `s[right]`, check whether it already exists in the hash map and whether its last occurrence is still inside the current window.
5. If the character is inside the current window, move `left` to one position after the previous occurrence. This removes the duplicate from the window.
6. Update the character's latest position in the hash map.
7. Compute the current window length using:

`right - left + 1`
8. Update `max_length` if the current window is larger than any previously seen window.
9. Continue until the entire string has been processed.

The reason this works efficiently is that the `left` pointer only moves forward. No character is revisited unnecessarily. The hash map provides constant time lookups for previous character positions, making the overall algorithm linear.

## Python Solution

```
class Solution:
    def lengthOfLongestSubstring(self, s: str) -> int:
        last_seen: dict[str, int] = {}
        
        left = 0
        max_length = 0
        
        for right, char in enumerate(s):
            if char in last_seen and last_seen[char] >= left:
                left = last_seen[char] + 1
            
            last_seen[char] = right
            
            current_length = right - left + 1
            max_length = max(max_length, current_length)
        
        return max_length
```

The implementation begins by creating a dictionary called `last_seen`. This dictionary maps each character to the most recent index where it appeared.

The variable `left` tracks the beginning of the current valid substring. Initially, the window is empty, so `left` starts at `0`.

The loop iterates through the string using `enumerate`, which provides both the index `right` and the current character `char`.

When a character has already appeared inside the current window, the window becomes invalid because it now contains duplicates. To restore validity, `left` moves to one position after the previous occurrence of that character.

The condition `last_seen[char] >= left` is very important. It ensures that only duplicates inside the current window matter. A character that appeared before the current window started does not affect the current substring.

After updating the window, the algorithm stores the current character's latest index in `last_seen`.

The current window length is calculated using `right - left + 1`, and `max_length` is updated whenever a larger valid substring is found.

Finally, the function returns the largest substring length discovered during the scan.

## Go Solution

```
func lengthOfLongestSubstring(s string) int {
    lastSeen := make(map[byte]int)

    left := 0
    maxLength := 0

    for right := 0; right < len(s); right++ {
        char := s[right]

        if prevIndex, exists := lastSeen[char]; exists && prevIndex >= left {
            left = prevIndex + 1
        }

        lastSeen[char] = right

        currentLength := right - left + 1

        if currentLength > maxLength {
            maxLength = currentLength
        }
    }

    return maxLength
}
```

The Go implementation follows the same logic as the Python version. The main difference is that Go strings are indexed as byte arrays. Since the problem statement limits input to English letters, digits, symbols, and spaces, treating characters as bytes is safe here.

The hash map is implemented using `map[byte]int`, where each byte character maps to its latest index.

Go does not have Python's `enumerate`, so the loop explicitly uses integer indices.

The solution does not require special handling for empty strings because the loop naturally executes zero times and returns `0`.

There are no integer overflow concerns because the maximum string length is only 50,000, well within Go's integer range.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each character is processed at most twice |
| Space | O(min(n, charset)) | Hash map stores unique characters currently seen |

The algorithm runs in linear time because both pointers only move forward. The `right` pointer visits each character once, and the `left` pointer never moves backward.

The space complexity depends on the number of unique characters stored in the hash map. In the worst case, all characters are distinct, so the map stores up to `n` entries. If the character set is fixed and small, the practical space usage is bounded by the character set size.

## Edge Cases

An empty string is an important edge case because some implementations accidentally assume at least one character exists. In this solution, the loop never executes for an empty string, so `max_length` remains `0`, which is the correct answer.

A string containing all identical characters can easily break incorrect sliding window implementations. For example, `"aaaaa"` repeatedly forces the left boundary to move forward. The algorithm correctly maintains a window of size `1` throughout execution because duplicates are immediately removed from the current window.

A string where duplicates appear after a long gap is another common source of bugs. Consider `"abba"`. When processing the final `'a'`, the earlier `'a'` is no longer inside the current window. Incorrect implementations may move `left` backward, which would corrupt the window. The condition `last_seen[char] >= left` prevents this issue by ensuring only duplicates inside the active window affect the result.

Strings containing spaces or symbols are also important because the problem does not restrict input to alphabetic characters. Since the solution uses a hash map keyed by characters, it works uniformly for letters, digits, spaces, and symbols without requiring any special handling.
