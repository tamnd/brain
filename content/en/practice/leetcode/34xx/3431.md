## Solution

## Problem Understanding

This problem asks us to determine the minimum number of indices we must unlock in a binary array `locked` to make a small integer array `nums` sortable using a very specific set of allowed swaps. The array `nums` only contains integers from 1 to 3, and `locked[i]` tells us whether the element at position `i` is fixed (`1`) or can be swapped (`0`). The allowed swaps are strictly adjacent swaps where the left element is exactly one larger than the right element (`nums[i] - nums[i + 1] == 1`), and the left index must be unlocked (`locked[i] == 0`).

The input consists of two arrays of the same length, up to `10^5` elements. The output is a single integer, representing the minimal number of indices we must unlock to make `nums` sortable according to these swap rules. If sorting is impossible even after unlocking every index, the function should return `-1`.

Edge cases that could trip a naive implementation include arrays that already require no unlocks, arrays where all elements are unlocked but unsortable due to the arrangement of numbers, and arrays where only a few specific locked positions prevent sorting. The problem guarantees valid input lengths and values, but does not guarantee that sorting is initially possible without unlocking.

## Approaches

A brute-force approach would try every combination of unlocking indices and performing all possible allowed swaps until the array is sorted. One could perform recursive backtracking, exploring every subset of indices to unlock, and simulate the sorting process with allowed swaps. While this guarantees a correct answer, it is computationally infeasible for large arrays (`O(2^n)` for unlocking possibilities times `O(n)` for swaps) and would time out quickly.

The key insight for an optimal solution comes from recognizing that only local inversions (where `nums[i] > nums[i + 1]`) block sorting, and the problem is constrained to three possible numbers. For each contiguous segment of numbers, we can analyze whether the locked positions force the segment to be unsortable. In other words, we can count how many indices must be unlocked to remove "blocking inversions" in a left-to-right pass. We also observe that a segment containing all the same number or sorted in ascending order does not require unlocking. Only positions where a larger number is left of a smaller one and the swap is locked need attention. This reduces the problem to linear analysis of segments, counting necessary unlocks.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n * n) | O(n) | Tries all subsets of unlocked indices and simulates swaps |
| Optimal | O(n) | O(1) | Linear pass to count blocking inversions and minimum unlocks, leverages the 1-2-3 value constraint |

## Algorithm Walkthrough

1. Start by scanning `nums` from left to right, comparing each pair `nums[i]` and `nums[i + 1]`. Identify all adjacent inversions where `nums[i] > nums[i + 1]`.
2. For each inversion, check whether it is immediately swappable (`nums[i] - nums[i + 1] == 1` and `locked[i] == 0`). If yes, no unlocking is needed.
3. If the inversion cannot be swapped due to `locked[i] == 1`, mark this position as a required unlock. Increment a counter of unlock operations.
4. Repeat this process across all segments of consecutive inversions, since unlocking one index may allow multiple swaps in a chain.
5. After scanning the entire array, check if the array is sortable with the proposed unlocks. If any inversion cannot be resolved even after unlocking all candidates (for example, `nums[i] - nums[i + 1] > 1`), return `-1`.
6. Return the total number of unlocks identified.

**Why it works:** The algorithm works because any unsortable inversion in a 1-2-3 array must involve a local difference of 1 or 2. Swaps only remove differences of 1, and a locked position prevents that swap. By counting the minimal positions to unlock to eliminate all 1-difference inversions, we ensure that all possible allowed swaps can occur. If a difference of 2 exists and cannot be resolved with any swaps, the array is unsortable.

## Python Solution

```python
from typing import List

class Solution:
    def minUnlockedIndices(self, nums: List[int], locked: List[int]) -> int:
        n = len(nums)
        unlocks_needed = 0
        
        for i in range(n - 1):
            if nums[i] > nums[i + 1]:
                diff = nums[i] - nums[i + 1]
                if diff > 1:
                    return -1
                if locked[i] == 1:
                    unlocks_needed += 1
        return unlocks_needed
```

This implementation iterates through `nums`, comparing each adjacent pair. If a left element is larger than the right, it checks the difference. Differences greater than 1 are impossible to fix with allowed swaps, immediately returning `-1`. If the difference is exactly 1 and the left index is locked, we increment our counter of unlock operations. At the end, the counter represents the minimal number of unlocks needed.

## Go Solution

```go
func minUnlockedIndices(nums []int, locked []int) int {
    n := len(nums)
    unlocksNeeded := 0

    for i := 0; i < n-1; i++ {
        if nums[i] > nums[i+1] {
            diff := nums[i] - nums[i+1]
            if diff > 1 {
                return -1
            }
            if locked[i] == 1 {
                unlocksNeeded++
            }
        }
    }
    return unlocksNeeded
}
```

The Go implementation mirrors the Python logic, using a simple for loop and conditional checks. The main difference is syntax: Go uses `int` for the counter, explicit loop bounds, and array indexing.
