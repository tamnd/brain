## Solution

## Problem Understanding

The problem gives us an array `maxHeights`, where `maxHeights[i]` represents the maximum allowed height for the tower at index `i`. We must construct another array called `heights` such that every tower height satisfies:

- `1 <= heights[i] <= maxHeights[i]`

Additionally, the final `heights` array must form a mountain array.

A mountain array means there exists some peak index `p` such that:

- Moving from left to right up to `p`, heights never decrease.
- Moving from `p` to the end, heights never increase.

In other words, the sequence first goes upward or stays flat, then goes downward or stays flat.

Our goal is to maximize the total sum of all tower heights while respecting both the maximum height constraints and the mountain property.

The important observation is that the chosen peak determines the structure of the entire array. Once a peak is fixed, every tower on the left side can be at most the minimum of:

- its own `maxHeights[i]`
- the height immediately to its right

Similarly, every tower on the right side can be at most the minimum of:

- its own `maxHeights[i]`
- the height immediately to its left

This creates a constrained propagation effect outward from the peak.

The constraints are large:

- `n` can be up to `100000`
- `maxHeights[i]` can be as large as `10^9`

These limits immediately rule out quadratic solutions. Any algorithm with `O(n^2)` complexity will time out for the largest inputs. This strongly suggests that we need a linear or near linear solution.

Several edge cases are important:

- Arrays with a single element.
- Strictly increasing arrays.
- Strictly decreasing arrays.
- Arrays with repeated values.
- Cases where the optimal peak is not the maximum element.
- Very large values that require 64-bit integer arithmetic.

Because sums can exceed the range of a 32-bit integer, we must use `int64` in Go and rely on Python's arbitrary precision integers.

## Approaches

### Brute Force Approach

The most straightforward idea is to try every index as the mountain peak.

For a chosen peak `p`:

- Set `heights[p] = maxHeights[p]`
- Expand leftward:

- `heights[i] = min(maxHeights[i], heights[i + 1])`
- Expand rightward:

- `heights[i] = min(maxHeights[i], heights[i - 1])`

This produces the maximum valid mountain for that specific peak because every tower is made as tall as possible while maintaining monotonicity.

After constructing the full mountain, compute its sum. Repeat for every peak and take the maximum.

This approach is correct because for a fixed peak, greedily assigning the largest possible value at each position maximizes the total sum.

However, it is too slow.

For each peak, we may scan the entire array, leading to `O(n)` work per peak. Since there are `n` possible peaks, the total complexity becomes `O(n^2)`.

With `n = 100000`, this is infeasible.

### Optimal Approach

The key insight is that the contribution of each side of the mountain can be computed efficiently using a monotonic stack.

Suppose we want:

- `left[i]` = maximum possible sum of a valid nondecreasing sequence ending at `i`
- `right[i]` = maximum possible sum of a valid nonincreasing sequence starting at `i`

Then if `i` is chosen as the peak:

```
total = left[i] + right[i] - maxHeights[i]
```

We subtract `maxHeights[i]` once because it gets counted in both arrays.

The difficult part is computing `left` and `right` efficiently.

A monotonic increasing stack helps determine how far a height can extend before a smaller height blocks it. This transforms repeated minimum computations into amortized `O(1)` operations.

The technique is similar to problems involving histogram spans and subarray minimums.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Construct mountain separately for every peak |
| Optimal | O(n) | O(n) | Uses monotonic stacks to compute contributions efficiently |

## Algorithm Walkthrough

### Step 1: Compute Left Contributions

We define `left[i]` as the maximum total sum for the subarray `[0...i]` if index `i` is the highest point on the right boundary.

We process the array from left to right using a monotonic increasing stack.

The stack stores indices whose heights are increasing.

For each index `i`:

1. Pop elements from the stack while their height is greater than `maxHeights[i]`.
2. After popping:

- If the stack is empty:

- Every position from `0` to `i` can use height `maxHeights[i]`
- `left[i] = maxHeights[i] * (i + 1)`
- Otherwise:

- Let `prev` be the stack top.
- Positions after `prev` can use `maxHeights[i]`
- `left[i] = left[prev] + maxHeights[i] * (i - prev)`
3. Push `i` onto the stack.

This efficiently computes the optimal nondecreasing prefix sums.

### Step 2: Compute Right Contributions

Now compute `right[i]`, the maximum total sum for subarray `[i...n-1]` assuming `i` is the highest point on the left boundary.

This is symmetric to the previous computation.

We traverse from right to left using another monotonic increasing stack.

For each index `i`:

1. Pop elements while their height is greater than `maxHeights[i]`.
2. After popping:

- If the stack is empty:

- Every position from `i` to `n-1` can use `maxHeights[i]`
- `right[i] = maxHeights[i] * (n - i)`
- Otherwise:

- Let `next` be the stack top.
- `right[i] = right[next] + maxHeights[i] * (next - i)`
3. Push `i`.

### Step 3: Evaluate Every Peak

For each index `i`:

```
candidate = left[i] + right[i] - maxHeights[i]
```

Track the maximum candidate.

### Step 4: Return the Maximum Sum

The largest candidate value is the answer.

### Why it works

For a fixed peak, the optimal mountain is uniquely determined by greedily limiting neighboring heights using the minimum constraint. The monotonic stack efficiently groups ranges where the same limiting height applies.

The stack invariant guarantees that indices are maintained in increasing height order. This allows each element to find the nearest smaller boundary in amortized constant time. Since every index is pushed and popped at most once, the total runtime remains linear.

## Python Solution

```python
from typing import List

class Solution:
    def maximumSumOfHeights(self, maxHeights: List[int]) -> int:
        n = len(maxHeights)

        left = [0] * n
        stack = []

        for i in range(n):
            while stack and maxHeights[stack[-1]] > maxHeights[i]:
                stack.pop()

            if not stack:
                left[i] = maxHeights[i] * (i + 1)
            else:
                prev = stack[-1]
                left[i] = left[prev] + maxHeights[i] * (i - prev)

            stack.append(i)

        right = [0] * n
        stack = []

        for i in range(n - 1, -1, -1):
            while stack and maxHeights[stack[-1]] > maxHeights[i]:
                stack.pop()

            if not stack:
                right[i] = maxHeights[i] * (n - i)
            else:
                nxt = stack[-1]
                right[i] = right[nxt] + maxHeights[i] * (nxt - i)

            stack.append(i)

        answer = 0

        for i in range(n):
            total = left[i] + right[i] - maxHeights[i]
            answer = max(answer, total)

        return answer
```

The implementation follows the exact algorithm described earlier.

The `left` array stores the best possible sum for prefixes ending at each index. The monotonic stack ensures that when we process a new height, we quickly find the nearest previous height that is less than or equal to it.

If no such element exists, the current height becomes the limiting height for the entire prefix.

Otherwise, the current height only affects the segment after the nearest smaller element. Everything before that has already been optimally computed.

The `right` array is computed symmetrically from right to left.

Finally, every index is considered as the mountain peak. Since both arrays include the peak value, we subtract it once to avoid double counting.

## Go Solution

```go
func maximumSumOfHeights(maxHeights []int) int64 {
	n := len(maxHeights)

	left := make([]int64, n)
	stack := []int{}

	for i := 0; i < n; i++ {
		for len(stack) > 0 && maxHeights[stack[len(stack)-1]] > maxHeights[i] {
			stack = stack[:len(stack)-1]
		}

		if len(stack) == 0 {
			left[i] = int64(maxHeights[i]) * int64(i+1)
		} else {
			prev := stack[len(stack)-1]
			left[i] = left[prev] + int64(maxHeights[i])*int64(i-prev)
		}

		stack = append(stack, i)
	}

	right := make([]int64, n)
	stack = []int{}

	for i := n - 1; i >= 0; i-- {
		for len(stack) > 0 && maxHeights[stack[len(stack)-1]] > maxHeights[i] {
			stack = stack[:len(stack)-1]
		}

		if len(stack) == 0 {
			right[i] = int64(maxHeights[i]) * int64(n-i)
		} else {
			next := stack[len(stack)-1]
			right[i] = right[next] + int64(maxHeights[i])*int64(next-i)
		}

		stack = append(stack, i)
	}

	var answer int64 = 0

	for i := 0; i < n; i++ {
		total := left[i] + right[i] - int64(maxHeights[i])

		if total > answer {
			answer = total
		}
	}

	return answer
}
```

The Go implementation is structurally identical to the Python solution, but several language-specific details are important.

The answer can exceed 32-bit integer limits, so all accumulated sums use `int64`.

Slices are used for both stacks and DP arrays. Stack popping is implemented using slice truncation:

```go
stack = stack[:len(stack)-1]
```

Unlike Python, Go does not support arbitrary precision integers automatically, so explicit casting to `int64` is necessary during multiplication.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every index is pushed and popped from the stack at most once |
| Space | O(n) | Arrays and stacks store up to n elements |

The algorithm is linear because each stack operation is amortized constant time. Although the code contains nested `while` loops, every element can only be removed from the stack once. Therefore, the total number of stack operations across the entire algorithm is proportional to `n`.

## Edge Cases

### Single Element Array

When the array contains only one element, that tower is automatically the mountain peak. A buggy implementation might incorrectly assume both left and right neighbors exist.

The implementation handles this naturally because both stack passes still work correctly for `n = 1`, and the final formula becomes:

```
left[0] + right[0] - maxHeights[0]
```

which simplifies to the correct value.

### Strictly Increasing or Strictly Decreasing Arrays

These cases are important because the optimal peak lies at one end of the array.

For example:

```
[1,2,3,4,5]
```

The best peak is the last element.

Similarly:

```
[5,4,3,2,1]
```

The best peak is the first element.

Some incorrect solutions assume the peak must lie in the middle. The monotonic stack approach evaluates every index independently, so endpoint peaks are handled correctly.

### Duplicate Heights and Plateaus

Arrays such as:

```
[3,2,5,5,2,3]
```

contain multiple valid peak positions.

Handling equal values incorrectly in the stack condition can lead to overcounting or broken spans. The implementation uses:

```
while stack and maxHeights[stack[-1]] > maxHeights[i]:
```

Notice the strict `>` comparison rather than `>=`.

This preserves equal values in the stack and correctly supports plateau-shaped mountains.
