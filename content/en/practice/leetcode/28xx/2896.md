## Solution

## Problem Understanding

The problem gives us two binary strings, `s1` and `s2`, both of the same length `n`. Our goal is to transform `s1` into `s2` using the minimum possible cost.

We are allowed to perform two kinds of operations on `s1`.

The first operation lets us choose any two indices `i` and `j`, then flip both bits at those positions. This operation costs `x`.

The second operation lets us choose adjacent indices `i` and `i + 1`, then flip both bits together. This operation costs `1`.

A flip changes `'0'` into `'1'` and `'1'` into `'0'`.

The important observation is that every operation flips exactly two bits. That means the parity of the number of mismatched positions never changes. If the number of mismatches between `s1` and `s2` is odd, then it is impossible to make the strings equal, because we can only fix mismatches in pairs.

The constraints are relatively small:

- `1 <= n <= 500`
- `1 <= x <= 500`

Since `n` is only 500, an `O(n^2)` dynamic programming solution is completely acceptable. However, brute force exploration of all operation sequences would still be far too expensive because the state space grows exponentially.

The key challenge is deciding when to use:

- the expensive but flexible operation with cost `x`
- the cheap adjacent operation with cost `1`

Adjacent flips can effectively move mismatches toward each other at a cumulative cost equal to the distance between them. Sometimes this is cheaper than paying `x`, and sometimes it is not.

Several edge cases are important:

- If there are no mismatches, the answer is `0`.
- If the mismatch count is odd, the answer is `-1`.
- If two mismatches are close together, repeated adjacent operations may be cheaper than using the global operation.
- If mismatches are far apart and `x` is small, the global operation may be optimal.
- When `x = 1`, the first operation becomes extremely powerful because any pair can be fixed at minimal cost.

## Approaches

### Brute Force Approach

A brute force solution would model every possible string reachable from `s1` and attempt all valid operations recursively or using BFS/Dijkstra.

At every step, we could:

- choose any pair `(i, j)` and apply the first operation
- choose any adjacent pair `(i, i + 1)` and apply the second operation

We would continue exploring states until reaching `s2`.

This approach is correct because it explicitly searches all possible operation sequences and therefore eventually finds the minimum cost.

However, it is computationally infeasible. A binary string of length `n` has `2^n` possible states. Even for moderate values of `n`, the number of reachable configurations becomes enormous. Since `n` can be 500, this approach is completely impractical.

### Optimal Dynamic Programming Approach

The crucial insight is that only mismatched positions matter.

Suppose we collect all indices where:

```
s1[i] != s2[i]
```

Call this array `positions`.

Every operation flips two bits, so we always resolve mismatches in pairs.

Now consider two mismatches at indices `a` and `b`.

We have two possible ways to fix them:

1. Use the first operation directly, cost `x`
2. Use adjacent swaps repeatedly to move the mismatch across the interval, cost `b - a`

Therefore, the effective cost of pairing these mismatches is:

```
min(x, b - a)
```

This transforms the problem into a pairing DP problem.

We process mismatches from left to right and decide how to pair them optimally.

The standard DP formulation is:

- `dp[i]` = minimum cost to fix mismatches starting from index `i` in the mismatch array

Then we pair mismatch `i` with mismatch `i + 1`.

Because optimal pairing in ordered intervals is non crossing, adjacent pairing in the mismatch list is sufficient.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Explores all reachable string states |
| Optimal Dynamic Programming | O(m) | O(m) | Uses mismatch pairing, where `m` is mismatch count |

Here, `m` is the number of mismatched positions.

## Algorithm Walkthrough

1. Traverse both strings simultaneously and collect all indices where the characters differ.

These are the only positions we actually need to fix.
2. Let the mismatch indices be stored in an array called `positions`.

For example:

```
s1 = 1100011000
s2 = 0101001010
```

Mismatch positions become:

```
[0, 3, 5, 8]
```
3. If the number of mismatches is odd, return `-1`.

Every operation flips exactly two bits, so mismatches can only disappear in pairs.
4. Define dynamic programming over the mismatch array.

Let:

```
dp[i]
```

represent the minimum cost needed to resolve mismatches starting from index `i`.
5. The transition pairs mismatch `i` with mismatch `i + 1`.

Suppose:

```
distance = positions[i + 1] - positions[i]
```

We can fix this pair either:

- directly using the first operation with cost `x`
- using adjacent operations repeatedly with cost `distance`

Therefore:

```
pair_cost = min(x, distance)
```
6. The recurrence becomes:

```
dp[i] = pair_cost + dp[i + 2]
```
7. Initialize:

```
dp[m] = 0
```

because no mismatches remain.
8. Fill the DP array backward.
9. Return `dp[0]`.

### Why it works

The important property is that mismatch pairings in an optimal solution never need to cross.

If we pair mismatches out of order, we can always rearrange the pairings into non crossing adjacent pairings without increasing cost.

Since each operation fixes exactly two mismatches, and the cost only depends on the pair distance or fixed value `x`, dynamic programming over adjacent mismatch pairs guarantees the minimum total cost.

## Python Solution

```python
class Solution:
    def minOperations(self, s1: str, s2: str, x: int) -> int:
        positions = []

        for i in range(len(s1)):
            if s1[i] != s2[i]:
                positions.append(i)

        m = len(positions)

        if m % 2 == 1:
            return -1

        if m == 0:
            return 0

        dp = [0] * (m + 1)

        for i in range(m - 2, -1, -2):
            distance = positions[i + 1] - positions[i]
            pair_cost = min(x, distance)

            dp[i] = pair_cost + dp[i + 2]

        return dp[0]
```

The implementation begins by collecting all mismatch positions into the `positions` array. Since only mismatches matter, this reduces the problem size significantly.

Next, we immediately check whether the mismatch count is odd. If it is, the answer is impossible because every operation flips exactly two bits.

The dynamic programming array stores the minimum cost to resolve mismatches from a given point onward. Since we always process mismatches in pairs, we iterate backward in steps of two.

For every adjacent mismatch pair, we compute:

```python
min(x, distance)
```

This captures the cheaper option between:

- directly fixing the pair with the global operation
- resolving it using repeated adjacent operations

Finally, `dp[0]` contains the minimum total cost.

## Go Solution

```go
func minOperations(s1 string, s2 string, x int) int {
    positions := []int{}

    for i := 0; i < len(s1); i++ {
        if s1[i] != s2[i] {
            positions = append(positions, i)
        }
    }

    m := len(positions)

    if m%2 == 1 {
        return -1
    }

    if m == 0 {
        return 0
    }

    dp := make([]int, m+1)

    for i := m - 2; i >= 0; i -= 2 {
        distance := positions[i+1] - positions[i]

        pairCost := x
        if distance < pairCost {
            pairCost = distance
        }

        dp[i] = pairCost + dp[i+2]
    }

    return dp[0]
}
```

The Go implementation follows the exact same logic as the Python version.

Slices are used instead of Python lists, and the minimum comparison is written manually because Go does not provide a built in integer `min` function.

Since all values are small and bounded by the constraints, integer overflow is not a concern.

## Edge Cases

One important edge case occurs when the strings are already equal. In this situation, the mismatch array is empty. A buggy implementation might still attempt to run dynamic programming or access invalid indices. The implementation handles this explicitly by returning `0` immediately when there are no mismatches.

Another important case is when the number of mismatches is odd. Since every allowed operation flips exactly two bits, parity never changes. If the mismatch count is odd, the transformation is impossible. The implementation detects this early and returns `-1` before performing any DP work.

A third tricky case happens when two mismatches are very far apart. Using adjacent operations repeatedly may become extremely expensive because the cost grows with distance. The implementation correctly compares the distance cost against `x` and always chooses the cheaper option.

A final subtle case occurs when `x` is very small, especially `x = 1`. In that scenario, the global operation becomes extremely efficient because any pair of mismatches can be fixed at constant cost regardless of distance. The algorithm naturally handles this through the `min(x, distance)` transition.
