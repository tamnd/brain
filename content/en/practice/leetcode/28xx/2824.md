## Solution

## Problem Understanding

The problem gives us an integer array `nums` and an integer `target`. We must count how many index pairs `(i, j)` satisfy two conditions:

1. `i < j`, meaning we only consider unique unordered pairs.
2. `nums[i] + nums[j] < target`, meaning the sum of the two selected values must be strictly smaller than the target value.

The output is simply the number of valid pairs.

For example, if `nums = [-1,1,2,3,1]` and `target = 2`, we examine every pair of positions and count only those whose sum is less than `2`.

The array length is at most `50`, which is very small. This means even an `O(n²)` solution is acceptable in practice. However, the problem is categorized under sorting and two pointers, so it is useful to understand both the straightforward brute-force solution and the more optimized pattern.

The values inside the array can be negative, positive, or zero. This matters because sorting and comparison logic must work correctly across mixed signs.

Several edge cases are important:

- Arrays with only one element cannot form any pair.
- Arrays where no pair satisfies the condition should return `0`.
- Arrays where every pair satisfies the condition should count all possible combinations.
- Duplicate numbers must still be treated as separate indices.
- Negative values can create many valid pairs even when the target is also negative.

The problem guarantees:

- `1 <= n <= 50`
- All values are small integers
- The input is always valid

Because the constraints are tiny, correctness and clarity matter more than extreme optimization.

## Approaches

### Brute-Force Approach

The most direct solution is to examine every possible pair `(i, j)` where `i < j`.

We use two nested loops:

- The outer loop chooses the first element.
- The inner loop chooses the second element after the first.

For every pair, we compute:

```
nums[i] + nums[j]
```

If the sum is less than `target`, we increment the answer.

This works because every possible valid pair is checked exactly once.

The downside is that we perform comparisons for all pairs, leading to `O(n²)` time complexity. Although this is completely acceptable for `n <= 50`, we can still improve the logic using sorting and the two-pointer technique.

### Optimal Two-Pointer Approach

The key observation is that sorting creates order relationships we can exploit.

Suppose the array is sorted.

If:

```
nums[left] + nums[right] < target
```

then every element between `left` and `right` paired with `nums[left]` will also satisfy the condition, because all those elements are less than or equal to `nums[right]`.

That means instead of counting pairs one by one, we can count an entire block of valid pairs at once.

We maintain:

- `left` at the beginning
- `right` at the end

If the sum is valid:

- all pairs from `left+1` through `right` are valid with `left`
- add `right - left` to the answer
- move `left` forward

Otherwise:

- the sum is too large
- move `right` backward to reduce the sum

This avoids checking every pair individually after sorting.

## Algorithm Walkthrough

### Optimal Two-Pointer Algorithm

1. Sort the array in nondecreasing order.

Sorting is essential because the two-pointer strategy depends on ordered values. Once sorted, we can reason about how sums change as pointers move.
2. Initialize two pointers.

Set:

- `left = 0`
- `right = len(nums) - 1`

Also initialize:

- `count = 0`
3. Evaluate the current pair.

Compute:

```
nums[left] + nums[right]
```
4. If the sum is less than the target, count multiple pairs at once.

Since the array is sorted:

```
nums[left] + nums[right] < target
```

implies:

```
nums[left] + nums[k] < target
```

for every index:

```
left < k <= right
```

Therefore, we can add:

```
right - left
```

directly to the answer.

Then move:

```
left += 1
```

We move the left pointer because we have already counted every valid pair starting with that index.
5. Otherwise, decrease the sum.

If:

```
nums[left] + nums[right] >= target
```

then the current sum is too large.

Because the array is sorted, decreasing `right` is the only way to reduce the sum.

Move:

```
right -= 1
```
6. Continue until the pointers cross.

Stop when:

```
left >= right
```

At that point, all valid pairs have been processed.

### Why it works

The algorithm works because sorting establishes monotonic behavior.

When a pair `(left, right)` satisfies the condition, every index between them also forms a valid pair with `left`. This allows us to count many pairs simultaneously instead of checking them individually.

When the sum is too large, decreasing `right` is guaranteed to move toward smaller sums because the array is sorted.

Every possible pair is either counted exactly once or discarded correctly, ensuring the final answer is accurate.

## Python Solution

```python
from typing import List

class Solution:
    def countPairs(self, nums: List[int], target: int) -> int:
        nums.sort()

        left = 0
        right = len(nums) - 1
        count = 0

        while left < right:
            current_sum = nums[left] + nums[right]

            if current_sum < target:
                count += right - left
                left += 1
            else:
                right -= 1

        return count
```

The implementation begins by sorting the array so the two-pointer strategy becomes valid.

The `left` pointer starts at the beginning of the array, while `right` starts at the end. During each iteration, the code computes the sum of the two pointed values.

If the sum is smaller than the target, every element between `left` and `right` also forms a valid pair with `left`, because the array is sorted. Therefore, the implementation adds `right - left` directly to the count.

After counting those pairs, the algorithm advances `left`, since all valid pairs beginning with that position have already been handled.

If the sum is too large, the algorithm decreases `right` to try a smaller value.

The loop continues until the pointers meet.

## Go Solution

```go
package main

import "sort"

func countPairs(nums []int, target int) int {
	sort.Ints(nums)

	left := 0
	right := len(nums) - 1
	count := 0

	for left < right {
		currentSum := nums[left] + nums[right]

		if currentSum < target {
			count += right - left
			left++
		} else {
			right--
		}
	}

	return count
}
```

The Go implementation follows the same logic as the Python solution.

The main difference is that Go uses `sort.Ints(nums)` to sort the slice in place. Variables are explicitly typed through Go's type inference system.

There are no special concerns about integer overflow here because the constraints are extremely small. Empty and single-element slices are handled naturally because the loop condition `left < right` immediately fails.

## Edge Cases

### Single Element Array

If the array contains only one element, there are no possible pairs because a pair requires two distinct indices.

A naive implementation might accidentally attempt invalid indexing or incorrectly count something. The current implementation handles this naturally because the loop condition `left < right` immediately fails.

### Sum Equal to Target

The condition requires the pair sum to be strictly less than the target.

This is a common source of off-by-one mistakes. For example:

```
nums = [1,2]
target = 3
```

The sum equals `3`, so the answer must be `0`.

The implementation correctly uses:

```
if current_sum < target:
```

rather than `<=`.

### Duplicate Values

Arrays may contain repeated numbers, and each index combination counts separately.

For example:

```
nums = [2,2,2,2]
target = 5
```

Every distinct pair is valid.

A buggy implementation might accidentally remove duplicates or count only unique values. The two-pointer approach works correctly because it counts based on indices, not distinct numbers.

### Negative Numbers

Negative values can significantly change pair behavior.

For example:

```
nums = [-7,-6,-2,-1]
target = -2
```

Many sums become valid because adding negatives decreases the total.

The sorted two-pointer strategy still works correctly because ordering remains valid regardless of sign.
