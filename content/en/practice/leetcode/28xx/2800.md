## Solution

## Problem Understanding

The problem gives us three strings, `a`, `b`, and `c`, and asks us to construct the shortest possible string that contains all three as substrings.

A substring must appear contiguously inside the final answer. The three strings may overlap with each other, and those overlaps are the key to minimizing the final length.

For example, if we have:

```
a = "abc"
b = "bcd"
```

then instead of concatenating them into `"abcbcd"`, we can overlap `"bc"` and produce `"abcd"`.

The challenge is not only to minimize the length, but also to break ties lexicographically. If multiple shortest strings exist, we must return the lexicographically smallest one.

The input size is very small:

```
1 <= length <= 100
```

Since there are only three strings, we can afford to try every ordering of the strings. There are only `3! = 6` permutations, which is tiny.

However, a naive brute-force generation of all possible merged strings would still be inefficient if done carelessly. The important observation is that merging two strings optimally can be done greedily by maximizing overlap.

Several edge cases are especially important:

- One string may already contain another entirely.
- Multiple merge orders may produce the same minimal length.
- Different overlaps may lead to the same length but different lexicographical order.
- Two strings may have no overlap at all.
- All three strings may already collapse into one string.

For example:

```
a = "ab"
b = "ba"
c = "aba"
```

Since `"aba"` already contains both `"ab"` and `"ba"`, the answer is simply `"aba"`.

## Approaches

### Brute Force Approach

The most direct idea is to generate every possible way to combine the three strings and then check which resulting string is shortest.

One naive strategy is:

1. Try every permutation of the three strings.
2. Concatenate them in that order.
3. Attempt every possible overlap placement manually.
4. Check whether all strings appear as substrings.
5. Keep the shortest valid answer.

This works because the number of strings is only three. However, manually exploring every overlap position becomes unnecessarily complicated and inefficient.

If each string has length `n`, then checking all placements between strings can approach quadratic work per merge, and the implementation becomes messy.

The key improvement is realizing that when merging two strings, the optimal strategy is always:

- If one string already contains the other, keep the larger one.
- Otherwise, maximize the suffix-prefix overlap.

That observation lets us compute the best merge between two strings directly.

### Optimal Greedy Enumeration Approach

Since there are only six possible orders of the three strings, we can simply:

1. Enumerate all permutations.
2. Merge the first two strings optimally.
3. Merge the result with the third string optimally.
4. Track the shortest result.
5. Use lexicographical comparison to break ties.

The merge operation itself is greedy:

- If `x` already contains `y`, return `x`.
- Otherwise, find the largest suffix of `x` matching a prefix of `y`.
- Append only the non-overlapping part.

This works because maximizing overlap always minimizes the merged length.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n³) to O(n⁴) | O(n) | Tries many overlap placements explicitly |
| Optimal | O(n²) | O(n) | Enumerates only 6 orders and greedily merges |

## Algorithm Walkthrough

### Step 1: Define a merge function

We first create a helper function `merge(x, y)` that returns the shortest string containing both `x` and `y`.

There are two important cases:

- If `x` already contains `y`, then the answer is simply `x`.
- Otherwise, we search for the largest overlap where:

```
suffix of x == prefix of y
```

Then we append only the remaining part of `y`.

For example:

```
x = "abc"
y = "bcd"
```

The largest overlap is `"bc"`.

So:

```
"abc" + "d" = "abcd"
```

### Step 2: Generate all permutations

There are only six possible orders:

```
(a,b,c)
(a,c,b)
(b,a,c)
(b,c,a)
(c,a,b)
(c,b,a)
```

We must test every order because overlap opportunities depend on ordering.

### Step 3: Merge sequentially

For each permutation:

1. Merge the first two strings.
2. Merge the result with the third string.

Example:

```
a = "abc"
b = "bca"
c = "aaa"
```

Order:

```
("aaa", "abc", "bca")
```

First merge:

```
merge("aaa", "abc") = "aaabc"
```

Second merge:

```
merge("aaabc", "bca") = "aaabca"
```

### Step 4: Track the best answer

We maintain the current best string.

A candidate replaces the current best if:

1. Its length is smaller, or
2. The lengths are equal and it is lexicographically smaller.

### Step 5: Return the final answer

After testing all six permutations, the stored best string is the answer.

### Why it works

For any fixed ordering of the strings, greedily maximizing overlap between adjacent strings produces the shortest possible merged result for that order.

Since we exhaustively test all six orderings, we are guaranteed to consider the globally optimal arrangement.

Finally, lexicographical comparison ensures that among all shortest valid strings, we return the smallest one.

## Python Solution

```python
from itertools import permutations

class Solution:
    def minimumString(self, a: str, b: str, c: str) -> str:
        def merge(x: str, y: str) -> str:
            # If y is already inside x, no need to add anything
            if y in x:
                return x

            max_overlap = 0

            # Find largest suffix-prefix overlap
            limit = min(len(x), len(y))

            for k in range(1, limit + 1):
                if x[-k:] == y[:k]:
                    max_overlap = k

            return x + y[max_overlap:]

        best = None

        for order in permutations([a, b, c]):
            merged = merge(merge(order[0], order[1]), order[2])

            if (
                best is None
                or len(merged) < len(best)
                or (len(merged) == len(best) and merged < best)
            ):
                best = merged

        return best
```

The implementation starts by defining the `merge` helper function. This function computes the shortest string containing both inputs.

The first check handles substring containment. If `y` is already inside `x`, we simply return `x`.

Otherwise, we search for the largest overlap between the suffix of `x` and the prefix of `y`. The variable `max_overlap` stores the best overlap size found so far.

After determining the overlap, we append only the unmatched suffix of `y`.

Next, we iterate through all six permutations using `itertools.permutations`.

For each ordering:

1. Merge the first two strings.
2. Merge the result with the third string.

We then compare the generated candidate against the current best answer. The comparison prioritizes:

1. Smaller length
2. Lexicographically smaller order when lengths tie

Finally, we return the best candidate found.

## Go Solution

```go
package main

import (
	"strings"
)

func minimumString(a string, b string, c string) string {
	merge := func(x, y string) string {
		// If y already exists inside x
		if strings.Contains(x, y) {
			return x
		}

		maxOverlap := 0
		limit := min(len(x), len(y))

		// Find largest suffix-prefix overlap
		for k := 1; k <= limit; k++ {
			if x[len(x)-k:] == y[:k] {
				maxOverlap = k
			}
		}

		return x + y[maxOverlap:]
	}

	orders := [][]string{
		{a, b, c},
		{a, c, b},
		{b, a, c},
		{b, c, a},
		{c, a, b},
		{c, b, a},
	}

	best := ""

	for i, order := range orders {
		merged := merge(merge(order[0], order[1]), order[2])

		if i == 0 ||
			len(merged) < len(best) ||
			(len(merged) == len(best) && merged < best) {
			best = merged
		}
	}

	return best
}

func min(a, b int) int {
	if a < b {
		return a
	}
	return b
}
```

The Go implementation follows the same logic as the Python version.

One notable difference is substring checking. Python uses:

```go
if y in x
```

while Go uses:

```go
strings.Contains(x, y)
```

Go also requires explicit slice operations for substring comparison:

```go
x[len(x)-k:]
y[:k]
```

Since Go does not provide a built-in `min` function for integers, we define a helper manually.

The lexicographical comparison works naturally with Go string comparison operators.

## Edge Cases

One important edge case occurs when one string already contains another. A naive implementation might still attempt to append the contained string, producing unnecessarily long results. For example:

```
a = "aba"
b = "ab"
```

The correct merge is simply `"aba"`. The implementation explicitly checks substring containment before computing overlaps.

Another tricky case involves repeated characters. Consider:

```
a = "aaaa"
b = "aaab"
```

There are multiple possible overlaps of different lengths. A buggy implementation might stop at the first overlap instead of finding the largest one. Our algorithm scans all overlap lengths and keeps the maximum valid overlap.

Lexicographical tie-breaking is another subtle source of bugs. Different permutations can produce answers with identical lengths. For example:

```
"abca"
"bcab"
```

Both may have the same size, but the problem requires the lexicographically smallest result. The implementation explicitly compares strings when lengths match.

Finally, completely disjoint strings require correct concatenation behavior. If no overlap exists, the algorithm should append the entire second string. The overlap search naturally handles this case because `max_overlap` remains zero.
