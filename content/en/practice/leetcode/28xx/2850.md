## Solution

## Problem Understanding

The problem gives us a `3 x 3` grid where each cell contains some number of stones. The total number of stones across the entire grid is always exactly `9`, which matches the total number of cells in the grid.

Our goal is to redistribute the stones so that every cell ends up containing exactly one stone. We are allowed to move stones one step at a time between adjacent cells that share a side, meaning movement is restricted to up, down, left, and right.

Each movement of a single stone across one edge counts as one move. The task is to compute the minimum number of moves needed to achieve a valid final configuration.

The important observation is that some cells may initially contain too many stones, while others may contain none. A cell with more than one stone has "extra" stones that must be moved elsewhere. A cell with zero stones needs exactly one incoming stone.

Because movement cost is based on Manhattan distance, moving a stone from `(r1, c1)` to `(r2, c2)` costs:

$|r_1-r_2|+|c_1-c_2|$

The grid size is extremely small. There are only `9` cells total, which means even exponential or backtracking solutions are feasible. This constraint is the key reason dynamic programming with bitmasking works efficiently here.

Several edge cases are important:

- A grid may already be valid, meaning every cell already contains one stone. The answer should then be `0`.
- One cell may contain many extra stones while several cells are empty.
- Multiple assignments of extra stones to empty cells may exist, but only one gives the minimum total movement cost.
- Greedy nearest-cell assignment is not always optimal globally, so we must consider combinations carefully.

## Approaches

### Brute Force Approach

A brute force solution would try every possible way to move stones from surplus cells to empty cells.

First, we identify all empty cells and all extra stones. For example, if a cell contains `4` stones, then it contributes `3` extra movable stones. We can think of these extra stones as independent resources that must be assigned to empty cells.

The brute force method generates every permutation of assigning extra stones to empty cells, computes the total Manhattan distance for each assignment, and returns the minimum.

This approach is correct because it exhaustively checks every possible redistribution plan. However, factorial growth becomes expensive as the number of empty cells increases.

Even though the grid is small, a cleaner and more scalable approach is preferable.

### Optimal Approach, Backtracking with Bitmask Dynamic Programming

The key insight is that the problem is fundamentally an assignment problem.

Each empty cell needs exactly one stone, and each extra stone can fill exactly one empty cell. The movement cost between them is simply the Manhattan distance.

Instead of exploring every arrangement redundantly, we use dynamic programming with bitmasking.

We collect:

- A list of empty cells
- A list of extra stones

Then we recursively assign stones one empty cell at a time.

A bitmask tracks which extra stones have already been used. At each step:

1. Choose an unused extra stone
2. Assign it to the current empty cell
3. Add the Manhattan distance cost
4. Recurse for the next empty cell

Memoization avoids recomputing identical states.

Because there are at most `8` movable stones, the total number of DP states is very small.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(k!) | O(k) | Tries every assignment permutation |
| Optimal | O(k × 2^k) | O(2^k) | Bitmask DP with memoization |

Here, `k` is the number of empty cells, which is also the number of extra stones.

## Algorithm Walkthrough

### Step 1: Identify empty cells and extra stones

Traverse the grid.

If a cell contains `0`, store its coordinates in an `empty_cells` list.

If a cell contains more than `1`, add its coordinates multiple times into an `extra_stones` list. For example:

- `3` stones contributes `2` extra entries
- `4` stones contributes `3` extra entries

This transforms the problem into matching individual extra stones with empty cells.

### Step 2: Define the DP state

We use a recursive function:

```
dp(index, mask)
```

Where:

- `index` represents which empty cell we are currently filling
- `mask` indicates which extra stones have already been used

If bit `i` in the mask is set, then the `i-th` extra stone has already been assigned.

### Step 3: Handle the base case

If `index == len(empty_cells)`, then all empty cells have been filled.

Return `0` because no more moves are needed.

### Step 4: Try assigning every unused extra stone

For the current empty cell:

1. Iterate through every extra stone
2. Skip stones already used in the mask
3. Compute Manhattan distance
4. Recurse with updated mask
5. Keep the minimum total cost

The transition cost is:

$dp(i,mask)=\min_j\left(dist(i,j)+dp(i+1,mask\cup(1\ll j))\right)$

### Step 5: Memoize results

Many recursive states repeat.

Memoization stores computed answers for each `(index, mask)` pair so they are only solved once.

### Why it works

Every valid final arrangement corresponds to assigning each extra stone to exactly one empty cell. The DP explores all such assignments while ensuring no stone is reused.

Since the algorithm checks every valid assignment and always chooses the minimum total cost, it guarantees the optimal answer.

## Python Solution

```python
from typing import List
from functools import lru_cache

class Solution:
    def minimumMoves(self, grid: List[List[int]]) -> int:
        empty_cells = []
        extra_stones = []

        for r in range(3):
            for c in range(3):
                if grid[r][c] == 0:
                    empty_cells.append((r, c))
                elif grid[r][c] > 1:
                    for _ in range(grid[r][c] - 1):
                        extra_stones.append((r, c))

        @lru_cache(None)
        def dp(index: int, mask: int) -> int:
            if index == len(empty_cells):
                return 0

            target_r, target_c = empty_cells[index]
            best = float("inf")

            for i in range(len(extra_stones)):
                if mask & (1 << i):
                    continue

                stone_r, stone_c = extra_stones[i]

                distance = abs(target_r - stone_r) + abs(target_c - stone_c)

                best = min(
                    best,
                    distance + dp(index + 1, mask | (1 << i))
                )

            return best

        return dp(0, 0)
```

The implementation begins by scanning the grid and separating cells into two groups.

`empty_cells` stores every location that needs a stone. `extra_stones` stores every movable stone individually. If a cell contains multiple surplus stones, that coordinate appears multiple times.

The recursive `dp` function represents the minimum remaining cost after filling the first `index` empty cells while using the stones indicated by `mask`.

For each recursive call, the algorithm tries every unused extra stone. It computes the Manhattan distance to the current empty cell and recursively solves the remaining subproblem.

The `lru_cache` decorator memoizes results automatically, which prevents exponential recomputation of identical states.

Finally, `dp(0, 0)` starts the search with no empty cells filled and no stones used.

## Go Solution

```go
package main

import "math"

func minimumMoves(grid [][]int) int {
	var emptyCells [][2]int
	var extraStones [][2]int

	for r := 0; r < 3; r++ {
		for c := 0; c < 3; c++ {
			if grid[r][c] == 0 {
				emptyCells = append(emptyCells, [2]int{r, c})
			} else if grid[r][c] > 1 {
				for k := 0; k < grid[r][c]-1; k++ {
					extraStones = append(extraStones, [2]int{r, c})
				}
			}
		}
	}

	memo := make(map[[2]int]int)

	var dp func(int, int) int

	dp = func(index int, mask int) int {
		if index == len(emptyCells) {
			return 0
		}

		key := [2]int{index, mask}

		if val, exists := memo[key]; exists {
			return val
		}

		target := emptyCells[index]
		best := math.MaxInt32

		for i := 0; i < len(extraStones); i++ {
			if (mask & (1 << i)) != 0 {
				continue
			}

			stone := extraStones[i]

			distance := abs(target[0]-stone[0]) + abs(target[1]-stone[1])

			cost := distance + dp(index+1, mask|(1<<i))

			if cost < best {
				best = cost
			}
		}

		memo[key] = best
		return best
	}

	return dp(0, 0)
}

func abs(x int) int {
	if x < 0 {
		return -x
	}
	return x
}
```

The Go implementation follows the same algorithmic structure as the Python version.

Because Go does not provide built-in memoization decorators, we explicitly store computed states in a map. The key is a pair containing `(index, mask)`.

Slices of fixed-size arrays are used to store coordinates efficiently. Integer overflow is not a concern because the maximum answer is very small.

| Space | O(2^k) | Memoization table stores bitmask states |

Here, `k` is the number of empty cells.

The number of DP states is bounded by all possible bitmasks. For each state, we may iterate through all stones. Since `k ≤ 8`, the algorithm is extremely fast in practice.

## Edge Cases

One important edge case is when the grid is already balanced. Every cell already contains exactly one stone, so there are no empty cells and no extra stones. A buggy implementation might still attempt recursion or indexing operations on empty arrays. This implementation handles the case naturally because the base condition immediately returns `0`.

Another important case occurs when one cell contains many stones, such as `9`, while all other cells are empty. This creates the maximum number of extra stones and requires the algorithm to evaluate many assignments. The implementation correctly expands each surplus stone into a separate entry in `extra_stones`, which allows the DP to treat each movable stone independently.

A third tricky scenario is when multiple surplus cells can fill the same empty cell with similar costs. Greedy approaches may choose a locally optimal movement that blocks a better global assignment later. The DP solution avoids this issue because it evaluates every possible assignment combination and memoizes the best outcome.

A final subtle edge case is duplicate coordinates inside `extra_stones`. If a cell has several surplus stones, the same coordinate appears multiple times. The bitmask still treats them as separate movable stones, which is exactly what we want. This guarantees that all surplus stones are accounted for independently.
