## Solution

## Problem Understanding

The problem gives us a list of cars parked on a number line. Each car occupies every integer point from `starti` to `endi`, inclusive. For example, a car represented by `[3, 6]` covers the integer points `3, 4, 5, 6`.

Our task is to determine how many distinct integer points are covered by at least one car. If multiple cars overlap at the same point, that point should only be counted once.

The input is a 2D array `nums`, where every element contains two integers representing the inclusive interval occupied by a car. The output is a single integer representing the total number of covered integer points.

The constraints are very small:

- There are at most 100 cars.
- Each coordinate is between 1 and 100.
- Every interval is valid because `starti <= endi`.

These constraints are important because they allow us to use very direct approaches without worrying about performance issues. Since the coordinate range is capped at 100, even iterating over every possible point is completely acceptable.

There are several edge cases worth noticing early:

- Multiple cars may overlap heavily, and overlapping points must only be counted once.
- Cars may be completely disjoint, meaning we count all intervals separately.
- A car may occupy only a single point, such as `[5,5]`.
- Intervals may be nested inside other intervals.
- The cars are not guaranteed to be sorted.

The problem guarantees that all intervals are valid and remain within a very small coordinate range, which simplifies implementation significantly.

## Approaches

### Brute Force Approach

A straightforward solution is to examine every interval and mark every integer point covered by that interval.

For each car `[start, end]`, we iterate through every integer from `start` to `end` and store the point in a set. Since sets automatically remove duplicates, overlapping points are counted only once.

After processing all cars, the size of the set gives the number of distinct covered points.

This approach is correct because every covered point is explicitly recorded, and duplicates are automatically ignored.

Although this is called a brute force solution, it is actually fast enough for the problem constraints because the coordinate range is extremely small.

### Optimal Approach

A slightly more structured approach uses a prefix sum technique, also known as a difference array approach.

Instead of marking every point individually for every interval, we record where coverage starts and where it ends.

For an interval `[start, end]`:

- Add `1` at `start`
- Subtract `1` at `end + 1`

Then we compute a running prefix sum across the number line. If the running sum at a position is greater than zero, that point is covered by at least one car.

The key insight is that intervals can be represented compactly using incremental changes rather than explicitly marking every point repeatedly.

Because the coordinate range is tiny, both methods are efficient. However, the prefix sum approach demonstrates an important interval-processing technique commonly used in larger problems.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × range) | O(range) | Explicitly marks every covered point using a set |
| Optimal | O(n + range) | O(range) | Uses difference array and prefix sums to compute coverage efficiently |

Here, `range` refers to the coordinate range, which is at most 100.

## Algorithm Walkthrough

### Optimal Prefix Sum Algorithm

1. Create a difference array large enough to store all coordinate changes.

Since coordinates go up to 100, we create an array slightly larger than that so we can safely access `end + 1`.
2. Process each interval `[start, end]`.

For every car:

- Increment `difference[start]`
- Decrement `difference[end + 1]`

This records where coverage begins and where it stops.
3. Compute the prefix sum across the number line.

Maintain a running value called `activeCars`.

At every point:

- Add the current difference value to `activeCars`
- If `activeCars > 0`, then at least one car covers this point
4. Count covered points.

Every position whose prefix sum is positive contributes one to the answer.
5. Return the final count.

### Why it works

The difference array stores how coverage changes at each coordinate. When we compute the prefix sum, we reconstruct how many intervals currently cover each point.

If the running total is greater than zero, then at least one interval includes that point. Since every interval contributes `+1` at its start and `-1` after its end, the prefix sum exactly tracks active coverage across the number line.

## Python Solution

```python
from typing import List

class Solution:
    def numberOfPoints(self, nums: List[List[int]]) -> int:
        difference = [0] * 102

        for start, end in nums:
            difference[start] += 1
            difference[end + 1] -= 1

        covered_points = 0
        active_cars = 0

        for point in range(1, 101):
            active_cars += difference[point]

            if active_cars > 0:
                covered_points += 1

        return covered_points
```

The implementation begins by creating a difference array of size 102. This extra space ensures that accessing `end + 1` is always safe, even when `end` equals 100.

For every interval, we record where coverage starts and where it stops. Adding `1` at `start` indicates that a new interval becomes active there. Subtracting `1` at `end + 1` indicates that the interval is no longer active after `end`.

Next, we iterate through every possible coordinate from 1 to 100 while maintaining a running prefix sum in `active_cars`. This value represents how many intervals currently cover the point.

Whenever `active_cars` is greater than zero, the current point is covered by at least one car, so we increment the answer.

Finally, the total number of covered points is returned.

## Go Solution

```go
func numberOfPoints(nums [][]int) int {
    difference := make([]int, 102)

    for _, interval := range nums {
        start := interval[0]
        end := interval[1]

        difference[start]++
        difference[end+1]--
    }

    coveredPoints := 0
    activeCars := 0

    for point := 1; point <= 100; point++ {
        activeCars += difference[point]

        if activeCars > 0 {
            coveredPoints++
        }
    }

    return coveredPoints
}
```

The Go implementation follows the same logic as the Python version. The main difference is syntax and array handling.

Go uses slices instead of Python lists, so we allocate the difference array using `make([]int, 102)`.

There are no integer overflow concerns because all values remain extremely small under the problem constraints.

The iteration over intervals uses Go's `range` syntax, and the prefix sum logic is identical to the Python solution.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n + range) | Process intervals once, then scan the coordinate range |
| Space | O(range) | Difference array stores coordinate coverage changes |

The algorithm processes each interval exactly once, which costs `O(n)`. After that, it scans all possible coordinates from 1 to 100, which costs `O(range)`.

Since the coordinate range is bounded by 100, the algorithm is effectively linear and extremely efficient.

The space complexity comes from the difference array, whose size is proportional to the coordinate range.

## Edge Cases

One important edge case occurs when multiple intervals overlap completely. For example, `[[1,5],[1,5],[1,5]]` should still return `5`, not `15`. A buggy implementation might accidentally count duplicate coverage multiple times. The prefix sum approach handles this naturally because it only checks whether the running coverage count is greater than zero.

Another important case is single-point intervals such as `[[7,7]]`. Since the interval includes both endpoints, point `7` must be counted. Some implementations make mistakes with inclusive ranges and accidentally exclude the endpoint. Our implementation correctly handles this by decrementing at `end + 1`, ensuring the endpoint itself remains covered.

A third edge case involves intervals that only touch at boundaries, such as `[[1,2],[2,3]]`. Point `2` belongs to both intervals but should only be counted once. The prefix sum method correctly tracks active coverage and counts each coordinate exactly once regardless of how many intervals overlap there.

A final edge case is the maximum boundary value, such as `[1,100]`. Since we access `end + 1`, implementations that allocate insufficient array space may trigger out-of-bounds errors. Our solution avoids this by creating an array of size 102.
