**Difficulty:** 🔴 Hard  
**Topics:** Array, Hash Table, Stack, Greedy, Sorting, Heap (Priority Queue)

## Solution

## Problem Understanding

We are given an array called `items`, where every element contains two integers:

- `profit`
- `category`

We must choose exactly `k` items and maximize a special score called the elegance.

The elegance formula is:

$$\text{elegance} = \text{total profit} + (\text{number of distinct categories})^2$$

The first part of the score rewards choosing items with large profits. The second part rewards diversity of categories, and because the number of distinct categories is squared, increasing category diversity can become extremely valuable.

The key difficulty is that these two goals compete with each other:

- Picking the highest-profit items may produce many duplicate categories.
- Picking many distinct categories may force us to give up large profits.

We need the best balance.

The input size is large:

- `n` can be up to `10^5`
- profits can be as large as `10^9`

This immediately rules out exponential or quadratic subset enumeration approaches.

Another important detail is that the problem asks for a subsequence, but the relative order does not matter for the elegance formula. Since we only care about which items are selected, we can freely sort the items without changing the answer.

Several edge cases are important:

- All items may belong to the same category.
- Every item may belong to a different category.
- The optimal solution may intentionally replace a high-profit duplicate-category item with a smaller-profit new-category item because the square bonus becomes large enough.
- Profits are large, so integer overflow matters in some languages such as Go.

## Approaches

### Brute Force

The most direct approach is to try every possible subsequence of size `k`.

For each subset:

1. Compute the sum of profits.
2. Count distinct categories.
3. Compute the elegance score.
4. Track the maximum.

This works because it literally evaluates every valid selection.

However, the number of subsets is:

$$\binom{n}{k}$$

This becomes astronomically large even for moderate values of `n`. With `n = 10^5`, brute force is completely impossible.

### Key Insight

The important observation is that the profit contribution and category contribution can be separated strategically.

Suppose we first choose the `k` highest-profit items. This maximizes profit, but it may contain many duplicate categories.

Now consider improving category diversity.

If we want to introduce a new category, we must remove one existing item because the subsequence size must remain exactly `k`.

Which item should we remove?

We should remove the duplicate-category item with the smallest profit, because:

- Removing a unique-category item would reduce the distinct-category count.
- Removing the smallest-profit duplicate sacrifices the least profit.

This naturally suggests:

- Sort items by profit descending.
- Initially take the first `k` items.
- Track duplicate-category profits.
- Then scan remaining items looking for unseen categories.
- Replace the smallest duplicate-profit item whenever doing so increases diversity.

A min-heap is perfect for storing removable duplicate profits efficiently.

| Optimal | O(n log n) | O(n) | Greedy with sorting and min-heap |

## Algorithm Walkthrough

1. Sort all items in descending order of profit.

We want to prioritize high-profit items first. Sorting allows us to greedily start with the most profitable subsequence possible.
2. Select the first `k` items.

These are the highest-profit items available.

While processing them:

- Add their profits to `total_profit`.
- Track categories in a hash set.
- If a category appears more than once, store that item's profit in a min-heap.

The heap represents removable duplicate-category items.
3. Compute the initial elegance.

If there are `d` distinct categories:

$$\text{elegance} = \text{total\_profit} + d^2$$
4. Process the remaining items one by one.

For each item:

- If its category already exists in the selected subsequence, skip it because it does not increase category diversity.
- Otherwise, this item introduces a new category.
5. Replace the smallest duplicate-category item.

If the heap is not empty:

- Remove the smallest duplicate profit from the heap.
- Subtract it from `total_profit`.
- Add the new item's profit.
- Add the new category.

This operation increases the distinct category count by one while minimizing profit loss.
6. Recompute elegance after each replacement.

Update the answer with the maximum elegance seen so far.

### Why it works

The algorithm maintains an optimal invariant:

- At any moment, among all selected items with the current set of categories, the chosen subsequence has maximum possible profit.

Whenever we add a new category, the best item to remove is always the smallest-profit duplicate-category item. Removing anything else would either:

- reduce distinct categories, or
- lose more profit unnecessarily.

Because we process items in descending profit order, the first unseen item encountered for a category is the best possible representative of that category.

Therefore, every category-expansion step is greedily optimal.

## Python Solution

```python
from typing import List
import heapq

class Solution:
    def findMaximumElegance(self, items: List[List[int]], k: int) -> int:
        # Sort by profit descending
        items.sort(key=lambda x: -x[0])

        total_profit = 0
        category_set = set()

        # Min-heap of duplicate-category profits
        duplicate_profits = []

        # Initial selection of first k items
        for i in range(k):
            profit, category = items[i]

            total_profit += profit

            if category in category_set:
                heapq.heappush(duplicate_profits, profit)
            else:
                category_set.add(category)

        distinct_count = len(category_set)

        best = total_profit + distinct_count * distinct_count

        # Try introducing new categories
        for i in range(k, len(items)):
            profit, category = items[i]

            # Skip if category already exists
            if category in category_set:
                continue

            # Need a duplicate item to replace
            if not duplicate_profits:
                break

            removed_profit = heapq.heappop(duplicate_profits)

            total_profit -= removed_profit
            total_profit += profit

            category_set.add(category)

            distinct_count = len(category_set)

            elegance = total_profit + distinct_count * distinct_count

            best = max(best, elegance)

        return best
```

The implementation begins by sorting all items in descending order of profit. This ensures that the initial selection of `k` items maximizes total profit.

The variable `total_profit` stores the sum of the currently selected items. The `category_set` tracks distinct categories currently included.

Whenever we encounter a duplicate category during the initial selection, we push its profit into a min-heap called `duplicate_profits`. These are candidates for future removal.

The heap is important because when introducing a new category, we want to sacrifice the smallest possible duplicate profit.

After computing the initial elegance, we scan the remaining items. If an item belongs to a new category and we still have removable duplicates, we perform a replacement.

Each replacement:

- increases distinct categories by one,
- minimizes profit loss,
- potentially improves elegance.

The answer is updated after every successful replacement.

## Go Solution

```go
package main

import (
	"container/heap"
	"sort"
)

type MinHeap []int

func (h MinHeap) Len() int {
	return len(h)
}

func (h MinHeap) Less(i, j int) bool {
	return h[i] < h[j]
}

func (h MinHeap) Swap(i, j int) {
	h[i], h[j] = h[j], h[i]
}

func (h *MinHeap) Push(x interface{}) {
	*h = append(*h, x.(int))
}

func (h *MinHeap) Pop() interface{} {
	old := *h
	n := len(old)

	value := old[n-1]

	*h = old[:n-1]

	return value
}

func findMaximumElegance(items [][]int, k int) int64 {
	sort.Slice(items, func(i, j int) bool {
		return items[i][0] > items[j][0]
	})

	totalProfit := int64(0)

	categorySet := make(map[int]bool)

	duplicateProfits := &MinHeap{}
	heap.Init(duplicateProfits)

	// Initial selection
	for i := 0; i < k; i++ {
		profit := items[i][0]
		category := items[i][1]

		totalProfit += int64(profit)

		if categorySet[category] {
			heap.Push(duplicateProfits, profit)
		} else {
			categorySet[category] = true
		}
	}

	distinctCount := int64(len(categorySet))

	best := totalProfit + distinctCount*distinctCount

	// Try adding new categories
	for i := k; i < len(items); i++ {
		profit := items[i][0]
		category := items[i][1]

		if categorySet[category] {
			continue
		}

		if duplicateProfits.Len() == 0 {
			break
		}

		removedProfit := heap.Pop(duplicateProfits).(int)

		totalProfit -= int64(removedProfit)
		totalProfit += int64(profit)

		categorySet[category] = true

		distinctCount = int64(len(categorySet))

		elegance := totalProfit + distinctCount*distinctCount

		if elegance > best {
			best = elegance
		}
	}

	return best
}
```

The Go implementation follows the same algorithmic structure as the Python version, but several language-specific details are important.

Go does not include a built-in heap type for integers, so we implement the `heap.Interface` manually using `container/heap`.

The result type is `int64` because profits can be as large as `10^9`, and the total sum can exceed 32-bit integer limits.

The category set is implemented using `map[int]bool`.

## Edge Cases

One important edge case occurs when all items belong to the same category. In this situation, increasing category diversity is impossible because no new categories exist. A buggy implementation might still attempt replacements or incorrectly update the distinct category count. This implementation handles the case naturally because every later item is skipped as an already-seen category.

Another important case happens when every selected item already has a unique category. In this scenario, the duplicate heap becomes empty. Once there are no duplicate-category items available for removal, no further category expansion is possible. The implementation correctly stops processing because replacing a unique-category item would reduce distinct categories instead of increasing them.

A third tricky case is when a lower-profit item should replace a higher-profit duplicate because the square bonus compensates for the lost profit. Greedy algorithms that maximize only profit fail here. The heap-based strategy handles this correctly because it explicitly evaluates category expansion opportunities while minimizing the sacrificed profit.

A final subtle issue involves integer overflow. Profits can reach `10^9`, and the total profit may exceed the range of 32-bit integers. The Go solution uses `int64` throughout all arithmetic operations to avoid overflow bugs.
