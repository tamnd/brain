## Problem Understanding

This problem asks us to simulate a simple purchase with rounding and compute the resulting bank account balance. You start with a fixed balance of 100 dollars. When a purchase occurs, the purchase amount must first be **rounded to the nearest multiple of 10**, where values ending in 5 or more are rounded **up**. For example, 9 rounds to 10, 15 rounds to 20, and 10 remains 10. After rounding, this rounded amount is subtracted from the balance, and the resulting value is returned.

The input `purchaseAmount` is an integer between 0 and 100 inclusive. The output is a single integer representing the remaining balance. Edge cases include values that are already multiples of 10, values at the lower boundary (0), and values at the upper boundary (100). The problem guarantees that the input is valid, so we do not need to handle negative values or values greater than 100.

## Approaches

The brute-force approach would involve iterating through every integer between 0 and `purchaseAmount` and manually rounding each number to the nearest multiple of 10, then summing or subtracting. While correct, this is unnecessary because the rounding operation can be computed directly using simple arithmetic. Given the small input range (0 to 100), performance is not an issue, but a direct computation is both simpler and more readable.

The optimal approach leverages the observation that rounding to the nearest multiple of 10 is equivalent to computing `(purchaseAmount + 5) // 10 * 10`. This formula works because adding 5 ensures that numbers with a last digit of 5 or higher round up, while lower digits round down when integer division truncates. Once the rounded value is calculated, subtracting it from 100 yields the remaining balance.

| Approach | Time Complexity | Space Complexity | Notes |

| Brute Force | O(purchaseAmount) | O(1) | Iterates through all numbers up to purchaseAmount to compute rounding |
| Optimal | O(1) | O(1) | Uses arithmetic rounding formula for direct computation |

## Algorithm Walkthrough

1. Start with the initial balance of 100 dollars. This is fixed and does not depend on the input.
2. Compute the rounded purchase amount using the formula `roundedAmount = ((purchaseAmount + 5) // 10) * 10`. This ensures standard rounding to the nearest multiple of 10.
3. Subtract `roundedAmount` from the initial balance to get the remaining balance.
4. Return the computed remaining balance.

This algorithm works because integer arithmetic correctly handles rounding to the nearest multiple of 10. The invariant is that the rounded purchase amount will always be the closest multiple of 10 to the original `purchaseAmount`, and subtracting this from 100 gives the correct balance.

## Python Solution

```python
class Solution:
    def accountBalanceAfterPurchase(self, purchaseAmount: int) -> int:
        initial_balance = 100
        # Round the purchaseAmount to nearest multiple of 10
        roundedAmount = ((purchaseAmount + 5) // 10) * 10
        # Subtract the rounded amount from the balance
        remaining_balance = initial_balance - roundedAmount
        return remaining_balance
```

This implementation first defines the starting balance. It then calculates the rounded purchase amount using integer division. Finally, it computes the remaining balance by simple subtraction and returns the result. The logic is direct and matches the step-by-step algorithm above.

## Go Solution

func accountBalanceAfterPurchase(purchaseAmount int) int {
    initialBalance := 100
    // Round purchaseAmount to the nearest multiple of 10
    roundedAmount := ((purchaseAmount + 5) / 10) * 10
    // Subtract the rounded amount from the initial balance

The Go implementation mirrors the Python logic. Note that integer division in Go also truncates toward zero, which correctly implements the rounding formula. Variable names are chosen to clearly indicate their role, similar to the Python version.

## Edge Cases

One important edge case is `purchaseAmount = 0`. Without proper handling, subtracting zero might still work, but the rounding formula must not break; it correctly returns 0. Another is `purchaseAmount = 100`, which is the upper limit. The rounding formula should handle this correctly and return 100, giving a remaining balance of 0. A third edge case is `purchaseAmount` values that are exactly 5, 15, 25, etc., which are on the boundary for rounding up. Using `(purchaseAmount + 5) // 10 * 10` ensures these round correctly. The implementation handles all of these correctly by using integer arithmetic consistently.
