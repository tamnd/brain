## Solution

## Problem Understanding

The problem asks us to find the maximum possible value of:

$$(nums[i] - nums[j]) \times nums[k]$$

subject to the ordering constraint:

$$i < j < k$$

The array is zero indexed, and we must choose three different indices in strictly increasing order.

For every valid triplet `(i, j, k)`, we compute the expression and take the maximum value among all possible triplets. If every possible triplet produces a negative value, we return `0`.

The input is an integer array `nums` with length up to `10^5`. Each element can be as large as `10^6`.

The constraints are important because they immediately rule out any cubic solution. A brute force solution that tries every triplet would require:

$$O(n^3)$$

operations, which is far too slow for `n = 100000`.

The key observation is that for each middle index `j`, we want:

- the largest possible `nums[i]` on the left side
- the largest possible `nums[k]` on the right side

because maximizing both factors helps maximize the product.

Several edge cases are important:

- All triplets may produce negative values, in which case we must return `0`.
- The best triplet may not use adjacent elements.
- Since values are large, the product may exceed 32 bit integer range, so we must use 64 bit arithmetic.
- Arrays with strictly increasing values often produce negative differences because `nums[i] < nums[j]`.

## Approaches

### Brute Force Approach

The most direct solution is to try every possible ordered triplet `(i, j, k)`.

We use three nested loops:

- The first loop selects `i`
- The second loop selects `j`
- The third loop selects `k`

For every valid triplet, we compute:

$$(nums[i] - nums[j]) \times nums[k]$$

and update the answer if the value is larger.

This approach is correct because it exhaustively checks every valid triplet. No candidate is missed.

However, the time complexity is:

$$O(n^3)$$

With `n = 100000`, this is completely infeasible.

### Optimal Approach

The expression can be rewritten conceptually as:

$$(\text{best left value before } j - nums[j]) \times (\text{best right value after } j)$$

For a fixed middle index `j`, the optimal choice is:

- the maximum value on the left side for `nums[i]`
- the maximum value on the right side for `nums[k]`

This suggests preprocessing.

We can build:

- `prefixMax[i]` = maximum value from `nums[0]` to `nums[i]`
- `suffixMax[i]` = maximum value from `nums[i]` to `nums[n-1]`

Then for every middle index `j`:

- best left value is `prefixMax[j - 1]`
- best right value is `suffixMax[j + 1]`

So the candidate value becomes:

$$(prefixMax[j-1] - nums[j]) \times suffixMax[j+1]$$

We evaluate this in constant time for each `j`.

This reduces the complexity to linear time.

## Algorithm Walkthrough

1. Create a prefix maximum array.

For each position `i`, store the maximum value seen from the beginning of the array up to index `i`.

This allows us to instantly know the best possible `nums[i]` for any middle position `j`.
2. Create a suffix maximum array.

For each position `i`, store the maximum value seen from index `i` to the end of the array.

This allows us to instantly know the best possible `nums[k]` for any middle position `j`.
3. Iterate through every possible middle index `j`.

Since `i < j < k`, the middle index can range from `1` to `n - 2`.
4. For each `j`, compute:

$$(prefixMax[j-1] - nums[j]) \times suffixMax[j+1]$$

This represents the best possible triplet value using `j` as the middle element.

1. Track the maximum value encountered.
2. Return the maximum value if it is positive, otherwise return `0`.

### Why it works

For a fixed middle index `j`, the expression depends on two independent choices:

- maximizing `nums[i]` on the left
- maximizing `nums[k]` on the right

Since multiplication by a larger positive number increases the result, choosing the maximum right value is always optimal. Similarly, choosing the maximum left value maximizes the difference `(nums[i] - nums[j])`.

Therefore, evaluating only the best left and right candidates for each `j` guarantees the globally optimal answer.

## Python Solution

```python
from typing import List

class Solution:
    def maximumTripletValue(self, nums: List[int]) -> int:
        n = len(nums)

        prefix_max = [0] * n
        suffix_max = [0] * n

        prefix_max[0] = nums[0]
        for i in range(1, n):
            prefix_max[i] = max(prefix_max[i - 1], nums[i])

        suffix_max[n - 1] = nums[n - 1]
        for i in range(n - 2, -1, -1):
            suffix_max[i] = max(suffix_max[i + 1], nums[i])

        answer = 0

        for j in range(1, n - 1):
            left_max = prefix_max[j - 1]
            right_max = suffix_max[j + 1]

            triplet_value = (left_max - nums[j]) * right_max

            answer = max(answer, triplet_value)

        return answer
```

The implementation begins by constructing the prefix maximum array. Each position stores the largest value encountered so far while scanning from left to right.

Next, the suffix maximum array is built by scanning from right to left. Each position stores the largest value that appears from that index onward.

Once these helper arrays are prepared, the algorithm iterates through every valid middle index `j`.

For each `j`:

- `prefix_max[j - 1]` gives the best possible left value
- `suffix_max[j + 1]` gives the best possible right value

The candidate expression is computed and compared against the current best answer.

The answer variable starts at `0`, which automatically handles cases where all triplets produce negative values.

## Go Solution

```go
func maximumTripletValue(nums []int) int64 {
    n := len(nums)

    prefixMax := make([]int, n)
    suffixMax := make([]int, n)

    prefixMax[0] = nums[0]
    for i := 1; i < n; i++ {
        if nums[i] > prefixMax[i-1] {
            prefixMax[i] = nums[i]
        } else {
            prefixMax[i] = prefixMax[i-1]
        }
    }

    suffixMax[n-1] = nums[n-1]
    for i := n - 2; i >= 0; i-- {
        if nums[i] > suffixMax[i+1] {
            suffixMax[i] = nums[i]
        } else {
            suffixMax[i] = suffixMax[i+1]
        }
    }

    var answer int64 = 0

    for j := 1; j < n-1; j++ {
        leftMax := prefixMax[j-1]
        rightMax := suffixMax[j+1]

        tripletValue := int64(leftMax-nums[j]) * int64(rightMax)

        if tripletValue > answer {
            answer = tripletValue
        }
    }

    return answer
}
```

The Go implementation follows the same logic as the Python solution.

The most important Go-specific detail is integer overflow handling. Since values can reach:

$$(10^6 - 1) \times 10^6$$

the result may exceed 32 bit integer range. Therefore, the multiplication is explicitly converted to `int64`.

Slices are used for the prefix and suffix arrays, and no special handling for empty arrays is needed because the constraints guarantee at least three elements.

## Edge Cases

One important edge case occurs when all triplets produce negative values. This happens when the array is strictly increasing, such as `[1,2,3]`. In that case, every difference `(nums[i] - nums[j])` is negative, making the overall product negative as well. The implementation handles this by initializing `answer = 0` and never allowing smaller negative values to replace it.

Another important case involves repeated values, such as `[5,5,5,5]`. Here, every difference becomes zero, so every triplet value is zero. A buggy implementation might incorrectly assume a strictly positive answer exists. The current implementation correctly computes and returns zero.

Large integer overflow is another critical concern. Since values can reach `10^6`, the product can approach `10^12`. This exceeds 32 bit integer range. Python handles large integers automatically, but the Go implementation explicitly converts values to `int64` before multiplication to prevent overflow bugs.

A final subtle case occurs when the optimal triplet is not locally obvious. For example, in `[1,10,3,4,19]`, the best answer uses indices `(1,2,4)`, not adjacent elements. The prefix and suffix maximum approach correctly captures globally optimal left and right choices for every middle index.
