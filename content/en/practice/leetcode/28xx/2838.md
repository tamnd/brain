[LeetCode Problem 2838](https://leetcode.com/problems/maximum-coins-heroes-can-collect/)

**Difficulty:** 🟡 Medium  
**Topics:** Array, Two Pointers, Binary Search, Sorting, Prefix Sum

## Solution

## Problem Understanding

The problem gives us three arrays:

- `heroes[i]` represents the power level of the `i-th` hero.
- `monsters[j]` represents the power level of the `j-th` monster.
- `coins[j]` represents how many coins are earned for defeating the `j-th` monster.

A hero can defeat every monster whose power is less than or equal to the hero's power. Each hero acts independently, meaning multiple heroes may defeat the same monster and earn its coins separately.

For every hero, we need to calculate the total number of coins obtainable from all monsters that hero can defeat.

The key observation is that a hero does not choose a subset of monsters. Since every monster gives positive coins, the maximum coins a hero can collect is simply the sum of coins from all monsters the hero can defeat.

The constraints are important:

- `n` and `m` can both be as large as `10^5`
- Monster powers and coin values can be up to `10^9`

These limits immediately tell us that an `O(n * m)` brute-force solution will be too slow. In the worst case:

```
10^5 * 10^5 = 10^10
```

operations, which is far beyond acceptable limits.

We therefore need a solution closer to `O(n log m)` or `O((n + m) log m)`.

Several edge cases are worth noting early:

- A hero may not be able to defeat any monster, resulting in `0`.
- A hero may be able to defeat all monsters.
- Multiple monsters can have the same power.
- Multiple heroes can have the same power.
- Coin totals may become very large, especially in Go, so `int64` should be used.

## Approaches

### Brute Force Approach

The most direct solution is to process every hero independently.

For each hero:

1. Iterate through all monsters.
2. Check whether `monsterPower <= heroPower`.
3. If true, add the corresponding coin value to the total.

This works because the problem directly defines whether a hero can defeat a monster.

However, this approach performs `m` checks for each of the `n` heroes, leading to `O(n * m)` time complexity.

With both arrays potentially having size `10^5`, this becomes too slow.

### Key Insight for the Optimal Solution

The important observation is that the only thing that matters is:

```
All monsters with power <= hero power
```

This suggests sorting.

If we sort monsters by power, then:

- All defeatable monsters for a hero form a contiguous prefix.
- We can precompute prefix sums of coins.
- For each hero, we only need to determine how many monsters fall within the allowed power range.

This can be done efficiently using binary search.

The overall process becomes:

1. Pair each monster with its coin value.
2. Sort by monster power.
3. Build a prefix sum array of coins.
4. For each hero, binary search to find the last defeatable monster.
5. Use the prefix sum to compute the total instantly.

This reduces the complexity dramatically.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * m) | O(1) | Checks every monster for every hero |
| Optimal | O(m log m + n log m) | O(m) | Uses sorting, prefix sums, and binary search |

## Algorithm Walkthrough

### Step 1: Pair monster powers with coin values

We first combine each monster power with its corresponding coin value:

```
(monsterPower, coinValue)
```

This keeps the relationship intact after sorting.

For example:

```
monsters = [1,1,5,2,3]
coins    = [2,3,4,5,6]
```

becomes:

```
[(1,2), (1,3), (5,4), (2,5), (3,6)]
```

### Step 2: Sort monsters by power

We sort the pairs by monster power.

After sorting:

```
[(1,2), (1,3), (2,5), (3,6), (5,4)]
```

Now all monsters a hero can defeat appear as a prefix of the array.

### Step 3: Build arrays for powers and prefix sums

We extract:

```
powers = [1,1,2,3,5]
```

Then construct a prefix sum array for coins:

```
prefix = [2,5,10,16,20]
```

Here:

- `prefix[i]` represents total coins from monsters `0..i`

This allows constant-time range sum queries.

### Step 4: Process each hero using binary search

For each hero power:

1. Find the rightmost monster whose power is `<= heroPower`
2. Use the prefix sum to get the total coins

Example:

Hero power = `4`

Binary search finds index `3`:

```
powers[3] = 3 <= 4
powers[4] = 5 > 4
```

So the answer is:

```
prefix[3] = 16
```

### Step 5: Handle heroes who cannot defeat any monster

If binary search returns no valid index, the hero earns `0`.

This occurs when all monster powers exceed the hero's power.

### Why it works

After sorting monsters by power, every monster a hero can defeat appears in one contiguous prefix of the sorted array. Binary search correctly identifies the boundary between defeatable and undefeatable monsters. Since the prefix sum array stores cumulative coin totals, retrieving the total coins for that prefix gives the exact maximum coins the hero can collect.

## Python Solution

```python
from bisect import bisect_right
from typing import List

class Solution:
    def maximumCoins(
        self,
        heroes: List[int],
        monsters: List[int],
        coins: List[int]
    ) -> List[int]:

        monster_data = sorted(zip(monsters, coins))

        sorted_powers = []
        prefix_coins = []

        running_sum = 0

        for power, coin in monster_data:
            sorted_powers.append(power)

            running_sum += coin
            prefix_coins.append(running_sum)

        answer = []

        for hero_power in heroes:
            index = bisect_right(sorted_powers, hero_power) - 1

            if index >= 0:
                answer.append(prefix_coins[index])
            else:
                answer.append(0)

        return answer
```

The implementation begins by combining monster powers and coin values using `zip`. This ensures that after sorting, each coin value still corresponds to the correct monster.

The monsters are then sorted by power. Once sorted, the code constructs two arrays:

- `sorted_powers`, which stores monster powers in ascending order
- `prefix_coins`, which stores cumulative coin totals

The variable `running_sum` maintains the cumulative total while building the prefix array.

For each hero, the solution uses `bisect_right` to find the insertion point for the hero's power in the sorted monster powers array. Subtracting one gives the index of the strongest monster the hero can defeat.

If the index is valid, the corresponding prefix sum gives the total obtainable coins. Otherwise, the hero cannot defeat any monster and receives `0`.

## Go Solution

```go
package main

import (
	"sort"
)

func maximumCoins(heroes []int, monsters []int, coins []int) []int64 {
	type Monster struct {
		power int
		coin  int
	}

	m := len(monsters)

	monsterData := make([]Monster, m)

	for i := 0; i < m; i++ {
		monsterData[i] = Monster{
			power: monsters[i],
			coin:  coins[i],
		}
	}

	sort.Slice(monsterData, func(i, j int) bool {
		return monsterData[i].power < monsterData[j].power
	})

	sortedPowers := make([]int, m)
	prefixCoins := make([]int64, m)

	var runningSum int64 = 0

	for i := 0; i < m; i++ {
		sortedPowers[i] = monsterData[i].power

		runningSum += int64(monsterData[i].coin)
		prefixCoins[i] = runningSum
	}

	answer := make([]int64, len(heroes))

	for i, heroPower := range heroes {
		index := sort.Search(len(sortedPowers), func(j int) bool {
			return sortedPowers[j] > heroPower
		}) - 1

		if index >= 0 {
			answer[i] = prefixCoins[index]
		} else {
			answer[i] = 0
		}
	}

	return answer
}
```

The Go implementation follows the same algorithmic structure as the Python version.

A custom `Monster` struct is used to pair monster powers with coin values before sorting.

The major Go-specific detail is integer overflow handling. Since the sum of coins can become very large, the prefix sums and final answers are stored as `int64`.

Go does not have a built-in equivalent of Python's `bisect_right`, so `sort.Search` is used instead. The search locates the first monster with power greater than the hero's power, and subtracting one gives the last defeatable monster.

## Edge Cases

### Heroes Cannot Defeat Any Monster

A common source of bugs occurs when a hero's power is smaller than every monster power. In this situation, binary search effectively returns an invalid position.

For example:

```
heroes = [1]
monsters = [5,6,7]
```

The implementation handles this by checking whether the computed index is negative. If so, it appends `0`.

### Duplicate Monster Powers

Multiple monsters may share the same power value. A naive implementation could accidentally stop at the first matching power instead of including all matching monsters.

For example:

```
monsters = [2,2,2]
```

Using `bisect_right` is critical because it returns the insertion point after all matching values, ensuring every valid monster is counted.

### Large Coin Totals

The total sum of coins may become very large:

```
10^5 monsters * 10^9 coins
```

which can reach `10^14`.

Python handles this naturally because integers are arbitrary precision. In Go, however, a regular `int` may overflow on some systems. The implementation therefore uses `int64` for prefix sums and answers to ensure correctness.
