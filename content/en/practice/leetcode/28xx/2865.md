## Solution

## Problem Understanding

The problem gives us an array `heights`, where `heights[i]` represents the maximum possible height of the `i-th` tower. We are allowed to remove bricks from any tower, but we are not allowed to add bricks. After removing bricks, we want the final tower arrangement to form a valid mountain shape.

A mountain-shaped arrangement means:

- Heights are non-decreasing as we move toward the peak.
- Heights are non-increasing after the peak.
- The peak can consist of one tower or multiple adjacent towers with the same height.

The important detail is that every final height must satisfy:

```
0 < finalHeight[i] <= heights[i]
```

We want to maximize the total sum of all final tower heights.

The core challenge is deciding where the peak should be and how tall each surrounding tower can become while maintaining the mountain property.

For example, if we choose index `i` as the peak, then:

- Moving left from `i`, heights cannot increase beyond the next tower.
- Moving right from `i`, heights also cannot increase beyond the previous tower.

This creates a constrained propagation effect where the chosen peak determines the maximum legal heights on both sides.

The constraints are relatively small:

- `n <= 1000`

This means even an `O(n^2)` solution is acceptable. However, the problem is still designed to encourage monotonic stack thinking because the same idea appears in the harder version of the problem where `n` is much larger.

Several edge cases are important:

- Strictly increasing arrays, where the best peak is usually near the end.
- Strictly decreasing arrays, where the best peak is near the beginning.
- Multiple equal peaks, where flat plateaus are valid.
- Very small arrays like length `1`.
- Large height values up to `10^9`, which means sums may exceed 32-bit integer range.

## Approaches

### Brute Force Approach

The brute force idea is to try every index as the mountain peak.

Suppose we choose index `i` as the peak. Then:

- The peak height becomes `heights[i]`.
- Moving left, each tower can be at most the minimum of:

- its original height
- the height of the tower immediately to its right
- Moving right, each tower can be at most the minimum of:

- its original height
- the height of the tower immediately to its left

This guarantees the final arrangement is mountain-shaped.

For every possible peak:

1. Build the left side greedily.
2. Build the right side greedily.
3. Compute the total sum.
4. Track the maximum.

This works because once the peak is fixed, the optimal choice for every other tower is simply the largest valid height possible.

The brute force solution requires scanning the entire array for every peak, resulting in `O(n^2)` time complexity.

### Optimal Insight

The key observation is that when a tower becomes smaller than previous towers, it limits how far large heights can extend.

This is a classic monotonic stack pattern.

We compute:

- `left[i]` = maximum sum for subarray `[0...i]` if `i` is the highest point from the left perspective
- `right[i]` = maximum sum for subarray `[i...n-1]` if `i` is the highest point from the right perspective

Then:

```
total = left[i] + right[i] - heights[i]
```

We subtract `heights[i]` once because the peak gets counted twice.

A monotonic increasing stack helps efficiently determine how far a height can extend before a smaller value blocks it.

| --- | --- | --- | --- |

| Optimal | O(n) | O(n) | Uses monotonic stacks to compute contribution ranges efficiently |

## Algorithm Walkthrough

### Step 1: Compute left contributions

We create an array `left` where:

```
left[i] = maximum possible sum for subarray [0...i]
```

assuming tower `i` is the rightmost boundary.

We use a monotonic increasing stack.

For every index `i`:

- While the stack top has height greater than `heights[i]`, pop it.
- This finds the nearest smaller element on the left.
- If no smaller element exists, then `heights[i]` can extend all the way to index `0`.
- Otherwise, it extends only after the previous smaller element.

The recurrence becomes:

```
left[i] = left[prev] + heights[i] * (i - prev)
```

or:

```
left[i] = heights[i] * (i + 1)
```

if no previous smaller element exists.

### Step 2: Compute right contributions

We repeat the same process from right to left.

Now:

```
right[i] = maximum possible sum for subarray [i...n-1]
```

We again maintain a monotonic increasing stack.

This computes how far each height can extend toward the right.

### Step 3: Combine results

For every index `i`:

```
total = left[i] + right[i] - heights[i]
```

The subtraction avoids double-counting the peak.

We return the maximum value among all peaks.

### Why it works

The monotonic stack guarantees that for every tower, we quickly locate the nearest smaller height that limits expansion.

For any chosen peak, the optimal mountain arrangement always uses the maximum valid height at every position. Reducing a tower further would only decrease the total sum without helping satisfy any additional constraints.

The stack-based dynamic programming correctly accumulates the largest achievable sums for every prefix and suffix, so combining them produces the globally optimal mountain configuration.

## Python Solution

```python
from typing import List

class Solution:
    def maximumSumOfHeights(self, heights: List[int]) -> int:
        n = len(heights)

        left = [0] * n
        stack = []

        # Compute left contributions
        for i in range(n):
            while stack and heights[stack[-1]] > heights[i]:
                stack.pop()

            if stack:
                prev = stack[-1]
                left[i] = left[prev] + heights[i] * (i - prev)
            else:
                left[i] = heights[i] * (i + 1)

            stack.append(i)

        right = [0] * n
        stack = []

        # Compute right contributions
        for i in range(n - 1, -1, -1):
            while stack and heights[stack[-1]] > heights[i]:
                stack.pop()

            if stack:
                nxt = stack[-1]
                right[i] = right[nxt] + heights[i] * (nxt - i)
            else:
                right[i] = heights[i] * (n - i)

            stack.append(i)

        answer = 0

        for i in range(n):
            total = left[i] + right[i] - heights[i]
            answer = max(answer, total)

        return answer
```

The implementation begins by computing the `left` array. The monotonic stack stores indices with non-decreasing heights. Whenever we encounter a smaller height, larger heights are popped because they can no longer extend past the current position.

For each index, we determine how many positions can legally use the current height as their limiting value. Instead of recomputing sums repeatedly, we reuse previously computed prefix results stored in `left`.

The same logic is mirrored from right to left to build the `right` array.

Finally, each position is treated as the mountain peak. Since both arrays include the peak itself, we subtract one copy of `heights[i]` to avoid double-counting.

## Go Solution

```go
func maximumSumOfHeights(heights []int) int64 {
	n := len(heights)

	left := make([]int64, n)
	stack := []int{}

	// Compute left contributions
	for i := 0; i < n; i++ {
		for len(stack) > 0 && heights[stack[len(stack)-1]] > heights[i] {
			stack = stack[:len(stack)-1]
		}

		if len(stack) > 0 {
			prev := stack[len(stack)-1]
			left[i] = left[prev] + int64(heights[i]) * int64(i-prev)
		} else {
			left[i] = int64(heights[i]) * int64(i+1)
		}

		stack = append(stack, i)
	}

	right := make([]int64, n)
	stack = []int{}

	// Compute right contributions
	for i := n - 1; i >= 0; i-- {
		for len(stack) > 0 && heights[stack[len(stack)-1]] > heights[i] {
			stack = stack[:len(stack)-1]
		}

		if len(stack) > 0 {
			next := stack[len(stack)-1]
			right[i] = right[next] + int64(heights[i]) * int64(next-i)
		} else {
			right[i] = int64(heights[i]) * int64(n-i)
		}

		stack = append(stack, i)
	}

	var answer int64 = 0

	for i := 0; i < n; i++ {
		total := left[i] + right[i] - int64(heights[i])

		if total > answer {
			answer = total
		}
	}

	return answer
}
```

The Go version follows the exact same algorithmic structure as the Python solution. The main difference is explicit integer handling. Since sums may exceed 32-bit integer range, all accumulated values use `int64`.

Slices are used for both stacks and dynamic programming arrays. Stack popping is implemented using slice truncation.

## Edge Cases

### Single Tower

When the array contains only one tower, that tower automatically forms a valid mountain. A buggy implementation might incorrectly assume both left and right neighbors exist. The solution handles this naturally because both stack passes work correctly even for length `1`.

### Strictly Increasing Heights

Consider:

```
[1,2,3,4]
```

The optimal mountain is the entire array itself. Some implementations incorrectly reduce earlier values unnecessarily. The monotonic stack correctly recognizes that every tower can extend without restriction.

### Equal Height Plateaus

Consider:

```
[5,5,5,5]
```

A valid mountain may contain multiple adjacent peak towers. Incorrect solutions sometimes enforce a strictly increasing then strictly decreasing shape. This implementation uses `>` instead of `>=` in stack comparisons, allowing equal heights to remain and correctly supporting plateaus.

### Deep Valleys

Consider:

```
[10,1,10]
```

The center tower severely limits both sides. A naive greedy approach might incorrectly keep one of the `10`s unchanged. The algorithm properly propagates the minimum constraint outward, producing the valid optimal mountain `[1,1,1]` or `[10,1,1]` depending on chosen peak constraints, with maximum sum `12`.
