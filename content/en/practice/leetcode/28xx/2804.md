## Solution

## Problem Understanding

The problem asks us to implement our own version of JavaScript's `Array.prototype.forEach` method. The goal is to enhance all arrays so that every array instance can call:

```
array.forEach(callback, context)
```

The method should iterate through every element in the array and execute the provided `callback` function for each element. Unlike methods such as `map`, this function does not create or return a new array. Its purpose is purely side effects, such as modifying the original array or interacting with external state.

The callback receives three arguments during each iteration:

```
callback(currentValue, index, array)
```

The arguments mean:

- `currentValue` is the current element in the iteration
- `index` is the current position
- `array` is the original array itself

The second parameter, `context`, determines what the `this` keyword refers to inside the callback function. This is important because JavaScript functions can execute with different `this` bindings depending on how they are invoked.

For example:

```
callback.call(context, value, index, array)
```

ensures that inside the callback, `this === context`.

The constraints tell us that the array length can be as large as `10^5`. This immediately suggests that the solution must run in linear time. Any nested iteration or repeated copying of the array would be unnecessary and inefficient.

Several edge cases are important:

- The array may be empty
- The callback may mutate the array during iteration
- The callback may depend on the `this` context
- Elements can contain any valid JSON values
- The method should return `undefined`, just like the native `forEach`

The problem also explicitly asks us not to rely on built in array iteration methods, so we should implement the traversal manually using a loop.

## Approaches

### Brute Force Approach

A naive implementation could create a temporary copy of the array and then repeatedly process elements from the copy while updating the original array. This approach still works correctly because every element is visited and the callback executes with the proper arguments and context.

However, copying the array introduces unnecessary overhead. Since the problem only requires iterating through the original array once, allocating extra memory provides no benefit.

This approach is correct because every index is still processed exactly once, but it wastes both time and space.

### Optimal Approach

The key observation is that `forEach` fundamentally performs a single sequential traversal of the array.

We simply need to:

1. Iterate from index `0` to `arr.length - 1`
2. Invoke the callback on each element
3. Bind the callback's `this` value to the provided context

The cleanest way to achieve the correct context binding is to use:

```
callback.call(context, value, index, array)
```

This directly invokes the function while explicitly setting `this`.

Since we only traverse the array once and do not allocate additional data structures, this solution is optimal.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Copies the array before iteration |
| Optimal | O(n) | O(1) | Single traversal with direct callback execution |

## Algorithm Walkthrough

1. Define a new method on `Array.prototype` named `forEach`.

This allows every JavaScript array to access the method automatically.
2. Accept two parameters: `callback` and `context`.

The callback is the function to execute for each element, and the context determines the `this` binding.
3. Start a loop from index `0` to `this.length - 1`.

The keyword `this` refers to the array instance on which `forEach` was called.
4. During each iteration, retrieve the current element.

The value is stored as:

```
const value = this[index]
```
5. Execute the callback using `call`.

```
callback.call(context, value, index, this)
```

This guarantees:

- `value` becomes `currentValue`
- `index` becomes the current index
- `this` becomes the original array
- inside the callback, `this === context`
6. Continue until every index has been processed.
7. Do not return anything.

The native `forEach` method returns `undefined`, so our implementation should behave the same way.

### Why it works

The algorithm works because it processes every valid index exactly once in increasing order. Each callback invocation receives the correct value, index, and array reference. Using `Function.prototype.call` guarantees that the callback executes with the required `this` binding. Since no indices are skipped and no extra transformations occur, the behavior matches the expected semantics of `forEach`.

## Python Solution

Although the original problem is written for JavaScript array prototypes, below is a Python equivalent that demonstrates the same logic and behavior.

```python
from typing import Any, Callable, List

class Solution:
    def forEach(
        self,
        arr: List[Any],
        callback: Callable[[Any, int, List[Any], Any], None],
        context: Any,
    ) -> None:
        for index in range(len(arr)):
            callback(arr[index], index, arr, context)
```

The implementation performs a simple linear traversal of the array.

The loop iterates over every valid index from `0` to `len(arr) - 1`. During each iteration, the callback is executed with:

- the current value
- the current index
- the original array
- the provided context

Python does not have JavaScript style `this` binding, so the context is passed explicitly as an additional argument. Conceptually, however, this mirrors the JavaScript behavior.

The method intentionally does not return anything, matching the semantics of `forEach`.

## Go Solution

```go
package main

type Callback func(value interface{}, index int, arr []interface{}, context interface{})

func forEach(arr []interface{}, callback Callback, context interface{}) {
	for index := 0; index < len(arr); index++ {
		callback(arr[index], index, arr, context)
	}
}
```

The Go implementation mirrors the same iteration logic.

Go does not support JavaScript style prototype extension or dynamic `this` binding, so the callback receives the context explicitly as a parameter.

Slices in Go already behave similarly to dynamic arrays, so iterating with a standard `for` loop is both efficient and idiomatic.

No additional memory is allocated beyond the loop variables.

## Edge Cases

### Empty Array

An empty array is one of the most common edge cases for iteration problems. A buggy implementation might accidentally attempt to access elements or invoke the callback unnecessarily.

Our implementation handles this naturally because the loop condition immediately fails when the array length is zero. No callback executions occur, and the function exits correctly.

### Callback Mutates the Array

The callback may modify elements while iteration is still ongoing. This can be dangerous if the implementation relies on cached values or copied arrays.

Our implementation directly accesses `arr[index]` during each iteration, so any updates made by earlier iterations are reflected naturally in later iterations. This matches standard `forEach` behavior.

### Context Dependent Logic

Some callbacks rely heavily on the `this` value. Forgetting to bind the context properly would cause incorrect behavior or runtime errors.

Using:

```
callback.call(context, value, index, array)
```

ensures that the callback always executes with the expected `this` binding. This guarantees correctness even for callbacks that depend entirely on context state.
