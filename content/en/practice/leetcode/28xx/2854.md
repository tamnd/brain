## Solution

## Problem Understanding

The problem gives us a database table named `Steps`. Each row represents the number of steps a particular user walked on a specific date. The table contains three columns:

| Column | Meaning |
| --- | --- |
| `user_id` | The identifier of the user |
| `steps_count` | The number of steps walked on that date |
| `steps_date` | The date associated with the step count |

The pair `(user_id, steps_date)` is unique, which means a user can only have one entry per day.

The task is to compute a `3-day rolling average` for each user. A rolling average ending on a given day is defined as the average of the step counts for that day and the two immediately preceding consecutive days.

The key requirement is that the three days must be consecutive calendar days. If even one day is missing, the rolling average is not defined for that ending date.

For example, suppose a user has entries for:

| Date | Steps |
| --- | --- |
| Sep 4 | 395 |
| Sep 5 | 499 |
| Sep 6 | 712 |

These are three consecutive days, so the rolling average for Sep 6 is:

$$(395 + 499 + 712) / 3 = 535.33$$

However, if the dates were Sep 2, Sep 4, and Sep 5, then no rolling average could be computed for Sep 5 because Sep 3 is missing.

The output should contain:

| Column | Meaning |
| --- | --- |
| `user_id` | The user identifier |
| `steps_date` | The ending date of the 3-day window |
| `rolling_average` | The average step count over the 3 consecutive days |

The result must be ordered by `user_id` and `steps_date` in ascending order.

The problem is fundamentally about identifying valid consecutive 3-day windows and computing averages only for those windows.

A naive implementation can easily make mistakes in several areas. One common bug is checking only whether there are three rows in sequence, instead of verifying that the dates themselves are consecutive. Another issue is incorrectly mixing data from different users. Since rolling averages are computed independently for each user, every calculation must remain within a single user's records.

The input guarantees that `(user_id, steps_date)` is unique, so duplicate dates for the same user do not need special handling. The problem does not specify extremely large constraints, but this is a database problem intended to test SQL window functions and date arithmetic efficiently.

## Approaches

### Brute Force Approach

The brute force approach examines every row independently and tries to determine whether the previous two consecutive dates exist for the same user.

For each row:

1. Find the row for `steps_date - 1 day`
2. Find the row for `steps_date - 2 days`
3. If both exist, compute the average of all three rows
4. Otherwise, skip the current row

This approach is correct because it explicitly verifies the exact requirement of three consecutive dates. If all three dates exist, the rolling average is computed directly from those values.

However, this method becomes inefficient when implemented naively because each row may need to scan the table multiple times to locate the previous dates. If there are `n` rows, and each lookup costs `O(n)`, the total complexity becomes `O(n^2)`.

Even if indexes improve practical performance in SQL databases, conceptually this is still a repetitive and inefficient strategy.

### Optimal Approach

The key insight is that rolling averages naturally fit the concept of window functions.

If we sort each user's records by date, we can examine sliding windows of three rows at a time. However, simply taking three adjacent rows is not enough because the dates must be consecutive.

The optimal solution therefore combines two ideas:

1. Use window functions to access the previous two rows efficiently
2. Verify that the date differences are exactly one day apart

For each row, we retrieve:

- The previous date and step count
- The date before that and its step count

Then we check:

- Current date minus previous date equals 1 day
- Previous date minus second previous date equals 1 day

If both conditions hold, we compute the average.

This approach avoids repeated table scans and processes the data efficiently in a single ordered traversal per user.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Repeatedly searches for previous dates |
| Optimal | O(n log n) | O(n) | Uses sorting and window functions efficiently |

The sorting cost dominates the complexity because window functions operate over ordered partitions.

## Algorithm Walkthrough

1. Partition the table by `user_id`.

Rolling averages must be computed independently for each user. Partitioning ensures that data from different users never mixes together.
2. Sort each user's rows by `steps_date`.

Rolling averages depend on chronological order. Sorting guarantees that previous rows correspond to earlier dates.
3. Use window functions to retrieve the previous two rows.

For every row, retrieve:

- The previous day's date and step count
- The date two rows earlier and its step count

This is typically done using `LAG()`.
4. Verify date consecutiveness.

Adjacent rows are not necessarily consecutive dates. We must check:

- `current_date = previous_date + 1 day`
- `previous_date = second_previous_date + 1 day`

Only when both conditions are true do we have a valid 3-day window.
5. Compute the rolling average.

Add the three step counts together and divide by 3.
6. Round the result to two decimal places.

The problem explicitly requires formatting to two decimal places.
7. Return the final rows ordered by `user_id` and `steps_date`.

### Why it works

The algorithm works because every valid rolling average must end at a row whose previous two rows represent the immediately preceding calendar days for the same user.

By partitioning by user and sorting by date, the window functions correctly expose the relevant neighboring rows. The consecutive date checks guarantee that no missing days exist in the window. Therefore, every produced average corresponds exactly to a valid 3-day consecutive sequence, and every valid sequence is discovered once.

## Python Solution

```python
# Write your MySQL query statement below

SELECT
    user_id,
    steps_date,
    ROUND(
        (
            steps_count +
            prev_steps_1 +
            prev_steps_2
        ) / 3,
        2
    ) AS rolling_average
FROM (
    SELECT
        user_id,
        steps_date,
        steps_count,
        LAG(steps_count, 1) OVER (
            PARTITION BY user_id
            ORDER BY steps_date
        ) AS prev_steps_1,
        LAG(steps_count, 2) OVER (
            PARTITION BY user_id
            ORDER BY steps_date
        ) AS prev_steps_2,
        LAG(steps_date, 1) OVER (
            PARTITION BY user_id
            ORDER BY steps_date
        ) AS prev_date_1,
        LAG(steps_date, 2) OVER (
            PARTITION BY user_id
            ORDER BY steps_date
        ) AS prev_date_2
    FROM Steps
) AS ordered_steps
WHERE
    DATEDIFF(steps_date, prev_date_1) = 1
    AND DATEDIFF(prev_date_1, prev_date_2) = 1
ORDER BY user_id, steps_date;
```

The solution begins by constructing a derived table named `ordered_steps`. Inside this derived table, we use multiple `LAG()` window functions.

The `PARTITION BY user_id` clause ensures that each user's records are processed independently. The `ORDER BY steps_date` clause guarantees chronological ordering within each user partition.

We retrieve:

- The previous step count
- The step count two rows earlier
- The previous date
- The date two rows earlier

Once these values are available, the outer query filters the rows using two `DATEDIFF()` checks. These conditions ensure that the three rows correspond to consecutive calendar days.

Finally, the rolling average is computed by summing the three step counts and dividing by 3. The `ROUND(..., 2)` function formats the result to two decimal places as required.

Since this is a database problem, the submitted solution is SQL rather than a traditional Go function implementation. The same SQL query is used regardless of language selection on LeetCode.

One important implementation detail is that `LAG()` returns `NULL` when insufficient previous rows exist. The `DATEDIFF()` conditions automatically eliminate such rows because comparisons involving `NULL` do not evaluate to true.

## Edge Cases

### Missing Dates Between Rows

One of the most important edge cases occurs when a user has three rows that are adjacent after sorting, but the dates are not consecutive.

For example:

| Date | Steps |
| --- | --- |
| Sep 1 | 100 |
| Sep 3 | 200 |
| Sep 4 | 300 |

A naive implementation that simply averages every three adjacent rows would incorrectly produce a result for Sep 4.

The solution prevents this by explicitly checking date differences using `DATEDIFF()`. Since Sep 3 minus Sep 1 equals 2 days, the row is rejected.

### Users With Fewer Than Three Records

A user may have only one or two rows in the table.

In this situation, `LAG()` returns `NULL` for missing previous rows. The date difference checks therefore fail automatically, and no rolling average is produced.

This avoids the need for special-case logic.

### Consecutive Dates Across Month or Year Boundaries

Calendar arithmetic can be tricky around month endings and year endings.

For example:

| Date |
| --- |
| 2020-12-30 |
| 2020-12-31 |
| 2021-01-01 |

These are still consecutive days even though the month and year changed.

Using `DATEDIFF()` is important because it handles actual calendar arithmetic correctly. Manual string comparisons or numeric day extraction could easily introduce bugs here.

### Multiple Overlapping Windows

A long consecutive sequence should produce overlapping rolling averages.

For example:

| Date |
| --- |
| Sep 1 |
| Sep 2 |
| Sep 3 |
| Sep 4 |

This should generate:

- Window ending Sep 3
- Window ending Sep 4

The window function approach naturally supports overlapping windows because every row independently examines its two previous rows.
