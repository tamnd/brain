## Solution

## Problem Understanding

The problem gives us an integer array `nums` and an integer `k`. We are allowed to delete at most `k` elements from the array. After performing those deletions, we want to find the longest possible subarray in which every remaining element is identical.

The important detail is that deletions do not need to happen only inside the final subarray. Instead, we can delete elements anywhere so that some contiguous section becomes composed entirely of the same value.

Another way to think about the problem is this: for any chosen value `x`, we want to find the largest contiguous range that can be transformed into all `x` values by deleting at most `k` non-`x` elements.

For example, consider:

```
nums = [1,3,2,3,1,3], k = 3
```

If we focus on the value `3`, the positions containing `3` are:

```
[1, 3, 5]
```

Between these positions, there are non-`3` values that must be deleted if we want all `3`s to become contiguous. The total number of deletions required determines whether the range is valid.

The constraints are important:

- `nums.length` can be as large as `10^5`
- Values can also be up to `10^5`

This immediately tells us that quadratic solutions will be too slow. Any solution near `O(n^2)` will time out for the largest inputs. We need a solution that is approximately linear or `O(n log n)`.

There are several edge cases worth noticing early:

- Arrays where all elements are already equal
- `k = 0`, meaning no deletions are allowed
- Arrays where every element is distinct
- Very large contiguous blocks of the same value
- Cases where the optimal subarray is formed only after deleting elements between distant matching values

The problem guarantees that the array is non-empty, so the minimum possible answer is always at least `1`.

## Approaches

### Brute Force Approach

A straightforward approach is to examine every possible subarray and determine whether it can become equal after deleting at most `k` elements.

For each subarray:

1. Count the frequency of each value.
2. Find the maximum frequency inside the subarray.
3. The number of deletions needed is:

```
subarray_length - maximum_frequency
```

If that value is at most `k`, then the subarray can be transformed into an equal subarray.

This works because we can keep the most frequent value and delete everything else.

However, there are `O(n^2)` subarrays, and counting frequencies for each one can cost another `O(n)` in the worst case. Even with optimizations, the complexity remains too large for `n = 10^5`.

### Optimal Approach

The key insight is that we do not need to examine arbitrary subarrays directly. Instead, we can process each distinct value independently.

Suppose we focus on a single value `x`.

We store all indices where `x` appears. For example:

```
nums = [1,3,2,3,1,3]

positions of 3 = [1,3,5]
```

Now consider a window inside this positions array.

If we choose positions:

```
[1,3,5]
```

then the total span in the original array is:

```
5 - 1 + 1 = 5
```

Inside this span, we only have `3` occurrences equal to:

```
window_size = 3
```

So the number of deletions required is:

```
5 - 3 = 2
```

This formula generalizes nicely.

For positions array `pos`:

```
deletions_needed =
(pos[right] - pos[left] + 1) - (right - left + 1)
```

The first part is the full span length in the original array.

The second part is how many target values already exist.

The difference equals how many non-target values must be removed.

Since deletions must be at most `k`, we can use a sliding window over each positions list.

This gives an efficient linear-time solution overall.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) to O(n³) | O(n) | Checks every subarray and counts frequencies |
| Optimal | O(n) | O(n) | Uses grouped indices and sliding window |

## Algorithm Walkthrough

1. Create a hash map where each key is a number from `nums`, and the value is a list of all indices where that number appears.

For example:

```
nums = [1,1,2,2,1,1]

positions:
1 -> [0,1,4,5]
2 -> [2,3]
```
2. Initialize a variable `best` to store the maximum equal subarray length found so far.
3. Process each positions list independently.

Suppose we are processing:

```
pos = [0,1,4,5]
```
4. Use a sliding window with two pointers, `left` and `right`.

The window represents a group of occurrences of the same value that we want to keep.
5. For each `right` index, compute how many deletions are needed to make all occurrences inside the window contiguous.

The formula is:

```
deletions =
(pos[right] - pos[left] + 1)
- (right - left + 1)
```

Explanation:

- `pos[right] - pos[left] + 1` is the total span length
- `right - left + 1` is how many matching values we already have
- Their difference is how many unwanted elements lie between them
6. If deletions exceed `k`, move `left` forward until the window becomes valid again.
7. After each adjustment, compute the current window size:

```
current_length = right - left + 1
```

Update `best` if this is larger.
8. Continue until all positions lists are processed.
9. Return `best`.

### Why it works

For any chosen value, the only elements we want to keep are occurrences of that value. Any different element between two kept occurrences must be deleted. The sliding window maintains the largest valid group of occurrences whose required deletions do not exceed `k`. Since every possible valid group is examined exactly once, the algorithm always finds the optimal answer.

## Python Solution

```python
from collections import defaultdict
from typing import List

class Solution:
    def longestEqualSubarray(self, nums: List[int], k: int) -> int:
        positions = defaultdict(list)

        for index, value in enumerate(nums):
            positions[value].append(index)

        best = 0

        for pos in positions.values():
            left = 0

            for right in range(len(pos)):
                while (
                    pos[right] - pos[left] + 1
                    - (right - left + 1)
                ) > k:
                    left += 1

                best = max(best, right - left + 1)

        return best
```

The implementation begins by grouping indices for each distinct number using a hash map. This allows us to process each value independently instead of repeatedly scanning the full array.

For every positions list, we apply a sliding window. The `left` and `right` pointers define which occurrences of the current value are included.

The critical calculation is:

```python
pos[right] - pos[left] + 1 - (right - left + 1)
```

This computes how many non-matching elements exist between the chosen occurrences. If that count exceeds `k`, the window is invalid, so we shrink it from the left.

Once the window becomes valid again, the number of kept equal elements is simply:

```python
right - left + 1
```

We continuously update the global maximum answer.

## Go Solution

```go
func longestEqualSubarray(nums []int, k int) int {
	positions := make(map[int][]int)

	for index, value := range nums {
		positions[value] = append(positions[value], index)
	}

	best := 0

	for _, pos := range positions {
		left := 0

		for right := 0; right < len(pos); right++ {
			for (pos[right]-pos[left]+1)-(right-left+1) > k {
				left++
			}

			length := right - left + 1
			if length > best {
				best = length
			}
		}
	}

	return best
}
```

The Go implementation follows the same logic as the Python version. A map from integer to slice of indices replaces Python's `defaultdict`.

Slices in Go are dynamic and work naturally for storing positions. Integer overflow is not a concern because the maximum index difference is at most `10^5`.

Unlike Python, Go requires explicit maximum comparisons instead of using the built-in `max` function.

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every index enters and leaves a sliding window at most once |

The total number of stored indices across all lists equals `n`. During sliding window processing, both pointers move only forward, so the combined work across all lists is linear.

## Edge Cases

One important edge case occurs when all elements are already equal. For example:

```
[7,7,7,7]
```

No deletions are necessary, and the answer should simply be the array length. Some implementations incorrectly attempt unnecessary shrinking or overcount deletions. This solution handles the case naturally because the deletion formula always evaluates to zero.

Another important edge case is when `k = 0`. In this scenario, no elements may be removed, so the algorithm must find the longest already-contiguous block of equal values. The sliding window correctly enforces this because any gap immediately makes the deletion count positive and forces the window to shrink.

A third tricky case appears when matching values are far apart. Consider:

```
[1,2,3,4,1]
```

Even though there are two `1`s, connecting them requires deleting three elements. If `k < 3`, they cannot belong to the same equal subarray. The sliding window formula precisely measures the number of required deletions, preventing invalid windows from being counted.

A final edge case occurs when every value is distinct. In that situation, every positions list contains only one index. The algorithm still works correctly because every window size is exactly `1`, which is the maximum achievable equal subarray length.
