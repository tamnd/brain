# LeetCode 2803 - Factorial Generator

## Problem Understanding

This problem asks us to implement a generator function that produces the factorial sequence from `1!` up to `n!`.

The factorial of a number is defined as:

$n! = n \times (n-1) \times (n-2) \times \cdots \times 2 \times 1$

Additionally, the problem guarantees that:

$0! = 1$

The input `n` represents the largest factorial we need to generate. Instead of returning an array directly, we must return a generator object. Every time the caller invokes `next()`, the generator should yield the next factorial value in sequence.

For example, if `n = 5`, the generator should yield:

- `1! = 1`
- `2! = 2`
- `3! = 6`
- `4! = 24`
- `5! = 120`

The output shown in the examples is represented as an array only for illustration purposes. Internally, the implementation must lazily produce the values one at a time.

The constraints are very small:

- `0 <= n <= 18`

This tells us several important things:

- Performance is not a major concern because at most 19 values are generated.
- Factorial values remain within JavaScript's safe integer range up to `18!`.
- A straightforward iterative solution is completely sufficient.

There are a few edge cases worth identifying early:

- `n = 0` is special because the factorial sequence still contains one value, `1`.
- Small values like `n = 1` can expose off by one mistakes in loops.
- A naive recursive factorial computation repeated for every yield could introduce unnecessary repeated work.

The safest and cleanest approach is to maintain a running factorial value and update it incrementally as the generator progresses.

## Approaches

### Brute Force Approach

A brute force solution would compute every factorial independently from scratch.

For each number `k` from `1` to `n`, we could compute:

$k! = 1 \times 2 \times 3 \times \cdots \times k$

This works because factorials are mathematically independent and each value can always be recomputed directly.

However, this approach performs repeated multiplication work. For example:

- `5!` recomputes the work already done for `4!`
- `4!` recomputes the work already done for `3!`

Even though the constraints are small, this redundancy makes the solution less elegant and less efficient.

### Optimal Approach

The key observation is that factorials build directly on previous factorials.

Specifically:

$n! = n \times (n-1)!$

This means that once we know `(n-1)!`, we can compute `n!` with only one additional multiplication.

Instead of recomputing each factorial from scratch, we maintain a running product:

- Start with `current = 1`
- Multiply by the next integer each time
- Yield the updated value

This gives a clean iterative generator solution with linear time complexity.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Recomputes each factorial independently |
| Optimal | O(n) | O(1) | Maintains a running factorial incrementally |

## Algorithm Walkthrough

1. Initialize a variable called `currentFactorial` to `1`.

This variable stores the latest factorial value that has already been computed.
2. Handle the special case where `n == 0`.

Since `0! = 1`, the generator should yield `1` immediately and stop.
3. Iterate from `1` to `n`.

Each iteration represents computing the next factorial in the sequence.
4. Multiply the running factorial by the current number.

If the current number is `i`, then:

$i! = i \times (i-1)!$

Since `(i-1)!` is already stored in `currentFactorial`, we only need one multiplication.
5. Yield the updated factorial value.

The generator produces the value lazily, one at a time.
6. Continue until all factorials through `n!` have been generated.

### Why it works

The algorithm maintains the invariant that before processing integer `i`, the variable `currentFactorial` stores `(i-1)!`.

After multiplying by `i`, the variable becomes `i!`, which is exactly the next factorial value we need to yield.

Because factorials follow the recurrence relation:

$i! = i \times (i-1)!$

the algorithm correctly generates every factorial in order without recomputation.

## Python Solution

```python
from typing import Generator

class Solution:
    def factorial(self, n: int) -> Generator[int, None, None]:
        current_factorial = 1

        if n == 0:
            yield 1
            return

        for number in range(1, n + 1):
            current_factorial *= number
            yield current_factorial
```

The implementation begins by creating a running factorial variable initialized to `1`.

The special case `n == 0` is handled separately because the factorial sequence for zero contains exactly one value, `1`.

For all other values, the loop iterates from `1` through `n`. During each iteration, the current number is multiplied into the running factorial, producing the next factorial value in sequence.

The `yield` keyword makes this a generator function, allowing values to be produced lazily instead of storing them all in memory at once.

## Go Solution

```go
func factorial(n int) <-chan int64 {
    ch := make(chan int64)

    go func() {
        defer close(ch)

        var currentFactorial int64 = 1

        if n == 0 {
            ch <- 1
            return
        }

        for number := 1; number <= n; number++ {
            currentFactorial *= int64(number)
            ch <- currentFactorial
        }
    }()

    return ch
}
```

Go does not support generators natively in the same way Python does, so the implementation uses a channel to simulate generator behavior.

A goroutine computes factorial values and sends them through the channel one at a time. The caller can iterate over the channel lazily, which closely matches generator semantics.

The implementation uses `int64` because factorial values grow quickly and could overflow smaller integer types.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | One multiplication and one yield per factorial |
| Space | O(1) | Only a few variables are stored |

The algorithm processes each integer from `1` through `n` exactly once. No nested loops or repeated factorial recomputation occur.

The space complexity remains constant because the algorithm stores only the running factorial and loop variable. The yielded values are produced lazily instead of stored in a separate collection.

## Edge Cases

### Edge Case 1: `n = 0`

This is the most important special case in the problem. Many implementations accidentally return an empty generator because there are no positive numbers to iterate through.

However, mathematically:

$0! = 1$

The implementation explicitly checks for `n == 0`, yields `1`, and terminates immediately.

### Edge Case 2: `n = 1`

Small inputs often expose off by one errors in loops.

For `n = 1`, the generator must produce exactly one value:

```
[1]
```

The loop correctly starts at `1` and includes `n`, ensuring that `1!` is yielded properly.

### Edge Case 3: Large Values Near the Constraint Limit

Factorials grow extremely quickly. Even though the constraint only goes up to `18`, careless implementations using small integer types could overflow.

The Go solution uses `int64` to safely store values up to `18!`, while Python integers automatically expand to arbitrary precision.

This guarantees correct results throughout the full input range.
