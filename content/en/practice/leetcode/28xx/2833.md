## Solution

## Problem Understanding

The problem gives us a string called `moves`, where each character represents one move on a number line. We always start at position `0`.

Each character can mean the following:

- `'L'` means we must move one step left.
- `'R'` means we must move one step right.
- `'_'` means we are allowed to choose either left or right for that move.

After performing all moves, we want to maximize the distance from the origin. The important detail is that the answer is the absolute distance from `0`, not necessarily the final coordinate itself.

For example, if we end at `-5`, the distance from the origin is `5`. If we end at `+5`, the distance is also `5`.

The key freedom in the problem comes from the `'_'` characters. Every underscore can be converted into either `'L'` or `'R'` in whatever way helps us move farther from the origin.

The constraints are very small:

- `1 <= n <= 50`
- The string contains only `'L'`, `'R'`, and `'_'`

Because the input size is tiny, even brute force solutions are technically feasible. However, the problem has a very clean mathematical observation that leads to a much simpler and more efficient solution.

Several edge cases are important:

- A string containing only `'_'`, because every move can be chosen freely.
- A string where left and right moves already cancel each other out.
- A string with no underscores, where the result is simply the final absolute position.
- Cases where choosing all underscores in the same direction is optimal, which turns out to always be true.

## Approaches

### Brute Force Approach

A brute force solution would try every possible replacement for the `'_'` characters.

Suppose there are `k` underscores. Each underscore has two choices:

- replace with `'L'`
- replace with `'R'`

That creates `2^k` possible movement sequences.

For each sequence, we could simulate the moves and compute the final position. Then we take the maximum absolute value among all possibilities.

This works because it exhaustively checks every valid configuration. Since all possibilities are explored, the maximum distance found must be correct.

However, this approach becomes inefficient when the number of underscores increases. In the worst case, all 50 characters could be underscores, producing `2^50` combinations, which is completely impractical.

### Optimal Approach

The important observation is that every underscore should be used to strengthen whichever direction already dominates.

Suppose:

- `L` appears `left_count` times
- `R` appears `right_count` times
- `'_'` appears `blank_count` times

Without underscores, the current displacement is:

```
right_count - left_count
```

The absolute distance is:

```
|right_count - left_count|
```

Every underscore can increase this distance by exactly `1` if we choose its direction correctly.

Therefore, the best strategy is:

- if right moves dominate, convert all underscores into `'R'`
- if left moves dominate, convert all underscores into `'L'`

That gives the formula:

```
abs(right_count - left_count) + blank_count
```

This works because every underscore contributes positively toward the already larger side instead of canceling movement.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^k × n) | O(1) | Tries every replacement for underscores |
| Optimal | O(n) | O(1) | Counts characters and applies a mathematical observation |

## Algorithm Walkthrough

1. Initialize three counters:

- `left_count`
- `right_count`
- `blank_count`

These counters track how many times each type of move appears.
2. Iterate through every character in the string.

For each character:

- increment `left_count` if the character is `'L'`
- increment `right_count` if the character is `'R'`
- increment `blank_count` if the character is `'_'`
3. Compute the current imbalance between left and right moves.

The net displacement without underscores is:

```
right_count - left_count
```

Since we only care about distance from the origin, take the absolute value:

```
abs(right_count - left_count)
```
4. Use every underscore to increase the distance.

Every underscore can be assigned to the stronger direction, increasing the distance by `1`.

Therefore:

```
answer = abs(right_count - left_count) + blank_count
```
5. Return the computed answer.

### Why it works

The key invariant is that each underscore can always be chosen independently to maximize the final displacement.

If the current position trends right, converting an underscore into `'R'` increases the displacement by `1`. Converting it into `'L'` would reduce the displacement instead. The same logic applies symmetrically when left moves dominate.

Therefore, the optimal strategy is always to assign every underscore to the same direction as the current majority side, guaranteeing the maximum possible distance.

## Python Solution

```python
class Solution:
    def furthestDistanceFromOrigin(self, moves: str) -> int:
        left_count = 0
        right_count = 0
        blank_count = 0

        for move in moves:
            if move == 'L':
                left_count += 1
            elif move == 'R':
                right_count += 1
            else:
                blank_count += 1

        return abs(right_count - left_count) + blank_count
```

The implementation follows the algorithm directly.

The first section initializes counters for left moves, right moves, and undecided moves. We then scan through the string once and update the appropriate counter depending on the character.

After counting, the code computes the existing imbalance between left and right moves using:

```python
abs(right_count - left_count)
```

Finally, every underscore is added to that imbalance because each one can be chosen to extend the dominant direction.

The solution is concise because the mathematical observation eliminates the need for simulation or dynamic programming.

## Go Solution

```go
func furthestDistanceFromOrigin(moves string) int {
    leftCount := 0
    rightCount := 0
    blankCount := 0

    for _, move := range moves {
        if move == 'L' {
            leftCount++
        } else if move == 'R' {
            rightCount++
        } else {
            blankCount++
        }
    }

    difference := rightCount - leftCount
    if difference < 0 {
        difference = -difference
    }

    return difference + blankCount
}
```

The Go implementation is almost identical to the Python version.

One small difference is that Go does not provide a built in integer `abs()` function for plain integers in the standard language syntax, so we manually convert negative values into positive ones.

The iteration uses Go's `range` loop over the string, which yields runes. Since the input contains only ASCII characters, rune handling is straightforward.

## Edge Cases

One important edge case is when the string contains only underscores. In this situation, there are no fixed directions at all, so every move can be chosen freely. A buggy implementation might incorrectly assume there must already be a dominant direction. The correct behavior is to choose the same direction for every move, producing a distance equal to the string length.

Another important edge case is when left and right moves perfectly cancel each other out. For example, `"LRLR"` results in a net displacement of zero. If underscores exist in such a case, they become entirely responsible for the final distance. The formula correctly handles this because the imbalance becomes zero and only the underscore count remains.

A third edge case is when the input contains no underscores. In that scenario, there is no flexibility at all. The answer must simply be the absolute difference between right and left moves. The implementation naturally handles this because `blank_count` becomes zero.

A final subtle case occurs when one direction already dominates heavily. For example, `"LLLL_R"` strongly favors left movement. A naive strategy might attempt to balance moves, but balancing actually reduces distance from the origin. The optimal algorithm always assigns underscores to the already dominant direction, maximizing the absolute displacement correctly.
