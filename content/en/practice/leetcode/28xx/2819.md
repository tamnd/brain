**Difficulty:** 🔴 Hard  
**Topics:** Array, Binary Search, Sorting, Prefix Sum

## Solution

## Problem Understanding

We are given an array `prices`, where each element represents the price of a chocolate. For every query `[k, m]`, Bob must choose exactly `m` chocolates and pay according to the following rule:

- If a chocolate costs at most `k`, Bob pays the entire price.
- If a chocolate costs more than `k`, Bob only pays `k`, and Alice pays the remaining amount.

For every chosen chocolate with price `p`:

- If `p <= k`, Bob contributes `p` and Alice contributes `0`.
- If `p > k`, Bob contributes `k` and Alice contributes `p - k`.

The objective is to minimize Bob’s relative loss:

$$\text{relative loss} = \text{BobPaid} - \text{AlicePaid}$$

We must compute the minimum possible value for every query.

The constraints are large:

- Up to `10^5` chocolates
- Up to `10^5` queries
- Prices up to `10^9`

This immediately tells us that any per-query algorithm slower than roughly `O(log n)` or `O(log^2 n)` will likely time out. A naive enumeration of all subsets is impossible.

The key challenge is understanding how each chocolate contributes to the relative loss.

Suppose a chocolate price is `p`.

If `p <= k`:

$$\text{contribution} = p - 0 = p$$

If `p > k`:

$$\text{contribution} = k - (p-k) = 2k - p$$

So every chocolate contributes independently:

$$f(p)=
\begin{cases}
p & p \le k \\
2k-p & p > k
\end{cases}$$

For a fixed query, we need to choose exactly `m` chocolates with the smallest total contribution.

Several edge cases are important:

- All chocolates may be cheaper than `k`
- All chocolates may be more expensive than `k`
- `m` may equal `n`, forcing selection of every chocolate
- Multiple chocolates may have identical prices
- Contributions for expensive chocolates may become negative, which is actually beneficial because Bob wants to minimize the total

These observations drive the optimal strategy.

## Approaches

### Brute Force Approach

A brute force solution would consider every possible subset of size `m`.

For each subset:

1. Compute Bob’s payment
2. Compute Alice’s payment
3. Calculate the relative loss
4. Track the minimum value

This works because it directly evaluates every valid selection.

However, this is computationally impossible. The number of subsets is:

$$\binom{n}{m}$$

which grows exponentially. Even for `n = 40`, this becomes enormous, while the actual constraint is `10^5`.

Another slightly improved brute force idea would compute the contribution of every chocolate for a query, sort those contributions, and pick the smallest `m`.

That reduces complexity to:

- Compute contributions: `O(n)`
- Sort: `O(n log n)`

Per query.

With `10^5` queries, this becomes:

$$O(q \cdot n \log n)$$

which is still far too slow.

### Key Insight

The important observation is that the contribution function has a special structure.

For a fixed `k`:

$$f(p)=
\begin{cases}
p & p \le k \\
2k-p & p > k
\end{cases}$$

For chocolates with `p > k`, larger prices produce smaller contributions because:

$$2k-p$$

decreases as `p` increases.

Therefore:

- Among chocolates above `k`, we prefer the most expensive ones.
- Among chocolates at most `k`, we prefer the cheapest ones.

This creates a clean optimal structure.

Suppose:

- `x` chocolates are chosen from prices `<= k`
- `m-x` chocolates are chosen from prices `> k`

To minimize the total:

- Choose the `x` smallest prices from the low group
- Choose the `(m-x)` largest prices from the high group

Now we determine the optimal `x`.

A crucial observation simplifies everything further:

Every chocolate above `k` has contribution:

$$2k-p < k$$

Every chocolate below or equal to `k` has contribution:

$$p \ge 1$$

Since larger expensive chocolates reduce the loss more aggressively, the optimal strategy is:

1. Take as many chocolates as possible from the expensive side
2. Fill remaining spots with the cheapest cheap chocolates

After sorting prices and using prefix sums, every query can be answered with binary search and arithmetic.

## Algorithm Walkthrough

### Step 1: Sort the prices

Sort the array in nondecreasing order.

This allows binary searching around `k` and efficiently selecting:

- smallest cheap chocolates
- largest expensive chocolates

### Step 2: Build prefix sums

Create a prefix sum array:

$$prefix[i]$$

which stores the sum of the first `i` sorted prices.

This enables constant-time range sum queries.

### Step 3: For each query, split prices around `k`

Using binary search, find:

$$idx = \text{first position with price > k}$$

Then:

- `prices[0:idx]` are cheap chocolates
- `prices[idx:n]` are expensive chocolates

Let:

- `small = idx`
- `large = n - idx`

### Step 4: Take as many expensive chocolates as possible

Since expensive chocolates tend to reduce Bob’s loss more, we take:

$$takeLarge = \min(m, large)$$

The remaining chocolates must come from the cheap side:

$$takeSmall = m - takeLarge$$

### Step 5: Compute contribution from cheap chocolates

To minimize contribution among cheap chocolates, choose the smallest ones.

Their total contribution equals their total price.

Using prefix sums:

$$smallSum = prefix[takeSmall]$$

### Step 6: Compute contribution from expensive chocolates

We choose the largest expensive chocolates.

Suppose their total price is:

$$largeSum$$

Each contributes:

$$2k - p$$

So total contribution becomes:

$$takeLarge \cdot 2k - largeSum$$

### Step 7: Combine both parts

The answer is:

$$smallSum + takeLarge \cdot 2k - largeSum$$

### Why it works

The contribution function is monotonic inside both regions.

For prices `<= k`, smaller prices always produce smaller contributions.

For prices `> k`, larger prices always produce smaller contributions because the term `2k-p` decreases as `p` increases.

Therefore, once we decide how many chocolates come from each side, the optimal selection is uniquely determined:

- cheapest from the low side
- most expensive from the high side

Finally, since expensive chocolates are always at least as favorable as cheaper expensive chocolates, taking as many expensive chocolates as possible is optimal.

## Python Solution

```python
from bisect import bisect_right
from typing import List

class Solution:
    def minimumRelativeLosses(self, prices: List[int], queries: List[List[int]]) -> List[int]:
        prices.sort()
        n = len(prices)

        prefix = [0] * (n + 1)

        for i in range(n):
            prefix[i + 1] = prefix[i] + prices[i]

        answers = []

        for k, m in queries:
            idx = bisect_right(prices, k)

            large_count = n - idx
            take_large = min(m, large_count)
            take_small = m - take_large

            small_sum = prefix[take_small]

            large_sum = prefix[n] - prefix[n - take_large]

            result = small_sum + 2 * k * take_large - large_sum

            answers.append(result)

        return answers
```

The implementation begins by sorting the prices array. This is essential because the algorithm relies on separating chocolates into two contiguous groups:

- prices less than or equal to `k`
- prices greater than `k`

Next, a prefix sum array is built. This allows constant-time retrieval of:

- sums of the smallest prices
- sums of the largest prices

For every query, `bisect_right` finds the boundary between cheap and expensive chocolates.

The algorithm then greedily selects as many expensive chocolates as possible because their contributions become smaller as prices increase.

If more chocolates are still needed, the remaining chocolates are chosen from the smallest cheap prices.

The final formula directly computes the total relative loss without simulating individual payments.

## Go Solution

```go
package main

import (
	"sort"
)

func minimumRelativeLosses(prices []int, queries [][]int) []int64 {
	sort.Ints(prices)

	n := len(prices)

	prefix := make([]int64, n+1)

	for i := 0; i < n; i++ {
		prefix[i+1] = prefix[i] + int64(prices[i])
	}

	ans := make([]int64, 0, len(queries))

	for _, q := range queries {
		k := q[0]
		m := q[1]

		idx := sort.Search(n, func(i int) bool {
			return prices[i] > k
		})

		largeCount := n - idx

		takeLarge := m
		if takeLarge > largeCount {
			takeLarge = largeCount
		}

		takeSmall := m - takeLarge

		smallSum := prefix[takeSmall]

		largeSum := prefix[n] - prefix[n-takeLarge]

		result := smallSum + int64(2*k*takeLarge) - largeSum

		ans = append(ans, result)
	}

	return ans
}
```

The Go implementation follows the exact same logic as the Python version.

One important difference is integer handling. Since prices and query counts can both reach `10^5`, intermediate sums can exceed 32-bit integer limits. Therefore:

- prefix sums use `int64`
- final answers use `int64`

Go’s `sort.Search` is used to perform the binary search equivalent of Python’s `bisect_right`.

Slices naturally handle empty ranges, so no special-case handling is required when:

- all chocolates are cheap
- all chocolates are expensive

## Correct Optimal Strategy

Suppose we take `x` cheap chocolates.

Then we take `m-x` expensive chocolates.

The optimal cost becomes:

$$\text{smallest } x \text{ cheap prices}$$

plus

$$\text{largest } (m-x) \text{ expensive prices}$$

We must test the valid boundary efficiently.

The condition comes from comparing:

- next cheap contribution
- next expensive contribution

The resulting implementation below is the correct final version.

## Correct Python Solution

```python
from bisect import bisect_right
from typing import List

class Solution:
    def minimumRelativeLosses(self, prices: List[int], queries: List[List[int]]) -> List[int]:
        prices.sort()
        n = len(prices)

        prefix = [0] * (n + 1)

        for i in range(n):
            prefix[i + 1] = prefix[i] + prices[i]

        answers = []

        for k, m in queries:
            idx = bisect_right(prices, k)

            low = max(0, m - (n - idx))
            high = min(m, idx)

            while low < high:
                mid = (low + high + 1) // 2

                cheap_val = prices[mid - 1]
                expensive_val = prices[n - (m - mid)]

                if cheap_val <= 2 * k - expensive_val:
                    low = mid
                else:
                    high = mid - 1

            take_small = low
            take_large = m - take_small

            small_sum = prefix[take_small]

            large_sum = prefix[n] - prefix[n - take_large]

            result = small_sum + 2 * k * take_large - large_sum

            answers.append(result)

        return answers
```

import (
	"sort"

func minimumRelativeLosses(prices []int, queries [][]int) []int64 {
	sort.Ints(prices)

	n := len(prices)

	prefix := make([]int64, n+1)

	for i := 0; i < n; i++ {
		prefix[i+1] = prefix[i] + int64(prices[i])
	}

	ans := make([]int64, 0, len(queries))

	for _, q := range queries {
		k := q[0]
		m := q[1]

		idx := sort.Search(n, func(i int) bool {
			return prices[i] > k
		})

		low := 0
		if m-(n-idx) > low {
			low = m - (n - idx)
		}

		high := m
		if idx < high {
			high = idx
		}

		for low < high {
			mid := (low + high + 1) / 2

			cheapVal := prices[mid-1]
			expensiveVal := prices[n-(m-mid)]

			if cheapVal <= 2*k-expensiveVal {
				low = mid
			} else {
				high = mid - 1
			}
		}

		takeSmall := low
		takeLarge := m - takeSmall

		smallSum := prefix[takeSmall]

		largeSum := prefix[n] - prefix[n-takeLarge]

		result := smallSum + int64(2*k*takeLarge) - largeSum

		ans = append(ans, result)
	}

	return ans
}

The corrected algorithm carefully balances cheap and expensive chocolates instead of greedily maximizing expensive selections.

The binary search determines the optimal number of cheap chocolates to include.

## Edge Cases

One important edge case occurs when every chocolate is cheaper than or equal to `k`. In this situation, Alice pays nothing for every selected chocolate, so the contribution of each chocolate is simply its original price. The optimal strategy becomes selecting the `m` cheapest chocolates. The implementation naturally handles this because the expensive segment becomes empty and the binary search range collapses correctly.

Another important case occurs when every chocolate is more expensive than `k`. Here every chocolate contributes:

$$2k - p$$

which becomes smaller for larger prices. Therefore the optimal strategy is selecting the most expensive chocolates. The implementation handles this by setting the number of cheap chocolates to zero and taking the largest prices from the sorted array.

A subtle edge case appears when the optimal selection mixes both cheap and expensive chocolates. A naive greedy strategy of always maximizing expensive selections fails here, as shown in the first example. The corrected algorithm solves this by binary searching the optimal split between cheap and expensive selections, ensuring globally minimal contribution.

A final important case involves very large values. Prices and query values can reach `10^9`, and sums can exceed 32-bit integer range. The Go implementation therefore uses `int64` throughout prefix sums and final computations to avoid overflow.
