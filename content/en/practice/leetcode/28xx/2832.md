## Solution

## Problem Understanding

The problem gives us an array `nums` containing distinct integers. For every index `i`, we must determine the largest possible subarray in which `nums[i]` is the maximum element.

More formally, for each position `i`, we want the maximum length of a contiguous subarray `nums[l..r]` such that:

- `l <= i <= r`
- `nums[i]` is the largest value in that entire subarray

Because all elements are distinct, there is always exactly one maximum element in every subarray. This uniqueness simplifies the logic because we never need to handle ties between equal values.

The output array `ans` has the same length as `nums`, where:

- `ans[i]` equals the maximum size of a subarray where `nums[i]` remains the maximum value

The key observation is that if we want `nums[i]` to stay the maximum, then the subarray cannot include any element larger than `nums[i]`.

That means:

- On the left side, we can extend until we hit the first element greater than `nums[i]`
- On the right side, we can extend until we hit the first element greater than `nums[i]`

Everything between those two boundaries can safely belong to the subarray.

The constraints are important:

- `nums.length` can be as large as `10^5`
- A quadratic solution would be too slow
- We need an algorithm close to linear time

The fact that all numbers are distinct is also important because it guarantees that comparisons are strictly greater or smaller, with no ambiguity from duplicates.

Several edge cases are worth noticing immediately:

- A strictly increasing array
- A strictly decreasing array
- A single-element array
- The global maximum element, which can always expand across the entire array
- Very small local minima, which may only form subarrays of length `1`

These cases help validate whether boundary calculations are implemented correctly.

## Approaches

### Brute Force Approach

A straightforward solution is to process each index independently.

For every `nums[i]`, we can try every possible subarray containing index `i`. For each candidate subarray, we compute whether `nums[i]` is the maximum element. If it is, we update the best length.

One implementation would look like this conceptually:

- Fix index `i`
- Expand left boundary `l`
- Expand right boundary `r`
- Check whether every element inside `nums[l..r]` is smaller than `nums[i]`

This works because it explicitly checks all valid subarrays.

However, this approach is extremely slow.

There are `O(n^2)` possible subarrays, and validating each subarray may take another `O(n)` scan. Even with optimizations, the total runtime becomes at least `O(n^2)`, which is too large for `n = 10^5`.

We need a way to determine the valid expansion boundaries much more efficiently.

### Optimal Approach, Monotonic Stack

The important insight is that for each element `nums[i]`, the answer depends only on the nearest greater element on both sides.

Suppose:

- `leftGreater[i]` is the index of the nearest greater element to the left
- `rightGreater[i]` is the index of the nearest greater element to the right

Then the largest valid subarray is:

- from `leftGreater[i] + 1`
- to `rightGreater[i] - 1`

So the answer becomes:

$$ans[i] = rightGreater[i] - leftGreater[i] - 1$$

The remaining challenge is efficiently computing nearest greater elements.

This is exactly where a monotonic stack is useful.

A monotonic decreasing stack allows us to process each element once while maintaining candidates for nearest greater elements.

Because every element is pushed and popped at most once, the total complexity becomes linear.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) to O(n³) | O(1) | Explicitly checks many subarrays |
| Optimal | O(n) | O(n) | Uses monotonic stacks to find nearest greater elements |

## Algorithm Walkthrough

### Step 1: Compute nearest greater element on the left

We create an array `left_greater`.

For every index `i`, we want the closest index to the left whose value is greater than `nums[i]`.

We maintain a decreasing stack of indices.

While the top of the stack contains a value smaller than `nums[i]`, we remove it because it can no longer serve as a greater boundary for future elements.

After removals:

- If the stack is empty, there is no greater element on the left
- Otherwise, the stack top is the nearest greater element

Then we push the current index onto the stack.

### Step 2: Compute nearest greater element on the right

We repeat the same idea from right to left.

We create an array `right_greater`.

Again, we use a decreasing stack.

For each index `i`:

- Remove smaller elements from the stack
- The remaining top becomes the nearest greater element on the right
- Push the current index

### Step 3: Compute the answer

Now each index has:

- Left boundary blocker
- Right boundary blocker

The valid subarray spans all positions between them.

So:

$$ans[i] = rightGreater[i] - leftGreater[i] - 1$$

### Why it works

The algorithm works because any subarray containing an element larger than `nums[i]` would immediately invalidate `nums[i]` as the maximum.

Therefore, the farthest valid extension on each side is limited by the nearest greater element.

The monotonic stack guarantees that we efficiently discover these nearest greater boundaries in linear time.

Each element enters and leaves the stack at most once, preserving the `O(n)` runtime.

## Python Solution

```python
from typing import List

class Solution:
    def maximumLengthOfRanges(self, nums: List[int]) -> List[int]:
        n = len(nums)

        left_greater = [-1] * n
        right_greater = [n] * n

        stack = []

        # Nearest greater element on the left
        for i in range(n):
            while stack and nums[stack[-1]] < nums[i]:
                stack.pop()

            if stack:
                left_greater[i] = stack[-1]

            stack.append(i)

        stack.clear()

        # Nearest greater element on the right
        for i in range(n - 1, -1, -1):
            while stack and nums[stack[-1]] < nums[i]:
                stack.pop()

            if stack:
                right_greater[i] = stack[-1]

            stack.append(i)

        ans = [0] * n

        for i in range(n):
            ans[i] = right_greater[i] - left_greater[i] - 1

        return ans
```

The implementation begins by initializing two boundary arrays:

- `left_greater`
- `right_greater`

These arrays store the nearest indices containing larger values.

The first traversal processes elements from left to right. The stack remains monotonically decreasing in value. Whenever we encounter a larger element, smaller elements are removed because they can never serve as future greater boundaries.

After this pass, every position knows the closest larger value to its left.

The second traversal repeats the same logic from right to left, producing the nearest larger value on the right.

Finally, the answer for each index is simply the size of the interval between those two greater boundaries.

The subtraction formula works because the blocking indices themselves cannot be included in the valid subarray.

## Go Solution

```go
func maximumLengthOfRanges(nums []int) []int {
	n := len(nums)

	leftGreater := make([]int, n)
	rightGreater := make([]int, n)

	for i := 0; i < n; i++ {
		leftGreater[i] = -1
		rightGreater[i] = n
	}

	stack := []int{}

	// Nearest greater element on the left
	for i := 0; i < n; i++ {
		for len(stack) > 0 && nums[stack[len(stack)-1]] < nums[i] {
			stack = stack[:len(stack)-1]
		}

		if len(stack) > 0 {
			leftGreater[i] = stack[len(stack)-1]
		}

		stack = append(stack, i)
	}

	stack = []int{}

	// Nearest greater element on the right
	for i := n - 1; i >= 0; i-- {
		for len(stack) > 0 && nums[stack[len(stack)-1]] < nums[i] {
			stack = stack[:len(stack)-1]
		}

		if len(stack) > 0 {
			rightGreater[i] = stack[len(stack)-1]
		}

		stack = append(stack, i)
	}

	ans := make([]int, n)

	for i := 0; i < n; i++ {
		ans[i] = rightGreater[i] - leftGreater[i] - 1
	}

	return ans
}
```

The Go implementation closely mirrors the Python version.

Slices are used to implement the monotonic stack. Popping is performed by reslicing:

```go
stack = stack[:len(stack)-1]
```

Unlike Python, Go requires explicit initialization of boundary values because slices default to zero values.

There are no integer overflow concerns because all computations stay within array index ranges.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each index is pushed and popped at most once |
| Space | O(n) | Boundary arrays and stack storage |

The monotonic stack guarantees amortized constant work per element.

Although the inner `while` loops appear nested, every element can only be removed once throughout the entire execution. This makes the total number of stack operations linear.

The auxiliary space consists of:

- Two boundary arrays
- One stack

All of them scale proportionally with the input size.

## Edge Cases

### Single Element Array

When the array contains only one element, that element is trivially the maximum in the only possible subarray.

A buggy implementation might incorrectly compute boundaries or produce `0` due to off by one mistakes.

This implementation handles it naturally because:

- `leftGreater = -1`
- `rightGreater = n`

So the formula becomes:

$$1 - (-1) - 1 = 1$$

### Strictly Increasing Array

In a strictly increasing array, every element has no greater value on the left.

This is an important case because the left boundary always expands to the beginning of the array.

The algorithm handles this correctly because smaller elements are continuously popped from the stack, leaving it empty before inserting each new larger value.

### Strictly Decreasing Array

In a strictly decreasing array, every element immediately encounters a greater element on the left.

This creates very small valid ranges for later elements.

Incorrect stack logic can easily produce wrong nearest greater indices here.

The monotonic decreasing stack preserves all earlier larger elements, ensuring the nearest greater boundary is always correctly identified.
