# LeetCode 2830 - Maximize the Profit as the Salesman

## Problem Understanding

The problem gives us `n` houses arranged on a number line, numbered from `0` to `n - 1`. We are also given a list of purchase offers. Each offer is represented as:

```
[start, end, gold]
```

This means a buyer wants to purchase every house from `start` to `end`, inclusive, and will pay `gold` amount of money.

The important restriction is that no house can be sold more than once. Therefore, if we accept one offer covering houses `[2, 5]`, then we cannot accept any other offer that overlaps with any house in that interval.

Our task is to choose a subset of non-overlapping offers such that the total earned gold is maximized.

This is fundamentally an interval scheduling optimization problem. Every offer is an interval on the number line, and intervals cannot overlap. Among all valid combinations of intervals, we want the one with the maximum total value.

The constraints are large:

- `n <= 10^5`
- `offers.length <= 10^5`

These limits immediately tell us that brute force enumeration of subsets is impossible. Since there can be up to `100000` offers, even an `O(m^2)` solution may be too slow, where `m = len(offers)`.

A few important observations and edge cases are worth identifying early:

- Multiple offers may have the same start or end.
- One large profitable interval may outperform many smaller intervals.
- Some houses may remain unsold, which means we are never forced to accept an offer.
- Offers can consist of a single house, such as `[3,3,7]`.
- Offers may be completely nested inside larger intervals.
- The optimal answer may involve skipping seemingly profitable offers in favor of a better future combination.

Because intervals cannot overlap and we want maximum total value, this strongly suggests dynamic programming combined with sorting and binary search.

## Approaches

### Brute Force Approach

A naive solution would try every possible subset of offers.

For each subset, we would check whether any two selected offers overlap. If they do not overlap, we compute the total gold and track the maximum.

This approach is correct because it exhaustively explores every possible valid combination of offers. Eventually, it would discover the optimal one.

However, the number of subsets of `m` offers is:

```
2^m
```

With `m` up to `100000`, this is astronomically large and completely infeasible.

Even a slightly optimized backtracking approach would still be far too slow because the search space grows exponentially.

### Key Insight

The crucial observation is that this problem is equivalent to the classic Weighted Interval Scheduling problem.

Each offer is an interval:

```
[start, end]
```

with a profit:

```
gold
```

If we sort offers by ending position, then for every offer we can decide:

- Skip the offer
- Take the offer and combine it with the best compatible previous offer

The challenge becomes efficiently finding the last non-overlapping offer.

Suppose we sort by ending house. For an offer `[s, e, g]`, we need the best profit achievable using offers that end before `s`.

This naturally leads to dynamic programming:

```
dp[i] = maximum profit considering offers up to index i
```

To efficiently find the last compatible offer, we use binary search on sorted end positions.

This reduces the complexity from exponential time to:

```
O(m log m)
```

which easily handles the constraints.

| --- | --- | --- | --- |
| Brute Force | O(2^m) | O(m) | Enumerates every subset of offers |
| Optimal | O(m log m) | O(m) | Dynamic programming with sorting and binary search |

Here, `m` is the number of offers.

## Algorithm Walkthrough

### Optimal Dynamic Programming with Binary Search

1. Sort all offers by their ending house.

Sorting by end position allows us to process offers in chronological order. Once offers are ordered this way, every compatible earlier offer appears before the current one.
2. Create an array containing only the end positions.

This array enables binary search. For each current offer, we want to quickly locate the last offer whose ending house is strictly smaller than the current offer's starting house.
3. Define the DP state.

Let:

```
dp[i]
```

represent the maximum profit obtainable using offers from index `0` through `i` in the sorted list.
4. Process offers one by one.

For each offer `[start, end, gold]`, there are two choices:

- Skip the current offer
- Take the current offer
5. Compute the profit if we skip the offer.

If we skip the current offer, then:

```
skip = dp[i - 1]
```
6. Find the last compatible offer.

We binary search the sorted end positions to locate the rightmost offer whose end is less than the current offer's start.

This ensures the intervals do not overlap.
7. Compute the profit if we take the offer.

If the compatible offer index is `j`, then:

```
take = gold + dp[j]
```

If no compatible offer exists:

```
take = gold
```
8. Store the best result.

The optimal result for this position is:

```
dp[i] = max(skip, take)
```
9. Return the final answer.

After processing all offers:

```
dp[m - 1]
```

contains the maximum achievable profit.

### Why it works

The algorithm works because every offer creates an independent decision:

- Either include it
- Or exclude it

After sorting by end position, all compatible earlier offers lie before the current offer. Binary search efficiently identifies the best compatible boundary.

The DP invariant is:

```sql
dp[i] stores the optimal profit using offers from 0 through i
```

Since every state is computed from already-correct smaller states, dynamic programming guarantees the final answer is optimal.

## Python Solution

```python
from bisect import bisect_left
from typing import List

class Solution:
    def maximizeTheProfit(self, n: int, offers: List[List[int]]) -> int:
        offers.sort(key=lambda offer: offer[1])

        ends = [offer[1] for offer in offers]
        m = len(offers)

        dp = [0] * m

        for i, (start, end, gold) in enumerate(offers):
            # Profit if we take this offer
            take = gold

            # Find last offer with end < start
            prev_index = bisect_left(ends, start) - 1

            if prev_index >= 0:
                take += dp[prev_index]

            # Profit if we skip this offer
            skip = dp[i - 1] if i > 0 else 0

            dp[i] = max(take, skip)

        return dp[-1]
```

The implementation begins by sorting all offers according to their ending house. This ordering is essential because it allows us to use dynamic programming on progressively larger prefixes of the offer list.

Next, we build the `ends` array, which stores only the end values from the sorted offers. This array is used with `bisect_left` to efficiently locate the last compatible interval.

The `dp` array stores the best achievable profit up to each offer index.

Inside the loop, we compute two possibilities for every offer:

- Accept the offer
- Reject the offer

To accept the offer, we binary search for the last offer ending before the current start position. If such an offer exists, we add its optimal DP value.

To reject the offer, we simply inherit the previous best answer.

Finally, we store the larger of the two values.

The final DP entry contains the optimal total profit.

## Go Solution

```go
package main

import (
	"sort"
)

func maximizeTheProfit(n int, offers [][]int) int {
	sort.Slice(offers, func(i, j int) bool {
		return offers[i][1] < offers[j][1]
	})

	m := len(offers)

	ends := make([]int, m)
	for i := 0; i < m; i++ {
		ends[i] = offers[i][1]
	}

	dp := make([]int, m)

	for i := 0; i < m; i++ {
		start := offers[i][0]
		gold := offers[i][2]

		take := gold

		// Find first end >= start
		prevIndex := sort.Search(len(ends), func(j int) bool {
			return ends[j] >= start
		}) - 1

		if prevIndex >= 0 {
			take += dp[prevIndex]
		}

		skip := 0
		if i > 0 {
			skip = dp[i-1]
		}

		if take > skip {
			dp[i] = take
		} else {
			dp[i] = skip
		}
	}

	return dp[m-1]
}
```

The Go implementation follows the exact same algorithmic structure as the Python solution.

Instead of Python's `bisect_left`, Go uses `sort.Search` to perform binary search. The logic is identical: locate the first interval whose end is greater than or equal to the current start, then move one position left to obtain the last compatible interval.

Go slices are used instead of Python lists, and manual maximum comparison replaces Python's built-in `max`.

Since the maximum possible profit is at most:

```go
100000 * 1000 = 10^8
```

standard Go `int` values are completely safe from overflow.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m log m) | Sorting plus binary search for each offer |
| Space | O(m) | DP array and end-position array |

Here, `m` is the number of offers.

Sorting requires `O(m log m)` time. Then, for every offer, we perform one binary search costing `O(log m)`. Since there are `m` offers, the total processing time is also `O(m log m)`.

The additional memory usage comes from the DP array and the list of end positions, both proportional to the number of offers.

## Edge Cases

One important edge case occurs when all offers overlap with each other. In this situation, the algorithm must correctly select only the single most profitable offer. A buggy implementation might accidentally combine overlapping intervals or fail to compare all possibilities properly. The DP transition handles this safely because every offer independently compares "take" versus "skip".

Another important case involves intervals that touch at boundaries. For example, `[0,1]` and `[1,2]` overlap because house `1` belongs to both intervals. The binary search specifically looks for intervals whose end is strictly less than the next interval's start. This guarantees overlapping endpoints are never combined incorrectly.

A third tricky case involves deeply nested intervals, such as one large interval covering many smaller ones. Sometimes the large interval is optimal, and sometimes multiple smaller intervals produce greater total profit. Greedy strategies often fail here because local choices do not guarantee a global optimum. Dynamic programming correctly evaluates both possibilities at every step and always preserves the best achievable profit.

A final edge case is when offers contain only single-house ranges like `[3,3,10]`. These intervals behave exactly the same as larger ranges, and the algorithm naturally handles them without requiring any special logic.
