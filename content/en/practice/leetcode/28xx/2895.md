## Solution

## Problem Understanding

In this problem, we are given two arrays:

- `processorTime`, where each value represents the time when a processor becomes available
- `tasks`, where each value represents how long a task takes to execute

Every processor has exactly 4 cores, and every core can execute exactly one task. Since there are `n` processors, the total number of tasks is guaranteed to be `4 * n`.

A task assigned to a processor can only start after that processor becomes available. If a processor becomes available at time `p`, and a task assigned to it takes `t` time, then that task finishes at time:

$$p + t$$

Because the processor has 4 cores, it can execute 4 tasks in parallel. Therefore, the completion time for a processor is determined by the longest task assigned to it.

The goal is to assign tasks to processors so that the overall finishing time across all processors is minimized.

In other words, we want to minimize:

$$\max(p_i + \text{largest assigned task to processor } i)$$

The constraints are important:

- Up to `25,000` processors
- Up to `100,000` tasks
- Task durations and processor times can be as large as `10^9`

These limits immediately rule out exponential or combinatorial assignment strategies. We need an algorithm close to `O(n log n)`.

The problem guarantees:

- Every processor gets exactly 4 tasks
- The number of tasks is exactly `4 * n`
- Every task must be assigned exactly once

Important edge cases include:

- All processors having the same availability time
- All tasks having the same duration
- Extremely large task values
- Processors with very different availability times
- One processor being very late, forcing careful assignment of small tasks to it

A naive greedy assignment can easily fail if it does not properly balance large tasks against processor availability times.

## Approaches

### Brute Force Approach

A brute-force solution would try all possible ways to distribute tasks among processors.

Since each processor must receive exactly 4 tasks, we could generate every grouping of tasks and compute the resulting completion time. For each assignment, we would calculate:

$$\max(\text{processorTime}[i] + \max(\text{assigned tasks}))$$

Then we would return the minimum possible value over all assignments.

This approach is correct because it explores every possible valid assignment, guaranteeing that the optimal one is found.

However, it is computationally impossible for the given constraints. Even for relatively small inputs, the number of possible task distributions becomes enormous. The problem size can reach `100,000` tasks, making exhaustive search infeasible.

### Key Insight for the Optimal Solution

The important observation is that for each processor, only the largest assigned task matters.

If a processor gets tasks `[1, 2, 5, 8]`, the processor finishes at:

$$processorTime + 8$$

The smaller tasks do not affect the completion time because all four cores run in parallel.

This means the problem becomes:

- Assign large tasks to processors with small availability times
- Assign small tasks to processors with large availability times

This is a classic greedy matching strategy.

To achieve this:

1. Sort processors in ascending order
2. Sort tasks in descending order
3. Give the largest remaining 4 tasks to the earliest available processor

Why does this work?

The largest tasks are the most dangerous because they dominate completion time. Assigning them to the earliest processors minimizes their contribution to the final answer.

Meanwhile, later processors should receive smaller tasks to compensate for their larger starting times.

## Algorithm Walkthrough

1. Sort `processorTime` in ascending order.

The earliest available processors should handle the heaviest workloads. Sorting allows us to process processors from best to worst availability.
2. Sort `tasks` in descending order.

The largest tasks are the most important because they determine processor completion times.
3. Process processors one by one.

Since each processor must receive exactly 4 tasks, processor `i` receives tasks from indices:

$$4i \text{ to } 4i + 3$$

in the descending task list.
4. For each processor, determine its completion time.

Since tasks are sorted descending, the first task in each group of 4 is the largest one. Therefore:

$$completion = processorTime[i] + tasks[4i]$$
5. Track the maximum completion time across all processors.

The overall answer is the slowest processor completion time.
6. Return the maximum value.

### Why it works

The greedy strategy works because the largest tasks have the greatest impact on the final answer. Pairing the largest tasks with the earliest processors minimizes the maximum completion time.

This follows the same principle as minimizing maximum pair sums, where the largest values should be matched with the smallest complementary values.

Any alternative assignment that places a larger task on a later processor can only increase the maximum completion time.

## Edge Cases

One important edge case is when there is only a single processor. In this situation, all four tasks must go to the same processor, and the answer is simply the processor availability time plus the largest task. Some implementations incorrectly attempt more complicated balancing logic even though no assignment flexibility exists. This solution handles the case naturally because the loop runs exactly once.

Another important edge case occurs when processor availability times differ dramatically. For example, one processor may become available at time `1` while another becomes available at time `100`. A poor greedy strategy might accidentally assign large tasks to the later processor, dramatically increasing the final answer. Sorting processors ascending and tasks descending guarantees that the largest tasks go to the earliest processors.

A third edge case involves duplicate values. Processors or tasks may all have the same value. Some algorithms accidentally depend on uniqueness assumptions and produce unstable behavior. This implementation works correctly because it only depends on sorted ordering, not uniqueness.

Large integer values are also important. Since both processor times and task durations can reach `10^9`, the final answer can exceed `10^9`. The implementation safely handles this because Python integers are unbounded, and Go integers comfortably support the required range on modern systems.
