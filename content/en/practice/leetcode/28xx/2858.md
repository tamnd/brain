# LeetCode 2858 - Minimum Edge Reversals So Every Node Is Reachable

## Problem Understanding

This problem gives us a directed graph with `n` nodes and exactly `n - 1` edges. The important detail is that if we ignore the edge directions, the graph forms a tree. That means the underlying structure is connected and contains no cycles.

Each edge is directed, meaning travel is only possible in one direction unless we reverse that edge.

For every node `i`, we must determine the minimum number of edge reversals required so that starting from node `i`, we can reach every other node in the graph following the edge directions.

The key detail is that every node is evaluated independently. The reversals required for node `0` do not affect the answer for node `1`.

Suppose we choose a node as the root. To make all nodes reachable from that root, every edge in the tree must point away from the root. Any edge pointing toward the root must be reversed.

The input consists of:

- `n`, the number of nodes
- `edges`, where `edges[i] = [u, v]` means there is a directed edge from `u` to `v`

The output is an array `answer` where:

- `answer[i]` is the minimum number of reversals needed so node `i` can reach every other node

The constraints are large:

- `n` can be as large as `100000`
- The graph contains `n - 1` edges

These constraints immediately rule out expensive algorithms such as running a full graph traversal from every node while recomputing reversals each time.

Because the undirected version forms a tree, there is exactly one simple path between any pair of nodes. This property is extremely important because it allows us to reroot efficiently using dynamic programming on trees.

Several edge cases are important:

- A graph where all edges already point away from a node, meaning its answer is `0`
- A graph where every edge points toward a node, meaning many reversals are needed
- A chain-shaped tree, where moving the root changes the answer gradually
- Very deep trees, where recursive implementations may hit recursion depth limits in some languages
- The smallest valid input, `n = 2`

The problem guarantees:

- No duplicate edges
- No cycles in the undirected version
- Full connectivity in the undirected version

These guarantees make tree DP possible.

## Approaches

### Brute Force Approach

A straightforward approach is to compute the answer independently for every node.

For each node `start`:

1. Treat `start` as the root.
2. Traverse the tree.
3. Count how many edges point toward the root instead of away from it.
4. Those edges must be reversed.

To implement this, we can convert the graph into an undirected adjacency list while storing whether traversing an edge follows the original direction or goes against it.

During DFS from a chosen root:

- If an edge already points away from the current node, no reversal is needed.
- If the edge points toward the current node, we must reverse it.

Since we repeat a full traversal for every node, the total complexity becomes:

- `O(n)` work per root
- `n` roots total

This gives `O(n^2)` time complexity, which is too slow for `n = 100000`.

### Optimal Approach

The key insight is that answers for neighboring roots are closely related.

Suppose we already know the answer for node `u`.

Now consider moving the root from `u` to one of its neighbors `v`.

Only one edge changes its contribution, the edge between `u` and `v`.

There are two possibilities:

- Original edge is `u -> v`

- This edge was correctly oriented when rooted at `u`
- When rooted at `v`, it must be reversed
- Answer increases by `1`
- Original edge is `v -> u`

- This edge required reversal when rooted at `u`
- When rooted at `v`, it becomes correct
- Answer decreases by `1`

This rerooting relationship allows us to compute all answers in linear time.

The algorithm works in two DFS passes:

1. First DFS computes the answer for one fixed root, usually node `0`
2. Second DFS reroots dynamically to compute answers for all other nodes

This is a classic rerooting DP technique on trees.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Run DFS from every node independently |
| Optimal | O(n) | O(n) | Use rerooting DP on the tree |

## Algorithm Walkthrough

### Step 1: Build an Augmented Adjacency List

We treat the graph as undirected for traversal purposes, but we store extra information about edge direction.

For each directed edge `u -> v`:

- Add `(v, 0)` to `u`
- Add `(u, 1)` to `v`

The meaning is:

- Cost `0` means traversing this edge follows the original direction
- Cost `1` means traversing this edge goes against the original direction, so reversal would be needed

Example:

```
2 -> 0
```

Adjacency entries:

```
2: (0, 0)
0: (2, 1)
```

From node `2` to `0`, no reversal is needed.

From node `0` to `2`, reversal is required.

### Step 2: Compute the Answer for Root 0

Run DFS starting from node `0`.

Whenever we traverse an edge with cost `1`, it means the edge points toward the current root and must be reversed.

Accumulate these costs across the traversal.

At the end:

```
answer[0] = total reversals needed for root 0
```

### Step 3: Reroot the Tree

Now perform another DFS.

Suppose we move the root from `u` to neighbor `v`.

If the original edge direction was:

- `u -> v`

- Rooting at `v` makes this edge incorrect
- Add `1`
- `v -> u`

- Rooting at `v` makes this edge correct
- Subtract `1`

Using our encoding:

- Cost `0` means original edge is `u -> v`

- New answer becomes:

```
answer[v] = answer[u] + 1
```

- Cost `1` means original edge is `v -> u`

- New answer becomes:

```
answer[v] = answer[u] - 1
```

Repeat this DFS until all answers are computed.

### Why it works

For any chosen root, every edge must point away from that root so all nodes become reachable.

When rerooting from `u` to `v`, every edge in the tree keeps the same correctness status except the edge between `u` and `v`.

Therefore, the total reversal count changes by exactly `1` or `-1`, depending on the original direction of that edge.

Because the graph is a tree, every node is visited exactly once in each DFS, guaranteeing correctness and linear complexity.

## Python Solution

```python
from typing import List
import sys

sys.setrecursionlimit(1 << 20)

class Solution:
    def minEdgeReversals(self, n: int, edges: List[List[int]]) -> List[int]:
        graph = [[] for _ in range(n)]

        for u, v in edges:
            graph[u].append((v, 0))
            graph[v].append((u, 1))

        answer = [0] * n

        def dfs1(node: int, parent: int) -> int:
            reversals = 0

            for neighbor, cost in graph[node]:
                if neighbor == parent:
                    continue

                reversals += cost
                reversals += dfs1(neighbor, node)

            return reversals

        answer[0] = dfs1(0, -1)

        def dfs2(node: int, parent: int) -> None:
            for neighbor, cost in graph[node]:
                if neighbor == parent:
                    continue

                if cost == 0:
                    answer[neighbor] = answer[node] + 1
                else:
                    answer[neighbor] = answer[node] - 1

                dfs2(neighbor, node)

        dfs2(0, -1)

        return answer
```

The implementation begins by constructing an adjacency list where every edge is represented twice. One representation follows the original direction with cost `0`, and the reverse representation has cost `1`.

The first DFS computes how many reversals are needed when node `0` is the root. Every edge traversed against its original direction contributes one reversal.

After computing `answer[0]`, the second DFS performs rerooting DP. Moving from parent to child changes the answer by exactly one depending on the original edge orientation.

If the edge already points from parent to child, the child root needs one extra reversal. Otherwise, the child root saves one reversal.

Both traversals visit every node exactly once.

## Go Solution

```go
package main

func minEdgeReversals(n int, edges [][]int) []int {
	graph := make([][][2]int, n)

	for _, edge := range edges {
		u, v := edge[0], edge[1]

		graph[u] = append(graph[u], [2]int{v, 0})
		graph[v] = append(graph[v], [2]int{u, 1})
	}

	answer := make([]int, n)

	var dfs1 func(int, int) int

	dfs1 = func(node int, parent int) int {
		reversals := 0

		for _, edge := range graph[node] {
			neighbor := edge[0]
			cost := edge[1]

			if neighbor == parent {
				continue
			}

			reversals += cost
			reversals += dfs1(neighbor, node)
		}

		return reversals
	}

	answer[0] = dfs1(0, -1)

	var dfs2 func(int, int)

	dfs2 = func(node int, parent int) {
		for _, edge := range graph[node] {
			neighbor := edge[0]
			cost := edge[1]

			if neighbor == parent {
				continue
			}

			if cost == 0 {
				answer[neighbor] = answer[node] + 1
			} else {
				answer[neighbor] = answer[node] - 1
			}

			dfs2(neighbor, node)
		}
	}

	dfs2(0, -1)

	return answer
}
```

The Go implementation follows exactly the same logic as the Python version.

Instead of tuples, Go uses fixed-size arrays `[2]int` to store `(neighbor, cost)` pairs.

Go does not have recursion depth issues as aggressively as Python in competitive programming environments, but extremely deep trees can still be problematic in theory. The implementation remains recursive because this is the standard and accepted solution pattern for this problem.

Slices are used for adjacency lists and the answer array, which naturally handle dynamic sizing.

## Edge Cases

### Smallest Possible Tree

When `n = 2`, the graph contains exactly one edge. This is the simplest possible input and is useful for verifying base logic.

For example:

```
0 -> 1
```

Node `0` already reaches every node, so answer is `0`.

Node `1` cannot reach `0` unless the edge is reversed, so answer is `1`.

The implementation handles this naturally because both DFS traversals still work correctly on a single-edge tree.

### Deep Chain Structure

A chain-shaped tree creates the maximum possible recursion depth.

Example:

```
0 -> 1 -> 2 -> 3 -> 4
```

The rerooting answers become:

```
[0,1,2,3,4]
```

This case is important because it verifies the reroot transition logic.

The Python implementation explicitly increases the recursion limit to safely handle deep trees up to the problem constraints.

### All Edges Point Toward One Node

Consider:

```
1 -> 0
2 -> 0
3 -> 0
```

Node `0` cannot reach anyone without reversing every edge.

However, leaves already have some correctly oriented paths relative to themselves.

This case is easy to mishandle if the reroot transition formulas are incorrect.

The implementation correctly adjusts answers by exactly `+1` or `-1` during rerooting, ensuring every edge contributes properly depending on the current root.
