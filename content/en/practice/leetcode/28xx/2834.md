## Solution

## Problem Understanding

The problem asks us to construct an array of length `n` using distinct positive integers such that no two different elements add up to `target`. Among all such valid arrays, we must return the minimum possible sum of the elements.

A valid array is called "beautiful". The conditions are:

1. The array contains exactly `n` numbers.
2. Every number is positive.
3. All numbers are distinct.
4. No pair of distinct elements sums to `target`.

The input consists of two integers:

- `n`, the required size of the array.
- `target`, the forbidden pair sum.

The output is the smallest possible sum of a beautiful array, modulo `10^9 + 7`.

The constraints are extremely large:

- `1 <= n <= 10^9`
- `1 <= target <= 10^9`

These limits immediately tell us that we cannot explicitly build the array in most cases. Any algorithm with linear complexity in `n` would already be too slow when `n` is close to one billion. We need a mathematical or greedy observation that allows us to compute the answer directly using formulas.

The key restriction is the forbidden sum condition. If we include a number `x`, then we cannot include `target - x`, because together they would sum to `target`.

For example, if `target = 10`:

- Choosing `1` forbids `9`
- Choosing `2` forbids `8`
- Choosing `3` forbids `7`

To minimize the total sum, we naturally want to choose the smallest available positive integers. However, once we pick one number from a conflicting pair, we must avoid the other.

Some important edge cases include:

- `n = 1`, where any single positive integer works because there is no pair at all.
- Very small targets such as `target = 1` or `target = 2`, where the forbidden structure behaves differently.
- Large `n`, where explicitly storing or iterating through all selected values is impossible.
- Even and odd targets, since `target / 2` behaves specially when the target is even.

## Approaches

### Brute Force Approach

A straightforward idea is to greedily build the array from the smallest positive integers upward.

We maintain a set of selected numbers. For each candidate integer `x` starting from `1`:

- Check whether `target - x` already exists in the set.
- If it does not, add `x` to the array.
- Continue until we collect `n` numbers.

This works because we always attempt to include the smallest possible valid number, so the resulting sum is minimized.

For example, with `n = 4` and `target = 5`:

- Pick `1`
- Skip `4`, because `1 + 4 = 5`
- Pick `2`
- Skip `3`, because `2 + 3 = 5`
- Pick `5`
- Pick `6`

Final array: `[1, 2, 5, 6]`

Although correct, this approach becomes impractical for very large inputs. We may need to iterate through many integers, and maintaining a hash set also uses additional memory.

### Key Insight for the Optimal Solution

The important observation is that the smallest valid numbers come from the range:

$$1, 2, 3, \dots, \left\lfloor \frac{target}{2} \right\rfloor$$

Why?

Because any two numbers in this range cannot sum to `target`.

For example, if `target = 10`, then:

$$1 + 4 = 5,\quad 2 + 4 = 6,\quad 4 + 4 = 8$$

More generally, if both numbers are strictly less than `target / 2`, their sum is always less than `target`.

This means we can safely take all numbers from `1` up to:

$$m = \left\lfloor \frac{target}{2} \right\rfloor$$

without creating a forbidden pair.

If `n <= m`, then the optimal answer is simply the sum of the first `n` positive integers.

If `n > m`, then after taking all safe small numbers, we must avoid the conflicting region near `target`. The next smallest safe numbers start from `target` itself.

Why is `target` safe?

Because if we choose a number `x < target`, then:

$$target + x > target$$

so no forbidden pair can be formed.

Thus the optimal array structure is:

$$[1, 2, \dots, m] \cup [target, target+1, target+2, \dots]$$

This allows us to compute the answer directly using arithmetic progression formulas.

| --- | --- | --- | --- |
| Brute Force | O(n) to O(target + n) | O(n) | Simulates construction using a hash set |
| Optimal | O(1) | O(1) | Uses greedy observation and arithmetic sums |

## Algorithm Walkthrough

1. Compute:

$$m = \left\lfloor \frac{target}{2} \right\rfloor$$

This is the count of smallest integers we can safely use without conflicts.

1. Check whether `n <= m`.

If true, then we only need the first `n` positive integers:

$$1 + 2 + \dots + n$$

The sum formula is:

$$\frac{n(n+1)}{2}$$

1. Otherwise, take all integers from `1` to `m`.

Their sum is:

$$\frac{m(m+1)}{2}$$

1. We still need:

$$remaining = n - m$$

additional numbers.

1. The next valid numbers begin at `target`.

So we take:

$$target,\ target+1,\ target+2,\ \dots,\ target+remaining-1$$

1. Compute this arithmetic progression sum.

The first term is `target`, and the last term is:

$$target + remaining - 1$$

The sum becomes:

$$\frac{remaining \times (first + last)}{2}$$

1. Add both sums together.
2. Return the result modulo:

$$10^9 + 7$$

### Why it works

The algorithm is correct because minimizing the total sum requires choosing the smallest possible valid integers. For every pair `(x, target - x)`, we can choose at most one number. The smaller number is always preferable because it contributes less to the sum.

Therefore, the optimal strategy is to always select the smaller member of each conflicting pair, which yields all integers from `1` through `floor(target / 2)`. Once those are exhausted, every integer below `target` conflicts with an already selected smaller number, so the next smallest safe values begin at `target`.

This greedy structure uniquely minimizes the total sum.

## Python Solution

```python
class Solution:
    def minimumPossibleSum(self, n: int, target: int) -> int:
        MOD = 10**9 + 7

        half = target // 2

        if n <= half:
            return (n * (n + 1) // 2) % MOD

        first_part = half * (half + 1) // 2

        remaining = n - half

        first = target
        last = target + remaining - 1

        second_part = remaining * (first + last) // 2

        return (first_part + second_part) % MOD
```

The implementation directly follows the mathematical derivation from the algorithm.

First, we compute `half = target // 2`. These are the smallest numbers we can safely include without any forbidden pair.

If `n` is small enough to fit entirely within this safe range, then the answer is simply the sum of the first `n` integers.

Otherwise, we split the construction into two parts:

- The safe prefix from `1` to `half`
- The remaining numbers starting from `target`

Both sums are arithmetic progressions, so we compute them using constant time formulas instead of loops.

Finally, we apply the modulo operation before returning the result.

## Go Solution

```go
func minimumPossibleSum(n int, target int) int {
	const MOD int64 = 1_000_000_007

	half := target / 2

	if n <= half {
		sum := int64(n) * int64(n+1) / 2
		return int(sum % MOD)
	}

	firstPart := int64(half) * int64(half+1) / 2

	remaining := n - half

	first := int64(target)
	last := int64(target + remaining - 1)

	secondPart := int64(remaining) * (first + last) / 2

	answer := (firstPart + secondPart) % MOD

	return int(answer)
}
```

The Go implementation is nearly identical to the Python version, but we must carefully handle integer overflow.

Since `n` and `target` can be as large as `10^9`, intermediate multiplication results can exceed the range of a 32 bit integer. Therefore, all arithmetic calculations use `int64`.

The final answer is converted back to `int` before returning.

## Edge Cases

One important edge case occurs when `n = 1`. Since the condition only forbids pairs of distinct indices, a single number can never violate the rule. A buggy implementation might unnecessarily avoid certain values, but the optimal answer is always simply `1`. The current implementation handles this naturally through the arithmetic formulas.

Another important case is when `target` is very small, especially `target = 1` or `target = 2`. In these situations, the safe prefix size `target // 2` becomes `0` or `1`. Many implementations accidentally assume there will always be a meaningful safe range. This solution correctly handles these cases because the formulas still work when `half = 0`.

A third important edge case is when `n <= target // 2`. In this situation, the optimal array consists entirely of the smallest integers starting from `1`, and we never need to jump to values starting at `target`. Incorrect implementations sometimes continue skipping numbers unnecessarily. The conditional branch in the solution correctly returns the triangular number sum immediately.

Large values near `10^9` are also critical. Any implementation using iterative construction or 32 bit arithmetic risks either timing out or overflowing. The mathematical approach avoids loops entirely, and the Go solution explicitly uses `int64` to guarantee correctness for large intermediate calculations.
