## Solution

## Problem Understanding

The problem gives us a circular operation called a right shift on an array of distinct positive integers. In one right shift, every element moves one position to the right, and the last element wraps around to the front. For example, applying one right shift to `[3,4,5,1,2]` produces `[2,3,4,5,1]`.

Our goal is to determine the minimum number of right shifts needed to transform the array into strictly increasing sorted order. If no number of right shifts can produce a sorted array, we must return `-1`.

The input consists of:

- A 0-indexed integer array `nums`
- All integers are distinct
- The array length is between `1` and `100`

The output is:

- The minimum number of right shifts required to sort the array
- `0` if the array is already sorted
- `-1` if sorting through right shifts is impossible

Because right shifts only rotate the array, the relative order of elements never changes. This observation is extremely important. A rotation can only move where the sequence starts, it cannot rearrange elements arbitrarily.

The constraints are small, so even a brute-force simulation would work comfortably. However, the problem has a very elegant linear-time observation that leads to a simpler and more efficient solution.

Several edge cases are important:

- Arrays of length `1` are already sorted
- Arrays already in increasing order should return `0`
- Arrays with exactly one "drop" in ordering may be sortable
- Arrays with multiple drops cannot become sorted through rotation
- The smallest element might already be at index `0`
- The rotation count must be minimized

For example:

- `[3,4,5,1,2]` is sortable because the sequence only breaks once
- `[2,1,4]` is not sortable because the order breaks in a way that rotation cannot fix

## Approaches

### Brute Force Approach

A straightforward approach is to simulate every possible number of right shifts.

For each possible shift count from `0` to `n - 1`, we can generate the rotated array and check whether it is sorted. The first shift count that produces a sorted array is the answer.

This works because an array of length `n` has exactly `n` unique rotations. If none of them are sorted, then sorting through right shifts is impossible.

To check whether an array is sorted, we compare every adjacent pair and ensure that each element is smaller than the next one.

Although this approach is perfectly acceptable for the given constraints, it repeatedly constructs rotated arrays and performs repeated scans, making it less elegant than necessary.

### Optimal Approach

The key observation is that a sortable rotated array must already look like a sorted array that has been rotated.

In a strictly increasing sorted array, every adjacent pair satisfies:

```
nums[i] < nums[i + 1]
```

If the array has been rotated, then there will be exactly one place where the ordering decreases. This position represents the point where the largest element wraps around to the smallest element.

For example:

```
[3,4,5,1,2]
```

The only decreasing pair is:

```
5 > 1
```

This means the sorted sequence starts at index `3`.

If there is more than one decreasing pair, then no rotation can restore sorted order.

Once we find the position of the decrease, we can directly compute how many right shifts are needed. If the smallest element starts at index `k`, then we need `n - k` right shifts to move it to the front.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Try every rotation and check if sorted |
| Optimal | O(n) | O(1) | Count ordering breaks and compute rotation directly |

## Algorithm Walkthrough

1. Start by storing the length of the array in `n`.
2. Initialize a variable called `drop_index` to `-1`. This variable will record where the array decreases.
3. Traverse the array from index `0` to `n - 2`.
4. For each adjacent pair, compare `nums[i]` and `nums[i + 1]`.
5. If `nums[i] > nums[i + 1]`, then we found a decreasing point.

- If this is the first decreasing point, store `i`.
- If we already found another decreasing point earlier, return `-1` immediately because the array cannot be sorted through rotation.
6. After the loop finishes, there are two possibilities.
7. If no decreasing point was found, the array is already sorted. Return `0`.
8. Otherwise, verify that the last element is not greater than the first element.

This check is necessary because rotation creates a circular ordering. For a valid rotated sorted array:

```
nums[n - 1] < nums[0]
```

If this condition is not satisfied, return `-1`.
9. The sorted order begins at index `drop_index + 1`.
10. Compute the minimum right shifts:

```
n - (drop_index + 1)
```
11. Return the result.

### Why it works

A rotated sorted array preserves the increasing order everywhere except at exactly one boundary where the largest element wraps around to the smallest element. Therefore, a valid array must contain at most one decreasing adjacent pair. If there are multiple decreases, no rotation can produce a fully sorted array. Once the rotation boundary is identified, the number of shifts needed is determined directly by how far the smallest element is from the front.

## Python Solution

```python
from typing import List

class Solution:
    def minimumRightShifts(self, nums: List[int]) -> int:
        n = len(nums)
        drop_index = -1

        for i in range(n - 1):
            if nums[i] > nums[i + 1]:
                if drop_index != -1:
                    return -1
                drop_index = i

        # Already sorted
        if drop_index == -1:
            return 0

        # Validate circular ordering
        if nums[-1] > nums[0]:
            return -1

        # Number of right shifts needed
        return n - (drop_index + 1)
```

The implementation begins by scanning the array to locate any position where the ordering decreases. Since a rotated sorted array can only have one such position, the code immediately returns `-1` if a second decrease is found.

If no decrease exists, the array is already sorted, so the answer is `0`.

After identifying the rotation boundary, the implementation verifies that the circular relationship is valid. Specifically, the last element must be smaller than the first element. If this condition fails, the array cannot represent a rotated sorted sequence.

Finally, the algorithm computes how many right shifts are required to bring the smallest element to the front. The smallest element begins immediately after the decreasing position, so the required number of shifts is calculated directly.

## Go Solution

```go
func minimumRightShifts(nums []int) int {
	n := len(nums)
	dropIndex := -1

	for i := 0; i < n-1; i++ {
		if nums[i] > nums[i+1] {
			if dropIndex != -1 {
				return -1
			}
			dropIndex = i
		}
	}

	// Already sorted
	if dropIndex == -1 {
		return 0
	}

	// Validate circular ordering
	if nums[n-1] > nums[0] {
		return -1
	}

	// Number of right shifts needed
	return n - (dropIndex + 1)
}
```

The Go implementation follows the exact same logic as the Python version. Since Go slices already carry their length information, there is no need for additional handling of empty arrays beyond normal indexing precautions.

Integer overflow is not a concern because the constraints are extremely small. The algorithm uses only constant extra memory and performs a single scan of the array.

## Edge Cases

One important edge case is an array that is already sorted, such as `[1,2,3,4]`. A naive implementation might incorrectly attempt to rotate the array or fail to distinguish between zero shifts and impossible cases. The implementation handles this by checking whether any decreasing pair exists. If none are found, it immediately returns `0`.

Another important edge case is an array with multiple decreasing points, such as `[4,1,3,2]`. This kind of input cannot be repaired through rotation because the relative ordering is fundamentally inconsistent with a rotated sorted sequence. The algorithm detects this during the scan and immediately returns `-1` as soon as a second decrease appears.

A third important edge case is when there is exactly one decrease, but the circular relationship is invalid. For example, `[2,1,4]` contains only one decreasing pair, but the last element is greater than the first element. This means the array does not represent a proper rotated sorted sequence. The additional circular validation step correctly catches this situation and prevents an incorrect answer.

A final edge case is a single-element array such as `[7]`. Since the array is trivially sorted, the correct answer is `0`. The loop does not execute because there are no adjacent pairs, and the implementation naturally returns `0` through the already-sorted condition.
