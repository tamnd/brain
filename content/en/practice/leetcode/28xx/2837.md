## Solution

## Problem Understanding

This problem asks us to compute the total distance traveled by every user in the system. The data is stored across two tables:

- The `Users` table contains basic user information such as `user_id` and `name`.
- The `Rides` table contains individual ride records, where each row represents one completed ride and its traveled `distance`.

The goal is to return a result table containing:

- `user_id`
- `name`
- total traveled distance for that user

The important detail is that every user must appear in the output, even if they never completed a ride. For users with no rides, the traveled distance should be reported as `0`.

The final output must be sorted by `user_id` in ascending order.

The problem is fundamentally an aggregation problem. We need to group rides by user and compute the sum of their distances. However, since some users may not appear in the `Rides` table at all, we cannot simply aggregate the rides table alone. We must ensure all users are preserved in the result.

The example demonstrates this clearly:

- User `2` has two rides, so their total becomes `197 + 196 = 393`.
- User `10` has no rides, so their total becomes `0`.

The constraints are relatively small because this is an Easy SQL problem, but the important conceptual challenge is correctly handling users without rides. A naive implementation that only aggregates rows from `Rides` would completely miss those users.

Several edge cases are important:

- A user may have zero rides.
- A user may have many rides that need aggregation.
- Multiple users may have the same total distance.
- The `Rides` table could even be empty.
- The `Users` table guarantees unique `user_id` values.

The correct solution must therefore combine aggregation with a join that preserves all users.

## Approaches

### Brute Force Approach

A brute force approach would process each user individually. For every user in the `Users` table, we would scan the entire `Rides` table and sum all ride distances that belong to that user.

Conceptually, the algorithm works like this:

1. Iterate through each user.
2. Initialize their total distance to `0`.
3. Scan every ride in the `Rides` table.
4. If the ride belongs to the current user, add its distance.
5. Store the computed total.

This approach is correct because every ride is checked against every user, guaranteeing that all matching rides are included in the total.

However, this is inefficient because the `Rides` table is repeatedly scanned for every user. If there are `U` users and `R` rides, the total work becomes `O(U × R)`.

### Optimal Approach

The key insight is that we should aggregate ride distances only once.

Instead of repeatedly scanning the rides table, we can:

1. Group rides by `user_id`
2. Compute the total distance for each user
3. Join the aggregated results with the `Users` table

In SQL, this is naturally expressed using:

- `LEFT JOIN` to preserve all users
- `SUM()` to aggregate distances
- `GROUP BY` to group rides per user
- `COALESCE()` to replace `NULL` totals with `0`

The `LEFT JOIN` is the most important part of the solution. It guarantees that users without rides are still included in the result set.

| --- | --- | --- | --- |

| Optimal | O(U + R) | O(U) | Aggregates rides once using grouping |

## Algorithm Walkthrough

1. Start with the `Users` table because every user must appear in the final output.
2. Perform a `LEFT JOIN` between `Users` and `Rides` using `user_id`. This ensures that:

- Users with rides get matched ride rows.
- Users without rides still appear with `NULL` ride values.
3. Group the joined rows by:

- `user_id`
- `name`

Grouping is necessary because one user may have multiple rides.
4. Use `SUM(distance)` to compute the total traveled distance for each user.
5. Users without rides will produce `NULL` when summed, because there are no matching ride rows. Use `COALESCE(SUM(distance), 0)` to convert these `NULL` values into `0`.
6. Sort the final result by `user_id` in ascending order.

### Why it works

The algorithm works because the `LEFT JOIN` preserves every row from the `Users` table, even when no matching rides exist. The grouping step combines all rides belonging to the same user, and `SUM()` correctly accumulates their total traveled distance. Finally, `COALESCE()` ensures that missing ride data is represented as `0` instead of `NULL`.

## Python Solution

LeetCode database problems are solved using SQL rather than standard Python classes. The following query is the complete accepted solution.

```python
SELECT
    u.user_id,
    u.name,
    COALESCE(SUM(r.distance), 0) AS travelled_distance
FROM Users u
LEFT JOIN Rides r
    ON u.user_id = r.user_id
GROUP BY u.user_id, u.name
ORDER BY u.user_id;
```

The query begins with the `Users` table because every user must appear in the output. The `LEFT JOIN` connects rides to their corresponding users while still preserving users who have no rides.

After joining, multiple ride rows may exist for a single user. The `GROUP BY` clause merges those rows into one result per user. The `SUM(r.distance)` expression accumulates all traveled distances.

For users without rides, `SUM()` returns `NULL`, so `COALESCE()` converts it into `0`.

Finally, the result is ordered by `user_id` in ascending order to satisfy the problem requirements.

## Edge Cases

### Users With No Rides

This is the most important edge case in the problem. A naive inner join would completely exclude users who never completed a ride. The implementation avoids this issue by using a `LEFT JOIN`, which preserves all rows from the `Users` table. `COALESCE()` then converts missing sums from `NULL` to `0`.

### Multiple Rides Per User

A user may have many ride records. Without proper grouping, the output would contain duplicate rows for the same user. The solution handles this by grouping on both `user_id` and `name`, ensuring one final row per user with the correct aggregated distance.

### Empty Rides Table

If the `Rides` table contains no rows, every user should still appear in the result with distance `0`. The `LEFT JOIN` guarantees that all users remain visible even when there are no matching ride records at all.
