[LeetCode Problem 2867](https://leetcode.com/problems/count-valid-paths-in-a-tree/)

**Difficulty:** 🔴 Hard  
**Topics:** Math, Dynamic Programming, Tree, Depth-First Search, Number Theory

## Solution

## Problem Understanding

We are given an undirected tree with `n` nodes labeled from `1` to `n`. A tree is a connected graph with no cycles, so there is exactly one simple path between every pair of nodes.

Each node label is also its numeric value, and we care about whether that value is prime or not. A path is considered valid if the sequence of node labels along the path contains exactly one prime number.

The task is to count how many distinct unordered node pairs `(a, b)` produce a valid path.

Since `(a, b)` and `(b, a)` represent the same path, we count each pair only once.

The input consists of:

- An integer `n`, the number of nodes
- An array `edges` containing `n - 1` undirected edges

The constraints are large:

- `n` can be as large as `100000`
- The graph is guaranteed to be a valid tree

These constraints immediately tell us that quadratic solutions are too slow. Any algorithm close to `O(n^2)` will time out.

The main difficulty is efficiently determining how many paths contain exactly one prime node.

Some important observations:

- Every path in a tree is unique
- Prime status depends only on node labels
- A valid path must contain one and only one prime node
- Paths containing zero primes or multiple primes must not be counted

Important edge cases include:

- Trees where every node is prime
- Trees where no node except one is prime
- Long chain trees, which can expose recursion depth or traversal inefficiencies
- Star-shaped trees, where many paths share the same center
- Single-node trees

The problem guarantees that the graph is connected and acyclic, so we never need to handle disconnected components or duplicate paths.

## Approaches

### Brute Force Approach

A straightforward approach is to examine every pair of nodes `(u, v)` and determine the unique path between them.

For each path:

1. Find the nodes on the path
2. Count how many prime-valued nodes appear
3. Increment the answer if the count equals exactly one

We could use DFS or BFS to reconstruct each path.

This works because trees guarantee a unique path between every pair of nodes.

However, there are `O(n^2)` node pairs, and each path traversal can take `O(n)` time in the worst case. That leads to `O(n^3)` complexity.

Even with preprocessing techniques like Lowest Common Ancestor, the solution still remains far too slow for `n = 100000`.

### Key Insight

Instead of examining every path individually, we should think about how valid paths are structured.

A valid path contains exactly one prime node.

Suppose we choose a prime node `p` as the unique prime on the path.

Then:

- Every other node on the path must be non-prime
- The path can extend into different non-prime connected regions adjacent to `p`

This suggests a component-based strategy:

1. Remove all prime nodes conceptually
2. The remaining non-prime nodes form connected components
3. For each prime node:

- Find the sizes of neighboring non-prime components
- Count all paths that use this prime as the unique prime

This transforms the problem into counting combinations between component sizes.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | `O(n^3)` | `O(n)` | Check every pair and reconstruct paths |
| Optimal | `O(n log log n + n)` | `O(n)` | Sieve primes and count paths using non-prime components |

## Algorithm Walkthrough

### Step 1: Build the adjacency list

Since the graph is a tree, an adjacency list is the most efficient representation.

For every edge `[u, v]`:

- Add `v` to `u`'s neighbor list
- Add `u` to `v`'s neighbor list

This allows efficient DFS traversal.

### Step 2: Precompute prime numbers using the Sieve of Eratosthenes

We need fast primality checks for node labels from `1` to `n`.

Using the sieve:

- Initialize all numbers as prime
- Mark multiples as non-prime

This preprocessing runs in nearly linear time.

The resulting array lets us check primality in `O(1)` time.

### Step 3: Find connected components of non-prime nodes

We now ignore prime nodes and only traverse non-prime nodes.

Using DFS:

- Start from every unvisited non-prime node
- Traverse only through non-prime neighbors
- Assign all reachable nodes the same component ID
- Record the component size

For example:

```
non-prime component A: size 4
non-prime component B: size 2
```

We store:

- `component_id[node]`
- `component_size[id]`

This preprocessing is critical because later we can instantly determine how many non-prime nodes are reachable from a prime node through a particular branch.

### Step 4: Process each prime node independently

Now consider a prime node `p`.

Each neighboring non-prime component represents one direction the path can extend without encountering another prime.

Suppose the neighboring component sizes are:

```
[3, 5, 2]
```

These represent groups of non-prime nodes reachable from `p`.

We count two kinds of valid paths.

#### Type 1: Paths where the prime is one endpoint

Examples:

```
(p, x)
```

where `x` is any non-prime node in a neighboring component.

This contributes:

```
3 + 5 + 2 = 10
```

paths.

#### Type 2: Paths passing through the prime

Examples:

```
x -> p -> y
```

where `x` and `y` come from different components.

We compute pair combinations incrementally.

If we process component sizes one by one:

```
sizes = [3, 5, 2]
```

Then:

- `3 * 5`
- `3 * 2`
- `5 * 2`

Total:

```
15 + 6 + 10 = 31
```

We avoid nested loops by maintaining a running prefix sum.

### Step 5: Accumulate the total answer

Add all contributions from every prime node.

The final result is the total number of valid paths.

## Why it works

Every valid path must contain exactly one prime node.

If we fix that prime node:

- Every other node on the path must be non-prime
- The path can only move through non-prime connected regions
- Any valid path either:

- Starts or ends at the prime
- Passes through the prime connecting two distinct non-prime components

Because trees contain unique paths, every valid path is counted exactly once by its unique prime node.

Therefore the algorithm is both complete and non-overcounting.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Solution:
    def countPaths(self, n: int, edges: List[List[int]]) -> int:
        graph = [[] for _ in range(n + 1)]

        for u, v in edges:
            graph[u].append(v)
            graph[v].append(u)

        # Sieve of Eratosthenes
        is_prime = [True] * (n + 1)
        is_prime[0] = False
        is_prime[1] = False

        i = 2
        while i * i <= n:
            if is_prime[i]:
                j = i * i
                while j <= n:
                    is_prime[j] = False
                    j += i
            i += 1

        component_id = [0] * (n + 1)
        component_size = defaultdict(int)

        current_component = 0

        def dfs(node: int, parent: int, cid: int) -> int:
            component_id[node] = cid
            size = 1

            for neighbor in graph[node]:
                if neighbor == parent:
                    continue

                if is_prime[neighbor]:
                    continue

                if component_id[neighbor] != 0:
                    continue

                size += dfs(neighbor, node, cid)

            return size

        # Build connected components of non-prime nodes
        for node in range(1, n + 1):
            if is_prime[node]:
                continue

            if component_id[node] != 0:
                continue

            current_component += 1
            size = dfs(node, -1, current_component)
            component_size[current_component] = size

        answer = 0

        # Process each prime node
        for prime_node in range(1, n + 1):
            if not is_prime[prime_node]:
                continue

            total_seen = 0

            used_components = set()

            for neighbor in graph[prime_node]:
                if is_prime[neighbor]:
                    continue

                cid = component_id[neighbor]

                if cid in used_components:
                    continue

                used_components.add(cid)

                size = component_size[cid]

                # Paths connecting this component with previous ones
                answer += total_seen * size

                # Paths where prime_node is one endpoint
                answer += size

                total_seen += size

        return answer
```

The implementation begins by constructing the adjacency list representation of the tree.

Next, the Sieve of Eratosthenes computes primality for all labels from `1` to `n`. This allows constant-time prime checks throughout the algorithm.

The DFS phase identifies connected components formed exclusively by non-prime nodes. Each component receives a unique ID, and its size is stored.

The main counting logic iterates through every prime node. For each neighboring non-prime component, the algorithm counts:

- Paths from the prime node directly into that component
- Paths connecting nodes from different components through the prime node

The `total_seen` variable efficiently accumulates previous component sizes so that all cross-component pairs can be counted in linear time.

The `used_components` set prevents double counting if multiple neighbors belong to the same component.

## Go Solution

```go
package main

func countPaths(n int, edges [][]int) int64 {
	graph := make([][]int, n+1)

	for _, edge := range edges {
		u, v := edge[0], edge[1]
		graph[u] = append(graph[u], v)
		graph[v] = append(graph[v], u)
	}

	// Sieve of Eratosthenes
	isPrime := make([]bool, n+1)

	for i := 2; i <= n; i++ {
		isPrime[i] = true
	}

	for i := 2; i*i <= n; i++ {
		if isPrime[i] {
			for j := i * i; j <= n; j += i {
				isPrime[j] = false
			}
		}
	}

	componentID := make([]int, n+1)
	componentSize := make(map[int]int)

	currentComponent := 0

	var dfs func(int, int, int) int

	dfs = func(node int, parent int, cid int) int {
		componentID[node] = cid

		size := 1

		for _, neighbor := range graph[node] {
			if neighbor == parent {
				continue
			}

			if isPrime[neighbor] {
				continue
			}

			if componentID[neighbor] != 0 {
				continue
			}

			size += dfs(neighbor, node, cid)
		}

		return size
	}

	// Build non-prime connected components
	for node := 1; node <= n; node++ {
		if isPrime[node] {
			continue
		}

		if componentID[node] != 0 {
			continue
		}

		currentComponent++

		size := dfs(node, -1, currentComponent)
		componentSize[currentComponent] = size
	}

	var answer int64 = 0

	// Process each prime node
	for primeNode := 1; primeNode <= n; primeNode++ {
		if !isPrime[primeNode] {
			continue
		}

		totalSeen := 0
		usedComponents := make(map[int]bool)

		for _, neighbor := range graph[primeNode] {
			if isPrime[neighbor] {
				continue
			}

			cid := componentID[neighbor]

			if usedComponents[cid] {
				continue
			}

			usedComponents[cid] = true

			size := componentSize[cid]

			answer += int64(totalSeen * size)
			answer += int64(size)

			totalSeen += size
		}
	}

	return answer
}
```

The Go implementation closely mirrors the Python logic.

Some Go-specific considerations:

- The result uses `int64` because the number of valid paths can exceed 32-bit integer limits.
- Maps are used for component sizes and duplicate component detection.
- Recursive DFS is implemented using a function variable closure.
- Slices are used for adjacency lists.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | `O(n log log n + n)` | Sieve plus DFS traversals |
| Space | `O(n)` | Adjacency list, component arrays, recursion stack |

The sieve computation costs `O(n log log n)`, which is standard for prime preprocessing.

All DFS traversals together visit each node and edge at most once, giving linear complexity.

The counting phase also processes each edge a constant number of times.

Overall, the algorithm is efficient enough for `n = 100000`.

## Edge Cases

### Single-node tree

A tree with only one node can never produce a valid path unless that single node itself forms a valid path. However, the problem counts unordered node pairs, and there is no meaningful path containing exactly one prime across two distinct nodes.

The implementation handles this naturally because no component combinations are formed and the answer remains zero.

### Multiple neighbors belonging to the same non-prime component

A prime node may connect to several nodes that are already part of the same non-prime connected component.

Without protection, the algorithm could count the same component multiple times and overcount paths.

The `used_components` set ensures each component contributes only once per prime node.

### Trees where every node is prime

If all nodes are prime, then every path with length greater than zero contains at least two primes.

No valid paths should exist.

The implementation correctly handles this because:

- No non-prime components are created
- Every prime node finds no usable neighboring components
- The answer remains zero
