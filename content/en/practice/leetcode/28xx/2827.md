# LeetCode 2827 - Number of Beautiful Integers in the Range

## Problem Understanding

The problem asks us to count how many integers in the inclusive range `[low, high]` satisfy two conditions simultaneously:

1. The number contains an equal count of even digits and odd digits.
2. The number is divisible by `k`.

A digit is considered even if it belongs to `{0, 2, 4, 6, 8}` and odd if it belongs to `{1, 3, 5, 7, 9}`.

For example, the number `12` contains one even digit (`2`) and one odd digit (`1`), so the counts are balanced. Since `12 % 3 == 0`, it is considered beautiful.

The input consists of three integers:

- `low`, the lower bound of the range
- `high`, the upper bound of the range
- `k`, the divisor requirement

The output is a single integer representing how many numbers in `[low, high]` are beautiful.

The constraints are important:

- `high <= 10^9`
- `k <= 20`

Since the range can contain up to one billion numbers, iterating through every value directly is far too slow. This immediately suggests that we need a more advanced counting technique.

A few important observations arise from the problem definition:

- A number can only have equal counts of even and odd digits if its total number of digits is even.
- Leading zeros should not count as digits.
- Divisibility depends on the full numeric value, not the digit counts alone.
- The answer for a range `[low, high]` is naturally expressed as:

$$count(high) - count(low - 1)$$

where `count(x)` returns the number of beautiful integers in `[0, x]`.

These observations point strongly toward digit dynamic programming, commonly called Digit DP.

## Approaches

### Brute Force Approach

The brute force method checks every integer from `low` to `high`.

For each number:

1. Extract all digits.
2. Count how many digits are even.
3. Count how many digits are odd.
4. Check whether the counts are equal.
5. Check whether the number is divisible by `k`.

If both conditions hold, increment the answer.

This approach is straightforward and correct because it explicitly verifies the problem definition for every number.

However, it is too slow for the constraints. In the worst case, the range size can approach `10^9`, making direct iteration impossible within time limits.

### Optimal Approach, Digit DP

The key insight is that we do not actually need to enumerate every number individually.

Instead, we can construct valid numbers digit by digit while tracking enough information to determine whether the final number will be beautiful.

This is a classic Digit DP setup because:

- We need to count numbers within a numeric bound.
- The property depends on the digits.
- The divisibility condition can be tracked incrementally.

While building a number from left to right, we maintain:

- Current position in the digit string
- Difference between even and odd digit counts
- Current remainder modulo `k`
- Whether we are still restricted by the upper bound
- Whether the number has started yet

This allows us to explore all valid digit combinations efficiently without enumerating the entire numeric range.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O((high - low + 1) * log(high)) | O(1) | Checks every number individually |
| Optimal | O(digits * k * state_space * 10) | O(digits * k * state_space) | Uses memoized Digit DP |

## Algorithm Walkthrough

### Step 1: Convert the Range Query into Prefix Queries

Define a helper function:

```
count(x)
```

which returns the number of beautiful integers in `[0, x]`.

Then the final answer becomes:

```
count(high) - count(low - 1)
```

This is a standard trick for range counting problems.

### Step 2: Represent the Upper Bound as Digits

Convert `x` into a string so we can process digits from left to right.

For example:

```
x = 528
digits = "528"
```

Digit DP works by deciding each digit one position at a time.

### Step 3: Define the DP State

We use DFS with memoization.

The state contains:

1. `position`

- Which digit index we are currently filling.
2. `balance`

- Difference between even and odd digit counts.
- We add `+1` for an even digit.
- We subtract `1` for an odd digit.
3. `remainder`

- Current number modulo `k`.
4. `tight`

- Whether the current prefix exactly matches the upper bound.
- If true, the next digit cannot exceed the bound digit.
5. `started`

- Whether we have placed any non-leading-zero digit yet.

These states uniquely determine all future possibilities.

### Step 4: Handle Leading Zeros Carefully

Leading zeros should not contribute to digit counts.

If the number has not started yet and we place another `0`:

- The balance does not change.
- The remainder stays `0`.
- The number is still considered not started.

This prevents artificial digit counting from leading zeros.

### Step 5: Transition Through Digits

At each position:

1. Determine the maximum allowed digit.

- If `tight == true`, we cannot exceed the corresponding bound digit.
- Otherwise, we may use `9`.
2. Try every possible digit.
3. Update:

- Tightness
- Started flag
- Balance
- Modulo remainder

The new remainder is:

$$new\_remainder = (old\_remainder \times 10 + digit) \bmod k$$

### Step 6: Define the Base Case

When all digits are processed:

- The number is valid if:

- We actually formed a number
- `balance == 0`
- `remainder == 0`

If all conditions hold, return `1`, otherwise `0`.

### Step 7: Use Memoization

Many states repeat during recursion.

Memoization ensures each state is computed only once, reducing exponential recursion to manageable complexity.

### Why it works

The DP explores every valid number within the bound exactly once through digit-by-digit construction. The state fully captures all information needed to determine whether a partially constructed number can eventually become beautiful. Since every transition correctly updates digit parity counts and modulo remainder, the final states precisely correspond to beautiful integers. Memoization guarantees efficient reuse of repeated subproblems.

## Python Solution

```python
from functools import lru_cache
from typing import *

class Solution:
    def numberOfBeautifulIntegers(self, low: int, high: int, k: int) -> int:
        
        def count_beautiful(limit: int) -> int:
            if limit <= 0:
                return 0
            
            digits = str(limit)
            n = len(digits)

            @lru_cache(None)
            def dfs(
                pos: int,
                balance: int,
                remainder: int,
                tight: bool,
                started: bool
            ) -> int:
                
                if pos == n:
                    return int(started and balance == 0 and remainder == 0)

                upper = int(digits[pos]) if tight else 9
                total = 0

                for digit in range(upper + 1):
                    next_tight = tight and (digit == upper)

                    if not started and digit == 0:
                        total += dfs(
                            pos + 1,
                            balance,
                            remainder,
                            next_tight,
                            False
                        )
                    else:
                        next_balance = balance + 1 if digit % 2 == 0 else balance - 1
                        next_remainder = (remainder * 10 + digit) % k

                        total += dfs(
                            pos + 1,
                            next_balance,
                            next_remainder,
                            next_tight,
                            True
                        )

                return total

            return dfs(0, 0, 0, True, False)

        return count_beautiful(high) - count_beautiful(low - 1)
```

The implementation begins by defining `count_beautiful(limit)`, which computes how many beautiful integers exist in `[0, limit]`.

If `limit <= 0`, the answer is immediately zero because no positive beautiful integers exist.

The recursive `dfs` function performs the digit DP. The parameters capture the full DP state:

- `pos` tracks the current digit position.
- `balance` tracks the difference between counts of even and odd digits.
- `remainder` stores the current modulo `k`.
- `tight` indicates whether the current prefix is still bounded.
- `started` distinguishes leading zeros from actual digits.

At each recursive step, the code iterates through all valid digits.

If the number has not started and the digit is zero, the recursion continues without affecting parity counts or modulo state.

Otherwise:

- Even digits increment the balance.
- Odd digits decrement the balance.
- The modulo remainder is updated incrementally.

The base case validates that:

- A number was formed.
- The balance is exactly zero.
- The number is divisible by `k`.

Finally, the range answer is computed using prefix subtraction.

## Go Solution

```go
package main

func numberOfBeautifulIntegers(low int, high int, k int) int {

	countBeautiful := func(limit int) int {
		if limit <= 0 {
			return 0
		}

		digits := []byte{}
		for _, ch := range []byte(string(runeSlice(limit))) {
			digits = append(digits, ch)
		}

		n := len(digits)

		type State struct {
			pos       int
			balance   int
			remainder int
			tight     bool
			started   bool
		}

		memo := make(map[State]int)

		var dfs func(int, int, int, bool, bool) int

		dfs = func(pos, balance, remainder int, tight, started bool) int {

			if pos == n {
				if started && balance == 0 && remainder == 0 {
					return 1
				}
				return 0
			}

			state := State{
				pos,
				balance,
				remainder,
				tight,
				started,
			}

			if !tight {
				if val, ok := memo[state]; ok {
					return val
				}
			}

			upper := 9
			if tight {
				upper = int(digits[pos] - '0')
			}

			total := 0

			for digit := 0; digit <= upper; digit++ {

				nextTight := tight && (digit == upper)

				if !started && digit == 0 {

					total += dfs(
						pos+1,
						balance,
						remainder,
						nextTight,
						false,
					)

				} else {

					nextBalance := balance
					if digit%2 == 0 {
						nextBalance++
					} else {
						nextBalance--
					}

					nextRemainder := (remainder*10 + digit) % k

					total += dfs(
						pos+1,
						nextBalance,
						nextRemainder,
						nextTight,
						true,
					)
				}
			}

			if !tight {
				memo[state] = total
			}

			return total
		}

		return dfs(0, 0, 0, true, false)
	}

	return countBeautiful(high) - countBeautiful(low-1)
}

func runeSlice(num int) []rune {
	if num == 0 {
		return []rune{'0'}
	}

	digits := []rune{}

	for num > 0 {
		digits = append([]rune{rune('0' + num%10)}, digits...)
		num /= 10
	}

	return digits
}
```

The Go implementation follows the same DP structure as the Python solution, but there are several language-specific differences.

Python uses `lru_cache` automatically for memoization, while Go uses a custom `map[State]int`.

Go structs are used to represent memoization keys efficiently.

Unlike Python strings, Go strings are immutable byte slices, so digit extraction is handled manually.

The memoization optimization stores results only when `tight == false`. Tight states depend on the exact upper bound path and are less reusable.

All integer arithmetic safely fits within Go's standard `int` type because the total state space is relatively small.

## Edge Cases

One important edge case is handling leading zeros correctly. In Digit DP, numbers shorter than the upper bound length are represented using leading zeros. If those zeros were counted as even digits, many invalid numbers would incorrectly appear balanced. The implementation avoids this by using the `started` flag. Until the first nonzero digit appears, zeros do not affect balance or modulo state.

Another important case is single-digit numbers. A single digit can never have equal counts of even and odd digits because the total digit count is odd. The algorithm naturally handles this because the final balance cannot become zero after exactly one digit contribution.

A third subtle case is when `low = 1`. The computation uses:

```
count(high) - count(low - 1)
```

which becomes:

```
count(high) - count(0)
```

The implementation explicitly returns zero when the limit is nonpositive, preventing invalid recursion on negative values.

A final edge case involves large values of `k`. Since divisibility must be tracked incrementally, incorrect modulo transitions can easily produce wrong answers. The implementation correctly updates remainders using:

$$(remainder \times 10 + digit) \bmod k$$

which guarantees accurate divisibility tracking for all prefixes.
