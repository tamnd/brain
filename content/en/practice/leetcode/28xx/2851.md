## Solution

## Problem Understanding

The operation described in the problem is a cyclic rotation of the string. In one move, we choose a suffix of length `l`, remove it from the end, and place it at the beginning. If the string length is `n`, then choosing a suffix of length `l` is equivalent to rotating the string to the right by `l` positions.

For example:

- `"abcd"` rotated right by `1` becomes `"dabc"`
- `"abcd"` rotated right by `2` becomes `"cdab"`
- `"abcd"` rotated right by `3` becomes `"bcda"`

Since `0 < l < n`, every operation must perform a non-trivial rotation.

The problem asks for the number of distinct sequences of exactly `k` operations that transform the initial string `s` into the target string `t`.

The important detail is that we count sequences of operations, not just whether the transformation is possible. Two different choices of suffix lengths count as different ways, even if they lead through the same intermediate states.

The constraints are extremely large:

- `n` can be up to `5 * 10^5`
- `k` can be up to `10^15`

These limits immediately rule out any simulation over all operations. We need a mathematical characterization of the transitions.

Another critical observation is that every operation preserves the cyclic structure of the string. Therefore, `t` must be some rotation of `s`, otherwise the answer is automatically `0`.

Several edge cases are important:

- `t` may not be a rotation of `s`
- `s == t`, but we still must perform exactly `k` non-trivial rotations
- Multiple rotations may produce the same string when the string is periodic, such as `"ababab"`
- `k` is extremely large, so repeated simulation is impossible
- The number of valid rotations can be large when repeated patterns exist

## Approaches

### Brute Force Approach

A direct approach would model every possible operation explicitly.

From a string of length `n`, there are `n - 1` possible suffix lengths we can choose in each step. We could recursively generate all possible sequences of operations up to depth `k`, tracking how many ways lead to `t`.

At each step:

- Choose a suffix length `l`
- Rotate the string
- Continue recursively

This is correct because it explores every legal sequence exactly once.

However, the branching factor is `n - 1`, and the depth is `k`, so the total number of states becomes enormous:

$$(n-1)^k$$

Even for small inputs this becomes infeasible. Since `k` may be `10^{15}`, brute force is completely impossible.

### Key Insight

The crucial observation is that every operation only changes the current rotation offset.

Suppose we number the `n` cyclic rotations of `s`:

$$R_0, R_1, R_2, \dots, R_{n-1}$$

A rotation by suffix length `l` moves from rotation `i` to:

$$(i + l) \bmod n$$

Since `1 \le l < n`, we can move from any rotation to any other rotation except staying in place in one move.

This creates a complete directed graph on `n` states without self-loops.

Now the problem becomes:

- Start at rotation `0`
- After exactly `k` transitions, how many ways reach a rotation corresponding to `t`?

We also need to know how many rotation indices produce `t`. This can be found efficiently using KMP string matching on:

$$s + s$$

If `t` appears starting at index `i`, then rotation `i` transforms `s` into `t`.

The graph structure is highly symmetric, so instead of tracking all `n` states individually, we only track two categories:

- `same`, the number of ways to be at rotation `0`
- `diff`, the number of ways to be at any non-zero rotation

This leads to a simple linear recurrence.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | $O((n-1)^k)$ | $O(k)$ | Enumerates all operation sequences |
| Optimal | $O(n + \log k)$ | $O(n)$ | Uses KMP and matrix exponentiation |

## Algorithm Walkthrough

### Step 1: Model rotations as states

Every possible string obtainable from `s` is one of its cyclic rotations.

Define rotation `i` as:

$$s[i:] + s[:i]$$

There are `n` possible rotation indices.

### Step 2: Find all rotations equal to `t`

We must determine which rotation indices produce `t`.

To do this efficiently, concatenate:

$$text = s + s$$

Then search for occurrences of `t` inside `text[:-1]`.

Each occurrence starting at index `i` corresponds to rotation `i`.

KMP is used because it finds all matches in linear time.

### Step 3: Build the transition recurrence

From any rotation:

- There are `n - 1` valid moves
- We may move to any different rotation
- We may not stay where we are

Let:

- `same[k]` = ways to be at rotation `0` after `k` operations
- `diff[k]` = ways to be at a particular non-zero rotation after `k` operations

Transitions:

If we want to end at rotation `0`:

- We must come from one of the `n - 1` non-zero rotations

So:

$$same' = (n-1) \cdot diff$$

If we want to end at a non-zero rotation:

- We may come from rotation `0`
- Or from one of the other `n - 2` non-zero rotations

So:

$$diff' = same + (n-2)\cdot diff$$

### Step 4: Express as matrix multiplication

The recurrence becomes:

$$\begin{bmatrix}
same' \\
diff'
\end{bmatrix}
=
\begin{bmatrix}
0 & n-1 \\
1 & n-2
\end{bmatrix}
\begin{bmatrix}
same \\
diff
\end{bmatrix}$$

Initial state:

$$same = 1,\quad diff = 0$$

We need the matrix raised to the `k`th power.

### Step 5: Use fast exponentiation

Since `k` may be as large as `10^{15}`, we compute the matrix power using binary exponentiation in:

$$O(\log k)$$

### Step 6: Compute the final answer

Suppose:

- `cnt_same` is `1` if rotation `0` equals `t`
- `cnt_diff` is the number of non-zero rotations equal to `t`

Then:

$$answer =
same_k \cdot cnt_same +
diff_k \cdot cnt_diff$$

Take everything modulo:

$$10^9 + 7$$

### Why it works

The key invariant is that every non-zero rotation is structurally identical in the transition graph.

From any state:

- There are exactly `n - 1` outgoing edges
- Every other state is reachable in exactly one move

Therefore all non-zero rotations always have identical counts, allowing us to compress the entire graph into only two states, `same` and `diff`. Matrix exponentiation correctly computes repeated transitions over exactly `k` operations.

## Python Solution

```python
class Solution:
    def numberOfWays(self, s: str, t: str, k: int) -> int:
        MOD = 10**9 + 7
        n = len(s)

        def kmp_search(text: str, pattern: str):
            m = len(pattern)

            lps = [0] * m
            j = 0

            for i in range(1, m):
                while j > 0 and pattern[i] != pattern[j]:
                    j = lps[j - 1]

                if pattern[i] == pattern[j]:
                    j += 1
                    lps[i] = j

            matches = []
            j = 0

            for i in range(len(text)):
                while j > 0 and text[i] != pattern[j]:
                    j = lps[j - 1]

                if text[i] == pattern[j]:
                    j += 1

                if j == m:
                    matches.append(i - m + 1)
                    j = lps[j - 1]

            return matches

        rotations = kmp_search(s + s[:-1], t)

        if not rotations:
            return 0

        same_target = 1 if 0 in rotations else 0
        diff_target = len(rotations) - same_target

        def mat_mult(a, b):
            return [
                [
                    (a[0][0] * b[0][0] + a[0][1] * b[1][0]) % MOD,
                    (a[0][0] * b[0][1] + a[0][1] * b[1][1]) % MOD,
                ],
                [
                    (a[1][0] * b[0][0] + a[1][1] * b[1][0]) % MOD,
                    (a[1][0] * b[0][1] + a[1][1] * b[1][1]) % MOD,
                ],
            ]

        def mat_pow(mat, power):
            result = [[1, 0], [0, 1]]

            while power > 0:
                if power & 1:
                    result = mat_mult(result, mat)

                mat = mat_mult(mat, mat)
                power >>= 1

            return result

        transition = [
            [0, n - 1],
            [1, n - 2]
        ]

        power_matrix = mat_pow(transition, k)

        same_k = power_matrix[0][0]
        diff_k = power_matrix[1][0]

        answer = (
            same_k * same_target +
            diff_k * diff_target
        ) % MOD

        return answer
```

The implementation begins with a KMP string matching routine. This efficiently finds all cyclic shifts of `s` that equal `t`.

The search is performed on `s + s[:-1]` instead of `s + s` to avoid counting the full wraparound duplicate rotation.

After identifying matching rotations, the code separates them into:

- rotation `0`
- non-zero rotations

The transition matrix represents the recurrence relation between the compressed states `same` and `diff`.

Matrix exponentiation computes the state after exactly `k` operations in logarithmic time.

Finally, the answer is assembled by multiplying the number of ways to reach each category by how many target rotations belong to that category.

## Go Solution

```go
package main

const MOD int64 = 1_000_000_007

func numberOfWays(s string, t string, k int64) int {
	n := len(s)

	rotations := kmpSearch(s+s[:n-1], t)

	if len(rotations) == 0 {
		return 0
	}

	sameTarget := int64(0)
	if rotations[0] == 0 {
		sameTarget = 1
	}

	diffTarget := int64(len(rotations)) - sameTarget

	type Matrix [2][2]int64

	matMult := func(a, b Matrix) Matrix {
		var res Matrix

		for i := 0; i < 2; i++ {
			for j := 0; j < 2; j++ {
				for x := 0; x < 2; x++ {
					res[i][j] = (res[i][j] + a[i][x]*b[x][j]) % MOD
				}
			}
		}

		return res
	}

	matPow := func(mat Matrix, power int64) Matrix {
		result := Matrix{
			{1, 0},
			{0, 1},
		}

		for power > 0 {
			if power&1 == 1 {
				result = matMult(result, mat)
			}

			mat = matMult(mat, mat)
			power >>= 1
		}

		return result
	}

	transition := Matrix{
		{0, int64(n - 1)},
		{1, int64(n - 2)},
	}

	powerMatrix := matPow(transition, k)

	sameK := powerMatrix[0][0]
	diffK := powerMatrix[1][0]

	answer := (sameK*sameTarget + diffK*diffTarget) % MOD

	return int(answer)
}

func kmpSearch(text string, pattern string) []int {
	m := len(pattern)

	lps := make([]int, m)

	j := 0

	for i := 1; i < m; i++ {
		for j > 0 && pattern[i] != pattern[j] {
			j = lps[j-1]
		}

		if pattern[i] == pattern[j] {
			j++
			lps[i] = j
		}
	}

	var matches []int
	j = 0

	for i := 0; i < len(text); i++ {
		for j > 0 && text[i] != pattern[j] {
			j = lps[j-1]
		}

		if text[i] == pattern[j] {
			j++
		}

		if j == m {
			matches = append(matches, i-m+1)
			j = lps[j-1]
		}
	}

	return matches
}
```

The Go implementation follows the same logic as the Python version, but uses fixed-size arrays for the `2 x 2` matrix.

All arithmetic uses `int64` to avoid overflow before applying the modulus.

Slices are used naturally for KMP arrays and match storage.

Possible rotations:

| Rotation Index | String |

| 1 | bcda |
| 2 | cdab |
| 3 | dabc |

Target rotation indices:

$$\{2\}$$

So:

- `same_target = 0`
- `diff_target = 1`

Initial state:

| Step | same | diff |
| --- | --- | --- |
| 0 | 1 | 0 |

After one operation:

$$same = 3 \cdot 0 = 0$$

$$diff = 1 + 2 \cdot 0 = 1$$

| Step | same | diff |
| --- | --- | --- |
| 1 | 0 | 1 |

After two operations:

$$same = 3 \cdot 1 = 3$$

$$diff = 0 + 2 \cdot 1 = 2$$

| Step | same | diff |
| --- | --- | --- |
| 2 | 3 | 2 |

Since the target is a non-zero rotation:

$$answer = 2$$

- `same_target = 1`

$$0 \cdot 1 + 1 \cdot 2 = 2$$

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | $O(n + \log k)$ | KMP is linear, matrix exponentiation is logarithmic |
| Space | $O(n)$ | KMP prefix table and match storage |

The dominant linear cost comes from KMP preprocessing and matching over strings of size proportional to `n`.

The matrix exponentiation operates on a constant-size `2 x 2` matrix, so each multiplication is constant time. Since binary exponentiation performs `O(log k)` multiplications, the total additional cost is logarithmic.

## Edge Cases

One important edge case occurs when `t` is not a rotation of `s`. Since every operation preserves cyclic order, no sequence of operations can ever create a string outside the rotation class of `s`. The implementation handles this by using KMP to search for `t` inside `s + s[:-1]`. If no matches are found, the function immediately returns `0`.

Another tricky case occurs when `s == t`. A naive implementation might incorrectly assume there is always exactly one way to remain unchanged. However, the operation cannot leave the string untouched directly because every move must rotate by at least one position. The only valid ways are sequences whose total rotation offset becomes `0 mod n`. The recurrence correctly models this behavior through the `same` state.

Periodic strings create another subtle issue. For example, `"ababab"` has multiple different rotation indices producing the same string. If we only tracked whether a string equals `t`, we would undercount valid transformations. The implementation instead counts all rotation indices matching `t`, separating them into rotation `0` and non-zero rotations. This guarantees every valid operation sequence is counted correctly.
