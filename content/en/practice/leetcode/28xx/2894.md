## Solution

## Problem Understanding

The problem gives us two positive integers, `n` and `m`. We must examine every integer in the inclusive range `[1, n]` and divide those numbers into two groups.

The first group contains numbers that are **not divisible** by `m`. The sum of these numbers is called `num1`.

The second group contains numbers that **are divisible** by `m`. The sum of these numbers is called `num2`.

Our task is to return the value:

$$num1 - num2$$

In simpler terms, we iterate through all integers from `1` to `n`, decide whether each number is divisible by `m`, and add it to the appropriate sum. After processing all numbers, we subtract the divisible sum from the non-divisible sum.

For example, if `n = 10` and `m = 3`, the numbers divisible by `3` are:

$$3, 6, 9$$

Their sum is:

$$18$$

The remaining numbers are:

$$1, 2, 4, 5, 7, 8, 10$$

Their sum is:

$$37$$

The final answer is:

$$37 - 18 = 19$$

The constraints are small:

$$1 \le n, m \le 1000$$

This tells us that even a straightforward iteration through all numbers is completely acceptable. A loop from `1` to `n` performs at most `1000` iterations, which is trivial for modern computers.

There are still several important edge cases to consider carefully.

If `m > n`, then no number in `[1, n]` is divisible by `m`. In that case, `num2 = 0`, and the answer becomes the sum of all numbers from `1` to `n`.

If `m = 1`, then every number is divisible by `1`. That means `num1 = 0`, and the answer becomes the negative of the total sum.

Another important case is the smallest valid input, such as `n = 1` and `m = 1`. The implementation must still correctly classify the single value.

## Approaches

### Brute Force Approach

The most direct solution is to iterate through every integer from `1` to `n`.

For each number:

- If the number is divisible by `m`, add it to `num2`
- Otherwise, add it to `num1`

At the end, return:

$$num1 - num2$$

This approach is correct because every integer in the range is examined exactly once and placed into the correct category.

Even though this is called a brute force approach, it is already efficient enough for the given constraints because `n` is at most `1000`.

### Key Observation for an Optimal Approach

We can optimize further using mathematical formulas.

The sum of integers from `1` to `n` is:

$$\frac{n(n+1)}{2}$$

We can also compute the sum of all numbers divisible by `m`.

The divisible numbers form this sequence:

$$m, 2m, 3m, \dots, km$$

Where:

$$k = \left\lfloor \frac{n}{m} \right\rfloor$$

The sum of these divisible numbers is:

$$m \times \frac{k(k+1)}{2}$$

Let:

- `totalSum` = sum of all numbers from `1` to `n`
- `divisibleSum` = sum of numbers divisible by `m`

Then:

$$num1 = totalSum - divisibleSum$$

So the final result becomes:

$$(totalSum - divisibleSum) - divisibleSum$$

$$totalSum - 2 \times divisibleSum$$

This removes the need for iteration entirely.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Iterates through every number and classifies it |
| Optimal | O(1) | O(1) | Uses arithmetic series formulas |

## Algorithm Walkthrough

### Optimal Mathematical Algorithm

1. Compute the total sum of integers from `1` to `n`.

We use the arithmetic series formula:

$$totalSum = \frac{n(n+1)}{2}$$

This gives the sum of all numbers in constant time.
2. Determine how many numbers are divisible by `m`.

Every multiple of `m` inside `[1, n]` looks like:

$$m, 2m, 3m, \dots$$

The largest multiple is:

$$\left\lfloor \frac{n}{m} \right\rfloor \times m$$

So the count of divisible numbers is:

$$count = n // m$$
3. Compute the sum of all divisible numbers.

The divisible sequence is:

$$m(1 + 2 + 3 + \dots + count)$$

Using the arithmetic series formula:

$$divisibleSum = m \times \frac{count(count+1)}{2}$$
4. Compute the non-divisible sum.

Since all numbers belong to exactly one of the two groups:

$$nonDivisibleSum = totalSum - divisibleSum$$
5. Return the required difference.

The answer is:

$$nonDivisibleSum - divisibleSum$$

Which simplifies to:

$$totalSum - 2 \times divisibleSum$$

### Why it works

Every integer from `1` to `n` belongs to exactly one category, either divisible by `m` or not divisible by `m`. The arithmetic series formulas correctly compute the total sum and the divisible-number sum. Subtracting the divisible sum from the total gives the non-divisible sum, and the final subtraction produces the required answer. Since no numbers are omitted or double-counted, the algorithm is correct.

## Python Solution

```python
class Solution:
    def differenceOfSums(self, n: int, m: int) -> int:
        total_sum = n * (n + 1) // 2

        divisible_count = n // m
        divisible_sum = m * divisible_count * (divisible_count + 1) // 2

        return total_sum - 2 * divisible_sum
```

The implementation begins by computing the sum of all integers from `1` to `n` using the arithmetic progression formula.

Next, the code determines how many multiples of `m` exist within the range `[1, n]`. This is done using integer division:

```python
divisible_count = n // m
```

For example, if `n = 10` and `m = 3`, then:

```python
10 // 3 = 3
```

meaning the divisible numbers are:

```python
3, 6, 9
```

The sum of these divisible numbers is then computed using another arithmetic series formula. Since the divisible numbers are simply `m` multiplied by `1` through `divisible_count`, we multiply the standard sum formula by `m`.

Finally, the result is computed using the simplified equation:

```python
total_sum - 2 * divisible_sum
```

This works because:

```python
(non_divisible_sum - divisible_sum)
```

is equivalent to:

```python
(total_sum - divisible_sum - divisible_sum)
```

## Go Solution

```go
func differenceOfSums(n int, m int) int {
    totalSum := n * (n + 1) / 2

    divisibleCount := n / m
    divisibleSum := m * divisibleCount * (divisibleCount + 1) / 2

    return totalSum - 2*divisibleSum
}
```

The Go implementation follows the exact same mathematical approach as the Python solution.

Go uses integer arithmetic automatically when working with `int` values, so division operations remain safe and precise for this problem. Since the constraints are very small, integer overflow is not a concern here.

No additional data structures are required, and the implementation uses only a few integer variables.

## Edge Cases

### Case 1: `m` is Larger Than `n`

When `m > n`, there are no multiples of `m` inside the range `[1, n]`.

For example:

```
n = 5, m = 6
```

No numbers are divisible by `6`, so the divisible sum is `0`. The implementation handles this naturally because:

```
divisible_count = n // m
```

becomes:

```
5 // 6 = 0
```

which correctly produces a divisible sum of `0`.

### Case 2: `m = 1`

Every integer is divisible by `1`.

For example:

```
n = 5, m = 1
```

The non-divisible group is empty, so the result must be negative. This is a common source of bugs if the implementation assumes both groups contain values.

The formula-based solution handles this correctly because:

```
divisible_count = n
```

and the divisible sum becomes the total sum.

### Case 3: Smallest Possible Input

The smallest valid input is:

```
n = 1, m = 1
```

There is only one number to classify. Since `1` is divisible by `1`, the answer is:

```
0 - 1 = -1
```

Small inputs often expose off-by-one errors or incorrect loop boundaries. The mathematical solution avoids these issues because it does not rely on iteration boundaries at all.
