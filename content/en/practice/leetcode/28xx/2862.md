## Solution

## Problem Understanding

The problem asks us to select a subset of elements from a **1-indexed** array `nums` such that for every pair of selected indices $i$ and $j$, the product $i \cdot j$ is a perfect square. The goal is to maximize the sum of the elements at the selected indices.

In other words, if we imagine picking indices from `nums`, we can only select a set where multiplying any two indices produces a perfect square. Once a valid subset is determined, we sum the corresponding elements and want the largest possible sum.

The input `nums` is an array of integers with length up to $10^4$ and values up to $10^9$. The constraints indicate that a brute-force check of all subsets is computationally infeasible. Since indices must be considered for perfect-square products, the structure of indices is as important as the values themselves.

Important edge cases include arrays of length 1 (only one element can be selected), arrays where all numbers are equal, or arrays where only certain combinations of indices can form perfect-square products.

## Approaches

A naive brute-force solution would try all subsets of indices, check if every pair in the subset forms a perfect square, and compute the sum. While correct in theory, the time complexity is $O(2^n \cdot n^2)$, which is infeasible for $n = 10^4$.

The key insight comes from observing the property of **indices forming perfect-square products**. If $i \cdot j$ is a perfect square, then $i$ and $j$ can be written as $i = a \cdot x^2$ and $j = a \cdot y^2$ for some integer $a$ and integers $x$ and $y$. Essentially, indices in the valid subset must have the **same square-free part**. By grouping indices by their square-free parts, we only need to sum elements corresponding to indices in each group and pick the group with the maximum sum. This reduces the problem to a grouping and aggregation problem.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n * n^2) | O(n) | Check every subset, verify perfect-square condition for all pairs |
| Optimal | O(n * sqrt(n)) | O(n) | Group indices by square-free part and sum values within groups |

## Algorithm Walkthrough

1. Define a helper function to compute the **square-free part** of a number. This is done by dividing out squares of prime numbers from the number, leaving only the product of primes appearing to an odd power.
2. Initialize a dictionary (hash map) that maps each square-free part to the cumulative sum of elements whose indices share that square-free part.
3. Iterate through all 1-indexed positions in `nums`. For each index, compute its square-free part.
4. Add the corresponding element value to the sum stored in the dictionary for that square-free part.
5. After processing all indices, the maximum sum among all groups is the result.

**Why it works:** By grouping indices with the same square-free part, we guarantee that the product of any two indices in the group is a perfect square. Therefore, summing the values in a group gives a valid complete subset, and taking the maximum sum over all groups gives the optimal answer.

## Python Solution

```python
from typing import List
from collections import defaultdict
import math

class Solution:
    def maximumSum(self, nums: List[int]) -> int:
        def square_free_part(n: int) -> int:
            result = 1
            d = 2
            while d * d <= n:
                count = 0
                while n % d == 0:
                    n //= d
                    count += 1
                if count % 2 == 1:
                    result *= d
                d += 1
            if n > 1:
                result *= n
            return result

        groups = defaultdict(int)
        for i, num in enumerate(nums, 1):
            key = square_free_part(i)
            groups[key] += num

        return max(groups.values())
```

This implementation first computes the square-free part for each 1-indexed position and aggregates the sum of numbers in each group. The final maximum sum is returned. Using a hash map avoids redundant calculations and ensures efficiency.

## Go Solution

```go
package main

import (
	"math"
)

func maximumSum(nums []int) int64 {
	squareFree := func(n int) int {
		result := 1
		for d := 2; d*d <= n; d++ {
			count := 0
			for n%d == 0 {
				n /= d
				count++
			}
			if count%2 == 1 {
				result *= d
			}
		}
		if n > 1 {
			result *= n
		}
		return result
	}

	groups := make(map[int]int64)
	for i, num := range nums {
		key := squareFree(i + 1)
		groups[key] += int64(num)
	}

	var maxSum int64
	for _, v := range groups {
		if v > maxSum {
			maxSum = v
		}
	}

	return maxSum
}
```

The Go version mirrors the Python approach, handling integer overflow carefully by using `int64` for sums. Map keys are the square-free parts, and aggregation is performed in the loop.

## Worked Examples

For `nums = [8,7,3,5,7,2,4,9]`, indices 2 and 8 have a perfect-square product since `2*8=16`. Their sum is `7+9=16`, which is the maximum. The grouping by square-free part produces `{1:8,2:16,3:3,...}` and the max sum is 16.

For `nums = [8,10,3,8,1,13,7,9,4]`, indices 1, 4, and 9 form a valid subset since their square-free parts match. The sum `8+8+4=20` is maximum.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n * sqrt(n)) | Each index requires factorization up to sqrt(n) |
| Space | O(n) | Hash map stores sums for at most n different square-free parts |

The time complexity is driven by computing square-free parts, and space is proportional to the number of distinct groups.

## Edge Cases

One important edge case is when the array contains only one element. The algorithm correctly handles it because the single index forms a valid subset by itself. Another tricky case is when multiple indices share the same square-free part but some have zeros; summing them still works because zero does not affect the grouping logic. Finally, indices that are perfect squares themselves need correct factorization, and the algorithm ensures their square-free part is computed accurately. All these cases are handled efficiently by the grouping mechanism.
