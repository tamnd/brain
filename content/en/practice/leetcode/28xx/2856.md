# Minimum Array Length After Pair Removals

## Problem Understanding

The problem gives us a sorted integer array `nums` in non-decreasing order. We are allowed to repeatedly remove pairs of elements, but only when the first element is strictly smaller than the second element.

In other words, for any operation we must choose two indices `i` and `j` such that:

```
nums[i] < nums[j]
```

Then both elements are removed from the array.

The goal is to minimize the number of elements remaining after performing as many valid removals as possible.

Because the array is already sorted, equal values appear together in contiguous blocks. This ordering is extremely important because it allows us to reason about pairing smaller values with larger values efficiently.

The constraints are large:

- `1 <= nums.length <= 100000`
- `1 <= nums[i] <= 10^9`

This tells us that any solution worse than roughly `O(n log n)` may be too slow, and quadratic solutions are definitely infeasible for the upper limit.

A crucial observation is that two equal numbers cannot be removed together because the condition requires strict inequality. This means that if one value appears too many times, some copies may remain unmatched forever.

Some important edge cases include:

- Arrays where all elements are equal, because no removals are possible.
- Arrays with completely distinct values, because every element can potentially be paired.
- Arrays where one value appears more than half the time, because not all copies can be matched with larger values.
- Very small arrays such as length `1` or `2`.

The problem guarantees that the array is already sorted, so we do not need to spend time sorting it ourselves.

## Approaches

### Brute Force Approach

A brute-force strategy would repeatedly search for any valid pair `(i, j)` where `nums[i] < nums[j]`, remove them, and continue until no more valid pairs exist.

One possible implementation could:

1. Try every possible pair.
2. Remove one valid pair.
3. Rebuild the array.
4. Repeat until no valid pair remains.

This approach eventually produces the correct answer because it simulates the operations directly.

However, it is far too slow. Each removal may require scanning the array to find a valid pair, and removing elements from arrays is also expensive. In the worst case, this can become `O(n^2)` or even `O(n^3)` depending on implementation details.

With `n = 100000`, this is not practical.

### Optimal Greedy Two Pointer Approach

The key insight is that the array is sorted.

To maximize removals, we want to pair smaller numbers with larger numbers as efficiently as possible.

A very effective strategy is:

- Split the array into two halves.
- Use one pointer in the left half and another pointer in the right half.
- Try to match each smaller value with the smallest possible larger value.

Why does this work?

Suppose the array length is `n`.

At most, we can remove `2 * floor(n/2)` elements because each operation removes exactly two elements.

The left half contains candidate smaller values, and the right half contains candidate larger values.

If:

```
nums[left] < nums[right]
```

then we can form a valid pair and advance both pointers.

Otherwise, the right value is not large enough, so we move only the right pointer forward.

This greedy strategy always preserves the best future pairing opportunities.

## Algorithm Walkthrough

### Step 1: Determine the array length

Let:

```
n = len(nums)
```

We will attempt to maximize the number of valid pairs.

### Step 2: Split the array conceptually into two halves

We place:

- `left` pointer at index `0`
- `right` pointer at index `(n + 1) // 2`

The left pointer represents smaller elements that need partners.

The right pointer represents candidate larger elements.

We start the right pointer in the second half because every valid pair must use two distinct positions, and we want enough room to form maximum pairings.

### Step 3: Try to greedily form pairs

While both pointers remain inside the array:

- If `nums[left] < nums[right]`, we found a valid pair.

- Increase the count of matched pairs.
- Move both pointers forward.
- Otherwise:

- The current right value is not large enough.
- Move only the right pointer forward to search for a larger value.

### Step 4: Compute remaining length

If we formed `pairs` valid pairs, then we removed:

```
2 * pairs
```

elements.

So the remaining length is:

```
n - 2 * pairs
```

### Why it works

The greedy strategy always matches the current smallest unmatched element with the smallest possible valid larger element.

This is optimal because using a larger-than-necessary partner early could prevent future matches.

Since the array is sorted, once `nums[right]` becomes large enough for `nums[left]`, pairing them immediately is always safe and never reduces future possibilities.

Therefore, the algorithm maximizes the number of removals.

## Python Solution

```python
from typing import List

class Solution:
    def minLengthAfterRemovals(self, nums: List[int]) -> int:
        n = len(nums)

        left = 0
        right = (n + 1) // 2

        pairs = 0

        while left < n // 2 and right < n:
            if nums[left] < nums[right]:
                pairs += 1
                left += 1
                right += 1
            else:
                right += 1

        return n - 2 * pairs
```

The implementation starts by computing the array length and initializing two pointers.

The `left` pointer begins at the start of the array and tracks the current smaller element that still needs a partner.

The `right` pointer begins in the second half of the array. This ensures we never reuse positions and always search among larger-indexed elements.

The loop continues while both pointers remain valid.

When a valid pair is found, both pointers move forward because both elements are now used.

If the current right value is not strictly larger, only the right pointer advances because we still need to find a valid partner for the current left value.

Finally, the algorithm subtracts twice the number of formed pairs from the original length to obtain the minimum remaining array size.

## Go Solution

```go
func minLengthAfterRemovals(nums []int) int {
	n := len(nums)

	left := 0
	right := (n + 1) / 2

	pairs := 0

	for left < n/2 && right < n {
		if nums[left] < nums[right] {
			pairs++
			left++
			right++
		} else {
			right++
		}
	}

	return n - 2*pairs
}
```

The Go implementation follows exactly the same logic as the Python version.

Slices in Go already behave similarly to dynamic arrays, so no extra data structures are needed.

There are no integer overflow concerns because the values involved are well within Go's `int` range for the given constraints.

The solution uses only constant extra memory.

## Edge Cases

### All Elements Are Equal

Consider:

```
[5,5,5,5]
```

No two elements satisfy the strict inequality requirement because every pair is equal.

A naive implementation might accidentally allow equal values to pair if it uses `<=` instead of `<`.

This implementation correctly checks:

```
nums[left] < nums[right]
```

so no invalid removals occur.

### Odd Length Arrays

Consider:

```
[1,1,2,3,4]
```

Even with optimal pairing, one element must remain because the array length is odd.

The algorithm naturally handles this because each operation removes exactly two elements, and the final formula:

```
n - 2 * pairs
```

correctly preserves one leftover element.

### One Value Appears Too Frequently

Consider:

```
[1,1,1,1,2,3]
```

There are four copies of `1` but only two larger elements available.

Only two copies of `1` can be removed.

A buggy greedy strategy might incorrectly assume all smaller elements can eventually be paired.

This implementation avoids that issue because the right pointer only advances when a strictly larger value exists, ensuring every pair is valid and unique.
