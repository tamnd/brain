# LeetCode 2812 - Find the Safest Path in a Grid

## Problem Understanding

The problem gives us an `n x n` grid where each cell is either empty (`0`) or contains a thief (`1`). We start at the top-left corner `(0, 0)` and want to reach the bottom-right corner `(n - 1, n - 1)` by moving up, down, left, or right.

The important part is the definition of the safeness factor.

For every cell on a path, we compute its Manhattan distance to the nearest thief. The safeness factor of the entire path is the minimum of those distances across all cells in the path.

We are not trying to minimize path length. Instead, we want a path whose weakest point is as safe as possible.

In other words:

- Every cell has a "safety score", equal to its distance from the nearest thief.
- A path's quality is determined by the minimum safety score along that path.
- We must maximize that minimum value.

This is a classic "maximize the minimum" problem.

The constraints are important:

- `n` can be as large as `400`
- The grid can therefore contain up to `160,000` cells

Any algorithm worse than roughly `O(n^2 log n)` will likely time out.

A few edge cases immediately stand out:

- If the starting cell or ending cell contains a thief, the answer is automatically `0`, because the path must include that cell.
- There may be many thieves, so repeatedly computing nearest-thief distances naively would be far too slow.
- The optimal path may be longer than the shortest path, because safety matters more than distance.
- The grid always contains at least one thief, so every cell has a finite distance to some thief.

## Approaches

### Brute Force Approach

A brute force solution would attempt to enumerate all possible paths from `(0, 0)` to `(n - 1, n - 1)`.

For each path:

1. Compute the minimum distance from each path cell to any thief
2. Take the minimum value across the path
3. Keep the maximum among all paths

This is completely infeasible.

The number of possible paths grows exponentially with the grid size. Even for relatively small grids, the number of valid paths becomes enormous.

A slightly less terrible brute force improvement would precompute the distance from every cell to the nearest thief, then run DFS over all paths while tracking the minimum safety value. However, path enumeration is still exponential.

### Key Insight

The key observation is that this problem can be transformed into a graph problem.

First, compute for every cell:

- `dist[r][c] = distance to nearest thief`

This can be done efficiently using multi-source BFS:

- Start BFS simultaneously from all thief cells
- Expand outward layer by layer
- The first time a cell is reached gives its nearest-thief distance

Now each cell has a safety value.

The problem becomes:

> Find a path from start to end that maximizes the minimum cell value along the path.

This is extremely similar to:

- Maximum bottleneck path
- Widest path problem

We can solve it using a priority queue variant of Dijkstra's algorithm.

Instead of minimizing path cost, we maximize the minimum safety encountered so far.

At each step:

- The path value becomes:

`min(current_path_safety, neighbor_safety)`
- We always process the currently safest path first

This guarantees that when we first reach the destination, we have found the optimal answer.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Enumerates all possible paths |
| Optimal | O(n² log n) | O(n²) | Multi-source BFS + priority queue |

## Algorithm Walkthrough

### Step 1: Compute distance to nearest thief

We create a matrix `dist` of size `n x n`.

Every thief cell starts with distance `0`.

We push all thief cells into a queue simultaneously and run BFS.

This works because BFS explores in increasing distance order. Since all thieves are inserted initially, every cell gets assigned the shortest distance to any thief.

### Step 2: Prepare the priority queue

We now treat the grid as a weighted graph.

The value of a cell is its safeness score from `dist`.

We use a max-heap where each entry stores:

- current path safeness
- row
- column

The path safeness means:

> the minimum safeness value seen along the path so far

We start with:

- position `(0, 0)`
- safeness `dist[0][0]`

### Step 3: Always expand the safest path first

At each step:

1. Pop the state with the highest current safeness
2. If this is the destination, return its safeness
3. Explore all 4 neighbors
4. Compute:

```
new_safeness = min(current_safeness, dist[nr][nc])
```

This reflects the weakest cell along the extended path.

### Step 4: Avoid revisiting worse states

We maintain a visited matrix.

Once a cell is processed from the heap, we do not need to revisit it because the heap guarantees we reached it with the best possible safeness value.

### Step 5: Return the answer

The first time we pop the destination cell from the heap, we have found the maximum safeness factor.

## Why it works

The algorithm works because the priority queue always processes the currently best available path first.

This is analogous to Dijkstra's algorithm, except:

- Dijkstra minimizes total distance
- Here we maximize the minimum edge or node quality

Once a cell is removed from the heap, no future path can reach it with a better minimum safeness value. Therefore, when the destination is popped, its value is optimal.

## Python Solution

```python
from typing import List
from collections import deque
import heapq

class Solution:
    def maximumSafenessFactor(self, grid: List[List[int]]) -> int:
        n = len(grid)

        # Step 1: Multi-source BFS from all thieves
        dist = [[-1] * n for _ in range(n)]
        queue = deque()

        for r in range(n):
            for c in range(n):
                if grid[r][c] == 1:
                    dist[r][c] = 0
                    queue.append((r, c))

        directions = [(1, 0), (-1, 0), (0, 1), (0, -1)]

        while queue:
            r, c = queue.popleft()

            for dr, dc in directions:
                nr, nc = r + dr, c + dc

                if 0 <= nr < n and 0 <= nc < n and dist[nr][nc] == -1:
                    dist[nr][nc] = dist[r][c] + 1
                    queue.append((nr, nc))

        # Step 2: Maximum bottleneck path using max-heap
        max_heap = [(-dist[0][0], 0, 0)]
        visited = [[False] * n for _ in range(n)]

        while max_heap:
            neg_safe, r, c = heapq.heappop(max_heap)
            safe = -neg_safe

            if visited[r][c]:
                continue

            visited[r][c] = True

            if r == n - 1 and c == n - 1:
                return safe

            for dr, dc in directions:
                nr, nc = r + dr, c + dc

                if 0 <= nr < n and 0 <= nc < n and not visited[nr][nc]:
                    new_safe = min(safe, dist[nr][nc])
                    heapq.heappush(max_heap, (-new_safe, nr, nc))

        return 0
```

The implementation has two major phases.

The first phase computes the nearest-thief distance for every cell. Instead of running BFS from every empty cell individually, which would be extremely slow, we run one multi-source BFS beginning from all thief cells simultaneously. This guarantees optimal distances in linear time.

The second phase searches for the safest path. The heap always prioritizes the path with the highest current safeness factor. Whenever we move to a neighbor, the new path safeness becomes the minimum of the current path value and the neighbor's cell safety.

Because we always process the best available state first, the first time we reach the destination gives the optimal answer.

## Go Solution

```go
package main

import (
	"container/heap"
)

type State struct {
	safe int
	r    int
	c    int
}

type MaxHeap []State

func (h MaxHeap) Len() int {
	return len(h)
}

func (h MaxHeap) Less(i, j int) bool {
	return h[i].safe > h[j].safe
}

func (h MaxHeap) Swap(i, j int) {
	h[i], h[j] = h[j], h[i]
}

func (h *MaxHeap) Push(x interface{}) {
	*h = append(*h, x.(State))
}

func (h *MaxHeap) Pop() interface{} {
	old := *h
	n := len(old)
	item := old[n-1]
	*h = old[:n-1]
	return item
}

func maximumSafenessFactor(grid [][]int) int {
	n := len(grid)

	dist := make([][]int, n)
	for i := range dist {
		dist[i] = make([]int, n)
		for j := range dist[i] {
			dist[i][j] = -1
		}
	}

	type Cell struct {
		r int
		c int
	}

	queue := []Cell{}

	for r := 0; r < n; r++ {
		for c := 0; c < n; c++ {
			if grid[r][c] == 1 {
				dist[r][c] = 0
				queue = append(queue, Cell{r, c})
			}
		}
	}

	directions := [][2]int{
		{1, 0},
		{-1, 0},
		{0, 1},
		{0, -1},
	}

	// Multi-source BFS
	head := 0

	for head < len(queue) {
		cell := queue[head]
		head++

		for _, d := range directions {
			nr := cell.r + d[0]
			nc := cell.c + d[1]

			if nr >= 0 && nr < n && nc >= 0 && nc < n && dist[nr][nc] == -1 {
				dist[nr][nc] = dist[cell.r][cell.c] + 1
				queue = append(queue, Cell{nr, nc})
			}
		}
	}

	// Max-heap search
	pq := &MaxHeap{}
	heap.Init(pq)

	heap.Push(pq, State{
		safe: dist[0][0],
		r:    0,
		c:    0,
	})

	visited := make([][]bool, n)
	for i := range visited {
		visited[i] = make([]bool, n)
	}

	for pq.Len() > 0 {
		cur := heap.Pop(pq).(State)

		r := cur.r
		c := cur.c
		safe := cur.safe

		if visited[r][c] {
			continue
		}

		visited[r][c] = true

		if r == n-1 && c == n-1 {
			return safe
		}

		for _, d := range directions {
			nr := r + d[0]
			nc := c + d[1]

			if nr >= 0 && nr < n && nc >= 0 && nc < n && !visited[nr][nc] {
				newSafe := safe
				if dist[nr][nc] < newSafe {
					newSafe = dist[nr][nc]
				}

				heap.Push(pq, State{
					safe: newSafe,
					r:    nr,
					c:    nc,
				})
			}
		}
	}

	return 0
}
```

The Go version follows the same algorithmic structure as the Python implementation.

The main difference is heap management. Go's `container/heap` package requires implementing the heap interface manually. We define a custom `MaxHeap` type and reverse the comparison in `Less()` so that larger safeness values are processed first.

The BFS queue is implemented using a slice with a moving head index, which avoids the overhead of repeatedly removing from the front of the slice.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n² log n) | BFS is O(n²), heap operations add logarithmic factor |
| Space | O(n²) | Distance matrix, visited matrix, queue, and heap |

The BFS phase visits every cell exactly once, so it costs `O(n²)`.

The heap phase may push each cell multiple times, but each heap operation costs `O(log(n²)) = O(log n)`. Since there are `n²` cells, the total complexity becomes `O(n² log n)`.

The auxiliary data structures all store at most `n²` elements, so the space complexity is `O(n²)`.

## Edge Cases

One important edge case occurs when the starting cell or destination cell contains a thief. Since the path must include those cells, the minimum safeness immediately becomes zero. A buggy implementation might try to avoid the thief conceptually, but the correct answer must still be `0`. The algorithm handles this naturally because the BFS assigns distance `0` to thief cells, and the heap starts with that value.

Another important edge case is a grid where thieves form a barrier across the board. Even if some cells have large distances from thieves, every valid path may eventually be forced through a dangerous region. A greedy shortest-path algorithm would fail here because path length is irrelevant. The priority-queue bottleneck approach correctly optimizes the minimum safeness value along the entire route.

A third important case is a very large grid with many thieves. Running BFS separately from every cell would be far too slow. The multi-source BFS is critical because it computes all nearest-thief distances simultaneously in linear time. This ensures the solution remains efficient even at the maximum constraint size of `400 x 400`.
