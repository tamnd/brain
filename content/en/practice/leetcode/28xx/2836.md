**Difficulty:** 🔴 Hard  
**Topics:** Array, Dynamic Programming, Bit Manipulation

## Solution

## Problem Understanding

This problem describes a deterministic ball-passing game between `n` players. Each player has exactly one outgoing connection defined by the array `receiver`, where `receiver[i]` tells us which player receives the ball after player `i` passes it.

If we start from player `i`, then the sequence of ball holders is:

```
i
receiver[i]
receiver[receiver[i]]
...
```

The game performs exactly `k` passes. Since the starting player also counts as touching the ball, the total number of touched players in the score is `k + 1`.

The score for a starting player `i` is:

```
i + receiver[i] + receiver(receiver[i]) + ...
```

including repeated visits to the same player if the graph contains cycles.

Our task is to compute the maximum possible score over all possible starting players.

The constraints are extremely important:

- `n <= 10^5`
- `k <= 10^10`

A direct simulation of all `k` passes for every starting node would require:

```
O(n * k)
```

which is completely impossible when `k` is as large as ten billion.

The structure of the graph also matters. Since every node has exactly one outgoing edge, the graph is a functional graph. Every connected component eventually enters a cycle. This property allows efficient repeated jumping techniques.

Several edge cases are important:

- `k` is extremely large, so repeated simulation is impossible.
- A node may point to itself.
- Multiple nodes may point to the same receiver.
- The graph may consist entirely of cycles.
- The optimal path may revisit nodes many times.
- Scores can become very large, so 64-bit integers are required.

The key challenge is efficiently computing both:

- where we end up after many jumps
- the sum accumulated along those jumps

for every starting node.

## Approaches

### Brute Force Approach

The most straightforward solution is to try every starting player independently.

For each player `i`:

1. Initialize the score with `i`.
2. Repeat `k` times:

- move to `receiver[current]`
- add that node index to the score
3. Track the maximum score across all starting nodes.

This approach is correct because it directly simulates the exact process described in the problem statement.

However, the time complexity is disastrous. For every one of the `n` nodes, we may perform up to `k` transitions.

Since:

- `n` can be `10^5`
- `k` can be `10^10`

the runtime becomes astronomically large.

### Optimal Approach, Binary Lifting

The key observation is that repeated transitions in a functional graph can be accelerated using binary lifting.

Binary lifting is commonly used for ancestor queries in trees, but the same principle works here because every node deterministically transitions to exactly one next node.

Instead of simulating one pass at a time, we precompute:

- where we land after `2^j` jumps
- the sum collected during those `2^j` jumps

for every node.

Then we decompose `k` into binary form.

For example:

```
k = 13 = 8 + 4 + 1
```

Instead of performing 13 individual transitions, we combine:

- one precomputed jump of length 8
- one jump of length 4
- one jump of length 1

This reduces the complexity dramatically from linear in `k` to logarithmic in `k`.

The additional challenge is that we need both:

- destination nodes
- cumulative sums

So each binary lifting table stores two values:

- next node
- score accumulated along that jump length

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × k) | O(1) | Simulates every pass directly |
| Optimal | O(n × log k) | O(n × log k) | Uses binary lifting for fast jumps and score accumulation |

## Algorithm Walkthrough

### Step 1: Determine Maximum Bit Length

We need enough binary lifting levels to represent `k`.

If:

```
k <= 10^10
```

then approximately 35 bits are sufficient.

We compute:

```
max_bits = k.bit_length()
```

This tells us how many powers of two we must precompute.

### Step 2: Build the First Binary Lifting Layer

We create two tables:

```
next_node[j][i]
jump_sum[j][i]
```

where:

- `next_node[j][i]` is the node reached after `2^j` jumps from `i`
- `jump_sum[j][i]` is the sum of node indices collected during those jumps

For `j = 0`:

- one jump means moving directly to `receiver[i]`

So:

```
next_node[0][i] = receiver[i]
jump_sum[0][i] = receiver[i]
```

Notice carefully:

- the starting node itself is not included here
- it will be added separately during evaluation

This design simplifies transitions.

### Step 3: Build Higher Binary Lifting Levels

Suppose we already know:

- results for `2^(j-1)` jumps

Then:

- `2^j` jumps can be constructed by combining two consecutive `2^(j-1)` jumps.

So:

```
mid = next_node[j - 1][i]

next_node[j][i] =
    next_node[j - 1][mid]

jump_sum[j][i] =
    jump_sum[j - 1][i]
    + jump_sum[j - 1][mid]
```

This dynamic programming recurrence builds all powers of two efficiently.

### Step 4: Evaluate Every Starting Node

For each starting node:

1. Initialize:

- current node
- score
2. Decompose `k` into binary form.
3. For every set bit:

- add the corresponding precomputed sum
- move to the corresponding destination node

The initial score starts with the starting node itself:

```
score = start
```

because the starting player touches the ball before any pass occurs.

### Step 5: Track the Maximum

After evaluating all starting nodes, return the largest score found.

## Why it works

Binary lifting works because any integer can be represented as a sum of powers of two.

The preprocessing step correctly computes:

- the destination after every power-of-two jump length
- the exact score accumulated during those jumps

When evaluating a starting node, combining the selected binary segments reconstructs exactly the same sequence of passes as direct simulation.

Since every jump contribution is included exactly once, the computed score is correct.

Because we evaluate every possible starting node, the global maximum is guaranteed to be found.

## Python Solution

```python
from typing import List

class Solution:
    def getMaxFunctionValue(self, receiver: List[int], k: int) -> int:
        n = len(receiver)
        max_bits = k.bit_length()

        next_node = [[0] * n for _ in range(max_bits)]
        jump_sum = [[0] * n for _ in range(max_bits)]

        # Base case: 2^0 = 1 jump
        for i in range(n):
            next_node[0][i] = receiver[i]
            jump_sum[0][i] = receiver[i]

        # Build binary lifting tables
        for bit in range(1, max_bits):
            for node in range(n):
                mid = next_node[bit - 1][node]

                next_node[bit][node] = next_node[bit - 1][mid]

                jump_sum[bit][node] = (
                    jump_sum[bit - 1][node]
                    + jump_sum[bit - 1][mid]
                )

        answer = 0

        # Try every starting node
        for start in range(n):
            current = start
            total = start

            for bit in range(max_bits):
                if (k >> bit) & 1:
                    total += jump_sum[bit][current]
                    current = next_node[bit][current]

            answer = max(answer, total)

        return answer
```

The implementation begins by determining how many binary lifting levels are required. Since each level represents powers of two, the number of levels is simply the number of bits needed to represent `k`.

Two DP tables are constructed:

- `next_node`
- `jump_sum`

The first table tracks destinations after power-of-two jumps. The second tracks the accumulated score contributions.

The base layer corresponds to exactly one jump. From there, higher layers are built using dynamic programming. Each larger jump is composed of two smaller jumps of equal size.

Finally, the algorithm evaluates every possible starting node. By iterating through the bits of `k`, it efficiently reconstructs the full sequence of passes using only logarithmic work.

The result is the maximum score observed across all starting nodes.

## Go Solution

```go
func getMaxFunctionValue(receiver []int, k int64) int64 {
	n := len(receiver)

	maxBits := 0
	temp := k
	for temp > 0 {
		maxBits++
		temp >>= 1
	}

	nextNode := make([][]int, maxBits)
	jumpSum := make([][]int64, maxBits)

	for i := 0; i < maxBits; i++ {
		nextNode[i] = make([]int, n)
		jumpSum[i] = make([]int64, n)
	}

	// Base case
	for i := 0; i < n; i++ {
		nextNode[0][i] = receiver[i]
		jumpSum[0][i] = int64(receiver[i])
	}

	// Build binary lifting tables
	for bit := 1; bit < maxBits; bit++ {
		for node := 0; node < n; node++ {
			mid := nextNode[bit-1][node]

			nextNode[bit][node] = nextNode[bit-1][mid]

			jumpSum[bit][node] =
				jumpSum[bit-1][node] +
					jumpSum[bit-1][mid]
		}
	}

	var answer int64 = 0

	// Evaluate every starting node
	for start := 0; start < n; start++ {
		current := start
		total := int64(start)

		for bit := 0; bit < maxBits; bit++ {
			if ((k >> bit) & 1) == 1 {
				total += jumpSum[bit][current]
				current = nextNode[bit][current]
			}
		}

		if total > answer {
			answer = total
		}
	}

	return answer
}
```

The Go implementation follows the same algorithmic structure as the Python version, but several language-specific details matter.

Go requires explicit use of `int64` because the score can become much larger than a 32-bit integer. The variable `k` is also already provided as `int64`, so all score computations should remain in 64-bit arithmetic.

Slices are used instead of Python lists, but the binary lifting tables are structurally identical.

Unlike Python's `bit_length()` helper, Go computes the number of bits manually using repeated right shifts.

The preprocessing phase constructs approximately `log2(k)` layers, each containing `n` entries.

The evaluation phase also processes each node across all bit positions.

Since `log2(10^10)` is only about 34, the algorithm is extremely efficient for the given constraints.

## Edge Cases

### Self-Loops

A node may point to itself:

```
receiver[i] = i
```

This can easily cause bugs if the implementation incorrectly assumes movement always changes the current node.

The binary lifting tables handle this naturally because repeated jumps simply continue returning the same node while accumulating the same value repeatedly.

### Extremely Large k

The value of `k` can reach `10^10`, making direct simulation impossible.

A naive implementation would timeout immediately. Binary lifting avoids this by processing only the set bits of `k`, reducing the number of operations per node to approximately 34.

### Multiple Nodes Sharing the Same Receiver

Several nodes may point to the same destination:

```
receiver = [2,2,2]
```

This means the graph is not a tree and contains merged paths.

The algorithm works correctly because binary lifting depends only on deterministic transitions, not on uniqueness of incoming edges.

### Chains Entering Cycles

Functional graphs often contain long chains that eventually enter cycles.

For example:

```
0 -> 1 -> 2 -> 3 -> 3
```

A buggy implementation might incorrectly stop after detecting a cycle.

This solution never relies on cycle detection. It simply computes exact jump transitions mathematically, so both chain traversal and repeated cycle traversal are handled uniformly and correctly.
