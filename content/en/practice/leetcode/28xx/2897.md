# LeetCode 2897 - Apply Operations on Array to Maximize Sum of Squares

## Problem Understanding

The problem gives us an integer array `nums` and an integer `k`. We are allowed to repeatedly perform a special operation on any two distinct elements:

- One element becomes the bitwise AND of the two values.
- The other element becomes the bitwise OR of the two values.

After performing any number of operations, we must choose exactly `k` elements from the resulting array and maximize the sum of their squares.

The important part of the problem is understanding what the operation actually does to the binary representation of the numbers.

Suppose we choose two numbers:

- `a`
- `b`

After the operation:

- One becomes `a & b`
- The other becomes `a | b`

For every individual bit position:

- If both bits are `0`, the total number of `1`s stays `0`
- If both bits are `1`, the total number of `1`s stays `2`
- If one bit is `1` and the other is `0`, the AND gets `0` and the OR gets `1`, so the total number of `1`s still stays `1`

This means the operation never changes how many times each bit appears across the entire array. It only redistributes those bits among numbers.

The goal therefore becomes:

- Redistribute the available set bits so that the chosen `k` numbers produce the maximum possible sum of squares.

The constraints are large:

- `nums.length` can be up to `10^5`
- `nums[i]` can be up to `10^9`

A brute-force simulation of operations is completely infeasible because the number of reachable states is enormous.

Important observations from the constraints:

- Since `nums[i] <= 10^9`, we only need to consider about 30 bits.
- We need an algorithm close to linear time.
- The problem is fundamentally about bit distribution, not explicit operation simulation.

Important edge cases include:

- Arrays where all numbers are identical
- Arrays with only one element
- Cases where `k = n`
- Cases where all bits are concentrated in a few numbers
- Cases where every number contributes only a single bit

The problem guarantees:

- `k` is always valid
- All numbers are positive
- The array is non-empty

## Approaches

### Brute Force Approach

A naive approach would attempt to simulate operations between pairs of indices and explore all possible resulting arrays.

At each step:

1. Choose two indices
2. Apply the AND/OR transformation
3. Recurse into future states
4. Compute the best possible sum of squares

This eventually explores the state graph of all reachable arrays.

The approach is theoretically correct because it examines every possible configuration reachable through operations. However, it is completely impractical.

Even for small arrays, the number of possible operation sequences explodes exponentially. Since each operation can be repeated indefinitely and there are `O(n^2)` pair choices at every step, the search space becomes astronomically large.

This approach cannot work for `n = 10^5`.

### Optimal Greedy Bit Construction

The key insight is that the operation preserves the total count of set bits at every bit position.

Instead of thinking about operations directly, we can think about redistribution.

For every bit position:

- Count how many numbers contain that bit.
- Those set bits can be reassigned arbitrarily among the final numbers.

To maximize the sum of squares, we want numbers to be as large as possible.

This follows from the convex nature of squaring:

$$(a+b)^2 > a^2 + b^2$$

for positive `a` and `b`.

Therefore, concentrating bits into fewer numbers produces a larger sum of squares.

The optimal strategy is:

1. Count how many times each bit appears.
2. Construct numbers greedily:

- For each new number, take every available bit once if possible.
3. Square each constructed number and add it to the answer.

This creates the largest possible numbers first, which maximizes the total squared contribution.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Explores all reachable states |
| Optimal | O(n × 30 + k × 30) | O(30) | Redistributes bits greedily |

## Algorithm Walkthrough

### Step 1: Count set bits at every position

Create an array `bit_count[30]`.

For every number in `nums`:

- Check each bit from `0` to `29`
- If the bit is set, increment its count

This tells us how many copies of each bit exist globally.

### Step 2: Construct numbers greedily

We will build exactly `k` numbers.

For each constructed number:

1. Start with value `0`
2. Iterate through all bit positions
3. If a bit is still available:

- Set that bit in the current number
- Decrease its remaining count

This greedily packs as many bits as possible into earlier numbers.

### Step 3: Add square contribution

After building a number:

- Compute `value * value`
- Add it to the answer modulo `10^9 + 7`

### Step 4: Return the final result

After constructing `k` numbers, return the accumulated answer.

### Why it works

The operation preserves the total number of set bits at every position, so any valid final arrangement must use exactly the same bit counts.

Because squaring rewards concentration of value, the optimal arrangement is to make numbers as large as possible. Greedily assigning every available bit into the current number maximizes that number before moving to the next one.

This produces the maximum possible sum of squares.

## Python Solution

```python
from typing import List

class Solution:
    def maxSum(self, nums: List[int], k: int) -> int:
        MOD = 10**9 + 7
        
        bit_count = [0] * 30
        
        # Count occurrences of each bit
        for num in nums:
            for bit in range(30):
                if num & (1 << bit):
                    bit_count[bit] += 1
        
        answer = 0
        
        # Construct k numbers greedily
        for _ in range(k):
            current = 0
            
            for bit in range(30):
                if bit_count[bit] > 0:
                    current |= (1 << bit)
                    bit_count[bit] -= 1
            
            answer = (answer + current * current) % MOD
        
        return answer
```

The implementation begins by counting how many times every bit appears across the input array. Since numbers are bounded by `10^9`, only 30 bit positions are necessary.

The second phase constructs numbers greedily. For each of the `k` numbers:

- We try to include every available bit once.
- If a bit still has remaining occurrences, we place it into the current number and decrement the count.

This process ensures earlier numbers become as large as possible.

Finally, each constructed number contributes its square to the result, computed modulo `10^9 + 7`.

## Go Solution

```go
package main

func maxSum(nums []int, k int) int {
	const MOD int = 1_000_000_007

	bitCount := make([]int, 30)

	// Count occurrences of each bit
	for _, num := range nums {
		for bit := 0; bit < 30; bit++ {
			if (num & (1 << bit)) != 0 {
				bitCount[bit]++
			}
		}
	}

	answer := 0

	// Construct k numbers greedily
	for i := 0; i < k; i++ {
		current := 0

		for bit := 0; bit < 30; bit++ {
			if bitCount[bit] > 0 {
				current |= (1 << bit)
				bitCount[bit]--
			}
		}

		answer = (answer + int((int64(current) * int64(current))%MOD)) % MOD
	}

	return answer
}
```

The Go implementation follows the same logic as the Python version.

One important difference is integer overflow handling. Since squaring can exceed 32-bit integer range, the multiplication is performed using `int64` before applying the modulo.

Slices are used instead of fixed arrays for convenience, though either would work because the size is constant at 30.

| --- | --- | --- |

| Space | O(30) | Fixed-size bit counter |

The algorithm processes each number across 30 bit positions, giving `O(n × 30)` complexity. Then it constructs `k` numbers, again iterating through 30 bits each time.

Since 30 is constant, the complexity is effectively linear in `n + k`.

The auxiliary space usage is constant because the bit counter always has size 30.

## Edge Cases

### Single Element Array

When the array contains only one number, no operations can be performed because the operation requires two distinct indices.

A buggy implementation might assume redistribution is always possible. Our solution handles this naturally because the bit counts simply reconstruct the same number.

### Sparse Bits Across Many Numbers

Consider:

```
nums = [1,2,4,8]
```

Every number contributes a unique bit.

The optimal strategy combines all bits into one large number because squaring favors concentration. A naive greedy based only on original numbers would miss this redistribution opportunity.

Our implementation correctly reconstructs:

```
15
```

as the first number.

### Large Values and Overflow

Numbers can approach `10^9`, and squaring them can exceed 32-bit integer limits.

In Go especially, this can cause overflow bugs if multiplication is performed using `int`.

The implementation explicitly converts to `int64` before multiplication to ensure correctness.
