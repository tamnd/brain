# LeetCode 2801 - Count Stepping Numbers in Range

## Problem Understanding

This problem asks us to count how many integers within the inclusive range `[low, high]` are valid stepping numbers.

A stepping number is defined as a number where every pair of adjacent digits differs by exactly `1`. For example:

- `1234543` is valid because every neighboring pair differs by `1`
- `10` is valid because `|1 - 0| = 1`
- `98` is valid because `|9 - 8| = 1`
- `122` is invalid because `|2 - 2| = 0`
- `135` is invalid because `|1 - 3| = 2`

The input values `low` and `high` are provided as strings instead of integers because their size can be extremely large, up to `10^100`. This immediately tells us that normal integer iteration is impossible. We cannot simply loop from `low` to `high`.

The goal is to return the total count of stepping numbers inside the range, modulo `10^9 + 7`.

The constraints are the most important clue:

- `high` may contain up to `100` digits
- The numeric range can therefore be astronomically large
- Brute force enumeration is completely infeasible

This strongly suggests that we need a digit dynamic programming approach, often called digit DP.

Several edge cases are important:

- Single digit numbers are always stepping numbers
- Numbers cannot contain leading zeros
- The range boundaries themselves may be stepping numbers
- `low` and `high` may have different digit lengths
- Very large values require string-based processing rather than numeric conversion

The key challenge is counting valid numbers efficiently without enumerating every value in the range.

## Approaches

### Brute Force Approach

The brute force solution would iterate through every integer from `low` to `high`, check whether each number is a stepping number, and count the valid ones.

To check whether a number is a stepping number:

1. Convert the number to a string
2. Compare every adjacent pair of digits
3. Verify that their absolute difference equals `1`

This approach is correct because it directly verifies every number in the range.

However, it is far too slow. The range size may approach `10^100`, which is impossible to iterate through. Even for much smaller ranges, repeatedly checking every number would still be inefficient.

The brute force method only works for tiny inputs.

### Optimal Approach, Digit DP

The key observation is that we do not need to generate every number individually.

Instead, we can count how many valid stepping numbers can be formed digit by digit while respecting an upper bound.

This is a classic digit DP problem.

The idea is:

- Count how many stepping numbers are less than or equal to `high`
- Count how many stepping numbers are less than `low`
- Subtract the two counts
- Add `1` back if `low` itself is a stepping number

The digit DP constructs numbers from left to right while tracking:

- Current digit position
- Previous digit chosen
- Whether the current prefix is already smaller than the bound
- Whether we have started building the number yet

At each step:

- If the number has not started yet, we may skip digits
- Otherwise, the next digit must differ from the previous digit by exactly `1`

Because the DP only depends on a small state space, it runs efficiently even for 100 digit numbers.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O((high - low) × D) | O(D) | Checks every number individually |
| Optimal | O(D × 10 × 2 × 2) | O(D × 10 × 2 × 2) | Digit DP with memoization |

Here, `D` is the number of digits, at most `100`.

## Algorithm Walkthrough

### Step 1, Define a helper function `count_up_to(limit)`

This function returns the number of stepping numbers in the range `[0, limit]`.

Instead of enumerating all numbers, we use digit DP to count valid constructions.

### Step 2, Use DFS with memoization

We define a recursive function with these parameters:

- `position`, current digit index
- `previous_digit`, last digit chosen
- `tight`, whether the current prefix exactly matches the limit prefix
- `started`, whether we have started forming the number

Memoization avoids recomputing identical states.

### Step 3, Determine the digit range

If `tight` is true, the current digit cannot exceed the corresponding digit in the limit.

Otherwise, digits from `0` to `9` are allowed.

### Step 4, Handle leading zeros

If `started` is false and we choose digit `0`, we remain in the "not started" state.

This allows shorter numbers to be represented naturally.

For example, while processing a 5 digit limit, the number `98` is represented as:

```
00098
```

without treating leading zeros as actual digits.

### Step 5, Enforce the stepping property

Once the number has started:

- The next digit is only valid if

```
abs(current_digit - previous_digit) == 1
```

Otherwise, that transition is skipped.

### Step 6, Count valid endings

When we reach the end of the digits:

- Return `1` if we formed at least one digit
- Return `0` otherwise

This prevents counting the empty number.

### Step 7, Compute the final answer

We compute:

```
count(high) - count(low - 1)
```

Instead of implementing subtraction on huge integers directly, we compute:

```
count(high) - count(low)
```

and then add `1` if `low` itself is a stepping number.

### Step 8, Check whether `low` is a stepping number

We directly validate `low` by checking adjacent digit differences.

If valid, add `1` to the final result.

### Why it works

The DP explores every possible digit sequence that satisfies the stepping property while never exceeding the limit. Because every valid number corresponds to exactly one DP path, and every invalid number is excluded by the transition rules, the algorithm counts precisely the valid stepping numbers.

Memoization guarantees that identical subproblems are solved only once, making the solution efficient even for 100 digit inputs.

## Python Solution

```python
from functools import lru_cache

MOD = 10**9 + 7

class Solution:
    def countSteppingNumbers(self, low: str, high: str) -> int:

        def is_stepping(num: str) -> bool:
            for i in range(1, len(num)):
                if abs(int(num[i]) - int(num[i - 1])) != 1:
                    return False
            return True

        def count_up_to(limit: str) -> int:
            digits = list(map(int, limit))
            n = len(digits)

            @lru_cache(None)
            def dfs(pos: int, prev_digit: int, tight: bool, started: bool) -> int:
                if pos == n:
                    return 1 if started else 0

                upper_bound = digits[pos] if tight else 9
                total = 0

                for digit in range(upper_bound + 1):

                    next_tight = tight and (digit == upper_bound)

                    if not started:
                        if digit == 0:
                            total += dfs(pos + 1, -1, next_tight, False)
                        else:
                            total += dfs(pos + 1, digit, next_tight, True)
                    else:
                        if abs(digit - prev_digit) == 1:
                            total += dfs(pos + 1, digit, next_tight, True)

                    total %= MOD

                return total

            return dfs(0, -1, True, False)

        answer = (
            count_up_to(high)
            - count_up_to(low)
            + (1 if is_stepping(low) else 0)
        ) % MOD

        return answer
```

The implementation begins with `is_stepping`, which directly validates whether a given string represents a stepping number.

The main logic is inside `count_up_to`. This function counts all stepping numbers less than or equal to a limit.

The recursive DFS processes digits from left to right. The `tight` flag ensures we never exceed the limit. The `started` flag distinguishes leading zeros from actual digits.

When the number has not started yet, choosing digit `0` simply skips the current position. Once a nonzero digit is chosen, the stepping constraint becomes active.

Memoization with `lru_cache` is critical because many states repeat during recursion. Without caching, the recursion tree would grow exponentially.

Finally, we use inclusion and exclusion:

```python
count(high) - count(low)
```

and add `1` back if `low` itself is valid.

## Go Solution

```go
package main

const MOD int = 1000000007

func countSteppingNumbers(low string, high string) int {

	isStepping := func(num string) bool {
		for i := 1; i < len(num); i++ {
			diff := int(num[i]) - int(num[i-1])
			if diff < 0 {
				diff = -diff
			}

			if diff != 1 {
				return false
			}
		}
		return true
	}

	countUpTo := func(limit string) int {

		digits := make([]int, len(limit))
		for i := 0; i < len(limit); i++ {
			digits[i] = int(limit[i] - '0')
		}

		type State struct {
			pos        int
			prevDigit  int
			tight      int
			started    int
		}

		memo := map[State]int{}

		var dfs func(pos int, prevDigit int, tight int, started int) int

		dfs = func(pos int, prevDigit int, tight int, started int) int {

			if pos == len(digits) {
				if started == 1 {
					return 1
				}
				return 0
			}

			state := State{pos, prevDigit, tight, started}

			if value, exists := memo[state]; exists {
				return value
			}

			upperBound := 9
			if tight == 1 {
				upperBound = digits[pos]
			}

			total := 0

			for digit := 0; digit <= upperBound; digit++ {

				nextTight := 0
				if tight == 1 && digit == upperBound {
					nextTight = 1
				}

				if started == 0 {

					if digit == 0 {
						total += dfs(pos+1, -1, nextTight, 0)
					} else {
						total += dfs(pos+1, digit, nextTight, 1)
					}

				} else {

					diff := digit - prevDigit
					if diff < 0 {
						diff = -diff
					}

					if diff == 1 {
						total += dfs(pos+1, digit, nextTight, 1)
					}
				}

				total %= MOD
			}

			memo[state] = total
			return total
		}

		return dfs(0, -1, 1, 0)
	}

	answer := countUpTo(high) - countUpTo(low)

	if isStepping(low) {
		answer++
	}

	answer %= MOD

	if answer < 0 {
		answer += MOD
	}

	return answer
}
```

The Go implementation mirrors the Python solution closely.

Since Go does not provide built in memoization decorators, we explicitly store computed states inside a map keyed by a custom `State` struct.

Boolean states are represented as integers because integers are simpler and slightly more efficient for struct map keys.

The recursive DFS logic is otherwise identical.

Care is taken to normalize negative modulo results at the end.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(D × 10 × 2 × 2 × 10) | DP states times digit transitions |
| Space | O(D × 10 × 2 × 2) | Memoization table |

Here, `D` is the number of digits in the input, at most `100`.

The DP state space is small:

- `D` positions
- `10` possible previous digits
- `2` values for tight
- `2` values for started

For each state, we iterate over at most `10` candidate digits.

This makes the algorithm extremely efficient even for 100 digit numbers.

## Edge Cases

### Single Digit Numbers

Every single digit number is automatically a stepping number because there are no adjacent digits to compare.

A buggy implementation might accidentally reject these numbers if it assumes at least one digit comparison must exist.

The implementation handles this naturally because the adjacency check loop starts from index `1`, so single digit numbers immediately return true.

### Leading Zero States in Digit DP

Digit DP commonly uses leading zeros to represent shorter numbers inside longer digit lengths.

For example, while processing a 5 digit limit, the number `98` internally appears as:

```
00098
```

A buggy implementation might incorrectly compare the leading zeros against real digits, causing invalid transitions.

The `started` flag prevents this. The stepping condition is only enforced after the first nonzero digit is placed.

### Large Inputs Beyond Integer Range

The inputs may contain up to 100 digits, far exceeding standard integer types.

A naive solution using integer conversion would overflow immediately.

This implementation processes the numbers as strings throughout the algorithm, making it safe for arbitrarily large values within the problem constraints.
