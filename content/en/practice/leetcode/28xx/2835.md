# LeetCode 2835 - Minimum Operations to Form Subsequence With Target Sum

## Problem Understanding

The problem gives us an array `nums` where every element is a power of two. We are also given a target value `target`. Our goal is to determine the minimum number of split operations needed so that some subsequence of the array sums exactly to `target`.

A split operation works like this:

- Pick a number greater than `1`
- Remove it
- Replace it with two copies of half its value

For example:

- `8 -> 4 + 4`
- `4 -> 2 + 2`
- `2 -> 1 + 1`

Since every number is a power of two, repeatedly splitting preserves this property.

The key observation is that splitting does not change the total sum of the array. It only changes how that sum is distributed among smaller powers of two.

The subsequence condition means we can choose any subset of elements while preserving order, but because only the sum matters, the order is irrelevant. Effectively, we are asking:

> Can we create enough powers of two at each bit position so that their total equals `target`?

The constraints are important:

- `nums.length <= 1000`
- `nums[i] <= 2^30`
- `target < 2^31`

These limits strongly suggest that a bit manipulation solution is intended. There are only at most 31 relevant bit positions, which allows an efficient greedy strategy.

An important edge case is when the total sum of all numbers is smaller than `target`. Since splitting never increases total sum, such cases are immediately impossible.

Another subtle case occurs when the needed bit is unavailable, but a larger bit exists. In that situation, we must split larger powers down until the required bit becomes available. Minimizing operations means we should always split the nearest larger power.

## Approaches

### Brute Force Approach

A brute force solution would simulate every possible sequence of split operations and check whether some subsequence sums to `target`.

At every step, we could:

1. Choose any splittable element
2. Split it
3. Recursively continue exploring

After each configuration, we could run a subset sum check to determine whether `target` can be formed.

This approach is theoretically correct because it explores all possible transformations of the array. Eventually, if a valid configuration exists, brute force will find it.

However, the branching factor is enormous. Every split creates more numbers, which creates even more future choices. The state space grows exponentially and becomes completely infeasible even for modest input sizes.

### Key Insight

The important observation is that powers of two behave independently across bit positions.

Suppose we count how many copies of each power of two we currently have:

- how many `1`s
- how many `2`s
- how many `4`s
- how many `8`s
- etc.

To form `target`, we simply need enough bits at every position.

If a needed bit is missing, we should split the smallest larger power available. This is optimal because:

- splitting a farther larger power requires more operations
- each split reduces the exponent by exactly one

This naturally leads to a greedy bitwise solution.

| --- | --- | --- | --- |

| Optimal | O(31) | O(31) | Greedy bit manipulation using frequency counts |

## Algorithm Walkthrough

### Step 1: Check Total Sum

First compute the total sum of all numbers.

If:

```
sum(nums) < target
```

then it is impossible to form the target, even after unlimited splitting.

Return `-1`.

### Step 2: Count Powers of Two

Create an array `count[32]` where:

```
count[i] = number of occurrences of 2^i
```

For example:

```
nums = [1,2,8]

count[0] = 1
count[1] = 1
count[3] = 1
```

This representation is much easier to work with than the original array.

### Step 3: Process Bits from Low to High

Iterate through bit positions from `0` to `30`.

At each bit:

- determine whether `target` needs this bit
- ensure enough copies exist

If the current bit is required but unavailable, we must borrow from a higher bit.

### Step 4: Split the Nearest Larger Power

Suppose bit `i` is needed but `count[i] == 0`.

Search upward for the nearest `j > i` such that:

```
count[j] > 0
```

Then repeatedly split:

```
2^j -> 2^(j-1) + 2^(j-1)
```

Each split costs one operation.

Continue until we create a `2^i`.

### Step 5: Carry Extra Bits Forward

After satisfying the current bit, any remaining copies can combine upward.

Two copies of `2^i` equal one copy of `2^(i+1)`.

So:

```
count[i+1] += count[i] // 2
```

This simulates unused lower powers contributing to higher powers.

### Step 6: Return Total Operations

The accumulated split count is the minimum number of operations.

## Why it works

The greedy strategy is optimal because every split reduces the exponent by exactly one. If we need a missing bit at position `i`, borrowing from the nearest larger bit minimizes the number of required splits.

Processing bits from low to high guarantees that all lower requirements are permanently satisfied before moving upward. Since powers of two combine perfectly through binary representation, the algorithm exactly mirrors binary addition and decomposition.

## Python Solution

```python
from typing import List

class Solution:
    def minOperations(self, nums: List[int], target: int) -> int:
        if sum(nums) < target:
            return -1

        count = [0] * 32

        for num in nums:
            bit = num.bit_length() - 1
            count[bit] += 1

        operations = 0

        for bit in range(31):
            # If target needs this bit
            if (target >> bit) & 1:
                if count[bit] > 0:
                    count[bit] -= 1
                else:
                    higher = bit + 1

                    while higher < 32 and count[higher] == 0:
                        higher += 1

                    while higher > bit:
                        count[higher] -= 1
                        count[higher - 1] += 2
                        operations += 1
                        higher -= 1

                    count[bit] -= 1

            # Carry extra powers upward
            count[bit + 1] += count[bit] // 2

        return operations
```

The implementation begins with an impossibility check using the total sum. Since splitting preserves sum, any target larger than the total is unreachable.

The `count` array stores frequencies of powers of two by exponent index. Using `bit_length() - 1` efficiently computes which power each number represents.

The main loop processes bits from least significant to most significant. If the target requires the current bit and we already have one available, we consume it immediately.

If the bit is unavailable, we search for the nearest higher power and repeatedly split downward. Every split creates two smaller powers and increments the operation count.

Finally, unused powers are merged upward using integer division by two. This simulates binary carrying and ensures higher bits correctly reflect leftover resources.

## Go Solution

```go
func minOperations(nums []int, target int) int {
	total := 0
	for _, num := range nums {
		total += num
	}

	if total < target {
		return -1
	}

	count := make([]int, 32)

	for _, num := range nums {
		bit := 0
		for (1 << bit) != num {
			bit++
		}
		count[bit]++
	}

	operations := 0

	for bit := 0; bit < 31; bit++ {
		if ((target >> bit) & 1) == 1 {
			if count[bit] > 0 {
				count[bit]--
			} else {
				higher := bit + 1

				for higher < 32 && count[higher] == 0 {
					higher++
				}

				for higher > bit {
					count[higher]--
					count[higher-1] += 2
					operations++
					higher--
				}

				count[bit]--
			}
		}

		count[bit+1] += count[bit] / 2
	}

	return operations
}
```

The Go implementation follows exactly the same logic as the Python version. Since Go does not provide a direct equivalent of Python's `bit_length()`, the exponent is computed using a loop.

The constraints guarantee values fit safely inside Go's integer range on standard 64 bit systems, so no special overflow handling is required.

Slices are initialized with fixed size `32`, which is sufficient because all numbers are at most `2^30`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(31) | At most 31 bit positions are processed |
| Space | O(31) | Frequency array has constant size |

Although nested loops appear in the implementation, the total number of bit transitions is bounded by the fixed integer width. Since all values fit within 31 bits, the runtime is effectively constant relative to input magnitude.

The only input-dependent work is building the frequency table, which requires `O(n)` time for `n = len(nums)`. Therefore, the full complexity is technically `O(n + 31)`, which simplifies to `O(n)`.

## Edge Cases

### Total Sum Smaller Than Target

One of the most important edge cases occurs when the sum of all numbers is smaller than `target`.

For example:

```
nums = [1,2]
target = 10
```

No sequence of splits can increase the total sum. Since splitting only redistributes value into smaller powers, the target is fundamentally impossible.

The implementation handles this immediately with:

```
if sum(nums) < target:
    return -1
```

This prevents unnecessary processing.

### Missing Lower Bit Requiring Multiple Splits

Another tricky case occurs when the needed bit is absent and only a much larger power exists.

Example:

```
nums = [32]
target = 1
```

To create a `1`, we must repeatedly split:

```
32 -> 16
16 -> 8
8 -> 4
4 -> 2
2 -> 1
```

The algorithm correctly counts one operation per split level and always chooses the nearest available higher bit.

### Excess Lower Bits Combining Upward

A subtle scenario occurs when many smaller powers naturally combine into larger powers.

Example:

```
nums = [1,1,1,1]
target = 4
```

Even though no `4` initially exists, the four `1`s combine into:

```
1+1 = 2
2+2 = 4
```

No split operations are required.

The carry step:

```
count[bit + 1] += count[bit] // 2
```

correctly simulates this binary merging behavior.
