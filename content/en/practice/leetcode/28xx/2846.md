## Solution

## Problem Understanding

This problem gives us a weighted undirected tree with `n` nodes. Since the graph is a tree, there is exactly one simple path between any pair of nodes. Every edge has a weight between `1` and `26`.

For each query `[a, b]`, we must examine the unique path between node `a` and node `b`. The goal is to make every edge on that path have the same weight. In one operation, we may change the weight of any edge to any value we want.

The important observation is that we do not care what the final weight becomes. We only care about minimizing the number of edge modifications.

Suppose the path contains these weights:

```
[1, 1, 1, 2, 2]
```

The best strategy is to keep the most frequent weight unchanged and modify all other edges to match it. In this case:

- Weight `1` appears `3` times
- Weight `2` appears `2` times

We keep all `1`s and change the two `2`s, or vice versa. The minimum number of operations is:

```
total_edges_on_path - maximum_frequency
= 5 - 3
= 2
```

Therefore, every query reduces to two tasks:

1. Determine the number of edges on the path.
2. Determine the frequency of each weight on that path.

The constraints are important:

- `n <= 10^4`
- `queries <= 2 * 10^4`

A naive traversal for every query could become too slow because each query might walk through an entire path of length `O(n)`.

The edge weights are limited to only `1` through `26`, which is a very small constant. This strongly hints that we should maintain frequency information for all weights during preprocessing.

Several edge cases are important:

- Queries where `a == b`, meaning the path contains no edges.
- Paths where all weights are already equal.
- Paths where every edge has a different weight.
- Deep trees that resemble linked lists.
- Large numbers of queries that require efficient repeated processing.

The problem guarantees the graph is a valid tree, so:

- The graph is connected.
- There are exactly `n - 1` edges.
- There is exactly one path between any two nodes.

These guarantees simplify the solution significantly because we never need to consider multiple possible paths.

## Approaches

### Brute Force Approach

The brute force method processes each query independently.

For a query `[a, b]`, we can perform a DFS or BFS to find the path from `a` to `b`. Once we discover the path, we collect all edge weights along it and count their frequencies.

If the path contains `k` edges and the maximum frequency among all weights is `f`, then the answer is:

```
k - f
```

This works because we keep all edges with the most common weight and change every other edge.

Although this approach is correct, it is inefficient. In the worst case, each query may traverse nearly the entire tree, costing `O(n)` time per query. With up to `2 * 10^4` queries, the total complexity becomes too large.

### Optimal Approach

The key insight is that we repeatedly need information about paths in a tree. This suggests using Lowest Common Ancestor, LCA, preprocessing.

For any two nodes `u` and `v`:

```
path(u, v) = path(root, u) + path(root, v) - 2 * path(root, lca)
```

We preprocess:

- The depth of every node
- Binary lifting tables for LCA queries
- Prefix frequency counts for all edge weights from the root to every node

Let:

```
count[node][w]
```

represent how many edges with weight `w` appear on the path from the root to `node`.

Then for a query `(u, v)`:

```
frequency_of_w_on_path =
count[u][w] + count[v][w] - 2 * count[lca][w]
```

We compute this for all `26` possible weights.

The total number of edges on the path is:

```
depth[u] + depth[v] - 2 * depth[lca]
```

Then:

```
answer = path_length - maximum_frequency
```

Because the number of weights is only `26`, each query becomes very efficient.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m * n) | O(n) | DFS/BFS for every query |
| Optimal | O((n + m) * log n) | O(n log n + 26n) | Binary lifting with prefix frequency counts |

## Algorithm Walkthrough

1. Build the adjacency list for the tree.

Since the graph is undirected, each edge is stored in both directions. Every adjacency entry stores both the neighboring node and the edge weight.
2. Choose a root node.

We root the tree at node `0`. Root choice does not matter because the tree structure is fixed.
3. Perform DFS preprocessing.

During DFS, compute:

- `depth[node]`
- `parent[node][0]`, the direct parent
- `count[node][w]`, the number of occurrences of weight `w` from root to node

When moving from parent to child through an edge with weight `w`, copy the parent's frequency array and increment `w`.
4. Build the binary lifting table.

Binary lifting allows efficient LCA computation.

Define:

```
parent[node][j]
```

as the `2^j`th ancestor of `node`.

Build this table bottom up:

```
parent[node][j] =
parent[parent[node][j-1]][j-1]
```
5. Process each query.

For nodes `u` and `v`:

- Compute `lca(u, v)`
- Compute path length:

```
depth[u] + depth[v] - 2 * depth[lca]
```
6. Compute frequency counts on the path.

For each weight `1` to `26`:

```
freq =
count[u][w] + count[v][w] - 2 * count[lca][w]
```

Track the maximum frequency.
7. Compute the answer.

The minimum operations needed are:

```
path_length - max_frequency
```
8. Return all answers.

### Why it works

The path between two nodes in a tree can always be decomposed using their LCA. Prefix frequency arrays allow us to reconstruct weight frequencies on any path using inclusion and exclusion.

Since changing all non-majority weights is always optimal, minimizing operations reduces to maximizing the number of unchanged edges. Therefore:

```
minimum_operations =
total_edges - most_frequent_weight_count
```

This guarantees correctness for every query.

## Python Solution

```python
from typing import List
import math

class Solution:
    def minOperationsQueries(
        self,
        n: int,
        edges: List[List[int]],
        queries: List[List[int]]
    ) -> List[int]:

        graph = [[] for _ in range(n)]

        for u, v, w in edges:
            graph[u].append((v, w))
            graph[v].append((u, w))

        log = math.ceil(math.log2(n)) + 1

        parent = [[-1] * log for _ in range(n)]
        depth = [0] * n

        # count[node][w] = occurrences of weight w
        # on path from root to node
        count = [[0] * 27 for _ in range(n)]

        def dfs(node: int, par: int) -> None:
            for nei, weight in graph[node]:
                if nei == par:
                    continue

                depth[nei] = depth[node] + 1
                parent[nei][0] = node

                count[nei] = count[node][:]  # copy frequencies
                count[nei][weight] += 1

                dfs(nei, node)

        dfs(0, -1)

        # Build binary lifting table
        for j in range(1, log):
            for node in range(n):
                prev = parent[node][j - 1]

                if prev != -1:
                    parent[node][j] = parent[prev][j - 1]

        def lca(u: int, v: int) -> int:
            if depth[u] < depth[v]:
                u, v = v, u

            # Lift u to same depth
            diff = depth[u] - depth[v]

            for j in range(log):
                if diff & (1 << j):
                    u = parent[u][j]

            if u == v:
                return u

            # Lift both together
            for j in range(log - 1, -1, -1):
                if parent[u][j] != parent[v][j]:
                    u = parent[u][j]
                    v = parent[v][j]

            return parent[u][0]

        answers = []

        for u, v in queries:
            ancestor = lca(u, v)

            path_length = (
                depth[u] + depth[v] - 2 * depth[ancestor]
            )

            max_frequency = 0

            for weight in range(1, 27):
                freq = (
                    count[u][weight]
                    + count[v][weight]
                    - 2 * count[ancestor][weight]
                )

                max_frequency = max(max_frequency, freq)

            answers.append(path_length - max_frequency)

        return answers
```

The implementation begins by constructing an adjacency list representation of the tree. Each entry stores both the neighboring node and the edge weight.

The DFS preprocessing stage computes three critical pieces of information:

- Node depths
- Immediate parents
- Prefix frequency counts for edge weights

The `count` array is the most important optimization. Instead of recomputing frequencies for every query, we store cumulative counts from the root to each node.

The binary lifting table is then constructed so that LCA queries can be answered in logarithmic time. Each level stores ancestors at exponentially increasing distances.

The `lca` function first equalizes node depths, then lifts both nodes upward until their parents converge.

For each query, the algorithm computes:

- The LCA
- The path length
- The frequency of each weight on the path

Finally, the answer is calculated by subtracting the maximum frequency from the total path length.

## Go Solution

```go
package main

import "math"

func minOperationsQueries(n int, edges [][]int, queries [][]int) []int {
	graph := make([][][2]int, n)

	for _, e := range edges {
		u, v, w := e[0], e[1], e[2]

		graph[u] = append(graph[u], [2]int{v, w})
		graph[v] = append(graph[v], [2]int{u, w})
	}

	log := int(math.Ceil(math.Log2(float64(n)))) + 1

	parent := make([][]int, n)
	for i := range parent {
		parent[i] = make([]int, log)

		for j := range parent[i] {
			parent[i][j] = -1
		}
	}

	depth := make([]int, n)

	count := make([][]int, n)
	for i := range count {
		count[i] = make([]int, 27)
	}

	var dfs func(node, par int)

	dfs = func(node, par int) {
		for _, edge := range graph[node] {
			nei := edge[0]
			weight := edge[1]

			if nei == par {
				continue
			}

			depth[nei] = depth[node] + 1
			parent[nei][0] = node

			copy(count[nei], count[node])
			count[nei][weight]++

			dfs(nei, node)
		}
	}

	dfs(0, -1)

	for j := 1; j < log; j++ {
		for node := 0; node < n; node++ {
			prev := parent[node][j-1]

			if prev != -1 {
				parent[node][j] = parent[prev][j-1]
			}
		}
	}

	lca := func(u, v int) int {
		if depth[u] < depth[v] {
			u, v = v, u
		}

		diff := depth[u] - depth[v]

		for j := 0; j < log; j++ {
			if (diff & (1 << j)) != 0 {
				u = parent[u][j]
			}
		}

		if u == v {
			return u
		}

		for j := log - 1; j >= 0; j-- {
			if parent[u][j] != parent[v][j] {
				u = parent[u][j]
				v = parent[v][j]
			}
		}

		return parent[u][0]
	}

	ans := make([]int, 0, len(queries))

	for _, q := range queries {
		u, v := q[0], q[1]

		ancestor := lca(u, v)

		pathLength := depth[u] + depth[v] - 2*depth[ancestor]

		maxFreq := 0

		for weight := 1; weight <= 26; weight++ {
			freq := count[u][weight] +
				count[v][weight] -
				2*count[ancestor][weight]

			if freq > maxFreq {
				maxFreq = freq
			}
		}

		ans = append(ans, pathLength-maxFreq)
	}

	return ans
}
```

The Go implementation closely mirrors the Python solution. Since Go does not support dynamic nested list copying as conveniently as Python, the `copy` function is used to duplicate frequency arrays during DFS preprocessing.

The adjacency list uses `[2]int` pairs to store neighbors and weights compactly. All integers safely fit within Go's standard `int` type because the constraints are small.

The binary lifting table is initialized with `-1` values to represent nonexistent ancestors.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n + m log n + 26m) | Preprocessing plus query handling |
| Space | O(n log n + 26n) | Binary lifting table and frequency arrays |

The DFS preprocessing visits every node once and constructs frequency arrays. Building the binary lifting table requires `O(n log n)` time.

Each query performs an LCA lookup in `O(log n)` time and iterates through all `26` possible weights, which is effectively constant time.

The space complexity comes primarily from the ancestor table and the prefix frequency arrays.

## Edge Cases

One important edge case occurs when `a == b`. In this situation, the path contains no edges at all. A careless implementation might incorrectly attempt to compute frequencies or subtract invalid values. The implementation handles this naturally because the path length becomes zero, and therefore the answer is also zero.

Another critical edge case is when all edge weights on the path are already identical. In this case, the maximum frequency equals the total path length, so the answer becomes zero. The algorithm correctly identifies this because frequency counting is exact.

A third important case involves paths where every edge weight is different. For example:

```
[1,2,3,4]
```

The maximum frequency is only `1`, meaning all other edges must change. The implementation correctly computes:

```
4 - 1 = 3
```

Deep trees that resemble linked lists are another common source of bugs, especially in LCA implementations. Binary lifting handles these efficiently because ancestor jumps remain logarithmic regardless of tree shape.

Finally, trees with many branches require careful LCA computation. Incorrect ancestor lifting can easily produce wrong path decompositions. The implementation avoids this by first equalizing depths and then lifting both nodes simultaneously until their parents converge.
