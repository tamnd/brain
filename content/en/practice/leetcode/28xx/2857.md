## Solution

## Problem Understanding

In this problem, we are given a list of 2D points, where each point is represented as `[x, y]`. We are also given an integer `k`.

For every pair of points `(i, j)` such that `i < j`, we define the distance between them as:

$$(x_i \oplus x_j) + (y_i \oplus y_j)$$

where `⊕` denotes the bitwise XOR operation.

The goal is to count how many pairs of points have a distance exactly equal to `k`.

The input size is important here. The number of points can be as large as `50,000`, which immediately tells us that checking every pair directly may be too expensive. A naive pairwise comparison would require:

$$O(n^2)$$

comparisons, which becomes far too slow when `n = 50,000`.

Another important constraint is:

$$0 \le k \le 100$$

This small upper bound on `k` is the key observation that enables an efficient solution.

The coordinates themselves can be as large as `10^6`, so we cannot use a dense 2D grid or DP table indexed by coordinates. Instead, we need a sparse structure such as a hash map.

There are several important edge cases to consider:

- When `k = 0`, the condition becomes:

$$(x_1 \oplus x_2) + (y_1 \oplus y_2) = 0$$

Since XOR values are non-negative, both XOR terms must individually be `0`, meaning the two points must be identical.
- Multiple duplicate points may exist, and all valid pair combinations must be counted correctly.
- Coordinates may be very large, so solutions depending on coordinate ranges are not feasible.
- Because XOR is involved, geometric intuition about Euclidean or Manhattan distance does not apply here.

## Approaches

### Brute Force Approach

The most direct solution is to check every possible pair of points.

For each pair `(i, j)`:

1. Compute:

$$(x_i \oplus x_j) + (y_i \oplus y_j)$$
2. If the result equals `k`, increment the answer.

This works because it explicitly evaluates the condition for every pair, guaranteeing correctness.

However, the time complexity is:

$$O(n^2)$$

With `50,000` points, this means roughly:

$$\frac{50,000 \times 49,999}{2}$$

pair checks, which is over one billion operations and far too slow.

### Optimal Approach

The key observation is that:

$$(x_1 \oplus x_2) + (y_1 \oplus y_2) = k$$

Since `k` is at most `100`, there are only a small number of possible XOR splits.

Suppose:

$$x_1 \oplus x_2 = a$$

Then necessarily:

$$y_1 \oplus y_2 = k - a$$

For every possible value of `a` from `0` to `k`, we can determine exactly what coordinates a matching point must have.

If the current point is `(x, y)`, then a valid previous point must satisfy:

$$x' = x \oplus a$$

and

$$y' = y \oplus (k - a)$$

We can store previously seen points in a hash map. Then for each point, we enumerate all `k + 1` possible XOR decompositions and check whether the corresponding matching point already exists.

Because `k ≤ 100`, each point requires only about `101` checks, making the solution efficient.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Checks every pair directly |
| Optimal | O(n × k) | O(n) | Uses hash map and XOR decomposition |

## Algorithm Walkthrough

1. Create a hash map that stores how many times each point has already appeared.

The key will be a tuple `(x, y)` and the value will be its frequency.
2. Initialize `answer = 0`.
3. Process the points one by one.

We only look for matching points among previously processed points. This naturally guarantees `i < j` and avoids double counting.
4. For the current point `(x, y)`, iterate through all possible values of:

$$a = 0 \text{ to } k$$

Here:

$$a = x_1 \oplus x_2$$
5. Compute the required XOR value for the y-coordinate:

$$b = k - a$$
6. Determine the coordinates of the matching point.

Using XOR properties:

$$x' = x \oplus a$$

$$y' = y \oplus b$$
7. Look up `(x', y')` in the hash map.

If it exists, add its frequency to the answer because every occurrence forms a valid pair with the current point.
8. After checking all possible XOR decompositions, insert the current point into the hash map.
9. Continue until all points are processed.

### Why it works

For every valid pair, there exists exactly one decomposition:

$$(x_1 \oplus x_2) = a$$

and

$$(y_1 \oplus y_2) = k-a$$

The algorithm enumerates all such decompositions for each point and reconstructs the exact coordinates a matching previous point must have.

Because the hash map contains only previously processed points, every valid pair is counted exactly once.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Solution:
    def countPairs(self, coordinates: List[List[int]], k: int) -> int:
        point_count = defaultdict(int)
        answer = 0

        for x, y in coordinates:
            for xor_x in range(k + 1):
                xor_y = k - xor_x

                target_x = x ^ xor_x
                target_y = y ^ xor_y

                answer += point_count[(target_x, target_y)]

            point_count[(x, y)] += 1

        return answer
```

The implementation follows the algorithm directly.

We use `defaultdict(int)` so that missing points automatically return frequency `0`.

For every point `(x, y)`, we enumerate all possible XOR contributions to the x-coordinate. Since the total distance must equal `k`, the y-coordinate XOR contribution is determined automatically as `k - xor_x`.

Using XOR properties, we reconstruct the exact coordinates a matching point must have. If that point has already appeared, its frequency contributes directly to the answer.

Finally, after all matches are counted, we add the current point to the hash map so future points can pair with it.

## Go Solution

```go
func countPairs(coordinates [][]int, k int) int {
	pointCount := make(map[[2]int]int)
	answer := 0

	for _, point := range coordinates {
		x := point[0]
		y := point[1]

		for xorX := 0; xorX <= k; xorX++ {
			xorY := k - xorX

			targetX := x ^ xorX
			targetY := y ^ xorY

			answer += pointCount[[2]int{targetX, targetY}]
		}

		pointCount[[2]int{x, y}]++
	}

	return answer
}
```

The Go implementation is structurally identical to the Python version.

Since Go does not have tuple keys like Python dictionaries, we use a fixed-size array `[2]int` as the map key. Arrays are comparable in Go and therefore valid map keys.

Go maps return the zero value for missing keys automatically, so nonexistent points contribute `0` without requiring special handling.

Integer overflow is not an issue because the maximum number of pairs is:

$$\frac{50000 \times 49999}{2}$$

which comfortably fits inside a 32-bit signed integer.

## Edge Cases

### Edge Case 1: `k = 0`

When `k` is zero, the equation becomes:

$$(x_1 \oplus x_2) + (y_1 \oplus y_2) = 0$$

Since XOR values are always non-negative, both terms must individually equal zero. That means:

$$x_1 = x_2$$

and

$$y_1 = y_2$$

A common bug is accidentally counting non-identical points. The implementation handles this correctly because the only decomposition considered is:

```
xor_x = 0
xor_y = 0
```

which reconstructs only identical coordinates.

### Edge Case 2: Many Duplicate Points

Suppose the input contains thousands of identical coordinates.

A naive implementation might recount pairs multiple times or fail to account for multiplicity correctly.

The hash map stores frequencies, not just existence. When a matching point appears multiple times, all previous copies contribute valid pairs automatically.

For example, if a point has appeared `4` times already, the next identical point contributes `4` new pairs immediately.

### Edge Case 3: Very Large Coordinates

Coordinates can be as large as `10^6`.

Solutions using coordinate grids or large matrices would consume excessive memory.

This implementation uses a hash map keyed by coordinate tuples, so memory usage depends only on the number of unique points, not the coordinate magnitude.

The XOR computations also remain efficient regardless of coordinate size.
