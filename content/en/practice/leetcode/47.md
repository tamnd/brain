## Solution

## Problem Understanding

This problem asks us to generate every possible permutation of the input array, while ensuring that duplicate permutations are not included in the final result.

A permutation is an arrangement of all elements in a particular order. For example, the array `[1,2,3]` has six permutations because there are six distinct ways to arrange the three numbers.

The complication in this problem is that the input array may contain duplicate values. If we generate permutations naively, duplicate numbers will produce repeated permutations. For example, with `[1,1,2]`, swapping the two `1`s does not create a new arrangement, even though a brute-force algorithm might treat them as different positions.

The input is an integer array `nums`, and the output should be a list containing every unique permutation. The order of the returned permutations does not matter.

The constraints are important:

- `1 <= nums.length <= 8`
- `-10 <= nums[i] <= 10`

The array length is very small, which strongly suggests that exponential or factorial algorithms are acceptable. Since permutations inherently involve factorial growth, any correct solution will have complexity related to `n!`.

However, even with only eight elements, duplicate handling matters. Without careful pruning, the algorithm may generate many redundant permutations and waste significant time.

Some important edge cases include:

- Arrays where all elements are identical, such as `[1,1,1]`
- Arrays with no duplicates, such as `[1,2,3]`
- Arrays with multiple duplicate groups, such as `[1,1,2,2]`
- Arrays of length `1`
- Negative numbers, since values are not restricted to positive integers

The problem guarantees that the array length is at least one, so we never need to handle an empty input.

## Approaches

### Brute Force Approach

A straightforward approach is to generate all possible permutations first, then remove duplicates afterward.

One way to do this is:

1. Generate every permutation using standard backtracking.
2. Convert each permutation into a tuple or string representation.
3. Store permutations inside a set to eliminate duplicates.
4. Convert the unique results back into a list.

This works because sets only keep unique values.

For example, with `[1,1,2]`, a naive permutation generator would produce:

```
[1,1,2]
[1,2,1]
[1,1,2]
[1,2,1]
[2,1,1]
[2,1,1]
```

The set removes duplicates afterward.

The problem with this method is inefficiency. The algorithm still explores every possible arrangement, including duplicate branches that produce identical results. When duplicates are common, much of the work becomes redundant.

### Optimal Approach

The key insight is that duplicate permutations arise because identical values can be chosen interchangeably at the same recursion level.

To avoid generating duplicates in the first place:

1. Sort the array.
2. Use a `used` array to track which indices are already included in the current permutation.
3. Skip duplicate values during recursion when they would produce the same branch.

The critical pruning rule is:

```
If nums[i] == nums[i-1] and nums[i-1] has not been used,
skip nums[i]
```

This rule ensures that duplicate values are selected in a consistent order.

For example, after sorting `[1,1,2]`:

```
[1,1,2]
```

At a given recursion depth, if the first `1` has not been used yet, we do not allow the second `1` to start another equivalent branch.

This avoids duplicate permutations entirely, making the algorithm significantly more efficient.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × n!) | O(n × n!) | Generates all permutations, then removes duplicates using a set |
| Optimal | O(n × n!) | O(n) excluding output | Uses sorting and pruning to avoid duplicate branches during generation |

## Algorithm Walkthrough

1. Sort the input array.

Sorting groups duplicate values together. This is essential because the duplicate-skipping logic depends on adjacent equal values.
2. Create a `used` array.

The `used` array keeps track of which indices are already included in the current permutation. This prevents reusing the same element twice.
3. Create a recursive backtracking function.

The function builds permutations one element at a time.
4. Check whether the current permutation is complete.

If the current permutation length equals `len(nums)`, we have formed a valid unique permutation. Add a copy of it to the result list.
5. Iterate through all indices.

At each recursion level, try choosing every unused element.
6. Skip already used elements.

If `used[i]` is `True`, the element is already part of the current permutation.
7. Skip duplicate branches.

The important pruning condition is:

```
if i > 0 and nums[i] == nums[i - 1] and not used[i - 1]:
```

This means:

- The current value equals the previous value
- The previous duplicate has not been used in this branch
- Therefore choosing the current duplicate would create a redundant permutation
8. Choose the current element.

Mark it as used and append it to the current permutation.
9. Recurse deeper.

Continue building the permutation.
10. Backtrack.

Remove the element from the current permutation and mark it unused again.

This restores the state so other branches can be explored.

## Python Solution

```
from typing import List

class Solution:
    def permuteUnique(self, nums: List[int]) -> List[List[int]]:
        nums.sort()
        
        results = []
        current_permutation = []
        used = [False] * len(nums)

        def backtrack():
            if len(current_permutation) == len(nums):
                results.append(current_permutation[:])
                return

            for i in range(len(nums)):
                if used[i]:
                    continue

                if i > 0 and nums[i] == nums[i - 1] and not used[i - 1]:
                    continue

                used[i] = True
                current_permutation.append(nums[i])

                backtrack()

                current_permutation.pop()
                used[i] = False

        backtrack()
        return results
```

The implementation begins by sorting the input array. Sorting ensures duplicate values appear next to each other, which allows the pruning condition to work correctly.

The `results` list stores all completed unique permutations. The `current_permutation` list represents the permutation currently being built during recursion.

The `used` array tracks whether each index has already been included in the current branch. This is important because permutations depend on positions, not just values.

The recursive `backtrack` function handles the depth-first search. Whenever the current permutation length matches the original array length, a complete permutation has been formed, so a copy is appended to the result.

Inside the loop, the algorithm first skips already used indices. Then it applies the duplicate-pruning condition. If the current value equals the previous value and the previous value has not been used yet, the branch is skipped because it would generate a duplicate ordering.

When a valid element is chosen, it is added to the current permutation and marked as used. After recursion returns, the algorithm removes the element and resets the used flag. This backtracking step restores the state for exploring other possibilities.

## Go Solution

```
package main

import "sort"

func permuteUnique(nums []int) [][]int {
	sort.Ints(nums)

	var results [][]int
	currentPermutation := []int{}
	used := make([]bool, len(nums))

	var backtrack func()

	backtrack = func() {
		if len(currentPermutation) == len(nums) {
			permutationCopy := make([]int, len(currentPermutation))
			copy(permutationCopy, currentPermutation)
			results = append(results, permutationCopy)
			return
		}

		for i := 0; i < len(nums); i++ {
			if used[i] {
				continue
			}

			if i > 0 && nums[i] == nums[i-1] && !used[i-1] {
				continue
			}

			used[i] = true
			currentPermutation = append(currentPermutation, nums[i])

			backtrack()

			currentPermutation = currentPermutation[:len(currentPermutation)-1]
			used[i] = false
		}
	}

	backtrack()

	return results
}
```

The Go implementation follows the same algorithmic structure as the Python version.

One important difference is that slices in Go reference shared underlying arrays. Because of this, when a complete permutation is found, the code must create a fresh copy before appending it to `results`. Otherwise, future modifications during backtracking would overwrite previously stored permutations.

The `used` slice functions similarly to the Python boolean list. The recursive closure `backtrack` captures the surrounding variables and performs the depth-first search.

Go does not require special handling for integer overflow here because the input constraints are very small.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n × n!) | There are up to `n!` permutations, and copying each permutation costs `O(n)` |
| Space | O(n) excluding output | Recursion depth, current permutation, and used array each require linear space |

The factorial term dominates because every valid permutation must be generated. Even with pruning, the worst case occurs when all elements are distinct, producing `n!` permutations.

The auxiliary space usage is linear because the recursion stack, current permutation, and `used` array all scale with the input size. The output itself requires much more space, but output storage is usually excluded from auxiliary space analysis.

## Edge Cases

### All Elements Identical

Consider:

```
[1,1,1]
```

A naive solution would generate multiple identical permutations. The pruning condition prevents duplicate branches from being explored. Only one valid permutation is produced:

```
[[1,1,1]]
```

### Single Element Array

Consider:

```
[5]
```

The recursion immediately reaches the base case after selecting the only element. The algorithm correctly returns:

```
[[5]]
```

This verifies that the implementation handles the minimum allowed input size.

### Multiple Duplicate Groups

Consider:

```
[1,1,2,2]
```

This case is more complex because duplicates appear in multiple groups. Without careful pruning, duplicate permutations would explode combinatorially.

Sorting ensures duplicates are adjacent, and the condition:

```
nums[i] == nums[i - 1] and not used[i - 1]
```

guarantees that each duplicate group is explored in a consistent order. This prevents redundant branches while still generating every distinct permutation exactly once.
