## Solution

## Problem Understanding

This problem asks us to determine, for each person arriving at a specific time, how many flowers are in full bloom at that exact moment.

Each flower is represented by an interval `[starti, endi]`, meaning that flower is blooming at every integer time point from `starti` to `endi`, inclusive. A person arrives at time `people[i]`, and we must count how many of these intervals include that time.

In other words, for each query time `t`, we want to count how many intervals satisfy `starti <= t <= endi`.

The input consists of up to `5 * 10^4` flowers and `5 * 10^4` people queries, and each time value can be as large as `10^9`. This immediately implies that any solution with quadratic complexity over flowers and people will be too slow.

The key constraints and implications are that we need a solution that can efficiently answer many interval containment queries over a large set of intervals without scanning all intervals for each query.

Important edge cases include:

- All flowers overlapping all times, producing maximum counts for every query.
- Non-overlapping intervals where counts are mostly 0 or 1.
- Large coordinate values requiring sorting or binary search rather than array indexing.
- Queries occurring before all flowers start or after all flowers end.

The problem guarantees valid intervals with `starti <= endi`, and all values are positive integers.

## Approaches

### Brute Force Approach

The most direct way to solve this problem is to iterate over every person and, for each person, scan all flowers and count how many intervals include that person's arrival time. This is straightforward: for each query time `t`, we check every flower interval and increment a counter if `starti <= t <= endi`.

This works correctly because it directly models the definition of the problem. However, it is too slow because it performs `O(n * m)` operations, where `n` is number of people and `m` is number of flowers. With both up to `5 * 10^4`, this results in up to `2.5 * 10^9` checks, which is not feasible.

### Optimal Approach: Sorting + Binary Search

The key insight is that for a given time `t`, the number of active flowers can be computed as:

number of flowers started by time `t` minus number of flowers ended before time `t`.

If we separate all start times and end times into two sorted arrays, we can answer each query using binary search:

- Count of starts `<= t` gives how many flowers have begun blooming.
- Count of ends `< t` gives how many have already finished before `t`.

The difference gives the number of active intervals.

This reduces the problem to two sorted arrays and fast range counting using binary search.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × m) | O(1) | Check every flower for every person |
| Optimal (Sorting + Binary Search) | O((n + m) log n) | O(n) | Pre-sort start/end arrays, answer queries via binary search |

## Algorithm Walkthrough

1. First, extract all flower start times into one array and all end times into another array. This separation allows us to independently reason about when flowers become active and when they stop being active.
2. Sort both the start times array and the end times array. Sorting is essential because it enables binary search operations, which allow us to count how many elements fall within a condition in logarithmic time.
3. For each person's arrival time `t`, compute how many flowers have started blooming by using binary search to find the first index in the sorted start array where value is greater than `t`. This index gives the count of starts `<= t`.
4. Similarly, compute how many flowers have already ended before time `t` by finding the first index in the end array where value is greater than or equal to `t`. This index gives the count of ends `< t`.
5. Subtract the number of ended flowers from the number of started flowers to get the number of flowers currently in bloom at time `t`.
6. Store this result in the answer array corresponding to the current person.
7. Repeat for all queries independently.

### Why it works

The correctness comes from partitioning all intervals into three disjoint categories relative to time `t`: those that have not started, those currently active, and those that have ended. Every interval falls into exactly one category, and binary search allows us to compute the boundaries of these categories efficiently. The difference between "started" and "ended" intervals exactly equals the number of active intervals.

## Python Solution

```python
from typing import List
import bisect

class Solution:
    def fullBloomFlowers(self, flowers: List[List[int]], people: List[int]) -> List[int]:
        starts = []
        ends = []
        
        for s, e in flowers:
            starts.append(s)
            ends.append(e)
        
        starts.sort()
        ends.sort()
        
        result = []
        
        for t in people:
            started = bisect.bisect_right(starts, t)
            ended_before = bisect.bisect_left(ends, t)
            result.append(started - ended_before)
        
        return result
```

### Implementation Explanation

We first split each flower interval into two separate arrays: `starts` and `ends`. Sorting them enables binary search operations. For each person, we compute how many flowers have started using `bisect_right`, which counts all values `<= t`. We compute how many have ended before `t` using `bisect_left`, which counts values `< t`. The difference gives active flowers.

This directly implements the mathematical decomposition of intervals into started minus ended.

## Go Solution

```go
import (
    "sort"
)

func fullBloomFlowers(flowers [][]int, people []int) []int {
    n := len(flowers)
    starts := make([]int, n)
    ends := make([]int, n)

    for i, f := range flowers {
        starts[i] = f[0]
        ends[i] = f[1]
    }

    sort.Ints(starts)
    sort.Ints(ends)

    result := make([]int, len(people))

    for i, t := range people {
        started := upperBound(starts, t)
        endedBefore := lowerBound(ends, t)
        result[i] = started - endedBefore
    }

    return result
}

func upperBound(arr []int, target int) int {
    left, right := 0, len(arr)
    for left < right {
        mid := left + (right-left)/2
        if arr[mid] <= target {
            left = mid + 1
        } else {
            right = mid
        }
    }
    return left
}

func lowerBound(arr []int, target int) int {
    left, right := 0, len(arr)
    for left < right {
        mid := left + (right-left)/2
        if arr[mid] < target {
            left = mid + 1
        } else {
            right = mid
        }
    }
    return left
}
```

### Go-specific Notes

In Go, we do not have built-in bisect utilities, so we implement `lowerBound` and `upperBound` manually using binary search. We also explicitly manage slices using `make`. Integer handling is straightforward since all values fit within `int`.

The logic remains identical to Python, but boundary handling in binary search must be carefully implemented to avoid off-by-one errors.

## Edge Cases

One important edge case is when all flowers overlap completely across the entire range of queries. In this case, every query should return the total number of flowers, and the binary search logic must correctly count all starts and ends without overflow or misclassification.

Another edge case occurs when queries fall before all flowers start or after all flowers end. In these cases, either the started count is zero or the ended count equals all flowers, producing zero active intervals. Correct handling depends on proper use of `bisect_right` and `bisect_left` boundaries.

A final edge case involves flowers with identical start and end times, such as `[t, t]`. These should only count as active at exactly time `t`, and the inclusive interval logic is preserved by using `<=` for starts and `<` for ends in the binary search decomposition.
