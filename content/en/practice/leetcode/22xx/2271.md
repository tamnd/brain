## Solution

## Problem Understanding

In this problem, we are given several disjoint intervals representing white tiles on a number line. Each interval `[li, ri]` means that every integer position from `li` to `ri`, inclusive, is white.

We are also given a carpet of fixed length `carpetLen`. The carpet can be placed anywhere on the number line, and it covers every position in its range continuously. Our goal is to determine the maximum number of white tiles that can be covered by a single placement of the carpet.

The important detail is that the tile intervals are non-overlapping, but they are not necessarily adjacent. There may be gaps between intervals that contain no white tiles. Since the carpet covers a continuous segment, some portion of it may end up covering gaps, which contributes nothing to the total.

For example, if we place a carpet of length `10` starting at position `10`, then it covers positions `10` through `19` inclusive. We must count how many of those positions belong to white tile intervals.

The constraints are large:

- Up to `5 * 10^4` intervals
- Coordinates can be as large as `10^9`

These constraints immediately tell us that iterating over every individual tile position is impossible. A naive coordinate-based simulation would be far too slow and memory intensive.

Another important guarantee is that the intervals are non-overlapping. This property is extremely useful because it allows us to process intervals in sorted order and use sliding window techniques efficiently.

Several edge cases are important:

- The carpet may completely cover one or more intervals.
- The carpet may partially cover an interval.
- The carpet may fit entirely inside a single interval.
- There may be large gaps between intervals.
- The carpet may be longer than the total number of white tiles.
- Intervals may already be sorted or may require sorting.

The core challenge is efficiently determining the best carpet placement without checking every possible coordinate.

## Approaches

### Brute Force Approach

A brute force solution would try placing the carpet at many candidate positions and compute how many white tiles are covered each time.

One possible implementation is:

1. Enumerate every possible carpet starting position.
2. For each placement, compute the overlap with every interval.
3. Sum all covered portions.
4. Track the maximum result.

This works because overlap between two intervals can always be computed correctly using interval intersection logic.

However, this approach is far too slow.

The coordinate values go up to `10^9`, so checking every possible carpet position is impossible. Even if we only test interval boundaries as starting positions, we still end up with an `O(n^2)` solution because for each starting point we may scan all intervals again.

With `n = 5 * 10^4`, quadratic complexity is too expensive.

### Key Insight

The key observation is that after sorting the intervals, we can process them using a sliding window.

Suppose we place the carpet starting at the beginning of some interval. Then the carpet covers:

- Several complete intervals
- Possibly one partially covered interval at the end

Because intervals are non-overlapping and sorted, once an interval becomes fully covered, it stays fully covered as we expand the right side of the window.

This structure makes a two-pointer sliding window approach ideal.

We maintain:

- A left pointer representing the carpet start interval
- A right pointer representing intervals fully covered
- A running sum of fully covered white tiles

Then we compute whether the next interval is partially covered.

This reduces the problem to linear time after sorting.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Try many carpet positions and recompute overlaps |
| Optimal | O(n log n) | O(1) | Sort intervals, then use sliding window |

## Algorithm Walkthrough

1. Sort the intervals by starting coordinate.

Even though the intervals are non-overlapping, they are not guaranteed to be sorted. Sorting ensures we can process them from left to right.
2. Initialize two pointers.

Let `left` represent the interval where the carpet starts.

Let `right` represent the furthest interval currently considered.
3. Maintain a running total of fully covered tiles.

As we expand the right pointer, we add the full size of intervals that are completely inside the carpet range.
4. For each left interval, determine the carpet's coverage range.

If the carpet starts at `tiles[left][0]`, then the carpet ends at:

```
carpetEnd = tiles[left][0] + carpetLen - 1
```
5. Expand the right pointer while intervals are fully covered.

An interval `[start, end]` is fully covered if:

```
end <= carpetEnd
```

While this condition holds, add the interval's size to the running total and move `right` forward.
6. Handle partial overlap.

After fully covered intervals are processed, the next interval may still overlap partially.

Suppose the next interval is `[start, end]`.

The partial overlap length is:

```
carpetEnd - start + 1
```

but only if `start <= carpetEnd`.
7. Compute the current answer.

The total covered tiles are:

```
fullyCovered + partialCovered
```

Update the maximum answer.
8. Move the left pointer forward.

Before advancing `left`, remove its contribution from the running total if it was fully covered.
9. Continue until all intervals are processed.

### Why it works

The sliding window maintains the invariant that every interval between `left` and `right - 1` is fully covered by the carpet placement starting at `tiles[left][0]`.

Because intervals are sorted and non-overlapping, the carpet coverage boundary moves monotonically to the right. Each interval enters and leaves the window at most once, which guarantees linear scanning after sorting.

The algorithm always checks:

- All fully covered intervals
- The only possible partially covered interval

Therefore every valid carpet placement anchored at an interval start is evaluated correctly.

## Python Solution

```python
from typing import List

class Solution:
    def maximumWhiteTiles(self, tiles: List[List[int]], carpetLen: int) -> int:
        tiles.sort()

        n = len(tiles)

        left = 0
        right = 0

        fully_covered = 0
        answer = 0

        while left < n:
            carpet_end = tiles[left][0] + carpetLen - 1

            while right < n and tiles[right][1] <= carpet_end:
                fully_covered += tiles[right][1] - tiles[right][0] + 1
                right += 1

            current_cover = fully_covered

            if right < n and tiles[right][0] <= carpet_end:
                partial_cover = carpet_end - tiles[right][0] + 1
                current_cover += partial_cover

            answer = max(answer, current_cover)

            if left < right:
                fully_covered -= (
                    tiles[left][1] - tiles[left][0] + 1
                )
            else:
                right = left + 1

            left += 1

        return answer
```

The implementation begins by sorting the intervals so they can be processed sequentially.

The `left` pointer represents the interval where the carpet begins. For every such starting point, the algorithm computes the carpet's ending coordinate.

The `right` pointer expands as long as intervals are fully covered. Their lengths are accumulated in `fully_covered`.

Once expansion stops, there are only two possibilities:

- The next interval does not overlap at all.
- The next interval overlaps partially.

The partial overlap is computed directly using interval arithmetic.

After updating the answer, the algorithm slides the window forward by removing the interval at `left` from the running total.

Each interval is added and removed at most once, giving linear scanning behavior.

## Go Solution

```go
package main

import (
	"sort"
)

func maximumWhiteTiles(tiles [][]int, carpetLen int) int {
	sort.Slice(tiles, func(i, j int) bool {
		return tiles[i][0] < tiles[j][0]
	})

	n := len(tiles)

	left := 0
	right := 0

	fullyCovered := 0
	answer := 0

	for left < n {
		carpetEnd := tiles[left][0] + carpetLen - 1

		for right < n && tiles[right][1] <= carpetEnd {
			fullyCovered += tiles[right][1] - tiles[right][0] + 1
			right++
		}

		currentCover := fullyCovered

		if right < n && tiles[right][0] <= carpetEnd {
			partialCover := carpetEnd - tiles[right][0] + 1
			currentCover += partialCover
		}

		if currentCover > answer {
			answer = currentCover
		}

		if left < right {
			fullyCovered -= tiles[left][1] - tiles[left][0] + 1
		} else {
			right = left + 1
		}

		left++
	}

	return answer
}
```

The Go implementation follows the same logic as the Python version.

The primary language-specific difference is sorting. In Go, `sort.Slice` is used with a custom comparator.

All arithmetic safely fits within Go's `int` type because the maximum covered length cannot exceed `10^9`, which is within the range of a 64-bit Go integer on LeetCode environments.

Slices are used directly for interval access, and no additional data structures are required.

## Edge Cases

One important edge case occurs when the carpet is longer than the total number of white tiles. A buggy implementation might accidentally count gaps between intervals as covered white tiles. The sliding window solution avoids this because it only adds actual interval lengths, never raw coordinate distances.

Another tricky case happens when the carpet partially overlaps the next interval. Many incorrect solutions either forget partial overlap entirely or compute it off by one because intervals are inclusive. This implementation correctly computes:

```
carpetEnd - intervalStart + 1
```

which properly counts inclusive endpoints.

A third important edge case occurs when intervals are unsorted. Since the sliding window relies on monotonic ordering, failing to sort first would produce incorrect behavior. The implementation explicitly sorts the intervals before processing, guaranteeing that the two-pointer technique works correctly.

Another subtle case involves large coordinate values up to `10^9`. Any solution attempting coordinate compression over the full range or direct simulation would fail due to memory or runtime limits. This implementation only processes intervals themselves, making it scalable regardless of coordinate magnitude.
