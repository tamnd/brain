[LeetCode Problem 2247](https://leetcode.com/problems/maximum-cost-of-trip-with-k-highways/)

**Difficulty:** 🔴 Hard  
**Topics:** Dynamic Programming, Bit Manipulation, Graph Theory, Bitmask

## Solution

## Problem Understanding

In this problem, we are given an undirected weighted graph representing cities and highways. Each city is identified by an integer from `0` to `n - 1`, and every highway connects two cities with an associated toll cost.

The goal is to find a trip that uses exactly `k` highways while maximizing the total toll collected along the route. There are two important restrictions:

1. The trip may start from any city.
2. Each city may be visited at most once.

Because each city can only appear once in the trip, the path must be a simple path with exactly `k` edges. Since a path with `k` edges contains `k + 1` distinct cities, the problem is fundamentally asking us to search for the highest-cost simple path of length exactly `k`.

The input consists of:

- `n`, the number of cities
- `highways`, where each element is `[u, v, toll]`
- `k`, the exact number of highways that must be used

The expected output is:

- The maximum possible total toll among all valid trips
- `-1` if no valid trip exists

The constraints are the key to determining the correct algorithmic approach:

- `n <= 15`
- `highways.length <= 50`
- `k <= 50`

The small value of `n` is extremely important. Although `k` can be as large as `50`, a simple path cannot contain more than `n - 1` edges because cities cannot repeat. Since `n <= 15`, the actual meaningful path length is at most `14`.

This immediately suggests that exponential algorithms over subsets of cities may be feasible. Specifically, bitmask dynamic programming becomes attractive because the total number of subsets is:

```
2^15 = 32768
```

which is manageable.

There are several important edge cases:

- The graph may be disconnected, meaning no path of length `k` exists.
- `k` may exceed `n - 1`, which automatically makes the answer impossible.
- Highways may have toll `0`, so maximizing cost is not equivalent to maximizing path length.
- The best path may start from any city, not necessarily city `0`.
- Since the graph is undirected, every edge must be added in both directions.

A naive DFS that explores every possible path can become prohibitively expensive, especially when many cities are connected. We therefore need a more structured dynamic programming solution.

## Approaches

### Brute Force Approach

A straightforward solution is to try every possible simple path of length exactly `k`.

We can start DFS from every city. During DFS:

- Keep track of visited cities
- Keep track of current total cost
- Stop when exactly `k` edges have been used
- Update the global maximum answer

This works because DFS explores every valid simple path. Since every possible trip is checked, the maximum valid cost will eventually be found.

However, this approach becomes expensive because the number of simple paths grows exponentially. In the worst case, if the graph is dense, each city may branch into many unvisited neighbors.

The worst-case complexity is approximately:

```
O(n!)
```

or more precisely:

```
O(n * 2^n * n)
```

depending on how the search tree is analyzed.

Although `n = 15` is relatively small, unrestricted DFS still performs redundant computations. The same subset-state combinations get recomputed many times.

### Optimal Approach, Bitmask Dynamic Programming

The key observation is that the only information needed to continue building a path is:

- Which cities have already been visited
- Which city we are currently at

This naturally leads to state compression dynamic programming.

We define:

```
dp[mask][u]
```

as the maximum toll cost of a path that:

- Visits exactly the cities represented in `mask`
- Ends at city `u`

The bitmask compactly stores visited cities. Since `n <= 15`, we can represent every subset using an integer.

From state `(mask, u)`, we attempt to extend the path to an unvisited neighbor `v`.

Transition:

```
new_mask = mask | (1 << v)

dp[new_mask][v] =
    max(dp[new_mask][v],
        dp[mask][u] + weight(u, v))
```

At the end, we examine all states whose number of visited cities is exactly `k + 1`, because a path with `k` edges contains `k + 1` vertices.

This avoids recomputing identical subproblems and efficiently explores all valid simple paths.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n!) | O(n) | Explores every simple path recursively |
| Optimal | O(2^n * n^2) | O(2^n * n) | Uses bitmask DP to reuse overlapping states |

## Algorithm Walkthrough

1. Build an adjacency list for the graph.

Since the highways are undirected, every edge must be inserted in both directions. The adjacency list allows efficient traversal of neighboring cities.
2. Handle impossible cases early.

A simple path with `k` edges requires `k + 1` distinct cities. If `k >= n`, the answer is immediately `-1`.
3. Create the DP table.

Define:

```
dp[mask][u]
```

where:

- `mask` is the set of visited cities
- `u` is the current ending city
- value = maximum toll collected so far

Initialize every state to `-1` to represent unreachable states.
4. Initialize starting positions.

Since the trip may begin at any city, initialize:

```
dp[1 << i][i] = 0
```

for every city `i`.

A path containing only one city has cost `0`.
5. Iterate through all masks.

For every subset of visited cities:

- For every possible ending city `u`
- Skip invalid states where `dp[mask][u] == -1`
6. Try extending the path.

For each neighbor `v` of `u`:

- If `v` is already visited, skip it
- Otherwise create:

```
new_mask = mask | (1 << v)
```

Update:

```
dp[new_mask][v] =
    max(dp[new_mask][v],
        dp[mask][u] + toll)
```
7. Check path length.

A path using exactly `k` highways contains exactly `k + 1` cities.

Therefore, whenever:

```
popcount(mask) == k + 1
```

we consider the state as a candidate answer.
8. Return the maximum answer found.

If no valid path exists, return `-1`.

### Why it works

The DP state uniquely captures all information necessary to continue constructing a valid simple path:

- The visited set ensures no city repeats.
- The ending city determines which highways may be taken next.

Every valid simple path corresponds to exactly one DP state progression, and every DP transition preserves correctness by extending paths only to unvisited cities. Since all subsets and endpoints are explored, the algorithm guarantees that the maximum valid trip cost is found.

## Python Solution

```python
from typing import List

class Solution:
    def maximumCost(self, n: int, highways: List[List[int]], k: int) -> int:
        if k >= n:
            return -1

        graph = [[] for _ in range(n)]

        for u, v, toll in highways:
            graph[u].append((v, toll))
            graph[v].append((u, toll))

        size = 1 << n

        dp = [[-1] * n for _ in range(size)]

        for city in range(n):
            dp[1 << city][city] = 0

        answer = -1

        for mask in range(size):
            visited_count = mask.bit_count()

            if visited_count > k + 1:
                continue

            for u in range(n):
                if dp[mask][u] == -1:
                    continue

                current_cost = dp[mask][u]

                if visited_count == k + 1:
                    answer = max(answer, current_cost)
                    continue

                for v, toll in graph[u]:
                    if mask & (1 << v):
                        continue

                    new_mask = mask | (1 << v)

                    dp[new_mask][v] = max(
                        dp[new_mask][v],
                        current_cost + toll
                    )

        return answer
```

The implementation begins by handling the impossible scenario where `k >= n`. Since cities cannot repeat, a path cannot contain more than `n - 1` edges.

Next, the graph is converted into an adjacency list. Each highway is inserted twice because the graph is undirected.

The DP table stores the maximum toll cost for every combination of visited cities and ending city. All entries begin as `-1`, indicating unreachable states.

The initialization phase treats every city as a potential starting point. A single-city path has cost `0`.

The main loop iterates through every subset mask. For each reachable state:

- If the path already contains `k + 1` cities, update the answer.
- Otherwise, attempt to extend the path to every unvisited neighbor.

The bitmask efficiently tracks visited cities using bit operations, which keeps transitions fast and memory-efficient.

Finally, the algorithm returns the maximum valid cost found, or `-1` if no such path exists.

## Go Solution

```go
package main

func maximumCost(n int, highways [][]int, k int) int {
	if k >= n {
		return -1
	}

	type Edge struct {
		to   int
		cost int
	}

	graph := make([][]Edge, n)

	for _, h := range highways {
		u, v, toll := h[0], h[1], h[2]

		graph[u] = append(graph[u], Edge{v, toll})
		graph[v] = append(graph[v], Edge{u, toll})
	}

	size := 1 << n

	dp := make([][]int, size)

	for i := range dp {
		dp[i] = make([]int, n)

		for j := range dp[i] {
			dp[i][j] = -1
		}
	}

	for city := 0; city < n; city++ {
		dp[1<<city][city] = 0
	}

	answer := -1

	for mask := 0; mask < size; mask++ {
		visitedCount := popcount(mask)

		if visitedCount > k+1 {
			continue
		}

		for u := 0; u < n; u++ {
			if dp[mask][u] == -1 {
				continue
			}

			currentCost := dp[mask][u]

			if visitedCount == k+1 {
				if currentCost > answer {
					answer = currentCost
				}
				continue
			}

			for _, edge := range graph[u] {
				v := edge.to
				toll := edge.cost

				if mask&(1<<v) != 0 {
					continue
				}

				newMask := mask | (1 << v)

				newCost := currentCost + toll

				if newCost > dp[newMask][v] {
					dp[newMask][v] = newCost
				}
			}
		}
	}

	return answer
}

func popcount(x int) int {
	count := 0

	for x > 0 {
		x &= (x - 1)
		count++
	}

	return count
}
```

The Go implementation follows the same structure as the Python solution but uses slices instead of lists.

Since Go does not provide a built-in `bit_count()` method for integers, a custom `popcount` helper function is implemented using Brian Kernighan’s algorithm.

The DP table is initialized manually because Go slices default to `0`, while we need `-1` to indicate unreachable states.

Integer overflow is not a concern because the maximum possible toll is small:

```go
14 edges * 100 toll = 1400
```

which easily fits inside a standard Go `int`.

## Edge Cases

### Case 1, `k` Larger Than Maximum Possible Path Length

A simple path cannot revisit cities. Therefore, with `n` cities, the longest possible simple path contains `n - 1` edges.

If `k >= n`, no valid trip can exist.

This is easy to overlook because the problem allows `k` up to `50`, while `n` is only `15`. The implementation handles this immediately with:

```
if k >= n:
    return -1
```

which prevents unnecessary computation.

### Case 2, Disconnected Components

The graph may contain multiple disconnected regions. Even if the graph has enough total cities, there may not exist a connected simple path of length `k`.

For example:

```
0 -- 1

2 -- 3
```

cannot produce a path of length `2`.

The DP naturally handles this because unreachable states remain `-1`. No special connectivity algorithm is required.

### Case 3, Revisiting Cities

A common bug is accidentally allowing cycles.

For example:

```
0 -> 1 -> 0
```

would incorrectly appear valid if visited tracking is missing.

The bitmask guarantees correctness because before moving to a neighbor `v`, the algorithm checks:

```
if mask & (1 << v):
    continue
```

which prevents revisiting any city already included in the path.
