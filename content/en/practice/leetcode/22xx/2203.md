**Difficulty:** 🔴 Hard  
**Topics:** Graph Theory, Heap (Priority Queue), Shortest Path

## Solution

## Problem Understanding

The problem asks us to find the minimum-weight subgraph of a weighted directed graph such that two given source nodes, `src1` and `src2`, can both reach a common destination node `dest`. The input consists of the number of nodes `n`, a list of weighted directed edges `edges`, and the three distinct nodes `src1`, `src2`, and `dest`. Each edge `[from, to, weight]` represents a directed connection from node `from` to node `to` with a positive weight.

The goal is to select a subset of the edges (a subgraph) such that there exists a path from `src1` to `dest` and from `src2` to `dest`, and the sum of the weights of the edges included in the subgraph is minimized. If no such subgraph exists, we must return `-1`.

Constraints tell us the graph can be large: up to `10^5` nodes and `10^5` edges, which rules out any solution with superlinear time complexity in the number of nodes or edges. Edge weights are positive, so shortest path algorithms like Dijkstra can be safely applied.

Important edge cases include disconnected graphs, graphs where one source cannot reach the destination, or graphs where the optimal paths from `src1` and `src2` overlap only partially.

## Approaches

A brute-force approach would try every possible subgraph and check whether `src1` and `src2` can reach `dest`. This is impractical because the number of possible subgraphs grows exponentially with the number of edges. Even attempting all paths from `src1` and `src2` independently and combining them would be too slow due to the graph size.

The key insight is that we do not need to consider all subgraphs; instead, we can model this as a shortest path problem from three perspectives: from `src1`, from `src2`, and to `dest` (using a reversed graph). For each node `x`, the potential minimum subgraph that allows paths from both sources to the destination can include the paths `src1 -> x`, `src2 -> x`, and `x -> dest`. Summing these gives the total weight of a candidate subgraph if `x` serves as a meeting point. By evaluating all nodes as potential meeting points, we can find the minimal total weight efficiently using Dijkstra's algorithm.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^E) | O(E) | Examine all subsets of edges; infeasible for large graphs |
| Optimal | O(E log N) | O(N + E) | Use three Dijkstra runs: `src1`, `src2`, and `dest` (reversed graph) and combine distances |

## Algorithm Walkthrough

1. **Graph Construction**: Build a standard adjacency list for the original graph and a reversed adjacency list for computing distances to the destination. This allows efficient traversal for Dijkstra in both forward and backward directions.
2. **Dijkstra from `src1`**: Compute the shortest distance from `src1` to every other node using a priority queue. Store these distances in `dist_from_src1`.
3. **Dijkstra from `src2`**: Similarly, compute the shortest distance from `src2` to every other node, storing them in `dist_from_src2`.
4. **Dijkstra to `dest` (reverse graph)**: On the reversed graph, compute the shortest distance from `dest` to every other node. This effectively gives the shortest distance from each node to `dest` in the original graph, stored in `dist_to_dest`.
5. **Combine Distances**: For each node `x` from `0` to `n-1`, compute the sum `dist_from_src1[x] + dist_from_src2[x] + dist_to_dest[x]`. Keep track of the minimum sum encountered where all three distances are finite (paths exist).
6. **Return Result**: If the minimum sum is finite, return it as the minimum subgraph weight. Otherwise, return `-1` if no such subgraph exists.

**Why it works**: The algorithm works because any subgraph that allows both sources to reach the destination must have a “meeting point” node `x` that lies on the paths from both sources to the destination. By considering all nodes as potential meeting points and computing shortest paths efficiently, we ensure that we find the minimum-weight combination. Dijkstra's correctness guarantees that the shortest path distances are accurate and sumable.

## Python Solution

```python
import heapq
from typing import List

class Solution:
    def minimumWeight(self, n: int, edges: List[List[int]], src1: int, src2: int, dest: int) -> int:
        def dijkstra(start: int, graph: List[List[List[int]]]) -> List[int]:
            dist = [float('inf')] * n
            dist[start] = 0
            heap = [(0, start)]
            while heap:
                cost, node = heapq.heappop(heap)
                if cost > dist[node]:
                    continue
                for nei, w in graph[node]:
                    if dist[nei] > cost + w:
                        dist[nei] = cost + w
                        heapq.heappush(heap, (dist[nei], nei))
            return dist
        
        graph = [[] for _ in range(n)]
        rev_graph = [[] for _ in range(n)]
        for u, v, w in edges:
            graph[u].append([v, w])
            rev_graph[v].append([u, w])
        
        dist_from_src1 = dijkstra(src1, graph)
        dist_from_src2 = dijkstra(src2, graph)
        dist_to_dest = dijkstra(dest, rev_graph)
        
        min_weight = float('inf')
        for i in range(n):
            total = dist_from_src1[i] + dist_from_src2[i] + dist_to_dest[i]
            if total < min_weight:
                min_weight = total
        
        return min_weight if min_weight != float('inf') else -1
```

**Explanation**: We first construct both the original and reversed graphs. The `dijkstra` function computes shortest paths from a start node to all other nodes using a priority queue. We compute distances from `src1`, `src2`, and to `dest` using the reversed graph. Finally, we combine the distances for each node to find the minimal total weight of a subgraph connecting both sources to the destination.

## Go Solution

```go
package main

import (
    "container/heap"
    "math"
)

type Edge struct {
    to, weight int
}

type Item struct {
    node, cost int
}

type PriorityQueue []Item

func (pq PriorityQueue) Len() int { return len(pq) }
func (pq PriorityQueue) Less(i, j int) bool { return pq[i].cost < pq[j].cost }
func (pq PriorityQueue) Swap(i, j int) { pq[i], pq[j] = pq[j], pq[i] }
func (pq *PriorityQueue) Push(x any) { *pq = append(*pq, x.(Item)) }
func (pq *PriorityQueue) Pop() any {
    old := *pq
    n := len(old)
    item := old[n-1]
    *pq = old[0 : n-1]
    return item
}

func dijkstra(n int, graph [][]Edge, start int) []int64 {
    dist := make([]int64, n)
    for i := range dist { dist[i] = math.MaxInt64 }
    dist[start] = 0
    pq := &PriorityQueue{}
    heap.Init(pq)
    heap.Push(pq, Item{node: start, cost: 0})
    
    for pq.Len() > 0 {
        cur := heap.Pop(pq).(Item)
        if int64(cur.cost) > dist[cur.node] {
            continue
        }
        for _, e := range graph[cur.node] {
            newDist := dist[cur.node] + int64(e.weight)
            if newDist < dist[e.to] {
                dist[e.to] = newDist
                heap.Push(pq, Item{node: e.to, cost: int(newDist)})
            }
        }
    }
    return dist
}

func minimumWeight(n int, edges [][]int, src1 int, src2 int, dest int) int64 {
    graph := make([][]Edge, n)
    revGraph := make([][]Edge, n)
    for _, e := range edges {
        u, v, w := e[0], e[1], e[2]
        graph[u] = append(graph[u], Edge{to: v, weight: w})
        revGraph[v] = append(revGraph[v], Edge{to: u, weight: w})
    }

    dist1 := dijkstra(n, graph, src1)
    dist2 := dijkstra(n, graph, src2)
    distToDest := dijkstra(n, revGraph, dest)
    
    minWeight := int64(math.MaxInt64)
    for i := 0; i < n; i++ {
        total := dist1[i] + dist2[i] + distToDest[i]
        if total < minWeight {
            minWeight = total
        }
    }
    if minWeight == math.MaxInt64 {
        return -1
    }
    return minWeight
}
```

**Go Notes**: We use a min-heap with a priority queue implementation, and int64 for distances to avoid overflow. The reversed graph allows computing distances to `dest` efficiently.
