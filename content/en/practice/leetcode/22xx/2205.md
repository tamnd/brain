## Solution

## Problem Understanding

The problem asks us to determine how many users are eligible for a discount based on their purchase history. Each user can have multiple purchases recorded in the `Purchases` table, with each purchase having a timestamp and an amount. A user is eligible if **at least one of their purchases falls within a given date range `[startDate, endDate]` and the amount for that purchase is at least `minAmount`.**

The input is a database table `Purchases` with columns `user_id`, `time_stamp`, and `amount`. The `user_id` identifies the user, `time_stamp` records the date and time of the purchase, and `amount` indicates the purchase amount. The output should be a single integer representing the number of unique users that meet the eligibility criteria.

Key considerations include that the dates in the query are interpreted as the **start of the day**. Therefore, purchases occurring exactly at `endDate 00:00:00` are included. There are no constraints on the number of users or purchases, but since this is an easy database problem, the solution should scale reasonably to several thousand records.

Important edge cases include users with multiple purchases, purchases exactly at the start or end date, users with all purchases below `minAmount`, and a table with no qualifying purchases at all.

## Approaches

The brute-force approach is straightforward: iterate over all rows in the `Purchases` table, check for each purchase if it falls within the date range and meets the minimum amount, and then collect the distinct users that satisfy these conditions. This approach is correct but may require scanning all rows and maintaining a list of unique users, which could be inefficient for very large datasets.

The optimal approach leverages SQL's filtering and aggregation capabilities. Instead of scanning all purchases manually, we filter purchases directly using a `WHERE` clause for the date range and `amount >= minAmount`. Then we use `COUNT(DISTINCT user_id)` to efficiently count only the unique eligible users.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(u) | Scan all purchases and maintain a set of eligible users, where n is the number of rows and u is the number of unique users |
| Optimal | O(n) | O(1) | Let the database handle filtering and aggregation with `COUNT(DISTINCT user_id)` for maximum efficiency |

## Algorithm Walkthrough

1. Filter the `Purchases` table to select only those rows where `time_stamp` is between `startDate` and `endDate`. Use the `>=` and `<=` operators for inclusivity.
2. Further filter these rows to retain only purchases where `amount >= minAmount`.
3. From the filtered results, identify the unique `user_id` values. SQL's `COUNT(DISTINCT user_id)` function handles this aggregation efficiently.
4. Return the count as `user_cnt`.

**Why it works:** By filtering purchases by both date and amount, we ensure only eligible purchases are considered. Counting distinct users guarantees that multiple qualifying purchases by the same user do not inflate the result. The database engine handles indexing and optimization, ensuring correct and efficient computation.

## Python Solution

```python
import sqlite3
from typing import List, Tuple

def count_eligible_users(connection: sqlite3.Connection, startDate: str, endDate: str, minAmount: int) -> int:
    cursor = connection.cursor()
    query = """
    SELECT COUNT(DISTINCT user_id) AS user_cnt
    FROM Purchases
    WHERE time_stamp >= ? AND time_stamp <= ?
      AND amount >= ?
    """
    cursor.execute(query, (startDate, endDate, minAmount))
    result = cursor.fetchone()
    return result[0] if result else 0
```

The Python implementation uses parameterized SQL queries to avoid SQL injection and ensure correct type handling. We filter the `Purchases` table by `time_stamp` and `amount`, then count the distinct `user_id` values. The result is returned as an integer.

## Go Solution

```go
package main

import (
    "database/sql"
    _ "github.com/mattn/go-sqlite3"
    "log"
)

func CountEligibleUsers(db *sql.DB, startDate string, endDate string, minAmount int) int {
    query := `
    SELECT COUNT(DISTINCT user_id) AS user_cnt
    FROM Purchases
    WHERE time_stamp >= ? AND time_stamp <= ?
      AND amount >= ?
    `
    var userCnt int
    err := db.QueryRow(query, startDate, endDate, minAmount).Scan(&userCnt)
    if err != nil {
        log.Fatal(err)
    }
    return userCnt
}
```

The Go solution mirrors the Python solution. The main differences are the use of `QueryRow` and `Scan` for reading a single value, and explicit error handling to ensure any database issue is caught.

## Edge Cases

One important edge case is when all purchases are below `minAmount`. This tests whether the filtering logic correctly excludes ineligible users and returns zero.

Another edge case is when multiple purchases by the same user qualify. The distinct count must ensure the user is only counted once, preventing overcounting.

A third edge case is purchases exactly on `startDate` or `endDate`. Our implementation handles this inclusively by using `>= startDate` and `<= endDate`, ensuring these boundary values are included as intended.
