## Solution

## Problem Understanding

The problem gives us a collection of stock price points, where each point is represented as `[day, price]`. The `day` value is the x-coordinate and the `price` value is the y-coordinate on a graph. After plotting all points, adjacent points are connected in chronological order to form a line chart.

Our task is to determine the minimum number of straight line segments needed to represent the entire chart.

The key detail is that multiple consecutive segments can belong to the same line if they all have the same slope. In other words, if three or more consecutive points are collinear, they can all be represented using one continuous line segment instead of several smaller ones.

For example, consider these points:

```
(1,7), (2,6), (3,5), (4,4)
```

Each consecutive pair has slope `-1`, so all four points lie on the same straight line. Therefore, they contribute only one line to the answer.

However, if the slope changes at some point, a new line segment must begin.

One important detail is that the input array is not guaranteed to be sorted by day. Before analyzing slopes, we must sort the points chronologically.

The constraints are large:

- Up to `10^5` points
- Coordinate values up to `10^9`

These constraints immediately rule out expensive geometric approaches. We need an algorithm close to `O(n log n)` time, where sorting dominates the complexity.

There are also several important edge cases:

- If there is only one point, no line is needed because a single point does not form a segment.
- Multiple consecutive points may have identical slopes and should be merged into one line.
- Slopes must be compared carefully to avoid floating point precision errors.
- Coordinate values are large, so multiplication during slope comparison can overflow in some languages unless handled properly.

The problem guarantees that all days are distinct, which means there will never be vertical lines with division by zero concerns after sorting.

## Approaches

### Brute Force Approach

A brute force approach would examine every possible maximal segment of collinear points.

After sorting the points by day, we could repeatedly attempt to extend a line from point `i` through all subsequent points while checking whether each point lies on the same line. Whenever the slope changes, we would start a new segment.

One naive implementation might compute slopes using floating point division:

```
(price2 - price1) / (day2 - day1)
```

and compare them directly.

Although this works conceptually, floating point arithmetic introduces precision issues. Two mathematically identical slopes may not compare equal because of rounding behavior.

Additionally, repeatedly recomputing slopes for large groups of points can become inefficient.

### Optimal Approach

The key observation is that we only need to compare slopes between consecutive pairs of points.

Suppose we have three consecutive points:

```
A, B, C
```

If:

```
slope(A,B) == slope(B,C)
```

then all three points lie on the same line segment.

Instead of performing division, we compare slopes using cross multiplication:

```
(y2 - y1) * (x3 - x2) == (y3 - y2) * (x2 - x1)
```

This avoids floating point precision problems entirely.

The algorithm becomes:

1. Sort points by day.
2. Start with one line segment if at least two points exist.
3. Traverse consecutive pairs.
4. Whenever the slope changes, increment the answer.

This is optimal because each point is processed only once after sorting.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Repeatedly checks collinearity across many ranges |
| Optimal | O(n log n) | O(1) or O(log n) | Sort once, then compare consecutive slopes |

## Algorithm Walkthrough

1. First, sort the `stockPrices` array by the `day` value.

The line chart connects points in chronological order, not in the order given by the input. Sorting ensures adjacent points represent consecutive days on the chart.
2. Handle the smallest edge case.

If there is only one point, return `0` because a single point does not require any line segment.
3. Initialize the answer to `1`.

Once there are at least two points, at least one line segment is always necessary.
4. Compute the slope between the first two points.

Instead of storing the slope as a floating point value, store the differences:

```
dx = x2 - x1
dy = y2 - y1
```
5. Iterate through the remaining points starting from index `2`.

For each new point, compute the slope between the current point and the previous point.
6. Compare the current slope with the previous slope using cross multiplication.

Suppose:

```
previous slope = dy1 / dx1
current slope = dy2 / dx2
```

They are equal if:

```
dy1 * dx2 == dy2 * dx1
```
7. If the slopes differ, increment the answer.

A slope change means the previous line can no longer continue, so a new line segment must begin.
8. Update the stored slope values and continue processing.
9. Return the final answer after processing all points.

### Why it works

A straight line is uniquely determined by its slope. Consecutive chart segments belong to the same line exactly when their slopes are equal. By scanning the sorted points and counting how many times the slope changes, we partition the chart into the minimum number of maximal collinear groups. Each group corresponds to one line segment, which guarantees correctness.

## Python Solution

```python
from typing import List

class Solution:
    def minimumLines(self, stockPrices: List[List[int]]) -> int:
        n = len(stockPrices)

        if n == 1:
            return 0

        stockPrices.sort()

        lines = 1

        prev_dx = stockPrices[1][0] - stockPrices[0][0]
        prev_dy = stockPrices[1][1] - stockPrices[0][1]

        for i in range(2, n):
            curr_dx = stockPrices[i][0] - stockPrices[i - 1][0]
            curr_dy = stockPrices[i][1] - stockPrices[i - 1][1]

            if prev_dy * curr_dx != curr_dy * prev_dx:
                lines += 1

            prev_dx = curr_dx
            prev_dy = curr_dy

        return lines
```

The implementation begins by checking whether only one point exists. Since a line segment requires at least two points, the answer is `0` in that case.

Next, the points are sorted by day. This is necessary because the chart connects points chronologically.

The variable `lines` starts at `1` because any dataset with at least two points requires at least one line segment.

The algorithm stores the slope of the previous segment using `dx` and `dy` instead of performing division. This prevents floating point precision issues.

During iteration, the current slope is compared with the previous slope using cross multiplication:

```python
prev_dy * curr_dx == curr_dy * prev_dx
```

If the equality fails, the slope changed and a new line segment is required.

Finally, the function returns the total number of line segments counted.

## Go Solution

```go
package main

import (
	"sort"
)

func minimumLines(stockPrices [][]int) int {
	n := len(stockPrices)

	if n == 1 {
		return 0
	}

	sort.Slice(stockPrices, func(i, j int) bool {
		return stockPrices[i][0] < stockPrices[j][0]
	})

	lines := 1

	prevDx := int64(stockPrices[1][0] - stockPrices[0][0])
	prevDy := int64(stockPrices[1][1] - stockPrices[0][1])

	for i := 2; i < n; i++ {
		currDx := int64(stockPrices[i][0] - stockPrices[i-1][0])
		currDy := int64(stockPrices[i][1] - stockPrices[i-1][1])

		if prevDy*currDx != currDy*prevDx {
			lines++
		}

		prevDx = currDx
		prevDy = currDy
	}

	return lines
}
```

The Go implementation follows the same logic as the Python solution.

One important difference is the use of `int64` during multiplication. Coordinate values can reach `10^9`, and multiplying two differences may exceed the range of a 32-bit integer. Using `int64` prevents overflow.

The input is sorted using `sort.Slice`, which provides a custom comparator for sorting by day.

## Edge Cases

A very important edge case is when the input contains only one point. Since a line segment requires two endpoints, no line is needed. A naive implementation that initializes the answer to `1` without checking this case would return the wrong result. The implementation explicitly checks `n == 1` and returns `0`.

Another important edge case involves floating point precision. Consider slopes like `1/3` and `2/6`. They represent the same mathematical slope, but floating point arithmetic may produce slightly different binary representations. Comparing floating point values directly can therefore produce incorrect answers. The implementation avoids division entirely and instead uses exact integer cross multiplication.

Large coordinate values are also a potential source of bugs. Differences between coordinates may reach `10^9`, and multiplying two such values produces numbers around `10^18`. In languages with fixed-width integers, this can overflow 32-bit arithmetic. The Go implementation uses `int64` to safely handle these calculations.

A final subtle edge case occurs when the input is not sorted. The problem statement does not guarantee chronological order. If we process points in their original order, slope comparisons become meaningless because adjacent input elements may not correspond to adjacent chart points. Sorting by day ensures the algorithm correctly reconstructs the chart before counting line segments.
