**Difficulty:** 🔴 Hard  
**Topics:** Array, Tree, Depth-First Search, Breadth-First Search

## Solution

## Problem Understanding

The problem gives us an undirected tree with `n` nodes numbered from `0` to `n - 1`. Because the graph is a tree, there is exactly one simple path between any pair of nodes.

For every query `[start, end, node]`, we must examine the unique path from `start` to `end`. Among all nodes on that path, we need to find the node whose distance to `node` is minimum.

The key detail is that we are not trying to find the closest node in the entire tree. We are only allowed to choose from the nodes that lie on the path between `start` and `end`.

The distance between two nodes in a tree is the number of edges on the unique path connecting them.

For example, suppose the path from `start` to `end` is:

```
5 -> 2 -> 0 -> 3
```

If `node = 4`, then we compute:

```
dist(4,5)
dist(4,2)
dist(4,0)
dist(4,3)
```

and choose the node with the smallest value.

The constraints are relatively small:

- `n <= 1000`
- `query.length <= 1000`

These limits are important because they allow preprocessing approaches with quadratic complexity. A solution with `O(n^2)` preprocessing and `O(1)` or `O(log n)` query answering is completely acceptable.

There are several important edge cases:

- The path may contain only one node, such as when `start == end`
- The queried `node` itself may lie on the path
- The closest node may be one of the endpoints
- The tree may be highly unbalanced, like a linked list
- Multiple nodes could appear equally close, but the tree structure guarantees a unique projection point onto the path, so ambiguity does not create correctness issues

Because the graph is guaranteed to be a valid tree, we never need to handle disconnected components or cycles.

## Approaches

### Brute Force Approach

The most direct solution is to process each query independently.

For every query `[start, end, node]`, we can:

1. Find the path from `start` to `end`
2. Iterate through every node on that path
3. Compute the distance from `node` to each candidate
4. Return the node with the minimum distance

To compute distances, we could run BFS from `node` every time, or precompute all-pairs shortest distances first.

Finding the path itself can also be done with DFS or parent tracking.

This approach is correct because it explicitly checks every valid candidate on the path and selects the best one.

However, it is inefficient if implemented naively. If each query requires traversing large portions of the tree and recomputing distances repeatedly, the total runtime becomes unnecessarily large.

### Key Insight

The crucial observation is a classic tree property:

> The node on the path from `start` to `end` closest to `node` is exactly the intersection point of the three paths:
> 
> 
> 
> 
> - `start -> end`
> - `start -> node`
> - `end -> node`

Another equivalent interpretation is:

> Among the LCAs:
> 
> 
> 
> 
> - `lca(start, end)`
> - `lca(start, node)`
> - `lca(end, node)`
> 
> 
> 
> 
> the deepest one is the answer.

Why does this work?

In a tree, the closest point from a node to a path is the unique projection of that node onto the path. That projection is precisely the deepest common ancestor relationship shared among these three paths.

This allows us to preprocess Lowest Common Ancestor information and answer each query efficiently.

Since `n <= 1000`, we can even use a simpler LCA implementation based on parent climbing rather than binary lifting.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(q * n) | O(n) | Finds path and checks every node on it |
| Optimal | O(n + q * h) | O(n) | Uses LCA relationships to directly compute answer |

Here:

- `q` is the number of queries
- `h` is the tree height
- In the worst case, `h = n`

Because constraints are small, even parent-climbing LCA is fast enough.

## Algorithm Walkthrough

### Step 1: Build the adjacency list

We first convert the edge list into an adjacency list representation.

An adjacency list is ideal for trees because:

- Traversing neighbors becomes efficient
- DFS and BFS become straightforward
- Space usage is linear in the number of edges

For every edge `[u, v]`, we add:

```
u -> v
v -> u
```

because the tree is undirected.

### Step 2: Run DFS to compute parent and depth

We root the tree at node `0`.

During DFS, we compute:

- `parent[node]`, the immediate parent
- `depth[node]`, the distance from the root

These arrays allow us to answer LCA queries later.

For example:

```
depth[0] = 0
depth[child] = depth[parent] + 1
```

### Step 3: Implement Lowest Common Ancestor

To compute `lca(u, v)`:

1. Move the deeper node upward until both nodes have the same depth
2. Move both upward together until they meet

The meeting point is the LCA.

Since the constraints are small, this simple implementation is sufficient.

### Step 4: Process each query

For query `[a, b, c]`, compute:

```
x = lca(a, b)
y = lca(a, c)
z = lca(b, c)
```

The answer is whichever of `x`, `y`, or `z` has the greatest depth.

Why?

Because the deepest of these LCAs represents the projection of `c` onto the path from `a` to `b`.

### Step 5: Store answers

Append the chosen node to the result list.

After processing all queries, return the list.

### Why it works

In a tree, every pair of nodes has exactly one simple path.

The node closest to `c` on the path from `a` to `b` is the unique branching point where the paths:

- `a -> c`
- `b -> c`
- `a -> b`

interact.

Among the three pairwise LCAs, the deepest one identifies this branching point precisely. Since depth measures closeness to the leaves, the deepest LCA corresponds to the actual projection of `c` onto the path.

Therefore, selecting the deepest among:

```
lca(a,b)
lca(a,c)
lca(b,c)
```

always produces the correct answer.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Solution:
    def closestNode(
        self,
        n: int,
        edges: List[List[int]],
        query: List[List[int]]
    ) -> List[int]:

        graph = defaultdict(list)

        for u, v in edges:
            graph[u].append(v)
            graph[v].append(u)

        parent = [-1] * n
        depth = [0] * n

        def dfs(node: int, par: int) -> None:
            parent[node] = par

            for neighbor in graph[node]:
                if neighbor == par:
                    continue

                depth[neighbor] = depth[node] + 1
                dfs(neighbor, node)

        dfs(0, -1)

        def lca(u: int, v: int) -> int:
            while depth[u] > depth[v]:
                u = parent[u]

            while depth[v] > depth[u]:
                v = parent[v]

            while u != v:
                u = parent[u]
                v = parent[v]

            return u

        answer = []

        for a, b, c in query:
            x = lca(a, b)
            y = lca(a, c)
            z = lca(b, c)

            best = x

            if depth[y] > depth[best]:
                best = y

            if depth[z] > depth[best]:
                best = z

            answer.append(best)

        return answer
```

The implementation begins by constructing the adjacency list representation of the tree.

Next, DFS computes the parent and depth arrays. The tree is rooted at node `0`, although any node would work because LCAs are independent of root choice once the structure is fixed.

The `lca()` function uses parent climbing. It first equalizes depths, then moves both nodes upward simultaneously until they meet.

For each query, the code computes the three LCAs:

- `lca(a, b)`
- `lca(a, c)`
- `lca(b, c)`

The deepest among them is selected using the `depth` array.

Finally, all answers are returned in order.

## Go Solution

```go
package main

func closestNode(n int, edges [][]int, query [][]int) []int {
	graph := make([][]int, n)

	for _, edge := range edges {
		u := edge[0]
		v := edge[1]

		graph[u] = append(graph[u], v)
		graph[v] = append(graph[v], u)
	}

	parent := make([]int, n)
	depth := make([]int, n)

	var dfs func(int, int)

	dfs = func(node int, par int) {
		parent[node] = par

		for _, neighbor := range graph[node] {
			if neighbor == par {
				continue
			}

			depth[neighbor] = depth[node] + 1
			dfs(neighbor, node)
		}
	}

	dfs(0, -1)

	lca := func(u int, v int) int {
		for depth[u] > depth[v] {
			u = parent[u]
		}

		for depth[v] > depth[u] {
			v = parent[v]
		}

		for u != v {
			u = parent[u]
			v = parent[v]
		}

		return u
	}

	answer := make([]int, 0, len(query))

	for _, q := range query {
		a := q[0]
		b := q[1]
		c := q[2]

		x := lca(a, b)
		y := lca(a, c)
		z := lca(b, c)

		best := x

		if depth[y] > depth[best] {
			best = y
		}

		if depth[z] > depth[best] {
			best = z
		}

		answer = append(answer, best)
	}

	return answer
}
```

The Go implementation closely mirrors the Python version.

Slices are used for adjacency lists, parent storage, and depths. The DFS is implemented with a recursive closure.

One small difference is that Go requires explicit slice initialization sizes, while Python lists grow dynamically.

There are no integer overflow concerns because node indices and depths are small.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n + q * h) | DFS preprocessing plus three LCA computations per query |
| Space | O(n) | Adjacency list, parent array, and depth array |

The DFS preprocessing traverses every edge once, so it requires `O(n)` time.

Each LCA operation may climb up the tree height `h`. Since each query performs three LCAs, query processing costs `O(h)` overall.

In the worst case, a skewed tree has height `n`, giving worst-case complexity `O(q * n)`. However, with `n <= 1000`, this is completely acceptable.

The space usage is linear because we store only adjacency information and a few arrays.

## Edge Cases

### Path contains only one node

When `start == end`, the path consists of exactly one node.

A buggy implementation might still attempt to traverse edges or compare multiple candidates unnecessarily.

This implementation handles the case naturally because:

```
lca(a,a) = a
```

and all three LCAs become identical.

### Highly unbalanced trees

A tree shaped like a linked list creates the maximum possible height.

This is important because the LCA implementation uses parent climbing rather than binary lifting.

Even in the worst case, the constraints are small enough that `O(n)` LCA operations remain efficient.

### Queried node already lies on the path

If `node` itself belongs to the path from `start` to `end`, then the answer must be `node` because the distance is zero.

The LCA property automatically captures this situation. The deepest LCA becomes the queried node itself, so the implementation returns it correctly without requiring special handling.
