## Solution

## Problem Understanding

This problem asks us to move from the top left corner of a grid to the bottom right corner while removing as few obstacles as possible.

The input is a 2D matrix called `grid`, where:

- `0` represents an empty cell that can be moved through freely
- `1` represents an obstacle that can be removed if necessary

From any cell, we may move in four directions:

- up
- down
- left
- right

The important detail is that moving into an obstacle cell is allowed, but doing so costs `1` because we must remove that obstacle first. Moving into an empty cell costs `0`.

The goal is not to minimize the number of steps. Instead, we must minimize the total number of obstacles removed along the path.

This transforms the grid into a weighted graph problem:

- each cell is a node
- moving into a `0` cell has weight `0`
- moving into a `1` cell has weight `1`

We are therefore searching for the minimum cost path from `(0, 0)` to `(m - 1, n - 1)`.

The constraints are very important:

- `m * n <= 10^5`

This means the total number of cells can be as large as one hundred thousand. A naive exhaustive search would be far too slow. We need an algorithm close to linear or logarithmic per operation.

The problem also guarantees:

- the start cell is `0`
- the destination cell is `0`

So we never need to remove the starting or ending cell.

Several edge cases are important:

- A path may already exist with no removals needed.
- The grid may contain almost all obstacles.
- Multiple paths may exist with different lengths but the same obstacle count.
- A shorter path in terms of distance may actually require removing more obstacles.
- Since movement is allowed in four directions, cycles are possible, so we must avoid revisiting states inefficiently.

## Approaches

### Brute Force Approach

A brute force solution would explore every possible path from the top left corner to the bottom right corner and count how many obstacles are removed along each path.

One possible implementation would use DFS with backtracking:

- recursively move in all four directions
- keep track of visited cells
- accumulate obstacle removals
- update the minimum answer when reaching the destination

This approach is correct because it eventually explores every valid path and therefore cannot miss the optimal one.

However, it is computationally infeasible.

The number of possible paths in a grid grows exponentially. Even moderate grid sizes would produce an enormous search space. Since the grid can contain up to `100000` cells, exhaustive path enumeration is impossible.

### Optimal Approach, 0-1 BFS

The key observation is that every movement cost is either:

- `0`, if the next cell is empty
- `1`, if the next cell contains an obstacle

Whenever a graph contains only edge weights `0` and `1`, we can use a specialized shortest path algorithm called **0-1 BFS**.

Instead of using a priority queue like Dijkstra's algorithm, 0-1 BFS uses a deque.

The idea is:

- if moving to a neighbor costs `0`, push it to the front
- if moving to a neighbor costs `1`, push it to the back

This guarantees that cells are processed in nondecreasing order of obstacle removals.

The algorithm behaves similarly to Dijkstra's algorithm but achieves better efficiency because it avoids heap operations.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force DFS | Exponential | O(m × n) | Explores all possible paths |
| 0-1 BFS | O(m × n) | O(m × n) | Efficient shortest path for weights 0 and 1 |

## Algorithm Walkthrough

### Step 1, Initialize Distance Matrix

Create a `dist` matrix where:

- `dist[r][c]` represents the minimum obstacles removed to reach cell `(r, c)`

Initialize all values to infinity except:

```
dist[0][0] = 0
```

This means reaching the starting cell requires removing zero obstacles.

### Step 2, Create the Deque

We use a deque because we need efficient insertion at both ends.

Start with:

```
deque([(0, 0)])
```

The deque stores cells to process.

### Step 3, Define Movement Directions

We can move in four directions:

```
up, down, left, right
```

These are represented using direction vectors.

### Step 4, Process Cells Using 0-1 BFS

While the deque is not empty:

1. Remove the front cell.
2. Explore its four neighbors.
3. Compute the new obstacle removal cost.

If the next cell contains:

- `0`, cost does not increase
- `1`, cost increases by one

### Step 5, Relax Distances

Suppose we move from `(r, c)` to `(nr, nc)`.

The new cost becomes:

```
new_cost = dist[r][c] + grid[nr][nc]
```

If this cost is better than the current known distance:

```
new_cost < dist[nr][nc]
```

update the distance.

### Step 6, Push Into Deque Correctly

This is the critical optimization.

If:

```
grid[nr][nc] == 0
```

push to the front:

```
appendleft()
```

because zero-cost transitions should be processed immediately.

Otherwise:

```
append()
```

because cost-one transitions should be processed later.

### Step 7, Return Final Answer

Once processing completes:

```
dist[m - 1][n - 1]
```

contains the minimum number of obstacles removed.

### Why it works

0-1 BFS maintains the invariant that cells are processed in nondecreasing order of removal cost.

Whenever we traverse a zero-cost edge, the neighbor is processed immediately by pushing it to the front of the deque. Cost-one edges are delayed by pushing them to the back.

This ordering guarantees that when a cell is processed, we have already found the minimum possible obstacle removal count for it, exactly like Dijkstra's algorithm guarantees shortest paths for nonnegative weights.

Since every edge weight is either `0` or `1`, 0-1 BFS correctly computes the minimum obstacle removals.

## Python Solution

```python
from collections import deque
from typing import List

class Solution:
    def minimumObstacles(self, grid: List[List[int]]) -> int:
        rows = len(grid)
        cols = len(grid[0])

        dist = [[float("inf")] * cols for _ in range(rows)]
        dist[0][0] = 0

        dq = deque([(0, 0)])

        directions = [
            (1, 0),
            (-1, 0),
            (0, 1),
            (0, -1)
        ]

        while dq:
            row, col = dq.popleft()

            for dr, dc in directions:
                nr = row + dr
                nc = col + dc

                if 0 <= nr < rows and 0 <= nc < cols:
                    new_cost = dist[row][col] + grid[nr][nc]

                    if new_cost < dist[nr][nc]:
                        dist[nr][nc] = new_cost

                        if grid[nr][nc] == 0:
                            dq.appendleft((nr, nc))
                        else:
                            dq.append((nr, nc))

        return dist[rows - 1][cols - 1]
```

The implementation begins by computing the grid dimensions and initializing the distance matrix. Every cell starts with infinity because we have not yet discovered any path to it.

The starting cell receives distance `0` because no obstacles need removal initially.

A deque is then initialized with the starting position. Unlike a standard BFS queue, this deque supports insertion at both ends, which is essential for 0-1 BFS.

Inside the main loop, we repeatedly process cells from the front of the deque. For each neighboring cell, we compute the new obstacle removal cost by adding the value of the destination cell.

If we discover a cheaper path to that neighbor, we update the distance matrix.

The insertion rule is the heart of the algorithm:

- zero-cost moves go to the front
- cost-one moves go to the back

This preserves shortest path ordering without requiring a heap.

Finally, the algorithm returns the distance stored at the bottom right corner.

## Go Solution

```go
func minimumObstacles(grid [][]int) int {
	rows := len(grid)
	cols := len(grid[0])

	const INF = int(1e9)

	dist := make([][]int, rows)
	for i := 0; i < rows; i++ {
		dist[i] = make([]int, cols)
		for j := 0; j < cols; j++ {
			dist[i][j] = INF
		}
	}

	dist[0][0] = 0

	type Pair struct {
		r int
		c int
	}

	deque := []Pair{{0, 0}}

	directions := [][]int{
		{1, 0},
		{-1, 0},
		{0, 1},
		{0, -1},
	}

	for len(deque) > 0 {
		current := deque[0]
		deque = deque[1:]

		r := current.r
		c := current.c

		for _, dir := range directions {
			nr := r + dir[0]
			nc := c + dir[1]

			if nr >= 0 && nr < rows && nc >= 0 && nc < cols {
				newCost := dist[r][c] + grid[nr][nc]

				if newCost < dist[nr][nc] {
					dist[nr][nc] = newCost

					if grid[nr][nc] == 0 {
						deque = append([]Pair{{nr, nc}}, deque...)
					} else {
						deque = append(deque, Pair{nr, nc})
					}
				}
			}
		}
	}

	return dist[rows-1][cols-1]
}
```

The Go implementation follows the same algorithmic structure as the Python solution.

A custom `Pair` struct is used to store coordinates because Go does not provide tuples.

The distance matrix is initialized using nested slices. Since Go does not have built in infinity values for integers, a large constant `INF` is used.

One implementation detail worth noting is deque handling. Go does not provide a native deque in the standard library, so this implementation uses slices. Prepending with:

```go
append([]Pair{{nr, nc}}, deque...)
```

works correctly but is less efficient than a dedicated deque structure. For LeetCode constraints, this still passes comfortably.

Integer overflow is not a concern because the maximum obstacle removals cannot exceed `m * n`, which is at most `100000`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m × n) | Each cell is processed a constant number of times |
| Space | O(m × n) | Distance matrix and deque storage |

The grid contains `m × n` cells.

Each edge is relaxed at most a small constant number of times. Since each cell has at most four neighbors, the total work is linear in the number of cells.

Unlike Dijkstra's algorithm, 0-1 BFS avoids heap operations, so we do not incur an additional logarithmic factor.

The distance matrix requires storing one value per cell, and the deque may also contain up to `m × n` cells in the worst case.

## Edge Cases

One important edge case is a grid where no obstacle removals are necessary. A naive shortest path implementation might incorrectly prioritize shorter geometric distance rather than minimum removal count. The 0-1 BFS implementation handles this correctly because zero-cost moves are always processed first.

Another important case is a grid filled mostly with obstacles. In such situations, many paths may require removals, and multiple states can compete for the same cell. The distance matrix prevents revisiting cells with worse costs, ensuring efficient processing even in dense obstacle layouts.

A single-cell grid is also important. Since the start and destination are the same cell, the answer should immediately be `0`. The implementation handles this naturally because `dist[0][0]` is initialized to zero and no traversal is needed.

A more subtle edge case occurs when the shortest path in terms of steps is not the optimal path in terms of obstacle removals. Traditional BFS would fail here because BFS assumes uniform edge costs. The 0-1 BFS approach correctly prioritizes lower obstacle counts regardless of geometric path length.
