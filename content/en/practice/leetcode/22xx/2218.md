[LeetCode Problem 2218](https://leetcode.com/problems/maximum-value-of-k-coins-from-piles/)

**Difficulty:** 🔴 Hard  
**Topics:** Array, Dynamic Programming, Prefix Sum

## Solution

## Problem Understanding

The problem gives us several piles of coins, where each pile is represented from top to bottom. We may only remove coins from the top of a pile, which means we cannot skip coins in the middle. If we want the third coin in a pile, we must also take the first and second coins above it.

Our goal is to pick exactly `k` coins total across all piles while maximizing the total value collected.

For example, suppose a pile is:

```
[1, 100, 3]
```

If we want to take the coin worth `100`, we are forced to first take the top coin worth `1`. Therefore:

- Taking 0 coins gives value `0`
- Taking 1 coin gives value `1`
- Taking 2 coins gives value `101`
- Taking 3 coins gives value `104`

This structure is important because each pile behaves like a sequence of prefix choices.

The input consists of:

- `piles`, a list of coin piles
- `k`, the exact number of coins we must collect

The output is a single integer, the maximum total value achievable.

The constraints are critical for determining the correct algorithm:

- Up to `1000` piles
- Total number of coins across all piles is at most `2000`
- `k <= 2000`

A brute-force recursive search would explode combinatorially because every pile can contribute multiple possible counts of coins. However, the relatively small total number of coins suggests that dynamic programming is appropriate.

Several edge cases are important:

- A pile may contain only one coin
- `k` may equal the total number of coins, forcing us to take everything
- The optimal solution may involve taking many coins from one pile and none from others
- Large values deeper in a pile may require taking smaller prefix values first
- Multiple piles may compete for the same limited coin budget

The problem guarantees that `k` is valid, meaning it never exceeds the total number of available coins.

## Approaches

### Brute Force

A straightforward approach is to recursively try every possible number of coins taken from each pile.

For each pile, we can choose:

- Take 0 coins
- Take 1 coin
- Take 2 coins
- And so on, up to the pile size

Then recursively solve the remaining piles with the remaining number of coins.

This approach is correct because it explores every valid combination of choices. Eventually, it will find the optimal one.

However, it is far too slow. If each pile has several possible choices, the total number of combinations grows exponentially. Even with memoization, recomputing prefix sums repeatedly would still be inefficient.

### Optimal Dynamic Programming

The key observation is that this problem is very similar to a grouped knapsack problem.

Each pile represents a group of choices:

- Take 0 coins for value 0
- Take 1 coin for some prefix sum
- Take 2 coins for another prefix sum
- And so on

For every pile, we must choose exactly one option from that group.

This naturally leads to dynamic programming.

We define:

```
dp[c] = maximum value achievable using exactly c coins
```

Then, for each pile, we compute all possible prefix sums and update the DP array.

Since the total number of coins is at most `2000`, an `O(k^2)` style DP is feasible.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | O(n) recursion | Tries all combinations of choices |
| Optimal | O(n * k * m) | O(k) | Dynamic programming with prefix sums |

Here:

- `n` is the number of piles
- `m` is the average pile size
- Since total coins ≤ 2000, the practical upper bound is about `O(k * totalCoins)`

## Algorithm Walkthrough

### Step 1: Initialize the DP Array

We create a DP array where:

```
dp[c] = maximum value obtainable using exactly c coins
```

Initially:

- `dp[0] = 0`
- All other states start at `0`

This represents that taking zero coins gives zero value.

### Step 2: Process Each Pile Independently

For every pile, we calculate prefix sums.

For example:

```
pile = [1, 100, 3]
prefix sums:
0 coins -> 0
1 coin  -> 1
2 coins -> 101
3 coins -> 104
```

This allows us to instantly know the value of taking the first `x` coins from a pile.

### Step 3: Create a New DP State

For each pile, we create a fresh DP array called `next_dp`.

This is important because while processing a pile, we must not reuse updated states from the same pile multiple times.

We are effectively performing grouped transitions.

### Step 4: Try All Coin Counts

For every possible current coin count `used`:

- Try taking `take` coins from the current pile
- Ensure `used + take <= k`
- Update:

```
next_dp[used + take]
=
max(
    next_dp[used + take],
    dp[used] + prefix[take]
)
```

This transition means:

- We already achieved value `dp[used]`
- We add `take` coins from the current pile
- The added value is the prefix sum

### Step 5: Replace DP

After finishing the pile:

```
dp = next_dp
```

Now the DP represents all optimal solutions considering piles processed so far.

### Step 6: Return the Answer

The final answer is:

```
dp[k]
```

because we need exactly `k` coins.

### Why it works

The DP invariant is:

```
After processing the first i piles,
dp[c] stores the maximum value obtainable
using exactly c coins.
```

For every pile, we consider every valid number of coins we could take from that pile. Since all possibilities are explored and transitions preserve optimal substructure, the algorithm guarantees the optimal final answer.

## Python Solution

```python
from typing import List

class Solution:
    def maxValueOfCoins(self, piles: List[List[int]], k: int) -> int:
        dp = [0] * (k + 1)

        for pile in piles:
            prefix = [0]

            current_sum = 0
            for coin in pile:
                current_sum += coin
                prefix.append(current_sum)

            next_dp = dp[:]

            for used in range(k + 1):
                for take in range(1, min(len(pile), k - used) + 1):
                    next_dp[used + take] = max(
                        next_dp[used + take],
                        dp[used] + prefix[take]
                    )

            dp = next_dp

        return dp[k]
```

The implementation follows the dynamic programming approach directly.

The `dp` array stores the best possible value for every exact coin count. Each pile is processed independently.

For every pile, we first compute prefix sums. This is essential because taking coins from a pile always means taking a prefix of that pile. Prefix sums let us retrieve the value of taking the first `x` coins in constant time.

The `next_dp` array prevents accidental reuse of states from the same pile. Without it, updates from the current pile could cascade incorrectly and effectively allow taking from the same pile multiple times.

The nested loops enumerate:

- Current number of used coins
- Number of coins taken from the current pile

Every valid transition updates the best achievable value.

Finally, `dp[k]` contains the optimal answer.

## Go Solution

```go
func maxValueOfCoins(piles [][]int, k int) int {
	dp := make([]int, k+1)

	for _, pile := range piles {
		prefix := make([]int, len(pile)+1)

		for i, coin := range pile {
			prefix[i+1] = prefix[i] + coin
		}

		nextDP := make([]int, k+1)
		copy(nextDP, dp)

		for used := 0; used <= k; used++ {
			limit := len(pile)
			if k-used < limit {
				limit = k - used
			}

			for take := 1; take <= limit; take++ {
				candidate := dp[used] + prefix[take]

				if candidate > nextDP[used+take] {
					nextDP[used+take] = candidate
				}
			}
		}

		dp = nextDP
	}

	return dp[k]
}
```

The Go implementation mirrors the Python solution closely.

Slices are used for the DP arrays and prefix sums. Since Go slices are mutable references, we explicitly allocate `nextDP` and copy the old values into it to avoid overwriting states during iteration.

Go integers are sufficient because the maximum possible sum fits comfortably within 64-bit integer limits, and Go's default `int` type is adequate on LeetCode platforms.

## Edge Cases

One important edge case occurs when a highly valuable coin is buried deep inside a pile. A naive greedy algorithm might incorrectly try to take only that coin. However, the rules require taking all coins above it first. The prefix sum approach handles this naturally because every choice corresponds to taking a valid prefix only.

Another important edge case occurs when `k` equals the total number of available coins. In this scenario, the algorithm must effectively take everything. The DP transitions still work correctly because every pile can contribute all its coins, and the final state `dp[k]` accumulates the full total.

A third edge case involves many small piles. For example:

```
[[10],[20],[30],[40]]
```

with `k = 2`.

The optimal strategy is selecting the two best piles. The DP formulation correctly evaluates all combinations of pile selections without assuming coins must come from contiguous piles.

Another subtle case involves piles where taking fewer coins is actually better overall because it leaves room for stronger choices elsewhere. The algorithm handles this because every possible prefix length is explored independently, allowing globally optimal combinations rather than locally greedy decisions.
