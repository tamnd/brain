## Solution

## Problem Understanding

The problem gives us a binary string `s`, where each character represents a building type along a street.

- `'0'` means the building is an office
- `'1'` means the building is a restaurant

We must choose exactly three buildings such that no two consecutive selected buildings are of the same type.

This means the selected sequence must alternate. Since we are selecting exactly three buildings from a binary alphabet, the only valid patterns are:

- `"010"`
- `"101"`

Any selection that produces `"000"`, `"001"`, `"011"`, `"100"`, `"110"`, or `"111"` is invalid.

The buildings must preserve their original order because we are selecting indices from the string. We are effectively counting subsequences of length three that form alternating patterns.

The constraints are important:

- `3 <= s.length <= 10^5`

A length of `10^5` immediately tells us that an `O(n^3)` brute-force solution will be far too slow. Even an `O(n^2)` solution may be risky depending on constants. We should aim for a linear-time solution.

Several edge cases are worth identifying early:

- Strings containing only one type, such as `"00000"` or `"11111"`, should return `0`.
- Very short strings of length `3` need careful handling because there is only one possible subsequence.
- Highly alternating strings like `"010101"` produce many valid combinations.
- Consecutive repeated characters can reduce the number of valid subsequences significantly.
- Since the answer can become large, especially for long strings, we must use 64-bit integers in Go.

## Approaches

### Brute Force Approach

The most direct solution is to generate every possible combination of three indices `(i, j, k)` where:

- `i < j < k`

For each triplet, we form the corresponding three-character sequence and check whether it equals `"010"` or `"101"`.

This approach is correct because it explicitly examines every possible valid selection. Nothing is missed, and invalid selections are filtered out.

However, the time complexity is extremely poor. There are `O(n^3)` triplets, and with `n = 10^5`, this becomes computationally impossible.

### Key Insight for the Optimal Solution

Instead of trying every triplet, we can focus on the middle building.

Suppose we fix some position `j` as the middle building.

If `s[j] == '0'`, then we need:

- a `'1'` somewhere to the left
- a `'1'` somewhere to the right

Every combination of those choices forms a valid `"101"` subsequence centered at `j`.

Similarly, if `s[j] == '1'`, then we need:

- a `'0'` on the left
- a `'0'` on the right

Every pair creates a valid `"010"` subsequence.

This transforms the problem into counting:

- how many opposite characters exist before each index
- how many opposite characters exist after each index

Then we multiply the counts together.

This avoids enumerating triplets directly and reduces the complexity to linear time.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n³) | O(1) | Checks every triplet of indices |
| Optimal | O(n) | O(1) | Counts valid subsequences using left/right frequencies |

## Algorithm Walkthrough

1. Count the total number of `'0'` and `'1'` characters in the string.

We need these totals so we can determine how many of each character remain on the right side while scanning the string.
2. Initialize counters for characters seen on the left side.

We maintain:

- `leftZeros`
- `leftOnes`

These represent how many characters have already appeared before the current position.
3. Iterate through the string character by character.

At each index, treat the current character as the middle building of the subsequence.
4. If the current character is `'0'`:

- We need a `'1'` before it
- We need a `'1'` after it

The number of `'1'` characters after the current index is:

```
totalOnes - leftOnes - 1
```

We subtract one because the current character itself contributes to totals only if it is `'1'`. Since current is `'0'`, we simply use remaining ones on the right.

The number of valid subsequences centered here is:

```
leftOnes * rightOnes
```
5. If the current character is `'1'`:

- We need a `'0'` before it
- We need a `'0'` after it

The number of valid subsequences is:

```
leftZeros * rightZeros
```
6. Add the computed value to the answer.

Each multiplication represents all valid ways to choose matching buildings on both sides.
7. Update the left-side counters.

After processing the current index, include it in the left-side counts before moving forward.

### Why It Works

Every valid subsequence of length three must have either pattern `"010"` or `"101"`.

For any valid subsequence, the middle character uniquely determines the pattern:

- middle `'0'` corresponds to `"101"`
- middle `'1'` corresponds to `"010"`

By fixing each position as the middle building and counting all valid left/right choices independently, we count every valid subsequence exactly once.

## Python Solution

```python
class Solution:
    def numberOfWays(self, s: str) -> int:
        total_zeros = s.count('0')
        total_ones = s.count('1')

        left_zeros = 0
        left_ones = 0

        result = 0

        for char in s:
            if char == '0':
                right_ones = total_ones - left_ones
                result += left_ones * right_ones
                left_zeros += 1
            else:
                right_zeros = total_zeros - left_zeros
                result += left_zeros * right_zeros
                left_ones += 1

        return result
```

The implementation begins by counting the total number of zeros and ones in the string. These totals allow us to determine how many matching characters remain on the right side during iteration.

We then maintain two running counters:

- `left_zeros`
- `left_ones`

As we scan the string, each character becomes the middle building of a potential subsequence.

If the current character is `'0'`, then we compute how many `'1'` characters exist on both sides and multiply those counts together. Every combination forms a valid `"101"` subsequence.

If the current character is `'1'`, we similarly count valid `"010"` subsequences.

Finally, we update the left-side counts so future positions can use them.

## Go Solution

```go
func numberOfWays(s string) int64 {
	var totalZeros, totalOnes int64

	for _, ch := range s {
		if ch == '0' {
			totalZeros++
		} else {
			totalOnes++
		}
	}

	var leftZeros, leftOnes int64
	var result int64

	for _, ch := range s {
		if ch == '0' {
			rightOnes := totalOnes - leftOnes
			result += leftOnes * rightOnes
			leftZeros++
		} else {
			rightZeros := totalZeros - leftZeros
			result += leftZeros * rightZeros
			leftOnes++
		}
	}

	return result
}
```

The Go implementation follows the same logic as the Python solution.

The main Go-specific detail is the use of `int64` for all counters and the final answer. The number of valid subsequences can exceed the range of a standard 32-bit integer when the string length approaches `10^5`.

Go strings are iterated using `range`, which returns runes. Since the input only contains `'0'` and `'1'`, rune comparison works naturally.

## Edge Cases

One important edge case is when the string contains only one type of character, such as `"00000"` or `"11111"`. A naive implementation might accidentally count invalid subsequences because every triplet technically exists. However, the algorithm correctly returns `0` because there are never opposite characters available on both sides of any middle position.

Another important case is the minimum input size of length `3`. Since only one subsequence exists in such strings, the algorithm must avoid off-by-one errors when computing right-side counts. Inputs like `"010"` and `"101"` correctly produce `1`, while `"000"` and `"111"` correctly produce `0`.

A third important edge case involves heavily repeated blocks, such as `"001100"`. These cases can produce many valid combinations because multiple left-side and right-side choices exist simultaneously. The multiplication approach handles this naturally by counting every valid pair combination without explicitly enumerating them.

Finally, very large inputs near `10^5` characters are important from a performance perspective. The linear-time solution processes these efficiently, while brute force would be infeasible. The Go implementation additionally uses `int64` to safely store large answers without overflow.
