## Solution

## Problem Understanding

The problem asks us to find users who have made at least two purchases with a maximum gap of seven days between any pair of their purchases. The input is a `Purchases` table with three columns: `purchase_id`, `user_id`, and `purchase_date`. Each row represents a single purchase made by a user. The output is a table of distinct `user_id`s that satisfy the condition, sorted in ascending order.

The key constraints are that `purchase_id` is unique, and `purchase_date` is a standard date. The table may contain multiple purchases per user on the same day, which counts as two purchases within zero days (a valid result). Important edge cases include users with only one purchase, users with multiple purchases separated by more than seven days, and users with multiple purchases on the same day.

## Approaches

The brute-force approach involves comparing all pairs of purchases for each user. For each user, we would iterate through their purchase dates and check every combination of two dates to see if the difference is at most seven days. This approach guarantees correctness because it explicitly checks every possible pair, but it can be extremely slow for users with many purchases, giving a time complexity of O(n²) per user.

The optimal approach leverages SQL's ability to join tables efficiently. The key observation is that we only need to check pairs of purchases for the same user, so a self-join on `user_id` allows us to compare purchase dates directly. We can then filter for date differences that are greater than zero and less than or equal to seven. By selecting distinct `user_id`s from this filtered result, we get the correct output. This approach avoids nested loops in application code and leverages the database engine for set operations.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) per user | O(n) | Compare all pairs of purchases for each user |
| Optimal | O(n log n) | O(n) | Use a self-join and date difference filtering, letting SQL handle comparisons efficiently |

## Algorithm Walkthrough

1. Start by performing a self-join of the `Purchases` table with itself on the `user_id` column. This generates all pairs of purchases for each user.
2. Filter out pairs where the two purchases are the same using the condition `p1.purchase_id != p2.purchase_id`.
3. Compute the absolute difference between `purchase_date` values using `DATEDIFF`. Keep only those pairs where the difference is at most 7 days.
4. Select the distinct `user_id`s from the filtered pairs to ensure no duplicate user IDs in the result.
5. Order the final list of `user_id`s in ascending order.

Why it works: Each pair generated in the self-join represents a potential two-purchase sequence for a user. By filtering for a date difference within seven days and removing identical purchases, we capture all valid users. Selecting distinct IDs ensures uniqueness, and ordering produces the required output format.

## Python Solution

```python
class Solution:
    def usersWithTwoPurchasesWithinSevenDays(self) -> str:
        return """
        SELECT DISTINCT p1.user_id
        FROM Purchases p1
        JOIN Purchases p2
          ON p1.user_id = p2.user_id
         AND p1.purchase_id != p2.purchase_id
         AND ABS(DATEDIFF(p1.purchase_date, p2.purchase_date)) <= 7
        ORDER BY p1.user_id;
        """
```

The Python solution simply returns a SQL query as a string because this is a database problem. The SQL query implements the optimal algorithm: it performs a self-join, filters for valid pairs within seven days, and selects distinct users, ordered by `user_id`.

## Edge Cases

The first edge case is a user with multiple purchases all on the same day. This could trip up naive implementations if they only check `>` instead of `>= 0`. Our implementation handles this by using absolute difference, capturing zero-day gaps.

The second edge case is a user with only one purchase. Naive solutions might include them accidentally, but the self-join inherently requires two distinct purchases, so these users are automatically excluded.

The third edge case is a user with multiple purchases separated by more than seven days. Without proper date filtering, these users might be included incorrectly. The `ABS(DATEDIFF(...)) <= 7` condition ensures only valid pairs are considered, excluding those with longer gaps.
