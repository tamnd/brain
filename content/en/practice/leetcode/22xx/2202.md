## Solution

## Problem Understanding

The problem gives us a stack-like pile represented as an array `nums`, where `nums[0]` is the current top element. We must perform exactly `k` moves, and in each move we may either remove the top element from the pile or add back one previously removed element onto the top.

The goal is to maximize the value of the topmost element after exactly `k` operations.

The important detail is that we are not required to preserve the original order of removed elements when adding them back. Any removed element may be placed back on top at any time. This flexibility is the key observation that makes the greedy solution possible.

The constraints are large:

- `nums.length` can be up to `10^5`
- `k` can be up to `10^9`

These constraints immediately rule out any brute-force simulation of all possible move sequences. We need a direct mathematical or greedy observation.

Several edge cases are especially important:

- A pile with only one element
- `k = 0`
- `k = 1`
- `k` larger than the array length
- Situations where the pile becomes empty
- Situations where we must use exactly all moves, even if additional moves seem unnecessary

The single-element case is particularly tricky because removing the only element empties the pile, and we must carefully consider parity and exact move counts.

## Approaches

### Brute Force Approach

A brute-force solution would explore every possible sequence of moves. At each move we could either:

- Remove the current top element, if the pile is non-empty
- Reinsert any previously removed element

This creates a huge branching factor. Even for small `k`, the number of states grows exponentially.

The brute-force method is correct because it enumerates all reachable configurations and selects the maximum top value among them. However, it is computationally infeasible for the given constraints.

### Key Insight for the Optimal Solution

The crucial observation is that after exactly `k` moves, the final top element can only come from two possibilities:

- One of the first `k - 1` removed elements, because we can remove it earlier and place it back on the last move
- The element originally at index `k`, because we could remove exactly `k` elements and expose that element naturally

This dramatically simplifies the problem.

Suppose we want some element to be the final top:

- If we want to restore a previously removed value, we must spend one move putting it back
- Therefore, we can only remove at most `k - 1` elements beforehand

So among the first `k - 1` elements, we may choose the maximum and place it back on top.

Separately, if we remove exactly `k` elements without reinserting anything, the new top becomes `nums[k]`, assuming that index exists.

The answer is simply the maximum of these candidates.

The only exceptional scenario is when the array has length `1`.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Explores all possible move sequences |
| Optimal | O(min(n, k)) | O(1) | Uses greedy observations about reachable top elements |

## Algorithm Walkthrough

1. First, handle the special case where the array contains only one element.

If `n == 1`, the behavior becomes constrained because there is only one removable element.

- If `k` is odd, the pile ends empty after the final move, so return `-1`
- If `k` is even, we can alternate remove/add operations and end with the original element on top
2. Handle the case where `k == 0`.

No operations are performed, so the answer is simply `nums[0]`.
3. Consider all elements among the first `k - 1` positions.

These are the elements we could remove earlier and then place back onto the top during the final move.

Compute:

```
max(nums[0], nums[1], ..., nums[k - 2])
```

Of course, we only iterate while indices remain inside the array bounds.
4. Consider the element at index `k`.

If `k < n`, then removing exactly `k` elements exposes `nums[k]` as the new top.
5. Return the maximum among all valid candidates.

### Why it works

The algorithm works because every valid final top element must arise from one of two legal constructions:

- Either it was removed earlier and restored on the last move
- Or it becomes exposed after exactly `k` removals

No other element can possibly become the top after exactly `k` operations. Therefore, checking these candidates exhausts all reachable optimal states.

## Python Solution

```python
from typing import List

class Solution:
    def maximumTop(self, nums: List[int], k: int) -> int:
        n = len(nums)

        # Special case: only one element
        if n == 1:
            if k % 2 == 1:
                return -1
            return nums[0]

        # No moves
        if k == 0:
            return nums[0]

        answer = -1

        # We can reinsert any of the first k - 1 removed elements
        limit = min(n, k - 1)

        for i in range(limit):
            answer = max(answer, nums[i])

        # We can also expose nums[k] directly
        if k < n:
            answer = max(answer, nums[k])

        return answer
```

The implementation begins by computing the array length because several edge cases depend on it.

The first conditional handles the single-element scenario. If the number of moves is odd, the pile must end empty because every removal toggles between empty and non-empty states. If the number of moves is even, we can repeatedly remove and restore the same value.

The `k == 0` case is straightforward because no operations occur.

The main loop scans the first `k - 1` elements. These are exactly the elements that could be restored on the final move. We track the maximum candidate value.

Finally, if index `k` exists, we also consider `nums[k]`, since removing exactly `k` elements exposes it naturally.

The maximum among these possibilities is returned.

## Go Solution

```go
func maximumTop(nums []int, k int) int {
	n := len(nums)

	// Special case: only one element
	if n == 1 {
		if k%2 == 1 {
			return -1
		}
		return nums[0]
	}

	// No moves
	if k == 0 {
		return nums[0]
	}

	answer := -1

	// Consider first k - 1 elements
	limit := k - 1
	if limit > n {
		limit = n
	}

	for i := 0; i < limit; i++ {
		if nums[i] > answer {
			answer = nums[i]
		}
	}

	// Consider nums[k]
	if k < n && nums[k] > answer {
		answer = nums[k]
	}

	return answer
}
```

The Go implementation closely mirrors the Python version. Since Go does not provide a built-in `max` function for integers, comparisons are written explicitly with `if` statements.

Slices in Go already behave like dynamic views into arrays, so no extra memory allocation is required. Integer overflow is not a concern because the problem constraints fit comfortably inside Go's standard `int` type.

We evaluate all legal candidates.

The first `k - 1 = 3` elements are:

```

The maximum among them is `5`.

We also consider `nums[4]`:

```
nums[4] = 0
```

Now compare candidates.

| Candidate Source | Value |
| --- | --- |
| Max among first 3 elements | 5 |
| Element at index 4 | 0 |

Final answer:

```
max(5, 0) = 5
```

- Move 1 removes `2`

```

## Edge Cases

### Single Element Array

When `nums` contains only one element, the parity of `k` completely determines the answer.

If `k` is odd, the final move leaves the pile empty because every removal toggles the state. Since an empty pile has no top element, the answer is `-1`.

If `k` is even, we can repeatedly remove and restore the same element, ending with the original value still on top.

This case is easy to mishandle if the implementation assumes a top element always exists.

### `k` Larger Than Array Length

When `k` exceeds `n`, we cannot expose `nums[k]` because that index does not exist.

However, we may still restore one of the previously removed elements. The algorithm correctly handles this by only scanning valid indices and skipping the `nums[k]` candidate when `k >= n`.

### `k == 0`

If no operations are allowed, the pile remains unchanged.

Some implementations accidentally attempt to scan the first `k - 1` elements, which becomes problematic because `k - 1` is negative. The explicit early return avoids this issue entirely.
