**Difficulty:** 🔴 Hard  
**Topics:** String, Dynamic Programming, Prefix Sum

## Solution

## Problem Understanding

The problem presents a binary string `floor` representing a row of tiles. Each character in `floor` is either `'0'` for a black tile or `'1'` for a white tile. The goal is to cover white tiles using a limited number of black carpets, each of a fixed length, in a way that **minimizes the number of white tiles still visible** after placing the carpets. Carpets may overlap, so optimal placement does not require strictly non-overlapping coverage.

The inputs are:

- `floor`: a string of length up to 1000.
- `numCarpets`: up to 1000 carpets available.
- `carpetLen`: length of each carpet, up to the length of `floor`.

The output is a single integer, the minimum number of white tiles still visible after placing carpets optimally.

Key edge cases include:

1. `floor` consisting entirely of `'0'` tiles, which should return 0 regardless of carpets.
2. `numCarpets` or `carpetLen` sufficient to cover all `'1'` tiles, producing a 0 result.
3. Carpets longer than the remaining floor at the end, where careful handling avoids index errors.
4. Small floors, carpets, or just one carpet, which test boundary conditions.

The constraints indicate the input size is small enough (≤1000) to allow a dynamic programming approach without running into performance issues.

## Approaches

**Brute-force Approach:**

A brute-force method would attempt every possible way to place the carpets. For each starting tile, we could either place a carpet or skip it, recursively computing the minimum visible white tiles for the remaining portion of the floor. While correct, this method is **exponentially slow** because the number of placement combinations grows rapidly with `numCarpets` and `floor.length`.

**Optimal Approach:**

A key observation is that the problem has **overlapping subproblems and an optimal substructure**, making it ideal for dynamic programming. We define a DP state `dp[i][k]` as the minimum number of visible white tiles from tile `i` to the end using `k` carpets. At each step, there are two choices: either leave the current tile uncovered, contributing 1 if it is white, or cover a segment of length `carpetLen` starting at the current tile. Using prefix sums of white tiles allows quick calculation of how many white tiles a carpet covers. This reduces the problem to O(n * numCarpets) time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^(n*numCarpets)) | O(n*numCarpets) | Recursively try every placement of carpets; impractical for n ~ 1000 |
| Optimal | O(n * numCarpets) | O(n * numCarpets) | Dynamic programming with prefix sums to compute visible white tiles efficiently |

## Algorithm Walkthrough

1. Compute a **prefix sum array** `prefixWhite` where `prefixWhite[i]` stores the total number of white tiles from the start to index `i-1`. This allows constant-time computation of the number of white tiles in any segment.
2. Initialize a DP table `dp` of size `(floor.length + 1) x (numCarpets + 1)` to store the minimum number of white tiles visible starting from tile `i` with `k` carpets remaining.
3. Iterate over `i` from `floor.length - 1` down to 0, representing the current starting tile for decisions.
4. For each `k` from 0 to `numCarpets`, compute the minimum visible white tiles if we either skip the current tile or place a carpet starting at the current tile.

- **Skip current tile:** If `floor[i]` is `'1'`, add 1 to the result from `dp[i+1][k]`.
- **Place a carpet:** Only if `k > 0`, compute `end = min(i + carpetLen, floor.length)` and use the prefix sum to get white tiles covered by the carpet, then add `dp[end][k-1]`.
5. The answer is stored in `dp[0][numCarpets]`, representing the minimum number of white tiles visible starting from the beginning with all carpets available.

**Why it works:**

The DP state captures the minimum visible white tiles for every subproblem, ensuring overlapping subproblems are reused. Each decision either leaves a tile uncovered or covers a segment, guaranteeing the substructure is optimal. Prefix sums ensure constant-time calculation of white tiles covered by a carpet, which keeps the overall complexity manageable.

## Python Solution

```python
class Solution:
    def minimumWhiteTiles(self, floor: str, numCarpets: int, carpetLen: int) -> int:
        n = len(floor)
        prefixWhite = [0] * (n + 1)
        for i in range(n):
            prefixWhite[i + 1] = prefixWhite[i] + (1 if floor[i] == '1' else 0)
        
        dp = [[0] * (numCarpets + 1) for _ in range(n + 1)]
        
        for i in range(n - 1, -1, -1):
            for k in range(numCarpets + 1):
                # Option 1: skip the current tile
                dp[i][k] = dp[i + 1][k] + (1 if floor[i] == '1' else 0)
                # Option 2: use a carpet if available
                if k > 0:
                    end = min(i + carpetLen, n)
                    dp[i][k] = min(dp[i][k], dp[end][k - 1])
        
        return dp[0][numCarpets]
```

**Implementation Walkthrough:**

The prefix sum array `prefixWhite` allows rapid calculation of white tiles covered. The DP table `dp[i][k]` fills from the end to the beginning, considering both choices at each position. Skipping a tile accounts for the visible white tile, while placing a carpet reduces visible white tiles over the carpet segment. Using `min` ensures the optimal choice is always selected.

## Go Solution

```go
func minimumWhiteTiles(floor string, numCarpets int, carpetLen int) int {
    n := len(floor)
    prefixWhite := make([]int, n+1)
    for i := 0; i < n; i++ {
        prefixWhite[i+1] = prefixWhite[i]
        if floor[i] == '1' {
            prefixWhite[i+1]++
        }
    }
    
    dp := make([][]int, n+1)
    for i := range dp {
        dp[i] = make([]int, numCarpets+1)
    }
    
    for i := n - 1; i >= 0; i-- {
        for k := 0; k <= numCarpets; k++ {
            dp[i][k] = dp[i+1][k]
            if floor[i] == '1' {
                dp[i][k]++
            }
            if k > 0 {
                end := i + carpetLen
                if end > n {
                    end = n
                }
                if dp[i][k] > dp[end][k-1] {
                    dp[i][k] = dp[end][k-1]
                }
            }
        }
    }
    
    return dp[0][numCarpets]
}
```

**Go-specific Notes:**

Slices are used for both prefix sums and the DP table. Index handling for `end` ensures we do not go out of bounds. Integer overflow is not a concern because inputs are small. Unlike Python, Go requires explicit initialization of the 2D slice.

## Worked Examples

**Example 1:** `floor = "10110101", numCarpets = 2, carpetLen = 2`

Step through `dp` for each tile and carpet:

| i | k=2 | k=1 | k=0 | Explanation |
| --- | --- | --- | --- | --- |
| 7 | 1 | 1 | 1 | Last tile is '1', uncovered counts as 1 |
| 6 | 1 | 1 | 1 | Tile '0', no additional white |
| 5 | 2 | 2 | 2 | Tile '1', can cover with carpet for min 1 |
| ... | ... | ... | ... | Continue backward filling dp |

Final result `dp[0][2] = 2`.

**Example 2:** `floor = "11111", numCarpets = 2, carpetLen = 3`

Optimal carpet placement covers all tiles, so `dp[0][2] = 0`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n * numCarpets) | Nested loops over floor length and number of carpets, constant-time prefix sum operations |
| Space | O(n * numCarpets) | DP table stores values for each tile and number of carpets |

The prefix sum is O(n) and does not dominate the overall complexity. The DP approach avoids exponential recursion, making it feasible for the constraints.
