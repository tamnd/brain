**Difficulty:** 🟡 Medium  
**Topics:** Array, Hash Table, String, Counting

## Solution

## Problem Understanding

The problem gives us two arrays of equal length:

- `messages[i]` represents the text of the `i-th` message.
- `senders[i]` represents the sender who sent that message.

Each message contains words separated by single spaces, with no leading or trailing spaces. The word count of a message is simply the number of words inside it. The total word count for a sender is the sum of the word counts of all messages sent by that sender.

Our task is to determine which sender has sent the largest total number of words.

There is one important tie-breaking rule. If multiple senders have the same maximum total word count, we must return the lexicographically largest sender name.

Lexicographical comparison works similarly to dictionary ordering, except uppercase letters come before lowercase letters in ASCII ordering. For example:

- `"Charlie" > "Bob"`
- `"alice" > "Alice"` because lowercase letters have larger ASCII values than uppercase letters.

The constraints are relatively small:

- Up to `10^4` messages
- Each message length is at most `100`

This tells us that even straightforward string operations are completely acceptable. Counting words by splitting each message is efficient enough.

Several edge cases are important:

- Multiple messages can belong to the same sender.
- Different senders may end with the same total word count.
- A sender name may contain uppercase or lowercase letters, and comparisons are case-sensitive.
- Messages may contain only one word.
- There is guaranteed to be at least one message.

The problem guarantees valid formatting for messages, so we do not need to worry about extra spaces or malformed input.

## Approaches

### Brute Force Approach

A naive solution would repeatedly recompute the total word count for every sender.

For each sender, we could scan through the entire list of messages and sum the word counts of messages belonging to that sender. After computing totals for every sender, we would determine the sender with the maximum total.

This approach is correct because every sender’s total is computed exactly according to the definition in the problem statement. However, it is inefficient because the same messages are repeatedly scanned for different senders.

If there are `n` messages and many unique senders, this repeated scanning can lead to quadratic behavior.

### Optimal Approach

The key observation is that we only need to process each message once.

As we iterate through the arrays:

1. Count the number of words in the current message.
2. Add that count to the sender’s running total using a hash map.
3. Keep track of the current best sender.

A hash map is ideal because it allows constant-time updates for sender totals.

Word counting is simple because the input guarantees single spaces between words. The number of words in a message equals:

```
number of spaces + 1
```

We can either split the string or count spaces directly. Both approaches are fast enough. Using `split()` is simpler and more readable.

Whenever a sender’s total increases, we compare it against the current maximum:

- If the new total is larger, update the answer.
- If the totals are equal, choose the lexicographically larger sender name.

This gives us a clean linear-time solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(k) | Recomputes totals repeatedly for each sender |
| Optimal | O(n * m) | O(k) | Processes each message once, where `m` is message length |

Here:

- `n` is the number of messages
- `m` is the average message length
- `k` is the number of unique senders

## Algorithm Walkthrough

1. Create a hash map called `word_counts`.

This map stores the cumulative number of words sent by each sender. The key is the sender name, and the value is the total word count so far.
2. Initialize variables for the current best sender and maximum word count.

We keep:

- `best_sender`
- `max_words`

These help us avoid scanning the hash map again later.
3. Iterate through all messages and senders together.

For each index:

- Read the current message.
- Read the corresponding sender.
4. Count the number of words in the current message.

Since words are separated by single spaces, we can compute the count using:

```
len(message.split())
```
5. Add the word count to the sender’s total.

Update the hash map:

```
word_counts[sender] += words
```
6. Compare the sender’s updated total with the current maximum.

There are two cases:

- The sender now has a strictly larger total.
- The sender ties the maximum total but has a lexicographically larger name.

In either case, update:

- `best_sender`
- `max_words`
7. Continue until all messages are processed.
8. Return `best_sender`.

### Why it works

The algorithm maintains an invariant during iteration:

- `word_counts[sender]` always equals the total number of words processed so far for that sender.
- `best_sender` always represents the sender with the largest total among all processed messages.

Every message contributes exactly once to its sender’s total, so the final totals are correct. The tie-breaking rule is enforced immediately whenever totals are equal, guaranteeing that the final answer is the lexicographically largest valid sender.

## Python Solution

```python
from collections import defaultdict
from typing import List

class Solution:
    def largestWordCount(self, messages: List[str], senders: List[str]) -> str:
        word_counts = defaultdict(int)

        best_sender = ""
        max_words = 0

        for message, sender in zip(messages, senders):
            words = len(message.split())
            word_counts[sender] += words

            current_total = word_counts[sender]

            if (
                current_total > max_words
                or (
                    current_total == max_words
                    and sender > best_sender
                )
            ):
                max_words = current_total
                best_sender = sender

        return best_sender
```

The implementation begins by creating a `defaultdict(int)` so every sender automatically starts with a word count of zero.

We then maintain two tracking variables:

- `max_words`, the largest total seen so far
- `best_sender`, the sender associated with that maximum

The loop processes messages and senders together using `zip()`.

For each message:

- `message.split()` separates the words.
- `len(...)` gives the number of words.
- The sender’s running total is updated in the hash map.

After updating the total, the algorithm immediately checks whether this sender should become the new answer. The sender becomes the new answer if:

- Their total is larger than the current maximum.
- Their total ties the current maximum, but their name is lexicographically larger.

At the end, `best_sender` contains the correct result.

## Go Solution

```go
package main

import "strings"

func largestWordCount(messages []string, senders []string) string {
	wordCounts := make(map[string]int)

	bestSender := ""
	maxWords := 0

	for i := 0; i < len(messages); i++ {
		words := len(strings.Split(messages[i], " "))
		sender := senders[i]

		wordCounts[sender] += words
		currentTotal := wordCounts[sender]

		if currentTotal > maxWords ||
			(currentTotal == maxWords && sender > bestSender) {
			maxWords = currentTotal
			bestSender = sender
		}
	}

	return bestSender
}
```

The Go implementation follows the same logic as the Python version.

Instead of `defaultdict`, Go uses a regular map:

```go
map[string]int
```

Missing keys automatically return zero values, which makes accumulation straightforward.

Word counting is performed with:

```go
strings.Split(message, " ")
```

Lexicographical comparison between strings works directly using standard comparison operators like `>`.

There are no integer overflow concerns because the total number of words is small relative to Go’s integer range.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n * m) | Each message is processed once, and splitting a message takes proportional time to its length |
| Space | O(k) | Hash map stores totals for unique senders |

The algorithm processes each message exactly once. Word counting requires scanning the message string, so the total runtime depends on the total size of all message strings.

The additional memory usage comes from storing sender totals in the hash map. If there are `k` unique senders, the map requires `O(k)` space.

## Edge Cases

### Multiple senders with identical totals

One of the most important edge cases occurs when several senders end with the same total word count. A buggy implementation might simply return the first sender that reaches the maximum.

The problem explicitly requires returning the lexicographically largest sender in case of a tie. The implementation handles this by checking:

```
current_total == max_words and sender > best_sender
```

This guarantees the correct sender is selected.

### A sender appearing many times

A sender may send multiple messages spread throughout the input. A naive implementation could accidentally overwrite counts instead of accumulating them.

The hash map solves this cleanly by maintaining a running total:

```
word_counts[sender] += words
```

Every message contributes exactly once to the sender’s cumulative total.

### Messages containing only one word

Some messages may contain no spaces at all. Incorrect word-count logic based purely on counting spaces could fail if not handled carefully.

Using:

```
len(message.split())
```

works correctly because splitting a one-word string produces a list containing one element.

### Uppercase and lowercase lexicographical ordering

Lexicographical comparison in this problem is case-sensitive. ASCII ordering places uppercase letters before lowercase letters.

For example:

```
"alice" > "Alice"
```

The implementation relies directly on the language’s built-in string comparison operators, which already follow the required ordering rules.
