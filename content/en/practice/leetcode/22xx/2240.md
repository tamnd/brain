## Solution

## Problem Understanding

The problem gives us a total amount of money, along with the individual costs of pens and pencils. We may buy any non-negative number of pens and any non-negative number of pencils, as long as the total amount spent does not exceed the available money.

More formally, we are looking for the number of integer pairs `(pens, pencils)` such that:

$pens \cdot cost1 + pencils \cdot cost2 \le total$

Both `pens` and `pencils` must be greater than or equal to zero.

The input consists of:

- `total`, the maximum amount of money available
- `cost1`, the price of one pen
- `cost2`, the price of one pencil

The output should be the total number of distinct valid combinations.

A distinct way means a different pair of quantities. For example, buying 1 pen and 2 pencils is different from buying 2 pens and 1 pencil.

The constraints are important:

- `1 <= total, cost1, cost2 <= 10^6`

Since the values can be as large as one million, we need an efficient solution. A naive double nested loop could potentially iterate over billions of combinations in the worst case, which would be too slow.

There are several important edge cases to keep in mind:

- Both items may cost more than `total`, meaning the only valid answer is buying nothing.
- One item may be very cheap while the other is very expensive.
- The total may be exactly divisible by one or both costs.
- We must count the `(0, 0)` case, because buying nothing is allowed.

## Approaches

### Brute Force Approach

The most straightforward approach is to try every possible number of pens and pencils.

We could iterate through every valid pen count from `0` to `total // cost1`, and for each pen count, iterate through every valid pencil count from `0` to `total // cost2`. For each pair, we would check whether the total cost stays within the budget.

This approach is correct because it explicitly examines every possible combination. Any valid pair is counted exactly once.

However, the time complexity is too large. In the worst case, if both costs are `1`, we would iterate roughly:

$10^6 \times 10^6 = 10^{12}$

operations, which is completely impractical.

### Optimal Approach

The key observation is that once we choose a number of pens, the remaining money automatically determines how many pencil choices are possible.

Suppose we buy `p` pens. The remaining money is:

$remaining = total - p \cdot cost1$

The number of pencils we can buy ranges from `0` up to:

$\left\lfloor \frac{remaining}{cost2} \right\rfloor$

That means the number of valid pencil choices is:

$\left\lfloor \frac{remaining}{cost2} \right\rfloor + 1$

The `+1` accounts for buying zero pencils.

Instead of checking every `(pens, pencils)` pair individually, we count all valid pencil choices at once for each pen count.

This reduces the complexity dramatically.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O((total / cost1) * (total / cost2)) | O(1) | Tries every pair explicitly |
| Optimal | O(total / cost1) | O(1) | Counts all pencil choices in constant time per pen count |

## Algorithm Walkthrough

1. Initialize a variable `ways` to store the total number of valid combinations.
2. Iterate through every possible number of pens from `0` up to `total // cost1`.
3. For the current pen count, compute how much money remains after buying those pens.
4. Using the remaining money, calculate the maximum number of pencils we can afford.
5. Since pencil counts can range from `0` to that maximum value, add `(max_pencils + 1)` to the answer.
6. Continue until all possible pen counts have been processed.
7. Return the accumulated total.

### Why it works

For every possible number of pens, the algorithm counts all compatible pencil counts exactly once. Every valid `(pens, pencils)` pair belongs to exactly one iteration of the outer loop because the pen count uniquely identifies that iteration. Since the algorithm considers every feasible pen count and counts every feasible pencil count for that pen count, all valid combinations are counted exactly once.

## Python Solution

```python
class Solution:
    def waysToBuyPensPencils(self, total: int, cost1: int, cost2: int) -> int:
        ways = 0

        max_pens = total // cost1

        for pens in range(max_pens + 1):
            remaining_money = total - (pens * cost1)

            max_pencils = remaining_money // cost2

            ways += max_pencils + 1

        return ways
```

The implementation begins by computing the maximum number of pens that can possibly be purchased. This defines the range for the loop.

For each pen count, we calculate the remaining money after purchasing those pens. Using integer division, we determine the maximum number of pencils that fit within the remaining budget.

If the maximum number of pencils is `k`, then valid pencil counts are:

- `0`
- `1`
- `2`
- ...
- `k`

This gives exactly `k + 1` valid choices, which we add directly to the answer.

The algorithm uses only a few integer variables, so the space complexity remains constant.

## Go Solution

```go
func waysToBuyPensPencils(total int, cost1 int, cost2 int) int64 {
    var ways int64 = 0

    maxPens := total / cost1

    for pens := 0; pens <= maxPens; pens++ {
        remainingMoney := total - (pens * cost1)

        maxPencils := remainingMoney / cost2

        ways += int64(maxPencils + 1)
    }

    return ways
}
```

The Go implementation follows the same logic as the Python solution. The primary difference is that the return type is `int64`, so the accumulated answer must also use `int64`.

This is important because the number of combinations can become large. For example, when both costs are `1`, the number of valid combinations is approximately:

$\frac{(10^6 + 1)(10^6 + 2)}{2}$

which exceeds the range of a 32-bit integer.

## Edge Cases

One important edge case occurs when both item costs are greater than the available budget. In this scenario, no pens or pencils can be purchased. A careless implementation might incorrectly return `0`, forgetting that buying nothing is still a valid choice. This implementation handles the case naturally because the loop still runs once for zero pens, and zero pencils is counted.

Another important edge case occurs when one item is extremely cheap and the other is very expensive. For example, if pens cost `100` and pencils cost `1`, almost all combinations involve pencils only. The algorithm still works efficiently because it only iterates over feasible pen counts and computes pencil possibilities directly.

A third edge case involves very large inputs. Since `total` can reach `10^6`, the number of combinations may exceed the range of a 32-bit integer. The Go implementation explicitly uses `int64` to avoid overflow. Python integers automatically expand as needed, so no special handling is required there.

A final subtle edge case is exact divisibility. If the remaining money divides evenly by `cost2`, the maximum pencil count itself is valid and must be included. Using integer division and adding `1` ensures that the implementation correctly counts the zero-pencil case and the exact-maximum-pencil case.
