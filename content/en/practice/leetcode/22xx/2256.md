## Solution

## Problem Understanding

The problem gives us a 0-indexed integer array `nums`, and for every index `i`, we must compute something called the "average difference".

For a chosen index `i`, the array is divided into two parts:

- The left part contains elements from index `0` through `i`
- The right part contains elements from index `i + 1` through `n - 1`

We then calculate:

1. The average of the left part
2. The average of the right part
3. The absolute difference between those two averages

Both averages use integer division, which means the values are rounded down automatically.

The goal is to return the index whose average difference is the smallest. If multiple indices produce the same minimum difference, we return the smallest index among them.

The constraints are important:

- `1 <= nums.length <= 10^5`
- `0 <= nums[i] <= 10^5`

An array length of `10^5` means an `O(n^2)` solution will be too slow. We need something close to linear time.

A key detail is that the average of zero elements is defined as `0`. This matters when `i == n - 1`, because the right side becomes empty.

Several edge cases are important:

- A single-element array like `[0]`
- Cases where the minimum difference occurs multiple times
- Very large sums that could overflow 32-bit integers in some languages
- The last index, where the right-side average must be treated as `0`

Understanding these details correctly is essential for building a correct implementation.

## Approaches

### Brute Force Approach

The most direct approach is to evaluate every index independently.

For each index `i`:

1. Compute the sum of elements from `0` to `i`
2. Compute the sum of elements from `i + 1` to `n - 1`
3. Compute both averages
4. Compute the absolute difference
5. Track the smallest difference and corresponding index

This works because it directly follows the problem definition.

However, recomputing sums repeatedly is extremely inefficient. For every index, we may scan almost the entire array again.

If the array length is `n`, then:

- We process `n` indices
- Each index may require `O(n)` work

That produces `O(n^2)` time complexity, which is too slow for `n = 10^5`.

### Optimal Prefix Sum Approach

The key observation is that repeated summation is unnecessary.

If we know:

- The total sum of the array
- The running prefix sum up to index `i`

Then we can compute:

- Left sum directly from the prefix sum
- Right sum as `totalSum - leftSum`

This allows every index to be processed in constant time.

Instead of recalculating sums repeatedly, we maintain a running prefix sum while traversing the array once.

This transforms the solution into an `O(n)` algorithm.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Recomputes left and right sums for every index |
| Optimal | O(n) | O(1) | Uses running prefix sums and total sum |

## Algorithm Walkthrough

### Optimal Prefix Sum Algorithm

1. Compute the total sum of the entire array.

This allows us to derive any suffix sum later without rescanning the array.
2. Initialize a running prefix sum variable.

As we move through the array, this variable stores the sum of elements from index `0` through `i`.
3. Initialize variables to track:

- The minimum average difference seen so far
- The index that produced it
4. Iterate through every index `i` in the array.

During each iteration:

- Add `nums[i]` to the running prefix sum
- The left sum becomes the current prefix sum
- The right sum becomes `totalSum - leftSum`
5. Compute the left average.

The left side contains `i + 1` elements, so:

$\text{leftAvg} = \left\lfloor \frac{\text{leftSum}}{i+1} \right\rfloor$
6. Compute the right average.

If the right side is non-empty:

$\text{rightAvg} = \left\lfloor \frac{\text{rightSum}}{n-i-1} \right\rfloor$

Otherwise, the right average is `0`.
7. Compute the absolute difference between the two averages.

$\text{difference} = |\text{leftAvg} - \text{rightAvg}|$
8. If this difference is smaller than the current minimum:

- Update the minimum difference
- Store the current index
9. Continue until all indices are processed.
10. Return the stored index.

### Why it works

The algorithm works because every index's average difference depends only on:

- The sum of elements before and including the index
- The sum of elements after the index

Using a running prefix sum guarantees that the left sum is always correct for the current index. Since the total sum is fixed, subtracting the left sum gives the exact right sum.

Every index is evaluated exactly once, and the algorithm always keeps track of the smallest difference encountered so far. Because indices are processed from left to right, ties automatically preserve the smallest index.

## Python Solution

```python
from typing import List

class Solution:
    def minimumAverageDifference(self, nums: List[int]) -> int:
        total_sum = sum(nums)
        prefix_sum = 0
        
        min_difference = float("inf")
        result_index = 0
        
        n = len(nums)
        
        for i in range(n):
            prefix_sum += nums[i]
            
            left_average = prefix_sum // (i + 1)
            
            right_sum = total_sum - prefix_sum
            right_count = n - i - 1
            
            if right_count > 0:
                right_average = right_sum // right_count
            else:
                right_average = 0
            
            current_difference = abs(left_average - right_average)
            
            if current_difference < min_difference:
                min_difference = current_difference
                result_index = i
        
        return result_index
```

The implementation begins by computing the total sum of the array. This value never changes and allows suffix sums to be derived instantly.

The variable `prefix_sum` maintains the running sum from the beginning of the array through the current index.

Inside the loop:

- `left_average` is computed using integer division
- `right_sum` is derived by subtracting the prefix sum from the total sum
- `right_count` determines whether the right side exists

If the right side is empty, the problem explicitly defines its average as `0`.

The algorithm then computes the absolute difference and updates the answer whenever a smaller value is found.

Because the loop processes indices in ascending order, ties naturally keep the smallest index.

## Go Solution

```go
package main

import "math"

func minimumAverageDifference(nums []int) int {
	n := len(nums)

	var totalSum int64
	for _, num := range nums {
		totalSum += int64(num)
	}

	var prefixSum int64

	minDifference := int64(math.MaxInt64)
	resultIndex := 0

	for i := 0; i < n; i++ {
		prefixSum += int64(nums[i])

		leftAverage := prefixSum / int64(i+1)

		rightSum := totalSum - prefixSum
		rightCount := n - i - 1

		var rightAverage int64
		if rightCount > 0 {
			rightAverage = rightSum / int64(rightCount)
		} else {
			rightAverage = 0
		}

		difference := leftAverage - rightAverage
		if difference < 0 {
			difference = -difference
		}

		if difference < minDifference {
			minDifference = difference
			resultIndex = i
		}
	}

	return resultIndex
}
```

The Go implementation follows the same algorithm as the Python version, but there are several language-specific considerations.

The most important detail is integer overflow. Since the array can contain up to `10^5` elements and each value can also be `10^5`, the total sum may reach `10^10`, which exceeds 32-bit integer range.

To handle this safely, the implementation uses `int64` for all sum and average calculations.

Unlike Python, Go does not provide a built-in absolute value function for integers, so the absolute difference is computed manually.

## Edge Cases

### Single Element Array

When the array contains only one element, the right side is empty for index `0`.

A common mistake is attempting to divide by zero while computing the right average. The problem explicitly defines the average of zero elements as `0`.

The implementation handles this by checking whether the right partition size is greater than zero before performing division.

### Minimum Difference Appears Multiple Times

The problem requires returning the smallest index if several indices share the same minimum difference.

A subtle bug would occur if the implementation updated the answer when the new difference was less than or equal to the current minimum.

Instead, the algorithm updates only when the new difference is strictly smaller. Since iteration proceeds from left to right, the earliest index is preserved automatically.

### Last Index Handling

At the last index:

- The left partition contains the entire array
- The right partition is empty

This is another situation where division by zero could occur.

The implementation explicitly checks whether the right partition size is zero and assigns the right average to `0` in that case.

### Large Sum Values

The total array sum may exceed the range of 32-bit integers.

For example:

```
100000 elements × 100000 value = 10^10
```

This exceeds standard 32-bit integer capacity.

The Go implementation uses `int64` to safely store large sums and averages. Python integers already support arbitrary precision, so no extra handling is required there.
