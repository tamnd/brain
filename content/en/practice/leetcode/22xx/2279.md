## Solution

## Problem Understanding

This problem gives us two arrays, `capacity` and `rocks`, where each index represents a bag. The value `capacity[i]` tells us the maximum number of rocks the `i-th` bag can hold, and `rocks[i]` tells us how many rocks are already inside that bag.

We are also given an integer `additionalRocks`, which represents extra rocks that we are free to distribute among the bags in any way we choose.

The goal is to maximize the number of bags that become completely full after distributing the extra rocks.

A bag is considered full when:

```
rocks[i] == capacity[i]
```

For every bag, we can calculate how many more rocks are needed to fill it:

```
needed[i] = capacity[i] - rocks[i]
```

Some bags may already be full, meaning their required amount is `0`. Other bags may need several additional rocks.

The challenge is to decide how to distribute the limited number of extra rocks so that the number of fully filled bags is as large as possible.

The constraints are important:

- There can be up to `5 * 10^4` bags.
- Capacities and rock counts can be as large as `10^9`.

These limits immediately tell us that:

- We cannot use exponential or combinatorial search.
- We need an efficient algorithm, likely `O(n log n)` or better.
- Since values can be large, we should avoid unnecessary arithmetic complexity, but standard integer operations are completely safe.

There are several edge cases worth recognizing early:

- Some bags may already be full.
- We may have enough rocks to fill every bag.
- We may not have enough rocks to fill even one partially filled bag.
- Multiple bags may require the same number of rocks.
- `additionalRocks` can be much larger than the total remaining capacity.

The problem guarantees that:

```
0 <= rocks[i] <= capacity[i]
```

So we never need to handle invalid states such as overfilled bags.

## Approaches

### Brute Force Approach

A brute force strategy would try all possible ways to distribute the additional rocks among the bags.

For example, we could recursively decide how many rocks to place into each bag, then count how many bags end up full.

This approach is theoretically correct because it explores every possible allocation and therefore cannot miss the optimal answer.

However, it is completely impractical.

Suppose there are `n` bags and `additionalRocks` rocks. The number of possible distributions grows extremely quickly. Even for relatively small inputs, the number of combinations becomes enormous.

A slightly less naive brute force method could repeatedly simulate different filling orders and compare outcomes, but this still becomes inefficient because there are too many possible choices.

Given that `n` can reach `50,000`, any exponential or high polynomial solution is impossible.

### Key Insight

The important observation is that every bag has a "cost" to become full:

```
cost = capacity[i] - rocks[i]
```

If our goal is to maximize the number of completed bags, then we should always fill the cheapest bags first.

This is a classic greedy strategy.

Why does this work?

Suppose one bag needs `1` rock and another needs `10` rocks. If we spend rocks on the expensive bag first, we may lose the opportunity to fully complete several cheaper bags.

To maximize the count of completed bags, we should prioritize bags requiring the fewest additional rocks.

Therefore:

1. Compute how many rocks each bag still needs.
2. Sort these requirements in ascending order.
3. Fill bags from smallest requirement to largest until we run out of rocks.

This greedy choice is optimal because every time we spend rocks, we gain the maximum possible number of completed bags for the amount spent.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Tries all rock distributions, infeasible for large inputs |
| Optimal Greedy + Sorting | O(n log n) | O(n) | Fill bags requiring the fewest rocks first |

## Algorithm Walkthrough

1. Create an array called `remaining` that stores how many rocks each bag still needs to become full.

For every bag:

```
remaining[i] = capacity[i] - rocks[i]
```

This transforms the problem into a simpler form. Instead of thinking about capacities and current rocks separately, we only care about how many rocks are still needed.
2. Sort the `remaining` array in ascending order.

This is the critical greedy step.

By sorting, we ensure that bags requiring the fewest rocks are processed first. Since our goal is to maximize the number of completed bags, smaller requirements are always more efficient.
3. Initialize a counter called `full_bags` to track how many bags become full.
4. Iterate through the sorted `remaining` array.

For each value:

- If we have enough `additionalRocks` to fill the bag:

- Subtract the required rocks from `additionalRocks`
- Increment `full_bags`
- Otherwise, stop immediately.

Since the array is sorted, every remaining bag requires at least as many rocks as the current one. If we cannot fill the current bag, we cannot fill any later bag either.
5. Return `full_bags`.

### Why it works

The greedy strategy is optimal because filling a cheaper bag can never reduce the number of bags we can complete later.

Suppose we skip a bag requiring `2` rocks and instead fill one requiring `5` rocks. We spend more resources while gaining the same benefit, exactly one completed bag.

By always choosing the smallest remaining requirement first, we maximize the number of bags completed for the available budget of rocks.

This is the same principle used in many scheduling and resource allocation greedy problems.

## Python Solution

```python
from typing import List

class Solution:
    def maximumBags(
        self,
        capacity: List[int],
        rocks: List[int],
        additionalRocks: int
    ) -> int:
        
        remaining = []

        for i in range(len(capacity)):
            remaining.append(capacity[i] - rocks[i])

        remaining.sort()

        full_bags = 0

        for needed in remaining:
            if needed <= additionalRocks:
                additionalRocks -= needed
                full_bags += 1
            else:
                break

        return full_bags
```

The implementation begins by computing the remaining capacity for every bag. This converts the original problem into a simpler list of costs.

Next, the list is sorted so that the cheapest bags to complete appear first. This enables the greedy strategy.

The loop then processes bags one at a time in sorted order. Whenever enough rocks remain to fill the current bag, those rocks are spent and the completed bag count increases.

The moment we encounter a bag that cannot be filled, the algorithm stops. Because the array is sorted, every remaining bag would require at least as many rocks.

Finally, the function returns the number of completed bags.

## Go Solution

```go
package main

import "sort"

func maximumBags(capacity []int, rocks []int, additionalRocks int) int {

	n := len(capacity)

	remaining := make([]int, n)

	for i := 0; i < n; i++ {
		remaining[i] = capacity[i] - rocks[i]
	}

	sort.Ints(remaining)

	fullBags := 0

	for _, needed := range remaining {
		if needed <= additionalRocks {
			additionalRocks -= needed
			fullBags++
		} else {
			break
		}
	}

	return fullBags
}
```

The Go implementation follows exactly the same logic as the Python version.

The primary language-specific difference is that Go requires explicit slice allocation using `make`. Sorting is performed using `sort.Ints`.

Integer overflow is not a concern because Go's `int` type easily handles values up to the problem constraints on standard LeetCode environments.

The algorithm performs three major operations:

- Building the `remaining` array, `O(n)`
- Sorting the array, `O(n log n)`
- Iterating through the sorted values, `O(n)`

The sorting step dominates the total runtime.

The extra space comes from storing the `remaining` array separately from the input arrays.

## Edge Cases

One important edge case occurs when some or all bags are already full. In this situation, the remaining requirement for those bags becomes zero. A buggy implementation might skip them or incorrectly spend rocks on them. This implementation handles the case naturally because zero values appear at the front after sorting and are counted immediately without consuming any additional rocks.

Another important case is when there are not enough additional rocks to complete even the cheapest bag. A naive solution might continue checking later bags unnecessarily. Since the requirements are sorted, the implementation stops immediately once a bag cannot be filled. This guarantees correctness and avoids wasted work.

A third critical edge case occurs when `additionalRocks` is much larger than the total remaining capacity of all bags combined. Some implementations may accidentally overrun indices or continue unnecessary processing. This solution simply fills every bag in sorted order and returns the total count once iteration finishes.

A subtle case involves multiple bags requiring the same number of rocks. The greedy algorithm remains correct because any ordering among equal-cost bags produces the same number of completed bags. Sorting naturally handles duplicates without requiring special logic.
