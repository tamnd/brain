**Difficulty:** 🟡 Medium  
**Topics:** Array, Hash Table, Sorting, Counting

## Solution

## Problem Understanding

The problem gives a list of match results, where each match is represented as a pair `[winner, loser]`. This means that the player `winner` defeated the player `loser` exactly once in that match.

Our task is to analyze all match results and produce two sorted lists:

1. All players who participated in at least one match and never lost.
2. All players who participated in at least one match and lost exactly one time.

The final answer must be returned as a list containing these two lists:

```
[
    players_with_zero_losses,
    players_with_one_loss
]
```

Both inner lists must be sorted in increasing order.

A very important detail is that we only consider players who actually appeared in the matches array. If a player never appears as either a winner or loser, they should not be included.

The constraints are large enough to require an efficient solution:

- There can be up to `10^5` matches.
- Player IDs can also be as large as `10^5`.

Because of this scale, repeatedly scanning the entire match list for every player would be too slow. We need a solution that processes matches efficiently in a single pass or close to it.

Another important observation is that the same match outcome will never appear twice. This simplifies counting because we never need to handle duplicate match records.

Several edge cases are important:

- A player may only appear as a winner and never as a loser.
- A player may lose many times.
- No player may have exactly one loss.
- Every player may have at least one loss except a few winners.
- The smallest valid input contains only one match.

These cases can easily expose bugs if the implementation does not track all participating players correctly.

## Approaches

### Brute Force Approach

A straightforward approach would be:

1. Collect every unique player from the matches list.
2. For each player, scan through all matches and count how many times that player appears as a loser.
3. Based on the loss count:

- Add the player to the zero-loss list if the count is `0`.
- Add the player to the one-loss list if the count is `1`.
4. Sort both lists before returning.

This works because every player's loss count is computed directly from the full match history.

However, this approach is inefficient. If there are `P` players and `M` matches, then for every player we may scan all matches again. In the worst case:

```
Time Complexity = O(P × M)
```

Since both values can be up to `10^5`, this becomes far too slow.

### Optimal Approach

The key observation is that we only care about one statistic for each player:

```
How many matches did this player lose?
```

We do not actually need to store every match after processing it.

A hash map is ideal here because it allows us to maintain each player's loss count in constant average time.

While iterating through the matches:

- Ensure every winner exists in the map with at least `0` losses.
- Increment the loser's loss count by `1`.

After processing all matches, we simply iterate through the map:

- Players with value `0` go into the zero-loss list.
- Players with value `1` go into the one-loss list.

Finally, sort both lists.

This reduces the problem to essentially one pass over the matches plus sorting.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(P × M) | O(P) | Recounts losses for every player by rescanning matches |
| Optimal | O(M + P log P) | O(P) | Uses a hash map to count losses efficiently |

Here:

- `M` is the number of matches.
- `P` is the number of unique players.

## Algorithm Walkthrough

1. Create a hash map called `loss_count`.

The keys will be player IDs, and the values will represent how many times that player has lost.
2. Iterate through every match in `matches`.

For each match:

- Let `winner = match[0]`
- Let `loser = match[1]`
3. Ensure the winner exists in the hash map.

Winners may never lose any games, but they still need to appear in the final result.

If the winner is not already in the map, initialize:

```
loss_count[winner] = 0
```
4. Increment the loser's loss count.

Since the loser lost one more game:

```
loss_count[loser] += 1
```

If the loser is not already in the map, initialize it first.
5. After processing all matches, create two result lists:

- `zero_losses`
- `one_loss`
6. Iterate through every `(player, losses)` pair in the hash map.

- If `losses == 0`, append the player to `zero_losses`.
- If `losses == 1`, append the player to `one_loss`.
7. Sort both result lists in increasing order.
8. Return:

```
[zero_losses, one_loss]
```

### Why it works

The algorithm maintains the invariant that after processing every match, `loss_count[player]` equals the exact number of matches that player has lost so far.

Each match contributes exactly one loss to exactly one player, and we process every match once. Therefore, after the full traversal, the map contains the correct final loss counts for all participating players.

Filtering players by counts `0` and `1` therefore produces exactly the required answer.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Solution:
    def findWinners(self, matches: List[List[int]]) -> List[List[int]]:
        loss_count = defaultdict(int)

        for winner, loser in matches:
            # Ensure winners are included even if they never lose
            if winner not in loss_count:
                loss_count[winner] = 0

            # Increment loser loss count
            loss_count[loser] += 1

        zero_losses = []
        one_loss = []

        for player, losses in loss_count.items():
            if losses == 0:
                zero_losses.append(player)
            elif losses == 1:
                one_loss.append(player)

        zero_losses.sort()
        one_loss.sort()

        return [zero_losses, one_loss]
```

The implementation directly follows the algorithm described earlier.

The `defaultdict(int)` automatically initializes missing keys to `0`, which simplifies counting losses. However, winners still need explicit initialization because a winner who never loses would otherwise never appear in the map.

The first loop processes every match exactly once. During this pass:

- Winners are guaranteed to exist in the map.
- Losers have their loss counts incremented.

After all matches are processed, the map contains complete loss statistics for every participating player.

The second loop filters players into the two required categories:

- Zero losses
- Exactly one loss

Finally, both lists are sorted because the problem requires increasing order.

## Go Solution

```go
package main

import "sort"

func findWinners(matches [][]int) [][]int {
	lossCount := make(map[int]int)

	for _, match := range matches {
		winner := match[0]
		loser := match[1]

		// Ensure winner exists in map
		if _, exists := lossCount[winner]; !exists {
			lossCount[winner] = 0
		}

		// Increment loser loss count
		lossCount[loser]++
	}

	zeroLosses := []int{}
	oneLoss := []int{}

	for player, losses := range lossCount {
		if losses == 0 {
			zeroLosses = append(zeroLosses, player)
		} else if losses == 1 {
			oneLoss = append(oneLoss, player)
		}
	}

	sort.Ints(zeroLosses)
	sort.Ints(oneLoss)

	return [][]int{zeroLosses, oneLoss}
}
```

The Go implementation mirrors the Python solution closely.

One important difference is that Go maps return the zero value for missing keys automatically. However, winners still need explicit insertion into the map so that undefeated players are included in the final result.

Slices are used for the result lists, and `sort.Ints()` is used to satisfy the sorted-order requirement.

There are no integer overflow concerns because the maximum loss count is at most `10^5`, which easily fits inside Go's `int` type.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(M + P log P) | One pass to count losses, plus sorting result lists |
| Space | O(P) | Hash map stores loss counts for all unique players |

The counting phase processes each match exactly once, giving `O(M)` time complexity.

Sorting dominates the remaining work. In the worst case, many players may belong to the output lists, leading to `O(P log P)` sorting time.

The hash map stores one entry per unique player, so the auxiliary space complexity is `O(P)`.

## Edge Cases

One important edge case is when a player only appears as a winner and never as a loser. A buggy implementation might only track losers, causing undefeated players to disappear entirely from the result. This implementation avoids that by explicitly inserting every winner into the hash map with an initial loss count of zero.

Another important edge case occurs when no player has exactly one loss. In this situation, the second output list should be empty rather than omitted or set to `null`. The implementation correctly initializes an empty list and only appends players whose loss count equals one.

A third important edge case is when a player loses many times. Some incorrect implementations may accidentally include players with multiple losses in the one-loss list if counts are not updated properly. Since this solution increments the loss count for every loss and only selects players with value exactly equal to one, players with two or more losses are correctly excluded.

A final subtle case involves sorting. Hash maps do not preserve ordering in either Python or Go. Without explicit sorting, the output order would be nondeterministic and would fail the problem requirements. The implementation therefore sorts both result lists before returning them.
