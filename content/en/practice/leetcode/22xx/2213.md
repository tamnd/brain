[LeetCode Problem 2213](https://leetcode.com/problems/longest-substring-of-one-repeating-character/)

**Difficulty:** 🔴 Hard  
**Topics:** Array, String, Segment Tree, Ordered Set

## Solution

## Problem Understanding

The problem gives us a string `s` and a sequence of update queries. Each query changes exactly one character in the string. After every update, we must determine the length of the longest contiguous substring that consists of only one repeated character.

For example, if the string becomes `"aaabbbbcc"`, then the answer is `4` because the substring `"bbbb"` has length `4`.

The important detail is that every query permanently modifies the string. Queries are not independent. Each update changes the current state of the string, and the next query operates on that updated version.

The input consists of:

- A string `s` of length up to `10^5`
- A string `queryCharacters`, where `queryCharacters[i]` is the character used in the `i`th update
- An array `queryIndices`, where `queryIndices[i]` is the index to modify in the `i`th update

For each query:

1. Replace `s[queryIndices[i]]` with `queryCharacters[i]`
2. Compute the maximum length of any contiguous block of equal characters
3. Store that value in the result array

The constraints are large:

- Up to `10^5` characters
- Up to `10^5` queries

These constraints immediately rule out any solution that scans the entire string after every update. A naive `O(n)` scan per query would become `O(n * k)`, which could reach `10^10` operations in the worst case.

Several edge cases are important:

- Updating a character to the same value
- Merging two large blocks into one
- Splitting an existing large block
- Single-character strings
- Strings where all characters are already identical
- Alternating character patterns such as `"abababab"`

A correct solution must efficiently maintain contiguous segment information after each update.

## Approaches

### Brute Force Approach

The most straightforward solution is to apply each query directly to the string, then scan the entire string to find the longest contiguous run of equal characters.

After every update:

1. Modify the character at the specified index
2. Traverse the string from left to right
3. Track the current run length
4. Maintain the maximum run length encountered

This works because scanning the string guarantees that every contiguous block is examined exactly once.

However, the complexity is too large. Each query requires `O(n)` work, and there can be `10^5` queries. The total complexity becomes `O(n * k)`, which is far too slow.

### Optimal Approach, Segment Tree

The key observation is that every query changes only one character. Most of the string remains unchanged after an update.

Instead of recomputing the answer from scratch, we can maintain enough information in a segment tree so that updates only affect `O(log n)` nodes.

For every segment tree node, we store:

- The leftmost character
- The rightmost character
- The longest repeating prefix length
- The longest repeating suffix length
- The maximum repeating substring inside the segment

When combining two child segments:

- If the boundary characters match, repeating runs may merge across the middle
- Otherwise, the runs remain separate

This allows us to recompute segment information efficiently after updates.

The root node always contains the answer for the entire string.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(nk) | O(1) | Scan the entire string after every query |
| Optimal | O((n + k) log n) | O(n) | Segment tree maintains longest repeating runs dynamically |

## Algorithm Walkthrough

### Segment Tree Node Information

Each node in the segment tree represents a substring.

For every node, we maintain:

- `left_char`, the first character in the segment
- `right_char`, the last character in the segment
- `prefix`, the length of the longest repeating prefix
- `suffix`, the length of the longest repeating suffix
- `best`, the longest repeating substring anywhere inside the segment
- `length`, the total segment length

This information is enough to merge two child segments into their parent.

### Step 1: Build the Segment Tree

We recursively split the string into halves until reaching individual characters.

A leaf node represents a single character:

- `prefix = 1`
- `suffix = 1`
- `best = 1`

Then we merge children upward to compute parent information.

### Step 2: Merge Two Child Segments

Suppose we have a left child and a right child.

We compute:

- Parent left character from the left child
- Parent right character from the right child

The initial maximum is:

```
max(left.best, right.best)
```

If the boundary characters match:

```
left.right_char == right.left_char
```

then a repeating substring can extend across the boundary.

The merged middle length becomes:

```
left.suffix + right.prefix
```

We update the maximum accordingly.

We also compute merged prefixes and suffixes carefully:

- The prefix expands if the entire left segment is uniform and matches the right prefix
- The suffix expands symmetrically

### Step 3: Process Updates

For each query:

1. Update the target character in the leaf node
2. Recompute all affected ancestors
3. Read the answer from the root node's `best`

Only `O(log n)` nodes are updated.

### Step 4: Store Results

After every update, append the root node's `best` value to the result array.

### Why it works

The segment tree maintains complete information about repeating runs inside every segment.

The crucial invariant is that every node correctly stores:

- The longest repeating prefix
- The longest repeating suffix
- The maximum repeating substring inside the segment

When merging two children, the only possible new repeating substring that did not already exist must cross the boundary between the two children. The merge operation explicitly checks this case.

Since updates only affect one root-to-leaf path, recomputing along that path preserves correctness globally.

## Python Solution

```python
from typing import List

class Solution:
    def longestRepeating(
        self,
        s: str,
        queryCharacters: str,
        queryIndices: List[int]
    ) -> List[int]:

        n = len(s)
        chars = list(s)

        size = 4 * n

        left_char = [''] * size
        right_char = [''] * size
        prefix = [0] * size
        suffix = [0] * size
        best = [0] * size
        length = [0] * size

        def merge(node: int) -> None:
            left_node = node * 2
            right_node = node * 2 + 1

            length[node] = (
                length[left_node] + length[right_node]
            )

            left_char[node] = left_char[left_node]
            right_char[node] = right_char[right_node]

            prefix[node] = prefix[left_node]
            if (
                prefix[left_node] == length[left_node]
                and right_char[left_node] == left_char[right_node]
            ):
                prefix[node] += prefix[right_node]

            suffix[node] = suffix[right_node]
            if (
                suffix[right_node] == length[right_node]
                and right_char[left_node] == left_char[right_node]
            ):
                suffix[node] += suffix[left_node]

            best[node] = max(
                best[left_node],
                best[right_node]
            )

            if right_char[left_node] == left_char[right_node]:
                best[node] = max(
                    best[node],
                    suffix[left_node] + prefix[right_node]
                )

        def build(node: int, left: int, right: int) -> None:
            if left == right:
                ch = chars[left]

                left_char[node] = ch
                right_char[node] = ch

                prefix[node] = 1
                suffix[node] = 1
                best[node] = 1
                length[node] = 1
                return

            mid = (left + right) // 2

            build(node * 2, left, mid)
            build(node * 2 + 1, mid + 1, right)

            merge(node)

        def update(
            node: int,
            left: int,
            right: int,
            index: int,
            value: str
        ) -> None:

            if left == right:
                chars[index] = value

                left_char[node] = value
                right_char[node] = value

                prefix[node] = 1
                suffix[node] = 1
                best[node] = 1
                return

            mid = (left + right) // 2

            if index <= mid:
                update(node * 2, left, mid, index, value)
            else:
                update(node * 2 + 1, mid + 1, right, index, value)

            merge(node)

        build(1, 0, n - 1)

        answer = []

        for ch, idx in zip(queryCharacters, queryIndices):
            update(1, 0, n - 1, idx, ch)
            answer.append(best[1])

        return answer
```

The implementation begins by converting the string into a mutable character list. Since Python strings are immutable, updates would otherwise be expensive.

Several arrays store segment tree node information:

- `left_char`
- `right_char`
- `prefix`
- `suffix`
- `best`
- `length`

Using parallel arrays instead of custom objects improves performance and avoids object allocation overhead.

The `build` function constructs the segment tree recursively. Leaf nodes correspond to single characters, while internal nodes are formed by merging children.

The `merge` function is the core of the solution. It computes how repeating substrings combine across child boundaries. This is where the segment tree maintains the invariant needed for correctness.

The `update` function modifies one leaf node and recomputes all ancestors on the path back to the root.

After every update, the answer for the whole string is stored in `best[1]`, the root node.

## Go Solution

```go
package main

func longestRepeating(s string, queryCharacters string, queryIndices []int) []int {
	n := len(s)

	chars := []byte(s)

	size := 4 * n

	leftChar := make([]byte, size)
	rightChar := make([]byte, size)

	prefix := make([]int, size)
	suffix := make([]int, size)
	best := make([]int, size)
	length := make([]int, size)

	var merge func(int)

	merge = func(node int) {
		leftNode := node * 2
		rightNode := node*2 + 1

		length[node] = length[leftNode] + length[rightNode]

		leftChar[node] = leftChar[leftNode]
		rightChar[node] = rightChar[rightNode]

		prefix[node] = prefix[leftNode]

		if prefix[leftNode] == length[leftNode] &&
			rightChar[leftNode] == leftChar[rightNode] {
			prefix[node] += prefix[rightNode]
		}

		suffix[node] = suffix[rightNode]

		if suffix[rightNode] == length[rightNode] &&
			rightChar[leftNode] == leftChar[rightNode] {
			suffix[node] += suffix[leftNode]
		}

		if best[leftNode] > best[rightNode] {
			best[node] = best[leftNode]
		} else {
			best[node] = best[rightNode]
		}

		if rightChar[leftNode] == leftChar[rightNode] {
			combined := suffix[leftNode] + prefix[rightNode]

			if combined > best[node] {
				best[node] = combined
			}
		}
	}

	var build func(int, int, int)

	build = func(node int, left int, right int) {
		if left == right {
			ch := chars[left]

			leftChar[node] = ch
			rightChar[node] = ch

			prefix[node] = 1
			suffix[node] = 1
			best[node] = 1
			length[node] = 1

			return
		}

		mid := (left + right) / 2

		build(node*2, left, mid)
		build(node*2+1, mid+1, right)

		merge(node)
	}

	var update func(int, int, int, int, byte)

	update = func(node int, left int, right int, index int, value byte) {
		if left == right {
			chars[index] = value

			leftChar[node] = value
			rightChar[node] = value

			prefix[node] = 1
			suffix[node] = 1
			best[node] = 1

			return
		}

		mid := (left + right) / 2

		if index <= mid {
			update(node*2, left, mid, index, value)
		} else {
			update(node*2+1, mid+1, right, index, value)
		}

		merge(node)
	}

	build(1, 0, n-1)

	result := make([]int, 0, len(queryIndices))

	for i := 0; i < len(queryIndices); i++ {
		update(1, 0, n-1, queryIndices[i], queryCharacters[i])
		result = append(result, best[1])
	}

	return result
}
```

The Go implementation follows the same logic as the Python version, but uses slices and byte arrays for efficiency.

Strings in Go are immutable, so the code converts the input string into a `[]byte` slice before processing updates.

Go does not support nested functions with implicit recursion as conveniently as Python, so recursive closures are declared with `var functionName func(...)` before assignment.

Integer overflow is not a concern because all values remain within `10^5`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O((n + k) log n) | Building costs O(n), each update costs O(log n) |
| Space | O(n) | Segment tree storage requires linear memory |

The segment tree contains a constant amount of information per node, and the total number of nodes is proportional to `4n`.

Each query updates only one path from the root to a leaf, so only `O(log n)` nodes need recomputation.

## Edge Cases

One important edge case occurs when the update does not actually change the character. For example, changing `"aaaa"` at index `2` to `'a'`. A careless implementation might still incorrectly recompute values or break invariants. This implementation handles the case naturally because the leaf node is simply rewritten with identical information, and merging still preserves correctness.

Another tricky case occurs when an update merges two large blocks together. For example, changing `"aabaa"` at index `2` from `'b'` to `'a'` creates `"aaaaa"`. The longest repeating substring suddenly jumps from `2` to `5`. The segment tree correctly detects this because the merge operation explicitly checks whether the rightmost character of the left child matches the leftmost character of the right child.

A third important edge case occurs when an update splits an existing maximal block. For example, changing `"aaaaa"` at index `2` to `'b'` creates `"aabaa"`. The maximum repeating substring decreases dramatically. The segment tree handles this correctly because the affected leaf is updated first, then all ancestor nodes recompute their prefix, suffix, and best values from scratch using current child information.

Another subtle case involves alternating strings such as `"abababab"`. Many updates may repeatedly merge and split distant regions. Since every operation affects only `O(log n)` nodes, the implementation remains efficient even under worst-case query sequences.
