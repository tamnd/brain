## Solution

## Problem Understanding

This problem describes a linear sequence of game levels, each of which reduces your health by a fixed amount given in the array `damage`. You start with some initial health, and after completing each level, your health is reduced by that level’s damage. At any point during the game, your health must remain strictly greater than zero in order to continue.

You are also given a single-use armor ability that can be applied to at most one level. When used on a level, it reduces the damage of that level by up to `armor` points. If the level’s damage is smaller than `armor`, the armor reduces it to zero; otherwise, it reduces it by exactly `armor`.

The task is to determine the minimum starting health required so that, after applying damage sequentially and optimally using the armor once, you never drop to zero or below.

The constraints indicate that `n` can be as large as `10^5`, and damage values can be up to `10^5`, which strongly suggests an O(n) solution is required. Any solution that tries to simulate all possible placements of armor in a naive way will be too slow.

Important edge cases include scenarios where armor is zero, where armor is larger than any single damage value, where all damage values are identical, and where the array has only one element. Another subtle case is when the maximum damage occurs at multiple indices, although only one usage of armor is allowed so only one such occurrence can be reduced.

## Approaches

### Brute Force Approach

The brute force idea is to try placing the armor at every possible level and compute the resulting total damage for each choice. For each index `i`, we reduce `damage[i]` by `min(armor, damage[i])`, then compute the total damage of the entire array and determine how much initial health is required, which is total damage plus one.

This approach is correct because it exhaustively explores all possible positions for the armor and picks the best one. However, recomputing the total damage for every index leads to inefficiency. If we recompute the sum for each position, the complexity becomes O(n^2), which is too slow for large inputs.

### Key Insight for Optimal Solution

The key observation is that the total damage is fixed regardless of where armor is applied. The only thing that changes is how much damage we are able to negate, and that negation is maximized when the armor is applied to the largest single damage value in the array.

This is because reducing a larger number yields a greater benefit than reducing a smaller number, and the armor can only be applied once. Therefore, the optimal strategy is to find the maximum element in the array and reduce it by `min(armor, max_damage)`.

This transforms the problem into a simple linear scan: compute total damage and subtract the best possible armor benefit.

### Comparison of Approaches

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^2) | O(1) | Try armor at every index and recompute total damage each time |
| Optimal | O(n) | O(1) | Sum all damage and subtract armor applied to maximum element |

## Algorithm Walkthrough

1. First, compute the total damage by summing all elements in the `damage` array. This represents the health loss if no armor is used at all, and it forms the baseline requirement.
2. While computing the total, also track the maximum single damage value in the array. This is important because the armor should be applied where it provides the highest reduction.
3. Determine the effective reduction provided by armor. Since armor can reduce at most `armor` damage, and cannot reduce more than the value of a single hit, the actual reduction is `min(armor, max_damage)`.
4. Subtract this effective reduction from the total damage. This gives the minimum possible total damage after optimally using armor.
5. Add 1 to the resulting value. This ensures that the health remains strictly greater than zero after taking all damage.
6. Return this final value as the minimum required starting health.

### Why it works

The correctness relies on the invariant that total damage is independent of armor placement, and the only optimization available is reducing exactly one element by up to `armor`. Since the final health requirement depends only on the total damage, maximizing the reduction on a single largest element guarantees the minimal required starting health.

## Python Solution

```python
from typing import List

class Solution:
    def minimumHealth(self, damage: List[int], armor: int) -> int:
        total_damage = 0
        max_damage = 0

        for d in damage:
            total_damage += d
            if d > max_damage:
                max_damage = d

        reduction = min(armor, max_damage)
        return total_damage - reduction + 1
```

The implementation first computes both the total damage and the maximum single damage in one pass through the array. It then computes how much damage the armor can actually negate, which is bounded by both `armor` and the maximum element. Finally, it adjusts the total damage accordingly and adds one to ensure survival.

## Go Solution

```go
func minimumHealth(damage []int, armor int) int64 {
    var total int64 = 0
    var maxDamage int64 = 0

    for _, d := range damage {
        total += int64(d)
        if int64(d) > maxDamage {
            maxDamage = int64(d)
        }
    }

    reduction := int64(armor)
    if reduction > maxDamage {
        reduction = maxDamage
    }

    return total - reduction + 1
}
```

In the Go implementation, we explicitly use `int64` to avoid overflow because the sum of damage values can exceed the range of a 32-bit integer. The logic mirrors the Python version exactly: compute total damage, track the maximum element, compute effective armor reduction, and return the adjusted result.

## Worked Examples

### Example 1

`damage = [2,7,4,3], armor = 4`

We compute total damage as 2 + 7 + 4 + 3 = 16. The maximum damage is 7. Armor can reduce at most 4, so reduction = 4.

Final required health = 16 - 4 + 1 = 13.

## Edge Cases

One important edge case is when the `damage` array contains only a single element. In this situation, the armor should be applied directly to that element, and the result becomes `damage[0] - min(armor, damage[0]) + 1`. This case is important because some implementations mistakenly assume multiple levels exist and overcomplicate the logic.

Another edge case occurs when `armor` is zero. In this case, the problem reduces to simply summing all damage values and adding one. Any attempt to apply reduction logic must correctly handle zero without introducing negative reductions or incorrect clamping.

A third edge case is when `armor` is larger than all elements in the array. Here, the optimal use of armor fully cancels the largest element. The implementation must ensure that the reduction is clamped to the maximum element rather than subtracting the full armor value, which would incorrectly reduce total damage below the theoretical minimum.
