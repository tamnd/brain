# LeetCode 2270, Number of Ways to Split Array

## Problem Understanding

The problem gives us a 0-indexed integer array `nums` and asks us to count how many indices can be used to split the array into two non-empty parts such that:

- The sum of the left part is greater than or equal to the sum of the right part.
- The split must leave at least one element on the right side.

If we split at index `i`, then:

- The left subarray is `nums[0:i+1]`
- The right subarray is `nums[i+1:n]`

We must determine whether:

$$\text{leftSum} \geq \text{rightSum}$$

and count how many indices satisfy this condition.

The array length can be as large as $10^5$, which means we cannot repeatedly compute sums for every split using nested loops. A quadratic solution would be too slow.

The values in the array may also be negative, which is important because we cannot make assumptions such as sums always increasing. Every split must be evaluated carefully.

A few important edge cases immediately stand out:

- Arrays containing negative values, because prefix sums may decrease.
- Arrays where every split is valid.
- Arrays where no split is valid.
- Very small arrays of length 2, where there is only one possible split.
- Large arrays near the constraint limit, where efficiency becomes critical.

The problem guarantees that the array contains at least two elements, so there is always at least one possible split position.

## Approaches

### Brute Force Approach

The most straightforward method is to examine every possible split index and explicitly compute the sum of the left and right parts.

For every index `i` from `0` to `n - 2`:

1. Compute the sum of elements from `0` to `i`
2. Compute the sum of elements from `i + 1` to `n - 1`
3. Compare the two sums
4. Increment the answer if the condition is satisfied

This approach is correct because it directly follows the definition of a valid split.

However, computing sums repeatedly is expensive. Each split may require traversing much of the array again, leading to a total complexity of:

$$O(n^2)$$

With $n = 10^5$, this becomes far too slow.

### Optimal Prefix Sum Approach

The key observation is that the left and right sums are closely related.

If we already know the total sum of the entire array, then:

$$\text{rightSum} = \text{totalSum} - \text{leftSum}$$

So instead of recomputing sums repeatedly, we can:

1. Compute the total array sum once
2. Traverse the array while maintaining a running prefix sum
3. At each split position:

- `leftSum` is the running prefix sum
- `rightSum` is `totalSum - leftSum`
4. Check whether `leftSum >= rightSum`

This reduces the complexity to linear time.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Recomputes left and right sums for every split |
| Optimal | O(n) | O(1) | Uses running prefix sum and total sum |

## Algorithm Walkthrough

### Optimal Prefix Sum Algorithm

1. Compute the total sum of the array.

This allows us to derive the right-side sum at any split position without recalculating it from scratch.
2. Initialize two variables:

- `prefixSum = 0`
- `validSplits = 0`

The `prefixSum` stores the sum of elements seen so far.
3. Iterate through the array from index `0` to `n - 2`.

We stop at `n - 2` because the right side must contain at least one element.
4. Add the current element to `prefixSum`.

After this update, `prefixSum` represents the sum of the left partition.
5. Compute the right-side sum.

Since the total array sum is already known:

$$\text{rightSum} = \text{totalSum} - \text{prefixSum}$$
6. Compare the two sums.

If:

$$\text{prefixSum} \geq \text{rightSum}$$

then the current split is valid, so increment the answer.
7. Continue until all possible split positions are processed.
8. Return the number of valid splits.

### Why it works

The algorithm maintains the invariant that `prefixSum` always equals the sum of the left partition for the current split index.

Because the total sum never changes, subtracting `prefixSum` from `totalSum` always gives the exact sum of the right partition.

Every valid split is checked exactly once, and no split is skipped or double-counted, which guarantees correctness.

## Python Solution

```python
from typing import List

class Solution:
    def waysToSplitArray(self, nums: List[int]) -> int:
        total_sum = sum(nums)
        
        prefix_sum = 0
        valid_splits = 0
        
        for i in range(len(nums) - 1):
            prefix_sum += nums[i]
            
            right_sum = total_sum - prefix_sum
            
            if prefix_sum >= right_sum:
                valid_splits += 1
        
        return valid_splits
```

The implementation begins by computing `total_sum`, which represents the sum of the entire array.

The variable `prefix_sum` is updated as we move through the array. At each position, it represents the sum of the left partition.

The loop stops at `len(nums) - 1` because the right side must remain non-empty.

For every split position, the right-side sum is derived using:

```python
right_sum = total_sum - prefix_sum
```

This avoids recomputing sums repeatedly.

Whenever the left sum is greater than or equal to the right sum, the counter `valid_splits` is incremented.

Finally, the method returns the total number of valid splits.

## Go Solution

```go
func waysToSplitArray(nums []int) int {
    totalSum := 0
    
    for _, num := range nums {
        totalSum += num
    }
    
    prefixSum := 0
    validSplits := 0
    
    for i := 0; i < len(nums)-1; i++ {
        prefixSum += nums[i]
        
        rightSum := totalSum - prefixSum
        
        if prefixSum >= rightSum {
            validSplits++
        }
    }
    
    return validSplits
}
```

The Go implementation follows the same logic as the Python version.

One implementation detail worth noting is integer handling. Go's `int` type is sufficient here because the maximum possible array sum is within safe bounds:

$$10^5 \times 10^5 = 10^{10}$$

On modern 64-bit systems, `int` safely supports this range. If absolute portability were required across 32-bit systems, `int64` could be used instead.

The slice traversal uses Go's standard indexed loop because we need to stop before the last index.

## Edge Cases

### Minimum Length Array

An array of length 2 has exactly one possible split. A buggy implementation might accidentally skip this case or incorrectly iterate beyond bounds.

For example:

```
[1, 1]
```

The implementation handles this correctly because the loop iterates from `0` to `len(nums) - 2`, which includes exactly one iteration.

### Arrays Containing Negative Numbers

Negative values are important because prefix sums are not guaranteed to increase.

For example:

```
[10, 4, -8, 7]
```

The left sum decreases after including `-8`.

A naive assumption that sums always grow would fail here. The implementation recomputes the comparison at every position using exact arithmetic, so negative values are handled naturally.

### All Splits Valid

Some arrays satisfy the condition at every possible split.

For example:

```
[0, 0, 0, 0]
```

Every split produces equal sums, and equality is allowed because the condition is:

```
leftSum >= rightSum
```

The implementation correctly counts every valid position.

### No Valid Splits

Some arrays may have no valid split positions.

For example:

```
[1, 10, 20]
```

Every left-side sum is smaller than the corresponding right-side sum.

The algorithm correctly returns `0` because the counter is only incremented when the condition is satisfied.
