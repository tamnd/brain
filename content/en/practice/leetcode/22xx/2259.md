# LeetCode 2259 - Remove Digit From Number to Maximize Result

## Problem Understanding

The problem gives us a string called `number` that represents a positive integer, along with a character `digit`. The task is to remove exactly one occurrence of `digit` from the string so that the resulting number is as large as possible.

Since the input number is provided as a string, we are not performing arithmetic operations directly. Instead, we are manipulating characters inside the string. Every removal produces a new string with length `number.length - 1`, and we want the lexicographically largest valid result, which also corresponds to the numerically largest value because all strings contain only digits and have the same length after removal.

For example, if `number = "1231"` and `digit = "1"`, there are two possible removals:

- Remove the first `'1'` → `"231"`
- Remove the second `'1'` → `"123"`

Since `231 > 123`, the correct answer is `"231"`.

The constraints are very small:

- `2 <= number.length <= 100`
- `digit` always appears at least once

These limits tell us that even a brute-force solution is completely acceptable. However, there is also a cleaner greedy observation that leads to a more elegant optimal solution.

Several edge cases are important:

- The digit may appear only once.
- Multiple removals may produce the same result.
- The best removal is not always the first occurrence.
- The best removal depends on the digits that come after the removed position.
- All digits are between `'1'` and `'9'`, so there are no leading zero complications.

## Approaches

### Brute Force Approach

The most direct solution is to try removing every occurrence of `digit`.

For each index where `number[i] == digit`, we construct a new string:

```
number[:i] + number[i+1:]
```

We compare all generated candidates and keep the largest one.

This works because we explicitly evaluate every valid possibility, so the maximum candidate must be the correct answer.

Since the maximum string length is only 100, generating all candidates is inexpensive.

### Greedy Observation

We can do better conceptually with a greedy strategy.

Suppose we are scanning the string from left to right. If we find an occurrence of `digit` where the next digit is larger than the current one, removing the current digit immediately increases the value of the number.

For example:

```
5512, remove 5
```

If we remove the first `5`, we get:

```
512
```

If we remove the second `5`, we get:

```
552
```

The second result is larger because keeping the earlier larger prefix is beneficial.

The key insight is:

- If the next digit is larger than the current `digit`, removing the current digit shifts that larger digit forward into a more significant position.
- Therefore, the first position where `number[i] < number[i+1]` after matching `digit` is the optimal removal.
- If no such position exists, we should remove the last occurrence of `digit`.

This produces a linear-time greedy solution.

| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Generate every possible removal and choose the maximum |
| Optimal | O(n) | O(n) | Greedily remove the best occurrence during one scan |

## Algorithm Walkthrough

### Optimal Greedy Algorithm

1. Traverse the string from left to right.
2. Whenever the current character equals `digit`, check the next character.
3. If the next character exists and is larger than the current digit, remove the current occurrence immediately.

This works because moving a larger digit toward the front increases the overall value of the number.
4. If no such position is found during the scan, remove the last occurrence of `digit`.

In this case, digits after each occurrence are not larger, so preserving earlier digits gives the largest result.
5. Return the constructed string.

### Why it works

The algorithm relies on positional significance in decimal numbers. Earlier digits contribute more to the final value than later digits. When a digit smaller than the following digit is removed, the larger following digit shifts into a more significant position, increasing the number as much as possible.

If no occurrence satisfies this condition, then the digits after each occurrence are less than or equal to the removed digit. In that situation, keeping earlier occurrences preserves a larger prefix, so removing the last occurrence gives the maximum result.

## Python Solution

```python
class Solution:
    def removeDigit(self, number: str, digit: str) -> str:
        last_index = -1

        for i in range(len(number)):
            if number[i] == digit:
                last_index = i

                if i + 1 < len(number) and number[i + 1] > digit:
                    return number[:i] + number[i + 1:]

        return number[:last_index] + number[last_index + 1:]
```

The implementation maintains the index of the last occurrence of `digit` while scanning the string.

During traversal, whenever the current character matches `digit`, the algorithm checks whether the next digit is larger. If it is, the current occurrence is immediately removed and returned because this is guaranteed to produce the optimal answer.

If the scan finishes without finding such a position, the algorithm removes the last occurrence stored in `last_index`.

String slicing is used to construct the result:

```python
number[:i] + number[i + 1:]
```

This creates a new string without the selected character.

## Go Solution

```go
func removeDigit(number string, digit byte) string {
    lastIndex := -1

    for i := 0; i < len(number); i++ {
        if number[i] == digit {
            lastIndex = i

            if i+1 < len(number) && number[i+1] > digit {
                return number[:i] + number[i+1:]
            }
        }
    }

    return number[:lastIndex] + number[lastIndex+1:]
}
```

The Go implementation follows the same greedy logic as the Python solution.

A `byte` is used for the digit because Go strings are byte-indexable and the input contains only ASCII digits. String slicing works similarly to Python slicing, producing substrings without the selected character.

There are no integer overflow concerns because the algorithm never converts the string into a numeric type.

## Edge Cases

One important edge case occurs when the digit appears exactly once. In this situation, there is no decision to make because only one valid removal exists. A buggy implementation might assume multiple occurrences exist or fail to track the only valid index correctly. The current implementation naturally handles this because the single occurrence becomes the stored `last_index`.

Another important case happens when every occurrence of the digit is followed by a smaller or equal digit. In this scenario, the greedy condition never triggers. Removing the last occurrence is correct because it preserves the largest possible prefix. The implementation explicitly handles this by storing the latest matching index and removing it after the scan completes.

A third edge case involves repeated identical digits, such as `"1111"` with digit `"1"`. Every possible removal produces the same result. Some implementations may incorrectly depend on strict comparisons or produce inconsistent behavior. The current algorithm works correctly because no greedy improvement exists, so the last occurrence is removed, still yielding the optimal answer.

A final subtle case occurs when removing an earlier occurrence allows a larger digit to move forward immediately. For example, in `"1231"` removing the first `'1'` shifts `'2'` into the leading position. The greedy condition captures this precisely by checking whether the next digit is larger than the current removable digit.
