## Solution

## Problem Understanding

The problem asks us to transform an integer array into a “beautiful” array using the minimum number of deletions. A beautiful array is defined by two conditions. First, its final length must be even. Second, for every valid index `i` where `i % 2 == 0`, the element at position `i` must not be equal to the next element, meaning `nums[i] != nums[i + 1]`.

We are allowed to delete any elements from the original array, and when we delete an element, the remaining elements shift left while preserving their relative order. The goal is to minimize the number of deletions so that the resulting array satisfies both conditions simultaneously.

The constraints indicate that `nums.length` can be as large as 100,000, so any solution worse than linear time will not be feasible. This immediately suggests that we need a greedy or single-pass approach rather than any combinational or dynamic programming over subsets.

Important edge cases include arrays of length 1, where we must delete the single element to achieve even length (empty array is valid), arrays where all elements are identical, which forces many deletions due to adjacency constraints, and already valid arrays where no deletions are needed. Another subtle case is when the array becomes valid structurally but has odd length at the end, requiring a final adjustment.

## Approaches

The brute-force approach considers every possible subset of elements, checks whether each subset is beautiful, and tracks the minimum deletions required. While this is conceptually correct, it is computationally infeasible because there are `2^n` subsets, and checking each one costs additional linear time.

The key insight for optimization is that we do not actually need to explore subsets. Instead, we can construct the resulting array greedily from left to right, maintaining the constraints as we go. The only dependency we need to enforce is that whenever we are at an even index in the resulting array, the current element must not equal the previous kept element. This allows us to decide locally whether to keep or delete each element.

The second key observation is that we do not need to explicitly build the full array; we only need to simulate its construction and count deletions.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n · n) | O(n) | Try all subsets and validate beauty condition |
| Optimal Greedy | O(n) | O(1) | Single pass construction with parity-aware selection |

## Algorithm Walkthrough

The optimal solution works by simulating the construction of the final array while scanning the input from left to right.

1. We maintain a conceptual “result array length” that reflects how many elements we have kept so far. We also maintain a deletion counter initialized to zero. The key idea is that we never explicitly need to store the result array; we only track how many elements would be kept and whether they satisfy the condition.
2. As we iterate through each element in `nums`, we decide whether to keep or delete it based on the current length of the constructed array. If the current length is even, this element will become a candidate at an even index in the final array, and there is no constraint involving a previous element at this step, so we can tentatively keep it.
3. If the current constructed length is odd, then the last kept element occupies an even index in the final array, and therefore we must ensure the new element is not equal to that last kept element. If it is equal, we must delete the current element and increment the deletion counter. Otherwise, we keep it.
4. Every time we decide to keep an element, we increment the conceptual result length. Every time we delete, we only increment the deletion counter and do not change the result length.
5. After processing all elements, we check whether the constructed length is odd. If it is, we must delete one more element from the end to satisfy the requirement that the final length is even, so we increment the deletion counter by one.

The reason we can safely adjust at the end is that removing the last element never violates earlier constraints, since it does not affect any previously validated adjacent pairs.

### Why it works

This algorithm works because it enforces the only constraint that can be violated incrementally, which is the equality condition between adjacent elements in the constructed sequence at odd positions. By greedily skipping only when necessary, we ensure that we maximize the number of kept elements while maintaining validity. The final parity adjustment ensures the even-length requirement is satisfied with minimal additional deletion.

## Python Solution

```python
from typing import List

class Solution:
    def minDeletion(self, nums: List[int]) -> int:
        deletions = 0
        result_length = 0

        for num in nums:
            if result_length % 2 == 0:
                result_length += 1
            else:
                if result_length > 0 and num == nums_used_last:
                    deletions += 1
                else:
                    result_length += 1
                    nums_used_last = num

        if result_length % 2 == 1:
            deletions += 1

        return deletions
```

The implementation above follows the greedy simulation idea. We track how many elements we have effectively kept using `result_length`. When the current position in the constructed array is even, we always accept the element. When it is odd, we only accept it if it does not match the last kept element; otherwise, we increment deletions. Finally, we ensure the length is even by possibly deleting one extra element.

However, the above snippet has an issue: `nums_used_last` must be initialized properly. A cleaner implementation avoids this complication by explicitly maintaining the last kept value.

Corrected version:

```python
from typing import List

class Solution:
    def minDeletion(self, nums: List[int]) -> int:
        deletions = 0
        result_length = 0
        last = None

        for num in nums:
            if result_length % 2 == 0:
                last = num
                result_length += 1
            else:
                if num == last:
                    deletions += 1
                else:
                    last = num
                    result_length += 1

        if result_length % 2 == 1:
            deletions += 1

        return deletions
```

The corrected version maintains `last` as the most recently kept element. The decision logic directly follows the parity rule described in the algorithm walkthrough.

## Go Solution

```go
func minDeletion(nums []int) int {
    deletions := 0
    length := 0
    var last int

    hasLast := false

    for _, num := range nums {
        if length%2 == 0 {
            last = num
            hasLast = true
            length++
        } else {
            if hasLast && num == last {
                deletions++
            } else {
                last = num
                length++
            }
        }
    }

    if length%2 == 1 {
        deletions++
    }

    return deletions
}
```

In Go, we explicitly track whether `last` is valid using a boolean flag `hasLast`, since Go does not have nullable integers. The rest of the logic mirrors the Python solution exactly, using integer counters for the constructed length and deletions.

## Worked Examples

For `nums = [1,1,2,3,5]`, we track `(length, last, deletions)`:

At `1`, length is 0 so we take it, length becomes 1, last is 1. At the next `1`, length is 1 so we compare with last which is 1, so we delete it and increment deletions to 1. At `2`, we take it since it differs from last, length becomes 2, last is 2. At `3`, length is 2 so we take it, length becomes 3, last is 3. At `5`, length is 3 so we compare with last 3, so we take it, length becomes 4, last is 5. Final length is 4, which is even, so no extra deletion is needed. Total deletions is 1.

For `nums = [1,1,2,2,3,3]`, we proceed similarly. First `1` is taken, second `1` is deleted. Then `2` is taken, second `2` is deleted. Then `3` is taken, second `3` is deleted, but since we only need a valid even-length structure, the greedy process naturally avoids unnecessary keeps. The final adjustment ensures even length, resulting in total deletions equal to 2.

## Edge Cases

One important edge case is when the array has only one element. Since a beautiful array must have even length, the only valid transformation is an empty array, so the answer is always one deletion. The algorithm correctly handles this by performing a final parity check.

Another edge case occurs when all elements are identical. In this case, every second element must be deleted during the greedy scan because the adjacency constraint is repeatedly violated. The algorithm ensures this by always comparing against the last kept value and deleting duplicates in odd positions.

A third edge case is when the array is already valid except for length parity. For example, if the constructed result ends with an odd number of elements, the final adjustment deletes exactly one element. This works safely because removing the last element cannot invalidate earlier adjacency constraints.

A final subtle edge case is alternating sequences that appear valid but break the even-index constraint after deletions shift indices. The greedy construction avoids this by always enforcing rules based on the evolving constructed sequence rather than original indices.
