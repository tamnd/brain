## Solution

## Problem Understanding

The problem gives us a string `s` consisting of lowercase English letters. We must design a custom keypad with `9` buttons, where each button can contain at most `3` characters. Every lowercase English letter must appear on exactly one button.

Typing cost depends on the position of a character on its assigned button:

- The first character on a button costs `1` keypress.
- The second character costs `2` keypresses.
- The third character costs `3` keypresses.

Our goal is to arrange the letters across the buttons so that typing the given string `s` requires the minimum total number of keypresses.

The important detail is that we are free to choose which characters go onto which buttons and in what order. Since the typing cost depends only on the position inside the button, we want the most frequently used characters to require the fewest presses.

The constraints are important:

- `1 <= s.length <= 10^5`
- Only lowercase English letters appear.

Although the input string may be large, the alphabet size is fixed at `26`. This strongly suggests that counting frequencies and sorting characters will be efficient enough.

There are several edge cases worth thinking about upfront.

If the string contains only one unique character, then that character should obviously be assigned a single keypress position. If many characters have the same frequency, multiple optimal assignments may exist, but the minimum total cost remains the same. Another important case is when the string uses all `26` letters. Since only `9` buttons exist and each button holds at most `3` letters, the capacity is exactly `27`, so all letters fit.

A naive implementation might incorrectly distribute letters without considering frequency order, which would waste low-cost positions on rare characters instead of common ones.

## Approaches

### Brute Force Approach

A brute-force solution would attempt every possible assignment of letters to keypad positions and compute the resulting typing cost.

There are up to `27` available positions:

- `9` positions costing `1`
- `9` positions costing `2`
- `9` positions costing `3`

We would need to assign up to `26` letters into these positions in every possible arrangement. Even restricting ourselves to used letters, the number of permutations becomes astronomically large.

For each arrangement, we could compute the total typing cost by multiplying each character frequency by its assigned press count.

This approach is correct because it explicitly checks all possible keypad layouts and chooses the minimum total cost. However, it is computationally impossible for realistic inputs.

### Key Insight

The critical observation is that the exact button number does not matter. Only the press count matters.

There are:

- `9` slots costing `1`
- `9` slots costing `2`
- `9` slots costing `3`

To minimize the total typing cost, the most frequent characters should occupy the cheapest positions.

This becomes a classic greedy assignment problem:

1. Count how often each character appears.
2. Sort frequencies in descending order.
3. Assign:

- The first `9` frequencies to cost `1`
- The next `9` frequencies to cost `2`
- The remaining frequencies to cost `3`

This greedy strategy is optimal because swapping a higher-frequency character into a cheaper position can only reduce or maintain the total cost.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Tries all keypad arrangements |
| Optimal | O(n + 26 log 26) | O(26) | Count frequencies and greedily assign cheapest costs |

## Algorithm Walkthrough

1. Create a frequency map for all characters in the string.

We count how many times each character appears because frequently used characters should receive lower typing costs.
2. Extract all character frequencies into a list.

We do not actually care which character is which after counting. Only the frequencies matter for optimization.
3. Sort the frequency list in descending order.

The largest frequencies should receive the smallest press counts.
4. Iterate through the sorted frequencies.

The index determines the press cost:

- Indices `0` through `8` cost `1`
- Indices `9` through `17` cost `2`
- Indices `18` and beyond cost `3`
5. Accumulate the total typing cost.

Multiply each frequency by its assigned press count and add it to the answer.

### Why it works

The greedy strategy works because lower press counts are strictly better than higher press counts. If a higher-frequency character were assigned a more expensive position while a lower-frequency character occupied a cheaper one, swapping them would reduce the total cost.

Therefore, sorting frequencies in descending order and assigning the cheapest available costs first always produces an optimal solution.

## Python Solution

```python
class Solution:
    def minimumKeypresses(self, s: str) -> int:
        frequency = {}

        for char in s:
            frequency[char] = frequency.get(char, 0) + 1

        frequencies = sorted(frequency.values(), reverse=True)

        total_presses = 0

        for index, freq in enumerate(frequencies):
            press_count = (index // 9) + 1
            total_presses += freq * press_count

        return total_presses
```

The implementation begins by building a frequency map using a dictionary. Each character in the string increases its count by one.

Next, we extract all frequency values and sort them in descending order. This ensures the most common characters are processed first.

The loop then assigns costs greedily. The expression `(index // 9) + 1` determines the typing cost group:

- Indices `0` to `8` map to cost `1`
- Indices `9` to `17` map to cost `2`
- Indices `18` to `26` map to cost `3`

For each frequency, we multiply by the assigned press count and accumulate the total.

Finally, the function returns the minimum typing cost.

## Go Solution

```go
package main

import "sort"

func minimumKeypresses(s string) int {
	frequency := make(map[rune]int)

	for _, char := range s {
		frequency[char]++
	}

	frequencies := make([]int, 0, len(frequency))

	for _, freq := range frequency {
		frequencies = append(frequencies, freq)
	}

	sort.Sort(sort.Reverse(sort.IntSlice(frequencies)))

	totalPresses := 0

	for index, freq := range frequencies {
		pressCount := (index / 9) + 1
		totalPresses += freq * pressCount
	}

	return totalPresses
}
```

The Go implementation follows the same logic as the Python solution.

A `map[rune]int` stores character frequencies. The frequencies are copied into a slice and sorted in descending order using `sort.Reverse`.

The greedy assignment logic is identical. Integer division naturally groups indices into blocks of `9`.

No special overflow handling is required because the maximum possible answer is well within Go's integer range for the given constraints.

## Edge Cases

One important edge case occurs when the string contains only one unique character, such as `"aaaaaa"`. A buggy implementation might unnecessarily assign that character to a higher-cost position. The greedy sorting ensures the most frequent character always receives a cost of `1`, producing the optimal result.

Another important case happens when there are more than `9` unique characters. The algorithm must correctly transition from `1` press positions to `2` press positions. The formula `(index // 9) + 1` handles this automatically and cleanly without requiring special conditional logic.

A third edge case is when all `26` lowercase letters appear. Since the keypad capacity is `27`, every letter still fits. The algorithm properly assigns:

- `9` letters to cost `1`
- `9` letters to cost `2`
- `8` letters to cost `3`

This guarantees all characters are assigned valid positions while still minimizing the total cost.

A subtle edge case involves ties in frequency. Multiple letters may appear the same number of times. The algorithm does not care about the exact ordering among equal frequencies because swapping equal-frequency characters does not change the total typing cost. This ensures correctness even when sorting order among ties differs between implementations.
