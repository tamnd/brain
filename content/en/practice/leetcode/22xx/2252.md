## Solution

## Problem Understanding

The problem asks us to transform a normalized database table `Products` into a pivoted table, where each row corresponds to a unique `product_id` and each column corresponds to a store. In the input table, each row contains a `product_id`, a `store`, and the `price` of that product at that store. Not all products are sold in every store, so some cells in the output will be `null`. The store columns must be ordered lexicographically, meaning sorted alphabetically.

The input has at most 30 distinct stores, which is a small enough number to allow dynamic SQL or pivoting techniques. The primary key constraint guarantees that for each `(product_id, store)` pair, there is only one row, so no ambiguity arises in the price.

Important edge cases include products that are sold in only one store, stores with no products, and situations where the store names require proper lexicographical sorting. We must also ensure that missing entries appear as `null` rather than zero or any default value.

## Approaches

A brute-force approach would manually iterate through all products and stores, filling in the corresponding price. For each product, we would check every store and insert the price if it exists or `null` otherwise. While correct, this is inefficient and cumbersome in SQL, as it requires multiple subqueries or joins for each store.

The optimal approach leverages dynamic SQL to construct a pivot query at runtime. The key observation is that because the number of stores is small (≤30), we can dynamically generate a SQL query that selects each store as a column, applying a `MAX(CASE WHEN store = '<store>' THEN price END)` construct for each. This method works efficiently because the database engine handles aggregation and pivoting internally.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(P * S) | O(P * S) | Iterates through all products P and stores S, joining manually |
| Optimal | O(P * S) | O(S) | Uses dynamic SQL pivot, minimal extra memory, relies on built-in aggregation |

## Algorithm Walkthrough

1. Identify all unique store names from the `Products` table. Sort them lexicographically to ensure proper column order.
2. Construct a dynamic SQL query string. For each store, create a column using `MAX(CASE WHEN store = '<store>' THEN price END) AS "<store>"`. This ensures that if a product does not exist in a store, the column will return `null`.
3. Combine these columns into a single SELECT query that groups by `product_id`.
4. Execute the dynamic SQL query using `EXECUTE` in the procedure.
5. Return the resulting pivoted table.

Why it works: The algorithm guarantees correctness because each product is grouped by `product_id`, and each column computes the maximum price for that store, which is effectively the single price due to the primary key. Sorting the store names ensures the columns are in the required lexicographical order. Using dynamic SQL allows the procedure to adapt to any set of stores without hardcoding column names.

## Python Solution

LeetCode SQL problems do not use Python for procedural dynamic SQL, but for completeness, we can illustrate the logic in Python as dynamic query generation.

```python
def PivotProducts(products: list[tuple[int, str, int]]) -> list[dict[str, int | None]]:
    # Collect all unique stores and sort them
    stores = sorted({store for _, store, _ in products})
    
    # Initialize a dictionary mapping product_id to store-price dict
    product_map = {}
    for product_id, store, price in products:
        if product_id not in product_map:
            product_map[product_id] = {s: None for s in stores}
        product_map[product_id][store] = price
    
    # Build result as list of dictionaries
    result = []
    for product_id, store_prices in product_map.items():
        row = {'product_id': product_id}
        row.update(store_prices)
        result.append(row)
    
    return result
```

This Python code simulates the pivot by creating a dictionary of dictionaries, filling `None` for missing stores, and producing the final list of dictionaries. It mirrors what dynamic SQL would do in a database.

## Go Solution

For completeness, here is a Go version using maps.

```go
package main

type Product struct {
	ProductID int
	Store     string
	Price     int
}

func PivotProducts(products []Product) []map[string]interface{} {
	storeSet := map[string]struct{}{}
	for _, p := range products {
		storeSet[p.Store] = struct{}{}
	}
	stores := []string{}
	for s := range storeSet {
		stores = append(stores, s)
	}
	sort.Strings(stores)

	productMap := map[int]map[string]interface{}{}
	for _, p := range products {
		if _, exists := productMap[p.ProductID]; !exists {
			productMap[p.ProductID] = map[string]interface{}{}
			for _, s := range stores {
				productMap[p.ProductID][s] = nil
			}
		}
		productMap[p.ProductID][p.Store] = p.Price
	}

	result := []map[string]interface{}{}
	for pid, sp := range productMap {
		row := map[string]interface{}{"product_id": pid}
		for _, s := range stores {
			row[s] = sp[s]
		}
		result = append(result, row)
	}

	return result
}
```

Go-specific considerations: We use `interface{}` to allow `nil` values representing missing prices. We also explicitly initialize all store columns for each product to `nil` before assigning actual prices.
