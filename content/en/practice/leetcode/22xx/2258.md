[LeetCode Problem 2258](https://leetcode.com/problems/escape-the-spreading-fire/)

**Difficulty:** 🔴 Hard  
**Topics:** Array, Binary Search, Breadth-First Search, Matrix

## Solution

## Problem Understanding

This problem combines two simultaneous processes on a grid:

1. Your movement from the top-left corner to the bottom-right corner.
2. The spread of fire throughout the grid over time.

The grid contains three types of cells:

- `0` means open grass, both you and the fire may move through it.
- `1` means the cell initially contains fire.
- `2` means a wall, neither you nor the fire may pass through it.

You start at position `(0, 0)` and want to reach the safehouse at `(m - 1, n - 1)`.

Every minute follows this exact sequence:

1. You move first, to one adjacent grass cell.
2. Then all fire cells spread simultaneously to adjacent non-wall cells.

The key detail is that reaching the safehouse is considered successful even if the fire spreads into the safehouse immediately afterward during the same minute.

The question asks for the maximum number of minutes you may wait at the starting position before beginning your movement, while still being able to safely reach the safehouse.

There are three possible outcomes:

- Return `-1` if escape is impossible even when starting immediately.
- Return `10^9` if escape is always possible no matter how long you wait.
- Otherwise, return the largest valid waiting time.

The constraints are important:

- `m * n <= 2 * 10^4`
- The grid can be as large as `300 x 300`

A naive simulation for every possible waiting time would be too expensive. We need a more efficient method.

Several edge cases are especially important:

- The fire may block all paths immediately.
- The fire may never reach the safehouse because walls isolate it.
- The player may arrive at the safehouse at the exact same minute the fire reaches it, which is allowed.
- Intermediate cells cannot be entered at the same time as the fire, only strictly before the fire arrives.
- The starting cell itself may eventually catch fire while waiting.

These rules make the timing logic subtle and easy to implement incorrectly.

## Approaches

### Brute Force Approach

A direct brute-force strategy would try every possible waiting time:

1. Wait `t` minutes.
2. Simulate fire spreading for `t` minutes.
3. Run a BFS to determine whether the player can still reach the safehouse safely.

We could repeat this process for increasing values of `t` until escape becomes impossible.

This approach is correct because it explicitly simulates the entire process exactly as described in the problem statement.

However, it is far too slow.

The maximum possible waiting time could be very large, and each simulation requires BFS traversals over the entire grid. Recomputing fire spread repeatedly becomes prohibitively expensive.

### Key Insight

The crucial observation is that fire spread times can be precomputed once.

Instead of repeatedly simulating the fire, we can calculate:

- `fireTime[r][c] = earliest minute when fire reaches cell (r, c)`

This can be computed efficiently using a multi-source BFS starting from all fire cells simultaneously.

Once we know the fire arrival times, we can determine whether a particular waiting time is feasible.

For a chosen delay `wait`:

- You start moving at minute `wait`.
- If you reach a normal cell at time `t`, then:

- `t < fireTime[cell]`
- If the cell is the safehouse:

- `t <= fireTime[safehouse]`

This transforms the problem into a decision problem:

> "Can we escape if we wait `X` minutes?"

That decision problem is monotonic:

- If waiting `X` minutes works,
- Then any smaller waiting time also works.

This monotonic property allows binary search over the answer.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O((m × n)²) or worse | O(m × n) | Repeatedly simulates fire spread and pathfinding |
| Optimal | O(m × n × log(m × n)) | O(m × n) | Uses multi-source BFS plus binary search |

## Algorithm Walkthrough

### Step 1: Precompute Fire Arrival Times

We first compute the earliest time the fire reaches every cell.

We perform a multi-source BFS:

1. Insert all initial fire cells into the queue.
2. Set their fire arrival time to `0`.
3. Expand outward level by level.
4. Walls are ignored because fire cannot pass through them.

This gives us a matrix:

```
fireTime[r][c]
```

where each entry stores the first minute the fire reaches that location.

If a cell is never reached by fire, its value remains infinity.

### Step 2: Define the Feasibility Check

We now create a function:

```
canEscape(wait)
```

This determines whether escaping is possible after waiting `wait` minutes at the start.

We perform another BFS for the player.

The queue stores:

```
(row, col, currentTime)
```

where:

```
currentTime = wait + number of moves taken
```

### Step 3: Validate the Starting Position

If the fire reaches `(0, 0)` before or at the waiting time, escape is impossible.

Specifically:

```
wait >= fireTime[0][0]
```

means the player burns before moving.

### Step 4: BFS for Player Movement

From each cell:

1. Try all four directions.
2. Ignore walls and out-of-bounds cells.
3. Compute the arrival time to the next cell.

For normal cells:

```
arrivalTime < fireTime[next]
```

must hold.

For the safehouse:

```
arrivalTime <= fireTime[safehouse]
```

is allowed because simultaneous arrival is valid there.

### Step 5: Binary Search the Maximum Waiting Time

The answer is monotonic:

- Smaller waits are easier.
- Larger waits are harder.

So we binary search over the waiting time.

Typical bounds are:

```
0 to 10^9
```

If a very large wait is feasible, we return `10^9`.

Otherwise, binary search finds the maximum valid wait.

### Why it works

The algorithm works because BFS guarantees shortest-time traversal in an unweighted grid.

The fire BFS computes the earliest possible fire arrival for every cell. Since fire spreads uniformly each minute, these times are exact.

The player BFS checks whether every move occurs before the fire arrives. Since we only allow valid transitions, any discovered path is guaranteed safe.

The monotonic property of waiting times guarantees binary search correctness. If waiting `t` minutes is possible, then waiting fewer than `t` minutes cannot make the situation worse.

## Python Solution

```python
from collections import deque
from typing import List

class Solution:
    def maximumMinutes(self, grid: List[List[int]]) -> int:
        rows = len(grid)
        cols = len(grid[0])

        INF = 10**18
        directions = [(1, 0), (-1, 0), (0, 1), (0, -1)]

        # Compute earliest fire arrival times
        fire_time = [[INF] * cols for _ in range(rows)]
        queue = deque()

        for r in range(rows):
            for c in range(cols):
                if grid[r][c] == 1:
                    fire_time[r][c] = 0
                    queue.append((r, c))

        while queue:
            r, c = queue.popleft()

            for dr, dc in directions:
                nr = r + dr
                nc = c + dc

                if (
                    0 <= nr < rows
                    and 0 <= nc < cols
                    and grid[nr][nc] != 2
                    and fire_time[nr][nc] == INF
                ):
                    fire_time[nr][nc] = fire_time[r][c] + 1
                    queue.append((nr, nc))

        def can_escape(wait: int) -> bool:
            # Fire reaches start before or at waiting time
            if wait >= fire_time[0][0]:
                return False

            queue = deque([(0, 0, wait)])
            visited = [[False] * cols for _ in range(rows)]
            visited[0][0] = True

            while queue:
                r, c, time = queue.popleft()

                if (r, c) == (rows - 1, cols - 1):
                    return True

                for dr, dc in directions:
                    nr = r + dr
                    nc = c + dc
                    nt = time + 1

                    if not (0 <= nr < rows and 0 <= nc < cols):
                        continue

                    if visited[nr][nc]:
                        continue

                    if grid[nr][nc] == 2:
                        continue

                    # Safehouse allows equal arrival time
                    if (nr, nc) == (rows - 1, cols - 1):
                        if nt > fire_time[nr][nc]:
                            continue
                    else:
                        if nt >= fire_time[nr][nc]:
                            continue

                    visited[nr][nc] = True
                    queue.append((nr, nc, nt))

            return False

        # If waiting forever works
        if can_escape(10**9):
            return 10**9

        left = 0
        right = 10**9
        answer = -1

        while left <= right:
            mid = (left + right) // 2

            if can_escape(mid):
                answer = mid
                left = mid + 1
            else:
                right = mid - 1

        return answer
```

The implementation begins by computing fire arrival times using multi-source BFS. Every initial fire cell enters the queue simultaneously, ensuring the shortest fire arrival time for every reachable cell.

The `can_escape` function then performs a BFS for the player. Each state tracks both position and current time. The time value already includes the initial waiting period.

The transition rules enforce the timing constraints carefully:

- Regular cells require strict inequality.
- The safehouse allows equality.

Finally, binary search exploits the monotonic nature of the problem to locate the maximum feasible waiting time efficiently.

## Go Solution

```go
package main

import "container/list"

func maximumMinutes(grid [][]int) int {
	rows := len(grid)
	cols := len(grid[0])

	const INF int = 1 << 60

	directions := [][2]int{
		{1, 0},
		{-1, 0},
		{0, 1},
		{0, -1},
	}

	fireTime := make([][]int, rows)
	for i := range fireTime {
		fireTime[i] = make([]int, cols)
		for j := range fireTime[i] {
			fireTime[i][j] = INF
		}
	}

	queue := list.New()

	for r := 0; r < rows; r++ {
		for c := 0; c < cols; c++ {
			if grid[r][c] == 1 {
				fireTime[r][c] = 0
				queue.PushBack([2]int{r, c})
			}
		}
	}

	// Multi-source BFS for fire
	for queue.Len() > 0 {
		front := queue.Remove(queue.Front()).([2]int)
		r, c := front[0], front[1]

		for _, d := range directions {
			nr := r + d[0]
			nc := c + d[1]

			if nr < 0 || nr >= rows || nc < 0 || nc >= cols {
				continue
			}

			if grid[nr][nc] == 2 {
				continue
			}

			if fireTime[nr][nc] != INF {
				continue
			}

			fireTime[nr][nc] = fireTime[r][c] + 1
			queue.PushBack([2]int{nr, nc})
		}
	}

	canEscape := func(wait int) bool {
		if wait >= fireTime[0][0] {
			return false
		}

		type State struct {
			r, c int
			t    int
		}

		visited := make([][]bool, rows)
		for i := range visited {
			visited[i] = make([]bool, cols)
		}

		q := list.New()
		q.PushBack(State{0, 0, wait})
		visited[0][0] = true

		for q.Len() > 0 {
			cur := q.Remove(q.Front()).(State)

			if cur.r == rows-1 && cur.c == cols-1 {
				return true
			}

			for _, d := range directions {
				nr := cur.r + d[0]
				nc := cur.c + d[1]
				nt := cur.t + 1

				if nr < 0 || nr >= rows || nc < 0 || nc >= cols {
					continue
				}

				if visited[nr][nc] {
					continue
				}

				if grid[nr][nc] == 2 {
					continue
				}

				if nr == rows-1 && nc == cols-1 {
					if nt > fireTime[nr][nc] {
						continue
					}
				} else {
					if nt >= fireTime[nr][nc] {
						continue
					}
				}

				visited[nr][nc] = true
				q.PushBack(State{nr, nc, nt})
			}
		}

		return false
	}

	if canEscape(1000000000) {
		return 1000000000
	}

	left, right := 0, 1000000000
	answer := -1

	for left <= right {
		mid := left + (right-left)/2

		if canEscape(mid) {
			answer = mid
			left = mid + 1
		} else {
			right = mid - 1
		}
	}

	return answer
}
```

The Go implementation mirrors the Python logic closely.

A few Go-specific details are worth noting:

- Go does not provide a built-in deque, so `container/list` is used for BFS queues.
- A large integer constant is used for infinity.
- Structs are used to store BFS states cleanly.
- Integer overflow is avoided by computing the midpoint as:

```go
mid := left + (right-left)/2
```

rather than `(left + right) / 2`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m × n × log(m × n)) | BFS for fire plus binary search with BFS checks |
| Space | O(m × n) | Fire matrix, visited matrix, and BFS queues |

The fire BFS visits each cell at most once.

Each feasibility check also performs a BFS over the grid, costing `O(m × n)`.

Binary search performs approximately `log(10^9)`, around 30 iterations.

Therefore the total complexity is:

```
O(m × n × log(10^9))
```

which is effectively:

```
O(m × n × 30)
```

and easily fits within the constraints.

## Edge Cases

### Fire Reaches the Starting Cell Before Movement

One subtle case occurs when the player waits too long and the fire reaches `(0,0)` before movement begins.

A naive implementation might still allow BFS to start, which would incorrectly permit escape.

The implementation prevents this with:

```
if wait >= fire_time[0][0]:
    return False
```

This correctly models the player burning before making the first move.

### Simultaneous Arrival at the Safehouse

Most cells require the player to arrive strictly before the fire.

However, the destination is special. Arriving at exactly the same minute as the fire is still considered safe.

This creates an easy source of off-by-one bugs.

The implementation handles this explicitly:

```
if nt > fire_time[nr][nc]:
    continue
```

for the safehouse only.

### Fire Never Reaches Certain Cells

Walls may completely isolate the fire from parts of the grid.

In such cases, fire arrival times remain infinity.

Without careful handling, some solutions mistakenly overflow or incorrectly reject valid paths.

The implementation uses a very large constant for infinity and naturally allows movement into those cells forever.

This also enables correct detection of the special `10^9` return condition.
