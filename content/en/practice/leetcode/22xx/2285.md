## Solution

## Problem Understanding

The problem gives us `n` cities labeled from `0` to `n - 1` and a list of bidirectional roads connecting pairs of cities. We must assign every city a unique value from `1` to `n`, using each value exactly once.

Once values are assigned, the importance of a road becomes the sum of the values of its two endpoint cities. The final score is the sum of the importance values across all roads.

Our goal is to maximize this total sum.

In other words, we are free to permute the numbers `1` through `n` among the cities in whatever way produces the largest possible total road importance.

The important observation is that a city contributes its assigned value once for every road connected to it. Therefore, cities with more roads should receive larger values.

Suppose a city has degree `d`, meaning it is connected to `d` roads. If we assign value `v` to that city, then that city contributes `d * v` to the total result. This transforms the problem into assigning larger numbers to cities with larger degrees.

The constraints are important:

- `n` can be as large as `5 * 10^4`
- `roads.length` can also be as large as `5 * 10^4`

Because of these limits, factorial or exponential approaches are impossible. Even quadratic solutions may be risky. We need an efficient algorithm, ideally `O(n log n)` or better.

The problem also guarantees:

- No duplicate roads
- No self loops
- Roads are bidirectional

Important edge cases include cities with zero connections, graphs with very few roads, and graphs where multiple cities share the same degree. A correct solution must handle all of these cleanly.

## Approaches

### Brute Force Approach

A brute force solution would try every possible assignment of values `1` through `n` to the cities.

For each permutation:

1. Assign the values to cities.
2. Compute the total importance of all roads.
3. Track the maximum result.

This approach is correct because it explicitly checks every possible assignment and therefore cannot miss the optimal one.

However, the number of assignments is `n!`, which becomes astronomically large even for moderate `n`.

For example:

- `10! = 3,628,800`
- `15!` is already over one trillion

Since `n` can be `50,000`, brute force is completely infeasible.

### Optimal Greedy Approach

The key insight is that every city's assigned value contributes once for every incident road.

If a city has degree `d`, assigning value `v` contributes:

```
d * v
```

to the final total.

Therefore:

- Higher degree cities should receive larger values.
- Lower degree cities should receive smaller values.

This becomes a classic greedy assignment problem.

The algorithm is:

1. Count the degree of every city.
2. Sort cities by degree.
3. Assign values `1` through `n` in ascending order of degree.
4. Compute the final contribution.

This works because swapping a larger value onto a higher degree city can only improve or preserve the total score.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n! * m) | O(n) | Tries every possible assignment |
| Optimal | O(n log n + m) | O(n) | Assigns largest values to highest degree cities |

Here, `m = len(roads)`.

## Algorithm Walkthrough

### Step 1: Compute the degree of every city

Create an array `degree` of size `n`, initialized to `0`.

For every road `[u, v]`:

- Increment `degree[u]`
- Increment `degree[v]`

This tells us how many roads each city participates in.

### Step 2: Sort cities by degree

Create a list of city indices:

```
cities = [0, 1, 2, ..., n-1]
```

Sort this list according to each city's degree.

Cities with smaller degrees appear first, and cities with larger degrees appear later.

### Step 3: Assign values greedily

Assign values from `1` to `n` in sorted order.

That means:

- The city with the smallest degree gets `1`
- The next gets `2`
- ...
- The city with the largest degree gets `n`

Store these assigned values in an array.

### Step 4: Compute the final total

Traverse all roads again.

For every road `[u, v]`:

```
importance += assigned[u] + assigned[v]
```

Return the total.

### Why it works

The total contribution of a city equals:

```
degree(city) * assigned_value(city)
```

To maximize the sum of these products, the largest assigned values must pair with the largest degrees.

This follows the same principle as the rearrangement inequality, where matching larger values together maximizes the total sum of products.

Therefore, sorting by degree and assigning values greedily is optimal.

## Python Solution

```python
from typing import List

class Solution:
    def maximumImportance(self, n: int, roads: List[List[int]]) -> int:
        degree = [0] * n

        # Count degree of each city
        for u, v in roads:
            degree[u] += 1
            degree[v] += 1

        # Sort cities by degree
        cities = list(range(n))
        cities.sort(key=lambda city: degree[city])

        # Assign values from 1 to n
        assigned = [0] * n

        for value, city in enumerate(cities, start=1):
            assigned[city] = value

        # Compute total importance
        total_importance = 0

        for u, v in roads:
            total_importance += assigned[u] + assigned[v]

        return total_importance
```

The implementation begins by building the `degree` array. Each road contributes one connection to both endpoint cities because the graph is bidirectional.

Next, we sort the city indices according to their degree counts. We do not sort the degree array itself because we still need to know which degree belongs to which city.

After sorting, we assign values incrementally from `1` to `n`. Since the cities are ordered from smallest degree to largest degree, the highest degree cities naturally receive the largest values.

Finally, we iterate through every road again and add the assigned values of both endpoints to the answer.

The implementation uses only arrays and sorting, which keeps the solution efficient and straightforward.

## Go Solution

```go
package main

import "sort"

func maximumImportance(n int, roads [][]int) int64 {
	degree := make([]int, n)

	// Count degrees
	for _, road := range roads {
		u, v := road[0], road[1]
		degree[u]++
		degree[v]++
	}

	// Create city indices
	cities := make([]int, n)
	for i := 0; i < n; i++ {
		cities[i] = i
	}

	// Sort by degree
	sort.Slice(cities, func(i, j int) bool {
		return degree[cities[i]] < degree[cities[j]]
	})

	// Assign values
	assigned := make([]int, n)

	for i, city := range cities {
		assigned[city] = i + 1
	}

	// Compute total importance
	var total int64 = 0

	for _, road := range roads {
		u, v := road[0], road[1]
		total += int64(assigned[u] + assigned[v])
	}

	return total
}
```

The Go implementation follows the exact same logic as the Python version.

One important detail is the use of `int64` for the final answer. The total importance can become quite large because there may be up to `50,000` roads and city values can also reach `50,000`. Using `int64` prevents overflow.

Go uses `sort.Slice` to sort the city indices based on degree values.

| --- | --- | --- |

The degree counting step takes `O(m)` because we process each road once.

Sorting the `n` cities takes `O(n log n)`.

Finally, computing the total importance requires another `O(m)` traversal.

The extra memory usage comes from storing the degree array, sorted city list, and assigned values.

## Edge Cases

### Cities with zero connections

Some cities may not appear in any road. Their degree is zero, meaning their assigned value does not contribute to the final sum at all.

A buggy implementation might accidentally prioritize these cities incorrectly. Our solution naturally handles this because zero degree cities appear first in the sorted order and therefore receive the smallest values.

### Multiple cities with the same degree

Several cities may have identical degrees. In that situation, any ordering among them is equally optimal because swapping equal degree cities does not change the total contribution.

Our implementation handles this automatically through sorting. Even if the internal order differs, the final answer remains optimal.

### Extremely dense graphs

A graph may contain many roads, causing the final answer to become large.

For example, in a near complete graph, many high valued cities contribute repeatedly. In Go specifically, using regular `int` could risk overflow depending on platform size.

The Go implementation therefore uses `int64` for the accumulated result to ensure correctness for all valid inputs.
