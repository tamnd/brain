## Solution

## Problem Understanding

The problem gives us a database table named `Purchases`. Each row represents a purchase made by a user, along with the timestamp of the purchase and the amount paid.

The goal is to identify all users who are eligible for a discount. A user becomes eligible if they satisfy two conditions with at least one purchase:

1. The purchase timestamp falls inside the inclusive interval `[startDate, endDate]`.
2. The purchase amount is greater than or equal to `minAmount`.

An important detail is how the dates are interpreted. Both `startDate` and `endDate` are treated as the start of the day, meaning:

- `2022-03-08` becomes `2022-03-08 00:00:00`
- `2022-03-20` becomes `2022-03-20 00:00:00`

This means the interval includes timestamps from the exact beginning of `startDate` up to the exact beginning of `endDate`.

The output should contain only the `user_id` values of users who qualify. The result must be ordered by `user_id`.

The table guarantees that `(user_id, time_stamp)` is unique, so no two rows have the same user and timestamp combination.

Since this is a database problem, efficiency comes from filtering rows directly in SQL rather than processing everything manually.

Several edge cases are important:

- A purchase exactly at `startDate 00:00:00` should be included.
- A purchase exactly at `endDate 00:00:00` should also be included.
- A purchase one second after `endDate 00:00:00` should not be included.
- Users may have multiple purchases, but only one valid purchase is needed.
- Multiple qualifying purchases for the same user should still produce only one row in the result.
- Users with qualifying amounts outside the time range should not qualify.
- Users inside the time range but below `minAmount` should not qualify.

## Approaches

### Brute Force Approach

A brute force solution would scan every purchase row and manually check both conditions:

1. Whether the timestamp lies within the allowed range.
2. Whether the amount is at least `minAmount`.

For every valid purchase, we would store the corresponding `user_id` in a set to avoid duplicates. After processing all rows, we would sort the resulting user IDs.

This approach is correct because every purchase is individually validated against the rules. Any user with at least one qualifying purchase will eventually be added to the result set.

However, this approach is inefficient in an application-layer context because it requires loading all purchase records into memory and performing filtering manually. Databases are optimized for filtering and deduplication operations, so pushing the work into SQL is preferable.

### Optimal Approach

The key observation is that SQL databases already provide efficient mechanisms for:

- Filtering rows with `WHERE`
- Removing duplicates with `DISTINCT`
- Ordering results with `ORDER BY`

Instead of processing records outside the database, we can directly ask the database for all unique users whose purchases satisfy both conditions.

The query simply filters rows where:

- `time_stamp` is between `startDate` and `endDate`
- `amount >= minAmount`

Then we select distinct `user_id` values and sort them.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) | O(n) | Manual filtering and sorting outside the database |
| Optimal | O(n log n) | O(n) | Database performs filtering, deduplication, and ordering efficiently |

The exact complexity depends on indexing and database implementation details, but conceptually both approaches examine the rows once.

## Algorithm Walkthrough

1. Start by scanning the `Purchases` table.
2. For each row, check whether the purchase timestamp lies inside the interval `[startDate, endDate]`. Since both dates are interpreted as the start of the day, comparisons are performed directly against those datetime boundaries.
3. Check whether the purchase amount is greater than or equal to `minAmount`.
4. If both conditions are satisfied, keep the corresponding `user_id`.
5. Use `DISTINCT` to ensure each user appears only once, even if they have multiple qualifying purchases.
6. Sort the final result by `user_id` in ascending order.

### Why it works

The algorithm works because every purchase row is independently checked against the exact eligibility rules. If a user has at least one qualifying purchase, their ID is included in the result. Using `DISTINCT` guarantees uniqueness, and `ORDER BY` guarantees the required ordering.

## Python Solution

Although this is a SQL problem on LeetCode, the equivalent logic in Python can be expressed as follows.

```python
from typing import List, Dict
from datetime import datetime

class Solution:
    def eligibleUsers(
        self,
        purchases: List[Dict],
        startDate: str,
        endDate: str,
        minAmount: int
    ) -> List[int]:

        start_time = datetime.strptime(startDate, "%Y-%m-%d")
        end_time = datetime.strptime(endDate, "%Y-%m-%d")

        eligible_users = set()

        for purchase in purchases:
            timestamp = datetime.strptime(
                purchase["time_stamp"],
                "%Y-%m-%d %H:%M:%S"
            )

            amount = purchase["amount"]

            if start_time <= timestamp <= end_time and amount >= minAmount:
                eligible_users.add(purchase["user_id"])

        return sorted(eligible_users)
```

### Implementation Explanation

The implementation begins by converting the input date strings into Python `datetime` objects. This allows direct comparison between timestamps.

A set named `eligible_users` is used because sets automatically eliminate duplicates. If a user has multiple qualifying purchases, their ID will still appear only once.

The algorithm iterates through every purchase record. For each purchase:

- The timestamp string is parsed into a `datetime`.
- The amount is extracted.
- Both eligibility conditions are checked.

If the purchase qualifies, the user ID is inserted into the set.

Finally, the result is converted into a sorted list before returning.

## Go Solution

```go
package main

import (
	"sort"
	"time"
)

type Purchase struct {
	UserID   int
	TimeStamp string
	Amount   int
}

func eligibleUsers(
	purchases []Purchase,
	startDate string,
	endDate string,
	minAmount int,
) []int {

	startTime, _ := time.Parse("2006-01-02", startDate)
	endTime, _ := time.Parse("2006-01-02", endDate)

	eligibleSet := make(map[int]bool)

	for _, purchase := range purchases {

		timestamp, _ := time.Parse(
			"2006-01-02 15:04:05",
			purchase.TimeStamp,
		)

		if (timestamp.Equal(startTime) || timestamp.After(startTime)) &&
			(timestamp.Equal(endTime) || timestamp.Before(endTime)) &&
			purchase.Amount >= minAmount {

			eligibleSet[purchase.UserID] = true
		}
	}

	result := make([]int, 0, len(eligibleSet))

	for userID := range eligibleSet {
		result = append(result, userID)
	}

	sort.Ints(result)

	return result
}
```

### Go-Specific Notes

The Go version uses a `map[int]bool` to simulate a set because Go does not provide a built-in set type.

Time parsing uses Go's reference layout format:

```
"2006-01-02"
```

and

```
"2006-01-02 15:04:05"
```

Sorting is handled using `sort.Ints`.

Unlike Python, Go requires explicit slice allocation and iteration over the map to construct the final result.

## SQL Solution

```sql
SELECT DISTINCT user_id
FROM Purchases
WHERE time_stamp BETWEEN startDate AND endDate
    AND amount >= minAmount
ORDER BY user_id;
```

### SQL Explanation

The `WHERE` clause filters purchases that satisfy both eligibility conditions.

`BETWEEN` is inclusive in SQL, which matches the problem requirements.

`DISTINCT` removes duplicate users in case multiple purchases qualify.

Finally, `ORDER BY user_id` ensures ascending order.

## Edge Cases

### Purchases Exactly on the Date Boundaries

A common mistake is incorrectly treating the interval as exclusive instead of inclusive. The problem explicitly states that purchases at `startDate 00:00:00` and `endDate 00:00:00` should qualify.

The implementation handles this correctly by using inclusive comparisons:

```
start_time <= timestamp <= end_time
```

and by using SQL `BETWEEN`, which is also inclusive.

### Multiple Qualifying Purchases for the Same User

A user may make several qualifying purchases. A naive implementation might accidentally return duplicate user IDs.

This implementation avoids duplicates by using:

- `DISTINCT` in SQL
- A `set` in Python
- A `map[int]bool` in Go

As a result, each eligible user appears exactly once.

### Valid Amount but Invalid Time

Another subtle bug occurs when implementations check only the amount condition and forget the timestamp condition.

For example:

```
amount = 5000
time_stamp = 2022-04-01
```

Even though the amount is large enough, the purchase must still fall inside the allowed interval.

The implementation correctly requires both conditions simultaneously before adding the user to the result set.
