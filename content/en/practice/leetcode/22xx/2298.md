## Problem Understanding

The problem asks us to compute the number of tasks submitted during the weekend and the number submitted on working days, based on a `Tasks` table containing task IDs, assignee IDs, and submission dates. The input is a database table where each row represents a single task with a submission date. The output is a single-row table with two columns: `weekend_cnt` for tasks submitted on Saturday or Sunday, and `working_cnt` for tasks submitted from Monday to Friday.

The key constraints are that `task_id` is unique, and all dates are valid. The dataset could be large, so a solution needs to efficiently determine whether each date falls on a weekend or a weekday. Edge cases include tables with no tasks, all tasks on weekends, all tasks on weekdays, and leap-year dates.

## Approaches

A brute-force approach would involve reading every task, determining its day of the week, and incrementing two counters accordingly. While correct, this approach may be inefficient if done outside the database or without leveraging SQL date functions, especially for large datasets.

The optimal approach leverages SQL built-in date functions to directly classify each task as weekend or working day. Most relational databases provide functions like `DAYOFWEEK` or `EXTRACT(DOW FROM date)` which return an integer corresponding to the day of the week. Using these, we can sum conditional counts in a single query, which is both concise and efficient.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Iterate through each task in application code and classify the date. |
| Optimal | O(n) | O(1) | Use SQL aggregation with conditional counting, leveraging database functions. |

## Algorithm Walkthrough

1. Access the `Tasks` table from the database.
2. Use a built-in SQL function to extract the day of the week from `submit_date`. For example, `DAYOFWEEK(submit_date)` in MySQL returns 1 for Sunday through 7 for Saturday.
3. Use a `CASE` expression to determine if the day corresponds to a weekend (Saturday or Sunday) or a working day (Monday to Friday).
4. Use `SUM` over the `CASE` expressions to count the number of tasks for each category.
5. Return a single-row table with the counts labeled `weekend_cnt` and `working_cnt`.

**Why it works:** The SQL `CASE` expression guarantees that each task is counted exactly once in either the weekend or working day sum. Summing over the boolean conditions ensures accurate totals without missing or double-counting tasks.

## Python Solution

```python
import sqlite3

def tasks_count_in_weekend(db_connection: sqlite3.Connection) -> tuple[int, int]:
    cursor = db_connection.cursor()
    query = """
    SELECT
        SUM(CASE WHEN strftime('%w', submit_date) IN ('0', '6') THEN 1 ELSE 0 END) AS weekend_cnt,
        SUM(CASE WHEN strftime('%w', submit_date) NOT IN ('0', '6') THEN 1 ELSE 0 END) AS working_cnt
    FROM Tasks
    """
    cursor.execute(query)
    result = cursor.fetchone()
    return result
```

This Python implementation uses SQLite. The `strftime('%w', date)` function returns '0' for Sunday and '6' for Saturday. The `CASE` statements categorize each task and sum the counts. Fetching a single row gives the final `weekend_cnt` and `working_cnt`.

## Go Solution

```go
package main

import (
    "database/sql"
    _ "github.com/mattn/go-sqlite3"
    "fmt"
)

func TasksCountInWeekend(db *sql.DB) (int, int, error) {
    query := `
    SELECT
        SUM(CASE WHEN strftime('%w', submit_date) IN ('0', '6') THEN 1 ELSE 0 END) AS weekend_cnt,
        SUM(CASE WHEN strftime('%w', submit_date) NOT IN ('0', '6') THEN 1 ELSE 0 END) AS working_cnt
    FROM Tasks;
    `
    var weekendCnt, workingCnt int
    err := db.QueryRow(query).Scan(&weekendCnt, &workingCnt)
    if err != nil {
        return 0, 0, err
    }
    return weekendCnt, workingCnt, nil
}
```

The Go version mirrors the Python logic. `QueryRow` is used to fetch a single row result, and `Scan` assigns the counts to integer variables. SQLite functions behave the same in Go.

## Edge Cases

One edge case is an empty `Tasks` table, which should return 0 for both counts. Another is when all tasks occur on the same day, ensuring the code correctly counts only that category. A third case involves leap year dates to ensure day-of-week computation remains accurate, which is handled naturally by SQL date functions.

This approach reliably handles any valid date without requiring manual calculation of the day of the week.
