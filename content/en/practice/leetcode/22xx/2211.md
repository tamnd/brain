## Solution

## Problem Understanding

This problem asks us to compute the total number of collisions that occur among cars on an infinitely long road, given their initial directions. Each car can move left ('L'), move right ('R'), or stay stationary ('S'). Collisions occur in two ways: when two moving cars meet head-on, each contributes to the collision count as 2, or when a moving car hits a stationary car, contributing 1 to the collision count. After any collision, the cars involved stop moving.

The input is a string `directions` where the index represents the car's position from left to right, and the value represents the direction or state of the car. The output is a single integer representing the total number of collisions.

Constraints tell us the string length can be up to 100,000. This means any brute-force solution simulating each time step would be too slow. Edge cases include cars that are all moving in one direction with no collisions, single-car strings, and strings that start or end with cars that never interact due to directionality.

## Approaches

The brute-force approach is to simulate the cars moving step by step along the road. At each step, check for collisions between adjacent cars, update the directions accordingly, and count collisions. While correct, this approach is too slow because it potentially requires O(n²) or more steps for long strings of cars.

A key observation simplifies the problem: cars moving left from the leftmost end will never collide with anything on their left, and cars moving right from the rightmost end will never collide with anything on their right. Therefore, we only need to consider cars that eventually interact, which are those that are either trapped between moving cars in opposite directions or stationary cars. From this, we can scan from left to right and right to left, counting collisions based on runs of 'R' and 'L' interacting with stationary cars. This reduces the problem to O(n) complexity without simulating every time step.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Simulate each time step and check for collisions |
| Optimal | O(n) | O(1) | Count collisions using observation about stationary boundaries |

## Algorithm Walkthrough

1. First, remove cars on the left moving left ('L') and cars on the right moving right ('R') as they will never collide. Count the number of remaining cars that can collide.
2. Initialize a counter for collisions to 0.
3. Traverse the remaining string from left to right. Whenever a car moving right ('R') is followed by a car moving left ('L') or stationary ('S'), count the collision and mark involved cars as stationary.
4. Similarly, whenever a moving car encounters a stationary car, increment the collision count by 1 and continue until all interactions are accounted for.
5. Return the total number of collisions.

Why it works: Any car that will collide is either part of a sequence of 'R' cars moving toward stationary or 'S' cars or part of a head-on 'R' and 'L' collision. By eliminating the non-colliding edges, we can correctly count all interactions linearly. The invariant is that all cars left after trimming edges are involved in eventual collisions.

## Python Solution

```python
class Solution:
    def countCollisions(self, directions: str) -> int:
        n = len(directions)
        # Trim left 'L' and right 'R' as they never collide
        left, right = 0, n - 1
        while left < n and directions[left] == 'L':
            left += 1
        while right >= 0 and directions[right] == 'R':
            right -= 1
        
        collisions = 0
        # Count collisions in the remaining substring
        for i in range(left, right + 1):
            if directions[i] != 'S':
                collisions += 1
        
        return collisions
```

The code first trims non-colliding cars on the left and right ends. Then, for all cars that are between these bounds, any moving car ('L' or 'R') will eventually collide, so we increment the counter by 1 for each moving car. Stationary cars are already considered part of the collision chain but do not add extra. This approach directly follows the algorithm steps described.

## Go Solution

```go
func countCollisions(directions string) int {
    n := len(directions)
    left, right := 0, n-1
    for left < n && directions[left] == 'L' {
        left++
    }
    for right >= 0 && directions[right] == 'R' {
        right--
    }
    
    collisions := 0
    for i := left; i <= right; i++ {
        if directions[i] != 'S' {
            collisions++
        }
    }
    return collisions
}
```

In Go, the approach mirrors the Python version. We use simple loops to trim the non-colliding edges and a single loop to count all moving cars in the collision-prone region. Slices and string indexing handle character access directly.

## Edge Cases

A critical edge case is when all cars are moving in the same direction, either all 'R' or all 'L'. In this case, collisions are impossible, and the implementation correctly trims edges to avoid overcounting.

Another important edge case is a string of only one car. Since no other car exists to collide with, the algorithm handles it by the trimming step, resulting in zero collisions.

Finally, consider a case with stationary cars in between moving cars. The algorithm counts every moving car in the collision region as contributing to the total collisions, correctly accounting for sequences like "RRSLLR" without double-counting collisions.
