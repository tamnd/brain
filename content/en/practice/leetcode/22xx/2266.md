## Solution

## Problem Understanding

The problem gives us a string called `pressedKeys`, which represents the sequence of numeric button presses Bob received from Alice's old mobile phone keypad. Each digit corresponds to a group of letters, and typing a specific letter requires pressing the digit multiple times.

For example:

- Digit `2` maps to `"abc"`

- `'a'` requires one press of `2`
- `'b'` requires two presses of `2`
- `'c'` requires three presses of `2`
- Digit `7` maps to `"pqrs"`

- `'p'` requires one press
- `'q'` requires two presses
- `'r'` requires three presses
- `'s'` requires four presses

The important detail is that Bob only receives the raw sequence of key presses, not the intended grouping. We must determine how many different valid text messages Alice could have intended.

Consider the input:

```
"22233"
```

This sequence could be split in multiple ways:

- `"2" + "2" + "2" + "33"` → `"aaae"`
- `"22" + "2" + "33"` → `"bae"`
- `"222" + "33"` → `"ce"`

and several more.

The task is to count every valid way to partition the string into legal button press groups.

The constraints are very important:

- `pressedKeys.length` can be as large as `10^5`
- A brute force recursive exploration of all partitions would be far too slow
- We must return the answer modulo `10^9 + 7`

Another critical observation is that:

- Digits `2`, `3`, `4`, `5`, `6`, and `8` each support at most 3 consecutive presses
- Digits `7` and `9` support at most 4 consecutive presses

This means:

- `"222"` is valid
- `"2222"` is invalid because digit `2` only has 3 letters
- `"7777"` is valid because digit `7` has 4 letters

Important edge cases include long runs of identical digits, especially runs of `7` and `9`, because they allow more partition choices. Another edge case is a single character string, which always has exactly one valid interpretation.

## Approaches

### Brute Force Approach

A straightforward solution is to recursively try every possible partition of the string.

At each position, we attempt to take:

- 1 digit
- 2 equal digits
- 3 equal digits
- Possibly 4 equal digits if the digit is `7` or `9`

For every valid choice, we recursively solve the remaining suffix.

This works because every valid message corresponds to exactly one valid partitioning of the pressed key sequence.

However, this approach is extremely inefficient because the number of possible partitions grows exponentially. For long runs like:

```
"222222222222..."
```

the recursive tree becomes enormous.

Even memoization alone is not enough unless we structure the recurrence carefully as dynamic programming.

### Key Insight

The crucial observation is that the answer depends only on the current index in the string.

Suppose we define:

- `dp[i]` = number of valid ways to interpret the substring starting at index `i`

Then:

- We try extending a group from index `i`
- The group must:

- contain only identical digits
- not exceed the allowed maximum length
- For every valid group ending at `j`, we add:

- `dp[j + 1]`

This creates a classic dynamic programming recurrence.

Since each index only checks up to 4 future positions, the algorithm becomes linear.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(4^n) | O(n) | Explores every partition recursively |
| Optimal Dynamic Programming | O(n) | O(n) | Computes valid partitions efficiently using DP |

## Algorithm Walkthrough

### Step 1: Define the DP State

We create an array:

```
dp[i]
```

where `dp[i]` represents the number of valid interpretations for the substring:

```
pressedKeys[i:]
```

The final answer will be:

```
dp[0]
```

because that represents the whole string.

### Step 2: Initialize the Base Case

An empty suffix has exactly one valid interpretation:

```
dp[n] = 1
```

This acts as the termination condition for transitions.

### Step 3: Process the String Backward

We iterate from the end of the string toward the beginning.

Why backward?

Because:

```
dp[i]
```

depends on future states like:

```
dp[i + 1], dp[i + 2], ...
```

which must already be computed.

### Step 4: Determine Maximum Allowed Group Size

For each digit:

- digits `2,3,4,5,6,8` allow up to 3 presses
- digits `7,9` allow up to 4 presses

So:

```
limit = 4 if digit in "79" else 3
```

### Step 5: Try All Valid Group Lengths

Starting at index `i`, we try extending the group:

- length 1
- length 2
- length 3
- possibly length 4

We stop when:

- we exceed the limit
- digits stop matching
- we reach the end of the string

For every valid group:

```
dp[i] += dp[next_index]
```

because every valid continuation contributes additional interpretations.

### Step 6: Apply Modulo

Since the count can become very large, we take:

```
MOD = 10**9 + 7
```

after every addition.

### Step 7: Return the Result

The final answer is:

```
dp[0]
```

## Why it works

The algorithm works because every valid interpretation corresponds to a unique partition of the string into valid groups of repeated digits.

At each position, the dynamic programming transition enumerates every legal group that could begin there. Since the remaining suffix is solved optimally by `dp[next_index]`, summing all such possibilities gives the total number of interpretations for the current position.

The recurrence considers all valid partitions exactly once, so no interpretations are missed or double counted.

## Python Solution

```python
class Solution:
    def countTexts(self, pressedKeys: str) -> int:
        MOD = 10**9 + 7
        n = len(pressedKeys)

        dp = [0] * (n + 1)
        dp[n] = 1

        for i in range(n - 1, -1, -1):
            digit = pressedKeys[i]
            limit = 4 if digit in "79" else 3

            for length in range(1, limit + 1):
                next_index = i + length

                if next_index > n:
                    break

                if pressedKeys[i:next_index] != digit * length:
                    break

                dp[i] = (dp[i] + dp[next_index]) % MOD

        return dp[0]
```

The implementation begins by creating a DP array of size `n + 1`. The extra position represents the empty suffix.

The base case:

```python
dp[n] = 1
```

means there is exactly one way to interpret an empty string.

The main loop processes indices from right to left. For each position, we determine whether the current digit belongs to the 3-letter group or the 4-letter group.

The inner loop tries every possible group length. A group is valid only if:

- it stays within bounds
- all digits in the group are identical

Whenever a valid group is found, we add the number of interpretations for the remaining suffix.

Finally, we return `dp[0]`, which represents the number of interpretations for the full string.

## Go Solution

```go
func countTexts(pressedKeys string) int {
    const MOD int = 1000000007

    n := len(pressedKeys)
    dp := make([]int, n+1)

    dp[n] = 1

    for i := n - 1; i >= 0; i-- {
        digit := pressedKeys[i]

        limit := 3
        if digit == '7' || digit == '9' {
            limit = 4
        }

        for length := 1; length <= limit; length++ {
            nextIndex := i + length

            if nextIndex > n {
                break
            }

            valid := true

            for j := i; j < nextIndex; j++ {
                if pressedKeys[j] != digit {
                    valid = false
                    break
                }
            }

            if !valid {
                break
            }

            dp[i] = (dp[i] + dp[nextIndex]) % MOD
        }
    }

    return dp[0]
}
```

The Go implementation follows the same logic as the Python version.

One notable difference is that Go strings are indexed as bytes, so comparisons are done using byte values such as:

```go
digit == '7'
```

instead of string membership checks.

The DP array uses `int`, which is safe because modulo operations prevent overflow from growing unbounded.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each index checks at most 4 transitions |
| Space | O(n) | DP array stores one value per index |

Although there is a nested loop, the inner loop runs at most 4 times because keypad mappings have a fixed maximum size. Therefore, the total work grows linearly with the length of the string.

The space complexity is linear because we store one DP value for every position in the input string.

## Edge Cases

### Long Runs of the Same Digit

A very long sequence such as:

```
"222222222222..."
```

creates a huge number of valid partitions. A recursive brute force solution would explode exponentially and likely time out.

The DP solution handles this efficiently because each position is computed once, and each state checks at most four transitions.

### Digits With Four Letters

Digits `7` and `9` are special because they allow four consecutive presses instead of three.

This is a common source of bugs because treating all digits uniformly would incorrectly reject valid cases such as:

```
"7777"
```

The implementation explicitly checks:

```
limit = 4 if digit in "79" else 3
```

so these digits are handled correctly.

### Transitions Between Different Digits

A sequence like:

```
"22233"
```

contains multiple runs of different digits.

A buggy implementation might accidentally allow mixed groups such as:

```
"23"
```

which is invalid because groups must contain only repeated presses of the same key.

The implementation prevents this by verifying:

```
pressedKeys[i:next_index] == digit * length
```

which guarantees every group consists entirely of the same digit.
