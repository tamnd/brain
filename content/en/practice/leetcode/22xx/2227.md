## Solution

## Problem Understanding

The problem asks us to design a data structure that supports two operations:

1. Encrypting a string using a fixed character-to-string mapping.
2. Decrypting an encrypted string and counting how many valid original words from a given dictionary could produce it.

We are given:

- `keys`, an array of unique characters.
- `values`, an array of strings of length 2.
- `dictionary`, a list of valid original words.

Each character in `keys[i]` maps to the encrypted string `values[i]`.

For encryption, we process the input string character by character. Every character must exist in `keys`. If even one character does not exist, encryption fails and returns an empty string. Otherwise, we concatenate the corresponding two-character strings together.

The tricky part is decryption.

Different characters may map to the same encrypted value. For example:

```
'a' -> "ei"
'c' -> "ei"
```

This means the encrypted substring `"ei"` is ambiguous during decryption because it could represent either `'a'` or `'c'`.

When decrypting a string, we split it into chunks of length 2 and determine all possible original strings that could generate it. However, we only care about strings that exist in `dictionary`.

The key observation is that the problem does not ask us to generate every possible decrypted string. It only asks for the count of valid dictionary words.

The constraints are important:

- `dictionary.length <= 100`
- `dictionary[i].length <= 100`
- At most 200 total calls to `encrypt` and `decrypt`

These are relatively small constraints. Even though decryption can theoretically branch exponentially because of ambiguous mappings, the small dictionary size gives us a much better strategy.

An important guarantee is that all characters in `word1` during encryption appear in `keys`. This means valid encryption calls will always succeed according to the constraints, though implementing the failure case is still correct and robust.

Some important edge cases include:

- Multiple characters mapping to the same encrypted value.
- Dictionary words producing identical encrypted strings.
- Encrypting a word with an unmapped character.
- Decrypting a string that matches no dictionary word.
- Very long strings where brute force decryption would explode combinatorially.

## Approaches

### Brute Force Approach

The brute force idea is to explicitly generate all possible decrypted strings.

To decrypt `word2`, we split it into chunks of size 2. For each chunk, we determine every possible character that could map to it. Then we recursively build all possible candidate strings.

For example:

```
"eizf"
```

might split into:

```
"ei" -> ['a', 'c']
"zf" -> ['b']
```

This produces:

```
"ab"
"cb"
```

After generating every possible string, we check whether each exists in the dictionary.

This works correctly because it explores every valid decoding path. However, it becomes extremely expensive when many encrypted pairs are ambiguous.

If every encrypted pair maps to multiple characters, the number of possible strings grows exponentially.

For a string of length `n`, the complexity can become:

```
O(k^(n/2))
```

where `k` is the average number of characters per encrypted value.

This is too slow conceptually and unnecessary given the small dictionary size.

### Optimal Approach

The key insight is that we do not actually need to perform decryption.

Instead, we can precompute the encrypted version of every dictionary word during initialization.

Then:

- `encrypt(word)` works normally.
- `decrypt(word2)` simply returns how many dictionary words encrypt to `word2`.

This transforms the problem completely.

Instead of exploring exponentially many decryptions, we only perform direct hash map lookups.

The idea works because the question only asks:

> How many dictionary words could decrypt to this encrypted string?

A dictionary word could decrypt to `word2` if and only if encrypting that dictionary word produces `word2`.

So we preprocess all dictionary words:

```
encrypted_word -> frequency
```

Then decryption becomes:

```
return frequency_map[word2]
```

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(k^(n/2)) | O(k^(n/2)) | Generates all possible decryptions |
| Optimal | O(D × L) preprocessing, O(L) encrypt, O(1) decrypt | O(D × L) | Precomputes encrypted dictionary words |

Where:

- `D` = number of dictionary words
- `L` = average dictionary word length
- `k` = average branching factor during decryption

## Algorithm Walkthrough

### Step 1: Build the encryption map

We first create a hash map:

```
character -> encrypted string
```

Example:

```
'a' -> "ei"
'b' -> "zf"
'c' -> "ei"
'd' -> "am"
```

This allows constant time lookup during encryption.

### Step 2: Precompute encrypted dictionary words

We iterate through every word in `dictionary`.

For each word:

1. Encrypt it using the mapping.
2. Store the resulting encrypted string in a frequency map.

Example:

```
"abcd" -> "eizfeiam"
"abad" -> "eizfeiam"
```

The frequency map becomes:

```
"eizfeiam" -> 2
```

We use a frequency map instead of a set because multiple dictionary words may encrypt to the same result.

### Step 3: Encrypt operation

To encrypt a word:

1. Initialize an empty result list.
2. For each character:

- If the character does not exist in the map, return `""`.
- Otherwise append its encrypted value.
3. Join the pieces together.

Using a list and join is efficient because repeated string concatenation is expensive in many languages.

### Step 4: Decrypt operation

To decrypt a word:

1. Look up the encrypted string in the precomputed frequency map.
2. Return the stored count.
3. If it does not exist, return `0`.

This works because every dictionary word that encrypts to `word2` represents one valid decryption.

### Why it works

The algorithm relies on the equivalence:

```
A dictionary word can decrypt to word2
if and only if
encrypt(dictionary_word) == word2
```

Instead of reversing the encryption process, we precompute the forward encryption for every valid dictionary word.

Since decryption only asks for the number of valid dictionary matches, counting encrypted dictionary frequencies gives the exact answer.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Encrypter:

    def __init__(self, keys: List[str], values: List[str], dictionary: List[str]):
        self.encrypt_map = {}

        for key, value in zip(keys, values):
            self.encrypt_map[key] = value

        self.encrypted_count = defaultdict(int)

        for word in dictionary:
            encrypted_word = self.encrypt(word)

            if encrypted_word:
                self.encrypted_count[encrypted_word] += 1

    def encrypt(self, word1: str) -> str:
        encrypted_parts = []

        for char in word1:
            if char not in self.encrypt_map:
                return ""

            encrypted_parts.append(self.encrypt_map[char])

        return "".join(encrypted_parts)

    def decrypt(self, word2: str) -> int:
        return self.encrypted_count[word2]
```

The implementation begins by constructing `encrypt_map`, which provides constant time access from characters to encrypted strings.

During initialization, every dictionary word is encrypted once and stored in `encrypted_count`. This preprocessing step is the core optimization of the solution.

The `encrypt` method iterates through the input word and converts each character into its corresponding encrypted string. If any character does not exist in the map, encryption fails immediately and returns an empty string.

The `decrypt` method is extremely efficient because it only performs a hash map lookup. Since all valid dictionary encryptions were precomputed earlier, the answer is available in constant time.

## Go Solution

```go
package main

type Encrypter struct {
	encryptMap    map[byte]string
	encryptedFreq map[string]int
}

func Constructor(keys []byte, values []string, dictionary []string) Encrypter {
	encryptMap := make(map[byte]string)

	for i := 0; i < len(keys); i++ {
		encryptMap[keys[i]] = values[i]
	}

	encryptedFreq := make(map[string]int)

	encrypter := Encrypter{
		encryptMap:    encryptMap,
		encryptedFreq: encryptedFreq,
	}

	for _, word := range dictionary {
		encrypted := encrypter.Encrypt(word)

		if encrypted != "" {
			encrypter.encryptedFreq[encrypted]++
		}
	}

	return encrypter
}

func (this *Encrypter) Encrypt(word1 string) string {
	result := make([]byte, 0)

	for i := 0; i < len(word1); i++ {
		value, exists := this.encryptMap[word1[i]]

		if !exists {
			return ""
		}

		result = append(result, value...)
	}

	return string(result)
}

func (this *Encrypter) Decrypt(word2 string) int {
	return this.encryptedFreq[word2]
}
```

The Go implementation follows the same strategy as the Python version.

One important difference is that Go strings are immutable, so the solution builds the encrypted result using a byte slice for efficiency.

Go maps automatically return zero values for missing keys, so:

```go
return this.encryptedFreq[word2]
```

naturally returns `0` if the encrypted string does not exist.

Since all encrypted values have fixed length 2, there are no integer overflow concerns or complex memory management issues.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(D × L) preprocessing, O(L) encrypt, O(1) decrypt | Dictionary preprocessing dominates initialization |
| Space | O(D × L) | Stores encrypted dictionary strings |

Where:

- `D` is the number of dictionary words
- `L` is the average word length

The preprocessing step encrypts every dictionary word exactly once. Each encryption takes linear time in the word length.

The `encrypt` operation processes each character once, so it runs in linear time.

The `decrypt` operation is a single hash map lookup, which is constant time on average.

The space complexity comes from storing encrypted dictionary strings in the frequency map.

## Edge Cases

### Multiple characters mapping to the same encrypted value

This is the most important edge case in the problem.

For example:

```
'a' -> "ei"
'c' -> "ei"
```

A naive decryption implementation might incorrectly assume one-to-one mappings and miss valid possibilities.

The solution handles this naturally because it does not attempt explicit decryption. Instead, it counts how many dictionary words produce the encrypted string. If multiple words map to the same encrypted result, they are all counted correctly.

### Dictionary words producing identical encrypted strings

Different dictionary words can encrypt to exactly the same output.

Example:

```
"abcd" -> "eizfeiam"
"abad" -> "eizfeiam"
```

Using a set would lose duplicate counts and produce incorrect answers.

The implementation uses a frequency map:

```
encrypted_count[encrypted_word] += 1
```

This guarantees every valid dictionary word contributes to the final answer.

### Encryption with unmapped characters

Although the constraints guarantee valid input during official calls, a robust implementation should still handle invalid characters.

Example:

```
keys = ['a']
encrypt("ab")
```

Since `'b'` does not exist in the mapping, encryption should fail.

The implementation checks:

```
if char not in self.encrypt_map:
    return ""
```

This prevents incorrect partial encryptions and follows the problem specification exactly.
