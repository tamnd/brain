## Solution

## Problem Understanding

The problem gives us an array of positive integers, and in one operation we may pick any number and replace it with exactly half of its current value. The important detail is that the halved value remains in the array and may be selected again in future operations.

We want to determine the minimum number of operations required so that the total sum of the array decreases by at least half of its original value.

Suppose the original array sum is `S`. Our goal is to reduce the total by at least `S / 2`. Each operation contributes some reduction amount. If we halve a number `x`, then the sum decreases by `x / 2`, because the value changes from `x` to `x / 2`.

The problem is therefore equivalent to repeatedly choosing numbers that produce the largest possible reduction until the accumulated reduction reaches at least half of the original total sum.

The constraints are important:

- The array can contain up to `10^5` elements.
- Each value can be as large as `10^7`.

This immediately rules out any solution that repeatedly scans the entire array for the maximum value in every operation, because the number of operations can also become fairly large.

There are several key observations:

- Every operation on a value `x` reduces the total sum by exactly `x / 2`.
- After halving, the resulting number may still be large enough to be selected again.
- To minimize the number of operations, we should always maximize the reduction gained in the current step.
- Since the largest current number always provides the largest reduction, we repeatedly need fast access to the current maximum element.

Important edge cases include:

- Arrays with only one element.
- Arrays where all numbers are equal.
- Arrays where one number dominates the total sum.
- Cases requiring repeated halvings of the same number.
- Very large arrays, where inefficient repeated scans would time out.

The problem guarantees all numbers are positive integers, so we never deal with zeros or negative values.

## Approaches

### Brute Force Approach

A straightforward approach is to repeatedly perform the following process:

1. Find the current largest value in the array.
2. Halve it.
3. Add the reduction amount to a running total.
4. Repeat until the reduction reaches at least half of the original sum.

This approach is correct because choosing the current maximum always gives the greatest immediate reduction.

However, the inefficiency comes from repeatedly searching for the largest element. If we scan the array every time, each operation costs `O(n)` time.

In the worst case, we may perform many operations, leading to a total complexity close to `O(n * k)`, where `k` is the number of operations performed.

With `n = 10^5`, this becomes too slow.

### Optimal Approach, Greedy + Max Heap

The key insight is that every operation should target the current largest number.

Why?

If we halve a number `x`, the reduction is `x / 2`. Therefore, the larger the number, the larger the reduction obtained from the operation.

This creates a greedy strategy:

- Always pick the largest available number.
- Halve it.
- Put the new value back because it may still remain large.

To efficiently retrieve the largest number repeatedly, we use a max heap.

A heap allows:

- Extracting the maximum value in `O(log n)`
- Reinserting the halved value in `O(log n)`

Since every operation only performs heap removal and insertion, the overall algorithm becomes efficient enough for the constraints.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * k) | O(1) | Repeatedly scans the array to find the largest value |
| Optimal | O((n + k) log n) | O(n) | Uses a max heap to efficiently retrieve the largest value |

Here, `k` is the number of operations required to reduce the sum by half.

## Algorithm Walkthrough

1. Compute the total sum of the array.

We first calculate the original sum because our target reduction is exactly half of this value.
2. Determine the target reduction.

If the original sum is `S`, then we must reduce the array sum by at least `S / 2`.
3. Build a max heap containing all numbers.

Python's `heapq` is a min heap, so we store negative values to simulate max heap behavior.
4. Initialize a variable to track how much total reduction we have achieved.

Every operation contributes some reduction amount, and we continue until this accumulated reduction reaches the target.
5. Repeatedly extract the current maximum number from the heap.

This is the greedy choice. The largest number always gives the greatest reduction for the current operation.
6. Halve the extracted value.

If the current value is `x`, then:

- The new value becomes `x / 2`
- The reduction gained is also `x / 2`
7. Add the reduction amount to the accumulated reduction total.

This tracks progress toward our goal.
8. Insert the halved value back into the heap.

The reduced number may still be among the largest values later.
9. Increment the operation counter.
10. Continue until the accumulated reduction is at least half of the original sum.
11. Return the operation count.

### Why it works

The greedy strategy is optimal because each operation's benefit is exactly proportional to the chosen number's current value. Choosing any number smaller than the maximum would produce a smaller reduction in the current operation and could never improve the total number of operations needed. The max heap ensures we always make the best possible local decision, which also leads to the globally minimal number of operations.

## Python Solution

```python
from typing import List
import heapq

class Solution:
    def halveArray(self, nums: List[int]) -> int:
        total_sum = sum(nums)
        target_reduction = total_sum / 2

        max_heap = [-num for num in nums]
        heapq.heapify(max_heap)

        current_reduction = 0.0
        operations = 0

        while current_reduction < target_reduction:
            largest = -heapq.heappop(max_heap)

            halved = largest / 2
            current_reduction += halved

            heapq.heappush(max_heap, -halved)

            operations += 1

        return operations
```

The implementation begins by computing the total array sum and the target reduction amount.

Next, we construct a max heap. Since Python's `heapq` only supports min heaps, we store all numbers as negative values. This allows the smallest negative number to correspond to the largest original value.

The variable `current_reduction` tracks how much total sum reduction has been achieved so far. The variable `operations` counts how many halvings have been performed.

Inside the loop, we repeatedly remove the largest value from the heap, halve it, and add the reduction amount to the running total. The halved value is then pushed back into the heap because it can still participate in future operations.

The loop stops once the accumulated reduction reaches at least half of the original sum.

## Go Solution

```go
package main

import (
	"container/heap"
)

type MaxHeap []float64

func (h MaxHeap) Len() int {
	return len(h)
}

func (h MaxHeap) Less(i, j int) bool {
	return h[i] > h[j]
}

func (h MaxHeap) Swap(i, j int) {
	h[i], h[j] = h[j], h[i]
}

func (h *MaxHeap) Push(x interface{}) {
	*h = append(*h, x.(float64))
}

func (h *MaxHeap) Pop() interface{} {
	old := *h
	n := len(old)
	value := old[n-1]
	*h = old[:n-1]
	return value
}

func halveArray(nums []int) int {
	totalSum := 0.0

	h := &MaxHeap{}

	for _, num := range nums {
		totalSum += float64(num)
		*h = append(*h, float64(num))
	}

	heap.Init(h)

	targetReduction := totalSum / 2
	currentReduction := 0.0
	operations := 0

	for currentReduction < targetReduction {
		largest := heap.Pop(h).(float64)

		halved := largest / 2
		currentReduction += halved

		heap.Push(h, halved)

		operations++
	}

	return operations
}
```

The Go solution uses the `container/heap` package, which requires implementing the heap interface manually.

Unlike Python, Go does not provide a built in max heap, so we define a custom `MaxHeap` type and reverse the comparison inside the `Less` function.

We use `float64` because repeated halving produces fractional values.

The rest of the algorithm matches the Python solution exactly.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O((n + k) log n) | Heap construction costs O(n), each operation performs heap pop and push |
| Space | O(n) | The heap stores all elements |

The heap initially stores all `n` numbers. Each operation requires one extraction and one insertion, both costing `O(log n)`.

If `k` operations are needed, the total complexity becomes `O((n + k) log n)`.

In practice, `k` remains manageable because each operation substantially reduces the largest value.

## Edge Cases

One important edge case is an array containing only one element. For example, `[10]`. Halving the number once reduces the total sum from `10` to `5`, which already achieves a reduction of half the original sum. A buggy implementation might incorrectly continue processing due to floating point comparisons. This implementation safely stops once the accumulated reduction reaches the target.

Another important case is when one value dominates the entire array, such as `[100,1,1,1]`. The optimal solution is to halve the largest value once, immediately reducing the total by `50`, which already exceeds half the original sum. A non greedy implementation could waste operations reducing smaller numbers first. The max heap guarantees the largest value is always selected.

A third important edge case is when repeated halvings of the same value are necessary. For example, in `[20,3,3]`, the value `20` remains the largest even after being halved once. Some incorrect solutions remove the value permanently after one operation. This implementation correctly reinserts the halved value into the heap, allowing it to be selected again later.

Another subtle edge case involves floating point values after repeated halvings. Since numbers become fractional, integer arithmetic would produce incorrect results. Both implementations use floating point types, ensuring accurate tracking of reductions throughout the process.
