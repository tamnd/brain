## Solution

## Problem Understanding

This problem asks us to design a simplified text editor that supports four operations around a movable cursor:

1. Insert text at the cursor position.
2. Delete characters immediately to the left of the cursor.
3. Move the cursor left.
4. Move the cursor right.

The cursor behaves exactly like a cursor in a real text editor. It always exists between characters, and its position is constrained to remain within the valid range of the current text.

For example, if the text is:

```
hello|
```

the cursor is at the end of the text. If we move it left twice:

```
hel|lo
```

then inserting `"X"` produces:

```
helX|lo
```

The problem also requires that after every cursor movement operation, we return the last up to 10 characters to the left of the cursor.

The constraints are important:

- `text.length <= 40`
- `k <= 40`
- At most `2 * 10^4` total operations

The relatively small operation size suggests that moving characters one by one is acceptable. However, rebuilding entire strings repeatedly would still be inefficient over many operations.

The follow-up asks for `O(k)` per operation, which strongly hints that we should avoid shifting entire strings whenever the cursor moves or text is inserted.

Several edge cases are important:

- Deleting more characters than exist to the left of the cursor.
- Moving the cursor beyond the beginning of the text.
- Moving the cursor beyond the end of the text.
- Returning fewer than 10 characters when the left side is short.
- Handling operations on an initially empty editor.

A naive implementation using a single string and slicing can work functionally, but repeated insertions and deletions in the middle of a string become expensive because strings are immutable in many languages.

## Approaches

### Brute Force Approach

The most straightforward implementation is to maintain:

- The entire text as a string.
- An integer cursor index.

For insertion:

- Split the string into left and right parts around the cursor.
- Insert the new text in between.
- Rebuild the string.

For deletion:

- Remove up to `k` characters before the cursor.
- Rebuild the string again.

For cursor movement:

- Adjust the cursor index carefully within bounds.

This approach is easy to understand and implement. However, string rebuilding is expensive because every insertion or deletion may require copying almost the entire string.

If the text length becomes large, each operation can cost `O(n)` where `n` is the current text length.

### Optimal Approach

The key insight is that the cursor naturally divides the text into two parts:

- Characters to the left of the cursor.
- Characters to the right of the cursor.

Instead of storing one string, we store:

- A stack for the left side.
- A stack for the right side.

For example:

```
hello|world
```

becomes:

```
left  = ['h','e','l','l','o']
right = ['d','l','r','o','w']
```

The right stack is reversed so that the character nearest the cursor is always on top.

This representation makes all operations efficient:

- Adding text pushes onto the left stack.
- Deleting text pops from the left stack.
- Moving left transfers characters from left to right.
- Moving right transfers characters from right to left.

Each operation only moves or modifies at most `k` characters, giving `O(k)` complexity per call.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) per operation | O(n) | Rebuilds strings repeatedly |
| Optimal | O(k) per operation | O(n) | Uses two stacks around cursor |

## Algorithm Walkthrough

### Data Structure Design

We maintain two stacks:

- `left`, characters to the left of the cursor.
- `right`, characters to the right of the cursor.

The cursor conceptually exists between these two stacks.

For example:

```
abc|xyz
```

is stored as:

```
left  = ['a', 'b', 'c']
right = ['z', 'y', 'x']
```

The top of each stack represents the character closest to the cursor.

### Step-by-Step Algorithm

1. Initialize two empty stacks.

The editor starts with empty text and the cursor at position 0.
2. Handle `addText(text)`.

Iterate through every character in `text` and push it onto `left`.

This works because inserted text always appears immediately to the left of the cursor, and the cursor moves to the end of the inserted text.
3. Handle `deleteText(k)`.

Repeatedly pop characters from `left` until:

- Either `k` characters are removed,
- Or the left stack becomes empty.

Return the number of characters actually deleted.
4. Handle `cursorLeft(k)`.

Moving the cursor left means characters cross from the left side to the right side.

Repeat up to `k` times:

- Pop from `left`
- Push onto `right`

Stop early if `left` becomes empty.
5. Handle `cursorRight(k)`.

Moving the cursor right means characters cross from the right side back to the left side.

Repeat up to `k` times:

- Pop from `right`
- Push onto `left`

Stop early if `right` becomes empty.
6. Return the last 10 characters left of the cursor.

Since `left` already stores characters in correct order, simply join:

```
left[-10:]
```

### Why it works

The invariant is that:

- `left` always contains exactly the characters to the left of the cursor.
- `right` always contains exactly the characters to the right of the cursor, in reverse order.

Every operation preserves this invariant:

- Insertions append to the left side.
- Deletions remove from the left side.
- Cursor movement transfers characters between sides.

Because the representation always mirrors the actual editor state, the returned results are always correct.

## Python Solution

```python
class TextEditor:

    def __init__(self):
        self.left = []
        self.right = []

    def addText(self, text: str) -> None:
        for ch in text:
            self.left.append(ch)

    def deleteText(self, k: int) -> int:
        deleted = 0

        while self.left and deleted < k:
            self.left.pop()
            deleted += 1

        return deleted

    def cursorLeft(self, k: int) -> str:
        moves = min(k, len(self.left))

        for _ in range(moves):
            self.right.append(self.left.pop())

        return "".join(self.left[-10:])

    def cursorRight(self, k: int) -> str:
        moves = min(k, len(self.right))

        for _ in range(moves):
            self.left.append(self.right.pop())

        return "".join(self.left[-10:])

# Your TextEditor object will be instantiated and called as such:
# obj = TextEditor()
# obj.addText(text)
# param_2 = obj.deleteText(k)
# param_3 = obj.cursorLeft(k)
# param_4 = obj.cursorRight(k)
```

The implementation directly follows the two-stack design.

The constructor initializes empty stacks for both sides of the cursor.

The `addText` method pushes every inserted character onto the left stack because inserted text always ends up immediately before the cursor.

The `deleteText` method repeatedly removes characters from the left stack. The loop stops either after deleting `k` characters or when no characters remain to the left of the cursor.

The `cursorLeft` method transfers characters from the left stack to the right stack. This simulates the cursor moving left through the text.

The `cursorRight` method performs the opposite transfer, moving characters from the right stack back onto the left stack.

Finally, both cursor movement methods return the last up to 10 characters from the left stack, which represents the visible text immediately before the cursor.

## Go Solution

```go
type TextEditor struct {
	left  []byte
	right []byte
}

func Constructor() TextEditor {
	return TextEditor{
		left:  []byte{},
		right: []byte{},
	}
}

func (this *TextEditor) AddText(text string) {
	for i := 0; i < len(text); i++ {
		this.left = append(this.left, text[i])
	}
}

func (this *TextEditor) DeleteText(k int) int {
	deleted := 0

	for len(this.left) > 0 && deleted < k {
		this.left = this.left[:len(this.left)-1]
		deleted++
	}

	return deleted
}

func (this *TextEditor) CursorLeft(k int) string {
	moves := k
	if moves > len(this.left) {
		moves = len(this.left)
	}

	for i := 0; i < moves; i++ {
		ch := this.left[len(this.left)-1]
		this.left = this.left[:len(this.left)-1]
		this.right = append(this.right, ch)
	}

	start := 0
	if len(this.left) > 10 {
		start = len(this.left) - 10
	}

	return string(this.left[start:])
}

func (this *TextEditor) CursorRight(k int) string {
	moves := k
	if moves > len(this.right) {
		moves = len(this.right)
	}

	for i := 0; i < moves; i++ {
		ch := this.right[len(this.right)-1]
		this.right = this.right[:len(this.right)-1]
		this.left = append(this.left, ch)
	}

	start := 0
	if len(this.left) > 10 {
		start = len(this.left) - 10
	}

	return string(this.left[start:])
}

/**
 * Your TextEditor object will be instantiated and called as such:
 * obj := Constructor();
 * obj.AddText(text);
 * param_2 := obj.DeleteText(k);
 * param_3 := obj.CursorLeft(k);
 * param_4 := obj.CursorRight(k);
 */
```

The Go implementation mirrors the Python logic closely.

Instead of Python lists, Go uses `[]byte` slices because the input contains only lowercase English letters. Using bytes is both memory-efficient and fast.

Stack operations are implemented using slice append and slice truncation:

- Push: `append(slice, value)`
- Pop: `slice[:len(slice)-1]`

String conversion is performed only when returning the final visible substring.

### Understanding Cursor Transfers

When performing:

```
cursorLeft(8)
```

characters move one by one:

```
left  -> right
```

Result:

```
leet|practice
```

Internally:

```
left  = ['l','e','e','t']
right = ['e','c','i','t','c','a','r','p']
```

The right side is reversed because the nearest cursor character must remain on top.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(k) per operation | Each operation moves or removes at most `k` characters |
| Space | O(n) | All characters are stored across two stacks |

The important observation is that no operation scans the entire text unless `k` itself is large. Since each call only transfers or modifies at most `k` characters, the follow-up requirement is satisfied.

The total memory usage is linear in the total text size because every character exists in exactly one of the two stacks.

## Edge Cases

### Deleting More Characters Than Exist

A common bug is assuming exactly `k` characters always exist to the left of the cursor. For example:

```
text = "abc|"
deleteText(10)
```

Only three characters can actually be deleted.

The implementation handles this safely by stopping deletion once the left stack becomes empty. The returned value is the actual number removed.

### Moving Cursor Beyond Text Boundaries

Cursor movement must never exceed valid bounds.

For example:

```
|hello
cursorLeft(5)
```

The cursor should remain at the beginning.

Similarly:

```
hello|
cursorRight(5)
```

should keep the cursor at the end.

The implementation solves this by limiting movement using:

```
min(k, len(stack))
```

This guarantees safe transfers.

### Returning Fewer Than 10 Characters

The problem requires returning:

```
last min(10, len)
```

characters to the left of the cursor.

If only four characters exist:

```
abcd|
```

the result must be `"abcd"`, not an invalid substring access.

Using:

```
self.left[-10:]
```

works naturally in Python because slicing automatically clamps boundaries safely.

In Go, the implementation computes a safe starting index before slicing.
