## Solution

## Problem Understanding

The problem asks us to design an ATM machine that supports two operations:

1. Depositing banknotes into the machine.
2. Withdrawing a requested amount of money.

The ATM supports exactly five denominations:

- `$20`
- `$50`
- `$100`
- `$200`
- `$500`

The important rule is that the ATM must always prioritize larger denominations during withdrawal. This is not optional. The machine is required to behave greedily.

For example, suppose the ATM contains:

- one `$500`
- three `$200`

If a user requests `$600`, the ATM must first try to use the `$500` bill because it is the largest denomination available. After using `$500`, the remaining amount is `$100`. Since the ATM cannot form `$100` using the remaining notes, the withdrawal fails, even though three `$200` bills could have formed `$600`.

This detail is the central part of the problem. We are not trying to find any valid combination. We are specifically simulating the ATM's forced greedy behavior.

The `deposit(banknotesCount)` function receives an array of length `5`, where:

- index `0` represents the number of `$20` bills
- index `1` represents the number of `$50` bills
- index `2` represents the number of `$100` bills
- index `3` represents the number of `$200` bills
- index `4` represents the number of `$500` bills

The function adds those notes into the ATM inventory.

The `withdraw(amount)` function attempts to withdraw the requested amount while following the greedy rule. If successful, it returns an array describing how many notes of each denomination were dispensed. If not possible, it returns `[-1]`.

The constraints are relatively small in terms of operation count:

- At most `5000` operations total
- Only `5` denominations

This tells us that efficiency is not primarily about large loops or advanced optimization. Instead, correctness and careful state management are the important parts.

Several edge cases are important:

- A withdrawal may fail even when another valid combination exists.
- The ATM state must remain unchanged after a failed withdrawal.
- The machine may contain extremely large note counts, so we should avoid unnecessary expensive computation.
- The withdrawal amount may not be divisible by available denominations.

## Approaches

### Brute Force Approach

A brute force solution would attempt to generate all possible combinations of banknotes that sum to the target amount.

For example, we could try every possible count of `$500` bills, then every possible count of `$200` bills, and so on until we either find a valid combination or exhaust all possibilities.

This approach would certainly find a valid solution if one exists. However, it does not match the problem requirements because the ATM is not allowed to choose arbitrary combinations. It must follow the greedy rule.

Even if we modified brute force to prioritize larger notes first, exploring combinations is unnecessary and inefficient. The number of possible combinations can become enormous when note counts are large.

Since there are only five denominations and the ATM behavior is deterministic, brute force is excessive.

### Optimal Greedy Simulation

The key observation is that the ATM behavior is completely defined by the problem statement:

- Always try larger denominations first.
- Use as many as possible of each denomination.
- Move downward to smaller denominations.

This means we do not need search or backtracking. We simply simulate exactly what the ATM is instructed to do.

For each denomination from largest to smallest:

1. Compute how many bills are needed.
2. Limit that count by how many bills actually exist.
3. Deduct the corresponding value from the remaining amount.

After processing all denominations:

- If the remaining amount becomes zero, the withdrawal succeeds.
- Otherwise, it fails.

This greedy simulation exactly matches the ATM rules.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | O(1) to O(n) | Tries many combinations of banknotes |
| Optimal | O(1) | O(1) | Simulates the required greedy ATM behavior |

## Algorithm Walkthrough

### Step 1: Store the Denominations

Create a fixed array containing the bill values:

```
[20, 50, 100, 200, 500]
```

We also maintain another array storing how many notes of each denomination currently exist in the ATM.

### Step 2: Handle Deposits

When `deposit(banknotesCount)` is called:

1. Iterate through all five denominations.
2. Add the deposited counts to the ATM inventory.

This operation is straightforward because the denominations are fixed.

### Step 3: Start Withdrawal From Largest Denomination

When `withdraw(amount)` is called:

1. Create a result array initialized with zeros.
2. Process denominations from right to left:

- `$500`
- `$200`
- `$100`
- `$50`
- `$20`

We process largest first because the ATM is forced to prioritize larger notes.

### Step 4: Determine Maximum Bills Usable

For each denomination:

1. Compute how many bills are theoretically needed:

```
amount // denomination
```

1. Compare that with how many bills are available.
2. Use the smaller of the two.

This ensures we never use more notes than the ATM contains.

### Step 5: Reduce Remaining Amount

After choosing how many notes to use:

1. Store the chosen count in the result array.
2. Subtract the corresponding value from the remaining amount.

Then continue to the next smaller denomination.

### Step 6: Check Whether Withdrawal Succeeded

After processing all denominations:

- If remaining amount is `0`, the withdrawal succeeded.
- Otherwise, return `[-1]`.

A nonzero remainder means the ATM could not complete the withdrawal using the required greedy process.

### Step 7: Update ATM State Only On Success

If withdrawal succeeds:

1. Subtract dispensed note counts from the ATM inventory.
2. Return the result array.

If withdrawal fails:

- Do not modify the inventory.

This rollback behavior is essential.

### Why it works

The ATM behavior is deterministic. At every denomination, the machine must take the maximum possible number of larger bills before considering smaller ones. Our algorithm exactly simulates that rule.

Because we process denominations in descending order and always take the maximum usable amount, the produced result is precisely the one the ATM is required to use. If this process cannot complete the withdrawal, then the ATM must reject the request.

## Python Solution

```python
from typing import List

class ATM:

    def __init__(self):
        self.denominations = [20, 50, 100, 200, 500]
        self.counts = [0] * 5

    def deposit(self, banknotesCount: List[int]) -> None:
        for i in range(5):
            self.counts[i] += banknotesCount[i]

    def withdraw(self, amount: int) -> List[int]:
        used = [0] * 5

        for i in range(4, -1, -1):
            denomination = self.denominations[i]

            max_needed = amount // denomination
            notes_to_use = min(max_needed, self.counts[i])

            used[i] = notes_to_use
            amount -= notes_to_use * denomination

        if amount != 0:
            return [-1]

        for i in range(5):
            self.counts[i] -= used[i]

        return used

# Your ATM object will be instantiated and called as such:
# obj = ATM()
# obj.deposit(banknotesCount)
# param_2 = obj.withdraw(amount)
```

The implementation begins by storing two arrays:

- `denominations` stores the fixed bill values.
- `counts` stores how many notes currently exist.

The `deposit` method simply adds incoming notes into the inventory.

The `withdraw` method performs the greedy simulation. It iterates from largest denomination to smallest denomination and computes the maximum usable number of bills at each step.

The `used` array records how many notes are selected for withdrawal. The remaining amount is reduced after every denomination.

After processing all bills, the implementation checks whether the remaining amount is zero. If not, the withdrawal fails and `[-1]` is returned without modifying the ATM inventory.

If the withdrawal succeeds, the selected notes are deducted from the inventory and the result array is returned.

## Go Solution

```go
type ATM struct {
	denominations []int
	counts        []int
}

func Constructor() ATM {
	return ATM{
		denominations: []int{20, 50, 100, 200, 500},
		counts:        make([]int, 5),
	}
}

func (this *ATM) Deposit(banknotesCount []int) {
	for i := 0; i < 5; i++ {
		this.counts[i] += banknotesCount[i]
	}
}

func (this *ATM) Withdraw(amount int) []int {
	used := make([]int, 5)

	for i := 4; i >= 0; i-- {
		denomination := this.denominations[i]

		maxNeeded := amount / denomination

		notesToUse := maxNeeded
		if notesToUse > this.counts[i] {
			notesToUse = this.counts[i]
		}

		used[i] = notesToUse
		amount -= notesToUse * denomination
	}

	if amount != 0 {
		return []int{-1}
	}

	for i := 0; i < 5; i++ {
		this.counts[i] -= used[i]
	}

	return used
}

/**
 * Your ATM object will be instantiated and called as such:
 * obj := Constructor();
 * obj.Deposit(banknotesCount);
 * param_2 := obj.Withdraw(amount);
 */
```

The Go implementation mirrors the Python logic closely. Slices are used for both denomination values and banknote counts.

One small difference is that Go does not provide a built in `min` function for integers, so we manually compare values using an `if` statement.

The problem constraints fit comfortably within Go's `int` type because the total number of deposited notes is bounded by `10^9`.

## Edge Cases

### Greedy Failure Despite Existing Solution

One of the most important edge cases occurs when a valid combination exists, but the greedy rule prevents the ATM from using it.

For example:

```
1 x 500
3 x 200
withdraw 600
```

A human might choose three `$200` bills, but the ATM is forced to use `$500` first. That leaves `$100`, which cannot be formed. The request must fail.

A naive implementation that searches for any valid combination would incorrectly accept this request. Our implementation avoids this bug by strictly processing denominations from largest to smallest.

### Failed Withdrawal Must Not Change State

A common implementation mistake is partially deducting notes during withdrawal and forgetting to restore them if the withdrawal later fails.

For example:

```
ATM has:
1 x 500
1 x 50

withdraw 600
```

The ATM may temporarily select `$500`, but the remaining `$100` cannot be formed.

Our implementation prevents state corruption by storing selected notes in a temporary `used` array. The ATM inventory is only updated after the entire withdrawal succeeds.

### Amount Cannot Be Formed Exactly

Another tricky case occurs when the remaining amount cannot be represented using available denominations.

For example:

```
ATM has:
1 x 20

withdraw 30
```

Even though the ATM contains money, it cannot form the exact amount.

Our algorithm naturally handles this because the remaining amount will still be nonzero after processing all denominations. In that case, the method returns `[-1]`.
