## Solution

## Problem Understanding

The problem gives us an array `nums` containing distinct positive integers. We are also given a sequence of replacement operations. Each operation is a pair `[oldValue, newValue]`, meaning that every occurrence of `oldValue` in the array should be replaced with `newValue`.

Because all numbers in `nums` are distinct, each value appears exactly once at any moment. The problem guarantees two very important properties for every operation:

1. `operations[i][0]` always exists in the array when the operation is applied.
2. `operations[i][1]` does not already exist in the array.

These guarantees eliminate ambiguity and ensure that each replacement affects exactly one position in the array.

The task is to apply all operations in order and return the final version of the array.

For example, if the array is:

```
[1,2,4,6]
```

and we replace `1 -> 3`, then `4 -> 7`, then `6 -> 1`, the array evolves as:

```
[1,2,4,6]
[3,2,4,6]
[3,2,7,6]
[3,2,7,1]
```

The final answer is:

```
[3,2,7,1]
```

The constraints are large:

- `n` and `m` can both be up to `100000`
- Values can be as large as `10^6`

These limits tell us that repeatedly scanning the entire array for each operation would likely be too slow. A solution with quadratic behavior, such as `O(n * m)`, will not pass efficiently for the maximum input sizes.

The most important observation is that every value is unique. This means we can maintain a direct mapping from a number to its index in the array. Then each replacement can be processed in constant time.

Some important edge cases include:

- A single-element array
- Operations that reuse previously removed numbers later
- Long chains of replacements
- Operations where the replacement value later becomes the source of another operation

The problem guarantees consistency, so we never need to handle invalid operations or duplicate values appearing simultaneously.

## Approaches

### Brute Force Approach

The most straightforward solution is to simulate the operations directly.

For each operation `[oldValue, newValue]`, we scan through the array until we find `oldValue`. Once found, we replace it with `newValue`.

This works because the array always contains the value we are looking for, and values are distinct.

However, this approach is inefficient for large inputs. If the array has size `n` and there are `m` operations, then in the worst case we may scan almost the entire array for every operation.

That gives a total complexity of:

```
O(n * m)
```

With both `n` and `m` up to `100000`, this could require up to `10^10` operations, which is far too slow.

### Optimal Approach

The key insight is that values are distinct.

Instead of searching for a value every time, we can maintain a hash map:

```
value -> index in nums
```

This lets us immediately locate where a number exists in the array.

When processing an operation `[oldValue, newValue]`:

1. Use the hash map to find the index of `oldValue`
2. Update `nums[index] = newValue`
3. Remove `oldValue` from the map
4. Insert `newValue -> index` into the map

Each operation now takes constant average time.

This reduces the overall complexity to linear time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * m) | O(1) | Scan the array for every operation |
| Optimal | O(n + m) | O(n) | Use a hash map from value to index |

## Algorithm Walkthrough

1. Create a hash map called `value_to_index`.

This map stores where each number currently exists in the array.

For example:

```
nums = [1,2,4,6]
```

produces:

```
{
    1: 0,
    2: 1,
    4: 2,
    6: 3
}
```
2. Process each operation one by one.

Suppose the operation is:

```
[oldValue, newValue]
```
3. Look up the index of `oldValue` using the hash map.

Because the problem guarantees the value exists, the lookup is always valid.
4. Replace the value in the array.

If `oldValue` is at index `i`, then update:

```
nums[i] = newValue
```
5. Update the hash map.

Remove the old mapping:

```
del value_to_index[oldValue]
```

Then insert the new one:

```
value_to_index[newValue] = i
```
6. Continue until all operations are processed.
7. Return the modified array.

### Why it works

The hash map always maintains the invariant that every current value in the array maps to its exact index.

Whenever a replacement occurs, both the array and the map are updated together. Because every operation replaces exactly one existing value with one new distinct value, the mapping remains correct after every step.

Since each operation updates the correct index directly, the final array is exactly the result of applying all operations in order.

## Python Solution

```python
from typing import List

class Solution:
    def arrayChange(self, nums: List[int], operations: List[List[int]]) -> List[int]:
        value_to_index = {}

        for index, value in enumerate(nums):
            value_to_index[value] = index

        for old_value, new_value in operations:
            index = value_to_index[old_value]

            nums[index] = new_value

            del value_to_index[old_value]
            value_to_index[new_value] = index

        return nums
```

The implementation begins by constructing the `value_to_index` hash map. This preprocessing step records the current position of every number in the array.

Next, the algorithm iterates through each operation. For every replacement, the map provides immediate access to the correct index, avoiding any array scan.

After updating the array element, the hash map is synchronized with the new value. The old mapping is removed because that value no longer exists in the array, and the new value is inserted with the same index.

Finally, the modified array is returned.

## Go Solution

```go
func arrayChange(nums []int, operations [][]int) []int {
    valueToIndex := make(map[int]int)

    for index, value := range nums {
        valueToIndex[value] = index
    }

    for _, operation := range operations {
        oldValue := operation[0]
        newValue := operation[1]

        index := valueToIndex[oldValue]

        nums[index] = newValue

        delete(valueToIndex, oldValue)
        valueToIndex[newValue] = index
    }

    return nums
}
```

The Go implementation follows the same logic as the Python version.

The primary difference is syntax. Go uses `make(map[int]int)` to create the hash map and uses the built-in `delete` function to remove keys.

Slices in Go are mutable, so updating `nums[index]` directly modifies the underlying array. No additional copying is required.

Integer overflow is not a concern because the values are bounded by `10^6`, well within Go's standard integer range.

## Edge Cases

One important edge case is a single-element array. With only one value present, every operation modifies the same position repeatedly. Some implementations accidentally assume multiple indices exist or mishandle map updates after deletion. This implementation works correctly because the map always tracks exactly one valid index.

Another important case is when previously removed values reappear later as replacement targets. For example:

```
[1,2]
[[1,3],[2,1]]
```

After the first operation, `1` disappears. Later, `1` is inserted again. A buggy implementation might incorrectly think `1` is still mapped somewhere old. By deleting the old key before inserting the new one, the implementation guarantees that the map always reflects the current array state.

A third important edge case involves long chains of replacements, such as:

```
[[1,2],[2,3],[3,4],[4,5]]
```

In these scenarios, the replacement target of one operation becomes the source of the next operation. The algorithm handles this naturally because every operation updates both the array and the hash map immediately, ensuring the next lookup uses the newest value.

A final subtle case is maintaining distinctness throughout execution. Since the problem guarantees that the replacement value does not already exist, the map never contains duplicate keys. This allows every value lookup to remain unambiguous and guarantees constant-time access throughout the algorithm.
