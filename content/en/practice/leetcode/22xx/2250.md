## Solution

## Problem Understanding

This problem gives us a collection of axis-aligned rectangles and a collection of query points. Every rectangle has its bottom-left corner fixed at `(0, 0)` and its top-right corner at `(li, hi)`.

That means each rectangle covers all coordinates:

```
0 <= x <= li
0 <= y <= hi
```

For every point `(xj, yj)`, we must determine how many rectangles contain that point.

A rectangle contains a point if both conditions are true:

```
xj <= li
yj <= hi
```

So for each point, we need to count all rectangles whose width is large enough and whose height is large enough.

The input sizes are large:

```
1 <= rectangles.length, points.length <= 5 * 10^4
```

A naive comparison of every point against every rectangle would require:

```
5 * 10^4 * 5 * 10^4 = 2.5 * 10^9
```

operations in the worst case, which is far too slow.

One extremely important constraint changes the nature of the problem:

```
1 <= hi, yj <= 100
```

Even though rectangle widths and point x-coordinates can be as large as `10^9`, heights are very small, only from `1` to `100`.

This allows us to organize rectangles by height and answer queries efficiently.

There are several edge cases worth noticing immediately.

A point may lie exactly on the rectangle boundary. Since boundaries count as contained, comparisons must use `<=` rather than `<`.

Some points may not be contained in any rectangle. In that case, the answer is `0`.

Some rectangles may contain every point. The algorithm must still remain efficient even when many rectangles overlap heavily.

The coordinates can be extremely large, so we cannot build a 2D grid or coordinate matrix.

The rectangles and points are guaranteed to be unique, which simplifies bookkeeping slightly, but uniqueness is not essential to the algorithm.

## Approaches

### Brute Force Approach

The most direct solution is to check every rectangle against every point.

For each point `(x, y)`:

1. Iterate through all rectangles `(l, h)`
2. Check whether:

```
x <= l and y <= h
```
3. Count how many rectangles satisfy the condition

This works because the containment condition is exactly the definition given in the problem.

However, the performance is terrible.

If there are `R` rectangles and `P` points, the complexity becomes:

```
O(R * P)
```

With both up to `5 * 10^4`, this becomes billions of operations.

### Optimal Approach

The key observation is that rectangle heights are very small:

```
1 <= hi <= 100
```

Instead of comparing every rectangle with every point, we group rectangles by height.

For each possible height `h` from `1` to `100`, we store all rectangle widths whose rectangle has that height.

Then:

1. Sort the widths inside each height bucket
2. For a query point `(x, y)`:

- Only heights `h >= y` can contain the point
- For each valid height bucket:

- Count how many widths are `>= x`
- Use binary search to do this efficiently

This transforms the problem into repeated binary searches over sorted arrays.

Because there are only 100 possible heights, the algorithm becomes extremely efficient.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(R * P) | O(1) | Checks every rectangle against every point |
| Optimal | O(R log R + P * 100 * log R) | O(R) | Groups rectangles by height and uses binary search |

## Algorithm Walkthrough

1. Create an array of buckets indexed by height.

Since heights are limited to `1..100`, we create 101 lists:

```
height_groups = [[] for _ in range(101)]
```

Each bucket stores rectangle widths for rectangles with that exact height.
2. Insert rectangle widths into their corresponding height bucket.

For every rectangle `(l, h)`:

```
height_groups[h].append(l)
```

We only store widths because the height is already represented by the bucket index.
3. Sort every height bucket.

Binary search requires sorted data.

After sorting, each bucket contains widths in ascending order.
4. Process each query point independently.

For a point `(x, y)`, only rectangles with:

```
h >= y
```

can contain it.

Therefore, we iterate over heights from `y` to `100`.
5. Use binary search to count valid widths.

Inside a height bucket, we need widths satisfying:

```
l >= x
```

Since the bucket is sorted, we use `bisect_left` to find the first width `>= x`.

If:

```
idx = bisect_left(bucket, x)
```

then:

```
len(bucket) - idx
```

gives the number of valid rectangles in that bucket.
6. Sum contributions from all valid heights.

Add together counts from every height bucket `h >= y`.
7. Store the result for the current point.

### Why it works

The algorithm works because every rectangle that contains a point must satisfy two independent conditions:

```
width >= x
height >= y
```

Grouping rectangles by height guarantees that when we iterate from `y` to `100`, we only consider rectangles whose heights are large enough.

Within each height group, binary search correctly counts all rectangles whose widths are also large enough.

Since every valid rectangle is counted exactly once, and every invalid rectangle is excluded by either the height filter or width filter, the final count is correct.

## Python Solution

```python
from typing import List
from bisect import bisect_left

class Solution:
    def countRectangles(self, rectangles: List[List[int]], points: List[List[int]]) -> List[int]:
        height_groups = [[] for _ in range(101)]

        # Group rectangle widths by height
        for length, height in rectangles:
            height_groups[height].append(length)

        # Sort each bucket for binary search
        for h in range(101):
            height_groups[h].sort()

        result = []

        # Process each point
        for x, y in points:
            count = 0

            # Only heights >= y can contain the point
            for h in range(y, 101):
                widths = height_groups[h]

                # Find first width >= x
                idx = bisect_left(widths, x)

                # Count valid rectangles
                count += len(widths) - idx

            result.append(count)

        return result
```

The implementation begins by creating 101 buckets, one for each possible rectangle height.

Each rectangle contributes its width into the bucket corresponding to its height. This transforms the original 2D containment problem into multiple 1D search problems.

After grouping, every bucket is sorted. Sorting enables binary search, which is crucial for efficiency.

For every query point, the algorithm iterates through all heights that are tall enough to contain the point. Since heights are capped at 100, this loop is always small and bounded.

Inside each bucket, `bisect_left` finds the first width that is at least `x`. Everything to the right of that index represents rectangles wide enough to contain the point.

The counts from all valid height buckets are accumulated into the final answer.

## Go Solution

```go
package main

import (
	"sort"
)

func countRectangles(rectangles [][]int, points [][]int) []int {
	heightGroups := make([][]int, 101)

	// Group rectangle widths by height
	for _, rect := range rectangles {
		length := rect[0]
		height := rect[1]

		heightGroups[height] = append(heightGroups[height], length)
	}

	// Sort each bucket
	for h := 0; h <= 100; h++ {
		sort.Ints(heightGroups[h])
	}

	result := make([]int, len(points))

	// Process each point
	for i, point := range points {
		x := point[0]
		y := point[1]

		count := 0

		// Only heights >= y can contain the point
		for h := y; h <= 100; h++ {
			widths := heightGroups[h]

			// Find first width >= x
			idx := sort.SearchInts(widths, x)

			// Count valid rectangles
			count += len(widths) - idx
		}

		result[i] = count
	}

	return result
}
```

The Go implementation follows the same strategy as the Python solution.

Instead of `bisect_left`, Go uses `sort.SearchInts`, which performs binary search on a sorted slice.

Slices are dynamically sized, so height buckets are represented naturally as `[][]int`.

There are no integer overflow concerns because the maximum answer is at most `5 * 10^4`, which easily fits inside Go's `int`.

Empty buckets are handled naturally because binary search on an empty slice returns `0`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(R log R + P * 100 * log R) | Sorting buckets plus binary searches for each point |
| Space | O(R) | Stores all rectangle widths in height buckets |

The preprocessing phase inserts every rectangle into exactly one bucket and sorts the buckets.

Across all buckets, the total number of stored widths is `R`, so the total sorting cost is bounded by `O(R log R)`.

For every point, we iterate over at most 100 heights. Each iteration performs a binary search on a sorted bucket, giving:

```
O(100 * log R)
```

per point.

Since 100 is a constant, the solution is very efficient in practice.

## Edge Cases

One important edge case occurs when a point lies exactly on the boundary of a rectangle. Since the problem explicitly states that edges count as contained, comparisons must use inclusive logic. The implementation handles this correctly because binary search looks for widths greater than or equal to `x`, and the height iteration includes heights equal to `y`.

Another important edge case is when no rectangles can contain a point. This can happen when the point's x-coordinate exceeds all rectangle widths or its y-coordinate exceeds all rectangle heights. In this situation, every binary search contribution becomes zero, so the final count correctly remains zero.

A third important edge case involves many rectangles sharing the same height. A poorly designed solution might repeatedly scan large groups linearly. This implementation avoids that inefficiency by sorting each height bucket once and using binary search to count valid widths in logarithmic time.

A final edge case involves extremely large coordinate values such as `10^9`. Since the algorithm never creates a coordinate grid and only performs comparisons and binary searches, it handles large coordinate ranges efficiently without excessive memory usage.
