## Solution

## Problem Understanding

The problem gives us an array named `tasks`, where each integer represents the difficulty level of a task. Multiple tasks may share the same difficulty. In one round, we are allowed to complete either exactly 2 tasks or exactly 3 tasks, but all tasks completed in the same round must have the same difficulty level.

Our goal is to determine the minimum number of rounds needed to complete every task. If it is impossible to group the tasks according to the rules, we must return `-1`.

The important detail is that tasks of different difficulties cannot be mixed together. This means we can process each difficulty independently. For example, if difficulty `4` appears 5 times, we only need to figure out the minimum number of rounds needed to complete those 5 tasks.

The constraints are large:

- `tasks.length` can be up to `10^5`
- task values can be as large as `10^9`

The large task values tell us we should not use an array indexed by difficulty. Instead, we should use a hash map or dictionary to count frequencies efficiently.

The size limit of `10^5` also tells us that exponential or quadratic solutions will be too slow. We need a solution close to linear time.

There are several important edge cases:

- A difficulty appearing exactly once is impossible to complete, because each round must contain either 2 or 3 tasks.
- Counts divisible by 3 are straightforward because using groups of 3 minimizes the number of rounds.
- Counts leaving a remainder of 1 when divided by 3 require special handling. For example, 4 tasks cannot be completed as `3 + 1`, so we must instead use `2 + 2`.
- Very small counts like 2 or 3 should still work correctly.
- Large frequencies should be handled efficiently without simulation.

## Approaches

### Brute Force Approach

A brute force strategy would attempt all possible ways to partition each task frequency into groups of 2 and 3. For a frequency `f`, we could recursively try:

- taking one group of 2 and solving for `f - 2`
- taking one group of 3 and solving for `f - 3`

Then we would choose the minimum valid answer.

This approach is correct because it explores every possible grouping combination. However, it is extremely inefficient because the number of recursive possibilities grows exponentially with the frequency count.

For example, a count of 100 would generate a massive recursion tree with many repeated subproblems.

Even if memoization is added, we still do unnecessary work compared to the much simpler mathematical observation available in this problem.

### Optimal Greedy Counting Approach

The key observation is that groups of 3 are always more efficient than groups of 2 because they complete more tasks per round.

Therefore, we should use as many groups of 3 as possible.

However, there is one tricky case:

- If the count leaves remainder `1` when divided by 3, we cannot leave a single task alone.

For example:

- `4 = 2 + 2`
- `7 = 3 + 2 + 2`
- `10 = 3 + 3 + 2 + 2`

In these cases, we convert one group of 3 into two groups of 2.

This leads to a simple mathematical formula:

- If frequency is `1`, return `-1`
- Otherwise, minimum rounds = `(frequency + 2) // 3`

This formula works because it effectively computes the ceiling of `frequency / 3`.

## Algorithm Walkthrough

1. Create a hash map to count how many times each difficulty appears.

We process the input array once and store frequencies in a dictionary. This lets us independently solve the problem for each difficulty level.
2. Initialize a variable `total_rounds` to 0.

This variable accumulates the minimum rounds required across all difficulties.
3. Iterate through every frequency in the hash map.

Each frequency represents the number of tasks for one difficulty.
4. If a frequency equals 1, immediately return `-1`.

A single task cannot form a valid round because rounds require either 2 or 3 tasks.
5. Otherwise, compute the minimum rounds for this frequency.

We want to maximize groups of 3 because they minimize the number of rounds.

The formula:

```
(frequency + 2) // 3
```

computes the ceiling of `frequency / 3`.
6. Add the computed rounds to `total_rounds`.
7. After processing all frequencies, return `total_rounds`.

### Why it works

The greedy strategy works because groups of 3 are always more efficient than groups of 2. Every time we replace two groups of 2 with one group of 3 and one leftover task, we reduce efficiency and create invalid states.

The only problematic remainder is `1`. In that case, we convert one group of 3 into two groups of 2. This guarantees all tasks are grouped validly while still minimizing the number of rounds.

Therefore, the formula always produces the minimum number of rounds for every valid frequency.

## Python Solution

```python
from collections import Counter
from typing import List

class Solution:
    def minimumRounds(self, tasks: List[int]) -> int:
        frequency = Counter(tasks)
        
        total_rounds = 0
        
        for count in frequency.values():
            if count == 1:
                return -1
            
            total_rounds += (count + 2) // 3
        
        return total_rounds
```

The implementation begins by using `Counter` from the `collections` module to count how many times each task difficulty appears.

We then iterate through every frequency. If any frequency equals `1`, we immediately return `-1` because it is impossible to create a valid round.

For all other counts, we compute the minimum rounds using `(count + 2) // 3`. This expression performs ceiling division by 3, which matches the optimal greedy strategy.

Finally, we accumulate the result and return the total number of rounds.

## Go Solution

```go
func minimumRounds(tasks []int) int {
    frequency := make(map[int]int)

    for _, task := range tasks {
        frequency[task]++
    }

    totalRounds := 0

    for _, count := range frequency {
        if count == 1 {
            return -1
        }

        totalRounds += (count + 2) / 3
    }

    return totalRounds
}
```

The Go implementation follows the same logic as the Python version.

Instead of using `Counter`, Go uses a `map[int]int` to store frequencies. Integer division in Go automatically truncates toward zero, so `(count + 2) / 3` correctly computes ceiling division for positive integers.

No special handling for nil or empty input is necessary because the problem guarantees at least one task.

## Edge Cases

One important edge case occurs when a difficulty appears exactly once. Since every round must contain either 2 or 3 tasks, a single task can never be completed. A naive implementation might incorrectly attempt to process it mathematically and produce an invalid answer. The implementation explicitly checks for `count == 1` and immediately returns `-1`.

Another important case occurs when the frequency leaves remainder `1` after division by 3. For example, a count of `4` cannot be handled as `3 + 1`. The algorithm correctly transforms this into `2 + 2`. The ceiling division formula `(count + 2) // 3` naturally handles this situation without requiring separate branching logic.

A third edge case involves very large inputs with many repeated tasks. Since the constraints allow up to `10^5` tasks, inefficient recursive or simulation-based approaches could time out. The implementation uses a hash map and constant-time arithmetic operations, ensuring the solution remains efficient even at maximum input size.
