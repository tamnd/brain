# LeetCode 2234 - Maximum Total Beauty of the Gardens

## Problem Understanding

The problem gives us `n` gardens, where `flowers[i]` represents the number of flowers already planted in the `i-th` garden. We are allowed to add at most `newFlowers` additional flowers across all gardens. Existing flowers cannot be removed.

A garden becomes **complete** if it contains at least `target` flowers. The final beauty score is calculated using two components:

1. Every complete garden contributes `full` beauty points.
2. Among the remaining incomplete gardens, we look at the minimum flower count, and multiply it by `partial`.

The goal is to distribute the new flowers in a way that maximizes the total beauty.

A key detail is that the partial contribution only depends on the minimum value among incomplete gardens. This means that if we decide to leave some gardens incomplete, we should try to raise the smallest incomplete gardens as much as possible.

The constraints are large:

- `n` can be up to `10^5`
- `newFlowers` can be as large as `10^10`

These limits immediately rule out exponential or quadratic brute-force approaches. We need an algorithm around `O(n log n)`.

Several edge cases are important:

- All gardens may already be complete.
- We may have enough flowers to complete every garden.
- It may be better not to complete all gardens if `partial` is large.
- Gardens already above `target` should be treated carefully because additional flowers beyond `target` provide no extra benefit.
- The minimum incomplete value must always stay strictly below `target`, because otherwise that garden would also become complete.

## Approaches

### Brute Force Approach

A brute-force strategy would attempt every possible distribution of `newFlowers` among gardens and compute the resulting beauty score.

For each configuration, we would:

1. Determine which gardens become complete.
2. Find the minimum incomplete garden.
3. Compute the total beauty.

This approach is correct because it explores all valid allocations, guaranteeing the optimal answer is found.

However, the number of ways to distribute flowers grows combinatorially. Even for small inputs, this becomes infeasible. With `n = 10^5`, brute force is completely impossible.

### Key Insight for the Optimal Solution

The important observation is that the final arrangement has a very structured form.

If we choose some gardens to become complete, then among the remaining incomplete gardens, the optimal strategy is to raise the smallest gardens evenly until we run out of flowers.

This suggests:

1. Sort the gardens.
2. Try every possible number of complete gardens.
3. For the remaining incomplete gardens, maximize their minimum value.

To efficiently maximize the minimum incomplete value, we use:

- Prefix sums
- Binary search
- Greedy reasoning

Another crucial optimization is that any garden with more than `target` flowers can be capped at `target`, since extra flowers beyond `target` never increase beauty.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Tries every flower distribution |
| Optimal | O(n log n) | O(n) | Sorting, prefix sums, binary search |

## Algorithm Walkthrough

### Step 1: Cap flower counts at `target`

Any garden with more than `target` flowers is effectively equivalent to exactly `target` flowers, because extra flowers provide no additional benefit.

So we replace:

```
flowers[i] = min(flowers[i], target)
```

This simplifies later calculations.

### Step 2: Sort the gardens

Sorting allows us to efficiently:

- Complete the largest gardens first
- Raise the smallest incomplete gardens uniformly

After sorting, gardens with fewer flowers appear first.

### Step 3: Build a prefix sum array

We create:

```
prefix[i]
```

which stores the sum of the first `i` flower counts.

This lets us quickly compute the total flowers needed to raise a range of gardens to a certain level.

### Step 4: Compute completion costs

We iterate from the end of the sorted array and calculate how many flowers are needed to make the largest gardens complete.

For example:

- To complete one garden:

```
target - flowers[n-1]
```

- To complete two gardens:

```
(target - flowers[n-1]) +
(target - flowers[n-2])
```

We progressively track this cumulative cost.

### Step 5: Try every number of complete gardens

Suppose we decide to make `k` gardens complete.

Then:

- The last `k` gardens become complete.
- The remaining `n-k` gardens stay incomplete.

We compute how many flowers remain after completing those `k` gardens.

If the remaining flower budget is negative, this configuration is impossible.

### Step 6: Maximize the minimum incomplete value

Now we want to maximize the smallest value among the incomplete gardens.

Suppose we want every incomplete garden to be at least `x`.

We can binary search for the largest feasible `x`.

For each candidate `x`:

1. Find how many gardens are below `x`
2. Compute the cost to raise them all to `x`
3. Check whether the remaining flowers are enough

Prefix sums allow this check in constant time after binary search indexing.

The incomplete minimum cannot reach `target`, because then those gardens would become complete.

So the maximum possible incomplete value is:

```
target - 1
```

### Step 7: Compute beauty

For every valid configuration:

```
beauty =
(complete gardens * full) +
(min incomplete * partial)
```

We track the maximum value across all configurations.

### Why it works

The algorithm works because the optimal arrangement always has a monotonic structure after sorting.

If some gardens remain incomplete, then maximizing the minimum incomplete value is equivalent to leveling the smallest gardens as evenly as possible. Any uneven allocation would waste flowers without improving the minimum.

Trying every possible number of complete gardens guarantees that we never miss the globally optimal balance between complete and incomplete contributions.

Sorting and prefix sums allow these evaluations to be performed efficiently.

## Python Solution

```python
from bisect import bisect_right
from typing import List

class Solution:
    def maximumBeauty(
        self,
        flowers: List[int],
        newFlowers: int,
        target: int,
        full: int,
        partial: int
    ) -> int:

        n = len(flowers)

        flowers = [min(f, target) for f in flowers]
        flowers.sort()

        prefix = [0] * (n + 1)

        for i in range(n):
            prefix[i + 1] = prefix[i] + flowers[i]

        if flowers[0] == target:
            return n * full

        complete_cost = [0] * (n + 1)

        for k in range(1, n + 1):
            idx = n - k
            complete_cost[k] = (
                complete_cost[k - 1]
                + (target - flowers[idx])
            )

        answer = 0

        for complete in range(n + 1):

            cost_to_complete = complete_cost[complete]

            if cost_to_complete > newFlowers:
                continue

            remaining = newFlowers - cost_to_complete

            incomplete_count = n - complete

            if incomplete_count == 0:
                answer = max(answer, complete * full)
                continue

            left = 0
            right = target - 1
            best_partial = 0

            while left <= right:

                mid = (left + right) // 2

                pos = bisect_right(
                    flowers,
                    mid,
                    0,
                    incomplete_count
                )

                needed = mid * pos - prefix[pos]

                if needed <= remaining:
                    best_partial = mid
                    left = mid + 1
                else:
                    right = mid - 1

            beauty = (
                complete * full
                + best_partial * partial
            )

            answer = max(answer, beauty)

        return answer
```

The implementation begins by capping every garden at `target`, because excess flowers above the threshold never improve the answer.

The gardens are then sorted so that incomplete gardens can be processed from smallest to largest. Prefix sums are constructed to allow constant-time range sum calculations.

The `complete_cost` array stores how many flowers are required to make the largest `k` gardens complete. This allows each configuration to be evaluated quickly.

For every possible number of complete gardens, the algorithm computes the remaining flower budget. If some gardens remain incomplete, a binary search is performed to maximize the minimum incomplete value.

The binary search uses `bisect_right` to determine how many gardens must be raised to the candidate value. Prefix sums then compute the exact flower cost efficiently.

Finally, the beauty score is computed and the global maximum is updated.

## Go Solution

```go
package main

import (
	"sort"
)

func maximumBeauty(flowers []int, newFlowers int64, target int, full int, partial int) int64 {

	n := len(flowers)

	for i := 0; i < n; i++ {
		if flowers[i] > target {
			flowers[i] = target
		}
	}

	sort.Ints(flowers)

	if flowers[0] == target {
		return int64(n * full)
	}

	prefix := make([]int64, n+1)

	for i := 0; i < n; i++ {
		prefix[i+1] = prefix[i] + int64(flowers[i])
	}

	completeCost := make([]int64, n+1)

	for k := 1; k <= n; k++ {
		idx := n - k
		completeCost[k] = completeCost[k-1] +
			int64(target-flowers[idx])
	}

	var answer int64 = 0

	for complete := 0; complete <= n; complete++ {

		cost := completeCost[complete]

		if cost > newFlowers {
			continue
		}

		remaining := newFlowers - cost

		incompleteCount := n - complete

		if incompleteCount == 0 {
			beauty := int64(complete * full)

			if beauty > answer {
				answer = beauty
			}

			continue
		}

		left := 0
		right := target - 1
		bestPartial := 0

		for left <= right {

			mid := (left + right) / 2

			pos := sort.Search(
				incompleteCount,
				func(i int) bool {
					return flowers[i] > mid
				},
			)

			needed := int64(mid*pos) - prefix[pos]

			if needed <= remaining {
				bestPartial = mid
				left = mid + 1
			} else {
				right = mid - 1
			}
		}

		beauty := int64(complete*full) +
			int64(bestPartial*partial)

		if beauty > answer {
			answer = beauty
		}
	}

	return answer
}
```

The Go implementation follows the same logic as the Python version, but uses `int64` extensively because `newFlowers` can reach `10^10`, which exceeds the range of a standard 32-bit integer.

Go's `sort.Search` is used instead of Python's `bisect_right`. Prefix sums are stored as `[]int64` to avoid overflow during large arithmetic operations.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | Sorting plus binary searches |
| Space | O(n) | Prefix sums and auxiliary arrays |

The dominant cost is sorting the gardens, which requires `O(n log n)` time. For each possible number of complete gardens, we perform a binary search over the incomplete minimum value. Since there are `n` iterations and each binary search is logarithmic, the overall complexity remains `O(n log n)`.

The extra space comes from the prefix sum array and completion cost array.

## Edge Cases

One important edge case occurs when all gardens are already complete before adding any flowers. In this situation, the partial contribution must be zero because there are no incomplete gardens. A buggy implementation might still attempt to compute a minimum incomplete value. The solution handles this by immediately returning `n * full`.

Another tricky case happens when it is possible to complete every garden, but doing so is not optimal. If `partial` is very large, leaving exactly one garden incomplete with a high minimum value can produce a better total beauty than completing everything. The algorithm correctly explores every possible number of completed gardens, so it naturally considers both possibilities.

A third important edge case involves gardens already exceeding `target`. Since flowers above `target` do not increase beauty, failing to cap these values can produce incorrect completion costs and distort binary search behavior. The implementation safely converts all values larger than `target` into exactly `target` before processing begins.
