# LeetCode 2265 - Count Nodes Equal to Average of Subtree

## Problem Understanding

This problem gives us the root of a binary tree and asks us to count how many nodes satisfy a specific condition.

For every node in the tree, we consider its entire subtree. A subtree includes the node itself and all of its descendants. We then compute the average value of all nodes inside that subtree. If the node's value is equal to that average, we count it.

The important detail is that the average is rounded down to the nearest integer. In practice, since all values are integers, this is equivalent to using integer division.

For example, suppose a subtree contains values `[5, 6]`. The sum is `11`, the number of nodes is `2`, and the average is:

$\left\lfloor \frac{11}{2} \right\rfloor = 5$

Since the root node of that subtree has value `5`, it contributes to the answer.

The input is a standard binary tree structure where each node contains:

- An integer value
- A left child
- A right child

The output is a single integer representing how many nodes satisfy the condition.

The constraints are relatively small:

- At most 1000 nodes
- Node values range from 0 to 1000

Even though the constraints are not huge, we still want an efficient solution because repeatedly recomputing subtree sums can become unnecessarily expensive.

Several edge cases are important:

- A tree with only one node, every single-node subtree always has an average equal to its own value.
- Trees with missing children, we must correctly handle `None` or `nil`.
- Integer division behavior, we must use floor division rather than floating point averages.
- Deep recursive traversal, although the constraints are small enough that recursion is safe in both Python and Go.

## Approaches

### Brute Force Approach

A straightforward solution is to process every node independently.

For each node:

1. Traverse its entire subtree.
2. Compute the subtree sum.
3. Compute the subtree node count.
4. Calculate the average using integer division.
5. Compare the average with the node value.

This approach is correct because every node explicitly recomputes the information needed for its own subtree.

However, it is inefficient because the same nodes are visited many times. For example, a leaf node may be revisited while processing each ancestor above it.

In the worst case, such as a skewed tree:

- We compute subtree information for `n` nodes.
- Each computation may visit up to `n` nodes.

This leads to quadratic time complexity.

### Optimal Approach

The key observation is that subtree information can be reused.

For every subtree, we only need two pieces of information:

- The total sum of values
- The total number of nodes

If we already know these values for the left and right subtrees, we can immediately compute them for the current node.

This naturally suggests a postorder depth-first traversal:

1. Process the left subtree.
2. Process the right subtree.
3. Compute the current subtree information.

This avoids repeated work because each node contributes to the computation exactly once.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(h) | Recomputes subtree sums and counts repeatedly |
| Optimal | O(n) | O(h) | Single DFS traversal computes all subtree information |

Here:

- `n` is the number of nodes
- `h` is the height of the tree

## Algorithm Walkthrough

1. Start a depth-first traversal from the root using postorder traversal.
2. For each node, recursively process the left subtree first. The recursive call returns:

- The sum of all node values in the left subtree
- The number of nodes in the left subtree
3. Recursively process the right subtree and obtain the same information for the right side.
4. Combine the information from both children with the current node:

- Current subtree sum = left sum + right sum + current node value
- Current subtree count = left count + right count + 1
5. Compute the subtree average using integer division.

$\text{average} = \left\lfloor \frac{\text{subtree sum}}{\text{subtree count}} \right\rfloor$

1. Compare the computed average with the current node value.

- If they are equal, increment the answer counter.
2. Return the subtree sum and subtree count to the parent call.
3. Continue until the entire tree has been processed.

### Why it works

The algorithm works because every subtree is processed after its children have already been fully evaluated. This guarantees that when processing a node, we already know the exact sum and node count for both subtrees.

By combining these values correctly, we obtain the exact subtree statistics for the current node. Since every node is checked exactly once using accurate subtree information, the final count is correct.

## Python Solution

```python
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import Optional, Tuple

class Solution:
    def averageOfSubtree(self, root: Optional[TreeNode]) -> int:
        result = 0

        def dfs(node: Optional[TreeNode]) -> Tuple[int, int]:
            nonlocal result

            if not node:
                return 0, 0

            left_sum, left_count = dfs(node.left)
            right_sum, right_count = dfs(node.right)

            subtree_sum = left_sum + right_sum + node.val
            subtree_count = left_count + right_count + 1

            average = subtree_sum // subtree_count

            if average == node.val:
                result += 1

            return subtree_sum, subtree_count

        dfs(root)

        return result
```

The implementation uses a nested helper function called `dfs`.

The helper function returns two values:

- The sum of the current subtree
- The number of nodes in the current subtree

The base case handles `None` nodes by returning `(0, 0)`. This allows parent nodes to combine subtree information cleanly without special handling.

The recursive calls compute the left and right subtree information first. Once both are available, the current subtree statistics are calculated.

The average is computed using integer floor division with `//`, which matches the problem requirements exactly.

Whenever the node value matches the computed average, the global counter `result` is incremented.

Finally, the traversal starts at the root and the total count is returned.

## Go Solution

```go
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */

func averageOfSubtree(root *TreeNode) int {
    result := 0

    var dfs func(node *TreeNode) (int, int)

    dfs = func(node *TreeNode) (int, int) {
        if node == nil {
            return 0, 0
        }

        leftSum, leftCount := dfs(node.Left)
        rightSum, rightCount := dfs(node.Right)

        subtreeSum := leftSum + rightSum + node.Val
        subtreeCount := leftCount + rightCount + 1

        average := subtreeSum / subtreeCount

        if average == node.Val {
            result++
        }

        return subtreeSum, subtreeCount
    }

    dfs(root)

    return result
}
```

The Go implementation follows the same logic as the Python version.

A recursive closure named `dfs` returns two integers:

- The subtree sum
- The subtree node count

Go automatically performs integer division for integers, so `subtreeSum / subtreeCount` already gives the floor value required by the problem.

The implementation uses `nil` checks instead of Python's `None`. Since the constraints are small, integer overflow is not a concern in Go.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every node is visited exactly once |
| Space | O(h) | Recursive call stack depends on tree height |

The traversal processes each node a single time and performs only constant-time work per node.

The auxiliary space comes from recursion. In a balanced tree, the recursion depth is `O(log n)`. In the worst case of a skewed tree, it becomes `O(n)`.

## Edge Cases

### Single Node Tree

A tree containing only one node is the smallest valid input. The subtree contains exactly that node, so the average is always equal to the node value.

This case can expose bugs if the implementation incorrectly handles empty child subtrees. The recursive base case returning `(0, 0)` ensures the calculation remains correct.

### Highly Unbalanced Tree

A skewed tree behaves almost like a linked list. Recursive algorithms must still correctly aggregate subtree sums and counts while traversing deeply nested structures.

The implementation handles this naturally because each recursive call independently computes its subtree information before returning upward.

### Integer Division Behavior

The problem explicitly requires rounding down when computing averages. Using floating point division followed by rounding could introduce unnecessary complexity or mistakes.

The implementation avoids this issue entirely by using integer division:

- Python uses `//`
- Go uses integer `/`

This guarantees the correct floor behavior for every subtree.
