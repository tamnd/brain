## Solution

## Problem Understanding

This problem asks us to compute the minimum cost to reach the last index of an array while respecting a set of specialized jump rules. The array `nums` represents heights or values at each index, and `costs` represents the cost of jumping **to** a particular index.

You start at index 0. A jump from index `i` to index `j` is allowed if either of the following conditions holds:

1. `nums[i] <= nums[j]` and all intermediate numbers between `i` and `j` are strictly less than `nums[i]`.
2. `nums[i] > nums[j]` and all intermediate numbers between `i` and `j` are greater than or equal to `nums[i]`.

The task is to find the path from index 0 to index `n-1` that minimizes the sum of the costs associated with the indices visited (excluding the starting index since cost is for jumping **to** an index).

Key constraints:

- `1 <= n <= 10^5`, which rules out any naive `O(n^2)` approach.
- Values in `nums` and `costs` are non-negative integers up to `10^5`.

Important edge cases include small arrays (length 1 or 2), arrays with strictly increasing or decreasing sequences, and sequences with repeated numbers, which could affect jump eligibility.

## Approaches

The brute-force approach would be to explore all valid jumps from each index recursively, keeping track of accumulated costs. This guarantees correctness because it checks all paths, but it is infeasible for large `n` due to exponential time complexity.

The optimal approach leverages the fact that the jump conditions define **monotonic segments**. We can think of the array as a graph where nodes are indices and edges are valid jumps. By using a **monotonic stack** strategy, we can efficiently find all valid next jumps without checking every intermediate index. Additionally, we can apply **dynamic programming (DP)** where `dp[i]` stores the minimum cost to reach index `i`. The stack helps maintain candidates for jumps in a monotonic fashion (one for increasing sequences and one for decreasing sequences).

The key insight is that for a strictly increasing jump, only the last smaller number in the potential jump window matters, and symmetrically for decreasing jumps. Monotonic stacks let us track these efficiently.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(n) | Explore all jump sequences recursively. Too slow. |
| Optimal | O(n) | O(n) | Use dynamic programming with two monotonic stacks to track valid jump candidates. Efficient for large `n`. |

## Algorithm Walkthrough

1. Initialize a DP array `dp` of size `n` with `dp[0] = 0` (starting index) and infinity for other indices.
2. Maintain two stacks: `inc_stack` for tracking indices forming increasing sequences, `dec_stack` for decreasing sequences.
3. Iterate over each index `i` from 1 to `n-1`:

1. While `inc_stack` is not empty and `nums[i] >= nums[inc_stack[-1]]`, pop from `inc_stack`. This ensures that the stack only contains indices for which a valid increasing jump is possible.
2. If `inc_stack` is not empty, update `dp[i] = min(dp[i], dp[inc_stack[-1]] + costs[i])`.
3. Push `i` to `inc_stack`.
4. Repeat symmetric steps for `dec_stack` for decreasing jumps.
4. After processing all indices, `dp[n-1]` contains the minimum cost to reach the last index.

**Why it works:** The stacks maintain candidates for valid jumps by monotonic property. Each index only considers the most relevant previous jump positions. DP ensures that the minimum cost is propagated efficiently along valid paths.

## Python Solution

```python
from typing import List
import math

class Solution:
    def minCost(self, nums: List[int], costs: List[int]) -> int:
        n = len(nums)
        dp = [math.inf] * n
        dp[0] = 0
        
        inc_stack = []
        dec_stack = []
        
        for i in range(n):
            while inc_stack and nums[i] >= nums[inc_stack[-1]]:
                j = inc_stack.pop()
                dp[i] = min(dp[i], dp[j] + costs[i])
            if inc_stack:
                dp[i] = min(dp[i], dp[inc_stack[-1]] + costs[i])
            inc_stack.append(i)
            
            while dec_stack and nums[i] <= nums[dec_stack[-1]]:
                j = dec_stack.pop()
                dp[i] = min(dp[i], dp[j] + costs[i])
            if dec_stack:
                dp[i] = min(dp[i], dp[dec_stack[-1]] + costs[i])
            dec_stack.append(i)
        
        return dp[-1]
```

**Implementation Explanation:** We use two stacks to maintain potential jump origins for increasing and decreasing sequences. Popping from the stack ensures we only consider valid jumps according to the problem rules. DP updates the minimum cost to reach each index. By the end, `dp[-1]` contains the minimum cost to reach the last index.

## Go Solution

```go
package main

import (
    "math"
)

func minCost(nums []int, costs []int) int64 {
    n := len(nums)
    dp := make([]int64, n)
    for i := 1; i < n; i++ {
        dp[i] = math.MaxInt64
    }
    
    incStack := []int{}
    decStack := []int{}
    
    for i := 0; i < n; i++ {
        for len(incStack) > 0 && nums[i] >= nums[incStack[len(incStack)-1]] {
            j := incStack[len(incStack)-1]
            incStack = incStack[:len(incStack)-1]
            if dp[j]+int64(costs[i]) < dp[i] {
                dp[i] = dp[j] + int64(costs[i])
            }
        }
        if len(incStack) > 0 && dp[incStack[len(incStack)-1]]+int64(costs[i]) < dp[i] {
            dp[i] = dp[incStack[len(incStack)-1]] + int64(costs[i])
        }
        incStack = append(incStack, i)
        
        for len(decStack) > 0 && nums[i] <= nums[decStack[len(decStack)-1]] {
            j := decStack[len(decStack)-1]
            decStack = decStack[:len(decStack)-1]
            if dp[j]+int64(costs[i]) < dp[i] {
                dp[i] = dp[j] + int64(costs[i])
            }
        }
        if len(decStack) > 0 && dp[decStack[len(decStack)-1]]+int64(costs[i]) < dp[i] {
            dp[i] = dp[decStack[len(decStack)-1]] + int64(costs[i])
        }
        decStack = append(decStack, i)
    }
    
    return dp[n-1]
}
```

**Go-Specific Notes:** The main differences are handling integer overflow with `int64` for `dp` and using slices as stacks. The algorithm logic remains identical to Python.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each index is pushed and popped from each stack at most once. |
| Space | O(n) | DP array and two stacks of size up to n. |

The monotonic stacks allow us to consider only relevant jump origins efficiently, avoiding O(n^2) comparisons.
