## Solution

## Problem Understanding

The problem gives us a 2D grid called `heights`, where each cell represents the height of a person standing at that position. For every person in the grid, we need to determine how many other people they can see either to their right in the same row or downward in the same column.

Visibility follows a strict rule. A person at position `(r1, c1)` can see another person `(r2, c2)` only if:

- They are aligned horizontally or vertically in the allowed direction.
- Every person between them has a height strictly smaller than both endpoint people.

This condition is very important. It means visibility is blocked as soon as we encounter someone tall enough. More specifically, if a person between two positions has height greater than or equal to the shorter endpoint, then visibility fails.

The output must be another matrix of the same size, where `answer[i][j]` contains the total number of visible people from that position.

The constraints are:

- Up to `400 x 400 = 160000` cells.
- Heights can be as large as `10^5`.

The grid size tells us that a naive solution checking every pair of cells would be too slow. An `O((mn)^2)` or even `O(mn(m+n))` solution may become expensive near the upper bound. We need something closer to linear processing per row and column.

Several edge cases are important:

- A single cell grid should return `0`.
- Strictly increasing rows or columns produce many visible people.
- Strictly decreasing rows or columns behave differently because the first taller person immediately blocks further visibility.
- Equal heights are especially tricky because the rule requires intermediate people to be strictly shorter than both endpoints.

## Approaches

### Brute Force Approach

The most direct solution is to process every cell independently.

For each position `(i, j)`:

- Scan to the right cell by cell.
- Keep checking whether all intermediate heights are smaller than both endpoint heights.
- Scan downward similarly.

This approach is correct because it explicitly checks the visibility condition for every candidate pair.

However, it is inefficient. For each cell, we may scan an entire row and column. Since there are `m * n` cells, the total complexity becomes very large.

In the worst case:

- Each cell scans `O(m + n)` positions.
- Total complexity becomes `O(mn(m+n))`.

With `m = n = 400`, this can become too slow.

### Key Insight

This problem is fundamentally about finding the next blocking height while preserving visibility relationships.

A monotonic stack works perfectly here because:

- Shorter people remain visible until a taller or equal person appears.
- Once a taller person appears, all smaller people behind them become irrelevant.
- Every person is pushed and popped at most once.

We can process rows and columns independently because visibility only occurs horizontally or vertically.

For each row:

- Traverse from right to left.
- Maintain a decreasing monotonic stack of heights.
- Pop shorter heights because the current person can see them.
- If the stack still contains someone afterward, the current person can also see the first taller or equal person.

The same logic applies to columns by traversing bottom to top.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(mn(m+n)) | O(1) | Explicitly checks all visibility relationships |
| Optimal | O(mn) | O(m+n) | Uses monotonic stacks for rows and columns |

## Algorithm Walkthrough

1. Initialize an answer matrix filled with zeros.

Every cell will accumulate the number of visible people from both row and column directions.
2. Process each row independently from right to left.

We use a monotonic decreasing stack. The stack stores heights of people to the right that are still relevant for future visibility checks.
3. For the current cell, repeatedly pop all shorter heights from the stack.

Each popped height represents a visible person because:

- They are shorter than the current person.
- No taller person blocked them before reaching the current position.

For every popped element, increment the visibility count.
4. After popping shorter heights, check whether the stack is still nonempty.

If it is, then the current person can also see the first taller or equal person remaining on the stack.

Increment the visibility count once more.
5. Push the current height onto the stack.

This person may become visible to future positions further left.
6. Repeat the same procedure for every column.

Traverse bottom to top instead of right to left.
7. Return the completed answer matrix.

### Why it works

The monotonic stack maintains a decreasing sequence of heights. When processing a new person:

- All shorter people are directly visible and get popped.
- The first remaining taller or equal person is also visible because nobody between them is tall enough to block visibility.
- Any people hidden behind a taller person can never become visible from the current position.

Since every element enters and leaves the stack at most once, the algorithm remains linear.

## Python Solution

```python
from typing import List

class Solution:
    def seePeople(self, heights: List[List[int]]) -> List[List[int]]:
        rows = len(heights)
        cols = len(heights[0])

        answer = [[0] * cols for _ in range(rows)]

        # Process rows
        for r in range(rows):
            stack = []

            for c in range(cols - 1, -1, -1):
                current_height = heights[r][c]

                while stack and stack[-1] < current_height:
                    answer[r][c] += 1
                    stack.pop()

                if stack:
                    answer[r][c] += 1

                stack.append(current_height)

        # Process columns
        for c in range(cols):
            stack = []

            for r in range(rows - 1, -1, -1):
                current_height = heights[r][c]

                while stack and stack[-1] < current_height:
                    answer[r][c] += 1
                    stack.pop()

                if stack:
                    answer[r][c] += 1

                stack.append(current_height)

        return answer
```

The implementation starts by creating the result matrix initialized to zeros.

The first major section processes each row. We traverse from right to left because visibility only extends to the right. The stack maintains decreasing heights. Whenever a shorter person is popped, that person is visible to the current position.

After removing all shorter heights, if the stack still contains someone, then the current person can see exactly one more person, the first taller or equal person.

The second section applies the exact same logic to columns. The only difference is traversal direction. We move bottom to top because visibility extends downward.

Finally, the completed answer matrix is returned.

## Go Solution

```go
package main

func seePeople(heights [][]int) [][]int {
	rows := len(heights)
	cols := len(heights[0])

	answer := make([][]int, rows)
	for i := 0; i < rows; i++ {
		answer[i] = make([]int, cols)
	}

	// Process rows
	for r := 0; r < rows; r++ {
		stack := []int{}

		for c := cols - 1; c >= 0; c-- {
			currentHeight := heights[r][c]

			for len(stack) > 0 && stack[len(stack)-1] < currentHeight {
				answer[r][c]++
				stack = stack[:len(stack)-1]
			}

			if len(stack) > 0 {
				answer[r][c]++
			}

			stack = append(stack, currentHeight)
		}
	}

	// Process columns
	for c := 0; c < cols; c++ {
		stack := []int{}

		for r := rows - 1; r >= 0; r-- {
			currentHeight := heights[r][c]

			for len(stack) > 0 && stack[len(stack)-1] < currentHeight {
				answer[r][c]++
				stack = stack[:len(stack)-1]
			}

			if len(stack) > 0 {
				answer[r][c]++
			}

			stack = append(stack, currentHeight)
		}
	}

	return answer
}
```

The Go implementation closely mirrors the Python version.

Slices are used as stacks, with the end of the slice representing the stack top. Popping is implemented by reslicing with `stack[:len(stack)-1]`.

Go does not require special handling for integer overflow here because the maximum visibility count is bounded by `m + n`, which easily fits inside a standard `int`.

### Example 1


### Row Processing

We process from right to left.

| Position | Height | Stack Before | Visible Count Added | Stack After |
| --- | --- | --- | --- | --- |
| 4 | 5 | [] | 0 | [5] |
| 3 | 2 | [5] | 1 | [5,2] |
| 2 | 4 | [5,2] | pop 2, +1, see 5, +1 | [5,4] |
| 1 | 1 | [5,4] | see 4, +1 | [5,4,1] |
| 0 | 3 | [5,4,1] | pop 1, +1, see 4, +1 | [5,4,3] |

Final answer:

```
[[2,1,2,1,0]]
```

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(mn) | Every cell is pushed and popped at most once in row and column traversals |
| Space | O(m+n) | Maximum stack size is one row or one column |

The key observation is that monotonic stacks guarantee amortized constant work per element. Although a loop may pop multiple elements at once, each height can only be removed once overall. Therefore, total work across all operations remains linear.

## Edge Cases

### Single Cell Grid

A `1 x 1` grid contains no people to the right or below. This case is easy to mishandle if loops assume neighbors always exist.

The implementation naturally handles this because both traversal loops execute once, but the stack starts empty, so no visibility increments occur.

### Equal Heights

Equal heights are subtle because visibility requires all intermediate people to be strictly shorter than both endpoints.

For example:

```
[2,2,2]
```

The first person can only see the second person. The third is blocked because the middle person is not strictly shorter.

The algorithm handles this correctly because the stack only pops strictly smaller heights. Equal heights remain on the stack and act as blockers.

### Strictly Increasing Heights

Consider:

```
[1,2,3,4]
```

Each person only sees the next taller person. A naive implementation might incorrectly allow visibility through taller people.

The monotonic stack prevents this because once a taller person remains on the stack, visibility stops after counting that person.

### Strictly Decreasing Heights

Consider:

```
[4,3,2,1]
```

Each person sees only one adjacent neighbor. Even though everyone farther away is shorter, the closest person already blocks visibility because they are not shorter than both endpoints.

The stack logic models this naturally because the nearest person remains on the stack and blocks further visibility.
