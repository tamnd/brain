## Solution

## Problem Understanding

The problem asks us to count how many distinct hills and valleys exist in an integer array. A hill is a position whose closest different values on both sides are smaller than the current value. A valley is a position whose closest different values on both sides are larger than the current value.

The important detail is that consecutive equal values belong to the same hill or valley. For example, in the array `[1,1,1]`, all indices represent the same flat segment. If that segment forms a valley or hill relative to its surrounding values, it should only be counted once.

The phrase "closest non-equal neighbors" is the key part of the problem. We cannot simply compare `nums[i-1]`, `nums[i]`, and `nums[i+1]`, because equal adjacent values must be skipped. For example:

```
[2, 1, 1, 1, 3]
```

All three `1`s belong to the same valley because their closest non-equal neighbors are `2` and `3`.

The input is a small array with length between `3` and `100`, and values between `1` and `100`. Since the constraints are small, even less efficient approaches would work in practice. However, the goal is to design a clean and optimal solution that correctly handles duplicate values.

Several edge cases are important:

- Arrays with many consecutive duplicates.
- Strictly increasing or strictly decreasing arrays, which contain no hills or valleys.
- Hills or valleys formed by long flat regions.
- The first and last elements, which can never be hills or valleys because they do not have non-equal neighbors on both sides.

## Approaches

### Brute Force Approach

A straightforward approach is to examine every index individually. For each position `i`, we scan left until we find the closest value different from `nums[i]`, and we scan right until we find the closest different value on the other side.

Once both neighbors are found, we compare them with `nums[i]`:

- If both are smaller, the position belongs to a hill.
- If both are larger, the position belongs to a valley.

To avoid counting duplicate plateau regions multiple times, we must carefully ensure that consecutive equal elements are treated as one group.

This approach is correct because it directly follows the definition from the problem statement. However, each index may require scanning across multiple equal values, making the worst-case complexity `O(n^2)`.

### Optimal Approach

The key observation is that consecutive equal values do not affect hill or valley transitions. Instead of repeatedly skipping duplicates during comparisons, we can first compress the array into unique consecutive values.

For example:

```
[2,4,1,1,6,5]
```

becomes:

```
[2,4,1,6,5]
```

Now the problem becomes much simpler. A value is:

- A hill if it is greater than both neighbors.
- A valley if it is smaller than both neighbors.

After compression, each plateau appears exactly once, so counting becomes straightforward.

This reduces the complexity to linear time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Scans left and right for every index |
| Optimal | O(n) | O(n) | Compresses consecutive duplicates first |

## Algorithm Walkthrough

### Step 1: Remove consecutive duplicates

Create a new array that keeps only the first occurrence of each consecutive group.

For example:

```
[2,4,1,1,6,5]
```

becomes:

```
[2,4,1,6,5]
```

This step is important because adjacent equal values belong to the same hill or valley.

### Step 2: Iterate through the compressed array

Traverse the compressed array from index `1` to `len(compressed) - 2`.

We skip the first and last elements because they cannot have neighbors on both sides.

### Step 3: Check for hills

If:

```
compressed[i] > compressed[i-1]
and
compressed[i] > compressed[i+1]
```

then the current value is a hill.

Increment the answer.

### Step 4: Check for valleys

If:

```
compressed[i] < compressed[i-1]
and
compressed[i] < compressed[i+1]
```

then the current value is a valley.

Increment the answer.

### Step 5: Return the result

After processing all valid indices, return the total count.

### Why it works

Removing consecutive duplicates preserves all hill and valley relationships because equal adjacent values belong to the same feature. After compression, every remaining element represents exactly one distinct segment. A hill or valley can then be identified using only immediate neighbors, since there are no equal adjacent values left.

## Python Solution

```python
from typing import List

class Solution:
    def countHillValley(self, nums: List[int]) -> int:
        compressed = [nums[0]]

        for num in nums[1:]:
            if num != compressed[-1]:
                compressed.append(num)

        count = 0

        for i in range(1, len(compressed) - 1):
            if (
                compressed[i] > compressed[i - 1]
                and compressed[i] > compressed[i + 1]
            ):
                count += 1
            elif (
                compressed[i] < compressed[i - 1]
                and compressed[i] < compressed[i + 1]
            ):
                count += 1

        return count
```

The implementation begins by building a compressed version of the input array. The first element is always included, and every subsequent element is added only if it differs from the previous compressed value.

This compression step ensures that plateau regions appear only once.

Next, the algorithm scans the compressed array while ignoring the first and last positions. Each middle element is compared against its neighbors.

If the current value is larger than both neighbors, it forms a hill. If it is smaller than both neighbors, it forms a valley.

The final count is returned after the traversal completes.

## Go Solution

```go
func countHillValley(nums []int) int {
    compressed := []int{nums[0]}

    for i := 1; i < len(nums); i++ {
        if nums[i] != compressed[len(compressed)-1] {
            compressed = append(compressed, nums[i])
        }
    }

    count := 0

    for i := 1; i < len(compressed)-1; i++ {
        if compressed[i] > compressed[i-1] &&
            compressed[i] > compressed[i+1] {
            count++
        } else if compressed[i] < compressed[i-1] &&
            compressed[i] < compressed[i+1] {
            count++
        }
    }

    return count
}
```

The Go implementation follows the same logic as the Python solution. A slice named `compressed` stores the deduplicated consecutive values.

Go slices dynamically resize during `append`, making them suitable for this problem.

No special handling for integer overflow is needed because the values are very small.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | One pass for compression and one pass for counting |
| Space | O(n) | Compressed array may store all elements |

The algorithm performs two linear traversals. The first builds the compressed array, and the second counts hills and valleys. Since each element is processed a constant number of times, the total time complexity is linear.

The extra space comes from storing the compressed array. In the worst case, when all elements are distinct, the compressed array has the same size as the original input.

## Edge Cases

One important edge case is an array containing only equal values, such as:

```
[1,1,1,1]
```

A naive implementation that only compares adjacent elements might incorrectly identify hills or valleys. After compression, the array becomes `[1]`, which contains no valid middle elements, so the algorithm correctly returns `0`.

Another important case is a plateau hill or valley, such as:

```
[1,2,2,2,1]
```

All the `2`s belong to the same hill. Without compression, an implementation could accidentally count multiple hills. Compressing the array into `[1,2,1]` guarantees the hill is counted exactly once.

Strictly monotonic arrays are another common source of mistakes. For example:

```
[1,2,3,4,5]
```

contains no hills or valleys because every middle value has one neighbor larger and one neighbor smaller. The algorithm correctly avoids counting any positions because no value satisfies the hill or valley conditions.

Finally, very short valid arrays such as `[1,2,1]` or `[2,1,2]` must still work correctly. These are the smallest arrays capable of containing a hill or valley, and the implementation handles them naturally through the middle-element traversal logic.
