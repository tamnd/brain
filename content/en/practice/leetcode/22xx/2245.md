## Solution

## Problem Understanding

The problem gives us a two dimensional grid of positive integers. We must find a cornered path whose product contains the largest possible number of trailing zeros.

A trailing zero appears in a number when the number is divisible by 10. Since:

$10 = 2 \times 5$

every trailing zero requires one factor of 2 and one factor of 5. Therefore, for any product, the number of trailing zeros equals:

$\min(\text{count of }2,\ \text{count of }5)$

The input grid contains positive integers up to 1000, and the total number of cells satisfies:

$m \times n \le 10^5$

This constraint is extremely important. It tells us that an algorithm that examines every possible path cell by cell will be too slow. We need something close to linear in the number of cells.

A cornered path is a path with at most one turn. That means the path can look like:

- straight horizontal
- straight vertical
- an L shape

The path cannot revisit cells and cannot contain multiple turns.

For every possible turning point, we may extend:

- up + left
- up + right
- down + left
- down + right

The answer is the maximum number of trailing zeros among all such paths.

Several edge cases are important:

- A grid may contain no factors of 2 or 5 at all, producing answer 0.
- A single row or single column grid still forms valid paths.
- The turning cell belongs to both path segments, so it must not be counted twice.
- Numbers like 125 or 64 contribute multiple factors of 5 or 2 respectively, so simple divisibility checks are insufficient.

## Approaches

### Brute Force Approach

A brute force solution would enumerate every possible cornered path in the grid.

For each starting point, we could extend in one direction, optionally turn once, and continue in the perpendicular direction. For every path, we would compute the product's factor counts of 2 and 5, then determine the number of trailing zeros.

This approach is correct because it explicitly evaluates every valid path. However, it is far too slow.

A grid may contain up to $10^5$ cells. The number of possible L shaped paths grows very quickly because every cell can potentially connect to many row and column segments. Recomputing factor counts repeatedly would lead to excessive time complexity.

### Key Insight

The important observation is that trailing zeros depend only on counts of prime factors 2 and 5.

Instead of multiplying numbers directly, we can preprocess each cell:

- how many times 2 divides the number
- how many times 5 divides the number

Then every path simply needs the total counts of 2 and 5 along that path.

This immediately suggests prefix sums.

If we build prefix sums for rows and columns separately, we can compute the total number of 2s and 5s for any straight segment in constant time.

Then for every cell as the turning point, we evaluate the four possible L shapes in constant time.

Because each cell is processed once, the algorithm becomes efficient enough for the constraints.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(mn(m+n)) | O(1) | Enumerates all cornered paths directly |
| Optimal | O(mn) | O(mn) | Uses prefix sums of factors 2 and 5 |

## Algorithm Walkthrough

1. First, compute the number of factors of 2 and 5 for every cell.

For a value like 40:

$40 = 2^3 \times 5$

so it contributes:

- three factors of 2
- one factor of 5

Store these counts separately.
2. Build row prefix sums.

For each row, maintain cumulative counts of:

- factors of 2
- factors of 5

This allows us to retrieve the factor counts for any horizontal segment in constant time.
3. Build column prefix sums.

Similarly, maintain cumulative counts vertically for every column.

This allows constant time queries for vertical segments.
4. Treat every cell as a turning point.

Suppose the current cell is `(r, c)`.

There are four possible cornered paths:

- up + left
- up + right
- down + left
- down + right
5. Compute factor totals for each shape.

For example, for up + left:

- take vertical prefix from top to `(r,c)`
- take horizontal prefix from left to `(r,c)`
- subtract the turning cell once because it was included twice
6. Compute trailing zeros for the path.

The number of trailing zeros is:

$\min(\text{total twos},\ \text{total fives})$
7. Track the maximum across all cells and all four directions.

### Why it works

Every valid cornered path consists of exactly one vertical segment and one horizontal segment sharing a turning point. Prefix sums allow us to retrieve the factor counts for each segment exactly. By subtracting the turning cell once, we obtain the correct total factor counts for the full path. Since trailing zeros depend only on the minimum of total factors of 2 and 5, evaluating all four orientations at every cell guarantees we find the optimal answer.

## Python Solution

```python
from typing import List

class Solution:
    def maxTrailingZeros(self, grid: List[List[int]]) -> int:
        rows = len(grid)
        cols = len(grid[0])

        def count_factors(num: int, factor: int) -> int:
            count = 0
            while num % factor == 0:
                count += 1
                num //= factor
            return count

        twos = [[0] * cols for _ in range(rows)]
        fives = [[0] * cols for _ in range(rows)]

        for r in range(rows):
            for c in range(cols):
                twos[r][c] = count_factors(grid[r][c], 2)
                fives[r][c] = count_factors(grid[r][c], 5)

        row2 = [[0] * (cols + 1) for _ in range(rows)]
        row5 = [[0] * (cols + 1) for _ in range(rows)]

        for r in range(rows):
            for c in range(cols):
                row2[r][c + 1] = row2[r][c] + twos[r][c]
                row5[r][c + 1] = row5[r][c] + fives[r][c]

        col2 = [[0] * cols for _ in range(rows + 1)]
        col5 = [[0] * cols for _ in range(rows + 1)]

        for c in range(cols):
            for r in range(rows):
                col2[r + 1][c] = col2[r][c] + twos[r][c]
                col5[r + 1][c] = col5[r][c] + fives[r][c]

        answer = 0

        for r in range(rows):
            for c in range(cols):
                center2 = twos[r][c]
                center5 = fives[r][c]

                up2 = col2[r + 1][c]
                up5 = col5[r + 1][c]

                down2 = col2[rows][c] - col2[r][c]
                down5 = col5[rows][c] - col5[r][c]

                left2 = row2[r][c + 1]
                left5 = row5[r][c + 1]

                right2 = row2[r][cols] - row2[r][c]
                right5 = row5[r][cols] - row5[r][c]

                paths = [
                    (
                        up2 + left2 - center2,
                        up5 + left5 - center5
                    ),
                    (
                        up2 + right2 - center2,
                        up5 + right5 - center5
                    ),
                    (
                        down2 + left2 - center2,
                        down5 + left5 - center5
                    ),
                    (
                        down2 + right2 - center2,
                        down5 + right5 - center5
                    )
                ]

                for total2, total5 in paths:
                    answer = max(answer, min(total2, total5))

        return answer
```

The implementation begins by extracting the counts of prime factors 2 and 5 from every grid value. This preprocessing transforms the original multiplication problem into a counting problem.

Next, the code builds row and column prefix sums. Row prefix sums support horizontal segment queries, while column prefix sums support vertical segment queries.

For every cell, the implementation computes cumulative counts in all four directions:

- up
- down
- left
- right

Then it combines these counts into the four possible L shaped paths. Since the turning cell belongs to both segments, its contribution is subtracted once.

Finally, the algorithm computes the trailing zeros for each candidate path and updates the global maximum.

## Go Solution

```go
package main

func maxTrailingZeros(grid [][]int) int {
	rows := len(grid)
	cols := len(grid[0])

	countFactors := func(num int, factor int) int {
		count := 0
		for num%factor == 0 {
			count++
			num /= factor
		}
		return count
	}

	twos := make([][]int, rows)
	fives := make([][]int, rows)

	for r := 0; r < rows; r++ {
		twos[r] = make([]int, cols)
		fives[r] = make([]int, cols)

		for c := 0; c < cols; c++ {
			twos[r][c] = countFactors(grid[r][c], 2)
			fives[r][c] = countFactors(grid[r][c], 5)
		}
	}

	row2 := make([][]int, rows)
	row5 := make([][]int, rows)

	for r := 0; r < rows; r++ {
		row2[r] = make([]int, cols+1)
		row5[r] = make([]int, cols+1)

		for c := 0; c < cols; c++ {
			row2[r][c+1] = row2[r][c] + twos[r][c]
			row5[r][c+1] = row5[r][c] + fives[r][c]
		}
	}

	col2 := make([][]int, rows+1)
	col5 := make([][]int, rows+1)

	for r := 0; r <= rows; r++ {
		col2[r] = make([]int, cols)
		col5[r] = make([]int, cols)
	}

	for c := 0; c < cols; c++ {
		for r := 0; r < rows; r++ {
			col2[r+1][c] = col2[r][c] + twos[r][c]
			col5[r+1][c] = col5[r][c] + fives[r][c]
		}
	}

	answer := 0

	max := func(a, b int) int {
		if a > b {
			return a
		}
		return b
	}

	min := func(a, b int) int {
		if a < b {
			return a
		}
		return b
	}

	for r := 0; r < rows; r++ {
		for c := 0; c < cols; c++ {
			center2 := twos[r][c]
			center5 := fives[r][c]

			up2 := col2[r+1][c]
			up5 := col5[r+1][c]

			down2 := col2[rows][c] - col2[r][c]
			down5 := col5[rows][c] - col5[r][c]

			left2 := row2[r][c+1]
			left5 := row5[r][c+1]

			right2 := row2[r][cols] - row2[r][c]
			right5 := row5[r][cols] - row5[r][c]

			paths := [][2]int{
				{up2 + left2 - center2, up5 + left5 - center5},
				{up2 + right2 - center2, up5 + right5 - center5},
				{down2 + left2 - center2, down5 + left5 - center5},
				{down2 + right2 - center2, down5 + right5 - center5},
			}

			for _, path := range paths {
				answer = max(answer, min(path[0], path[1]))
			}
		}
	}

	return answer
}
```

The Go implementation follows the same algorithmic structure as the Python version. The main differences come from Go's explicit slice allocation and helper functions for `min` and `max`.

Go integers are sufficient because we never compute actual products. Instead, we only count factors of 2 and 5, which remain small.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(mn) | Each cell is processed a constant number of times |
| Space | O(mn) | Prefix sum arrays store information for every cell |

The factor counting phase processes every cell once. Building prefix sums also requires linear work. Finally, every cell evaluates exactly four candidate paths, which is still constant work per cell. Therefore the total running time remains linear in the number of grid cells.

The space complexity comes from storing:

- factor count matrices
- row prefix sums
- column prefix sums

All of these structures are proportional to the size of the grid.

## Edge Cases

A very important edge case is a single row or single column grid. In these cases, there may be no actual turn in the path. The problem allows paths with at most one turn, so straight paths are valid. The implementation naturally handles this because one of the directional segments may contain only the turning cell itself.

Another important edge case occurs when the grid contains many factors of 2 but no factors of 5, or vice versa. Since trailing zeros require matched pairs of 2 and 5, the result must be zero. A naive implementation that only tracks divisibility by 10 could fail here. This solution correctly computes the minimum of total factor counts.

The turning cell is another common source of bugs. When combining a horizontal and vertical segment, the turning cell belongs to both segments. If we simply add the prefix sums together, the turning cell would be counted twice. The implementation explicitly subtracts the center cell's factor counts once to avoid this error.

Finally, values may contain repeated prime factors. For example:

$125 = 5^3$

A naive implementation checking only divisibility by 5 would miss extra trailing zero contributions. The helper function repeatedly divides by the factor, ensuring all multiplicities are counted correctly.
