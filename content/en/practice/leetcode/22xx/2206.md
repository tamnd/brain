## Solution

## Problem Understanding

The problem gives an array `nums` containing exactly `2 * n` integers. Our goal is to determine whether the array can be split into `n` pairs where both elements inside every pair are equal.

In simpler terms, every number in the array must appear an even number of times. If a value appears 2 times, we can form one pair. If it appears 4 times, we can form two pairs. If it appears 6 times, we can form three pairs, and so on. However, if any value appears an odd number of times, one occurrence will always remain unmatched, making it impossible to divide the array completely into equal pairs.

For example, consider:

```
[3,2,3,2,2,2]
```

The number `3` appears twice, and the number `2` appears four times. Both counts are even, so the array can be partitioned into equal pairs.

Now consider:

```
[1,2,3,4]
```

Every value appears exactly once. Since each occurrence count is odd, there is no way to create pairs of equal elements.

The constraints are small:

- `1 <= n <= 500`
- `nums.length == 2 * n`
- `1 <= nums[i] <= 500`

This means the maximum array size is only `1000`, so even relatively inefficient solutions could pass. However, the problem is fundamentally a counting problem, and there is a very direct optimal solution.

One important observation is that the input length is always even. Therefore, we never need to handle arrays with odd length. Another key guarantee is that all values are positive integers between `1` and `500`, which means a frequency array is also possible. Still, using a hash map or dictionary is cleaner and more general.

Important edge cases include arrays where:

- Every element is already perfectly paired.
- Exactly one number has an odd frequency.
- Only one distinct number exists.
- All numbers are unique.
- A number appears many times.

These cases are where incorrect pairing logic or incomplete counting implementations often fail.

## Approaches

### Brute Force Approach

A brute force solution would repeatedly search for a matching element for every unpaired number.

The algorithm could work like this:

1. Take the first unused element.
2. Scan the remaining array to find another equal unused element.
3. Mark both as used.
4. Continue until all elements are processed or a match cannot be found.

This approach is correct because it explicitly constructs valid pairs whenever possible. If any element cannot find a matching partner, then the answer must be `false`.

However, this method is inefficient because every element may require scanning a large portion of the array. In the worst case, this leads to quadratic time complexity.

### Optimal Approach

The key observation is that pairing is only possible when every distinct number appears an even number of times.

Instead of actually constructing pairs, we only need to count frequencies.

We can use a hash map where:

- The key is the number.
- The value is the number of times it appears.

After building the frequency table, we iterate through all counts:

- If every count is even, return `true`.
- If any count is odd, return `false`.

This works because equal pairs consume elements two at a time. An even frequency can always be fully partitioned into pairs, while an odd frequency always leaves one leftover element.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeatedly searches for matching unused elements |
| Optimal | O(n) | O(n) | Counts frequencies and checks parity |

## Algorithm Walkthrough

1. Create an empty hash map called `frequency`.

This map stores how many times each number appears in the array. A hash map is ideal because it provides average O(1) insertion and lookup time.
2. Iterate through every number in `nums`.

For each value:

- If the value is not already in the map, initialize its count.
- Increment its frequency count by one.

After this step, the map contains the exact occurrence count for every distinct number.
3. Iterate through all frequency counts in the map.

For each count:

- Check whether the count is odd using `count % 2 != 0`.
- If an odd count is found, immediately return `false`.

An odd count means one element cannot be paired.
4. If all counts are even, return `true`.

This guarantees that every occurrence can be grouped into equal pairs.

### Why it works

The algorithm relies on a simple invariant:

A value can be fully partitioned into equal pairs if and only if its frequency is even.

Since all elements must belong to exactly one pair, every distinct number must contribute an even number of elements. The algorithm verifies this condition directly for every number, so it correctly determines whether the array can be divided into equal pairs.

## Python Solution

```python
from collections import Counter
from typing import List

class Solution:
    def divideArray(self, nums: List[int]) -> bool:
        frequency = Counter(nums)

        for count in frequency.values():
            if count % 2 != 0:
                return False

        return True
```

The implementation begins by using `Counter` from Python's `collections` module. This automatically builds a frequency table for the array.

For example:

```python
Counter([3, 2, 3, 2, 2, 2])
```

produces:

```python
{3: 2, 2: 4}
```

Next, the code iterates through all frequency values. The expression:

```python
count % 2 != 0
```

checks whether the frequency is odd. If any odd frequency exists, the function immediately returns `False` because one element would remain unmatched.

If the loop finishes without finding an odd count, every number appears an even number of times, so the function returns `True`.

The implementation is concise because the problem reduces entirely to frequency parity checking.

## Go Solution

```go
func divideArray(nums []int) bool {
    frequency := make(map[int]int)

    for _, num := range nums {
        frequency[num]++
    }

    for _, count := range frequency {
        if count%2 != 0 {
            return false
        }
    }

    return true
}
```

The Go solution follows the same logic as the Python version.

A `map[int]int` is used to store frequencies. As the array is traversed, each number's count is incremented.

Go maps automatically return the zero value for missing keys, so this line works safely:

```go
frequency[num]++
```

Unlike Python, there is no built-in `Counter` type in the standard library, so the counting logic is written manually.

There are no integer overflow concerns because the maximum frequency is at most `1000`, which easily fits inside Go's `int` type.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | One pass to count frequencies and one pass to check them |
| Space | O(n) | Hash map may store up to all distinct numbers |

The algorithm performs two linear scans. The first builds the frequency table, and the second checks parity. Since hash map operations are O(1) on average, the total runtime is linear relative to the array size.

The extra memory usage comes from storing frequencies for distinct values. In the worst case, every number is unique, so the map size becomes O(n).

## Edge Cases

One important edge case is when every number is unique. For example:

```
[1,2,3,4]
```

Every frequency equals `1`, which is odd. A naive implementation that only checks whether duplicates exist somewhere in the array could incorrectly return `true`. The frequency-based approach avoids this by verifying parity for every distinct value.

Another important case is when only one distinct number exists, such as:

```
[5,5,5,5,5,5]
```

Some implementations mistakenly assume pairs must involve different indices or different values. In reality, repeated identical values are perfectly valid as long as the count is even. The algorithm correctly returns `true` because the frequency `6` is divisible by `2`.

A third critical edge case occurs when almost all counts are even except one:

```
[1,1,2,2,3]
```

The value `3` appears once, leaving one unmatched element. This is a common source of bugs when implementations attempt greedy pairing without complete counting. The frequency check guarantees correctness because every value must independently satisfy the even-count condition.

Another subtle case is the minimum input size:

```
[1,1]
```

The implementation handles this naturally. The frequency map contains one entry with count `2`, which is even, so the function correctly returns `true`.
