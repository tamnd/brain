## Solution

## Problem Understanding

The problem asks us to transform a given array `nums` into a **non-decreasing** array through repeated steps. Each step removes elements that are smaller than their immediate predecessor. More formally, for each index `i` (where `0 < i < nums.length`), if `nums[i - 1] > nums[i]`, `nums[i]` is removed in that step. The task is to count how many steps are required until no such removals are needed, i.e., the array is non-decreasing.

The input is a list of integers `nums` with length up to `10^5`, and each integer can be as large as `10^9`. This tells us that a brute-force simulation of all steps may be too slow for large arrays, since repeated scans of `nums` can lead to `O(n^2)` complexity. The output is a single integer representing the number of steps needed.

Edge cases to note include arrays that are already non-decreasing, arrays with all equal elements, and arrays that strictly decrease, which might require many steps. The input guarantees at least one element, so empty arrays are not a concern.

## Approaches

The **brute-force approach** directly simulates each step: iterate over the array, identify elements where `nums[i - 1] > nums[i]`, remove them, and count the steps until the array becomes non-decreasing. This works because it directly implements the problem description, but it is inefficient. Each step may remove very few elements, and scanning the array repeatedly can lead to `O(n^2)` worst-case time complexity.

The **optimal approach** uses a **monotonic stack** to track elements and the number of steps each element would take to be removed. The key insight is that an element’s removal depends on the largest number of steps any preceding smaller element takes. By iterating from left to right and maintaining a stack of elements with their associated "removal steps," we can compute the total number of steps in **one pass**.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^2) | O(n) | Simulate each step, removing elements that break non-decreasing order |
| Optimal | O(n) | O(n) | Use monotonic stack to track removal steps efficiently |

## Algorithm Walkthrough

1. Initialize a stack to hold pairs `(value, steps)`, where `value` is the array element and `steps` is the number of steps until it would be removed.
2. Initialize a variable `max_steps` to track the maximum steps needed across all elements.
3. Iterate through the array from left to right. For each element `num`:

1. Set `current_steps` to 0.
2. While the stack is not empty and the top element of the stack has a value less than or equal to `num`:

1. Pop the element and update `current_steps` to `max(current_steps, popped_element_steps)`.
3. If the stack is not empty after popping, increment `current_steps` by 1 because `num` is still smaller than its predecessor and will be removed in the next step.
4. Push `(num, current_steps)` onto the stack.
5. Update `max_steps` to `max(max_steps, current_steps)`.
4. Return `max_steps`.

**Why it works:** The stack maintains a non-increasing sequence of values from the left, with the number of steps needed for each element’s removal. Each pop propagates the removal steps, ensuring that the maximum number of steps required is tracked correctly. This guarantees that the algorithm calculates the exact number of steps until the array is non-decreasing.

## Python Solution

```python
from typing import List

class Solution:
    def totalSteps(self, nums: List[int]) -> int:
        stack = []
        max_steps = 0
        
        for num in nums:
            current_steps = 0
            while stack and stack[-1][0] <= num:
                current_steps = max(current_steps, stack.pop()[1])
            if stack:
                current_steps += 1
            stack.append((num, current_steps))
            max_steps = max(max_steps, current_steps)
        
        return max_steps
```

In this Python implementation, we maintain a stack of tuples `(value, steps)`. Each element's removal step is calculated based on the stack elements to its left. The while loop propagates the maximum steps needed from the elements removed, and if the stack still has elements after popping, it means the current element would be removed one step later. Finally, `max_steps` records the total number of removal steps.

## Go Solution

```go
func totalSteps(nums []int) int {
    type pair struct {
        val, steps int
    }
    
    stack := []pair{}
    maxSteps := 0
    
    for _, num := range nums {
        currentSteps := 0
        for len(stack) > 0 && stack[len(stack)-1].val <= num {
            currentSteps = max(currentSteps, stack[len(stack)-1].steps)
            stack = stack[:len(stack)-1]
        }
        if len(stack) > 0 {
            currentSteps++
        }
        stack = append(stack, pair{num, currentSteps})
        if currentSteps > maxSteps {
            maxSteps = currentSteps
        }
    }
    
    return maxSteps
}

func max(a, b int) int {
    if a > b {
        return a
    }
    return b
}
```

The Go implementation mirrors the Python logic, using a struct `pair` to store the value and steps. Slices are used as a stack, with careful handling of slice bounds for popping. The helper function `max` simplifies comparisons.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each element is pushed and popped at most once on the stack, giving linear time |
| Space | O(n) | Stack can hold all elements in the worst case |

The linear time is guaranteed by the monotonic stack, as each element is processed once, and all stack operations are constant time. The space complexity is proportional to the array length because the stack may store all elements in strictly decreasing sequences.
