## Solution

## Problem Understanding

The problem gives us an integer array `nums` and asks us to find the number whose value is closest to `0`. In other words, we want the element with the smallest absolute value.

The absolute value of a number measures its distance from zero. For example:

- `|-5| = 5`
- `|3| = 3`
- `|0| = 0`

The challenge comes from the tie breaking rule. If two numbers are equally close to zero, we must return the larger number. This means:

- Between `-1` and `1`, we return `1`
- Between `-5` and `5`, we return `5`

The input array contains between `1` and `1000` integers, and each integer can range from `-100000` to `100000`. These constraints are small enough that even a simple linear scan over the array is completely efficient.

The array is guaranteed to contain at least one element, so we never need to handle an empty input. Important edge cases include:

- Arrays containing both positive and negative numbers with equal absolute values
- Arrays containing only negative numbers
- Arrays containing only positive numbers
- Arrays containing `0`
- Arrays with a single element

A naive implementation might forget the tie breaking rule and incorrectly return the negative number when two values are equally close to zero.

## Approaches

### Brute Force Approach

The most direct approach is to compare every number against every other number to determine which one is closest to zero.

For each element, we could compute its absolute value and check whether any other element has a smaller absolute value. If there is a tie, we choose the larger number.

This approach works because eventually every pair of numbers is compared, guaranteeing that the best candidate is found. However, it performs unnecessary repeated comparisons. With `n` elements, comparing every element against every other element requires `O(n^2)` time.

Although the constraints are small enough that this would still pass, it is not efficient or elegant.

### Optimal Approach

The key observation is that we only need to keep track of the current best answer while scanning the array once.

As we iterate through the array:

- If the current number has a smaller absolute value than the best number so far, we update the answer.
- If the absolute values are equal, we choose the larger number.

This works because the decision about whether a number is better depends only on the current best candidate, not on the rest of the array.

Since we examine each element exactly once, the algorithm runs in linear time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Compares every pair of elements unnecessarily |
| Optimal | O(n) | O(1) | Single pass while tracking the best candidate |

## Algorithm Walkthrough

1. Initialize the answer using the first element of the array.

This guarantees that we always have a valid candidate to compare against. Since the array is non empty, this is safe.
2. Iterate through every number in the array.

For each number, compute its absolute value and compare it with the absolute value of the current best answer.
3. If the current number is closer to zero, update the answer.

This happens when:

```
abs(current) < abs(answer)
```

A smaller absolute value means the number is closer to zero.
4. If both numbers are equally close to zero, apply the tie breaking rule.

This happens when:

```
abs(current) == abs(answer)
```

In this case, choose the larger number:

```
current > answer
```

This ensures that positive numbers are preferred over their negative counterparts.
5. Continue until all elements are processed.
6. Return the final answer.

### Why it works

The algorithm maintains the invariant that `answer` is always the closest number to zero among all elements processed so far.

Whenever a new number is examined:

- If it is strictly closer to zero, it becomes the new best answer.
- If it is equally close but numerically larger, it replaces the previous answer.

By the end of the iteration, every element has been considered exactly once, so the stored answer must be the correct result.

## Python Solution

```python
from typing import List

class Solution:
    def findClosestNumber(self, nums: List[int]) -> int:
        closest = nums[0]

        for num in nums:
            if abs(num) < abs(closest):
                closest = num
            elif abs(num) == abs(closest) and num > closest:
                closest = num

        return closest
```

The implementation starts by assigning the first array element to `closest`. This gives us an initial candidate for the answer.

The loop then examines every number in the array. For each number:

- `abs(num) < abs(closest)` checks whether the current number is closer to zero.
- `abs(num) == abs(closest) and num > closest` handles the tie breaking rule.

If either condition is true, `closest` is updated.

After the loop finishes, `closest` contains the correct answer and is returned.

## Go Solution

```go
func findClosestNumber(nums []int) int {
    closest := nums[0]

    for _, num := range nums {
        if abs(num) < abs(closest) {
            closest = num
        } else if abs(num) == abs(closest) && num > closest {
            closest = num
        }
    }

    return closest
}

func abs(x int) int {
    if x < 0 {
        return -x
    }
    return x
}
```

The Go implementation follows the same logic as the Python version. Since Go does not provide a built in integer `abs` function for plain integers, we define a helper function manually.

The solution uses a range based loop to iterate through the slice efficiently. No additional memory is allocated beyond a few integer variables, so the space complexity remains constant.

## Edge Cases

One important edge case occurs when two numbers have the same absolute value, such as `-1` and `1`. A buggy implementation might return whichever number appears first in the array. The problem explicitly requires returning the larger value, so the algorithm includes an equality check and compares the actual numeric values to ensure the positive number is selected.

Another important case is when the array contains only negative numbers. In this scenario, the answer is the negative number with the smallest absolute value. The implementation handles this naturally because it compares absolute values directly rather than relying on positivity.

A third edge case is when the array contains `0`. Since `0` has absolute value `0`, it is automatically the closest possible number to zero. Once the algorithm encounters `0`, no other value can beat it. The implementation correctly keeps `0` as the answer.

A final edge case involves arrays with only one element. Since the array is guaranteed to be non empty, initializing `closest` with the first element is always safe. The loop still works correctly and simply returns that element.
