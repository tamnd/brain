# LeetCode 2219 - Maximum Sum Score of Array

## Problem Understanding

The problem gives us a 0-indexed integer array `nums` and asks us to compute the maximum possible "sum score" across all indices.

For a given index `i`, the sum score is defined as the larger of these two values:

1. The sum of the prefix ending at index `i`
2. The sum of the suffix starting at index `i`

In other words:

- Prefix sum at `i` = `nums[0] + nums[1] + ... + nums[i]`
- Suffix sum at `i` = `nums[i] + nums[i+1] + ... + nums[n-1]`

The score at index `i` is:

$$\max(\text{prefixSum}(i), \text{suffixSum}(i))$$

We must compute this score for every index and return the maximum among them.

The input array can contain both positive and negative numbers. This detail is important because sums do not necessarily increase as we move through the array. A larger prefix does not guarantee a larger suffix, and vice versa.

The constraints tell us that:

- `n` can be as large as `10^5`
- Each element can range from `-10^5` to `10^5`

An `O(n^2)` solution would be far too slow because recomputing prefix and suffix sums repeatedly would require billions of operations in the worst case. This strongly suggests that we need a linear-time solution using prefix sum techniques.

Several edge cases are important to recognize early:

- Arrays containing only negative numbers
- Arrays of length `1`
- Arrays where the best score comes from a suffix rather than a prefix
- Arrays where the total sum itself is the answer
- Arrays with alternating positive and negative values

The problem guarantees that the array contains at least one element, so we never need to handle an empty array.

## Approaches

### Brute Force Approach

The most direct solution is to evaluate every index independently.

For each index `i`, we can:

1. Compute the prefix sum from index `0` to `i`
2. Compute the suffix sum from index `i` to `n - 1`
3. Take the maximum of those two values
4. Track the largest score overall

This approach is correct because it explicitly follows the problem definition.

However, the inefficiency comes from repeatedly recomputing sums. For every index, we may scan a large portion of the array again. Since there are `n` indices and each computation may take `O(n)` time, the total complexity becomes `O(n^2)`.

With `n = 100000`, this approach is too slow.

### Optimal Prefix Sum Approach

The key observation is that prefix and suffix sums overlap heavily between indices.

Instead of recomputing sums repeatedly, we can compute them incrementally.

Let:

- `totalSum` be the sum of the entire array
- `prefixSum` be the running prefix sum while iterating

At index `i`:

$$\text{prefixSum} = nums[0] + \dots + nums[i]$$

The suffix sum can then be derived without another loop:

$$\text{suffixSum} = totalSum - prefixSum + nums[i]$$

Why does this formula work?

- `totalSum - prefixSum` removes everything up to index `i`
- This accidentally removes `nums[i]` as well
- Adding `nums[i]` back gives the suffix starting at `i`

This allows us to compute both values in constant time per index.

Since we process the array only once after computing the total sum, the final complexity becomes `O(n)`.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Recomputes prefix and suffix sums for every index |
| Optimal | O(n) | O(1) | Uses running prefix sum and total sum |

## Algorithm Walkthrough

1. Compute the total sum of the array.

This value allows us to derive suffix sums without traversing the array repeatedly.
2. Initialize a variable `prefix_sum` to `0`.

This variable will maintain the running sum from the beginning of the array up to the current index.
3. Initialize `best_score` to negative infinity.

We will update this whenever we find a larger sum score.
4. Iterate through the array from left to right.

At each index:

- Add the current value to `prefix_sum`
- Compute the suffix sum using:

$$\text{suffixSum} = totalSum - prefixSum + nums[i]$$
- Compute the current score:

$$\max(prefixSum, suffixSum)$$
- Update `best_score` if this score is larger
5. After processing all indices, return `best_score`.

### Why it works

The algorithm works because every valid score depends only on two quantities:

- Prefix sum ending at `i`
- Suffix sum starting at `i`

The running prefix sum always correctly represents the first quantity. The suffix formula derives the second quantity directly from the total sum. Since every index is evaluated exactly once and we keep the maximum score encountered, the final result is guaranteed to be correct.

## Python Solution

```python
from typing import List

class Solution:
    def maximumSumScore(self, nums: List[int]) -> int:
        total_sum = sum(nums)

        prefix_sum = 0
        best_score = float("-inf")

        for num in nums:
            prefix_sum += num

            suffix_sum = total_sum - prefix_sum + num

            current_score = max(prefix_sum, suffix_sum)

            best_score = max(best_score, current_score)

        return best_score
```

The implementation begins by computing `total_sum`, which is used later to derive suffix sums efficiently.

The variable `prefix_sum` tracks the running prefix while iterating through the array. Each iteration updates this running total with the current element.

The suffix sum is then computed using the derived formula:

$$\text{suffixSum} = totalSum - prefixSum + num$$

This avoids constructing a separate suffix array or performing additional loops.

For every position, the algorithm computes the larger of the prefix and suffix sums, then updates `best_score` if needed.

Because the solution performs only a single traversal after computing the total sum, it runs in linear time and constant extra space.

## Go Solution

```go
func maximumSumScore(nums []int) int64 {
    var totalSum int64 = 0

    for _, num := range nums {
        totalSum += int64(num)
    }

    var prefixSum int64 = 0
    var bestScore int64 = -(1 << 60)

    for _, num := range nums {
        value := int64(num)

        prefixSum += value

        suffixSum := totalSum - prefixSum + value

        currentScore := prefixSum
        if suffixSum > currentScore {
            currentScore = suffixSum
        }

        if currentScore > bestScore {
            bestScore = currentScore
        }
    }

    return bestScore
}
```

The Go implementation follows the exact same algorithmic logic as the Python version.

One important difference is integer handling. Since sums may exceed the range of a 32-bit integer, the implementation uses `int64` throughout the computation. Every array element is explicitly converted before arithmetic operations.

Go does not provide built-in `max` functions for integers, so comparisons are performed manually using `if` statements.

The solution uses only a few scalar variables, so the extra space complexity remains constant.

## Edge Cases

### Single Element Array

When the array contains only one element, the prefix sum and suffix sum are identical because both cover the entire array.

For example:

```
nums = [5]
```

Both prefix and suffix sums equal `5`, so the answer is `5`.

The implementation handles this naturally because the loop processes the single element exactly once.

### All Negative Numbers

Arrays containing only negative values can expose bugs in implementations that incorrectly initialize the answer to `0`.

For example:

```
nums = [-3, -5]
```

The correct answer is `-3`, not `0`.

The implementation avoids this issue by initializing `best_score` to negative infinity, ensuring negative results are handled correctly.

### Large Positive and Negative Values

The constraints allow values up to `100000` in magnitude, and the array length can reach `100000`. This means cumulative sums can become very large.

For example:

```
nums = [100000, 100000, ..., 100000]
```

The total sum may reach `10^10`, which exceeds 32-bit integer limits.

The Go implementation uses `int64` to safely store these sums. Python integers automatically support arbitrary precision, so no special handling is required there.
