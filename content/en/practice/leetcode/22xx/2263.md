# LeetCode 2263 - Make Array Non-decreasing or Non-increasing

## Problem Understanding

The problem gives us an integer array `nums`, and we are allowed to perform operations that increment or decrement any element by `1`. Each increment or decrement counts as a single operation.

Our goal is to transform the array into either:

- A non-decreasing array, where `nums[i] <= nums[i+1]`
- A non-increasing array, where `nums[i] >= nums[i+1]`

We may choose whichever direction requires fewer operations.

The important detail is that every change has a linear cost. If we change a value from `7` to `3`, that costs `4` operations. Similarly, changing `2` to `5` costs `3` operations.

So the real problem becomes:

> Find another array that is monotonic and minimizes the total absolute difference from the original array.

The constraints are relatively small:

- `nums.length <= 1000`
- `nums[i] <= 1000`

These constraints strongly suggest that a dynamic programming solution over possible values is feasible. Since values are bounded between `0` and `1000`, we can consider every possible target value during DP transitions.

An important observation is that making the array non-increasing is equivalent to reversing the array and making it non-decreasing. This allows us to reuse the same logic for both cases.

Several edge cases are important:

- Arrays of length `1` are already monotonic.
- Arrays already sorted in either direction should return `0`.
- Arrays with repeated values must still be handled correctly.
- Large oscillations like `[1,1000,1,1000]` can trick greedy approaches.
- The optimal transformed values do not necessarily appear in the original array.

## Approaches

### Brute Force Approach

A brute force solution would attempt to generate all possible monotonic arrays and compute the total cost required to transform `nums` into each one.

For a non-decreasing array, every position could potentially take values from `0` to `1000`, subject to monotonic constraints. Even with pruning, the number of possible arrays grows exponentially.

For example, if we tried every sequence:

- `a[0] <= a[1] <= ... <= a[n-1]`
- each `a[i]` in `[0,1000]`

the state space becomes enormous.

Although this approach would eventually find the optimal answer, it is computationally infeasible.

### Optimal Dynamic Programming Approach

The key insight is that the values are bounded.

Instead of constructing all monotonic arrays explicitly, we use dynamic programming.

Define:

- `dp[v]` = minimum cost to process elements so far where the current transformed value is exactly `v`

For a non-decreasing array:

- if current value becomes `v`
- previous transformed value must be `<= v`

So while processing each number, we maintain prefix minimums to efficiently compute transitions.

The transition becomes:

```
new_dp[v] = min(previous_dp[0...v]) + abs(nums[i] - v)
```

Using prefix minimums reduces the transition complexity from `O(1000^2)` to `O(1000)` per element.

To compute the non-increasing case, we reverse the array and run the same procedure again.

Finally, we return the minimum of the two results.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Tries all monotonic arrays |
| Optimal Dynamic Programming | O(n * V) | O(V) | Uses prefix minimum optimization, where `V = 1001` |

## Algorithm Walkthrough

### Step 1: Define the DP Meaning

We create a DP array where:

```
dp[v]
```

represents the minimum cost after processing some prefix of the array, with the last transformed value equal to `v`.

Since `nums[i] <= 1000`, we only need values from `0` to `1000`.

### Step 2: Initialize the First Element

For the first number:

```
dp[v] = abs(nums[0] - v)
```

because transforming the first element to `v` costs exactly that many operations.

### Step 3: Process Each Remaining Element

For every next number:

- we compute prefix minimums
- `prefix_min[v]` stores the minimum DP value among all indices `<= v`

This is necessary because for a non-decreasing array:

```
previous_value <= current_value
```

So the transition becomes:

```
new_dp[v] = prefix_min[v] + abs(nums[i] - v)
```

### Step 4: Replace the DP Array

After computing all `new_dp[v]`, we assign:

```
dp = new_dp
```

and continue.

### Step 5: Compute the Final Answer

After processing the full array:

```
min(dp)
```

is the minimum cost for making the array non-decreasing.

### Step 6: Handle the Non-increasing Case

A non-increasing array becomes non-decreasing after reversing.

So we:

1. Run DP on the original array
2. Reverse the array
3. Run DP again
4. Return the minimum result

### Why it works

The DP maintains the invariant that every state already satisfies the monotonic condition up to the current position.

For each possible transformed value `v`, we only transition from valid previous values:

- `<= v` for non-decreasing order

The prefix minimum ensures we always choose the cheapest valid predecessor.

Since every possible transformed value is considered, and every valid transition is evaluated, the algorithm guarantees the globally minimum total cost.

## Python Solution

```python
from typing import List

class Solution:
    def convertArray(self, nums: List[int]) -> int:
        MAX_VALUE = 1000

        def min_cost(arr: List[int]) -> int:
            dp = [0] * (MAX_VALUE + 1)

            for value in range(MAX_VALUE + 1):
                dp[value] = abs(arr[0] - value)

            for num in arr[1:]:
                prefix_min = [0] * (MAX_VALUE + 1)

                prefix_min[0] = dp[0]
                for value in range(1, MAX_VALUE + 1):
                    prefix_min[value] = min(prefix_min[value - 1], dp[value])

                new_dp = [0] * (MAX_VALUE + 1)

                for value in range(MAX_VALUE + 1):
                    new_dp[value] = prefix_min[value] + abs(num - value)

                dp = new_dp

            return min(dp)

        non_decreasing = min_cost(nums)
        non_increasing = min_cost(nums[::-1])

        return min(non_decreasing, non_increasing)
```

The implementation follows the dynamic programming strategy directly.

The helper function `min_cost()` computes the minimum operations required to make an array non-decreasing.

The `dp` array stores the minimum cost for every possible ending value. Initially, each value is simply the cost of converting the first element.

For every subsequent number:

- `prefix_min` is constructed to efficiently answer:

"What is the cheapest previous state with value less than or equal to the current candidate value?"

Without prefix minimums, each transition would require scanning all smaller values, leading to `O(V^2)` work per element.

After processing all positions, the minimum value in `dp` is the answer.

To compute the non-increasing case, the array is reversed and the same function is reused.

## Go Solution

```go
package main

import "math"

func convertArray(nums []int) int {
	const MAX_VALUE = 1000

	minCost := func(arr []int) int {
		dp := make([]int, MAX_VALUE+1)

		for value := 0; value <= MAX_VALUE; value++ {
			dp[value] = abs(arr[0] - value)
		}

		for i := 1; i < len(arr); i++ {
			prefixMin := make([]int, MAX_VALUE+1)

			prefixMin[0] = dp[0]
			for value := 1; value <= MAX_VALUE; value++ {
				prefixMin[value] = min(prefixMin[value-1], dp[value])
			}

			newDP := make([]int, MAX_VALUE+1)

			for value := 0; value <= MAX_VALUE; value++ {
				newDP[value] = prefixMin[value] + abs(arr[i]-value)
			}

			dp = newDP
		}

		answer := math.MaxInt32

		for _, cost := range dp {
			answer = min(answer, cost)
		}

		return answer
	}

	reversed := make([]int, len(nums))
	for i := 0; i < len(nums); i++ {
		reversed[i] = nums[len(nums)-1-i]
	}

	nonDecreasing := minCost(nums)
	nonIncreasing := minCost(reversed)

	return min(nonDecreasing, nonIncreasing)
}

func abs(x int) int {
	if x < 0 {
		return -x
	}
	return x
}

func min(a, b int) int {
	if a < b {
		return a
	}
	return b
}
```

The Go implementation mirrors the Python logic closely.

Since Go does not provide built in `min` or `abs` functions for integers, helper functions are implemented explicitly.

The reversed array is constructed manually because slices are views rather than automatically reversible collections.

Integer overflow is not a concern because the maximum total cost is bounded by:

```go
1000 elements * 1000 change per element = 1,000,000
```

which easily fits inside Go's `int`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n * V) | For each element, we process all values from `0` to `1000` |
| Space | O(V) | Only DP arrays of size `1001` are stored |

Here:

```
V = 1001
```

because values range from `0` to `1000`.

The prefix minimum optimization is critical. Without it, transitions would require scanning all previous values, producing `O(n * V^2)` complexity.

Instead, each DP layer is computed in linear time relative to `V`.

## Edge Cases

### Single Element Arrays

An array with one element is automatically both non-decreasing and non-increasing.

Naive implementations sometimes assume at least one comparison exists between neighboring elements, which can produce index errors. This implementation handles the case naturally because the DP initializes once and never enters the transition loop.

### Arrays Already Monotonic

Arrays like:

```
[1,2,3,4]
```

or:

```
[5,4,4,2]
```

should return `0`.

Some incorrect solutions perform unnecessary modifications because they fail to preserve already optimal states. The DP always includes the option of keeping a value unchanged, so zero-cost solutions remain available throughout computation.

### Large Oscillations

Arrays such as:

```
[1,1000,1,1000]
```

can break greedy strategies.

A local decision that seems optimal may create expensive future adjustments. Dynamic programming avoids this issue by evaluating every possible transformed value at each position and preserving globally optimal states.

### Duplicate Values

Repeated numbers are important because monotonicity allows equality.

Some implementations mistakenly enforce strictly increasing or strictly decreasing order. This solution correctly allows equal neighboring values because transitions use:

```
previous <= current
```

rather than strict inequality.
