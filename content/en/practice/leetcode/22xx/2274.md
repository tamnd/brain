# LeetCode 2274 - Maximum Consecutive Floors Without Special Floors

## Problem Understanding

The problem describes a building where Alice has rented every floor from `bottom` to `top`, inclusive. Among these rented floors, some floors are marked as "special floors" and cannot be counted as part of a normal working sequence.

We are asked to find the largest number of consecutive floors that do not contain any special floor.

The input consists of:

- `bottom`, the lowest rented floor
- `top`, the highest rented floor
- `special`, an array containing all special floors

Every floor in `special` is guaranteed to lie between `bottom` and `top`, and all values are unique.

The key observation is that special floors break the building into segments of normal floors. We need to determine the length of the largest segment.

For example, if:

```
bottom = 2
top = 9
special = [4, 6]
```

then the floors are divided like this:

```
2 3 | 4 | 5 | 6 | 7 8 9
```

The longest consecutive non-special segment is `7 8 9`, which has length `3`.

The constraints are important:

- `special.length` can be as large as `10^5`
- Floor numbers can be as large as `10^9`

Because the range of floors can be extremely large, iterating through every floor individually is not feasible. We need an approach that depends primarily on the number of special floors rather than the size of the floor range.

Some important edge cases include:

- Every rented floor is special
- There is only one special floor
- The longest valid segment occurs before the first special floor
- The longest valid segment occurs after the last special floor
- Special floors are provided in random order

The problem guarantees that all special floors are unique, which simplifies the implementation because we do not need to handle duplicates.

## Approaches

### Brute Force Approach

A brute-force solution would examine every floor from `bottom` to `top`.

One way to implement this is:

1. Put all special floors into a hash set.
2. Iterate through every floor.
3. Count consecutive non-special floors.
4. Reset the count whenever a special floor is encountered.
5. Track the maximum streak.

This approach is correct because it directly simulates the definition of the problem. However, it is too slow for the constraints.

The floor range can be extremely large:

```
top - bottom can approach 10^9
```

Iterating over a billion floors is not practical.

### Optimal Approach

The key insight is that only the positions of special floors matter.

If we sort the special floors, then the valid non-special segments are:

- Between `bottom` and the first special floor
- Between consecutive special floors
- Between the last special floor and `top`

For two consecutive special floors `a` and `b`, the number of non-special floors between them is:

```
b - a - 1
```

This works because both endpoints are special and must be excluded.

After sorting, we only need to scan through the special floors once.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(top - bottom + 1) | O(n) | Iterates through every floor |
| Optimal | O(n log n) | O(1) or O(n) depending on sorting | Sorts special floors and checks gaps |

## Algorithm Walkthrough

1. Sort the `special` array in ascending order.

Sorting allows us to examine gaps between consecutive special floors in order.
2. Compute the gap before the first special floor.

The floors from `bottom` to `special[0] - 1` are valid non-special floors.

Their count is:

```
special[0] - bottom
```
3. Initialize the answer with this first gap.

This ensures we consider segments that appear before the first special floor.
4. Iterate through consecutive pairs of special floors.

For each pair:

```
previous_special
current_special
```

compute:

```
current_special - previous_special - 1
```

This represents the number of consecutive non-special floors between them.
5. Update the maximum answer whenever a larger gap is found.
6. Compute the gap after the last special floor.

The floors from:

```
special[-1] + 1
```

through `top` are non-special.

Their count is:

```
top - special[-1]
```
7. Return the maximum value found.

### Why it works

After sorting, every possible consecutive non-special segment lies either:

- before the first special floor,
- between two adjacent special floors,
- or after the last special floor.

The algorithm explicitly checks all three categories. Since every valid segment is examined exactly once, the maximum computed value must be the correct answer.

## Python Solution

```python
from typing import List

class Solution:
    def maxConsecutive(self, bottom: int, top: int, special: List[int]) -> int:
        special.sort()

        max_consecutive = special[0] - bottom

        for i in range(1, len(special)):
            gap = special[i] - special[i - 1] - 1
            max_consecutive = max(max_consecutive, gap)

        max_consecutive = max(max_consecutive, top - special[-1])

        return max_consecutive
```

The implementation begins by sorting the `special` array so that adjacent special floors can be processed in order.

The variable `max_consecutive` is initialized using the gap before the first special floor. This immediately accounts for sequences starting at `bottom`.

The loop then processes every pair of adjacent special floors. For each pair, the code computes the number of ordinary floors between them using:

```python
special[i] - special[i - 1] - 1
```

The subtraction by `1` is necessary because the special floors themselves are excluded.

Finally, the code checks the segment after the last special floor and returns the largest gap found.

## Go Solution

```go
package main

import "sort"

func maxConsecutive(bottom int, top int, special []int) int {
	sort.Ints(special)

	maxConsecutive := special[0] - bottom

	for i := 1; i < len(special); i++ {
		gap := special[i] - special[i-1] - 1

		if gap > maxConsecutive {
			maxConsecutive = gap
		}
	}

	lastGap := top - special[len(special)-1]

	if lastGap > maxConsecutive {
		maxConsecutive = lastGap
	}

	return maxConsecutive
}
```

The Go implementation follows the same logic as the Python version.

The primary language-specific difference is the use of `sort.Ints` to sort the slice in place. Go also requires explicit comparisons instead of Python's built-in `max` function for integers.

There are no integer overflow concerns because the constraints fit safely within Go's standard `int` type on LeetCode systems.

The main cost comes from sorting the `special` array. After sorting, the algorithm performs a single linear scan through the array.

The scan itself is `O(n)`, but sorting requires `O(n log n)` time, which becomes the overall complexity.

The additional space usage depends on the language's sorting implementation. Python's sort may use auxiliary memory internally, while Go's implementation may also allocate temporary space.

## Edge Cases

One important edge case occurs when every rented floor is special. In this scenario, there are no valid non-special floors at all. A careless implementation might incorrectly return a negative value or fail to handle zero-length gaps. This implementation correctly computes all gaps as zero and returns `0`.

Another important case is when the largest valid segment appears before the first special floor or after the last special floor. Some implementations only examine gaps between consecutive special floors and forget the edges of the building. This solution explicitly checks both boundary regions separately.

A third critical edge case involves unsorted input. The `special` array can appear in arbitrary order. Without sorting, gap calculations between adjacent elements would be meaningless. By sorting first, the algorithm guarantees that every computed gap corresponds to a real consecutive floor segment.

A final subtle edge case occurs when special floors are consecutive, such as `[4, 5, 6]`. The gap formula:

```
current - previous - 1
```

correctly produces zero, ensuring that no invalid negative or incorrect counts appear.
