## Solution

## Problem Understanding

The problem gives us an integer array called `cards`, where each element represents the value written on a card. We need to find the minimum number of consecutive cards that must be picked so that among those picked cards, there exists at least one pair of matching values.

In other words, we are looking for the shortest contiguous subarray that contains at least two equal numbers.

Suppose we encounter the same card value at indices `i` and `j`, where `i < j`. Then the consecutive segment from `i` to `j` already contains two matching cards. The number of cards picked would therefore be:

$$j - i + 1$$

Our goal is to compute the minimum such length across all matching pairs in the array.

If no card value appears more than once, then it is impossible to pick a consecutive sequence containing a matching pair, so we return `-1`.

The constraints are important:

- `1 <= cards.length <= 10^5`
- `0 <= cards[i] <= 10^6`

An array size of `10^5` immediately tells us that quadratic solutions are too slow. Any algorithm with time complexity near `O(n^2)` will likely exceed time limits. This strongly suggests we need a linear or near-linear approach.

Several edge cases are important:

- Arrays with no duplicates at all should return `-1`.
- Arrays where duplicates are adjacent should return `2`, which is the smallest possible valid answer.
- Arrays with many repeated values require us to correctly track the closest repeated occurrence.
- Very small arrays such as length `1` cannot contain duplicates.

## Approaches

### Brute Force Approach

A straightforward solution is to examine every possible subarray and determine whether it contains duplicate values.

For every starting index `i`, we can extend the subarray one element at a time toward the right. While expanding, we maintain a set of seen values. As soon as we encounter a duplicate, we compute the subarray length and update the minimum answer.

This works because every possible consecutive segment is considered, so the smallest valid one will eventually be found.

However, this approach is too slow for large inputs. In the worst case, we may inspect almost every pair of indices, leading to quadratic time complexity.

With `n = 10^5`, an `O(n^2)` algorithm would require roughly `10^10` operations, which is far beyond acceptable limits.

### Optimal Approach

The key observation is that a valid consecutive segment is completely determined by two equal values.

If a card value appears at indices `i` and `j`, then the shortest segment containing those two equal cards is simply the interval between them:

$$j - i + 1$$

Therefore, instead of checking every subarray, we only need to track the most recent position where each card value appeared.

As we iterate through the array:

- If we see a value for the first time, we store its index.
- If we see the value again, we compute the distance between the current index and the previous occurrence.
- We update the minimum answer if this segment is smaller.

A hash map is ideal for this because it allows constant-time lookup of the previous occurrence of each card value.

This reduces the time complexity from `O(n^2)` to `O(n)`.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Checks many subarrays explicitly |
| Optimal | O(n) | O(n) | Tracks last seen index using a hash map |

## Algorithm Walkthrough

1. Initialize a hash map called `last_seen`.

This map stores the most recent index where each card value appeared. The key is the card value, and the value is its latest index.
2. Initialize the answer variable.

Set `minimum_length` to infinity. This represents the best answer found so far.
3. Iterate through the array using index and value.

For each card:

- Let the current index be `i`
- Let the current value be `card`
4. Check whether the current card value has appeared before.

If `card` already exists in `last_seen`, then we found a matching pair.
5. Compute the length of the consecutive segment.

The segment starts at the previous occurrence and ends at the current index.

The length is:

$$i - \text{last\_seen}[card] + 1$$
6. Update the minimum answer.

Compare this segment length with the current best answer and keep the smaller one.
7. Update the latest occurrence.

Store the current index as the newest occurrence of this card value.
8. Finish processing the array.

After the loop:

- If `minimum_length` is still infinity, no duplicates were found, so return `-1`
- Otherwise return `minimum_length`

### Why it works

For every repeated card value, the shortest valid consecutive segment containing that pair is exactly the interval between two occurrences. By always comparing the current index with the most recent previous occurrence, we examine the smallest possible segment ending at the current position. Since every repeated occurrence is processed exactly once, the algorithm guarantees that the global minimum segment length is found.

## Python Solution

```python
from typing import List

class Solution:
    def minimumCardPickup(self, cards: List[int]) -> int:
        last_seen = {}
        minimum_length = float("inf")

        for index, card in enumerate(cards):
            if card in last_seen:
                current_length = index - last_seen[card] + 1
                minimum_length = min(minimum_length, current_length)

            last_seen[card] = index

        return minimum_length if minimum_length != float("inf") else -1
```

The implementation begins by creating a dictionary named `last_seen`. This dictionary tracks the latest index where each card value appeared.

The variable `minimum_length` is initialized to infinity so that any valid segment length found later will replace it.

The loop processes the array from left to right using `enumerate`, which provides both the index and the card value.

Whenever the current card value already exists in `last_seen`, we know we found a matching pair. The segment length is computed using:

```python
index - last_seen[card] + 1
```

We then compare this value against the best answer seen so far.

After processing the current position, we update `last_seen[card]` to the current index. This ensures future duplicate checks always use the most recent occurrence.

Finally, if no duplicate was ever found, the answer remains infinity, and the method returns `-1`.

## Go Solution

```go
func minimumCardPickup(cards []int) int {
    lastSeen := make(map[int]int)
    minimumLength := int(^uint(0) >> 1)

    for index, card := range cards {
        if previousIndex, exists := lastSeen[card]; exists {
            currentLength := index - previousIndex + 1

            if currentLength < minimumLength {
                minimumLength = currentLength
            }
        }

        lastSeen[card] = index
    }

    if minimumLength == int(^uint(0)>>1) {
        return -1
    }

    return minimumLength
}
```

The Go solution follows exactly the same logic as the Python implementation.

The main difference is that Go requires explicit map creation using `make(map[int]int)`.

Since Go does not provide a built-in infinity value for integers, the implementation initializes `minimumLength` using the maximum possible integer value:

```go
int(^uint(0) >> 1)
```

The map lookup uses Go's two-value assignment form:

```go
previousIndex, exists := lastSeen[card]
```

This cleanly checks whether the card value has appeared before.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each element is processed once |
| Space | O(n) | Hash map may store every distinct card value |

The algorithm performs a single pass through the array. Every hash map lookup and update runs in average constant time, so the total running time is linear.

The space usage depends on the number of distinct card values. In the worst case, all values are unique, so the hash map stores `n` entries.

## Edge Cases

One important edge case occurs when the array contains no duplicate values at all. A naive implementation might accidentally return a large default value or `0` instead of `-1`. This implementation avoids that problem by initializing the answer to infinity and checking whether it was ever updated.

Another important case is when duplicates are adjacent, such as `[5,5]`. The shortest valid answer possible is `2`, since two cards are required to form a pair. Some incorrect implementations may accidentally compute `1` because of off-by-one errors in subarray length calculation. This solution correctly uses:

```
index - previous_index + 1
```

which properly counts both endpoints.

A third important edge case is when a value appears many times, such as `[1,2,1,2,1]`. The algorithm must always consider the closest repeated occurrence, not necessarily the first occurrence. By updating the stored index every time a value is seen, the implementation guarantees that future comparisons use the nearest previous occurrence, which minimizes the segment length correctly.
