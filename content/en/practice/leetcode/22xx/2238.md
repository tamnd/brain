## Solution

## Problem Understanding

The problem gives us a database table named `Rides`. Each row represents a single ride and contains three values:

- `ride_id`, the unique identifier for the ride
- `driver_id`, the person driving during that ride
- `passenger_id`, the person riding as the passenger

The key detail is that every row describes a relationship between one driver and one passenger, and the driver and passenger are guaranteed to be different people.

The task is to produce a result table that contains:

- every unique driver who appears in the `driver_id` column
- the number of times that same driver appeared as a passenger in the `passenger_id` column

In other words, for each person who has ever been a driver, we want to count how many rides they participated in as a passenger.

The important distinction is that we only report IDs that appear as drivers. If someone appears only as a passenger and never as a driver, they should not appear in the output.

Consider the sample input:

| ride_id | driver_id | passenger_id |
| --- | --- | --- |
| 1 | 7 | 1 |
| 2 | 7 | 2 |
| 3 | 11 | 1 |
| 4 | 11 | 7 |
| 5 | 11 | 7 |
| 6 | 11 | 3 |

The unique drivers are `7` and `11`.

- Driver `7` appears twice as a passenger, in rides `4` and `5`
- Driver `11` never appears as a passenger

So the result becomes:

| driver_id | cnt |
| --- | --- |
| 7 | 2 |
| 11 | 0 |

The constraints are relatively small for a database aggregation problem, but the important observation is that the task is fundamentally about counting occurrences and preserving rows even when the count is zero. This immediately suggests using SQL aggregation with joins.

A common mistake is forgetting drivers who never appear as passengers. If we only count passenger occurrences directly, those drivers would disappear from the result. Another possible bug is counting passengers who were never drivers, even though the output should include only driver IDs.

## Approaches

### Brute Force Approach

A brute force solution would first gather all distinct drivers. Then, for each driver, scan the entire table again and count how many rows contain that driver in the `passenger_id` column.

Conceptually, this works because we directly verify every ride for every driver. If a driver appears as a passenger in a row, we increment their count.

For example:

1. Extract unique drivers: `{7, 11}`
2. For driver `7`, scan all rides and count passenger occurrences
3. For driver `11`, scan all rides and count passenger occurrences

This approach is correct because every possible ride is checked for every driver. However, it is inefficient because it repeatedly scans the same table.

If there are `D` distinct drivers and `N` rides, the complexity becomes `O(D × N)`.

### Optimal Approach

The better solution is to use SQL aggregation.

The key insight is that we do not need to repeatedly scan the table for each driver. Instead:

1. Extract all unique drivers
2. Count passenger occurrences grouped by passenger ID
3. Join the two results together

A `LEFT JOIN` is essential because some drivers may never appear as passengers. The join preserves all drivers while attaching passenger counts when they exist.

This transforms the problem into two aggregation steps plus one join, which is much more efficient and aligns naturally with relational database operations.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(D × N) | O(D) | Repeatedly scans all rides for every driver |
| Optimal | O(N) | O(D) | Uses grouping and a LEFT JOIN for efficient counting |

## Algorithm Walkthrough

1. First, extract all distinct driver IDs from the `Rides` table.

This gives the complete set of people that must appear in the final answer. We use `DISTINCT` because the same driver may appear in multiple rides.
2. Next, count how many times each passenger appears in the `passenger_id` column.

We use `GROUP BY passenger_id` and compute `COUNT(*)`. This produces a mapping from passenger ID to passenger frequency.
3. Perform a `LEFT JOIN` between the distinct drivers and the passenger counts.

The join condition matches:

```
driver_id = passenger_id
```

A `LEFT JOIN` is important because every driver must remain in the result even if they never appeared as a passenger.
4. Replace missing counts with zero.

If a driver never appeared as a passenger, the joined count becomes `NULL`. We use `COALESCE` or `IFNULL` to convert that value to `0`.
5. Return the final result.

The output contains each driver ID and the number of times they were a passenger.

### Why it works

The algorithm works because it separates the problem into two independent facts:

- who qualifies to appear in the output, all distinct drivers
- how many times each person appeared as a passenger

The `LEFT JOIN` merges these facts correctly. Every driver is preserved, and passenger frequencies are attached whenever they exist. Drivers without matching passenger records receive a count of zero.

## Python Solution

Although this is a database problem and normally solved in SQL, the following demonstrates the equivalent logic in Python.

```python
from collections import Counter
from typing import List

class Solution:
    def numberOfTimes(self, rides: List[List[int]]) -> List[List[int]]:
        drivers = set()
        passenger_count = Counter()

        for ride_id, driver_id, passenger_id in rides:
            drivers.add(driver_id)
            passenger_count[passenger_id] += 1

        result = []

        for driver_id in drivers:
            result.append([driver_id, passenger_count.get(driver_id, 0)])

        return result
```

The implementation closely follows the algorithm described earlier.

The `drivers` set stores every unique driver ID. This ensures that each driver appears exactly once in the final result.

The `Counter` named `passenger_count` tracks how many times each passenger ID appears. Every ride increments the count for its passenger.

After processing all rides, we iterate through the distinct drivers and look up their passenger frequency in the counter. If a driver never appeared as a passenger, the counter lookup defaults to zero.

The final list contains pairs of `[driver_id, count]`.

For the actual LeetCode database problem, the expected solution is SQL rather than Python code execution.

A standard SQL solution is:

```python
SELECT
    d.driver_id,
    COALESCE(p.cnt, 0) AS cnt
FROM
    (
        SELECT DISTINCT driver_id
        FROM Rides
    ) d
LEFT JOIN
    (
        SELECT
            passenger_id,
            COUNT(*) AS cnt
        FROM Rides
        GROUP BY passenger_id
    ) p
ON d.driver_id = p.passenger_id;
```

## Go Solution

```go
package main

import "fmt"

func numberOfTimes(rides [][]int) [][]int {
	driverSet := make(map[int]bool)
	passengerCount := make(map[int]int)

	for _, ride := range rides {
		driverID := ride[1]
		passengerID := ride[2]

		driverSet[driverID] = true
		passengerCount[passengerID]++
	}

	result := [][]int{}

	for driverID := range driverSet {
		count := passengerCount[driverID]
		result = append(result, []int{driverID, count})
	}

	return result
}

func main() {
	rides := [][]int{
		{1, 7, 1},
		{2, 7, 2},
		{3, 11, 1},
		{4, 11, 7},
		{5, 11, 7},
		{6, 11, 3},
	}

	fmt.Println(numberOfTimes(rides))
}
```

The Go implementation mirrors the Python approach closely.

A `map[int]bool` is used to track distinct drivers, while `map[int]int` stores passenger frequencies.

Go maps automatically return the zero value for missing keys, so looking up a driver who never appeared as a passenger naturally returns `0`. This eliminates the need for explicit default handling like Python's `.get()` method.

The result is built as a slice of integer slices, where each inner slice contains `[driverID, count]`.

## Edge Cases

One important edge case occurs when a driver never appears as a passenger. A naive implementation using only grouped passenger counts could accidentally omit such drivers entirely. The correct implementation uses a `LEFT JOIN` in SQL, or a default value lookup in code, ensuring these drivers still appear with count `0`.

Another edge case occurs when someone appears many times as a passenger but never as a driver. These people should not appear in the result at all. The implementation handles this correctly because the final iteration is based only on distinct driver IDs.

A third important edge case involves repeated rides with the same passenger. An incorrect implementation might accidentally count only distinct passenger appearances rather than total occurrences. The solution correctly increments the passenger count for every matching row, ensuring duplicate appearances are counted separately.

A final subtle case occurs when drivers and passengers form cyclic relationships, such as driver `1` riding with driver `2`, and later driver `2` riding with driver `1`. The implementation handles this naturally because passenger counting is independent of driver extraction.
