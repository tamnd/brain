## Solution

## Problem Understanding

This problem asks us to transform a binary array so that every element becomes `1` using the minimum number of operations. Each operation is a **k-bit flip**, meaning we choose a contiguous subarray of length `k` and invert every bit inside it. A `0` becomes `1`, and a `1` becomes `0`.

The input consists of two values:

- `nums`, a binary array containing only `0`s and `1`s
- `k`, the fixed size of every flip operation

The goal is to determine the **minimum number of flips** required so that no `0` remains in the array. If it is impossible to achieve an all-ones array, we must return `-1`.

The key challenge is that flips overlap and interact with one another. Flipping a range changes future decisions because a bit may be flipped multiple times. A naive simulation that repeatedly flips bits directly can become extremely expensive.

The constraints are important:

- `nums.length <= 10^5`
- `k <= nums.length`

Since the array can contain up to one hundred thousand elements, any quadratic approach such as repeatedly flipping ranges directly will be too slow. This strongly suggests that we need a near-linear solution, ideally `O(n)`.

Several edge cases are especially important:

- If `k = 1`, every `0` can simply be flipped individually, so the answer is the number of zeros.
- If `k = nums.length`, we only get one possible flip range, so the answer depends on whether one full-array flip can solve the problem.
- Near the end of the array, we may encounter a `0` that cannot start a valid flip because there are fewer than `k` positions remaining. In that case, the answer must be `-1`.
- Multiple overlapping flips may affect a position, so we cannot rely only on the original value of `nums[i]`.

## Approaches

### Brute Force

A straightforward idea is to process the array from left to right. Whenever we encounter a `0`, we immediately flip the next `k` elements to turn that bit into a `1`.

This greedy idea is actually correct, because once we move past an index, we can never modify it again. If the current bit is effectively `0`, the only possible choice is to flip starting at that position.

However, a naive implementation performs the flip explicitly by iterating through the next `k` elements and toggling every bit. Since we may perform up to `O(n)` flips and each flip costs `O(k)`, the worst-case complexity becomes `O(n × k)`.

For `n = 10^5`, this can easily exceed time limits.

### Optimal Approach, Sliding Window with Difference Tracking

The important observation is that we do **not actually need to flip every bit physically**.

Instead, we only need to know whether a position has been flipped an **odd** or **even** number of times.

Suppose we process the array left to right. At each index, we want to know the bit's **effective value** after all currently active flips.

We can track active flips using a sliding window idea:

- When we start a flip at index `i`, it affects the next `k` elements.
- When we move beyond `i + k - 1`, that flip expires.
- Instead of modifying the entire range, we record where flips start and remove their effect later.

We maintain:

- `current_flips`, the parity of active flips
- an auxiliary array `flip_started` to remember where flips began

At index `i`:

- If a flip started at `i - k`, its effect ends, so we remove it.
- Compute the effective bit using parity:

`effective_bit = nums[i] XOR (current_flips % 2)`
- If the effective bit is `0`, we must flip starting here.
- If there are fewer than `k` elements remaining, return `-1`.

This lets us process each element only once.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | `O(n × k)` | `O(1)` | Explicitly flips each range |
| Optimal | `O(n)` | `O(n)` | Tracks active flips using a sliding window |

## Algorithm Walkthrough

1. Create an auxiliary array called `flip_started` of size `n`, initialized with `0`. This records whether we started a flip at each index.
2. Maintain two variables:

- `current_flips`, which stores how many active flips currently affect the current position
- `total_flips`, which counts the answer
3. Iterate through the array from left to right.
4. At each index `i`, first remove expired flips. If `i >= k`, check whether a flip started at `i - k`. If so, subtract its effect from `current_flips` because that flip window has ended.
5. Compute the effective value of the current bit. A bit changes only if it has been flipped an odd number of times:

- even flips → unchanged
- odd flips → inverted

Therefore:

`effective_bit = nums[i] XOR (current_flips % 2)`
6. If the effective bit is `0`, we must flip starting at index `i`. This is forced because future flips cannot affect earlier positions.
7. Before starting the flip, check whether `i + k > n`. If true, there are not enough elements left, so return `-1`.
8. Otherwise:

- mark `flip_started[i] = 1`
- increment `current_flips`
- increment `total_flips`
9. Continue until the array ends.
10. Return `total_flips`.

### Why it works

The algorithm relies on a greedy invariant: when processing index `i`, if the effective bit is `0`, the only valid move is to start a flip at `i`. No future flip can affect this position once we move past it. Therefore, every decision is forced and locally optimal.

Because we process left to right and only flip when necessary, we never perform unnecessary operations. Since each forced decision is optimal, the final answer is minimal.

## Python Solution

```
from typing import List

class Solution:
    def minKBitFlips(self, nums: List[int], k: int) -> int:
        n = len(nums)
        flip_started = [0] * n

        current_flips = 0
        total_flips = 0

        for i in range(n):
            # Remove expired flip effect
            if i >= k:
                current_flips -= flip_started[i - k]

            # Determine effective value after flips
            effective_bit = nums[i] ^ (current_flips % 2)

            # Must flip if effective bit is 0
            if effective_bit == 0:
                if i + k > n:
                    return -1

                flip_started[i] = 1
                current_flips += 1
                total_flips += 1

        return total_flips
```

The implementation follows the algorithm directly.

We first initialize `flip_started`, which tracks where flips begin. This lets us efficiently remove a flip's effect once its window expires.

The variable `current_flips` keeps track of how many flips currently affect the current index. Since only parity matters, an odd count means the bit is inverted and an even count means it remains unchanged.

For every index, we first remove expired flips by subtracting `flip_started[i - k]`. Then we compute the effective bit using XOR. If that bit is `0`, we are forced to start a flip.

Before starting the flip, we verify that enough positions remain in the array. If not, it is impossible to solve the problem, so we return `-1`.

Otherwise, we record the flip and update the counters.

## Go Solution

```
func minKBitFlips(nums []int, k int) int {
	n := len(nums)
	flipStarted := make([]int, n)

	currentFlips := 0
	totalFlips := 0

	for i := 0; i < n; i++ {
		// Remove expired flip effect
		if i >= k {
			currentFlips -= flipStarted[i-k]
		}

		// Effective bit after flips
		effectiveBit := nums[i] ^ (currentFlips % 2)

		// Must flip if effective bit is 0
		if effectiveBit == 0 {
			if i+k > n {
				return -1
			}

			flipStarted[i] = 1
			currentFlips++
			totalFlips++
		}
	}

	return totalFlips
}
```

The Go implementation mirrors the Python solution closely.

Since Go does not have Python's dynamic lists, we allocate `flipStarted` with `make([]int, n)`.

Integer overflow is not a concern here because the maximum number of flips is at most `n`, which fits comfortably inside Go's `int`.

The solution uses slices naturally and does not require any special handling for empty input because the constraints guarantee `nums.length >= 1`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | `O(n)` | Each index is processed exactly once |
| Space | `O(n)` | Auxiliary array stores flip start positions |

The algorithm is linear because every element is visited once, and each operation inside the loop takes constant time. We never explicitly flip `k` elements, which avoids the `O(n × k)` cost of brute force.

The additional space comes from the `flip_started` array, which stores whether a flip began at each index.

## Edge Cases

### A zero appears too close to the end

A common bug occurs when a `0` is encountered near the end of the array, but there are not enough elements left for a `k`-length flip. For example, `nums = [1,1,0]` with `k = 2`.

A naive implementation might attempt an out-of-bounds flip or continue incorrectly. This implementation explicitly checks `i + k > n` before flipping and immediately returns `-1`.

### Multiple overlapping flips

Bits can be affected by many overlapping operations. For example, in `[0,0,0,1,0,1,1,0]` with `k = 3`, some positions are influenced by several flips.

A buggy solution may physically mutate the array incorrectly or lose track of overlapping effects. This implementation avoids that problem by tracking only the parity of active flips, ensuring the effective bit is always computed correctly.

### `k = 1`

When `k = 1`, every flip affects only one element. The problem reduces to counting zeros.

Some implementations accidentally overcomplicate this case or mishandle window expiration. Here, the same general logic works naturally without any special handling, because every flip expires immediately after one position.
