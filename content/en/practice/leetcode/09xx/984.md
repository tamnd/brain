# LeetCode 984 - String Without AAA or BBB

## Problem Understanding

The problem gives us two integers, `a` and `b`, representing how many `'a'` and `'b'` characters we must place into a final string. The goal is to construct any valid string that satisfies three conditions simultaneously:

1. The string must contain exactly `a` occurrences of `'a'`
2. The string must contain exactly `b` occurrences of `'b'`
3. The string must never contain `"aaa"` or `"bbb"` as a substring

The output does not need to be unique. Any valid arrangement is acceptable.

For example, if `a = 1` and `b = 2`, then valid outputs include `"abb"`, `"bab"`, and `"bba"`. All of them contain one `'a'`, two `'b'`, and avoid three consecutive identical characters.

The constraints are small, with both `a` and `b` between `0` and `100`. Because the input size is tiny, many approaches are technically feasible. However, the challenge is designing a clean and correct construction algorithm.

The most important observation is that the problem guarantees a valid answer always exists. This removes the need to detect impossible cases.

Several edge cases are important:

- One character count may be much larger than the other, such as `a = 4, b = 1`
- One count may be zero, although the guarantee ensures no invalid impossible configuration appears
- The counts may be equal
- The algorithm must avoid accidentally creating three consecutive identical characters while greedily appending characters

A naive greedy strategy that always places the more frequent character can fail if it creates `"aaa"` or `"bbb"` near the end of construction.

## Approaches

### Brute Force Approach

A brute force solution would generate all possible strings of length `a + b` containing exactly `a` `'a'` characters and `b` `'b'` characters. For each generated string, we would check whether it contains `"aaa"` or `"bbb"`.

This approach is correct because it exhaustively examines every candidate arrangement and returns one that satisfies all constraints.

However, the number of possible strings is combinatorial:

$$\binom{a+b}{a}$$

This grows exponentially with the input size. Even though the constraints are relatively small, brute force becomes impractical quickly.

### Optimal Greedy Approach

The key insight is that we should always try to place the character with the larger remaining count, unless doing so would create three consecutive identical characters.

Suppose we currently have more `'a'` characters remaining than `'b'`. If we do not place `'a'` aggressively enough, we may later be forced into creating `"aaa"` because too many `'a'` characters remain clustered together.

At every step:

- Prefer the character with the larger remaining count
- But if the last two characters are already the same, we must place the opposite character instead

This greedy strategy works because it continuously balances the remaining counts while enforcing the no-three-consecutive rule locally.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^(a+b)) | O(a+b) | Generates all candidate strings |
| Optimal Greedy | O(a+b) | O(a+b) | Builds the answer incrementally while avoiding invalid triples |

## Algorithm Walkthrough

1. Initialize an empty result list that will store the constructed characters.
2. Continue building the string while either `a` or `b` is still greater than zero.
3. At each step, first determine whether adding `'a'` or `'b'` would create three consecutive identical characters.
4. If the last two characters in the result are both `'a'`, then we are forbidden from adding another `'a'`. We must add `'b'`.
5. Similarly, if the last two characters are both `'b'`, then we must add `'a'`.
6. If neither restriction applies, choose the character with the larger remaining count. This greedy decision helps distribute the more frequent character early, reducing the risk of being forced into an invalid sequence later.
7. Append the selected character to the result and decrement its corresponding count.
8. Repeat until all characters are used.
9. Join the result list into a string and return it.

### Why it works

The algorithm maintains an important invariant: the constructed prefix never contains `"aaa"` or `"bbb"`.

Whenever two identical trailing characters already exist, the algorithm forces the opposite character, preventing invalid triples. Otherwise, it greedily consumes the more frequent remaining character, which avoids accumulating too many unused copies of one character. Because a valid answer is guaranteed to exist, this strategy successfully completes the construction without getting stuck.

## Python Solution

```
class Solution:
    def strWithout3a3b(self, a: int, b: int) -> str:
        result = []

        while a > 0 or b > 0:
            n = len(result)

            # Check whether adding 'a' would create "aaa"
            if n >= 2 and result[-1] == result[-2] == 'a':
                result.append('b')
                b -= 1

            # Check whether adding 'b' would create "bbb"
            elif n >= 2 and result[-1] == result[-2] == 'b':
                result.append('a')
                a -= 1

            # Otherwise greedily choose the larger remaining count
            elif a >= b:
                result.append('a')
                a -= 1

            else:
                result.append('b')
                b -= 1

        return "".join(result)
```

The implementation closely follows the greedy strategy described earlier.

The `result` list stores characters incrementally because appending to a list is efficient in Python. At the end, `"".join(result)` converts the list into the final string.

Inside the loop, the code first checks whether the last two characters are identical. If they are both `'a'`, then adding another `'a'` would create `"aaa"`, so the algorithm is forced to place `'b'`. The symmetric logic applies for `'b'`.

If neither restriction applies, the algorithm greedily chooses the character with the larger remaining count. This helps balance the remaining characters and avoids future conflicts.

The loop continues until both counts reach zero.

## Go Solution

```
func strWithout3a3b(a int, b int) string {
    result := make([]byte, 0)

    for a > 0 || b > 0 {
        n := len(result)

        // Prevent "aaa"
        if n >= 2 && result[n-1] == 'a' && result[n-2] == 'a' {
            result = append(result, 'b')
            b--

        // Prevent "bbb"
        } else if n >= 2 && result[n-1] == 'b' && result[n-2] == 'b' {
            result = append(result, 'a')
            a--

        // Greedily choose larger count
        } else if a >= b {
            result = append(result, 'a')
            a--

        } else {
            result = append(result, 'b')
            b--
        }
    }

    return string(result)
}
```

The Go implementation mirrors the Python logic almost exactly.

Instead of using a Python list of characters, Go uses a `[]byte` slice for efficient character appending. At the end, the byte slice is converted into a string using `string(result)`.

There are no integer overflow concerns because the counts are at most `100`.

## Edge Cases

### One Character Dominates the Other

A case like `a = 5, b = 2` is dangerous because a naive greedy strategy might produce `"aaab..."`, violating the rules immediately. The implementation avoids this by checking the last two characters before appending. If two `'a'` characters already exist consecutively, the algorithm is forced to insert `'b'`.

### Equal Counts

When `a` and `b` are equal, such as `a = 3, b = 3`, multiple valid solutions exist. A buggy implementation might overcomplicate the balancing logic. This solution naturally alternates characters because neither count dominates for long.

### Very Small Inputs

Cases like `a = 1, b = 0` or `a = 0, b = 1` can expose index-access bugs if the algorithm blindly checks the last two characters without verifying the current length first. The implementation safely checks `n >= 2` before accessing `result[-1]` and `result[-2]`.

### Near Triple Formation

A subtle case occurs when the current string already ends in two identical characters. For example, after building `"aa"`, the algorithm must not append another `'a'` even if more `'a'` characters remain. The explicit triple-prevention checks ensure correctness in these situations.
