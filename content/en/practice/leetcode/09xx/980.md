## Solution

## Problem Understanding

LeetCode 980, Unique Paths III, asks us to count how many valid paths exist in a grid under very strict movement rules.

The grid contains four types of cells:

- `1`, the starting position
- `2`, the ending position
- `0`, normal empty cells
- `-1`, blocked obstacle cells

We can move only in four directions: up, down, left, and right. A valid path must satisfy two conditions simultaneously:

1. The path starts at the start cell and ends at the end cell.
2. Every non-obstacle cell must be visited exactly once.

This second condition is the most important part of the problem. We are not simply finding any route from start to end. We are effectively searching for a Hamiltonian path over all traversable cells in the grid.

The input size is deceptively small. While `m` and `n` can each be as large as 20, the constraint `m * n <= 20` is the key observation. The total number of cells is at most 20, which strongly suggests that exponential algorithms involving backtracking or bitmasking are acceptable.

The output is a single integer representing the number of distinct valid walks.

Several edge cases are important:

- The start and end cells may be adjacent, but the path is invalid unless all walkable cells are visited.
- Obstacles may isolate parts of the grid, making a complete traversal impossible.
- A path may reach the ending cell too early before all cells are visited, which must not count as valid.
- The grid can be as small as a single row or single column.
- There is guaranteed to be exactly one start cell and exactly one end cell.

## Approaches

### Brute Force Approach

The most straightforward approach is to generate every possible path from the start cell to the end cell while avoiding revisiting cells. For each generated path, we check whether every non-obstacle square was visited exactly once.

This can be implemented using recursive depth-first search. At every step, we attempt to move in all four directions. If a neighboring cell is inside the grid, not blocked, and not already visited, we continue recursively.

The brute-force method is correct because it exhaustively explores every legal path configuration. Any valid solution must appear somewhere in this search tree.

However, the search space grows explosively. In the worst case, each step branches into up to four directions, and the recursion depth can reach nearly 20 cells. The total number of possibilities becomes exponential.

Although the input size is small enough that exponential behavior is expected, naive brute force still performs unnecessary work unless we carefully track visitation state and prune invalid branches efficiently.

### Optimal Approach, Backtracking with Visited State

The key insight is that this problem naturally forms a constrained traversal problem where each walkable cell must be visited exactly once.

Because the grid size is at most 20 cells, we can use backtracking efficiently. Instead of storing entire paths, we only maintain:

- Current position
- Number of remaining cells to visit
- Which cells are already visited

At each recursive step:

- Mark the current cell as visited
- Explore all four neighboring directions
- Backtrack afterward by unmarking the cell

The crucial optimization is tracking how many cells remain unvisited. When we reach the ending cell, the path is valid only if no remaining cells exist.

This pruning dramatically reduces unnecessary work because paths that reach the end too early are discarded immediately.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(4^K) | O(K) | Explores all possible paths without efficient pruning |
| Optimal | O(4^K) | O(K) | Backtracking with visitation tracking and pruning, where K is number of non-obstacle cells |

Even though both are exponential, the optimized backtracking solution is practical because `K <= 20`.

## Algorithm Walkthrough

1. Scan the entire grid to locate the starting position and count how many non-obstacle cells exist.

We need the total number of traversable cells because a valid path must visit every one exactly once. This count includes the start and end cells.
2. Start a depth-first search from the starting cell.

The recursive DFS function will track:

- Current row and column
- Number of remaining cells that still need to be visited
3. When entering a cell, decrement the remaining cell count.

This represents consuming one traversable square.
4. If the current cell is the ending square, check whether all cells have been visited.

If the remaining count becomes zero exactly when we arrive at the end cell, then we found a valid path and return `1`.

Otherwise, the path is invalid because we reached the end too early.
5. Temporarily mark the current cell as visited.

A simple technique is to overwrite the cell value with `-1`. This prevents revisiting the same square in the current recursive path.
6. Explore all four directions.

For each neighboring position:

- Check grid boundaries
- Skip obstacles
- Skip already visited cells

Valid neighbors continue recursively.
7. After exploring all neighbors, restore the cell.

This is the core idea of backtracking. Other recursive branches must be allowed to reuse the cell.
8. Return the total number of valid paths found from all recursive branches.

### Why it works

The algorithm maintains the invariant that every recursive call represents a partial path where all visited cells are unique. Since visited cells are marked immediately and restored only during backtracking, no recursive branch can revisit a cell illegally.

Additionally, the remaining-cell counter guarantees that we count only paths that visit every non-obstacle square exactly once before reaching the ending cell.

Because the DFS explores every legal traversal possibility exactly once, the final count is correct.

## Python Solution

```
from typing import List

class Solution:
    def uniquePathsIII(self, grid: List[List[int]]) -> int:
        rows = len(grid)
        cols = len(grid[0])

        start_row = 0
        start_col = 0
        empty_cells = 0

        # Count all non-obstacle cells
        # Also locate the starting position
        for r in range(rows):
            for c in range(cols):
                if grid[r][c] != -1:
                    empty_cells += 1

                if grid[r][c] == 1:
                    start_row = r
                    start_col = c

        directions = [(1, 0), (-1, 0), (0, 1), (0, -1)]

        def dfs(row: int, col: int, remaining: int) -> int:
            # Out of bounds or blocked
            if (
                row < 0
                or row >= rows
                or col < 0
                or col >= cols
                or grid[row][col] == -1
            ):
                return 0

            # Reached ending cell
            if grid[row][col] == 2:
                return 1 if remaining == 1 else 0

            # Mark visited
            temp = grid[row][col]
            grid[row][col] = -1

            total_paths = 0

            for dr, dc in directions:
                next_row = row + dr
                next_col = col + dc

                total_paths += dfs(next_row, next_col, remaining - 1)

            # Backtrack
            grid[row][col] = temp

            return total_paths

        return dfs(start_row, start_col, empty_cells)
```

The implementation begins by scanning the grid once. During this scan, we count every non-obstacle square and identify the starting position.

The DFS function represents the recursive backtracking process. Each call corresponds to standing on one cell while trying to complete a valid traversal of all remaining cells.

The first conditional handles invalid positions such as out-of-bounds coordinates, obstacles, or already visited cells.

The second conditional handles the ending cell. A path counts as valid only when the ending cell is reached with exactly one remaining cell left, namely the ending cell itself.

The current cell is then marked as visited by temporarily replacing its value with `-1`. This avoids needing a separate visited matrix.

The algorithm recursively explores all four neighboring directions and accumulates the number of valid paths returned.

Finally, the original value is restored before returning. This restoration is essential because different recursive branches must see the cell as available again.

## Go Solution

```
func uniquePathsIII(grid [][]int) int {
	rows := len(grid)
	cols := len(grid[0])

	startRow, startCol := 0, 0
	emptyCells := 0

	for r := 0; r < rows; r++ {
		for c := 0; c < cols; c++ {
			if grid[r][c] != -1 {
				emptyCells++
			}

			if grid[r][c] == 1 {
				startRow = r
				startCol = c
			}
		}
	}

	directions := [][]int{
		{1, 0},
		{-1, 0},
		{0, 1},
		{0, -1},
	}

	var dfs func(int, int, int) int

	dfs = func(row int, col int, remaining int) int {
		if row < 0 || row >= rows || col < 0 || col >= cols || grid[row][col] == -1 {
			return 0
		}

		if grid[row][col] == 2 {
			if remaining == 1 {
				return 1
			}
			return 0
		}

		temp := grid[row][col]
		grid[row][col] = -1

		totalPaths := 0

		for _, dir := range directions {
			nextRow := row + dir[0]
			nextCol := col + dir[1]

			totalPaths += dfs(nextRow, nextCol, remaining-1)
		}

		grid[row][col] = temp

		return totalPaths
	}

	return dfs(startRow, startCol, emptyCells)
}
```

The Go implementation follows the same recursive structure as the Python version.

One difference is that Go does not support nested named functions directly without variable declaration first, so the DFS function is declared using `var dfs func(...)`.

The grid is modified in place during recursion exactly like the Python implementation. Since Go slices reference underlying arrays, the updates are visible across recursive calls automatically.

No integer overflow concerns exist because the maximum number of valid paths is small relative to Go's integer range.

## Edge Cases

One important edge case occurs when the ending cell is reachable immediately from the starting cell. A naive implementation might incorrectly count such a path as valid. However, the problem requires visiting every non-obstacle cell exactly once. The implementation prevents this bug by checking whether the remaining cell count equals exactly one when arriving at the ending square.

Another subtle case involves disconnected regions caused by obstacles. Some empty cells may become impossible to reach from the start. A careless DFS might continue exploring indefinitely or count incomplete traversals. In this implementation, any branch that cannot eventually reach all cells naturally terminates with zero valid paths because the DFS reaches dead ends before satisfying the completion condition.

Single-row and single-column grids are also easy sources of indexing mistakes. Movement logic must still correctly handle boundaries when only one dimension exists. The implementation uses explicit boundary checks for every recursive move, so these degenerate grid shapes work correctly without requiring special-case logic.

A final important edge case involves revisiting cells accidentally. Since the problem requires every cell to be visited exactly once, revisits invalidate the path immediately. The implementation handles this safely by marking cells as `-1` during recursion and restoring them during backtracking. This guarantees that each recursive branch maintains a correct visitation state independently.
