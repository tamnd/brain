## Solution

## Problem Understanding

The problem asks us to find the maximum width ramp in an array of integers.

A ramp is defined as a pair of indices `(i, j)` such that:

- `i < j`
- `nums[i] <= nums[j]`

The width of the ramp is simply the distance between the two indices:

$$\text{width} = j - i$$

Our goal is to find the largest possible width among all valid ramps.

In other words, for every position `j`, we want to find the earliest position `i` before it where the value at `i` is less than or equal to the value at `j`. The farther apart those indices are, the larger the ramp width becomes.

The input is an integer array `nums`, and the output is a single integer representing the maximum width of any valid ramp. If no valid ramp exists, the answer is `0`.

The constraints are important:

- `2 <= nums.length <= 5 * 10^4`
- `0 <= nums[i] <= 5 * 10^4`

An array length of up to fifty thousand elements immediately rules out inefficient quadratic solutions for production use. An `O(n^2)` solution could require billions of comparisons in the worst case, which would be too slow.

Several edge cases are worth identifying early:

- A strictly decreasing array such as `[5,4,3,2,1]` contains no valid ramp, so the answer is `0`.
- A strictly increasing array such as `[1,2,3,4,5]` has the widest possible ramp from index `0` to `n-1`.
- Arrays with duplicate values are important because equality is allowed. For example, `[5,5,5]` produces a valid ramp with width `2`.
- The optimal ramp may not involve the minimum or maximum values in the array. We must carefully track candidate starting indices.

## Approaches

### Brute Force Approach

The most direct solution is to examine every possible pair `(i, j)` where `i < j`.

For each pair:

- Check whether `nums[i] <= nums[j]`
- If the condition is true, compute `j - i`
- Track the maximum width encountered

This approach is guaranteed to produce the correct answer because it explicitly checks every possible ramp.

However, the time complexity is too large. There are approximately:

$$\frac{n(n-1)}{2}$$

possible pairs, which leads to `O(n^2)` time complexity.

With `n = 50,000`, this becomes far too slow.

### Optimal Monotonic Stack Approach

The key insight is that not every index needs to be considered as a starting point.

Suppose we already have an earlier index with a smaller value. Then a later index with a larger value is never better as a left endpoint because:

- it appears later, reducing width
- it has a value that is not smaller

This means we only care about indices that form a strictly decreasing sequence of values from left to right.

We can store these candidate starting indices in a monotonic decreasing stack.

Then, by scanning from right to left, we try to match each position `j` with the earliest valid `i` from the stack. Since we process from the far right first, the first successful match for a stack element automatically gives its maximum possible width.

This reduces the complexity to linear time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Checks every pair of indices |
| Optimal | O(n) | O(n) | Uses a monotonic decreasing stack |

## Algorithm Walkthrough

### Step 1: Build a Monotonic Decreasing Stack

We iterate through the array from left to right.

The stack stores indices, not values.

For each index `i`:

- If the stack is empty, push `i`
- Otherwise, compare `nums[i]` with the value at the index on top of the stack
- Push `i` only if `nums[i]` is smaller

This creates a stack of indices whose values form a strictly decreasing sequence.

The reason this works is that only the smallest value seen so far is useful as a potential left endpoint. Any larger value appearing later is dominated by an earlier smaller value.

For example:

```
nums = [6,0,8,2,1,5]

stack construction:
index 0 -> value 6 -> push
index 1 -> value 0 -> push
index 2 -> value 8 -> skip
index 3 -> value 2 -> skip
index 4 -> value 1 -> skip
index 5 -> value 5 -> skip

stack = [0,1]
```

### Step 2: Scan From Right to Left

Now we traverse the array from the end toward the beginning.

For each index `j`:

- While the stack is not empty
- And `nums[stack[-1]] <= nums[j]`
- We have found a valid ramp

We compute:

$$j - stack[-1]$$

and update the maximum width.

After that, we pop the stack element.

We pop because scanning from right to left guarantees that this is the largest possible width for that starting index.

### Step 3: Continue Until the Stack Is Empty

Eventually, all candidate starting positions are matched or discarded.

The maximum width found during the scan is the answer.

### Why it works

The stack contains only indices that could potentially produce the widest ramp. Every stored index has a smaller value than all earlier stored indices.

When scanning from right to left, we try to pair each candidate left endpoint with the farthest possible valid right endpoint first. Once a valid match is found, no later scan can produce a larger width for that left index, so it is safe to remove it.

This guarantees that every optimal ramp is considered exactly once.

## Python Solution

```
from typing import List

class Solution:
    def maxWidthRamp(self, nums: List[int]) -> int:
        n = len(nums)

        decreasing_stack = []

        # Build monotonic decreasing stack
        for i in range(n):
            if not decreasing_stack or nums[i] < nums[decreasing_stack[-1]]:
                decreasing_stack.append(i)

        max_width = 0

        # Traverse from right to left
        for j in range(n - 1, -1, -1):
            while decreasing_stack and nums[decreasing_stack[-1]] <= nums[j]:
                i = decreasing_stack.pop()
                max_width = max(max_width, j - i)

        return max_width
```

The implementation follows the exact algorithm described earlier.

The first loop constructs the monotonic decreasing stack. Each stored index represents a potentially optimal left endpoint. We only push indices whose values are strictly smaller than the current minimum because larger values are never better candidates.

The second loop scans from right to left. For each position `j`, we repeatedly check whether the top stack index forms a valid ramp.

If it does:

- we compute the width
- update the answer
- remove the index from the stack

Removing the index is important because the current `j` is already the farthest possible valid partner for that index.

The algorithm processes every index at most twice:

- once when pushed
- once when popped

This is why the solution runs in linear time.

## Go Solution

```
func maxWidthRamp(nums []int) int {
	n := len(nums)

	decreasingStack := []int{}

	// Build monotonic decreasing stack
	for i := 0; i < n; i++ {
		if len(decreasingStack) == 0 ||
			nums[i] < nums[decreasingStack[len(decreasingStack)-1]] {

			decreasingStack = append(decreasingStack, i)
		}
	}

	maxWidth := 0

	// Traverse from right to left
	for j := n - 1; j >= 0; j-- {
		for len(decreasingStack) > 0 &&
			nums[decreasingStack[len(decreasingStack)-1]] <= nums[j] {

			i := decreasingStack[len(decreasingStack)-1]
			decreasingStack = decreasingStack[:len(decreasingStack)-1]

			if j-i > maxWidth {
				maxWidth = j - i
			}
		}
	}

	return maxWidth
}
```

The Go implementation is structurally identical to the Python version.

The stack is implemented using a slice of integers. Pushing is done with `append`, and popping is handled by slicing off the final element.

There are no integer overflow concerns because the maximum index difference is at most `5 * 10^4`, which easily fits within Go's `int` type.

Unlike Python, Go does not have built-in stack operations, so slice manipulation is used directly.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each index is pushed and popped at most once |
| Space | O(n) | The stack may contain up to all indices |

The algorithm is linear because every array index participates in at most two stack operations:

- one push
- one pop

No nested repeated scanning occurs. Even though there is a nested `while` loop, the total number of pops across the entire execution is at most `n`.

The space complexity is linear in the worst case, such as a strictly decreasing array where every index is stored in the stack.

## Edge Cases

A strictly decreasing array is one of the most important edge cases. In an input such as `[5,4,3,2,1]`, no pair satisfies `nums[i] <= nums[j]` for `i < j`. A buggy implementation might accidentally return a negative width or incorrectly match equal indices. This implementation correctly returns `0` because no stack element is ever popped during the right-to-left scan.

Arrays with all equal values are another important case. Since equality is allowed, every pair forms a valid ramp. For example, `[7,7,7,7]` should return `3`. Some implementations mistakenly use a strict inequality and would fail here. This solution uses `<=`, ensuring equal values are handled correctly.

Very small arrays can expose off-by-one mistakes. For an array like `[2,2]`, the correct answer is `1`. The algorithm handles this naturally because the first index enters the stack, and the second index successfully forms a ramp during the reverse scan.

Arrays where the optimal left endpoint is not the smallest overall value are also important. For example, `[10,9,8,1,2,3,4]` has its best ramp starting at index `3`. The monotonic stack correctly preserves all decreasing minima, ensuring later smaller values remain available as candidates.
