## Solution

## Problem Understanding

The problem asks us to redistribute coins in a binary tree so that each node ends up with exactly one coin. The input is the root of a binary tree where each node contains a certain number of coins. The total number of coins across the tree equals the total number of nodes, ensuring that it is theoretically possible for every node to have exactly one coin after redistribution.

The output is the minimum number of moves required to achieve this state. A move consists of transferring a single coin between two adjacent nodes (parent to child or child to parent).

The constraints indicate that the tree has up to 100 nodes, so efficiency matters, but we are not dealing with extremely large datasets. Each node can have zero coins, which introduces the need to sometimes move coins to a node, and each node can have more than one coin, which introduces surplus coins that need to be moved elsewhere. Edge cases include nodes with zero coins, leaf nodes with surplus coins, and an unbalanced tree structure.

## Approaches

A brute-force approach would attempt to simulate every possible coin transfer, perhaps by performing a BFS or DFS traversal and repeatedly moving coins from nodes with surplus to adjacent nodes with deficits. While correct in principle, this approach is highly inefficient because it does not leverage the structure of the tree, and it would require considering multiple paths for coin redistribution. The complexity grows combinatorially as we try different sequences of moves, which is not feasible even for moderate tree sizes.

The key observation for an optimal solution is that we do not need to simulate individual coin moves. Instead, we can think in terms of **net coin balance** at each node. If a node has more than one coin, the surplus can be passed up to its parent. If it has fewer than one coin, the deficit must be received from its parent. This insight allows us to use a post-order DFS traversal: compute the balance for left and right subtrees first, then calculate the moves required to balance the current node relative to its parent. Each subtree independently contributes its excess or deficit to the parent, and the total moves are the sum of absolute values of these contributions.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(n) | Simulate every possible coin transfer, combinatorial explosion makes it infeasible |
| Optimal | O(n) | O(h) | Use post-order DFS to compute net coin balance at each node; h is tree height |

## Algorithm Walkthrough

1. Define a recursive DFS function that returns the **net balance** of coins for a node. The net balance is the number of coins minus one (the number the node should keep) plus the balances of its left and right children.
2. Recursively traverse the left subtree to compute the left balance. The absolute value of the left balance represents the number of moves needed to balance the left child with the current node.
3. Recursively traverse the right subtree to compute the right balance. Similarly, the absolute value of the right balance represents the number of moves needed to balance the right child with the current node.
4. Add the absolute values of the left and right balances to a global counter tracking total moves. This counts all coin transfers required at this node.
5. Return the net balance of the current node to the parent. This allows the parent to account for surplus or deficit when performing its own calculation.
6. Call the DFS function on the root node. The global move counter at the end contains the minimum number of moves required.

**Why it works**: Each node’s balance is computed independently after accounting for its children. Since moves are counted based on absolute surplus/deficit, every coin transfer between parent and child is counted exactly once. The post-order traversal ensures that all subtrees are balanced before moving coins to the parent, guaranteeing minimal moves.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right
class Solution:
    def distributeCoins(self, root: Optional[TreeNode]) -> int:
        self.moves = 0

        def dfs(node: Optional[TreeNode]) -> int:
            if not node:
                return 0
            left_balance = dfs(node.left)
            right_balance = dfs(node.right)
            self.moves += abs(left_balance) + abs(right_balance)
            return node.val + left_balance + right_balance - 1

        dfs(root)
        return self.moves
```

The Python implementation defines a DFS helper function that computes the balance of coins at each node. The absolute values of left and right balances are summed into `self.moves` to count the moves. The returned balance propagates up to the parent to maintain correctness.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func distributeCoins(root *TreeNode) int {
    moves := 0

    var dfs func(node *TreeNode) int
    dfs = func(node *TreeNode) int {
        if node == nil {
            return 0
        }
        leftBalance := dfs(node.Left)
        rightBalance := dfs(node.Right)
        moves += abs(leftBalance) + abs(rightBalance)
        return node.Val + leftBalance + rightBalance - 1
    }

    dfs(root)
    return moves
}

func abs(x int) int {
    if x < 0 {
        return -x
    }
    return x
}
```

The Go version mirrors the Python logic. A recursive `dfs` function returns the net balance and accumulates moves in the outer `moves` variable. Nil nodes are handled gracefully to avoid errors, and an `abs` helper is used.

## Worked Examples

**Example 1: root = [3,0,0]**

Start DFS at root (3).

| Step | Node | Left Balance | Right Balance | Moves | Return Balance |
| --- | --- | --- | --- | --- | --- |
| 1 | Left child (0) | 0 | 0 | 0 | 0 + 0 + 0 - 1 = -1 |
| 2 | Right child (0) | 0 | 0 | 0 | -1 |
| 3 | Root (3) | -1 | -1 | abs(-1)+abs(-1)=2 | 3 -1 -1 -1 = 0 |

Total moves = 2

**Example 2: root = [0,3,0]**

| Step | Node | Left Balance | Right Balance | Moves | Return Balance |
| --- | --- | --- | --- | --- | --- |
| 1 | Left child (3) | 0 | 0 | 0 | 3 - 1 = 2 |
| 2 | Right child (0) | 0 | 0 | 0 | 0 - 1 = -1 |
| 3 | Root (0) | 2 | -1 | abs(2)+abs(-1)=3 | 0+2-1-1=0 |

Total moves = 3

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each node is visited exactly once in a DFS traversal |
| Space | O(h) | Recursive call stack depth equals the height of the tree, h ≤ n |

Since each node is processed once, and all calculations at a node are O(1), the algorithm is linear in the number of nodes. Space is dominated by recursion stack depth.

## Edge Cases

One important edge case is a single-node tree. Since it already has exactly one coin, no moves are required, and the implementation correctly returns 0.

Another edge case is when all coins are concentrated in a single leaf node. Moves must propagate up through every ancestor to reach nodes that need coins. Our DFS approach handles this naturally because balances propagate from leaves to the root.

A third edge case is a highly unbalanced tree, such as a linked-list-like structure. The recursive DFS stack may grow linearly with the number of nodes, but the algorithm still correctly computes moves because it processes nodes in post-order, ensuring all subtrees are balanced before coins are moved up.
