## Solution

## Problem Understanding

The problem gives us an integer array `nums` and asks us to divide it into two contiguous parts, called `left` and `right`.

The partition must satisfy three conditions:

1. Every element in `left` must be less than or equal to every element in `right`.
2. Both parts must contain at least one element.
3. Among all valid partitions, `left` must be as small as possible.

The output is not the partition itself, but the length of the `left` subarray.

For example, in the array `[5,0,3,8,6]`, if we split after index `2`, then:

- `left = [5,0,3]`
- `right = [8,6]`

The largest value in `left` is `5`, and the smallest value in `right` is `6`, so every value in `left` is less than or equal to every value in `right`. The size of `left` is `3`, so the answer is `3`.

The constraints are important:

- The array length can be as large as `10^5`.
- Values can be as large as `10^6`.

These constraints immediately tell us that an `O(n^2)` solution may be too slow in the worst case. We should aim for a linear or near linear solution.

There are several edge cases worth thinking about upfront.

An array may already be almost sorted, such as `[1,2,3,4]`, where the answer is simply `1` because the first element can stand alone in the left partition.

Another tricky case is when a smaller value appears later and forces the left partition to expand. For example, `[1,1,1,0,6,12]` initially looks partitionable early, but the `0` invalidates earlier splits.

Arrays with repeated values are also important because the condition is `<=`, not strictly `<`. Equal values are allowed across the partition boundary.

The problem guarantees that at least one valid partition always exists, so we never need to handle an impossible case.

## Approaches

### Brute Force Approach

A straightforward solution is to try every possible partition point.

For each split position `i`:

- `left = nums[0:i]`
- `right = nums[i:]`

We compute:

- the maximum value in `left`
- the minimum value in `right`

If:

```
max(left) <= min(right)
```

then the partition is valid, and since we test from left to right, the first valid partition gives the smallest possible left size.

This approach is correct because it explicitly checks the exact condition required by the problem. However, it is inefficient because for every partition point we repeatedly scan subarrays to compute maxima and minima.

If there are `n` positions and each check costs `O(n)`, the total complexity becomes `O(n^2)`.

With `n = 10^5`, this is far too slow.

### Key Insight for the Optimal Solution

The expensive part of the brute force solution is repeatedly recomputing:

- maximum values on the left
- minimum values on the right

We can avoid this repeated work with preprocessing.

Suppose we precompute:

- `left_max[i]` = maximum value from index `0` to `i`
- `right_min[i]` = minimum value from index `i` to the end

Then a partition after index `i` is valid if:

```
left_max[i] <= right_min[i + 1]
```

Now each partition check becomes `O(1)`.

This transforms the overall solution into `O(n)` time.

We scan once to build prefix maximums, once to build suffix minimums, and once more to find the earliest valid split.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Tries every split and rescans subarrays each time |
| Optimal | O(n) | O(n) | Uses prefix maximums and suffix minimums for constant time checks |

## Algorithm Walkthrough

1. Create an array `left_max` of size `n`.

`left_max[i]` will store the largest value seen from the beginning of the array up to index `i`.

This helps us quickly know the maximum value inside the left partition for any split point.
2. Fill the `left_max` array.

Start with:

```
left_max[0] = nums[0]
```

Then for every later index:

```
left_max[i] = max(left_max[i - 1], nums[i])
```

This continuously carries forward the largest value seen so far.
3. Create an array `right_min` of size `n`.

`right_min[i]` will store the smallest value from index `i` to the end of the array.

This allows us to quickly know the minimum value inside the right partition.
4. Fill the `right_min` array from right to left.

Start with:

```
right_min[n - 1] = nums[n - 1]
```

Then move backward:

```
right_min[i] = min(right_min[i + 1], nums[i])
```

This keeps track of the smallest value seen from the current index onward.
5. Scan for the earliest valid partition.

For every index `i` from `0` to `n - 2`:

- Left partition is `nums[0:i+1]`
- Right partition is `nums[i+1:]`

The partition is valid if:

```
left_max[i] <= right_min[i + 1]
```

The first valid index gives the smallest possible left partition.
6. Return `i + 1`.

The answer is the size of the left partition, not the index itself.

### Why it works

The algorithm works because `left_max[i]` precisely captures the largest value in the left partition, while `right_min[i + 1]` precisely captures the smallest value in the right partition.

If the maximum value on the left is less than or equal to the minimum value on the right, then every element in the left partition must be less than or equal to every element in the right partition.

By scanning from left to right and returning the first valid split, we guarantee the smallest possible left partition size.

## Python Solution

```
from typing import List

class Solution:
    def partitionDisjoint(self, nums: List[int]) -> int:
        n = len(nums)

        left_max = [0] * n
        right_min = [0] * n

        left_max[0] = nums[0]
        for i in range(1, n):
            left_max[i] = max(left_max[i - 1], nums[i])

        right_min[n - 1] = nums[n - 1]
        for i in range(n - 2, -1, -1):
            right_min[i] = min(right_min[i + 1], nums[i])

        for i in range(n - 1):
            if left_max[i] <= right_min[i + 1]:
                return i + 1

        return -1
```

The implementation begins by allocating two auxiliary arrays, `left_max` and `right_min`.

The first loop computes the running maximum from left to right. After this loop finishes, `left_max[i]` contains the maximum value in the prefix ending at index `i`.

The second loop computes the running minimum from right to left. After it finishes, `right_min[i]` contains the minimum value in the suffix starting at index `i`.

The final loop tests each possible partition point. For a split after index `i`, we compare:

```
left_max[i]
```

with:

```
right_min[i + 1]
```

If the condition holds, the partition is valid, so we return the size of the left partition, which is `i + 1`.

The problem guarantees a valid answer exists, but the function still includes a fallback `return -1` for completeness.

## Go Solution

```
func partitionDisjoint(nums []int) int {
	n := len(nums)

	leftMax := make([]int, n)
	rightMin := make([]int, n)

	leftMax[0] = nums[0]
	for i := 1; i < n; i++ {
		if nums[i] > leftMax[i-1] {
			leftMax[i] = nums[i]
		} else {
			leftMax[i] = leftMax[i-1]
		}
	}

	rightMin[n-1] = nums[n-1]
	for i := n - 2; i >= 0; i-- {
		if nums[i] < rightMin[i+1] {
			rightMin[i] = nums[i]
		} else {
			rightMin[i] = rightMin[i+1]
		}
	}

	for i := 0; i < n-1; i++ {
		if leftMax[i] <= rightMin[i+1] {
			return i + 1
		}
	}

	return -1
}
```

The Go implementation follows the same logic as the Python solution.

Slices are used instead of arrays because Go slices are dynamically sized and are the standard container for sequence problems in LeetCode.

Unlike Python, Go does not provide built in `max` and `min` functions for integers, so the implementation uses explicit `if` statements to compute running maximums and minimums.

Integer overflow is not a concern because the constraints are well within Go's integer range.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Three linear scans through the array |
| Space | O(n) | Two auxiliary arrays store prefix maximums and suffix minimums |

The algorithm performs a constant amount of work per element while building `left_max`, building `right_min`, and searching for the partition point.

Although there are three separate loops, they are sequential rather than nested, so the total runtime remains linear.

The additional memory usage comes from the two helper arrays, each of size `n`.

## Edge Cases

One important edge case is the smallest valid input size, such as `[1,2]`. Since both partitions must be non empty, the only possible split is after the first element. A buggy implementation might accidentally allow an empty partition or mishandle indexing near array boundaries. The implementation correctly iterates only up to `n - 2`, ensuring the right partition always contains at least one element.

Another important edge case involves repeated values, such as `[2,2,2,2]`. The condition uses `<=`, not `<`. Some incorrect implementations mistakenly require strict inequality and would reject valid partitions containing equal values across the boundary. The algorithm correctly checks:

```
left_max[i] <= right_min[i + 1]
```

so equal values are handled properly.

A third tricky case occurs when a very small value appears later in the array, such as `[1,1,1,0,6,12]`. A naive greedy strategy might attempt to partition too early after seeing several small values at the beginning. However, the later `0` invalidates those earlier partitions. The prefix maximum and suffix minimum approach avoids this mistake because every split is validated against the minimum value in the entire remaining suffix.

Another subtle case is when the partition must occur near the end of the array, such as `[5,4,3,2,6]`. Many earlier splits fail because the right side still contains smaller values. The algorithm systematically checks all possible partition points in order and eventually finds the correct split at the last valid position.
