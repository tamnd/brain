## Solution

## Problem Understanding

This problem models the spread of rot through a grid of oranges over time. We are given a two dimensional matrix `grid`, where each cell represents one of three possible states:

- `0` means the cell is empty.
- `1` means the cell contains a fresh orange.
- `2` means the cell contains a rotten orange.

The key rule is that every minute, a rotten orange infects all fresh oranges directly adjacent to it in the four cardinal directions: up, down, left, and right. Diagonal neighbors do not count.

The goal is to determine the **minimum number of minutes** required until there are no fresh oranges left in the grid. If some fresh oranges can never become rotten because they are isolated from all rotten oranges, we return `-1`.

Another way to think about the problem is as a time based spreading process. Every rotten orange acts like a source of infection, and the infection expands outward simultaneously each minute.

The input consists of an `m x n` matrix where both dimensions are at most `10`. This means the total number of cells is at most `100`, which is relatively small. Even though brute force solutions may still pass due to the small constraint, we should still aim for the most efficient and clean solution.

There are several important edge cases to consider upfront.

If there are **no fresh oranges initially**, the answer is immediately `0`, because nothing needs to rot.

If there are **fresh oranges but no rotten oranges**, then rotting can never begin, so the answer must be `-1`.

Another important case occurs when some fresh oranges are completely blocked off by empty spaces. Since rot only spreads through adjacent cells, those isolated oranges will never rot, making the answer `-1`.

Finally, because rotting happens **simultaneously**, we must be careful not to process newly rotten oranges immediately in the same minute. A naive simulation can accidentally allow chain reactions inside a single time step, which would produce incorrect results.

## Approaches

### Brute Force Approach

A straightforward approach is to simulate the grid minute by minute.

At each minute, we scan the entire matrix and locate all currently rotten oranges. For every rotten orange, we mark adjacent fresh oranges to become rotten in the next round. Since changes should happen simultaneously, we cannot modify the grid immediately during the scan. Instead, we would first record which oranges should rot, then update them after the scan finishes.

We repeat this process until either:

1. No fresh oranges remain, in which case we return the elapsed time.
2. No new oranges rot during a round while fresh oranges still exist, meaning some fresh oranges are unreachable, so we return `-1`.

This method works because it correctly simulates the problem statement minute by minute. However, it repeatedly scans the entire grid even when only a few oranges are actively spreading rot. That repeated work makes it inefficient.

### Optimal Approach

The key observation is that this problem is naturally modeled as a **Breadth First Search (BFS)**.

Each rotten orange acts as a starting point for infection, and the infection spreads level by level through the grid. Since all rotten oranges spread simultaneously every minute, we should start BFS from **all rotten oranges at once**, not just one.

This is called **multi source BFS**.

The intuition is simple:

- Minute `0`: all initially rotten oranges are active.
- Minute `1`: oranges adjacent to them become rotten.
- Minute `2`: oranges adjacent to those newly rotten oranges become rotten.
- And so on.

BFS is perfect because each layer of traversal naturally represents one minute of elapsed time.

We initialize a queue with all rotten oranges and count the number of fresh oranges. Then, during BFS, each layer processes all oranges that are rotten at the same time step. Whenever a fresh orange becomes rotten, we reduce the fresh count and add it to the queue for the next minute.

At the end:

- If all fresh oranges became rotten, return the total minutes.
- Otherwise, return `-1`.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O((m × n)²) | O(m × n) | Repeatedly scans the whole grid each minute |
| Optimal | O(m × n) | O(m × n) | Multi source BFS processes each cell once |

## Algorithm Walkthrough

1. Traverse the entire grid once to gather initial information.

During this scan, count the number of fresh oranges and place every rotten orange into a queue. The queue is essential because BFS processes elements in the order they spread infection.
2. Handle the trivial case where no fresh oranges exist.

If the fresh orange count is already zero, return `0` immediately. No time needs to pass because the grid is already complete.
3. Define the four possible movement directions.

Since rot spreads only vertically and horizontally, we store the directional offsets for up, down, left, and right.
4. Begin the BFS process.

While the queue is not empty and fresh oranges still remain, process one BFS level at a time. Each level represents one minute.
5. Process all rotten oranges for the current minute.

Before processing, record the current queue size. This ensures we only process oranges that were rotten at the beginning of the minute.
6. Attempt to infect neighboring cells.

For each rotten orange, check its four neighbors:

- Skip cells outside the grid.
- Ignore empty cells.
- Ignore already rotten cells.
- If a neighboring cell contains a fresh orange, convert it to rotten, decrement the fresh count, and add it to the queue.
7. Increment the elapsed time.

After finishing one BFS layer, increase the minute counter by one. This accurately reflects one full minute of simultaneous spreading.
8. Finish processing.

After BFS ends, check whether fresh oranges remain.

- If `fresh_count == 0`, return the total minutes.
- Otherwise, return `-1`.

### Why it works

The correctness comes from the BFS level order property. Every layer of BFS represents all oranges that become rotten at the same minute. Because we process the queue level by level, newly rotten oranges only begin spreading during the next iteration, exactly matching the problem's simultaneous spreading rule. Since BFS explores cells in increasing order of distance from the initial rotten oranges, the first time an orange becomes rotten is guaranteed to be the minimum possible minute.

## Python Solution

```
from collections import deque
from typing import List

class Solution:
    def orangesRotting(self, grid: List[List[int]]) -> int:
        rows, cols = len(grid), len(grid[0])
        queue = deque()
        fresh_count = 0

        # Collect all rotten oranges and count fresh ones
        for row in range(rows):
            for col in range(cols):
                if grid[row][col] == 2:
                    queue.append((row, col))
                elif grid[row][col] == 1:
                    fresh_count += 1

        # No fresh oranges to rot
        if fresh_count == 0:
            return 0

        directions = [(1, 0), (-1, 0), (0, 1), (0, -1)]
        minutes = 0

        # Multi-source BFS
        while queue and fresh_count > 0:
            level_size = len(queue)

            for _ in range(level_size):
                row, col = queue.popleft()

                for row_offset, col_offset in directions:
                    new_row = row + row_offset
                    new_col = col + col_offset

                    # Check bounds and fresh orange
                    if (
                        0 <= new_row < rows
                        and 0 <= new_col < cols
                        and grid[new_row][new_col] == 1
                    ):
                        grid[new_row][new_col] = 2
                        fresh_count -= 1
                        queue.append((new_row, new_col))

            minutes += 1

        return minutes if fresh_count == 0 else -1
```

The implementation begins by scanning the grid once. During this pass, every rotten orange is added to the BFS queue, while fresh oranges are counted. This preprocessing step avoids rescanning the matrix later.

The early return for `fresh_count == 0` handles cases where the answer is immediately known.

The `directions` list makes neighbor traversal cleaner and avoids repetitive logic. Instead of writing separate conditions for up, down, left, and right, we loop through directional offsets.

The BFS loop runs level by level. The variable `level_size` ensures that only oranges rotten at the start of the current minute are processed. This preserves the simultaneous spreading behavior required by the problem.

Whenever a fresh orange is found, it is immediately marked rotten to prevent duplicate processing, the fresh counter decreases, and the orange is added to the queue for future spreading.

Finally, after BFS finishes, we determine whether every fresh orange was infected. If not, at least one orange was unreachable.

## Go Solution

```
func orangesRotting(grid [][]int) int {
    rows, cols := len(grid), len(grid[0])

    type Cell struct {
        row int
        col int
    }

    queue := []Cell{}
    freshCount := 0

    // Collect rotten oranges and count fresh ones
    for row := 0; row < rows; row++ {
        for col := 0; col < cols; col++ {
            if grid[row][col] == 2 {
                queue = append(queue, Cell{row, col})
            } else if grid[row][col] == 1 {
                freshCount++
            }
        }
    }

    // No fresh oranges
    if freshCount == 0 {
        return 0
    }

    directions := [][]int{
        {1, 0},
        {-1, 0},
        {0, 1},
        {0, -1},
    }

    minutes := 0
    front := 0

    // Multi-source BFS
    for front < len(queue) && freshCount > 0 {
        levelSize := len(queue) - front

        for i := 0; i < levelSize; i++ {
            current := queue[front]
            front++

            for _, direction := range directions {
                newRow := current.row + direction[0]
                newCol := current.col + direction[1]

                if newRow >= 0 &&
                    newRow < rows &&
                    newCol >= 0 &&
                    newCol < cols &&
                    grid[newRow][newCol] == 1 {

                    grid[newRow][newCol] = 2
                    freshCount--
                    queue = append(queue, Cell{newRow, newCol})
                }
            }
        }

        minutes++
    }

    if freshCount == 0 {
        return minutes
    }

    return -1
}
```

The Go implementation follows the same BFS logic but uses a custom `Cell` struct instead of tuples. Since Go slices do not support efficient popping from the front, we use a `front` index to simulate queue behavior in O(1) time.

Unlike Python, Go does not have a built in deque in the standard library, so slice indexing is the cleanest approach for queue management here.

Integer overflow is not a concern because the grid size is extremely small, at most `10 × 10`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m × n) | Each cell is visited at most once |
| Space | O(m × n) | Queue can hold up to all cells |

The algorithm scans the matrix once during initialization and processes each orange at most one time during BFS. Once an orange becomes rotten, it never enters the queue again. Therefore, total work is proportional to the number of grid cells.

The queue may contain every cell in the worst case, so the auxiliary space complexity is `O(m × n)`.

## Edge Cases

One important edge case occurs when the grid contains **no fresh oranges initially**. A buggy implementation might still start BFS and increment minutes unnecessarily. For example, `[[0,2]]` should immediately return `0`. Our implementation explicitly checks `fresh_count == 0` before starting BFS.

Another tricky case happens when **fresh oranges exist but there are no rotten oranges**. Since infection has no starting point, rotting can never begin. For example, `[[1]]` must return `-1`. In our solution, the queue starts empty, so BFS never runs, and the remaining fresh count correctly causes a `-1` return.

A third important edge case involves **isolated fresh oranges** blocked by empty cells. Since movement is only four directional and cannot pass through `0`, some oranges may be unreachable. For example, `[[2,0,1]]` leaves the rightmost orange permanently fresh. Our BFS naturally handles this because unreachable oranges never enter the queue, leaving `fresh_count > 0` at the end.

A subtle but critical edge case concerns **simultaneous spreading**. Newly rotten oranges must not infect others in the same minute. If we processed the queue continuously without separating BFS levels, the infection could spread too quickly. Using `level_size` guarantees that oranges infected during the current minute only spread infection during the next minute, matching the problem specification exactly.
