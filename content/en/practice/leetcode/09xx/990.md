## Solution

## Problem Understanding

The problem gives us a list of equations involving lowercase English letters. Each equation has one of two forms:

```
"a==b"
"a!=b"
```

The equality equation means the two variables must have the same value. The inequality equation means the two variables must have different values.

Our task is to determine whether there exists some assignment of integer values to all variables such that every equation is satisfied simultaneously.

The important detail is that we are not required to actually produce the values. We only need to determine whether such an assignment is possible.

For example:

```
["a==b","b!=a"]
```

The first equation requires `a` and `b` to be equal, while the second requires them to be different. Both conditions cannot be true at the same time, so the answer is `false`.

The variables are restricted to lowercase English letters, which means there are only 26 possible variables from `a` to `z`. This is a very small fixed universe, which strongly suggests that grouping or connectivity techniques can work efficiently.

The constraints are:

- At most 500 equations
- Each equation has exactly length 4
- Variables are lowercase letters

Even though the number of equations can be relatively large, the number of unique variables is tiny. This means we can use algorithms that repeatedly merge or compare groups without worrying about scalability issues.

Several edge cases are important:

- A variable may compare against itself, such as `"a!=a"`, which is immediately impossible.
- Multiple equality chains may exist, such as `"a==b"` and `"b==c"`, which implies `"a==c"` even if not explicitly stated.
- Equations may appear in any order. An inequality might appear before the equalities that eventually create a contradiction.
- Duplicate equations should not cause issues.

## Approaches

### Brute Force Approach

A brute force approach would attempt to assign values to variables and verify whether all equations are satisfied.

Since there are 26 variables, we could theoretically try assigning integers to them and checking every possible configuration. However, even with a small number of variables, the number of assignments grows exponentially. This becomes infeasible very quickly.

Another brute force style approach would repeatedly propagate equality relationships manually. For every equality, we could update groups and repeatedly scan the equations until no changes occur. Then we would verify inequalities afterward. While this works conceptually, it is inefficient and unnecessarily complicated.

The key challenge is that equality relationships are transitive:

```
a == b
b == c
```

implies:

```
a == c
```

Handling these transitive relationships efficiently is exactly what the Union-Find data structure is designed for.

### Optimal Approach, Union-Find

The key observation is that equality equations form connected groups.

If:

```
a == b
b == c
```

then all three variables belong to the same connected component.

Union-Find, also called Disjoint Set Union (DSU), efficiently supports two operations:

- Merge two variables into the same group
- Determine whether two variables belong to the same group

The strategy is:

1. Process all equality equations first and merge their variables.
2. Process all inequality equations afterward.
3. If an inequality connects two variables already in the same group, the equations are impossible to satisfy.

This works because equality constraints define the groups first, and inequalities simply verify that conflicting variables are not grouped together.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^26) or worse | O(26) | Attempts assignments or repeated propagation |
| Optimal | O(n * α(26)) | O(26) | Uses Union-Find to group equal variables efficiently |

Here, `α` is the inverse Ackermann function, which grows so slowly that it is effectively constant in practice.

## Algorithm Walkthrough

1. Initialize a Union-Find structure for all 26 lowercase letters.

Each variable initially belongs to its own group. For example:

```
parent[a] = a
parent[b] = b
...
```
2. Process all equality equations first.

For every equation of the form:

```
x==y
```

merge the sets containing `x` and `y`.

This step builds connected components representing variables that must share the same value.
3. Process all inequality equations.

For every equation of the form:

```
x!=y
```

check whether `x` and `y` belong to the same set.

- If they do, we have a contradiction because the same group cannot simultaneously be equal and unequal.
- If they do not, the inequality is valid.
4. If no contradiction is found, return `true`.

This means all equalities and inequalities can coexist consistently.

### Why it works

Union-Find maintains the invariant that all variables connected by equality relationships belong to the same set.

After processing all equalities, every set represents variables that must share identical values. Therefore, any inequality between variables in the same set is impossible to satisfy.

If no such contradiction exists, we can assign a unique value to each connected component, guaranteeing that all equations are satisfied.

## Python Solution

```
from typing import List

class Solution:
    def equationsPossible(self, equations: List[str]) -> bool:
        parent = list(range(26))

        def find(x: int) -> int:
            if parent[x] != x:
                parent[x] = find(parent[x])
            return parent[x]

        def union(x: int, y: int) -> None:
            root_x = find(x)
            root_y = find(y)

            if root_x != root_y:
                parent[root_x] = root_y

        # First pass: process equality equations
        for equation in equations:
            if equation[1:3] == "==":
                left = ord(equation[0]) - ord('a')
                right = ord(equation[3]) - ord('a')
                union(left, right)

        # Second pass: process inequality equations
        for equation in equations:
            if equation[1:3] == "!=":
                left = ord(equation[0]) - ord('a')
                right = ord(equation[3]) - ord('a')

                if find(left) == find(right):
                    return False

        return True
```

The implementation begins by creating a `parent` array of size 26, one entry for each lowercase letter.

The `find` function determines the representative of a variable's set. It uses path compression, which flattens the structure of the tree during lookups and makes future operations faster.

The `union` function merges two sets by connecting their roots.

The algorithm then performs two passes through the equations:

- The first pass processes all equality equations and merges variables into groups.
- The second pass checks inequality equations for contradictions.

If an inequality connects two variables with the same root, the equations are unsatisfiable and the function immediately returns `False`.

Otherwise, after all checks pass, the function returns `True`.

## Go Solution

```
func equationsPossible(equations []string) bool {
	parent := make([]int, 26)

	for i := 0; i < 26; i++ {
		parent[i] = i
	}

	var find func(int) int

	find = func(x int) int {
		if parent[x] != x {
			parent[x] = find(parent[x])
		}
		return parent[x]
	}

	union := func(x int, y int) {
		rootX := find(x)
		rootY := find(y)

		if rootX != rootY {
			parent[rootX] = rootY
		}
	}

	// First pass: process equalities
	for _, equation := range equations {
		if equation[1:3] == "==" {
			left := int(equation[0] - 'a')
			right := int(equation[3] - 'a')

			union(left, right)
		}
	}

	// Second pass: process inequalities
	for _, equation := range equations {
		if equation[1:3] == "!=" {
			left := int(equation[0] - 'a')
			right := int(equation[3] - 'a')

			if find(left) == find(right) {
				return false
			}
		}
	}

	return true
}
```

The Go implementation follows the same logic as the Python version.

One notable difference is that Go does not support nested named functions in the same way Python does, so the `find` function is declared as a variable holding an anonymous recursive function.

The parent structure is implemented using a fixed-size integer slice of length 26.

Go strings are byte-indexable, so extracting characters like `equation[0]` works naturally for lowercase ASCII letters.

Since the variable count is tiny, there are no concerns about recursion depth or memory usage.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n * α(26)) | Each union/find operation is nearly constant time |
| Space | O(26) | Parent array stores one entry per variable |

The algorithm processes each equation at most twice, once during equality processing and once during inequality checking.

Union-Find with path compression makes each operation extremely efficient. Technically, the complexity is amortized `O(α(n))`, where `α` is the inverse Ackermann function. Since there are only 26 variables, this behaves like constant time in practice.

The space usage is constant because the number of variables never exceeds 26.

## Edge Cases

One important edge case is self-inequality:

```
["a!=a"]
```

A variable can never be unequal to itself. This is an immediate contradiction. In the implementation, both sides map to the same root, so the inequality check correctly returns `False`.

Another important edge case is transitive equality relationships:

```
["a==b", "b==c", "a!=c"]
```

A naive implementation that only compares direct equations might miss the contradiction. Union-Find correctly groups all connected variables into the same component, so the final inequality check detects the issue.

A third edge case is when inequalities appear before equalities:

```
["a!=c", "a==b", "b==c"]
```

If processed sequentially in a naive way, the contradiction might be missed because the equality chain is not fully known yet. The two-pass approach avoids this problem entirely by processing all equalities first before validating inequalities.

Another subtle edge case involves duplicate or redundant equations:

```
["a==b", "a==b", "b==a"]
```

The implementation handles this naturally because repeated union operations on already-connected variables have no negative effect.
