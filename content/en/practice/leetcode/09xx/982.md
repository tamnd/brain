## Solution

## Problem Understanding

The problem asks us to count how many ordered triples of indices `(i, j, k)` satisfy:

$nums[i] \& nums[j] \& nums[k] = 0$

The array `nums` contains non-negative integers, and the bitwise AND operation must evaluate to zero across all three selected elements.

An important detail is that the indices are ordered and can repeat. This means `(0,1,2)` and `(2,1,0)` are considered different triples, and `(0,0,0)` is also allowed. We are not selecting combinations, we are counting every valid ordered triple.

The input size is up to `1000`, and every value is smaller than `2^16`. The upper bound on the values is extremely important because it means every number fits into exactly 16 bits. That small bit-width creates opportunities for optimization using bit manipulation techniques.

A naive implementation would examine all possible triples. Since there are `n^3` triples and `n` can be `1000`, that would require up to:

$1000^3 = 10^9$

operations, which is far too slow.

Several edge cases matter immediately:

- Arrays containing many zeros, because zero AND anything is zero
- Arrays where all numbers share common bits, causing very few valid triples
- Arrays with repeated values, because duplicates dramatically increase the number of valid ordered triples
- Arrays of size `1`, where only a single triple exists

The constraints guarantee that all numbers are valid non-negative integers and fit within 16 bits, which enables a much faster solution than brute force.

## Approaches

### Brute Force Approach

The most direct solution is to iterate through every possible triple `(i, j, k)` and compute:

$nums[i] \& nums[j] \& nums[k]$

If the result equals zero, we increment the answer.

This approach is correct because it explicitly checks every possible ordered triple. Nothing is skipped, so every valid triple is counted exactly once.

However, the runtime is cubic. With `n = 1000`, we would perform up to one billion iterations, which is too slow for LeetCode constraints.

### Optimal Approach

The key observation is that bitwise AND is associative:

$(a \& b) \& c = a \& (b \& c)$

Instead of evaluating all triples directly, we can first precompute all pairwise AND results.

Suppose we compute every value:

$nums[i] \& nums[j]$

and store how many times each result occurs.

Then, for every number `x = nums[k]`, we only need to determine how many pairwise AND results satisfy:

$pairAND \& x = 0$

This transforms the problem from iterating over triples to:

1. Precompute all pairwise AND frequencies
2. Match each number against those frequencies

Since there are at most `2^16 = 65536` distinct AND values, the search space becomes manageable.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n³) | O(1) | Checks every ordered triple directly |
| Optimal | O(n² + n × 2¹⁶) | O(2¹⁶) | Precomputes pairwise AND frequencies |

## Algorithm Walkthrough

1. Create a frequency array called `pair_count` of size `65536`.

Since every number fits within 16 bits, every possible AND result also lies in the range `[0, 65535]`. We use an array instead of a hash map because array indexing is faster and the range is small and fixed.
2. Compute all pairwise AND results.

For every pair `(a, b)` in `nums`, calculate:

$a \& b$

Increment the frequency of that result inside `pair_count`.

After this step, `pair_count[v]` tells us how many ordered pairs produce AND result `v`.
3. Iterate through every number `x` in `nums`.

We now want to count how many pairwise results satisfy:

$v \& x = 0$
4. Check every possible AND value.

Iterate through all integers `v` from `0` to `65535`.

If:

$v \& x = 0$

then every pair contributing to `pair_count[v]` forms a valid triple with `x`.
5. Add those frequencies to the answer.

Accumulate:

$answer += pair\_count[v]$
6. Return the final answer.

After processing all numbers, the accumulated total equals the number of valid ordered triples.

### Why it works

The algorithm works because every valid triple can be decomposed into:

1. A pairwise AND result
2. A third number compatible with that result

For every ordered pair `(i, j)`, we store:

$nums[i] \& nums[j]$

Then for each `k`, we verify whether:

$(nums[i] \& nums[j]) \& nums[k] = 0$

By associativity of bitwise AND, this exactly matches the original condition. Every ordered triple is counted once and only once.

## Python Solution

```
from typing import List

class Solution:
    def countTriplets(self, nums: List[int]) -> int:
        MAX_MASK = 1 << 16

        pair_count = [0] * MAX_MASK

        # Count all pairwise AND results
        for a in nums:
            for b in nums:
                pair_count[a & b] += 1

        answer = 0

        # For each number, find compatible pairwise AND values
        for num in nums:
            for mask in range(MAX_MASK):
                if (mask & num) == 0:
                    answer += pair_count[mask]

        return answer
```

The implementation follows the algorithm directly.

The `pair_count` array stores how many ordered pairs produce each possible AND value. Since all numbers fit within 16 bits, we allocate an array of size `2^16`.

The nested loop over `nums` computes every ordered pair `(a, b)` and increments the count for `a & b`.

After preprocessing, we iterate through each number in the array and test every possible AND result. Whenever a mask satisfies `(mask & num) == 0`, it means that combining the pair producing `mask` with `num` creates a valid triple.

The final accumulated value is returned as the answer.

## Go Solution

```
func countTriplets(nums []int) int {
	const MAX_MASK = 1 << 16

	pairCount := make([]int, MAX_MASK)

	// Count all pairwise AND results
	for _, a := range nums {
		for _, b := range nums {
			pairCount[a&b]++
		}
	}

	answer := 0

	// Count valid triples
	for _, num := range nums {
		for mask := 0; mask < MAX_MASK; mask++ {
			if (mask & num) == 0 {
				answer += pairCount[mask]
			}
		}
	}

	return answer
}
```

The Go implementation is nearly identical to the Python version.

Go uses a fixed-size integer slice for `pairCount`, which provides efficient indexed access. Since the answer can grow up to `1000^3 = 10^9`, a standard Go `int` is sufficient on modern systems.

Unlike Python, Go requires explicit slice allocation using `make`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n² + n × 2¹⁶) | Pair preprocessing plus mask checking |
| Space | O(2¹⁶) | Frequency array for all possible AND results |

The preprocessing step computes all pairwise AND results in `O(n²)` time. The second phase iterates over every number and every possible 16-bit mask, giving `O(n × 65536)` complexity.

Since `65536` is a fixed upper bound, the algorithm is efficient enough for `n = 1000`.

The space complexity comes entirely from the frequency array storing counts for all possible 16-bit values.

## Edge Cases

One important edge case is an array containing only zeros. Since:

$0 \& x = 0$

for any `x`, every ordered triple is valid. A buggy implementation might accidentally treat repeated indices incorrectly or undercount permutations. The current solution correctly counts every ordered pair and every ordered third element, producing exactly `n³`.

Another critical edge case is arrays where all numbers share at least one common bit. For example:

```
[7,7,7]
```

Every AND result remains nonzero, so the answer should be zero. The implementation naturally handles this because no pairwise mask satisfies the compatibility condition.

A third important edge case involves the maximum possible value:

```
65535
```

which corresponds to all 16 bits being set. Since the problem guarantees all numbers fit within 16 bits, our frequency array size of `65536` safely covers every possible AND result. This prevents out-of-bounds access and guarantees correctness for the entire input range.

A final subtle edge case is repeated values. Since triples are ordered and indices may repeat, duplicates dramatically increase the count. The preprocessing step stores frequencies of pairwise AND values rather than unique pairs, ensuring every ordered combination is counted correctly.
