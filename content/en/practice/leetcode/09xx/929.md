## Solution

## Problem Understanding

The problem asks us to determine how many unique email addresses actually receive mail after applying specific normalization rules to the local part of each email address.

Every email address contains two parts separated by the `'@'` symbol:

- The local name, which appears before `'@'`
- The domain name, which appears after `'@'`

The domain name always stays unchanged. The local name, however, follows two transformation rules:

1. All periods `'.'` inside the local name are ignored.
2. If a plus sign `'+'` appears in the local name, everything after the first `'+'` is ignored.

These rules apply only to the local name. The domain must remain exactly the same.

For example:

- `"test.email+alex@leetcode.com"` becomes `"testemail@leetcode.com"`
- `"m.y+name@email.com"` becomes `"my@email.com"`

The input is an array of email strings. We send one email to each address in the array, and we must compute how many distinct normalized addresses actually receive mail.

The constraints are relatively small:

- At most 100 emails
- Each email length is at most 100 characters

These limits mean that even moderately inefficient approaches would still pass, but the problem is fundamentally a string-processing and deduplication problem, so we should still aim for a clean linear-time solution.

Several edge cases are important:

- Emails may contain multiple periods in the local name.
- Emails may contain a plus sign with additional characters afterward.
- Some emails may normalize to exactly the same address.
- The domain name must never be modified.
- The input guarantees exactly one `'@'` symbol.
- Local names never start with `'+'`, so we do not need to handle malformed cases.

A naive implementation can easily make mistakes by applying the dot-removal rule to the domain name or by ignoring characters after every `'+'` instead of only the first one.

## Approaches

### Brute Force Approach

A brute-force solution would normalize every email and then compare it manually against all previously processed normalized emails to determine whether it is unique.

For each email:

1. Split the address into local and domain parts.
2. Remove everything after the first `'+'` in the local part.
3. Remove all periods from the remaining local part.
4. Reconstruct the normalized email.
5. Compare it against a list of already processed normalized emails.

This approach is correct because every email is transformed according to the exact problem rules before comparison. However, checking uniqueness using repeated linear searches makes the solution inefficient.

If there are `n` emails and each normalized email comparison costs up to `O(n)`, the total runtime becomes quadratic.

### Optimal Approach

The key insight is that we only care about distinct normalized email addresses. This naturally suggests using a hash set.

A hash set provides average-case `O(1)` insertion and lookup. Instead of manually checking whether a normalized email already exists, we can simply insert each normalized email into the set. Duplicate addresses automatically collapse into one entry.

The algorithm becomes:

1. Normalize each email.
2. Insert the normalized result into a hash set.
3. Return the size of the set.

Because each email is processed once and each character is examined a constant number of times, the solution runs efficiently in linear time relative to the total input size.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n² × m) | O(n × m) | Normalizes emails but uses repeated linear comparisons |
| Optimal | O(n × m) | O(n × m) | Uses a hash set for constant-time uniqueness checks |

Here:

- `n` is the number of emails
- `m` is the average email length

## Algorithm Walkthrough

1. Create an empty hash set to store normalized email addresses.

We use a hash set because we only care about unique addresses. Sets automatically eliminate duplicates and provide efficient insertion and lookup.
2. Iterate through each email in the input array.

Each email must be processed independently according to the normalization rules.
3. Split the email into local and domain parts using `'@'`.

The problem guarantees exactly one `'@'`, so the split operation is always valid.
4. Process the local name.

First, check whether the local name contains a `'+'`. If it does, keep only the substring before the first `'+'`.

This works because the problem states that everything after the first plus sign is ignored.
5. Remove all periods from the local name.

Periods do not affect delivery in the local name, so they must be removed entirely.
6. Reconstruct the normalized email.

Combine the processed local name with the unchanged domain using `'@'`.
7. Insert the normalized email into the hash set.

If the email already exists, the set ignores the duplicate automatically.
8. After processing all emails, return the size of the hash set.

The set size represents the number of distinct addresses that actually receive mail.

### Why it works

The algorithm works because every email is transformed into its canonical form according to the exact rules given in the problem statement. Two emails that normalize to the same canonical string must represent the same mailbox, while emails that normalize differently must represent different mailboxes. Since the hash set stores only unique canonical forms, its final size is exactly the number of distinct recipients.

## Python Solution

```
from typing import List

class Solution:
    def numUniqueEmails(self, emails: List[str]) -> int:
        unique_emails = set()

        for email in emails:
            local_name, domain_name = email.split("@")

            if "+" in local_name:
                local_name = local_name.split("+")[0]

            local_name = local_name.replace(".", "")

            normalized_email = local_name + "@" + domain_name

            unique_emails.add(normalized_email)

        return len(unique_emails)
```

The solution begins by creating a set called `unique_emails`. This set stores all normalized email addresses.

For each email, the code splits the string into `local_name` and `domain_name` using the `'@'` separator. Because the input guarantees exactly one `'@'`, this operation is safe.

Next, the code checks whether the local name contains a `'+'`. If it does, only the portion before the plus sign is kept. This matches the problem rule that everything after the first plus sign is ignored.

After handling the plus sign, all periods are removed from the local name using `replace(".", "")`.

The normalized email is then reconstructed by concatenating the cleaned local name with the original domain name.

Finally, the normalized email is inserted into the set. Since sets automatically remove duplicates, only unique normalized addresses remain. The method returns the size of the set.

## Go Solution

```
package main

import "strings"

func numUniqueEmails(emails []string) int {
	uniqueEmails := make(map[string]bool)

	for _, email := range emails {
		parts := strings.Split(email, "@")

		localName := parts[0]
		domainName := parts[1]

		if plusIndex := strings.Index(localName, "+"); plusIndex != -1 {
			localName = localName[:plusIndex]
		}

		localName = strings.ReplaceAll(localName, ".", "")

		normalizedEmail := localName + "@" + domainName

		uniqueEmails[normalizedEmail] = true
	}

	return len(uniqueEmails)
}
```

The Go implementation follows the same logic as the Python version but uses Go-specific standard library functions.

Instead of a set, Go uses a `map[string]bool` to simulate set behavior. The keys of the map represent unique normalized emails.

`strings.Split` separates the email into local and domain parts. `strings.Index` locates the first plus sign, and substring slicing removes everything afterward. `strings.ReplaceAll` removes periods from the local name.

Go does not require any special handling for integer overflow because the result is bounded by the number of input emails, which is at most 100.

## Edge Cases

One important edge case occurs when multiple emails differ only by periods in the local name. For example, `"a.b.c@leetcode.com"` and `"abc@leetcode.com"` should normalize to the same address. A buggy implementation might forget to remove all periods or remove periods only once. The solution correctly handles this by using `replace(".", "")`, which removes every period from the local name.

Another important edge case involves plus signs. Emails such as `"alex+work@leetcode.com"` and `"alex+home@leetcode.com"` should both normalize to `"alex@leetcode.com"`. A common mistake is processing characters after every plus sign instead of stopping at the first one. The implementation correctly uses the substring before the first `'+'`, ensuring the rule is applied exactly as specified.

A third important edge case is ensuring that the domain name remains untouched. For example, `"test.email@leetcode.com"` and `"test.email@lee.tcode.com"` must remain distinct because the domains differ. A flawed solution might accidentally apply period removal to the entire email address. This implementation explicitly separates local and domain parts before processing, guaranteeing that only the local name is modified.
