## Solution

## Problem Understanding

The problem gives us an array of strings where every string has the same length. If we place these strings vertically, character by character, they form a rectangular grid. Each column contains one character from every string.

Our task is to determine how many columns are not sorted in lexicographical order from top to bottom. A column is considered sorted if every character is less than or equal to the character directly below it.

For example, suppose we have:

```
abc
bce
cae
```

Column `0` contains `a, b, c`, which is sorted. Column `1` contains `b, c, a`, which is not sorted because `a` comes after `c`. Column `2` contains `c, e, e`, which is sorted because equal characters are allowed.

The input consists of:

- `strs`, an array of strings
- All strings have equal length
- Only lowercase English letters are used

The output is a single integer representing the number of columns that must be deleted because they are not sorted.

The constraints are relatively small:

- Up to `100` strings
- Each string length can be up to `1000`

This tells us that an algorithm with time complexity around `O(rows × cols)` is completely acceptable.

There are several important edge cases to consider:

- A single row is always sorted, because there is nothing below any character.
- A single column still needs comparison between rows.
- Equal adjacent characters are valid and should not count as unsorted.
- Every column may already be sorted, resulting in `0`.
- Every column may be unsorted, resulting in the full column count.

The problem also guarantees that all strings have the same length, so we never need to worry about index mismatches while traversing columns.

## Approaches

### Brute Force Approach

The brute force idea is straightforward. For every column, we collect all characters in that column into a temporary list or string, then check whether that list is sorted.

For example:

```
cba
daf
ghi
```

For column `1`, we build:

```
["b", "a", "h"]
```

We then compare adjacent characters one by one to determine whether the sequence is sorted.

This approach is correct because a column is sorted if and only if every adjacent pair satisfies:

```
current <= next
```

The drawback is that constructing temporary column arrays creates unnecessary extra work and memory usage.

### Optimal Approach

The key observation is that we never actually need to build the columns explicitly.

Instead, we can iterate column by column and directly compare characters between adjacent rows.

For each column:

- Compare `strs[row][col]` with `strs[row + 1][col]`
- If any comparison violates sorted order, increment the answer
- Stop checking that column immediately, because we already know it must be deleted

This avoids creating temporary structures and performs only the comparisons we truly need.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(rows × cols) | O(rows) | Builds temporary column arrays |
| Optimal | O(rows × cols) | O(1) | Directly compares adjacent rows |

## Algorithm Walkthrough

1. Determine the number of rows and columns from the input array.
2. Initialize a counter called `deletions` to `0`. This will store how many columns are not sorted.
3. Iterate through each column index from left to right.
4. For the current column, iterate through adjacent row pairs:

- Compare the character in the current row with the character in the next row.
- If the current character is greater than the next character, the column is not sorted.
5. Once an unsorted pair is found:

- Increment `deletions`
- Stop checking the current column immediately using `break`

Continuing further would waste time because the column already needs deletion.
6. After all columns have been processed, return `deletions`.

### Why it works

A column is sorted exactly when every adjacent pair of characters satisfies nondecreasing order. If even one adjacent pair violates this condition, the entire column becomes invalid and must be deleted. By checking all adjacent row pairs for every column, the algorithm precisely identifies every unsorted column and counts it exactly once.

## Python Solution

```
from typing import List

class Solution:
    def minDeletionSize(self, strs: List[str]) -> int:
        rows = len(strs)
        cols = len(strs[0])

        deletions = 0

        for col in range(cols):
            for row in range(rows - 1):
                if strs[row][col] > strs[row + 1][col]:
                    deletions += 1
                    break

        return deletions
```

The implementation first determines the number of rows and columns. Since all strings have equal length, the number of columns is simply the length of the first string.

The outer loop iterates through each column. The inner loop compares adjacent rows within that column.

If the current character is greater than the next character, the column violates lexicographical order. We increment the deletion counter and immediately stop checking that column using `break`.

Finally, after all columns are processed, the function returns the total number of invalid columns.

## Go Solution

```
func minDeletionSize(strs []string) int {
	rows := len(strs)
	cols := len(strs[0])

	deletions := 0

	for col := 0; col < cols; col++ {
		for row := 0; row < rows-1; row++ {
			if strs[row][col] > strs[row+1][col] {
				deletions++
				break
			}
		}
	}

	return deletions
}
```

The Go implementation follows the exact same logic as the Python solution.

One important Go detail is that strings are indexed as bytes. Since the input only contains lowercase English letters, byte indexing is perfectly safe and efficient.

No additional slices or arrays are needed, so the solution maintains constant extra space usage.

## Edge Cases

A single row is an important edge case because no vertical comparisons exist. Every column is automatically sorted. A buggy implementation might incorrectly assume at least two rows exist and attempt invalid comparisons. This solution handles it naturally because the inner loop runs `rows - 1` times, which becomes zero iterations when there is only one row.

Equal adjacent characters are another important case. Lexicographical sorting allows equality, so comparisons must use `>` instead of `>=`. For example, in `["aaa", "aaa"]`, every column is valid. Using the wrong comparison operator would incorrectly delete all columns.

A fully descending grid is another common source of mistakes. In inputs like `["zyx","wvu","tsr"]`, every column fails immediately at the first comparison. The implementation correctly increments the deletion counter once per column and stops checking further rows using `break`, which avoids unnecessary work.

A single column input also deserves attention. Even though there is only one column, vertical ordering still matters. The algorithm handles this seamlessly because the outer loop still processes column `0`, and the inner loop performs the required row comparisons.
