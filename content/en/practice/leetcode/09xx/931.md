## Solution

## Problem Understanding

The problem gives us an `n x n` matrix of integers and asks for the minimum possible sum of a falling path. A falling path begins at any cell in the first row and moves downward one row at a time. From position `(row, col)`, the next step may go to one of three positions in the next row:

- Directly below: `(row + 1, col)`
- Diagonally left: `(row + 1, col - 1)`
- Diagonally right: `(row + 1, col + 1)`

The goal is to find the smallest total sum among all valid falling paths.

In other words, we must choose exactly one element from each row while respecting the movement constraints between rows. The answer is the minimum sum achievable.

The input matrix contains integers that may be positive, zero, or negative. Since values can be negative, the minimum path is not necessarily the shortest-looking path locally. Greedy decisions can fail because a larger value early may lead to much smaller values later.

The constraints are important:

- `1 <= n <= 100`
- `-100 <= matrix[i][j] <= 100`

An `n` value of 100 means there are at most 10,000 cells. This is small enough for dynamic programming solutions with `O(n^2)` time complexity, but too large for exponential brute-force recursion.

Several edge cases are important:

- A `1 x 1` matrix, where the only element is the answer.
- Matrices containing negative values, where the minimum path sum can also be negative.
- Paths that touch the leftmost or rightmost columns, where diagonal movement may go out of bounds.
- Multiple paths producing the same minimum sum.

The problem guarantees the matrix is square and non-empty, so we do not need to handle malformed inputs.

## Approaches

### Brute Force Approach

The brute-force solution explores every possible falling path starting from each column in the first row.

From each cell, the recursion branches into up to three directions:

- Down-left
- Down
- Down-right

For every possible path, we compute the total sum and track the minimum.

This approach is correct because it explicitly checks every valid path. Since the optimal path must be one of those paths, the minimum found is guaranteed to be correct.

However, this method becomes extremely expensive as the matrix grows. Each row can branch into three possibilities, producing roughly `3^n` possible paths in the worst case. Even for `n = 100`, this is computationally infeasible.

The main issue is repeated computation. The minimum falling path from a particular cell is recalculated many times across different recursive branches.

### Dynamic Programming Insight

The key observation is that the minimum falling path ending at a cell depends only on the minimum falling paths ending in the previous row.

For a cell `(row, col)`, the best path reaching it must come from:

- `(row - 1, col - 1)`
- `(row - 1, col)`
- `(row - 1, col + 1)`

Therefore, we can define a dynamic programming state:

$$dp[row][col]$$

This represents the minimum falling path sum that ends at `(row, col)`.

The transition becomes:

$dp[r][c] = matrix[r][c] + \min(dp[r-1][c-1],\ dp[r-1][c],\ dp[r-1][c+1])$

This eliminates redundant computation because each subproblem is solved exactly once.

Since there are `n^2` cells and each transition checks at most three previous cells, the total complexity becomes `O(n^2)`.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(3^n) | O(n) | Recursively explores every falling path |
| Optimal Dynamic Programming | O(n^2) | O(n^2) or O(n) | Reuses overlapping subproblems efficiently |

## Algorithm Walkthrough

1. Create a dynamic programming table.

We create a `dp` matrix of the same size as the input matrix. Each entry stores the minimum falling path sum ending at that position.
2. Initialize the first row.

The first row has no previous row above it, so the minimum path ending at each cell is simply the cell's own value.

$$dp[0][col] = matrix[0][col]$$
3. Process each remaining row.

For every cell `(row, col)`, we examine the three reachable cells from the previous row:

- Upper-left
- Upper
- Upper-right

We take the minimum among those valid positions.
4. Handle boundary conditions carefully.

Cells on the left edge do not have an upper-left neighbor.

Cells on the right edge do not have an upper-right neighbor.

We must avoid out-of-bounds access by checking indices before using them.
5. Compute the current cell's best value.

The minimum path ending at the current cell equals:

- The current matrix value
- Plus the minimum valid predecessor value
6. Continue row by row.

By the time we reach row `r`, all values from row `r - 1` are already finalized. This guarantees each state can be computed correctly.
7. Return the best value from the last row.

A falling path may end in any column of the final row, so the answer is:

$$\min(dp[n-1])$$

### Why it works

The algorithm works because of optimal substructure. Any minimum falling path ending at `(row, col)` must extend from the minimum falling path ending at one of the three reachable cells in the previous row. If a better predecessor existed, replacing the chosen predecessor would produce a smaller total path, contradicting optimality. Therefore, building solutions row by row guarantees correctness.

## Python Solution

```
from typing import List

class Solution:
    def minFallingPathSum(self, matrix: List[List[int]]) -> int:
        n = len(matrix)

        dp = [[0] * n for _ in range(n)]

        # Initialize first row
        for col in range(n):
            dp[0][col] = matrix[0][col]

        # Fill DP table
        for row in range(1, n):
            for col in range(n):
                min_previous = dp[row - 1][col]

                if col > 0:
                    min_previous = min(min_previous, dp[row - 1][col - 1])

                if col < n - 1:
                    min_previous = min(min_previous, dp[row - 1][col + 1])

                dp[row][col] = matrix[row][col] + min_previous

        return min(dp[n - 1])
```

The implementation begins by determining the matrix size and creating a dynamic programming table with the same dimensions.

The first row is initialized directly from the input matrix because no moves have occurred yet. Each value already represents the minimum path ending at that position.

The nested loops process the matrix row by row. For each cell, the code checks all valid predecessor positions from the previous row. Boundary checks prevent invalid index access on the leftmost and rightmost columns.

The variable `min_previous` stores the smallest reachable predecessor value. After finding it, the current matrix value is added to produce the best path sum ending at the current cell.

Finally, the algorithm returns the minimum value from the last row because any column may serve as the endpoint of the optimal falling path.

## Go Solution

```
func minFallingPathSum(matrix [][]int) int {
	n := len(matrix)

	dp := make([][]int, n)
	for i := 0; i < n; i++ {
		dp[i] = make([]int, n)
	}

	// Initialize first row
	for col := 0; col < n; col++ {
		dp[0][col] = matrix[0][col]
	}

	// Fill DP table
	for row := 1; row < n; row++ {
		for col := 0; col < n; col++ {
			minPrevious := dp[row-1][col]

			if col > 0 && dp[row-1][col-1] < minPrevious {
				minPrevious = dp[row-1][col-1]
			}

			if col < n-1 && dp[row-1][col+1] < minPrevious {
				minPrevious = dp[row-1][col+1]
			}

			dp[row][col] = matrix[row][col] + minPrevious
		}
	}

	// Find minimum in last row
	result := dp[n-1][0]

	for col := 1; col < n; col++ {
		if dp[n-1][col] < result {
			result = dp[n-1][col]
		}
	}

	return result
}
```

The Go implementation follows the same dynamic programming strategy as the Python version. Since Go does not provide built-in helper functions like Python's `min`, comparisons are performed manually using `if` statements.

Slices are used to represent the 2D dynamic programming table. Integer overflow is not a concern because the maximum possible absolute path sum is small:

$$100 \times 100 = 10000$$

which easily fits within Go's `int` type.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n^2) | Every cell is processed exactly once |
| Space | O(n^2) | The DP table stores one value per cell |

The matrix contains `n^2` cells, and each cell performs only constant work by checking at most three predecessors. Therefore, the runtime is quadratic.

The dynamic programming table also contains `n^2` entries, producing quadratic auxiliary space usage. This can be optimized to `O(n)` using rolling rows because each state depends only on the previous row.

## Edge Cases

A `1 x 1` matrix is the smallest possible input. This case is important because there are no transitions between rows. A careless implementation might incorrectly assume a previous row exists and attempt invalid indexing. The implementation handles this naturally because the first row is initialized directly, and the transition loop never executes.

Matrices with negative numbers are another important edge case. Greedy approaches often fail here because choosing the locally smallest value may not produce the globally smallest total path. The dynamic programming solution correctly evaluates all valid predecessor states and therefore always finds the true minimum path sum.

Boundary columns are a common source of off-by-one bugs. Cells in the leftmost column do not have an upper-left predecessor, and cells in the rightmost column do not have an upper-right predecessor. The implementation explicitly checks column boundaries before accessing diagonal neighbors, preventing invalid index access while still considering all legal moves.

Another subtle edge case occurs when multiple falling paths produce the same minimum sum. The algorithm does not rely on path uniqueness. Since it stores only the minimum achievable sum for each cell, ties are handled naturally without any special logic.
