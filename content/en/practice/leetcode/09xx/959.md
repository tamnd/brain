## Solution

## Problem Understanding

The problem gives us an `n x n` grid where every cell contains one of three possible characters:

- `'/'`
- `'\'`
- `' '` , a blank space

Each character divides the square differently:

- `'/'` creates a diagonal from bottom-left to top-right
- `'\'` creates a diagonal from top-left to bottom-right
- `' '` does not divide the square at all

These slashes partition the entire grid into several connected regions. Two areas belong to the same region if they are connected through open space without crossing a slash boundary.

The goal is to compute how many distinct connected regions exist after considering all slash boundaries.

A critical challenge is that slashes divide cells diagonally, while grids are naturally represented with horizontal and vertical adjacency. Standard flood-fill on the original grid is not enough because diagonal boundaries create partial-cell regions.

For example, a single `'/'` in a cell separates the cell into two triangular regions. A naive algorithm treating each cell as one unit loses this information.

The constraints are relatively small:

- `1 <= n <= 30`
- Maximum grid size is `30 x 30`

Even though the grid is small, we still need a correct representation of diagonal partitions. The small constraints allow solutions with expanded grids or graph modeling.

Several edge cases are important:

- Completely blank grids should produce exactly one region.
- Slashes forming enclosed loops create additional regions.
- Adjacent slashes can connect in non-obvious ways across cells.
- A single cell may already contain multiple regions if divided by a slash.
- Backslashes are escaped in the input representation, so `"\\ "` actually means a string containing `\`.

## Approaches

### Brute Force Approach

A brute-force idea is to treat the problem geometrically and explicitly model every fractional triangular area created by slashes.

One possible brute-force strategy is to subdivide each cell into many smaller pixels, such as a `3 x 3` expansion. Then:

- Draw slashes as blocked pixels.
- Treat remaining pixels as open space.
- Run DFS or BFS to count connected components.

This works because increasing resolution converts diagonal boundaries into standard blocked cells in a larger grid.

For example:

- `'/'` blocks the anti-diagonal
- `'\'` blocks the main diagonal

Then standard graph traversal correctly counts regions.

The approach is correct because the finer-resolution grid preserves all geometric separations created by slashes.

However, this method is inefficient conceptually because it artificially inflates the grid size. A `30 x 30` grid becomes `90 x 90`, increasing traversal overhead.

### Optimal Approach

The key insight is that each cell can naturally be divided into four triangular subregions.

Instead of expanding the entire grid geometrically, we model connectivity directly.

Each cell is split into:

- top triangle
- right triangle
- bottom triangle
- left triangle

We then connect triangles:

- inside the same cell, depending on the slash type
- across neighboring cells

This transforms the problem into counting connected components in a graph.

Union-Find is ideal here because:

- we repeatedly merge connected regions
- we only need the final number of connected components
- graph connectivity operations are efficient

Each triangle becomes a node in Union-Find.

Since there are `n * n * 4` triangles total, the graph size remains manageable.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O((3n)²) = O(n²) | O(n²) | Expands grid into pixels and flood-fills |
| Optimal | O(n² * α(n)) | O(n²) | Uses Union-Find on triangular subregions |

Although both are effectively quadratic for this constraint size, the Union-Find approach is more elegant and directly models the topology of the regions.

## Algorithm Walkthrough

### Step 1: Split Every Cell into Four Triangles

Each cell is divided into four parts:

- `0` = top
- `1` = right
- `2` = bottom
- `3` = left

So every cell contributes four graph nodes.

If a cell index is `(r, c)`, its base node ID is:

```
base = (r * n + c) * 4
```

Then:

- top = `base + 0`
- right = `base + 1`
- bottom = `base + 2`
- left = `base + 3`

This representation allows diagonal partitions to be modeled precisely.

### Step 2: Connect Internal Triangles

The slash determines which triangles inside the cell are connected.

#### Blank Space `' '`

All four triangles connect together because there is no divider.

Connections:

- top ↔ right
- right ↔ bottom
- bottom ↔ left

#### Slash `'/'`

The slash separates:

- top + left
- right + bottom

Connections:

- top ↔ left
- right ↔ bottom

#### Backslash `'\'`

The backslash separates:

- top + right
- bottom + left

Connections:

- top ↔ right
- bottom ↔ left

These unions correctly model the internal geometry.

### Step 3: Connect Neighboring Cells

Triangles touching neighboring cells must also connect.

We connect:

- current bottom ↔ below top
- current right ↔ right neighbor left

These are sufficient because every adjacency is handled exactly once.

### Step 4: Count Connected Components

Initially every triangle is its own component.

Each successful union reduces the component count.

At the end, the remaining number of disjoint sets equals the number of regions.

### Why it Works

The algorithm works because every possible open area in the grid corresponds exactly to a connected set of triangles.

Internal unions preserve connectivity inside a cell, while neighbor unions preserve connectivity across cells.

Slashes prevent certain unions, which correctly creates barriers between regions.

Therefore, every connected component in the Union-Find structure corresponds to one distinct region in the final partitioned grid.

## Python Solution

```
from typing import List

class UnionFind:
    def __init__(self, size: int):
        self.parent = list(range(size))
        self.rank = [0] * size
        self.components = size

    def find(self, x: int) -> int:
        if self.parent[x] != x:
            self.parent[x] = self.find(self.parent[x])
        return self.parent[x]

    def union(self, x: int, y: int) -> None:
        root_x = self.find(x)
        root_y = self.find(y)

        if root_x == root_y:
            return

        if self.rank[root_x] < self.rank[root_y]:
            root_x, root_y = root_y, root_x

        self.parent[root_y] = root_x

        if self.rank[root_x] == self.rank[root_y]:
            self.rank[root_x] += 1

        self.components -= 1

class Solution:
    def regionsBySlashes(self, grid: List[str]) -> int:
        n = len(grid)
        total_parts = n * n * 4

        uf = UnionFind(total_parts)

        for row in range(n):
            for col in range(n):
                base = (row * n + col) * 4

                top = base + 0
                right = base + 1
                bottom = base + 2
                left = base + 3

                cell = grid[row][col]

                # Internal connections
                if cell == ' ':
                    uf.union(top, right)
                    uf.union(right, bottom)
                    uf.union(bottom, left)

                elif cell == '/':
                    uf.union(top, left)
                    uf.union(right, bottom)

                else:  # '\'
                    uf.union(top, right)
                    uf.union(bottom, left)

                # Connect with bottom neighbor
                if row + 1 < n:
                    neighbor_base = ((row + 1) * n + col) * 4
                    neighbor_top = neighbor_base + 0
                    uf.union(bottom, neighbor_top)

                # Connect with right neighbor
                if col + 1 < n:
                    neighbor_base = (row * n + (col + 1)) * 4
                    neighbor_left = neighbor_base + 3
                    uf.union(right, neighbor_left)

        return uf.components
```

The implementation begins by constructing a Union-Find structure containing one node for every triangle in every cell.

The `UnionFind` class supports:

- path compression in `find`
- union by rank in `union`

These optimizations make operations nearly constant time.

Inside the nested grid traversal:

- each cell is mapped to four triangle IDs
- internal unions depend on the character
- neighboring cells are connected through shared borders

The final number of connected components stored in `uf.components` equals the number of regions.

## Go Solution

```
package main

type UnionFind struct {
	parent     []int
	rank       []int
	components int
}

func NewUnionFind(size int) *UnionFind {
	parent := make([]int, size)
	rank := make([]int, size)

	for i := 0; i < size; i++ {
		parent[i] = i
	}

	return &UnionFind{
		parent:     parent,
		rank:       rank,
		components: size,
	}
}

func (uf *UnionFind) Find(x int) int {
	if uf.parent[x] != x {
		uf.parent[x] = uf.Find(uf.parent[x])
	}
	return uf.parent[x]
}

func (uf *UnionFind) Union(x int, y int) {
	rootX := uf.Find(x)
	rootY := uf.Find(y)

	if rootX == rootY {
		return
	}

	if uf.rank[rootX] < uf.rank[rootY] {
		rootX, rootY = rootY, rootX
	}

	uf.parent[rootY] = rootX

	if uf.rank[rootX] == uf.rank[rootY] {
		uf.rank[rootX]++
	}

	uf.components--
}

func regionsBySlashes(grid []string) int {
	n := len(grid)
	totalParts := n * n * 4

	uf := NewUnionFind(totalParts)

	for row := 0; row < n; row++ {
		for col := 0; col < n; col++ {
			base := (row*n + col) * 4

			top := base + 0
			right := base + 1
			bottom := base + 2
			left := base + 3

			cell := grid[row][col]

			// Internal connections
			if cell == ' ' {
				uf.Union(top, right)
				uf.Union(right, bottom)
				uf.Union(bottom, left)
			} else if cell == '/' {
				uf.Union(top, left)
				uf.Union(right, bottom)
			} else {
				// '\'
				uf.Union(top, right)
				uf.Union(bottom, left)
			}

			// Bottom neighbor
			if row+1 < n {
				neighborBase := ((row + 1) * n + col) * 4
				neighborTop := neighborBase + 0
				uf.Union(bottom, neighborTop)
			}

			// Right neighbor
			if col+1 < n {
				neighborBase := (row*n + (col + 1)) * 4
				neighborLeft := neighborBase + 3
				uf.Union(right, neighborLeft)
			}
		}
	}

	return uf.components
}
```

The Go implementation follows the same logic as the Python version.

Some Go-specific details are important:

- Strings in Go are byte arrays, so `grid[row][col]` returns a byte.
- Slices are initialized with `make`.
- Recursive path compression is implemented identically.
- Integer overflow is not a concern because the maximum node count is only `30 * 30 * 4 = 3600`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n² * α(n)) | Each triangle participates in a constant number of Union-Find operations |
| Space | O(n²) | Stores parent and rank arrays for all triangles |

The total number of nodes is `4 * n²`.

Each node participates in only a few union operations:

- internal unions
- neighbor unions

Union-Find with path compression and union by rank gives amortized inverse Ackermann complexity, which is effectively constant in practice.

Therefore the entire algorithm is essentially linear in the number of triangles.

## Edge Cases

### Single Cell with a Slash

A single cell containing `'/'` or `'\'` already creates two regions. A naive algorithm that treats cells as indivisible would incorrectly return one region.

The triangle-based modeling handles this naturally because the slash prevents certain internal unions, leaving two connected components.

### Completely Empty Grid

If every cell contains `' '`, the entire board should form one connected region.

This case is important because over-segmentation bugs can accidentally leave disconnected components between neighboring cells.

The implementation correctly unions all internal triangles and neighboring boundaries, producing exactly one component.

### Complex Cyclic Boundaries

Patterns like:

```
["/\\",
 "\\/"]
```

create enclosed loops and tiny triangular regions.

These are difficult for naive adjacency logic because diagonal partitions interact across multiple cells.

The Union-Find representation handles this correctly because every triangle boundary is modeled explicitly, preserving all geometric separations.
