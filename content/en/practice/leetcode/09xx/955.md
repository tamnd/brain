## Solution

## Problem Understanding

LeetCode 955, LeetCode "Delete Columns to Make Sorted II", asks us to determine the minimum number of columns that must be deleted from an array of equal length strings so that the resulting array becomes lexicographically sorted.

Each string represents a row, and every character position represents a column. When we delete a column, that column is removed from every string simultaneously. The goal is not to sort individual strings internally, but to ensure that the entire array satisfies:

```
strs[0] <= strs[1] <= strs[2] <= ...
```

in lexicographic order after deletions.

The important detail is that lexicographic ordering compares strings from left to right. The first differing character determines the order. This means some columns become irrelevant once a pair of strings has already been strictly ordered by an earlier column.

For example, consider:

```
["ca", "bb", "ac"]
```

The first column produces:

```
c, b, a
```

which violates sorted order because `"ca" > "bb"` and `"bb" > "ac"` at the first character. If we delete that column, we get:

```
["a", "b", "c"]
```

which is sorted.

The constraints are relatively small:

- Up to 100 strings
- Each string length up to 100

This means an $O(n \times m^2)$ solution is acceptable, where:

- $n$ = number of strings
- $m$ = length of each string

However, brute force over all subsets of columns would still be infeasible because there are $2^m$ possible deletion combinations.

Several edge cases are important:

- The array may already be sorted, requiring zero deletions.
- Every column may need deletion.
- Some rows may remain equal for many columns before eventually diverging.
- A later column cannot repair an ordering violation introduced earlier unless the problematic earlier column is deleted.
- Equal adjacent rows must remain allowed because lexicographic order uses `<=`, not `<`.

Understanding how lexicographic comparison works incrementally is the key to the optimal solution.

## Approaches

### Brute Force Approach

A brute force solution would try every possible subset of columns to delete. For each subset, we would reconstruct all strings without those columns and then verify whether the resulting array is lexicographically sorted.

This approach is correct because it explores all valid deletion combinations and chooses the minimum deletion count among valid candidates.

However, the number of subsets is exponential. If the string length is $m$, there are:

$2^m$

possible deletion sets.

For each subset, reconstructing and validating strings costs additional work. With $m \le 100$, this quickly becomes impossible.

### Optimal Greedy Approach

The key insight is that lexicographic ordering can be resolved progressively from left to right.

Suppose we compare adjacent strings:

```
strs[i]
strs[i + 1]
```

If we already know that:

```
strs[i] < strs[i + 1]
```

based on previous columns, then future columns no longer matter for this pair. Their relative order is permanently fixed.

This allows a greedy strategy:

- Process columns from left to right.
- Tentatively keep the current column.
- If keeping it would create an ordering violation for any unresolved adjacent pair, delete the column.
- Otherwise, keep the column and mark newly resolved pairs as permanently ordered.

The algorithm works because lexicographic order depends on the first differing character. Once a pair differs correctly, later characters cannot invalidate that ordering.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^m × n × m) | O(n × m) | Tries every deletion combination |
| Optimal Greedy | O(n × m) | O(n) | Processes columns left to right while tracking resolved pairs |

## Algorithm Walkthrough

### Step 1: Initialize Tracking State

Create a boolean array called `sorted_pairs` of size `n - 1`.

```
sorted_pairs[i]
```

represents whether:

```
strs[i] < strs[i + 1]
```

has already been permanently established by previous columns.

Initially, all values are `False` because no comparisons have been resolved yet.

### Step 2: Process Columns Left to Right

Iterate through each column index from `0` to `m - 1`.

For every column, we test whether keeping it would violate lexicographic ordering.

### Step 3: Check for Violations

For every unresolved adjacent pair:

```
strs[i], strs[i + 1]
```

check the current column.

If:

```
strs[i][col] > strs[i + 1][col]
```

then keeping this column would force the array out of lexicographic order.

Since lexicographic comparison depends on the first differing character, this violation cannot be repaired later.

Therefore, we must delete the current column.

Increment the deletion counter and skip updating pair states for this column.

### Step 4: Keep Safe Columns

If no violation exists, the column can safely remain.

Now update the `sorted_pairs` array.

Whenever:

```
strs[i][col] < strs[i + 1][col]
```

that pair becomes permanently resolved.

Set:

```
sorted_pairs[i] = True
```

Future columns no longer matter for this pair.

### Step 5: Continue Until All Columns Are Processed

Repeat the process for every column.

At the end, the deletion counter contains the minimum number of columns that must be removed.

### Why it works

The algorithm maintains an important invariant:

- Every kept column preserves lexicographic validity for all unresolved pairs.
- Once a pair becomes strictly ordered, future columns cannot change that ordering.

Because columns are processed left to right, any violation discovered at a column means that column must be deleted immediately. Keeping it would introduce an irreversible lexicographic contradiction.

This greedy choice is always optimal because deleting a violating column is unavoidable.

## Python Solution

```
from typing import List

class Solution:
    def minDeletionSize(self, strs: List[str]) -> int:
        n = len(strs)
        m = len(strs[0])

        deleted_columns = 0

        # sorted_pairs[i] means:
        # strs[i] < strs[i + 1] has already been established
        sorted_pairs = [False] * (n - 1)

        for col in range(m):
            should_delete = False

            # Check whether keeping this column causes a violation
            for row in range(n - 1):
                if not sorted_pairs[row]:
                    if strs[row][col] > strs[row + 1][col]:
                        should_delete = True
                        break

            if should_delete:
                deleted_columns += 1
                continue

            # Update resolved pairs
            for row in range(n - 1):
                if not sorted_pairs[row]:
                    if strs[row][col] < strs[row + 1][col]:
                        sorted_pairs[row] = True

        return deleted_columns
```

The implementation closely follows the greedy algorithm.

The variable `sorted_pairs` tracks which adjacent string pairs already have a confirmed lexicographic relationship. This optimization is essential because once a pair is resolved, future columns no longer affect it.

For each column, the algorithm first performs a validation pass. This ensures that keeping the column would not create an irreversible ordering violation.

If a violation exists, the column is deleted immediately by incrementing `deleted_columns`.

If the column is safe, a second pass updates which adjacent pairs are now permanently ordered.

This separation between validation and updating is important. We must never partially update state before knowing whether the column is valid.

## Go Solution

```
func minDeletionSize(strs []string) int {
	n := len(strs)
	m := len(strs[0])

	deletedColumns := 0

	// sortedPairs[i] means:
	// strs[i] < strs[i+1] is already established
	sortedPairs := make([]bool, n-1)

	for col := 0; col < m; col++ {
		shouldDelete := false

		// Check whether keeping this column causes a violation
		for row := 0; row < n-1; row++ {
			if !sortedPairs[row] {
				if strs[row][col] > strs[row+1][col] {
					shouldDelete = true
					break
				}
			}
		}

		if shouldDelete {
			deletedColumns++
			continue
		}

		// Update resolved pairs
		for row := 0; row < n-1; row++ {
			if !sortedPairs[row] {
				if strs[row][col] < strs[row+1][col] {
					sortedPairs[row] = true
				}
			}
		}
	}

	return deletedColumns
}
```

The Go implementation mirrors the Python logic almost exactly.

One small language difference is that Go strings are byte indexed. Since the problem guarantees lowercase English letters only, direct indexing like:

```
strs[row][col]
```

is safe and efficient.

The `sortedPairs` slice is initialized using `make([]bool, n-1)`, which automatically fills the slice with `false`.

No special handling for integer overflow is required because the constraints are very small.

| --- | --- | --- |
| Time | O(n × m) | Each column scans adjacent row pairs at most twice |
| Space | O(n) | The `sorted_pairs` array stores pair states |

The algorithm processes every column sequentially. For each column, it may scan all adjacent row pairs once to validate and once to update states.

Since there are `m` columns and `n - 1` adjacent pairs:

$O(n \times m)$

time complexity follows naturally.

The only additional memory used is the boolean tracking array.

## Edge Cases

One important edge case occurs when the array already satisfies lexicographic ordering before any deletions. A naive implementation might still delete unnecessary columns if it only checks local column ordering independently. The greedy solution avoids this by tracking resolved pairs and only deleting columns that create genuine lexicographic contradictions.

Another critical edge case involves identical strings or identical prefixes. For example:

```
["aaa", "aaa"]
```

or

```
["abcx", "abcy"]
```

In lexicographic comparison, equal prefixes are perfectly valid. The implementation handles this correctly because pairs are only marked resolved when a strictly smaller character is found. Equal characters leave the pair unresolved.

A more subtle edge case appears when an earlier column permanently determines ordering. Consider:

```
["ax", "by"]
```

After the first column:

```
a < b
```

the second column no longer matters. Even if:

```
x > y
```

the strings remain lexicographically sorted. A naive implementation that checks every column independently would incorrectly delete the second column. The `sorted_pairs` tracking prevents this mistake by ignoring already resolved pairs.
