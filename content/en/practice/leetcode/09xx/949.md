## Problem Understanding

This problem asks us to construct the largest possible valid 24 hour time using exactly four given digits. Each digit from the input array must be used once and only once. The final result must follow the `"HH:MM"` format, where the hour portion ranges from `00` to `23` and the minute portion ranges from `00` to `59`.

The input is an array of four integers, each between `0` and `9`. Since the array always contains exactly four digits, the total number of possible arrangements is limited. Our task is to determine which arrangement forms the latest valid time chronologically.

For example, given `[1,2,3,4]`, we can rearrange the digits into several valid times such as `"12:34"`, `"21:43"`, and `"23:41"`. Among all valid possibilities, `"23:41"` is the latest, so that is the correct answer.

The constraints are very small. Since there are only four digits, the total number of permutations is `4! = 24`. This immediately suggests that enumeration or backtracking is practical. Even a brute force solution will run extremely quickly because the search space is tiny.

Several edge cases are important to consider:

- Some digit combinations cannot form any valid time at all. For example, `[5,5,5,5]` only produces `"55:55"`, which is invalid.
- Digits may contain duplicates, such as `[0,0,1,0]`. The algorithm must still treat each occurrence as a separate position.
- Leading zeros are allowed. Times like `"01:23"` and `"00:59"` are perfectly valid.
- A naive implementation might incorrectly compare time strings lexicographically without validating the hour and minute ranges first.

The problem guarantees that the array length is always exactly four, so we never need to handle arrays of different sizes.

## Approaches

### Brute Force Approach

The brute force strategy is to generate every possible ordering of the four digits, interpret each ordering as a time, validate it, and keep track of the largest valid one.

For a permutation `[a,b,c,d]`, we form:

- Hour = `10 * a + b`
- Minute = `10 * c + d`

The time is valid if:

- `0 <= hour < 24`
- `0 <= minute < 60`

Whenever we encounter a valid time, we compare it against the best time found so far. One convenient way to compare times is to convert them into total minutes:

```
total_minutes = hour * 60 + minute
```

The permutation with the largest total minutes represents the latest time.

This approach is guaranteed to work because it exhaustively checks every possible arrangement. Since there are only 24 permutations, the runtime is extremely small.

### Optimized Backtracking Approach

A cleaner and more structured solution uses backtracking to build permutations incrementally.

Instead of generating all permutations upfront, we recursively construct candidate times digit by digit. At each recursion step, we choose one unused digit and append it to the current path. Once the path contains four digits, we evaluate whether it forms a valid time.

The key insight is that the input size is fixed and tiny. Backtracking naturally explores all valid arrangements while keeping the implementation organized and easy to reason about.

Although the asymptotic complexity remains effectively constant because there are only four digits, backtracking avoids unnecessary intermediate storage and fits naturally with permutation generation problems.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(4!) | O(4!) | Generate all permutations explicitly |
| Optimal | O(4!) | O(4!) | Backtracking permutation generation with validation |

## Algorithm Walkthrough

1. Initialize a variable to store the maximum valid time found so far. We can store this as the total number of minutes since midnight. Initially, set it to `-1` to indicate that no valid time has been discovered yet.
2. Create a backtracking function that builds permutations one digit at a time. The function keeps track of:

- The current permutation being constructed
- Which indices have already been used
3. At each recursive step, iterate through all four digits. If a digit has not been used yet, mark it as used and append it to the current path.
4. Continue recursion until the path contains exactly four digits. At this point, interpret the digits as:

- `hour = path[0] * 10 + path[1]`
- `minute = path[2] * 10 + path[3]`
5. Validate the generated time:

- The hour must be less than `24`
- The minute must be less than `60`
6. If the time is valid, convert it into total minutes:

```
total = hour * 60 + minute
```
7. Compare this value with the current best result. If it is larger, update the maximum.
8. After exploring a choice, backtrack by removing the last digit and marking it as unused again. This allows other permutations to be explored correctly.
9. Once all recursive calls finish, check whether a valid time was found:

- If the maximum remains `-1`, return an empty string
- Otherwise, convert the stored total minutes back into `"HH:MM"` format

### Why it works

The algorithm systematically explores every possible arrangement of the four digits exactly once. Since every valid permutation is evaluated, the algorithm cannot miss the optimal answer. By storing the maximum total minutes encountered, we guarantee that the final result represents the latest valid time.

## Python Solution

```
from typing import List

class Solution:
    def largestTimeFromDigits(self, arr: List[int]) -> str:
        max_minutes = -1
        used = [False] * 4

        def backtrack(path: List[int]) -> None:
            nonlocal max_minutes

            if len(path) == 4:
                hour = path[0] * 10 + path[1]
                minute = path[2] * 10 + path[3]

                if hour < 24 and minute < 60:
                    total_minutes = hour * 60 + minute
                    max_minutes = max(max_minutes, total_minutes)

                return

            for i in range(4):
                if used[i]:
                    continue

                used[i] = True
                path.append(arr[i])

                backtrack(path)

                path.pop()
                used[i] = False

        backtrack([])

        if max_minutes == -1:
            return ""

        hour = max_minutes // 60
        minute = max_minutes % 60

        return f"{hour:02d}:{minute:02d}"
```

The implementation begins by initializing `max_minutes` to `-1`, which acts as a sentinel value indicating that no valid time has been found yet.

The `used` array tracks which indices have already been included in the current permutation. This prevents reuse of digits during permutation generation.

The recursive `backtrack` function builds permutations incrementally. Whenever the path reaches length four, the digits are interpreted as an hour and minute pair. The implementation validates the ranges before updating the maximum.

The backtracking process works by choosing a digit, recursing deeper, and then undoing the choice afterward. This ensures that every possible ordering is explored correctly.

Finally, if a valid time exists, the total minutes are converted back into hour and minute components and formatted using two digit padding.

## Go Solution

```
package main

import "fmt"

func largestTimeFromDigits(arr []int) string {
	maxMinutes := -1
	used := make([]bool, 4)
	path := make([]int, 0, 4)

	var backtrack func()

	backtrack = func() {
		if len(path) == 4 {
			hour := path[0]*10 + path[1]
			minute := path[2]*10 + path[3]

			if hour < 24 && minute < 60 {
				totalMinutes := hour*60 + minute

				if totalMinutes > maxMinutes {
					maxMinutes = totalMinutes
				}
			}

			return
		}

		for i := 0; i < 4; i++ {
			if used[i] {
				continue
			}

			used[i] = true
			path = append(path, arr[i])

			backtrack()

			path = path[:len(path)-1]
			used[i] = false
		}
	}

	backtrack()

	if maxMinutes == -1 {
		return ""
	}

	hour := maxMinutes / 60
	minute := maxMinutes % 60

	return fmt.Sprintf("%02d:%02d", hour, minute)
}
```

The Go implementation follows the same logic as the Python solution. Instead of nested functions with automatic closure behavior, Go uses a function variable declaration for recursion.

Slices are used to maintain the current permutation path. Backtracking is implemented by appending a digit before recursion and shrinking the slice afterward.

Formatting is handled with `fmt.Sprintf("%02d:%02d", hour, minute)` to ensure leading zeros appear correctly.

Since all arithmetic values remain very small, integer overflow is not a concern in Go.

## Edge Cases

One important edge case occurs when no valid time can be formed at all. Inputs like `[5,5,5,5]` or `[9,9,9,9]` generate only invalid hours and minutes. A buggy implementation might accidentally return an invalid formatted string instead of an empty string. This implementation avoids that problem by initializing `max_minutes` to `-1` and only updating it when a valid time is discovered.

Another critical edge case involves leading zeros. Times such as `"00:59"` or `"07:00"` are completely valid in 24 hour format. Some implementations incorrectly reject these because they treat leading zeros as invalid numeric representations. This solution formats the final answer using zero padded formatting, ensuring that valid times with leading zeros are handled correctly.

Duplicate digits are also a common source of bugs. For example, `[0,0,1,0]` contains three zeros. A naive set based permutation generator might accidentally eliminate distinct permutations because identical values appear multiple times. This implementation tracks usage by index rather than by value, so every occurrence of a digit is treated independently and all valid arrangements are explored correctly.

A final subtle edge case occurs when the lexicographically largest arrangement is not a valid time. For instance, `[2,4,0,0]` might tempt a greedy solution to attempt `"42:00"`, which is invalid. The algorithm correctly validates each constructed time before considering it as a candidate answer.
