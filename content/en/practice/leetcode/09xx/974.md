## Solution

## Problem Understanding

The problem asks us to count how many non-empty contiguous subarrays have a sum divisible by a given integer `k`.

We are given an integer array `nums` that may contain positive numbers, negative numbers, and zeros. We are also given an integer `k`. A subarray is any contiguous section of the array, meaning the elements must appear next to each other in their original order.

For every possible subarray, we compute its sum. If that sum is divisible by `k`, meaning:

$\text{sum} \bmod k = 0$

then that subarray contributes to the final answer.

The output is the total number of such subarrays.

The constraints are important because they immediately rule out extremely slow solutions. The array length can be as large as `3 * 10^4`, which means there can be roughly:

$\frac{n(n+1)}{2}$

possible subarrays. For `n = 30000`, this is far too many to examine using expensive repeated computations.

Another important detail is that the array can contain negative numbers. This matters because modulo operations with negative values behave differently across programming languages. A correct implementation must normalize modulo results carefully.

Some important edge cases include arrays containing only negative numbers, arrays containing many zeros, cases where no valid subarray exists, and situations where the prefix sum becomes negative. The problem guarantees that `k >= 2`, so division by zero is never an issue.

## Approaches

### Brute Force Approach

The most direct solution is to examine every possible subarray.

We can use two nested loops. The outer loop chooses the starting index, and the inner loop extends the subarray one element at a time while maintaining the running sum. For every subarray, we check whether the sum is divisible by `k`.

This works because it explicitly evaluates every possible contiguous subarray, so no valid answer can be missed.

However, there are too many subarrays for large inputs. An array of length `n` has approximately `n^2 / 2` subarrays. With `n = 30000`, this becomes prohibitively expensive.

Even if we maintain a running sum inside the inner loop to avoid recomputing sums from scratch, the time complexity remains quadratic.

### Optimal Approach, Prefix Sum with Modulo Counting

The key insight comes from prefix sums and modular arithmetic.

Suppose we define a prefix sum:

$prefix[i] = nums[0] + nums[1] + \dots + nums[i]$

The sum of a subarray from index `l` to `r` can be written as:

$subarraySum = prefix[r] - prefix[l-1]$

We want this value to be divisible by `k`:

$(prefix[r] - prefix[l-1]) \bmod k = 0$

This means:

$prefix[r] \bmod k = prefix[l-1] \bmod k$

This is the crucial observation.

If two prefix sums have the same remainder when divided by `k`, then the subarray between them has a sum divisible by `k`.

So instead of checking every subarray directly, we track how many times each remainder has appeared before. Every time we encounter a remainder again, we know that all previous occurrences form valid subarrays ending at the current position.

A hash map is ideal for this because it allows constant-time frequency lookup and updates.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Checks every possible subarray |
| Optimal | O(n) | O(k) | Uses prefix sums and remainder frequency counting |

## Algorithm Walkthrough

1. Initialize a hash map called `remainder_count`.

This map stores how many times each prefix sum remainder has appeared so far. The key is the remainder, and the value is the frequency.
2. Insert the value `{0: 1}` into the map before processing the array.

This is extremely important. It represents an empty prefix sum with remainder `0`. It allows subarrays starting from index `0` to be counted correctly.
3. Initialize two variables:

- `prefix_sum = 0`
- `result = 0`

The prefix sum tracks the cumulative total as we iterate through the array. The result stores the number of valid subarrays found.
4. Iterate through each number in `nums`.

At every step:

- Add the current number to `prefix_sum`
- Compute the remainder:

$remainder = prefix\_sum \bmod k$
5. Normalize negative remainders if necessary.

Some languages produce negative modulo results for negative numbers. To guarantee a remainder in the range `[0, k-1]`, use:

$remainder = (remainder + k) \bmod k$
6. Check how many times this remainder has appeared before.

If the same remainder has already appeared `x` times, then there are `x` subarrays ending at the current index whose sums are divisible by `k`.

Add that count to `result`.
7. Update the hash map.

Increment the frequency for the current remainder so future indices can use it.
8. After processing the entire array, return `result`.

### Why it works

The algorithm relies on the property that if two prefix sums leave the same remainder when divided by `k`, their difference must be divisible by `k`.

Every repeated remainder identifies one or more valid subarrays. By counting remainder frequencies as we iterate, we efficiently count all valid subarrays in linear time.

## Python Solution

```
from typing import List
from collections import defaultdict

class Solution:
    def subarraysDivByK(self, nums: List[int], k: int) -> int:
        remainder_count = defaultdict(int)
        remainder_count[0] = 1

        prefix_sum = 0
        result = 0

        for num in nums:
            prefix_sum += num

            remainder = prefix_sum % k

            result += remainder_count[remainder]

            remainder_count[remainder] += 1

        return result
```

The implementation starts by creating a hash map called `remainder_count`. This tracks how many times each modulo remainder has appeared. We initialize `remainder_count[0] = 1` so that subarrays starting at index `0` are handled correctly.

As we iterate through the array, we continuously update the running prefix sum. The remainder of the prefix sum modulo `k` determines whether previous prefix sums can form a divisible subarray with the current position.

Whenever the same remainder appears again, all earlier occurrences correspond to valid subarrays ending at the current index. We immediately add that frequency to the answer.

Finally, we increment the count for the current remainder so it can contribute to future subarrays.

Python already guarantees non-negative modulo results for positive divisors, so explicit normalization is unnecessary here.

## Go Solution

```
func subarraysDivByK(nums []int, k int) int {
	remainderCount := make(map[int]int)
	remainderCount[0] = 1

	prefixSum := 0
	result := 0

	for _, num := range nums {
		prefixSum += num

		remainder := prefixSum % k

		if remainder < 0 {
			remainder += k
		}

		result += remainderCount[remainder]

		remainderCount[remainder]++
	}

	return result
}
```

The Go implementation follows the same logic as the Python version, but there is one important difference involving modulo behavior.

In Go, the `%` operator can produce negative results when the dividend is negative. Because of this, we explicitly normalize the remainder by adding `k` when necessary.

Go maps return the zero value for missing keys, so accessing `remainderCount[remainder]` is safe even if the remainder has not appeared before.

Since the constraints are relatively small, standard Go integers are sufficient and overflow is not a concern.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each element is processed once |
| Space | O(k) | At most `k` distinct remainders are stored |

The algorithm performs a single pass through the array. Every iteration does constant-time work involving hash map lookups and updates.

The space complexity is bounded by the number of possible remainders modulo `k`, which ranges from `0` to `k - 1`. Therefore, the hash map can contain at most `k` entries.

## Edge Cases

One important edge case is arrays containing many zeros. Every zero contributes heavily to the number of valid subarrays because adding zero does not change the prefix sum remainder. A naive implementation can easily undercount these overlapping subarrays. The hash map approach handles this naturally because repeated remainders accumulate frequency counts correctly.

Another important case involves negative numbers. Some programming languages produce negative modulo values for negative prefix sums. If the remainder is not normalized, logically equivalent remainders may appear different, causing missed matches. The Go solution explicitly fixes this by adding `k` when necessary.

A third edge case occurs when valid subarrays begin at index `0`. Without initializing the remainder map with `{0: 1}`, these subarrays would never be counted because there would be no earlier prefix sum to pair with. The initialization effectively represents an empty prefix before the array starts.

A fourth subtle case is when all subarrays are valid. For example, if every element is divisible by `k`, the number of valid subarrays becomes:

$\frac{n(n+1)}{2}$

The algorithm still handles this efficiently because it only tracks remainder frequencies instead of explicitly generating every subarray.
