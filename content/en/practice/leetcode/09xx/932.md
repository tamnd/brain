## Solution

## Problem Understanding

The problem asks us to construct a **beautiful array** of length `n`. A beautiful array is a permutation of integers from `1` to `n` that satisfies a mathematical property: for any two elements `nums[i]` and `nums[j]` with `i < j`, there should **not exist an element `nums[k]` between them (i < k < j) such that it is the exact midpoint**, i.e., `2 * nums[k] == nums[i] + nums[j]`.

In simpler terms, the array must avoid forming arithmetic sequences of length 3. For example, `[1, 3, 2]` is beautiful because no element sits exactly halfway between two others. The input `n` represents the length of the array and sets the range of integers from `1` to `n`. The output is any permutation of these numbers that satisfies the beautiful property.

Constraints tell us that `n` is at most 1000, so brute-force checking all permutations would be computationally infeasible. Edge cases to consider include very small arrays like `n = 1` or `n = 2`, where the property trivially holds, and `n` large, which requires an efficient construction method rather than testing all permutations.

## Approaches

A naive brute-force approach would generate all permutations of numbers from `1` to `n` and then check each permutation to see if it satisfies the beautiful property. While this would eventually produce a correct answer, the number of permutations grows factorially with `n`, i.e., `n!`, which is infeasible for `n` up to 1000. Checking each permutation for the midpoint property requires an O(n^3) check in the worst case.

The key insight to solve this problem efficiently is based on **divide and conquer** and **mathematical construction**. Observe that if we split a beautiful array into odd and even numbers, then **recursively constructing beautiful arrays for odd indices and even indices separately** preserves the beautiful property when merged. Specifically, for any two numbers `a` (odd) and `b` (even), `2*k` will never equal `a + b` since one is odd and the other is even. This allows us to build the array recursively without checking all permutations.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n! * n^3) | O(n) | Generate all permutations and check the arithmetic sequence condition |
| Optimal | O(n log n) | O(n) | Recursive divide and conquer splitting into odd and even subsequences |

## Algorithm Walkthrough

1. If `n` is 1, return `[1]`. This is the base case because a single-element array trivially satisfies the beautiful property.
2. Split the target array into **odd indices** and **even indices** using the formula: for a beautiful array `B` of length `n`, the odd part is `[2*x - 1 for x in B]` and the even part is `[2*x for x in B]`.
3. Recursively generate beautiful arrays for lengths `(n + 1) // 2` (odd numbers) and `n // 2` (even numbers). These lengths ensure that we cover all numbers from `1` to `n`.
4. Merge the two arrays by concatenating the odd array followed by the even array. This combined array is guaranteed to be beautiful because no midpoint can exist between odd and even numbers.
5. Return the merged array.

**Why it works**: By always separating numbers into odd and even sequences recursively, we ensure no element will sit exactly halfway between any two others across the odd-even boundary. Within the subarrays, recursion guarantees each smaller array is beautiful, maintaining the property throughout.

## Python Solution

```
from typing import List

class Solution:
    def beautifulArray(self, n: int) -> List[int]:
        if n == 1:
            return [1]
        
        odd_part = self.beautifulArray((n + 1) // 2)
        even_part = self.beautifulArray(n // 2)
        
        result = [2 * x - 1 for x in odd_part] + [2 * x for x in even_part]
        return result
```

The implementation starts with the base case for `n = 1`. It recursively constructs the odd and even parts, scales the numbers accordingly, and merges them. Each recursion guarantees the subarray is beautiful, and the final merge preserves the property across the two halves.

## Go Solution

```
func beautifulArray(n int) []int {
    if n == 1 {
        return []int{1}
    }
    
    oddPart := beautifulArray((n + 1) / 2)
    evenPart := beautifulArray(n / 2)
    
    result := make([]int, 0, n)
    for _, x := range oddPart {
        result = append(result, 2*x-1)
    }
    for _, x := range evenPart {
        result = append(result, 2*x)
    }
    return result
}
```

In Go, slices are used to dynamically build the array. The recursive structure mirrors Python, and the slice is preallocated with capacity `n` for efficiency.

## Worked Examples

### Example 1: `n = 4`

1. Recursive call for `odd_part = beautifulArray(2)`

- `n = 2` → split into `odd_part = beautifulArray(1) = [1]`, `even_part = beautifulArray(1) = [1]`
- Transform: `odd = [1*2-1] = [1]`, `even = [1*2] = [2]` → merge → `[1,2]`
2. Recursive call for `even_part = beautifulArray(2)` → same as above → `[1,2]`
3. Scale for `n=4`: `odd_part = [1*2-1, 2*2-1] = [1,3]`, `even_part = [1*2, 2*2] = [2,4]` → merge → `[1,3,2,4]`

### Example 2: `n = 5`

1. Recursive call for `odd_part = beautifulArray(3)` → `[1,3,2]`
2. Recursive call for `even_part = beautifulArray(2)` → `[1,2]`
3. Scale and merge: `odd = [1,5,3]`, `even = [2,4]` → `[1,5,3,2,4]`

## Complexity Analysis

The recursion depth is O(log n), and each merge takes O(n), giving overall O(n log n) time. Space is dominated by the result array of size `n` and stack depth of O(log n).

## Edge Cases

**Single-element array (`n=1`)**: This is trivially beautiful, but a naive implementation might attempt to split it further and fail. The base case ensures correct handling.

**Even vs Odd `n`**: Recursive splitting handles odd and even lengths differently. For `n=5`, the odd part has length `(5+1)//2=3` and even part `5//2=2`. Correct calculation ensures all integers are included without overlap or gaps.

**Large `n` (e.g., 1000)**: Recursive depth could be significant, but Python and Go handle recursion for depth ~10 recursively log-scaled, and preallocating slices/arrays ensures memory efficiency. The algorithm avoids factorial-level computation, making large input feasible.
