## Solution

## Problem Understanding

The problem asks us to determine whether two string representations of rational numbers describe the exact same numeric value.

Each input string may contain three different parts:

- An integer part
- A non-repeating decimal part
- A repeating decimal part enclosed in parentheses

For example:

- `"12"` means the integer 12
- `"0.125"` means a terminating decimal
- `"0.1(6)"` means `0.166666...`
- `"1.(9)"` means `1.999999...`

The challenge is that the same rational number can have multiple valid decimal representations. For example:

- `0.(52)` and `0.5(25)` both represent `0.52525252...`
- `0.9(9)` equals `1.0`
- `0.1666(6)` and `0.166(66)` are identical

This means we cannot simply compare the strings directly. We must compare their mathematical values.

The constraints are very small. Each section has at most 4 digits, so the entire string is short. This tells us that performance is not the primary challenge. The real difficulty is handling decimal repetition correctly and avoiding floating point precision errors.

Several edge cases are especially important:

- Repeating `9`s, such as `0.(9)`, which mathematically equal the next integer
- Different placements of the repeating boundary, such as `0.1666(6)` versus `0.166(66)`
- Pure integers like `"1."`
- Numbers with no repeating section
- Repeating zeros, such as `"1.0(0)"`

A naive floating point comparison can easily fail because many rational numbers cannot be represented exactly in binary floating point format.

## Approaches

### Brute Force Approach

A straightforward idea is to simulate the decimal expansion of each number and compare many generated digits.

For example, we could:

1. Parse the number
2. Expand the repeating portion repeatedly
3. Generate perhaps 20 to 30 decimal digits
4. Compare the resulting decimal strings

This works for many cases because the constraints are small and repeating cycles are short.

However, this approach is fundamentally unreliable. Rational numbers can differ very far into the decimal expansion. Worse, numbers like `0.(9)` and `1.` require special mathematical handling because infinite repeating decimals have equivalence relationships that finite truncation cannot perfectly capture.

Another issue is precision. Using floating point values introduces rounding errors, especially for repeating decimals.

### Optimal Approach

The key insight is that every rational decimal representation can be converted into an exact fraction.

Instead of comparing decimal expansions, we convert both strings into reduced rational numbers of the form:

$$\frac{numerator}{denominator}$$

Two rational numbers are equal if and only if their reduced fractions are equal.

This avoids all floating point issues and naturally handles repeating decimals.

For a decimal representation:

$$I.N(R)$$

where:

- `I` is the integer part
- `N` is the non-repeating decimal part
- `R` is the repeating decimal part

we can derive an exact mathematical formula.

Suppose:

- `len(N) = n`
- `len(R) = r`

Then:

$$value =
I +
\frac{N}{10^n} +
\frac{R}{10^n(10^r - 1)}$$

This formula exactly represents the repeating decimal.

For example:

$$0.1(6)
=
\frac{1}{10} + \frac{6}{10(9)}
=
\frac{1}{10} + \frac{1}{15}
=
\frac{1}{6}$$

Once both numbers are converted into reduced fractions, equality becomes trivial.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(k) | O(k) | Expands decimal digits and compares approximations |
| Optimal | O(1) | O(1) | Converts directly into exact fractions |

Here, `k` represents the number of generated decimal digits in the brute force simulation.

## Algorithm Walkthrough

### Step 1: Parse the String Components

We separate the string into:

- Integer part
- Non-repeating decimal part
- Repeating decimal part

For example:

```
"12.34(56)"
```

becomes:

- Integer = `"12"`
- Non-repeating = `"34"`
- Repeating = `"56"`

If parentheses are absent, then the repeating portion is empty.

### Step 2: Initialize the Fraction with the Integer Part

We begin with:

$$fraction = integer\_part$$

represented as:

$$\frac{integer}{1}$$

### Step 3: Add the Non-Repeating Decimal Portion

Suppose the non-repeating digits are `"34"`.

This contributes:

$$\frac{34}{100}$$

because there are two decimal digits.

In general:

$$\frac{N}{10^n}$$

where `n` is the length of the non-repeating section.

### Step 4: Add the Repeating Portion

Suppose the repeating digits are `"56"`.

The repeating decimal:

$$0.(56)
=
\frac{56}{99}$$

If it starts after `n` non-repeating digits, then we divide additionally by `10^n`.

So the contribution becomes:

$$\frac{R}{10^n(10^r - 1)}$$

where:

- `r` is the repeating length
- `10^r - 1` produces `9`, `99`, `999`, etc.

### Step 5: Combine Fractions

We combine all parts into one fraction using standard fraction arithmetic.

Using Python's `Fraction` class greatly simplifies this process because it automatically reduces fractions.

### Step 6: Compare the Two Fractions

Once both strings are converted into exact fractions, equality comparison is exact and reliable.

### Why it works

Every terminating or repeating decimal corresponds to a unique rational number. The formula used for the repeating section derives directly from infinite geometric series:

$$0.(abc)
=
\frac{abc}{999}$$

By converting both inputs into exact reduced fractions, we remove all ambiguity from decimal notation. Two strings represent the same rational number if and only if their reduced fractions are identical.

## Python Solution

```
from fractions import Fraction

class Solution:
    def isRationalEqual(self, s: str, t: str) -> bool:

        def convert(value: str) -> Fraction:
            if '.' in value:
                integer_part, decimal_part = value.split('.')
            else:
                integer_part, decimal_part = value, ""

            repeating_part = ""

            if '(' in decimal_part:
                non_repeating, repeating = decimal_part.split('(')
                repeating_part = repeating[:-1]
            else:
                non_repeating = decimal_part

            result = Fraction(int(integer_part), 1)

            if non_repeating:
                numerator = int(non_repeating)
                denominator = 10 ** len(non_repeating)
                result += Fraction(numerator, denominator)

            if repeating_part:
                numerator = int(repeating_part)
                denominator = (
                    (10 ** len(non_repeating)) *
                    (10 ** len(repeating_part) - 1)
                )
                result += Fraction(numerator, denominator)

            return result

        return convert(s) == convert(t)
```

The implementation begins by parsing the string into its structural components. We first separate the integer portion from the decimal portion using the decimal point.

Next, we determine whether a repeating section exists by checking for parentheses. If parentheses are present, we split the decimal section into the non-repeating and repeating parts.

The integer portion is converted directly into a `Fraction`.

The non-repeating portion contributes a standard decimal fraction:

$$\frac{digits}{10^{length}}$$

The repeating portion contributes:

$$\frac{digits}{10^n(10^r - 1)}$$

where:

- `n` is the number of non-repeating digits
- `r` is the number of repeating digits

Python's `Fraction` class automatically reduces fractions to lowest terms, which makes equality comparison exact and simple.

## Go Solution

```
package main

func gcd(a, b int64) int64 {
	for b != 0 {
		a, b = b, a%b
	}
	return a
}

type Fraction struct {
	num int64
	den int64
}

func reduce(f Fraction) Fraction {
	g := gcd(f.num, f.den)
	return Fraction{
		num: f.num / g,
		den: f.den / g,
	}
}

func add(a, b Fraction) Fraction {
	return reduce(Fraction{
		num: a.num*b.den + b.num*a.den,
		den: a.den * b.den,
	})
}

func pow10(n int) int64 {
	result := int64(1)
	for i := 0; i < n; i++ {
		result *= 10
	}
	return result
}

func parseInt(s string) int64 {
	var result int64
	for _, ch := range s {
		result = result*10 + int64(ch-'0')
	}
	return result
}

func convert(value string) Fraction {
	integerPart := ""
	decimalPart := ""

	dotIndex := -1
	for i, ch := range value {
		if ch == '.' {
			dotIndex = i
			break
		}
	}

	if dotIndex == -1 {
		integerPart = value
	} else {
		integerPart = value[:dotIndex]
		decimalPart = value[dotIndex+1:]
	}

	nonRepeating := decimalPart
	repeating := ""

	for i, ch := range decimalPart {
		if ch == '(' {
			nonRepeating = decimalPart[:i]
			repeating = decimalPart[i+1 : len(decimalPart)-1]
			break
		}
	}

	result := Fraction{
		num: parseInt(integerPart),
		den: 1,
	}

	if len(nonRepeating) > 0 {
		num := parseInt(nonRepeating)
		den := pow10(len(nonRepeating))

		result = add(result, Fraction{
			num: num,
			den: den,
		})
	}

	if len(repeating) > 0 {
		num := parseInt(repeating)

		den := pow10(len(nonRepeating)) *
			(pow10(len(repeating)) - 1)

		result = add(result, Fraction{
			num: num,
			den: den,
		})
	}

	return reduce(result)
}

func isRationalEqual(s string, t string) bool {
	a := convert(s)
	b := convert(t)

	return a.num == b.num && a.den == b.den
}
```

The Go implementation mirrors the Python logic but requires manual fraction handling because Go does not provide a built-in rational arithmetic class like Python's `Fraction`.

We implement:

- Greatest common divisor for reduction
- Fraction addition
- Fraction normalization

All arithmetic uses `int64`, which is completely safe under the problem constraints because the denominators remain small.

Unlike Python, Go requires explicit parsing logic for integers and powers of ten.

## Edge Cases

One important edge case is repeating `9`s. Infinite repeating `9`s are mathematically equal to the next terminating decimal. For example:

$$0.(9) = 1$$

A floating point implementation can easily mishandle this because approximations never become exactly 1. By converting into exact fractions, the implementation correctly computes both sides as the same reduced rational number.

Another tricky case is different placements of the repeating boundary. Inputs like:

```
0.1666(6)
0.166(66)
```

look structurally different but represent identical infinite decimals. A naive parser might compare repeating blocks directly and conclude they differ. The fraction-based approach avoids this entirely because both expressions reduce to the same exact rational value.

A third important edge case is repeating zeros. Inputs such as:

```
1.0(0)
```

should equal:

```
1.
```

Some implementations accidentally treat the repeating section as meaningful even when it contributes nothing. In this solution, the repeating fraction becomes zero naturally:

$$\frac{0}{10^n(10^r - 1)} = 0$$

so the result remains mathematically correct.
