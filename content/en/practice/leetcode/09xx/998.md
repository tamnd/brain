## Solution

## Problem Understanding

The problem gives us the root of an existing maximum binary tree and a new integer value `val` that should be appended to the original array used to build that tree.

A maximum binary tree has a very specific construction rule. For any subarray, the largest value becomes the root, everything to the left becomes the left subtree, and everything to the right becomes the right subtree. Because all values are unique, the structure of the tree is uniquely determined.

The important detail is that we are not given the original array directly. Instead, we are only given the already constructed tree. We are told that a new array `b` is formed by appending `val` to the end of the original array `a`. We must return the maximum binary tree that would result from reconstructing the tree from `b`.

The key observation is that appending a value to the end of the array only affects the right side of the tree. The existing left relationships inside the tree remain valid. This allows us to avoid reconstructing the entire tree from scratch.

The constraints are small, at most 100 nodes, so even a less efficient solution would pass. However, the goal is to understand the structure of maximum binary trees and derive the optimal insertion strategy.

Several edge cases are important:

- If `val` is larger than the current root, the new value becomes the new root of the entire tree.
- If `val` is smaller than the root, it must be inserted somewhere in the right subtree.
- If the tree only contains one node, insertion logic must still work correctly.
- Because values are unique, we never need to handle duplicate comparisons.

## Approaches

### Brute Force Approach

The brute force solution reconstructs the original array from the tree, appends `val`, and then rebuilds the maximum binary tree from scratch.

To recover the original array, we perform an inorder traversal of the maximum binary tree. This works because the inorder traversal of a maximum binary tree reproduces the original array used to build it.

After obtaining the array:

1. Append `val`
2. Rebuild the maximum binary tree recursively
3. Return the new root

This solution is correct because it exactly follows the original construction definition from the problem statement.

However, rebuilding the tree repeatedly searches for maximum elements inside subarrays, which can become inefficient.

### Optimal Approach

The optimal solution uses the structural property of maximum binary trees.

Since `val` is appended to the end of the array, it can only appear somewhere along the right spine of the tree.

There are only two possibilities:

- `val` is greater than the current root:

- Then `val` becomes the new root
- The entire existing tree becomes its left child
- `val` is smaller than the current root:

- Then the insertion must occur inside the right subtree
- We recursively continue searching on the right side

This works because appending to the array never changes the relative structure of elements on the left side.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Recover array and rebuild tree from scratch |
| Optimal | O(h) | O(h) | Traverse only the right spine of the tree |

Here, `h` is the height of the tree. In the worst case, `h = n`.

## Algorithm Walkthrough

1. Start at the current root node.

We compare `val` with the current node's value because the maximum tree property depends entirely on relative ordering.
2. If `root` is `None`, create a new node with `val`.

This is the insertion base case. Eventually, recursion reaches an empty right child position.
3. If `val` is greater than `root.val`, create a new node containing `val`.

Since `val` is larger than every value in the current subtree, it must become the new root of this subtree.
4. Attach the old subtree as the left child of the new node.

This preserves the maximum tree structure because all previous elements appeared before `val` in the array.
5. Return the new node.

The subtree has now been correctly rebuilt.
6. Otherwise, recurse into the right subtree.

Since `val` was appended to the end of the original array, it cannot affect the left subtree. We only continue searching on the right side.
7. Assign the recursive result back to `root.right`.

This reconnects the modified subtree after insertion.
8. Return the current root.

The current subtree remains valid after updating its right child.

### Why it works

The correctness comes from the construction rule of maximum binary trees. Appending a value at the end of the array only affects nodes that lie along the rightmost path. If the appended value is larger than a subtree root, it becomes the new root of that subtree. Otherwise, it belongs somewhere deeper in the right subtree. This recursive rule exactly mirrors how the maximum tree would be rebuilt from the appended array.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import Optional

class Solution:
    def insertIntoMaxTree(
        self,
        root: Optional[TreeNode],
        val: int
    ) -> Optional[TreeNode]:

        if not root:
            return TreeNode(val)

        if val > root.val:
            new_root = TreeNode(val)
            new_root.left = root
            return new_root

        root.right = self.insertIntoMaxTree(root.right, val)

        return root
```

The implementation directly follows the recursive logic from the algorithm walkthrough.

The first condition handles the empty subtree case. When recursion reaches a `None` node, we insert the new value there.

The second condition checks whether the new value is larger than the current root. If it is, the new value must become the new root of the subtree. The old subtree becomes the left child because all existing elements appeared earlier in the original array.

If the new value is smaller, we recursively process the right subtree. We assign the recursive result back to `root.right` to preserve tree connections.

Finally, we return the current root after updating its right subtree.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */

func insertIntoMaxTree(root *TreeNode, val int) *TreeNode {
    if root == nil {
        return &TreeNode{Val: val}
    }

    if val > root.Val {
        newRoot := &TreeNode{Val: val}
        newRoot.Left = root
        return newRoot
    }

    root.Right = insertIntoMaxTree(root.Right, val)

    return root
}
```

The Go implementation is structurally identical to the Python version.

The main difference is explicit pointer handling. Go uses `nil` instead of `None`, and tree nodes are manipulated through pointers. A new node is allocated using `&TreeNode{Val: val}`.

There are no integer overflow concerns because values are very small, at most 100.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(h) | Only nodes along the right spine are visited |
| Space | O(h) | Recursive call stack depth equals tree height |

The algorithm never traverses the entire tree unless the tree is completely skewed. Because insertion only affects the right side, the runtime depends on the height of the tree rather than the total number of nodes.

In the worst case, the tree may degenerate into a linked list, giving `O(n)` complexity. In balanced cases, the complexity is closer to `O(log n)`.

## Edge Cases

One important edge case occurs when `val` is larger than the current root. This can easily lead to bugs if the old tree is not properly attached as the left child of the new root. The implementation handles this by explicitly creating a new node and assigning the previous root to `new_root.left`.

Another edge case is when the insertion must travel all the way down the right side of the tree. A naive implementation might forget to reconnect recursive return values. The solution avoids this by assigning the recursive result back to `root.right`.

A third important edge case is a tree containing only one node. In this scenario, either the new value becomes the new root or it becomes the right child. The recursive base case and comparison logic naturally handle both situations without requiring special-case code.

A subtle edge case involves subtree replacement. Suppose `val` is larger than a node inside the right subtree but smaller than the overall root. In that situation, `val` must become the new root of that subtree while preserving the previous subtree as its left child. The recursive structure of the algorithm correctly maintains this relationship.
