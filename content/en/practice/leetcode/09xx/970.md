## Solution

## Problem Understanding

The problem asks us to generate every integer of the form:

$$x^i + y^j$$

where both exponents satisfy:

$$i \ge 0,\quad j \ge 0$$

and the resulting value must be less than or equal to `bound`.

A number that satisfies this condition is called a powerful integer.

The input consists of three integers:

- `x`, the base for the first power sequence
- `y`, the base for the second power sequence
- `bound`, the maximum allowed value

The output should contain every distinct value that can be written as a sum of powers of `x` and `y` within the bound. The order does not matter, but duplicates are not allowed.

The key detail is that exponents can start at zero. Since any number raised to the power of zero equals one, the smallest possible powerful integer is:

$$x^0 + y^0 = 1 + 1 = 2$$

unless the bound is smaller than 2.

The constraints are relatively small:

- `x` and `y` are at most 100
- `bound` is at most $10^6$

These limits strongly suggest that brute force over all integers up to `bound` is unnecessary. Instead, we should directly enumerate powers of `x` and `y`.

An important observation is that powers grow exponentially. Even for small bases like 2 or 3, the number of powers less than $10^6$ is quite small. For example:

$$2^{20} = 1,048,576$$

so there are only about 20 relevant powers of 2.

There are several important edge cases:

- When `bound < 2`, no valid powerful integer exists because the minimum possible sum is 2.
- When `x == 1`, powers of `x` never change because $1^i = 1$ for all `i`. A naive loop multiplying by `x` forever would become an infinite loop.
- Similarly, when `y == 1`, powers of `y` remain constant.
- Different exponent pairs can produce the same result, so duplicates must be removed.

Because of these edge cases, careful loop control and duplicate handling are essential.

## Approaches

### Brute Force Approach

A straightforward brute force solution would examine every integer from 1 to `bound` and determine whether it can be expressed as:

$$x^i + y^j$$

for some non-negative integers `i` and `j`.

To do this, for each target number we could try many combinations of exponents and check whether the sum matches the target.

This approach is correct because it exhaustively checks all possibilities. However, it is inefficient because it repeatedly recomputes powers and performs unnecessary searches for every number up to `bound`.

With `bound` as large as $10^6$, iterating through all numbers becomes wasteful, especially since the number of actual power combinations is very small.

### Optimal Approach

The key insight is that powers grow exponentially.

Instead of iterating through every possible integer, we can directly generate all relevant powers of `x` and `y` that are less than or equal to `bound`.

For example, if `x = 2`:

$$1, 2, 4, 8, 16, 32, \dots$$

There are only logarithmically many such values before exceeding the bound.

We can therefore:

1. Generate every power of `x` within the bound.
2. Generate every power of `y` within the bound.
3. Compute every possible pairwise sum.
4. Keep only sums less than or equal to `bound`.
5. Store results in a set to avoid duplicates.

This dramatically reduces the search space.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(bound × log(bound)^2) | O(1) excluding output | Checks many unnecessary numbers |
| Optimal | O(log(bound)^2) | O(log(bound)^2) | Enumerates only relevant powers |

## Algorithm Walkthrough

1. Create an empty set to store powerful integers.

A set is used because different exponent combinations can generate the same value. Using a set automatically removes duplicates.
2. Generate powers of `x`.

Start with `x_power = 1`, representing $x^0$.

Continue multiplying by `x` until the value exceeds `bound`.

If `x == 1`, stop after the first iteration because repeated multiplication would never change the value.
3. For each power of `x`, generate powers of `y`.

Start with `y_power = 1`, representing $y^0$.

Continue multiplying by `y` while the sum:

$$x\_power + y\_power$$

remains within the bound.
4. Compute the current sum.

For every pair `(x_power, y_power)`:

$$current = x\_power + y\_power$$

If `current <= bound`, insert it into the set.
5. Handle the `y == 1` edge case.

If `y == 1`, stop after one iteration to avoid an infinite loop.
6. Move to the next power of `x`.

Multiply `x_power` by `x`.
7. Handle the `x == 1` edge case.

If `x == 1`, stop after one iteration.
8. Convert the set into a list and return it.

### Why it works

The algorithm systematically generates every valid power of `x` and every valid power of `y` within the bound. Every possible combination of these powers is considered exactly once, so every powerful integer is discovered.

Because the set stores unique sums, duplicates are automatically removed. Since powers larger than the bound can never contribute to a valid answer, limiting enumeration to powers within the bound guarantees completeness while avoiding unnecessary work.

## Python Solution

```
from typing import List

class Solution:
    def powerfulIntegers(self, x: int, y: int, bound: int) -> List[int]:
        result = set()

        x_power = 1

        while x_power <= bound:
            y_power = 1

            while x_power + y_power <= bound:
                result.add(x_power + y_power)

                if y == 1:
                    break

                y_power *= y

            if x == 1:
                break

            x_power *= x

        return list(result)
```

The implementation follows the algorithm directly.

The `result` set stores all unique powerful integers. Using a set avoids duplicate values automatically, which is important because multiple exponent pairs can produce the same sum.

The outer loop generates powers of `x`. It begins with `1`, which corresponds to $x^0$. After each iteration, the value is multiplied by `x` to move to the next power.

Inside the outer loop, the inner loop generates powers of `y` in the same way. For every pair of powers, the algorithm computes the sum and inserts it into the set if it does not exceed the bound.

The checks for `x == 1` and `y == 1` are critical. Without them, multiplying by 1 would never change the power value, resulting in infinite loops.

Finally, the set is converted to a list because LeetCode expects a list return type.

## Go Solution

```
func powerfulIntegers(x int, y int, bound int) []int {
    resultSet := make(map[int]bool)

    xPower := 1

    for xPower <= bound {
        yPower := 1

        for xPower+yPower <= bound {
            resultSet[xPower+yPower] = true

            if y == 1 {
                break
            }

            yPower *= y
        }

        if x == 1 {
            break
        }

        xPower *= x
    }

    result := []int{}

    for value := range resultSet {
        result = append(result, value)
    }

    return result
}
```

The Go solution closely mirrors the Python implementation.

Since Go does not have a built-in set type, a `map[int]bool` is used to simulate a set. Keys represent unique powerful integers.

Slices are used for the final return value. The order is arbitrary, which matches the problem requirements.

Integer overflow is not an issue under the given constraints because all values remain below $10^6$.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(log(bound)^2) | Number of powers for each base is logarithmic |
| Space | O(log(bound)^2) | Set stores all generated sums |

The number of powers generated for `x` is approximately:

$$\log_x(bound)$$

and similarly for `y`.

Since we examine every pair of generated powers, the total work becomes the product of the two logarithmic counts:

$$O(\log(bound)^2)$$

This is extremely efficient because exponential growth keeps the number of powers small.

## Edge Cases

One important edge case occurs when `bound < 2`. Since the smallest possible powerful integer is:

$$1 + 1 = 2$$

no valid answer exists. A careless implementation might still attempt unnecessary computation or incorrectly include values larger than the bound. The current implementation naturally handles this because the inner loop condition immediately fails.

Another critical edge case is when `x == 1`. In this scenario:

$$1^i = 1$$

for every exponent. Without special handling, repeatedly multiplying by `x` would never change the value, creating an infinite loop. The implementation prevents this by explicitly breaking after the first iteration.

The same issue exists for `y == 1`. The inner loop would otherwise never terminate because `y_power` would remain equal to 1 forever. The implementation includes a dedicated break condition to safely stop iteration.

A further subtle edge case involves duplicate sums. Different exponent combinations can produce the same integer. For example:

$$2^1 + 2^1 = 4$$

and

$$2^0 + 2^2 = 5$$

In larger examples, duplicate sums become common. Using a set guarantees that each powerful integer appears only once in the final output.
