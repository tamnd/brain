## Solution

## Problem Understanding

The problem asks whether two binary trees can be made identical by performing any number of flip operations. A flip operation consists of choosing a node and swapping its left and right children.

Two trees are considered flip equivalent if there exists some sequence of flips that transforms one tree into the other. The important detail is that flips can occur independently at any nodes, including multiple levels of the tree.

The input consists of two binary tree roots, `root1` and `root2`. Each tree node contains an integer value and pointers to left and right children. The goal is to return `true` if the trees are flip equivalent, otherwise return `false`.

The constraints are relatively small. Each tree contains at most 100 nodes, and every node value is unique within a tree. The uniqueness guarantee is very important because it removes ambiguity when comparing child subtrees. If node values were duplicated, determining which subtree should correspond to which would be much harder.

Several edge cases are important to consider:

- Both trees may be empty. Two empty trees are trivially equivalent.
- One tree may be empty while the other is not. In that case they cannot be equivalent.
- The roots may contain different values. Since flips only swap children and never change node values, the trees cannot become equal.
- A node may require a flip while another node at a different level does not. The algorithm must independently evaluate both possibilities at every node.
- Trees may already be identical without any flips. The algorithm should recognize this efficiently.

## Approaches

### Brute Force Approach

A brute-force solution would attempt every possible combination of flips in one tree and then compare the resulting structure against the other tree.

At each node, there are two choices:

- Do not flip the children
- Flip the children

If a tree contains `n` nodes, there can be up to `2^n` different flip configurations. For each configuration, we would still need to compare the resulting tree against the target tree, which takes `O(n)` time.

This approach is correct because it explicitly explores every possible transformed version of the tree. If any transformed tree matches the second tree, then the trees are flip equivalent.

However, the approach becomes exponentially expensive and is not practical even for moderate tree sizes.

### Optimal Recursive DFS Approach

The key insight is that we do not need to physically generate all flipped trees. Instead, we can recursively determine whether two subtrees are equivalent under either of the following conditions:

1. Their left children match and their right children match.
2. Their left child matches the other subtree's right child, and vice versa.

This naturally leads to a depth-first recursive solution.

At every pair of nodes:

- If both nodes are `None`, they are equivalent.
- If only one is `None`, they are not equivalent.
- If their values differ, they are not equivalent.
- Otherwise, recursively check both the non-flipped and flipped possibilities.

Because each node pair is processed once and each recursive call only performs constant additional work, the solution is efficient.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × 2^n) | O(n) | Tries every possible flip combination |
| Optimal | O(n) | O(h) | Recursive DFS comparing flipped and non-flipped states |

## Algorithm Walkthrough

1. Start with the two root nodes.

The recursion always compares corresponding nodes from the two trees. The goal is to determine whether the subtrees rooted at these nodes can be made identical.
2. Handle the base cases first.

If both nodes are `None`, the subtrees are equivalent because they are both empty.

If exactly one node is `None`, the structures differ and equivalence is impossible.

If the node values differ, the trees cannot become equal because flips only rearrange children and never modify values.
3. Check the non-flipped configuration.

Recursively compare:

- `root1.left` with `root2.left`
- `root1.right` with `root2.right`

If both comparisons return `true`, then the current subtrees are equivalent without flipping.
4. Check the flipped configuration.

Recursively compare:

- `root1.left` with `root2.right`
- `root1.right` with `root2.left`

If both comparisons return `true`, then the current subtrees are equivalent after flipping.
5. Return whether either configuration succeeds.

If either the non-flipped or flipped arrangement works, then the current subtree pair is flip equivalent.
6. Continue recursively until all nodes have been processed.

The recursion naturally traverses the entire tree structure.

### Why it works

The correctness comes from recursively verifying all valid structural possibilities at every node. A node can only match another node in two ways: either its children remain in the same order, or they are swapped. Since the algorithm exhaustively checks both valid configurations and recursively validates all descendant subtrees, it guarantees a correct result.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import Optional

class Solution:
    def flipEquiv(
        self,
        root1: Optional[TreeNode],
        root2: Optional[TreeNode]
    ) -> bool:

        if root1 is root2:
            return True

        if not root1 or not root2:
            return False

        if root1.val != root2.val:
            return False

        no_flip = (
            self.flipEquiv(root1.left, root2.left)
            and
            self.flipEquiv(root1.right, root2.right)
        )

        flip = (
            self.flipEquiv(root1.left, root2.right)
            and
            self.flipEquiv(root1.right, root2.left)
        )

        return no_flip or flip
```

The implementation directly follows the recursive reasoning from the algorithm walkthrough.

The first condition checks whether both node references are identical. This efficiently handles the case where both are `None`.

The second condition verifies whether exactly one node is missing. If so, the trees cannot be equivalent.

Next, the node values are compared. Since values are unique and flips do not alter values, mismatched values immediately invalidate equivalence.

The variable `no_flip` represents the case where both subtrees align naturally without swapping children.

The variable `flip` represents the case where the children must be swapped to achieve equivalence.

Finally, the function returns `True` if either arrangement works.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func flipEquiv(root1 *TreeNode, root2 *TreeNode) bool {
    if root1 == root2 {
        return true
    }

    if root1 == nil || root2 == nil {
        return false
    }

    if root1.Val != root2.Val {
        return false
    }

    noFlip := flipEquiv(root1.Left, root2.Left) &&
        flipEquiv(root1.Right, root2.Right)

    flip := flipEquiv(root1.Left, root2.Right) &&
        flipEquiv(root1.Right, root2.Left)

    return noFlip || flip
}
```

The Go implementation is almost identical to the Python version conceptually.

The primary difference is the use of `nil` instead of `None`. Pointer comparisons are straightforward in Go because tree nodes are represented as pointers.

Go does not require explicit type hints because the function signature already specifies parameter and return types.

There are no integer overflow concerns because node values are very small and arithmetic is not performed.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each node pair is processed once |
| Space | O(h) | Recursive call stack height |

The time complexity is linear because every node is visited once during the recursive traversal. At each node, only constant work is performed outside the recursive calls.

The space complexity depends on the recursion depth. In the worst case of a completely skewed tree, the recursion stack may contain `n` calls. In a balanced tree, the height is approximately `log n`.

## Edge Cases

One important edge case occurs when both trees are empty. A naive implementation might incorrectly treat empty input as invalid, but two empty trees are structurally identical. The implementation handles this immediately through the `root1 is root2` or `root1 == root2` check.

Another important case is when one subtree exists while the corresponding subtree in the other tree is missing. This can easily lead to null pointer errors if not checked carefully. The implementation explicitly verifies whether exactly one node is missing before attempting further recursion.

A particularly tricky edge case involves trees that only become equivalent after flips at multiple levels. Some incorrect solutions only attempt a single flip or only compare immediate children. The recursive approach correctly evaluates flipping independently at every node, allowing complex transformations throughout the tree.

A final edge case occurs when node values differ even though the structures appear compatible. Since flips only rearrange children and cannot change values, any value mismatch immediately makes equivalence impossible. The implementation checks values before exploring recursive branches, which also improves efficiency by pruning impossible cases early.
