## Solution

## Problem Understanding

In this problem, we are given a list of log strings. Each log contains an identifier followed by one or more words separated by spaces. The identifier is always the first token in the string, and everything after it represents the content of the log.

The logs fall into two categories:

- Letter-logs, where every word after the identifier consists only of lowercase English letters
- Digit-logs, where every word after the identifier consists only of digits

The task is to reorder the logs according to three specific rules.

First, all letter-logs must appear before any digit-log. Second, the letter-logs themselves must be sorted lexicographically based on their content, meaning everything after the identifier. If two letter-logs have identical content, then their identifiers are used as a tie-breaker. Third, digit-logs must preserve their original relative order from the input. This requirement means digit-logs behave like a stable sequence.

The input is an array of strings, and the output is another array of strings containing the reordered logs.

The constraints are relatively small, with at most 100 logs and each log having length at most 100 characters. Because the input size is modest, we do not need highly specialized optimization techniques. A sorting-based approach is more than sufficient.

Several edge cases are important to recognize upfront. Multiple letter-logs may share identical content, requiring identifier comparison as a secondary sort key. The input may contain only digit-logs or only letter-logs. Another important detail is that digit-logs must preserve their original ordering, so any solution that accidentally reorders them is incorrect. The problem also guarantees that every log contains an identifier and at least one additional token, so we never need to handle malformed input.

## Approaches

### Brute Force Approach

A straightforward brute-force strategy would be to separate letter-logs and digit-logs manually, then repeatedly compare and reorder the letter-logs using a simple sorting algorithm such as bubble sort or selection sort.

For each pair of letter-logs, we would:

1. Extract the identifier and content
2. Compare contents lexicographically
3. If contents are equal, compare identifiers
4. Swap logs if necessary

After sorting the letter-logs, we would append the digit-logs in their original order.

This approach works because it explicitly follows the problem's ordering rules. However, using a quadratic sorting algorithm is inefficient compared to built-in sorting methods. While the constraints are small enough that this would still pass, it is not considered optimal engineering practice.

### Optimal Approach

The key observation is that the problem naturally maps to custom sorting.

We can define a sorting key for each log:

- Letter-logs should come first
- Letter-logs should sort by content, then identifier
- Digit-logs should remain in original order

Python's built-in sorting is stable, which is extremely important here. Stability guarantees that equal elements retain their original relative order. By assigning all digit-logs the same sorting priority, they automatically remain in their original sequence.

The optimal solution uses a custom comparator or sorting key:

- For a letter-log, return a tuple like `(0, content, identifier)`
- For a digit-log, return `(1,)`

Since tuples are compared lexicographically, all letter-logs come before digit-logs because `0 < 1`. Among letter-logs, sorting proceeds by content and then identifier.

This approach is concise, efficient, and directly matches the problem requirements.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n² × m) | O(n) | Uses manual quadratic sorting of letter-logs |
| Optimal | O(n log n × m) | O(n) | Uses stable sorting with a custom key |

Here, `n` is the number of logs and `m` is the average log length.

## Algorithm Walkthrough

1. Iterate through each log and split it into two parts: the identifier and the remaining content. We only need to split once because the identifier is always the first token.
2. Determine whether the log is a letter-log or digit-log. We can inspect the first character of the content. If it is a digit, then the entire content is numeric because the problem guarantees consistency.
3. Define a sorting key for each log. For letter-logs, return a tuple containing:

- `0` to ensure letter-logs come first
- The content portion for primary lexicographical sorting
- The identifier for tie-breaking
4. For digit-logs, return a tuple containing only `1`. Since all digit-logs receive the same key and Python sorting is stable, they remain in their original order.
5. Apply the built-in sorting function using the custom key function.
6. Return the sorted list.

### Why it works

The algorithm works because the sorting key directly encodes the problem requirements. Letter-logs receive smaller priority values than digit-logs, ensuring they appear first. Among letter-logs, lexicographical ordering of content and identifiers guarantees the correct ordering. Stable sorting preserves the relative order of digit-logs because all digit-logs share the same sorting key.

## Python Solution

```
from typing import List

class Solution:
    def reorderLogFiles(self, logs: List[str]) -> List[str]:

        def sort_key(log: str):
            identifier, content = log.split(" ", 1)

            if content[0].isdigit():
                return (1,)
            
            return (0, content, identifier)

        return sorted(logs, key=sort_key)
```

The implementation begins by defining a helper function called `sort_key`. This function computes the sorting priority for each log.

The statement `log.split(" ", 1)` separates the identifier from the remainder of the content. Using `1` as the second argument ensures that only the first space is used for splitting, preserving the remainder exactly as written.

The check `content[0].isdigit()` determines whether the log is a digit-log. Because the problem guarantees that all words after the identifier belong to the same category, checking only the first character is sufficient.

Digit-logs return the tuple `(1,)`, which places them after all letter-logs. Letter-logs return `(0, content, identifier)`, which ensures correct lexicographical ordering.

Finally, Python's built-in `sorted()` function applies stable sorting using the custom key.

## Go Solution

```
package main

import (
	"sort"
	"strings"
	"unicode"
)

func reorderLogFiles(logs []string) []string {

	sort.SliceStable(logs, func(i, j int) bool {

		id1, content1 := splitLog(logs[i])
		id2, content2 := splitLog(logs[j])

		isDigit1 := unicode.IsDigit(rune(content1[0]))
		isDigit2 := unicode.IsDigit(rune(content2[0]))

		if !isDigit1 && !isDigit2 {

			if content1 == content2 {
				return id1 < id2
			}

			return content1 < content2
		}

		if !isDigit1 && isDigit2 {
			return true
		}

		if isDigit1 && !isDigit2 {
			return false
		}

		return false
	})

	return logs
}

func splitLog(log string) (string, string) {

	parts := strings.SplitN(log, " ", 2)

	return parts[0], parts[1]
}
```

The Go implementation uses `sort.SliceStable`, which is essential because digit-logs must preserve their original ordering. Unlike Python, Go does not provide tuple-based sorting keys directly, so we implement explicit comparison logic inside the comparator function.

The helper function `splitLog` separates the identifier from the content using `strings.SplitN`.

The comparator handles four cases:

- Both logs are letter-logs
- First is a letter-log and second is a digit-log
- First is a digit-log and second is a letter-log
- Both are digit-logs

When both are digit-logs, the comparator returns `false`, allowing stable sorting to preserve their original order.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n × m) | Sorting `n` logs, each comparison may examine up to `m` characters |
| Space | O(n) | Sorting and tuple creation require additional memory |

The dominant cost comes from sorting. Each comparison between logs may involve comparing strings whose lengths are proportional to the log size. Since sorting performs `O(n log n)` comparisons, the total complexity becomes `O(n log n × m)`.

The auxiliary space usage is linear because sorting implementations and temporary tuples require additional memory proportional to the number of logs.

## Edge Cases

One important edge case occurs when multiple letter-logs share identical content. A naive implementation might sort only by content and ignore identifiers, producing nondeterministic ordering. The implementation handles this correctly by including the identifier as the second sorting criterion.

Another critical edge case is when the input contains only digit-logs. Since digit-logs must preserve their original relative order, any unstable sorting method could accidentally rearrange them. The solution avoids this issue by assigning all digit-logs the same sorting priority and relying on stable sorting behavior.

A third edge case is when the input contains only letter-logs. In this scenario, the algorithm effectively becomes a normal lexicographical sort based on content and identifier. The implementation naturally handles this because every log receives the letter-log sorting tuple.

A subtle edge case involves logs with very similar prefixes, such as `"art can"` and `"art zero"`. Lexicographical comparison must examine characters beyond the shared prefix. Built-in string comparison correctly handles this without requiring manual parsing logic.

Finally, the smallest valid input size, a single log, must also work correctly. Since sorting a one-element list is trivial, the algorithm immediately returns the same list unchanged.
