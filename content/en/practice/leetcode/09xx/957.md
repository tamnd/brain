## Solution

## Problem Understanding

The problem gives us a row of exactly eight prison cells, where each cell is either occupied (`1`) or vacant (`0`). Every day, the state of the prison changes simultaneously according to a fixed rule based on neighboring cells.

For any cell in the middle of the row, meaning indices `1` through `6`, the next state depends on its immediate left and right neighbors:

- If both neighbors are the same, either both occupied or both vacant, the current cell becomes occupied (`1`)
- Otherwise, it becomes vacant (`0`)

The first and last cells are special because they do not have two neighbors. Since the rule requires two adjacent neighbors, these edge cells always become vacant (`0`) after the first day.

The input consists of:

- `cells`, an array of length `8`
- `n`, the number of days to simulate

The output should be the exact configuration of the prison after applying the transition rule for `n` days.

The main challenge comes from the constraint:

```
1 <= n <= 10^9
```

A billion simulation steps is far too large for straightforward day-by-day simulation. Even though there are only eight cells, a naive approach that loops `n` times would be too slow for large inputs.

An important observation is that the number of possible states is actually very small. Since each of the eight cells can only be `0` or `1`, there are at most:

$2^8 = 256$

possible configurations.

In practice, the first and last cells always become `0`, so only the six middle cells vary after day one. That means the true number of recurring states is at most:

$2^6 = 64$

This immediately suggests that the system must eventually enter a cycle.

Several edge cases are important:

- Very large values of `n`, where brute-force simulation becomes impossible
- Repeated states, which must be detected correctly to avoid unnecessary work
- Inputs where the cycle begins quickly
- Inputs where `n` becomes zero immediately after cycle reduction
- Correct handling of the first and last cells, which must always become `0`

## Approaches

### Brute-Force Simulation

The most direct solution is to simulate the prison day by day.

For each day:

- Create a new array of length `8`
- Set the first and last cells to `0`
- For each middle cell, compare its left and right neighbors
- If the neighbors match, set the new value to `1`
- Otherwise set it to `0`
- Replace the current state with the new state

This approach is easy to understand and guaranteed to produce the correct answer because it directly follows the problem rules.

However, the time complexity is proportional to `n`. Since `n` can be as large as `10^9`, this approach is not feasible.

### Optimal Approach, Cycle Detection

The key insight is that the prison has only a limited number of possible states.

Once a state repeats, the sequence of future states will repeat forever because the transition function is deterministic. That means the process forms a cycle.

Instead of simulating all `n` days, we can:

- Store every previously seen state
- Detect when a state repeats
- Compute the cycle length
- Skip unnecessary iterations using modulo arithmetic

Because there are at most 64 meaningful states after the first day, the cycle appears very quickly.

This transforms a potentially billion-step simulation into a tiny bounded computation.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Simulates every day directly |
| Optimal | O(min(n, 64)) | O(64) | Uses cycle detection to skip repeated states |

## Algorithm Walkthrough

1. Start with the initial prison configuration and create a hash map to track previously seen states.

The hash map stores:

- The prison state as the key
- The remaining number of days when that state was first seen

We use a tuple representation because lists are not hashable in Python.
2. Repeat the simulation while `n > 0`.

At each iteration:

- Convert the current state into a tuple
- Check whether this state has already appeared
3. If the state has been seen before, a cycle has been detected.

Since the prison evolution is deterministic, revisiting the same state means all future transitions will repeat in the same order.

The cycle length is:

$\text{cycle length} = \text{previous remaining days} - \text{current remaining days}$
4. Reduce the remaining simulation work using modulo arithmetic.

Instead of simulating every remaining day, compute:

$n = n \bmod \text{cycle length}$

This skips all full repetitions of the cycle.
5. Store the current state if it has not been seen before.

This allows future cycle detection.
6. If `n` is still greater than zero, simulate one day transition.

Create a new array initialized with zeros.

For each middle index `i` from `1` to `6`:

- If `cells[i-1] == cells[i+1]`, set `next_cells[i] = 1`
- Otherwise leave it as `0`
7. Replace the current state with the newly computed state and decrement `n`.
8. Continue until `n == 0`.

At this point, the current state is the final answer.

### Why it works

The algorithm works because the prison state transition is deterministic. A given state always produces exactly one next state.

Since there are finitely many possible states, repeated simulation must eventually revisit a previous configuration. Once that happens, all future states repeat cyclically.

By detecting the cycle and reducing the remaining number of steps modulo the cycle length, we avoid redundant work while still arriving at the exact same final configuration.

## Python Solution

```
from typing import List

class Solution:
    def prisonAfterNDays(self, cells: List[int], n: int) -> List[int]:
        seen = {}

        while n > 0:
            state = tuple(cells)

            if state in seen:
                cycle_length = seen[state] - n
                n %= cycle_length

            seen[state] = n

            if n > 0:
                n -= 1

                next_cells = [0] * 8

                for i in range(1, 7):
                    if cells[i - 1] == cells[i + 1]:
                        next_cells[i] = 1

                cells = next_cells

        return cells
```

The implementation begins by creating a dictionary called `seen`. This dictionary records previously encountered prison states along with the remaining value of `n` when the state appeared.

Inside the loop, the current state is converted into a tuple so it can be used as a dictionary key. If the state already exists in `seen`, a cycle has been detected. The cycle length is computed using the difference between the previously recorded remaining days and the current remaining days.

The algorithm then reduces `n` using modulo arithmetic. This effectively skips complete repetitions of the cycle.

After cycle handling, the current state is stored in the dictionary.

The next section computes the following day's prison configuration. A fresh array initialized to zeros is used because the edge cells must always remain vacant. For indices `1` through `6`, the code compares neighboring cells and assigns `1` when the neighbors match.

Finally, the current state is updated and the loop continues until all remaining days have been processed.

## Go Solution

```
package main

func prisonAfterNDays(cells []int, n int) []int {
	seen := make(map[[8]int]int)

	for n > 0 {
		var state [8]int

		for i := 0; i < 8; i++ {
			state[i] = cells[i]
		}

		if prevN, exists := seen[state]; exists {
			cycleLength := prevN - n
			n %= cycleLength
		}

		seen[state] = n

		if n > 0 {
			n--

			nextCells := make([]int, 8)

			for i := 1; i <= 6; i++ {
				if cells[i-1] == cells[i+1] {
					nextCells[i] = 1
				}
			}

			cells = nextCells
		}
	}

	return cells
}
```

The Go implementation follows the same logic as the Python version, but there are some language-specific differences.

Go slices cannot be used as hash map keys, so the current prison state is copied into a fixed-size array `[8]int`. Arrays in Go are comparable and can therefore be used as map keys safely.

The transition computation uses `make([]int, 8)` to allocate the next state. Since Go initializes integers to zero automatically, the edge cells naturally remain vacant.

Integer overflow is not a concern because all computations remain within standard integer ranges.

## Edge Cases

One important edge case is when `n` is extremely large, such as `10^9`. A naive implementation that simulates every day would time out. The cycle detection logic prevents this by identifying repeating states and reducing the remaining work using modulo arithmetic.

Another important case is the handling of the first and last cells. Since these cells never have two neighbors, they must always become `0`. A buggy implementation might accidentally update them using invalid neighbor checks. This implementation avoids the issue by initializing the next state array entirely with zeros and only updating indices `1` through `6`.

A third subtle case occurs when a cycle is detected and `n % cycle_length == 0`. In this situation, the algorithm must avoid performing extra transitions. The implementation handles this naturally because once `n` becomes zero, the simulation loop terminates immediately and returns the current state.

A final edge case involves repeated states appearing very early. Because the state space is so small, cycles can begin almost immediately. The hash map correctly records every encountered configuration, ensuring that cycles are detected regardless of when they begin.
