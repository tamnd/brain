## Solution

## Problem Understanding

This problem gives us an `8 x 8` chessboard represented as a matrix of characters. Each cell contains one of four possible values:

- `'R'` represents the white rook
- `'p'` represents a black pawn
- `'B'` represents a white bishop
- `'.'` represents an empty square

The board always contains exactly one rook.

The rook moves like a standard chess rook. It can move horizontally or vertically in four directions:

- Up
- Down
- Left
- Right

The rook may continue moving until:

- It reaches the edge of the board
- It encounters another piece

A rook can capture a pawn only if the pawn is the first piece encountered in that direction. If a bishop blocks the path before the pawn, the pawn cannot be captured.

The goal is to return the total number of pawns the rook can attack in one move.

The constraints are very small. The board size is fixed at `8 x 8`, which means there are at most 64 cells. Because of this, even relatively inefficient solutions would still run quickly. However, the problem is really testing careful grid traversal and simulation logic.

Several edge cases are important:

- The rook may have no capturable pawns at all.
- Bishops may immediately block movement in one or more directions.
- Multiple pawns may exist in the same direction, but only the first reachable one matters.
- The rook may be positioned near the edge of the board.
- A pawn may be directly adjacent to the rook.
- A bishop may be directly adjacent to the rook, completely blocking that direction.

The problem guarantees that there is exactly one rook, so we never need to handle missing or multiple rooks.

## Approaches

### Brute Force Approach

A straightforward brute force solution is to simulate every possible rook move from the rook's position.

First, we locate the rook on the board. Then, for each of the four directions, we continue moving one square at a time until:

- We leave the board
- We hit a bishop
- We hit a pawn

If we encounter a pawn before any bishop, we count it as capturable and stop searching further in that direction because the rook cannot move through pieces.

This approach is correct because chess rook movement is deterministic. Every square along a direction must be checked in order until movement is blocked.

Even though this solution is technically brute force, the board size is constant, so the runtime is extremely small in practice.

### Key Insight for the Optimal Approach

The important observation is that the rook only cares about the first non-empty square encountered in each of the four directions.

We do not need to explore arbitrary paths or perform graph traversal. The rook's movement is linear and constrained to rows and columns.

Therefore, the optimal solution is simply:

1. Find the rook.
2. Scan outward in four directions.
3. Stop immediately when:

- A bishop blocks the path
- A pawn is captured
- The board boundary is reached

Because each direction is scanned at most once, the implementation is both simple and efficient.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(8 × 8) | O(1) | Simulates movement from the rook in all directions |
| Optimal | O(8 × 8) | O(1) | Stops scanning immediately once a blocker or pawn is found |

## Algorithm Walkthrough

1. Traverse the entire board to locate the rook.

Since there is exactly one rook, we scan row by row until we find the cell containing `'R'`. We store its row and column coordinates.
2. Define the four movement directions.

The rook can move:

- Up: `(-1, 0)`
- Down: `(1, 0)`
- Left: `(0, -1)`
- Right: `(0, 1)`

These direction vectors allow us to write one generic scanning loop.
3. For each direction, move outward one square at a time.

Starting from the rook's position, repeatedly add the direction offsets to the current coordinates.
4. Check board boundaries.

If the next position falls outside the board, stop scanning in that direction because the rook cannot move further.
5. Check whether the square contains a bishop.

If the square contains `'B'`, stop immediately. Bishops block the rook's movement, so no squares beyond the bishop are reachable.
6. Check whether the square contains a pawn.

If the square contains `'p'`, increment the answer by one and stop scanning in that direction. The rook can capture that pawn, but cannot continue moving beyond it.
7. Continue scanning empty squares.

If the square contains `'.'`, continue moving in the same direction.
8. After processing all four directions, return the total number of capturable pawns.

### Why it works

The algorithm works because rook movement is strictly linear along rows and columns. For any direction, only the first non-empty square matters:

- If it is a bishop, movement is blocked.
- If it is a pawn, the pawn is capturable.

By scanning outward in order from the rook, we exactly simulate legal rook movement according to chess rules.

## Python Solution

```
from typing import List

class Solution:
    def numRookCaptures(self, board: List[List[str]]) -> int:
        rook_row = 0
        rook_col = 0

        # Find the rook
        for row in range(8):
            for col in range(8):
                if board[row][col] == 'R':
                    rook_row = row
                    rook_col = col
                    break

        captures = 0

        # Four movement directions
        directions = [
            (-1, 0),  # up
            (1, 0),   # down
            (0, -1),  # left
            (0, 1)    # right
        ]

        # Scan in each direction
        for dr, dc in directions:
            row = rook_row + dr
            col = rook_col + dc

            while 0 <= row < 8 and 0 <= col < 8:
                if board[row][col] == 'B':
                    break

                if board[row][col] == 'p':
                    captures += 1
                    break

                row += dr
                col += dc

        return captures
```

The implementation begins by locating the rook through a nested loop over the board.

After finding the rook's coordinates, we define the four possible movement directions. Using direction vectors avoids duplicated code and keeps the logic compact.

For each direction, we move outward step by step. The `while` loop continues as long as the current position remains inside the board.

Inside the loop:

- A bishop immediately stops movement.
- A pawn increases the capture count and also stops movement.
- Empty squares allow scanning to continue.

At the end, the total number of capturable pawns is returned.

## Go Solution

```
func numRookCaptures(board [][]byte) int {
    rookRow, rookCol := 0, 0

    // Find the rook
    for row := 0; row < 8; row++ {
        for col := 0; col < 8; col++ {
            if board[row][col] == 'R' {
                rookRow = row
                rookCol = col
            }
        }
    }

    captures := 0

    directions := [][]int{
        {-1, 0}, // up
        {1, 0},  // down
        {0, -1}, // left
        {0, 1},  // right
    }

    // Scan in each direction
    for _, dir := range directions {
        row := rookRow + dir[0]
        col := rookCol + dir[1]

        for row >= 0 && row < 8 && col >= 0 && col < 8 {
            if board[row][col] == 'B' {
                break
            }

            if board[row][col] == 'p' {
                captures++
                break
            }

            row += dir[0]
            col += dir[1]
        }
    }

    return captures
}
```

The Go implementation follows the same logic as the Python version.

One difference is that the board is represented as `[][]byte` instead of a list of strings. Character comparisons therefore use byte literals such as `'R'`, `'B'`, and `'p'`.

Go does not require special handling for integer overflow here because all coordinates remain within the small fixed board size.

## Edge Cases

One important edge case occurs when a bishop is directly adjacent to the rook. A buggy implementation might continue scanning beyond the bishop and incorrectly count pawns behind it. The solution avoids this by immediately breaking out of the directional scan when a bishop is encountered.

Another edge case happens when a pawn is immediately adjacent to the rook. Some implementations mistakenly continue scanning after capturing a pawn. In chess, the rook cannot move through captured pieces during the same move. The algorithm handles this correctly by incrementing the capture count and immediately stopping further scanning in that direction.

A third important edge case is when the rook is positioned near the border of the board. Poor boundary handling can easily cause index out of range errors. The implementation carefully checks that row and column coordinates remain within `[0, 7]` before accessing the board.

Another subtle case involves multiple pawns in the same direction. Only the first pawn is reachable because the rook cannot move through pieces. Since the algorithm stops after the first pawn encountered, it naturally enforces this rule correctly.
