# LeetCode 912 - Sort an Array

## Problem Understanding

The problem gives us an integer array `nums` and asks us to sort it in ascending order. The final result must contain exactly the same elements as the original array, but arranged from smallest to largest.

The important restriction is that we cannot use built in sorting functions such as Python's `sort()` or Go's `sort.Ints()`. We must implement the sorting algorithm ourselves.

The problem also requires an `O(n log n)` time complexity solution. This immediately rules out inefficient quadratic algorithms like Bubble Sort, Selection Sort, and Insertion Sort for large inputs. Since the array length can be as large as `5 * 10^4`, an `O(n^2)` solution could require billions of operations in the worst case.

The constraints also tell us several important things:

- The input size is large enough that efficiency matters.
- Values may be negative.
- Duplicate values are allowed.
- The array always contains at least one element.

Because duplicates and negative numbers exist, the sorting algorithm must correctly preserve all values and compare them normally. The problem does not require stable sorting, but many efficient sorting algorithms naturally provide it.

Some important edge cases include:

- Arrays with only one element.
- Arrays already sorted in ascending order.
- Arrays sorted in descending order.
- Arrays containing many duplicate values.
- Arrays containing negative numbers.
- Arrays where all elements are identical.

A naive implementation may perform poorly on already sorted data or may mishandle duplicate values if partition logic is incorrect.

## Approaches

### Brute Force Approach

A straightforward solution is to repeatedly compare adjacent elements and swap them if they are out of order. This is the idea behind Bubble Sort.

For example, in every pass through the array, the largest remaining unsorted value "bubbles" toward the end. After enough passes, the entire array becomes sorted.

This approach works because every inversion in the array is eventually corrected through swaps. However, its time complexity is too slow for the problem constraints.

In the worst case, Bubble Sort performs approximately:

$$n + (n-1) + (n-2) + \dots + 1$$

comparisons, which simplifies to `O(n^2)`.

With `n = 50,000`, this becomes impractical.

### Optimal Approach

An efficient solution is Merge Sort.

Merge Sort uses the divide and conquer strategy:

1. Split the array into two halves.
2. Recursively sort each half.
3. Merge the two sorted halves together.

The key insight is that merging two already sorted arrays can be done efficiently in linear time. Since the array is repeatedly divided in half, the recursion depth becomes `log n`, and each level processes all `n` elements once.

This gives an overall complexity of:

$$O(n \log n)$$

which satisfies the problem requirements.

Merge Sort is also predictable because its worst case complexity remains `O(n log n)` regardless of input ordering.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Bubble Sort repeatedly swaps adjacent elements |
| Optimal | O(n log n) | O(n) | Merge Sort divides the array and merges sorted halves |

## Algorithm Walkthrough

### Merge Sort Steps

1. Check whether the current array has one or zero elements.

If an array contains at most one element, it is already sorted. This becomes the base case of the recursion and prevents infinite recursive calls.
2. Split the array into two halves.

Compute the midpoint and divide the array into:

- Left half
- Right half

Splitting reduces the problem into smaller subproblems that are easier to solve recursively.
3. Recursively sort the left half.

Apply Merge Sort again to the left portion until it becomes fully sorted.
4. Recursively sort the right half.

Apply the same process to the right portion.
5. Merge the two sorted halves.

Use two pointers:

- One pointer for the left array
- One pointer for the right array

Compare the current values from both halves:

- Append the smaller value to the merged result
- Move the corresponding pointer forward

This works because both halves are already sorted.
6. Append remaining elements.

One side may still contain leftover elements after the other side is exhausted. Since the remaining elements are already sorted, append them directly.
7. Return the merged sorted array.

Each recursive call returns a fully sorted version of its subarray.

### Why it works

Merge Sort works because of recursive decomposition and sorted merging.

The recursive calls guarantee that every subarray eventually becomes sorted. Once two smaller sorted arrays are available, the merge step constructs a larger sorted array by always choosing the smallest available element from the front of either half.

Since every merge preserves sorted order, the final result is a completely sorted array.

## Python Solution

```
from typing import List

class Solution:
    def sortArray(self, nums: List[int]) -> List[int]:
        def merge_sort(arr: List[int]) -> List[int]:
            if len(arr) <= 1:
                return arr

            mid = len(arr) // 2

            left_sorted = merge_sort(arr[:mid])
            right_sorted = merge_sort(arr[mid:])

            merged = []
            left_index = 0
            right_index = 0

            while left_index < len(left_sorted) and right_index < len(right_sorted):
                if left_sorted[left_index] <= right_sorted[right_index]:
                    merged.append(left_sorted[left_index])
                    left_index += 1
                else:
                    merged.append(right_sorted[right_index])
                    right_index += 1

            while left_index < len(left_sorted):
                merged.append(left_sorted[left_index])
                left_index += 1

            while right_index < len(right_sorted):
                merged.append(right_sorted[right_index])
                right_index += 1

            return merged

        return merge_sort(nums)
```

The implementation begins by defining an inner recursive function called `merge_sort`.

The base case checks whether the array length is less than or equal to one. Such arrays are already sorted, so they are returned immediately.

The array is then divided into two halves using the midpoint index. Recursive calls sort both halves independently.

After both recursive calls return sorted arrays, the merge phase begins. Two indices track the current position inside each sorted half.

The algorithm repeatedly compares the current elements from both halves and appends the smaller one into the `merged` list. This guarantees that elements are added in ascending order.

After one half becomes exhausted, the remaining elements from the other half are appended directly because they are already sorted.

Finally, the merged sorted array is returned upward through the recursion stack until the entire input becomes sorted.

## Go Solution

```
func sortArray(nums []int) []int {
    return mergeSort(nums)
}

func mergeSort(arr []int) []int {
    if len(arr) <= 1 {
        return arr
    }

    mid := len(arr) / 2

    leftSorted := mergeSort(arr[:mid])
    rightSorted := mergeSort(arr[mid:])

    merged := make([]int, 0, len(arr))

    leftIndex := 0
    rightIndex := 0

    for leftIndex < len(leftSorted) && rightIndex < len(rightSorted) {
        if leftSorted[leftIndex] <= rightSorted[rightIndex] {
            merged = append(merged, leftSorted[leftIndex])
            leftIndex++
        } else {
            merged = append(merged, rightSorted[rightIndex])
            rightIndex++
        }
    }

    for leftIndex < len(leftSorted) {
        merged = append(merged, leftSorted[leftIndex])
        leftIndex++
    }

    for rightIndex < len(rightSorted) {
        merged = append(merged, rightSorted[rightIndex])
        rightIndex++
    }

    return merged
}
```

The Go implementation follows the same recursive Merge Sort structure as the Python version.

One important difference is slice handling. Go slices reference underlying arrays, so recursive calls use sub-slices like `arr[:mid]` and `arr[mid:]`.

The merged result is created using:

```
make([]int, 0, len(arr))
```

This preallocates capacity for efficiency and reduces repeated memory reallocations during appends.

Unlike Python, Go does not have built in dynamic list concatenation with the same flexibility, so `append` is used throughout the merge phase.

The implementation safely handles all edge cases, including empty subarrays and duplicate values.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | The array is divided into log n levels, and each level processes all n elements |
| Space | O(n) | Extra arrays are used during merging |

The recursion depth is `log n` because the array is repeatedly halved. At every recursion level, the merge operation touches every element exactly once. Multiplying these together gives `O(n log n)` time complexity.

The algorithm requires additional memory for temporary merged arrays, resulting in `O(n)` auxiliary space usage.

## Edge Cases

### Single Element Array

An array containing only one element is already sorted. Recursive algorithms can fail if they do not properly define this base case. Without it, recursion would continue infinitely.

The implementation handles this correctly using:

```
if len(arr) <= 1:
    return arr
```

This immediately terminates recursion for trivial arrays.

### Arrays with Duplicate Values

Duplicate values can expose bugs in merge logic, especially when pointer movement is implemented incorrectly.

For example:

```
[2,2,1,1]
```

The merge step must preserve all occurrences and avoid skipping elements accidentally.

The implementation compares values using `<=`, ensuring that duplicates are merged correctly and all elements appear in the final result.

### Reverse Sorted Arrays

A descending array such as:

```
[5,4,3,2,1]
```

is often a worst case input for simpler sorting algorithms like Bubble Sort or Insertion Sort.

Merge Sort handles this efficiently because its complexity does not depend on initial ordering. The recursive splitting and merging process still performs in `O(n log n)` time.

### Arrays Containing Negative Numbers

Some specialized sorting algorithms fail if they assume only nonnegative integers.

For example:

```
[-5,-1,-3]
```

Merge Sort relies only on comparisons between values, so negative numbers are naturally handled without any special logic.
