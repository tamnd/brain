## Solution

## Problem Understanding

This problem asks us to find the size of the largest connected component in a graph defined by an array of unique positive integers, `nums`. Each number in `nums` represents a node in the graph. There is an undirected edge between two nodes if the corresponding numbers share a common factor greater than 1. The expected output is a single integer representing the number of nodes in the largest group of numbers connected by shared factors.

The constraints tell us that the array can have up to 20,000 elements and each element can be as large as 100,000. All numbers are unique, which simplifies tracking nodes. A naive approach of checking every pair for common factors would be too slow because it would result in O(n²) comparisons, which is unacceptable for n = 20,000.

Important edge cases include numbers that are prime (which can only connect to multiples of themselves), numbers that are all powers of the same prime (which will form a fully connected component), and numbers that are pairwise coprime (which would result in many isolated nodes).

## Approaches

The brute-force approach would be to check every pair of numbers for a common factor greater than 1 and build the graph explicitly. This would involve iterating over all pairs of numbers, calculating the greatest common divisor (GCD), and then performing a graph traversal to find connected components. While correct, this approach is too slow for large inputs due to its O(n² * log(max_num)) time complexity, where log(max_num) comes from computing the GCD.

The optimal approach relies on a key insight: rather than checking every pair of numbers, we can group numbers by their prime factors. If two numbers share a prime factor, they are connected. This allows us to use a Union-Find (Disjoint Set Union, DSU) data structure to efficiently merge numbers that share any factor. First, we factorize each number into its prime factors, then union numbers through these factors. Finally, we count the size of each connected component using a hash map.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n² * log(max_num)) | O(n²) | Checks GCD for all pairs, too slow for n=20,000 |
| Optimal | O(n * sqrt(max_num) + n * α(n)) | O(n + max_num) | Uses prime factorization and Union-Find for efficient grouping |

## Algorithm Walkthrough

1. Initialize a Union-Find (DSU) data structure for all numbers up to the maximum in `nums`. Each number starts as its own component.
2. For each number in `nums`, factorize it into its prime factors. Each factor represents a potential connection point.
3. For each number, union it with all of its prime factors in the DSU. This ensures that numbers sharing a factor are in the same component.
4. After processing all numbers, iterate over `nums` again and find the representative (root) of each number using the DSU.
5. Use a hash map to count how many numbers belong to each root. This gives the size of each connected component.
6. Return the maximum value from the hash map, which represents the largest component size.

Why it works: The algorithm works because two numbers are connected if they share at least one prime factor. By unioning numbers through their prime factors, we maintain the invariant that all numbers in the same connected component have a path of shared factors between them. Counting the size of each root gives the correct largest connected component.

## Python Solution

```
from typing import List
from collections import defaultdict
import math

class UnionFind:
    def __init__(self, size: int):
        self.parent = list(range(size))
        self.rank = [0] * size

    def find(self, x: int) -> int:
        if self.parent[x] != x:
            self.parent[x] = self.find(self.parent[x])
        return self.parent[x]

    def union(self, x: int, y: int):
        px, py = self.find(x), self.find(y)
        if px == py:
            return
        if self.rank[px] < self.rank[py]:
            self.parent[px] = py
        elif self.rank[px] > self.rank[py]:
            self.parent[py] = px
        else:
            self.parent[py] = px
            self.rank[px] += 1

class Solution:
    def largestComponentSize(self, nums: List[int]) -> int:
        max_num = max(nums)
        uf = UnionFind(max_num + 1)
        
        def factors(x: int):
            for i in range(2, int(math.sqrt(x)) + 1):
                if x % i == 0:
                    yield i
                    while x % i == 0:
                        x //= i
            if x > 1:
                yield x
        
        for num in nums:
            for f in factors(num):
                uf.union(num, f)
        
        count = defaultdict(int)
        for num in nums:
            root = uf.find(num)
            count[root] += 1
        
        return max(count.values())
```

The Python implementation first defines a Union-Find class with path compression and union by rank to ensure nearly constant time operations. The `factors` function yields all prime factors of a number efficiently. Each number is unioned with its prime factors. Finally, the code counts the size of each connected component and returns the largest.

## Go Solution

```
package main

import (
    "math"
)

type UnionFind struct {
    parent []int
    rank   []int
}

func NewUnionFind(size int) *UnionFind {
    parent := make([]int, size)
    rank := make([]int, size)
    for i := 0; i < size; i++ {
        parent[i] = i
    }
    return &UnionFind{parent, rank}
}

func (uf *UnionFind) Find(x int) int {
    if uf.parent[x] != x {
        uf.parent[x] = uf.Find(uf.parent[x])
    }
    return uf.parent[x]
}

func (uf *UnionFind) Union(x, y int) {
    px, py := uf.Find(x), uf.Find(y)
    if px == py {
        return
    }
    if uf.rank[px] < uf.rank[py] {
        uf.parent[px] = py
    } else if uf.rank[px] > uf.rank[py] {
        uf.parent[py] = px
    } else {
        uf.parent[py] = px
        uf.rank[px]++
    }
}

func factors(x int) []int {
    res := []int{}
    for i := 2; i*i <= x; i++ {
        if x%i == 0 {
            res = append(res, i)
            for x%i == 0 {
                x /= i
            }
        }
    }
    if x > 1 {
        res = append(res, x)
    }
    return res
}

func largestComponentSize(nums []int) int {
    maxNum := 0
    for _, num := range nums {
        if num > maxNum {
            maxNum = num
        }
    }
    uf := NewUnionFind(maxNum + 1)

    for _, num := range nums {
        for _, f := range factors(num) {
            uf.Union(num, f)
        }
    }

    count := make(map[int]int)
    maxSize := 0
    for _, num := range nums {
        root := uf.Find(num)
        count[root]++
        if count[root] > maxSize {
            maxSize = count[root]
        }
    }

    return maxSize
}
```

The Go implementation follows the same logic. Differences include explicit slice creation for the Union-Find parent and rank arrays, and the factors function returns a slice instead of a generator. Map operations are used to count component sizes, and maxSize is tracked directly to avoid extra iteration.

## Worked Examples

Example 1: nums = [4,6,15,35]

1. Prime factors: 4 → [2], 6 → [2,3], 15 → [3,5], 35 → [5,7]
2. Union steps: union(4,2), union(6,2), union(6,3), union(15,3), union(15,5), union(35,5)
3. Connected components after unions: {4,6,15,35} all connected through shared factors 2,3,5
4. Count: {root → size} = {root=2 → 4}
5. Return 4

Similar step-by-step reasoning applies for examples 2 and 3, yielding output 2 and 8 respectively.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n * sqrt(max_num) + n * α(n)) | Factorization takes O(sqrt(max_num)) per number, Union-Find operations take nearly O(1) per operation |
| Space | O(n + max_num) | Union-Find stores parent and rank arrays, hash map stores component sizes |

The reasoning: Prime factorization dominates the time complexity, but union-find operations are efficient due to path compression and rank heuristics. Space is proportional to the range of numbers and the number of input numbers.
