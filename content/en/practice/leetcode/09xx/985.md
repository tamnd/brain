## Solution

## Problem Understanding

The problem gives us an integer array `nums` and a list of update operations called `queries`. Each query contains two values:

- `vali`, the value to add
- `indexi`, the index in the array to update

For every query, we must:

1. Update the array element:

```
nums[indexi] = nums[indexi] + vali
```
2. Compute the sum of all even numbers currently present in the array.
3. Store that sum in the result array.

The final output is an array where each entry corresponds to the sum of even numbers after processing the respective query.

The important detail is that queries are processed sequentially. Each update permanently changes `nums`, and future queries operate on the modified array.

For example, if the array starts as:

```
[1,2,3,4]
```

the initial even sum is:

```
2 + 4 = 6
```

If a query adds `1` to index `0`, the array becomes:

```
[2,2,3,4]
```

Now the even sum becomes:

```
2 + 2 + 4 = 8
```

The constraints are important:

- `nums.length <= 10^4`
- `queries.length <= 10^4`

A solution that recomputes the entire even sum after every query would require:

```
O(n * q)
```

which could reach:

```
10^4 * 10^4 = 10^8
```

operations. That is unnecessarily expensive and risks time limit issues.

Several edge cases matter:

- An even number may become odd after an update.
- An odd number may become even after an update.
- Negative even numbers still contribute to the even sum.
- Adding zero should still behave correctly.
- Arrays may contain only odd or only even values.
- The array may contain only one element.

The key challenge is efficiently maintaining the running sum of even numbers after each update.

## Approaches

### Brute Force Approach

The simplest approach is to process each query independently and recompute the sum of even numbers from scratch every time.

For each query:

1. Apply the update to the array.
2. Iterate through the entire array.
3. Add every even number to a running total.
4. Store the result.

This works because after every query we explicitly calculate the exact even-number sum for the current array state.

However, this approach is inefficient. If there are `q` queries and the array size is `n`, then each query requires scanning all `n` elements.

That gives a total complexity of:

```
O(n * q)
```

With both values potentially reaching `10^4`, this can become too slow.

### Optimal Incremental Approach

The important observation is that a single query changes only one array element.

That means the total sum of even numbers changes only because of that one modified value. Every other element remains unchanged.

Instead of recalculating the entire even sum after each query, we can maintain a running even sum and update it intelligently.

For each query:

- If the old value was even, remove it from the even sum before modification.
- Update the value.
- If the new value is even, add it to the even sum afterward.

This allows us to process each query in constant time.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * q) | O(1) | Recomputes even sum after every query |
| Optimal | O(n + q) | O(1) | Maintains running even sum incrementally |

## Algorithm Walkthrough

### Step-by-Step Process

1. Compute the initial sum of all even numbers in `nums`.

We begin by scanning the array once and adding every even number to a variable called `even_sum`.

This gives us the current even-number total before any queries are processed.
2. Create a result array.

We will append the updated even sum after every query.
3. Process each query one at a time.

Each query contains:

- `value_to_add`
- `index`
4. Check whether the current number at `nums[index]` is even.

If it is even, it currently contributes to `even_sum`.

Since the value is about to change, we must first remove its contribution:

```
even_sum -= nums[index]
```
5. Apply the update.

Modify the array element:

```
nums[index] += value_to_add
```
6. Check whether the updated number is even.

If the new value is even, it now contributes to the even-number sum.

Add it back:

```
even_sum += nums[index]
```
7. Append the current `even_sum` to the answer array.
8. Continue until all queries are processed.

### Why it works

The algorithm maintains an invariant:

> `even_sum` always equals the sum of all even numbers currently present in `nums`.

Before modifying a value, we remove its old contribution if necessary. After modification, we add its new contribution if necessary.

Since only one array element changes per query, updating the running total is sufficient to maintain correctness without rescanning the entire array.

## Python Solution

```
from typing import List

class Solution:
    def sumEvenAfterQueries(
        self,
        nums: List[int],
        queries: List[List[int]]
    ) -> List[int]:

        even_sum = 0

        for num in nums:
            if num % 2 == 0:
                even_sum += num

        result = []

        for value_to_add, index in queries:

            # Remove old contribution if current value is even
            if nums[index] % 2 == 0:
                even_sum -= nums[index]

            # Apply update
            nums[index] += value_to_add

            # Add new contribution if updated value is even
            if nums[index] % 2 == 0:
                even_sum += nums[index]

            result.append(even_sum)

        return result
```

The implementation begins by calculating the initial sum of all even numbers in the array. This establishes the running state that will be maintained throughout the algorithm.

For every query, the code first checks whether the current value contributes to the even sum. If it does, that contribution is removed before the update occurs.

After applying the update, the algorithm checks whether the new value is even. If it is, the value is added back into the running total.

This approach guarantees that `even_sum` always reflects the exact sum of even numbers in the current array state.

The result array stores the running even sum after each query.

## Go Solution

```
func sumEvenAfterQueries(nums []int, queries [][]int) []int {
	evenSum := 0

	for _, num := range nums {
		if num%2 == 0 {
			evenSum += num
		}
	}

	result := make([]int, 0, len(queries))

	for _, query := range queries {
		valueToAdd := query[0]
		index := query[1]

		// Remove old contribution if current value is even
		if nums[index]%2 == 0 {
			evenSum -= nums[index]
		}

		// Apply update
		nums[index] += valueToAdd

		// Add new contribution if updated value is even
		if nums[index]%2 == 0 {
			evenSum += nums[index]
		}

		result = append(result, evenSum)
	}

	return result
}
```

The Go implementation follows the same logic as the Python version.

A slice named `result` is initialized with capacity equal to the number of queries to reduce reallocations during appends.

Go integer arithmetic is safe here because the constraints are small enough that overflow is not a concern with standard `int`.

Unlike Python, Go does not have dynamic list typing, so explicit slice management is required.

- parity checks

Since there are `q` queries, total query processing takes `O(q)` time.

The algorithm uses only a few integer variables beyond the output array, so auxiliary space usage is constant.

## Edge Cases

### Even Number Becoming Odd

A common source of bugs occurs when an even number becomes odd after an update.

For example:

```
nums = [4]
query = [1, 0]
```

The value changes from `4` to `5`.

If the algorithm forgets to remove the old even contribution before updating, the running sum becomes incorrect.

The implementation handles this by subtracting the old value from `even_sum` before applying the update.

### Odd Number Becoming Even

Another tricky situation occurs when an odd number becomes even.

For example:

```
nums = [3]
query = [1, 0]
```

The value changes from `3` to `4`.

If the algorithm does not check the updated parity afterward, the new even number will never be included in the running total.

The implementation explicitly checks parity again after modification and adds the updated value if it is even.

### Negative Even Numbers

Negative values can still be even and must contribute to the sum correctly.

For example:

```
nums = [-2, -4]
```

The even sum is:

```
-6
```

Some incorrect implementations assume only positive even numbers matter.

The solution correctly uses the modulus operator to test parity and includes all even values, whether positive or negative.
