## Solution

## Problem Understanding

The problem gives an integer array `nums` whose length is exactly `2 * n`. Among all the values in the array, there are `n + 1` distinct numbers. Out of those distinct numbers, exactly one number appears repeatedly, and it appears exactly `n` times. Every other number appears only once.

The goal is to return the value that is repeated `n` times.

In simpler terms, the array is mostly made of unique values, except for one special value that occurs many times. Since the array length is `2n`, and one value appears `n` times, the remaining `n` positions are filled by `n` unique elements that each occur once.

For example, consider:

```
nums = [2,1,2,5,3,2]
```

The array length is `6`, so `n = 3`. The number `2` appears three times, while `1`, `5`, and `3` each appear once. Therefore, the correct answer is `2`.

The constraints are relatively small:

- `2 <= n <= 5000`
- Maximum array size is `10000`
- Values are between `0` and `10000`

These constraints tell us that even an `O(n²)` solution would technically pass, since the worst case is only about 100 million comparisons. However, the problem is designed to encourage a more efficient solution.

An important observation is that the input is guaranteed to contain exactly one repeated element, and it is guaranteed to appear exactly `n` times. This means we never need to handle invalid input or situations where no answer exists.

Some edge cases that could cause issues in naive implementations include very small arrays, repeated values appearing early in the traversal, and repeated values being spread apart throughout the array. Fortunately, the guarantees in the problem ensure that the repeated value always exists and can always be detected.

## Approaches

### Brute-Force Approach

The brute-force solution compares every element with every other element. For each index `i`, we scan all later indices `j`. If `nums[i] == nums[j]`, then we have found the repeated element and can return it immediately.

This works because the repeated number appears multiple times, while every other number appears only once. Therefore, the first duplicate we encounter must be the correct answer.

Although correct, this approach is inefficient because it performs many unnecessary comparisons. In the worst case, it checks nearly every pair of elements, resulting in quadratic time complexity.

### Optimal Approach

The better solution uses a hash set to track values we have already seen.

As we iterate through the array:

- If the current number has not been seen before, we add it to the set.
- If the current number is already in the set, we immediately return it.

The key insight is that all non-repeated elements appear exactly once, so the first value we encounter twice must be the repeated element.

A hash set allows constant average-time lookup and insertion, making the overall solution linear in time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Compare every pair of elements |
| Optimal | O(n) | O(n) | Use a hash set to detect duplicates efficiently |

## Algorithm Walkthrough

### Optimal Algorithm Using a Hash Set

1. Create an empty hash set called `seen`.

The purpose of this set is to store all numbers we have already processed. A hash set provides very fast lookup operations, typically `O(1)` on average.
2. Iterate through each number in `nums`.

We process the array from left to right, examining one value at a time.
3. For the current number, check whether it already exists in `seen`.

If it does, then this is the second time we have encountered the number. Since all other numbers appear exactly once, this duplicate must be the repeated element.
4. If the number is already in the set, return it immediately.

There is no need to continue processing because the problem guarantees exactly one repeated element.
5. Otherwise, add the current number to `seen`.

This records that we have now encountered this value.
6. Continue until the repeated element is found.

Because the repeated value appears multiple times, we are guaranteed to eventually encounter a duplicate.

### Why it works

The algorithm relies on the problem guarantee that every non-repeated number appears exactly once. Therefore, the only way we can encounter the same number twice is if it is the element repeated `n` times. The hash set efficiently tracks previously seen values, allowing immediate duplicate detection.

## Python Solution

```
from typing import List

class Solution:
    def repeatedNTimes(self, nums: List[int]) -> int:
        seen = set()

        for number in nums:
            if number in seen:
                return number

            seen.add(number)

        return -1
```

The implementation begins by creating an empty set named `seen`. This set stores all values encountered so far during the traversal.

The loop iterates through each element in `nums`. For every number, the algorithm first checks whether it already exists in the set. If it does, that means we have found the repeated element, so the function immediately returns it.

If the number is not yet in the set, it is added so future iterations can detect duplicates.

The final `return -1` statement is technically unnecessary because the problem guarantees a valid answer exists. However, including it satisfies Python's requirement that every execution path returns a value.

## Go Solution

```
func repeatedNTimes(nums []int) int {
    seen := make(map[int]bool)

    for _, number := range nums {
        if seen[number] {
            return number
        }

        seen[number] = true
    }

    return -1
}
```

The Go implementation uses a map from `int` to `bool` instead of a dedicated set type, since Go does not provide a built-in hash set.

The logic is otherwise identical to the Python solution. Each number is checked against the map. If it already exists, the repeated element has been found. Otherwise, the number is added to the map.

There are no integer overflow concerns because the values are very small. Go slices are used naturally for array traversal, and no special handling for nil or empty slices is necessary because the problem guarantees valid input.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each element is processed once |
| Space | O(n) | The hash set may store up to `n + 1` unique values |

The algorithm performs a single pass through the array. Each lookup and insertion into the hash set takes constant average time, so the total runtime grows linearly with the size of the input.

The space complexity is linear because the set may contain every unique value before the duplicate is detected.

## Edge Cases

One important edge case is the smallest possible input size:

```
nums = [0,0]
```

Here, `n = 1`, and the repeated element appears exactly once more after its first occurrence. Some implementations incorrectly assume the array contains at least three or four elements. The hash set solution handles this naturally because it immediately detects the duplicate during the second iteration.

Another important case occurs when the duplicate appears very early in the array:

```
nums = [2,1,2,5,3,2]
```

A poorly designed implementation might continue scanning even after finding the repeated value, wasting unnecessary work. The optimal solution immediately returns once a duplicate is found, making it efficient even in best-case scenarios.

A third edge case is when repeated values are spread far apart:

```
nums = [5,1,2,3,4,5]
```

Some incorrect approaches rely on duplicates being adjacent. However, the problem does not guarantee any ordering. The hash set solution works regardless of position because it tracks all previously seen values, not just neighboring elements.
