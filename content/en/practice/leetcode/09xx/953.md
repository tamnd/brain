# LeetCode 953, Verifying an Alien Dictionary

## Problem Understanding

This problem asks us to determine whether a list of words is sorted according to a custom alphabet order instead of the normal English alphabetical order.

Normally, English words are compared lexicographically using the standard alphabet:

```
a < b < c < ... < z
```

In this problem, the alien language still uses the same lowercase English letters, but the order of those letters may be different. The input string `order` defines this custom alphabet.

For example:

```
order = "hlabcdefgijkmnopqrstuvwxyz"
```

means:

```
h < l < a < b < c < ...
```

The input consists of:

- `words`, an array of strings written in the alien language
- `order`, a permutation of the 26 lowercase English letters

We must return `true` if the words are sorted lexicographically according to this alien alphabet. Otherwise, return `false`.

Lexicographical comparison works similarly to dictionary ordering in English:

- Compare characters from left to right
- The first differing character determines the ordering
- If all compared characters are equal, the shorter word comes first

For example:

```
"app" < "apple"
```

because the shorter word is considered smaller when one word is a prefix of the other.

The constraints are small:

- At most 100 words
- Each word has length at most 20

This means even relatively straightforward solutions are fast enough. However, we still want a clean and efficient implementation.

Several edge cases are important:

- One word may be a prefix of another
- Consecutive words may be identical
- The first differing character may appear late in the strings
- The custom order may completely rearrange the alphabet

A naive implementation using normal string comparison will fail because Python and Go use the standard English alphabet ordering internally.

## Approaches

### Brute Force Approach

The brute force idea is to repeatedly compare every pair of adjacent words character by character using the alien alphabet.

For each comparison:

1. Scan both words from left to right
2. Find the first position where characters differ
3. Use the alien ordering to decide which word is smaller
4. If no differing character exists, compare lengths

To support these comparisons, we can search for each character's position inside the `order` string whenever needed.

For example, to compare `'h'` and `'l'`, we could call:

```
order.index('h')
order.index('l')
```

This approach is correct because lexicographical ordering depends only on the first differing character. However, repeatedly searching the `order` string is inefficient because each lookup costs `O(26)` time.

Even though the constraints are small, this repeated scanning is unnecessary.

### Optimal Approach

The key optimization is to preprocess the alien alphabet into a hash map that stores the rank of every character.

Example:

```
{
    'h': 0,
    'l': 1,
    'a': 2,
    ...
}
```

Now each character comparison becomes `O(1)` instead of searching through the order string every time.

We then compare each adjacent pair of words:

- Find the first differing character
- Compare their ranks
- Handle the prefix case carefully

This produces an efficient and clean solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(N × M × 26) | O(1) | Repeatedly searches the order string |
| Optimal | O(N × M) | O(26) | Uses a hash map for constant time rank lookup |

Where:

- `N` = number of words
- `M` = maximum word length

## Algorithm Walkthrough

1. Create a hash map that stores the alien rank of every character.

For example:

```
rank['h'] = 0
rank['l'] = 1
rank['a'] = 2
```

This allows constant time character comparisons.
2. Iterate through all adjacent word pairs.

Since a sorted list only requires every neighboring pair to be in correct order, we compare:

```
words[0] vs words[1]
words[1] vs words[2]
...
```
3. Compare the two words character by character.

Use the minimum length of the two words because comparison stops once one word ends.
4. Find the first differing character.

If:

```
word1[j] != word2[j]
```

then their alien ranks determine ordering.
5. If the first word's character rank is greater than the second word's rank, return `False`.

This means the words are out of order.
6. If the first word's character rank is smaller, the pair is correctly ordered.

We can stop comparing this pair and move to the next pair.
7. If all compared characters are equal, handle the prefix case.

Example:

```
"apple"
"app"
```

Here, `"app"` is a prefix of `"apple"`.

Lexicographical rules require the shorter word to come first. Therefore:

```
len(word1) > len(word2)
```

means the ordering is invalid.
8. If all adjacent pairs are valid, return `True`.

### Why it works

Lexicographical ordering depends entirely on the first differing character between two strings. If all characters match up to the shorter length, then the shorter string must come first.

The algorithm checks exactly these conditions for every adjacent pair. Since a sequence is sorted if and only if every neighboring pair is sorted, the algorithm correctly determines whether the entire list is sorted.

## Python Solution

```
from typing import List

class Solution:
    def isAlienSorted(self, words: List[str], order: str) -> bool:
        # Build character rank map
        rank = {char: index for index, char in enumerate(order)}

        # Compare adjacent words
        for i in range(len(words) - 1):
            word1 = words[i]
            word2 = words[i + 1]

            found_difference = False

            # Compare characters
            for j in range(min(len(word1), len(word2))):
                if word1[j] != word2[j]:
                    if rank[word1[j]] > rank[word2[j]]:
                        return False

                    found_difference = True
                    break

            # Handle prefix case
            if not found_difference and len(word1) > len(word2):
                return False

        return True
```

The implementation begins by constructing a dictionary called `rank`. This maps every character to its position in the alien alphabet. Using a dictionary gives constant time lookups during comparisons.

Next, the algorithm iterates through each adjacent pair of words. Only neighboring pairs need to be checked because lexicographical sorting is transitive.

For each pair, the code compares characters from left to right. Once a differing character is found, their alien ranks determine whether the pair is valid.

If the first differing character in `word1` has a larger rank than the corresponding character in `word2`, the words are incorrectly ordered, so the function immediately returns `False`.

If no differing characters are found, the words may still be invalid if the first word is longer. This handles prefix cases such as:

```
"apple"
"app"
```

Finally, if all pairs pass validation, the function returns `True`.

## Go Solution

```
func isAlienSorted(words []string, order string) bool {
	// Build character rank map
	rank := make(map[byte]int)

	for i := 0; i < len(order); i++ {
		rank[order[i]] = i
	}

	// Compare adjacent words
	for i := 0; i < len(words)-1; i++ {
		word1 := words[i]
		word2 := words[i+1]

		foundDifference := false

		limit := len(word1)
		if len(word2) < limit {
			limit = len(word2)
		}

		// Compare characters
		for j := 0; j < limit; j++ {
			if word1[j] != word2[j] {
				if rank[word1[j]] > rank[word2[j]] {
					return false
				}

				foundDifference = true
				break
			}
		}

		// Handle prefix case
		if !foundDifference && len(word1) > len(word2) {
			return false
		}
	}

	return true
}
```

The Go implementation follows the same logic as the Python version. The primary difference is that Go strings are indexed as bytes, which works perfectly here because the input only contains lowercase English letters.

The rank mapping uses:

```
map[byte]int
```

instead of a Python dictionary.

Go does not provide a built in `min` function for integers, so the implementation manually computes the smaller length before comparing characters.

No special handling for integer overflow is needed because the constraints are very small.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(N × M) | Each character is compared at most once across adjacent pairs |
| Space | O(26) | The rank map stores one entry per lowercase letter |

The preprocessing step creates a hash map of size 26, which is constant space. The main work comes from comparing adjacent words character by character.

In the worst case, every comparison scans the full length of the shorter word, producing `O(N × M)` time complexity.

## Edge Cases

### Prefix relationships

One of the most important edge cases occurs when one word is a prefix of another.

Example:

```
["apple", "app"]
```

A buggy implementation may incorrectly treat these words as equal because all compared characters match. However, lexicographical ordering requires the shorter word to come first.

The implementation handles this by checking:

```
if not found_difference and len(word1) > len(word2):
```

which correctly identifies invalid prefix ordering.

### Identical words

Another edge case occurs when two consecutive words are exactly the same.

Example:

```
["same", "same"]
```

No differing characters exist, but the words are still correctly sorted. The implementation naturally handles this because the prefix condition only fails when the first word is longer.

### First differing character appears late

Sometimes words share a long common prefix before differing.

Example:

```
["abcdefx", "abcdefy"]
```

A flawed implementation might stop comparison too early or compare full strings incorrectly.

The algorithm correctly scans characters left to right until the first differing position is found. Only that position matters for lexicographical ordering.

### Completely custom alphabet order

The alien alphabet may drastically differ from English ordering.

Example:

```
order = "zyxwvutsrqponmlkjihgfedcba"
```

Normal string comparison functions would fail in this situation.

The implementation avoids this issue entirely by using the custom rank mapping instead of built in lexicographical comparison.
