## Solution

## Problem Understanding

LeetCode 968, Binary Tree Cameras, asks us to determine the minimum number of cameras required to monitor every node in a binary tree.

A camera can be installed on any node. Once installed, that camera monitors three categories of nodes:

- The node where the camera is placed
- Its parent
- Its immediate left and right children

The goal is not simply to cover most nodes, but to guarantee that every node in the tree is monitored while using as few cameras as possible.

The input is the root of a binary tree. Each node contains only the value `0`, so the actual values are irrelevant. What matters is the structure of the tree. The output is a single integer representing the smallest number of cameras needed for complete coverage.

The constraints tell us the tree contains between 1 and 1000 nodes. This is small enough that recursive depth-first traversal is practical, but large enough that exponential brute-force solutions become infeasible. Since the problem involves hierarchical parent-child relationships and optimal local decisions, tree dynamic programming and greedy DFS are natural fits.

Several edge cases are important to recognize early:

A tree with only one node requires exactly one camera, because the node must monitor itself.

A highly skewed tree, where every node only has one child, can expose flaws in greedy logic if camera placement is handled incorrectly.

Leaf nodes are especially important because they cannot monitor their parents. Their parents often become the optimal camera locations.

The root node requires special handling because it has no parent. If traversal finishes and the root remains unmonitored, we must add one final camera.

The problem guarantees the input is a valid binary tree and that at least one node exists.

## Approaches

### Brute Force Approach

The brute-force idea is to try every possible subset of nodes as camera locations and check whether all nodes become monitored.

Suppose the tree has `n` nodes. Each node has two possibilities:

- Place a camera
- Do not place a camera

That produces `2^n` possible configurations. For each configuration, we would need to simulate coverage and verify whether every node is monitored. Among all valid configurations, we would choose the minimum number of cameras.

This approach is guaranteed to find the optimal answer because it exhaustively explores every possible placement strategy.

However, it is computationally impractical. Even with only 20 nodes, there are already over one million configurations. With up to 1000 nodes, exhaustive search becomes impossible.

The key issue is that brute force ignores the structure of the tree and repeatedly recomputes overlapping coverage states.

### Optimal Approach

The optimal solution uses postorder depth-first traversal combined with greedy state management.

The key insight is that camera placement decisions should be made from the bottom upward.

If a child node is not monitored, then its parent must contain a camera. Waiting until we process the parent gives us maximum information about both children before deciding whether a camera is necessary.

Instead of explicitly storing all coverage combinations, we classify each node into one of three states:

- The node has a camera
- The node is covered but has no camera
- The node is not covered

These states are sufficient to make optimal local decisions.

The traversal proceeds in postorder because a node's state depends entirely on the states of its children.

This creates an elegant greedy strategy:

- If either child is uncovered, place a camera here
- Else if either child has a camera, this node is covered
- Else this node remains uncovered

The greedy rule works because cameras cover parents and children simultaneously, making parent placement especially efficient for leaf coverage.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n * n) | O(n) | Tries every camera placement configuration |
| Optimal | O(n) | O(h) | Postorder DFS with greedy state transitions |

Here, `n` is the number of nodes and `h` is the tree height.

## Algorithm Walkthrough

### State Definitions

We define three states for each node:

- `0` = node is not covered
- `1` = node has a camera
- `2` = node is covered but has no camera

These states allow the parent to determine what action is necessary.

### Step-by-Step Algorithm

1. Perform a postorder DFS traversal.

We process the left subtree first, then the right subtree, and finally the current node. This order is essential because the current node's decision depends entirely on the states returned by its children.
2. Treat null nodes as already covered.

A missing child does not require monitoring, so returning state `2` for null nodes simplifies the logic and avoids unnecessary camera placement.
3. Recursively obtain the states of the left and right children.

Each recursive call tells us whether the child:

- Needs coverage
- Already has a camera
- Is safely covered
4. If either child is uncovered, place a camera at the current node.

This is the most important greedy step.

If a child returns state `0`, that child has no camera and is not covered by its descendants. The only remaining node capable of covering it efficiently is the current node.

We increment the camera counter and return state `1`.
5. If either child has a camera, the current node is covered.

A child camera automatically monitors its parent, so the current node does not need its own camera.

We return state `2`.
6. If both children are covered but neither has a camera, this node becomes uncovered.

Since the children are already safe and no child camera covers the current node, we leave this node uncovered for now and allow its parent to decide whether to place a camera.

We return state `0`.
7. After DFS completes, check the root.

The root has no parent, so if it remains uncovered after traversal, we must install one final camera at the root.

### Why it works

The correctness comes from the fact that every uncovered node forces its parent to place a camera. Since cameras cover both parent and child simultaneously, placing cameras as high as possible while still covering uncovered children minimizes redundancy.

Postorder traversal guarantees that when we process a node, all information about its descendants is already finalized. Therefore, every camera placement decision is locally optimal and collectively produces a globally optimal solution.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import Optional

class Solution:
    def minCameraCover(self, root: Optional[TreeNode]) -> int:
        NOT_COVERED = 0
        HAS_CAMERA = 1
        COVERED = 2

        camera_count = 0

        def dfs(node: Optional[TreeNode]) -> int:
            nonlocal camera_count

            if not node:
                return COVERED

            left_state = dfs(node.left)
            right_state = dfs(node.right)

            if left_state == NOT_COVERED or right_state == NOT_COVERED:
                camera_count += 1
                return HAS_CAMERA

            if left_state == HAS_CAMERA or right_state == HAS_CAMERA:
                return COVERED

            return NOT_COVERED

        if dfs(root) == NOT_COVERED:
            camera_count += 1

        return camera_count
```

The implementation closely follows the state-machine logic described earlier.

We first define symbolic constants for the three possible node states. Using named constants instead of raw integers improves readability and makes the transitions easier to understand.

The recursive `dfs` function performs a postorder traversal. It processes both children before making a decision about the current node.

The base case returns `COVERED` for null nodes. This prevents leaf parents from incorrectly thinking they need to place cameras for missing children.

After collecting the child states, the function evaluates three conditions in priority order.

The first condition checks whether either child is uncovered. If so, the current node must install a camera. We increment the shared counter and return `HAS_CAMERA`.

The second condition checks whether either child already has a camera. In that case, the current node is automatically monitored, so it returns `COVERED`.

If neither condition applies, the node itself is uncovered. We return `NOT_COVERED` and allow the parent to potentially place a camera.

Finally, after DFS finishes, we verify whether the root is uncovered. Since the root has no parent, we add one final camera if necessary.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func minCameraCover(root *TreeNode) int {
	const (
		NOT_COVERED = 0
		HAS_CAMERA  = 1
		COVERED     = 2
	)

	cameraCount := 0

	var dfs func(node *TreeNode) int

	dfs = func(node *TreeNode) int {
		if node == nil {
			return COVERED
		}

		leftState := dfs(node.Left)
		rightState := dfs(node.Right)

		if leftState == NOT_COVERED || rightState == NOT_COVERED {
			cameraCount++
			return HAS_CAMERA
		}

		if leftState == HAS_CAMERA || rightState == HAS_CAMERA {
			return COVERED
		}

		return NOT_COVERED
	}

	if dfs(root) == NOT_COVERED {
		cameraCount++
	}

	return cameraCount
}
```

The Go implementation mirrors the Python logic almost exactly.

Go uses constants for the three states and a closure-based recursive DFS function. Since Go does not support nested functions with implicit variable capture like Python's `nonlocal`, the closure directly updates `cameraCount` from the outer scope.

Nil pointers naturally represent missing children, so the base case checks `node == nil`.

Integer overflow is not a concern because the maximum number of nodes is only 1000.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each node is visited exactly once |
| Space | O(h) | Recursive call stack depth equals tree height |

The DFS traversal processes every node a single time and performs only constant-time work per node, producing linear time complexity.

The auxiliary space comes entirely from recursion. In a balanced tree, the height is approximately `log n`. In the worst case, a completely skewed tree has height `n`, producing worst-case recursion depth of `O(n)`.

## Edge Cases

A single-node tree is one of the most important edge cases. Since the root has no parent, it cannot rely on another node for coverage. A buggy implementation might incorrectly leave the root uncovered after DFS completes. This solution explicitly checks the root state after traversal and adds a final camera if necessary.

Highly skewed trees can also create problems. In a chain-like structure, naive greedy placement may install too many cameras because local decisions appear optimal without considering parent coverage. The postorder traversal prevents this issue by always evaluating children first and only placing cameras when a child is genuinely uncovered.

Leaf nodes are another subtle case. Leaves themselves do not usually receive cameras in the optimal solution. Instead, their parents should contain cameras because those cameras simultaneously cover more nodes. Returning `NOT_COVERED` for leaf nodes enables this behavior naturally and avoids inefficient placements.

Null children are easy to mishandle. If null nodes were treated as uncovered, every leaf parent would incorrectly place cameras for missing children. Returning `COVERED` for null nodes ensures that only real nodes affect placement decisions.
