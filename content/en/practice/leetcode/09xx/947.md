## Solution

## Problem Understanding

The problem presents a 2D grid with `n` stones placed at integer coordinates. Each stone occupies a unique coordinate, meaning no two stones share the same position. The key operation allowed is removing a stone if it shares either the same row or the same column with **another stone that has not been removed**. The goal is to remove as many stones as possible, leaving a minimum number of stones on the board.

The input, `stones`, is an array of pairs `[x, y]` representing the coordinates of each stone. The output is an integer representing the maximum number of stones that can be removed.

Constraints inform us that `stones.length` can go up to 1000, with coordinate values as large as 10,000. This means a brute-force approach iterating over every pair repeatedly will likely be too slow. The problem guarantees no duplicate stones, which simplifies connectivity checks. Edge cases include a single stone or stones that are all isolated in terms of row and column connections.

The underlying challenge is to identify **connected groups of stones** where each stone is indirectly or directly connected via rows or columns. Each group can be reduced to **one remaining stone**, so the maximum number of removable stones is the total number of stones minus the number of connected groups.

## Approaches

### Brute Force

A brute-force solution would repeatedly scan the list of stones to remove any stone that shares a row or column with another remaining stone. Each removal requires updating the list of remaining stones and checking all other stones again. This approach is correct because it simulates the removal process directly. However, it is inefficient because for each stone removal, it may need to iterate through the remaining stones, leading to O(n²) or worse in repeated iterations. With n up to 1000, this becomes computationally expensive.

### Optimal Approach

The key insight is that stones form **connected components** in a graph where edges exist between stones in the same row or column. Using either Depth-First Search (DFS) or Union-Find, we can efficiently identify connected components. Once identified, the maximum stones removable is the total number of stones minus the number of connected components, because each component must retain at least one stone.

Union-Find (Disjoint Set Union, DSU) is particularly efficient here, as it allows fast merging of nodes (stones) that share a row or column. By mapping rows and columns to unique identifiers, we can union stones connected via rows or columns. The number of unique root parents after all unions represents the number of connected components.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n² * n) | O(n) | Repeatedly scan and remove stones; inefficient for large n |
| Optimal (Union-Find / DFS) | O(n * α(n)) | O(n) | Union-Find with path compression or DFS to count connected components |

## Algorithm Walkthrough

1. **Map rows and columns**: Treat each stone as a node and map its row and column to unique identifiers. To avoid collisions, offset column indices by a large constant (e.g., 10001).
2. **Initialize Union-Find**: Create a parent mapping for each unique row and column identifier.
3. **Union stones by row and column**: For each stone at `[x, y]`, union `x` (row) with `y + offset` (column). This connects stones sharing rows or columns.
4. **Count connected components**: After union operations, find the number of unique root parents. Each root represents a connected component.
5. **Compute removable stones**: Subtract the number of connected components from the total number of stones to get the maximum removable stones.

**Why it works:** Each connected component represents a set of stones where any stone can eventually be removed except one, because removal requires a neighboring stone in the same row or column. Union-Find ensures we efficiently capture these connections.

## Python Solution

```
from typing import List

class Solution:
    def removeStones(self, stones: List[List[int]]) -> int:
        parent = {}

        def find(x):
            if x != parent.setdefault(x, x):
                parent[x] = find(parent[x])
            return parent[x]

        def union(x, y):
            parent[find(x)] = find(y)

        offset = 10001
        for x, y in stones:
            union(x, y + offset)

        unique_roots = set(find(x) for x in parent)
        return len(stones) - len(unique_roots)
```

This implementation uses a dictionary to store the parent of each row or column. `find` performs path compression for efficiency, and `union` merges connected nodes. After all unions, counting unique roots gives the number of connected components. Subtracting from the total stones gives the maximum number removable.

## Go Solution

```
func removeStones(stones [][]int) int {
    parent := make(map[int]int)

    var find func(x int) int
    find = func(x int) int {
        if _, ok := parent[x]; !ok {
            parent[x] = x
        }
        if parent[x] != x {
            parent[x] = find(parent[x])
        }
        return parent[x]
    }

    union := func(x, y int) {
        parent[find(x)] = find(y)
    }

    offset := 10001
    for _, stone := range stones {
        x, y := stone[0], stone[1]
        union(x, y+offset)
    }

    uniqueRoots := make(map[int]struct{})
    for k := range parent {
        root := find(k)
        uniqueRoots[root] = struct{}{}
    }

    return len(stones) - len(uniqueRoots)
}
```

In Go, we use a map to track parent relationships, similar to Python. We handle initialization of missing keys explicitly and use a map of empty structs to count unique roots efficiently.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n * α(n)) | Each union/find operation is near constant with path compression |
| Space | O(n) | Stores parent mapping for rows and columns |

The `α(n)` is the inverse Ackermann function, practically constant for all reasonable n.

## Edge Cases

**Single Stone:** Only one stone exists. The algorithm correctly returns 0 because no removal is possible, and the union-find structure handles this trivially.

**Disconnected Stones:** Stones that do not share a row or column with any other stone form separate components. The algorithm counts each as a unique root, ensuring no removals beyond isolated stones.

**Large Coordinates:** Stones can have coordinates up to 10,000. By offsetting column indices in union operations, we avoid collisions between row and column indices while maintaining unique identifiers in the union-find structure.
