# LeetCode 933 - Number of Recent Calls

## Problem Understanding

This problem asks us to design a data structure that tracks recent requests over a moving time window. Every time the `ping(t)` method is called, we receive a timestamp `t` in milliseconds. We must return how many requests occurred within the inclusive range `[t - 3000, t]`.

In other words, whenever a new request arrives, we need to count all requests that happened during the last 3000 milliseconds, including the current request itself.

The key detail is that timestamps are strictly increasing. This means every new request comes later than all previous requests. That property is extremely important because it allows us to process requests efficiently in chronological order.

The input is not given as a traditional array. Instead, we are implementing a class that supports repeated method calls. The class must maintain internal state between calls to `ping`.

The constraints tell us several important things:

- There can be at most `10^4` calls, which is not extremely large, but still large enough that inefficient repeated scans could become expensive.
- Timestamps can be as large as `10^9`, so we cannot use an indexed array based on time.
- Since timestamps are strictly increasing, old requests that fall outside the valid window will never become relevant again.

Several edge cases are important to consider:

- The very first `ping` call should always return `1`.
- Multiple requests may still fall inside the same 3000 millisecond window.
- Requests exactly at `t - 3000` must be included because the range is inclusive.
- Requests smaller than `t - 3000` must be removed or ignored.
- The timestamps are guaranteed to increase strictly, so we never need to handle out-of-order insertions.

## Approaches

### Brute Force Approach

A straightforward solution is to store every timestamp in a list. Whenever `ping(t)` is called, we iterate through the entire list and count how many timestamps satisfy:

```
t - 3000 <= timestamp <= t
```

This approach is correct because it explicitly checks every request and counts the valid ones.

However, this method becomes inefficient as the number of requests grows. If there are `n` calls to `ping`, and each call scans all previous timestamps, the total complexity becomes `O(n^2)` in the worst case.

Although `10^4` operations might still pass in practice, the problem is clearly designed to encourage a more efficient sliding window solution.

### Optimal Approach

The key observation is that timestamps arrive in strictly increasing order. Because of this, once a timestamp becomes too old, it will never become valid again in future calls.

This naturally suggests using a queue.

We maintain a queue containing only timestamps inside the current valid window. For each new `ping(t)`:

1. Add `t` to the queue.
2. Remove timestamps from the front while they are smaller than `t - 3000`.
3. The remaining queue size is the answer.

This works efficiently because each timestamp is added once and removed once.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^2) | O(n) | Scan all previous timestamps for every ping |
| Optimal | O(n) total, O(1) amortized per call | O(n) | Queue maintains only valid timestamps |

## Algorithm Walkthrough

1. Initialize an empty queue to store timestamps.
2. When `ping(t)` is called, append `t` to the queue. The current request must always be included in the valid range.
3. Compute the smallest valid timestamp, which is `t - 3000`.
4. While the front of the queue contains a timestamp smaller than `t - 3000`, remove it. These requests are too old and can never become valid again because future timestamps will only increase.
5. After removing all expired timestamps, every remaining timestamp lies within the inclusive range `[t - 3000, t]`.
6. Return the size of the queue because it now represents exactly the valid recent requests.

### Why it works

The queue always maintains an important invariant: every timestamp inside it belongs to the current valid window.

Since timestamps are strictly increasing, expired timestamps always appear at the front of the queue. Removing them preserves the invariant efficiently. Because each timestamp enters and leaves the queue at most once, the algorithm achieves amortized constant time per operation.

## Python Solution

```
from collections import deque
from typing import Deque

class RecentCounter:

    def __init__(self):
        self.requests: Deque[int] = deque()

    def ping(self, t: int) -> int:
        self.requests.append(t)

        minimum_valid_time = t - 3000

        while self.requests and self.requests[0] < minimum_valid_time:
            self.requests.popleft()

        return len(self.requests)

# Your RecentCounter object will be instantiated and called as such:
# obj = RecentCounter()
# param_1 = obj.ping(t)
```

The implementation uses Python's `deque` from the `collections` module because it supports efficient insertion at the back and removal from the front in `O(1)` time.

The constructor initializes an empty queue named `requests`.

Inside `ping`, the current timestamp is appended immediately because the current request always belongs to the valid range.

Next, we compute `minimum_valid_time`, which represents the smallest timestamp still allowed in the window.

The `while` loop removes outdated timestamps from the front of the queue. Since timestamps are stored in increasing order, once we encounter a valid timestamp, all later timestamps are also valid.

Finally, the queue length is returned because the queue contains exactly the recent requests.

## Go Solution

```
type RecentCounter struct {
    requests []int
}

func Constructor() RecentCounter {
    return RecentCounter{
        requests: []int{},
    }
}

func (this *RecentCounter) Ping(t int) int {
    this.requests = append(this.requests, t)

    minimumValidTime := t - 3000

    for len(this.requests) > 0 && this.requests[0] < minimumValidTime {
        this.requests = this.requests[1:]
    }

    return len(this.requests)
}

/**
 * Your RecentCounter object will be instantiated and called as such:
 * obj := Constructor();
 * param_1 := obj.Ping(t);
 */
```

The Go implementation uses a slice as a queue. New timestamps are appended to the end using `append`.

Expired timestamps are removed by re-slicing:

```
this.requests = this.requests[1:]
```

Go slices do not automatically free underlying memory immediately, but with the problem constraints this is completely acceptable.

Unlike Python, Go does not have a built-in deque in the standard library, so using slices is the most common solution.

### Detailed explanation for `ping(3002)`

The valid range is:

```
[3002 - 3000, 3002]
[2, 3002]
```

Timestamp `1` is outside this range, so it is removed.

The remaining timestamps are:

```
[100, 3001, 3002]
```

The count is `3`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(1) amortized per call, O(n) total | Each timestamp is inserted and removed at most once |
| Space | O(n) | In the worst case, all timestamps may remain inside the window |

The important observation is that although the `while` loop may remove multiple elements during a single call, each timestamp can only be removed once across the entire execution. This gives amortized constant time complexity.

The space complexity is linear because the queue may temporarily store many recent timestamps.

## Edge Cases

One important edge case occurs when the very first request arrives. Since there are no previous requests, the queue starts empty. A buggy implementation might accidentally return `0` if it removes elements before inserting the current timestamp. Our implementation appends the new timestamp first, guaranteeing the correct result of `1`.

Another critical edge case involves timestamps exactly equal to `t - 3000`. The problem explicitly states that the valid range is inclusive. This means timestamps equal to the lower boundary must remain in the queue. Our implementation removes timestamps only when they are strictly smaller than `t - 3000`, which correctly preserves boundary values.

A third important case happens when many old requests expire at once. For example, after many tightly packed requests, a large jump in time may invalidate nearly the entire queue. Some implementations remove only one outdated request instead of all outdated requests. Our `while` loop continues removing timestamps until the queue contains only valid entries.

A final edge case involves very large timestamp values near `10^9`. Since we only perform subtraction and comparison operations, and both Python integers and Go integers comfortably support these values, the implementation handles them safely without overflow concerns.
