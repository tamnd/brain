## Solution

## Problem Understanding

The problem asks us to perform a vertical traversal of a binary tree. Every node in the tree is assigned a coordinate `(row, col)` based on its position relative to the root.

The root starts at position `(0, 0)`.

For any node located at `(row, col)`:

- Its left child is located at `(row + 1, col - 1)`
- Its right child is located at `(row + 1, col + 1)`

The goal is to group nodes by their column index, from the smallest column to the largest column. Inside each column, nodes must appear from top to bottom, meaning smaller row values come first.

There is one additional rule that makes the problem more subtle. If two nodes share the exact same position, meaning they have the same row and the same column, then they must be ordered by their node values in ascending order.

The input is the root node of a binary tree. The output is a list of lists, where each inner list contains the node values belonging to one vertical column.

The constraints tell us that the tree contains at most 1000 nodes. This is small enough that an `O(n log n)` sorting-based solution is completely acceptable. However, we still need to carefully handle ordering rules because traversal order alone is not enough.

Several edge cases are important:

- Multiple nodes may share the same `(row, col)` position.
- The tree may be highly skewed, creating many negative or positive columns.
- A simple BFS or DFS traversal without sorting can produce incorrect ordering.
- The tree contains at least one node, so the root is never `None`.

## Approaches

### Brute Force Approach

A brute force strategy would be to explicitly compute every node's `(row, col)` coordinates, store them, and then repeatedly scan the collection to build columns one by one.

For example, we could:

1. Traverse the tree and collect all nodes with coordinates.
2. Find the minimum and maximum column values.
3. For each column:

- Scan every node again
- Select nodes belonging to that column
- Sort them by row and value
- Append them to the result

This approach works because it directly follows the problem definition. However, it is inefficient because each column reconstruction repeatedly scans all nodes.

If there are `n` nodes and many distinct columns, the repeated scans can approach `O(n^2)` behavior.

### Optimal Approach

The key insight is that we only need one traversal to collect all positional information.

During DFS traversal, we can record each node as:

```
(column, row, value)
```

Once all nodes are collected, we sort the entire list using:

1. Column ascending
2. Row ascending
3. Value ascending

This single sorting step automatically enforces every ordering rule required by the problem.

After sorting, we simply group consecutive entries that share the same column.

This works efficiently because sorting gives us the exact traversal order the problem demands.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeatedly scans all nodes for each column |
| Optimal | O(n log n) | O(n) | Single traversal plus global sorting |

## Algorithm Walkthrough

1. Create an empty list called `nodes`.

This list will store every node together with its position information. Each entry will contain:

```
(column, row, value)
```
2. Perform a DFS traversal starting from the root at `(0, 0)`.

DFS is convenient because we naturally propagate row and column values through recursive calls.
3. For each visited node:

- Append `(col, row, node.val)` to `nodes`
- Recurse to the left child using:

```
row + 1, col - 1
```
- Recurse to the right child using:

```
row + 1, col + 1
```
4. After traversal completes, sort `nodes`.

The sorting order is critical:

- First by column
- Then by row
- Then by value

Since Python and Go tuple-like sorting are lexicographical, this ordering happens naturally.
5. Iterate through the sorted list and group values by column.

Maintain:

- `current_column`
- `current_values`

Whenever the column changes:

- Push the previous group into the answer
- Start a new group
6. Return the grouped result.

### Why it works

Every node is assigned the exact coordinates defined in the problem statement. Sorting by `(column, row, value)` guarantees:

- Columns appear from left to right
- Nodes inside a column appear from top to bottom
- Nodes sharing the same position are ordered by value

Since every node appears exactly once in the sorted order, the grouped result is the correct vertical traversal.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import List, Optional

class Solution:
    def verticalTraversal(self, root: Optional[TreeNode]) -> List[List[int]]:
        nodes = []

        def dfs(node: Optional[TreeNode], row: int, col: int) -> None:
            if not node:
                return

            nodes.append((col, row, node.val))

            dfs(node.left, row + 1, col - 1)
            dfs(node.right, row + 1, col + 1)

        dfs(root, 0, 0)

        nodes.sort()

        result = []
        current_column = None
        current_values = []

        for col, row, value in nodes:
            if col != current_column:
                if current_values:
                    result.append(current_values)

                current_column = col
                current_values = []

            current_values.append(value)

        if current_values:
            result.append(current_values)

        return result
```

The implementation closely follows the algorithm walkthrough.

The `dfs` function traverses the tree while maintaining row and column coordinates. Each node contributes a tuple `(col, row, value)` to the `nodes` list.

After traversal, `nodes.sort()` performs lexicographical sorting automatically. This is important because Python sorts tuples element by element. That means:

```
(col, row, value)
```

naturally enforces the exact ordering rules required by the problem.

The final loop groups adjacent entries with the same column into separate result lists.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */

import "sort"

func verticalTraversal(root *TreeNode) [][]int {
	type NodeInfo struct {
		col   int
		row   int
		value int
	}

	nodes := []NodeInfo{}

	var dfs func(node *TreeNode, row int, col int)

	dfs = func(node *TreeNode, row int, col int) {
		if node == nil {
			return
		}

		nodes = append(nodes, NodeInfo{
			col:   col,
			row:   row,
			value: node.Val,
		})

		dfs(node.Left, row+1, col-1)
		dfs(node.Right, row+1, col+1)
	}

	dfs(root, 0, 0)

	sort.Slice(nodes, func(i, j int) bool {
		if nodes[i].col != nodes[j].col {
			return nodes[i].col < nodes[j].col
		}

		if nodes[i].row != nodes[j].row {
			return nodes[i].row < nodes[j].row
		}

		return nodes[i].value < nodes[j].value
	})

	result := [][]int{}

	currentCol := nodes[0].col
	currentValues := []int{}

	for _, node := range nodes {
		if node.col != currentCol {
			result = append(result, currentValues)

			currentCol = node.col
			currentValues = []int{}
		}

		currentValues = append(currentValues, node.value)
	}

	result = append(result, currentValues)

	return result
}
```

The Go solution follows the same logic as the Python version, but there are a few implementation differences.

Go does not provide built in tuple sorting, so we define a custom `NodeInfo` struct and implement sorting manually using `sort.Slice`.

The DFS function is implemented as a recursive closure.

Unlike Python lists, Go slices require explicit appending and initialization. Also, Go uses `nil` checks instead of Python's truthiness checks.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | DFS traversal is O(n), sorting dominates |
| Space | O(n) | Storage for node tuples and recursion stack |

The DFS traversal visits each node exactly once, which costs `O(n)` time.

The expensive operation is sorting the collected node list. Since there are `n` tuples, sorting costs `O(n log n)`.

The space complexity is `O(n)` because we store all node coordinates and values in memory. The recursion stack can also reach `O(n)` in a skewed tree.

## Edge Cases

### Multiple Nodes Sharing the Same Position

The most important edge case occurs when two nodes occupy the same `(row, col)` coordinates. A naive DFS or BFS implementation may accidentally preserve traversal order instead of sorting by value.

For example:

```
5 and 6 both at (2,0)
```

The problem explicitly requires smaller values first. The implementation handles this correctly by sorting tuples using:

```
(column, row, value)
```

This guarantees deterministic ordering.

### Highly Skewed Trees

A tree may contain only left children or only right children. This creates many distinct column indices.

For example:

```
1
/
2
/
3
```

produces columns:

```
0, -1, -2
```

The implementation handles this naturally because column indices are stored as integers and sorting automatically orders them from smallest to largest.

### Duplicate Node Values

The tree may contain repeated values.

For example:

```
    1
   / \
  1   1
```

A buggy solution might incorrectly rely on unique values during grouping or sorting.

This implementation never assumes uniqueness. Every node is represented independently using its coordinates and value, so duplicate values are processed correctly.
