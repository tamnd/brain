## Solution

## Problem Understanding

This problem asks us to determine whether a special person called the "town judge" exists in a group of `n` people.

Each person is labeled from `1` to `n`, and the input array `trust` describes directed trust relationships. A pair `[a, b]` means person `a` trusts person `b`.

The town judge must satisfy two strict conditions:

1. The judge trusts nobody.
2. Every other person trusts the judge.

Additionally, there must be exactly one such person.

The task is to return the label of the judge if that person exists. Otherwise, we return `-1`.

The trust relationships form a directed graph. Each person is a node, and each trust relationship is a directed edge from `a` to `b`.

The important observation is that the judge has:

- Incoming trust count = `n - 1`
- Outgoing trust count = `0`

This immediately suggests tracking how many people trust each person and how many people each person trusts.

The constraints are relatively small:

- `n <= 1000`
- `trust.length <= 10^4`

Even an `O(n^2)` solution would technically pass, but we can do much better with a linear solution.

Several edge cases are important:

- When `n = 1`, the single person is automatically the judge if no trust relationships exist.
- Cyclic trust relationships invalidate the judge property.
- Multiple people trusted by many others do not necessarily qualify if they themselves trust someone.
- Some people may not appear in the `trust` array at all.

Understanding these edge cases helps avoid subtle mistakes.

## Approaches

### Brute Force Approach

A straightforward approach is to check every person individually and verify whether they satisfy the judge conditions.

For each candidate person:

1. Check whether they trust nobody.
2. Check whether every other person trusts them.

To do this efficiently, we can scan the entire `trust` array for every candidate.

If we test all `n` people and each test scans all trust relationships, the total complexity becomes `O(n * trust.length)`.

This works because we explicitly verify the exact judge definition for every person. However, it performs unnecessary repeated scans.

### Optimal Approach

The key insight is that the judge can be identified using trust counts.

For every trust relationship `[a, b]`:

- Person `a` loses one point because they trust someone.
- Person `b` gains one point because someone trusts them.

If we maintain a score array:

- Outgoing trust contributes `-1`
- Incoming trust contributes `+1`

Then the judge must end up with score:

```
(n - 1)
```

Why?

- Everyone else trusts the judge, contributing `n - 1`
- The judge trusts nobody, so no subtraction occurs

Any non-judge person either:

- Trusts someone, lowering their score
- Or is not trusted by everyone

This allows us to solve the problem in a single pass through the trust relationships.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × m) | O(1) | Repeatedly scans all trust relationships for every candidate |
| Optimal | O(n + m) | O(n) | Uses trust score counting to identify the judge efficiently |

Here, `m = trust.length`.

## Algorithm Walkthrough

1. Create an integer array called `trust_score` of size `n + 1`.

We use `n + 1` because people are labeled from `1` to `n`, so index `0` remains unused.
2. Traverse every trust relationship `[a, b]`.

Person `a` trusts someone, so decrement their score:

```
trust_score[a] -= 1
```

Person `b` is trusted by someone, so increment their score:

```
trust_score[b] += 1
```
3. After processing all relationships, scan through people `1` to `n`.

The judge must have score exactly:

```
n - 1
```

This means:

- Trusted by everybody else
- Trusts nobody
4. If such a person exists, return their label.
5. If no such person exists, return `-1`.

### Why it works

Each trust relationship contributes one outgoing edge and one incoming edge.

The judge has:

- `n - 1` incoming edges
- `0` outgoing edges

So their final score becomes:

```
(n - 1) - 0 = n - 1
```

Any other person either trusts someone or is not trusted by everyone, preventing them from reaching this score.

Therefore, a person with score `n - 1` is uniquely the judge.

## Python Solution

```
from typing import List

class Solution:
    def findJudge(self, n: int, trust: List[List[int]]) -> int:
        trust_score = [0] * (n + 1)

        for person_a, person_b in trust:
            trust_score[person_a] -= 1
            trust_score[person_b] += 1

        for person in range(1, n + 1):
            if trust_score[person] == n - 1:
                return person

        return -1
```

The implementation closely follows the algorithm.

First, we create the `trust_score` array. Each index represents one person in the town.

While iterating through the `trust` array:

- The trusting person loses one point.
- The trusted person gains one point.

This compact scoring mechanism simultaneously tracks outgoing and incoming trust relationships.

Finally, we scan all people and return the person whose score equals `n - 1`.

If nobody satisfies that condition, the function returns `-1`.

## Go Solution

```
func findJudge(n int, trust [][]int) int {
    trustScore := make([]int, n+1)

    for _, relation := range trust {
        personA := relation[0]
        personB := relation[1]

        trustScore[personA]--
        trustScore[personB]++
    }

    for person := 1; person <= n; person++ {
        if trustScore[person] == n-1 {
            return person
        }
    }

    return -1
}
```

The Go implementation mirrors the Python logic very closely.

Slices in Go are initialized using `make`, and indexing remains straightforward because labels start from `1`.

No special handling for integer overflow is needed because the maximum score is at most `999`, which easily fits within Go's `int` type.

An empty `trust` slice is naturally handled correctly. For example, when `n == 1`, the lone person has score `0`, which equals `n - 1`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n + m) | One pass over trust relationships and one pass over people |
| Space | O(n) | Stores one score per person |

Here, `m` represents the number of trust relationships.

The algorithm is linear because each trust pair is processed exactly once, and the final scan over all people is also linear.

The additional memory comes entirely from the score array.

## Edge Cases

One important edge case occurs when `n = 1`.

If there is only one person and no trust relationships, that person automatically satisfies the judge conditions. They trust nobody, and there are no other people who fail to trust them. The implementation handles this naturally because the person's score remains `0`, and `n - 1` is also `0`.

Another tricky case occurs when a candidate is trusted by everyone but also trusts someone else.

For example:

```
trust = [[1,3],[2,3],[3,1]]
```

At first glance, person `3` appears to be the judge because multiple people trust them. However, they also trust person `1`, violating the judge definition. The score system correctly subtracts one point for outgoing trust, preventing person `3` from reaching score `n - 1`.

A third important case occurs when there are no trust relationships at all.

For example:

```
n = 4
trust = []
```

Nobody trusts anybody, so no person can satisfy the requirement of being trusted by all others. Every score remains `0`, but the required judge score is `3`. The algorithm correctly returns `-1`.

Another subtle case involves disconnected trust relationships.

For example:

```
trust = [[1,2],[3,2]]
```

Person `2` is trusted by two people, but if `n = 4`, they still are not trusted by everyone. The algorithm correctly checks for exactly `n - 1` incoming trust score, preventing false positives.
