# LeetCode 988 - Smallest String Starting From Leaf

## Problem Understanding

This problem gives us the root of a binary tree where every node stores an integer value between `0` and `25`. Each number maps directly to a lowercase English character:

- `0 -> 'a'`
- `1 -> 'b'`
- `2 -> 'c'`
- ...
- `25 -> 'z'`

The goal is to construct strings that begin at a **leaf node** and end at the **root**, then return the **lexicographically smallest** among all possible leaf-to-root strings.

The direction of traversal is extremely important. Although we naturally traverse a binary tree from root to leaf, the string must be built in the opposite direction, from leaf back to root.

For example, suppose a path in the tree is:

```
root -> b -> a -> d (leaf)
```

The resulting string is not `"bad"` because that represents root to leaf order. Instead, we reverse the direction and form:

```
"dab"
```

We must compare all such leaf-to-root strings and return the lexicographically smallest one.

Lexicographical comparison behaves like dictionary ordering. Characters earlier in the alphabet are considered smaller. If two strings share a prefix, the shorter one is smaller.

For example:

```
"ab" < "aba"
"abc" < "abd"
```

The input guarantees:

- At least one node exists.
- At most `8500` nodes exist.
- Node values are always valid lowercase letter indices.

The tree size tells us that a brute-force solution may still work, but we should avoid unnecessarily rebuilding strings repeatedly if possible. Since `8500` nodes is moderate, a depth-first traversal is very suitable.

Several edge cases are important:

A tree containing only a single node should return the corresponding one-character string.

Multiple leaves may generate strings with long shared prefixes, meaning lexicographical comparison must be performed correctly.

The lexicographically smallest answer may come from a deeper subtree, not necessarily the shortest path or leftmost traversal.

Because the string direction is leaf-to-root, a naive implementation that compares root-to-leaf strings will produce incorrect results.

## Approaches

### Brute Force Approach

The brute-force idea is straightforward. We perform a depth-first search and collect **every possible leaf-to-root string**.

For every root-to-leaf path:

1. Build the path characters.
2. Reverse the path when reaching a leaf.
3. Store the resulting string.
4. After traversing the entire tree, sort or compare all collected strings to find the smallest.

This approach is correct because it explicitly enumerates every candidate string. Since every leaf-to-root path is examined, the smallest valid answer cannot be missed.

However, it is inefficient because it stores all strings in memory. If the tree contains many leaves and long paths, repeated string creation and storage increases memory usage unnecessarily.

### Optimal Approach

The key insight is that we do not need to store every candidate string.

Instead, while performing DFS, we maintain the current root-to-node path. Whenever we reach a leaf:

1. Convert the path into a leaf-to-root string.
2. Compare it immediately with the best answer found so far.
3. Update the best answer if the new string is smaller.

Backtracking is important here. Since DFS explores one branch at a time, we can reuse the same path array:

- Add the current character when moving downward.
- Remove it when returning upward.

This avoids repeatedly allocating large path structures.

Because the answer only depends on the smallest string encountered, we only keep one best candidate in memory.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × h) | O(n × h) | Stores every leaf-to-root string |
| Optimal DFS + Backtracking | O(n × h) | O(h) | Reuses one path and stores only best answer |

Here:

- `n` = number of nodes
- `h` = tree height

In the worst case, every leaf requires building a string of length `h`.

## Algorithm Walkthrough

### Optimal DFS + Backtracking Algorithm

1. Create a variable to store the smallest string found so far.

Initially, no answer exists, so we can use `None` (or an empty marker).
2. Create a DFS function.

This function traverses the tree recursively while maintaining the current path from root to current node.
3. Convert node values into characters.

Since values range from `0` to `25`, convert using:

```
chr(node.val + ord('a'))
```

This transforms numbers into lowercase letters.
4. Add the current character to the path.

Since we are traversing from root downward, the path naturally becomes a root-to-current sequence.
5. Check whether the current node is a leaf.

A node is a leaf if both children are `None`.

When a leaf is reached:

- Reverse the path to obtain leaf-to-root order.
- Join characters into a string.
- Compare against the best answer.
- Replace the answer if the new string is lexicographically smaller.
6. Continue DFS into left and right children.

This guarantees every possible leaf path is explored.
7. Backtrack after processing children.

Remove the current character from the path.

This step is critical because the same list is reused for sibling branches.
8. Return the best answer after DFS completes.

### Why it works

The algorithm works because DFS guarantees that every root-to-leaf path is visited exactly once. At each leaf, we construct the exact leaf-to-root string required by the problem and compare it against the current best answer. Since lexicographical comparison in Python and Go follows dictionary ordering, repeatedly keeping the smallest candidate ensures the final answer is globally optimal. Backtracking guarantees the path always represents the current recursion branch and never mixes nodes from unrelated subtrees.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import Optional

class Solution:
    def smallestFromLeaf(self, root: Optional[TreeNode]) -> str:
        smallest = None
        path = []

        def dfs(node: Optional[TreeNode]) -> None:
            nonlocal smallest

            if not node:
                return

            path.append(chr(node.val + ord('a')))

            # Leaf node
            if not node.left and not node.right:
                current = ''.join(reversed(path))

                if smallest is None or current < smallest:
                    smallest = current

            dfs(node.left)
            dfs(node.right)

            # Backtrack
            path.pop()

        dfs(root)

        return smallest
```

The implementation follows the algorithm exactly.

We maintain two shared variables:

- `smallest`, which stores the best leaf-to-root string found so far.
- `path`, which stores characters along the current DFS branch.

Inside `dfs`, we first add the current node's character to the path. This represents descending deeper into the tree.

When we encounter a leaf node, we reverse the path because the required direction is leaf-to-root. We join the characters into a string and compare it with the current best answer.

After recursively exploring both children, we remove the current character from the path. This backtracking step restores the path state before returning to the parent.

Without backtracking, sibling subtrees would accidentally inherit characters from previously explored paths.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */

func smallestFromLeaf(root *TreeNode) string {
	var smallest string
	hasAnswer := false
	path := []byte{}

	var dfs func(node *TreeNode)

	dfs = func(node *TreeNode) {
		if node == nil {
			return
		}

		path = append(path, byte('a'+node.Val))

		// Leaf node
		if node.Left == nil && node.Right == nil {
			current := make([]byte, len(path))

			for i := 0; i < len(path); i++ {
				current[i] = path[len(path)-1-i]
			}

			currentStr := string(current)

			if !hasAnswer || currentStr < smallest {
				smallest = currentStr
				hasAnswer = true
			}
		}

		dfs(node.Left)
		dfs(node.Right)

		// Backtrack
		path = path[:len(path)-1]
	}

	dfs(root)

	return smallest
}
```

The Go implementation mirrors the Python logic closely.

Instead of Python lists, Go uses a `[]byte` slice to represent characters efficiently.

Go does not have `reversed(path)` or direct string comparison with `None`, so we:

- Manually reverse the path into a temporary byte slice.
- Use a boolean `hasAnswer` to track whether a valid answer has been initialized.

Backtracking is performed using slice truncation:

```
path = path[:len(path)-1]
```

This efficiently removes the most recent character.

Integer overflow is not a concern because node values remain between `0` and `25`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n × h) | Every node is visited once, each leaf may construct a string of length `h` |
| Space | O(h) | DFS recursion stack and path storage |

The DFS traversal itself touches every node exactly once, giving an `O(n)` base cost. However, every leaf requires creating a reversed string whose length may be up to the tree height `h`. In the worst case, many leaves exist, producing `O(n × h)` time.

The auxiliary space comes from recursion depth and the path array. Since both grow proportionally to the tree height, the extra memory usage is `O(h)`.

## Edge Cases

### Single Node Tree

A tree may contain only one node. Since the root is also a leaf, the answer should simply be the character represented by that node.

This can easily cause bugs if a solution assumes at least one child exists before generating a candidate string. Our implementation correctly detects the root as a leaf and immediately constructs the answer.

### Lexicographically Smaller but Deeper Path

A shorter path is not always smaller. Likewise, the left subtree is not always better.

For example:

```
    z
   / \
  b   a
```

The strings become:

```
"bz"
"az"
```

Even though the left subtree is visited first, `"az"` is smaller. Because we compare every candidate string globally, the implementation always chooses the true minimum.

### Shared Prefixes

Two candidate strings may share long prefixes:

```
"abcc"
"abc"
```

Naive comparisons sometimes fail if length is ignored. Lexicographical ordering states that when one string is a prefix of another, the shorter string is smaller.

Python and Go string comparison handle this naturally, so the implementation produces the correct answer automatically.

### Highly Skewed Tree

A tree may resemble a linked list with height close to `8500`.

This stresses recursion depth and path maintenance. The algorithm still works correctly because the path always contains only the current branch, and backtracking removes nodes cleanly after returning from recursion.
