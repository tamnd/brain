## Solution

## Problem Understanding

The problem asks us to multiply two matrices, `mat1` and `mat2`, and return the resulting matrix. The matrices are described as sparse matrices, which means that most of their elements are zero.

If `mat1` has dimensions `m x k` and `mat2` has dimensions `k x n`, then the resulting matrix will have dimensions `m x n`. Matrix multiplication works by taking the dot product between rows of `mat1` and columns of `mat2`.

For every position `(i, j)` in the result matrix:

$$result[i][j] = \sum_{t=0}^{k-1} mat1[i][t] \times mat2[t][j]$$

The important detail is that the matrices are sparse. In a normal dense matrix multiplication algorithm, we still iterate through every element even when many values are zero. Since multiplying by zero contributes nothing to the final answer, sparse matrices allow us to skip unnecessary operations.

The constraints are relatively small, with dimensions up to `100 x 100`, so even a cubic solution can technically pass. However, the problem specifically tests whether we recognize and exploit sparsity to avoid wasted computation.

There are several important edge cases to keep in mind:

- Entire rows or columns may contain only zeros.
- One or both matrices may contain only zeros.
- Negative numbers are allowed, so accumulation logic must correctly handle subtraction.
- Matrices may not be square, so the implementation must carefully use dimensions instead of assuming equal sizes.
- Sparse optimization should not accidentally skip valid non-zero contributions.

## Approaches

### Brute Force Approach

The straightforward solution follows the standard matrix multiplication algorithm directly.

For every cell `(i, j)` in the result matrix, we iterate through all possible intermediate indices `t` and compute:

$$result[i][j] += mat1[i][t] \times mat2[t][j]$$

This guarantees correctness because it exactly follows the mathematical definition of matrix multiplication.

However, this approach performs many unnecessary operations when matrices are sparse. Even if `mat1[i][t]` is zero, the algorithm still multiplies it against every corresponding value in `mat2[t][j]`. Since zero contributes nothing to the sum, these computations are wasted.

The brute-force approach runs in cubic time:

$$O(m \times k \times n)$$

This is acceptable for small inputs, but inefficient for sparse matrices.

### Optimal Sparse Matrix Approach

The key observation is that multiplication involving zero never changes the result.

Instead of iterating through every possible element, we only process non-zero values.

Suppose `mat1[i][t]` is non-zero. Only then does it make sense to look at row `t` in `mat2`. Furthermore, if `mat2[t][j]` is also non-zero, then the product contributes to the answer.

This allows us to skip most operations when the matrices are sparse.

A common optimization is:

- Iterate through every non-zero element in `mat1`
- For each such element, iterate only through relevant values in `mat2`
- Accumulate contributions directly into the result matrix

This dramatically reduces work when many entries are zero.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m × k × n) | O(m × n) | Computes every possible multiplication, including many useless zero multiplications |
| Optimal | O(nonZero(mat1) × n) worst-case O(m × k × n) | O(m × n) | Skips zero entries and performs only meaningful multiplications |

## Algorithm Walkthrough

### Optimal Sparse Matrix Multiplication

1. Initialize the result matrix with dimensions `m x n`, filled with zeros.

We need a matrix to accumulate all multiplication results. Since every cell begins at zero, we create a zero-initialized matrix.
2. Iterate through every row `i` in `mat1`.

Each row contributes to the corresponding row in the result matrix.
3. Iterate through every column `t` in `mat1`.

The value `mat1[i][t]` determines whether row `t` in `mat2` contributes anything to the result.
4. Skip processing if `mat1[i][t] == 0`.

This is the core sparse optimization. If the value is zero, then:

$$0 \times mat2[t][j] = 0$$

for every `j`, so no contribution exists.
5. If `mat1[i][t]` is non-zero, iterate through every column `j` in `mat2`.

Now we compute contributions to the result matrix.
6. Skip processing if `mat2[t][j] == 0`.

Again, multiplying by zero contributes nothing.
7. Update the result matrix.

Add:

$$mat1[i][t] \times mat2[t][j]$$

into:

$$result[i][j]$$
8. Continue until all rows and columns are processed.
9. Return the completed result matrix.

### Why it works

Matrix multiplication is defined as the sum of products between matching row and column elements. The optimized algorithm still computes every valid non-zero contribution exactly once. The only difference is that it skips products involving zero, which mathematically contribute nothing to the final answer. Therefore, the optimized algorithm produces exactly the same result as standard matrix multiplication.

## Python Solution

```
from typing import List

class Solution:
    def multiply(self, mat1: List[List[int]], mat2: List[List[int]]) -> List[List[int]]:
        m = len(mat1)
        k = len(mat1[0])
        n = len(mat2[0])

        result = [[0] * n for _ in range(m)]

        for i in range(m):
            for t in range(k):
                if mat1[i][t] == 0:
                    continue

                for j in range(n):
                    if mat2[t][j] == 0:
                        continue

                    result[i][j] += mat1[i][t] * mat2[t][j]

        return result
```

The implementation begins by determining the dimensions of the matrices. `m` is the number of rows in `mat1`, `k` is the shared inner dimension, and `n` is the number of columns in `mat2`.

The result matrix is initialized with zeros because matrix multiplication accumulates sums into each cell.

The outer loop iterates through rows of `mat1`. The middle loop iterates through the shared dimension `k`. At this point, we check whether `mat1[i][t]` is zero. If it is, we skip all computations involving that value because it cannot contribute anything meaningful.

If the value is non-zero, we iterate through all columns of `mat2`. We also skip entries where `mat2[t][j]` is zero for the same reason.

Whenever both values are non-zero, we multiply them and add the product into the corresponding result cell.

This implementation preserves the exact mathematical behavior of matrix multiplication while avoiding unnecessary work.

## Go Solution

```
func multiply(mat1 [][]int, mat2 [][]int) [][]int {
    m := len(mat1)
    k := len(mat1[0])
    n := len(mat2[0])

    result := make([][]int, m)

    for i := 0; i < m; i++ {
        result[i] = make([]int, n)
    }

    for i := 0; i < m; i++ {
        for t := 0; t < k; t++ {
            if mat1[i][t] == 0 {
                continue
            }

            for j := 0; j < n; j++ {
                if mat2[t][j] == 0 {
                    continue
                }

                result[i][j] += mat1[i][t] * mat2[t][j]
            }
        }
    }

    return result
}
```

The Go implementation closely mirrors the Python version. The primary difference is explicit slice allocation.

The result matrix is created using nested `make` calls. Since Go slices are not automatically initialized as nested arrays, each row must be allocated individually.

Go integers are sufficient for this problem because the constraints guarantee values remain well within 32-bit integer range.

Unlike Python, Go does not support list comprehensions, so matrix initialization requires explicit loops.

## Edge Cases

One important edge case occurs when an entire row in `mat1` contains only zeros. A naive implementation might still iterate through every column in `mat2`, performing unnecessary multiplications. In the optimized solution, the condition:

```
if mat1[i][t] == 0:
    continue
```

immediately skips all work associated with that value. As a result, rows with only zeros produce zero rows in the result matrix efficiently.

Another critical edge case is when both matrices contain only zeros. Sparse matrices often exhibit this pattern. Some implementations accidentally assume at least one non-zero contribution exists and fail to initialize the result matrix properly. This implementation initializes every result cell to zero at the start, so even if all multiplications are skipped, the correct all-zero matrix is returned.

Negative values are another common source of bugs. Since matrix multiplication accumulates sums, implementations must correctly add both positive and negative contributions. For example:

```
(-1 × 3) + (2 × 4) = 5
```

The algorithm handles this naturally because it performs standard integer multiplication and accumulation without making assumptions about sign.

A final edge case involves rectangular matrices. Many incorrect implementations implicitly assume square matrices and accidentally use the wrong dimensions in loops. This solution explicitly computes `m`, `k`, and `n`, and uses each dimension correctly in the corresponding loop, ensuring compatibility with all valid matrix shapes.
