## Solution

## Problem Understanding

The problem gives us a string `s` consisting only of lowercase English letters. Our task is to remove duplicate characters so that every character appears exactly once in the final result. However, there is an additional and very important constraint: among all valid results, we must return the lexicographically smallest one.

Lexicographical order is dictionary order. For example, `"abc"` is smaller than `"acb"` because the first differing character is `'b'` versus `'c'`, and `'b'` comes first alphabetically.

The challenge is not simply removing duplicates. If we only kept the first occurrence of every character, we would not always obtain the smallest possible string. Likewise, keeping the last occurrence also does not always work. The order of characters matters because the resulting string must still preserve the relative ordering imposed by the original string.

For example:

- Input: `"bcabc"`
- Possible unique-letter results:

- `"bca"`
- `"cab"`
- `"abc"`
- The smallest lexicographically is `"abc"`.

The constraints tell us that:

- `1 <= s.length <= 10^4`
- Only lowercase English letters are used

A length of `10^4` is large enough that exponential or factorial approaches are not feasible. We need an algorithm close to linear time.

Several edge cases are important:

- Strings where all characters are already unique, such as `"abc"`.
- Strings where all characters are identical, such as `"aaaaa"`.
- Cases where removing a larger character earlier enables a smaller lexicographical result later.
- Cases where a character must be kept because it will not appear again later in the string.

The last point is the key difficulty of the problem. We can only safely remove a character if we know another copy exists later.

## Approaches

### Brute Force Approach

A brute force solution would generate all subsequences of the string, then filter them according to two conditions:

1. Every distinct character appears exactly once.
2. The subsequence contains all unique characters from the original string.

Among all valid subsequences, we would select the lexicographically smallest one.

This approach is correct because it exhaustively checks every possible candidate. However, it is computationally infeasible. A string of length `n` has `2^n` subsequences, which becomes enormous even for moderate values of `n`.

For `n = 10^4`, brute force is completely impossible.

### Key Insight for the Optimal Solution

The optimal solution relies on a greedy strategy combined with a monotonic stack.

The critical observation is this:

When constructing the answer, if the current character is smaller than the top of the stack, then replacing the larger character with the smaller one improves lexicographical order. However, we can only remove the larger character if it appears again later in the string.

This gives us a safe greedy rule:

- While the current character is smaller than the stack top,
- and the stack top appears later again,
- remove the stack top.

This ensures the result remains valid while becoming lexicographically smaller.

To implement this efficiently, we need:

- A frequency map or last occurrence map to know whether a character appears later.
- A stack to build the result.
- A set to track which characters are already included.

The stack naturally maintains the current best lexicographical sequence.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(2^n) | Generates all subsequences and filters valid ones |
| Optimal | O(n) | O(1) or O(n) | Greedy monotonic stack with character tracking |

## Algorithm Walkthrough

1. Count how many times each character appears in the string.

We need to know whether a character will appear again later. This allows us to safely remove a character from the stack if necessary.
2. Create an empty stack.

The stack will store the characters of our final answer in order.
3. Create a set called `in_stack`.

This set helps us quickly determine whether a character has already been included in the result. Since every character may appear only once, duplicates should be skipped.
4. Iterate through each character in the string.

For every character:

- Decrease its remaining frequency count because we are currently processing it.
- If the character is already in the stack, skip it. We already included it in the result.
5. While the stack can be improved, pop characters.

The stack can be improved if:

- The stack is not empty.
- The current character is smaller than the top of the stack.
- The top character appears again later.

If all conditions hold, removing the larger top character produces a lexicographically smaller result without losing that character permanently.
6. Push the current character onto the stack.

After removing unnecessary larger characters, we insert the current character and mark it as present in the stack.
7. Join the stack into a string.

The stack now contains the lexicographically smallest valid sequence.

### Why it works

The algorithm maintains an important invariant:

The stack always contains the smallest lexicographical subsequence possible for the characters processed so far, while still allowing completion into a valid final answer.

Whenever a larger character appears before a smaller one, we remove the larger character only if it can be added again later. This greedy decision never prevents a valid solution and always improves lexicographical order. Because every character is pushed and popped at most once, the algorithm remains linear.

## Python Solution

```
class Solution:
    def removeDuplicateLetters(self, s: str) -> str:
        remaining_count = {}
        
        for char in s:
            remaining_count[char] = remaining_count.get(char, 0) + 1
        
        stack = []
        in_stack = set()
        
        for char in s:
            remaining_count[char] -= 1
            
            if char in in_stack:
                continue
            
            while (
                stack
                and char < stack[-1]
                and remaining_count[stack[-1]] > 0
            ):
                removed_char = stack.pop()
                in_stack.remove(removed_char)
            
            stack.append(char)
            in_stack.add(char)
        
        return "".join(stack)
```

The implementation begins by counting the frequency of every character. This allows the algorithm to determine whether a character can safely be removed from the stack because another occurrence still exists later.

The `stack` stores the current best lexicographical sequence. The `in_stack` set ensures that each character appears only once.

During iteration, the current character's remaining count is reduced immediately because we are processing that occurrence now.

If the character is already present in the stack, it is skipped. This guarantees uniqueness.

The `while` loop is the heart of the greedy logic. It removes characters from the top of the stack while all three conditions hold:

- The stack is not empty.
- The current character is smaller.
- The top character appears later again.

Removing such characters improves lexicographical order without losing them permanently.

Finally, the current character is pushed onto the stack, and the final answer is created by joining all stack characters together.

## Go Solution

```
func removeDuplicateLetters(s string) string {
	remainingCount := make(map[byte]int)

	for i := 0; i < len(s); i++ {
		remainingCount[s[i]]++
	}

	stack := make([]byte, 0)
	inStack := make(map[byte]bool)

	for i := 0; i < len(s); i++ {
		char := s[i]
		remainingCount[char]--

		if inStack[char] {
			continue
		}

		for len(stack) > 0 &&
			char < stack[len(stack)-1] &&
			remainingCount[stack[len(stack)-1]] > 0 {

			removedChar := stack[len(stack)-1]
			stack = stack[:len(stack)-1]
			delete(inStack, removedChar)
		}

		stack = append(stack, char)
		inStack[char] = true
	}

	return string(stack)
}
```

The Go implementation follows the same algorithmic structure as the Python solution.

A `map[byte]int` stores remaining character counts because the input contains only lowercase English letters. The stack is implemented using a byte slice, which provides efficient append and pop operations.

Unlike Python sets, Go uses `map[byte]bool` to track membership in the stack.

Go slices naturally handle dynamic resizing, making them a good fit for stack behavior. Since the input size is small relative to integer limits, overflow concerns do not arise.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each character is pushed and popped at most once |
| Space | O(1) or O(n) | Stack, set, and frequency map store at most 26 characters |

Although the algorithm contains a nested `while` loop, the total number of stack pops across the entire execution is bounded by the number of pushes. Every character can enter and leave the stack only once, giving amortized linear complexity.

The space usage is technically bounded by the alphabet size of lowercase English letters, which is 26, so it can be considered O(1). More generally, relative to arbitrary character sets, it would be O(n).

## Edge Cases

One important edge case occurs when all characters are identical, such as `"aaaaa"`. A naive implementation might accidentally include duplicates or repeatedly process unnecessary stack operations. The algorithm handles this correctly because once a character is added to `in_stack`, all future occurrences are skipped.

Another important case is when the input is already lexicographically optimal, such as `"abc"`. Some incorrect greedy implementations may unnecessarily rearrange characters or attempt pops that should not happen. In this implementation, the `while` loop only removes characters when the current character is smaller, so already optimal sequences remain unchanged.

A particularly tricky edge case occurs when a character cannot safely be removed because it will not appear again later. Consider `"acdb"`. Even if a smaller character appears later, removing a character that has no future occurrences would make it impossible to include all distinct letters exactly once. The frequency map prevents this bug by ensuring pops only happen when the top character still exists later in the string.

Another subtle case involves repeated opportunities for popping, such as `"cbacdcbc"`. Multiple larger characters may need removal before inserting a smaller one. The `while` loop continues popping until the stack becomes valid again, ensuring the globally smallest lexicographical result instead of only making a single local improvement.
