## Solution

## Problem Understanding

The problem asks us to design a Tic-Tac-Toe game that supports efficient move processing on an `n x n` board. Two players take turns placing their marks on the board, and after every move we must determine whether that move causes the current player to win.

A player wins if they completely fill one of the following with their own marks:

- Any row
- Any column
- The main diagonal, from top-left to bottom-right
- The anti-diagonal, from top-right to bottom-left

The class exposes two operations:

- `TicTacToe(n)` initializes a board of size `n x n`
- `move(row, col, player)` places the player's mark at the specified location and returns:

- `0` if nobody has won yet
- `1` if player 1 wins
- `2` if player 2 wins

The important detail is the follow-up question asking whether we can do better than `O(n²)` per move. This strongly suggests that a naive board scan after every move is not optimal enough, and the intended solution is to maintain incremental state that allows winner detection in constant time.

The constraints are relatively small, with `n <= 100`, but the problem is a design problem rather than a pure performance problem. The goal is to demonstrate how to structure the data efficiently.

Several guarantees simplify the implementation:

- Every move is valid
- No move is repeated
- Players alternate correctly
- Once a winner exists, no more moves occur

These guarantees mean we do not need to validate moves or maintain complicated game state transitions. We only need to track progress toward victory conditions.

There are several edge cases that can cause mistakes in naive implementations. A move may belong simultaneously to a row, a column, and one or both diagonals. The center cell of an odd-sized board belongs to both diagonals. A player may win on the very last move of the game. Small board sizes such as `2 x 2` can produce victories very quickly, so the implementation must correctly detect wins immediately after every move.

## Approaches

### Brute Force Approach

The most straightforward solution is to explicitly store the entire board and check for a winner after every move.

When a player makes a move, we place their mark into the matrix. Then we scan:

- The entire row
- The entire column
- The main diagonal if applicable
- The anti-diagonal if applicable

Alternatively, some implementations scan the entire board after every move to check every row, column, and diagonal.

This approach is correct because it directly evaluates the game's winning conditions after every move. If any row, column, or diagonal contains only the current player's marks, then the player has won.

However, this method is inefficient. Even if we optimize by scanning only the affected row and column, each move still requires up to `O(n)` work. A full-board scan would be `O(n²)` per move.

The key inefficiency is that we repeatedly recompute information that changes only slightly between moves.

### Optimal Approach

The key observation is that we do not need the entire board to determine whether someone has won.

Instead, we can maintain running counts for:

- Every row
- Every column
- The main diagonal
- The anti-diagonal

We treat player 1 as `+1` and player 2 as `-1`.

When a player makes a move:

- Add `+1` or `-1` to the corresponding row count
- Add the same value to the column count
- Update diagonals if the move belongs to them

If any count reaches:

- `+n`, player 1 has filled that line
- `-n`, player 2 has filled that line

This works because mixed rows cancel each other out numerically, while a fully completed row accumulates an absolute value of `n`.

This allows winner detection in constant time.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) or O(n²) per move | O(n²) | Stores the full board and rescans lines repeatedly |
| Optimal | O(1) per move | O(n) | Tracks row, column, and diagonal counts incrementally |

## Algorithm Walkthrough

1. Initialize four tracking structures.

We maintain:

- An array `rows` of size `n`
- An array `cols` of size `n`
- An integer `diagonal`
- An integer `anti_diagonal`

Each value represents the cumulative score for that line.
2. Represent players numerically.

We assign:

- Player 1 → `+1`
- Player 2 → `-1`

This allows both players to share the same counters.
3. Process a move.

When a move `(row, col, player)` occurs:

- Compute the player's value
- Add it to `rows[row]`
- Add it to `cols[col]`

This updates the affected row and column immediately.
4. Update diagonals when necessary.

If `row == col`, the move lies on the main diagonal, so update `diagonal`.

If `row + col == n - 1`, the move lies on the anti-diagonal, so update `anti_diagonal`.
5. Check for a winner.

After updating the counters, check whether any affected counter has absolute value `n`.

If:

- `abs(rows[row]) == n`
- `abs(cols[col]) == n`
- `abs(diagonal) == n`
- `abs(anti_diagonal) == n`

then the current player has won.
6. Return the result.

Return:

- `player` if a winning line exists
- `0` otherwise

### Why it works

The invariant is that every row, column, and diagonal counter always equals the sum of contributions from the moves placed on that line.

Player 1 contributes `+1` and player 2 contributes `-1`.

A line can only reach absolute value `n` if all `n` cells in that line belong to the same player. Mixed ownership prevents the total from reaching `n` or `-n`.

Therefore, checking whether a counter reaches absolute value `n` exactly captures the winning condition.

## Python Solution

```
class TicTacToe:

    def __init__(self, n: int):
        self.n = n
        self.rows = [0] * n
        self.cols = [0] * n
        self.diagonal = 0
        self.anti_diagonal = 0

    def move(self, row: int, col: int, player: int) -> int:
        value = 1 if player == 1 else -1

        self.rows[row] += value
        self.cols[col] += value

        if row == col:
            self.diagonal += value

        if row + col == self.n - 1:
            self.anti_diagonal += value

        if (
            abs(self.rows[row]) == self.n
            or abs(self.cols[col]) == self.n
            or abs(self.diagonal) == self.n
            or abs(self.anti_diagonal) == self.n
        ):
            return player

        return 0

# Your TicTacToe object will be instantiated and called as such:
# obj = TicTacToe(n)
# param_1 = obj.move(row,col,player)
```

The constructor initializes the tracking structures. We only store cumulative counters rather than the entire board because the problem only requires efficient winner detection.

The `rows` and `cols` arrays track the cumulative values for each row and column. The two diagonal variables handle the special diagonals separately.

Inside `move`, we first convert the player identifier into either `+1` or `-1`. This encoding is the central optimization because it allows both players to share the same counters.

We then update the relevant row and column. If the move lies on either diagonal, we update those counters as well.

Finally, we check whether any affected counter has reached absolute value `n`. If so, the current player has completed a full line and wins immediately.

Because every operation involves only direct array access and constant-time arithmetic, the entire method runs in `O(1)` time.

## Go Solution

```
type TicTacToe struct {
	n             int
	rows          []int
	cols          []int
	diagonal      int
	antiDiagonal  int
}

func Constructor(n int) TicTacToe {
	return TicTacToe{
		n:            n,
		rows:         make([]int, n),
		cols:         make([]int, n),
		diagonal:     0,
		antiDiagonal: 0,
	}
}

func (this *TicTacToe) Move(row int, col int, player int) int {
	value := 1
	if player == 2 {
		value = -1
	}

	this.rows[row] += value
	this.cols[col] += value

	if row == col {
		this.diagonal += value
	}

	if row+col == this.n-1 {
		this.antiDiagonal += value
	}

	if abs(this.rows[row]) == this.n ||
		abs(this.cols[col]) == this.n ||
		abs(this.diagonal) == this.n ||
		abs(this.antiDiagonal) == this.n {
		return player
	}

	return 0
}

func abs(x int) int {
	if x < 0 {
		return -x
	}
	return x
}

/**
 * Your TicTacToe object will be instantiated and called as such:
 * obj := Constructor(n);
 * param_1 := obj.Move(row,col,player);
 */
```

The Go implementation follows the same algorithmic structure as the Python version. The main difference is that Go requires explicit slice allocation using `make`.

Go does not provide a built-in integer absolute value function for plain `int`, so we implement a small helper function `abs`.

The solution uses slices instead of fixed arrays because the board size `n` is only known at runtime.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(1) | Each move updates a constant number of counters |
| Space | O(n) | We store row and column counters |

The algorithm never scans the board. Every move updates at most one row, one column, and two diagonals. Each operation is direct indexing or arithmetic, so the runtime is constant.

The space usage comes from the two arrays that store row and column counts. The diagonal counters require only constant extra memory.

## Edge Cases

One important edge case occurs when a move belongs to both diagonals simultaneously. This only happens at the center cell of an odd-sized board. A buggy implementation may update only one diagonal or accidentally double-count incorrectly. The solution handles this cleanly by independently checking both diagonal conditions. If both conditions are true, both counters are updated.

Another important case is a very small board such as `2 x 2`. On small boards, victories happen extremely quickly, sometimes after only two moves by the same player. Implementations that assume a minimum number of turns before checking for victory can fail here. This solution checks for victory immediately after every move, so it works correctly regardless of board size.

A third edge case is when a line contains marks from both players. Naive counting approaches sometimes require separate counters per player and can become error-prone. Using `+1` and `-1` naturally cancels mixed ownership. For example, a row containing one move from each player produces a sum of zero, making it impossible to falsely detect a win.

Another subtle case occurs when the final move of the game creates a victory. Since the algorithm evaluates only the affected row, column, and diagonals after every move, it correctly detects wins even on the last possible turn without needing any additional full-board scan.
