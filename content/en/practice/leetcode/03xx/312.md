## Solution

## Problem Understanding

The problem gives us an array `nums`, where each element represents a balloon with a value written on it. When we burst a balloon at index `i`, we gain coins equal to:

$\text{coins} = nums[i-1] \cdot nums[i] \cdot nums[i+1]$

The important detail is that after a balloon is burst, it disappears from the array. This means the neighboring balloons change dynamically over time. The order in which balloons are burst completely changes the total number of coins collected.

If a balloon is at the boundary of the array and does not have a left or right neighbor, the missing neighbor is treated as a balloon with value `1`.

The task is to determine the maximum possible number of coins that can be collected by choosing the optimal bursting order.

The constraints are significant:

- `1 <= n <= 300`
- `0 <= nums[i] <= 100`

A naive brute-force solution that tries every bursting order becomes infeasible very quickly because the number of possible orders grows factorially. With `n = 300`, we need a much more efficient approach.

There are several important observations and edge cases:

- A balloon can contain `0`, which can drastically reduce the value of nearby bursts.
- Bursting order matters more than local greediness.
- Arrays of size `1` and `2` should still work correctly.
- Since neighbors change dynamically, directly simulating all possibilities is extremely expensive.
- The optimal substructure is not obvious when thinking about which balloon to burst first.

The key challenge is finding a way to structure the problem so that overlapping subproblems can be reused.

## Approaches

### Brute Force

A brute-force solution would try every possible order of bursting balloons.

For each recursive step:

- Choose one balloon to burst.
- Compute the coins gained from bursting it.
- Remove it from the array.
- Recursively solve the remaining problem.

Since there are `n` choices for the first balloon, `n - 1` for the next, and so on, the total number of orders is:

$n!$

This becomes astronomically large even for moderate `n`.

The brute-force solution is correct because it explores every possible sequence and therefore cannot miss the optimal answer. However, it is far too slow for `n = 300`.

### Optimal Dynamic Programming Approach

The critical insight is that it is difficult to reason about which balloon to burst first, but much easier to reason about which balloon is burst last within a subarray.

Suppose we consider a range `(left, right)` and ask:

"What is the maximum number of coins obtainable by bursting all balloons strictly between `left` and `right`?"

Now imagine balloon `k` is the last balloon burst in this interval.

At that moment:

- All balloons between `left` and `k` are already gone.
- All balloons between `k` and `right` are already gone.
- Therefore, the neighbors of `k` are exactly `left` and `right`.

This means the coins earned from bursting `k` last are:

$nums[left] \cdot nums[k] \cdot nums[right]$

The remaining work splits cleanly into two independent subproblems:

- Burst balloons in `(left, k)`
- Burst balloons in `(k, right)`

This gives us a classic interval dynamic programming recurrence.

We also add virtual boundary balloons with value `1` at both ends to simplify edge handling.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n!) | O(n) | Tries every bursting order recursively |
| Optimal Dynamic Programming | O(n³) | O(n²) | Uses interval DP and chooses the last balloon to burst |

## Algorithm Walkthrough

1. Create a new array with virtual boundary balloons.

We prepend and append `1` to the original array:

```
balloons = [1] + nums + [1]
```

These virtual balloons simplify edge calculations because every real balloon will always have valid left and right neighbors.
2. Define the DP state.

Let:

```
dp[left][right]
```

represent the maximum coins obtainable by bursting all balloons strictly between `left` and `right`.

The interval is open, meaning `left` and `right` themselves are not burst within this subproblem.
3. Initialize the DP table.

Create a 2D table initialized to `0`.

If there are no balloons between `left` and `right`, then the result is naturally `0`.
4. Process intervals by increasing length.

We must solve smaller intervals before larger ones because larger intervals depend on smaller subproblems.
5. Try every possible last balloon.

For every interval `(left, right)`, try every `k` such that:

```
left < k < right
```

Treat `k` as the last balloon burst in this interval.
6. Compute the coins for this choice.

The total becomes:

$dp[left][k] + dp[k][right] + balloons[left] \cdot balloons[k] \cdot balloons[right]$

This works because:

- `dp[left][k]` handles the left side
- `dp[k][right]` handles the right side
- Bursting `k` last earns the product of its current neighbors
7. Take the maximum over all choices of `k`.

Store the best possible value in `dp[left][right]`.
8. Return the answer.

The final answer is:

```
dp[0][n - 1]
```

where `n` is the length of the padded array.

### Why it works

The correctness comes from optimal substructure.

For any interval `(left, right)`, the final balloon burst must be some `k` inside that interval. Once `k` is chosen as the last balloon, the remaining balloons split into two completely independent intervals. Since every possible `k` is considered, and each subproblem is solved optimally, the algorithm guarantees the globally optimal result.

## Python Solution

```
from typing import List

class Solution:
    def maxCoins(self, nums: List[int]) -> int:
        balloons = [1] + nums + [1]
        n = len(balloons)

        dp = [[0] * n for _ in range(n)]

        for length in range(2, n):
            for left in range(n - length):
                right = left + length

                for k in range(left + 1, right):
                    coins = (
                        dp[left][k]
                        + dp[k][right]
                        + balloons[left] * balloons[k] * balloons[right]
                    )

                    dp[left][right] = max(dp[left][right], coins)

        return dp[0][n - 1]
```

The implementation begins by padding the array with virtual balloons containing value `1`. This removes the need for special boundary checks during the DP transitions.

The `dp` table stores solutions for every interval. Initially, all entries are `0` because intervals with no balloons contribute no coins.

The outer loop iterates over interval lengths from small to large. This ordering is essential because each state depends on smaller intervals that must already be computed.

For every interval `(left, right)`, the algorithm tries every possible `k` as the final balloon burst. The total coins are computed by combining:

- The optimal solution for the left subinterval
- The optimal solution for the right subinterval
- The coins earned from bursting `k` last

The maximum value across all possible `k` choices becomes the optimal answer for that interval.

Finally, `dp[0][n - 1]` contains the solution for the entire array.

## Go Solution

```
func maxCoins(nums []int) int {
	balloons := make([]int, 0, len(nums)+2)
	balloons = append(balloons, 1)
	balloons = append(balloons, nums...)
	balloons = append(balloons, 1)

	n := len(balloons)

	dp := make([][]int, n)
	for i := range dp {
		dp[i] = make([]int, n)
	}

	for length := 2; length < n; length++ {
		for left := 0; left < n-length; left++ {
			right := left + length

			for k := left + 1; k < right; k++ {
				coins := dp[left][k] +
					dp[k][right] +
					balloons[left]*balloons[k]*balloons[right]

				if coins > dp[left][right] {
					dp[left][right] = coins
				}
			}
		}
	}

	return dp[0][n-1]
}
```

The Go implementation follows the same interval DP structure as the Python version.

Slices are used instead of Python lists, and the 2D DP table is created manually using nested slices. Go integers are sufficient because the maximum possible answer fits within standard integer ranges for the given constraints.

Unlike Python, Go does not provide a built-in `max` function for integers, so the comparison is handled explicitly with an `if` statement.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n³) | Three nested loops over interval length, left boundary, and last balloon |
| Space | O(n²) | DP table stores results for all interval pairs |

The algorithm uses interval dynamic programming. There are `O(n²)` possible intervals, and for each interval we try every possible last balloon, resulting in `O(n)` work per state. Therefore the total runtime becomes `O(n³)`.

The DP table contains one entry for every `(left, right)` pair, requiring `O(n²)` memory.

## Edge Cases

One important edge case is an array containing only one balloon. In this situation, the algorithm must correctly apply the virtual boundary balloons. For example, `[9]` should produce:

$1 \cdot 9 \cdot 1 = 9$

The implementation handles this naturally because the padded array becomes `[1,9,1]`, and the interval DP computes the correct single transition.

Another critical edge case is arrays containing zeros. A greedy approach can easily fail here because bursting a zero-valued balloon at the wrong time may destroy opportunities for larger products elsewhere. The interval DP avoids this issue because it systematically evaluates every possible last balloon configuration.

Arrays with repeated values are also important. Cases like `[1,1,1,1]` can expose bugs where intervals are processed in the wrong order or where overlapping subproblems are not combined correctly. Since the DP table is filled strictly from smaller intervals to larger intervals, all dependencies are guaranteed to be available before use.

A final subtle edge case involves very small intervals. When there are no balloons between `left` and `right`, the value should be `0`. The implementation handles this implicitly because the DP table is initialized with zeros and only updated when valid interior balloons exist.
