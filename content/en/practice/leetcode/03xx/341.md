## Solution

## Problem Understanding

The problem gives us a nested list structure where each element can either be a single integer or another list containing additional `NestedInteger` objects. Those nested objects can themselves contain integers or more lists, potentially to many levels of depth.

We need to implement an iterator that behaves like a normal sequential iterator over a flat list of integers, even though the underlying structure is deeply nested.

For example, if the input is:

```
[[1,1],2,[1,1]]
```

the iterator should return values in this order:

```
1, 1, 2, 1, 1
```

Similarly, for:

```
[1,[4,[6]]]
```

the iterator should produce:

```
1, 4, 6
```

The important detail is that the iterator must preserve the original left-to-right traversal order of the nested structure. This is effectively a depth-first traversal where lists are expanded in place.

The interface provides three important methods on `NestedInteger`:

- `isInteger()` tells us whether the object stores a single integer
- `getInteger()` returns the integer value
- `getList()` returns the nested list

We are not supposed to implement this interface ourselves. We only use it.

The constraints are fairly moderate. The outer list can contain up to 500 elements, but nesting can be arbitrarily deep. That means recursion depth and traversal strategy matter. Since the iterator may be called repeatedly through `next()` and `hasNext()`, we also need to think carefully about how work is distributed across those calls.

Several edge cases are important:

- Deeply nested single values such as `[[[[1]]]]`
- Empty nested lists such as `[[],[[]],1]`
- Multiple consecutive nested lists before reaching an integer
- Calling `hasNext()` multiple times before `next()`
- Inputs containing only one integer

A naive implementation can easily mishandle traversal order or repeatedly reprocess nested structures.

## Approaches

### Brute Force Approach

The most straightforward solution is to completely flatten the nested structure during initialization.

We perform a depth-first traversal of the entire nested list and store all integers into a simple array. After preprocessing, the iterator becomes trivial because `next()` simply returns the next element from the flattened array.

This approach is correct because a recursive DFS naturally visits integers in the same left-to-right order required by the problem.

For example:

```
[1,[4,[6]]]
```

would be flattened into:

```
[1,4,6]
```

Then the iterator only needs an index pointer.

The downside is that we eagerly process the entire structure up front, even if only a few elements are ever requested. This increases initialization cost and requires storing every integer simultaneously.

### Optimal Approach

A more elegant solution uses a stack to perform lazy flattening.

Instead of flattening everything immediately, we only expand nested lists when needed. The key insight is that `hasNext()` can prepare the stack so that the top element is always an integer ready to be returned.

We maintain a stack of `NestedInteger` objects. Initially, we push the input list onto the stack in reverse order so the leftmost element appears on top.

Whenever the top of the stack is a list rather than an integer, we pop it and push its contents back onto the stack in reverse order. This effectively expands one nesting level while preserving traversal order.

This approach is more memory efficient in practice because it only keeps the currently active traversal state instead of storing every integer separately.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Fully flatten entire structure during initialization |
| Optimal | O(n) | O(d) average, O(n) worst case | Lazy stack-based DFS traversal |

Here, `n` is the total number of nested elements and integers, while `d` represents nesting depth.

## Algorithm Walkthrough

1. Initialize an empty stack that will simulate iterative depth-first traversal.
2. Push all elements from `nestedList` onto the stack in reverse order. Reversing is important because stacks are LIFO structures. By pushing from right to left, the leftmost element becomes the next item processed.
3. Implement `hasNext()` so it prepares the stack before returning.
4. While the stack is not empty:

- Look at the top element without removing it.
- If the top element is an integer, we are ready to return `True` because `next()` can safely consume it.
- If the top element is a nested list, pop it from the stack and expand it.
5. To expand a nested list:

- Retrieve its internal list using `getList()`.
- Push those elements onto the stack in reverse order.
- Reversing preserves correct left-to-right traversal order.
6. If the stack eventually becomes empty, there are no remaining integers, so return `False`.
7. Implement `next()` by first ensuring `hasNext()` is true, then popping the top integer and returning its value.

The reason we use a stack is that the traversal is naturally depth-first. A stack mirrors the recursive call stack that DFS would normally use.

### Why it works

The key invariant is that after `hasNext()` completes, either the stack is empty or the top of the stack is guaranteed to be an integer.

Every nested list encountered is expanded exactly once, and every integer is returned exactly once. Because elements are always pushed in reverse order, the iterator preserves the original left-to-right traversal order of the nested structure.

## Python Solution

```
# """
# This is the interface that allows for creating nested lists.
# You should not implement it, or speculate about its implementation
# """
# class NestedInteger:
#     def isInteger(self) -> bool:
#         pass
#
#     def getInteger(self) -> int:
#         pass
#
#     def getList(self) -> [NestedInteger]:
#         pass

from typing import List

class NestedIterator:
    def __init__(self, nestedList: [NestedInteger]):
        self.stack = []

        # Push in reverse order so leftmost element is processed first
        for item in reversed(nestedList):
            self.stack.append(item)

    def next(self) -> int:
        if self.hasNext():
            return self.stack.pop().getInteger()

    def hasNext(self) -> bool:
        while self.stack:
            top = self.stack[-1]

            # If top is an integer, iterator is ready
            if top.isInteger():
                return True

            # Otherwise expand the nested list
            self.stack.pop()
            nested = top.getList()

            for item in reversed(nested):
                self.stack.append(item)

        return False

# Your NestedIterator object will be instantiated and called as such:
# i, v = NestedIterator(nestedList), []
# while i.hasNext():
#     v.append(i.next())
```

The implementation begins by creating a stack that stores traversal state. During initialization, all top-level elements are pushed in reverse order so iteration begins with the leftmost element.

The `hasNext()` method performs the core lazy flattening logic. Instead of immediately flattening everything, it repeatedly checks the stack's top element. If that element is already an integer, iteration can continue immediately.

If the top element is a list, the algorithm removes it and expands its contents onto the stack. Reversing the insertion order ensures correct traversal order.

The `next()` method becomes very small because `hasNext()` guarantees the stack top is an integer before retrieval.

This design cleanly separates traversal preparation from value extraction.

## Go Solution

```
/**
 * // This is the interface that allows for creating nested lists.
 * // You should not implement it, or speculate about its implementation
 * type NestedInteger struct {
 * }
 *
 * func (this NestedInteger) IsInteger() bool {}
 * func (this NestedInteger) GetInteger() int {}
 * func (this NestedInteger) GetList() []*NestedInteger {}
 */

type NestedIterator struct {
	stack []*NestedInteger
}

func Constructor(nestedList []*NestedInteger) *NestedIterator {
	stack := []*NestedInteger{}

	// Push in reverse order
	for i := len(nestedList) - 1; i >= 0; i-- {
		stack = append(stack, nestedList[i])
	}

	return &NestedIterator{
		stack: stack,
	}
}

func (this *NestedIterator) Next() int {
	if this.HasNext() {
		n := len(this.stack)
		val := this.stack[n-1].GetInteger()
		this.stack = this.stack[:n-1]
		return val
	}

	return -1
}

func (this *NestedIterator) HasNext() bool {
	for len(this.stack) > 0 {
		top := this.stack[len(this.stack)-1]

		if top.IsInteger() {
			return true
		}

		// Remove nested list from stack
		this.stack = this.stack[:len(this.stack)-1]

		nested := top.GetList()

		// Push children in reverse order
		for i := len(nested) - 1; i >= 0; i-- {
			this.stack = append(this.stack, nested[i])
		}
	}

	return false
}
```

The Go implementation follows exactly the same logic as the Python version, but uses slices to represent the stack.

Go does not provide built-in stack structures, so stack operations are implemented using slice append and truncation operations.

Unlike Python, Go requires explicit handling of slice lengths and indexing. The last element of the slice represents the top of the stack.

The implementation returns `-1` in `Next()` if no element exists, although the LeetCode judge guarantees valid usage patterns.

| --- |
| [1,1] |

| 1 |

Next `next()` returns another `1`.

Stack becomes:

| Stack Bottom -> Top |
| --- |
| [1,1] |
| 2 |

Next `next()` returns `2`.

Remaining nested list expands similarly and produces:

```
1, 1
```

[1,1,2,1,1]

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each nested element and integer is pushed and popped at most once |
| Space | O(d) average, O(n) worst case | Stack stores active traversal state |

The runtime is linear because every element participates in at most one expansion and one removal from the stack.

The space complexity depends on nesting depth. In balanced cases, only the current traversal path is stored. In the worst case, such as a very wide structure, the stack may temporarily contain all elements.

## Edge Cases

One important edge case is empty nested lists. Inputs such as `[[],[[]],1]` can easily break naive implementations if they assume every list contains at least one element. The stack-based solution handles this naturally because expanding an empty list simply adds nothing to the stack, and traversal continues normally.

Another tricky case is deeply nested structures like `[[[[[1]]]]]`. Recursive brute-force solutions risk recursion depth problems in some languages. The iterative stack-based approach avoids recursion entirely, making it more robust for arbitrarily deep nesting.

A third important edge case involves repeated calls to `hasNext()` before calling `next()`. A poorly designed iterator might accidentally advance state multiple times. In this implementation, `hasNext()` only expands nested lists until the top is an integer. Once that invariant is established, repeated calls to `hasNext()` do not modify the stack further, ensuring consistent iterator behavior.
