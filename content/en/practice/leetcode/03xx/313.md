## Solution

## Problem Understanding

The problem asks us to generate the sequence of "super ugly numbers" and return the `n`th element in that sequence.

A super ugly number is defined as a positive integer whose prime factors all belong to the given array `primes`. This definition is similar to the classic "Ugly Number II" problem, except instead of being limited to the primes `[2, 3, 5]`, we are given an arbitrary list of valid prime factors.

The sequence always starts with `1`. Even though `1` has no prime factors, it is considered valid because it does not violate the rule that all prime factors must come from `primes`.

For example, if:

```
primes = [2,7,13,19]
```

then valid super ugly numbers include:

```
1, 2, 4, 7, 8, 13, 14, 16, 19, 26, ...
```

because every number can be factorized entirely using only `2`, `7`, `13`, and `19`.

The input consists of:

- `n`, the position in the sequence we want to compute
- `primes`, the list of allowed prime factors

The output is the `n`th super ugly number in ascending order.

The constraints are important:

- `n` can be as large as `100000`
- `primes.length` can be up to `100`
- Each prime is at most `1000`

These constraints immediately tell us that brute-force factorization of every integer would be far too slow. We need an algorithm that generates valid numbers directly rather than checking every integer one by one.

Several edge cases are important:

- `n = 1`, where the answer must always be `1`
- Multiple primes generating the same value, such as:

- `2 * 7 = 14`
- `7 * 2 = 14`

A naive implementation could accidentally insert duplicates.
- Large `n`, where inefficient approaches will time out
- A single prime in the array, such as `[2]`, where the sequence becomes powers of two

The problem guarantees:

- All primes are unique
- Primes are sorted
- The answer fits in a 32-bit signed integer

These guarantees simplify implementation because we do not need to deduplicate the input or handle overflow beyond standard integer limits.

## Approaches

### Brute Force Approach

A straightforward solution would be to iterate through positive integers one by one and determine whether each number is a super ugly number.

To check whether a number is valid, we repeatedly divide it by every allowed prime factor. If the number can eventually be reduced to `1`, then all of its prime factors belong to the `primes` array.

For example, with:

```
primes = [2,7,13,19]
```

to check whether `28` is valid:

```
28 ÷ 2 = 14
14 ÷ 2 = 7
7 ÷ 7 = 1
```

Since we reach `1`, the number is valid.

This approach is correct because every number is explicitly verified against the definition. However, it is extremely inefficient. Most integers are not super ugly numbers, so we waste enormous time checking irrelevant values.

When `n` reaches `100000`, the required answer may be very large, making brute force completely impractical.

### Optimal Dynamic Programming Approach

The key insight is that every super ugly number is generated from an earlier super ugly number multiplied by one of the allowed primes.

Suppose we already know:

```
1, 2, 4, 7, 8
```

Then future numbers can be generated by multiplying these values with the primes.

This observation allows us to build the sequence incrementally in sorted order.

We maintain:

- A DP array `ugly`, where `ugly[i]` stores the `(i + 1)`th super ugly number
- One pointer for each prime
- A candidate value for each prime

Each pointer indicates which previously generated ugly number should next be multiplied by that prime.

This technique is similar to merging multiple sorted lists:

```
2 × ugly[]
7 × ugly[]
13 × ugly[]
19 × ugly[]
```

At every step:

- Choose the smallest candidate
- Append it to the sequence
- Advance every pointer that produced that minimum

Advancing all matching pointers is crucial for avoiding duplicates.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Extremely large, impractical | O(1) | Checks every integer individually |
| Optimal Dynamic Programming | O(n × k) | O(n + k) | Efficiently generates numbers directly |

Here:

- `n` is the target index
- `k` is `len(primes)`

## Algorithm Walkthrough

1. Create an array `ugly` of size `n`.

This array stores the super ugly numbers in sorted order. Initialize:

```
ugly[0] = 1
```

because the sequence always starts with `1`.
2. Create a pointer array `indices`.

Each prime has its own pointer indicating which ugly number should be multiplied next.

Initially:

```
indices = [0, 0, 0, ...]
```
3. Create a candidate array `next_values`.

Each candidate is:

```
primes[i] * ugly[indices[i]]
```

Initially:

```
next_values = primes
```

because every pointer starts at index `0`, and:

```
ugly[0] = 1
```
4. Repeat until the DP array is filled.

For every new position:

- Find the minimum value in `next_values`
- Append it to `ugly`

This minimum is guaranteed to be the next smallest valid super ugly number.
5. Advance all matching pointers.

Multiple primes may generate the same value.

Example:

```
2 × 7 = 14
7 × 2 = 14
```

If we only advance one pointer, duplicates would appear.

Therefore, for every prime whose candidate equals the chosen minimum:

- Increment its pointer
- Recompute its next candidate
6. Continue until `ugly[n - 1]` is computed.

Return the final value.

### Why it works

The algorithm maintains the invariant that the `ugly` array is always sorted and contains only valid super ugly numbers.

Every new number is generated by multiplying an already valid super ugly number by one allowed prime, so every generated value remains valid.

Because we always choose the smallest available candidate across all primes, numbers are generated in ascending order without missing any valid values.

Advancing all matching pointers prevents duplicates from entering the sequence.

## Python Solution

```
from typing import List

class Solution:
    def nthSuperUglyNumber(self, n: int, primes: List[int]) -> int:
        k = len(primes)

        ugly = [0] * n
        ugly[0] = 1

        indices = [0] * k
        next_values = primes[:]

        for i in range(1, n):
            next_ugly = min(next_values)
            ugly[i] = next_ugly

            for j in range(k):
                if next_values[j] == next_ugly:
                    indices[j] += 1
                    next_values[j] = (
                        ugly[indices[j]] * primes[j]
                    )

        return ugly[-1]
```

The implementation begins by initializing the dynamic programming array `ugly`. This array stores the sequence in sorted order, and the first value is set to `1`.

The `indices` array tracks which ugly number each prime should currently multiply. Initially, every prime points to index `0`.

The `next_values` array stores the next candidate generated by each prime. At the start, each prime multiplies `1`, so the candidate values are simply the primes themselves.

Inside the loop, the algorithm repeatedly selects the smallest candidate value. That value becomes the next super ugly number.

After inserting the value into the sequence, the code scans through all primes. Any prime whose candidate matches the chosen minimum must advance its pointer. This ensures duplicate values are skipped correctly.

Finally, after generating all `n` numbers, the last element of the DP array is returned.

## Go Solution

```
func nthSuperUglyNumber(n int, primes []int) int {
    k := len(primes)

    ugly := make([]int, n)
    ugly[0] = 1

    indices := make([]int, k)
    nextValues := make([]int, k)

    copy(nextValues, primes)

    for i := 1; i < n; i++ {
        nextUgly := nextValues[0]

        for j := 1; j < k; j++ {
            if nextValues[j] < nextUgly {
                nextUgly = nextValues[j]
            }
        }

        ugly[i] = nextUgly

        for j := 0; j < k; j++ {
            if nextValues[j] == nextUgly {
                indices[j]++
                nextValues[j] = ugly[indices[j]] * primes[j]
            }
        }
    }

    return ugly[n-1]
}
```

The Go implementation follows the same logic as the Python version. The primary difference is manual handling of minimum selection because Go does not provide a built-in `min()` function for slices.

Slices are used instead of dynamic arrays, and `copy()` initializes the `nextValues` array from `primes`.

Integer overflow is not a concern because the problem guarantees the result fits inside a 32-bit signed integer.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n × k) | Each iteration scans all primes |
| Space | O(n + k) | Stores ugly numbers and pointer arrays |

Here:

- `n` is the number of super ugly numbers to generate
- `k` is the number of primes

For every generated ugly number, the algorithm scans all `k` primes to find and update candidate values. Since this happens `n` times, the total runtime is `O(n × k)`.

The space usage comes primarily from:

- The `ugly` array of size `n`
- The pointer and candidate arrays of size `k`

## Edge Cases

One important edge case occurs when `n = 1`. Since the sequence always begins with `1`, the correct answer must immediately be `1` regardless of the primes array. Implementations that try to start generating values before handling this case can accidentally skip the initial value or return the first prime instead.

Another critical edge case involves duplicate generation paths. Multiple primes can generate the same super ugly number through different multiplication combinations. For example:

```
2 × 7 = 14
7 × 2 = 14
```

If the implementation advances only one pointer after generating `14`, the number would appear twice in the sequence. The solution avoids this by advancing every pointer whose candidate equals the chosen minimum.

A third important edge case is when the primes array contains only one element. In that situation, the sequence becomes repeated powers of a single prime. For example:

```
primes = [2]
```

produces:

```
1, 2, 4, 8, 16, ...
```

Some implementations incorrectly assume multiple primes exist or rely on merging behavior that fails with a single candidate stream. This implementation works correctly because the logic naturally handles any number of primes, including one.

Another subtle edge case involves large values of `n`. Since `n` can reach `100000`, inefficient approaches that repeatedly factorize integers or use expensive heap deduplication carelessly may time out. The dynamic programming approach guarantees efficient sequential generation in `O(n × k)` time, which comfortably fits within the problem constraints.
