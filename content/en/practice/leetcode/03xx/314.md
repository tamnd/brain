## Solution

## Problem Understanding

The problem asks us to compute the **vertical order traversal** of a binary tree. In simpler terms, we are given a binary tree and need to group its nodes based on their vertical columns when viewed from the top. Each column is defined by the horizontal distance from the root: the root is at column `0`, its left child at column `-1`, its right child at column `1`, and so on. Within the same column, nodes are ordered **top-to-bottom** based on their row depth. If multiple nodes share the same row and column, they should appear in **left-to-right order**, preserving the natural order of tree traversal.

The input is the root of a binary tree. The output is a list of lists of integers, where each inner list contains all node values in the same vertical column, ordered from leftmost column to rightmost column.

Constraints indicate that the tree has at most 100 nodes, with node values in the range `[-100, 100]`. This small size allows us to use BFS or DFS without worrying about performance, though we still want an efficient solution.

Key edge cases include:

1. An **empty tree** (`root = None`), which should return an empty list.
2. Trees where all nodes are aligned vertically (linked-list shaped), which could test column indexing.
3. Trees with multiple nodes sharing the same column and row, ensuring the **left-to-right order** is respected.

## Approaches

The **brute-force approach** would involve using DFS to traverse the tree, recording both the column and row of each node. Then, we would sort all nodes first by column, then by row, and finally by left-to-right order for nodes sharing the same row and column. While correct, this approach requires **explicit sorting of all nodes**, which is unnecessary and adds overhead.

The **optimal approach** uses **Breadth-First Search (BFS)**. BFS naturally traverses nodes level by level, which ensures that nodes in the same row are processed left to right. During BFS, we maintain a mapping from **column indices to a list of node values**. We also track the **minimum and maximum column indices** seen so we can assemble the result efficiently in left-to-right column order. BFS avoids sorting and respects both row and left-to-right ordering naturally.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(N log N) | O(N) | DFS traversal + sorting all nodes by column and row |
| Optimal | O(N) | O(N) | BFS traversal with a column hashmap, no sorting required |

## Algorithm Walkthrough

1. If the root is `None`, return an empty list immediately.
2. Initialize a queue for BFS, storing tuples of `(node, column_index)`.
3. Create a hash map (`defaultdict`) to map `column_index` to a list of node values.
4. Track the minimum and maximum column indices observed during traversal.
5. While the queue is not empty, pop the first element `(node, column_index)`.
6. Append `node.val` to the hash map entry corresponding to `column_index`.
7. If the node has a left child, enqueue `(node.left, column_index - 1)`.
8. If the node has a right child, enqueue `(node.right, column_index + 1)`.
9. After BFS completes, iterate from `min_column` to `max_column` and collect values from the hash map in that order to form the final result.

**Why it works:** BFS ensures that nodes are processed level by level. By enqueuing left children before right children, we respect left-to-right ordering within each row. Mapping nodes to columns guarantees that nodes in the same vertical column are grouped together. Tracking min and max columns ensures the result is returned from leftmost to rightmost column.

## Python Solution

```
from collections import defaultdict, deque
from typing import Optional, List

# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

class Solution:
    def verticalOrder(self, root: Optional[TreeNode]) -> List[List[int]]:
        if not root:
            return []
        
        column_table = defaultdict(list)
        min_column = max_column = 0
        queue = deque([(root, 0)])
        
        while queue:
            node, column = queue.popleft()
            column_table[column].append(node.val)
            min_column = min(min_column, column)
            max_column = max(max_column, column)
            
            if node.left:
                queue.append((node.left, column - 1))
            if node.right:
                queue.append((node.right, column + 1))
        
        return [column_table[col] for col in range(min_column, max_column + 1)]
```

The Python implementation follows BFS logic directly. The queue maintains nodes along with their column indices. Each node value is appended to its respective column list. After traversal, the result is assembled using the recorded min and max column indices.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func verticalOrder(root *TreeNode) [][]int {
    if root == nil {
        return [][]int{}
    }

    columnTable := make(map[int][]int)
    type NodeInfo struct {
        node   *TreeNode
        column int
    }

    queue := []NodeInfo{{root, 0}}
    minColumn, maxColumn := 0, 0

    for len(queue) > 0 {
        current := queue[0]
        queue = queue[1:]
        node, column := current.node, current.column

        columnTable[column] = append(columnTable[column], node.Val)
        if column < minColumn {
            minColumn = column
        }
        if column > maxColumn {
            maxColumn = column
        }

        if node.Left != nil {
            queue = append(queue, NodeInfo{node.Left, column - 1})
        }
        if node.Right != nil {
            queue = append(queue, NodeInfo{node.Right, column + 1})
        }
    }

    result := [][]int{}
    for col := minColumn; col <= maxColumn; col++ {
        result = append(result, columnTable[col])
    }

    return result
}
```

In Go, we explicitly define a `NodeInfo` struct to store the node and its column index. A slice simulates the BFS queue. Map operations and slice appends maintain the column grouping. Handling `nil` nodes prevents runtime panics.

## Worked Examples

**Example 1:** `[3,9,20,null,null,15,7]`

BFS queue progression:

| Step | Queue | Column Table | Min | Max |
| --- | --- | --- | --- | --- |
| 1 | [(3,0)] | {0:[3]} | 0 | 0 |
| 2 | [(9,-1),(20,1)] | {-1:[9],0:[3],1:[20]} | -1 | 1 |
| 3 | [(20,1),(15,0),(7,2)] | {-1:[9],0:[3,15],1:[20],2:[7]} | -1 | 2 |

Result: `[[9],[3,15],[20],[7]]`

**Example 2:** `[3,9,8,4,0,1,7]`

Result: `[[4],[9],[3,0,1],[8],[7]]`

**Example 3:** `[1,2,3,4,10,9,11,null,5,null,null,null,null,null,null,null,6]`

Result: `[[4],[2,5],[1,10,9,6],[3],[11]]`

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(N) | Each node is visited exactly once in BFS |
| Space | O(N) | Queue can contain up to all nodes in one level; column table stores all node values |

The algorithm scales linearly with the number of nodes. The extra space is proportional to the number of nodes because we store all node values in the column table and potentially in the BFS queue simultaneously.
