## Solution

## Problem Understanding

This problem asks us to find the best empty land cell in a grid where we can build a house such that the sum of distances to all buildings is minimized.

The grid contains three kinds of cells:

- `0` represents empty land, which can be walked through and can also potentially hold the new house.
- `1` represents a building, which blocks movement.
- `2` represents an obstacle, which also blocks movement.

Movement is restricted to the four cardinal directions: up, down, left, and right.

The task is not to find the shortest path between two points. Instead, we must evaluate empty cells and determine which one minimizes the total distance to every building combined.

For example, suppose there are three buildings. If an empty cell is distance `3` from the first building, `3` from the second, and `1` from the third, then its total distance is `7`.

A crucial detail is that the chosen empty land must be reachable from every building. If even one building cannot reach that land because obstacles or walls block the path, then that location is invalid.

The constraints are relatively small:

- `1 <= m, n <= 50`
- Maximum grid size is `2500` cells.

This size is large enough that repeatedly running expensive searches from every empty cell to every building can become too slow. However, it is small enough that performing Breadth-First Search several times is practical.

Several edge cases are important:

- A grid with only buildings and no empty land should return `-1`.
- A grid where obstacles isolate some buildings from others should return `-1`.
- Multiple empty cells may exist, but only those reachable from all buildings are valid.
- There is guaranteed to be at least one building.
- A single building next to an empty cell should return the Manhattan distance directly.

## Approaches

### Brute Force Approach

The brute-force strategy is to try every empty land cell as a possible house location.

For each empty cell:

1. Run a BFS to every building.
2. Compute the shortest distance to each building.
3. Sum all distances.
4. Keep the minimum valid total.

This approach is correct because it explicitly evaluates every possible candidate location and computes the exact shortest-path distance to every building.

However, it is inefficient because the same paths are recomputed repeatedly. If there are many empty cells and many buildings, we perform a massive number of redundant BFS traversals.

Suppose:

- `E` is the number of empty cells
- `B` is the number of buildings
- Grid size is `m * n`

The brute-force approach can degrade toward:

- `O(E * B * m * n)`

In dense grids, this becomes unnecessarily expensive.

### Optimal Approach

The key insight is that BFS should start from buildings, not from empty lands.

Why?

Because the number of buildings is usually smaller than the number of empty cells, and each BFS from a building naturally computes shortest distances to all reachable empty lands simultaneously.

We maintain two matrices:

- `distance_sum[r][c]`

- Total accumulated distance from all buildings.
- `reach_count[r][c]`

- Number of buildings that can reach this cell.

For every building:

1. Run BFS outward.
2. Whenever an empty land is reached:

- Add the current BFS distance to `distance_sum`
- Increment `reach_count`

After processing all buildings, the valid answer is the minimum:

```
distance_sum[r][c]
```

among cells where:

```
reach_count[r][c] == total_buildings
```

This avoids redundant work because each BFS computes shortest distances to all reachable cells in one traversal.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(E * B * m * n) | O(m * n) | Runs BFS from every empty cell |
| Optimal | O(B * m * n) | O(m * n) | Runs BFS from each building once |

## Algorithm Walkthrough

1. Initialize bookkeeping structures.

Create two matrices with the same dimensions as the grid:

- `distance_sum[r][c]`

- Stores cumulative distance from all buildings.
- `reach_count[r][c]`

- Stores how many buildings can reach this empty land.

Also count the total number of buildings.
2. Iterate through the grid to find buildings.

Every cell with value `1` becomes a BFS starting point.

We process buildings one at a time because BFS guarantees shortest-path distances in an unweighted grid.
3. Perform BFS from the current building.

Use a queue containing:

```
(row, col, distance)
```

Also maintain a `visited` matrix for this BFS run so we do not revisit cells.
4. Explore neighboring cells.

For each popped cell, try moving in four directions:

- up
- down
- left
- right

A move is valid only if:

- it stays within bounds
- the cell is empty land (`0`)
- it has not already been visited in this BFS
5. Update distance information.

Whenever an empty land is reached:

- add BFS distance to `distance_sum`
- increment `reach_count`

This works because BFS explores cells level by level, so the first time we reach a cell is always the shortest path from the current building.
6. Repeat BFS for every building.

Each building contributes its shortest distances to all reachable empty cells.
7. Find the best valid empty land.

After all BFS traversals finish, scan every empty land cell.

A cell is valid only if:

```
reach_count[r][c] == total_buildings
```

Among valid cells, choose the minimum accumulated distance.
8. Return the result.

- If at least one valid cell exists, return the minimum distance.
- Otherwise return `-1`.

### Why it works

Breadth-First Search guarantees shortest distances in an unweighted grid because nodes are explored level by level.

By running BFS from every building, we ensure:

- every reachable empty land receives the shortest distance contribution from that building
- `distance_sum` accumulates the correct total distance
- `reach_count` verifies that all buildings can reach the cell

Therefore, any cell considered in the final step has the exact total shortest distance from all buildings, and taking the minimum gives the optimal answer.

## Python Solution

```
from collections import deque
from typing import List

class Solution:
    def shortestDistance(self, grid: List[List[int]]) -> int:
        rows = len(grid)
        cols = len(grid[0])

        distance_sum = [[0] * cols for _ in range(rows)]
        reach_count = [[0] * cols for _ in range(rows)]

        total_buildings = 0

        directions = [(1, 0), (-1, 0), (0, 1), (0, -1)]

        def bfs(start_row: int, start_col: int) -> None:
            visited = [[False] * cols for _ in range(rows)]
            queue = deque()

            queue.append((start_row, start_col, 0))
            visited[start_row][start_col] = True

            while queue:
                row, col, dist = queue.popleft()

                for dr, dc in directions:
                    new_row = row + dr
                    new_col = col + dc

                    if (
                        0 <= new_row < rows
                        and 0 <= new_col < cols
                        and not visited[new_row][new_col]
                        and grid[new_row][new_col] == 0
                    ):
                        visited[new_row][new_col] = True

                        distance_sum[new_row][new_col] += dist + 1
                        reach_count[new_row][new_col] += 1

                        queue.append((new_row, new_col, dist + 1))

        for row in range(rows):
            for col in range(cols):
                if grid[row][col] == 1:
                    total_buildings += 1
                    bfs(row, col)

        result = float("inf")

        for row in range(rows):
            for col in range(cols):
                if (
                    grid[row][col] == 0
                    and reach_count[row][col] == total_buildings
                ):
                    result = min(result, distance_sum[row][col])

        return result if result != float("inf") else -1
```

The implementation begins by creating two matrices:

- `distance_sum`
- `reach_count`

These matrices accumulate information across all BFS traversals.

The `bfs` helper function performs a standard level-order traversal starting from one building. The queue stores coordinates together with the current distance from the source building.

A local `visited` matrix is necessary because each building's BFS traversal must independently track visited cells. Reusing visitation across buildings would incorrectly block valid traversals.

Whenever an empty land is reached, we update:

```
distance_sum[new_row][new_col] += dist + 1
reach_count[new_row][new_col] += 1
```

The distance increment uses `dist + 1` because neighboring cells are exactly one step farther away.

After all buildings are processed, the final scan identifies cells reachable from every building and selects the minimum total distance.

## Go Solution

```
package main

import "container/list"

func shortestDistance(grid [][]int) int {
	rows := len(grid)
	cols := len(grid[0])

	distanceSum := make([][]int, rows)
	reachCount := make([][]int, rows)

	for i := 0; i < rows; i++ {
		distanceSum[i] = make([]int, cols)
		reachCount[i] = make([]int, cols)
	}

	totalBuildings := 0

	directions := [][]int{
		{1, 0},
		{-1, 0},
		{0, 1},
		{0, -1},
	}

	type State struct {
		row  int
		col  int
		dist int
	}

	bfs := func(startRow, startCol int) {
		visited := make([][]bool, rows)

		for i := 0; i < rows; i++ {
			visited[i] = make([]bool, cols)
		}

		queue := list.New()
		queue.PushBack(State{startRow, startCol, 0})

		visited[startRow][startCol] = true

		for queue.Len() > 0 {
			front := queue.Front()
			queue.Remove(front)

			current := front.Value.(State)

			for _, dir := range directions {
				newRow := current.row + dir[0]
				newCol := current.col + dir[1]

				if newRow >= 0 &&
					newRow < rows &&
					newCol >= 0 &&
					newCol < cols &&
					!visited[newRow][newCol] &&
					grid[newRow][newCol] == 0 {

					visited[newRow][newCol] = true

					distanceSum[newRow][newCol] += current.dist + 1
					reachCount[newRow][newCol]++

					queue.PushBack(
						State{
							newRow,
							newCol,
							current.dist + 1,
						},
					)
				}
			}
		}
	}

	for row := 0; row < rows; row++ {
		for col := 0; col < cols; col++ {
			if grid[row][col] == 1 {
				totalBuildings++
				bfs(row, col)
			}
		}
	}

	result := int(^uint(0) >> 1)

	for row := 0; row < rows; row++ {
		for col := 0; col < cols; col++ {
			if grid[row][col] == 0 &&
				reachCount[row][col] == totalBuildings {

				if distanceSum[row][col] < result {
					result = distanceSum[row][col]
				}
			}
		}
	}

	if result == int(^uint(0)>>1) {
		return -1
	}

	return result
}
```

The Go implementation follows the same algorithmic structure as the Python version.

A few Go-specific details are worth noting:

- Go does not have a built-in deque, so `container/list` is used as the BFS queue.
- A custom `State` struct stores row, column, and distance together.
- Go does not provide `infinity` for integers, so the maximum integer value is computed using:

```
int(^uint(0) >> 1)
```

- Boolean visitation matrices must be explicitly allocated row by row.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(B * m * n) | Each building performs one BFS over the grid |
| Space | O(m * n) | Distance matrices, reach matrices, and BFS visitation |

The key observation is that each BFS touches each cell at most once. If there are `B` buildings, total work becomes:

```
B * (m * n)
```

Since the maximum grid size is only `2500`, this solution comfortably fits within limits.

## Edge Cases

One important edge case occurs when there is no empty land at all. For example:

```
[[1]]
```

A naive implementation might accidentally return `0` because no traversal is required. However, the problem specifically requires building on an empty land cell. Since none exists, the correct answer is `-1`. The implementation handles this naturally because no candidate cell satisfies the final validation condition.

Another tricky case happens when some buildings are isolated by obstacles. Consider:

```
[
  [1,2,0],
  [2,2,2],
  [0,2,1]
]
```

Although empty cells exist, no empty land is reachable from both buildings. A buggy solution might compute partial distances and mistakenly accept them. The `reach_count` matrix prevents this problem because a cell is only considered valid if every building can reach it.

A third important edge case involves narrow grids such as a single row or single column. These cases often expose indexing mistakes or incorrect boundary checks. Since the implementation consistently validates coordinates using:

```
0 <= new_row < rows
0 <= new_col < cols
```

it correctly handles grids with dimensions equal to `1`.

Finally, another subtle case occurs when multiple empty cells produce the same minimum total distance. The algorithm does not rely on uniqueness. It simply computes the minimum over all valid cells, so ties are handled automatically and correctly.
