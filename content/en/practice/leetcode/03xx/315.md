## Solution

## Problem Understanding

The problem asks us to compute, for every position in the array, how many numbers appearing to its right are strictly smaller than the current value.

Given an array `nums`, we must produce another array `counts` of the same length. For each index `i`, `counts[i]` should contain the number of elements `nums[j]` such that:

- `j > i`
- `nums[j] < nums[i]`

For example, in the array `[5,2,6,1]`, the value `5` has two smaller elements to its right, `2` and `1`, so the answer for index `0` is `2`.

The important detail is that we are only interested in elements to the right of the current index. Elements to the left do not matter. We also only count strictly smaller values, not equal values.

The constraints are large:

- `nums.length` can be up to `100000`
- Values range from `-10000` to `10000`

These constraints immediately rule out slow quadratic solutions. A naive nested loop solution would require up to `10^10` comparisons in the worst case, which is far too slow.

The problem belongs to the class of inversion-counting style problems. We need an efficient way to determine how many smaller values appear after each element while processing the array.

Several edge cases are important:

- Arrays with duplicate values, because equal elements should not be counted as smaller.
- Arrays already sorted in increasing order, where every answer is `0`.
- Arrays sorted in decreasing order, where every position has the maximum possible count.
- Negative numbers, since ordering still matters regardless of sign.
- Single-element arrays, where the answer is always `[0]`.

## Approaches

### Brute Force Approach

The simplest solution is to examine every pair of indices.

For each index `i`, we iterate through all indices `j > i` and count how many values satisfy `nums[j] < nums[i]`.

This works because it directly follows the definition of the problem. Every possible pair is checked, so the counts are guaranteed to be correct.

However, this approach is extremely inefficient for large inputs. With `n` elements, we perform roughly:

$$\frac{n(n-1)}{2}$$

comparisons.

For `n = 100000`, this becomes far too large.

### Optimal Approach, Merge Sort Counting

The key insight is that this problem resembles inversion counting.

During merge sort, when two sorted halves are merged together, we can determine how many elements from the right half move ahead of elements from the left half. Those moved elements are exactly the smaller numbers appearing after the current element.

The important observation is:

- Left half elements originally appear before right half elements in the array.
- If a right-half element is smaller during merging, it contributes to the count for remaining left-half elements.

By augmenting merge sort with counting logic, we can compute all answers in `O(n log n)` time.

Instead of sorting plain numbers, we sort pairs:

- `(value, original_index)`

The original index allows us to place counts back into the correct answer positions.

| --- | --- | --- | --- |

| Optimal Merge Sort | O(n log n) | O(n) | Counts smaller elements during merge operations |

## Algorithm Walkthrough

### Optimal Merge Sort Counting Algorithm

1. Create an array of pairs containing both the value and its original index.

We cannot lose the original positions because the final answer must correspond to the original array order. Each element becomes:

```
(value, original_index)
```
2. Initialize a result array filled with zeros.

This array stores the number of smaller elements appearing after each index.
3. Perform merge sort on the pair array.

Merge sort recursively divides the array into smaller halves until each segment contains only one element.
4. During the merge step, compare elements from the left and right halves.

Since both halves are already sorted, we can efficiently determine how many right-half elements are smaller than a left-half element.
5. Track how many right-half elements have been placed before the current left-half element.

Suppose we are merging:

```
Left:  [5]
Right: [2,1]
```

When `1` and `2` are merged before `5`, they are both smaller and originally to the right of `5`.

Therefore, `5` gains a count of `2`.
6. When taking an element from the left half, add the number of already-merged right-half elements to its count.

This works because all previously merged right-half elements are smaller than the current left element.
7. Continue recursively merging until the entire array is sorted.

By the time merge sort completes, every element has accumulated the total number of smaller elements appearing after it.

### Why it works

The correctness comes from the merge sort invariant.

At every merge step:

- The left half contains elements originally appearing earlier in the array.
- The right half contains elements originally appearing later.
- Both halves are individually sorted.

Whenever a right-half element is merged before a left-half element, it means:

```
right_value < left_value
```

and the right element originally appeared after the left element.

Therefore, every such event contributes exactly one valid smaller-after-self count.

Because merge sort processes all such cross-half relationships exactly once, every valid pair is counted exactly once.

## Python Solution

```
from typing import List

class Solution:
    def countSmaller(self, nums: List[int]) -> List[int]:
        n = len(nums)
        result = [0] * n
        
        indexed_nums = [(nums[i], i) for i in range(n)]
        
        def merge_sort(left: int, right: int) -> None:
            if left >= right:
                return
            
            mid = (left + right) // 2
            
            merge_sort(left, mid)
            merge_sort(mid + 1, right)
            
            merge(left, mid, right)
        
        def merge(left: int, mid: int, right: int) -> None:
            temp = []
            
            i = left
            j = mid + 1
            
            right_count = 0
            
            while i <= mid and j <= right:
                if indexed_nums[j][0] < indexed_nums[i][0]:
                    temp.append(indexed_nums[j])
                    right_count += 1
                    j += 1
                else:
                    value, original_index = indexed_nums[i]
                    result[original_index] += right_count
                    temp.append(indexed_nums[i])
                    i += 1
            
            while i <= mid:
                value, original_index = indexed_nums[i]
                result[original_index] += right_count
                temp.append(indexed_nums[i])
                i += 1
            
            while j <= right:
                temp.append(indexed_nums[j])
                j += 1
            
            for k in range(len(temp)):
                indexed_nums[left + k] = temp[k]
        
        merge_sort(0, n - 1)
        
        return result
```

The implementation begins by pairing every value with its original index. This is essential because merge sort rearranges elements, but the final counts must correspond to the original positions.

The `result` array stores the answer for every index.

The `merge_sort` function recursively divides the array into smaller subarrays until each subarray contains a single element. Single-element arrays are already sorted.

The important logic happens inside the `merge` function.

Two pointers traverse the sorted left and right halves. Whenever an element from the right half is smaller than the current left-half element, it is merged first and `right_count` increases.

`right_count` represents how many smaller right-side elements have already moved ahead of the current left-side element.

When a left element is finally merged, we add `right_count` to its answer because all those right-side elements are smaller and originally appeared after it.

Finally, the merged sorted segment is copied back into the original array structure so future merges operate on sorted data.

## Go Solution

```
func countSmaller(nums []int) []int {
	n := len(nums)

	type Pair struct {
		value int
		index int
	}

	result := make([]int, n)
	indexedNums := make([]Pair, n)

	for i := 0; i < n; i++ {
		indexedNums[i] = Pair{
			value: nums[i],
			index: i,
		}
	}

	var mergeSort func(int, int)

	mergeSort = func(left int, right int) {
		if left >= right {
			return
		}

		mid := (left + right) / 2

		mergeSort(left, mid)
		mergeSort(mid+1, right)

		merge(left, mid, right)
	}

	var merge func(int, int, int)

	merge = func(left int, mid int, right int) {
		temp := make([]Pair, 0, right-left+1)

		i := left
		j := mid + 1

		rightCount := 0

		for i <= mid && j <= right {
			if indexedNums[j].value < indexedNums[i].value {
				temp = append(temp, indexedNums[j])
				rightCount++
				j++
			} else {
				result[indexedNums[i].index] += rightCount
				temp = append(temp, indexedNums[i])
				i++
			}
		}

		for i <= mid {
			result[indexedNums[i].index] += rightCount
			temp = append(temp, indexedNums[i])
			i++
		}

		for j <= right {
			temp = append(temp, indexedNums[j])
			j++
		}

		for k := 0; k < len(temp); k++ {
			indexedNums[left+k] = temp[k]
		}
	}

	mergeSort(0, n-1)

	return result
}
```

The Go implementation closely mirrors the Python solution, but there are several language-specific differences.

Go does not support tuples directly, so a custom `Pair` struct stores both the value and original index.

Slices are used instead of Python lists. The temporary merge array is created with a predefined capacity to reduce reallocations.

Go integers are sufficient for this problem because counts never exceed `100000`, well within integer limits.

An empty input is naturally handled because `mergeSort(0, -1)` immediately returns due to the base condition.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | Merge sort performs log n levels of recursion with linear work per level |
| Space | O(n) | Temporary merge arrays and recursion stack require additional memory |

The algorithm achieves optimal asymptotic performance for this problem class.

Each merge step processes every element exactly once per recursion level, and merge sort has `log n` levels. Therefore the total runtime is:

$$O(n \log n)$$

The additional memory comes from temporary arrays used during merging and recursive function calls.

## Edge Cases

### Duplicate Values

Duplicate numbers are one of the most common sources of bugs in this problem.

The problem asks for strictly smaller elements, not smaller-or-equal elements. If the merge logic uses `<=` incorrectly, equal values may be counted accidentally.

The implementation avoids this by only incrementing `rightCount` when:

```
right_value < left_value
```

Equal values are merged from the left side first, preventing overcounting.

### Already Sorted Increasing Array

An increasing array such as:

```
[1,2,3,4,5]
```

should produce all zeros because no element has a smaller value to its right.

Some incorrect implementations accidentally count elements simply because they appear later.

In this implementation, `rightCount` only increases when a right-half element is strictly smaller, so increasing arrays correctly remain all zeros.

### Strictly Decreasing Array

A decreasing array such as:

```
[5,4,3,2,1]
```

represents the worst-case counting scenario.

Every element has all remaining elements smaller than itself.

This stresses whether the algorithm correctly accumulates multiple smaller elements during merges. The merge-sort counting logic handles this naturally because every right-half element moves ahead of left-half elements during merging.

### Negative Numbers

Negative values can expose bugs in algorithms relying on array indexing or coordinate assumptions.

This implementation compares values directly and never depends on positivity, so negative numbers work identically to positive numbers.

### Single Element Input

A single-element array has no elements to its right.

The recursion base case:

```
left >= right
```

immediately terminates processing, and the initialized zero result is returned unchanged.
