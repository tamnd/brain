## Solution

## Problem Understanding

The problem asks us to reverse a string that is represented as an array of characters. Instead of receiving a normal immutable string, the function receives a mutable character array such as `["h","e","l","l","o"]`. The goal is to rearrange the characters so that the array becomes reversed in-place.

The phrase "in-place" is extremely important here. It means we are not allowed to allocate another array of the same size and copy elements into it. Instead, we must directly modify the original input array. The problem also explicitly requires `O(1)` extra memory, which means the algorithm may only use a constant amount of additional storage regardless of input size.

The input represents a sequence of printable ASCII characters. Each element in the array contains a single character. The output is not returned as a new array. Instead, the original array itself must be modified so that its contents appear in reverse order.

The constraints tell us that the array length can be as large as `10^5`. This size is large enough that inefficient approaches could become problematic. However, since reversing a string is fundamentally a linear operation, an `O(n)` solution is ideal and expected.

Several edge cases are important to recognize before implementing the algorithm. A single-character array such as `["a"]` should remain unchanged after reversal. Arrays with even lengths and odd lengths must both work correctly. Repeated characters such as `["a","a","a"]` should not cause issues. The problem guarantees that the array contains at least one character, so we do not need to handle completely empty input.

## Approaches

### Brute Force Approach

A straightforward approach is to create a new array and fill it with characters from the original array in reverse order. For example, if the input is:

```
["h","e","l","l","o"]
```

we could iterate from the end of the array to the beginning and append each character into a new array:

```
["o","l","l","e","h"]
```

Finally, we could copy the reversed values back into the original array.

This approach produces the correct answer because every character at position `i` is moved to position `n - 1 - i`, which is exactly the definition of reversal.

However, this method uses an additional array of size `n`, which violates the requirement that the algorithm use only `O(1)` extra memory. Even though the runtime is acceptable, the extra space makes this solution invalid for the problem constraints.

### Optimal Approach

The key insight is that reversing a sequence only requires swapping symmetric elements.

The first character should swap with the last character, the second character should swap with the second-to-last character, and so on. This naturally leads to the two pointers technique.

We maintain:

- One pointer at the beginning of the array
- One pointer at the end of the array

At each step, we swap the characters at these positions and move the pointers toward the center.

Because each swap places two characters into their final positions, we only need to process half of the array. Most importantly, swapping is done directly inside the original array, so the algorithm satisfies the in-place requirement with constant extra memory.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Creates an additional reversed array |
| Optimal | O(n) | O(1) | Uses two pointers and swaps characters in-place |

## Algorithm Walkthrough

1. Initialize two pointers, `left` and `right`.

The `left` pointer starts at index `0`, which represents the beginning of the array. The `right` pointer starts at index `len(s) - 1`, which represents the end of the array.
2. Continue processing while `left < right`.

Once the pointers cross each other, every necessary swap has already been completed. For odd-length arrays, the middle character does not need to move.
3. Swap the characters at `left` and `right`.

This operation places both characters into their correct reversed positions simultaneously.
4. Move the pointers inward.

Increment `left` by `1` and decrement `right` by `1`.

This advances the algorithm toward the center of the array.
5. Repeat until the pointers meet or cross.

At that point, the entire array has been reversed in-place.

### Why it works

The algorithm maintains the invariant that all positions outside the range `[left, right]` are already correctly reversed. Each swap fixes exactly two positions, one at the beginning and one at the end. Because the pointers move inward after every swap, the unresolved region continuously shrinks until no elements remain to process. Therefore, the final array is fully reversed.

## Python Solution

```
from typing import List

class Solution:
    def reverseString(self, s: List[str]) -> None:
        """
        Do not return anything, modify s in-place instead.
        """
        
        left = 0
        right = len(s) - 1

        while left < right:
            s[left], s[right] = s[right], s[left]

            left += 1
            right -= 1
```

The implementation begins by initializing two pointers. `left` starts at the first index and `right` starts at the last index.

The `while left < right` condition ensures that swapping continues only while there are still unmatched pairs remaining. During each iteration, the tuple assignment:

```
s[left], s[right] = s[right], s[left]
```

swaps the two characters in-place without needing a temporary variable.

After the swap, the pointers move inward toward the center of the array. Eventually, the pointers meet or cross, meaning all characters have been reversed.

The function does not return anything because the input array itself is modified directly, which matches the problem requirements.

## Go Solution

```
func reverseString(s []byte) {
    left := 0
    right := len(s) - 1

    for left < right {
        s[left], s[right] = s[right], s[left]

        left++
        right--
    }
}
```

The Go implementation follows exactly the same logic as the Python version. Since Go strings are immutable, the problem uses a `[]byte` slice instead. This allows direct modification of individual characters.

The swap operation is concise because Go supports simultaneous assignment:

```
s[left], s[right] = s[right], s[left]
```

No special handling is needed for empty slices because the constraints guarantee at least one character.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each character is processed at most once |
| Space | O(1) | Only two pointer variables are used |

The algorithm performs one swap for each pair of characters, which means approximately `n / 2` iterations. Since constants are ignored in Big O notation, the runtime is `O(n)`.

The space complexity is `O(1)` because the algorithm only uses two integer pointers regardless of input size. No additional arrays or dynamic data structures are allocated.

## Edge Cases

A single-character array is an important edge case because no swaps are required. A buggy implementation might accidentally attempt unnecessary operations or mishandle pointer boundaries. In this implementation, the condition `left < right` immediately fails when both pointers start at index `0`, so the array remains unchanged correctly.

Odd-length arrays require careful handling because the middle character should remain in place. For example, in `["a","b","c","d","e"]`, the character `"c"` does not move. The algorithm naturally handles this because the pointers eventually meet at the middle index, and the loop stops before any unnecessary swap occurs.

Arrays with repeated characters can hide logical mistakes because incorrect swaps may still appear visually correct. For instance, reversing `["x","x","x"]` should still produce the same array. Since the algorithm swaps based purely on position rather than character uniqueness, repeated values are handled safely and consistently.

Even-length arrays are another common source of off-by-one errors. In arrays like `["a","b","c","d"]`, every character participates in a swap, and the pointers cross after the final valid swap. The condition `left < right` guarantees the loop stops at the correct moment without overshooting.
