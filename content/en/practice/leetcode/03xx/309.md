## Solution

## Problem Understanding

The problem gives an array `prices` where `prices[i]` represents the stock price on day `i`. You may buy and sell the stock multiple times in order to maximize total profit, but there are two important restrictions.

First, you cannot hold multiple stocks at the same time. This means you must sell your current stock before buying another one.

Second, after selling a stock, there is a mandatory cooldown period of one day. If you sell on day `i`, then you are not allowed to buy on day `i + 1`.

The goal is to compute the maximum possible profit that can be achieved while respecting these rules.

For example, with:

```
prices = [1,2,3,0,2]
```

One optimal sequence is:

- Buy at price 1
- Sell at price 2
- Cooldown for one day
- Buy at price 0
- Sell at price 2

This produces a total profit of `1 + 2 = 3`.

The constraints are important:

- `1 <= prices.length <= 5000`
- `0 <= prices[i] <= 1000`

An input size of up to 5000 means exponential solutions are not feasible. We need an algorithm that is approximately linear or quadratic at worst. Since stock trading decisions naturally depend on previous states, this problem strongly suggests a dynamic programming approach.

Several edge cases are important:

- An array with only one day, no transaction is possible.
- Strictly decreasing prices, the best answer is zero because buying would never help.
- Alternating rises and drops, where the cooldown restriction changes which transactions are optimal.
- Consecutive profitable opportunities, where a naive greedy algorithm may fail because taking one trade can block another due to cooldown.

## Approaches

### Brute Force Approach

A brute force solution explores every possible decision on every day.

At each index, we can choose to:

- Buy the stock
- Sell the stock
- Skip the day

The recursion tracks whether we currently hold a stock and whether we are in a cooldown state. By trying all combinations, we eventually discover the maximum possible profit.

This approach is correct because it exhaustively checks every legal sequence of transactions. However, it is extremely inefficient because the number of possible decision paths grows exponentially with the number of days.

With up to 5000 days, an exponential solution becomes completely impractical.

### Key Insight for the Optimal Solution

The important observation is that the future only depends on a small amount of information about the current day.

Specifically, on each day we only care about three states:

- Holding a stock
- Not holding a stock and free to buy
- In cooldown after selling

Instead of exploring all transaction histories, we store the best possible profit for each state on each day.

This transforms the problem into dynamic programming.

The cooldown rule creates a dependency between days, because after selling we cannot immediately buy again. Dynamic programming works well here because the optimal choice for today depends only on optimal states from previous days.

We can update the states iteratively in linear time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(n) | Recursively explores every transaction sequence |
| Optimal | O(n) | O(1) | Dynamic programming with rolling state variables |

## Algorithm Walkthrough

### Dynamic Programming States

We define three states for each day:

- `hold`: maximum profit when currently holding a stock
- `sold`: maximum profit when we just sold a stock today
- `rest`: maximum profit when not holding a stock and not in cooldown

These states completely describe every valid situation.

### Step-by-Step Algorithm

1. Initialize the states for day 0.

On the first day:

- If we buy the stock, profit becomes `-prices[0]`
- We cannot sell yet, so `sold = 0`
- Doing nothing gives `rest = 0`
2. Iterate through the remaining days.

For each price, compute the next values for all three states.
3. Update the `hold` state.

We can either:

- Continue holding the previous stock
- Buy today from the `rest` state

So:

```
hold = max(previous_hold, previous_rest - price)
```
4. Update the `sold` state.

The only way to sell today is if we were previously holding a stock.

```
sold = previous_hold + price
```
5. Update the `rest` state.

We can remain resting, or enter rest after cooldown.

```
rest = max(previous_rest, previous_sold)
```
6. Continue until all days are processed.
7. Return the best non-holding state.

At the end, holding a stock is not useful because unsold stock does not contribute profit.

Therefore:

```
answer = max(sold, rest)
```

### Why it works

The algorithm works because the three states fully capture all legal transaction situations at every day. Every valid action transitions between states in exactly one correct way. Since each state stores the maximum achievable profit for that situation, the transitions preserve optimality. By processing days sequentially, we guarantee that every decision is built on previously optimal subproblems.

## Python Solution

```
from typing import List

class Solution:
    def maxProfit(self, prices: List[int]) -> int:
        hold = -prices[0]
        sold = 0
        rest = 0

        for price in prices[1:]:
            previous_hold = hold
            previous_sold = sold
            previous_rest = rest

            hold = max(previous_hold, previous_rest - price)
            sold = previous_hold + price
            rest = max(previous_rest, previous_sold)

        return max(sold, rest)
```

The implementation directly follows the state transition logic described earlier.

The variable `hold` represents the best profit achievable while currently owning a stock. Initially, buying on day 0 produces a negative profit equal to the stock price.

The variable `sold` represents the profit immediately after selling a stock. This state is important because it enforces the cooldown restriction.

The variable `rest` represents being free to buy again.

Inside the loop, we save the previous values before updating them. This is necessary because all transitions for the current day must depend on the previous day's states, not partially updated values.

The transitions mirror the recurrence relations exactly:

- `hold` either keeps holding or buys today
- `sold` always comes from selling a held stock
- `rest` either continues resting or finishes cooldown

Finally, we return the maximum profit among non-holding states.

## Go Solution

```
func maxProfit(prices []int) int {
    hold := -prices[0]
    sold := 0
    rest := 0

    for i := 1; i < len(prices); i++ {
        price := prices[i]

        previousHold := hold
        previousSold := sold
        previousRest := rest

        if previousRest-price > previousHold {
            hold = previousRest - price
        } else {
            hold = previousHold
        }

        sold = previousHold + price

        if previousSold > previousRest {
            rest = previousSold
        } else {
            rest = previousRest
        }
    }

    if sold > rest {
        return sold
    }

    return rest
}
```

The Go implementation is nearly identical to the Python version, but Go does not provide a built-in `max` function for integers, so comparisons are handled with explicit `if` statements.

Go slices are naturally safe for iteration, and integer overflow is not a concern here because the maximum possible profit is small relative to Go's integer range.

Unlike Python, Go requires explicit variable declarations and type inference through `:=`.

## Edge Cases

A single-day input is an important edge case because no sale can occur. A naive implementation might incorrectly assume at least two days exist and access invalid indices. This implementation handles the case naturally because the loop over remaining days never executes, and the final result remains zero.

Strictly decreasing prices are another common source of mistakes. Some greedy solutions may incorrectly perform transactions that produce negative profit. The dynamic programming transitions prevent this because the `rest` state always preserves the option of doing nothing, ensuring the result never becomes negative.

Cooldown-sensitive transaction chains are particularly tricky. For example, in `[1,2,3,0,2]`, a naive algorithm might attempt consecutive transactions without respecting the cooldown rule. The explicit `sold` and `rest` states correctly enforce the restriction because buying is only allowed from the `rest` state, never directly after selling.

Arrays with repeated or zero prices can also expose logical errors. In cases like `[0,0,0,0]`, buying and selling never produces profit. The implementation safely handles this because all state transitions naturally preserve the best achievable value, which remains zero throughout execution.
