## Solution

## Problem Understanding

The problem is asking us to design a class `MovingAverage` that computes the moving average of the most recent `size` integers from a data stream. In other words, as integers are received one by one, we must maintain a "sliding window" of the last `size` values and return the average of the values currently in that window. If fewer than `size` values have been received, the average is calculated over all received values so far.

The input consists of two parts: the initialization with an integer `size` defining the window, and repeated calls to `next(val)` with integers from the stream. The output is the moving average after each `next` call, which is a floating-point number.

Constraints clarify the problem scale: `size` is between 1 and 1000, each value `val` is within `-10^5` to `10^5`, and at most `10^4` calls will be made. This indicates that while the window is relatively small, the number of operations can be moderately large, so efficiency matters. Importantly, this guarantees the stream always contains integers and never invalid input, and the window size is always positive, so we do not have to handle division by zero.

Edge cases include when the number of received elements is smaller than the window size, when all values are negative or zero, and when the values approach the upper or lower limits of `val`. Naive implementations may recalculate the sum of the entire window on each call, which could be inefficient for larger streams.

## Approaches

The brute-force approach is straightforward: maintain a list of all received values, and on each call to `next(val)`, append the new value and calculate the sum of the last `size` elements to compute the average. This works correctly but is inefficient because calculating the sum of the window repeatedly takes O(size) time per call. Over many calls, this results in O(N * size) total time, where N is the number of calls, which can be slow if `size` is large.

The optimal approach leverages a queue (or deque) to maintain the current window and a running sum. Instead of recomputing the sum each time, we update the sum incrementally: add the new value to the sum, and if the queue exceeds the window size, subtract the oldest value. This reduces the time complexity of each `next` call to O(1) while using O(size) additional space for the queue.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(N * size) | O(N) | Maintains all values, sums last `size` values on each call |
| Optimal | O(1) per call | O(size) | Uses queue and running sum for constant-time update |

## Algorithm Walkthrough

1. Initialize the `MovingAverage` object with a fixed `size`. Store this size, initialize an empty queue to track the current window, and set a variable `window_sum` to 0 to track the sum of elements in the window.
2. On each call to `next(val)`, add `val` to the queue and increment `window_sum` by `val`. This step keeps the sum of elements in the window updated without recalculating it entirely.
3. Check if the queue exceeds the maximum window size. If it does, remove the oldest element from the queue and subtract it from `window_sum`. This maintains the invariant that the sum always represents only the last `size` elements.
4. Compute the moving average by dividing `window_sum` by the length of the queue. Initially, the queue may contain fewer elements than `size`, so divide by the actual queue length.
5. Return the moving average as a float.

**Why it works**: The queue always contains the most recent elements up to the window size, and `window_sum` always reflects the sum of the elements currently in the queue. This ensures that the average is accurate at every step without recomputing the sum from scratch.

## Python Solution

```
from collections import deque

class MovingAverage:

    def __init__(self, size: int):
        self.size = size
        self.queue = deque()
        self.window_sum = 0

    def next(self, val: int) -> float:
        self.queue.append(val)
        self.window_sum += val
        if len(self.queue) > self.size:
            self.window_sum -= self.queue.popleft()
        return self.window_sum / len(self.queue)
```

The Python implementation uses a `deque` to efficiently add and remove elements from the window. We maintain a running sum `window_sum` to avoid recalculating the sum each time. Adding a new value and possibly removing an old value ensures the queue never exceeds the maximum size, while computing the average is a simple division by the current queue length.

## Go Solution

```
package main

type MovingAverage struct {
    size      int
    window    []int
    windowSum int
}

func Constructor(size int) MovingAverage {
    return MovingAverage{
        size:   size,
        window: []int{},
    }
}

func (this *MovingAverage) Next(val int) float64 {
    this.window = append(this.window, val)
    this.windowSum += val
    if len(this.window) > this.size {
        this.windowSum -= this.window[0]
        this.window = this.window[1:]
    }
    return float64(this.windowSum) / float64(len(this.window))
}
```

In Go, we use a slice as a queue. Appending is simple, but removing the oldest element requires creating a new slice excluding the first element. We maintain a running sum to achieve O(1) updates. We also cast integers to `float64` when calculating the average to avoid integer division.

## Edge Cases

One important edge case is when the number of elements received is smaller than the window size. The implementation correctly divides by the actual number of elements in the queue rather than the maximum size, avoiding incorrect averages.

A second edge case involves negative numbers. The running sum may become negative, and integer arithmetic could produce errors if not handled properly. Maintaining the sum as an integer and converting to float when dividing ensures accurate results.

A third edge case is the maximum window size with the maximum number of calls. Efficient O(1) updates and O(size) space usage ensure that the algorithm remains performant and does not exceed memory limits, even at the upper bounds specified in the constraints.
