## Solution

## Problem Understanding

This problem gives us a collection of mathematical relationships between variables. Each equation represents a division relationship such as:

```
a / b = 2.0
```

This means that `a` is twice as large as `b`, or equivalently:

```
a = 2 * b
```

We are then asked to answer additional division queries such as:

```
a / c = ?
```

The challenge is that the answer may not be directly provided. Instead, we may need to derive it indirectly using other equations.

For example:

```
a / b = 2
b / c = 3
```

From these two equations we can infer:

```
a / c = (a / b) * (b / c) = 2 * 3 = 6
```

The input consists of three arrays:

- `equations`, where each entry contains two variables
- `values`, where `values[i]` corresponds to the ratio represented by `equations[i]`
- `queries`, where each query asks for a ratio between two variables

The output should contain one result for each query.

If the relationship between two variables cannot be determined, we return `-1.0`.

A few important observations come from the constraints:

- The number of equations and queries is small, at most 20
- Variables are strings, not numeric IDs
- Relationships are guaranteed to be consistent
- Division by zero never occurs

Even though the constraints are small, the structure of the problem strongly suggests graph traversal.

The important edge cases include:

- Queries involving variables that never appear in any equation
- Queries where both variables are identical
- Queries where no path exists between the variables
- Multi-step relationships requiring traversal through several intermediate variables
- Reverse lookups such as `b / a` when only `a / b` is given

These cases are where naive implementations often fail.

## Approaches

### Brute Force Approach

A brute-force solution would try to evaluate each query independently by recursively testing every possible equation combination until the target variable is found.

For every query:

- Start from the numerator variable
- Try all equations containing that variable
- Recursively continue searching
- Multiply ratios along the path
- Backtrack when dead ends are reached

This approach works because every valid answer corresponds to some chain of equations connecting the two variables.

However, the brute-force method repeatedly scans the equations array for every recursive step. Since the same searches may be repeated many times across different queries, it becomes inefficient.

The biggest weakness is that the equations are not organized for fast lookup.

### Key Insight

The equations naturally form a weighted directed graph.

If:

```
a / b = 2
```

then we can create:

- an edge from `a` to `b` with weight `2`
- an edge from `b` to `a` with weight `1/2`

Now the problem becomes:

> Find a path between two variables and multiply all edge weights along that path.

This is a classic graph traversal problem.

Because the graph is small and sparse, Depth-First Search works extremely well.

We first build an adjacency list so neighbors can be found efficiently. Then for each query we perform DFS from the source variable to the target variable while accumulating the product of weights along the traversal path.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(Q * E²) | O(E) | Repeatedly scans equations for each recursive search |
| Optimal Graph DFS | O(Q * (V + E)) | O(V + E) | Builds adjacency list and traverses efficiently |

Where:

- `Q` = number of queries
- `V` = number of variables
- `E` = number of equations

## Algorithm Walkthrough

### 1. Build the Graph

We convert every equation into two directed edges.

If:

```
a / b = 2
```

then:

```
a -> b : 2
b -> a : 0.5
```

We use a hash map where:

- the key is a variable
- the value is a list of neighboring variables and edge weights

This structure allows fast lookup of connected variables.

### 2. Process Each Query Independently

For each query `[src, dst]`:

- If either variable does not exist in the graph, return `-1.0`
- If `src == dst`, return `1.0`
- Otherwise perform DFS

The reason `src / src = 1.0` is because any nonzero quantity divided by itself equals 1.

### 3. Perform DFS Traversal

We recursively search for a path from `src` to `dst`.

During traversal:

- Keep track of visited nodes to avoid cycles
- Multiply edge weights as we move through the graph
- If we reach the destination, return the accumulated product

For example:

```
a -> b = 2
b -> c = 3
```

DFS path:

```
a -> b -> c
```

Accumulated value:

```
1 * 2 * 3 = 6
```

### 4. Backtrack When Necessary

If a path does not lead to the destination:

- return failure
- continue searching other neighbors

This ensures all reachable paths are explored safely.

### Why it works

The graph encodes all known division relationships. Every path between two variables represents a valid chain of multiplicative transformations.

Because multiplication of ratios is associative:

```
(a / b) * (b / c) = a / c
```

the product of edge weights along any valid path gives the correct answer. DFS guarantees that if such a path exists, it will eventually be found.

## Python Solution

```
from typing import List
from collections import defaultdict

class Solution:
    def calcEquation(
        self,
        equations: List[List[str]],
        values: List[float],
        queries: List[List[str]]
    ) -> List[float]:

        graph = defaultdict(list)

        # Build graph
        for (a, b), value in zip(equations, values):
            graph[a].append((b, value))
            graph[b].append((a, 1 / value))

        def dfs(current, target, product, visited):
            if current == target:
                return product

            visited.add(current)

            for neighbor, weight in graph[current]:
                if neighbor not in visited:
                    result = dfs(
                        neighbor,
                        target,
                        product * weight,
                        visited
                    )

                    if result != -1.0:
                        return result

            return -1.0

        results = []

        for src, dst in queries:

            if src not in graph or dst not in graph:
                results.append(-1.0)

            elif src == dst:
                results.append(1.0)

            else:
                results.append(
                    dfs(src, dst, 1.0, set())
                )

        return results
```

The implementation begins by constructing the adjacency list representation of the graph.

We use `defaultdict(list)` because each variable may connect to multiple neighbors. Every equation generates two edges:

- forward edge with the given ratio
- reverse edge with the reciprocal ratio

The DFS function performs recursive graph traversal.

Its parameters are:

- `current`, the node currently being explored
- `target`, the destination node
- `product`, the accumulated multiplication result
- `visited`, a set used to avoid revisiting nodes

Whenever DFS moves across an edge, the edge weight is multiplied into the running product.

If the destination is reached, the accumulated value is returned immediately.

If no valid path exists, the function returns `-1.0`.

Finally, we process every query independently and collect the results.

## Go Solution

```
package main

func calcEquation(equations [][]string, values []float64, queries [][]string) []float64 {

	graph := make(map[string][]Edge)

	// Build graph
	for i, eq := range equations {
		a := eq[0]
		b := eq[1]
		value := values[i]

		graph[a] = append(graph[a], Edge{b, value})
		graph[b] = append(graph[b], Edge{a, 1.0 / value})
	}

	var dfs func(string, string, float64, map[string]bool) float64

	dfs = func(current string, target string, product float64, visited map[string]bool) float64 {

		if current == target {
			return product
		}

		visited[current] = true

		for _, edge := range graph[current] {
			if !visited[edge.neighbor] {

				result := dfs(
					edge.neighbor,
					target,
					product*edge.weight,
					visited,
				)

				if result != -1.0 {
					return result
				}
			}
		}

		return -1.0
	}

	results := []float64{}

	for _, query := range queries {

		src := query[0]
		dst := query[1]

		_, srcExists := graph[src]
		_, dstExists := graph[dst]

		if !srcExists || !dstExists {
			results = append(results, -1.0)
		} else if src == dst {
			results = append(results, 1.0)
		} else {
			results = append(
				results,
				dfs(src, dst, 1.0, make(map[string]bool)),
			)
		}
	}

	return results
}

type Edge struct {
	neighbor string
	weight   float64
}
```

The Go implementation follows the same overall structure as the Python version.

One notable difference is that Go requires an explicit `Edge` struct to store neighbor-weight pairs.

The DFS function is declared as a function variable because recursive anonymous functions in Go require prior declaration.

Go maps return two values during lookup:

```
value, exists := map[key]
```

This is used to check whether variables exist in the graph.

Slices are used for adjacency lists because they provide efficient append operations and dynamic sizing.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(Q * (V + E)) | Each query may traverse the entire graph |
| Space | O(V + E) | Graph storage plus DFS visited set |

The graph construction step takes `O(E)` time because each equation creates two edges.

For each query, DFS may visit every node and edge once in the worst case. Therefore each query costs `O(V + E)`.

Since there are `Q` queries, total complexity becomes:

```
O(Q * (V + E))
```

The adjacency list stores every directed edge, requiring `O(V + E)` space.

The recursion stack and visited set together also require at most `O(V)` additional memory.

## Edge Cases

### Querying Undefined Variables

A very common bug occurs when queries contain variables that never appeared in any equation.

For example:

```
queries = [["x", "y"]]
```

If neither variable exists in the graph, there is no mathematical relationship available.

The implementation explicitly checks graph membership before running DFS:

```
if src not in graph or dst not in graph:
```

This prevents invalid traversal attempts and correctly returns `-1.0`.

### Self Division Queries

Queries such as:

```
a / a
```

can be tricky.

Mathematically, any nonzero value divided by itself equals `1.0`.

However, the variable must actually exist in the known equations. If `a` is undefined, the result should still be `-1.0`.

The implementation handles this carefully by first checking existence in the graph before returning `1.0`.

### Disconnected Components

The graph may contain multiple independent groups of variables.

Example:

```
a / b = 2
x / y = 3
```

There is no relationship between `a` and `y`.

A naive implementation might accidentally traverse indefinitely or assume connectivity.

The DFS solution uses a visited set and naturally fails when no path exists, returning `-1.0`.

### Cycles in the Graph

Even though the input guarantees consistency, cycles may still exist:

```
a -> b
b -> c
c -> a
```

Without a visited set, DFS could recurse forever.

The implementation prevents revisiting nodes by storing explored variables in `visited`, ensuring safe traversal even in cyclic graphs.
