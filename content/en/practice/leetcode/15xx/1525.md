## Solution

## Problem Understanding

The problem asks us to determine how many ways we can split a given string `s` into two non-empty parts such that the number of **distinct letters** in the left substring (`sleft`) is equal to the number of distinct letters in the right substring (`sright`). The input is a single string `s` consisting of lowercase English letters, and the output is an integer representing the number of "good" splits that satisfy the distinct letter condition.

A "good split" requires splitting the string at positions from `1` to `len(s)-1` because both resulting substrings must be non-empty. The constraints indicate that the length of `s` can go up to `10^5`, which is large enough to make a brute-force solution-one that counts distinct letters for each possible split-inefficient. The input guarantees only lowercase letters, so we can leverage this fixed alphabet size (26 letters) to optimize counting distinct characters efficiently.

Important edge cases include strings where all characters are identical, strings with all unique characters, very short strings, and strings where good splits occur only at specific positions.

## Approaches

### Brute Force Approach

A naive approach is to iterate through every possible split of the string, and for each split, count the number of distinct characters in the left and right substrings using a set or hash map. If the counts are equal, increment a counter. This approach is straightforward and guaranteed to be correct, but it is computationally expensive because each split requires iterating over parts of the string, leading to a worst-case time complexity of $O(n^2)$, which is too slow for $n \leq 10^5$.

### Optimal Approach

The key observation is that we can track the number of distinct letters in both the left and right substrings incrementally. First, traverse the string once from left to right, maintaining a running count of distinct letters in `sleft` at each split point. Then, traverse from right to left to maintain the number of distinct letters in `sright` for each split point. By storing these counts in arrays, we can check each possible split in constant time, resulting in an overall linear time complexity, $O(n)$, and space complexity $O(n)$.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^2) | O(n) | Counts distinct letters for each split separately, slow for large strings |
| Optimal | O(n) | O(n) | Uses prefix and suffix distinct count arrays to check splits in linear time |

## Algorithm Walkthrough

1. Initialize an array `left_count` to store the number of distinct letters from the start up to each position. Use a set to track letters seen so far as you traverse the string from left to right.
2. Initialize another array `right_count` to store the number of distinct letters from the end up to each position. Use a separate set to track letters seen as you traverse the string from right to left.
3. Iterate through the string indices from `0` to `len(s)-2` (valid split positions). For each index `i`, compare `left_count[i]` with `right_count[i+1]`. If they are equal, increment the good split counter.
4. Return the total count of good splits.

Why it works: By precomputing the number of distinct letters for all prefixes and suffixes, we avoid redundant counting. The invariant is that `left_count[i]` always accurately reflects the number of distinct characters in `s[0:i+1]` and `right_count[i+1]` in `s[i+1:]`, ensuring correctness.

## Python Solution

```python
class Solution:
    def numSplits(self, s: str) -> int:
        n = len(s)
        left_count = [0] * n
        right_count = [0] * n
        
        seen_left = set()
        for i in range(n):
            seen_left.add(s[i])
            left_count[i] = len(seen_left)
        
        seen_right = set()
        for i in range(n - 1, -1, -1):
            seen_right.add(s[i])
            right_count[i] = len(seen_right)
        
        good_splits = 0
        for i in range(n - 1):
            if left_count[i] == right_count[i + 1]:
                good_splits += 1
        
        return good_splits
```

The Python solution first computes prefix and suffix distinct letter counts using sets. Then it iterates through possible split positions to check equality. Each logical section is directly aligned with the algorithm steps above.

## Go Solution

```go
func numSplits(s string) int {
    n := len(s)
    leftCount := make([]int, n)
    rightCount := make([]int, n)
    
    seenLeft := make(map[byte]struct{})
    for i := 0; i < n; i++ {
        seenLeft[s[i]] = struct{}{}
        leftCount[i] = len(seenLeft)
    }
    
    seenRight := make(map[byte]struct{})
    for i := n - 1; i >= 0; i-- {
        seenRight[s[i]] = struct{}{}
        rightCount[i] = len(seenRight)
    }
    
    goodSplits := 0
    for i := 0; i < n-1; i++ {
        if leftCount[i] == rightCount[i+1] {
            goodSplits++
        }
    }
    
    return goodSplits
}
```

The Go implementation mirrors Python's logic. Maps are used instead of sets, and slices replace arrays. Empty struct values save memory while tracking distinct characters.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each character is visited twice: once for prefix counts, once for suffix counts, plus a linear check over splits |
| Space | O(n) | Arrays for prefix and suffix counts and temporary sets/maps for distinct letters |

The complexity is efficient because the size of the alphabet is constant, and all operations scale linearly with the string length.

## Edge Cases

A key edge case is a string with a single character. There are no possible splits, and the solution correctly returns 0. Another edge case is a string with all identical characters, where any split will not balance distinct letters; the algorithm handles this because the distinct count never increases after the first character. Finally, strings where all characters are unique create a single valid split in the middle, which is correctly captured by comparing prefix and suffix counts. All these edge cases are handled gracefully by precomputing counts and checking each split in constant time.
