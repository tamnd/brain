## Solution

## Problem Understanding

The problem gives us an integer array `arr` along with two integers, `m` and `k`. We need to determine whether there exists a contiguous subarray of length `m` that repeats consecutively at least `k` times.

A pattern is considered valid only if the repetitions occur one immediately after another, without gaps or overlaps. For example, if `m = 2` and the pattern is `[1,2]`, then the sequence:

```
[1,2,1,2,1,2]
```

contains the pattern repeated 3 times consecutively.

The task is to return `true` if at least one such pattern exists anywhere in the array. Otherwise, we return `false`.

The input array contains only positive integers. The constraints are relatively small:

- `arr.length <= 100`
- `m <= 100`
- `k <= 100`

Because the array is very small, even moderately inefficient solutions are acceptable. However, we still want to design a clean and optimal solution.

One important observation is that the repeated segments must be exactly identical. If even one value differs, the repetition breaks immediately.

Another important detail is that the repetition must be consecutive. A pattern appearing multiple times in different locations does not count unless those appearances are directly adjacent.

There are several edge cases worth thinking about before implementation:

- The array may be too short to contain `k` repetitions of length `m`. In that case, the answer is automatically `false`.
- The pattern may overlap incorrectly in a naive implementation. We must compare corresponding positions carefully.
- The repeated sequence might appear at the very end of the array, so boundary calculations must be correct.
- `m` may equal `1`, meaning we are simply checking whether a value appears consecutively at least `k` times.

## Approaches

### Brute Force Approach

The most direct solution is to try every possible starting index in the array and check whether the next `k` blocks of size `m` are identical.

Suppose we start at index `i`. The first block is:

```
arr[i : i + m]
```

We then compare it with:

```
arr[i + m : i + 2m]
arr[i + 2m : i + 3m]
...
```

If all `k` blocks match exactly, we return `true`.

This works because it explicitly checks the exact definition of the problem. However, it repeatedly compares many overlapping sections, which can become inefficient in larger inputs.

Even though the constraints are small enough for this method to pass comfortably, we can simplify the logic further.

### Optimal Approach

The key observation is that if a pattern of length `m` repeats consecutively, then:

```
arr[i] == arr[i + m]
```

for every position participating in the repetition.

For example:

```
[1,2,1,2,1,2]
```

with `m = 2` gives:

```
arr[0] == arr[2]
arr[1] == arr[3]
arr[2] == arr[4]
arr[3] == arr[5]
```

This means we only need to count how many consecutive positions satisfy:

```
arr[i] == arr[i + m]
```

If we find `m * (k - 1)` consecutive matching comparisons, then we have `k` repeated blocks.

This transforms the problem into a simple linear scan.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * m * k) | O(1) | Explicitly compares every repeated block |
| Optimal | O(n) | O(1) | Counts consecutive matches offset by `m` |

## Algorithm Walkthrough

### Optimal Linear Scan Algorithm

1. Initialize a counter called `matched` to `0`.

This counter tracks how many consecutive positions satisfy:

```
arr[i] == arr[i + m]
```
2. Iterate through the array from index `0` to `len(arr) - m - 1`.

We stop here because `i + m` must remain within bounds.
3. For each index `i`, compare:

```
arr[i] and arr[i + m]
```
4. If the values are equal, increment `matched`.

This means the current position continues a repeating structure.
5. If the values are different, reset `matched` to `0`.

A mismatch breaks the consecutive repetition sequence.
6. After updating `matched`, check whether:

```
matched == m * (k - 1)
```

Why this formula works:

- Repeating a block `k` times requires `k - 1` transitions between blocks.
- Each transition contributes `m` matching positions.
- Therefore, we need exactly `m * (k - 1)` consecutive matches.
7. If the condition is satisfied, return `true`.
8. If the loop finishes without success, return `false`.

### Why it works

A repeated pattern of length `m` means every element in one block matches the corresponding element exactly `m` positions ahead. By counting consecutive successful comparisons between `arr[i]` and `arr[i + m]`, we effectively verify that multiple adjacent blocks are identical. Reaching `m * (k - 1)` consecutive matches guarantees the existence of `k` consecutive copies of the same pattern.

## Python Solution

```python
from typing import List

class Solution:
    def containsPattern(self, arr: List[int], m: int, k: int) -> bool:
        matched = 0

        for i in range(len(arr) - m):
            if arr[i] == arr[i + m]:
                matched += 1

                if matched == m * (k - 1):
                    return True
            else:
                matched = 0

        return False
```

The implementation follows the linear scan approach directly.

The variable `matched` stores the number of consecutive successful comparisons between elements separated by distance `m`.

The loop iterates only while `i + m` remains valid. For each position, we compare `arr[i]` with `arr[i + m]`.

If the values match, we extend the current repetition streak by incrementing `matched`. If they differ, the repeating structure is broken, so we reset `matched` back to `0`.

The moment `matched` reaches `m * (k - 1)`, we know that enough consecutive aligned matches exist to form `k` repeated blocks of length `m`, so we return `True`.

If the scan completes without reaching the required count, the function returns `False`.

## Go Solution

```go
func containsPattern(arr []int, m int, k int) bool {
    matched := 0

    for i := 0; i < len(arr)-m; i++ {
        if arr[i] == arr[i+m] {
            matched++

            if matched == m*(k-1) {
                return true
            }
        } else {
            matched = 0
        }
    }

    return false
}
```

The Go implementation is almost identical to the Python version. The logic remains the same because the algorithm only requires integer comparisons and a counter.

Go slices already provide efficient indexed access, so no additional data structures are necessary. Integer overflow is not a concern because all constraints are extremely small.

## Edge Cases

One important edge case occurs when the array is too short to contain the required repetitions. For example:

```
arr = [1,2,3]
m = 2
k = 2
```

A pattern of length `2` repeated twice would require at least `4` elements. The implementation naturally handles this because the loop never accumulates enough matches to satisfy the required threshold.

Another tricky case is when repetitions are interrupted midway. Consider:

```
[1,1,1,2,1,1]
```

with `m = 1` and `k = 3`.

A naive implementation might incorrectly count total occurrences of `1`, but the problem requires consecutive repetition. The algorithm correctly resets `matched` to `0` after encountering the `2`, preventing a false positive.

A third edge case involves patterns located at the very end of the array. For example:

```
[9,8,7,6,6,6]
```

with `m = 1` and `k = 3`.

Boundary handling can easily produce off by one errors in these situations. The implementation avoids this by iterating only while `i + m` remains within bounds, ensuring all comparisons are valid while still checking the final possible repetition window correctly.
