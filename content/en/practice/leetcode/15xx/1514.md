# LeetCode 1514, Path with Maximum Probability

## Problem Understanding

This problem gives us an undirected weighted graph where each edge represents a probability of successfully traveling between two nodes. The graph contains `n` nodes labeled from `0` to `n - 1`.

The input is split into several parts:

- `edges[i] = [a, b]` means there is an undirected edge between node `a` and node `b`
- `succProb[i]` gives the probability of successfully traversing that edge
- `start_node` is the starting node
- `end_node` is the destination node

The goal is to find a path from `start_node` to `end_node` such that the product of probabilities along the path is maximized.

For example, suppose we have the path:

```
0 -> 1 -> 2
```

with probabilities:

```
0.5 and 0.5
```

Then the probability of successfully completing the entire path is:

```
0.5 × 0.5 = 0.25
```

The problem asks us to return the highest possible probability among all valid paths between the two nodes.

If no path exists, we return `0`.

The constraints are important:

- Up to `10^4` nodes
- Up to `2 * 10^4` edges

These limits are too large for exhaustive path exploration. A brute-force search through all possible paths would become exponentially expensive in dense graphs.

Several edge cases are important:

- The destination may be unreachable
- Some edges may have probability `0`
- The graph is undirected, so every edge works both ways
- There is at most one edge between any pair of nodes
- The optimal path may contain more edges if their combined probability is larger

The key challenge is that we are maximizing a product of probabilities, not minimizing a sum of distances.

## Approaches

### Brute Force Approach

A straightforward solution would be to explore every possible path from `start_node` to `end_node` using DFS or backtracking.

For each path:

1. Multiply the probabilities along the edges
2. Track the maximum product seen so far
3. Continue until all paths have been explored

This approach is correct because it checks every possible path, guaranteeing that the maximum probability is found.

However, the number of paths in a graph can grow exponentially. In the worst case, the algorithm becomes far too slow for the given constraints.

### Key Insight

This problem is structurally very similar to the shortest path problem.

In classic Dijkstra's algorithm:

- We minimize total distance
- We repeatedly expand the node with the smallest known distance

Here:

- We maximize probability
- We repeatedly expand the node with the largest known probability

Instead of storing minimum distances, we store the best probability discovered so far for each node.

The important observation is:

If we have already reached a node with probability `p`, then any future path reaching that node with probability smaller than `p` cannot possibly lead to a better answer.

This greedy property allows us to use a priority queue exactly like Dijkstra's algorithm, except we use a max heap instead of a min heap.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^V) | O(V) | Explores all possible paths |
| Optimal | O((V + E) log V) | O(V + E) | Modified Dijkstra using max heap |

## Algorithm Walkthrough

### Step 1, Build the Graph

We first convert the edge list into an adjacency list.

Each node stores:

- Its neighboring node
- The probability associated with the edge

For an undirected graph:

```
a <-> b
```

we add both:

```
a -> b
b -> a
```

This structure allows efficient traversal of neighboring nodes.

### Step 2, Initialize Best Probabilities

We maintain an array:

```
best_probability[node]
```

which stores the highest probability found so far for reaching each node.

Initially:

- All values are `0`
- The start node has probability `1.0`

because reaching the starting node requires no traversal.

### Step 3, Use a Max Heap

We process nodes in descending order of probability.

A max heap always gives us the node currently reachable with the highest probability.

The heap stores:

```
(current_probability, node)
```

Python's `heapq` is a min heap, so we store negative probabilities to simulate max heap behavior.

### Step 4, Process the Most Promising Node

Repeatedly:

1. Pop the node with the highest probability
2. If it is the destination, return immediately
3. Explore all neighbors

For each neighbor:

```
new_probability = current_probability × edge_probability
```

If this new probability is better than the previously known probability:

- Update the best value
- Push the new state into the heap

### Step 5, Handle Unreachable Destinations

If the heap becomes empty before reaching the destination, then no path exists.

Return `0`.

## Why it works

The algorithm relies on the same greedy property as Dijkstra's algorithm.

When a node is removed from the max heap, we already know the highest possible probability for reaching it. Any alternative path discovered later would necessarily have a smaller probability because probabilities are multiplied by values between `0` and `1`.

Therefore, once the destination node is popped from the heap, its probability is guaranteed to be optimal.

## Python Solution

```python
from typing import List
import heapq
from collections import defaultdict

class Solution:
    def maxProbability(
        self,
        n: int,
        edges: List[List[int]],
        succProb: List[float],
        start_node: int,
        end_node: int
    ) -> float:

        graph = defaultdict(list)

        for (u, v), probability in zip(edges, succProb):
            graph[u].append((v, probability))
            graph[v].append((u, probability))

        best_probability = [0.0] * n
        best_probability[start_node] = 1.0

        max_heap = [(-1.0, start_node)]

        while max_heap:
            current_probability, node = heapq.heappop(max_heap)

            current_probability = -current_probability

            if node == end_node:
                return current_probability

            if current_probability < best_probability[node]:
                continue

            for neighbor, edge_probability in graph[node]:
                new_probability = current_probability * edge_probability

                if new_probability > best_probability[neighbor]:
                    best_probability[neighbor] = new_probability
                    heapq.heappush(
                        max_heap,
                        (-new_probability, neighbor)
                    )

        return 0.0
```

The implementation begins by constructing an adjacency list representation of the graph. Since the graph is undirected, every edge is added in both directions.

The `best_probability` array stores the best probability found so far for each node. Initially, every node has probability `0`, except the starting node which has probability `1`.

The heap stores pairs of `(probability, node)`. Because Python's `heapq` implements a min heap, probabilities are negated so that larger probabilities are processed first.

Inside the main loop, the algorithm repeatedly extracts the node with the current highest probability. If the destination node is reached, the answer is returned immediately because the greedy property guarantees optimality.

For each neighboring node, the algorithm computes the probability of reaching that neighbor through the current path. If this probability improves upon the previously recorded value, the neighbor is updated and inserted into the heap.

If traversal completes without reaching the destination, the graph contains no valid path, so the algorithm returns `0`.

## Go Solution

```go
package main

import (
	"container/heap"
)

type State struct {
	probability float64
	node        int
}

type MaxHeap []State

func (h MaxHeap) Len() int {
	return len(h)
}

func (h MaxHeap) Less(i, j int) bool {
	return h[i].probability > h[j].probability
}

func (h MaxHeap) Swap(i, j int) {
	h[i], h[j] = h[j], h[i]
}

func (h *MaxHeap) Push(x interface{}) {
	*h = append(*h, x.(State))
}

func (h *MaxHeap) Pop() interface{} {
	old := *h
	n := len(old)

	item := old[n-1]
	*h = old[:n-1]

	return item
}

func maxProbability(
	n int,
	edges [][]int,
	succProb []float64,
	start_node int,
	end_node int,
) float64 {

	graph := make([][][2]float64, n)

	for i, edge := range edges {
		u := edge[0]
		v := edge[1]
		prob := succProb[i]

		graph[u] = append(graph[u], [2]float64{float64(v), prob})
		graph[v] = append(graph[v], [2]float64{float64(u), prob})
	}

	bestProbability := make([]float64, n)
	bestProbability[start_node] = 1.0

	maxHeap := &MaxHeap{}
	heap.Init(maxHeap)

	heap.Push(maxHeap, State{
		probability: 1.0,
		node:        start_node,
	})

	for maxHeap.Len() > 0 {
		current := heap.Pop(maxHeap).(State)

		currentProbability := current.probability
		node := current.node

		if node == end_node {
			return currentProbability
		}

		if currentProbability < bestProbability[node] {
			continue
		}

		for _, neighborInfo := range graph[node] {
			neighbor := int(neighborInfo[0])
			edgeProbability := neighborInfo[1]

			newProbability := currentProbability * edgeProbability

			if newProbability > bestProbability[neighbor] {
				bestProbability[neighbor] = newProbability

				heap.Push(maxHeap, State{
					probability: newProbability,
					node:        neighbor,
				})
			}
		}
	}

	return 0.0
}
```

The Go implementation follows the same algorithmic structure as the Python solution but requires explicit heap implementation through the `container/heap` package.

A custom `MaxHeap` type is defined because Go's heap package provides only the heap framework, not a built-in priority queue.

Unlike Python, Go does not require negating probabilities because the custom `Less` method directly defines max heap behavior.

The adjacency list uses slices, and neighbor information is stored as fixed-size arrays containing the destination node and edge probability.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O((V + E) log V) | Each node and edge may be processed through the heap |
| Space | O(V + E) | Adjacency list, heap, and probability array |

The adjacency list requires `O(E)` space because every edge is stored twice in an undirected graph.

The heap may contain up to `O(E)` entries in the worst case due to repeated updates. Each insertion and removal operation costs `O(log V)`.

Overall, this matches the standard complexity of Dijkstra's algorithm.

## Edge Cases

### Unreachable Destination

One important edge case occurs when the destination node cannot be reached from the start node.

A naive implementation might accidentally return an uninitialized value or crash while searching for a path that does not exist.

This implementation handles the case naturally. If no valid path exists, the heap eventually becomes empty without reaching the destination node, and the algorithm returns `0.0`.

### Paths with More Edges Can Be Better

Many shortest path problems favor paths with fewer edges, but this problem does not.

For example:

```
Direct path: 0.2
Two-edge path: 0.5 × 0.5 = 0.25
```

The longer path is better.

A greedy strategy based only on edge count would fail. The implementation correctly evaluates complete path probabilities by multiplying edge probabilities at every step.

### Zero Probability Edges

An edge may have probability `0`.

Such an edge effectively blocks successful traversal through that route.

Some implementations may incorrectly divide probabilities or mishandle floating-point values. This solution only performs multiplication, so zero-valued probabilities naturally propagate through the calculation and produce the correct result.

### Multiple Heap Entries for the Same Node

A node may appear in the heap multiple times with different probabilities.

For example:

```
Node 2 inserted with 0.2
Later improved to 0.25
```

The earlier heap entry becomes stale.

The implementation safely handles this using:

```
if current_probability < best_probability[node]:
    continue
```

This ensures outdated states are skipped efficiently without affecting correctness.
