## Solution

## Problem Understanding

The problem gives us an integer array `arr` and asks whether the array contains three odd numbers that appear consecutively. Consecutive means the numbers must appear next to each other in the array, without any gaps.

An odd number is any integer that is not divisible by 2. In programming terms, a number is odd if `number % 2 == 1`.

The task is therefore straightforward:

- Scan through the array
- Check whether any sequence of three adjacent elements are all odd
- Return `true` if such a sequence exists
- Otherwise return `false`

For example, in the array `[1,2,34,3,4,5,7,23,12]`, the subarray `[5,7,23]` contains three odd numbers in a row, so the answer is `true`.

The constraints are small:

- `1 <= arr.length <= 1000`
- `1 <= arr[i] <= 1000`

These constraints tell us that performance is not a major concern. Even a simple solution that checks every possible group of three numbers will run comfortably within limits. Still, we should aim for a clean and efficient linear-time solution.

There are several important edge cases to consider:

- Arrays shorter than 3 elements cannot possibly contain three consecutive odd numbers.
- Arrays containing only even numbers should return `false`.
- Arrays containing exactly three odd numbers should return `true`.
- The sequence can appear anywhere in the array, including at the beginning or end.
- An even number should reset the consecutive odd count.

Because the problem guarantees all values are positive integers, we do not need to worry about negative modulo behavior.

## Approaches

### Brute Force Approach

The brute-force solution checks every possible group of three consecutive elements in the array.

For each index `i`, we examine:

- `arr[i]`
- `arr[i + 1]`
- `arr[i + 2]`

If all three numbers are odd, we immediately return `true`.

If we finish checking all valid triplets and never find three consecutive odds, we return `false`.

This approach is correct because every possible consecutive group of length 3 is examined exactly once. If a valid sequence exists, the algorithm will eventually inspect it.

Although this method is already efficient enough for the constraints, it repeatedly performs parity checks on overlapping elements. For example, an element may be checked multiple times as part of different triplets.

### Optimal Approach

The optimal solution uses a running counter to track how many consecutive odd numbers have been seen so far.

We iterate through the array once:

- If the current number is odd, increment the counter.
- If the current number is even, reset the counter to 0.
- Whenever the counter reaches 3, return `true`.

This works because we only care about consecutive odd values. An even number breaks the sequence, so resetting the counter is sufficient.

This approach is cleaner, requires only one pass through the array, and avoids repeatedly checking overlapping windows.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Checks every consecutive triplet directly |
| Optimal | O(n) | O(1) | Tracks consecutive odd count in one pass |

## Algorithm Walkthrough

1. Initialize a variable `consecutive_odds` to `0`.

This variable keeps track of how many odd numbers have appeared consecutively up to the current position.
2. Traverse the array from left to right.

We examine each element exactly once.
3. For each number, check whether it is odd.

A number is odd if `number % 2 == 1`.
4. If the number is odd, increment `consecutive_odds`.

This extends the current streak of consecutive odd numbers.
5. If the number is even, reset `consecutive_odds` to `0`.

An even number breaks the consecutive sequence, so we must start counting again from scratch.
6. After updating the counter, check whether `consecutive_odds == 3`.

If this condition becomes true, we immediately return `True` because we found three consecutive odd numbers.
7. If the loop finishes without reaching 3 consecutive odds, return `False`.

### Why it works

The algorithm maintains the invariant that `consecutive_odds` always equals the number of consecutive odd numbers ending at the current index.

Whenever an even number appears, the streak is broken and the counter resets to zero. Therefore, if the counter ever reaches 3, we know there are exactly three consecutive odd numbers somewhere in the array. Since every element is processed once in order, no valid sequence can be missed.

## Python Solution

```python
from typing import List

class Solution:
    def threeConsecutiveOdds(self, arr: List[int]) -> bool:
        consecutive_odds = 0

        for number in arr:
            if number % 2 == 1:
                consecutive_odds += 1
            else:
                consecutive_odds = 0

            if consecutive_odds == 3:
                return True

        return False
```

The implementation starts by initializing `consecutive_odds` to zero. This variable tracks the current streak of odd numbers.

The loop iterates through every number in the array. For each value, the code checks whether it is odd using the modulo operation.

If the number is odd, the streak increases by one. If the number is even, the streak resets because consecutive odds must be uninterrupted.

After each update, the code checks whether the streak has reached three. As soon as this happens, the function returns `True` immediately because the condition has been satisfied.

If the loop completes without finding a valid streak, the function returns `False`.

## Go Solution

```go
func threeConsecutiveOdds(arr []int) bool {
    consecutiveOdds := 0

    for _, number := range arr {
        if number%2 == 1 {
            consecutiveOdds++
        } else {
            consecutiveOdds = 0
        }

        if consecutiveOdds == 3 {
            return true
        }
    }

    return false
}
```

The Go implementation follows the same logic as the Python version.

The `range` loop iterates through the slice, and the modulo operator checks whether each number is odd. The consecutive count is incremented or reset exactly as in Python.

There are no special concerns about integer overflow because the values are very small. Go slices naturally handle arrays of varying lengths, and no additional memory allocation is required.

## Edge Cases

One important edge case is an array with fewer than three elements. Since the problem requires three consecutive odd numbers, such arrays can never satisfy the condition. A careless implementation might accidentally access indices outside the array bounds. This implementation avoids that problem because it simply iterates through the array once without relying on fixed index windows.

Another important case is when an even number interrupts a streak of odd numbers. For example, `[1,3,2,5,7]` contains two odd numbers, then an even number, then two more odd numbers. A buggy implementation might incorrectly accumulate all odd numbers together instead of resetting after the even number. The counter reset ensures each streak is tracked independently.

A third edge case occurs when there are more than three consecutive odd numbers, such as `[1,3,5,7,9]`. The algorithm correctly returns `True` as soon as the counter reaches three. It does not matter whether the streak continues beyond that point because the condition has already been satisfied.
