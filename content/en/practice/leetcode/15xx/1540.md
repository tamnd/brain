# LeetCode 1540 - Can Convert String in K Moves

## Problem Understanding

The problem gives us two lowercase English strings, `s` and `t`, along with an integer `k`. We need to determine whether it is possible to transform `s` into `t` using at most `k` moves under a very specific set of rules.

On the `i-th` move, where `1 <= i <= k`, we may either do nothing or choose one index in the string that has not been used before and shift that character exactly `i` times forward in the alphabet. The alphabet wraps around, so shifting `'z'` once becomes `'a'`.

The important detail is that the number of shifts applied during a move is fixed by the move number itself. During move `1`, we can only apply exactly one shift. During move `5`, we can only apply exactly five shifts. We cannot choose an arbitrary number of shifts during a move.

The task is to determine whether every character in `s` can be converted into the corresponding character in `t` while respecting these movement constraints.

For example, converting `'a'` to `'d'` requires a shift of `3`. That means we must schedule that character on a move whose number is congruent to `3 mod 26`, because shifting by `29` or `55` also effectively shifts by `3` due to alphabet wraparound.

The constraints are large:

- String lengths can be up to `10^5`
- `k` can be as large as `10^9`

These limits immediately tell us that we need an efficient linear-time solution. Any brute-force simulation of moves would be far too slow.

There are several important edge cases to notice early:

- If `s` and `t` have different lengths, conversion is impossible immediately.
- Characters that are already equal require no move at all.
- Multiple characters may require the same shift amount. Since each move number can only be used once, these shifts must be spaced apart by multiples of `26`.
- Very large `k` values are possible, so arithmetic reasoning is more important than simulation.

## Approaches

### Brute Force Approach

A naive solution would attempt to simulate the process move by move.

For each move from `1` to `k`, we could try to find a character that still needs a shift matching the current move number modulo `26`. Once assigned, that character would be marked as used.

This approach is conceptually correct because every move corresponds to a specific effective alphabet shift. However, it becomes impractical when `k` is very large. Since `k` can reach `10^9`, iterating through every move is impossible within time limits.

Additionally, repeatedly searching for matching characters would make the implementation unnecessarily expensive.

### Optimal Approach

The key observation is that we do not need to simulate moves individually.

For each position:

- Compute how many shifts are needed to transform `s[i]` into `t[i]`
- This shift value is between `0` and `25`

If the shift is `0`, no move is needed.

If multiple characters require the same shift amount, they cannot all use the same move number. For example:

- The first character needing shift `3` can use move `3`
- The second must use move `29`
- The third must use move `55`

In general, if a shift value `d` appears `count` times, the required move numbers are:

```
d, d + 26, d + 52, ...
```

Therefore, for the `count-th` occurrence of shift `d`, the required move is:

```
d + 26 * (count - 1)
```

If this required move exceeds `k`, conversion is impossible.

This transforms the problem into simple counting and arithmetic, giving us a linear-time solution.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(k + n) or worse | O(n) | Simulates moves directly, infeasible for very large `k` |
| Optimal | O(n) | O(1) | Counts required shifts and validates scheduling mathematically |

## Algorithm Walkthrough

1. First, check whether the lengths of `s` and `t` are equal. If they differ, return `False` immediately because characters can only be shifted, not inserted or removed.
2. Create a frequency structure to count how many times each shift amount has already appeared. Since shifts range only from `0` to `25`, an array of size `26` is sufficient.
3. Iterate through both strings character by character.
4. For each position, compute the required shift:

```
shift = (ord(t[i]) - ord(s[i])) % 26
```

The modulo handles wraparound correctly. For example:

- `'a' -> 'c'` gives `2`
- `'z' -> 'b'` also gives `2`
5. If `shift == 0`, no move is needed, so continue to the next character.
6. Otherwise, determine which move number this occurrence would require.

Suppose this is the third character needing shift `5`.

The required moves for shift `5` are:

```
5, 31, 57, ...
```

Since two previous occurrences already used `5` and `31`, this one needs:

```
5 + 26 * 2 = 57
```
7. Compute the required move:

```
required_move = shift + 26 * shift_count[shift]
```
8. If `required_move > k`, return `False`.
9. Otherwise, increment the frequency count for this shift and continue.
10. If all characters can be scheduled successfully, return `True`.

### Why it works

The algorithm works because every move number corresponds to exactly one effective shift modulo `26`.

Two characters requiring the same shift cannot share a move number, so their moves must differ by multiples of `26`. By assigning shifts greedily in increasing order:

```
d, d + 26, d + 52, ...
```

we always use the earliest available valid move. If even the earliest valid move exceeds `k`, then no alternative schedule could work either. Therefore, the algorithm correctly determines feasibility.

## Python Solution

```python
class Solution:
    def canConvertString(self, s: str, t: str, k: int) -> bool:
        if len(s) != len(t):
            return False

        shift_count = [0] * 26

        for i in range(len(s)):
            shift = (ord(t[i]) - ord(s[i])) % 26

            if shift == 0:
                continue

            required_move = shift + 26 * shift_count[shift]

            if required_move > k:
                return False

            shift_count[shift] += 1

        return True
```

The implementation begins with the length check because unequal strings can never be transformed into one another using only shifts.

The `shift_count` array tracks how many times each shift amount has already been used. Since there are only `26` possible shifts, this structure uses constant space.

For every character pair, we compute the cyclic shift distance using modulo arithmetic. This handles alphabet wraparound naturally.

If a character already matches its target, we skip it because no move is required.

Otherwise, we compute the earliest valid move number for this occurrence of the shift. The formula:

```python
shift + 26 * shift_count[shift]
```

ensures that repeated uses of the same shift are spaced by `26`.

If the required move exceeds `k`, we immediately return `False`. Otherwise, we reserve this shift occurrence by incrementing its counter.

If the loop completes successfully, every character can be scheduled within the move limit.

## Go Solution

```go
func canConvertString(s string, t string, k int) bool {
    if len(s) != len(t) {
        return false
    }

    shiftCount := make([]int, 26)

    for i := 0; i < len(s); i++ {
        shift := (int(t[i]) - int(s[i]) + 26) % 26

        if shift == 0 {
            continue
        }

        requiredMove := shift + 26*shiftCount[shift]

        if requiredMove > k {
            return false
        }

        shiftCount[shift]++
    }

    return true
}
```

The Go implementation follows the same logic as the Python version.

One small difference is how modulo arithmetic is handled. Since subtraction can become negative in Go, we add `26` before taking `% 26`:

```go
(int(t[i]) - int(s[i]) + 26) % 26
```

This guarantees a non-negative result.

The `shiftCount` slice serves the same purpose as the Python list and remains fixed at size `26`, giving constant extra space usage.

## Edge Cases

### Different String Lengths

If `s` and `t` have different lengths, conversion is impossible because operations only modify characters, not string size. A naive implementation might forget this check and attempt to process characters positionally, leading to index errors or incorrect results. The solution handles this immediately with an early return.

### Multiple Characters Requiring the Same Shift

This is the most important edge case in the problem. Two characters that both require shift `1` cannot both use move `1`. The second occurrence must wait until move `27`, the third until `53`, and so on. A buggy implementation might only check whether `shift <= k`, which is insufficient. The frequency-counting approach correctly spaces repeated shifts by multiples of `26`.

### Alphabet Wraparound

Converting `'z'` to `'b'` requires wrapping around the alphabet. Without modulo arithmetic, subtraction would produce a negative value or an incorrect shift count. The implementation uses:

```
(ord(t[i]) - ord(s[i])) % 26
```

which guarantees the correct cyclic distance.

### Characters Already Equal

If `s[i] == t[i]`, no move is necessary. Treating these as requiring shift `0` moves could incorrectly consume scheduling slots. The implementation explicitly skips shift `0`, ensuring unnecessary moves are never assigned.

### Exact Boundary Values

Cases where the required move is exactly equal to `k` are valid. For example, if a conversion requires move `27` and `k == 27`, the answer should be `True`. The implementation correctly checks:

```
required_move > k
```

rather than using `>=`.
