## Solution

## Problem Understanding

This problem asks us to retrieve the most recent order or orders for every product that has ever been ordered.

We are given three database tables:

- `Customers`, which stores customer information.
- `Orders`, which stores each order placed by customers.
- `Products`, which stores product information.

The key requirement is to identify, for each product, the latest `order_date`. Once we know that latest date, we must return every order for that product that occurred on that date.

This detail is important because multiple orders for the same product can happen on the same most recent day. The example demonstrates this clearly with the `keyboard` product, where two separate orders occurred on `2020-08-01`, which is the latest date for that product. Both rows must appear in the output.

The result must include:

- `product_name`
- `product_id`
- `order_id`
- `order_date`

The output ordering rules are also important:

1. Sort by `product_name` ascending.
2. If tied, sort by `product_id` ascending.
3. If still tied, sort by `order_id` ascending.

The problem also states that products that were never ordered should not appear in the result. This means we only consider products that exist in the `Orders` table.

An important guarantee is that the same customer cannot order the same product more than once on the same day. However, different customers can still order the same product on the same day, which is why multiple rows can share the same latest date.

Conceptually, the problem reduces to:

1. Find the maximum `order_date` for each `product_id`.
2. Return all orders whose `order_date` matches that maximum date for their product.

Since this is a database problem, the natural solution uses SQL aggregation and joins.

## Approaches

### Brute Force Approach

A brute force approach would examine every product individually and repeatedly scan the `Orders` table to determine its most recent order date.

For each product:

1. Scan all orders to find the latest date.
2. Scan all orders again to collect rows matching that date.

This approach is correct because it explicitly computes the latest date for each product and retrieves all matching rows. However, it is inefficient because the `Orders` table may be scanned many times.

If there are `P` products and `O` orders, repeatedly scanning the orders table leads to approximately `O(P * O)` work.

### Optimal Approach

The key insight is that the latest order date for each product can be computed once using aggregation.

We can:

1. Group the `Orders` table by `product_id`.
2. Compute `MAX(order_date)` for each product.
3. Join this result back with the `Orders` table to retrieve all orders occurring on that latest date.
4. Join with `Products` to obtain the `product_name`.

This is efficient because each table is processed only a small number of times.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(P * O) | O(P) | Repeatedly scans orders for every product |
| Optimal | O(O + P) | O(P) | Uses grouping and joins efficiently |

## Algorithm Walkthrough

1. First, compute the latest order date for every product.

We group the `Orders` table by `product_id` and calculate:

```
MAX(order_date)
```

This gives us one row per product containing its most recent order date.
2. Store this intermediate result as a derived table or common table expression.

The table conceptually looks like this:

| product_id | latest_date |
| --- | --- |
| 1 | 2020-08-01 |
| 2 | 2020-08-03 |
| 3 | 2020-08-29 |
3. Join the derived table back with the `Orders` table.

We match rows where:

- `Orders.product_id = latest.product_id`
- `Orders.order_date = latest.latest_date`

This retrieves every order occurring on the most recent date for that product.
4. Join with the `Products` table.

This allows us to include `product_name` in the final output.
5. Sort the final result.

The required ordering is:

- `product_name ASC`
- `product_id ASC`
- `order_id ASC`

### Why it works

The aggregation step guarantees that we correctly identify the maximum order date for every product. By joining this result back with the original `Orders` table on both `product_id` and `order_date`, we retrieve exactly the orders that occurred on that maximum date. Since every qualifying row satisfies the latest-date condition, and no older rows can satisfy it, the result is correct.

## Python Solution

```python
# Write your MySQL query statement below

SELECT
    p.product_name,
    o.product_id,
    o.order_id,
    o.order_date
FROM Orders o
JOIN (
    SELECT
        product_id,
        MAX(order_date) AS latest_date
    FROM Orders
    GROUP BY product_id
) latest
ON o.product_id = latest.product_id
AND o.order_date = latest.latest_date
JOIN Products p
ON o.product_id = p.product_id
ORDER BY
    p.product_name ASC,
    o.product_id ASC,
    o.order_id ASC;
```

The query begins by creating a derived table named `latest`. This table groups all orders by `product_id` and computes the most recent `order_date` for each product using `MAX(order_date)`.

Next, the query joins this derived table with the original `Orders` table. The join condition ensures that only orders whose date matches the latest date for that product are included.

After retrieving the qualifying orders, the query joins with the `Products` table so the output can include the product name.

Finally, the query applies the required sorting order.

## Edge Cases

One important edge case occurs when multiple orders share the same latest date for a product. A naive solution might use `LIMIT 1` or select only one row after finding the maximum date. That would incorrectly discard valid rows. The implementation handles this correctly by joining on both `product_id` and `order_date`, allowing all matching rows to be returned.

Another edge case involves products that were never ordered. Since such products have no entries in the `Orders` table, they should not appear in the output. The query naturally handles this because the aggregation begins from the `Orders` table itself, so only products with at least one order participate.

A third edge case occurs when orders are not inserted chronologically. The latest order cannot be determined using `order_id`, because higher IDs do not necessarily imply newer dates. The implementation correctly uses `MAX(order_date)` rather than relying on order identifiers.

A fourth subtle edge case involves ties in sorting. Multiple rows can share the same product name and product ID, particularly when several orders occurred on the same latest date. The final `ORDER BY` clause includes `order_id` to ensure deterministic and fully correct ordering.
