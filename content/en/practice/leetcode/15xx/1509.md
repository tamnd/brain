## Solution

## Problem Understanding

The problem gives us an integer array `nums`, and we are allowed to perform at most three moves. In a single move, we can select any element and replace it with any value we want. The goal is to minimize the difference between the largest and smallest elements in the final array.

In other words, after changing up to three elements arbitrarily, we want the remaining spread of the array to be as small as possible. The final answer is:

```
maximum value in array - minimum value in array
```

after performing at most three modifications.

The key observation is that changing an element to any value is extremely powerful. We are not restricted to incrementing or decrementing by one. A single move can completely eliminate an inconveniently large or small value.

The constraints are important:

- `1 <= nums.length <= 10^5`
- Values may be as small as `-10^9` or as large as `10^9`

The large input size means we need an efficient algorithm. Anything exponential or quadratic will be too slow for arrays of size `100000`.

There are also several important edge cases:

- If the array has `4` or fewer elements, we can modify every element except possibly one. That means we can always make all values equal, so the answer is `0`.
- Arrays containing negative numbers must still work correctly.
- Arrays with repeated values should not confuse the logic.
- Extremely large and small integers require careful handling, although standard integer arithmetic is sufficient in both Python and Go.

## Approaches

### Brute Force Approach

A brute force solution would attempt every possible combination of up to three elements to modify and determine the minimum achievable difference.

The reasoning behind this approach is straightforward. Since we can modify any element to any value, we could try:

- changing zero elements,
- changing one element,
- changing two elements,
- changing three elements,

and evaluate every possible resulting configuration.

However, this becomes infeasible very quickly. Even choosing which three elements to modify requires examining combinations of indices. Worse, determining the optimal replacement values adds another layer of complexity.

For an array of size `n`, the number of ways to choose up to three elements is approximately:

```
O(n^3)
```

and each configuration may require additional processing.

With `n = 100000`, this is completely impractical.

### Key Insight

The important observation is that only the smallest and largest elements matter.

Suppose the array is sorted. Since we can modify at most three elements, we can effectively "remove" up to three problematic extremes.

There are only four meaningful possibilities:

1. Modify the three largest elements
2. Modify the two largest and one smallest
3. Modify the one largest and two smallest
4. Modify the three smallest elements

After sorting, we only need to compare the remaining endpoints for each case.

For example, if we remove the three largest values, the new maximum becomes the fourth largest original value.

This reduces the problem to checking only four candidate differences.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n³) or worse | O(1) | Tries many modification combinations |
| Optimal | O(n log n) | O(1) or O(log n) depending on sorting | Sort and evaluate only four cases |

## Algorithm Walkthrough

1. Check whether the array length is less than or equal to `4`.

If the array has at most four elements, we can modify enough elements to make all numbers equal. Therefore, the minimum possible difference is `0`.
2. Sort the array.

Sorting places the smallest values at the beginning and the largest values at the end. This allows us to reason about which extremes to remove.

Suppose the sorted array is:

```
nums[0] <= nums[1] <= ... <= nums[n-1]
```
3. Evaluate the four possible ways to use three moves.

Since we can change any value arbitrarily, changing an extreme element is equivalent to removing it from consideration.

The four possibilities are:

- Remove the three largest values

Difference becomes:

```
nums[n-4] - nums[0]
```
- Remove two largest and one smallest:

```
nums[n-3] - nums[1]
```
- Remove one largest and two smallest:

```
nums[n-2] - nums[2]
```
- Remove the three smallest values:

```
nums[n-1] - nums[3]
```
4. Compute the minimum among these four differences.

This gives the smallest achievable range after at most three moves.

### Why it works

After sorting, the only elements that can affect the final range are the smallest and largest remaining values. Since we may modify at most three elements, the only meaningful operations are removing some number of smallest elements and some number of largest elements, where the total removed is three.

Every optimal solution must correspond to one of these four distributions:

```
0 small + 3 large
1 small + 2 large
2 small + 1 large
3 small + 0 large
```

Because these are the only possibilities, checking all four guarantees we find the optimal answer.

## Python Solution

```python
from typing import List

class Solution:
    def minDifference(self, nums: List[int]) -> int:
        n = len(nums)

        if n <= 4:
            return 0

        nums.sort()

        answer = float("inf")

        for left_removed in range(4):
            right_removed = 3 - left_removed

            current_difference = (
                nums[n - 1 - right_removed]
                - nums[left_removed]
            )

            answer = min(answer, current_difference)

        return answer
```

The implementation begins by checking whether the array size is at most four. In that situation, we can always make all elements identical using at most three moves, so the answer is immediately `0`.

Next, the array is sorted so that the smallest and largest elements are easy to access.

The loop iterates through the four valid ways to distribute the three removals between the left side and right side of the sorted array. For each configuration:

- `left_removed` represents how many smallest elements are ignored.
- `right_removed` represents how many largest elements are ignored.

The remaining range is computed using the new smallest and largest surviving elements.

The algorithm keeps track of the minimum difference across all four possibilities and returns it.

## Go Solution

```go
package main

import (
	"math"
	"sort"
)

func minDifference(nums []int) int {
	n := len(nums)

	if n <= 4 {
		return 0
	}

	sort.Ints(nums)

	answer := math.MaxInt32

	for leftRemoved := 0; leftRemoved < 4; leftRemoved++ {
		rightRemoved := 3 - leftRemoved

		currentDifference := nums[n-1-rightRemoved] - nums[leftRemoved]

		if currentDifference < answer {
			answer = currentDifference
		}
	}

	return answer
}
```

The Go implementation follows the same logic as the Python version.

The main difference is that Go requires explicit imports for sorting and integer constants. The solution uses `sort.Ints(nums)` to sort the slice in place and initializes the answer with `math.MaxInt32`.

Go slices already behave dynamically, so no additional array handling is needed.

## Edge Cases

One important edge case occurs when the array length is less than or equal to four. Since we can perform at most three moves, we can always transform all elements into the same value. A naive implementation that blindly accesses indices assuming at least five elements may produce index errors. The solution handles this immediately with an early return of `0`.

Another important case involves duplicate values. For example:

```
[10, 10, 10, 1000]
```

The optimal solution is to modify the single outlier. Some implementations incorrectly assume distinct values or mishandle repeated numbers when calculating ranges. Sorting and directly computing endpoint differences avoids this issue entirely.

Negative values are another potential source of bugs. Arrays like:

```
[-10, -5, 0, 5, 10]
```

still follow the same logic after sorting. Since the algorithm relies only on ordering and subtraction, negative numbers are handled naturally without any special cases.

A final subtle edge case occurs when the optimal strategy removes elements from both ends simultaneously. Some incorrect solutions only consider removing the three largest or three smallest values. However, the best answer may require a mixed strategy, such as removing two small values and one large value. The algorithm explicitly checks all four valid distributions, guaranteeing correctness.
