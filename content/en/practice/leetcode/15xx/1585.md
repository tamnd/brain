## Solution

## Problem Understanding

This problem asks whether we can transform one digit string, `s`, into another digit string, `t`, using a very specific operation. In one operation, we may choose any non-empty contiguous substring of `s` and sort that substring in ascending order.

The key detail is that sorting a substring only allows smaller digits to move left and larger digits to move right within that substring. A digit can never "jump over" a smaller digit that appears before it, because sorting preserves ascending order.

We are given two strings of equal length consisting only of digits from `'0'` to `'9'`. We must determine whether repeated applications of the allowed operation can transform `s` exactly into `t`.

The constraints are important:

- `1 <= s.length <= 10^5`
- Both strings contain only digits
- The strings have equal length

The input size immediately rules out brute force simulation of all possible substring sorts. There are far too many substrings and too many possible intermediate states. We need a near linear-time solution.

Several edge cases are especially important:

- The two strings may contain different digit frequencies. In that case, transformation is impossible immediately.
- Digits may repeat many times, so we must carefully track which occurrence of a digit is being matched.
- A larger digit can move right easily, but moving a smaller digit left may be blocked by another smaller digit appearing earlier in the string.
- Strings that already match should return `true`.
- Very long inputs require an efficient data structure and careful implementation.

## Approaches

### Brute Force Approach

A brute force solution would attempt to simulate all possible substring sorting operations starting from `s`.

At every step, we could:

1. Choose every possible substring.
2. Sort that substring.
3. Generate a new string state.
4. Continue searching until we either reach `t` or exhaust all possibilities.

This approach is theoretically correct because it explores every reachable transformation. However, it is completely impractical.

A string of length `n` has `O(n^2)` substrings. Each transformation creates a new state, and the number of reachable states grows explosively. Even for relatively small inputs, the search space becomes enormous.

With `n = 10^5`, brute force is impossible.

### Key Insight

The important observation is that sorting a substring only allows a digit to move left if there are no smaller digits blocking it.

Suppose we want to place digit `d` from `s` into its next required position in `t`.

If some smaller digit already appears before that occurrence of `d` in `s`, then `d` can never move ahead of it. Sorting always preserves ascending order, so smaller digits stay before larger digits.

This creates a crucial invariant:

> A digit `d` can only move left across digits larger than `d`, never across smaller digits.

We can process `t` from left to right and greedily match each required digit with the next unused occurrence in `s`.

To do this efficiently:

- Store all indices of each digit from `s`
- Use queues so we can always retrieve the next unused occurrence
- Before using digit `d`, check whether any smaller digit still has an unused occurrence positioned earlier in the string

If such a smaller digit exists, transformation is impossible.

This leads to an efficient greedy solution.

## Algorithm Walkthrough

1. Create ten queues, one for each digit from `0` to `9`.

Each queue stores the indices where that digit appears in `s`. For example, if `s = "84532"`, then:

- `2 -> [4]`
- `3 -> [3]`
- `4 -> [1]`
- `5 -> [2]`
- `8 -> [0]`

We use queues because we always want the leftmost unused occurrence of each digit.
2. Process the target string `t` from left to right.

For each character `char` in `t`, convert it into digit `d`.
3. Check whether there is any unused occurrence of `d`.

If the queue for digit `d` is empty, then `t` requires more copies of `d` than `s` contains, so return `false`.
4. Let `current_index` be the leftmost unused occurrence of digit `d` in `s`.

This is the occurrence we want to move into the current target position.
5. Check all smaller digits from `0` to `d - 1`.

If any smaller digit still has an unused occurrence positioned before `current_index`, then digit `d` cannot move ahead of it.

In that situation, transformation is impossible, so return `false`.
6. If no smaller digit blocks movement, consume this occurrence of `d`.

Remove the front element from digit `d`'s queue.
7. Continue until all characters in `t` are processed.
8. If every digit placement succeeds, return `true`.

### Why it works

The algorithm maintains the invariant that every processed prefix of `t` is achievable from `s`.

When we select the next occurrence of digit `d`, we verify that no smaller unused digit appears earlier in `s`. Since substring sorting preserves ascending order, a larger digit can never cross a smaller one. Therefore, if such a smaller digit exists, the transformation is impossible.

Conversely, if no smaller digit blocks `d`, then repeated substring sorts can move `d` left across larger digits as needed. Thus every accepted placement is valid.

Because we process digits greedily from left to right and always use the earliest available occurrence, the algorithm correctly determines transformability.

## Python Solution

```python
from collections import deque
from typing import List

class Solution:
    def isTransformable(self, s: str, t: str) -> bool:
        positions = [deque() for _ in range(10)]

        for index, char in enumerate(s):
            digit = int(char)
            positions[digit].append(index)

        for char in t:
            digit = int(char)

            if not positions[digit]:
                return False

            current_index = positions[digit][0]

            for smaller_digit in range(digit):
                if (
                    positions[smaller_digit]
                    and positions[smaller_digit][0] < current_index
                ):
                    return False

            positions[digit].popleft()

        return True
```

The implementation begins by creating ten queues using `deque`, one for each possible digit.

We iterate through `s` and record the index of every digit. This allows us to quickly retrieve the leftmost unused occurrence of any digit.

Next, we process `t` character by character. For each required digit:

- We verify that an unused occurrence exists.
- We inspect all smaller digits to ensure none appear earlier in `s`.
- If a blocking smaller digit exists, we return `False`.
- Otherwise, we consume the occurrence by removing it from the queue.

If the loop completes successfully, every digit in `t` can be formed legally from `s`, so we return `True`.

The solution is efficient because each index is inserted and removed exactly once.

## Go Solution

```go
func isTransformable(s string, t string) bool {
	positions := make([][]int, 10)

	for i := 0; i < len(s); i++ {
		digit := int(s[i] - '0')
		positions[digit] = append(positions[digit], i)
	}

	used := make([]int, 10)

	for i := 0; i < len(t); i++ {
		digit := int(t[i] - '0')

		if used[digit] >= len(positions[digit]) {
			return false
		}

		currentIndex := positions[digit][used[digit]]

		for smaller := 0; smaller < digit; smaller++ {
			if used[smaller] < len(positions[smaller]) &&
				positions[smaller][used[smaller]] < currentIndex {
				return false
			}
		}

		used[digit]++
	}

	return true
}
```

The Go implementation uses slices instead of deques.

For each digit, `positions[digit]` stores all indices where that digit appears in `s`.

Instead of removing elements from the front of a queue, the Go solution keeps a pointer array named `used`. The value `used[digit]` indicates how many occurrences of that digit have already been consumed.

This approach avoids expensive slice shifting operations and keeps the implementation efficient.

No integer overflow concerns exist because indices are bounded by `10^5`.

## Edge Cases

One important edge case occurs when the strings contain different digit frequencies. For example, `s = "123"` and `t = "122"`. Even before considering movement constraints, transformation is impossible because `t` requires more copies of digit `2` than `s` provides. The implementation handles this naturally by checking whether the queue for a digit is empty before attempting to use it.

Another critical edge case involves repeated digits. Consider `s = "5522"` and `t = "2255"`. A naive implementation might incorrectly track only digit counts rather than individual occurrences. The queue-based approach correctly processes occurrences in left-to-right order, ensuring that movement constraints are validated for every specific copy of each digit.

A third important edge case involves smaller digits blocking movement. For example, `s = "12345"` and `t = "12435"`. Although the strings contain the same digits, digit `4` cannot move ahead of digit `3` because substring sorting always preserves ascending order. The algorithm explicitly checks for smaller unused digits positioned earlier in the source string, preventing invalid transformations.

A final edge case is when the strings are already identical. In that situation, every digit matches immediately without encountering blockers. The algorithm still processes all characters correctly and returns `true` without requiring any special handling.
