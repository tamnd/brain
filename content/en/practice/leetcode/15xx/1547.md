## Solution

## Problem Understanding

This problem asks us to determine the minimum cost of cutting a stick at specified positions. We are given a stick of length `n`, labelled from 0 to `n`, and an array `cuts` which contains positions where cuts need to be made. Each cut costs the current length of the stick being cut. Importantly, we can choose the order in which to perform the cuts, and this order affects the total cost.

The goal is to return the minimum total cost required to perform all the cuts. The key insight is that making a cut splits the stick into two smaller sticks, and the cost of future cuts depends on the lengths of these smaller sticks. The input constraints are significant: `n` can be up to 10^6, but `cuts.length` is limited to at most 100, which suggests that an algorithm exponential in `cuts.length` is feasible, but iterating over all stick lengths directly is not.

Edge cases to consider include: having a single cut, having cuts at positions near the ends of the stick, or having cuts that are consecutive. The problem guarantees that all cut positions are distinct and within the bounds of the stick.

## Approaches

### Brute Force Approach

A brute-force solution would involve trying every possible order of cuts and computing the total cost for each sequence. For each permutation of cuts, we would simulate cutting the stick, summing the costs. This approach is correct because it exhaustively searches all possibilities, but it is prohibitively slow. With up to 100 cuts, the number of permutations is 100!, which is astronomically large.

### Optimal Approach - Dynamic Programming

The key observation is that this is a classic interval DP problem. The order of cuts matters locally: if we know the minimum cost for cutting a stick between two positions `i` and `j`, we can build up the solution for larger sticks. We first sort the `cuts` array and add the stick boundaries `0` and `n` to it. Then we define `dp[i][j]` as the minimum cost to perform all cuts between `cuts[i]` and `cuts[j]`. For each interval, we try every possible cut within it, adding the cost of cutting the full interval plus the costs of the left and right subintervals recursively.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(c!) where c = cuts.length | O(c) | Try every permutation of cuts, not feasible for c ~ 100 |
| Optimal | O(c^3) | O(c^2) | Interval DP over sorted cuts with recursion or tabulation |

## Algorithm Walkthrough

1. Sort the cuts array and append the stick boundaries 0 and n. This ensures that all subproblems are aligned with the original stick endpoints.
2. Initialize a 2D DP table `dp[i][j]` to store the minimum cost of cutting the stick between cuts[i] and cuts[j]. Set initial values to zero.
3. Iterate over intervals of increasing length. For each interval `(i, j)`, consider every possible cut `k` that lies between `i` and `j`.
4. For each possible cut `k`, compute the cost as the length of the current interval `cuts[j] - cuts[i]` plus `dp[i][k] + dp[k][j]`.
5. Update `dp[i][j]` to be the minimum of all possible cut placements within this interval.
6. The final answer is `dp[0][m-1]` where `m` is the length of the augmented cuts array, representing the minimum cost to cut the full stick.

Why it works: Each subproblem computes the minimal cost of cutting a segment, assuming optimal solutions for smaller segments. By systematically combining these optimal sub-solutions, the algorithm guarantees a global minimum cost.

## Python Solution

```python
from typing import List

class Solution:
    def minCost(self, n: int, cuts: List[int]) -> int:
        cuts = sorted(cuts)
        cuts = [0] + cuts + [n]
        m = len(cuts)
        
        dp = [[0] * m for _ in range(m)]
        
        for length in range(2, m):
            for i in range(m - length):
                j = i + length
                dp[i][j] = float('inf')
                for k in range(i + 1, j):
                    cost = cuts[j] - cuts[i] + dp[i][k] + dp[k][j]
                    if cost < dp[i][j]:
                        dp[i][j] = cost
        
        return dp[0][m-1]
```

The Python implementation first sorts the cut positions and adds the stick boundaries to simplify the interval calculations. It uses a DP table where `dp[i][j]` represents the minimal cost to cut the segment between cuts[i] and cuts[j]. For each interval, it tries all possible cuts and picks the one minimizing the total cost. This directly follows the interval DP algorithm explained above.

## Go Solution

```go
func minCost(n int, cuts []int) int {
    cuts = append(cuts, 0, n)
    sort.Ints(cuts)
    m := len(cuts)
    
    dp := make([][]int, m)
    for i := range dp {
        dp[i] = make([]int, m)
    }
    
    for length := 2; length < m; length++ {
        for i := 0; i+length < m; i++ {
            j := i + length
            dp[i][j] = 1 << 30
            for k := i + 1; k < j; k++ {
                cost := cuts[j] - cuts[i] + dp[i][k] + dp[k][j]
                if cost < dp[i][j] {
                    dp[i][j] = cost
                }
            }
        }
    }
    
    return dp[0][m-1]
}
```

In Go, we handle slices explicitly and initialize a 2D slice for the DP table. The logic follows the Python version closely, with attention to Go-specific syntax for slices and the large integer used as infinity.

## Worked Examples

### Example 1: `n = 7, cuts = [1,3,4,5]`

After sorting and adding boundaries: `[0,1,3,4,5,7]`

| Interval | Possible Cuts | dp[i][j] Value |
| --- | --- | --- |
| 0-2 | cut at 1 | 1-0 + 0 + 0 = 1 |
| 0-5 | try cuts 1,3,4,5 | min total cost = 16 |

The DP table eventually computes `dp[0][5] = 16`.

### Example 2: `n = 9, cuts = [5,6,1,4,2]`

After sorting and adding boundaries: `[0,1,2,4,5,6,9]`

Applying interval DP results in `dp[0][6] = 22`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(c^3) | We iterate over O(c^2) intervals and for each interval try O(c) cuts |
| Space | O(c^2) | 2D DP table of size (c+2) x (c+2) |

Even though `n` is large, the DP only depends on the number of cuts, making it efficient for the given constraints.

## Edge Cases

The first edge case is when there is only one cut. The solution handles this naturally, returning the length of the stick as the cost. The second edge case is cuts at the extremes, near 0 or n. Since we add 0 and n explicitly to the cuts array, the interval calculations are correct. The third edge case is a large number of cuts close together, which could create many overlapping intervals. The interval DP still correctly computes the minimum cost for each interval, ensuring the global minimum cost is found.
