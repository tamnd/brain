## Solution

## Problem Understanding

The problem gives us a target array and asks us to build it starting from an array of all zeros. In one operation, we may choose any contiguous subarray and increment every element in that subarray by exactly one.

The goal is to determine the minimum number of such operations needed to transform the zero array into the target array.

For example, if the target is `[1,2,3,2,1]`, we can think of the array as layers of height. One operation paints a horizontal strip across a contiguous region. We want to cover the entire structure using as few strips as possible.

The input array `target` has length up to `10^5`, and each value can also be as large as `10^5`. These constraints immediately tell us that expensive simulation approaches will not work. Any solution worse than roughly `O(n log n)` is likely too slow, and an `O(n)` solution is ideal.

A key observation is that operations affect contiguous ranges uniformly. When moving from left to right through the array, increases in height require new operations, while decreases do not. If the current element is smaller than or equal to the previous one, earlier operations can already account for the necessary increments.

Several edge cases are important:

- A single-element array such as `[5]`
- Strictly increasing arrays such as `[1,2,3,4]`
- Strictly decreasing arrays such as `[5,4,3,2]`
- Arrays with repeated values such as `[2,2,2,2]`
- Large arrays where efficiency matters

The problem guarantees that all values are positive integers and that the final answer fits within a 32-bit integer.

## Approaches

### Brute Force Approach

A brute force strategy would simulate the process directly.

One possible method is:

1. Find contiguous positive regions.
2. Increment those regions one layer at a time.
3. Decrease the corresponding target values until all become zero.
4. Count how many operations were used.

For example:

```
[1,2,3]
```

We could subtract one from the entire range:

```
[0,1,2]
```

Then again from the remaining positive suffix:

```
[0,0,1]
```

Then once more:

```
[0,0,0]
```

This uses 3 operations.

While this works conceptually, it is inefficient because each operation may scan large portions of the array repeatedly. In the worst case, such as a strictly increasing array of size `10^5`, the number of simulated operations becomes enormous.

This leads to time complexity close to `O(n * max(target[i]))`, which is far too slow.

### Optimal Greedy Observation

The key insight is that we only need new operations when the height increases compared to the previous element.

Suppose we process the array from left to right.

- The first element `target[0]` always requires `target[0]` operations because we start from zero.
- For every later position:

- If `target[i] > target[i-1]`, we need additional operations equal to the increase.
- If `target[i] <= target[i-1]`, no extra operations are needed because previous operations can extend into this position.

Mathematically:

$$\text{answer} = target[0] + \sum_{i=1}^{n-1} \max(0, target[i] - target[i-1])$$

This works because every upward slope introduces new layers that must start at that index.

The algorithm only requires one linear scan.

| Optimal Greedy | O(n) | O(1) | Counts only positive height increases |

## Algorithm Walkthrough

### Optimal Greedy Algorithm

1. Initialize the answer with the value of the first element.

The first element starts from zero, so we need exactly `target[0]` operations to build it.
2. Traverse the array from index `1` to `n - 1`.

At each position, compare the current value with the previous value.
3. If the current value is greater than the previous value, add the difference to the answer.

This difference represents new layers that must begin at the current index because earlier operations cannot create additional height beyond what already existed at the previous position.
4. If the current value is less than or equal to the previous value, do nothing.

Existing operations from earlier indices can already cover this position.
5. Return the accumulated answer after processing the entire array.

### Why it works

Every operation creates a horizontal layer over a contiguous segment. When the array height rises from one index to the next, new layers must begin there. When the height stays the same or decreases, existing layers can continue or stop naturally without requiring extra operations.

Therefore, the minimum number of operations equals the total amount of positive increases encountered while scanning from left to right.

## Python Solution

```python
from typing import List

class Solution:
    def minNumberOperations(self, target: List[int]) -> int:
        operations = target[0]

        for i in range(1, len(target)):
            if target[i] > target[i - 1]:
                operations += target[i] - target[i - 1]

        return operations
```

The implementation directly follows the greedy observation.

The variable `operations` is initialized with `target[0]` because the first position must be built entirely from zero.

The loop starts from index `1` and compares each element with its predecessor. Whenever the current value exceeds the previous value, the difference is added to the total number of operations.

If the current value is smaller or equal, no additional work is needed because previous operations already provide sufficient coverage.

The algorithm performs a single pass through the array and uses constant extra memory.

## Go Solution

```go
func minNumberOperations(target []int) int {
    operations := target[0]

    for i := 1; i < len(target); i++ {
        if target[i] > target[i-1] {
            operations += target[i] - target[i-1]
        }
    }

    return operations
}
```

The Go implementation is almost identical to the Python version.

The solution uses integer arithmetic throughout, and the problem guarantees the result fits within a 32-bit integer, so the standard `int` type is sufficient.

Go slices already provide efficient indexed access, so no additional data structures are necessary.

## Edge Cases

### Single Element Array

A single-element array such as `[5]` is the simplest possible case. A buggy implementation might incorrectly initialize the answer to zero or mishandle loop boundaries.

The solution handles this correctly because it initializes the answer with `target[0]` and does not require the loop to execute.

### Strictly Decreasing Arrays

Consider `[5,4,3,2,1]`.

A naive implementation might incorrectly add operations at every index. In reality, the initial five operations can already cover the entire decreasing structure by simply ending earlier as needed.

The algorithm correctly avoids adding anything when the current value is less than or equal to the previous value.

### Large Positive Jumps

Arrays such as `[1,100000]` are important because they contain a very large increase.

The algorithm correctly computes the difference:

```
100000 - 1 = 99999
```

and adds it to the initial operation count of `1`, producing the correct answer of `100000`.

### Repeated Values

Plateaus such as `[2,2,2,2]` can expose bugs where implementations unnecessarily count equal neighboring values as additional operations.

The condition:

```
if target[i] > target[i - 1]:
```

ensures that equal values do not increase the operation count.
