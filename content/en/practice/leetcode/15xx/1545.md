## Solution

## Problem Understanding

The problem defines a recursively generated binary string sequence:

- `S1 = "0"`
- `Si = Si-1 + "1" + reverse(invert(Si-1))`

For every new level, we take the previous string, append `"1"`, then append a transformed version of the previous string where:

- every bit is inverted,
- and the result is reversed.

We are given two integers:

- `n`, which specifies which string in the sequence we care about,
- `k`, which specifies the position of the bit we want to retrieve.

The task is to return the `kth` bit of `Sn`.

The important observation is that the strings grow exponentially. Their lengths follow:

- `len(S1) = 1`
- `len(S2) = 3`
- `len(S3) = 7`
- `len(S4) = 15`

In general:

$$|S_n| = 2^n - 1$$

Even though `n <= 20` is not extremely large, directly constructing strings repeatedly is inefficient and unnecessary. The largest possible string length is:

$$2^{20} - 1 \approx 1,048,575$$

This is still manageable in memory, but the recursive structure gives us a much cleaner and more elegant solution.

The input guarantees that `k` is always valid, meaning:

$$1 \le k \le 2^n - 1$$

Some edge cases are especially important:

- When `n = 1`, the answer is always `"0"`.
- When `k` points exactly to the middle position of a string, the answer is always `"1"`.
- Positions in the second half of the string correspond to mirrored and inverted positions in the first half, which is where most implementation mistakes happen.

## Approaches

### Brute Force Approach

The most straightforward solution is to literally build the strings from `S1` up to `Sn`.

At each step:

1. Take the previous string.
2. Invert every bit.
3. Reverse the inverted result.
4. Concatenate:

- original string,
- `"1"`,
- reversed inverted string.

After constructing `Sn`, simply return `Sn[k - 1]`.

This works because it follows the problem definition exactly. However, each string doubles in size at every level, so repeated string construction becomes increasingly expensive.

Although the constraints are small enough that this method technically passes, it wastes both time and memory because we only need one bit, not the entire string.

### Optimal Recursive Approach

The recursive structure of the strings reveals a much more efficient strategy.

For any `Sn`:

$$S_n = S_{n-1} + "1" + reverse(invert(S_{n-1}))$$

The middle position is always:

$$mid = 2^{n-1}$$

This immediately gives three cases:

1. If `k == mid`, the answer is `"1"`.
2. If `k < mid`, the answer lies directly in `S_{n-1}`.
3. If `k > mid`, the answer lies in the mirrored portion:

- map the index back into the first half,
- recursively compute that bit,
- invert the result.

The mirrored index is:

$$mirror = 2^n - k$$

This avoids constructing the string entirely.

## Algorithm Walkthrough

1. Compute the length of `Sn`.

The length of `Sn` is:

$$2^n - 1$$

The middle position is therefore:

$$mid = 2^{n-1}$$

This position always contains `"1"`.
2. Handle the base case.

If `n == 1`, then the string is simply `"0"`.
3. Check whether `k` is the middle position.

If `k == mid`, immediately return `"1"` because the recursive construction always inserts a `"1"` in the center.
4. If `k` lies in the left half, recurse normally.

The left half of `Sn` is exactly `S_{n-1}`.

Therefore:

$$find(n, k) = find(n-1, k)$$
5. If `k` lies in the right half, map it back.

The right half is:

$$reverse(invert(S_{n-1}))$$

Because it is reversed, positions mirror around the center.

The mirrored index is:

$$mirror = 2^n - k$$
6. Recursively compute the mirrored bit.

Compute:

$$bit = find(n-1, mirror)$$
7. Invert the result.

Since the right half is inverted:

- `"0"` becomes `"1"`
- `"1"` becomes `"0"`

Return the inverted value.

### Why it works

The algorithm relies on the exact recursive definition of the string. Every position in the right half corresponds to a mirrored position in the left half after inversion. By repeatedly reducing the problem into smaller subproblems, we eventually reach either:

- the middle bit, which is always `"1"`,
- or the base string `S1 = "0"`.

Because each recursive step preserves the structure of the original definition, the algorithm always returns the correct bit.

## Python Solution

```python
class Solution:
    def findKthBit(self, n: int, k: int) -> str:
        # Base case
        if n == 1:
            return "0"

        mid = 1 << (n - 1)

        # Middle bit is always 1
        if k == mid:
            return "1"

        # Left half
        if k < mid:
            return self.findKthBit(n - 1, k)

        # Right half
        mirrored_index = (1 << n) - k
        bit = self.findKthBit(n - 1, mirrored_index)

        return "1" if bit == "0" else "0"
```

The implementation follows the recursive structure directly.

The first condition handles the smallest possible string, `S1 = "0"`.

Next, the middle position is computed using bit shifting. Since powers of two appear frequently in this problem, bit operations make the implementation concise and efficient.

If `k` equals the middle index, the function immediately returns `"1"` because every generated string inserts a `"1"` at the center.

If `k` lies in the left half, the function recursively searches the same position in `S_{n-1}`.

If `k` lies in the right half, the code computes the mirrored position using:

```python
mirrored_index = (1 << n) - k
```

The recursive call retrieves the corresponding bit from the left half, and the final step inverts that bit.

## Go Solution

```go
func findKthBit(n int, k int) byte {
    // Base case
    if n == 1 {
        return '0'
    }

    mid := 1 << (n - 1)

    // Middle bit
    if k == mid {
        return '1'
    }

    // Left half
    if k < mid {
        return findKthBit(n-1, k)
    }

    // Right half
    mirroredIndex := (1 << n) - k
    bit := findKthBit(n-1, mirroredIndex)

    if bit == '0' {
        return '1'
    }

    return '0'
}
```

The Go implementation mirrors the Python logic closely.

One notable difference is that Go returns a `byte` rather than a string. Character literals therefore use single quotes, such as `'0'` and `'1'`.

Bit shifting works naturally with integers in Go, making it straightforward to compute powers of two efficiently.

Since the recursion depth is at most `20`, there is no concern about stack overflow.

## Edge Cases

One important edge case occurs when `n = 1`. In this situation, the string contains only one character, `"0"`. Recursive implementations can accidentally recurse infinitely if they do not explicitly stop here. The solution handles this immediately with the base condition:

```
if n == 1:
    return "0"
```

Another important edge case is when `k` equals the middle position of the string. Every recursively generated string inserts a `"1"` exactly in the center. Many incorrect solutions unnecessarily recurse further instead of recognizing this structural property. The implementation computes the midpoint and returns `"1"` directly.

A third critical edge case happens when `k` lies in the right half of the string. The right half is not simply inverted, it is both reversed and inverted. Forgetting the reversal leads to incorrect mirrored indices. The implementation carefully maps positions using:

```
mirrored_index = (1 << n) - k
```

This correctly reflects the index across the center before inversion is applied.

A final subtle case involves the largest constraint values, especially `n = 20`. A brute force solution may allocate strings with over one million characters repeatedly, which is unnecessary overhead. The recursive solution avoids constructing strings entirely, so performance remains efficient even at maximum input size.
