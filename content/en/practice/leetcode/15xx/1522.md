## Solution

## Problem Understanding

The problem gives us the root of an N-ary tree and asks us to compute the tree's diameter. An N-ary tree is a tree where each node can have any number of children, unlike a binary tree where each node has at most two children.

The diameter of a tree is defined as the length of the longest path between any two nodes in the tree. The length is measured in edges, not nodes. This path does not necessarily need to pass through the root. The two endpoints of the diameter can exist anywhere in the tree.

For example, consider a path like:

```
5 -> 3 -> 1 -> 4
```

This path contains 3 edges, so its length is 3.

The input is represented as an N-ary tree using level order traversal with `null` markers separating groups of children. However, in the actual implementation, LeetCode already constructs the tree for us and passes the root node directly into the function.

The constraints are important:

- The tree depth is at most 1000.
- The number of nodes is at most `10^4`.

These constraints tell us several things:

- A quadratic solution may become too slow for large trees.
- Recursive DFS is acceptable because the depth is bounded by 1000.
- We should aim for a linear time traversal.

Several edge cases are important to consider:

- A tree with only one node has diameter 0 because there are no edges.
- A completely linear tree behaves like a linked list, where the diameter equals the number of edges from one end to the other.
- The longest path may exist entirely inside a subtree and never pass through the root.
- Nodes may have many children, so we cannot rely on binary-tree-specific logic.

## Approaches

### Brute Force Approach

A brute force solution would compute the longest path between every pair of nodes in the tree.

One way to do this is:

1. Treat the tree as an undirected graph.
2. For every node:

- Run DFS or BFS to find the farthest node from it.
- Track the maximum distance found.

This works because the diameter is simply the maximum shortest-path distance between any two nodes.

The approach is correct because it explicitly checks every possible path indirectly through repeated traversals. However, it is inefficient.

If there are `N` nodes, performing a DFS or BFS from every node costs:

- `O(N)` per traversal
- `O(N)` traversals

This results in `O(N^2)` time complexity.

With up to `10^4` nodes, quadratic complexity becomes expensive.

### Optimal Approach

The key observation is that the diameter can be determined during a single DFS traversal.

For every node, we compute:

- The height of each child subtree
- The two largest child heights

Why do we need the two largest heights?

Because the longest path passing through the current node is formed by:

```
largest child depth + second largest child depth
```

Imagine a node connecting two deep subtrees. The diameter passing through that node uses the deepest path from one subtree and the deepest path from another subtree.

During DFS:

- Each recursive call returns the height of the subtree rooted at that node.
- At every node, we update the global diameter using the two greatest child heights.

This allows us to compute the answer in a single traversal.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(N²) | O(N) | Run DFS/BFS from every node |
| Optimal | O(N) | O(H) | Single DFS traversal using subtree heights |

Here:

- `N` is the number of nodes.
- `H` is the tree height.

## Algorithm Walkthrough

1. Start a depth-first traversal from the root node.
2. For every node, recursively compute the height of each child subtree.

The height returned by a child represents the maximum number of edges from that child down to a leaf.
3. While processing children, keep track of the two largest subtree heights.

We need the two largest because a diameter passing through the current node may connect two different child branches.
4. For each child:

- Compute its subtree height recursively.
- Add 1 to account for the edge connecting the current node to the child.
- Update the largest and second-largest heights if necessary.
5. After all children are processed:

- The longest path passing through the current node is:

```
largest_height + second_largest_height
```

1. Update the global diameter if this value is larger than the current maximum.
2. Return the largest height upward to the parent.

The parent only needs the single longest downward path from this node.
3. Continue until the DFS finishes.
4. Return the global diameter.

### Why it works

For every node, the longest path that passes through it must use its two deepest child subtrees. By computing subtree heights bottom-up, we guarantee that when processing a node, we already know the maximum depths of all descendant paths.

Since every possible diameter either:

- passes through some node, or
- lies entirely inside a subtree,

checking the candidate diameter at every node guarantees that the global maximum is found.

## Python Solution

```python
"""
# Definition for a Node.
class Node:
    def __init__(self, val: Optional[int] = None, children: Optional[List['Node']] = None):
        self.val = val
        self.children = children if children is not None else []
"""

from typing import Optional, List

class Solution:
    def diameter(self, root: 'Node') -> int:
        self.max_diameter = 0

        def dfs(node: 'Node') -> int:
            if not node:
                return 0

            longest = 0
            second_longest = 0

            for child in node.children:
                child_height = dfs(child) + 1

                if child_height > longest:
                    second_longest = longest
                    longest = child_height
                elif child_height > second_longest:
                    second_longest = child_height

            self.max_diameter = max(
                self.max_diameter,
                longest + second_longest
            )

            return longest

        dfs(root)
        return self.max_diameter
```

The implementation uses recursive DFS to process the tree bottom-up.

The `dfs()` function returns the height of the subtree rooted at the current node. More specifically, it returns the maximum number of edges from the current node down to any descendant leaf.

Inside each DFS call, we maintain two variables:

- `longest`
- `second_longest`

These store the two greatest child subtree heights.

For every child:

1. Recursively compute its subtree height.
2. Add 1 for the connecting edge.
3. Update the two largest heights.

After processing all children, the candidate diameter through the current node becomes:

```python
longest + second_longest
```

This value is compared against the global answer.

Finally, the function returns `longest` upward because the parent only needs the single deepest branch extending downward from this node.

## Go Solution

```go
/**
 * Definition for a Node.
 * type Node struct {
 *     Val int
 *     Children []*Node
 * }
 */

func diameter(root *Node) int {
    maxDiameter := 0

    var dfs func(node *Node) int

    dfs = func(node *Node) int {
        if node == nil {
            return 0
        }

        longest := 0
        secondLongest := 0

        for _, child := range node.Children {
            childHeight := dfs(child) + 1

            if childHeight > longest {
                secondLongest = longest
                longest = childHeight
            } else if childHeight > secondLongest {
                secondLongest = childHeight
            }
        }

        if longest+secondLongest > maxDiameter {
            maxDiameter = longest + secondLongest
        }

        return longest
    }

    dfs(root)

    return maxDiameter
}
```

The Go implementation follows the same logic as the Python version.

Some Go-specific details are worth noting:

- Go uses `nil` instead of Python's `None`.
- Recursive helper functions are commonly implemented using function variables.
- Slices are used for child lists.
- Integer overflow is not a concern because the maximum diameter is at most `10^4 - 1`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(N) | Every node is visited exactly once |
| Space | O(H) | Recursive call stack depth equals tree height |

The DFS traversal processes each node one time and performs constant work per child relationship, resulting in linear complexity.

The space complexity comes from recursion depth. In the worst case, a completely skewed tree may have height `N`, though the problem guarantees the depth does not exceed 1000.

## Edge Cases

A single-node tree is the simplest possible input. Since the diameter counts edges rather than nodes, the correct answer is 0. A buggy implementation might incorrectly return 1 if it confuses node count with edge count. This implementation correctly returns 0 because leaf nodes contribute height 0 and no edges are added.

A completely skewed tree behaves like a linked list. In this case, recursion depth becomes large and the diameter equals the tree height. Some implementations incorrectly assume branching structure and fail to propagate heights properly. Here, the DFS naturally handles skewed trees because every node simply forwards its single longest path upward.

Another important edge case occurs when the diameter does not pass through the root. Many incorrect solutions only compute paths involving the root node. This implementation avoids that issue by evaluating a candidate diameter at every node independently. As a result, deep subtree diameters are correctly detected even when the root is irrelevant to the longest path.

A node with many children is another possible source of bugs. In binary trees, tracking two child heights is straightforward because there are only two children. In an N-ary tree, we must dynamically track the largest and second-largest heights while iterating through all children. This implementation correctly updates those values regardless of the number of children.
