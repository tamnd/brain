## Solution

## Problem Understanding

The problem asks us to find the `kth` positive integer that is **missing** from a sorted array of positive integers `arr`. In other words, some positive integers are skipped in `arr`, and we need to identify the one that is missing at the `kth` position in this sequence of missing numbers.

The input array `arr` is strictly increasing, meaning no duplicates exist, and every element is guaranteed to be greater than the previous. The integer `k` specifies which missing number to return. The constraints ensure that the array is not excessively large (maximum length 1000), all elements are positive and ≤1000, and `k` is also ≤1000, so brute-force solutions are feasible but may not be optimal.

Important edge cases include situations where all missing numbers occur before the first element, missing numbers appear between elements, or the `kth` missing number is beyond the last element of the array. For instance, if `arr = [2,3,4,7,11]` and `k = 1`, the first missing number is before the first element, which is 1. The problem guarantees that `arr` is sorted and strictly increasing, which simplifies counting missing numbers.

## Approaches

The **brute-force approach** iterates through every positive integer starting from 1 and counts how many are missing from the array. When the counter reaches `k`, we return the current number. This approach works because it directly tracks missing numbers, but it can be slow if `arr` has large numbers and `k` is large, as it could require scanning up to 2000 numbers to find the 1000th missing.

The **optimal approach** relies on the insight that for a sorted array, the number of missing numbers up to index `i` can be calculated as `arr[i] - (i + 1)`. This formula works because if there were no missing numbers, `arr[i]` would equal `i + 1`. Using this, we can perform a binary search to efficiently find the first index where the number of missing numbers is greater than or equal to `k`. If `k` exceeds the number of missing numbers in the array, the answer is simply beyond the last element.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(max(arr[-1], k)) | O(1) | Count each positive number until the kth missing |
| Optimal | O(log n) | O(1) | Binary search using missing count formula |

## Algorithm Walkthrough

1. Compute the number of missing numbers at the last index as `missing_last = arr[-1] - len(arr)`. If `k` exceeds `missing_last`, the `kth` missing number is beyond the last element. Return `arr[-1] + (k - missing_last)`.
2. Initialize a binary search with `left = 0` and `right = len(arr) - 1`. The goal is to find the first index `idx` where the number of missing numbers is greater than or equal to `k`.
3. While `left < right`, compute `mid = left + (right - left) // 2` and calculate `missing = arr[mid] - (mid + 1)`. If `missing < k`, the `kth` missing number is to the right, so set `left = mid + 1`. Otherwise, set `right = mid`.
4. After the loop, `left` points to the first element for which the number of missing numbers is ≥ k. Compute the answer as `arr[left - 1] + (k - (arr[left - 1] - left))`. This formula adjusts for how many numbers are missing before the current element.

**Why it works:** The algorithm maintains the invariant that at each step, we know the number of missing numbers up to the current midpoint. By binary searching for the first index where the missing count reaches or exceeds `k`, we narrow down the exact location of the `kth` missing number efficiently. This guarantees correctness without scanning every number.

## Python Solution

```python
from typing import List

class Solution:
    def findKthPositive(self, arr: List[int], k: int) -> int:
        n = len(arr)
        # If the kth missing number is beyond the last element
        if k > arr[-1] - n:
            return arr[-1] + (k - (arr[-1] - n))
        
        left, right = 0, n - 1
        while left < right:
            mid = left + (right - left) // 2
            missing = arr[mid] - (mid + 1)
            if missing < k:
                left = mid + 1
            else:
                right = mid
        
        # kth missing is before arr[left]
        return arr[left - 1] + (k - (arr[left - 1] - (left - 1)))
```

The Python code first checks whether the `kth` missing number is beyond the array. If not, it uses binary search to locate the position where the `kth` missing number fits. The formula at the end adjusts for the offset caused by previously missing numbers.

## Go Solution

```go
func findKthPositive(arr []int, k int) int {
    n := len(arr)
    missingLast := arr[n-1] - n
    if k > missingLast {
        return arr[n-1] + (k - missingLast)
    }
    
    left, right := 0, n-1
    for left < right {
        mid := left + (right - left)/2
        missing := arr[mid] - (mid + 1)
        if missing < k {
            left = mid + 1
        } else {
            right = mid
        }
    }
    
    return arr[left-1] + (k - (arr[left-1] - (left - 1)))
}
```

The Go version mirrors the Python solution closely. Go uses explicit integer division and does not require a separate `len` function for arrays; slices are used instead. Edge cases, such as when `k` exceeds the total missing count, are handled identically.

## Worked Examples

**Example 1:** `arr = [2,3,4,7,11], k = 5`

Step through missing numbers:

| i | arr[i] | Missing count `arr[i] - (i+1)` |
| --- | --- | --- |
| 0 | 2 | 1 |
| 1 | 3 | 1 |
| 2 | 4 | 1 |
| 3 | 7 | 3 |
| 4 | 11 | 6 |

Binary search finds `left = 4` (first index where missing ≥ 5). Previous missing count: `arr[3] - 4 = 3`. Compute result: `arr[3] + (5 - 3) = 7 + 2 = 9`.

**Example 2:** `arr = [1,2,3,4], k = 2`

Missing numbers are beyond the last element. `missingLast = 4 - 4 = 0`. Since `k > missingLast`, return `4 + (2 - 0) = 6`.

The complexity is efficient because it avoids scanning all numbers up to the `kth` missing number. Binary search reduces the search space logarithmically relative to the array size.
