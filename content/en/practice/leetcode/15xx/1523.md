## Solution

## Problem Understanding

The problem asks us to count how many odd integers exist in the inclusive range from `low` to `high`.

An integer is odd if it is not divisible by 2. Examples of odd numbers are `1`, `3`, `5`, and `7`. Even numbers, such as `2`, `4`, and `6`, are divisible by 2.

The input consists of two non-negative integers:

- `low`, the beginning of the interval
- `high`, the end of the interval

The interval is inclusive, which means both `low` and `high` are considered part of the range.

For example, if `low = 3` and `high = 7`, the numbers in the range are:

```
3, 4, 5, 6, 7
```

Among these, the odd numbers are:

```
3, 5, 7
```

So the answer is `3`.

The constraints are important:

```
0 <= low <= high <= 10^9
```

The upper bound can be as large as one billion. This tells us that iterating through every number in the interval could potentially be very expensive if the interval is large. A direct loop from `low` to `high` could require up to `10^9` iterations, which is far too slow for an efficient solution.

Several edge cases are worth identifying early:

- The interval may contain exactly one number.
- Both endpoints may be even.
- Both endpoints may be odd.
- One endpoint may be odd and the other even.
- The interval may start at `0`.
- The interval may be extremely large.

A correct solution must handle all of these efficiently.

## Approaches

### Brute Force Approach

The most straightforward solution is to iterate through every integer from `low` to `high` and count how many are odd.

For each number:

- Check whether `number % 2 == 1`
- If true, increment a counter

This works because it directly evaluates every value in the range and counts exactly the numbers that satisfy the condition.

However, this approach is inefficient for very large ranges. If the interval contains millions or billions of numbers, iterating through all of them becomes too slow.

For example:

```
low = 0
high = 1,000,000,000
```

This would require over one billion iterations.

### Optimal Mathematical Approach

The key observation is that odd and even numbers alternate.

In any consecutive sequence of integers:

- Roughly half are odd
- Roughly half are even

We can determine the number of integers in the range:

```
length = high - low + 1
```

If the length is even, then exactly half of the numbers are odd.

If the length is odd, then the parity of the starting number determines whether there is one extra odd number.

Another elegant mathematical shortcut is:

```sql
count of odds from 0 to n = (n + 1) // 2
```

Therefore:

```
odds in [low, high]
= odds up to high - odds up to (low - 1)
```

Which becomes:

```
(high + 1) // 2 - low // 2
```

This gives the answer in constant time without iterating through the range.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(high - low + 1) | O(1) | Iterates through every number and checks parity |
| Optimal | O(1) | O(1) | Uses mathematical properties of odd numbers |

## Algorithm Walkthrough

### Optimal Algorithm

1. Compute how many odd numbers exist from `0` to `high`.

The formula is:

$\left\lfloor\frac{high+1}{2}\right\rfloor$

This works because every pair of consecutive integers contributes exactly one odd number.
2. Compute how many odd numbers exist from `0` to `low - 1`.

The formula is:

$\left\lfloor\frac{low}{2}\right\rfloor$
3. Subtract the second quantity from the first.

This leaves only the odd numbers inside the interval `[low, high]`.

The final formula is:

$\left\lfloor\frac{high+1}{2}\right\rfloor-\left\lfloor\frac{low}{2}\right\rfloor$
4. Return the computed value.

### Why it works

The formula works because it converts the problem into counting cumulative odd numbers.

The number of odd integers from `0` through `n` is always `(n + 1) // 2`. By subtracting the count before the interval starts, we isolate exactly the odd numbers inside the desired range.

This is analogous to prefix sums:

```
count in interval
= prefix(high) - prefix(low - 1)
```

Since the formulas are mathematically exact, the algorithm always produces the correct result.

## Python Solution

```python
class Solution:
    def countOdds(self, low: int, high: int) -> int:
        return (high + 1) // 2 - low // 2
```

The implementation is extremely compact because the mathematical observation eliminates the need for loops or extra data structures.

The expression:

```python
(high + 1) // 2
```

computes how many odd numbers exist from `0` through `high`.

The expression:

```python
low // 2
```

computes how many odd numbers exist before the interval begins.

Subtracting them gives the number of odd integers inside the inclusive interval `[low, high]`.

Python integer arithmetic safely handles the problem constraints because Python integers automatically expand as needed.

## Go Solution

```go
func countOdds(low int, high int) int {
    return (high + 1) / 2 - low / 2
}
```

The Go implementation is nearly identical to the Python version.

Since Go integer division automatically truncates toward zero for positive integers, it behaves exactly like floor division in this problem.

The constraints are small enough that standard Go `int` values safely handle all calculations.

## Worked Examples

```

The odd numbers are:

```
3, 5, 7
```

Expected answer:

```

```

Step-by-step calculation:

| Expression | Value |
| --- | --- |
| `(high + 1) // 2` | `(7 + 1) // 2 = 4` |
| `low // 2` | `3 // 2 = 1` |
| Result | `4 - 1 = 3` |

Final answer:

```
3
```

### Example 2

```

The numbers are:

```

```

Only `9` is odd.

Step-by-step calculation:

| Expression | Value |
| --- | --- |
| `(high + 1) // 2` | `(10 + 1) // 2 = 5` |
| `low // 2` | `8 // 2 = 4` |
| Result | `5 - 4 = 1` |

Final answer:

```
1
```

## Edge Cases

One important edge case occurs when the interval contains exactly one number. For example, `low = high = 1`. A naive implementation could incorrectly assume intervals always contain multiple values. The mathematical formula handles this naturally because it correctly computes whether that single value is odd or even.

Another important edge case is when both endpoints are even, such as `low = 4` and `high = 8`. In this situation, the odd numbers are entirely inside the interval and not located at either boundary. Incorrect formulas sometimes overcount by assuming an odd endpoint exists. The subtraction formula correctly counts only the internal odd values.

A third edge case involves very large ranges, such as `low = 0` and `high = 10^9`. A brute-force solution would require one billion iterations and would likely time out. The constant-time mathematical solution avoids iteration entirely, so performance remains fast regardless of interval size.

A final edge case occurs when the interval starts at `0`. Some implementations using prefix counting can accidentally mishandle `low - 1` when `low` is zero. The chosen formula avoids negative values entirely and safely handles intervals beginning at zero.
