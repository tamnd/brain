## Solution

## Problem Understanding

This problem asks us to construct the lexicographically smallest possible integer string after performing at most `k` adjacent swaps on the digits of `num`.

An adjacent swap means we may swap positions `i` and `i + 1` in one operation. Since only neighboring digits can be swapped, moving a digit from position `j` to position `i` costs exactly `j - i` swaps.

The goal is not to minimize the numeric value mathematically with arbitrary rearrangement. Instead, we must respect the swap budget and the restriction that swaps are adjacent only.

The input size is large:

- `num.length` can be up to `3 * 10^4`
- `k` can be as large as `10^9`

These constraints immediately rule out brute force permutation generation or repeated simulation of swaps.

The output should be returned as a string. Leading zeros are allowed in the result, even though the original input does not contain leading zeros.

The important observation is that lexicographical minimization behaves greedily. At each position, we want to bring the smallest possible digit forward, as long as the swap cost does not exceed the remaining budget.

Several edge cases are important:

- Large `k`, where we may effectively perform every beneficial swap.
- Strings with repeated digits, where choosing the correct occurrence matters.
- Inputs already minimal, where no swaps improve the result.
- Inputs containing zeros, since moving a zero forward often creates the smallest answer.
- Very large strings, where inefficient shifting or repeated rebuilding becomes too slow.

## Approaches

### Brute Force Approach

A naive solution would repeatedly try every reachable adjacent swap sequence and search for the smallest resulting string.

One possible brute force strategy is:

1. Generate all states reachable within `k` swaps.
2. Use BFS or DFS to explore permutations.
3. Keep the lexicographically smallest result.

This approach is correct because it explores every valid transformation.

However, it is completely infeasible.

The number of reachable states grows explosively. Even a string of length 20 already has an enormous state space under adjacent swaps. With length up to `30000`, brute force is impossible.

Another less extreme brute force method is:

1. For each position `i`,
2. Search the next `k` positions,
3. Find the smallest digit,
4. Physically rotate the substring to move that digit forward.

This works conceptually, but repeated substring rotations cost `O(n)` each. Since this may happen for every position, the total complexity becomes `O(n^2)`.

With `n = 30000`, quadratic complexity is too slow.

### Key Insight for the Optimal Solution

The critical insight is that we do not actually need to physically simulate every adjacent swap.

Suppose a digit originally sits at index `j`. If some earlier digits have already been removed and placed into the answer, then the effective current position of this digit becomes smaller.

We need a way to efficiently answer:

> How many digits before index `j` are still unused?

This is exactly what a Binary Indexed Tree, also called a Fenwick Tree, can provide.

The greedy strategy is:

- Build the answer from left to right.
- At each step, try digits from `0` to `9`.
- For each digit, consider its leftmost unused occurrence.
- Compute how many swaps are needed to bring it to the current front.
- If affordable, take it immediately.

Because lexicographical order is dominated by earlier positions, always taking the smallest feasible digit guarantees optimality.

The Fenwick Tree allows us to compute effective movement costs in `O(log n)` time.

## Algorithm Walkthrough

### Step 1: Store Positions of Every Digit

We create queues for digits `0` through `9`.

For every character in `num`, we store its index inside the corresponding queue.

For example:

```
num = "4321"

digit positions:
1 -> [3]
2 -> [2]
3 -> [1]
4 -> [0]
```

We use queues because we always want the leftmost unused occurrence of a digit.

### Step 2: Initialize the Fenwick Tree

The Fenwick Tree tracks which indices are still available.

Initially every position contains `1`, meaning the digit at that index still exists in the remaining string.

As digits are chosen and removed, we update their positions to `0`.

This allows efficient prefix sum queries.

### Step 3: Build the Result Greedily

We construct the answer one character at a time.

For every answer position:

1. Try digits from `0` to `9`.
2. Skip digits whose queue is empty.
3. Let `idx` be the leftmost occurrence of the current digit.
4. Compute how many unused digits exist before `idx`.

That value equals the number of swaps needed to move this digit to the front.

### Step 4: Compute the Swap Cost

If the Fenwick Tree stores active positions, then:

```
cost = active positions before idx
```

More precisely:

```
cost = query(idx) - 1
```

because the digit itself is included in the prefix sum.

This gives the exact number of adjacent swaps required.

### Step 5: Choose the Smallest Affordable Digit

If `cost <= k`:

1. Append this digit to the answer.
2. Subtract `cost` from `k`.
3. Remove the digit from its queue.
4. Update the Fenwick Tree to mark this position as removed.
5. Move to the next answer position.

We stop checking larger digits because choosing the smallest feasible digit is always optimal.

### Step 6: Continue Until All Digits Are Used

Repeat until the result length equals `len(num)`.

The constructed string is the minimum possible integer.

### Why it works

The algorithm relies on a greedy invariant:

> At every position, the smallest digit that can be moved forward within the remaining swap budget must appear in the optimal answer.

If a smaller feasible digit existed and we skipped it, the resulting number would become lexicographically larger immediately, which can never be repaired later.

The Fenwick Tree guarantees that movement costs are computed correctly even after earlier digits have been removed.

Therefore, each greedy choice is globally optimal.

## Python Solution

```python
from collections import deque
from typing import List

class FenwickTree:
    def __init__(self, size: int):
        self.n = size
        self.tree = [0] * (size + 1)

    def update(self, index: int, delta: int) -> None:
        index += 1

        while index <= self.n:
            self.tree[index] += delta
            index += index & -index

    def query(self, index: int) -> int:
        index += 1
        result = 0

        while index > 0:
            result += self.tree[index]
            index -= index & -index

        return result

class Solution:
    def minInteger(self, num: str, k: int) -> str:
        n = len(num)

        positions = [deque() for _ in range(10)]

        for index, digit in enumerate(num):
            positions[int(digit)].append(index)

        fenwick = FenwickTree(n)

        for i in range(n):
            fenwick.update(i, 1)

        result = []

        for _ in range(n):

            for digit in range(10):

                if not positions[digit]:
                    continue

                index = positions[digit][0]

                cost = fenwick.query(index) - 1

                if cost <= k:
                    k -= cost

                    result.append(str(digit))

                    fenwick.update(index, -1)

                    positions[digit].popleft()

                    break

        return "".join(result)
```

The implementation begins by storing all indices for each digit inside queues. This allows efficient access to the leftmost unused occurrence.

The `FenwickTree` class supports two operations:

- `update(index, delta)` modifies whether a position is active.
- `query(index)` returns how many active positions exist from `0` through `index`.

Initially every index is active, so we insert `1` into every position.

The main loop builds the answer greedily. For each position in the result, we test digits from `0` to `9`.

For a candidate digit:

- We inspect its earliest unused index.
- We compute how many active digits remain before it.
- That count equals the number of adjacent swaps required.

If the digit is affordable, we:

- Append it,
- Deduct swaps from `k`,
- Remove it from the Fenwick Tree,
- Remove its index from the queue.

This continues until all digits are placed.

## Go Solution

```go
package main

import (
	"strings"
)

type FenwickTree struct {
	tree []int
	n    int
}

func NewFenwickTree(size int) *FenwickTree {
	return &FenwickTree{
		tree: make([]int, size+1),
		n:    size,
	}
}

func (f *FenwickTree) Update(index int, delta int) {
	index++

	for index <= f.n {
		f.tree[index] += delta
		index += index & -index
	}
}

func (f *FenwickTree) Query(index int) int {
	index++

	result := 0

	for index > 0 {
		result += f.tree[index]
		index -= index & -index
	}

	return result
}

func minInteger(num string, k int) string {
	n := len(num)

	positions := make([][]int, 10)

	for i, ch := range num {
		digit := int(ch - '0')
		positions[digit] = append(positions[digit], i)
	}

	ptr := make([]int, 10)

	fenwick := NewFenwickTree(n)

	for i := 0; i < n; i++ {
		fenwick.Update(i, 1)
	}

	var result strings.Builder

	for step := 0; step < n; step++ {

		for digit := 0; digit <= 9; digit++ {

			if ptr[digit] >= len(positions[digit]) {
				continue
			}

			index := positions[digit][ptr[digit]]

			cost := fenwick.Query(index) - 1

			if cost <= k {

				k -= cost

				result.WriteByte(byte(digit + '0'))

				fenwick.Update(index, -1)

				ptr[digit]++

				break
			}
		}
	}

	return result.String()
}
```

The Go implementation follows the same algorithmic structure as the Python version.

One notable difference is queue handling. Instead of `deque`, Go uses slices plus pointer indices (`ptr`) to track the current front element efficiently.

The result string is built using `strings.Builder`, which avoids repeated string reallocations.

All arithmetic safely fits inside Go's `int` type because the maximum number of swaps is bounded by roughly `n²`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | Each digit selection performs Fenwick operations |
| Space | O(n) | Queues and Fenwick Tree store linear data |

Each digit is inserted and removed exactly once.

For every result position, we test at most 10 digits, which is constant.

Fenwick Tree queries and updates each cost `O(log n)`.

Therefore the total complexity is:

```
O(n * log n)
```

The space usage is linear because we store:

- All digit indices,
- The Fenwick Tree,
- The output string.

## Edge Cases

### Leading Zeros After Swaps

One subtle case occurs when moving a zero to the front creates a leading zero in the answer.

For example:

```
num = "100"
k = 1
```

A naive implementation might incorrectly forbid leading zeros because the original input guarantees none.

However, the problem explicitly allows leading zeros in the output.

The greedy algorithm naturally handles this because `0` is simply the smallest feasible digit.

### Repeated Digits

Repeated digits are another common source of bugs.

Consider:

```
num = "221"
```

When choosing a digit, we must always use the leftmost unused occurrence.

Using a later occurrence first would waste swaps and potentially block better future moves.

The queues guarantee that occurrences are consumed in correct order.

### Extremely Large k

The constraint allows:

```
k <= 10^9
```

A naive simulation of swaps would become prohibitively expensive.

The Fenwick Tree solution never performs physical swaps. Instead, it computes how many swaps would be required mathematically.

This allows the algorithm to remain efficient even when `k` is enormous.
