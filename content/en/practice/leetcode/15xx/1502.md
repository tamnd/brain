## Solution

## Problem Understanding

The problem asks us to determine whether an input array of integers, `arr`, can be rearranged to form an **arithmetic progression**. An arithmetic progression is a sequence where the difference between consecutive elements is constant. In other words, after sorting or reordering the array, the difference between any two adjacent elements must be the same across the entire sequence.

The input is a list of integers of length between 2 and 1000, with each element in the range `-10^6` to `10^6`. The expected output is a boolean value: `true` if the array can be rearranged to form an arithmetic progression and `false` otherwise.

Important edge cases include arrays with only two elements (which trivially form an arithmetic progression), arrays with duplicate values, and arrays with large positive or negative numbers. The constraints guarantee that the array is non-empty and has at least two elements, so we do not need to handle empty arrays.

## Approaches

### Brute-Force Approach

The most straightforward method is to consider all possible permutations of the array and check if any permutation forms an arithmetic progression. This involves generating all factorial-length permutations (`n!`) of the array and, for each permutation, verifying that the difference between consecutive elements is consistent. While this approach is correct because it exhaustively checks every possible ordering, it is infeasible for large arrays since `n!` grows extremely rapidly (e.g., for `n=10`, `n! = 3,628,800`). Therefore, this method does not scale to arrays with up to 1000 elements.

### Optimal Approach

The key observation is that if the array can be rearranged into an arithmetic progression, then after sorting it, the difference between consecutive elements must be the same. Sorting the array guarantees that any valid arithmetic progression will appear in order. After sorting, we can compute the difference between the first two elements as the common difference and verify that every subsequent pair of consecutive elements has the same difference. If all differences match, the array forms an arithmetic progression; otherwise, it does not.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n!) | O(n) | Generate all permutations and check each |
| Optimal | O(n log n) | O(1) or O(n) depending on sort | Sort array, check consecutive differences |

## Algorithm Walkthrough

1. Sort the input array `arr`. Sorting ensures that if the array can form an arithmetic progression, it will appear sequentially after sorting.
2. Compute the common difference `diff` as the difference between the first two elements, i.e., `arr[1] - arr[0]`. This is the difference every consecutive pair must have.
3. Iterate through the array from index 2 to the end.
4. For each element `arr[i]`, check if `arr[i] - arr[i-1]` equals the common difference `diff`.
5. If any consecutive difference does not match `diff`, return `false` immediately because the array cannot form an arithmetic progression.
6. If the loop completes without finding any mismatch, return `true`.

**Why it works**: Sorting the array guarantees that any valid arithmetic progression appears in order. The common difference must be constant across all consecutive elements. If any pair breaks this difference, the property of an arithmetic progression is violated.

## Python Solution

```python
from typing import List

class Solution:
    def canMakeArithmeticProgression(self, arr: List[int]) -> bool:
        arr.sort()
        if len(arr) < 2:
            return True  # Trivial case
        diff = arr[1] - arr[0]
        for i in range(2, len(arr)):
            if arr[i] - arr[i - 1] != diff:
                return False
        return True
```

The Python implementation first sorts the array, then calculates the common difference between the first two elements. It iterates through the array starting from the third element, checking if the difference with the previous element matches the initial difference. If a mismatch is found, it immediately returns `false`. Otherwise, it returns `true` after verifying all elements.

## Go Solution

```go
import "sort"

func canMakeArithmeticProgression(arr []int) bool {
    sort.Ints(arr)
    if len(arr) < 2 {
        return true
    }
    diff := arr[1] - arr[0]
    for i := 2; i < len(arr); i++ {
        if arr[i]-arr[i-1] != diff {
            return false
        }
    }
    return true
}
```

In Go, we use `sort.Ints` to sort the array. Integer subtraction works safely within the constraints. The loop checks consecutive differences and returns early if a mismatch is detected. The handling of slices in Go is similar to arrays in Python, but Go requires explicit indexing syntax.

## Edge Cases

**Case 1: Two elements**

Any array with exactly two elements trivially forms an arithmetic progression because there is only one difference to check. The algorithm handles this by computing the difference and confirming there are no mismatches.

**Case 2: Duplicate elements**

If duplicates are present, they may or may not allow an arithmetic progression. For example, `[1,1,1]` works with difference `0`, but `[1,1,2]` cannot form an arithmetic progression. The difference check in the algorithm ensures this is handled correctly.

**Case 3: Large magnitude numbers**

The input range allows values up to ±10^6. Sorting and difference computation are safe with 32-bit integers in Go and Python integers, which are arbitrary-precision, ensuring no overflow occurs.

**Case 4: Negative differences**

The common difference may be negative if the array is decreasing. Sorting ensures we always compute the correct positive or negative difference. For instance, `[5,3,1]` after sorting becomes `[1,3,5]`, yielding a positive difference of 2, consistent with arithmetic progression rules.
