# LeetCode 1594 - Maximum Non Negative Product in a Matrix

## Problem Understanding

The problem gives us an `m x n` matrix called `grid`, where each cell contains an integer that may be positive, negative, or zero. We start at the top-left corner `(0, 0)` and must reach the bottom-right corner `(m - 1, n - 1)`.

At every step, we are only allowed to move either:

- one cell to the right
- one cell downward

Every path forms a sequence of visited cells, and the product of the path is the multiplication of all values along that route.

The goal is to find the largest product that is non-negative. If every possible path produces a negative product, we return `-1`. Otherwise, we return the maximum non-negative product modulo `10^9 + 7`.

A very important detail is that the modulo operation is applied only after determining the true maximum product. We cannot take modulo during the dynamic programming process because negative values and magnitude comparisons would become incorrect.

The constraints are relatively small:

- `1 <= m, n <= 15`
- each value is between `-4` and `4`

Even though the grid dimensions are small, the number of possible paths grows exponentially. A path from the top-left to bottom-right consists of exactly `(m - 1) + (n - 1)` moves, and each move can branch into two choices. This makes brute force impractical for the upper constraint range.

The presence of negative numbers creates the core challenge. A negative value can flip the sign of the product:

- positive × negative = negative
- negative × negative = positive

This means that a very small negative product can suddenly become a very large positive product later if multiplied by another negative number.

Because of this behavior, tracking only the maximum product at each position is not enough.

Important edge cases include:

- paths containing zeros, because zero can become the best non-negative answer
- grids where all possible products are negative
- single-cell grids
- grids with many negative values where the optimal path intentionally accumulates negatives to flip signs later

## Approaches

### Brute Force Approach

The brute force solution explores every possible path from the top-left corner to the bottom-right corner.

At each cell, we recursively try:

- moving right
- moving down

We maintain the running product along the path. When we reach the destination, we check whether the product is non-negative and update the answer if it is larger than the current best.

This approach is correct because it explicitly evaluates every valid path, guaranteeing that the maximum non-negative product will eventually be found.

However, the number of paths grows combinatorially. The total number of paths is:

$$\binom{m+n-2}{m-1}$$

For a `15 x 15` grid, this becomes far too large for exhaustive exploration.

### Key Insight for the Optimal Solution

The important observation is that negative numbers completely change the nature of the problem.

Suppose we only store the maximum product reaching a cell. That fails because a large negative product might later become the optimal positive product after multiplication by another negative value.

For example:

```
Current max = 10
Current min = -100

Next cell = -5

10 × -5 = -50
-100 × -5 = 500
```

The minimum product became the best possible result.

Therefore, for every cell we must track:

- the maximum product reaching that cell
- the minimum product reaching that cell

This naturally leads to dynamic programming.

At every cell:

- we consider transitions from top and left
- we multiply both the maximum and minimum products by the current value
- we compute the new maximum and minimum among all possibilities

This works because the future behavior of a path depends entirely on both extremes.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^(m+n)) | O(m+n) | Explores every possible path recursively |
| Optimal Dynamic Programming | O(m × n) | O(m × n) | Tracks both maximum and minimum products at each cell |

## Algorithm Walkthrough

### Step 1: Create DP Tables

We create two matrices:

- `max_dp[r][c]`, the maximum product reaching cell `(r, c)`
- `min_dp[r][c]`, the minimum product reaching cell `(r, c)`

Both tables have the same dimensions as the input grid.

### Step 2: Initialize the Starting Cell

The starting cell `(0, 0)` has only one possible product:

```
grid[0][0]
```

So:

```
max_dp[0][0] = grid[0][0]
min_dp[0][0] = grid[0][0]
```

### Step 3: Fill the First Row

Cells in the first row can only be reached from the left.

For each cell:

1. multiply the previous product by the current value
2. store the result in both max and min tables

There is only one path to each first-row cell.

### Step 4: Fill the First Column

Similarly, cells in the first column can only be reached from above.

Again, there is exactly one path to each such cell.

### Step 5: Process Internal Cells

For each remaining cell:

1. take the maximum and minimum products from the top cell
2. take the maximum and minimum products from the left cell
3. multiply all four values by the current grid value
4. the largest candidate becomes the new maximum
5. the smallest candidate becomes the new minimum

This ensures we preserve both positive and negative extremes.

### Step 6: Compute the Final Answer

At the destination cell:

- if the maximum product is negative, return `-1`
- otherwise return:

```
max_product % (10^9 + 7)
```

### Why it works

The algorithm works because every path reaching a cell can be summarized by two values:

- the largest possible product
- the smallest possible product

Any future multiplication depends only on these extremes. A future negative number can transform the smallest negative product into the largest positive product. By preserving both values at every step, the dynamic programming state contains all information necessary to build the optimal final answer.

## Python Solution

```python
from typing import List

class Solution:
    def maxProductPath(self, grid: List[List[int]]) -> int:
        MOD = 10**9 + 7

        rows = len(grid)
        cols = len(grid[0])

        max_dp = [[0] * cols for _ in range(rows)]
        min_dp = [[0] * cols for _ in range(rows)]

        max_dp[0][0] = grid[0][0]
        min_dp[0][0] = grid[0][0]

        # First row
        for col in range(1, cols):
            value = max_dp[0][col - 1] * grid[0][col]
            max_dp[0][col] = value
            min_dp[0][col] = value

        # First column
        for row in range(1, rows):
            value = max_dp[row - 1][0] * grid[row][0]
            max_dp[row][0] = value
            min_dp[row][0] = value

        # Remaining cells
        for row in range(1, rows):
            for col in range(1, cols):
                current = grid[row][col]

                candidates = [
                    max_dp[row - 1][col] * current,
                    min_dp[row - 1][col] * current,
                    max_dp[row][col - 1] * current,
                    min_dp[row][col - 1] * current,
                ]

                max_dp[row][col] = max(candidates)
                min_dp[row][col] = min(candidates)

        result = max_dp[rows - 1][cols - 1]

        if result < 0:
            return -1

        return result % MOD
```

The implementation directly follows the dynamic programming strategy.

The two DP tables track the maximum and minimum products reaching each cell. The starting cell is initialized with the grid value itself because no movement has occurred yet.

The first row and first column are handled separately because they each have only one incoming direction. This simplifies the transition logic for the rest of the matrix.

For every internal cell, the code evaluates four possible candidate products:

- maximum from top
- minimum from top
- maximum from left
- minimum from left

Each is multiplied by the current cell value. The largest candidate becomes the best achievable product at that position, while the smallest candidate becomes the worst.

Finally, the algorithm checks whether the destination cell contains a negative maximum product. If so, no non-negative path exists.

## Go Solution

```go
func maxProductPath(grid [][]int) int {
    const MOD int64 = 1000000007

    rows := len(grid)
    cols := len(grid[0])

    maxDP := make([][]int64, rows)
    minDP := make([][]int64, rows)

    for i := 0; i < rows; i++ {
        maxDP[i] = make([]int64, cols)
        minDP[i] = make([]int64, cols)
    }

    start := int64(grid[0][0])

    maxDP[0][0] = start
    minDP[0][0] = start

    // First row
    for col := 1; col < cols; col++ {
        value := maxDP[0][col-1] * int64(grid[0][col])

        maxDP[0][col] = value
        minDP[0][col] = value
    }

    // First column
    for row := 1; row < rows; row++ {
        value := maxDP[row-1][0] * int64(grid[row][0])

        maxDP[row][0] = value
        minDP[row][0] = value
    }

    // Remaining cells
    for row := 1; row < rows; row++ {
        for col := 1; col < cols; col++ {
            current := int64(grid[row][col])

            candidates := []int64{
                maxDP[row-1][col] * current,
                minDP[row-1][col] * current,
                maxDP[row][col-1] * current,
                minDP[row][col-1] * current,
            }

            maxValue := candidates[0]
            minValue := candidates[0]

            for _, value := range candidates {
                if value > maxValue {
                    maxValue = value
                }

                if value < minValue {
                    minValue = value
                }
            }

            maxDP[row][col] = maxValue
            minDP[row][col] = minValue
        }
    }

    result := maxDP[rows-1][cols-1]

    if result < 0 {
        return -1
    }

    return int(result % MOD)
}
```

The Go implementation mirrors the Python logic closely. The main difference is the use of `int64` to safely handle intermediate products. Although the grid values are small, repeated multiplication across up to 29 cells can exceed standard 32-bit integer limits.

Go slices are used for the DP tables, and explicit loops compute the minimum and maximum values because Go does not provide built-in `min` and `max` functions for slices.

## Edge Cases

### Edge Case 1: Zero in the Grid

A zero can completely change the answer because multiplying by zero resets the product to zero. This is important because zero is non-negative and may be preferable to all negative products.

For example:

```
[[1, 3],
 [0, -4]]
```

One path produces `-12`, while another produces `0`. The algorithm correctly keeps track of both maximum and minimum products, allowing zero to become the optimal answer.

### Edge Case 2: Negative Products Becoming Positive

A path with the smallest negative value can later become the largest positive value after multiplication by another negative number.

For example:

```
[-100] × [-5] = 500
```

A naive solution that only tracks maximum products would lose this possibility. The implementation avoids this issue by storing both minimum and maximum products for every cell.

### Edge Case 3: Single-Cell Grids

When the grid contains only one cell, there is no movement at all. The answer depends entirely on that single value.

Examples:

```
[[5]] → 5
[[-2]] → -1
[[0]] → 0
```

The implementation handles this naturally because the starting cell is initialized directly into the DP tables, and no additional processing is required.

### Edge Case 4: Entirely Negative Outcomes

Some grids produce only negative path products. In those situations, the problem requires returning `-1`.

The implementation checks the final maximum product after DP completion. If it is still negative, that means no valid non-negative path exists.
