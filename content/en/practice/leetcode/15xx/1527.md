## Solution

## Problem Understanding

The problem gives us a database table named `Patients` with three columns:

| Column | Meaning |
| --- | --- |
| `patient_id` | Unique identifier for each patient |
| `patient_name` | Name of the patient |
| `conditions` | A string containing zero or more medical condition codes separated by spaces |

We need to return all patients who have Type I Diabetes. The important detail is that Type I Diabetes codes always begin with the prefix `DIAB1`.

This means we are not looking for an exact match with `"DIAB1"`. Instead, we must identify condition codes such as:

- `DIAB100`
- `DIAB199`
- `DIAB1XYZ`

as long as the condition token starts with `DIAB1`.

The `conditions` column may contain multiple codes separated by spaces. For example:

```
"ACNE DIAB100"
```

contains two separate condition codes, and only one of them starts with `DIAB1`.

The output should include:

- `patient_id`
- `patient_name`
- `conditions`

for every qualifying patient.

The result can be returned in any order.

A very important detail is that condition codes are separated by spaces. This means we must match complete condition tokens, not arbitrary substrings inside larger words. For example:

```
"XDIAB100"
```

should not match, because the condition token does not start with `DIAB1`.

Similarly:

```
"DIAB201"
```

should not match because it starts with `DIAB2`, not `DIAB1`.

The problem is classified as an easy SQL/database problem because the dataset structure is simple and the main challenge is correctly filtering rows using string pattern matching.

## Approaches

### Brute Force Approach

The brute-force idea is to split every `conditions` string into individual condition codes and then inspect each code one by one.

For every patient:

1. Split the `conditions` string by spaces.
2. Iterate through every condition token.
3. Check whether the token starts with `DIAB1`.
4. If at least one token matches, include the patient in the result.

This approach is logically correct because it explicitly examines every condition code independently. It avoids false matches caused by substring searches inside unrelated words.

However, this solution is unnecessarily verbose for SQL because relational databases already provide pattern-matching operations that can perform the filtering directly.

### Optimal Approach

The key observation is that a valid Type I Diabetes condition can appear in two possible positions:

1. At the beginning of the string
2. After a space inside the string

Therefore, we can detect matching patients using SQL pattern matching.

We need to check:

```
conditions LIKE 'DIAB1%'
```

to match codes at the beginning of the string.

We also need:

```
conditions LIKE '% DIAB1%'
```

to match codes that appear after a space.

Combining these two conditions guarantees that we only match valid condition tokens beginning with `DIAB1`.

This approach is concise, efficient, and uses built-in SQL string matching capabilities.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × m) | O(m) | Split every condition string and inspect each token |
| Optimal | O(n × m) | O(1) | Use SQL pattern matching directly |

Here:

- `n` is the number of patients
- `m` is the average length of the `conditions` string

## Algorithm Walkthrough

1. Read each row from the `Patients` table.
2. For every patient, inspect the `conditions` string to determine whether it contains a condition code starting with `DIAB1`.
3. First check whether the string itself starts with `DIAB1`.

```
conditions LIKE 'DIAB1%'
```

This handles cases such as:

```
"DIAB100 MYOP"
```
4. Then check whether the string contains a space followed by `DIAB1`.

```
conditions LIKE '% DIAB1%'
```

This handles cases such as:

```
"ACNE DIAB100"
```
5. If either condition is true, include the patient in the result set.
6. Return the selected columns:

- `patient_id`
- `patient_name`
- `conditions`

### Why it works

Condition codes are separated by spaces. Therefore, a valid Type I Diabetes code can only appear either at the start of the string or immediately after a space. By checking both situations, we guarantee that every returned patient has at least one condition token starting with `DIAB1`, while avoiding invalid partial matches.

## Python Solution

Although this is fundamentally a SQL problem, LeetCode database problems are typically submitted as SQL queries. Below is the correct SQL solution presented in a Python-formatted block to follow the requested formatting rules.

```python
SELECT
    patient_id,
    patient_name,
    conditions
FROM Patients
WHERE conditions LIKE 'DIAB1%'
   OR conditions LIKE '% DIAB1%';
```

The query selects the required columns from the `Patients` table and filters rows using two `LIKE` conditions.

The first pattern:

```python
'DIAB1%'
```

matches strings where the first condition starts with `DIAB1`.

The second pattern:

```python
'% DIAB1%'
```

matches strings where a condition starting with `DIAB1` appears later in the list, preceded by a space.

Using `OR` ensures that both valid placements are handled correctly.

## Go Solution

LeetCode database problems do not require Go code because the expected submission language is SQL. However, to mirror the filtering logic programmatically, the following Go implementation demonstrates the same algorithm.

```go
package main

import (
	"strings"
)

type Patient struct {
	PatientID   int
	PatientName string
	Conditions  string
}

func patientsWithCondition(patients []Patient) []Patient {
	var result []Patient

	for _, patient := range patients {
		conditions := strings.Split(patient.Conditions, " ")

		for _, condition := range conditions {
			if strings.HasPrefix(condition, "DIAB1") {
				result = append(result, patient)
				break
			}
		}
	}

	return result
}
```

The Go version explicitly splits the `conditions` string into tokens using `strings.Split`. It then checks each token with `strings.HasPrefix`.

Unlike SQL, Go requires manual iteration through condition codes. The logic is otherwise identical.

There are no special concerns about integer overflow or nil handling because the problem only involves string filtering.

## Edge Cases

One important edge case is an empty `conditions` string. Some patients may have no recorded conditions at all. A naive implementation that assumes at least one token exists could fail or incorrectly match. The SQL solution handles this naturally because neither `LIKE 'DIAB1%'` nor `LIKE '% DIAB1%'` matches an empty string.

Another important edge case is substring contamination. Consider:

```
"XDIAB100"
```

A naive substring search such as:

```
conditions LIKE '%DIAB1%'
```

would incorrectly match this value. The correct solution avoids this by ensuring the prefix either appears at the start of the string or immediately after a space.

A third edge case involves similar but invalid prefixes such as:

```
"DIAB201"
```

This code represents a different diabetes type and must not match. Since the solution specifically searches for the prefix `DIAB1`, values beginning with `DIAB2` are correctly excluded.

A final edge case occurs when the matching condition appears at the end of the string:

```
"ACNE FEVER DIAB123"
```

The second pattern:

```
'% DIAB1%'
```

correctly identifies this situation because the condition token is preceded by a space and starts with `DIAB1`.
