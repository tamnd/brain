# LeetCode 1510, Stone Game IV

## Problem Understanding

This problem describes a two player game between Alice and Bob. There is a pile containing `n` stones, and the players alternate turns removing stones from the pile. On each turn, the player may remove any non-zero perfect square number of stones. Examples of valid moves are removing `1`, `4`, `9`, `16`, and so on, as long as that many stones remain in the pile.

Alice always moves first. If a player cannot make a move, that player loses the game. Since removing at least one stone is required on every move, the only losing terminal state is when the pile contains `0` stones at the start of a player's turn.

The input is a single integer `n`, representing the number of stones initially in the pile. The output should be `true` if Alice can force a win assuming both players play optimally, otherwise `false`.

The constraint is:

```
1 <= n <= 10^5
```

This constraint is important because it immediately rules out extremely expensive recursive exploration without optimization. A naive recursive solution would repeatedly recompute the same game states and become far too slow. Since `n` can reach one hundred thousand, we need a solution that efficiently determines whether each position is winning or losing.

The key observation is that this is a classic impartial game theory problem. Every game state depends only on smaller states, because every move reduces the number of stones. That structure strongly suggests dynamic programming.

Several edge cases are important:

- When `n = 1`, Alice can immediately remove one stone and win.
- When `n` itself is a perfect square, Alice may be able to win in one move by removing all stones.
- Small values such as `2` or `3` can be tricky because only the move `1` is available.
- Large values near `10^5` require efficient iteration over square numbers.

## Approaches

### Brute Force Approach

The brute force strategy is to recursively simulate every possible move from the current number of stones.

For a given state `n`, we try every square number `k^2` such that `k^2 <= n`. After removing `k^2` stones, the opponent faces the state `n - k^2`.

If any move causes the opponent to lose, then the current player wins. Otherwise, the current player loses.

This recursive definition is correct because it directly models optimal play. However, the problem is that many states are recomputed repeatedly. For example, the state `n = 50` may be reached from many different recursive paths, causing exponential growth in computation.

Without memoization, the solution becomes far too slow for `n = 10^5`.

### Optimal Dynamic Programming Approach

The crucial insight is that every state depends only on smaller states.

Define:

```
dp[i] = whether the current player can force a win with i stones remaining
```

For each `i`, we examine every square number `square` such that `square <= i`.

If there exists a move where:

```
dp[i - square] == false
```

then the current state is winning, because we can move the opponent into a losing position.

Otherwise, if every possible move leads to a winning state for the opponent, then the current state is losing.

This is a standard game theory transition:

- A state is winning if at least one move leads to a losing state.
- A state is losing if all moves lead to winning states.

Because each state is computed exactly once, the algorithm becomes efficient enough for the constraint.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | O(n) recursion depth | Recursively explores all game states repeatedly |
| Optimal Dynamic Programming | O(n * sqrt(n)) | O(n) | Computes each state once using previously solved states |

## Algorithm Walkthrough

1. Create a boolean array `dp` of size `n + 1`.

Each index `i` represents whether the current player can force a win when there are `i` stones remaining.
2. Initialize the base case.

```
dp[0] = false
```

If there are zero stones, the current player cannot make a move and therefore loses.
3. Iterate through all states from `1` to `n`.

For every number of stones `i`, determine whether it is a winning or losing position.
4. For each `i`, try all possible square numbers.

Start with `square = 1^2`, then `2^2`, `3^2`, and continue while the square is less than or equal to `i`.
5. Check whether removing that square creates a losing state for the opponent.

If:

```
dp[i - square] == false
```

then the current player can force a win by choosing that move.
6. Mark the current state as winning and stop searching further.

Once a winning move is found, no further checking is necessary.
7. After processing all states, return `dp[n]`.

This tells us whether Alice can force a win starting from `n` stones.

### Why it works

The algorithm works because every game state is completely determined by smaller game states. A position is winning if the player can move to at least one losing position. A position is losing if every legal move leads to a winning position for the opponent.

Since the algorithm computes states in increasing order from `0` to `n`, every required smaller state has already been solved before it is needed.

## Python Solution

```python
class Solution:
    def winnerSquareGame(self, n: int) -> bool:
        dp = [False] * (n + 1)

        for stones in range(1, n + 1):
            square = 1

            while square * square <= stones:
                remove = square * square

                if not dp[stones - remove]:
                    dp[stones] = True
                    break

                square += 1

        return dp[n]
```

The implementation begins by creating the `dp` array. Every position is initially marked `False`, meaning losing, until proven otherwise.

The loop iterates from `1` through `n`. For each number of stones, the code tries every possible square number that can be removed.

The condition:

```python
if not dp[stones - remove]:
```

checks whether the opponent would face a losing state after the move. If such a move exists, the current state becomes winning immediately, and the loop exits early using `break`.

The algorithm fills the table incrementally, ensuring every smaller state has already been computed before it is referenced.

Finally, `dp[n]` is returned as the answer.

## Go Solution

```go
func winnerSquareGame(n int) bool {
    dp := make([]bool, n+1)

    for stones := 1; stones <= n; stones++ {
        for square := 1; square*square <= stones; square++ {
            remove := square * square

            if !dp[stones-remove] {
                dp[stones] = true
                break
            }
        }
    }

    return dp[n]
}
```

The Go implementation follows exactly the same dynamic programming logic as the Python version.

A boolean slice is created using `make`. Go initializes boolean values to `false` automatically, which conveniently matches the losing-state default.

The nested loops iterate through all stone counts and all valid square removals. Once a winning move is discovered, the inner loop terminates early with `break`.

Since the maximum value of `n` is only `10^5`, integer overflow is not a concern for `square * square`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n * sqrt(n)) | For each state, we try all square numbers up to sqrt(n) |
| Space | O(n) | The dynamic programming table stores one value per state |

The outer loop runs `n` times. For each value of `stones`, the inner loop checks all perfect squares less than or equal to `stones`. The number of such squares is approximately `sqrt(stones)`, giving an overall complexity of:

```
O(1 + sqrt(2) + sqrt(3) + ... + sqrt(n))
```

which simplifies to:

```
O(n * sqrt(n))
```

The space complexity is linear because the algorithm stores one boolean value for every integer from `0` to `n`.

## Edge Cases

One important edge case is when `n` itself is a perfect square. In such situations, the first player can remove all stones immediately and win. A buggy implementation might fail to consider removing the entire pile if it incorrectly restricts moves to strictly smaller squares. The implementation correctly handles this because it checks every square satisfying `square * square <= stones`.

Another important edge case is very small values such as `n = 1` or `n = 2`. Small cases are often where recursive or DP base conditions fail. For `n = 1`, the player wins immediately. For `n = 2`, only one move exists, and it leads to a winning state for the opponent. The initialization `dp[0] = False` ensures these cases are evaluated correctly.

A third important edge case involves large values near `10^5`. Recursive brute force solutions may exceed time limits or recursion depth limits. The iterative dynamic programming approach avoids recursion entirely and computes each state once, making it efficient and safe for the maximum constraint.

Another subtle edge case occurs when every legal move leads to a winning state for the opponent. In these situations, the current state must remain losing. Since the DP array is initialized with `False`, and we only change it to `True` when a winning move is found, the implementation naturally handles this scenario correctly.
