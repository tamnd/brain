## Solution

## Problem Understanding

The problem gives us a binary string `target`, and we start with another binary string `s` of the same length that contains only `'0'` characters.

We are allowed to perform a special operation:

- Choose an index `i`
- Flip every bit from index `i` to the end of the string

A flip changes:

- `'0'` into `'1'`
- `'1'` into `'0'`

The goal is to determine the minimum number of such operations needed to transform the initial all-zero string into `target`.

The key detail is that every operation affects an entire suffix of the string, not just a single position. Because flipping at index `i` changes all positions from `i` onward, every decision has cascading effects on later characters.

For example, suppose:

```
target = "10111"
```

We start from:

```
"00000"
```

If we flip starting at index `0`, the string becomes:

```
"11111"
```

If we then flip starting at index `1`, the string becomes:

```
"10000"
```

Each operation changes the parity of all later positions.

The constraints are important:

- `1 <= n <= 10^5`
- `target[i]` is either `'0'` or `'1'`

Since the input size can be as large as one hundred thousand characters, any algorithm worse than linear time may become too slow. This immediately suggests that we should look for a greedy or state-tracking solution instead of explicitly simulating every suffix flip.

Several edge cases are worth identifying early:

- A string containing only `'0'` characters requires zero operations.
- A string alternating between `'0'` and `'1'` causes many flips.
- A string beginning with `'1'` requires an immediate flip.
- Consecutive identical characters should not require repeated flips.
- Very large inputs require an efficient `O(n)` approach with minimal extra memory.

## Approaches

### Brute Force Approach

A straightforward solution is to explicitly simulate the string transformation process.

We begin with a mutable version of the all-zero string. Then, scanning from left to right, whenever the current character does not match the target character, we perform a suffix flip starting at that position.

For example:

```
Current: 00000
Target : 10111
```

At index `0`, the characters differ, so we flip the suffix:

```
11111
```

At index `1`, the characters differ again, so we flip:

```
10000
```

At index `2`, we flip once more:

```
10111
```

This method is correct because every mismatch must eventually be fixed, and flipping at the earliest mismatch is always necessary.

However, each suffix flip may touch up to `O(n)` characters, and we may perform up to `O(n)` flips. Therefore, the total complexity becomes `O(n^2)` in the worst case.

With `n = 10^5`, this is too slow.

### Optimal Greedy Observation

The important insight is that we do not actually need to simulate the string after every flip.

Instead, we only need to know whether the current position has been flipped an even or odd number of times.

Suppose we process the string from left to right.

- Initially, every position is effectively `'0'`
- Each suffix flip toggles the interpretation of all later positions
- Therefore, the current effective bit depends only on the parity of the number of flips performed so far

If we have performed:

- An even number of flips, current effective bit is `'0'`
- An odd number of flips, current effective bit is `'1'`

Whenever the effective bit does not match `target[i]`, we must flip at position `i`.

This leads to a very elegant greedy rule:

- Count how many times the target string changes between adjacent characters
- Add one more flip if the string starts with `'1'`

Equivalently, we can track the current effective bit while scanning.

## Algorithm Walkthrough

### Optimal Greedy Algorithm

1. Initialize a variable `flips = 0`.

This variable stores how many suffix flips we have performed so far.
2. Iterate through each character in `target` from left to right.

At every position, we determine what the current effective bit should be after considering all previous flips.
3. Compute the effective current bit.

Since the initial string is all zeros:

- If `flips` is even, the effective current bit is `'0'`
- If `flips` is odd, the effective current bit is `'1'`
4. Compare the effective current bit with `target[i]`.

- If they already match, no action is needed.
- If they differ, we must perform a suffix flip starting at index `i`.
5. Increment `flips` whenever a mismatch occurs.

This updates the parity for all future positions.
6. Continue until all characters are processed.
7. Return `flips`.

### Why it works

The greedy strategy is optimal because once we reach index `i`, the only operation that can change position `i` is a suffix flip beginning at `i` or earlier. All earlier decisions are already fixed. Therefore, if the current effective bit differs from `target[i]`, flipping at `i` is mandatory.

Since every flip affects all remaining positions uniformly, tracking only flip parity is sufficient. The algorithm never performs unnecessary flips and always fixes mismatches at the earliest possible point.

## Python Solution

```python
class Solution:
    def minFlips(self, target: str) -> int:
        flips = 0
        
        for char in target:
            current_bit = '1' if flips % 2 else '0'
            
            if current_bit != char:
                flips += 1
        
        return flips
```

The implementation directly follows the greedy logic.

The variable `flips` tracks how many suffix flips have occurred so far. Since every flip toggles all future bits, the parity of `flips` determines the current effective bit.

Inside the loop:

- If `flips` is even, the effective bit is `'0'`
- If `flips` is odd, the effective bit is `'1'`

Whenever the effective bit differs from the target character, we increment `flips`, representing a new suffix flip operation.

No actual string modification is needed, which keeps the solution both fast and memory efficient.

## Go Solution

```go
func minFlips(target string) int {
    flips := 0

    for _, ch := range target {
        currentBit := '0'

        if flips%2 == 1 {
            currentBit = '1'
        }

        if ch != currentBit {
            flips++
        }
    }

    return flips
}
```

The Go implementation mirrors the Python approach closely.

A few Go-specific details are worth noting:

- Strings in Go are immutable, but this solution never modifies the string anyway.
- The loop iterates over runes using `range`.
- Only integer counters are used, so there are no overflow concerns within the given constraints.
- The solution uses constant extra space because no auxiliary arrays or slices are allocated.

## Edge Cases

### All Zeros

A target string containing only `'0'` characters is an important edge case because the initial string already matches the target. A buggy implementation might incorrectly perform flips while scanning.

For example:

```
target = "00000"
```

The algorithm correctly detects that the effective bit always matches the target bit, so `flips` remains zero throughout the entire traversal.

### Alternating Characters

Strings such as:

```
"101010"
```

are challenging because every character transition forces another suffix flip. This creates the maximum possible number of flips relative to the string length.

The implementation handles this naturally because every mismatch increments the flip counter, alternating the effective bit each time.

### Large Consecutive Groups

Inputs like:

```
"11110000"
```

can expose bugs in solutions that over-flip repeated identical characters.

Once the effective bit matches the current group, no additional flips are necessary until the target character changes again. The greedy parity-based implementation correctly avoids unnecessary operations.

### Single Character Strings

The smallest possible inputs:

```
"0"
```

and

```
"1"
```

are useful boundary cases.

The algorithm correctly returns:

- `0` for `"0"`
- `1` for `"1"`

because the effective initial bit is `'0'` before any flips occur.
