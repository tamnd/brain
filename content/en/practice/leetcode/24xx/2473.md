## Solution

## Problem Understanding

This problem involves a weighted undirected graph where nodes represent cities and edges represent roads with traversal costs. Each city has a cost associated with buying an apple. You start at a given city, travel along roads to buy **exactly one apple** in any city, and then return to the starting city. The catch is that the **return trip costs are multiplied by a factor `k`**.

The input consists of:

- `n`, the number of cities (nodes)
- `roads`, a list of `[ai, bi, costi]` defining bidirectional road costs
- `appleCost`, a list of integers where `appleCost[i]` is the cost to buy an apple in city `i+1`
- `k`, a multiplier for the return journey

The output is a list where each element is the **minimum total cost to buy one apple starting from that city**, taking the cheapest path to some city to purchase the apple and returning with the multiplied road cost.

Key observations:

- The problem can be seen as **finding shortest paths** from each city to all others, weighted by the trip multiplier for the return journey.
- Brute-force solutions that explore all paths for each starting city would be too slow given `n` up to 1000 and `roads.length` up to 2000.

Edge cases include:

- Cheapest to buy in the starting city itself
- All roads having the same cost
- `k = 1`, making forward and return costs symmetric
- Highly disconnected cities or sparse graphs

## Approaches

### Brute Force

A brute-force approach would iterate over all cities `i` as the starting point. For each starting city, we would:

1. Enumerate every possible city `j` to buy an apple.
2. Compute all paths from `i` to `j` and back, applying the `k` multiplier on the return path.
3. Sum the road costs plus `appleCost[j]` and take the minimum.

This works correctly but is extremely inefficient: enumerating all paths explicitly is `O(2^n)` in a naive DFS approach, or `O(n^3)` if using Floyd-Warshall to precompute all-pairs shortest paths without considering the multiplier cleverly.

### Optimal Approach

The optimal solution uses a **modified Dijkstra’s algorithm** with the following insight:

1. Because only **one city will be chosen for apple purchase**, we can treat the problem as a **single-source shortest path** from each city to all others, but with **forward and return costs separated**.
2. To simplify, notice the total cost to buy an apple at city `j` starting from city `i` is:

```
total_cost = distance(i -> j) + appleCost[j] + distance(i -> j) * k
           = distance(i -> j) * (1 + k) + appleCost[j]
```

Thus, if we precompute `distance(i -> j)` using Dijkstra for each starting city, we can compute the minimal total cost efficiently.

Given constraints (`n <= 1000`, `roads.length <= 2000`), using Dijkstra from each node separately with a priority queue gives `O(n * (m log n))` which is feasible.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^3) | O(n^2) | Enumerates all paths; too slow |
| Optimal (Dijkstra) | O(n * m log n) | O(n + m) | Uses single-source shortest path from each node |

## Algorithm Walkthrough

1. Construct a graph representation using an adjacency list. Each city maps to a list of `(neighbor, cost)` tuples.
2. Define a helper function `dijkstra(start)` that computes shortest distances from the starting city to all others using a min-heap priority queue.
3. For each city `i` (starting point):

a. Run Dijkstra from `i` to compute `dist[i][j]` for all cities `j`.

b. For each city `j`, compute `total_cost = dist[i][j] * (1 + k) + appleCost[j]`.

c. Track the minimum `total_cost` for starting city `i`.
4. Store the minimum costs for all cities in an array and return it.

Why it works: Dijkstra guarantees the shortest path from `i` to every other city. Multiplying the forward distance by `(1 + k)` correctly accounts for the return trip. Adding `appleCost[j]` ensures we include the purchase cost, and taking the minimum over all `j` guarantees the cheapest apple purchase.

## Python Solution

```python
from typing import List
import heapq

class Solution:
    def minCost(self, n: int, roads: List[List[int]], appleCost: List[int], k: int) -> List[int]:
        graph = [[] for _ in range(n)]
        for u, v, cost in roads:
            graph[u-1].append((v-1, cost))
            graph[v-1].append((u-1, cost))
        
        def dijkstra(start: int) -> List[int]:
            dist = [float('inf')] * n
            dist[start] = 0
            heap = [(0, start)]
            
            while heap:
                d, node = heapq.heappop(heap)
                if d > dist[node]:
                    continue
                for neighbor, cost in graph[node]:
                    if dist[neighbor] > d + cost:
                        dist[neighbor] = d + cost
                        heapq.heappush(heap, (dist[neighbor], neighbor))
            return dist
        
        answer = []
        for i in range(n):
            dist = dijkstra(i)
            min_cost = float('inf')
            for j in range(n):
                total = dist[j] * (1 + k) + appleCost[j]
                min_cost = min(min_cost, total)
            answer.append(min_cost)
        return answer
```

This Python code constructs the graph, runs Dijkstra for each starting city, and calculates the minimum total cost considering both forward and return trips with the multiplier applied. The heap ensures efficient extraction of the next closest node.

## Go Solution

```go
package main

import (
    "container/heap"
    "math"
)

type Edge struct {
    to, cost int
}

type Item struct {
    node int
    dist int
}

type PriorityQueue []Item

func (pq PriorityQueue) Len() int { return len(pq) }
func (pq PriorityQueue) Less(i, j int) bool { return pq[i].dist < pq[j].dist }
func (pq PriorityQueue) Swap(i, j int) { pq[i], pq[j] = pq[j], pq[i] }
func (pq *PriorityQueue) Push(x any) { *pq = append(*pq, x.(Item)) }
func (pq *PriorityQueue) Pop() any {
    old := *pq
    n := len(old)
    item := old[n-1]
    *pq = old[0:n-1]
    return item
}

func minCost(n int, roads [][]int, appleCost []int, k int) []int64 {
    graph := make([][]Edge, n)
    for _, r := range roads {
        u, v, c := r[0]-1, r[1]-1, r[2]
        graph[u] = append(graph[u], Edge{v, c})
        graph[v] = append(graph[v], Edge{u, c})
    }
    
    dijkstra := func(start int) []int {
        dist := make([]int, n)
        for i := 0; i < n; i++ { dist[i] = math.MaxInt64 }
        dist[start] = 0
        pq := &PriorityQueue{}
        heap.Init(pq)
        heap.Push(pq, Item{start, 0})
        
        for pq.Len() > 0 {
            curr := heap.Pop(pq).(Item)
            if curr.dist > dist[curr.node] { continue }
            for _, e := range graph[curr.node] {
                if dist[e.to] > curr.dist + e.cost {
                    dist[e.to] = curr.dist + e.cost
                    heap.Push(pq, Item{e.to, dist[e.to]})
                }
            }
        }
        return dist
    }
    
    answer := make([]int64, n)
    for i := 0; i < n; i++ {
        dist := dijkstra(i)
        minCost := int64(math.MaxInt64)
        for j := 0; j < n; j++ {
            total := int64(dist[j]*(k+1) + appleCost[j])
            if total < minCost { minCost = total }
        }
        answer[i] = minCost
    }
    return answer
}
```

Go-specific notes: Uses a custom `PriorityQueue` type and the standard `heap` package. `math.MaxInt64` is used for initial distances. Integer overflows are avoided by converting to `int64` for total cost computations.
