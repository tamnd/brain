# LeetCode 2497 - Maximum Star Sum of a Graph

## Problem Understanding

The problem gives us an undirected graph with `n` nodes. Each node has an integer value stored in the array `vals`, where `vals[i]` represents the value of node `i`.

The graph connections are provided through the `edges` array. Since the graph is undirected, an edge `[a, b]` means node `a` is connected to node `b`, and node `b` is also connected to node `a`.

We must construct a star graph. A star graph consists of:

- One center node
- Zero or more neighboring nodes directly connected to the center
- At most `k` edges

The star sum is the sum of:

- The center node's value
- The values of all selected neighboring nodes

Our goal is to compute the maximum possible star sum among all valid star graphs.

An important detail is that we are allowed to choose fewer than `k` neighbors. This matters because some neighboring nodes may have negative values, and adding them would reduce the total sum.

The constraints are large:

- Up to `10^5` nodes
- Up to `10^5` edges

This immediately tells us that quadratic solutions are too slow. We need an algorithm close to linear or `O(n log n)`.

Several edge cases are especially important:

- `k = 0`, meaning we cannot include any neighbors
- All node values are negative
- Nodes with many neighbors
- Graphs with no edges
- Neighbors with negative values that should be skipped

The problem guarantees that the graph is valid, node indices are in range, and there are no self-loops.

## Approaches

### Brute Force Approach

A brute-force solution would consider every node as the center of a star graph. For each center, we could generate all subsets of its neighbors containing at most `k` nodes, compute their sums, and track the maximum.

This approach is correct because it explicitly checks every possible valid star graph.

However, it is far too slow. If a node has degree `d`, then there are exponentially many subsets of neighbors. In dense graphs, this becomes completely infeasible.

Even a slightly improved brute-force method that sorts all neighbors for every node repeatedly would still be inefficient.

### Key Insight

The key observation is that negative neighbors are never useful.

If a neighboring node has a negative value, adding it decreases the star sum. Since the problem allows using fewer than `k` edges, we should only include neighbors with positive values.

Therefore, for each node:

1. Collect the values of all neighboring nodes
2. Keep only positive neighbor values
3. Select the largest at most `k` values
4. Add them to the center node value

This transforms the problem into a local optimization problem for each node independently.

We can efficiently build adjacency information and process each node once.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | O(n) | Enumerates all neighbor subsets |
| Optimal | O(n + m log m) | O(n + m) | Keeps only top positive neighbors |

Here, `m` represents the number of edges.

## Algorithm Walkthrough

### Step 1: Build adjacency lists

Create an adjacency list for every node.

For every edge `[u, v]`:

- Add `vals[v]` to node `u`'s neighbor list
- Add `vals[u]` to node `v`'s neighbor list

We store neighbor values directly because only the values matter for the final sum.

### Step 2: Process each node independently

Treat each node as a potential center of the star graph.

Initialize the current star sum as:

```
current_sum = vals[node]
```

This represents the star containing only the center node.

### Step 3: Keep only positive neighbors

Negative neighbors decrease the total sum, so they should never be selected.

Filter the neighbor list to retain only positive values.

### Step 4: Select the top `k` neighbors

Sort the positive neighbor values in descending order.

Take the first `k` values and add them to the center value.

This gives the best possible star sum for that center node.

### Step 5: Track the global maximum

Maintain a variable storing the maximum star sum seen so far.

Update it after processing every node.

### Why it works

For a fixed center node, the best star graph is obtained by choosing the highest-value neighboring nodes.

Including any negative neighbor can only reduce the total sum, so excluding them is always optimal.

Since each node is evaluated independently as a center, and for each center we choose the optimal set of neighbors, the algorithm guarantees the global maximum star sum.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Solution:
    def maxStarSum(self, vals: List[int], edges: List[List[int]], k: int) -> int:
        graph = defaultdict(list)

        # Build adjacency lists using neighbor values
        for u, v in edges:
            graph[u].append(vals[v])
            graph[v].append(vals[u])

        answer = max(vals)

        for node in range(len(vals)):
            neighbors = graph[node]

            # Keep only positive neighbors
            positive_neighbors = [value for value in neighbors if value > 0]

            # Take largest values first
            positive_neighbors.sort(reverse=True)

            current_sum = vals[node]

            # Add at most k neighbors
            for i in range(min(k, len(positive_neighbors))):
                current_sum += positive_neighbors[i]

            answer = max(answer, current_sum)

        return answer
```

The implementation begins by constructing an adjacency list. Instead of storing neighboring node indices, it directly stores neighboring node values. This simplifies later computations because we only care about values when calculating star sums.

The variable `answer` is initialized with `max(vals)` because a valid star graph may contain only a single node with no neighbors.

For each node, the algorithm filters out negative neighbor values. This is an important optimization and correctness step because negative neighbors never improve the result.

The remaining positive neighbor values are sorted in descending order so the largest contributions are added first. Since we can choose at most `k` neighbors, we only add the first `k` values.

Finally, the global maximum is updated after processing each node.

## Go Solution

```go
package main

import "sort"

func maxStarSum(vals []int, edges [][]int, k int) int {
	n := len(vals)

	graph := make([][]int, n)

	// Build adjacency lists using neighbor values
	for _, edge := range edges {
		u, v := edge[0], edge[1]

		graph[u] = append(graph[u], vals[v])
		graph[v] = append(graph[v], vals[u])
	}

	answer := vals[0]

	for _, value := range vals {
		if value > answer {
			answer = value
		}
	}

	for node := 0; node < n; node++ {
		neighbors := []int{}

		// Keep only positive neighbors
		for _, value := range graph[node] {
			if value > 0 {
				neighbors = append(neighbors, value)
			}
		}

		// Sort descending
		sort.Sort(sort.Reverse(sort.IntSlice(neighbors)))

		currentSum := vals[node]

		limit := k
		if len(neighbors) < limit {
			limit = len(neighbors)
		}

		for i := 0; i < limit; i++ {
			currentSum += neighbors[i]
		}

		if currentSum > answer {
			answer = currentSum
		}
	}

	return answer
}
```

The Go implementation follows the same logic as the Python solution.

Go slices are used for adjacency lists. Since Go does not have Python's convenient descending sort built in, we use:

```go
sort.Sort(sort.Reverse(sort.IntSlice(neighbors)))
```

to sort in descending order.

Go integers are sufficient because the maximum possible sum fits comfortably within 32-bit integer range.

| --- | --- | --- |

The graph construction requires traversing every edge once, which costs `O(m)`.

For each node, we sort its positive neighbors. Across all nodes, the total sorting cost is bounded by `O(m log m)` in the worst case.

The adjacency lists store every undirected edge twice, giving `O(n + m)` space usage.

## Edge Cases

### Case 1: `k = 0`

When `k = 0`, no neighbors may be included in the star graph. A buggy implementation might still try to add neighbors accidentally or mishandle empty selections.

The implementation handles this naturally because the loop:

```
for i in range(min(k, len(positive_neighbors))):
```

executes zero iterations when `k = 0`.

### Case 2: All values are negative

If every node and neighbor has a negative value, the best solution is simply the largest single node value.

A common mistake is forcing the algorithm to include neighbors. Since our implementation filters out non-positive neighbors, no negative neighbor is ever added.

The initialization:

```
answer = max(vals)
```

guarantees correctness even if every star with neighbors is worse.

### Case 3: `k` larger than node degree

Some nodes may have fewer than `k` neighbors.

Attempting to access nonexistent neighbors could cause index errors.

The implementation safely handles this with:

```
min(k, len(positive_neighbors))
```

which ensures we only iterate over available neighbors.
