## Solution

## Problem Understanding

The problem asks us to compute, for every element in the array, the value of its second greater element to the right.

For an index `i`, we are not simply looking for the second larger value anywhere after it. The definition is stricter. We need to find an index `j` such that:

- `j > i`
- `nums[j] > nums[i]`
- Between `i` and `j`, there exists exactly one element greater than `nums[i]`

In other words, if we scan to the right of `nums[i]`, the first element greater than `nums[i]` is the first greater element, and the next greater element after that becomes the second greater element.

For example, consider:

```
nums = [2,4,0,9,6]
```

For `2`:

- `4` is the first greater element
- `9` is the second greater element

So the answer for index `0` is `9`.

For `4`:

- `9` is the first greater element
- `6` is the second greater element

So the answer is `6`.

The constraints are large:

- `nums.length` can be up to `100000`
- Values can be as large as `10^9`

These constraints immediately rule out inefficient quadratic solutions. Any algorithm with nested scans over the array will likely time out.

Several edge cases are important:

- Arrays with all equal values, such as `[3,3,3]`
- Strictly decreasing arrays, where no greater elements exist
- Strictly increasing arrays, where many second greater elements exist
- Duplicate values, since the condition requires strictly greater, not greater-or-equal
- Very small arrays, especially length `1`

The problem guarantees non-negative integers and a valid array length, so we do not need to handle malformed input.

## Approaches

### Brute Force Approach

The most direct approach is to process each index independently.

For every `i`, we scan all indices to the right and count how many elements are greater than `nums[i]`. The moment we encounter the second such element, we store it as the answer.

For example:

```
nums = [2,4,0,9,6]
```

For index `0`:

- `4 > 2`, count becomes `1`
- `0` is ignored
- `9 > 2`, count becomes `2`

Therefore the second greater element is `9`.

This approach is correct because it directly follows the problem definition. However, it is too slow for large inputs.

In the worst case, for every index we scan almost the entire remainder of the array.

This leads to `O(n^2)` time complexity, which is not feasible for `n = 100000`.

### Optimal Approach

The key observation is that we can process elements in a single left-to-right pass while maintaining information about elements waiting for:

- their first greater element
- their second greater element

This naturally suggests using monotonic stacks.

We use two stacks:

- `first_stack`, elements waiting for their first greater element
- `second_stack`, elements that already found their first greater element and are now waiting for their second greater element

As we process each new number:

- Any element in `second_stack` smaller than the current number has found its second greater element
- Any element in `first_stack` smaller than the current number has now found its first greater element, so we move them into `second_stack`
- Finally, we add the current index into `first_stack`

This allows every index to move through the stacks only a constant number of times.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Scan right side for every index |
| Optimal | O(n) | O(n) | Two monotonic stacks process each index once |

## Algorithm Walkthrough

### Optimal Algorithm Using Two Monotonic Stacks

1. Initialize the answer array with `-1`.

Every element defaults to `-1` because some indices may never find a second greater element.
2. Create two stacks.

The first stack stores indices waiting for their first greater element.

The second stack stores indices that already found their first greater element and are now waiting for their second greater element.
3. Iterate through the array from left to right.

Suppose the current element is `nums[i]`.
4. Resolve second greater elements first.

While the top of `second_stack` contains an index whose value is smaller than `nums[i]`, we have found that index's second greater element.

We assign:

```
answer[index] = nums[i]
```

Then remove that index from `second_stack`.
5. Move indices from the first stack to the second stack.

Any index in `first_stack` with value smaller than `nums[i]` has now found its first greater element.

Those indices should now begin waiting for their second greater element.

We temporarily collect them, then push them into `second_stack` in reverse order.

The reverse order preserves the monotonic property.
6. Push the current index into `first_stack`.

This index has not yet found any greater element.
7. Continue until the entire array has been processed.
8. Return the answer array.

### Why it works

The algorithm maintains two invariants:

- `first_stack` contains indices that have not yet encountered any greater element
- `second_stack` contains indices that have encountered exactly one greater element

When a new value arrives:

- Elements removed from `second_stack` have now seen exactly two greater elements, so the current value is their answer
- Elements moved from `first_stack` into `second_stack` have just encountered their first greater element

Because every index transitions through these states in order, the algorithm correctly identifies the second greater element for every position.

## Python Solution

```python
from typing import List

class Solution:
    def secondGreaterElement(self, nums: List[int]) -> List[int]:
        n = len(nums)
        answer = [-1] * n

        first_stack = []
        second_stack = []

        for i, value in enumerate(nums):

            # Resolve second greater elements
            while second_stack and nums[second_stack[-1]] < value:
                index = second_stack.pop()
                answer[index] = value

            # Move elements that found first greater
            temp = []

            while first_stack and nums[first_stack[-1]] < value:
                temp.append(first_stack.pop())

            while temp:
                second_stack.append(temp.pop())

            # Current index waits for first greater
            first_stack.append(i)

        return answer
```

The implementation closely follows the algorithm description.

The `answer` array is initialized with `-1` because some elements may never find a second greater element.

The `first_stack` tracks indices still waiting for their first greater element. The stack is monotonic decreasing by value.

The `second_stack` tracks indices that already found one greater element and are waiting for the second one.

For every new value:

- We first resolve indices in `second_stack`
- Then we move eligible indices from `first_stack` into `second_stack`
- Finally, we push the current index into `first_stack`

The temporary list is important because it preserves ordering when transferring indices between stacks.

Without reversing the order, the monotonic structure would break and future processing would become incorrect.

## Go Solution

```go
func secondGreaterElement(nums []int) []int {
	n := len(nums)

	answer := make([]int, n)
	for i := range answer {
		answer[i] = -1
	}

	firstStack := []int{}
	secondStack := []int{}

	for i, value := range nums {

		// Resolve second greater elements
		for len(secondStack) > 0 &&
			nums[secondStack[len(secondStack)-1]] < value {

			index := secondStack[len(secondStack)-1]
			secondStack = secondStack[:len(secondStack)-1]

			answer[index] = value
		}

		// Move elements that found first greater
		temp := []int{}

		for len(firstStack) > 0 &&
			nums[firstStack[len(firstStack)-1]] < value {

			index := firstStack[len(firstStack)-1]
			firstStack = firstStack[:len(firstStack)-1]

			temp = append(temp, index)
		}

		for len(temp) > 0 {
			index := temp[len(temp)-1]
			temp = temp[:len(temp)-1]

			secondStack = append(secondStack, index)
		}

		// Current index waits for first greater
		firstStack = append(firstStack, i)
	}

	return answer
}
```

The Go implementation mirrors the Python logic almost exactly.

Go slices are used as stacks. Appending pushes an element, and reslicing removes the top.

The answer array is initialized manually because Go defaults integers to `0`, while the problem requires `-1` when no second greater element exists.

There are no overflow concerns because the values fit comfortably inside Go's `int` type.

First resolve second greater elements.

`nums[0]=2 < 9`, so:

answer[0] = 9

- `0 < 9`, move index `2`
- `4 < 9`, move index `1`

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each index is pushed and popped at most once from each stack |
| Space | O(n) | Stacks and answer array may store all indices |

The time complexity is linear because every index transitions through the stacks only once. There are no nested rescans of the array.

The space complexity is linear because, in the worst case, both stacks together may contain almost all indices.

## Edge Cases

### Arrays With No Greater Elements

A strictly decreasing array such as:

```
[5,4,3,2,1]
```

is an important edge case because no element ever finds even a first greater element.

A buggy implementation might accidentally assign values due to incorrect stack transitions.

This implementation handles the case correctly because indices remain in `first_stack` until the end, and the answer array remains initialized to `-1`.

### Duplicate Values

The problem requires strictly greater values.

For example:

```
[3,3,3]
```

No element should count another `3` as greater.

This is a common source of bugs if `<=` is mistakenly used instead of `<`.

The implementation always checks:

```
nums[stack[-1]] < value
```

which correctly enforces strict inequality.

### Elements With Only One Greater Value

Consider:

```
[2,1,3]
```

The value `1` has exactly one greater element, namely `3`.

Since a second greater element does not exist, the answer must remain `-1`.

The implementation naturally handles this because indices only receive an answer when removed from `second_stack`. If an index never encounters a second greater value, its answer remains unchanged.
