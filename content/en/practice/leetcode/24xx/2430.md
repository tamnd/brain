## Solution

## Problem Understanding

The problem gives us a string `s` containing lowercase English letters. We want to completely delete the string using the maximum possible number of operations.

At every operation, we have two choices:

1. Delete the entire remaining string immediately.
2. Delete only the first `i` characters, but only if:

- `1 <= i <= len(s) / 2`
- The first `i` characters are exactly equal to the next `i` characters.

The second operation means we can only remove a prefix if it appears twice consecutively at the start of the current string.

The goal is not to minimize operations, but to maximize them. This is important because sometimes deleting a small valid prefix leads to more future operations.

For example, in `"aaaaa"`, we can repeatedly delete one `"a"` at a time because the next character is always also `"a"`. That produces five operations total, which is optimal.

The input size is the key challenge:

- `1 <= s.length <= 4000`

A string length of 4000 is large enough that naive substring comparisons inside recursive branching solutions become too slow. A brute-force search would repeatedly compare substrings and revisit the same suffix states many times.

This strongly suggests we need:

- Dynamic programming to avoid recomputation
- Efficient substring comparison, typically rolling hash or longest common prefix preprocessing

Several edge cases are important:

- Strings with no valid repeated prefix at all, such as `"abcdef"`, where the answer is always `1`
- Strings made entirely of the same character, such as `"aaaaa"`, which maximize the number of operations
- Strings where greedy deletion is not optimal
- Very long repetitive strings, where repeated substring comparisons could become quadratic or cubic if implemented poorly

## Approaches

### Brute Force Approach

A direct recursive solution tries every valid deletion.

Suppose we are currently at substring `s[i:]`. We check every possible length `length` such that:

- `i + 2 * length <= n`
- `s[i:i+length] == s[i+length:i+2*length]`

If the condition holds, we recursively solve the remaining suffix after deleting the first block.

The recurrence becomes:

```
dp[i] = max(
    1,
    1 + dp[i + length]
)
```

The `1` represents deleting the entire remaining string immediately.

This approach is correct because it explores every legal sequence of deletions and chooses the maximum.

However, naive substring comparison is expensive.

Each comparison can cost `O(length)`, and we may perform:

- `O(n)` states
- `O(n)` candidate lengths per state
- `O(n)` comparison cost

This leads to `O(n^3)` time complexity, which is too slow for `n = 4000`.

### Key Insight

The expensive part is repeatedly comparing substrings.

Instead of comparing character by character every time, we can preprocess the longest common prefix between every pair of suffixes.

Define:

```
lcp[i][j]
```

as the length of the longest common prefix between:

```
s[i:]
and
s[j:]
```

Then two adjacent substrings of length `length` are equal if:

```
lcp[i][i + length] >= length
```

This converts substring comparison from `O(length)` into `O(1)`.

We can compute all LCP values using dynamic programming from the end of the string:

```
if s[i] == s[j]:
    lcp[i][j] = 1 + lcp[i+1][j+1]
else:
    lcp[i][j] = 0
```

Once LCP preprocessing is complete, the main DP becomes efficient.

## Algorithm Walkthrough

### Step 1: Define the DP State

Let:

```
dp[i]
```

represent the maximum number of operations needed to delete the substring:

```
s[i:]
```

The final answer will be:

```
dp[0]
```

### Step 2: Precompute Longest Common Prefix Values

We create a 2D array:

```
lcp[i][j]
```

which stores the length of the common prefix between:

```
s[i:]
and
s[j:]
```

We fill this table from the end toward the beginning.

If:

```
s[i] == s[j]
```

then:

```
lcp[i][j] = 1 + lcp[i+1][j+1]
```

Otherwise:

```
lcp[i][j] = 0
```

This works because matching prefixes extend only if the current characters match.

### Step 3: Initialize DP

Every substring can always be deleted entirely in one move.

So initially:

```
dp[i] = 1
```

for all positions.

### Step 4: Try All Valid Prefix Lengths

For every starting index `i`, we test every possible deletion length `length`.

The two adjacent blocks are:

```
s[i:i+length]
s[i+length:i+2*length]
```

They are equal if:

```
lcp[i][i + length] >= length
```

If they match, we can delete the first block and continue solving from:

```
i + length
```

So:

```
dp[i] = max(dp[i], 1 + dp[i + length])
```

### Step 5: Return the Result

After processing all states, return:

```
dp[0]
```

### Why it works

The algorithm works because every valid operation depends only on the current remaining suffix of the string. The DP state fully captures that information.

The LCP table guarantees correct substring equality checks in constant time. Since the recurrence explores every valid deletion choice and takes the maximum, the computed value is the optimal number of operations.

## Python Solution

```python
class Solution:
    def deleteString(self, s: str) -> int:
        n = len(s)

        # lcp[i][j] = longest common prefix length of s[i:] and s[j:]
        lcp = [[0] * (n + 1) for _ in range(n + 1)]

        # Build LCP table from bottom-right to top-left
        for i in range(n - 1, -1, -1):
            for j in range(n - 1, i, -1):
                if s[i] == s[j]:
                    lcp[i][j] = 1 + lcp[i + 1][j + 1]

        # dp[i] = maximum operations to delete s[i:]
        dp = [1] * n

        # Process from back to front
        for i in range(n - 1, -1, -1):

            # Try every possible prefix length
            for length in range(1, (n - i) // 2 + 1):

                # Check whether the two adjacent substrings are equal
                if lcp[i][i + length] >= length:
                    dp[i] = max(dp[i], 1 + dp[i + length])

        return dp[0]
```

The implementation begins by allocating the LCP table. The extra row and column simplify boundary handling because accessing `lcp[i+1][j+1]` is always safe.

The nested reverse loops fill the LCP table. We process from the end because each state depends on diagonally lower-right values.

Next, the DP array is initialized with `1`, because every substring can always be removed entirely in one operation.

The outer DP loop iterates backward so that when we compute `dp[i]`, all future states `dp[i + length]` are already available.

For each possible deletion length, the LCP table allows constant-time equality checking. If the adjacent substrings match, we update the best answer for that suffix.

Finally, `dp[0]` represents the optimal answer for the whole string.

## Go Solution

```go
func deleteString(s string) int {
	n := len(s)

	// lcp[i][j] = longest common prefix length of s[i:] and s[j:]
	lcp := make([][]int, n+1)
	for i := range lcp {
		lcp[i] = make([]int, n+1)
	}

	// Build LCP table
	for i := n - 1; i >= 0; i-- {
		for j := n - 1; j > i; j-- {
			if s[i] == s[j] {
				lcp[i][j] = 1 + lcp[i+1][j+1]
			}
		}
	}

	// dp[i] = maximum operations to delete s[i:]
	dp := make([]int, n)

	for i := 0; i < n; i++ {
		dp[i] = 1
	}

	// Fill DP from back to front
	for i := n - 1; i >= 0; i-- {

		for length := 1; length <= (n-i)/2; length++ {

			if lcp[i][i+length] >= length {
				if 1+dp[i+length] > dp[i] {
					dp[i] = 1 + dp[i+length]
				}
			}
		}
	}

	return dp[0]
}
```

The Go implementation mirrors the Python solution closely.

The primary difference is explicit allocation of the 2D slices for the LCP table. Go does not support automatic multidimensional array initialization like Python lists.

The DP logic is otherwise identical. Integer overflow is not a concern because the maximum answer is at most `4000`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n²) | LCP preprocessing and DP each use nested loops |
| Space | O(n²) | The LCP table stores all pairwise suffix matches |

The dominant cost comes from the `n × n` LCP table. The DP itself is also quadratic because each index tests multiple possible deletion lengths.

Despite the quadratic space usage, this is acceptable for `n <= 4000`.

## Edge Cases

### Single Character String

A string of length `1` cannot satisfy the repeated-prefix condition because there is no room for two adjacent substrings.

For example:

```
"a"
```

The only valid move is deleting the whole string. The implementation handles this naturally because the inner loop never executes, leaving `dp[i] = 1`.

### Completely Non-Repetitive Strings

Strings like:

```
"abcdef"
```

have no valid repeated adjacent prefixes.

A buggy implementation might incorrectly identify partial overlaps as matches. The LCP table prevents this by requiring the common prefix length to be at least the entire candidate deletion length.

Every state remains `1`, which correctly represents deleting the remaining substring immediately.

### Highly Repetitive Strings

Strings such as:

```
"aaaaaaaaaa"
```

create many valid deletion paths.

This is dangerous for naive recursion because the branching factor becomes very large. Without memoization or efficient substring comparison, performance degrades rapidly.

The DP solution handles this efficiently because:

- Every suffix state is computed once
- Every substring comparison becomes `O(1)` using the LCP table

This keeps the total runtime quadratic instead of exponential or cubic.
