## Solution

## Problem Understanding

The problem gives us an array of positive integers and allows us to repeatedly merge any two adjacent elements into their sum. Each merge operation reduces the array length by one.

Our goal is to transform the array into a palindrome using the minimum number of merge operations.

A palindrome is an array that reads the same from left to right and right to left. For example:

- `[1,2,3,2,1]` is a palindrome
- `[4,5,5,4]` is a palindrome
- `[1,2,3]` is not a palindrome

The important detail is that we are only allowed to merge adjacent elements. After merging two adjacent values, they are replaced by a single value equal to their sum.

For example:

```
[1,2,3,1]
```

If we merge `2` and `3`, the array becomes:

```
[1,5,1]
```

which is now a palindrome.

The task is not to produce the palindrome itself. We only need to compute the minimum number of merge operations required.

The constraints are important:

- `1 <= nums.length <= 10^5`
- `1 <= nums[i] <= 10^6`

An array size of `10^5` means we need a highly efficient algorithm, ideally linear time. Any brute-force search or repeated array reconstruction approach would be too slow.

The fact that all numbers are positive is also extremely important. Since every merge increases the value of a segment, sums grow monotonically. This property makes a greedy two-pointer strategy possible.

Several edge cases are important to consider:

- Arrays that are already palindromes should return `0`
- Arrays of length `1` are automatically palindromes
- Arrays where every merge must eventually collapse into one element
- Cases where one side repeatedly needs merges before matching the other side
- Large arrays where repeated shifting or rebuilding would be inefficient

## Approaches

### Brute Force Approach

A brute-force solution would try every possible merge sequence and search for the minimum number of operations that leads to a palindrome.

At each step, we could:

1. Choose any adjacent pair
2. Merge them
3. Recursively continue until the array becomes a palindrome

This approach is correct because it explores all possible valid operations and therefore cannot miss the optimal answer.

However, it is computationally infeasible. Every state can branch into many new states, and the number of possible merge sequences grows exponentially with the array size.

Even for relatively small arrays, the number of possibilities becomes enormous. With `n = 10^5`, this approach is completely impossible.

### Optimal Greedy Two-Pointer Approach

The key observation is that we only care about making the left and right sides equal.

Suppose we compare:

```
nums[left] and nums[right]
```

There are three possibilities:

1. They are equal

Then these positions already satisfy the palindrome condition, so we move both pointers inward.
2. Left side is smaller

Since all values are positive, the only way to increase the left side is to merge it with its neighbor.
3. Right side is smaller

Symmetrically, we merge elements on the right side.

This greedy strategy works because:

- A smaller side must eventually grow
- Merging on the larger side would only make the imbalance worse
- Positive integers guarantee merged sums only increase

We never need to reconsider previous choices, which allows a linear-time solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Tries all merge sequences |
| Optimal | O(n) | O(1) | Greedy two-pointer merging |

## Algorithm Walkthrough

1. Initialize two pointers:

- `left = 0`
- `right = len(nums) - 1`

These pointers represent the current elements we are trying to match.
2. Initialize an operation counter:

- `operations = 0`
3. While `left < right`, compare the values at both ends.
4. If `nums[left] == nums[right]`, no merge is needed for these positions.

Move both pointers inward:

```
left += 1
right -= 1
```
5. If `nums[left] < nums[right]`, the left side is too small.

Merge the left element with the next element:

```
nums[left + 1] += nums[left]
left += 1
operations += 1
```

This effectively combines two adjacent elements into one larger value.
6. If `nums[left] > nums[right]`, the right side is too small.

Merge the right element with the previous element:

```
nums[right - 1] += nums[right]
right -= 1
operations += 1
```
7. Continue until the pointers meet or cross.
8. Return the total number of operations.

### Why it works

The algorithm maintains the invariant that everything outside the current `[left, right]` window has already been matched into a valid palindrome structure.

If one side is smaller, that side must eventually absorb more elements to reach the other side's value. Because all numbers are positive, merging only increases sums. Therefore, greedily merging the smaller side is always safe and never prevents an optimal solution.

This guarantees that every merge operation is necessary and contributes directly toward balancing the palindrome.

## Python Solution

```python
from typing import List

class Solution:
    def minimumOperations(self, nums: List[int]) -> int:
        left = 0
        right = len(nums) - 1
        operations = 0

        while left < right:
            if nums[left] == nums[right]:
                left += 1
                right -= 1

            elif nums[left] < nums[right]:
                nums[left + 1] += nums[left]
                left += 1
                operations += 1

            else:
                nums[right - 1] += nums[right]
                right -= 1
                operations += 1

        return operations
```

The implementation directly follows the greedy two-pointer algorithm.

We begin with pointers at both ends of the array. During each iteration, we compare the current left and right values.

When both values are equal, they already satisfy the palindrome condition, so we move inward.

When the left value is smaller, we merge it into its neighbor on the right. Instead of physically removing elements from the array, we simply accumulate the sum into the adjacent element and advance the pointer. This simulates the merge efficiently without expensive array operations.

The same logic applies symmetrically when the right side is smaller.

The operation counter increases every time we perform a merge.

Because each pointer moves inward at most `n` times, the algorithm runs in linear time.

## Go Solution

```go
func minimumOperations(nums []int) int {
    left := 0
    right := len(nums) - 1
    operations := 0

    for left < right {
        if nums[left] == nums[right] {
            left++
            right--
        } else if nums[left] < nums[right] {
            nums[left+1] += nums[left]
            left++
            operations++
        } else {
            nums[right-1] += nums[right]
            right--
            operations++
        }
    }

    return operations
}
```

The Go implementation mirrors the Python solution closely.

Go slices are mutable, so we can modify elements in place exactly as we do in Python.

No special handling is needed for empty arrays because the constraints guarantee at least one element.

Integer overflow is not an issue for typical LeetCode constraints because Go's `int` type easily handles the possible sums produced by the input limits.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each pointer moves inward at most once per iteration |
| Space | O(1) | Only a few variables are used |

The algorithm is linear because every operation advances either the left pointer or the right pointer. No pointer ever moves backward, so the total number of iterations is bounded by `n`.

The solution uses constant extra space because all merging is simulated directly inside the input array.

## Edge Cases

### Single Element Array

An array with one element is always a palindrome because it reads the same forward and backward.

A careless implementation might still attempt pointer comparisons or merges unnecessarily. In this solution, the loop condition is `left < right`, so a single-element array immediately returns `0`.

### Already Palindromic Arrays

Arrays such as:

```
[1,2,3,2,1]
```

require no operations.

A buggy implementation might accidentally merge equal values or continue processing after finding symmetry. This algorithm correctly recognizes equality at each step and simply moves both pointers inward without modifying the array.

### Repeated Merging on One Side

Consider:

```
[1,2,3,10]
```

The left side must repeatedly merge to catch up with the large right value.

A naive strategy that merges arbitrarily could perform unnecessary operations or fail to minimize merges. The greedy rule of always merging the smaller side guarantees the minimum number of operations because only the smaller cumulative sum needs to grow.

### Entire Array Collapses Into One Element

Some arrays cannot form a multi-element palindrome and must eventually become a single value.

For example:

```
[1,2,3,4]
```

After enough merges, the array effectively becomes:

```
[10]
```

which is trivially a palindrome.

The algorithm naturally handles this because the pointers eventually meet after repeated merges.
