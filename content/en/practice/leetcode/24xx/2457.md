# LeetCode 2457 - Minimum Addition to Make Integer Beautiful

## Problem Understanding

The problem gives us two positive integers, `n` and `target`.

An integer is considered beautiful when the sum of its digits is less than or equal to `target`. Our task is to find the minimum non-negative integer `x` such that:

$$\text{digitSum}(n + x) \leq target$$

We are not asked to construct the beautiful number directly. Instead, we must determine the smallest amount we need to add to `n` so that the resulting number becomes beautiful.

For example, if `n = 467` and `target = 6`, the digit sum of `467` is:

$$4 + 6 + 7 = 17$$

Since `17 > 6`, the number is not beautiful. By adding `33`, we get `500`, whose digit sum is:

$$5 + 0 + 0 = 5$$

This satisfies the condition, and `33` is the minimum valid addition.

The constraints are important:

- `1 <= n <= 10^12`
- `1 <= target <= 150`

The value of `n` can be very large, so iterating one number at a time is not feasible. A brute-force solution that repeatedly checks `n + 1`, `n + 2`, and so on could require billions of iterations in the worst case.

The problem also guarantees that a solution always exists. Since we can always eventually reach a power of 10, whose digit sum is `1`, we know a valid answer must exist.

Several edge cases are important:

- `n` may already be beautiful, in which case the answer is `0`.
- Carry propagation can affect multiple digits, such as turning `999` into `1000`.
- The minimum addition may require modifying higher place values, not just the last digit.
- Large values near `10^12` require careful handling of integer arithmetic.

## Approaches

### Brute Force Approach

The most direct solution is to try every possible addition starting from `0`.

For each candidate `x`, we compute:

$$n + x$$

Then we calculate the digit sum of the resulting number. If the digit sum is less than or equal to `target`, we return `x`.

This approach is correct because it checks all possible additions in increasing order, so the first valid one must be minimal.

However, this method is far too slow. Consider a case where we need to jump from a number like `999999999999` to `1000000000000`. The number of iterations could become enormous.

### Optimal Greedy Approach

The key observation is that the optimal solution always comes from rounding part of the number upward.

Suppose a number is not beautiful. We can improve the digit sum by converting some suffix digits into zeros through carrying.

For example:

- `467 -> 470`
- `470 -> 500`
- `999 -> 1000`

Every time we round upward at a position, the lower digits become zero. This often dramatically reduces the digit sum.

The greedy strategy is:

1. Start from the least significant digit.
2. Round the current digit upward to the next multiple of 10.
3. Check whether the resulting number is beautiful.
4. If not, continue to the next digit position.

Because each step makes the smallest possible change at that digit position, the first valid result is guaranteed to be minimal.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(answer × log n) | O(1) | Checks every possible addition one by one |
| Optimal Greedy | O((log n)^2) | O(1) | Progressively rounds digits upward |

## Algorithm Walkthrough

### Step 1: Check Whether `n` Is Already Beautiful

Compute the digit sum of `n`.

If the digit sum is already less than or equal to `target`, return `0` immediately because no addition is necessary.

### Step 2: Initialize Variables

We maintain:

- `current`, the evolving number after rounding operations
- `multiplier`, representing the current place value (`10`, `100`, `1000`, and so on)

Initially:

- `current = n`
- `multiplier = 10`

### Step 3: Round Up at the Current Digit Position

At each iteration, we round `current` upward to the nearest multiple of `multiplier`.

This is done using:

```
current = ((current // multiplier) + 1) * multiplier
```

For example:

- If `current = 467` and `multiplier = 10`

- `467 -> 470`
- Then with `multiplier = 100`

- `470 -> 500`

This operation zeroes out all lower digits.

### Step 4: Check the Digit Sum Again

After rounding, compute the digit sum of `current`.

If the digit sum is now less than or equal to `target`, we have found the smallest valid beautiful number reachable through greedy rounding.

The answer is:

```
current - n
```

### Step 5: Move to the Next Higher Digit

If the number is still not beautiful:

```
multiplier *= 10
```

This allows us to round at the next place value.

### Step 6: Repeat Until Successful

Eventually, we must reach a beautiful number because powers of 10 have digit sum `1`.

## Why It Works

The algorithm works because each rounding operation produces the smallest possible increase that eliminates a suffix of digits.

If changing the last digit is insufficient, then any valid solution must modify a higher digit. The greedy process explores these possibilities in increasing order of impact.

By always performing the smallest upward rounding at each digit position, we guarantee that the first beautiful number found corresponds to the minimum possible addition.

## Python Solution

```python
class Solution:
    def makeIntegerBeautiful(self, n: int, target: int) -> int:
        def digit_sum(value: int) -> int:
            total = 0
            
            while value > 0:
                total += value % 10
                value //= 10
            
            return total
        
        if digit_sum(n) <= target:
            return 0
        
        current = n
        multiplier = 10
        
        while digit_sum(current) > target:
            current = ((current // multiplier) + 1) * multiplier
            multiplier *= 10
        
        return current - n
```

The helper function `digit_sum` computes the sum of digits using repeated modulo and division operations.

We first check whether the original number is already beautiful. If so, we immediately return `0`.

The variable `current` stores the progressively rounded value. The variable `multiplier` determines which digit position we are currently modifying.

Inside the loop, we round `current` upward to the nearest multiple of `multiplier`. This removes lower digits by turning them into zeros.

After each rounding step, we recompute the digit sum. Once the digit sum becomes valid, we return the difference between the new number and the original number.

## Go Solution

```go
func makeIntegerBeautiful(n int64, target int) int64 {
	digitSum := func(value int64) int {
		sum := 0

		for value > 0 {
			sum += int(value % 10)
			value /= 10
		}

		return sum
	}

	if digitSum(n) <= target {
		return 0
	}

	current := n
	var multiplier int64 = 10

	for digitSum(current) > target {
		current = ((current / multiplier) + 1) * multiplier
		multiplier *= 10
	}

	return current - n
}
```

The Go implementation follows the same logic as the Python version.

The primary difference is explicit integer typing. Since `n` can reach `10^12`, we use `int64` throughout the computation to avoid overflow.

Go does not support nested named functions as cleanly as Python, so we use an inline function literal for `digitSum`.

Otherwise, the control flow and rounding logic remain identical.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O((log n)^2) | We process at most one iteration per digit, and each digit sum computation takes O(log n) |
| Space | O(1) | Only a few integer variables are used |

The number of iterations is proportional to the number of digits in `n`. During each iteration, we compute the digit sum, which itself scans all digits of the current number. Therefore, the total complexity becomes:

$$O((\log n)^2)$$

Since `n <= 10^{12}`, this is extremely efficient.

## Edge Cases

### The Number Is Already Beautiful

A common mistake is continuing to modify the number even when its digit sum already satisfies the condition.

For example:

```
n = 123
target = 6
```

The digit sum is already:

$$1 + 2 + 3 = 6$$

The correct answer is `0`.

The implementation handles this with an early return before entering the loop.

### Cascading Carry Operations

Numbers like `999` can trigger multiple carry operations:

```
999 -> 1000
```

A naive implementation that only adjusts one digit may fail to propagate carries correctly.

The rounding formula:

```
((current // multiplier) + 1) * multiplier
```

naturally handles carry propagation across all digits.

### Large Numbers Near the Constraint Limit

Since `n` can be as large as `10^12`, implementations using smaller integer types may overflow.

The Go solution explicitly uses `int64`, ensuring all intermediate computations remain safe.

The algorithm also remains efficient because it only processes digit positions, not every integer value between `n` and the answer.
