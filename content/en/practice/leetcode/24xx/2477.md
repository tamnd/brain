# LeetCode 2477 - Minimum Fuel Cost to Report to the Capital

## Problem Understanding

This problem describes a country road network that forms a tree. A tree is a connected graph with no cycles, which means there is exactly one simple path between any two cities. The capital city is always city `0`.

Each city has exactly one representative who must travel to the capital for a meeting. Every city also has a car, and each car can carry at most `seats` people at a time. Representatives are allowed to switch cars during the journey, which means people can combine into fewer vehicles as they move closer to the capital.

The goal is to compute the minimum total amount of fuel required for all representatives to reach city `0`. Traveling across one road consumes exactly one liter of fuel, regardless of how many people are inside the car, as long as the number does not exceed the seat capacity.

The input consists of:

- `roads`, a list of undirected edges describing the tree
- `seats`, the maximum capacity of every car

The output is a single integer representing the minimum fuel cost.

The constraints are important:

- The number of cities can be as large as `10^5`
- The graph is guaranteed to be a valid tree
- A brute-force simulation of all possible ride-sharing combinations is infeasible

These constraints strongly suggest that the solution must run in approximately linear time.

Several edge cases are important:

- A country with only the capital city, where `roads = []`
- A seat count of `1`, where every representative must effectively travel independently
- A highly unbalanced tree, which can expose recursion or traversal mistakes
- Large seat counts, where many representatives can combine into very few cars

The problem guarantees that the graph is connected and acyclic, so we never need to handle disconnected components or invalid structures.

## Approaches

### Brute Force Approach

A brute-force solution would attempt to simulate every possible way representatives could combine into cars while traveling toward the capital. For each city, we could explore different merge strategies and compute the resulting fuel cost.

This approach is theoretically correct because it considers every valid transportation configuration. However, the number of possible combinations grows exponentially as the number of cities increases. Since `n` can be as large as `100000`, such a solution is completely impractical.

Even attempting dynamic programming over subsets or states would be far too expensive.

### Key Insight

The crucial observation is that every representative must eventually travel along the unique path from their city to the capital.

Because the graph is a tree, every subtree must send all its representatives upward through exactly one edge, the edge connecting the subtree to its parent.

This means the problem becomes much simpler:

- For every subtree, determine how many representatives it contains
- Compute how many cars are needed to move those representatives one edge upward
- Add that fuel cost to the total

If a subtree contains `people` representatives, and each car can carry `seats` people, then the minimum number of cars required is:

$\left\lceil \frac{people}{seats} \right\rceil$

Each of those cars must cross exactly one edge to move toward the capital, so this directly gives the fuel cost contribution for that subtree.

A postorder Depth-First Search works perfectly because we need child subtree sizes before computing the parent result.

## Algorithm Walkthrough

### Step 1: Build the Adjacency List

Convert the edge list into an adjacency list representation of the tree.

Since the graph is undirected, every road `[a, b]` is added in both directions.

This structure allows efficient traversal of neighboring cities.

### Step 2: Perform a DFS from the Capital

Start a DFS traversal from city `0`.

The DFS will return the number of representatives in the current subtree.

Every city contributes one representative, itself.

### Step 3: Process Child Subtrees First

For every neighboring city that is not the parent:

1. Recursively compute the number of representatives in that child's subtree
2. Determine how many cars are needed to move those representatives upward
3. Add the fuel cost for that edge
4. Add the subtree representative count to the current node

This is a postorder traversal because children are processed before parents.

### Step 4: Compute Cars Needed

Suppose a subtree contains `subtree_people` representatives.

The minimum number of cars required is:

$\left\lceil \frac{subtree\_people}{seats} \right\rceil$

This can be implemented efficiently using integer arithmetic:

```
(subtree_people + seats - 1) // seats
```

Each car crossing the edge costs one liter of fuel.

### Step 5: Skip Fuel Calculation for the Capital

The capital does not need to travel anywhere, so we never add fuel cost for city `0`.

Only edges leading upward toward the capital contribute fuel.

### Why it Works

The algorithm works because every representative must traverse the unique path from their city to the capital. For every subtree, all representatives must cross exactly one edge to move upward.

The optimal strategy is always to fill cars as efficiently as possible before crossing that edge. Since representatives can freely switch cars, the exact routing inside the subtree does not matter. Only the total number of representatives matters.

The DFS guarantees that when processing a node, we already know the complete representative count of each child subtree. This allows us to compute the exact minimum number of required cars for every edge independently.

Because every edge is processed exactly once, the total fuel cost is optimal.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Solution:
    def minimumFuelCost(self, roads: List[List[int]], seats: int) -> int:
        n = len(roads) + 1

        graph = defaultdict(list)

        for a, b in roads:
            graph[a].append(b)
            graph[b].append(a)

        total_fuel = 0

        def dfs(node: int, parent: int) -> int:
            nonlocal total_fuel

            representatives = 1

            for neighbor in graph[node]:
                if neighbor == parent:
                    continue

                subtree_people = dfs(neighbor, node)

                cars_needed = (subtree_people + seats - 1) // seats

                total_fuel += cars_needed

                representatives += subtree_people

            return representatives

        dfs(0, -1)

        return total_fuel
```

The implementation begins by constructing an adjacency list for the tree. Since the roads are bidirectional, each edge is inserted in both directions.

The `dfs` function computes the total number of representatives in the current subtree. Every node contributes one representative initially.

For every child subtree, the recursive call returns the number of representatives below that child. Once we know that number, we calculate how many cars are required to move those people upward by one edge.

The fuel contribution is added immediately because every required car must cross the edge connecting the child subtree to the current node.

Finally, the subtree representative count is returned upward so parent nodes can continue the aggregation.

The traversal starts at the capital city `0`, and the final accumulated fuel value is returned.

## Go Solution

```go
package main

func minimumFuelCost(roads [][]int, seats int) int64 {
	n := len(roads) + 1

	graph := make([][]int, n)

	for _, road := range roads {
		a := road[0]
		b := road[1]

		graph[a] = append(graph[a], b)
		graph[b] = append(graph[b], a)
	}

	var totalFuel int64

	var dfs func(int, int) int

	dfs = func(node int, parent int) int {
		representatives := 1

		for _, neighbor := range graph[node] {
			if neighbor == parent {
				continue
			}

			subtreePeople := dfs(neighbor, node)

			carsNeeded := (subtreePeople + seats - 1) / seats

			totalFuel += int64(carsNeeded)

			representatives += subtreePeople
		}

		return representatives
	}

	dfs(0, -1)

	return totalFuel
}
```

The Go implementation follows exactly the same logic as the Python version.

One important difference is that the return type must be `int64`, because the total fuel cost can exceed the range of a standard 32-bit integer when the tree is large.

The adjacency list is implemented using a slice of integer slices instead of a hash map. Since city indices range from `0` to `n - 1`, this is efficient and avoids unnecessary hashing overhead.

Go also requires explicit recursive function declaration using a variable assignment before definition.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every node and edge is visited once |
| Space | O(n) | Adjacency list and recursion stack |

The DFS traverses each edge exactly twice, once in each direction. Since a tree with `n` nodes contains `n - 1` edges, the total traversal cost is linear.

The adjacency list requires linear space, and the recursion stack can also reach linear depth in the worst case of a skewed tree.

## Edge Cases

### Single City Country

When `roads = []`, the country contains only the capital city. No representatives need to travel because the capital's representative is already at the destination.

This case can easily cause bugs if the implementation assumes at least one edge exists. The solution handles it naturally because the DFS starts at node `0` and finds no neighbors, so the fuel total remains `0`.

### Seat Count Equal to One

When `seats = 1`, no car can carry additional passengers. Every representative effectively travels independently.

This is an important stress case because it maximizes fuel usage. The formula:

$\left\lceil \frac{people}{1} \right\rceil = people$

correctly forces one car per representative.

### Highly Unbalanced Trees

A tree shaped like a linked list creates the deepest possible recursion structure.

Naive implementations may incorrectly aggregate representatives or double-count fuel along the chain. The DFS solution correctly processes the tree bottom-up, ensuring every subtree contributes exactly once to the fuel total.

### Very Large Seat Counts

When `seats` is larger than the number of representatives in a subtree, all representatives can share a single car.

The ceiling division formula correctly handles this case, producing exactly one required car instead of overestimating fuel usage.
