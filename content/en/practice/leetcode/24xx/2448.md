# LeetCode 2448 - Minimum Cost to Make Array Equal

## Problem Understanding

The problem gives us two arrays, `nums` and `cost`, both of length `n`.

Each value in `nums[i]` represents a number we want to modify. Each value in `cost[i]` represents how expensive it is to change `nums[i]` by one unit.

We may repeatedly perform operations where we either increase or decrease an element by exactly `1`. Every unit change on index `i` costs `cost[i]`.

Our goal is to make every number in `nums` equal while spending the minimum total cost.

For example, if:

```
nums = [1, 3]
cost = [10, 1]
```

then changing `1` into `3` costs `20`, because we must increase it twice and each increase costs `10`.

However, changing `3` into `1` costs only `2`, because decreasing twice costs `1` each time.

This demonstrates the key idea of the problem: not all numbers are equally expensive to move. The optimal target value depends heavily on the weights in `cost`.

The constraints are large:

- `n` can be up to `100000`
- `nums[i]` and `cost[i]` can each be up to `1000000`

These limits immediately rule out many naive approaches. Any algorithm significantly worse than `O(n log n)` will likely time out.

The problem also guarantees that the final answer fits within `2^53 - 1`, but intermediate calculations can still become large, so we must use 64-bit integer arithmetic.

Several edge cases are important:

- All numbers may already be equal
- One element may have an extremely large cost, forcing the optimal answer toward that value
- Multiple target values may produce the same minimum cost
- The optimal target might not exist in the original array
- Large values require careful overflow handling

## Approaches

### Brute Force Approach

A straightforward solution is to try every possible target value and compute the total cost required to transform every element into that target.

If we choose a target `x`, then the total cost is:

```
sum(abs(nums[i] - x) * cost[i])
```

We could iterate through all target values from `min(nums)` to `max(nums)` and compute the total cost each time.

This works because the optimal target must lie somewhere within that range.

However, the range of values can be as large as `10^6`. For each target, we would scan all `n` elements.

That leads to:

```
O(n * range)
```

In the worst case:

```
100000 * 1000000 = 10^11
```

which is far too slow.

### Key Insight

The cost function is:

$$f(x) = \sum |nums[i] - x| \cdot cost[i]$$

This is a weighted absolute distance function.

A fundamental property of weighted absolute deviations is that the minimum occurs at the **weighted median**.

The weighted median is the value where the total weight on the left side and right side are both at most half of the total weight.

Intuitively:

- Numbers with larger `cost[i]` are more expensive to move
- Therefore, the optimal target gets pulled toward heavily weighted values
- The weighted median balances these movement costs optimally

Once we sort `(nums[i], cost[i])` pairs by value, we can find the weighted median efficiently.

After finding the weighted median, we compute the total transformation cost in one final pass.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × range) | O(1) | Tries every target value |
| Optimal | O(n log n) | O(n) | Uses weighted median after sorting |

## Algorithm Walkthrough

### Optimal Algorithm

1. Pair each number with its corresponding cost.

We combine the arrays into pairs:

```
(nums[i], cost[i])
```

This makes it easier to sort while preserving the relationship between values and weights.
2. Sort the pairs by the number value.

The weighted median logic depends on processing values in sorted order.
3. Compute the total weight.

We calculate:

```
total_cost_weight = sum(cost)
```

This represents the total influence across all numbers.
4. Find the weighted median.

We scan the sorted pairs while maintaining a running sum of weights.

The first value where:

```
prefix_weight * 2 >= total_weight
```

is the weighted median.

This works because the weighted median balances the movement cost on both sides.
5. Compute the final transformation cost.

Once the target value is known, we compute:

```
abs(nums[i] - target) * cost[i]
```

for every element and sum the results.
6. Return the total.

This value is the minimum possible cost.

### Why it works

The total cost function is convex because it is a sum of weighted absolute value functions. Convex functions have a global minimum where the slope changes sign.

For weighted absolute deviations, that minimum occurs at the weighted median. Moving the target left or right from the weighted median increases the total weighted distance, so the weighted median guarantees the minimum possible cost.

## Python Solution

```python
from typing import List

class Solution:
    def minCost(self, nums: List[int], cost: List[int]) -> int:
        pairs = sorted(zip(nums, cost))
        
        total_weight = sum(cost)
        current_weight = 0
        target = 0
        
        for value, weight in pairs:
            current_weight += weight
            
            if current_weight * 2 >= total_weight:
                target = value
                break
        
        total_cost = 0
        
        for value, weight in zip(nums, cost):
            total_cost += abs(value - target) * weight
        
        return total_cost
```

The implementation begins by combining the arrays into `(value, weight)` pairs and sorting them by value.

Next, the code computes the total weight across all elements. This allows us to determine when we have crossed the halfway point while scanning the sorted array.

The loop that finds the weighted median maintains a running prefix sum of weights. As soon as the prefix reaches at least half of the total weight, the current value becomes the optimal target.

Finally, the algorithm computes the total weighted movement cost using the chosen target value.

The implementation uses integer arithmetic throughout, which is important because the total cost may become very large.

## Go Solution

```go
package main

import (
	"sort"
)

func minCost(nums []int, cost []int) int64 {
	type Pair struct {
		value  int
		weight int
	}

	n := len(nums)

	pairs := make([]Pair, n)

	var totalWeight int64

	for i := 0; i < n; i++ {
		pairs[i] = Pair{
			value:  nums[i],
			weight: cost[i],
		}

		totalWeight += int64(cost[i])
	}

	sort.Slice(pairs, func(i, j int) bool {
		return pairs[i].value < pairs[j].value
	})

	var currentWeight int64
	target := 0

	for _, pair := range pairs {
		currentWeight += int64(pair.weight)

		if currentWeight*2 >= totalWeight {
			target = pair.value
			break
		}
	}

	var totalCost int64

	for i := 0; i < n; i++ {
		diff := nums[i] - target

		if diff < 0 {
			diff = -diff
		}

		totalCost += int64(diff) * int64(cost[i])
	}

	return totalCost
}
```

The Go solution follows the exact same logic as the Python implementation.

The main difference is careful handling of integer overflow. Since costs can become very large, the implementation uses `int64` for cumulative sums and the final answer.

Go also requires an explicit struct type for storing `(value, weight)` pairs, and sorting is performed using `sort.Slice`.

## Edge Cases

### All elements are already equal

If every number in `nums` is identical, then no operations are required. A buggy implementation might still perform unnecessary calculations or incorrectly shift the target value.

This implementation handles the case naturally because the weighted median equals the existing value, producing zero total distance.

### One element has an extremely large cost

Consider:

```
nums = [1, 100]
cost = [1000000, 1]
```

Moving the first element is extremely expensive, so the optimal solution should keep the target near `1`.

A naive implementation that uses the ordinary median instead of the weighted median would produce the wrong answer.

The weighted median logic correctly prioritizes heavily weighted elements.

### Large integer calculations

The expression:

```
abs(nums[i] - target) * cost[i]
```

can become very large.

For example:

```
1000000 * 1000000 = 10^12
```

Summing many such values requires 64-bit arithmetic.

The Go solution explicitly uses `int64`, and Python automatically handles large integers safely.
