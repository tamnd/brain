# LeetCode 2411 - Smallest Subarrays With Maximum Bitwise OR

## Problem Understanding

The problem gives us an array `nums`, where each element is a non-negative integer. For every starting index `i`, we must determine the shortest possible subarray beginning at `i` whose bitwise OR is as large as possible among all subarrays that start at `i`.

For a fixed starting position `i`, imagine extending the subarray farther and farther to the right:

- `nums[i...i]`
- `nums[i...i+1]`
- `nums[i...i+2]`
- and so on

Each extension can only keep the OR value the same or increase it, because once a bit becomes `1` in a bitwise OR, it never turns back to `0`.

Eventually, we reach the maximum possible OR obtainable from index `i`. The task is not merely to compute that maximum OR value, but to find the minimum subarray length that achieves it.

For example, if the full suffix from index `i` has OR value `15`, we want the shortest prefix of that suffix whose OR is also `15`.

The constraints are important:

- `n` can be as large as `100000`
- `nums[i]` can be up to `10^9`

A quadratic solution that checks every subarray would be far too slow. We need something close to linear time.

An important observation is that integers up to `10^9` require at most 30 bits, because:

$2^{30} \approx 10^9$

This small fixed bit count becomes the key to the optimal solution.

Some edge cases that can easily cause bugs include:

- Arrays containing only zeros
- Arrays where the first element already has the maximum OR
- Arrays where the needed bit appears very far to the right
- Single-element arrays
- Repeated values that do not introduce new bits

The problem guarantees that all numbers are non-negative, which simplifies bit manipulation logic.

## Approaches

### Brute Force Approach

The most direct solution is to process every starting index independently.

For each index `i`:

1. Start with OR value `0`
2. Extend the subarray one element at a time
3. Update the running OR
4. Track the maximum OR encountered
5. Find the first position where that maximum OR appears

This works because checking all subarrays guarantees correctness.

However, there are `O(n^2)` possible subarrays. Since `n` can be `100000`, this approach becomes far too slow.

Even though the OR operation itself is constant time, iterating through every pair `(i, j)` is not feasible.

### Key Insight for the Optimal Solution

The crucial observation is that the maximum OR starting from index `i` is simply the OR of the entire suffix:

$nums[i] \mid nums[i+1] \mid \dots \mid nums[n-1]$

Why?

Because OR only gains bits as we extend the subarray. Therefore, the full suffix contains every possible bit that can ever appear.

Now consider what is required to achieve that maximum OR.

For every bit that appears somewhere in the suffix, our chosen subarray must include at least one occurrence of that bit.

This transforms the problem into:

- For each starting index `i`
- Find the farthest position needed to collect all required bits

We can process the array from right to left while maintaining:

- The nearest future position where each bit appears

Since there are only about 30 bits, this becomes extremely efficient.

| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Checks every subarray starting at each index |
| Optimal | O(n × 30) | O(30) | Tracks nearest occurrence of each bit |

## Algorithm Walkthrough

### Step 1: Create the Result Array

We create an array `answer` of size `n`.

Each position `answer[i]` will store the minimum subarray length starting at index `i`.

### Step 2: Track the Latest Position of Every Bit

We maintain an array `last_seen` of size 32.

`last_seen[b]` stores the closest index to the right where bit `b` appears.

Initially, all values are `-1`, meaning the bit has not been seen yet.

### Step 3: Process the Array from Right to Left

We iterate backward because future information is needed to determine how far a subarray must extend.

At index `i`, we examine all bits of `nums[i]`.

If bit `b` is set in `nums[i]`, then:

```
last_seen[b] = i
```

This means the nearest occurrence of bit `b` from position `i` onward is now `i` itself.

### Step 4: Determine the Required Ending Position

For the current index `i`, the maximum OR contains every bit that appears somewhere in the suffix.

To obtain all those bits, we must extend the subarray far enough to include the nearest required occurrence of every active bit.

We compute:

```
farthest = maximum(last_seen[b])
```

over all bits that have appeared.

This farthest position is the minimum ending index that guarantees the subarray contains every necessary bit.

### Step 5: Compute the Length

The subarray length is:

$\text{length} = \text{farthest} - i + 1$

Store this value in `answer[i]`.

### Step 6: Continue Until the Beginning

By moving right to left, every index already has access to all future bit positions.

Eventually, the entire answer array is filled.

### Why It Works

The algorithm relies on the monotonic nature of bitwise OR.

A bit contributes to the maximum OR for index `i` if and only if that bit appears somewhere in the suffix starting at `i`.

To achieve the maximum OR, the chosen subarray must include at least one occurrence of every such bit.

For each bit, the nearest occurrence is sufficient. Therefore, the shortest valid subarray ends at the farthest among those nearest required positions.

This guarantees both:

- The OR is maximal
- The subarray is minimal in length

## Python Solution

```python
from typing import List

class Solution:
    def smallestSubarrays(self, nums: List[int]) -> List[int]:
        n = len(nums)
        answer = [1] * n
        
        # last_seen[b] = nearest index at or after current position
        # where bit b appears
        last_seen = [-1] * 32
        
        for i in range(n - 1, -1, -1):
            # Update bit positions using nums[i]
            for bit in range(32):
                if nums[i] & (1 << bit):
                    last_seen[bit] = i
            
            # Find the farthest required position
            farthest = i
            
            for bit in range(32):
                if last_seen[bit] != -1:
                    farthest = max(farthest, last_seen[bit])
            
            answer[i] = farthest - i + 1
        
        return answer
```

The implementation closely follows the algorithm described earlier.

The array is processed from right to left so that future bit information is already known when handling index `i`.

The `last_seen` array tracks the nearest occurrence of every bit. Whenever the current number contains a bit, we update its nearest position.

After updating all relevant bits, we scan through all bit positions again to determine the farthest required index. That farthest position represents the smallest subarray end capable of collecting every necessary bit.

Finally, the subarray length is computed and stored.

Because there are only 32 possible bit positions, the nested loops remain efficient even for large inputs.

## Go Solution

```go
func smallestSubarrays(nums []int) []int {
	n := len(nums)
	answer := make([]int, n)

	lastSeen := make([]int, 32)

	for i := 0; i < 32; i++ {
		lastSeen[i] = -1
	}

	for i := n - 1; i >= 0; i-- {
		// Update bit positions
		for bit := 0; bit < 32; bit++ {
			if nums[i]&(1<<bit) != 0 {
				lastSeen[bit] = i
			}
		}

		// Find farthest required position
		farthest := i

		for bit := 0; bit < 32; bit++ {
			if lastSeen[bit] != -1 && lastSeen[bit] > farthest {
				farthest = lastSeen[bit]
			}
		}

		answer[i] = farthest - i + 1
	}

	return answer
}
```

The Go implementation mirrors the Python logic almost exactly.

Slices are used instead of Python lists. Since Go arrays are initialized with zero values, we explicitly fill `lastSeen` with `-1` to indicate unseen bits.

Go integer operations are safe here because the problem constraints stay well within 32-bit integer limits.

## Edge Cases

### Arrays Containing Only Zeros

If every element is `0`, then every subarray OR is also `0`.

A common mistake is to unnecessarily extend the subarray searching for new bits. Since no bits ever appear, the shortest valid subarray at every position is simply the element itself.

The implementation handles this correctly because `last_seen` never updates for any bit, so `farthest` remains equal to `i`.

### Single Element Arrays

When `n = 1`, the only possible subarray is the single element itself.

Some implementations accidentally produce length `0` because of incorrect initialization or off-by-one errors.

This solution initializes `farthest = i`, ensuring the minimum valid length is always at least `1`.

### Required Bits Appearing Far to the Right

Consider:

```
[1,0,0,0,8]
```

The maximum OR from index `0` is:

$1 \mid 8 = 9$

To obtain bit `8`, the subarray must extend all the way to the final element.

A buggy implementation might stop too early after seeing the current OR increase once.

This solution avoids that issue by explicitly tracking the nearest occurrence of every required bit and taking the farthest necessary position among them.
