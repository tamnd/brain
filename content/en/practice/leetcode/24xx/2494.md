# LeetCode 2494 - Merge Overlapping Events in the Same Hall

## Problem Understanding

This problem gives us a database table named `HallEvents`. Each row represents an event occurring in a specific hall, with a starting date and an ending date.

The goal is to merge all overlapping events that belong to the same hall. Two events are considered overlapping if they share at least one common day. That means if one event ends on the exact same day another event starts, they still overlap and should be merged together.

The output should contain the minimal set of non-overlapping intervals for every hall after all possible merges are completed.

For example, suppose Hall 1 has these two events:

- `2023-01-13` to `2023-01-14`
- `2023-01-14` to `2023-01-17`

These overlap because both include `2023-01-14`. Therefore, they should be merged into:

- `2023-01-13` to `2023-01-17`

An important detail is that merging only happens within the same hall. Events from different halls are completely independent.

The problem does not explicitly provide constraints, but this is a database problem intended for SQL optimization. That means we should expect potentially large datasets, making inefficient repeated comparisons undesirable.

There are several important edge cases:

- Multiple events may completely contain smaller events.
- Duplicate rows may exist.
- Events may overlap transitively. For example:

- A overlaps B
- B overlaps C
- Therefore A, B, and C must all merge together.
- Adjacent events that share a boundary day must merge.
- Different halls may contain identical date ranges, but they must still be processed separately.

## Approaches

### Brute Force Approach

A brute force solution would repeatedly compare every event with every other event in the same hall and merge intervals whenever overlap is detected.

The algorithm would continue looping until no more merges are possible. This eventually produces the correct result because every overlapping chain is eventually collapsed into one interval.

However, this approach is inefficient because each merge can trigger additional scans through the dataset. In the worst case, repeated interval comparisons lead to quadratic or even cubic behavior depending on implementation details.

For large datasets, this becomes impractical.

### Optimal Approach

The key insight is that interval merging becomes much easier after sorting.

If we sort events by:

1. `hall_id`
2. `start_day`
3. `end_day`

then all potentially overlapping intervals for a hall appear next to each other.

Once sorted, we can process events sequentially:

- If the current interval overlaps the previous merged interval, extend the merged interval.
- Otherwise, start a new merged interval.

This is the classic interval merging pattern.

In SQL, we can implement this efficiently using window functions. The important observation is that a new group starts only when the current event starts after the maximum ending date seen so far in that hall.

We can:

1. Compute the running maximum `end_day`
2. Detect where a new non-overlapping group begins
3. Assign a group number using cumulative sums
4. Aggregate intervals within each group

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeated interval comparisons and merges |
| Optimal | O(n log n) | O(n) | Sorting plus linear interval grouping |

## Algorithm Walkthrough

1. First, partition events by `hall_id` because events in different halls must never interact.
2. Within each hall, sort events by `start_day` and `end_day`. Sorting guarantees that overlapping intervals appear consecutively.
3. For every row, compute the maximum `end_day` seen so far among all previous rows in the same hall. This tells us how far the currently merged interval extends.
4. Compare the current `start_day` with the previous running maximum `end_day`.

- If `start_day` is less than or equal to the previous maximum end date, the event overlaps the existing merged group.
- Otherwise, a new merged group begins.
5. Create a flag indicating whether a row starts a new group.
6. Use a cumulative sum over the flags to assign a unique group identifier for each merged interval.
7. Finally, group by:

- `hall_id`
- generated group id

and compute:

- minimum `start_day`
- maximum `end_day`

These values represent the merged intervals.

### Why it works

After sorting, every overlapping interval chain becomes contiguous. The running maximum end date always represents the farthest boundary reached by the current merged interval. Whenever a new interval starts after that boundary, overlap is impossible, so a new group must begin. This invariant guarantees correct grouping of all overlapping events.

## SQL Solution

```sql
WITH OrderedEvents AS (
    SELECT
        hall_id,
        start_day,
        end_day,
        MAX(end_day) OVER (
            PARTITION BY hall_id
            ORDER BY start_day, end_day
            ROWS BETWEEN UNBOUNDED PRECEDING AND 1 PRECEDING
        ) AS prev_max_end
    FROM HallEvents
),
GroupedEvents AS (
    SELECT
        hall_id,
        start_day,
        end_day,
        SUM(
            CASE
                WHEN prev_max_end IS NULL
                     OR start_day > prev_max_end
                THEN 1
                ELSE 0
            END
        ) OVER (
            PARTITION BY hall_id
            ORDER BY start_day, end_day
        ) AS group_id
    FROM OrderedEvents
)
SELECT
    hall_id,
    MIN(start_day) AS start_day,
    MAX(end_day) AS end_day
FROM GroupedEvents
GROUP BY hall_id, group_id;
```

## Solution Explanation

The solution uses two Common Table Expressions.

The first CTE, `OrderedEvents`, computes the maximum `end_day` among all previously processed rows in the same hall. This gives us the farthest reach of the current merged interval before processing the current row.

The second CTE, `GroupedEvents`, determines whether each event begins a new merged interval. If the current event starts after the previous maximum end date, it cannot overlap and therefore starts a new group.

The cumulative `SUM()` assigns stable group identifiers to overlapping chains.

Finally, the outer query aggregates each group into a single merged interval using:

- `MIN(start_day)`
- `MAX(end_day)`

This produces the desired merged ranges.

## Edge Cases

One important edge case is boundary overlap. Since intervals overlap even if they share exactly one day, intervals like `[2023-01-01, 2023-01-03]` and `[2023-01-03, 2023-01-05]` must merge. A common bug is using a strict inequality instead of an inclusive comparison. The solution correctly uses `start_day > prev_max_end` to detect separation.

Another important edge case is transitive overlap. Suppose interval A overlaps B, and B overlaps C, but A does not directly overlap C. All three still belong to one merged interval. The running maximum end date handles this naturally because it continually extends the active merged boundary.

Duplicate rows are also important. Since the table may contain duplicates, the algorithm must still produce valid merged intervals without duplication issues. Sorting and grouping naturally absorb duplicates into the same merged range.

A final edge case involves multiple halls with identical dates. Events must only merge within the same hall. The use of `PARTITION BY hall_id` guarantees complete separation between halls during window function computation and grouping.
