## Solution

## Problem Understanding

The problem gives us two strings, `s` and `t`, and asks for the minimum number of characters we must append to the end of `s` so that `t` becomes a subsequence of the resulting string.

A subsequence means we can delete characters from a string without changing the order of the remaining characters. The characters do not need to be contiguous. For example, `"coding"` is a subsequence of `"coachingding"` because we can skip some characters while still matching the letters in order.

The important detail is that we are only allowed to append characters to the end of `s`. We cannot modify existing characters, insert characters in the middle, or reorder anything. That restriction greatly simplifies the problem.

The key question becomes:

> How much of `t` is already matched as a subsequence inside `s`?

If the first `k` characters of `t` can already be matched in order inside `s`, then we only need to append the remaining `len(t) - k` characters.

For example:

- `s = "abcde"`
- `t = "acefg"`

The characters `"ace"` already appear in order inside `s`, so we only need to append `"fg"`. The answer is `2`.

The constraints are important:

- `1 <= s.length, t.length <= 10^5`

This means we need a very efficient solution. Any algorithm worse than linear or near-linear time could become too slow. In particular, quadratic approaches such as checking many subsequences repeatedly would not scale.

Several edge cases are important:

- `t` may already be a subsequence of `s`, so the answer is `0`.
- `s` may contain none of the characters from `t`, so we may need to append all of `t`.
- `s` and `t` may partially overlap in complex ways.
- Repeated characters can easily break incorrect greedy implementations if pointer movement is mishandled.

The problem guarantees that both strings contain only lowercase English letters, so character comparison is straightforward.

## Approaches

### Brute Force Approach

A brute force strategy would try to determine the longest prefix of `t` that is already a subsequence of `s`.

One possible brute force method is:

1. Try every prefix length `k` from `len(t)` down to `0`.
2. For each prefix `t[:k]`, check whether it is a subsequence of `s`.
3. The first valid `k` gives the answer `len(t) - k`.

Checking whether a string of length `k` is a subsequence of `s` takes `O(len(s))` time using two pointers. Since we may repeat this for many different prefix lengths, the total complexity becomes:

- `O(len(s) * len(t))`

With lengths up to `100000`, this is far too slow.

The brute force approach is correct because it explicitly searches for the largest prefix already matched, but it performs too many repeated scans.

### Optimal Approach

The key observation is that we do not need to repeatedly test prefixes of `t`.

Instead, we can greedily match characters from left to right using two pointers:

- One pointer moves through `s`
- One pointer moves through `t`

Whenever characters match, we advance the pointer in `t` because we have successfully matched one more character of the target subsequence.

At the end:

- The pointer position in `t` tells us how many characters were matched.
- Any remaining suffix of `t` must be appended.

This works because subsequence matching is inherently greedy. If a character in `t` can be matched earlier in `s`, delaying the match never helps. Matching as early as possible always leaves the maximum room for future matches.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × m) | O(1) | Repeatedly checks prefixes of `t` as subsequences |
| Optimal | O(n + m) | O(1) | Single left-to-right two-pointer scan |

Here:

- `n = len(s)`
- `m = len(t)`

## Algorithm Walkthrough

1. Initialize two pointers:

- `i` for traversing `s`
- `j` for traversing `t`
2. Start scanning through `s` from left to right.
3. At each step, compare `s[i]` and `t[j]`.
4. If the characters match:

- We have successfully matched one more character of `t`.
- Increment `j`.
5. Regardless of whether the characters matched, increment `i` because we always move forward through `s`.
6. Continue until:

- We finish scanning `s`, or
- We already matched all characters in `t`.
7. After the scan:

- `j` represents how many characters of `t` were matched.
- The remaining unmatched characters are `t[j:]`.
8. The answer is:

- `len(t) - j`

### Why it works

The algorithm maintains the invariant that `t[:j]` is the longest prefix of `t` that has been matched as a subsequence using the processed portion of `s`.

Whenever we see a matching character, greedily consuming it is always optimal because subsequences only require order preservation, not contiguous placement. Matching earlier can never reduce future matching opportunities.

By the time we finish scanning `s`, we have matched the maximum possible prefix of `t`. Therefore, the only way to make `t` a subsequence is to append the unmatched suffix, whose length is `len(t) - j`.

## Python Solution

```python
class Solution:
    def appendCharacters(self, s: str, t: str) -> int:
        t_index = 0

        for char in s:
            if t_index < len(t) and char == t[t_index]:
                t_index += 1

        return len(t) - t_index
```

The implementation directly follows the two-pointer strategy.

The variable `t_index` tracks how many characters of `t` have already been matched.

We iterate through every character in `s`. Whenever the current character matches the next required character in `t`, we advance `t_index`.

At the end of the loop:

- `t_index` equals the number of matched characters
- `len(t) - t_index` equals the number of unmatched characters still needed

The implementation uses constant extra memory and performs only a single pass through `s`.

## Go Solution

```go
func appendCharacters(s string, t string) int {
    tIndex := 0

    for i := 0; i < len(s); i++ {
        if tIndex < len(t) && s[i] == t[tIndex] {
            tIndex++
        }
    }

    return len(t) - tIndex
}
```

The Go implementation is nearly identical to the Python version.

Since strings in Go can be indexed directly by byte and the problem guarantees lowercase English letters, byte indexing is completely safe here.

No special handling for empty strings is needed because the constraints guarantee both strings have length at least `1`.

The algorithm also uses constant extra memory in Go.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n + m) | Each string is traversed at most once |
| Space | O(1) | Only a few pointer variables are used |

The algorithm performs a single pass through `s`, while `t_index` only moves forward and never backward. No auxiliary data structures proportional to input size are allocated.

Although we conceptually refer to both strings, the actual work done is linear in their combined size.

## Edge Cases

One important edge case occurs when `t` is already a subsequence of `s`. A buggy implementation might continue processing incorrectly or append unnecessary characters. In this situation, the pointer for `t` reaches `len(t)` before the scan finishes, and the algorithm correctly returns `0`.

Another important case is when no characters match at all. For example, `s = "xyz"` and `t = "abc"`. Some incorrect solutions attempt partial matching logic that assumes at least one shared character exists. In this implementation, `t_index` never advances, so the answer naturally becomes `len(t)`.

Repeated characters are another common source of mistakes. Consider `s = "aaaaa"` and `t = "aaab"`. Incorrect pointer movement can accidentally skip valid matches or reuse characters improperly. Because the algorithm strictly advances both pointers only when characters match, every matched character corresponds to a unique position in `s`, which preserves subsequence correctness.

A final subtle edge case is when the subsequence match is non-contiguous. For example, `"abcd"` is a subsequence of `"axbxcxd"`, even though the matching characters are separated by unrelated characters. The algorithm handles this naturally because unmatched characters in `s` are simply skipped while maintaining order.
