## Solution

## Problem Understanding

The problem asks us to design a data structure that tracks uploaded videos and continuously reports the length of the longest uploaded prefix.

The videos are numbered from `1` to `n`, and uploads happen in arbitrary order. A prefix of length `i` is considered valid only if every video from `1` through `i` has already been uploaded.

For example, suppose the uploaded videos are:

```
[1, 2, 3]
```

Then the longest uploaded prefix is `3`.

If the uploaded videos are:

```
[2, 3, 5]
```

Then the longest uploaded prefix is `0`, because video `1` is still missing.

The important detail is that we are not trying to count how many videos have been uploaded. We specifically care about the longest consecutive sequence starting from `1`.

The class must support three operations:

- `LUPrefix(n)` initializes the system.
- `upload(video)` marks a video as uploaded.
- `longest()` returns the largest valid prefix length.

The constraints are important:

- `n` can be as large as `100000`
- There can be up to `200000` total operations

This means every operation should ideally be close to `O(1)` time. A slow scan across all videos after every upload would become too expensive.

The problem also guarantees:

- Every uploaded video number is distinct
- Uploaded video numbers are valid integers in `[1, n]`

This guarantee simplifies the implementation because we do not need to handle duplicate uploads.

Several edge cases are important:

- Uploads may happen completely out of order
- The first uploaded video may not be `1`
- The final missing video in the prefix may appear very late
- All videos may already form a complete prefix
- The longest prefix may remain unchanged for many uploads

A correct solution must efficiently detect when the prefix can expand.

## Approaches

### Brute Force Approach

A straightforward solution is to store uploaded videos in a set or boolean array and recompute the longest prefix every time `longest()` is called.

For example:

1. Maintain an array `uploaded`
2. Mark `uploaded[video] = True` during upload
3. In `longest()`, start from `1` and scan forward until a missing video is found

This works because the definition of the prefix is simple: every number from `1` onward must exist consecutively.

However, this approach becomes inefficient when many calls are made.

Suppose:

- `n = 100000`
- `longest()` is called many times

Each `longest()` call may scan almost the entire array, leading to `O(n)` per query.

With up to `200000` operations, the worst case becomes too large.

### Optimal Approach

The key observation is that the longest uploaded prefix only moves forward, never backward.

If we already know the current longest prefix length, say `current = 5`, then after uploading a new video:

- If video `6` is still missing, the answer remains `5`
- If video `6` becomes available, we can extend the prefix
- Once extended, we never need to revisit smaller indices again

This suggests maintaining:

- A boolean array indicating uploaded videos
- A pointer representing the current longest prefix

Whenever a video is uploaded:

1. Mark it uploaded
2. While the next expected video exists, move the pointer forward

Because each video advances the pointer at most once, the total work across all operations is linear.

This gives extremely efficient performance.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | `O(n)` per `longest()` | `O(n)` | Scans from the beginning each query |
| Optimal | `O(1)` amortized per operation | `O(n)` | Pointer only moves forward once per video |

## Algorithm Walkthrough

### Optimal Algorithm

1. Create a boolean array called `uploaded` of size `n + 1`.

We use `n + 1` because video numbering starts at `1`, not `0`.
2. Maintain an integer pointer called `longest_prefix`.

This stores the current largest valid prefix length.
3. During `upload(video)`:

- Mark `uploaded[video] = True`
- Check whether the next required video has now become available
4. While `uploaded[longest_prefix + 1]` is true:

- Increment `longest_prefix`

This loop expands the prefix as far as possible.
5. During `longest()`:

- Simply return `longest_prefix`

This approach is efficient because every video contributes to pointer advancement only once.

### Why it works

The invariant is:

> `longest_prefix` always equals the largest integer `k` such that every video from `1` through `k` has been uploaded.

Whenever a new upload occurs, the only possible change is that the prefix may expand forward. Since the pointer only moves forward and never backward, every index is processed at most once.

Therefore, the algorithm always maintains the correct answer efficiently.

## Python Solution

```python
class LUPrefix:

    def __init__(self, n: int):
        self.uploaded = [False] * (n + 2)
        self.longest_prefix = 0

    def upload(self, video: int) -> None:
        self.uploaded[video] = True

        while self.uploaded[self.longest_prefix + 1]:
            self.longest_prefix += 1

    def longest(self) -> int:
        return self.longest_prefix

# Your LUPrefix object will be instantiated and called as such:
# obj = LUPrefix(n)
# obj.upload(video)
# param_2 = obj.longest()
```

The implementation uses two pieces of state.

The `uploaded` array stores whether each video has been uploaded. Since videos are numbered from `1` to `n`, the array is slightly oversized for convenience.

The variable `longest_prefix` stores the current answer.

Inside `upload()`, we first mark the video as uploaded. Then we repeatedly check whether the next expected video is available. If it is, we advance the prefix pointer.

The `while` loop is important because one upload can unlock multiple consecutive videos. For example:

```python
Already uploaded: 1,2,4,5
Upload: 3
```

After uploading `3`, the prefix immediately expands from `2` to `5`.

The `longest()` method is extremely fast because the answer is already maintained incrementally.

## Go Solution

```go
type LUPrefix struct {
	uploaded      []bool
	longestPrefix int
}

func Constructor(n int) LUPrefix {
	return LUPrefix{
		uploaded:      make([]bool, n+2),
		longestPrefix: 0,
	}
}

func (this *LUPrefix) Upload(video int) {
	this.uploaded[video] = true

	for this.uploaded[this.longestPrefix+1] {
		this.longestPrefix++
	}
}

func (this *LUPrefix) Longest() int {
	return this.longestPrefix
}

/**
 * Your LUPrefix object will be instantiated and called as such:
 * obj := Constructor(n);
 * obj.Upload(video);
 * param_2 := obj.Longest();
 */
```

The Go implementation mirrors the Python logic closely.

A boolean slice stores upload status, and an integer tracks the current prefix length.

Go slices are initialized with default `false` values, which works perfectly for this problem.

The slice size is `n + 2` to safely allow access to `longestPrefix + 1` without boundary issues.

No integer overflow concerns exist because the maximum value is only `100000`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | `O(1)` amortized | Each video advances the pointer at most once |
| Space | `O(n)` | Boolean array stores upload status |

Although the `while` loop inside `upload()` may appear expensive, the pointer only moves from `0` to `n` once across the entire lifetime of the object.

This means the total work done by all loops combined is `O(n)`.

Therefore, averaged across operations, each operation runs in constant amortized time.

## Edge Cases

### Uploads begin with videos larger than 1

A common bug is incorrectly assuming the first upload creates a valid prefix.

Example:

```
upload(5)
```

The correct answer is still `0` because video `1` is missing.

The implementation handles this correctly because the pointer only advances while the next required index exists.

### One upload unlocks many videos

Another tricky case occurs when many later videos are already uploaded.

Example:

```
Uploaded: 1,2,4,5,6
Then upload: 3
```

The prefix should jump immediately from `2` to `6`.

The `while` loop correctly handles this by continuously advancing until the next missing video is found.

### Entire sequence uploaded

When every video becomes uploaded, the pointer should eventually reach `n`.

Some implementations accidentally stop early because of boundary mistakes.

This implementation avoids out of bounds access by allocating an array of size `n + 2`, making checks like:

```
uploaded[longest_prefix + 1]
```

always safe.
