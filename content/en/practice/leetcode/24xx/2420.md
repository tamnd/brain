**Difficulty:** 🟡 Medium  
**Topics:** Array, Dynamic Programming, Prefix Sum

## Solution

## Problem Understanding

The problem gives us an integer array `nums` and an integer `k`. We need to find every index `i` such that two separate conditions are true around that index.

For an index `i` to be considered good:

- The `k` elements immediately before `i` must be in non-increasing order.
- The `k` elements immediately after `i` must be in non-decreasing order.

More concretely:

- The left segment is `nums[i-k] ... nums[i-1]`
- The right segment is `nums[i+1] ... nums[i+k]`

The left segment is valid if every adjacent pair satisfies:

```
nums[j] >= nums[j+1]
```

The right segment is valid if every adjacent pair satisfies:

```
nums[j] <= nums[j+1]
```

The task is to return all such indices in increasing order.

The array size can be as large as `10^5`, which immediately tells us that quadratic solutions will not work. Any solution that repeatedly scans `k` elements for every index may become too slow because `k` itself can also be large.

The constraints also guarantee:

- `k >= 1`
- `k <= n / 2`
- `n >= 3`

This means every candidate index always has enough elements on both sides.

Several edge cases are important:

- Arrays with all equal values, because equality is allowed in both non-increasing and non-decreasing sequences.
- Strictly increasing or strictly decreasing arrays.
- Small values of `k`, especially `k = 1`, because every single-element segment is automatically valid.
- Large arrays where repeated scanning would cause time limit issues.
- Arrays with alternating patterns that frequently break monotonicity.

## Approaches

### Brute Force Approach

The most direct solution is to examine every possible index `i` from `k` to `n-k-1`.

For each index:

1. Check whether the previous `k` elements are non-increasing.
2. Check whether the next `k` elements are non-decreasing.

This requires scanning up to `k-1` adjacent pairs on each side.

For every candidate index, we may perform `O(k)` work. Since there are `O(n)` candidate indices, the total complexity becomes `O(nk)`.

This approach is correct because it directly implements the problem definition. However, it is too slow when both `n` and `k` are large. In the worst case:

```
n = 100000
k = 50000
```

The algorithm would require billions of comparisons.

### Optimal Approach

The key observation is that many checks overlap.

Suppose we already know that a long segment ending at index `i` is non-increasing. Then when moving to `i+1`, we should reuse previous information instead of rescanning the entire segment.

We can preprocess two arrays:

- `non_increasing[i]`

- Length of the longest non-increasing sequence ending at `i`
- `non_decreasing[i]`

- Length of the longest non-decreasing sequence starting at `i`

These arrays allow us to answer whether a valid segment of length `k` exists around an index in constant time.

For index `i`:

- The left side is valid if:

```
non_increasing[i-1] >= k
```

- The right side is valid if:

```
non_decreasing[i+1] >= k
```

This transforms repeated scanning into simple lookups.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(nk) | O(1) | Directly checks every candidate index |
| Optimal | O(n) | O(n) | Uses preprocessing for constant-time validation |

## Algorithm Walkthrough

1. Create an array `non_increasing` of size `n`.

`non_increasing[i]` stores the number of consecutive elements ending at index `i` that form a non-increasing sequence.

Initialize every value to `1`, because every single element is trivially non-increasing.
2. Fill `non_increasing` from left to right.

For every index `i` from `1` to `n-1`:

- If `nums[i-1] >= nums[i]`, then the sequence continues.
- Extend the previous length:

```
non_increasing[i] = non_increasing[i-1] + 1
```
- Otherwise, restart the sequence length at `1`.
3. Create an array `non_decreasing` of size `n`.

`non_decreasing[i]` stores the number of consecutive elements starting at index `i` that form a non-decreasing sequence.

Again, initialize every value to `1`.
4. Fill `non_decreasing` from right to left.

For every index `i` from `n-2` down to `0`:

- If `nums[i] <= nums[i+1]`, then the sequence continues.
- Extend the sequence:

```
non_decreasing[i] = non_decreasing[i+1] + 1
```
- Otherwise, reset to `1`.
5. Iterate through all possible good indices.

Valid indices range from:

```
k <= i < n-k
```
6. For each candidate index `i`:

- Check whether:

```
non_increasing[i-1] >= k
```
- Check whether:

```
non_decreasing[i+1] >= k
```

If both conditions are true, append `i` to the answer.
7. Return the result list.

### Why it works

The preprocessing arrays encode exactly the information we need.

If `non_increasing[i-1] >= k`, then the `k` elements ending at `i-1` form a non-increasing segment, which is precisely the left-side requirement for index `i`.

Similarly, if `non_decreasing[i+1] >= k`, then the `k` elements starting at `i+1` form a non-decreasing segment, which matches the right-side requirement.

Since every candidate index is checked against these precomputed properties, the algorithm correctly identifies all good indices.

## Python Solution

```python
from typing import List

class Solution:
    def goodIndices(self, nums: List[int], k: int) -> List[int]:
        n = len(nums)

        non_increasing = [1] * n
        for i in range(1, n):
            if nums[i - 1] >= nums[i]:
                non_increasing[i] = non_increasing[i - 1] + 1

        non_decreasing = [1] * n
        for i in range(n - 2, -1, -1):
            if nums[i] <= nums[i + 1]:
                non_decreasing[i] = non_decreasing[i + 1] + 1

        result = []

        for i in range(k, n - k):
            if (
                non_increasing[i - 1] >= k
                and non_decreasing[i + 1] >= k
            ):
                result.append(i)

        return result
```

The implementation begins by computing the `non_increasing` array. Each entry represents how long the current non-increasing run extends backward from that position. This preprocessing allows the algorithm to instantly determine whether the left side of a candidate index satisfies the requirement.

Next, the code computes the `non_decreasing` array by traversing from right to left. Each entry stores how long the non-decreasing run extends forward from that position.

After preprocessing, the algorithm iterates through every valid candidate index. Instead of rescanning segments repeatedly, it performs two constant-time checks using the precomputed arrays.

If both conditions hold, the index is added to the result list.

Finally, the result list is returned in increasing order because the indices are processed from left to right.

## Go Solution

```go
func goodIndices(nums []int, k int) []int {
	n := len(nums)

	nonIncreasing := make([]int, n)
	nonDecreasing := make([]int, n)

	for i := 0; i < n; i++ {
		nonIncreasing[i] = 1
		nonDecreasing[i] = 1
	}

	for i := 1; i < n; i++ {
		if nums[i-1] >= nums[i] {
			nonIncreasing[i] = nonIncreasing[i-1] + 1
		}
	}

	for i := n - 2; i >= 0; i-- {
		if nums[i] <= nums[i+1] {
			nonDecreasing[i] = nonDecreasing[i+1] + 1
		}
	}

	result := []int{}

	for i := k; i < n-k; i++ {
		if nonIncreasing[i-1] >= k &&
			nonDecreasing[i+1] >= k {
			result = append(result, i)
		}
	}

	return result
}
```

The Go implementation follows the same logic as the Python solution. Slices are used instead of Python lists, but the preprocessing strategy remains identical.

Go requires explicit initialization of the arrays and uses `append` to build the result slice dynamically.

There are no integer overflow concerns because the stored sequence lengths never exceed `n`, which is at most `100000`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each preprocessing pass and validation pass scans the array once |
| Space | O(n) | Two auxiliary arrays of size `n` are stored |

The algorithm performs three linear passes over the input array:

- One pass to compute non-increasing lengths
- One pass to compute non-decreasing lengths
- One pass to collect valid indices

Each pass takes `O(n)` time, so the total remains linear.

The additional memory usage comes from the two helper arrays, each storing `n` integers.

## Edge Cases

### Case 1: `k = 1`

When `k = 1`, every single-element segment is automatically both non-increasing and non-decreasing. This can easily lead to off-by-one mistakes.

For example:

```
nums = [5,1,3]
k = 1
```

Every valid middle index should be accepted because both adjacent windows contain only one element.

The implementation handles this naturally because every entry in both preprocessing arrays starts at `1`.

### Case 2: All Elements Equal

Arrays like:

```
[7,7,7,7,7]
```

are important because equality must count as valid monotonicity.

A common bug is using strict comparisons (`>` or `<`) instead of non-strict comparisons (`>=` and `<=`).

The implementation correctly uses:

```
nums[i - 1] >= nums[i]
nums[i] <= nums[i + 1]
```

which allows equal neighboring values.

### Case 3: No Candidate Indices

When:

```
n = 2k
```

there may be no valid indices to examine.

For example:

```
nums = [2,1,1,2]
k = 2
```

The valid range becomes:

```
2 <= i < 2
```

which is empty.

The loop:

```
for i in range(k, n - k):
```

naturally handles this situation by executing zero iterations.
