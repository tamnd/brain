## Solution

## Problem Understanding

The problem asks us to design a very small in-memory SQL-like database system that supports four operations on multiple tables:

1. Insert a row into a table
2. Remove a row from a table
3. Select a single cell value
4. Export all rows from a table in CSV-style format

Each table has a fixed number of columns, defined during initialization. Every inserted row automatically receives a unique integer id. The ids are table-specific and increase monotonically.

A very important detail is that row ids are never reused. Even if a row is deleted, the next inserted row still receives a larger id than the previously inserted row.

For example:

- Insert first row into table `"users"` → id = 1
- Delete row 1
- Insert another row → id = 2, not 1

This means we need to separately track the next available id for every table.

The constructor receives:

- `names[i]`, the name of the `i-th` table
- `columns[i]`, the required number of columns for that table

The operations behave as follows:

- `ins(name, row)` inserts a row if the table exists and the row length matches the expected number of columns
- `rmv(name, rowId)` removes a row if it exists
- `sel(name, rowId, columnId)` returns a specific cell value
- `exp(name)` exports all existing rows as CSV strings

The exported format includes the row id as the first field. For example:

```
1,alice,engineer
```

means:

- row id = 1
- column 1 = alice
- column 2 = engineer

The constraints are relatively small:

- At most 2000 insertions/removals
- At most 10000 selections
- At most 500 exports

These limits tell us we do not need advanced database indexing structures or balanced trees. A carefully designed hash map based solution is more than sufficient.

There are several important edge cases:

- Inserting into a non-existent table
- Inserting rows with incorrect column counts
- Selecting from deleted rows
- Selecting invalid column ids
- Exporting a non-existent table
- Ensuring deleted row ids are never reused

A naive implementation can easily fail if it physically compacts rows after deletions, because row ids must remain stable forever.

## Approaches

### Brute Force Approach

A brute force solution could store rows in simple lists for every table.

For insertion:

- Append rows to the list

For deletion:

- Linearly scan for the row id and remove it

For selection:

- Linearly scan all rows until the desired row id is found

For export:

- Iterate through all rows and generate CSV strings

This works correctly because every operation directly searches through the stored data. However, lookup operations become inefficient because finding a row requires scanning potentially many rows.

If there are many operations, especially `sel`, repeated linear scans become wasteful.

### Optimal Approach

The key observation is that rows are uniquely identified by their row id. This naturally suggests using a hash map.

For every table, we store:

- The expected column count
- The next auto-increment id
- A hash map:

```
rowId -> row data
```

This allows:

- O(1) insertion
- O(1) deletion
- O(1) selection

For export, we iterate through row ids in increasing order and include only rows that still exist.

This approach directly models how a real database might internally manage records with primary keys.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) per lookup/removal | O(n) | Uses linear scans through row lists |
| Optimal | O(1) average for insert/remove/select | O(n) | Uses hash maps keyed by row id |

## Algorithm Walkthrough

1. Create a table structure for every table name.

Each table stores:

- Number of columns
- Next available row id
- Hash map of rows

Example:

```
{
    "users": {
        columns: 3,
        next_id: 1,
        rows: {}
    }
}
```
2. For insertion, first validate the request.

We check:

- Does the table exist?
- Does the row length match the required number of columns?

If either check fails, return `False`.
3. If insertion is valid, assign the current `next_id`.

Store:

```
rowId -> row
```

Then increment `next_id`.

This guarantees auto-increment behavior even after deletions.
4. For removal, validate the table name.

If the table exists and the row id exists in the row map, delete it.

We do not modify `next_id`.
5. For selection, validate all conditions.

We must verify:

- Table exists
- Row exists
- Column id is valid

If any condition fails, return `"<null>"`.

Otherwise return:

```
row[columnId - 1]
```

because column ids are 1-indexed.
6. For export, validate the table name.

If invalid, return an empty list.

Otherwise iterate through row ids in increasing order.

For each existing row:

- Convert row id to string
- Append row values
- Join using commas
7. Return the final exported list.

### Why it works

The core invariant is that every table maintains a unique mapping:

```
rowId -> row data
```

Row ids are never reused because `next_id` only increases. Since rows are stored in a hash map, every row can be efficiently inserted, removed, and queried independently without affecting other rows.

## Python Solution

```python
from typing import List, Dict

class SQL:

    def __init__(self, names: List[str], columns: List[int]):
        self.tables = {}

        for name, col_count in zip(names, columns):
            self.tables[name] = {
                "columns": col_count,
                "next_id": 1,
                "rows": {}
            }

    def ins(self, name: str, row: List[str]) -> bool:
        if name not in self.tables:
            return False

        table = self.tables[name]

        if len(row) != table["columns"]:
            return False

        row_id = table["next_id"]
        table["rows"][row_id] = row
        table["next_id"] += 1

        return True

    def rmv(self, name: str, rowId: int) -> None:
        if name not in self.tables:
            return

        table = self.tables[name]

        if rowId in table["rows"]:
            del table["rows"][rowId]

    def sel(self, name: str, rowId: int, columnId: int) -> str:
        if name not in self.tables:
            return "<null>"

        table = self.tables[name]

        if rowId not in table["rows"]:
            return "<null>"

        if columnId < 1 or columnId > table["columns"]:
            return "<null>"

        return table["rows"][rowId][columnId - 1]

    def exp(self, name: str) -> List[str]:
        if name not in self.tables:
            return []

        table = self.tables[name]
        result = []

        for row_id in sorted(table["rows"].keys()):
            row = table["rows"][row_id]
            csv_row = ",".join([str(row_id)] + row)
            result.append(csv_row)

        return result
```

The implementation begins by constructing a dictionary called `tables`. Each table name maps to a structure containing:

- The expected number of columns
- The next available row id
- A dictionary of stored rows

The `ins` method validates both the table name and the column count before insertion. The row is stored using the current `next_id`, then the counter is incremented.

The `rmv` method safely removes rows only if both the table and row exist.

The `sel` method performs careful validation before accessing the requested cell. Since the problem uses 1-indexed column ids, we convert them to Python's 0-indexed array access.

The `exp` method generates CSV-style strings by iterating through existing row ids in sorted order. Deleted rows are automatically skipped because they no longer exist in the hash map.

## Go Solution

```go
package main

import (
	"strconv"
	"strings"
)

type Table struct {
	columns int
	nextID  int
	rows    map[int][]string
}

type SQL struct {
	tables map[string]*Table
}

func Constructor(names []string, columns []int) SQL {
	tables := make(map[string]*Table)

	for i := 0; i < len(names); i++ {
		tables[names[i]] = &Table{
			columns: columns[i],
			nextID:  1,
			rows:    make(map[int][]string),
		}
	}

	return SQL{
		tables: tables,
	}
}

func (this *SQL) Ins(name string, row []string) bool {
	table, exists := this.tables[name]

	if !exists {
		return false
	}

	if len(row) != table.columns {
		return false
	}

	rowID := table.nextID
	table.rows[rowID] = row
	table.nextID++

	return true
}

func (this *SQL) Rmv(name string, rowId int) {
	table, exists := this.tables[name]

	if !exists {
		return
	}

	delete(table.rows, rowId)
}

func (this *SQL) Sel(name string, rowId int, columnId int) string {
	table, exists := this.tables[name]

	if !exists {
		return "<null>"
	}

	row, exists := table.rows[rowId]

	if !exists {
		return "<null>"
	}

	if columnId < 1 || columnId > table.columns {
		return "<null>"
	}

	return row[columnId-1]
}

func (this *SQL) Exp(name string) []string {
	table, exists := this.tables[name]

	if !exists {
		return []string{}
	}

	result := []string{}

	for rowID := 1; rowID < table.nextID; rowID++ {
		row, exists := table.rows[rowID]

		if !exists {
			continue
		}

		values := append([]string{strconv.Itoa(rowID)}, row...)
		result = append(result, strings.Join(values, ","))
	}

	return result
}

/**
 * Your SQL object will be instantiated and called as such:
 * obj := Constructor(names, columns);
 * param_1 := obj.Ins(name,row);
 * obj.Rmv(name,rowId);
 * param_3 := obj.Sel(name,rowId,columnId);
 * param_4 := obj.Exp(name);
 */
```

The Go implementation mirrors the Python logic closely but uses explicit structs for stronger typing.

Instead of Python dictionaries, Go uses:

```go
map[string]*Table
```

for tables and:

```go
map[int][]string
```

for row storage.

The export method iterates from `1` to `nextID - 1` because Go maps do not preserve ordering. This guarantees rows are exported in ascending row id order.

Go slices naturally handle row storage, and no integer overflow concerns exist because the constraints are very small.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(1) average for insert/remove/select, O(k) for export | Hash map operations are constant time on average |
| Space | O(total rows stored) | We store all existing rows in memory |

Insertion, removal, and selection all rely on direct hash map lookups, which are O(1) on average. Export must iterate through all existing rows, so its complexity is proportional to the number of rows in the table.

The memory usage grows linearly with the number of currently stored rows.

## Edge Cases

One important edge case is inserting rows with the wrong number of columns. A naive implementation might still insert the row and create inconsistent table state. Our implementation validates the row length before insertion and immediately returns `False` if the sizes do not match.

Another critical edge case involves deleted rows. After removing a row, future selections on that row id must return `"<null>"`. Because rows are stored in a hash map and physically deleted using `del` or `delete`, removed rows completely disappear from the table state.

A subtle edge case is the auto-increment behavior after deletions. Many incorrect implementations attempt to reuse deleted ids. The problem explicitly forbids this. Our solution avoids reuse by maintaining a dedicated `next_id` counter that only increases and is never decremented.

Another edge case occurs when selecting invalid columns. Since column ids are 1-indexed, requests such as column `0` or a column greater than the table's column count must return `"<null>"`. Our implementation explicitly checks these bounds before accessing the row array.

Finally, exporting tables after many deletions can become tricky. Some implementations accidentally include deleted rows or produce unordered output. By iterating only over existing rows and exporting in increasing row id order, the implementation guarantees correct formatting and ordering.
