## Solution

## Problem Understanding

The problem gives us a binary matrix `grid` with `m` rows and `n` columns. Every cell contains either `0` or `1`.

We must construct another matrix called `diff` of the same size. For every position `(i, j)`, we compute:

$$diff[i][j] = onesRow_i + onesCol_j - zerosRow_i - zerosCol_j$$

This means each cell depends on four values:

- The number of `1`s in row `i`
- The number of `1`s in column `j`
- The number of `0`s in row `i`
- The number of `0`s in column `j`

The task is not to modify the original matrix, but to calculate these values efficiently for every cell and return the resulting matrix.

The input size constraints are important:

- `1 <= m, n <= 10^5`
- `1 <= m * n <= 10^5`

The product constraint means the total number of cells is at most `100000`, which tells us that an `O(m * n)` solution is expected and feasible. However, anything close to `O((m * n)^2)` would likely time out for the largest inputs.

An important observation is that the number of zeros in a row or column can be derived from the number of ones:

- `zerosRow[i] = n - onesRow[i]`
- `zerosCol[j] = m - onesCol[j]`

This allows us to avoid explicitly counting zeros separately.

Several edge cases are worth considering:

- A matrix containing all zeros
- A matrix containing all ones
- A single row matrix
- A single column matrix
- The smallest possible matrix, `1 x 1`

A naive implementation may repeatedly scan rows and columns for every cell, causing unnecessary repeated work. The key challenge is avoiding this repeated counting.

## Approaches

### Brute Force Approach

The most direct solution is to compute the answer for every cell independently.

For each position `(i, j)`:

1. Scan row `i` and count the number of ones and zeros.
2. Scan column `j` and count the number of ones and zeros.
3. Apply the formula.

This approach is correct because it exactly follows the problem definition. However, it performs a large amount of repeated work.

Suppose the matrix has `m * n` cells. For every cell, we scan an entire row and an entire column:

- Row scan costs `O(n)`
- Column scan costs `O(m)`

Therefore, each cell requires `O(m + n)` work.

Since there are `m * n` cells, the total complexity becomes:

$$O(m * n * (m + n))$$

This is far too slow for large matrices.

### Optimal Approach

The key insight is that row counts and column counts are reused many times.

For example:

- Every cell in row `i` uses the same `onesRow[i]`
- Every cell in column `j` uses the same `onesCol[j]`

Instead of recomputing these values repeatedly, we precompute them once.

The optimized strategy is:

1. Count the number of ones in every row.
2. Count the number of ones in every column.
3. Use the matrix dimensions to derive zero counts.
4. Compute every answer in constant time.

We can simplify the formula algebraically:

$$diff[i][j]
=
onesRow[i] + onesCol[j] - (n - onesRow[i]) - (m - onesCol[j])$$

Rearranging:

$$diff[i][j]
=
2 * onesRow[i] + 2 * onesCol[j] - n - m$$

This means once row and column one-counts are known, each cell can be computed instantly.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m * n * (m + n)) | O(1) excluding output | Repeatedly scans rows and columns |
| Optimal | O(m * n) | O(m + n) excluding output | Precomputes row and column counts once |

## Algorithm Walkthrough

1. Determine the matrix dimensions.

We store:

- `m = number of rows`
- `n = number of columns`

These values are needed for both iteration and zero-count calculations.
2. Create arrays to store row and column one-counts.

We use:

- `row_ones[m]`
- `col_ones[n]`

Each entry starts at `0`.
3. Traverse the matrix once to count ones.

For every cell `(i, j)`:

- If `grid[i][j] == 1`

- Increment `row_ones[i]`
- Increment `col_ones[j]`

After this pass:

- `row_ones[i]` contains the number of ones in row `i`
- `col_ones[j]` contains the number of ones in column `j`
4. Create the result matrix.

Initialize a matrix `diff` of size `m x n`.
5. Compute each cell using the precomputed counts.

For each position `(i, j)`:

- Number of zeros in row `i`:

$$n - row\_ones[i]$$
- Number of zeros in column `j`:

$$m - col\_ones[j]$$

Then compute:

$$diff[i][j]
=
row\_ones[i]
+
col\_ones[j]
-
(n - row\_ones[i])
-
(m - col\_ones[j])$$
6. Return the completed matrix.

### Why it works

The algorithm works because every value required by the formula depends only on row-level and column-level statistics. By precomputing the number of ones in each row and column, we can derive all remaining values directly.

Each `diff[i][j]` is computed using exactly the same formula defined in the problem statement, so the resulting matrix is guaranteed to be correct.

## Python Solution

```python
from typing import List

class Solution:
    def onesMinusZeros(self, grid: List[List[int]]) -> List[List[int]]:
        m = len(grid)
        n = len(grid[0])

        row_ones = [0] * m
        col_ones = [0] * n

        # Count ones in rows and columns
        for i in range(m):
            for j in range(n):
                if grid[i][j] == 1:
                    row_ones[i] += 1
                    col_ones[j] += 1

        # Build result matrix
        diff = [[0] * n for _ in range(m)]

        for i in range(m):
            for j in range(n):
                zeros_row = n - row_ones[i]
                zeros_col = m - col_ones[j]

                diff[i][j] = (
                    row_ones[i]
                    + col_ones[j]
                    - zeros_row
                    - zeros_col
                )

        return diff
```

The implementation begins by determining the matrix dimensions. It then allocates two arrays to store the number of ones in each row and column.

The first nested loop scans the matrix once. Whenever a `1` is encountered, both the corresponding row count and column count are incremented. This efficiently gathers all required statistics in a single traversal.

Next, the solution initializes the output matrix `diff`. A second nested loop computes each result cell independently using the precomputed counts.

Instead of explicitly counting zeros, the implementation derives them mathematically:

- `zeros_row = n - row_ones[i]`
- `zeros_col = m - col_ones[j]`

This avoids unnecessary storage and computation.

Finally, the completed matrix is returned.

## Go Solution

```go
func onesMinusZeros(grid [][]int) [][]int {
    m := len(grid)
    n := len(grid[0])

    rowOnes := make([]int, m)
    colOnes := make([]int, n)

    // Count ones in rows and columns
    for i := 0; i < m; i++ {
        for j := 0; j < n; j++ {
            if grid[i][j] == 1 {
                rowOnes[i]++
                colOnes[j]++
            }
        }
    }

    // Build result matrix
    diff := make([][]int, m)

    for i := 0; i < m; i++ {
        diff[i] = make([]int, n)

        for j := 0; j < n; j++ {
            zerosRow := n - rowOnes[i]
            zerosCol := m - colOnes[j]

            diff[i][j] = rowOnes[i] +
                colOnes[j] -
                zerosRow -
                zerosCol
        }
    }

    return diff
}
```

The Go implementation follows the same algorithmic structure as the Python version.

Slices are used for both the row and column count arrays. The result matrix is constructed row by row using nested `make` calls.

Since Go integers are large enough for the problem constraints, integer overflow is not a concern here. The maximum possible value in any cell is bounded by `m + n`, which is at most `200000`.

Unlike Python, Go requires explicit allocation for every row of the 2D slice.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m * n) | The matrix is traversed twice |
| Space | O(m + n) | Stores row and column counts |

The algorithm performs two full passes over the matrix:

1. One pass to compute row and column one-counts
2. One pass to build the answer matrix

Each traversal touches every cell exactly once, resulting in linear complexity relative to the number of cells.

The auxiliary space consists of:

- One array of size `m`
- One array of size `n`

The output matrix itself is not counted as extra space because it is required by the problem.

## Edge Cases

### Single Cell Matrix

A `1 x 1` matrix is the smallest valid input. This case is important because row and column calculations overlap completely.

For input `[[0]]`:

- Ones in row = 0
- Ones in column = 0
- Zeros in row = 1
- Zeros in column = 1

Result:

```
0 + 0 - 1 - 1 = -2
```

The implementation handles this naturally because the counting arrays still work correctly for size one.

### All Zeros Matrix

If every value is `0`, then every row and column contains zero ones.

This produces the most negative possible results because both zero counts are maximized.

A buggy implementation might incorrectly initialize counts or forget to subtract zero counts properly. Since the solution derives zeros mathematically using matrix dimensions, it handles this case correctly without any special logic.

### Non Square Matrix

Many matrix problems accidentally assume `m == n`, but this problem explicitly allows rectangular matrices.

For example:

```
2 x 5
5 x 2
```

The implementation keeps row and column dimensions separate:

- Rows use `m`
- Columns use `n`

This prevents indexing mistakes and ensures correct zero calculations for rectangular inputs.
