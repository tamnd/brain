# LeetCode 2427 - Number of Common Factors

## Problem Understanding

The problem gives us two positive integers, `a` and `b`, and asks us to determine how many integers divide both numbers evenly.

A number `x` is considered a common factor if:

- `a % x == 0`
- `b % x == 0`

In other words, `x` must divide both numbers without leaving a remainder.

For example, if `a = 12` and `b = 6`, the numbers that divide both are:

- `1`
- `2`
- `3`
- `6`

So the answer is `4`.

The input consists of two integers in the range:

```
1 <= a, b <= 1000
```

These constraints are very small. Since the maximum value is only `1000`, even relatively simple enumeration approaches are fast enough. This tells us that we do not need advanced optimization techniques, although we can still improve the solution with a useful mathematical observation.

One important detail is that both numbers are guaranteed to be positive. This means we never need to handle division by zero or negative divisors.

Several edge cases are worth thinking about early:

- When both numbers are `1`, the only common factor is `1`.
- When the numbers are coprime, meaning their greatest common divisor is `1`, the answer will always be `1`.
- When the numbers are equal, every factor of one number is also a factor of the other.
- When one number is much larger than the other, we only need to check up to the smaller number.

Understanding these cases helps ensure the implementation is both correct and efficient.

## Approaches

### Brute Force Approach

The most straightforward solution is to check every integer from `1` to `min(a, b)`.

For each candidate number `i`:

- Check whether `a % i == 0`
- Check whether `b % i == 0`

If both conditions are true, then `i` is a common factor, so we increment our answer counter.

This works because no number larger than the smaller of `a` and `b` can divide both numbers. By testing every possible divisor, we guarantee that we count every common factor exactly once.

Since the constraints are small, this solution is already fast enough for LeetCode.

### Optimal Observation

A more mathematical observation is that every common factor of `a` and `b` must also be a factor of `gcd(a, b)`.

For example:

- `gcd(12, 6) = 6`
- Factors of `6` are `1, 2, 3, 6`
- These are exactly the common factors of `12` and `6`

So instead of checking divisibility against both numbers repeatedly, we can:

1. Compute `gcd(a, b)`
2. Count how many divisors the gcd has

This reduces the problem to counting factors of a single number.

We can optimize further by checking divisors only up to the square root of the gcd. Whenever we find a divisor `i`, we also get a paired divisor `gcd_value // i`.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(min(a, b)) | O(1) | Checks every possible divisor directly |
| Optimal | O(sqrt(gcd(a, b))) | O(1) | Counts divisors of the gcd using divisor pairs |

## Algorithm Walkthrough

### Optimal Algorithm

1. Compute the greatest common divisor of `a` and `b`.

The gcd contains all shared divisibility information between the two numbers. Any common factor must divide the gcd.
2. Initialize a counter variable called `count`.

This variable stores how many divisors of the gcd we have found.
3. Iterate from `1` to `sqrt(gcd_value)`.

We only need to search up to the square root because divisors come in pairs.
4. For each number `i`, check whether `gcd_value % i == 0`.

If true, then `i` is a divisor.
5. When a divisor is found, determine whether it forms a distinct pair.

- If `i * i == gcd_value`, then `i` is the square root and should only be counted once.
- Otherwise, both `i` and `gcd_value // i` are distinct divisors, so add `2` to the count.
6. Continue until all possible divisors up to the square root are checked.
7. Return the final count.

### Why it works

The key mathematical property is that the set of common factors of `a` and `b` is exactly the set of factors of `gcd(a, b)`.

Every divisor of the gcd divides both numbers, and every common divisor must divide the gcd. Therefore, counting divisors of the gcd gives the exact number of common factors.

The square root optimization works because divisors always appear in complementary pairs. If `i` divides `n`, then `n / i` also divides `n`.

## Python Solution

```python
class Solution:
    def commonFactors(self, a: int, b: int) -> int:
        def gcd(x: int, y: int) -> int:
            while y != 0:
                x, y = y, x % y
            return x

        gcd_value = gcd(a, b)
        count = 0

        divisor = 1

        while divisor * divisor <= gcd_value:
            if gcd_value % divisor == 0:
                if divisor * divisor == gcd_value:
                    count += 1
                else:
                    count += 2

            divisor += 1

        return count
```

The implementation begins with the Euclidean algorithm to compute the greatest common divisor. The Euclidean algorithm repeatedly replaces the larger number with the remainder until the remainder becomes zero. The remaining value is the gcd.

After computing the gcd, the solution counts its divisors. The loop runs while `divisor * divisor <= gcd_value`, which efficiently limits the search space to the square root.

Whenever a divisor is found, the code determines whether it forms a unique square root divisor or a normal divisor pair:

- Perfect square divisor, count once
- Regular divisor pair, count twice

Finally, the function returns the total number of divisors, which equals the number of common factors.

## Go Solution

```go
func commonFactors(a int, b int) int {
    gcd := func(x int, y int) int {
        for y != 0 {
            x, y = y, x%y
        }
        return x
    }

    gcdValue := gcd(a, b)
    count := 0

    for divisor := 1; divisor*divisor <= gcdValue; divisor++ {
        if gcdValue%divisor == 0 {
            if divisor*divisor == gcdValue {
                count++
            } else {
                count += 2
            }
        }
    }

    return count
}
```

The Go implementation follows the same logic as the Python version. One small difference is that Go commonly uses anonymous functions for helper methods like `gcd`.

Since the input range is very small, integer overflow is not a concern here. All operations fit comfortably within Go's standard integer type.

The divisor counting logic is identical:

- Count one divisor for perfect squares
- Count two divisors for normal divisor pairs

## Edge Cases

### Both numbers are 1

When `a = 1` and `b = 1`, the only divisor is `1`. Some implementations accidentally return `0` if they assume divisors start from `2`. This solution correctly checks from `1`, so the answer becomes `1`.

### The gcd is a perfect square

Consider `a = 9` and `b = 27`.

The gcd is `9`, whose divisors are:

```
1, 3, 9
```

The divisor `3` is both the divisor and its paired divisor because `3 * 3 = 9`. Without special handling, the algorithm would double count it. The implementation avoids this by adding only one count when `divisor * divisor == gcd_value`.

### Coprime numbers

For inputs like `a = 7` and `b = 13`, the gcd is `1`. The only common factor is `1`.

Some incorrect solutions may accidentally skip checking `1` or assume there are no common factors. This implementation handles the case naturally because `1` is always checked first.
