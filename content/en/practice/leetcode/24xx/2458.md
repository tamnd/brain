**Topics:** Array, Tree, Depth-First Search, Breadth-First Search, Binary Tree

## Solution

## Problem Understanding

This problem asks us to process multiple independent subtree removal operations on a binary tree. For every query, we temporarily remove the entire subtree rooted at a specific node, then determine the height of the remaining tree.

The binary tree contains `n` nodes, where every node has a unique value between `1` and `n`. The queries array contains node values, and each query represents the root of a subtree that should be removed.

The important detail is that queries are independent. After answering one query, the tree is restored to its original form before the next query begins. This means we do not permanently modify the tree.

The height of a binary tree is defined as the number of edges in the longest path from the root to any node. This is important because many implementations accidentally count nodes instead of edges.

For example:

- A single node tree has height `0`
- A root with one child has height `1`
- A chain of four nodes has height `3`

The constraints are large:

- `n` can be up to `100000`
- `queries.length` can be up to `10000`

These constraints immediately tell us that recomputing the height of the tree from scratch for every query would be too slow. A naive solution that traverses the tree for every query could easily become `O(n * m)`, which may reach `10^9` operations.

Several edge cases are important:

- Removing a subtree that contains the deepest path
- Removing a leaf node
- Removing an entire large branch close to the root
- Trees that are highly unbalanced, such as linked-list shaped trees
- Queries that remove nodes whose siblings determine the new height

The problem guarantees that the queried node is never the root, so the resulting tree is always non-empty.

## Approaches

### Brute Force Approach

The brute force solution processes each query independently.

For every query:

1. Temporarily remove the specified subtree
2. Recompute the height of the remaining tree using DFS
3. Restore the tree

The recomputation step requires traversing most of the tree in the worst case. Since this happens for every query, the complexity becomes extremely expensive.

The approach is correct because each query explicitly rebuilds the resulting tree and computes its height directly. However, it repeats almost the same work many times.

With `n = 100000` and `m = 10000`, this approach is far too slow.

### Optimal Approach

The key observation is that removing a subtree only affects paths that pass through that subtree.

Suppose we know:

- The depth of every node
- The height of every subtree
- The maximum tree height achievable without using a specific subtree

Then we can answer each query in constant time.

The core idea is rerooting style dynamic programming.

For every node, we compute:

- `subtree_height[node]`
- `answer[node]`, meaning the height of the remaining tree after removing this node's subtree

We use two DFS traversals:

1. First DFS computes subtree heights
2. Second DFS propagates the best possible height from ancestors and sibling subtrees

This allows every query to be answered instantly after preprocessing.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × m) | O(n) | Recompute tree height for every query |
| Optimal | O(n + m) | O(n) | Two DFS traversals with preprocessing |

## Algorithm Walkthrough

### Step 1: Compute subtree heights

We perform a postorder DFS.

For every node:

- Compute the height of the left subtree
- Compute the height of the right subtree
- Store the node's subtree height as:

```
1 + max(left_height, right_height)
```

Since the problem defines height in edges, we use:

- Null node height = `-1`
- Leaf node height = `0`

This makes the formulas work naturally.

We store subtree heights in a hash map keyed by node value.

### Step 2: Compute node depths

During traversal, we also store the depth of every node.

Depth means the number of edges from the root to that node.

This is necessary because when considering an alternate path through a sibling subtree, the total height becomes:

```
depth(current node) + 1 + sibling subtree height
```

### Step 3: Perform rerooting DFS

Now we compute the answer for every node.

The idea is:

When moving from parent to child, we need to determine the best tree height available if the child's subtree is removed.

There are two possibilities:

1. The best path comes from ancestors
2. The best path comes from the sibling subtree

Suppose we move from parent to left child.

If the left subtree is removed, the best remaining height may come from:

- The already known best height from above
- The right sibling subtree

The candidate through the sibling becomes:

```
depth(parent) + 1 + height(right subtree)
```

We propagate the maximum of these values downward.

### Step 4: Store answers

For every node:

- `result[node]` stores the height of the remaining tree after removing that subtree

Then every query becomes a simple lookup.

### Why it works

For any node, every valid remaining path after subtree removal must avoid entering that subtree.

Such paths can only come from:

- Ancestor paths already computed earlier
- Sibling subtrees

The rerooting DFS systematically propagates the best possible height excluding each subtree. Since all possible remaining paths are considered exactly once, the computed answer is correct for every node.

## Python Solution

```python
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import Optional, List

class Solution:
    def treeQueries(
        self,
        root: Optional['TreeNode'],
        queries: List[int]
    ) -> List[int]:

        subtree_height = {}
        depth = {}
        answer = {}

        def compute_heights(node: Optional['TreeNode'], d: int) -> int:
            if not node:
                return -1

            depth[node.val] = d

            left_height = compute_heights(node.left, d + 1)
            right_height = compute_heights(node.right, d + 1)

            current_height = 1 + max(left_height, right_height)
            subtree_height[node.val] = current_height

            return current_height

        compute_heights(root, 0)

        def dfs(
            node: Optional['TreeNode'],
            best_without_subtree: int
        ) -> None:

            if not node:
                return

            answer[node.val] = best_without_subtree

            current_depth = depth[node.val]

            if node.left:
                sibling_height = (
                    subtree_height[node.right.val]
                    if node.right
                    else -1
                )

                candidate = current_depth + 1 + sibling_height

                dfs(
                    node.left,
                    max(best_without_subtree, candidate)
                )

            if node.right:
                sibling_height = (
                    subtree_height[node.left.val]
                    if node.left
                    else -1
                )

                candidate = current_depth + 1 + sibling_height

                dfs(
                    node.right,
                    max(best_without_subtree, candidate)
                )

        dfs(root, 0)

        return [answer[q] for q in queries]
```

The implementation uses two DFS traversals.

The first DFS computes two important values:

- `depth[node]`
- `subtree_height[node]`

The subtree height is computed bottom up using postorder traversal. A null node returns `-1`, ensuring leaves correctly receive height `0`.

The second DFS propagates the best possible height outside each subtree.

The parameter `best_without_subtree` represents the maximum tree height achievable without entering the current subtree.

When moving to a child, we calculate whether the sibling subtree provides a better path. The maximum of the inherited value and the sibling-based candidate becomes the child's answer.

Finally, each query is answered in constant time using the precomputed `answer` map.

## Go Solution

```go
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func treeQueries(root *TreeNode, queries []int) []int {

    subtreeHeight := map[int]int{}
    depth := map[int]int{}
    answer := map[int]int{}

    var computeHeights func(node *TreeNode, d int) int

    computeHeights = func(node *TreeNode, d int) int {
        if node == nil {
            return -1
        }

        depth[node.Val] = d

        leftHeight := computeHeights(node.Left, d+1)
        rightHeight := computeHeights(node.Right, d+1)

        currentHeight := 1
        if leftHeight > rightHeight {
            currentHeight += leftHeight
        } else {
            currentHeight += rightHeight
        }

        subtreeHeight[node.Val] = currentHeight

        return currentHeight
    }

    computeHeights(root, 0)

    var dfs func(node *TreeNode, bestWithoutSubtree int)

    dfs = func(node *TreeNode, bestWithoutSubtree int) {
        if node == nil {
            return
        }

        answer[node.Val] = bestWithoutSubtree

        currentDepth := depth[node.Val]

        if node.Left != nil {
            siblingHeight := -1

            if node.Right != nil {
                siblingHeight = subtreeHeight[node.Right.Val]
            }

            candidate := currentDepth + 1 + siblingHeight

            nextBest := bestWithoutSubtree
            if candidate > nextBest {
                nextBest = candidate
            }

            dfs(node.Left, nextBest)
        }

        if node.Right != nil {
            siblingHeight := -1

            if node.Left != nil {
                siblingHeight = subtreeHeight[node.Left.Val]
            }

            candidate := currentDepth + 1 + siblingHeight

            nextBest := bestWithoutSubtree
            if candidate > nextBest {
                nextBest = candidate
            }

            dfs(node.Right, nextBest)
        }
    }

    dfs(root, 0)

    result := make([]int, len(queries))

    for i, q := range queries {
        result[i] = answer[q]
    }

    return result
}
```

The Go implementation follows the same logic as the Python version.

A few Go-specific details are important:

- Go does not have default hash maps like Python dictionaries, so maps are explicitly initialized
- `nil` checks replace Python truthiness checks
- Helper functions are declared as function variables before assignment to support recursion
- Integer overflow is not a concern because heights are at most `100000`

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n + m) | Two DFS traversals plus constant-time query lookups |
| Space | O(n) | Hash maps and recursion stack |

The algorithm visits every node a constant number of times.

The first DFS computes subtree heights and depths. The second DFS computes answers for all nodes. Each traversal is linear.

After preprocessing, every query is answered in `O(1)` time through direct lookup in the answer map.

The auxiliary space comes from:

- Depth map
- Height map
- Answer map
- Recursive call stack

All are linear in the number of nodes.

## Edge Cases

One important edge case is removing a leaf node. A naive implementation might accidentally reduce the tree height even when another equally deep branch still exists. The rerooting approach handles this correctly because it always considers sibling and ancestor paths when computing the remaining maximum height.

Another important case is a completely skewed tree, where every node has only one child. In this structure, removing a subtree can dramatically reduce the height. Since sibling subtrees do not exist, the algorithm correctly falls back to the best ancestor-based height propagated through the DFS.

A third important case is removing a subtree directly below the root. This can eliminate almost the entire tree. Some solutions incorrectly assume the remaining height must involve the removed branch's parent. The rerooting DFS avoids this issue because it computes answers independently using only paths outside the removed subtree.

A subtle implementation detail involves height definitions. Since the problem defines height as edges rather than nodes, null nodes are assigned height `-1`. This ensures leaves naturally receive height `0`. Forgetting this detail often causes off-by-one errors throughout the solution.
