# LeetCode 2403 - Minimum Time to Kill All Monsters

## Problem Understanding

This problem asks us to determine the minimum number of days required to defeat all monsters, given a very specific mana accumulation system.

We are given an array `power`, where `power[i]` represents the mana required to defeat the `ith` monster. Initially, we start with:

- `0` mana
- `gain = 1`

At the start of every day, our current mana increases by `gain`.

After gaining mana for the day, we may choose to defeat a monster if our current mana is at least that monster's required power. When we defeat a monster:

- our mana immediately resets to `0`
- `gain` increases by `1`

The goal is to find the minimum total number of days needed to defeat every monster.

The important detail is that the order in which monsters are defeated matters significantly. Defeating monsters earlier increases `gain`, which allows mana to accumulate faster for future monsters. Because of this, the problem becomes an optimization problem over all possible monster orders.

The constraints are especially important here:

- `1 <= power.length <= 17`
- `1 <= power[i] <= 10^9`

The number of monsters is small, at most 17, which strongly suggests that exponential algorithms involving subsets or bitmasks are intended. However, the monster powers can be extremely large, so we cannot simulate day-by-day behavior inefficiently.

A key observation is that once we know the current `gain`, the number of days needed to accumulate enough mana for a monster can be computed directly using ceiling division.

Several edge cases are important:

- A single monster, where we simply wait enough days to accumulate its required mana.
- Monsters with identical power values.
- Very large power values, where integer overflow could become a concern in some languages.
- Orders where defeating a weaker monster first dramatically reduces the total time because it increases `gain` earlier.

## Approaches

### Brute Force Approach

The most straightforward approach is to try every possible order in which the monsters can be defeated.

Suppose there are `n` monsters. We can generate all `n!` permutations. For each permutation:

1. Start with `gain = 1`
2. For each monster in the chosen order:

- Compute how many days are needed to accumulate enough mana
- Add those days to the answer
- Increase `gain`

This approach is correct because it explicitly evaluates every possible strategy and chooses the minimum.

However, the time complexity becomes prohibitively large. With `n = 17`, we would need to evaluate:

$$17! \approx 3.56 \times 10^{14}$$

which is completely infeasible.

### Optimal Approach, Bitmask Dynamic Programming

The key insight is that the only thing that matters at any point is:

- which monsters have already been defeated
- how many monsters have been defeated so far

If we have already defeated `k` monsters, then:

$$gain = k + 1$$

This means the exact order history does not matter beyond the current subset of defeated monsters.

That observation allows us to use dynamic programming over subsets.

We define:

$$dp[mask]$$

as the minimum number of days needed to defeat all monsters represented by `mask`.

For every state:

- count how many monsters are already defeated
- determine the current gain
- try defeating each remaining monster next

The transition cost can be computed directly using ceiling division:

$$days = \left\lceil \frac{power[i]}{gain} \right\rceil$$

This transforms the problem from factorial complexity into subset DP complexity.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n! × n) | O(n) | Tries every monster order |
| Optimal | O(n × 2^n) | O(2^n) | Bitmask DP over subsets |

## Algorithm Walkthrough

1. Let `n` be the number of monsters.
2. Create a DP array of size `2^n`.

Each bitmask represents a subset of monsters already defeated.

If bit `i` is set, monster `i` has been defeated.
3. Initialize all states to infinity.

Set:

```
dp[0] = 0
```

because defeating no monsters requires zero days.
4. Iterate through every mask from `0` to `(1 << n) - 1`.
5. Count how many monsters are already defeated in the current mask.

If `k` monsters are defeated, then:

```
gain = k + 1
```
6. Try defeating every monster not yet included in the mask.
7. For each remaining monster:

- compute days needed using ceiling division

$$days = \left\lceil \frac{power[i]}{gain} \right\rceil$$

In code:

```
days = (power[i] + gain - 1) // gain
```
8. Transition to the next mask:

```
next_mask = mask | (1 << i)
```
9. Update the DP value:

```
dp[next_mask] = min(dp[next_mask], dp[mask] + days)
```
10. After processing all states, return:

```
dp[(1 << n) - 1]
```

which represents defeating all monsters.

### Why It Works

The DP state captures every relevant piece of information needed for future decisions. Once we know which monsters have already been defeated, we also know the current `gain`, because it depends only on the number of defeated monsters.

The transition explores every valid next monster choice, and the recurrence stores the minimum total days among all possibilities. Since every subset is processed and every transition is considered, the algorithm guarantees the optimal answer.

## Python Solution

```python
from typing import List

class Solution:
    def minimumTime(self, power: List[int]) -> int:
        n = len(power)
        total_masks = 1 << n

        dp = [float("inf")] * total_masks
        dp[0] = 0

        for mask in range(total_masks):
            defeated = mask.bit_count()
            gain = defeated + 1

            for i in range(n):
                if mask & (1 << i):
                    continue

                days_needed = (power[i] + gain - 1) // gain
                next_mask = mask | (1 << i)

                dp[next_mask] = min(
                    dp[next_mask],
                    dp[mask] + days_needed
                )

        return dp[total_masks - 1]
```

The implementation begins by determining the number of monsters and allocating a DP array for all subset states.

Each state in `dp` represents the minimum days required to defeat the monsters indicated by the bitmask. Initially, only `dp[0]` is zero because no days are needed before defeating any monsters.

For every mask, the code computes how many monsters are already defeated using `bit_count()`. Since the gain increases by one after every kill and starts at one, the current gain is simply:

```python
gain = defeated + 1
```

The inner loop considers every monster not yet defeated. The required waiting time is computed using ceiling division, which efficiently determines how many days are needed to accumulate sufficient mana.

The DP transition updates the next subset state with the minimum achievable value.

Finally, the algorithm returns the value corresponding to the mask where all monsters are defeated.

## Go Solution

```go
package main

import "math"

func checkMin(a, b int64) int64 {
	if a < b {
		return a
	}
	return b
}

func minimumTime(power []int) int64 {
	n := len(power)
	totalMasks := 1 << n

	dp := make([]int64, totalMasks)

	for i := range dp {
		dp[i] = math.MaxInt64
	}

	dp[0] = 0

	for mask := 0; mask < totalMasks; mask++ {
		defeated := 0

		for temp := mask; temp > 0; temp >>= 1 {
			if temp&1 == 1 {
				defeated++
			}
		}

		gain := defeated + 1

		for i := 0; i < n; i++ {
			if mask&(1<<i) != 0 {
				continue
			}

			daysNeeded := int64((power[i] + gain - 1) / gain)
			nextMask := mask | (1 << i)

			dp[nextMask] = checkMin(
				dp[nextMask],
				dp[mask]+daysNeeded,
			)
		}
	}

	return dp[totalMasks-1]
}
```

The Go implementation follows the same logic as the Python solution but uses `int64` for safety because cumulative day counts may exceed the standard 32-bit integer range.

Go does not provide a built-in `bit_count()` method for integers in the same way Python does, so the implementation manually counts set bits using a loop.

The DP array is initialized with `math.MaxInt64` to represent infinity.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n × 2^n) | Each subset processes all monsters |
| Space | O(2^n) | DP table stores one value per subset |

There are `2^n` subset states. For every state, we iterate through all `n` monsters to determine possible transitions. Therefore, the total complexity is:

$$O(n \times 2^n)$$

The space complexity comes from the DP array containing one entry for every subset.

With `n <= 17`, this complexity is efficient enough:

$$17 \times 2^{17} \approx 2.2 \text{ million operations}$$

which is easily manageable.

## Edge Cases

A very important edge case is when there is only one monster. In this situation, the gain never increases because no monster has yet been defeated. The answer is simply the monster's power value itself, since mana grows by exactly one per day. The implementation handles this naturally because the DP starts with gain equal to one.

Another important edge case is when monster powers are extremely large, such as `10^9`. A naive simulation that increments mana day by day would be far too slow. The implementation avoids this issue entirely by using direct ceiling division to compute the required number of days in constant time.

A third critical edge case involves ordering sensitivity. Consider `[8,1]`. If we defeat the stronger monster first, we spend eight days immediately. However, defeating the weaker monster first increases gain sooner, reducing the total time substantially. The DP correctly explores all possible orders and guarantees the optimal choice.

A final subtle case is when multiple monsters have identical power values. Some greedy strategies might incorrectly assume the order does not matter, but future gain changes can still affect the optimal schedule. The subset DP does not rely on greedy assumptions and therefore handles these cases correctly.
