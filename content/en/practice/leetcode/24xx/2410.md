**Difficulty:** 🟡 Medium  
**Topics:** Array, Two Pointers, Greedy, Sorting

## Solution

## Problem Understanding

The problem gives us two arrays:

- `players`, where each value represents the ability of a player
- `trainers`, where each value represents the training capacity of a trainer

A player can only be matched with a trainer if:

```
player ability <= trainer capacity
```

Each player can be matched at most once, and each trainer can also be matched at most once. Our goal is to maximize the total number of valid matches.

In other words, we want to pair as many players and trainers as possible while respecting the ability constraint and the one-to-one matching restriction.

For example, if we have:

```
players = [4,7,9]
trainers = [8,2,5,8]
```

then:

- Player `4` can match with trainers `5`, `8`, or `8`
- Player `7` can match with trainers `8` or `8`
- Player `9` cannot match with any trainer

The best possible answer is `2`.

The constraints are important:

- Both arrays can contain up to `10^5` elements
- Values can be as large as `10^9`

These limits immediately tell us that quadratic solutions are too slow. Any solution close to `O(n * m)` will time out when both arrays are large. We should aim for something around `O(n log n)` or better.

Several edge cases are worth noticing early:

- There may be more players than trainers
- There may be more trainers than players
- No player may be matchable
- Every player may be matchable
- Arrays can contain duplicate values
- Arrays may already be sorted or completely unsorted

The problem guarantees all values are positive integers, so we do not need to handle negative values or empty arrays.

## Approaches

### Brute Force Approach

A straightforward idea is to try matching every player with every trainer.

One possible brute-force strategy is:

- Iterate through each player
- Search all unused trainers
- Assign the first trainer that can satisfy the player
- Mark that trainer as used

This approach can produce a valid matching, but it is inefficient because for every player we may scan the entire trainer array.

The time complexity becomes `O(n * m)`, where:

- `n = len(players)`
- `m = len(trainers)`

With both values up to `10^5`, the worst case becomes `10^10` operations, which is far too slow.

Another issue is that greedy matching without sorting can make poor decisions. For example, assigning a very strong trainer to a weak player too early may prevent a stronger player from being matched later.

### Optimal Greedy + Sorting Approach

The key insight is that we should match the weakest possible trainer with the weakest possible player that can be satisfied.

This naturally suggests sorting both arrays.

After sorting:

- The smallest players appear first
- The smallest trainers appear first

We then use two pointers:

- One pointer for players
- One pointer for trainers

If the current trainer can satisfy the current player:

```
trainer >= player
```

then we form a match and move both pointers forward.

Otherwise, the trainer is too weak, so we skip that trainer and move only the trainer pointer.

This greedy strategy works because using the smallest sufficient trainer preserves larger trainers for stronger players later.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * m) | O(1) or O(m) | Checks many unnecessary pairings |
| Optimal | O(n log n + m log m) | O(1) extra, ignoring sort space | Sorting enables efficient greedy matching |

## Algorithm Walkthrough

1. Sort the `players` array in non-decreasing order.

Sorting allows us to process players from weakest to strongest. This is important because weaker players have fewer requirements and are easier to place early.
2. Sort the `trainers` array in non-decreasing order.

Sorting trainers allows us to efficiently find the smallest trainer capable of handling each player.
3. Initialize two pointers:

- `player_index = 0`
- `trainer_index = 0`

Also initialize a variable `matches = 0`.
4. While both pointers are inside their arrays, compare the current player and trainer.

If:

```
trainers[trainer_index] >= players[player_index]
```

then the trainer can handle the player.
5. When a match is possible:

- Increment `matches`
- Move `player_index` forward
- Move `trainer_index` forward

We move both pointers because both the player and trainer are now used.
6. If the trainer is too weak:

```
trainers[trainer_index] < players[player_index]
```

then this trainer cannot match the current player or any stronger player later.

Therefore, we discard this trainer by moving only `trainer_index` forward.
7. Continue until either all players or all trainers have been processed.
8. Return `matches`.

### Why it works

The greedy choice is always safe because we match each player with the smallest trainer capable of satisfying them.

Suppose we used a larger trainer instead of the smallest valid one. That larger trainer might be necessary later for a stronger player. By preserving stronger trainers whenever possible, we maximize future matching opportunities.

Sorting ensures that once a trainer is too weak for the current player, it will also be too weak for every later player, since later players are at least as strong.

## Python Solution

```python
from typing import List

class Solution:
    def matchPlayersAndTrainers(self, players: List[int], trainers: List[int]) -> int:
        players.sort()
        trainers.sort()

        player_index = 0
        trainer_index = 0
        matches = 0

        while player_index < len(players) and trainer_index < len(trainers):
            if trainers[trainer_index] >= players[player_index]:
                matches += 1
                player_index += 1
                trainer_index += 1
            else:
                trainer_index += 1

        return matches
```

The implementation starts by sorting both arrays. This transforms the problem into an ordered matching process where weaker players and weaker trainers are handled first.

The two pointers track the current unmatched player and trainer.

Inside the loop, the current trainer is checked against the current player:

- If the trainer is strong enough, a match is formed
- Otherwise, the trainer is discarded because it cannot help with this player or any stronger player later

The loop stops once either all players are matched or all trainers are exhausted.

The variable `matches` keeps track of the total number of successful pairings.

## Go Solution

```go
package main

import "sort"

func matchPlayersAndTrainers(players []int, trainers []int) int {
	sort.Ints(players)
	sort.Ints(trainers)

	playerIndex := 0
	trainerIndex := 0
	matches := 0

	for playerIndex < len(players) && trainerIndex < len(trainers) {
		if trainers[trainerIndex] >= players[playerIndex] {
			matches++
			playerIndex++
			trainerIndex++
		} else {
			trainerIndex++
		}
	}

	return matches
}
```

The Go implementation follows the exact same logic as the Python version.

The main difference is the use of `sort.Ints()` for sorting slices in place. Go slices behave similarly to dynamic arrays, so pointer-style index traversal works naturally.

Integer overflow is not a concern here because the algorithm only compares integers and counts matches up to `10^5`.

## Edge Cases

One important edge case occurs when no trainer can satisfy any player. For example:

```
players = [10, 20]
trainers = [1, 2]
```

A naive implementation might accidentally count invalid matches if comparison logic is incorrect. In this solution, trainers that are too weak are skipped immediately, so the final answer correctly remains `0`.

Another important case happens when multiple players or trainers have identical values. For example:

```
players = [2,2,2]
trainers = [2,2]
```

Duplicate values can expose pointer movement bugs. If pointers are advanced incorrectly, the algorithm may reuse trainers or skip valid matches. This implementation advances both pointers only after a successful match, guaranteeing each trainer and player is used at most once.

A third critical edge case is when the arrays are highly unbalanced in size. For example:

```
players = [1,2,3,4,5]
trainers = [100]
```

The algorithm must stop safely when one pointer reaches the end of its array. The loop condition ensures no out-of-bounds access occurs, and only one valid match is counted.

A final subtle case is completely unsorted input. Since the greedy strategy depends on processing smallest values first, failing to sort would produce incorrect results. The implementation explicitly sorts both arrays before any matching logic begins, ensuring the greedy invariant always holds.
