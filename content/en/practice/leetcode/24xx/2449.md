## Solution

## Problem Understanding

The problem asks us to transform one array `nums` into another array `target` using a specific type of operation. In each operation, you can choose two **distinct indices** `i` and `j` and simultaneously increase `nums[i]` by 2 and decrease `nums[j]` by 2. The goal is to make `nums` **similar** to `target`, meaning the frequency of each element in `nums` must match the frequency in `target`.

The input arrays are of the same length `n` (up to $10^5$), and elements can be as large as $10^6$. This means a naive approach that simulates operations step by step will be far too slow. The problem guarantees that it is always possible to transform `nums` into `target` with these operations.

Key insights include that increasing and decreasing by 2 preserves **parity**: even numbers stay even, and odd numbers stay odd. This means elements in `nums` must be matched with elements in `target` of the same parity. Any solution must handle parity carefully, otherwise operations may never achieve similarity.

Important edge cases include arrays that are already similar (requiring 0 operations), arrays where all elements are even or odd, and arrays where large differences exist between corresponding elements.

## Approaches

The brute-force approach would try to simulate every possible operation, checking which pairs to increment and decrement. While correct, this would take an enormous number of steps, especially because `n` can be $10^5$, leading to a time complexity far beyond feasible limits.

The optimal approach comes from two key observations:

1. Since operations only change numbers by 2, **parity must match**. Therefore, we can separate the numbers into **even** and **odd** groups and handle each group independently.
2. We can match elements of `nums` and `target` by sorting both groups. Once sorted, the difference between the corresponding elements in each group can be computed. Each operation changes two elements by 2, which means one operation can decrease the difference by 4 (increase one by 2, decrease one by 2). Therefore, the total number of operations required for a group is the sum of differences divided by 4.

This insight allows us to compute the minimum number of operations efficiently without simulating each move.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O((max(nums) - min(nums)) * n) | O(n) | Simulate operations directly, too slow for large n |
| Optimal | O(n log n) | O(n) | Separate evens and odds, sort, sum differences, divide by 4 |

## Algorithm Walkthrough

1. Separate `nums` into `nums_even` and `nums_odd` arrays according to parity. Do the same for `target` to obtain `target_even` and `target_odd`.
2. Sort all four arrays in non-decreasing order. Sorting ensures we pair the smallest numbers with the smallest targets to minimize operations.
3. Initialize a variable `operations` to zero.
4. For each corresponding pair `(num, tar)` in the even arrays, compute the absolute difference `abs(num - tar)` and sum it across the group. Repeat for the odd arrays.
5. Each operation changes two numbers by 2, which effectively reduces the total difference by 4. Therefore, divide the sum of differences by 4 to get the number of operations.
6. Return the total number of operations.

**Why it works:** Sorting guarantees we minimize the total number of operations because pairing larger discrepancies with smaller ones would require extra moves. Separating by parity ensures all operations are valid, and dividing by 4 correctly accounts for the fact that each operation reduces the total sum of differences by 4.

## Python Solution

```python
from typing import List

class Solution:
    def makeSimilar(self, nums: List[int], target: List[int]) -> int:
        # Separate nums and target into even and odd numbers
        nums_even, nums_odd = [], []
        target_even, target_odd = [], []

        for n in nums:
            (nums_even if n % 2 == 0 else nums_odd).append(n)
        for t in target:
            (target_even if t % 2 == 0 else target_odd).append(t)

        # Sort arrays to match elements optimally
        nums_even.sort()
        nums_odd.sort()
        target_even.sort()
        target_odd.sort()

        # Compute the total number of operations needed for each group
        operations = 0
        for n, t in zip(nums_even, target_even):
            operations += abs(n - t) // 2
        for n, t in zip(nums_odd, target_odd):
            operations += abs(n - t) // 2

        # Each operation changes two elements, so divide by 2
        return operations // 2
```

**Explanation:** We first split `nums` and `target` into even and odd arrays. Sorting ensures optimal pairing, and summing absolute differences divided by 2 accounts for the +2/-2 operations. Dividing by 2 again converts total moves into the number of operations, as each operation adjusts two numbers simultaneously.

## Go Solution

```go
import "sort"

func makeSimilar(nums []int, target []int) int64 {
    var numsEven, numsOdd, targetEven, targetOdd []int

    for _, n := range nums {
        if n%2 == 0 {
            numsEven = append(numsEven, n)
        } else {
            numsOdd = append(numsOdd, n)
        }
    }
    for _, t := range target {
        if t%2 == 0 {
            targetEven = append(targetEven, t)
        } else {
            targetOdd = append(targetOdd, t)
        }
    }

    sort.Ints(numsEven)
    sort.Ints(numsOdd)
    sort.Ints(targetEven)
    sort.Ints(targetOdd)

    var operations int64 = 0
    for i := range numsEven {
        diff := numsEven[i] - targetEven[i]
        if diff < 0 {
            diff = -diff
        }
        operations += int64(diff / 2)
    }
    for i := range numsOdd {
        diff := numsOdd[i] - targetOdd[i]
        if diff < 0 {
            diff = -diff
        }
        operations += int64(diff / 2)
    }

    return operations / 2
}
```

**Explanation:** Go handles slices and integer arithmetic efficiently. We explicitly manage absolute differences and convert the final count to `int64` to avoid overflow. The logic mirrors the Python solution.

## Worked Examples

**Example 1:** `nums = [8,12,6], target = [2,14,10]`

Step 1: Separate parity

`nums_even = [8,12,6]`, `nums_odd = []`

`target_even = [2,14,10]`, `target_odd = []`

Step 2: Sort

`nums_even = [6,8,12]`, `target_even = [2,10,14]`

Step 3: Compute differences

`abs(6-2)/2 = 2`, `abs(8-10)/2 = 1`, `abs(12-14)/2 = 1` → sum = 4

Step 4: Divide by 2 → `4 / 2 = 2` operations

**Example 2:** `nums = [1,2,5], target = [4,1,3]`

Evens: `nums_even = [2]`, `target_even = [4]` → `abs(2-4)/2 = 1`

Odds: `nums_odd = [1,5]`, `target_odd = [1,3]` → `abs(1-1)/2 = 0`, `abs(5-3)/2 = 1` → sum = 1 → total sum = 2 → divide by 2 → 1 operation

**Example 3:** `nums = [1,1,1,1,1], target = [1,1,1,1,1]`

All differences = 0 → operations = 0

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | Sorting each parity group dominates, linear separation is O(n) |
| Space | O(n) | Additional arrays for even and odd numbers |

Separating the numbers into even and odd arrays is linear, and the final summation is linear. Sorting is the bottleneck at O(n log n).
