**Difficulty:** 🔴 Hard  
**Topics:** Array, Dynamic Programming, Sorting

## Solution

## Problem Understanding

This problem asks us to assign robots to factories in a way that minimizes the total travel distance.

Each robot starts at a unique position on the X-axis. Each factory also has a unique position and can repair only a limited number of robots. A robot may move either left or right, and once it reaches a factory with remaining capacity, it gets repaired and stops moving.

The important observation is that robots do not interfere with each other. Even if two robots cross paths, there is no collision. That means we only care about the final assignment of robots to factories, not the exact movement process.

The input consists of:

- `robot[i]`, the position of the `i-th` robot
- `factory[j] = [positionj, limitj]`, where:

- `positionj` is the factory position
- `limitj` is the maximum number of robots that factory can repair

The goal is to minimize:

$$\sum |robot_i - assignedFactory_i|$$

The constraints are relatively small:

- Up to 100 robots
- Up to 100 factories

This strongly suggests that a dynamic programming solution is feasible.

Several edge cases are important:

- A factory may have limit `0`
- Robots and factories may already share the same position
- Multiple robots may need to travel to the same factory
- Coordinates can be very large, so we must avoid any coordinate-based DP
- A greedy nearest-factory assignment does not always produce the optimal answer

The problem guarantees that every robot can eventually be repaired, so we never need to handle impossible states in the final answer.

## Approaches

### Brute Force Approach

A brute-force solution would try every possible assignment of robots to factories while respecting factory capacities.

For each robot, we could choose a factory that still has remaining capacity, recursively assign the next robot, and compute the total distance. The minimum among all valid assignments would be the answer.

This approach is correct because it exhaustively explores every possible assignment.

However, it is far too slow. Suppose there are `n` robots and `m` factories. Even with pruning, the number of valid assignments grows exponentially. In the worst case, each robot may choose among many factories, leading to roughly:

$$O(m^n)$$

This is completely infeasible for `n = 100`.

### Key Insight

The crucial observation is that the robots and factories lie on a line.

If we sort both robots and factories by position, then in an optimal solution, robots assigned to a factory form contiguous groups.

Why?

Suppose robot `A` is left of robot `B`, but `A` is assigned to a factory to the right of the factory assigned to `B`. Their assignments would cross. Swapping the assignments can only reduce or preserve total distance.

This means the optimal assignments preserve order.

After sorting:

- We process robots from left to right
- For each factory, we decide how many consecutive robots it repairs

This naturally leads to dynamic programming.

We define:

- `dp(i, j)` = minimum distance to repair robots starting from index `i` using factories starting from index `j`

At each factory, we try assigning:

- 0 robots
- 1 robot
- 2 robots
- ...
- up to `limit`

Then recursively solve the remaining subproblem.

Because there are only 100 robots and factories, this DP is efficient enough.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Tries every valid assignment |
| Optimal Dynamic Programming | O(n × m × L) | O(n × m) | Uses sorted order and contiguous assignment property |

Here:

- `n` = number of robots
- `m` = number of factories
- `L` = maximum factory limit

Since all values are at most 100, this is efficient.

## Algorithm Walkthrough

1. Sort the `robot` array in ascending order.

Sorting allows us to process robots from left to right and exploit the ordered structure of the optimal solution.
2. Sort the `factory` array by factory position.

This ensures factories are also processed from left to right.
3. Define a memoized DP function:

```
dp(robotIndex, factoryIndex)
```

This represents the minimum distance needed to repair all robots starting from `robotIndex` using factories starting from `factoryIndex`.
4. Handle base cases carefully.

- If all robots are assigned, return `0`
- If factories are exhausted while robots remain, return infinity
5. For the current factory, try assigning different numbers of robots.

Suppose the current factory is:

```
[position, limit]
```

We can assign:

- 0 robots
- 1 robot
- 2 robots
- ...
- up to `limit` robots

Each assigned robot must be the next unassigned robot because assignments remain contiguous after sorting.
6. Incrementally compute travel cost.

As we assign robots to the current factory, we maintain:

```
totalDistance += abs(robot[k] - position)
```

This avoids recomputing costs repeatedly.
7. Recursively solve the remaining problem.

After assigning `x` robots to the current factory:

```
totalDistance + dp(nextRobot, nextFactory)
```

We take the minimum over all choices.
8. Use memoization.

Many subproblems repeat, so caching results reduces complexity dramatically.

### Why it works

After sorting, any optimal solution can be rearranged so that assignments never cross. Therefore, robots assigned to a factory form a contiguous segment in sorted order.

The DP explores every possible contiguous allocation while respecting capacities. Since every valid ordered assignment is considered and the minimum is taken, the algorithm always finds the globally optimal total distance.

## Python Solution

```python
from typing import List
from functools import lru_cache
import math

class Solution:
    def minimumTotalDistance(self, robot: List[int], factory: List[List[int]]) -> int:
        robot.sort()
        factory.sort()

        n = len(robot)
        m = len(factory)

        @lru_cache(None)
        def dp(robot_index: int, factory_index: int) -> int:
            if robot_index == n:
                return 0

            if factory_index == m:
                return math.inf

            factory_position, limit = factory[factory_index]

            answer = dp(robot_index, factory_index + 1)

            distance_sum = 0

            for next_robot in range(
                robot_index,
                min(n, robot_index + limit)
            ):
                distance_sum += abs(
                    robot[next_robot] - factory_position
                )

                answer = min(
                    answer,
                    distance_sum +
                    dp(next_robot + 1, factory_index + 1)
                )

            return answer

        return dp(0, 0)
```

The implementation begins by sorting both robots and factories. This is essential because the DP relies on the ordered assignment property.

The memoized `dp` function represents the minimum cost for repairing the remaining robots using the remaining factories.

Two base cases are used:

- If all robots are assigned, the remaining cost is `0`
- If factories run out first, the state is impossible, so infinity is returned

For each factory, the code first considers skipping the factory entirely. Then it progressively assigns robots to that factory while accumulating distance incrementally.

The loop:

```python
for next_robot in range(robot_index, min(n, robot_index + limit)):
```

tries every valid number of robots that the factory can repair.

Memoization ensures each DP state is computed only once.

## Go Solution

```go
package main

import (
	"math"
	"sort"
)

func minimumTotalDistance(robot []int, factory [][]int) int64 {
	sort.Ints(robot)

	sort.Slice(factory, func(i, j int) bool {
		return factory[i][0] < factory[j][0]
	})

	n := len(robot)
	m := len(factory)

	memo := make([][]int64, n+1)

	for i := range memo {
		memo[i] = make([]int64, m+1)

		for j := range memo[i] {
			memo[i][j] = -1
		}
	}

	var dp func(int, int) int64

	dp = func(robotIndex int, factoryIndex int) int64 {
		if robotIndex == n {
			return 0
		}

		if factoryIndex == m {
			return math.MaxInt64 / 2
		}

		if memo[robotIndex][factoryIndex] != -1 {
			return memo[robotIndex][factoryIndex]
		}

		position := factory[factoryIndex][0]
		limit := factory[factoryIndex][1]

		answer := dp(robotIndex, factoryIndex+1)

		var distanceSum int64 = 0

		for nextRobot := robotIndex;
			nextRobot < n && nextRobot < robotIndex+limit;
			nextRobot++ {

			diff := robot[nextRobot] - position

			if diff < 0 {
				diff = -diff
			}

			distanceSum += int64(diff)

			candidate := distanceSum +
				dp(nextRobot+1, factoryIndex+1)

			if candidate < answer {
				answer = candidate
			}
		}

		memo[robotIndex][factoryIndex] = answer

		return answer
	}

	return dp(0, 0)
}
```

The Go solution follows the same DP structure as the Python version.

The primary implementation difference is memoization. Since Go does not have a built-in memoization decorator like Python's `lru_cache`, we manually maintain a 2D memo table initialized with `-1`.

Another important difference is integer handling. Distances may become large, so the solution uses `int64` throughout the DP.

The infinity value is represented with:

```go
math.MaxInt64 / 2
```

Using division by 2 prevents overflow during addition.

Now only factory `6` remains.

Distances:

Distance:

```

```

Now solve:

```
2 + dp(1,1)
```

At factory `6`:

- Robot `4` travels `2`
- Robot `6` travels `0`

Total:

```

```

Distances:

```
|0 - 2| = 2
|4 - 2| = 2

Total so far:

4
```

Now:

```
4 + dp(2,1)
```

Robot `6` goes to factory `6`:

```
|6 - 6| = 0
```

Final total:

```
4
```

Minimum answer:

4

| --- | --- | --- |
| Time | O(n × m × L) | Each DP state tries up to `L` robot assignments |
| Space | O(n × m) | Memoization table stores DP states |

Here:

- `n` = number of robots
- `m` = number of factories
- `L` = maximum factory limit

Since all constraints are at most 100, the total number of operations is comfortably manageable.

The memoized DP contains at most `n × m` states. For each state, we iterate through at most `limit` robots, leading to the total complexity.

## Edge Cases

One important edge case occurs when a factory has capacity zero. A naive implementation might still attempt to assign robots to it or fail because no progress is made. This solution handles the case naturally because the assignment loop runs zero times, and the DP simply moves to the next factory.

Another important edge case is when robots and factories share the same position. The distance contribution becomes zero, and some implementations accidentally mishandle this because they assume positive movement. Since the solution always uses absolute difference, zero-distance assignments work correctly.

Large coordinate values are another potential source of bugs. Coordinates may reach `10^9`, so total distances can become very large. In Python this is naturally safe because integers are unbounded. In Go, the implementation explicitly uses `int64` to avoid overflow.

A subtle edge case involves optimal assignments that are not locally greedy. Assigning each robot to the nearest available factory can produce suboptimal results because factory capacities affect future choices. The DP avoids this mistake by exploring every valid contiguous allocation and choosing the global minimum.
