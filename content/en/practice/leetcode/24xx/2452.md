## Solution

## Problem Understanding

The problem gives us two arrays of strings, `queries` and `dictionary`. Every string contains only lowercase English letters, and all strings have the same length.

For each word in `queries`, we want to determine whether it can be transformed into at least one word in `dictionary` using at most two character changes. A single edit means replacing one character with another character. Insertions and deletions are not allowed because every word already has the same fixed length.

Another way to think about the problem is that we are comparing two words position by position. Every position where the characters differ requires one edit. Therefore, the number of edits required to transform one word into another is simply the number of mismatched positions.

If a query word differs from some dictionary word in two or fewer positions, then that query should appear in the output. The output must preserve the original order of `queries`.

The constraints are relatively small:

- Up to 100 query words
- Up to 100 dictionary words
- Word length up to 100

This tells us that even an approach that compares every query against every dictionary word is acceptable. The worst case involves:

- 100 queries
- 100 dictionary words
- 100 characters per word

That produces at most `100 * 100 * 100 = 1,000,000` character comparisons, which is easily manageable.

There are several important edge cases to keep in mind. A query may already exactly match a dictionary word, which requires zero edits and should therefore be included. A query might differ from every dictionary word in more than two positions, in which case it should not appear in the result. Another subtle case occurs when multiple dictionary words exist, because the query only needs to match one of them within two edits. Finally, since all words have equal length, we never need to handle insertions, deletions, or unequal-length comparisons.

## Approaches

### Brute Force Approach

The most direct approach is to compare every query word against every dictionary word. For each pair of words, we count how many character positions differ.

If the mismatch count is at most two, we immediately know the query qualifies and should be added to the result list. If none of the dictionary words match within two edits, we skip that query.

This approach is correct because it explicitly checks all possible query-dictionary pairs. Since the problem asks whether a valid dictionary match exists, exhaustive comparison guarantees we do not miss any valid transformation.

Although this method is technically brute force, the constraints are small enough that it is already efficient enough to pass comfortably.

### Key Insight

The important observation is that the edit distance here is extremely simplified. Because all words have the same length and only replacements are allowed, the required number of edits is exactly the Hamming distance between two strings, meaning the number of differing positions.

That means we do not need dynamic programming or a full edit distance algorithm like Levenshtein distance. We only need a simple linear scan through corresponding characters.

An additional optimization is early stopping. As soon as the mismatch count exceeds two, we can stop comparing the current pair because it can never become valid again.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(Q × D × N) | O(1) excluding output | Compare every query with every dictionary word character by character |
| Optimal | O(Q × D × N) | O(1) excluding output | Same asymptotic complexity, but uses early stopping once mismatches exceed two |

Here:

- `Q` = number of query words
- `D` = number of dictionary words
- `N` = length of each word

## Algorithm Walkthrough

1. Initialize an empty result list that will store all valid query words.
2. Iterate through each word in `queries`. For every query word, we want to determine whether at least one dictionary word can be reached using at most two edits.
3. For the current query word, iterate through every word in `dictionary`.
4. Compare the query word and dictionary word character by character. Maintain a mismatch counter.
5. Whenever two characters at the same index differ, increment the mismatch counter because one edit would be required for that position.
6. If the mismatch counter becomes greater than two, stop checking the current pair immediately. Even if the remaining characters match, the pair is already invalid.
7. After finishing the comparison, if the mismatch count is at most two, add the query word to the result list and stop checking additional dictionary words. We only need one successful match.
8. Continue until all query words have been processed.

### Why it works

The algorithm works because the only allowed operation is replacing characters. Since all words have equal length, the minimum number of edits required to transform one word into another is exactly the number of positions where the characters differ.

By comparing every query against every dictionary word, we guarantee that any valid transformation will be found. Early stopping does not affect correctness because once more than two mismatches exist, the pair can never satisfy the problem condition.

## Python Solution

```python
from typing import List

class Solution:
    def twoEditWords(self, queries: List[str], dictionary: List[str]) -> List[str]:
        result = []

        for query in queries:
            for word in dictionary:
                differences = 0

                for q_char, w_char in zip(query, word):
                    if q_char != w_char:
                        differences += 1

                    if differences > 2:
                        break

                if differences <= 2:
                    result.append(query)
                    break

        return result
```

The implementation closely follows the algorithm described earlier.

The outer loop processes each query word one at a time. For every query, the inner loop checks all dictionary candidates until a valid match is found.

Inside the character comparison loop, `zip(query, word)` allows us to iterate through corresponding characters simultaneously. Whenever characters differ, we increment `differences`.

The early termination condition is important for efficiency. Once the mismatch count exceeds two, there is no reason to continue comparing the remaining characters.

If a dictionary word satisfies the condition, we append the query to `result` and immediately break out of the dictionary loop because only one successful match is needed.

Finally, after all queries are processed, we return the accumulated result list.

## Go Solution

```go
func twoEditWords(queries []string, dictionary []string) []string {
    result := []string{}

    for _, query := range queries {
        found := false

        for _, word := range dictionary {
            differences := 0

            for i := 0; i < len(query); i++ {
                if query[i] != word[i] {
                    differences++
                }

                if differences > 2 {
                    break
                }
            }

            if differences <= 2 {
                found = true
                break
            }
        }

        if found {
            result = append(result, query)
        }
    }

    return result
}
```

The Go implementation mirrors the Python logic very closely.

One small difference is that Go strings are indexed directly using byte access, which is safe here because the problem guarantees lowercase English letters only. If Unicode characters were possible, rune handling would be necessary.

The `found` boolean tracks whether the current query has already matched a dictionary word. Once a valid match is found, the inner loop terminates early.

Slices in Go naturally handle dynamic result accumulation using `append`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(Q × D × N) | Compare each query against each dictionary word, scanning all characters |
| Space | O(1) excluding output | Only a few counters and variables are used |

The algorithm performs nested iteration over queries and dictionary words. For each pair, it may scan up to `N` characters.

Although early stopping can reduce the average runtime in practice, the worst case still occurs when most comparisons differ in at most two positions or only exceed two mismatches near the end of the string.

The space usage remains constant because no auxiliary data structures proportional to input size are created.

## Edge Cases

One important edge case occurs when a query word already exists in the dictionary unchanged. Since the problem allows a maximum of two edits, zero edits are perfectly valid. The implementation handles this naturally because the mismatch counter remains zero, which satisfies the `differences <= 2` condition.

Another subtle edge case involves words that differ in exactly two positions. It is easy to accidentally write a strict comparison such as `differences < 2`, which would incorrectly reject valid cases. The implementation explicitly uses `<= 2`, ensuring the boundary condition is handled correctly.

A third important case happens when the first few characters already exceed the mismatch limit. Without early stopping, the algorithm would continue scanning unnecessary characters. While correctness would still hold, efficiency would suffer. The implementation immediately breaks once the mismatch count exceeds two, preventing wasted work while preserving correctness.

A final edge case involves very short strings, especially length one. Since any single-character word differs from another in at most one position, all such pairs automatically qualify. The implementation handles this naturally because the comparison loop simply checks the single character and computes the correct mismatch count.
