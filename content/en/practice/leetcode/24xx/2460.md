## Solution

## Problem Understanding

The problem gives us a 0-indexed integer array `nums` containing non-negative integers. We must process the array from left to right and apply a sequence of operations.

For every index `i` from `0` to `n - 2`, we compare `nums[i]` with `nums[i + 1]`.

If the two adjacent values are equal, we:

1. Double the value at `nums[i]`
2. Replace `nums[i + 1]` with `0`

If the two values are different, we do nothing for that index.

An important detail is that the operations happen sequentially. This means every operation modifies the array immediately, and later operations use the updated values rather than the original ones.

After all operations are complete, we must move all zero values to the end of the array while preserving the relative order of all non-zero values.

The final transformed array is returned.

The constraints are relatively small:

- `2 <= nums.length <= 2000`
- `0 <= nums[i] <= 1000`

Since the input size is at most 2000, even quadratic solutions would technically pass. However, the problem is designed for a linear-time simulation approach.

Several edge cases are important:

- Arrays that already contain many zeros
- Consecutive equal numbers
- Situations where merging creates new values that should not merge again immediately
- Arrays with no valid operations
- Arrays consisting entirely of zeros

For example, after transforming `[2,2,2]`:

- At index `0`, we get `[4,0,2]`
- At index `1`, we compare `0` and `2`, not `4` and `2`

This sequential behavior is critical and can easily cause incorrect implementations.

## Approaches

### Brute Force Approach

A straightforward approach is to directly simulate the operations exactly as described.

First, iterate through the array and apply the merge rules:

- If two adjacent values are equal, double the left one and set the right one to zero.

After all operations are complete, repeatedly shift zeros toward the end. One way to do this is to repeatedly scan the array and swap zeros with later non-zero elements until all zeros are grouped at the end.

This method works correctly because it follows the problem statement literally. However, repeatedly shifting zeros can become inefficient. In the worst case, every zero may require many swaps.

### Optimal Approach

The key insight is that the problem naturally separates into two independent phases:

1. Apply all merge operations
2. Compact all non-zero elements to the front

Instead of repeatedly shifting zeros, we can build the final arrangement in one pass using a two-pointer technique.

After performing all merge operations, we iterate through the array and collect all non-zero values in order. Then we fill the remaining positions with zeros.

This preserves the relative order of non-zero elements while moving all zeros to the end efficiently.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Repeatedly shifts zeros after simulation |
| Optimal | O(n) | O(1) | Uses simulation plus efficient compaction |

## Algorithm Walkthrough

### Step 1: Apply the merge operations

Iterate from index `0` to `n - 2`.

For every position:

- Compare `nums[i]` and `nums[i + 1]`
- If they are equal:

- Set `nums[i] *= 2`
- Set `nums[i + 1] = 0`

This directly simulates the required operations in order.

### Step 2: Compact non-zero values

After all merge operations, the array may contain zeros scattered throughout.

Use a write pointer called `insert_position`.

Traverse the array again:

- If the current value is non-zero:

- Write it into `nums[insert_position]`
- Increment `insert_position`

This moves all non-zero values to the front while preserving order.

### Step 3: Fill the remaining positions with zeros

After all non-zero values are written, every remaining position from `insert_position` to the end should become zero.

Fill these positions with `0`.

### Step 4: Return the modified array

The array now satisfies all requirements:

- Operations applied sequentially
- Non-zero order preserved
- All zeros moved to the end

### Why it works

The algorithm is correct because it exactly follows the two logical phases required by the problem.

The first pass performs every operation sequentially in left-to-right order, so later operations always observe updated values.

The second pass preserves the order of all non-zero elements while removing gaps created by zeros. Since every non-zero value is copied forward exactly once and zeros are appended afterward, the final arrangement matches the required output.

## Python Solution

```python
from typing import List

class Solution:
    def applyOperations(self, nums: List[int]) -> List[int]:
        n = len(nums)

        # Apply operations sequentially
        for i in range(n - 1):
            if nums[i] == nums[i + 1]:
                nums[i] *= 2
                nums[i + 1] = 0

        # Move non-zero values forward
        insert_position = 0

        for num in nums:
            if num != 0:
                nums[insert_position] = num
                insert_position += 1

        # Fill remaining positions with zeros
        while insert_position < n:
            nums[insert_position] = 0
            insert_position += 1

        return nums
```

The implementation follows the algorithm directly.

The first loop simulates the required operations exactly as described in the problem statement. Because updates happen immediately, later iterations naturally see the modified array.

The second loop performs the compaction phase. Instead of repeatedly swapping elements, we overwrite positions from left to right using `insert_position`.

Finally, the remaining unused positions are filled with zeros. This guarantees all zeros end up at the end of the array.

The algorithm modifies the input array in place, which keeps the space complexity constant.

## Go Solution

```go
func applyOperations(nums []int) []int {
	n := len(nums)

	// Apply operations sequentially
	for i := 0; i < n-1; i++ {
		if nums[i] == nums[i+1] {
			nums[i] *= 2
			nums[i+1] = 0
		}
	}

	// Move non-zero values forward
	insertPosition := 0

	for _, num := range nums {
		if num != 0 {
			nums[insertPosition] = num
			insertPosition++
		}
	}

	// Fill remaining positions with zeros
	for insertPosition < n {
		nums[insertPosition] = 0
		insertPosition++
	}

	return nums
}
```

The Go implementation is nearly identical to the Python version.

Slices in Go behave similarly to dynamic arrays, so the algorithm can modify the input slice directly in place.

There are no integer overflow concerns because the maximum value is small. Even after doubling, values remain well within Go's integer range.

Unlike Python, Go requires explicit index-based loops and variable declarations, but the underlying logic remains the same.

| Space | O(1) | All operations are performed in place |

The algorithm performs a constant amount of work for each element in the array. The first traversal handles the merge operations, and the second traversal compacts non-zero values.

No auxiliary data structures proportional to input size are used, so the space complexity remains constant.

## Edge Cases

### Consecutive Equal Values

An input like `[2,2,2]` is easy to mishandle if the implementation incorrectly reuses merged values immediately.

The correct sequence is:

- Merge first pair: `[4,0,2]`
- Compare `0` and `2`, not `4` and `2`

The implementation handles this correctly because operations proceed strictly left to right using the updated array state.

### Arrays Containing Many Zeros

Inputs such as `[0,0,1,0]` can expose bugs in zero-shifting logic.

A naive implementation might accidentally reorder non-zero values or fail to move all zeros to the end.

The compaction phase avoids this problem by explicitly copying non-zero values in order and then filling remaining positions with zeros.

### Zero Merging With Zero

The problem statement allows `0 == 0`, so two adjacent zeros technically satisfy the merge condition.

For example:

```
[0,0]
```

becomes:

```
[0,0]
```

because doubling zero still produces zero.

Some implementations incorrectly skip zero comparisons. The provided solution applies the same rule uniformly to all values, including zero, which guarantees correctness.
