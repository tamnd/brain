## Solution

## Problem Understanding

The problem presents a sequence of stones in a river represented by a sorted integer array `stones`, where each value indicates the position of a stone. A frog starts on the first stone (`stones[0]`) and wants to travel to the last stone (`stones[-1]`) and then return to the first stone, but each stone can be **visited at most once**. The frog’s movement between stones is measured by the absolute difference in positions, which defines the "jump length." The goal is to determine the **minimum cost path**, where the cost is defined as the **maximum jump length in the path**.

Key points to note:

- The stones are sorted in strictly increasing order, which ensures no duplicates and allows for predictable traversal.
- Each stone can be jumped on at most once, which prevents simple back-and-forth movement between neighboring stones multiple times.
- The input size can be up to 105 stones, with positions as large as 10^9, so any solution must be efficient and avoid naive brute force.

Important edge cases include:

- Only two stones, where the frog simply jumps to the end and back.
- Large gaps between stones that could make a single jump the limiting factor in cost.
- Uneven spacing where optimal paths require skipping stones to minimize the maximum jump length.

The expected output is a single integer representing the minimum cost path for the frog.

## Approaches

### Brute Force Approach

A brute force approach would attempt to generate **all possible paths** from the first stone to the last and back, computing the maximum jump length for each path. This guarantees the correct answer because it exhaustively explores every sequence. However, this approach is infeasible because the number of possible paths grows factorially with the number of stones, making it extremely slow for the maximum input size of 105 stones.

### Key Insight for Optimal Solution

A critical observation is that the frog can **jump over stones**, and the **order in which it visits the stones can be optimized by alternating jumps**. Specifically, the frog can take **all odd-indexed stones first and then the even-indexed stones** (or vice versa) to minimize the largest jump. This works because:

- The largest jumps in an alternating pattern are minimized compared to sequential hopping.
- In essence, the frog "zig-zags" across the stones, preventing a huge jump from the first to the last stone directly.

This reduces the problem to computing the maximum jump in a **specific alternating sequence** rather than checking all permutations.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n!) | O(n) | Generate all possible paths, compute max jump for each. |
| Optimal | O(n) | O(1) | Traverse stones in alternating pattern to compute max jump. |

## Algorithm Walkthrough

1. Initialize a variable `max_jump` to zero. This will track the largest jump in the optimal path.
2. Iterate through the stones starting from the third stone (index 2) to the last stone. Compute the jump length between the current stone and the stone **two positions before**. This simulates the frog jumping in a zig-zag pattern, skipping every other stone.
3. Update `max_jump` if the current jump length is greater than the current value of `max_jump`.
4. After iterating through all stones, also check the jump from the second stone to the first stone, as the frog may return from the last stone.
5. Return `max_jump`.

Why it works: This pattern guarantees that the maximum jump is minimized because it evenly distributes the frog's jumps across the largest gaps. By always considering every second stone, the algorithm prevents a single huge jump at the end, which would occur if the frog hopped sequentially without skipping stones.

## Python Solution

```python
from typing import List

class Solution:
    def maxJump(self, stones: List[int]) -> int:
        max_jump = 0
        n = len(stones)
        for i in range(2, n):
            max_jump = max(max_jump, stones[i] - stones[i - 2])
        max_jump = max(max_jump, stones[1] - stones[0])
        return max_jump
```

In this Python implementation, the loop computes the jumps between stones separated by one stone, effectively simulating the optimal zig-zag path. The final `max` call ensures we include the first jump, which is a special case since the loop starts from index 2.

## Go Solution

```go
func maxJump(stones []int) int {
    maxJump := 0
    n := len(stones)
    for i := 2; i < n; i++ {
        jump := stones[i] - stones[i-2]
        if jump > maxJump {
            maxJump = jump
        }
    }
    firstJump := stones[1] - stones[0]
    if firstJump > maxJump {
        maxJump = firstJump
    }
    return maxJump
}
```

The Go implementation mirrors the Python logic, with explicit integer comparisons for updating `maxJump`. It handles edge cases such as the first jump separately, just like the Python version.

## Worked Examples

### Example 1

Input: `[0,2,5,6,7]`

1. Initialize `max_jump = 0`.
2. Compute jumps: `stones[2] - stones[0] = 5 - 0 = 5` → `max_jump = 5`.
3. `stones[3] - stones[1] = 6 - 2 = 4` → `max_jump = 5`.
4. `stones[4] - stones[2] = 7 - 5 = 2` → `max_jump = 5`.
5. Check first jump: `stones[1] - stones[0] = 2` → `max_jump = 5`.
6. Return `5`.

### Example 2

Input: `[0,3,9]`

1. `max_jump = 0`.
2. Compute `stones[2] - stones[0] = 9 - 0 = 9` → `max_jump = 9`.
3. First jump: `stones[1] - stones[0] = 3` → `max_jump = 9`.
4. Return `9`.

## Edge Cases

First, the **minimum stones** case `[0,1]` could trip up algorithms that assume at least three stones. Our implementation handles it by checking the first jump separately.

Second, **large position values** like `[0, 10^9]` could cause integer overflow in some languages, but both Python and Go handle these safely with 64-bit integers.

Third, **uneven stone spacing** might tempt a solution that always picks adjacent stones; however, the alternating approach ensures the maximum jump is minimized even when gaps vary widely. This edge case is handled inherently by the zig-zag computation.

This solution is robust, efficient, and directly applicable to the constraints of the problem.
