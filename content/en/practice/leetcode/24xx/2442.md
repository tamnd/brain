## Solution

## Problem Understanding

The problem gives us an array of positive integers called `nums`. For every number in the array, we must reverse its digits and append the reversed value to the array. This operation is performed only on the original elements, not on the newly appended reversed numbers.

After all reversed numbers have been added, we must determine how many distinct integers exist in the final array.

For example, if we start with:

```
[1, 13, 10]
```

their reversals are:

```
1  -> 1
13 -> 31
10 -> 1
```

so the final array becomes:

```
[1, 13, 10, 1, 31, 1]
```

The distinct values are:

```
1, 10, 13, 31
```

which gives an answer of `4`.

An important detail is that reversed integers do not preserve leading zeros. Reversing `10` gives `"01"`, which becomes integer `1`. This means multiple numbers can collapse into the same reversed value.

The constraints are fairly large:

- `nums.length` can be as large as `100000`
- `nums[i]` can be as large as `1000000`

These limits tell us that we need a solution close to linear time. Any quadratic approach that repeatedly scans the array or checks uniqueness inefficiently would be too slow.

There are several edge cases worth noticing immediately:

- Numbers ending in zero, such as `10` or `100`, lose leading zeros after reversal.
- Duplicate values may already exist in the input.
- A number may reverse to itself, such as `121` or `2`.
- Different numbers can reverse into the same value, such as `10` and `1`.

The problem guarantees that all numbers are positive integers, so we never need to handle negative signs.

## Approaches

### Brute Force Approach

A straightforward solution is to explicitly build the final array.

For every number in `nums`, we reverse its digits and append the result to another array. Once the final array is complete, we iterate through it and count distinct values manually.

One naive way to count distinct values is:

- Maintain a list of unique numbers.
- For every value in the final array, scan the unique list to see whether it already exists.
- If not, add it.

This approach is correct because it directly simulates the process described in the problem statement. However, checking whether a number already exists in a list takes linear time.

If the final array contains `2n` elements, uniqueness checking can degrade to `O(n^2)` time complexity in the worst case.

With `n = 100000`, this becomes too slow.

### Optimal Approach

The key observation is that we do not actually care about the final array itself. We only care about how many distinct numbers appear after adding reversals.

A hash set is perfect for this problem because:

- Sets automatically maintain uniqueness.
- Insertion and lookup are approximately `O(1)` on average.

The algorithm becomes:

1. Create an empty set.
2. For each number:

- Add the original number to the set.
- Reverse the digits.
- Add the reversed number to the set.
3. Return the size of the set.

This avoids constructing unnecessary arrays and efficiently handles duplicates.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Uses manual duplicate checking |
| Optimal | O(n * d) | O(n) | Uses a hash set, where `d` is digit count |

Since each number has at most 7 digits, `d` is effectively constant, making the optimal solution linear in practice.

## Algorithm Walkthrough

1. Create an empty hash set called `distinct_numbers`.

The set is used because it automatically stores only unique values. If the same number is inserted multiple times, the set still keeps only one copy.
2. Iterate through every number in `nums`.

We process each original element exactly once because the problem specifies that reversal should only be applied to the original array elements.
3. Insert the original number into the set.

This guarantees that every original value contributes to the final distinct count.
4. Reverse the digits of the current number.

We can reverse digits mathematically:

- Extract the last digit using `% 10`
- Append it to the reversed number
- Remove the last digit using integer division

For example, reversing `120`:

```
reverse = 0

digit = 0
reverse = 0

digit = 2
reverse = 2

digit = 1
reverse = 21
```
5. Insert the reversed number into the set.

If it already exists, the set ignores duplicates automatically.
6. After processing all numbers, return the size of the set.

The set now contains every distinct integer that appears in the final array.

### Why it works

The algorithm works because every number that can appear in the final array belongs to exactly one of two categories:

- An original number from `nums`
- The reversal of an original number

The algorithm inserts both values into the set for every element. Since sets store unique values only, the final size of the set is exactly the number of distinct integers in the final array.

## Python Solution

```python
from typing import List

class Solution:
    def countDistinctIntegers(self, nums: List[int]) -> int:
        def reverse_number(number: int) -> int:
            reversed_number = 0

            while number > 0:
                digit = number % 10
                reversed_number = reversed_number * 10 + digit
                number //= 10

            return reversed_number

        distinct_numbers = set()

        for number in nums:
            distinct_numbers.add(number)
            distinct_numbers.add(reverse_number(number))

        return len(distinct_numbers)
```

The solution begins by defining a helper function called `reverse_number`. This function reverses digits mathematically instead of converting the number to a string. It repeatedly extracts the last digit and builds the reversed result.

The main logic uses a set named `distinct_numbers`. For every value in `nums`, the algorithm inserts both:

- The original number
- Its reversed form

Because sets automatically remove duplicates, we do not need any extra logic for uniqueness.

Finally, the function returns the size of the set, which represents the number of distinct integers after all reverse operations are completed.

## Go Solution

```go
func countDistinctIntegers(nums []int) int {
	reverseNumber := func(number int) int {
		reversedNumber := 0

		for number > 0 {
			digit := number % 10
			reversedNumber = reversedNumber*10 + digit
			number /= 10
		}

		return reversedNumber
	}

	distinctNumbers := make(map[int]bool)

	for _, number := range nums {
		distinctNumbers[number] = true
		distinctNumbers[reverseNumber(number)] = true
	}

	return len(distinctNumbers)
}
```

The Go implementation follows the same logic as the Python solution.

Since Go does not have a built in set type, we simulate one using `map[int]bool`. The map keys represent unique numbers, and the boolean values are irrelevant placeholders.

The digit reversal logic is identical and uses integer arithmetic throughout. Integer overflow is not a concern because reversing a number up to `10^6` still fits comfortably inside Go's `int` type.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n * d) | Each number is processed once and reversed digit by digit |
| Space | O(n) | The set may store up to all unique original and reversed numbers |

The digit reversal operation takes `O(d)` time, where `d` is the number of digits in the number. Since the maximum value is `10^6`, each number has at most 7 digits, so `d` is effectively constant.

Therefore, the overall runtime behaves like linear time, `O(n)`.

The space complexity is `O(n)` because the set can contain up to roughly `2n` unique values in the worst case.

## Edge Cases

One important edge case involves numbers ending in zeros, such as `10` or `1000`. Reversing these numbers produces values like `1`. A buggy implementation might accidentally preserve leading zeros as strings like `"001"`. The mathematical reversal approach avoids this issue naturally because integers never store leading zeros.

Another important case is when a number reverses to itself. Examples include `121`, `44`, and `7`. A naive implementation might accidentally count both the original and reversed version separately. Using a set guarantees that duplicates are stored only once.

A third edge case occurs when different numbers produce the same reversed value. For example:

```
10 -> 1
1  -> 1
```

Even though two operations generate the same number, the set correctly stores only one copy. This ensures the distinct count remains accurate.

Finally, arrays containing many duplicates, such as `[2,2,2,2]`, are handled efficiently because repeated insertions into a hash set remain constant time on average and do not increase memory usage unnecessarily.
