# LeetCode 2499 - Minimum Total Cost to Make Arrays Unequal

## Problem Understanding

We are given two arrays, `nums1` and `nums2`, both of length `n`. We may perform operations only on `nums1`. In one operation, we choose any two indices `i` and `j` and swap `nums1[i]` with `nums1[j]`. The cost of that swap is `i + j`.

Our goal is to make sure that after all swaps are complete:

```
nums1[i] != nums2[i]
```

for every index `i`.

The task is to compute the minimum total swap cost needed to achieve this condition. If it is impossible, we return `-1`.

The key detail is that swaps may involve any indices, not only adjacent ones. Also, the swap cost depends only on the indices involved, not on the values being swapped.

The constraints are large:

- `n` can be as large as `100000`
- values are between `1` and `n`

This immediately tells us that exponential search or simulation of all swap sequences is impossible. We need a near linear solution.

The most important observation is that only indices where:

```
nums1[i] == nums2[i]
```

are problematic. These are the positions that already violate the desired condition.

Another important detail is that swapping values can accidentally create new conflicts. For example, if too many problematic indices contain the same value, we may not be able to rearrange them safely.

Important edge cases include:

- Arrays already valid, answer should be `0`
- All values identical, impossible case
- A dominant value appearing in too many conflicting positions
- Cases where extra non-conflicting indices are needed to break cycles
- Large arrays requiring efficient counting

## Approaches

### Brute Force Approach

A brute force solution would try all possible swap sequences on `nums1` and search for the minimum cost arrangement where every index satisfies:

```
nums1[i] != nums2[i]
```

This could be modeled as a graph search or state-space search where each permutation of `nums1` is a state.

The brute force method is correct because it explores every reachable arrangement and can therefore find the minimum cost valid configuration.

However, the number of permutations is `n!`, which becomes astronomically large even for moderate values of `n`. Even using BFS or Dijkstra on permutation states is completely infeasible.

This approach cannot work under the given constraints.

### Optimal Greedy Counting Approach

The key insight is that we only care about indices where:

```
nums1[i] == nums2[i]
```

These indices are already invalid and must participate in swaps.

Suppose we collect all such indices. Their indices automatically contribute to the answer because each of them must be involved in at least one swap.

Now consider the values appearing at those conflicting indices. If one value appears too frequently, then swapping only among conflicting indices may still leave some positions equal.

For example:

```
nums1 = [1,1,1]
nums2 = [1,1,1]
```

No rearrangement can avoid equality everywhere because every value is identical.

More generally, if one value appears more than half the time among the conflicting positions, then we need additional helper indices. These helper indices must:

- not already be conflicting
- contain a different value in both arrays from the dominant value

We greedily add such indices with minimal extra cost.

This produces an `O(n)` solution.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n!) | O(n!) | Explores all permutations or swap sequences |
| Optimal | O(n) | O(n) | Uses greedy counting and frequency analysis |

## Algorithm Walkthrough

1. Traverse the arrays and identify all indices where:

```
nums1[i] == nums2[i]
```

These indices are already invalid and must be fixed.
2. For every conflicting index:

- Add the index to the running cost
- Count the frequency of the value appearing there
- Track which value appears most frequently
3. Let:

- `totalBad` be the number of conflicting indices
- `maxFreq` be the highest frequency among conflicting values
- `dominantValue` be the value with that frequency
4. If:

```
maxFreq <= totalBad / 2
```

then the conflicting indices can be rearranged among themselves without creating unavoidable equalities.

In this case, the current accumulated cost is already optimal.
5. Otherwise, the dominant value appears too many times. We need extra helper indices.

The number of additional indices needed is:

```
needed = 2 * maxFreq - totalBad
```

This is the minimum number required to ensure the dominant value no longer exceeds half the positions.
6. Traverse the arrays again looking for helper indices where:

- `nums1[i] != nums2[i]`
- `nums1[i] != dominantValue`
- `nums2[i] != dominantValue`

These indices are safe because bringing them into swaps will not introduce new dominant-value conflicts.
7. For every valid helper index:

- Add its index to the total cost
- Decrease `needed`
8. If after scanning all indices:

```
needed > 0
```

then no valid arrangement exists, so return `-1`.
9. Otherwise, return the accumulated cost.

### Why it works

The critical invariant is that no value may occupy more than half of the participating swap positions. If a value exceeds half, then by the pigeonhole principle at least one position would remain equal after rearrangement.

The algorithm first gathers all mandatory conflicting indices. If one value dominates these positions, we introduce additional safe indices until the dominant value no longer exceeds half the total positions. At that point, a valid rearrangement is always possible.

Because we always choose the smallest available indices greedily during a left-to-right scan, the total cost is minimized.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Solution:
    def minimumTotalCost(self, nums1: List[int], nums2: List[int]) -> int:
        frequency = defaultdict(int)

        total_cost = 0
        total_bad = 0

        dominant_value = -1
        max_frequency = 0

        n = len(nums1)

        # Collect conflicting indices
        for i in range(n):
            if nums1[i] == nums2[i]:
                value = nums1[i]

                total_cost += i
                total_bad += 1

                frequency[value] += 1

                if frequency[value] > max_frequency:
                    max_frequency = frequency[value]
                    dominant_value = value

        # Additional indices needed
        needed = max(0, 2 * max_frequency - total_bad)

        # Add helper indices
        for i in range(n):
            if needed == 0:
                break

            if nums1[i] != nums2[i]:
                if nums1[i] != dominant_value and nums2[i] != dominant_value:
                    total_cost += i
                    needed -= 1

        return total_cost if needed == 0 else -1
```

The implementation begins by scanning all indices and collecting the conflicting positions where the arrays currently match.

For every conflicting position, the index is immediately added to the answer because that index must participate in some swap. At the same time, we count how frequently each value appears among conflicting positions.

The algorithm tracks the most frequent conflicting value because this value determines whether rearrangement is possible using only the conflicting indices.

If the dominant value appears too often, additional helper indices are required. The second pass finds such helper indices greedily from left to right so the added cost is minimized.

Finally, if enough helper indices were found, we return the accumulated cost. Otherwise, the arrangement is impossible.

## Go Solution

```go
package main

func minimumTotalCost(nums1 []int, nums2 []int) int64 {
	frequency := make(map[int]int)

	var totalCost int64
	totalBad := 0

	dominantValue := -1
	maxFrequency := 0

	n := len(nums1)

	// Collect conflicting indices
	for i := 0; i < n; i++ {
		if nums1[i] == nums2[i] {
			value := nums1[i]

			totalCost += int64(i)
			totalBad++

			frequency[value]++

			if frequency[value] > maxFrequency {
				maxFrequency = frequency[value]
				dominantValue = value
			}
		}
	}

	needed := 2*maxFrequency - totalBad
	if needed < 0 {
		needed = 0
	}

	// Add helper indices
	for i := 0; i < n && needed > 0; i++ {
		if nums1[i] != nums2[i] &&
			nums1[i] != dominantValue &&
			nums2[i] != dominantValue {

			totalCost += int64(i)
			needed--
		}
	}

	if needed > 0 {
		return -1
	}

	return totalCost
}
```

The Go implementation follows the same logic as the Python version. The main difference is that Go requires explicit integer type handling. Since the answer can exceed the range of a 32-bit integer, the function uses `int64` for the accumulated cost.

The frequency map is implemented using Go's built-in `map[int]int`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Two linear scans through the arrays |
| Space | O(n) | Frequency map may store up to n distinct values |

The algorithm performs only two passes over the arrays. Every operation inside the loops is constant time, giving overall linear complexity.

The hash map stores frequencies of values appearing at conflicting indices. In the worst case, all values are distinct, requiring linear additional space.

## Edge Cases

### Already Valid Arrays

If every index already satisfies:

```
nums1[i] != nums2[i]
```

then no swaps are needed and the answer should be `0`.

A buggy implementation might still attempt swaps unnecessarily. This solution correctly handles the case because `totalBad` remains zero and no cost is accumulated.

### Dominant Value Appears Too Often

Suppose one value dominates the conflicting indices.

Example:

```
nums1 = [2,2,2]
nums2 = [2,2,1]
```

The value `2` appears too frequently among conflicts. Rearranging only conflicting positions cannot avoid at least one equality.

The algorithm detects this through:

```
2 * maxFrequency - totalBad
```

and searches for additional helper indices.

### Impossible Configurations

Some cases simply cannot be solved.

Example:

```
nums1 = [1,1,1]
nums2 = [1,1,1]
```

No matter how we rearrange `nums1`, every position will still contain `1`.

The implementation correctly returns `-1` because no helper indices exist to reduce the dominant frequency below half the participating positions.
