**Difficulty:** 🔴 Hard  
**Topics:** Tree, Depth-First Search, Graph Theory, Trie

## Solution

## Problem Understanding

We are given a rooted tree with `n` nodes. The tree is rooted at node `0`, and each node has an associated integer value. For every node, its subtree consists of the node itself and all of its descendants in the rooted tree.

The task is to choose two subtrees that do not overlap, meaning they share no nodes at all. For those two chosen subtrees, we compute:

$$\text{subtreeSum}(u) \oplus \text{subtreeSum}(v)$$

where `⊕` is the bitwise XOR operator.

We must return the maximum possible XOR value among all valid pairs of non-overlapping subtrees.

The important detail is that subtree overlap is determined by ancestry. If one node is an ancestor of another, then their subtrees overlap. Otherwise, their subtrees are disjoint.

The input consists of:

- `n`, the number of nodes
- `edges`, an undirected edge list describing a valid tree
- `values`, where `values[i]` is the value stored at node `i`

The constraints are large:

- `n` can be as large as `5 * 10^4`
- Node values can be up to `10^9`

These constraints immediately rule out any quadratic solution over all subtree pairs. Since a tree with `5 * 10^4` nodes has roughly `2.5 * 10^9` pairs, an `O(n^2)` approach will time out.

Several edge cases are important:

- A chain-like tree may make recursion depth large.
- There may be no two disjoint subtrees at all, such as a path of length 2.
- Very large subtree sums require careful handling of bit operations.
- Two nodes can only form a valid pair if neither lies inside the other’s subtree.

The core challenge is efficiently checking disjointness while maximizing XOR.

## Approaches

### Brute Force

The most direct approach is:

1. Root the tree at node `0`.
2. Compute the subtree sum for every node.
3. For every pair of nodes `(u, v)`:

- Check whether their subtrees overlap.
- If they do not overlap, compute:

$$\text{subtreeSum}(u) \oplus \text{subtreeSum}(v)$$
- Track the maximum.

To determine whether two subtrees overlap, we can use DFS entry and exit times. A node `u` is an ancestor of `v` if:

$$tin[u] \le tin[v] \le tout[u]$$

So two subtrees are disjoint exactly when neither node is an ancestor of the other.

This approach is correct because it explicitly checks every valid pair. However, it is too slow. There are `O(n^2)` pairs, and even constant-time overlap checks still produce quadratic complexity.

With `n = 5 * 10^4`, this is infeasible.

### Key Insight

The important observation is that subtree overlap is entirely determined by DFS ordering.

If we process nodes during DFS traversal, then when we finish processing a subtree, every previously completed subtree that is not inside the current subtree must be disjoint from it.

This allows us to transform the problem into an online XOR maximization problem.

Suppose we process nodes in DFS order. For each node:

- We know its subtree sum.
- We want the maximum XOR with any previously completed disjoint subtree.

The natural data structure for maximizing XOR is a binary trie.

A trie allows us to:

- Insert subtree sums
- Query the maximum XOR partner for a given value

in `O(log MAX_VALUE)` time.

The remaining challenge is ensuring we only compare disjoint subtrees. This is handled by carefully ordering insertions during DFS.

We use a second DFS:

- Before exploring a child subtree, we query against subtree sums from already processed sibling subtrees.
- After finishing a child subtree, we insert all sums from that subtree into the trie.

This guarantees all queried subtree sums are disjoint.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Checks every subtree pair explicitly |
| Optimal | O(n log M) | O(n log M) | Uses DFS ordering and a binary trie for XOR maximization |

Here, `M` is the maximum subtree sum value, which fits within about 46 bits because:

$$5 \times 10^4 \times 10^9 = 5 \times 10^{13}$$

## Algorithm Walkthrough

### Step 1: Build the Tree

Convert the edge list into an adjacency list so DFS traversal becomes efficient.

Since the input tree is undirected, we must track parent nodes during DFS to avoid revisiting nodes.

### Step 2: Compute Subtree Sums

Run a DFS from root `0`.

For every node:

1. Start with its own value.
2. Recursively compute subtree sums of all children.
3. Add child subtree sums to the current node’s sum.

Store:

- `subtree_sum[u]`

This gives the sum of every subtree.

### Step 3: Flatten the Tree with Euler Tour

During DFS, assign each node an entry time:

- `tin[u]`

and an exit time:

- `tout[u]`

A subtree becomes a contiguous interval in Euler order.

This property allows efficient subtree grouping later.

### Step 4: Build a Binary Trie

We need fast maximum XOR queries.

The trie stores binary representations of subtree sums.

Each trie node has:

- child `0`
- child `1`

To maximize XOR:

- At each bit position, prefer the opposite bit if available.

Since subtree sums may reach about `5 * 10^13`, we use 46 bits.

### Step 5: DFS for Non-Overlapping Pairs

Now comes the key idea.

For a node `u`, different child subtrees are disjoint from each other.

Suppose we process children one by one.

For each child:

1. Query every node inside that child subtree against the trie.
2. The trie currently contains subtree sums from previously processed sibling subtrees.
3. Therefore, every queried pair is guaranteed disjoint.

After querying:

1. Insert all subtree sums from the current child subtree into the trie.

This ordering guarantees correctness.

### Step 6: Track the Maximum XOR

Each query returns the best XOR achievable with an already inserted disjoint subtree.

Update the global answer accordingly.

### Why it works

The algorithm relies on the fact that sibling subtrees in a rooted tree are always disjoint.

During DFS processing of a node:

- Previously inserted subtree sums belong only to already completed sibling subtrees.
- Current queries come only from the active child subtree.

Therefore, every compared pair is guaranteed non-overlapping.

The trie guarantees we always obtain the maximum XOR among all inserted subtree sums, so every valid pair is considered optimally.

Together, these properties ensure correctness.

## Python Solution

```python
from typing import List
import sys

sys.setrecursionlimit(1 << 20)

class TrieNode:
    def __init__(self):
        self.children = {}

class Trie:
    def __init__(self):
        self.root = TrieNode()
        self.BITS = 46

    def insert(self, num: int) -> None:
        node = self.root

        for bit in range(self.BITS, -1, -1):
            current = (num >> bit) & 1

            if current not in node.children:
                node.children[current] = TrieNode()

            node = node.children[current]

    def query(self, num: int) -> int:
        node = self.root
        result = 0

        for bit in range(self.BITS, -1, -1):
            current = (num >> bit) & 1
            opposite = 1 - current

            if opposite in node.children:
                result |= (1 << bit)
                node = node.children[opposite]
            else:
                node = node.children[current]

        return result

class Solution:
    def maxXor(self, n: int, edges: List[List[int]], values: List[int]) -> int:
        graph = [[] for _ in range(n)]

        for u, v in edges:
            graph[u].append(v)
            graph[v].append(u)

        subtree_sum = [0] * n
        tin = [0] * n
        tout = [0] * n
        euler = []

        timer = 0

        def dfs_sum(node: int, parent: int) -> None:
            nonlocal timer

            tin[node] = timer
            euler.append(node)
            timer += 1

            total = values[node]

            for neighbor in graph[node]:
                if neighbor == parent:
                    continue

                dfs_sum(neighbor, node)
                total += subtree_sum[neighbor]

            subtree_sum[node] = total
            tout[node] = timer - 1

        dfs_sum(0, -1)

        answer = 0

        def collect_nodes(root: int) -> List[int]:
            result = []

            for idx in range(tin[root], tout[root] + 1):
                result.append(euler[idx])

            return result

        def dfs(node: int, parent: int) -> None:
            nonlocal answer

            trie = Trie()

            for neighbor in graph[node]:
                if neighbor == parent:
                    continue

                current_nodes = collect_nodes(neighbor)

                for x in current_nodes:
                    if trie.root.children:
                        answer = max(
                            answer,
                            trie.query(subtree_sum[x])
                        )

                for x in current_nodes:
                    trie.insert(subtree_sum[x])

            for neighbor in graph[node]:
                if neighbor == parent:
                    continue

                dfs(neighbor, node)

        dfs(0, -1)

        return answer
```

The implementation begins by constructing the adjacency list representation of the tree. This allows efficient DFS traversal.

The first DFS computes three important pieces of information:

- `subtree_sum[node]`
- Euler tour entry time
- Euler tour exit time

The Euler ordering makes every subtree correspond to a contiguous range in the traversal array.

The trie implementation supports two operations:

- `insert`
- `query`

The `query` operation greedily attempts to take the opposite bit at each level to maximize XOR.

The second DFS performs the actual optimization logic. For every node, we process child subtrees one at a time.

The trie initially contains subtree sums from already processed child subtrees. When we query nodes from the current child subtree, all trie values are guaranteed to come from disjoint subtrees.

After querying, we insert the current subtree sums into the trie so later siblings can use them.

This maintains the required invariant throughout traversal.

## Go Solution

```go
package main

type TrieNode struct {
	children [2]*TrieNode
}

type Trie struct {
	root *TrieNode
}

func NewTrie() *Trie {
	return &Trie{
		root: &TrieNode{},
	}
}

func (t *Trie) Insert(num int64) {
	node := t.root

	for bit := 46; bit >= 0; bit-- {
		current := (num >> bit) & 1

		if node.children[current] == nil {
			node.children[current] = &TrieNode{}
		}

		node = node.children[current]
	}
}

func (t *Trie) Query(num int64) int64 {
	node := t.root
	var result int64 = 0

	for bit := 46; bit >= 0; bit-- {
		current := (num >> bit) & 1
		opposite := 1 - current

		if node.children[opposite] != nil {
			result |= (1 << bit)
			node = node.children[opposite]
		} else {
			node = node.children[current]
		}
	}

	return result
}

func maxXor(n int, edges [][]int, values []int) int64 {
	graph := make([][]int, n)

	for _, edge := range edges {
		u := edge[0]
		v := edge[1]

		graph[u] = append(graph[u], v)
		graph[v] = append(graph[v], u)
	}

	subtreeSum := make([]int64, n)
	tin := make([]int, n)
	tout := make([]int, n)
	euler := make([]int, 0)

	timer := 0

	var dfsSum func(int, int)

	dfsSum = func(node int, parent int) {
		tin[node] = timer
		euler = append(euler, node)
		timer++

		total := int64(values[node])

		for _, neighbor := range graph[node] {
			if neighbor == parent {
				continue
			}

			dfsSum(neighbor, node)
			total += subtreeSum[neighbor]
		}

		subtreeSum[node] = total
		tout[node] = timer - 1
	}

	dfsSum(0, -1)

	var answer int64 = 0

	collectNodes := func(root int) []int {
		result := []int{}

		for i := tin[root]; i <= tout[root]; i++ {
			result = append(result, euler[i])
		}

		return result
	}

	var dfs func(int, int)

	dfs = func(node int, parent int) {
		trie := NewTrie()

		for _, neighbor := range graph[node] {
			if neighbor == parent {
				continue
			}

			currentNodes := collectNodes(neighbor)

			for _, x := range currentNodes {
				if trie.root.children[0] != nil || trie.root.children[1] != nil {
					value := trie.Query(subtreeSum[x])

					if value > answer {
						answer = value
					}
				}
			}

			for _, x := range currentNodes {
				trie.Insert(subtreeSum[x])
			}
		}

		for _, neighbor := range graph[node] {
			if neighbor == parent {
				continue
			}

			dfs(neighbor, node)
		}
	}

	dfs(0, -1)

	return answer
}
```

The Go implementation closely mirrors the Python version, but several language-specific details are important.

Go requires explicit use of `int64` because subtree sums may exceed the range of a 32-bit integer. The trie methods therefore operate entirely on `int64`.

Unlike Python dictionaries, the trie uses fixed-size child arrays:

```go
children [2]*TrieNode
```

This is both faster and more memory efficient in Go.

Slices are used for adjacency lists and Euler traversal storage. Since Go does not support nested functions with implicit closure behavior exactly like Python, recursive DFS functions are declared as variables first and then assigned.

| Query Value | Best XOR Partner | XOR |

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log M) | Each subtree sum is inserted and queried once in the trie |
| Space | O(n log M) | Trie storage plus DFS structures |

The trie operations require traversing approximately 47 bits because subtree sums can reach about `5 * 10^13`.

Each node participates in a constant number of trie insertions and queries, so the total complexity becomes `O(n log M)`.

The DFS arrays, adjacency list, Euler tour, and trie together require linear space relative to the input size.

## Edge Cases

One important edge case is a degenerate chain-shaped tree. In such a structure, every node is an ancestor of all nodes below it, meaning every pair of subtrees overlaps. A naive implementation might incorrectly compare unrelated DFS intervals or assume different nodes always represent disjoint subtrees. The algorithm handles this correctly because sibling subtree comparisons never occur in a pure chain, so the answer remains `0`.

Another important case involves extremely large subtree sums. Since node values may reach `10^9` and there may be `5 * 10^4` nodes, subtree sums can exceed `32-bit` integer range. The implementation explicitly uses sufficiently large integer handling. Python integers naturally support arbitrary precision, while the Go solution uses `int64`.

A third subtle case occurs when the optimal answer involves internal subtree nodes rather than immediate children of the root. A simplistic solution that only compares root children would miss such cases. The DFS traversal recursively processes every node as a local root, ensuring all possible disjoint subtree pairs are considered.

A fourth edge case is repeated subtree sums. Multiple subtrees may produce identical sums, making XOR equal to zero. The trie logic still works correctly because maximizing XOR naturally handles equal values without requiring uniqueness.
