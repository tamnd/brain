# LeetCode 2414, Length of the Longest Alphabetical Continuous Substring

## Problem Understanding

The problem asks us to find the length of the longest substring in a given lowercase English string where every character is exactly one letter after the previous character in the alphabet.

An alphabetical continuous substring means the characters must appear consecutively in alphabetical order. For example:

- `"abc"` is valid because `'b'` comes immediately after `'a'`, and `'c'` comes immediately after `'b'`
- `"xyz"` is valid for the same reason
- `"abd"` is not valid because `'d'` does not immediately follow `'b'`
- `"za"` is not valid because wrapping around the alphabet is not allowed

The input is a string `s` containing only lowercase English letters. We need to return the maximum length among all valid continuous substrings.

The constraints are important:

- `1 <= s.length <= 10^5`
- Only lowercase English letters are used

A string length of `10^5` tells us that quadratic solutions are likely too slow. We need a solution that processes the string efficiently, ideally in linear time.

Several edge cases are worth considering immediately:

- A string with only one character, such as `"a"`
- A string with no consecutive alphabetical pairs, such as `"zazaz"`
- A string where the entire input is already continuous, such as `"abcdefghijklmnopqrstuvwxyz"`
- Multiple continuous segments separated by breaks, such as `"abcabdefg"`

The problem guarantees the string is non-empty, so we do not need to handle empty input.

## Approaches

### Brute Force Approach

The brute force solution checks every possible substring and determines whether it is an alphabetical continuous substring.

For every starting index `i`, we can extend the substring character by character toward the right. For each substring `s[i:j]`, we verify that every adjacent pair differs by exactly `1` in ASCII value.

For example, in `"abac"`:

- `"a"` is valid
- `"ab"` is valid
- `"aba"` is invalid because `'a'` does not follow `'b'`
- `"bac"` is invalid because `'c'` does not follow `'a'`

We keep track of the maximum valid length found.

This approach is correct because it explicitly examines all possible substrings. However, there are `O(n^2)` substrings, and validating each substring can take additional time. Even with optimization, the overall complexity is too large for `n = 10^5`.

### Optimal Approach

The key observation is that we only care about adjacent characters.

If `s[i]` is exactly one character after `s[i-1]`, then the current continuous substring can be extended. Otherwise, the current sequence breaks and we must start a new substring from the current character.

This means we do not need to examine every substring independently. Instead, we can scan the string once while maintaining:

- The length of the current valid continuous substring
- The maximum length seen so far

This transforms the problem into a simple linear traversal.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Checks every substring explicitly |
| Optimal | O(n) | O(1) | Single pass using consecutive character tracking |

## Algorithm Walkthrough

1. Initialize two variables:

- `current_length = 1`, representing the length of the current alphabetical continuous substring
- `max_length = 1`, representing the best answer found so far
2. Start iterating through the string from index `1` to `n - 1`.
3. For each character `s[i]`, compare it with the previous character `s[i - 1]`.
4. Check whether:

```
ord(s[i]) == ord(s[i - 1]) + 1
```

This condition means the current character is exactly the next letter in the alphabet.
5. If the condition is true:

- Extend the current continuous substring
- Increment `current_length`
6. Otherwise:

- The sequence is broken
- Reset `current_length` to `1` because the current character itself starts a new substring
7. After each iteration, update:

```
max_length = max(max_length, current_length)
```
8. After processing the entire string, return `max_length`.

### Why it works

The algorithm maintains the invariant that `current_length` always represents the length of the longest valid continuous substring ending at the current index.

Whenever two adjacent characters are consecutive alphabetically, we extend the existing substring. Otherwise, no valid continuous substring can cross that boundary, so we restart the count.

Because every character is processed exactly once and every possible continuous segment is evaluated during traversal, the algorithm correctly finds the global maximum length.

## Python Solution

```python
class Solution:
    def longestContinuousSubstring(self, s: str) -> int:
        current_length = 1
        max_length = 1

        for i in range(1, len(s)):
            if ord(s[i]) == ord(s[i - 1]) + 1:
                current_length += 1
            else:
                current_length = 1

            max_length = max(max_length, current_length)

        return max_length
```

The implementation directly follows the algorithm described earlier.

We initialize both `current_length` and `max_length` to `1` because a single character is always a valid alphabetical continuous substring.

The loop begins from index `1` because each character must be compared with its predecessor.

The expression:

```python
ord(s[i]) == ord(s[i - 1]) + 1
```

checks whether the characters are consecutive in the alphabet. For example:

- `'b'` follows `'a'`
- `'c'` follows `'b'`

If the condition is true, the current substring grows by one character. Otherwise, the sequence breaks and we restart from length `1`.

At every step, we update `max_length` so that it always stores the best answer encountered during traversal.

## Go Solution

```go
func longestContinuousSubstring(s string) int {
    currentLength := 1
    maxLength := 1

    for i := 1; i < len(s); i++ {
        if s[i] == s[i-1]+1 {
            currentLength++
        } else {
            currentLength = 1
        }

        if currentLength > maxLength {
            maxLength = currentLength
        }
    }

    return maxLength
}
```

The Go implementation is very similar to the Python version.

One useful difference is that Go strings are byte slices internally. Since the input contains only lowercase English letters, comparing bytes directly is safe and efficient.

The condition:

```go
s[i] == s[i-1] + 1
```

works because characters are stored using ASCII values.

No additional memory allocation is needed, so the implementation remains constant-space.

## Edge Cases

### Single Character Strings

A string containing only one character is always a valid alphabetical continuous substring of length `1`.

This case can expose bugs when implementations assume at least one adjacent comparison exists. Our implementation handles this correctly because both counters start at `1`, and the loop simply does not execute.

### Repeated Characters

Inputs like `"aaaaaa"` can be tricky because repeated letters are not consecutive alphabetically.

A common mistake is checking whether characters are non-decreasing instead of exactly consecutive. Our implementation specifically checks:

```
ord(s[i]) == ord(s[i - 1]) + 1
```

so repeated characters correctly reset the sequence length.

### Multiple Separate Continuous Segments

Strings such as `"abcxyzab"` contain several valid continuous regions.

An incorrect implementation might only track the first valid segment or fail to reset properly after a break. Our solution continuously updates the global maximum while independently restarting sequences whenever continuity breaks.

### Descending or Wrapped Sequences

Cases like `"cba"` or `"za"` are invalid because the letters do not increase by exactly one.

The implementation correctly rejects these because:

- `'b'` is not `'c' + 1`
- `'a'` is not `'z' + 1`

The problem does not allow alphabet wrapping.
