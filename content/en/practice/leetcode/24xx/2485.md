# LeetCode 2485 - Find the Pivot Integer

## Problem Understanding

The problem gives us a positive integer `n`, which represents the sequence of integers from `1` to `n`. We need to find an integer `x` such that the sum of all integers from `1` through `x` is equal to the sum of all integers from `x` through `n`.

In other words, we are looking for a number that balances the sequence into two equal sum parts, with the pivot number itself included in both sums.

Mathematically, we want:

$$1 + 2 + \dots + x = x + (x+1) + \dots + n$$

The input consists of a single integer `n`, and the output should be the pivot integer if one exists. If no such integer exists, we return `-1`.

The constraints are very small:

$$1 \le n \le 1000$$

This tells us that even relatively inefficient solutions will still run comfortably within limits. However, the problem is designed to test mathematical reasoning and prefix sum observations rather than raw optimization.

An important guarantee is that there is at most one valid pivot integer. This means we do not need to worry about multiple answers or choosing among several candidates.

Several edge cases are important to think about early:

- When `n = 1`, the only number is trivially the pivot.
- Small values such as `n = 2` or `n = 3` may not have any valid pivot.
- The pivot could appear near the beginning or near the end of the range.
- Many inputs do not have a valid pivot at all, so the algorithm must correctly return `-1`.

## Approaches

### Brute Force Approach

The most straightforward solution is to try every possible pivot integer from `1` to `n`.

For each candidate `x`:

1. Compute the sum from `1` to `x`.
2. Compute the sum from `x` to `n`.
3. Compare the two sums.
4. If they are equal, return `x`.

This works because we explicitly test every valid candidate and directly verify the condition required by the problem.

However, this approach repeatedly recalculates sums. For each candidate pivot, we may iterate through up to `n` numbers twice. Since there are `n` possible pivots, the total complexity becomes:

$$O(n^2)$$

Although this is acceptable for `n <= 1000`, we can do better using mathematical observations.

### Optimal Approach

The key observation comes from using the formula for the sum of the first `k` integers:

$$1 + 2 + \dots + k = \frac{k(k+1)}{2}$$

Let the total sum from `1` to `n` be:

$$\text{total} = \frac{n(n+1)}{2}$$

If `x` is the pivot integer, then:

$$1 + 2 + \dots + x = x + (x+1) + \dots + n$$

The left side equals:

$$\frac{x(x+1)}{2}$$

The right side can be written as:

$$\text{total} - \frac{(x-1)x}{2}$$

Setting them equal and simplifying leads to:

$$x^2 = \text{total}$$

This means the pivot integer exists only if the total sum is a perfect square. If the square root of the total sum is an integer, that integer is the pivot.

This reduces the problem to a simple mathematical check.

## Algorithm Walkthrough

1. Compute the total sum of integers from `1` to `n` using the arithmetic series formula.

$$\text{total} = \frac{n(n+1)}{2}$$

This gives the sum of the entire range in constant time.
2. Compute the integer square root of `total`.

If `total` is a perfect square, then its square root is the pivot integer.
3. Check whether the square of the computed root equals `total`.

This confirms whether `total` is actually a perfect square.
4. If the check succeeds, return the square root.

Otherwise, return `-1`.

### Why it works

The derivation shows that the pivot condition simplifies exactly to:

$$x^2 = \frac{n(n+1)}{2}$$

Therefore, a pivot integer exists if and only if the total sum is a perfect square. The square root of that sum is guaranteed to be the unique pivot integer.

## Python Solution

```python
class Solution:
    def pivotInteger(self, n: int) -> int:
        total_sum = n * (n + 1) // 2
        
        pivot = int(total_sum ** 0.5)
        
        if pivot * pivot == total_sum:
            return pivot
        
        return -1
```

The implementation starts by calculating the total sum from `1` to `n` using the arithmetic progression formula. This avoids iterating through the numbers manually.

Next, the code computes the square root of the total sum. Since Python's exponentiation with `0.5` produces a floating point result, we convert it to an integer.

The important verification step is checking whether squaring the candidate root recreates the original total sum. If it does, the total sum is a perfect square, and the root is the pivot integer.

If the check fails, there is no valid pivot, so the function returns `-1`.

## Go Solution

```go
package main

import "math"

func pivotInteger(n int) int {
    totalSum := n * (n + 1) / 2

    pivot := int(math.Sqrt(float64(totalSum)))

    if pivot*pivot == totalSum {
        return pivot
    }

    return -1
}
```

The Go implementation follows the same mathematical logic as the Python version.

One Go specific detail is that `math.Sqrt` operates on `float64`, so we explicitly convert the integer sum to `float64` before computing the square root. Afterward, we convert the result back to `int`.

Since the constraints are very small, integer overflow is not a concern in this problem.

First compute the total sum:

$$\text{total} = \frac{8 \times 9}{2} = 36$$

Now compute the square root:

$$\sqrt{36} = 6$$

Check whether:

$$6^2 = 36$$

This is true, so the pivot integer is `6`.

Verification:

```
n = 1
```

Compute total sum:

$$\text{total} = \frac{1 \times 2}{2} = 1$$

Square root:

$$\sqrt{1} = 1$$

Check:

$$1^2 = 1$$

The pivot integer is `1`.

### Example 3

Input:

```
n = 4
```

Compute total sum:

$$\text{total} = \frac{4 \times 5}{2} = 10$$

Square root:

$$\sqrt{10} \approx 3.16$$

Integer portion is `3`.

Check:

$$3^2 = 9$$

Since `9 != 10`, the total sum is not a perfect square, so there is no pivot integer.

Return:

```
-1
```

## Edge Cases

### Edge Case 1: Single Element Range

When `n = 1`, the sequence contains only one number. Both the left sum and right sum are simply `1`, so the pivot exists immediately.

This case is important because some implementations may incorrectly assume the pivot must divide the sequence into two non-empty sections. The mathematical solution handles it naturally because the total sum is `1`, which is a perfect square.

### Edge Case 2: No Valid Pivot

Many values of `n` do not produce a valid pivot integer. For example:

```
n = 4
```

The total sum is `10`, which is not a perfect square. A buggy implementation might accidentally return the truncated square root or incorrectly compare floating point values.

The implementation avoids this by explicitly checking:

```
pivot * pivot == total_sum
```

This guarantees correctness.

### Edge Case 3: Larger Inputs Near the Constraint Limit

Although `n <= 1000` is small, larger values still test whether the mathematical derivation is implemented correctly.

For example:

```
n = 1000
```

The total sum becomes:

$$500500$$

Since this is not a perfect square, the answer should be `-1`.

The implementation handles this efficiently in constant time without any iteration, making it robust even if the constraints were significantly larger.
