## Solution

## Problem Understanding

In this problem, we manage a set of meeting rooms and simulate how meetings are assigned over time.

We are given `n` rooms labeled from `0` to `n - 1`, and a list of meetings where each meeting has a start time and an end time. Every meeting duration is fixed as:

```
duration = end - start
```

The scheduling rules are extremely important:

First, if one or more rooms are currently free when a meeting starts, the meeting must be assigned to the free room with the smallest room number.

Second, if every room is occupied, the meeting cannot be discarded. Instead, it must wait until the earliest room becomes available. Once that room becomes free, the delayed meeting begins immediately and keeps the same duration as originally planned.

Third, if multiple meetings are waiting, the one with the earlier original start time gets priority. Fortunately, the problem guarantees that all start times are unique, which simplifies tie-breaking.

The goal is to count how many meetings each room hosts, then return the room that handled the most meetings. If multiple rooms handled the same maximum number, we return the smallest room index.

The constraints are important:

- `n <= 100`
- `meetings.length <= 100000`

Even though the number of rooms is small, the number of meetings is very large. This immediately tells us that repeatedly scanning schedules or simulating time unit by unit would be too slow. We need an efficient event-based simulation.

Several edge cases are important:

- Multiple meetings may become delayed many times in a row.
- Several rooms may free up at the same timestamp.
- Meetings may start exactly when another ends, since intervals are half-open `[start, end)`.
- Multiple rooms may tie for maximum usage count.
- A meeting might be delayed far beyond its original start time.

The uniqueness of start times guarantees that delayed meetings can always be processed in chronological order without ambiguity.

## Approaches

### Brute Force Approach

A straightforward simulation would process meetings in chronological order and explicitly track the schedule for every room.

For each meeting, we could scan all rooms to determine:

- Which rooms are currently available.
- If none are free, which room becomes available earliest.

We would then update that room's schedule and increment its usage count.

This approach is correct because it directly follows the problem rules. However, it becomes inefficient because every meeting may scan all rooms and repeatedly inspect scheduling information.

If implemented naively with room schedules stored as lists, the complexity can degrade significantly when there are many meetings.

### Key Insight for the Optimal Solution

The core observation is that we only need two kinds of information efficiently:

- Which rooms are currently free, and among them, the smallest room number.
- Which room becomes available next.

These are classic priority queue operations.

We use two heaps:

- An `available_rooms` min-heap storing free room numbers.
- A `busy_rooms` min-heap storing `(end_time, room_number)`.

The busy heap lets us quickly identify the earliest room that becomes free.

As we process meetings in sorted order:

- First, we release every room whose meeting has already ended.
- If a room is available, we assign the meeting immediately.
- Otherwise, we delay the meeting until the earliest room becomes free.

This gives an efficient event simulation with logarithmic heap operations.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m * n) or worse | O(n) | Scans rooms repeatedly for each meeting |
| Optimal | O(m log n) | O(n) | Uses two priority queues for efficient scheduling |

Here, `m` is the number of meetings.

## Algorithm Walkthrough

1. Sort all meetings by their start time.

The scheduling rules depend on chronological order, so processing meetings from earliest to latest guarantees correctness.
2. Create a min-heap of available rooms.

Initially, every room is free, so the heap contains:

```
[0, 1, 2, ..., n - 1]
```

Using a min-heap ensures we always select the smallest available room number.
3. Create another min-heap for busy rooms.

Each entry stores:

```
(meeting_end_time, room_number)
```

The heap is ordered primarily by end time, which lets us efficiently find the next room that becomes available.
4. Maintain an array `meeting_count`.

`meeting_count[i]` stores how many meetings room `i` has hosted.
5. Process each meeting `(start, end)` in sorted order.

Before assigning the current meeting, release every room whose meeting has already ended.

While the earliest ending meeting satisfies:

```
busy_end_time <= current_start
```

remove it from `busy_rooms` and push its room number into `available_rooms`.
6. If at least one room is available, assign the meeting immediately.

Pop the smallest room number from `available_rooms`.

The meeting runs from:

```
[start, end)
```

Push:

```
(end, room)
```

into `busy_rooms`.
7. If no rooms are available, delay the meeting.

Pop the room with the earliest finishing time from `busy_rooms`.

Suppose:

```
earliest_end = t
duration = end - start
```

Then the delayed meeting runs from:

```
[t, t + duration)
```

Push:

```
(t + duration, room)
```

back into the heap.
8. Increment the meeting count for the assigned room.
9. After processing all meetings, scan `meeting_count` and return the room with the maximum count.

If multiple rooms tie, choose the smallest index.

### Why it works

The algorithm maintains two critical invariants:

- `available_rooms` always contains exactly the currently free rooms.
- `busy_rooms` always contains exactly the occupied rooms ordered by earliest finishing time.

Because meetings are processed in chronological order, and delayed meetings always start at the earliest possible release time, the simulation exactly matches the scheduling rules. The heaps guarantee that we always choose the correct room according to the problem constraints.

## Python Solution

```python
from typing import List
import heapq

class Solution:
    def mostBooked(self, n: int, meetings: List[List[int]]) -> int:
        meetings.sort()

        available_rooms = list(range(n))
        heapq.heapify(available_rooms)

        # (end_time, room_number)
        busy_rooms = []

        meeting_count = [0] * n

        for start, end in meetings:
            # Free all rooms that have completed meetings
            while busy_rooms and busy_rooms[0][0] <= start:
                end_time, room = heapq.heappop(busy_rooms)
                heapq.heappush(available_rooms, room)

            duration = end - start

            # If a room is available, use it immediately
            if available_rooms:
                room = heapq.heappop(available_rooms)
                heapq.heappush(busy_rooms, (end, room))
            else:
                # Delay the meeting until the earliest room becomes free
                earliest_end, room = heapq.heappop(busy_rooms)
                new_end = earliest_end + duration
                heapq.heappush(busy_rooms, (new_end, room))

            meeting_count[room] += 1

        max_meetings = max(meeting_count)

        for room in range(n):
            if meeting_count[room] == max_meetings:
                return room

        return 0
```

The implementation follows the algorithm directly.

First, meetings are sorted by start time so they can be processed chronologically.

The `available_rooms` heap stores free room numbers. Since Python's `heapq` is a min-heap, the smallest room number is always chosen automatically.

The `busy_rooms` heap stores `(end_time, room)` pairs. The room with the earliest ending meeting appears at the top.

Before handling each meeting, the code releases every room whose end time is less than or equal to the current meeting's start time. Those rooms are pushed back into the available heap.

If at least one room is free, the meeting starts immediately in the smallest-numbered available room.

Otherwise, the meeting is delayed. We remove the room with the earliest finishing time and extend its schedule by the current meeting's duration.

Finally, we count meetings per room and return the smallest room with the maximum count.

## Go Solution

```go
package main

import (
	"container/heap"
	"sort"
)

type IntHeap []int

func (h IntHeap) Len() int {
	return len(h)
}

func (h IntHeap) Less(i, j int) bool {
	return h[i] < h[j]
}

func (h IntHeap) Swap(i, j int) {
	h[i], h[j] = h[j], h[i]
}

func (h *IntHeap) Push(x interface{}) {
	*h = append(*h, x.(int))
}

func (h *IntHeap) Pop() interface{} {
	old := *h
	n := len(old)
	val := old[n-1]
	*h = old[:n-1]
	return val
}

type BusyRoom struct {
	endTime int64
	room    int
}

type BusyHeap []BusyRoom

func (h BusyHeap) Len() int {
	return len(h)
}

func (h BusyHeap) Less(i, j int) bool {
	if h[i].endTime == h[j].endTime {
		return h[i].room < h[j].room
	}
	return h[i].endTime < h[j].endTime
}

func (h BusyHeap) Swap(i, j int) {
	h[i], h[j] = h[j], h[i]
}

func (h *BusyHeap) Push(x interface{}) {
	*h = append(*h, x.(BusyRoom))
}

func (h *BusyHeap) Pop() interface{} {
	old := *h
	n := len(old)
	val := old[n-1]
	*h = old[:n-1]
	return val
}

func mostBooked(n int, meetings [][]int) int {
	sort.Slice(meetings, func(i, j int) bool {
		return meetings[i][0] < meetings[j][0]
	})

	available := &IntHeap{}
	for i := 0; i < n; i++ {
		*available = append(*available, i)
	}
	heap.Init(available)

	busy := &BusyHeap{}
	heap.Init(busy)

	count := make([]int, n)

	for _, meeting := range meetings {
		start := int64(meeting[0])
		end := int64(meeting[1])

		for busy.Len() > 0 && (*busy)[0].endTime <= start {
			roomInfo := heap.Pop(busy).(BusyRoom)
			heap.Push(available, roomInfo.room)
		}

		duration := end - start

		var room int

		if available.Len() > 0 {
			room = heap.Pop(available).(int)
			heap.Push(busy, BusyRoom{
				endTime: end,
				room:    room,
			})
		} else {
			roomInfo := heap.Pop(busy).(BusyRoom)
			room = roomInfo.room

			newEnd := roomInfo.endTime + duration

			heap.Push(busy, BusyRoom{
				endTime: newEnd,
				room:    room,
			})
		}

		count[room]++
	}

	bestRoom := 0

	for i := 1; i < n; i++ {
		if count[i] > count[bestRoom] {
			bestRoom = i
		}
	}

	return bestRoom
}
```

The Go solution mirrors the Python logic closely, but Go requires explicit heap implementations using the `container/heap` package.

Two custom heap types are defined:

- `IntHeap` for available room numbers.
- `BusyHeap` for occupied rooms ordered by end time.

The Go version uses `int64` for time calculations because delayed meetings can accumulate large end times after many postponements.

Unlike Python, Go does not have built-in priority queues, so all heap interface methods must be implemented manually.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m log n) | Each meeting performs heap operations |
| Space | O(n) | The heaps store at most `n` rooms |

Here, `m` is the number of meetings.

Sorting meetings costs:

```
O(m log m)
```

However, each meeting only performs a small number of heap operations on at most `n` rooms. Since `n <= 100`, heap operations are extremely cheap.

The dominant simulation work is:

```
O(m log n)
```

The total complexity is therefore:

```
O(m log m + m log n)
```

which is easily fast enough for `100000` meetings.

## Edge Cases

One important edge case occurs when every meeting overlaps and only one room exists. In this situation, every meeting after the first must be delayed. A buggy implementation may accidentally reuse the original end time instead of preserving the meeting duration. This implementation correctly computes:

```
new_end = earliest_end + duration
```

which preserves the original meeting length after delay.

Another important case occurs when meetings start exactly when another meeting ends. Since the intervals are half-open `[start, end)`, a room finishing at time `t` is immediately available for another meeting starting at `t`. The implementation handles this correctly using:

```
while busy_rooms and busy_rooms[0][0] <= start:
```

The `<=` is essential. Using `<` would incorrectly delay valid meetings.

A third tricky case happens when multiple rooms become free at the same time. The problem requires choosing the smallest available room number. Because freed rooms are pushed into the `available_rooms` min-heap, the next assignment automatically picks the smallest room index.

Another subtle edge case involves ties in the final answer. Multiple rooms may host the same maximum number of meetings. The implementation scans rooms in ascending order and returns the first room with the maximum count, guaranteeing the smallest valid index is returned.
