## Solution

## Problem Understanding

We are given a binary string `s` consisting only of `'0'` and `'1'`, along with two coprime integers `num1` and `num2`. The task is to count how many non-empty substrings satisfy a very specific condition:

- The number of `0`s in the substring divided by the number of `1`s in the substring must equal `num1 : num2`.

In other words, for a substring to be valid:

$$\frac{\text{count of 0s}}{\text{count of 1s}} = \frac{\text{num1}}{\text{num2}}$$

Since `num1` and `num2` are coprime, this ratio is already in reduced form. A substring is valid whenever its counts are proportional to these two values.

For example, if `num1 = 1` and `num2 = 2`, then valid substrings include:

- one `0` and two `1`s
- two `0`s and four `1`s
- three `0`s and six `1`s

and so on.

The input size can be as large as `10^5`, which immediately tells us that an `O(n^2)` or `O(n^3)` solution will be too slow. We need something close to linear time.

Several edge cases are important:

- Strings containing only one character
- Strings with only `0`s or only `1`s
- Ratios that are impossible to achieve
- Multiple overlapping valid substrings
- Large inputs where performance matters

The problem guarantees that `num1` and `num2` are coprime, which simplifies the ratio handling because we do not need additional reduction logic.

## Approaches

### Brute Force Approach

The most direct solution is to examine every possible substring.

For each substring:

1. Count how many `0`s and `1`s it contains.
2. Check whether:

$$\text{zeros} \times \text{num2} = \text{ones} \times \text{num1}$$

This works because cross multiplication avoids floating point precision issues.

A naive implementation would generate all substrings and count characters inside each substring independently. That leads to `O(n^3)` complexity.

We can improve it slightly using prefix sums:

- Build prefix counts of zeros and ones.
- Compute counts for any substring in `O(1)` time.

That reduces the complexity to `O(n^2)`.

Although correct, `O(n^2)` is still too slow for `n = 100000`.

### Optimal Approach

The key observation is that we can transform the ratio condition into a prefix-sum equality problem.

Suppose:

- `z` = number of zeros
- `o` = number of ones

We want:

$$\frac{z}{o} = \frac{num1}{num2}$$

Cross multiplying:

$$z \cdot num2 = o \cdot num1$$

Now define a transformed prefix value:

$$value = zeros \cdot num2 - ones \cdot num1$$

For a substring between indices `l` and `r`, the substring is valid if:

$$(prefix_r - prefix_l) = 0$$

which means:

$$prefix_r = prefix_l$$

This becomes identical to the classic "count subarrays with equal prefix sums" pattern.

We maintain:

- a running transformed prefix value
- a hash map storing how many times each value has appeared

Whenever the same transformed value appears again, every previous occurrence forms a valid substring ending at the current position.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Uses prefix counts to evaluate every substring |
| Optimal | O(n) | O(n) | Uses transformed prefix sums and a hash map |

## Algorithm Walkthrough

1. Initialize a hash map called `frequency`.

This map stores how many times each transformed prefix value has appeared so far.
2. Insert the value `0` into the map with frequency `1`.

This handles substrings that start at index `0`.
3. Maintain two running counts:

- `zeros`
- `ones`
4. Iterate through the string character by character.

For each character:

- If the character is `'0'`, increment `zeros`
- Otherwise increment `ones`
5. Compute the transformed prefix value:

$$current = zeros \cdot num2 - ones \cdot num1$$

1. Check how many times this value has appeared before.

If it has appeared `k` times, then there are `k` valid substrings ending at the current position.
2. Add that frequency to the answer.
3. Increment the frequency of the current transformed value in the hash map.
4. Continue until the entire string is processed.

### Why it works

The transformed prefix value represents the imbalance between zeros and ones relative to the desired ratio.

If two prefix positions have the same transformed value, subtracting them gives:

$$(z_2 - z_1)\cdot num2 = (o_2 - o_1)\cdot num1$$

which means the substring between those positions has exactly the required ratio.

Thus, counting equal transformed prefix values is equivalent to counting valid substrings.

## Python Solution

```python
from collections import defaultdict

class Solution:
    def fixedRatio(self, s: str, num1: int, num2: int) -> int:
        frequency = defaultdict(int)
        frequency[0] = 1

        zeros = 0
        ones = 0
        answer = 0

        for ch in s:
            if ch == '0':
                zeros += 1
            else:
                ones += 1

            current = zeros * num2 - ones * num1

            answer += frequency[current]

            frequency[current] += 1

        return answer
```

The implementation follows the mathematical transformation directly.

The `frequency` hash map stores how many times each transformed prefix value has occurred. We initialize `frequency[0] = 1` because before processing any characters, the transformed value is zero once.

As we scan the string:

- encountering `'0'` increases the zero count
- encountering `'1'` increases the one count

We compute:

```python
current = zeros * num2 - ones * num1
```

If this same value has appeared previously, every earlier occurrence corresponds to a valid substring ending at the current position.

The answer accumulates all such matches.

Because each character is processed exactly once and all hash map operations are average `O(1)`, the algorithm runs efficiently for very large inputs.

## Go Solution

```go
func fixedRatio(s string, num1 int, num2 int) int64 {
	frequency := make(map[int]int64)
	frequency[0] = 1

	zeros := 0
	ones := 0
	var answer int64 = 0

	for _, ch := range s {
		if ch == '0' {
			zeros++
		} else {
			ones++
		}

		current := zeros*num2 - ones*num1

		answer += frequency[current]

		frequency[current]++
	}

	return answer
}
```

The Go implementation mirrors the Python logic closely.

One important difference is that the return type is `int64`. The number of substrings can become very large, potentially exceeding the range of a 32-bit integer. Therefore both the answer and the map frequencies are stored as `int64`.

Go maps return the zero value automatically for missing keys, which simplifies the implementation because we can directly read `frequency[current]` even if the key has never appeared before.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each character is processed once |
| Space | O(n) | The hash map may store up to `n + 1` distinct prefix values |

The algorithm performs a constant amount of work per character:

- update counts
- compute transformed value
- hash map lookup
- hash map update

All operations are average `O(1)`, leading to overall linear complexity.

The hash map can contain at most one entry per prefix position, so the space complexity is linear as well.

## Edge Cases

One important edge case is a string containing only one type of character, such as `"00000"` or `"11111"`. In these situations, one side of the ratio is always zero, which can never match a positive coprime ratio like `1:1` or `2:3`. A buggy implementation might accidentally divide by zero or incorrectly treat these as valid. The prefix transformation approach avoids this entirely because it uses only integer arithmetic.

Another critical edge case involves substrings beginning at index `0`. For example, `"01"` with ratio `1:1` should count the entire string. Without initializing the frequency map with `frequency[0] = 1`, these substrings would be missed because there would be no earlier prefix to pair with the current one.

A third important edge case is overlapping valid substrings. In strings like `"0101"` with ratio `1:1`, multiple valid substrings share characters. A naive implementation that greedily matches substrings could undercount. The prefix-frequency method correctly counts every valid pair of equal transformed values, ensuring all overlapping substrings are included.

A final edge case concerns very large outputs. For strings of length `100000`, the number of valid substrings can become extremely large, approaching quadratic magnitude in the worst case. The Go solution uses `int64` to avoid integer overflow, ensuring correctness for all valid inputs.
