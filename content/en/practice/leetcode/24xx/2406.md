# LeetCode 2406, Divide Intervals Into Minimum Number of Groups

## Problem Understanding

The problem gives us a list of inclusive intervals, where each interval is represented as `[left, right]`. We must divide all intervals into groups such that no two intervals inside the same group intersect.

The important detail is that the intervals are inclusive. This means two intervals overlap even if they only share a single endpoint. For example:

- `[1, 5]` and `[5, 8]` overlap because both include `5`
- `[1, 5]` and `[6, 8]` do not overlap

The goal is to compute the minimum number of groups required so that every interval belongs to exactly one group and no overlaps exist within a group.

Another way to think about the problem is:

- Each group represents a collection of non-overlapping intervals
- Whenever multiple intervals overlap at the same time, they must belong to different groups
- Therefore, the answer is actually the maximum number of simultaneously active intervals

The constraints are large:

- Up to `100000` intervals
- Interval values up to `10^6`

This immediately tells us that quadratic solutions are too slow. Any approach near `O(n^2)` will time out. Efficient sorting-based solutions around `O(n log n)` are expected.

Several edge cases are especially important:

- Intervals that touch at endpoints, because the intervals are inclusive
- Completely nested intervals such as `[1,10]`, `[2,9]`, `[3,8]`
- Many identical intervals
- Already non-overlapping intervals
- Large inputs where efficiency matters

The inclusive overlap condition is the most common source of mistakes. Since `[1,5]` intersects `[5,7]`, we cannot reuse a group if the previous interval ends exactly where the next one starts.

## Approaches

### Brute Force Approach

A straightforward solution is to process intervals one by one and try placing each interval into an existing group.

For every interval:

1. Check every existing group
2. Determine whether the interval overlaps with any interval already inside that group
3. If it fits, place it there
4. Otherwise create a new group

To check whether an interval can fit into a group, we may need to compare it against multiple intervals already stored there.

This approach is correct because it explicitly enforces the grouping rules. However, it becomes extremely slow when the number of intervals grows large. In the worst case, we may compare every interval against almost every other interval, leading to quadratic complexity.

With `100000` intervals, this is not feasible.

### Optimal Approach, Sweep Line with Min Heap

The key observation is that the answer equals the maximum number of overlapping intervals at any moment.

Instead of explicitly building groups, we can track when groups become available.

The strategy is:

1. Sort intervals by start time
2. Use a min heap to store the ending times of currently active groups
3. For each interval:

- If the smallest ending time is strictly smaller than the current start, that group can be reused
- Otherwise we need a new group

The heap efficiently tells us which group becomes free first.

The inclusive overlap condition changes the reuse rule slightly:

- We can reuse a group only if `previous_end < current_start`
- If `previous_end == current_start`, the intervals still overlap

This greedy strategy works because reusing the group with the earliest ending interval leaves maximum flexibility for future intervals.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Tries placing each interval into existing groups manually |
| Optimal | O(n log n) | O(n) | Uses sorting and a min heap to track active groups |

## Algorithm Walkthrough

1. Sort all intervals by their starting value.

Sorting ensures we process intervals in chronological order. This allows us to reason about which previously used groups may now be available.
2. Create a min heap.

The heap stores the ending times of intervals currently occupying groups. The smallest ending time is always at the top.
3. Process intervals one by one.

For each interval `[start, end]`, check the smallest end value in the heap.
4. Determine whether a group can be reused.

If the smallest end value is strictly smaller than `start`, then that interval no longer overlaps with the current interval.

Since intervals are inclusive, equality still means overlap.

Therefore:

- Reusable if `heap[0] < start`
- Not reusable if `heap[0] >= start`
5. Reuse a group if possible.

If a reusable group exists:

- Remove the smallest ending time from the heap
- This means that group is now free
6. Assign the current interval to a group.

Push the current interval's end value into the heap.

This marks the group as occupied until `end`.
7. Continue until all intervals are processed.
8. Return the heap size.

The heap size represents the number of simultaneously active groups. The maximum size reached during processing is the minimum number of groups required.

### Why it works

The heap always contains the ending times of intervals currently occupying groups. By always reusing the group with the earliest ending interval, we greedily free resources as soon as possible.

If even the earliest-ending group still overlaps with the current interval, then every active group overlaps, meaning a new group is unavoidable.

Thus, the algorithm creates a new group only when absolutely necessary, guaranteeing the minimum number of groups.

## Python Solution

```python
from typing import List
import heapq

class Solution:
    def minGroups(self, intervals: List[List[int]]) -> int:
        intervals.sort()

        min_heap = []

        for start, end in intervals:
            if min_heap and min_heap[0] < start:
                heapq.heappop(min_heap)

            heapq.heappush(min_heap, end)

        return len(min_heap)
```

The implementation begins by sorting the intervals by their starting values. Python's default list sorting handles this naturally for lists of two integers.

A min heap is then used to store the ending times of active groups. The heap property allows us to efficiently access the smallest ending value in `O(1)` time and update the structure in `O(log n)` time.

For each interval, the algorithm checks whether the earliest-ending group can be reused. Because the intervals are inclusive, reuse is only possible when the stored end value is strictly smaller than the current start.

If reuse is possible, the old ending time is removed from the heap. Then the current interval's end is inserted, representing that the group is now occupied until the new interval finishes.

At the end, the heap size equals the number of groups needed.

## Go Solution

```go
package main

import (
	"container/heap"
	"sort"
)

type MinHeap []int

func (h MinHeap) Len() int {
	return len(h)
}

func (h MinHeap) Less(i, j int) bool {
	return h[i] < h[j]
}

func (h MinHeap) Swap(i, j int) {
	h[i], h[j] = h[j], h[i]
}

func (h *MinHeap) Push(x interface{}) {
	*h = append(*h, x.(int))
}

func (h *MinHeap) Pop() interface{} {
	old := *h
	n := len(old)
	value := old[n-1]
	*h = old[:n-1]
	return value
}

func minGroups(intervals [][]int) int {
	sort.Slice(intervals, func(i, j int) bool {
		return intervals[i][0] < intervals[j][0]
	})

	minHeap := &MinHeap{}
	heap.Init(minHeap)

	for _, interval := range intervals {
		start := interval[0]
		end := interval[1]

		if minHeap.Len() > 0 && (*minHeap)[0] < start {
			heap.Pop(minHeap)
		}

		heap.Push(minHeap, end)
	}

	return minHeap.Len()
}
```

The Go solution follows the same logic as the Python version but requires an explicit heap implementation using Go's `container/heap` package.

The custom `MinHeap` type implements the required heap interface methods. The algorithm itself remains identical:

- Sort intervals by start
- Reuse a group if the smallest ending time is strictly smaller than the current start
- Push the current ending time into the heap

No special integer overflow handling is necessary because the interval bounds are well within Go's standard integer range.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | Sorting costs O(n log n), each heap operation costs O(log n) |
| Space | O(n) | Heap may store all intervals in worst case |

The sorting step dominates the runtime complexity. Each interval is inserted into the heap exactly once, and possibly removed once. Since heap operations take logarithmic time, the overall complexity remains `O(n log n)`.

The heap can grow to size `n` in the worst case, such as when all intervals overlap.

## Edge Cases

### Intervals Touching at Endpoints

A very common mistake is treating intervals like half-open ranges instead of inclusive ranges.

For example:

```
[1,5] and [5,10]
```

These intervals overlap because both contain `5`.

The implementation correctly handles this by only reusing a group when:

```
previous_end < current_start
```

Using `<=` would produce incorrect results.

### Completely Nested Intervals

Nested intervals such as:

```
[1,10], [2,9], [3,8]
```

all overlap simultaneously. None of them can share a group.

The heap naturally handles this case because every new interval overlaps with the earliest-ending active interval, forcing creation of additional groups.

### Large Numbers of Identical Intervals

If many intervals are identical:

```
[1,100], [1,100], [1,100]
```

then every interval overlaps with every other interval.

The heap grows to size `n`, correctly representing that every interval requires its own group.

This also validates that the implementation handles worst-case memory usage safely within constraints.
