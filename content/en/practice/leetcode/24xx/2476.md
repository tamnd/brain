## Solution

## Problem Understanding

This problem asks us to find, for a given list of queries, the closest nodes in a binary search tree (BST). Specifically, for each query value, we need to find two numbers in the BST: the largest value that is **less than or equal** to the query (`mini`) and the smallest value that is **greater than or equal** to the query (`maxi`). If no such value exists in either case, we use `-1`.

The input is a BST, represented as a root node, and an array of query integers. The output is a 2D array where each subarray corresponds to the result for each query.

Key constraints include up to 100,000 nodes in the BST and up to 100,000 queries, which immediately rules out naive approaches that traverse the tree for every query individually. Each node value and query value is within the range 1 to 1,000,000. We must therefore use an approach that leverages the properties of a BST or preprocessed structures for fast lookups.

Important edge cases include queries smaller than the smallest node, larger than the largest node, or exactly equal to some node values. The problem guarantees at least two nodes in the tree, so we do not have to handle empty trees.

## Approaches

The **brute-force approach** would involve, for each query, performing a traversal of the tree to find the closest nodes. For `n` queries and `m` nodes, this results in a time complexity of O(n * m), which is infeasible for the maximum constraints (`n` and `m` up to 10^5).

The **optimal approach** uses the property of a BST that an in-order traversal yields sorted node values. By performing a single in-order traversal, we can extract all node values into a sorted array. Then, for each query, we perform **binary search** on this sorted array to find the largest number ≤ query and the smallest number ≥ query. This reduces the query processing to O(log m) per query, giving a total complexity of O(m + n log m).

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * m) | O(1) | Traverse BST for each query, too slow for large inputs |
| Optimal | O(m + n log m) | O(m) | In-order traversal + binary search on sorted array of node values |

## Algorithm Walkthrough

1. Perform an in-order traversal of the BST to collect all node values into a sorted array `values`.
2. Initialize an empty array `answer` to store results.
3. For each query `q`:

1. Use binary search to find the index of the smallest number ≥ `q` (`right_idx`).
2. Use binary search or careful index manipulation to find the largest number ≤ `q` (`left_idx`).
3. If no value exists on either side, use `-1`.
4. Append `[mini, maxi]` to `answer`.
4. Return `answer`.

Why it works: the in-order traversal guarantees a sorted array of BST node values. Binary search ensures efficient retrieval of closest smaller or equal and closest larger or equal elements for each query.

## Python Solution

```python
from typing import List, Optional
import bisect

# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

class Solution:
    def closestNodes(self, root: Optional[TreeNode], queries: List[int]) -> List[List[int]]:
        def inorder(node):
            if not node:
                return
            inorder(node.left)
            values.append(node.val)
            inorder(node.right)
        
        values = []
        inorder(root)
        
        answer = []
        for q in queries:
            idx_right = bisect.bisect_left(values, q)
            if idx_right < len(values) and values[idx_right] == q:
                mini = maxi = values[idx_right]
            else:
                mini = values[idx_right - 1] if idx_right > 0 else -1
                maxi = values[idx_right] if idx_right < len(values) else -1
            answer.append([mini, maxi])
        
        return answer
```

The Python solution first generates a sorted array `values` from the BST using in-order traversal. For each query, it uses `bisect_left` to efficiently find the insertion point, then determines `mini` and `maxi` based on the index and edge cases.

## Go Solution

```go
package main

import "sort"

/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func closestNodes(root *TreeNode, queries []int) [][]int {
    var values []int
    var inorder func(node *TreeNode)
    inorder = func(node *TreeNode) {
        if node == nil {
            return
        }
        inorder(node.Left)
        values = append(values, node.Val)
        inorder(node.Right)
    }
    inorder(root)
    
    answer := make([][]int, len(queries))
    for i, q := range queries {
        idx := sort.SearchInts(values, q)
        var mini, maxi int
        if idx < len(values) && values[idx] == q {
            mini, maxi = values[idx], values[idx]
        } else {
            if idx > 0 {
                mini = values[idx-1]
            } else {
                mini = -1
            }
            if idx < len(values) {
                maxi = values[idx]
            } else {
                maxi = -1
            }
        }
        answer[i] = []int{mini, maxi}
    }
    return answer
}
```

In Go, we use `sort.SearchInts` which behaves like `bisect_left` in Python. `nil` checks are used to safely traverse the BST. Slice operations handle appending and storing the final answer.

## Worked Examples

**Example 1:** `root = [6,2,13,1,4,9,15,null,null,null,null,null,null,14], queries = [2,5,16]`

In-order traversal produces `[1,2,4,6,9,13,14,15]`.

| Query | Index | mini | maxi |
| --- | --- | --- | --- |
| 2 | 1 | 2 | 2 |
| 5 | 3 | 4 | 6 |
| 16 | 8 | 15 | -1 |

Answer: `[[2,2],[4,6],[15,-1]]`

**Example 2:** `root = [4,null,9], queries = [3]`

In-order traversal produces `[4,9]`.

| Query | Index | mini | maxi |
| --- | --- | --- | --- |
| 3 | 0 | -1 | 4 |

Answer: `[[-1,4]]`

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m + n log m) | O(m) for in-order traversal to collect values, O(n log m) for binary search on sorted array for n queries |
| Space | O(m) | Store the sorted array of BST node values |

The dominant cost is the binary search for each query. In-order traversal guarantees the BST property is fully leveraged, and no extra tree traversals are needed per query.

## Edge Cases

**Query smaller than all nodes:** For a BST like `[5,3,7]`, querying `2` would require returning `[-1,3]`. Our binary search correctly returns `-1` for `mini` when the index is 0.

**Query larger than all nodes:** For the same BST, querying `8` returns `[7,-1]`.
