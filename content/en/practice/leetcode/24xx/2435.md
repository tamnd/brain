## Solution

## Problem Understanding

The problem gives us a matrix `grid` of size `m x n` and an integer `k`. We start at the top-left cell `(0, 0)` and want to reach the bottom-right cell `(m - 1, n - 1)`. At every step, we are only allowed to move either one cell to the right or one cell downward.

Every path accumulates a sum consisting of all values visited along that path. Our goal is to count how many valid paths produce a total sum divisible by `k`.

More formally, if the sum of values along a path is `S`, then the path is considered valid if:

$S \bmod k = 0$

The answer can become extremely large because the number of possible paths grows combinatorially, so we return the result modulo:

$10^9 + 7$

The constraints are very important here:

- `1 <= m * n <= 5 * 10^4`
- `1 <= k <= 50`

The product `m * n` can be as large as 50,000, meaning the grid may be very wide or very tall. Any exponential solution is completely infeasible.

The small bound on `k` is the key observation. Since `k <= 50`, we only care about path sums modulo `k`, not the full sum itself. This strongly suggests a dynamic programming approach where the DP state tracks the current remainder modulo `k`.

Several edge cases matter:

- A single-cell grid where the answer depends entirely on whether `grid[0][0] % k == 0`
- `k = 1`, where every number is divisible by `1`, meaning every path is valid
- Large grids with many possible paths, where integer overflow or inefficient recursion could become problematic
- Grids filled with zeros, where every path produces remainder `0`

## Approaches

### Brute Force Approach

A straightforward solution is to generate every possible path from the top-left corner to the bottom-right corner using either DFS or backtracking.

At each step, we can move:

- right
- down

For every complete path, we compute the path sum and check whether it is divisible by `k`.

This approach is correct because it explicitly enumerates every valid path and tests the divisibility condition directly.

However, the number of paths is enormous. The total number of paths in an `m x n` grid is:

$\binom{m+n-2}{m-1}$

This grows exponentially for large grids. Even a modest `100 x 100` grid would have an astronomically large number of paths.

Therefore, brute force is far too slow.

### Optimal Dynamic Programming Approach

The key insight is that we do not need the exact path sum. We only care about its remainder modulo `k`.

Suppose we already know:

- how many ways reach cell `(r, c)`
- with a current remainder `rem`

Then after adding `grid[r][c]`, we can compute the new remainder efficiently:

$newRem = (rem + grid[r][c]) \bmod k$

This allows us to define a DP state:

- `dp[r][c][rem]`
- number of ways to reach `(r, c)` with path sum remainder `rem`

Since `k <= 50`, the remainder dimension is small enough to manage efficiently.

Instead of tracking huge sums, we compress all possibilities into only `k` remainder classes.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | O(m + n) | Enumerates every path recursively |
| Optimal Dynamic Programming | O(m × n × k) | O(m × n × k) | Tracks path sums modulo `k` |

## Algorithm Walkthrough

1. Create a 3D DP table where `dp[r][c][rem]` stores the number of ways to reach cell `(r, c)` such that the path sum modulo `k` equals `rem`.
2. Initialize the starting position. The only path initially is the single starting cell itself. Its remainder is:

$grid[0][0] \bmod k$

Therefore:

- `dp[0][0][grid[0][0] % k] = 1`
3. Iterate through every cell in row-major order.
4. For each cell `(r, c)`, examine all possible remainders `rem`.
5. If there are `count` ways to reach `(r, c)` with remainder `rem`, then propagate those paths:

- downward to `(r + 1, c)`
- rightward to `(r, c + 1)`
6. When moving into a neighboring cell containing value `val`, compute the updated remainder:

$nextRem = (rem + val) \bmod k$
7. Add the current count into the neighbor's DP state:

- `dp[nextRow][nextCol][nextRem] += count`
8. Always apply modulo `10^9 + 7` to avoid overflow.
9. After processing all cells, the final answer is:

- `dp[m - 1][n - 1][0]`

because remainder `0` means divisibility by `k`.

### Why it works

The DP invariant is:

- `dp[r][c][rem]` always represents the exact number of valid paths reaching `(r, c)` whose path sum modulo `k` equals `rem`.

Every path reaching a neighbor must come from either the top or left cell. Since we propagate all remainder states correctly and update remainders using modular arithmetic, every possible path is counted exactly once.

Because modulo arithmetic preserves divisibility relationships, tracking only remainders is sufficient.

## Python Solution

```python
from typing import List

class Solution:
    def numberOfPaths(self, grid: List[List[int]], k: int) -> int:
        MOD = 10**9 + 7
        
        rows = len(grid)
        cols = len(grid[0])
        
        dp = [[[0] * k for _ in range(cols)] for _ in range(rows)]
        
        start_remainder = grid[0][0] % k
        dp[0][0][start_remainder] = 1
        
        for r in range(rows):
            for c in range(cols):
                for rem in range(k):
                    current_count = dp[r][c][rem]
                    
                    if current_count == 0:
                        continue
                    
                    # Move down
                    if r + 1 < rows:
                        next_rem = (rem + grid[r + 1][c]) % k
                        dp[r + 1][c][next_rem] += current_count
                        dp[r + 1][c][next_rem] %= MOD
                    
                    # Move right
                    if c + 1 < cols:
                        next_rem = (rem + grid[r][c + 1]) % k
                        dp[r][c + 1][next_rem] += current_count
                        dp[r][c + 1][next_rem] %= MOD
        
        return dp[rows - 1][cols - 1][0]
```

The implementation begins by allocating a 3D DP table. The third dimension corresponds to all possible remainders from `0` to `k - 1`.

The starting cell initializes exactly one state, corresponding to the remainder of the starting value.

The nested loops iterate through:

- every row
- every column
- every remainder state

For each reachable state, we propagate the path count to neighboring cells.

The modulo operation serves two purposes:

- computing remainder transitions
- preventing integer overflow in the path counts

At the end, we return the number of paths ending with remainder `0`.

## Go Solution

```go
package main

func numberOfPaths(grid [][]int, k int) int {
	const MOD int = 1_000_000_007

	rows := len(grid)
	cols := len(grid[0])

	dp := make([][][]int, rows)

	for r := 0; r < rows; r++ {
		dp[r] = make([][]int, cols)

		for c := 0; c < cols; c++ {
			dp[r][c] = make([]int, k)
		}
	}

	startRem := grid[0][0] % k
	dp[0][0][startRem] = 1

	for r := 0; r < rows; r++ {
		for c := 0; c < cols; c++ {
			for rem := 0; rem < k; rem++ {
				currentCount := dp[r][c][rem]

				if currentCount == 0 {
					continue
				}

				// Move down
				if r+1 < rows {
					nextRem := (rem + grid[r+1][c]) % k
					dp[r+1][c][nextRem] =
						(dp[r+1][c][nextRem] + currentCount) % MOD
				}

				// Move right
				if c+1 < cols {
					nextRem := (rem + grid[r][c+1]) % k
					dp[r][c+1][nextRem] =
						(dp[r][c+1][nextRem] + currentCount) % MOD
				}
			}
		}
	}

	return dp[rows-1][cols-1][0]
}
```

The Go implementation mirrors the Python solution closely.

A 3D slice is used instead of nested Python lists. Since Go arrays require fixed sizes, slices are the natural choice.

The modulo constant is explicitly typed as `int` to avoid type mismatch issues.

Go does not automatically resize multidimensional structures, so every level of the DP slice must be allocated manually.

## Worked Examples

### Example 1

```
grid = [
    [5, 2, 4],
    [3, 0, 5],
    [0, 7, 2]
]
k = 3
```

We track remainders modulo `3`.

### Initial State

| Cell | Remainder Counts |
| --- | --- |
| (0,0) | [0,0,1] |

Because:

$5 \bmod 3 = 2$

### From (0,0)

Moving right to `2`:

$(2 + 2) \bmod 3 = 1$

Moving down to `3`:

$(2 + 3) \bmod 3 = 2$

| Cell | Remainder Counts |
| --- | --- |
| (0,1) | [0,1,0] |
| (1,0) | [0,0,1] |

The process continues similarly for all cells.

At the end:

| Cell | Remainder Counts |
| --- | --- |
| (2,2) | [2,2,2] |

The answer is the count for remainder `0`:

```
2
```

### Example 2

```
grid = [[0,0]]
k = 5
```

Initial remainder:

$0 \bmod 5 = 0$

DP state:

| Cell | Remainder Counts |
| --- | --- |
| (0,0) | [1,0,0,0,0] |

Move right:

| Cell | Remainder Counts |
| --- | --- |
| (0,1) | [1,0,0,0,0] |

Final answer:

```
1
```

### Example 3

```
grid = [
    [7,3,4,9],
    [2,3,6,2],
    [2,3,7,0]
]
k = 1
```

Every number modulo `1` equals `0`.

Therefore every path automatically satisfies divisibility.

The number of paths from top-left to bottom-right is:

$\binom{5}{2} = 10$

So the answer is:

```
10
```

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m × n × k) | Each cell processes all `k` remainder states |
| Space | O(m × n × k) | DP table stores `k` states per cell |

The algorithm processes every cell exactly once, and for each cell it iterates through all possible remainders from `0` to `k - 1`.

Since `k <= 50`, this is efficient enough even when the grid contains 50,000 cells.

## Edge Cases

A single-cell grid is an important corner case because there are no moves at all. The algorithm must correctly interpret the starting cell as a complete path. Our initialization handles this naturally because `dp[0][0][grid[0][0] % k] = 1`.

The case `k = 1` is another subtle scenario. Since every integer is divisible by `1`, every path should count. A buggy implementation might overcomplicate remainder logic or accidentally divide by zero. Our solution works correctly because every remainder becomes `0`.

Very large grids can expose performance issues. A recursive DFS solution would either time out or overflow the recursion stack. The iterative DP avoids recursion entirely and processes states efficiently in `O(m × n × k)` time.

Grids containing many zeros are also important. Multiple distinct paths can produce identical remainders repeatedly. Our DP accumulates counts rather than overwriting them, ensuring all valid paths are preserved correctly.
