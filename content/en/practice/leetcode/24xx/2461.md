## Solution

## Problem Understanding

The problem gives us an integer array `nums` and an integer `k`. We must examine every contiguous subarray of length exactly `k`, but only consider the subarrays where all elements are distinct. Among all valid subarrays, we return the maximum possible sum.

In other words, we are searching for a window of size `k` that satisfies two conditions:

1. The window length is exactly `k`
2. No value appears more than once inside the window

For every valid window, we compute its sum and keep track of the largest one.

The input array can contain up to `10^5` elements, which is large enough that we cannot repeatedly scan subarrays or recompute sums from scratch. The values themselves are also up to `10^5`, which means a hash-based frequency structure works well.

The important observation is that the problem asks specifically for contiguous subarrays, not arbitrary subsets or subsequences. This makes sliding window techniques especially appropriate.

There are several edge cases that matter:

- If every length-`k` subarray contains duplicates, the answer must be `0`
- If `k = 1`, every single element is automatically distinct, so the answer becomes the maximum element
- If the entire array length equals `k`, we only check one window
- Large values and large array sizes mean repeated summation would be too slow
- Duplicate-heavy arrays can easily break naive distinctness checks if frequencies are not maintained carefully

Because of the input size constraints, an efficient linear-time solution is required.

## Approaches

### Brute Force Approach

The most direct solution is to examine every subarray of length `k`.

For each starting index:

1. Extract the subarray of length `k`
2. Check whether all elements are distinct
3. If they are distinct, compute the sum
4. Update the maximum answer

To verify distinctness, we could insert elements into a set and check whether duplicates appear. Computing the sum could either be done directly in `O(k)` time or with a prefix sum array.

This approach is correct because it explicitly checks every possible candidate window. However, it is too slow for the constraints.

There are approximately `n` windows, and each one requires up to `O(k)` work. In the worst case:

- `n = 10^5`
- `k = 10^5`

This leads to `O(n * k)` complexity, which can become `10^10` operations.

### Optimal Sliding Window Approach

The key observation is that consecutive windows overlap heavily.

Suppose we move from:

```
[1,5,4]
```

to

```
[5,4,2]
```

Most elements remain the same. Only one element leaves the window and one new element enters.

Instead of rebuilding the window information from scratch every time, we can maintain it incrementally.

We use:

- A sliding window of size `k`
- A hash map that stores frequencies of elements inside the current window
- A running window sum

As we expand the window:

- Add the incoming number
- Remove the outgoing number once the window exceeds size `k`
- If the window size is `k` and the number of distinct keys equals `k`, then all elements are distinct

This transforms the solution into linear time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * k) | O(k) | Rechecks every window independently |
| Optimal | O(n) | O(k) | Sliding window with frequency map |

## Algorithm Walkthrough

1. Initialize a frequency map to track how many times each number appears inside the current window.
2. Maintain a variable `window_sum` that stores the sum of elements currently inside the window. This avoids recomputing sums repeatedly.
3. Maintain a variable `max_sum` to store the best valid window sum found so far.
4. Iterate through the array using a right pointer.
5. For each new element:

- Add it to `window_sum`
- Increment its frequency in the hash map
6. If the window size becomes larger than `k`, remove the leftmost element:

- Subtract it from `window_sum`
- Decrease its frequency
- Remove the key entirely if its frequency becomes zero
7. Once the window size reaches exactly `k`, check whether all elements are distinct.

This condition is true when:

```
len(frequency_map) == k
```

because the window already has exactly `k` elements. If there are `k` distinct values, every value appears exactly once.
8. If the window is valid, update `max_sum`.
9. Continue until all windows have been processed.

### Why it works

The sliding window always represents exactly one contiguous subarray. The frequency map accurately tracks how many times each value appears inside that window. Because the window size is fixed at `k`, having exactly `k` distinct keys guarantees that every element is unique. Since every possible length-`k` window is processed exactly once, the algorithm correctly finds the maximum valid sum.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Solution:
    def maximumSubarraySum(self, nums: List[int], k: int) -> int:
        frequency = defaultdict(int)
        
        window_sum = 0
        max_sum = 0
        left = 0
        
        for right in range(len(nums)):
            current = nums[right]
            
            window_sum += current
            frequency[current] += 1
            
            # Shrink window if it becomes larger than k
            if right - left + 1 > k:
                outgoing = nums[left]
                
                window_sum -= outgoing
                frequency[outgoing] -= 1
                
                if frequency[outgoing] == 0:
                    del frequency[outgoing]
                
                left += 1
            
            # Check valid window
            if right - left + 1 == k:
                if len(frequency) == k:
                    max_sum = max(max_sum, window_sum)
        
        return max_sum
```

The implementation follows the sliding window strategy directly.

The `frequency` hash map stores the count of each element currently inside the window. This allows duplicate detection in constant time.

The `window_sum` variable is updated incrementally. When a new value enters the window, we add it. When a value leaves, we subtract it. This keeps every update efficient.

The `left` pointer tracks the beginning of the window, while the loop variable `right` expands the window from left to right.

Whenever the window exceeds size `k`, we shrink it from the left side. This guarantees the window never becomes unnecessarily large.

Finally, when the window size equals `k`, we verify whether all values are distinct by checking the number of keys in the frequency map.

## Go Solution

```go
func maximumSubarraySum(nums []int, k int) int64 {
    frequency := make(map[int]int)

    var windowSum int64 = 0
    var maxSum int64 = 0

    left := 0

    for right := 0; right < len(nums); right++ {
        current := nums[right]

        windowSum += int64(current)
        frequency[current]++

        // Shrink window if needed
        if right-left+1 > k {
            outgoing := nums[left]

            windowSum -= int64(outgoing)
            frequency[outgoing]--

            if frequency[outgoing] == 0 {
                delete(frequency, outgoing)
            }

            left++
        }

        // Check valid window
        if right-left+1 == k {
            if len(frequency) == k {
                if windowSum > maxSum {
                    maxSum = windowSum
                }
            }
        }
    }

    return maxSum
}
```

The Go implementation is structurally identical to the Python version. The main difference is integer handling.

The problem can produce sums larger than a 32-bit integer, so the function returns `int64`. The running sum variables are also stored as `int64`.

Go maps require explicit deletion using `delete(map, key)` when a frequency becomes zero.

Unlike Python, Go does not have a built-in default dictionary, so frequencies are managed manually using `map[int]int`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each element enters and leaves the window at most once |
| Space | O(k) | The frequency map stores at most `k` distinct values |

The algorithm is linear because both pointers move forward only once. Every insertion and deletion in the hash map is amortized `O(1)`.

The extra space comes from the frequency map. In the best case it may contain fewer than `k` elements, but in the worst case all `k` elements are distinct.

## Edge Cases

One important edge case occurs when no valid subarray exists. Arrays such as `[4,4,4]` with `k = 3` never produce a window containing distinct values. A buggy implementation might accidentally return the sum of an invalid window. This solution avoids that problem by updating the answer only when `len(frequency) == k`.

Another important case is when `k = 1`. Every single-element subarray is automatically distinct because duplicates cannot exist inside a window of size one. The algorithm handles this naturally because every window of size one will always have exactly one distinct key.

A third important edge case is when the entire array length equals `k`. In this situation there is only one candidate window. Some implementations accidentally shrink the window too early or mishandle boundary conditions. This solution checks the window only after it reaches size `k`, ensuring the single full-length window is evaluated correctly.

A subtle edge case involves repeated values entering and leaving the window frequently, such as `[1,2,1,2,1,2]`. Incorrect frequency management can leave stale keys inside the map, causing the distinctness check to fail. This implementation deletes keys whenever their frequency reaches zero, keeping the map accurate at all times.
