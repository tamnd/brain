# LeetCode 2436 - Minimum Split Into Subarrays With GCD Greater Than One

## Problem Understanding

This problem asks us to divide an array into the smallest possible number of contiguous subarrays such that the GCD of every subarray is strictly greater than `1`.

The input array `nums` contains positive integers. We must partition the array into one or more contiguous pieces, where every element belongs to exactly one piece. For each resulting subarray, we compute the GCD of all elements inside it. A subarray is valid only if that GCD is greater than `1`.

The goal is to minimize the number of subarrays.

A key detail is that the subarrays must be contiguous. We are not allowed to rearrange elements or arbitrarily group them. Every split point divides the array into consecutive sections.

The constraints are important:

- `nums.length <= 2000`
- `nums[i] <= 10^9`

The array size is moderate, which means an `O(n^2)` solution is acceptable. However, the values themselves can be very large, so any approach involving factorization of every number must be designed carefully.

The most important mathematical observation is how GCD behaves when extending a subarray. As more numbers are added, the GCD can only stay the same or decrease. Once the GCD becomes `1`, adding more elements can never increase it again.

Several edge cases are important:

- If the entire array already has GCD greater than `1`, the answer is `1`.
- If adjacent elements frequently produce GCD `1`, many splits may be required.
- Since every number is at least `2`, every single-element subarray is always valid because its GCD equals the number itself.
- Large values up to `10^9` require efficient GCD computation rather than factorization-heavy solutions.

## Approaches

### Brute Force Approach

A brute force solution would try every possible way to split the array and check whether all resulting subarrays have GCD greater than `1`.

For an array of length `n`, there are `2^(n-1)` possible ways to place split points between elements. For each partitioning, we would compute the GCD of every subarray and verify validity.

This approach is correct because it explores every possible partition and selects the one with the minimum number of subarrays.

However, the number of possible partitions grows exponentially. Even for moderate values of `n`, this becomes computationally infeasible.

### Optimal Greedy Approach

The key observation is that we should extend the current subarray as long as its GCD remains greater than `1`.

Suppose we maintain the GCD of the current subarray while scanning from left to right.

- If adding the next number keeps the GCD greater than `1`, then extending the subarray is always beneficial because it reduces the total number of splits.
- If adding the next number makes the GCD equal to `1`, then the current subarray can no longer remain valid. Since adding more numbers cannot increase the GCD again, we must split before this element.

This leads to a simple greedy strategy:

- Maintain the running GCD of the current segment.
- Continue extending while the GCD stays above `1`.
- Whenever the GCD becomes `1`, start a new subarray from the current element.

This greedy approach is optimal because it always creates the longest possible valid subarray before splitting.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n * n) | O(n) | Tries every possible partition |
| Optimal Greedy | O(n log M) | O(1) | Maintains running GCD while scanning |

Here, `M` represents the maximum value in `nums`.

## Algorithm Walkthrough

1. Initialize the answer as `1` because at least one subarray is always needed.
2. Set `current_gcd` to the first element of the array. This represents the GCD of the current subarray being built.
3. Iterate through the remaining elements in the array.
4. For each number, compute the GCD between `current_gcd` and the current number. This gives the GCD of the extended subarray.
5. If the updated GCD is greater than `1`, continue extending the current subarray because it is still valid.
6. If the updated GCD becomes `1`, then the current subarray cannot include this element. Since GCD cannot increase after adding more numbers, we must split here.
7. Increment the answer because a new subarray is required.
8. Start the new subarray using the current element alone, so set `current_gcd` to the current number.
9. Continue until all elements are processed.
10. Return the total number of subarrays.

### Why it works

The algorithm relies on the monotonic behavior of GCD. As elements are added to a subarray, the GCD can only decrease or remain unchanged.

Once the GCD becomes `1`, no future extension can restore it to a value greater than `1`. Therefore, splitting immediately is necessary and optimal.

Conversely, if the GCD remains greater than `1`, extending the current subarray is always beneficial because it avoids unnecessary splits. This greedy strategy always produces the minimum number of valid subarrays.

## Python Solution

```python
from typing import List
from math import gcd

class Solution:
    def minimumSplits(self, nums: List[int]) -> int:
        splits = 1
        current_gcd = nums[0]

        for i in range(1, len(nums)):
            current_gcd = gcd(current_gcd, nums[i])

            if current_gcd == 1:
                splits += 1
                current_gcd = nums[i]

        return splits
```

The solution begins by initializing the number of required subarrays as `1`, because at least one valid segment must exist.

The variable `current_gcd` stores the GCD of the current subarray being extended.

As we iterate through the array, we continuously update the running GCD using Python's built-in `gcd` function from the `math` module.

Whenever the GCD becomes `1`, the current subarray is no longer valid. At that moment, we increment the split counter and begin a new subarray starting from the current element.

Because every element is at least `2`, every single-element subarray is valid, so restarting from the current element is always safe.

## Go Solution

```go
package main

func gcd(a int, b int) int {
	for b != 0 {
		a, b = b, a%b
	}
	return a
}

func minimumSplits(nums []int) int {
	splits := 1
	currentGCD := nums[0]

	for i := 1; i < len(nums); i++ {
		currentGCD = gcd(currentGCD, nums[i])

		if currentGCD == 1 {
			splits++
			currentGCD = nums[i]
		}
	}

	return splits
}
```

The Go implementation follows exactly the same logic as the Python version.

Since Go does not provide a built-in GCD function for integers, we implement the Euclidean algorithm manually.

The solution uses constant extra memory and iterates through the array only once.

There are no integer overflow concerns because GCD operations only reduce values and all numbers fit comfortably within Go's `int` range.

## Edge Cases

One important edge case occurs when the entire array already has a GCD greater than `1`. In this situation, no split is necessary, and the answer should be `1`. The implementation handles this naturally because the running GCD never becomes `1`, so the split counter is never incremented.

Another important case occurs when every adjacent pair produces GCD `1`. In that scenario, every element must become its own subarray. This can easily cause off-by-one mistakes if the split counter is initialized incorrectly. The implementation starts with one subarray already counted and increments only when a split becomes necessary.

A third edge case involves very large values close to `10^9`. Factorization-based approaches may become unnecessarily expensive or complicated. The implementation avoids this entirely by relying only on Euclid's algorithm for GCD computation, which remains extremely efficient even for large integers.

A subtle case appears when the GCD decreases gradually but remains above `1` for several steps before eventually becoming `1`. The greedy algorithm correctly keeps extending the subarray until the exact moment validity is lost, ensuring the minimum number of splits.
