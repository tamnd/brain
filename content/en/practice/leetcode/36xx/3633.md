# LeetCode 3633 - Earliest Finish Time for Land and Water Rides I

## Problem Understanding

The problem gives us two groups of rides:

- Land rides, where each ride has an opening time and a duration.
- Water rides, where each ride has an opening time and a duration.

A tourist must ride exactly one land ride and exactly one water ride. The rides can be taken in either order:

- Land ride first, then water ride.
- Water ride first, then land ride.

A ride cannot start before its opening time, but it may start at any later time. Once a ride starts, it runs for its duration and finishes at:

$$\text{start time} + \text{duration}$$

After completing the first ride, the tourist immediately attempts to board the second ride. If the second ride is already open, it can be started immediately. Otherwise, the tourist must wait until its opening time.

The goal is to find the earliest possible finishing time after completing one ride from each category.

The input consists of four arrays:

- `landStartTime[i]` is the opening time of land ride `i`.
- `landDuration[i]` is the duration of land ride `i`.
- `waterStartTime[j]` is the opening time of water ride `j`.
- `waterDuration[j]` is the duration of water ride `j`.

The output is a single integer representing the minimum possible completion time.

The constraints are very small:

- At most 100 land rides.
- At most 100 water rides.

This means there are at most:

$$100 \times 100 = 10,000$$

possible land-water pairs, which is a very manageable number.

Some important edge cases include situations where one ride opens much later than the other, forcing waiting time. Another important case is when the optimal solution requires taking the rides in the less obvious order. We must evaluate both possible orders for every pair of rides. The problem guarantees that there is at least one land ride and one water ride, so a valid answer always exists.

## Approaches

### Brute Force

The most direct solution is to try every possible combination of:

- One land ride.
- One water ride.
- Both possible ride orders.

For a chosen land ride `i` and water ride `j`, we can compute:

1. Land first, then water.
2. Water first, then land.

For each scenario we calculate the final completion time and keep the minimum over all possibilities.

Since there are only `n * m` ride pairs and two orders per pair, this approach is already efficient enough.

### Key Insight

The important observation is that the constraints are extremely small.

For every land-water pair, the completion time can be computed directly in constant time. There is no need for sorting, binary search, or advanced optimization.

For a land ride:

$$\text{landFinish}
=
\text{landStartTime}[i]
+
\text{landDuration}[i]$$

If the water ride is taken second, it can start at:

$$\max(\text{landFinish}, \text{waterStartTime}[j])$$

and finish at:

$$\max(\text{landFinish}, \text{waterStartTime}[j])
+
\text{waterDuration}[j]$$

The reverse order is computed symmetrically.

Because every valid schedule is represented by some pair and one of the two orders, enumerating all possibilities guarantees the optimal answer.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(nm) | O(1) | Check every land-water pair and both ride orders |
| Optimal | O(nm) | O(1) | Same approach, constraints are small enough that exhaustive enumeration is optimal |

## Algorithm Walkthrough

1. Initialize an answer variable with a very large value.
2. Iterate through every land ride index `i`.
3. For each land ride, iterate through every water ride index `j`.
4. Compute the finish time of the land ride if started at its opening time:

$$\text{landFinish}
=
\text{landStartTime}[i]
+
\text{landDuration}[i]$$
5. Compute the earliest time the water ride can start after finishing the land ride:

$$\text{waterStart}
=
\max(\text{landFinish}, \text{waterStartTime}[j])$$
6. Compute the completion time for the order:

$$\text{land} \rightarrow \text{water}$$

and update the answer.
7. Compute the finish time of the water ride if started at its opening time:

$$\text{waterFinish}
=
\text{waterStartTime}[j]
+
\text{waterDuration}[j]$$
8. Compute the earliest time the land ride can start after finishing the water ride:

$$\text{landStart}
=
\max(\text{waterFinish}, \text{landStartTime}[i])$$
9. Compute the completion time for the order:

$$\text{water} \rightarrow \text{land}$$

and update the answer.
10. After all pairs have been processed, return the minimum answer found.

### Why it works

Every valid itinerary consists of exactly one land ride and one water ride, taken in one of two possible orders. The algorithm explicitly evaluates every land-water pair and both possible orders. For each case it computes the earliest achievable completion time while respecting ride opening times. Since all feasible schedules are examined and the minimum completion time is recorded, the returned value must be the globally optimal answer.

## Python Solution

```python
from typing import List

class Solution:
    def earliestFinishTime(
        self,
        landStartTime: List[int],
        landDuration: List[int],
        waterStartTime: List[int],
        waterDuration: List[int]
    ) -> int:
        answer = float("inf")

        for i in range(len(landStartTime)):
            land_finish = landStartTime[i] + landDuration[i]

            for j in range(len(waterStartTime)):
                water_finish = waterStartTime[j] + waterDuration[j]

                # Land -> Water
                finish_time = max(land_finish, waterStartTime[j]) + waterDuration[j]
                answer = min(answer, finish_time)

                # Water -> Land
                finish_time = max(water_finish, landStartTime[i]) + landDuration[i]
                answer = min(answer, finish_time)

        return answer
```

The implementation follows the algorithm directly.

The outer loop selects a land ride and precomputes its finishing time when started at its opening time. The inner loop selects a water ride and similarly computes its finishing time.

For the `land -> water` order, the water ride can only begin after both conditions are satisfied: the land ride has finished and the water ride has opened. This is captured by the `max(...)` expression.

For the `water -> land` order, the same logic applies with the ride categories reversed.

The minimum completion time across all examined schedules is stored in `answer` and returned at the end.

## Go Solution

```go
func earliestFinishTime(landStartTime []int, landDuration []int, waterStartTime []int, waterDuration []int) int {
	ans := int(1 << 60)

	for i := 0; i < len(landStartTime); i++ {
		landFinish := landStartTime[i] + landDuration[i]

		for j := 0; j < len(waterStartTime); j++ {
			waterFinish := waterStartTime[j] + waterDuration[j]

			// Land -> Water
			finishTime := max(landFinish, waterStartTime[j]) + waterDuration[j]
			if finishTime < ans {
				ans = finishTime
			}

			// Water -> Land
			finishTime = max(waterFinish, landStartTime[i]) + landDuration[i]
			if finishTime < ans {
				ans = finishTime
			}
		}
	}

	return ans
}

func max(a, b int) int {
	if a > b {
		return a
	}
	return b
}
```

The Go implementation is essentially identical to the Python version. Since all values are small, integer overflow is not a concern. A helper `max` function is used because Go does not provide a built-in integer maximum function. No additional slices or data structures are required, so the space complexity remains constant.

The algorithm performs a constant amount of work for each land-water pair. Since there are `n` land rides and `m` water rides, the total running time is `O(nm)`. No auxiliary data structures proportional to the input size are created, so the space usage is constant.

## Edge Cases

One important edge case occurs when one ride opens much later than the other. A solution that always takes land first or always takes water first could miss the optimal schedule. The implementation explicitly evaluates both ride orders for every pair, ensuring the correct answer is found.

Another edge case occurs when the second ride has not opened by the time the first ride finishes. In this situation the tourist must wait. The expressions using `max(...)` correctly model this waiting period by delaying the second ride's start until its opening time.

A third edge case involves multiple rides with identical or very similar finishing times. It is easy to accidentally stop after finding a good candidate. The implementation examines every possible pair and continuously updates the global minimum, guaranteeing that the absolute best schedule is returned.

A final edge case is the smallest input size, where there is exactly one land ride and one water ride. The algorithm still works correctly because the nested loops execute exactly once and evaluate both possible orders before returning the better result.
