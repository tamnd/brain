# LeetCode 3615 - Longest Palindromic Path in Graph

## Problem Understanding

We are given an undirected graph with `n` nodes. Each node has a lowercase character label. A valid path may start at any node and move along graph edges, with the restriction that every node can be visited at most once.

For any valid simple path, we can read the characters associated with the visited nodes in path order and obtain a string. The goal is to find the maximum possible length of a path whose corresponding string is a palindrome.

A palindrome reads the same forward and backward. For example, `"aba"`, `"bb"`, and `"racecar"` are palindromes, while `"abc"` and `"abca"` are not.

The graph contains at most `14` nodes. This constraint is extremely important because it suggests that algorithms involving subsets of vertices are feasible. Since:

$$2^{14} = 16384$$

bitmask dynamic programming becomes a natural candidate.

Several edge cases are worth noticing immediately.

A graph may contain only one node, in which case the answer is always `1`.

A graph may contain many edges but no multi-node palindromic path. In that situation, any individual node forms a palindrome of length `1`.

A palindrome may have odd length, with a single center node, or even length, with a center edge connecting two equal characters.

The graph is not necessarily a tree. It may contain cycles, which means a naive DFS that enumerates all simple paths can revisit the same structural situations many times.

The graph guarantees no duplicate edges and no self-loops.

## Approaches

### Brute Force

The most direct approach is to enumerate every simple path in the graph.

For every starting node, we perform a DFS that explores all possible simple paths. Whenever a path is extended, we construct its string and check whether it is a palindrome.

This approach is correct because every valid simple path is examined, so the longest palindromic one will eventually be found.

Unfortunately, the number of simple paths in a graph can grow exponentially. Even with only fourteen vertices, the total number of simple paths can become extremely large. Repeated palindrome checks further increase the cost.

As a result, brute force is impractical.

### Optimal Observation

A palindrome can be constructed from the center outward.

For an odd-length palindrome, the center is a single node.

For an even-length palindrome, the center is an edge whose two endpoint characters are equal.

Once a valid palindromic path exists, we can extend it by adding one node to each end. The newly added endpoint characters must be equal, otherwise the resulting string cannot remain a palindrome.

This suggests a dynamic programming state that represents an already constructed palindromic path. We then grow the palindrome symmetrically from both ends.

Because `n ≤ 14`, we can represent the set of used vertices with a bitmask.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential, roughly $O(n!)$ in dense graphs | $O(n)$ | Enumerates all simple paths |
| Optimal | $O(2^n \cdot n^2)$ states, each expanded over endpoint neighbors | $O(2^n \cdot n^2)$ | Bitmask DP over palindromic path endpoints |

## Algorithm Walkthrough

### State Definition

We define a DP state:

$$(mask, u, v)$$

where:

- `mask` contains exactly the vertices used by the path.
- `u` and `v` are the two endpoints of the path.
- The path represented by this state forms a palindrome.

If such a state exists, then there is at least one simple palindromic path using exactly the vertices in `mask` and having endpoints `u` and `v`.

### Initialization

1. Every single node forms a palindrome of length `1`.

Therefore, for every vertex `i`, create the state:

$$(1 \ll i,\ i,\ i)$$
2. Every edge connecting equal characters forms a palindrome of length `2`.

If an edge `(a,b)` exists and:

$$label[a] = label[b]$$

then create the state:

$$((1 \ll a) \mid (1 \ll b),\ a,\ b)$$

### Transition

1. Consider a valid state `(mask, u, v)`.
2. Let `nu` be a neighbor of `u` not already in `mask`.
3. Let `nv` be a neighbor of `v` not already in `mask`.
4. The two newly added endpoint characters must match:

$$label[nu] = label[nv]$$
5. The two added vertices must be distinct:

$$nu \ne nv$$

because every node may be visited at most once.
6. If the conditions hold, extend the palindrome:

$$newMask = mask \cup \{nu, nv\}$$

and create the new state:

$$(newMask, nu, nv)$$

### Answer Maintenance

1. Every reachable state corresponds to a valid palindromic path.
2. The length of that path equals the number of vertices contained in `mask`.
3. Track the maximum popcount among all reachable states.

### Why it works

Every palindrome can be reduced inward by repeatedly removing matching endpoint characters. Eventually it reaches either a single center node or a center edge consisting of two equal characters. Those are exactly our base states.

Conversely, every transition adds equal characters to both ends of an already valid palindrome, preserving the palindrome property while maintaining a simple path because newly added vertices are not already present in the mask.

Thus the DP generates exactly the set of all palindromic simple paths, and the largest reachable mask size is the answer.

## Python Solution

```python
from typing import List
from collections import deque

class Solution:
    def maxLen(self, n: int, edges: List[List[int]], label: str) -> int:
        graph = [[] for _ in range(n)]

        for u, v in edges:
            graph[u].append(v)
            graph[v].append(u)

        queue = deque()
        visited = set()

        answer = 1

        # Odd-length centers.
        for i in range(n):
            mask = 1 << i
            state = (mask, i, i)

            visited.add(state)
            queue.append(state)

        # Even-length centers.
        for u, v in edges:
            if label[u] == label[v]:
                mask = (1 << u) | (1 << v)

                a, b = (u, v) if u <= v else (v, u)
                state = (mask, a, b)

                if state not in visited:
                    visited.add(state)
                    queue.append(state)

                answer = 2

        while queue:
            mask, u, v = queue.popleft()

            answer = max(answer, mask.bit_count())

            for nu in graph[u]:
                if mask & (1 << nu):
                    continue

                for nv in graph[v]:
                    if mask & (1 << nv):
                        continue

                    if nu == nv:
                        continue

                    if label[nu] != label[nv]:
                        continue

                    new_mask = mask | (1 << nu) | (1 << nv)

                    a, b = (nu, nv) if nu <= nv else (nv, nu)
                    state = (new_mask, a, b)

                    if state not in visited:
                        visited.add(state)
                        queue.append(state)

        return answer
```

### Implementation Explanation

The graph is stored as an adjacency list because we repeatedly need to enumerate neighbors of the current endpoints.

The `visited` set stores all reachable DP states. Each state contains the vertex mask and the two current endpoints.

All single-node palindromes are inserted first. These represent odd-length palindrome centers.

Next, every edge whose endpoint characters match is inserted. These represent even-length palindrome centers.

A BFS-style traversal processes all reachable states. For each state, we try to extend the palindrome from both ends simultaneously. The candidate vertices must be unused and must carry the same character.

Whenever a new state is discovered, it is added to both the queue and the visited set. The answer is updated using the number of set bits in the mask, which equals the path length represented by that state.

## Go Solution

```go
package main

type State struct {
	mask int
	u    int
	v    int
}

func maxLen(n int, edges [][]int, label string) int {
	graph := make([][]int, n)

	for _, e := range edges {
		u, v := e[0], e[1]
		graph[u] = append(graph[u], v)
		graph[v] = append(graph[v], u)
	}

	visited := make(map[State]bool)
	queue := make([]State, 0)

	answer := 1

	// Odd-length centers.
	for i := 0; i < n; i++ {
		state := State{
			mask: 1 << i,
			u:    i,
			v:    i,
		}

		visited[state] = true
		queue = append(queue, state)
	}

	// Even-length centers.
	for _, e := range edges {
		u, v := e[0], e[1]

		if label[u] == label[v] {
			mask := (1 << u) | (1 << v)

			a, b := u, v
			if a > b {
				a, b = b, a
			}

			state := State{
				mask: mask,
				u:    a,
				v:    b,
			}

			if !visited[state] {
				visited[state] = true
				queue = append(queue, state)
			}

			answer = 2
		}
	}

	head := 0

	for head < len(queue) {
		cur := queue[head]
		head++

		length := 0
		x := cur.mask
		for x > 0 {
			x &= x - 1
			length++
		}

		if length > answer {
			answer = length
		}

		for _, nu := range graph[cur.u] {
			if (cur.mask & (1 << nu)) != 0 {
				continue
			}

			for _, nv := range graph[cur.v] {
				if (cur.mask & (1 << nv)) != 0 {
					continue
				}

				if nu == nv {
					continue
				}

				if label[nu] != label[nv] {
					continue
				}

				newMask := cur.mask | (1 << nu) | (1 << nv)

				a, b := nu, nv
				if a > b {
					a, b = b, a
				}

				nextState := State{
					mask: newMask,
					u:    a,
					v:    b,
				}

				if !visited[nextState] {
					visited[nextState] = true
					queue = append(queue, nextState)
				}
			}
		}
	}

	return answer
}
```

### Go-Specific Notes

The Go solution mirrors the Python version almost exactly.

A struct is used as the DP state key because structs are comparable and can be used directly inside a map.

Instead of Python's `bit_count()`, the implementation counts set bits using the classic `x &= x - 1` technique.

The queue is implemented with a slice and a moving head index, avoiding expensive front removals.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | $O(2^n \cdot n^2)$ states, with neighbor expansion on each state | Every reachable mask-endpoint combination is processed once |
| Space | $O(2^n \cdot n^2)$ | Storage for visited DP states |

The number of masks is at most `2^14 = 16384`. For each mask, there can be at most `n²` endpoint pairs. Therefore the theoretical state space is `O(2^n · n²)`, which is manageable for `n ≤ 14`.

## Edge Cases

### Single Vertex Graph

When `n = 1`, there are no edges and the only valid path consists of the single vertex itself. A common bug is assuming every palindrome must be built from an edge or expanded from two endpoints. The implementation avoids this by inserting every vertex as an initial length-1 palindrome state.

### Even-Length Palindromes

Many palindrome algorithms focus only on a single center node. That would miss paths such as `"aa"` or `"abba"`. The implementation explicitly initializes every edge whose endpoint characters are equal, allowing even-length palindromes to be generated correctly.

### Dense Graphs with Cycles

In graphs containing cycles, many different exploration orders can reach the same logical palindrome state. Without memoization, the search could repeatedly process equivalent situations. The `(mask, u, v)` visited structure guarantees that each state is explored at most once, preventing exponential duplication.

### Reusing the Same Vertex

A subtle bug would be allowing both ends of an expansion to choose the same unused vertex. That would place one node at two positions in the path simultaneously, violating the requirement that every node is visited at most once. The implementation explicitly checks `nu != nv` before creating a new state.
