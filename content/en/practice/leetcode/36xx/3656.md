# Determine if a Simple Graph Exists (LeetCode 3656)

## Problem Understanding

This problem asks whether a given degree sequence can correspond to some **undirected simple graph**.

We are given an array `degrees` where `degrees[i]` is the desired degree of vertex `i`. A degree represents the number of edges incident to that vertex.

Our goal is to determine whether there exists **at least one** simple graph whose vertex degrees match the array exactly.

A simple graph has two important restrictions:

1. No self-loops, meaning a vertex cannot connect to itself.
2. No parallel edges, meaning there can be at most one edge between any pair of vertices.

If such a graph exists, we return `true`. Otherwise, we return `false`.

The constraints are significant:

- `n` can be as large as `100,000`.
- Each degree is between `0` and `n - 1`.

The large value of `n` immediately tells us that any solution involving explicit graph construction or repeated degree updates in quadratic time will be too slow.

Mathematically, this is the classical problem of determining whether a degree sequence is **graphical**, meaning whether it can be realized by some simple undirected graph.

Several edge cases are important:

- A single vertex with degree `0` is valid.
- A single vertex with degree `1` is impossible because self-loops are forbidden.
- The sum of all degrees must be even, because every edge contributes exactly two to the total degree count.
- Large degree values near `n - 1` can force many connections and may make the sequence impossible.
- Many zero-degree vertices can coexist with highly connected vertices only if the constraints remain consistent.

The key challenge is efficiently testing whether the sequence is graphical.

## Approaches

### Brute Force

A brute-force solution would attempt to construct every possible simple graph on `n` vertices and check whether any of them has the desired degree sequence.

This is clearly correct because it examines all possible graphs. If a valid graph exists, it will eventually be found.

Unfortunately, the number of simple graphs on `n` vertices is

$$2^{\frac{n(n-1)}{2}}$$

since every possible edge may either exist or not exist.

Even for relatively small values of `n`, this becomes astronomically large, making brute force completely infeasible.

### Key Insight

This is a classic graph theory problem.

A degree sequence is graphical if and only if it satisfies the **Erdős-Gallai theorem**.

After sorting degrees in non-increasing order:

$$d_1 \ge d_2 \ge \cdots \ge d_n$$

the sequence is graphical exactly when:

1. The total degree sum is even.
2. For every $k$ from $1$ to $n$,

$$\sum_{i=1}^{k} d_i \le k(k-1) + \sum_{i=k+1}^{n} \min(d_i,k)$$

The theorem gives a complete characterization of graphical sequences.

The challenge is evaluating all $n$ inequalities efficiently.

A direct implementation would require $O(n^2)$, which is too slow for $n = 100,000$.

The optimization comes from:

- Sorting the degrees.
- Computing prefix sums.
- Using binary search to locate where values become smaller than $k$.

This reduces the complexity to $O(n \log n)$.

## Algorithm Walkthrough

### Step 1: Sort degrees in descending order

The Erdős-Gallai theorem assumes the degree sequence is ordered:

$$d_1 \ge d_2 \ge \cdots \ge d_n$$

Therefore we first sort the array in decreasing order.

### Step 2: Verify the degree sum is even

Compute:

$$S = \sum d_i$$

Every edge contributes exactly two to the total degree count.

If `S` is odd, the sequence is immediately impossible.

### Step 3: Build prefix sums

Create:

$$prefix[i] = d_1+d_2+\cdots+d_i$$

This allows us to obtain any range sum in constant time.

### Step 4: Check every Erdős-Gallai inequality

For each `k` from `1` to `n`, compute:

$$L=\sum_{i=1}^{k} d_i$$

using the prefix sums.

We then need:

$$R= k(k-1) + \sum_{i=k+1}^{n}\min(d_i,k)$$

### Step 5: Use binary search

Since the degrees are sorted, among indices greater than `k` there is a boundary where values change from at least `k` to less than `k`.

Using binary search, find the first index `pos` such that:

$$d_{pos}<k$$

within the suffix.

Then:

- Degrees from `k+1` through `pos-1` contribute exactly `k`.
- Remaining degrees contribute their actual values.

This lets us compute the entire right side in constant time after the binary search.

### Step 6: Validate the inequality

If

$$L > R$$

for any `k`, the sequence cannot be graphical.

Return `false`.

### Step 7: Return success

If every inequality holds, the sequence is graphical.

Return `true`.

### Why it works

The Erdős-Gallai theorem is both necessary and sufficient.

The even-sum condition ensures the degree sequence could correspond to an undirected graph. The Erdős-Gallai inequalities ensure that no subset of vertices demands more edges than can possibly exist inside that subset and between that subset and the remaining vertices.

Because the theorem completely characterizes graphical sequences, passing all inequalities guarantees that a simple graph exists, while failing any inequality proves that no such graph can exist.

## Python Solution

```python
from typing import List
from bisect import bisect_left

class Solution:
    def simpleGraphExists(self, degrees: List[int]) -> bool:
        n = len(degrees)

        deg = sorted(degrees, reverse=True)

        total = sum(deg)
        if total % 2:
            return False

        prefix = [0] * (n + 1)
        for i in range(n):
            prefix[i + 1] = prefix[i] + deg[i]

        neg_deg = [-x for x in deg]

        for k in range(1, n + 1):
            left = prefix[k]

            pos = bisect_left(neg_deg, -k, lo=k)

            count_ge_k = pos - k

            right = (
                k * (k - 1)
                + count_ge_k * k
                + (prefix[n] - prefix[pos])
            )

            if left > right:
                return False

        return True
```

### Implementation Explanation

The degrees are first sorted in descending order because the Erdős-Gallai theorem assumes this ordering.

The parity condition is checked immediately. If the sum is odd, no graph can exist.

A prefix-sum array is then constructed so that sums of degree ranges can be obtained in constant time.

For each value of `k`, the left side of the inequality is simply `prefix[k]`.

The right side requires evaluating:

$$\sum_{i=k+1}^{n}\min(d_i,k)$$

Rather than scanning the suffix, binary search finds the boundary between degrees that are at least `k` and those that are smaller than `k`.

The contribution of the large degrees is computed as `count_ge_k * k`, while the contribution of the smaller degrees comes directly from the prefix sums.

If any inequality fails, the function immediately returns `False`. Otherwise all inequalities pass and the function returns `True`.

## Go Solution

```go
import "sort"

func simpleGraphExists(degrees []int) bool {
	n := len(degrees)

	deg := make([]int, n)
	copy(deg, degrees)

	sort.Slice(deg, func(i, j int) bool {
		return deg[i] > deg[j]
	})

	total := 0
	for _, d := range deg {
		total += d
	}

	if total%2 != 0 {
		return false
	}

	prefix := make([]int64, n+1)
	for i := 0; i < n; i++ {
		prefix[i+1] = prefix[i] + int64(deg[i])
	}

	for k := 1; k <= n; k++ {
		left := prefix[k]

		pos := sort.Search(n-k, func(idx int) bool {
			return deg[k+idx] < k
		}) + k

		countGeK := pos - k

		right := int64(k*(k-1))
		right += int64(countGeK * k)
		right += prefix[n] - prefix[pos]

		if left > right {
			return false
		}
	}

	return true
}
```

### Go-Specific Notes

The Go implementation follows the same logic as the Python version.

The main difference is the use of `sort.Search` for binary search. Since degree sums can grow to roughly $10^{10}$, prefix sums are stored in `int64` rather than `int` to avoid overflow on platforms where `int` is only 32 bits.

The input slice is copied before sorting so that the original slice is not modified.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | Sorting dominates, each Erdős-Gallai check uses binary search |
| Space | O(n) | Prefix sum array and sorted copy |

The sorting step requires `O(n log n)` time. The algorithm then performs `n` iterations, each containing a binary search, resulting in another `O(n log n)` term. Therefore the overall complexity remains `O(n log n)`.

The prefix-sum array and sorted degree array require linear extra memory, giving `O(n)` space complexity.

## Edge Cases

### Single Vertex

When `n = 1`, the only valid degree sequence is `[0]`. A degree of `1` would require a self-loop, which is forbidden in a simple graph. The implementation correctly rejects `[1]` because the Erdős-Gallai inequality fails.

### Odd Degree Sum

A common mistake is forgetting the handshaking lemma. Every edge contributes two to the total degree count, so the sum of all degrees must be even. Sequences such as `[2,2,1]` are impossible regardless of any other considerations. The implementation performs this check immediately.

### Vertices Demanding Too Many Connections

Sequences such as `[4,4,1,1,0]` can look plausible because every degree is within the range `[0,n-1]`. However, the two vertices of degree `4` must connect to every other vertex, forcing the low-degree vertices to have larger degrees than specified. The Erdős-Gallai inequalities detect exactly this contradiction.

### All Zero Degrees

A sequence like `[0,0,0,0]` represents a graph with no edges. Such cases are valid and can sometimes expose bugs in implementations that assume every graph contains edges. The algorithm handles this naturally because all inequalities hold with equality.

### Large Inputs

With `n = 100,000`, a naive implementation of the Erdős-Gallai theorem would be quadratic and far too slow. The binary-search optimization ensures each inequality is checked in `O(log n)` time, keeping the overall complexity at `O(n log n)` and making the solution practical for the largest allowed inputs.
