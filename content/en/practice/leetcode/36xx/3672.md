# LeetCode 3672 - Sum of Weighted Modes in Subarrays

## Problem Understanding

The problem gives an integer array `nums` and a window size `k`. We must examine every contiguous subarray whose length is exactly `k`.

For each such subarray, we determine its mode. The mode is the value that appears most frequently inside that window. If multiple values have the same highest frequency, we choose the smallest value among them.

Once the mode is determined, its contribution is not just the mode itself. Instead, the weight of the window is defined as:

- `weight = mode × frequency(mode)`

The final answer is the sum of these weights over all windows of length `k`.

For example, if a window contains:

- `1 → 2 times`
- `3 → 2 times`
- `5 → 1 time`

then both `1` and `3` have the highest frequency of `2`. Since ties are broken by choosing the smaller value, the mode is `1`, and the weight becomes:

- `1 × 2 = 2`

The constraints are important:

- `1 ≤ nums.length ≤ 10^5`
- `1 ≤ nums[i] ≤ 10^5`

There are up to one hundred thousand windows, so recomputing frequencies from scratch for every window would be too slow. An efficient sliding window approach is required.

Some important edge cases include:

- `k = 1`, where every element is its own mode.
- All elements identical, where every window has the same mode and maximum frequency.
- Many ties in frequency, requiring correct tie-breaking toward the smallest value.
- `k = n`, where there is only one window.
- Windows where removing one element and adding another changes the mode.

## Approaches

### Brute Force

The straightforward method processes each window independently.

For every subarray of length `k`, we count the frequency of all elements inside that window, determine the maximum frequency, find the smallest element having that frequency, compute its weight, and add it to the answer.

This approach is correct because every window is evaluated independently according to the problem definition.

However, there are `n-k+1` windows, and each window requires `O(k)` work to build frequencies plus another pass to determine the mode. In the worst case this becomes `O(nk)`, which can reach `10^10` operations and is far too slow.

### Key Insight

Adjacent windows differ by only two elements:

- one element leaves the window,
- one element enters the window.

Therefore, frequencies can be maintained incrementally with a sliding window.

The remaining challenge is determining the mode efficiently after every update.

For each value, we know its current frequency. We maintain an ordered structure containing pairs:

- `(-frequency, value)`

Sorting by negative frequency means higher frequencies come first. Sorting by value means ties automatically favor smaller numbers.

Thus the smallest pair in this ordering always corresponds to the current mode.

Insertion and deletion take logarithmic time, giving an overall complexity of `O(n log n)`.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(nk) | O(k) | Recompute frequencies for every window |
| Optimal | O(n log n) | O(k) | Sliding window with ordered set |

## Algorithm Walkthrough

1. Create a frequency map that stores the count of each value currently inside the window.
2. Maintain an ordered set containing pairs `(-frequency, value)` for every value whose frequency is positive. Since higher frequencies have smaller negative values, the first element of the set represents the mode.
3. Process the first window of size `k`. Update frequencies and keep the ordered structure synchronized.
4. Read the smallest pair from the ordered structure. If the pair is `(-f, x)`, then:

- frequency of the mode is `f`
- mode value is `x`
- weight equals `x × f`
5. Add this weight to the answer.
6. Slide the window one position to the right.
7. Before changing the frequency of the outgoing element, remove its current pair from the ordered structure.
8. Decrease its frequency. If the new frequency remains positive, insert its updated pair.
9. For the incoming element, remove its old pair if it already exists, increase its frequency, and insert the new pair.
10. The smallest pair in the ordered structure again gives the mode for the new window.
11. Repeat until all windows have been processed.

### Why it works

The frequency map always reflects exactly the elements inside the current window. Every element with positive frequency contributes one pair `(-frequency, value)` to the ordered structure. Because ordering prioritizes larger frequencies and then smaller values, the first pair always corresponds to the mode defined by the problem. Thus every window contributes the correct weight.

## Python Solution

```python
from typing import List
from collections import defaultdict
from sortedcontainers import SortedList

class Solution:
    def modeWeight(self, nums: List[int], k: int) -> int:
        freq = defaultdict(int)
        ordered = SortedList()

        def add(x: int) -> None:
            if freq[x] > 0:
                ordered.remove((-freq[x], x))
            freq[x] += 1
            ordered.add((-freq[x], x))

        def remove(x: int) -> None:
            ordered.remove((-freq[x], x))
            freq[x] -= 1
            if freq[x] > 0:
                ordered.add((-freq[x], x))

        for i in range(k):
            add(nums[i])

        neg_freq, mode_val = ordered[0]
        answer = mode_val * (-neg_freq)

        for right in range(k, len(nums)):
            left = right - k

            remove(nums[left])
            add(nums[right])

            neg_freq, mode_val = ordered[0]
            answer += mode_val * (-neg_freq)

        return answer
```

The implementation uses a hash map to maintain frequencies and a `SortedList` to maintain ordering by frequency and value.

Whenever an element changes frequency, its old pair is removed from the ordered structure and the updated pair is inserted. Therefore, the ordered structure always contains one entry for each distinct value currently present in the window.

The first entry of the `SortedList` corresponds to the highest frequency and smallest value among ties, allowing us to compute the weight in constant time.

## Go Solution

```go
import (
	"github.com/emirpasic/gods/trees/redblacktree"
)

func modeWeight(nums []int, k int) int64 {
	type Pair struct {
		freq int
		val  int
	}

	tree := redblacktree.NewWith(func(a, b interface{}) int {
		x := a.(Pair)
		y := b.(Pair)

		if x.freq != y.freq {
			if x.freq > y.freq {
				return -1
			}
			return 1
		}

		if x.val < y.val {
			return -1
		}
		if x.val > y.val {
			return 1
		}
		return 0
	})

	freq := map[int]int{}

	add := func(x int) {
		if freq[x] > 0 {
			tree.Remove(Pair{freq[x], x})
		}
		freq[x]++
		tree.Put(Pair{freq[x], x}, true)
	}

	remove := func(x int) {
		tree.Remove(Pair{freq[x], x})
		freq[x]--
		if freq[x] > 0 {
			tree.Put(Pair{freq[x], x}, true)
		}
	}

	for i := 0; i < k; i++ {
		add(nums[i])
	}

	first := tree.Left().Key.(Pair)
	ans := int64(first.freq * first.val)

	for right := k; right < len(nums); right++ {
		left := right - k

		remove(nums[left])
		add(nums[right])

		first = tree.Left().Key.(Pair)
		ans += int64(first.freq * first.val)
	}

	return ans
}
```

The Go implementation follows the same logic as the Python version. Since Go does not provide a built-in ordered set, a balanced binary search tree is used. The comparator first orders by decreasing frequency and then by increasing value, ensuring that the leftmost node always represents the mode.

The answer is stored as `int64` to avoid overflow because the sum of all weights can exceed the range of a 32-bit integer.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log k) | Every insertion and deletion in the ordered structure costs logarithmic time |
| Space | O(k) | At most k distinct values exist in a window |

Since every element enters and leaves the sliding window exactly once, there are `O(n)` updates. Each update performs a constant number of balanced-tree operations, leading to `O(n log k)` overall time complexity.

## Edge Cases

### Window Size Equals One

When `k = 1`, every window contains only one element. The frequency of that element is always one, so the weight equals the element itself. A common mistake is overcomplicating tie handling, but the implementation naturally handles this case because the ordered structure contains exactly one pair.

### All Values Have Equal Frequency

Windows such as `[1,2,3,4]` produce frequencies of one for every value. The problem requires choosing the smallest value among ties. By storing pairs ordered as `(-frequency, value)`, ties automatically resolve correctly without additional logic.

### Mode Changes After Sliding

When an element leaves the window, its frequency decreases and another value may become the new mode. Incorrect update order can leave stale entries in the ordered structure. The implementation always removes the old pair before modifying the frequency and inserts the new pair afterward, ensuring the ordered structure remains consistent.

### All Elements Are Identical

For arrays such as `[7,7,7,7]`, the frequency of the single value grows to `k` and remains high throughout all windows. Since only one pair exists in the ordered structure, mode queries remain correct and efficient.
