## Solution

## Problem Understanding

This problem gives us a list of threats, where each threat is represented as:

- `IDi`, a unique identifier
- `sevi`, the severity value
- `expi`, the exploitability value

For every threat, we compute a derived metric called the **score**:

$$score = 2 \times sev_i + exp_i$$

The goal is to reorder the entire `threats` array according to two sorting rules:

1. Higher score should come first, meaning descending order of `score`.
2. If two threats produce the same score, the one with the smaller `ID` should appear first.

The output should contain the original threat entries, only rearranged into the required order.

The constraints are important:

- Up to `10^5` threats
- Severity and exploitability values can be as large as `10^9`
- IDs are guaranteed unique

Because the input size reaches one hundred thousand rows, any algorithm worse than `O(n log n)` becomes impractical. At the same time, the score computation itself is constant time, so the dominant cost should come from ordering the rows.

A few edge cases stand out immediately.

If there is only one threat, sorting should leave the array unchanged.

Multiple threats may produce identical scores, so tie breaking by ascending ID must be implemented correctly.

Severity and exploitability can be large, meaning score values can exceed normal small integer assumptions, although both Python integers and Go `int` on modern platforms comfortably handle the maximum possible value of `3 × 10^9`.

## Approaches

### Brute Force

A straightforward approach would be to repeatedly select the highest priority threat and build the answer incrementally.

For each position in the result:

- Scan all remaining threats.
- Compute scores.
- Find the largest score.
- If scores tie, choose the smaller ID.
- Remove that threat and continue.

This approach is correct because each step places the globally best remaining threat into the next position.

However, every selection requires scanning the remaining elements, leading to approximately:

$$n + (n-1) + (n-2) + \dots + 1 = O(n^2)$$

With `n = 100000`, this becomes far too slow.

### Optimal Approach, Custom Sorting

The key observation is that the problem is fundamentally a sorting problem.

Each threat has a deterministic ordering key:

$$(-score,\ ID)$$

Using `-score` converts descending score order into ascending comparator logic.

Since modern sorting algorithms run in `O(n log n)`, we can:

1. Compute the score when comparing elements.
2. Sort using:

- descending score
- ascending ID for ties

No extra data structures are required.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeatedly select highest priority threat |
| Optimal | O(n log n) | O(log n) to O(n) depending on sort implementation | Single custom sort using score and ID |

## Algorithm Walkthrough

1. Iterate through the `threats` array conceptually during sorting rather than precomputing scores.
2. For each comparison between two threats:

- Extract `id`, `severity`, and `exploitability`.
- Compute:

$$score = 2 \times severity + exploitability$$
3. Compare scores.

- If scores differ, place the larger score first.
- This enforces descending score order.
4. If scores are equal, compare IDs.

- Place the smaller ID first.
- This enforces ascending ID order.
5. Return the sorted array.

### Why it works

The sorting comparator defines a complete ordering relation for every pair of threats. Every comparison prioritizes larger scores and resolves equal scores with smaller IDs. Since sorting guarantees that all elements are placed consistently according to that ordering relation, the final array satisfies the exact specification.

## Python Solution

```python
from typing import List

class Solution:
    def sortThreats(self, threats: List[List[int]]) -> List[List[int]]:
        threats.sort(
            key=lambda threat: (
                -(2 * threat[1] + threat[2]),
                threat[0]
            )
        )
        return threats
```

The implementation uses Python's built in `sort()` method, which performs an efficient `O(n log n)` stable sort.

The sorting key is a tuple.

The first component is `-(2 * severity + exploitability)`, which ensures higher scores appear earlier.

The second component is `ID`, which naturally produces ascending ID ordering when scores match.

Because sorting happens in place, no additional result array is needed.

## Go Solution

```go
import "sort"

func sortThreats(threats [][]int) [][]int {
	sort.Slice(threats, func(i, j int) bool {
		scoreI := 2*threats[i][1] + threats[i][2]
		scoreJ := 2*threats[j][1] + threats[j][2]

		if scoreI != scoreJ {
			return scoreI > scoreJ
		}

		return threats[i][0] < threats[j][0]
	})

	return threats
}
```

The Go implementation uses `sort.Slice`, which accepts a comparator function.

Unlike Python, Go does not expose tuple sorting directly, so the comparison logic is written explicitly.

The score values remain safely within integer limits because the maximum score is `3 × 10^9`.

## Edge Cases

### Single Threat

When the input contains exactly one threat, sorting should return the same array. Implementations that unnecessarily rebuild structures or apply incorrect indexing can fail on this smallest valid input.

The presented solution naturally handles this because sorting one element performs no changes.

### Multiple Equal Scores

Several threats may generate identical scores. A common mistake is sorting only by score and relying on unstable ordering.

This implementation explicitly includes `ID` as the secondary sorting key, guaranteeing deterministic output.

### Very Large Severity and Exploitability Values

Severity and exploitability can each reach `10^9`, producing scores near `3 × 10^9`.

Implementations that assume small numeric ranges could overflow in some languages.

Python integers grow automatically, and Go integer arithmetic remains safe for these constraints on standard LeetCode environments.

### Already Sorted Input

If threats already satisfy the required ordering, the algorithm should preserve correctness without extra handling.

Sorting algorithms still perform comparisons but keep the output unchanged, maintaining `O(n log n)` worst case performance.
