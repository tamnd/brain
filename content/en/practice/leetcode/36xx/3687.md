## Solution

## Problem Understanding

This problem asks us to calculate the total library late fee across multiple returned books.

We are given an integer array `daysLate`, where each element represents how many days late a particular book was returned. For every book, we compute an individual penalty according to a fixed set of rules, then sum all penalties together and return the final total.

The penalty rules are piecewise:

- If a book is exactly `1` day late, its penalty is `1`.
- If a book is between `2` and `5` days late inclusive, its penalty is `2 × daysLate[i]`.
- If a book is more than `5` days late, its penalty is `3 × daysLate[i]`.

The input is simply a list of integers. The output is a single integer representing the accumulated fee for all books.

The constraints are small:

- The array length is at most `100`
- Each late day value is at most `100`

These limits immediately suggest that efficiency is not a concern. Even a straightforward solution that examines every element individually will run comfortably within limits.

Several edge cases are worth identifying upfront.

A list containing only `1`s ensures the special fixed penalty path is applied correctly instead of multiplying. Values exactly at the category boundaries, such as `2`, `5`, and `6`, verify that conditions are implemented with correct inclusive and exclusive comparisons. A single-element array checks that accumulation logic does not assume multiple books.

## Approaches

### Brute Force

The brute-force approach is to evaluate every book independently.

For each value in `daysLate`, determine which fee category applies, calculate the penalty, and add it into a running total.

Although this is called brute force, there is no unnecessary work. Every input element must be inspected at least once because every book contributes independently to the result.

This approach is correct because the total penalty is simply the sum of penalties for all books, and each book's penalty depends only on its own late days.

### Key Insight for the Optimal Solution

The important observation is that there is no interaction between books.

Unlike problems that require sorting, dynamic programming, or additional data structures, each element can be processed independently and immediately converted into its contribution to the final answer.

That means a single linear scan is sufficient.

As we iterate through `daysLate`, we apply the matching pricing rule and update the total. No extra storage is required.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Evaluate each book individually and accumulate penalties |
| Optimal | O(n) | O(1) | Single pass with direct fee calculation |

In this problem, the brute-force and optimal approaches are effectively identical because every element must be examined once.

## Algorithm Walkthrough

1. Initialize a variable `total_penalty` to `0`.

This variable stores the running sum of penalties as books are processed.
2. Iterate through every value `days` in `daysLate`.

Each iteration handles exactly one book.
3. Check whether `days` equals `1`.

If true, add `1` to `total_penalty`.
4. Otherwise, check whether `days` is between `2` and `5` inclusive.

If true, compute `2 * days` and add that value.
5. Otherwise, the value must be greater than `5`.

Compute `3 * days` and add it to the running total.
6. After processing all books, return `total_penalty`.

### Why it works

The algorithm maintains a simple invariant: after processing the first `k` books, `total_penalty` equals the correct total fee for exactly those `k` books.

Each iteration computes the exact fee for one additional book according to the problem rules and adds it to the existing correct subtotal. After the final iteration, all books have been included, so the result is the complete total fee.

## Python Solution

```python
from typing import List

class Solution:
    def lateFee(self, daysLate: List[int]) -> int:
        total_penalty = 0

        for days in daysLate:
            if days == 1:
                total_penalty += 1
            elif days <= 5:
                total_penalty += 2 * days
            else:
                total_penalty += 3 * days

        return total_penalty
```

The implementation starts by creating `total_penalty`, which tracks the accumulated fee.

The loop processes one book at a time. The first condition handles the special rule for exactly one day late. The second condition captures values from `2` through `5`. Since the first branch already excluded `1`, checking `days <= 5` is sufficient. Any remaining values automatically belong to the final category and receive triple the number of late days.

At the end, the accumulated total is returned.

## Go Solution

```go
func lateFee(daysLate []int) int {
	totalPenalty := 0

	for _, days := range daysLate {
		if days == 1 {
			totalPenalty += 1
		} else if days <= 5 {
			totalPenalty += 2 * days
		} else {
			totalPenalty += 3 * days
		}
	}

	return totalPenalty
}
```

The Go implementation follows the same logic as Python.

The `range` loop iterates over the slice values directly. Go slices can safely be empty, although this problem guarantees at least one element. Integer overflow is not a concern because the maximum possible total is `100 × (3 × 100) = 30000`, well within Go's integer limits.

## Edge Cases

One important edge case is when every book is exactly one day late. This can expose implementations that incorrectly apply multiplication rules universally. The solution explicitly checks `days == 1` first, ensuring the special penalty of `1` is used.

Another important case is values near category boundaries, especially `5` and `6`. An off-by-one mistake could incorrectly charge `5` days at triple rate or `6` days at double rate. The implementation uses `elif days <= 5`, making the transition precise.

A final edge case is the largest valid input, where there are `100` books and each is `100` days late. Even though the total remains small, testing this confirms the algorithm scales linearly and correctly accumulates large totals without overflow concerns.
