## Solution

## Problem Understanding

We are given two integers, `n` and `m`.

The first task is to generate the first `m` prime numbers. From those primes, we may choose any prime any number of times because the problem explicitly allows repeated usage. The chosen primes form a multiset, meaning duplicates are allowed.

Our goal is to make the sum of the selected primes exactly equal to `n`, while minimizing how many primes are used. If no combination of the first `m` primes can produce exactly `n`, we return `-1`.

This is very similar to a coin change problem. The first `m` primes act as coin denominations, each denomination can be used an unlimited number of times, and we want the minimum number of coins needed to reach a target sum `n`.

The constraints are relatively small:

- `1 <= n <= 1000`
- `1 <= m <= 1000`

Although `m` can be as large as 1000, we only care about primes that are at most `n`. Any prime larger than `n` can never participate in a sum equal to `n`. Since `n` is only 1000, the effective number of useful primes is quite small.

Several edge cases are important:

- `n = 1` is impossible because the smallest prime is 2.
- The first `m` primes may all be larger than the remaining amount during DP transitions.
- Some targets cannot be formed from the available primes. For example, if `m = 1`, only prime 2 is available, so odd targets are impossible.
- A target may itself be one of the first `m` primes, in which case the answer is `1`.
- Large values of `m` do not significantly increase difficulty because primes larger than `n` are irrelevant.

## Approaches

### Brute Force

A straightforward approach is to recursively try every possible prime at every step.

For a remaining target `remaining`, we could attempt to subtract every available prime and recursively solve the smaller subproblem. We would then take the minimum number of primes among all valid choices.

This approach is correct because it explores every possible combination of allowed primes. However, it repeatedly recomputes the same subproblems. For example, many different paths may eventually ask for the minimum number of primes needed to form the same remaining sum.

Without memoization, the number of states grows exponentially, making the solution impractical.

### Key Insight

The problem is an unbounded coin change problem.

Each prime acts as a coin denomination:

- Prime values are the coin denominations.
- Each denomination can be used unlimited times.
- We need the minimum number of coins to reach target `n`.

The classic dynamic programming solution maintains:

`dp[x] = minimum number of primes needed to form sum x`

Initially:

- `dp[0] = 0`
- All other states are set to infinity

For every available prime, we update all reachable sums. Since each prime may be reused indefinitely, we iterate forward through the target sums, exactly as in the unbounded knapsack / coin change minimum-coins formulation.

Before running DP, we must generate the first `m` prime numbers.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | O(n) recursion depth | Tries all combinations recursively |
| Optimal | O(P × n) | O(n) | Unbounded coin change DP, where P is the number of usable primes |

## Algorithm Walkthrough

### Step 1: Generate the first `m` prime numbers

We need the first `m` primes because only those primes are allowed to participate in the sum.

A simple primality test is sufficient because the constraints are small. We keep testing integers starting from 2 until we have collected `m` primes.

### Step 2: Discard primes larger than `n`

Any prime greater than `n` cannot contribute to a sum equal to `n`.

Removing such primes slightly reduces the DP work.

### Step 3: Initialize the DP array

Create an array:

```
dp[0...n]
```

where:

```
dp[x] = minimum primes needed to form x
```

Initialize:

```
dp[0] = 0
dp[1...n] = infinity
```

The value infinity represents an unreachable sum.

### Step 4: Process each prime

For every available prime `p`, update all sums from `p` to `n`.

For each sum:

```
dp[sum] = min(
    dp[sum],
    dp[sum - p] + 1
)
```

The logic is:

- If `sum - p` is reachable,
- then adding one more prime `p`
- creates a candidate solution for `sum`.

### Step 5: Read the answer

After all transitions are complete:

- If `dp[n]` is still infinity, return `-1`.
- Otherwise return `dp[n]`.

### Why it works

The DP invariant is that after processing primes, `dp[x]` stores the minimum number of allowed primes required to form sum `x`.

Every valid representation of a sum must end with some prime `p`. Removing that final prime leaves a smaller sum `x - p`. Since the DP already stores the optimal solution for `x - p`, adding one prime produces a candidate solution for `x`. Taking the minimum over all possible final primes guarantees that `dp[x]` becomes the optimal answer. Therefore `dp[n]` is the minimum number of primes needed to form `n`.

## Python Solution

```python
class Solution:
    def minNumberOfPrimes(self, n: int, m: int) -> int:
        def is_prime(x: int) -> bool:
            if x < 2:
                return False
            d = 2
            while d * d <= x:
                if x % d == 0:
                    return False
                d += 1
            return True

        primes = []
        candidate = 2

        while len(primes) < m:
            if is_prime(candidate):
                primes.append(candidate)
            candidate += 1

        primes = [p for p in primes if p <= n]

        INF = float("inf")
        dp = [INF] * (n + 1)
        dp[0] = 0

        for prime in primes:
            for total in range(prime, n + 1):
                if dp[total - prime] != INF:
                    dp[total] = min(
                        dp[total],
                        dp[total - prime] + 1
                    )

        return -1 if dp[n] == INF else dp[n]
```

The implementation begins by generating the first `m` prime numbers. A helper function performs trial division up to the square root of the candidate number.

After collecting the primes, we remove those larger than `n` because they can never contribute to the target sum.

The DP array is then created. `dp[0] = 0` represents the empty selection, while all other states start as unreachable.

For each prime, we iterate forward from that prime value up to `n`. Forward iteration is important because it allows the same prime to be reused multiple times, matching the unbounded nature of the problem.

Finally, if `dp[n]` remains infinite, the target is unreachable and we return `-1`. Otherwise, `dp[n]` contains the minimum number of primes required.

## Go Solution

```go
func minNumberOfPrimes(n int, m int) int {
	isPrime := func(x int) bool {
		if x < 2 {
			return false
		}
		for d := 2; d*d <= x; d++ {
			if x%d == 0 {
				return false
			}
		}
		return true
	}

	primes := make([]int, 0)

	for candidate := 2; len(primes) < m; candidate++ {
		if isPrime(candidate) {
			primes = append(primes, candidate)
		}
	}

	filtered := make([]int, 0)
	for _, p := range primes {
		if p <= n {
			filtered = append(filtered, p)
		}
	}

	const INF = int(1e9)

	dp := make([]int, n+1)
	for i := range dp {
		dp[i] = INF
	}
	dp[0] = 0

	for _, prime := range filtered {
		for total := prime; total <= n; total++ {
			if dp[total-prime] != INF {
				candidate := dp[total-prime] + 1
				if candidate < dp[total] {
					dp[total] = candidate
				}
			}
		}
	}

	if dp[n] == INF {
		return -1
	}

	return dp[n]
}
```

The Go implementation follows the same algorithm. An integer sentinel value (`1e9`) is used instead of floating-point infinity. Slices are used for both the prime list and the DP table. Integer overflow is not a concern because the maximum answer can never exceed `n`, which is at most 1000.

## Edge Cases

### Target Equals One

The value `1` cannot be represented as a sum of prime numbers because the smallest prime is `2`. A careless implementation might accidentally treat the empty selection as valid or initialize DP incorrectly. In this solution, `dp[1]` remains unreachable and the algorithm correctly returns `-1`.

### Only Prime 2 Is Available

When `m = 1`, the only usable prime is `2`. Every reachable sum must therefore be even. Odd targets such as `5` are impossible. The DP naturally handles this because odd positions are never updated from infinity.

### Target Is Already an Allowed Prime

If `n` itself appears among the first `m` primes, the optimal answer is always `1`. The transition from `dp[0]` to `dp[n]` immediately sets `dp[n] = 1`, and no later solution can improve upon using a single prime.

### Very Large m

The value of `m` can be as large as 1000, producing many primes that exceed `n`. Such primes are useless because they cannot participate in a sum equal to `n`. The implementation filters them out before running DP, preventing unnecessary work while preserving correctness.
