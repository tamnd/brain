## Solution

## Problem Understanding

This problem asks us to determine how much rainwater can be trapped between vertical bars after rainfall. We are given an array `height`, where each element represents the height of a bar in an elevation map. Every bar has a width of `1`.

The key observation is that water can only be trapped when there are taller bars on both sides of a position. For any index `i`, the amount of water trapped above `height[i]` depends on the shortest boundary between the tallest bar to its left and the tallest bar to its right.

In other words, for each position:

$$\text{water at i} = \min(\text{max left}, \text{max right}) - \text{height[i]}$$

If this value is negative, no water can be trapped at that position.

For example, consider:

```
height = [0,1,0,2]
```

At index `2`, the height is `0`. The tallest bar to the left is `1`, and the tallest bar to the right is `2`. Since the limiting boundary is `1`, we can trap:

```
min(1,2) - 0 = 1
```

unit of water.

The constraints tell us that:

- `1 <= n <= 2 * 10^4`
- `0 <= height[i] <= 10^5`

The input size can be as large as 20,000 elements, which means inefficient quadratic approaches may become too slow. We should aim for an `O(n)` or `O(n log n)` solution.

Several edge cases are important:

- Arrays with fewer than three bars cannot trap water because at least two boundaries are needed.
- Strictly increasing or strictly decreasing arrays trap no water.
- Flat surfaces such as `[3,3,3]` trap no water.
- Deep valleys like `[5,0,5]` can trap large amounts of water.
- Multiple peaks and valleys require careful handling because local decisions may be misleading.

## Approaches

### Brute Force Approach

A straightforward solution is to calculate the trapped water at every index independently.

For each position `i`, we scan all elements to the left to find the maximum left boundary, then scan all elements to the right to find the maximum right boundary. The water trapped at index `i` is determined by the smaller of these two maximum heights.

This works because water level at any position is fundamentally constrained by the shortest wall surrounding it.

However, the issue is efficiency. For every index, we perform two additional scans over the array. Since we repeat this for all `n` positions, the total time complexity becomes `O(n²)`.

With `n = 20,000`, quadratic performance is unnecessarily expensive.

### Optimal Approach, Two Pointers

The key insight is that we do not need to repeatedly recompute left and right maximum heights.

Instead, we can maintain:

- A left pointer starting at the beginning
- A right pointer starting at the end
- The maximum height seen so far from the left
- The maximum height seen so far from the right

The reasoning is subtle but powerful.

At any point, if `leftMax < rightMax`, then the amount of water trapped at the current left position is fully determined by `leftMax`.

Why?

Because even if there are taller bars further right, we already know the right side has at least `rightMax`, and `rightMax` is greater than `leftMax`. The limiting factor is therefore `leftMax`.

Similarly, when `rightMax <= leftMax`, the right side becomes the limiting factor, and we can safely compute trapped water for the right pointer.

This allows us to process the array in a single pass without additional storage.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | For every index, scan left and right to find boundaries |
| Optimal | O(n) | O(1) | Two pointers with running maximum heights |

## Algorithm Walkthrough

1. Initialize two pointers, `left` at the beginning of the array and `right` at the end.
2. Maintain two variables, `left_max` and `right_max`, to track the tallest wall encountered from each side.
3. Maintain a variable `trapped_water` to accumulate the total trapped water.
4. While `left < right`, compare `height[left]` and `height[right]`.
5. If `height[left] < height[right]`, process the left side:

- If `height[left]` is greater than or equal to `left_max`, update `left_max`.
- Otherwise, water can be trapped:

```
left_max - height[left]
```
- Move the left pointer one step right.
6. Otherwise, process the right side:

- If `height[right]` is greater than or equal to `right_max`, update `right_max`.
- Otherwise, water can be trapped:

```
right_max - height[right]
```
- Move the right pointer one step left.
7. Continue until both pointers meet.
8. Return `trapped_water`.

The reason this works is that we always process the side whose maximum boundary is guaranteed to be the limiting factor.

## Python Solution

```
from typing import List

class Solution:
    def trap(self, height: List[int]) -> int:
        if len(height) < 3:
            return 0

        left = 0
        right = len(height) - 1

        left_max = 0
        right_max = 0
        trapped_water = 0

        while left < right:
            if height[left] < height[right]:
                if height[left] >= left_max:
                    left_max = height[left]
                else:
                    trapped_water += left_max - height[left]

                left += 1
            else:
                if height[right] >= right_max:
                    right_max = height[right]
                else:
                    trapped_water += right_max - height[right]

                right -= 1

        return trapped_water
```

The implementation begins with a quick edge case check. If the array contains fewer than three bars, trapping water is impossible, so we immediately return `0`.

We initialize two pointers, `left` and `right`, at opposite ends of the array. These pointers gradually move toward one another.

The variables `left_max` and `right_max` store the tallest wall seen so far from each direction. These values are continuously updated as we scan inward.

Inside the loop, we decide which side to process based on comparing `height[left]` and `height[right]`. When the left bar is smaller, the left boundary is the limiting factor, so we calculate trapped water using `left_max`. Otherwise, we process the right side using `right_max`.

The accumulated water is stored in `trapped_water`, which is returned at the end.

## Go Solution

```
func trap(height []int) int {
    if len(height) < 3 {
        return 0
    }

    left := 0
    right := len(height) - 1

    leftMax := 0
    rightMax := 0
    trappedWater := 0

    for left < right {
        if height[left] < height[right] {
            if height[left] >= leftMax {
                leftMax = height[left]
            } else {
                trappedWater += leftMax - height[left]
            }

            left++
        } else {
            if height[right] >= rightMax {
                rightMax = height[right]
            } else {
                trappedWater += rightMax - height[right]
            }

            right--
        }
    }

    return trappedWater
}
```

The Go implementation follows the same logic as the Python version. Since Go slices can safely have length `0`, we simply check whether `len(height) < 3`.

All arithmetic uses `int`, which is sufficient given the constraints. Even in the worst case, the maximum trapped water fits comfortably within integer limits.

The primary syntactic difference is variable naming style, using camelCase such as `leftMax` and `trappedWater`, which is idiomatic in Go.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each pointer moves across the array at most once |
| Space | O(1) | Only a fixed number of variables are used |

The time complexity is `O(n)` because each element is processed at most once by either the left pointer or the right pointer. No nested iteration occurs.

The space complexity is `O(1)` because we only store a few integer variables regardless of input size. No auxiliary arrays or data structures proportional to `n` are used.

## Edge Cases

### Arrays With Fewer Than Three Elements

An array such as `[1]` or `[2,1]` cannot trap water because there are not enough bars to form a container. A naive implementation might still attempt calculations and produce incorrect values. Our implementation explicitly checks for this and immediately returns `0`.

### Strictly Increasing or Decreasing Heights

Inputs like `[1,2,3,4]` or `[4,3,2,1]` trap no water because every position lacks a sufficient boundary on one side. The two pointer method naturally handles this by updating maximum heights without ever accumulating trapped water.

### Flat Terrain

An input like `[3,3,3,3]` may cause confusion because there are equal height walls everywhere, but no valleys exist to hold water. Since every height equals the current maximum boundary, no positive difference is added.

### Deep Valleys Between Tall Walls

Cases such as `[5,0,0,0,5]` are useful for testing correctness because large volumes of water are trapped. Our algorithm correctly accumulates water at every interior position:

```
5 + 5 + 5 = 15
```

because the limiting boundary on both sides is height `5`.

### Multiple Peaks and Valleys

Complex inputs like `[4,2,0,3,2,5]` contain several local minima and maxima. A greedy local approach could fail here. The two pointer technique works because it relies on guaranteed boundary information rather than local patterns, ensuring correctness across all configurations.
