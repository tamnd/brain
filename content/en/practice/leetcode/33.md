# LeetCode 33, Search in Rotated Sorted Array

## Problem Understanding

The problem gives us an array of integers that was originally sorted in ascending order, but may have been rotated at some pivot point. A rotation means that some prefix of the array was moved to the end while preserving the relative ordering of the elements.

For example, the sorted array:

```
[0,1,2,4,5,6,7]
```

could become:

```
[4,5,6,7,0,1,2]
```

after rotating at index `3`.

The important detail is that all numbers are distinct. This guarantee simplifies the logic because we never have to deal with duplicate values when determining which side of the array is sorted.

The task is to find the index of a target value in this rotated array. If the target exists, we return its index. Otherwise, we return `-1`.

The problem explicitly requires an algorithm with `O(log n)` runtime complexity. This immediately rules out linear scanning as the intended final solution. Since the original array was sorted before rotation, binary search is the natural direction to explore.

The constraints are relatively small, with `nums.length <= 5000`, but the logarithmic runtime requirement is the real challenge. The problem is designed to test whether we can adapt binary search to work even when the array is no longer globally sorted.

Several edge cases are important to think about upfront:

- The array may not actually be rotated at all, for example `[1,2,3,4,5]`.
- The array may contain only one element.
- The target may not exist in the array.
- The rotation pivot may split the array near the beginning or near the end.
- The target may lie on either side of the pivot.

A naive binary search fails because the array is not completely sorted anymore. However, one half of the array is always sorted, and that observation is the key to the optimal solution.

## Approaches

### Brute Force Approach

The simplest solution is to iterate through the array from left to right and compare every element with the target.

If we find the target, we return its index immediately. If we finish the traversal without finding it, we return `-1`.

This approach is guaranteed to work because every element is checked exactly once.

However, the runtime complexity is `O(n)` because in the worst case we may need to inspect every element. This violates the problem requirement of `O(log n)` runtime complexity.

### Optimal Binary Search Approach

The key observation is that although the entire array is not sorted, at least one half of the array is always sorted.

Suppose we pick a middle index during binary search.

There are only two possibilities:

1. The left half is sorted.
2. The right half is sorted.

Because all values are distinct, we can determine which half is sorted by comparing `nums[left]` and `nums[mid]`.

If:

```
nums[left] <= nums[mid]
```

then the left side is sorted.

Otherwise, the right side must be sorted.

Once we know which half is sorted, we can determine whether the target lies inside that sorted range. If it does, we continue searching there. Otherwise, we discard that half and search the other side.

This preserves the logarithmic behavior of binary search while handling the rotation correctly.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Scan every element sequentially |
| Optimal | O(log n) | O(1) | Modified binary search using sorted-half detection |

## Algorithm Walkthrough

1. Initialize two pointers, `left` and `right`, at the beginning and end of the array.
2. Continue searching while `left <= right`. This represents the current search space.
3. Compute the middle index using:

```
mid = (left + right) // 2
```

This splits the current search space into two halves.

1. If `nums[mid] == target`, return `mid` immediately because we found the target.
2. Determine which half is sorted.

If:

```
nums[left] <= nums[mid]
```

then the left half is sorted.

Otherwise, the right half is sorted.

1. If the left half is sorted, check whether the target lies inside this range.

Specifically, test:

```
nums[left] <= target < nums[mid]
```

If true, move `right = mid - 1` because the target must be in the left half.

Otherwise, move `left = mid + 1` because the target must be in the right half.

1. If the right half is sorted, check whether the target lies inside that range.

Specifically, test:

```
nums[mid] < target <= nums[right]
```

If true, move `left = mid + 1`.

Otherwise, move `right = mid - 1`.

1. Repeat until the search space becomes empty.
2. If the loop ends without finding the target, return `-1`.

This approach works because each iteration removes half of the remaining search space, just like standard binary search.

## Python Solution

```
from typing import List

class Solution:
    def search(self, nums: List[int], target: int) -> int:
        left = 0
        right = len(nums) - 1

        while left <= right:
            mid = (left + right) // 2

            if nums[mid] == target:
                return mid

            # Left half is sorted
            if nums[left] <= nums[mid]:
                if nums[left] <= target < nums[mid]:
                    right = mid - 1
                else:
                    left = mid + 1

            # Right half is sorted
            else:
                if nums[mid] < target <= nums[right]:
                    left = mid + 1
                else:
                    right = mid - 1

        return -1
```

The implementation begins by initializing two pointers, `left` and `right`, which define the current search range.

Inside the loop, we calculate the midpoint and immediately check whether it matches the target.

The next step is the critical insight of the algorithm. We determine which half is sorted by comparing the values at `left` and `mid`.

If the left half is sorted, we check whether the target falls within that sorted interval. If it does, we discard the right half. Otherwise, we discard the left half.

If the left half is not sorted, then the right half must be sorted. We perform the same range check on the right side and narrow the search accordingly.

Eventually, either the target is found or the search range becomes empty.

The algorithm uses only constant extra space because all operations are performed in place using pointers.

## Go Solution

```
func search(nums []int, target int) int {
    left := 0
    right := len(nums) - 1

    for left <= right {
        mid := (left + right) / 2

        if nums[mid] == target {
            return mid
        }

        // Left half is sorted
        if nums[left] <= nums[mid] {
            if nums[left] <= target && target < nums[mid] {
                right = mid - 1
            } else {
                left = mid + 1
            }
        } else {
            // Right half is sorted
            if nums[mid] < target && target <= nums[right] {
                left = mid + 1
            } else {
                right = mid - 1
            }
        }
    }

    return -1
}
```

The Go implementation follows exactly the same logic as the Python version.

Slices in Go behave similarly to dynamic arrays, so index access works naturally for binary search. Since the constraints are small, integer overflow when calculating `mid` is not a concern here, although in production systems some developers prefer:

```
mid := left + (right-left)/2
```

to avoid overflow in very large arrays.

Go does not have Python style type hints, but the function signature clearly specifies that the input is a slice of integers and the return value is an integer index.

## Worked Examples

### Example 1

```
Input: nums = [4,5,6,7,0,1,2], target = 0
```

Initial state:

| left | mid | right | nums[mid] |
| --- | --- | --- | --- |
| 0 | 3 | 6 | 7 |

The left half `[4,5,6,7]` is sorted because:

```
nums[left] <= nums[mid]
```

becomes:

```
4 <= 7
```

The target `0` is not inside `[4,7)`.

Move right:

```
left = mid + 1 = 4
```

New state:

| left | mid | right | nums[mid] |
| --- | --- | --- | --- |
| 4 | 5 | 6 | 1 |

The left half `[0,1]` is sorted.

The target `0` lies inside `[0,1)`.

Move left:

```
right = mid - 1 = 4
```

New state:

| left | mid | right | nums[mid] |
| --- | --- | --- | --- |
| 4 | 4 | 4 | 0 |

We found the target.

Return:

```
4
```

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(log n) | Each iteration removes half of the remaining search space |
| Space | O(1) | Only a few pointer variables are used |

The runtime is logarithmic because the algorithm behaves like binary search. At every step, one half of the array is discarded completely.

The space complexity is constant because no auxiliary data structures are created. The algorithm only stores a few integer variables.

## Edge Cases

### Array With One Element

A single-element array can easily expose off-by-one errors in binary search implementations.

For example:

```
nums = [1], target = 0
```

The implementation handles this correctly because the loop condition uses `left <= right`, allowing the single index to be checked properly before terminating.

### Array Not Rotated

The array may already be fully sorted:

```
nums = [1,2,3,4,5]
```

In this case, the algorithm still works because the left half will always appear sorted, reducing the logic to ordinary binary search behavior.

### Target Does Not Exist

If the target is absent, the algorithm must terminate cleanly without infinite looping.

For example:

```
nums = [4,5,6,7,0,1,2], target = 8
```

Each iteration still removes half the search space, and eventually `left` becomes greater than `right`. At that point the algorithm correctly returns `-1`.

### Rotation Near the Ends

Rotations near the boundaries can be tricky:

```
[7,1,2,3,4,5,6]
```

or

```
[2,3,4,5,6,7,1]
```

The implementation handles these correctly because it does not rely on finding the pivot explicitly. Instead, it only checks which half is currently sorted at each iteration.

### Target at the Pivot

The target itself may be the smallest element where the rotation occurs.

Example:

```
nums = [4,5,6,7,0,1,2], target = 0
```

The algorithm correctly identifies that the target lies in the unsorted side after the first split and continues searching there until it is found.
