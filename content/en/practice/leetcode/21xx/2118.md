## Solution

## Problem Understanding

This problem asks us to construct a formatted polynomial equation string from the rows stored in the `Terms` table.

Each row represents one term in a polynomial expression. The `power` column tells us the exponent of `X`, and the `factor` column tells us the coefficient. Our goal is to transform all rows into a single equation string that follows very strict formatting rules.

The final equation must always be written in the form:

```
<terms>=0
```

The left-hand side contains every polynomial term sorted by descending power, while the right-hand side is always `0`.

Each term must follow these formatting rules:

- Positive coefficients start with `+`
- Negative coefficients start with `-`
- The coefficient itself is written using its absolute value
- Terms with `power > 1` use the format:

```
+3X^2
```
- Terms with `power == 1` omit the exponent:

```
-4X
```
- Terms with `power == 0` omit both `X` and the exponent:

```
+7
```

The table guarantees that:

- `power` is unique
- `factor` is never zero
- `power` is between `0` and `100`

Because powers are unique, each power contributes exactly one term to the equation. This simplifies the problem significantly because we do not need to combine like terms.

The main challenges are formatting correctness and handling the special cases for powers `0` and `1`.

Several edge cases are important:

- Constant terms where `power = 0`
- Linear terms where `power = 1`
- Negative coefficients
- Ensuring powers are sorted descending
- Correct placement of signs for positive and negative terms
- Avoiding invalid formats like `X^1` or `X^0`

Although the constraints are small, correctness of string construction is critical.

## Approaches

### Brute Force Approach

A straightforward solution is to iterate through every row, generate a formatted string for each term, store all terms in a temporary list, sort the list by power descending, and finally concatenate everything together.

This approach works because every row is independent. We can fully format each term without needing information from any other row.

The implementation would typically:

1. Read every row
2. Convert it into its textual polynomial representation
3. Store `(power, formatted_term)`
4. Sort descending by power
5. Concatenate all strings
6. Append `=0`

This approach is already efficient enough for the given constraints because the number of powers is at most `101`.

### Optimal Approach

The optimal SQL solution leverages SQL string functions and aggregation directly inside the database engine.

The key observation is that we do not need procedural construction outside SQL. Instead, we can:

- Build each term using conditional logic
- Sort rows by descending power
- Concatenate all formatted terms using `GROUP_CONCAT`

Because powers are unique and the table is small, the database can efficiently process everything in a single query.

This produces a concise and highly efficient solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) | O(n) | Requires sorting formatted terms |
| Optimal | O(n log n) | O(n) | Uses SQL aggregation and ordering directly |

## Algorithm Walkthrough

1. Read all rows from the `Terms` table.

Each row contains a coefficient (`factor`) and exponent (`power`). Every row corresponds to one polynomial term.
2. Sort the rows by descending `power`.

The problem explicitly requires that terms appear from highest exponent to lowest exponent.
3. Determine the sign for each term.

If `factor > 0`, prepend `"+"`.

If `factor < 0`, prepend `"-"`.
4. Append the absolute value of the coefficient.

The coefficient itself must never include the negative sign because the sign is handled separately.
5. Handle the power formatting.

If `power > 1`, append:

```
X^<power>
```

If `power == 1`, append:

```
X
```

If `power == 0`, append nothing extra.
6. Concatenate all formatted terms together.

Since every term already includes its sign, simple concatenation produces the full left-hand side.
7. Append `=0`.

The problem requires the equation to always equal zero.

### Why it works

The algorithm works because each polynomial term is independent and the formatting rules are deterministic based solely on `factor` and `power`.

Sorting guarantees the required order. Conditional formatting guarantees the correct textual representation for powers `0`, `1`, and powers greater than `1`. Concatenating all valid term strings therefore produces exactly the required polynomial equation.

## Python Solution

```python
from typing import List

class Solution:
    def buildEquation(self, terms: List[List[int]]) -> str:
        formatted_terms = []

        terms.sort(key=lambda x: -x[0])

        for power, factor in terms:
            sign = "+" if factor > 0 else "-"
            coefficient = str(abs(factor))

            if power == 0:
                term = f"{sign}{coefficient}"
            elif power == 1:
                term = f"{sign}{coefficient}X"
            else:
                term = f"{sign}{coefficient}X^{power}"

            formatted_terms.append(term)

        return "".join(formatted_terms) + "=0"
```

The implementation begins by sorting the terms in descending order of power. This guarantees that the final equation respects the ordering requirement.

For each term, the code first determines the sign. The coefficient is converted to its absolute value because the sign is handled separately.

The implementation then branches into three cases:

- `power == 0`, constant term
- `power == 1`, linear term
- `power > 1`, general polynomial term

Each formatted term is appended into a list, and finally all terms are concatenated together with `=0` appended at the end.

In the actual LeetCode problem, the expected answer is written in SQL rather than Python. The Python version here demonstrates the exact algorithmic logic behind the formatting process.

## Go Solution

```go
package main

import (
	"fmt"
	"sort"
	"strconv"
)

func buildEquation(terms [][]int) string {
	sort.Slice(terms, func(i, j int) bool {
		return terms[i][0] > terms[j][0]
	})

	result := ""

	for _, term := range terms {
		power := term[0]
		factor := term[1]

		sign := "+"
		if factor < 0 {
			sign = "-"
		}

		coefficient := strconv.Itoa(abs(factor))

		if power == 0 {
			result += sign + coefficient
		} else if power == 1 {
			result += sign + coefficient + "X"
		} else {
			result += sign + coefficient + "X^" + strconv.Itoa(power)
		}
	}

	return result + "=0"
}

func abs(x int) int {
	if x < 0 {
		return -x
	}
	return x
}

func main() {
	terms := [][]int{
		{2, 1},
		{1, -4},
		{0, 2},
	}

	fmt.Println(buildEquation(terms))
}
```

The Go implementation follows the same structure as the Python version. The primary differences come from Go's explicit typing and string conversion requirements.

Go requires `sort.Slice` for custom sorting and `strconv.Itoa` for integer-to-string conversion. Since Go does not provide a built-in integer absolute value function for plain integers, a small helper function is included.

String concatenation is done directly with `+=`, which is acceptable here because the maximum input size is extremely small.

## Edge Cases

One important edge case is when `power = 0`. In this situation, the term must not contain either `X` or an exponent. A naive implementation might incorrectly generate something like `+5X^0`. The solution explicitly checks for `power == 0` and formats only the coefficient.

Another important edge case is when `power = 1`. The exponent must be omitted entirely. A careless implementation could produce `+3X^1`, which violates the formatting rules. The implementation handles this with a dedicated conditional branch.

Negative coefficients are another common source of bugs. The sign must appear separately from the coefficient value, and the coefficient itself must use the absolute value. Without careful handling, an implementation might accidentally generate invalid strings such as `--4X` or `+-4X`. The solution avoids this by computing the sign independently and always formatting the coefficient using `abs(factor)`.

A final edge case involves sorting. Even if the input arrives in arbitrary order, the output must always be sorted by descending powers. The implementation explicitly sorts before formatting, ensuring deterministic and correct output ordering every time.
