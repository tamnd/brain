## Solution

## Problem Understanding

In this problem, we are given an array called `differences`, where each value represents the difference between two consecutive elements in some unknown array called `hidden`.

If:

```
differences[i] = hidden[i + 1] - hidden[i]
```

then once we know the first value of the hidden sequence, every remaining value becomes completely determined.

For example:

```
differences = [1, -3, 4]
```

If we choose:

```
hidden[0] = 3
```

then:

```
hidden[1] = 3 + 1 = 4
hidden[2] = 4 - 3 = 1
hidden[3] = 1 + 4 = 5
```

So the sequence becomes:

```
[3, 4, 1, 5]
```

The problem also gives two integers, `lower` and `upper`, which define the allowed inclusive range for every element in the hidden sequence.

Our task is to determine how many valid starting values can produce an entire sequence where every number stays inside:

```
[lower, upper]
```

The constraints are important:

- `n` can be as large as `10^5`
- Each difference can also be large
- A brute-force simulation over all possible sequences would be too slow

These constraints strongly suggest that we need an `O(n)` solution.

An important observation is that the entire sequence is determined solely by the starting value. Therefore, instead of constructing many sequences independently, we should determine the range of valid starting values mathematically.

Several edge cases are important:

- The sequence may exceed the allowed range immediately.
- The valid range of starting values may become empty.
- All differences may be zero, meaning every sequence element is identical.
- The range `[lower, upper]` may contain only one value.
- Large positive and negative prefix movements can shrink the valid starting interval to zero.

## Approaches

### Brute Force Approach

A straightforward idea is to try every possible starting value from `lower` to `upper`.

For each starting value:

1. Build the hidden sequence using the differences array.
2. Check whether every generated value stays within `[lower, upper]`.
3. Count the valid sequences.

This approach is correct because every possible hidden sequence corresponds to exactly one starting value.

However, it is inefficient.

The range size can be as large as:

```
upper - lower + 1 = 200001
```

For each starting value, we may process up to `10^5` differences.

That leads to a worst-case complexity of:

```
O((upper - lower) * n)
```

which is too slow.

### Optimal Prefix Sum Approach

The key insight is that every hidden value can be represented relative to the first element.

Suppose:

```
hidden[0] = x
```

Then:

```
hidden[1] = x + differences[0]
hidden[2] = x + differences[0] + differences[1]
...
```

The accumulated sums of `differences` are prefix sums.

If we define:

```
prefix[i] = differences[0] + ... + differences[i-1]
```

then every hidden value is:

```
x + prefix[i]
```

To keep every value inside `[lower, upper]`, we require:

```
lower <= x + prefix[i] <= upper
```

Rearranging:

```
lower - prefix[i] <= x <= upper - prefix[i]
```

Each prefix sum creates a valid interval for `x`.

The intersection of all these intervals gives all valid starting values.

Instead of checking every possible sequence individually, we only need:

- the minimum prefix sum
- the maximum prefix sum

If:

```
minPrefix = minimum prefix sum
maxPrefix = maximum prefix sum
```

then:

```
x + minPrefix >= lower
x + maxPrefix <= upper
```

So:

```
x >= lower - minPrefix
x <= upper - maxPrefix
```

The count of valid integers in this interval is:

```
(upper - maxPrefix) - (lower - minPrefix) + 1
```

if positive, otherwise `0`.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O((upper - lower) × n) | O(1) | Tries every starting value independently |
| Optimal | O(n) | O(1) | Uses prefix sum range analysis |

## Algorithm Walkthrough

1. Initialize a running prefix sum called `currentPrefix` to `0`.

This represents how far the current sequence element is from the starting value.
2. Initialize two variables:

- `minPrefix = 0`
- `maxPrefix = 0`

These track the minimum and maximum prefix sums encountered.
3. Traverse the `differences` array.

For each difference:

- Add it to `currentPrefix`
- Update `minPrefix`
- Update `maxPrefix`

This tells us the smallest and largest offsets that any hidden value can have relative to the starting value.
4. Compute the valid range of the starting value.

Since every sequence value must remain within `[lower, upper]`:

```
startingValue + minPrefix >= lower
```

therefore:

```
startingValue >= lower - minPrefix
```

Similarly:

```
startingValue + maxPrefix <= upper
```

therefore:

```
startingValue <= upper - maxPrefix
```
5. Compute the number of integers inside the valid interval.

Let:

```
leftBound = lower - minPrefix
rightBound = upper - maxPrefix
```

Then the number of valid starting values is:

```
rightBound - leftBound + 1
```
6. If the interval is invalid, return `0`.

Specifically, if:

```
rightBound < leftBound
```

there are no valid sequences.

### Why it works

Every element of the hidden sequence equals:

```
startingValue + prefixSum
```

The smallest prefix sum determines how low the sequence can go, and the largest prefix sum determines how high it can go.

Therefore, ensuring that the starting value keeps both extremes inside the allowed range automatically guarantees that every intermediate value also remains valid.

The algorithm works because it transforms the problem from generating sequences into finding the intersection of valid intervals for the starting value.

## Python Solution

```python
from typing import List

class Solution:
    def numberOfArrays(self, differences: List[int], lower: int, upper: int) -> int:
        current_prefix = 0
        min_prefix = 0
        max_prefix = 0

        for diff in differences:
            current_prefix += diff
            min_prefix = min(min_prefix, current_prefix)
            max_prefix = max(max_prefix, current_prefix)

        left_bound = lower - min_prefix
        right_bound = upper - max_prefix

        return max(0, right_bound - left_bound + 1)
```

The implementation follows the mathematical derivation directly.

We maintain a running prefix sum while scanning through `differences`. During this scan, we record the smallest and largest prefix sums encountered.

After the traversal:

- `min_prefix` represents the lowest deviation from the starting value.
- `max_prefix` represents the highest deviation from the starting value.

Using these values, we derive the valid interval for the starting value.

Finally, we compute the number of integers inside that interval. If the interval is empty, we return `0`.

The implementation uses only constant extra space and processes the array once.

## Go Solution

```go
func numberOfArrays(differences []int, lower int, upper int) int {
	currentPrefix := 0
	minPrefix := 0
	maxPrefix := 0

	for _, diff := range differences {
		currentPrefix += diff

		if currentPrefix < minPrefix {
			minPrefix = currentPrefix
		}

		if currentPrefix > maxPrefix {
			maxPrefix = currentPrefix
		}
	}

	leftBound := lower - minPrefix
	rightBound := upper - maxPrefix

	if rightBound < leftBound {
		return 0
	}

	return rightBound - leftBound + 1
}
```

The Go implementation mirrors the Python logic closely.

Go does not provide built-in `min` and `max` functions for integers, so we use explicit comparisons.

All calculations safely fit within Go's `int` type under the given constraints.

No additional slices or arrays are needed, so the space complexity remains constant.

## Edge Cases

One important edge case occurs when all differences are zero. In this situation, every element in the hidden sequence is identical to the starting value. A buggy implementation might overcomplicate the prefix handling, but the correct logic naturally handles it because both `minPrefix` and `maxPrefix` remain zero. The answer becomes simply the number of integers in `[lower, upper]`.

Another important case occurs when the valid interval becomes empty. This happens when the sequence fluctuates too much and no starting value can keep every element inside bounds. For example, a large positive prefix sum combined with a large negative prefix sum may force contradictory requirements on the starting value. The implementation correctly detects this by checking whether:

```
rightBound < leftBound
```

and returning `0`.

A third important edge case involves very large prefix movements near the constraint limits. Since differences can reach `±100000`, prefix sums may grow substantially. A naive implementation that repeatedly reconstructs sequences could become inefficient. Our implementation only tracks the minimum and maximum prefix sums, so it remains efficient and safe even for the largest inputs.

Another subtle case is when the valid interval contains exactly one value. In this scenario:

```
leftBound == rightBound
```

The implementation correctly returns:

```
1
```

because there is exactly one valid starting value that satisfies all constraints.
