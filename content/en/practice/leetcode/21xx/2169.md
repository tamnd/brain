## Solution

## Problem Understanding

The problem gives us two non-negative integers, `num1` and `num2`. In each operation, we compare the two values:

- If `num1 >= num2`, we subtract `num2` from `num1`
- Otherwise, we subtract `num1` from `num2`

We continue performing these operations until at least one of the numbers becomes `0`. The task is to return the total number of operations performed.

In simpler terms, we repeatedly reduce the larger number by the smaller number. Every subtraction counts as one operation. Eventually, one number reaches zero because the values continuously decrease and never become negative.

The input consists of two integers within the range:

```
0 <= num1, num2 <= 100000
```

The constraints are fairly small, so even a straightforward simulation is acceptable. Since the values only decrease, we are guaranteed that the process terminates.

There are several important edge cases to consider:

- One number may already be `0` initially. In that case, no operations are needed.
- Both numbers may be equal. A single subtraction immediately makes one value `0`.
- One number may be much larger than the other, causing many repeated subtraction steps.
- Since the inputs are non-negative, we never need to handle negative values.

The problem is essentially a simulation of repeated subtraction, closely related to the subtraction-based form of the Euclidean algorithm.

## Approaches

### Brute Force Approach

The most direct solution is to simulate exactly what the problem statement describes.

At each step:

- Compare `num1` and `num2`
- Subtract the smaller value from the larger value
- Increment an operation counter
- Stop when either number becomes `0`

This works because the problem explicitly defines the process. Every operation strictly decreases one of the values, so the algorithm eventually terminates.

However, this approach can be inefficient when one number is much larger than the other. For example, if `num1 = 100000` and `num2 = 1`, we would perform `100000` separate subtraction operations.

Even though this still fits comfortably within the given constraints, we can improve the solution by recognizing a mathematical pattern.

### Optimal Approach

The key observation is that repeatedly subtracting the smaller number from the larger number is equivalent to integer division.

Suppose:

```
num1 = 20
num2 = 5
```

Instead of subtracting `5` four separate times:

```
20 -> 15 -> 10 -> 5 -> 0
```

we can directly determine that:

```
20 // 5 = 4
```

This means we can perform four operations at once.

After adding `4` to the operation count, we update:

```
num1 = 20 % 5 = 0
```

This transforms the repeated subtraction process into the modulo-based Euclidean algorithm.

The algorithm becomes:

- Add `larger // smaller` to the operation count
- Replace the larger value with `larger % smaller`
- Repeat until one number becomes `0`

This significantly reduces the number of iterations.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(max(num1, num2)) | O(1) | Simulates every subtraction individually |
| Optimal | O(log(min(num1, num2))) | O(1) | Uses division and modulo like the Euclidean algorithm |

## Algorithm Walkthrough

1. Initialize a variable `operations` to `0`.

This variable keeps track of the total number of operations performed.
2. Continue processing while both numbers are greater than `0`.

Once either number becomes `0`, the process stops according to the problem statement.
3. Compare `num1` and `num2`.

We always reduce the larger number because the rules require subtracting the smaller value from the larger value.
4. If `num1 >= num2`, compute how many times `num2` fits into `num1`.

```
quotient = num1 // num2
```

This tells us how many subtraction operations would occur consecutively.
5. Add the quotient to the operation count.

```
operations += quotient
```
6. Replace `num1` with the remainder.

```
num1 %= num2
```

This simulates all repeated subtractions at once.
7. Otherwise, if `num2 > num1`, do the symmetric operation.

```
operations += num2 // num1
num2 %= num1
```
8. Repeat until one number becomes `0`.
9. Return the total operation count.

### Why it works

Each subtraction operation reduces the larger number by exactly the smaller number. Repeating subtraction multiple times is mathematically identical to division with remainder.

For example:

```
a = b * q + r
```

means:

- `q` subtraction operations occur
- `r` is the remaining value afterward

Using integer division and modulo preserves the exact behavior of the original process while skipping unnecessary repeated steps. Therefore, the algorithm produces exactly the same operation count as the direct simulation.

## Python Solution

```python
class Solution:
    def countOperations(self, num1: int, num2: int) -> int:
        operations = 0

        while num1 > 0 and num2 > 0:
            if num1 >= num2:
                operations += num1 // num2
                num1 %= num2
            else:
                operations += num2 // num1
                num2 %= num1

        return operations
```

The implementation starts by initializing an operation counter.

The `while` loop continues as long as both numbers remain positive. Inside the loop, we determine which number is larger.

If `num1` is larger or equal, we compute how many times `num2` can be subtracted from it using integer division. That quotient directly increases the operation count. We then replace `num1` with the remainder using modulo.

The opposite case is handled symmetrically when `num2` is larger.

Eventually, one value becomes `0`, causing the loop to terminate. The final operation count is returned.

This implementation closely mirrors the Euclidean algorithm and avoids unnecessary repeated subtraction steps.

## Go Solution

```go
func countOperations(num1 int, num2 int) int {
    operations := 0

    for num1 > 0 && num2 > 0 {
        if num1 >= num2 {
            operations += num1 / num2
            num1 %= num2
        } else {
            operations += num2 / num1
            num2 %= num1
        }
    }

    return operations
}
```

The Go implementation follows the same logic as the Python version.

Go uses integer division automatically when dividing two integers, so `num1 / num2` behaves exactly as needed. The modulo operator `%` also works identically.

Since the constraints are small, integer overflow is not a concern. All variables comfortably fit within Go's standard `int` type.

## Edge Cases

One important edge case occurs when one of the inputs is already zero at the beginning. Since the problem stops as soon as either number becomes zero, the correct answer is immediately `0`. The implementation handles this naturally because the `while` loop condition fails immediately.

Another important case is when both numbers are equal. In that situation, exactly one subtraction operation occurs, reducing one value directly to zero. A buggy implementation might accidentally continue processing after the first operation, but this implementation correctly exits because one number becomes zero after the modulo operation.

A third important edge case occurs when one number is much larger than the other, such as `(100000, 1)`. A naive repeated subtraction implementation would perform many individual iterations. The optimized solution handles this efficiently using integer division, computing all operations in a single step.

A final subtle case involves consecutive reductions where the remainder becomes zero immediately. For example, `(20, 5)` should terminate after four operations. The modulo operation correctly produces zero, and the loop exits immediately without extra processing.
