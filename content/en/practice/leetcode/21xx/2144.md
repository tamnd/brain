# LeetCode 2144 - Minimum Cost of Buying Candies With Discount

## Problem Understanding

The problem gives us an array called `cost`, where each element represents the price of a candy. A store offers a promotion: whenever you buy two candies, you may get one additional candy for free. However, the free candy must have a price less than or equal to the cheaper of the two candies you purchased.

Our goal is to buy all candies while spending the smallest possible amount of money.

The important detail is that we are allowed to choose which candies are paid for and which candies become free. Since the free candy must not cost more than the minimum of the two purchased candies, we want to structure purchases so that the most expensive possible candies are paid for, while still maximizing the value of the free candies.

The input is a list of integers:

- `cost[i]` is the price of the `i-th` candy.
- We must include every candy exactly once, either as purchased or free.

The output is a single integer representing the minimum total amount of money needed to obtain all candies.

The constraints are small:

- `1 <= cost.length <= 100`
- `1 <= cost[i] <= 100`

These constraints tell us that even less efficient algorithms could work in practice, but the problem is fundamentally a greedy optimization problem, and there is a very clean optimal solution.

There are several edge cases worth identifying early:

- If there are fewer than 3 candies, no candy can be free.
- If the number of candies is not divisible by 3, some candies will remain outside a full discount group.
- Multiple candies may have the same value.
- The free candy must always be less than or equal to both purchased candies, which affects how groups can be formed.

## Approaches

### Brute Force Approach

A brute-force solution would attempt every possible grouping of candies into sets of three, where two candies are paid for and one is free. For each grouping, we would verify whether the free candy satisfies the discount condition and then compute the total cost.

This approach is correct because it explores all possible valid purchase combinations and therefore eventually finds the minimum cost.

However, the number of possible groupings grows extremely quickly as the number of candies increases. Even with only 100 candies, the number of combinations becomes astronomically large. This makes brute force computationally infeasible.

### Key Greedy Insight

The key observation is that the free candy should always be as expensive as possible.

Suppose we sort the candies in descending order:

- The two largest candies in a group must be paid for.
- The third candy, which is smaller or equal, can become free.

Therefore, after sorting from largest to smallest, every third candy can be taken for free.

For example:

```
Sorted descending: [9, 7, 6, 5, 2, 2]
```

We can form groups like:

```
(9, 7) -> 6 free
(5, 2) -> 2 free
```

This greedy strategy maximizes the value of each free candy.

| --- | --- | --- | --- |

| Optimal Greedy | O(n log n) | O(1) or O(log n) depending on sorting | Sort descending and skip every third candy |

## Algorithm Walkthrough

1. Sort the `cost` array in descending order.

We place the most expensive candies first because expensive candies should be paired together to maximize the value of the free candy.
2. Initialize a variable `total_cost` to `0`.

This variable stores the amount we actually pay.
3. Traverse the sorted array using an index.

Every group of three candies follows this pattern:

- First candy: paid
- Second candy: paid
- Third candy: free
4. Add the cost of candies whose indices are not multiples of 3 after the first two positions.

More specifically:

- Indices `0` and `1` are paid
- Index `2` is free
- Indices `3` and `4` are paid
- Index `5` is free
5. Skip every third candy because it becomes the free candy.
6. Return the accumulated total.

### Why it works

After sorting in descending order, every candy is less than or equal to the two candies before it. Therefore, when we skip every third candy, that candy is guaranteed to satisfy the discount rule.

This strategy is optimal because the free candy in each group is the most expensive candy that can legally be free. Any other arrangement would either waste the discount on a cheaper candy or force a more expensive candy to be paid for.

## Python Solution

```python
from typing import List

class Solution:
    def minimumCost(self, cost: List[int]) -> int:
        cost.sort(reverse=True)

        total_cost = 0

        for index, candy_cost in enumerate(cost):
            if index % 3 != 2:
                total_cost += candy_cost

        return total_cost
```

The implementation begins by sorting the candies in descending order. This guarantees that every third candy is less than or equal to the previous two candies.

We then iterate through the sorted list using `enumerate`, which provides both the index and the candy price.

The expression `index % 3 != 2` identifies candies that must be paid for:

- Index `0`: paid
- Index `1`: paid
- Index `2`: free
- Index `3`: paid
- Index `4`: paid
- Index `5`: free

Whenever the condition is true, we add the candy price to `total_cost`.

Finally, we return the accumulated result.

## Go Solution

```go
package main

import (
	"sort"
)

func minimumCost(cost []int) int {
	sort.Slice(cost, func(i, j int) bool {
		return cost[i] > cost[j]
	})

	totalCost := 0

	for i, candyCost := range cost {
		if i%3 != 2 {
			totalCost += candyCost
		}
	}

	return totalCost
}
```

The Go solution follows the same logic as the Python version.

The main difference is sorting. In Go, `sort.Slice` is used with a custom comparison function to sort in descending order.

Go slices are dynamic references to arrays, so sorting modifies the original slice in place. Since all values are small integers and the result easily fits inside a standard integer type, there are no integer overflow concerns here.

## Edge Cases

One important edge case occurs when the array contains fewer than three candies. Since the promotion requires buying two candies to receive a third for free, no discount can be applied. The implementation handles this naturally because the loop never reaches an index where `index % 3 == 2`.

Another important case is when the number of candies is not divisible by three. In such situations, some candies remain outside a complete discount group and must be paid for normally. The algorithm handles this correctly because only every third candy is skipped.

A third edge case involves duplicate candy prices. For example, `[2,2,2]` still allows one candy to be free because the free candy only needs to be less than or equal to the cheaper purchased candy. Sorting and skipping every third element continues to work correctly even when values are repeated.

A final subtle edge case is already sorted or reverse-sorted input. Since the algorithm explicitly sorts the array before processing, the original ordering never affects correctness.
