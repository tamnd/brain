## Solution

## Problem Understanding

The problem asks us to count how many distinct houses of cards can be built using exactly `n` cards.

A valid house of cards is made of rows of triangles and horizontal support cards. Each triangle uses exactly 2 cards. Between every pair of adjacent triangles in the same row, there must be 1 horizontal card. Higher rows must sit correctly on top of horizontal cards from the row below.

The key observation is that a row containing `k` triangles requires:

- `2 * k` cards for the triangles
- `k - 1` cards for the horizontal supports between them

So a row with `k` triangles uses:

$$2k + (k - 1) = 3k - 1$$

cards.

The structure also imposes a stacking rule. Since every upper triangle must rest on a horizontal card from the previous row, the number of triangles must strictly decrease as we move upward. That means if a lower row has `k` triangles, the next row can have at most `k - 1` triangles.

The input is a single integer `n`, representing the exact number of cards available. The output is the number of distinct valid houses that use all `n` cards.

The constraint `1 <= n <= 500` is relatively small. This strongly suggests that a dynamic programming solution is appropriate. Exhaustive recursion may work with memoization, but brute force enumeration without pruning would become inefficient.

Several edge cases are important:

- Small values like `n = 1` or `n = 4`, where no valid arrangement exists
- Exact row-cost matches such as `n = 2`, where a single triangle is valid
- Multiple-row combinations where rows must strictly decrease
- Values that can be decomposed into row costs in several distinct ways

## Approaches

### Brute Force Approach

A brute force solution would recursively try every possible row size at every level.

For each recursive call:

- Choose a valid number of triangles for the current row
- Subtract that row's card cost from the remaining cards
- Continue building upper rows with strictly fewer triangles

This approach is correct because it explores every legal arrangement and counts all valid constructions.

However, without memoization, the same subproblems appear repeatedly. For example, many recursive paths may eventually ask:

- "How many ways can I build using 20 cards with at most 4 triangles in the next row?"

Recomputing these states repeatedly leads to exponential complexity.

### Key Insight

The important observation is that each row has a deterministic cost:

$$\text{rowCost}(k) = 3k - 1$$

Additionally, rows must strictly decrease in triangle count.

This transforms the problem into a constrained partition problem:

- Select distinct positive integers `k1 > k2 > k3 ...`
- Such that:

$$(3k_1 - 1) + (3k_2 - 1) + \cdots = n$$

This naturally leads to dynamic programming.

We define a recursive state:

- Remaining cards
- Maximum number of triangles allowed in the next row

Memoization ensures each state is solved only once.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | O(depth) | Recomputes many overlapping states |
| Optimal Dynamic Programming | O(n²) | O(n²) | Memoized recursion over remaining cards and row size |

## Algorithm Walkthrough

### Step 1: Define the row cost formula

A row with `k` triangles requires:

$$3k - 1$$

cards.

This lets us compute exactly how many cards are consumed by any row choice.

### Step 2: Define the recursive state

We define:

```
dfs(remaining, maxTriangles)
```

where:

- `remaining` is the number of cards left to place
- `maxTriangles` is the maximum number of triangles allowed in the next row

The maximum is necessary because upper rows must contain strictly fewer triangles than lower rows.

### Step 3: Handle base cases

If `remaining == 0`, we found a valid complete house, so return `1`.

If `remaining < 0`, the configuration is invalid, so return `0`.

### Step 4: Try every valid next row

For every possible row size:

```
1 <= k <= maxTriangles
```

we compute:

```
cost = 3*k - 1
```

If the row fits within the remaining cards, recursively solve:

```
dfs(remaining - cost, k - 1)
```

The next row can contain at most `k - 1` triangles.

### Step 5: Memoize results

Many states repeat during recursion. We cache results for:

```
(remaining, maxTriangles)
```

This reduces the complexity dramatically.

### Step 6: Initialize the search

The largest possible first row occurs when:

$$3k - 1 \le n$$

So:

$$k \le \frac{n + 1}{3}$$

We start recursion with this maximum possible row size.

### Why it works

The algorithm works because every valid house can be uniquely represented as a strictly decreasing sequence of row sizes.

At every recursive step:

- We choose exactly one valid row
- We enforce the decreasing-row constraint
- We reduce the remaining cards correctly

Memoization guarantees each distinct subproblem is solved once, and every legal construction is counted exactly once.

## Python Solution

```python
class Solution:
    def houseOfCards(self, n: int) -> int:
        from functools import lru_cache

        max_first_row = (n + 1) // 3

        @lru_cache(None)
        def dfs(remaining: int, max_triangles: int) -> int:
            if remaining == 0:
                return 1

            if remaining < 0 or max_triangles == 0:
                return 0

            ways = 0

            for triangles in range(1, max_triangles + 1):
                cost = 3 * triangles - 1

                if cost <= remaining:
                    ways += dfs(remaining - cost, triangles - 1)

            return ways

        return dfs(n, max_first_row)
```

The implementation begins by computing the maximum possible number of triangles in the first row. Since a row with `k` triangles costs `3k - 1` cards, no larger row can fit within `n`.

The recursive `dfs` function explores all valid row choices. The parameter `remaining` tracks unused cards, while `max_triangles` enforces the strictly decreasing row condition.

The base case `remaining == 0` means we successfully used all cards in a valid arrangement. Invalid states return `0`.

For every possible row size, we compute its card cost and recursively solve the smaller subproblem. Memoization via `lru_cache` prevents repeated computation.

## Go Solution

```go
func houseOfCards(n int) int {
    maxFirstRow := (n + 1) / 3

    memo := make(map[[2]int]int)

    var dfs func(int, int) int

    dfs = func(remaining int, maxTriangles int) int {
        if remaining == 0 {
            return 1
        }

        if remaining < 0 || maxTriangles == 0 {
            return 0
        }

        key := [2]int{remaining, maxTriangles}

        if val, exists := memo[key]; exists {
            return val
        }

        ways := 0

        for triangles := 1; triangles <= maxTriangles; triangles++ {
            cost := 3*triangles - 1

            if cost <= remaining {
                ways += dfs(remaining-cost, triangles-1)
            }
        }

        memo[key] = ways
        return ways
    }

    return dfs(n, maxFirstRow)
}
```

The Go version follows the same recursive memoized strategy as the Python solution.

Since Go does not provide built-in memoization decorators like Python's `lru_cache`, we explicitly store computed states in a map. The key is a two-element array containing:

- Remaining cards
- Maximum allowed triangles

Go integers are sufficient because the answer size remains small for `n <= 500`.

## Edge Cases

One important edge case occurs when `n` is too small to form even a single valid row. Since the smallest row requires 2 cards, values like `n = 1` must immediately return `0`. The implementation handles this naturally because no valid recursive choices exist.

Another tricky case occurs when cards remain but no smaller row can legally be placed. For example, after building a row with 1 triangle, the next row must contain fewer than 1 triangle, which is impossible. The recursion correctly stops because `max_triangles` becomes `0`.

A third important case involves exact row-cost matches such as `n = 5`, `8`, or `11`. These represent single-row houses with 2, 3, and 4 triangles respectively. The implementation correctly counts these because reaching `remaining == 0` contributes one valid arrangement.

A subtle source of bugs is double-counting arrangements. Since rows must strictly decrease in size, every valid house has a unique recursive construction order. The `max_triangles` restriction guarantees that permutations of the same rows are not counted multiple times.
