# LeetCode 2187 - Minimum Time to Complete Trips

## Problem Understanding

The problem gives us an array called `time`, where each element represents how long a specific bus takes to complete exactly one trip. A bus can immediately begin another trip after finishing one, and all buses operate independently from one another.

We are also given an integer `totalTrips`, which represents the minimum number of trips that all buses combined must complete.

The goal is to determine the smallest amount of time required so that the total number of completed trips across all buses is at least `totalTrips`.

For example, if a bus takes 2 units of time per trip, then in 6 units of time it can complete:

```
6 // 2 = 3 trips
```

Similarly, if another bus takes 3 units of time, then in 6 units of time it can complete:

```
6 // 3 = 2 trips
```

Together, these buses complete 5 trips in 6 units of time.

The constraints are important:

- The number of buses can be as large as `10^5`
- Each bus time can be as large as `10^7`
- `totalTrips` can also be as large as `10^7`

These constraints immediately tell us that we cannot simulate time unit by time unit. A naive simulation could require billions or trillions of operations.

The problem guarantees:

- Every bus time is positive
- At least one bus exists
- At least one trip is required

Several edge cases are important:

- Only one bus exists
- All buses have identical speeds
- One bus is dramatically faster than the others
- `totalTrips` is extremely large
- The answer may exceed 32-bit integer range

Because the answer can become very large, we must use 64-bit integers in Go.

## Approaches

### Brute Force Approach

A straightforward approach is to simulate time starting from `1` and repeatedly check how many trips all buses can complete at that moment.

At each time `t`, we compute:

```
trips = sum(t // bus_time for each bus)
```

If the total trips are at least `totalTrips`, we return `t`.

This approach is correct because we eventually encounter the first valid time. However, it is far too slow. The minimum required time could be extremely large, potentially up to:

```
min(time) * totalTrips
```

With constraints up to `10^7`, the answer could reach `10^14`.

Iterating through every time value up to `10^14` is infeasible.

### Optimal Approach

The key insight is that the number of completed trips increases monotonically as time increases.

If the buses can complete at least `totalTrips` trips at time `t`, then they can also complete at least `totalTrips` trips at any later time.

This monotonic property makes the problem ideal for binary search.

Instead of checking every possible time, we binary search over the answer space.

For a candidate time `mid`, we calculate:

```
completed_trips = sum(mid // bus_time)
```

If `completed_trips >= totalTrips`, then `mid` might be the answer, so we try smaller times.

Otherwise, we need more time, so we search the larger half.

This reduces the search dramatically from linear time to logarithmic time.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(answer × n) | O(1) | Checks every possible time value |
| Optimal | O(n × log(answer)) | O(1) | Uses binary search on time |

## Algorithm Walkthrough

1. Determine the search boundaries for binary search.

The minimum possible time is `1`.

The maximum possible time occurs when the fastest bus completes all trips alone:

```
min(time) * totalTrips
```

This guarantees the real answer lies within the search range.
2. Perform binary search on the time range.

For each midpoint `mid`, interpret it as a candidate amount of time.
3. Compute how many trips all buses can complete within `mid`.

For each bus with trip duration `t`:

```
mid // t
```

gives the number of trips that bus can finish.

Sum these values across all buses.
4. Compare the total completed trips with `totalTrips`.

- If the completed trips are at least the target, then `mid` is a valid answer candidate.
- Store it and continue searching the left half for a potentially smaller valid time.
- Otherwise, search the right half because more time is needed.
5. Continue until the binary search range collapses.

The smallest valid time found is the answer.

### Why it works

The correctness relies on monotonicity.

If a certain amount of time is sufficient to complete the required trips, then any larger amount of time is also sufficient. Conversely, if a time value is insufficient, then all smaller values are also insufficient.

Binary search works perfectly on such monotonic conditions because it repeatedly eliminates half of the remaining search space while preserving the location of the first valid answer.

## Python Solution

```python
from typing import List

class Solution:
    def minimumTime(self, time: List[int], totalTrips: int) -> int:
        left = 1
        right = min(time) * totalTrips

        while left < right:
            mid = (left + right) // 2

            completed_trips = 0

            for bus_time in time:
                completed_trips += mid // bus_time

            if completed_trips >= totalTrips:
                right = mid
            else:
                left = mid + 1

        return left
```

The implementation starts by initializing the binary search boundaries.

The lower bound is `1` because time cannot be zero when at least one trip must be completed.

The upper bound is computed as:

```python
min(time) * totalTrips
```

This represents the scenario where the fastest bus completes every required trip alone.

Inside the binary search loop, we calculate the midpoint `mid`. This midpoint represents a candidate time value.

We then compute how many trips can be completed by all buses during this time. Each bus contributes:

```python
mid // bus_time
```

trips.

If the total completed trips meet or exceed `totalTrips`, we know the answer is at most `mid`, so we shrink the search range to the left half.

Otherwise, we need more time, so we move the lower boundary upward.

Eventually, `left` and `right` converge to the smallest valid time.

## Go Solution

```go
func minimumTime(time []int, totalTrips int) int64 {
    minTime := time[0]

    for _, t := range time {
        if t < minTime {
            minTime = t
        }
    }

    left := int64(1)
    right := int64(minTime) * int64(totalTrips)

    for left < right {
        mid := left + (right-left)/2

        var completedTrips int64 = 0

        for _, t := range time {
            completedTrips += mid / int64(t)
        }

        if completedTrips >= int64(totalTrips) {
            right = mid
        } else {
            left = mid + 1
        }
    }

    return left
}
```

The Go implementation follows the same algorithm as the Python version, but there are several language-specific considerations.

The most important difference is integer overflow handling. Since the answer can grow as large as `10^14`, we must use `int64` instead of regular `int`.

The binary search variables `left`, `right`, and `mid` are therefore all stored as `int64`.

The midpoint calculation uses:

```go
mid := left + (right-left)/2
```

instead of:

```go
(left + right) / 2
```

This avoids overflow in more general binary search implementations.

Go slices are used directly, and no additional data structures are needed.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n × log(min(time) × totalTrips)) | Binary search over the answer space, each step scans all buses |
| Space | O(1) | Only constant extra variables are used |

The binary search operates over a range whose size is approximately:

```
min(time) * totalTrips
```

Binary search reduces this range logarithmically.

For each binary search iteration, we iterate through all buses once to compute the total completed trips.

Therefore, the total complexity becomes:

```
O(n × log(answer))
```

which easily fits within the constraints.

## Edge Cases

### Single Bus

When only one bus exists, the answer is simply:

```
time[0] * totalTrips
```

A naive implementation might still work, but binary search must correctly handle a search space where only one bus contributes trips. The implementation handles this naturally because the trip calculation loop still works with one element.

### Extremely Large Answer

The maximum possible answer can reach:

```
10^7 * 10^7 = 10^14
```

This exceeds 32-bit integer limits. In Go, using regular `int` could cause overflow on some systems. The implementation explicitly uses `int64` for all binary search computations and trip totals.

### One Bus Much Faster Than Others

Consider:

```
time = [1, 10000000]
```

The faster bus dominates almost all trip production. A poor implementation might attempt unnecessary simulation of all buses over time. The binary search approach avoids this entirely because it only computes trip counts mathematically using division.

### Minimum Required Trips

When `totalTrips = 1`, the answer should simply be the minimum bus time. Binary search still works correctly because the first valid time encountered will be the smallest bus duration.

### All Buses Have Same Speed

For input like:

```
time = [2,2,2]
```

all buses contribute equally. The implementation correctly sums contributions independently, ensuring no special-case handling is needed.
