# LeetCode 2127 - Maximum Employees to Be Invited to a Meeting

## Problem Understanding

This problem describes a directed graph where each employee points to exactly one other employee, their favorite person. The company wants to arrange invited employees around a circular table so that every invited employee sits next to their favorite employee.

The input array `favorite` has length `n`, where:

- Employee IDs range from `0` to `n - 1`
- `favorite[i]` is the person employee `i` wants to sit next to
- No employee favorites themselves

The task is to compute the maximum number of employees that can be invited while satisfying all favorite relationships simultaneously.

The key detail is the circular seating arrangement. Every employee has exactly two neighbors at the table. Therefore, an employee can only be satisfied if one of those two neighbors is their favorite person.

Because every node has exactly one outgoing edge, the graph is a collection of directed cycles with trees feeding into those cycles.

The constraints are large:

- `n` can be as large as `10^5`

This immediately rules out brute-force permutation generation or exhaustive seating simulation. Any acceptable solution must run close to linear time.

Several edge cases are important:

- A graph may contain one large cycle
- The graph may contain multiple disconnected components
- There may be cycles of length `2`, which behave differently from larger cycles
- Long chains may feed into mutual pairs
- Some nodes may never be part of an optimal seating arrangement

The most subtle part of the problem is understanding why cycles of length `2` are special and can be extended with incoming chains, while larger cycles cannot.

## Approaches

### Brute Force Approach

A brute-force solution would try every possible subset of employees and every possible circular ordering of that subset. For each arrangement, we would verify whether every employee sits adjacent to their favorite person.

This approach is correct because it explicitly checks every valid seating arrangement. However, it is computationally impossible for `n = 10^5`.

Even generating all permutations for relatively small values of `n` becomes infeasible. The number of circular permutations grows factorially.

The brute-force approach therefore cannot satisfy the problem constraints.

### Key Insight

The graph formed by the `favorite` array has a very special structure:

- Every node has exactly one outgoing edge
- Therefore, every connected component contains exactly one directed cycle
- Trees may feed into the cycle

There are only two kinds of structures that can contribute to the answer:

1. A cycle of length greater than `2`
2. A mutual pair, meaning a cycle of length `2`

A cycle of length greater than `2` must remain intact as a closed circular arrangement. No extra employees can be attached to it because every seat around the cycle is already occupied by required neighbors.

A cycle of length `2` is different.

Suppose:

- `a` likes `b`
- `b` likes `a`

Then we can place chains leading into `a` and `b` on opposite sides:

```
chain -> a <-> b <- chain
```

This allows us to extend the seating arrangement with the longest incoming chain for each member of the pair.

The final answer is therefore:

- The size of the largest cycle with length at least `3`
- Or the sum of all extended mutual pairs

We compute both and take the maximum.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n!) | O(n) | Tries all seatings |
| Optimal | O(n) | O(n) | Uses graph structure and topological processing |

## Algorithm Walkthrough

1. Build the indegree array and reverse graph.

Every employee points to one favorite. We count how many incoming edges each node has, and we also store reverse edges so we can later traverse incoming chains.
2. Perform topological pruning.

Nodes with indegree `0` cannot belong to cycles. We place them into a queue and repeatedly remove them.

During this process, we compute:

- `depth[node]`

which represents the longest chain ending at that node.

If `u -> v`, then:

```
depth[v] = max(depth[v], depth[u] + 1)
```
3. After pruning, only cycle nodes remain.

Any node whose indegree is still positive belongs to a cycle.
4. Detect all cycles.

We traverse remaining nodes and compute each cycle length.

If the cycle length is greater than `2`, we update:

```
maxCycle
```
5. Handle mutual pairs separately.

For every pair:

```
a -> b
b -> a
```

we can attach the longest incoming chain to both sides.

The contribution becomes:

```
depth[a] + depth[b]
```

because `depth` already includes the node itself.
6. Sum contributions of all mutual pairs.

Unlike larger cycles, multiple mutual pairs can coexist in the final arrangement because they can be placed independently around the table.
7. Return the larger value.

The answer is:

```
max(maxCycle, pairSum)
```

### Why it works

Every connected component in a functional graph contains exactly one directed cycle.

For cycles longer than `2`, every node already uses both adjacent seats to satisfy cycle neighbors, so no additional chains can be attached.

For a cycle of length `2`, each member only requires the other member as one neighbor, leaving one free seat on each side. This allows exactly one incoming chain to attach to each node.

The algorithm correctly computes:

- The largest standalone cycle
- The total contribution of all extendable mutual pairs

Therefore, it always produces the optimal answer.

## Python Solution

```python
from collections import deque
from typing import List

class Solution:
    def maximumInvitations(self, favorite: List[int]) -> int:
        n = len(favorite)

        indegree = [0] * n

        for v in favorite:
            indegree[v] += 1

        queue = deque()

        for i in range(n):
            if indegree[i] == 0:
                queue.append(i)

        # depth[i] = longest chain ending at i, including i itself
        depth = [1] * n

        while queue:
            node = queue.popleft()
            nxt = favorite[node]

            depth[nxt] = max(depth[nxt], depth[node] + 1)

            indegree[nxt] -= 1

            if indegree[nxt] == 0:
                queue.append(nxt)

        # Find largest cycle
        visited = [False] * n
        max_cycle = 0

        for i in range(n):
            if indegree[i] > 0 and not visited[i]:
                cycle_length = 0
                curr = i

                while not visited[curr]:
                    visited[curr] = True
                    curr = favorite[curr]
                    cycle_length += 1

                max_cycle = max(max_cycle, cycle_length)

        # Sum contributions from all mutual pairs
        pair_sum = 0

        for i in range(n):
            j = favorite[i]

            if favorite[j] == i and i < j:
                pair_sum += depth[i] + depth[j]

        return max(max_cycle, pair_sum)
```

The implementation begins by computing indegrees for all nodes. This allows us to identify nodes that are definitely not part of cycles.

The queue-based topological process removes all tree nodes layer by layer. While removing nodes, we propagate chain lengths forward. The `depth` array stores the longest incoming chain that can reach each node.

After pruning, all remaining nodes belong to cycles. We traverse each cycle once and measure its size.

The final loop identifies every mutual pair. The condition:

```python
favorite[j] == i and i < j
```

ensures that:

- `i` and `j` form a 2-cycle
- each pair is counted exactly once

Finally, we return the larger value between the largest long cycle and the sum of all extended mutual pairs.

## Go Solution

```go
package main

func maximumInvitations(favorite []int) int {
	n := len(favorite)

	indegree := make([]int, n)

	for _, v := range favorite {
		indegree[v]++
	}

	queue := make([]int, 0)

	for i := 0; i < n; i++ {
		if indegree[i] == 0 {
			queue = append(queue, i)
		}
	}

	depth := make([]int, n)

	for i := 0; i < n; i++ {
		depth[i] = 1
	}

	head := 0

	for head < len(queue) {
		node := queue[head]
		head++

		next := favorite[node]

		if depth[node]+1 > depth[next] {
			depth[next] = depth[node] + 1
		}

		indegree[next]--

		if indegree[next] == 0 {
			queue = append(queue, next)
		}
	}

	visited := make([]bool, n)

	maxCycle := 0

	for i := 0; i < n; i++ {
		if indegree[i] > 0 && !visited[i] {
			cycleLength := 0
			curr := i

			for !visited[curr] {
				visited[curr] = true
				curr = favorite[curr]
				cycleLength++
			}

			if cycleLength > maxCycle {
				maxCycle = cycleLength
			}
		}
	}

	pairSum := 0

	for i := 0; i < n; i++ {
		j := favorite[i]

		if favorite[j] == i && i < j {
			pairSum += depth[i] + depth[j]
		}
	}

	if maxCycle > pairSum {
		return maxCycle
	}

	return pairSum
}
```

The Go implementation follows the same logic as the Python version.

Instead of Python's `deque`, the queue is implemented using a slice and a moving head index, which avoids costly front removals.

Go slices are dynamically sized, so they work naturally for queue operations. Integer overflow is not a concern because the answer never exceeds `n`, which is at most `10^5`.

## Edge Cases

### Mutual Pair With Long Chains

A very common bug is mishandling cycles of length `2`.

For example:

```
0 <-> 1
2 -> 0
3 -> 2
4 -> 1
```

The optimal arrangement includes all five employees because chains can attach to both sides of the pair.

The implementation handles this correctly using the `depth` array, which stores the longest incoming chain ending at each pair member.

### Large Cycles Cannot Be Extended

Another subtle case is a cycle longer than `2`.

For example:

```
0 -> 1 -> 2 -> 0
```

No extra chains can be attached because every node already requires both neighboring seats.

A naive implementation might incorrectly try to append incoming chains. The algorithm avoids this by treating cycles of length greater than `2` separately.

### Multiple Incoming Chains Competing

Consider:

```
2 -> 0
3 -> 0
0 <-> 1
```

Both chains cannot simultaneously attach to employee `0`.

Only the longest chain may be used.

The implementation correctly handles this using:

```
depth[next] = max(depth[next], depth[node] + 1)
```

This guarantees that only the best chain contributes to the final answer.
