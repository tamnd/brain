## Solution

## Problem Understanding

The problem gives us two arrays, `plantTime` and `growTime`, where each index represents one flower seed.

For every seed:

- We must spend `plantTime[i]` total days planting it.
- After planting is completely finished, the seed requires `growTime[i]` additional days before blooming.
- A flower blooms only after both planting and growing are complete.

The important detail is that planting work is shared across all seeds. On any single day, we can work on planting exactly one seed. However, planting for a seed does not need to happen consecutively. We can pause one seed and continue another later.

The objective is to minimize the earliest day when every flower has bloomed.

In other words, we want to schedule the planting order so that the maximum bloom completion time among all seeds is as small as possible.

If a seed finishes planting on day `d`, then it blooms on:

```
d + growTime[i]
```

So the final answer is:

```
max(completionPlantDay[i] + growTime[i])
```

The constraints are large:

- `n` can be up to `100000`
- Planting and growing times can each be up to `10000`

This immediately tells us that exponential or factorial solutions are impossible. Any solution that tries every planting order will be far too slow. We need an algorithm around `O(n log n)`.

Several edge cases are important:

- A single seed, where the answer is simply `plantTime[0] + growTime[0]`
- Seeds with very large grow times, because delaying them can significantly increase the final answer
- Seeds with identical grow times, where ordering among them does not matter
- Seeds with tiny planting times but huge growth times, which should likely be planted early
- Seeds with large planting times but tiny growth times, which are safer to delay

## Approaches

### Brute Force Approach

A brute force solution would try every possible planting order.

For each permutation:

1. Simulate planting the seeds in that order
2. Track the cumulative planting time
3. Compute each seed's bloom day
4. Take the maximum bloom day
5. Return the minimum result across all permutations

This works because every valid planting schedule corresponds to some ordering of seeds.

However, there are `n!` possible permutations.

Even for `n = 15`, this becomes computationally infeasible:

```
15! ≈ 1.3 trillion
```

Since the actual constraint is `100000`, brute force is completely impossible.

### Key Insight for the Optimal Solution

The key observation is that seeds with larger growth times should be planted earlier.

Why?

While one seed is growing, we can continue planting other seeds. Growth happens passively. Therefore, long growth periods should begin as soon as possible.

Suppose we have two seeds:

- Seed A with large `growTime`
- Seed B with small `growTime`

If we delay Seed A, its long growth period also gets delayed, increasing the final completion time significantly.

If we instead plant Seed A earlier, its long growth phase overlaps with later planting work for other seeds.

This is a classic greedy scheduling strategy.

Therefore:

- Sort seeds in descending order of `growTime`
- Plant seeds with larger growth times first

After sorting, we simulate the planting process:

- Maintain cumulative planting days
- For each seed:

- Add its planting time
- Compute bloom day
- Track the maximum bloom day

This produces the minimum possible final blooming day.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n! × n) | O(n) | Tries every planting order |
| Optimal Greedy + Sorting | O(n log n) | O(n) | Sort by descending grow time |

## Algorithm Walkthrough

1. Pair each seed's planting and growing times together.

We create pairs like:

```
(growTime[i], plantTime[i])
```

Keeping them together makes sorting and processing easier.
2. Sort the seeds by decreasing `growTime`.

Seeds with larger growth durations should start growing as early as possible.

If two seeds have the same growth time, their relative order does not matter.
3. Initialize two variables:

- `current_plant_days = 0`
- `earliest_full_bloom = 0`

The first tracks total planting work completed so far.

The second tracks the latest bloom day among all processed seeds.
4. Process the sorted seeds one by one.

For each seed:

- Add its planting time to `current_plant_days`
- Compute:

```
bloom_day = current_plant_days + grow_time
```

- Update:

```
earliest_full_bloom = max(earliest_full_bloom, bloom_day)
```
5. After all seeds are processed, return `earliest_full_bloom`.

This represents the earliest day when every flower has bloomed.

### Why it works

The greedy strategy works because long growth periods are the hardest to hide behind other work.

If a seed has a very large growth time, delaying its planting delays its bloom significantly. Planting it earlier allows its growth phase to overlap with later planting work for other seeds.

An exchange argument proves correctness:

- Suppose two adjacent seeds are ordered incorrectly:

- smaller grow time before larger grow time
- Swapping them cannot increase the final completion time
- Repeatedly applying this swap leads to descending grow time order

Therefore, sorting by descending `growTime` is optimal.

## Python Solution

```python
from typing import List

class Solution:
    def earliestFullBloom(self, plantTime: List[int], growTime: List[int]) -> int:
        seeds = list(zip(growTime, plantTime))

        # Plant seeds with larger grow times first
        seeds.sort(reverse=True)

        current_plant_days = 0
        earliest_full_bloom = 0

        for grow, plant in seeds:
            current_plant_days += plant
            bloom_day = current_plant_days + grow

            earliest_full_bloom = max(
                earliest_full_bloom,
                bloom_day
            )

        return earliest_full_bloom
```

The implementation first combines each seed's growth and planting times into tuples. This allows sorting the seeds directly by growth time.

The call to `sort(reverse=True)` sorts tuples primarily by the first element, which is `growTime`. As a result, seeds with larger growth durations are processed first.

The variable `current_plant_days` tracks how many planting days have been spent so far. After planting a seed completely, we calculate when it blooms by adding its growth time.

The answer is the maximum bloom day among all seeds.

The algorithm performs only one sort and one linear pass, making it efficient enough for the large constraints.

## Go Solution

```go
package main

import "sort"

func earliestFullBloom(plantTime []int, growTime []int) int {
	type Seed struct {
		plant int
		grow  int
	}

	n := len(plantTime)

	seeds := make([]Seed, n)

	for i := 0; i < n; i++ {
		seeds[i] = Seed{
			plant: plantTime[i],
			grow:  growTime[i],
		}
	}

	// Sort by descending grow time
	sort.Slice(seeds, func(i, j int) bool {
		return seeds[i].grow > seeds[j].grow
	})

	currentPlantDays := 0
	earliestFullBloom := 0

	for _, seed := range seeds {
		currentPlantDays += seed.plant

		bloomDay := currentPlantDays + seed.grow

		if bloomDay > earliestFullBloom {
			earliestFullBloom = bloomDay
		}
	}

	return earliestFullBloom
}
```

The Go implementation follows the exact same greedy strategy as the Python version.

A custom `Seed` struct stores planting and growing times together. The `sort.Slice` function sorts the seeds in descending order of growth time.

Go slices are dynamic views over arrays, so no special handling for arrays versus lists is needed.

Integer overflow is not an issue because the maximum total planting time is:

```go
100000 × 10000 = 1,000,000,000
```

which fits safely inside Go's 64 bit integer representation, and also within standard signed 32 bit range.

## Edge Cases

### Single Seed

When there is only one seed, the answer is simply:

```
plantTime[0] + growTime[0]
```

This case is easy to mishandle if the implementation assumes multiple iterations or relies on comparisons between seeds. The current implementation handles it naturally because the loop executes once and computes the bloom day directly.

### Multiple Seeds with Equal Grow Times

If several seeds have the same growth duration, their relative order does not affect the final answer.

A buggy implementation might attempt unnecessary tie-breaking logic or accidentally rely on unstable ordering. The greedy strategy does not require any special handling here because identical growth times are interchangeable.

### Very Large Grow Time Mixed with Small Ones

Consider:

```
plantTime = [5,1]
growTime = [1,10]
```

If the seed with grow time `10` is delayed, the total completion time becomes much larger.

This is exactly the scenario the greedy strategy is designed for. Sorting by descending `growTime` ensures the long growth period starts as early as possible.

### Large Input Size

With `100000` seeds, inefficient approaches quickly fail.

Any quadratic or factorial algorithm would time out. The implemented solution performs only:

- One sort
- One linear scan

which fits comfortably within the problem constraints.
