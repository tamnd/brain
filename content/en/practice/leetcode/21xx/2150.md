## Solution

## Problem Understanding

The problem asks us to identify all "lonely" numbers in an integer array. A number is considered lonely if it satisfies two conditions simultaneously:

1. It appears exactly once in the array.
2. Neither its immediate neighboring values, meaning `x - 1` nor `x + 1`, appear anywhere in the array.

The input is an array `nums` containing integers. The output should be a list containing every number that satisfies the loneliness condition. The order of the returned numbers does not matter.

For example, if the array contains `10`, but neither `9` nor `11` exist in the array, and `10` appears only once, then `10` is lonely.

The constraints are important because they guide the algorithm design:

- The array length can be as large as `10^5`
- Values can be as large as `10^6`

These constraints immediately suggest that an inefficient nested loop solution could become too slow. A quadratic time solution would perform up to `10^10` operations in the worst case, which is far beyond acceptable limits for LeetCode.

The problem naturally suggests using a counting structure such as a hash map because we repeatedly need to answer questions like:

- How many times does this number appear?
- Does `x - 1` exist?
- Does `x + 1` exist?

These are exactly the kinds of operations hash tables handle efficiently.

Several edge cases are worth considering upfront:

- Arrays with only one element. That single value is automatically lonely because no adjacent numbers exist.
- Arrays with duplicates. Even if neighboring numbers do not exist, duplicates disqualify the number.
- Arrays containing consecutive values. Numbers next to each other invalidate each other.
- Arrays containing zero. Since values are non-negative, checking `x - 1` for `0` becomes `-1`, which simply does not exist in the map and is safe to check.

## Approaches

### Brute Force Approach

The brute force method checks every number against all other numbers in the array.

For each value `nums[i]`, we would:

1. Count how many times it appears.
2. Check whether `nums[i] - 1` exists.
3. Check whether `nums[i] + 1` exists.

A naive implementation would use nested loops for these checks. For every element, we scan the entire array to count occurrences and search for adjacent values.

This approach is correct because it explicitly verifies the exact conditions from the problem statement. However, it is inefficient. With `n` elements, each element may require scanning the full array, resulting in `O(n^2)` time complexity.

Given that `n` can be `10^5`, this approach is too slow.

### Optimal Approach

The key observation is that we repeatedly need fast membership and frequency checks.

A hash map allows us to:

- Count frequencies in `O(1)` average time
- Check whether neighboring values exist in `O(1)` average time

The algorithm works in two passes:

1. Build a frequency map for all numbers.
2. Iterate through the array again and identify numbers that:

- Have frequency exactly `1`
- Do not have `x - 1`
- Do not have `x + 1`

This reduces the total runtime to linear time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Repeatedly scans the array for every element |
| Optimal | O(n) | O(n) | Uses a hash map for frequency and existence checks |

## Algorithm Walkthrough

1. Create a hash map called `frequency`.

This map stores how many times each number appears in the array. The key is the number itself, and the value is its count.
2. Traverse the array once to populate the frequency map.

For every number `num`:

- If it has not been seen before, initialize its count.
- Increment its frequency.

After this pass, we can instantly determine whether a number appears once or multiple times.
3. Create an empty result list.

This list will store all lonely numbers.
4. Traverse the array again.

For each number `num`, check three conditions:

- `frequency[num] == 1`
- `num - 1` is not in the frequency map
- `num + 1` is not in the frequency map

If all conditions are true, append `num` to the result list.
5. Return the result list.

### Why it works

The frequency map guarantees accurate occurrence counts for every value. Since the map also provides constant time existence checks, we can efficiently determine whether adjacent values exist.

A number is added to the answer only if:

- It appears exactly once
- Neither neighboring value exists

These conditions exactly match the problem definition, so every returned number is lonely, and every lonely number is returned.

## Python Solution

```python
from typing import List
from collections import Counter

class Solution:
    def findLonely(self, nums: List[int]) -> List[int]:
        frequency = Counter(nums)

        lonely_numbers = []

        for num in nums:
            if (
                frequency[num] == 1
                and (num - 1) not in frequency
                and (num + 1) not in frequency
            ):
                lonely_numbers.append(num)

        return lonely_numbers
```

The implementation starts by importing `Counter` from Python's `collections` module. `Counter` automatically builds a frequency map from the input array.

The variable `frequency` stores how many times each number appears. For example, if the input is `[1, 3, 5, 3]`, the map becomes:

```python
{
    1: 1,
    3: 2,
    5: 1
}
```

Next, we create an empty list called `lonely_numbers` to store valid answers.

We then iterate through every number in `nums`. For each value, we verify the three loneliness conditions:

- The number appears once.
- The previous number does not exist.
- The next number does not exist.

If all checks succeed, the number is appended to the result list.

Finally, the result list is returned.

## Go Solution

```go
func findLonely(nums []int) []int {
	frequency := make(map[int]int)

	for _, num := range nums {
		frequency[num]++
	}

	result := []int{}

	for _, num := range nums {
		_, hasPrev := frequency[num-1]
		_, hasNext := frequency[num+1]

		if frequency[num] == 1 && !hasPrev && !hasNext {
			result = append(result, num)
		}
	}

	return result
}
```

The Go implementation follows the same logic as the Python version but uses a `map[int]int` instead of `Counter`.

Go maps return two values during lookup:

- The value
- A boolean indicating whether the key exists

This is why the code uses:

```go
_, hasPrev := frequency[num-1]
```

The underscore ignores the actual stored value because only existence matters.

Slices in Go automatically grow when using `append`, making them suitable for collecting results dynamically.

There are no integer overflow concerns because the constraints stay well within Go's integer range.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | One pass to count frequencies and one pass to collect answers |
| Space | O(n) | The hash map may store every unique number |

The algorithm performs two linear scans of the array. Hash map operations such as insertion and lookup take constant time on average, so the total runtime remains linear.

The extra space comes from storing frequencies for up to `n` unique numbers.

## Edge Cases

One important edge case is an array with a single element. Since there are no other numbers in the array, the lone value automatically satisfies both loneliness conditions. A careless implementation might accidentally fail when checking neighboring values, but the hash map lookup safely handles missing keys.

Another important case involves duplicates. Consider `[5, 5, 5]`. Even though neither `4` nor `6` exists, `5` is not lonely because it appears more than once. The frequency check `frequency[num] == 1` ensures duplicates are excluded immediately.

A third edge case is consecutive values such as `[1, 2, 3]`. Every number has at least one adjacent neighbor, so none should be returned. The implementation correctly checks both `num - 1` and `num + 1`, preventing any consecutive number from being incorrectly classified as lonely.

A subtle edge case involves zero. Since the constraints allow `0`, checking `num - 1` results in `-1`. Some implementations might incorrectly assume all values are non-negative and mishandle this case. The hash map approach works naturally because looking up `-1` simply returns "not found".
