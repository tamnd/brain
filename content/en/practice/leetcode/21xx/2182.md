## Solution

## Problem Understanding

The problem gives us a string `s` consisting only of lowercase English letters and an integer `repeatLimit`. We must construct a new string using characters from `s` such that no character appears more than `repeatLimit` times consecutively.

Among all valid strings, we want the lexicographically largest one possible.

Lexicographical order works the same way dictionary order works, except larger letters are considered better here. For example, `"z"` is larger than `"y"`, and `"zz"` is larger than `"za"` because the first differing character is larger.

An important detail is that we are not required to use all characters from `s`. If some remaining characters would violate the repeat limit and there is no smaller character available to break the sequence, we are allowed to stop early.

The constraints are significant:

- `1 <= s.length <= 10^5`
- `s` only contains lowercase English letters
- Only 26 possible characters exist

The input size tells us that exponential or quadratic construction approaches are too slow. We need something close to linear time.

The fact that only lowercase English letters are used is extremely important. Since there are only 26 possible characters, we can efficiently maintain frequency counts and greedily select the largest valid character at every step.

Several edge cases are important:

- A character may appear far more times than `repeatLimit`
- The largest character may need interruption by a smaller character
- We may become unable to place remaining characters legally
- `repeatLimit` may be `1`, meaning no adjacent equal characters are allowed
- The answer may intentionally leave characters unused

For example, with `s = "aaaaa"` and `repeatLimit = 2`, the best valid answer is `"aa"`, because there is no smaller character available to separate additional `'a'` characters.

## Approaches

### Brute Force Approach

A brute force solution would attempt to generate all possible valid strings using the characters from `s`, then select the lexicographically largest valid one.

One possible implementation would use backtracking:

- Try appending every available character
- Ensure the repeat limit is not violated
- Recursively continue building the string
- Track the lexicographically largest valid result

This works because it explores every valid construction possibility.

However, this approach is completely impractical for the given constraints. The number of possible permutations grows factorially with the string size. Even with pruning, the search space becomes enormous for strings approaching length `10^5`.

### Key Insight

The key observation is that lexicographical maximization strongly favors using the largest possible character at every step.

Suppose we currently can place several characters legally. To maximize the resulting string, we should always choose the largest available character that does not violate the repeat constraint.

If the current largest character has already been used `repeatLimit` consecutive times, we temporarily insert the next-largest available character to break the streak. Then we can continue using the larger character again.

This naturally leads to a greedy strategy.

Because only 26 lowercase letters exist, we can maintain character frequencies in an array and repeatedly search from `'z'` down to `'a'`.

The greedy choice is correct because lexicographical order is determined from left to right. Making the current position as large as possible always dominates any future optimization.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | O(n) recursion | Tries all valid constructions |
| Optimal Greedy Counting | O(n) | O(1) | Always places the largest valid character |

## Algorithm Walkthrough

### Optimal Greedy Algorithm

1. Count the frequency of every character in the string.

Since there are only 26 lowercase English letters, we can store counts in an array of size 26. This gives constant-space frequency tracking.
2. Maintain a result array that will store the constructed string.

Using a list of characters is more efficient than repeated string concatenation because strings are immutable in Python.
3. Start from the largest character, `'z'`, and work downward.

At every step, we try to place the lexicographically largest possible character because earlier positions dominate lexicographical order.
4. For the current character, append it up to `repeatLimit` times.

If the character appears fewer than `repeatLimit` times, append all remaining occurrences.

This greedily maximizes the current prefix while staying valid.
5. After using the current character `repeatLimit` times, check whether more copies remain.

If no copies remain, move to the next smaller character.

If copies do remain, we cannot place it again immediately because that would violate the repeat limit.
6. Find the next smaller character with a positive remaining frequency.

This character acts as a separator or breaker.

We append exactly one copy of this smaller character.
7. After inserting the breaker character, continue using the larger character again.

The separator resets the consecutive repetition count, allowing the larger character to appear again legally.
8. If no smaller character exists to break the sequence, stop construction.

At this point, any remaining characters would violate the repeat limit, so the current result is optimal.

### Why it works

The algorithm maintains the invariant that the constructed prefix is always the lexicographically largest valid prefix possible.

At every position, choosing a smaller character while a larger legal character exists would immediately produce a lexicographically smaller result. Therefore, the greedy decision is always optimal.

When the repeat limit blocks the current largest character, using the next-largest available separator minimizes the lexicographical penalty while allowing future use of the larger character again.

## Python Solution

```python
class Solution:
    def repeatLimitedString(self, s: str, repeatLimit: int) -> str:
        counts = [0] * 26
        
        for ch in s:
            counts[ord(ch) - ord('a')] += 1
        
        result = []
        
        current = 25
        
        while current >= 0:
            if counts[current] == 0:
                current -= 1
                continue
            
            use_count = min(counts[current], repeatLimit)
            
            result.append(chr(current + ord('a')) * use_count)
            counts[current] -= use_count
            
            if counts[current] == 0:
                continue
            
            smaller = current - 1
            
            while smaller >= 0 and counts[smaller] == 0:
                smaller -= 1
            
            if smaller < 0:
                break
            
            result.append(chr(smaller + ord('a')))
            counts[smaller] -= 1
        
        return "".join(result)
```

The implementation begins by counting the frequency of each lowercase letter using a fixed-size array.

The variable `current` tracks the largest character currently being processed. Since `'z'` corresponds to index `25`, we begin there and move downward.

For each character, we greedily append as many copies as possible without exceeding `repeatLimit`. This maximizes the lexicographical value of the current prefix.

If additional copies of the same character remain, we search for the next smaller available character. This smaller character acts as a separator that resets the repetition constraint.

If no smaller character exists, construction stops because any further additions would violate the rules.

Finally, we join the accumulated character pieces into the resulting string.

## Go Solution

```go
func repeatLimitedString(s string, repeatLimit int) string {
    counts := make([]int, 26)

    for _, ch := range s {
        counts[ch-'a']++
    }

    result := make([]byte, 0)

    current := 25

    for current >= 0 {
        if counts[current] == 0 {
            current--
            continue
        }

        useCount := counts[current]
        if useCount > repeatLimit {
            useCount = repeatLimit
        }

        for i := 0; i < useCount; i++ {
            result = append(result, byte(current)+'a')
        }

        counts[current] -= useCount

        if counts[current] == 0 {
            continue
        }

        smaller := current - 1

        for smaller >= 0 && counts[smaller] == 0 {
            smaller--
        }

        if smaller < 0 {
            break
        }

        result = append(result, byte(smaller)+'a')
        counts[smaller]--
    }

    return string(result)
}
```

The Go implementation follows the same greedy logic as the Python version.

Instead of Python lists and strings, Go uses a `[]byte` slice for efficient mutable string construction.

Character indexing is handled using byte arithmetic. Since the input contains only lowercase English letters, converting between character indices and ASCII values is straightforward.

No integer overflow concerns exist because counts never exceed `10^5`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each character is processed a constant number of times |
| Space | O(1) | Frequency array size is fixed at 26 |

The algorithm processes each character at most once while building the answer. Although we occasionally scan downward to find a smaller separator character, the alphabet size is fixed at 26, making this effectively constant work.

The frequency array always uses exactly 26 integers regardless of input size, so auxiliary space is constant.

## Edge Cases

One important edge case occurs when the string contains only one unique character. For example, `"aaaaaa"` with `repeatLimit = 2`. A naive implementation might attempt to place all characters and violate the constraint. The correct behavior is to stop once the limit is reached because no separator character exists. The implementation handles this by searching for a smaller character and terminating if none is found.

Another tricky case happens when `repeatLimit = 1`. In this situation, identical adjacent characters are completely forbidden. The algorithm still works naturally because it only appends one copy of a character before requiring a separator. This effectively turns the problem into alternating between the largest available characters whenever possible.

A third important edge case involves a highly dominant largest character, such as `"zzzzzzza"` with `repeatLimit = 2`. We can only use `"zzazz"` before becoming stuck. A naive greedy solution might incorrectly continue appending `'z'` after exhausting separators. The implementation correctly stops construction once no smaller character remains.

A final subtle case occurs when multiple smaller separators exist. For example, `"zzzzccbb"` with `repeatLimit = 2`. The optimal strategy is always to use the largest possible separator, not the smallest one. Using a larger separator preserves lexicographical optimality. The algorithm guarantees this by always searching downward from the current character and choosing the first available smaller character.
