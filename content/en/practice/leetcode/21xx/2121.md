## Solution

## Problem Understanding

The problem asks us to compute, for every index in the array, the total distance between that index and every other index containing the same value.

More concretely, suppose we are at index `i`. We look through the array and find every position `j` where `arr[j] == arr[i]`. For each matching position, we calculate the interval `|i - j|`, which is simply the absolute difference between their indices. The final answer for index `i` is the sum of all such intervals.

For example, if `arr[i] = 3` and the value `3` appears at indices `[2, 5, 6]`, then:

- At index `2`, the sum is `|2 - 5| + |2 - 6| = 3 + 4 = 7`
- At index `5`, the sum is `|5 - 2| + |5 - 6| = 3 + 1 = 4`
- At index `6`, the sum is `|6 - 2| + |6 - 5| = 4 + 1 = 5`

The input is a `0-indexed` integer array `arr`, where `n` can be as large as `10^5`. The output must be an array of the same length where each position stores the sum of distances to identical values.

The constraints are important here:

- `n <= 10^5`
- `arr[i] <= 10^5`

Since the array can contain up to one hundred thousand elements, a naive solution that compares every pair of equal elements would likely be too slow. Any approach worse than roughly `O(n log n)` or `O(n)` risks timing out.

Several edge cases immediately stand out:

- An element appears only once, meaning its answer must be `0`.
- Every element is identical, producing many pairwise distances.
- The array contains no repeated values, meaning the entire result is zeros.
- Repeated elements are spread far apart, making distance calculations large.
- Large inputs near `10^5`, where efficiency becomes critical.

The problem guarantees valid input sizes and values, so we do not need to worry about empty arrays or invalid integers.

## Approaches

### Brute Force Approach

The most straightforward solution is to compute the answer independently for every index.

For each position `i`, we scan the entire array and check every other position `j`. Whenever `arr[j] == arr[i]`, we add `|i - j|` to the total.

This approach is correct because it explicitly follows the definition of the problem. Every valid matching index is considered, and every distance is summed exactly once.

However, this method is far too slow for the given constraints. Since we iterate through all `n` indices for every element, the total complexity becomes `O(n²)`. With `n = 10^5`, this would require roughly `10^10` operations, which is computationally infeasible.

### Optimal Approach, Grouping Indices with Prefix Sums

The key observation is that identical values can be processed together.

Instead of repeatedly scanning the entire array, we first group indices by value using a hash map.

For example:

```
arr = [2,1,3,1,2,3,3]

Groups:
2 -> [0,4]
1 -> [1,3]
3 -> [2,5,6]
```

Now consider a group of indices:

```
positions = [2,5,6]
```

For index `5`, we want:

```
|5-2| + |5-6|
```

Because the indices are already sorted naturally as we traverse the array, we can exploit arithmetic relationships.

Suppose we process a position `positions[i]`.

The contribution from elements to the left is:

```
(current_index * left_count) - left_sum
```

The contribution from elements to the right is:

```
right_sum - (current_index * right_count)
```

By maintaining prefix sums of indices, we can compute each answer in constant time instead of recomputing distances repeatedly.

This transforms the solution from quadratic time into linear time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Compare every index with every other index |
| Optimal | O(n) | O(n) | Group indices and compute distances using prefix sums |

## Algorithm Walkthrough

1. Create a hash map where each value maps to a list of indices where it appears.

As we iterate through `arr`, we append each index to the list corresponding to its value. This groups all identical elements together.

Example:

```
arr = [2,1,3,1,2,3,3]

groups = {
    2: [0,4],
    1: [1,3],
    3: [2,5,6]
}
```
2. Initialize the result array.

Create an output array of size `n` filled with zeros.

```
result = [0] * n
```
3. Process each group independently.

For each value's list of indices:

```
positions = [2,5,6]
```

Compute the total sum of all indices:

```
total_sum = 2 + 5 + 6 = 13
```

Also maintain a running prefix sum:

```
prefix_sum = 0
```
4. Compute distances for each position.

For each index `i` in `positions`:

Let:

```
current = positions[i]
```

Compute left contribution:

```
left_distance =
current * i - prefix_sum
```

This works because there are `i` indices to the left, and their total sum is `prefix_sum`.
5. Compute right contribution.

Remove the current index from the remaining total:

```
remaining_sum =
total_sum - prefix_sum - current
```

Then compute:

```
right_distance =
remaining_sum -
current * (remaining_count)
```
6. Store the answer.

The final result for this index is:

```
result[current] =
left_distance + right_distance
```
7. Update prefix sum.

Add the current index so future positions can treat it as part of the left side.

```
prefix_sum += current
```
8. Repeat for every group until all indices are processed.

### Why it works

The algorithm works because every identical value is processed independently, and the indices for each value are naturally sorted. For any position, the total distance can be decomposed into contributions from indices to the left and indices to the right.

Using prefix sums ensures we can calculate these contributions mathematically rather than iterating through all matching indices repeatedly. Every valid interval is counted exactly once, and no intervals are omitted, guaranteeing correctness.

## Python Solution

```python
from collections import defaultdict
from typing import List

class Solution:
    def getDistances(self, arr: List[int]) -> List[int]:
        index_groups = defaultdict(list)

        # Group indices by value
        for index, value in enumerate(arr):
            index_groups[value].append(index)

        result = [0] * len(arr)

        # Process each group independently
        for positions in index_groups.values():
            total_sum = sum(positions)
            prefix_sum = 0
            group_size = len(positions)

            for i, current_index in enumerate(positions):
                left_distance = current_index * i - prefix_sum

                remaining_sum = (
                    total_sum
                    - prefix_sum
                    - current_index
                )

                right_count = group_size - i - 1
                right_distance = (
                    remaining_sum
                    - current_index * right_count
                )

                result[current_index] = (
                    left_distance
                    + right_distance
                )

                prefix_sum += current_index

        return result
```

The implementation begins by grouping indices using a `defaultdict(list)`. This avoids repeated scanning of the array because every identical value is collected into a single list.

Next, we initialize the result array with zeros. This automatically handles values that appear only once because no distance calculations will contribute anything.

For each group of indices, we compute `total_sum`, which stores the sum of all indices in the group. During iteration, `prefix_sum` tracks the sum of indices already processed.

At each position, we split the distance calculation into two parts. The left side computes distances to all previously seen matching indices, while the right side computes distances to future matching indices. Because all required sums are maintained incrementally, each calculation takes constant time.

Finally, the computed distance is stored in the result array at the original index.

## Go Solution

```go
func getDistances(arr []int) []int64 {
	indexGroups := make(map[int][]int)

	// Group indices by value
	for index, value := range arr {
		indexGroups[value] = append(indexGroups[value], index)
	}

	result := make([]int64, len(arr))

	// Process each group
	for _, positions := range indexGroups {
		var totalSum int64 = 0

		for _, position := range positions {
			totalSum += int64(position)
		}

		var prefixSum int64 = 0
		groupSize := len(positions)

		for i, currentIndex := range positions {
			current := int64(currentIndex)

			leftDistance :=
				current*int64(i) - prefixSum

			remainingSum :=
				totalSum -
					prefixSum -
					current

			rightCount :=
				int64(groupSize - i - 1)

			rightDistance :=
				remainingSum -
					current*rightCount

			result[currentIndex] =
				leftDistance +
					rightDistance

			prefixSum += current
		}
	}

	return result
}
```

The Go implementation follows the same algorithmic structure as the Python version. A `map[int][]int` stores grouped indices, and each group is processed independently.

One important difference is integer handling. Since the sum of distances can become large, the function returns `[]int64` rather than `[]int`. Intermediate calculations such as `totalSum`, `prefixSum`, and distance computations also use `int64` to avoid overflow.

Go slices are dynamically resized using `append`, making them a natural replacement for Python lists.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each index is added to a group once and processed once |
| Space | O(n) | Hash map stores all indices grouped by value |

The time complexity is linear because every array index participates in exactly two phases: grouping and computation. No nested scans across the entire array occur.

The space complexity is also linear because the hash map stores every index exactly once across all groups, and the output array also requires `O(n)` memory.

## Edge Cases

### Elements That Appear Only Once

A common source of bugs is forgetting that some values may appear exactly once. In such cases, there are no matching indices, so the total interval sum must be `0`.

For example:

```
arr = [1,2,3]
```

Every value appears once, so the output is:

```
[0,0,0]
```

The implementation handles this naturally because a group of size one has no left or right contribution.

### All Elements Are Identical

When every element is the same, every index contributes distances to every other index.

Example:

```
arr = [5,5,5,5]
```

A naive implementation may repeatedly recompute the same distances, leading to quadratic performance. Our prefix sum method efficiently computes all answers in linear time while avoiding duplicate work.

### Large Sparse Repetitions

Repeated values may be spread very far apart.

Example:

```
arr = [1,0,0,0,0,1]
```

The first and last positions are separated by a large distance:

```
|0 - 5| = 5
```

It is easy to make off by one mistakes when computing left and right contributions. The implementation avoids this by explicitly tracking how many elements remain on each side and using exact arithmetic formulas.

### Very Large Inputs

Since `n` can be `10^5`, quadratic solutions will time out.

The implementation processes each index only a constant number of times, ensuring the algorithm remains efficient even at the maximum input size.
