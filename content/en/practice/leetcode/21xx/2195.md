## Solution

## Problem Understanding

The problem gives us an integer array `nums` and an integer `k`. We must append exactly `k` positive integers to the array such that:

- Every appended integer is unique
- None of the appended integers already exist in `nums`
- The sum of the appended integers is as small as possible

The return value is not the final sum of the array. Instead, we only return the sum of the newly appended integers.

In simpler terms, we want to choose the smallest possible positive integers that are missing from `nums`. Since smaller numbers contribute less to the total sum, the optimal strategy is always to greedily take the smallest available positive integers.

The constraints are important:

- `nums.length` can be up to `10^5`
- `nums[i]` can be as large as `10^9`
- `k` can be as large as `10^8`

These constraints immediately tell us that we cannot simulate every number one by one up to very large ranges. A naive solution that repeatedly checks integers individually would become too slow.

Another important observation is that `nums` may contain duplicates. Duplicates do not matter for exclusion purposes because we only care whether a number exists at least once. This means deduplication is useful.

Several edge cases are especially important:

- `nums` may already contain many small positive integers, forcing us to skip them
- `nums` may contain duplicates, which should not incorrectly reduce the number of values we can append
- `k` may still remain large after processing all values in `nums`, requiring us to append numbers larger than every existing value
- Large values such as `10^9` require arithmetic formulas instead of iterative summation to avoid timeouts

## Approaches

### Brute Force Approach

The brute force solution would repeatedly try positive integers starting from `1`.

For each integer:

- Check whether it exists in `nums`
- If it does not exist, add it to the answer
- Continue until we have chosen `k` integers

A hash set can make membership checks efficient. However, the main problem is that we may need to scan through a huge range of integers before collecting all `k` missing values.

For example, if `nums` contains every number from `1` to `100000`, we must skip all of them before finding valid candidates. Since `k` itself can also be very large, iterating one number at a time becomes inefficient.

This approach is correct because it always picks the smallest available missing integers, but it is not scalable enough for the constraints.

### Optimal Greedy + Sorting Approach

The key insight is that missing numbers appear in contiguous ranges.

Suppose after sorting and deduplicating we have:

```
[1, 4, 10]
```

Then the missing ranges are:

```
2 to 3
5 to 9
11 onward
```

Instead of processing numbers individually, we can process entire missing ranges at once.

If a missing range contains enough numbers to satisfy all remaining `k`, we can directly compute the sum using the arithmetic progression formula.

The sum of integers from `a` to `b` is:

$\sum_{x=a}^{b} x = \frac{(a+b)(b-a+1)}{2}$

This allows us to compute large sums in constant time.

The algorithm therefore becomes:

- Sort and deduplicate `nums`
- Track the smallest candidate integer we can still use
- For each number in sorted order:

- Identify the missing gap before it
- Take as many numbers from that gap as possible
- Compute their contribution with arithmetic formulas
- If `k` still remains afterward, append numbers larger than the maximum element

## Approaches Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(maxValue + k) | O(n) | Iterates through integers one by one |
| Optimal | O(n log n) | O(n) | Uses sorting and range summation |

## Algorithm Walkthrough

1. Convert `nums` into a sorted list of unique values.

Duplicates do not matter because a number is either forbidden or not. Using unique values simplifies the logic and avoids processing the same number multiple times.
2. Initialize a variable `current` to `1`.

This represents the smallest positive integer we are currently allowed to consider.
3. Initialize `answer` to `0`.

This variable accumulates the sum of appended integers.
4. Iterate through each number `num` in the sorted unique array.

For every value, check whether there is a gap between `current` and `num - 1`.
5. If `current < num`, then there exists a missing range.

The missing numbers are:

```
current, current + 1, ..., num - 1
```

The size of this range is:

```
num - current
```
6. Determine how many numbers we should take from this range.

We cannot take more than:

- The number of available missing integers
- The remaining value of `k`

So:

```
count = min(k, num - current)
```
7. Compute the sum of the chosen numbers using arithmetic progression formulas.

If we take `count` numbers starting from `current`, then the last chosen number is:

```
end = current + count - 1
```

The sum is:

$\frac{(current + end) \cdot count}{2}$
8. Add this contribution to `answer`.
9. Reduce `k` by `count`.

This tracks how many more integers we still need to append.
10. Move `current` to `num + 1`.

Since `num` exists in the array, we cannot use it.
11. After processing all numbers, `k` may still be greater than zero.

In that case, all remaining valid integers begin from `current`.
12. Append the next `k` integers starting from `current`.

Again use arithmetic progression formulas to compute the sum efficiently.

### Why it works

The algorithm always chooses the smallest possible unused positive integers. Because smaller integers always produce a smaller total sum than larger integers, any optimal solution must greedily consume missing numbers in ascending order.

Sorting allows us to identify missing ranges efficiently, and arithmetic progression formulas let us sum entire ranges in constant time. Therefore, the algorithm computes exactly the minimal possible sum.

## Python Solution

```python
from typing import List

class Solution:
    def minimalKSum(self, nums: List[int], k: int) -> int:
        unique_nums = sorted(set(nums))

        answer = 0
        current = 1

        for num in unique_nums:
            if current < num:
                count = min(k, num - current)
                end = current + count - 1

                answer += (current + end) * count // 2
                k -= count

                if k == 0:
                    return answer

            current = num + 1

        end = current + k - 1
        answer += (current + end) * k // 2

        return answer
```

The implementation begins by converting `nums` into a sorted set. This removes duplicates and guarantees ascending order, which is necessary for detecting missing ranges.

The variable `current` tracks the smallest candidate integer we can still use. Initially, it is `1` because we want the smallest positive integers.

Inside the loop, whenever `current < num`, there exists a gap of missing integers before `num`. We calculate how many values can be taken from that gap using `min(k, num - current)`.

Instead of iterating through the numbers individually, we compute their total directly using the arithmetic progression formula. This is critical for efficiency because `k` can be as large as `10^8`.

After consuming numbers from a gap, we update `k`. If `k` becomes zero, we immediately return the answer because we already selected enough integers.

Finally, if some values are still needed after processing all elements in `nums`, we append consecutive integers starting from `current`.

## Go Solution

```go
package main

import (
	"sort"
)

func minimalKSum(nums []int, k int) int64 {
	sort.Ints(nums)

	uniqueNums := make([]int, 0)

	for _, num := range nums {
		if len(uniqueNums) == 0 || uniqueNums[len(uniqueNums)-1] != num {
			uniqueNums = append(uniqueNums, num)
		}
	}

	var answer int64 = 0
	current := 1

	for _, num := range uniqueNums {
		if current < num {
			count := min(k, num-current)
			end := current + count - 1

			answer += int64(current+end) * int64(count) / 2
			k -= count

			if k == 0 {
				return answer
			}
		}

		current = num + 1
	}

	end := current + k - 1
	answer += int64(current+end) * int64(k) / 2

	return answer
}

func min(a, b int) int {
	if a < b {
		return a
	}
	return b
}
```

The Go implementation follows the same logic as the Python solution, but there are several language-specific considerations.

Go does not have a built-in set type, so duplicates are removed manually after sorting. We construct a `uniqueNums` slice by only appending values different from the previous one.

Another important difference is integer overflow handling. Since sums can become very large, the return type must be `int64`. Intermediate arithmetic is also explicitly cast to `int64` before multiplication.

Slices in Go are dynamic views over arrays, so `uniqueNums` is built efficiently using `append`.

## Edge Cases

One important edge case occurs when `nums` contains duplicates. A naive implementation might process duplicate values repeatedly and incorrectly skip ranges multiple times. For example, with `nums = [1,1,1]`, we should still only consider `1` as forbidden once. The implementation handles this by converting the array into a sorted set in Python, and by manually deduplicating after sorting in Go.

Another critical edge case happens when the smallest positive integers are already occupied. Consider `nums = [1,2,3,4]`. In this scenario, the optimal appended numbers begin only after the existing prefix. The algorithm correctly advances `current` past each occupied number until it reaches the first missing value.

A third important edge case is extremely large values in `nums`, such as `10^9`. Iterating one number at a time would be impossible at this scale. The implementation avoids iteration over large ranges entirely by computing sums mathematically using arithmetic progression formulas.

A final edge case occurs when `k` remains positive after all elements in `nums` have been processed. For example, with `nums = [5,6]` and `k = 6`, we still need additional numbers after consuming the initial missing range. The implementation handles this by appending consecutive integers starting from `current` after the loop completes.
