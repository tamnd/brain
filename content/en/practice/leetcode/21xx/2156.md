## Solution

## Problem Understanding

We are given a string `s` and a polynomial rolling hash function. For every substring of length `k`, the hash is computed as:

$\text{hash}(s)=\left(\sum_{i=0}^{k-1} \text{val}(s[i]) \cdot p^i\right) \bmod m$

Here:

- `val('a') = 1`
- `val('b') = 2`
- ...
- `val('z') = 26`

The problem asks us to find the first substring of length `k` whose hash equals `hashValue`.

The important detail is that the hash uses increasing powers from left to right. For a substring `"abc"` with `power = p`, the hash is:

$1\cdot p^0 + 2\cdot p^1 + 3\cdot p^2$

The string length can be as large as `2 * 10^4`, and both `power` and `modulo` can be very large, up to `10^9`. This immediately tells us that recomputing the hash for every substring from scratch would be too expensive.

The problem also guarantees that at least one valid answer exists. If multiple substrings produce the same target hash, we must return the one that appears earliest in the string.

A subtle but extremely important observation is that the polynomial hash is directional. Since powers increase from left to right, removing the leftmost character from a rolling hash is difficult in the usual forward traversal. However, if we traverse the string from right to left, the rolling hash becomes much easier to maintain.

Several edge cases matter:

- When `k == 1`, every substring is a single character, so the rolling logic must still work correctly.
- When the entire string length equals `k`, the answer is simply the whole string.
- Multiple substrings may share the same hash value, and we must return the earliest one.
- Large values of `power` and `modulo` require careful modular arithmetic to avoid overflow issues in languages like Go.

## Approaches

### Brute Force Approach

The most direct solution is to generate every substring of length `k`, compute its hash from scratch, and compare it against `hashValue`.

For each substring:

1. Convert each character into its alphabet value.
2. Compute the polynomial hash using repeated powers of `power`.
3. Take the result modulo `modulo`.
4. Return the first substring whose hash matches `hashValue`.

This works because it exactly follows the definition of the problem.

However, computing the hash for one substring takes `O(k)` time, and there are `O(n)` substrings. Therefore, the total complexity becomes `O(nk)`.

With `n` up to `2 * 10^4`, this can become far too slow.

### Optimal Approach, Reverse Rolling Hash

The key insight is that this is a classic rolling hash problem.

Normally, rolling hashes are easiest when the lowest power corresponds to the newest character entering the window. Unfortunately, this hash definition assigns increasing powers from left to right, which makes a standard left-to-right rolling update awkward.

The trick is to process the string in reverse.

Suppose we process substrings from right to left. Then:

- The character entering the window naturally receives power `p^0`
- Existing characters shift one power higher
- We can efficiently remove the outgoing character using a precomputed `p^(k-1)`

This transforms the hash maintenance into a clean sliding window update.

Instead of recomputing every substring hash independently, we update the previous hash in constant time.

## Algorithm Walkthrough

### Optimal Reverse Rolling Hash Algorithm

1. Define a helper function that converts a character into its alphabet index.

For example:

- `'a' -> 1`
- `'b' -> 2`
- `'z' -> 26`
2. Precompute:

$p^{k-1} \bmod modulo$

This value is needed when removing the outgoing character from the rolling window.
3. Traverse the string from right to left.

We maintain:

- `current_hash`, the hash of the current window
- a sliding window of size at most `k`
4. For each character entering the window:

Multiply the current hash by `power`, then add the new character value.

Conceptually:

$newHash=(oldHash\cdot power+val(c))\bmod modulo$

This shifts all previous characters one power higher.
5. If the window size exceeds `k`, remove the outgoing character contribution.

The outgoing character contributed:

$val(outgoing)\cdot p^{k-1}$

So we subtract it modulo `modulo`.
6. Whenever the window size becomes exactly `k`, compare the current hash with `hashValue`.

If they match, store the current starting index.
7. Continue scanning all the way to the beginning.

Since we move from right to left, later matches correspond to earlier substrings in the original string. Therefore, every valid match should overwrite the previous answer.
8. Return the substring starting at the stored index with length `k`.

### Why it works

The rolling hash invariant is that `current_hash` always equals the hash of the current length-`k` substring under the exact polynomial definition required by the problem.

Traversing from right to left aligns perfectly with the increasing powers in the hash formula. Every step shifts existing powers upward by multiplying with `power`, while the newly added character naturally receives exponent `0`.

Because every substring is examined exactly once and the rolling updates preserve the correct polynomial structure, the algorithm always finds the correct earliest substring.

## Python Solution

```python
class Solution:
    def subStrHash(
        self,
        s: str,
        power: int,
        modulo: int,
        k: int,
        hashValue: int
    ) -> str:
        
        n = len(s)

        def char_value(ch: str) -> int:
            return ord(ch) - ord('a') + 1

        power_k = pow(power, k - 1, modulo)

        current_hash = 0
        answer_index = 0

        for i in range(n - 1, -1, -1):
            current_hash = (
                current_hash * power + char_value(s[i])
            ) % modulo

            # Remove outgoing character if window exceeds size k
            if i + k < n:
                outgoing = char_value(s[i + k])
                current_hash = (
                    current_hash
                    - outgoing * power_k
                ) % modulo

            # Check window of exact size k
            if n - i >= k and current_hash == hashValue:
                answer_index = i

        return s[answer_index:answer_index + k]
```

The implementation closely follows the rolling hash algorithm.

The helper function converts characters into values from `1` to `26`.

`power_k` stores:

$power^{k-1} \bmod modulo$

which is used when removing the outgoing character contribution.

The loop iterates from right to left. At every step:

- Existing powers are shifted upward by multiplying the hash by `power`
- The new character enters with exponent `0`
- If the window grows too large, the outgoing character is removed

The condition `n - i >= k` checks whether the current window has reached size `k`.

Whenever the hash matches `hashValue`, we update `answer_index`. Since we scan backward, the final stored index is the earliest valid substring.

## Go Solution

```go
func subStrHash(s string, power int, modulo int, k int, hashValue int) string {
	n := len(s)

	charValue := func(ch byte) int {
		return int(ch-'a') + 1
	}

	powerK := 1
	for i := 0; i < k-1; i++ {
		powerK = (powerK * power) % modulo
	}

	currentHash := 0
	answerIndex := 0

	for i := n - 1; i >= 0; i-- {
		currentHash = (currentHash*power + charValue(s[i])) % modulo

		if i+k < n {
			outgoing := charValue(s[i+k])

			currentHash = (
				currentHash -
					(outgoing*powerK)%modulo +
					modulo
			) % modulo
		}

		if n-i >= k && currentHash == hashValue {
			answerIndex = i
		}
	}

	return s[answerIndex : answerIndex+k]
}
```

The Go implementation mirrors the Python solution very closely.

One important Go-specific detail is modular subtraction. Since Go can produce negative values after subtraction, we add `modulo` before taking `% modulo` again.

Another difference is integer handling. Since the constraints fit comfortably within Go's standard integer range on LeetCode, regular `int` arithmetic is sufficient.

## Edge Cases

### Case 1, `k = 1`

When the substring length is `1`, every substring consists of a single character. This is easy to mishandle because rolling hash formulas often assume larger windows.

The implementation handles this naturally. The precomputed value becomes:

$power^0 = 1$

and the rolling updates still function correctly.

### Case 2, Multiple Valid Substrings

Several substrings may produce the same hash value. The problem specifically asks for the earliest occurrence.

This is why we traverse from right to left and overwrite `answer_index` whenever a match is found. Earlier substrings are encountered later during reverse traversal, ensuring the final stored answer is the correct one.

### Case 3, Modular Subtraction Becoming Negative

When removing the outgoing character contribution, subtraction can produce a negative number.

For example:

$3 - 10 = -7$

In modular arithmetic, this must wrap correctly. The implementation always applies modulo after subtraction, and the Go version explicitly adds `modulo` before `% modulo` to guarantee non-negative results.
