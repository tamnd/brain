# LeetCode 2164 - Sort Even and Odd Indices Independently

## Problem Understanding

The problem gives us a 0-indexed integer array `nums` and asks us to rearrange its values according to two independent sorting rules.

The values located at odd indices, meaning positions `1, 3, 5, ...`, must be sorted in non-increasing order, which means descending order. At the same time, the values located at even indices, meaning positions `0, 2, 4, ...`, must be sorted in non-decreasing order, which means ascending order.

An important detail is that the positions themselves do not change. Only the values assigned to those positions are rearranged. Values originally at even indices will remain at even indices after sorting, and values originally at odd indices will remain at odd indices after sorting.

For example, if the array is:

```
[4,1,2,3]
```

The even-indexed values are:

```
indices 0 and 2 -> [4,2]
```

Sorting them in ascending order gives:

```
[2,4]
```

The odd-indexed values are:

```
indices 1 and 3 -> [1,3]
```

Sorting them in descending order gives:

```
[3,1]
```

Rebuilding the array using those sorted values produces:

```
[2,3,4,1]
```

The constraints are very small:

- `1 <= nums.length <= 100`
- `1 <= nums[i] <= 100`

Since the array length is at most 100, even solutions involving sorting are easily fast enough. We do not need advanced optimization techniques or specialized data structures.

Several edge cases are important to recognize early:

- Arrays of length 1 contain only one even index and no odd indices.
- Arrays of length 2 contain exactly one even index and one odd index, so no visible change occurs after sorting.
- Arrays with repeated values must still preserve the sorting rules correctly.
- Arrays where the even or odd subsequence is already sorted should remain unchanged.
- Arrays with alternating high and low values can expose indexing mistakes during reconstruction.

## Approaches

### Brute Force Approach

A brute-force strategy would repeatedly compare and swap values among even indices and separately compare and swap values among odd indices.

For the even positions, we could run a nested loop similar to selection sort or bubble sort to place smaller values earlier. Then we could do another nested loop for the odd positions to place larger values earlier.

This approach works because it explicitly enforces the required ordering for each parity group. However, it performs unnecessary repeated comparisons and swaps.

Although the constraints are small enough that this would still pass, it is not the cleanest or most efficient solution.

### Optimal Approach

The key observation is that even-indexed elements and odd-indexed elements are completely independent from each other.

This means we can:

1. Extract all even-indexed values into one list.
2. Extract all odd-indexed values into another list.
3. Sort the even list in ascending order.
4. Sort the odd list in descending order.
5. Reconstruct the array by placing values back into their original parity positions.

This approach is simple, readable, and efficient. Since sorting is already highly optimized in most languages, using built-in sorting functions makes the implementation concise and reliable.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Uses repeated comparisons and swaps among parity groups |
| Optimal | O(n log n) | O(n) | Separates values by parity, sorts them, then rebuilds the array |

## Algorithm Walkthrough

1. Create two separate lists, one for even-indexed values and one for odd-indexed values.

We iterate through the array once. If the current index is even, we append the value to the even list. Otherwise, we append it to the odd list.
2. Sort the even-indexed values in ascending order.

This satisfies the requirement that values at even positions appear in non-decreasing order.
3. Sort the odd-indexed values in descending order.

This satisfies the requirement that values at odd positions appear in non-increasing order.
4. Reconstruct the array.

We iterate through the original indices again. For each even index, we place the next value from the sorted even list. For each odd index, we place the next value from the sorted odd list.
5. Return the modified array.

At this point, every even position contains values in ascending order, and every odd position contains values in descending order.

### Why it works

The algorithm works because even-indexed positions and odd-indexed positions are independent subsets of the array. By extracting and sorting each subset separately, we guarantee that each parity group satisfies its required ordering. Reconstructing the array while preserving index parity ensures that no even value is placed into an odd position and vice versa.

## Python Solution

```python
from typing import List

class Solution:
    def sortEvenOdd(self, nums: List[int]) -> List[int]:
        even_values = []
        odd_values = []

        # Separate values based on index parity
        for index, value in enumerate(nums):
            if index % 2 == 0:
                even_values.append(value)
            else:
                odd_values.append(value)

        # Sort according to requirements
        even_values.sort()
        odd_values.sort(reverse=True)

        even_pointer = 0
        odd_pointer = 0

        # Rebuild the array
        for index in range(len(nums)):
            if index % 2 == 0:
                nums[index] = even_values[even_pointer]
                even_pointer += 1
            else:
                nums[index] = odd_values[odd_pointer]
                odd_pointer += 1

        return nums
```

The implementation begins by creating two separate lists, `even_values` and `odd_values`. During the first traversal of the array, values are grouped according to whether their indices are even or odd.

Next, the even-indexed values are sorted in ascending order using `sort()`, while the odd-indexed values are sorted in descending order using `sort(reverse=True)`.

Two pointers are then used during reconstruction. `even_pointer` tracks the next value to place from the sorted even list, and `odd_pointer` tracks the next value from the sorted odd list.

Finally, the algorithm iterates through the original indices again and places the correct sorted value into each position based on parity.

## Go Solution

```go
package main

import "sort"

func sortEvenOdd(nums []int) []int {
    evenValues := []int{}
    oddValues := []int{}

    // Separate values based on index parity
    for i, value := range nums {
        if i%2 == 0 {
            evenValues = append(evenValues, value)
        } else {
            oddValues = append(oddValues, value)
        }
    }

    // Sort even indices ascending
    sort.Ints(evenValues)

    // Sort odd indices descending
    sort.Sort(sort.Reverse(sort.IntSlice(oddValues)))

    evenPointer := 0
    oddPointer := 0

    // Rebuild the array
    for i := 0; i < len(nums); i++ {
        if i%2 == 0 {
            nums[i] = evenValues[evenPointer]
            evenPointer++
        } else {
            nums[i] = oddValues[oddPointer]
            oddPointer++
        }
    }

    return nums
}
```

The Go solution follows the same overall structure as the Python version. The main language-specific difference is sorting in descending order. Go's standard library uses:

```go
sort.Sort(sort.Reverse(sort.IntSlice(oddValues)))
```

to reverse the default ascending sort order.

Slices are used naturally for dynamic storage, and the array is modified in place before being returned.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | Sorting dominates the runtime |
| Space | O(n) | Extra lists store separated even and odd values |

The algorithm performs three main operations: extracting values, sorting them, and rebuilding the array. Extraction and reconstruction are both linear operations. The sorting operations dominate the runtime complexity, resulting in `O(n log n)` overall complexity. Additional space is required for the two temporary lists storing even-indexed and odd-indexed values.

## Edge Cases

One important edge case is an array with only one element. In this situation, there is only an even index and no odd indices. A buggy implementation might incorrectly assume both groups contain elements. This solution handles the case naturally because the odd list simply remains empty, and reconstruction only uses the even list.

Another important edge case is when there is exactly one value in each parity group, such as `[2,1]`. Sorting single-element lists should not change their order. Some implementations accidentally over-process or mis-handle index reconstruction in these tiny cases. This solution safely reconstructs the array using parity checks and separate pointers.

Arrays containing duplicate values are also important. Since equal values may appear in both parity groups, the implementation must not accidentally merge or mix them. By storing even-indexed and odd-indexed values separately throughout the algorithm, duplicates remain correctly assigned to their respective parity positions.

A final important edge case is when the even or odd subsequence is already correctly sorted. For example:

```
[1,9,3,7,5,5]
```

The even indices are already ascending, and the odd indices are already descending. The algorithm still works correctly because sorting an already sorted sequence leaves it unchanged.
