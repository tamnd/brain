## Solution

## Problem Understanding

The problem describes a scenario where a planet with an initial `mass` can collide with a sequence of asteroids, each with a given mass. The key rule is that a collision can only destroy an asteroid if the planet’s mass is greater than or equal to the asteroid’s mass. When a collision occurs and the asteroid is destroyed, the planet gains the asteroid’s mass. The objective is to determine if the planet can destroy all asteroids, assuming we can choose the collision order arbitrarily.

The input consists of an integer `mass` representing the planet’s starting mass and an array `asteroids` of integers representing the mass of each asteroid. The output is a boolean: `true` if there exists an ordering of asteroid collisions where the planet survives all, and `false` otherwise.

The constraints indicate that the size of the input array can be up to 100,000, and asteroid masses can be as high as 100,000. This suggests that a brute-force approach trying all permutations would be infeasible. Edge cases include situations where the planet starts with a mass smaller than the smallest asteroid or where the planet can sequentially destroy all but the largest asteroid, which is still too heavy.

Important considerations include checking for integer overflow when summing masses, and ensuring that the solution handles arrays of length one, arrays with all equal asteroids, or cases where the initial planet mass is extremely small relative to the asteroids.

## Approaches

The brute-force approach would involve trying every permutation of asteroid collisions. For each permutation, we simulate the planet colliding with asteroids in that order, updating its mass after each collision. If at any point the planet cannot destroy an asteroid, that permutation fails. This guarantees correctness because all possibilities are explored, but its time complexity is O(n!) which is far too large for n up to 100,000.

The key observation for an optimal solution is that we want to collide with smaller asteroids first. By destroying smaller asteroids first, the planet accumulates mass, increasing the likelihood it can destroy larger asteroids later. Therefore, sorting the asteroids in ascending order and then simulating collisions is sufficient. This greedy strategy works because choosing a smaller asteroid never blocks the planet from destroying a larger asteroid later; delaying a smaller asteroid could only hurt.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n!) | O(n) | Tries all permutations; guarantees correctness but infeasible for large n |
| Optimal | O(n log n) | O(1) | Sort asteroids ascending and simulate collisions; uses constant extra space |

## Algorithm Walkthrough

1. **Sort the asteroids** array in ascending order. This ensures that the planet always collides with the smallest available asteroid first, maximizing its chance of survival.
2. **Iterate through the sorted array** of asteroids. For each asteroid, compare its mass with the current planet mass.
3. **Check the collision condition**: if the planet mass is smaller than the asteroid mass, immediately return `false` because the planet cannot destroy this asteroid.
4. **Update the planet mass** by adding the asteroid mass after a successful collision. This simulates the planet growing after destroying an asteroid.
5. **Repeat** until all asteroids are processed. If the loop completes without returning `false`, return `true`.

**Why it works:** The invariant is that at each step, the planet has destroyed all smaller or equal asteroids encountered so far and gained their mass. Sorting ensures the planet never encounters an asteroid it cannot currently destroy if it could destroy all previous smaller ones. Thus, if the planet can destroy the sorted sequence, it can destroy all asteroids in some order.

## Python Solution

```python
from typing import List

class Solution:
    def asteroidsDestroyed(self, mass: int, asteroids: List[int]) -> bool:
        asteroids.sort()
        for asteroid in asteroids:
            if mass < asteroid:
                return False
            mass += asteroid
        return True
```

The implementation first sorts the asteroids to ensure the planet tackles smaller asteroids first. It then iterates through each asteroid, checking if the planet's mass is sufficient. If it is, the planet gains the asteroid's mass. If it is not, the function immediately returns `false`. The final return statement confirms that all asteroids have been successfully destroyed.

## Go Solution

```go
import "sort"

func asteroidsDestroyed(mass int, asteroids []int) bool {
    sort.Ints(asteroids)
    for _, asteroid := range asteroids {
        if mass < asteroid {
            return false
        }
        mass += asteroid
    }
    return true
}
```

The Go version mirrors the Python logic. The `sort.Ints` function sorts the slice in ascending order. The loop iterates over the slice, updating the mass and returning `false` if the planet cannot destroy an asteroid. There are no special considerations for nil slices because an empty slice is trivially handled by the loop.

## Edge Cases

One edge case is when the planet’s initial mass is smaller than all asteroids. In this case, the algorithm immediately returns `false` because it will fail on the first asteroid. This is correctly handled by the `if mass < asteroid` check in the loop.

A second edge case is when there is only one asteroid. The algorithm must correctly compare the planet’s mass to that asteroid. Sorting a single-element array has no effect, and the loop correctly decides whether the planet survives.

A third edge case involves extremely large arrays with repeated small asteroids. The planet accumulates mass incrementally. The solution handles this without overflow in Python due to its arbitrary-precision integers. In Go, care must be taken to avoid exceeding `int` limits, but the constraints ensure that `mass` and asteroid values are within 10^5, so integer overflow does not occur with 32-bit integers.
