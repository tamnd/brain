## Solution

## Problem Understanding

The problem asks us to count how many different sturdy brick walls can be built using an unlimited supply of given brick sizes.

Each brick has height `1` and a width taken from the `bricks` array. A wall has:

- A fixed number of rows, given by `height`
- A fixed row width, given by `width`

Every row must exactly fill the total width. Bricks cannot be rotated, so only their given widths matter.

The important constraint is what makes the wall "sturdy". If two adjacent rows have brick boundaries aligned at the same internal position, the wall is considered weak. The only allowed shared boundaries are the left edge and right edge of the wall.

For example, suppose a row of width `3` is built as:

```
[1][2]
```

This row has an internal seam at position `1`.

Another row:

```
[2][1]
```

has an internal seam at position `2`.

These two rows are compatible because their seams do not overlap.

However:

```
[1][2]
[1][2]
```

is invalid because both rows have a seam at position `1`.

The goal is to return the total number of valid sturdy walls modulo `10^9 + 7`.

The constraints are very important:

- `height <= 100`
- `width <= 10`
- `bricks.length <= 10`

The small width is the key observation. Even though the wall height can be large, the width is tiny, which means the number of possible row patterns is manageable. This strongly suggests preprocessing all valid row configurations and then using dynamic programming across rows.

Several edge cases are important:

- No brick can fit into the wall width, resulting in zero valid rows.
- Some widths may only allow one row configuration, making sturdiness impossible for height greater than `1`.
- Widths can often be formed in many ways, creating many seam patterns.
- A seam at the very end of the row does not count as an internal seam and must not be stored.

## Approaches

### Brute Force Approach

The brute force solution would generate every possible row configuration, then recursively try every possible combination of rows for the full wall height.

For each completed wall, we would verify whether every adjacent pair of rows is sturdy by checking whether they share internal seam positions.

This approach is correct because it explicitly enumerates all possible walls and validates each one.

However, it becomes extremely expensive. If there are `R` valid row patterns, then a wall of height `H` has `R^H` possible combinations. Even with moderate values, this becomes infeasible.

For example, if there are 50 row patterns and height is 100, brute force exploration is impossible.

### Optimal Approach

The key insight is that the sturdiness condition depends only on adjacent rows.

This transforms the problem into a graph-style dynamic programming problem:

- Each valid row pattern becomes a state
- Two states are connected if their seam masks do not overlap
- We count how many ways to build walls row by row

Another crucial optimization is representing seam positions as bitmasks.

Because `width <= 10`, there are at most 9 internal seam positions. We can encode seam locations using bits:

- Bit `i` indicates whether there is a seam after position `i`

This allows compatibility checks in constant time using bitwise AND:

```
mask1 & mask2 == 0
```

If the result is zero, the rows are compatible.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(R^H) | O(H) | Tries every possible wall configuration |
| Optimal | O(R² × H) | O(R² + R) | Uses row-state DP with bitmask compatibility |

Where `R` is the number of valid row configurations.

## Algorithm Walkthrough

### Step 1: Generate All Valid Row Configurations

We use backtracking to generate every possible way to fill a row of total width `width`.

During construction:

- We track the current filled length
- Whenever we place a brick and are not at the end, we record a seam
- Seams are encoded in a bitmask

For example:

```
width = 4
row = [1,3]
```

The seam occurs after position `1`.

Mask:

```
0010
```

If we reach exactly `width`, we store the completed mask.

### Step 2: Build Compatibility Relationships

For every pair of row masks:

- They are compatible if they share no internal seams
- We check:

```
mask1 & mask2 == 0
```

If true, the rows can be stacked.

We build an adjacency list where:

```
compatible[i] = all rows that can be placed above row i
```

### Step 3: Initialize Dynamic Programming

Define:

```
dp[i]
```

as the number of ways to build the current height ending with row pattern `i`.

Initially:

- Any valid row can be the first row
- Therefore:

```
dp[i] = 1
```

for all row patterns.

### Step 4: Build the Wall Row by Row

For each additional layer:

- Create a new DP array
- Transition between compatible rows

If row `j` can sit above row `i`:

```
next_dp[j] += dp[i]
```

Take modulo `10^9 + 7` after every update.

### Step 5: Return Final Answer

After processing all `height` rows:

- Sum all DP values
- Return modulo `10^9 + 7`

### Why it works

The algorithm works because every valid wall can be uniquely represented as a sequence of compatible row states. The DP transition exactly captures the sturdiness condition by only allowing transitions between rows whose seam masks do not overlap. Since every possible row configuration is generated and every valid transition is counted, the DP enumerates all sturdy walls exactly once.

## Python Solution

```python
from typing import List

class Solution:
    def buildWall(self, height: int, width: int, bricks: List[int]) -> int:
        MOD = 10**9 + 7

        row_masks = []

        def generate_rows(position: int, mask: int) -> None:
            if position == width:
                row_masks.append(mask)
                return

            for brick in bricks:
                next_position = position + brick

                if next_position > width:
                    continue

                next_mask = mask

                if next_position < width:
                    next_mask |= (1 << next_position)

                generate_rows(next_position, next_mask)

        generate_rows(0, 0)

        n = len(row_masks)

        compatible = [[] for _ in range(n)]

        for i in range(n):
            for j in range(n):
                if row_masks[i] & row_masks[j] == 0:
                    compatible[i].append(j)

        dp = [1] * n

        for _ in range(1, height):
            next_dp = [0] * n

            for i in range(n):
                for j in compatible[i]:
                    next_dp[j] = (next_dp[j] + dp[i]) % MOD

            dp = next_dp

        return sum(dp) % MOD
```

The implementation follows the algorithm directly.

The `generate_rows` function recursively constructs every possible row layout. The `position` parameter tracks how much width has already been filled, while `mask` stores all internal seam positions.

When a brick placement reaches exactly the wall width, the row is complete and its mask is stored.

After generating all rows, the code computes compatibility between every pair of masks. The bitwise AND operation efficiently determines whether two rows share seam positions.

The dynamic programming phase iteratively builds taller walls. The `dp` array tracks how many ways each row pattern can appear at the current height. For every compatible transition, counts are propagated into `next_dp`.

Finally, the total number of sturdy walls is the sum of all valid ending states.

## Go Solution

```go
package main

func buildWall(height int, width int, bricks []int) int {
	const MOD = 1000000007

	rowMasks := []int{}

	var generateRows func(int, int)

	generateRows = func(position int, mask int) {
		if position == width {
			rowMasks = append(rowMasks, mask)
			return
		}

		for _, brick := range bricks {
			nextPosition := position + brick

			if nextPosition > width {
				continue
			}

			nextMask := mask

			if nextPosition < width {
				nextMask |= (1 << nextPosition)
			}

			generateRows(nextPosition, nextMask)
		}
	}

	generateRows(0, 0)

	n := len(rowMasks)

	compatible := make([][]int, n)

	for i := 0; i < n; i++ {
		for j := 0; j < n; j++ {
			if rowMasks[i]&rowMasks[j] == 0 {
				compatible[i] = append(compatible[i], j)
			}
		}
	}

	dp := make([]int, n)

	for i := 0; i < n; i++ {
		dp[i] = 1
	}

	for level := 1; level < height; level++ {
		nextDP := make([]int, n)

		for i := 0; i < n; i++ {
			for _, j := range compatible[i] {
				nextDP[j] = (nextDP[j] + dp[i]) % MOD
			}
		}

		dp = nextDP
	}

	result := 0

	for _, value := range dp {
		result = (result + value) % MOD
	}

	return result
}
```

The Go implementation mirrors the Python logic closely. Slices are used for dynamic arrays, and recursive row generation is implemented using a function variable.

One important difference is integer handling. Go integers are fixed-width, so modulo operations are applied consistently to avoid overflow accumulation. Since all calculations stay below the modulo range, standard `int` works safely within constraints.

## Edge Cases

One important edge case occurs when no brick can fit into the wall width. For example, `width = 1` and `bricks = [5]`. In this situation, the recursive row generator never reaches the target width, so no row states are created. The DP naturally handles this by operating on an empty state set and returning `0`.

Another subtle case is when the wall width is `1`. A row of width `1` has no internal seams because the only boundaries are the edges of the wall. This means every row is automatically compatible with every other row. The implementation handles this correctly because seam masks remain `0`, and `0 & 0 == 0`.

A third important edge case happens when only one row pattern exists, but it contains seams. For example:

```
width = 2
bricks = [1]
```

The only row is `[1,1]`, which contains a seam at position `1`. Since every adjacent row would repeat that seam, walls taller than one row are impossible. The compatibility graph correctly contains no self-edge for this mask, causing the DP counts to become zero after the first layer.
