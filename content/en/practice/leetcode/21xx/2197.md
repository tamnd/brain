## Solution

## Problem Understanding

The problem gives an integer array `nums` and asks us to repeatedly merge adjacent numbers whenever those two numbers are non-coprime. Two numbers are non-coprime if their greatest common divisor, `gcd(x, y)`, is greater than `1`.

Whenever we find such a pair, we remove both numbers and replace them with their least common multiple, `lcm(x, y)`.

The process continues until no adjacent pair has a gcd greater than `1`. At that point, we return the final array.

The important detail is that we may choose any valid adjacent pair at each step, but the problem guarantees that the final result is always the same regardless of the order of operations. This means we do not need to worry about trying all possible merge sequences.

The input size is large:

- `1 <= nums.length <= 10^5`
- `1 <= nums[i] <= 10^5`

A quadratic simulation that repeatedly scans the array and rebuilds it after each merge would be too slow for the upper bound.

The final values are guaranteed to remain at most `10^8`, which means standard integer arithmetic is safe in both Python and Go.

Several edge cases are important:

- Arrays where no adjacent numbers are non-coprime, such as `[2,3,5,7]`
- Arrays where one merge creates another merge opportunity, such as `[6,4,3]`
- Arrays containing `1`, because `gcd(1, x) = 1`, so `1` never merges with anything
- Long chains of cascading merges, where repeated reductions collapse many elements into one value

The main challenge is efficiently handling these cascading merges.

## Approaches

### Brute Force Approach

A straightforward approach is to repeatedly scan the array looking for adjacent non-coprime pairs.

Whenever we find a pair:

1. Compute their LCM
2. Remove the two elements
3. Insert the LCM in their place
4. Restart scanning from the beginning

This method is correct because it directly simulates the problem statement.

However, it is inefficient because every merge may require rebuilding the array, and after each rebuild we may need another full scan. In the worst case, this can lead to quadratic or even worse behavior.

For example, an array of many repeated even numbers could repeatedly collapse one pair at a time, forcing many rescans.

### Key Insight for the Optimal Solution

The crucial observation is that merging only affects neighboring elements.

Suppose we process the array from left to right while maintaining the already processed result in a stack.

When we add a new number:

- If it is coprime with the top of the stack, no merge is possible, so we simply push it.
- If it is non-coprime with the top, we merge them into their LCM.
- That newly formed LCM may again be non-coprime with the next stack element below it, so we continue merging until the top becomes coprime.

This behavior is naturally modeled using a stack because every new merge only interacts with the most recent surviving element.

Each number is pushed once and removed at most once, making the approach efficient.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeated rescanning and rebuilding of the array |
| Optimal | O(n log M) | O(n) | Stack-based merging using GCD and LCM operations |

Here, `M` represents the magnitude of the numbers involved, and the logarithmic factor comes from the Euclidean GCD algorithm.

## Algorithm Walkthrough

1. Create an empty stack.

The stack will store the partially processed final array. At every moment, the stack represents a valid reduced prefix where no adjacent elements are mergeable.
2. Iterate through each number in `nums`.

We process the array from left to right because merges only depend on neighboring elements.
3. Push the current number into a temporary variable called `current`.

This value may grow as repeated merges occur.
4. While the stack is not empty and `gcd(stack[-1], current) > 1`, merge them.

If the top of the stack and the current number are non-coprime, they must eventually merge in the final answer.
5. Compute the LCM.

Use the formula:

$\mathrm{lcm}(a,b)=\frac{a\times b}{\gcd(a,b)}$

We divide before multiplying when possible to avoid unnecessary overflow risk.
6. Replace `current` with the computed LCM.

The merged value may still be non-coprime with earlier elements in the stack.
7. Continue merging while possible.

This handles cascading reductions such as:

```
[6,4,3]
```

First:

```
lcm(6,4) = 12
```

Then:

```
gcd(12,3) = 3
```

So another merge occurs immediately.
8. After all possible merges are complete, push `current` onto the stack.
9. After processing all numbers, return the stack.

### Why it works

The stack always maintains the invariant that no two adjacent elements inside it are non-coprime.

Whenever a new number arrives, the only possible invalid adjacency is between that number and the stack top. If they merge, the newly formed LCM can only conflict with the next element below it. Therefore, repeatedly merging with the stack top completely resolves all necessary interactions.

Because every required merge is performed immediately when it becomes possible, the final stack exactly matches the unique final array guaranteed by the problem.

## Python Solution

```python
from typing import List
from math import gcd

class Solution:
    def replaceNonCoprimes(self, nums: List[int]) -> List[int]:
        stack = []

        for num in nums:
            current = num

            while stack:
                top = stack[-1]
                current_gcd = gcd(top, current)

                if current_gcd == 1:
                    break

                stack.pop()
                current = (top // current_gcd) * current

            stack.append(current)

        return stack
```

The implementation uses a stack to maintain the current reduced array.

For every number in the input, we attempt to merge it with the top of the stack. The `gcd` function determines whether two numbers are non-coprime.

If the gcd is greater than `1`, we compute the LCM using:

```python
(top // gcd) * current
```

This formulation avoids unnecessary intermediate growth.

After merging, we continue checking against the new stack top because the merged value may create another merge opportunity.

Once no more merges are possible, the final value is pushed onto the stack.

At the end, the stack itself is the answer.

## Go Solution

```go
package main

func gcd(a, b int) int {
	for b != 0 {
		a, b = b, a%b
	}
	return a
}

func replaceNonCoprimes(nums []int) []int {
	stack := []int{}

	for _, num := range nums {
		current := num

		for len(stack) > 0 {
			top := stack[len(stack)-1]
			currentGCD := gcd(top, current)

			if currentGCD == 1 {
				break
			}

			stack = stack[:len(stack)-1]
			current = (top / currentGCD) * current
		}

		stack = append(stack, current)
	}

	return stack
}
```

The Go solution follows the same logic as the Python version.

Slices naturally work as stacks using `append` and slicing operations.

Go does not include a built-in `gcd` function, so we implement the Euclidean algorithm manually.

The expression:

```go
(top / currentGCD) * current
```

computes the LCM safely and avoids larger intermediate multiplication values.

Because the final values are guaranteed to stay within `10^8`, standard `int` arithmetic is sufficient.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log M) | Each element is pushed and popped at most once, each merge requires a GCD computation |
| Space | O(n) | The stack may store all elements in the worst case |

The Euclidean algorithm computes GCD in logarithmic time relative to the magnitude of the numbers. Since every element can only enter and leave the stack once, the total number of stack operations is linear.

Therefore, the overall complexity is dominated by the GCD computations, giving `O(n log M)` time complexity.

## Edge Cases

One important edge case is arrays containing `1`. Since `gcd(1, x) = 1` for every integer `x`, a value of `1` can never participate in a merge. A buggy implementation might accidentally merge values when using incorrect GCD logic. The solution handles this naturally because the merge loop only activates when the gcd is greater than `1`.

Another critical edge case is cascading merges. Consider `[6,4,3]`. After merging `6` and `4` into `12`, the new value `12` must immediately be checked again against `3`. A naive implementation that only checks one merge step would incorrectly stop early. The stack-based loop continues merging until the top becomes coprime.

A third important edge case is when the entire array collapses into a single number, such as `[4,8,16]`. Some implementations may incorrectly preserve intermediate values or mishandle repeated popping from the stack. Because the algorithm repeatedly removes merged elements and updates the current value before pushing it back, it correctly reduces the whole array into one final LCM.

A final edge case involves already valid arrays such as `[2,3,5,7]`. The algorithm must avoid unnecessary modifications. Since every adjacent gcd equals `1`, each number is simply pushed once, and the original array is returned unchanged.
