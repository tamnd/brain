## Solution

## Problem Understanding

The problem gives us a directed acyclic graph, also called a DAG, with `n` nodes numbered from `0` to `n - 1`. Each directed edge `[u, v]` means there is a one way connection from node `u` to node `v`.

Our task is to compute, for every node `i`, all nodes that can eventually reach `i` through one or more directed edges. These nodes are called the ancestors of `i`.

The output must be a list called `answer` where:

- `answer[i]` contains every ancestor of node `i`
- the ancestors must be sorted in ascending order

For example, if there is a path:

```
0 -> 3 -> 5
```

then node `0` is an ancestor of node `5`, because it can reach `5` through intermediate nodes.

The graph is guaranteed to be acyclic, which is extremely important. Since there are no cycles, we can process nodes in topological order without worrying about infinite loops or repeated circular dependencies.

The constraints are:

- `n <= 1000`
- `edges.length <= 2000`

These constraints are moderate. A fully quadratic or cubic algorithm may still be acceptable depending on implementation details, but we should still aim for something efficient and clean.

Several edge cases are important:

- A graph may contain isolated nodes with no incoming or outgoing edges
- Some nodes may have no ancestors at all
- The graph may be completely disconnected
- A node may have many indirect ancestors through long dependency chains
- Multiple paths may lead to the same ancestor, so duplicate handling is necessary
- Since the graph is acyclic, we never need cycle detection

The requirement that the graph is a DAG strongly suggests using topological sorting or propagation techniques.

## Approaches

### Brute Force Approach

The brute force idea is to run a graph traversal from every node.

For each node `u`:

1. Start a DFS or BFS from `u`
2. Visit every node reachable from `u`
3. For every visited node `v`, add `u` to `answer[v]`

This works because if `u` can reach `v`, then `u` is an ancestor of `v`.

Although this approach is conceptually simple, it can become expensive because we potentially traverse large portions of the graph `n` times.

If we perform DFS or BFS from every node independently, the complexity becomes:

- `O(n * (n + e))`

where:

- `n` is the number of nodes
- `e` is the number of edges

With `n = 1000` and `e = 2000`, this is still technically feasible, but we can do better using the DAG property.

### Key Insight for the Optimal Solution

Because the graph is acyclic, we can process nodes in topological order.

Suppose we already know all ancestors of node `u`. When we process an edge:

```
u -> v
```

then:

- `u` itself is an ancestor of `v`
- every ancestor of `u` is also an ancestor of `v`

So instead of repeatedly traversing the graph from scratch, we can propagate ancestor information forward through the DAG.

This turns the problem into a dynamic propagation process over a topological ordering.

We use:

- adjacency lists for graph traversal
- indegree counts for Kahn's topological sort
- sets to efficiently merge ancestor information without duplicates

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force DFS/BFS from every node | O(n × (n + e)) | O(n + e) | Repeated traversals cause redundant work |
| Topological Sort + Ancestor Propagation | O(n² + e) | O(n²) | Efficient reuse of previously computed ancestor sets |

## Algorithm Walkthrough

### Optimal Algorithm: Topological Sort with Ancestor Propagation

1. Build the graph using an adjacency list.

For every edge `[u, v]`, add `v` to the neighbors of `u`.

At the same time, compute the indegree of every node. The indegree represents how many incoming edges a node has.
2. Create an ancestor set for every node.

We use a set because:

- ancestor relationships may arrive from multiple paths
- sets automatically prevent duplicates
- insertion is efficient
3. Initialize a queue with all nodes whose indegree is `0`.

These nodes have no parents, so they also have no ancestors.
4. Process nodes in topological order using Kahn's algorithm.

Repeatedly:

- remove a node `u` from the queue
- visit every neighbor `v`
5. Propagate ancestor information through each edge.

For edge:

```
u -> v
```

do two things:

- add `u` into `ancestors[v]`
- add all ancestors of `u` into `ancestors[v]`

This works because every ancestor of `u` can also reach `v`.
6. Reduce indegree of each neighbor.

Since one prerequisite has been processed, decrement `indegree[v]`.

If `indegree[v]` becomes `0`, push it into the queue.
7. Convert each ancestor set into a sorted list.

The problem requires ascending order, so sort every set before returning.

### Why it works

The key invariant is:

> When a node `u` is processed in topological order, its ancestor set is already complete.

This is guaranteed because all incoming edges to `u` must come from earlier nodes in the topological ordering.

Therefore, when we propagate ancestor information from `u` to its neighbors, we are propagating fully correct information. Since every edge is processed exactly once in topological order, every reachable ancestor relationship is eventually transferred to all descendants.

## Python Solution

```python
from collections import deque
from typing import List

class Solution:
    def getAncestors(self, n: int, edges: List[List[int]]) -> List[List[int]]:
        graph = [[] for _ in range(n)]
        indegree = [0] * n

        for u, v in edges:
            graph[u].append(v)
            indegree[v] += 1

        ancestors = [set() for _ in range(n)]

        queue = deque()

        for node in range(n):
            if indegree[node] == 0:
                queue.append(node)

        while queue:
            current = queue.popleft()

            for neighbor in graph[current]:
                ancestors[neighbor].add(current)
                ancestors[neighbor].update(ancestors[current])

                indegree[neighbor] -= 1

                if indegree[neighbor] == 0:
                    queue.append(neighbor)

        return [sorted(list(ancestor_set)) for ancestor_set in ancestors]
```

The implementation starts by constructing the adjacency list and indegree array. The adjacency list allows efficient traversal of outgoing edges, while the indegree array enables Kahn's topological sorting algorithm.

Next, we create an array of sets called `ancestors`. Each index stores all ancestors for that node. Sets are important because a node may be reachable through multiple different paths, and we must avoid duplicates automatically.

The queue initially contains all nodes with indegree `0`. These nodes have no incoming edges, so they have no ancestors.

The main loop processes nodes in topological order. For every directed edge `current -> neighbor`, we propagate ancestor information forward:

```python
ancestors[neighbor].add(current)
ancestors[neighbor].update(ancestors[current])
```

The first line adds the direct parent. The second line adds all indirect ancestors.

After processing an edge, we decrement the indegree of the neighbor. Once the indegree becomes `0`, all prerequisite nodes have already been processed, so the neighbor can safely enter the queue.

Finally, we convert every ancestor set into a sorted list because the problem requires ascending order.

## Go Solution

```go
package main

import "sort"

func getAncestors(n int, edges [][]int) [][]int {
	graph := make([][]int, n)
	indegree := make([]int, n)

	for _, edge := range edges {
		u := edge[0]
		v := edge[1]

		graph[u] = append(graph[u], v)
		indegree[v]++
	}

	ancestors := make([]map[int]bool, n)

	for i := 0; i < n; i++ {
		ancestors[i] = make(map[int]bool)
	}

	queue := []int{}

	for i := 0; i < n; i++ {
		if indegree[i] == 0 {
			queue = append(queue, i)
		}
	}

	for len(queue) > 0 {
		current := queue[0]
		queue = queue[1:]

		for _, neighbor := range graph[current] {
			ancestors[neighbor][current] = true

			for ancestor := range ancestors[current] {
				ancestors[neighbor][ancestor] = true
			}

			indegree[neighbor]--

			if indegree[neighbor] == 0 {
				queue = append(queue, neighbor)
			}
		}
	}

	result := make([][]int, n)

	for i := 0; i < n; i++ {
		for ancestor := range ancestors[i] {
			result[i] = append(result[i], ancestor)
		}

		sort.Ints(result[i])
	}

	return result
}
```

The Go implementation follows the same logic as the Python version, but there are some language specific differences.

Go does not have a built in set type, so we simulate sets using `map[int]bool`.

Queues are implemented using slices. We repeatedly remove the first element using:

```go
queue = queue[1:]
```

The final answer must be sorted, so we use:

```go
sort.Ints(result[i])
```

Go slices default to `nil`, but appending to a nil slice works correctly, so no special handling is needed.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n² + e) | Ancestor propagation across sets can total up to O(n²) |
| Space | O(n²) | In the worst case, every node may store nearly all other nodes as ancestors |

The graph traversal itself is linear in the number of edges and nodes. However, ancestor propagation dominates the complexity.

In the worst case, the graph forms a chain:

```
0 -> 1 -> 2 -> 3 -> ...
```

Then node `i` stores `i` ancestors, leading to approximately:

```
1 + 2 + 3 + ... + n = O(n²)
```

total stored ancestor relationships.

## Edge Cases

### Graph With No Edges

A graph may contain nodes with absolutely no connections.

Example:

```
n = 4
edges = []
```

In this situation, every node has zero ancestors. A buggy implementation might incorrectly initialize ancestor sets or fail to process nodes with indegree `0`.

Our implementation handles this correctly because all nodes enter the queue immediately, but no ancestor propagation occurs.

### Multiple Paths to the Same Ancestor

Consider:

```
0 -> 1
0 -> 2
1 -> 3
2 -> 3
```

Node `0` can reach node `3` through two different paths.

A naive implementation using lists instead of sets could accidentally insert duplicate ancestors. Our implementation uses sets, so duplicates are automatically removed.

### Long Dependency Chains

Consider:

```
0 -> 1 -> 2 -> 3 -> 4
```

Each node accumulates all previous nodes as ancestors.

This case stresses transitive propagation logic. If we only added direct parents and forgot to propagate indirect ancestors, the output would be incorrect.

Our algorithm explicitly propagates:

```
ancestors[neighbor].update(ancestors[current])
```

which guarantees all indirect ancestor relationships are preserved.
