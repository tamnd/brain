# LeetCode 2148 - Count Elements With Strictly Smaller and Greater Elements

## Problem Understanding

The problem gives us an integer array `nums` and asks us to count how many elements satisfy a very specific condition:

- There must exist at least one element in the array that is strictly smaller.
- There must also exist at least one element in the array that is strictly greater.

In other words, we only count numbers that are neither the minimum value nor the maximum value in the array.

For example, consider the array `[11,7,2,15]`.

- `2` is the smallest value, so there is no smaller element than `2`, therefore it does not qualify.
- `15` is the largest value, so there is no greater element than `15`, therefore it does not qualify.
- `7` has `2` smaller and `11` or `15` greater, so it qualifies.
- `11` has `7` or `2` smaller and `15` greater, so it qualifies.

The answer is `2`.

An important detail is that duplicates count individually. In the example `[-3,3,3,90]`, both occurrences of `3` qualify independently, so the answer is `2`.

The constraints are small:

- `1 <= nums.length <= 100`
- `-10^5 <= nums[i] <= 10^5`

Because the array size is at most 100, even an `O(n^2)` solution would work comfortably. However, the problem is simple enough that we can derive a cleaner and more efficient linear-time solution.

Several edge cases are important:

- Arrays with only one element.
- Arrays where all values are identical.
- Arrays with exactly two distinct values.
- Arrays containing duplicates of the minimum or maximum value.
- Negative numbers.

The problem guarantees that the input array is non-empty, so we never need to handle a completely empty array.

## Approaches

### Brute Force Approach

The brute force method checks every element individually.

For each number `nums[i]`, we scan the entire array again and determine:

- Does there exist a value smaller than `nums[i]`?
- Does there exist a value greater than `nums[i]`?

If both conditions are true, we increment the answer.

This approach is correct because it directly follows the problem definition. Every element is explicitly verified against all other elements.

However, this requires nested loops. For every element, we potentially compare against every other element, resulting in `O(n^2)` time complexity.

Although the constraints are small enough for this to pass, we can do better.

### Optimal Approach

The key observation is that an element qualifies if and only if:

- It is strictly greater than the minimum value in the array.
- It is strictly smaller than the maximum value in the array.

This means we do not need to compare every element with every other element.

Instead:

1. Find the minimum value in the array.
2. Find the maximum value in the array.
3. Count how many elements lie strictly between them.

This works because:

- Any occurrence of the minimum value can never have a smaller element.
- Any occurrence of the maximum value can never have a greater element.
- Every value strictly between the minimum and maximum automatically satisfies both conditions.

This reduces the solution to linear time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Compare every element against all others |
| Optimal | O(n) | O(1) | Find min/max once, then count elements between them |

## Algorithm Walkthrough

1. Initialize two variables, `minimum_value` and `maximum_value`, using the smallest and largest values in the array.

We compute these first because they completely determine which elements qualify.
2. Initialize a counter variable called `count` to `0`.

This variable stores the number of valid elements.
3. Iterate through each number in the array.

For every element `num`, check whether:

- `num > minimum_value`
- `num < maximum_value`
4. If both conditions are true, increment `count`.

Such an element has at least one smaller element and at least one greater element.
5. After processing the entire array, return `count`.

### Why it works

The minimum value cannot qualify because no array element is strictly smaller than it. Similarly, the maximum value cannot qualify because no array element is strictly greater than it.

Any value strictly between the minimum and maximum automatically has:

- A smaller value, namely the minimum.
- A greater value, namely the maximum.

Therefore, counting all elements strictly between the global minimum and maximum produces the correct answer.

## Python Solution

```python
from typing import List

class Solution:
    def countElements(self, nums: List[int]) -> int:
        minimum_value = min(nums)
        maximum_value = max(nums)

        count = 0

        for num in nums:
            if minimum_value < num < maximum_value:
                count += 1

        return count
```

The implementation starts by computing the minimum and maximum values of the array using Python's built-in `min()` and `max()` functions.

Next, we initialize a counter variable to track how many valid elements exist.

We then iterate through every number in the array. The chained comparison:

```python
minimum_value < num < maximum_value
```

checks both conditions simultaneously:

- The current number must be greater than the minimum.
- The current number must be smaller than the maximum.

Whenever both conditions are satisfied, we increment the counter.

Finally, the method returns the total count.

The implementation is concise because the problem's core logic is based entirely on the relationship between each element and the global minimum and maximum.

## Go Solution

```go
func countElements(nums []int) int {
    minimumValue := nums[0]
    maximumValue := nums[0]

    for _, num := range nums {
        if num < minimumValue {
            minimumValue = num
        }

        if num > maximumValue {
            maximumValue = num
        }
    }

    count := 0

    for _, num := range nums {
        if num > minimumValue && num < maximumValue {
            count++
        }
    }

    return count
}
```

The Go implementation follows the same logic as the Python version, but Go does not provide built-in `min()` and `max()` functions for slices.

Instead, we initialize both `minimumValue` and `maximumValue` with the first element and update them during iteration.

After determining the minimum and maximum values, we perform a second pass through the slice to count elements strictly between them.

Since the constraints are small and values fit easily within Go's `int` type, integer overflow is not a concern here.

## Edge Cases

### All Elements Are Identical

An array like `[4,4,4,4]` can easily cause logical mistakes if we only think about positions rather than values.

Although there are multiple elements, none of them have a strictly smaller or strictly greater value because every number is equal. The implementation handles this naturally because no element satisfies:

```
minimum_value < num < maximum_value
```

when the minimum and maximum are the same.

### Array Contains Only Two Distinct Values

Consider `[1,1,5,5]`.

Every element is either the minimum or the maximum. There is no value strictly between them, so the answer should be `0`.

A buggy implementation might incorrectly count duplicates as valid simply because other elements exist. Our solution avoids this by requiring strict inequalities.

### Single Element Array

For input like `[10]`, the only element is simultaneously the minimum and maximum.

Since there is no smaller or greater value, the correct answer is `0`.

The implementation handles this correctly because:

```
minimum_value < num < maximum_value
```

can never be true when all three values are equal.
