## Solution

## Problem Understanding

The problem gives us an array `candies`, where each value represents the flavor of a candy. We must give away exactly `k` consecutive candies to a younger sister. After removing that contiguous block from the array, we keep all remaining candies.

Our goal is to maximize the number of unique candy flavors that remain in our possession.

In other words, for every possible subarray of length `k`, we imagine removing it from the array and counting how many distinct flavors are left outside that window. We then return the maximum such value.

For example, suppose:

```
candies = [1,2,2,3,4,3]
k = 3
```

If we give away the consecutive candies `[2,2,3]`, we keep `[1,4,3]`. The remaining distinct flavors are `{1,3,4}`, so the answer is `3`.

The constraints are important:

- The array length can be as large as `10^5`
- Flavor values can also be as large as `10^5`

These limits immediately tell us that an `O(n^2)` solution will be too slow. We need something close to linear time.

Several edge cases are especially important:

- `k = 0`, meaning we give away nothing
- `k = len(candies)`, meaning we give away all candies
- Arrays with all identical flavors
- Arrays where every candy has a unique flavor
- Empty arrays

A correct solution must handle all of these efficiently.

## Approaches

### Brute Force Approach

A straightforward solution is to try every possible contiguous subarray of length `k` and simulate giving those candies away.

For each window:

1. Remove the candies in that window.
2. Build a set of the remaining candies.
3. Count the number of unique flavors.
4. Track the maximum value.

This works because it directly evaluates every valid choice of candies to share.

However, this approach is inefficient. There are `O(n)` possible windows, and for each one we may scan almost the entire array again to count distinct flavors. That leads to `O(n^2)` time complexity in the worst case.

With `n = 10^5`, this is far too slow.

### Optimal Sliding Window Approach

The key observation is that instead of repeatedly rebuilding the remaining candies from scratch, we can maintain the flavor counts dynamically.

Think about the problem from the opposite perspective:

- Initially, we own all candies.
- Then we slide a window of size `k` across the array.
- Every candy inside the window is temporarily removed from our collection.
- Every candy outside the window remains with us.

We can maintain a frequency map representing the candies we currently keep.

Initially:

- Count frequencies of all candies.

Then, as the sliding window expands:

- When a candy enters the shared window, decrease its count in our map.
- If its frequency becomes zero, remove it from the map.
- The number of keys in the map equals the number of distinct flavors we still keep.

As the window slides forward:

- Add back the candy leaving the window.
- Remove the candy entering the window.

This lets us process every element only a constant number of times, giving an `O(n)` solution.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Recomputes distinct flavors for every window |
| Optimal | O(n) | O(n) | Uses sliding window with frequency counting |

## Algorithm Walkthrough

1. Create a frequency map containing counts of all candies.

This map represents the candies we currently keep before sharing anything.
2. Initialize the answer as the current number of unique flavors.

If `k = 0`, this value is immediately the final answer.
3. Start sliding a window of size `k` across the array.

The window represents the candies we give away.
4. For each candy entering the window:

Decrease its frequency in the map because we no longer keep that candy.
5. If a candy's frequency becomes zero:

Remove it from the map entirely.

This is important because the number of keys in the map represents the number of unique flavors still available to us.
6. Once the window size reaches `k`:

Update the answer using the current number of distinct flavors.
7. Move the window forward by one position:

Add back the candy leaving the window because we now keep it again.
8. Continue until all windows have been processed.

### Why it works

The sliding window always represents the exact candies being shared. The frequency map always represents the exact candies still owned after removing that window.

Because we correctly update frequencies whenever candies enter or leave the shared window, the map remains accurate throughout the process. Therefore, the number of keys in the map always equals the number of unique flavors we keep for the current choice of shared candies.

Taking the maximum over all windows guarantees the optimal answer.

## Python Solution

```python
from typing import List
from collections import Counter

class Solution:
    def shareCandies(self, candies: List[int], k: int) -> int:
        frequency = Counter(candies)

        max_unique = len(frequency)

        left = 0

        for right in range(len(candies)):
            flavor = candies[right]

            frequency[flavor] -= 1

            if frequency[flavor] == 0:
                del frequency[flavor]

            if right - left + 1 == k:
                max_unique = max(max_unique, len(frequency))

                left_flavor = candies[left]
                frequency[left_flavor] = frequency.get(left_flavor, 0) + 1

                left += 1

        return max_unique
```

The implementation begins by building a frequency table using `Counter`. This table tracks how many copies of each flavor we currently keep.

Initially, we keep every candy, so the number of unique flavors is simply the number of keys in the frequency map.

The sliding window is controlled using two pointers:

- `right` expands the window
- `left` shrinks the window

Whenever a candy enters the shared window, its frequency is reduced because we no longer keep it.

If a frequency becomes zero, that flavor disappears entirely from our remaining candies, so we remove the key from the dictionary.

Once the window size becomes exactly `k`, we evaluate the current number of unique flavors. Then we slide the window forward by restoring the candy at the left boundary.

Because every element enters and leaves the window at most once, the algorithm runs efficiently in linear time.

## Go Solution

```go
func shareCandies(candies []int, k int) int {
	frequency := make(map[int]int)

	for _, candy := range candies {
		frequency[candy]++
	}

	maxUnique := len(frequency)

	left := 0

	for right := 0; right < len(candies); right++ {
		flavor := candies[right]

		frequency[flavor]--

		if frequency[flavor] == 0 {
			delete(frequency, flavor)
		}

		if right-left+1 == k {
			if len(frequency) > maxUnique {
				maxUnique = len(frequency)
			}

			leftFlavor := candies[left]
			frequency[leftFlavor]++

			left++
		}
	}

	return maxUnique
}
```

The Go implementation follows the same logic as the Python solution. A `map[int]int` is used instead of Python's `Counter`.

One small difference is that Go requires explicit deletion using `delete(map, key)` when a frequency reaches zero.

Go slices naturally handle empty arrays, so no additional edge-case handling is needed.

Integer overflow is not a concern because all counts remain within the array size limit of `10^5`.

The extra space depends on the number of distinct flavors. In the worst case, every candy has a unique flavor, requiring `O(n)` space.

## Edge Cases

One important edge case occurs when `k = 0`. In this situation, no candies are shared, so we keep the entire array. A buggy implementation might accidentally remove candies anyway or mishandle the sliding window size. This solution handles it naturally because the frequency map already contains all candies initially, and the answer starts as the total number of distinct flavors.

Another critical case is when `k = len(candies)`. Here, every candy is given away, meaning we keep nothing. The sliding window eventually covers the entire array, reducing all frequencies to zero and leaving an empty frequency map. The implementation correctly returns `0`.

Arrays containing only one unique flavor can also expose logic errors. For example:

```
[1,1,1,1]
```

Even after removing some candies, the remaining candies may still contain the same flavor. The algorithm correctly maintains frequencies and only removes a flavor from the map when its count truly becomes zero.

Finally, empty arrays deserve careful handling. When `candies = []`, the frequency map starts empty and the loop never executes. The implementation correctly returns `0` without any special-case code.
