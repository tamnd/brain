## Solution

## Problem Understanding

This problem asks us to determine which topics are associated with each social media post based on keyword matching.

We are given two database tables:

The `Keywords` table maps words to topic IDs. A topic may have multiple keywords, and the same keyword may belong to multiple topics.

The `Posts` table contains posts and their textual content.

For every post, we must identify all topics whose keywords appear inside the post content. The matching rules are important:

1. Matching is case insensitive.
2. Only whole words count.
3. Partial matches are invalid.

The third rule is especially important. For example:

- `"war"` matches `"war"`
- `"war"` does not match `"warning"`
- `"vaccine"` does not match `"vaccinated"`

So we are effectively performing token-based matching, not substring matching.

If a post contains one or more matching keywords, we return all distinct matching topic IDs sorted in ascending order and joined by commas.

If no keywords match, we return `"Ambiguous!"`.

The result table should contain:

| Column | Meaning |
| --- | --- |
| `post_id` | The ID of the post |
| `topic` | Either comma-separated topic IDs or `"Ambiguous!"` |

The problem is categorized as a database problem, so the expected solution is SQL based.

### Important Observations

The post content consists only of English letters and spaces. This simplifies tokenization because we do not need to worry about punctuation.

Since matching must be case insensitive, both keywords and post content should be normalized to lowercase before comparison.

Whole-word matching means we cannot use a naive substring search like:

```
content LIKE '%war%'
```

because that would incorrectly match `"warning"`.

Instead, we need a technique that enforces word boundaries.

### Important Edge Cases

A keyword may belong to multiple topics. If a post contains that keyword, all associated topics must appear.

A post may contain multiple keywords for the same topic. The output must not contain duplicate topic IDs.

A post may contain no matching keywords at all. In that case, we must return `"Ambiguous!"`.

Posts may contain uppercase or lowercase variations of keywords, so normalization is required.

## Approaches

### Brute Force Approach

A straightforward approach is to compare every keyword against every post.

For each `(post, keyword)` pair:

1. Convert both strings to lowercase.
2. Split the post into words.
3. Check whether the keyword exists in the token list.
4. If it does, record the associated topic ID.

After processing all pairs, gather distinct topic IDs for each post, sort them, and concatenate them into a comma-separated string.

This approach is correct because every keyword is explicitly checked against every post. Since we only compare complete tokens, we avoid false substring matches.

However, this method becomes expensive because every post must be repeatedly tokenized and scanned for every keyword.

### Optimal Approach

The better solution is to use SQL joins with proper word-boundary matching.

The key insight is that we can normalize each post into a searchable format with explicit boundaries.

A common trick is:

```
CONCAT(' ', LOWER(content), ' ')
```

Then we search for:

```
CONCAT('% ', LOWER(word), ' %')
```

This guarantees whole-word matching because the keyword must be surrounded by spaces.

For example:

```
" stop the war and play "
```

contains:

```
" war "
```

but:

```
" warning "
```

does not contain:

```
" war "
```

After finding matching `(post_id, topic_id)` pairs, we:

1. Remove duplicates using `DISTINCT`
2. Group by `post_id`
3. Sort topic IDs
4. Use `GROUP_CONCAT`
5. Replace missing results with `"Ambiguous!"`

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(P × K × W) | O(W) | Repeatedly scans tokenized words |
| Optimal | O(P × K) | O(P × K) | Uses SQL joins and aggregation efficiently |

Where:

- `P` = number of posts
- `K` = number of keywords
- `W` = average number of words per post

## Algorithm Walkthrough

### Step 1: Normalize Post Content

For each post, convert the content to lowercase and add spaces at both ends.

Example:

```
"We Love Football"
```

becomes:

```
" we love football "
```

This allows us to enforce whole-word matching.

### Step 2: Normalize Keywords

Convert every keyword to lowercase as well.

This ensures case-insensitive matching.

### Step 3: Match Posts with Keywords

Join the `Posts` and `Keywords` tables using a condition like:

```
LOWER(content) LIKE CONCAT('% ', LOWER(word), ' %')
```

This checks whether the keyword appears as an independent word.

### Step 4: Collect Matching Topics

For every successful match, store:

```
(post_id, topic_id)
```

Since multiple keywords may map to the same topic, we use `DISTINCT`.

### Step 5: Aggregate Topic IDs

Group all matching topic IDs for the same post.

Sort them in ascending order and concatenate them with commas.

Example:

```
1,3
```

### Step 6: Handle Posts Without Matches

If a post has no matching topics, return:

```
Ambiguous!
```

This is typically done with `COALESCE`.

### Why it works

The algorithm works because every keyword is checked against every post using exact word-boundary matching. Lowercasing guarantees case insensitivity, and surrounding words with spaces prevents substring false positives. Aggregation collects all valid topics while removing duplicates and preserving ascending order.

## Python Solution

LeetCode database problems are solved using SQL queries. The following Python snippet simply returns the SQL query string in the format commonly used by LeetCode's database interface.

```python
class Solution:
    def solve(self):
        return """
        WITH matched_topics AS (
            SELECT DISTINCT
                p.post_id,
                k.topic_id
            FROM Posts p
            JOIN Keywords k
                ON CONCAT(' ', LOWER(p.content), ' ')
                   LIKE CONCAT('% ', LOWER(k.word), ' %')
        )

        SELECT
            p.post_id,
            COALESCE(
                GROUP_CONCAT(
                    DISTINCT mt.topic_id
                    ORDER BY mt.topic_id
                    SEPARATOR ','
                ),
                'Ambiguous!'
            ) AS topic
        FROM Posts p
        LEFT JOIN matched_topics mt
            ON p.post_id = mt.post_id
        GROUP BY p.post_id;
        """
```

### Implementation Explanation

The solution begins with a Common Table Expression named `matched_topics`.

Inside this CTE, we join `Posts` and `Keywords` using case-insensitive whole-word matching. The expression:

```
CONCAT(' ', LOWER(p.content), ' ')
```

adds word boundaries around the content.

The pattern:

```
CONCAT('% ', LOWER(k.word), ' %')
```

ensures the keyword is matched as a standalone word.

We use `DISTINCT` because a post may contain multiple keywords associated with the same topic.

After constructing all valid `(post_id, topic_id)` pairs, we perform a `LEFT JOIN` with the original `Posts` table. This guarantees that even posts without matches remain in the final result.

`GROUP_CONCAT` aggregates all topic IDs into a sorted comma-separated string.

Finally, `COALESCE` converts `NULL` values into `"Ambiguous!"`.

## Edge Cases

### Keywords Appearing as Substrings

One major source of bugs is substring matching. For example, `"war"` appears inside `"warning"`, but this should not count as a valid match.

A naive implementation using:

```
LIKE '%war%'
```

would produce incorrect results.

The implementation avoids this by adding spaces around both the content and the keyword pattern, ensuring only standalone words match.

### Multiple Keywords Mapping to the Same Topic

A post may contain several keywords that all belong to the same topic.

For example:

```
handball football
```

Both keywords map to topic `1`.

Without deduplication, the output could incorrectly become:

```
1,1
```

The query uses `DISTINCT` to guarantee every topic ID appears only once.

### Posts Without Any Matching Keywords

Some posts may contain no recognized keywords at all.

In these cases, aggregation functions return `NULL`.

Without explicit handling, the result would contain null values instead of the required string `"Ambiguous!"`.

The solution correctly handles this with:

```
COALESCE(..., 'Ambiguous!')
```

which replaces missing results with the required output string.
