## Solution

## Problem Understanding

The problem asks us to divide a string into consecutive groups, where every group must contain exactly `k` characters. We process the string from left to right, taking characters in chunks of size `k`.

If the final chunk does not contain enough characters to reach size `k`, we must append the given `fill` character until the group reaches the required size.

The input consists of:

- `s`, the original string
- `k`, the required size for each group
- `fill`, a single character used to pad the last group if necessary

The output is an array of strings, where each string has length exactly `k`.

For example, if `s = "abcdefghij"` and `k = 3`, we divide the string like this:

- `"abc"`
- `"def"`
- `"ghi"`
- `"j"`

The final group only contains one character, so we append the fill character twice:

- `"jxx"`

The final answer becomes:

```
["abc", "def", "ghi", "jxx"]
```

The constraints are small:

- `s.length <= 100`
- `k <= 100`

This tells us that performance is not a major concern. Even less efficient solutions would work comfortably within the limits. However, the problem is fundamentally linear because we only need to scan the string once.

Several edge cases are important:

- The string length may already be divisible by `k`, meaning no padding is needed.
- The string may be shorter than `k`, meaning the entire string becomes one padded group.
- `k` may equal `1`, meaning every character forms its own group.
- The final group may require multiple fill characters.

The problem guarantees valid lowercase English letters and positive values for `k`, so we do not need to handle invalid input.

## Approaches

### Brute Force Approach

A brute-force solution could simulate the grouping process character by character.

We could maintain a temporary string builder and append characters one at a time while iterating through `s`. Whenever the temporary group reaches size `k`, we add it to the result array and reset the builder. After processing the entire string, if the current group is incomplete, we repeatedly append the `fill` character until its size becomes `k`.

This approach is correct because it directly follows the problem statement. Every character is assigned to exactly one group, and the final incomplete group is padded appropriately.

However, this implementation performs unnecessary manual character management. It is more verbose and less elegant than necessary because modern languages already support substring slicing.

### Optimal Approach

The key observation is that the groups are simply consecutive substrings of length `k`.

Instead of building groups character by character, we can iterate through the string in steps of size `k`:

- Extract the substring from index `i` to `i + k`
- If the substring length is less than `k`, append enough `fill` characters
- Add the completed group to the result

This works efficiently because every character belongs to exactly one group, and slicing directly gives us the required partitions.

Since we traverse the string once, the algorithm runs in linear time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Builds groups character by character |
| Optimal | O(n) | O(n) | Uses substring slicing in chunks of size `k` |

## Algorithm Walkthrough

1. Initialize an empty result list that will store all groups.
2. Iterate through the string using an index that increases by `k` each time. This ensures every iteration processes exactly one group-sized segment.
3. For each position `i`, extract the substring:

```
s[i:i+k]
```

This gives the next group candidate.
4. Check whether the extracted substring has fewer than `k` characters. This only happens for the final group when the string length is not divisible by `k`.
5. If the group is too short, calculate how many characters are missing:

```
k - len(group)
```
6. Append the `fill` character that many times to complete the group.
7. Add the completed group to the result list.
8. Continue until the entire string has been processed.
9. Return the result list.

### Why it works

The algorithm always processes the string in non-overlapping chunks of size `k`. Every character belongs to exactly one chunk because the index advances by `k` after each iteration. If the final chunk is incomplete, padding guarantees its length becomes exactly `k`. Therefore, the resulting array satisfies all conditions required by the problem.

## Python Solution

```python
from typing import List

class Solution:
    def divideString(self, s: str, k: int, fill: str) -> List[str]:
        groups = []

        for i in range(0, len(s), k):
            group = s[i:i + k]

            if len(group) < k:
                group += fill * (k - len(group))

            groups.append(group)

        return groups
```

The implementation begins by creating an empty list called `groups` to store the final answer.

The loop iterates through the string in increments of `k`. At each iteration, the slice `s[i:i+k]` extracts the next group.

If the extracted group is shorter than `k`, the code appends the required number of `fill` characters using string multiplication:

```python
fill * (k - len(group))
```

Finally, each completed group is appended to the result list.

The algorithm directly mirrors the logical steps described earlier, making the implementation concise and easy to understand.

## Go Solution

```go
func divideString(s string, k int, fill byte) []string {
    result := []string{}

    for i := 0; i < len(s); i += k {
        end := i + k

        if end > len(s) {
            end = len(s)
        }

        group := s[i:end]

        for len(group) < k {
            group += string(fill)
        }

        result = append(result, group)
    }

    return result
}
```

The Go implementation follows the same overall logic as the Python version, but there are a few language-specific differences.

Go does not support Python-style slicing that automatically clamps indices, so we explicitly ensure that `end` does not exceed `len(s)`.

Padding is performed using a loop because Go does not support multiplying strings by integers like Python does.

The result is stored in a slice of strings, which dynamically grows as groups are appended.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each character is processed once |
| Space | O(n) | The output stores all characters from the input plus possible padding |

The algorithm visits the string in chunks of size `k`, but every character is still processed only once overall. The padding operation adds at most `k - 1` extra characters, which does not change the linear complexity.

The output itself requires linear space because all groups together contain essentially the same total number of characters as the original string, plus any padding.

## Edge Cases

One important edge case occurs when the string length is already divisible by `k`. In this situation, the algorithm must avoid adding unnecessary fill characters. The implementation handles this naturally because the padding step only runs when the extracted group length is smaller than `k`.

Another important case is when the string length is smaller than `k`. A careless implementation might incorrectly return multiple groups or fail to add enough padding. Here, the algorithm extracts the entire string as one group and appends exactly the required number of fill characters.

A third edge case occurs when `k = 1`. In this case, every character should become its own group, and padding should never occur. Since the loop advances one character at a time and every extracted substring already has length `1`, the implementation works correctly without requiring special handling.

A final subtle case involves multiple padding characters being required. For example, if the final group contains only one character and `k = 5`, four fill characters must be appended. The implementation correctly computes the missing amount using:

```
k - len(group)
```

This guarantees the final group always reaches the required size exactly.
