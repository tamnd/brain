## Solution

## Problem Understanding

The problem asks us to determine all the recipes that can be made given a set of initial supplies. Each recipe has a list of ingredients, and some ingredients may themselves be recipes. The goal is to output all recipes that can be fully constructed using the supplies and other recipes that we can create.

The input consists of a list of recipe names, a corresponding list of ingredient lists, and a list of initial supplies. Each recipe is unique, each supply is unique, and ingredients for a recipe do not contain duplicates. The constraints indicate that the input sizes are manageable (`n <= 100` and `ingredients[i].length <= 100`), but there could be dependencies between recipes that form chains or cycles. We are guaranteed that supplies and recipe names combined are unique, which simplifies lookups.

Important edge cases include a recipe depending on another recipe that is not initially available, circular dependencies between recipes, and recipes that require only initial supplies.

## Approaches

The **brute-force approach** is to attempt to create each recipe by checking whether all its ingredients are either in the supplies or can themselves be created. This could be done recursively, but a naive recursive implementation risks recomputing the same sub-recipes multiple times and could fail if there are circular dependencies. Its time complexity is exponential in the worst case because for each recipe you may recursively traverse many dependent recipes repeatedly.

The **optimal approach** recognizes this as a **topological sort problem**. If we model each recipe as a node in a graph where edges point from ingredients to the recipe that uses them, then a recipe can be made when all its dependencies are satisfied. We can use a queue to perform a **BFS-like process**, starting with the initial supplies and progressively marking recipes as creatable once all their ingredients are available. This handles chains of dependencies efficiently and avoids recomputation.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * 2^n) | O(n) | Recursively checks all dependencies, may recompute sub-recipes repeatedly |
| Optimal | O(n * m) | O(n * m) | Topological sort using BFS, where n = number of recipes, m = max ingredients per recipe |

## Algorithm Walkthrough

1. Create a mapping from each recipe to its ingredient list. This allows fast lookup of what each recipe requires.
2. Create a dictionary to track the **in-degree** of each recipe, representing how many ingredients are still missing before it can be made.
3. Create a dictionary mapping each ingredient to a list of recipes that depend on it. This allows us to propagate availability efficiently.
4. Initialize a queue with all initial supplies since these are immediately available.
5. While the queue is not empty, pop an available item. For each recipe that depends on this item, decrement its in-degree. If the in-degree reaches zero, it means all ingredients are available, so the recipe can now be made. Add this recipe to the queue and to the result list.
6. Return the list of recipes that were successfully created.

**Why it works:** By keeping track of in-degrees and propagating availability through a BFS, we ensure that recipes are only marked as creatable once all their ingredients are available. This approach naturally handles dependency chains and prevents infinite loops due to cycles because a recipe is only processed once its in-degree reaches zero.

## Python Solution

```python
from collections import deque, defaultdict
from typing import List

class Solution:
    def findAllRecipes(self, recipes: List[str], ingredients: List[List[str]], supplies: List[str]) -> List[str]:
        # Map each recipe to its remaining number of ingredients required
        in_degree = {recipe: len(ingredients[i]) for i, recipe in enumerate(recipes)}
        # Map each ingredient to the list of recipes that depend on it
        ingredient_to_recipes = defaultdict(list)
        for i, recipe in enumerate(recipes):
            for ing in ingredients[i]:
                ingredient_to_recipes[ing].append(recipe)
        # Initialize queue with initial supplies
        queue = deque(supplies)
        result = []
        # Process queue
        while queue:
            item = queue.popleft()
            for recipe in ingredient_to_recipes[item]:
                in_degree[recipe] -= 1
                if in_degree[recipe] == 0:
                    queue.append(recipe)
                    result.append(recipe)
        return result
```

The Python code creates in-degree counts for each recipe, maps ingredients to recipes that need them, and processes a queue of available items. When a recipe's in-degree reaches zero, it is added to the queue and result list, ensuring we only produce it when all dependencies are met.

## Go Solution

```go
package main

func findAllRecipes(recipes []string, ingredients [][]string, supplies []string) []string {
    inDegree := make(map[string]int)
    ingredientToRecipes := make(map[string][]string)
    
    for i, recipe := range recipes {
        inDegree[recipe] = len(ingredients[i])
        for _, ing := range ingredients[i] {
            ingredientToRecipes[ing] = append(ingredientToRecipes[ing], recipe)
        }
    }
    
    queue := make([]string, len(supplies))
    copy(queue, supplies)
    res := []string{}
    
    for head := 0; head < len(queue); head++ {
        item := queue[head]
        for _, recipe := range ingredientToRecipes[item] {
            inDegree[recipe]--
            if inDegree[recipe] == 0 {
                queue = append(queue, recipe)
                res = append(res, recipe)
            }
        }
    }
    
    return res
}
```

In Go, we use slices to implement the queue, iterating with a pointer to simulate deque behavior. Maps are used for in-degrees and reverse ingredient-to-recipe mappings. The logic mirrors the Python implementation closely.

## Worked Examples

**Example 1:** `recipes = ["bread"], ingredients = [["yeast","flour"]], supplies = ["yeast","flour","corn"]`

Queue starts as `["yeast","flour","corn"]`.

Processing "yeast" and "flour" reduces `bread` in-degree to 0, so `bread` is added to queue and result. Final result is `["bread"]`.

**Example 2:** `recipes = ["bread","sandwich"], ingredients = [["yeast","flour"],["bread","meat"]], supplies = ["yeast","flour","meat"]`

Queue starts as `["yeast","flour","meat"]`.

"yeast" and "flour" reduce `bread` in-degree to 0 → add `bread`.

Processing `bread` reduces `sandwich` in-degree to 0 → add `sandwich`.

Final result is `["bread","sandwich"]`.

**Example 3:** `recipes = ["bread","sandwich","burger"], ingredients = [["yeast","flour"],["bread","meat"],["sandwich","meat","bread"]], supplies = ["yeast","flour","meat"]`

Queue starts as `["yeast","flour","meat"]`.

"yeast" and "flour" → `bread` in-degree 0 → add `bread`.

"bread" and "meat" → `sandwich` in-degree 0 → add `sandwich`.

"bread","sandwich","meat" → `burger` in-degree 0 → add `burger`.

Final result is `["bread","sandwich","burger"]`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n * m) | We process each ingredient of each recipe once, where n = number of recipes, m = max ingredients per recipe |
| Space | O(n * m) | Storage for in-degree map, ingredient-to-recipe map, queue, and result list |

The BFS approach ensures each recipe is visited at most once when its in-degree becomes zero, so we do not revisit recipes unnecessarily. The data structures scale linearly with the number of recipes and ingredients.
