**Difficulty:** 🔴 Hard

## Solution

## Problem Understanding

The problem gives us two arrays, `nums1` and `nums2`, each containing every integer from `0` to `n - 1` exactly once. In other words, both arrays are permutations of the same set of values.

We need to count how many triplets `(x, y, z)` satisfy the following condition:

- In `nums1`, the positions of `x`, `y`, and `z` appear in increasing order
- In `nums2`, the positions of `x`, `y`, and `z` also appear in increasing order

Formally:

- `pos1[x] < pos1[y] < pos1[z]`
- `pos2[x] < pos2[y] < pos2[z]`

The important detail is that we are comparing relative ordering, not actual numeric values.

For example, suppose:

```
nums1 = [2,0,1,3]
nums2 = [0,1,2,3]
```

The triplet `(0,1,3)` is valid because:

- In `nums1`, positions are `1 < 2 < 3`
- In `nums2`, positions are `0 < 1 < 3`

So the ordering is preserved in both arrays.

The constraints are extremely important:

- `n` can be as large as `100000`
- A naive cubic solution would require roughly `10^15` operations in the worst case

That immediately tells us brute force is impossible.

The permutation property is also very useful. Since every value appears exactly once in each array, every value maps to a unique position. This lets us transform the problem into a sequence ordering problem.

Some important edge cases include:

- Arrays where no triplets are valid
- Arrays where every triplet is valid
- Very large inputs where efficiency matters
- Inputs with reversed ordering, which minimize valid triplets
- Inputs with nearly identical ordering, which maximize valid triplets

The problem guarantees that both arrays are valid permutations, so we never need to handle duplicates or missing values.

## Approaches

### Brute Force

The most direct solution is to generate every possible triplet `(x, y, z)` and check whether the ordering condition holds in both arrays.

We first compute the positions of every value in both arrays. Then for every combination of three distinct values:

- Check whether their positions are increasing in `nums1`
- Check whether their positions are increasing in `nums2`

If both conditions hold, increment the answer.

This works because it explicitly verifies the exact definition of a good triplet.

However, there are `O(n^3)` triplets. With `n = 100000`, this is completely infeasible.

### Key Insight

The crucial observation is that the actual values do not matter. Only their relative positions matter.

We can transform the problem into:

> Count increasing subsequences of length 3.

First, map every value to its index in `nums2`.

Then convert `nums1` into a new array:

```
transformed[i] = position of nums1[i] in nums2
```

Now the problem becomes:

> Count triples `i < j < k` such that:
> 
> 
> 
> 
> `transformed[i] < transformed[j] < transformed[k]`

This is now a classic increasing subsequence counting problem.

For every middle element `j`, we need:

- How many smaller elements exist on the left
- How many larger elements exist on the right

If:

- `left_smaller[j] = count of values < transformed[j] before j`
- `right_larger[j] = count of values > transformed[j] after j`

Then every combination forms a valid triplet:

```
answer += left_smaller[j] * right_larger[j]
```

To compute these efficiently, we use a Binary Indexed Tree, also called a Fenwick Tree.

## Algorithm Walkthrough

### Step 1: Map values to positions in `nums2`

We create an array or hash map:

```
position[value] = index in nums2
```

This allows constant time lookup of where each value appears in `nums2`.

For example:

```
nums2 = [0,1,2,3]

position:
0 -> 0
1 -> 1
2 -> 2
3 -> 3
```

### Step 2: Transform `nums1`

Replace every value in `nums1` with its position in `nums2`.

Example:

```
nums1 = [2,0,1,3]

transformed = [2,0,1,3]
```

Now the problem becomes finding increasing triplets in this transformed array.

Why does this work?

Because:

- The order in `nums1` is already fixed by the array indices
- The transformed values encode the order in `nums2`

So increasing subsequences in `transformed` correspond exactly to good triplets.

### Step 3: Compute left smaller counts

For every index `j`, we want:

```
How many transformed[i] < transformed[j] exist for i < j?
```

We process the array from left to right.

A Fenwick Tree stores how many values we have already seen.

For each value:

1. Query how many smaller values exist
2. Store that count
3. Insert the current value into the tree

This gives:

```
left_smaller[j]
```

### Step 4: Compute right larger counts

Now process from right to left.

For every index `j`, we want:

```
How many transformed[k] > transformed[j] exist for k > j?
```

Again use a Fenwick Tree.

This time:

1. Query how many values greater than current already exist
2. Store that count
3. Insert current value

This gives:

```
right_larger[j]
```

### Step 5: Count all triplets

Each index `j` acts as the middle element of a triplet.

The number of valid triplets centered at `j` is:

```
left_smaller[j] * right_larger[j]
```

Sum this across all indices.

### Why it works

For a fixed middle element `j`:

- Any smaller element before it can serve as the left endpoint
- Any larger element after it can serve as the right endpoint

Because the transformed array encodes the ordering in `nums2`, every such combination corresponds to a valid triplet in both arrays.

The Fenwick Tree efficiently maintains prefix frequency counts, allowing us to compute these quantities in logarithmic time.

## Python Solution

```python
from typing import List

class FenwickTree:
    def __init__(self, size: int):
        self.size = size
        self.tree = [0] * (size + 1)

    def update(self, index: int, delta: int) -> None:
        index += 1

        while index <= self.size:
            self.tree[index] += delta
            index += index & -index

    def query(self, index: int) -> int:
        index += 1
        result = 0

        while index > 0:
            result += self.tree[index]
            index -= index & -index

        return result

class Solution:
    def goodTriplets(self, nums1: List[int], nums2: List[int]) -> int:
        n = len(nums1)

        position = [0] * n

        for index, value in enumerate(nums2):
            position[value] = index

        transformed = [position[value] for value in nums1]

        left_smaller = [0] * n
        fenwick = FenwickTree(n)

        for i, value in enumerate(transformed):
            left_smaller[i] = fenwick.query(value - 1)
            fenwick.update(value, 1)

        right_larger = [0] * n
        fenwick = FenwickTree(n)

        for i in range(n - 1, -1, -1):
            value = transformed[i]

            total_seen = fenwick.query(n - 1)
            less_or_equal = fenwick.query(value)

            right_larger[i] = total_seen - less_or_equal

            fenwick.update(value, 1)

        answer = 0

        for i in range(n):
            answer += left_smaller[i] * right_larger[i]

        return answer
```

The implementation begins by constructing the `position` array, which maps each value to its index in `nums2`. This lets us transform `nums1` into a sequence representing the ordering inside `nums2`.

The `FenwickTree` class supports two operations efficiently:

- `update(index, delta)` inserts a value
- `query(index)` returns the prefix sum from `0` to `index`

The first traversal computes `left_smaller`. Before inserting the current value into the tree, we query how many smaller values already exist.

The second traversal computes `right_larger`. Since we move from right to left, the Fenwick Tree contains all values to the right of the current index. We compute how many of them are larger than the current value.

Finally, each index contributes:

```python
left_smaller[i] * right_larger[i]
```

to the answer.

## Go Solution

```go
package main

type FenwickTree struct {
	tree []int
	size int
}

func NewFenwickTree(size int) *FenwickTree {
	return &FenwickTree{
		tree: make([]int, size+1),
		size: size,
	}
}

func (f *FenwickTree) Update(index int, delta int) {
	index++

	for index <= f.size {
		f.tree[index] += delta
		index += index & -index
	}
}

func (f *FenwickTree) Query(index int) int {
	index++

	result := 0

	for index > 0 {
		result += f.tree[index]
		index -= index & -index
	}

	return result
}

func goodTriplets(nums1 []int, nums2 []int) int64 {
	n := len(nums1)

	position := make([]int, n)

	for index, value := range nums2 {
		position[value] = index
	}

	transformed := make([]int, n)

	for i, value := range nums1 {
		transformed[i] = position[value]
	}

	leftSmaller := make([]int, n)

	fenwick := NewFenwickTree(n)

	for i, value := range transformed {
		leftSmaller[i] = fenwick.Query(value - 1)
		fenwick.Update(value, 1)
	}

	rightLarger := make([]int, n)

	fenwick = NewFenwickTree(n)

	for i := n - 1; i >= 0; i-- {
		value := transformed[i]

		totalSeen := fenwick.Query(n - 1)
		lessOrEqual := fenwick.Query(value)

		rightLarger[i] = totalSeen - lessOrEqual

		fenwick.Update(value, 1)
	}

	var answer int64 = 0

	for i := 0; i < n; i++ {
		answer += int64(leftSmaller[i] * rightLarger[i])
	}

	return answer
}
```

The Go implementation follows the same algorithmic structure as the Python version.

One important difference is integer overflow handling. The number of triplets can be large, potentially close to `n choose 3`, so the final answer must use `int64`.

The Fenwick Tree uses slices for storage, and methods are attached to a struct for clarity and encapsulation.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | Each Fenwick Tree operation takes O(log n), and we perform O(n) operations |
| Space | O(n) | Arrays and Fenwick Trees each require linear storage |

The dominant cost comes from Fenwick Tree updates and prefix sum queries. Since each operation is logarithmic and we perform a constant number per element, the overall runtime becomes `O(n log n)`.

The memory usage is linear because we store:

- The transformed array
- Left and right count arrays
- Fenwick Tree storage
- Position mapping

All of these scale proportionally with `n`.

## Edge Cases

### Completely Reversed Order

A common bug occurs when implementations accidentally count triplets based on numeric values instead of relative positions.

Example:

```
nums1 = [0,1,2,3]
nums2 = [3,2,1,0]
```

No triplet preserves ordering in both arrays, so the answer must be zero.

The transformed array becomes strictly decreasing, meaning:

```
left_smaller > 0
right_larger = 0
```

or vice versa, so no contributions are added.

### All Triplets Are Valid

When both arrays are identical:

```
nums1 = nums2
```

every increasing index triplet is valid.

For `n = 5`, the answer becomes:

```
C(5,3) = 10
```

This case is important because it produces the maximum possible answer size. The implementation correctly uses large integer types:

- Python integers are unbounded
- Go uses `int64`

to avoid overflow.

### Very Large Inputs

With `n = 100000`, brute force solutions are impossible.

A common mistake is using nested loops for left and right counting, which leads to `O(n²)` complexity and time limit exceeded errors.

The Fenwick Tree guarantees:

- `O(log n)` updates
- `O(log n)` queries

so the full algorithm remains efficient even at maximum constraints.
