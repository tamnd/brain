# LeetCode 2165 - Smallest Value of the Rearranged Number

## Problem Understanding

The problem gives us an integer `num`, which may be positive, negative, or zero. Our task is to rearrange its digits so that the resulting number is as small as possible while preserving the original sign.

For positive numbers, "smallest" means the numerically smallest positive value. For example, rearranging `310` into `103` produces the minimum possible valid number because numbers like `013` are invalid due to leading zeros.

For negative numbers, the interpretation is slightly different. Since more negative values are smaller, we actually want to maximize the absolute value of the number while keeping the negative sign. For example, among all rearrangements of `7605`, the arrangement `7650` gives the largest magnitude, so `-7650` is the smallest possible value.

The problem also states that the rearranged number cannot contain leading zeros. This restriction is important because simply sorting digits in ascending order for positive numbers could place zeros at the front, creating an invalid representation.

The constraints are relatively small:

```
-10^15 <= num <= 10^15
```

This means the number contains at most 16 digits, including possible zeros. Because the digit count is small, sorting digits is completely feasible.

Several edge cases are important:

- `num = 0`
- Numbers containing multiple zeros such as `100200`
- Negative numbers
- Numbers where all digits are identical
- Numbers where the smallest digit is zero
- Very large values near the constraint boundaries

A naive implementation can easily fail by accidentally introducing leading zeros or by handling negative numbers incorrectly.

## Approaches

### Brute Force Approach

A brute force solution would generate every possible permutation of the digits, convert each permutation into a number, discard any arrangement with leading zeros, and then select the minimum valid value.

For example, for `310`, we would generate:

```
013, 031, 103, 130, 301, 310
```

After removing invalid leading-zero arrangements, we would choose `103`.

This approach is correct because it checks every possible arrangement. However, it is far too slow. If the number contains `n` digits, there are `n!` permutations. Even with only 15 digits, this becomes computationally impossible.

### Optimal Approach

The key insight is that we do not need to test every permutation. We can construct the optimal arrangement directly using sorting.

For positive numbers:

- To minimize the value, digits should appear in ascending order.
- However, leading zeros are forbidden.
- Therefore:

- Sort digits ascending.
- Place the smallest non-zero digit first.
- Place all zeros immediately after it.
- Append the remaining digits.

For negative numbers:

- A more negative number is smaller.
- Therefore, we want the largest possible magnitude.
- This means sorting digits in descending order and then applying the negative sign.

This greedy strategy works because digit positions have positional value. Smaller digits placed earlier reduce positive numbers, while larger digits placed earlier increase magnitude for negative numbers.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n! × n) | O(n!) | Generates every permutation |
| Optimal | O(n log n) | O(n) | Sorts digits and constructs answer greedily |

## Algorithm Walkthrough

1. Check whether the number is zero.

If `num == 0`, return `0` immediately. Rearranging zero still produces zero.
2. Determine whether the number is negative.

Store whether `num < 0`. Then work with the absolute value of the number so digit extraction becomes easier.
3. Convert the number into a list of digits.

Convert the absolute value into a string, then into a list of characters. This makes sorting straightforward.
4. Handle the positive number case.

If the original number was positive:

- Sort digits in ascending order.
- Find the first non-zero digit.
- Move that digit to the front.
- Append all remaining digits afterward.

This guarantees:

- The leading digit is as small as possible.
- No leading zeros appear.
- Remaining digits stay as small as possible.
5. Handle the negative number case.

If the original number was negative:

- Sort digits in descending order.
- Join them into the largest possible magnitude.
- Apply the negative sign.
6. Convert the resulting digit sequence back into an integer.

Join the digits into a string and convert it into an integer before returning.

### Why it works

For positive numbers, the leftmost digits contribute the most to the value. Therefore, placing the smallest available non-zero digit first minimizes the number while avoiding invalid leading zeros. After that, placing remaining digits in ascending order keeps every subsequent position as small as possible.

For negative numbers, a smaller numerical value means a larger magnitude with the negative sign. Sorting digits in descending order maximizes the magnitude, producing the smallest overall signed value.

## Python Solution

```python
class Solution:
    def smallestNumber(self, num: int) -> int:
        if num == 0:
            return 0

        is_negative = num < 0
        digits = list(str(abs(num)))

        if is_negative:
            digits.sort(reverse=True)
            return -int("".join(digits))

        digits.sort()

        first_non_zero = 0
        while digits[first_non_zero] == '0':
            first_non_zero += 1

        digits[0], digits[first_non_zero] = (
            digits[first_non_zero],
            digits[0],
        )

        return int("".join(digits))
```

The implementation begins by handling the special case where the number is zero. This prevents unnecessary processing and avoids issues with digit manipulation.

Next, the code determines whether the original number is negative and extracts the digits from the absolute value. Using strings simplifies sorting because each digit becomes an individual sortable element.

For negative numbers, the digits are sorted in descending order. This creates the largest possible magnitude, which becomes the smallest numerical value once the negative sign is restored.

For positive numbers, the digits are sorted ascending. However, this may place zeros at the front. The code scans for the first non-zero digit and swaps it into the leading position. This preserves the smallest possible arrangement while satisfying the no-leading-zero requirement.

Finally, the digits are joined back into a string and converted into an integer.

## Go Solution

```go
package main

import (
	"sort"
	"strconv"
)

func smallestNumber(num int64) int64 {
	if num == 0 {
		return 0
	}

	isNegative := num < 0

	if isNegative {
		num = -num
	}

	digits := []byte(strconv.FormatInt(num, 10))

	if isNegative {
		sort.Slice(digits, func(i, j int) bool {
			return digits[i] > digits[j]
		})

		result, _ := strconv.ParseInt(string(digits), 10, 64)
		return -result
	}

	sort.Slice(digits, func(i, j int) bool {
		return digits[i] < digits[j]
	})

	firstNonZero := 0
	for digits[firstNonZero] == '0' {
		firstNonZero++
	}

	digits[0], digits[firstNonZero] =
		digits[firstNonZero], digits[0]

	result, _ := strconv.ParseInt(string(digits), 10, 64)

	return result
}
```

The Go implementation follows the same logic as the Python version but uses Go-specific utilities.

Instead of Python's built-in sorting methods, Go uses `sort.Slice` with custom comparison functions. Digits are stored as a byte slice because strings in Go are immutable.

The conversion between integers and strings uses the `strconv` package. Since the constraints fit within `int64`, `strconv.ParseInt` safely reconstructs the final number.

Unlike Python, Go requires explicit handling of integer types and conversions, but the overall algorithm remains identical.

## Edge Cases

### Zero Input

When the input is `0`, rearranging its digits still produces `0`. A careless implementation might attempt to sort digits and search for a non-zero digit, which would fail because none exists. The implementation handles this immediately with an early return.

### Positive Numbers with Leading Zeros After Sorting

Inputs like `1002` become `0012` after sorting. This representation is invalid because integers cannot contain leading zeros. The algorithm fixes this by locating the first non-zero digit and swapping it into the front position, producing `1002` instead.

### Negative Numbers

Negative numbers are easy to mishandle because minimizing a negative value means maximizing its magnitude. For example, sorting `-7605` in ascending order would produce `-0567`, which equals `-567` and is not the smallest possible value. The implementation correctly sorts digits in descending order for negative numbers, producing `-7650`.

### Numbers with Repeated Digits

Inputs such as `1111` or `1000000` contain repeated digits. Some permutation-based approaches can generate duplicate arrangements unnecessarily. The sorting-based solution avoids this issue entirely because it constructs the answer directly instead of enumerating permutations.
