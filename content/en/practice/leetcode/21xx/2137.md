## Solution

## Problem Understanding

The problem asks us to equalize the amount of water in a collection of buckets while accounting for spillage. Each bucket has an initial amount of water represented by the array `buckets`. You can pour water from one bucket to another, but a fixed percentage of any poured water is lost during the transfer, represented by `loss`. The goal is to determine the **maximum possible water level** that all buckets can reach after redistributing water optimally, considering the spillage.

The input consists of an array of integers, `buckets`, with length up to `10^5` and values up to `10^5`, and an integer `loss` between 0 and 99. The output is a floating-point number representing the highest water level that can be achieved in all buckets, accurate to within `10^-5`.

Key insights include recognizing that not all water can be perfectly transferred due to loss, and the solution requires checking feasible water levels rather than trying every possible transfer explicitly.

Important edge cases include a single bucket (trivial case), all buckets already equal (no operations needed), `loss = 0` (no water lost), and `loss = 99` (most water is lost during transfer).

## Approaches

A brute-force approach would attempt to simulate all possible pour operations between buckets, adjusting water levels iteratively until no further equalization is possible. This is computationally infeasible because the number of operations could be enormous for large `n` and high initial water volumes.

The optimal approach uses a **binary search** on the potential maximum water level. The key insight is that for a given target level `x`, we can check if it is achievable by calculating the total water that must be poured from buckets above `x` and comparing it to the water needed by buckets below `x`, considering the loss. This reduces the problem to a series of feasibility checks that are efficient and precise.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^2) | O(1) | Simulate every possible pour, very slow for n = 10^5 |
| Optimal | O(n log(max_bucket)) | O(1) | Binary search on target water level with linear feasibility check |

## Algorithm Walkthrough

1. Identify the range of possible water levels. The minimum is `0`, and the maximum is `max(buckets)` since no bucket can have more water than the largest initial bucket.
2. Perform a binary search on the water level `target` within this range. Maintain a high precision, e.g., `10^-6`, as the stopping criterion.
3. For each candidate `target` level, calculate the total excess water from buckets above `target` and the total water needed for buckets below `target`. When computing excess, only the portion that reaches the target (after loss) counts.
4. If the excess after spillage is sufficient to satisfy the buckets that are below `target`, then the target level is feasible; otherwise, it is not.
5. Adjust the binary search bounds based on feasibility: increase `low` if feasible, decrease `high` if not.
6. Continue the search until the precision criterion is met. The resulting `low` or `high` will be the maximum achievable equal water level.

Why it works: The algorithm maintains the invariant that any target water level checked can be precisely validated for feasibility by linear scanning of buckets. Binary search ensures we efficiently converge on the maximum achievable value.

## Python Solution

```python
from typing import List

class Solution:
    def equalizeWater(self, buckets: List[int], loss: int) -> float:
        def can_achieve(target: float) -> bool:
            excess = 0.0
            needed = 0.0
            for water in buckets:
                if water > target:
                    excess += (water - target) * (1 - loss / 100)
                else:
                    needed += target - water
            return excess >= needed

        low, high = 0.0, max(buckets)
        precision = 1e-6
        while high - low > precision:
            mid = (low + high) / 2
            if can_achieve(mid):
                low = mid
            else:
                high = mid
        return low
```

The `can_achieve` helper function iterates through all buckets, summing the transferable excess water and the water needed. Binary search then adjusts the candidate water level based on feasibility. Using `precision = 1e-6` ensures the result meets the required `10^-5` accuracy.

## Go Solution

```go
func equalizeWater(buckets []int, loss int) float64 {
    canAchieve := func(target float64) bool {
        excess, needed := 0.0, 0.0
        for _, water := range buckets {
            w := float64(water)
            if w > target {
                excess += (w - target) * (1 - float64(loss)/100)
            } else {
                needed += target - w
            }
        }
        return excess >= needed
    }

    low, high := 0.0, 0.0
    for _, water := range buckets {
        if float64(water) > high {
            high = float64(water)
        }
    }

    precision := 1e-6
    for high-low > precision {
        mid := (low + high) / 2
        if canAchieve(mid) {
            low = mid
        } else {
            high = mid
        }
    }
    return low
}
```

The Go solution follows the same logic as Python. Note that `float64` conversions are necessary to handle the division and loss percentage accurately. The slice iteration ensures O(n) traversal for each binary search step.

## Worked Examples

**Example 1:** `buckets = [1,2,7], loss = 80`

1. Set low = 0, high = 7. Binary search begins.
2. Test mid = 3. `can_achieve(3)` computes excess = 7-3 = 4, spilled = 4*0.8 = 3.2 lost, received = 0.8? Total needed for buckets below 3: 2 (bucket[0]) + 1 (bucket[1]) = 2. Feasible.
3. Continue narrowing the range; final result converges to 2.0.

**Example 2:** `buckets = [2,4,6], loss = 50`

1. low = 0, high = 6, mid = 3
2. `can_achieve(3)` computes: excess = (4-3)*0.5 + (6-3)*0.5 = 0.5 + 1.5 = 2. Needed = 1 (bucket[0]) = 1. Feasible.
3. Binary search continues, final result = 3.5.

**Example 3:** `buckets = [3,3,3,3], loss = 40`

1. All equal, any mid <= 3 is feasible.
2. Result converges immediately to 3.0.
