# LeetCode 2178, Maximum Split of Positive Even Integers

## Problem Understanding

The problem gives us a single integer called `finalSum`, and asks us to split it into a collection of unique positive even integers whose total sum equals `finalSum`.

The important requirement is not just to find any valid split, but to maximize the number of integers in the split.

For example, if `finalSum = 12`, several valid decompositions exist:

- `12`
- `2 + 10`
- `4 + 8`
- `2 + 4 + 6`

Among these, `2 + 4 + 6` uses the largest number of distinct even integers, so it is the correct answer.

The integers must satisfy three conditions simultaneously:

1. Every number must be positive.
2. Every number must be even.
3. Every number must be unique.

The input size is relatively small in this problem statement, but the intended greedy solution works efficiently even for very large values. The important observation is mathematical rather than dependent on brute force search.

One critical constraint immediately stands out: if `finalSum` is odd, then the answer is impossible.

This is because the sum of even numbers is always even. No collection of even integers can add up to an odd number.

Some important edge cases include:

- `finalSum` is odd, such as `7`
- `finalSum` is very small, such as `2`
- `finalSum` is itself an even number but cannot support many distinct values
- The remaining unused sum at the end duplicates an already chosen number

The last case is especially important because uniqueness must always be preserved.

## Approaches

### Brute Force Approach

A brute force solution would try every possible subset of positive even integers whose sum equals `finalSum`.

We could generate all combinations recursively using backtracking. At each step, we either include or exclude an even number. Whenever the running sum reaches `finalSum`, we check whether the current combination contains more integers than the best solution found so far.

This approach is correct because it exhaustively explores all valid possibilities. Eventually, it will discover the optimal split.

However, the number of possible subsets grows exponentially. Even though the input constraint in the statement is modest, this approach becomes inefficient quickly and is unnecessary because the problem has a strong greedy structure.

### Key Insight

To maximize the number of unique even integers, we should use the smallest possible even numbers first.

Why?

Suppose we start with large numbers. Large values consume too much of the total sum, leaving less room for additional integers.

Using the smallest distinct even numbers allows us to fit as many numbers as possible into the sum.

So we greedily take:

- `2`
- `4`
- `6`
- `8`
- and so on

We continue until adding the next even number would exceed `finalSum`.

At that point, there may still be some leftover value. Instead of creating a duplicate number, we simply add the leftover amount to the last chosen number.

This preserves uniqueness while ensuring the total sum becomes exactly `finalSum`.

For example:

- `finalSum = 28`
- Take `2`, remaining `26`
- Take `4`, remaining `22`
- Take `6`, remaining `16`
- Take `8`, remaining `8`

Now the next candidate would be `10`, which is too large.

The leftover is `8`, so we add it to the last chosen number:

- Last number becomes `8 + 8 = 16`

Final result:

- `[2, 4, 6, 16]`

All numbers remain unique and even.

## Algorithm Walkthrough

1. First, check whether `finalSum` is odd.

If it is odd, immediately return an empty list because an odd number cannot be represented as a sum of even integers.
2. Create an empty result list.

This list will store the chosen unique even integers.
3. Start with the smallest positive even integer, `2`.

This is the best choice because smaller numbers allow more integers to fit into the total sum.
4. While the current even number is less than or equal to the remaining sum:

- Add the current even number to the result list.
- Subtract it from `finalSum`.
- Increase the current even number by `2`.
5. Eventually, the next even number will be too large to fit.

At this point, there may still be some leftover value remaining in `finalSum`.
6. Add the leftover amount to the last element of the result list.

This step ensures:

- The total sum becomes exactly correct.
- All integers remain even.
- No duplicates are introduced.
7. Return the result list.

### Why it works

The greedy strategy works because choosing smaller distinct even integers always maximizes how many numbers can fit into the total sum.

If we replaced a smaller chosen number with a larger one, we would consume more of the available sum and reduce the number of integers we could include.

The final leftover adjustment is safe because the leftover is always even, and adding it to the largest chosen number cannot create duplicates. The modified last number becomes strictly larger than all previous numbers.

## Python Solution

```python
from typing import List

class Solution:
    def maximumEvenSplit(self, finalSum: int) -> List[int]:
        if finalSum % 2 == 1:
            return []

        result = []
        current_even = 2

        while current_even <= finalSum:
            result.append(current_even)
            finalSum -= current_even
            current_even += 2

        if result:
            result[-1] += finalSum

        return result
```

The implementation begins by checking whether `finalSum` is odd. If it is, the function immediately returns an empty list because no valid decomposition exists.

The `result` list stores the chosen even integers. The variable `current_even` tracks the next smallest unused even number.

Inside the loop, we greedily add the current even number to the answer and subtract it from the remaining sum. Then we move to the next even integer by adding `2`.

The loop stops once the next candidate would exceed the remaining sum.

At that moment, some leftover value may still remain in `finalSum`. Instead of adding another number that might duplicate an existing value, we merge the leftover into the final element of the result list.

This guarantees the sum is exact while preserving uniqueness.

## Go Solution

```go
func maximumEvenSplit(finalSum int64) []int64 {
    if finalSum%2 == 1 {
        return []int64{}
    }

    result := []int64{}
    currentEven := int64(2)

    for currentEven <= finalSum {
        result = append(result, currentEven)
        finalSum -= currentEven
        currentEven += 2
    }

    if len(result) > 0 {
        result[len(result)-1] += finalSum
    }

    return result
}
```

The Go implementation follows exactly the same greedy logic as the Python version.

One small difference is that Go uses `int64` because the function signature requires it. The result slice is initialized as an empty slice rather than `nil`, which matches the expected LeetCode behavior cleanly.

The final leftover adjustment modifies the last element directly using index access.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(sqrt(finalSum)) | The sequence of even numbers grows linearly while their cumulative sum grows quadratically |
| Space | O(sqrt(finalSum)) | The result list stores roughly sqrt(finalSum) integers |

The algorithm keeps adding consecutive even numbers:

```
2 + 4 + 6 + ... + 2k
```

The sum of the first `k` even numbers is:

$2+4+6+\cdots+2k = k(k+1)$

This means the number of iterations is approximately proportional to the square root of `finalSum`.

## Edge Cases

One important edge case occurs when `finalSum` is odd. Since the sum of even integers is always even, no valid decomposition can exist. A naive implementation might still attempt to search for combinations unnecessarily. The solution handles this immediately with an early return.

Another important case is the smallest valid input, `finalSum = 2`. The algorithm correctly adds `2` to the result list and terminates. This validates that the loop logic works even for minimal values.

A more subtle edge case occurs when the remaining leftover value equals an already chosen number. For example, with `finalSum = 20`, the greedy process produces `[2, 4, 6]` with leftover `8`. Appending `8` directly would still work here, but in other cases direct appending could create duplicates. The implementation avoids this issue entirely by always merging the leftover into the last element.

Another edge case happens when the loop consumes the entire sum exactly. For example, `12` becomes `[2, 4, 6]` with zero leftover. The final merge step safely adds `0` to the last element, leaving the answer unchanged.
