## Solution

## Problem Understanding

This problem asks us to measure how the global ranking of national teams changes after each team's points are updated.

We are given two database tables:

The `TeamPoints` table stores the current information for every national team. Each row contains:

- `team_id`, the unique identifier of the team
- `name`, the country name
- `points`, the current ranking points

The `PointsChange` table stores how many points each team gains or loses. Each row contains:

- `team_id`
- `points_change`, which may be positive, negative, or zero

The ranking rules are extremely important:

1. Teams are sorted by `points` in descending order
2. If two teams have the same number of points, then the tie is broken using `name` in lexicographical ascending order

We first compute the original rankings using the original points. Then we apply the point changes and compute the updated rankings. Finally, for each team, we calculate:

```
rank_diff = old_rank - new_rank
```

A positive `rank_diff` means the team improved in the rankings because its new rank number became smaller.

A negative `rank_diff` means the team dropped in the rankings because its new rank number became larger.

A value of `0` means the ranking position stayed the same.

The output must contain:

- `team_id`
- `name`
- `rank_diff`

The problem guarantees that:

- Every `team_id` in `TeamPoints` appears exactly once in `PointsChange`
- `team_id` values are unique
- Team names are unique because no two teams represent the same country

An important subtlety is the tie-breaking rule. Even if two teams have identical points after the update, their relative order depends on the lexicographical order of their names. A naive implementation that ignores this secondary sort condition will produce incorrect rankings.

Another important edge case is when point changes do not alter the ordering at all. The algorithm still needs to correctly compute a `rank_diff` of `0`.

We must also carefully handle negative point changes because a team can move downward in the rankings even after originally being near the top.

## Approaches

### Brute Force Approach

The brute force solution would repeatedly compare every team against every other team to determine rankings both before and after the updates.

For each team, we could count how many teams should appear before it according to the sorting rules. This would require pairwise comparisons between all teams.

The algorithm would work like this:

1. For every team, compare it with every other team to determine its original rank
2. Apply the point updates
3. Again compare every pair of teams to determine the new rankings
4. Compute the difference between the two rankings

This approach is correct because rankings are ultimately defined by relative ordering between teams.

However, it is inefficient because determining rankings through pairwise comparisons requires `O(n²)` work.

### Optimal Approach

The key insight is that rankings are completely determined by sorting order.

Instead of repeatedly comparing every pair of teams, we can:

1. Sort teams once using original points
2. Assign ranks directly from the sorted order
3. Update the points
4. Sort again using updated points
5. Assign new ranks
6. Compute the differences

Sorting already solves the ordering problem efficiently.

The ranking rules map naturally to SQL window functions, specifically `ROW_NUMBER()`.

We can:

- Compute original ranks using `ROW_NUMBER() OVER (ORDER BY points DESC, name ASC)`
- Compute updated ranks using the updated point totals
- Join the two results together
- Subtract the ranks

This approach is significantly more efficient and much cleaner.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeated pairwise comparisons between teams |
| Optimal | O(n log n) | O(n) | Uses sorting or SQL window functions to compute rankings efficiently |

## Algorithm Walkthrough

1. Start with the original `TeamPoints` table and compute the initial rankings.

We sort teams by:

- `points` descending
- `name` ascending

Then we assign row numbers starting from `1`. These row numbers represent the original global rankings.
2. Join `TeamPoints` with `PointsChange`.

This allows us to calculate each team's updated points:

```
updated_points = points + points_change
```
3. Compute the updated rankings.

Using the updated point totals, we again sort by:

- updated points descending
- name ascending

Then assign new row numbers.
4. Match each team's original rank with its updated rank.

Since `team_id` uniquely identifies teams, we can join the two ranking results on `team_id`.
5. Calculate the ranking difference.

The formula is:

```
rank_diff = old_rank - new_rank
```

If the value is positive, the team improved.

If negative, the team dropped.

If zero, the team stayed in the same position.
6. Return the final result.

The output includes:

- `team_id`
- `name`
- `rank_diff`

### Why it works

The algorithm works because rankings are entirely determined by sorted order. The `ROW_NUMBER()` window function assigns unique rankings according to the exact ordering rules required by the problem. Since we compute rankings both before and after the point updates using the same ordering logic, subtracting the two rankings correctly measures the change in position.

## Python Solution

```python
# Write your MySQL query statement below

WITH original_ranks AS (
    SELECT
        team_id,
        name,
        ROW_NUMBER() OVER (
            ORDER BY points DESC, name ASC
        ) AS old_rank
    FROM TeamPoints
),
updated_ranks AS (
    SELECT
        tp.team_id,
        tp.name,
        ROW_NUMBER() OVER (
            ORDER BY (tp.points + pc.points_change) DESC,
                     tp.name ASC
        ) AS new_rank
    FROM TeamPoints tp
    JOIN PointsChange pc
        ON tp.team_id = pc.team_id
)

SELECT
    o.team_id,
    o.name,
    (o.old_rank - u.new_rank) AS rank_diff
FROM original_ranks o
JOIN updated_ranks u
    ON o.team_id = u.team_id;
```

The solution uses two Common Table Expressions.

The first CTE, `original_ranks`, computes the rankings before any updates happen. The `ROW_NUMBER()` window function assigns rankings according to the required sorting rules.

The second CTE, `updated_ranks`, joins the points changes table so that updated point totals can be calculated. Rankings are again assigned using the updated values.

Finally, the main query joins both ranking tables using `team_id` and computes the ranking difference.

Using `ROW_NUMBER()` is important because rankings must remain unique even when teams tie on points. The secondary sorting condition using `name ASC` guarantees deterministic ordering.

## Edge Cases

### Teams With Equal Updated Points

A very important edge case occurs when multiple teams end up with the same updated score. A naive implementation may assume equal points imply equal rankings or may produce nondeterministic ordering.

The problem explicitly states that ties are broken using lexicographical order on the team name. The implementation handles this correctly by including:

```
ORDER BY updated_points DESC, name ASC
```

inside the `ROW_NUMBER()` window function.

### No Ranking Changes

Some teams may gain or lose points without changing their relative position in the rankings.

For example, the top-ranked team may lose a few points but still remain ahead of everyone else. The implementation correctly computes:

```
old_rank - new_rank = 0
```

for such teams.

### Large Ranking Swings

A team may move dramatically upward or downward after the update.

For example:

- A low-ranked team may receive a huge positive increase
- A top-ranked team may receive a large negative decrease

Because the algorithm recomputes the entire sorted ranking after updates, it naturally handles arbitrary movement in either direction without requiring any special-case logic.
