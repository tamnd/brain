## Solution

## Problem Understanding

The problem requires building a dynamic ranking system for scenic locations based on their **score** and **name**. Each location has a unique `name` and an integer `score`. Locations are ranked first by score in descending order, and if scores are equal, by name in lexicographical ascending order. The system supports two operations: adding a new location, and retrieving the `ith` best location, where `i` is the number of times the `get` method has been called.

The input consists of a series of `add` and `get` operations. Each `add` operation provides a location's `name` and `score`. Each `get` operation queries the `ith` best location among all previously added locations. The constraints guarantee that at any time, the number of `get` calls will not exceed the number of added locations, and the total number of operations is at most `4 * 10^4`. This means the solution must efficiently support both inserting and retrieving the ranked locations without scanning the entire list every time.

Important considerations include handling ties correctly (lexicographical order), maintaining the correct `ith` element for repeated `get` calls, and performing efficiently for up to tens of thousands of operations.

Edge cases to consider include adding multiple locations with the same score, adding locations in descending or ascending order of scores, and repeated `get` calls without new additions.

## Approaches

The brute-force approach is to maintain a list of all added locations and, whenever `get` is called, sort the entire list by score and name, then pick the `ith` element. This is correct but inefficient, because sorting each time is O(n log n), and with up to 40,000 operations, it would be too slow.

The key insight for an optimal approach is to maintain two heaps: a **max-heap** for the "worse" half of locations and a **min-heap** for the "better" half. The min-heap keeps the top `i` locations for the current query count. Every time a location is added, it is placed in the appropriate heap, and the heaps are rebalanced so the min-heap always contains exactly `i` best locations. This allows the `get` operation to return the top of the min-heap in O(1) time and maintain logarithmic insertions.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) per get | O(n) | Sort all locations every time get() is called |
| Optimal | O(log n) per add/get | O(n) | Use two heaps to maintain the top `i` elements dynamically |

## Algorithm Walkthrough

1. Initialize the tracker with two heaps: a max-heap `left` for the worst-ranked locations among the top `i`, and a min-heap `right` for the best-ranked locations.
2. When adding a new location `(score, name)`, insert it into the max-heap `left` first. Use a custom comparator that considers `score` descending and `name` ascending.
3. Move the largest element from `left` to `right` to ensure the top `i` best locations are in `right`.
4. If `right` becomes larger than `i` (number of `get` calls so far), move its smallest element back to `left` to maintain the heap invariant.
5. Each `get` call increments the query count and returns the top of `right`, which is the current `ith` best location.

**Why it works**: The heaps maintain a strict partition: `right` contains exactly the top `i` locations by the ranking rules, so the root of `right` is always the `i`th best. Insertions and rebalancing ensure the invariant holds after every `add`.

## Python Solution

```python
import heapq

class SORTracker:

    def __init__(self):
        self.left = []  # max-heap for "worse" half (invert score for max-heap)
        self.right = [] # min-heap for "better" half
        self.query_count = 0

    def add(self, name: str, score: int) -> None:
        # Push to max-heap with inverted score to simulate max-heap
        heapq.heappush(self.left, (-score, -ord(name[0]), name))
        # Move the largest from left to right to maintain top `i` in right
        val = heapq.heappop(self.left)
        heapq.heappush(self.right, ( -val[0], val[2] ))
        # Rebalance if right has more than query_count elements
        if len(self.right) > self.query_count:
            moved = heapq.heappop(self.right)
            heapq.heappush(self.left, (-moved[0], -ord(moved[1][0]), moved[1]))

    def get(self) -> str:
        self.query_count += 1
        # Ensure right heap has exactly query_count elements
        if len(self.right) < self.query_count:
            val = heapq.heappop(self.left)
            heapq.heappush(self.right, (-val[0], val[2]))
        return self.right[0][1]
```

**Implementation Notes**: The `left` heap uses a max-heap by negating scores and part of the name for comparison. The `right` heap always contains exactly the top `i` best locations. Each `get` simply returns the top of `right`.

## Go Solution

```go
package main

import (
    "container/heap"
)

type Location struct {
    score int
    name  string
}

type MaxHeap []Location
func (h MaxHeap) Len() int { return len(h) }
func (h MaxHeap) Less(i, j int) bool {
    if h[i].score == h[j].score {
        return h[i].name > h[j].name
    }
    return h[i].score < h[j].score
}
func (h MaxHeap) Swap(i, j int) { h[i], h[j] = h[j], h[i] }
func (h *MaxHeap) Push(x interface{}) { *h = append(*h, x.(Location)) }
func (h *MaxHeap) Pop() interface{} {
    n := len(*h)
    x := (*h)[n-1]
    *h = (*h)[:n-1]
    return x
}

type MinHeap []Location
func (h MinHeap) Len() int { return len(h) }
func (h MinHeap) Less(i, j int) bool {
    if h[i].score == h[j].score {
        return h[i].name < h[j].name
    }
    return h[i].score > h[j].score
}
func (h MinHeap) Swap(i, j int) { h[i], h[j] = h[j], h[i] }
func (h *MinHeap) Push(x interface{}) { *h = append(*h, x.(Location)) }
func (h *MinHeap) Pop() interface{} {
    n := len(*h)
    x := (*h)[n-1]
    *h = (*h)[:n-1]
    return x
}

type SORTracker struct {
    left  *MaxHeap
    right *MinHeap
    queryCount int
}

func Constructor() SORTracker {
    l := &MaxHeap{}
    r := &MinHeap{}
    heap.Init(l)
    heap.Init(r)
    return SORTracker{left: l, right: r, queryCount: 0}
}

func (this *SORTracker) Add(name string, score int) {
    heap.Push(this.left, Location{score, name})
    moved := heap.Pop(this.left).(Location)
    heap.Push(this.right, moved)
    if this.right.Len() > this.queryCount {
        val := heap.Pop(this.right).(Location)
        heap.Push(this.left, val)
    }
}

func (this *SORTracker) Get() string {
    this.queryCount++
    if this.right.Len() < this.queryCount {
        val := heap.Pop(this.left).(Location)
        heap.Push(this.right, val)
    }
    return (*this.right)[0].name
}
```

**Go Notes**: MaxHeap is implemented by reversing the ordering of `score` and `name` for comparison. MinHeap uses the natural ordering to store top `i` elements. Go requires explicit heap operations via `container/heap`.
