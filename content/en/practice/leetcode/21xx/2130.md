## Solution

## Problem Understanding

The problem gives us a singly linked list with an even number of nodes. For every node at position `i`, there exists a corresponding "twin" node at position `n - 1 - i`, where `n` is the total number of nodes in the list.

We are asked to compute the twin sum for every such pair and return the maximum twin sum.

For example, in the list:

```
[5,4,2,1]
```

The pairs are:

- Index `0` and index `3` → `5 + 1 = 6`
- Index `1` and index `2` → `4 + 2 = 6`

The answer is therefore `6`.

The input is provided as the head of a singly linked list. Since singly linked lists only allow traversal in one direction, accessing the twin node directly is not easy because the twin lies symmetrically from the end of the list.

The constraints are important:

- The number of nodes can be as large as `10^5`
- The list length is guaranteed to be even
- Node values can also be as large as `10^5`

These limits tell us that:

- An `O(n^2)` solution would be too slow
- We should aim for an `O(n)` solution
- Memory usage should ideally be minimized

Several edge cases are already guaranteed away by the problem:

- The list is never empty
- The list length is always even
- Every node has exactly one twin

However, some inputs can still expose implementation bugs:

- A list with only two nodes
- Very large values
- Lists where the maximum twin sum appears in the middle rather than at the ends
- Incorrect midpoint handling when splitting the list

## Approaches

### Brute Force Approach

A straightforward idea is to first store all node values into an array. Once the linked list values are in an indexed structure, computing twin sums becomes easy.

For every index `i` in the first half of the array, we compute:

```
values[i] + values[n - 1 - i]
```

We track the maximum sum encountered.

This approach works because arrays support constant time random access, making it easy to locate both members of a twin pair.

Although this solution is efficient enough for the given constraints, it uses additional memory proportional to the size of the list.

A truly brute force linked list approach without extra storage would repeatedly traverse the list to find the twin node for each position. That would require nested traversals and would result in `O(n^2)` time complexity, which is too slow for `10^5` nodes.

### Optimal Approach

The key observation is that twin nodes are mirrored around the center of the list.

If we reverse the second half of the linked list, then corresponding twin nodes become aligned in the same traversal direction.

For example:

```
Original:
5 -> 4 -> 2 -> 1

Second half reversed:
5 -> 4 -> 1 -> 2
```

Now:

- `5` pairs with `1`
- `4` pairs with `2`

We can walk through both halves simultaneously and compute twin sums in one pass.

To achieve this efficiently:

1. Find the middle of the linked list using slow and fast pointers
2. Reverse the second half
3. Traverse both halves together
4. Compute the maximum twin sum

This gives us linear time complexity with constant extra space.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Repeatedly traverses list to locate twin nodes |
| Optimal | O(n) | O(1) | Reverse second half and compare mirrored nodes |

## Algorithm Walkthrough

1. Use two pointers, `slow` and `fast`, to locate the midpoint of the linked list.

The `slow` pointer moves one step at a time, while the `fast` pointer moves two steps at a time. Since the list length is guaranteed to be even, when `fast` reaches the end, `slow` will point to the start of the second half.
2. Reverse the second half of the linked list.

Starting from `slow`, reverse all links in the second half. After reversal, the tail node becomes the head of the reversed half.

This transformation aligns twin nodes so they can be traversed simultaneously.
3. Initialize two traversal pointers.

One pointer starts from the original head of the list, and the other starts from the head of the reversed second half.
4. Traverse both halves together.

At each step:

- Compute the twin sum
- Update the maximum answer
- Advance both pointers
5. Return the maximum twin sum found.

### Why it works

Twin nodes are symmetric around the center of the linked list. Reversing the second half converts this symmetric relationship into a parallel relationship.

After reversal:

```
First half:   a1 -> a2 -> a3
Second half:  b1 -> b2 -> b3
```

Where:

```
a1 pairs with b1
a2 pairs with b2
a3 pairs with b3
```

Thus, traversing both lists simultaneously guarantees that every twin pair is processed exactly once.

## Python Solution

```python
# Definition for singly-linked list.
# class ListNode:
#     def __init__(self, val=0, next=None):
#         self.val = val
#         self.next = next

class Solution:
    def pairSum(self, head: Optional[ListNode]) -> int:
        slow = head
        fast = head

        # Find the start of the second half
        while fast and fast.next:
            slow = slow.next
            fast = fast.next.next

        # Reverse the second half
        prev = None
        curr = slow

        while curr:
            next_node = curr.next
            curr.next = prev
            prev = curr
            curr = next_node

        # Compute maximum twin sum
        max_sum = 0
        first = head
        second = prev

        while second:
            max_sum = max(max_sum, first.val + second.val)
            first = first.next
            second = second.next

        return max_sum
```

The implementation follows the algorithm directly.

The first loop finds the midpoint using the classic slow and fast pointer technique. Because the list length is even, `slow` ends exactly at the beginning of the second half.

The second loop reverses the second half in place. Three pointers are used:

- `curr` tracks the current node
- `prev` tracks the reversed portion
- `next_node` temporarily stores the next node before links are changed

After reversal, `prev` points to the new head of the reversed half.

Finally, two pointers traverse both halves simultaneously. Since the second half was reversed, corresponding twin nodes now appear in matching order. We compute each twin sum and maintain the maximum value.

## Go Solution

```go
/**
 * Definition for singly-linked list.
 * type ListNode struct {
 *     Val int
 *     Next *ListNode
 * }
 */
func pairSum(head *ListNode) int {
    slow := head
    fast := head

    // Find midpoint
    for fast != nil && fast.Next != nil {
        slow = slow.Next
        fast = fast.Next.Next
    }

    // Reverse second half
    var prev *ListNode
    curr := slow

    for curr != nil {
        nextNode := curr.Next
        curr.Next = prev
        prev = curr
        curr = nextNode
    }

    // Compute maximum twin sum
    maxSum := 0
    first := head
    second := prev

    for second != nil {
        sum := first.Val + second.Val

        if sum > maxSum {
            maxSum = sum
        }

        first = first.Next
        second = second.Next
    }

    return maxSum
}
```

The Go implementation closely mirrors the Python solution.

Go uses explicit pointer types, so reversing the linked list requires careful pointer manipulation with `*ListNode`.

Unlike Python, Go does not have built in `max()` support for integers, so the maximum comparison is handled manually with an `if` statement.

Integer overflow is not an issue because the maximum possible twin sum is:

```go
100000 + 100000 = 200000
```

which safely fits within Go's `int` type.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each node is visited a constant number of times |
| Space | O(1) | Reversal is performed in place |

The algorithm performs three linear traversals:

1. Finding the midpoint
2. Reversing the second half
3. Computing twin sums

Each traversal touches each node at most once, so the total runtime remains linear.

No auxiliary arrays, stacks, or hash maps are used, so the extra memory usage is constant.

## Edge Cases

### Minimum Length List

The smallest valid linked list contains exactly two nodes. This case is easy to mishandle because midpoint logic and reversal logic often assume larger lists.

For example:

```
[1,100000]
```

The implementation correctly handles this because the slow pointer lands on the second node, and reversing a single node works naturally without special cases.

### Very Large Node Values

Node values can be as large as `100000`, so the maximum twin sum can reach `200000`.

Some languages require overflow consideration when adding large integers. Both Python and Go safely handle this range using standard integer types.

### Maximum Twin Sum Not at the Ends

A naive implementation might incorrectly assume the largest values appear near the outermost pairs.

For example:

```
[9,8,7,6,5,4]
```

Twin sums are:

```
9 + 4 = 13
8 + 5 = 13
7 + 6 = 13
```

Or another example:

```
[1,100,50,2]
```

Twin sums:

```
1 + 2 = 3
100 + 50 = 150
```

The implementation correctly checks every pair after reversal, ensuring the true maximum is always found.

### Incorrect Midpoint Handling

Linked list midpoint algorithms are a common source of off by one errors.

Because the list length is guaranteed to be even, the slow and fast pointer technique reliably places `slow` at the start of the second half. This ensures that every node in the first half aligns perfectly with its twin after reversal.
