## Solution

## Problem Understanding

The problem gives us a list of parent-child relationships that together describe a valid binary tree. Each entry in `descriptions` has the form:

```
[parent, child, isLeft]
```

This means:

- `parent` is the parent node value
- `child` is the child node value
- `isLeft == 1` means the child is the left child
- `isLeft == 0` means the child is the right child

All node values are unique, which is very important because it means every value corresponds to exactly one tree node.

Our task is to reconstruct the entire binary tree and return the root node.

The input does not explicitly tell us which node is the root. Instead, we must determine it ourselves. In a tree, the root is the only node that never appears as a child. Every other node has exactly one parent.

The constraints tell us that:

- There can be up to `10^4` relationships
- Node values can be as large as `10^5`
- The tree is guaranteed to be valid

Because the tree is valid, we do not need to worry about cycles, duplicate parents, or malformed structures. Every child belongs to exactly one parent, and the resulting structure is a proper binary tree.

A naive implementation can easily make mistakes in a few places:

- Creating duplicate `TreeNode` objects for the same value
- Forgetting to track which nodes are children
- Returning the wrong root
- Incorrectly assigning left and right children

The guarantees simplify the problem significantly because we know:

- Every value is unique
- There is exactly one root
- The tree structure is valid
- A child cannot have multiple parents

## Approaches

### Brute Force Approach

A brute-force solution would repeatedly scan the entire list of descriptions while trying to reconstruct parent-child relationships manually.

For example, we could:

1. Create nodes gradually
2. For every node, repeatedly search the descriptions list to find its children
3. Separately determine the root by checking whether each node appears as a child

This works because eventually every relationship is processed correctly.

However, repeatedly scanning the entire descriptions array becomes inefficient. If there are `n` descriptions and we repeatedly search through them for many nodes, the overall complexity can become quadratic.

With up to `10^4` entries, an `O(n^2)` solution is unnecessarily slow.

### Optimal Approach

The key insight is that every node value should map to exactly one `TreeNode` object.

A hash map allows us to:

- Create nodes only once
- Retrieve nodes in constant time
- Efficiently connect parents and children

At the same time, we can maintain a set of all child nodes. Since the root never appears as a child, we can identify the root after processing all descriptions.

This allows us to build the entire tree in a single pass.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeatedly scans descriptions to build relationships |
| Optimal | O(n) | O(n) | Uses hash map and child tracking for efficient construction |

## Algorithm Walkthrough

1. Create a hash map called `nodes`.

The keys are integer values from the tree, and the values are `TreeNode` objects. This ensures that every node value corresponds to exactly one node instance.
2. Create a set called `children`.

Every node that appears as a child will be inserted into this set. Later, the root will be the only node absent from this set.
3. Iterate through every description `[parent, child, isLeft]`.

For each relationship:

- If the parent node does not exist in the map, create it.
- If the child node does not exist in the map, create it.
- Connect the child to the parent:

- If `isLeft == 1`, assign it to `parent.left`
- Otherwise assign it to `parent.right`
- Add the child value to the `children` set.
4. Find the root node.

Iterate through all node values in the hash map. The root is the node whose value never appears in the `children` set.
5. Return the root node.

Since the tree is guaranteed to be valid, exactly one such node exists.

### Why it works

The algorithm works because every non-root node appears exactly once as a child, while the root never appears as a child at all.

The hash map guarantees that all references to the same value point to the same `TreeNode` object, preventing duplicate nodes from being created.

By processing every description exactly once, all parent-child relationships are correctly established.

## Python Solution

```python
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import List, Optional

class Solution:
    def createBinaryTree(
        self,
        descriptions: List[List[int]]
    ) -> Optional[TreeNode]:

        nodes = {}
        children = set()

        for parent_val, child_val, is_left in descriptions:

            if parent_val not in nodes:
                nodes[parent_val] = TreeNode(parent_val)

            if child_val not in nodes:
                nodes[child_val] = TreeNode(child_val)

            parent_node = nodes[parent_val]
            child_node = nodes[child_val]

            if is_left == 1:
                parent_node.left = child_node
            else:
                parent_node.right = child_node

            children.add(child_val)

        for node_val in nodes:
            if node_val not in children:
                return nodes[node_val]

        return None
```

The implementation closely follows the algorithm steps.

The `nodes` dictionary stores one `TreeNode` object for each unique value. Before connecting nodes, we check whether they already exist in the dictionary. This prevents duplicate node creation.

The `children` set tracks all values that appear as children. After processing all relationships, the only node not present in this set must be the root.

During each iteration:

- We retrieve the parent and child nodes
- We assign the child to either the left or right pointer
- We record the child value

Finally, we scan all created nodes and return the one that never appeared as a child.

## Go Solution

```go
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func createBinaryTree(descriptions [][]int) *TreeNode {

    nodes := make(map[int]*TreeNode)
    children := make(map[int]bool)

    for _, desc := range descriptions {

        parentVal := desc[0]
        childVal := desc[1]
        isLeft := desc[2]

        if _, exists := nodes[parentVal]; !exists {
            nodes[parentVal] = &TreeNode{Val: parentVal}
        }

        if _, exists := nodes[childVal]; !exists {
            nodes[childVal] = &TreeNode{Val: childVal}
        }

        parentNode := nodes[parentVal]
        childNode := nodes[childVal]

        if isLeft == 1 {
            parentNode.Left = childNode
        } else {
            parentNode.Right = childNode
        }

        children[childVal] = true
    }

    for val, node := range nodes {
        if !children[val] {
            return node
        }
    }

    return nil
}
```

The Go implementation mirrors the Python logic closely.

Instead of Python dictionaries and sets, Go uses:

- `map[int]*TreeNode` for node storage
- `map[int]bool` to simulate a set of child values

Pointers are used naturally in Go tree structures, so child assignments directly connect nodes.

The final root search checks which node value does not exist in the `children` map.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each description is processed once |
| Space | O(n) | Hash map and child set store up to n nodes |

The algorithm performs a single pass through the descriptions array. Every hash map and set operation runs in average constant time.

The space usage is linear because we store:

- One `TreeNode` per unique value
- One child-set entry per child node

Since the number of nodes is proportional to the number of descriptions, the total auxiliary space is `O(n)`.

## Edge Cases

One important edge case is when the tree contains only a single parent-child relationship. A buggy implementation might incorrectly assume there are multiple levels or fail to identify the root correctly. The current solution handles this naturally because the child is recorded in the child set, leaving the parent as the only node not marked as a child.

Another important edge case is a completely skewed tree, where every node has only one child. For example, a chain like `1 -> 2 -> 3 -> 4`. Some implementations accidentally overwrite children or mishandle missing left/right pointers. This implementation correctly assigns only the specified child pointer and leaves the other pointer as `None` or `nil`.

A third important edge case is when the root node appears late in the descriptions array. The input order is arbitrary, so we cannot assume the first parent encountered is the root. By tracking every child value separately, the implementation correctly identifies the true root regardless of ordering.

A fourth subtle case is repeated references to previously created nodes. Since nodes are reused through the hash map, the implementation guarantees that every value corresponds to exactly one `TreeNode` object. Without this safeguard, duplicate node objects could be created, leading to disconnected or incorrect trees.
