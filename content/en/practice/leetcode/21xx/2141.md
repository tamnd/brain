## Solution

## Problem Understanding

The problem asks us to determine the **maximum number of minutes `n` computers can run simultaneously** given a collection of batteries with fixed runtimes. Each computer can initially take **at most one battery**, and later batteries can be swapped in from other computers or unused batteries at any integer time. Batteries are **non-rechargeable**, so once depleted, they cannot be reused.

The input consists of an integer `n` representing the number of computers, and an array `batteries` where each element is the runtime in minutes of that battery. The output is a single integer representing the maximum duration in minutes that all `n` computers can run at the same time.

Constraints indicate that the number of computers `n` is at least 1 and at most the number of batteries, and each battery's runtime is at least 1 minute and can go up to 10^9 minutes. The problem guarantees that `n <= len(batteries)` and all battery values are positive. Important edge cases include situations where some batteries are extremely large compared to others, or when the number of batteries exactly equals the number of computers.

The core challenge is **efficiently distributing the battery power** so that the runtime of all computers is maximized, which requires considering **combining partial battery runtimes across multiple batteries** and **balancing power consumption**. A naive approach that simulates minute by minute is infeasible due to high input constraints.

## Approaches

A brute-force approach would attempt to simulate each minute of computer runtime, decrementing battery power and swapping batteries as necessary. This guarantees correctness but is too slow because each battery can last up to 10^9 minutes, making a per-minute simulation prohibitively expensive.

The key observation is that **the problem reduces to distributing total battery power evenly among all `n` computers**. If the sum of all battery runtimes is `total_power`, the maximum runtime for `n` computers cannot exceed `total_power // n`. Some batteries may be larger than the target runtime, in which case we can cap them at the target, as extra minutes beyond the target do not help other computers. This allows us to use **binary search** on the possible runtime range `[1, total_power // n]` and **check feasibility** efficiently by summing `min(battery, target_runtime)` for all batteries.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(total_power) | O(1) | Simulate each minute and battery swap; too slow for large inputs |
| Optimal | O(m log(sum(batteries)/n)) | O(1) | Binary search the maximum feasible runtime; check feasibility using sum of capped battery contributions |

## Algorithm Walkthrough

1. **Compute total power:** Calculate the sum of all battery runtimes. This gives an upper bound for how many total minutes all computers can run if the power is distributed perfectly.
2. **Initialize binary search:** Set `low = 1` and `high = total_power // n`. The lower bound is trivially 1 minute, and the upper bound is the ideal scenario where power is evenly split.
3. **Binary search loop:** While `low <= high`, compute `mid = (low + high) // 2` as the candidate runtime. Check if it is possible to run all `n` computers for `mid` minutes by summing contributions from each battery, using `min(battery, mid)` for each. If the total is at least `n * mid`, the candidate is feasible.
4. **Adjust search bounds:** If `mid` is feasible, it means we can potentially run longer, so set `low = mid + 1`. Otherwise, `mid` is too ambitious, so set `high = mid - 1`.
5. **Return the result:** After the loop, `high` contains the maximum feasible runtime.

**Why it works:** The algorithm maintains the invariant that all candidate runtimes less than or equal to `high` are feasible. Binary search efficiently narrows down the largest feasible runtime without simulating every minute, leveraging the fact that runtime feasibility is monotonic.

## Python Solution

```python
from typing import List

class Solution:
    def maxRunTime(self, n: int, batteries: List[int]) -> int:
        total_power = sum(batteries)
        low, high = 1, total_power // n
        
        def can_run(target: int) -> bool:
            power_available = sum(min(b, target) for b in batteries)
            return power_available >= n * target
        
        while low <= high:
            mid = (low + high) // 2
            if can_run(mid):
                low = mid + 1
            else:
                high = mid - 1
                
        return high
```

The Python implementation first calculates the total power and sets up the binary search range. The helper `can_run` checks feasibility by summing the capped contributions of each battery. The main loop adjusts `low` and `high` based on feasibility, and the final answer is `high` which is the largest feasible runtime.

## Go Solution

```go
func maxRunTime(n int, batteries []int) int64 {
    var totalPower int64 = 0
    for _, b := range batteries {
        totalPower += int64(b)
    }
    
    low, high := int64(1), totalPower/int64(n)
    
    canRun := func(target int64) bool {
        var sum int64 = 0
        for _, b := range batteries {
            if int64(b) < target {
                sum += int64(b)
            } else {
                sum += target
            }
        }
        return sum >= target*int64(n)
    }
    
    for low <= high {
        mid := (low + high) / 2
        if canRun(mid) {
            low = mid + 1
        } else {
            high = mid - 1
        }
    }
    
    return high
}
```

The Go version mirrors the Python logic but uses `int64` to handle potential overflow from large battery sums. The helper function `canRun` checks feasibility using the same capped sum approach.

## Worked Examples

### Example 1

`n = 2`, `batteries = [3,3,3]`

Total power = 9. Binary search range: `[1, 4]`.

1. mid = 2 → sum(min(3,2) for all) = 2+2+2=6 ≥ 2*2 → feasible, low = 3
2. mid = 3 → sum(min(3,3) for all) = 9 ≥ 2*3=6 → feasible, low = 4
3. mid = 4 → sum(min(3,4) for all) = 9 ≥ 2*4=8 → feasible, low = 5
4. mid = 5 → sum(min(3,5) for all) = 9 < 10 → not feasible, high = 4

Return 4.

### Example 2

`n = 2`, `batteries = [1,1,1,1]`

Total power = 4. Binary search range: `[1,2]`.

1. mid = 1 → sum = 1+1+1+1=4 ≥ 2*1=2 → feasible, low=2
2. mid = 2 → sum = 1+1+1+1=4 ≥ 4 → feasible, low=3
3. mid = 3 → sum = 4 < 2*3=6 → not feasible, high=2

Return 2.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m log(sum(batteries)/n)) | m = len(batteries); binary search over the range [1, total_power//n], each step summing over m batteries |
| Space | O(1) | Only a few integer variables are used; no extra data structures |

The binary search reduces the number of candidate runtimes to roughly log(total_power/n), making it efficient even when battery runtimes are very large.
