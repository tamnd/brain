**Topics:** Array, Segment Tree, Ordered Set

## Solution

## Problem Understanding

The problem describes a painting process on a number line. Each day, we are given an interval `[start, end)` that represents a segment of the line we want to paint. The important detail is that repainting an already painted region does not count as new work. For every day, we must determine how much previously unpainted area becomes painted for the first time.

The input is a 2D array `paint` where:

- `paint[i][0]` is the starting coordinate
- `paint[i][1]` is the ending coordinate
- The interval is half-open, meaning it includes `start` but excludes `end`

For each interval, we must return the amount of newly painted area introduced on that day.

For example, if we already painted `[1, 5)` and later try to paint `[2, 6)`, only `[5, 6)` contributes new area, so the answer for that day is `1`.

The constraints are large:

- Up to `10^5` intervals
- Coordinates up to `5 * 10^4`

A naive solution that checks every previously painted unit for every interval could easily become too slow in worst-case scenarios. The relatively small coordinate range suggests that interval compression or jump-based traversal techniques may work efficiently.

There are several important edge cases to consider:

- Completely overlapping intervals, where the answer becomes `0`
- Intervals that only partially overlap
- Adjacent intervals such as `[1,4)` and `[4,7)`, which do not overlap because the right endpoint is excluded
- Large chains of overlapping intervals that could make repeated scanning expensive
- Repainting tiny uncovered gaps inside large painted ranges

The problem guarantees that `start < end`, so we never need to handle empty intervals.

## Approaches

### Brute Force Approach

The most direct solution is to treat the number line as individual unit segments. Since coordinates are at most `50000`, we could maintain a boolean array where `painted[x]` indicates whether unit segment `x -> x+1` has already been painted.

For every interval `[start, end)`:

- Iterate through every integer coordinate `x` from `start` to `end - 1`
- If `painted[x]` is false, mark it true and increment the answer for that day

This works because each unit interval corresponds exactly to one length of paint on the number line.

The problem with this approach is efficiency. In the worst case, each interval may span almost the entire coordinate range, producing roughly:

$$O(n \cdot M)$$

where:

- `n = 10^5`
- `M = 5 * 10^4`

That can become too expensive.

### Optimal Approach

The key observation is that once a coordinate has already been painted, we never need to visit it again.

Instead of repeatedly scanning painted positions, we can use a "next unpainted position" structure. This behaves similarly to path compression in Disjoint Set Union.

The idea is:

- If position `x` is already painted, jump directly to the next candidate position that may still be unpainted
- After painting position `x`, link it to `x + 1`
- Future traversals skip over already processed regions immediately

This transforms repeated linear scanning into nearly constant amortized jumps.

We maintain a hash map:

```
next_unpainted[x] = next position to check after x
```

Using path compression, every coordinate gets skipped efficiently after its first visit.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * M) | O(M) | Scans every coordinate repeatedly |
| Optimal | O(total painted coordinates * α(M)) | O(M) | Uses path compression to skip painted regions |

Here, `α(M)` is the inverse Ackermann function, which is effectively constant in practice.

## Algorithm Walkthrough

### Optimal Algorithm Using Path Compression

1. Create a hash map called `next_pos`.

This map stores jump pointers. If a coordinate has already been painted, the map tells us the next coordinate worth checking.
2. Define a helper function `find(x)`.

The purpose of `find(x)` is to return the next unpainted coordinate starting from `x`.

If `x` does not exist in `next_pos`, then `x` itself is still unpainted.

Otherwise, recursively follow jump pointers until reaching an unpainted coordinate.

Apply path compression while returning so future lookups become faster.
3. Process each interval `[start, end)` independently.

Initialize:

- `current = find(start)`
- `new_area = 0`
4. While `current < end`:

This means we found an unpainted coordinate inside the interval.

Since this unit segment is newly painted:

- Increment `new_area`
- Mark it painted by setting:

```
next_pos[current] = current + 1
```

Then jump to the next available unpainted coordinate:

```
current = find(current)
```
5. Store `new_area` in the result array.
6. Continue for all days.

### Why it works

The invariant is that `find(x)` always returns the leftmost coordinate at or after `x` that has not yet been painted.

Whenever we paint coordinate `x`, we redirect it to `x + 1`. Path compression ensures that chains of already painted positions collapse into direct jumps. Therefore, every coordinate is effectively processed only once, guaranteeing correctness and efficiency.

## Python Solution

```python
from typing import List

class Solution:
    def amountPainted(self, paint: List[List[int]]) -> List[int]:
        next_pos = {}

        def find(x: int) -> int:
            if x not in next_pos:
                return x

            next_pos[x] = find(next_pos[x])
            return next_pos[x]

        result = []

        for start, end in paint:
            painted_today = 0

            current = find(start)

            while current < end:
                painted_today += 1

                next_pos[current] = current + 1

                current = find(current)

            result.append(painted_today)

        return result
```

The implementation revolves around the `find()` function. This function behaves similarly to a Disjoint Set Union "find with path compression" operation.

If a coordinate is not present in `next_pos`, it means that coordinate has never been painted before, so it is immediately usable.

When a coordinate becomes painted, we redirect it to the next coordinate:

```python
next_pos[current] = current + 1
```

This effectively removes the coordinate from future consideration.

The recursive compression step:

```python
next_pos[x] = find(next_pos[x])
```

ensures that long chains collapse into short jumps, dramatically improving performance.

The outer loop processes each day's interval independently, while the inner loop paints only truly unpainted coordinates.

## Go Solution

```go
package main

func amountPainted(paint [][]int) []int {
	nextPos := make(map[int]int)

	var find func(int) int

	find = func(x int) int {
		next, exists := nextPos[x]

		if !exists {
			return x
		}

		nextPos[x] = find(next)
		return nextPos[x]
	}

	result := make([]int, 0, len(paint))

	for _, interval := range paint {
		start := interval[0]
		end := interval[1]

		paintedToday := 0

		current := find(start)

		for current < end {
			paintedToday++

			nextPos[current] = current + 1

			current = find(current)
		}

		result = append(result, paintedToday)
	}

	return result
}
```

The Go implementation follows the same logic as the Python version.

A `map[int]int` is used instead of a Python dictionary. The recursive `find` function performs path compression exactly the same way.

Go slices are initialized with capacity equal to the number of intervals to avoid unnecessary reallocations.

Integer overflow is not a concern because coordinates are bounded by `5 * 10^4`.

## Edge Cases

### Completely Overlapping Intervals

An interval may already be fully painted by previous days. For example:

```
[[1,10],[2,5]]
```

After painting `[1,10)`, every coordinate inside `[2,5)` is already covered. A naive implementation might still scan every coordinate repeatedly. The path compression approach skips directly past painted regions, causing the second interval to contribute `0`.

### Adjacent Intervals

Intervals such as `[1,4)` and `[4,7)` touch at the boundary but do not overlap because the right endpoint is excluded.

This distinction is easy to mishandle if intervals are treated as closed ranges. The implementation correctly iterates while `current < end`, preserving half-open interval semantics.

### Deep Chains of Painted Coordinates

Consider many heavily overlapping intervals:

```
[[1,10000],[2,10000],[3,10000],...]
```

Without path compression, repeatedly traversing painted coordinates would become extremely slow. The `find()` compression step collapses long chains into direct jumps, ensuring efficient amortized performance even under worst-case overlap patterns.
