## Solution

## Problem Understanding

The problem asks us to transform an array into an alternating array using the minimum number of element changes.

An array is considered alternating when two conditions hold:

1. Every element matches the element two positions before it.
2. Every adjacent pair of elements is different.

These two conditions together imply a very specific structure:

- All elements at even indices must be identical.
- All elements at odd indices must be identical.
- The chosen even value and odd value must be different.

For example:

- `[3,1,3,1,3,1]` is alternating because:

- All even positions contain `3`
- All odd positions contain `1`
- `3 != 1`

The input consists of:

- A `0-indexed` integer array `nums`
- `1 <= nums.length <= 10^5`
- `1 <= nums[i] <= 10^5`

The output should be the minimum number of operations required to transform the array into a valid alternating array. In one operation, we may replace any element with any positive integer.

The constraints are large enough that we cannot try every possible pair of values for even and odd positions naively. An `O(n^2)` or worse approach would be too slow for `n = 100000`.

Several edge cases are important:

- Arrays of length `1` are already alternating, because there are no adjacent violations.
- The most frequent value on even indices might equal the most frequent value on odd indices. In that case, we cannot use both simultaneously.
- Arrays where all values are identical require changing either all odd positions or all even positions.
- Very unbalanced distributions can make the second-most-frequent value important.

## Approaches

### Brute Force Approach

A brute-force solution would try every possible value for even positions and every possible value for odd positions.

For each pair `(evenValue, oddValue)` where the values are different:

- Count how many even indices already equal `evenValue`
- Count how many odd indices already equal `oddValue`
- The remaining positions must be modified

This guarantees correctness because every valid alternating array must consist of exactly one repeated even value and one repeated odd value.

However, this approach is too slow. If there are up to `10^5` distinct values, trying all pairs becomes extremely expensive.

### Key Insight

The crucial observation is that we only care about the most frequent values at even indices and odd indices.

Why?

Because minimizing operations is equivalent to maximizing how many elements we keep unchanged.

Suppose:

- `evenCount[x]` is how many even positions already contain `x`
- `oddCount[y]` is how many odd positions already contain `y`

If we choose:

- even positions -> value `x`
- odd positions -> value `y`

then the number of unchanged elements is:

```
evenCount[x] + oddCount[y]
```

So we want the pair `(x, y)` with:

- `x != y`
- maximum retained elements

Instead of checking every value combination, we only need the top few most frequent candidates from each side.

The optimal answer will always involve either:

- the most frequent even value
- or the second most frequent even value

and similarly for odd positions.

This reduces the problem to a linear-time counting problem.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(U² + n) | O(U) | Tries all distinct value pairs |
| Optimal | O(n) | O(U) | Uses frequency counting and greedy selection |

Here, `U` represents the number of distinct values.

## Algorithm Walkthrough

1. Split the array into even-indexed and odd-indexed positions.

We treat these separately because all even indices must share one value, and all odd indices must share another value.
2. Count frequencies for even and odd positions.

Use two hash maps:

- `even_freq`
- `odd_freq`

For every index:

- if index is even, increment `even_freq[nums[i]]`
- otherwise increment `odd_freq[nums[i]]`
3. Find the two most frequent values in each frequency map.

We need:

- most frequent even value
- second most frequent even value
- most frequent odd value
- second most frequent odd value

The second-best candidate matters because the most frequent even and odd values might be identical.
4. Check whether the most frequent even value differs from the most frequent odd value.

If they are different, we can use both directly.

The number of unchanged elements becomes:

```
bestEvenCount + bestOddCount
```
5. Handle the conflict case.

If both sides want the same value, we cannot use it for both even and odd positions.

We have two choices:

- Use the best even value and second-best odd value
- Use the second-best even value and best odd value

We choose whichever keeps more elements unchanged.
6. Compute the minimum operations.

The answer is:

```
n - maximum_kept_elements
```

### Why it works

Every alternating array must use exactly one value for even positions and one different value for odd positions. Therefore, the optimal solution is the pair of distinct values that preserves the largest number of existing elements.

The algorithm explicitly searches for this optimal pair using frequency counts. Since only the top frequency candidates can maximize preserved elements, considering the best and second-best frequencies is sufficient to guarantee correctness.

## Python Solution

```python
from typing import List
from collections import Counter

class Solution:
    def minimumOperations(self, nums: List[int]) -> int:
        n = len(nums)

        if n == 1:
            return 0

        even_freq = Counter()
        odd_freq = Counter()

        for i, num in enumerate(nums):
            if i % 2 == 0:
                even_freq[num] += 1
            else:
                odd_freq[num] += 1

        even_common = even_freq.most_common(2)
        odd_common = odd_freq.most_common(2)

        # Pad with dummy values if fewer than 2 entries exist
        while len(even_common) < 2:
            even_common.append((0, 0))

        while len(odd_common) < 2:
            odd_common.append((0, 0))

        even_value_1, even_count_1 = even_common[0]
        even_value_2, even_count_2 = even_common[1]

        odd_value_1, odd_count_1 = odd_common[0]
        odd_value_2, odd_count_2 = odd_common[1]

        if even_value_1 != odd_value_1:
            kept = even_count_1 + odd_count_1
        else:
            kept = max(
                even_count_1 + odd_count_2,
                even_count_2 + odd_count_1
            )

        return n - kept
```

The implementation begins by handling the smallest edge case. Arrays of length one are already alternating, so the answer is immediately `0`.

Next, the algorithm separates frequency counting for even and odd indices using two `Counter` objects. This allows independent analysis of both parity groups.

The `most_common(2)` call retrieves the top two frequency candidates from each side. The second candidate is necessary because the most frequent values may conflict.

If the best even value and best odd value differ, the solution is straightforward. We keep all occurrences of both.

If they are identical, we evaluate both possible alternatives and choose the better one.

Finally, the minimum operations equal the total array length minus the number of elements we can preserve.

## Go Solution

```go
package main

func minimumOperations(nums []int) int {
	n := len(nums)

	if n == 1 {
		return 0
	}

	evenFreq := make(map[int]int)
	oddFreq := make(map[int]int)

	for i, num := range nums {
		if i%2 == 0 {
			evenFreq[num]++
		} else {
			oddFreq[num]++
		}
	}

	evenVal1, evenCnt1 := topTwo(evenFreq)[0][0], topTwo(evenFreq)[0][1]
	evenVal2, evenCnt2 := topTwo(evenFreq)[1][0], topTwo(evenFreq)[1][1]

	oddVal1, oddCnt1 := topTwo(oddFreq)[0][0], topTwo(oddFreq)[0][1]
	oddVal2, oddCnt2 := topTwo(oddFreq)[1][0], topTwo(oddFreq)[1][1]

	kept := 0

	if evenVal1 != oddVal1 {
		kept = evenCnt1 + oddCnt1
	} else {
		option1 := evenCnt1 + oddCnt2
		option2 := evenCnt2 + oddCnt1

		if option1 > option2 {
			kept = option1
		} else {
			kept = option2
		}
	}

	return n - kept
}

func topTwo(freq map[int]int) [][2]int {
	first := [2]int{0, 0}
	second := [2]int{0, 0}

	for value, count := range freq {
		if count > first[1] {
			second = first
			first = [2]int{value, count}
		} else if count > second[1] {
			second = [2]int{value, count}
		}
	}

	return [][2]int{first, second}
}
```

The Go implementation follows the same logic as the Python version but requires a manual helper function to compute the top two frequencies.

Go does not provide a built-in equivalent of Python's `Counter.most_common()`, so the `topTwo` helper scans the map once while maintaining the two best candidates.

There are no integer overflow concerns because counts never exceed `10^5`, which fits comfortably inside Go's `int`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each element is processed a constant number of times |
| Space | O(U) | Frequency maps store distinct values |

The algorithm performs a single pass to build the frequency maps and another pass over distinct keys to determine the top candidates. Since every operation is linear relative to the input size, the total runtime is `O(n)`.

The space usage depends on how many distinct values appear in the array. In the worst case, every number is unique, requiring `O(U)` additional space.

## Edge Cases

One important edge case is an array of length `1`. Since there are no adjacent elements, the array automatically satisfies the alternating conditions. A careless implementation might incorrectly attempt frequency comparisons or assume both even and odd groups exist. The implementation explicitly checks for this case and returns `0`.

Another critical edge case occurs when the most frequent even value equals the most frequent odd value. This is the main source of logical bugs. Since adjacent elements must differ, we cannot assign the same value to both parity groups. The implementation handles this by considering second-best candidates and selecting the better alternative.

A third important case is when one side has only a single distinct value. For example:

```
[2,2,2,2,2]
```

The odd side may have only one candidate available. If we do not safely handle missing second-best frequencies, index errors or incorrect calculations may occur. The implementation pads missing entries with `(0, 0)` so the logic remains consistent and safe.

A fourth subtle case is when the array is already alternating. In such cases, the optimal answer is `0`, and the algorithm naturally discovers this because the best even and odd values already maximize preserved elements equal to `n`.
