## Solution

## Problem Understanding

The problem asks us to determine the **minimum number of moves** needed to transform the integer `1` into a given `target` integer.

At every move, we have exactly two possible operations:

1. **Increment by 1**: `x = x + 1`
2. **Double the current value**: `x = 2 * x`

The important restriction is that the doubling operation can only be used **at most `maxDoubles` times**, while increments may be used without limit.

The goal is to find the smallest number of operations required to reach exactly `target`.

At first glance, it may seem natural to simulate the process forward from `1` to `target`, deciding when to increment and when to double. However, making locally optimal choices while moving forward is difficult because doubling too early or too late may increase the total number of moves.

A much cleaner perspective is to work **backwards**, starting from `target` and reducing it to `1`.

When reversing the operations:

- A forward **increment (`+1`)** becomes a backward **decrement (`-1`)**
- A forward **double (`*2`)** becomes a backward **division by 2**

This reverse perspective makes greedy decisions obvious.

The input consists of:

- `target`: the number we want to reach
- `maxDoubles`: the maximum number of doubling operations allowed

The output is a single integer representing the minimum number of moves.

### Constraints Analysis

The constraints are:

- `1 <= target <= 10^9`
- `0 <= maxDoubles <= 100`

The target value can be very large, up to one billion. This immediately tells us that any brute force search over states or dynamic programming across all numbers up to `target` would likely be too expensive.

However, `maxDoubles` is relatively small, at most `100`, which hints that an efficient greedy strategy involving halving operations may work well.

### Important Edge Cases

One important edge case occurs when `maxDoubles = 0`. In this case, doubling is impossible, so the only valid operation is incrementing. The answer must simply be `target - 1`.

Another important case is when `target = 1`. Since we already start at `1`, no operations are needed.

A subtle case happens when `target` is odd during the reverse process. Since odd numbers cannot be evenly divided by `2`, we must first subtract `1` before halving. Missing this condition often causes bugs.

Finally, we must handle the situation where we run out of allowed doubles before reaching `1`. At that point, the only remaining option is repeated decrements, which can be computed directly.

## Approaches

### Brute Force Approach

A brute force solution would explore all possible sequences of operations starting from `1`.

At every step, we can either increment or double, creating a branching decision tree. We could use techniques such as breadth-first search (BFS) to explore all reachable states and stop once we reach `target`.

This approach guarantees correctness because BFS finds the shortest path in an unweighted graph, and each operation costs exactly one move.

However, the state space becomes extremely large. Since `target` can be as large as `10^9`, exploring all possibilities is computationally infeasible. Even memoized recursion or dynamic programming would still struggle because the range of values is too large.

### Optimal Greedy Approach

The key observation is that solving the problem **backwards** turns a difficult optimization problem into a greedy one.

Suppose we are currently at `target` while moving backward:

- If the number is **even** and we still have remaining doubles, dividing by `2` is always optimal because it removes a large amount of progress in one move.
- If the number is **odd**, division is impossible, so we must subtract `1`.
- If we have exhausted all doubles, the only remaining choice is repeated subtraction until we reach `1`.

Why is division greedy-safe?

Whenever division is available, using it immediately is always beneficial because halving reduces the number significantly in just one move. Delaying division would only require additional decrement operations and never improve the result.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(target) | O(target) | BFS or DP over all reachable numbers |
| Optimal | O(log target) | O(1) | Greedy reverse simulation using halving |

## Algorithm Walkthrough

### Optimal Greedy Algorithm

1. Initialize a counter `moves = 0`.
2. While `target > 1` and we still have remaining doubles:

- If `target` is **even**, divide it by `2`. This corresponds to reversing a doubling operation.
- If `target` is **odd**, subtract `1` to make it even. This corresponds to reversing an increment operation.
- Increment the move counter after every operation.
3. Once we run out of doubles, we can no longer halve numbers. The only remaining valid operation is decrementing repeatedly until we reach `1`.
4. Since decrementing from `target` to `1` takes exactly `target - 1` operations, add that directly to `moves`.
5. Return the total number of moves.

### Why it works

The correctness comes from a greedy property. Whenever the current number is even and division is available, dividing immediately is always optimal because it reduces the target faster than any sequence of decrements could. For odd values, subtraction is forced because division is impossible. Once no doubles remain, repeated subtraction is the only legal move. Since every step makes the uniquely optimal choice, the algorithm always produces the minimum number of operations.

## Python Solution

```python
class Solution:
    def minMoves(self, target: int, maxDoubles: int) -> int:
        moves = 0

        while target > 1 and maxDoubles > 0:
            if target % 2 == 0:
                target //= 2
                maxDoubles -= 1
            else:
                target -= 1

            moves += 1

        moves += target - 1

        return moves
```

The implementation directly follows the reverse greedy strategy.

We begin with `moves = 0` to track the number of operations performed.

The main loop continues while two conditions hold: the target is greater than `1`, and we still have doubling operations available. Inside the loop, we inspect whether the target is even or odd.

If the number is even, we reverse a doubling operation by dividing by `2`, and we decrement `maxDoubles` because one doubling has been consumed.

If the number is odd, division is impossible, so we subtract `1`.

Each operation counts as one move, so we increment `moves`.

Once doubling is no longer available, we stop simulating step by step. The remaining distance to `1` must be covered entirely with decrements, which takes exactly `target - 1` moves.

## Go Solution

```go
func minMoves(target int, maxDoubles int) int {
    moves := 0

    for target > 1 && maxDoubles > 0 {
        if target%2 == 0 {
            target /= 2
            maxDoubles--
        } else {
            target--
        }

        moves++
    }

    moves += target - 1

    return moves
}
```

The Go implementation mirrors the Python logic closely.

Since Go integers are already sufficient for the constraint range (`10^9`), no special overflow handling is needed.

The loop condition and greedy logic remain identical. Integer division in Go naturally truncates toward zero, which is exactly what we want when halving an even number.

Unlike Python, Go does not use type hints, but the function signature already specifies integer parameters and return types.

## Edge Cases

### Edge Case 1: `target = 1`

If the target is already `1`, we start at the desired value and no operations are needed. A buggy implementation might accidentally perform unnecessary work or return a negative value when computing `target - 1`. This implementation handles the case naturally because the loop never executes, and `moves += target - 1` adds `0`.

### Edge Case 2: `maxDoubles = 0`

When no doubling operations are allowed, the answer is simply `target - 1` because incrementing is the only legal move. A naive implementation may still attempt greedy halving logic and fail. Here, the loop condition immediately fails, and we directly compute the remaining moves.

### Edge Case 3: Odd Targets During Reverse Processing

Odd numbers cannot be divided evenly by `2`. A common mistake is trying to halve odd numbers directly or using integer division incorrectly. The implementation explicitly checks parity using `target % 2`. For odd values, it subtracts `1` first, guaranteeing that future division becomes valid.

### Edge Case 4: Doubles Exhausted Before Reaching `1`

Sometimes we use all allowed doubles while the target is still large. For example, with `target = 100` and `maxDoubles = 1`, we can halve once to `50`, but afterward only decrements remain. Instead of simulating each decrement one by one, the implementation efficiently adds `target - 1` in one step.
