## Solution

## Problem Understanding

The problem gives us an array `nums` of length `2n`. This array was formed by combining two hidden arrays:

- `lower[i] = arr[i] - k`
- `higher[i] = arr[i] + k`

for some unknown positive integer `k`.

The original array `arr` has length `n`, and every value in `nums` belongs to either `lower` or `higher`. The order was lost, so `nums` is simply a shuffled mixture of both arrays.

Our task is to reconstruct any valid original array `arr`.

A very important observation is that every original value `arr[i]` creates exactly two numbers in `nums`:

```
arr[i] - k
arr[i] + k
```

The difference between these two numbers is:

```
(arr[i] + k) - (arr[i] - k) = 2k
```

That means if we sort the array, every valid pair must differ by the same even value `2k`.

The constraints are relatively small:

- `nums.length <= 2000`
- Therefore `n <= 1000`

This size allows us to try several candidate values of `k` and validate them efficiently.

Some important edge cases immediately stand out:

- Duplicate numbers, such as `[1,1,3,3]`
- Very large integers up to `10^9`
- Multiple valid answers
- Smallest possible input size, where `n = 1`
- Cases where many candidate differences are invalid because `k` must be positive

The problem guarantees that at least one valid solution exists, so we never need to handle the impossible case.

## Approaches

### Brute Force Approach

A brute force strategy would attempt every possible partition of the `2n` elements into two groups of size `n`.

For each partition:

- Treat one group as `lower`
- Treat the other as `higher`
- Check whether all paired values produce the same positive `k`

This works because it explores all possible assignments.

However, the number of partitions is enormous:

```
C(2n, n)
```

For `n = 1000`, this is completely infeasible.

Even much smaller inputs would already explode combinatorially.

### Key Insight

The key observation is that after sorting the array:

- The smallest element must belong to `lower`
- Its matching element in `higher` must be larger by exactly `2k`

Suppose the smallest value is `nums[0]`.

If we pair it with some other value `nums[i]`, then:

```
nums[i] - nums[0] = 2k
```

This immediately gives us a candidate value for `k`.

Not every difference works:

- The difference must be positive
- The difference must be even

Once we choose a candidate `k`, we can greedily attempt to pair numbers:

- For each unused number `x`
- Find another unused number `x + 2k`

If successful for all elements, then:

```
arr value = x + k
```

This transforms the problem into pairing numbers with a fixed difference.

Because the array size is only 2000, trying all possible candidates for `k` is efficient enough.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Tries all possible partitions |
| Optimal | O(n²) | O(n) | Enumerates valid `k` values and greedily pairs |

## Algorithm Walkthrough

1. Sort the input array.

Sorting is crucial because it allows us to process numbers from smallest to largest and greedily form valid pairs.
2. Let the smallest number be `nums[0]`.

Since `k > 0`, this value must belong to the `lower` array. It cannot belong to `higher`, because then there would need to exist a smaller corresponding `lower` value.
3. Try every possible pairing between `nums[0]` and `nums[i]`.

For every index `i > 0`, compute:

```
diff = nums[i] - nums[0]
```

Since:

```
diff = 2k
```

valid candidates must satisfy:

- `diff > 0`
- `diff % 2 == 0`
4. Compute the candidate value of `k`.

```
k = diff // 2
```
5. Attempt to reconstruct the original array using this `k`.

We use a frequency counter to track unused numbers.
6. Process numbers from smallest to largest.

For each number `x`:

- If it has already been fully used, skip it
- Otherwise we must pair it with `x + 2k`
7. Validate the pairing.

If `x + 2k` does not exist in the remaining frequencies, then this `k` is invalid.
8. Build the original array.

Once we successfully pair:

```
x
x + 2k
```

the original value is:

```
x + k
```
9. Return the constructed array as soon as a valid reconstruction is found.

The problem allows any valid answer.

### Why it works

The correctness comes from the invariant that every original value contributes exactly two numbers differing by `2k`.

After sorting, the smallest unused value must always belong to `lower`, because no smaller value exists to pair with it as a `higher` element.

Therefore, greedily pairing each smallest unused number with `x + 2k` is forced. If at any point the pair does not exist, the candidate `k` cannot be correct.

Since we try every possible valid `k` derived from the smallest element, we are guaranteed to eventually find a correct reconstruction.

## Python Solution

```python
from collections import Counter
from typing import List

class Solution:
    def recoverArray(self, nums: List[int]) -> List[int]:
        nums.sort()
        n = len(nums) // 2

        for i in range(1, len(nums)):
            diff = nums[i] - nums[0]

            if diff <= 0 or diff % 2 == 1:
                continue

            k = diff // 2
            freq = Counter(nums)
            original = []
            valid = True

            for x in nums:
                if freq[x] == 0:
                    continue

                y = x + 2 * k

                if freq[y] == 0:
                    valid = False
                    break

                freq[x] -= 1
                freq[y] -= 1

                original.append(x + k)

            if valid and len(original) == n:
                return original

        return []
```

The implementation starts by sorting the input array. This guarantees that we process candidate pairs in ascending order, which is necessary for the greedy strategy to work correctly.

Next, we iterate through every possible partner for the smallest element. Each difference gives a possible value for `2k`. Invalid differences are skipped immediately because `k` must be positive and integral.

For each candidate `k`, we create a frequency counter. This counter allows us to efficiently track which numbers are still available for pairing.

The main reconstruction loop processes numbers in sorted order. If a number has already been consumed by previous pairings, we skip it. Otherwise, we attempt to find its matching value `x + 2k`.

If the matching value is unavailable, the candidate `k` is invalid and we stop early.

When a valid pair is found, we decrement both frequencies and append the midpoint `x + k` to the reconstructed array.

As soon as a full valid reconstruction is built, we return it immediately.

## Go Solution

```go
package main

import "sort"

func recoverArray(nums []int) []int {
	sort.Ints(nums)

	n := len(nums) / 2

	for i := 1; i < len(nums); i++ {
		diff := nums[i] - nums[0]

		if diff <= 0 || diff%2 == 1 {
			continue
		}

		k := diff / 2

		freq := make(map[int]int)
		for _, num := range nums {
			freq[num]++
		}

		original := []int{}
		valid := true

		for _, x := range nums {
			if freq[x] == 0 {
				continue
			}

			y := x + 2*k

			if freq[y] == 0 {
				valid = false
				break
			}

			freq[x]--
			freq[y]--

			original = append(original, x+k)
		}

		if valid && len(original) == n {
			return original
		}
	}

	return []int{}
}
```

The Go implementation follows the same logic as the Python version.

A `map[int]int` is used instead of Python's `Counter`. Go slices are dynamically appended to while constructing the answer.

Integer overflow is not an issue because the constraints remain safely within Go's standard integer range on LeetCode platforms.

The algorithm also returns immediately once a valid reconstruction is found.

## Edge Cases

One important edge case involves duplicate numbers. For example, `[1,1,3,3]` requires pairing identical values multiple times. A naive implementation that only tracks presence rather than frequency could incorrectly reuse numbers. The frequency counter solves this by carefully decrementing counts after every successful pairing.

Another tricky case occurs when the difference between two candidate numbers is odd. Since the difference equals `2k`, it must always be even. If we forget this condition, we could incorrectly compute a fractional `k`. The implementation explicitly skips odd differences before attempting reconstruction.

Very small arrays also require careful handling. When `n = 1`, there is only one possible pair. The algorithm still works naturally because the single difference immediately determines `k`, and the reconstruction loop processes exactly one pair.

Large values near `10^9` can expose overflow bugs in some languages. The solution avoids unnecessary arithmetic complexity and only performs additions and subtractions within safe integer bounds. Both Python and Go handle these operations correctly under the given constraints.

Finally, inputs with multiple valid solutions are fully supported. The problem only requires returning any valid original array. The algorithm returns the first successful reconstruction it finds, which satisfies the requirement.
