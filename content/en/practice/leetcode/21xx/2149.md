## Solution

## Problem Understanding

The problem gives us an array `nums` that contains an equal number of positive and negative integers. The length of the array is always even, and every element is guaranteed to be non-zero because `1 <= |nums[i]|`.

Our goal is to rearrange the elements so that:

1. Adjacent elements always have opposite signs.
2. The relative ordering among positive numbers remains unchanged.
3. The relative ordering among negative numbers also remains unchanged.
4. The final array must start with a positive number.

This means the problem is not asking us to sort the array by value. Instead, it asks us to interleave positive and negative numbers while preserving the original ordering within each group.

For example, if the positive numbers appear in the order:

```
[3, 1, 2]
```

and the negative numbers appear in the order:

```
[-2, -5, -4]
```

then the final arrangement must alternate signs while preserving those sequences:

```
[3, -2, 1, -5, 2, -4]
```

The constraints are important:

- `nums.length` can be as large as `2 * 10^5`
- We must process large arrays efficiently
- The number of positive and negative values is guaranteed to be equal

Because of these constraints, any quadratic solution would be too slow. We need a linear time solution.

Several edge cases are already handled by the problem guarantees:

- The array always contains equal counts of positive and negative numbers
- The array length is always even
- No value is zero

However, there are still implementation pitfalls:

- Preserving relative order is essential
- The array must start with a positive number
- Alternation must be exact
- Small arrays such as length `2` must work correctly
- Large arrays require efficient memory and time usage

## Approaches

### Brute Force Approach

A brute force approach would repeatedly search for the next unused positive number and the next unused negative number, appending them alternately into the result array.

One way to implement this would be:

- Maintain a visited array
- For every even index in the result, scan the original array to find the next unused positive number
- For every odd index, scan again to find the next unused negative number

This approach works because it explicitly preserves ordering by always choosing the next unused matching element.

However, the repeated scanning is extremely inefficient. In the worst case, every selection may require traversing almost the entire array. Since there are `n` placements and each may cost `O(n)`, the total complexity becomes `O(n^2)`.

With `n` up to `200,000`, this is far too slow.

### Key Insight for the Optimal Solution

The most important observation is that the output pattern is completely predetermined:

- Even indices must contain positive numbers
- Odd indices must contain negative numbers

Since the relative order must remain unchanged, we can simply process the original array once:

- Place each positive number into the next available even index
- Place each negative number into the next available odd index

Because the number of positive and negative integers is equal, we are guaranteed that both sequences fit perfectly.

This leads naturally to a linear time solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeatedly scans for next unused positive or negative |
| Optimal | O(n) | O(n) | Single pass, directly places values into correct positions |

## Algorithm Walkthrough

### Optimal Algorithm

1. Create a result array `answer` of the same length as `nums`.

We need a separate array because rearranging in-place while preserving order is much more complicated. Since extra space is allowed, using a new array simplifies the solution considerably.
2. Initialize two pointers:

- `positive_index = 0`
- `negative_index = 1`

These pointers represent the next available positions for positive and negative numbers.

Since the array must start with a positive number:

- Positive numbers go to even indices
- Negative numbers go to odd indices
3. Traverse the input array from left to right.

Processing the array in original order guarantees stability, meaning the relative ordering of positive and negative numbers remains unchanged.
4. For each number:

- If the number is positive:

- Place it at `answer[positive_index]`
- Increase `positive_index` by `2`
- Otherwise:

- Place it at `answer[negative_index]`
- Increase `negative_index` by `2`
5. Continue until every element has been placed.
6. Return the result array.

### Why it works

The algorithm maintains two important invariants:

- Every positive number is placed into the next unused even index
- Every negative number is placed into the next unused odd index

Because we process elements in their original order, the relative ordering among numbers with the same sign is preserved.

Because even indices receive only positives and odd indices receive only negatives, every adjacent pair automatically has opposite signs.

Finally, since index `0` is even, the array always begins with a positive number.

## Python Solution

```python
from typing import List

class Solution:
    def rearrangeArray(self, nums: List[int]) -> List[int]:
        n = len(nums)
        answer = [0] * n
        
        positive_index = 0
        negative_index = 1
        
        for num in nums:
            if num > 0:
                answer[positive_index] = num
                positive_index += 2
            else:
                answer[negative_index] = num
                negative_index += 2
        
        return answer
```

The implementation begins by creating an output array `answer` with the same size as the input.

Two pointers are then initialized:

- `positive_index` starts at `0`
- `negative_index` starts at `1`

These pointers track the next valid insertion positions for positive and negative values.

The algorithm processes the input array exactly once. For each element:

- Positive values are inserted into even indices
- Negative values are inserted into odd indices

After insertion, the corresponding pointer advances by `2`, ensuring the next value of the same sign goes into the next valid slot.

Because the traversal order matches the original array order, stability is preserved automatically.

Finally, the completed rearranged array is returned.

## Go Solution

```go
func rearrangeArray(nums []int) []int {
	n := len(nums)
	answer := make([]int, n)

	positiveIndex := 0
	negativeIndex := 1

	for _, num := range nums {
		if num > 0 {
			answer[positiveIndex] = num
			positiveIndex += 2
		} else {
			answer[negativeIndex] = num
			negativeIndex += 2
		}
	}

	return answer
}
```

The Go implementation follows the same logic as the Python version.

The result slice is created using `make([]int, n)`.

Go slices automatically manage bounds safely as long as indices are valid. Since the problem guarantees equal numbers of positive and negative integers, both insertion pointers always remain within bounds.

There are no integer overflow concerns because the values are small relative to Go's integer capacity.

The `range` loop cleanly iterates through the input while preserving original ordering.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each element is processed exactly once |
| Space | O(n) | A separate output array of size `n` is created |

The algorithm performs a single traversal of the input array. Every operation inside the loop is constant time, so the total running time grows linearly with the input size.

The extra space comes entirely from the output array. Since the problem explicitly allows non in-place modification, this is acceptable and optimal for a simple stable solution.

## Edge Cases

One important edge case is the minimum allowed array length, where the array contains only two elements. A careless implementation might assume more indices exist or incorrectly advance pointers. The current solution handles this safely because each element is inserted exactly once into either index `0` or `1`.

Another important case occurs when all positive numbers appear before all negative numbers, or vice versa. Some naive alternating approaches might fail to preserve relative ordering. Since this solution processes elements in original order and appends them into fixed sign positions, ordering stability is guaranteed automatically.

A third critical edge case is when the input already satisfies the required arrangement. Some implementations unnecessarily swap or reorder elements, potentially breaking stability. This algorithm does not modify already placed elements and simply reconstructs the correct arrangement directly, so already valid inputs remain unchanged.

A final subtle edge case involves large input sizes near `200,000` elements. Quadratic approaches become too slow under these constraints. Because this solution performs only a single linear traversal and constant time insertions, it remains efficient even for the maximum input size.
