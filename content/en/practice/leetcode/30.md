## Solution

## Problem Understanding

The problem gives us a string `s` and a list of words called `words`. Every word in `words` has the same length. We must find every starting index in `s` where a substring exists that is formed by concatenating all the words exactly once, in any order, without extra characters in between.

Suppose `words = ["foo", "bar"]`. Since both words must appear exactly once and each word has length 3, any valid substring must have total length 6. The substring `"foobar"` is valid, and `"barfoo"` is also valid because the order does not matter. However, `"foofoo"` is invalid because `"bar"` is missing.

The output should contain all starting positions in `s` where such a valid concatenation begins.

A key observation is that every valid substring has a fixed size:

$$\text{totalLength} = \text{numberOfWords} \times \text{wordLength}$$

Because all words have identical lengths, we can split candidate substrings into equal-sized chunks and check whether those chunks match the required word frequencies.

The constraints are important:

- `s.length <= 10^4`
- `words.length <= 5000`
- `words[i].length <= 30`

These limits immediately tell us that generating all permutations of words is impossible. Even checking every substring with expensive operations could become too slow. We need an efficient method that avoids repeatedly rebuilding frequency counts from scratch.

Several edge cases are especially important:

- Duplicate words in `words`, such as `["foo", "foo", "bar"]`
- Strings where partial matches appear frequently
- Cases where no valid concatenation exists
- Very small inputs
- Situations where `s` is shorter than the required concatenation length

The problem guarantees that all words have the same length, which enables a sliding window strategy based on fixed-size chunks.

## Approaches

### Brute Force Approach

The most direct solution is to examine every possible starting index in `s`.

For each starting position:

1. Extract a substring whose length equals the total concatenation length.
2. Split that substring into chunks of size `wordLength`.
3. Count the frequency of each chunk.
4. Compare the chunk frequencies with the frequency map of `words`.

This approach is correct because every possible candidate substring is checked exhaustively. If a valid concatenation exists, it will eventually be examined.

However, this solution is inefficient. For every index in `s`, we rebuild a frequency map from scratch. If the substring length is large and there are many words, the repeated counting becomes expensive.

### Optimal Sliding Window Approach

The key insight is that all words have the same length.

Instead of checking every substring independently, we can process the string in aligned word-sized chunks. This allows us to use a sliding window that moves one word at a time rather than rebuilding counts repeatedly.

We maintain:

- A frequency map of required words
- A frequency map for the current window
- A left pointer and right pointer that move in steps of `wordLength`

We process the string starting from every possible offset:

- Offset 0
- Offset 1
- ...
- Offset `wordLength - 1`

This ensures we examine every valid word alignment.

As we expand the window:

- If the current word is valid, we add it to the window.
- If a word appears too many times, we shrink the window from the left until frequencies become valid again.
- If the number of matched words equals the required count, we record the starting index.

This avoids recomputing frequencies from scratch and gives a much more efficient solution.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × k × m) | O(k) | Rebuilds frequency map for every starting position |
| Optimal | O(n × m) | O(k) | Sliding window processes each word chunk efficiently |

Where:

- `n = len(s)`
- `k = len(words)`
- `m = len(words[0])`

## Algorithm Walkthrough

1. First, handle edge cases. If `s` or `words` is empty, return an empty list immediately because no valid concatenation can exist.
2. Compute the important constants:

- `wordLength`, the size of each word
- `wordCount`, the number of words
- `windowLength`, the total length of a valid concatenation
3. Build a frequency map called `targetCount`. This stores how many times each word must appear.
4. We now process the string using multiple sliding windows. Since words must align perfectly, we try every possible starting offset from `0` to `wordLength - 1`.
5. For each offset:

- Initialize `left` and `right` pointers.
- Create a hash map called `windowCount` to store word frequencies inside the current window.
- Track `matchedWords`, the number of words currently inside the window.
6. Move the `right` pointer forward one word at a time:

- Extract the next word using substring slicing.
- If the word is not in `targetCount`, reset the window because no valid concatenation can include this word.
- Otherwise, add the word to `windowCount`.
7. After adding a word, the frequency may exceed the allowed count. If that happens:

- Remove words from the left side of the window.
- Decrease their counts.
- Move `left` forward by one word each time.
- Continue until the window becomes valid again.
8. If `matchedWords == wordCount`, then:

- The current window contains exactly all required words.
- Record `left` as a valid starting index.
9. Continue processing until every aligned window has been examined.

The reason this works efficiently is that each word enters and leaves the window at most once per offset. Instead of repeatedly rebuilding counts, we update them incrementally.

## Python Solution

```
from typing import List
from collections import Counter, defaultdict

class Solution:
    def findSubstring(self, s: str, words: List[str]) -> List[int]:
        if not s or not words:
            return []

        word_length = len(words[0])
        word_count = len(words)
        total_length = word_length * word_count

        if len(s) < total_length:
            return []

        target_count = Counter(words)
        result = []

        for offset in range(word_length):
            left = offset
            right = offset

            window_count = defaultdict(int)
            matched_words = 0

            while right + word_length <= len(s):
                current_word = s[right:right + word_length]
                right += word_length

                if current_word not in target_count:
                    window_count.clear()
                    matched_words = 0
                    left = right
                    continue

                window_count[current_word] += 1
                matched_words += 1

                while window_count[current_word] > target_count[current_word]:
                    left_word = s[left:left + word_length]
                    window_count[left_word] -= 1
                    matched_words -= 1
                    left += word_length

                if matched_words == word_count:
                    result.append(left)

                    left_word = s[left:left + word_length]
                    window_count[left_word] -= 1
                    matched_words -= 1
                    left += word_length

        return result
```

The implementation begins by validating edge cases. If the input string or word list is empty, there is no possible answer.

Next, the code computes the fixed sizes that define the problem structure. Since every word has equal length, we can safely move through the string in increments of `word_length`.

`Counter(words)` builds the required frequency map. This allows duplicate words to be handled naturally.

The outer loop iterates through every alignment offset. This is necessary because valid words can start only at positions aligned with the word boundaries.

Inside the sliding window:

- `right` expands the window
- `left` shrinks the window when needed
- `window_count` stores frequencies inside the current window
- `matched_words` tracks how many valid chunks are currently included

When an invalid word appears, the entire window is reset because no concatenation can cross that boundary.

When a word appears too frequently, the left side of the window is contracted until the frequency becomes valid again.

Once the window contains exactly `word_count` valid words, the current `left` index is added to the answer.

After recording a valid window, the algorithm immediately removes the leftmost word so it can continue searching for overlapping matches efficiently.

## Go Solution

```
package main

func findSubstring(s string, words []string) []int {
	if len(s) == 0 || len(words) == 0 {
		return []int{}
	}

	wordLength := len(words[0])
	wordCount := len(words)
	totalLength := wordLength * wordCount

	if len(s) < totalLength {
		return []int{}
	}

	targetCount := make(map[string]int)
	for _, word := range words {
		targetCount[word]++
	}

	result := []int{}

	for offset := 0; offset < wordLength; offset++ {
		left := offset
		right := offset

		windowCount := make(map[string]int)
		matchedWords := 0

		for right+wordLength <= len(s) {
			currentWord := s[right : right+wordLength]
			right += wordLength

			if _, exists := targetCount[currentWord]; !exists {
				windowCount = make(map[string]int)
				matchedWords = 0
				left = right
				continue
			}

			windowCount[currentWord]++
			matchedWords++

			for windowCount[currentWord] > targetCount[currentWord] {
				leftWord := s[left : left+wordLength]
				windowCount[leftWord]--
				matchedWords--
				left += wordLength
			}

			if matchedWords == wordCount {
				result = append(result, left)

				leftWord := s[left : left+wordLength]
				windowCount[leftWord]--
				matchedWords--
				left += wordLength
			}
		}
	}

	return result
}
```

The Go implementation follows the same logic as the Python solution. The main difference is manual handling of maps and slices.

Go maps return zero values automatically for missing keys, which simplifies frequency counting. Resetting the window is done by allocating a new map rather than clearing an existing one.

String slicing in Go works efficiently because substrings are represented as views into the original string. Since the problem uses only lowercase English letters, byte indexing is completely safe.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n × m) | We process each alignment separately, but each word enters and leaves the window at most once |
| Space | O(k) | Hash maps store frequencies of at most all unique words |

Where:

- `n = len(s)`
- `m = wordLength`
- `k = len(words)`

Although there are `wordLength` separate passes through the string, each pass moves the sliding window efficiently. The total work remains linear relative to the size of the input string.

The space complexity comes from the frequency maps used to track required words and current window contents.

## Edge Cases

### Duplicate Words in the Input

A common source of bugs is handling duplicate words correctly.

For example:

```
words = ["foo", "foo", "bar"]
```

A naive implementation that only checks whether words exist would fail because frequency matters. The solution uses hash maps that store exact counts, ensuring duplicates are handled properly.

### String Shorter Than Required Concatenation

If:

```
len(s) < len(words) * len(words[0])
```

then a valid substring is impossible.

Without this check, the algorithm could waste time processing impossible windows or accidentally access invalid ranges. The implementation immediately returns an empty list in this situation.

### Invalid Words Breaking the Window

Consider:

```
s = "barxyzfoo"
words = ["bar", "foo"]
```

The substring `"xyz"` is not a valid word. Any concatenation crossing this section must be invalid.

The sliding window handles this by completely resetting whenever an unknown word appears. This avoids carrying invalid state forward.

### Overlapping Matches

Some valid answers overlap.

Example:

```
s = "aaaaaa"
words = ["aa", "aa"]
```

Valid answers occur at multiple nearby positions.

The implementation correctly handles overlaps because after recording a valid window, it removes only the leftmost word and continues sliding normally instead of resetting completely.

### Different Alignment Offsets

A subtle issue is that valid concatenations may begin at positions not divisible by `wordLength`.

For example, if `wordLength = 3`, valid windows could begin at indices:

```
0, 1, or 2
```

The algorithm handles this by running separate sliding windows for every possible offset. Without this step, some valid answers would be missed.
