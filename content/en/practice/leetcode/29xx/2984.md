## Solution

## Problem Understanding

This problem asks us to analyze phone call activity grouped by city and hour of the day. Each row in the `Calls` table represents a single call between two users, along with the timestamp of when the call happened and the city associated with that call.

The goal is to determine, for every city, which hour or hours had the highest number of calls. These are called the "peak calling hours".

The important detail is that multiple hours can tie for the maximum number of calls in a city. In that situation, all tied hours must appear in the final result.

The output must contain:

- `city`
- `peak_calling_hour`
- `number_of_calls`

The result should be sorted by:

1. `peak_calling_hour` in descending order
2. `city` in descending order

The input data contains a `datetime` column called `call_time`. We only care about the hour portion of the timestamp, meaning values from `0` to `23`.

For example:

- `2021-08-24 22:46:07` belongs to hour `22`
- `2021-08-17 13:07:00` belongs to hour `13`

The problem is fundamentally a grouping and aggregation problem. We need to:

1. Count how many calls happened for each `(city, hour)` pair.
2. Find the maximum count within each city.
3. Return every hour whose count equals that maximum.

The table guarantees uniqueness using the composite primary key `(caller_id, recipient_id, call_time)`, so duplicate rows are not a concern.

Several edge cases are important:

- A city may have only one call total.
- Multiple hours may tie for the highest count.
- Calls may occur at midnight (`0`) or late night (`23`).
- Different cities are processed independently, meaning the peak hour in one city has no effect on another.

Because this is a database problem, the intended solution uses SQL aggregation functions such as `COUNT`, `GROUP BY`, and ranking/window functions.

## Approaches

### Brute Force Approach

A brute force solution would process each city independently. For every city, we could iterate through all possible hours from `0` to `23` and count how many calls occurred during each hour by scanning the entire table repeatedly.

After computing all hourly counts for a city, we would determine the maximum count and return all matching hours.

This approach is correct because every city-hour combination is evaluated explicitly. However, it is inefficient because the table may be scanned many times. If there are `C` cities and `N` rows, repeatedly filtering the dataset leads to unnecessary repeated work.

The core inefficiency is recomputing counts from scratch instead of aggregating them once.

### Optimal Approach

The optimal solution uses SQL aggregation and window functions.

The key insight is that we only need two phases:

1. Aggregate the number of calls for each `(city, hour)` pair.
2. Within each city, determine which hours have the maximum count.

We can first use `GROUP BY city, HOUR(call_time)` to compute call frequencies.

Then, using a window function such as `DENSE_RANK()` or `RANK()`, we rank the hours inside each city based on call count in descending order.

All rows with rank `1` represent peak calling hours.

This avoids repeated scans and lets the database engine efficiently perform grouping and ranking operations.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(C × 24 × N) | O(C × 24) | Repeatedly scans the table for each city and hour |
| Optimal | O(N log N) | O(N) | Uses aggregation and window functions efficiently |

## Algorithm Walkthrough

1. Extract the hour from each `call_time` value using the SQL `HOUR()` function.

This converts timestamps into hourly buckets ranging from `0` to `23`.
2. Group the rows by both `city` and extracted hour.

For every `(city, hour)` pair, compute the total number of calls using `COUNT(*)`.

At this stage, we have a table like:

| city | hour | calls |
| --- | --- | --- |
| Houston | 22 | 3 |
| Houston | 21 | 1 |
| New York | 13 | 1 |
| New York | 14 | 1 |
3. Use a window ranking function partitioned by city.

We apply:

```
DENSE_RANK() OVER (
    PARTITION BY city
    ORDER BY number_of_calls DESC
)
```

This ranks hours within each city based on call frequency.
4. Select only rows where the rank equals `1`.

These are the peak calling hours because they have the highest number of calls in their city.
5. Sort the final output.

The problem requires ordering by:

- `peak_calling_hour` descending
- `city` descending

### Why it works

The algorithm works because the aggregation step computes the exact number of calls for every city-hour pair. The ranking step then compares only rows within the same city and assigns rank `1` to the highest frequency values. Since `DENSE_RANK()` gives the same rank to ties, all peak hours are correctly included.

## Python Solution

```python
# Write your MySQL query statement below

WITH hourly_calls AS (
    SELECT
        city,
        HOUR(call_time) AS peak_calling_hour,
        COUNT(*) AS number_of_calls
    FROM Calls
    GROUP BY city, HOUR(call_time)
),
ranked_calls AS (
    SELECT
        city,
        peak_calling_hour,
        number_of_calls,
        DENSE_RANK() OVER (
            PARTITION BY city
            ORDER BY number_of_calls DESC
        ) AS ranking
    FROM hourly_calls
)

SELECT
    city,
    peak_calling_hour,
    number_of_calls
FROM ranked_calls
WHERE ranking = 1
ORDER BY peak_calling_hour DESC, city DESC;
```

The solution begins by creating a common table expression called `hourly_calls`. This step groups all records by city and hour, then counts the number of calls in each group.

The second common table expression, `ranked_calls`, applies a `DENSE_RANK()` window function. The ranking is reset for every city because of `PARTITION BY city`.

The rows with the highest call counts receive rank `1`. Since `DENSE_RANK()` preserves ties, multiple peak hours are retained automatically.

Finally, the outer query filters to only rank `1` rows and sorts the results according to the problem requirements.

## Edge Cases

One important edge case occurs when multiple hours have the same maximum number of calls within a city. A naive solution using `ROW_NUMBER()` would incorrectly keep only one of them. Using `DENSE_RANK()` ensures all tied peak hours receive rank `1` and are included in the output.

Another edge case involves cities with only a single recorded call. In this situation, that hour is automatically the peak hour because its count is the maximum by default. The aggregation and ranking logic naturally handles this without requiring special conditions.

A third important edge case is calls occurring at boundary hours such as `0` and `23`. Incorrect datetime extraction logic could mishandle midnight or late-night timestamps. The use of the SQL `HOUR()` function guarantees correct extraction of the hour component for all valid datetime values.

A fourth edge case occurs when multiple cities share the same peak calling hour. Rankings must remain isolated per city. The `PARTITION BY city` clause ensures that peak detection is performed independently for every city rather than globally across the entire table.
