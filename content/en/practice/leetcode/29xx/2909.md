## Solution

## Problem Understanding

The problem asks us to find a special triplet of indices `(i, j, k)` in the array such that:

- `i < j < k`
- `nums[i] < nums[j]`
- `nums[k] < nums[j]`

This means the middle element must be strictly larger than both the left and right elements. The sequence forms a "mountain" shape, where the value rises to `nums[j]` and then falls afterward.

Among all valid mountain triplets, we must return the minimum possible sum:

$$nums[i] + nums[j] + nums[k]$$

If no such triplet exists, we return `-1`.

The input array can contain up to `10^5` elements, which is extremely important for choosing an algorithm. A naive cubic solution that checks all triplets would perform roughly:

$$O(n^3)$$

operations, which is far too slow for `n = 100000`.

The values themselves can be as large as `10^8`, but that does not significantly affect algorithm choice because we only perform comparisons and additions.

Several edge cases are important:

- Arrays that are strictly increasing cannot form a mountain because no right element is smaller than the middle.
- Arrays that are strictly decreasing cannot form a mountain because no left element is smaller than the middle.
- Duplicate values require careful handling because the inequalities are strict. Equal values do not satisfy the condition.
- The optimal triplet may not use the nearest smaller elements, but rather the globally smallest valid elements on both sides.

The key challenge is efficiently finding, for every position `j`, the smallest valid value on the left and the smallest valid value on the right.

## Approaches

### Brute Force Approach

The most direct solution is to try every possible triplet `(i, j, k)`.

We use three nested loops:

- The first loop selects `i`
- The second loop selects `j`
- The third loop selects `k`

For every triplet, we check whether:

```
nums[i] < nums[j]
nums[k] < nums[j]
```

If both conditions hold, we compute the sum and keep track of the minimum.

This approach is correct because it explicitly examines every possible triplet, so no valid answer can be missed.

However, the time complexity is extremely poor. With three nested loops, the runtime is:

$$O(n^3)$$

For `n = 100000`, this is completely infeasible.

### Optimal Approach

The key observation is that for every middle index `j`, we only care about:

- The smallest value to the left of `j` that is smaller than `nums[j]`
- The smallest value to the right of `j` that is smaller than `nums[j]`

If we know those two values, then the best mountain centered at `j` is immediately determined.

We can preprocess:

- `leftMin[j]` = smallest element seen before index `j`
- `rightMin[j]` = smallest element seen after index `j`

Then for each index `j`, we check:

```
leftMin[j] < nums[j]
rightMin[j] < nums[j]
```

If both are true, then:

```
leftMin[j] + nums[j] + rightMin[j]
```

is a valid mountain sum.

This reduces the problem to linear time because each preprocessing pass takes `O(n)`.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n³) | O(1) | Checks every possible triplet |
| Optimal | O(n) | O(n) | Uses prefix and suffix minimum arrays |

## Algorithm Walkthrough

1. Create an array `leftMin` of size `n`.

For every index `j`, `leftMin[j]` stores the minimum value that appears before index `j`.

We build it by scanning from left to right while maintaining the smallest value seen so far.
2. Create an array `rightMin` of size `n`.

For every index `j`, `rightMin[j]` stores the minimum value that appears after index `j`.

We build it by scanning from right to left while maintaining the smallest value seen so far.
3. Initialize the answer as infinity.

This allows us to track the minimum valid mountain sum.
4. Iterate through every possible middle index `j`.

For each index:

- Check whether `leftMin[j] < nums[j]`
- Check whether `rightMin[j] < nums[j]`

These conditions ensure the mountain property is satisfied.
5. If both conditions hold:

Compute:

$$leftMin[j] + nums[j] + rightMin[j]$$

Update the global minimum answer if this sum is smaller.
6. After processing all indices:

- If the answer is still infinity, no mountain triplet exists, so return `-1`
- Otherwise return the minimum sum found

### Why it works

For a fixed middle index `j`, the minimum possible mountain sum must use the smallest valid value on the left and the smallest valid value on the right. Any larger choice would only increase the total sum.

The preprocessing arrays guarantee that we can retrieve those smallest candidates in constant time for every `j`. Since every valid middle index is examined exactly once, the algorithm always finds the globally minimum mountain triplet.

## Python Solution

```python
from typing import List

class Solution:
    def minimumSum(self, nums: List[int]) -> int:
        n = len(nums)

        left_min = [0] * n
        right_min = [0] * n

        current_min = nums[0]
        left_min[0] = current_min

        for i in range(1, n):
            left_min[i] = current_min
            current_min = min(current_min, nums[i])

        current_min = nums[-1]
        right_min[-1] = current_min

        for i in range(n - 2, -1, -1):
            right_min[i] = current_min
            current_min = min(current_min, nums[i])

        answer = float("inf")

        for j in range(1, n - 1):
            if left_min[j] < nums[j] and right_min[j] < nums[j]:
                triplet_sum = left_min[j] + nums[j] + right_min[j]
                answer = min(answer, triplet_sum)

        return answer if answer != float("inf") else -1
```

The implementation begins by constructing the `left_min` array. During the left-to-right traversal, `current_min` always stores the smallest value encountered so far. Before updating it with the current element, we assign it to `left_min[i]`, which means `left_min[i]` represents the smallest value strictly before index `i`.

The same idea is used for `right_min`, except we traverse from right to left.

Once both preprocessing arrays are built, the algorithm checks every possible middle index. If both sides contain smaller values, then a valid mountain exists, and we compute its sum.

Finally, we return the smallest valid sum found, or `-1` if no mountain exists.

## Go Solution

```go
package main

import "math"

func minimumSum(nums []int) int {
	n := len(nums)

	leftMin := make([]int, n)
	rightMin := make([]int, n)

	currentMin := nums[0]
	leftMin[0] = currentMin

	for i := 1; i < n; i++ {
		leftMin[i] = currentMin

		if nums[i] < currentMin {
			currentMin = nums[i]
		}
	}

	currentMin = nums[n-1]
	rightMin[n-1] = currentMin

	for i := n - 2; i >= 0; i-- {
		rightMin[i] = currentMin

		if nums[i] < currentMin {
			currentMin = nums[i]
		}
	}

	answer := math.MaxInt32

	for j := 1; j < n-1; j++ {
		if leftMin[j] < nums[j] && rightMin[j] < nums[j] {
			sum := leftMin[j] + nums[j] + rightMin[j]

			if sum < answer {
				answer = sum
			}
		}
	}

	if answer == math.MaxInt32 {
		return -1
	}

	return answer
}
```

The Go implementation follows the same algorithmic structure as the Python version.

Slices are used for the prefix and suffix minimum arrays. Since Go does not have a built in infinity value for integers, `math.MaxInt32` is used as the initial answer sentinel.

Go integers are sufficient because the maximum possible sum is:

$$3 \times 10^8$$

which comfortably fits within a 32 bit signed integer.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Three linear passes through the array |
| Space | O(n) | Two auxiliary arrays are stored |

The algorithm performs:

- One pass to build `leftMin`
- One pass to build `rightMin`
- One pass to evaluate candidate middle indices

Each pass is linear, so the total runtime remains `O(n)`.

The additional memory comes from the two preprocessing arrays, each of size `n`.

## Edge Cases

One important edge case is an array with no valid mountain triplet. Arrays that are entirely increasing or entirely decreasing cannot satisfy both mountain conditions simultaneously. A careless implementation might accidentally return an uninitialized minimum value. This solution prevents that by initializing the answer to infinity and explicitly returning `-1` if no valid triplet is found.

Another important case involves duplicate values. The problem requires strict inequalities, meaning equal values are not allowed. For example, `[1,2,2,1]` does not allow the middle `2` values to use each other as valid smaller neighbors. The implementation correctly uses `<` rather than `<=`, ensuring duplicates are handled properly.

A third important edge case occurs when the smallest values are not adjacent to the middle index. For example:

```
[1,10,2,9,1]
```

The best mountain centered at `9` uses the distant `1` values rather than nearby larger values. The prefix and suffix minimum arrays guarantee that the globally smallest valid candidates are always considered, regardless of distance.
