## Solution

## Problem Understanding

The problem asks us to count how many distinct sub-multisets of the array `nums` have a sum within the inclusive range `[l, r]`.

A sub-multiset is different from a normal subset. In a normal subset problem, elements are usually distinguished by position. Here, elements are distinguished only by value and frequency. If the array contains duplicates, choosing the first `2` or the second `2` does not create different sub-multisets.

For example, if `nums = [1,2,2,3]`, then:

- `{1,2,3}` is one valid sub-multiset
- Choosing the first `2` versus the second `2` does not matter
- `{2}` appears only once as a sub-multiset, even though there are two copies of `2`

This means the problem is fundamentally about counting combinations of frequencies, not combinations of indices.

The input consists of:

- `nums`, a list of non-negative integers
- `l` and `r`, defining the allowed sum range

The output is the number of distinct sub-multisets whose total sum lies between `l` and `r`.

The constraints are extremely important:

- `nums.length <= 2 * 10^4`
- Sum of all elements is at most `2 * 10^4`
- Values themselves are also bounded by `2 * 10^4`

The critical observation is that the total possible sum is relatively small. Whenever subset-sum style problems have bounded total sum, dynamic programming over sums becomes feasible.

There are several edge cases that can easily cause bugs:

- Arrays containing many duplicates
- Arrays containing zeros
- `l = 0`, where the empty multiset may count
- Very large duplicate counts
- Situations where many different frequency selections lead to the same sum

Zeros are especially tricky because selecting additional zeros changes the multiset but does not change the sum. Since sub-multisets are defined by frequencies, different counts of zero produce different valid sub-multisets.

## Approaches

### Brute Force

A brute-force solution would generate every possible sub-multiset.

Suppose a value `x` appears `k` times. Then we may choose:

- `0` copies
- `1` copy
- `2` copies
- ...
- `k` copies

If there are many distinct values, the total number of combinations becomes:

$$\prod (freq[x] + 1)$$

For each generated multiset, we compute its sum and check whether it lies in `[l, r]`.

This approach is correct because it explicitly enumerates every valid frequency configuration exactly once.

However, it is far too slow. In the worst case, the number of multisets grows exponentially.

### Key Insight

The key insight is that this is actually a bounded knapsack counting problem.

For every distinct value `v` with frequency `f`, we may use the value:

- `0` times
- `1` time
- ...
- `f` times

We want to count how many ways produce each sum.

This naturally suggests dynamic programming.

Define:

$$dp[s]$$

as the number of sub-multisets with sum exactly `s`.

The transition becomes:

$$newDp[s] = \sum_{k=0}^{freq[v]} dp[s - k \cdot v]$$

A naive implementation of this transition would still be too slow because for every value and every sum we might iterate through all frequencies.

The optimization comes from using a sliding window over residue classes modulo `v`.

This reduces the transition from:

$$O(freq[v] \cdot r)$$

to:

$$O(r)$$

per distinct value.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Enumerates every frequency combination |
| Optimal | O(S) | O(S) | Dynamic programming with sliding window optimization |

Here, `S` is the maximum possible sum, bounded by `2 * 10^4`.

## Algorithm Walkthrough

### Step 1: Count Frequencies

First, count how many times each value appears.

For example:

```
nums = [1,2,2,3]
```

becomes:

```
1 -> 1
2 -> 2
3 -> 1
```

This converts the problem from dealing with array positions into dealing with value frequencies.

### Step 2: Initialize DP

Create a DP array:

```
dp[s] = number of ways to create sum s
```

Initially:

```
dp[0] = 1
```

because the empty multiset has sum `0`.

All other entries start as `0`.

### Step 3: Handle Zero Separately

If value `0` appears `z` times, then for every existing multiset we may choose:

- `0` zeros
- `1` zero
- ...
- `z` zeros

All of these choices keep the same sum.

Therefore every DP state is multiplied by:

$$z + 1$$

This is much simpler than including zero in the sliding window transitions.

### Step 4: Process Each Nonzero Value

Suppose value `v` appears `f` times.

We need:

$$newDp[s]
=
dp[s]
+
dp[s-v]
+
dp[s-2v]
+
...
+
dp[s-fv]$$

A direct implementation would be too slow.

### Step 5: Use Sliding Window Optimization

Observe that all transitions for a fixed value move in steps of `v`.

For each remainder:

$$0, 1, 2, ..., v-1$$

we process sums:

$$rem, rem+v, rem+2v, ...$$

using a sliding window.

The window maintains the sum of the last `f+1` DP states.

This transforms each transition into constant time amortized work.

### Step 6: Accumulate the Answer

After processing all values:

```
answer = sum(dp[l:r+1])
```

Return the result modulo:

$$10^9 + 7$$

### Why it works

The DP invariant is:

$$dp[s]$$

always equals the number of distinct sub-multisets that produce sum `s` using only the processed values.

When processing a value `v`, we consider all legal counts from `0` to `freq[v]`. The sliding window computes exactly this bounded sum efficiently. Since every distinct frequency configuration is counted exactly once, the final DP array contains the correct counts.

## Python Solution

```python
from typing import List
from collections import Counter

class Solution:
    def countSubMultisets(self, nums: List[int], l: int, r: int) -> int:
        MOD = 10**9 + 7
        
        freq = Counter(nums)
        
        dp = [0] * (r + 1)
        dp[0] = 1
        
        # Handle zeros separately
        zero_count = freq.get(0, 0)
        if zero_count:
            multiplier = zero_count + 1
            for s in range(r + 1):
                dp[s] = (dp[s] * multiplier) % MOD
        
        # Process nonzero values
        for value, count in freq.items():
            if value == 0:
                continue
            
            new_dp = [0] * (r + 1)
            
            for remainder in range(value):
                window_sum = 0
                
                for current_sum in range(remainder, r + 1, value):
                    window_sum += dp[current_sum]
                    
                    remove_index = current_sum - (count + 1) * value
                    if remove_index >= 0:
                        window_sum -= dp[remove_index]
                    
                    window_sum %= MOD
                    new_dp[current_sum] = window_sum
            
            dp = new_dp
        
        answer = sum(dp[l:r + 1]) % MOD
        return answer
```

The implementation starts by converting the input array into a frequency map. This is necessary because the problem counts multisets by value frequencies rather than array positions.

The DP array stores the number of ways to form every possible sum up to `r`. Since sums larger than `r` are irrelevant, we never compute them.

Zeros are processed separately because they do not affect the sum. If there are `z` zeros, every existing multiset can be extended in `z + 1` different ways while keeping the same sum.

For each nonzero value, we build a new DP array. The transition is optimized using sliding windows grouped by remainder modulo `value`.

The variable `window_sum` maintains:

$$dp[s] + dp[s-v] + ... + dp[s-fv]$$

for the current sum. As the window moves forward, old states are removed once they exceed the allowed frequency range.

Finally, we sum all valid sums in `[l, r]`.

## Go Solution

```go
package main

func countSubMultisets(nums []int, l int, r int) int {
	const MOD int = 1_000_000_007

	freq := map[int]int{}
	for _, num := range nums {
		freq[num]++
	}

	dp := make([]int, r+1)
	dp[0] = 1

	// Handle zeros separately
	if zeroCount, exists := freq[0]; exists {
		multiplier := zeroCount + 1

		for i := 0; i <= r; i++ {
			dp[i] = (dp[i] * multiplier) % MOD
		}
	}

	// Process nonzero values
	for value, count := range freq {
		if value == 0 {
			continue
		}

		newDP := make([]int, r+1)

		for remainder := 0; remainder < value; remainder++ {
			windowSum := 0

			for currentSum := remainder; currentSum <= r; currentSum += value {
				windowSum += dp[currentSum]

				removeIndex := currentSum - (count+1)*value
				if removeIndex >= 0 {
					windowSum -= dp[removeIndex]
				}

				windowSum %= MOD
				if windowSum < 0 {
					windowSum += MOD
				}

				newDP[currentSum] = windowSum
			}
		}

		dp = newDP
	}

	answer := 0
	for s := l; s <= r; s++ {
		answer += dp[s]
		answer %= MOD
	}

	return answer
}
```

The Go implementation follows exactly the same logic as the Python version.

One important Go-specific detail is modular arithmetic with negative numbers. After subtraction, `windowSum` may become negative, so we explicitly add `MOD` before storing the value.

Slices are used instead of dynamic arrays, and maps are used for frequency counting.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(S) | Each sum is processed a constant number of times |
| Space | O(S) | DP arrays of size `r + 1` |

Here:

$$S = r \le 2 \times 10^4$$

The sliding window optimization ensures that each DP state participates in constant amortized work. Without this optimization, the complexity would become much larger because every frequency count would need explicit iteration.

## Edge Cases

### Arrays Containing Only Zeros

This is one of the trickiest cases because zeros do not affect the sum. If there are `k` zeros, then there are actually `k + 1` distinct multisets corresponding to choosing different counts of zeros.

A naive subset-sum implementation often collapses all of these into one configuration because the sum never changes. The implementation handles this by multiplying all DP states by `zeroCount + 1`.

### Large Duplicate Frequencies

Suppose the input contains many copies of the same number, such as:

```
[2,2,2,2,2,...]
```

A naive bounded knapsack transition would iterate over every possible count repeatedly, causing large time complexity.

The sliding window optimization prevents this by maintaining the bounded transition sum incrementally.

### Empty Multiset

The empty multiset always exists and has sum `0`.

If `l = 0`, the answer must include it. Many implementations accidentally exclude the empty multiset by initializing DP incorrectly.

This solution correctly sets:

```
dp[0] = 1
```

before processing any values, ensuring the empty multiset is counted exactly once.
