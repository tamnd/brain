**Difficulty:** 🔴 Hard  
**Topics:** Math, Dynamic Programming, Combinatorics

## Solution

## Problem Understanding

The problem gives us an `n x m` grid where every cell is identified using 1-indexed coordinates `[x, y]`. We start at the cell `source` and want to reach the cell `dest` in exactly `k` moves.

A move is valid if:

- The next cell is in the same row, or
- The next cell is in the same column,
- But the next cell cannot be the current cell itself.

This means that from any cell, we may jump to any other cell in the same row or same column in a single move.

The task is to count how many distinct sequences of exactly `k` moves lead from `source` to `dest`. Since the number of paths can become extremely large, we return the answer modulo `10^9 + 7`.

The constraints are very important:

- `n` and `m` can be as large as `10^9`
- `k` can be up to `10^5`

The enormous grid dimensions immediately tell us that iterating over cells is impossible. Even storing the grid is infeasible. The solution must avoid any dependence on `n * m`.

The value of `k` is much smaller, so the intended solution likely performs dynamic programming over the number of moves rather than over grid cells.

A naive implementation could easily fail on several edge cases:

- When `source == dest`
- When only the row matches
- When only the column matches
- When neither coordinate matches
- When `n = 2` or `m = 2`, where transition counts become very small
- When `k = 1`, where only direct transitions matter

The problem guarantees that all coordinates are valid and inside the grid.

## Approaches

### Brute Force

The brute force approach would simulate every possible path of length `k`.

From a given cell:

- We can move to `m - 1` cells in the same row
- We can move to `n - 1` cells in the same column

So each state has roughly `n + m - 2` outgoing moves.

A recursive depth first search would try every sequence of moves and count the paths ending at `dest`.

This approach is correct because it explicitly enumerates all valid paths. However, its complexity grows exponentially with `k`.

For large values like `k = 10^5`, this is completely impossible.

### Key Insight

The critical observation is that we do not care about the exact cell at every step. We only care about the relationship between the current cell and the destination.

Every cell belongs to one of four categories relative to `dest`:

1. Exactly at `dest`
2. Same row as `dest`, but different column
3. Same column as `dest`, but different row
4. Different row and different column

All cells within the same category behave identically in terms of transition counts.

This allows us to compress the entire grid into only four dynamic programming states.

Instead of tracking positions, we track how many ways exist to be in each category after each move.

Because there are only four states, each transition step becomes constant time.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O((n + m)^k) | O(k) | Enumerates all possible paths recursively |
| Optimal | O(k) | O(1) | Uses four compressed DP states |

## Algorithm Walkthrough

We define four DP states:

- `dp0`: ways to be exactly at `dest`
- `dp1`: ways to be in the same row as `dest`, different column
- `dp2`: ways to be in the same column as `dest`, different row
- `dp3`: ways to be in a different row and different column

### State Initialization

We initialize the starting state based on the relationship between `source` and `dest`.

- If `source == dest`, then `dp0 = 1`
- Else if rows match, then `dp1 = 1`
- Else if columns match, then `dp2 = 1`
- Else `dp3 = 1`

All other states start at zero.

### Transition Rules

At each move, we compute the next values for all four states.

#### Transition into `dp0`

To end exactly at `dest`:

- From `dp1`, we can move horizontally into `dest`
- From `dp2`, we can move vertically into `dest`

So:

```
new0 = dp1 + dp2
```

### Transition into `dp1`

To end in the same row as `dest`, different column:

- From `dp0`, we can move to any of `m - 1` cells in the row
- From another `dp1` cell, we can move to `m - 2` other same-row cells
- From `dp3`, we can move vertically into the destination row

So:

```
new1 =
    dp0 * (m - 1)
    + dp1 * (m - 2)
    + dp3
```

### Transition into `dp2`

Symmetrically:

```
new2 =
    dp0 * (n - 1)
    + dp2 * (n - 2)
    + dp3
```

### Transition into `dp3`

To end in a completely different row and column:

- From `dp1`, move vertically outside destination row
- From `dp2`, move horizontally outside destination column
- From `dp3`, remain outside both

So:

```
new3 =
    dp1 * (n - 1)
    + dp2 * (m - 1)
    + dp3 * (n + m - 4)
```

We repeat these transitions exactly `k` times.

The final answer is `dp0`.

### Why it works

The algorithm works because every cell inside the same category has identical transition behavior.

For example, every cell sharing the destination row but not the destination column has exactly:

- One move into `dest`
- `m - 2` moves within the same category
- `n - 1` moves into the fully different category

Since transitions depend only on the category and not the exact coordinates, the four-state DP perfectly captures all possible paths without losing information.

## Python Solution

```python
from typing import List

class Solution:
    def numberOfWays(
        self,
        n: int,
        m: int,
        k: int,
        source: List[int],
        dest: List[int]
    ) -> int:
        
        MOD = 10**9 + 7

        sx, sy = source
        dx, dy = dest

        dp0 = dp1 = dp2 = dp3 = 0

        if sx == dx and sy == dy:
            dp0 = 1
        elif sx == dx:
            dp1 = 1
        elif sy == dy:
            dp2 = 1
        else:
            dp3 = 1

        for _ in range(k):
            new0 = (dp1 + dp2) % MOD

            new1 = (
                dp0 * (m - 1)
                + dp1 * (m - 2)
                + dp3
            ) % MOD

            new2 = (
                dp0 * (n - 1)
                + dp2 * (n - 2)
                + dp3
            ) % MOD

            new3 = (
                dp1 * (n - 1)
                + dp2 * (m - 1)
                + dp3 * (n + m - 4)
            ) % MOD

            dp0, dp1, dp2, dp3 = new0, new1, new2, new3

        return dp0
```

The implementation directly follows the four-state dynamic programming model.

First, we classify the starting position into one of the four categories relative to the destination.

The variables `dp0`, `dp1`, `dp2`, and `dp3` store the number of ways to be in each category after the current number of moves.

For every iteration, we compute the next state values using the transition equations derived earlier.

Because all transitions only depend on the previous step, we only need four integers instead of a full DP table.

Every arithmetic operation is taken modulo `10^9 + 7` to prevent overflow and satisfy the problem requirements.

Finally, after exactly `k` moves, `dp0` contains the number of ways to land exactly on `dest`.

## Go Solution

```go
func numberOfWays(n int, m int, k int, source []int, dest []int) int {
	const MOD int64 = 1_000_000_007

	sx, sy := source[0], source[1]
	dx, dy := dest[0], dest[1]

	var dp0, dp1, dp2, dp3 int64

	if sx == dx && sy == dy {
		dp0 = 1
	} else if sx == dx {
		dp1 = 1
	} else if sy == dy {
		dp2 = 1
	} else {
		dp3 = 1
	}

	for i := 0; i < k; i++ {
		new0 := (dp1 + dp2) % MOD

		new1 := (
			dp0*int64(m-1) +
				dp1*int64(m-2) +
				dp3
		) % MOD

		new2 := (
			dp0*int64(n-1) +
				dp2*int64(n-2) +
				dp3
		) % MOD

		new3 := (
			dp1*int64(n-1) +
				dp2*int64(m-1) +
				dp3*int64(n+m-4)
		) % MOD

		dp0, dp1, dp2, dp3 = new0, new1, new2, new3
	}

	return int(dp0)
}
```

The Go implementation mirrors the Python logic almost exactly.

The main difference is integer handling. Since intermediate values can become very large, the implementation uses `int64` for all DP variables and multiplication operations.

Slices are used for `source` and `dest`, matching the LeetCode function signature.

Go does not support tuple unpacking in the same way Python does, but simultaneous assignment is still possible for updating the DP states.

## Edge Cases

One important edge case occurs when `source == dest`. A naive implementation may incorrectly assume that staying in place is allowed, but the problem explicitly forbids moving to the same cell. Our DP initialization correctly places the starting configuration into `dp0`, and all future transitions require actual movement.

Another tricky case appears when `n = 2` or `m = 2`. In these situations, terms like `m - 2` or `n - 2` become zero. Incorrect implementations sometimes produce negative counts or invalid transitions. Our formulas naturally handle these cases because the transition counts correctly reduce to zero.

A third important edge case occurs when the destination cannot be reached in one move. For example, if the source and destination differ in both row and column and `k = 1`, the answer must be zero because every valid move changes only one coordinate relationship at a time. The DP transitions correctly produce zero for `dp0` after one iteration in this scenario.

Finally, very large values of `n`, `m`, and `k` can easily overflow standard integers. The implementation consistently applies modulo arithmetic during every update step, and the Go solution uses `int64` for all intermediate calculations to avoid overflow bugs.
