# LeetCode 2974 - Minimum Number Game

## Problem Understanding

The problem gives us an integer array `nums` with an even number of elements. Alice and Bob repeatedly play rounds using the following rules:

First, Alice removes the smallest number currently in `nums`. Then Bob removes the next smallest number. After that, Bob appends his removed number into a result array `arr`, and finally Alice appends hers.

This process continues until all numbers are removed from `nums`.

The important detail is the order in which numbers are appended. Alice removes first, but Bob appends first. Because both always remove the smallest available elements, each round effectively works on the two smallest remaining numbers.

Suppose the two smallest numbers are:

```
a <= b
```

Alice removes `a`, Bob removes `b`, and the result array receives:

```
[b, a]
```

So every pair of smallest elements gets swapped before being added to the answer.

The input constraints are very small:

```
2 <= nums.length <= 100
1 <= nums[i] <= 100
```

This tells us performance is not a major concern. Even relatively inefficient solutions will pass comfortably. Still, we should aim for the cleanest and most natural solution.

An important guarantee is that the array length is always even. That means every round will always have exactly two numbers available for Alice and Bob.

Some edge cases worth noticing:

- The smallest valid input has only two elements.
- The array may already be sorted.
- The array may contain duplicate values.
- The array may be in completely random order.

Because duplicates are allowed, we must make sure the algorithm handles equal values correctly.

## Approaches

### Brute Force Approach

A direct simulation follows the game rules literally.

In every round:

1. Find the minimum element for Alice and remove it.
2. Find the next minimum element for Bob and remove it.
3. Append Bob's number first, then Alice's number.

We repeat until the array becomes empty.

This works correctly because it exactly mirrors the rules of the game. However, removing minimum elements repeatedly from a normal array is inefficient. Each minimum search costs `O(n)`, and removing from the middle of an array also costs `O(n)`.

With `n/2` rounds, the total complexity becomes `O(n^2)`.

Even though the constraints are tiny and this would still pass, there is a much simpler observation that leads to cleaner code.

### Optimal Approach

The key observation is that every round always uses the two smallest remaining numbers.

If we sort the array first, then the numbers naturally appear in the exact order they would be removed during the game.

For a sorted array:

```
[a, b, c, d, e, f]
```

The rounds become:

- Alice removes `a`, Bob removes `b` → append `[b, a]`
- Alice removes `c`, Bob removes `d` → append `[d, c]`
- Alice removes `e`, Bob removes `f` → append `[f, e]`

So after sorting, we only need to swap every adjacent pair.

This produces a very clean solution.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeatedly searches and removes minimum elements |
| Optimal | O(n log n) | O(1) extra, excluding output | Sort once, then swap adjacent pairs |

## Algorithm Walkthrough

1. Sort the array in non-decreasing order.

After sorting, the elements appear in the same order they would be removed during the game. The first two elements are the smallest pair for the first round, the next two are for the second round, and so on.
2. Traverse the array two elements at a time.

At indices `i` and `i + 1`, we have:

```
nums[i] <= nums[i + 1]
```

Alice would remove `nums[i]`, and Bob would remove `nums[i + 1]`.
3. Append the pair in reversed order.

Since Bob appends first, we place:

```
nums[i + 1], nums[i]
```

into the result array.
4. Continue until all pairs are processed.

Because the array length is guaranteed to be even, every iteration safely processes exactly two elements.

### Why it works

Sorting guarantees that the numbers appear in the exact order they would be removed from the game. Each round always consumes the two smallest remaining elements. By processing adjacent pairs in the sorted array and reversing each pair, we exactly simulate Bob appending before Alice. Therefore, the constructed array is identical to the result of the game.

## Python Solution

```python
from typing import List

class Solution:
    def numberGame(self, nums: List[int]) -> List[int]:
        nums.sort()
        
        result = []
        
        for i in range(0, len(nums), 2):
            result.append(nums[i + 1])
            result.append(nums[i])
        
        return result
```

The implementation starts by sorting `nums`. This transforms the repeated "remove minimum" operation into a simple left-to-right traversal.

The `result` array stores the final answer.

The loop increments by 2 because each game round consumes exactly two numbers. For every adjacent pair, the larger element is appended first to simulate Bob's move, followed by the smaller element to simulate Alice's move.

Finally, the completed result array is returned.

## Go Solution

```go
package main

import "sort"

func numberGame(nums []int) []int {
	sort.Ints(nums)

	result := make([]int, 0, len(nums))

	for i := 0; i < len(nums); i += 2 {
		result = append(result, nums[i+1])
		result = append(result, nums[i])
	}

	return result
}
```

The Go solution follows the same logic as the Python version.

The main difference is that Go uses `sort.Ints(nums)` for sorting and explicitly creates a result slice with preallocated capacity using:

```go
make([]int, 0, len(nums))
```

This avoids unnecessary reallocations while appending.

There are no integer overflow concerns because values are very small. The input length is guaranteed to be even, so accessing `nums[i+1]` is always safe.

## Edge Cases

One important edge case is the smallest possible input size, where the array contains only two elements. In this scenario there is only one round of the game. A buggy implementation might incorrectly assume more rounds exist or mishandle indexing. The solution handles this naturally because the loop processes exactly one pair.

Another important case is duplicate values. For example:

```
[1,1,2,2]
```

Some implementations may incorrectly assume distinct values when searching for minimum elements. Sorting and swapping adjacent pairs works perfectly even when values are equal, because equal numbers remain adjacent and are still processed correctly.

A third edge case is reverse sorted input such as:

```
[4,3,2,1]
```

If an implementation forgets to sort before pairing, it would produce the wrong answer. Sorting guarantees that the game simulation always processes the true smallest remaining elements in every round.

Finally, arrays where all values are identical can expose unnecessary assumptions about ordering. For example:

```
[7,7]
```

The algorithm still works because swapping equal values leaves the array unchanged, which correctly matches the game behavior.
