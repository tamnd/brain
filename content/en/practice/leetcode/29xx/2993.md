## Solution

## Problem Understanding

The problem provides a database table named `Purchases`, where each row represents a single purchase made by a user during November 2023. Every row contains three fields:

- `user_id`, the identifier of the user making the purchase
- `purchase_date`, the exact date of the purchase
- `amount_spend`, the dollar amount spent in that transaction

The primary key is the combination `(user_id, purchase_date, amount_spend)`, which guarantees that no two rows are identical.

The task is to calculate the total amount spent on every Friday in November 2023. However, there is an important condition: we only include Fridays that actually have at least one purchase. If a Friday exists in the calendar but no transactions occurred on that date, it should not appear in the output.

For each qualifying Friday, we must output:

- `week_of_month`, representing which week of November the Friday belongs to
- `purchase_date`, the actual Friday date
- `total_amount`, the sum of all purchases made on that Friday

The result must be ordered by `week_of_month` in ascending order.

The weeks of November 2023 are interpreted naturally based on the calendar:

| Week of Month | Friday |
| --- | --- |
| 1 | 2023-11-03 |
| 2 | 2023-11-10 |
| 3 | 2023-11-17 |
| 4 | 2023-11-24 |

The input size is relatively small because the dates are restricted to November 2023 only. This means performance is not a major concern, but correctness in date calculations is essential.

Several edge cases are important:

- Some Fridays may have no purchases and therefore must be excluded.
- Multiple users may purchase on the same Friday, requiring aggregation with `SUM`.
- Purchases on non-Friday dates must be ignored completely.
- The final ordering must be based on the computed week number, not the purchase amount or date string ordering alone.

## Approaches

### Brute Force Approach

A brute-force solution would explicitly iterate through every Friday in November 2023 and scan the entire `Purchases` table for matching rows.

For example, we could:

1. Generate all Friday dates in November.
2. For each Friday, iterate through every purchase row.
3. Check whether the row's `purchase_date` matches the Friday.
4. Sum all matching `amount_spend` values.
5. Add the result if the sum is greater than zero.

This approach is correct because every purchase is checked against every Friday. However, it performs unnecessary repeated scans of the table.

If there are `F` Fridays and `N` purchase rows, the complexity becomes `O(F × N)`. While acceptable for small inputs, it is inefficient compared to directly grouping relevant rows.

### Optimal Approach

The key observation is that we only care about rows where the purchase occurred on a Friday. Instead of repeatedly scanning the table, we can:

1. Filter rows to keep only Fridays.
2. Compute the week number directly from the date.
3. Group by Friday date and aggregate the spending.

SQL databases are designed specifically for this kind of filtering and aggregation. Functions like `DAYOFWEEK`, `DAY`, and `SUM` allow us to express the solution compactly and efficiently.

The week number can be computed using:

```
WEEK_OF_MONTH = CEIL(DAY(purchase_date) / 7)
```

For example:

| Date | Day Number | Week |
| --- | --- | --- |
| 2023-11-03 | 3 | 1 |
| 2023-11-10 | 10 | 2 |
| 2023-11-17 | 17 | 3 |
| 2023-11-24 | 24 | 4 |

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(F × N) | O(1) | Repeatedly scans all purchases for each Friday |
| Optimal | O(N) | O(1) excluding output | Single pass filtering and aggregation using SQL grouping |

## Algorithm Walkthrough

### Optimal SQL Algorithm

1. Scan all rows in the `Purchases` table.

We must inspect every purchase because any row could potentially belong to a Friday.
2. Filter rows where the purchase date is a Friday.

In MySQL, `DAYOFWEEK(date)` returns:

| Day | Value |
| --- | --- |
| Sunday | 1 |
| Monday | 2 |
| Tuesday | 3 |
| Wednesday | 4 |
| Thursday | 5 |
| Friday | 6 |
| Saturday | 7 |

Therefore, we keep only rows where:

```sql
DAYOFWEEK(purchase_date) = 6
```
3. Compute the week of the month.

We derive the week number using:

```sql
CEIL(DAY(purchase_date) / 7)
```

This converts calendar days into week buckets.
4. Group rows by the Friday date.

Every Friday should produce one output row containing the total spending for that day.
5. Aggregate the spending using `SUM(amount_spend)`.

This combines all purchases that occurred on the same Friday.
6. Sort the result by `week_of_month`.

The problem explicitly requires ascending week order.

### Why it works

The algorithm works because every purchase row is examined exactly once. Non-Friday rows are excluded immediately, while Friday rows are grouped by their exact date. Since each Friday belongs to exactly one week of the month, the computed `week_of_month` value is correct. Aggregating with `SUM` guarantees that all spending on the same Friday is combined into a single total.

## Python Solution

Although this is a database problem and LeetCode expects SQL, the following Python implementation demonstrates the same logic programmatically.

```python
from collections import defaultdict
from datetime import datetime
from typing import List, Dict

class Solution:
    def fridayPurchases(self, purchases: List[List]) -> List[Dict]:
        friday_totals = defaultdict(int)

        for user_id, purchase_date, amount_spend in purchases:
            date_obj = datetime.strptime(purchase_date, "%Y-%m-%d")

            # Monday = 0, Friday = 4
            if date_obj.weekday() == 4:
                friday_totals[purchase_date] += amount_spend

        result = []

        for purchase_date in sorted(friday_totals.keys()):
            day_of_month = int(purchase_date[-2:])
            week_of_month = (day_of_month - 1) // 7 + 1

            result.append({
                "week_of_month": week_of_month,
                "purchase_date": purchase_date,
                "total_amount": friday_totals[purchase_date]
            })

        return result
```

The implementation begins by iterating through every purchase row. Each date string is converted into a `datetime` object so we can determine whether the date is a Friday.

Python's `weekday()` method returns integers from `0` to `6`, where Friday corresponds to `4`. Only Friday purchases are accumulated.

A `defaultdict(int)` is used because it simplifies aggregation. Every Friday date automatically starts with a total of zero.

After aggregation, we compute the week number using integer arithmetic:

```python
(day_of_month - 1) // 7 + 1
```

This correctly maps days 1 to 7 into week 1, days 8 to 14 into week 2, and so on.

Finally, the results are sorted chronologically and returned in the required structure.

## Go Solution

```go
package main

import (
	"sort"
	"time"
)

type Purchase struct {
	UserID      int
	PurchaseDate string
	AmountSpend int
}

type Result struct {
	WeekOfMonth int
	PurchaseDate string
	TotalAmount int
}

func fridayPurchases(purchases []Purchase) []Result {
	fridayTotals := make(map[string]int)

	for _, purchase := range purchases {
		dateObj, _ := time.Parse("2006-01-02", purchase.PurchaseDate)

		if dateObj.Weekday() == time.Friday {
			fridayTotals[purchase.PurchaseDate] += purchase.AmountSpend
		}
	}

	var dates []string

	for date := range fridayTotals {
		dates = append(dates, date)
	}

	sort.Strings(dates)

	var results []Result

	for _, date := range dates {
		day := 0
		timeObj, _ := time.Parse("2006-01-02", date)
		day = timeObj.Day()

		weekOfMonth := (day-1)/7 + 1

		results = append(results, Result{
			WeekOfMonth: weekOfMonth,
			PurchaseDate: date,
			TotalAmount: fridayTotals[date],
		})
	}

	return results
}
```

The Go solution follows the same logical structure as the Python implementation. A `map[string]int` is used for aggregation because Go does not have a built-in defaultdict equivalent.

Date parsing uses Go's reference date format:

```go
"2006-01-02"
```

Go's `Weekday()` method provides named constants like `time.Friday`, making the Friday check very readable.

The final results are sorted lexicographically by date string, which works correctly because the dates are formatted as `YYYY-MM-DD`.

## SQL Solution

```sql
SELECT
    CEIL(DAY(purchase_date) / 7) AS week_of_month,
    purchase_date,
    SUM(amount_spend) AS total_amount
FROM Purchases
WHERE DAYOFWEEK(purchase_date) = 6
GROUP BY purchase_date
ORDER BY week_of_month;
```

This SQL solution directly implements the optimal approach:

- `DAYOFWEEK(purchase_date) = 6` filters Fridays
- `CEIL(DAY(purchase_date) / 7)` computes the week number
- `SUM(amount_spend)` aggregates spending
- `GROUP BY purchase_date` combines all purchases on the same Friday
- `ORDER BY week_of_month` ensures correct output order

## Edge Cases

### No Friday Purchases

One important edge case occurs when the table contains purchases, but none happen on a Friday. A naive implementation might incorrectly output zero-value rows for every Friday in November. The correct behavior is to return an empty result set because only Fridays with at least one purchase should appear.

The implementation handles this naturally by filtering for Fridays first. If no Friday rows exist, the aggregation structure remains empty and no output rows are generated.

### Multiple Purchases on the Same Friday

Another common source of bugs is failing to aggregate multiple transactions occurring on the same Friday. If each row were output independently, the result would contain duplicate Friday dates instead of a single combined total.

The solution avoids this issue by grouping rows using the purchase date as the key and summing the `amount_spend` values.

### Purchases on Non-Friday Dates

A naive implementation might accidentally include all dates in the aggregation. Since the problem specifically requests Friday spending only, every other weekday must be ignored completely.

The implementation explicitly checks the weekday before aggregation. Only rows passing the Friday condition contribute to the totals.

### Correct Week Number Calculation

Week calculation can easily become incorrect if calendar logic is misunderstood. The problem defines weeks based on day ranges within November, not ISO week numbering.

Using:

```
(day - 1) // 7 + 1
```

correctly maps:

- Days 1 to 7 into week 1
- Days 8 to 14 into week 2
- Days 15 to 21 into week 3
- Days 22 to 28 into week 4
- Days 29 to 30 into week 5

This guarantees accurate week labeling for all November dates.
