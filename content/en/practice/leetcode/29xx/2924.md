## Solution

## Problem Understanding

This problem models a tournament as a directed acyclic graph, or DAG. Each team is represented as a node, and each directed edge represents a strength relationship between two teams.

If there is a directed edge from `u` to `v`, it means team `u` is stronger than team `v`.

The task is to identify the tournament champion. A team is considered the champion if no other team is stronger than it. In graph terminology, this means we are looking for a node with an incoming degree of `0`, meaning no edges point into it.

The problem adds one important condition: the champion must be unique. If multiple teams have no incoming edges, then there is no single strongest team, so we must return `-1`.

The input consists of:

- `n`, the number of teams labeled from `0` to `n - 1`
- `edges`, a list of directed edges where `edges[i] = [ui, vi]`

The output should be:

- The unique champion team index, if exactly one exists
- `-1` otherwise

The constraints are small:

- `1 <= n <= 100`
- The graph is guaranteed to be a DAG
- The graph also satisfies transitive consistency

These guarantees simplify the problem significantly. Since the graph has no cycles, strength relationships are valid and non-contradictory. We never need to worry about situations where team A beats B, B beats C, and C beats A.

Several edge cases are important:

- A graph with no edges at all, every node has indegree `0`, so there is no unique champion
- A graph with exactly one node, that node is trivially the champion
- Multiple disconnected components may produce multiple indegree-zero nodes
- A fully ordered DAG will always have exactly one source node

The key observation is that the champion corresponds exactly to the unique node with indegree `0`.

## Approaches

### Brute Force Approach

A brute force solution would examine every team and determine whether any other team is stronger than it.

For each node `i`, we scan all edges and check whether there exists an edge ending at `i`. If no such edge exists, then `i` is a candidate champion.

At the end, if exactly one candidate exists, we return it. Otherwise, we return `-1`.

This works because a champion is defined as a team with no stronger incoming relationship.

However, this approach repeatedly scans the edge list for every node, which is inefficient.

### Optimal Approach

The better solution comes from recognizing that we only care about indegrees.

The indegree of a node is the number of incoming edges. A champion must have indegree `0`.

Instead of repeatedly scanning the edge list, we compute the indegree of every node in a single pass through `edges`.

After building the indegree array:

- Count how many nodes have indegree `0`
- If exactly one exists, return that node
- Otherwise, return `-1`

This is efficient because each edge is processed only once.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * m) | O(1) | Repeatedly scans all edges for every node |
| Optimal | O(n + m) | O(n) | Computes indegrees once and finds unique source |

## Algorithm Walkthrough

### Optimal Algorithm

1. Create an indegree array of size `n`, initialized to `0`.

The indegree array tracks how many incoming edges each node has. Since incoming edges represent stronger teams, a node with indegree `0` has no stronger team above it.
2. Traverse every edge `[u, v]` in `edges`.

Since the edge points into `v`, increment `indegree[v]`.
3. After processing all edges, scan the indegree array.

Any node with indegree `0` is a potential champion.
4. Count how many nodes have indegree `0`.

If more than one node has indegree `0`, then multiple teams are not weaker than anyone else, so there is no unique champion.
5. If exactly one node has indegree `0`, return that node.
6. Otherwise, return `-1`.

### Why it works

A champion is defined as a node with no stronger incoming relationship. In graph terms, that is exactly a node with indegree `0`.

If multiple nodes have indegree `0`, then multiple teams are undefeated by stronger teams, so no unique champion exists.

Since every edge is processed correctly and indegrees accurately represent incoming strength relationships, the algorithm always identifies the correct answer.

## Python Solution

```python
from typing import List

class Solution:
    def findChampion(self, n: int, edges: List[List[int]]) -> int:
        indegree = [0] * n

        for u, v in edges:
            indegree[v] += 1

        champion = -1

        for team in range(n):
            if indegree[team] == 0:
                if champion != -1:
                    return -1
                champion = team

        return champion
```

The implementation begins by creating an `indegree` array initialized with zeros.

Each directed edge `[u, v]` means `v` has one stronger team pointing to it, so we increment `indegree[v]`.

After building the indegree array, we iterate through all teams and look for nodes with indegree `0`.

The variable `champion` stores the current candidate. If we encounter another indegree-zero node after already finding one, we immediately return `-1` because the champion is not unique.

If exactly one such node exists, its index is returned.

## Go Solution

```go
func findChampion(n int, edges [][]int) int {
    indegree := make([]int, n)

    for _, edge := range edges {
        v := edge[1]
        indegree[v]++
    }

    champion := -1

    for team := 0; team < n; team++ {
        if indegree[team] == 0 {
            if champion != -1 {
                return -1
            }
            champion = team
        }
    }

    return champion
}
```

The Go implementation follows the same logic as the Python version.

The `indegree` slice stores incoming edge counts for each team. Go slices are automatically initialized to zero values, so no extra setup is needed.

The algorithm processes all edges once, then scans for nodes with indegree `0`.

No special handling for integer overflow is necessary because the constraints are very small.

## Edge Cases

### Graph With No Edges

If the graph contains no edges, then every node has indegree `0`.

This is an important edge case because a naive implementation might incorrectly return the first node encountered. However, the problem requires a unique champion.

The implementation correctly detects multiple indegree-zero nodes and returns `-1`.

### Single Node Graph

When `n = 1`, there is only one team and no edges.

That single node trivially has indegree `0`, so it must be the champion.

The implementation handles this naturally because the indegree array contains exactly one zero entry.

### Multiple Disconnected Components

A graph may contain several disconnected sections.

For example:

```
0 -> 1

2 -> 3
```

Both `0` and `2` have indegree `0`.

This case is easy to mishandle if the algorithm assumes the graph is fully connected. The implementation avoids this issue by scanning all nodes and ensuring exactly one indegree-zero node exists before returning a result.
