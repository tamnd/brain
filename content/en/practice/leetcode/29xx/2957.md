## Solution

## Problem Understanding

The problem asks us to transform a string so that no two adjacent characters are "almost-equal". Two characters are considered almost-equal if they are either exactly the same or next to each other in the alphabet.

For example:

- `'a'` and `'a'` are almost-equal because they are identical.
- `'c'` and `'d'` are almost-equal because their alphabet positions differ by `1`.
- `'m'` and `'o'` are not almost-equal because their difference is greater than `1`.

We are allowed to perform operations where we change any character in the string to any lowercase English letter. Each replacement costs exactly one operation. The goal is to compute the minimum number of operations required so that the final string contains no adjacent almost-equal pairs.

The input consists of a single string `word` containing only lowercase English letters. The output is a single integer representing the minimum number of replacements needed.

The constraints are relatively small:

- `1 <= word.length <= 100`

This immediately tells us that even moderately expensive algorithms are acceptable. However, the problem has an important structural property that allows for a very clean greedy solution.

Several edge cases are important:

- Strings of length `1` are already valid because they contain no adjacent pairs.
- Long runs of identical characters such as `"aaaaaa"` can create overlapping conflicts.
- Alternating near-characters like `"ababab"` may still be invalid because `'a'` and `'b'` are adjacent alphabetically.
- Changing one character can fix multiple conflicts simultaneously, so naive local modifications may overcount operations.

Understanding how overlapping conflicts interact is the key insight of the problem.

## Approaches

### Brute Force Approach

A brute-force solution would try every possible combination of character replacements and determine whether the resulting string is valid.

For each character, we have 26 possible letters. A string of length `n` therefore has `26^n` possible transformed versions. We could generate all possibilities, check whether adjacent pairs are valid, and track the minimum number of changes.

This approach is guaranteed to find the optimal answer because it explores the entire search space. However, it is computationally infeasible even for relatively small strings. For `n = 100`, the number of possible strings is astronomically large.

A slightly improved brute-force method would use dynamic programming over states representing previous characters and replacement counts. While this becomes practical, it is still more complicated than necessary.

### Optimal Greedy Approach

The key observation is that whenever two adjacent characters form an almost-equal pair, at least one of them must be changed.

Suppose we encounter a bad pair at indices `i` and `i + 1`.

Instead of trying many possibilities, we can greedily change the second character and skip the next position. This works because:

- One modification can always break the current conflict.
- Modifying the second character is never worse than modifying the first.
- After fixing the pair, the next comparison involving the modified character becomes irrelevant because we conceptually replace it with a safe value.

This transforms the problem into counting how many disjoint adjacent conflicts exist.

The greedy strategy becomes:

- Scan from left to right.
- If `word[i]` and `word[i + 1]` are almost-equal:

- Increment the answer.
- Skip the next character because this operation already resolves conflicts involving it.
- Otherwise continue normally.

This produces the minimum number of operations.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(26^n × n) | O(n) | Tries all possible transformed strings |
| Optimal Greedy | O(n) | O(1) | Greedily fixes adjacent conflicts |

## Algorithm Walkthrough

1. Initialize a counter `operations = 0`.
2. Start scanning the string from left to right using index `i`.
3. For every adjacent pair `word[i]` and `word[i + 1]`, compute whether they are almost-equal.
4. Two characters are almost-equal if:

- They are identical, or
- Their alphabet distance is exactly `1`.

We can compute this using:

```
abs(ord(word[i]) - ord(word[i + 1])) <= 1
```
5. If the pair is almost-equal:

- We must perform an operation.
- Increment `operations`.
- Skip the next character by moving `i` forward by `2`.
6. If the pair is already valid:

- Simply move to the next position by incrementing `i` by `1`.
7. Continue until the scan reaches the end of the string.
8. Return the total number of operations.

### Why it works

Whenever two adjacent characters are almost-equal, at least one character must change. The greedy strategy always modifies the second character in the conflicting pair. This immediately resolves the current conflict.

Skipping the next position is safe because the modified character can always be changed into a value that avoids conflicts with both neighbors. Since the alphabet contains 26 letters and each neighbor only blocks at most 3 letters, there is always a valid replacement available.

Thus, every operation fixes one independent conflict segment optimally, and the total number of operations is minimized.

## Python Solution

```python
class Solution:
    def removeAlmostEqualCharacters(self, word: str) -> int:
        operations = 0
        index = 0
        n = len(word)

        while index < n - 1:
            difference = abs(ord(word[index]) - ord(word[index + 1]))

            if difference <= 1:
                operations += 1
                index += 2
            else:
                index += 1

        return operations
```

The implementation follows the greedy algorithm directly.

The variable `operations` stores the number of required replacements.

We iterate through the string using `index`. At each step, we compare the current character with the next character.

The expression:

```python
abs(ord(word[index]) - ord(word[index + 1]))
```

computes the alphabet distance between the two letters.

If the distance is less than or equal to `1`, the pair is almost-equal and requires a modification. We increment the answer and skip the next position because that character is considered modified.

Otherwise, we continue scanning normally.

The implementation uses constant extra memory and processes the string in a single pass.

## Go Solution

```go
func removeAlmostEqualCharacters(word string) int {
    operations := 0
    n := len(word)

    for i := 0; i < n-1; {
        difference := int(word[i]) - int(word[i+1])

        if difference < 0 {
            difference = -difference
        }

        if difference <= 1 {
            operations++
            i += 2
        } else {
            i++
        }
    }

    return operations
}
```

The Go implementation mirrors the Python solution closely.

Since Go does not provide a built-in absolute value function for integers in the standard language syntax, we compute the absolute difference manually.

Strings in Go are byte slices for ASCII lowercase letters, so direct indexing is safe here.

The algorithm still runs in linear time with constant extra space.

## Edge Cases

One important edge case is a string of length one. Since there are no adjacent pairs, the string is automatically valid. The algorithm handles this correctly because the loop condition `index < n - 1` immediately fails.

Another important case is overlapping conflicts such as `"abc"`. Both pairs `(a,b)` and `(b,c)` are invalid. A naive algorithm might incorrectly count two operations. However, changing the middle character alone can resolve both conflicts simultaneously. The greedy algorithm correctly counts only one operation because it skips the next character after a modification.

A third tricky case involves long runs of identical characters such as `"aaaaaa"`. Every adjacent pair is invalid, but changing every character would not be optimal. The minimum number of operations is actually half the length of the run rounded down. The greedy strategy naturally achieves this optimal behavior by fixing one pair at a time and skipping the modified character afterward.
