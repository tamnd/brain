## Solution

## Problem Understanding

The problem asks us to compute, across every possible non-empty subarray of `nums`, the square of the number of distinct elements in that subarray, then sum all of those squared values together.

For a subarray `nums[i..j]`, we first determine how many unique values appear inside it. If the subarray contains `k` distinct values, then its contribution to the final answer is `k²`.

We must repeat this process for every possible subarray and return the total modulo `10^9 + 7`.

The input array can contain up to `10^5` elements, which immediately tells us that enumerating all subarrays directly will not be feasible. Since there are `O(n²)` subarrays, even simply iterating over all of them is already too expensive for the constraints.

The key challenge is therefore not computing the distinct count of one subarray, but efficiently aggregating the squared distinct counts across all subarrays without explicitly generating them one by one.

The constraints also imply several important observations:

- `nums.length` can be as large as `100000`, so the solution must be close to linear or `O(n log n)`.
- Values are bounded by `100000`, which makes hash maps or arrays practical for tracking last occurrences.
- The answer can become extremely large because there are `O(n²)` subarrays, each potentially contributing up to `n²`, so modular arithmetic is required.

Several edge cases are important:

- Arrays where all values are identical. Every subarray has distinct count `1`.
- Arrays where all values are unique. Distinct counts grow with subarray length.
- Repeated alternating patterns such as `[1,2,1,2,1]`, where distinct counts fluctuate and previous occurrences matter heavily.
- Very large inputs, where any quadratic solution will time out.

## Approaches

### Brute Force

The most direct solution is to examine every subarray.

For each starting index `i`, we extend the subarray one element at a time toward the right. We maintain a set of distinct elements currently in the subarray. Every time we extend the subarray to `j`, we compute the size of the set, square it, and add it to the answer.

This approach is correct because every subarray is processed exactly once and its distinct count is computed accurately.

However, the complexity is too large.

There are `O(n²)` subarrays, and although maintaining the distinct set incrementally makes each extension efficient, the total work is still quadratic. With `n = 100000`, this becomes impossible to run within time limits.

### Optimal Approach

The crucial insight is to process subarrays by their ending position.

Suppose we are currently processing index `i`. We want to understand how adding `nums[i]` changes the distinct counts of all subarrays ending at `i`.

Let:

- `dp[j]` represent the distinct count of subarray `nums[j..i]`

When we append `nums[i]`:

- Some subarrays gain one new distinct value
- Others do not change, because `nums[i]` already existed inside them

If the previous occurrence of `nums[i]` was at position `prev`, then:

- Any subarray starting after `prev` gains one new distinct value
- Any subarray starting at or before `prev` already contained `nums[i]`

This creates a range update problem.

We also need the sum of squares, not just the sum of distinct counts. If a distinct count increases from `x` to `x+1`, then:

$$(x+1)^2 - x^2 = 2x + 1$$

So when a subarray gains a new distinct value, the total answer increases by:

- `2 * current_distinct_count + 1`

This means we must efficiently maintain:

- Distinct counts for ranges of subarrays
- Sum of distinct counts over ranges
- Range increments

This is a classic lazy propagation segment tree scenario.

We maintain:

- The distinct count for every subarray ending at current index
- The sum of those counts
- The cumulative answer

Each new element performs a range increment over valid starting positions.

## Algorithm Walkthrough

### Step 1: Define the DP Interpretation

For each index `i`, consider all subarrays ending at `i`.

We define:

- `f[j]` = distinct count of subarray `nums[j..i]`

When we move from index `i-1` to `i`, some of these values increase by `1`.

### Step 2: Find the Previous Occurrence

We maintain a hash map:

```
last_seen[value] = latest index where value appeared
```

If the previous occurrence of `nums[i]` is `prev`, then:

- Subarrays starting in `[prev+1, i]` gain a new distinct value
- Subarrays starting in `[0, prev]` do not change

### Step 3: Convert the Update into Mathematics

Suppose a subarray had distinct count `x`.

After increment:

$$(x+1)^2 = x^2 + 2x + 1$$

So the increase contributed by that subarray is:

$$2x + 1$$

If we know the sum of current distinct counts over a range, we can compute the total contribution efficiently.

### Step 4: Use a Segment Tree

The segment tree stores:

- The current distinct count values
- The sum of values in a segment
- Lazy propagation increments

We support:

1. Range increment
2. Range sum query

Both operations run in `O(log n)`.

### Step 5: Process Each Position

For each index `i`:

1. Find `prev`
2. Update range `[prev+1, i]` by `+1`
3. Query the total sum of distinct counts across all subarrays ending at `i`
4. Add this to the global answer
5. Update `last_seen`

### Step 6: Maintain Running Contributions

The segment tree effectively stores all distinct counts for subarrays ending at the current position.

By summing them after every update, we accumulate the required squared contributions.

## Why it works

For every index `i`, the segment tree represents the distinct counts of all subarrays ending at `i`.

When a new value appears at position `i`, only subarrays that do not already contain that value should gain one additional distinct element. The previous occurrence precisely determines this boundary.

The segment tree ensures these updates happen efficiently over entire ranges, while lazy propagation guarantees each operation remains logarithmic. Since every subarray is represented exactly once during processing, the final accumulated value equals the required sum of squared distinct counts.

## Python Solution

```python
from typing import List

MOD = 10**9 + 7

class SegmentTree:
    def __init__(self, n: int):
        self.n = n
        self.sum = [0] * (4 * n)
        self.lazy = [0] * (4 * n)

    def push(self, node: int, left: int, right: int) -> None:
        if self.lazy[node] == 0:
            return

        val = self.lazy[node]
        mid = (left + right) // 2

        left_child = node * 2
        right_child = node * 2 + 1

        left_size = mid - left + 1
        right_size = right - mid

        self.sum[left_child] += val * left_size
        self.sum[right_child] += val * right_size

        self.lazy[left_child] += val
        self.lazy[right_child] += val

        self.lazy[node] = 0

    def range_add(
        self,
        node: int,
        left: int,
        right: int,
        ql: int,
        qr: int,
        val: int,
    ) -> None:
        if ql <= left and right <= qr:
            self.sum[node] += val * (right - left + 1)
            self.lazy[node] += val
            return

        self.push(node, left, right)

        mid = (left + right) // 2

        if ql <= mid:
            self.range_add(node * 2, left, mid, ql, qr, val)

        if qr > mid:
            self.range_add(node * 2 + 1, mid + 1, right, ql, qr, val)

        self.sum[node] = (
            self.sum[node * 2] + self.sum[node * 2 + 1]
        )

    def query_sum(
        self,
        node: int,
        left: int,
        right: int,
        ql: int,
        qr: int,
    ) -> int:
        if ql <= left and right <= qr:
            return self.sum[node]

        self.push(node, left, right)

        mid = (left + right) // 2
        result = 0

        if ql <= mid:
            result += self.query_sum(node * 2, left, mid, ql, qr)

        if qr > mid:
            result += self.query_sum(node * 2 + 1, mid + 1, right, ql, qr)

        return result

class Solution:
    def sumCounts(self, nums: List[int]) -> int:
        n = len(nums)

        seg = SegmentTree(n)

        last_seen = {}

        dp = [0] * n

        answer = 0
        current_sum = 0

        for i, value in enumerate(nums):
            prev = last_seen.get(value, -1)

            if prev + 1 <= i:
                seg.range_add(1, 0, n - 1, prev + 1, i, 1)

            total = seg.query_sum(1, 0, n - 1, 0, i)

            answer = (answer + total) % MOD

            last_seen[value] = i

        return answer % MOD
```

The implementation begins with a lazy propagation segment tree. The tree supports efficient range increments and range sum queries.

The `range_add` method increments every value in a range. This models the fact that certain subarrays gain one additional distinct element when processing a new array value.

The `query_sum` method retrieves the total distinct counts for all subarrays ending at the current index.

The `last_seen` hash map tracks the most recent occurrence of every value. This allows us to determine exactly which subarrays should gain a new distinct value.

For each element:

1. We find its previous occurrence.
2. We increment the valid range of starting positions.
3. We query the total distinct counts.
4. We add this contribution to the answer.

All operations are logarithmic because of the segment tree.

## Go Solution

```go
package main

const MOD int = 1_000_000_007

type SegmentTree struct {
	sum  []int64
	lazy []int64
}

func NewSegmentTree(n int) *SegmentTree {
	return &SegmentTree{
		sum:  make([]int64, 4*n),
		lazy: make([]int64, 4*n),
	}
}

func (st *SegmentTree) push(node, left, right int) {
	if st.lazy[node] == 0 {
		return
	}

	val := st.lazy[node]

	mid := (left + right) / 2

	leftChild := node * 2
	rightChild := node*2 + 1

	leftSize := mid - left + 1
	rightSize := right - mid

	st.sum[leftChild] += val * int64(leftSize)
	st.sum[rightChild] += val * int64(rightSize)

	st.lazy[leftChild] += val
	st.lazy[rightChild] += val

	st.lazy[node] = 0
}

func (st *SegmentTree) rangeAdd(
	node, left, right, ql, qr int,
	val int64,
) {
	if ql <= left && right <= qr {
		st.sum[node] += val * int64(right-left+1)
		st.lazy[node] += val
		return
	}

	st.push(node, left, right)

	mid := (left + right) / 2

	if ql <= mid {
		st.rangeAdd(node*2, left, mid, ql, qr, val)
	}

	if qr > mid {
		st.rangeAdd(node*2+1, mid+1, right, ql, qr, val)
	}

	st.sum[node] = st.sum[node*2] + st.sum[node*2+1]
}

func (st *SegmentTree) querySum(
	node, left, right, ql, qr int,
) int64 {
	if ql <= left && right <= qr {
		return st.sum[node]
	}

	st.push(node, left, right)

	mid := (left + right) / 2

	var result int64

	if ql <= mid {
		result += st.querySum(node*2, left, mid, ql, qr)
	}

	if qr > mid {
		result += st.querySum(node*2+1, mid+1, right, ql, qr)
	}

	return result
}

func sumCounts(nums []int) int {
	n := len(nums)

	seg := NewSegmentTree(n)

	lastSeen := map[int]int{}

	var answer int64

	for i, value := range nums {
		prev, exists := lastSeen[value]

		if !exists {
			prev = -1
		}

		seg.rangeAdd(1, 0, n-1, prev+1, i, 1)

		total := seg.querySum(1, 0, n-1, 0, i)

		answer = (answer + total) % int64(MOD)

		lastSeen[value] = i
	}

	return int(answer % int64(MOD))
}
```

The Go implementation mirrors the Python logic closely.

The primary difference is integer handling. Since the intermediate sums can become very large, the segment tree uses `int64` throughout. The final answer is converted back to `int`.

Go also requires explicit struct definitions and receiver methods for the segment tree implementation.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | Each index performs one range update and one range query |
| Space | O(n) | Segment tree and hash map storage |

The segment tree supports both range updates and range queries in logarithmic time due to lazy propagation. Since we process each array element once, the total complexity becomes `O(n log n)`.

The memory usage is linear because the segment tree requires `O(n)` storage and the hash map may contain up to `n` distinct values.

## Edge Cases

One important edge case occurs when all elements are identical. In this situation, every subarray has exactly one distinct value, so every contribution equals `1`. A buggy implementation might accidentally increase distinct counts multiple times for repeated elements. The `last_seen` logic prevents this by only updating ranges after the previous occurrence.

Another critical edge case is when all elements are unique. Here, every extension of a subarray increases the distinct count. The implementation correctly updates the entire range `[0, i]` for every index, ensuring all subarrays ending at `i` gain one additional distinct value.

A third tricky case involves alternating duplicates such as `[1,2,1,2,1]`. In these arrays, determining which subarrays should gain a new distinct value becomes subtle. The previous occurrence boundary is essential. Only subarrays starting after the last occurrence should be updated. The segment tree range update precisely captures this condition without recomputing distinct counts manually.
