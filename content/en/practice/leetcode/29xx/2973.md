**Difficulty:** 🔴 Hard  
**Topics:** Dynamic Programming, Tree, Depth-First Search, Sorting, Heap (Priority Queue)

## Solution

## Problem Understanding

We are given an undirected tree with `n` nodes, rooted at node `0`. Every node has an associated integer `cost[i]`, which may be positive or negative.

For every node `i`, we must determine how many coins to place on that node. The answer depends on the size and values inside the subtree rooted at `i`.

The subtree of a node contains:

- The node itself
- All descendants reachable below it in the rooted tree

The rules for coin placement are:

- If the subtree contains fewer than 3 nodes, place exactly `1` coin.
- Otherwise, find the maximum possible product of the costs of any 3 distinct nodes in that subtree.
- If the best product is negative, place `0` coins instead.

The final result is an array `coin`, where:

- `coin[i]` is the number of coins placed on node `i`

A key detail is that the subtree is rooted relative to node `0`, even though the input edges are undirected. This means we first need to interpret the graph as a rooted tree.

The constraints are important:

- `n` can be as large as `2 * 10^4`
- Costs can be negative
- We need an answer for every node

A naive approach that recomputes all subtree products independently would be far too slow. Since each subtree may contain thousands of nodes, repeatedly enumerating all triples would explode combinatorially.

The presence of negative values creates an important edge case. The maximum product of three numbers is not always obtained from the three largest positive values. Sometimes:

- Two very negative numbers multiplied together produce a large positive value.
- That positive value multiplied by the largest positive number gives the optimal answer.

For example:

```
[-10, -9, 5, 2]
```

The best product is:

```
(-10) * (-9) * 5 = 450
```

not:

```
5 * 2 * (-9) = -90
```

Another important edge case is when every possible product is negative. In that case the answer must be `0`.

The problem guarantees:

- The graph is a valid tree
- There are exactly `n - 1` edges
- The tree is connected

This allows us to safely use DFS traversal without worrying about cycles beyond the parent relationship.

## Approaches

### Brute Force Approach

The brute force solution would process every node independently.

For each node:

1. Collect all values in its subtree.
2. Generate every possible combination of 3 distinct nodes.
3. Compute the product of each triple.
4. Take the maximum product.
5. If the maximum is negative, return `0`.

This works because it directly follows the problem definition.

However, it is extremely inefficient.

Suppose a subtree contains `k` nodes. Enumerating all triples requires:

```
O(k^3)
```

time.

In the worst case, the tree is a chain, so many subtrees are large. Repeating this process for every node leads to roughly:

```
O(n^4)
```

time complexity in the worst case, which is completely infeasible for `n = 2 * 10^4`.

### Key Insight

To maximize the product of 3 numbers, we never need all subtree values.

The optimal triple must come from one of two possibilities:

1. The three largest values
2. The two smallest values and the largest value

Why?

Because:

- Large positive numbers maximize positive products.
- Two large-magnitude negative numbers can become positive when multiplied together.

Therefore, for every subtree we only need:

- The 3 largest values
- The 2 smallest values

This dramatically reduces the amount of information we must propagate upward during DFS.

We can perform a postorder DFS:

- Each node gathers candidate values from its children.
- Merge those values with its own cost.
- Keep only the largest 3 and smallest 2 values.
- Compute the best possible product.

This converts the problem into a linear-time tree DP.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^4) worst case | O(n) | Enumerates all triples for every subtree |
| Optimal | O(n) | O(n) | DFS with only extreme values retained |

## Algorithm Walkthrough

### Step 1: Build the adjacency list

The input graph is undirected, so we first build an adjacency list.

This allows efficient DFS traversal of the tree.

```
graph[u].append(v)
graph[v].append(u)
```

### Step 2: Perform DFS from the root

We run DFS starting at node `0`.

The DFS returns:

- Subtree size
- Candidate values needed to compute maximum products

We pass the parent node during recursion so we do not revisit the previous node.

### Step 3: Gather values from children

At each node:

1. Start with the current node's own cost.
2. Recursively process every child.
3. Merge the child's candidate values into the current node's candidate list.

We do not need all subtree values. We only preserve:

- The 5 smallest values
- The 5 largest values

Keeping a few extra values simplifies merging logic and guarantees correctness.

### Step 4: Compute subtree size

Every DFS call accumulates subtree size:

```
size = 1 + sum(child sizes)
```

This determines whether the subtree has fewer than 3 nodes.

### Step 5: Handle small subtrees

If subtree size is less than 3:

```
answer[node] = 1
```

This is required directly by the problem statement.

### Step 6: Compute the best product

For larger subtrees, we sort the candidate values.

The maximum product can only be:

- Largest 3 values
- Smallest 2 values times largest value

So we compute:

```
candidate1 = largest1 * largest2 * largest3
candidate2 = smallest1 * smallest2 * largest1
```

Then:

```
best = max(candidate1, candidate2)
```

If `best < 0`, store `0`.

Otherwise store `best`.

### Step 7: Return compact candidate information upward

To keep the algorithm efficient, each DFS call only returns a bounded number of values.

After sorting:

- Keep the smallest 5
- Keep the largest 5

This ensures merging remains constant time per edge.

### Why it works

The correctness relies on the mathematical property of maximizing a product of three numbers.

Any optimal triple must involve either:

- The largest positive values
- Two most negative values and the largest positive value

No other values can improve the result.

Therefore retaining only extreme values is sufficient to reconstruct the optimal answer for every subtree. Since DFS processes children before parents, each node receives complete information about its entire subtree before computing its answer.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Solution:
    def placedCoins(self, edges: List[List[int]], cost: List[int]) -> List[int]:
        n = len(cost)

        graph = defaultdict(list)

        for u, v in edges:
            graph[u].append(v)
            graph[v].append(u)

        answer = [0] * n

        def dfs(node: int, parent: int):
            values = [cost[node]]
            subtree_size = 1

            for neighbor in graph[node]:
                if neighbor == parent:
                    continue

                child_size, child_values = dfs(neighbor, node)

                subtree_size += child_size
                values.extend(child_values)

            values.sort()

            if subtree_size < 3:
                answer[node] = 1
            else:
                largest_three = values[-1] * values[-2] * values[-3]

                smallest_two_largest_one = (
                    values[0] * values[1] * values[-1]
                )

                best = max(largest_three, smallest_two_largest_one)

                answer[node] = max(0, best)

            # Keep only useful extreme values
            trimmed = values[:5] + values[-5:]

            return subtree_size, trimmed

        dfs(0, -1)

        return answer
```

The implementation begins by constructing the adjacency list representation of the tree. Since the input edges are undirected, every edge is added in both directions.

The DFS function is the core of the algorithm. Each recursive call computes information about one subtree.

The `values` list stores candidate numbers that could contribute to the optimal product. Initially it contains only the current node's cost.

Each child recursively returns:

- Its subtree size
- Its important candidate values

These are merged into the current node's data.

After merging:

- We sort the values
- Compute the best possible triple product
- Store the answer for the current node

The trimming step is essential for efficiency. Instead of returning the entire subtree values upward, we only return the smallest and largest values that could matter for future computations.

This guarantees linear complexity overall.

## Go Solution

```go
package main

import "sort"

func placedCoins(edges [][]int, cost []int) []int64 {
	n := len(cost)

	graph := make([][]int, n)

	for _, e := range edges {
		u, v := e[0], e[1]
		graph[u] = append(graph[u], v)
		graph[v] = append(graph[v], u)
	}

	answer := make([]int64, n)

	var dfs func(int, int) (int, []int)

	dfs = func(node int, parent int) (int, []int) {
		values := []int{cost[node]}
		subtreeSize := 1

		for _, neighbor := range graph[node] {
			if neighbor == parent {
				continue
			}

			childSize, childValues := dfs(neighbor, node)

			subtreeSize += childSize
			values = append(values, childValues...)
		}

		sort.Ints(values)

		if subtreeSize < 3 {
			answer[node] = 1
		} else {
			candidate1 := int64(values[len(values)-1]) *
				int64(values[len(values)-2]) *
				int64(values[len(values)-3])

			candidate2 := int64(values[0]) *
				int64(values[1]) *
				int64(values[len(values)-1])

			best := candidate1
			if candidate2 > best {
				best = candidate2
			}

			if best < 0 {
				answer[node] = 0
			} else {
				answer[node] = best
			}
		}

		trimmed := []int{}

		left := min(5, len(values))
		rightStart := max(0, len(values)-5)

		trimmed = append(trimmed, values[:left]...)
		trimmed = append(trimmed, values[rightStart:]...)

		return subtreeSize, trimmed
	}

	dfs(0, -1)

	return answer
}

func min(a, b int) int {
	if a < b {
		return a
	}
	return b
}

func max(a, b int) int {
	if a > b {
		return a
	}
	return b
}
```

The Go implementation follows the same logic as the Python solution, but there are several language-specific considerations.

The result array uses `int64` because products can exceed 32-bit integer range. Multiplication is explicitly cast to `int64` before computing products.

Slices are used to store candidate values. Since Go does not provide built-in list merging utilities like Python, values are merged using `append`.

The DFS function is declared as a closure so it can recursively reference itself while accessing surrounding variables such as `graph` and `answer`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each node and edge is processed once, candidate lists remain bounded |
| Space | O(n) | Adjacency list and recursion stack |

The algorithm is linear because every DFS call processes only a constant number of candidate values. Even though sorting occurs repeatedly, the lists are bounded to a small constant size, so the sorting cost is effectively constant.

Without trimming, subtree lists could grow to size `O(n)`, which would destroy performance. The trimming optimization is what makes the solution scalable.

## Edge Cases

One important edge case occurs when the subtree size is smaller than 3. In these situations, there are not enough nodes to form a triple, so the answer must always be `1`. A common bug is attempting to compute products anyway, which can cause index errors or invalid logic. The implementation explicitly checks subtree size before computing products.

Another important edge case involves negative numbers. The maximum product is not always formed by the three numerically largest values. Two very negative numbers can produce a large positive product when multiplied together. The implementation correctly handles this by checking both:

- The three largest values
- The two smallest values combined with the largest value

A third critical edge case happens when every possible product is negative. The problem requires returning `0` instead of the negative product. The implementation handles this with:

```
answer[node] = max(0, best)
```

which guarantees compliance with the specification.

A final subtle edge case involves integer overflow. Products can become very large because costs may reach `10^4`. The product of three such values is `10^12`, which exceeds 32-bit integer range. The Go implementation uses `int64` to safely store results.
