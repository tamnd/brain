## Solution

# Problem Understanding

The problem asks us to count how many valid ways exist to distribute exactly `n` candies among `3` children, under the restriction that no child may receive more than `limit` candies.

Each distribution can be represented as a triple:

$$(a, b, c)$$

where:

- `a` is the number of candies given to the first child
- `b` is the number of candies given to the second child
- `c` is the number of candies given to the third child

The distribution is valid only if all of the following conditions hold:

$$a + b + c = n$$

and

$$0 \le a, b, c \le limit$$

The task is not to generate the distributions themselves, but simply to return the total count of valid triples.

The constraints are very small:

- `1 <= n <= 50`
- `1 <= limit <= 50`

Since both values are tiny, even moderately inefficient approaches will still run comfortably within limits. This means we can prioritize simplicity and correctness over advanced optimization.

There are several important edge cases to consider.

If `n` is larger than `3 * limit`, then it is impossible to distribute all candies while keeping every child at or below the limit. In that situation, the answer must be `0`.

If `n` is very small, such as `1`, many children may receive `0` candies. The solution must correctly include such cases.

If `limit >= n`, then the upper bound effectively does not matter, because no child would ever need more than `n` candies. In this situation, the problem becomes equivalent to counting all nonnegative integer solutions to:

$$a + b + c = n$$

A naive implementation can also accidentally double count or miss distributions if the loops are not structured carefully.

# Approaches

## Brute Force Approach

The most direct solution is to try every possible number of candies for each child.

We can loop through all possible values for the first child from `0` to `limit`. For each choice, we loop through all possible values for the second child from `0` to `limit`. Then we compute how many candies remain for the third child.

If the remaining value is between `0` and `limit`, then we have found a valid distribution.

This approach is correct because it explicitly checks every possible assignment that satisfies the constraints. Nothing is skipped, and every valid configuration is counted exactly once.

The downside is that it performs unnecessary work. We iterate through many combinations even when they cannot possibly sum to `n`.

Still, because the constraints are small, this brute force method is actually acceptable for this problem.

## Optimal Approach

We can improve the brute force solution slightly by observing that once we choose values for the first two children, the third child's value is completely determined.

Suppose:

$$a + b + c = n$$

If we already know `a` and `b`, then:

$$c = n - a - b$$

So instead of iterating over all three children independently, we only need two loops.

For every pair `(a, b)`, we compute `c` directly and check whether it satisfies:

$$0 \le c \le limit$$

This reduces the complexity from three nested loops to two nested loops.

Because the constraints are tiny, this solution is both efficient and very easy to understand.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(limit³) | O(1) | Tries every possible value for all three children |
| Optimal | O(limit²) | O(1) | Computes the third child's candies directly |

# Algorithm Walkthrough

## Optimal Algorithm

1. Initialize a variable `ways = 0`.

This variable will store the total number of valid distributions.
2. Iterate through all possible candy counts for the first child.

The first child can receive anywhere from `0` to `limit` candies.
3. For each value of the first child, iterate through all possible candy counts for the second child.

Again, the second child can receive anywhere from `0` to `limit`.
4. Compute the remaining candies for the third child.

Since the total must equal `n`, the third child's candies are:

$$third = n - first - second$$
5. Check whether the third child's value is valid.

The value is valid if:

$$0 \le third \le limit$$

If this condition is true, then the current distribution satisfies all constraints.
6. Increment the answer.

Every valid triple contributes exactly one to the final count.
7. After all iterations finish, return `ways`.

## Why it works

The algorithm systematically checks every possible assignment for the first two children. For each such assignment, there is exactly one value that can make the total sum equal to `n`, namely:

$$c = n - a - b$$

If this computed value lies within the allowed range, then the distribution is valid.

Because every pair `(a, b)` is examined exactly once, every valid distribution is counted exactly once. No valid configurations are skipped, and no invalid configurations are included.

# Python Solution

```python
class Solution:
    def distributeCandies(self, n: int, limit: int) -> int:
        ways = 0

        for first in range(limit + 1):
            for second in range(limit + 1):
                third = n - first - second

                if 0 <= third <= limit:
                    ways += 1

        return ways
```

The implementation begins by initializing the `ways` counter to zero.

The outer loop iterates through every possible candy count for the first child. The inner loop does the same for the second child.

For each pair of values, the remaining candies are assigned to the third child using:

```python
third = n - first - second
```

This guarantees that the total number of candies equals `n`.

The conditional statement verifies whether the third child's candy count stays within the allowed range. If it does, the configuration is valid, and the counter is incremented.

Finally, the function returns the total number of valid distributions.

# Go Solution

```go
func distributeCandies(n int, limit int) int {
    ways := 0

    for first := 0; first <= limit; first++ {
        for second := 0; second <= limit; second++ {
            third := n - first - second

            if third >= 0 && third <= limit {
                ways++
            }
        }
    }

    return ways
}
```

The Go implementation follows exactly the same logic as the Python version.

The main difference is syntax. Go uses explicit variable declarations and standard `for` loops instead of Python's `range`.

Integer overflow is not a concern here because the constraints are extremely small. All computations easily fit inside Go's `int` type.

No additional slices, arrays, or maps are required, so the space usage remains constant.

# Edge Cases

## Case 1: Impossible Distribution

Consider:

```
n = 10
limit = 2
```

Each child can receive at most `2` candies, so the maximum total number of candies that can be distributed is:

$$2 + 2 + 2 = 6$$

Since `10 > 6`, no valid distribution exists.

A buggy implementation might still count invalid states if it does not properly enforce the upper bound. Our implementation handles this correctly because the computed third value must satisfy:

```
0 <= third <= limit
```

No combination will pass this condition, so the answer remains `0`.

## Case 2: All Candies Must Be Evenly Distributed

Consider:

```
n = 3
limit = 1
```

Since each child can receive at most one candy, the only valid configuration is:

```
(1, 1, 1)
```

This case is important because many candidate pairs produce negative values for the third child. Our implementation correctly rejects those states.

## Case 3: Large Limit Compared to n

Consider:

```
n = 2
limit = 10
```

Here, the upper bound is effectively irrelevant because no child could ever receive more than `2` candies anyway.

The algorithm still works correctly because it simply checks whether the third child's value falls within the valid range. Every valid nonnegative solution is counted exactly once.

## Case 4: Zero Candies Remaining

Consider:

```
n = 6
limit = 2
```

The only valid configuration is:

```
(2, 2, 2)
```

This case ensures the implementation properly handles situations where the third child receives exactly the limit value. The condition:

```
0 <= third <= limit
```

correctly includes boundary values.
