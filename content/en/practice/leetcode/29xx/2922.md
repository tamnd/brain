## Solution

## Problem Understanding

This problem asks us to analyze sellers, their favorite brands, and the items they have sold. We need to identify the seller or sellers who sold the highest number of unique items whose brands are different from that seller’s favorite brand.

There are three tables involved:

- The `Users` table stores seller information, including each seller’s favorite brand.
- The `Items` table maps each item to its brand.
- The `Orders` table records which seller sold which item.

The important detail is that we only count items whose brand is different from the seller’s favorite brand. In addition, the count must be based on unique items, not total sales. If a seller sells the same qualifying item multiple times, it should only contribute once to the count.

The output should contain:

- `seller_id`
- `num_items`, the number of distinct qualifying items sold

We only return the seller or sellers whose count is the maximum among all sellers. If multiple sellers tie for the highest value, all of them must be returned. The result must be ordered by `seller_id` in ascending order.

Consider the example carefully:

- Seller `2` has favorite brand `Samsung`.
- Seller `2` sold item `4` twice.
- Item `4` has brand `HP`, which is different from `Samsung`.
- Even though it was sold twice, it counts only once because we need unique items.

The constraints are not explicitly listed in the statement, but this is a database problem designed for SQL. That means we should think in terms of joins, filtering, grouping, distinct counting, and aggregation.

Several edge cases are important:

- A seller may sell only favorite-brand items, resulting in a count of `0`.
- A seller may sell the same non-favorite item many times, but it still counts once.
- Multiple sellers may tie for the maximum count.
- Some sellers may have no orders at all.
- Every item has exactly one brand, and every order references valid rows in the other tables.

## Approaches

### Brute Force Approach

A brute force solution would process each seller independently.

For every seller, we could:

1. Scan all orders.
2. Select orders belonging to the current seller.
3. For each order, look up the item brand.
4. Compare the item brand against the seller’s favorite brand.
5. Store qualifying item IDs in a set to ensure uniqueness.
6. Compute the size of the set.

After processing all sellers, we would determine the maximum count and return every seller matching that maximum.

This approach is correct because it explicitly checks every relevant condition for every seller and every order. However, it is inefficient because it repeatedly scans the orders table for each seller.

If there are `U` users and `O` orders, this approach can degrade to `O(U * O)` time complexity.

### Optimal Approach

The key observation is that SQL databases are optimized for set-based operations.

Instead of processing sellers one by one, we can:

1. Join the three tables once.
2. Filter rows where the item brand differs from the favorite brand.
3. Group by seller.
4. Count distinct item IDs.
5. Find the maximum count.
6. Return all sellers matching that maximum.

This approach avoids repeated scans and leverages aggregation directly in SQL.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(U * O) | O(O) | Repeatedly scans all orders for every seller |
| Optimal | O(O log O) approximately | O(O) | Uses joins, grouping, and DISTINCT efficiently |

## Algorithm Walkthrough

1. Join the `Orders` table with the `Users` table using `seller_id`.

This gives us access to each seller’s favorite brand alongside every order.
2. Join the result with the `Items` table using `item_id`.

This lets us determine the brand of each sold item.
3. Filter rows where `item_brand != favorite_brand`.

We only care about items whose brand differs from the seller’s favorite brand.
4. Group the remaining rows by `seller_id`.

Each group now represents all qualifying sales for one seller.
5. Count `DISTINCT item_id` for every seller.

DISTINCT is necessary because the same item may have been sold multiple times, but it should count only once.
6. Store these counts in a temporary result.
7. Find the maximum count across all sellers.
8. Return every seller whose count equals that maximum.
9. Order the final output by `seller_id` ascending.

### Why it works

The algorithm works because every qualifying order is included exactly once after filtering, and grouping ensures that all qualifying sales for a seller are processed together. Using `COUNT(DISTINCT item_id)` guarantees that duplicate sales of the same item are not overcounted. Selecting sellers whose count equals the global maximum ensures the correct top sellers are returned.

```python
WITH seller_counts AS (
    SELECT
        o.seller_id,
        COUNT(DISTINCT o.item_id) AS num_items
    FROM Orders o
    JOIN Users u
        ON o.seller_id = u.seller_id
    JOIN Items i
        ON o.item_id = i.item_id
    WHERE i.item_brand <> u.favorite_brand
    GROUP BY o.seller_id
),
max_count AS (
    SELECT MAX(num_items) AS mx
    FROM seller_counts
)
SELECT
    sc.seller_id,
    sc.num_items
FROM seller_counts sc
JOIN max_count mc
    ON sc.num_items = mc.mx
ORDER BY sc.seller_id;
```

The solution starts by joining the three tables so that every order can be evaluated alongside the seller’s favorite brand and the item’s actual brand.

The `WHERE` clause removes all orders where the sold item matches the seller’s favorite brand. Only non-favorite-brand items remain.

Next, the query groups rows by seller and computes:

```python
COUNT(DISTINCT o.item_id)
```

This is the most important part of the solution because it prevents duplicate counting when the same item is sold multiple times.

The `seller_counts` common table expression stores each seller’s final count.

The second common table expression computes the largest count among all sellers.

Finally, the outer query returns every seller whose count equals that maximum value.

## Edge Cases

### Repeated Sales of the Same Item

A seller may sell the same qualifying item many times. A naive implementation using `COUNT(*)` would overcount these sales. The solution avoids this by using:

```
COUNT(DISTINCT o.item_id)
```

This guarantees every item contributes at most once per seller.

### Sellers With Only Favorite-Brand Sales

A seller might only sell items matching their favorite brand. In this situation, none of their orders pass the filter condition. The query correctly excludes these rows during aggregation.

If every seller has zero qualifying items, handling may depend on whether the database expects zero-count sellers to appear explicitly. The provided solution follows the standard interpretation used in LeetCode.

### Multiple Sellers Tying for Maximum

The problem explicitly states that all top sellers should be returned if there is a tie. A common bug is returning only one seller using `LIMIT 1`. This solution instead computes the maximum count separately and returns every seller whose count matches that maximum.

### Sellers With No Orders

Some sellers may never appear in the `Orders` table. Since they have sold no items, they naturally contribute no qualifying rows after joins. The aggregation handles this correctly without special logic.
