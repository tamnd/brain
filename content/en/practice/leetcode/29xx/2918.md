## Solution

## Problem Understanding

The problem gives us two integer arrays, `nums1` and `nums2`. Some elements may be `0`, and every zero must eventually be replaced with a strictly positive integer. After replacing all zeros, the total sum of both arrays must become exactly equal.

Our goal is not just to make the sums equal, but to make that equal sum as small as possible. If there is no way to make the sums equal, we must return `-1`.

The important detail is that every replacement value must be greater than or equal to `1`. We cannot leave zeros unchanged, and we cannot replace them with negative numbers or additional zeros.

Suppose an array currently has:

- A non-zero sum of `baseSum`
- `zeroCount` zeros

Then the minimum possible sum we can achieve for that array is:

- `baseSum + zeroCount`

because every zero must contribute at least `1`.

The constraints are large:

- Up to `10^5` elements in each array
- Values up to `10^6`

This immediately tells us that exponential or simulation-based approaches are impossible. We need a linear-time solution.

Several edge cases are especially important:

- One array may contain no zeros at all, meaning its sum is fixed forever.
- Both arrays may already have equal sums.
- One side may have a smaller minimum achievable sum but no flexibility to increase further.
- Arrays consisting entirely of zeros require careful handling.
- Large sums require attention to integer overflow in Go, which is why the Go signature returns `int64`.

## Approaches

### Brute Force Approach

A brute-force approach would try different positive replacements for each zero and check whether the two resulting sums can become equal.

For example, if an array contains two zeros, we could try assigning:

- `(1,1)`
- `(1,2)`
- `(2,1)`
- `(3,5)`

and so on.

This approach is clearly infeasible because each zero can take infinitely many positive values. Even if we artificially bounded the search space, the number of combinations would explode exponentially.

The brute-force method is correct because it explores all possible assignments, but it is far too slow for arrays of size up to `10^5`.

### Optimal Greedy Observation

The key observation is that we only care about sums, not the exact replacement values.

For any array:

- Let `sum` be the sum of non-zero elements.
- Let `zeros` be the number of zeros.

Then the smallest possible final sum is:

$$sum + zeros$$

because each zero contributes at least `1`.

After reaching this minimum, we can increase the total sum arbitrarily by increasing any replaced value.

This leads to two critical cases:

1. If an array has at least one zero, its final sum can be any value greater than or equal to its minimum achievable sum.
2. If an array has no zeros, its sum is fixed permanently.

Therefore:

- If both arrays can reach some common value, the minimum valid answer is the larger of the two minimum achievable sums.
- If one array has no zeros and its fixed sum is smaller than the other array's minimum achievable sum, equality is impossible.

This reduces the entire problem to simple arithmetic.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Tries all possible zero replacements |
| Optimal | O(n + m) | O(1) | Uses minimum achievable sums and feasibility checks |

## Algorithm Walkthrough

1. Traverse `nums1` and compute:

- `sum1`, the sum of all elements
- `zeros1`, the number of zeros
2. Traverse `nums2` and compute:

- `sum2`, the sum of all elements
- `zeros2`, the number of zeros
3. Compute the minimum achievable sums:

- `min1 = sum1 + zeros1`
- `min2 = sum2 + zeros2`

This works because every zero must become at least `1`.
4. If `min1 == min2`, return that value immediately.

Both arrays can already reach the same minimum sum.
5. Suppose `min1 < min2`.

Then array 1 must increase its sum until it reaches `min2`.

This is only possible if `nums1` contains at least one zero. Without zeros, its sum is fixed forever.

Therefore:

- If `zeros1 == 0`, return `-1`
- Otherwise return `min2`
6. Similarly, if `min2 < min1`:

- If `zeros2 == 0`, return `-1`
- Otherwise return `min1`

### Why it works

Each array has a minimum achievable sum determined by replacing every zero with `1`. Any array containing at least one zero can increase its total sum arbitrarily by assigning a larger replacement value to one of those zeros. Therefore, the smallest common achievable sum is simply the larger of the two minimum sums, provided the smaller side still has flexibility to increase. If the smaller side has no zeros, its sum is fixed and equality becomes impossible.

## Python Solution

```python
from typing import List

class Solution:
    def minSum(self, nums1: List[int], nums2: List[int]) -> int:
        sum1 = sum(nums1)
        sum2 = sum(nums2)

        zeros1 = nums1.count(0)
        zeros2 = nums2.count(0)

        min1 = sum1 + zeros1
        min2 = sum2 + zeros2

        if min1 == min2:
            return min1

        if min1 < min2:
            if zeros1 == 0:
                return -1
            return min2

        if zeros2 == 0:
            return -1

        return min1
```

The implementation begins by computing the total sums and zero counts for both arrays.

The variables `min1` and `min2` represent the smallest achievable sums after replacing every zero with `1`.

If these minimum sums are already equal, that value is the optimal answer because no smaller equal sum can exist.

If one minimum sum is smaller, that side must increase to match the larger side. This is only possible if the smaller side contains at least one zero, because zeros provide flexibility to increase the total sum.

If the smaller side has no zeros, its sum is fixed permanently, so equality cannot be achieved and we return `-1`.

Otherwise, the larger minimum sum becomes the answer.

## Go Solution

```go
func minSum(nums1 []int, nums2 []int) int64 {
    var sum1 int64 = 0
    var sum2 int64 = 0

    zeros1 := 0
    zeros2 := 0

    for _, num := range nums1 {
        sum1 += int64(num)
        if num == 0 {
            zeros1++
        }
    }

    for _, num := range nums2 {
        sum2 += int64(num)
        if num == 0 {
            zeros2++
        }
    }

    min1 := sum1 + int64(zeros1)
    min2 := sum2 + int64(zeros2)

    if min1 == min2 {
        return min1
    }

    if min1 < min2 {
        if zeros1 == 0 {
            return -1
        }
        return min2
    }

    if zeros2 == 0 {
        return -1
    }

    return min1
}
```

The Go implementation follows the same logic as the Python version. The main difference is the use of `int64` for sums, because the total sum may exceed the range of a 32-bit integer when arrays are large.

Go slices naturally handle empty ranges, so no special handling is required for empty arrays, although the constraints guarantee at least one element anyway.

### Example 1

```
nums1 = [3,2,0,1,0]
nums2 = [6,5,0]
```

First compute sums and zero counts.

| Array | Raw Sum | Zero Count | Minimum Achievable Sum |
| --- | --- | --- | --- |
| nums1 | 6 | 2 | 8 |
| nums2 | 11 | 1 | 12 |

We now compare:

- `min1 = 8`
- `min2 = 12`

`nums1` must increase from `8` to `12`.

Since `nums1` has zeros, we can increase its replacements.

One valid assignment:

- Replace zeros in `nums1` with `2` and `4`
- Replace zero in `nums2` with `1`

Final arrays:

```
nums1 = [3,2,2,1,4]  -> sum = 12
nums2 = [6,5,1]      -> sum = 12
```

Answer:

```
12
```

### Example 2

```
nums1 = [2,0,2,0]
nums2 = [1,4]
```

Compute values.

| Array | Raw Sum | Zero Count | Minimum Achievable Sum |
| --- | --- | --- | --- |
| nums1 | 4 | 2 | 6 |
| nums2 | 5 | 0 | 5 |

We compare:

- `min1 = 6`
- `min2 = 5`

`nums2` would need to increase from `5` to `6`.

But `nums2` has no zeros, so its sum is fixed permanently.

Therefore equality is impossible.

Answer:

```
-1
```

## Edge Cases

One important edge case occurs when neither array contains zeros. In this situation, both sums are completely fixed. If the sums are already equal, we return that value. Otherwise, equality is impossible. A buggy implementation might incorrectly assume sums can always be adjusted.

Another critical case is when one array contains zeros and the other does not. The array without zeros has a fixed sum, while the other array can only increase from its minimum achievable value upward. If the flexible array's minimum already exceeds the fixed sum, the problem becomes impossible because sums can never decrease.

Arrays consisting entirely of zeros are also important. For example, `[0,0]` has a minimum achievable sum of `2`. Since every zero must become at least `1`, implementations that accidentally allow replacement with `0` would produce incorrect answers.

Large input sizes create another potential issue. Since arrays may contain up to `10^5` elements with values up to `10^6`, total sums can become very large. The Go solution uses `int64` to avoid overflow problems that could occur with standard 32-bit integers.
