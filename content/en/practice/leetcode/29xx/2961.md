## Solution

## Problem Understanding

The problem gives us a list called `variables`, where each element contains four integers:

```
[ai, bi, ci, mi]
```

For every index `i`, we must evaluate the following mathematical expression:

$$((a_i^{b_i} \bmod 10)^{c_i}) \bmod m_i$$

If the result equals `target`, then index `i` is considered a good index.

Our task is to return all good indices in any order.

The important detail is that this is a double modular exponentiation problem. We first compute:

$$a_i^{b_i} \bmod 10$$

Then we take that result and raise it to the power `ci`:

$$(a_i^{b_i} \bmod 10)^{c_i}$$

Finally, we take modulo `mi`.

Even though the exponents can become large mathematically, the constraints are actually small:

- `variables.length <= 100`
- `ai, bi, ci, mi <= 1000`

Because of these limits, we do not need advanced number theory. However, we still need to avoid directly computing enormous powers like `1000^1000`, since that would be inefficient and unnecessary.

The key observation is that modular exponentiation allows us to compute powers efficiently while keeping intermediate numbers small.

Several edge cases are important:

- `mi` can be `1`, which means the final result is always `0`
- `a^b % 10` always produces a value between `0` and `9`
- Large exponents such as `1000` could overflow or become slow if computed naively
- The answer may be empty if no index satisfies the condition

## Approaches

### Brute Force Approach

A straightforward approach is to directly compute:

$$(a_i^{b_i} \% 10)^{c_i} \% m_i$$

for every index.

Conceptually, this works because it follows the formula exactly. For each element:

1. Compute `ai^bi`
2. Take modulo `10`
3. Raise the result to `ci`
4. Take modulo `mi`
5. Compare with `target`

The problem is that exponentiation grows extremely quickly. Even values like `1000^1000` are astronomically large. Computing such numbers directly is inefficient and may overflow in some languages.

Although Python supports arbitrarily large integers, repeatedly constructing giant numbers is still unnecessary and slower than modular exponentiation.

### Optimal Approach

The better solution uses modular exponentiation.

The key mathematical insight is:

$$(a^b) \bmod m$$

can be computed efficiently without ever constructing the huge value `a^b`.

Most languages provide a fast modular exponentiation implementation:

- Python: `pow(base, exponent, mod)`
- Go: implement fast exponentiation manually

We compute the expression in two stages:

1. Compute:

$$x = a_i^{b_i} \bmod 10$$

1. Compute:

$$y = x^{c_i} \bmod m_i$$

1. If `y == target`, add the index to the answer.

Because modular exponentiation runs in logarithmic time relative to the exponent, this solution is efficient and clean.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × bi × ci) or worse | O(1) | Direct exponentiation creates huge intermediate numbers |
| Optimal | O(n × log bi + n × log ci) | O(1) | Uses fast modular exponentiation |

## Algorithm Walkthrough

1. Initialize an empty result array.

This array will store all indices that satisfy the required formula.
2. Iterate through every element in `variables`.

Each entry contains four integers:

```
[a, b, c, m]
```
3. Compute the first modular exponentiation.

We calculate:

$$x = a^b \bmod 10$$

Instead of computing `a^b` directly, we use modular exponentiation to keep the computation efficient.
4. Compute the second modular exponentiation.

Using the previous result:

$$y = x^c \bmod m$$

Again, modular exponentiation avoids large intermediate values.
5. Compare the result with `target`.

If:

```
y == target
```

then the current index is good, so we append it to the result array.
6. Continue until all indices are processed.
7. Return the result array.

### Why it works

Modular exponentiation preserves correctness because of the mathematical property:

$$(a^b) \bmod m$$

can be computed incrementally while reducing modulo `m` at each step.

The algorithm evaluates the exact expression specified in the problem, but does so efficiently without constructing enormous integers. Since every index is checked independently and added only when the computed value equals `target`, the returned list contains exactly all good indices.

## Python Solution

```python
from typing import List

class Solution:
    def getGoodIndices(self, variables: List[List[int]], target: int) -> List[int]:
        good_indices = []

        for index, (a, b, c, m) in enumerate(variables):
            first_mod = pow(a, b, 10)
            final_value = pow(first_mod, c, m)

            if final_value == target:
                good_indices.append(index)

        return good_indices
```

The implementation closely follows the algorithm described earlier.

We first create an empty list called `good_indices` to store valid indices.

The loop iterates through every entry in `variables`. Python tuple unpacking makes the code cleaner by directly assigning:

```python
a, b, c, m
```

The first computation:

```python
first_mod = pow(a, b, 10)
```

efficiently calculates:

$$a^b \bmod 10$$

Python's three-argument `pow` function performs modular exponentiation internally in logarithmic time.

Next:

```python
final_value = pow(first_mod, c, m)
```

computes:

$$(first\_mod)^c \bmod m$$

If the result matches `target`, we append the current index to the answer list.

Finally, the completed result list is returned.

## Go Solution

```go
func modPow(base int, exponent int, mod int) int {
	result := 1
	base %= mod

	for exponent > 0 {
		if exponent%2 == 1 {
			result = (result * base) % mod
		}

		base = (base * base) % mod
		exponent /= 2
	}

	return result
}

func getGoodIndices(variables [][]int, target int) []int {
	result := []int{}

	for i, values := range variables {
		a := values[0]
		b := values[1]
		c := values[2]
		m := values[3]

		firstMod := modPow(a, b, 10)
		finalValue := modPow(firstMod, c, m)

		if finalValue == target {
			result = append(result, i)
		}
	}

	return result
}
```

The Go solution uses a helper function called `modPow` because Go does not provide built-in modular exponentiation for integers.

The helper implements binary exponentiation, also called fast exponentiation. At each step:

- If the current exponent bit is set, multiply the result by the current base
- Square the base
- Divide the exponent by two

This reduces the complexity from linear to logarithmic time.

Go uses fixed-size integers, but the values remain small because modulo is applied after every multiplication, preventing overflow under the given constraints.

The result slice is initialized as an empty slice rather than `nil`, though both would work correctly.

#### Index 0

$$2^3 \bmod 10 = 8$$

$$8^3 \bmod 10 = 512 \bmod 10 = 2$$

Matches target, include index `0`.

#### Index 1

$$3^3 \bmod 10 = 27 \bmod 10 = 7$$

Then:

$$7^3 \bmod 1 = 0$$

Does not match target.

#### Index 2

$$6^1 \bmod 10 = 6$$

Then:

$$6^1 \bmod 4 = 2$$

Matches target, include index `2`.

Final answer:

```
[0, 2]
```

## Edge Cases

One important edge case occurs when `mi = 1`. Any number modulo `1` equals `0`, regardless of the exponentiation result before it. A buggy implementation might attempt unnecessary calculations or mishandle division-by-one style logic. This implementation handles it naturally because modular exponentiation correctly returns `0`.

Another important case involves very large exponents such as `1000`. Direct exponentiation could create extremely large intermediate values, especially in languages with fixed-size integers. The implementation avoids this entirely by using modular exponentiation, which continuously reduces values modulo `m`.

A third edge case is when no indices satisfy the condition. Some implementations accidentally return `None` or an uninitialized structure instead of an empty list. Here, the result array is initialized empty and returned unchanged if no matches are found.

A subtle edge case appears when `a^b % 10` becomes `0`. Once the intermediate value is `0`, all further positive powers remain `0`. The implementation still works correctly because modular exponentiation handles zero bases naturally.

Finally, there can be multiple valid indices. The algorithm checks every index independently and appends all matching positions, ensuring none are skipped.
