## Solution

# Problem Understanding

The problem asks us to count how many valid ways exist to distribute exactly `n` candies among `3` children, under the restriction that no child receives more than `limit` candies.

In mathematical terms, we want to count the number of integer solutions to:

$x_1 + x_2 + x_3 = n$

subject to:

$0 \le x_i \le \text{limit}$

for all three children.

Each valid distribution is an ordered triple `(x1, x2, x3)`. Order matters because giving `(1,2,2)` is considered different from `(2,1,2)`.

The input contains two integers:

- `n`, the total number of candies
- `limit`, the maximum number of candies any single child may receive

The output should be the total number of valid ordered distributions.

The constraints are extremely important:

- `1 <= n <= 10^8`
- `1 <= limit <= 10^8`

These values are far too large for any brute-force iteration over all possibilities. A cubic or even quadratic solution would be completely infeasible. This strongly suggests that the intended solution relies on mathematics or combinatorics rather than simulation.

There are several important edge cases to keep in mind:

- If `n > 3 * limit`, then distributing all candies is impossible because even giving every child the maximum amount is insufficient.
- If `n <= limit`, then many unrestricted combinations are automatically valid.
- Very small values such as `n = 1` or `limit = 1` can expose off-by-one errors.
- Large values require careful handling of arithmetic and combinatorial formulas.

## Approaches

### Brute Force Approach

A straightforward solution is to try every possible number of candies for the first child and every possible number for the second child. The third child then receives whatever candies remain.

For every pair `(x1, x2)`:

1. Compute `x3 = n - x1 - x2`
2. Check whether:

- `0 <= x3 <= limit`
- `x1 <= limit`
- `x2 <= limit`
3. Count the distribution if valid.

This approach is correct because it explicitly enumerates every possible ordered distribution and filters out invalid ones.

However, the time complexity is far too large. We would iterate up to `limit^2` combinations, which becomes impossible when `limit` can be as large as `10^8`.

### Optimal Approach

The key observation is that this is a classic bounded integer composition problem.

Without the upper bound restriction, the number of nonnegative integer solutions to:

$x_1 + x_2 + x_3 = n$

is given by the stars and bars formula:

$\binom{n+2}{2}$

This counts all nonnegative solutions, including invalid ones where one or more children exceed `limit`.

To enforce the upper bound, we use the Inclusion-Exclusion Principle.

We:

1. Count all unrestricted solutions.
2. Subtract solutions where at least one child exceeds `limit`.
3. Add back solutions where at least two children exceed `limit`, because they were subtracted twice.
4. Continue inclusion-exclusion appropriately.

This transforms the problem into a constant-time combinatorial calculation.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(limit²) | O(1) | Enumerates all possible distributions |
| Optimal | O(1) | O(1) | Uses combinatorics and inclusion-exclusion |

## Algorithm Walkthrough

### Step 1: Count all unrestricted solutions

The number of nonnegative integer solutions to:

$x_1 + x_2 + x_3 = n$

is:

$\binom{n+2}{2}$

We store this as the initial answer.

### Step 2: Subtract solutions where one child exceeds the limit

Suppose child 1 exceeds the limit.

Then:

$x_1 \ge \text{limit}+1$

We substitute:

$x_1' = x_1 - (\text{limit}+1)$

Now the equation becomes:

$x_1' + x_2 + x_3 = n-(\text{limit}+1)$

The number of such solutions is:

$\binom{n-\text{limit}+1}{2}$

There are 3 choices for which child violates the limit, so we subtract:

$3\cdot\binom{n-\text{limit}+1}{2}$

### Step 3: Add back solutions where two children exceed the limit

Suppose both child 1 and child 2 exceed the limit.

We substitute:

$x_1' = x_1-(\text{limit}+1),\quad x_2' = x_2-(\text{limit}+1)$

The equation becomes:

$x_1'+x_2'+x_3 = n-2(\text{limit}+1)$

The number of such solutions is:

$\binom{n-2\text{limit}}{2}$

There are:

$\binom{3}{2}=3$

ways to choose the violating pair, so we add back:

$3\cdot\binom{n-2\text{limit}}{2}$

### Step 4: Subtract solutions where all three children exceed the limit

Now all three exceed the limit.

After substitution, the equation becomes:

$x_1'+x_2'+x_3' = n-3(\text{limit}+1)$

The number of such solutions is:

$\binom{n-3\text{limit}-1}{2}$

We subtract this term.

### Step 5: Handle invalid combinatorial arguments

Whenever the upper value of a combination becomes smaller than 2, the number of solutions should be treated as 0.

We therefore implement a helper:

$\binom{k}{2}=0\quad\text{if }k<2$

### Why it works

The algorithm works because the Inclusion-Exclusion Principle guarantees that every invalid configuration is counted exactly the correct number of times.

We begin by counting all possible nonnegative solutions. Then we remove configurations violating the upper bound. Since configurations with multiple violations get removed multiple times, we add them back appropriately. Continuing this alternating correction process ensures that only valid distributions remain in the final count.

## Python Solution

```python
class Solution:
    def distributeCandies(self, n: int, limit: int) -> int:
        def comb2(x: int) -> int:
            if x < 2:
                return 0
            return x * (x - 1) // 2

        total = comb2(n + 2)

        total -= 3 * comb2(n - limit + 1)

        total += 3 * comb2(n - 2 * limit)

        total -= comb2(n - 3 * limit - 1)

        return total
```

The implementation follows the inclusion-exclusion derivation directly.

The helper function `comb2(x)` computes:

$\binom{x}{2}$

using the formula:

$\frac{x(x-1)}{2}$

If `x < 2`, the helper returns `0` because no valid combinations exist.

The variable `total` starts with the unrestricted count. We then apply the inclusion-exclusion corrections in alternating subtraction and addition steps.

Since the solution uses only arithmetic operations, the runtime is constant.

## Go Solution

```go
func distributeCandies(n int, limit int) int64 {
    comb2 := func(x int64) int64 {
        if x < 2 {
            return 0
        }
        return x * (x - 1) / 2
    }

    N := int64(n)
    L := int64(limit)

    total := comb2(N + 2)

    total -= 3 * comb2(N - L + 1)

    total += 3 * comb2(N - 2*L)

    total -= comb2(N - 3*L - 1)

    return total
}
```

The Go implementation mirrors the Python solution almost exactly.

The main Go-specific consideration is integer overflow. Since intermediate values can become large, the implementation uses `int64` for all arithmetic. The function signature already returns `int64`, so converting the inputs at the start keeps calculations consistent and safe.

## Edge Cases

One important edge case occurs when `n > 3 * limit`. In this situation, even giving every child the maximum allowed candies still does not reach `n`. A naive implementation might accidentally produce a negative intermediate count or invalid combinatorial value. The inclusion-exclusion formula handles this naturally because all remaining terms cancel correctly and the final result becomes zero.

Another important edge case is when `limit >= n`. In this scenario, the upper bound never actually restricts the distribution because no child can receive more than the total number of candies anyway. The answer should reduce to the unrestricted stars and bars formula. The implementation handles this correctly because all subtraction terms become zero automatically.

A third subtle edge case involves negative or very small combinatorial arguments. During inclusion-exclusion, expressions like:

$\binom{n-3\text{limit}-1}{2}$

may become negative. Directly applying the arithmetic formula would produce incorrect results. The helper function explicitly returns `0` whenever the upper argument is smaller than `2`, ensuring mathematical correctness.

Finally, extremely large inputs require careful arithmetic handling. Since both `n` and `limit` may reach `10^8`, intermediate combinatorial values become very large. Python naturally supports arbitrary precision integers, while the Go implementation uses `int64` to safely store all values within range.
