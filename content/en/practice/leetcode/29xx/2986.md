# LeetCode 2986 - Find Third Transaction

## Problem Understanding

The problem provides a database table named `Transactions` that stores information about user purchases. Each row contains a `user_id`, the amount spent in the transaction, and the exact `transaction_date`.

The task is to identify the third transaction for every user, but only if that third transaction satisfies a specific condition. The spending amount of the third transaction must be strictly greater than the spending amounts of the user's first two transactions.

To solve this correctly, we must first determine the chronological order of transactions for each user. The ordering is based on `transaction_date`. Once ordered, we examine the third transaction and compare its `spend` value against the previous two transactions.

The output should contain:

- `user_id`
- `third_transaction_spend`
- `third_transaction_date`

Only users whose third transaction is greater than both preceding transactions should appear in the result.

The output must be sorted by `user_id` in ascending order.

The table guarantees that `(user_id, transaction_date)` is unique, which means no user can have two transactions at the exact same timestamp. This removes ambiguity when assigning transaction order.

Several edge cases are important:

- Users with fewer than three transactions must be ignored.
- Users whose third transaction equals one of the previous transactions should not be included, because the comparison must be strictly greater.
- Transaction ordering must always use `transaction_date`, not insertion order.
- Users may have many transactions, but only the first three matter for this problem.

## Approaches

### Brute Force Approach

A straightforward solution is to group all transactions by user, sort each user's transactions by date, and then manually inspect the third transaction.

For each user:

1. Collect all transactions.
2. Sort them chronologically.
3. Check whether a third transaction exists.
4. Compare the third transaction's spending amount with the first and second transactions.
5. If the condition holds, add the user to the result.

This approach is correct because it explicitly reconstructs the transaction history for every user and directly evaluates the required condition.

However, in SQL, manually grouping and sorting records can become cumbersome and inefficient on large datasets. We want a more scalable and declarative solution.

### Optimal Approach

The key insight is that SQL window functions are designed exactly for problems involving ordered records within partitions.

We can use:

- `ROW_NUMBER()` to determine the chronological position of each transaction within a user's history.
- `LAG()` to access the spending values of the two preceding transactions.

Once each row knows:

- its transaction order,
- the previous transaction spend,
- the transaction before that,

we simply filter for rows where:

- the transaction number is `3`
- the current spend is greater than both previous spends.

This solution is efficient, concise, and leverages database optimizations for window operations.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) | O(n) | Group and sort transactions manually |
| Optimal | O(n log n) | O(n) | Uses SQL window functions for ordered comparisons |

## Algorithm Walkthrough

1. Partition transactions by `user_id`.

We process each user's transactions independently because transaction order only matters within the same user.
2. Order transactions by `transaction_date`.

Chronological ordering is essential because we need to identify the first, second, and third transactions accurately.
3. Assign row numbers using `ROW_NUMBER()`.

This gives every transaction a sequential index:

- first transaction → row number 1
- second transaction → row number 2
- third transaction → row number 3
4. Use `LAG()` to retrieve previous spending values.

`LAG(spend, 1)` gives the immediately previous transaction spend.

`LAG(spend, 2)` gives the spend from two transactions earlier.
5. Filter rows where:

- the row number equals `3`
- current spend is greater than both previous spends
6. Return the required columns ordered by `user_id`.

### Why it works

The algorithm works because window functions preserve the chronological ordering of transactions within each user partition. By assigning row numbers and retrieving preceding spending values directly from the ordered sequence, we can accurately determine whether the third transaction exceeds both earlier transactions. Since every transaction is evaluated relative to its true historical position, the result is correct for all valid inputs.

## SQL Solution

```sql
WITH ordered_transactions AS (
    SELECT
        user_id,
        spend,
        transaction_date,
        ROW_NUMBER() OVER (
            PARTITION BY user_id
            ORDER BY transaction_date
        ) AS transaction_number,
        LAG(spend, 1) OVER (
            PARTITION BY user_id
            ORDER BY transaction_date
        ) AS previous_spend_1,
        LAG(spend, 2) OVER (
            PARTITION BY user_id
            ORDER BY transaction_date
        ) AS previous_spend_2
    FROM Transactions
)

SELECT
    user_id,
    spend AS third_transaction_spend,
    transaction_date AS third_transaction_date
FROM ordered_transactions
WHERE transaction_number = 3
    AND spend > previous_spend_1
    AND spend > previous_spend_2
ORDER BY user_id;
```

## Implementation Explanation

The Common Table Expression named `ordered_transactions` prepares all the information we need.

`ROW_NUMBER()` assigns each transaction a chronological position inside its user group.

The two `LAG()` calls retrieve the spending amounts from the previous two transactions. These values are aligned automatically with the current row.

After constructing this enriched dataset, the outer query filters only the rows representing the third transaction.

The conditions:

```
spend > previous_spend_1
AND spend > previous_spend_2
```

ensure that the third transaction is strictly larger than both earlier transactions.

Finally, the result is sorted by `user_id` as required.

## Worked Example

Consider the following transactions for user `1`:

| user_id | spend | transaction_date |
| --- | --- | --- |
| 1 | 7.44 | 2023-11-02 |
| 1 | 49.78 | 2023-11-12 |
| 1 | 65.56 | 2023-11-18 |
| 1 | 96.00 | 2023-11-30 |

After applying window functions:

| transaction_number | spend | previous_spend_1 | previous_spend_2 |
| --- | --- | --- | --- |
| 1 | 7.44 | NULL | NULL |
| 2 | 49.78 | 7.44 | NULL |
| 3 | 65.56 | 49.78 | 7.44 |
| 4 | 96.00 | 65.56 | 49.78 |

We only inspect row number `3`.

For this row:

```
65.56 > 49.78
65.56 > 7.44
```

Both conditions are true, so user `1` is included in the result.

Now consider user `3`:

| transaction_number | spend | previous_spend_1 | previous_spend_2 |
| --- | --- | --- | --- |
| 1 | 37.33 | NULL | NULL |
| 2 | 13.89 | 37.33 | NULL |
| 3 | 7.00 | 13.89 | 37.33 |

For the third transaction:

```
7.00 > 13.89
```

This condition fails, so the user is excluded.

## Edge Cases

### Users With Fewer Than Three Transactions

A user may only have one or two transactions. In such cases, there is no third transaction to evaluate. The implementation handles this naturally because `ROW_NUMBER()` never produces a row numbered `3` for those users, so they are excluded automatically.

### Third Transaction Equal To Previous Transactions

The condition requires the third transaction to be strictly greater than the first two. If the third transaction equals either earlier value, the row must not appear in the result. The implementation uses the `>` operator instead of `>=`, ensuring correct behavior.

### Transactions Inserted Out Of Chronological Order

Database rows are not guaranteed to be stored chronologically. A naive implementation that relies on insertion order would produce incorrect results. The solution explicitly orders transactions using `ORDER BY transaction_date`, guaranteeing correct transaction sequencing regardless of physical row order.
