# LeetCode 2998 - Minimum Number of Operations to Make X and Y Equal

## Problem Understanding

This problem gives us two positive integers, `x` and `y`, and asks for the minimum number of operations required to transform `x` into `y`.

We are allowed to perform four operations on `x`:

1. Divide by `11`, but only if `x` is divisible by `11`
2. Divide by `5`, but only if `x` is divisible by `5`
3. Increment by `1`
4. Decrement by `1`

The important detail is that every operation has equal cost, exactly one step. Our goal is therefore to find the shortest sequence of operations that converts `x` into `y`.

The input consists of two integers between `1` and `10^4`. Since the values are relatively small, we can consider graph based shortest path techniques such as Breadth-First Search. Every integer value can be viewed as a node in a graph, and every valid operation creates an edge to another number.

The expected output is a single integer representing the minimum number of operations needed.

A few edge cases are important to think about immediately.

If `x == y`, the answer is `0` because no operations are needed.

If `x < y`, division operations are usually not useful because they decrease the value further away from the target. In such cases, repeatedly incrementing may already be optimal.

Another subtle case occurs when temporarily increasing `x` creates a better division opportunity. For example, in the second sample, `54` becomes `55`, then divides by `11`, then by `5`. A greedy strategy that only reduces the number would fail here.

Because operations can both increase and decrease values, naive recursion can revisit the same states repeatedly and become inefficient or even infinite without memoization or visited tracking.

## Approaches

### Brute Force Approach

A brute force solution would recursively try every possible operation from the current value until it reaches `y`.

From any value, we could:

- Increment
- Decrement
- Divide by `5` if divisible
- Divide by `11` if divisible

We would recursively explore all possible paths and keep the minimum number of operations.

This approach is correct because it eventually explores every valid sequence of operations. However, it is extremely inefficient because the same states are recomputed many times. Since operations can increase and decrease numbers, the recursion tree can grow exponentially and may even loop forever without additional safeguards.

For example, from `10`, we could go to `11`, then back to `10`, repeatedly.

### Optimal Approach

The key insight is that this problem is actually a shortest path problem on an implicit graph.

Each integer value represents a node, and each operation represents an edge with weight `1`.

Since every operation costs the same, Breadth-First Search naturally finds the shortest path. BFS explores all states reachable in `1` operation, then `2` operations, and so on. The first time we reach `y`, we are guaranteed to have found the minimum number of operations.

We maintain:

- A queue for BFS traversal
- A visited set to avoid revisiting states
- The current distance, representing operation count

We also place practical bounds on explored numbers. Since `x` and `y` are at most `10^4`, we do not need to explore extremely large values. A reasonable upper bound such as `2 * max(x, y) + 10` safely contains all useful intermediate states.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Explores all operation sequences recursively |
| Optimal BFS | O(M) | O(M) | Uses shortest path traversal over reachable states |

Here, `M` represents the number of reachable states within the bounded search space.

## Algorithm Walkthrough

1. Start by handling the simplest case. If `x == y`, return `0` immediately because no operations are required.
2. Create a queue for Breadth-First Search. Each queue entry stores the current number and the number of operations used to reach it.
3. Create a `visited` set and insert the starting value `x`. This prevents revisiting the same state multiple times and avoids infinite loops.
4. Define an upper exploration limit. Since useful states never need to grow arbitrarily large, we use:

```
limit = 2 * max(x, y) + 10
```

This safely covers all meaningful intermediate values.
5. Begin BFS traversal:

- Remove the front element from the queue.
- If the current number equals `y`, return the current operation count.
6. Generate all possible next states:

- `current + 1`
- `current - 1`
- `current // 5` if divisible by `5`
- `current // 11` if divisible by `11`
7. For every generated state:

- Ignore it if it is negative.
- Ignore it if it exceeds the chosen limit.
- Ignore it if already visited.
- Otherwise, mark it visited and add it to the queue with operation count incremented by `1`.
8. Continue BFS until `y` is reached.

### Why it works

Breadth-First Search explores states in increasing order of distance from the starting value. Since every operation costs exactly one step, the first time BFS reaches `y` must correspond to the shortest possible sequence of operations. The visited set guarantees that each state is processed only once, preventing cycles and redundant work.

## Python Solution

```python
from collections import deque

class Solution:
    def minimumOperationsToMakeEqual(self, x: int, y: int) -> int:
        if x == y:
            return 0
        
        limit = 2 * max(x, y) + 10
        
        queue = deque([(x, 0)])
        visited = {x}
        
        while queue:
            current, operations = queue.popleft()
            
            if current == y:
                return operations
            
            next_states = [
                current + 1,
                current - 1
            ]
            
            if current % 5 == 0:
                next_states.append(current // 5)
            
            if current % 11 == 0:
                next_states.append(current // 11)
            
            for next_value in next_states:
                if 0 <= next_value <= limit and next_value not in visited:
                    visited.add(next_value)
                    queue.append((next_value, operations + 1))
        
        return -1
```

The implementation begins with the trivial equality check. If `x` already equals `y`, the algorithm returns immediately.

The queue stores pairs of `(current_value, operation_count)`. This allows BFS to track how many operations were required to reach each state.

The `visited` set is critical because operations like increment and decrement can easily create cycles. Without visited tracking, the search could revisit the same values indefinitely.

For every current value, the algorithm constructs all valid next states. Division operations are only added when the current value is divisible by the required divisor.

Each valid unseen state is inserted into the queue with operation count increased by one. Because BFS processes states layer by layer, the first time we encounter `y` is guaranteed to be optimal.

## Go Solution

```go
package main

import "container/list"

func minimumOperationsToMakeEqual(x int, y int) int {
	if x == y {
		return 0
	}

	limit := 2*max(x, y) + 10

	type State struct {
		value      int
		operations int
	}

	queue := list.New()
	queue.PushBack(State{x, 0})

	visited := make(map[int]bool)
	visited[x] = true

	for queue.Len() > 0 {
		front := queue.Front()
		queue.Remove(front)

		current := front.Value.(State)

		if current.value == y {
			return current.operations
		}

		nextStates := []int{
			current.value + 1,
			current.value - 1,
		}

		if current.value%5 == 0 {
			nextStates = append(nextStates, current.value/5)
		}

		if current.value%11 == 0 {
			nextStates = append(nextStates, current.value/11)
		}

		for _, nextValue := range nextStates {
			if nextValue >= 0 && nextValue <= limit && !visited[nextValue] {
				visited[nextValue] = true
				queue.PushBack(State{nextValue, current.operations + 1})
			}
		}
	}

	return -1
}

func max(a int, b int) int {
	if a > b {
		return a
	}
	return b
}
```

The Go implementation follows the same BFS strategy as the Python solution.

Since Go does not provide a built in deque, the implementation uses `container/list` as the queue structure. A custom `State` struct stores both the current value and operation count.

The visited set is implemented using `map[int]bool`. Integer overflow is not a concern because the problem constraints are small and the search space is explicitly bounded.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(M) | Each reachable state is processed once |
| Space | O(M) | Queue and visited set store reachable states |

The BFS explores every reachable integer state at most once. Each state generates at most four neighboring states. Because the search space is bounded, the total runtime grows linearly with the number of explored states.

The visited set and queue together also require linear space proportional to the number of processed states.

## Edge Cases

One important edge case occurs when `x` already equals `y`. A careless implementation might still start BFS unnecessarily or even produce incorrect results if no early return exists. The implementation handles this immediately by returning `0`.

Another tricky case occurs when increasing first leads to a better division path. For example, `54 -> 55 -> 5 -> 1 -> 2`. A greedy algorithm that only decreases values would miss this optimal solution. BFS correctly explores all shortest possibilities simultaneously, including temporary increases.

A third important edge case involves cycles created by increment and decrement operations. Without a visited set, the algorithm could repeatedly move between adjacent numbers forever. The implementation prevents this by marking every processed value and never revisiting it.

Finally, large values near the upper constraint could cause excessive exploration if no bounds are enforced. The implementation limits the search space using `2 * max(x, y) + 10`, which safely includes all useful states while preventing runaway growth.
