## Solution

## Problem Understanding

The problem gives us an integer array `nums`, along with two constraints:

- The distance between two indices must be at least `indexDifference`
- The difference between the values at those indices must be at least `valueDifference`

More formally, we must find indices `i` and `j` such that:

$|i-j|\geq \text{indexDifference}$

and

$|\text{nums}[i]-\text{nums}[j]|\geq \text{valueDifference}$

If such a pair exists, we return `[i, j]`. Otherwise, we return `[-1, -1]`.

An important detail is that `i` and `j` are allowed to be equal. This matters when both required differences are zero, because then any index automatically satisfies both conditions.

The input array can contain up to `10^5` elements, which is large enough that quadratic solutions will be too slow. The values inside the array can also be very large, up to `10^9`, so we should avoid solutions that depend on value ranges such as counting arrays or bucket arrays.

The problem only asks for any valid pair, not necessarily the first, smallest, or lexicographically smallest pair. This gives us flexibility in the implementation.

Several edge cases are important:

- `indexDifference = 0`, meaning the same index can be used
- `valueDifference = 0`, meaning any pair with enough index distance works
- Arrays of length `1`
- Strictly increasing or decreasing arrays
- Cases where no valid pair exists
- Multiple valid answers, where the algorithm may return any one of them

The large constraint on `n` strongly suggests we need an `O(n)` or `O(n log n)` solution.

## Approaches

### Brute Force Approach

The most straightforward solution is to check every possible pair of indices `(i, j)`.

For every pair:

1. Compute the index distance
2. Compute the value difference
3. If both conditions are satisfied, return the pair immediately

This works because it exhaustively examines every possible combination, guaranteeing that a valid pair will be found if one exists.

However, the time complexity is too high. There are `n^2` possible pairs, and with `n = 10^5`, this becomes approximately `10^10` comparisons in the worst case, which is far beyond acceptable limits.

### Optimal Observation

The key observation is that while scanning the array from left to right, we only need to compare the current element with elements that are at least `indexDifference` positions away.

Suppose we are currently at index `j`. Then all valid candidate indices `i` must satisfy:

$i\leq j-\text{indexDifference}$

Among all those candidates, we do not need every value individually. We only care about:

- The minimum value seen so far
- The maximum value seen so far

Why? Because if the current value can form a sufficiently large difference with any earlier value, then it must also form a sufficiently large difference with either the minimum or maximum earlier value.

So while iterating through the array:

- Maintain the smallest eligible value and its index
- Maintain the largest eligible value and its index
- For each current index, check whether either extreme satisfies the value difference condition

This reduces the problem to linear time.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Checks every pair of indices |
| Optimal | O(n) | O(1) | Tracks running minimum and maximum eligible values |

## Algorithm Walkthrough

1. Initialize two variables to track the minimum eligible value and maximum eligible value seen so far. Also store their corresponding indices.
2. Start iterating through the array using index `j`.
3. For each `j`, determine which index becomes newly eligible for comparison. That index is:

$k=j-\text{indexDifference}$

Any index less than or equal to `k` satisfies the required index distance.

1. Update the running minimum value if `nums[k]` is smaller than the current minimum. Also store index `k` as the minimum index.
2. Update the running maximum value if `nums[k]` is larger than the current maximum. Also store index `k` as the maximum index.
3. Compare the current value `nums[j]` with the running minimum value. If their difference is at least `valueDifference`, return the corresponding pair.
4. Compare the current value `nums[j]` with the running maximum value. If their difference is at least `valueDifference`, return the corresponding pair.
5. Continue scanning until a valid pair is found.
6. If the loop finishes without finding any valid pair, return `[-1, -1]`.

### Why it works

At index `j`, every valid partner index must belong to the eligible prefix ending at `j - indexDifference`.

If there exists some eligible value whose difference from `nums[j]` is large enough, then either the minimum eligible value or the maximum eligible value must also produce a difference large enough. This is because the largest absolute deviation from a number occurs at one of the extremes.

Therefore, tracking only the minimum and maximum eligible values is sufficient to detect every possible valid pair.

## Python Solution

```python
from typing import List

class Solution:
    def findIndices(
        self,
        nums: List[int],
        indexDifference: int,
        valueDifference: int
    ) -> List[int]:

        n = len(nums)

        min_index = 0
        max_index = 0

        for j in range(indexDifference, n):
            candidate_index = j - indexDifference

            if nums[candidate_index] < nums[min_index]:
                min_index = candidate_index

            if nums[candidate_index] > nums[max_index]:
                max_index = candidate_index

            if abs(nums[j] - nums[min_index]) >= valueDifference:
                return [min_index, j]

            if abs(nums[j] - nums[max_index]) >= valueDifference:
                return [max_index, j]

        return [-1, -1]
```

The implementation maintains two running indices:

- `min_index`, pointing to the smallest eligible value
- `max_index`, pointing to the largest eligible value

The loop begins at `indexDifference`, because earlier indices cannot satisfy the required index distance with any previous position.

For each position `j`, the algorithm first adds the newly eligible index into consideration. This keeps the tracked minimum and maximum consistent with all valid earlier positions.

After updating the extremes, the algorithm checks whether the current value differs enough from either extreme. If so, the pair is immediately returned.

The use of indices instead of storing raw values allows the function to directly return the required answer without additional bookkeeping.

## Go Solution

```go
func findIndices(nums []int, indexDifference int, valueDifference int) []int {
	n := len(nums)

	minIndex := 0
	maxIndex := 0

	for j := indexDifference; j < n; j++ {
		candidateIndex := j - indexDifference

		if nums[candidateIndex] < nums[minIndex] {
			minIndex = candidateIndex
		}

		if nums[candidateIndex] > nums[maxIndex] {
			maxIndex = candidateIndex
		}

		if abs(nums[j]-nums[minIndex]) >= valueDifference {
			return []int{minIndex, j}
		}

		if abs(nums[j]-nums[maxIndex]) >= valueDifference {
			return []int{maxIndex, j}
		}
	}

	return []int{-1, -1}
}

func abs(x int) int {
	if x < 0 {
		return -x
	}
	return x
}
```

The Go implementation closely mirrors the Python solution.

One notable difference is that Go does not provide a built in integer absolute value function for plain `int`, so we define a small helper function `abs`.

The returned slices are created directly using literal syntax such as `[]int{minIndex, j}`.

Since the constraints fit safely within 32 bit integers, integer overflow is not an issue in Go for subtraction operations involving these values.

## Worked Examples

### Example 1

Input:

```
nums = [5,1,4,1]
indexDifference = 2
valueDifference = 4
```

We iterate through the array while maintaining eligible minimum and maximum values.

| j | candidate_index | Eligible Values | min_index | max_index | Check Result |
| --- | --- | --- | --- | --- | --- |
| 2 | 0 | [5] | 0 | 0 | abs(4 - 5) = 1 |
| 3 | 1 | [5,1] | 1 | 0 | abs(1 - 5) = 4 |

At `j = 3`:

- `abs(nums[3] - nums[0])`
- `abs(1 - 5) = 4`

Both conditions are satisfied, so we return:

```
[0, 3]
```

### Example 2

Input:

```
nums = [2,1]
indexDifference = 0
valueDifference = 0
```

Since `indexDifference = 0`, the same index is allowed.

| j | candidate_index | min_index | max_index | Result |
| --- | --- | --- | --- | --- |
| 0 | 0 | 0 | 0 | abs(2 - 2) = 0 |

The condition is immediately satisfied.

Output:

```
[0,0]
```

### Example 3

Input:

```
nums = [1,2,3]
indexDifference = 2
valueDifference = 4
```

| j | candidate_index | Eligible Values | Difference Checks |
| --- | --- | --- | --- |
| 2 | 0 | [1] | abs(3 - 1) = 2 |

The value difference never reaches `4`.

Therefore:

```
[-1,-1]
```

## Edge Cases

One important edge case occurs when `indexDifference = 0`. In this situation, an index may pair with itself. A naive implementation might incorrectly assume the indices must be distinct and skip valid solutions. This implementation naturally handles the case because the loop begins at `j = 0`, making index `0` immediately eligible.

Another subtle edge case occurs when all array values are identical. If `valueDifference > 0`, then no pair can ever satisfy the condition. Some implementations may accidentally return a pair because the index condition succeeds. Here, the algorithm explicitly checks the absolute value difference before returning.

A third important case is when `indexDifference` exceeds the array length. In such situations, no two indices can satisfy the required distance. The loop simply never executes because the starting value is already outside the array bounds, and the algorithm correctly returns `[-1, -1]`.

A final edge case involves very large integer values near `10^9`. Since the algorithm only performs subtraction and absolute value operations, and does not use multiplication or nested structures, it safely handles large numbers without overflow concerns in either Python or Go.
