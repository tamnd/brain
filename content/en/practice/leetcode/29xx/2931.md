**Difficulty:** 🔴 Hard  
**Topics:** Array, Greedy, Sorting, Heap (Priority Queue), Matrix

## Solution

## Problem Understanding

We are given an `m x n` matrix called `values`, where each row represents a shop and each column represents an item in that shop. Every row is sorted in non-increasing order, meaning larger values appear on the left and smaller values appear on the right.

Each day, we must buy exactly one item. However, there is an important restriction: from any shop, we can only buy the current rightmost remaining item. Since rows are sorted in descending order, this means we always buy items from smallest to largest within the same shop.

If we buy an item with value `x` on day `d`, the amount spent is:

$\text{cost} = x \cdot d$

Our goal is to maximize the total money spent after purchasing all `m * n` items.

The key observation is that the multiplier `d` increases every day. Therefore, larger values should ideally be purchased later so they receive larger multipliers.

The constraints are important:

- `m <= 10`
- `n <= 10^4`
- Total items can reach `10^5`
- Values can be as large as `10^6`

A brute-force simulation that repeatedly searches all available items would become too slow if implemented inefficiently. We need an approach close to `O(m * n log m)` or `O(m * n log(m * n))`.

The problem guarantees that each row is sorted in descending order. This structure is the foundation of the optimal solution.

Several edge cases are important:

- A single shop
- A single item
- Rows containing very different magnitudes
- Rows where values are already globally ordered
- Large input sizes requiring efficient heap operations

The problem also guarantees that all items are distinct, so we never need tie-breaking logic.

## Approaches

### Brute Force Approach

A straightforward solution is to simulate every day independently.

On each day, we inspect every shop and determine which rightmost item is currently available. Among all currently available items, we choose the smallest one because smaller values should receive smaller day multipliers, leaving larger values for later days.

This greedy choice is correct because assigning larger multipliers to larger values maximizes the weighted sum.

However, the naive implementation repeatedly scans all shops to find the minimum available item. Since there are `m * n` total purchases and each search may cost `O(m)`, the total complexity becomes:

$O(m^2 n)$

Although `m` is small here, this approach is still not ideal conceptually and does not scale well.

### Optimal Approach

The important insight is that every shop behaves like a sorted stream of items.

Because we can only buy from right to left, each row effectively exposes items in ascending order:

- Rightmost item first
- Then the next rightmost
- And so on

This means each row can be treated as a sorted ascending sequence during the purchasing process.

Now the problem becomes:

- Merge `m` sorted streams
- Repeatedly extract the globally smallest available value
- Assign increasing day multipliers

A min-heap is perfect for this.

We initialize the heap with the rightmost item from every row. Then:

1. Extract the smallest value
2. Add `value * day` to the answer
3. Move left in that same row
4. Push the next available item into the heap

This guarantees that values are purchased globally in ascending order.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m²n) | O(m) | Repeatedly scans all shops to find smallest available item |
| Optimal | O(mn log m) | O(m) | Uses a min-heap to efficiently merge sorted shop streams |

## Algorithm Walkthrough

1. Create a min-heap.

Each heap entry stores:

- Item value
- Row index
- Column index

The heap always contains the next available item from each shop.
2. Initialize the heap.

Since purchases happen from right to left, the first available item in every row is the rightmost element.

Push:

```
(values[i][n - 1], i, n - 1)
```

for every row `i`.
3. Initialize:

- `day = 1`
- `answer = 0`
4. Repeat until the heap becomes empty.

At each step:

- Pop the smallest value from the heap
- Add `value * day` to the answer
- Increment the day counter
5. After removing an item from row `r` and column `c`, check whether another item exists to its left.

If `c > 0`, then:

- The next available item is `values[r][c - 1]`
- Push it into the heap
6. Continue until all items are purchased.

### Why it works

The algorithm always purchases the smallest currently available value first.

Suppose two values `a < b` are assigned to days `x < y`.

Their contribution is:

$a \cdot x + b \cdot y$

If we swap them:

$a \cdot y + b \cdot x$

The original arrangement is better because:

$(a \cdot x + b \cdot y) - (a \cdot y + b \cdot x) = (b-a)(y-x) > 0$

Therefore, smaller values should always receive smaller multipliers. The heap guarantees this ordering globally.

## Python Solution

```python
from typing import List
import heapq

class Solution:
    def maxSpending(self, values: List[List[int]]) -> int:
        m = len(values)
        n = len(values[0])

        min_heap = []

        # Start with the rightmost item from each shop
        for row in range(m):
            heapq.heappush(
                min_heap,
                (values[row][n - 1], row, n - 1)
            )

        day = 1
        total_spending = 0

        while min_heap:
            value, row, col = heapq.heappop(min_heap)

            total_spending += value * day
            day += 1

            # Push the next available item from the same shop
            if col > 0:
                heapq.heappush(
                    min_heap,
                    (values[row][col - 1], row, col - 1)
                )

        return total_spending
```

The implementation directly follows the heap-based merge strategy.

The heap stores the next available item from each shop. Initially, this is the rightmost item because purchases must proceed from right to left.

During each iteration:

- We pop the globally smallest available value
- Multiply it by the current day
- Add the result to the running total
- Push the next item from the same row if one exists

Since the heap size never exceeds `m`, heap operations are very efficient.

## Go Solution

```go
package main

import (
	"container/heap"
)

type Item struct {
	value int
	row   int
	col   int
}

type MinHeap []Item

func (h MinHeap) Len() int {
	return len(h)
}

func (h MinHeap) Less(i, j int) bool {
	return h[i].value < h[j].value
}

func (h MinHeap) Swap(i, j int) {
	h[i], h[j] = h[j], h[i]
}

func (h *MinHeap) Push(x interface{}) {
	*h = append(*h, x.(Item))
}

func (h *MinHeap) Pop() interface{} {
	old := *h
	n := len(old)

	item := old[n-1]
	*h = old[:n-1]

	return item
}

func maxSpending(values [][]int) int64 {
	m := len(values)
	n := len(values[0])

	h := &MinHeap{}
	heap.Init(h)

	// Push rightmost item from each row
	for row := 0; row < m; row++ {
		heap.Push(h, Item{
			value: values[row][n-1],
			row:   row,
			col:   n - 1,
		})
	}

	var total int64 = 0
	var day int64 = 1

	for h.Len() > 0 {
		item := heap.Pop(h).(Item)

		total += int64(item.value) * day
		day++

		// Push next item from same row
		if item.col > 0 {
			heap.Push(h, Item{
				value: values[item.row][item.col-1],
				row:   item.row,
				col:   item.col - 1,
			})
		}
	}

	return total
}
```

The Go solution uses the `container/heap` package to implement the priority queue.

One important detail is integer overflow. Since values and multipliers can become large, the final answer must use `int64`.

The heap stores custom `Item` structs containing:

- Value
- Row index
- Column index

Otherwise, the logic is identical to the Python implementation.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(mn log m) | Each item is pushed and popped once from a heap of size at most `m` |
| Space | O(m) | Heap stores at most one active item per row |

The heap never grows beyond the number of rows because only one candidate from each shop is active at a time.

There are exactly `m * n` heap removals and at most `m * n` insertions. Each heap operation costs `O(log m)`.

## Edge Cases

### Single Shop

If there is only one shop, the algorithm must simply buy items from right to left. Since the row is already sorted in descending order, buying from the right automatically processes values in ascending order.

The implementation handles this naturally because the heap initially contains only one item.

### Single Column

When each row contains only one item, every row contributes exactly one heap entry and no further insertions occur.

This case is important because incorrect column handling could produce out-of-bounds access when attempting to move left.

The condition:

```
if col > 0:
```

prevents invalid accesses.

### Very Large Values

The final answer can become large because:

- Values can be up to `10^6`
- There can be `10^5` items
- Day multipliers also grow to `10^5`

The Go solution explicitly uses `int64` to avoid overflow. Python integers automatically expand, so no extra handling is necessary there.
