## Solution

## Problem Understanding

We are given three integers, `a`, `b`, and `n`. We must choose an integer `x` such that:

- `0 <= x < 2^n`

Then we compute:

$$(a \oplus x) \times (b \oplus x)$$

where `⊕` is the bitwise XOR operation.

Our goal is to maximize this product and return the result modulo:

$$10^9 + 7$$

The important detail is that `x` is restricted to only the lowest `n` bits. Since:

$$0 \le x < 2^n$$

the binary representation of `x` can only affect bit positions `0` through `n - 1`. Any higher bits of `a` and `b` remain unchanged.

The constraints are:

- `0 <= a, b < 2^50`
- `0 <= n <= 50`

This immediately tells us that brute forcing all values of `x` is impossible when `n` is large, because there are:

$$2^n$$

possible choices.

For `n = 50`, that is more than one quadrillion candidates.

The problem is fundamentally about bit manipulation and greedy optimization.

Several edge cases are important:

- `n = 0`, meaning only `x = 0` is allowed
- `a = b`, where symmetry matters
- one value much larger than the other
- bits above position `n - 1`, which cannot be changed
- maximizing a product rather than a sum, which means balancing the two numbers is important

The key challenge is deciding, for each bit position, whether setting that bit in `x` improves the final product.

## Approaches

### Brute Force Approach

The most direct solution is to try every possible value of `x`.

For each candidate:

1. Compute `A = a XOR x`
2. Compute `B = b XOR x`
3. Compute `A * B`
4. Track the maximum

This works because it explicitly evaluates every valid choice of `x`, so the best answer is guaranteed to be found.

However, the number of candidates is:

$$2^n$$

When `n = 50`, this is computationally impossible.

Even if each evaluation took only a nanosecond, the runtime would still be enormous.

So we need a smarter approach.

### Key Insight

Only the lowest `n` bits can change.

Consider a single bit position `i`.

There are two possibilities:

- `a_i == b_i`
- `a_i != b_i`

If the bits are equal:

- both `0`
- or both `1`

then we can make both resulting bits become `1`.

For example:

| Original | Choose `x_i` | Result |
| --- | --- | --- |
| `(0,0)` | `1` | `(1,1)` |
| `(1,1)` | `0` | `(1,1)` |

Since setting both numbers to `1` at a high bit increases both multiplicands, it is always optimal.

Now consider when the bits differ:

| Original | Result if `x_i=0` | Result if `x_i=1` |
| --- | --- | --- |
| `(0,1)` | `(0,1)` | `(1,0)` |

The total contribution remains the same, but the `1` bit moves from one number to the other.

This means we should greedily assign the larger bit to whichever current number is smaller, because balanced numbers maximize the product.

This follows from the mathematical principle that for a fixed sum, the product is maximized when the numbers are as close as possible.

So the strategy becomes:

- if bits are equal, make both `1`
- if bits differ, give the `1` bit to the currently smaller number

This leads to an optimal greedy bit-by-bit construction.

| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(1) | Tries every possible x |
| Optimal Greedy Bit Manipulation | O(n) | O(1) | Processes each bit independently |

## Algorithm Walkthrough

### Step 1: Separate immutable higher bits

Bits above position `n - 1` cannot be changed because `x` only contains `n` bits.

So we initialize:

$$A = a >> n << n$$

$$B = b >> n << n$$

These preserve all higher bits.

### Step 2: Process bits from high to low

We iterate from bit `n - 1` down to `0`.

Higher bits matter more because their numeric contribution is larger.

At each bit:

```
bit = 1 << i
```

### Step 3: Extract current bits

We check whether bit `i` is set in `a` and `b`.

```
abit = a & bit
bbit = b & bit
```

### Step 4: If bits are equal

If both bits are the same:

- both `0`
- or both `1`

then we can always make both resulting bits equal to `1`.

So we set:

```
A |= bit
B |= bit
```

### Step 5: If bits differ

Exactly one of the numbers can receive the `1` bit.

To maximize the final product, we assign the `1` to the currently smaller value.

```
if A < B:
    A |= bit
else:
    B |= bit
```

This greedily balances the two numbers.

### Step 6: Compute final answer

After processing all bits:

```
(A * B) % MOD
```

where:

```
MOD = 10^9 + 7
```

### Why it works

The algorithm works because each bit can be optimized independently.

When bits are equal, making both resulting bits `1` strictly improves the product.

When bits differ, the total contribution of that bit is fixed, exactly one number receives the `1`. Since product maximization favors balanced multiplicands, assigning the larger bit to the currently smaller number is always optimal.

Processing from high bits to low bits ensures that the most significant decisions are made first.

## Python Solution

```python
class Solution:
    def maximumXorProduct(self, a: int, b: int, n: int) -> int:
        MOD = 10**9 + 7

        # Preserve bits above n-1
        A = (a >> n) << n
        B = (b >> n) << n

        # Process mutable bits from high to low
        for i in range(n - 1, -1, -1):
            bit = 1 << i

            abit = a & bit
            bbit = b & bit

            # If bits are equal, make both 1
            if abit == bbit:
                A |= bit
                B |= bit
            else:
                # Give the 1-bit to the smaller number
                if A < B:
                    A |= bit
                else:
                    B |= bit

        return (A * B) % MOD
```

The implementation starts by extracting the immutable higher bits. Since `x` only affects the lowest `n` bits, everything above that remains fixed.

Next, the loop processes bits from most significant to least significant. This ordering is important because higher bits contribute more to the final product.

For equal bits, both numbers can safely receive a `1` in that position, so the code sets the bit in both `A` and `B`.

For differing bits, only one number can receive the `1`. The implementation compares the current values of `A` and `B` and gives the bit to the smaller one. This greedily keeps the numbers balanced.

Finally, the product is returned modulo `10^9 + 7`.

## Go Solution

```go
package main

func maximumXorProduct(a int64, b int64, n int) int {
	const MOD int64 = 1_000_000_007

	// Preserve bits above n-1
	A := (a >> n) << n
	B := (b >> n) << n

	// Process mutable bits from high to low
	for i := n - 1; i >= 0; i-- {
		bit := int64(1) << i

		abit := a & bit
		bbit := b & bit

		// If bits are equal, make both 1
		if abit == bbit {
			A |= bit
			B |= bit
		} else {
			// Give the 1-bit to the smaller number
			if A < B {
				A |= bit
			} else {
				B |= bit
			}
		}
	}

	return int((A % MOD) * (B % MOD) % MOD)
}
```

The Go version follows the same logic as the Python implementation.

One important difference is integer handling. Go uses fixed-width integer types, so the implementation uses `int64` throughout to safely store values up to approximately `2^50`.

The modulo multiplication is also performed using `int64` before converting back to `int`.

## Edge Cases

One important edge case is when `n = 0`. In this scenario, the only valid value of `x` is `0`, because:

$$0 \le x < 2^0 = 1$$

A naive implementation might still attempt to process bits or modify values incorrectly. This implementation handles it naturally because the loop runs zero times, leaving `A = a` and `B = b`.

Another critical edge case occurs when `a` and `b` already have identical bits in many positions. In such situations, the optimal strategy is always to make both resulting bits equal to `1`. A buggy implementation might unnecessarily flip bits or fail to maximize both numbers simultaneously. This solution explicitly checks for equal bits and always sets them in both values.

A third important edge case involves immutable higher bits. Since only the lowest `n` bits may change, bits above position `n - 1` must remain untouched. Forgetting this detail can produce invalid states that cannot actually be obtained using XOR with a restricted `x`. The implementation correctly preserves these higher bits before processing the mutable region.

Another subtle case is when one number becomes much larger than the other during processing. Since the objective is maximizing a product rather than a sum, balance matters significantly. The greedy rule that assigns differing bits to the smaller current value ensures the numbers remain as close as possible, which maximizes the product.
