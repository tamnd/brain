# LeetCode 2964 - Number of Divisible Triplet Sums

## Problem Understanding

The problem gives us an integer array `nums` and another integer `d`. We must count how many triplets `(i, j, k)` satisfy two conditions:

1. The indices are strictly increasing, meaning `i < j < k`
2. The sum of the three selected elements is divisible by `d`

In mathematical terms, we need to count every triplet where:

$$(nums[i] + nums[j] + nums[k]) \bmod d = 0$$

The array is 0-indexed, so valid indices range from `0` to `n - 1`.

The output is a single integer representing the total number of valid triplets.

The constraints are important:

- `nums.length <= 1000`
- `nums[i] <= 10^9`
- `d <= 10^9`

A brute-force solution that checks every triplet would require three nested loops, leading to `O(n^3)` complexity. With `n = 1000`, this means potentially around:

$$\frac{1000 \times 999 \times 998}{6} \approx 166 \text{ million triplets}$$

That is too slow for an efficient interview-quality solution.

The key observation is that divisibility depends only on remainders modulo `d`. Instead of checking every third element explicitly, we can compute the required remainder and count matching candidates efficiently.

Some important edge cases include:

- Arrays with fewer than 3 meaningful combinations
- Cases where every triplet is valid
- Cases where no triplet is valid
- Large element values, where modulo arithmetic becomes essential
- Duplicate numbers, where different index combinations still count separately

The problem guarantees valid input sizes and positive `d`, so division by zero is never an issue.

## Approaches

### Brute Force Approach

The most direct solution uses three nested loops.

We iterate through every possible combination of indices `(i, j, k)` such that `i < j < k`. For each triplet, we compute:

```
(nums[i] + nums[j] + nums[k]) % d
```

If the result is `0`, we increment the answer.

This approach is correct because it explicitly checks every possible triplet exactly once. However, it performs too many operations for larger inputs.

With `n = 1000`, the cubic complexity becomes impractical.

### Optimal Approach

The key insight is that modular arithmetic allows us to transform the problem.

Suppose we fix indices `i` and `j`. Then we already know:

$$current = (nums[i] + nums[j]) \bmod d$$

To make the total divisible by `d`, we need:

$$(current + nums[k]) \bmod d = 0$$

Rearranging:

$$nums[k] \bmod d = (d - current) \bmod d$$

So instead of searching every `k` individually, we can count how many future numbers have the required remainder.

We use a hash map to store frequencies of remainders for elements to the right of the current pair.

This reduces the complexity from `O(n^3)` to `O(n^2)`.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n³) | O(1) | Checks every possible triplet explicitly |
| Optimal | O(n²) | O(n) | Uses remainder frequencies to count valid third elements efficiently |

## Algorithm Walkthrough

### Optimal Algorithm

1. Initialize a variable `answer` to store the total number of valid triplets.
2. Iterate through the array using index `j` as the middle element of the triplet.
3. For every fixed `j`, create a hash map called `remainder_count`.
4. Populate `remainder_count` using all elements after index `j`, meaning all possible `k > j`.
5. Now iterate through all indices `i` where `i < j`.
6. Compute the current remainder:

$$current = (nums[i] + nums[j]) \bmod d$$
7. Compute the remainder needed from the third element:

$$needed = (d - current) \bmod d$$
8. Add `remainder_count[needed]` to the answer, because every element with that remainder forms a valid triplet.
9. Continue until all pairs `(i, j)` are processed.
10. Return the final answer.

### Why it works

For every pair `(i, j)`, the algorithm computes exactly which remainder is needed from `nums[k]` to make the total divisible by `d`.

The hash map stores how many valid future elements exist with that remainder. Since every triplet satisfies exactly one required remainder condition, every valid triplet is counted once and only once.

Because we always maintain `i < j < k`, the ordering requirement is automatically enforced.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Solution:
    def divisibleTripletCount(self, nums: List[int], d: int) -> int:
        n = len(nums)
        answer = 0

        for j in range(1, n - 1):
            remainder_count = defaultdict(int)

            for k in range(j + 1, n):
                remainder = nums[k] % d
                remainder_count[remainder] += 1

            for i in range(j):
                current = (nums[i] + nums[j]) % d
                needed = (d - current) % d

                answer += remainder_count[needed]

        return answer
```

The implementation follows the algorithm directly.

The outer loop fixes the middle index `j`. For each `j`, we first collect all possible third elements `k` into a frequency map keyed by remainder modulo `d`.

Then we iterate over all valid `i` positions before `j`. For every pair `(i, j)`, we compute the remainder required from `nums[k]`.

Instead of searching manually, we simply look up how many future numbers already have that remainder.

The `defaultdict(int)` simplifies frequency counting because missing keys automatically default to zero.

## Go Solution

```go
func divisibleTripletCount(nums []int, d int) int {
	n := len(nums)
	answer := 0

	for j := 1; j < n-1; j++ {
		remainderCount := make(map[int]int)

		for k := j + 1; k < n; k++ {
			remainder := nums[k] % d
			remainderCount[remainder]++
		}

		for i := 0; i < j; i++ {
			current := (nums[i] + nums[j]) % d
			needed := (d - current) % d

			answer += remainderCount[needed]
		}
	}

	return answer
}
```

The Go implementation mirrors the Python logic closely.

Go uses `map[int]int` instead of `defaultdict`. Missing keys automatically return the zero value for integers, so frequency lookup remains straightforward.

Integer overflow is not a concern here because the maximum number of triplets is:

$$\binom{1000}{3} = 166,167,000$$

which safely fits inside a 32-bit integer.

Slices are used naturally since Go arrays are rarely used directly for dynamic input problems.

We process each middle index `j`.

Next:

| i | nums[i] + nums[j] | current % 5 | needed | matches |

| 0 | 3 + 4 = 7 | 2 | 3 | 1 |
| 1 | 3 + 4 = 7 | 2 | 3 | 1 |

Next:

| j | Possible k values | Remainder Map |
| --- | --- | --- |
| 3 | 8 | {3:1} |

Check all `i`.

| i | nums[i] + nums[j] | current % 5 | needed | matches |
| --- | --- | --- | --- | --- |
| 0 | 3 + 7 = 10 | 0 | 0 | 0 |
| 1 | 3 + 7 = 10 | 0 | 0 | 0 |
| 2 | 4 + 7 = 11 | 1 | 4 | 0 |

Final answer is `3`.

### Example 2

```
Input: nums = [3,3,3,3], d = 3
```

Every number has remainder `0`.

Every triplet sum becomes:

$$3 + 3 + 3 = 9$$

and:

$$9 \bmod 3 = 0$$

Total triplets:

$$\binom{4}{3} = 4$$

Answer is `4`.

### Example 3

```
Input: nums = [3,3,3,3], d = 6
```

Every triplet sum is still `9`.

But:

$$9 \bmod 6 = 3$$

No triplet is divisible by `6`.

Answer is `0`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n²) | Two nested loops process pairs efficiently |
| Space | O(n) | Hash map stores remainder frequencies |

The algorithm avoids the cubic enumeration of triplets by converting the third-element search into a constant-time hash lookup.

For each middle index `j`, we process all future indices once and all previous indices once, leading to quadratic complexity overall.

The extra space comes from the remainder frequency map.

## Edge Cases

### Arrays With Fewer Than Three Elements

If the array length is less than `3`, no triplet can exist. This can easily cause index errors in careless implementations.

The algorithm handles this naturally because the outer loop:

```
for j in range(1, n - 1):
```

does not execute when `n < 3`.

### All Triplets Valid

When every number has remainder `0` modulo `d`, every triplet sum is divisible by `d`.

This case is important because the algorithm must count all index combinations correctly without missing duplicates or overcounting.

Since the hash map stores frequencies rather than unique values, every valid index combination is counted separately.

### Large Values in nums

The array elements can be as large as `10^9`. Summing multiple values directly could raise overflow concerns in some languages.

The implementation avoids issues by reducing values modulo `d` whenever necessary. In Go, the result still fits safely within integer limits because only remainders are used during divisibility checks.
