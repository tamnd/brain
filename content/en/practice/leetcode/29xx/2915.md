## Solution

## Problem Understanding

The problem gives us an array `nums` and an integer `target`. We must find a subsequence of `nums` whose elements add up exactly to `target`, and among all such subsequences, we want the one with the maximum possible length.

A subsequence preserves the original ordering of elements, but we are allowed to skip elements. For example, from `[1,2,3,4,5]`, the subsequence `[1,3,5]` is valid because the relative order is maintained.

The important detail is that we are not trying to minimize or maximize the sum. The sum must equal exactly `target`. Among all subsequences that satisfy this condition, we choose the one with the greatest number of elements.

The input constraints are important:

- `nums.length <= 1000`
- `target <= 1000`
- `nums[i] <= 1000`

These limits strongly suggest that a dynamic programming solution based on the target sum is feasible. Since the target is at most `1000`, we can maintain DP states for sums from `0` to `target` efficiently.

A naive brute-force approach would consider every subsequence. Since an array of length `n` has `2^n` subsequences, that approach becomes impossible for `n = 1000`.

There are several edge cases to keep in mind:

- It may be impossible to form `target`, in which case we return `-1`.
- Multiple subsequences may produce the same sum, so we must keep the one with the largest length.
- Since all numbers are positive, once a sum exceeds `target`, it can never become useful again.
- A subsequence can contain repeated values if those values appear multiple times in the array.

## Approaches

### Brute Force Approach

The brute-force solution is to generate every possible subsequence and compute its sum.

For each subsequence:

1. Compute the total sum.
2. If the sum equals `target`, record its length.
3. Return the maximum valid length found.

This approach is correct because it exhaustively checks every possible subsequence, so it cannot miss the optimal answer.

However, the time complexity is exponential. With `n` elements, there are `2^n` subsequences. For `n = 1000`, this is completely infeasible.

### Optimal Dynamic Programming Approach

The key observation is that this problem resembles the classic subset sum dynamic programming problem.

Normally, subset sum DP asks whether a sum is achievable. Here, instead of storing a boolean, we store the maximum subsequence length that achieves each sum.

Define:

- `dp[s] = maximum length of a subsequence whose sum is s`

If a sum cannot be formed, we store `-1`.

We initialize:

- `dp[0] = 0`

A sum of `0` can always be formed using the empty subsequence.

For each number `num` in `nums`, we update the DP array in reverse order:

- For every sum `s` from `target` down to `num`

- If `dp[s - num]` is reachable
- Then we can form `s` using one additional element
- Update:

- `dp[s] = max(dp[s], dp[s - num] + 1)`

The reverse iteration is crucial because each number can only be used once. Iterating backward prevents reusing the same element multiple times in a single iteration.

At the end:

- If `dp[target] == -1`, return `-1`
- Otherwise return `dp[target]`

## Comparison of Approaches

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(n) | Generates every subsequence |
| Optimal Dynamic Programming | O(n × target) | O(target) | Uses subset-sum style DP storing maximum lengths |

## Algorithm Walkthrough

1. Create a DP array of size `target + 1`.

Each index represents a possible sum. The value stored at that index is the maximum subsequence length that can produce the sum.
2. Initialize every entry to `-1`.

A value of `-1` means the sum is currently impossible to form.
3. Set `dp[0] = 0`.

The empty subsequence has sum `0` and length `0`.
4. Process each number in `nums`.

For every number `num`, attempt to extend previously reachable sums.
5. Iterate sums backward from `target` down to `num`.

Backward traversal ensures each array element is used at most once. If we iterated forward, the same number could accidentally be reused multiple times during the same iteration.
6. Check whether `dp[current_sum - num]` is reachable.

If it is not `-1`, then we already know how to build the smaller sum.
7. Update the current sum.

If we can build `current_sum - num`, then adding `num` creates `current_sum`.

Update:

```
dp[current_sum] =
    max(dp[current_sum],
        dp[current_sum - num] + 1)
```
8. Continue until all numbers are processed.
9. Return the result.

If `dp[target]` is still `-1`, no valid subsequence exists. Otherwise, return the stored maximum length.

### Why it works

The DP invariant is:

- After processing the first `i` elements, `dp[s]` stores the maximum length of any subsequence from those elements whose sum equals `s`.

Each number gives us two choices:

- Exclude it
- Include it

The DP transition explores both possibilities implicitly. Since we always keep the maximum length for every reachable sum, the final value at `dp[target]` is guaranteed to be the length of the longest valid subsequence.

## Python Solution

```python
from typing import List

class Solution:
    def lengthOfLongestSubsequence(self, nums: List[int], target: int) -> int:
        dp = [-1] * (target + 1)
        dp[0] = 0

        for num in nums:
            for current_sum in range(target, num - 1, -1):
                if dp[current_sum - num] != -1:
                    dp[current_sum] = max(
                        dp[current_sum],
                        dp[current_sum - num] + 1
                    )

        return dp[target]
```

The implementation begins by creating the DP array. Every value is initialized to `-1` because initially no positive sum is reachable.

We then set `dp[0] = 0`, representing the empty subsequence.

For every number in the array, we iterate backward through possible sums. The backward iteration is the standard technique used in 0/1 knapsack style problems to prevent using the same element multiple times.

Whenever `current_sum - num` is reachable, we know we can extend that subsequence by adding `num`. We therefore update the best known length for `current_sum`.

Finally, the algorithm returns `dp[target]`. If no subsequence can form the target, the value remains `-1`.

## Go Solution

```go
func lengthOfLongestSubsequence(nums []int, target int) int {
	dp := make([]int, target+1)

	for i := 0; i <= target; i++ {
		dp[i] = -1
	}

	dp[0] = 0

	for _, num := range nums {
		for currentSum := target; currentSum >= num; currentSum-- {
			if dp[currentSum-num] != -1 {
				candidate := dp[currentSum-num] + 1

				if candidate > dp[currentSum] {
					dp[currentSum] = candidate
				}
			}
		}
	}

	return dp[target]
}
```

The Go implementation follows the same logic as the Python version.

Since Go does not provide automatic array initialization to `-1`, we explicitly fill the DP slice with `-1`.

The backward iteration is implemented using a descending `for` loop. Integer overflow is not a concern because all values remain very small under the problem constraints.

| Space | O(target) | Only one DP array of size `target + 1` is stored |

The algorithm processes every element exactly once, and for each element it may scan all sums from `target` down to the element value. Since both `n` and `target` are at most `1000`, this solution is easily fast enough.

The space complexity is efficient because we use a one-dimensional DP array rather than a full two-dimensional table.

## Edge Cases

One important edge case is when no subsequence can form the target. A naive implementation might accidentally return `0` or another incorrect value. This implementation avoids that issue by initializing unreachable sums to `-1`. If the target remains unreachable after processing all numbers, the algorithm correctly returns `-1`.

Another important case involves repeated values. For example, `[1,1,1,1]` with target `2` has multiple valid subsequences. The algorithm must distinguish between different occurrences of the same value. Since each array position is processed independently and backward iteration prevents reuse within the same iteration, duplicates are handled correctly.

A subtle edge case occurs when the longest subsequence is not the first valid subsequence discovered. For example, in `[4,1,3,2,1,5]` with target `7`, the subsequence `[4,3]` has length `2`, but `[1,3,2,1]` has length `4`. The DP state stores the maximum length for every sum, ensuring better solutions replace shorter ones.

Another important implementation detail is preventing reuse of the same element multiple times. If the DP loop iterated forward instead of backward, a number could contribute to multiple updates during the same iteration, effectively allowing unlimited reuse. The reverse traversal guarantees each number is used at most once.
