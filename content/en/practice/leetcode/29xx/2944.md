# LeetCode 2944 - Minimum Number of Coins for Fruits

## Problem Understanding

You are given an array `prices`, where `prices[i]` represents the number of coins required to buy the `(i + 1)`th fruit.

The important part of the problem is the reward system. When you buy fruit `i + 1`, you unlock a promotion that allows you to take any number of the next `i` fruits for free. Since indexing is 0-based, buying fruit at index `i` lets you freely acquire fruits in the range:

$$[i + 1, \, i + i]$$

Another subtle but extremely important detail is that even if a fruit can already be taken for free, you are still allowed to buy it anyway. Doing so may be beneficial because purchasing that fruit activates its own reward, potentially covering many additional fruits later.

The goal is to determine the minimum total number of coins needed to obtain all fruits.

The constraints are relatively small:

- `1 <= prices.length <= 1000`
- `1 <= prices[i] <= 10^5`

Since `n` is at most `1000`, an `O(n^2)` dynamic programming solution is completely acceptable. However, exponential brute force would still be far too slow.

Several edge cases are important:

- A single fruit array, where we must buy the only fruit.
- Cases where buying a cheap fruit later is more valuable than taking it for free.
- Cases where one purchase covers almost the remainder of the array.
- Situations where overlapping reward ranges create multiple possible optimal paths.

The main challenge is deciding which fruits should actually be purchased in order to maximize future free coverage while minimizing total cost.

## Approaches

### Brute Force Approach

A brute force solution would recursively explore every possible decision:

- Either take the current fruit for free if it is covered.
- Or buy it and activate its reward.

At every fruit, multiple choices may exist, leading to a large branching factor. Since each fruit may independently be purchased or skipped, the number of possible states grows exponentially.

Although brute force eventually finds the optimal answer because it explores every valid combination, it becomes computationally infeasible even for moderate input sizes.

### Key Insight

The crucial observation is that the problem naturally forms a dynamic programming structure.

Suppose we define:

$$dp[i]$$

as the minimum cost needed to acquire all fruits starting from index `i`.

If we decide to buy fruit `i`, then:

- We pay `prices[i]`
- Buying fruit `i` allows us to cover the next `i + 1` fruits total, meaning the next uncovered position becomes:

$$2i + 2$$

So after buying fruit `i`, we may continue from any index between:

$$i + 1 \text{ and } 2i + 2$$

depending on how many fruits we choose to take for free.

This gives the transition:

$$dp[i] = prices[i] + \min(dp[j])$$

where:

$$j \in [i+1, \, 2i+2]$$

A naive implementation of this recurrence would require scanning a range for every state, resulting in `O(n^2)` complexity.

We can optimize this using a monotonic queue to maintain the minimum DP value over sliding ranges efficiently.

## Algorithm Walkthrough

### Optimal DP With Monotonic Queue

We process fruits from right to left.

We define:

$$dp[i]$$

as the minimum cost to obtain all fruits starting from index `i`.

The transition is:

$$dp[i] = prices[i] + \min(dp[j])$$

where:

$$j \in [i+1, 2i+2]$$

If `2i + 2 >= n`, then buying fruit `i` can cover all remaining fruits, so:

$$dp[i] = prices[i]$$

To efficiently compute the minimum over sliding ranges, we maintain a monotonic deque.

### Step-by-step Process

1. Create a DP array of size `n + 1`.

We define:

$$dp[n] = 0$$

because no cost is needed once all fruits are acquired.
2. Process indices from right to left.

This ensures all future DP states are already computed when evaluating `dp[i]`.
3. Maintain a monotonic deque.

The deque stores indices whose DP values are increasing. The front of the deque always contains the minimum DP value for the current valid range.
4. Remove indices outside the valid transition range.

For position `i`, valid next states are:

$$[i+1, 2i+2]$$

Any deque entries larger than `2i + 2` are removed.
5. Compute the current DP value.

If the deque is empty, then buying fruit `i` covers all remaining fruits:

$$dp[i] = prices[i]$$

Otherwise:

$$dp[i] = prices[i] + dp[deque[0]]$$
6. Insert the current index into the deque.

Before insertion, remove all indices from the back whose DP values are greater than or equal to `dp[i]`.

This preserves monotonic increasing order.
7. Continue until index `0`.

The answer is:

$$dp[0]$$

### Why it works

The DP recurrence correctly models every valid purchasing strategy. Buying fruit `i` determines the furthest reachable free range, and the remaining uncovered fruits form a smaller subproblem.

The monotonic deque guarantees that the minimum DP value over the required range is always available in constant time. Since each index enters and leaves the deque at most once, the total complexity remains linear.

## Python Solution

```python
from collections import deque
from typing import List

class Solution:
    def minimumCoins(self, prices: List[int]) -> int:
        n = len(prices)

        dp = [0] * (n + 1)

        dq = deque([n])

        for i in range(n - 1, -1, -1):
            right = min(n, 2 * i + 2)

            while dq and dq[0] > right:
                dq.popleft()

            dp[i] = prices[i] + dp[dq[0]]

            while dq and dp[dq[-1]] >= dp[i]:
                dq.pop()

            dq.append(i)

        return dp[0]
```

The implementation starts by initializing the DP array and setting `dp[n] = 0`, representing the base case where no fruits remain.

The deque is initialized with index `n`, since this represents the empty suffix state.

The algorithm processes fruits from right to left. For each index `i`, the furthest reachable uncovered position after buying fruit `i` is:

$$2i + 2$$

Any deque entries beyond this limit are removed because they are not valid transitions.

The front of the deque always stores the index with the minimum DP value in the valid range. Therefore:

```python
dp[i] = prices[i] + dp[dq[0]]
```

correctly computes the optimal cost.

After computing `dp[i]`, the deque is updated to maintain increasing DP values. Any worse candidates at the back are removed before inserting the current index.

Finally, `dp[0]` contains the minimum total cost to obtain all fruits.

## Go Solution

```go
func minimumCoins(prices []int) int {
	n := len(prices)

	dp := make([]int, n+1)

	deque := []int{n}

	for i := n - 1; i >= 0; i-- {
		right := 2*i + 2
		if right > n {
			right = n
		}

		for len(deque) > 0 && deque[0] > right {
			deque = deque[1:]
		}

		dp[i] = prices[i] + dp[deque[0]]

		for len(deque) > 0 && dp[deque[len(deque)-1]] >= dp[i] {
			deque = deque[:len(deque)-1]
		}

		deque = append(deque, i)
	}

	return dp[0]
}
```

The Go implementation follows the same logic as the Python version.

Slices are used to simulate the deque. Front removals are handled using:

```go
deque = deque[1:]
```

and back removals use slice truncation.

Integer overflow is not an issue because the maximum possible answer is well within Go's standard integer range.

## Edge Cases

One important edge case is a single fruit array. Since there are no future fruits to cover, the only possible action is to buy the fruit directly. The implementation handles this naturally because `dp[n] = 0`, so:

$$dp[0] = prices[0]$$

Another tricky case occurs when a fruit can already be taken for free, but buying it is still beneficial. This is the core subtlety of the problem. A naive greedy solution might always prefer free acquisition, but that can prevent unlocking a much larger reward range later. The DP formulation explicitly considers buying every fruit as a possible decision, ensuring the optimal strategy is always found.

A third important edge case is when buying a fruit covers all remaining fruits. In this scenario, the transition range reaches beyond the array boundary. The implementation safely handles this using:

```
right = min(n, 2 * i + 2)
```

This ensures all array accesses remain valid while still correctly modeling complete coverage of the remaining suffix.
