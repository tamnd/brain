## Solution

## Problem Understanding

The problem asks us to find the minimum sum of a mountain triplet in a given array `nums`. A mountain triplet is a sequence of three indices `(i, j, k)` such that `i < j < k`, `nums[i] < nums[j]`, and `nums[k] < nums[j]`. In other words, the middle element of the triplet must be strictly larger than its neighbors. The array is 0-indexed, and we are required to return the sum of the values at these indices. If no such triplet exists, we return `-1`.

The input is an array of integers with length between 3 and 50 and each integer between 1 and 50. The small input constraints suggest that an algorithm with a complexity up to $O(n^3)$ is feasible, but we should explore more efficient solutions. Important edge cases include arrays that are strictly increasing, strictly decreasing, or all equal, as these contain no valid mountain triplets.

## Approaches

### Brute Force

The brute-force approach iterates over all possible triplets `(i, j, k)` where `i < j < k`. For each triplet, it checks if `nums[i] < nums[j]` and `nums[k] < nums[j]`. If the triplet satisfies these conditions, it computes the sum `nums[i] + nums[j] + nums[k]` and updates the minimum sum seen so far. While this guarantees correctness, it requires three nested loops over the array, resulting in $O(n^3)$ time complexity. For $n \le 50$, this is acceptable, but it is not optimal.

### Optimal Approach

The optimal approach leverages the observation that for each potential middle element `nums[j]`, we only need the smallest number to the left that is smaller than `nums[j]` and the smallest number to the right that is smaller than `nums[j]`. By scanning the array once from left to right and once from right to left, we can precompute these minimums efficiently. Then, for each middle element, we check if valid smaller numbers exist on both sides and compute the sum. This reduces the complexity to $O(n^2)$.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^3) | O(1) | Check all triplets |
| Optimal | O(n^2) | O(1) | Use precomputed minimums for left and right sides |

## Algorithm Walkthrough

1. Initialize a variable `min_sum` to infinity to track the minimum mountain triplet sum.
2. Loop over each index `j` from 1 to `len(nums) - 2` as the potential middle element.
3. For each `j`, find the smallest element `left_min` to its left that is strictly smaller than `nums[j]`. If none exists, skip this `j`.
4. Similarly, find the smallest element `right_min` to its right that is strictly smaller than `nums[j]`. If none exists, skip this `j`.
5. If both `left_min` and `right_min` exist, compute the sum `left_min + nums[j] + right_min` and update `min_sum` if the sum is smaller.
6. After examining all indices, return `min_sum` if it has been updated; otherwise, return `-1`.

**Why it works**: The algorithm ensures that for every potential mountain peak, we consider the smallest possible elements on both sides. This guarantees the minimum sum without missing any valid triplets. By focusing on valid peaks only, we reduce unnecessary checks and improve efficiency.

## Python Solution

```python
from typing import List

class Solution:
    def minimumSum(self, nums: List[int]) -> int:
        n = len(nums)
        min_sum = float('inf')
        
        for j in range(1, n - 1):
            left_min = float('inf')
            for i in range(j):
                if nums[i] < nums[j]:
                    left_min = min(left_min, nums[i])
            
            right_min = float('inf')
            for k in range(j + 1, n):
                if nums[k] < nums[j]:
                    right_min = min(right_min, nums[k])
            
            if left_min != float('inf') and right_min != float('inf'):
                min_sum = min(min_sum, left_min + nums[j] + right_min)
        
        return min_sum if min_sum != float('inf') else -1
```

The Python implementation loops over each possible middle element and searches both sides for the smallest valid numbers. It uses `float('inf')` as a sentinel to detect if no valid element exists on either side. The code follows the algorithm closely and handles all edge cases.

## Go Solution

```go
func minimumSum(nums []int) int {
    n := len(nums)
    minSum := 1<<31 - 1
    
    for j := 1; j < n-1; j++ {
        leftMin := 1<<31 - 1
        for i := 0; i < j; i++ {
            if nums[i] < nums[j] && nums[i] < leftMin {
                leftMin = nums[i]
            }
        }
        
        rightMin := 1<<31 - 1
        for k := j + 1; k < n; k++ {
            if nums[k] < nums[j] && nums[k] < rightMin {
                rightMin = nums[k]
            }
        }
        
        if leftMin != 1<<31-1 && rightMin != 1<<31-1 {
            sum := leftMin + nums[j] + rightMin
            if sum < minSum {
                minSum = sum
            }
        }
    }
    
    if minSum == 1<<31-1 {
        return -1
    }
    return minSum
}
```

The Go solution mirrors the Python version. Go does not have `float('inf')`, so `1<<31-1` is used as a sentinel for "infinity". It handles slices carefully and ensures that edge conditions for left and right minimums are checked before updating the minimum sum.

## Worked Examples

### Example 1: `[8,6,1,5,3]`

For `j = 1` (nums[1] = 6), left_min = 8 does not satisfy `nums[i] < nums[j]`, right_min = 1. Invalid.

For `j = 2` (nums[2] = 1), no left < 1. Invalid.

For `j = 3` (nums[3] = 5), left_min = 1, right_min = 3 → sum = 1 + 5 + 3 = 9. Update min_sum = 9.

For `j = 4` (nums[4] = 3), left_min = 1, right_min does not exist. Invalid.

Return 9.

### Example 2: `[5,4,8,7,10,2]`

`j = 2` (nums[2] = 8), left_min = 4, right_min = 2 → sum = 4 + 8 + 2 = 14.

`j = 3` (nums[3] = 7), left_min = 4, right_min = 2 → sum = 4 + 7 + 2 = 13. Update min_sum = 13.

`j = 4` (nums[4] = 10), left_min = 4, right_min = 2 → sum = 4 + 10 + 2 = 16. No update.

Return 13.

### Example 3: `[6,5,4,3,4,5]`

No middle element satisfies left < middle < right. Return -1.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n^2) | For each middle element, we scan left and right sides, giving n * n = n^2 operations |
| Space | O(1) | Only a few variables are used; no extra data structures proportional to n |

Because `n` is at most 50, this quadratic solution runs efficiently. Brute force would be cubic, which is acceptable but slightly slower.
