## Solution

## Problem Understanding

We are given an integer array `nums` and a target value `k`. We may increment any element any number of times, where each increment operation increases a chosen element by exactly `1`.

The goal is to make the array "beautiful".

An array is considered beautiful if every subarray of length at least `3` contains at least one element greater than or equal to `k`.

This condition is extremely important to interpret correctly. It does not require every element to become at least `k`. Instead, every window of size `3` or larger must contain at least one "good" element, meaning an element whose value is at least `k`.

Another way to think about the condition is this:

- There cannot exist three consecutive elements that are all strictly less than `k`.

Why? Because if there are three consecutive elements below `k`, then that length-3 subarray violates the requirement immediately.

This observation simplifies the problem dramatically. We only need to ensure that every consecutive block of length `3` contains at least one element that reaches `k`.

For any index `i`, the cost to make `nums[i]` become at least `k` is:

```
max(0, k - nums[i])
```

We want to choose some indices to upgrade so that every window of length `3` is covered, while minimizing the total increment cost.

The constraints are large:

- `n` can be up to `10^5`
- `nums[i]` and `k` can be up to `10^9`

These constraints immediately rule out exponential or quadratic solutions. We need a linear or near-linear algorithm.

Several edge cases are important:

- Arrays that are already beautiful should return `0`
- Arrays where every value is far below `k`
- Very small arrays, especially `n = 3`
- Cases where choosing one upgraded element can cover multiple windows
- Cases where upgrading a larger existing value is cheaper than upgrading smaller values

Because the answer can become large, especially when `k` is large, the Go solution must use `int64`.

## Approaches

### Brute Force Approach

A brute force solution would try every possible subset of indices to upgrade.

For each subset:

1. Compute the cost required to raise those selected indices to at least `k`
2. Check whether every subarray of length at least `3` contains at least one upgraded element
3. Track the minimum valid cost

This works because it explores all possibilities, guaranteeing the optimal answer.

However, there are `2^n` possible subsets of indices, which is completely infeasible for `n = 10^5`.

Even optimizing the validation step would not help enough because the state space itself is exponential.

### Key Insight

The critical observation is that the condition only depends on windows of size `3`.

If every length-3 window contains at least one element greater than or equal to `k`, then every larger window automatically does too.

Therefore, the problem becomes:

> Choose positions to upgrade so that every consecutive group of three indices contains at least one upgraded index.

This creates a local dependency structure, which is ideal for dynamic programming.

We process the array from left to right while tracking how many consecutive positions have remained below `k`.

If we ever allow three consecutive unselected positions, the configuration becomes invalid.

At each position:

- Either upgrade the current element
- Or leave it unchanged

This leads naturally to a DP with only three states.

| --- | --- | --- | --- |
| Brute Force | O(2^n * n) | O(1) | Tries every subset of upgraded indices |
| Optimal Dynamic Programming | O(n) | O(1) | Tracks consecutive positions below `k` |

## Algorithm Walkthrough

### Dynamic Programming State

We define:

```
dp[j]
```

where `j` represents how many consecutive positions at the end are still below `k`.

Possible values:

- `0` = current position is upgraded to at least `k`
- `1` = one consecutive position below `k`
- `2` = two consecutive positions below `k`

We never allow `3`.

### Step-by-Step Process

1. Initialize the DP array.

At the beginning:

```
dp[0] = 0
dp[1] = inf
dp[2] = inf
```

No positions have been processed yet, and the cost is zero.
2. Process each element from left to right.

For the current element `nums[i]`, compute:

```
cost = max(0, k - nums[i])
```

This is the number of increments needed to make this element reach `k`.
3. Option 1, upgrade the current element.

If we upgrade it, the streak of consecutive "bad" elements resets to `0`.

We can transition from any previous valid state:

```
new_dp[0] = min(dp[0], dp[1], dp[2]) + cost
```
4. Option 2, do not upgrade the current element.

Then the consecutive-bad streak increases by one.

Transitions:

```
new_dp[1] = dp[0]
new_dp[2] = dp[1]
```

We do not allow transitions from `dp[2]` because that would create three consecutive bad positions.
5. Replace the old DP with the new DP.
6. After processing all elements, return:

```
min(dp[0], dp[1], dp[2])
```

Any remaining valid state is acceptable.

### Why it works

The DP maintains a strict invariant:

> `dp[j]` always stores the minimum cost after processing the current prefix while ending with exactly `j` consecutive elements below `k`.

Because every invalid configuration necessarily contains three consecutive bad elements, forbidding transitions into state `3` guarantees validity.

The DP explores all valid possibilities while always keeping the minimum achievable cost for each state. Therefore, the final minimum among all valid ending states is the optimal answer.

## Python Solution

```python
from typing import List

class Solution:
    def minIncrementOperations(self, nums: List[int], k: int) -> int:
        INF = float('inf')

        dp0 = 0
        dp1 = INF
        dp2 = INF

        for num in nums:
            cost = max(0, k - num)

            new_dp0 = min(dp0, dp1, dp2) + cost
            new_dp1 = dp0
            new_dp2 = dp1

            dp0, dp1, dp2 = new_dp0, new_dp1, new_dp2

        return min(dp0, dp1, dp2)
```

The implementation uses three variables instead of an array because only the previous state is needed at each step.

`dp0` represents the minimum cost when the current position has been upgraded to at least `k`.

`dp1` represents the minimum cost when there is one consecutive position below `k`.

`dp2` represents the minimum cost when there are two consecutive positions below `k`.

For each element:

- We compute the upgrade cost
- We calculate the new DP states
- We shift the states forward

The transition:

```python
new_dp0 = min(dp0, dp1, dp2) + cost
```

means we may upgrade the current element regardless of the previous streak length.

The transitions:

```python
new_dp1 = dp0
new_dp2 = dp1
```

mean we skip upgrading the current element, increasing the consecutive-bad streak by one.

No transition into a streak length of `3` is allowed, which enforces the beauty condition.

## Go Solution

```go
package main

func minIncrementOperations(nums []int, k int) int64 {
	const INF int64 = 1 << 60

	dp0 := int64(0)
	dp1 := INF
	dp2 := INF

	for _, num := range nums {
		cost := int64(0)
		if num < k {
			cost = int64(k - num)
		}

		newDp0 := min3(dp0, dp1, dp2) + cost
		newDp1 := dp0
		newDp2 := dp1

		dp0, dp1, dp2 = newDp0, newDp1, newDp2
	}

	return min3(dp0, dp1, dp2)
}

func min3(a, b, c int64) int64 {
	if a > b {
		a = b
	}
	if a > c {
		a = c
	}
	return a
}
```

The Go implementation follows the same DP logic as the Python version.

The main difference is the use of `int64` to avoid overflow. Since `k` and array length can both be large, the total increment cost may exceed the range of a standard 32-bit integer.

Go also requires an explicit helper function for taking the minimum of three values.

## Edge Cases

One important edge case is the minimum possible array size, where `n = 3`. In this situation, there is exactly one relevant subarray, the entire array itself. The algorithm still works naturally because allowing three consecutive positions below `k` becomes immediately invalid. The DP transitions correctly force at least one upgraded element.

Another critical edge case occurs when the array is already beautiful. For example:

```
nums = [5,1,1,5]
k = 5
```

The implementation handles this correctly because the upgrade cost for already-valid elements is zero. The DP can choose those positions without paying any additional cost.

A particularly tricky scenario is when one upgraded position can satisfy many overlapping windows. For example:

```
[0,0,5,0,0]
```

The middle element already satisfies every length-3 window. A greedy approach that upgrades locally without considering overlap could overpay. The DP avoids this by globally minimizing cost across all valid states.

Another subtle case is when there are multiple cheap upgrade candidates close together. The algorithm must choose the globally optimal placement rather than simply upgrading the first invalid window it encounters. The DP naturally evaluates all valid possibilities and retains only the minimum cost for each state configuration.
