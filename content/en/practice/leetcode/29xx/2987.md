# LeetCode 2987 - Find Expensive Cities

## Problem Understanding

This problem asks us to identify which cities have an average home listing price that is strictly greater than the overall national average listing price.

The input comes from a database table named `Listings`. Each row represents a single property listing and contains three fields:

| Column | Meaning |
| --- | --- |
| `listing_id` | Unique identifier for a property listing |
| `city` | Name of the city where the property is located |
| `price` | Price of the property |

The national average is computed using every listing in the table, regardless of city. After computing that overall average, we must compute the average listing price for each individual city. Any city whose average exceeds the national average should appear in the output.

The result should contain only one column, `city`, and the output must be sorted alphabetically in ascending order.

A very important detail is that the comparison is strictly greater than the national average. Cities whose average is exactly equal to the national average must not be included.

The problem is categorized as a database problem, so the intended solution is SQL. Conceptually, though, this is a grouping and aggregation problem involving averages.

The constraints are not explicitly listed, but database interview problems like this are generally designed for efficient aggregation queries. Since averages can be computed with aggregate functions, we should avoid repeatedly scanning the table unnecessarily.

Some important edge cases are worth considering upfront.

If a city contains only one listing, that single price is still considered the city's average. The query must handle this naturally.

If every city has the same average as the national average, the result should be empty.

If multiple cities exceed the national average, all must be returned in sorted order.

Because averages may produce decimal values, the database engine must correctly handle floating point or decimal arithmetic during comparison.

## Approaches

### Brute Force Approach

A brute-force strategy would recompute averages repeatedly.

For every distinct city, we could:

1. Scan the entire table to compute that city's average.
2. Scan the entire table again to compute the national average.
3. Compare the two averages.
4. Include the city if its average is larger.

This approach is correct because it explicitly computes the required values for each city. However, it is inefficient because the national average is recomputed over and over again, even though it never changes.

If there are `n` listings and `k` cities, this approach may scan the table roughly `k + 1` times.

### Optimal Approach

The key insight is that the national average only needs to be computed once.

We can first calculate the national average price using an aggregate query over the entire table. Then we group listings by city and compute each city's average price. Finally, we filter cities whose average exceeds the previously computed national average.

SQL aggregate functions are specifically designed for this type of problem. Using `GROUP BY` and `HAVING` allows us to express the logic cleanly and efficiently.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(k * n) | O(1) | Recomputes averages repeatedly |
| Optimal | O(n) | O(k) | Computes national average once, then groups by city |

## Algorithm Walkthrough

1. Compute the national average listing price across the entire `Listings` table using the `AVG(price)` aggregate function.
2. Group all rows by `city` using `GROUP BY city`. This creates one group per city.
3. For each city group, compute the average listing price using `AVG(price)`.
4. Use a `HAVING` clause to keep only cities whose average price is strictly greater than the national average computed earlier.
5. Sort the remaining cities alphabetically using `ORDER BY city ASC`.

### Why it works

The algorithm works because every listing belongs to exactly one city, and the SQL aggregation functions correctly compute averages over those groups.

The national average is computed across all rows exactly once. The city averages are computed independently for each grouped subset. Comparing the grouped averages against the overall average directly matches the problem definition.

## Python Solution

Even though this is a database problem, LeetCode database questions are solved using SQL. The "Python solution" section here explains the SQL query in a Python-style presentation format.

```python
SELECT city
FROM Listings
GROUP BY city
HAVING AVG(price) > (
    SELECT AVG(price)
    FROM Listings
)
ORDER BY city ASC;
```

The query has three major logical parts.

The subquery computes the national average price:

```python
SELECT AVG(price)
FROM Listings
```

This value becomes the threshold for comparison.

Next, the outer query groups rows by city:

```python
GROUP BY city
```

This allows SQL to compute one average per city.

The `HAVING` clause filters grouped results:

```python
HAVING AVG(price) > (...)
```

Unlike `WHERE`, which filters individual rows, `HAVING` filters aggregated groups.

Finally, the output is sorted alphabetically:

```python
ORDER BY city ASC
```

## Edge Cases

One important edge case occurs when there is only a single city in the table. In that scenario, the city average and national average are identical because all listings belong to the same city. Since the condition requires the city average to be strictly greater than the national average, the result should be empty. The implementation handles this naturally through the `HAVING AVG(price) > (...)` comparison.

Another important edge case occurs when a city has only one listing. A naive implementation might incorrectly assume averages require multiple rows. SQL aggregation functions correctly treat a single value as a valid average, so the query works without any special handling.

A third edge case involves floating point comparisons. National averages often contain decimal values, while city averages may also produce decimals. Using SQL's built-in `AVG()` function ensures the database performs the comparison using proper numeric arithmetic instead of integer division, preventing truncation bugs.
