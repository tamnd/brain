**Topics:** Array, String, Dynamic Programming, Graph Theory, Trie, Shortest Path

## Solution

## Problem Understanding

This problem asks us to transform one string into another while minimizing the total conversion cost. At first glance, it looks similar to a normal string transformation problem, but the operation rules make it significantly more complex.

We are given two strings, `source` and `target`, both with the same length `n`. We are also given a list of allowed substring transformations. Each transformation says that we can convert one string into another string of the same length for some cost.

For example, if we have:

```
"abc" -> "def" with cost 5
```

then any substring `"abc"` inside the current string can be replaced with `"def"` by paying 5.

The important detail is that operations are restricted by index ranges. If one operation modifies indices `[a, b]`, then every other operation must either:

1. Use a completely disjoint interval, or
2. Use exactly the same interval `[a, b]`

This rule prevents partially overlapping operations.

The second rule is extremely important because it allows chaining transformations on the exact same substring. For example:

```
"abc" -> "xyz"
"xyz" -> "pqr"
```

can both be applied to the same interval.

The goal is to determine the minimum total cost required to convert the entire `source` string into `target`. If it cannot be done, we return `-1`.

The constraints provide several important hints:

- `source.length <= 1000`, which means quadratic dynamic programming is possible.
- There are at most 100 transformation rules, which is small enough for graph algorithms such as Floyd Warshall.
- Transformation strings can have varying lengths.
- Multiple rules may exist for the same transformation pair, so we must keep the minimum cost.
- Transformations may need to be chained indirectly through intermediate strings.

A naive solution that explores all possible operation sequences would explode combinatorially. The key challenge is combining substring matching, shortest path preprocessing, and interval dynamic programming efficiently.

Some important edge cases include:

- Multiple transformations between the same strings with different costs.
- Indirect transformations that are cheaper than direct ones.
- Impossible conversions where no valid sequence exists.
- Overlapping intervals that are illegal.
- Positions where `source[i] == target[i]`, meaning no operation is needed.
- Transformations involving substrings longer than one character.

## Approaches

### Brute Force Approach

A brute force solution would recursively try every possible substring operation at every step.

At each recursive state, we would:

1. Pick every possible substring interval.
2. Try every valid transformation rule.
3. Apply the transformation.
4. Continue recursively until the string becomes `target`.

This eventually finds the optimal answer because it explores all legal operation sequences.

However, the number of states becomes enormous. Every substring could potentially be transformed multiple times, and chained transformations create even more branching. Since the string length can reach 1000, brute force is completely infeasible.

The main inefficiency is repeated recomputation of equivalent substring conversions.

### Key Insight

The crucial observation is that operations on different intervals are independent because overlapping intervals are forbidden.

That means we can process the string from left to right using dynamic programming.

Another important insight is that transformations form a weighted directed graph:

```
original[i] -> changed[i]
```

with edge weight `cost[i]`.

Since we may chain operations on the same interval, converting one substring into another becomes a shortest path problem over this graph.

For example:

```
"a" -> "b" cost 5
"b" -> "c" cost 2
```

implies:

```
"a" -> "c" cost 7
```

even if no direct rule exists.

We therefore:

1. Build a graph of all unique transformation strings.
2. Run Floyd Warshall to compute minimum conversion costs between every pair.
3. Use dynamic programming over string positions:

- Either keep matching characters unchanged.
- Or apply a substring transformation for some interval length.

This converts the problem into an interval DP combined with shortest path preprocessing.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Tries all operation sequences recursively |
| Optimal | O(m³ + n × L) | O(m² + n) | Floyd Warshall plus DP over string positions |

Here:

- `m` is the number of unique strings in transformations.
- `n` is the source length.
- `L` is the number of distinct substring lengths.

## Algorithm Walkthrough

### Step 1: Collect All Unique Transformation Strings

Every string appearing in `original` or `changed` becomes a node in our graph.

We assign each unique string an integer ID.

For example:

```
"a" -> 0
"b" -> 1
"c" -> 2
```

This allows us to represent the graph as a matrix.

### Step 2: Build the Cost Graph

We create a distance matrix:

```
dist[u][v]
```

which stores the minimum known cost to convert string `u` into string `v`.

Initially:

- `dist[i][i] = 0`
- All other values are infinity.

For every rule:

```
original[i] -> changed[i] with cost c
```

we update:

```
dist[u][v] = min(dist[u][v], c)
```

because multiple rules may exist for the same pair.

### Step 3: Run Floyd Warshall

Now we compute the shortest path between every pair of strings.

If:

```
A -> B cost 5
B -> C cost 2
```

then Floyd Warshall discovers:

```
A -> C cost 7
```

This preprocessing gives the cheapest possible cost between any two transformation strings.

### Step 4: Group Rules by Length

A transformation only works between substrings of equal length.

We therefore collect all distinct transformation lengths.

For example:

```
length 1
length 3
length 5
```

This allows efficient substring checks during DP.

### Step 5: Define Dynamic Programming

Let:

```
dp[i]
```

represent the minimum cost to convert:

```
source[i:]
```

into:

```
target[i:]
```

We initialize:

```
dp[n] = 0
```

because converting empty suffixes costs nothing.

### Step 6: Transition for Matching Characters

If:

```
source[i] == target[i]
```

then we can skip this position:

```
dp[i] = dp[i + 1]
```

because no operation is needed.

### Step 7: Try Every Transformation Length

For every allowed substring length `length`:

1. Extract:

```
s = source[i:i+length]
t = target[i:i+length]
```
2. Check whether both strings exist in the graph.
3. If a path exists between them:

```
conversion_cost = dist[id(s)][id(t)]
```
4. Update:

```
dp[i] = min(dp[i], conversion_cost + dp[i + length])
```

This works because intervals are processed disjointly from left to right.

### Step 8: Return the Final Answer

If:

```
dp[0]
```

is still infinity, conversion is impossible.

Otherwise return `dp[0]`.

### Why it works

The algorithm works because every valid operation sequence can be decomposed into non-overlapping intervals processed independently.

For any interval, the cheapest way to transform one substring into another is exactly the shortest path in the transformation graph. Floyd Warshall computes these optimal interval costs.

The dynamic programming then combines optimal interval solutions into an optimal whole-string solution. Since every DP state considers all valid interval lengths and transitions, no legal transformation sequence is missed.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Solution:
    def minimumCost(
        self,
        source: str,
        target: str,
        original: List[str],
        changed: List[str],
        cost: List[int]
    ) -> int:
        
        INF = 10**18
        
        # Collect all unique strings
        unique_strings = set(original + changed)
        
        string_to_id = {}
        idx = 0
        
        for s in unique_strings:
            string_to_id[s] = idx
            idx += 1
        
        m = len(string_to_id)
        
        # Initialize distance matrix
        dist = [[INF] * m for _ in range(m)]
        
        for i in range(m):
            dist[i][i] = 0
        
        # Build graph
        for o, c, w in zip(original, changed, cost):
            u = string_to_id[o]
            v = string_to_id[c]
            dist[u][v] = min(dist[u][v], w)
        
        # Floyd Warshall
        for k in range(m):
            for i in range(m):
                if dist[i][k] == INF:
                    continue
                
                for j in range(m):
                    if dist[k][j] == INF:
                        continue
                    
                    new_cost = dist[i][k] + dist[k][j]
                    
                    if new_cost < dist[i][j]:
                        dist[i][j] = new_cost
        
        # Store valid transformation lengths
        lengths = set()
        
        for s in unique_strings:
            lengths.add(len(s))
        
        n = len(source)
        
        # dp[i] = minimum cost to convert source[i:]
        dp = [INF] * (n + 1)
        dp[n] = 0
        
        for i in range(n - 1, -1, -1):
            
            # Case 1: characters already match
            if source[i] == target[i]:
                dp[i] = dp[i + 1]
            
            # Case 2: try substring transformations
            for length in lengths:
                if i + length > n:
                    continue
                
                s_sub = source[i:i + length]
                t_sub = target[i:i + length]
                
                if s_sub not in string_to_id:
                    continue
                
                if t_sub not in string_to_id:
                    continue
                
                u = string_to_id[s_sub]
                v = string_to_id[t_sub]
                
                if dist[u][v] == INF:
                    continue
                
                dp[i] = min(
                    dp[i],
                    dist[u][v] + dp[i + length]
                )
        
        return dp[0] if dp[0] != INF else -1
```

The implementation begins by assigning every unique transformation string an integer ID. This lets us represent transformations as graph edges inside a matrix.

The `dist` matrix stores the minimum known conversion cost between every pair of strings. Initially, each string can convert to itself with cost zero, while all other conversions are impossible.

Next, the algorithm processes all transformation rules. If multiple rules exist for the same pair, we keep only the cheapest direct cost.

Floyd Warshall then computes the cheapest indirect transformation paths. This step is critical because optimal conversions may require intermediate strings.

After preprocessing, the algorithm performs dynamic programming from right to left. Each state `dp[i]` represents the cheapest way to convert the suffix beginning at index `i`.

At every position, the algorithm first checks whether characters already match. If they do, we can advance without cost.

Then it tries every valid transformation length. If both substrings exist in the transformation graph and a conversion path exists between them, we update the DP state using the interval cost plus the best solution for the remaining suffix.

Finally, if `dp[0]` remains infinite, no valid conversion exists.

## Go Solution

```go
package main

import "math"

func minimumCost(
	source string,
	target string,
	original []string,
	changed []string,
	cost []int,
) int64 {

	const INF int64 = 1e18

	// Collect unique strings
	stringToID := make(map[string]int)

	addString := func(s string) {
		if _, exists := stringToID[s]; !exists {
			stringToID[s] = len(stringToID)
		}
	}

	for _, s := range original {
		addString(s)
	}

	for _, s := range changed {
		addString(s)
	}

	m := len(stringToID)

	// Initialize distance matrix
	dist := make([][]int64, m)

	for i := 0; i < m; i++ {
		dist[i] = make([]int64, m)

		for j := 0; j < m; j++ {
			if i == j {
				dist[i][j] = 0
			} else {
				dist[i][j] = INF
			}
		}
	}

	// Build graph
	for i := 0; i < len(cost); i++ {
		u := stringToID[original[i]]
		v := stringToID[changed[i]]

		w := int64(cost[i])

		if w < dist[u][v] {
			dist[u][v] = w
		}
	}

	// Floyd Warshall
	for k := 0; k < m; k++ {
		for i := 0; i < m; i++ {

			if dist[i][k] == INF {
				continue
			}

			for j := 0; j < m; j++ {

				if dist[k][j] == INF {
					continue
				}

				newCost := dist[i][k] + dist[k][j]

				if newCost < dist[i][j] {
					dist[i][j] = newCost
				}
			}
		}
	}

	// Collect valid lengths
	lengthSet := make(map[int]bool)

	for s := range stringToID {
		lengthSet[len(s)] = true
	}

	lengths := []int{}

	for length := range lengthSet {
		lengths = append(lengths, length)
	}

	n := len(source)

	dp := make([]int64, n+1)

	for i := 0; i <= n; i++ {
		dp[i] = INF
	}

	dp[n] = 0

	for i := n - 1; i >= 0; i-- {

		// Matching characters
		if source[i] == target[i] {
			dp[i] = dp[i+1]
		}

		// Try transformations
		for _, length := range lengths {

			if i+length > n {
				continue
			}

			sSub := source[i : i+length]
			tSub := target[i : i+length]

			u, ok1 := stringToID[sSub]
			v, ok2 := stringToID[tSub]

			if !ok1 || !ok2 {
				continue
			}

			if dist[u][v] == INF {
				continue
			}

			dp[i] = min(
				dp[i],
				dist[u][v]+dp[i+length],
			)
		}
	}

	if dp[0] == INF {
		return -1
	}

	return dp[0]
}

func min(a, b int64) int64 {
	return int64(math.Min(float64(a), float64(b)))
}
```

The Go implementation follows the same algorithmic structure as the Python version.

One important difference is integer handling. Since costs can become large after chaining multiple operations, the implementation uses `int64` throughout the DP and graph computations.

Go does not provide a built in infinity value for integers, so we use a large sentinel constant:

```go
const INF int64 = 1e18
```

Maps are used for string to ID conversion, and slices are used for the Floyd Warshall matrix and DP table.

The Floyd Warshall preprocessing dominates when many unique strings exist. Since there are at most 200 unique strings, this remains practical.

## Edge Cases

One important edge case occurs when the source and target strings are already identical. A careless implementation might still try to apply transformations unnecessarily. The DP handles this naturally by allowing transitions that simply advance one character when the characters already match.

Another tricky case involves indirect transformations being cheaper than direct ones. For example:

```
a -> b cost 10
a -> c cost 2
c -> b cost 2
```

A greedy solution using direct edges would incorrectly choose cost 10. Floyd Warshall correctly computes the shortest path cost 4.

A third important edge case is overlapping intervals. The problem explicitly forbids partial overlap. For example, intervals `[1,3]` and `[3,5]` cannot both be used. The DP avoids this issue entirely because it processes intervals left to right without revisiting covered positions.

Another subtle case is duplicate rules between the same strings. If two rules exist:

```
a -> b cost 10
a -> b cost 3
```

the algorithm must preserve only the cheaper edge. The graph initialization handles this with:

```
dist[u][v] = min(dist[u][v], w)
```

Finally, some substring pairs may never appear in the transformation graph at all. Attempting to transform such substrings would cause lookup failures or invalid transitions. The implementation explicitly checks whether both substrings exist in the graph before attempting a DP update.
