**Difficulty:** 🔴 Hard  
**Topics:** Array, Dynamic Programming, Bit Manipulation, Tree, Depth-First Search, Memoization

## Solution

## Problem Understanding

The problem gives us a rooted tree with `n` nodes, where node `0` is the root. Every node contains some number of coins, and we must collect coins from every node while respecting the parent-to-child order. A node can only be processed after all of its ancestors have already been processed.

At each node, we have exactly two choices for how to collect its coins:

1. Take the coins directly and gain `coins[i] - k` points.
2. Take half the coins and gain `floor(coins[i] / 2)` points. If we choose this option, then every node in the current node's subtree also has its coin value permanently halved.

The important detail is that halving operations accumulate. If multiple ancestors use the second option, then a node's coin value may already have been divided by powers of two before we reach it.

The goal is to maximize the total score after processing all nodes.

The input consists of:

- `edges`, which defines the tree structure
- `coins`, where `coins[i]` is the original value at node `i`
- `k`, the penalty subtracted when using the first collection method

The output is a single integer representing the maximum achievable score.

The constraints are large:

- `n` can be as large as `10^5`
- `coins[i]` can be up to `10^4`

These constraints immediately tell us that exponential exploration of all choices is impossible. Since each node has two possible decisions, a naive approach would consider `2^n` states, which is far too large.

Another important observation is that repeated halving quickly drives values toward zero. Since `coins[i] <= 10^4`, repeated division by two becomes zero after at most about 14 operations:

$$2^{14} = 16384$$

This observation becomes the key optimization.

Some important edge cases include:

- Nodes with `0` coins
- `k = 0`, where direct collection is always attractive
- Deep trees where many halving operations stack together
- Cases where taking a negative score now leads to better subtree outcomes later
- Star-shaped trees versus chain-shaped trees

The problem guarantees the graph is a valid tree, so there are no cycles and exactly one path between any two nodes.

## Approaches

### Brute Force

A brute force solution would recursively explore both choices at every node.

For each node:

- Either collect directly with `coins[i] - k`
- Or halve the subtree and collect `floor(coins[i] / 2)`

The difficulty is that choosing the second option changes the state of every descendant. Therefore, the future value of each node depends on how many ancestors previously chose the halving operation.

A brute force recursive search would attempt every possible combination of choices across all nodes. Since each node has two options, this leads to roughly:

$$O(2^n)$$

states.

This approach is correct because it explores every possible strategy, but it is completely infeasible for `n = 10^5`.

### Key Insight

The crucial observation is that the only information that matters when processing a node is:

- Which node we are at
- How many times the current coin value has already been halved

Suppose a node has already been affected by `shift` halving operations from its ancestors. Then its effective value is:

$$\left\lfloor \frac{coins[i]}{2^{shift}} \right\rfloor$$

At this point, we again have two choices:

- Take directly:

$$value - k$$

- Halve again:

$$\left\lfloor \frac{value}{2} \right\rfloor$$

and pass `shift + 1` to all descendants.

This naturally leads to tree dynamic programming with memoization.

Because `coins[i] <= 10^4`, after about 14 halvings all values become zero. Therefore the number of distinct `shift` states is very small, around 14 to 15.

This transforms the problem into:

$$O(n \cdot 15)$$

which is efficient enough.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(n) | Explores every combination of decisions |
| Optimal DFS + DP | O(n × log(maxCoin)) | O(n × log(maxCoin)) | Memoizes subtree results by node and halving count |

## Algorithm Walkthrough

1. Build the adjacency list for the tree.

Since the input is given as edges, we first convert it into an adjacency list so we can efficiently traverse children during DFS.
2. Define the DP state.

Let:

$$dp(node, parent, shift)$$

represent the maximum points obtainable from the subtree rooted at `node`, assuming the node's coin value has already been halved `shift` times by its ancestors.
3. Compute the effective coin value.

The current node's usable value becomes:

$$current = coins[node] >> shift$$

Using bit shifting is equivalent to floor division by powers of two.
4. Evaluate option one.

If we collect directly, we gain:

$$current - k$$

Then all children continue with the same `shift`.
5. Evaluate option two.

If we choose the halving operation, we gain:

$$current >> 1$$

Then all descendants receive one additional halving, so children use `shift + 1`.
6. Recursively process children.

For each child:

- Add `dp(child, node, shift)` for option one
- Add `dp(child, node, shift + 1)` for option two
7. Take the better result.

The answer for this state is:

$$\max(option1, option2)$$
8. Memoize results.

Since many subtree states repeat, memoization avoids recomputation.
9. Start DFS from the root.

The final answer is:

$$dp(0, -1, 0)$$

### Why it works

The algorithm works because every node's future behavior depends only on how many times its value has been halved. The actual sequence of ancestor choices does not matter beyond this count.

The recursive DP correctly considers both legal decisions at every node and combines optimal subtree results. Since every subtree is solved optimally for every possible halving count, the overall solution is globally optimal.

## Python Solution

```python
from typing import List
from functools import lru_cache

class Solution:
    def maximumPoints(self, edges: List[List[int]], coins: List[int], k: int) -> int:
        n = len(coins)

        graph = [[] for _ in range(n)]

        for u, v in edges:
            graph[u].append(v)
            graph[v].append(u)

        MAX_SHIFT = 15

        @lru_cache(None)
        def dfs(node: int, parent: int, shift: int) -> int:
            current_value = coins[node] >> shift

            # Option 1: collect directly
            option1 = current_value - k

            for neighbor in graph[node]:
                if neighbor == parent:
                    continue

                option1 += dfs(neighbor, node, shift)

            # Option 2: halve subtree again
            halved_value = current_value >> 1
            option2 = halved_value

            next_shift = min(MAX_SHIFT, shift + 1)

            for neighbor in graph[node]:
                if neighbor == parent:
                    continue

                option2 += dfs(neighbor, node, next_shift)

            return max(option1, option2)

        return dfs(0, -1, 0)
```

The implementation begins by constructing an adjacency list for the tree. Since the graph is undirected, every edge is added in both directions.

The DFS function represents the memoized dynamic programming state. The `shift` parameter tracks how many cumulative halving operations have affected the current node.

The expression:

```python
coins[node] >> shift
```

efficiently computes:

$$\left\lfloor \frac{coins[node]}{2^{shift}} \right\rfloor$$

For each node, the code evaluates both legal choices.

The first option collects the current value directly and subtracts `k`. Children continue with the same shift count because no new halving operation occurs.

The second option halves the subtree again. The node gains half its current value, and descendants use `shift + 1`.

Memoization is essential because many subtree states repeat. Without caching, the recursion would become exponential.

The `MAX_SHIFT` cap works because values become zero after enough divisions by two. Additional halvings produce identical states.

## Go Solution

```go
package main

func maximumPoints(edges [][]int, coins []int, k int) int {
	n := len(coins)

	graph := make([][]int, n)

	for _, edge := range edges {
		u := edge[0]
		v := edge[1]

		graph[u] = append(graph[u], v)
		graph[v] = append(graph[v], u)
	}

	const MAX_SHIFT = 15

	type State struct {
		node   int
		parent int
		shift  int
	}

	memo := make(map[State]int)

	var dfs func(int, int, int) int

	dfs = func(node int, parent int, shift int) int {
		state := State{node, parent, shift}

		if val, exists := memo[state]; exists {
			return val
		}

		currentValue := coins[node] >> shift

		// Option 1
		option1 := currentValue - k

		for _, neighbor := range graph[node] {
			if neighbor == parent {
				continue
			}

			option1 += dfs(neighbor, node, shift)
		}

		// Option 2
		option2 := currentValue >> 1

		nextShift := shift + 1
		if nextShift > MAX_SHIFT {
			nextShift = MAX_SHIFT
		}

		for _, neighbor := range graph[node] {
			if neighbor == parent {
				continue
			}

			option2 += dfs(neighbor, node, nextShift)
		}

		if option1 > option2 {
			memo[state] = option1
		} else {
			memo[state] = option2
		}

		return memo[state]
	}

	return dfs(0, -1, 0)
}
```

The Go implementation follows the same dynamic programming structure as the Python solution.

Since Go does not provide built in memoization decorators like Python's `lru_cache`, a custom map is used with a struct key representing the DP state.

Bit shifting behaves the same way in Go and efficiently computes division by powers of two.

The recursion depth can become large for chain shaped trees. In competitive programming environments this is usually acceptable in Go, though iterative DFS would be safer for extremely deep recursion limits.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n × log(maxCoin)) | Each node is evaluated for a bounded number of shift states |
| Space | O(n × log(maxCoin)) | Memoization table plus recursion stack |

The number of possible `shift` values is small because repeated halving quickly reduces all values to zero. Since `coins[i] <= 10^4`, at most about 14 meaningful shift states exist.

Therefore each node participates in only a small constant number of DP states, making the solution effectively linear.

## Edge Cases

One important edge case occurs when `k` is very large relative to all coin values. In this situation, direct collection may produce heavily negative scores at every node. A buggy implementation might greedily choose direct collection anyway. The DP correctly evaluates both possibilities and can repeatedly choose the halving operation until values become zero, avoiding unnecessary penalties.

Another tricky case is repeated halving across many ancestors. A node deep in the tree may already have undergone several divisions before it is processed. If the implementation does not correctly propagate the `shift` state through recursion, descendant values will be incorrect. The memoized DFS explicitly carries the halving count as part of the state, ensuring every node uses the proper effective value.

Zero valued nodes are another potential source of bugs. After enough halving operations, many values become zero. An implementation that continues increasing shift indefinitely could waste memory or time. This solution caps the shift count because once values reach zero, additional halving no longer changes anything.

A chain shaped tree is also important because recursion depth becomes maximal. The algorithm still remains efficient because each node is processed for only a small number of shift states, preventing exponential blowup even in highly unbalanced trees.
