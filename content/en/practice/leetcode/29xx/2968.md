## Solution

## Problem Understanding

The problem gives us an integer array `nums` and a budget `k`. Each operation allows us to either increase or decrease any element by exactly `1`. We may perform at most `k` total operations across the entire array.

Our goal is to maximize the frequency of some value after performing these operations. In other words, we want to make as many elements as possible equal, while spending no more than `k` total changes.

A key observation is that changing a number from `a` to `b` costs `|a - b|` operations, because each increment or decrement changes the value by one. Therefore, for any chosen target value, the total cost to convert several elements into that target is the sum of their absolute differences from the target.

The problem asks for the maximum number of elements that can be transformed into the same value within the allowed budget.

The constraints are large:

- `nums.length` can be up to `10^5`
- `nums[i]` can be as large as `10^9`
- `k` can be as large as `10^14`

These limits immediately rule out quadratic solutions. Any algorithm that compares every subarray or every target independently will be too slow.

There are several important details and edge cases to recognize early:

- Since both increments and decrements are allowed, the optimal target is not necessarily one of the largest values.
- The best target for minimizing total absolute distance is the median of the chosen elements.
- Large values of `k` require careful handling of integer overflow, especially in Go.
- Arrays of length `1` always return `1`.
- When `k = 0`, the answer is simply the maximum existing frequency.
- Sorting changes the problem into finding the largest window whose elements can be converted into a common value cheaply enough.

## Approaches

### Brute Force Approach

A brute force solution would try every possible subset or window of elements and determine whether those elements can all be transformed into a common target value within budget `k`.

After sorting, we could examine every subarray `[l, r]`. For each subarray, we would compute the minimum cost required to make all values equal. Since the minimum sum of absolute differences is achieved at the median, we could choose the median element as the target and calculate:

$$\sum |nums[i] - median|$$

If the cost is at most `k`, then the subarray length is a valid answer.

However, there are `O(n^2)` subarrays. Even if we compute each cost efficiently using prefix sums, the total complexity becomes `O(n^2)`, which is far too slow for `n = 10^5`.

### Optimal Approach

The crucial insight is that after sorting the array, the optimal group of elements will always form a contiguous window.

For any sorted window, the cheapest target value is the median element. This is a classic property of absolute deviation minimization.

Therefore, the problem becomes:

- Find the largest window where the cost to convert all values into the median is at most `k`.

We can solve this efficiently using:

- Sorting
- Prefix sums
- Sliding window

The prefix sums allow us to compute transformation costs in constant time for any window. The sliding window lets us expand and shrink candidate ranges efficiently.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Checks every subarray independently |
| Optimal | O(n log n) | O(n) | Uses sorting, prefix sums, and sliding window |

## Algorithm Walkthrough

### Step 1: Sort the Array

We first sort `nums`.

Sorting is important because elements that are close in value become adjacent. Any optimal set of elements can then be represented as a contiguous subarray.

For example:

```
nums = [1,2,6,4]
sorted = [1,2,4,6]
```

### Step 2: Build Prefix Sums

We create a prefix sum array where:

$$prefix[i] = \text{sum of first } i \text{ elements}$$

This allows us to compute sums of arbitrary ranges in constant time.

For a range `[l, r]`:

$$sum(l,r) = prefix[r+1] - prefix[l]$$

### Step 3: Use a Sliding Window

We maintain a window `[left, right]`.

For every `right` index, we try to expand the window as much as possible.

The question becomes:

- Can all elements in this window be converted into the median value within cost `k`?

### Step 4: Compute the Median

For a sorted window:

```
[left ... right]
```

the optimal target is:

```
mid = (left + right) // 2
target = nums[mid]
```

### Step 5: Compute Left Cost

All elements on the left side must increase to the median.

The cost is:

$$target \times countLeft - sumLeft$$

where:

- `countLeft = mid - left`
- `sumLeft` is the sum of values before the median

### Step 6: Compute Right Cost

All elements on the right side must decrease to the median.

The cost is:

$$sumRight - target \times countRight$$

where:

- `countRight = right - mid`
- `sumRight` is the sum of values after the median

### Step 7: Check Budget

Total cost:

$$leftCost + rightCost$$

If this exceeds `k`, we shrink the window from the left.

Otherwise, the current window is valid, and we update the answer.

### Step 8: Continue Until Completion

We repeat this process for every `right` index and keep track of the largest valid window.

### Why it works

The correctness relies on two properties:

1. In a sorted array, the minimum total absolute distance to a single value is achieved at the median.
2. Any optimal group of elements becomes a contiguous window after sorting.

The sliding window guarantees that every candidate window is examined exactly once, while prefix sums ensure that cost computations remain efficient.

## Python Solution

```python
from typing import List

class Solution:
    def maxFrequencyScore(self, nums: List[int], k: int) -> int:
        nums.sort()
        n = len(nums)

        prefix = [0] * (n + 1)

        for i in range(n):
            prefix[i + 1] = prefix[i] + nums[i]

        def cost(left: int, right: int) -> int:
            mid = (left + right) // 2
            median = nums[mid]

            left_sum = prefix[mid] - prefix[left]
            left_count = mid - left
            left_cost = median * left_count - left_sum

            right_sum = prefix[right + 1] - prefix[mid + 1]
            right_count = right - mid
            right_cost = right_sum - median * right_count

            return left_cost + right_cost

        answer = 1
        left = 0

        for right in range(n):
            while cost(left, right) > k:
                left += 1

            answer = max(answer, right - left + 1)

        return answer
```

The implementation begins by sorting the array, which enables the sliding window strategy.

Next, the prefix sum array is constructed. This allows fast range sum queries, which are essential for efficient cost calculations.

The helper function `cost(left, right)` computes the minimum number of operations needed to make all elements in the window equal. The median element is chosen automatically because it minimizes the total absolute difference.

Inside the sliding window loop:

- `right` expands the window
- If the cost exceeds `k`, the `left` pointer moves forward until the window becomes valid again
- The maximum valid window size is stored in `answer`

Every element enters and leaves the window at most once, making the sliding window efficient.

## Go Solution

```go
package main

import (
	"sort"
)

func maxFrequencyScore(nums []int, k int64) int {
	sort.Ints(nums)

	n := len(nums)

	prefix := make([]int64, n+1)

	for i := 0; i < n; i++ {
		prefix[i+1] = prefix[i] + int64(nums[i])
	}

	cost := func(left, right int) int64 {
		mid := (left + right) / 2
		median := int64(nums[mid])

		leftSum := prefix[mid] - prefix[left]
		leftCount := int64(mid - left)
		leftCost := median*leftCount - leftSum

		rightSum := prefix[right+1] - prefix[mid+1]
		rightCount := int64(right - mid)
		rightCost := rightSum - median*rightCount

		return leftCost + rightCost
	}

	answer := 1
	left := 0

	for right := 0; right < n; right++ {
		for cost(left, right) > k {
			left++
		}

		windowSize := right - left + 1

		if windowSize > answer {
			answer = windowSize
		}
	}

	return answer
}
```

The Go solution follows the same algorithmic structure as the Python implementation.

The main difference is integer handling. Since `k` may be as large as `10^14`, all arithmetic involving sums and costs must use `int64`. Using regular `int` could overflow on some systems.

The prefix sums are stored as `[]int64`, and every arithmetic expression related to costs is explicitly converted to `int64`.

Go slices naturally handle dynamic window ranges, and the sorting step uses `sort.Ints`.

## Edge Cases

### Single Element Arrays

If the array contains only one element, the answer must always be `1`, regardless of `k`. A careless implementation might assume at least two elements exist when computing medians or window costs. This implementation handles the case naturally because the sliding window begins with size `1`, and the cost function correctly returns zero.

### Zero Operations Allowed

When `k = 0`, no modifications are possible. The problem reduces to finding the maximum existing frequency in the original array. The algorithm still works correctly because only windows with zero transformation cost remain valid.

### Very Large Numbers and Budgets

The constraints allow values up to `10^9` and budgets up to `10^14`. Intermediate sums can therefore become extremely large. In Go, using regular `int` arithmetic may overflow. The implementation avoids this by using `int64` for prefix sums and cost computations.

### Even Length Windows

For windows with an even number of elements, there are technically two median candidates. Either one produces the same minimum total absolute deviation. The implementation consistently chooses:

```
mid = (left + right) // 2
```

This guarantees deterministic behavior while still preserving correctness.

### Large Invalid Windows

Some windows may initially exceed the budget by a large margin. The sliding window logic repeatedly increments `left` until the cost becomes valid again. This ensures the algorithm never misses a valid candidate window and maintains linear scanning behavior.
