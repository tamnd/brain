## Solution

## Problem Understanding

The problem gives us an array `nums`, where each value represents a possible side length. We want to choose some subset of these values, using at least three numbers, so that they can form a valid polygon. Among all valid polygons, we must return the largest possible perimeter.

A polygon with sides sorted in nondecreasing order:

$$a_1 \le a_2 \le \dots \le a_k$$

is valid if:

$$a_1 + a_2 + \dots + a_{k-1} > a_k$$

This condition says that the largest side must be strictly smaller than the sum of all the remaining sides. This is the generalized polygon inequality. It is similar to the triangle inequality, except it applies to any number of sides greater than or equal to three.

The input size can be very large:

- `n` can be up to `10^5`
- each side length can be up to `10^9`

These constraints immediately tell us that exponential or quadratic subset exploration is impossible. We need an efficient algorithm, ideally dominated only by sorting.

An important observation is that all numbers are positive. This matters because adding more valid sides always increases the perimeter. There are no negative or zero values that could reduce the sum.

Some important edge cases include:

- Arrays where no polygon can be formed at all, such as `[5,5,50]`
- Arrays where only a smaller subset works, even though larger elements exist
- Arrays with many equal values
- Cases where the optimal polygon uses more than three sides
- Very large numbers that require 64 bit integer handling in Go

The problem guarantees:

- At least 3 elements exist
- Every value is positive
- Any valid answer fits within 64 bit integer range

## Approaches

### Brute Force Approach

A brute force solution would examine every possible subset of size at least three. For each subset, we would:

1. Sort the subset
2. Check whether the polygon inequality holds
3. Compute its perimeter
4. Track the maximum valid perimeter

This works because it explicitly checks every possible polygon configuration.

However, the number of subsets is exponential:

$$2^n$$

With `n = 100000`, this is completely infeasible. Even generating all subsets for much smaller values would already time out.

The brute force approach is useful conceptually because it reveals the key property we need to optimize: validating the polygon inequality efficiently.

### Key Insight

Suppose the array is sorted:

$$nums[0] \le nums[1] \le \dots \le nums[n-1]$$

For a sorted set of sides, the only condition we need to verify is:

$$\text{sum of smaller sides} > \text{largest side}$$

If we already know the total sum of the current prefix, then checking validity becomes very cheap.

Another critical observation is:

- Since all values are positive, including more valid sides always increases the perimeter.
- Therefore, for any valid polygon, we prefer using as many sides as possible.

This suggests a greedy strategy:

- Sort the array
- Maintain the running prefix sum
- At each position, check whether the current number can serve as the largest side of a valid polygon
- If it can, then the entire prefix forms a valid polygon, and its perimeter is the current prefix sum

Because we process numbers in sorted order, the last valid prefix automatically gives the maximum perimeter.

## Algorithm Walkthrough

1. Sort the array in nondecreasing order.

Sorting is important because the polygon condition only depends on comparing the largest side against the sum of all smaller sides. With sorting, the current element is always the largest side of the current prefix.
2. Initialize a running prefix sum.

This variable stores the total sum of all processed elements so far.
3. Iterate through the sorted array.

At each index `i`, treat `nums[i]` as the largest side of a potential polygon.
4. Before adding the current number to the prefix sum, check whether:

$$\text{prefix\_sum} > nums[i]$$

Here, `prefix_sum` represents the sum of all smaller sides.

1. If the condition holds, then the polygon formed by all elements up to index `i` is valid.

Its perimeter equals:

$$\text{prefix\_sum} + nums[i]$$

Record this as the current best answer.

1. Add the current number to the running prefix sum.
2. Continue until the array ends.
3. If no valid polygon was ever found, return `-1`.

### Why it works

After sorting, every prefix has its largest side at the end. The polygon inequality only requires checking whether the sum of previous sides exceeds this largest side.

Because all side lengths are positive, adding additional sides always increases the perimeter. Therefore, among all valid polygons, the best one is simply the largest valid prefix in sorted order.

The algorithm checks every possible prefix exactly once, so it cannot miss the optimal answer.

## Python Solution

```python
from typing import List

class Solution:
    def largestPerimeter(self, nums: List[int]) -> int:
        nums.sort()

        prefix_sum = 0
        best = -1

        for side in nums:
            if prefix_sum > side:
                best = prefix_sum + side

            prefix_sum += side

        return best
```

The implementation begins by sorting the array so that every element can be treated as the largest side of the current prefix.

The variable `prefix_sum` tracks the sum of all previously processed sides. Before adding the current side, we check whether the polygon inequality holds. If it does, then the current prefix forms a valid polygon.

The variable `best` stores the largest perimeter seen so far. Since we process sides in increasing order and all values are positive, later valid prefixes automatically have larger perimeters.

Finally, if no valid polygon exists, `best` remains `-1`.

## Go Solution

```go
package main

import (
	"sort"
)

func largestPerimeter(nums []int) int64 {
	sort.Ints(nums)

	var prefixSum int64 = 0
	var best int64 = -1

	for _, side := range nums {
		current := int64(side)

		if prefixSum > current {
			best = prefixSum + current
		}

		prefixSum += current
	}

	return best
}
```

The Go implementation follows exactly the same logic as the Python solution.

One important difference is integer handling. Since the perimeter may exceed the range of a 32 bit integer, the running sum and final answer use `int64`.

The array is sorted using `sort.Ints`, and iteration is performed using a range loop.

## Edge Cases

One important edge case occurs when the largest side equals the sum of all smaller sides, such as `[1,2,3]`. A common mistake is using `>=` instead of `>`. The polygon inequality is strict, so equality does not produce a valid polygon. The implementation correctly checks `prefix_sum > side`.

Another critical edge case is when very large numbers are involved. Since side lengths can reach `10^9` and there may be up to `10^5` elements, the perimeter can become extremely large. The Go solution uses `int64` to prevent overflow.

A third important edge case is when the optimal polygon excludes some large values. For example:

```
[1,12,1,2,5,50,3]
```

A naive strategy might always try to include the largest side. However, `50` and `12` cannot participate in any valid polygon here. The sorted prefix approach naturally handles this because it validates every prefix independently.

Another subtle case is arrays where all elements are usable. Since all numbers are positive, adding more valid sides always improves the perimeter. The algorithm correctly keeps updating the answer whenever a larger valid prefix is found.
