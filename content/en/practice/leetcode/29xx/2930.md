## Solution

## Problem Understanding

We are given an integer `n`, and we must count how many strings of length `n` satisfy a special property.

A string is considered good if its characters can be rearranged so that the substring `"leet"` appears somewhere in the rearranged string.

The key detail is that we are allowed to rearrange the string arbitrarily before checking for `"leet"`. That means the original order of characters does not matter at all. Instead, only the character frequencies matter.

The substring `"leet"` contains:

- one `'l'`
- two `'e'`
- one `'t'`

Therefore, a string is good if and only if:

- it contains at least one `'l'`
- it contains at least two `'e'`
- it contains at least one `'t'`

All remaining characters can be anything.

The input consists of a single integer `n`, which represents the string length. We must return the total number of valid strings modulo `10^9 + 7`.

The constraint `1 <= n <= 10^5` is extremely important. A brute force approach that generates all possible strings is impossible because there are `26^n` total strings. Even for moderate values of `n`, this becomes astronomically large.

This constraint strongly suggests that we need a mathematical counting technique, typically combinatorics or dynamic programming.

Several edge cases are important:

- If `n < 4`, the answer must be `0`, because `"leet"` itself requires four characters.
- Strings may contain extra copies of `'l'`, `'e'`, or `'t'`, which are completely valid.
- The order of characters in the original string does not matter because rearrangement is allowed.
- Since the answer grows very quickly, modular arithmetic is required throughout the computation.

## Approaches

### Brute Force Approach

The most direct idea is to generate every possible string of length `n` using lowercase English letters.

There are `26^n` such strings.

For each string, we could count character frequencies and check whether:

- count(`'l'`) >= 1
- count(`'e'`) >= 2
- count(`'t'`) >= 1

If all conditions hold, we increment the answer.

This approach is correct because it explicitly checks every possible candidate string.

However, it is completely infeasible for the given constraints. Even for `n = 10`, there are already:

```
26^10 = 141,167,095,653,376
```

possible strings.

The runtime explodes exponentially, making brute force unusable.

### Optimal Approach, Inclusion-Exclusion Principle

The key observation is that a string is good exactly when it satisfies all required frequency conditions simultaneously.

Instead of directly counting good strings, we count:

```
All strings
minus
Strings missing at least one requirement
```

This is a classic Inclusion-Exclusion Principle problem.

Define the following bad sets:

- `A` = strings with no `'l'`
- `B` = strings with fewer than two `'e'`
- `C` = strings with no `'t'`

We want:

```
Total good strings = Total strings - |A ∪ B ∪ C|
```

Using inclusion-exclusion:

```
|A ∪ B ∪ C|
=
|A| + |B| + |C|
- |A ∩ B| - |A ∩ C| - |B ∩ C|
+ |A ∩ B ∩ C|
```

The challenge is computing each term carefully.

This reduces the problem from exponential enumeration to a small number of mathematical expressions involving fast exponentiation.

Since `n` can be as large as `10^5`, computing powers efficiently with modular exponentiation is critical.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(26^n * n) | O(n) | Generates every possible string |
| Optimal | O(log n) | O(1) | Uses inclusion-exclusion with fast exponentiation |

## Algorithm Walkthrough

### Step 1: Count all possible strings

Each position has 26 choices.

So the total number of strings is:

```
26^n
```

We compute this modulo `10^9 + 7`.

### Step 2: Count strings missing `'l'`

If a string contains no `'l'`, every position has only 25 choices.

So:

```
|A| = 25^n
```

### Step 3: Count strings missing `'t'`

This is symmetric to the previous case:

```
|C| = 25^n
```

### Step 4: Count strings with fewer than two `'e'`

There are two possibilities:

#### Case 1: No `'e'`

Every position has 25 choices:

```
25^n
```

#### Case 2: Exactly one `'e'`

Choose the position of the `'e'`:

```
n choices
```

Every remaining position has 25 choices:

```
n * 25^(n-1)
```

Therefore:

```
|B|
=
25^n + n * 25^(n-1)
```

### Step 5: Count pairwise intersections

#### Missing both `'l'` and `'t'`

Only 24 characters available:

```
|A ∩ C| = 24^n
```

#### Missing `'l'` and having fewer than two `'e'`

Again split into two cases.

No `'e'`:

```
24^n
```

Exactly one `'e'`:

```
n * 24^(n-1)
```

So:

```
|A ∩ B|
=
24^n + n * 24^(n-1)
```

By symmetry:

```
|B ∩ C|
=
24^n + n * 24^(n-1)
```

### Step 6: Count triple intersection

Strings:

- missing `'l'`
- missing `'t'`
- having fewer than two `'e'`

Again split into cases.

No `'e'`:

```
23^n
```

Exactly one `'e'`:

```
n * 23^(n-1)
```

Thus:

```
|A ∩ B ∩ C|
=
23^n + n * 23^(n-1)
```

### Step 7: Apply inclusion-exclusion

The final answer becomes:

```
26^n
- (|A| + |B| + |C|)
+ (|A ∩ B| + |A ∩ C| + |B ∩ C|)
- |A ∩ B ∩ C|
```

After simplifying:

```
26^n
- (3 * 25^n + n * 25^(n-1))
+ (3 * 24^n + 2 * n * 24^(n-1))
- (23^n + n * 23^(n-1))
```

All computations are done modulo `10^9 + 7`.

### Why it works

The inclusion-exclusion principle guarantees correctness because it precisely compensates for overcounting when multiple bad conditions overlap.

Every invalid string belongs to at least one bad set. Inclusion-exclusion ensures that each invalid string is counted exactly once overall, regardless of how many conditions it violates.

Subtracting this total from all strings leaves exactly the valid good strings.

## Python Solution

```python
class Solution:
    def stringCount(self, n: int) -> int:
        MOD = 10**9 + 7

        total = pow(26, n, MOD)

        p25_n = pow(25, n, MOD)
        p25_n1 = pow(25, n - 1, MOD) if n >= 1 else 1

        p24_n = pow(24, n, MOD)
        p24_n1 = pow(24, n - 1, MOD) if n >= 1 else 1

        p23_n = pow(23, n, MOD)
        p23_n1 = pow(23, n - 1, MOD) if n >= 1 else 1

        bad_single = (3 * p25_n + n * p25_n1) % MOD

        bad_pair = (
            3 * p24_n +
            2 * n * p24_n1
        ) % MOD

        bad_triple = (
            p23_n +
            n * p23_n1
        ) % MOD

        ans = (
            total
            - bad_single
            + bad_pair
            - bad_triple
        ) % MOD

        return ans
```

The implementation directly follows the inclusion-exclusion derivation.

First, we compute the total number of strings, `26^n`.

Next, we precompute all required powers:

- `25^n`
- `25^(n-1)`
- `24^n`
- `24^(n-1)`
- `23^n`
- `23^(n-1)`

Python's built-in `pow(base, exp, mod)` performs fast modular exponentiation in logarithmic time.

The code then computes:

- the sum of single bad sets
- the sum of pairwise intersections
- the triple intersection

Finally, inclusion-exclusion is applied exactly as derived mathematically.

The modulo operation at the end ensures the result remains non-negative.

## Go Solution

```go
func stringCount(n int) int {
	const MOD int64 = 1_000_000_007

	modPow := func(base, exp int64) int64 {
		result := int64(1)
		base %= MOD

		for exp > 0 {
			if exp&1 == 1 {
				result = (result * base) % MOD
			}
			base = (base * base) % MOD
			exp >>= 1
		}

		return result
	}

	total := modPow(26, int64(n))

	p25n := modPow(25, int64(n))
	p25n1 := modPow(25, int64(n-1))

	p24n := modPow(24, int64(n))
	p24n1 := modPow(24, int64(n-1))

	p23n := modPow(23, int64(n))
	p23n1 := modPow(23, int64(n-1))

	badSingle := (
		3*p25n +
		int64(n)*p25n1,
	) % MOD

	badPair := (
		3*p24n +
		2*int64(n)*p24n1,
	) % MOD

	badTriple := (
		p23n +
		int64(n)*p23n1,
	) % MOD

	ans := (
		total -
		badSingle +
		badPair -
		badTriple,
	) % MOD

	if ans < 0 {
		ans += MOD
	}

	return int(ans)
}
```

The Go solution mirrors the Python logic closely.

Since Go does not provide built-in modular exponentiation, we implement binary exponentiation manually.

All calculations use `int64` to avoid overflow during multiplication.

Unlike Python, Go's modulo operator can produce negative values for negative operands, so we explicitly normalize the result by adding `MOD` when necessary.

## Edge Cases

### Edge Case 1: `n < 4`

A valid string must contain at least four characters because `"leet"` itself has length four.

This case can easily cause logical mistakes if the implementation assumes every formula naturally evaluates to zero. The inclusion-exclusion formula correctly produces zero automatically for these values, so no special branching is required.

### Edge Case 2: Exactly two `'e'` characters

The requirement is not merely to contain `'e'`, but to contain at least two copies.

A common bug is forgetting the duplicate requirement and treating `"leet"` as requiring four distinct characters. The inclusion-exclusion derivation explicitly handles this by defining the bad set as strings with fewer than two `'e'`.

### Edge Case 3: Very large `n`

For `n = 100000`, direct exponentiation would overflow standard integers and naive repeated multiplication would be too slow.

The implementation avoids both problems by:

- using modular arithmetic throughout
- using fast exponentiation
- storing intermediate results modulo `10^9 + 7`

This guarantees both correctness and efficiency even at the maximum constraint.
