## Solution

## Problem Understanding

The problem asks us to count how many valid ways exist to distribute exactly `n` candies among `3` children, under the restriction that no child may receive more than `limit` candies.

In other words, we want to count the number of integer triplets:

$$(a, b, c)$$

such that:

$$a + b + c = n$$

and:

$$0 \le a, b, c \le limit$$

Each child is distinct, so distributions like `(1,2,2)` and `(2,1,2)` are considered different.

The input consists of two integers:

- `n`, the total number of candies
- `limit`, the maximum candies any one child may receive

The output should be the total number of valid distributions satisfying all constraints.

The constraints are important:

- `1 <= n <= 10^6`
- `1 <= limit <= 10^6`

These large values immediately tell us that exponential or deeply nested brute-force enumeration is not feasible. We need a solution that runs in constant time or linear time at worst.

Several edge cases are important:

- When `n > 3 * limit`, distributing all candies is impossible because even giving every child the maximum still does not reach `n`.
- When `limit >= n`, the upper bound effectively disappears, and the problem reduces to counting all non-negative solutions to `a + b + c = n`.
- When `n = 0`, there would be exactly one solution `(0,0,0)`, although the constraints guarantee `n >= 1`.
- Small values near boundaries can easily produce off-by-one errors in counting logic.

The key challenge is efficiently counting constrained integer solutions.

## Approaches

### Brute Force Approach

The most direct solution is to try every possible amount for the first child and every possible amount for the second child. The third child's amount is then determined automatically.

We can write:

- Choose `a` from `0` to `limit`
- Choose `b` from `0` to `limit`
- Compute:

$$c = n - a - b$$

If `c` is between `0` and `limit`, then the distribution is valid.

This works because every valid triplet is checked exactly once.

However, the time complexity is too large. The nested loops each run up to `limit`, giving:

$$O(limit^2)$$

Since `limit` can be as large as `10^6`, this could require up to `10^{12}` iterations, which is completely infeasible.

### Optimal Approach

The key observation is that we do not actually need to enumerate every triplet.

Suppose we fix the candies given to the first child as `a`.

Then the remaining candies are:

$$remaining = n - a$$

Now we only need to count how many pairs `(b,c)` satisfy:

$$b + c = remaining$$

with:

$$0 \le b,c \le limit$$

Instead of iterating through all possibilities, we can compute the number of valid `b` values mathematically.

For a fixed `a`:

$$c = remaining - b$$

To ensure `c` stays within bounds:

$$0 \le remaining - b \le limit$$

Rearranging gives:

$$remaining - limit \le b \le remaining$$

Since `b` itself must also satisfy:

$$0 \le b \le limit$$

the valid range becomes:

$$\max(0, remaining - limit)
\le b \le
\min(limit, remaining)$$

The number of integers in this range is:

$$right - left + 1$$

if `left <= right`.

This lets us compute the answer in linear time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(limit²) | O(1) | Enumerates every pair `(a,b)` |
| Optimal | O(limit) | O(1) | Uses mathematical counting for valid ranges |

## Algorithm Walkthrough

1. Initialize a variable `ways` to store the total number of valid distributions.
2. Iterate through all possible candy counts for the first child, `a`, from `0` to `min(limit, n)`.

We never need to consider values larger than `n` because distributing more than all candies is impossible.
3. Compute the remaining candies after assigning candies to the first child:

$$remaining = n - a$$

1. Determine the smallest valid value for the second child `b`.

Since the third child receives:

$$c = remaining - b$$

and must satisfy:

$$c \le limit$$

we get:

$$remaining - b \le limit$$

which becomes:

$$b \ge remaining - limit$$

Also, `b` cannot be negative, so:

$$left = \max(0, remaining - limit)$$

1. Determine the largest valid value for `b`.

Since `b` itself cannot exceed `limit`, and also cannot exceed `remaining`:

$$right = \min(limit, remaining)$$

1. If `left <= right`, then every integer in that interval corresponds to exactly one valid pair `(b,c)`.

Add:

$$right - left + 1$$

to the answer.

1. After processing all possible values of `a`, return the accumulated count.

### Why it works

For every fixed value of `a`, the algorithm counts all valid values of `b` that produce a legal value for `c`. Each valid triplet `(a,b,c)` corresponds to exactly one iteration and one valid `b` choice, so every distribution is counted exactly once. The range calculation guarantees that all constraints are satisfied simultaneously.

## Python Solution

```python
class Solution:
    def distributeCandies(self, n: int, limit: int) -> int:
        ways = 0

        for first_child in range(min(limit, n) + 1):
            remaining = n - first_child

            left = max(0, remaining - limit)
            right = min(limit, remaining)

            if left <= right:
                ways += right - left + 1

        return ways
```

The implementation follows the mathematical derivation directly.

The variable `first_child` represents the candies assigned to the first child. Once this value is fixed, the remaining candies are determined automatically.

The interval `[left, right]` represents all valid values for the second child. Every value in this interval produces exactly one valid value for the third child.

The expression:

```python
right - left + 1
```

counts how many integers exist in the interval. This allows the algorithm to count many valid distributions at once instead of enumerating them individually.

The solution uses only a few integer variables, so the memory usage remains constant.

## Go Solution

```go
func distributeCandies(n int, limit int) int64 {
    var ways int64 = 0

    maxFirst := limit
    if n < limit {
        maxFirst = n
    }

    for firstChild := 0; firstChild <= maxFirst; firstChild++ {
        remaining := n - firstChild

        left := remaining - limit
        if left < 0 {
            left = 0
        }

        right := remaining
        if right > limit {
            right = limit
        }

        if left <= right {
            ways += int64(right - left + 1)
        }
    }

    return ways
}
```

The Go implementation is almost identical to the Python version, but there are a few language-specific differences.

The return type is `int64`, matching the required function signature. This avoids overflow issues because the number of valid distributions can exceed the range of a 32-bit integer.

Since Go does not provide built-in `min` and `max` functions for integers, we implement the comparisons manually using `if` statements.

Other than these differences, the algorithm remains the same.

## Edge Cases

One important edge case occurs when `n > 3 * limit`. In this situation, even if every child receives the maximum allowed candies, the total still cannot reach `n`. A naive implementation might accidentally count invalid configurations. The range calculations in this solution naturally prevent that, because no valid intervals will exist for the second child, resulting in a final answer of zero.

Another important case occurs when `limit >= n`. Here, the upper bound effectively disappears because no child could ever exceed the limit while still summing to `n`. The algorithm still works correctly because the computed ranges expand naturally to include all valid non-negative solutions.

A third tricky case involves very small limits, especially when only one or very few configurations exist. For example, `n = 6` and `limit = 2` has exactly one valid distribution: `(2,2,2)`. Off-by-one errors in interval counting are common in such cases. Using the inclusive interval formula:

$$right - left + 1$$

ensures the count is computed correctly.

Another subtle edge case is when the valid interval becomes empty. For example, if `left > right`, there are no valid values for the second child. The explicit condition:

```
if left <= right:
```

prevents adding negative counts or invalid values to the answer.
