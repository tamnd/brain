## Solution

## Problem Understanding

This problem asks us to find the **median** of two individually sorted arrays, `nums1` and `nums2`, without fully merging them in linear time. The key challenge is that the required time complexity is **O(log(m+n))**, where `m` and `n` are the lengths of the two arrays. This requirement immediately rules out straightforward approaches such as merging the arrays and sorting them again.

The median of a sorted sequence is the middle value. If the total number of elements is odd, the median is the single middle element. If the total number of elements is even, the median is the average of the two middle elements.

For example:

- `[1,2,3]` has median `2`
- `[1,2,3,4]` has median `(2 + 3) / 2 = 2.5`

The input consists of two already sorted integer arrays. Since both arrays are sorted, we should exploit this property instead of treating them as unsorted data.

The constraints are important:

- Each array length is between `0` and `1000`
- At least one array contains elements
- Combined length is at most `2000`
- Values range from `-10^6` to `10^6`

Although the total size is relatively small, the problem explicitly requires **O(log(m+n))** complexity. Even an `O(m+n)` merge solution, while fast enough in practice for these constraints, does not satisfy the intended requirement.

Several edge cases stand out immediately. One array may be empty, which means the median comes entirely from the other array. The arrays may have very different sizes, which can cause indexing mistakes if handled carelessly. Duplicate values are allowed, so comparisons must not assume uniqueness. Negative numbers also appear, meaning we cannot rely on positivity assumptions. Finally, odd and even total lengths require different median calculations.

## Approaches

### Brute Force Approach

The most intuitive solution is to merge the two sorted arrays into a single sorted array and then compute the median.

Since both arrays are already sorted, we can perform a classic merge operation similar to merge sort. We compare the front elements of both arrays, append the smaller one, and continue until all elements are processed.

Once merged, finding the median becomes trivial:

- If total length is odd, return the middle element.
- If total length is even, return the average of the two middle elements.

This approach is correct because the merged array preserves sorted order, meaning the median definition applies directly.

However, this solution takes **O(m+n)** time and **O(m+n)** extra space, which violates the problem requirement of **O(log(m+n))**.

### Optimal Approach, Binary Search on Partition

The key insight is that we do **not** need the entire merged array. We only care about the middle position.

Instead of merging, we can think about dividing both arrays into two halves such that:

- The left half contains exactly half the elements.
- Every element in the left half is less than or equal to every element in the right half.

If we can find such a partition, the median becomes easy to calculate.

Suppose we partition the arrays like this:

```
nums1: [1, 3 | 8, 9]
nums2: [2, 7 | 10, 11]
```

The left side contains `[1,3,2,7]`, and the right side contains `[8,9,10,11]`.

For a valid partition:

```
max(left side) <= min(right side)
```

In terms of boundary values:

```
left1 <= right2
left2 <= right1
```

Because the arrays are sorted, if these conditions hold, we know the partition is correct.

Instead of searching both arrays, we binary search only on the smaller array. This gives us an efficient **O(log(min(m,n)))** solution, which satisfies the required complexity.

| Approach | Time Complexity | Space Complexity | Notes |

| Brute Force | O(m+n) | O(m+n) | Merge arrays completely, then compute median |
| Optimal | O(log(min(m,n))) | O(1) | Binary search on partition of smaller array |

## Algorithm Walkthrough

1. **Ensure binary search happens on the smaller array**

We always binary search on the shorter array. This minimizes search space and prevents partition indices from going out of bounds.
2. **Compute total length and left partition size**

Let:

```
total = m + n
half = (total + 1) // 2
```

The `+1` ensures odd-length cases work naturally, placing the extra element on the left side.
3. **Binary search for a valid partition**

Use binary search over `nums1`.

Let:

```
partition1 = (left + right) // 2
partition2 = half - partition1
```

These values determine how many elements belong to the left partition from each array.
4. **Determine partition boundary values**

For each partition:

```
left1  = largest value on left of nums1
right1 = smallest value on right of nums1
left2  = largest value on left of nums2
right2 = smallest value on right of nums2
```

When a partition reaches an edge, use sentinel values:

```
-infinity
+infinity
```

This avoids special-case logic.
5. **Check partition validity**

A valid partition satisfies:

```
left1 <= right2
left2 <= right1
```

If true, we found the correct split.
6. **Compute the median**

If total length is odd:

```
median = max(left1, left2)
```

If total length is even:

```
median = (
    max(left1, left2) +
    min(right1, right2)
) / 2
```
7. **Adjust binary search**

If:

```
left1 > right2
```

then we took too many elements from `nums1`, move left.

Otherwise:

```
left2 > right1
```

meaning we need more elements from `nums1`, move right.

## Python Solution

```
from typing import List

class Solution:
    def findMedianSortedArrays(
        self,
        nums1: List[int],
        nums2: List[int]
    ) -> float:

        # Always binary search on the smaller array
        if len(nums1) > len(nums2):
            nums1, nums2 = nums2, nums1

        m, n = len(nums1), len(nums2)
        total_length = m + n
        half_length = (total_length + 1) // 2

        left = 0
        right = m

        while left <= right:
            partition1 = (left + right) // 2
            partition2 = half_length - partition1

            left1 = (
                float("-inf")
                if partition1 == 0
                else nums1[partition1 - 1]
            )

            right1 = (
                float("inf")
                if partition1 == m
                else nums1[partition1]
            )

            left2 = (
                float("-inf")
                if partition2 == 0
                else nums2[partition2 - 1]
            )

            right2 = (
                float("inf")
                if partition2 == n
                else nums2[partition2]
            )

            # Valid partition found
            if left1 <= right2 and left2 <= right1:

                # Odd total length
                if total_length % 2 == 1:
                    return float(max(left1, left2))

                # Even total length
                return (
                    max(left1, left2) +
                    min(right1, right2)
                ) / 2.0

            # Too many elements taken from nums1
            elif left1 > right2:
                right = partition1 - 1

            # Too few elements taken from nums1
            else:
                left = partition1 + 1

        return 0.0
```

The implementation begins by ensuring `nums1` is the smaller array. This is important because binary search complexity depends on the search space size, and it prevents partition calculations from becoming invalid.

Next, the algorithm computes the combined length and determines how many elements should belong to the left partition. Binary search boundaries are initialized from `0` to `m`, allowing partitions anywhere in the smaller array.

Inside the loop, partition positions are computed for both arrays. The partition in the second array is derived automatically so that the total left side always contains exactly half the elements.

Boundary values around each partition are then calculated. Sentinel values (`-inf` and `inf`) are used whenever a partition reaches an array edge. This simplifies comparisons because edge cases naturally fit into the same logic.

Once a valid partition is found, the median calculation depends on parity. For odd total lengths, the maximum value from the left side becomes the answer. For even lengths, the median is the average of the middle pair.

If the partition is invalid, binary search adjusts left or right depending on which side violates the ordering condition.

## Go Solution

```
import "math"

func findMedianSortedArrays(nums1 []int, nums2 []int) float64 {

	// Always binary search on the smaller array
	if len(nums1) > len(nums2) {
		return findMedianSortedArrays(nums2, nums1)
	}

	m, n := len(nums1), len(nums2)
	totalLength := m + n
	halfLength := (totalLength + 1) / 2

	left, right := 0, m

	for left <= right {

		partition1 := (left + right) / 2
		partition2 := halfLength - partition1

		left1 := math.Inf(-1)
		if partition1 > 0 {
			left1 = float64(nums1[partition1-1])
		}

		right1 := math.Inf(1)
		if partition1 < m {
			right1 = float64(nums1[partition1])
		}

		left2 := math.Inf(-1)
		if partition2 > 0 {
			left2 = float64(nums2[partition2-1])
		}

		right2 := math.Inf(1)
		if partition2 < n {
			right2 = float64(nums2[partition2])
		}

		// Valid partition found
		if left1 <= right2 && left2 <= right1 {

			// Odd total length
			if totalLength%2 == 1 {
				return math.Max(left1, left2)
			}

			// Even total length
			return (
				math.Max(left1, left2) +
				math.Min(right1, right2),
			) / 2.0
		}

		// Too many elements from nums1
		if left1 > right2 {
			right = partition1 - 1
		} else {
			// Too few elements from nums1
			left = partition1 + 1
		}
	}

	return 0.0
}
```

The Go implementation follows the same logic as Python, but there are some language-specific differences. Go does not have built-in infinity literals like Python, so `math.Inf(-1)` and `math.Inf(1)` are used for sentinel values. Since median calculations involve division, values are converted to `float64` before arithmetic to avoid integer truncation. Go slices naturally handle empty arrays, so no special handling for `nil` versus empty slices is required.

## Edge Cases

### One Array Is Empty

A common source of bugs occurs when one array has length `0`. For example:

```
nums1 = []
nums2 = [1,2,3]
```

Naive implementations may fail because they assume both arrays contain elements. This solution handles the case naturally using sentinel values. If a partition reaches an array boundary, `-inf` and `inf` ensure comparisons still work correctly. The algorithm effectively computes the median directly from the non-empty array.

### Arrays With Very Different Sizes

Consider:

```
nums1 = [1]
nums2 = [2,3,4,5,6,7]
```

Partition calculations can become error-prone when array lengths differ significantly. Searching the smaller array prevents out-of-bounds partition indices and keeps binary search efficient.

### Even Total Length

When the total number of elements is even, the median is not a single number but the average of the two middle elements. For example:

```
[1,2,3,4]
```

has median:

```
(2 + 3) / 2
```

A common mistake is returning only one middle element. The implementation explicitly checks parity and computes the average of the maximum left value and minimum right value.

### Duplicate Values

Inputs such as:

```
nums1 = [1,2,2]
nums2 = [2,2,3]
```

can expose incorrect comparison logic. Since the algorithm uses `<=` rather than strict `<`, duplicate boundary values are handled correctly and still produce a valid partition.

### Negative Numbers

Values may be negative:

```
nums1 = [-5,-3]
nums2 = [-2,-1]
```

Because the algorithm only depends on ordering comparisons, negative values work naturally. Sentinel infinities ensure comparisons remain valid even near array boundaries.
