## Solution

## Problem Understanding

The problem asks us to simulate an alternating deletion process on a sequence of integers from `1` to `n` until only one integer remains. Specifically, we perform deletions in two alternating directions: first from the left, removing every second number, then from the right, again removing every second number, and repeat. The input `n` represents the number of integers in the initial sequence. The output is the last integer remaining after repeatedly applying these deletion operations.

Constraints indicate that `n` can be as large as `10^15`, which is far too large for a naive simulation using arrays or lists. This forces us to reason mathematically or recursively about the sequence rather than explicitly storing and removing elements. Important edge cases include `n = 1` (trivial single-element sequence), small odd numbers like `n = 5` where deletion patterns from left and right alternate asymmetrically, and very large `n` to ensure efficiency.

## Approaches

### Brute Force

The brute-force approach would be to generate the sequence `[1, 2, ..., n]` in an array or list, then repeatedly remove every second element from the left, then from the right, alternating until one element remains. This approach is correct for small `n` because it explicitly follows the process, but it requires `O(n)` space and roughly `O(n log n)` or worse time, because each deletion operation shrinks the array but still touches many elements. For `n = 10^15`, this is completely infeasible.

### Optimal Approach

The key insight is that the sequence follows a predictable pattern that can be expressed recursively or iteratively. Let `f(n, left)` denote the last remaining number for a sequence of length `n` with the next deletion starting from the `left` (True if starting from left, False if from right). Observing the pattern:

- If we delete from the left, the first number always moves, so `f(n, True) = 2 * f(n // 2, False)`.
- If we delete from the right, the first number only moves if `n` is odd, so `f(n, False) = 2 * f(n // 2, True) - 1 + (n % 2)`.

This allows a recursive or iterative solution in `O(log n)` time without storing the sequence, since each step halves `n`.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) | O(n) | Simulate deletions explicitly |
| Optimal | O(log n) | O(1) | Recursive/iterative computation using mathematical pattern |

## Algorithm Walkthrough

1. Start with `head = 1`, representing the first element of the current sequence.
2. Track `step = 1`, the interval between elements in the current effective sequence.
3. Track `remaining = n`, the number of elements remaining.
4. Track a boolean `left = True` to indicate the next deletion direction.
5. While `remaining > 1`:

1. If deleting from left or `remaining` is odd, move `head` forward by `step`.
2. Double the `step` because each deletion removes half the elements.
3. Halve `remaining` to reflect the number of elements left.
4. Toggle `left` to switch deletion direction.
6. Return `head` as the last remaining integer.

**Why it works**: The algorithm works because each deletion pattern consistently moves the head of the sequence in a predictable way. By tracking only the head, step, remaining count, and deletion direction, we avoid storing the full sequence. The invariant is that `head` always points to the first element of the current sequence, and `step` is the distance between consecutive surviving numbers.

## Python Solution

```python
class Solution:
    def lastInteger(self, n: int) -> int:
        head = 1
        step = 1
        remaining = n
        left = True

        while remaining > 1:
            if left or remaining % 2 == 1:
                head += step
            step *= 2
            remaining //= 2
            left = not left

        return head
```

The Python implementation initializes `head` to 1. `step` keeps track of the gap between elements after each deletion. `remaining` tracks the number of integers left. In each iteration, if we delete from the left or the remaining count is odd (deleting from the right requires moving the head only if odd), we increment `head` by `step`. After each round, we double the step, halve the remaining count, and toggle the deletion direction. Finally, `head` holds the last remaining integer.

## Go Solution

```go
func lastInteger(n int64) int64 {
    head := int64(1)
    step := int64(1)
    remaining := n
    left := true

    for remaining > 1 {
        if left || remaining%2 == 1 {
            head += step
        }
        step *= 2
        remaining /= 2
        left = !left
    }

    return head
}
```

The Go version mirrors the Python solution with type `int64` to handle large `n` up to `10^15`. The logic is identical: track `head`, `step`, `remaining`, and `left`. The only differences are Go syntax and explicit integer type handling.

## Edge Cases

The first edge case is `n = 1`. Here, the sequence has only one element. Our algorithm correctly returns `head = 1` immediately without entering the loop. The second edge case is `n = 2`. Deleting from the left removes the second element, leaving `2` as the last integer. The third edge case is sequences where `n` is odd. Deletion from the right only moves the `head` if `remaining` is odd; this subtlety ensures correctness for sequences like `n = 5` or `n = 7`. By handling the `remaining % 2 == 1` condition, the algorithm accommodates both odd and even sequences seamlessly.
