# Count Subarrays With Majority Element I

## Problem Understanding

We are given an integer array `nums` and a value `target`. We must count how many contiguous subarrays have `target` as their majority element.

A majority element in a subarray means that it appears **strictly more than half** of the subarray's length. For a subarray of length `L`, the frequency of `target` must satisfy:

$$\text{count(target)} > \frac{L}{2}$$

The task is not to find the majority element of the entire array. Instead, we must examine every possible contiguous subarray and count only those where `target` is the majority.

For example, in `[1,2,2,3]` with `target = 2`, the subarray `[2,2]` has length 2 and contains two occurrences of `2`, so `2 > 2/2`, making it valid. The subarray `[1,2,2]` has length 3 and contains two occurrences of `2`, so `2 > 3/2`, making it valid as well.

The constraints are:

- `1 <= nums.length <= 1000`
- `1 <= nums[i] <= 10^9`
- `1 <= target <= 10^9`

The array length is relatively small, only up to 1000. This means an `O(n²)` solution is feasible, while `O(n³)` becomes too expensive.

Several edge cases are important:

- The target may not appear at all. In that case the answer is immediately 0.
- Every element may equal the target. Then every subarray is valid.
- A subarray may contain the target exactly half the time. This does **not** qualify because the definition requires strictly more than half.
- Single-element subarrays containing the target are always valid because `1 > 1/2`.

## Approaches

### Brute Force

The most direct solution is to enumerate every subarray.

For each starting index `i` and ending index `j`, we compute:

- Length of the subarray.
- Number of occurrences of `target`.

If the target frequency is greater than half of the subarray length, we increment the answer.

A naive implementation would scan the subarray each time to count occurrences, leading to `O(n³)` complexity:

- `O(n²)` subarrays.
- `O(n)` work to count elements inside each subarray.

This is correct but unnecessarily slow.

### Key Observation

The majority condition can be transformed into a prefix-sum problem.

Define a transformed array:

- `+1` if `nums[i] == target`
- `-1` otherwise

For any subarray:

\text{sum} = (\text{#target}) - (\text{#non-target})

Suppose:

- `t = #target`
- `o = #non-target`

Then:

$$\text{length} = t + o$$

Target is a majority iff:

$$t > \frac{t+o}{2}$$

Multiplying by 2:

$$2t > t + o$$

$$t > o$$

$$t - o > 0$$

But the transformed subarray sum is exactly:

$$t - o$$

Therefore:

**A subarray has target as majority iff its transformed sum is positive.**

Now the problem becomes:

> Count subarrays whose transformed sum is greater than zero.

Using prefix sums:

$$prefix[j] - prefix[i] > 0$$

which is equivalent to:

$$prefix[i] < prefix[j]$$

So we need to count pairs of prefix sums where an earlier prefix is strictly smaller than a later prefix.

This becomes a classic counting problem solvable with merge sort inversion-style counting.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n³) | O(1) | Check every subarray and recount target occurrences |
| Optimal | O(n log n) | O(n) | Transform into counting prefix-sum pairs using merge sort |

## Algorithm Walkthrough

1. Create a transformed array where each element becomes:

- `+1` if it equals `target`
- `-1` otherwise.
2. Build a prefix sum array of length `n + 1`.

The first prefix value is `0`.

If the transformed array is:

```
[1, -1, 1]
```

then prefixes become:

```
[0, 1, 0, 1]
```
3. For every subarray `(l, r)`:

```
subarray_sum = prefix[r+1] - prefix[l]
```

The subarray is valid when:

```
subarray_sum > 0
```
4. Rearrange the condition:

```
prefix[l] < prefix[r+1]
```
5. Therefore count all ordered pairs `(i, j)` with:

```
i < j
prefix[i] < prefix[j]
```
6. Use divide and conquer merge sort.

During merging:

- Left half and right half are already sorted.
- For each right-side value, count how many left-side values are strictly smaller.
- Add that amount to the answer.
7. Merge the two sorted halves and continue recursively.
8. Return the total count.

### Why it works

The transformation converts every occurrence of `target` into `+1` and every other value into `-1`. The sum of a subarray then equals:

(\text{#target}) - (\text{#non-target})

A positive sum means the target appears more often than all non-target values combined, which is exactly the definition of a majority element. Counting positive-sum subarrays is equivalent to counting pairs of prefix sums where the earlier prefix is smaller than the later prefix. The merge sort counting procedure counts all such pairs exactly once, so the final result is correct.

## Python Solution

```python
from typing import List

class Solution:
    def countMajoritySubarrays(self, nums: List[int], target: int) -> int:
        n = len(nums)

        prefix = [0]
        current = 0

        for num in nums:
            if num == target:
                current += 1
            else:
                current -= 1
            prefix.append(current)

        def merge_sort(arr: List[int]) -> int:
            length = len(arr)

            if length <= 1:
                return 0

            mid = length // 2

            left = arr[:mid]
            right = arr[mid:]

            count = merge_sort(left) + merge_sort(right)

            j = 0
            left_size = len(left)

            for value in right:
                while j < left_size and left[j] < value:
                    j += 1
                count += j

            i = 0
            j = 0
            k = 0

            while i < len(left) and j < len(right):
                if left[i] <= right[j]:
                    arr[k] = left[i]
                    i += 1
                else:
                    arr[k] = right[j]
                    j += 1
                k += 1

            while i < len(left):
                arr[k] = left[i]
                i += 1
                k += 1

            while j < len(right):
                arr[k] = right[j]
                j += 1
                k += 1

            return count

        return merge_sort(prefix)
```

### Implementation Explanation

The first section constructs the transformed prefix sums. Every occurrence of `target` contributes `+1`, while every other element contributes `-1`.

The recursive `merge_sort` function performs two tasks simultaneously:

- Sort the prefix sums.
- Count valid prefix pairs.

After recursively sorting both halves, the left and right arrays are individually sorted. For each value in the right half, we count how many values in the left half are strictly smaller. Since both halves are sorted, a single advancing pointer is sufficient.

The accumulated count equals the number of pairs `(i, j)` such that:

$$i < j,\quad prefix[i] < prefix[j]$$

Finally, a standard merge step restores sorted order for higher recursion levels.

## Go Solution

```go
func countMajoritySubarrays(nums []int, target int) int {
	prefix := make([]int, len(nums)+1)

	cur := 0
	for i, num := range nums {
		if num == target {
			cur++
		} else {
			cur--
		}
		prefix[i+1] = cur
	}

	var mergeSort func([]int) int

	mergeSort = func(arr []int) int {
		if len(arr) <= 1 {
			return 0
		}

		mid := len(arr) / 2

		left := append([]int(nil), arr[:mid]...)
		right := append([]int(nil), arr[mid:]...)

		count := mergeSort(left) + mergeSort(right)

		j := 0
		for _, value := range right {
			for j < len(left) && left[j] < value {
				j++
			}
			count += j
		}

		i, k := 0, 0
		j = 0

		for i < len(left) && j < len(right) {
			if left[i] <= right[j] {
				arr[k] = left[i]
				i++
			} else {
				arr[k] = right[j]
				j++
			}
			k++
		}

		for i < len(left) {
			arr[k] = left[i]
			i++
			k++
		}

		for j < len(right) {
			arr[k] = right[j]
			j++
			k++
		}

		return count
	}

	return mergeSort(prefix)
}
```

### Go-Specific Notes

The logic is identical to the Python version. Since Go slices share backing arrays, copies of the left and right halves are created before recursion. The answer comfortably fits inside a Go `int` because the maximum number of subarrays is:

$$\frac{1000 \times 1001}{2} = 500500$$

which is far below integer limits.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | Merge sort counting over `n + 1` prefix sums |
| Space | O(n) | Recursion and temporary merge arrays |

The transformed prefix array contains `n + 1` elements. The divide-and-conquer counting process performs the same amount of work as merge sort, resulting in `O(n log n)` time. Additional temporary arrays created during merging require `O(n)` extra space.

## Edge Cases

### Target Does Not Appear

If the target never appears, every transformed value becomes `-1`. All prefix sums form a strictly decreasing sequence. There can never be a pair where an earlier prefix is smaller than a later prefix, so the answer is correctly computed as zero.

### Target Appears Exactly Half the Time

The definition requires a strict majority. For example, in subarray `[2,1]`, the target appears once in a length-two subarray. Since `1` is not greater than `2/2`, the subarray is invalid.

The transformed sum becomes:

```
+1 + (-1) = 0
```

Our algorithm counts only pairs with:

```
prefix[i] < prefix[j]
```

which corresponds to positive subarray sums. Zero sums are excluded automatically.

### Every Element Is the Target

When all elements equal the target, every transformed value is `+1`. Prefix sums become strictly increasing. Every pair of prefix sums contributes to the count, producing exactly:

$$\frac{n(n+1)}{2}$$

which equals the total number of subarrays. The implementation naturally handles this case without any special logic.

### Single-Element Arrays

A single-element subarray containing the target is always valid because one occurrence is more than half of one. If the lone element is not the target, no valid subarray exists. The prefix-sum formulation correctly produces answers of `1` and `0` respectively.
