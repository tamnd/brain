# LeetCode 3791 - Number of Balanced Integers in a Range

## Problem Understanding

The problem asks us to count how many integers in the inclusive range `[low, high]` are **balanced**.

A number is considered balanced if it satisfies two requirements:

1. It contains at least two digits.
2. The sum of digits at odd positions is equal to the sum of digits at even positions, where positions are counted from left to right and the leftmost digit is position `1`.

For example:

- `121` is balanced because:

- Odd positions: `1 + 1 = 2`
- Even positions: `2`
- Both sums are equal.
- `1234` is not balanced because:

- Odd positions: `1 + 3 = 4`
- Even positions: `2 + 4 = 6`

The input consists of two integers, `low` and `high`, and we must return the number of balanced integers in that range.

The most important constraint is:

- `1 <= low <= high <= 10^15`

This means numbers can have up to 16 digits. A range could contain trillions of values, making it impossible to check every number individually.

Several edge cases are important:

- Single digit numbers are never balanced because the problem requires at least two digits.
- Numbers with different digit lengths must be handled correctly.
- The parity of a position depends on the total digit layout from the left side, so leading zeros in a digit-DP state must be treated carefully.
- The upper bound can be as large as `10^15`, so any solution must work efficiently for 16-digit numbers.

## Approaches

### Brute Force

The most direct solution is to iterate through every number in `[low, high]`.

For each number:

1. Convert it to a string.
2. Skip it if it contains fewer than two digits.
3. Compute the sum of digits at odd positions.
4. Compute the sum of digits at even positions.
5. Count it if the sums are equal.

This approach is obviously correct because it directly checks the definition of a balanced number.

However, it is far too slow. The range can be nearly `10^15` numbers wide, making enumeration impossible.

### Optimal Approach: Digit DP

The key observation is that we are not interested in individual numbers. We only care about the digit sums at odd and even positions.

Whenever a problem asks:

- Count numbers in a range.
- Subject to digit-related constraints.
- Upper bound up to `10^15` or larger.

Digit DP is usually the appropriate technique.

Instead of enumerating numbers, we count how many valid numbers can be formed digit by digit.

Define:

- `F(x)` = number of balanced integers in `[1, x]`.

Then the answer is:

$$F(high) - F(low - 1)$$

While constructing a number digit by digit, we keep track of:

- Current digit position.
- Whether we are still limited by the upper bound.
- Whether a non-leading-zero digit has appeared.
- Number length already determined.
- Difference between odd-position sum and even-position sum.

The tricky part is that positions are counted from the leftmost actual digit, not including leading zeros.

To handle this cleanly, we process numbers of a fixed length. For a given length `L`, the parity of every position is known in advance:

- Position 1 is odd.
- Position 2 is even.
- Position 3 is odd.
- And so on.

Digit DP then counts all valid numbers of length `L` whose final difference equals zero.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O((high - low + 1) × D) | O(1) | Checks every number individually |
| Optimal Digit DP | O(D × S × 10) | O(D × S) | Counts valid numbers without enumeration |
|  |  |  | `D ≤ 16`, `S ≤ 144` |

Here `S` denotes the possible digit-sum difference range.

## Algorithm Walkthrough

We define a helper function `count_up_to(limit)` that returns the number of balanced integers not exceeding `limit`.

For each possible digit length from `2` up to the length of `limit`, we count balanced numbers of exactly that length.

### 1. Convert the limit into digits

Store the decimal representation of `limit`.

This allows us to perform digit DP while respecting the upper bound.

### 2. Handle shorter lengths

Any number whose digit length is smaller than the length of `limit` is automatically below the limit.

For these lengths, we can run an unrestricted digit DP.

### 3. Handle the maximum length

For numbers having the same length as `limit`, we use a tight constraint so that generated numbers never exceed the limit.

### 4. Define DP state

For a fixed length `L`, define:

`dfs(pos, diff, tight)`

where:

- `pos` is the current digit index.
- `diff` is:

$$\text{oddSum} - \text{evenSum}$$

- `tight` indicates whether the constructed prefix is equal to the limit prefix.

### 5. Place a digit

At position `pos`, choose a digit.

If the position is odd:

$$diff += digit$$

Otherwise:

$$diff -= digit$$

### 6. Move to the next position

Continue recursively until all positions are assigned.

### 7. Check validity

When all digits have been processed:

- Return `1` if `diff == 0`.
- Otherwise return `0`.

### 8. Compute the final answer

Use:

$$\text{answer} = count\_up\_to(high) - count\_up\_to(low-1)$$

### Why it works

For every digit length, the DP enumerates all valid digit combinations exactly once. The state records the complete information needed to determine whether the final odd-position sum equals the final even-position sum. Since every valid number corresponds to exactly one path through the DP and every path represents exactly one number, the count returned by the DP is exactly the number of balanced integers.

## Python Solution

```python
from functools import lru_cache

class Solution:
    def countBalanced(self, low: int, high: int) -> int:
        def count_up_to(limit: int) -> int:
            if limit < 10:
                return 0

            digits = list(map(int, str(limit)))
            n = len(digits)

            def count_length(length: int, bound_digits=None) -> int:
                @lru_cache(None)
                def dfs(pos: int, diff: int, tight: bool) -> int:
                    if pos == length:
                        return 1 if diff == 0 else 0

                    upper = (
                        bound_digits[pos]
                        if tight and bound_digits is not None
                        else 9
                    )

                    total = 0

                    lower = 1 if pos == 0 else 0

                    for digit in range(lower, upper + 1):
                        next_tight = (
                            tight
                            and bound_digits is not None
                            and digit == upper
                        )

                        if (pos + 1) % 2 == 1:
                            total += dfs(
                                pos + 1,
                                diff + digit,
                                next_tight
                            )
                        else:
                            total += dfs(
                                pos + 1,
                                diff - digit,
                                next_tight
                            )

                    return total

                return dfs(0, 0, True)

            result = 0

            for length in range(2, n):
                result += count_length(length)

            result += count_length(n, digits)

            return result

        return count_up_to(high) - count_up_to(low - 1)
```

The implementation follows the algorithm directly.

The outer function `count_up_to()` computes how many balanced integers are at most a given limit. Any limit below `10` immediately returns zero because balanced numbers require at least two digits.

For each digit length, a separate digit DP is executed. The state stores the current position, the difference between odd and even digit sums, and whether the constructed prefix remains equal to the bound.

The first digit is forced to be nonzero, ensuring that every generated number has exactly the intended length. When all positions have been assigned, the number is balanced precisely when the difference equals zero.

Finally, the standard digit-DP range-counting formula is applied:

$$F(high) - F(low - 1)$$

## Go Solution

```go
package main

import "strconv"

func countBalanced(low int64, high int64) int64 {
	countUpTo := func(limit int64) int64 {
		if limit < 10 {
			return 0
		}

		digitsStr := strconv.FormatInt(limit, 10)
		n := len(digitsStr)

		var countLength func(length int, bound []int) int64

		countLength = func(length int, bound []int) int64 {
			type State struct {
				Pos   int
				Diff  int
				Tight bool
			}

			memo := make(map[State]int64)

			var dfs func(pos int, diff int, tight bool) int64

			dfs = func(pos int, diff int, tight bool) int64 {
				if pos == length {
					if diff == 0 {
						return 1
					}
					return 0
				}

				state := State{pos, diff, tight}
				if val, ok := memo[state]; ok {
					return val
				}

				upper := 9
				if tight && bound != nil {
					upper = bound[pos]
				}

				lower := 0
				if pos == 0 {
					lower = 1
				}

				var total int64

				for digit := lower; digit <= upper; digit++ {
					nextTight := false

					if tight && bound != nil && digit == upper {
						nextTight = true
					}

					if (pos+1)%2 == 1 {
						total += dfs(pos+1, diff+digit, nextTight)
					} else {
						total += dfs(pos+1, diff-digit, nextTight)
					}
				}

				memo[state] = total
				return total
			}

			return dfs(0, 0, true)
		}

		var result int64

		for length := 2; length < n; length++ {
			result += countLength(length, nil)
		}

		bound := make([]int, n)
		for i := 0; i < n; i++ {
			bound[i] = int(digitsStr[i] - '0')
		}

		result += countLength(n, bound)

		return result
	}

	return countUpTo(high) - countUpTo(low-1)
}
```

The Go implementation mirrors the Python solution. A struct is used as the memoization key because Go does not support tuple hashing directly. The result type is `int64` because the count can exceed the range of a 32-bit integer.

The difference value ranges from approximately `-144` to `144` because a 16-digit number can contribute at most `16 × 9 = 144` to either side. Consequently, the total number of DP states is quite small, making the solution extremely efficient.

## Edge Cases

### Single-Digit Numbers

A common mistake is to count numbers such as `5` or `8` as balanced because their odd-position sum equals their even-position sum plus zero. The problem explicitly requires at least two digits. The implementation handles this by returning zero whenever the limit is below `10` and only processing lengths starting from `2`.

### Numbers With Different Lengths

The range may span multiple digit lengths, for example from `90` to `10000`. Position parity depends on the leftmost digit of each number, so a three-digit number and a four-digit number cannot be treated identically. The implementation processes each length separately, guaranteeing correct position assignments.

### Upper-Bound Restriction

When counting numbers up to a limit such as `123456`, it is easy to accidentally count numbers larger than the limit. The `tight` flag ensures that once a chosen digit becomes smaller than the corresponding limit digit, subsequent positions may use any digit, but as long as the prefix remains equal, future choices remain restricted. This guarantees that every counted number is less than or equal to the limit.

### Large Digit-Sum Differences

For a 16-digit number, the difference between odd and even sums can become quite large. A naive fixed-size array may be indexed incorrectly or overflow its bounds. The memoized DP stores the difference directly as part of the state, avoiding offset-management bugs and naturally handling the full range of possible values.

### Exact Boundary Queries

Queries such as `low = high = 121` require correctly determining whether a single number qualifies. The range-counting formula `F(high) - F(low - 1)` naturally handles this situation, reducing the problem to counting numbers up to two adjacent limits.
