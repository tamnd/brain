# LeetCode 3739 - Count Subarrays With Majority Element II

## Problem Understanding

We are given an array `nums` and a value `target`. We must count how many contiguous subarrays have `target` as their majority element.

A majority element in a subarray means that it appears strictly more than half of the subarray's length. If a subarray has length `L`, then `target` must appear more than `L / 2` times.

The input array can contain up to `100,000` elements, which immediately tells us that any algorithm that examines every subarray explicitly will be far too slow. Since there are `O(n²)` possible subarrays, we need a solution that is close to linear or `O(n log n)`.

The key challenge is that majority is defined relative to the length of each subarray. Naively counting occurrences of `target` inside every possible interval would require too much work.

Several edge cases are important:

- The target may not appear in the array at all. In that case, no valid subarray exists.
- Every element may equal the target. Then every subarray is valid.
- The array may alternate between target and non-target values, creating many borderline cases where the target appears exactly half the time, which must **not** be counted because the definition requires strictly more than half.
- The answer can be as large as `n(n+1)/2`, which is roughly `5 × 10^9` when `n = 100000`, so 64-bit arithmetic is required.

## Approaches

### Brute Force

The most direct solution is to examine every subarray.

For every starting position `i`, we extend the ending position `j` and maintain the number of occurrences of `target`. Let:

- `targetCount` = occurrences of `target`
- `length = j - i + 1`

The subarray is valid if:

$$targetCount > \frac{length}{2}$$

This approach is correct because it explicitly checks the definition for every subarray.

However, there are `O(n²)` subarrays. Even if we maintain counts incrementally, the overall complexity remains `O(n²)`, which is far too slow for `n = 100000`.

### Key Observation

Instead of working directly with occurrence counts, we can transform the problem.

Define a new array:

- `+1` if `nums[i] == target`
- `-1` otherwise

Consider a subarray.

Let:

- `T` = number of target elements
- `O` = number of non-target elements

The transformed subarray sum becomes:

$$T - O$$

The target is a majority element exactly when:

$$T > O$$

which is equivalent to:

$$T - O > 0$$

Therefore, the original problem becomes:

**Count the number of subarrays whose transformed sum is strictly positive.**

Now introduce prefix sums.

Let:

$$prefix[i]$$

be the transformed prefix sum up to position `i`.

A subarray `(l, r)` has positive sum when:

$$prefix[r+1] - prefix[l] > 0$$

which rearranges to:

$$prefix[r+1] > prefix[l]$$

So the problem becomes:

**Count pairs of prefix sums `(i, j)` with `i < j` and `prefix[j] > prefix[i]`.**

This is a classic order-statistics problem that can be solved efficiently using a Fenwick Tree (Binary Indexed Tree).

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Check every subarray directly |
| Optimal | O(n log n) | O(n) | Transform to positive-sum subarray counting and use Fenwick Tree |

## Algorithm Walkthrough

1. Create a transformed array where each element becomes:

- `+1` if it equals `target`
- `-1` otherwise
2. Build prefix sums. Start with `prefix = 0`. For every transformed value, add it to the running prefix sum.
3. Observe that a subarray has target as majority exactly when its transformed sum is positive.
4. Using prefix sums, a positive subarray sum corresponds to:

$$prefix[j] > prefix[i]$$

for some earlier prefix sum `prefix[i]`.
5. Therefore, while scanning prefix sums from left to right, we need to know how many previously seen prefix sums are strictly smaller than the current one.
6. Prefix sums can be negative, so we coordinate-compress all possible prefix values.
7. Use a Fenwick Tree to maintain frequencies of previously seen compressed prefix sums.
8. For each prefix sum:

- Query how many previous prefix sums are strictly smaller.
- Add that count to the answer.
- Insert the current prefix sum into the Fenwick Tree.
9. After processing all prefix sums, return the accumulated answer.

### Why it works

After transformation, every target contributes `+1` and every non-target contributes `-1`. A subarray has target as majority precisely when the transformed sum is positive. Prefix sums convert subarray sums into differences of prefixes. Therefore, counting valid subarrays becomes counting pairs of prefix sums where the later prefix is larger than the earlier prefix. The Fenwick Tree efficiently counts how many earlier prefix sums are smaller than the current one, ensuring every valid subarray is counted exactly once.

## Python Solution

```python
from typing import List

class Fenwick:
    def __init__(self, size: int):
        self.tree = [0] * (size + 1)

    def update(self, index: int, delta: int) -> None:
        n = len(self.tree)
        while index < n:
            self.tree[index] += delta
            index += index & -index

    def query(self, index: int) -> int:
        result = 0
        while index > 0:
            result += self.tree[index]
            index -= index & -index
        return result

class Solution:
    def countMajoritySubarrays(self, nums: List[int], target: int) -> int:
        prefix_sums = [0]
        current = 0

        for value in nums:
            current += 1 if value == target else -1
            prefix_sums.append(current)

        values = sorted(set(prefix_sums))
        rank = {value: i + 1 for i, value in enumerate(values)}

        fenwick = Fenwick(len(values))
        answer = 0

        for prefix in prefix_sums:
            idx = rank[prefix]

            answer += fenwick.query(idx - 1)

            fenwick.update(idx, 1)

        return answer
```

The implementation begins by constructing the transformed prefix sums. Every occurrence of `target` contributes `+1`, while every other value contributes `-1`.

Next, coordinate compression maps all distinct prefix sums into a compact range `[1, m]`, allowing efficient Fenwick Tree operations.

The Fenwick Tree stores frequencies of prefix sums encountered so far. For each current prefix sum, querying `idx - 1` returns the number of earlier prefix sums that are strictly smaller. Each such prefix forms a valid subarray ending at the current position.

Finally, the current prefix sum is inserted into the Fenwick Tree so it can participate in future counts.

## Go Solution

```go
package main

import "sort"

type Fenwick struct {
	tree []int64
}

func NewFenwick(size int) *Fenwick {
	return &Fenwick{
		tree: make([]int64, size+1),
	}
}

func (f *Fenwick) Update(index int, delta int64) {
	for index < len(f.tree) {
		f.tree[index] += delta
		index += index & -index
	}
}

func (f *Fenwick) Query(index int) int64 {
	var result int64

	for index > 0 {
		result += f.tree[index]
		index -= index & -index
	}

	return result
}

func countMajoritySubarrays(nums []int, target int) int64 {
	prefixSums := make([]int, 1)
	current := 0

	for _, value := range nums {
		if value == target {
			current++
		} else {
			current--
		}
		prefixSums = append(prefixSums, current)
	}

	values := append([]int(nil), prefixSums...)
	sort.Ints(values)

	unique := make([]int, 0)
	for _, v := range values {
		if len(unique) == 0 || unique[len(unique)-1] != v {
			unique = append(unique, v)
		}
	}

	rank := make(map[int]int)
	for i, v := range unique {
		rank[v] = i + 1
	}

	fenwick := NewFenwick(len(unique))

	var answer int64

	for _, prefix := range prefixSums {
		idx := rank[prefix]

		answer += fenwick.Query(idx - 1)

		fenwick.Update(idx, 1)
	}

	return answer
}
```

The Go solution follows exactly the same logic as the Python version. The primary difference is that the answer and Fenwick frequencies use `int64`, because the number of valid subarrays can exceed the range of a 32-bit integer. Coordinate compression is implemented using sorting and deduplication of the prefix sums.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | Coordinate compression plus Fenwick operations |
| Space | O(n) | Prefix sums, compression map, and Fenwick Tree |

The algorithm processes `n + 1` prefix sums. Each Fenwick Tree update and query takes `O(log n)` time. Coordinate compression requires sorting at most `n + 1` values, which also costs `O(n log n)`. Therefore the total running time is `O(n log n)`, and the auxiliary memory usage is `O(n)`.

## Edge Cases

### Target Does Not Appear

If `target` never appears, every transformed value becomes `-1`. Every subarray sum is therefore non-positive, and no valid subarray exists. The prefix-sum formulation naturally produces zero increasing prefix pairs, yielding the correct answer without requiring special handling.

### Target Appears Exactly Half the Time

A common mistake is counting subarrays where the target appears at least half the time. The problem requires strictly more than half. In the transformed representation, exactly-half cases produce a subarray sum of zero. Since the algorithm counts only pairs satisfying `prefix[j] > prefix[i]`, zero-sum intervals are correctly excluded.

### All Elements Equal Target

When every element equals `target`, every transformed value is `+1`. The prefix sums become strictly increasing. Every pair of prefix sums contributes to the answer, which correctly counts all possible subarrays. The Fenwick Tree efficiently handles this case and returns `n(n+1)/2`.

### Very Large Answers

For `n = 100000`, the total number of subarrays is approximately five billion. This exceeds the range of a 32-bit integer. The Go solution uses `int64`, and Python integers automatically expand as needed, ensuring correct results for the largest inputs.
