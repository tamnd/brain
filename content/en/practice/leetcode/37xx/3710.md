## Solution

## Problem Understanding

We are given `n` points on a 2D plane. The distance between any two points is measured using Manhattan distance:

$$|x_i - x_j| + |y_i - y_j|$$

We must divide all points into exactly two non-empty groups. For a particular split, we look at every pair of points that ends up in the same group. Among those intra-group pairs, we take the minimum Manhattan distance. That value is called the partition factor of the split.

Our goal is to choose the split that maximizes this partition factor.

Another way to think about the problem is that we want points that are "too close" to each other to end up in different groups. The larger the partition factor, the larger the minimum distance we successfully enforce inside both groups.

The constraints are important:

- `2 <= n <= 500`
- Coordinates can be as large as `10^8` in magnitude.

Since `n = 500`, there are at most:

$$\binom{500}{2} = 124,750$$

distinct point pairs. This strongly suggests that an $O(n^2)$ preprocessing or checking step is acceptable, but enumerating all possible partitions is impossible.

A few important edge cases immediately stand out.

When `n = 2`, the only valid split places one point in each group. Neither group contains an intra-group pair, so the problem explicitly defines the answer as `0`.

When many points are very close together, a large candidate partition factor may become impossible because too many pairs must be separated simultaneously.

When there are no distance restrictions for a candidate value, the graph constructed by the optimal solution contains no edges. We must still ensure that two non-empty groups can be formed.

## Approaches

### Brute Force

A brute-force solution would enumerate every possible partition of the points into two non-empty groups.

There are $2^n$ possible assignments of points to groups. For each partition, we would compute the minimum Manhattan distance among all intra-group pairs and keep the best result.

This approach is correct because it examines every valid split. However, it is completely infeasible.

Even for `n = 30`, the number of partitions is already enormous. For `n = 500`, this approach is impossible.

### Key Insight

Suppose we want to know whether a partition factor of at least `D` is achievable.

If two points have Manhattan distance less than `D`, they cannot belong to the same group. Otherwise the minimum intra-group distance would become smaller than `D`.

Therefore:

- Create a graph whose vertices are points.
- Connect two vertices if their Manhattan distance is less than `D`.

Every edge represents a pair of points that must be placed in different groups.

Now the question becomes:

> Can we assign every vertex to one of two groups so that every edge connects vertices in different groups?

This is exactly the definition of a bipartite graph.

Therefore:

- `D` is feasible if and only if the graph formed by all pairs with distance `< D` is bipartite.
- Feasibility is monotonic.

- If a large `D` is feasible, every smaller value is also feasible.
- If a value is infeasible, every larger value is also infeasible.

Monotonicity allows binary search on the answer.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | $O(2^n \cdot n^2)$ | $O(n)$ | Enumerates all partitions |
| Optimal | $O(n^2 \log M)$ | $O(n^2)$ | Binary search + bipartite checking |

Here `M` is the range of possible Manhattan distances, at most about `4 * 10^8`.

## Algorithm Walkthrough

### Step 1: Handle the special case `n = 2`

The statement explicitly defines the answer as `0` because both groups are singletons and no intra-group pair exists.

### Step 2: Precompute all pairwise Manhattan distances

For every pair `(i, j)`, compute:

$$|x_i - x_j| + |y_i - y_j|$$

Store these distances in a matrix.

This allows every feasibility check to reuse previously computed distances.

### Step 3: Binary search the answer

Let:

- `low = 0`
- `high = maxDistance + 1`

We binary search for the largest feasible value.

### Step 4: Check whether a candidate value `D` is feasible

Construct the graph implicitly:

- Vertex `i` represents point `i`.
- There is an edge `(i, j)` if `distance[i][j] < D`.

Instead of explicitly building adjacency lists, we simply scan the distance matrix whenever needed.

### Step 5: Test bipartiteness

Maintain a color array:

- `-1` = unvisited
- `0` and `1` = two graph colors

For every unvisited vertex:

- Start a BFS or DFS.
- Assign an initial color.
- Whenever an edge is encountered, force the opposite color.

If an edge ever connects two vertices with the same color, the graph is not bipartite and `D` is infeasible.

Otherwise `D` is feasible.

### Step 6: Update the binary search

If `D` is feasible:

- Record it as a candidate answer.
- Search larger values.

Otherwise:

- Search smaller values.

### Step 7: Return the largest feasible value

The binary search eventually finds the maximum partition factor.

### Why it works

For a candidate value `D`, every pair of points whose distance is less than `D` must belong to different groups. Representing such pairs as graph edges transforms the problem into a two-coloring problem.

A valid partition exists exactly when all these constraints can be satisfied simultaneously, which is equivalent to the graph being bipartite.

Because increasing `D` only adds more constraints, feasibility is monotonic. Binary search therefore correctly finds the largest feasible value. Since the bipartite test exactly captures the partition requirements, the returned value is the maximum possible partition factor.

## Python Solution

```python
from typing import List
from collections import deque

class Solution:
    def maxPartitionFactor(self, points: List[List[int]]) -> int:
        n = len(points)

        if n == 2:
            return 0

        dist = [[0] * n for _ in range(n)]
        max_dist = 0

        for i in range(n):
            x1, y1 = points[i]
            for j in range(i + 1, n):
                x2, y2 = points[j]
                d = abs(x1 - x2) + abs(y1 - y2)
                dist[i][j] = d
                dist[j][i] = d
                max_dist = max(max_dist, d)

        def feasible(limit: int) -> bool:
            color = [-1] * n

            for start in range(n):
                if color[start] != -1:
                    continue

                color[start] = 0
                queue = deque([start])

                while queue:
                    u = queue.popleft()

                    for v in range(n):
                        if u == v or dist[u][v] >= limit:
                            continue

                        if color[v] == -1:
                            color[v] = color[u] ^ 1
                            queue.append(v)
                        elif color[v] == color[u]:
                            return False

            return True

        left, right = 0, max_dist + 1
        answer = 0

        while left <= right:
            mid = (left + right) // 2

            if feasible(mid):
                answer = mid
                left = mid + 1
            else:
                right = mid - 1

        return answer
```

### Implementation Explanation

The first step handles the special case `n = 2`, which is defined by the problem to have answer `0`.

Next, all pairwise Manhattan distances are computed and stored in a matrix. Since every feasibility check repeatedly needs distance information, precomputing avoids recomputation.

The `feasible()` function performs the key graph test. Two vertices are connected whenever their distance is smaller than the candidate partition factor. Instead of explicitly storing edges, the function scans all vertices and checks the distance condition on demand.

A standard BFS-based bipartite check colors vertices using two colors. If an edge ever connects vertices of the same color, the candidate value is impossible.

The outer binary search uses the monotonic nature of feasibility. Whenever a value works, the search moves upward. Otherwise it moves downward. The largest feasible value is returned.

## Go Solution

```go
package main

func maxPartitionFactor(points [][]int) int {
	n := len(points)

	if n == 2 {
		return 0
	}

	dist := make([][]int, n)
	for i := range dist {
		dist[i] = make([]int, n)
	}

	maxDist := 0

	for i := 0; i < n; i++ {
		x1, y1 := points[i][0], points[i][1]

		for j := i + 1; j < n; j++ {
			x2, y2 := points[j][0], points[j][1]

			d := abs(x1-x2) + abs(y1-y2)

			dist[i][j] = d
			dist[j][i] = d

			if d > maxDist {
				maxDist = d
			}
		}
	}

	feasible := func(limit int) bool {
		color := make([]int, n)

		for i := 0; i < n; i++ {
			color[i] = -1
		}

		queue := make([]int, 0)

		for start := 0; start < n; start++ {
			if color[start] != -1 {
				continue
			}

			color[start] = 0
			queue = queue[:0]
			queue = append(queue, start)

			for head := 0; head < len(queue); head++ {
				u := queue[head]

				for v := 0; v < n; v++ {
					if u == v || dist[u][v] >= limit {
						continue
					}

					if color[v] == -1 {
						color[v] = color[u] ^ 1
						queue = append(queue, v)
					} else if color[v] == color[u] {
						return false
					}
				}
			}
		}

		return true
	}

	left, right := 0, maxDist+1
	answer := 0

	for left <= right {
		mid := left + (right-left)/2

		if feasible(mid) {
			answer = mid
			left = mid + 1
		} else {
			right = mid - 1
		}
	}

	return answer
}

func abs(x int) int {
	if x < 0 {
		return -x
	}
	return x
}
```

### Go-Specific Notes

The Go version follows exactly the same algorithm. A distance matrix is stored using a two-dimensional slice. BFS uses a slice-based queue with a moving head index, which avoids repeatedly removing elements from the front.

All distance calculations fit comfortably within Go's `int` type because the maximum Manhattan distance is at most `4 * 10^8`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | $O(n^2 \log M)$ | Each feasibility check scans all vertex pairs, binary search performs $O(\log M)$ checks |
| Space | $O(n^2)$ | Distance matrix storage |

The dominant cost comes from the bipartite test. For each binary search iteration we may inspect every pair of vertices. Since there are $O(n^2)$ pairs and approximately 30 binary search iterations, the total complexity is $O(n^2 \log M)$, which is easily fast enough for `n = 500`.

## Edge Cases

### Exactly Two Points

This is the only situation where every valid partition consists of two singleton groups. Since there are no intra-group pairs, the problem explicitly defines the answer as `0`. Without a special case, a graph-based solution would incorrectly consider arbitrarily large thresholds feasible.

### Duplicate Points

Two distinct points may occupy exactly the same coordinates, producing Manhattan distance `0`. For any positive candidate partition factor, those points must be separated. The bipartite graph naturally models this because an edge is added whenever the distance is smaller than the candidate threshold.

### Graph With No Edges

For very small candidate values, it is possible that no pair has distance below the threshold. The graph then contains only isolated vertices. Such a graph is bipartite, and a valid split always exists because `n > 2` guarantees that we can place at least one vertex into each group. The coloring algorithm correctly treats isolated vertices as independent bipartite components.

### Odd Cycle Formation

A candidate threshold may create an odd cycle among several points. In that situation, the separation constraints become contradictory because every edge requires opposite groups. The BFS coloring detects this immediately when it encounters an edge connecting two vertices of the same color, correctly marking the threshold as infeasible.
