# LeetCode 3756 - Concatenate Non-Zero Digits and Multiply by Sum II

## Problem Understanding

We are given a digit string `s` and many range queries. For each query `[l, r]`, we look only at the substring `s[l..r]`.

Inside that substring, we remove every zero and keep all non-zero digits in their original order. The remaining digits are concatenated to form an integer `x`.

For example:

- `"10203004"` becomes `"1234"`, so `x = 1234`
- `"020"` becomes `"2"`, so `x = 2`
- `"000"` contains no non-zero digits, so `x = 0`

After constructing `x`, we compute the sum of its digits. Since `x` consists exactly of the non-zero digits extracted from the substring, this digit sum is simply the sum of all non-zero digits inside the queried range.

The answer for a query is:

$$x \times \text{sumDigits}(x)$$

and the result must be returned modulo:

$$10^9 + 7$$

The constraints are very important:

- `|s| ≤ 100000`
- `queries.length ≤ 100000`

A naive solution that processes every substring independently would be far too slow. Even an `O(length of substring)` solution per query could degrade to `O(nq)`, which is approximately `10^{10}` operations in the worst case.

Therefore, we need a preprocessing strategy that allows each query to be answered in logarithmic or constant time.

### Important Observations

The digit sum is easy to handle because it is simply the sum of non-zero digits in the queried range.

The difficult part is computing the concatenated value `x`.

For example:

```
digits: 1 2 3 4
x = 1234
```

This can be viewed as:

$$1 \cdot 10^3 + 2 \cdot 10^2 + 3 \cdot 10^1 + 4$$

The challenge is extracting the contribution of only the non-zero digits that lie inside a query interval.

### Edge Cases

A few situations deserve special attention:

- The queried substring may contain only zeros.
- The queried substring may contain exactly one non-zero digit.
- The entire string may be non-zero.
- Queries may cover almost the entire string.
- The resulting integer may have tens of thousands of digits, so we must work modulo `10^9 + 7` throughout.

## Approaches

### Brute Force

For each query:

1. Extract the substring.
2. Scan it character by character.
3. Build the non-zero digit string.
4. Compute its numeric value.
5. Compute its digit sum.
6. Return the product.

This is clearly correct because it directly follows the problem definition.

However, if a query spans `O(n)` characters and there are `O(q)` queries, the total complexity becomes:

$$O(nq)$$

which is far too slow for `n = q = 100000`.

### Key Insight

The query only depends on the non-zero digits.

Suppose we compress the original string and keep only:

- the value of each non-zero digit,
- its original position.

Let:

- `digits[]` = sequence of all non-zero digits.
- `positions[]` = their original indices.

Now every query `[l, r]` corresponds to a contiguous segment inside `digits[]`.

If the first relevant non-zero digit is at compressed index `L` and the last is at compressed index `R`, then:

$$x = d_L d_{L+1} \cdots d_R$$

This becomes a substring query on the compressed sequence.

We can answer such queries efficiently using polynomial-hash style preprocessing:

$$prefix[i]
=
prefix[i-1]\cdot10 + digit_i$$

Then any concatenated value can be extracted using powers of ten.

At the same time, a simple prefix sum provides the digit sum.

Each query then becomes:

- two binary searches,
- a few prefix computations,

which gives logarithmic time per query.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(q·n) worst case | O(n) | Rebuilds the number for every query |
| Optimal | O(n + q log n) | O(n) | Compresses non-zero digits and uses prefix preprocessing |

## Algorithm Walkthrough

### Step 1: Extract all non-zero digits

Scan the string once.

For every non-zero character:

- store its original position,
- store its digit value.

For example:

```
s = "10203004"

positions = [0,2,4,7]
digits    = [1,2,3,4]
```

### Step 2: Precompute powers of ten

Build:

```
pow10[i] = 10^i mod MOD
```

for all required lengths.

These powers allow us to remove unwanted prefixes from a concatenated number.

### Step 3: Build digit-sum prefix array

Let:

```
digitPrefix[i]
```

store the sum of the first `i` non-zero digits.

This allows range digit sums in constant time.

### Step 4: Build concatenation prefix array

Define:

```
valuePrefix[0] = 0

valuePrefix[i]
=
valuePrefix[i-1] * 10 + digits[i]
```

modulo `MOD`.

This stores the concatenated value of the first `i` non-zero digits.

### Step 5: Locate relevant non-zero digits

For query `[l,r]`:

Use binary search on `positions`.

Find:

- `L = first non-zero position >= l`
- `R = last non-zero position <= r`

If no such indices exist, return `0`.

### Step 6: Compute the concatenated number

Suppose the compressed segment contains:

```
k = R - L + 1
```

digits.

The concatenated value is:

$$valuePrefix[R+1]
-
valuePrefix[L]\cdot10^k$$

modulo `MOD`.

This is the standard substring extraction formula.

### Step 7: Compute the digit sum

Using the prefix sums:

$$sum =
digitPrefix[R+1] - digitPrefix[L]$$

### Step 8: Compute the answer

Return:

$$x \cdot sum \pmod{10^9+7}$$

for every query.

### Why it works

The compressed array contains exactly the non-zero digits in their original order. Every query corresponds to a contiguous segment of this compressed sequence. The concatenation prefix behaves identically to a rolling polynomial representation in base 10, allowing any contiguous digit segment to be extracted in constant time. The digit-sum prefix independently provides the sum of those same digits. Since both quantities exactly match the problem definition, multiplying them yields the correct answer.

## Python Solution

```python
from typing import List
from bisect import bisect_left, bisect_right

class Solution:
    def sumAndMultiply(self, s: str, queries: List[List[int]]) -> List[int]:
        MOD = 10**9 + 7

        positions = []
        digits = []

        for i, ch in enumerate(s):
            if ch != '0':
                positions.append(i)
                digits.append(int(ch))

        k = len(digits)

        pow10 = [1] * (k + 1)
        for i in range(1, k + 1):
            pow10[i] = (pow10[i - 1] * 10) % MOD

        digit_prefix = [0] * (k + 1)
        value_prefix = [0] * (k + 1)

        for i in range(k):
            digit_prefix[i + 1] = digit_prefix[i] + digits[i]
            value_prefix[i + 1] = (
                value_prefix[i] * 10 + digits[i]
            ) % MOD

        answer = []

        for l, r in queries:
            left = bisect_left(positions, l)
            right = bisect_right(positions, r) - 1

            if left > right:
                answer.append(0)
                continue

            length = right - left + 1

            x = (
                value_prefix[right + 1]
                - value_prefix[left] * pow10[length]
            ) % MOD

            digit_sum = (
                digit_prefix[right + 1]
                - digit_prefix[left]
            )

            answer.append((x * digit_sum) % MOD)

        return answer
```

The implementation begins by compressing the string into only its non-zero digits and their positions. This immediately removes all irrelevant characters from future processing.

The `pow10` array stores powers of ten modulo `MOD`. These values are needed when extracting a digit segment from the concatenation prefix.

The `digit_prefix` array stores cumulative digit sums, while `value_prefix` stores cumulative concatenated values. The recurrence

```python
value_prefix[i + 1] =
value_prefix[i] * 10 + digits[i]
```

constructs the same number that would result from concatenating digits one by one.

For each query, binary searches identify which compressed digits belong to the original interval. If no non-zero digit exists inside the range, the answer is immediately zero.

Otherwise, the substring extraction formula obtains the concatenated value in constant time, the digit sum is obtained from the prefix array, and the final product is returned modulo `MOD`.

## Go Solution

```go
package main

import "sort"

func sumAndMultiply(s string, queries [][]int) []int {
	const MOD int64 = 1000000007

	positions := make([]int, 0)
	digits := make([]int64, 0)

	for i, ch := range s {
		if ch != '0' {
			positions = append(positions, i)
			digits = append(digits, int64(ch-'0'))
		}
	}

	k := len(digits)

	pow10 := make([]int64, k+1)
	pow10[0] = 1
	for i := 1; i <= k; i++ {
		pow10[i] = (pow10[i-1] * 10) % MOD
	}

	digitPrefix := make([]int64, k+1)
	valuePrefix := make([]int64, k+1)

	for i := 0; i < k; i++ {
		digitPrefix[i+1] = digitPrefix[i] + digits[i]
		valuePrefix[i+1] =
			(valuePrefix[i]*10 + digits[i]) % MOD
	}

	ans := make([]int, len(queries))

	for idx, q := range queries {
		l, r := q[0], q[1]

		left := sort.Search(len(positions), func(i int) bool {
			return positions[i] >= l
		})

		right := sort.Search(len(positions), func(i int) bool {
			return positions[i] > r
		}) - 1

		if left > right {
			ans[idx] = 0
			continue
		}

		length := right - left + 1

		x := (valuePrefix[right+1] -
			valuePrefix[left]*pow10[length]) % MOD

		if x < 0 {
			x += MOD
		}

		digitSum := digitPrefix[right+1] -
			digitPrefix[left]

		ans[idx] = int((x * digitSum) % MOD)
	}

	return ans
}
```

### Go-Specific Notes

Go does not provide built-in binary search helpers equivalent to Python's `bisect_left` and `bisect_right`, so `sort.Search` is used.

All modular arithmetic is performed using `int64` to avoid overflow during multiplication. After subtraction, the intermediate result may become negative, so we add `MOD` when necessary before taking the final answer.

## Edge Cases

### Query Contains No Non-Zero Digits

A common source of bugs is attempting to extract a concatenated value from an empty digit range. For example:

```
s = "00000"
query = [1,3]
```

There are no non-zero digits inside the interval, so the answer must be `0`. The implementation detects this when the binary search result satisfies `left > right`, and immediately returns zero.

### Query Contains Exactly One Non-Zero Digit

When only one non-zero digit exists in the range, the concatenated number should equal that digit itself.

For example:

```
s = "020"
query = [0,2]
```

The compressed segment contains only digit `2`, giving:

```
x = 2
sum = 2
answer = 4
```

The substring extraction formula naturally handles length-one ranges without any special cases.

### Very Large Concatenated Numbers

A query may contain tens of thousands of non-zero digits. Constructing the actual integer would be impossible within normal integer limits.

The implementation never stores the full number. Instead, it maintains every concatenation modulo `10^9 + 7` using the prefix recurrence. This keeps all values bounded while preserving the correctness of modular arithmetic.

### Entire String Is Non-Zero

If every character is already non-zero, the compressed representation has size `n`. The algorithm still works efficiently because preprocessing remains linear and each query still requires only two binary searches and constant-time arithmetic. No special handling is needed.
