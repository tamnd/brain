## Solution

## Problem Understanding

The problem asks us to identify **active businesses** based on events data. Each row in the `Events` table records the number of occurrences of a particular `event_type` for a `business_id`. The key challenge is to first compute the **average occurrences** for each event type across all businesses, and then determine which businesses exceed this average for **more than one event type**.

In other words, we are tasked with analyzing the events data in two stages. First, compute the mean `occurrences` for each `event_type`. Second, count for each business how many of its event types have `occurrences` strictly greater than the corresponding average. If this count is greater than one, the business is considered active.

The input is a relational table with three columns: `business_id`, `event_type`, and `occurrences`. The output is a single-column table listing the `business_id`s of active businesses. The order of output does not matter.

Constraints and observations include:

- `business_id` and `event_type` together are unique, which simplifies our analysis since there are no duplicate events for the same business and type.
- `occurrences` are positive integers.
- We need to handle multiple event types per business, and only businesses exceeding the average in **more than one type** are active.

Important edge cases to consider are:

1. Businesses with only one event type cannot be active, no matter how high their occurrences.
2. Event types with occurrences that are all equal; in this case, exceeding the average is impossible.
3. Businesses that exceed the average for exactly one event type; they are **not active**.

## Approaches

The **brute-force approach** would be to iterate over each event type, compute the average occurrences, and then iterate over each business-event pair to count the number of event types exceeding the average. This works because it directly implements the problem statement, but it involves scanning the table multiple times and may be inefficient for large datasets.

The **optimal approach** leverages SQL aggregation and joins to reduce the number of passes over the data. The key insight is that we can compute the average occurrences per event type using a `GROUP BY`, and then join this result back to the original table to directly count, per business, the number of event types where the business exceeds the average. Filtering on this count gives the active businesses efficiently.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * m) | O(n) | Iterate over events and compute average for each type, then scan all businesses. |
| Optimal | O(n) | O(n) | Use SQL aggregation and join to count exceeding averages per business in a single query. |

## Algorithm Walkthrough

1. Compute the average `occurrences` for each `event_type` across all businesses using a `GROUP BY` query. This gives a table mapping each event type to its mean activity.
2. Join the original `Events` table with this average table on `event_type`.
3. For each row in the join, compare `occurrences` with the average for that event type. Keep only rows where `occurrences` is strictly greater than the average.
4. Group the filtered rows by `business_id` and count the number of event types exceeding the average.
5. Filter businesses where this count is greater than one. These are the active businesses.
6. Return the `business_id`s of these active businesses.

**Why it works:** By joining the table with the average occurrences and counting only the events above the average, we ensure that each business is correctly evaluated. Grouping and filtering on the count ensures the "more than one" condition is met.

## Python Solution

```python
import pandas as pd

def active_businesses(events: pd.DataFrame) -> pd.DataFrame:
    # Step 1: Compute average occurrences per event_type
    avg_occurrences = events.groupby('event_type', as_index=False)['occurrences'].mean()
    avg_occurrences.rename(columns={'occurrences': 'avg_occurrences'}, inplace=True)
    
    # Step 2: Join original events with averages
    merged = events.merge(avg_occurrences, on='event_type')
    
    # Step 3: Filter events where occurrences > average
    above_avg = merged[merged['occurrences'] > merged['avg_occurrences']]
    
    # Step 4: Count number of event types above average per business
    counts = above_avg.groupby('business_id', as_index=False)['event_type'].count()
    
    # Step 5: Filter businesses with more than one event type above average
    active = counts[counts['event_type'] > 1][['business_id']]
    
    return active
```

The Python implementation follows the algorithm closely. We first compute averages using `groupby`, then merge with the original events to compare occurrences. Filtering and counting ensure that only businesses exceeding the average in more than one event type are selected.

## Go Solution

```go
package main

import (
    "database/sql"
    _ "github.com/go-sql-driver/mysql"
    "fmt"
)

func ActiveBusinesses(db *sql.DB) ([]int, error) {
    query := `
    WITH AvgOccurrences AS (
        SELECT event_type, AVG(occurrences) AS avg_occurrences
        FROM Events
        GROUP BY event_type
    ),
    AboveAvg AS (
        SELECT e.business_id
        FROM Events e
        JOIN AvgOccurrences a
        ON e.event_type = a.event_type
        WHERE e.occurrences > a.avg_occurrences
    )
    SELECT business_id
    FROM AboveAvg
    GROUP BY business_id
    HAVING COUNT(*) > 1;
    `
    rows, err := db.Query(query)
    if err != nil {
        return nil, err
    }
    defer rows.Close()
    
    var results []int
    for rows.Next() {
        var businessID int
        if err := rows.Scan(&businessID); err != nil {
            return nil, err
        }
        results = append(results, businessID)
    }
    return results, nil
}
```

In Go, we leverage SQL directly via a `WITH` clause for readability. The logic mirrors Python: compute averages, filter occurrences above average, group by business, and count. The main difference is handling SQL connections, `rows.Scan`, and slice appending.

## Worked Examples

Using the problem statement input:

```
+-------------+------------+-------------+
| business_id | event_type | occurrences |
+-------------+------------+-------------+
| 1           | reviews    | 7           |
| 3           | reviews    | 3           |
| 1           | ads        | 11          |
| 2           | ads        | 7           |
| 3           | ads        | 6           |
| 1           | page views | 3           |
| 2           | page views | 12          |
```

**Step 1:** Compute averages:

| event_type | avg_occurrences |
| --- | --- |
| reviews | 5 |
| ads | 8 |
| page views | 7.5 |

**Step 2:** Merge and filter events above average:

| business_id | event_type | occurrences | avg_occurrences |
| --- | --- | --- | --- |
| 1 | reviews | 7 | 5 |
| 1 | ads | 11 | 8 |
| 2 | page views | 12 | 7.5 |

**Step 3:** Count per business:

| business_id | count_above_avg |
| --- | --- |
| 1 | 2 |
| 2 | 1 |

**Step 4:** Filter count > 1 → `business_id = 1` is active.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Computing averages, merging, filtering, and grouping each scan the table at most once. |
| Space | O(n) | Storage of the merged table and intermediate counts grows linearly with the number of events. |

The solution is efficient because it uses aggregation and avoids nested iteration over businesses and event types.
