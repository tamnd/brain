## Solution

## Problem Understanding

In this problem, we are given a two dimensional matrix called `grid`, where each cell contains a non negative integer value. We start at the top left corner `(0, 0)` and want to reach the bottom right corner `(m - 1, n - 1)` by moving only in the four cardinal directions: up, down, left, and right.

The important part is how a path is scored.

A path’s score is defined as the minimum value encountered anywhere along that path. For example, if a path contains values:

```
8 → 4 → 5 → 9
```

then the score of that path is `4`, because `4` is the smallest value on the path.

Our goal is not to minimize distance or maximize sum. Instead, we want to find the path whose minimum value is as large as possible.

Another way to think about it is:

- Every path has a weakest cell
- We want the path whose weakest cell is stronger than the weakest cell of every other possible path

This is sometimes called a "maximum bottleneck path" problem.

The input constraints are:

- `1 <= m, n <= 100`
- `0 <= grid[i][j] <= 10^9`

Since the grid dimensions are at most `100 x 100`, there can be up to `10,000` cells. This is large enough that exhaustive enumeration of all paths is infeasible, because the number of possible paths can grow exponentially.

The value range is very large, up to `10^9`, which suggests we should not rely on techniques that require iterating over all possible values directly.

Several edge cases are important:

- A `1 x 1` grid, where the answer is simply the single cell value
- Grids containing many zeroes, where the optimal path may still be forced through a zero
- Situations where the shortest path is not the best path
- Situations where a longer route avoids small bottleneck values
- Large plateaus of equal values, which can create many equivalent optimal paths

The problem guarantees that the grid is non empty and rectangular, so we do not need to handle malformed input.

## Approaches

### Brute Force Approach

The brute force solution would explore every possible path from the top left corner to the bottom right corner using DFS backtracking.

For each path:

1. Track the minimum value encountered along that path
2. When the destination is reached, compare the path score with the global maximum
3. Return the largest score among all valid paths

This approach is correct because it literally evaluates every possible path, so it cannot miss the optimal one.

However, the number of paths in a grid grows exponentially. Since movement is allowed in four directions, the search space becomes enormous even for moderately sized grids. A `100 x 100` grid would be completely impossible to process this way.

The brute force approach is therefore far too slow.

### Key Insight

The important observation is this:

If we always expand the currently highest valued position first, then we maximize our chances of preserving a large minimum value along the path.

This resembles Dijkstra’s algorithm, but instead of minimizing path cost, we maximize the minimum value along the route.

For each cell:

- The best score reaching that cell is the maximum possible bottleneck value
- When moving to a neighbor, the new path score becomes:

```
min(current_path_score, neighbor_value)
```

This is because the path’s bottleneck is determined by the smallest value encountered so far.

A max heap is perfect here because it allows us to always process the most promising path first.

The first time we reach the destination, we are guaranteed to have found the optimal answer.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | O(m × n) | Explores every possible path |
| Optimal Heap-Based BFS | O(m × n × log(m × n)) | O(m × n) | Uses a max heap to prioritize better bottleneck paths |

## Algorithm Walkthrough

### Optimal Algorithm Using a Max Heap

1. Initialize a max heap and place the starting cell into it.

Since Python’s `heapq` is a min heap, we store negative values to simulate max heap behavior.

Each heap entry stores:

- current path score
- row index
- column index
2. Initialize a `visited` set.

Once a cell is processed with the best possible bottleneck score, revisiting it is unnecessary.
3. Start from `(0, 0)`.

The initial path score is simply `grid[0][0]`, because the path currently contains only one cell.
4. Repeatedly pop the best candidate from the heap.

The heap always gives us the path with the highest current minimum value.
5. If the popped cell is the destination, return its score immediately.

This works because the heap guarantees that no future path can produce a better bottleneck score.
6. Explore all four neighboring cells.

For every valid neighbor:

- compute the new path score:

```
new_score = min(current_score, neighbor_value)
```

- push the neighbor into the heap

1. Mark neighbors as visited when pushing them.

This prevents duplicate processing and keeps the runtime efficient.
2. Continue until the destination is reached.

### Why it works

The algorithm maintains the invariant that cells are processed in descending order of path bottleneck score.

When a cell is removed from the max heap, we already know there cannot exist another undiscovered path to that cell with a better minimum value. This is the same correctness principle used in Dijkstra’s algorithm.

Therefore, when the destination is popped from the heap, its score is guaranteed to be the maximum possible minimum value among all paths.

## Python Solution

```
from typing import List
import heapq

class Solution:
    def maximumMinimumPath(self, grid: List[List[int]]) -> int:
        rows = len(grid)
        cols = len(grid[0])

        max_heap = [(-grid[0][0], 0, 0)]
        visited = set()
        visited.add((0, 0))

        directions = [(1, 0), (-1, 0), (0, 1), (0, -1)]

        while max_heap:
            negative_score, row, col = heapq.heappop(max_heap)

            current_score = -negative_score

            if row == rows - 1 and col == cols - 1:
                return current_score

            for dr, dc in directions:
                new_row = row + dr
                new_col = col + dc

                if (
                    0 <= new_row < rows
                    and 0 <= new_col < cols
                    and (new_row, new_col) not in visited
                ):
                    visited.add((new_row, new_col))

                    new_score = min(current_score, grid[new_row][new_col])

                    heapq.heappush(
                        max_heap,
                        (-new_score, new_row, new_col)
                    )

        return -1
```

The implementation begins by determining the grid dimensions. We then initialize a max heap containing the starting position.

Because Python’s built in heap implementation is a min heap, negative values are stored so that larger scores are processed first.

The `visited` set ensures that each cell is processed only once. This keeps the algorithm efficient and prevents cycles caused by four directional movement.

Inside the main loop, we pop the cell with the best current bottleneck score. If this cell is the destination, we immediately return the score.

For each neighboring cell, we compute the updated bottleneck value using:

```
min(current_score, grid[new_row][new_col])
```

This reflects the definition of the path score, which is the minimum value along the entire path.

Each valid neighbor is then inserted into the heap for future processing.

If the algorithm somehow finishes without reaching the destination, `-1` is returned, although the problem guarantees a valid path exists.

## Go Solution

```
package main

import (
	"container/heap"
)

type State struct {
	score int
	row   int
	col   int
}

type MaxHeap []State

func (h MaxHeap) Len() int {
	return len(h)
}

func (h MaxHeap) Less(i, j int) bool {
	return h[i].score > h[j].score
}

func (h MaxHeap) Swap(i, j int) {
	h[i], h[j] = h[j], h[i]
}

func (h *MaxHeap) Push(x interface{}) {
	*h = append(*h, x.(State))
}

func (h *MaxHeap) Pop() interface{} {
	old := *h
	n := len(old)

	item := old[n-1]

	*h = old[:n-1]

	return item
}

func minimum(a, b int) int {
	if a < b {
		return a
	}
	return b
}

func maximumMinimumPath(grid [][]int) int {
	rows := len(grid)
	cols := len(grid[0])

	visited := make([][]bool, rows)

	for i := 0; i < rows; i++ {
		visited[i] = make([]bool, cols)
	}

	maxHeap := &MaxHeap{}
	heap.Init(maxHeap)

	heap.Push(maxHeap, State{
		score: grid[0][0],
		row:   0,
		col:   0,
	})

	visited[0][0] = true

	directions := [][]int{
		{1, 0},
		{-1, 0},
		{0, 1},
		{0, -1},
	}

	for maxHeap.Len() > 0 {
		current := heap.Pop(maxHeap).(State)

		score := current.score
		row := current.row
		col := current.col

		if row == rows-1 && col == cols-1 {
			return score
		}

		for _, dir := range directions {
			newRow := row + dir[0]
			newCol := col + dir[1]

			if newRow >= 0 &&
				newRow < rows &&
				newCol >= 0 &&
				newCol < cols &&
				!visited[newRow][newCol] {

				visited[newRow][newCol] = true

				newScore := minimum(
					score,
					grid[newRow][newCol],
				)

				heap.Push(maxHeap, State{
					score: newScore,
					row:   newRow,
					col:   newCol,
				})
			}
		}
	}

	return -1
}
```

The Go implementation follows the same algorithmic structure as the Python solution, but Go requires explicit heap implementation using the `container/heap` package.

A custom `MaxHeap` type is created by implementing the heap interface methods:

- `Len`
- `Less`
- `Swap`
- `Push`
- `Pop`

The `Less` method is reversed so that the heap behaves as a max heap rather than a min heap.

Instead of a hash set, Go uses a two dimensional boolean slice for visited tracking. This is both memory efficient and fast.

Since Go integers are already large enough for the constraint range, no overflow concerns exist here.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m × n × log(m × n)) | Each cell may be inserted into the heap once, heap operations cost logarithmic time |
| Space | O(m × n) | Heap and visited structures may store every cell |

The grid contains `m × n` cells.

Each cell is processed at most once because we mark cells visited when inserting them into the heap.

Every heap insertion and removal costs:

```
O(log(m × n))
```

Therefore, the total runtime becomes:

```
O(m × n × log(m × n))
```

The heap and visited structures can both contain up to all cells in the grid, giving linear auxiliary space usage.

## Edge Cases

### Single Cell Grid

A `1 x 1` grid is the smallest valid input. The start and destination are the same cell, so the answer must simply be that cell’s value.

This case can expose bugs in algorithms that assume at least one movement step exists. Our implementation handles it naturally because the starting node is immediately recognized as the destination when popped from the heap.

### Paths Forced Through Zero

Some grids may contain unavoidable zeroes. Since the score of a path is its minimum value, any path containing a zero has score zero.

A buggy implementation might incorrectly prioritize large neighboring values and miss the fact that every valid route eventually crosses a zero. Our algorithm correctly computes the bottleneck incrementally using `min(current_score, neighbor_value)`.

### Longer Path With Better Bottleneck

The optimal path is not necessarily the shortest path.

For example:

```
[
  [10, 1, 1],
  [9,  8, 7],
  [1,  1, 6]
]
```

A short route across the top immediately encounters small values, while a longer route around the middle preserves a much larger minimum value.

Algorithms based purely on BFS shortest distance would fail here. Our max heap approach always prioritizes the best bottleneck score rather than path length.

### Large Equal Value Regions

Grids with many repeated values can generate many equivalent candidate paths.

If visited handling is implemented incorrectly, the algorithm may repeatedly process the same cells and become inefficient. By marking cells visited when pushing them into the heap, we guarantee each cell enters the heap only once.
