## Solution

## Problem Understanding

In this problem, we are given a string `text`, and we want to split it into the maximum possible number of non-empty pieces such that the sequence of pieces forms a palindrome at the substring level.

This is different from checking whether the whole string itself is a palindrome. Instead, we are decomposing the string into chunks where:

- The first chunk equals the last chunk
- The second chunk equals the second-to-last chunk
- And so on

The goal is to maximize the number of chunks.

For example:

```
"ghiabcdefhelloadamhelloabcdefghi"
```

can be split into:

```
(ghi)(abcdef)(hello)(adam)(hello)(abcdef)(ghi)
```

The chunks mirror each other around the center. Since there are 7 chunks total, the answer is 7.

The input size is relatively small:

```
1 <= text.length <= 1000
```

This constraint allows solutions up to roughly `O(n^2)` comfortably, but exponential approaches become impractical.

A very important detail is that chunks do not need to be the same length. The only requirement is that symmetric chunks match exactly.

Another key observation is that the center chunk, if one exists, does not need a matching partner. For example:

```
(a)(bc)(xyz)(bc)(a)
```

Here, `"xyz"` is the middle unmatched chunk.

Important edge cases include:

- Strings with no matching prefix and suffix at all
- Strings where every character can become its own chunk
- Odd-length decompositions with a single middle chunk
- Repeated characters, which can trick naive greedy implementations
- Very small inputs such as length 1

The problem guarantees that the string contains only lowercase English letters, which simplifies comparisons and hashing strategies.

## Approaches

### Brute Force Approach

A brute force solution tries every possible prefix and suffix pair recursively.

For every possible chunk size:

1. Compare the prefix and suffix of that size
2. If they match, recursively solve the remaining middle substring
3. Take the decomposition with the maximum number of chunks

For example:

```
merchant
```

We would try:

- `"m"` vs `"t"`
- `"me"` vs `"nt"`
- `"mer"` vs `"ant"`
- and so on

Since no prefix equals its corresponding suffix, the answer becomes 1.

This approach is correct because it explores every valid decomposition possibility. However, it is extremely inefficient because many overlapping subproblems are recomputed repeatedly.

The number of recursive states grows exponentially in the worst case.

### Key Insight for the Optimal Solution

The crucial observation is that we should greedily take the smallest matching prefix and suffix whenever possible.

Why?

Suppose we find a matching pair early:

```
prefix == suffix
```

Choosing smaller matching chunks increases the total number of chunks we can obtain.

If we delay and take larger chunks instead, we potentially lose opportunities to split the string further.

This greedy property allows us to process the string from both ends simultaneously.

We build:

- A growing prefix string from the left
- A growing suffix string from the right

Whenever they become equal:

- We count two chunks
- Reset both builders
- Continue inward

If a middle unmatched portion remains at the end, it contributes one additional chunk.

This produces an elegant `O(n^2)` greedy solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(n) | Recursively tries all decompositions |
| Optimal | O(n^2) | O(n) | Greedily matches smallest prefix and suffix |

## Algorithm Walkthrough

1. Initialize two empty strings:

- `left_chunk`
- `right_chunk`

These will gradually build candidate prefix and suffix chunks.
2. Use two pointers:

- `left` starting from the beginning
- `right` starting from the end
3. At each iteration:

- Append `text[left]` to `left_chunk`
- Prepend `text[right]` to `right_chunk`

We prepend on the right side because we are building the suffix in correct order.
4. Compare the two constructed chunks.

If:

```
left_chunk == right_chunk
```

then we found a valid mirrored pair.
5. When a match is found:

- If `left == right`, this is the middle chunk, so add 1
- Otherwise, add 2 because we matched both sides
6. Reset both chunk builders to empty strings and continue searching for the next pair.
7. Continue until the pointers cross.
8. Return the total chunk count.

### Why it works

The algorithm relies on a greedy invariant:

Whenever the smallest possible matching prefix and suffix are chosen, the remaining middle substring can still be decomposed optimally.

Choosing smaller matching chunks always leaves more flexibility for future splits. Therefore, greedily taking the first available match maximizes the total number of chunks.

## Python Solution

```python
class Solution:
    def longestDecomposition(self, text: str) -> int:
        left_chunk = ""
        right_chunk = ""

        left = 0
        right = len(text) - 1

        chunks = 0

        while left <= right:
            left_chunk += text[left]
            right_chunk = text[right] + right_chunk

            if left_chunk == right_chunk:
                if left == right:
                    chunks += 1
                else:
                    chunks += 2

                left_chunk = ""
                right_chunk = ""

            left += 1
            right -= 1

        return chunks
```

The implementation follows the greedy strategy directly.

Two strings are used to accumulate candidate chunks from both sides of the input. The `left_chunk` grows normally, while `right_chunk` is built in reverse order so that the resulting string matches the actual suffix order.

Each iteration processes one character from the left and one from the right.

When the two constructed chunks become equal, we immediately count them as a valid decomposition pair. If both pointers point to the same character, then we are processing the center chunk of an odd-length decomposition, so only one chunk is added.

After counting a match, both chunk builders are reset so the next decomposition segment can begin.

The algorithm terminates once all characters have been processed.

## Go Solution

```go
func longestDecomposition(text string) int {
	leftChunk := ""
	rightChunk := ""

	left := 0
	right := len(text) - 1

	chunks := 0

	for left <= right {
		leftChunk += string(text[left])
		rightChunk = string(text[right]) + rightChunk

		if leftChunk == rightChunk {
			if left == right {
				chunks++
			} else {
				chunks += 2
			}

			leftChunk = ""
			rightChunk = ""
		}

		left++
		right--
	}

	return chunks
}
```

The Go implementation mirrors the Python solution closely.

Since Go strings are immutable, concatenation creates new strings each time, similar to Python. Given the small constraint of at most 1000 characters, this is perfectly acceptable.

Character extraction requires conversion:

```go
string(text[left])
```

because indexing a Go string returns a byte rather than a string.

No special handling for nil values is needed because strings in Go default to empty strings.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n^2) | String concatenation and comparisons can each cost O(n) |
| Space | O(n) | Temporary chunk strings store up to O(n) characters |

Although the main loop runs only `O(n)` iterations, string concatenation and equality checks are not constant time. In the worst case, each comparison may examine many characters, leading to overall quadratic complexity.

Given the constraint `n <= 1000`, this complexity is efficient enough.

## Edge Cases

### Single Character String

A string such as:

```
"a"
```

can only form one chunk.

This case is important because both pointers refer to the same index immediately. Incorrect implementations may accidentally count two chunks instead of one.

The implementation handles this correctly using:

```
if left == right:
    chunks += 1
```

### No Matching Prefix and Suffix

Inputs like:

```
"merchant"
```

contain no matching prefix and suffix at any stage.

A buggy greedy solution might incorrectly split partial matches or fail to count the remaining substring.

In this implementation, no chunk reset occurs, so the entire string naturally becomes one chunk.

### Repeated Characters

Strings like:

```
"aaaaaa"
```

can expose flaws in decomposition logic because many different valid chunkings exist.

A non-greedy approach might choose large chunks early and miss the maximum decomposition.

The greedy strategy always selects the smallest matching prefix and suffix first, allowing every character to become its own chunk and producing the optimal answer of 6.
