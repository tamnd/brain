## Solution

## Problem Understanding

The problem gives us an array `hours`, where each element represents the number of hours worked on a particular day.

A day is classified as:

- a tiring day if `hours[i] > 8`
- a non-tiring day otherwise

We need to find the length of the longest contiguous subarray where the number of tiring days is strictly greater than the number of non-tiring days.

In other words, for any interval `[l, r]`, we want:

```
count(tiring days) > count(non-tiring days)
```

The output is the maximum possible interval length satisfying this condition.

A useful way to reinterpret the problem is to convert each day into a score:

- `+1` for a tiring day
- `-1` for a non-tiring day

Then the problem becomes:

> Find the longest subarray whose total sum is strictly positive.

This transformation is the key observation that enables an efficient solution.

The constraints tell us that:

- `1 <= hours.length <= 10^4`
- brute force solutions with `O(n^2)` complexity may be borderline or too slow in practice
- we should aim for an `O(n)` solution

Several edge cases are important:

- all days are non-tiring, answer should be `0`
- all days are tiring, answer should be the entire array length
- intervals may begin at index `0`
- multiple prefix sums may repeat
- the longest valid interval may appear near the end of the array

A naive implementation can easily miss intervals that depend on earlier prefix sums, which is why careful tracking is required.

## Approaches

### Brute Force

The brute-force approach checks every possible subarray.

For each interval `[i, j]`, we count how many tiring and non-tiring days exist. If tiring days exceed non-tiring days, we update the maximum length.

A straightforward implementation requires:

- choosing all start indices
- choosing all end indices
- computing the balance for each interval

Even with prefix sums to optimize interval evaluation, we still examine all `O(n^2)` subarrays.

This works correctly because every possible interval is explicitly checked, so the longest valid one cannot be missed.

However, with `n = 10^4`, an `O(n^2)` solution may require around 50 million interval checks, which is unnecessarily expensive.

### Optimal Approach, Prefix Sum with Earliest Occurrence Hash Map

The key insight is that we only care whether an interval has a positive transformed sum.

Convert the array as follows:

```
hours[i] > 8  -> +1
hours[i] <= 8 -> -1
```

Define a prefix sum:

```
prefix[i] = sum of transformed values from 0 to i
```

For an interval `(j + 1 ... i)`:

```
interval_sum = prefix[i] - prefix[j]
```

We need:

```
prefix[i] - prefix[j] > 0
```

which means:

```
prefix[j] < prefix[i]
```

Now consider two important cases:

- If `prefix[i] > 0`, then the entire range `[0...i]` is valid.
- Otherwise, we want the earliest index where the prefix sum was `prefix[i] - 1`.

Why `prefix[i] - 1`?

Because prefix sums change only by `+1` or `-1`. If we can find an earlier prefix sum strictly smaller than the current one, then the interval between them has positive sum.

To maximize interval length, we store the earliest occurrence of each prefix sum in a hash map.

This gives us an `O(n)` solution.

## Algorithm Walkthrough

1. Initialize a running prefix sum `score = 0`.

Each tiring day contributes `+1`, and each non-tiring day contributes `-1`.
2. Create a hash map called `first_seen`.

This stores the earliest index where each prefix sum first appeared.

We only store the first occurrence because earlier indices create longer intervals.
3. Initialize `max_length = 0`.

This keeps track of the longest valid interval found so far.
4. Iterate through the array.

For each day:

- if `hours[i] > 8`, add `1`
- otherwise subtract `1`
5. If the current prefix sum is positive, then the entire interval from `0` to `i` is well-performing.

Update:

```
max_length = i + 1
```
6. Otherwise, check whether `score - 1` exists in the hash map.

If it does, suppose its earliest index is `j`.

Then:

```
prefix[i] - prefix[j] > 0
```

which means the interval `(j + 1 ... i)` is well-performing.

Update the maximum length accordingly.
7. Store the current prefix sum in the hash map if it has not appeared before.

We preserve the earliest occurrence because it produces the largest interval lengths later.
8. Continue until the array is fully processed.
9. Return `max_length`.

### Why it works

The algorithm relies on the prefix sum identity:

```
subarray_sum = prefix[right] - prefix[left]
```

A well-performing interval requires a positive transformed sum.

For every position `i`, we search for the earliest earlier prefix sum smaller than the current prefix sum. Because prefix sums only change by `±1`, checking `score - 1` is sufficient.

By storing earliest occurrences, every valid interval ending at `i` is maximized in length.

## Python Solution

```
from typing import List

class Solution:
    def longestWPI(self, hours: List[int]) -> int:
        score = 0
        max_length = 0

        # Maps prefix sum -> earliest index
        first_seen = {}

        for index, hour in enumerate(hours):
            if hour > 8:
                score += 1
            else:
                score -= 1

            # Entire prefix is valid
            if score > 0:
                max_length = index + 1
            else:
                # Look for an earlier prefix sum smaller than current
                if (score - 1) in first_seen:
                    interval_length = index - first_seen[score - 1]
                    max_length = max(max_length, interval_length)

            # Store earliest occurrence only
            if score not in first_seen:
                first_seen[score] = index

        return max_length
```

The implementation directly follows the prefix sum strategy.

The variable `score` represents the transformed cumulative balance between tiring and non-tiring days.

For each day:

- `+1` is added if the day is tiring
- `-1` is added otherwise

When `score > 0`, the entire prefix from index `0` to the current index is automatically valid because tiring days outnumber non-tiring days overall.

If `score <= 0`, the algorithm searches for a previously seen prefix sum equal to `score - 1`. Finding such a prefix guarantees a positive interval sum between the two positions.

The dictionary `first_seen` stores only the first occurrence of each prefix sum. This is essential because earlier indices create longer candidate intervals.

The algorithm processes the array once, resulting in linear time complexity.

## Go Solution

```
func longestWPI(hours []int) int {
	score := 0
	maxLength := 0

	// prefix sum -> earliest index
	firstSeen := make(map[int]int)

	for i, hour := range hours {
		if hour > 8 {
			score++
		} else {
			score--
		}

		// Entire prefix is valid
		if score > 0 {
			maxLength = i + 1
		} else {
			// Check for smaller previous prefix sum
			if idx, exists := firstSeen[score-1]; exists {
				length := i - idx
				if length > maxLength {
					maxLength = length
				}
			}
		}

		// Store earliest occurrence only
		if _, exists := firstSeen[score]; !exists {
			firstSeen[score] = i
		}
	}

	return maxLength
}
```

The Go implementation mirrors the Python logic closely.

A `map[int]int` is used for storing the earliest occurrence of each prefix sum.

Go does not have Python's built-in `max` function for integers, so comparisons are handled explicitly with `if` statements.

There are no concerns about integer overflow because the maximum prefix sum magnitude is bounded by `n <= 10^4`.

An empty slice is not possible because the constraints guarantee at least one element.

## Edge Cases

### All Days Are Non-Tiring

Consider:

```
[6,6,6,6]
```

Every transformed value becomes `-1`, so every prefix sum is negative.

A buggy implementation might incorrectly treat equal counts as valid or accidentally return a nonzero interval.

The current implementation correctly requires the interval sum to be strictly positive, so the result remains `0`.

### Entire Array Is Valid

Consider:

```
[9,9,9,6]
```

The transformed array is:

```
[+1,+1,+1,-1]
```

The total sum is still positive, so the whole array is well-performing.

The implementation handles this through the condition:

```
if score > 0:
    max_length = index + 1
```

Without this special handling, intervals beginning at index `0` could be missed.

### Multiple Repeated Prefix Sums

Consider:

```
[9,6,9,6,9]
```

The prefix sums repeatedly revisit earlier values.

If the algorithm overwrote earlier hash map entries, it would lose the longest possible interval.

The implementation avoids this bug by storing only the earliest occurrence:

```
if score not in first_seen:
    first_seen[score] = index
```

This guarantees maximal interval lengths later in the traversal.
