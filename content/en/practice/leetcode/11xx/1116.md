## Solution

## Problem Understanding

This problem asks us to coordinate three threads to print numbers in a very specific interleaved order: zeros alternating with consecutive integers starting from 1 up to `n`. Specifically, Thread A prints zeros, Thread B prints even numbers, and Thread C prints odd numbers. Each thread calls its respective method (`zero`, `even`, or `odd`) on the same `ZeroEvenOdd` instance, but the order in which threads run is not guaranteed, so we must synchronize them.

The input `n` specifies how many numbers (not zeros) we want to print. For example, if `n = 5`, the output sequence should be `"0102030405"`. Each zero is immediately followed by either the next odd or even number. The length of the final string will be `2 * n`, since every number is preceded by a zero.

Constraints tell us `1 <= n <= 1000`. This implies the solution must be efficient in terms of concurrency management but does not need extreme optimizations for large data. The main challenge is thread synchronization, not computation.

Important edge cases include `n = 1` (minimum input), which should produce `"01"`, and ensuring even and odd numbers alternate correctly when `n` is even versus odd. The implementation must also ensure no race conditions occur and each thread prints exactly what it should without skipping or duplicating output.

## Approaches

A naive or brute-force approach might attempt to use busy-waiting loops in each thread to check whose turn it is, or a global counter without proper synchronization. While this could eventually produce correct output, it is inefficient and prone to race conditions, deadlocks, or incorrect sequences, especially under high concurrency.

The key insight for an optimal solution is to use synchronization primitives such as **semaphores** or **condition variables** to control which thread is allowed to run at a given time. We can maintain a strict turn-taking system:

- The `zero` thread prints `0` and then signals either the `odd` or `even` thread depending on the current number.
- The `odd` and `even` threads print their number and then signal `zero` to continue.

This guarantees the correct interleaving and avoids busy waiting.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Uses busy waiting or global state without proper synchronization, may fail under concurrency |
| Optimal | O(n) | O(1) | Uses semaphores/locks to coordinate threads precisely, ensures correct ordering without unnecessary CPU usage |

## Algorithm Walkthrough

1. Initialize three semaphores: `zero_sem` starting at 1, `odd_sem` and `even_sem` starting at 0. The `zero` thread starts first.
2. Start a loop in the `zero` thread from 1 to `n`. In each iteration:

- Acquire `zero_sem`.
- Print `0`.
- Release `odd_sem` if the current number is odd, otherwise release `even_sem`.
3. Start the `odd` thread:

- Loop over all odd numbers from 1 to `n`.
- Acquire `odd_sem`.
- Print the number.
- Release `zero_sem`.
4. Start the `even` thread:

- Loop over all even numbers from 2 to `n`.
- Acquire `even_sem`.
- Print the number.
- Release `zero_sem`.

Why it works: At any moment, only one semaphore allows a thread to run. The zero thread always runs first, prints `0`, and signals the correct number thread. After the number thread prints, it signals zero to continue. This cycle guarantees the pattern `"010203..."` and prevents race conditions.

## Python Solution

```
from threading import Semaphore
from typing import Callable

class ZeroEvenOdd:
    def __init__(self, n: int):
        self.n = n
        self.zero_sem = Semaphore(1)
        self.odd_sem = Semaphore(0)
        self.even_sem = Semaphore(0)

    def zero(self, printNumber: 'Callable[[int], None]') -> None:
        for i in range(1, self.n + 1):
            self.zero_sem.acquire()
            printNumber(0)
            if i % 2 == 1:
                self.odd_sem.release()
            else:
                self.even_sem.release()

    def even(self, printNumber: 'Callable[[int], None]') -> None:
        for i in range(2, self.n + 1, 2):
            self.even_sem.acquire()
            printNumber(i)
            self.zero_sem.release()

    def odd(self, printNumber: 'Callable[[int], None]') -> None:
        for i in range(1, self.n + 1, 2):
            self.odd_sem.acquire()
            printNumber(i)
            self.zero_sem.release()
```

The Python implementation uses `Semaphore` objects to control access. `zero_sem` allows zero to run first. After printing zero, it releases the correct number semaphore depending on whether the next number is odd or even. Each number thread prints its number and releases `zero_sem` for the next cycle.

## Go Solution

```
package main

import (
	"sync"
)

type ZeroEvenOdd struct {
	n        int
	zeroSem  chan struct{}
	oddSem   chan struct{}
	evenSem  chan struct{}
}

func NewZeroEvenOdd(n int) *ZeroEvenOdd {
	zeo := &ZeroEvenOdd{
		n:       n,
		zeroSem: make(chan struct{}, 1),
		oddSem:  make(chan struct{}, 1),
		evenSem: make(chan struct{}, 1),
	}
	zeo.zeroSem <- struct{}{} // allow zero to start
	return zeo
}

func (z *ZeroEvenOdd) Zero(printNumber func(int)) {
	for i := 1; i <= z.n; i++ {
		<-z.zeroSem
		printNumber(0)
		if i%2 == 1 {
			z.oddSem <- struct{}{}
		} else {
			z.evenSem <- struct{}{}
		}
	}
}

func (z *ZeroEvenOdd) Even(printNumber func(int)) {
	for i := 2; i <= z.n; i += 2 {
		<-z.evenSem
		printNumber(i)
		z.zeroSem <- struct{}{}
	}
}

func (z *ZeroEvenOdd) Odd(printNumber func(int)) {
	for i := 1; i <= z.n; i += 2 {
		<-z.oddSem
		printNumber(i)
		z.zeroSem <- struct{}{}
	}
}
```

The Go solution mirrors the Python approach but uses buffered channels of size 1 to implement semaphores. Channels are acquired with `<-` and released by sending an empty struct. Go channels naturally handle synchronization without explicit locking.
