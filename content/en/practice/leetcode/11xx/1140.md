# LeetCode 1140 - Stone Game II

## Problem Understanding

This problem describes a two-player game between Alice and Bob. The game uses an array called `piles`, where each element represents the number of stones in a pile. The piles are arranged in a row, and players always take piles from the front of the remaining sequence.

Alice moves first. On each turn, the current player may take the first `X` remaining piles, where `1 <= X <= 2M`. After the move, the value of `M` becomes `max(M, X)`. Initially, `M = 1`.

Both players play optimally, meaning each player always chooses the move that maximizes their own final number of stones. The task is to compute the maximum number of stones Alice can collect by the end of the game.

The input is:

- `piles`, an integer array of positive values

The output is:

- The maximum number of stones Alice can guarantee if both players play optimally

The constraints are important:

- `1 <= piles.length <= 100`
- `1 <= piles[i] <= 10^4`

The array length is relatively small, which strongly suggests that a dynamic programming solution is appropriate. A naive recursive exploration of all possible game states would grow exponentially and become infeasible. Since the number of states is manageable, memoization or bottom-up DP is likely the intended approach.

Several edge cases are important:

- A single pile, where Alice immediately takes everything
- Situations where a player can take all remaining piles because `2M` is large enough
- Arrays with very uneven pile sizes, where choosing fewer piles now may lead to a better future position
- Cases where greedy choices fail, because taking the largest immediate amount is not always optimal

The key difficulty is that every move changes the future range of valid moves through the evolving value of `M`.

## Approaches

### Brute Force Approach

The brute-force solution recursively simulates every possible game state. At each turn, the current player tries every valid choice of `X`, recursively computes the result of the remaining game, and chooses the move that maximizes their final score.

The recursive state is determined by:

- The current index in the pile array
- The current value of `M`
- Whose turn it is

For every state, we explore all possible moves from `1` to `2M`.

This approach is correct because it exhaustively evaluates every legal sequence of moves and assumes both players make optimal decisions.

However, it is far too slow because many identical states are recomputed repeatedly. The number of recursive branches grows exponentially, making the solution impractical even for `n = 100`.

### Optimal Dynamic Programming Approach

The key insight is that the game contains overlapping subproblems.

If we know:

- The current starting index `i`
- The current value `M`

then the remaining game outcome is completely determined.

We define a DP state:

- `dp(i, M)` = maximum stones the current player can obtain starting from index `i` with parameter `M`

Instead of directly tracking both players' scores separately, we use a stronger observation:

If the total remaining stones from index `i` onward is known, then:

```
current player's gain =
remaining stones - opponent's best possible gain
```

This transforms the problem into a minimax-style DP.

To efficiently compute remaining sums, we precompute suffix sums:

```
suffix[i] = total stones from i to end
```

This allows constant-time calculation of remaining stones.

Memoization ensures each `(i, M)` state is solved only once.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | O(n) recursion stack | Explores every possible game sequence |
| Optimal Dynamic Programming | O(n³) | O(n²) | Uses memoization and suffix sums |

## Algorithm Walkthrough

1. Precompute suffix sums.

We create an array `suffix` where `suffix[i]` stores the total number of stones from pile `i` to the end of the array. This allows us to quickly compute how many stones remain at any position.

For example:

```
piles = [2,7,9,4,4]

suffix[4] = 4
suffix[3] = 8
suffix[2] = 17
suffix[1] = 24
suffix[0] = 26
```
2. Define the recursive DP state.

Let:

```
dp(i, M)
```

represent the maximum stones the current player can collect starting from pile index `i` when the current value of `M` is `M`.
3. Handle the base case.

If the current player can take all remaining piles, then that is always optimal.

This condition is:

```
i + 2M >= n
```

because the player may take up to `2M` piles.

In this case:

```
dp(i, M) = suffix[i]
```
4. Try every valid move.

The player may choose any `X` such that:

```
1 <= X <= 2M
```

For each choice:

- The player takes the first `X` piles
- The opponent then starts at index `i + X`
- The new value of `M` becomes `max(M, X)`
5. Compute the opponent's best result.

After choosing `X`, the opponent can optimally collect:

```
dp(i + X, max(M, X))
```
6. Derive the current player's score.

Since `suffix[i]` is the total remaining stones, if the opponent gets some portion of them, the current player gets the rest:

```
current_gain =
suffix[i] - opponent_gain
```
7. Choose the best move.

We maximize the current player's result over all possible `X`.
8. Memoize results.

Since many states repeat, we cache each `(i, M)` result to avoid recomputation.
9. Return the result.

The answer is:

```
dp(0, 1)
```

because Alice starts at index `0` with `M = 1`.

### Why it works

At every state, the total remaining stones are fixed. Whatever stones the opponent eventually collects cannot also belong to the current player. Therefore, maximizing the current player's final score is equivalent to minimizing the opponent's optimal future score.

The recursive relation correctly models optimal play from both sides, and memoization guarantees that every state is computed consistently and only once.

## Python Solution

```
from typing import List
from functools import lru_cache

class Solution:
    def stoneGameII(self, piles: List[int]) -> int:
        n = len(piles)

        suffix = [0] * (n + 1)

        for i in range(n - 1, -1, -1):
            suffix[i] = suffix[i + 1] + piles[i]

        @lru_cache(None)
        def dp(index: int, m: int) -> int:
            if index + 2 * m >= n:
                return suffix[index]

            best = 0

            for x in range(1, 2 * m + 1):
                opponent = dp(index + x, max(m, x))
                current = suffix[index] - opponent
                best = max(best, current)

            return best

        return dp(0, 1)
```

The implementation begins by computing suffix sums. This allows constant-time lookup of the total remaining stones from any index onward.

The recursive function `dp(index, m)` represents the optimal number of stones the current player can collect from the remaining piles starting at `index` with parameter `m`.

The base case checks whether the player can take all remaining piles. If so, the function immediately returns the remaining sum.

Otherwise, the function iterates through every valid move size `x`. For each move:

- The opponent's optimal result is recursively computed
- The current player's result becomes:

```
remaining stones - opponent's result
```

The best possible value among all moves is stored and returned.

The `@lru_cache` decorator ensures that repeated states are computed only once, dramatically reducing runtime.

## Go Solution

```
func stoneGameII(piles []int) int {
	n := len(piles)

	suffix := make([]int, n+1)

	for i := n - 1; i >= 0; i-- {
		suffix[i] = suffix[i+1] + piles[i]
	}

	memo := make(map[[2]int]int)

	var dp func(int, int) int

	dp = func(index int, m int) int {
		if index+2*m >= n {
			return suffix[index]
		}

		key := [2]int{index, m}

		if val, exists := memo[key]; exists {
			return val
		}

		best := 0

		for x := 1; x <= 2*m; x++ {
			opponent := dp(index+x, max(m, x))
			current := suffix[index] - opponent

			if current > best {
				best = current
			}
		}

		memo[key] = best
		return best
	}

	return dp(0, 1)
}

func max(a int, b int) int {
	if a > b {
		return a
	}
	return b
}
```

The Go implementation follows the same logic as the Python version.

Since Go does not provide built-in memoization decorators, we explicitly store computed states in a map. The key is a two-element array containing `(index, m)`.

Go slices are used for suffix sums. Integer overflow is not a concern because the maximum total number of stones is well within Go's integer range.

Unlike Python, Go requires explicit helper functions such as `max`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n³) | There are O(n²) states, each exploring up to O(n) moves |
| Space | O(n²) | Memoization table stores O(n²) states |

There are at most `n` possible indices and at most `n` meaningful values for `M`, giving approximately `O(n²)` DP states.

For each state, we iterate over up to `2M` possible moves, which in the worst case becomes `O(n)` work per state.

Therefore, the total runtime is `O(n³)`.

The suffix array uses `O(n)` space, while memoization dominates with `O(n²)` storage.

## Edge Cases

One important edge case occurs when there is only one pile. In this scenario, Alice immediately takes the entire pile because `M = 1` allows taking up to two piles. A naive recursive solution might incorrectly continue recursion or mishandle indexing. The implementation correctly detects that all remaining piles can be taken and returns the suffix sum immediately.

Another important edge case occurs when `2M` becomes large enough to cover all remaining piles. In these situations, the optimal move is always to take everything immediately. Without this base case, the recursion would continue exploring unnecessary states and potentially produce incorrect results. The implementation handles this with:

```
if index + 2 * m >= n
```

which directly returns the total remaining stones.

A third subtle edge case involves non-greedy optimal play. Sometimes taking fewer piles initially leads to a better overall result because it restricts the opponent's future choices. For example:

```
[1,2,3,4,5,100]
```

A greedy approach that always maximizes the current move would fail badly here. The DP formulation correctly evaluates all future consequences before selecting the best move.

Another edge case involves repeated states. Different move sequences can lead to the same `(index, M)` configuration. Without memoization, the recursion would repeatedly recompute identical subproblems, causing exponential runtime. The cache guarantees each state is solved only once.
