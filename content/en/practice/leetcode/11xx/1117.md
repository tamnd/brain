## Solution

## Problem Understanding

This problem asks us to coordinate multiple concurrent threads so they form water molecules correctly. Each water molecule requires exactly two hydrogen atoms and one oxygen atom. Threads arrive independently, and we must ensure that they are released in valid groups of three.

The important requirement is not just counting atoms overall, but synchronizing them so that every molecule is formed as a complete unit before the next molecule begins bonding. If we divide the output into chunks of three characters, every chunk must contain exactly two `H` characters and one `O` character.

The input conceptually represents a stream of concurrently executing threads. Hydrogen threads call the `hydrogen()` method, and oxygen threads call the `oxygen()` method. Each method receives a callback function that prints its corresponding atom.

The output is not a single fixed ordering. Multiple valid outputs exist because thread scheduling is nondeterministic. For example:

```
HHO
HOH
OHH
```

are all valid for a single molecule because they all contain two hydrogens and one oxygen.

The constraints are relatively small, with at most 20 molecules, meaning at most 60 threads total. However, this is a concurrency problem, not a traditional data size problem. The challenge is correctness of synchronization rather than algorithmic scalability.

Several edge cases are important:

- Many hydrogen threads may arrive before any oxygen thread.
- Many oxygen threads may arrive before hydrogens.
- Threads may interleave unpredictably.
- Multiple molecules may be formed back to back.
- We must prevent threads from different molecules from mixing together.

The problem guarantees that the total counts are valid: exactly `2 * n` hydrogens and `n` oxygens exist. Therefore, a complete set of molecules can always eventually form.

## Approaches

### Brute Force Approach

A naive approach would use shared counters and busy waiting. For example, hydrogen threads could continuously check whether fewer than two hydrogens have entered the current molecule, and oxygen threads could continuously check whether an oxygen slot is available.

The algorithm would look conceptually like this:

- Maintain counters for hydrogens and oxygens in the current molecule.
- Threads repeatedly poll the shared state.
- Once the molecule becomes complete, reset counters for the next molecule.

This approach can produce correct results if implemented carefully with locks. However, it is inefficient because threads actively spin while waiting. Busy waiting wastes CPU cycles and creates unnecessary contention.

More importantly, synchronization becomes fragile. Without proper barriers, one molecule may overlap with the next, causing invalid groupings.

### Optimal Approach

The better solution uses semaphores and a barrier.

The key observation is that each molecule always requires:

- exactly 2 hydrogen threads
- exactly 1 oxygen thread

This maps naturally to semaphores:

- A hydrogen semaphore initialized to 2 allows at most two hydrogen threads into a molecule.
- An oxygen semaphore initialized to 1 allows exactly one oxygen thread into a molecule.

A barrier ensures that all three participating threads finish bonding before any thread from the next molecule proceeds.

This solution is efficient because waiting threads sleep instead of spinning, and synchronization primitives enforce the molecule structure directly.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Uses polling or busy waiting, inefficient and error prone |
| Optimal | O(n) | O(1) | Uses semaphores and barriers for precise synchronization |

## Algorithm Walkthrough

### Optimal Algorithm

1. Create a semaphore for hydrogen initialized with value `2`.

This semaphore represents the two hydrogen slots available in each molecule. Only two hydrogen threads may proceed before reset.
2. Create a semaphore for oxygen initialized with value `1`.

This semaphore ensures only one oxygen thread participates in each molecule.
3. Create a barrier initialized with value `3`.

The barrier forces exactly three threads to synchronize before moving forward.
4. When a hydrogen thread arrives:

- Acquire the hydrogen semaphore.
- Print `"H"` using `releaseHydrogen()`.
- Wait at the barrier.
- After the barrier releases all three threads, release the hydrogen semaphore permit.
5. When an oxygen thread arrives:

- Acquire the oxygen semaphore.
- Print `"O"` using `releaseOxygen()`.
- Wait at the barrier.
- After synchronization completes, release the oxygen semaphore permit.
6. The barrier guarantees that:

- exactly three threads participate together
- all three finish before the next molecule begins
7. Releasing semaphore permits after the barrier allows the next molecule to form safely.

### Why it works

The invariant is that at any moment:

- at most two hydrogen threads are active in a molecule
- at most one oxygen thread is active in a molecule

The barrier guarantees that all three threads synchronize before any permits are returned. Therefore, molecules cannot overlap, and every group of three threads contains exactly two hydrogens and one oxygen.

## Python Solution

```
from threading import Semaphore, Barrier
from typing import Callable

class H2O:
    def __init__(self):
        self.hydrogen_semaphore = Semaphore(2)
        self.oxygen_semaphore = Semaphore(1)
        self.barrier = Barrier(3)

    def hydrogen(self, releaseHydrogen: 'Callable[[], None]') -> None:
        self.hydrogen_semaphore.acquire()

        # releaseHydrogen() outputs "H". Do not change or remove this line.
        releaseHydrogen()

        self.barrier.wait()

        self.hydrogen_semaphore.release()

    def oxygen(self, releaseOxygen: 'Callable[[], None]') -> None:
        self.oxygen_semaphore.acquire()

        # releaseOxygen() outputs "O". Do not change or remove this line.
        releaseOxygen()

        self.barrier.wait()

        self.oxygen_semaphore.release()
```

The implementation starts by creating two semaphores and one barrier.

The hydrogen semaphore starts with value `2`, meaning two hydrogen threads can participate in the current molecule. The oxygen semaphore starts with value `1`, allowing only one oxygen thread.

When a hydrogen thread enters, it first acquires a hydrogen slot. If two hydrogens are already active, the thread blocks automatically until the next molecule begins.

After printing `"H"`, the thread waits at the barrier. The barrier pauses execution until exactly three threads arrive.

The oxygen method behaves similarly, except only one oxygen thread may acquire the semaphore at a time.

Once all three threads reach the barrier, the barrier releases them together. Only after synchronization completes are semaphore permits returned, enabling formation of the next molecule.

This ordering is critical because it prevents threads from different molecules from interleaving.

## Go Solution

```
package main

import (
	"sync"
)

type H2O struct {
	hydrogenSem chan struct{}
	oxygenSem   chan struct{}
	barrier     sync.WaitGroup
	mutex       sync.Mutex
	count       int
}

func NewH2O() *H2O {
	h := &H2O{
		hydrogenSem: make(chan struct{}, 2),
		oxygenSem:   make(chan struct{}, 1),
	}

	h.hydrogenSem <- struct{}{}
	h.hydrogenSem <- struct{}{}

	h.oxygenSem <- struct{}{}

	return h
}

func (h *H2O) waitBarrier() {
	h.mutex.Lock()
	h.count++

	if h.count == 3 {
		h.count = 0
		h.barrier.Done()
		h.barrier.Done()
		h.barrier.Done()
	}

	h.mutex.Unlock()

	h.barrier.Wait()
}

func (h *H2O) Hydrogen(releaseHydrogen func()) {
	<-h.hydrogenSem

	h.barrier.Add(1)

	// releaseHydrogen() outputs "H". Do not change or remove this line.
	releaseHydrogen()

	h.waitBarrier()

	h.hydrogenSem <- struct{}{}
}

func (h *H2O) Oxygen(releaseOxygen func()) {
	<-h.oxygenSem

	h.barrier.Add(1)

	// releaseOxygen() outputs "O". Do not change or remove this line.
	releaseOxygen()

	h.waitBarrier()

	h.oxygenSem <- struct{}{}
}
```

The Go implementation uses buffered channels as semaphores because Go does not provide a built in semaphore type.

The hydrogen channel has capacity `2`, and the oxygen channel has capacity `1`. Receiving from the channel acquires a permit, and sending back into the channel releases it.

Go does not include a reusable barrier primitive like Python's `Barrier`, so the implementation manually coordinates synchronization using `sync.WaitGroup`, a mutex, and a shared counter.

The mutex protects the shared `count` variable so increments remain thread safe.

## Edge Cases

### Many Hydrogen Threads Arrive First

A common bug is allowing more than two hydrogen threads into the same molecule. If many hydrogen threads arrive before oxygen, a naive implementation may print multiple hydrogens consecutively without proper grouping.

The semaphore prevents this by allowing only two hydrogen threads to proceed. Additional hydrogen threads block automatically until the next molecule begins.

### Many Oxygen Threads Arrive First

Another tricky scenario occurs when oxygen threads arrive before hydrogens. Without synchronization, multiple oxygen threads could proceed incorrectly.

The oxygen semaphore solves this problem by allowing only one oxygen thread at a time. Extra oxygen threads wait until the current molecule completes.

### Molecule Boundary Interleaving

The hardest concurrency issue is preventing one molecule from overlapping with another.

For example, this invalid sequence could happen without a barrier:

```
H H O H
```

The fourth hydrogen belongs to the next molecule, but it appears before the first molecule fully completes.

The barrier prevents this issue completely. All three threads in a molecule must arrive before any semaphore permits are returned. This creates a strict synchronization boundary between molecules.
