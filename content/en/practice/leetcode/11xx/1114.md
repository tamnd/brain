## Solution

## Problem Understanding

This problem asks us to coordinate the execution order of three methods across different threads. We are given a `Foo` class with three methods: `first()`, `second()`, and `third()`. These methods are called by three independent threads, and the operating system may schedule those threads in any order.

The challenge is that even if `third()` starts running before `first()`, the final execution order must always be:

```
first
second
third
```

In other words, we must guarantee that:

- `second()` executes only after `first()` has completed.
- `third()` executes only after `second()` has completed.

The input examples, such as `[1,2,3]` or `[1,3,2]`, only describe the order in which threads are started. They do not guarantee execution order because thread scheduling is nondeterministic.

For example, in:

```
nums = [1,3,2]
```

thread A calls `first()`, thread B calls `third()`, and thread C calls `second()`. Since threads are asynchronous, `third()` might start before `second()`. Our implementation must prevent that and still ensure the output becomes:

```
"firstsecondthird"
```

The constraints are intentionally small because there are always exactly three operations. This means algorithmic efficiency is not the primary challenge. Instead, this is a concurrency synchronization problem. The main difficulty is ensuring proper ordering without relying on thread scheduling behavior.

An important edge case is when the methods are invoked in the worst possible order, such as `third()` starting first, followed by `second()`, and finally `first()`. A naive implementation would print out of order. The problem guarantees that all three methods will be called exactly once, which simplifies synchronization because we do not need to handle repeated calls or missing threads.

## Approaches

### Brute Force Approach

A brute force approach would repeatedly check whether previous methods have finished using shared boolean variables and busy waiting.

For example:

- `first()` sets `first_done = True`
- `second()` continuously loops until `first_done == True`
- `third()` continuously loops until `second_done == True`

This works because each method waits until its dependency has completed before proceeding.

However, this approach is inefficient because it wastes CPU cycles through active polling, also called busy waiting. Threads repeatedly check conditions instead of sleeping efficiently while waiting for a signal. In real concurrent systems, this leads to unnecessary CPU consumption and poor scalability.

### Key Insight

The key observation is that we need synchronization, not repeated checking.

Modern concurrency libraries provide synchronization primitives designed exactly for this problem. We can use **locks/events/semaphores** to block a thread until another thread signals completion.

In Python, `threading.Event` is a natural fit:

- `first()` signals an event after completion.
- `second()` waits for that signal, then signals another event.
- `third()` waits for the second signal.

In Go, channels are ideal synchronization primitives:

- `First()` closes or signals a channel.
- `Second()` waits on that channel before running, then signals another channel.
- `Third()` waits for the second signal.

This eliminates busy waiting and guarantees correct execution ordering.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(∞) worst case | O(1) | Busy waiting repeatedly checks flags, wastes CPU |
| Optimal | O(1) | O(1) | Uses synchronization primitives to block efficiently |

## Algorithm Walkthrough

We will use synchronization signals to enforce ordering.

### Python Version Using `threading.Event`

1. During initialization, create two synchronization events:

- One event indicates that `first()` has completed.
- Another event indicates that `second()` has completed.
2. When `first()` executes, it calls `printFirst()` immediately because nothing depends on it. After printing, it signals the first event to notify waiting threads that execution can continue.
3. When `second()` executes, it waits on the first event. If `first()` has not finished yet, the thread blocks efficiently. Once the event is signaled, `second()` prints `"second"` and signals the second event.
4. When `third()` executes, it waits on the second event. It cannot proceed until `second()` finishes. Once the signal arrives, it prints `"third"`.

### Go Version Using Channels

1. Create two channels during initialization:

- One channel signals completion of `First()`.
- Another channel signals completion of `Second()`.
2. `First()` executes immediately and then closes the first channel to notify waiting goroutines.
3. `Second()` waits to receive from the first channel before proceeding. Once unblocked, it prints and closes the second channel.
4. `Third()` waits on the second channel before printing.

### Why it works

The correctness comes from enforced dependencies.

`second()` cannot proceed until `first()` signals completion, and `third()` cannot proceed until `second()` signals completion. Since the synchronization primitives block execution rather than polling, the ordering is guaranteed regardless of thread scheduling. Even if `third()` starts first, it immediately blocks until `second()` finishes.

## Python Solution

```
from typing import Callable
from threading import Event

class Foo:
    def __init__(self):
        self.first_done = Event()
        self.second_done = Event()

    def first(self, printFirst: 'Callable[[], None]') -> None:
        # printFirst() outputs "first". Do not change or remove this line.
        printFirst()

        # Signal that first() has completed
        self.first_done.set()

    def second(self, printSecond: 'Callable[[], None]') -> None:
        # Wait until first() finishes
        self.first_done.wait()

        # printSecond() outputs "second". Do not change or remove this line.
        printSecond()

        # Signal that second() has completed
        self.second_done.set()

    def third(self, printThird: 'Callable[[], None]') -> None:
        # Wait until second() finishes
        self.second_done.wait()

        # printThird() outputs "third". Do not change or remove this line.
        printThird()
```

The implementation begins by creating two `Event` objects in the constructor. These events act like signals between threads.

The `first()` method executes immediately because it has no dependency. After printing `"first"`, it calls `set()` on `first_done`, which wakes any thread waiting for `first()` to finish.

The `second()` method calls `wait()` on `first_done`. If `first()` has not executed yet, the thread pauses without consuming CPU. Once unblocked, it prints `"second"` and signals `second_done`.

Finally, `third()` waits for `second_done`. This guarantees it cannot execute before `second()` completes.

The synchronization structure directly mirrors the dependency chain:

```
first → second → third
```

## Go Solution

```
type Foo struct {
	firstDone  chan struct{}
	secondDone chan struct{}
}

func NewFoo() *Foo {
	return &Foo{
		firstDone:  make(chan struct{}),
		secondDone: make(chan struct{}),
	}
}

func (f *Foo) First(printFirst func()) {
	// Do not change this line
	printFirst()

	// Signal that First() is complete
	close(f.firstDone)
}

func (f *Foo) Second(printSecond func()) {
	// Wait for First()
	<-f.firstDone

	/// Do not change this line
	printSecond()

	// Signal that Second() is complete
	close(f.secondDone)
}

func (f *Foo) Third(printThird func()) {
	// Wait for Second()
	<-f.secondDone

	// Do not change this line
	printThird()
}
```

The Go implementation uses channels instead of events. A `chan struct{}` is ideal here because it transmits a signal without carrying data, minimizing memory overhead.

Closing a channel acts as a broadcast signal. Any goroutine waiting to receive from the channel becomes unblocked. This makes channels a clean synchronization mechanism for enforcing execution order.

Unlike Python, Go does not require an explicit `wait()` method. Reading from a channel naturally blocks until data becomes available or the channel closes.

There are no concerns about integer overflow, nil handling, or slice behavior because the solution only uses synchronization channels.

### Example 1

**Input:**

```
nums = [1,2,3]
```

This means:

- Thread A calls `first()`
- Thread B calls `second()`

| Step | Action | first_done | second_done | Output |
| --- | --- | --- | --- | --- |

```
"firstsecondthird"
```

**Input:**

```

```

This means thread scheduling may happen in this order:

```
third() → second() → first()
```

| Step | Action | first_done | second_done | Output |
| --- | --- | --- | --- | --- |
| 1 | `third()` starts | Not Set | Not Set | `""` |
| 2 | `third()` blocks | Not Set | Not Set | `""` |
| 3 | `second()` starts | Not Set | Not Set | `""` |
| 4 | `second()` blocks | Not Set | Not Set | `""` |
| 5 | `first()` runs | Set | Not Set | `"first"` |
| 6 | `second()` unblocks | Set | Set | `"firstsecond"` |
| 7 | `third()` unblocks | Set | Set | `"firstsecondthird"` |

Final output:

```
"firstsecondthird"
```

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(1) | Each method performs constant work and synchronization |
| Space | O(1) | Only two synchronization primitives are stored |

Since there are always exactly three methods, the amount of work never scales with input size. Each thread performs a constant number of operations and synchronization calls. The additional memory usage is also constant because we only maintain two signals.

## Test Cases

```
from threading import Thread

def run_test(order):
    foo = Foo()
    output = []

    def print_first():
        output.append("first")

    def print_second():
        output.append("second")

    def print_third():
        output.append("third")

    methods = {
        1: lambda: foo.first(print_first),
        2: lambda: foo.second(print_second),
        3: lambda: foo.third(print_third),
    }

    threads = [Thread(target=methods[x]) for x in order]

    for thread in threads:
        thread.start()

    for thread in threads:
        thread.join()

    return "".join(output)

# Provided examples
assert run_test([1, 2, 3]) == "firstsecondthird"  # Already ordered
assert run_test([1, 3, 2]) == "firstsecondthird"  # Third starts before second

# Reverse order
assert run_test([3, 2, 1]) == "firstsecondthird"  # Worst scheduling order

# Mixed order
assert run_test([2, 1, 3]) == "firstsecondthird"  # Second starts before first
assert run_test([2, 3, 1]) == "firstsecondthird"  # Both dependent threads start first
assert run_test([3, 1, 2]) == "firstsecondthird"  # Third starts first

# Multiple runs for stability
for _ in range(100):
    assert run_test([3, 2, 1]) == "firstsecondthird"  # Stress concurrency
```

| Test | Why |
| --- | --- |
| `[1,2,3]` | Verifies normal execution order |
| `[1,3,2]` | Ensures `third()` waits for `second()` |
| `[3,2,1]` | Tests worst-case scheduling |
| `[2,1,3]` | Ensures `second()` blocks correctly |
| `[2,3,1]` | Validates multiple blocked threads |
| `[3,1,2]` | Confirms `third()` can safely start first |
| 100 repeated runs | Stress tests synchronization reliability |

## Edge Cases

One important edge case occurs when `third()` starts before both `first()` and `second()`. A naive implementation may immediately print `"third"`, producing an incorrect result. Our implementation avoids this because `third()` blocks on the synchronization signal from `second()`.

Another tricky case is when `second()` starts before `first()`. Without proper synchronization, `second()` may print prematurely. By calling `wait()` in Python or receiving from a channel in Go, execution pauses until `first()` signals completion.

A final edge case is extreme thread scheduling unpredictability. In concurrent systems, the operating system may pause or reorder threads arbitrarily. Even if threads execute in reverse order, the synchronization primitives guarantee that the dependency chain is preserved, ensuring the output is always `"firstsecondthird"`.
