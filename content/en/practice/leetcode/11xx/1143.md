## Solution

## Problem Understanding

The problem asks us to compute the length of the longest common subsequence between two strings. A subsequence is formed by deleting zero or more characters from a string without changing the order of the remaining characters. The characters do not need to be contiguous.

For example, given the string `"abcde"`, the string `"ace"` is a valid subsequence because we can remove the characters `'b'` and `'d'` while preserving the relative order of the remaining characters.

We are given two input strings, `text1` and `text2`. The goal is to find the maximum possible length of any subsequence that appears in both strings.

It is important to understand that this is not asking for a substring. Substrings must occupy consecutive positions, while subsequences only require relative ordering.

For example:

- `"ace"` is a subsequence of `"abcde"`
- `"aec"` is not a subsequence because the relative order changes
- `"bc"` is both a substring and a subsequence

The output is a single integer representing the length of the longest common subsequence, commonly abbreviated as LCS.

The constraints are important:

- Each string length can be up to `1000`
- Strings contain only lowercase English letters

A brute-force solution that tries every subsequence would be infeasible because the number of subsequences grows exponentially. A string of length `1000` has `2^1000` possible subsequences, which is astronomically large.

The constraints strongly suggest that we need a polynomial-time dynamic programming solution.

Several edge cases are worth identifying early:

- The two strings may have no characters in common, producing an answer of `0`
- The two strings may be identical, producing an answer equal to the full string length
- One string may be much longer than the other
- Repeated characters can create multiple possible matching paths
- The optimal subsequence may skip many characters in both strings

These cases can easily break naive greedy approaches.

## Approaches

### Brute-Force Approach

The brute-force idea is to generate every possible subsequence of one string and check whether it is also a subsequence of the other string.

For a string of length `n`, there are `2^n` possible subsequences. We could:

1. Generate all subsequences of `text1`
2. For each subsequence, check whether it appears in `text2`
3. Track the maximum valid length

This approach is correct because it examines every possible candidate subsequence, so it cannot miss the optimal answer.

However, it is far too slow. Even for strings of length `50`, the number of subsequences becomes impractically large. With lengths up to `1000`, brute force is completely impossible.

The key issue is overlapping subproblems. While exploring different subsequences, we repeatedly solve the same smaller matching problems again and again.

### Key Insight

The important observation is that the problem has optimal substructure.

Suppose we compare two positions:

- `text1[i]`
- `text2[j]`

There are two possibilities:

1. The characters match
2. The characters do not match

If they match, then this character can be part of the LCS, and we move both pointers forward.

If they do not match, then one of the characters must be excluded. We try both possibilities:

- Skip `text1[i]`
- Skip `text2[j]`

This naturally leads to a recursive structure with overlapping subproblems, which makes dynamic programming the ideal solution.

We define:

`dp[i][j]` = length of the longest common subsequence between:

- `text1[0:i]`
- `text2[0:j]`

This lets us build larger answers from smaller already-computed answers.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n × m) or worse | O(2^n) | Generates all subsequences and checks matches |
| Optimal Dynamic Programming | O(n × m) | O(n × m) | Reuses overlapping subproblems efficiently |

## Algorithm Walkthrough

### Dynamic Programming Strategy

We use a 2D dynamic programming table.

Let:

- `n = len(text1)`
- `m = len(text2)`

We create a table `dp` of size `(n + 1) × (m + 1)`.

Each cell represents:

`dp[i][j] = LCS length between first i characters of text1 and first j characters of text2`

The extra row and column simplify boundary conditions.

### Step-by-Step Process

1. Initialize the DP table with zeros.

If either substring is empty, the LCS length is `0`. Therefore:

- `dp[0][j] = 0`
- `dp[i][0] = 0`
2. Iterate through both strings.

For every `i` from `1` to `n` and every `j` from `1` to `m`, compare:

- `text1[i - 1]`
- `text2[j - 1]`
3. Handle matching characters.

If:

```
text1[i - 1] == text2[j - 1]
```

then this character contributes to the common subsequence.

We extend the previous subsequence:

```
dp[i][j] = dp[i - 1][j - 1] + 1
```
4. Handle non-matching characters.

If the characters differ, one of them cannot belong to the current optimal subsequence.

We try both possibilities:

- Skip current character from `text1`
- Skip current character from `text2`

Then take the better result:

```
dp[i][j] = max(dp[i - 1][j], dp[i][j - 1])
```
5. Continue until the table is filled.

Each cell depends only on previously computed cells, so the table can be filled row by row.
6. Return the final answer.

The bottom-right cell:

```
dp[n][m]
```

contains the LCS length for the full strings.

### Why it works

The dynamic programming recurrence works because every optimal subsequence decision reduces to smaller optimal subsequence problems.

If two characters match, any optimal solution including them must extend the optimal solution for the prefixes before those characters.

If two characters do not match, at least one character must be excluded. The optimal answer must therefore come from one of the two smaller subproblems formed by skipping one character.

Because every subproblem is solved exactly once and reused afterward, the algorithm efficiently computes the globally optimal solution.

## Python Solution

```python
class Solution:
    def longestCommonSubsequence(self, text1: str, text2: str) -> int:
        n = len(text1)
        m = len(text2)

        dp = [[0] * (m + 1) for _ in range(n + 1)]

        for i in range(1, n + 1):
            for j in range(1, m + 1):
                if text1[i - 1] == text2[j - 1]:
                    dp[i][j] = dp[i - 1][j - 1] + 1
                else:
                    dp[i][j] = max(dp[i - 1][j], dp[i][j - 1])

        return dp[n][m]
```

The implementation begins by storing the lengths of the two input strings. These dimensions determine the size of the DP table.

The table `dp` is initialized with zeros. The extra row and column represent empty prefixes, which naturally produce an LCS length of `0`.

The nested loops iterate through every pair of character positions. Since the DP table uses 1-based indexing relative to substring lengths, we compare characters using `i - 1` and `j - 1`.

When the characters match, we extend the previously computed subsequence length from the diagonal cell `dp[i - 1][j - 1]`.

When the characters differ, we choose the larger result between:

- skipping a character from `text1`
- skipping a character from `text2`

Finally, the answer is returned from the bottom-right corner of the table.

## Go Solution

```go
func longestCommonSubsequence(text1 string, text2 string) int {
	n := len(text1)
	m := len(text2)

	dp := make([][]int, n+1)
	for i := 0; i <= n; i++ {
		dp[i] = make([]int, m+1)
	}

	for i := 1; i <= n; i++ {
		for j := 1; j <= m; j++ {
			if text1[i-1] == text2[j-1] {
				dp[i][j] = dp[i-1][j-1] + 1
			} else {
				if dp[i-1][j] > dp[i][j-1] {
					dp[i][j] = dp[i-1][j]
				} else {
					dp[i][j] = dp[i][j-1]
				}
			}
		}
	}

	return dp[n][m]
}
```

The Go implementation follows the same logic as the Python version but uses slices to build the 2D DP table.

Go strings are byte-indexable, which is safe here because the problem guarantees lowercase English letters only. If Unicode characters were possible, we would need rune conversion.

Go does not provide a built-in `max` function for integers, so the comparison is implemented manually using an `if` statement.

The DP table is initialized with default zero values automatically.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n × m) | Every pair of positions is processed once |
| Space | O(n × m) | The DP table stores all subproblem results |

The nested loops iterate through every cell in the DP table exactly once. Since the table dimensions are `n + 1` by `m + 1`, the total number of operations is proportional to `n × m`.

The space complexity comes from storing the entire DP table. Since each cell holds one integer result, the memory usage is also `O(n × m)`.

## Edge Cases

One important edge case occurs when the two strings have no characters in common. A naive implementation might accidentally count partial matches incorrectly or fail to initialize the DP table properly. For example, `"abc"` and `"def"` should return `0`. The implementation handles this naturally because no matching transitions ever occur, so all DP values remain zero.

Another critical edge case involves repeated characters. Consider `"aaaa"` and `"aa"`. Greedy approaches can easily overcount or reuse characters incorrectly. The dynamic programming solution avoids this issue because every DP state represents a valid subsequence using prefixes only once, ensuring characters are matched in proper order without duplication.

A third important edge case occurs when one string is already a subsequence of the other. For example, `"ace"` inside `"abcde"`. Some incorrect solutions attempt to align characters positionally and miss valid non-contiguous matches. The DP recurrence correctly explores all alignment possibilities, allowing characters to be skipped freely while preserving order.

A fourth subtle edge case involves extremely small inputs such as single-character strings. These cases often expose indexing mistakes or off-by-one errors. The extra DP row and column prevent invalid index access and make the transitions consistent even for minimal input sizes.
