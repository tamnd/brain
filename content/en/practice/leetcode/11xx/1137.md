## Solution

## Problem Understanding

This problem asks us to compute the **n-th Tribonacci number**, which is part of a sequence similar to the Fibonacci sequence, but with one important difference. Instead of each number being formed from the sum of the previous **two** numbers, the Tribonacci sequence is formed from the sum of the previous **three** numbers.

The sequence starts with three predefined values:

- `T0 = 0`
- `T1 = 1`
- `T2 = 1`

After that, every new number is calculated using the recurrence relation:

$$T_{n+3} = T_n + T_{n+1} + T_{n+2}$$

In simpler terms, to calculate the next Tribonacci number, we add the previous three numbers together.

The input to the problem is a single integer `n`, representing the position in the Tribonacci sequence that we want to compute. The expected output is the value of `Tn`.

For example, if `n = 4`:

- `T0 = 0`
- `T1 = 1`
- `T2 = 1`
- `T3 = 0 + 1 + 1 = 2`
- `T4 = 1 + 1 + 2 = 4`

So the answer is `4`.

The constraints tell us several important things about the problem:

- `0 <= n <= 37`
- The result always fits inside a 32-bit signed integer.

Since `n` is very small, even relatively inefficient solutions might pass. However, the problem still tests whether we recognize the repeating subproblem structure and apply an efficient dynamic programming approach.

Several edge cases are important to identify immediately. If `n` is `0`, we must return `0`, since that is explicitly defined in the sequence. Similarly, `n = 1` and `n = 2` should both return `1`. These base cases can easily cause bugs if forgotten, especially because the recurrence only applies for values larger than `2`. Another consideration is avoiding unnecessary repeated calculations in recursive solutions, which can become inefficient due to overlapping subproblems.

## Approaches

### Brute Force Approach, Recursive Computation

A straightforward way to solve this problem is to directly follow the mathematical recurrence definition using recursion.

We can define:

$$T(n) = T(n-1) + T(n-2) + T(n-3)$$

and recursively compute each value until we reach one of the base cases (`0`, `1`, or `2`).

This approach is correct because it exactly mirrors the Tribonacci definition. Every recursive call reduces the problem into smaller subproblems until eventually reaching known values.

However, this method is highly inefficient because of repeated computation. For example, while computing `T(5)`, we compute `T(4)`, `T(3)`, and `T(2)`. But `T(4)` itself also computes `T(3)` and `T(2)` again. As `n` grows, the same values are recalculated many times, leading to exponential growth in runtime.

Even though the constraint `n <= 37` is small enough that memoized recursion would work comfortably, plain recursion is still unnecessarily slow.

### Optimal Approach, Iterative Dynamic Programming

The key insight is that every Tribonacci number depends only on the **previous three values**.

Instead of recomputing values repeatedly, we can compute the sequence iteratively from the bottom up. At every step, we only need to remember:

- The third previous value
- The second previous value
- The immediate previous value

This means we do not need an entire dynamic programming array. We can store only three variables and continuously update them as we move forward through the sequence.

This works because once we compute `T(i)`, older values beyond the last three are never needed again.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(3ⁿ) | O(n) | Recursive computation repeatedly recalculates the same subproblems |
| Optimal | O(n) | O(1) | Iterative dynamic programming using only three variables |

## Algorithm Walkthrough

### Optimal Iterative Dynamic Programming

1. First, handle the base cases directly. If `n == 0`, return `0`. If `n == 1` or `n == 2`, return `1`. These values are explicitly defined by the problem.
2. Initialize three variables to represent the last three Tribonacci numbers:

- `first = 0` represents `T0`
- `second = 1` represents `T1`
- `third = 1` represents `T2`

These variables will act as a sliding window over the sequence.
3. Start a loop from `3` through `n` inclusive. At each iteration, compute the next Tribonacci number:

```
next_value = first + second + third
```

This follows the Tribonacci recurrence formula.
4. Update the three tracking variables by shifting them forward:

```
first = second
second = third
third = next_value
```

This ensures the variables always represent the three most recent sequence values.
5. After the loop finishes, `third` contains `Tn`, so return it.

### Why it works

The algorithm maintains an invariant: before each iteration, `first`, `second`, and `third` always represent the previous three Tribonacci numbers needed to compute the next one. Since the recurrence relation depends only on these three values, each iteration correctly computes the next sequence element. By induction, after the final iteration, `third` equals `Tn`.

## Python Solution

```python
class Solution:
    def tribonacci(self, n: int) -> int:
        if n == 0:
            return 0

        if n == 1 or n == 2:
            return 1

        first, second, third = 0, 1, 1

        for _ in range(3, n + 1):
            next_value = first + second + third
            first = second
            second = third
            third = next_value

        return third
```

The implementation begins by handling the base cases. Since the first three values of the sequence are explicitly defined, returning them immediately avoids unnecessary computation and prevents incorrect behavior for small inputs.

Next, three variables are initialized to represent `T0`, `T1`, and `T2`. These variables form a rolling window over the sequence.

The loop starts at index `3` because the first three values are already known. During each iteration, the next Tribonacci number is computed as the sum of the current three tracked values. After computing the next value, the variables shift forward so that they continue representing the latest three numbers.

Finally, after the loop completes, `third` contains the desired `n-th` Tribonacci number and is returned.

## Go Solution

```go
func tribonacci(n int) int {
    if n == 0 {
        return 0
    }

    if n == 1 || n == 2 {
        return 1
    }

    first, second, third := 0, 1, 1

    for i := 3; i <= n; i++ {
        nextValue := first + second + third
        first = second
        second = third
        third = nextValue
    }

    return third
}
```

The Go implementation follows the same logic as the Python solution. Base cases are handled first, followed by iterative computation using three integer variables.

One small Go-specific difference is variable declaration syntax. Go uses `:=` for concise initialization. The loop variable `i` explicitly controls iteration from `3` through `n`.

Integer overflow is not a concern here because the problem guarantees that all answers fit within a 32-bit signed integer, and Go's `int` type comfortably handles these values.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | We iterate once from `3` to `n` |
| Space | O(1) | Only three variables are stored regardless of input size |

The runtime is linear because we compute each Tribonacci number exactly once. Since `n` is at most `37`, the algorithm is extremely fast in practice.

The space complexity is constant because no extra array or recursion stack is used. Only three integer variables are maintained throughout execution.

## Edge Cases

One important edge case is when `n = 0`. A naive implementation that jumps immediately into recurrence logic could accidentally return an incorrect result or even fail due to invalid indexing. Our implementation explicitly checks for `n == 0` and returns `0` immediately.

Another critical case is when `n = 1` or `n = 2`. Since these values are predefined and not generated through recurrence, attempting to run the iterative loop without handling them separately could produce incorrect answers. The implementation handles both values directly by returning `1`.

A third edge case is the maximum input value, `n = 37`. While this is still small, recursive brute-force solutions can become unnecessarily expensive due to repeated computation. The iterative dynamic programming solution avoids this issue entirely by computing each value exactly once while using constant memory.
