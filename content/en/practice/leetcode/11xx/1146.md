## Solution

## Problem Understanding

The problem asks us to design a data structure, `SnapshotArray`, which behaves like an array but also allows us to take snapshots and retrieve historical values efficiently. In essence, we want a versioned array where each snapshot is immutable, and we can query the array as it existed at the time of any snapshot.

The input consists of operations on this data structure. Initially, we define the length of the array, which is filled with zeros. We then perform `set` operations to update elements, `snap` operations to take a snapshot and get its ID, and `get` operations to query the value at a particular index in a past snapshot.

The expected output corresponds to the results of `snap()` and `get()` operations. `snap()` returns a unique incremental `snap_id`, and `get(index, snap_id)` returns the value at that index when the snapshot with `snap_id` was taken.

Constraints indicate that the array can be large (up to 50,000 elements), and there can be many operations (up to 50,000 total calls). This rules out solutions that store the entire array for every snapshot since that could require up to 2.5 billion elements in the worst case. Efficient storage and retrieval are necessary, making a naive full-copy approach infeasible.

Edge cases include multiple `set` operations on the same index before taking a snapshot, querying snapshots for indices that were never modified, and snapshots taken without any changes.

## Approaches

The brute-force approach is to maintain a list of full arrays for each snapshot. For each `set`, we update the current array. For each `snap`, we clone the current array and append it to a list of snapshots. For `get(index, snap_id)`, we simply access `snapshots[snap_id][index]`. This works correctly but requires O(length × number of snapshots) space, which is unacceptable for large inputs.

The optimal approach leverages the observation that most array elements do not change between snapshots. Instead of storing full arrays, we can maintain, for each index, a list of `(snap_id, value)` pairs representing only the snapshots where that index changed. This allows `set` to update the current value, `snap` to increment the snapshot ID, and `get` to use binary search over the changes list to find the latest value for that index at the requested snapshot. This reduces storage and enables efficient retrieval.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(1) `set`, O(length) `snap`, O(1) `get` | O(length × snaps) | Stores entire array for every snapshot, very high space usage |
| Optimal | O(1) `set`, O(1) `snap`, O(log S) `get` | O(S) | Stores only modified values per index with their `snap_id`, efficient for large arrays |

## Algorithm Walkthrough

1. Initialize the array by creating a list of lists, where each inner list stores `(snap_id, value)` pairs. Initially, each index list contains `(−1, 0)` representing the initial value before any snapshots.
2. Maintain a global variable `current_snap_id` starting at 0.
3. For `set(index, val)`, update the value for that index in a temporary current state. If the last recorded `snap_id` in the index's list is the same as `current_snap_id`, overwrite it. Otherwise, append `(current_snap_id, val)` to the list.
4. For `snap()`, increment `current_snap_id` and return the previous value as the snapshot ID.
5. For `get(index, snap_id)`, perform binary search in the list of `(snap_id, value)` for the index to find the greatest `snap_id` less than or equal to the requested `snap_id`. Return the corresponding value.

This works because for each index, the list of changes is always sorted by `snap_id`. The binary search guarantees that we retrieve the most recent value up to the requested snapshot.

## Python Solution

```
from bisect import bisect_right

class SnapshotArray:

    def __init__(self, length: int):
        self.data = [[(-1, 0)] for _ in range(length)]
        self.current_snap_id = 0

    def set(self, index: int, val: int) -> None:
        if self.data[index][-1][0] == self.current_snap_id:
            self.data[index][-1] = (self.current_snap_id, val)
        else:
            self.data[index].append((self.current_snap_id, val))

    def snap(self) -> int:
        snap_id = self.current_snap_id
        self.current_snap_id += 1
        return snap_id

    def get(self, index: int, snap_id: int) -> int:
        arr = self.data[index]
        i = bisect_right(arr, (snap_id, float('inf'))) - 1
        return arr[i][1]
```

The implementation maintains a list of changes for each index, efficiently updating or appending values during `set`. The `snap` function increments the snapshot ID and returns the previous ID. `get` uses binary search to find the closest value at or before the requested snapshot.

## Go Solution

```
import "sort"

type SnapshotArray struct {
    data [][][2]int
    snapID int
}

func Constructor(length int) SnapshotArray {
    data := make([][][2]int, length)
    for i := 0; i < length; i++ {
        data[i] = append(data[i], [2]int{-1, 0})
    }
    return SnapshotArray{data: data, snapID: 0}
}

func (this *SnapshotArray) Set(index int, val int)  {
    arr := this.data[index]
    n := len(arr)
    if arr[n-1][0] == this.snapID {
        arr[n-1][1] = val
    } else {
        this.data[index] = append(arr, [2]int{this.snapID, val})
    }
}

func (this *SnapshotArray) Snap() int {
    snap := this.snapID
    this.snapID++
    return snap
}

func (this *SnapshotArray) Get(index int, snap_id int) int {
    arr := this.data[index]
    l, r := 0, len(arr)
    for l < r {
        m := l + (r-l)/2
        if arr[m][0] <= snap_id {
            l = m + 1
        } else {
            r = m
        }
    }
    return arr[l-1][1]
}
```

In Go, slices replace Python lists. Binary search is implemented manually instead of using a library. The logic is otherwise identical to Python, ensuring efficient storage and retrieval.

## Worked Examples

Example: `["SnapshotArray","set","snap","set","get"]` with arguments `[[3],[0,5],[],[0,6],[0,0]]`.

| Step | Operation | snapID | Index 0 changes | Output |
| --- | --- | --- | --- | --- |
| 1 | SnapshotArray(3) | 0 | [(-1,0)] | null |
| 2 | set(0,5) | 0 | [(0,5)] | null |
| 3 | snap() | 0 | [(0,5)] | 0 |
| 4 | set(0,6) | 1 | [(0,5),(1,6)] | null |
| 5 | get(0,0) | 1 | [(0,5),(1,6)] | 5 |

The `get` function uses binary search to find the value with the largest snap_id ≤ 0, which is 5.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(1) `set`, O(1) `snap`, O(log S) `get` | `set` appends or overwrites, `snap` increments a counter, `get` performs binary search over changes per index |
| Space | O(S) | Stores only the changes for each index, where S is total number of `set` operations |

The approach is efficient because we avoid duplicating the entire array per snapshot and leverage the fact that only a few indices change between snapshots.
