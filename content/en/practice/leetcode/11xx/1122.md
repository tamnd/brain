# LeetCode 1122, Relative Sort Array

## Problem Understanding

This problem asks us to reorder the elements of `arr1` according to the ordering defined by `arr2`.

The key idea is that `arr2` acts like a custom priority list. Every element that appears in `arr2` must appear first in the final result, and they must appear in exactly the same relative order as in `arr2`.

For example, if `arr2 = [2,1,4,3]`, then:

- All `2`s must come first
- Then all `1`s
- Then all `4`s
- Then all `3`s

Any remaining elements from `arr1` that do not exist in `arr2` should be placed at the end in normal ascending order.

The input consists of:

- `arr1`, the array we must sort
- `arr2`, the array that defines the custom ordering

The problem guarantees several important things:

- Every element in `arr2` exists in `arr1`
- All elements in `arr2` are distinct
- Values are between `0` and `1000`

These guarantees simplify the implementation because we never need to handle missing keys from `arr2`, and the small value range makes counting techniques especially attractive.

An important detail is that duplicates matter. If a value appears multiple times in `arr1`, all occurrences must appear in the final output.

Some edge cases that could cause bugs in a naive implementation include:

- Elements appearing many times in `arr1`
- `arr1` containing values not present in `arr2`
- `arr2` containing only a small subset of values
- Arrays with only one element
- Remaining values needing ascending sorting after the relative ordering section

Because the constraints are small, several approaches are possible, but we can still design a clean and efficient solution.

## Approaches

### Brute Force Approach

A straightforward solution is to process each element in `arr2` one at a time and scan through `arr1` repeatedly.

For every value in `arr2`, we iterate through `arr1` and collect matching elements into the result. After processing all values from `arr2`, we scan `arr1` again to collect elements not present in `arr2`, sort them, and append them to the result.

This works because:

- We explicitly preserve the order defined in `arr2`
- We collect duplicates correctly
- We sort leftover elements normally

However, this approach repeatedly scans the entire array. If `arr1` has length `n` and `arr2` has length `m`, we may perform roughly `m * n` comparisons.

### Optimal Approach

A better solution uses counting.

Since all values are between `0` and `1000`, we can count how many times each number appears in `arr1`.

Then:

1. Process values in the order given by `arr2`
2. Append each value the required number of times
3. Remove those counts
4. Finally iterate through all remaining numbers in ascending order

The important insight is that we do not actually need to sort the portion controlled by `arr2`. The order is already predefined. We only need frequency information.

This transforms the problem into a counting and reconstruction problem instead of a general sorting problem.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m × n + k log k) | O(k) | Repeatedly scans `arr1`, then sorts leftovers |
| Optimal | O(n + maxValue) | O(maxValue) | Uses frequency counting for efficient reconstruction |

Here:

- `n` is the size of `arr1`
- `m` is the size of `arr2`
- `k` is the number of leftover elements
- `maxValue = 1001`

## Algorithm Walkthrough

### Optimal Counting Approach

1. Create a frequency array or hash map to count occurrences of every number in `arr1`.

We use counting because the value range is small and fixed. This allows constant-time frequency updates and retrieval.
2. Traverse `arr1` and increment the count for each value.

After this step, we know exactly how many times every number appears.
3. Create an empty result array.

This array will store the final relative ordering.
4. Iterate through each number in `arr2`.

Since `arr2` defines the required ordering, we process values in this exact sequence.
5. For each number in `arr2`, append it to the result as many times as its frequency count.

If `2` appears three times in `arr1`, we append three `2`s.
6. After appending all occurrences, reset that frequency count to zero.

This prevents duplicate insertion later.
7. Iterate through all possible values from `0` to `1000`.

Any remaining nonzero counts correspond to values not present in `arr2`.
8. Append remaining values in ascending order according to their frequencies.

Since we iterate from smallest to largest, the leftover section becomes naturally sorted.
9. Return the completed result array.

### Why it works

The algorithm works because it separates the array into two groups:

- Values controlled by the custom ordering in `arr2`
- Values not present in `arr2`

The first group is reconstructed exactly in the required order by processing `arr2` directly. The second group is reconstructed in ascending order by iterating through values from smallest to largest.

Since every element from `arr1` is inserted exactly as many times as it originally appeared, no elements are lost or duplicated incorrectly.

## Python Solution

```
from typing import List

class Solution:
    def relativeSortArray(self, arr1: List[int], arr2: List[int]) -> List[int]:
        frequency = [0] * 1001
        
        # Count occurrences of each number
        for num in arr1:
            frequency[num] += 1
        
        result = []
        
        # Add numbers in the order defined by arr2
        for num in arr2:
            while frequency[num] > 0:
                result.append(num)
                frequency[num] -= 1
        
        # Add remaining numbers in ascending order
        for num in range(1001):
            while frequency[num] > 0:
                result.append(num)
                frequency[num] -= 1
        
        return result
```

The implementation begins by creating a frequency array of size `1001`, because the constraints guarantee that values range from `0` to `1000`.

The first loop counts how many times each value appears in `arr1`.

Next, we process elements from `arr2`. For every value, we append it repeatedly according to its stored frequency. After each insertion, we decrement the count so that the remaining phase does not process it again.

Finally, we iterate through all possible numbers in ascending order. Any remaining positive frequency corresponds to values not included in `arr2`, so they are appended in sorted order automatically.

The result array is then returned.

## Go Solution

```
func relativeSortArray(arr1 []int, arr2 []int) []int {
    frequency := make([]int, 1001)

    // Count occurrences
    for _, num := range arr1 {
        frequency[num]++
    }

    result := make([]int, 0, len(arr1))

    // Add numbers according to arr2 order
    for _, num := range arr2 {
        for frequency[num] > 0 {
            result = append(result, num)
            frequency[num]--
        }
    }

    // Add remaining numbers in ascending order
    for num := 0; num <= 1000; num++ {
        for frequency[num] > 0 {
            result = append(result, num)
            frequency[num]--
        }
    }

    return result
}
```

The Go implementation follows the same logic as the Python solution.

A slice of size `1001` is used for counting frequencies. Go slices are initialized with zero values automatically, so no extra setup is required.

The result slice is initialized with capacity `len(arr1)` to reduce reallocations during appends.

Unlike Python lists, Go slices require explicit `append` operations for dynamic growth, but the overall algorithm remains identical.

There are no integer overflow concerns because values and array sizes are very small.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n + 1001) | Count frequencies and iterate through fixed value range |
| Space | O(1001) | Frequency array stores counts for values 0 to 1000 |

The counting phase processes every element of `arr1` exactly once, giving `O(n)` time.

The reconstruction phase iterates through:

- All elements of `arr2`
- The fixed range `0` to `1000`

Because `1001` is constant, the total complexity simplifies to linear time relative to the input size.

The space complexity comes from the frequency array. Since its size is fixed and independent of input size, it is technically constant space, though it is commonly written as `O(1001)` for clarity.

## Edge Cases

### Duplicate Values Appearing Many Times

A common source of bugs is handling repeated numbers incorrectly. For example, if `2` appears five times in `arr1`, all five occurrences must appear consecutively when processing `arr2`.

The frequency-counting approach handles this naturally because the count explicitly tracks how many times each value appears. The reconstruction phase inserts exactly that many copies.

### Elements Not Present in arr2

Another tricky case occurs when many elements are not included in `arr2`. These values must not follow original order from `arr1`. Instead, they must be sorted in ascending order.

The implementation handles this by processing all remaining frequency counts from `0` through `1000` after finishing the `arr2` section. Since iteration is ascending, the leftover portion is automatically sorted.

### Minimum and Maximum Boundary Values

Values can range from `0` to `1000`. Boundary handling mistakes are common when using arrays for counting.

The implementation allocates a frequency array of size `1001`, ensuring both endpoints are valid indices. Iteration also explicitly includes `1000`, preventing off-by-one errors.
