## Solution

## Problem Understanding

This problem gives us a database table named `Spending` that records how much each user spent on a shopping platform on a particular date and device type.

Each row contains:

| Column | Meaning |
| --- | --- |
| `user_id` | The unique user making purchases |
| `spend_date` | The date of the purchase |
| `platform` | Either `mobile` or `desktop` |
| `amount` | Amount spent on that platform that day |

The key detail is that `(user_id, spend_date, platform)` is unique. This means a user can have at most one row per platform per day.

The problem asks us to produce aggregated statistics for every date. For each date, we must classify users into exactly one of three categories:

| Category | Meaning |
| --- | --- |
| `mobile` | User used only mobile on that date |
| `desktop` | User used only desktop on that date |
| `both` | User used both mobile and desktop on that date |

For every date and category, we need to compute:

1. `total_amount` = total money spent by users in that category
2. `total_users` = number of users in that category

The important detail is that a user who used both platforms belongs only to the `both` category. Their spending from both platforms must be summed together.

The output must contain all three categories for every date, even if no users belong to one category. In that situation, the totals should be `0`.

For example, suppose a user spends:

| user_id | date | platform | amount |
| --- | --- | --- | --- |
| 1 | 2019-07-01 | mobile | 100 |
| 1 | 2019-07-01 | desktop | 100 |

This user belongs to the `both` category for `2019-07-01`, not separately to `mobile` and `desktop`.

The constraints are not explicitly listed in the statement, but because this is a database aggregation problem, we should expect potentially large datasets. That means we need a grouped aggregation solution instead of repeatedly scanning the table.

Several edge cases are important:

- A date may have users only on one platform.
- A date may have no `both` users, but we still must output a `both` row with zeros.
- A user may appear only once on a date.
- A user using both platforms must not be double counted in `mobile` or `desktop`.
- Multiple users can belong to the same category on the same date.

## Approaches

### Brute Force Approach

A brute force solution would process each date independently and repeatedly scan the table to determine which users belong to which category.

For every date:

1. Collect all users who made purchases.
2. For each user, scan all rows for that date.
3. Determine whether the user used:

- only mobile
- only desktop
- both
4. Aggregate totals accordingly.

This approach is correct because every user is explicitly analyzed for each date. However, it is inefficient because the table may be scanned many times.

If there are `n` rows and many distinct dates/users, repeated rescanning creates quadratic behavior.

### Optimal Approach

The key insight is that classification can be determined using grouped aggregation.

For every `(user_id, spend_date)` pair, we can:

1. Count how many distinct platforms were used.
2. Sum the spending amount.

If the distinct platform count is:

- `1` and platform is `mobile`, classify as `mobile`
- `1` and platform is `desktop`, classify as `desktop`
- `2`, classify as `both`

After classifying every user-date pair, we can aggregate again by:

- `spend_date`
- derived category

This transforms the problem into two clean aggregation stages.

The final requirement, ensuring every date contains all three categories, can be handled by generating all `(date, category)` combinations using a cross join and then left joining the computed aggregates.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeatedly rescans rows for each user/date |
| Optimal | O(n) | O(n) | Uses grouped aggregation and classification |

## Algorithm Walkthrough

1. First, group records by `(user_id, spend_date)`.

This lets us analyze each user's activity for a single day independently.
2. For every grouped user-date pair, compute:

- total spending amount
- distinct platform count
- platform name if only one exists

This step determines whether the user used:

- only mobile
- only desktop
- both platforms
3. Derive the category.

The logic is:

- If distinct platform count is `2`, category is `both`
- Otherwise, category equals the single platform value
4. Aggregate again by `(spend_date, category)`.

For each category/date pair:

- sum all user spending
- count users
5. Generate all required output rows.

The problem requires every date to contain:

- mobile
- desktop
- both

even if totals are zero.

To achieve this:

- extract all distinct dates
- create a static table containing the three platform labels
- cross join them
6. Left join the computed aggregates onto the complete set of rows.

Missing values become `NULL`, which are converted to `0` using `COALESCE`.

### Why it works

Every user-date pair is classified exactly once based on the set of platforms used that day. Since the categories are mutually exclusive, no user can belong to more than one category for the same date. Aggregating after classification guarantees correct user counts and spending totals.

## Python Solution

```
class Solution:
    def userPurchase(self):
        return """
        WITH user_daily AS (
            SELECT
                user_id,
                spend_date,
                CASE
                    WHEN COUNT(DISTINCT platform) = 2 THEN 'both'
                    ELSE MAX(platform)
                END AS platform,
                SUM(amount) AS total_amount
            FROM Spending
            GROUP BY user_id, spend_date
        ),
        
        aggregated AS (
            SELECT
                spend_date,
                platform,
                SUM(total_amount) AS total_amount,
                COUNT(*) AS total_users
            FROM user_daily
            GROUP BY spend_date, platform
        ),
        
        platforms AS (
            SELECT 'mobile' AS platform
            UNION ALL
            SELECT 'desktop'
            UNION ALL
            SELECT 'both'
        ),
        
        dates AS (
            SELECT DISTINCT spend_date
            FROM Spending
        )
        
        SELECT
            d.spend_date,
            p.platform,
            COALESCE(a.total_amount, 0) AS total_amount,
            COALESCE(a.total_users, 0) AS total_users
        FROM dates d
        CROSS JOIN platforms p
        LEFT JOIN aggregated a
            ON d.spend_date = a.spend_date
           AND p.platform = a.platform
        """
```

The solution is organized into several Common Table Expressions to make each stage easy to understand.

The `user_daily` CTE performs the most important transformation. It groups rows by `(user_id, spend_date)` and determines whether the user belongs to `mobile`, `desktop`, or `both`.

The expression:

```
COUNT(DISTINCT platform) = 2
```

detects whether the user used both platforms.

If not, the platform is simply the single platform value for that group.

The `aggregated` CTE then combines all users in the same category/date pair. It calculates:

- total spending
- total users

The `platforms` CTE creates the required fixed categories. This guarantees the final output always contains all three rows per date.

The `dates` CTE extracts all unique dates from the table.

Finally:

- `CROSS JOIN` creates every `(date, platform)` combination
- `LEFT JOIN` attaches computed results
- `COALESCE` converts missing values into zeros

## Edge Cases

### Users Who Use Both Platforms

A common mistake is double counting users who purchased on both mobile and desktop. A naive solution might include them once in `mobile` and once in `desktop`.

This implementation avoids that issue by first grouping by `(user_id, spend_date)` and classifying users before aggregation. Once a user is labeled `both`, they are counted only in that category.

### Dates Missing One or More Categories

Some dates may contain only mobile users or only desktop users. A naive aggregation would completely omit missing categories from the result.

The implementation handles this using a `CROSS JOIN` between all distinct dates and the fixed platform list. This guarantees every date has exactly three rows.

### Users With Only One Transaction

A user may appear only once for a given date. In that case:

- `COUNT(DISTINCT platform)` equals `1`
- the platform category becomes the actual platform value

The logic correctly classifies such users into either `mobile` or `desktop`.

### Multiple Users in the Same Category

Many users may belong to the same category on a single date. The second aggregation step correctly sums all spending and counts all users for that category/date combination.

### Empty Category Aggregates

When no rows exist for a category/date pair, the `LEFT JOIN` produces `NULL` values.

Using `COALESCE(..., 0)` ensures the output contains valid numeric zeros instead of `NULL`.
