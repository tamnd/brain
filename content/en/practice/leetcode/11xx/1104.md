## Solution

## Problem Understanding

This problem describes an infinite perfect binary tree where every node has exactly two children. Unlike a normal binary tree labeling scheme, the labels alternate direction on every level.

In a standard binary tree labeled level by level, the labels increase from left to right:

```
Level 1:               1
Level 2:            2     3
Level 3:          4  5  6  7
Level 4:        8 9 10 11 12 13 14 15
```

In this problem, the direction alternates:

- Odd numbered rows are labeled left to right
- Even numbered rows are labeled right to left

So the actual labeling becomes:

```
Level 1:                 1
Level 2:              3     2
Level 3:            4  5  6  7
Level 4:         15 14 13 12 11 10 9 8
```

The input is a single integer `label`, representing a node somewhere in this zigzag-labeled binary tree.

The goal is to return the complete path from the root node `1` down to the given node.

For example:

```
Input: 14
Output: [1,3,4,14]
```

This means that if we follow parent-child relationships in the zigzag tree, the path from the root to node `14` is:

```
1 -> 3 -> 4 -> 14
```

The constraint is:

```
1 <= label <= 10^6
```

This tells us the tree depth is relatively small. Since powers of two grow exponentially:

```
2^20 = 1,048,576
```

the deepest possible level is only around 20. That means even algorithms proportional to tree height are extremely efficient.

Several edge cases are important:

- `label = 1`, the root node itself
- Labels on reversed levels
- Labels at level boundaries such as powers of two
- Large labels near `10^6`
- Correctly converting between zigzag labels and normal binary tree positions

A naive implementation can easily make mistakes when computing parents on reversed levels because the visual order differs from the actual binary tree structure.

## Approaches

### Brute Force Approach

A brute force solution would explicitly build levels of the tree one by one while simulating the zigzag labeling pattern.

For each level:

1. Generate all node labels
2. Reverse the order if the level is even
3. Store parent-child relationships
4. Continue until reaching the target label
5. Backtrack from the target to the root

This works because it directly models the tree exactly as described.

However, it is inefficient because the number of nodes doubles every level. Building all nodes up to a large label wastes memory and computation.

Even though the maximum depth is small, constructing the full tree is unnecessary because the parent relationship can be computed mathematically.

### Key Insight

The important observation is that the underlying tree structure is still a normal binary tree.

Only the labels are reversed on alternating levels.

If we can convert a zigzag label into its corresponding normal binary tree label, then computing the parent becomes easy because in a normal binary tree:

```
parent = label // 2
```

For a given level:

- Minimum label is `2^(level-1)`
- Maximum label is `2^level - 1`

If a level is reversed, the mapping between zigzag and normal labels is:

```
normal = level_min + level_max - zigzag
```

This mirrors the value across the level.

Using this transformation repeatedly allows us to move upward through the tree without explicitly constructing it.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(label) | O(label) | Explicitly constructs tree levels and parent mappings |
| Optimal | O(log label) | O(log label) | Uses mathematical level transformations and parent computation |

## Algorithm Walkthrough

### Optimal Algorithm

1. First determine which level the given label belongs to.

In a binary tree:

```
Level 1: 1 node
Level 2: 2 nodes
Level 3: 4 nodes
```

The range for level `k` is:

```
[2^(k-1), 2^k - 1]
```

We repeatedly increase the level until the label fits within the range.
2. Create an empty result list.

We will move upward from the target node toward the root, storing labels along the way.
3. While the current label is greater than or equal to 1:

- Append the current zigzag label to the result.
- Compute the minimum and maximum labels for the current level.
- Convert the zigzag label into the corresponding normal binary tree label.

The conversion formula is:

```
normal_label = level_min + level_max - zigzag_label
```
4. Compute the parent in the normal binary tree.

In a standard binary tree:

```
parent_normal = normal_label // 2
```
5. Move to the previous level.

Since the parent level may also be reversed, convert the parent back into zigzag form using the same mirroring logic.
6. Repeat until reaching the root.
7. The path was collected from bottom to top, so reverse the result before returning it.

### Why it works

At every level, the zigzag ordering is simply a mirrored version of the normal left-to-right ordering. The formula:

```
level_min + level_max - value
```

correctly converts between the two representations because it reflects the label across the midpoint of the level.

Once converted into normal form, the parent relationship becomes identical to a standard binary tree, where the parent is always `label // 2`.

By repeatedly converting between zigzag and normal representations while moving upward, the algorithm follows the exact parent chain in the zigzag tree.

## Python Solution

```
from typing import List

class Solution:
    def pathInZigZagTree(self, label: int) -> List[int]:
        level = 1
        
        while (1 << level) - 1 < label:
            level += 1
        
        path = []
        current = label
        
        while level > 0:
            path.append(current)
            
            level_min = 1 << (level - 1)
            level_max = (1 << level) - 1
            
            normal_label = level_min + level_max - current
            parent_normal = normal_label // 2
            
            level -= 1
            
            if level > 0:
                parent_min = 1 << (level - 1)
                parent_max = (1 << level) - 1
                
                current = parent_min + parent_max - parent_normal
        
        return path[::-1]
```

The implementation begins by determining the level containing the target label. Since each level spans powers of two, bit shifting provides an efficient way to compute boundaries.

The `path` list stores the labels encountered while moving upward toward the root.

Inside the loop, the algorithm computes the current level's minimum and maximum labels. These boundaries allow conversion between zigzag and normal numbering systems.

The formula:

```
normal_label = level_min + level_max - current
```

maps the zigzag label to its normal binary tree equivalent.

After converting to normal form, the parent is computed with integer division by 2.

Because the parent level may also use reversed labeling, the algorithm mirrors the parent label back into zigzag form before continuing.

Finally, since the path was collected bottom-up, reversing the list produces the required root-to-node order.

## Go Solution

```
func pathInZigZagTree(label int) []int {
	level := 1

	for (1<<level)-1 < label {
		level++
	}

	path := []int{}
	current := label

	for level > 0 {
		path = append(path, current)

		levelMin := 1 << (level - 1)
		levelMax := (1 << level) - 1

		normalLabel := levelMin + levelMax - current
		parentNormal := normalLabel / 2

		level--

		if level > 0 {
			parentMin := 1 << (level - 1)
			parentMax := (1 << level) - 1

			current = parentMin + parentMax - parentNormal
		}
	}

	for i, j := 0, len(path)-1; i < j; i, j = i+1, j-1 {
		path[i], path[j] = path[j], path[i]
	}

	return path
}
```

The Go implementation follows the same logic as the Python solution.

Slices are used instead of Python lists. Since Go does not provide built-in list reversal, the solution manually reverses the slice in place using two pointers.

Integer operations are safe because the maximum label is only `10^6`, which easily fits inside Go's `int` type.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(log label) | One iteration per tree level |
| Space | O(log label) | The path stores one node per level |

The height of the binary tree grows logarithmically with respect to the label value. Since each loop iteration moves exactly one level upward, the algorithm performs only `O(log label)` operations.

The result list also contains one node per level, so the auxiliary space usage is logarithmic as well.

## Edge Cases

One important edge case is when `label = 1`. This represents the root node itself. A buggy implementation might attempt to compute a parent for the root or incorrectly reverse an empty path. The current implementation handles this naturally because the loop executes exactly once and immediately terminates after adding `1` to the result.

Another important edge case occurs at level boundaries, especially powers of two such as `8`, `16`, and `32`. These labels represent the first node on a level, and reversed levels can make parent calculations error-prone. The implementation avoids mistakes by always computing level ranges explicitly using powers of two before applying the mirroring transformation.

A third tricky case involves reversed levels themselves. For example, on level 4 the labels run from `15` down to `8` instead of left to right. A naive implementation might incorrectly apply standard parent relationships directly to zigzag labels. This solution correctly converts every zigzag label into its normal binary tree equivalent before computing the parent, ensuring the tree structure remains accurate.

Large inputs near `10^6` are also important. Recursive implementations could risk unnecessary overhead or stack growth. This solution uses an iterative approach with only logarithmic iterations, so it efficiently handles the largest valid labels.
