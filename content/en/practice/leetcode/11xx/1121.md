# LeetCode 1121 - Divide Array Into Increasing Subsequences

## Problem Understanding

This problem gives us a sorted integer array `nums` and an integer `k`. We must determine whether it is possible to divide the entire array into one or more disjoint increasing subsequences, where every subsequence has length at least `k`.

The word "disjoint" means every element from the original array must belong to exactly one subsequence. No element can be reused or skipped.

The subsequences must be increasing, which means each next element in a subsequence must be strictly greater than the previous one. Since the array is already sorted in non-decreasing order, duplicate values appear together. This detail becomes very important because duplicate values cannot appear in the same increasing subsequence.

For example, consider:

```
nums = [1,2,2,3,3,4,4], k = 3
```

One valid division is:

```
[1,2,3,4]
[2,3,4]
```

Both subsequences are strictly increasing, and both have length at least `3`.

The challenge is not to explicitly construct the subsequences, but simply to determine whether such a division is possible.

The constraints are large:

- `nums.length` can be up to `100000`
- values can also be up to `100000`

Because of this, any exponential or backtracking solution is infeasible. We need a linear or near-linear algorithm.

An important observation is that duplicate values create a lower bound on the number of subsequences we must form. If a value appears `m` times, then at least `m` different subsequences are required, because duplicates cannot coexist in the same strictly increasing subsequence.

Several edge cases are important:

- Arrays where all elements are identical
- Arrays where `k = 1`
- Arrays where one value appears many times
- Arrays that are already strictly increasing
- Arrays whose total length is too small to support enough subsequences

The problem guarantees the array is already sorted, which greatly simplifies the reasoning.

## Approaches

### Brute Force Approach

A brute-force approach would attempt to explicitly build all valid increasing subsequences.

We could recursively assign each number to one of several subsequences while ensuring:

- subsequences remain strictly increasing
- every subsequence eventually reaches length at least `k`

This approach is correct because it explores every possible partitioning arrangement.

However, it becomes computationally infeasible very quickly. Every element may have multiple possible subsequences it could join, leading to exponential branching. With up to `100000` elements, this is completely impractical.

### Key Insight

The critical observation is that duplicate values determine the minimum number of subsequences required.

Suppose some value appears `maxFrequency` times.

Because increasing subsequences cannot contain duplicates, each occurrence of that value must go into a different subsequence. Therefore:

```
minimum subsequences needed = max frequency of any value
```

Now suppose we must create `maxFrequency` subsequences.

Since all array elements must be distributed among them, the average subsequence length becomes:

```
len(nums) / maxFrequency
```

To ensure every subsequence has length at least `k`, we need:

```
len(nums) >= maxFrequency * k
```

That single condition is both necessary and sufficient.

If the array contains enough total elements to give every required subsequence at least `k` elements, then the sorted structure guarantees we can distribute them correctly.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) or worse | O(n) | Tries all subsequence assignments |
| Optimal | O(n) | O(1) | Only tracks maximum duplicate frequency |

## Algorithm Walkthrough

1. Initialize two variables:

- `currentFrequency`, to count consecutive equal elements
- `maxFrequency`, to track the largest frequency seen so far
2. Traverse the sorted array from left to right.
3. If the current value equals the previous value, increment `currentFrequency` because we are still counting duplicates of the same number.
4. Otherwise, reset `currentFrequency` to `1` because a new value begins.
5. After updating the current frequency, update `maxFrequency` if needed.
6. After processing the entire array, compute whether:

```
len(nums) >= maxFrequency * k
```
7. If this condition is true, return `True`. Otherwise, return `False`.

### Why it works

The key invariant is that duplicate values must belong to different increasing subsequences.

If some value appears `m` times, then at least `m` subsequences are required. Therefore, the maximum frequency across all values determines the minimum number of subsequences we must create.

Once we know how many subsequences are required, the only remaining question is whether the array contains enough total elements so that each subsequence can reach length `k`.

If:

```
len(nums) >= maxFrequency * k
```

then it is always possible to distribute elements across the subsequences while preserving increasing order.

## Python Solution

```
from typing import List

class Solution:
    def canDivideIntoSubsequences(self, nums: List[int], k: int) -> bool:
        max_frequency = 1
        current_frequency = 1

        for i in range(1, len(nums)):
            if nums[i] == nums[i - 1]:
                current_frequency += 1
            else:
                current_frequency = 1

            max_frequency = max(max_frequency, current_frequency)

        return len(nums) >= max_frequency * k
```

The implementation relies entirely on the fact that the array is sorted.

We scan the array once and count runs of identical values. Since duplicates appear consecutively in a sorted array, we can compute frequencies without using a hash map.

`current_frequency` tracks the size of the current duplicate block. Whenever the value changes, the count resets to `1`.

`max_frequency` stores the largest duplicate count encountered.

At the end, we verify whether the array contains enough total elements to support that many subsequences, each of length at least `k`.

## Go Solution

```
func canDivideIntoSubsequences(nums []int, k int) bool {
    maxFrequency := 1
    currentFrequency := 1

    for i := 1; i < len(nums); i++ {
        if nums[i] == nums[i-1] {
            currentFrequency++
        } else {
            currentFrequency = 1
        }

        if currentFrequency > maxFrequency {
            maxFrequency = currentFrequency
        }
    }

    return len(nums) >= maxFrequency*k
}
```

The Go implementation follows the same logic as the Python solution.

Go uses explicit integer variables instead of Python's dynamic integers. Integer overflow is not a concern here because the constraints are at most `100000`, well within Go's integer range.

Since the array is guaranteed to contain at least one element, initializing both frequencies to `1` is safe.

## Edge Cases

One important edge case occurs when all elements are identical. For example:

```
nums = [7,7,7,7]
```

Since increasing subsequences cannot contain duplicate values, every `7` must belong to a separate subsequence. This means four subsequences are required. If `k > 1`, the division becomes impossible because there are not enough elements to give each subsequence sufficient length. The implementation correctly detects this because the maximum frequency becomes `4`.

Another important edge case is when `k = 1`. In this situation, every individual element can form its own subsequence. Therefore, the answer is always `True`. The formula naturally handles this case because:

```
len(nums) >= maxFrequency * 1
```

is always true since `maxFrequency` can never exceed `len(nums)`.

A third important edge case involves arrays where one value appears much more frequently than others. Consider:

```
nums = [1,1,1,2,3]
k = 2
```

The value `1` appears three times, so at least three subsequences are required. However, the array only has five total elements, which is insufficient to create three subsequences of length at least `2`. The implementation correctly identifies this using the condition:

```
5 >= 3 * 2
```

which evaluates to false.

A final edge case is the smallest possible input:

```
nums = [1]
k = 1
```

The algorithm initializes frequencies to `1`, skips the loop because there is only one element, and correctly returns `True`.
