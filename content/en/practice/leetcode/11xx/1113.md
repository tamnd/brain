## Solution

## Problem Understanding

This problem is asking us to compute the number of reports made on posts for each report reason that occurred **yesterday** relative to a fixed date, `2019-07-05`. The `Actions` table records user interactions with posts, including views, likes, comments, and reports. Each row in this table has a `user_id`, `post_id`, `action_date`, `action` type, and an optional `extra` column, which for `report` actions contains the reason for reporting.

Our goal is to output a table with two columns: `report_reason` and `report_count`. Each row represents a distinct report reason with the count of reports made **only yesterday** (`2019-07-04`). Duplicate reports for the same post and reason from the same user are counted separately since the table may contain duplicates.

Important constraints include that the table may contain duplicate rows, so we should not assume uniqueness. Only rows where `action` equals `'report'` and `action_date` equals yesterday are relevant. If no reports exist for a given reason, it should be omitted. Edge cases include actions with null `extra` values, multiple reports for the same reason on the same post, and dates outside of the target date.

## Approaches

The **brute-force approach** would involve scanning the entire `Actions` table and filtering rows by `action = 'report'` and `action_date = '2019-07-04'`. Then, for each report, we would maintain a mapping of `extra` values to counts, incrementing every time we encounter the same reason. This is correct because it counts every relevant report accurately. However, if the table is very large, scanning all rows and maintaining counts in an unoptimized way could be slower, especially in SQL if not using aggregation functions.

The **optimal approach** leverages SQL’s aggregation capabilities. By using a `WHERE` clause to filter only relevant rows (`action = 'report'` and the date filter), we reduce the data early. Then we use `GROUP BY extra` and `COUNT(*)` to efficiently compute the number of reports per reason. This approach is optimal because it pushes computation to the database engine, which is designed for aggregations and filtering. Indexing on `action_date` and `action` can further accelerate performance.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(k) | Scan all rows, manually count reports per reason |
| Optimal | O(n) | O(k) | Filter and aggregate using SQL `GROUP BY`, database handles counting efficiently |

## Algorithm Walkthrough

1. **Filter rows**: Select rows from `Actions` where `action` equals `'report'` and `action_date` equals `'2019-07-04'`. This reduces the dataset to only relevant entries and avoids counting unrelated actions.
2. **Group by report reason**: Use the `extra` column to group the filtered rows. Each unique value of `extra` represents a distinct report reason.
3. **Count reports**: For each group, count the number of rows. This directly gives the number of reports for each reason.
4. **Return results**: Output two columns: `report_reason` (the `extra` value) and `report_count` (the computed count). No need for ordering, as the problem allows any order.

**Why it works**: Filtering ensures only relevant reports are considered, grouping guarantees that each reason is aggregated correctly, and counting rows in each group gives the exact number of reports for that reason. Database aggregation functions guarantee correctness even with duplicates.

## Python Solution

```
import sqlite3
from typing import List, Tuple

def reported_posts_yesterday(conn: sqlite3.Connection) -> List[Tuple[str, int]]:
    query = """
        SELECT extra AS report_reason, COUNT(*) AS report_count
        FROM Actions
        WHERE action = 'report' AND action_date = '2019-07-04'
        GROUP BY extra
    """
    cursor = conn.cursor()
    cursor.execute(query)
    results = cursor.fetchall()
    return results
```

The Python solution executes the SQL query using the SQLite database connection. First, it filters for `'report'` actions on `'2019-07-04'`, then groups by `extra` and counts each group. `fetchall()` returns the results as a list of tuples matching the required output format.

## Go Solution

```
package main

import (
    "database/sql"
)

func ReportedPostsYesterday(db *sql.DB) (*sql.Rows, error) {
    query := `
        SELECT extra AS report_reason, COUNT(*) AS report_count
        FROM Actions
        WHERE action = 'report' AND action_date = '2019-07-04'
        GROUP BY extra
    `
    return db.Query(query)
}
```

In Go, the solution uses the `database/sql` package. The query is identical to Python’s. Instead of returning a list, the function returns `*sql.Rows` and an error. Iterating over `*sql.Rows` can then extract each row, handling potential SQL-specific nil values. Go requires explicit error handling for database queries, unlike Python's `fetchall()`.

## Edge Cases

One important edge case is when the `extra` column is null. In this case, the query counts rows where `extra` is null as a distinct group, ensuring no reports are accidentally ignored. Another edge case is multiple reports for the same post by the same user; our algorithm counts each row independently, handling duplicates correctly. A third edge case is when there are no report actions at all; the result should be an empty table, and the SQL aggregation handles this naturally by returning zero rows. All these cases are covered by the filtering and `GROUP BY` logic.
