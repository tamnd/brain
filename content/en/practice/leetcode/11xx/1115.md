## Solution

## Problem Understanding

The problem asks us to coordinate two threads so that they alternately print `"foo"` and `"bar"` a total of `n` times. We are given a class `FooBar` with methods `foo()` and `bar()`. The `foo()` method is called by one thread, and `bar()` by another. Each method loops `n` times, but without proper synchronization, the two threads could interleave in any order, producing outputs like `"foofoobarbar"` or `"barfoo"`. The goal is to ensure the final output is exactly `"foobar"` repeated `n` times.

The input `n` represents the number of `"foobar"` pairs we need to print. The constraints (`1 <= n <= 1000`) indicate that the solution needs to handle moderate iteration counts efficiently, but extreme optimization is not critical. Edge cases to consider include `n = 1`, where only a single `"foobar"` is printed, and maximum `n`, to ensure no deadlocks occur in synchronization. The problem guarantees a positive integer, so we do not need to handle `n <= 0`.

## Approaches

A naive approach would be to simply call `printFoo()` and `printBar()` in a loop in each thread without any coordination. While this runs without errors, the output can be unpredictable because the threads may not execute in strict alternating order. This approach is incorrect because it ignores the concurrency requirement.

The key insight for a correct solution is to enforce ordering between the two threads. One thread should wait until the other has completed its action before proceeding. This can be achieved using synchronization primitives like locks, semaphores, or condition variables. By controlling access to shared resources, we can guarantee that `foo()` prints first, followed by `bar()`, and repeat this `n` times.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Threads run without synchronization; output may be incorrect |
| Optimal | O(n) | O(1) | Uses synchronization primitives to enforce correct alternating printing |

## Algorithm Walkthrough

1. Initialize two semaphores, `foo_sem` and `bar_sem`. Set `foo_sem` to 1 to allow the `foo()` thread to start first, and `bar_sem` to 0 to block `bar()` initially.
2. In the `foo()` method, loop `n` times. On each iteration, acquire `foo_sem`, print `"foo"`, then release `bar_sem` to signal that `bar()` can run.
3. In the `bar()` method, loop `n` times. On each iteration, acquire `bar_sem`, print `"bar"`, then release `foo_sem` to signal that `foo()` can run.
4. This alternating acquire-release mechanism ensures that `"foo"` always precedes `"bar"` and maintains strict alternation.

Why it works: The invariant is that at any given time, exactly one thread is allowed to print, and the other is blocked. `foo_sem` and `bar_sem` control this alternation, so no matter the thread scheduling, `"foobar"` is printed `n` times in order.

## Python Solution

```
from threading import Semaphore
from typing import Callable

class FooBar:
    def __init__(self, n: int):
        self.n = n
        self.foo_sem = Semaphore(1)
        self.bar_sem = Semaphore(0)

    def foo(self, printFoo: 'Callable[[], None]') -> None:
        for _ in range(self.n):
            self.foo_sem.acquire()
            printFoo()
            self.bar_sem.release()

    def bar(self, printBar: 'Callable[[], None]') -> None:
        for _ in range(self.n):
            self.bar_sem.acquire()
            printBar()
            self.foo_sem.release()
```

This implementation uses semaphores to control thread execution. `foo_sem` starts with a permit to allow `foo()` to run first. Each thread acquires its semaphore before printing and releases the other thread's semaphore after printing. This guarantees strict alternation.

## Go Solution

```
package main

import "sync"

type FooBar struct {
	n      int
	fooSem chan struct{}
	barSem chan struct{}
}

func NewFooBar(n int) *FooBar {
	return &FooBar{
		n:      n,
		fooSem: make(chan struct{}, 1),
		barSem: make(chan struct{}, 1),
	}
}

func (fb *FooBar) Foo(printFoo func()) {
	fb.fooSem <- struct{}{}
	for i := 0; i < fb.n; i++ {
		<-fb.fooSem
		printFoo()
		fb.barSem <- struct{}{}
	}
}

func (fb *FooBar) Bar(printBar func()) {
	for i := 0; i < fb.n; i++ {
		<-fb.barSem
		printBar()
		fb.fooSem <- struct{}{}
	}
}
```

In Go, we use buffered channels as semaphores. `fooSem` starts with one value to allow `Foo()` to execute first. Each iteration waits on its channel, executes the print, and signals the other thread by sending a value to the opposite channel. This maintains correct alternating behavior.

## Edge Cases

The first edge case is `n = 1`. The implementation must allow exactly one `"foobar"` pair and not deadlock by waiting on a semaphore that is never released. By initializing `foo_sem` to 1 and `bar_sem` to 0, this is handled correctly.

The second edge case is rapid context switching where the thread scheduler alternates threads in unpredictable ways. Using semaphores ensures strict alternation regardless of OS scheduling.

The third edge case is the maximum `n` value, 1000. This tests both the correctness of repeated semaphore operations and the absence of cumulative performance or memory issues. Our implementation maintains constant space and only loops `n` times, so it handles this scenario safely.
