## Solution

## Problem Understanding

The problem provides a database table named `Views`. Each row represents a viewing event where a specific viewer looked at a specific article on a particular date. Every article has an associated author, identified by `author_id`, and every viewer is identified by `viewer_id`.

The goal is to find all authors who viewed at least one of their own articles. In other words, we need to identify rows where the `author_id` and `viewer_id` are the same person. Once those authors are identified, the result should contain a single column named `id`, sorted in ascending order.

A key detail in the problem statement is that the table has no primary key. This means duplicate rows are allowed. Because of that, the same author may appear multiple times if they viewed their own article more than once. The final result must contain unique author IDs only, so duplicate values must be removed.

The input size is not explicitly stated, but this is a standard SQL filtering and deduplication problem. The important operation is identifying rows where two columns are equal and then returning distinct values in sorted order.

Several edge cases are important. Duplicate rows could cause repeated author IDs if `DISTINCT` is not used. Some authors may never view their own articles, so they should not appear in the output. Some viewers may not be authors at all. The query must correctly filter only rows where `author_id = viewer_id`.

## Approaches

A brute-force approach would conceptually involve scanning every row, storing all matching author IDs in a temporary structure, manually removing duplicates, and finally sorting the results. This works because every row is checked individually, ensuring that all self-view events are detected. However, implementing duplicate removal manually is unnecessary because SQL already provides efficient built-in operations for filtering, deduplication, and sorting.

The key insight is that the problem reduces to a very simple condition: an author viewed their own article exactly when `author_id = viewer_id`. SQL can directly express this condition using a `WHERE` clause. Since duplicate rows may exist, `DISTINCT` ensures each author appears only once. Finally, `ORDER BY` guarantees ascending order.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) | O(n) | Manual filtering, deduplication, and sorting |
| Optimal | O(n log n) | O(n) | SQL filtering with DISTINCT and ORDER BY |

The sorting step dominates the complexity because the matching IDs must be returned in ascending order.

## Algorithm Walkthrough

1. Scan the `Views` table row by row. Each row contains an article ID, author ID, viewer ID, and view date.
2. For every row, compare `author_id` and `viewer_id`. If they are equal, this means the author viewed their own article.
3. Extract the matching `author_id` values. Since the output column must be named `id`, alias the column appropriately.
4. Use `DISTINCT` to eliminate duplicate author IDs. This is necessary because the table may contain duplicate rows or multiple self-view events by the same author.
5. Sort the resulting IDs in ascending order using `ORDER BY`.

Why it works:

The query works because the definition of a self-view is precisely `author_id = viewer_id`. Filtering rows with this condition guarantees that every returned author viewed their own article. Using `DISTINCT` preserves uniqueness, and sorting ensures the output format matches the problem requirements.

## Python Solution

Although this is a database problem and LeetCode expects SQL, the equivalent logic in Python can be expressed as follows for conceptual understanding.

```
# complete solution here

from typing import List, Dict

class Solution:
    def articleViews(self, views: List[Dict]) -> List[int]:
        author_ids = set()

        for row in views:
            if row["author_id"] == row["viewer_id"]:
                author_ids.add(row["author_id"])

        return sorted(author_ids)
```

The implementation iterates through every row in the input data structure. For each row, it checks whether the author and viewer are the same person. Matching author IDs are inserted into a set so duplicates are automatically removed. Finally, the result is sorted before being returned.

This mirrors the SQL solution closely. The `if` statement corresponds to the `WHERE` clause, the set corresponds to `DISTINCT`, and `sorted()` corresponds to `ORDER BY`.

## Go Solution

```
// complete solution here

package main

import "sort"

func articleViews(views []map[string]int) []int {
	authorSet := make(map[int]bool)

	for _, row := range views {
		if row["author_id"] == row["viewer_id"] {
			authorSet[row["author_id"]] = true
		}
	}

	result := []int{}

	for id := range authorSet {
		result = append(result, id)
	}

	sort.Ints(result)

	return result
}
```

The Go implementation follows the same logic as the Python version. A map is used to simulate a set because Go does not provide a built-in set type. Matching author IDs are inserted into the map, ensuring uniqueness. The keys are then collected into a slice and sorted using `sort.Ints`.

Unlike Python, Go requires explicit handling of sorting and map iteration. Integer overflow is not a concern here because IDs are simple integer identifiers.

## SQL Solution

```
SELECT DISTINCT author_id AS id
FROM Views
WHERE author_id = viewer_id
ORDER BY id ASC;
```

This query directly captures the entire problem logic. The `WHERE` clause identifies self-views, `DISTINCT` removes duplicates, and `ORDER BY` ensures ascending order.

## Edge Cases

One important edge case is duplicate rows in the table. Since the problem explicitly states that no primary key exists, the same viewing event may appear multiple times. Without `DISTINCT`, the query would return repeated author IDs. The implementation handles this correctly by explicitly removing duplicates.

Another edge case occurs when no author views their own article. In this situation, the result should be an empty table. The filtering condition naturally handles this because no rows satisfy `author_id = viewer_id`.

A third edge case involves multiple valid authors appearing in unsorted order. SQL tables do not guarantee ordering unless explicitly requested. The implementation uses `ORDER BY id ASC` to ensure the final output always appears in ascending order, exactly as required by the problem statement.
