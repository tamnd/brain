## Solution

## Problem Understanding

This problem asks us to generate **every possible ordering** of the numbers in a given array of **distinct integers**. A permutation is simply a rearrangement of elements. Since the integers are guaranteed to be unique, every arrangement we generate will also be unique.

The input is an integer array `nums`, where each number is distinct. The goal is to return a list containing **all possible permutations** of those numbers. The order of the returned permutations does not matter, meaning any valid ordering of the results is acceptable.

For example, if the input is `[1,2,3]`, we must produce every way those three numbers can be arranged. Since there are `3! = 6` ways to order three distinct elements, the output contains six permutations.

The constraints tell us several important things:

- `1 <= nums.length <= 6`
- All integers are unique
- Values range from `-10` to `10`

The most important constraint here is the array length. Since the maximum size is only `6`, the total number of permutations is at most:

$$6! = 720$$

This is small enough to allow us to explicitly generate every permutation. In many problems, factorial growth would be prohibitively expensive, but here the constraints are intentionally small.

There are several important edge cases to think about upfront. A single-element array such as `[1]` should return `[[1]]`, because there is exactly one arrangement. A two-element array like `[0,1]` should correctly produce both possible orderings. Since the input guarantees distinct integers, we do not need to worry about duplicate permutations, which simplifies the implementation considerably.

## Approaches

### Brute Force Approach

A brute-force way to solve this problem would be to generate every possible arrangement by repeatedly trying all combinations of numbers and then filtering invalid results.

For example, for an array of length `n`, we could attempt every sequence of length `n` where numbers are chosen from the original array, then discard sequences that reuse elements or miss some values. While this would eventually produce all permutations, it performs a large amount of unnecessary work because many generated candidates would be invalid.

Another brute-force interpretation is to recursively try every unused number at every position, which already resembles backtracking. Since the number of valid permutations is inherently `n!`, any correct solution must at least spend time outputting them.

The brute-force approach is correct because it eventually enumerates all valid orderings, but it is inefficient if implemented naively because it explores many invalid states or performs redundant checks.

### Key Insight for the Optimal Approach

The key observation is that **a permutation can be built incrementally**.

At any point, we only need to decide:

> Which unused number should be placed next?

If we already built part of a permutation, we can recursively try each remaining unused number and continue until the permutation reaches length `n`.

This naturally leads to **backtracking**.

Backtracking works well here because:

1. We build a permutation one number at a time.
2. Each number can only be used once in the current permutation.
3. Once we finish exploring one choice, we undo it and try another.

The recursive structure perfectly mirrors the decision-making process of permutation generation.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n<sup>n</sup>) | O(n) | Generates many unnecessary candidate sequences |
| Optimal Backtracking | O(n × n!) | O(n) excluding output | Efficiently generates only valid permutations |

## Algorithm Walkthrough

The optimal solution uses **backtracking with a visited array** to track which numbers have already been included in the current permutation.

1. Create a result list that will store all completed permutations.
2. Maintain a temporary list called `current_permutation`. This stores the permutation we are currently building.
3. Create a boolean array called `used`, where `used[i]` tells us whether `nums[i]` has already been placed into the current permutation.
4. Start a recursive backtracking function.
5. At each recursive call, check whether the current permutation length equals the length of `nums`. If it does, we have completed one valid permutation, so we add a copy of it to the result list.
6. Otherwise, iterate through every number in `nums`.
7. If a number has already been used in the current permutation, skip it because permutations cannot reuse elements.
8. If the number is unused:

- Mark it as used.
- Add it to the current permutation.
- Recursively continue building the rest of the permutation.
9. After recursion finishes, undo the choice:

- Remove the last added number.
- Mark the number as unused again.
10. Continue trying other numbers until all possibilities have been explored.

The reason backtracking works is that each recursive level represents one position in the permutation. By trying every unused number at each level, we systematically generate all valid orderings.

## Python Solution

```
from typing import List

class Solution:
    def permute(self, nums: List[int]) -> List[List[int]]:
        result = []
        current_permutation = []
        used = [False] * len(nums)

        def backtrack():
            if len(current_permutation) == len(nums):
                result.append(current_permutation[:])
                return

            for index in range(len(nums)):
                if used[index]:
                    continue

                used[index] = True
                current_permutation.append(nums[index])

                backtrack()

                current_permutation.pop()
                used[index] = False

        backtrack()
        return result
```

The implementation begins by initializing three important structures.

The `result` list stores every completed permutation. The `current_permutation` list keeps track of the permutation currently being built. The `used` array prevents reuse of elements within the same permutation.

The nested `backtrack()` function performs the recursive search.

The base case occurs when the current permutation length matches the input length. At this point, we have formed a valid permutation, so we append a **copy** of it to `result`. A copy is necessary because lists are mutable, and later modifications would otherwise affect stored permutations.

The recursive step iterates through all indices in `nums`. If an element has already been used, it is skipped. Otherwise, we choose that number by marking it as used and appending it to the current permutation.

After recursively exploring deeper states, we undo the choice using `pop()` and reset the used flag. This backtracking step ensures the algorithm can explore alternative arrangements without interference from previous decisions.

## Go Solution

```
func permute(nums []int) [][]int {
	var result [][]int
	currentPermutation := []int{}
	used := make([]bool, len(nums))

	var backtrack func()

	backtrack = func() {
		if len(currentPermutation) == len(nums) {
			permutationCopy := make([]int, len(currentPermutation))
			copy(permutationCopy, currentPermutation)
			result = append(result, permutationCopy)
			return
		}

		for i := 0; i < len(nums); i++ {
			if used[i] {
				continue
			}

			used[i] = true
			currentPermutation = append(currentPermutation, nums[i])

			backtrack()

			currentPermutation = currentPermutation[:len(currentPermutation)-1]
			used[i] = false
		}
	}

	backtrack()
	return result
}
```

The Go implementation follows the same backtracking logic as the Python version, but there are some language-specific considerations.

Since slices in Go reference shared underlying memory, we must explicitly create a copy of `currentPermutation` before appending it to `result`. Without copying, later modifications would affect previously stored permutations.

Instead of `pop()`, Go removes the last element using slicing:

```
currentPermutation = currentPermutation[:len(currentPermutation)-1]
```

Go does not have nested functions in the same way Python does, so we declare `backtrack` as a function variable before assigning the recursive implementation.

Nil versus empty slices is not a concern here because returning an empty result is impossible under the problem constraints, since `nums.length >= 1`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n × n!) | There are `n!` permutations, and copying each permutation costs `O(n)` |
| Space | O(n) | Recursive stack, `used` array, and current permutation storage, excluding output |

The time complexity comes from the fact that there are exactly `n!` permutations of `n` distinct numbers. For each permutation, we spend `O(n)` time copying the current permutation into the result list, resulting in a total complexity of `O(n × n!)`.

The auxiliary space complexity is `O(n)` because the recursion depth is at most `n`, the `used` array has size `n`, and the temporary permutation also stores at most `n` elements. The output itself requires `O(n × n!)` space, but this is usually excluded from auxiliary space analysis.

## Edge Cases

### Single Element Array

An input like `[1]` is the smallest valid case. A buggy implementation might fail to recognize that a single number is already a complete permutation. Our implementation handles this naturally because the base case triggers immediately after adding the lone element.

### Two Elements

An input such as `[0,1]` tests whether the algorithm properly explores alternative branches. A common bug is forgetting to backtrack correctly, which could prevent generation of `[1,0]`. Since our implementation removes the last element and resets the `used` flag after recursion, both orderings are explored correctly.

### Negative Numbers

The array may contain negative integers, for example `[-1, 2, -3]`. Since the algorithm treats values purely as items to arrange and never relies on ordering or arithmetic, negative values work exactly the same as positive ones.

### Maximum Constraint Size

The largest input size is `6`, which produces `720` permutations. Although factorial growth is expensive in theory, the problem constraints intentionally keep the size small enough for backtracking to finish comfortably. Our implementation efficiently explores all valid permutations without generating invalid candidates.
