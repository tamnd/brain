## Solution

## Problem Understanding

The problem provides a `Customer` table that records restaurant transactions. Each row represents a single customer's visit on a specific date, along with the amount they spent. The primary key is `(customer_id, visited_on)`, which means a customer can appear at most once per day, but multiple customers can visit on the same day.

The goal is to compute a rolling seven day summary of restaurant revenue. For every day starting from the seventh recorded day, we must calculate two values:

1. The total revenue collected during the current day and the previous six days.
2. The average daily revenue during that seven day window.

The result must contain:

- `visited_on`, representing the ending date of the seven day window
- `amount`, representing the total revenue over the seven day period
- `average_amount`, representing the seven day average revenue rounded to two decimal places

The output should be sorted by `visited_on` in ascending order.

An important detail is that multiple customers may visit on the same date. Therefore, before computing rolling sums, we first need to aggregate all transactions for each date into a single daily revenue value.

Another important guarantee is that there is at least one customer every day. This means there are no gaps in the date sequence, which simplifies the moving window calculation because every seven consecutive dates correspond to exactly seven consecutive days.

The problem effectively asks us to compute a sliding window sum and average over daily aggregated revenue values.

Several edge cases are important:

- Multiple customers visiting on the same day, which requires daily aggregation first.
- Exactly seven days of data, where only one result row should be returned.
- Large transaction values, where cumulative sums must still be handled correctly.
- More than seven days of continuous data, where the rolling window must move efficiently instead of recomputing everything repeatedly.

## Approaches

### Brute Force Approach

The most direct approach is to first compute the total revenue for each day, then for every date beginning with the seventh one, recompute the sum of the previous seven days from scratch.

For example, suppose we have daily totals stored in chronological order. For each day `i`, we iterate backward through the previous six days and add all seven daily revenues together. Then we divide by seven to compute the average.

This approach is correct because every output row independently calculates the exact seven day window requested by the problem.

However, the inefficiency comes from repeated work. Consecutive windows overlap heavily. If one window contains days 1 through 7, the next contains days 2 through 8. Recomputing all seven values each time duplicates most of the calculations.

If there are `n` days, and each window recomputes seven values, the total work becomes `O(7n)`, which simplifies to `O(n)`. Since the window size is fixed at seven, this is technically linear. However, in database problems, repeated aggregation logic is less elegant and less scalable compared to proper window function techniques.

### Optimal Approach

The key insight is that adjacent seven day windows overlap almost completely. Instead of recomputing the entire sum every time, we can maintain a running window total.

In SQL, this is naturally expressed using a window function. We first aggregate daily revenue, then use a window frame that includes the current row and the previous six rows.

The window function automatically maintains the rolling sum efficiently.

The process becomes:

1. Aggregate all transactions per day.
2. Use a seven row sliding window to compute rolling sums.
3. Divide by seven and round to two decimal places.
4. Exclude rows before the first complete seven day window.

This approach is cleaner, more scalable, and maps directly to SQL analytical functions.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Recomputes each seven day window independently |
| Optimal | O(n) | O(n) | Uses SQL window functions for rolling aggregation |

## Algorithm Walkthrough

1. First, aggregate all customer transactions by date. Since multiple customers can visit on the same day, we need a single revenue value per day. We use `GROUP BY visited_on` and compute `SUM(amount)`.
2. Sort the aggregated daily revenue chronologically. Window calculations depend on date order, so ascending order is required.
3. Use a window function to compute the rolling seven day total. The window frame should include the current row and the previous six rows:

```
SUM(amount) OVER (
    ORDER BY visited_on
    ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
)
```
4. Compute the moving average by dividing the rolling sum by `7.0`. Using a decimal divisor ensures floating point division instead of integer division.
5. Round the average to two decimal places using `ROUND()`.
6. Exclude rows that do not yet have a full seven day window. This can be done using `ROW_NUMBER()` and filtering rows where the row number is at least 7.
7. Return the final columns ordered by date.

### Why it works

The algorithm works because every row's window frame always contains exactly the current day plus the previous six days. Since the dates are continuous and ordered chronologically, each frame corresponds exactly to the required seven day interval. Aggregating daily revenue first ensures all customer transactions for the same date are combined correctly before rolling calculations begin.

## Python Solution

Although this is a database problem, LeetCode expects a SQL query rather than executable Python code. The following solution represents the SQL logic as a Python multiline string.

```python
class Solution:
    def restaurantGrowth(self) -> str:
        return """
        WITH daily_revenue AS (
            SELECT
                visited_on,
                SUM(amount) AS amount
            FROM Customer
            GROUP BY visited_on
        ),
        rolling_window AS (
            SELECT
                visited_on,
                amount,
                SUM(amount) OVER (
                    ORDER BY visited_on
                    ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
                ) AS total_amount,
                ROW_NUMBER() OVER (
                    ORDER BY visited_on
                ) AS row_num
            FROM daily_revenue
        )
        SELECT
            visited_on,
            total_amount AS amount,
            ROUND(total_amount / 7, 2) AS average_amount
        FROM rolling_window
        WHERE row_num >= 7
        ORDER BY visited_on;
        """
```

The implementation begins by creating a common table expression named `daily_revenue`. This step aggregates all customer transactions occurring on the same day into a single daily revenue value.

The second common table expression, `rolling_window`, computes two analytical values:

- `total_amount`, which stores the rolling seven day revenue total
- `row_num`, which identifies how many chronological days have been processed so far

The window frame `ROWS BETWEEN 6 PRECEDING AND CURRENT ROW` ensures that each rolling sum contains exactly seven consecutive days.

Finally, the outer query filters out rows before the seventh day, because earlier rows do not yet contain a complete seven day window. The result is sorted chronologically before returning.

## Edge Cases

One important edge case occurs when multiple customers visit on the same date. A naive implementation might treat each row independently, causing duplicate dates in the rolling calculation. The correct approach aggregates all rows per day before applying the seven day window, ensuring each date contributes exactly one revenue value.

Another critical edge case is when the dataset contains exactly seven days. In this situation, only one output row should be produced. The filtering condition using `ROW_NUMBER() >= 7` handles this correctly because only the seventh row has a complete window.

A third important edge case involves very large transaction values. Rolling sums can become large if many customers spend substantial amounts. Using SQL aggregate functions avoids manual overflow concerns in most database systems, and dividing by `7.0` ensures precise decimal averaging instead of integer truncation.

A final edge case involves consecutive rolling windows with overlapping data. A naive recomputation strategy could repeatedly sum the same values, increasing complexity unnecessarily. The window function implementation efficiently reuses overlapping computation internally, making the solution scalable and clean.
