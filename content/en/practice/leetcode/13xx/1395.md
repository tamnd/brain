## Solution

## Problem Understanding

This problem asks us to count the number of valid teams of three soldiers from a given array of unique ratings. Each soldier has a distinct rating, and a team is valid if the ratings are strictly increasing or strictly decreasing. In other words, for indices `i < j < k`, the team `(rating[i], rating[j], rating[k])` must satisfy either `rating[i] < rating[j] < rating[k]` or `rating[i] > rating[j] > rating[k]`.

The input is an integer array `rating` of length `n` where each value represents the unique rating of a soldier. The output is a single integer representing the total number of valid teams.

The constraints `3 <= n <= 1000` and `1 <= rating[i] <= 10^5` tell us that a naive O(n^3) brute-force solution might just barely fit for smaller inputs but could be inefficient for the upper bounds of n. The uniqueness guarantee simplifies comparisons because we do not have to handle equal values.

Important edge cases include arrays where all numbers are strictly increasing or strictly decreasing, arrays with mixed random ratings, or arrays with exactly three elements where only one team is possible. Another consideration is when no valid teams exist.

## Approaches

The brute-force approach would iterate through all possible triplets `(i, j, k)` and check if the triplet forms an increasing or decreasing sequence. This is correct but has a cubic time complexity, which can be too slow for `n = 1000`.

A key observation for a more efficient solution is that we can fix the middle soldier `j` and count how many soldiers to the left are smaller or larger than `rating[j]` and how many soldiers to the right are larger or smaller than `rating[j]`. Multiplying these counts gives the total number of valid teams with `j` as the middle soldier. This reduces the problem to a quadratic O(n^2) solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^3) | O(1) | Checks all possible triplets explicitly |
| Optimal | O(n^2) | O(1) | Counts valid increasing and decreasing combinations using the middle element as pivot |

## Algorithm Walkthrough

1. Initialize a variable `total_teams` to zero to store the count of valid teams.
2. Loop through each soldier `j` from index `1` to `n-2`, treating `j` as the middle soldier.
3. For each `j`, count how many soldiers to the left of `j` have ratings smaller than `rating[j]` (`left_smaller`) and larger than `rating[j]` (`left_larger`).
4. Similarly, count how many soldiers to the right of `j` have ratings larger than `rating[j]` (`right_larger`) and smaller than `rating[j]` (`right_smaller`).
5. The number of valid increasing teams with `j` as middle is `left_smaller * right_larger`, and the number of valid decreasing teams is `left_larger * right_smaller`.
6. Add these two counts to `total_teams`.
7. After processing all soldiers as middle, return `total_teams`.

**Why it works:** By fixing the middle element, we can efficiently compute the number of valid combinations on either side. Multiplying the counts ensures that every valid left-right pair with the middle forms a valid team, which guarantees that we count all valid triplets exactly once.

## Python Solution

```python
from typing import List

class Solution:
    def numTeams(self, rating: List[int]) -> int:
        n = len(rating)
        total_teams = 0
        
        for j in range(1, n-1):
            left_smaller = left_larger = 0
            right_smaller = right_larger = 0
            
            # Count elements to the left
            for i in range(j):
                if rating[i] < rating[j]:
                    left_smaller += 1
                elif rating[i] > rating[j]:
                    left_larger += 1
            
            # Count elements to the right
            for k in range(j+1, n):
                if rating[k] > rating[j]:
                    right_larger += 1
                elif rating[k] < rating[j]:
                    right_smaller += 1
            
            total_teams += left_smaller * right_larger + left_larger * right_smaller
        
        return total_teams
```

**Implementation Walkthrough:** The code loops over each soldier as the middle element. It counts smaller and larger elements on both sides to determine how many valid increasing and decreasing teams can be formed. These counts are multiplied and summed for each `j` to compute the total number of valid teams.

## Go Solution

```go
func numTeams(rating []int) int {
    n := len(rating)
    totalTeams := 0
    
    for j := 1; j < n-1; j++ {
        leftSmaller, leftLarger := 0, 0
        rightSmaller, rightLarger := 0, 0
        
        for i := 0; i < j; i++ {
            if rating[i] < rating[j] {
                leftSmaller++
            } else if rating[i] > rating[j] {
                leftLarger++
            }
        }
        
        for k := j+1; k < n; k++ {
            if rating[k] > rating[j] {
                rightLarger++
            } else if rating[k] < rating[j] {
                rightSmaller++
            }
        }
        
        totalTeams += leftSmaller * rightLarger + leftLarger * rightSmaller
    }
    
    return totalTeams
}
```

**Go-specific Notes:** Go uses explicit integer initialization. Loops use `for` syntax with index ranges. There is no need to handle nil slices as the input is guaranteed to have at least 3 elements.

## Worked Examples

| j (middle) | left_smaller | left_larger | right_larger | right_smaller | teams added |
| --- | --- | --- | --- | --- | --- |
| 5 | 1 | 0 | 1 | 2 | 1_1+0_2=1 |
| 3 | 1 | 1 | 1 | 1 | 1_1+1_1=2 |
| 4 | 2 | 1 | 0 | 1 | 2_0+1_1=1 |

Total = 3 teams

**Example 2:** `rating = [2,1,3]`

No valid teams exist because no combination satisfies increasing or decreasing order. Total = 0.

**Example 3:** `rating = [1,2,3,4]`

All triplets `(1,2,3)`, `(1,2,4)`, `(1,3,4)`, `(2,3,4)` are increasing. Total = 4.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n^2) | For each middle element, we iterate over left and right sides, giving quadratic complexity. |
| Space | O(1) | Only constant extra variables are used for counting. |

The quadratic complexity is acceptable for `n <= 1000` because it results in at most 1,000,000 iterations, which is feasible for modern CPUs.
