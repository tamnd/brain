**Topics:** Tree, Depth-First Search, Breadth-First Search, Graph Theory

## Solution

## Problem Understanding

The problem gives us an undirected tree with `n` vertices numbered from `1` to `n`. A frog starts at vertex `1`, and every second it attempts to jump to one of its unvisited neighboring vertices. If there are multiple valid neighbors, the frog chooses uniformly at random among them. Once a vertex has been visited, the frog is never allowed to return to it.

The key detail is what happens when the frog reaches a dead end. If the current vertex has no unvisited neighbors left, the frog stays on that vertex forever. This means the frog does not disappear or stop existing after reaching a leaf node. Instead, its probability mass remains there for all remaining time steps.

We are asked to compute the probability that the frog is exactly on the `target` vertex after `t` seconds.

The input consists of:

- `n`, the number of vertices
- `edges`, the list of undirected edges describing the tree
- `t`, the number of seconds elapsed
- `target`, the vertex whose probability we want to compute

Because the graph is guaranteed to be a tree:

- There are exactly `n - 1` edges
- The graph is connected
- There is exactly one simple path between any two vertices

These properties are extremely important because they eliminate cycles other than the implicit back edge to the parent. Since revisiting is forbidden, the frog essentially performs a traversal outward from the root.

The constraints are relatively small:

- `n <= 100`
- `t <= 50`

This means we do not need extremely advanced optimization techniques. A graph traversal such as BFS or DFS is more than sufficient.

Several edge cases are important:

- The target may be vertex `1`
- The frog may reach the target before time `t`
- The target may be a leaf node
- The target may still have unvisited children when reached
- The frog may be forced to continue moving away from the target
- The frog may never reach the target within `t` seconds

A particularly subtle case occurs when the frog reaches the target too early. If the target still has unvisited neighbors, the frog must continue jumping away, so the final probability becomes `0`. However, if the target is a leaf, the frog remains there forever, and the probability stays valid for all remaining time.

## Approaches

### Brute Force Approach

A brute force solution would explicitly simulate every possible path the frog can take. Starting from node `1`, we recursively try every valid unvisited neighbor at each second. For each complete path of length `t`, we check whether the frog ends on the target and accumulate the corresponding probability.

This approach is correct because it enumerates every possible sequence of jumps and applies the probability rule exactly.

However, the number of possible paths grows exponentially with time. In the worst case, each node may branch into several children, leading to a very large recursion tree. Even though the constraints are moderate, this method performs unnecessary repeated work and becomes inefficient conceptually.

### Optimal Approach

The key observation is that this process naturally forms a probability propagation problem on a tree.

At every node:

- The frog distributes its current probability equally among all unvisited neighbors
- Because the graph is a tree, every node has exactly one parent except the root
- Once we move away from a node, we never revisit it

This means we can perform a BFS or DFS traversal while carrying:

- The current node
- The elapsed time
- The current probability
- The parent node to avoid revisiting

The traversal stops when:

- We reach time `t`
- Or we reach a node with no valid moves

The most elegant solution uses BFS because probabilities evolve level by level in time.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(branching^t) | O(t) | Explores every possible path recursively |
| Optimal | O(n) | O(n) | Traverses each node at most once in the tree |

## Algorithm Walkthrough

1. Build an adjacency list representation of the tree.

Since the graph is undirected, every edge `[u, v]` is added in both directions. An adjacency list allows efficient lookup of neighboring vertices.
2. Initialize a BFS queue.

Each queue entry stores:

- Current node
- Parent node
- Current probability
- Current elapsed time

We start with:

- Node `1`
- Parent `0`
- Probability `1.0`
- Time `0`
3. Process nodes level by level.

For each state, determine how many unvisited neighbors exist. Since the graph is a tree, the only already-visited neighbor is the parent.
4. Handle the target node carefully.

If the current node equals `target`, there are two valid situations where we return the probability:

- We reached exactly at time `t`
- Or the node is a leaf and the frog is forced to stay there forever

Otherwise, the probability is invalid because the frog would have moved away before time `t`.
5. Distribute probability to children.

If the node has `k` unvisited neighbors, each child receives:

```
current_probability / k
```

Push each child into the queue with incremented time.
6. Continue until the queue becomes empty.

If we never satisfy the target condition, return `0.0`.

### Why it works

The algorithm works because probability is conserved and distributed exactly according to the problem rules. Since the graph is a tree, every node has a unique path from the root, so each node is reached through exactly one probability flow. BFS guarantees that states are processed in increasing order of time, matching the frog's movement process exactly.

## Python Solution

```python
from collections import deque
from typing import List

class Solution:
    def frogPosition(self, n: int, edges: List[List[int]], t: int, target: int) -> float:
        graph = [[] for _ in range(n + 1)]

        for u, v in edges:
            graph[u].append(v)
            graph[v].append(u)

        queue = deque([(1, 0, 1.0, 0)])
        # (node, parent, probability, time)

        while queue:
            node, parent, probability, time = queue.popleft()

            children = 0

            for neighbor in graph[node]:
                if neighbor != parent:
                    children += 1

            if node == target:
                if time == t:
                    return probability

                if children == 0 and time < t:
                    return probability

                return 0.0

            if time == t:
                continue

            if children == 0:
                continue

            next_probability = probability / children

            for neighbor in graph[node]:
                if neighbor != parent:
                    queue.append(
                        (neighbor, node, next_probability, time + 1)
                    )

        return 0.0
```

The implementation begins by constructing an adjacency list for the tree. Since the graph is undirected, each edge is inserted in both directions.

A BFS queue is then initialized with the starting state at node `1`. Each queue entry tracks the current node, the parent node, the accumulated probability, and the elapsed time.

For every dequeued state, the algorithm counts the number of valid children. In a tree, the only forbidden neighbor is the parent, so every other adjacent node is an unvisited child.

The target node logic is the most important section. If we reach the target exactly at time `t`, the probability is valid and returned immediately. If we arrive earlier than `t`, the probability only remains valid if the target is a leaf node with no unvisited children. Otherwise, the frog would be forced to leave the target, making the final probability zero.

If movement is still possible and time remains, the probability is divided equally among all valid children, and new BFS states are pushed into the queue.

Because each node is processed at most once through its unique tree path, the traversal remains efficient.

## Go Solution

```go
package main

import "container/list"

func frogPosition(n int, edges [][]int, t int, target int) float64 {
	graph := make([][]int, n+1)

	for _, edge := range edges {
		u, v := edge[0], edge[1]
		graph[u] = append(graph[u], v)
		graph[v] = append(graph[v], u)
	}

	type State struct {
		node        int
		parent      int
		probability float64
		time        int
	}

	queue := list.New()
	queue.PushBack(State{1, 0, 1.0, 0})

	for queue.Len() > 0 {
		front := queue.Front()
		queue.Remove(front)

		state := front.Value.(State)

		node := state.node
		parent := state.parent
		probability := state.probability
		time := state.time

		children := 0

		for _, neighbor := range graph[node] {
			if neighbor != parent {
				children++
			}
		}

		if node == target {
			if time == t {
				return probability
			}

			if children == 0 && time < t {
				return probability
			}

			return 0.0
		}

		if time == t {
			continue
		}

		if children == 0 {
			continue
		}

		nextProbability := probability / float64(children)

		for _, neighbor := range graph[node] {
			if neighbor != parent {
				queue.PushBack(State{
					neighbor,
					node,
					nextProbability,
					time + 1,
				})
			}
		}
	}

	return 0.0
}
```

The Go implementation closely mirrors the Python solution. The adjacency list uses slices, and BFS is implemented using `container/list` as a queue.

A custom `State` struct stores traversal information cleanly. Probability calculations use `float64` to maintain precision.

Unlike Python's dynamic tuples, Go requires explicit struct definitions and type assertions when retrieving queue elements.

- `2`
- `3`
- `7`

```
1 / 3
```

At node `2`, the frog can move to:

- `4`
- `6`

Each receives:

```
(1/3) × (1/2) = 1/6
```

The frog reaches target `4` exactly at time `2`, so the answer is:

```
1/6 = 0.16666666666666666
```

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each node and edge is processed at most once |
| Space | O(n) | Adjacency list and BFS queue store up to all nodes |

The adjacency list contains `2 * (n - 1)` edge entries because the graph is undirected. During BFS, every node is inserted into the queue at most once due to the tree structure and parent tracking. Therefore, both time and space complexity scale linearly with the number of vertices.

## Edge Cases

One important edge case occurs when the target is reached before time `t`, but the target still has unvisited neighbors. This is subtle because the frog is not allowed to stay voluntarily. It must continue jumping if moves exist. A naive implementation may incorrectly return the current probability immediately upon reaching the target. The solution avoids this by checking whether the target node has any valid children remaining.

Another important case is when the target is a leaf node reached before time `t`. In this scenario, the frog has no available moves and remains on the target forever. Many incorrect solutions mistakenly require the frog to reach the target at exactly time `t`. The implementation correctly preserves the probability if the target has no unvisited neighbors.

A third edge case occurs when the target is the starting node `1`. If the root has children and `t > 0`, the frog must leave immediately, making the probability zero. However, if the tree contains only one node, the frog remains at the root forever. The algorithm handles both situations naturally through the same movement rules.

A final subtle case involves paths longer than `t`. Even if the target exists in the tree, the frog may not have enough time to reach it. BFS processes states level by level in time order, so unreachable targets automatically produce probability `0.0`.
