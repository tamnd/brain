## Solution

## Problem Understanding

The problem asks us to take an input binary search tree (BST) and return a new BST that is **balanced**, meaning that for every node, the height difference between its left and right subtrees is at most one. The key requirement is that the new tree must **contain exactly the same node values** as the original tree, though the shape can differ to achieve balance. The input is a standard binary search tree node structure, and the output is also a BST node structure, potentially restructured to satisfy the balancing constraint.

The constraints tell us that the input tree can have up to 10,000 nodes and that each node's value is positive and up to 100,000. This means we cannot use algorithms with poor time complexity such as repeated rotations without careful planning. The tree may be highly skewed, like a linked list, which is the most critical edge case because it is the worst-case scenario for balancing.

Important edge cases include a tree with only a single node, a tree where all nodes are in a straight line (all left children or all right children), and trees that are already balanced. The problem guarantees that the tree has at least one node, so we do not need to handle an empty input.

## Approaches

The **brute-force approach** is to try to balance the tree by repeatedly performing rotations similar to AVL or Red-Black tree insertions. While rotations preserve the BST property, finding the correct sequence of rotations to achieve balance from an arbitrary tree is complicated and inefficient, especially for large trees. The complexity is unpredictable and may be worse than O(n log n) because each rotation might involve subtree updates, and the number of rotations required depends heavily on the initial tree shape.

The **optimal approach** relies on a key property of BSTs: an **in-order traversal of a BST produces nodes in sorted order**. By first performing an in-order traversal, we can flatten the BST into a sorted array. Then, we can recursively construct a balanced BST from the sorted array by always choosing the middle element as the root, which ensures that the left and right subtrees are as equal in size as possible. This approach guarantees a balanced BST in a predictable and efficient manner.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) - O(n^2) | O(h) | Repeated rotations may rebalance the tree but complexity depends on tree shape |
| Optimal | O(n) | O(n) | Flatten BST via in-order traversal, then build balanced BST recursively |

## Algorithm Walkthrough

1. Perform an in-order traversal of the input BST to collect all node values in a sorted list. This ensures that the resulting array preserves the BST property.
2. Implement a recursive helper function that takes a subarray of the sorted values. Find the middle element of the subarray and make it the root of the current subtree.
3. Recursively construct the left subtree using the left half of the subarray.
4. Recursively construct the right subtree using the right half of the subarray.
5. Return the root node of the newly constructed balanced BST.

**Why it works:** By always selecting the middle element as the root for any subtree, the algorithm ensures that the depth of the left and right subtrees differs by at most one. The in-order traversal guarantees that the BST property is preserved, so the resulting tree is both balanced and valid.

## Python Solution

```python
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right
class Solution:
    def balanceBST(self, root: Optional[TreeNode]) -> Optional[TreeNode]:
        def inorder(node: Optional[TreeNode]) -> list[int]:
            if not node:
                return []
            return inorder(node.left) + [node.val] + inorder(node.right)
        
        def build_balanced_bst(nums: list[int], left: int, right: int) -> Optional[TreeNode]:
            if left > right:
                return None
            mid = (left + right) // 2
            node = TreeNode(nums[mid])
            node.left = build_balanced_bst(nums, left, mid - 1)
            node.right = build_balanced_bst(nums, mid + 1, right)
            return node
        
        sorted_vals = inorder(root)
        return build_balanced_bst(sorted_vals, 0, len(sorted_vals) - 1)
```

The `inorder` function flattens the BST into a sorted list. The `build_balanced_bst` function recursively constructs a balanced BST by selecting the middle element of each subarray. The recursion ensures that each subtree is balanced, and the original values are preserved.

## Go Solution

```go
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func balanceBST(root *TreeNode) *TreeNode {
    var vals []int
    
    var inorder func(node *TreeNode)
    inorder = func(node *TreeNode) {
        if node == nil {
            return
        }
        inorder(node.Left)
        vals = append(vals, node.Val)
        inorder(node.Right)
    }
    
    var build func(left, right int) *TreeNode
    build = func(left, right int) *TreeNode {
        if left > right {
            return nil
        }
        mid := (left + right) / 2
        node := &TreeNode{Val: vals[mid]}
        node.Left = build(left, mid-1)
        node.Right = build(mid+1, right)
        return node
    }
    
    inorder(root)
    return build(0, len(vals)-1)
}
```

In Go, `nil` is used instead of `None`. Slices are used to store the sorted values. The logic mirrors the Python solution closely.

## Worked Examples

**Example 1:**

Input: `[1,null,2,null,3,null,4,null,null]`

Step 1: In-order traversal → `[1,2,3,4]`

Step 2: Build balanced BST

- Mid of `[1,2,3,4]` → `2` becomes root
- Left subarray `[1]` → left child = `1`
- Right subarray `[3,4]` → mid = `3`, left = `nil`, right = `4`

Resulting tree: `[2,1,3,null,null,null,4]`

**Example 2:**

Input: `[2,1,3]`

In-order traversal → `[1,2,3]`

Mid = `2` → root

Left = `[1]`, Right = `[3]`

Tree remains `[2,1,3]`

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | In-order traversal takes O(n), building balanced BST also takes O(n) |
| Space | O(n) | O(n) for the sorted array, O(log n) recursion stack depth for balanced tree |

The algorithm is efficient because it processes each node exactly twice: once during in-order traversal and once during tree reconstruction. The recursion stack for building the balanced tree is at most O(log n) due to perfect balancing.

## Edge Cases

A **single-node tree** could trip up implementations that assume children exist. The algorithm handles it naturally because the in-order traversal returns `[node.val]`, and building the balanced BST with a single element produces the same node.

A **completely right-skewed tree** (like a linked list) is the worst-case scenario for balancing. The in-order traversal produces a sorted list of all nodes, and choosing middle elements recursively ensures the final tree is perfectly balanced.

A **tree with duplicate values** is not a concern here because the problem guarantees `1 <= Node.val <= 10^5`, but the implementation would still work correctly since duplicates are preserved in the sorted array and inserted at appropriate positions to maintain the BST property.
