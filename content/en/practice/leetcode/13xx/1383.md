[LeetCode Problem 1383](https://leetcode.com/problems/maximum-performance-of-a-team/)

**Difficulty:** 🔴 Hard  
**Topics:** Array, Greedy, Sorting, Heap (Priority Queue)

## Solution

## Problem Understanding

The problem gives us `n` engineers, where each engineer has two attributes:

- `speed[i]`, representing how fast the engineer works
- `efficiency[i]`, representing how efficient the engineer is

We want to select at most `k` engineers to form a team.

The performance of a team is defined as:

$$\text{performance} = \left(\text{sum of speeds}\right) \times \left(\text{minimum efficiency in the team}\right)$$

The important detail is that the minimum efficiency among selected engineers controls the entire team's efficiency contribution. Even if most engineers are highly efficient, a single low-efficiency engineer lowers the multiplier for the whole team.

We must return the maximum possible performance modulo:

$$10^9 + 7$$

The constraints are large:

- `n` can be as large as `100000`
- Speeds can reach `100000`
- Efficiencies can reach `100000000`

These constraints immediately rule out exponential or quadratic solutions. We need something close to:

$$O(n \log n)$$

Another important observation is that the result before modulo can become extremely large. For example:

- total speed could be around `10^10`
- efficiency could be around `10^8`

Their product can exceed 32-bit integer limits, so we must use large integer types.

Some important edge cases include:

- `k = 1`, where we only choose a single engineer
- `k = n`, where we may use every engineer
- Engineers with extremely high speed but low efficiency
- Engineers with low speed but extremely high efficiency
- Situations where removing a slow engineer is better because it allows a higher minimum efficiency

The problem guarantees:

- Arrays are valid and equal in length
- Every engineer has positive speed and efficiency
- At least one engineer can always be selected

## Approaches

### Brute Force Approach

A brute force solution would try every possible subset of engineers whose size is at most `k`.

For each subset:

1. Compute the sum of speeds
2. Compute the minimum efficiency
3. Calculate the performance
4. Track the maximum

This works because it directly evaluates every valid team configuration.

However, the number of subsets is enormous.

For `n = 100000`, the number of combinations is astronomically large:

$$\sum_{i=1}^{k} \binom{n}{i}$$

Even for much smaller values of `n`, this becomes computationally impossible.

### Key Insight for the Optimal Solution

The crucial observation is this:

If we process engineers in descending order of efficiency, then at any moment, the current engineer's efficiency becomes the minimum efficiency of the team.

Why?

Because every engineer processed earlier has efficiency greater than or equal to the current one.

This transforms the problem into:

> For each engineer treated as the minimum efficiency, maximize the sum of speeds among up to `k` engineers.

Now the problem becomes manageable.

To maximize the speed sum efficiently, we use a min-heap:

- Keep the fastest engineers seen so far
- If the team size exceeds `k`, remove the smallest speed
- Maintain the running sum of speeds

This allows us to efficiently compute the best possible team for every candidate minimum efficiency.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | O(k) | Tries every possible subset |
| Optimal | O(n log k) | O(k) | Uses sorting and a min-heap |

## Algorithm Walkthrough

### Step 1: Pair engineers by efficiency and speed

Create pairs:

```
(efficiency, speed)
```

This keeps the engineer data together during sorting.

### Step 2: Sort engineers by decreasing efficiency

We sort in descending order of efficiency.

This is the key transformation.

When processing an engineer:

- Every previously processed engineer has efficiency greater than or equal to the current engineer
- Therefore, the current engineer's efficiency is the minimum efficiency of the current team

This lets us fix the efficiency multiplier while optimizing the speed sum.

### Step 3: Maintain a min-heap of speeds

We use a min-heap to keep track of selected engineers' speeds.

The heap stores the current team's speeds.

We also maintain:

- `speed_sum`, the sum of speeds currently in the heap
- `max_performance`, the best answer found so far

### Step 4: Add the current engineer

For each engineer:

1. Add their speed to the heap
2. Add their speed to `speed_sum`

This temporarily includes them in the team.

### Step 5: Remove the slowest engineer if size exceeds `k`

If heap size becomes larger than `k`:

- Remove the smallest speed from the heap
- Subtract it from `speed_sum`

Why remove the slowest engineer?

Because for a fixed minimum efficiency, performance depends entirely on maximizing total speed.

Removing the smallest speed preserves the largest possible sum.

### Step 6: Compute current performance

The current engineer's efficiency is the team's minimum efficiency.

So:

$$\text{performance} = \text{speed\_sum} \times \text{current\_efficiency}$$

Update the global maximum if needed.

### Step 7: Return modulo result

Finally return:

$$\text{max\_performance} \bmod (10^9 + 7)$$

### Why it works

The algorithm works because every possible team has a unique minimum efficiency.

When we process engineers in descending efficiency order, we effectively enumerate every possible minimum efficiency exactly once.

For a fixed minimum efficiency, the optimal team is simply the group of at most `k` engineers with the largest total speed among engineers whose efficiency is at least that minimum.

The min-heap guarantees we always maintain this optimal speed set efficiently.

## Python Solution

```python
from typing import List
import heapq

class Solution:
    def maxPerformance(self, n: int, speed: List[int], efficiency: List[int], k: int) -> int:
        MOD = 10**9 + 7

        engineers = sorted(
            zip(efficiency, speed),
            reverse=True
        )

        min_heap = []
        speed_sum = 0
        max_performance = 0

        for current_efficiency, current_speed in engineers:
            heapq.heappush(min_heap, current_speed)
            speed_sum += current_speed

            if len(min_heap) > k:
                removed_speed = heapq.heappop(min_heap)
                speed_sum -= removed_speed

            current_performance = speed_sum * current_efficiency
            max_performance = max(max_performance, current_performance)

        return max_performance % MOD
```

The implementation starts by pairing efficiencies and speeds together using `zip`.

The engineers are sorted in descending order of efficiency. This ensures that when we process an engineer, their efficiency becomes the minimum efficiency of the current candidate team.

The min-heap stores selected speeds. Python's `heapq` module provides an efficient min-heap implementation.

`speed_sum` tracks the total speed of the current team without recomputing sums repeatedly.

For each engineer:

1. Add their speed to the heap
2. Update the running speed total
3. If the team becomes too large, remove the slowest engineer
4. Compute the resulting performance
5. Update the global maximum

The modulo operation is performed only once at the end because intermediate values must remain exact for comparison purposes.

## Go Solution

```go
package main

import (
	"container/heap"
	"sort"
)

const MOD int = 1e9 + 7

type MinHeap []int

func (h MinHeap) Len() int {
	return len(h)
}

func (h MinHeap) Less(i, j int) bool {
	return h[i] < h[j]
}

func (h MinHeap) Swap(i, j int) {
	h[i], h[j] = h[j], h[i]
}

func (h *MinHeap) Push(x interface{}) {
	*h = append(*h, x.(int))
}

func (h *MinHeap) Pop() interface{} {
	old := *h
	n := len(old)

	value := old[n-1]
	*h = old[:n-1]

	return value
}

func maxPerformance(n int, speed []int, efficiency []int, k int) int {
	type Engineer struct {
		eff int
		spd int
	}

	engineers := make([]Engineer, n)

	for i := 0; i < n; i++ {
		engineers[i] = Engineer{
			eff: efficiency[i],
			spd: speed[i],
		}
	}

	sort.Slice(engineers, func(i, j int) bool {
		return engineers[i].eff > engineers[j].eff
	})

	minHeap := &MinHeap{}
	heap.Init(minHeap)

	var speedSum int64 = 0
	var maxPerformance int64 = 0

	for _, engineer := range engineers {
		heap.Push(minHeap, engineer.spd)
		speedSum += int64(engineer.spd)

		if minHeap.Len() > k {
			removedSpeed := heap.Pop(minHeap).(int)
			speedSum -= int64(removedSpeed)
		}

		currentPerformance := speedSum * int64(engineer.eff)

		if currentPerformance > maxPerformance {
			maxPerformance = currentPerformance
		}
	}

	return int(maxPerformance % int64(MOD))
}
```

The Go version follows the same algorithmic structure as the Python solution.

The main difference is that Go requires explicitly implementing the heap interface from `container/heap`.

Another important difference is integer overflow handling. Since the performance values can become very large, the implementation uses `int64` for:

- `speedSum`
- `maxPerformance`
- intermediate multiplication

Without `int64`, overflow could occur on many systems.

Go slices are dynamic, so the heap is naturally implemented as a slice-backed structure.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log k) | Sorting plus heap operations |
| Space | O(k) | Heap stores at most k speeds |

Sorting the engineers takes:

$$O(n \log n)$$

For each engineer, we perform at most:

- one heap insertion
- one heap removal

Each heap operation costs:

$$O(\log k)$$

So heap processing contributes:

$$O(n \log k)$$

The dominant complexity is usually written as:

$$O(n \log n)$$

because sorting dominates when `k` approaches `n`.

The auxiliary space comes from the heap, which never stores more than `k` elements.

## Edge Cases

### Edge Case 1: `k = 1`

When only one engineer can be selected, the problem reduces to maximizing:

$$\text{speed} \times \text{efficiency}$$

A buggy implementation might incorrectly try combining engineers anyway.

The heap-based solution naturally handles this because the heap size is forced back to one after every insertion.

### Edge Case 2: Very Large Numbers

Speeds and efficiencies can both be extremely large.

For example:

```
speed = 100000
efficiency = 100000000
```

Their product exceeds 32-bit integer range.

The Go solution explicitly uses `int64`, and Python automatically supports arbitrary precision integers.

The modulo operation is deferred until the end so comparisons remain accurate.

### Edge Case 3: Removing Engineers Correctly

A subtle bug occurs if we remove the wrong engineer when the team exceeds size `k`.

Suppose:

```
speeds = [1, 100, 101]
```

If we accidentally remove a large speed instead of the smallest speed, performance becomes suboptimal.

Using a min-heap guarantees that the smallest speed is always removed, preserving the largest possible speed sum for the current minimum efficiency.

### Edge Case 4: Best Team Uses Fewer Than `k` Engineers

The problem says "at most `k` engineers", not exactly `k`.

Sometimes adding another engineer lowers the minimum efficiency too much and hurts overall performance.

The algorithm correctly handles this because it evaluates performance at every step, including teams smaller than `k`.
