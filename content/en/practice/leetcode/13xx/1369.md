## Solution

## Problem Understanding

The problem asks us to extract the **second most recent activity** of each user from a `UserActivity` table. Each row in the table represents a user's activity with its start and end dates. If a user has only one activity, that single activity should be returned. The key point is that the "most recent" activity is determined by the `endDate`. Therefore, we are looking for the activity with the **second-largest endDate** for each user.

The input is a relational table with four columns: `username`, `activity`, `startDate`, and `endDate`. It may contain duplicate rows but does not have overlapping activities for the same user. The output is a table in the same schema, where each user appears exactly once with either their second most recent activity or, if they have only one activity, that activity.

Important edge cases include users with only one activity, multiple activities on consecutive days, and duplicate rows. These require careful ordering and ranking.

## Approaches

A **brute-force approach** would be to select all activities for each user, order them by `endDate` descending, and then pick the second row. In SQL, this could be done with a nested query for each user. This works because ordering by `endDate` guarantees the correct "most recent" ordering, but it is inefficient because it may involve multiple subqueries or sorting large sets of data for each user. The time complexity can be prohibitive for large datasets, roughly O(n log n) per user due to sorting.

The **optimal approach** leverages window functions available in SQL, specifically `ROW_NUMBER()` or `RANK()`. By partitioning the data by `username` and ordering each partition by `endDate` descending, we can assign a rank to each activity. We then filter for rank 2 (the second most recent activity). If no rank 2 exists for a user, we fallback to rank 1. This approach is more efficient because it avoids repeated queries per user and uses built-in SQL optimizations for window functions.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(U * A log A) | O(U * A) | For each user, sort all activities by endDate descending and pick second |
| Optimal | O(n) | O(n) | Use window function ROW_NUMBER() partitioned by username and order by endDate |

## Algorithm Walkthrough

1. Partition the `UserActivity` table by `username`. This ensures that all activities for a single user are grouped together.
2. Within each partition, order the activities by `endDate` in descending order. This makes the most recent activity first.
3. Use `ROW_NUMBER()` to assign a sequential number to each activity in the partition. The most recent activity gets 1, the second most recent gets 2, and so on.
4. Select activities where the row number is 2. These are the second most recent activities for users with multiple entries.
5. Use a union or conditional logic to handle users who only have one activity. For these users, select the activity with row number 1.
6. Combine both sets to produce a table containing exactly one activity per user: the second most recent activity if it exists, otherwise the only activity.

Why it works: `ROW_NUMBER()` guarantees unique sequential numbering within each partition, so rank 2 corresponds exactly to the second most recent activity. The partition ensures users are processed independently, and ordering by `endDate` guarantees correct chronological order.

## Python Solution

```python
import sqlite3
from typing import List, Tuple

def get_second_most_recent_activity(conn: sqlite3.Connection) -> List[Tuple]:
    query = """
    WITH ranked AS (
        SELECT
            username,
            activity,
            startDate,
            endDate,
            ROW_NUMBER() OVER (PARTITION BY username ORDER BY endDate DESC) AS rn,
            COUNT(*) OVER (PARTITION BY username) AS cnt
        FROM UserActivity
    )
    SELECT username, activity, startDate, endDate
    FROM ranked
    WHERE rn = 2 OR (rn = 1 AND cnt = 1)
    """
    cursor = conn.cursor()
    cursor.execute(query)
    return cursor.fetchall()
```

The Python implementation connects to a database, executes a SQL query using a CTE (common table expression), and uses window functions. `ROW_NUMBER()` gives each activity a position per user, and `COUNT(*)` checks if the user has only one activity. We then select rank 2 or rank 1 for users with a single activity. This directly maps to the algorithm steps.

## Go Solution

```go
package main

import (
    "database/sql"
    _ "github.com/mattn/go-sqlite3"
    "log"
)

func getSecondMostRecentActivity(db *sql.DB) ([][4]string, error) {
    query := `
    WITH ranked AS (
        SELECT
            username,
            activity,
            startDate,
            endDate,
            ROW_NUMBER() OVER (PARTITION BY username ORDER BY endDate DESC) AS rn,
            COUNT(*) OVER (PARTITION BY username) AS cnt
        FROM UserActivity
    )
    SELECT username, activity, startDate, endDate
    FROM ranked
    WHERE rn = 2 OR (rn = 1 AND cnt = 1)
    `
    rows, err := db.Query(query)
    if err != nil {
        return nil, err
    }
    defer rows.Close()

    var results [][4]string
    for rows.Next() {
        var username, activity, startDate, endDate string
        if err := rows.Scan(&username, &activity, &startDate, &endDate); err != nil {
            return nil, err
        }
        results = append(results, [4]string{username, activity, startDate, endDate})
    }
    return results, nil
}
```

Go implementation uses similar SQL logic. Differences include explicit handling of row iteration and error checking. The result is a slice of fixed-size arrays, representing the four columns.

## Edge Cases

The first edge case is a user with a single activity. A naive implementation that always selects rank 2 would return nothing for this user. Using the `COUNT(*)` ensures that the first activity is returned in this case.

The second edge case involves duplicate rows. If multiple identical activities exist, the `ROW_NUMBER()` function will still assign unique sequential numbers, ensuring the algorithm picks the second occurrence correctly or the only unique activity.

The third edge case is users with consecutive activities ending on the same day. Since `ROW_NUMBER()` breaks ties arbitrarily, the implementation still guarantees one activity is returned, but the selection between tied end dates could be non-deterministic. For deterministic behavior, `ORDER BY endDate DESC, startDate ASC` can be added.
