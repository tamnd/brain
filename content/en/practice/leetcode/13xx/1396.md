## Solution

## Problem Understanding

This problem asks us to design a small tracking system for an underground railway network. The system must record when passengers enter and leave stations, then compute the average travel time between any pair of stations.

Each passenger is identified by a unique integer `id`. When a customer checks in, we are given:

- Their customer ID
- The station name where they entered
- The check-in time

Later, when they check out, we receive:

- The same customer ID
- The destination station
- The checkout time

The system must calculate the average duration for all direct trips between two stations.

The phrase "directly" is important. A trip from `"Leyton"` to `"Waterloo"` is treated separately from a trip from `"Waterloo"` to `"Leyton"`. These are considered two different routes.

The constraints tell us several important things:

- There are at most `2 * 10^4` operations total, which is relatively small.
- All events occur in chronological order.
- Every checkout is guaranteed to correspond to a valid prior check-in.
- A customer can only be checked into one station at a time.
- `getAverageTime` will only be called for routes that already exist.

These guarantees simplify the implementation because we never need to handle invalid states.

The core challenge is designing a data structure that supports:

1. Fast lookup of a customer's active trip
2. Fast aggregation of route statistics
3. Efficient average computation

A naive implementation that stores every trip individually would work functionally, but it is not the most efficient design.

Some important edge cases include:

- Multiple customers traveling the same route
- Multiple trips by the same customer over time
- Routes in opposite directions being treated independently
- Calling `getAverageTime` before new trips are added
- Large numbers of repeated queries on the same route

The problem guarantees consistency, so we do not need to validate malformed operations.

## Approaches

### Brute Force Approach

A brute force solution would store every completed trip individually.

When a customer checks in, we store their starting station and check-in time.

When they check out, we append the trip duration into a list associated with the route `(startStation, endStation)`.

Then, whenever `getAverageTime` is called, we iterate through the entire list of trip durations for that route and compute the average.

This approach is correct because all trip durations are preserved exactly, so averaging them produces the right result.

However, this becomes inefficient when many queries are made for the same route. Every call to `getAverageTime` would require scanning all previous trips again.

For example, if there are `n` trips between two stations and we repeatedly query the average, each query costs `O(n)` time.

### Optimal Approach

The key insight is that we do not actually need to store every trip duration individually.

To compute an average, we only need:

- The total accumulated travel time
- The number of trips

Using these two values, the average becomes:

$$\text{average} = \frac{\text{totalTime}}{\text{tripCount}}$$

So instead of storing all trip durations, we maintain running aggregates for each route.

We use two hash maps:

1. An `activeTrips` map that tracks customers currently checked in
2. A `routeStats` map that stores:

- Total travel time
- Number of completed trips

This allows every operation to run in constant time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | `O(n)` per average query | `O(n)` | Stores every trip duration |
| Optimal | `O(1)` per operation | `O(n)` | Stores only aggregated statistics |

## Algorithm Walkthrough

1. Initialize two hash maps.

The first map, `activeTrips`, stores currently checked-in customers. The key is the customer ID, and the value contains:

- Starting station
- Check-in time

The second map, `routeStats`, stores aggregated information for each route. The key is `(startStation, endStation)` and the value stores:

- Total accumulated travel time
- Number of trips
2. During `checkIn(id, stationName, t)`:

Insert the customer's check-in information into `activeTrips`.

This allows us to later retrieve the starting station and starting time when the customer checks out.
3. During `checkOut(id, stationName, t)`:

Retrieve the customer's check-in information from `activeTrips`.

Compute the travel duration:

$$\text{duration} = t - \text{startTime}$$

Build the route key:

$$(\text{startStation}, \text{endStation})$$

Update the corresponding route statistics:

- Add `duration` to total travel time
- Increment trip count

Finally, remove the customer from `activeTrips` because their trip has completed.
4. During `getAverageTime(startStation, endStation)`:

Retrieve:

- Total travel time
- Trip count

Return:

$$\frac{\text{totalTime}}{\text{tripCount}}$$

### Why it works

The algorithm maintains an invariant that for every route:

- `totalTime` always equals the sum of all completed trip durations
- `tripCount` always equals the number of completed trips

Since the average is mathematically defined as:

$$\frac{\text{sum of values}}{\text{number of values}}$$

the returned result is always correct.

Because every trip contributes exactly once during checkout, and no trip is lost or double-counted, the statistics remain accurate throughout execution.

## Python Solution

```python
class UndergroundSystem:

    def __init__(self):
        self.active_trips = {}
        self.route_stats = {}

    def checkIn(self, id: int, stationName: str, t: int) -> None:
        self.active_trips[id] = (stationName, t)

    def checkOut(self, id: int, stationName: str, t: int) -> None:
        start_station, start_time = self.active_trips[id]

        route = (start_station, stationName)
        duration = t - start_time

        if route not in self.route_stats:
            self.route_stats[route] = [0, 0]

        self.route_stats[route][0] += duration
        self.route_stats[route][1] += 1

        del self.active_trips[id]

    def getAverageTime(self, startStation: str, endStation: str) -> float:
        total_time, trip_count = self.route_stats[(startStation, endStation)]
        return total_time / trip_count

# Your UndergroundSystem object will be instantiated and called as such:
# obj = UndergroundSystem()
# obj.checkIn(id, stationName, t)
# obj.checkOut(id, stationName, t)
# param_3 = obj.getAverageTime(startStation, endStation)
```

The implementation closely follows the algorithm described earlier.

The `active_trips` dictionary tracks customers currently traveling. Each customer ID maps to a tuple containing the station name and check-in time.

The `route_stats` dictionary aggregates statistics for every route. The key is a tuple `(startStation, endStation)`. The value is a two-element list:

- Index `0` stores total travel time
- Index `1` stores trip count

During checkout, we retrieve the customer's check-in data, compute the trip duration, and update the aggregate statistics.

The average calculation is efficient because it only requires a single division using precomputed totals.

Removing the customer from `active_trips` after checkout ensures that the dictionary only contains active travelers.

## Go Solution

```go
type CheckInData struct {
	station string
	time    int
}

type RouteStats struct {
	totalTime int
	tripCount int
}

type UndergroundSystem struct {
	activeTrips map[int]CheckInData
	routeStats  map[string]RouteStats
}

func Constructor() UndergroundSystem {
	return UndergroundSystem{
		activeTrips: make(map[int]CheckInData),
		routeStats:  make(map[string]RouteStats),
	}
}

func (this *UndergroundSystem) CheckIn(id int, stationName string, t int) {
	this.activeTrips[id] = CheckInData{
		station: stationName,
		time:    t,
	}
}

func (this *UndergroundSystem) CheckOut(id int, stationName string, t int) {
	checkInData := this.activeTrips[id]

	routeKey := checkInData.station + "->" + stationName
	duration := t - checkInData.time

	stats := this.routeStats[routeKey]
	stats.totalTime += duration
	stats.tripCount++

	this.routeStats[routeKey] = stats

	delete(this.activeTrips, id)
}

func (this *UndergroundSystem) GetAverageTime(startStation string, endStation string) float64 {
	routeKey := startStation + "->" + endStation

	stats := this.routeStats[routeKey]

	return float64(stats.totalTime) / float64(stats.tripCount)
}

/**
 * Your UndergroundSystem object will be instantiated and called as such:
 * obj := Constructor();
 * obj.CheckIn(id, stationName, t);
 * obj.CheckOut(id, stationName, t);
 * param_3 := obj.GetAverageTime(startStation, endStation);
 */
```

The Go solution follows the same logic as the Python implementation but uses structs for clarity and type safety.

Go does not support tuple keys as conveniently as Python, so the route is encoded as a string like `"Leyton->Waterloo"`.

Maps in Go return zero-value structs automatically when a key does not exist, which simplifies route statistic updates.

The average calculation requires explicit conversion to `float64` because integer division would otherwise truncate the result.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | `O(1)` | Every operation performs constant-time hash map operations |
| Space | `O(n)` | Stores active trips and route statistics |

The time complexity is constant because hash map insertion, lookup, update, and deletion all operate in average `O(1)` time.

The space complexity depends on:

- The number of active passengers
- The number of unique station pairs

In the worst case, both grow linearly with the number of operations.

## Edge Cases

One important edge case is multiple trips on the same route. A buggy implementation might overwrite previous data instead of accumulating it. Our solution avoids this by storing both total travel time and trip count, then incrementally updating both values during every checkout.

Another important edge case is route directionality. The trip from `"A"` to `"B"` must not be merged with the trip from `"B"` to `"A"`. Since the route key explicitly stores both start and end stations in order, these are treated as completely separate entries in the hash map.

A third important edge case is customers making multiple trips over time. If we fail to remove completed trips from `activeTrips`, stale data could corrupt future calculations. The implementation correctly deletes the customer's active trip after checkout, ensuring each new trip starts cleanly.

A fourth edge case involves simultaneous active passengers. Multiple customers may be checked in at once, even at the same station. Since `activeTrips` uses customer ID as the key, every passenger's journey remains isolated and can be updated independently without collisions.
