## Solution

## Problem Understanding

The problem gives us a binary matrix where each row represents a group of people. A value of `1` represents a soldier, and a value of `0` represents a civilian. The important guarantee is that in every row, all soldiers appear before civilians. In other words, every row is sorted in non-increasing order, meaning the row looks like this:

```
[1,1,1,1,0,0,0]
```

and never like this:

```
[1,0,1,0]
```

We need to determine the `k` weakest rows in the matrix. A row is considered weaker if it contains fewer soldiers. If two rows contain the same number of soldiers, then the row with the smaller index is considered weaker.

The output should be a list of row indices ordered from weakest to strongest among the selected `k` rows.

For example, if the rows contain soldier counts:

```
Row 0 -> 2 soldiers
Row 1 -> 4 soldiers
Row 2 -> 1 soldier
Row 3 -> 2 soldiers
```

then the order from weakest to strongest is:

```
[2,0,3,1]
```

because row `2` has the fewest soldiers, and rows `0` and `3` both have two soldiers, but `0` comes first because it has the smaller index.

The constraints are relatively small:

- `m <= 100`
- `n <= 100`

This means even an `O(m * n)` solution is completely acceptable. However, the sorted nature of each row gives us an opportunity to improve soldier counting with binary search.

Several edge cases are important to recognize:

- Multiple rows may contain the same number of soldiers.
- A row may contain all soldiers.
- A row may contain all civilians.
- `k` may equal `m`, meaning we return every row.
- The matrix is guaranteed to contain only `0` and `1`.
- Every row is guaranteed to have soldiers before civilians, which makes binary search possible.

## Approaches

### Brute Force Approach

The most direct solution is to count the number of soldiers in every row by scanning each element from left to right. Since each row contains only `0`s and `1`s, we can simply sum the row or count consecutive `1`s until we encounter the first `0`.

After computing the soldier count for every row, we store pairs of:

```
(soldier_count, row_index)
```

We then sort these pairs. Python automatically sorts tuples lexicographically, meaning it first compares soldier counts and then row indices. This perfectly matches the problem definition.

Finally, we extract the first `k` row indices from the sorted result.

This approach is correct because it explicitly computes the ranking criteria defined in the problem statement. However, counting soldiers by scanning entire rows costs `O(n)` per row.

### Optimal Approach

The key observation is that every row is sorted:

```
1 1 1 1 0 0 0
```

This means the number of soldiers is exactly the position of the first `0`.

Instead of scanning every element, we can use binary search to locate the first `0` in each row. This reduces soldier counting from `O(n)` to `O(log n)`.

After determining the soldier count for each row, we again sort by:

```
(soldier_count, row_index)
```

and return the first `k` indices.

Even though the constraints are small, this approach demonstrates how sorted data can be leveraged to improve efficiency.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m × n + m log m) | O(m) | Scans every cell in every row |
| Optimal | O(m log n + m log m) | O(m) | Uses binary search to count soldiers efficiently |

## Algorithm Walkthrough

### Optimal Algorithm

1. Create an empty list called `strengths`.

This list will store information about each row in the form:

```
(soldier_count, row_index)
```

We need both values because rows are sorted primarily by soldier count and secondarily by index.
2. Iterate through every row in the matrix.

For each row, determine how many soldiers it contains.
3. Use binary search to count soldiers.

Since all `1`s appear before `0`s, we search for the first occurrence of `0`.

We maintain two pointers:

- `left`
- `right`

At each step:

- Compute the middle index.
- If `row[mid] == 1`, soldiers continue to the right.
- Otherwise, the first `0` is at `mid` or earlier.

When the search finishes, `left` equals the number of soldiers.
4. Store the result.

Append:

```
(soldier_count, row_index)
```

into the `strengths` list.
5. Sort the list.

Python tuple sorting automatically handles both rules:

- smaller soldier count first
- smaller row index first if counts are equal
6. Extract the first `k` row indices.

These correspond to the weakest rows.
7. Return the result.

### Why it works

The algorithm works because each row is independently evaluated according to the exact ranking criteria defined in the problem. Binary search correctly identifies the boundary between soldiers and civilians because rows are sorted. Sorting the `(soldier_count, row_index)` pairs guarantees that rows are ordered first by strength and then by index for tie-breaking.

## Python Solution

```python
from typing import List

class Solution:
    def kWeakestRows(self, mat: List[List[int]], k: int) -> List[int]:
        def count_soldiers(row: List[int]) -> int:
            left = 0
            right = len(row)

            while left < right:
                mid = (left + right) // 2

                if row[mid] == 1:
                    left = mid + 1
                else:
                    right = mid

            return left

        strengths = []

        for index, row in enumerate(mat):
            soldiers = count_soldiers(row)
            strengths.append((soldiers, index))

        strengths.sort()

        return [index for _, index in strengths[:k]]
```

The implementation begins with a helper function called `count_soldiers`. This function performs binary search on a single row to determine how many soldiers it contains.

The search space starts as the entire row:

```python
left = 0
right = len(row)
```

If the middle element is `1`, then soldiers continue further right, so we move `left` forward. Otherwise, we shrink the search space to the left half.

When the loop terminates, `left` points to the first `0`, which is also equal to the number of soldiers.

Next, we iterate through all rows using `enumerate` so we have both the row contents and the row index. For each row, we compute the soldier count and store:

```python
(soldiers, index)
```

inside the `strengths` list.

Sorting this list automatically orders rows correctly because tuples are compared lexicographically.

Finally, we return the indices of the first `k` rows.

## Go Solution

```go
package main

import "sort"

func kWeakestRows(mat [][]int, k int) []int {
	type RowInfo struct {
		soldiers int
		index    int
	}

	countSoldiers := func(row []int) int {
		left := 0
		right := len(row)

		for left < right {
			mid := (left + right) / 2

			if row[mid] == 1 {
				left = mid + 1
			} else {
				right = mid
			}
		}

		return left
	}

	strengths := make([]RowInfo, 0)

	for index, row := range mat {
		soldiers := countSoldiers(row)

		strengths = append(strengths, RowInfo{
			soldiers: soldiers,
			index:    index,
		})
	}

	sort.Slice(strengths, func(i, j int) bool {
		if strengths[i].soldiers == strengths[j].soldiers {
			return strengths[i].index < strengths[j].index
		}

		return strengths[i].soldiers < strengths[j].soldiers
	})

	result := make([]int, 0)

	for i := 0; i < k; i++ {
		result = append(result, strengths[i].index)
	}

	return result
}
```

The Go solution follows the same algorithmic structure as the Python version. Since Go does not have built-in tuple sorting, we define a custom `RowInfo` struct containing the soldier count and row index.

Sorting is handled using `sort.Slice` with a custom comparator function. The comparator explicitly applies the problem rules:

- fewer soldiers first
- smaller index first when counts are equal

Go slices are dynamically sized, so we use `append` to build both the `strengths` and `result` slices.

Integer overflow is not a concern because matrix dimensions are very small.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m log n + m log m) | Binary search for each row plus sorting |
| Space | O(m) | Stores `(soldier_count, row_index)` pairs |

The binary search runs once for every row, costing `O(log n)` each time. Across `m` rows, this becomes `O(m log n)`.

After counting soldiers, we sort `m` entries, which costs `O(m log m)`.

The additional space comes from storing the row metadata in the `strengths` list.

## Edge Cases

One important edge case occurs when multiple rows contain the same number of soldiers. A naive implementation might accidentally lose the original ordering or incorrectly prioritize later rows. The problem explicitly states that ties are broken by row index. Our implementation handles this naturally by sorting tuples of:

```
(soldier_count, row_index)
```

which guarantees smaller indices come first when counts are equal.

Another important case is rows containing all soldiers:

```
[1,1,1,1]
```

In binary search implementations, this can cause off-by-one errors because there is no `0` in the row. Our implementation correctly returns the row length because the search continues until `left == len(row)`.

Rows containing all civilians are also important:

```
[0,0,0,0]
```

A buggy binary search could incorrectly return `1` or produce an index error. In our solution, the search immediately converges to position `0`, correctly indicating zero soldiers.

Finally, the case where `k == m` is important because the algorithm must return every row in sorted order. Our implementation simply slices the sorted list using:

```
strengths[:k]
```

which works correctly whether `k` is small or equal to the total number of rows.
