## Solution

## Problem Understanding

The problem gives us a list of votes, where each vote is a string representing a complete ranking of teams from highest rank to lowest rank. Every character in a vote represents a unique team, and the order of characters determines that voter's preference.

Our task is to produce the final ranking of all teams according to a custom comparison rule:

1. Teams with more first-place votes rank higher.
2. If two teams tie in first-place votes, compare second-place votes.
3. Continue comparing position by position until the tie is broken.
4. If all positional vote counts are identical, rank alphabetically.

The output should be a single string containing all teams sorted from best to worst according to these rules.

For example, if team A receives more first-place votes than team B, then A must come before B regardless of lower rankings. If A and B tie in first-place votes, then we compare how many second-place votes they received, and so on.

The constraints are important:

- There can be at most 1000 votes.
- Each vote contains at most 26 teams.
- Every vote ranks all teams exactly once.
- All votes contain the same set of teams.

Since there are only at most 26 teams, the number of candidates is very small. This is a strong hint that we can afford to store detailed statistics for every team across every ranking position.

Several edge cases are important:

- Only one vote exists. In that case, the answer is simply that vote itself.
- Multiple teams may tie at every ranking position, requiring alphabetical ordering.
- All teams may receive extremely balanced rankings.
- The number of teams may be much smaller than 26.

A naive implementation can easily make mistakes in tie-breaking order or in handling alphabetical fallback correctly.

## Approaches

### Brute Force Approach

A brute force approach would compare every pair of teams directly. For each comparison, we would repeatedly scan all votes to count how many times each team appeared in every ranking position.

For example, when comparing team A and team B:

- Count how many first-place votes A and B have.
- If tied, count second-place votes.
- Continue until the tie is resolved.

This works because it follows the exact ranking rules described in the problem.

However, this approach is inefficient because each comparison repeatedly scans all votes. Sorting requires many comparisons, and each comparison may inspect every vote and every ranking position again.

Although the constraints are relatively small, repeatedly recomputing counts is unnecessary and inefficient.

### Optimal Approach

The key observation is that ranking information can be precomputed once.

For every team, we store:

- How many first-place votes it received
- How many second-place votes it received
- And so on for all positions

Once we build these frequency vectors, comparing two teams becomes straightforward:

- Compare their first-position counts
- Then second-position counts
- Continue lexicographically
- Finally use alphabetical order

This transforms the problem into a sorting problem with a custom comparator.

Because the number of teams is at most 26, storing an array of vote counts for each team is very efficient.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(V × T² × log T) | O(1) | Recomputes vote counts during comparisons |
| Optimal | O(V × T + T log T) | O(T²) | Precomputes positional counts once |

Where:

- `V` = number of votes
- `T` = number of teams

## Algorithm Walkthrough

1. Determine the number of teams from the first vote.

Every vote contains the same teams in the same quantity, so `len(votes[0])` gives the total number of teams.
2. Create a frequency table for every team.

For each team, we store an array of size `T`.

Example:

```
A -> [5, 0, 0]
B -> [0, 2, 3]
C -> [0, 3, 2]
```

Here:

- `counts[team][0]` represents first-place votes
- `counts[team][1]` represents second-place votes
- etc.
3. Process every vote.

For each vote string:

- Traverse each character and its position.
- Increment the corresponding position count for that team.

Example:

```
Vote = "ACB"

A gets one first-place vote
C gets one second-place vote
B gets one third-place vote
```
4. Extract the list of teams.

Since all teams appear in the first vote, we can simply use:

```
teams = list(votes[0])
```
5. Sort teams using a custom sorting rule.

Teams should rank higher if they have:

- More first-place votes
- Then more second-place votes
- And so on

Since Python sorts ascending by default, we sort using negative counts.

The sort key becomes:

```
(-count1, -count2, ..., team_letter)
```

The team letter at the end ensures alphabetical ordering when all vote counts tie.
6. Join the sorted teams into a string.

After sorting, concatenate all characters together to produce the final ranking.

### Why it works

Each team is represented by a vector containing its vote counts at every ranking position. The ranking rules in the problem exactly correspond to lexicographical comparison of these vectors from left to right.

Sorting teams by descending positional counts therefore produces the correct ordering. Alphabetical ordering is applied only when all counts are identical, matching the problem statement precisely.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Solution:
    def rankTeams(self, votes: List[str]) -> str:
        if not votes:
            return ""
        
        num_teams = len(votes[0])
        
        counts = defaultdict(lambda: [0] * num_teams)
        
        for vote in votes:
            for position, team in enumerate(vote):
                counts[team][position] += 1
        
        teams = list(votes[0])
        
        teams.sort(
            key=lambda team: (
                [-counts[team][i] for i in range(num_teams)],
                team
            )
        )
        
        return "".join(teams)
```

The implementation begins by handling the edge case of an empty input, although the constraints guarantee at least one vote.

We then determine the number of teams from the first vote. This value is used to size the frequency arrays correctly.

The `counts` dictionary maps each team to an array of positional vote frequencies. A `defaultdict` is convenient because it automatically initializes arrays when a team is encountered for the first time.

Next, we process every vote. For each position in the ranking, we increment that team's count for the corresponding rank.

After collecting all frequencies, we extract the list of teams from the first vote. Since every vote contains the same teams, this is sufficient.

The sorting step is the core of the solution. The key function produces:

- Negative vote counts for descending priority
- The team letter itself for alphabetical tie-breaking

Finally, the sorted teams are joined into a single string and returned.

## Go Solution

```go
package main

import (
	"sort"
)

func rankTeams(votes []string) string {
	if len(votes) == 0 {
		return ""
	}

	numTeams := len(votes[0])

	counts := make(map[byte][]int)

	for _, vote := range votes {
		for position := 0; position < numTeams; position++ {
			team := vote[position]

			if _, exists := counts[team]; !exists {
				counts[team] = make([]int, numTeams)
			}

			counts[team][position]++
		}
	}

	teams := []byte(votes[0])

	sort.Slice(teams, func(i, j int) bool {
		teamA := teams[i]
		teamB := teams[j]

		for position := 0; position < numTeams; position++ {
			if counts[teamA][position] != counts[teamB][position] {
				return counts[teamA][position] > counts[teamB][position]
			}
		}

		return teamA < teamB
	})

	return string(teams)
}
```

The Go implementation follows the same overall algorithm as the Python version.

One important difference is that Go does not support custom tuple comparison like Python. Instead, we implement the comparison logic manually inside `sort.Slice`.

The counts are stored in a `map[byte][]int`, since team names are uppercase English letters and can be represented as bytes efficiently.

The comparator explicitly checks each ranking position one by one and returns the appropriate ordering immediately when a difference is found. If all counts match, alphabetical ordering is applied.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(V × T + T log T) | Counting all votes takes O(V × T), sorting teams takes O(T log T) |
| Space | O(T²) | Each team stores an array of size T |

The counting phase visits every character in every vote exactly once. Since there are `V` votes and `T` teams per vote, this requires `O(V × T)` time.

The sorting phase operates on at most 26 teams, making it very small in practice. The comparator examines up to `T` positions during comparisons, but since `T ≤ 26`, this remains efficient.

The space complexity comes from storing a positional frequency array for every team.

## Edge Cases

### Single Vote

If only one vote exists, the final ranking must exactly match that vote. A buggy implementation might still attempt unnecessary comparisons or sorting logic that accidentally changes the order.

This implementation handles the case naturally because the frequency arrays directly mirror the single ranking, so sorting preserves the original order.

### Complete Tie Between Teams

It is possible for two or more teams to receive identical counts at every ranking position. In that case, the problem requires alphabetical ordering.

For example:

```
["ABC", "BAC", "CBA"]
```

All teams become tied in positional counts. The implementation explicitly includes the team letter as the final sorting key, guaranteeing correct alphabetical resolution.

### Only One Team

The smallest valid input contains only one team:

```
["A"]
```

Some implementations may incorrectly assume multiple teams exist or create invalid comparisons.

This solution correctly creates a single frequency array, sorts one element trivially, and returns `"A"`.
