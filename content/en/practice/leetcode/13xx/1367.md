## Solution

## Problem Understanding

This problem asks us to determine whether a linked list appears as a downward path inside a binary tree.

We are given two data structures:

1. A singly linked list whose first node is `head`
2. A binary tree whose root is `root`

We must check whether there exists some path in the binary tree such that:

- The path starts at any tree node
- The path moves only downward, meaning from parent to child
- The sequence of node values along that path exactly matches the linked list

The path does not need to start at the root of the tree. It can begin anywhere. Similarly, it does not need to end at a leaf node. The only requirement is that the linked list values appear consecutively along a downward path.

For example, if the linked list is:

```
4 -> 2 -> 8
```

then we need to find some tree node with value `4`, whose child path contains `2`, whose child path then contains `8`.

The constraints are important:

- The binary tree contains at most 2500 nodes
- The linked list contains at most 100 nodes

These limits are small enough that a recursive depth first search solution is feasible. Even an approach that tries matching from many tree nodes can still work within the limits.

Several edge cases are important:

- The linked list may contain only one node
- Multiple tree nodes may share the same value
- A partial match may fail midway and another match may start deeper in the tree
- The matching path may go entirely through left children, entirely through right children, or alternate between them
- The root itself may not be part of the matching path
- The tree may contain many repeated values, which can confuse naive matching logic

The problem guarantees that both the tree and linked list contain at least one node.

## Approaches

### Brute Force Approach

The brute force idea is straightforward.

For every node in the binary tree, we attempt to start matching the linked list from that node.

We define a helper function that checks:

- If the current linked list node matches the current tree node
- If so, recursively continue matching with either the left child or right child

If the linked list is completely consumed, we found a valid path.

If the tree node becomes `None` before the linked list ends, the match fails.

This works because every possible starting position in the tree is explored, and from each position every possible downward path is checked.

The downside is repeated work. Many tree paths may be explored multiple times, especially when the tree contains repeated values.

Still, because the constraints are moderate, this approach is acceptable.

### Key Insight for the Optimal Approach

The important observation is that this problem naturally splits into two recursive tasks:

1. Search the tree for possible starting positions
2. Once a start is found, verify whether the linked list matches downward

Instead of generating all root to leaf paths explicitly, we recursively compare nodes as we traverse the tree.

The optimal recursive DFS solution works efficiently because:

- Each tree node is considered as a potential starting point
- Matching stops immediately when values differ
- The linked list length is small, at most 100

This pruning keeps the runtime manageable.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(N × M) | O(H + M) | Try matching the linked list from every tree node |
| Optimal | O(N × M) | O(H + M) | DFS with recursive matching and early pruning |

Here:

- `N` is the number of tree nodes
- `M` is the number of linked list nodes
- `H` is the height of the tree

## Algorithm Walkthrough

### Step 1: Create a matching helper function

We define a recursive helper:

```
match(list_node, tree_node)
```

This function determines whether the linked list beginning at `list_node` matches a downward path beginning at `tree_node`.

### Step 2: Handle successful completion

If `list_node` becomes `None`, it means every linked list node matched successfully.

At this point we return `True`.

### Step 3: Handle failed matches

If `tree_node` becomes `None` before the linked list ends, there are no more tree nodes available to continue matching.

Return `False`.

Also, if:

```
list_node.val != tree_node.val
```

then the current path cannot match the linked list.

Return `False`.

### Step 4: Continue matching downward

If the current values match, we recursively continue in either direction:

```
match(list_node.next, tree_node.left)
or
match(list_node.next, tree_node.right)
```

Only one successful path is needed.

### Step 5: Search every tree node as a starting point

The main DFS function traverses the entire tree.

At each tree node:

1. Try matching the linked list starting there
2. If matching fails, recursively search the left subtree
3. Then recursively search the right subtree

### Step 6: Return whether any valid path exists

If any recursive search succeeds, return `True`.

Otherwise return `False`.

### Why it works

The algorithm is correct because it exhaustively checks every valid starting position in the tree and every valid downward continuation path.

The `match` function guarantees that:

- Values must match in order
- Paths only move downward
- The entire linked list must be consumed for success

Since every possible downward path is explored recursively, no valid solution can be missed.

## Python Solution

```python
# Definition for singly-linked list.
# class ListNode:
#     def __init__(self, val=0, next=None):
#         self.val = val
#         self.next = next

# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import Optional

class Solution:
    def isSubPath(
        self,
        head: Optional[ListNode],
        root: Optional[TreeNode]
    ) -> bool:

        def match(
            list_node: Optional[ListNode],
            tree_node: Optional[TreeNode]
        ) -> bool:

            if list_node is None:
                return True

            if tree_node is None:
                return False

            if list_node.val != tree_node.val:
                return False

            return (
                match(list_node.next, tree_node.left)
                or
                match(list_node.next, tree_node.right)
            )

        if root is None:
            return False

        return (
            match(head, root)
            or
            self.isSubPath(head, root.left)
            or
            self.isSubPath(head, root.right)
        )
```

The implementation contains two layers of recursion.

The `match` helper is responsible for validating whether a linked list sequence matches a downward tree path. It handles all matching logic and terminates immediately when a mismatch occurs.

The main `isSubPath` function performs the tree traversal. At every node, it attempts to start a match. If matching fails, it recursively searches the left and right subtrees for another starting point.

This separation of responsibilities keeps the solution clean and easy to reason about.

The recursive structure also naturally mirrors the structure of the binary tree and linked list.

## Go Solution

```go
/**
 * Definition for singly-linked list.
 * type ListNode struct {
 *     Val int
 *     Next *ListNode
 * }
 */

/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */

func isSubPath(head *ListNode, root *TreeNode) bool {

    var match func(*ListNode, *TreeNode) bool

    match = func(listNode *ListNode, treeNode *TreeNode) bool {

        if listNode == nil {
            return true
        }

        if treeNode == nil {
            return false
        }

        if listNode.Val != treeNode.Val {
            return false
        }

        return match(listNode.Next, treeNode.Left) ||
            match(listNode.Next, treeNode.Right)
    }

    if root == nil {
        return false
    }

    return match(head, root) ||
        isSubPath(head, root.Left) ||
        isSubPath(head, root.Right)
}
```

The Go implementation follows the exact same recursive structure as the Python solution.

The primary difference is explicit handling of `nil` pointers instead of Python's `None`.

Go also requires declaring the recursive closure variable before assigning the function body:

```go
var match func(*ListNode, *TreeNode) bool
```

This allows the function to recursively reference itself.

No integer overflow concerns exist because node values are very small.

## Worked Examples

### Example 1

```
head = [4,2,8]
```

Tree:

```
        1
       / \
      4   4
       \   \
        2   2
           / \
          6   8
```

We traverse the tree searching for value `4`.

| Current Tree Node | Action |
| --- | --- |
| 1 | Does not match head value 4 |
| Left 4 | Start matching |
| 2 | Matches second list node |
| 6 | Does not match 8 |
| Backtrack | Match failed |
| Right 4 | Start matching |
| 2 | Matches second list node |
| 8 | Matches third list node |
| Linked list exhausted | Return True |

Result:

```
True
```

### Example 2

```
head = [1,4,2,6]
```

Traversal begins at root `1`.

| Linked List Node | Tree Node | Result |
| --- | --- | --- |
| 1 | 1 | Match |
| 4 | Right child 4 | Match |
| 2 | Child 2 | Match |
| 6 | Left child 6 | Match |
| End of list | Success | True |

Result:

```
True
```

### Example 3

```
head = [1,4,2,6,8]
```

The algorithm finds partial matches:

```
1 -> 4 -> 2 -> 6
```

But after `6`, there is no child node with value `8`.

Every possible continuation fails.

Result:

```

```

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(N × M) | Each tree node may attempt to match up to the entire linked list |
| Space | O(H + M) | Recursive call stack for tree traversal and matching |

Here:

- `N` is the number of tree nodes
- `M` is the number of linked list nodes
- `H` is the tree height

In the worst case, every tree node becomes a potential starting point, and each attempt may compare up to all linked list nodes.

The recursive stack depth comes from two sources:

1. Tree traversal depth, up to `H`
2. Linked list matching depth, up to `M`

Since `M ≤ 100`, recursion depth remains manageable.

## Edge Cases

### Single Node Linked List

A linked list containing only one node is easy to mishandle if the implementation assumes recursive continuation always exists.

For example:

```
head = [5]
```

The algorithm correctly handles this because matching succeeds immediately after comparing the current node values. Once `list_node.next` becomes `None`, the helper returns `True`.

### Multiple Matching Starting Points

Trees with repeated values can create many possible starting positions.

For example:

```
1
/ \
1   1
```

with:

```
[1,1]
```

A buggy implementation might stop after the first failed continuation and miss another valid path.

This solution avoids that issue because it recursively searches every tree node independently.

### Partial Match Followed by Failure

A common bug occurs when a long prefix matches but the final node fails.

Example:

```
List: 1 -> 2 -> 3 -> 4
Tree path: 1 -> 2 -> 3 -> 5
```

The algorithm correctly detects failure when `4 != 5` and backtracks to continue searching elsewhere in the tree.

This guarantees that only complete matches succeed.
