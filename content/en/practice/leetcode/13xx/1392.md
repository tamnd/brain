## Solution

## Problem Understanding

The problem asks us to find the longest substring that satisfies two conditions simultaneously:

- It must be a prefix of the string.
- It must also be a suffix of the string.
- It cannot be equal to the entire string itself.

A prefix starts at index `0` and extends to the right. A suffix ends at index `n - 1` and extends to the left.

For example, in the string `"ababab"`:

- Prefixes are: `"a"`, `"ab"`, `"aba"`, `"abab"`, `"ababa"`
- Suffixes are: `"b"`, `"ab"`, `"bab"`, `"abab"`, `"babab"`

The longest substring appearing in both lists is `"abab"`.

The key detail is that the prefix and suffix are allowed to overlap. This often surprises people at first. For example, in `"aaaaa"`, the answer is `"aaaa"`, even though the prefix and suffix overlap heavily in the original string.

The input size is important:

- `1 <= s.length <= 10^5`

A string of length one hundred thousand immediately tells us that quadratic solutions will likely time out. Any approach that compares many substrings directly using slicing or repeated scans will become too slow.

The string contains only lowercase English letters, which simplifies character handling, but the real challenge is designing an efficient string matching algorithm.

Several edge cases are especially important:

- Strings with no matching prefix and suffix, such as `"abcde"`
- Strings where the entire structure repeats, such as `"ababab"`
- Strings with repeated single characters, such as `"aaaaa"`
- Very small strings like `"a"`
- Cases where multiple candidate prefixes exist and we must choose the longest

A naive implementation can easily become inefficient because substring comparisons are expensive when repeated many times.

## Approaches

### Brute Force Approach

The brute force solution checks every possible prefix length from largest to smallest.

For a string of length `n`, we try:

- Prefix of length `n - 1`
- Prefix of length `n - 2`
- ...
- Prefix of length `1`

For each length `k`, we compare:

- `s[:k]`
- `s[n-k:]`

If they are equal, we immediately return that prefix because we are checking lengths in descending order.

This approach is correct because it explicitly tests every possible valid answer. The first successful match must be the longest happy prefix.

However, it is inefficient. Each substring comparison can take `O(k)` time, and we may perform up to `O(n)` such comparisons. In the worst case, this leads to `O(n^2)` time complexity.

With `n = 10^5`, quadratic performance is far too slow.

### Optimal Approach, KMP Prefix Function

The key insight is that this problem is fundamentally about matching prefixes against suffixes efficiently.

Instead of repeatedly comparing substrings from scratch, we can reuse previously computed matching information.

This is exactly what the KMP prefix function, also called the LPS array (Longest Prefix Suffix), was designed for.

For every position `i`, the prefix function stores:

- The length of the longest proper prefix of `s[:i+1]`
- That is also a suffix of `s[:i+1]`

By the time we finish processing the entire string, the last value in the prefix array tells us the length of the longest happy prefix for the full string.

This works because KMP incrementally builds knowledge about prefix-suffix matches while avoiding redundant comparisons.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Repeated substring comparisons become too expensive |
| Optimal KMP Prefix Function | O(n) | O(n) | Efficiently reuses previously matched prefix information |

## Algorithm Walkthrough

### Building the Prefix Function

We create an array called `lps` where:

- `lps[i]` represents the length of the longest proper prefix of `s[:i+1]`
- That is also a suffix of `s[:i+1]`

For example:

| Prefix | Longest Matching Prefix/Suffix | Length |
| --- | --- | --- |
| `"a"` | `""` | 0 |
| `"ab"` | `""` | 0 |
| `"aba"` | `"a"` | 1 |
| `"abab"` | `"ab"` | 2 |

### Step-by-Step Process

1. Initialize an array `lps` of size `n` with all zeros.

Initially, we assume no prefix-suffix matches exist.
2. Maintain a variable `length`.

This variable stores the current longest matching prefix-suffix length while building the array.
3. Start iterating from index `1`.

The first character alone cannot have a proper prefix and suffix.
4. Compare `s[i]` with `s[length]`.

If they match, we can extend the current prefix-suffix match by one character.
5. When characters match:

- Increment `length`
- Set `lps[i] = length`
- Move to the next index
6. When characters do not match:

We do not restart from zero immediately.

Instead, we fall back to a smaller previously known valid prefix using:

```
length = lps[length - 1]
```

This is the core optimization of KMP.
7. Repeat fallback until:

- Characters match, or
- `length` becomes zero
8. After processing the entire string:

- `lps[-1]` contains the length of the longest happy prefix
- Return `s[:lps[-1]]`

### Why it works

The KMP prefix function maintains an invariant:

- At every step, `length` represents the longest valid prefix-suffix match for the substring processed so far.

When a mismatch occurs, KMP avoids recomputing from scratch by falling back to the next best previously known prefix length. Since every character is processed at most a constant number of times, the algorithm remains linear.

The final value `lps[n-1]` therefore correctly represents the longest proper prefix that is also a suffix of the full string.

## Python Solution

```python
class Solution:
    def longestPrefix(self, s: str) -> str:
        n = len(s)

        lps = [0] * n
        length = 0
        i = 1

        while i < n:
            if s[i] == s[length]:
                length += 1
                lps[i] = length
                i += 1
            else:
                if length != 0:
                    length = lps[length - 1]
                else:
                    lps[i] = 0
                    i += 1

        longest_length = lps[-1]
        return s[:longest_length]
```

The implementation begins by creating the `lps` array, which stores the longest prefix-suffix length for every prefix of the string.

The variable `length` tracks the current candidate prefix length. As we scan through the string, matching characters allow us to extend the current prefix-suffix relationship.

When a mismatch occurs, instead of restarting entirely, we reuse previously computed information through the `lps` array. This is the optimization that makes KMP linear rather than quadratic.

At the end of processing, the final value in the `lps` array gives the length of the longest happy prefix for the entire string. We simply slice the string using that length.

## Go Solution

```go
func longestPrefix(s string) string {
	n := len(s)

	lps := make([]int, n)

	length := 0
	i := 1

	for i < n {
		if s[i] == s[length] {
			length++
			lps[i] = length
			i++
		} else {
			if length != 0 {
				length = lps[length-1]
			} else {
				lps[i] = 0
				i++
			}
		}
	}

	longestLength := lps[n-1]
	return s[:longestLength]
}
```

The Go implementation follows the exact same algorithmic structure as the Python version.

One small difference is that Go strings are byte-indexable, which works perfectly here because the problem guarantees lowercase English letters only.

The `make([]int, n)` call initializes the `lps` slice with zeros automatically.

Returning an empty string is naturally handled because slicing with `s[:0]` produces `""`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each character is processed at most a constant number of times |
| Space | O(n) | The LPS array stores one integer per character |

The efficiency comes from avoiding repeated substring comparisons.

Although mismatches may trigger fallback operations, the `length` variable only moves backward a limited number of times overall. Across the entire execution, every increment and decrement contributes to a total linear amount of work.

This guarantees `O(n)` runtime even for very large strings.

## Edge Cases

### Single Character Strings

A string of length one cannot have a non-empty proper prefix because removing the entire string leaves nothing.

For example:

```
"a"
```

The implementation correctly returns `""` because the final LPS value remains `0`.

### Strings With All Identical Characters

Strings like:

```
"aaaaa"
```

can be tricky because every prefix is also a suffix.

A naive implementation may accidentally return the entire string itself, which is forbidden.

The KMP approach avoids this automatically because the LPS definition uses a proper prefix, meaning the full string is never considered.

The answer becomes `"aaaa"` rather than `"aaaaa"`.

### No Matching Prefix and Suffix

For inputs like:

```
"abcde"
```

there are no matching prefix-suffix pairs at all.

The algorithm handles this naturally because every mismatch eventually falls back to length `0`, leaving the final LPS value equal to zero.

The returned substring becomes:

```
s[:0]
```

which correctly produces the empty string.
