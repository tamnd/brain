# LeetCode 1352 - Product of the Last K Numbers

## Problem Understanding

This problem asks us to design a data structure that processes a stream of integers one value at a time. After numbers are added to the stream, we must efficiently answer queries asking for the product of the last `k` numbers that were inserted.

The class exposes two operations:

- `add(num)` appends a new integer to the stream.
- `getProduct(k)` returns the multiplication result of the most recent `k` numbers.

The key challenge is that these operations may be called up to `4 * 10^4` times, and the follow-up specifically asks whether both operations can be implemented in constant time.

At first glance, computing the product of the last `k` numbers seems straightforward. We could simply store every number and multiply the last `k` values whenever `getProduct` is called. However, if many queries ask for large `k`, repeatedly multiplying numbers becomes expensive.

The most important complication is the presence of zeros. Any product range containing a zero becomes zero. This breaks ordinary division-based prefix product approaches unless zeros are handled carefully.

The constraints also provide important guarantees:

- Every number is between `0` and `100`.
- The product of any contiguous sequence fits inside a signed 32-bit integer.
- We will never be asked for more numbers than currently exist in the stream.

These guarantees simplify implementation because we do not need to worry about integer overflow beyond normal language integer handling.

Several edge cases are especially important:

- Adding `0` resets multiplication behavior because any product spanning that zero becomes `0`.
- Multiple zeros may appear in the stream.
- `k` may equal the entire stream length.
- Queries may immediately follow zeros.
- The stream may contain many `1`s, which should not affect products but still must be processed correctly.

## Approaches

### Brute Force Approach

The simplest solution is to store every number in an array. Whenever `getProduct(k)` is called, we iterate backward through the last `k` elements and multiply them together.

This works because the definition of the problem directly asks for the product of the most recent `k` values. By explicitly multiplying those values, we always obtain the correct answer.

However, this approach is inefficient for large inputs. Each `getProduct(k)` call requires `O(k)` time. Since there may be up to `4 * 10^4` operations overall, repeated large queries can become too slow.

### Optimal Approach

The key observation is that repeated multiplication over ranges can be optimized using prefix products.

Suppose we maintain cumulative products:

- `prefix[i]` represents the product of all numbers up to index `i`.

Then the product of a suffix range can be computed using division:

$$\text{product}(l \dots r) = \frac{\text{prefix}[r]}{\text{prefix}[l-1]}$$

This allows range products to be answered in constant time.

The complication is zeros. Division does not work across zeros because prefix products become zero permanently after the first zero.

The solution is to reset the prefix product array whenever a zero is added.

After a zero:

- Any query spanning beyond that zero must return `0`.
- Any query entirely after the zero can still use prefix division normally.

So instead of storing all prefix products forever, we only store prefix products since the most recent zero.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | `O(k)` per query | `O(n)` | Multiply the last `k` numbers directly |
| Optimal | `O(1)` per operation | `O(n)` | Uses prefix products and resets after zeros |

## Algorithm Walkthrough

1. Initialize a prefix product list with a starting value of `1`.

This sentinel value simplifies division logic. The list begins as `[1]`, meaning the product of zero numbers is `1`.
2. When `add(num)` is called, check whether the number is zero.

If the number is zero, all products spanning this position must become zero. We reset the prefix product list back to `[1]`.
3. If the number is nonzero, multiply it with the most recent prefix product.

Suppose the current prefix list ends with value `P`. After adding `num`, append `P * num`.
4. When `getProduct(k)` is called, first determine whether the query range crosses a zero.

Since the prefix array only stores numbers after the latest zero, if `k` is greater than or equal to the current prefix array length, then the requested range includes a zero. In that case, return `0`.
5. Otherwise, compute the range product using division.

Let:

- `last = prefix[-1]`
- `before = prefix[-k-1]`

Then:

$$\text{answer} = \frac{last}{before}$$

This isolates the product of the last `k` numbers.

### Why it works

The algorithm maintains an invariant:

- The prefix array always contains cumulative products of numbers since the most recent zero.

Because cumulative products are multiplicative, dividing two prefix products isolates the desired suffix range. Resetting after zeros guarantees that no stored prefix product ever contains a zero, so division remains valid. If a query extends beyond the current prefix segment, then it necessarily includes a zero, so the answer must be `0`.

## Python Solution

```python
class ProductOfNumbers:

    def __init__(self):
        self.prefix_products = [1]

    def add(self, num: int) -> None:
        if num == 0:
            self.prefix_products = [1]
        else:
            new_product = self.prefix_products[-1] * num
            self.prefix_products.append(new_product)

    def getProduct(self, k: int) -> int:
        if k >= len(self.prefix_products):
            return 0

        return (
            self.prefix_products[-1] //
            self.prefix_products[-k - 1]
        )

# Your ProductOfNumbers object will be instantiated and called as such:
# obj = ProductOfNumbers()
# obj.add(num)
# param_2 = obj.getProduct(k)
```

The implementation stores cumulative products in `prefix_products`.

The constructor initializes the array with `[1]`. This sentinel value allows range products to be computed uniformly without needing special handling for the first element.

In `add`, zeros are handled specially. When a zero appears, the prefix list resets because every product containing that zero becomes zero. For nonzero values, we extend the cumulative product chain.

The `getProduct` method first checks whether the requested range extends beyond the current prefix segment. If it does, the query must include a zero, so we return `0`.

Otherwise, we compute the answer using prefix division. Integer division is safe because the denominator is always an exact divisor of the numerator.

## Go Solution

```go
type ProductOfNumbers struct {
    prefixProducts []int
}

func Constructor() ProductOfNumbers {
    return ProductOfNumbers{
        prefixProducts: []int{1},
    }
}

func (this *ProductOfNumbers) Add(num int) {
    if num == 0 {
        this.prefixProducts = []int{1}
    } else {
        newProduct := this.prefixProducts[len(this.prefixProducts)-1] * num
        this.prefixProducts = append(this.prefixProducts, newProduct)
    }
}

func (this *ProductOfNumbers) GetProduct(k int) int {
    if k >= len(this.prefixProducts) {
        return 0
    }

    last := this.prefixProducts[len(this.prefixProducts)-1]
    before := this.prefixProducts[len(this.prefixProducts)-k-1]

    return last / before
}

/**
 * Your ProductOfNumbers object will be instantiated and called as such:
 * obj := Constructor();
 * obj.Add(num);
 * param_2 := obj.GetProduct(k);
 */
```

The Go implementation follows the same logic as the Python version.

Slices are used to store prefix products. Resetting after a zero is done by assigning a new slice containing only `1`.

Go integer division behaves correctly here because the numerator is always divisible by the denominator. The problem guarantees that results fit within 32-bit integers, so normal `int` arithmetic is sufficient.

## Edge Cases

### Queries That Cross a Zero

This is the most important edge case in the problem. Any product range containing a zero must evaluate to zero. A naive prefix product implementation would fail because dividing prefix products after a zero does not recover valid results.

The implementation handles this by resetting the prefix array whenever zero is added. Then, if a query requests more numbers than currently exist in the prefix array, we know the range must include a zero, so we return `0`.

### Multiple Consecutive Zeros

Consecutive zeros can easily introduce bugs if reset logic is incomplete. For example:

```
add(0)
add(0)
add(5)
```

The implementation correctly resets the prefix array each time a zero appears. After the second zero, the structure still behaves correctly because the prefix array becomes `[1]` again.

### Querying the Entire Current Nonzero Segment

Suppose the current prefix array is:

```
[1, 2, 10, 40]
```

A query like `getProduct(3)` should return the product of all nonzero values in the current segment.

The implementation correctly computes:

```
40 / 1 = 40
```

The sentinel value `1` at the beginning of the prefix array makes this case work naturally without requiring special branching logic.
