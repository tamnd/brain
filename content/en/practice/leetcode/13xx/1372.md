## Solution

## Problem Understanding

The problem gives us the root of a binary tree and asks us to find the length of the longest ZigZag path inside the tree.

A ZigZag path alternates directions at every step. If one move goes to the left child, the next move must go to the right child. If one move goes to the right child, the next move must go to the left child.

The important detail is that the path can start at any node, not necessarily the root. We are also free to choose the first direction. The length of a ZigZag path is defined as the number of edges traveled, which is equivalent to the number of visited nodes minus one.

For example:

- A single node by itself has ZigZag length `0`
- A path `root -> left` has length `1`
- A path `root -> left -> right -> left` has length `3`

The input is a standard binary tree where each node has at most two children. The constraints allow up to `5 * 10^4` nodes, which is large enough that inefficient repeated traversals can become too slow.

These constraints strongly suggest that we should aim for a linear time solution, ideally visiting each node only once.

Several edge cases are important:

- A tree with only one node should return `0`
- A completely skewed tree, where every node only has one child in the same direction, produces only short ZigZag paths
- The longest ZigZag path may start in the middle of the tree rather than at the root
- The path does not need to end at a leaf
- We must correctly alternate directions after every move

A naive implementation can easily make mistakes by restarting paths incorrectly or by not considering every node as a possible starting point.

## Approaches

### Brute Force Approach

The brute force idea is to try every node as a starting point and attempt both possible directions:

1. Start by moving left
2. Start by moving right

From each starting state, we recursively continue alternating directions until the path cannot continue.

This approach is correct because it explicitly explores every possible ZigZag path in the tree. Since every valid path is examined, the maximum length found must be the answer.

However, the same subproblems are recomputed many times. For example, different starting points may repeatedly explore the same subtree with the same expected next direction.

In the worst case, for every node we may traverse a large portion of the tree again, leading to quadratic complexity.

With up to `50,000` nodes, an `O(n^2)` solution is too slow.

### Optimal DFS Dynamic Programming Approach

The key observation is that at every node, the only information we need is:

- What was the previous direction?
- What is the current ZigZag length?

If we moved left previously, the next valid move must go right.

If we moved right previously, the next valid move must go left.

This means we can perform a single DFS traversal while carrying:

- The current node
- The direction of the previous move
- The current ZigZag length

Whenever we continue in the correct alternating direction, we increase the length.

Whenever we restart from a child in the same direction, we reset the length back to `1`.

Because each node is processed only a constant number of times, the total complexity becomes linear.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(h) | Recomputes many overlapping ZigZag paths |
| Optimal DFS | O(n) | O(h) | Single DFS traversal with direction tracking |

Here, `h` is the height of the tree.

## Algorithm Walkthrough

### Step 1: Maintain a Global Maximum

We keep a variable called `max_length` that stores the best ZigZag path found so far.

Every time we visit a node, we compare the current path length against this maximum.

### Step 2: Define DFS State

Our DFS function needs three pieces of information:

1. The current node
2. The direction used to arrive at this node
3. The current ZigZag length

For example:

- If we came from the left child, the next valid move must go right
- If we came from the right child, the next valid move must go left

### Step 3: Continue the ZigZag

Suppose we previously moved left.

Then:

- Moving right continues the ZigZag
- Moving left breaks the pattern and starts a new path

So:

- Continuing direction alternation increases the length by `1`
- Restarting resets the length to `1`

### Step 4: Explore Both Children

At every node:

- Explore the left child
- Explore the right child

But update the path length differently depending on whether the move alternates direction correctly.

### Step 5: Start DFS from Root

The root has no previous direction, so we simulate both possible starts:

- Start by moving left
- Start by moving right

Both begin with length `0`.

### Step 6: Return the Maximum

After DFS completes, `max_length` contains the answer.

### Why it works

The algorithm works because every possible ZigZag path can be represented as a sequence of alternating left and right moves. During DFS, every edge is explored in both possible ZigZag contexts.

At each node, the DFS state precisely captures the only information needed to continue the path correctly:

- previous direction
- current length

Since the traversal explores every valid continuation exactly once and updates the global maximum whenever a longer path is found, the final result must be the longest ZigZag path in the tree.

## Python Solution

```python
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import Optional

class Solution:
    def longestZigZag(self, root: Optional[TreeNode]) -> int:
        max_length = 0

        def dfs(node: Optional[TreeNode], direction: str, length: int) -> None:
            nonlocal max_length

            if not node:
                return

            max_length = max(max_length, length)

            if direction == "left":
                # Continue zigzag by going right
                dfs(node.right, "right", length + 1)

                # Restart path by going left again
                dfs(node.left, "left", 1)
            else:
                # Continue zigzag by going left
                dfs(node.left, "left", length + 1)

                # Restart path by going right again
                dfs(node.right, "right", 1)

        dfs(root.left, "left", 1)
        dfs(root.right, "right", 1)

        return max_length
```

The implementation uses a recursive DFS helper function.

The `direction` parameter represents the direction used to arrive at the current node. This determines which child continues the ZigZag pattern.

The `length` parameter stores the current ZigZag length.

At each node:

- We update the global maximum
- We recursively continue the valid alternating move
- We also restart a new ZigZag path from the same-direction child

For example, if the previous move was left:

- Going right extends the ZigZag
- Going left starts over with length `1`

The recursion naturally explores all possible ZigZag paths while avoiding redundant full traversals.

## Go Solution

```go
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func longestZigZag(root *TreeNode) int {
    maxLength := 0

    var dfs func(node *TreeNode, direction string, length int)

    dfs = func(node *TreeNode, direction string, length int) {
        if node == nil {
            return
        }

        if length > maxLength {
            maxLength = length
        }

        if direction == "left" {
            // Continue zigzag
            dfs(node.Right, "right", length+1)

            // Restart
            dfs(node.Left, "left", 1)
        } else {
            // Continue zigzag
            dfs(node.Left, "left", length+1)

            // Restart
            dfs(node.Right, "right", 1)
        }
    }

    dfs(root.Left, "left", 1)
    dfs(root.Right, "right", 1)

    return maxLength
}
```

The Go implementation closely mirrors the Python solution.

The main difference is handling `nil` pointers explicitly instead of Python's `None`.

Go also requires defining the DFS function variable before assigning the recursive closure.

No special handling for integer overflow is needed because the maximum path length is bounded by the number of nodes, which fits comfortably inside Go's `int`.

## Edge Cases

### Single Node Tree

A tree containing only one node is an important boundary case. Since ZigZag length counts edges rather than nodes, the correct answer is `0`.

A buggy implementation might incorrectly return `1` because it counts the starting node itself. This implementation avoids that issue because DFS only begins from children of the root, each with initial length `1`. If no children exist, DFS immediately returns and the maximum remains `0`.

### Completely Skewed Tree

Consider a tree where every node only has a left child:

```
1
/
2
/
3
/
4
```

The longest valid ZigZag path is only `1` because after moving left once, the next move must go right, which is impossible.

A naive implementation may incorrectly count the entire chain. This solution correctly resets the path whenever the same direction repeats.

### Longest Path Starts Below the Root

The optimal ZigZag path does not necessarily begin at the root.

For example:

```
    1
   /
  2
 /
3
 \
  4
 /
5
```

The best ZigZag path starts at node `3`.

A flawed implementation that only tracks paths originating from the root would miss this case. Our DFS restarts paths at every node automatically, ensuring all possible starting positions are explored correctly.
