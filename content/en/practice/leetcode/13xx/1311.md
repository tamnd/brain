## Solution

## Problem Understanding

In this problem, we are given a social network represented as a graph. Each person has:

- A unique integer ID from `0` to `n - 1`
- A list of videos they have watched
- A list of friends

The `friends` array defines an undirected graph because if person `i` is friends with person `j`, then person `j` is also friends with person `i`.

We are also given:

- A starting person `id`
- A friendship `level`

The goal is to find all videos watched by people whose shortest friendship distance from the starting person is exactly equal to `level`.

The important phrase is "shortest path exactly equal to level". This means:

- Level `1` means direct friends
- Level `2` means friends of friends, excluding anyone already reachable in fewer steps
- Level `3` means people exactly three friendship edges away
- And so on

Once we collect all videos watched by people at the required level, we count how many times each video appears. The final output must be sorted:

1. By frequency in ascending order
2. Alphabetically if frequencies are equal

For example, if the frequencies are:

- `"B"` → 1
- `"C"` → 2

Then the answer is `["B", "C"]`.

The constraints are relatively small:

- `n <= 100`
- Each person can watch up to `100` videos

Because the graph is small, a graph traversal solution is more than fast enough. The natural way to compute shortest friendship distance is Breadth-First Search, because BFS explores nodes level by level.

There are several important edge cases:

- The graph may contain cycles, so we must track visited users
- Multiple users at the same level may watch the same video
- Videos can tie in frequency, requiring alphabetical sorting
- The target level may contain only one person
- Some users may have no friends

The problem guarantees that friendship relationships are symmetric, so the graph is undirected and valid.

## Approaches

### Brute Force Approach

A brute force approach would try to compute the shortest path from the starting user to every other user separately.

For each person:

1. Run a graph search from `id`
2. Compute the shortest distance
3. If the distance equals `level`, collect their videos
4. Count frequencies
5. Sort the result

This approach is correct because shortest paths are computed explicitly for every node. However, it is inefficient because we repeatedly traverse the same graph structure.

If we run BFS separately for every node, the complexity becomes unnecessarily large.

### Optimal Approach

The key observation is that Breadth-First Search naturally explores a graph level by level.

When BFS starts from the given user:

- The first layer contains all friends at distance `1`
- The second layer contains all users at distance `2`
- And so on

This exactly matches the definition of friendship level in the problem.

Therefore, instead of computing shortest paths individually, we can:

1. Run one BFS from the starting user
2. Stop when we reach the desired level
3. Collect videos from users currently in the queue
4. Count frequencies with a hash map
5. Sort by frequency and then alphabetically

This avoids repeated traversals and gives an efficient solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × (n + e)) | O(n) | Runs graph traversal repeatedly for many users |
| Optimal | O(n + e + v log v) | O(n + v) | Single BFS traversal with frequency sorting |

Here:

- `n` is the number of users
- `e` is the number of friendship edges
- `v` is the number of unique videos collected at the target level

## Algorithm Walkthrough

1. Create a queue for Breadth-First Search and initialize it with the starting user `id`.
2. Create a `visited` set to prevent revisiting users. This is necessary because the friendship graph may contain cycles.
3. Start BFS level traversal.
4. For each BFS layer:

- Process all users currently in the queue
- Explore their friends
- Add unvisited friends to the queue
- Mark them as visited immediately to avoid duplicate insertion
5. Repeat the BFS process until we have advanced exactly `level` steps from the starting user.
6. After completing the BFS for the target level, every user remaining in the queue is exactly `level` friendships away.
7. Create a frequency hash map:

- Iterate through all users in the queue
- For every watched video, increment its count
8. Convert the frequency map into a sortable list.
9. Sort the videos:

- First by frequency ascending
- Then alphabetically ascending
10. Extract the sorted video names and return them.

### Why it works

Breadth-First Search guarantees that nodes are visited in increasing order of shortest path distance from the starting node. Because we process the graph level by level, after exactly `level` BFS expansions, the queue contains precisely the users whose shortest friendship distance equals `level`.

The frequency map correctly counts video occurrences among those users, and the final sorting rule directly matches the problem specification.

## Python Solution

```python
from collections import deque, Counter
from typing import List

class Solution:
    def watchedVideosByFriends(
        self,
        watchedVideos: List[List[str]],
        friends: List[List[int]],
        id: int,
        level: int
    ) -> List[str]:

        visited = set([id])
        queue = deque([id])

        current_level = 0

        while queue and current_level < level:
            level_size = len(queue)

            for _ in range(level_size):
                person = queue.popleft()

                for friend in friends[person]:
                    if friend not in visited:
                        visited.add(friend)
                        queue.append(friend)

            current_level += 1

        frequency = Counter()

        for person in queue:
            for video in watchedVideos[person]:
                frequency[video] += 1

        sorted_videos = sorted(
            frequency.keys(),
            key=lambda video: (frequency[video], video)
        )

        return sorted_videos
```

The implementation begins by initializing a BFS queue with the starting user. The `visited` set ensures that every user is processed only once, which prevents infinite loops in cyclic friendship graphs.

The BFS proceeds level by level. The variable `current_level` tracks how far we are from the starting user. For each layer, we process exactly the current queue size, ensuring clean separation between friendship distances.

Once BFS reaches the required level, the queue contains only users at that exact distance. We then iterate through their watched videos and count frequencies using `Counter`.

Finally, the videos are sorted using a tuple sort key:

```python
(frequency[video], video)
```

This ensures ascending frequency order first, with alphabetical order as the tie breaker.

## Go Solution

```go
package main

import (
	"sort"
)

func watchedVideosByFriends(
	watchedVideos [][]string,
	friends [][]int,
	id int,
	level int,
) []string {

	n := len(friends)

	visited := make([]bool, n)
	visited[id] = true

	queue := []int{id}

	currentLevel := 0

	for len(queue) > 0 && currentLevel < level {
		size := len(queue)
		nextQueue := []int{}

		for i := 0; i < size; i++ {
			person := queue[i]

			for _, friend := range friends[person] {
				if !visited[friend] {
					visited[friend] = true
					nextQueue = append(nextQueue, friend)
				}
			}
		}

		queue = nextQueue
		currentLevel++
	}

	frequency := make(map[string]int)

	for _, person := range queue {
		for _, video := range watchedVideos[person] {
			frequency[video]++
		}
	}

	result := make([]string, 0, len(frequency))

	for video := range frequency {
		result = append(result, video)
	}

	sort.Slice(result, func(i, j int) bool {
		if frequency[result[i]] == frequency[result[j]] {
			return result[i] < result[j]
		}
		return frequency[result[i]] < frequency[result[j]]
	})

	return result
}
```

The Go implementation follows the same BFS strategy as the Python version. Instead of Python's `deque`, slices are used to simulate queue behavior.

A boolean slice is used for the `visited` structure because user IDs are contiguous integers from `0` to `n - 1`, making indexed lookup efficient.

The sorting logic uses `sort.Slice` with a custom comparator that implements the required ordering:

1. Frequency ascending
2. Alphabetical ascending for ties

Go does not require any special handling for integer overflow here because all counts remain very small under the problem constraints.

## Edge Cases

One important edge case is a cyclic friendship graph. Since friendships are bidirectional, cycles naturally exist. Without a `visited` set, BFS could repeatedly revisit the same users and either loop forever or overcount videos. The implementation marks users as visited immediately when they are added to the queue, ensuring every user is processed only once.

Another important edge case is when multiple videos have the same frequency. A naive implementation might sort only by frequency and produce nondeterministic ordering for ties. The problem explicitly requires alphabetical ordering as the secondary criterion. The implementation handles this correctly by sorting with the tuple:

```
(frequency[video], video)
```

A third important edge case occurs when the target friendship level contains only one user or very few users. Some implementations accidentally include users from previous BFS layers. This solution avoids that bug by processing BFS level by level using the current queue size. After exactly `level` iterations, the queue contains only users at the desired shortest-path distance.

A final subtle case is repeated videos watched by the same user. The problem counts all occurrences from the input lists, so repeated video names must increment the frequency multiple times. The frequency map correctly increments every appearance rather than assuming uniqueness.
