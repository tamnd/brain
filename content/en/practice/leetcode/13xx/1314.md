## Solution

## Problem Understanding

The problem gives us a two dimensional matrix `mat` of size `m x n` and an integer `k`. For every position `(i, j)` in the matrix, we must compute the sum of all elements within a square region centered at `(i, j)`.

The block around `(i, j)` includes every cell `(r, c)` such that:

- `i - k <= r <= i + k`
- `j - k <= c <= j + k`

However, the matrix boundaries still apply. If part of the square extends outside the matrix, we simply ignore those invalid positions.

The output is another matrix `answer` of the same size, where each `answer[i][j]` contains the sum of all valid cells inside that block.

For example, if `k = 1`, each cell considers a surrounding `3 x 3` neighborhood whenever possible. Near edges and corners, the neighborhood becomes smaller because some indices would fall outside the matrix.

The constraints are relatively small:

- `1 <= m, n <= 100`
- `1 <= k <= 100`
- `1 <= mat[i][j] <= 100`

Even though the matrix is not huge, a naive implementation can still become unnecessarily expensive because each output cell may repeatedly sum overlapping regions. Since there can be up to `10,000` cells and each block can also contain up to `10,000` cells, repeated summation would lead to significant redundant work.

Some important edge cases include matrices with only one row or one column, very large values of `k` that cover the entire matrix, and cells located on borders or corners where the valid block area shrinks automatically.

## Approaches

### Brute Force Approach

The most direct solution is to compute every block sum independently.

For each cell `(i, j)`, we iterate through every row from `i-k` to `i+k` and every column from `j-k` to `j+k`. We check whether each position is inside the matrix bounds before adding it to the total.

This approach is straightforward and obviously correct because it directly follows the problem definition. However, it performs a large amount of repeated work. Neighboring cells often have heavily overlapping regions, yet the brute force solution recalculates those sums from scratch every time.

If the matrix is `m x n`, and each block may contain up to `(2k+1)^2` cells, the total complexity becomes:

$$O(m \cdot n \cdot (2k+1)^2)$$

In the worst case, this approaches `O(m * n * m * n)`.

### Optimal Approach, 2D Prefix Sum

The key observation is that we repeatedly compute sums of rectangular regions.

Instead of summing each rectangle from scratch, we can preprocess the matrix using a two dimensional prefix sum array. A prefix sum allows us to compute the sum of any rectangle in constant time.

The idea is similar to one dimensional prefix sums:

- In one dimension, prefix sums allow range sums in `O(1)`
- In two dimensions, a 2D prefix sum allows rectangular sums in `O(1)`

For a prefix matrix `prefix`, each entry stores the sum of all elements from the top left corner `(0,0)` to `(i-1,j-1)`.

Once this preprocessing is complete, any rectangular region can be computed using inclusion exclusion.

This removes redundant repeated summation and reduces the overall complexity to `O(m * n)`.

## Algorithm Walkthrough

### Optimal Algorithm Using 2D Prefix Sum

1. Create a prefix sum matrix with dimensions `(m + 1) x (n + 1)`.

The extra row and column simplify boundary handling. `prefix[r][c]` will represent the sum of all elements in the rectangle from `(0,0)` to `(r-1,c-1)`.
2. Fill the prefix sum matrix.

For each cell `(r, c)` in the original matrix:

$$prefix[r+1][c+1] =
mat[r][c]
+ prefix[r][c+1]
+ prefix[r+1][c]
- prefix[r][c]$$

The subtraction avoids double counting the overlapping area.
3. For each output cell `(i, j)`, determine the block boundaries.

Compute:

- `top = max(0, i-k)`
- `bottom = min(m-1, i+k)`
- `left = max(0, j-k)`
- `right = min(n-1, j+k)`

These ensure we stay within matrix bounds.
4. Use the prefix sum matrix to compute the rectangle sum.

The rectangle sum formula is:

$$\text{sum} =
prefix[bottom+1][right+1]
- prefix[top][right+1]
- prefix[bottom+1][left]
+ prefix[top][left]$$

1. Store the computed sum in the answer matrix.
2. Repeat for every cell.

### Why it works

The prefix sum matrix stores cumulative sums for all top left rectangles. Using inclusion exclusion, we can isolate any rectangular region in constant time.

The formula:

$$A - B - C + D$$

works because:

- `A` contains the full large rectangle
- `B` removes rows above the target
- `C` removes columns left of the target
- `D` restores the overlapping area that was subtracted twice

Therefore, every block sum is computed exactly once and correctly.

## Python Solution

```python
from typing import List

class Solution:
    def matrixBlockSum(self, mat: List[List[int]], k: int) -> List[List[int]]:
        m = len(mat)
        n = len(mat[0])

        # Build prefix sum matrix
        prefix = [[0] * (n + 1) for _ in range(m + 1)]

        for r in range(m):
            for c in range(n):
                prefix[r + 1][c + 1] = (
                    mat[r][c]
                    + prefix[r][c + 1]
                    + prefix[r + 1][c]
                    - prefix[r][c]
                )

        # Build answer matrix
        answer = [[0] * n for _ in range(m)]

        for i in range(m):
            for j in range(n):
                top = max(0, i - k)
                bottom = min(m - 1, i + k)
                left = max(0, j - k)
                right = min(n - 1, j + k)

                block_sum = (
                    prefix[bottom + 1][right + 1]
                    - prefix[top][right + 1]
                    - prefix[bottom + 1][left]
                    + prefix[top][left]
                )

                answer[i][j] = block_sum

        return answer
```

The implementation begins by determining the matrix dimensions. A prefix sum matrix with one extra row and one extra column is created to simplify calculations near boundaries.

The first nested loop builds the prefix sum matrix. Each entry accumulates values from the top, left, and current cell while subtracting the overlapping top left region.

The second nested loop computes each block sum. The algorithm first clamps the rectangle boundaries to valid indices using `max` and `min`.

Finally, the rectangle sum formula extracts the desired region in constant time and stores it in the answer matrix.

## Go Solution

```go
func matrixBlockSum(mat [][]int, k int) [][]int {
	m := len(mat)
	n := len(mat[0])

	// Build prefix sum matrix
	prefix := make([][]int, m+1)
	for i := range prefix {
		prefix[i] = make([]int, n+1)
	}

	for r := 0; r < m; r++ {
		for c := 0; c < n; c++ {
			prefix[r+1][c+1] =
				mat[r][c] +
					prefix[r][c+1] +
					prefix[r+1][c] -
					prefix[r][c]
		}
	}

	// Build answer matrix
	answer := make([][]int, m)
	for i := range answer {
		answer[i] = make([]int, n)
	}

	for i := 0; i < m; i++ {
		for j := 0; j < n; j++ {
			top := max(0, i-k)
			bottom := min(m-1, i+k)
			left := max(0, j-k)
			right := min(n-1, j+k)

			blockSum := prefix[bottom+1][right+1] -
				prefix[top][right+1] -
				prefix[bottom+1][left] +
				prefix[top][left]

			answer[i][j] = blockSum
		}
	}

	return answer
}

func max(a, b int) int {
	if a > b {
		return a
	}
	return b
}

func min(a, b int) int {
	if a < b {
		return a
	}
	return b
}
```

The Go implementation follows the same logic as the Python version. Since Go does not provide built in `min` and `max` functions for integers, helper functions are added.

Slices are used to construct both the prefix sum matrix and the result matrix. Integer overflow is not a concern because the maximum possible matrix sum is:

$$100 \times 100 \times 100 = 1,000,000$$

which fits comfortably within Go's `int` type.

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m × n) | One pass to build prefix sums and one pass to compute answers |
| Space | O(m × n) | Prefix sum matrix requires additional storage |

The prefix sum matrix construction processes every matrix cell exactly once. Computing all block sums also touches each cell once, and each rectangle query takes constant time. Therefore the overall running time is linear in the number of matrix cells.

The additional memory comes from the prefix sum matrix, which stores `(m+1) x (n+1)` integers.

## Edge Cases

### Very Large k

One important edge case occurs when `k` is larger than the matrix dimensions. In this situation, every block should include the entire matrix.

A buggy implementation might attempt to access invalid indices outside the matrix bounds. This solution avoids that problem by clamping boundaries with:

```
top = max(0, i - k)
bottom = min(m - 1, i + k)
```

and similarly for columns.

As a result, all rectangle coordinates remain valid.

### Single Row or Single Column

Matrices with only one row or one column are easy places for off by one mistakes.

For example:

```
[[1,2,3,4]]
```

still requires correct left and right boundary handling even though there are no vertical neighbors.

The implementation works correctly because the prefix sum formula is dimension independent and treats rows and columns uniformly.

### k = 0

When `k = 0`, the block contains only the current cell itself.

Some implementations accidentally include neighboring cells due to incorrect boundary formulas. Here:

```
top = i
bottom = i
left = j
right = j
```

so the rectangle query extracts exactly one element.

The output therefore becomes identical to the original matrix.
