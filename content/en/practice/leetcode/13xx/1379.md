## Solution

## Problem Understanding

The problem gives us two binary trees, `original` and `cloned`. The `cloned` tree is an exact deep copy of the `original` tree, meaning both trees have the same structure and the same node values, but they are made up of different node objects in memory.

We are also given a direct reference to a specific node called `target`, and this node exists inside the `original` tree. Our goal is to find and return the corresponding node inside the `cloned` tree.

The important detail is that we are not supposed to return a node with the same value from the original tree. We must return the actual node object from the cloned tree that occupies the same structural position as the target node in the original tree.

For example, suppose the original tree contains a node with value `3` as the right child of the root. The cloned tree will also contain a node with value `3` in the same position. We must return the node from the cloned tree, not the original one.

The constraints tell us that the tree can contain up to `10^4` nodes, which is large enough that we should prefer a linear time solution. The problem also guarantees that node values are unique. This simplifies the problem because we could theoretically identify the node using its value alone. However, the follow up asks what happens if duplicate values are allowed, which means a robust solution should rely on structure and traversal position rather than values alone.

Several edge cases are important to consider:

- The tree may contain only one node. In that case, the root itself is the target.
- The target node may be deep in the tree, such as in a skewed tree that behaves like a linked list.
- The target node may be a leaf node.
- The target node may be the root node.
- Future variations may allow duplicate values, so relying only on `node.val` is not always safe.

The safest and most general approach is to traverse both trees simultaneously and compare node references from the original tree.

## Approaches

### Brute Force Approach

A straightforward brute force idea is to first locate the path from the root of the original tree to the target node. For example, we might record a sequence like `"left -> right -> left"`.

Once we know this path, we could start from the root of the cloned tree and follow the exact same sequence of moves to arrive at the corresponding node.

This works because the cloned tree has the exact same structure as the original tree.

However, implementing this requires extra bookkeeping. We need a separate traversal to discover the path, and then another traversal to replay it on the cloned tree. Although the time complexity is still linear in the number of nodes, the implementation becomes more complicated than necessary.

### Optimal Approach

The key insight is that both trees have identical structure. If we traverse both trees at the same time, then whenever we visit a node in the original tree, we are simultaneously visiting the corresponding node in the cloned tree.

This means that during traversal:

- If the current node in the original tree is the `target`, then the current node in the cloned tree must be the answer.
- We do not need to compare values.
- We do not need to store paths.
- We only need synchronized traversal.

A depth first search works naturally here. At every recursive call:

- We receive one node from the original tree.
- We receive the matching node from the cloned tree.
- If the original node equals `target`, we return the cloned node.

Because every pair of nodes is visited in corresponding order, the correct cloned node is guaranteed to be found.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(h) | Find path to target, then replay path in cloned tree |
| Optimal | O(n) | O(h) | Traverse both trees simultaneously using DFS |

Here, `n` is the number of nodes and `h` is the height of the tree.

## Algorithm Walkthrough

1. Start a depth first traversal on both trees simultaneously.

We always process nodes in pairs:

- `original_node`
- `cloned_node`

These two nodes represent the same structural position in their respective trees.
2. Check whether the current original node is the target.

We compare node references directly:

```
if original_node == target:
```

This is important because multiple nodes could theoretically share the same value in the follow up version of the problem.
3. If the current original node is the target, return the cloned node.

Since both traversals are synchronized, the current cloned node is guaranteed to be the correct corresponding node.
4. Recursively search the left subtree.

If the answer exists in the left subtree, immediately return it.
5. If the left subtree does not contain the answer, recursively search the right subtree.
6. Continue until the target node is found.

Since the problem guarantees the target exists in the tree, the search will eventually succeed.

### Why it works

The algorithm maintains the invariant that every recursive call receives two nodes occupying the same structural position in the original and cloned trees. Therefore, when the traversal reaches the target node in the original tree, the simultaneously visited node in the cloned tree must be its exact counterpart. Since every node is visited at most once, the algorithm always finds the correct answer.

## Python Solution

```python
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, x):
#         self.val = x
#         self.left = None
#         self.right = None

from typing import Optional

class Solution:
    def getTargetCopy(
        self,
        original: TreeNode,
        cloned: TreeNode,
        target: TreeNode
    ) -> TreeNode:

        def dfs(original_node: Optional[TreeNode],
                cloned_node: Optional[TreeNode]) -> Optional[TreeNode]:

            if original_node is None:
                return None

            if original_node == target:
                return cloned_node

            left_result = dfs(original_node.left, cloned_node.left)

            if left_result is not None:
                return left_result

            return dfs(original_node.right, cloned_node.right)

        return dfs(original, cloned)
```

The implementation follows the synchronized traversal idea directly.

The helper function `dfs` receives two nodes at every step:

- `original_node` from the original tree
- `cloned_node` from the cloned tree

The first base case checks whether the current original node is `None`. If so, there is nothing to search.

Next, we check whether the current original node is exactly the target node. If it is, we immediately return the corresponding cloned node.

The algorithm then recursively explores the left subtree first. If the target is found there, the result is returned immediately.

Otherwise, the search continues into the right subtree.

Because both trees are traversed in lockstep, the cloned node returned is guaranteed to correspond to the target node.

## Go Solution

```go
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */

func getTargetCopy(original *TreeNode, cloned *TreeNode, target *TreeNode) *TreeNode {

    var dfs func(*TreeNode, *TreeNode) *TreeNode

    dfs = func(originalNode *TreeNode, clonedNode *TreeNode) *TreeNode {

        if originalNode == nil {
            return nil
        }

        if originalNode == target {
            return clonedNode
        }

        leftResult := dfs(originalNode.Left, clonedNode.Left)

        if leftResult != nil {
            return leftResult
        }

        return dfs(originalNode.Right, clonedNode.Right)
    }

    return dfs(original, cloned)
}
```

The Go implementation is nearly identical to the Python version.

The main difference is that Go uses pointers explicitly. Tree nodes are represented as `*TreeNode`, and comparisons such as:

```go
if originalNode == target
```

compare pointer addresses directly.

Go also uses `nil` instead of Python's `None`.

No integer overflow concerns exist here because the algorithm only traverses pointers and does not perform arithmetic operations.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each node is visited at most once |
| Space | O(h) | Recursive call stack stores at most tree height |

The algorithm performs a standard depth first traversal. In the worst case, every node may need to be examined before the target is found, leading to `O(n)` time complexity.

The space complexity comes from recursion depth. In a balanced tree, the height is `O(log n)`. In a skewed tree, the height can become `O(n)`.

## Edge Cases

### Single Node Tree

The smallest possible input contains only one node. In this case, the root is automatically the target node. A buggy implementation might unnecessarily recurse into children or fail to handle the absence of subtrees correctly. The current implementation handles this naturally because the root comparison succeeds immediately.

### Skewed Tree

A tree may be completely skewed to one side, effectively behaving like a linked list. This creates the maximum possible recursion depth. Some implementations accidentally assume a balanced tree structure. The current solution correctly traverses node by node regardless of shape.

### Duplicate Values in Follow Up Version

Although the original problem guarantees unique values, the follow up asks about duplicate values. Any implementation that searches by value alone could return the wrong node if multiple nodes share the same value. This implementation avoids that issue entirely because it compares node references in the original tree rather than values.

### Target is a Leaf Node

The target node may have no children. Some implementations incorrectly stop traversal too early or mishandle `None` children. The recursive base case ensures null children are handled safely while still allowing leaf nodes to be matched correctly.
