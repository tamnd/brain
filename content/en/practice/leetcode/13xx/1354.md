## Solution

## Problem Understanding

The problem asks whether it is possible to construct a given `target` array starting from an array `arr` of the same length, initially filled with 1's, using a specific operation repeatedly. The operation is: compute the sum `x` of all elements in the current array, and then replace any single element with `x`. We repeat this operation as many times as needed. The output should be a boolean, `true` if it is possible to reach `target` and `false` otherwise.

The input is an array of integers, each between 1 and 10^9, and the length of the array `n` can be up to 50,000. This means the solution must be efficient and avoid simulating every operation naively. Important edge cases include arrays where all elements are already 1, arrays with very large numbers, and arrays where the sum of all elements cannot produce a specific element through repeated additions.

The key insight is that directly building up from `[1,1,...,1]` is computationally expensive. Instead, we can consider the problem in reverse: starting from `target`, can we reduce it to all 1's by repeatedly subtracting the sum of the other elements from the largest element? This allows us to efficiently simulate the process in reverse.

## Approaches

The brute-force approach is to simulate the array operations starting from `[1,1,...,1]`. At each step, compute the sum of the current array and try replacing each element with that sum, recursively or iteratively exploring all possibilities. While correct in principle, this approach is infeasible due to the exponential number of possible sequences when `n` is large.

The optimal approach works in reverse. We repeatedly identify the largest element in `target`, compute the sum of the rest of the elements, and attempt to subtract that sum from the largest element to simulate reversing the operation. Using a max-heap allows efficient access to the largest element at each step. If the largest element becomes 1 or all elements are reduced to 1, the target is reachable. This approach is efficient because each operation significantly reduces the largest element, and we only maintain a heap of size `n`.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(n) | Simulates all operation sequences, too slow for large n |
| Optimal | O(n log n) | O(n) | Uses a max-heap and reverse operation, efficient for n ≤ 50,000 |

## Algorithm Walkthrough

1. Compute the total sum of the `target` array.
2. Push all elements of `target` into a max-heap, storing the largest element at the top for quick access.
3. Repeat until the largest element is 1:

1. Pop the largest element from the heap.
2. Compute the sum of the rest of the elements as `total_sum - largest`.
3. If the rest of the sum is 1, we can construct `target`, return `true`.
4. If `largest <= rest_sum` or `rest_sum` is 0, return `false` because it is impossible to reduce `largest` to 1.
5. Compute the previous value of the largest element using `largest % rest_sum`.
6. Push this previous value back into the heap and update `total_sum`.
4. If the loop terminates with all elements 1, return `true`.

Why it works: The reverse approach works because the operation of replacing an element with the sum of all elements can be inverted as subtracting the sum of the rest of the elements from the largest element. By repeatedly applying this reverse step, we can efficiently check if `target` can be reduced to `[1,1,...,1]`. The invariants are that the largest element is always valid relative to the sum of the rest and that we never create negative or zero elements.

## Python Solution

```python
from typing import List
import heapq

class Solution:
    def isPossible(self, target: List[int]) -> bool:
        if len(target) == 1:
            return target[0] == 1
        
        # Use a max-heap by pushing negative values
        max_heap = [-x for x in target]
        heapq.heapify(max_heap)
        total_sum = sum(target)
        
        while True:
            largest = -heapq.heappop(max_heap)
            rest_sum = total_sum - largest
            
            if largest == 1 or rest_sum == 1:
                return True
            if rest_sum == 0 or largest < rest_sum:
                return False
            
            new_val = largest % rest_sum
            if new_val == 0:
                return False
            
            heapq.heappush(max_heap, -new_val)
            total_sum = rest_sum + new_val
```

The Python implementation uses a max-heap simulated with a min-heap of negative numbers. We maintain `total_sum` to compute the sum of the rest efficiently and apply the modulo operation to find the previous value of the largest element. Edge cases, such as arrays of length 1 or when `rest_sum` is 0, are handled explicitly.

## Go Solution

```go
package main

import (
	"container/heap"
)

type IntHeap []int

func (h IntHeap) Len() int           { return len(h) }
func (h IntHeap) Less(i, j int) bool { return h[i] > h[j] } // max-heap
func (h IntHeap) Swap(i, j int)      { h[i], h[j] = h[j], h[i] }
func (h *IntHeap) Push(x any)        { *h = append(*h, x.(int)) }
func (h *IntHeap) Pop() any {
	old := *h
	n := len(old)
	x := old[n-1]
	*h = old[:n-1]
	return x
}

func isPossible(target []int) bool {
	if len(target) == 1 {
		return target[0] == 1
	}

	h := &IntHeap{}
	totalSum := 0
	for _, val := range target {
		heap.Push(h, val)
		totalSum += val
	}
	heap.Init(h)

	for {
		largest := heap.Pop(h).(int)
		restSum := totalSum - largest

		if largest == 1 || restSum == 1 {
			return true
		}
		if restSum == 0 || largest < restSum {
			return false
		}

		newVal := largest % restSum
		if newVal == 0 {
			return false
		}

		heap.Push(h, newVal)
		totalSum = restSum + newVal
	}
}
```

The Go version uses the `container/heap` interface to implement a max-heap and mirrors the Python logic. Care is taken to handle type assertions and manage `totalSum` consistently. The modulo operation ensures efficiency in reducing the largest element.

## Worked Examples

**Example 1: target = [9,3,5]**

| Step | Max Element | Rest Sum | New Value | Heap State |
| --- | --- | --- | --- | --- |
| Initial | 9 | 8 | 1 | [5,3,1] |
| Step 2 | 5 | 4 | 1 | [3,1,1] |
| Step 3 | 3 | 2 | 1 | [1,1,1] |
| Termination | All 1s | - | - | [1,1,1] |

Result: `True`

**Example 2: target = [1,1,1,2]**

| Step | Max Element | Rest Sum | New Value | Heap State |
| --- | --- | --- | --- | --- |
| Initial | 2 | 3 | 2%3=2 | - |
| Condition fails (largest < rest_sum) | - | - | - | - |

Result: `False`

**Example 3: target = [8,5]**

| Step | Max Element | Rest Sum | New Value | Heap State |
| --- | --- | --- | --- | --- |
| Initial | 8 | 5 | 8%5=3 | [5,3] |
| Next | 5 | 3 | 5%3=2 | [3,2] |
| Next | 3 | 2 | 3%2=1 | [2,1] |
| Next | 2 | 1 | 2%1=0 | [1,1] |

Result: `True`

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | Each heap operation is O(log n), and each element is reduced significantly each step |
| Space | O(n) | Heap stores all n elements |

The efficiency comes from using a max-heap for constant access to the largest element and the modulo operation to jump steps instead of decrementing one by one.
