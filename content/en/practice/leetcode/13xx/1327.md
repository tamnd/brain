## Solution

## Problem Understanding

The problem asks us to identify products that were ordered in **February 2020** and whose total order quantity reached at least 100 units. The input consists of two tables: `Products` and `Orders`. The `Products` table contains information about each product, including a unique `product_id`, the `product_name`, and `product_category`. The `Orders` table contains order history with `product_id`, `order_date`, and the quantity ordered (`unit`).

The output should be a table showing the `product_name` and the total `unit` quantity for products meeting the threshold of 100 units in February 2020. The order of results is not specified, so any ordering is acceptable. The problem is constrained in that all `product_id` values in `Orders` are guaranteed to exist in `Products`, and all dates are valid, which simplifies filtering.

Edge cases to consider include products with exactly 100 units, products with multiple orders on the same day, and products that have orders outside February 2020, which must be excluded from the sum.

## Approaches

The brute-force approach would involve iterating over all rows in `Orders`, filtering by date, aggregating sums for each product, and then joining with the `Products` table to retrieve names. While correct, this approach relies on multiple nested loops if implemented outside SQL and is inefficient for large datasets.

The optimal approach leverages SQL aggregation and filtering. First, filter the `Orders` table to include only dates in February 2020. Then, group by `product_id` and sum the `unit` column. Next, filter groups where the total units are at least 100. Finally, join the result with the `Products` table to retrieve the product names. This approach is efficient because SQL engines optimize filtering and grouping operations.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n*m) | O(n) | Iterate over all orders, sum manually, join with products |
| Optimal | O(n) | O(k) | Filter, group, sum using SQL aggregation functions, then join |

## Algorithm Walkthrough

1. Filter the `Orders` table for rows where `order_date` falls in February 2020. This ensures we only consider relevant orders.
2. Group the filtered orders by `product_id`. Aggregating at this stage allows us to calculate the total number of units per product efficiently.
3. Compute the sum of the `unit` column for each `product_id`. This gives the total units ordered in February 2020 for each product.
4. Apply a `HAVING` clause to retain only products with at least 100 units. This directly satisfies the problem constraint.
5. Join the aggregated results with the `Products` table on `product_id` to retrieve the corresponding `product_name`.
6. Select the `product_name` and aggregated `unit` count as the output. No particular order is required.

**Why it works:** Filtering before aggregation ensures that only relevant dates are counted. Grouping by `product_id` and summing guarantees an accurate total per product. The `HAVING` clause ensures the threshold condition is met, and the join guarantees we retrieve the product names correctly.

## Python Solution

Since this is a database problem, in Python we can emulate SQL behavior using `pandas` if necessary, but on LeetCode, we submit pure SQL. For reference, here is a Pythonic approach using SQL execution:

```python
# This is a Python solution for LeetCode's SQL interface
# LeetCode executes SQL directly, so in practice we submit the query as a string

sql_query = """
SELECT
    p.product_name,
    SUM(o.unit) AS unit
FROM Orders o
JOIN Products p ON o.product_id = p.product_id
WHERE o.order_date >= '2020-02-01' AND o.order_date <= '2020-02-29'
GROUP BY o.product_id, p.product_name
HAVING SUM(o.unit) >= 100
"""
```

**Explanation:** We first filter `Orders` by the February 2020 date range. Then we join with `Products` on `product_id`. The aggregation (`SUM`) is grouped by `product_id` and `product_name`. The `HAVING` clause filters only those groups with 100 or more units.

## Go Solution

For Go, if interacting with a database, we would execute the same SQL query. A Go-specific implementation might look like this:

```go
package main

import (
    "database/sql"
    _ "github.com/go-sql-driver/mysql"
    "fmt"
)

func ListProducts(db *sql.DB) ([]struct {
    ProductName string
    Unit        int
}, error) {
    query := `
        SELECT
            p.product_name,
            SUM(o.unit) AS unit
        FROM Orders o
        JOIN Products p ON o.product_id = p.product_id
        WHERE o.order_date >= '2020-02-01' AND o.order_date <= '2020-02-29'
        GROUP BY o.product_id, p.product_name
        HAVING SUM(o.unit) >= 100
    `
    rows, err := db.Query(query)
    if err != nil {
        return nil, err
    }
    defer rows.Close()

    var results []struct {
        ProductName string
        Unit        int
    }
    for rows.Next() {
        var r struct {
            ProductName string
            Unit        int
        }
        if err := rows.Scan(&r.ProductName, &r.Unit); err != nil {
            return nil, err
        }
        results = append(results, r)
    }
    return results, nil
}
```

**Go-specific notes:** The SQL query is identical. We use `rows.Scan` to populate a slice of structs. Go handles integer sums naturally, and no special handling for nil values is needed because SQL `SUM` never returns null when filtered.
