**Difficulty:** 🟡 Medium  
**Topics:** Tree, Depth-First Search, Breadth-First Search

## Solution

## Problem Understanding

This problem models a company hierarchy as a tree. Every employee has exactly one direct manager, except for the company head, whose manager is `-1`. Since the relationships form a valid tree structure, there are no cycles, and every employee is reachable from the head.

The goal is to determine how long it takes for information to spread from the head of the company to every employee.

Each employee requires a certain amount of time, stored in `informTime[i]`, to notify all of their direct subordinates. This timing is important because subordinates cannot begin informing their own subordinates until their manager has completely finished informing them.

The input consists of four values:

- `n`, the total number of employees
- `headID`, the ID of the company head
- `manager`, where `manager[i]` gives employee `i`'s direct manager
- `informTime`, where `informTime[i]` is the time employee `i` needs to inform all direct subordinates

The expected output is the total number of minutes required until every employee has received the news.

The constraints are large, up to `10^5` employees, which means the solution must be close to linear time. Any algorithm with quadratic complexity would be too slow. The tree guarantee is important because it means there is exactly one path from the head to every employee.

Several edge cases are important:

- A company with only one employee, where the answer is immediately `0`
- A very deep hierarchy, similar to a linked list
- A very wide hierarchy, where the head manages everyone directly
- Employees with `informTime[i] = 0`, which occurs for leaf employees
- Large trees where recursive depth could become an issue in some languages

The key observation is that the total time to reach an employee equals the sum of inform times along the path from the head to that employee. Therefore, the answer is the maximum such path sum.

## Approaches

### Brute Force Approach

A naive solution would compute the time required for every employee individually by repeatedly walking upward through the `manager` chain until reaching the head.

For each employee:

1. Start from the employee
2. Move upward through managers
3. Accumulate the inform times along the path
4. Track the maximum total

This works because every employee has exactly one unique path to the head. By summing the inform times on that path, we obtain the exact time when the employee receives the news.

However, this approach is inefficient. In the worst case, the hierarchy forms a chain of length `n`. Computing the time for one employee could take `O(n)` time, and doing this for all employees leads to `O(n^2)` complexity.

With `n = 10^5`, this is far too slow.

### Optimal Approach

The better solution treats the hierarchy as a tree and traverses it starting from the head.

The key insight is that information flows downward through the hierarchy. Instead of repeatedly walking upward for each employee, we can propagate the accumulated time downward exactly once.

We first build an adjacency list representing the subordinates of each manager. Then we perform either Depth-First Search (DFS) or Breadth-First Search (BFS).

During traversal:

- Keep track of the current accumulated time
- When visiting a subordinate, add the current manager's inform time
- Update the global maximum time seen so far

Since every employee and edge is processed exactly once, the solution runs in linear time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Repeatedly walks upward from every employee |
| Optimal | O(n) | O(n) | Builds the tree once and traverses it once |

## Algorithm Walkthrough

### Optimal DFS Algorithm

1. Build an adjacency list representing the company hierarchy.

The `manager` array tells us each employee's parent. For traversal, we want the opposite direction, manager to subordinates. We create a dictionary or list where `children[manager]` contains all direct subordinates.

This transformation allows efficient downward traversal from the head.
2. Initialize a variable to store the maximum time.

As we traverse the tree, we will compute the time required to reach each employee. The final answer is the maximum of these values.
3. Start DFS from the head employee.

The head initially receives the news at time `0`.
4. During DFS, propagate accumulated time.

If employee `u` receives the news at time `currentTime`, then subordinate `v` receives it at:

```
currentTime + informTime[u]
```

This is because employee `u` must spend `informTime[u]` minutes informing all direct subordinates before they can continue spreading the news.
5. Recursively process all subordinates.

Continue traversing until reaching leaf employees.
6. Update the maximum time encountered.

Whenever we reach an employee, compare their receive time against the current maximum.
7. Return the maximum time after traversal completes.

### Why it works

The company hierarchy is a tree, so there is exactly one path from the head to every employee. The DFS traversal computes the total communication delay along that unique path.

At every step, the algorithm correctly propagates the accumulated time from manager to subordinate. Since every employee is visited exactly once and the maximum arrival time is tracked globally, the final result equals the time required for the last employee to receive the news.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Solution:
    def numOfMinutes(
        self,
        n: int,
        headID: int,
        manager: List[int],
        informTime: List[int]
    ) -> int:

        children = defaultdict(list)

        # Build manager -> subordinate relationships
        for employee in range(n):
            if manager[employee] != -1:
                children[manager[employee]].append(employee)

        max_time = 0

        def dfs(employee: int, current_time: int) -> None:
            nonlocal max_time

            max_time = max(max_time, current_time)

            for subordinate in children[employee]:
                dfs(
                    subordinate,
                    current_time + informTime[employee]
                )

        dfs(headID, 0)

        return max_time
```

The implementation begins by constructing the adjacency list. The `manager` array naturally represents parent pointers, but DFS traversal requires quick access to children. The `children` dictionary stores all direct subordinates for each manager.

The `max_time` variable tracks the largest accumulated communication time encountered during traversal.

The recursive `dfs` function receives two values:

- `employee`, the current employee being processed
- `current_time`, the time when that employee receives the news

Inside DFS, we first update the global maximum. Then we recursively visit all subordinates.

When moving to a subordinate, we add `informTime[employee]` because the current employee must finish informing subordinates before they can continue spreading the news.

The traversal starts at `headID` with time `0`.

Since every employee is visited exactly once, the algorithm is efficient even for very large trees.

## Go Solution

```go
func numOfMinutes(n int, headID int, manager []int, informTime []int) int {
	children := make([][]int, n)

	// Build manager -> subordinate relationships
	for employee := 0; employee < n; employee++ {
		if manager[employee] != -1 {
			children[manager[employee]] = append(
				children[manager[employee]],
				employee,
			)
		}
	}

	maxTime := 0

	var dfs func(int, int)

	dfs = func(employee int, currentTime int) {
		if currentTime > maxTime {
			maxTime = currentTime
		}

		for _, subordinate := range children[employee] {
			dfs(
				subordinate,
				currentTime+informTime[employee],
			)
		}
	}

	dfs(headID, 0)

	return maxTime
}
```

The Go implementation follows the same logic as the Python version. Instead of a dictionary, it uses a slice of slices for the adjacency list because the employee IDs are contiguous from `0` to `n - 1`.

Go does not support nested named functions in the same way as Python, so the DFS function is declared as a function variable before assignment.

All values fit safely inside Go's `int` type because the maximum possible accumulated time is well below typical integer limits.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every employee and edge is processed exactly once |
| Space | O(n) | Adjacency list and recursion stack may store up to n employees |

The adjacency list construction takes linear time because we iterate through the employee list once. The DFS traversal also visits every employee exactly once.

The space complexity comes from storing the tree structure and the recursion stack. In the worst case, the hierarchy could form a chain of length `n`, producing a recursion depth of `O(n)`.

## Edge Cases

### Single Employee Company

When `n = 1`, the head is the only employee and already knows the news. A buggy implementation might incorrectly add the head's inform time or assume at least one subordinate exists.

This implementation handles the case naturally because the adjacency list for the head is empty, so DFS immediately terminates and returns `0`.

### Deep Hierarchy Chain

A hierarchy like:

```
0 -> 1 -> 2 -> 3 -> 4
```

creates the worst possible depth. A naive repeated upward traversal becomes quadratic in this scenario.

The DFS solution processes each employee once, so the runtime remains linear. The accumulated time correctly becomes the sum of all inform times along the chain.

### Employees with Zero Inform Time

Leaf employees always have `informTime[i] = 0`, but internal employees may also have zero delay in some custom cases.

A common bug is adding the subordinate's inform time instead of the manager's inform time when propagating time downward.

This implementation correctly uses:

```
current_time + informTime[employee]
```

because the manager determines how long subordinates must wait before they can continue spreading the news.

### Very Wide Hierarchy

If the head directly manages nearly all employees, the tree becomes extremely wide.

Some implementations might inefficiently recompute times for each subordinate independently. This solution avoids redundant work because each employee is visited exactly once during DFS traversal.
