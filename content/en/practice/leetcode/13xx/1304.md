## Solution

## Problem Understanding

The problem asks us to construct an array of exactly `n` integers such that two conditions are satisfied:

1. Every integer in the array must be unique.
2. The sum of all integers in the array must equal `0`.

The input is a single integer `n`, where `1 <= n <= 1000`. The output can be any valid array that satisfies the requirements. This means there is no single correct answer. Multiple valid arrays may exist for the same input.

For example, when `n = 3`, one valid answer is `[-1, 0, 1]` because all numbers are distinct and their sum is:

```
-1 + 0 + 1 = 0
```

The important detail is that uniqueness matters. Returning `[0, 0, 0]` would sum to zero, but it would violate the uniqueness requirement.

The constraints are small, since `n` is at most `1000`. This tells us performance is not a major concern, but we should still aim for a clean and efficient construction. The problem is fundamentally mathematical rather than algorithmically complex.

There are several important edge cases to consider:

- When `n = 1`, the only valid answer is `[0]`.
- When `n` is even, we can pair positive and negative integers cleanly.
- When `n` is odd, we need one extra number that does not disturb the sum. Using `0` works perfectly because it contributes nothing to the total.
- We must ensure all numbers remain unique. Accidentally repeating values would invalidate the solution.

The problem guarantees that `n` is always positive, so we never need to handle `n = 0` or negative inputs.

## Approaches

### Brute Force Approach

A brute-force solution would try generating integers and checking whether a subset of size `n` satisfies both conditions:

- All numbers are unique.
- Their sum is zero.

For example, we could repeatedly generate candidate arrays and verify their validity. Another brute-force idea would be to recursively build arrays while tracking the running sum and uniqueness.

This approach is correct because eventually it would find a valid combination. However, it is unnecessarily expensive. The search space grows rapidly because there are many possible integer combinations.

Even though the constraints are small, brute force ignores the mathematical structure of the problem. Since the problem only asks for any valid construction, exhaustive searching is wasteful.

### Optimal Approach

The key observation is that integers can naturally cancel each other out using positive-negative pairs.

For every positive integer `x`, adding `-x` ensures:

```
x + (-x) = 0
```

This means we can construct the array using symmetric pairs:

```
1, -1
2, -2
3, -3
```

If `n` is even, we can fill the array entirely with these pairs.

If `n` is odd, we can include `0` once, because:

```
sum + 0 = sum
```

This gives us a direct mathematical construction with no searching required.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | O(n) | Tries many possible combinations until a valid one is found |
| Optimal | O(n) | O(n) | Uses positive-negative pairs that automatically cancel out |

## Algorithm Walkthrough

### Optimal Algorithm

1. Create an empty result array.

We need a container to store the constructed integers.
2. If `n` is odd, add `0` to the array.

This is done because `0` does not affect the total sum. It also helps us handle the extra element when `n` cannot be completely divided into pairs.
3. Iterate from `1` to `n // 2`.

Each iteration creates one positive-negative pair.
4. For each integer `i`, append both `i` and `-i`.

These two numbers are unique and cancel each other out:

```
i + (-i) = 0
```
5. Return the completed array.

The array now contains exactly `n` unique integers whose total sum is zero.

### Why it works

The correctness relies on the cancellation property of additive inverses.

Every pair `(i, -i)` contributes zero to the overall sum. If `n` is odd, the extra `0` also contributes nothing. Since every generated value is distinct, uniqueness is preserved.

Therefore:

- The array length is exactly `n`.
- All integers are unique.
- The total sum is zero.

This guarantees the solution is always valid.

## Python Solution

```python
from typing import List

class Solution:
    def sumZero(self, n: int) -> List[int]:
        result = []

        if n % 2 == 1:
            result.append(0)

        for value in range(1, n // 2 + 1):
            result.append(value)
            result.append(-value)

        return result
```

The implementation starts by creating an empty list called `result`.

The first conditional checks whether `n` is odd. If it is, the algorithm inserts `0` into the result list. This ensures we can still build the remainder of the array using symmetric pairs.

The loop runs from `1` through `n // 2`. For each number, the algorithm appends both the positive and negative versions of that value. Since these numbers cancel each other out, they preserve the total sum of zero.

Finally, the completed list is returned.

The implementation directly mirrors the mathematical construction described earlier, which keeps the code concise and easy to reason about.

## Go Solution

```go
func sumZero(n int) []int {
    result := []int{}

    if n%2 == 1 {
        result = append(result, 0)
    }

    for value := 1; value <= n/2; value++ {
        result = append(result, value)
        result = append(result, -value)
    }

    return result
}
```

The Go implementation follows the same logic as the Python solution.

One small language-specific detail is slice handling. In Go, we initialize an empty slice using:

```go
result := []int{}
```

Then we use `append` to dynamically grow the slice.

There are no integer overflow concerns because the generated values are extremely small compared to Go's integer limits. The maximum generated value is at most `500`.

## Edge Cases

### Edge Case 1: `n = 1`

This is the smallest possible input and can easily expose incorrect assumptions. A naive implementation that only creates positive-negative pairs would fail because there is no room for a pair.

The implementation handles this correctly by detecting odd values of `n` and inserting `0`. The result becomes:

```
[0]
```

which satisfies both uniqueness and zero-sum requirements.

### Edge Case 2: Even Values of `n`

When `n` is even, no extra neutral element is needed. A common bug is accidentally inserting `0` anyway, which could produce too many elements.

The implementation avoids this by only adding `0` when:

```
n % 2 == 1
```

For even inputs, the result consists entirely of balanced positive-negative pairs.

### Edge Case 3: Large Inputs

For large values such as `n = 1000`, inefficient solutions could become unnecessarily slow or memory-heavy.

The implementation scales efficiently because it generates values directly in linear time without searching or backtracking. Only one pass is needed to construct the array.

### Edge Case 4: Duplicate Values

A careless implementation might accidentally reuse numbers, especially when constructing pairs manually.

This solution guarantees uniqueness because every generated positive integer appears exactly once, and its negative counterpart also appears exactly once. Zero, when included, appears only once as well.
