## Solution

## Problem Understanding

The problem gives us an `m x n` matrix containing distinct integers. We must find every value that satisfies two conditions simultaneously:

1. It is the minimum value in its row.
2. It is the maximum value in its column.

Such a value is called a lucky number.

The input is a two dimensional matrix where:

- `m` is the number of rows
- `n` is the number of columns
- every element is unique

The output is a list containing all lucky numbers in any order.

To understand the requirement clearly, consider a value located at position `(row, col)`:

- We first check whether this value is the smallest element among all elements in `matrix[row]`.
- Then we check whether this same value is the largest element among all elements in column `col`.

Only if both conditions are true do we include it in the result.

The constraints are small:

- `1 <= m, n <= 50`
- matrix values are distinct
- values are positive integers

Since the matrix dimensions are at most `50 x 50`, even quadratic or cubic solutions are acceptable. However, we should still aim for a clean and efficient solution.

The fact that all numbers are distinct is important because it removes ambiguity. There cannot be two equal minimums in a row or two equal maximums in a column. This simplifies the logic considerably.

Some important edge cases include:

- A matrix with only one element, that element is automatically both the row minimum and column maximum.
- A single row matrix, the lucky number must be the smallest value in the row because every value is trivially the maximum in its own column only if no larger value exists.
- A single column matrix, the lucky number must be the largest value in the column.
- Matrices where no lucky number exists, although the problem setup often leads to at least one valid answer.

## Approaches

### Brute Force Approach

The most direct solution is to examine every element in the matrix and independently verify whether it is:

- the minimum value in its row
- the maximum value in its column

For every cell `(i, j)`:

- Scan the entire row `i` to check whether `matrix[i][j]` is the row minimum.
- Scan the entire column `j` to check whether `matrix[i][j]` is the column maximum.

If both conditions hold, add the value to the result.

This works because it explicitly verifies the exact definition of a lucky number. However, it repeatedly scans rows and columns for every element, which creates unnecessary repeated work.

### Optimal Approach

The key observation is that:

- every lucky number must be a row minimum
- every lucky number must also be a column maximum

Instead of checking every element fully, we can precompute:

- the minimum value of each row
- the maximum value of each column

Then we simply look for values that appear in both sets.

More specifically:

- Compute all row minimums.
- Compute all column maximums.
- Any value present in both collections is a lucky number.

This reduces redundant scanning and makes the implementation cleaner.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m × n × (m + n)) | O(1) | Checks every element by rescanning its row and column |
| Optimal | O(m × n) | O(m + n) | Precomputes row minimums and column maximums |

## Algorithm Walkthrough

### Optimal Algorithm

1. Traverse each row and compute its minimum value.

We store these row minimums because a lucky number must be the smallest value in its row. Any element that is not a row minimum can immediately be ignored.
2. Traverse each column and compute its maximum value.

We store these column maximums because a lucky number must also be the largest value in its column.
3. Convert the column maximums into a set.

A set allows constant time membership checks. We want to quickly determine whether a row minimum is also a column maximum.
4. Iterate through all row minimums.

For each row minimum:

- Check whether it exists in the column maximum set.
- If it does, append it to the answer list.
5. Return the result list.

### Why it works

A lucky number must satisfy two independent properties:

- minimum in its row
- maximum in its column

The algorithm explicitly computes all candidates for each property. The intersection of row minimums and column maximums therefore contains exactly the lucky numbers. Since every lucky number must belong to both groups, and every value in both groups satisfies the definition, the algorithm is correct.

## Python Solution

```python
from typing import List

class Solution:
    def luckyNumbers(self, matrix: List[List[int]]) -> List[int]:
        rows = len(matrix)
        cols = len(matrix[0])

        # Find minimum value from each row
        row_mins = [min(row) for row in matrix]

        # Find maximum value from each column
        col_maxs = []

        for col in range(cols):
            max_value = matrix[0][col]

            for row in range(1, rows):
                max_value = max(max_value, matrix[row][col])

            col_maxs.append(max_value)

        col_max_set = set(col_maxs)

        # Lucky numbers are values appearing in both collections
        result = []

        for value in row_mins:
            if value in col_max_set:
                result.append(value)

        return result
```

The implementation begins by determining the matrix dimensions. It then computes the minimum value for every row using Python's built in `min()` function.

Next, it computes the maximum value for every column. Since Python matrices are stored row wise, we manually iterate through each row for a given column index.

The column maximums are stored in a set because set lookup is efficient. Finally, we iterate through all row minimums and collect the values that also appear among the column maximums.

This directly mirrors the algorithm described earlier and avoids repeatedly rescanning rows and columns.

## Go Solution

```go
func luckyNumbers(matrix [][]int) []int {
    rows := len(matrix)
    cols := len(matrix[0])

    // Find minimum value from each row
    rowMins := make([]int, rows)

    for i := 0; i < rows; i++ {
        minValue := matrix[i][0]

        for j := 1; j < cols; j++ {
            if matrix[i][j] < minValue {
                minValue = matrix[i][j]
            }
        }

        rowMins[i] = minValue
    }

    // Find maximum value from each column
    colMaxSet := make(map[int]bool)

    for j := 0; j < cols; j++ {
        maxValue := matrix[0][j]

        for i := 1; i < rows; i++ {
            if matrix[i][j] > maxValue {
                maxValue = matrix[i][j]
            }
        }

        colMaxSet[maxValue] = true
    }

    // Find intersection
    result := []int{}

    for _, value := range rowMins {
        if colMaxSet[value] {
            result = append(result, value)
        }
    }

    return result
}
```

The Go implementation follows the same logic as the Python version, but uses slices and maps explicitly.

Instead of Python sets, Go uses a `map[int]bool` to simulate set behavior. Row minimums and column maximums are computed with nested loops because Go does not provide built in `min()` and `max()` functions for slices.

There are no integer overflow concerns because matrix values are at most `10^5`, which easily fits inside Go's `int` type.

The algorithm performs two full passes over the matrix:

- one for row minimums
- one for column maximums

Since each pass touches every element once, the total runtime is linear in the number of cells.

The additional memory comes from storing:

- one minimum per row
- one maximum per column
- the set for quick lookups

## Edge Cases

### Single Element Matrix

A `1 x 1` matrix is the smallest possible input. The only value is automatically both:

- the minimum in its row
- the maximum in its column

A buggy implementation might accidentally skip processing because loops assume multiple rows or columns. This solution handles the case naturally because the row minimum and column maximum computations still work correctly for size one.

### Single Row Matrix

In a single row matrix, the row minimum is easy to identify, but column checks become subtle because each column contains only one element.

For example:

```
[[1, 2, 3]]
```

The lucky number is `1` because:

- it is the row minimum
- it is also trivially the maximum in its own column

The implementation correctly computes column maximums even when each column has only one value.

### No Lucky Number Exists

Some matrices contain no value satisfying both conditions.

For example:

```
[
  [20, 15, 10],
  [18, 8, 7],
  [30, 16, 17]
]
```

A naive implementation might incorrectly assume that at least one lucky number always exists. This solution does not make that assumption. If no row minimum appears among the column maximums, the result remains an empty list.

### Distinct Values Guarantee

The problem guarantees all elements are distinct. This prevents ties for row minimums or column maximums.

Without this guarantee, special handling might be needed for duplicate values. Since the input contains unique numbers, our implementation can safely treat row minimums and column maximums as unique candidates without ambiguity.
