## Solution

## Problem Understanding

The problem asks us to compute the smaller angle formed between the hour hand and the minute hand on an analog clock.

We are given two integers:

- `hour`, representing the current hour on a 12 hour clock
- `minutes`, representing the current minute

The goal is to return the smaller angle between the two clock hands in degrees.

A key detail is that the hour hand does not stay fixed while the minutes advance. For example, at `3:30`, the hour hand is not exactly at the `3`, it has already moved halfway toward the `4`. Many incorrect solutions forget this continuous movement.

The clock is circular, so two angles are always formed between the hands. For example, if one angle is `210` degrees, the other angle is `150` degrees. The problem specifically asks for the smaller one.

The constraints are very small:

- `1 <= hour <= 12`
- `0 <= minutes <= 59`

Since the input size is constant and tiny, performance is not a concern. The challenge is purely mathematical correctness.

Several edge cases are important:

- `12:00` should return `0`, because both hands overlap.
- Times like `6:00` produce exactly `180` degrees.
- The hour hand moves gradually, so `3:15` is not `0` or `90`, it is `7.5`.
- When the direct angle exceeds `180`, we must take the complementary angle instead.

## Approaches

### Brute Force Approach

A brute force style approach would simulate the positions of the clock hands minute by minute or even degree by degree until the current time is reached.

For example, we could:

1. Start both hands at `12:00`
2. Advance the minute hand step by step
3. Move the hour hand proportionally
4. Compute the resulting angle

This works because clocks move predictably and continuously. Eventually, the simulated state matches the requested time.

However, this approach is unnecessarily complicated. The position of each hand can be computed directly using simple arithmetic formulas. Simulation introduces extra computation and implementation complexity without any benefit.

### Optimal Mathematical Approach

The optimal solution relies on directly computing the angular position of each hand.

The minute hand completes a full `360` degree rotation every `60` minutes.

So the minute hand angle is:

$\text{Minute Angle} = 6 \times \text{minutes}$

The hour hand completes a full `360` degree rotation every `12` hours.

That means each hour mark represents:

$\frac{360}{12} = 30^\circ$

But the hour hand also moves continuously as minutes pass. Each minute moves the hour hand by:

$\frac{30}{60} = 0.5^\circ$

Therefore:

$\text{Hour Angle} = 30 \times \text{hour} + 0.5 \times \text{minutes}$

Once we know both angles, we compute their absolute difference.

Finally, because a circle has two possible angles between any two rays, we return:

$\min(\theta, 360 - \theta)$

where `θ` is the absolute difference.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(60) | O(1) | Simulates hand movement minute by minute |
| Optimal | O(1) | O(1) | Uses direct mathematical formulas |

## Algorithm Walkthrough

1. Compute the angle of the minute hand.

Every minute corresponds to `6` degrees because the minute hand rotates `360` degrees over `60` minutes.

So:

`minute_angle = minutes * 6`
2. Compute the angle of the hour hand.

Every hour corresponds to `30` degrees because the hour hand rotates `360` degrees over `12` hours.

However, the hour hand also moves gradually as minutes pass.

Each minute contributes `0.5` degrees.

So:

`hour_angle = hour * 30 + minutes * 0.5`

One subtle detail is that `12` behaves like `0` on a circular clock. Multiplying by `30` still works correctly because `12 * 30 = 360`, which is equivalent to `0` degrees.
3. Compute the absolute difference between the two angles.

`difference = abs(hour_angle - minute_angle)`
4. Return the smaller angle.

Since the clock is circular, the second angle is:

`360 - difference`

The answer is:

`min(difference, 360 - difference)`

### Why it works

The algorithm works because each clock hand moves at a constant angular velocity. The formulas compute the exact angular position of each hand at the given time. The absolute difference gives one of the two possible angles between them, and taking the minimum ensures we return the smaller angle required by the problem.

## Python Solution

```python
class Solution:
    def angleClock(self, hour: int, minutes: int) -> float:
        minute_angle = minutes * 6
        hour_angle = (hour % 12) * 30 + minutes * 0.5

        difference = abs(hour_angle - minute_angle)

        return min(difference, 360 - difference)
```

The implementation directly follows the mathematical derivation.

First, the minute hand angle is computed using `minutes * 6`. Since each minute corresponds to `6` degrees, this gives the exact position of the minute hand.

Next, the hour hand angle is calculated. We use `(hour % 12)` so that `12` maps to `0`, which matches the geometry of an analog clock. The term `minutes * 0.5` accounts for the gradual movement of the hour hand between hour marks.

The absolute difference between the two angles is then computed. Finally, we compare that difference with its complementary angle `360 - difference` and return the smaller value.

The implementation uses only constant extra space and a fixed number of arithmetic operations.

## Go Solution

```go
func angleClock(hour int, minutes int) float64 {
    minuteAngle := float64(minutes * 6)
    hourAngle := float64((hour%12)*30) + float64(minutes)*0.5

    difference := hourAngle - minuteAngle
    if difference < 0 {
        difference = -difference
    }

    if difference > 180 {
        return 360 - difference
    }

    return difference
}
```

The Go solution follows the same logic as the Python version. Since Go distinguishes between integers and floating point values, explicit `float64` conversions are required.

Go does not provide a built in absolute value function for floating point numbers without importing the `math` package, so the implementation manually converts negative differences into positive values.

## Edge Cases

One important edge case is when both hands overlap exactly, such as `12:00`. A careless implementation might incorrectly treat `12` as `360` degrees instead of `0`, producing the wrong difference. Using `(hour % 12)` ensures that `12` maps correctly to the top of the clock.

Another critical case occurs when the direct angle exceeds `180` degrees. For example, if the raw difference is `250`, the smaller angle is actually `110`. Forgetting to compute `min(difference, 360 - difference)` would return the larger angle instead of the required smaller one.

A subtle edge case involves fractional hour hand movement. At `3:15`, the hour hand is not fixed at `90` degrees. It has moved an additional `7.5` degrees toward `4`. Naive solutions that only compute `hour * 30` would incorrectly return `0` instead of `7.5`. The `minutes * 0.5` term correctly models continuous movement.

Another tricky case is near wraparound times such as `11:59`. The hour hand is almost at `12`, and the minute hand is also almost at `12`. The actual angle is very small. The implementation handles this naturally because it computes exact angular positions and always selects the smaller circular difference.
