## Solution

## Problem Understanding

The problem asks us to sort an array of integers according to a custom rule based on the binary representation of each number.

Every integer can be represented in binary form. For example:

- `5` in binary is `101`
- `7` in binary is `111`
- `8` in binary is `1000`

The task is to count how many `1` bits appear in each binary representation, then sort the numbers in ascending order by that count.

If two numbers contain the same number of `1` bits, then they must be sorted in normal numerical ascending order.

For example:

- `2` is `10`, which contains one `1`
- `4` is `100`, which also contains one `1`
- `8` is `1000`, which also contains one `1`

Since all three numbers have the same bit count, they are ordered numerically as:

`[2, 4, 8]`

The input is an integer array `arr`, and the output must be the same array sorted according to these rules.

The constraints are small:

- The array length is at most `500`
- Each integer is at most `10^4`

These limits tell us that we do not need extremely advanced optimizations. Even an `O(n log n)` sorting solution will easily fit within the limits.

An important observation is that the comparison rule is deterministic and stable:

1. First compare the number of set bits
2. If tied, compare the actual integer values

Some edge cases that could cause bugs include:

- Arrays containing `0`, since `0` has zero set bits
- Arrays where all numbers have the same number of set bits
- Arrays already sorted numerically but not by bit count
- Duplicate values
- Large values near `10^4`

The problem guarantees valid non-negative integers, so we never need to handle negative binary representations.

## Approaches

### Brute Force Approach

A brute-force solution would manually compare every pair of numbers during sorting.

For each comparison:

1. Convert both integers into binary
2. Count the number of `1` bits
3. Compare their bit counts
4. If the counts are equal, compare the integers themselves

This approach works because it directly follows the problem definition.

However, repeatedly recomputing the number of set bits during every comparison is inefficient. Sorting algorithms perform many comparisons, and recalculating binary counts each time creates unnecessary overhead.

Although the constraints are small enough that this could still pass, it is not optimal design.

### Optimal Approach

The key insight is that each number's bit count never changes.

Instead of recomputing the number of `1` bits repeatedly, we can compute it once and use it as a sorting key.

Modern sorting functions allow sorting using a custom key. We can sort each number using a tuple:

`(number_of_set_bits, actual_number)`

This automatically enforces both rules:

- Primary sort by bit count
- Secondary sort by integer value

For example:

| Number | Bit Count | Sort Key |
| --- | --- | --- |
| 3 | 2 | (2, 3) |
| 8 | 1 | (1, 8) |
| 5 | 2 | (2, 5) |

Sorting by these tuples produces the required ordering.

## Comparison of Approaches

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n * k) | O(1) | Recomputes bit counts during comparisons |
| Optimal | O(n log n) | O(n) | Uses sorting keys with precomputed bit counts |

Here, `k` represents the number of bits required to represent a number, which is small for this problem.

## Algorithm Walkthrough

1. Iterate through each integer in the array.

For every number, determine how many `1` bits exist in its binary representation. This value represents the primary sorting priority.
2. Create a sorting key for every number.

The key is a pair:

`(bit_count, number)`

The first element ensures sorting by the number of set bits. The second element guarantees ascending numerical order when bit counts are equal.
3. Use the built-in sorting function.

Most modern languages provide highly optimized sorting implementations. By supplying the custom key, the sorting algorithm automatically applies the required ordering rules.
4. Return the sorted array.

After sorting completes, the array already satisfies both constraints from the problem statement.

### Why it works

The algorithm works because tuple comparison naturally matches the problem requirements.

When two elements are compared:

- Their bit counts are compared first
- If the counts differ, the smaller count comes first
- If the counts are equal, the integers themselves are compared

This exactly matches the required sorting behavior, so the resulting array is guaranteed to be correct.

## Python Solution

```python
from typing import List

class Solution:
    def sortByBits(self, arr: List[int]) -> List[int]:
        def count_bits(num: int) -> int:
            return bin(num).count("1")

        return sorted(arr, key=lambda num: (count_bits(num), num))
```

The implementation begins by defining a helper function called `count_bits`.

The expression `bin(num)` converts the integer into a binary string. For example:

- `5` becomes `"0b101"`
- `7` becomes `"0b111"`

The `.count("1")` operation counts how many set bits exist.

The `sorted()` function is then used with a custom key.

For every number, the key is:

```python
(count_bits(num), num)
```

Python sorts tuples lexicographically:

- First by the bit count
- Then by the integer value

This perfectly matches the required ordering.

The solution is concise because Python's sorting system handles the comparison logic automatically.

## Go Solution

```go
package main

import (
	"math/bits"
	"sort"
)

func sortByBits(arr []int) []int {
	sort.Slice(arr, func(i, j int) bool {
		bitCountI := bits.OnesCount(uint(arr[i]))
		bitCountJ := bits.OnesCount(uint(arr[j]))

		if bitCountI == bitCountJ {
			return arr[i] < arr[j]
		}

		return bitCountI < bitCountJ
	})

	return arr
}
```

The Go implementation uses `sort.Slice`, which allows custom comparison logic.

Unlike Python, Go does not provide tuple-based sorting keys directly, so the comparison function explicitly handles both rules:

1. Compare the number of set bits
2. If equal, compare the actual integers

The `math/bits` package provides `bits.OnesCount`, which efficiently counts set bits using optimized bit operations.

The function sorts the slice in place and returns it afterward.

Since all values are non-negative and small, integer overflow is not a concern here.

## Edge Cases

One important edge case is arrays containing `0`. The binary representation of `0` contains no set bits, so its bit count is `0`. Some implementations accidentally treat `"0"` as containing one bit because the digit exists in the string. Using `count("1")` correctly returns zero, so the implementation handles this safely.

Another important case occurs when all numbers have the same number of set bits. In that situation, the algorithm must fall back to normal numerical ordering. For example, powers of two all contain exactly one set bit. The tuple-based sorting key ensures that when bit counts tie, the integers themselves determine the order.

Duplicate values are also important. Sorting algorithms sometimes behave unexpectedly when duplicates appear alongside custom comparison logic. Since this implementation uses deterministic tuple keys, duplicate integers naturally remain grouped correctly and maintain valid ordering.

A final edge case involves already sorted arrays or reverse sorted arrays. Some buggy implementations incorrectly assume partial ordering properties and fail to reorder elements properly. Because this solution relies entirely on a full sort using explicit comparison keys, it correctly handles every possible initial arrangement of the input array.
