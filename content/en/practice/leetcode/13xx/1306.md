## Solution

## Problem Understanding

The problem gives us an array of non-negative integers called `arr` and a starting index called `start`. From any index `i`, we are allowed to jump either forward to `i + arr[i]` or backward to `i - arr[i]`. The goal is to determine whether it is possible to reach any index whose value is `0`.

The important detail is that the jump distance is determined by the value stored at the current index. This means the array effectively represents a directed graph where each index can have up to two outgoing edges. We are not allowed to jump outside the bounds of the array, so any move that produces an invalid index must be ignored.

The input consists of:

- `arr`, an array of integers where each value represents the jump length from that position
- `start`, the index where traversal begins

The expected output is a boolean:

- `true` if some sequence of valid jumps reaches an index containing `0`
- `false` otherwise

The constraints are important:

- The array length can be as large as `5 * 10^4`
- Each value in the array is between `0` and `arr.length - 1`

These limits immediately tell us that exponential exploration is not acceptable. Since the array can contain cycles, a naive recursive search may revisit the same indices repeatedly and potentially never terminate. Efficient cycle detection is therefore necessary.

Several edge cases are important:

- The starting index itself may already contain `0`
- The array may contain cycles that never reach a `0`
- Both possible jumps from an index may go out of bounds
- Multiple paths may revisit the same index
- Arrays of length `1` must be handled correctly

Because the problem only asks whether reaching a zero is possible, we do not need to compute the shortest path or reconstruct the route. We only need reachability.

## Approaches

### Brute Force Approach

A brute-force solution would recursively try every possible jump from the current index. From index `i`, we attempt:

- `i + arr[i]`
- `i - arr[i]`

If either path eventually reaches a `0`, we return `true`.

This approach is conceptually simple because it directly models the problem statement. However, it suffers from a major issue: cycles. Consider indices that continuously jump back and forth between each other. Without tracking visited indices, the recursion can loop forever.

Even if we attempt to bound the recursion depth, the same states may be recomputed many times, leading to exponential behavior in the worst case.

### Optimal Approach

The key insight is that each array index represents a node in a graph, and each node has at most two neighbors. The problem is therefore a graph traversal problem.

Once we recognize this, standard graph traversal techniques such as Depth-First Search (DFS) or Breadth-First Search (BFS) become natural solutions.

The most important optimization is maintaining a `visited` set or array. Once an index has already been explored, revisiting it cannot produce any new information. Skipping revisits prevents infinite loops and guarantees that each index is processed at most once.

Because every index is explored at most one time, the algorithm becomes linear in the size of the array.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(n) | Recursively explores all possible jump paths without pruning repeated states |
| Optimal | O(n) | O(n) | Uses DFS or BFS with visited tracking so each index is processed once |

## Algorithm Walkthrough

We will use Depth-First Search with a visited array.

1. Create a `visited` array of the same size as `arr`, initialized to `False`. This structure prevents revisiting indices and avoids infinite loops caused by cycles.
2. Start DFS from the given `start` index. The recursive function represents the question: "Can we reach a zero starting from this index?"
3. Inside the DFS function, first check whether the current index is out of bounds. If it is, return `False` because invalid positions cannot contribute to a valid solution.
4. Next, check whether this index has already been visited. If it has, return `False`. Revisiting the same index would only repeat work already done.
5. If `arr[index] == 0`, return `True` immediately because we have successfully reached a target index.
6. Mark the current index as visited before exploring neighbors. This is critical because it prevents cycles during recursive exploration.
7. Compute the two possible next positions:

- `index + arr[index]`
- `index - arr[index]`
8. Recursively explore both positions. If either recursive call returns `True`, propagate that result upward.
9. If neither path reaches a zero, return `False`.

### Why it works

The algorithm works because it systematically explores every reachable index exactly once. Since each valid move corresponds to an edge in the implicit graph, DFS guarantees that every reachable node is examined. The visited array ensures termination even in the presence of cycles. Therefore, if a zero-valued index is reachable from `start`, DFS will eventually discover it.

## Python Solution

```python
from typing import List

class Solution:
    def canReach(self, arr: List[int], start: int) -> bool:
        n = len(arr)
        visited = [False] * n

        def dfs(index: int) -> bool:
            if index < 0 or index >= n:
                return False

            if visited[index]:
                return False

            if arr[index] == 0:
                return True

            visited[index] = True

            jump = arr[index]

            return dfs(index + jump) or dfs(index - jump)

        return dfs(start)
```

The implementation begins by creating a `visited` array to track explored indices. This prevents revisiting nodes and guarantees that recursion terminates even when cycles exist.

The nested `dfs` function handles the recursive traversal. The first condition checks bounds because jumps may produce invalid indices. The second condition checks whether the node has already been explored.

The next condition checks whether the current value is zero. Since the problem only asks whether a zero is reachable, we can immediately return `True` once such an index is found.

After marking the current node as visited, we compute the jump distance and recursively explore both possible destinations. The logical `or` operator is useful because it short-circuits. As soon as one path succeeds, the search stops.

Finally, the algorithm starts the DFS from the `start` index.

## Go Solution

```go
func canReach(arr []int, start int) bool {
    n := len(arr)
    visited := make([]bool, n)

    var dfs func(int) bool

    dfs = func(index int) bool {
        if index < 0 || index >= n {
            return false
        }

        if visited[index] {
            return false
        }

        if arr[index] == 0 {
            return true
        }

        visited[index] = true

        jump := arr[index]

        return dfs(index+jump) || dfs(index-jump)
    }

    return dfs(start)
}
```

The Go implementation closely mirrors the Python version. A boolean slice is used for visited tracking. The recursive DFS function is declared as a variable before assignment because Go requires recursive anonymous functions to reference an already-declared variable.

Go slices naturally handle dynamic array behavior, so no special handling is needed beyond bounds checking. Integer overflow is not a concern here because the constraints guarantee all indices remain within manageable ranges.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each index is visited at most once |
| Space | O(n) | The visited array and recursion stack may each grow to size n |

The time complexity is linear because every index enters the DFS at most one time. Once visited, it is never processed again. Since each index performs only constant work, the total running time is proportional to the array length.

The space complexity is also linear. The visited array requires `O(n)` memory, and in the worst case the recursion depth may also reach `n`.

## Edge Cases

One important edge case occurs when the starting index already contains `0`. A careless implementation might immediately begin exploring neighbors without checking the current node first. In that situation, it would incorrectly miss the valid answer. The implementation handles this by checking `arr[index] == 0` before performing any recursive exploration.

Another critical edge case involves cycles. For example, an array may allow jumping repeatedly between the same two indices. Without visited tracking, recursion would continue indefinitely and eventually overflow the call stack. The solution prevents this by marking indices as visited before exploring neighbors.

A third important edge case occurs when jumps go out of bounds. Since jumps can move both forward and backward, many transitions may produce negative indices or indices beyond the array length. Attempting to access such positions would cause runtime errors. The implementation safely handles this with explicit bounds checks at the beginning of every DFS call.

A final subtle edge case involves arrays of length one. If the only value is `0`, the answer should be `true`. If the only value is nonzero, both possible jumps are invalid. The implementation naturally handles both scenarios because the base cases cover zero detection and bounds validation correctly.
