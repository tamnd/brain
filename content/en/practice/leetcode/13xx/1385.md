# LeetCode 1385 - Find the Distance Value Between Two Arrays

## Problem Understanding

The problem gives us two integer arrays, `arr1` and `arr2`, along with an integer `d`. We must compute how many elements in `arr1` are "far enough" from every element in `arr2`.

More specifically, for each value `arr1[i]`, we check every value `arr2[j]`. If there exists even one element in `arr2` such that:

$$|arr1[i] - arr2[j]| \le d$$

then `arr1[i]` does not contribute to the answer.

If no such element exists, then that element from `arr1` is counted toward the distance value.

In simpler terms, we are counting how many numbers in `arr1` stay more than distance `d` away from all numbers in `arr2`.

The input sizes are relatively small:

- `arr1.length <= 500`
- `arr2.length <= 500`

This means even a quadratic solution is acceptable because at worst we perform:

$$500 \times 500 = 250000$$

comparisons, which is easily manageable.

The values themselves range from `-1000` to `1000`, which means negative numbers are possible and absolute differences must be handled correctly.

Several edge cases are important:

- Arrays may contain duplicate values.
- `d` may be zero, meaning only exact matches are disallowed.
- Negative numbers can appear in both arrays.
- Every element in `arr1` may qualify, or none may qualify.
- Arrays can contain a single element.

A naive implementation could accidentally stop too early or incorrectly handle absolute differences with negative values.

## Approaches

### Brute Force Approach

The most direct solution is to compare every element in `arr1` against every element in `arr2`.

For each value in `arr1`:

1. Iterate through all elements in `arr2`.
2. Compute the absolute difference.
3. If any difference is less than or equal to `d`, mark the element as invalid.
4. Otherwise, count it in the final answer.

This approach is guaranteed to work because it explicitly checks every possible pair.

The downside is the nested loop structure. If both arrays had much larger constraints, this would become inefficient. However, with only 500 elements per array, the runtime remains acceptable.

### Optimized Approach Using Sorting and Binary Search

We can improve the search process by sorting `arr2`.

Once `arr2` is sorted, instead of scanning every element for each value in `arr1`, we can use binary search to find the closest candidate values.

The key observation is this:

If `arr2` is sorted, then only the nearby elements around the insertion position matter. We do not need to compare against the entire array.

For each value `x` in `arr1`:

1. Use binary search to find where `x` would be inserted in `arr2`.
2. Check the neighboring values around that insertion point.
3. If either neighbor differs from `x` by at most `d`, then `x` is invalid.
4. Otherwise, count it.

This reduces repeated scanning and gives a more scalable solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × m) | O(1) | Compare every pair directly |
| Optimal | O(m log m + n log m) | O(1) or O(log m) | Sort `arr2` and use binary search |

Where:

- `n = len(arr1)`
- `m = len(arr2)`

## Algorithm Walkthrough

### Optimal Algorithm Using Sorting and Binary Search

1. Sort `arr2`.

Sorting allows efficient binary search operations. Once sorted, nearby values can be located quickly instead of scanning the whole array.
2. Initialize a counter called `distance_value`.

This variable stores how many elements from `arr1` satisfy the condition.
3. Iterate through each value `num` in `arr1`.

We independently determine whether this element is sufficiently far from all elements in `arr2`.
4. Use binary search to locate the insertion position of `num` in `arr2`.

The insertion position tells us where `num` would fit while maintaining sorted order.
5. Check the element at the insertion position, if it exists.

This value is the first number greater than or equal to `num`. It could be the closest value on the right side.
6. Check the element immediately before the insertion position, if it exists.

This value is the closest candidate on the left side.
7. Compute the absolute differences.

If either neighboring value differs from `num` by at most `d`, then `num` is invalid.
8. If neither neighbor violates the condition, increment `distance_value`.

This means all elements in `arr2` are farther than `d` away.
9. Return the final counter.

### Why it works

Binary search identifies the closest position where the current number would appear in sorted order. Any value farther away than the immediate neighbors must have an even larger difference. Therefore, checking only the neighboring elements is sufficient to determine whether any value lies within distance `d`.

## Python Solution

```python
from bisect import bisect_left
from typing import List

class Solution:
    def findTheDistanceValue(self, arr1: List[int], arr2: List[int], d: int) -> int:
        arr2.sort()
        distance_value = 0

        for num in arr1:
            index = bisect_left(arr2, num)

            valid = True

            if index < len(arr2):
                if abs(arr2[index] - num) <= d:
                    valid = False

            if index > 0:
                if abs(arr2[index - 1] - num) <= d:
                    valid = False

            if valid:
                distance_value += 1

        return distance_value
```

The implementation begins by sorting `arr2`. This is necessary because binary search only works correctly on sorted data.

For each number in `arr1`, the code uses `bisect_left` to locate the insertion position. This gives the first index where the number could be inserted while preserving sorted order.

Two neighboring candidates are then checked:

- `arr2[index]`, the closest value on the right
- `arr2[index - 1]`, the closest value on the left

If either value lies within distance `d`, the current element is invalid.

If both checks pass, the element contributes to the answer.

The boolean variable `valid` makes the logic easier to read and clearly separates the validation process from the counting process.

## Go Solution

```go
package main

import (
	"sort"
)

func findTheDistanceValue(arr1 []int, arr2 []int, d int) int {
	sort.Ints(arr2)

	distanceValue := 0

	for _, num := range arr1 {
		index := sort.SearchInts(arr2, num)

		valid := true

		if index < len(arr2) {
			diff := arr2[index] - num
			if diff < 0 {
				diff = -diff
			}

			if diff <= d {
				valid = false
			}
		}

		if index > 0 {
			diff := arr2[index-1] - num
			if diff < 0 {
				diff = -diff
			}

			if diff <= d {
				valid = false
			}
		}

		if valid {
			distanceValue++
		}
	}

	return distanceValue
}
```

The Go implementation follows the same strategy as the Python version.

`sort.Ints` is used to sort `arr2`, and `sort.SearchInts` performs binary search.

Go does not provide a built in absolute value function for integers in the standard library, so the code manually converts negative differences into positive values.

Slices in Go behave similarly to dynamic arrays, and no special handling is needed for empty arrays because the constraints guarantee at least one element in each array.

## Edge Cases

One important edge case occurs when `d = 0`. In this situation, only exact matches are invalid. A careless implementation might accidentally use a strict inequality instead of `<=`, which would incorrectly count equal values as valid. The implementation explicitly checks `<= d`, so exact matches are handled correctly.

Another important case involves negative numbers. Since values may range from `-1000` to `1000`, subtraction can produce negative differences. Forgetting to use absolute values would produce incorrect comparisons. Both implementations explicitly compute absolute differences before comparing against `d`.

A third edge case appears when the binary search insertion point is at the beginning or end of the array. Accessing `arr2[index - 1]` when `index == 0`, or accessing `arr2[index]` when `index == len(arr2)`, would cause out of bounds errors. The implementation carefully checks boundaries before accessing neighboring elements.

A final subtle case involves duplicate values in either array. Each occurrence in `arr1` must be evaluated independently. The algorithm does not remove duplicates or convert arrays into sets, so repeated values are counted correctly according to the problem requirements.
