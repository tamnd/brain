## Solution

## Problem Understanding

The problem gives us an integer array `arr`, along with two integers, `k` and `threshold`. We need to count how many contiguous subarrays of length exactly `k` have an average value greater than or equal to `threshold`.

A subarray is a continuous portion of the array. Since the size must be exactly `k`, every valid candidate subarray contains exactly `k` consecutive elements.

For each such subarray, we calculate:

$$\text{average} = \frac{\text{sum of elements}}{k}$$

If this average is greater than or equal to `threshold`, we include it in the count.

The input constraints are important:

- The array length can be as large as `10^5`
- Each element can be up to `10^4`
- `k` can also be as large as the array itself

These constraints immediately suggest that inefficient nested-loop solutions may be too slow. In particular, recalculating the sum of every length-`k` subarray from scratch would lead to excessive repeated work.

One important observation is that we do not actually need floating point averages. Instead of checking:

$$\frac{\text{sum}}{k} \ge \text{threshold}$$

we can multiply both sides by `k`:

$$\text{sum} \ge k \times \text{threshold}$$

This avoids floating point precision concerns and makes the implementation cleaner and faster.

Several edge cases are worth considering:

- When `k == 1`, every individual element becomes its own subarray.
- When `k == len(arr)`, there is only one possible subarray.
- When `threshold == 0`, every subarray automatically satisfies the condition because all array values are positive.
- Arrays with repeated values can easily expose inefficient implementations if sums are repeatedly recomputed.

The problem guarantees that all inputs are valid, so we do not need to handle invalid array sizes or negative `k`.

## Approaches

### Brute Force Approach

The most direct solution is to examine every subarray of size `k`.

For each starting position:

1. Compute the sum of the next `k` elements.
2. Divide by `k` or compare against `k * threshold`.
3. Increment the answer if the condition is satisfied.

This approach is correct because it explicitly checks every valid subarray exactly once.

However, the inefficiency comes from recomputing sums repeatedly. Consecutive subarrays overlap heavily. For example:

```
[2,2,2,2,5,5,5,8]
```

The subarrays:

```
[2,2,2]
[2,2,2]
[2,2,5]
```

share most of their elements. Recalculating each sum from scratch wastes work.

If there are `n` elements, and each subarray sum requires `O(k)` work, the total complexity becomes `O(n * k)`. With `n = 10^5`, this can become too slow.

### Optimal Sliding Window Approach

The key observation is that adjacent subarrays differ by only two elements:

- One element leaves the window
- One new element enters the window

Instead of recomputing the entire sum every time, we can update the current sum incrementally.

Suppose the current window sum is:

```
sum(arr[i : i+k])
```

The next window sum becomes:

```
current_sum - arr[i] + arr[i+k]
```

This update takes constant time.

This pattern is called the sliding window technique. It works perfectly here because:

- The window size is fixed at `k`
- Consecutive windows overlap heavily
- Only one element changes at each step

As a result, we process the entire array in linear time.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * k) | O(1) | Recomputes every subarray sum from scratch |
| Optimal Sliding Window | O(n) | O(1) | Reuses previous window sum efficiently |

## Algorithm Walkthrough

1. Compute the required minimum sum.

Instead of checking averages directly, compute:

$$\text{required\_sum} = k \times threshold$$

Any subarray whose sum is at least this value satisfies the condition.
2. Build the first window sum.

Add the first `k` elements together. This represents the first subarray of size `k`.
3. Check the first window.

If the first window sum is greater than or equal to `required_sum`, increment the answer.
4. Slide the window through the array.

Starting from index `k`, repeatedly:

- Remove the element leaving the window
- Add the new entering element

Update the running sum in constant time.
5. Check each updated window.

After every update, compare the current sum against `required_sum`.
6. Continue until all windows are processed.

Since every valid subarray of size `k` corresponds to exactly one window position, the final count is the answer.

### Why it works

At every step, the sliding window maintains the exact sum of the current contiguous subarray of size `k`. Because each possible size-`k` subarray appears exactly once as the window moves from left to right, every valid candidate is checked exactly once. The incremental update preserves correctness because the new window differs from the previous window by only one removed element and one added element.

## Python Solution

```python
from typing import List

class Solution:
    def numOfSubarrays(self, arr: List[int], k: int, threshold: int) -> int:
        required_sum = k * threshold

        window_sum = sum(arr[:k])
        count = 0

        if window_sum >= required_sum:
            count += 1

        left = 0

        for right in range(k, len(arr)):
            window_sum += arr[right]
            window_sum -= arr[left]
            left += 1

            if window_sum >= required_sum:
                count += 1

        return count
```

The implementation begins by converting the average requirement into a sum requirement. This avoids floating point arithmetic entirely.

Next, the code computes the sum of the first window using `sum(arr[:k])`. This initializes the sliding window.

The first window is checked immediately because it is already a valid subarray of size `k`.

The variable `left` tracks the element that should be removed when the window slides forward. The `for` loop starts at index `k`, which is the first element outside the initial window.

For each iteration:

- The new element enters the window
- The oldest element leaves the window
- The running sum is updated in constant time

After updating the window, the algorithm checks whether the sum satisfies the threshold condition.

Finally, the accumulated count is returned.

## Go Solution

```go
func numOfSubarrays(arr []int, k int, threshold int) int {
	requiredSum := k * threshold

	windowSum := 0
	for i := 0; i < k; i++ {
		windowSum += arr[i]
	}

	count := 0

	if windowSum >= requiredSum {
		count++
	}

	left := 0

	for right := k; right < len(arr); right++ {
		windowSum += arr[right]
		windowSum -= arr[left]
		left++

		if windowSum >= requiredSum {
			count++
		}
	}

	return count
}
```

The Go implementation follows the exact same logic as the Python version.

One small difference is that Go does not provide a built-in `sum()` function for slices, so the first window sum is computed manually using a loop.

Since the constraints are small enough, integer overflow is not a concern in Go. The maximum possible window sum is:

$$10^5 \times 10^4 = 10^9$$

which fits safely inside Go's standard `int` type on LeetCode platforms.

| --- | --- | --- |

| Space | O(1) | Only a few variables are used |

The algorithm processes the array in a single pass after initializing the first window. Every update to the sliding window takes constant time, so the total runtime grows linearly with the size of the array.

The memory usage remains constant because no additional data structures proportional to input size are created.

## Edge Cases

One important edge case occurs when `k == 1`. In this situation, every individual element becomes its own subarray. A buggy implementation might incorrectly assume windows always contain multiple elements. The sliding window approach handles this naturally because each window simply contains one value, and the update logic still works correctly.

Another important case occurs when `k == len(arr)`. There is only one possible subarray, the entire array itself. Some implementations accidentally skip checking the initial window before sliding begins, which would fail here because there are no later windows. This implementation explicitly checks the first window before entering the sliding loop.

A third edge case involves very large arrays with repeated values. A brute-force implementation would repeatedly recompute nearly identical sums, causing unnecessary quadratic-style work. The sliding window solution avoids this by maintaining a running sum and updating it in constant time.

A final subtle edge case is when `threshold == 0`. Since all array elements are positive integers, every subarray automatically satisfies the condition. The implementation still works correctly because every window sum will be greater than or equal to zero.
