## Solution

## Problem Understanding

The problem asks us to group every integer from `1` to `n` based on the sum of its digits. Two numbers belong to the same group if their digit sums are equal.

For example:

- `14` has digit sum `1 + 4 = 5`
- `5` has digit sum `5`

Since both sums are equal, they belong to the same group.

On the other hand:

- `13` has digit sum `4`
- `3` has digit sum `3`

These belong to different groups.

After forming all groups, we must determine which group size is the largest. Then we return how many groups have that maximum size.

The input consists of a single integer `n`, representing the inclusive range `[1, n]`.

The output is a single integer:

- Count how many groups contain the maximum number of elements.

The constraint is:

```
1 <= n <= 10^4
```

This tells us the input size is relatively small. Even if we process every number individually and compute its digit sum directly, the total work remains manageable.

An important observation is that the maximum possible digit sum for numbers up to `10^4` is small. For example:

- `9999` has digit sum `36`
- `10000` has digit sum `1`

So the number of distinct groups is limited.

Important edge cases include:

- `n = 1`, where only one group exists.
- Small values like `n = 2`, where every group has size `1`.
- Cases where multiple groups tie for the largest size.
- Larger values where digit sums repeat frequently.

The problem guarantees that `n` is always positive, so we never need to handle empty ranges or invalid inputs.

## Approaches

### Brute Force Approach

A brute force solution would generate every number from `1` to `n`, compute its digit sum, and place it into an explicit group structure such as a list for each digit sum.

For example:

- `1` goes into group `1`
- `10` also goes into group `1`
- `11` goes into group `2`

After building all groups, we would iterate through every group to determine:

1. The largest group size.
2. How many groups have that size.

This approach is correct because every number is processed exactly once and assigned to the proper group.

However, explicitly storing all numbers inside lists is unnecessary. We only care about the size of each group, not the actual members.

### Optimal Approach

The key insight is that we only need frequency counts.

Instead of storing every number inside a group, we store:

```
digit_sum -> count
```

For every number from `1` to `n`:

1. Compute its digit sum.
2. Increment the count for that digit sum.

After processing all numbers:

- Find the maximum frequency.
- Count how many digit sums have that frequency.

A hash map works well because digit sums are natural keys and frequencies are values.

This avoids unnecessary storage and keeps the implementation simple and efficient.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * d) | O(n) | Stores every number inside explicit groups |
| Optimal | O(n * d) | O(1) | Stores only frequency counts of digit sums |

Here, `d` represents the number of digits in a number. Since `n <= 10^4`, `d` is at most `5`.

## Algorithm Walkthrough

1. Create a hash map called `group_counts`.

The keys represent digit sums, and the values represent how many numbers belong to that group.
2. Iterate through every integer from `1` to `n`.

Each number must be assigned to exactly one group.
3. Compute the digit sum of the current number.

This can be done by repeatedly taking the last digit using `% 10` and removing digits using `// 10`.
4. Increment the frequency for that digit sum in the hash map.

If the digit sum does not exist yet, initialize it with `0` first.
5. After processing all numbers, determine the largest group size.

This is simply the maximum value inside the hash map.
6. Count how many groups have that maximum size.

Iterate through all frequencies and count how many equal the maximum.
7. Return the count.

### Why it works

Every number from `1` to `n` belongs to exactly one digit-sum group. The algorithm processes every number once and correctly increments the frequency of its corresponding group. Since the hash map stores the exact size of every group, taking the maximum frequency identifies the largest group size, and counting how many groups match that size produces the correct answer.

## Python Solution

```python
class Solution:
    def countLargestGroup(self, n: int) -> int:
        group_counts = {}

        for number in range(1, n + 1):
            digit_sum = 0
            current = number

            while current > 0:
                digit_sum += current % 10
                current //= 10

            group_counts[digit_sum] = group_counts.get(digit_sum, 0) + 1

        largest_size = max(group_counts.values())

        result = 0

        for count in group_counts.values():
            if count == largest_size:
                result += 1

        return result
```

The implementation follows the algorithm directly.

The dictionary `group_counts` stores how many numbers belong to each digit-sum group.

The outer loop iterates from `1` through `n`. For every number, the inner `while` loop computes the digit sum by repeatedly extracting digits.

For example, for `123`:

- Add `3`
- Add `2`
- Add `1`

Resulting in a digit sum of `6`.

After computing the digit sum, the dictionary frequency is incremented.

Once all numbers are processed, `max(group_counts.values())` finds the largest group size.

Finally, another loop counts how many groups have that size and returns the result.

## Go Solution

```go
func countLargestGroup(n int) int {
	groupCounts := make(map[int]int)

	for number := 1; number <= n; number++ {
		digitSum := 0
		current := number

		for current > 0 {
			digitSum += current % 10
			current /= 10
		}

		groupCounts[digitSum]++
	}

	largestSize := 0

	for _, count := range groupCounts {
		if count > largestSize {
			largestSize = count
		}
	}

	result := 0

	for _, count := range groupCounts {
		if count == largestSize {
			result++
		}
	}

	return result
}
```

The Go implementation mirrors the Python logic closely.

Instead of a Python dictionary, Go uses `map[int]int`.

Go maps automatically return the zero value for missing keys, so:

```go
groupCounts[digitSum]++
```

works even when the key does not yet exist.

Integer overflow is not a concern because the constraints are very small.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n * d) | Each number requires digit-sum computation |
| Space | O(1) | The number of possible digit sums is bounded |

The digit-sum computation takes proportional time to the number of digits in each number. Since `n <= 10^4`, each number has at most `5` digits, making the runtime effectively linear.

The space usage is effectively constant because the number of distinct digit sums is very small and bounded independently of `n`.

## Edge Cases

One important edge case is the minimum input value, `n = 1`. In this scenario, only one number exists, so only one group can exist. A buggy implementation might incorrectly assume multiple groups or mishandle the maximum calculation on a nearly empty map. This implementation handles it correctly because the dictionary contains exactly one entry after processing.

Another important edge case occurs when many groups tie for the largest size. For example, with `n = 13`, four different groups each contain two elements. Some incorrect solutions only track the largest size and forget to count how many groups achieve it. This implementation explicitly performs a second pass through all frequencies to count every matching group.

A third edge case involves numbers with multiple digits, such as `9999`. Computing digit sums incorrectly is a common source of bugs, especially when repeatedly extracting digits. The implementation uses a standard `% 10` and `// 10` loop, ensuring every digit is processed exactly once and no digits are skipped.

Another subtle edge case is the transition between single-digit and multi-digit numbers, such as from `9` to `10`. The digit sum changes from `9` to `1`, which causes numbers like `1` and `10` to belong to the same group. The implementation correctly handles this because the digit-sum computation is based entirely on arithmetic rather than string assumptions.
