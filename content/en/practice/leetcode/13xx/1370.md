## Solution

## Problem Understanding

The problem asks us to repeatedly build a new string by alternating between increasing and decreasing character order.

We are given a lowercase English string `s`. At every stage, we perform two phases:

1. Pick characters in ascending alphabetical order, taking at most one occurrence of each character during that pass.
2. Then pick characters in descending alphabetical order, again taking at most one occurrence of each character during that pass.

We continue alternating between these two directions until every character from the original string has been used.

The important detail is that during the increasing phase, we always pick the smallest available character that is larger than the previously chosen character. Similarly, during the decreasing phase, we always pick the largest available character that is smaller than the previously chosen character.

For example, if the string is `"aaaabbbbcccc"`:

- First increasing pass gives `"abc"`
- First decreasing pass gives `"cba"`
- Remaining characters are still `"aabbcc"`
- Repeating the process produces `"abccbaabccba"`

The constraints are small:

- `1 <= s.length <= 500`
- Only lowercase English letters appear

Since there are only 26 possible characters, this immediately suggests that counting frequencies is likely more efficient than repeatedly sorting or scanning large dynamic structures.

Several edge cases are important:

- A string with only one distinct character, such as `"aaaa"`, should simply return the same string.
- A string already in sorted order still needs to follow the alternating increasing/decreasing process.
- A string with all unique characters, such as `"abcdef"`, will produce a straightforward increasing pass followed by no useful decreasing additions.
- Very uneven frequency distributions, such as `"zzza"`, must still correctly alternate between directions while respecting remaining counts.

The problem guarantees that all characters are lowercase English letters, which allows us to use a fixed-size counting array of length 26.

## Approaches

### Brute Force Approach

A direct simulation approach would repeatedly sort the remaining characters and manually construct the next increasing or decreasing sequence.

One way to implement this is:

1. Store all remaining characters in a list.
2. Sort the list.
3. Build the increasing sequence by scanning left to right and selecting distinct characters.
4. Remove used characters.
5. Build the decreasing sequence by scanning right to left.
6. Repeat until no characters remain.

This approach is correct because it follows the exact rules of the problem. However, repeatedly sorting and removing characters is inefficient.

If the string length is `n`, sorting may happen many times, producing roughly `O(n^2 log n)` behavior in the worst case.

### Optimal Approach

The key observation is that the input contains only lowercase English letters.

Instead of repeatedly sorting, we can count how many times each character appears. Then:

- During the increasing phase, iterate from `'a'` to `'z'`
- During the decreasing phase, iterate from `'z'` to `'a'`

Whenever a character still has remaining frequency:

- Append it once
- Decrease its count

This perfectly simulates the required process while avoiding repeated sorting.

Since the alphabet size is fixed at 26, each full scan is effectively constant time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n² log n) | O(n) | Repeatedly sorts remaining characters |
| Optimal | O(n) | O(1) | Uses fixed-size frequency counting |

## Algorithm Walkthrough

1. Create a frequency array of size 26.

Each index represents a lowercase letter:

- Index `0` represents `'a'`
- Index `1` represents `'b'`
- And so on

This lets us efficiently track how many copies of each character remain.
2. Count all characters in the input string.

For every character `ch` in `s`:

- Compute its index using `ord(ch) - ord('a')`
- Increment the corresponding frequency
3. Initialize an empty result list.

We use a list instead of repeated string concatenation because appending to a list is more efficient in Python.
4. Continue processing until the result length equals the original string length.

This guarantees that all characters are eventually used.
5. Perform the increasing phase.

Iterate from index `0` to `25`:

- If the current character frequency is greater than zero:

- Append that character once
- Decrease its frequency by one

This simulates choosing the smallest available characters in increasing order.
6. Perform the decreasing phase.

Iterate from index `25` down to `0`:

- If the current character frequency is greater than zero:

- Append that character once
- Decrease its frequency by one

This simulates choosing the largest available characters in decreasing order.
7. Repeat steps 5 and 6 until all characters are used.
8. Convert the result list into a string and return it.

### Why it works

The algorithm always maintains the invariant that the frequency array accurately represents the remaining unused characters.

During each increasing phase, characters are appended in strictly increasing alphabetical order because we scan from left to right and take each available character once.

During each decreasing phase, characters are appended in strictly decreasing alphabetical order because we scan from right to left and again take each available character once.

Repeating these two passes exactly matches the process described in the problem statement, guaranteeing correctness.

## Python Solution

```python
class Solution:
    def sortString(self, s: str) -> str:
        frequency = [0] * 26
        
        for ch in s:
            frequency[ord(ch) - ord('a')] += 1
        
        result = []
        
        while len(result) < len(s):
            # Increasing pass
            for i in range(26):
                if frequency[i] > 0:
                    result.append(chr(i + ord('a')))
                    frequency[i] -= 1
            
            # Decreasing pass
            for i in range(25, -1, -1):
                if frequency[i] > 0:
                    result.append(chr(i + ord('a')))
                    frequency[i] -= 1
        
        return "".join(result)
```

The implementation begins by creating a frequency array with 26 slots, one for each lowercase English letter.

Next, the code counts every character in the input string. The expression `ord(ch) - ord('a')` converts a character into its alphabet index.

The `result` list stores the characters as they are generated. Using a list avoids the inefficiency of repeated string concatenation.

The main loop continues until the result contains the same number of characters as the original string.

Inside the loop:

- The first `for` loop performs the increasing pass from `'a'` to `'z'`
- The second `for` loop performs the decreasing pass from `'z'` to `'a'`

Whenever a character still has remaining frequency:

- The character is appended once
- Its count is reduced

Finally, `"".join(result)` converts the list into the final string.

## Go Solution

```go
func sortString(s string) string {
    frequency := make([]int, 26)

    for _, ch := range s {
        frequency[ch-'a']++
    }

    result := make([]byte, 0, len(s))

    for len(result) < len(s) {
        // Increasing pass
        for i := 0; i < 26; i++ {
            if frequency[i] > 0 {
                result = append(result, byte(i)+'a')
                frequency[i]--
            }
        }

        // Decreasing pass
        for i := 25; i >= 0; i-- {
            if frequency[i] > 0 {
                result = append(result, byte(i)+'a')
                frequency[i]--
            }
        }
    }

    return string(result)
}
```

The Go solution follows the same logic as the Python implementation.

A fixed-size integer slice stores character frequencies. Since Go strings are immutable, a byte slice is used to efficiently build the result.

The expression `ch - 'a'` converts a rune into its alphabet index.

The result slice is initialized with capacity `len(s)` to avoid unnecessary reallocations.

Unlike Python, Go requires explicit conversion from `byte` to `string` when returning the final result.

There are no integer overflow concerns because the maximum string length is only 500.

## Edge Cases

One important edge case is a string containing only one unique character, such as `"aaaaaa"`. A naive implementation might incorrectly alternate directions or attempt invalid comparisons between characters. The counting-based solution handles this naturally because every increasing pass simply consumes one remaining `'a'`, and the decreasing pass finds nothing else to add.

Another important edge case is a string where every character is unique, such as `"zyxwv"`. In this case, the entire result is produced during the first increasing pass. The decreasing phase immediately finds no remaining characters. The implementation correctly stops once the result length equals the original string length.

A third edge case involves highly uneven frequencies, such as `"zzzzza"`. Some implementations incorrectly append too many copies of a character during a single pass. This solution avoids that issue because each pass only appends a character once before moving on. The frequency array guarantees that duplicates are distributed across multiple alternating passes exactly as required.
