## Solution

## Problem Understanding

The problem gives us a list of restaurants where each restaurant is represented as an array of five integers:

```
[id, rating, veganFriendly, price, distance]
```

Each field has a specific meaning:

- `id` is the unique identifier of the restaurant
- `rating` is the restaurant's rating
- `veganFriendly` is either `1` or `0`
- `price` is the restaurant's price level
- `distance` is how far the restaurant is from the user

We are also given three filters:

- `veganFriendly`
- `maxPrice`
- `maxDistance`

The task is to return the IDs of restaurants that satisfy all required filters.

The vegan filter behaves slightly differently from the others:

- If `veganFriendly == 1`, we must only include restaurants whose `veganFriendly` value is also `1`
- If `veganFriendly == 0`, we can include all restaurants regardless of their vegan status

The other two filters are straightforward:

- Restaurant price must be less than or equal to `maxPrice`
- Restaurant distance must be less than or equal to `maxDistance`

After filtering, we must sort the remaining restaurants:

1. Higher rating comes first
2. If ratings are equal, higher ID comes first

Finally, we return only the restaurant IDs in the sorted order.

The constraints are important:

- Up to `10^4` restaurants
- Values can be as large as `10^5`

A size of `10^4` is moderate. It is completely acceptable to scan the list once and then sort the valid restaurants. Since sorting `10^4` elements costs `O(n log n)`, this is efficient enough.

Several edge cases are important:

- No restaurants satisfy the filters
- All restaurants satisfy the filters
- Multiple restaurants share the same rating
- Vegan filtering is disabled with `veganFriendly = 0`
- Restaurants exactly equal to `maxPrice` or `maxDistance` must still be included
- IDs are guaranteed to be unique, so no tie-breaking ambiguity exists beyond rating and ID

## Approaches

### Brute Force Approach

A naive brute-force solution would generate all possible restaurant candidates, repeatedly compare them, and manually construct the sorted order through repeated selection.

One possible brute-force method is:

1. Check every restaurant against the filters
2. Store valid restaurants
3. Repeatedly scan the remaining restaurants to find the next highest-rated restaurant
4. Remove it and continue until all restaurants are processed

This works because every valid restaurant is eventually considered and selected in the proper order.

However, repeatedly searching for the maximum element is inefficient. If there are `k` valid restaurants, repeatedly finding the best remaining restaurant costs roughly:

```
O(k + (k-1) + (k-2) + ...)
= O(k^2)
```

In the worst case, `k` can be `10^4`, which makes quadratic behavior unnecessarily expensive.

### Optimal Approach

The key observation is that we only need two operations:

1. Filter restaurants
2. Sort the filtered results once

Modern sorting algorithms are highly optimized and run in `O(n log n)` time.

The optimal strategy is therefore:

- Iterate through the restaurants once
- Keep only restaurants satisfying all filters
- Sort them using a custom sorting key:

- Higher rating first
- Higher ID first for ties
- Extract and return IDs

This is efficient because filtering is linear and sorting dominates the runtime.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeatedly searches for highest-rated restaurant |
| Optimal | O(n log n) | O(n) | Filter once, then sort using custom comparator |

## Algorithm Walkthrough

1. Initialize an empty list to store restaurants that pass all filters.

We need a separate collection because not every restaurant will satisfy the constraints.
2. Iterate through every restaurant in the input array.

Each restaurant contains five values:

```
[id, rating, veganFriendly, price, distance]
```
3. Apply the vegan filter.

If the input parameter `veganFriendly` is `1`, then we only accept restaurants whose own vegan flag is also `1`.

If the parameter is `0`, we skip this restriction entirely.
4. Apply the price filter.

The restaurant's price must satisfy:

```
price <= maxPrice
```
5. Apply the distance filter.

The restaurant's distance must satisfy:

```
distance <= maxDistance
```
6. If all conditions are satisfied, add the restaurant to the filtered list.

We keep the entire restaurant record because we still need rating and ID for sorting.
7. Sort the filtered restaurants.

The sorting rule is:

- Higher rating first
- Higher ID first when ratings are equal

In Python, we can accomplish this by sorting with:

```
key=lambda r: (-r[1], -r[0])
```

Negative values reverse the ordering.
8. Extract only the IDs from the sorted restaurants.

The problem requires only the ordered list of restaurant IDs.

### Why it works

The algorithm is correct because every restaurant is checked exactly once against all required conditions. Any restaurant failing even one condition is excluded, while every valid restaurant is included.

The sorting step enforces the exact ordering required by the problem statement:

- Descending rating
- Descending ID for ties

Since sorting is deterministic and applied to all valid restaurants, the final output is guaranteed to match the specification.

## Python Solution

```python
from typing import List

class Solution:
    def filterRestaurants(
        self,
        restaurants: List[List[int]],
        veganFriendly: int,
        maxPrice: int,
        maxDistance: int
    ) -> List[int]:

        filtered_restaurants = []

        for restaurant in restaurants:
            restaurant_id = restaurant[0]
            rating = restaurant[1]
            vegan = restaurant[2]
            price = restaurant[3]
            distance = restaurant[4]

            if veganFriendly == 1 and vegan == 0:
                continue

            if price > maxPrice:
                continue

            if distance > maxDistance:
                continue

            filtered_restaurants.append(restaurant)

        filtered_restaurants.sort(
            key=lambda r: (-r[1], -r[0])
        )

        return [restaurant[0] for restaurant in filtered_restaurants]
```

The implementation begins by creating an empty list called `filtered_restaurants`. This list stores only restaurants that satisfy all required conditions.

The loop processes each restaurant individually. For readability, each field is unpacked into meaningful variable names such as `rating`, `price`, and `distance`.

The vegan filtering logic is handled carefully:

```python
if veganFriendly == 1 and vegan == 0:
    continue
```

This means vegan restrictions are enforced only when requested.

The next two conditions remove restaurants that exceed the allowed price or distance.

Once all filters pass, the restaurant is appended to the filtered list.

After filtering, the list is sorted using:

```python
key=lambda r: (-r[1], -r[0])
```

Here:

- `r[1]` is the rating
- `r[0]` is the ID

Negative values create descending order.

Finally, a list comprehension extracts only the restaurant IDs.

## Go Solution

```go
package main

import "sort"

func filterRestaurants(restaurants [][]int, veganFriendly int, maxPrice int, maxDistance int) []int {

	filtered := [][]int{}

	for _, restaurant := range restaurants {

		id := restaurant[0]
		rating := restaurant[1]
		vegan := restaurant[2]
		price := restaurant[3]
		distance := restaurant[4]

		if veganFriendly == 1 && vegan == 0 {
			continue
		}

		if price > maxPrice {
			continue
		}

		if distance > maxDistance {
			continue
		}

		filtered = append(filtered, []int{
			id, rating, vegan, price, distance,
		})
	}

	sort.Slice(filtered, func(i, j int) bool {

		if filtered[i][1] == filtered[j][1] {
			return filtered[i][0] > filtered[j][0]
		}

		return filtered[i][1] > filtered[j][1]
	})

	result := []int{}

	for _, restaurant := range filtered {
		result = append(result, restaurant[0])
	}

	return result
}
```

The Go implementation follows the same logic as the Python version.

The primary difference is sorting syntax. Go uses `sort.Slice` with a custom comparator function instead of Python's `key` parameter.

Slices are dynamically resized using `append`, which is analogous to Python lists.

Go does not distinguish between `nil` and empty slices in a problematic way here. Returning an empty slice is completely valid when no restaurants satisfy the filters.

Integer overflow is not a concern because all values are well within Go's `int` range.

The algorithm performs a single pass through all restaurants, which costs `O(n)`.

The filtered restaurants are then sorted. In the worst case, all restaurants pass the filters, so sorting costs:

The extra space comes from storing the filtered restaurants before sorting.

## Edge Cases

One important edge case occurs when no restaurants satisfy the constraints. A buggy implementation might accidentally return `None`, raise an error during sorting, or fail while extracting IDs. This implementation handles the case naturally because sorting an empty list is valid and the final list comprehension simply returns an empty list.

Another important case involves restaurants whose values exactly equal `maxPrice` or `maxDistance`. Since the problem requires values less than or equal to the limits, using strict comparisons like `<` instead of `<=` would incorrectly exclude valid restaurants. The implementation explicitly checks:

```
if price > maxPrice:
```

and

```
if distance > maxDistance:
```

which correctly preserves boundary values.

A third important edge case is when multiple restaurants share the same rating. If the secondary sorting condition is forgotten, the output order becomes incorrect and potentially nondeterministic. The implementation handles this correctly by sorting with both rating and ID:

```
key=lambda r: (-r[1], -r[0])
```

This guarantees that ties are resolved using descending restaurant ID exactly as required.
