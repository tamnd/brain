## Solution

## Problem Understanding

The problem gives us a two dimensional matrix called `grid`, where each row is sorted in non-increasing order from left to right, and each column is also sorted in non-increasing order from top to bottom.

Non-increasing order means values either stay the same or decrease as we move right or downward. For example:

```
4  3  1 -1
3  2  0 -2
1  0 -1 -3
```

In this matrix:

- Rows go from larger values to smaller values
- Columns also go from larger values to smaller values

Our task is to count how many negative numbers exist in the entire matrix.

The input represents:

- `m` rows
- `n` columns
- Each element is an integer between `-100` and `100`

The expected output is a single integer representing the total number of negative values.

The constraints are relatively small:

- `1 <= m, n <= 100`

This means even an `O(m * n)` brute-force solution is acceptable for the official constraints. However, the follow up asks for an `O(m + n)` solution, which suggests there is an important structural property in the matrix that we should exploit.

Several edge cases are important:

- A matrix with no negative numbers
- A matrix where every number is negative
- A single row matrix
- A single column matrix
- Rows where the transition from positive to negative happens at different positions
- Zeros, since zero is not negative

The sorting guarantees are crucial because they allow us to avoid checking every element individually in the optimal solution.

## Approaches

### Brute Force Approach

The simplest solution is to iterate through every cell in the matrix and count how many values are less than zero.

We examine each element one by one:

- If the value is negative, increment the counter
- Otherwise continue

This approach is correct because every element is inspected exactly once, so no negative number can be missed.

However, this solution ignores the sorted structure of the matrix. Even if an entire section of the matrix is guaranteed to contain negative numbers, we still check every element individually.

The time complexity is `O(m * n)` because every cell is visited.

### Optimal Approach, Top Right Traversal

The key observation is that both rows and columns are sorted in non-increasing order.

This means:

- If a number is negative, then every number to its right in the same row is also negative
- If a number is non-negative, then every number above it in the same column is also non-negative

We can exploit this property by starting from the top-right corner of the matrix.

At each position:

- If the current number is negative:

- Everything below it in the same column is also negative
- We can count all remaining rows in that column immediately
- Then move left
- If the current number is non-negative:

- Everything to the left in that row is also non-negative
- So we move downward

This allows us to process the matrix in linear time relative to the number of rows and columns.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m * n) | O(1) | Checks every element individually |
| Optimal | O(m + n) | O(1) | Uses matrix sorting properties to skip regions |

## Algorithm Walkthrough

### Optimal Algorithm

1. Initialize a counter `negative_count` to `0`.
2. Start from the top-right corner of the matrix:

- Row index `row = 0`
- Column index `col = n - 1`
3. While `row < m` and `col >= 0`, continue processing.
4. Examine the current value `grid[row][col]`.
5. If the current value is negative:

- Because columns are sorted in non-increasing order, every value below this cell in the same column is also negative.
- The number of negative values in this column from the current row downward is:

```
m - row
```

- Add this quantity to `negative_count`.
- Move left by decrementing `col`.
6. Otherwise, if the current value is non-negative:

- Since rows are sorted in non-increasing order, every value to the left is also non-negative.
- Therefore there are no negatives in the current row before this point.
- Move downward by incrementing `row`.
7. Continue until either:

- We move past the last row, or
- We move before the first column.
8. Return `negative_count`.

### Why it works

The algorithm relies on the sorted structure of the matrix.

When we encounter a negative number at position `(row, col)`, every element below it in the same column must also be negative because columns decrease downward. Therefore we can count all those values at once instead of checking them individually.

When we encounter a non-negative number, every element to its left in the same row must also be non-negative because rows decrease left to right. Therefore we can safely ignore the rest of that row segment and move downward.

Each move eliminates either one row or one column from consideration, guaranteeing linear traversal complexity.

## Python Solution

```python
from typing import List

class Solution:
    def countNegatives(self, grid: List[List[int]]) -> int:
        rows = len(grid)
        cols = len(grid[0])

        row = 0
        col = cols - 1

        negative_count = 0

        while row < rows and col >= 0:
            if grid[row][col] < 0:
                negative_count += rows - row
                col -= 1
            else:
                row += 1

        return negative_count
```

The implementation begins by determining the matrix dimensions.

We initialize the traversal at the top-right corner because this position gives us the ability to eliminate either an entire row segment or an entire column segment after each comparison.

The `while` loop continues as long as the indices remain inside the matrix boundaries.

Inside the loop:

- If the current value is negative, all cells below it in the same column are guaranteed to be negative, so we add `rows - row` to the answer and move left.
- Otherwise the current value is non-negative, meaning all values to its left are also non-negative, so we move downward.

The algorithm finishes once all relevant rows or columns have been processed.

## Go Solution

```go
func countNegatives(grid [][]int) int {
    rows := len(grid)
    cols := len(grid[0])

    row := 0
    col := cols - 1

    negativeCount := 0

    for row < rows && col >= 0 {
        if grid[row][col] < 0 {
            negativeCount += rows - row
            col--
        } else {
            row++
        }
    }

    return negativeCount
}
```

The Go implementation follows the same logic as the Python version.

Slices are used naturally for matrix representation. Since the constraints guarantee at least one row and one column, accessing `grid[0]` is safe.

There are no integer overflow concerns because the matrix size is at most `100 x 100`, so the maximum possible count is only `10,000`.

## Edge Cases

One important edge case is when the matrix contains no negative numbers at all. A naive implementation might accidentally count zeros as negative or move incorrectly through the matrix. In our implementation, values are counted only if they are strictly less than zero. The traversal simply moves downward until it exits the matrix.

Another important case is when every number in the matrix is negative. In this scenario, the optimal algorithm repeatedly counts all remaining rows in each column. This efficiently handles the case without scanning every cell individually, while still producing the correct total.

A third important edge case is a single row or single column matrix. Pointer-based traversal algorithms sometimes fail when dimensions collapse to one direction. Our implementation correctly handles these cases because the loop conditions independently check row and column bounds, allowing traversal in either dimension without special handling.

A fourth subtle edge case involves zeros. Since the matrix is sorted, zeros may appear near the transition between positive and negative values. It is important not to treat zero as negative. The condition `grid[row][col] < 0` ensures only strictly negative values are counted.
