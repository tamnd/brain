## Solution

## Problem Understanding

The problem gives us a positive integer `num` that contains only the digits `6` and `9`. We are allowed to change at most one digit. A `6` can become a `9`, and a `9` can become a `6`.

Our goal is to return the largest possible number that can be formed after performing zero or one modification.

The important detail is that we are trying to maximize the numeric value. Since higher digits in more significant positions contribute more to the overall number, changing a digit near the front of the number has a much larger impact than changing one near the end.

For example, in `9669`, changing the second digit from `6` to `9` gives `9969`, which is larger than changing the third digit to produce `9699`. The earlier digit matters more because it affects a higher place value.

The constraints are very small:

- `1 <= num <= 10^4`
- The number contains only digits `6` and `9`

Since the number has at most 5 digits, almost any reasonable approach will work efficiently. However, the problem is designed to test whether we recognize the greedy observation that only one specific change matters.

There are several important edge cases to consider:

- The number may already contain only `9`s, such as `9999`. In that case, no change should be made.
- The number may contain only one `6`, and it could appear at the end, such as `9996`.
- The first digit may be `6`, such as `6999`, where changing the first digit gives the largest improvement possible.
- Since we may change at most one digit, we should stop immediately after making the best beneficial change.

## Approaches

### Brute Force Approach

A straightforward solution is to try changing every digit one by one and compute the resulting number.

For each position:

- If the digit is `6`, temporarily change it to `9`
- If the digit is `9`, temporarily change it to `6`
- Compute the resulting integer
- Keep track of the maximum value seen

We also include the original number because we are allowed to make no changes.

This approach works because it explicitly checks every valid possibility and chooses the maximum result.

However, it performs unnecessary work. We do not need to evaluate every possible transformation once we recognize how place value affects the number.

### Key Insight for the Optimal Approach

To maximize the number, we should increase the most significant digit possible.

Changing a `6` to a `9` always increases the number.

Changing a `9` to a `6` always decreases the number.

Therefore, the best strategy is:

- Find the first `6` from left to right
- Change it to `9`
- Return the result immediately

Why the first `6`?

Digits farther to the left have greater place value. Changing an earlier `6` produces a larger increase than changing any later `6`.

For example:

- `9669 -> 9969` increases the hundreds place
- `9669 -> 9699` increases only the tens place

The first change is larger.

This is a classic greedy observation. We make the locally optimal choice, changing the earliest `6`, and that choice is also globally optimal.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(d²) | O(d) | Try flipping every digit and compute all possibilities |
| Optimal | O(d) | O(d) | Change the first `6` to `9` and stop |

Here, `d` represents the number of digits.

## Algorithm Walkthrough

1. Convert the integer into a string or list of characters so individual digits can be modified easily.
2. Traverse the digits from left to right because earlier digits have greater significance in the final numeric value.
3. As soon as the first `6` is found, change it to `9`.
4. Stop immediately after the modification because only one change is allowed, and changing the earliest `6` already gives the maximum possible improvement.
5. Convert the modified digit sequence back into an integer and return it.
6. If no `6` is found, the number already consists entirely of `9`s, so return the original number unchanged.

### Why it works

The algorithm relies on place value ordering. A digit farther left contributes more to the total number than any digit to its right. Since changing `6` to `9` increases the value by `3 * place_value`, the largest increase comes from modifying the leftmost `6`. Any later modification produces a strictly smaller improvement. Therefore, changing the first `6` guarantees the maximum possible result.

## Python Solution

```python
class Solution:
    def maximum69Number(self, num: int) -> int:
        digits = list(str(num))

        for i in range(len(digits)):
            if digits[i] == '6':
                digits[i] = '9'
                break

        return int("".join(digits))
```

The implementation begins by converting the integer into a list of digit characters. Using a list makes modification straightforward because strings in Python are immutable.

The loop scans from left to right. The moment a `6` is encountered, it is replaced with `9`. We then immediately `break` out of the loop because only one modification is allowed and the earliest `6` always gives the best result.

Finally, the modified list of characters is joined back into a string and converted into an integer.

The code directly mirrors the greedy algorithm described earlier.

## Go Solution

```go
func maximum69Number(num int) int {
	digits := []byte(fmt.Sprintf("%d", num))

	for i := 0; i < len(digits); i++ {
		if digits[i] == '6' {
			digits[i] = '9'
			break
		}
	}

	result := 0

	for i := 0; i < len(digits); i++ {
		result = result*10 + int(digits[i]-'0')
	}

	return result
}
```

The Go implementation follows the same logic as the Python version.

One small difference is that Go strings are immutable, so we convert the string representation into a byte slice for in-place modification.

After updating the first `6`, we reconstruct the integer manually using arithmetic. This avoids needing additional parsing functions, though using `strconv.Atoi` would also be valid.

## Edge Cases

One important edge case is when the number already contains only `9`s, such as `9999`. A buggy implementation might accidentally change a `9` into a `6`, decreasing the result. Our implementation only changes digits when a `6` is found, so numbers consisting entirely of `9`s remain unchanged.

Another important case is when the first digit is `6`, such as `6999`. Since the leftmost digit has the greatest place value, changing it produces the largest possible increase. The algorithm scans from left to right, guaranteeing that the most significant `6` is modified first.

A third edge case occurs when there are multiple `6`s, such as `6666`. A careless implementation might change every `6` into `9`, violating the requirement that at most one digit can be modified. Our implementation immediately breaks after the first replacement, ensuring exactly one change at most.

A final edge case is the smallest valid input, such as `6` or `9`. Single-digit numbers can sometimes expose indexing mistakes or incorrect loop handling. The implementation works naturally for these inputs because the traversal logic is identical regardless of length.
