## Solution

## Problem Understanding

This problem requires analyzing relationships between three tables in a relational database: `Customers`, `Contacts`, and `Invoices`. Each `Invoice` is linked to a `user_id`, which corresponds to a customer. Each customer may have multiple contacts listed in the `Contacts` table. Some of these contacts may themselves be registered customers.

The task is to generate a result table for **each invoice**, containing the `invoice_id`, the `customer_name` of the invoice owner, the `price`, the total number of contacts (`contacts_cnt`), and the number of **trusted contacts** (`trusted_contacts_cnt`). A trusted contact is defined as someone listed in `Contacts` whose email also exists in the `Customers` table. The final table must be sorted by `invoice_id`.

Important points include that some customers may have no contacts, some contacts may not exist in the `Customers` table, and multiple invoices may belong to the same customer. The input sizes are moderate (up to thousands of records), but a brute-force nested query might be inefficient.

Edge cases to anticipate include customers with zero contacts, invoices linked to customers with zero trusted contacts, and contacts that exist but do not correspond to a registered customer.

## Approaches

The **brute-force approach** would involve iterating over each invoice, then counting contacts and checking if each contact's email exists in the `Customers` table. This could be done with multiple nested subqueries or joins. While this approach produces correct results, it can be inefficient because it repeatedly scans the `Contacts` and `Customers` tables for each invoice, leading to O(n*m) complexity where n is the number of invoices and m is the number of contacts.

The **optimal approach** leverages SQL joins and aggregate functions. By first joining `Invoices` to `Customers` and then performing a left join to `Contacts`, we can compute the total number of contacts per customer efficiently. To count trusted contacts, we can perform another left join with `Customers` (matching emails) and use a conditional aggregation (COUNT with filter or CASE) to compute the number of trusted contacts. This approach reduces the need for repeated scans and leverages the database engine’s optimizations.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n*m) | O(1) | Iterates over invoices and counts contacts individually using subqueries. |
| Optimal | O(n + m + k) | O(m + k) | Uses SQL joins and aggregations; n = invoices, m = contacts, k = customers. |

## Algorithm Walkthrough

1. **Join Invoices to Customers**: Start by joining the `Invoices` table with the `Customers` table on `user_id` to get the `customer_name` for each invoice. This ensures that every invoice is associated with its customer information.
2. **Left Join to Contacts**: Perform a left join of the previous result with the `Contacts` table on `user_id` to attach all contacts for each invoice's customer. Using a left join ensures customers without contacts are still included.
3. **Left Join to Customers Again for Trusted Contacts**: Join the `Contacts` table with `Customers` again on `contact_email` matching `email`. This identifies which contacts are trusted (registered customers).
4. **Aggregate Counts**: Use `COUNT` and `COUNT(DISTINCT)` to calculate the total number of contacts per invoice and the number of trusted contacts. If a customer has no contacts, the counts naturally become zero due to the left join.
5. **Group By Invoice**: Group the results by `invoice_id` and `customer_name` to ensure each invoice appears once with the aggregate counts.
6. **Order by Invoice ID**: Finally, order the results by `invoice_id` ascending, as required by the problem.

**Why it works**: The algorithm works because it systematically joins all relevant tables and uses aggregation to compute counts. The left joins ensure that missing contacts or missing trusted contacts are correctly counted as zero rather than excluded. Conditional aggregation ensures only valid trusted contacts are counted.

## Python Solution

```python
class Solution:
    def trustedContacts(self) -> str:
        return """
        SELECT 
            i.invoice_id,
            c.customer_name,
            i.price,
            COUNT(ct.contact_email) AS contacts_cnt,
            COUNT(c2.customer_id) AS trusted_contacts_cnt
        FROM Invoices i
        JOIN Customers c ON i.user_id = c.customer_id
        LEFT JOIN Contacts ct ON i.user_id = ct.user_id
        LEFT JOIN Customers c2 ON ct.contact_email = c2.email
        GROUP BY i.invoice_id, c.customer_name, i.price
        ORDER BY i.invoice_id;
        """
```

**Explanation**: The SQL query follows the algorithm steps. It joins `Invoices` to `Customers` to get the customer name, then left joins to `Contacts` for all contacts, and finally left joins again to `Customers` to count trusted contacts. Aggregation ensures the counts per invoice, and the result is ordered by `invoice_id`.

## Worked Examples

Take `invoice_id = 77` for Alice:

1. Join `Invoices` to `Customers` → Alice, price 100.
2. Left join to `Contacts` → Bob, John, Jal.
3. Left join `Contacts` to `Customers` on email → Bob and John are registered customers.
4. Aggregate counts → `contacts_cnt = 3`, `trusted_contacts_cnt = 2`.

For `invoice_id = 55` for John:

1. Join `Invoices` to `Customers` → John, price 500.
2. Left join to `Contacts` → no contacts.
3. Left join to `Customers` → no trusted contacts.
4. Counts → `contacts_cnt = 0`, `trusted_contacts_cnt = 0`.

All other invoices follow the same pattern.

## Edge Cases

1. **Customer with zero contacts**: Some customers may have invoices but no contacts. A naive inner join would exclude them. Using a left join ensures they appear in the result with `contacts_cnt = 0` and `trusted_contacts_cnt = 0`.
2. **Contacts not registered**: Some contacts may exist in the `Contacts` table but not in `Customers`. Trusted contact count must exclude these. The algorithm handles this with a left join to `Customers` on email.
3. **Multiple invoices per customer**: A customer may have multiple invoices. Counts must be per invoice, not per customer overall. Grouping by `invoice_id` ensures each invoice gets its own contact counts independently.
