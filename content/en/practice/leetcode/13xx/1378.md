## Solution

## Problem Understanding

This problem asks us to combine information from two database tables, `Employees` and `EmployeeUNI`, in order to display a list of all employees along with their unique identifier, if it exists. The `Employees` table contains every employee's `id` and `name`, while the `EmployeeUNI` table maps certain employee IDs to a `unique_id`. Not all employees have a corresponding entry in `EmployeeUNI`, so some employees will have `null` as their `unique_id`.

The input represents relational data in two tables. The `Employees` table is guaranteed to have unique `id` values. The `EmployeeUNI` table has unique combinations of `(id, unique_id)`, but not every employee from `Employees` may appear in `EmployeeUNI`. The expected output is a table with two columns: `unique_id` and `name`. If an employee has no entry in `EmployeeUNI`, the `unique_id` should be `null`. The output order does not matter.

Edge cases include employees who do not have a unique ID, an empty `EmployeeUNI` table, or employees with consecutive or non-consecutive IDs. The problem guarantees valid tables and that `id` values are integers.

## Approaches

A **brute-force approach** would involve iterating over each employee in `Employees` and, for each one, scanning through `EmployeeUNI` to find a matching `id`. This works correctly because it will eventually find the correct `unique_id` if it exists, or return `null` if it does not. However, this approach has poor efficiency, O(n*m) time complexity, where n is the number of employees and m is the number of entries in `EmployeeUNI`. For large tables, this becomes computationally expensive.

The **optimal approach** leverages a SQL `LEFT JOIN` operation. The key observation is that we only need to match employees with their `unique_id` if it exists, and we want to keep all employees in the output. A `LEFT JOIN` naturally preserves all rows from `Employees` and brings in `unique_id` from `EmployeeUNI` where available, filling `null` automatically when there is no match. This reduces the complexity significantly and uses SQL’s internal indexing and join algorithms for efficiency.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n*m) | O(1) | Iterates every employee and scans `EmployeeUNI` for each |
| Optimal | O(n + m) | O(n + m) | Uses LEFT JOIN to match tables efficiently; space for temporary join structures |

## Algorithm Walkthrough

1. Begin with the `Employees` table as the base. This ensures every employee is included in the final output regardless of whether they have a unique ID.
2. Perform a `LEFT JOIN` operation with `EmployeeUNI` on the `id` column. This ensures that if a match exists, the `unique_id` is pulled in; otherwise, SQL will assign `null`.
3. Select the `unique_id` and `name` columns for output. The order of rows is irrelevant, so no additional sorting is necessary.
4. Return the resulting table.

Why it works: The `LEFT JOIN` guarantees that all employees are preserved in the output while safely including `unique_id` when available. SQL handles missing matches by returning `null`, which matches the problem requirements. This ensures correctness and efficiency.

## Python Solution

```python
# LeetCode SQL submission does not require Python, but if we were to simulate in Python:
from typing import List, Tuple

def replace_employee_id_with_unique(employees: List[Tuple[int, str]], employee_uni: List[Tuple[int, int]]) -> List[Tuple[int, str]]:
    # Build a dictionary from EmployeeUNI for O(1) lookups
    id_to_unique = {eid: uid for eid, uid in employee_uni}
    
    # Construct the output
    result = []
    for eid, name in employees:
        result.append((id_to_unique.get(eid, None), name))
    
    return result
```

The Python implementation first builds a dictionary mapping `id` to `unique_id` for constant-time lookup. Then it iterates over every employee and retrieves the `unique_id` if it exists; otherwise, it assigns `None` (Python equivalent of SQL `NULL`). This mirrors the SQL `LEFT JOIN` logic.

## Go Solution

```go
package main

func ReplaceEmployeeIDWithUnique(employees [][]interface{}, employeeUni [][]interface{}) [][]interface{} {
    idToUnique := make(map[int]int)
    for _, row := range employeeUni {
        id := row[0].(int)
        uniqueID := row[1].(int)
        idToUnique[id] = uniqueID
    }

    result := make([][]interface{}, len(employees))
    for i, row := range employees {
        id := row[0].(int)
        name := row[1].(string)
        if unique, ok := idToUnique[id]; ok {
            result[i] = []interface{}{unique, name}
        } else {
            result[i] = []interface{}{nil, name}
        }
    }
    return result
}
```

In Go, we create a map for fast lookups from `id` to `unique_id`. For each employee, we check if their `id` exists in the map. If it does, we use the mapped `unique_id`; otherwise, we store `nil` to represent a missing value. Go requires explicit type assertions and the use of `interface{}` for heterogeneous row storage.

## Edge Cases

One edge case is when the `EmployeeUNI` table is empty. In this case, all `unique_id` values should be `null`. Another edge case is when `Employees` has rows with no corresponding `id` in `EmployeeUNI`, which must correctly yield `null` instead of causing a runtime error. A third edge case is when `EmployeeUNI` contains duplicate `unique_id` values for different employees; our solution handles this because the mapping is keyed by `id`, ensuring each employee receives their correct `unique_id`. All these cases are handled naturally by the LEFT JOIN logic in SQL or dictionary/map lookups in code.
