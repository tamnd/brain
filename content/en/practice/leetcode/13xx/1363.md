## Solution

## Problem Understanding

The problem gives us an array of digits, where each element is an integer from `0` to `9`. We may choose any subset of these digits and arrange them in any order to form a number. The goal is to construct the numerically largest possible number that is divisible by `3`.

A number is divisible by `3` if and only if the sum of its digits is divisible by `3`. This divisibility rule is the central observation that makes the problem tractable.

The output must be returned as a string because the resulting number can be extremely large. The constraints allow up to `10^4` digits, which means the final number may contain thousands of characters and cannot safely fit inside standard integer types.

The phrase "largest multiple of three" means we care about lexicographical and numerical magnitude. To maximize the resulting number, we want:

- As many digits as possible
- Larger digits placed earlier
- Digits sorted in descending order after selecting the valid subset

The constraints strongly suggest that brute force enumeration is impossible. With `n = 10^4`, even checking all subsets would require `2^10000` operations, which is astronomically infeasible.

Several edge cases are especially important:

- The array may contain only digits whose sum is not divisible by `3`
- The best answer may require removing one or two digits
- The result could consist entirely of zeros
- There may be no valid non-empty multiple of `3`
- Removing digits greedily in the wrong order can produce a smaller final answer

For example, if the input is `[0,0,0]`, the correct answer is `"0"`, not `"000"`.

If the input is `[1]`, there is no non-empty multiple of `3`, so the answer is an empty string.

## Approaches

### Brute Force Approach

A brute force solution would generate every possible subset of digits, then generate every permutation of each subset, and finally check whether the resulting number is divisible by `3`.

This works because it explicitly examines every possible valid number that can be formed. Among all numbers divisible by `3`, we could keep track of the largest one.

However, this approach is completely impractical. For an array of length `n`, there are `2^n` subsets. Each subset may also require permutation generation, which introduces factorial complexity. Even for modest values of `n`, the runtime becomes impossible.

The constraints make it clear that we need a mathematical shortcut instead of exhaustive search.

### Key Insight

The critical observation is the divisibility rule for `3`:

$\text{A number is divisible by 3 iff the sum of its digits is divisible by 3}$

This changes the problem dramatically.

Instead of thinking about permutations, we only need to decide which digits to keep or remove.

To maximize the resulting number:

- We should keep as many digits as possible
- If removals are necessary, remove the smallest possible digits
- After choosing the digits, sort them in descending order

Every digit belongs to one of three remainder groups when divided by `3`:

- Remainder `0`
- Remainder `1`
- Remainder `2`

Suppose the total digit sum leaves remainder:

- `0`, then all digits can be used
- `1`, then remove:

- one smallest digit with remainder `1`, or
- two smallest digits with remainder `2`
- `2`, then remove:

- one smallest digit with remainder `2`, or
- two smallest digits with remainder `1`

The greedy strategy works because removing the smallest digits preserves the largest possible final number.

## Algorithm Walkthrough

1. Sort the digits in ascending order.

Sorting makes it easy to remove the smallest digits when adjustments are needed. Removing smaller digits preserves larger digits for the final answer.
2. Compute the total sum of all digits.

The divisibility rule for `3` depends entirely on the digit sum. We calculate:

$r = (\sum digits) \bmod 3$

If `r == 0`, the current digits already form a valid multiple of `3`.
3. Group digits by remainder.

While scanning the sorted digits, track indices of digits whose values produce:

- `digit % 3 == 1`
- `digit % 3 == 2`

These groups allow us to quickly identify the smallest removable digits.
4. Handle the remainder cases.

If the total remainder is `1`:

- Prefer removing the smallest digit with remainder `1`
- Otherwise remove the two smallest digits with remainder `2`

If the total remainder is `2`:

- Prefer removing the smallest digit with remainder `2`
- Otherwise remove the two smallest digits with remainder `1`

This is the key greedy step. Removing the fewest and smallest digits preserves the maximum number.
5. Build the remaining digits.

After removing invalid digits, collect all remaining digits and sort them in descending order.

Descending order guarantees the numerically largest possible value.
6. Handle leading zeros.

If the resulting number starts with `0`, then all remaining digits must also be `0`.

In that case, return `"0"` instead of multiple zeros.
7. Return the final string.

Concatenate the remaining digits into a string and return it.

### Why it works

The correctness depends on two facts:

- A number is divisible by `3` exactly when its digit sum is divisible by `3`
- To maximize the resulting number, we should remove the smallest possible digits while removing the minimum required amount

Because digits are ultimately sorted descending, preserving larger digits always produces a larger final number. The greedy removals therefore produce the optimal solution.

## Python Solution

```python
from typing import List

class Solution:
    def largestMultipleOfThree(self, digits: List[int]) -> str:
        digits.sort()
        
        total_sum = sum(digits)
        remainder = total_sum % 3
        
        remove = set()
        
        rem1_indices = []
        rem2_indices = []
        
        for i, digit in enumerate(digits):
            if digit % 3 == 1:
                rem1_indices.append(i)
            elif digit % 3 == 2:
                rem2_indices.append(i)
        
        if remainder == 1:
            if rem1_indices:
                remove.add(rem1_indices[0])
            elif len(rem2_indices) >= 2:
                remove.add(rem2_indices[0])
                remove.add(rem2_indices[1])
            else:
                return ""
        
        elif remainder == 2:
            if rem2_indices:
                remove.add(rem2_indices[0])
            elif len(rem1_indices) >= 2:
                remove.add(rem1_indices[0])
                remove.add(rem1_indices[1])
            else:
                return ""
        
        result = []
        
        for i in range(len(digits) - 1, -1, -1):
            if i not in remove:
                result.append(str(digits[i]))
        
        if not result:
            return ""
        
        if result[0] == "0":
            return "0"
        
        return "".join(result)
```

The implementation begins by sorting the digits in ascending order. This ordering is important because we want to remove the smallest digits whenever adjustments are necessary.

Next, the code computes the total digit sum and determines its remainder modulo `3`. This remainder tells us whether the current collection of digits is already valid or which digits must be removed.

Two arrays store indices of digits with remainder `1` and remainder `2`. Because the array is sorted, the earliest indices correspond to the smallest digits in those groups.

The removal logic directly mirrors the divisibility cases:

- If the total remainder is `1`, remove either:

- one smallest remainder-`1` digit, or
- two smallest remainder-`2` digits
- If the total remainder is `2`, remove either:

- one smallest remainder-`2` digit, or
- two smallest remainder-`1` digits

After deciding which indices to discard, the algorithm builds the answer from largest to smallest digit by iterating backward through the sorted array.

Finally, the implementation handles the special all-zero case. If the largest remaining digit is `"0"`, the entire number must be zero, so the correct answer is simply `"0"`.

## Go Solution

```go
package main

import (
	"sort"
	"strconv"
	"strings"
)

func largestMultipleOfThree(digits []int) string {
	sort.Ints(digits)

	totalSum := 0
	for _, digit := range digits {
		totalSum += digit
	}

	remainder := totalSum % 3

	rem1 := []int{}
	rem2 := []int{}

	for i, digit := range digits {
		if digit%3 == 1 {
			rem1 = append(rem1, i)
		} else if digit%3 == 2 {
			rem2 = append(rem2, i)
		}
	}

	remove := make(map[int]bool)

	if remainder == 1 {
		if len(rem1) >= 1 {
			remove[rem1[0]] = true
		} else if len(rem2) >= 2 {
			remove[rem2[0]] = true
			remove[rem2[1]] = true
		} else {
			return ""
		}
	} else if remainder == 2 {
		if len(rem2) >= 1 {
			remove[rem2[0]] = true
		} else if len(rem1) >= 2 {
			remove[rem1[0]] = true
			remove[rem1[1]] = true
		} else {
			return ""
		}
	}

	var builder strings.Builder

	for i := len(digits) - 1; i >= 0; i-- {
		if !remove[i] {
			builder.WriteString(strconv.Itoa(digits[i]))
		}
	}

	result := builder.String()

	if result == "" {
		return ""
	}

	if result[0] == '0' {
		return "0"
	}

	return result
}
```

The Go implementation follows the same logic as the Python solution, but there are a few language-specific details worth noting.

Go does not have Python's flexible list comprehensions or set type, so a `map[int]bool` is used to track removed indices.

String concatenation inside loops is inefficient in Go because strings are immutable. To avoid repeated allocations, the implementation uses `strings.Builder`.

The solution also uses `sort.Ints` for ascending sorting and iterates backward to construct the largest possible number.

Since the result can become extremely large, the implementation keeps everything as strings instead of attempting integer conversion.

## Edge Cases

One important edge case occurs when all digits are zero. A naive implementation might return `"0000"` or another string with leading zeros. However, the problem explicitly states that unnecessary leading zeros are not allowed. The implementation handles this by checking whether the largest remaining digit is `"0"`. If so, it returns exactly `"0"`.

Another tricky case happens when no multiple of `3` can be formed. For example, with input `[1]`, removing the offending digit leaves no digits remaining. Some implementations accidentally return `"0"` here, which would be incorrect because zero cannot be formed from the provided digits. The solution correctly returns an empty string when all digits must be removed.

A third subtle case involves choosing between removing one digit or two digits. Suppose the sum remainder is `1`. Removing a single remainder-`1` digit is preferable to removing two remainder-`2` digits because it preserves more digits overall. However, when multiple candidates exist, we must also remove the smallest possible digits. Sorting first ensures the greedy removal strategy always preserves the lexicographically largest final number.

A fourth edge case appears when many repeated digits exist. Inputs like `[7,7,7,7,7]` can expose indexing mistakes or incorrect remainder grouping logic. The implementation safely tracks removable indices and removes only the minimum required digits.

Finally, very large inputs stress both runtime and memory efficiency. Since the constraints allow `10^4` digits, factorial or exponential approaches are impossible. The sorting-based greedy solution scales efficiently within the required limits.
