# LeetCode 1362 - Closest Divisors

## Problem Understanding

The problem gives us a positive integer `num` and asks us to find two integers whose product equals either:

- `num + 1`
- `num + 2`

Among all valid divisor pairs from these two numbers, we must return the pair whose absolute difference is the smallest.

A divisor pair of a number `x` consists of two integers `a` and `b` such that:

```
a * b = x
```

The phrase "closest divisors" means the two numbers should be as close to each other as possible. In mathematical terms, we want to minimize:

```
|a - b|
```

For example, if `num = 8`:

- `num + 1 = 9`, divisor pairs are `(1, 9)` and `(3, 3)`
- `num + 2 = 10`, divisor pairs are `(1, 10)` and `(2, 5)`

The pair `(3, 3)` has the smallest difference, so it is returned.

The constraints tell us that:

```
1 <= num <= 10^9
```

This is important because it rules out inefficient factorization methods. Any algorithm that checks all numbers up to `num` would be far too slow in the worst case. We need a more mathematical observation to solve the problem efficiently.

A key mathematical fact is that for any number, the closest divisor pair is centered around its square root. For example:

```
36 = 6 * 6
40 = 5 * 8
45 = 5 * 9
```

As the factors move away from the square root, their difference grows larger.

Some important edge cases include:

- Very small values such as `num = 1`
- Prime numbers, where the only divisors may be `1` and the number itself
- Cases where `num + 1` is prime but `num + 2` has balanced factors
- Large inputs near `10^9`, where efficiency becomes critical

The problem guarantees that at least one valid answer always exists because every positive integer has divisor pair `(1, x)`.

## Approaches

### Brute Force Approach

A straightforward approach is to generate all divisor pairs for both `num + 1` and `num + 2`.

For each candidate number:

1. Iterate from `1` to the number itself
2. Check whether the current value divides the number evenly
3. If it does, compute the paired divisor
4. Track the pair with the smallest difference

This approach is correct because it examines every possible divisor pair. However, it is extremely inefficient.

If `num` is close to `10^9`, iterating all the way to the number itself would require billions of operations, which is not feasible.

### Key Insight for the Optimal Solution

The important observation is that the closest divisor pair of a number is always nearest to its square root.

Suppose:

```
a * b = x
```

If `a` is very small, then `b` must be very large. The difference between them becomes large.

The minimum difference occurs when the two factors are as balanced as possible, which happens near:

```
sqrt(x)
```

Therefore, instead of checking every possible divisor, we can:

1. Start from `sqrt(num + 2)`
2. Move downward until we find a divisor
3. The first divisor found automatically produces the closest pair for that number

We perform this process for both `num + 1` and `num + 2`, then compare the resulting pairs.

This reduces the complexity dramatically.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(num) | O(1) | Checks every possible divisor |
| Optimal | O(sqrt(num)) | O(1) | Starts from square root and finds closest factors immediately |

## Algorithm Walkthrough

1. Compute the two candidate numbers:

- `num + 1`
- `num + 2`
2. For each candidate number, find its closest divisor pair.
3. To find the closest divisor pair for a number `target`:

- Compute `floor(sqrt(target))`
- Start checking downward from this value
- If `target % divisor == 0`, then:

```
divisor * (target / divisor) = target
```
- Since we started from the square root and moved downward, this is automatically the closest divisor pair.
4. Repeat the process for both candidate numbers.
5. Compare the absolute differences of the two pairs.
6. Return the pair with the smaller difference.

### Why it works

For a fixed product `x`, divisor pairs become farther apart as one divisor moves away from `sqrt(x)`. By starting from the square root and moving downward, the first divisor found must produce the pair with the minimum absolute difference. Since we apply this logic independently to both `num + 1` and `num + 2`, comparing the two resulting pairs guarantees the correct final answer.

## Python Solution

```python
from typing import List
import math

class Solution:
    def closestDivisors(self, num: int) -> List[int]:
        def find_closest_pair(target: int) -> List[int]:
            divisor = int(math.sqrt(target))

            while divisor > 0:
                if target % divisor == 0:
                    return [divisor, target // divisor]

                divisor -= 1

            return [1, target]

        pair1 = find_closest_pair(num + 1)
        pair2 = find_closest_pair(num + 2)

        diff1 = abs(pair1[0] - pair1[1])
        diff2 = abs(pair2[0] - pair2[1])

        if diff1 <= diff2:
            return pair1

        return pair2
```

The solution defines a helper function called `find_closest_pair` that computes the closest divisor pair for a single number.

Inside the helper function, we begin from the integer square root of the target number. This is the most balanced possible starting point for a divisor pair. We then move downward until we find a divisor that evenly divides the target.

Once such a divisor is found, we immediately return the pair because it must be the closest possible pair for that target.

The main function evaluates both `num + 1` and `num + 2`, computes their closest pairs, compares the absolute differences, and returns the better one.

The implementation uses only constant extra space and avoids unnecessary computations.

## Go Solution

```go
package main

import "math"

func closestDivisors(num int) []int {
	findClosestPair := func(target int) []int {
		divisor := int(math.Sqrt(float64(target)))

		for divisor > 0 {
			if target%divisor == 0 {
				return []int{divisor, target / divisor}
			}

			divisor--
		}

		return []int{1, target}
	}

	pair1 := findClosestPair(num + 1)
	pair2 := findClosestPair(num + 2)

	diff1 := abs(pair1[0] - pair1[1])
	diff2 := abs(pair2[0] - pair2[1])

	if diff1 <= diff2 {
		return pair1
	}

	return pair2
}

func abs(x int) int {
	if x < 0 {
		return -x
	}

	return x
}
```

The Go implementation follows the same logic as the Python version. One difference is that Go requires explicit conversion between `float64` and `int` when using `math.Sqrt`.

Go does not provide a built in integer absolute value function, so a small helper function named `abs` is implemented.

The result is returned as a slice of integers.

## Edge Cases

### Very Small Input

When `num = 1`, the candidate numbers become `2` and `3`. Both are prime, so their only divisor pairs are `(1, 2)` and `(1, 3)`.

A buggy implementation might incorrectly assume there is always a divisor larger than `1`. This solution safely continues downward until it reaches `1`, guaranteeing a valid pair is found.

### Perfect Square Candidates

Sometimes `num + 1` or `num + 2` is a perfect square, such as:

```
num = 8
num + 1 = 9
```

The closest divisors are exactly the square root itself:

```
3 * 3 = 9
```

Starting directly from the square root ensures the algorithm finds this optimal pair immediately.

### Large Prime Numbers

If one candidate number is prime, its closest divisor pair is simply:

```
1 * prime
```

A naive implementation that searches upward from `1` would waste significant time. This solution searches downward from the square root, minimizing the number of checks before concluding that no larger divisor exists.
