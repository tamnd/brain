## Solution

## Problem Understanding

This problem asks us to transform sales periods into yearly sales totals.

We are given two database tables. The `Product` table stores basic product information, specifically the `product_id` and the corresponding `product_name`. The `Sales` table stores aggregated sales data over a continuous date range. Each row tells us that a product was sold from `period_start` to `period_end`, inclusive, and that the average sales amount per day during that period was `average_daily_sales`.

The key challenge is that a single sales period may span multiple calendar years. The output, however, must contain one row per product per year. That means we must split every sales interval into yearly segments and compute the sales contribution for each year separately.

For every year touched by a sales interval, we need to determine how many days of that interval fall inside that year. Since the average sales amount is constant throughout the interval, the total sales amount for that year is:

```
(number of days in that year segment) * average_daily_sales
```

The output must include:

- `product_id`
- `product_name`
- `report_year`
- `total_amount`

The results must be ordered by `product_id` and `report_year`.

An important detail is that both `period_start` and `period_end` are inclusive dates. This means that when calculating the number of days, we must include both endpoints. Forgetting this is a common source of off by one errors.

The constraints tell us that all dates fall between 2018 and 2020. This is extremely important because it limits the number of years any interval can span. At most, a row can overlap three years: 2018, 2019, and 2020. That allows us to use a very compact and efficient solution.

Several edge cases are important to consider. A sales period may exist entirely within one year, in which case no splitting is needed. Another interval may start at the very end of one year and continue into the next year, which requires precise boundary calculations. We must also correctly handle leap years, although the given range only includes 2020 as a leap year. Finally, we must ensure that intervals touching year boundaries, such as `2020-01-01`, are counted correctly because the dates are inclusive.

## Approaches

### Brute Force Approach

A straightforward brute force solution would iterate through every single day in every sales interval. For each day, we would determine its year and add `average_daily_sales` to that year's running total for the product.

Conceptually, this is simple and clearly correct. Every date contributes exactly once, and grouping by year naturally produces the desired output.

However, this approach is inefficient because it processes dates one by one. If intervals become large, the number of daily iterations grows significantly. Even though this particular problem only spans three years, iterating day by day is unnecessary work and does not scale well in more general settings.

The brute force method also requires repeated date arithmetic and grouping operations for every individual day.

### Key Insight

The important observation is that sales values are constant within each interval. We do not need to examine every day separately. Instead, we only need to determine how many days of the interval belong to each year.

Since the years are limited to 2018 through 2020, we can directly compute the overlap between a sales interval and each calendar year.

For a given year:

- The effective start date is the later of:

- the interval start
- January 1 of that year
- The effective end date is the earlier of:

- the interval end
- December 31 of that year

If the effective start is not after the effective end, then the interval overlaps that year. The number of overlapping days can then be computed directly using date arithmetic.

This transforms the problem from iterating through every day to iterating through at most three years per row.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(total_days) | O(number_of_outputs) | Iterates through every day individually |
| Optimal | O(n) | O(number_of_outputs) | Processes at most three yearly overlaps per sales row |

## Algorithm Walkthrough

1. Join the `Sales` and `Product` tables using `product_id`.

We need the product name in the final output, so every sales record must be matched with its corresponding product information.
2. For each sales interval, examine each possible year from 2018 to 2020.

Because the problem guarantees that all dates lie within this range, we only need to check these three years.
3. Construct the calendar boundaries for the current year.

For example, for 2019:

- year start = `2019-01-01`
- year end = `2019-12-31`
4. Compute the overlap between the sales interval and the current year.

The overlap starts at:

```
max(period_start, year_start)
```

The overlap ends at:

```
min(period_end, year_end)
```

This isolates the portion of the interval that lies inside the current year.
5. Check whether an overlap actually exists.

If:

```
overlap_start <= overlap_end
```

then the interval contributes sales to that year.
6. Compute the number of overlapping days.

Since the dates are inclusive:

```
days = DATEDIFF(overlap_end, overlap_start) + 1
```
7. Compute the total sales amount.

Multiply the overlapping day count by `average_daily_sales`.
8. Output the row.

Include:

- `product_id`
- `product_name`
- `report_year`
- `total_amount`
9. Sort the final result.

Order by:

- `product_id`
- `report_year`

### Why it works

The algorithm works because every day in a sales interval belongs to exactly one calendar year. By intersecting the interval with each year's boundaries, we partition the interval into non overlapping yearly segments. Each segment contributes:

```
(number of days in that segment) * average_daily_sales
```

Summing these contributions across all overlapping years exactly reconstructs the total sales distribution.

## Python Solution

```python
# complete solution here

SELECT
    s.product_id,
    p.product_name,
    YEAR(y.year_start) AS report_year,
    (
        DATEDIFF(
            LEAST(s.period_end, y.year_end),
            GREATEST(s.period_start, y.year_start)
        ) + 1
    ) * s.average_daily_sales AS total_amount
FROM Sales s
JOIN Product p
    ON s.product_id = p.product_id
JOIN (
    SELECT
        '2018-01-01' AS year_start,
        '2018-12-31' AS year_end
    UNION ALL
    SELECT
        '2019-01-01',
        '2019-12-31'
    UNION ALL
    SELECT
        '2020-01-01',
        '2020-12-31'
) y
ON s.period_start <= y.year_end
AND s.period_end >= y.year_start
ORDER BY s.product_id, report_year;
```

The solution begins by joining the `Sales` and `Product` tables so that each sales interval has access to the corresponding product name.

A derived table named `y` is then created. This table contains the boundaries for the three possible years in the problem: 2018, 2019, and 2020. Each row stores:

- the start date of the year
- the end date of the year

The join condition between `Sales` and `y` checks whether the sales interval overlaps the current year:

```python
s.period_start <= y.year_end
AND s.period_end >= y.year_start
```

If this condition is true, then the interval contributes sales to that year.

The overlapping segment is computed using:

- `GREATEST` for the effective start
- `LEAST` for the effective end

The day count is then calculated with `DATEDIFF + 1` because both endpoints are inclusive.

Finally, the total amount is computed by multiplying the overlapping day count by `average_daily_sales`.

## Edge Cases

One important edge case is a single day sales interval where `period_start == period_end`. This can easily expose off by one errors if the implementation forgets that dates are inclusive. The query handles this correctly by using `DATEDIFF(...) + 1`, ensuring that a one day interval contributes exactly one day's sales.

Another critical edge case occurs when an interval crosses a year boundary, such as `2019-12-31` to `2020-01-01`. A naive implementation might incorrectly assign both days to one year or lose one of the boundary dates entirely. By explicitly computing yearly intersections with `LEAST` and `GREATEST`, the implementation cleanly separates the interval into two independent yearly segments.

Leap years are also a subtle source of bugs. The year 2020 contains February 29, so intervals spanning that date must count an extra day. Because the solution relies on SQL date arithmetic rather than manual calculations, the database engine automatically handles leap year behavior correctly.

A final important edge case is an interval fully contained within one year. In this situation, only one yearly overlap exists, and the join condition naturally filters out all other years. This ensures that no unnecessary rows are produced.
