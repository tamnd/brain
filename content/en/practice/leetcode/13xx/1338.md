## Solution

## Problem Understanding

The problem asks us to reduce an array to at most half its size by removing all occurrences of some integers. The input is a list of integers `arr`. You can select any subset of integers from the array and remove all their occurrences. The goal is to determine the **minimum number of distinct integers** you must remove to reduce the array size by at least half.

For example, if an array has ten elements, we must remove integers whose combined frequency is at least five. The output is the number of distinct integers selected for removal, not the number of elements removed.

The constraints tell us that the array length can be large (up to `10^5`) and is always even. The values in the array are positive integers up to `10^5`. These constraints indicate that a solution using simple nested loops or repeated scanning will be too slow, and a more efficient approach using counting and sorting or a heap is necessary.

Important edge cases include arrays where all elements are the same, arrays where all elements are unique, and arrays where a few elements dominate the frequency distribution.

## Approaches

### Brute Force

The brute-force approach would involve generating all subsets of distinct integers and calculating the total frequency of each subset to see if it reduces the array by at least half. This guarantees correctness because it checks every possible combination, but the number of subsets grows exponentially (`2^n` for `n` unique integers), making it infeasible for `n` as high as `10^5`.

### Key Insight for Optimal Solution

The main observation is that removing integers with the **highest frequency first** is the most efficient way to reduce the array size quickly. Instead of testing all subsets, we can:

1. Count the frequency of each integer using a hash map.
2. Sort the frequencies in descending order.
3. Keep removing the integers with the highest frequencies until at least half of the array is removed.

This is a classic greedy approach: removing the most "impactful" elements first ensures the minimum number of integers removed to achieve the target array reduction.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n * n) | O(n) | Checks all subsets of unique integers; infeasible for large input |
| Optimal | O(n log n) | O(n) | Uses frequency counting and sorting or a max-heap to greedily remove highest frequencies |

## Algorithm Walkthrough

1. Count the frequency of each integer in `arr` using a hash map. This allows us to know how many elements each distinct integer contributes.
2. Extract all the frequency values and sort them in descending order. Sorting ensures that the integers contributing the most to the array size are considered first.
3. Initialize a counter for the number of elements removed and a counter for the number of integers selected.
4. Iterate over the sorted frequencies, adding each frequency to the removed elements counter, and increment the integer selection counter.
5. Stop once the removed elements counter reaches at least half the length of the array.
6. Return the integer selection counter as the minimum size of the set.

**Why it works:** By always selecting the integers with the highest frequency first, we guarantee that each selection maximizes the reduction in array size. This greedy choice is provably optimal because any other choice would remove fewer elements per integer and require more integers to reach half the array size.

## Python Solution

```python
from typing import List
from collections import Counter

class Solution:
    def minSetSize(self, arr: List[int]) -> int:
        n = len(arr)
        half = n // 2
        freq = Counter(arr)
        freq_values = sorted(freq.values(), reverse=True)
        
        removed = 0
        count = 0
        
        for f in freq_values:
            removed += f
            count += 1
            if removed >= half:
                break
        
        return count
```

The Python implementation first counts frequencies using `Counter`, which runs in O(n) time. Sorting the frequency values in descending order allows the greedy selection. We iterate over the sorted frequencies, accumulating the number of elements removed, and stop once we reach half the array size. Finally, we return the number of integers selected, which is the minimum set size.

## Go Solution

```go
import "sort"

func minSetSize(arr []int) int {
    freqMap := make(map[int]int)
    for _, num := range arr {
        freqMap[num]++
    }

    freqValues := make([]int, 0, len(freqMap))
    for _, count := range freqMap {
        freqValues = append(freqValues, count)
    }

    sort.Slice(freqValues, func(i, j int) bool {
        return freqValues[i] > freqValues[j]
    })

    removed, count, half := 0, 0, len(arr)/2
    for _, f := range freqValues {
        removed += f
        count++
        if removed >= half {
            break
        }
    }

    return count
}
```

In Go, the main differences are explicit map handling and slice creation. Sorting is done with `sort.Slice`. The logic mirrors Python: count frequencies, sort, and greedily remove the largest contributors until at least half of the array is removed.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | Counting frequencies is O(n), sorting frequencies is O(n log n) |
| Space | O(n) | Storing the frequency map requires O(n) space in the worst case |

Counting and sorting dominate the runtime. Space complexity is linear relative to the number of unique elements, which can be up to n.

## Edge Cases

One important edge case is when all elements are the same. The algorithm handles it naturally because the highest frequency is equal to the array length, so the minimum set size is one. Another edge case is when all elements are unique. Here, each integer contributes exactly one, and the algorithm correctly selects half the integers to remove half the array. Finally, arrays with equal frequencies of multiple integers can be tricky, but the sorted frequency approach ensures that any tie in frequency is resolved arbitrarily without affecting correctness. These cases verify that the greedy approach is robust for any distribution.
