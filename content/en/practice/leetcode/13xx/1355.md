## Solution

## Problem Understanding

This problem asks us to find all activities that have neither the maximum nor the minimum number of participants. We are given two tables: `Friends` and `Activities`. The `Friends` table lists each friend along with the activity they participate in. The `Activities` table lists all available activities.

The input represents a relational database structure where `Friends.activity` corresponds to `Activities.name`. The expected output is a table containing activity names that have a participant count strictly between the activity with the fewest participants and the activity with the most participants.

Constraints implied by the problem include that every activity listed in `Activities` will have at least one participant in `Friends`. Edge cases that can trip up a naive implementation include situations where multiple activities share the maximum or minimum count, activities with only one participant, or cases where there are only two activities (in which case no activity qualifies). The problem guarantees that activity names are unique in the `Activities` table, and IDs are unique in both tables.

## Approaches

The brute-force approach is to count participants for every activity by iterating through the `Friends` table and grouping by `activity`. Then, we find the maximum and minimum participant counts. Finally, we iterate through all activities again and select those with counts strictly between the min and max. This works correctly but is not optimal for large datasets because it requires multiple full table scans or subqueries.

The optimal approach leverages SQL aggregate functions and window functions. First, we count the number of participants per activity using `GROUP BY`. Next, we identify the maximum and minimum participant counts using `MAX()` and `MIN()` over the aggregated counts. Finally, we filter out activities whose counts match either extreme. This approach is efficient because it avoids multiple joins or redundant scans and operates directly on aggregated data.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n + m) | O(n) | Count participants manually, then filter by min/max. Works but requires multiple passes. |
| Optimal | O(n) | O(m) | Aggregate participants with GROUP BY, filter using MIN/MAX in a single query or derived table. |

## Algorithm Walkthrough

1. Join the `Activities` table with the `Friends` table on `Activities.name = Friends.activity` to ensure that all activities are considered even if they have zero participants.
2. Use `COUNT(Friends.id)` to determine the number of participants per activity. Group the results by `Activities.name` to get one row per activity.
3. Identify the maximum and minimum participant counts across all activities using aggregate functions `MAX()` and `MIN()`.
4. Filter the grouped activity counts to exclude activities that match either the maximum or minimum count.
5. Return the activity names that remain.

Why it works: By counting participants per activity and comparing each count to the global maximum and minimum, we guarantee that only activities strictly between the extremes are included. Using SQL aggregation ensures correctness even when multiple activities share the same count.

## Python Solution

```python
import sqlite3
from typing import List, Tuple

def activity_participants(conn: sqlite3.Connection) -> List[Tuple[str]]:
    query = """
    WITH activity_counts AS (
        SELECT a.name AS activity, COUNT(f.id) AS participant_count
        FROM Activities a
        LEFT JOIN Friends f ON a.name = f.activity
        GROUP BY a.name
    ),
    stats AS (
        SELECT MAX(participant_count) AS max_count,
               MIN(participant_count) AS min_count
        FROM activity_counts
    )
    SELECT activity
    FROM activity_counts, stats
    WHERE participant_count <> max_count
      AND participant_count <> min_count;
    """
    cursor = conn.cursor()
    cursor.execute(query)
    return cursor.fetchall()
```

This solution first calculates participant counts for each activity. The `WITH` clause (`activity_counts`) stores the count per activity. Another `WITH` clause (`stats`) calculates the overall maximum and minimum participant counts. Finally, we filter out activities with participant counts equal to the extremes and return the remaining activity names.

## Go Solution

```go
package main

import (
    "database/sql"
    _ "github.com/mattn/go-sqlite3"
)

func ActivityParticipants(db *sql.DB) ([]string, error) {
    query := `
    WITH activity_counts AS (
        SELECT a.name AS activity, COUNT(f.id) AS participant_count
        FROM Activities a
        LEFT JOIN Friends f ON a.name = f.activity
        GROUP BY a.name
    ),
    stats AS (
        SELECT MAX(participant_count) AS max_count,
               MIN(participant_count) AS min_count
        FROM activity_counts
    )
    SELECT activity
    FROM activity_counts, stats
    WHERE participant_count <> max_count
      AND participant_count <> min_count;
    `
    rows, err := db.Query(query)
    if err != nil {
        return nil, err
    }
    defer rows.Close()

    var activities []string
    for rows.Next() {
        var activity string
        if err := rows.Scan(&activity); err != nil {
            return nil, err
        }
        activities = append(activities, activity)
    }
    return activities, nil
}
```

The Go version is similar in logic to the Python solution. Go-specific considerations include explicit error handling, deferred row closure, and scanning each row into a string slice. SQL logic is identical.

3. Filter `activity_counts` for counts not equal to `max_count` or `min_count`:

Singing -> 2

4. Return `Singing`.

## Edge Cases

One edge case is when all activities have the same number of participants. In this case, no activity is strictly between the minimum and maximum, and the implementation correctly returns an empty set.

A second edge case is when there are only two activities. One is maximum and the other minimum, so no activity can be between them. The implementation returns empty, handling this scenario without errors.

A third edge case is when an activity has zero participants. Using `LEFT JOIN` ensures such activities are included with a count of zero, and they are correctly filtered as the minimum if applicable. This prevents missing any activities and ensures accurate filtering.
