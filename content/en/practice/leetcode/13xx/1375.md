## Solution

## Problem Understanding

The problem is asking us to simulate flipping bits in a binary string that starts entirely with zeros. We are given an array `flips`, which tells us the order in which each bit will be flipped from `0` to `1`. After each flip, we need to determine whether the binary string is **prefix-aligned**, which means that all bits from index `1` to the current step `i` are `1` and all bits beyond `i` are still `0`.

The input `flips` is a permutation of integers from `1` to `n`, ensuring that every bit will be flipped exactly once. This means we do not have to handle repeated flips or indices out of range. The length constraint `1 <= n <= 5 * 10^4` indicates that a naive approach that repeatedly scans the string at each step will likely be too slow.

The output is simply the count of how many times the string is prefix-aligned during this flipping process. An edge case is when the first flipped bit is not `1`, which guarantees that the first prefix cannot be aligned.

Key observations are that a prefix-aligned string after the `i`-th step must have all bits from `1` to `i` flipped. Therefore, the largest index flipped so far must be equal to the number of steps completed for the string to be aligned.

## Approaches

The **brute-force approach** would involve simulating the string itself as an array of zeros and ones. After each flip, we would check all bits from index `1` to `i` to see if they are all ones. This approach is correct but inefficient because checking all bits after each flip leads to a time complexity of O(n^2), which is too slow for `n` up to 50,000.

The **optimal approach** leverages a key observation: a binary string is prefix-aligned after step `i` if and only if the maximum index flipped so far equals `i`. Instead of maintaining the full string, we only need a variable to track the largest flipped index. Each flip updates this maximum, and we check whether it equals the step number. This reduces the problem to O(n) time and O(1) space.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^2) | O(n) | Maintain the full string and check prefix at each step |
| Optimal | O(n) | O(1) | Track only the maximum flipped index and compare with step number |

## Algorithm Walkthrough

1. Initialize a counter `prefix_count` to zero, which will track the number of times the string is prefix-aligned.
2. Initialize a variable `max_flipped` to zero to store the largest index flipped so far.
3. Iterate over the `flips` array with the index `i` (0-based) and value `flip`:

1. Update `max_flipped` to be the maximum of its current value and `flip`.
2. Check if `max_flipped` equals `i + 1` (because steps are 1-indexed). If it does, increment `prefix_count`.
4. Return `prefix_count` at the end of the iteration.

**Why it works:** The invariant is that if the largest flipped index at step `i` is equal to `i`, it implies all numbers from `1` to `i` have been flipped, because `flips` is a permutation and there are no missing flips. This guarantees the prefix is aligned.

## Python Solution

```python
from typing import List

class Solution:
    def numTimesAllBlue(self, flips: List[int]) -> int:
        prefix_count = 0
        max_flipped = 0
        
        for i, flip in enumerate(flips):
            max_flipped = max(max_flipped, flip)
            if max_flipped == i + 1:
                prefix_count += 1
        
        return prefix_count
```

**Explanation:** We iterate over each flip, update the maximum flipped index, and check if the current step aligns with the maximum. This efficiently counts all prefix-aligned steps without maintaining the full string.

## Go Solution

```go
func numTimesAllBlue(flips []int) int {
    prefixCount := 0
    maxFlipped := 0
    
    for i, flip := range flips {
        if flip > maxFlipped {
            maxFlipped = flip
        }
        if maxFlipped == i+1 {
            prefixCount++
        }
    }
    
    return prefixCount
}
```

**Explanation:** The Go implementation mirrors the Python approach. We maintain a simple integer for the maximum flipped index and a counter for prefix-aligned steps. Go handles slices naturally, and there are no issues with integer overflow for the problem constraints.

## Edge Cases

**Edge Case 1:** Single-element array `[1]`. This tests the minimal input size. The algorithm correctly identifies the flip as immediately prefix-aligned.

**Edge Case 2:** Array where the first flip is not `1`, such as `[2,1]`. This ensures that the algorithm does not incorrectly count steps where the prefix is incomplete. The solution handles this by comparing `max_flipped` to the step number.

**Edge Case 3:** Array in strictly descending order `[n, n-1, ..., 1]`. Only the final flip results in a prefix-aligned string. This tests the algorithm's ability to track the maximum flipped index correctly and avoid overcounting early steps.
