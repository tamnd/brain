## Solution

## Problem Understanding

This problem gives us two database tables, `Customers` and `Orders`. The `Customers` table stores information about each customer, specifically their unique `customer_id` and their `customer_name`. The `Orders` table stores purchases made by customers, where each row represents a single product purchased by a customer.

The goal is to find customers who satisfy three conditions simultaneously:

1. They purchased product `"A"`
2. They purchased product `"B"`
3. They did not purchase product `"C"`

The final output should contain the `customer_id` and `customer_name` of all such customers, sorted by `customer_id` in ascending order.

The important detail is that purchases are recorded row by row in the `Orders` table. A customer may appear multiple times because they can buy many products. We therefore need to aggregate or analyze all products associated with each customer before deciding whether they qualify.

For example, if a customer bought products `A`, `B`, and `D`, they qualify because they bought both required products and avoided `C`. However, if they bought `A`, `B`, and `C`, they must be excluded because the presence of `C` disqualifies them.

The input constraints are not explicitly stated in the problem statement, but this is a standard SQL aggregation problem. The number of rows in `Orders` could be large, so repeatedly scanning the table for each customer would be inefficient. The problem is fundamentally about grouping records by customer and evaluating membership conditions across their purchased products.

Several edge cases are important to consider:

- A customer may buy the same product multiple times. We should still treat that as simply "the customer bought this product."
- A customer may buy only `A` or only `B`, which is insufficient.
- A customer may buy `A`, `B`, and `C`, which must be excluded.
- Some customers may have no orders at all.
- There may be additional products besides `A`, `B`, and `C`, and they should not affect eligibility.

## Approaches

### Brute Force Approach

A brute-force solution would examine each customer individually and repeatedly scan the `Orders` table to determine whether the customer purchased products `A`, `B`, and `C`.

For every customer, we could:

- Search all their orders
- Track whether `A` exists
- Track whether `B` exists
- Track whether `C` exists
- Include the customer only if `A` and `B` are present and `C` is absent

This approach is correct because it explicitly verifies the required conditions for every customer. However, it is inefficient because the `Orders` table may be scanned repeatedly for every customer.

If there are `N` customers and `M` orders, the complexity becomes approximately `O(N × M)`, which is unnecessarily expensive.

### Optimal Approach

The key observation is that the problem is naturally suited for SQL aggregation using `GROUP BY`.

Instead of processing each customer independently, we group all orders by `customer_id` and compute aggregate conditions across the products they purchased.

We can use conditional aggregation:

- Count whether product `A` exists
- Count whether product `B` exists
- Count whether product `C` exists

Then we filter grouped customers using a `HAVING` clause:

- `A` count > 0
- `B` count > 0
- `C` count = 0

This is efficient because the database scans the `Orders` table only once during aggregation.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(N × M) | O(1) | Repeatedly scans all orders for every customer |
| Optimal | O(M) | O(N) | Uses grouping and conditional aggregation |

## Algorithm Walkthrough

1. Join the `Customers` and `Orders` tables using `customer_id`.

We need both customer information and purchase information in the final result. The join connects each customer with all of their purchased products.
2. Group all rows by customer.

Using `GROUP BY customer_id, customer_name` allows us to analyze all purchases belonging to the same customer together.
3. Compute whether each required product exists.

We use conditional aggregation with expressions like:

```
SUM(product_name = 'A')
```

This works because the boolean expression evaluates to `1` when true and `0` when false in MySQL.
4. Check the required conditions in the `HAVING` clause.

We require:

- At least one purchase of `A`
- At least one purchase of `B`
- Zero purchases of `C`
5. Sort the final result by `customer_id`.

The problem explicitly requires ordered output.

### Why it works

Grouping ensures that all orders for a single customer are processed together. The conditional sums correctly determine whether a customer purchased each target product. Because the `HAVING` clause filters groups after aggregation, only customers satisfying all required conditions remain in the result.

## Python Solution

Although this is a database problem and LeetCode expects SQL, the following Python implementation demonstrates the same logic programmatically.

```python
from collections import defaultdict
from typing import List, Dict

class Solution:
    def customersWhoBoughtProducts(self, customers: List[Dict], orders: List[Dict]) -> List[Dict]:
        customer_products = defaultdict(set)

        for order in orders:
            customer_id = order["customer_id"]
            product_name = order["product_name"]
            customer_products[customer_id].add(product_name)

        result = []

        for customer in customers:
            customer_id = customer["customer_id"]
            products = customer_products[customer_id]

            if "A" in products and "B" in products and "C" not in products:
                result.append({
                    "customer_id": customer_id,
                    "customer_name": customer["customer_name"]
                })

        result.sort(key=lambda x: x["customer_id"])

        return result
```

The implementation uses a dictionary that maps each customer to a set of products they purchased. A set is the ideal data structure here because membership checks such as `"A" in products` are efficient and duplicate purchases are automatically ignored.

The first loop processes all orders and builds the product set for each customer. The second loop iterates through all customers and checks whether the required inclusion and exclusion conditions are satisfied.

Finally, the result is sorted by `customer_id` to match the required output format.

For the actual LeetCode database submission, the SQL solution is:

```python
SELECT
    c.customer_id,
    c.customer_name
FROM Customers c
JOIN Orders o
    ON c.customer_id = o.customer_id
GROUP BY c.customer_id, c.customer_name
HAVING SUM(o.product_name = 'A') > 0
   AND SUM(o.product_name = 'B') > 0
   AND SUM(o.product_name = 'C') = 0
ORDER BY c.customer_id;
```

## Go Solution

```go
package main

import "sort"

type Customer struct {
	customer_id   int
	customer_name string
}

type Order struct {
	order_id     int
	customer_id  int
	product_name string
}

type Result struct {
	customer_id   int
	customer_name string
}

func customersWhoBoughtProducts(customers []Customer, orders []Order) []Result {
	customerProducts := make(map[int]map[string]bool)

	for _, order := range orders {
		if customerProducts[order.customer_id] == nil {
			customerProducts[order.customer_id] = make(map[string]bool)
		}

		customerProducts[order.customer_id][order.product_name] = true
	}

	var result []Result

	for _, customer := range customers {
		products := customerProducts[customer.customer_id]

		if products["A"] && products["B"] && !products["C"] {
			result = append(result, Result{
				customer_id:   customer.customer_id,
				customer_name: customer.customer_name,
			})
		}
	}

	sort.Slice(result, func(i, j int) bool {
		return result[i].customer_id < result[j].customer_id
	})

	return result
}
```

The Go implementation uses a nested map structure where each customer ID maps to another map representing purchased products. This emulates the behavior of a set because Go does not have a built-in set type.

One important Go-specific detail is checking whether the inner map exists before inserting into it. If it is `nil`, we initialize it using `make(map[string]bool)`.

Another detail is sorting. Unlike Python, Go does not provide a direct built-in sort for custom structs, so we use `sort.Slice` with a comparison function.

| --- | --- | --- |
| Time | O(M + N) | One pass through orders and one pass through customers |
| Space | O(N) | Stores purchased product sets per customer |

The dominant work comes from scanning the `Orders` table once to build the product mapping. After that, each customer is checked exactly once. The extra space comes from storing product sets for each customer.

For the SQL solution, the database engine performs aggregation internally, typically using hashing or sorting. The effective complexity is approximately linear in the number of order rows.

## Edge Cases

### Customers Who Buy Product C Along With A and B

A very common bug is checking only whether a customer bought `A` and `B`, while forgetting to explicitly exclude customers who also bought `C`.

For example, a customer with purchases `{A, B, C}` should not qualify. The implementation handles this correctly using the condition:

```
"C" not in products
```

or in SQL:

```
SUM(product_name = 'C') = 0
```

This guarantees exclusion whenever `C` appears even once.

### Duplicate Purchases of the Same Product

A customer may buy the same product multiple times. A naive implementation might incorrectly count duplicates or require exactly one occurrence.

For example:

| customer_id | product |
| --- | --- |
| 1 | A |
| 1 | A |
| 1 | B |

This customer should still qualify. The Python implementation uses a set, which automatically removes duplicates. The SQL implementation uses aggregate counts greater than zero, which also handles duplicates correctly.

### Customers With No Orders

Some customers may exist in the `Customers` table but never appear in `Orders`.

A naive implementation might accidentally include them or raise lookup errors. The Python solution safely handles this because missing customers receive an empty set from the `defaultdict`. Since neither `A` nor `B` exists in the set, they are naturally excluded.

The SQL solution also excludes them because the inner join requires matching order rows.
