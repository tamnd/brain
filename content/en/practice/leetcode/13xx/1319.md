**Difficulty:** 🟡 Medium  
**Topics:** Depth-First Search, Breadth-First Search, Union-Find, Graph Theory

## Solution

## Problem Understanding

The problem describes a network of `n` computers connected by ethernet cables. Each cable directly connects two computers, and because the network connections are undirected, communication can happen both ways.

The input `connections` is a list of edges in a graph. Each computer is a node, and each cable is an edge between two nodes. The task is to determine the minimum number of cable relocation operations needed to make the entire network fully connected.

A relocation operation works like this:

1. Remove an existing cable between two already connected computers.
2. Reattach that cable between two disconnected components.

The important observation is that some cables may be redundant. A redundant cable is one that connects two computers that are already connected through another path. Such cables can be reused elsewhere.

The output should be:

- The minimum number of operations needed to connect all computers.
- `-1` if it is impossible.

A graph with `n` nodes needs at least `n - 1` edges to become fully connected. This is a fundamental property of connected undirected graphs. Therefore, if the number of cables is less than `n - 1`, the answer is immediately impossible.

The constraints are large:

- `n` can be up to `100000`
- `connections.length` can also be up to `100000`

Because of this, an efficient near linear solution is required. Quadratic algorithms would time out.

Several edge cases are important:

- A network that is already connected should return `0`.
- A network with too few cables should immediately return `-1`.
- Multiple disconnected groups may require several reconnections.
- A single computer with no connections is already connected by definition.

## Approaches

### Brute Force Approach

A brute force solution would repeatedly search for disconnected components, then attempt to simulate removing every possible redundant cable and reconnecting components one at a time.

One possible brute force strategy is:

1. Build the graph.
2. Check whether the graph is connected.
3. If not connected, try removing each edge and reconnecting different components.
4. Recompute connectivity after every modification.

This approach is correct because it explicitly explores ways to reconnect the graph. However, it is extremely inefficient because connectivity checks themselves require graph traversals, and repeating them for many edges creates very large time complexity.

With up to `100000` nodes and edges, repeated graph reconstruction and traversal is not practical.

### Key Insight

The key insight is that we do not actually need to simulate cable movements.

Suppose the graph has:

- `k` connected components

To connect all components into one connected graph, we need exactly:

```
k - 1
```

connections between components.

The second insight is that a connected graph with `m` nodes only needs `m - 1` edges. Any extra edges are redundant and can be repurposed.

Therefore:

- If total edges `< n - 1`, return `-1`
- Otherwise:

- Count connected components
- Answer is `components - 1`

We can efficiently count connected components using either:

- DFS/BFS
- Union-Find

Union-Find is especially efficient for dynamic connectivity problems.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(E × (V + E)) | O(V + E) | Repeatedly recomputes connectivity after modifications |
| Optimal Union-Find | O(V + E α(V)) | O(V) | Efficiently tracks connected components |

Here:

- `V = n`
- `E = len(connections)`
- `α(V)` is the inverse Ackermann function, which is nearly constant.

## Algorithm Walkthrough

### Optimal Union-Find Algorithm

1. First, check whether the number of cables is sufficient.

A connected graph with `n` nodes requires at least `n - 1` edges. If:

```
len(connections) < n - 1
```

then it is impossible to connect the network, so return `-1`.
2. Initialize a Union-Find data structure.

Each computer initially belongs to its own set. We maintain:

- `parent[i]`, the representative of node `i`
- `rank[i]` or `size[i]`, used to optimize unions
3. Start with `components = n`.

Initially every computer is isolated, so there are `n` connected components.
4. Process each connection `[a, b]`.

For every edge:

- Find the root parent of `a`
- Find the root parent of `b`

If the roots are different:

- Merge the two sets
- Decrease `components` by 1

If the roots are already equal:

- The edge is redundant
- We do not need to count it explicitly because the earlier edge count check guarantees enough cables exist.
5. After processing all edges, return:

```
components - 1
```

because connecting `k` disconnected groups requires exactly `k - 1` operations.

### Why It Works

Union-Find groups together all computers that are already connected directly or indirectly. After processing every cable, each disjoint set represents one connected component.

If there are `k` components, connecting them into a single network always requires exactly `k - 1` edges. Since we already verified that the graph contains enough total cables, the redundant cables can always be repurposed to create those missing connections.

Therefore, the algorithm always returns the minimum number of required operations.

## Python Solution

```python
from typing import List

class Solution:
    def makeConnected(self, n: int, connections: List[List[int]]) -> int:
        if len(connections) < n - 1:
            return -1

        parent = list(range(n))
        rank = [0] * n

        def find(node: int) -> int:
            if parent[node] != node:
                parent[node] = find(parent[node])
            return parent[node]

        def union(a: int, b: int) -> bool:
            root_a = find(a)
            root_b = find(b)

            if root_a == root_b:
                return False

            if rank[root_a] < rank[root_b]:
                parent[root_a] = root_b
            elif rank[root_a] > rank[root_b]:
                parent[root_b] = root_a
            else:
                parent[root_b] = root_a
                rank[root_a] += 1

            return True

        components = n

        for a, b in connections:
            if union(a, b):
                components -= 1

        return components - 1
```

The implementation starts with the important feasibility check. If there are fewer than `n - 1` cables, no amount of rearrangement can fully connect the graph.

The `parent` array stores the representative element for each connected component. Initially every node is its own parent.

The `find` function uses path compression. Whenever we search for the root of a node, we directly connect the node to its root afterward. This significantly speeds up future operations.

The `union` function merges two components if they are different. Union by rank keeps the trees shallow, improving performance.

The variable `components` tracks how many disconnected groups remain. Every successful union reduces the number of components by one.

Finally, the answer is `components - 1`, which is the minimum number of cables needed to connect all remaining groups.

## Go Solution

```go
package main

func makeConnected(n int, connections [][]int) int {
	if len(connections) < n-1 {
		return -1
	}

	parent := make([]int, n)
	rank := make([]int, n)

	for i := 0; i < n; i++ {
		parent[i] = i
	}

	var find func(int) int

	find = func(node int) int {
		if parent[node] != node {
			parent[node] = find(parent[node])
		}
		return parent[node]
	}

	union := func(a int, b int) bool {
		rootA := find(a)
		rootB := find(b)

		if rootA == rootB {
			return false
		}

		if rank[rootA] < rank[rootB] {
			parent[rootA] = rootB
		} else if rank[rootA] > rank[rootB] {
			parent[rootB] = rootA
		} else {
			parent[rootB] = rootA
			rank[rootA]++
		}

		return true
	}

	components := n

	for _, connection := range connections {
		a := connection[0]
		b := connection[1]

		if union(a, b) {
			components--
		}
	}

	return components - 1
}
```

The Go implementation follows the same algorithmic structure as the Python version.

Slices are used for the `parent` and `rank` arrays. Go does not support nested functions with implicit recursion as naturally as Python, so the recursive `find` function is declared as a variable before assignment.

Go integers are sufficient for this problem because all values remain well within 32 bit integer limits.

## Edge Cases

### Insufficient Number of Cables

A very common mistake is attempting to process the graph before checking whether enough edges even exist.

For example:

```
n = 6
connections = 4 edges
```

A connected graph with six nodes requires at least five edges. No rearrangement strategy can overcome this limitation.

The implementation handles this immediately with:

```
if len(connections) < n - 1:
    return -1
```

This avoids unnecessary computation and guarantees correctness.

### Already Connected Network

Another important case is when the network is already fully connected.

Example:

```
n = 5
connections = [[0,1],[1,2],[2,3],[3,4]]
```

The graph already forms a connected tree. The algorithm correctly counts only one component, producing:

```
1 - 1 = 0
```

No extra operations are needed.

### Redundant Cycles

Graphs containing cycles can confuse naive implementations because some edges are unnecessary.

Example:

```
[[0,1],[1,2],[2,0]]
```

The third edge creates a cycle and is redundant.

Union-Find naturally detects this because both endpoints already share the same root. The algorithm therefore avoids incorrectly reducing the component count.

### Single Computer

A graph with only one computer is technically already connected.

Example:

```
n = 1
connections = []
```

The algorithm initializes `components = 1`, and the final result becomes:

```
1 - 1 = 0
```

which is correct.
