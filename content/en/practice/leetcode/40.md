## Solution

## Problem Understanding

The problem asks us to find **all unique combinations** of numbers from the `candidates` array such that their sum equals `target`.

There are two important constraints that make this problem different from the classic Combination Sum problem:

1. **Each number may only be used once.**

Unlike LeetCode 39 (Combination Sum), where a candidate can be reused unlimited times, here every element in the input array can only contribute to a combination at most once.
2. **Duplicate combinations are not allowed.**

Even if the input contains repeated values, the result cannot contain duplicate combinations.

For example, consider:

```
candidates = [10,1,2,7,6,1,5]
target = 8
```

The input contains two `1`s. Since each element can only be used once, a valid combination like `[1,1,6]` is allowed because it uses both occurrences of `1`. However, combinations that differ only in ordering are considered duplicates. For instance:

```
[1,7]
[7,1]
```

These represent the same combination, so only one should appear in the answer.

The input consists of:

- `candidates`, an array of positive integers
- `target`, a positive integer sum we want to achieve

The expected output is a list of unique combinations where:

- Every chosen number comes from `candidates`
- Every number is used at most once
- The sum equals `target`
- No duplicate combinations exist

The constraints provide important hints about the problem size:

- `candidates.length <= 100`
- `candidates[i] <= 50`
- `target <= 30`

Even though the candidate list can contain up to 100 numbers, the target is relatively small. This strongly suggests that a carefully pruned backtracking approach can work efficiently. Since combinations must be explored, brute force is possible but inefficient.

Several edge cases can trip up a naive implementation. Duplicate numbers are the biggest challenge because they can easily produce duplicate combinations if handled incorrectly. Cases where no valid combination exists must return an empty list. Inputs containing many repeated values, such as `[1,1,1,1]`, require careful duplicate skipping logic to avoid redundant work.

## Approaches

### Brute Force Approach

A brute force solution would generate **every possible subset** of `candidates` and check whether its sum equals `target`.

Since every element can either be included or excluded, there are `2^n` possible subsets for an input of size `n`.

For each subset:

1. Decide whether to include each element.
2. Compute the total sum.
3. If the sum equals `target`, store the combination.
4. Use a set to remove duplicates afterward.

This approach is correct because it exhaustively examines every possible valid subset. Since every combination is considered, no valid answer can be missed.

However, it is inefficient. With up to 100 elements, `2^100` possibilities become computationally infeasible. Additionally, duplicate handling introduces extra overhead because many equivalent combinations would be generated repeatedly.

### Optimal Approach, Sorting + Backtracking with Duplicate Skipping

The key observation is that we do **not** need to generate every subset blindly.

Since:

- All values are positive
- We only care about combinations summing to `target`
- Duplicate combinations are forbidden

We can sort the array first and then use **backtracking with pruning**.

Sorting helps in two important ways:

1. **Duplicate detection becomes easy.**

After sorting, identical values appear next to each other. During recursion, we can skip duplicate values at the same recursion depth to avoid generating identical combinations.
2. **Pruning becomes possible.**

Because the array is sorted, if a number exceeds the remaining target, all later numbers will also be too large. We can stop exploring that branch immediately.

The backtracking process works by:

- Trying candidates one by one
- Building a temporary combination
- Recursively exploring further choices
- Undoing the choice after recursion completes

To ensure each number is used only once, recursive calls always move to the **next index**.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2ⁿ × n) | O(2ⁿ × n) | Generates every subset and removes duplicates afterward |
| Optimal | O(2ⁿ) worst case | O(n) recursion stack, excluding output | Uses sorting, pruning, and duplicate skipping |

## Algorithm Walkthrough

1. **Sort the candidates array.**

Sorting groups duplicate values together and enables early stopping when numbers exceed the remaining target.
2. **Initialize the result list.**

This will store all valid combinations.
3. **Start backtracking from index `0`.**

The recursive function keeps track of:

- The current index to explore
- The remaining target
- The current combination being built
4. **Check if remaining target becomes `0`.**

If it does, we found a valid combination. Add a copy of the current path to the result.
5. **Iterate through candidates starting at the current index.**

Each recursive call explores possible next elements.
6. **Skip duplicates at the same recursion level.**

If:

```
i > start AND candidates[i] == candidates[i - 1]
```

we skip this element.

This rule is subtle but extremely important.

We only skip duplicates **at the same recursion depth**. We still allow duplicate numbers in a combination if they come from different indices.

For example:

```
[1,1,6]
```

is valid because the two `1`s originate from different positions.
7. **Prune impossible paths.**

Since the array is sorted:

```
if candidates[i] > remaining_target
```

we can stop immediately because all later numbers will also be too large.
8. **Choose the current number.**

Add it to the current combination.
9. **Recurse using `i + 1`.**

This ensures each element is used only once.
10. **Backtrack.**

Remove the last number so the next branch starts cleanly.

## Python Solution

```
from typing import List

class Solution:
    def combinationSum2(self, candidates: List[int], target: int) -> List[List[int]]:
        candidates.sort()
        result = []

        def backtrack(start_index: int, remaining_target: int, current_combination: List[int]) -> None:
            if remaining_target == 0:
                result.append(current_combination[:])
                return

            for index in range(start_index, len(candidates)):
                current_number = candidates[index]

                # Skip duplicates at the same recursion level
                if index > start_index and current_number == candidates[index - 1]:
                    continue

                # Prune search space
                if current_number > remaining_target:
                    break

                current_combination.append(current_number)

                backtrack(
                    index + 1,
                    remaining_target - current_number,
                    current_combination
                )

                current_combination.pop()

        backtrack(0, target, [])
        return result
```

The implementation begins by sorting `candidates`, which is essential for duplicate handling and pruning. Without sorting, identical numbers would not be adjacent, making duplicate detection much harder.

The `backtrack` function manages the recursive exploration. It accepts three parameters:

- `start_index`, which prevents reusing earlier elements
- `remaining_target`, which tracks how much sum is still needed
- `current_combination`, which stores the numbers chosen so far

Whenever `remaining_target` becomes zero, a valid combination has been found. Since lists are mutable, we append a copy using `[:]`.

The duplicate-skipping condition ensures uniqueness:

```
if index > start_index and current_number == candidates[index - 1]:
```

This prevents generating repeated combinations from identical numbers at the same recursion depth.

Pruning happens here:

```
if current_number > remaining_target:
    break
```

Because the array is sorted, no future number can fit once the current one becomes too large.

Finally, after recursion finishes, we remove the last added number with `pop()` to restore the previous state for the next branch.

## Go Solution

```
package main

import "sort"

func combinationSum2(candidates []int, target int) [][]int {
	sort.Ints(candidates)

	result := [][]int{}
	currentCombination := []int{}

	var backtrack func(startIndex int, remainingTarget int)

	backtrack = func(startIndex int, remainingTarget int) {
		if remainingTarget == 0 {
			combinationCopy := make([]int, len(currentCombination))
			copy(combinationCopy, currentCombination)
			result = append(result, combinationCopy)
			return
		}

		for index := startIndex; index < len(candidates); index++ {
			currentNumber := candidates[index]

			// Skip duplicates at the same recursion level
			if index > startIndex && currentNumber == candidates[index-1] {
				continue
			}

			// Prune search space
			if currentNumber > remainingTarget {
				break
			}

			currentCombination = append(currentCombination, currentNumber)

			backtrack(index+1, remainingTarget-currentNumber)

			currentCombination = currentCombination[:len(currentCombination)-1]
		}
	}

	backtrack(0, target)
	return result
}
```

The Go solution follows the exact same algorithmic structure as the Python version.

One important Go-specific detail is slice copying. Since slices reference underlying memory, we must explicitly create a new slice before appending to `result`:

```
combinationCopy := make([]int, len(currentCombination))
copy(combinationCopy, currentCombination)
```

Without this step, later modifications during backtracking would overwrite previously stored results.

Go slices are also resized manually during backtracking:

```
currentCombination = currentCombination[:len(currentCombination)-1]
```

This removes the last chosen element and restores the previous recursion state.

Integer overflow is not a concern because values are small and `target <= 30`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(2ⁿ) | In the worst case, backtracking explores subsets of candidates |
| Space | O(n) | Recursion depth and temporary combination storage |

The time complexity is exponential because the algorithm explores combinations recursively. In the worst case, many branches are explored, similar to subset generation.

However, practical performance is much better than brute force because of two optimizations: duplicate skipping and pruning. Sorting allows early termination when numbers exceed the remaining target, significantly reducing the search space.

The space complexity is `O(n)` excluding output storage because recursion depth can grow to at most the number of candidates. The temporary combination list also grows proportionally to recursion depth.

## Edge Cases

### Input Contains Many Duplicate Values

Consider:

```
candidates = [1,1,1,1]
target = 2
```

A naive implementation may generate duplicate answers repeatedly. Since the input contains repeated values, combinations like `[1,1]` can be formed through multiple index paths.

The duplicate-skipping condition ensures only one unique combination is generated.

### No Valid Combination Exists

Consider:

```
candidates = [4,5,6]
target = 3
```

No number can contribute to the target.

Because the array is sorted, the algorithm immediately stops when the first candidate exceeds the remaining target. The result remains empty:

```
[]
```

### Single Element Equals Target

Consider:

```
candidates = [8]
target = 8
```

The algorithm correctly selects the element, reaches remaining target `0`, and returns:

```
[[8]]
```

Since each element may only be used once, the recursion naturally stops after selecting it.

### Candidates Larger Than Remaining Target

Consider:

```
candidates = [1,2,50]
target = 3
```

After choosing `[1,2]`, the remaining target becomes `0`.

When exploring other branches, `50` immediately exceeds the remaining target, triggering pruning:

```
if current_number > remaining_target:
    break
```

This avoids unnecessary recursive exploration and improves efficiency.
