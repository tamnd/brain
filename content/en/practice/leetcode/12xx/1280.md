## Solution

## Problem Understanding

This problem asks us to generate a complete report showing how many times every student attended every subject examination.

We are given three database tables:

- `Students` contains the list of students.
- `Subjects` contains the list of subjects.
- `Examinations` records every exam attendance event.

The important detail is that the `Examinations` table may contain duplicate rows. That means a student can appear multiple times for the same subject, and every row represents one attendance. Therefore, we are not checking whether a student attended a subject at least once, instead, we must count the total number of attendance records.

The output must contain every possible combination of:

- each student
- each subject

For every `(student, subject)` pair, we compute how many matching rows exist in the `Examinations` table.

This means that even if a student never attended a particular subject, that combination must still appear in the output with `attended_exams = 0`.

Conceptually, the problem is asking us to:

1. Generate all student-subject pairs.
2. Count matching examination rows for each pair.
3. Return the results ordered by `student_id` and `subject_name`.

The constraints are relatively small because this is an Easy database problem, but the important observation is correctness rather than raw optimization. The key challenge is ensuring that missing attendance records still produce rows with count `0`.

Several edge cases are important:

- A student may never attend any exam.
- A subject may never appear in `Examinations`.
- Multiple identical attendance rows may exist and must all be counted.
- The `Examinations` table can be empty.
- Every student-subject pair must still appear exactly once in the output.

A naive implementation often fails because it only aggregates existing rows from `Examinations`, which omits zero-attendance combinations.

## Approaches

### Brute Force Approach

The brute force solution would iterate through every student and every subject, then scan the entire `Examinations` table to count matching rows.

For each `(student, subject)` pair:

- traverse all examination records
- count rows where both `student_id` and `subject_name` match

This approach is correct because it explicitly checks every possible attendance record for every pair.

However, it is inefficient because the `Examinations` table is repeatedly scanned. If:

- `S` is the number of students
- `T` is the number of subjects
- `E` is the number of examination rows

then the complexity becomes `O(S * T * E)`.

This repeated scanning is unnecessary because databases are optimized for grouped aggregation and joins.

### Optimal Approach

The key insight is that we can first generate all possible student-subject combinations using a `CROSS JOIN`.

A `CROSS JOIN` creates the Cartesian product of the two tables:

- every student paired with every subject

Once all pairs exist, we can attach examination records using a `LEFT JOIN`.

The `LEFT JOIN` is critical because it preserves combinations that have no matching examination rows. Those unmatched rows will produce `NULL`, which allows `COUNT()` to return `0`.

Finally, we group by:

- `student_id`
- `student_name`
- `subject_name`

and count the matching examination rows.

This avoids rescanning the entire examinations table repeatedly and uses relational database operations efficiently.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(S * T * E) | O(1) | Repeatedly scans the examinations table |
| Optimal | O(S * T + E) | O(S * T) | Uses CROSS JOIN and aggregation efficiently |

## Algorithm Walkthrough

1. Start with the `Students` table.

We need every student to appear in the final result, even those with no examinations.
2. Perform a `CROSS JOIN` with the `Subjects` table.

This generates every possible `(student, subject)` pair. If there are 4 students and 3 subjects, we create 12 rows.
3. Perform a `LEFT JOIN` with the `Examinations` table.

The join condition is:

```
student.student_id = examinations.student_id
AND subject.subject_name = examinations.subject_name
```

This attaches matching examination rows to each pair.
4. Group the results.

We group by:

- `student_id`
- `student_name`
- `subject_name`

Grouping allows us to aggregate all matching examination rows together.
5. Count examination rows.

We use:

```
COUNT(e.subject_name)
```

Since `COUNT(column)` ignores `NULL`, combinations with no matching examinations automatically produce `0`.
6. Sort the output.

The problem requires ordering by:

- `student_id`
- `subject_name`

### Why it works

The algorithm works because the `CROSS JOIN` guarantees that every possible student-subject pair exists before counting begins. The `LEFT JOIN` preserves pairs with no examination records, and aggregation correctly counts all matching rows, including duplicates. Since every valid combination is generated exactly once and all matching attendance rows are counted, the result is complete and correct.

## Python Solution

```python
# Write your MySQL query statement below

SELECT
    s.student_id,
    s.student_name,
    sub.subject_name,
    COUNT(e.subject_name) AS attended_exams
FROM Students s
CROSS JOIN Subjects sub
LEFT JOIN Examinations e
    ON s.student_id = e.student_id
    AND sub.subject_name = e.subject_name
GROUP BY
    s.student_id,
    s.student_name,
    sub.subject_name
ORDER BY
    s.student_id,
    sub.subject_name;
```

The query begins by generating all possible student-subject combinations using `CROSS JOIN`. This ensures no valid output row is missing.

Next, the `LEFT JOIN` connects examination records to each pair. If a student never attended a subject, the joined examination columns become `NULL`.

The `GROUP BY` clause collects all matching examination rows together for each pair.

Finally, `COUNT(e.subject_name)` counts only non-null matches. Since `COUNT()` ignores `NULL`, combinations without examinations correctly return `0`.

The `ORDER BY` clause guarantees the output format required by the problem statement.

## Edge Cases

One important edge case occurs when a student never attends any exam. A naive solution that aggregates only the `Examinations` table would completely omit that student from the result. The `CROSS JOIN` combined with `LEFT JOIN` guarantees that the student still appears for every subject with attendance count `0`.

Another important edge case is duplicate examination rows. Since the problem explicitly states that the `Examinations` table may contain duplicates, using `COUNT(DISTINCT ...)` would produce incorrect results. Our implementation uses a normal `COUNT()` so every attendance row contributes correctly to the final total.

A third edge case happens when the `Examinations` table is empty. In this situation, the correct output still contains every student-subject combination. Because the solution begins with the Cartesian product rather than the examinations table, all required rows still exist and their counts correctly evaluate to `0`.

A final subtle edge case involves subjects that no student attended. These subjects still must appear for every student. Since the algorithm generates all combinations first, subjects with no examination rows remain present in the final grouped result.
