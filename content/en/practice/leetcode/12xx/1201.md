# LeetCode 1201 - Ugly Number III

## Problem Understanding

The problem defines an ugly number as any positive integer that is divisible by at least one of the three integers `a`, `b`, or `c`.

We are given four integers:

- `n`, the position of the ugly number we want to find
- `a`, `b`, and `c`, the divisors that define whether a number is ugly

The task is to return the `n`th positive integer that is divisible by `a`, `b`, or `c`.

For example, if `a = 2`, `b = 3`, and `c = 5`, then the ugly numbers are:

`2, 3, 4, 5, 6, 8, 9, 10, ...`

The third ugly number is `4`, so the answer is `4`.

The important detail is that a number only needs to be divisible by one of the three values. If a number is divisible by multiple values, it should still only be counted once. For example, `6` is divisible by both `2` and `3`, but it is still a single ugly number.

The constraints are extremely important:

- `n`, `a`, `b`, and `c` can be as large as `10^9`
- `a * b * c` can be as large as `10^18`

These limits immediately tell us that generating ugly numbers one by one is not feasible. Even iterating through all numbers up to the answer would be too slow in the worst case.

The problem also guarantees that the answer fits within the range `[1, 2 * 10^9]`, which is useful because it gives us a safe upper bound for binary search.

Several edge cases are important:

- Some of `a`, `b`, and `c` may be equal
- One divisor may be a multiple of another
- Large values may cause overflow if least common multiples are computed carelessly
- Numbers divisible by multiple divisors must not be double counted

These cases make inclusion-exclusion and careful least common multiple computation essential.

## Approaches

### Brute Force Approach

A straightforward solution would iterate through positive integers one by one and count how many are divisible by `a`, `b`, or `c`.

For each number `x`:

- Check whether `x % a == 0`
- Or `x % b == 0`
- Or `x % c == 0`

If any condition is true, increment the ugly number count.

Once the count reaches `n`, return the current number.

This works because it directly follows the definition of an ugly number. Every valid number is counted in increasing order, so the `n`th one found is the correct answer.

However, this approach is far too slow. The answer itself may be close to `2 * 10^9`, so iterating through billions of numbers is infeasible.

### Optimal Approach

The key observation is that instead of generating ugly numbers directly, we can ask a different question:

> For a given number `x`, how many ugly numbers are less than or equal to `x`?

If we can efficiently compute this count, then we can binary search for the smallest number whose count is at least `n`.

To count how many numbers up to `x` are divisible by `a`, `b`, or `c`, we use the inclusion-exclusion principle.

The count is:

$$\left\lfloor \frac{x}{a} \right\rfloor +
\left\lfloor \frac{x}{b} \right\rfloor +
\left\lfloor \frac{x}{c} \right\rfloor$$

But numbers divisible by both `a` and `b` are counted twice, so we subtract:

$$\left\lfloor \frac{x}{\text{lcm}(a,b)} \right\rfloor$$

Similarly for `(a,c)` and `(b,c)`.

Then numbers divisible by all three were subtracted too many times, so we add back:

$$\left\lfloor \frac{x}{\text{lcm}(a,b,c)} \right\rfloor$$

This gives an efficient counting function.

Since the count increases monotonically as `x` increases, binary search works perfectly.

| Optimal | O(log(2 × 10^9)) | O(1) | Uses binary search with inclusion-exclusion counting |

## Algorithm Walkthrough

1. Compute the least common multiples needed for inclusion-exclusion.

We need:

- `lcm(a, b)`
- `lcm(a, c)`
- `lcm(b, c)`
- `lcm(a, lcm(b, c))`

These values allow us to correctly count overlaps between divisibility groups.
2. Define a helper function `count(x)`.

This function returns how many integers from `1` to `x` are divisible by at least one of `a`, `b`, or `c`.

Using inclusion-exclusion:

$$\text{count}(x)
=
\frac{x}{a}
+
\frac{x}{b}
+
\frac{x}{c}
-
\frac{x}{\text{lcm}(a,b)}
-
\frac{x}{\text{lcm}(a,c)}
-
\frac{x}{\text{lcm}(b,c)}
+
\frac{x}{\text{lcm}(a,b,c)}$$

1. Initialize the binary search range.

The smallest possible answer is `1`.

The largest possible answer is guaranteed to be at most `2 * 10^9`.
2. Perform binary search.

At each step:

- Compute `mid`
- Evaluate `count(mid)`

If `count(mid) >= n`, then the answer could be `mid` or smaller, so move the right boundary leftward.

Otherwise, there are not enough ugly numbers yet, so move the left boundary rightward.
3. Continue until the search converges.

When `left == right`, we have found the smallest number whose count is at least `n`.

That number is the `n`th ugly number.

### Why it works

The counting function correctly computes how many integers are divisible by at least one of `a`, `b`, or `c` using inclusion-exclusion. This prevents double counting overlaps.

The count function is monotonic, meaning it never decreases as `x` increases. Therefore, binary search can safely find the smallest `x` where the count reaches `n`.

That smallest `x` must be the `n`th ugly number.

## Python Solution

```python
from math import gcd

class Solution:
    def nthUglyNumber(self, n: int, a: int, b: int, c: int) -> int:
        
        def lcm(x: int, y: int) -> int:
            return x * y // gcd(x, y)

        ab = lcm(a, b)
        ac = lcm(a, c)
        bc = lcm(b, c)
        abc = lcm(a, bc)

        def count_ugly(limit: int) -> int:
            return (
                limit // a
                + limit // b
                + limit // c
                - limit // ab
                - limit // ac
                - limit // bc
                + limit // abc
            )

        left = 1
        right = 2 * 10**9

        while left < right:
            mid = (left + right) // 2

            if count_ugly(mid) >= n:
                right = mid
            else:
                left = mid + 1

        return left
```

The implementation starts by defining an `lcm` helper function. Least common multiples are necessary because the inclusion-exclusion principle depends on identifying overlaps between divisibility groups.

The code then precomputes all pairwise LCM values and the combined LCM for all three divisors. Precomputing them avoids repeated calculations during binary search.

The `count_ugly` function applies the inclusion-exclusion formula directly. It counts numbers divisible by each divisor, subtracts overlaps, and adds back triple overlaps.

The binary search maintains a search space containing the answer. At each step, the midpoint is tested using `count_ugly`.

If the midpoint already contains at least `n` ugly numbers, the answer may be smaller, so the right boundary moves left.

Otherwise, more numbers are needed, so the left boundary moves right.

Eventually the search converges to the smallest valid number, which is the `n`th ugly number.

## Go Solution

```go
package main

func gcd(x int64, y int64) int64 {
	for y != 0 {
		x, y = y, x%y
	}
	return x
}

func lcm(x int64, y int64) int64 {
	return x / gcd(x, y) * y
}

func nthUglyNumber(n int, a int, b int, c int) int {
	A := int64(a)
	B := int64(b)
	C := int64(c)

	ab := lcm(A, B)
	ac := lcm(A, C)
	bc := lcm(B, C)
	abc := lcm(A, bc)

	countUgly := func(limit int64) int64 {
		return limit/A +
			limit/B +
			limit/C -
			limit/ab -
			limit/ac -
			limit/bc +
			limit/abc
	}

	left := int64(1)
	right := int64(2 * 1000000000)

	for left < right {
		mid := left + (right-left)/2

		if countUgly(mid) >= int64(n) {
			right = mid
		} else {
			left = mid + 1
		}
	}

	return int(left)
}
```

The Go implementation follows the same logic as the Python version, but integer overflow requires additional care.

All arithmetic involving least common multiples and counting uses `int64` rather than `int`. This is necessary because intermediate LCM values can exceed the standard 32-bit integer range.

The binary search logic remains identical. The midpoint calculation uses:

```go
mid := left + (right-left)/2
```

This pattern avoids overflow in languages with fixed-width integers.

## Edge Cases

### All Divisors Are Equal

If `a = b = c = 2`, every ugly number is simply a multiple of `2`.

A buggy inclusion-exclusion implementation might overcount because every overlap is identical. The formula still works correctly because all pairwise and triple LCMs are also `2`, so duplicate counting cancels out properly.

The implementation handles this naturally through the inclusion-exclusion formula.

### One Divisor Is a Multiple of Another

Consider:

```
a = 2, b = 4, c = 8
```

Every multiple of `4` and `8` is already divisible by `2`.

A naive counting method might incorrectly treat these as separate groups and double count many numbers.

Using least common multiples ensures overlaps are subtracted correctly, so every ugly number is counted exactly once.

### Very Large Inputs

The constraints allow values up to `10^9`, and intermediate LCM computations may reach `10^18`.

In languages with fixed-width integers, multiplication before division can overflow.

The Go implementation uses `int64` everywhere for arithmetic involving LCMs and counts. This guarantees safe computation within the allowed range.

The binary search approach also ensures performance remains efficient even for the largest possible inputs.
