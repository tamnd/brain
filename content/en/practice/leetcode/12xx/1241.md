## Solution

## Problem Understanding

This problem is asking us to analyze a table of submissions that can represent either posts or comments and determine how many unique comments exist for each unique post. Each row in the `Submissions` table has a `sub_id` representing the identifier of the submission and a `parent_id` which is `NULL` if the row represents a post, and otherwise points to the `sub_id` of the post it comments on. The input can contain duplicates, meaning the same post or comment can appear multiple times, but these duplicates should be ignored in the counting.

The expected output is a table listing each unique post (`post_id`) along with the number of **unique comments** associated with that post (`number_of_comments`). Posts that have no comments should appear with a count of zero. The output must be sorted by `post_id` in ascending order. The key constraints and subtleties are: duplicates exist and should be handled, comments may point to non-existent posts and should be ignored, and the input size could be moderately large because it is a database problem.

Important edge cases include comments pointing to posts that are not in the table, duplicate comments for the same post, and duplicate posts themselves.

## Approaches

The brute-force approach would involve iterating through the table and, for each post, scanning all rows to count how many comments point to that post. This is correct but inefficient for large tables since it involves repeated scanning of the table and does not leverage SQL’s aggregation capabilities effectively.

The optimal approach relies on SQL set operations and aggregation. By first eliminating duplicates using `DISTINCT`, we can ensure that each post and comment is counted only once. Then, using a `LEFT JOIN` from the unique posts to the unique comments, we can count how many comments each post has efficiently. This approach is much faster because it leverages the database engine’s optimized join and aggregation operations.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^2) | O(1) | Scan for each post and count comments, inefficient for large tables |
| Optimal | O(n) | O(n) | Use DISTINCT and LEFT JOIN with GROUP BY for unique posts and comments |

## Algorithm Walkthrough

1. Extract all unique posts from the `Submissions` table by selecting rows where `parent_id` is `NULL` and applying `DISTINCT` on `sub_id`. This ensures we only consider one copy of each post.
2. Extract all unique comments by selecting rows where `parent_id` is not `NULL` and applying `DISTINCT` on the combination `(sub_id, parent_id)` to remove duplicate comments.
3. Perform a `LEFT JOIN` from the unique posts to the unique comments on the condition that the post’s `sub_id` matches the comment’s `parent_id`. This ensures every post is included, even if it has no comments.
4. Group the results by `post_id` and count the number of joined comment rows for each post. If there are no matching comments, the count defaults to zero.
5. Order the final result by `post_id` in ascending order.

**Why it works:** The algorithm works because the `DISTINCT` operations remove duplicates, the `LEFT JOIN` ensures all posts are included regardless of whether they have comments, and the `GROUP BY` with `COUNT` correctly aggregates the number of unique comments per post.

## Python Solution

```python
# Since this is a database problem, we provide a SQL query as a string for execution
def number_of_comments_per_post() -> str:
    return """
    WITH unique_posts AS (
        SELECT DISTINCT sub_id AS post_id
        FROM Submissions
        WHERE parent_id IS NULL
    ),
    unique_comments AS (
        SELECT DISTINCT sub_id, parent_id
        FROM Submissions
        WHERE parent_id IS NOT NULL
    )
    SELECT p.post_id,
           COUNT(c.sub_id) AS number_of_comments
    FROM unique_posts p
    LEFT JOIN unique_comments c
           ON p.post_id = c.parent_id
    GROUP BY p.post_id
    ORDER BY p.post_id ASC;
    """
```

The implementation first isolates the posts and comments uniquely. The `LEFT JOIN` connects posts to comments, ensuring even posts without comments are included. `COUNT(c.sub_id)` counts only matching comments, and `GROUP BY` aggregates counts per post. The final `ORDER BY` ensures ascending order as required.

- Post 1 → comments `(3,1), (4,1), (9,1)` → count 3

1       | 3

## Edge Cases

One important edge case is a comment pointing to a post that does not exist in the table. This should be ignored because only posts present in the table should be included in the output. Our `LEFT JOIN` ensures that only posts in `unique_posts` are considered, so such orphaned comments are automatically excluded.

Another edge case is multiple duplicates of the same post or comment. Without `DISTINCT`, these duplicates would inflate the count of comments. Using `DISTINCT` on both posts and `(sub_id, parent_id)` pairs ensures that duplicates do not affect the count.

A third edge case is posts with no comments at all. A naive implementation that only counts comments might exclude such posts. Using a `LEFT JOIN` from posts to comments guarantees that every post appears in the output, with `COUNT(c.sub_id)` naturally returning zero for posts with no comments.
