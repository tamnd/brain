## Solution

## Problem Understanding

The problem gives us three integer arrays, `arr1`, `arr2`, and `arr3`. Each array is already sorted in strictly increasing order. "Strictly increasing" means there are no duplicate values inside the same array, and every next value is larger than the previous one.

Our task is to return a new sorted array containing only the integers that appear in all three arrays simultaneously.

In other words, we are looking for the intersection of the three arrays. A number should only be included in the result if it exists in `arr1`, `arr2`, and `arr3`.

For example, if a value appears in the first two arrays but not the third, it should not be included in the output.

The constraints are relatively small:

- Each array has length between `1` and `1000`
- Values range from `1` to `2000`
- Arrays are sorted
- Arrays contain unique values internally

The sorted property is the most important observation. Since the arrays are already ordered, we can process them efficiently using pointer-based traversal instead of repeatedly searching for elements.

The fact that the arrays contain unique elements also simplifies the implementation because we never need to worry about duplicates inside a single array.

Several edge cases are important to consider:

- The arrays may have no common elements at all
- Every element may be common across all arrays
- Arrays may have completely different ranges
- One array may be much shorter than the others
- The common elements may appear only near the end of the arrays

A naive implementation might repeatedly search entire arrays for matches, leading to unnecessary work. The sorted nature of the input allows us to do much better.

## Approaches

### Brute Force Approach

A straightforward approach is to iterate through every element in one array and check whether it exists in the other two arrays.

For each value in `arr1`, we can search for it in `arr2` and `arr3`. Since the arrays are sorted, we could use binary search for those lookups.

This approach works because every candidate element is explicitly verified against the other arrays. If a number exists in all three arrays, we add it to the result.

However, this method performs repeated searches. Even with binary search, we still do unnecessary repeated work compared to a linear traversal approach.

If we used plain linear searches instead of binary search, the complexity would become much worse.

### Optimal Approach

The key observation is that all three arrays are sorted.

Whenever we compare elements from the arrays:

- If all three values are equal, we found a common element
- Otherwise, the smallest value cannot possibly match later elements in the other arrays, because arrays are strictly increasing

Therefore, we can safely move the pointer pointing to the smallest value forward.

This idea leads naturally to the three-pointer technique.

We maintain one pointer for each array:

- `i` for `arr1`
- `j` for `arr2`
- `k` for `arr3`

At every step:

- If all three values match, add the value to the answer and move all pointers
- Otherwise, advance the pointer whose value is smallest

Because each pointer only moves forward once, the entire algorithm runs in linear time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × log m + n × log p) | O(1) | For each element in one array, binary search in the other two |
| Optimal | O(n + m + p) | O(1) excluding output | Uses three pointers to traverse arrays once |

## Algorithm Walkthrough

1. Initialize three pointers, `i`, `j`, and `k`, all starting at index `0`.

Each pointer tracks the current position in one of the arrays. Since the arrays are sorted, we only need to move forward.
2. Create an empty result list.

This list will store the values that appear in all three arrays.
3. Continue looping while all pointers remain inside their arrays.

The loop stops as soon as one pointer reaches the end because no further common elements can exist after that.
4. Compare the current values:

- `arr1[i]`
- `arr2[j]`
- `arr3[k]`
5. If all three values are equal:

- Add the value to the result
- Increment all three pointers

We move all pointers because the arrays are strictly increasing, so this value will never appear again.
6. Otherwise, determine which current value is smallest.

The smallest value cannot possibly match future larger values in the other arrays, so we increment only that pointer.
7. Repeat the process until one array is exhausted.
8. Return the result list.

### Why it works

The correctness relies on the sorted order of the arrays.

At every step, if one value is smaller than the others, it can never become equal later because all future elements in that array are even larger. Therefore, the only useful action is to advance the pointer holding the smallest value.

Whenever all three values match, we have definitively found a number that exists in every array.

Since every pointer only moves forward and never backward, all possible common values are checked exactly once.

## Python Solution

```python
from typing import List

class Solution:
    def arraysIntersection(
        self,
        arr1: List[int],
        arr2: List[int],
        arr3: List[int]
    ) -> List[int]:

        i = 0
        j = 0
        k = 0

        result = []

        while i < len(arr1) and j < len(arr2) and k < len(arr3):

            value1 = arr1[i]
            value2 = arr2[j]
            value3 = arr3[k]

            if value1 == value2 == value3:
                result.append(value1)
                i += 1
                j += 1
                k += 1

            else:
                minimum_value = min(value1, value2, value3)

                if value1 == minimum_value:
                    i += 1

                elif value2 == minimum_value:
                    j += 1

                else:
                    k += 1

        return result
```

The implementation begins by initializing three pointers, one for each array. These pointers allow us to traverse all arrays simultaneously.

The `while` loop continues as long as all pointers remain within bounds. At each iteration, we retrieve the current values from all three arrays.

If the values are identical, we append the value to the result list because it appears in all arrays. Since arrays are strictly increasing, we immediately move all three pointers forward.

If the values differ, we compute the minimum among them. The pointer corresponding to the smallest value is incremented because that value cannot possibly match future larger values in the other arrays.

Eventually, one pointer reaches the end of its array, which means no additional intersections are possible. At that point, we return the accumulated result.

## Go Solution

```go
func arraysIntersection(arr1 []int, arr2 []int, arr3 []int) []int {
	i := 0
	j := 0
	k := 0

	result := []int{}

	for i < len(arr1) && j < len(arr2) && k < len(arr3) {

		value1 := arr1[i]
		value2 := arr2[j]
		value3 := arr3[k]

		if value1 == value2 && value2 == value3 {
			result = append(result, value1)

			i++
			j++
			k++

		} else {

			minimumValue := value1

			if value2 < minimumValue {
				minimumValue = value2
			}

			if value3 < minimumValue {
				minimumValue = value3
			}

			if value1 == minimumValue {
				i++
			} else if value2 == minimumValue {
				j++
			} else {
				k++
			}
		}
	}

	return result
}
```

The Go implementation follows the exact same algorithmic structure as the Python version.

One small difference is that Go does not provide a built in `min` function for integers in older standard library versions, so we compute the minimum manually.

The result is stored in a slice initialized as an empty slice rather than `nil`, although either would work correctly. Since the constraints are small and values are limited, integer overflow is not a concern.

The algorithm is linear because each array is traversed exactly once. No pointer ever moves backward, and each iteration advances at least one pointer.

The auxiliary space usage is constant because we only store a few integer pointers and temporary variables. The output list itself is not counted in auxiliary space complexity.

## Edge Cases

One important edge case occurs when there are no common elements at all. In this situation, the algorithm must correctly return an empty list rather than accidentally including partial matches. The pointer advancement logic guarantees this because values are only added when all three current elements are equal.

Another important case is when all arrays are identical. A buggy implementation might accidentally skip values by advancing pointers incorrectly. Our solution handles this safely because whenever a match is found, all three pointers move together, preserving synchronization.

Single-element arrays are also important. Since the constraints allow arrays of length one, the implementation must work correctly even when the loop executes only once. The condition in the `while` loop ensures safe bounds checking before every access.

A subtle edge case involves arrays with very different sizes. For example, one array may contain many extra values between matching elements. The smallest-value advancement strategy correctly skips irrelevant values without wasting work searching unnecessarily.

Finally, common elements may appear only near the end of the arrays. Some incorrect implementations terminate too early after partial mismatches. Since our algorithm only stops when one pointer fully exhausts its array, late intersections are still discovered correctly.
