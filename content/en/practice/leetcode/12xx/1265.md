## Solution

## Problem Understanding

This problem gives us access to a linked list through a restricted interface. Unlike a normal linked list problem where we can directly read or modify node fields, here the linked list is immutable. That means we cannot change pointers, reverse the list, or directly inspect internal node values. We are only allowed to interact with each node using two API methods:

- `printValue()`, which prints the current node’s value
- `getNext()`, which returns the next node

The task is to print all node values in reverse order.

For example, if the linked list contains:

```
1 -> 2 -> 3 -> 4
```

the output must be:

```
4 3 2 1
```

The challenge comes from the restriction that the list is immutable. In a standard linked list problem, reversing the list or traversing backward might be possible, but here we cannot modify anything. Since the list is singly linked and only exposes a `getNext()` method, traversal is naturally forward-only.

The constraints tell us the list length is at most 1000 nodes. This is small enough that recursive solutions are safe in most languages, including Python for this constraint size. The follow-up questions are important because they hint at multiple possible approaches:

- Can we solve it in constant extra space?
- Can we achieve linear time with less than linear space?

These hints suggest that the straightforward stack solution is acceptable, but there are more advanced trade-offs involving recursion or block decomposition.

Several edge cases are important to recognize early. A single-node list should still print correctly because reversing a one-element sequence produces the same order. Very long lists near the maximum size could stress recursive depth in some languages. Another subtle point is that because we can only print values through `printValue()`, we cannot build a separate array of integers unless we explicitly store node references themselves.

## Approaches

The brute-force approach repeatedly traverses the linked list to locate nodes from the end toward the beginning. To print the last node, we walk from the head to the tail. To print the second-last node, we walk from the head again until the node before the tail, and so on.

This works because every traversal eventually reaches the desired node, but it is inefficient. If the list contains `n` nodes, the first pass costs `O(n)`, the second pass costs `O(n-1)`, and so forth. The total runtime becomes:

```
n + (n - 1) + (n - 2) + ... + 1 = O(n²)
```

Although the space usage is constant, the repeated scanning makes this approach unnecessarily slow.

The key insight for a better solution is that reverse order naturally suggests either recursion or a stack. Since the linked list only supports forward traversal, we can first move forward through the list while storing nodes, then process them backward.

A stack works perfectly because it is a Last-In-First-Out data structure. As we traverse from head to tail, we push nodes onto the stack. The last node visited becomes the first node popped, which exactly matches the required reverse printing order.

Another elegant observation is that recursion implicitly creates a call stack. If we recursively move to the end of the list first, then print values while the recursive calls unwind, the output automatically appears in reverse order.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Repeatedly traverses the list to find nodes from the end |
| Optimal | O(n) | O(n) | Uses recursion or a stack to reverse traversal order efficiently |

## Algorithm Walkthrough

We will use the recursive solution because it is concise, elegant, and naturally fits reverse traversal.

1. Start from the head node and recursively move forward through the linked list using `getNext()`. The recursion continues until we reach the end of the list.
2. Each recursive call pauses execution and waits for deeper calls to finish. This creates an implicit stack of function calls.
3. Once the recursion reaches the last node, there are no further nodes to visit, so recursion begins unwinding.
4. During the unwinding phase, each node calls `printValue()`. Because the deepest call returns first, the last node is printed first.
5. The recursion continues unwinding back toward the head, printing nodes in reverse order automatically.
6. When the original call finishes, all nodes have been printed in reverse sequence.

### Why it works

The correctness comes from the order in which recursive calls return. Recursive descent visits nodes from head to tail, but printing occurs after the recursive call finishes. Therefore, printing happens from tail to head. Every node is printed exactly once, and the order is exactly reversed relative to the original traversal order.

## Python Solution

```python
# """
# This is the ImmutableListNode's API interface.
# You should not implement it, or speculate about its implementation.
# """
# class ImmutableListNode:
#     def printValue(self) -> None:
#         pass
#
#     def getNext(self) -> 'ImmutableListNode':
#         pass

class Solution:
    def printLinkedListInReverse(self, head: 'ImmutableListNode') -> None:
        def dfs(node: 'ImmutableListNode') -> None:
            if not node:
                return

            dfs(node.getNext())
            node.printValue()

        dfs(head)
```

The implementation defines a helper function called `dfs`, which performs a depth-first traversal of the linked list. The helper receives the current node as input.

The base case checks whether the node is `None`. If so, recursion stops because there are no more nodes to process.

For every valid node, the algorithm first recursively visits the next node using `node.getNext()`. This ensures traversal reaches the tail before any values are printed.

After the recursive call returns, the current node prints its value. Since recursive calls unwind from the deepest level first, printing naturally occurs in reverse order.

The main method simply starts the recursion from the head node.

## Go Solution

```go
/*   Below is the interface for ImmutableListNode, which is already defined for you.
 *
 *   type ImmutableListNode struct {
 *
 *   }
 *
 *   func (this *ImmutableListNode) getNext() ImmutableListNode {
 *      // return the next node.
 *   }
 *
 *   func (this *ImmutableListNode) printValue() {
 *      // print the value of this node.
 *   }
 */

func printLinkedListInReverse(head ImmutableListNode) {
    var dfs func(ImmutableListNode)

    dfs = func(node ImmutableListNode) {
        if node == nil {
            return
        }

        dfs(node.getNext())
        node.printValue()
    }

    dfs(head)
}
```

The Go implementation follows the same recursive strategy as the Python version. One important difference is handling `nil` values explicitly because Go uses `nil` to represent the absence of a node.

The recursive helper function is declared using a function variable so that it can recursively reference itself. Aside from syntax differences, the algorithmic behavior is identical.

## Worked Examples

### Example 1

Input:

```
[1,2,3,4]
```

Linked list structure:

```
1 -> 2 -> 3 -> 4
```

Recursive traversal proceeds forward first.

| Step | Current Node | Action |
| --- | --- | --- |
| 1 | 1 | Recurse to 2 |
| 2 | 2 | Recurse to 3 |
| 3 | 3 | Recurse to 4 |
| 4 | 4 | Recurse to None |
| 5 | None | Return |
| 6 | 4 | Print 4 |
| 7 | 3 | Print 3 |
| 8 | 2 | Print 2 |
| 9 | 1 | Print 1 |

Output:

```
4 3 2 1
```

### Example 2

Input:

```
[0,-4,-1,3,-5]
```

Linked list structure:

```
0 -> -4 -> -1 -> 3 -> -5
```

Recursive call stack grows until reaching `-5`.

| Recursive Depth | Node |
| --- | --- |
| 1 | 0 |
| 2 | -4 |
| 3 | -1 |
| 4 | 3 |
| 5 | -5 |

During unwinding:

| Return Order | Printed Value |
| --- | --- |
| 1 | -5 |
| 2 | 3 |
| 3 | -1 |
| 4 | -4 |
| 5 | 0 |

Output:

```
-5 3 -1 -4 0
```

### Example 3

Input:

```
[-2,0,6,4,4,-6]
```

Linked list structure:

```
-2 -> 0 -> 6 -> 4 -> 4 -> -6
```

Traversal order:

```
-2 → 0 → 6 → 4 → 4 → -6
```

Printing order during recursion unwinding:

```
-6 → 4 → 4 → 6 → 0 → -2
```

Output:

```
-6 4 4 6 0 -2
```

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each node is visited exactly once |
| Space | O(n) | Recursive call stack stores one frame per node |

The runtime is linear because the algorithm traverses the linked list only once. Every node participates in one recursive call and one print operation.

The extra space complexity comes from recursion depth. In the worst case, the recursion stack contains all `n` nodes simultaneously, resulting in `O(n)` auxiliary space.

## Edge Cases

A single-node linked list is the smallest valid input. This case is important because recursive algorithms sometimes incorrectly assume the existence of a next node. In this implementation, recursion immediately reaches `None` after the single node, then unwinds and prints the node correctly.

Lists containing duplicate values can expose bugs in algorithms that accidentally depend on uniqueness. Since this solution operates entirely on node traversal order and never uses hashing or comparisons, duplicates are naturally handled without issue.

Very large lists near the maximum constraint size are another important consideration. Recursive solutions consume stack space proportional to the list length. Since the constraint is capped at 1000 nodes, recursion remains safe for this problem. Each node contributes exactly one recursive call frame, and the implementation maintains linear complexity throughout.
