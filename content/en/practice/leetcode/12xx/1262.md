## Solution

## Problem Understanding

The problem gives an integer array `nums`, and asks for the maximum possible sum of a subset of elements such that the final sum is divisible by `3`.

A subset means we are allowed to choose any combination of elements, including possibly choosing none of them. The goal is not to maximize the number of elements selected, but rather to maximize the resulting sum under the divisibility constraint.

For example, if the array is `[3,6,5,1,8]`, the total sum of all numbers is `23`, which is not divisible by `3`. However, by choosing the subset `[3,6,1,8]`, we obtain a sum of `18`, which is divisible by `3` and is the largest such sum possible.

The input size is important:

- `1 <= nums.length <= 4 * 10^4`
- `1 <= nums[i] <= 10^4`

These constraints immediately tell us that exponential solutions are impossible. Since there can be up to 40,000 elements, any approach that explores all subsets would be far too slow. We need something close to linear time.

The key mathematical property in this problem is divisibility by `3`. Any integer can only have one of three possible remainders when divided by `3`:

- remainder `0`
- remainder `1`
- remainder `2`

This small remainder space strongly suggests a dynamic programming solution based on modulo states.

Several edge cases are important:

- A single element not divisible by `3`, such as `[4]`, should return `0`.
- Arrays where the total sum is already divisible by `3` should simply return the total.
- Arrays where removing one small number produces divisibility should correctly identify that optimal adjustment.
- Arrays with many repeated values must still run efficiently.
- Since all numbers are positive, adding more numbers generally increases the sum, unless doing so breaks divisibility.

## Approaches

### Brute Force Approach

The brute force solution would generate every possible subset of the array, compute the sum of each subset, and keep track of the maximum sum divisible by `3`.

For an array of length `n`, there are `2^n` subsets. For each subset, we may compute its sum, resulting in exponential complexity.

This approach is correct because it checks every possible combination, guaranteeing that the optimal subset will eventually be found.

However, with `n = 40,000`, this is completely infeasible. Even for `n = 30`, `2^30` is already over one billion subsets.

### Key Insight

The important observation is that divisibility by `3` only depends on the remainder modulo `3`.

Instead of tracking every possible subset sum, we only need to track:

- the maximum sum with remainder `0`
- the maximum sum with remainder `1`
- the maximum sum with remainder `2`

As we process each number, we update these three states.

Suppose we already know the best sums for each remainder. When adding a new number `num`, every existing state can generate a new state:

- `(old_remainder + num % 3) % 3`

We only keep the maximum sum for each remainder.

This dramatically reduces the state space from exponential to constant size.

## Algorithm Walkthrough

### Optimal Dynamic Programming Algorithm

1. Create a DP array of size `3`.

The meaning of `dp[r]` is:

- `dp[0]` = maximum sum seen so far with remainder `0`
- `dp[1]` = maximum sum seen so far with remainder `1`
- `dp[2]` = maximum sum seen so far with remainder `2`

Initially:

```
dp = [0, -infinity, -infinity]
```

We begin with sum `0`, which has remainder `0`.
2. Process each number in the array one at a time.

For each number, we consider how it can extend all previously reachable sums.
3. Copy the current DP state before updating.

This is important because updates for the current number should not immediately affect other updates in the same iteration.
4. For each remainder state `r`:

- If `dp[r]` is reachable,
- Compute:

```
new_sum = dp[r] + num
new_remainder = new_sum % 3
```

Then update:

```
dp[new_remainder] = max(dp[new_remainder], new_sum)
```
5. Continue until all numbers are processed.
6. Return `dp[0]`.

This represents the maximum sum divisible by `3`.

### Why it works

The algorithm maintains an invariant:

> After processing the first `i` numbers, `dp[r]` stores the maximum achievable sum with remainder `r`.

Every subset either includes the current number or does not. The DP transition explicitly considers both possibilities. Since we only keep the maximum sum for each remainder class, no optimal solution is lost.

Because divisibility by `3` depends only on the remainder, these three states are sufficient to represent all relevant information.

## Python Solution

```python
from typing import List

class Solution:
    def maxSumDivThree(self, nums: List[int]) -> int:
        dp = [0, float('-inf'), float('-inf')]

        for num in nums:
            current = dp[:]

            for remainder in range(3):
                if current[remainder] != float('-inf'):
                    new_sum = current[remainder] + num
                    new_remainder = new_sum % 3

                    dp[new_remainder] = max(
                        dp[new_remainder],
                        new_sum
                    )

        return dp[0]
```

The implementation starts by initializing the DP array. Only remainder `0` is initially reachable because the empty subset has sum `0`.

For each number, we create a copy of the current DP state. This prevents updates from interfering with other transitions during the same iteration.

The inner loop iterates through the three possible remainder states. If a state is reachable, we try adding the current number to it and compute the resulting remainder.

We then update the best known sum for that remainder.

At the end, `dp[0]` contains the maximum sum divisible by `3`.

## Go Solution

```go
func maxSumDivThree(nums []int) int {
	dp := []int{0, -1 << 30, -1 << 30}

	for _, num := range nums {
		current := make([]int, 3)
		copy(current, dp)

		for remainder := 0; remainder < 3; remainder++ {
			if current[remainder] != -1<<30 {
				newSum := current[remainder] + num
				newRemainder := newSum % 3

				if newSum > dp[newRemainder] {
					dp[newRemainder] = newSum
				}
			}
		}
	}

	return dp[0]
}
```

The Go implementation follows the same logic as the Python version.

Since Go does not have a built-in negative infinity value for integers, a very small sentinel value (`-1 << 30`) is used to represent unreachable states.

Slices are used for the DP array, and `copy()` creates a snapshot of the previous state before updates.

Integer overflow is not a concern because the maximum possible sum is:

```go
4 * 10^4 * 10^4 = 4 * 10^8
```

which safely fits inside Go's `int`.

## Edge Cases

One important edge case is when no non-empty subset is divisible by `3`. For example, `[4]` or `[1]`. A buggy implementation might incorrectly return the largest element instead of `0`. The DP initialization handles this correctly because only remainder `0` starts as reachable.

Another important case is when the total sum itself is divisible by `3`. For example, `[3,6,9]`. In this scenario, the optimal answer is simply the sum of all elements. The DP transitions naturally preserve this because adding numbers with remainder `0` keeps the sum divisible by `3`.

A subtle edge case occurs when removing a small element creates divisibility. Consider `[5,5,5,5]`. The total sum is `20`, which leaves remainder `2`. The optimal solution excludes one `5`, producing `15`. Greedy approaches that only try local decisions can fail here, but the DP method correctly tracks all remainder states.

Another tricky scenario involves many repeated numbers with large input size. Since `nums.length` can reach `40,000`, inefficient solutions may time out or consume too much memory. The constant-size DP array ensures that the implementation remains efficient regardless of input size.
