## Solution

## Problem Understanding

LeetCode 1263, LeetCode "Minimum Moves to Move a Box to Their Target", is a grid-based pathfinding and state-search problem. The grid represents a warehouse containing walls, open floor cells, a player, a movable box, and a target location. The goal is to determine the minimum number of times the player must push the box so that the box eventually reaches the target.

The important detail is that the answer counts only pushes, not player movement steps. The player may walk around freely on open floor cells, but walking does not increase the cost. The only costed operation is pushing the box into an adjacent cell.

The input is an `m x n` matrix where each cell contains one of the following characters:

- `'#'` for walls
- `'.'` for open floor
- `'S'` for the player's starting position
- `'B'` for the box
- `'T'` for the target

The player can move in the four cardinal directions as long as the destination cell is not a wall and is not occupied by the box. To push the box, the player must stand adjacent to the box and move into the box's cell, causing the box to slide one step in the same direction. The box cannot move into walls.

The constraints are relatively small, with both dimensions at most 20. That means the grid contains at most 400 cells. This is small enough that a graph search over states is feasible, but large enough that naive exhaustive simulation becomes expensive if states are revisited repeatedly.

The critical observation is that the game state is not determined only by the box position. The player's position also matters, because different player locations around the same box position may allow different future pushes. Therefore, the search state must include both positions.

Several edge cases are important:

- The target may be unreachable because walls trap the box.
- The player may be unable to reach the correct side of the box to perform a push.
- A configuration may revisit the same box location but with a different player location, producing different future possibilities.
- The optimal solution may require many player movements but very few pushes, so minimizing movement distance is incorrect.
- The box can never be pulled, only pushed, which dramatically restricts valid transitions.

## Approaches

### Brute Force Approach

A brute-force solution would simulate every possible player movement and every possible box movement recursively or with standard BFS over the entire board configuration. The state would include both the player and box positions, and every player step would be treated equally.

This approach is correct because it eventually explores every reachable configuration. However, it is inefficient because most player movements do not matter directly to the objective. The problem asks for the minimum number of pushes, not the minimum number of total steps.

A naive BFS over all player movements expands an enormous number of unnecessary states. The player may walk around the map many times before pushing the box once. Treating walking and pushing identically causes a massive explosion in the search space.

### Key Insight

The key insight is that pushes are the true cost metric, while ordinary movement is effectively free.

This transforms the problem into a shortest-path problem where:

- Moving the player without pushing the box has cost 0
- Pushing the box has cost 1

Rather than exploring all movements equally, we search primarily over box pushes. For each potential push direction, we check whether the player can reach the required pushing position without crossing walls or the box itself.

The state becomes:

- Box position
- Player position

The transition becomes:

- Can the player reach the square behind the box?
- If yes, push the box one step

Since every push costs exactly 1, standard BFS over pushes guarantees the minimum number of pushes.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O((mn)^2 * 4^(mn)) | O((mn)^2) | Explores all player movements and box states indiscriminately |
| Optimal | O((mn)^2 * (mn)) | O((mn)^2) | BFS over push states with reachability checks |

## Algorithm Walkthrough

1. Parse the grid and locate the player, box, and target positions.

We first scan the entire grid to identify:

- Player starting position
- Box starting position
- Target position

These positions define the initial search state.
2. Define valid movement directions.

We use the four cardinal directions:

```
up, down, left, right
```

Every push and every movement check uses these directions.
3. Create a helper function to test whether the player can reach a position.

Before pushing the box, the player must stand on the opposite side of the intended push direction.

For example, if we want to push the box upward:

- The player must stand below the box.

We run a BFS from the player's current position to determine whether the required pushing position is reachable while treating the box as an obstacle.
4. Use BFS over box-push states.

Each BFS state contains:

- Box row and column
- Player row and column
- Number of pushes used so far

BFS is appropriate because every push has equal cost.
5. For each state, try all four push directions.

Suppose the box is at `(bx, by)` and we try direction `(dx, dy)`.

The new box position becomes:

```
(bx + dx, by + dy)
```

The required player position becomes:

```
(bx - dx, by - dy)
```

This is where the player must stand before the push.
6. Validate the push.

A push is valid only if:

- The destination box cell is inside bounds.
- The destination is not a wall.
- The player can reach the required pushing position.

The reachability check is the most important part of the algorithm.
7. Perform the push.

After pushing:

- The box moves into the destination cell.
- The player moves into the old box position.

The new state becomes:

```
(new_box_position, old_box_position)
```
8. Avoid revisiting states.

We maintain a visited set containing:

```
(box_position, player_position)
```

Revisiting identical states would create cycles and unnecessary work.
9. Stop when the box reaches the target.

Since BFS explores states in increasing push count order, the first time the box reaches the target is guaranteed to be optimal.
10. Return `-1` if BFS completes without success.

This means the target is unreachable.

### Why it works

The algorithm works because BFS explores states in order of increasing pushes. Every edge in the search graph corresponds to exactly one push, so the first time the target box position is reached must use the minimum number of pushes.

The reachability BFS guarantees that every push transition is physically achievable by the player. Therefore, every explored transition corresponds to a valid sequence of game moves.

## Python Solution

```python
from collections import deque
from typing import List, Tuple, Set

class Solution:
    def minPushBox(self, grid: List[List[str]]) -> int:
        rows = len(grid)
        cols = len(grid[0])

        player = box = target = None

        for r in range(rows):
            for c in range(cols):
                if grid[r][c] == 'S':
                    player = (r, c)
                elif grid[r][c] == 'B':
                    box = (r, c)
                elif grid[r][c] == 'T':
                    target = (r, c)

        directions = [(-1, 0), (1, 0), (0, -1), (0, 1)]

        def in_bounds(r: int, c: int) -> bool:
            return 0 <= r < rows and 0 <= c < cols

        def can_reach(
            start: Tuple[int, int],
            target_pos: Tuple[int, int],
            box_pos: Tuple[int, int]
        ) -> bool:
            queue = deque([start])
            visited = {start}

            while queue:
                r, c = queue.popleft()

                if (r, c) == target_pos:
                    return True

                for dr, dc in directions:
                    nr, nc = r + dr, c + dc

                    if not in_bounds(nr, nc):
                        continue

                    if grid[nr][nc] == '#':
                        continue

                    if (nr, nc) == box_pos:
                        continue

                    if (nr, nc) in visited:
                        continue

                    visited.add((nr, nc))
                    queue.append((nr, nc))

            return False

        queue = deque([
            (box[0], box[1], player[0], player[1], 0)
        ])

        visited: Set[Tuple[int, int, int, int]] = {
            (box[0], box[1], player[0], player[1])
        }

        while queue:
            bx, by, px, py, pushes = queue.popleft()

            if (bx, by) == target:
                return pushes

            for dr, dc in directions:
                new_bx = bx + dr
                new_by = by + dc

                player_needed_r = bx - dr
                player_needed_c = by - dc

                if not in_bounds(new_bx, new_by):
                    continue

                if not in_bounds(player_needed_r, player_needed_c):
                    continue

                if grid[new_bx][new_by] == '#':
                    continue

                if grid[player_needed_r][player_needed_c] == '#':
                    continue

                if not can_reach(
                    (px, py),
                    (player_needed_r, player_needed_c),
                    (bx, by)
                ):
                    continue

                new_state = (new_bx, new_by, bx, by)

                if new_state in visited:
                    continue

                visited.add(new_state)

                queue.append((
                    new_bx,
                    new_by,
                    bx,
                    by,
                    pushes + 1
                ))

        return -1
```

The implementation begins by locating the player, box, and target positions. These coordinates define the initial state of the puzzle.

The `can_reach` helper function performs a standard BFS to determine whether the player can walk to a required square without crossing walls or the box. The current box position is treated as blocked because the player cannot walk through the box.

The main BFS explores only push operations. For each state, we attempt all four push directions. Before pushing, the algorithm checks whether the player can stand behind the box. If the player can reach that location, the push is valid.

The new player position after a push becomes the old box position, because the player steps into the square previously occupied by the box.

The visited set prevents infinite loops and redundant exploration. Since BFS processes states in increasing push count order, the first time the box reaches the target gives the minimum number of pushes.

## Go Solution

```go
package main

import "container/list"

func minPushBox(grid [][]byte) int {
	rows := len(grid)
	cols := len(grid[0])

	var player, box, target [2]int

	for r := 0; r < rows; r++ {
		for c := 0; c < cols; c++ {
			if grid[r][c] == 'S' {
				player = [2]int{r, c}
			} else if grid[r][c] == 'B' {
				box = [2]int{r, c}
			} else if grid[r][c] == 'T' {
				target = [2]int{r, c}
			}
		}
	}

	directions := [][2]int{
		{-1, 0},
		{1, 0},
		{0, -1},
		{0, 1},
	}

	inBounds := func(r, c int) bool {
		return r >= 0 && r < rows && c >= 0 && c < cols
	}

	canReach := func(
		start [2]int,
		targetPos [2]int,
		boxPos [2]int,
	) bool {
		queue := list.New()
		queue.PushBack(start)

		visited := map[[2]int]bool{
			start: true,
		}

		for queue.Len() > 0 {
			front := queue.Front()
			queue.Remove(front)

			cur := front.Value.([2]int)
			r, c := cur[0], cur[1]

			if cur == targetPos {
				return true
			}

			for _, d := range directions {
				nr := r + d[0]
				nc := c + d[1]

				next := [2]int{nr, nc}

				if !inBounds(nr, nc) {
					continue
				}

				if grid[nr][nc] == '#' {
					continue
				}

				if next == boxPos {
					continue
				}

				if visited[next] {
					continue
				}

				visited[next] = true
				queue.PushBack(next)
			}
		}

		return false
	}

	type State struct {
		bx, by int
		px, py int
		pushes int
	}

	queue := list.New()
	queue.PushBack(State{
		box[0], box[1],
		player[0], player[1],
		0,
	})

	visited := map[[4]int]bool{
		{box[0], box[1], player[0], player[1]}: true,
	}

	for queue.Len() > 0 {
		front := queue.Front()
		queue.Remove(front)

		cur := front.Value.(State)

		if cur.bx == target[0] && cur.by == target[1] {
			return cur.pushes
		}

		for _, d := range directions {
			dr, dc := d[0], d[1]

			newBx := cur.bx + dr
			newBy := cur.by + dc

			playerNeedR := cur.bx - dr
			playerNeedC := cur.by - dc

			if !inBounds(newBx, newBy) {
				continue
			}

			if !inBounds(playerNeedR, playerNeedC) {
				continue
			}

			if grid[newBx][newBy] == '#' {
				continue
			}

			if grid[playerNeedR][playerNeedC] == '#' {
				continue
			}

			if !canReach(
				[2]int{cur.px, cur.py},
				[2]int{playerNeedR, playerNeedC},
				[2]int{cur.bx, cur.by},
			) {
				continue
			}

			stateKey := [4]int{
				newBx, newBy,
				cur.bx, cur.by,
			}

			if visited[stateKey] {
				continue
			}

			visited[stateKey] = true

			queue.PushBack(State{
				newBx,
				newBy,
				cur.bx,
				cur.by,
				cur.pushes + 1,
			})
		}
	}

	return -1
}
```

The Go implementation closely mirrors the Python version, but several language-specific details differ.

Go does not have tuples, so fixed-size arrays such as `[2]int` and `[4]int` are used for coordinates and visited-state keys. The BFS queues are implemented using `container/list`, since Go does not provide a built-in deque structure.

The helper BFS uses a map keyed by coordinate arrays to track visited cells. Since the grid dimensions are small, this approach remains efficient and straightforward.

Integer overflow is not a concern because the maximum number of states is bounded by the small grid size.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O((mn)^3) | Each BFS state may run a reachability BFS over the entire grid |
| Space | O((mn)^2) | Stores visited box-player states |

Let `N = m * n`.

There are at most `N` possible box positions and `N` possible player positions, producing `O(N^2)` total states. For each state, we may perform a reachability BFS costing `O(N)`. Therefore, the total complexity becomes `O(N^3)`.

With the maximum grid size limited to 400 cells, this complexity is acceptable.

## Edge Cases

One important edge case occurs when the box becomes trapped against walls or corners. Since the box can only be pushed and never pulled, corner positions are often irreversible dead ends unless the corner contains the target. A naive implementation may continue exploring impossible states unnecessarily. This solution handles the issue naturally because no further valid pushes can be generated from trapped configurations.

Another subtle edge case occurs when the player cannot reach the correct side of the box. The box itself acts as a moving obstacle, so even if a push direction appears geometrically valid, the player may be blocked from positioning properly. The `can_reach` BFS explicitly checks whether the player can access the required pushing square while treating the box as blocked.

A third important edge case involves revisiting states. The same box position may be reached with different player positions, and those situations are not equivalent. A solution that tracks only box coordinates may incorrectly prune valid paths. This implementation stores both the box position and player position in the visited set, ensuring that all distinct reachable states are considered correctly.

A final edge case involves large open grids where the player can wander extensively. If player movement steps were counted equally with pushes, the search space would explode. By structuring BFS around pushes instead of ordinary movement, the implementation avoids this inefficiency and directly optimizes the correct objective function.
