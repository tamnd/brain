## Solution

## Problem Understanding

The problem gives us a sorted list of disjoint half open intervals. Each interval `[a, b)` represents all real numbers `x` such that `a <= x < b`. The intervals are guaranteed to be non overlapping and already sorted by their starting values.

We are also given another interval called `toBeRemoved`. The goal is to remove every number inside `toBeRemoved` from the original set represented by `intervals`.

The important detail is that the intervals are half open. An interval `[a, b)` includes `a` but excludes `b`. This matters when two intervals touch at boundaries. For example, `[0,2)` and `[2,5)` do not overlap because the value `2` is not part of the first interval.

We are not merging intervals or inserting new ones. Instead, we process each existing interval and determine how much of it remains after removing the overlap with `toBeRemoved`.

For each interval, several situations are possible:

- The interval lies completely outside `toBeRemoved`, so it remains unchanged.
- The interval lies completely inside `toBeRemoved`, so it disappears entirely.
- The interval partially overlaps `toBeRemoved`, so it may split into one or two smaller intervals.

The constraints are important:

- There can be up to `10^4` intervals.
- Interval endpoints can be very large, from `-10^9` to `10^9`.

Because the input size is moderate, an `O(n)` solution is ideal and easily efficient enough. Since the intervals are already sorted and disjoint, we can process them independently without any complicated data structures.

Several edge cases can easily introduce bugs:

- An interval may be completely removed.
- `toBeRemoved` may overlap only the left side or only the right side.
- Removing a middle portion may split one interval into two intervals.
- `toBeRemoved` may not overlap any interval at all.
- `toBeRemoved` may fully cover all intervals.
- Boundary touching must be handled correctly because the intervals are half open.

For example, `[0,2)` and `[2,3)` do not overlap, so removing `[2,3)` from `[0,2)` should leave `[0,2)` unchanged.

## Approaches

### Brute Force Approach

A brute force strategy would try to represent all individual real number ranges in smaller pieces and repeatedly subtract overlaps. One possible interpretation is to discretize the interval space into tiny segments and test whether each point belongs to the removal interval.

This approach is conceptually correct because every value inside `toBeRemoved` would eventually be excluded. However, it is completely impractical because interval endpoints can be as large as `10^9`, and the intervals represent continuous real numbers rather than discrete integers.

Another brute force interpretation would compare every interval against every possible overlap configuration and reconstruct the remaining pieces manually. While feasible, it still performs unnecessary work because the intervals are already sorted and disjoint.

The key inefficiency is that we do not need to examine every point inside the intervals. We only need to reason about interval boundaries.

### Optimal Approach

The optimal observation is that each interval can be processed independently.

Because the intervals are already sorted and non overlapping, removing a range from one interval cannot affect any other interval. For each interval `[start, end)`:

- If it does not overlap `toBeRemoved`, we keep it unchanged.
- If there is overlap, the remaining portion can only exist:

- to the left of `toBeRemoved`
- to the right of `toBeRemoved`

This means at most two intervals can survive from any original interval.

For an interval `[start, end)` and removal interval `[removeStart, removeEnd)`:

- If `start < removeStart`, then the left piece `[start, min(end, removeStart))` survives.
- If `end > removeEnd`, then the right piece `[max(start, removeEnd), end)` survives.

This produces a clean single pass `O(n)` algorithm.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(R) or worse | O(R) | Attempts to process individual ranges or excessive overlap handling |
| Optimal | O(n) | O(n) | Processes each interval independently in one pass |

Here, `n` is the number of intervals, and `R` represents the numerical range size, which is infeasible for large coordinates.

## Algorithm Walkthrough

1. Initialize an empty result list that will store the remaining intervals after removal.
2. Extract the boundaries of `toBeRemoved` into two variables, `removeStart` and `removeEnd`. This improves readability and avoids repeated indexing.
3. Iterate through each interval `[start, end)` in `intervals`.
4. Check whether the interval has a surviving left portion. A left portion exists when `start < removeStart`. In that case, the surviving left interval becomes `[start, min(end, removeStart))`.
5. Only add the left portion if its start is strictly smaller than its end. This avoids creating invalid empty intervals.
6. Check whether the interval has a surviving right portion. A right portion exists when `end > removeEnd`. In that case, the surviving right interval becomes `[max(start, removeEnd), end)`.
7. Again, only add the right portion if it is valid.
8. Continue processing all intervals until the traversal completes.
9. Return the result list.

### Why it works

The algorithm works because every interval is disjoint and independent. Removing one interval from another can only leave up to two remaining pieces, one on the left side and one on the right side. The algorithm systematically computes exactly those surviving pieces and ignores the overlapping portion. Since every interval is processed once and all possible overlap configurations are covered, the final result is correct.

## Python Solution

```python
from typing import List

class Solution:
    def removeInterval(
        self,
        intervals: List[List[int]],
        toBeRemoved: List[int]
    ) -> List[List[int]]:
        
        remove_start, remove_end = toBeRemoved
        result = []

        for start, end in intervals:
            
            # Left remaining portion
            if start < remove_start:
                left_end = min(end, remove_start)
                
                if start < left_end:
                    result.append([start, left_end])

            # Right remaining portion
            if end > remove_end:
                right_start = max(start, remove_end)
                
                if right_start < end:
                    result.append([right_start, end])

        return result
```

The implementation follows the algorithm directly.

The method first extracts the removal interval boundaries into `remove_start` and `remove_end`. This improves clarity throughout the loop.

For every interval, the code independently computes the possible surviving left segment and right segment.

The left segment exists only if the interval begins before the removal interval. The endpoint of this surviving piece is the smaller value between the interval end and the removal start.

The right segment exists only if the interval extends beyond the removal interval. The starting point of this piece is the larger value between the interval start and the removal end.

The validity checks ensure that empty intervals are never added. This is especially important because the intervals are half open, and equal boundaries represent an empty range.

Finally, the result list is returned in sorted order automatically because the input intervals were already sorted and processed sequentially.

## Go Solution

```go
func removeInterval(intervals [][]int, toBeRemoved []int) [][]int {
    removeStart := toBeRemoved[0]
    removeEnd := toBeRemoved[1]

    result := [][]int{}

    for _, interval := range intervals {
        start := interval[0]
        end := interval[1]

        // Left remaining portion
        if start < removeStart {
            leftEnd := min(end, removeStart)

            if start < leftEnd {
                result = append(result, []int{start, leftEnd})
            }
        }

        // Right remaining portion
        if end > removeEnd {
            rightStart := max(start, removeEnd)

            if rightStart < end {
                result = append(result, []int{rightStart, end})
            }
        }
    }

    return result
}

func min(a int, b int) int {
    if a < b {
        return a
    }
    return b
}

func max(a int, b int) int {
    if a > b {
        return a
    }
    return b
}
```

The Go implementation mirrors the Python solution closely.

Go does not provide built in `min` and `max` functions for integers, so helper functions are included.

The result is stored as a slice of integer slices. Appending intervals dynamically works naturally with Go slices.

There are no integer overflow concerns because the constraints fit comfortably inside Go's standard integer range on LeetCode systems.

Unlike Python lists, Go slices require explicit construction when appending intervals.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each interval is processed exactly once |
| Space | O(n) | The output may contain up to twice as many intervals as the input |

The algorithm performs a single linear scan over the intervals. Every interval requires only constant time computations involving comparisons and boundary calculations.

The extra space complexity comes from the output list. In the worst case, every interval could split into two smaller intervals, producing up to `2n` intervals overall. Since output space is required by the problem itself, this is optimal.

## Edge Cases

One important edge case occurs when `toBeRemoved` completely covers an interval. For example, removing `[0,10)` from `[2,5)` should produce no interval at all. A buggy implementation might accidentally append empty intervals such as `[5,5)`. The solution avoids this by checking that the interval start is strictly smaller than the interval end before appending.

Another critical edge case involves boundary touching. Since the intervals are half open, `[0,2)` and `[2,5)` do not overlap. An implementation that incorrectly uses `<=` comparisons could mistakenly remove valid intervals. The algorithm correctly handles this by using strict inequality checks.

A third tricky case occurs when removing a middle section from an interval, such as removing `[2,3)` from `[0,5)`. This produces two separate intervals: `[0,2)` and `[3,5)`. Some implementations incorrectly keep only one surviving portion. The algorithm explicitly computes both the left and right surviving segments independently, ensuring both are preserved when necessary.
