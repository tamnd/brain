# LeetCode 1231 - Divide Chocolate

## Problem Understanding

This problem gives us an array called `sweetness`, where each element represents the sweetness value of a consecutive chunk of chocolate. We must divide the chocolate bar into `k + 1` contiguous pieces by making exactly `k` cuts.

The important detail is that we are being "generous". After dividing the chocolate, we keep the piece with the minimum total sweetness, while all other pieces go to our friends. Since we want to maximize our own happiness, we want the minimum piece among all pieces to be as large as possible.

In other words, we are trying to maximize:

$$\min(\text{sum of each piece})$$

subject to making exactly `k` cuts.

The input consists of:

- `sweetness`, an array of positive integers
- `k`, the number of friends

Since we and our friends together need `k + 1` pieces, the chocolate must be partitioned into exactly `k + 1` contiguous subarrays.

The output is a single integer representing the largest possible sweetness value of the piece we are forced to take.

The constraints are important:

- `sweetness.length <= 10^4`
- `sweetness[i] <= 10^5`

A brute force search over all possible partitions would be far too expensive because the number of ways to place cuts grows combinatorially. This strongly suggests we need a more efficient strategy.

Several edge cases are important:

- When `k = 0`, we make no cuts and receive the entire chocolate bar.
- When `k = sweetness.length - 1`, every chunk becomes its own piece.
- Arrays may contain highly uneven values, which can trick greedy approaches that are not carefully designed.
- Since all sweetness values are positive, larger segments always have larger sums, which becomes a key property for the optimal solution.

## Approaches

### Brute Force Approach

A straightforward approach is to try every possible way to place `k` cuts among the `n - 1` available cut positions.

For each partitioning:

1. Compute the sum of every piece.
2. Find the minimum piece sum.
3. Track the maximum such minimum across all valid partitions.

This approach is correct because it explicitly checks every possible solution. However, it is computationally infeasible.

If the array has `n` elements, we must choose `k` cut positions from `n - 1` possibilities:

$$\binom{n-1}{k}$$

This grows exponentially in practice. Even for moderate input sizes, the runtime becomes enormous.

### Optimal Approach, Binary Search on the Answer

The key observation is that we are not directly searching for cut positions. Instead, we can search for the answer itself.

Suppose we ask:

> Can we divide the chocolate into at least `k + 1` pieces such that every piece has sweetness at least `X`?

This becomes a decision problem.

If the answer is yes for some value `X`, then all smaller values are also feasible.

If the answer is no for some value `X`, then all larger values are impossible.

This monotonic property makes binary search applicable.

To test whether a target minimum sweetness `X` is feasible:

- Traverse the array from left to right.
- Keep accumulating sweetness.
- Whenever the running sum becomes at least `X`, create a piece and reset the sum.
- Count how many pieces we can form.

If we can form at least `k + 1` pieces, then `X` is achievable.

Because all numbers are positive, this greedy strategy always maximizes the number of valid pieces.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | O(k) | Tries every possible partition |
| Optimal | O(n log S) | O(1) | Binary search on answer, where S is total sweetness |

Here:

- `n` is the length of the array
- `S` is the sum of all sweetness values

## Algorithm Walkthrough

### Step 1: Define the Search Space

The minimum sweetness we can guarantee is at least:

$$\min(sweetness)$$

The maximum possible answer is:

$$\left\lfloor \frac{\sum sweetness}{k+1} \right\rfloor$$

because if every piece had sweetness larger than this value, the total sum would not be enough.

So we binary search within this range.

### Step 2: Binary Search the Answer

We repeatedly test a candidate minimum sweetness `mid`.

- If `mid` is feasible, try a larger value.
- If `mid` is not feasible, try a smaller value.

This gradually converges to the largest achievable minimum sweetness.

### Step 3: Greedy Feasibility Check

To check whether `mid` is feasible:

1. Initialize:

- `current_sum = 0`
- `pieces = 0`
2. Traverse the array.
3. Add each chunk to `current_sum`.
4. Whenever `current_sum >= mid`:

- Form a piece
- Increment `pieces`
- Reset `current_sum` to `0`
5. After processing the entire array:

- If `pieces >= k + 1`, then `mid` is feasible.

The greedy strategy works because forming a valid piece as early as possible leaves maximum flexibility for remaining pieces.

### Step 4: Return the Best Feasible Value

The binary search eventually finishes when the search range collapses.

The largest feasible value is the answer.

### Why it works

The correctness relies on two properties.

First, feasibility is monotonic. If we can create `k + 1` valid pieces with minimum sweetness `X`, then we can also do so for every smaller value.

Second, the greedy construction is optimal for feasibility checking. Since all sweetness values are positive, delaying a cut can never help create more valid pieces. Therefore, cutting as soon as the running sum reaches the target maximizes the number of achievable pieces.

Together, these properties make binary search with greedy checking both correct and efficient.

## Python Solution

```python
from typing import List

class Solution:
    def maximizeSweetness(self, sweetness: List[int], k: int) -> int:
        def can_divide(min_sweetness: int) -> bool:
            current_sum = 0
            pieces = 0

            for value in sweetness:
                current_sum += value

                if current_sum >= min_sweetness:
                    pieces += 1
                    current_sum = 0

            return pieces >= k + 1

        left = min(sweetness)
        right = sum(sweetness) // (k + 1)

        while left <= right:
            mid = (left + right) // 2

            if can_divide(mid):
                left = mid + 1
            else:
                right = mid - 1

        return right
```

The implementation begins with the helper function `can_divide`, which determines whether a given minimum sweetness target is achievable.

Inside this helper:

- `current_sum` accumulates chunk sweetness values.
- Whenever the accumulated sum reaches the target, we form a piece.
- The piece counter tracks how many valid segments can be created.

If we can form at least `k + 1` pieces, the target is feasible.

The binary search range is initialized carefully:

- `left` starts at the minimum chunk value.
- `right` starts at the average possible sweetness per piece.

During binary search:

- A feasible `mid` means we can attempt a larger minimum sweetness.
- An infeasible `mid` means we must reduce the target.

Finally, `right` stores the largest feasible minimum sweetness.

## Go Solution

```go
func maximizeSweetness(sweetness []int, k int) int {
	canDivide := func(minSweetness int) bool {
		currentSum := 0
		pieces := 0

		for _, value := range sweetness {
			currentSum += value

			if currentSum >= minSweetness {
				pieces++
				currentSum = 0
			}
		}

		return pieces >= k+1
	}

	minValue := sweetness[0]
	total := 0

	for _, value := range sweetness {
		if value < minValue {
			minValue = value
		}
		total += value
	}

	left := minValue
	right := total / (k + 1)

	for left <= right {
		mid := left + (right-left)/2

		if canDivide(mid) {
			left = mid + 1
		} else {
			right = mid - 1
		}
	}

	return right
}
```

The Go implementation follows the same logic as the Python version.

A few Go-specific details are worth noting:

- Go does not provide built-in `min` for slices, so we compute the minimum manually.
- Integer division naturally truncates toward zero, which matches the desired behavior.
- The binary search midpoint uses:

```go
mid := left + (right-left)/2
```

to avoid potential overflow, although overflow is not a practical concern under these constraints.

## Edge Cases

### Edge Case 1, No Cuts

When `k = 0`, we do not divide the chocolate at all. The entire array becomes our piece.

This case can cause bugs if the implementation assumes at least one cut exists. Our solution handles it naturally because the feasibility check simply tests whether one piece can achieve the target sweetness.

For example:

```
sweetness = [2,3,5]
k = 0
```

The answer is:

```
10
```

### Edge Case 2, Maximum Number of Cuts

When `k = n - 1`, every chunk must become its own piece.

In this situation, the answer must equal the minimum element in the array, because we receive the least sweet piece.

For example:

```
sweetness = [5,2,9]
k = 2
```

The only partition is:

```
[5], [2], [9]
```

So the answer is:

```
2
```

The greedy feasibility check correctly handles this because every chunk individually forms a piece.

### Edge Case 3, Extremely Uneven Values

Large values mixed with tiny values can break incorrect greedy strategies.

For example:

```
sweetness = [100,1,1,1,1]
k = 1
```

An incorrect approach might greedily isolate the largest chunk immediately and assume the answer is large.

However, we are maximizing the minimum piece, so the optimal split is:

```
[100]
[1,1,1,1]
```

The minimum sweetness is:

```
4
```

Our binary search correctly evaluates feasibility rather than relying on local greedy decisions about cut placement.
