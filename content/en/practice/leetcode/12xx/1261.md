## Solution

## Problem Understanding

This problem gives us a binary tree whose values have been completely contaminated. Every node currently contains `-1`, but we know the exact rule that was originally used to generate the values.

The recovery rules are deterministic:

- The root must have value `0`
- If a node has value `x`:

- its left child must have value `2 * x + 1`
- its right child must have value `2 * x + 2`

We need to implement a `FindElements` class with two operations:

- The constructor receives the contaminated tree and must recover the original values.
- The `find(target)` method must efficiently determine whether a value exists somewhere in the recovered tree.

The important detail is that the structure of the tree is still intact. Only the values are corrupted. That means the left and right pointers are valid, and we can reconstruct every value uniquely by traversing the tree from the root.

The constraints tell us several important things:

- The tree height is at most `20`
- The number of nodes is at most `10^4`
- `find()` can be called up to `10^4` times

These constraints strongly suggest that preprocessing during construction is beneficial. If we recover all values once and store them in a hash set, then each query becomes extremely fast.

An important observation is that every node value in the recovered tree is unique. The formulas generate a one-to-one mapping between tree positions and values.

Several edge cases are important:

- A tree containing only the root node
- Highly skewed trees, where nodes exist only on one side
- Missing children in arbitrary positions
- Large target values that do not exist
- Repeated `find()` calls, where efficiency matters significantly

Because the tree height is limited to `20`, recursion depth is safe in both Python and Go.

## Approaches

### Brute Force Approach

A naive solution would recover the tree normally, but avoid storing recovered values in any auxiliary data structure. Then, every time `find(target)` is called, we would traverse the entire tree searching for the target value.

This works because the recovery formulas uniquely determine every node value. Once recovered, the tree behaves like a normal binary tree with known values.

However, this approach is inefficient for repeated queries. In the worst case:

- Each `find()` operation scans all `n` nodes
- There may be up to `10^4` calls to `find()`

This leads to potentially `O(n * q)` total work, which can become large when both the number of nodes and queries are near their limits.

### Optimal Approach

The key insight is that `find()` will be called many times. Instead of searching the tree repeatedly, we should recover the tree once and store every recovered value in a hash set.

A hash set provides average `O(1)` membership lookup. During recovery:

- Assign correct values to every node
- Insert each value into a set

Then `find(target)` becomes a simple hash lookup.

This transforms repeated expensive tree traversals into constant-time queries.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) per find | O(h) | Recover tree, then DFS/BFS search for every query |
| Optimal | O(n) preprocessing, O(1) per find | O(n) | Store recovered values in a hash set |

## Algorithm Walkthrough

1. Initialize an empty hash set to store all recovered values.

The hash set allows constant-time lookup for future `find()` calls. Since values are unique, a set is the ideal structure.
2. Set the root value to `0`.

The problem guarantees that the original root value was always `0`.
3. Perform a DFS traversal starting from the root.

DFS is natural here because each node determines the values of its children directly.
4. For each visited node:

- Insert its value into the hash set
- If the left child exists:

- assign `2 * current + 1`
- recurse into the left child
- If the right child exists:

- assign `2 * current + 2`
- recurse into the right child

This guarantees that every node receives the exact value defined by the recovery rules.
5. Implement `find(target)` by checking whether `target` exists in the hash set.

Since the set contains every recovered value, membership directly answers the query.

### Why it works

The algorithm works because the recovery rules uniquely determine the value of every node based solely on its parent. Starting from the guaranteed root value `0`, DFS systematically reconstructs every reachable node exactly once. Since all recovered values are inserted into the hash set, `find(target)` correctly returns whether the recovered tree contains that value.

## Python Solution

```python
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import Optional

class FindElements:

    def __init__(self, root: Optional[TreeNode]):
        self.values = set()
        self.root = root

        if root:
            root.val = 0
            self._recover(root)

    def _recover(self, node: Optional[TreeNode]) -> None:
        if not node:
            return

        self.values.add(node.val)

        if node.left:
            node.left.val = 2 * node.val + 1
            self._recover(node.left)

        if node.right:
            node.right.val = 2 * node.val + 2
            self._recover(node.right)

    def find(self, target: int) -> bool:
        return target in self.values

# Your FindElements object will be instantiated and called as such:
# obj = FindElements(root)
# param_1 = obj.find(target)
```

The constructor initializes a hash set named `values`, which stores every recovered node value.

If the root exists, its value is restored to `0`, and DFS recovery begins.

The `_recover()` helper function performs the recursive traversal. Each node value is inserted into the set immediately after recovery. Then the left and right child values are computed using the required formulas.

The `find()` method is intentionally simple. Since all valid values are already stored in the set, the method only performs a membership check.

This design is efficient because all expensive work happens once during initialization.

## Go Solution

```go
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */

type FindElements struct {
    values map[int]bool
}

func Constructor(root *TreeNode) FindElements {
    finder := FindElements{
        values: make(map[int]bool),
    }

    if root != nil {
        root.Val = 0
        finder.recover(root)
    }

    return finder
}

func (this *FindElements) recover(node *TreeNode) {
    if node == nil {
        return
    }

    this.values[node.Val] = true

    if node.Left != nil {
        node.Left.Val = 2*node.Val + 1
        this.recover(node.Left)
    }

    if node.Right != nil {
        node.Right.Val = 2*node.Val + 2
        this.recover(node.Right)
    }
}

func (this *FindElements) Find(target int) bool {
    return this.values[target]
}

/**
 * Your FindElements object will be instantiated and called as such:
 * obj := Constructor(root);
 * param_1 := obj.Find(target);
 */
```

The Go implementation closely mirrors the Python version.

The primary difference is that Go uses `map[int]bool` instead of a Python `set`. Presence in the map indicates that the value exists in the recovered tree.

Go also requires explicit `nil` checks before accessing child pointers.

Integer overflow is not a concern because the maximum possible node value under the constraints remains well within Go's `int` range.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) preprocessing, O(1) per find | DFS visits every node once, hash lookup is constant time |
| Space | O(n) | Hash set stores every recovered value |

The constructor performs exactly one traversal of the tree, so recovery takes linear time relative to the number of nodes. Each `find()` operation is a constant-time hash lookup on average.

The space complexity is linear because every recovered node value is stored in the hash set.

## Edge Cases

A single-node tree is the smallest valid input. In this case, the recovered tree contains only value `0`. Implementations that assume children exist may accidentally dereference null pointers. This solution safely checks whether children exist before assigning values or recursing.

Sparse trees with missing intermediate children are another important edge case. The recovery formulas only apply to existing nodes. A common bug is accidentally generating values for nonexistent nodes. This implementation only computes child values if the child pointer is not null.

Large target values can also cause mistakes. The target range extends up to `10^6`, but many of those values may not exist in the tree. Since the implementation stores only recovered node values in a hash set, nonexistent values are correctly rejected in constant time.

Repeated `find()` calls are especially important under the constraints. A naive implementation that traverses the tree for every query would repeatedly redo work. By preprocessing all node values during construction, this solution guarantees efficient repeated lookups.
