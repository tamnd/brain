## Solution

## Problem Understanding

This problem gives us the roots of **two Binary Search Trees (BSTs)**, `root1` and `root2`, along with an integer `target`. We need to determine whether there exists **one node from the first tree** and **one node from the second tree** such that their values add up exactly to `target`.

In simpler terms, imagine we are allowed to pick one number from the first BST and one number from the second BST. Our goal is to check whether any such pair sums to the target value. If at least one valid pair exists, we return `true`. Otherwise, we return `false`.

The input consists of two BST roots. Since these are binary search trees, they satisfy the BST ordering property:

- Every node in the left subtree contains a smaller value.
- Every node in the right subtree contains a larger value.

This property is important because it enables more efficient searching strategies than brute force.

The expected output is a boolean value:

- `true` if a valid pair exists
- `false` if no valid pair exists

The constraints tell us several important things:

- Each tree contains between `1` and `5000` nodes.
- Node values and the target can range from `-10^9` to `10^9`.

A naive quadratic solution over both trees would require checking every possible pair of nodes, which could mean up to `5000 × 5000 = 25,000,000` comparisons in the worst case. While this may still be borderline feasible, it is not efficient and ignores the BST structure.

Several edge cases are worth identifying early:

- Trees may contain negative values, so we cannot assume all numbers are positive.
- The target may be very large or very small.
- A valid pair may involve root nodes, leaf nodes, or values deep inside either tree.
- One tree may be heavily skewed, behaving more like a linked list than a balanced tree.
- There may be no valid pair at all, requiring a complete traversal.

The problem guarantees that each tree contains at least one node, so we never need to handle completely empty input trees.

## Approaches

### Brute Force Approach

The most straightforward solution is to compare **every node in the first tree with every node in the second tree**.

We could first traverse both trees and store all node values in two arrays. Then, for each value in the first array, we check every value in the second array to see if their sum equals `target`.

This approach works because it exhaustively checks every possible pair. If a valid combination exists, we are guaranteed to find it.

However, the drawback is efficiency. If both trees contain `n` and `m` nodes respectively, we perform `n × m` comparisons. With up to `5000` nodes per tree, this becomes as large as **25 million operations**, which is unnecessarily expensive.

### Optimal Approach, DFS + Hash Set

The key insight comes from rewriting the equation:

$$a + b = target$$

If we already know `a`, then:

$$b = target - a$$

Instead of checking every node pair explicitly, we can:

1. Traverse the first BST and store all values in a hash set.
2. Traverse the second BST and, for each node value `x`, check whether `target - x` exists in the set.

The hash set gives us **O(1)** average lookup time. This transforms the problem from checking all pairs into a much faster lookup process.

Although the trees are BSTs, we do not actually need BST-specific searching here. A hash set approach is simpler and still optimal enough for the constraints.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × m) | O(n + m) | Compare every node pair from both trees |
| Optimal | O(n + m) | O(n) | Store first tree values in a hash set, then search complements |

Here, `n` and `m` are the number of nodes in the two trees.

## Algorithm Walkthrough

### Optimal Algorithm, DFS + Hash Set

1. Create an empty hash set called `values`.

This set will store all values from the first BST. A hash set is chosen because it provides constant time membership checks on average.
2. Traverse the first tree using DFS.

For every visited node:

- Add its value into `values`.
- Continue recursively into the left and right children.

After this traversal, `values` contains every number present in the first BST.
3. Traverse the second tree using DFS.

For every node with value `x`:

- Compute the complement:

`needed = target - x`
- Check whether `needed` exists inside `values`.

If it exists, we immediately return `True` because we found a valid pair.
4. Continue traversing the second tree if no match is found.

Search both left and right subtrees recursively.
5. If traversal completes without finding a valid pair, return `False`.

This means no combination of nodes from the two trees sums to the target.

### Why it works

The correctness comes from the complement relationship. Every valid pair must satisfy:

$$a + b = target$$

When traversing the second tree, for every value `b`, we explicitly check whether `target - b` exists in the first tree. Since the hash set contains **all values from the first tree**, any valid `a` will be found immediately. Therefore, the algorithm returns `True` if and only if a valid pair exists.

## Python Solution

```python
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import Optional

class Solution:
    def twoSumBSTs(
        self,
        root1: Optional[TreeNode],
        root2: Optional[TreeNode],
        target: int
    ) -> bool:
        values = set()

        def collect(node: Optional[TreeNode]) -> None:
            if not node:
                return

            values.add(node.val)
            collect(node.left)
            collect(node.right)

        def search(node: Optional[TreeNode]) -> bool:
            if not node:
                return False

            if target - node.val in values:
                return True

            return search(node.left) or search(node.right)

        collect(root1)
        return search(root2)
```

The implementation is divided into two logical phases.

The `collect()` helper function performs a depth first traversal of the first tree and inserts every node value into the `values` hash set. Since a set only stores unique elements and provides constant time lookup, it is ideal for complement searching.

The `search()` helper function traverses the second tree. For every node, it computes the needed complement using `target - node.val`. If this complement exists inside the set, we immediately return `True`.

The recursion naturally handles tree traversal without requiring an explicit stack. If no matching pair exists after exploring all nodes in the second tree, the function returns `False`.

## Go Solution

```go
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func twoSumBSTs(root1 *TreeNode, root2 *TreeNode, target int) bool {
	values := make(map[int]bool)

	var collect func(node *TreeNode)
	collect = func(node *TreeNode) {
		if node == nil {
			return
		}

		values[node.Val] = true
		collect(node.Left)
		collect(node.Right)
	}

	var search func(node *TreeNode) bool
	search = func(node *TreeNode) bool {
		if node == nil {
			return false
		}

		if values[target-node.Val] {
			return true
		}

		return search(node.Left) || search(node.Right)
	}

	collect(root1)
	return search(root2)
}
```

The Go implementation follows the exact same logic as the Python solution.

Instead of Python's `set`, Go uses a `map[int]bool` to simulate hash set behavior. Checking membership is done with:

```go
values[target-node.Val]
```

If the key exists, it evaluates to `true`.

Go also uses `nil` instead of Python's `None` for missing child pointers. Recursive helper functions are declared using function variables because nested named functions are not supported directly in Go.

Integer overflow is not a concern here because Go's `int` comfortably supports the problem's range on LeetCode environments.

First, traverse `root1`:

Now traverse `root2`:

| Current Node | Needed Value | In Set? | Result |
| --- | --- | --- | --- |
| 1 | 4 | Yes | Return `True` |

Since `1 + 4 = 5`, the algorithm immediately returns `true`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n + m) | Traverse both trees once |
| Space | O(n) | Hash set stores first tree values |

Let `n` be the number of nodes in `root1` and `m` be the number of nodes in `root2`.

The first traversal visits every node in `root1` exactly once, which costs `O(n)`. The second traversal visits every node in `root2` exactly once, costing `O(m)`. Since hash set lookups take constant time on average, the total runtime becomes:

$$O(n + m)$$

The additional memory comes from storing values from the first tree in a hash set, which requires `O(n)` space in the worst case.

## Edge Cases

### Single Node Trees

The smallest possible input consists of one node in each tree. A careless implementation might assume multiple nodes exist or rely on subtree traversal logic that fails on minimal input. This implementation works correctly because DFS naturally handles trees of any size, including a single node.

### Negative Numbers

Since node values may be negative, assumptions such as "move left for smaller sums" can become error prone if applied incorrectly. The hash set approach avoids this issue entirely because it only checks complements mathematically, regardless of sign.

### Highly Unbalanced Trees

A BST can degrade into a linked list if values are inserted in sorted order. Recursive traversals still work correctly, but naive algorithms that depend on balanced height assumptions may perform poorly. This implementation visits every node exactly once, so even skewed trees remain `O(n + m)`.

### No Valid Pair Exists

Some implementations may terminate early incorrectly or miss valid search paths. In this case, the second DFS explores every node if necessary. Only after every complement check fails does the algorithm return `False`, guaranteeing correctness.
