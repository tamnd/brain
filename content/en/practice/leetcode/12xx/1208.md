## Solution

## Problem Understanding

The problem is asking for the maximum length of a contiguous substring from string `s` that can be converted to the corresponding substring in string `t` without exceeding a given cost `maxCost`. Each character in `s` can be changed to match `t` at the same index, and the cost of changing a single character is the absolute difference of their ASCII values.

The input consists of two strings, `s` and `t`, which are guaranteed to have the same length, and an integer `maxCost`. The output is a single integer representing the length of the longest substring that can be modified under the cost constraint. If no substring meets the requirement, the function should return `0`.

Key points include that the strings are up to 100,000 characters long, and `maxCost` can be up to 1,000,000. This scale implies that any brute-force solution iterating over all possible substrings will be too slow. Edge cases include when `maxCost` is `0`, when the strings are identical, or when a single change exceeds `maxCost`.

## Approaches

The brute-force approach would involve iterating through every possible substring of `s`, calculating the total cost to convert it to `t`, and tracking the maximum length that meets `maxCost`. While this is correct, it is inefficient: generating all substrings and computing costs results in O(n^2) or O(n^3) time complexity depending on implementation, which is unacceptable for n up to 100,000.

The key insight for an optimal solution is that we are looking for the longest contiguous substring under a cumulative cost constraint. This naturally suggests a **sliding window approach**: we maintain a window `[left, right]` on `s`, keep a running total of the conversion cost for the window, and expand or shrink the window to respect `maxCost`. The sliding window ensures that each character is processed at most twice, leading to linear time complexity.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^2) | O(1) | Check every possible substring and sum costs |
| Optimal | O(n) | O(1) | Sliding window maintains running cost and window length |

## Algorithm Walkthrough

1. Initialize two pointers, `left` and `right`, both starting at 0, representing the current window on `s`.
2. Initialize a variable `currentCost` to track the total cost of converting the substring `s[left:right]` to `t[left:right]`.
3. Iterate `right` over the string from 0 to n-1:

1. Compute the cost to convert `s[right]` to `t[right]` and add it to `currentCost`.
2. While `currentCost` exceeds `maxCost`, shrink the window from the left by subtracting the cost of `s[left]` to `t[left]` and increment `left`.
4. At each step, calculate the length of the current valid window as `right - left + 1` and track the maximum length seen so far.
5. After finishing the iteration, return the maximum length found.

Why it works: At every step, the algorithm maintains a valid window whose conversion cost does not exceed `maxCost`. By adjusting the left boundary when the cost exceeds the limit, the algorithm guarantees that all considered substrings are feasible and that the longest feasible substring is found.

## Python Solution

```python
class Solution:
    def equalSubstring(self, s: str, t: str, maxCost: int) -> int:
        left = 0
        currentCost = 0
        maxLength = 0
        
        for right in range(len(s)):
            currentCost += abs(ord(s[right]) - ord(t[right]))
            while currentCost > maxCost:
                currentCost -= abs(ord(s[left]) - ord(t[left]))
                left += 1
            maxLength = max(maxLength, right - left + 1)
        
        return maxLength
```

This implementation maintains the running conversion cost and adjusts the window to ensure `currentCost` does not exceed `maxCost`. The window length is updated dynamically as we iterate over `s`. Each character is only processed twice, once when added and once when removed, ensuring linear time.

## Go Solution

```go
func equalSubstring(s string, t string, maxCost int) int {
    left, currentCost, maxLength := 0, 0, 0

    for right := 0; right < len(s); right++ {
        currentCost += int(abs(int(s[right]) - int(t[right])))
        for currentCost > maxCost {
            currentCost -= int(abs(int(s[left]) - int(t[left])))
            left++
        }
        if right-left+1 > maxLength {
            maxLength = right - left + 1
        }
    }

    return maxLength
}

func abs(x int) int {
    if x < 0 {
        return -x
    }
    return x
}
```

The Go solution mirrors the Python approach. The only notable difference is explicit type conversions between `byte` and `int` for the ASCII difference, and using a helper `abs` function since Go does not have a built-in absolute function for integers in this context.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each character is added and removed from the window at most once |
| Space | O(1) | Only pointers and counters are used, no additional data structures proportional to n |

The linear time is guaranteed because the sliding window only moves forward, ensuring every character is processed a constant number of times.

## Edge Cases

One important edge case is when `maxCost` is zero. In this case, the algorithm must correctly identify that only identical characters can form a valid substring. The sliding window ensures this by shrinking the window whenever a cost greater than zero is encountered.

Another edge case occurs when the cost of changing any single character exceeds `maxCost`. Here, the algorithm must reduce the window to zero length and continue scanning. The while loop that reduces the window from the left guarantees this is handled without skipping characters.

Finally, when the strings are identical, any `maxCost` greater than zero should allow the entire string to be converted (effectively no cost), testing that the implementation correctly handles a scenario where the conversion cost is zero but the window can be maximal.
