## Solution

## Problem Understanding

This problem asks us to simulate a game of Tic-Tac-Toe on a standard `3 x 3` board and determine the final result after a sequence of valid moves.

The input is a 2D array called `moves`, where each element represents a move in the form `[row, col]`. The moves are played in order. Player `A` always plays first and uses the symbol `'X'`, while player `B` plays second and uses `'O'`.

The goal is to determine one of four possible outcomes:

- `"A"` if player A wins
- `"B"` if player B wins
- `"Draw"` if all 9 cells are filled and nobody wins
- `"Pending"` if the game is not finished and nobody has won yet

A player wins when they occupy all three cells in:

- Any row
- Any column
- The main diagonal
- The anti-diagonal

The problem guarantees that the input sequence is valid. This means:

- No move is repeated
- Players alternate correctly
- Nobody continues playing after the game has already ended

These guarantees simplify the implementation because we do not need to validate the moves themselves.

The constraints are extremely small:

- At most 9 moves
- Board size is fixed at `3 x 3`

Because the input size is tiny, almost any reasonable approach will run efficiently. However, the goal is to design a clean and optimal solution that efficiently tracks the game state without repeatedly scanning the board.

Several edge cases are important:

- A player may win before all 9 moves are played
- The game may end with fewer than 9 moves and no winner yet, resulting in `"Pending"`
- A draw only occurs when all cells are filled and no player wins
- Diagonal wins are easy to overlook in naive implementations

## Approaches

### Brute Force Approach

The brute force solution explicitly builds the `3 x 3` board and simulates each move one by one.

After every move, we check:

- All rows
- All columns
- Both diagonals

to see whether the current player has formed three matching symbols.

This approach works because Tic-Tac-Toe has a fixed board size, so checking the entire board after each move is straightforward. After processing all moves, if no winner exists, we return either `"Draw"` or `"Pending"` depending on whether the board is full.

Although this approach is perfectly acceptable for the given constraints, it performs unnecessary repeated scans of the board. Every move triggers another full validation pass.

### Optimal Approach

The key observation is that we do not actually need to store the full board.

A player wins only when they completely fill:

- One row
- One column
- One diagonal

Instead of storing symbols, we can maintain counters:

- `rows[3]`
- `cols[3]`
- `diagonal`
- `anti_diagonal`

We use:

- `+1` for player A
- `-1` for player B

Whenever a player makes a move:

- Add the player's value to the corresponding row and column
- Update diagonals if applicable

If any counter reaches:

- `3`, player A wins
- `-3`, player B wins

This works because only one player can contribute consistently to a winning line.

This approach avoids repeatedly scanning the board and keeps the logic concise.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Simulates board and checks all win conditions after each move |
| Optimal | O(n) | O(1) | Tracks row, column, and diagonal counts incrementally |

Since the board size is constant, both are effectively constant time in practice. However, the counter-based solution is cleaner and more scalable conceptually.

## Algorithm Walkthrough

1. Initialize arrays to track row and column sums.

We create:

- `rows = [0, 0, 0]`
- `cols = [0, 0, 0]`

We also maintain:

- `diagonal = 0`
- `anti_diagonal = 0`

These structures allow us to track how many marks each player has contributed to every possible winning line.
2. Process moves one by one.

For each move at index `i`:

- If `i` is even, it is player A's turn
- If `i` is odd, it is player B's turn

We assign:

- `+1` for A
- `-1` for B
3. Update row and column counters.

Suppose the move is `(row, col)`.

We update:

```
rows[row] += player
cols[col] += player
```

If a row or column total becomes:

- `3`, A occupies all three cells
- `-3`, B occupies all three cells
4. Update diagonal counters when necessary.

A move belongs to the main diagonal if:

```
row == col
```

A move belongs to the anti-diagonal if:

```
row + col == 2
```

We update the corresponding counters accordingly.
5. Check for a winner immediately after every move.

After updating all counters, we check:

```
abs(rows[row]) == 3
abs(cols[col]) == 3
abs(diagonal) == 3
abs(anti_diagonal) == 3
```

If any condition is true:

- Return `"A"` if current player is A
- Return `"B"` if current player is B
6. Determine final game state.

If no winner exists after processing all moves:

- Return `"Draw"` if 9 moves were played
- Otherwise return `"Pending"`

### Why it works

The algorithm maintains an invariant:

- Positive values represent marks from player A
- Negative values represent marks from player B

A row, column, or diagonal can only reach absolute value `3` if all three positions were occupied by the same player. Since moves are valid and players alternate correctly, this guarantees accurate winner detection.

## Python Solution

```python
from typing import List

class Solution:
    def tictactoe(self, moves: List[List[int]]) -> str:
        rows = [0] * 3
        cols = [0] * 3
        diagonal = 0
        anti_diagonal = 0

        for index, (row, col) in enumerate(moves):
            player = 1 if index % 2 == 0 else -1

            rows[row] += player
            cols[col] += player

            if row == col:
                diagonal += player

            if row + col == 2:
                anti_diagonal += player

            if (
                abs(rows[row]) == 3
                or abs(cols[col]) == 3
                or abs(diagonal) == 3
                or abs(anti_diagonal) == 3
            ):
                return "A" if player == 1 else "B"

        return "Draw" if len(moves) == 9 else "Pending"
```

The implementation directly follows the algorithm described earlier.

The `rows` and `cols` arrays track cumulative scores for each line. Instead of storing actual board characters, we encode players numerically:

- Player A contributes `+1`
- Player B contributes `-1`

This allows a winning line to be identified simply by checking whether the absolute value reaches `3`.

During each iteration:

- The current player is determined using the move index
- Row and column counters are updated
- Diagonal counters are updated if the move belongs to either diagonal
- Win conditions are checked immediately

If no winner is found after all moves are processed, the result depends on whether the board is full.

## Go Solution

```go
func tictactoe(moves [][]int) string {
    rows := make([]int, 3)
    cols := make([]int, 3)

    diagonal := 0
    antiDiagonal := 0

    for i, move := range moves {
        row := move[0]
        col := move[1]

        player := 1
        if i%2 == 1 {
            player = -1
        }

        rows[row] += player
        cols[col] += player

        if row == col {
            diagonal += player
        }

        if row+col == 2 {
            antiDiagonal += player
        }

        if abs(rows[row]) == 3 ||
            abs(cols[col]) == 3 ||
            abs(diagonal) == 3 ||
            abs(antiDiagonal) == 3 {

            if player == 1 {
                return "A"
            }
            return "B"
        }
    }

    if len(moves) == 9 {
        return "Draw"
    }

    return "Pending"
}

func abs(x int) int {
    if x < 0 {
        return -x
    }
    return x
}
```

The Go implementation mirrors the Python logic closely.

One notable difference is that Go does not provide a built-in integer `abs` function, so we implement a small helper function manually.

Slices are used for the row and column counters. Since the board size is fixed, memory usage remains constant.

## Edge Cases

One important edge case is a game that ends before all nine moves are played. A naive implementation might continue processing moves unnecessarily or incorrectly assume the game must fill the board. This implementation checks for a winner immediately after every move, so the correct player is returned as soon as a winning line appears.

Another tricky case is diagonal detection. Some implementations correctly track rows and columns but forget one of the diagonals. This solution explicitly handles both:

- Main diagonal using `row == col`
- Anti-diagonal using `row + col == 2`

This guarantees all winning patterns are covered.

A third edge case occurs when the game has not finished but no winner exists yet. For example, after only two or three moves, the board is clearly incomplete. The implementation correctly distinguishes between `"Pending"` and `"Draw"` by checking whether all 9 moves have been played.

A subtle edge case involves overlapping counters. The center cell `(1,1)` belongs to both diagonals. Incorrect implementations may update only one diagonal. This solution updates both counters whenever appropriate, ensuring correct winner detection for diagonal patterns involving the center square.
