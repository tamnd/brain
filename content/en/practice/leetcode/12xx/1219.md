## Solution

## Problem Understanding

The problem gives a two dimensional grid where each cell contains either some amount of gold or `0`. A value of `0` represents an empty cell that cannot be visited. Any positive value represents a gold cell that can be collected.

You may start from any gold cell in the grid. From a current position, you can move one step in four directions: up, down, left, or right. You are not allowed to move diagonally. Each time you visit a cell, you collect all the gold in that cell. However, once a cell has been visited, it cannot be visited again in the same path.

The goal is to determine the maximum total amount of gold that can be collected from any valid path.

The important detail is that the path does not need to cover all gold cells. You may stop at any point. The path only needs to follow the movement rules and avoid revisiting cells.

The input is a matrix `grid` of dimensions `m x n`.

- `grid[i][j] == 0` means the cell is blocked or empty.
- `grid[i][j] > 0` means the cell contains that amount of gold.

The output is a single integer representing the maximum total gold collectible from any valid path.

The constraints provide several important observations:

- `m` and `n` are at most `15`, so the grid itself is relatively small.
- More importantly, there are at most `25` cells containing gold.

This second constraint is the key. Even though the grid may contain up to `225` cells, only a small subset actually matters for traversal. That makes exhaustive exploration with pruning feasible.

Several edge cases are important:

- The grid may contain only one gold cell.
- The grid may contain no gold at all.
- Gold cells may be disconnected into multiple regions.
- A greedy strategy can fail because taking a large value early may block access to a larger total path later.
- Paths can branch heavily, so revisiting prevention is critical to avoid infinite recursion or double counting.

## Approaches

### Brute Force Approach

A brute force solution would try every possible path starting from every gold cell.

For each starting cell:

1. Move in all possible directions.
2. Continue recursively exploring all valid next moves.
3. Track visited cells so the same cell is not reused.
4. Compute the total gold collected for every possible path.
5. Return the maximum total among all explored paths.

This approach is correct because it enumerates every valid path allowed by the problem constraints. Since the answer must correspond to one of these paths, exhaustive search guarantees correctness.

However, the number of possible paths grows exponentially. In the worst case, every gold cell may connect to several others, producing a massive branching factor. Without careful pruning, the runtime becomes extremely expensive.

### Key Insight for the Optimal Approach

The problem naturally fits backtracking with depth first search.

The important observation is that:

- There are at most `25` gold cells.
- Each cell can only be visited once in a path.

This means the recursion depth is bounded, and the search space is manageable with proper visited tracking.

Instead of generating paths separately, we can:

- Start DFS from every gold cell.
- Temporarily mark a cell as visited by setting it to `0`.
- Explore all four directions recursively.
- Restore the original value after recursion finishes.

This technique is classic backtracking. It allows us to reuse the same grid structure for visited state management without needing extra memory for a separate visited matrix.

Because each recursive call explores all valid continuations of the current path, the algorithm guarantees that the maximum achievable gold is discovered.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(4^k) | O(k) | Explores all possible paths without optimization |
| Optimal | O(4^k) | O(k) | DFS with backtracking, practical because `k <= 25` |

Here, `k` is the number of gold cells.

Although the asymptotic bound appears similar, the optimized backtracking implementation is efficient enough for the given constraints because the recursion depth is small and invalid paths terminate quickly.

## Algorithm Walkthrough

1. Iterate through every cell in the grid.

Any cell containing gold can serve as a starting position. Since the optimal path may begin anywhere, we must attempt DFS from every positive cell.
2. When a gold cell is found, start a DFS traversal.

The DFS function represents:

- The current position
- The gold collected so far
- The remaining unvisited cells
3. Inside DFS, first store the current cell's gold value.

We need this value because we will temporarily modify the grid to mark the cell as visited.
4. Mark the current cell as visited.

The simplest method is:

```
grid[row][col] = 0
```

This prevents revisiting the same cell during the current recursive path.
5. Explore all four directions.

For each direction:

- Compute the neighbor coordinates.
- Check bounds.
- Ensure the neighbor contains gold.
- Recursively continue the search.
6. Track the maximum gold collectable from neighboring paths.

Each recursive call returns the best possible continuation from that neighbor. We keep the maximum among all directions.
7. Restore the current cell after exploration.

This is the backtracking step:

```
grid[row][col] = original_gold
```

Restoring the value allows other DFS paths to use this cell later.
8. Return the total gold from this path.

The result equals:

- Current cell's gold
- Plus the best continuation discovered recursively
9. Update the global maximum.

After DFS finishes for one starting cell, compare its result with the overall best answer.

### Why it works

The algorithm systematically explores every valid path exactly as defined by the problem rules. Backtracking guarantees that each recursive call operates with the correct visited state, while restoration ensures independent exploration of alternate paths. Since every possible starting point and continuation is examined, the maximum collected gold must eventually be found.

## Python Solution

```python
from typing import List

class Solution:
    def getMaximumGold(self, grid: List[List[int]]) -> int:
        rows = len(grid)
        cols = len(grid[0])

        directions = [
            (1, 0),
            (-1, 0),
            (0, 1),
            (0, -1)
        ]

        def dfs(row: int, col: int) -> int:
            current_gold = grid[row][col]

            # Mark as visited
            grid[row][col] = 0

            best_next = 0

            for dr, dc in directions:
                next_row = row + dr
                next_col = col + dc

                if (
                    0 <= next_row < rows
                    and 0 <= next_col < cols
                    and grid[next_row][next_col] > 0
                ):
                    best_next = max(
                        best_next,
                        dfs(next_row, next_col)
                    )

            # Backtrack
            grid[row][col] = current_gold

            return current_gold + best_next

        maximum_gold = 0

        for row in range(rows):
            for col in range(cols):
                if grid[row][col] > 0:
                    maximum_gold = max(
                        maximum_gold,
                        dfs(row, col)
                    )

        return maximum_gold
```

The implementation begins by storing the grid dimensions and defining the four possible movement directions.

The `dfs` function performs the recursive exploration. It first saves the current cell's gold value, then marks the cell as visited by setting it to `0`. This prevents revisiting during the current path.

The algorithm then checks all four neighboring cells. If a neighbor is inside bounds and contains gold, DFS continues recursively from that position.

The variable `best_next` stores the maximum gold obtainable from any continuation path. After all neighbors are explored, the current cell is restored to its original value. This restoration is the essence of backtracking and allows future DFS calls to reuse the cell.

Finally, the function returns:

```python
current_gold + best_next
```

which represents the best possible path beginning at the current position.

The outer loops attempt DFS from every gold-containing cell, ensuring all possible starting points are considered.

## Go Solution

```go
package main

func getMaximumGold(grid [][]int) int {
	rows := len(grid)
	cols := len(grid[0])

	directions := [][]int{
		{1, 0},
		{-1, 0},
		{0, 1},
		{0, -1},
	}

	var dfs func(int, int) int

	dfs = func(row int, col int) int {
		currentGold := grid[row][col]

		// Mark as visited
		grid[row][col] = 0

		bestNext := 0

		for _, dir := range directions {
			nextRow := row + dir[0]
			nextCol := col + dir[1]

			if nextRow >= 0 &&
				nextRow < rows &&
				nextCol >= 0 &&
				nextCol < cols &&
				grid[nextRow][nextCol] > 0 {

				candidate := dfs(nextRow, nextCol)

				if candidate > bestNext {
					bestNext = candidate
				}
			}
		}

		// Backtrack
		grid[row][col] = currentGold

		return currentGold + bestNext
	}

	maximumGold := 0

	for row := 0; row < rows; row++ {
		for col := 0; col < cols; col++ {
			if grid[row][col] > 0 {
				candidate := dfs(row, col)

				if candidate > maximumGold {
					maximumGold = candidate
				}
			}
		}
	}

	return maximumGold
}
```

The Go implementation follows the same logic as the Python solution.

A recursive closure is used for DFS. Since Go does not support nested named functions directly in the same way as Python, the DFS function is declared as a variable and then assigned.

The grid itself is modified in place to track visited cells, avoiding extra memory allocation for a separate visited matrix.

Go slices are reference types, so changes to the grid inside recursion are reflected across calls. Because of this, restoring the cell during backtracking is essential for correctness.

Integer overflow is not a concern because the maximum possible gold is small:

- At most `25` gold cells
- Each cell contains at most `100`

So the theoretical maximum total is only `2500`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(4^k) | Each gold cell may branch into up to four recursive directions |
| Space | O(k) | Recursion stack depth is bounded by the number of gold cells |

Here, `k` represents the number of cells containing gold.

The recursion depth can never exceed `k` because a cell cannot be revisited within the same path.

Although the theoretical upper bound is exponential, the constraint `k <= 25` keeps the runtime manageable in practice. Most branches terminate early because visited cells and zero-value cells quickly reduce the available moves.

## Edge Cases

### Grid With No Gold

A grid may contain only zeros. In this case, there are no valid starting positions.

A buggy implementation might attempt DFS on every cell or assume at least one positive value exists. This implementation safely skips zero cells during the outer traversal loops. Since no DFS calls occur, the result remains `0`.

### Single Gold Cell

If the grid contains exactly one gold cell, the answer is simply that cell's value.

This case is important because recursive exploration immediately terminates without neighbors. The DFS implementation correctly returns the current cell's gold even when no recursive calls are made.

### Revisiting Cells

One of the most common bugs in path traversal problems is accidentally revisiting cells, creating cycles or double counting.

For example:

```
1 2
3 4
```

Without proper visited tracking, recursion could move back and forth infinitely.

This implementation prevents revisits by temporarily setting the current cell to `0` before exploring neighbors. Since recursive calls only move into positive cells, revisiting becomes impossible during the active DFS path.

### Disconnected Gold Regions

Gold cells may exist in completely separate connected components.

A flawed solution might only start DFS once and therefore miss better paths in other regions. This implementation starts DFS from every gold cell, guaranteeing that all disconnected regions are explored independently.

### Backtracking Restoration

Another subtle issue is forgetting to restore cells after DFS finishes.

If restoration does not occur, later DFS calls incorrectly treat previously visited cells as permanently unavailable. The implementation restores the original gold value immediately after exploring all neighbors, ensuring each DFS traversal starts with a clean grid state.
