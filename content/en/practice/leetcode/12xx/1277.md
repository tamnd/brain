## Solution

## Problem Understanding

The problem gives us a binary matrix of size `m x n`, where each cell contains either `0` or `1`. We need to count how many square submatrices consist entirely of ones.

A square submatrix is any contiguous square region inside the matrix. Its side length can be `1`, `2`, `3`, and so on, as long as it stays within bounds. For every possible square, we must determine whether every cell inside that square is equal to `1`.

For example, a single cell containing `1` counts as a valid `1 x 1` square. A `2 x 2` square counts only if all four cells are ones. Similarly, a `3 x 3` square counts only if all nine cells are ones.

The output is the total number of valid squares of every possible size.

The constraints are important:

- `1 <= m, n <= 300`
- Each cell is either `0` or `1`

A `300 x 300` matrix contains 90,000 cells. This is large enough that an inefficient brute-force solution can become too slow, especially if we repeatedly scan large square regions.

Several edge cases are worth identifying early:

- A matrix containing all zeros should return `0`.
- A matrix containing all ones should count squares of many different sizes.
- Single-row or single-column matrices can only contain `1 x 1` squares.
- Sparse matrices with isolated ones should not accidentally form larger squares.
- Large connected regions of ones require efficient reuse of previously computed information.

The key challenge is efficiently determining how many larger squares can be formed without repeatedly checking every cell inside every candidate square.

## Approaches

### Brute Force Approach

The most direct solution is to try every possible square size starting from every possible top-left position.

For each cell `(r, c)`, we can attempt to build:

- a `1 x 1` square,
- a `2 x 2` square,
- a `3 x 3` square,
- and so on.

For every candidate square, we scan all cells inside that square to verify that every value is `1`.

This approach is correct because it explicitly checks every possible square submatrix and counts only those that satisfy the condition.

However, it is inefficient. Suppose we try all square sizes for all starting positions. Verifying each square requires scanning its entire area. In the worst case, this becomes extremely expensive.

With a `300 x 300` matrix, the brute-force method can degrade toward `O(m * n * min(m,n)^3)` time, which is too slow.

### Optimal Dynamic Programming Approach

The key observation is that a large square of ones can only exist if smaller squares already exist around it.

Consider a cell `(r, c)` as the bottom-right corner of a square. Suppose we want to know the size of the largest square ending at this cell.

If the current cell contains `1`, then the largest square ending here depends on three neighboring values:

- top: `(r-1, c)`
- left: `(r, c-1)`
- top-left diagonal: `(r-1, c-1)`

The current square can only expand if all three neighboring regions already support squares of ones.

Therefore:

- If `matrix[r][c] == 0`, then no square ends here.
- If `matrix[r][c] == 1`, then:

$$dp[r][c] = 1 + \min(
dp[r-1][c],
dp[r][c-1],
dp[r-1][c-1]
)$$

The value `dp[r][c]` represents the side length of the largest square ending at `(r, c)`.

This works because the smallest neighboring square limits how large the current square can become.

An important insight is that if the largest square ending at `(r,c)` has size `k`, then that cell also contributes:

- one `1 x 1` square,
- one `2 x 2` square,
- ...
- one `k x k` square.

So we can simply add `dp[r][c]` to the answer.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m * n * min(m,n)^3) | O(1) | Checks every possible square explicitly |
| Optimal Dynamic Programming | O(m * n) | O(m * n) | Reuses results from neighboring cells |

## Algorithm Walkthrough

1. Create a DP table with the same dimensions as the matrix.

Each `dp[r][c]` will store the size of the largest square whose bottom-right corner is at `(r, c)`.
2. Initialize the first row and first column.

Any cell containing `1` on the top row or left column can only form a `1 x 1` square because there are not enough neighboring cells to expand further.
3. Iterate through every cell in the matrix.

For each cell:

- If the cell contains `0`, then no square can end there, so `dp[r][c] = 0`.
- If the cell contains `1`, then compute:

$$dp[r][c] =
1 + \min(
dp[r-1][c],
dp[r][c-1],
dp[r-1][c-1]
)$$

1. Add the DP value to the answer.

If `dp[r][c] = 3`, then this cell contributes:

- one `1 x 1` square,
- one `2 x 2` square,
- one `3 x 3` square.

Therefore, we add `3` to the total count.
2. Continue until all cells are processed.
3. Return the accumulated total.

### Why It Works

The algorithm works because every larger square must contain smaller valid squares directly above, directly left, and diagonally above-left.

The minimum of those three neighboring DP values determines the largest square that can be extended to include the current cell. If any neighboring direction cannot support a larger square, expansion stops there.

Thus, the DP recurrence correctly computes the maximum square size ending at every position, and summing these values counts every valid square exactly once.

## Python Solution

```python
from typing import List

class Solution:
    def countSquares(self, matrix: List[List[int]]) -> int:
        rows = len(matrix)
        cols = len(matrix[0])

        dp = [[0] * cols for _ in range(rows)]
        total_squares = 0

        for r in range(rows):
            for c in range(cols):
                if matrix[r][c] == 1:
                    if r == 0 or c == 0:
                        dp[r][c] = 1
                    else:
                        dp[r][c] = 1 + min(
                            dp[r - 1][c],
                            dp[r][c - 1],
                            dp[r - 1][c - 1]
                        )

                    total_squares += dp[r][c]

        return total_squares
```

The implementation begins by determining the matrix dimensions. A DP table of identical size is created, where each entry stores the largest square ending at that position.

We iterate through every cell in row-major order. If the current matrix value is `0`, we skip it because no valid square can end there.

If the value is `1`, we first handle boundary cells separately. Cells in the first row or first column cannot form larger squares because they lack neighbors above or to the left. Therefore, their DP value is simply `1`.

For all other cells, we apply the recurrence relation using the minimum of the three neighboring DP values. This determines the largest square that can be extended.

Finally, we add the DP value directly into the running total because every square size from `1` through `dp[r][c]` contributes one valid square ending at this position.

## Go Solution

```go
func countSquares(matrix [][]int) int {
    rows := len(matrix)
    cols := len(matrix[0])

    dp := make([][]int, rows)
    for i := range dp {
        dp[i] = make([]int, cols)
    }

    totalSquares := 0

    for r := 0; r < rows; r++ {
        for c := 0; c < cols; c++ {
            if matrix[r][c] == 1 {
                if r == 0 || c == 0 {
                    dp[r][c] = 1
                } else {
                    dp[r][c] = 1 + min(
                        dp[r-1][c],
                        min(dp[r][c-1], dp[r-1][c-1]),
                    )
                }

                totalSquares += dp[r][c]
            }
        }
    }

    return totalSquares
}

func min(a, b int) int {
    if a < b {
        return a
    }
    return b
}
```

The Go implementation follows the same dynamic programming strategy as the Python version. Since Go does not provide a built-in integer minimum function, we define a helper `min` function.

The DP table is constructed using slices of slices. Integer overflow is not a concern because the maximum number of squares in a `300 x 300` matrix is well within Go's integer range.

Go slices naturally handle dynamic allocation, so no special memory handling is needed beyond initializing the 2D slice.

| Space | O(m * n) | DP table stores one value per cell |

The algorithm performs constant work for every matrix cell. Each DP state depends only on three neighboring states, so computation is efficient and linear in the number of cells.

The additional memory usage comes from the DP table. Although this can be optimized to use less space, the standard implementation uses a full `m x n` table for clarity.

## Edge Cases

One important edge case is a matrix containing only zeros. A naive implementation might accidentally count empty or partial regions as squares. In this implementation, any cell with value `0` immediately produces a DP value of `0`, so such cells never contribute to the answer.

Another important case is a single-row or single-column matrix. Larger squares require both height and width, so only `1 x 1` squares are possible. The implementation handles this naturally because boundary cells are initialized directly to `1` when the matrix value is `1`, and no expansion logic is attempted.

A third critical edge case is when a nearly complete square contains a single zero inside it. For example:

```
[
  [1,1,1],
  [1,0,1],
  [1,1,1]
]
```

A careless approach might incorrectly count a `3 x 3` square. The DP transition prevents this because the zero cell forces its DP value to `0`, which limits all neighboring expansions through the `min(...)` operation.

Another subtle case involves large regions of ones. Rechecking every square repeatedly would be extremely slow. The dynamic programming solution avoids redundant computation by storing the largest square ending at each cell and reusing those results efficiently.
