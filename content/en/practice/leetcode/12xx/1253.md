## Solution

## Problem Understanding

The problem gives us three pieces of information about a binary matrix with exactly two rows and `n` columns.

The first value, `upper`, represents how many `1`s must appear in the top row. The second value, `lower`, represents how many `1`s must appear in the bottom row. The array `colsum` describes the required sum of each column. Since the matrix is binary, every cell can only contain either `0` or `1`.

For every column `i`:

- `colsum[i] == 0` means both cells in that column must be `0`
- `colsum[i] == 1` means exactly one of the two cells must be `1`
- `colsum[i] == 2` means both cells must be `1`

The goal is to construct any valid `2 x n` binary matrix that satisfies all these conditions. If no such matrix exists, we must return an empty array.

The constraints are important:

- `n` can be as large as `10^5`
- Every `colsum[i]` is only `0`, `1`, or `2`

Because the input size is large, exponential or quadratic approaches are not practical. We need a linear-time solution.

There are several important edge cases to think about before designing the algorithm.

If the total sum of `colsum` is not equal to `upper + lower`, then reconstruction is impossible. The matrix would require a different total number of `1`s than the row sums allow.

Columns with value `2` are forced decisions because both rows must contain `1`. These columns immediately reduce both `upper` and `lower`.

Columns with value `1` are flexible because the `1` can go either to the top row or the bottom row. However, assigning them incorrectly can make the remaining row sums impossible to satisfy.

Another edge case occurs when either `upper` or `lower` becomes negative during construction. That means we assigned too many `1`s to a row and no valid matrix exists.

## Approaches

### Brute Force Approach

A brute force solution would try every possible assignment for columns whose `colsum[i] == 1`.

For each such column, we have two choices:

- Place the `1` in the upper row
- Place the `1` in the lower row

If there are `k` columns with value `1`, then there are `2^k` possible configurations. For every configuration, we could check whether the row sums match `upper` and `lower`.

This approach is correct because it exhaustively explores all valid placements for flexible columns. Eventually, if a solution exists, brute force will find it.

However, this method is far too slow. In the worst case, all columns may have value `1`, giving `2^n` possibilities. With `n` up to `100000`, this is completely infeasible.

### Optimal Greedy Approach

The key observation is that columns with value `2` are mandatory decisions. Both rows must receive a `1`, so there is no flexibility.

After handling all `2`s, the only remaining decisions involve columns with value `1`.

At that point, we can greedily assign each `1` column to whichever row still needs more `1`s. If the upper row still has remaining capacity, place the `1` there. Otherwise, place it in the lower row.

This works because every `1` column contributes exactly one `1` total, and we only care about satisfying the remaining row counts. There is no interaction between columns beyond the row totals.

The greedy strategy succeeds because:

- Forced columns are handled first
- Remaining columns are independent
- Choosing a row with remaining capacity never blocks a future valid assignment

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(n) | Tries every possible assignment for flexible columns |
| Optimal | O(n) | O(n) | Greedily assigns columns while tracking remaining row sums |

## Algorithm Walkthrough

1. Create two arrays representing the upper and lower rows, both initially filled with `0`.
2. Iterate through `colsum` and process every column with value `2`.

For these columns, both rows must contain `1`.

Set:

- `top[i] = 1`
- `bottom[i] = 1`

Then decrement both `upper` and `lower` because we used one `1` from each row.
3. After processing all `2`s, check whether either `upper` or `lower` became negative.

If either value is negative, we used more `1`s than allowed, so return an empty array.
4. Iterate through `colsum` again and process columns with value `1`.

These columns are flexible. We decide where to place the single `1`.

If `upper > 0`, place the `1` in the top row:

- `top[i] = 1`
- decrement `upper`

Otherwise, place the `1` in the bottom row:

- `bottom[i] = 1`
- decrement `lower`
5. After processing all columns, verify that both `upper` and `lower` are exactly `0`.

If either value is still positive, we failed to place enough `1`s, so no valid matrix exists.
6. Return the two constructed rows.

### Why it works

The algorithm first handles all forced decisions, namely columns with value `2`. After those assignments, only columns with value `1` remain, and each such column contributes exactly one remaining `1`.

At every step, the algorithm places the `1` into a row that still requires additional `1`s. Since the total required sum equals the total available sum, greedily filling remaining capacity cannot waste a needed placement.

Thus, if a valid solution exists, the algorithm constructs one successfully.

## Python Solution

```python
from typing import List

class Solution:
    def reconstructMatrix(self, upper: int, lower: int, colsum: List[int]) -> List[List[int]]:
        n = len(colsum)

        top = [0] * n
        bottom = [0] * n

        # Handle forced columns where colsum[i] == 2
        for i in range(n):
            if colsum[i] == 2:
                top[i] = 1
                bottom[i] = 1
                upper -= 1
                lower -= 1

        # Invalid if forced assignments exceed limits
        if upper < 0 or lower < 0:
            return []

        # Handle flexible columns where colsum[i] == 1
        for i in range(n):
            if colsum[i] == 1:
                if upper > 0:
                    top[i] = 1
                    upper -= 1
                else:
                    bottom[i] = 1
                    lower -= 1

        # Both row sums must be exactly satisfied
        if upper != 0 or lower != 0:
            return []

        return [top, bottom]
```

The implementation follows the greedy strategy directly.

The first loop processes all columns with value `2`. Since these columns are mandatory, both rows receive a `1`, and the remaining row capacities are reduced accordingly.

After this phase, the code checks whether either row sum became negative. A negative value means the forced assignments already exceeded the allowed number of `1`s.

The second loop handles columns with value `1`. If the upper row still needs `1`s, the algorithm places the `1` there. Otherwise, it assigns the `1` to the lower row.

Finally, the code verifies that both row requirements were exactly met. If either row still needs additional `1`s, the reconstruction is impossible.

## Go Solution

```go
func reconstructMatrix(upper int, lower int, colsum []int) [][]int {
	n := len(colsum)

	top := make([]int, n)
	bottom := make([]int, n)

	// Handle columns with colsum == 2
	for i := 0; i < n; i++ {
		if colsum[i] == 2 {
			top[i] = 1
			bottom[i] = 1
			upper--
			lower--
		}
	}

	// Invalid if forced assignments exceed limits
	if upper < 0 || lower < 0 {
		return [][]int{}
	}

	// Handle columns with colsum == 1
	for i := 0; i < n; i++ {
		if colsum[i] == 1 {
			if upper > 0 {
				top[i] = 1
				upper--
			} else {
				bottom[i] = 1
				lower--
			}
		}
	}

	// Final validation
	if upper != 0 || lower != 0 {
		return [][]int{}
	}

	return [][]int{top, bottom}
}
```

The Go implementation is structurally identical to the Python version.

The main difference is that Go uses slices created with `make`. Returning an empty result is done with `[][]int{}` rather than `nil`, which more closely matches LeetCode expectations for an empty matrix.

Integer overflow is not a concern because all values are bounded by `10^5`.

## Edge Cases

One important edge case occurs when the total sum of `colsum` does not equal `upper + lower`. In that situation, reconstruction is mathematically impossible because the matrix would require a different number of total `1`s than the row constraints allow. The implementation handles this implicitly through the final validation step. If the totals do not match, either `upper` or `lower` will not end at zero.

Another critical edge case involves columns with value `2`. These columns are forced assignments because both rows must contain `1`. A buggy implementation might delay handling them or treat them like flexible columns. This solution processes them first so the remaining row capacities are always accurate. If too many `2`s exist, `upper` or `lower` becomes negative immediately, and the algorithm correctly returns an empty matrix.

A third edge case happens when one row becomes completely filled before all `1` columns are processed. For example, if `upper` becomes zero early, all remaining `1`s must go into the bottom row. The greedy strategy handles this naturally because it only assigns to the upper row while `upper > 0`. Once exhausted, every remaining flexible column is assigned to the lower row.

Another subtle case is when all columns are zero. In this situation, the correct answer is simply two rows filled with zeros. The algorithm handles this automatically because neither processing loop modifies the arrays, and the final validation succeeds only if both `upper` and `lower` are also zero.
