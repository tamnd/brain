## Solution

## Problem Understanding

The problem asks us to repeatedly transform an array according to a very specific set of local rules until the array becomes stable, meaning no element changes anymore.

We are given an integer array `arr`. Every "day", we construct a new version of the array based on the values from the previous day. For every index `i` that is not the first or last element, we compare `arr[i]` with its immediate neighbors:

- If `arr[i]` is strictly smaller than both neighbors, we increment it by `1`.
- If `arr[i]` is strictly larger than both neighbors, we decrement it by `1`.
- Otherwise, the element stays unchanged.

The first and last elements are special, they never change regardless of their values.

The important detail is that all modifications for a given day must happen simultaneously. This means when deciding whether an element changes, we always compare against the values from the previous day's array, not partially updated values from the current day. In other words, updates cannot influence one another within the same iteration.

Our task is to keep simulating these daily transformations until the array stops changing, then return the final stable array.

The constraints are small:

- `3 <= arr.length <= 100`
- `1 <= arr[i] <= 100`

These limits tell us that performance requirements are not extremely demanding. Since the array size is at most `100`, even repeatedly simulating transformations is computationally feasible. This allows us to prioritize correctness and clarity.

There are several important edge cases to keep in mind. Arrays that are already stable should immediately return unchanged. Arrays with repeated values can be tricky because the rules require strict inequality, equal neighbors do not trigger increments or decrements. Another subtle case is ensuring simultaneous updates, because modifying the array in place while iterating can lead to incorrect results. Finally, local peaks and valleys may require multiple days to stabilize, so a single pass is not enough.

## Approaches

### Brute Force Approach

The most straightforward approach is to simulate the process exactly as described.

For every day, we create a fresh copy of the array and inspect every interior element. Based on the previous day's values, we determine whether the current element should increase, decrease, or remain unchanged. After processing the entire array, we compare the new version to the old one. If they are identical, the process has stabilized and we return the result. Otherwise, we repeat.

This approach is guaranteed to be correct because it directly mirrors the problem statement. Every transformation step follows the exact rules, and simultaneous updates are naturally enforced by using a copy of the array.

Although this may look inefficient at first glance, the constraints are very small. Since each transformation only changes elements by `1`, and values are bounded, the process stabilizes quickly in practice.

### Key Insight for an Improved Solution

The key observation is that we do not need anything more sophisticated than simulation.

Unlike many array problems where we search for a mathematical shortcut or advanced data structure, this problem's constraints are intentionally small enough that repeated simulation is optimal and clean. The important optimization is simply to avoid in-place modification during a day's transformation.

Instead of updating `arr` directly, we create a copy called `next_arr`. We use the original array to decide changes and write updates into the copy. This guarantees simultaneous updates and avoids subtle correctness bugs.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × d) | O(n) | Simulates every day until stabilization |
| Optimal | O(n × d) | O(n) | Same simulation, but carefully uses a copied array for simultaneous updates |

Here, `n` is the array length and `d` is the number of days until stabilization.

Because both `n` and values are bounded by `100`, `d` remains small, making this approach effectively efficient.

## Algorithm Walkthrough

### Step-by-Step Algorithm

1. Initialize a loop that continues until no changes occur.

We repeatedly simulate one day of transformations. Since the array eventually stabilizes, this loop is guaranteed to terminate.
2. Create a copy of the current array.

We store updates in a separate array called `next_arr`. This is essential because all updates must happen simultaneously. If we modified the original array directly, later comparisons in the same iteration could see already-updated values, producing incorrect results.
3. Iterate through all interior indices.

We only process indices from `1` to `len(arr) - 2` because the first and last elements never change.
4. Compare each element with its neighbors.

For index `i`:

- If `arr[i] < arr[i - 1]` and `arr[i] < arr[i + 1]`, increment `next_arr[i]`.
- If `arr[i] > arr[i - 1]` and `arr[i] > arr[i + 1]`, decrement `next_arr[i]`.
- Otherwise, leave it unchanged.
5. Check whether the array changed.

If `next_arr` is identical to `arr`, the array has stabilized, so return it.
6. Update the array for the next day.

Assign `arr = next_arr` and continue the simulation.

### Why it works

The correctness comes from maintaining the invariant that each day's updates are computed exclusively from the previous day's array. Since all modifications are applied simultaneously using a copied array, the simulation exactly matches the problem statement.

Because every transformation moves local minima upward and local maxima downward, and values remain bounded, the process must eventually stabilize. Once no element changes during an iteration, the array satisfies the equilibrium condition and is the required final answer.

## Python Solution

```python
from typing import List

class Solution:
    def transformArray(self, arr: List[int]) -> List[int]:
        while True:
            next_arr = arr[:]

            for i in range(1, len(arr) - 1):
                if arr[i] < arr[i - 1] and arr[i] < arr[i + 1]:
                    next_arr[i] += 1
                elif arr[i] > arr[i - 1] and arr[i] > arr[i + 1]:
                    next_arr[i] -= 1

            if next_arr == arr:
                return arr

            arr = next_arr
```

The implementation closely follows the algorithm.

The `while True` loop repeatedly simulates days until stabilization. At the beginning of each iteration, we create `next_arr = arr[:]`, which produces a shallow copy of the array. This copy ensures simultaneous updates.

We then iterate over interior indices only, skipping the first and last elements because they never change.

For each element, we compare it with its left and right neighbors. If it forms a valley, meaning smaller than both neighbors, we increment it. If it forms a peak, meaning larger than both neighbors, we decrement it.

After processing all elements, we compare `next_arr` and `arr`. If they are identical, no changes occurred and we return the stable array. Otherwise, we continue with the updated array.

## Go Solution

```go
func transformArray(arr []int) []int {
	for {
		nextArr := make([]int, len(arr))
		copy(nextArr, arr)

		for i := 1; i < len(arr)-1; i++ {
			if arr[i] < arr[i-1] && arr[i] < arr[i+1] {
				nextArr[i]++
			} else if arr[i] > arr[i-1] && arr[i] > arr[i+1] {
				nextArr[i]--
			}
		}

		unchanged := true
		for i := 0; i < len(arr); i++ {
			if arr[i] != nextArr[i] {
				unchanged = false
				break
			}
		}

		if unchanged {
			return arr
		}

		arr = nextArr
	}
}
```

The Go implementation follows the same logic as Python but uses Go slices.

Since slices are references to underlying arrays, we must explicitly create a new slice using `make()` and copy values with `copy()`. This ensures simultaneous updates rather than accidental in-place modification.

Unlike Python, Go does not support direct slice equality comparison, so we manually compare elements to determine whether the array stabilized.

There are no concerns about integer overflow because values remain very small under the given constraints.

## Edge Cases

### Already Stable Array

An array may already satisfy the equilibrium condition on day zero. For example:

```
[1,2,3,4,5]
```

No element is strictly larger or smaller than both neighbors. A buggy implementation might unnecessarily continue processing or accidentally modify values. Our implementation correctly detects no changes after the first simulation and immediately returns the original array.

### Equal Neighbor Values

The rules require _strict_ inequality. Consider:

```
[4,4,4,4]
```

No element is smaller or larger than both neighbors because equality does not count. A common bug is using `<=` or `>=` comparisons, which would incorrectly modify plateau regions. Our implementation uses `<` and `>` exclusively, preventing this issue.

### Simultaneous Updates

This is the most important subtle case. Consider:

```
[1,6,3,4,3,5]
```

If we updated the array in place, changing `6 → 5` could incorrectly affect later comparisons in the same day. The problem explicitly requires all updates to happen simultaneously. Our solution avoids this by using a copied array, ensuring every decision uses the previous day's values.

### Minimum Array Length

The smallest valid input size is `3`, meaning there is only one interior element:

```
[3,1,3]
```

Some implementations incorrectly assume multiple interior elements or mishandle loop boundaries. Since we iterate from `1` to `len(arr) - 2`, the single middle element is handled correctly without index errors.
