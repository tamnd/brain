## Solution

## Problem Understanding

The problem gives us a two dimensional grid representing a server center. Each cell in the grid contains either `1` or `0`.

A value of `1` means there is a server located at that position.

A value of `0` means the cell is empty.

Two servers are considered connected, or able to communicate, if they share either:

- the same row
- the same column

The task is to count how many servers can communicate with at least one other server.

The important detail is that we are not counting communication pairs. We are counting individual servers that have at least one partner in their row or column.

For example, if three servers exist in the same row, all three count because each one can communicate with the others.

The input size is constrained by:

- `1 <= m <= 250`
- `1 <= n <= 250`

This means the grid can contain at most `250 * 250 = 62,500` cells. That size is small enough for an `O(m * n)` or `O(m * n * something small)` solution, but large enough that repeatedly scanning entire rows and columns for every server could become inefficient.

Several edge cases are important:

- A grid with only isolated servers should return `0`
- A grid where every server shares a row or column should count all servers
- A single row or single column grid requires careful handling because all servers may automatically communicate
- Sparse grids with only one server must not incorrectly count that server
- Dense grids should still run efficiently without repeated rescanning

The problem guarantees that every grid cell contains only `0` or `1`, so we never need to validate the input format.

## Approaches

### Brute Force Approach

A straightforward solution is to inspect every server individually.

For each cell containing a server:

1. Scan the entire row to see whether another server exists
2. Scan the entire column to see whether another server exists
3. If either scan finds another server, count the current server

This approach is correct because it directly checks the communication rule defined in the problem.

However, it is inefficient because for every server we may scan:

- up to `m` cells in its column
- up to `n` cells in its row

If the grid is full of servers, this repeated rescanning becomes expensive.

The worst case complexity becomes:

- `O(m * n * (m + n))`

With dimensions up to `250 x 250`, this is still technically feasible, but unnecessarily slow.

### Optimal Approach

The key observation is that communication depends only on how many servers exist in each row and each column.

A server communicates if:

- its row contains more than one server
- or its column contains more than one server

Instead of repeatedly rescanning rows and columns, we can preprocess:

- the number of servers in every row
- the number of servers in every column

Then, during a second pass through the grid:

- if a server's row count is greater than `1`
- or its column count is greater than `1`

then that server can communicate.

This transforms repeated work into constant time lookups.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m × n × (m + n)) | O(1) | Repeatedly scans rows and columns for each server |
| Optimal | O(m × n) | O(m + n) | Precomputes row and column server counts |

## Algorithm Walkthrough

1. Create two arrays:

- `row_counts` of size `m`
- `col_counts` of size `n`

These arrays store how many servers exist in each row and column.
2. Traverse the entire grid once.

Whenever a server is found at position `(r, c)`:

- increment `row_counts[r]`
- increment `col_counts[c]`

After this pass:

- `row_counts[r]` tells us how many servers exist in row `r`
- `col_counts[c]` tells us how many servers exist in column `c`
3. Traverse the grid a second time.

For every server at `(r, c)`:

- check whether `row_counts[r] > 1`
- or `col_counts[c] > 1`

If either condition is true, this server can communicate with another server.
4. Increment the answer counter whenever a communicating server is found.
5. Return the final count.

### Why it works

A server can communicate if and only if another server exists in the same row or column.

The preprocessing step computes exactly how many servers belong to every row and column. Therefore:

- `row_counts[r] > 1` guarantees another server exists in row `r`
- `col_counts[c] > 1` guarantees another server exists in column `c`

Every communicating server is counted exactly once, and isolated servers are excluded. Thus the algorithm is correct.

## Python Solution

```python
from typing import List

class Solution:
    def countServers(self, grid: List[List[int]]) -> int:
        rows = len(grid)
        cols = len(grid[0])

        row_counts = [0] * rows
        col_counts = [0] * cols

        # First pass: count servers in each row and column
        for r in range(rows):
            for c in range(cols):
                if grid[r][c] == 1:
                    row_counts[r] += 1
                    col_counts[c] += 1

        communicating_servers = 0

        # Second pass: count servers that can communicate
        for r in range(rows):
            for c in range(cols):
                if grid[r][c] == 1:
                    if row_counts[r] > 1 or col_counts[c] > 1:
                        communicating_servers += 1

        return communicating_servers
```

The implementation follows the exact two pass strategy described in the algorithm walkthrough.

The first section initializes:

- `row_counts`
- `col_counts`

These arrays act as frequency tables.

The first nested loop processes every cell once. Whenever a server is encountered, the corresponding row and column counters are incremented.

After preprocessing is complete, the second traversal evaluates every server independently. Instead of rescanning rows and columns, the solution performs constant time lookups into the precomputed arrays.

If either the row or column contains more than one server, the current server is counted as communicating.

Finally, the total number of communicating servers is returned.

## Go Solution

```go
func countServers(grid [][]int) int {
    rows := len(grid)
    cols := len(grid[0])

    rowCounts := make([]int, rows)
    colCounts := make([]int, cols)

    // First pass: count servers in each row and column
    for r := 0; r < rows; r++ {
        for c := 0; c < cols; c++ {
            if grid[r][c] == 1 {
                rowCounts[r]++
                colCounts[c]++
            }
        }
    }

    communicatingServers := 0

    // Second pass: count communicating servers
    for r := 0; r < rows; r++ {
        for c := 0; c < cols; c++ {
            if grid[r][c] == 1 {
                if rowCounts[r] > 1 || colCounts[c] > 1 {
                    communicatingServers++
                }
            }
        }
    }

    return communicatingServers
}
```

The Go implementation is structurally identical to the Python solution.

Slices are used for `rowCounts` and `colCounts`, which provide dynamic array behavior. Since the constraints are small, integer overflow is not a concern because the maximum number of servers is only `62,500`.

Unlike Python, Go requires explicit allocation using `make`. Otherwise, the logic remains the same.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m × n) | The grid is traversed twice |
| Space | O(m + n) | Stores row and column counts |

The algorithm performs two complete passes over the grid.

Each pass processes every cell exactly once, leading to linear complexity relative to the number of cells.

The additional memory comes only from the row and column count arrays:

- `m` entries for rows
- `n` entries for columns

No additional structures proportional to the number of grid cells are required.

## Edge Cases

### A Single Isolated Server

A grid containing exactly one server is an important edge case because the server has no neighbors in either its row or column.

For example:

```
[[1]]
```

A buggy implementation might accidentally count the server simply because it exists. The correct logic requires at least one additional server in the same row or column.

The implementation handles this correctly because both:

- `row_counts[r]`
- `col_counts[c]`

equal `1`, not greater than `1`.

### Servers Only in One Row

Consider:

```
[[1,0,1,1]]
```

All servers communicate because they share the same row.

This edge case is important because there is only one row, so column counts alone are insufficient.

The implementation correctly identifies communication through:

```
row_counts[r] > 1
```

which becomes true for every server in that row.

### Servers Only in One Column

Consider:

```
[
 [1],
 [1],
 [0],
 [1]
]
```

All three servers communicate because they share the same column.

This tests whether the algorithm treats row and column communication symmetrically.

The implementation handles this correctly because:

```
col_counts[c] > 1
```

is true for every server in the column.

### Completely Empty Grid

Although the grid dimensions are at least `1 x 1`, the grid may contain no servers at all:

```
[
 [0,0],
 [0,0]
]
```

A careless implementation might assume at least one server exists and incorrectly increment counters.

The current solution only processes cells containing `1`, so the final answer correctly remains `0`.
