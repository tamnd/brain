## Solution

## Problem Understanding

The problem presents a scenario where we have `n` chips placed at various positions on a number line, represented by an array `position`. Each chip can be moved either by 2 units at zero cost or by 1 unit at a cost of 1. The goal is to move all chips to the same position while minimizing the total cost.

In other words, we need to find the minimum number of single-step moves (cost = 1) required to align all chips at one location, taking advantage of the zero-cost moves whenever possible. The constraints tell us that the array length is small (`1 <= position.length <= 100`), but the values themselves can be very large (`1 <= position[i] <= 10^9`). This indicates that we cannot rely on iterating over all possible positions because the positions themselves could be extremely large. Instead, we need a solution that works based on the properties of the positions, not their absolute values.

Key observations include that moving a chip by 2 units costs nothing. Therefore, moving chips between positions of the same parity (even-to-even or odd-to-odd) costs nothing, while moving between even and odd positions costs 1 per chip. This parity-based insight is the key to solving the problem efficiently.

Important edge cases include arrays with a single chip (cost should be 0), arrays where all chips are already on positions of the same parity, and arrays with positions spaced very far apart.

## Approaches

The brute-force approach would involve trying to move all chips to every possible position in the input array, computing the total cost each time. While this guarantees correctness, it is unnecessary and inefficient because the positions can be very large, and the cost calculation can be simplified using the parity observation.

The optimal approach relies on a simple insight: since moving by 2 costs nothing, we only need to consider the parity of positions. Chips on even positions can be gathered for free, and chips on odd positions can be gathered for free. The cost arises only when moving chips between even and odd positions. Therefore, the minimum cost is simply the smaller count between chips on even positions and chips on odd positions.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * max(position)) | O(1) | Attempt all positions; inefficient for large values |
| Optimal | O(n) | O(1) | Count even and odd positions; cost is min(even_count, odd_count) |

## Algorithm Walkthrough

1. Initialize two counters, `even_count` and `odd_count`, to zero. These will track how many chips are at even and odd positions, respectively.
2. Iterate through the array `position`. For each chip position: if the position is even, increment `even_count`; if odd, increment `odd_count`.
3. After the loop, the minimum cost to move all chips to the same position is the smaller of `even_count` and `odd_count`. This is because moving all chips to an even position costs `odd_count`, and moving all to an odd position costs `even_count`.
4. Return the minimum of the two counts as the result.

**Why it works:** The key invariant is that moving chips between positions of the same parity costs zero. Therefore, the only cost incurred is from moving chips from even positions to odd positions or vice versa. By counting the chips in each parity group, we can directly compute the minimum number of costly moves.

## Python Solution

```python
from typing import List

class Solution:
    def minCostToMoveChips(self, position: List[int]) -> int:
        even_count = 0
        odd_count = 0
        
        for pos in position:
            if pos % 2 == 0:
                even_count += 1
            else:
                odd_count += 1
        
        return min(even_count, odd_count)
```

In this Python implementation, we iterate through each chip's position and count how many are on even positions and how many are on odd positions. The final result is the minimum of the two counts, reflecting the cheapest cost to align all chips on positions of the same parity.

## Go Solution

```go
func minCostToMoveChips(position []int) int {
    evenCount := 0
    oddCount := 0
    
    for _, pos := range position {
        if pos % 2 == 0 {
            evenCount++
        } else {
            oddCount++
        }
    }
    
    if evenCount < oddCount {
        return evenCount
    }
    return oddCount
}
```

The Go implementation follows the same logic. We use two integer counters and iterate over the slice. Unlike Python, Go does not have a built-in `min` function for integers, so we explicitly compare `evenCount` and `oddCount` to return the smaller value.

## Edge Cases

1. **Single chip:** If there is only one chip, the cost is zero because no movement is required. Both Python and Go solutions correctly return zero in this scenario.
2. **All chips at the same position:** No moves are required if all chips are already aligned. The algorithm correctly counts all chips as either even or odd and returns zero if all chips share the same parity and position.
3. **Maximum integer values:** Positions can be as large as `10^9`. The solution does not iterate over positions or attempt to allocate arrays up to the maximum value, so it handles large numbers safely. Counting parity is sufficient.

This approach is robust, efficient, and works correctly for all input constraints.
