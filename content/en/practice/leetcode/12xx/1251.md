## Solution

## Problem Understanding

This problem asks us to compute the **average selling price** for each product based on two tables: `Prices` and `UnitsSold`. The `Prices` table defines price intervals for each product, indicating that a product may have different prices over time. The `UnitsSold` table records the number of units sold for each product on specific dates.

The key challenge is matching each sale with the correct price interval. Once matched, the average selling price is calculated as the **total revenue divided by total units sold** for each product. The result must be rounded to two decimal places, and if a product has no units sold, its average price defaults to 0.

Constraints ensure that price intervals do not overlap for the same product, simplifying the matching process. There may be duplicate rows in `UnitsSold`, so summing units correctly is essential.

Important edge cases include products with no sales, products with only one sale, and ensuring date ranges in `Prices` correctly cover the `purchase_date` in `UnitsSold`.

## Approaches

The **brute-force approach** would be to iterate through each sale in `UnitsSold` and then scan all price intervals in `Prices` for that product to find the matching interval. This guarantees correctness because each sale is matched to the correct price, but it is inefficient because it could result in an O(N*M) operation, where N is the number of sales and M is the number of price intervals. For large datasets, this approach would be too slow.

The **optimal approach** uses a SQL-style **join on conditions**: we can join `UnitsSold` and `Prices` based on `product_id` and where `purchase_date` falls within `[start_date, end_date]`. Once joined, we can compute total revenue (`units * price`) and sum the units grouped by `product_id`. Finally, dividing total revenue by total units sold gives the average price, rounded to two decimal places. This avoids nested loops and leverages the database engine for efficient computation.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(N*M) | O(1) | Check each unit sale against all price intervals |
| Optimal | O(N + M) | O(N + M) | SQL-style join on product_id and date ranges, then aggregation |

## Algorithm Walkthrough

1. Start by **joining `UnitsSold` and `Prices`** on `product_id` and ensure the `purchase_date` is between `start_date` and `end_date`. This aligns each sale with the correct price interval.
2. Compute **total revenue per row** as `units * price`.
3. **Aggregate by `product_id`**, summing the total revenue and total units sold.
4. Calculate the **average selling price** as `total_revenue / total_units`. Use `ROUND` to limit the result to two decimal places.
5. For products with no sales, ensure the **average selling price defaults to 0** by performing a LEFT JOIN with the set of all products from `Prices`.
6. Return the final table with `product_id` and `average_price` in any order.

**Why it works:**

The join guarantees that each unit sold is matched with the correct price for the date it was sold. Aggregating revenue and units ensures that the average price reflects weighted contributions of different sales at different prices. The rounding and zero-default rules handle formatting and edge cases.

## Python Solution

```python
import sqlite3

# Assuming this is a LeetCode SQL-style Python execution
def average_selling_price():
    query = """
    WITH Revenue AS (
        SELECT u.product_id,
               SUM(u.units * p.price) AS total_revenue,
               SUM(u.units) AS total_units
        FROM UnitsSold u
        JOIN Prices p
          ON u.product_id = p.product_id
         AND u.purchase_date BETWEEN p.start_date AND p.end_date
        GROUP BY u.product_id
    )
    SELECT p.product_id,
           ROUND(COALESCE(r.total_revenue / r.total_units, 0), 2) AS average_price
    FROM (SELECT DISTINCT product_id FROM Prices) p
    LEFT JOIN Revenue r
      ON p.product_id = r.product_id;
    """
    return query
```

**Implementation Walkthrough:**

The `WITH Revenue AS` clause computes total revenue and total units for each product. The join ensures that each sale is matched to its corresponding price period. Then, the main query ensures every product from `Prices` is included using `LEFT JOIN` and applies `COALESCE` to handle products without sales. `ROUND(..., 2)` formats the final result as required.
