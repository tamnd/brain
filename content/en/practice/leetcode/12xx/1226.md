# LeetCode 1226 - The Dining Philosophers

## Problem Understanding

This problem is a classic synchronization and concurrency challenge. Five philosophers sit around a circular table, and there is one fork placed between every pair of adjacent philosophers. Since there are five philosophers, there are also five forks total.

Each philosopher alternates between thinking and eating. To eat, a philosopher must hold both the left fork and the right fork at the same time. A fork can only be used by one philosopher at a time, so concurrent access must be coordinated carefully.

The challenge is to implement a thread-safe solution that avoids deadlock and starvation. Multiple threads will call the `wantsToEat` method concurrently, where each thread represents one philosopher attempting to eat. The implementation must guarantee that philosophers can continue eating indefinitely without the system becoming stuck.

The input to the method is not a traditional data structure like an array or string. Instead, the problem provides:

- The philosopher ID, from `0` to `4`
- Callback functions for picking up forks
- A callback function for eating
- Callback functions for putting forks down

The expected behavior is that the callbacks are executed in a valid order while maintaining synchronization correctness.

The constraints are relatively small, with each philosopher attempting to eat at most 60 times, but the difficulty comes from concurrent correctness rather than input size. The implementation must prevent:

- Deadlock, where every philosopher waits forever
- Race conditions, where multiple philosophers use the same fork simultaneously
- Starvation, where some philosopher never gets a chance to eat

One important edge case is when all philosophers attempt to eat simultaneously. A naive implementation can easily deadlock if every philosopher picks up one fork and waits forever for the second. Another subtle issue is repeated calls for the same philosopher before previous calls finish. The implementation must therefore be fully thread-safe and reentrant.

The problem guarantees that philosophers only interact through the provided shared object, which means synchronization primitives inside the class are sufficient to coordinate all activity.

## Approaches

### Brute Force Approach

A straightforward approach is to associate a mutex or lock with each fork. Each philosopher attempts to:

1. Lock the left fork
2. Lock the right fork
3. Eat
4. Unlock both forks

This solution appears reasonable because locks prevent simultaneous fork usage. However, it contains a serious flaw.

If all philosophers simultaneously pick up their left fork first, every philosopher will then wait forever for the right fork. Since each right fork is already held by a neighboring philosopher, the system reaches a circular wait condition, producing deadlock.

For example:

- Philosopher 0 holds fork 0 and waits for fork 1
- Philosopher 1 holds fork 1 and waits for fork 2
- Philosopher 2 holds fork 2 and waits for fork 3
- Philosopher 3 holds fork 3 and waits for fork 4
- Philosopher 4 holds fork 4 and waits for fork 0

No philosopher can proceed.

Although the brute-force approach correctly protects forks from concurrent access, it fails to guarantee progress.

### Optimal Approach

The key insight is that deadlock only occurs when all philosophers are allowed to compete for forks simultaneously.

To eliminate deadlock, we can limit the number of philosophers attempting to acquire forks at the same time. A common strategy is to use a semaphore initialized to `4`.

Since there are only five philosophers, allowing at most four philosophers to compete guarantees that at least one philosopher can always obtain both forks and complete eating. This breaks the circular wait condition required for deadlock.

The algorithm works as follows:

1. Use a semaphore with capacity `4`
2. Before attempting to pick up forks, a philosopher acquires the semaphore
3. The philosopher locks both required forks
4. The philosopher eats
5. The philosopher releases both forks
6. The philosopher releases the semaphore

This approach guarantees:

- Mutual exclusion on forks
- No deadlock
- No starvation under fair scheduling

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(1) per call | O(5) | Can deadlock when all philosophers grab one fork |
| Optimal | O(1) per call | O(5) | Uses semaphore to prevent circular wait |

## Algorithm Walkthrough

1. Create a lock for each fork.

Since there are five forks, we maintain an array of five mutexes or locks. Each lock represents exclusive ownership of one fork.
2. Create a semaphore initialized to `4`.

This semaphore controls how many philosophers may attempt to acquire forks simultaneously. Allowing at most four philosophers guarantees that at least one philosopher can acquire both forks and proceed.
3. Determine the indices of the left and right forks.

For philosopher `i`:

- Left fork = `i`
- Right fork = `(i + 1) % 5`

The modulo operation wraps around the circular table.
4. Acquire the semaphore before touching any forks.

This step prevents all five philosophers from simultaneously entering the fork acquisition phase.
5. Lock the left fork.

Once the philosopher is permitted to proceed, they acquire exclusive access to their left fork.
6. Execute `pickLeftFork()`.

This callback records the fork pickup action.
7. Lock the right fork.

The philosopher now acquires the second required fork.
8. Execute `pickRightFork()`.

This callback records the second fork pickup.
9. Execute `eat()`.

Since both forks are held, the philosopher may safely eat.
10. Execute `putRightFork()` and unlock the right fork.

The philosopher releases the right fork after finishing.
11. Execute `putLeftFork()` and unlock the left fork.

The philosopher releases the left fork.
12. Release the semaphore.

Another waiting philosopher may now attempt to eat.

### Why it works

The critical property is that deadlock requires all five philosophers to hold one fork while waiting for another. By limiting concurrency to four philosophers, at least one philosopher can always obtain both forks and complete eating. Once that philosopher releases forks, progress continues.

Fork locks guarantee mutual exclusion, while the semaphore removes the possibility of circular waiting.

## Python Solution

```python
from threading import Lock, Semaphore
from typing import Callable

class DiningPhilosophers:

    def __init__(self):
        self.forks = [Lock() for _ in range(5)]
        self.semaphore = Semaphore(4)

    # call the functions directly to execute, for example, eat()
    def wantsToEat(self,
                   philosopher: int,
                   pickLeftFork: 'Callable[[], None]',
                   pickRightFork: 'Callable[[], None]',
                   eat: 'Callable[[], None]',
                   putLeftFork: 'Callable[[], None]',
                   putRightFork: 'Callable[[], None]') -> None:

        left_fork = philosopher
        right_fork = (philosopher + 1) % 5

        self.semaphore.acquire()

        self.forks[left_fork].acquire()
        pickLeftFork()

        self.forks[right_fork].acquire()
        pickRightFork()

        eat()

        putRightFork()
        self.forks[right_fork].release()

        putLeftFork()
        self.forks[left_fork].release()

        self.semaphore.release()
```

The implementation begins by initializing five locks, one for each fork. These locks ensure that no two philosophers can use the same fork simultaneously.

A semaphore with capacity `4` is also created. This is the core deadlock prevention mechanism.

Inside `wantsToEat`, the philosopher first computes which forks they need. The left fork corresponds directly to the philosopher index, while the right fork wraps around the circular table using modulo arithmetic.

The philosopher acquires the semaphore before attempting to lock forks. This prevents all five philosophers from competing simultaneously.

Once permitted, the philosopher acquires the left fork lock and calls `pickLeftFork()`. The same process is repeated for the right fork.

After both forks are held, the philosopher calls `eat()`.

Finally, the philosopher releases both forks in reverse order and releases the semaphore so another philosopher may proceed.

## Go Solution

```go
package main

import "sync"

type DiningPhilosophers struct {
	forks [5]sync.Mutex
	room  chan struct{}
}

func Constructor() DiningPhilosophers {
	dp := DiningPhilosophers{
		room: make(chan struct{}, 4),
	}

	return dp
}

func (this *DiningPhilosophers) wantsToEat(
	philosopher int,
	pickLeftFork func(),
	pickRightFork func(),
	eat func(),
	putLeftFork func(),
	putRightFork func(),
) {

	leftFork := philosopher
	rightFork := (philosopher + 1) % 5

	this.room <- struct{}{}

	this.forks[leftFork].Lock()
	pickLeftFork()

	this.forks[rightFork].Lock()
	pickRightFork()

	eat()

	putRightFork()
	this.forks[rightFork].Unlock()

	putLeftFork()
	this.forks[leftFork].Unlock()

	<-this.room
}
```

The Go implementation uses `sync.Mutex` for fork synchronization. Instead of a semaphore class, Go uses a buffered channel with capacity `4` to simulate semaphore behavior.

Sending into the channel acquires a semaphore slot, while receiving from the channel releases it.

Unlike Python, Go requires an explicit constructor to initialize the buffered channel. Mutexes themselves do not require separate initialization.

Integer overflow is not a concern because philosopher indices are always in the range `0` through `4`.

## Edge Cases

One important edge case occurs when all five philosophers attempt to eat simultaneously. This is the classic deadlock scenario that breaks naive implementations. Without the semaphore restriction, each philosopher could acquire one fork and wait forever for the second. The semaphore prevents this by allowing at most four philosophers into the fork acquisition phase simultaneously.

Another important edge case is repeated concurrent calls for the same philosopher. The problem explicitly states that a philosopher's function may be called again before a previous call finishes. A non-thread-safe implementation could corrupt shared state or incorrectly release locks. Since the implementation relies only on synchronization primitives and does not store philosopher-specific mutable state, repeated calls remain safe.

A third edge case involves fork contention between neighboring philosophers. For example, philosophers `0` and `1` both need fork `1`. Without mutex protection, both philosophers might incorrectly use the fork simultaneously. The fork locks guarantee mutual exclusion, ensuring that only one philosopher owns a fork at any given moment.

A final subtle edge case is circular indexing for philosopher `4`. Philosopher `4` uses left fork `4` and right fork `(4 + 1) % 5`, which correctly wraps to fork `0`. Incorrect indexing logic could cause out-of-bounds access or incorrect fork assignment. The modulo operation handles the circular table structure safely and correctly.
