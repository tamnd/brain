**Difficulty:** 🔴 Hard  
**Topics:** Backtracking

## Solution

## Problem Understanding

The problem asks us to tile an `n x m` rectangle using the minimum possible number of squares, where every square must have integer side lengths. The entire rectangle must be covered exactly, with no overlaps and no uncovered cells.

The input consists of two integers:

- `n`, the height of the rectangle
- `m`, the width of the rectangle

The output is a single integer representing the smallest number of squares needed to completely tile the rectangle.

For example, a `2 x 3` rectangle can be tiled with one `2 x 2` square and two `1 x 1` squares, giving a total of `3` squares.

The constraints are very important:

```
1 <= n, m <= 13
```

These dimensions are small enough that exponential search with aggressive pruning becomes feasible. A naive brute-force search over all possible tilings would still be far too slow, but the small upper bound allows us to use optimized backtracking with pruning techniques.

Several edge cases are important:

- If `n == m`, the answer is always `1` because the entire rectangle is already a square.
- Thin rectangles such as `1 x 13` force the solution to use only `1 x 1` squares.
- Certain dimensions such as `11 x 13` are notoriously difficult because greedy placement strategies fail to produce the optimal answer.
- Symmetry matters. A `2 x 3` rectangle is equivalent to `3 x 2`, so implementations should normalize dimensions where possible.

The problem guarantees valid positive dimensions, so we never need to handle zero or negative inputs.

## Approaches

### Brute Force Approach

The brute-force strategy tries every possible way to place squares into the rectangle.

At each step:

1. Find an uncovered cell.
2. Try every square size that can fit there.
3. Recursively continue until the rectangle is fully tiled.
4. Track the minimum number of squares used across all valid configurations.

This approach is correct because it exhaustively explores every possible tiling arrangement. Since every legal placement is considered, the optimal arrangement must eventually be found.

However, the search space grows explosively. Even small rectangles can produce enormous branching factors because each uncovered position may allow multiple square sizes. Many recursive states are repeated in different orders, making naive brute force impractical.

### Optimal Approach, Backtracking with Height Profile Pruning

The key observation is that we do not need to track the entire 2D board explicitly.

Instead, we can represent the current tiling state using a height profile:

- Each column stores how much of that column has already been filled.
- The rectangle is complete when every column reaches height `n`.

At each recursive step:

1. Find the leftmost column with the minimum height.
2. Attempt to place the largest possible square at that position.
3. Recursively continue.
4. Backtrack afterward.

This dramatically reduces the state space because many equivalent board configurations collapse into the same height profile.

Several pruning optimizations make the solution fast enough:

- If the current square count is already worse than the best known answer, stop searching.
- Always place squares at the lowest uncovered position.
- Try larger squares first because they usually lead to better solutions sooner, improving pruning effectiveness.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential, extremely large | O(n × m) recursion state | Explores nearly all tilings |
| Optimal Backtracking | Exponential in worst case, heavily pruned | O(m + recursion depth) | Uses height compression and pruning |

## Algorithm Walkthrough

### Step 1: Normalize Dimensions

Swap the dimensions if necessary so that:

```
n <= m
```

This reduces redundant symmetric cases and slightly improves performance.

### Step 2: Represent the Rectangle Using Heights

Instead of storing every cell, maintain an array:

```
heights[col]
```

Each value represents how many rows in that column are already filled.

For example:

```
heights = [2, 2, 1, 1]
```

means:

- Columns 0 and 1 are filled up to row 2
- Columns 2 and 3 are filled up to row 1

### Step 3: Detect Completion

The rectangle is fully tiled when every column reaches height `n`.

Equivalent condition:

```
min(heights) == n
```

### Step 4: Find the Next Placement Position

Choose the column with the smallest height.

This gives the topmost-leftmost uncovered position.

Why this works:

- It prevents gaps from forming underneath later placements.
- It standardizes recursive exploration.
- It reduces duplicate states.

### Step 5: Compute Maximum Square Size

Suppose the minimum height occurs at column `idx`.

We attempt to place squares beginning at:

```
row = heights[idx]
col = idx
```

The largest possible square must satisfy:

1. It stays inside the rectangle bounds.
2. All involved columns have the same current height.

This guarantees the square fits cleanly.

### Step 6: Try Squares from Largest to Smallest

Attempt larger squares first.

Why:

- Large squares usually reduce the total count.
- Good solutions are discovered earlier.
- Earlier good solutions improve pruning.

For each candidate square:

1. Increase the affected column heights.
2. Recursively solve the remaining space.
3. Undo the changes afterward.

### Step 7: Prune Inefficient States

If:

```
used_squares >= best_answer
```

stop recursion immediately.

There is no reason to continue because the current branch cannot improve the result.

### Why it works

The algorithm always explores every valid tiling configuration reachable from the current state. Since every square placement is recursively considered, the optimal solution cannot be missed.

The height profile uniquely captures the current frontier of the partially tiled rectangle. By always expanding from the minimum-height position, the algorithm avoids invalid disconnected gaps and ensures all reachable states are systematically explored.

Pruning does not affect correctness because branches are only discarded when they are already guaranteed to be no better than the current best solution.

## Python Solution

```python
class Solution:
    def tilingRectangle(self, n: int, m: int) -> int:
        if n > m:
            n, m = m, n

        heights = [0] * m
        self.best = n * m

        def backtrack(used: int) -> None:
            if used >= self.best:
                return

            min_height = min(heights)

            if min_height == n:
                self.best = min(self.best, used)
                return

            idx = heights.index(min_height)

            max_size = 1

            while (
                idx + max_size <= m and
                min_height + max_size <= n and
                all(heights[idx + k] == min_height for k in range(max_size))
            ):
                max_size += 1

            max_size -= 1

            for size in range(max_size, 0, -1):
                for k in range(size):
                    heights[idx + k] += size

                backtrack(used + 1)

                for k in range(size):
                    heights[idx + k] -= size

        backtrack(0)

        return self.best
```

The implementation begins by normalizing the dimensions so the smaller side is always treated as the height. This slightly reduces the number of recursive states.

The `heights` array represents the current filled height of each column. Initially every column is empty, so all heights are zero.

The recursive `backtrack` function tracks how many squares have already been used. If this value is not better than the current best answer, recursion stops immediately.

The algorithm then finds the minimum column height. This identifies the next uncovered position that must be filled.

To determine the largest valid square:

- The square cannot exceed the rectangle boundary.
- Every column involved must have the same height.

The loop incrementally checks whether larger squares remain valid.

Squares are attempted from largest to smallest. Each placement temporarily increases the relevant heights, recursion continues, and afterward the state is restored through backtracking.

Eventually all columns reach height `n`, meaning the rectangle is completely tiled.

## Go Solution

```go
func tilingRectangle(n int, m int) int {
	if n > m {
		n, m = m, n
	}

	heights := make([]int, m)
	best := n * m

	var backtrack func(int)

	backtrack = func(used int) {
		if used >= best {
			return
		}

		minHeight := n + 1
		idx := 0

		for i := 0; i < m; i++ {
			if heights[i] < minHeight {
				minHeight = heights[i]
				idx = i
			}
		}

		if minHeight == n {
			if used < best {
				best = used
			}
			return
		}

		maxSize := 1

		for idx+maxSize <= m &&
			minHeight+maxSize <= n {

			valid := true

			for k := 0; k < maxSize; k++ {
				if heights[idx+k] != minHeight {
					valid = false
					break
				}
			}

			if !valid {
				break
			}

			maxSize++
		}

		maxSize--

		for size := maxSize; size >= 1; size-- {
			for k := 0; k < size; k++ {
				heights[idx+k] += size
			}

			backtrack(used + 1)

			for k := 0; k < size; k++ {
				heights[idx+k] -= size
			}
		}
	}

	backtrack(0)

	return best
}
```

The Go implementation follows the same algorithmic structure as the Python version.

A slice is used for the height profile. Since Go does not support nested closures with implicit variable capture as flexibly as Python, the recursive function is declared using a variable assignment:

```go
var backtrack func(int)
```

Integer overflow is not a concern because the constraints are very small. Backtracking state restoration is handled manually by subtracting the square size after recursion returns.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | Exponential | Backtracking explores many tilings |
| Space | O(m + recursion depth) | Height profile plus recursive stack |

The worst-case time complexity remains exponential because the problem fundamentally requires exploring combinations of square placements.

However, the practical runtime is dramatically reduced by:

- Height-profile compression
- Largest-square-first ordering
- Branch pruning
- Canonical placement selection

Since `m <= 13`, the recursive state space becomes manageable.

The space usage is relatively small because only the height profile and recursion stack are stored.

## Edge Cases

### Perfect Square Rectangles

When `n == m`, the entire rectangle is already a square. A naive recursive implementation might still attempt unnecessary subdivision, greatly increasing runtime.

This implementation handles the case naturally because the largest possible square immediately fills the entire rectangle, producing answer `1`.

### Thin Rectangles

Cases like `1 x 13` are deceptively important. Only `1 x 1` squares can fit, so the answer equals the longer dimension.

Algorithms that incorrectly assume larger square placements are always available can fail here. The height validation logic correctly restricts square sizes to legal placements only.

### Difficult Non-Greedy Cases

The classic `11 x 13` rectangle demonstrates why greedy placement alone is insufficient. Always placing the largest square can lead to suboptimal layouts.

This implementation remains correct because it explores all valid placements through backtracking. Large squares are merely attempted first for efficiency, not forced permanently.
