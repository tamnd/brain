**Difficulty:** 🔴 Hard  
**Topics:** Array, Breadth-First Search, Matrix

## Solution

## Problem Understanding

The problem gives us a two-dimensional grid where each cell contains either `0` or `1`. A value of `0` represents an empty cell that can be walked through normally, while a value of `1` represents an obstacle. We start at the top-left corner `(0, 0)` and want to reach the bottom-right corner `(m - 1, n - 1)` using the fewest possible moves.

Movement is restricted to the four cardinal directions: up, down, left, and right. Each move costs exactly one step.

The important twist is that we are allowed to eliminate up to `k` obstacles during the path. This means that if we move into a cell containing `1`, we may consume one elimination and continue moving through it as if it were empty. However, once we have used all `k` eliminations, we can no longer move through obstacle cells.

The goal is to compute the minimum number of steps required to reach the destination. If no valid path exists even after using up to `k` eliminations, we return `-1`.

The constraints are important because they strongly influence the algorithm choice:

- `m` and `n` are each at most `40`
- The total number of cells is at most `1600`
- `k` can be as large as `m * n`

These limits are too large for brute-force path enumeration because the number of possible paths grows exponentially. However, the grid is still small enough for graph traversal algorithms like Breadth-First Search with additional state tracking.

There are several important edge cases to consider:

- A grid with only one cell. Since the start is already the destination, the answer is `0`.
- Cases where `k` is very large. If we can eliminate enough obstacles, the shortest path becomes the Manhattan distance.
- Cases where obstacles completely block the destination unless enough eliminations are available.
- Revisiting the same cell with different remaining eliminations. This is the key subtlety that breaks naive BFS implementations.
- Dense obstacle layouts where using eliminations too early can lead to failure later.

## Approaches

### Brute Force Approach

A brute-force solution would attempt to explore every possible path from the start to the destination. At each step, we could move in four directions and decide whether to eliminate an obstacle if necessary.

This approach is conceptually correct because it eventually explores every reachable route. However, the number of possible paths grows exponentially with grid size. Since cycles are possible, we would also need additional bookkeeping to avoid infinite recursion.

Even with pruning, the search space becomes enormous. In the worst case, each cell could branch into multiple new states depending on how many eliminations remain. This quickly becomes computationally infeasible for a `40 x 40` grid.

### Key Insight

The important observation is that this is fundamentally a shortest path problem on an unweighted graph.

Breadth-First Search is ideal for finding the shortest path in an unweighted graph because BFS explores states level by level. The first time we reach the destination, we are guaranteed to have used the minimum number of steps.

However, there is a subtle complication. A state is not defined only by position `(row, col)`. The number of remaining eliminations also matters.

For example, reaching cell `(2, 3)` with `5` eliminations left is much better than reaching the same cell with `0` eliminations left. A naive visited set using only coordinates would incorrectly discard potentially optimal paths.

Therefore, the correct BFS state must include:

- Current row
- Current column
- Remaining obstacle eliminations

This transforms the problem into a graph search over a three-dimensional state space.

We can further optimize by observing that if we have already visited a cell with more remaining eliminations, then revisiting it with fewer eliminations is never useful.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(4^(m*n)) | O(m*n) | Explores all possible paths recursively |
| Optimal BFS | O(m * n * k) | O(m * n * k) | BFS over position plus remaining eliminations |

## Algorithm Walkthrough

1. First, handle the special optimization case where `k` is large enough to ignore most obstacles entirely.

The shortest possible path from the top-left corner to the bottom-right corner requires exactly:

$(m-1)+(n-1)$

moves.

If `k` is at least this value, then we can always travel directly toward the destination and eliminate any obstacles encountered along the way. In this case, we immediately return the Manhattan distance.
2. Initialize a BFS queue.

Each queue entry stores:

- Current row
- Current column
- Remaining eliminations
- Steps taken so far

BFS is appropriate because every move has equal cost, which guarantees shortest-path correctness.
3. Initialize a visited structure.

We cannot simply mark `(row, col)` as visited because reaching the same cell with different remaining eliminations represents different states.

Instead, we store the maximum remaining eliminations seen for each cell.

If we revisit a cell with fewer or equal eliminations remaining, we skip processing because that state is strictly worse than a previously explored one.
4. Start BFS from `(0, 0)`.

Initially:

- Position is `(0, 0)`
- Remaining eliminations is `k`
- Steps is `0`
5. Process the BFS queue level by level.

For each state:

- Try moving in the four directions
- Check bounds
- Determine whether the next cell is an obstacle
6. Handle obstacle traversal.

If the next cell contains `1`, we must spend one elimination.

The new remaining eliminations become:

$newK = currentK - grid[nextRow][nextCol]$

If `newK` becomes negative, this move is invalid.
7. Check whether the destination has been reached.

Because BFS explores states in increasing order of distance, the first time we reach the bottom-right corner is guaranteed to be the minimum number of steps.
8. Apply pruning using the visited structure.

If we have already reached this cell with at least as many remaining eliminations, skip the new state.

Otherwise, record the improved state and continue BFS.
9. If BFS finishes without reaching the destination, return `-1`.

### Why it works

BFS guarantees shortest paths in unweighted graphs because states are explored in order of increasing distance from the start.

The critical invariant is that each state consists of both position and remaining eliminations. Two visits to the same cell are not equivalent unless they also have the same elimination budget.

By only discarding states that are strictly worse than previously explored states, we preserve all potentially optimal paths while avoiding redundant work.

## Python Solution

```python
from collections import deque
from typing import List

class Solution:
    def shortestPath(self, grid: List[List[int]], k: int) -> int:
        rows = len(grid)
        cols = len(grid[0])

        # Minimum possible path length
        if k >= rows + cols - 2:
            return rows + cols - 2

        # visited[row][col] stores the maximum remaining k
        # seen so far at this position
        visited = [[-1] * cols for _ in range(rows)]
        visited[0][0] = k

        queue = deque([(0, 0, k, 0)])

        directions = [
            (1, 0),
            (-1, 0),
            (0, 1),
            (0, -1)
        ]

        while queue:
            row, col, remaining_k, steps = queue.popleft()

            # Reached destination
            if row == rows - 1 and col == cols - 1:
                return steps

            for dr, dc in directions:
                next_row = row + dr
                next_col = col + dc

                # Check bounds
                if (
                    next_row < 0 or
                    next_row >= rows or
                    next_col < 0 or
                    next_col >= cols
                ):
                    continue

                # Spend elimination if obstacle
                next_k = remaining_k - grid[next_row][next_col]

                # Cannot pass obstacle
                if next_k < 0:
                    continue

                # Skip worse states
                if visited[next_row][next_col] >= next_k:
                    continue

                visited[next_row][next_col] = next_k

                queue.append(
                    (next_row, next_col, next_k, steps + 1)
                )

        return -1
```

The implementation begins by computing the grid dimensions and checking the optimization condition. If `k` is large enough to eliminate obstacles along a direct shortest route, the answer can be returned immediately without running BFS.

The `visited` matrix stores the best remaining elimination count observed for each cell. This is the key optimization that prevents exponential revisiting.

The BFS queue stores full state information. Each iteration removes the next state in shortest-distance order.

For every neighboring cell, the algorithm checks bounds, computes the updated elimination count, and discards invalid states.

The pruning condition:

```python
if visited[next_row][next_col] >= next_k:
```

is crucial. If we have already reached the same cell with more eliminations remaining, the new state can never produce a better future path.

Because BFS processes states in increasing step order, the first time we reach the destination is guaranteed to be optimal.

## Go Solution

```go
package main

import "container/list"

type State struct {
	row   int
	col   int
	k     int
	steps int
}

func shortestPath(grid [][]int, k int) int {
	rows := len(grid)
	cols := len(grid[0])

	// Minimum possible path
	if k >= rows+cols-2 {
		return rows + cols - 2
	}

	visited := make([][]int, rows)
	for i := range visited {
		visited[i] = make([]int, cols)
		for j := range visited[i] {
			visited[i][j] = -1
		}
	}

	visited[0][0] = k

	queue := list.New()
	queue.PushBack(State{0, 0, k, 0})

	directions := [][]int{
		{1, 0},
		{-1, 0},
		{0, 1},
		{0, -1},
	}

	for queue.Len() > 0 {
		front := queue.Front()
		queue.Remove(front)

		current := front.Value.(State)

		if current.row == rows-1 && current.col == cols-1 {
			return current.steps
		}

		for _, dir := range directions {
			nextRow := current.row + dir[0]
			nextCol := current.col + dir[1]

			if nextRow < 0 ||
				nextRow >= rows ||
				nextCol < 0 ||
				nextCol >= cols {
				continue
			}

			nextK := current.k - grid[nextRow][nextCol]

			if nextK < 0 {
				continue
			}

			if visited[nextRow][nextCol] >= nextK {
				continue
			}

			visited[nextRow][nextCol] = nextK

			queue.PushBack(State{
				nextRow,
				nextCol,
				nextK,
				current.steps + 1,
			})
		}
	}

	return -1
}
```

The Go implementation follows the same logic as the Python version but uses Go-specific data structures.

The BFS queue is implemented using `container/list`, which provides efficient front removal and back insertion.

A custom `State` struct stores BFS state information. This makes the queue contents type-safe and readable.

The `visited` matrix is initialized with `-1` values to indicate unseen states.

Unlike Python, Go does not have built-in tuples or deque support, so explicit structs and linked lists are used instead.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m * n * k) | Each cell can be processed with different remaining elimination counts |
| Space | O(m * n * k) | BFS queue and visited states may store all reachable states |

The state space consists of:

$m \times n \times (k+1)$

possible states.

Each state is processed at most once in the optimized BFS because inferior revisits are pruned.

For each processed state, we examine at most four neighbors, giving total time complexity proportional to the total number of reachable states.

## Edge Cases

One important edge case is a single-cell grid such as `[[0]]`. In this situation, the start and destination are the same cell. A naive BFS implementation might incorrectly return `1` because it assumes movement is required. The implementation handles this naturally because the starting state is immediately recognized as the destination with `0` steps.

Another critical edge case occurs when `k` is extremely large. For example, if `k` exceeds the Manhattan distance between the start and destination, then obstacles effectively become irrelevant because we can eliminate every obstacle encountered along the shortest direct path. Without the optimization check, the algorithm would still work correctly, but it would perform unnecessary BFS exploration. The implementation avoids this overhead with the early return condition.

A subtle but extremely important edge case involves revisiting the same cell with different remaining elimination counts. Suppose we reach a cell once with `0` eliminations remaining and later reach it again with `2` eliminations remaining. A naive visited set based only on coordinates would incorrectly discard the second state, even though it is clearly more powerful for future exploration. The implementation solves this by storing the maximum remaining eliminations seen for each cell and only pruning strictly worse states.

Another tricky case involves grids where the shortest geometric route is not the optimal valid route. Sometimes using obstacle eliminations too early prevents reaching the destination later. BFS correctly handles this because it explores all shortest-distance states simultaneously rather than greedily committing to a single path.
