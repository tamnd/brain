# LeetCode 1225 - Report Contiguous Dates

## Problem Understanding

The problem provides two database tables, `Failed` and `Succeeded`, which together represent the result of a task that runs exactly once every day. Every day has exactly one outcome, either the task failed or succeeded. The important detail is that we only care about dates inside the range from `2019-01-01` to `2019-12-31`, inclusive. Any data outside this range must be ignored.

The goal is to report continuous periods where the task status remains the same. A continuous period means consecutive calendar days where every task has the same outcome. For each such interval, we must return three columns:

- `period_state`, which is either `'failed'` or `'succeeded'`
- `start_date`, the first day in the interval
- `end_date`, the last day in the interval

The result must be ordered by `start_date`.

In simpler terms, we want to reconstruct the system timeline for 2019 and compress consecutive days with the same status into contiguous ranges.

The input consists of two independent tables. `Failed` contains all days where tasks failed, while `Succeeded` contains all days where tasks succeeded. Since exactly one task runs every day and every task either succeeds or fails, the two tables together represent the full state of the system.

The output should merge neighboring days with identical status into a single interval. For example, if tasks succeeded on January 1, January 2, and January 3, we should not return three separate rows. Instead, we should report one interval from January 1 to January 3.

Although the problem does not explicitly list constraints, the date range is fixed to a single year. This strongly suggests that efficiency is less about raw scale and more about writing a clean SQL solution that correctly groups consecutive dates.

Several edge cases are important. Data outside 2019 must be ignored completely, even if it forms part of a larger continuous sequence. Another tricky case occurs when only one day exists in a period, because the start and end dates should be identical. We also need to correctly handle alternating success and failure days, where every interval spans only one day.

## Approaches

### Brute Force Approach

A brute force strategy would generate every date from `2019-01-01` through `2019-12-31`, determine whether each day belongs to the failed or succeeded table, and then iterate through the ordered timeline while manually building intervals whenever the state changes.

This works because every day has exactly one state. By reconstructing the entire calendar, we can compare each day to the previous one and decide whether to continue the current interval or start a new one.

However, this approach is cumbersome in SQL. Generating a calendar table, joining against both tables, and manually scanning transitions introduces unnecessary complexity.

### Optimal Approach

The key observation is that contiguous date intervals can be identified using a grouping trick with window functions.

If we sort dates within the same state and assign a row number, the difference between the date and its row number remains constant for consecutive days. This creates a natural grouping key.

For example:

| Date | Row Number | Date - Row Number |
| --- | --- | --- |
| 2019-01-01 | 1 | constant |
| 2019-01-02 | 2 | constant |
| 2019-01-03 | 3 | constant |

As long as dates are consecutive, this computed value does not change. Once there is a gap or the status changes, the grouping value changes as well.

We first combine failed and succeeded dates into one dataset using `UNION ALL`, label each row with its status, filter to 2019, and then use `ROW_NUMBER()` to construct contiguous groups. Finally, we aggregate each group using `MIN(date)` and `MAX(date)`.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Generate every date and manually track transitions |
| Optimal | O(n log n) | O(n) | Uses sorting and window functions to group contiguous dates |

## Algorithm Walkthrough

1. Combine the `Failed` and `Succeeded` tables into a single unified dataset. Add a column called `period_state` to distinguish whether the row represents `'failed'` or `'succeeded'`.
2. Filter the records so that only dates between `2019-01-01` and `2019-12-31` remain. This ensures we ignore irrelevant dates outside the required reporting period.
3. Use `ROW_NUMBER()` partitioned by `period_state` and ordered by date. This assigns sequential numbers to rows of the same status.
4. Compute a grouping key by subtracting the row number interval from the actual date. Consecutive dates will produce the same grouping value.
5. Group rows by `period_state` and the computed grouping key. Since consecutive dates share the same key, they naturally collapse into a contiguous interval.
6. Use `MIN(date)` to determine the interval start and `MAX(date)` to determine the interval end.
7. Sort the final result by `start_date` so the report appears in chronological order.

### Why it works

The correctness comes from the invariant that consecutive calendar dates increase by exactly one day. Since row numbers also increase by one for consecutive rows of the same state, subtracting the row number offset from the date yields a constant value for every uninterrupted interval. Whenever continuity breaks, either due to a missing day or a status change, the grouping key changes, which correctly separates intervals.

## Python Solution

This is a SQL database problem, so the Python section contains the SQL query exactly as it would be submitted on LeetCode.

# Write your MySQL query statement belowWITH combined AS (    SELECT fail_date AS dt, 'failed' AS period_state    FROM Failed    WHERE fail_date BETWEEN '2019-01-01' AND '2019-12-31'
    UNION ALL
    SELECT success_date AS dt, 'succeeded' AS period_state    FROM Succeeded    WHERE success_date BETWEEN '2019-01-01' AND '2019-12-31'),grouped AS (    SELECT        dt,        period_state,        DATE_SUB(            dt,            INTERVAL ROW_NUMBER() OVER (                PARTITION BY period_state                ORDER BY dt            ) DAY        ) AS grp    FROM combined)SELECT    period_state,    MIN(dt) AS start_date,    MAX(dt) AS end_dateFROM groupedGROUP BY period_state, grpORDER BY start_date;

The implementation begins by constructing a common table expression called `combined`, which merges both tables into a unified structure. Each row contains a date and its corresponding status.

The next common table expression, `grouped`, applies the window function `ROW_NUMBER()` separately for each status. The computed date difference becomes the grouping key.

Finally, rows are grouped by both status and grouping key. Using `MIN(dt)` and `MAX(dt)` produces the interval boundaries.

## Edge Cases

One important edge case occurs when a period contains exactly one day. A naive implementation may accidentally omit it or produce incorrect bounds. This solution handles it naturally because `MIN(dt)` and `MAX(dt)` both evaluate to the same value.

Another edge case happens when dates outside 2019 exist. Since the problem explicitly restricts reporting to the 2019 calendar year, including these dates would produce incorrect intervals. The filtering step prevents this issue.

A third tricky case involves interruptions in continuity. For example, if succeeded days occur on January 1 and January 3, these cannot be merged into one interval because January 2 is missing. The row number grouping trick correctly separates them because the computed grouping key changes when continuity breaks.
