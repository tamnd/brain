## Solution

## Problem Understanding

The problem asks us to implement a search suggestion system for a list of products. You are given an array of unique lowercase strings `products` representing product names and a string `searchWord` representing what the user is typing. After each character of `searchWord` is typed, we need to return up to three product names from `products` that start with the prefix typed so far. These suggestions must be the lexicographically smallest matches if more than three exist.

The input `products` can have up to 1000 strings, each up to 3000 characters, and the total sum of characters across all products is bounded by 2 * 10^4. `searchWord` can have up to 1000 characters. The problem guarantees all products are unique, and each character is a lowercase English letter.

Key edge cases include a single product in the list, a searchWord that does not match any product, and products with common prefixes that exceed three matches. The constraints indicate that sorting or prefix-based searching is feasible, but naive scanning of the entire list for every prefix could be too slow.

## Approaches

The **brute-force approach** is straightforward: for each prefix of `searchWord`, iterate through the entire `products` list and collect strings that start with that prefix, sort the matches lexicographically, and take the first three. This is correct but inefficient because it repeatedly scans the full list for each prefix, which could be up to 1000 prefixes, leading to a worst-case time complexity that is roughly O(P * N log N) where P is the length of `searchWord` and N is the number of products.

The **optimal approach** leverages sorting and binary search. First, sort the `products` list lexicographically. Then, for each prefix of `searchWord`, perform a binary search to find the first product that matches the prefix. From that position, we only need to consider the next three products to check if they also match the prefix. This avoids scanning the entire list repeatedly and reduces unnecessary comparisons.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(P * N * L + P * N log N) | O(N) | Check each prefix against all products, sort matches each time. |
| Optimal | O(N log N + P * log N) | O(N) | Sort once, use binary search for prefix matches. Only check next three products per prefix. |

## Algorithm Walkthrough

1. **Sort the products list lexicographically**. This ensures that the first three matching products for any prefix are always the first three occurrences in the sorted list.
2. **Initialize an empty result list**. This will store lists of suggestions after each character.
3. **Iterate through each prefix of `searchWord`**. For each character typed, append it to a current prefix string.
4. **Use binary search to find the first product that is greater than or equal to the prefix**. Python's `bisect_left` or Go's `sort.Search` can locate this position efficiently.
5. **Collect up to three consecutive products from the found index** that start with the prefix. Since the list is sorted, no other products earlier in the list will match.
6. **Append the list of matches** (up to three) to the result list.
7. **Return the result list** after processing all prefixes.

**Why it works:** Sorting guarantees that lexicographically smaller products come first. Binary search ensures that we find the starting point for each prefix efficiently. Since we only need three suggestions, checking three consecutive elements from the search index suffices.

## Python Solution

```python
from typing import List
import bisect

class Solution:
    def suggestedProducts(self, products: List[str], searchWord: str) -> List[List[str]]:
        products.sort()
        result = []
        prefix = ""
        for char in searchWord:
            prefix += char
            i = bisect.bisect_left(products, prefix)
            suggestions = []
            for j in range(i, min(i + 3, len(products))):
                if products[j].startswith(prefix):
                    suggestions.append(products[j])
                else:
                    break
            result.append(suggestions)
        return result
```

This implementation first sorts the products list. For each prefix of `searchWord`, it finds the first matching product using `bisect_left`. It then checks at most the next three products to see if they match the prefix and appends them to the results.

## Go Solution

```go
import (
    "sort"
    "strings"
)

func suggestedProducts(products []string, searchWord string) [][]string {
    sort.Strings(products)
    result := [][]string{}
    prefix := ""
    
    for _, char := range searchWord {
        prefix += string(char)
        i := sort.Search(len(products), func(j int) bool {
            return products[j] >= prefix
        })
        suggestions := []string{}
        for j := i; j < len(products) && j < i+3; j++ {
            if strings.HasPrefix(products[j], prefix) {
                suggestions = append(suggestions, products[j])
            } else {
                break
            }
        }
        result = append(result, suggestions)
    }
    return result
}
```

Go handles slices differently, and `strings.HasPrefix` checks prefixes efficiently. We use `sort.Search` for binary search and `append` for dynamic lists. There is no risk of nil vs empty slices in this case, as the result slice is always initialized.

## Edge Cases

One edge case is when `searchWord` contains characters that no product starts with. In this scenario, the result should be a list of empty lists for the remaining characters. The binary search handles this naturally, and the suggestions list remains empty.

Another case is when products share a prefix longer than 3 matches. The implementation ensures only the first three lexicographically smallest products are selected. Sorting beforehand guarantees correctness without additional filtering.

A third case is very long products or `searchWord` close to the maximum length. This tests string comparison efficiency. Python and Go handle string slices efficiently, and since only a few comparisons occur per prefix (3 at most), performance remains within constraints.
