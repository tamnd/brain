## Solution

## Problem Understanding

The problem gives us an array called `groupSizes`, where each index represents a person and each value represents the required size of the group that person belongs to.

If `groupSizes[i] = k`, then person `i` must appear in a group containing exactly `k` people. Every person must belong to exactly one group, and all people in the same group must agree on the group size.

For example, if we have:

```
groupSizes = [3,3,3,3,3,1,3]
```

then person `0` must belong to a group of size `3`, person `1` must also belong to a group of size `3`, and so on. Person `5` must belong to a group of size `1`, meaning they must be alone in their group.

The output is a list of groups, where each group is itself a list of person IDs. Multiple valid answers may exist, and the problem allows returning any valid grouping.

The constraints are relatively small:

- `1 <= n <= 500`
- `1 <= groupSizes[i] <= n`

Since `n` is at most `500`, we do not need extremely advanced optimization, but we should still aim for an efficient linear or near-linear solution.

A few important observations come from the guarantees:

- The input is always valid.
- There is always at least one correct grouping.
- Every person appears exactly once.
- People requiring the same group size can potentially be partitioned into multiple groups.

Some edge cases that can cause issues in naive implementations include:

- Many people requiring group size `1`, where each person must become their own group.
- Multiple independent groups of the same size, such as six people all requiring size `3`.
- Interleaved group sizes, where people with the same requirement are scattered throughout the array.
- Forgetting to reset temporary storage after a group reaches its required size.

## Approaches

### Brute Force Approach

A brute-force strategy would attempt to explicitly search for valid group combinations for each person.

One possible brute-force method is:

- Start with all people ungrouped.
- Pick a person.
- Search for enough other ungrouped people with the same required group size.
- Form a candidate group.
- Continue until all people are assigned.

This works because we only create groups containing compatible people, meaning all members require the same group size.

However, the implementation becomes cumbersome because we repeatedly scan the array searching for compatible members. In the worst case, for every person we may search through many others multiple times, leading to quadratic complexity.

Even though `n = 500` is not enormous, the brute-force method is unnecessarily inefficient and harder to reason about correctly.

### Optimal Approach

The key observation is that people can only be grouped with others who require the exact same group size.

That means we can process people independently by required size.

A hash map is ideal here:

- The key is the required group size.
- The value is the current partially constructed group for that size.

As we iterate through the array:

- Add each person to the appropriate temporary group.
- Once the temporary group reaches the required size, it becomes complete.
- Append it to the answer and reset the temporary storage for that size.

This works efficiently because each person is processed exactly once.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeatedly searches for compatible people |
| Optimal | O(n) | O(n) | Uses a hash map to build groups incrementally |

## Algorithm Walkthrough

1. Create an empty result list that will store all completed groups.
2. Create a hash map where:

- The key is a group size.
- The value is the current partially filled group for that size.

This structure allows us to efficiently track unfinished groups.
3. Iterate through the `groupSizes` array using both the person ID and the required group size.
4. For each person:

- Add the person's ID to the temporary group corresponding to their required size.
5. After adding the person, check whether the temporary group has reached the required size.
6. If the group is now complete:

- Append the group to the final result.
- Reset the temporary group for that size to an empty list.
7. Continue until all people have been processed.
8. Return the final result.

### Why it works

The algorithm maintains the invariant that every temporary group only contains people requiring the same group size.

Whenever a temporary group reaches exactly its required size, it is guaranteed to be valid because:

- Every member requested that exact size.
- The group contains exactly that many people.

Since every person is processed exactly once and added to exactly one completed group, the final result satisfies all problem constraints.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Solution:
    def groupThePeople(self, groupSizes: List[int]) -> List[List[int]]:
        groups = defaultdict(list)
        result = []

        for person_id, group_size in enumerate(groupSizes):
            groups[group_size].append(person_id)

            if len(groups[group_size]) == group_size:
                result.append(groups[group_size])
                groups[group_size] = []

        return result
```

The solution begins by creating a `defaultdict(list)` called `groups`. This allows us to automatically create empty lists for unseen group sizes.

The `result` list stores completed groups.

We iterate through the array using `enumerate`, which gives us both:

- `person_id`
- `group_size`

For each person, we append their ID to the temporary group associated with their required size.

After insertion, we immediately check whether the group has reached the required size. If it has, then the group is complete and valid, so we append it to the answer.

Finally, we reset that temporary group to an empty list so that future people with the same required size can form another group.

Each person is processed exactly once, which gives the algorithm linear time complexity.

## Go Solution

```go
func groupThePeople(groupSizes []int) [][]int {
    groups := make(map[int][]int)
    result := [][]int{}

    for personID, groupSize := range groupSizes {
        groups[groupSize] = append(groups[groupSize], personID)

        if len(groups[groupSize]) == groupSize {
            result = append(result, groups[groupSize])
            groups[groupSize] = []int{}
        }
    }

    return result
}
```

The Go implementation follows the exact same logic as the Python solution.

A `map[int][]int` is used instead of a Python dictionary. Since Go slices are dynamic, appending person IDs works naturally.

One subtle difference is that after completing a group, we explicitly reset the slice to an empty slice:

```go
groups[groupSize] = []int{}
```

This ensures future groups of the same size start fresh.

There are no integer overflow concerns because the constraints are very small.

## Edge Cases

One important edge case occurs when every person requires a group of size `1`. In this scenario, each person must be placed alone in their own group. A buggy implementation might accidentally combine multiple singleton users together. This implementation handles the case naturally because each temporary group immediately reaches size `1` and gets flushed into the result.

Another important case is when many groups share the same required size. For example:

```
[3,3,3,3,3,3]
```

Here, we must create two separate groups of size `3`. A common mistake is forgetting to reset the temporary storage after forming the first complete group. This implementation explicitly resets the list after appending a completed group, ensuring future users form a new group correctly.

A third edge case involves interleaved group sizes, such as:

```
[2,3,2,3,3,2]
```

People requesting the same group size are not necessarily adjacent. Any solution relying on contiguous segments would fail here. The hash map solves this cleanly because it groups people logically by required size rather than by position in the array.

A final subtle edge case is ensuring that every person appears exactly once. If we accidentally reused completed groups or appended references incorrectly, duplicate assignments could occur. Since each person is inserted into exactly one temporary group and each completed group is appended exactly once, the implementation guarantees uniqueness.
