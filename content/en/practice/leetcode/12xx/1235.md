## Solution

## Problem Understanding

This problem asks us to maximize the total profit earned by selecting a subset of non-overlapping jobs.

Each job has three properties:

- `startTime[i]`, the time when the job begins
- `endTime[i]`, the time when the job finishes
- `profit[i]`, the amount earned if we complete that job

We are allowed to choose multiple jobs, but their time intervals cannot overlap. The problem explicitly states that if one job ends at time `X`, another job may start at time `X`. That means intervals are considered compatible when:

```
previous_job.endTime <= next_job.startTime
```

The goal is to return the maximum total profit obtainable from a valid schedule.

The input arrays all have the same length `n`, where:

```
1 <= n <= 5 * 10^4
```

This is a very important constraint. A brute-force search over all subsets would require exponential time and is completely infeasible for `50,000` jobs.

The time values can be as large as `10^9`, which means we cannot build a time-based DP array indexed by timestamps. Instead, we need an approach based on sorting jobs and efficiently finding compatible intervals.

Several edge cases are important:

- Multiple jobs may have the same start or end times.
- Some jobs may completely contain other jobs.
- The most profitable solution may skip many smaller jobs.
- Jobs may chain exactly at boundaries, such as `[1,3]` followed by `[3,5]`.
- A greedy strategy based only on earliest ending time or highest profit does not always work.

The problem is a classic weighted interval scheduling problem, where each interval has an associated profit and we must maximize the total weight of non-overlapping intervals.

## Approaches

### Brute Force Approach

The most direct solution is to try every possible subset of jobs.

We could recursively decide for each job whether to include it or skip it. If we include a job, we can only continue with jobs whose start time is greater than or equal to the chosen job’s end time.

This approach is correct because it explores every valid combination and therefore cannot miss the optimal answer.

However, the number of subsets is exponential. For `n` jobs, there are `2^n` possible selections. Even with pruning, this becomes impossible for `n = 50,000`.

A slightly improved brute-force approach sorts jobs and recursively searches compatible next jobs, but without memoization it still recomputes the same states repeatedly.

### Key Insight

The critical observation is that after sorting jobs by start time, the remaining decision space becomes structured.

For every job, we only need to decide between two choices:

1. Skip the current job
2. Take the current job and jump directly to the next compatible job

If we know the best answer starting from every index, we can reuse those results instead of recomputing them.

This naturally leads to dynamic programming.

The remaining challenge is efficiently finding the next non-overlapping job. Since the jobs are sorted by start time, we can use binary search to locate the first job whose start time is greater than or equal to the current job’s end time.

This reduces the transition cost from linear time to logarithmic time.

The final solution combines:

- Sorting
- Binary search
- Dynamic programming with memoization

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(n) | Tries every subset recursively |
| Optimal | O(n log n) | O(n) | Sorting + DP + binary search |

## Algorithm Walkthrough

### Optimal Algorithm

1. Combine the three input arrays into a single list of jobs.

Each job will contain:

```
(start_time, end_time, profit)
```

Keeping related information together simplifies sorting and processing.
2. Sort all jobs by start time.

Sorting is important because it allows us to binary search for the next compatible job efficiently.
3. Create a separate array containing only the sorted start times.

Example:

```
starts = [1, 2, 3, 3]
```

This array will be used with binary search.
4. Define a DP function `dfs(index)`.

This function returns the maximum profit obtainable starting from job `index`.
5. Handle the base case.

If `index == n`, there are no jobs remaining, so return `0`.
6. Compute the "skip" option.

If we skip the current job, the answer becomes:

```
dfs(index + 1)
```
7. Compute the "take" option.

If we take the current job:

- Add its profit
- Find the next compatible job using binary search
- Continue recursively from that job
8. Use binary search to find the next compatible job.

We search for the first start time greater than or equal to the current job’s end time.

This works because the jobs are sorted by start time.
9. Store results in memoization.

Without memoization, many states would be recomputed repeatedly. Memoization ensures each index is solved only once.
10. Return the maximum of the two choices.

```
max(skip, take)
```

### Why it works

At every job index, the algorithm considers all valid possibilities through two exhaustive choices:

- either include the current job,
- or exclude it.

Because the jobs are sorted and binary search correctly finds the next compatible interval, every recursive branch remains valid. Memoization guarantees that each subproblem is solved exactly once, and the recurrence always chooses the maximum achievable profit. Therefore, the final result is optimal.

## Python Solution

```python
from bisect import bisect_left
from functools import lru_cache
from typing import List

class Solution:
    def jobScheduling(
        self,
        startTime: List[int],
        endTime: List[int],
        profit: List[int]
    ) -> int:
        
        jobs = sorted(zip(startTime, endTime, profit))
        starts = [job[0] for job in jobs]
        n = len(jobs)

        @lru_cache(None)
        def dfs(index: int) -> int:
            if index == n:
                return 0

            skip_profit = dfs(index + 1)

            start, end, current_profit = jobs[index]

            next_index = bisect_left(starts, end)

            take_profit = current_profit + dfs(next_index)

            return max(skip_profit, take_profit)

        return dfs(0)
```

The implementation begins by combining the three arrays into tuples so each job can be processed as a single unit. The jobs are then sorted by start time.

The `starts` array is extracted separately because Python’s `bisect_left` operates on a sorted list. This allows us to efficiently locate the next compatible job.

The recursive `dfs(index)` function represents the maximum profit obtainable beginning at position `index`.

Inside the function, we first compute the profit obtained by skipping the current job. Then we compute the profit obtained by taking the current job.

When taking a job, binary search finds the earliest job whose start time is greater than or equal to the current job’s end time. This guarantees compatibility.

Memoization through `@lru_cache` prevents repeated computations. Since each index is evaluated once, the algorithm remains efficient even for large inputs.

## Go Solution

```go
package main

import (
	"sort"
)

func jobScheduling(startTime []int, endTime []int, profit []int) int {
	n := len(startTime)

	type Job struct {
		start  int
		end    int
		profit int
	}

	jobs := make([]Job, n)

	for i := 0; i < n; i++ {
		jobs[i] = Job{
			start:  startTime[i],
			end:    endTime[i],
			profit: profit[i],
		}
	}

	sort.Slice(jobs, func(i, j int) bool {
		return jobs[i].start < jobs[j].start
	})

	starts := make([]int, n)

	for i := 0; i < n; i++ {
		starts[i] = jobs[i].start
	}

	dp := make([]int, n+1)

	for i := n - 1; i >= 0; i-- {
		nextIndex := sort.Search(n, func(j int) bool {
			return starts[j] >= jobs[i].end
		})

		skipProfit := dp[i+1]
		takeProfit := jobs[i].profit + dp[nextIndex]

		if takeProfit > skipProfit {
			dp[i] = takeProfit
		} else {
			dp[i] = skipProfit
		}
	}

	return dp[0]
}
```

The Go solution uses bottom-up dynamic programming instead of recursion with memoization. This avoids recursion depth concerns and fits naturally with Go’s iterative style.

The `sort.Search` function performs binary search and returns the first index satisfying the condition.

The DP array is filled from right to left because each state depends on future states.

Go slices automatically handle dynamic arrays, and standard `int` is sufficient because the maximum possible profit fits comfortably within integer limits.

The sorting step costs `O(n log n)`.

Each job is processed once in the DP. During processing, we perform one binary search costing `O(log n)`.

Therefore:

```
n * log n
```

dominates the runtime.

The space usage comes from storing the sorted jobs, memoization cache or DP array, and the start times array.

## Edge Cases

### Jobs That End Exactly When Another Starts

One subtle detail is that intervals touching at boundaries are allowed. A naive overlap check using strict inequality may incorrectly reject valid schedules.

For example:

```
[1,3] and [3,5]
```

These jobs are compatible.

The implementation handles this correctly because binary search looks for:

```
start >= current_end
```

rather than:

```
start > current_end
```

### Multiple Jobs Sharing the Same Start Time

Several jobs may begin at the exact same time but have different durations and profits.

A naive greedy approach might incorrectly select the shortest or most profitable local option without considering future consequences.

The DP solution evaluates both taking and skipping every job, ensuring the globally optimal combination is chosen regardless of duplicate start times.

### Large Nested Intervals

Some jobs may completely contain others.

Example:

```
Job A: [1,10], profit = 100
Job B: [2,5], profit = 40
Job C: [5,8], profit = 50
```

Although Job A is long, taking smaller nested jobs could potentially yield more total profit.

The implementation handles this correctly because every state compares:

```
take current job
vs
skip current job
```

This guarantees all combinations are evaluated efficiently through dynamic programming.
