# LeetCode 1223 - Dice Roll Simulation

## Problem Understanding

This problem asks us to count how many valid dice roll sequences of length `n` can be formed under a special restriction.

Normally, if we roll a six-sided die `n` times, every position has 6 choices, so there would be `6^n` total sequences. However, each face has a limit on how many times it may appear consecutively.

The array `rollMax` contains these limits:

- `rollMax[0]` is the maximum number of consecutive `1`s allowed
- `rollMax[1]` is the maximum number of consecutive `2`s allowed
- ...
- `rollMax[5]` is the maximum number of consecutive `6`s allowed

For example, if `rollMax[0] = 1`, then the sequence cannot contain `1,1` anywhere because `1` may appear only once consecutively.

The task is to count all distinct valid sequences of exactly `n` rolls. Since the number of possible sequences grows very quickly, we return the answer modulo:

$10^9 + 7$

The constraints are very important:

- `1 <= n <= 5000`
- `rollMax[i] <= 15`

The large value of `n` immediately tells us that brute force generation is impossible. Even for moderate values like `n = 20`, the total number of raw sequences would already be:

$6^{20}$

which is astronomically large.

Another important observation is that `rollMax[i]` is very small, at most 15. That suggests we can track consecutive streak lengths efficiently in dynamic programming states.

Several edge cases are important:

- `n = 1`, where every die face is valid
- All `rollMax[i] = 1`, where no number may repeat consecutively
- Large `n = 5000`, where inefficient recursion or exponential branching will time out
- Faces with different repetition limits, where some values can repeat longer than others
- Sequences ending with long streaks near the allowed maximum

These cases make careful state tracking necessary.

## Approaches

### Brute Force

The brute force solution generates every possible sequence of length `n`.

At each position, we try all six die faces. While building the sequence, we track the current consecutive streak of the latest value. If adding a new value violates its `rollMax` restriction, we discard that branch.

This approach is correct because it explicitly enumerates every valid sequence and counts them.

However, the time complexity is exponential. In the worst case, we explore nearly all:

$6^n$

possible sequences.

With `n = 5000`, this is completely infeasible.

### Optimal Dynamic Programming

The key insight is that the future validity of a sequence depends only on:

1. The last die face rolled
2. How many times it has appeared consecutively

We do not need the entire sequence history.

This naturally leads to dynamic programming.

We define a DP state representing:

- current sequence length
- last rolled number
- current consecutive streak length

From one state, we transition to another by either:

- extending the current streak if allowed
- switching to a different die face and resetting streak length to 1

Because there are only:

- `n` positions
- 6 possible faces
- at most 15 streak lengths

the total number of states is manageable.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(6^n) | O(n) | Generates all valid sequences recursively |
| Optimal DP | O(n × 6 × 15 × 6) | O(6 × 15) | Tracks only last face and streak length |

## Algorithm Walkthrough

### Step 1: Define the DP State

We use:

```
dp[face][count]
```

where:

- `face` is the last rolled number, from 0 to 5
- `count` is how many consecutive times that face has appeared

The value stored is the number of valid sequences ending in that state.

For example:

```
dp[2][3]
```

means:

- last die face is `3`
- it has appeared 3 times consecutively

### Step 2: Initialize Length 1 States

For sequences of length 1:

- every face can appear once
- streak length is 1

So:

```
dp[face][1] = 1
```

for all six faces.

### Step 3: Build Longer Sequences

For every additional roll:

- create a new DP table called `next_dp`
- iterate through every existing state

For each state:

- current last face = `face`
- current streak = `count`
- number of sequences = `ways`

we try all six next faces.

### Step 4: Continue Same Face

If the next face equals the current face:

- we increase streak length by 1
- but only if it does not exceed `rollMax[face]`

So we transition:

```
(face, count) -> (face, count + 1)
```

when valid.

### Step 5: Switch Faces

If the next face differs:

- streak resets to 1

So:

```
(face, count) -> (new_face, 1)
```

This transition is always valid.

### Step 6: Apply Modulo

Because counts become very large, every update is computed modulo:

$10^9 + 7$

### Step 7: Compute Final Answer

After processing all `n` rolls:

- sum every state in the DP table
- return the total modulo value

### Why it works

The DP state captures exactly the information needed to determine future validity:

- the last rolled face
- its current consecutive streak length

No earlier history matters. Any future choice depends only on whether continuing the streak would violate `rollMax`.

Because every valid sequence corresponds to exactly one DP path, and every DP transition preserves validity, the algorithm counts all valid sequences exactly once.

## Python Solution

```python
from typing import List

class Solution:
    def dieSimulator(self, n: int, rollMax: List[int]) -> int:
        MOD = 10**9 + 7
        
        # dp[face][count]
        # count is 1-indexed
        dp = [[0] * 16 for _ in range(6)]
        
        # Initialize sequences of length 1
        for face in range(6):
            dp[face][1] = 1
        
        # Build sequences from length 2 to n
        for _ in range(1, n):
            next_dp = [[0] * 16 for _ in range(6)]
            
            for face in range(6):
                for count in range(1, rollMax[face] + 1):
                    ways = dp[face][count]
                    
                    if ways == 0:
                        continue
                    
                    for next_face in range(6):
                        if next_face == face:
                            # Extend streak if allowed
                            if count + 1 <= rollMax[face]:
                                next_dp[face][count + 1] += ways
                                next_dp[face][count + 1] %= MOD
                        else:
                            # Switch face, streak resets
                            next_dp[next_face][1] += ways
                            next_dp[next_face][1] %= MOD
            
            dp = next_dp
        
        # Sum all valid ending states
        answer = 0
        
        for face in range(6):
            for count in range(1, rollMax[face] + 1):
                answer += dp[face][count]
                answer %= MOD
        
        return answer
```

The implementation begins by creating a DP table where rows represent die faces and columns represent consecutive streak lengths. Since `rollMax[i] <= 15`, we allocate 16 columns for convenience and ignore index 0.

The initialization step sets every face with streak length 1 to value 1. This represents all valid sequences of length 1.

For every additional roll, we construct a fresh `next_dp` table. This avoids overwriting states that are still needed during iteration.

For each current state, we attempt all six possible next faces. If the next face matches the current one, we increase the streak length only if it stays within the allowed limit. Otherwise, we reset the streak to 1.

Modulo arithmetic is applied during every update to prevent integer overflow and keep numbers bounded.

Finally, we sum all remaining valid ending states.

## Go Solution

```go
func dieSimulator(n int, rollMax []int) int {
	const MOD int = 1e9 + 7

	// dp[face][count]
	dp := make([][]int, 6)
	for i := 0; i < 6; i++ {
		dp[i] = make([]int, 16)
		dp[i][1] = 1
	}

	for roll := 1; roll < n; roll++ {
		nextDP := make([][]int, 6)
		for i := 0; i < 6; i++ {
			nextDP[i] = make([]int, 16)
		}

		for face := 0; face < 6; face++ {
			for count := 1; count <= rollMax[face]; count++ {
				ways := dp[face][count]

				if ways == 0 {
					continue
				}

				for nextFace := 0; nextFace < 6; nextFace++ {
					if nextFace == face {
						if count+1 <= rollMax[face] {
							nextDP[face][count+1] =
								(nextDP[face][count+1] + ways) % MOD
						}
					} else {
						nextDP[nextFace][1] =
							(nextDP[nextFace][1] + ways) % MOD
					}
				}
			}
		}

		dp = nextDP
	}

	answer := 0

	for face := 0; face < 6; face++ {
		for count := 1; count <= rollMax[face]; count++ {
			answer = (answer + dp[face][count]) % MOD
		}
	}

	return answer
}
```

The Go implementation mirrors the Python logic closely. Slices are used for the DP tables, and each iteration allocates a fresh `nextDP` matrix.

Go integers are sufficient because all updates are reduced modulo `1e9 + 7` immediately after addition.

Unlike Python, Go does not support dynamically resizing nested arrays automatically, so each DP row is explicitly initialized.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n × 6 × 15 × 6) | For each roll, we iterate through faces, streak lengths, and next faces |
| Space | O(6 × 15) | Only two DP tables are stored |

The streak length is bounded by 15, which is a constant. Therefore the practical runtime is essentially linear in `n`.

The algorithm uses rolling DP tables instead of storing all `n` layers, reducing memory usage significantly.

## Edge Cases

### Edge Case 1: Single Roll

When `n = 1`, every die face is valid because no consecutive repetition exists yet.

A buggy implementation might incorrectly attempt transitions or fail to initialize the DP table properly. Our initialization step directly assigns one valid sequence to each face with streak length 1, producing the correct answer of 6.

### Edge Case 2: No Consecutive Repetitions Allowed

When every value in `rollMax` equals 1, the same number may never appear twice in a row.

This is a common source of off-by-one mistakes because extending a streak from length 1 to 2 must immediately be rejected. Our transition logic explicitly checks:

```
count + 1 <= rollMax[face]
```

so invalid extensions are never added.

### Edge Case 3: Maximum Input Size

With `n = 5000`, recursive backtracking would be impossibly slow and could also cause stack overflow.

The DP solution handles this efficiently because the total state space remains small:

- only 6 faces
- only 15 possible streak lengths

Using rolling DP arrays also ensures memory usage stays constant relative to `n`.

### Edge Case 4: Different Limits Per Face

Some faces may allow long streaks while others do not.

For example:

```
rollMax = [1,1,1,2,2,15]
```

A careless implementation might accidentally use a global maximum streak limit for all faces. Our DP transitions always reference:

```
rollMax[face]
```

for the specific current face, ensuring every die value follows its own constraint correctly.
