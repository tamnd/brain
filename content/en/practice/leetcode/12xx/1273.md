## Solution

## Problem Understanding

The problem gives us a rooted tree with `nodes` total vertices. Every node has two important pieces of information:

- `parent[i]` tells us the parent of node `i`
- `value[i]` tells us the integer value stored at node `i`

The tree is always rooted at node `0`, and `parent[0] = -1` indicates that the root has no parent.

We are asked to repeatedly remove every subtree whose total sum of node values is exactly zero. After all such removals are complete, we must return how many nodes remain in the tree.

A subtree consists of a node and all of its descendants. This means that when a subtree is deleted, every node inside it disappears together.

The key detail is that subtree sums must be computed including all descendants before any deletion decision is made. For example, if a node has value `5` and its children contribute `-5`, then the entire subtree rooted at that node has sum `0`, so the whole subtree is removed.

The constraints tell us several important things:

- `nodes <= 10^4`, which is too large for repeatedly recomputing subtree sums inefficiently
- The input always forms a valid tree, so there are no cycles
- Values may be negative, positive, or zero
- The root itself can potentially be deleted, resulting in `0` remaining nodes

Because subtree calculations naturally depend on child information, this problem strongly suggests a postorder tree traversal, where children are processed before their parent.

Several edge cases are especially important:

- A single-node tree whose value is `0`, the answer should be `0`
- A subtree whose children cancel out the parent value exactly
- Multiple independent zero-sum subtrees
- The entire tree summing to zero
- Deep trees where recursive logic must still correctly aggregate values upward

## Approaches

### Brute Force Approach

A straightforward solution would repeatedly search the tree for zero-sum subtrees. For every node, we could compute its subtree sum by traversing all descendants. If the sum is zero, we delete that subtree. Then we repeat the process until no more deletions occur.

This approach is correct because every iteration recomputes subtree sums using the current structure of the tree, ensuring that all deletions are eventually discovered.

However, this is extremely inefficient. Computing a subtree sum from scratch for every node already costs `O(n^2)` in the worst case. Repeating this after deletions makes the runtime even worse.

With up to `10^4` nodes, repeatedly traversing large portions of the tree is too slow.

### Optimal Approach

The important observation is that each subtree sum only needs to be computed once.

If we process the tree using postorder DFS, we can gather two pieces of information for every subtree:

- The total sum of values in the subtree
- The number of nodes that remain in the subtree

When visiting a node:

1. Recursively process all children first
2. Combine their subtree sums and remaining node counts
3. Add the current node's value
4. If the resulting subtree sum is zero, return `(0, 0)` because the entire subtree is deleted
5. Otherwise return the computed sum and node count

This works because a subtree's fate depends entirely on the fully computed sums of its descendants.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) or worse | O(n) | Repeatedly recomputes subtree sums |
| Optimal | O(n) | O(n) | Single DFS traversal computes everything once |

## Algorithm Walkthrough

1. Build an adjacency list representing the tree structure.

The input only gives parent pointers, but DFS traversal is easier when every node knows its children. We create a list of children for every node by iterating through the `parent` array.
2. Start a postorder DFS from the root node `0`.

Postorder traversal is essential because the subtree sum of a node depends on the sums of all its descendants. We must process children before the parent.
3. For each node, initialize:

- `subtree_sum = value[node]`
- `subtree_count = 1`

This represents the contribution of the current node itself before adding child information.
4. Recursively process every child.

Each child returns:

- The sum of its remaining subtree
- The number of remaining nodes inside it

Add these values into the current node's running totals.
5. After all children are processed, check whether the subtree sum equals zero.

If `subtree_sum == 0`, the entire subtree is deleted. Return:

- Sum = `0`
- Count = `0`

Returning zero nodes ensures the parent does not count deleted descendants.
6. Otherwise, return the accumulated subtree sum and subtree node count.
7. The final answer is the remaining node count returned for the root.

### Why it works

The algorithm works because every subtree is evaluated exactly once after all descendants have already been processed. This guarantees that the computed subtree sum accurately reflects the final remaining structure beneath that node. If the sum becomes zero, deleting the entire subtree is correct because the problem definition requires removing every zero-sum subtree.

## Python Solution

```python
from typing import List

class Solution:
    def deleteTreeNodes(self, nodes: int, parent: List[int], value: List[int]) -> int:
        children = [[] for _ in range(nodes)]

        for node in range(1, nodes):
            parent_node = parent[node]
            children[parent_node].append(node)

        def dfs(node: int):
            subtree_sum = value[node]
            subtree_count = 1

            for child in children[node]:
                child_sum, child_count = dfs(child)
                subtree_sum += child_sum
                subtree_count += child_count

            if subtree_sum == 0:
                return 0, 0

            return subtree_sum, subtree_count

        _, remaining_nodes = dfs(0)
        return remaining_nodes
```

The implementation begins by constructing the `children` adjacency list. Since the input only provides parent references, this conversion allows efficient downward traversal during DFS.

The `dfs` function returns two values:

- The total subtree sum
- The number of nodes remaining in that subtree

Each node starts with its own value and count of `1`. Then every child subtree contributes its computed values upward.

The critical logic happens after processing children. If the subtree sum becomes zero, the entire subtree is removed by returning `(0, 0)`. This effectively erases the subtree from all future calculations.

Finally, the DFS result from the root contains the total number of remaining nodes.

## Go Solution

```go
func deleteTreeNodes(nodes int, parent []int, value []int) int {
	children := make([][]int, nodes)

	for node := 1; node < nodes; node++ {
		parentNode := parent[node]
		children[parentNode] = append(children[parentNode], node)
	}

	var dfs func(int) (int, int)

	dfs = func(node int) (int, int) {
		subtreeSum := value[node]
		subtreeCount := 1

		for _, child := range children[node] {
			childSum, childCount := dfs(child)
			subtreeSum += childSum
			subtreeCount += childCount
		}

		if subtreeSum == 0 {
			return 0, 0
		}

		return subtreeSum, subtreeCount
	}

	_, remainingNodes := dfs(0)
	return remainingNodes
}
```

The Go implementation closely mirrors the Python version. Since Go does not support nested named functions as naturally as Python, we declare `dfs` as a variable before assigning the recursive function.

Slices are used for the adjacency list representation. Returning two integers from DFS is straightforward in Go because functions can return multiple values directly.

Integer overflow is not a concern because the constraints remain safely within Go's `int` range.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every node is visited exactly once |
| Space | O(n) | Adjacency list and recursion stack require linear space |

The algorithm performs a single DFS traversal across the tree. Each edge and node is processed exactly once, giving linear time complexity.

The adjacency list stores all tree edges, which requires `O(n)` space. In the worst case, the recursion depth can also reach `O(n)` for a highly skewed tree.

## Edge Cases

One important edge case occurs when the entire tree sums to zero. A naive implementation might assume the root always survives, but the problem explicitly allows deleting the entire tree. In this implementation, if the root's subtree sum becomes zero, DFS returns `(0, 0)`, producing the correct answer of `0`.

Another tricky case is when child subtrees are deleted before computing the parent sum. If a child subtree sums to zero, it must contribute nothing to its parent afterward. Returning `(0, 0)` from DFS correctly removes both the sum and node count contribution from deleted subtrees.

A highly skewed tree is another potential source of bugs. For example, the tree may form a linked list of length `10^4`. The algorithm still works correctly because each node is processed exactly once in postorder order. The logic does not depend on tree balance.

A final subtle edge case involves nodes whose own value is nonzero but whose descendants cause the total subtree sum to become zero. The algorithm handles this naturally because subtree sums are computed after processing all descendants, ensuring the final deletion decision uses complete information.
