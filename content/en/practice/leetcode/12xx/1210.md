**Difficulty:** 🔴 Hard  
**Topics:** Array, Breadth-First Search, Matrix

## Solution

## Problem Understanding

This problem asks us to move a two-cell snake through an `n x n` grid while avoiding blocked cells. The snake always occupies exactly two adjacent cells, and it can either be positioned horizontally or vertically.

The snake starts in the top-left corner occupying:

- `(0, 0)`
- `(0, 1)`

The target position is the bottom-right corner in horizontal orientation:

- `(n - 1, n - 2)`
- `(n - 1, n - 1)`

Each move costs exactly one step. We must compute the minimum number of moves required to reach the target configuration. If the target cannot be reached, we return `-1`.

The grid contains:

- `0` for empty cells
- `1` for blocked cells

The snake can perform four kinds of operations:

1. Move right
2. Move down
3. Rotate clockwise, horizontal to vertical
4. Rotate counterclockwise, vertical to horizontal

The important detail is that rotations require additional free space. For example, a clockwise rotation from horizontal to vertical requires both cells directly below the snake to be empty.

The constraints are important:

- `2 <= n <= 100`
- The grid can contain up to `10,000` cells

A naive brute force search over all movement sequences would explode combinatorially. However, the total number of meaningful snake states is actually limited. Each state is determined by:

- the snake's anchor position
- its orientation

This observation makes graph traversal algorithms practical.

Several edge cases are important:

- The snake may become trapped immediately.
- Rotations near walls can easily cause index errors.
- Multiple paths may revisit the same configuration.
- A configuration may be reachable in many ways, but we only care about the minimum moves.
- The target requires a horizontal orientation specifically, not merely reaching the bottom-right cell.

Because every move has equal cost, this naturally becomes a shortest path problem on an implicit graph.

## Approaches

### Brute Force Approach

A brute force solution would recursively try every possible move from every configuration:

- move right
- move down
- rotate if possible

The algorithm would continue exploring until it either reaches the target or exhausts all possibilities.

This approach is correct because it explores every legal sequence of moves. Eventually, if a valid path exists, it will discover it.

However, the major problem is repeated states. The snake can revisit the same configuration through different sequences of moves. Without pruning, the search tree grows exponentially.

Even with memoization, using depth-first search is still inconvenient because we need the minimum number of moves. DFS would need to explore many longer paths before discovering the optimal one.

The state space is manageable, but the number of paths is enormous.

### Key Insight

The key insight is that this problem is actually a shortest path problem on a graph of snake states.

Each unique state can be represented by:

- row
- column
- orientation

We define the state using the snake's top-left or upper-most cell:

- Horizontal state:

- occupies `(r, c)` and `(r, c+1)`
- Vertical state:

- occupies `(r, c)` and `(r+1, c)`

From each state, there are only a few valid transitions.

Since every move costs exactly one step, Breadth-First Search is the ideal algorithm:

- BFS explores states level by level
- The first time we reach the target is guaranteed to be the minimum number of moves

The total number of states is small:

- At most `2 * n * n`
- Roughly `20,000` when `n = 100`

This is completely manageable.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Explores all possible movement sequences |
| Optimal BFS | O(n²) | O(n²) | Treats snake configurations as graph states |

## Algorithm Walkthrough

1. Represent each snake configuration as `(row, col, orientation)`.

The orientation is:

- `0` for horizontal
- `1` for vertical

For a horizontal snake:

- cells occupied are `(r, c)` and `(r, c+1)`

For a vertical snake:

- cells occupied are `(r, c)` and `(r+1, c)`
2. Initialize the BFS queue.

The starting state is:

```
(0, 0, horizontal)
```

This corresponds to the snake occupying:

```
(0,0) and (0,1)
```
3. Maintain a visited set.

Without tracking visited states, BFS would repeatedly revisit the same configurations, causing unnecessary work and possibly infinite loops.

Each state is uniquely identified by:

```
(row, col, orientation)
```
4. Process states level by level.

Each BFS layer corresponds to one additional move.

We repeatedly:

- pop states from the queue
- generate all legal next states
- add unvisited states to the queue
5. Generate moves for horizontal states.

Suppose the snake occupies:

```
(r,c) and (r,c+1)
```

We can move right if:

```
grid[r][c+2] == 0
```

We can move down if both below cells are empty:

```
grid[r+1][c] == 0
grid[r+1][c+1] == 0
```

We can rotate clockwise under the exact same condition because the rotation area must be empty.
6. Generate moves for vertical states.

Suppose the snake occupies:

```
(r,c) and (r+1,c)
```

We can move down if:

```
grid[r+2][c] == 0
```

We can move right if both right-side cells are empty:

```
grid[r][c+1] == 0
grid[r+1][c+1] == 0
```

We can rotate counterclockwise under the same condition.
7. Stop once the target state is reached.

The target state is:

```
(n-1, n-2, horizontal)
```
8. If BFS finishes without reaching the target, return `-1`.

### Why it works

Breadth-First Search guarantees optimality in unweighted graphs. Every snake configuration is a node, and every legal move is an edge with equal cost `1`.

BFS explores all states reachable in `k` moves before exploring states reachable in `k+1` moves. Therefore, the first time the target configuration is reached, it must be through the minimum possible number of moves.

Because we mark states as visited, each configuration is processed at most once, ensuring efficiency.

## Python Solution

```python
from collections import deque
from typing import List

class Solution:
    def minimumMoves(self, grid: List[List[int]]) -> int:
        n = len(grid)

        # 0 = horizontal
        # 1 = vertical
        start = (0, 0, 0)
        target = (n - 1, n - 2, 0)

        queue = deque([start])
        visited = {start}

        moves = 0

        while queue:
            for _ in range(len(queue)):
                row, col, orientation = queue.popleft()

                if (row, col, orientation) == target:
                    return moves

                if orientation == 0:
                    # Move right
                    if (
                        col + 2 < n
                        and grid[row][col + 2] == 0
                    ):
                        next_state = (row, col + 1, 0)

                        if next_state not in visited:
                            visited.add(next_state)
                            queue.append(next_state)

                    # Move down
                    if (
                        row + 1 < n
                        and grid[row + 1][col] == 0
                        and grid[row + 1][col + 1] == 0
                    ):
                        next_state = (row + 1, col, 0)

                        if next_state not in visited:
                            visited.add(next_state)
                            queue.append(next_state)

                        # Rotate clockwise
                        rotate_state = (row, col, 1)

                        if rotate_state not in visited:
                            visited.add(rotate_state)
                            queue.append(rotate_state)

                else:
                    # Move down
                    if (
                        row + 2 < n
                        and grid[row + 2][col] == 0
                    ):
                        next_state = (row + 1, col, 1)

                        if next_state not in visited:
                            visited.add(next_state)
                            queue.append(next_state)

                    # Move right
                    if (
                        col + 1 < n
                        and grid[row][col + 1] == 0
                        and grid[row + 1][col + 1] == 0
                    ):
                        next_state = (row, col + 1, 1)

                        if next_state not in visited:
                            visited.add(next_state)
                            queue.append(next_state)

                        # Rotate counterclockwise
                        rotate_state = (row, col, 0)

                        if rotate_state not in visited:
                            visited.add(rotate_state)
                            queue.append(rotate_state)

            moves += 1

        return -1
```

The implementation closely follows the BFS algorithm described earlier.

The queue stores all snake states that can be reached in the current number of moves. After processing an entire BFS layer, we increment the move counter.

The `visited` set prevents revisiting states. Since each state is processed once, the algorithm remains efficient.

The orientation determines which transitions are valid. Horizontal and vertical states have different movement rules, so the code separates them into two branches.

Rotations are handled together with movement validation because they require the same empty-space checks.

The target is checked immediately after removing a state from the queue. Since BFS explores in increasing order of distance, the first match is optimal.

## Go Solution

```go
package main

import "container/list"

type State struct {
	row         int
	col         int
	orientation int
}

func minimumMoves(grid [][]int) int {
	n := len(grid)

	start := State{0, 0, 0}
	target := State{n - 1, n - 2, 0}

	queue := list.New()
	queue.PushBack(start)

	visited := make(map[State]bool)
	visited[start] = true

	moves := 0

	for queue.Len() > 0 {
		size := queue.Len()

		for i := 0; i < size; i++ {
			front := queue.Front()
			queue.Remove(front)

			current := front.Value.(State)

			if current == target {
				return moves
			}

			row := current.row
			col := current.col
			orientation := current.orientation

			if orientation == 0 {
				// Move right
				if col+2 < n &&
					grid[row][col+2] == 0 {

					nextState := State{row, col + 1, 0}

					if !visited[nextState] {
						visited[nextState] = true
						queue.PushBack(nextState)
					}
				}

				// Move down + rotate clockwise
				if row+1 < n &&
					grid[row+1][col] == 0 &&
					grid[row+1][col+1] == 0 {

					nextState := State{row + 1, col, 0}

					if !visited[nextState] {
						visited[nextState] = true
						queue.PushBack(nextState)
					}

					rotateState := State{row, col, 1}

					if !visited[rotateState] {
						visited[rotateState] = true
						queue.PushBack(rotateState)
					}
				}

			} else {
				// Move down
				if row+2 < n &&
					grid[row+2][col] == 0 {

					nextState := State{row + 1, col, 1}

					if !visited[nextState] {
						visited[nextState] = true
						queue.PushBack(nextState)
					}
				}

				// Move right + rotate counterclockwise
				if col+1 < n &&
					grid[row][col+1] == 0 &&
					grid[row+1][col+1] == 0 {

					nextState := State{row, col + 1, 1}

					if !visited[nextState] {
						visited[nextState] = true
						queue.PushBack(nextState)
					}

					rotateState := State{row, col, 0}

					if !visited[rotateState] {
						visited[rotateState] = true
						queue.PushBack(rotateState)
					}
				}
			}
		}

		moves++
	}

	return -1
}
```

The Go implementation follows the same logic as the Python version, but there are some language-specific differences.

Go does not have a built-in deque, so `container/list` is used as the BFS queue.

A custom `State` struct is used instead of tuples. Structs are hashable in Go when all fields are comparable, so they can be used directly as map keys.

The visited structure is implemented with:

```go
map[State]bool
```

Go also requires explicit type assertions when retrieving values from the linked list queue.

No integer overflow concerns exist here because the grid size is at most `100 x 100`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n²) | Each state is processed at most once |
| Space | O(n²) | Queue and visited set store at most all states |

There are only two orientations for every grid position, so the total number of states is bounded by approximately `2 * n * n`.

Each state generates at most four transitions:

- move right
- move down
- rotate
- alternate orientation movement

Since every state is visited once, the total runtime is linear in the number of states.

The queue and visited set together also store at most all possible states, giving `O(n²)` space complexity.

## Edge Cases

### Snake Trapped at the Start

A grid may immediately block all legal moves from the starting position. For example:

```
[
 [0,0],
 [1,1]
]
```

The snake cannot move right, move down, or rotate. A naive implementation might enter unnecessary recursion or fail to terminate correctly.

The BFS implementation handles this naturally. The queue becomes empty after processing the initial state, and the algorithm returns `-1`.

### Rotations Near Grid Boundaries

Rotations require checking neighboring cells. Near the bottom or right edge of the board, careless implementations may access invalid indices.

For example, checking:

```
grid[row + 1][col + 1]
```

without validating bounds first can raise errors.

The implementation carefully verifies:

```
row + 1 < n
col + 1 < n
row + 2 < n
col + 2 < n
```

before accessing grid positions.

### Revisiting the Same State

Different movement sequences can lead to the same configuration.

Without a visited set, BFS would repeatedly process identical states, causing massive redundancy and potentially exponential runtime.

The implementation stores every processed configuration in:

```
visited = {(row, col, orientation)}
```

This guarantees each state is explored only once.

### Target Requires Horizontal Orientation

Reaching the bottom-right cell alone is not enough. The snake must specifically occupy:

```
(n-1, n-2) and (n-1, n-1)
```

in horizontal orientation.

Some incorrect implementations stop as soon as any part of the snake touches the goal area.

The solution avoids this bug by checking the exact target state:

```
(n - 1, n - 2, 0)
```

where `0` explicitly represents horizontal orientation.
