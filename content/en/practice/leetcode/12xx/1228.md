## Solution

## Problem Understanding

This problem gives us an array that originally formed an arithmetic progression, but exactly one value, excluding the first and last elements, has been removed.

An arithmetic progression is a sequence where the difference between consecutive elements is constant. For example:

```
2, 4, 6, 8, 10
```

has a common difference of `2`.

The input array `arr` is therefore almost a valid arithmetic progression, except one middle value is missing. Our task is to identify and return that missing value.

For example:

```
arr = [5,7,11,13]
```

The original progression must have been:

```
[5,7,9,11,13]
```

so the missing number is `9`.

The constraints are small:

- `3 <= arr.length <= 1000`
- `0 <= arr[i] <= 10^5`

Since the array length is at most 1000, even an `O(n^2)` solution would technically pass. However, the structure of arithmetic progressions allows us to derive a cleaner and more efficient linear solution.

Several important guarantees simplify the problem:

- Exactly one value is missing.
- The missing value is never the first or last element.
- The remaining array is still ordered according to the original progression.
- The input is always valid.

These guarantees eliminate ambiguous cases. For example, we never need to guess between multiple possible progressions.

Some edge cases that can trip up naive implementations include:

- Negative common differences, such as `[15,13,12]`
- Very small arrays, especially length `3`
- Cases where the missing value appears near the beginning or near the end
- Progressions with difference `1`, where detecting the anomaly requires careful comparison

A correct solution must handle both increasing and decreasing arithmetic progressions.

## Approaches

### Brute Force Approach

A brute force solution would attempt to reconstruct the arithmetic progression manually.

One way to do this is:

1. Try every possible insertion position.
2. Insert candidate values.
3. Check whether the resulting sequence forms a valid arithmetic progression.

To verify a progression, we compare consecutive differences across the entire array.

This works because the constraints are small, but it is inefficient and unnecessarily complicated. For every possible insertion position, we may scan the entire array again, resulting in quadratic time complexity.

### Key Insight for the Optimal Solution

In a valid arithmetic progression:

```
arr[i + 1] - arr[i]
```

is always the same.

Since exactly one number is missing, almost every adjacent difference in the given array will still equal the true common difference. Only one gap becomes abnormally large because it skips over the missing value.

For example:

```
[5,7,11,13]
```

Differences:

```
2, 4, 2
```

The correct common difference is clearly `2`, and the gap `4` reveals where the missing number belongs.

We can determine the true common difference using:

```
(arr[-1] - arr[0]) // len(arr)
```

Why does this work?

If the original progression had `n + 1` elements and one was removed, the current array has length `n`.

The total span from first to last element still covers exactly `n` equal intervals.

Once we know the expected difference, we scan the array until we find the first pair whose difference is incorrect. The missing number is then:

```
arr[i] + common_difference
```

This produces a clean linear-time solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Try reconstructing the progression at every position |
| Optimal | O(n) | O(1) | Compute expected difference and locate the broken gap |

## Algorithm Walkthrough

### Optimal Algorithm

1. Compute the expected common difference using:

```
diff = (arr[-1] - arr[0]) // len(arr)
```

The original arithmetic progression had one more element than the current array, so the total distance between the first and last elements is divided across `len(arr)` intervals.
2. Traverse the array from left to right.
3. For each adjacent pair, calculate:

```
arr[i + 1] - arr[i]
```
4. Compare the current difference with the expected common difference.
5. If the difference does not match, the missing number must lie between these two elements.
6. Return:

```
arr[i] + diff
```
7. Since the problem guarantees a valid input, the algorithm will always find a missing value.

### Why it works

An arithmetic progression has a constant adjacent difference. Removing one internal value affects exactly one adjacent gap, making it twice the expected distance in increasing sequences, or otherwise inconsistent in decreasing sequences.

All other gaps remain correct. Therefore, the first mismatch directly identifies the location of the missing number. Adding the expected common difference to the left element reconstructs the missing value.

## Python Solution

```python
from typing import List

class Solution:
    def missingNumber(self, arr: List[int]) -> int:
        diff = (arr[-1] - arr[0]) // len(arr)

        for i in range(len(arr) - 1):
            current_diff = arr[i + 1] - arr[i]

            if current_diff != diff:
                return arr[i] + diff

        return -1
```

The implementation begins by computing the expected common difference. Because one element is missing, the number of intervals between the first and last elements equals the current array length.

Next, the algorithm scans adjacent pairs in the array. For each pair, it computes the actual difference and compares it with the expected difference.

As soon as a mismatch is found, we know the missing value lies between those two numbers. Returning `arr[i] + diff` reconstructs the missing term.

The final `return -1` is technically unreachable because the problem guarantees a valid input, but it satisfies the function requirement that every execution path returns a value.

## Go Solution

```go
func missingNumber(arr []int) int {
    diff := (arr[len(arr)-1] - arr[0]) / len(arr)

    for i := 0; i < len(arr)-1; i++ {
        currentDiff := arr[i+1] - arr[i]

        if currentDiff != diff {
            return arr[i] + diff
        }
    }

    return -1
}
```

The Go implementation follows the same logic as the Python version.

Go integer division naturally truncates toward zero, which works correctly here because the arithmetic progression guarantees an exact integer difference.

No special handling for nil or empty slices is needed because the constraints guarantee at least three elements.

## Edge Cases

One important edge case is a decreasing arithmetic progression. Many implementations assume the sequence is increasing and incorrectly use absolute values for differences. For example:

```
[15,13,12]
```

has a negative common difference. The implementation handles this naturally because subtraction preserves the sign of the difference.

Another important case is the smallest valid input size, where the array contains only three elements. In such cases there is only one adjacent pair to inspect before identifying the anomaly. The algorithm still works because the computed common difference remains valid even for very small arrays.

A third edge case occurs when the missing value is very close to the beginning or end of the array. For example:

```
[2,6,8,10]
```

The missing value is `4`, immediately after the first element. Some naive solutions incorrectly assume the anomaly appears later in the array. This implementation scans from left to right and immediately detects the first incorrect gap.

Another subtle edge case is when the common difference equals `1` or `-1`. Since the numbers are very close together, off by one errors become common. The algorithm avoids this issue by relying directly on arithmetic progression properties rather than heuristic comparisons.
