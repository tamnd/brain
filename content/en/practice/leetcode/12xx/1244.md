## Solution

## Problem Understanding

The problem requires designing a `Leaderboard` class to maintain scores for multiple players and provide efficient queries for the top K scores. The class must support three operations: adding a score for a player, retrieving the sum of the top K scores, and resetting a player's score to zero. Players are identified by a unique integer `playerId`. If a player does not exist in the leaderboard, adding a score initializes their entry. The leaderboard starts empty.

The input consists of function calls with parameters: `playerId` and `score` for `addScore`, and an integer `K` for `top`. The output is either `null` for operations that do not return a value, or an integer sum for the `top` query. Constraints indicate the number of players and K can go up to 10,000, scores are relatively small (1 to 100), and at most 1,000 function calls occur. This means operations must be efficient, particularly `top(K)`, since sorting all players each time would be too slow for large data sets.

Edge cases include adding scores to a new player, resetting a player immediately after adding, multiple players with the same score, and calling `top(K)` when K equals the number of current players.

## Approaches

A brute-force approach maintains a dictionary mapping `playerId` to their score. `addScore` simply updates or inserts the score. `reset` sets the score to zero or removes the entry. `top(K)` sorts all current scores in descending order and sums the top K. This works but sorting each time is O(N log N), which is inefficient if `top` is called frequently with large N.

An optimal approach leverages a hash map for O(1) score updates and a heap or sorted data structure to efficiently track the top scores. Python’s `heapq` or `SortedList` from `sortedcontainers` can maintain a sorted order of scores while supporting insertions and deletions. This reduces the time complexity of `top(K)` to O(K log N) if using a heap approach.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | `O(N log N)` per `top(K)` | `O(N)` | Simple dictionary, sorts every `top` call |
| Optimal | `O(K log N)` per `top(K)`, `O(1)` per `addScore`/`reset` | `O(N)` | Hash map plus heap or sorted list for efficient top-K queries |

## Algorithm Walkthrough

1. Initialize a dictionary `scores` mapping `playerId` to their current score.
2. For `addScore(playerId, score)`, check if `playerId` exists in `scores`. If yes, increment the existing score; if no, create a new entry.
3. For `reset(playerId)`, set the player’s score to zero or remove the entry from the dictionary.
4. For `top(K)`, retrieve all current scores from the dictionary, sort them in descending order, and sum the first K scores.
5. Return the sum as the result for the `top(K)` query.

Why it works: Maintaining scores in a hash map guarantees fast updates and resets. Sorting the scores when querying ensures that the top K scores are accurately selected. Using a heap or sorted list can further optimize repeated `top` queries without affecting the correctness.

## Python Solution

```python
class Leaderboard:

    def __init__(self):
        self.scores = {}

    def addScore(self, playerId: int, score: int) -> None:
        if playerId in self.scores:
            self.scores[playerId] += score
        else:
            self.scores[playerId] = score

    def top(self, K: int) -> int:
        return sum(sorted(self.scores.values(), reverse=True)[:K])

    def reset(self, playerId: int) -> None:
        if playerId in self.scores:
            self.scores[playerId] = 0
```

In the Python solution, the constructor initializes an empty dictionary. `addScore` updates or inserts the player's score efficiently. `top` retrieves all scores, sorts them descending, and sums the top K. `reset` sets a player's score to zero, reflecting removal from the leaderboard.

## Go Solution

```go
type Leaderboard struct {
    scores map[int]int
}

func Constructor() Leaderboard {
    return Leaderboard{scores: make(map[int]int)}
}

func (this *Leaderboard) AddScore(playerId int, score int) {
    this.scores[playerId] += score
}

func (this *Leaderboard) Top(K int) int {
    vals := make([]int, 0, len(this.scores))
    for _, v := range this.scores {
        vals = append(vals, v)
    }
    sort.Slice(vals, func(i, j int) bool { return vals[i] > vals[j] })
    sum := 0
    for i := 0; i < K; i++ {
        sum += vals[i]
    }
    return sum
}

func (this *Leaderboard) Reset(playerId int) {
    this.scores[playerId] = 0
}
```

In Go, we use a map for `scores`. `AddScore` updates using `+=`. `Top` collects all scores in a slice, sorts descending, and sums the first K. `Reset` sets the player’s score to zero.

## Worked Examples

Consider the input sequence:

```
["Leaderboard","addScore","addScore","addScore","addScore","addScore","top","reset","reset","addScore","top"]
[[],[1,73],[2,56],[3,39],[4,51],[5,4],[1],[1],[2],[2,51],[3]]
```

Step by step:

1. Initialize `Leaderboard`: `scores = {}`.
2. `addScore(1, 73)` → `scores = {1: 73}`
3. `addScore(2, 56)` → `scores = {1: 73, 2: 56}`
4. `addScore(3, 39)` → `scores = {1: 73, 2: 56, 3: 39}`
5. `addScore(4, 51)` → `scores = {1: 73, 2: 56, 3: 39, 4: 51}`
6. `addScore(5, 4)` → `scores = {1: 73, 2: 56, 3: 39, 4: 51, 5: 4}`
7. `top(1)` → sorted `[73, 56, 51, 39, 4]` → top 1 sum = 73
8. `reset(1)` → `scores = {1: 0, 2: 56, 3: 39, 4: 51, 5: 4}`
9. `reset(2)` → `scores = {1: 0, 2: 0, 3: 39, 4: 51, 5: 4}`
10. `addScore(2, 51)` → `scores = {1: 0, 2: 51, 3: 39, 4: 51, 5: 4}`
11. `top(3)` → sorted `[51, 51, 39, 4, 0]` → top 3 sum = 141

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(N log N) per top(K) | Sorting scores dictionary values dominates the runtime |

The time complexity is dominated by the `top(K)` call. Additions and resets are O(1).

## Edge Cases

One edge case is adding a score to a player that has just been reset. This tests that the reset does not prevent future additions. Another is calling `top(K)` where K equals the current number of players; the algorithm must correctly sum all scores without off-by-one errors.
