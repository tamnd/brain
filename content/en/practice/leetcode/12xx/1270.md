## Problem Understanding

The problem provides a table `Employees` with three columns: `employee_id`, `employee_name`, and `manager_id`. Each row represents an employee, and `manager_id` tells you who that employee reports to. The head of the company has `employee_id = 1`.

The task is to return a list of all employees who either directly report to the head (`manager_id = 1`) or indirectly report to the head through a chain of managers. The problem specifies that indirect reporting chains will not exceed three managers.

In simpler terms, for every employee, we want to check if, by repeatedly following their manager, we eventually reach employee 1. The input is a small table, so we can assume `O(n^2)` approaches may be acceptable, but an efficient method is preferred.

Important edge cases include employees who are their own managers (loops), employees who do not report at all to the head, and the possibility of multiple disconnected reporting structures. The problem guarantees a limited depth of three for indirect reports, which simplifies the solution.

## Approaches

The brute-force approach iterates through every employee and repeatedly follows the `manager_id` until reaching either the head of the company or a dead end. While this works, it may involve repeated work if many employees share the same intermediate managers.

The key insight is that since the company hierarchy is small and limited to three levels, we can write a SQL query (or programmatically build a solution) using joins or recursive lookups. By constructing a chain of managers up to three levels and filtering those who eventually report to the head, we get an optimal solution without unnecessary repeated traversal.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * depth) | O(1) | Check each employee and traverse the reporting chain individually. |
| Optimal | O(n) | O(n) | Use hash map to record direct reports and traverse up to depth 3 efficiently. |

## Algorithm Walkthrough

1. Read all rows of the `Employees` table into memory. We need both `employee_id` and `manager_id` for fast lookups.
2. Construct a mapping (hash map) from each employee to their manager. This allows constant-time lookups of any employee's manager.
3. Initialize an empty list for employees who report directly or indirectly to the head.
4. Iterate through each employee in the table.
5. For each employee, follow the chain of managers up to three levels or until reaching the head (`employee_id = 1`).
6. If the head is reached within three steps, append the employee's `employee_id` to the result list.
7. Return the final list.

**Why it works:**

By following each employee’s manager chain up to the guaranteed maximum depth of three, we capture all employees reporting to the head either directly or indirectly. The hash map ensures efficient lookups and prevents redundant computations.

## Python Solution

```python
from typing import List
import collections

class Solution:
    def getAllEmployeesReportingToHead(self, employees: List[List[int]]) -> List[int]:
        manager_map = {emp[0]: emp[2] for emp in employees}
        head_id = 1
        result = []

        for emp_id in manager_map:
            if emp_id == head_id:
                continue
            current = emp_id
            steps = 0
            while steps < 3:
                manager = manager_map.get(current)
                if manager == head_id:
                    result.append(emp_id)
                    break
                if manager is None or manager == current:
                    break
                current = manager
                steps += 1

        return result
```

**Implementation Walkthrough:**

We first build `manager_map` for O(1) lookups of any employee’s manager. We then iterate through each employee, skipping the head itself. Using a while loop capped at three steps, we follow each employee’s reporting chain. If we reach the head, we add the employee to the result. If the chain ends early or loops on itself, we break.

## Go Solution

```go
package main

func getAllEmployeesReportingToHead(employees [][]int) []int {
    managerMap := make(map[int]int)
    for _, emp := range employees {
        managerMap[emp[0]] = emp[2]
    }

    headID := 1
    result := []int{}

    for empID := range managerMap {
        if empID == headID {
            continue
        }
        current := empID
        steps := 0
        for steps < 3 {
            manager, ok := managerMap[current]
            if !ok || manager == current {
                break
            }
            if manager == headID {
                result = append(result, empID)
                break
            }
            current = manager
            steps++
        }
    }

    return result
}
```

**Go-specific notes:**

We use a map for quick lookups. The loop handles missing or self-referencing managers safely with `ok` checks. Slice `result` is used instead of Python lists.

## Edge Cases

One important edge case is an employee reporting to themselves. If the code does not handle this, it could enter an infinite loop. Our solution breaks when `manager == current`.

Another edge case is an employee who does not have a manager in the table (missing entry). Using `get` in Python or `ok` check in Go ensures such employees are safely skipped.

Finally, a case with exactly three indirect managers tests the maximum depth allowed by the constraints. The solution iterates only three times, ensuring correctness without extra computation.
