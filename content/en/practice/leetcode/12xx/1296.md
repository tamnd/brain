## Solution

## Problem Understanding

The problem gives us an integer array `nums` and a positive integer `k`. We must determine whether the entire array can be partitioned into groups where:

- Each group contains exactly `k` numbers
- The numbers inside each group are consecutive integers
- Every element in the array is used exactly once

The order of elements in the original array does not matter because we are free to rearrange them into valid groups.

For example, if `nums = [1,2,3,3,4,4,5,6]` and `k = 4`, we can rearrange the numbers into:

- `[1,2,3,4]`
- `[3,4,5,6]`

Both groups contain 4 consecutive integers, and every number is used exactly once, so the answer is `true`.

The constraints are important:

- `nums.length` can be as large as `10^5`
- Values can be as large as `10^9`

These limits immediately tell us that an inefficient brute-force simulation will likely time out. We need a solution close to `O(n log n)` time.

One very important observation is that the total number of elements must be divisible by `k`. If the array length is not divisible by `k`, then it is impossible to partition the array into groups of equal size `k`.

Another important challenge is handling duplicates correctly. For example:

```
nums = [1,2,2,3,3,4], k = 3
```

This is valid because we can form:

```
[1,2,3]
[2,3,4]
```

A naive greedy approach can easily consume values in the wrong order and leave unusable leftovers.

We also need to handle sparse values correctly. If a required consecutive number is missing, the grouping immediately fails.

## Approaches

### Brute Force Approach

A brute-force strategy would repeatedly try to construct groups manually.

We could sort the array, then repeatedly:

1. Take the smallest unused number
2. Search for the next `k - 1` consecutive numbers
3. Mark them as used
4. Repeat until all elements are consumed

This approach is correct because every valid grouping must eventually use the smallest remaining number as the beginning of some consecutive sequence.

However, the problem is efficiency. If we repeatedly search through the array and remove elements, the implementation becomes very expensive. In the worst case, each grouping operation may scan large portions of the array.

Depending on the implementation, this can degrade to `O(n^2)` time complexity, which is too slow for `n = 10^5`.

### Optimal Greedy + Hash Map Approach

The key insight is that the smallest remaining number must always start a new consecutive group.

Suppose the smallest unused number is `x`.

There is no way to place `x` in the middle or at the end of a consecutive sequence because smaller numbers required before it do not exist anymore. Therefore, `x` must be the first element of a group:

```
[x, x+1, x+2, ..., x+k-1]
```

This observation leads naturally to a greedy strategy.

We count the frequency of every number using a hash map. Then we process numbers in sorted order. Whenever we encounter a number with remaining frequency, we try to build a consecutive sequence of length `k` starting from it.

If at any point a required number is unavailable, the answer is `false`.

This works efficiently because:

- Sorting costs `O(n log n)`
- Each number is consumed exactly once
- Hash map operations are `O(1)` on average

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeatedly searches and removes consecutive numbers |
| Optimal | O(n log n) | O(n) | Uses sorting and frequency counting with greedy construction |

## Algorithm Walkthrough

1. First, check whether the array length is divisible by `k`.

If `len(nums) % k != 0`, then forming equal-sized groups is impossible, so we immediately return `false`.
2. Count the frequency of every number.

We use a hash map where:

```
frequency[number] = how many times the number appears
```

This allows efficient tracking of remaining unused values.
3. Sort the array.

Processing numbers in ascending order is critical because the smallest unused number must begin a new sequence.
4. Iterate through the sorted numbers.

For each number `num`:

- If its frequency is already zero, it has already been used in previous groups, so we skip it.
- Otherwise, this number must start a new consecutive sequence.
5. Attempt to build a sequence of length `k`.

For every value:

```
current = num, num+1, ..., num+k-1
```

we check whether its frequency is positive.

- If any required number is missing, return `false`
- Otherwise, decrement the frequency of each number by 1
6. If all sequences are successfully formed, return `true`.

### Why it works

The correctness comes from the greedy invariant:

> The smallest remaining unused number must start a consecutive group.

There is no smaller value available to place before it, so any valid solution is forced to begin a group from that number.

Because we always process numbers in ascending order and consume exactly the required consecutive values, we never create conflicts with future groups. If a needed value is missing at any step, no valid arrangement exists.

## Python Solution

```python
from collections import Counter
from typing import List

class Solution:
    def isPossibleDivide(self, nums: List[int], k: int) -> bool:
        n = len(nums)

        if n % k != 0:
            return False

        frequency = Counter(nums)

        nums.sort()

        for num in nums:
            if frequency[num] == 0:
                continue

            for current in range(num, num + k):
                if frequency[current] == 0:
                    return False

                frequency[current] -= 1

        return True
```

The implementation begins with an early divisibility check. Since every group must contain exactly `k` elements, the total array size must be divisible by `k`.

Next, we build a frequency map using `Counter`. This gives constant-time access to how many unused copies of each number remain.

The array is then sorted so that we always process the smallest available number first. This ordering is essential for the greedy argument.

Inside the main loop, we skip numbers whose frequency has already dropped to zero because they were consumed in earlier sequences.

Whenever we encounter a number with remaining frequency, we attempt to form a consecutive sequence of length `k` starting from that number. We decrement frequencies as numbers are used.

If any required number is unavailable, we immediately return `False` because the current smallest unused number cannot be placed into any valid group.

If the loop finishes successfully, all numbers were partitioned correctly, so we return `True`.

## Go Solution

```go
package main

import (
	"sort"
)

func isPossibleDivide(nums []int, k int) bool {
	n := len(nums)

	if n%k != 0 {
		return false
	}

	frequency := make(map[int]int)

	for _, num := range nums {
		frequency[num]++
	}

	sort.Ints(nums)

	for _, num := range nums {
		if frequency[num] == 0 {
			continue
		}

		for current := num; current < num+k; current++ {
			if frequency[current] == 0 {
				return false
			}

			frequency[current]--
		}
	}

	return true
}
```

The Go implementation follows the same algorithmic structure as the Python version.

A `map[int]int` is used instead of Python's `Counter`. Sorting is done using `sort.Ints(nums)`.

Go does not distinguish between missing keys and zero values when reading from a map, which works perfectly here because a missing frequency naturally behaves as zero.

Integer overflow is not a concern because the constraints fit comfortably within Go's `int` type.

## Edge Cases

One important edge case occurs when the array length is not divisible by `k`. This is the simplest impossible scenario, yet many implementations forget to check it early. Without this validation, the algorithm may do unnecessary work or even incorrectly succeed. The implementation handles this immediately with:

```
if len(nums) % k != 0:
    return False
```

Another tricky case involves duplicate values with insufficient matching consecutive numbers. For example:

```
[1,1,2,2,2,3], k=3
```

At first glance, the array appears close to valid, but there are not enough `3`s to pair with all the `2`s. A naive implementation may accidentally reuse values incorrectly. The frequency map prevents this because each usage decrements the count, ensuring every number is consumed exactly once.

A third important edge case is sparse gaps between values. Consider:

```
[1,2,4,5,6,7], k=3
```

The number `3` is missing, so the smallest unused number `1` cannot form a valid sequence. The implementation detects this immediately when checking frequencies during sequence construction.

Another subtle edge case is `k = 1`. In this scenario, every individual element forms a valid group by itself, so every input should return `true`. The algorithm naturally handles this without requiring special-case logic because each sequence contains exactly one number.
