## Solution

## Problem Understanding

The problem gives us a sorted integer array in non-decreasing order. We are guaranteed that exactly one integer appears more than 25% of the time in the array, and we must return that integer.

In simpler terms, if the array has length `n`, there exists one value whose frequency is strictly greater than `n / 4`. Because the array is already sorted, identical values appear consecutively. This property is extremely important because it allows us to reason about frequencies without needing additional data structures like hash maps.

The input array represents a sequence of integers sorted from smallest to largest. The expected output is the single integer whose count exceeds 25% of the array length.

The constraints are relatively small:

- The array length is at most `10^4`
- Values range from `0` to `10^5`

Even though the constraints are not large enough to require highly advanced optimization, the sorted property strongly suggests that there is a more elegant solution than counting frequencies with brute force.

There are several important edge cases to consider. The array may contain only one element, in which case that element trivially appears more than 25% of the time. The frequent element may appear at the beginning or end of the array. The frequent element may also appear exactly one more time than the threshold, which means the implementation must use a strict greater-than comparison rather than greater-than-or-equal.

Because the problem guarantees that exactly one such element exists, we never need to handle the case where no answer exists.

## Approaches

### Brute Force Approach

A straightforward solution is to count how many times each number appears in the array. We can use a hash map where the key is the integer and the value is its frequency.

We iterate through the array once, updating the count for each number. After incrementing the count, we check whether the frequency exceeds `n // 4`. As soon as one number crosses the threshold, we return it.

This approach is correct because it explicitly tracks the exact frequency of every value. Since the problem guarantees one valid answer, eventually one count will exceed the threshold.

Although this solution is fast enough for the given constraints, it does not fully take advantage of the fact that the array is sorted. It also requires additional memory proportional to the number of distinct elements.

### Optimal Approach

The key observation is that the array is sorted. If an element appears more than 25% of the time, then there must exist some index `i` such that:

```
arr[i] == arr[i + threshold]
```

where:

```
threshold = n // 4
```

This works because identical elements form a continuous block in a sorted array. If the block length exceeds 25% of the array, then looking `threshold` positions ahead from the start of the block must still land on the same value.

We can therefore scan the array and check whether the current element matches the element `threshold` positions later. The first such match is the answer.

This approach is both elegant and efficient because it leverages the sorted structure directly.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Uses a hash map to count frequencies |
| Optimal | O(n) | O(1) | Exploits the sorted property of the array |

## Algorithm Walkthrough

1. Compute the array length `n`.
2. Compute the threshold value as `n // 4`. Any valid answer must appear more than this many times.
3. Iterate through the array from index `0` to `n - threshold - 1`.
4. For each index `i`, compare `arr[i]` with `arr[i + threshold]`.
5. If the two values are equal, then the current value appears in at least `threshold + 1` consecutive positions. Because the array is sorted, this guarantees the value appears more than 25% of the time.
6. Return that value immediately.
7. Since the problem guarantees a valid answer exists, the algorithm will always return a value before the loop finishes.

### Why it works

The correctness relies on the sorted nature of the array. Suppose a value appears more than `n / 4` times. Because all occurrences are contiguous, there must be a segment longer than `threshold`. Therefore, if we look `threshold` positions ahead from some occurrence inside that segment, we will still see the same value.

Conversely, if `arr[i] == arr[i + threshold]`, then the value spans at least `threshold + 1` positions, meaning its frequency exceeds 25% of the array length.

## Python Solution

```python
from typing import List

class Solution:
    def findSpecialInteger(self, arr: List[int]) -> int:
        n = len(arr)
        threshold = n // 4

        for i in range(n - threshold):
            if arr[i] == arr[i + threshold]:
                return arr[i]

        return -1
```

The implementation begins by computing the array length and the frequency threshold. Since any valid element must appear more than `n // 4` times, we only need to search for a repeated value separated by exactly that distance.

The loop iterates only until `n - threshold` because we access `i + threshold` inside the loop. At each step, the algorithm checks whether the current value matches the value `threshold` positions later.

If the two values are equal, then the sorted structure guarantees the element appears frequently enough to satisfy the problem condition. The algorithm immediately returns that value.

The final `return -1` statement is technically unnecessary because the problem guarantees a valid answer exists, but it is included to satisfy Python function completeness.

## Go Solution

```go
func findSpecialInteger(arr []int) int {
    n := len(arr)
    threshold := n / 4

    for i := 0; i < n-threshold; i++ {
        if arr[i] == arr[i+threshold] {
            return arr[i]
        }
    }

    return -1
}
```

The Go implementation follows exactly the same logic as the Python version. Since slices in Go already provide indexed access similar to arrays, no additional data structures are needed.

Go uses integer division automatically for `n / 4`, which matches the required threshold calculation. Like the Python version, the final `return -1` is included for completeness even though the problem guarantees a valid answer exists.

### Example 1


The array length is:

```
n = 9
```

The threshold is:

```
threshold = 9 // 4 = 2
```

We now compare elements `2` positions apart.

| i | arr[i] | arr[i + threshold] | Match? |
| --- | --- | --- | --- |
| 0 | 1 | 2 | No |
| 1 | 2 | 6 | No |
| 2 | 2 | 6 | No |
| 3 | 6 | 6 | Yes |

At index `3`, we find:

```
arr[3] == arr[5] == 6
```

This confirms that `6` appears more than 25% of the time, so we return `6`.

```

```

The threshold is:

```
threshold = 2 // 4 = 0
```

Now the loop compares:

| i | arr[i] | arr[i + threshold] | Match? |
| --- | --- | --- | --- |
| 0 | 1 | 1 | Yes |

Since the values match immediately, the algorithm returns `1`.

## Edge Cases

One important edge case is a single-element array such as `[5]`. In this situation, the threshold becomes `0`, and the algorithm compares the element with itself. Since the values match, the algorithm correctly returns the only element. Implementations that assume at least two elements could fail here.

Another important edge case is when the frequent element appears at the beginning or end of the array. For example, `[1,1,1,2,3,4]` or `[1,2,3,4,5,5,5]`. Some implementations accidentally skip boundary regions due to incorrect loop limits. This implementation avoids that issue by iterating until `n - threshold`.

A third important edge case occurs when the frequent element barely exceeds the required threshold. For example, in an array of length `8`, an element appearing `3` times qualifies because `3 > 8 / 4`. Off-by-one mistakes are common here, especially when developers confuse `>= 25%` with `> 25%`. The comparison strategy using `arr[i] == arr[i + threshold]` naturally handles the strict inequality correctly because it requires at least `threshold + 1` occurrences.
