## Solution

## Problem Understanding

The problem requires us to transform a given array `arr` such that each element is replaced with the largest element among those to its right. The last element of the array should always become `-1` because there are no elements to its right. The input is a list of integers, each within the range `[1, 10^5]`, and the array length can range from `1` to `10^4`. The output is an array of the same length where every element has been replaced according to the stated rule.

In simpler terms, for each index `i`, we need to find the maximum value in the subarray `arr[i+1:]` and set it as the new value for `arr[i]`. If there are no elements to the right, we set `-1`. The constraints imply that a naive solution iterating over the remaining elements for each position could be too slow for the maximum array size of `10^4`.

Edge cases include arrays of length `1`, arrays with identical elements, and arrays that are strictly increasing or decreasing. The input guarantees non-empty arrays with positive integers, so we do not need to handle empty arrays or negative values.

## Approaches

The brute-force approach involves iterating over each element in the array and, for each element, scanning all elements to its right to find the maximum. This approach is simple and correct, but it requires a nested loop, making it inefficient for large arrays.

The optimal solution leverages the insight that we do not need to recompute the maximum for every element. By processing the array from right to left, we can maintain a running `max_so_far` that represents the largest element seen to the right. This allows us to update each element in a single pass with constant extra space, achieving linear time complexity.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^2) | O(1) | For each element, scan the remainder of the array to find the max |
| Optimal | O(n) | O(1) | Traverse the array from right to left, keeping track of the max |

## Algorithm Walkthrough

1. Initialize a variable `max_so_far` to `-1`, which will track the maximum element to the right of the current position.
2. Traverse the array from the last element to the first element.
3. For the current element, temporarily store its original value in a variable `current`.
4. Replace the current element with `max_so_far`.
5. Update `max_so_far` to be the maximum of `current` and `max_so_far`.
6. Continue this process until all elements have been processed.

Why it works: By iterating from right to left, we ensure that at each step, `max_so_far` accurately represents the greatest element to the right of the current index. The last element is automatically set to `-1` as required.

## Python Solution

```python
from typing import List

class Solution:
    def replaceElements(self, arr: List[int]) -> List[int]:
        max_so_far = -1
        for i in range(len(arr) - 1, -1, -1):
            current = arr[i]
            arr[i] = max_so_far
            max_so_far = max(max_so_far, current)
        return arr
```

In this implementation, `max_so_far` starts as `-1` for the last element. The loop iterates backward, storing the current element before updating it. After replacing the element with the current maximum to its right, `max_so_far` is updated to include the current element, ensuring correctness for the next iteration.

## Go Solution

```go
func replaceElements(arr []int) []int {
    maxSoFar := -1
    for i := len(arr) - 1; i >= 0; i-- {
        current := arr[i]
        arr[i] = maxSoFar
        if current > maxSoFar {
            maxSoFar = current
        }
    }
    return arr
}
```

In Go, we follow the same logic. The main differences are syntax-related: Go requires explicit type handling, and the `max` function is manually implemented using a conditional statement instead of a built-in function like Python's `max`.

The reasoning: The algorithm only loops over the array once and does not allocate additional arrays or data structures, making it both time and space efficient.

## Edge Cases

One important edge case is an array of length 1. The function must return `[-1]` without attempting any further comparisons. The current implementation handles this naturally by starting `max_so_far` at `-1` and performing the backward iteration.

Another edge case is when all elements are identical. A naive forward approach might mistakenly leave some elements unchanged. By maintaining a `max_so_far` from right to left, the algorithm correctly replaces all elements except the last with the repeated maximum.

A third edge case is a strictly increasing array. Here, the last element always becomes `-1`, and each preceding element should be replaced with the maximum to its right. The backward traversal ensures that each replacement uses the correct maximum value dynamically updated during iteration. This avoids unnecessary scanning of all subsequent elements, which would be inefficient.
