# LeetCode 1250 - Check If It Is a Good Array

## Problem Understanding

This problem asks whether it is possible to create the value `1` using the numbers in the array under a very flexible operation model.

We are given an array `nums` of positive integers. We may choose any subset of the array, multiply each chosen number by any integer, including negative integers and zero, and then add all the resulting values together.

Mathematically, we are trying to determine whether there exist integers:

$$x_1, x_2, x_3, \dots, x_n$$

such that:

$$nums[0] \cdot x_1 + nums[1] \cdot x_2 + \dots + nums[n-1] \cdot x_n = 1$$

The coefficients can be positive, negative, or zero.

The output should be:

- `True` if such a combination exists
- `False` otherwise

The constraints are large:

- The array can contain up to `10^5` elements
- Each value can be as large as `10^9`

These constraints immediately tell us that exponential or subset enumeration approaches are impossible. We need something close to linear time.

The key detail is that we are allowed to use negative coefficients. That transforms the problem from a subset-sum style problem into a number theory problem.

Several edge cases are important:

- Arrays containing a `1` are automatically good, because `1 * 1 = 1`
- Arrays where all numbers share a common divisor greater than `1` can never produce `1`
- Arrays with only one element require that element itself to be `1`
- Very large arrays require an efficient iterative solution

The central challenge is discovering the mathematical property that determines whether `1` can be formed.

## Approaches

### Brute Force Approach

A brute force approach would attempt to try combinations of numbers and coefficients to see whether some linear combination equals `1`.

For example, we might try:

- Different subsets of the array
- Different coefficient values
- Positive and negative multipliers

This approach is theoretically correct because if a valid combination exists, exhaustive search would eventually find it.

However, the search space is enormous:

- There are `2^n` subsets
- Each subset can use infinitely many integer coefficients

Even if we artificially bounded the coefficients, the number of possibilities would still explode combinatorially.

This makes brute force completely infeasible for `n = 10^5`.

### Key Insight

This problem is actually an application of Bézout's Identity from number theory.

Bézout's Identity states:

For integers `a` and `b`, there exist integers `x` and `y` such that:

$$ax + by = gcd(a, b)$$

This extends to multiple numbers as well.

For an array:

$$a_1x_1 + a_2x_2 + \dots + a_nx_n = gcd(a_1, a_2, \dots, a_n)$$

Therefore:

- We can form exactly the multiples of the array's greatest common divisor
- We can form `1` if and only if the gcd of the entire array is `1`

So the problem reduces to:

1. Compute the gcd of all numbers in the array
2. Return `True` if the gcd equals `1`

This transforms the problem from exponential complexity into a simple linear scan.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential or very large | Attempts combinations of subsets and coefficients |
| Optimal | O(n log M) | O(1) | Computes gcd of entire array |

Here, `M` is the maximum value in the array.

## Algorithm Walkthrough

### Optimal Algorithm

1. Initialize a variable `current_gcd` with the first element of the array.

We need a running gcd across all elements. Starting with the first value allows us to iteratively combine gcd results.
2. Iterate through the remaining elements in the array.

For each number:

- Compute:

$$current\_gcd = gcd(current\_gcd, num)$$

This progressively reduces the gcd across the entire array.
3. Check whether the final gcd equals `1`.

If:

$$gcd(nums[0], nums[1], \dots, nums[n-1]) = 1$$

then Bézout's Identity guarantees that some integer combination produces `1`.
4. Return the result.

- Return `True` if gcd is `1`
- Otherwise return `False`

### Why it works

The algorithm works because Bézout's Identity guarantees that every integer linear combination of the array elements must be a multiple of the array's gcd.

If the gcd is greater than `1`, then every possible combination is divisible by that gcd, so producing `1` is impossible.

If the gcd is exactly `1`, then there always exists some combination of integer coefficients that produces `1`.

Therefore, checking whether the overall gcd equals `1` is both necessary and sufficient.

## Python Solution

```python
from typing import List
from math import gcd

class Solution:
    def isGoodArray(self, nums: List[int]) -> bool:
        current_gcd = nums[0]

        for num in nums[1:]:
            current_gcd = gcd(current_gcd, num)

            if current_gcd == 1:
                return True

        return current_gcd == 1
```

The implementation begins by initializing `current_gcd` with the first element of the array.

We then iterate through the remaining numbers and continuously update the running gcd using Python's built-in `math.gcd` function.

An optimization is included:

```python
if current_gcd == 1:
    return True
```

Once the gcd becomes `1`, it can never increase again. Since `gcd(1, x) = 1` for any integer `x`, we can terminate early.

Finally, the function returns whether the overall gcd equals `1`.

## Go Solution

```go
func gcd(a int, b int) int {
    for b != 0 {
        a, b = b, a%b
    }
    return a
}

func isGoodArray(nums []int) bool {
    currentGCD := nums[0]

    for i := 1; i < len(nums); i++ {
        currentGCD = gcd(currentGCD, nums[i])

        if currentGCD == 1 {
            return true
        }
    }

    return currentGCD == 1
}
```

The Go implementation follows the same logic as the Python version.

Since Go does not provide a built-in gcd function for integers, we implement the Euclidean algorithm manually.

The Euclidean algorithm repeatedly applies:

$$gcd(a, b) = gcd(b, a \bmod b)$$

until `b` becomes zero.

Go slices are used naturally for iteration, and integer overflow is not a concern because gcd operations only reduce values rather than increase them.

## Worked Examples

### Example 1

```
nums = [12, 5, 7, 23]
```

We compute the running gcd step by step.

| Step | Current Number | Running GCD |
| --- | --- | --- |
| Start | 12 | 12 |
| 1 | 5 | gcd(12, 5) = 1 |
| Early Stop | - | 1 |

Since the gcd becomes `1`, the array is good.

Indeed:

$$5 \times 3 + 7 \times (-2) = 1$$

So the answer is `True`.

### Example 2

```
nums = [29, 6, 10]
```

| Step | Current Number | Running GCD |
| --- | --- | --- |
| Start | 29 | 29 |
| 1 | 6 | gcd(29, 6) = 1 |
| Early Stop | - | 1 |

The gcd becomes `1`, so the array is good.

One valid combination is:

$$29 \times 1 + 6 \times (-3) + 10 \times (-1) = 1$$

The answer is `True`.

### Example 3

```
nums = [3, 6]
```

| Step | Current Number | Running GCD |
| --- | --- | --- |
| Start | 3 | 3 |
| 1 | 6 | gcd(3, 6) = 3 |

The final gcd is `3`, not `1`.

Every possible integer combination will therefore be divisible by `3`.

It is impossible to produce `1`.

The answer is `False`.

The Euclidean algorithm runs in logarithmic time relative to the input values. Since we perform one gcd computation per array element, the total complexity becomes `O(n log M)`.

The algorithm uses constant extra memory because it only stores the running gcd.

## Edge Cases

### Single Element Arrays

A single-element array can only form multiples of that element. Therefore:

- `[1]` is good
- `[x]` for `x > 1` is not good

This case is important because some implementations incorrectly assume multiple numbers are required. Our solution handles this naturally through the gcd computation.

### Arrays With a Common Divisor

Consider:

```
[6, 12, 18]
```

The gcd is `6`.

Every possible integer combination will therefore also be divisible by `6`. Since `1` is not divisible by `6`, forming `1` is impossible.

A naive implementation might incorrectly attempt coefficient searches without recognizing this invariant. The gcd-based approach immediately identifies the impossibility.

### Arrays Where GCD Becomes 1 Early

Consider:

```
[14, 25, 100, 200]
```

After computing:

$$gcd(14, 25) = 1$$

the result is already determined.

This is important for performance because very large arrays may contain thousands of remaining elements. Our implementation uses early termination to avoid unnecessary work.

### Large Numbers Near Constraint Limits

Inputs may contain values close to `10^9`.

A brute force coefficient search would be impossible for such values. The Euclidean gcd algorithm remains extremely efficient even for large integers, which is why the optimal solution scales comfortably within the problem constraints.
