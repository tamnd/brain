## Problem Understanding

The problem describes a scenario where `n` passengers are boarding an airplane with exactly `n` seats. The first passenger has lost their ticket and chooses a seat randomly. Every subsequent passenger will either sit in their assigned seat if it is free, or pick a seat randomly if their assigned seat is taken. The task is to compute the probability that the `nth` passenger ends up in their assigned seat.

The input `n` represents the total number of passengers (and seats). The output is a floating-point number between 0 and 1 representing the probability that the last passenger sits in their assigned seat. The constraints indicate that `n` can be as large as `10^5`, which means a naive simulation approach will be too slow.

Important edge cases include `n = 1` (only one passenger, must sit in their seat) and `n = 2` (a simple scenario that reveals the probability behavior). The problem guarantees that `n` is at least 1, so there are no empty plane scenarios.

## Approaches

The brute-force approach would simulate all possible seat choices for the first passenger, recursively determine how subsequent passengers choose seats, and calculate the probability that the last passenger sits in their seat. While this would give the correct answer, the number of permutations grows factorially with `n` (O(n!)), which is infeasible for large `n`.

The key insight for an optimal solution is that, after the first passenger chooses a seat, only two outcomes matter for the last passenger: either the first passenger picks their own seat (seat 1) or the last passenger's seat (seat `n`). If the first passenger picks seat 1, the last passenger will definitely get their seat. If the first passenger picks seat `n`, the last passenger will not get their seat. For any other seat `k`, the problem reduces to the same scenario with fewer seats. Using this recursive reasoning, the probability for `n >= 2` converges to 0.5. This remarkable property simplifies the solution dramatically.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n!) | O(n) | Simulates all permutations of seat choices, impractical for large n |
| Optimal | O(1) | O(1) | Uses the insight that probability converges to 0.5 for n >= 2 |

## Algorithm Walkthrough

1. Check if `n` equals 1. If so, return 1.0, since the only passenger will occupy their seat.
2. For any `n` greater than 1, return 0.5. This is based on the recursive probability observation: the first passenger randomly chooses either their own seat, the last passenger's seat, or another passenger's seat, but mathematically, the probability for the last passenger to get their seat stabilizes at 0.5 for `n >= 2`.

**Why it works:** The invariant is that, at each stage after the first passenger, either the last passenger's seat is still available or it has already been taken. The recursive scenario collapses such that each intermediate choice does not affect the overall probability beyond the first and last seat, leading to a constant 0.5 probability for all `n >= 2`.

## Python Solution

```python
class Solution:
    def nthPersonGetsNthSeat(self, n: int) -> float:
        if n == 1:
            return 1.0
        return 0.5
```

In the Python implementation, we first handle the trivial case `n = 1`, returning `1.0` directly. For all other cases, we return `0.5` based on the probability reasoning. There is no need for loops, recursion, or additional data structures, making this solution extremely efficient.

## Go Solution

The Go implementation mirrors the Python version. The `if` statement handles the base case, and the constant `0.5` is returned for all other values of `n`. Go does not require handling floating-point literals differently here because `0.5` is correctly interpreted as a `float64`.

One edge case is `n = 1`. This is trivial but important, as the algorithm must return `1.0`. Our solution handles this explicitly with an `if` statement.

Another edge case is `n = 2`, the smallest non-trivial number of passengers. Here, the probability is exactly 0.5, which our solution correctly returns. This edge case ensures the base observation is correctly applied.

Finally, very large `n` such as `n = 10^5` is an edge case that tests efficiency. A brute-force simulation would fail due to factorial growth, but our constant-time solution handles it effortlessly. This ensures scalability and confirms the mathematical reasoning holds for all valid inputs.
