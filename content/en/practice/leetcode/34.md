# LeetCode 34, Find First and Last Position of Element in Sorted Array

## Problem Understanding

This problem gives us a sorted integer array `nums` and a target value `target`. The array is sorted in non-decreasing order, which means elements can stay the same or increase as we move from left to right.

Our task is to find the first occurrence and the last occurrence of the target value inside the array. The result should be returned as a two-element array:

- The first value is the index of the first appearance of `target`
- The second value is the index of the last appearance of `target`

If the target does not exist in the array at all, we must return `[-1, -1]`.

For example:

- In `[5,7,7,8,8,10]`, the target `8` appears at indices `3` and `4`, so the answer is `[3,4]`
- If the target is `6`, it does not exist in the array, so the answer is `[-1,-1]`

The important requirement is the runtime complexity. The problem explicitly requires an `O(log n)` solution. This immediately rules out linear scanning approaches for the final implementation because scanning the entire array takes `O(n)` time.

The constraints tell us several important things:

- The array can contain up to `10^5` elements, so efficiency matters
- The array is already sorted, which strongly suggests binary search
- Values may be negative or positive, but that does not affect the algorithm because comparisons still work normally
- The array may be empty, so we must handle that safely

Several edge cases are especially important:

- The array is empty
- The target does not exist
- The target appears exactly once
- Every element equals the target
- The target appears at the beginning or end of the array
- Multiple duplicate values exist

A naive implementation can easily make mistakes around boundaries, especially when duplicates are involved.

## Approaches

### Brute Force Approach

The most straightforward solution is to scan the array from left to right.

We can keep track of:

- The first index where we encounter the target
- The last index where we encounter the target

As we iterate:

- When we see the target for the first time, store the index as the starting position
- Continue scanning and keep updating the ending position whenever we see the target again

At the end:

- If we never found the target, return `[-1, -1]`
- Otherwise return the stored indices

This approach is simple and correct because it checks every element directly. However, it takes `O(n)` time in the worst case, which violates the problem requirement of `O(log n)`.

### Optimal Approach

The key observation is that the array is already sorted.

In a sorted array, binary search allows us to efficiently locate boundaries. Instead of searching for just any occurrence of the target, we can perform two separate binary searches:

1. One binary search to find the leftmost occurrence
2. Another binary search to find the rightmost occurrence

The important insight is that duplicates appear contiguously in a sorted array. Once we find one occurrence of the target, all equal values must be grouped together.

To find the first occurrence:

- When we find the target, we continue searching to the left
- We want the smallest valid index

To find the last occurrence:

- When we find the target, we continue searching to the right
- We want the largest valid index

Each binary search takes `O(log n)` time, so the total runtime remains `O(log n)`.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Scan entire array and track first and last occurrence |
| Optimal | O(log n) | O(1) | Use two binary searches to find left and right boundaries |

## Algorithm Walkthrough

1. First, create a helper binary search function that can search for either the left boundary or the right boundary.
2. Initialize two pointers:

- `left = 0`
- `right = len(nums) - 1`
3. While `left <= right`, compute the middle index:

$$mid = left + \frac{right - left}{2}$$
4. Compare `nums[mid]` with the target.
5. If `nums[mid] < target`, the target can only exist on the right side, so move:

- `left = mid + 1`
6. If `nums[mid] > target`, the target can only exist on the left side, so move:

- `right = mid - 1`
7. If `nums[mid] == target`, store the current index as a possible answer.
8. If we are searching for the first occurrence:

- Continue searching left:
- `right = mid - 1`
9. If we are searching for the last occurrence:

- Continue searching right:
- `left = mid + 1`
10. After the loop finishes, return the stored index.
11. Run the helper function twice:

- Once for the first occurrence
- Once for the last occurrence

1. Return both results together.

This approach works because binary search repeatedly cuts the search space in half while still preserving the possibility of finding a better boundary index.

## Python Solution

```
from typing import List

class Solution:
    def searchRange(self, nums: List[int], target: int) -> List[int]:
        def binary_search(find_first: bool) -> int:
            left = 0
            right = len(nums) - 1
            result = -1

            while left <= right:
                mid = left + (right - left) // 2

                if nums[mid] < target:
                    left = mid + 1
                elif nums[mid] > target:
                    right = mid - 1
                else:
                    result = mid

                    if find_first:
                        right = mid - 1
                    else:
                        left = mid + 1

            return result

        first_position = binary_search(True)

        if first_position == -1:
            return [-1, -1]

        last_position = binary_search(False)

        return [first_position, last_position]
```

The solution uses a nested helper function called `binary_search`. This function performs a standard binary search with one important modification: when the target is found, it does not stop immediately.

Instead, it stores the current index in `result` and continues searching for a better boundary position.

When `find_first` is `True`, the search continues toward the left side of the array because we are trying to locate the earliest occurrence.

When `find_first` is `False`, the search continues toward the right side because we are trying to locate the latest occurrence.

The main function first searches for the first occurrence. If the result is `-1`, the target does not exist, so we immediately return `[-1, -1]`.

Otherwise, we perform the second binary search to find the last occurrence and return both indices.

The implementation uses only constant extra space and satisfies the required `O(log n)` runtime complexity.

## Go Solution

```
func searchRange(nums []int, target int) []int {
    binarySearch := func(findFirst bool) int {
        left := 0
        right := len(nums) - 1
        result := -1

        for left <= right {
            mid := left + (right-left)/2

            if nums[mid] < target {
                left = mid + 1
            } else if nums[mid] > target {
                right = mid - 1
            } else {
                result = mid

                if findFirst {
                    right = mid - 1
                } else {
                    left = mid + 1
                }
            }
        }

        return result
    }

    firstPosition := binarySearch(true)

    if firstPosition == -1 {
        return []int{-1, -1}
    }

    lastPosition := binarySearch(false)

    return []int{firstPosition, lastPosition}
}
```

The Go implementation follows the same logic as the Python version. The helper binary search is implemented as an anonymous function assigned to `binarySearch`.

Slices in Go naturally handle empty arrays safely because `len(nums)` becomes `0`, making `right = -1`. The loop condition `left <= right` immediately fails, so the function correctly returns `-1`.

The midpoint calculation uses:

```
mid := left + (right-left)/2
```

This form is preferred because it avoids potential integer overflow that could occur with:

```
(left + right) / 2
```

Although overflow is unlikely under these constraints, this is considered best practice for binary search implementations.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(log n) | Two binary searches are performed, each taking O(log n) |
| Space | O(1) | Only a few variables are used regardless of input size |

The runtime remains logarithmic because binary search repeatedly halves the search space. Even though we run binary search twice, constants are ignored in Big O notation:

$$O(\log n) + O(\log n) = O(\log n)$$

The space complexity is constant because no additional data structures proportional to input size are allocated.

## Edge Cases

### Empty Array

If the input array is empty, `right` becomes `-1`. The binary search loop never executes because `left <= right` is false immediately.

This prevents out-of-bounds access and correctly returns `[-1, -1]`.

### Target Does Not Exist

Consider:

```
nums = [1,2,3,4,5]
target = 6
```

Binary search eventually exhausts the search space without finding the target. Since `result` remains `-1`, the implementation correctly identifies that the target does not exist.

This is important because some incorrect implementations accidentally return nearby indices instead of confirming the exact value exists.

### All Elements Equal Target

Consider:

```
nums = [8,8,8,8,8]
target = 8
```

A standard binary search may stop at any index. However, this implementation continues searching after finding the target.

The first binary search keeps moving left until it finds index `0`.

The second binary search keeps moving right until it finds index `4`.

This ensures the full range is returned correctly.

### Single Element Array

Consider:

```
nums = [8]
target = 8
```

Both binary searches correctly return index `0`.

This case is important because boundary updates can easily create off-by-one errors in arrays of size one.

### Target at Array Boundaries

Consider:

```
nums = [2,2,2,3,4]
target = 2
```

The first occurrence is at the very beginning of the array.

Similarly:

```
nums = [1,2,3,4,4,4]
target = 4
```

The last occurrence is at the end of the array.

The implementation handles both correctly because the search continues toward the relevant boundary even after finding the target.
