## Solution

## Problem Understanding

This problem asks us to implement a custom version of JavaScript's built in `Function.prototype.call` behavior. Specifically, we need to add a new method named `callPolyfill` to all functions. This method should allow us to invoke a function while explicitly controlling what the `this` keyword refers to inside that function.

In JavaScript, the value of `this` depends on how a function is called. If a normal function is invoked without an explicit context, `this` may become `undefined` in strict mode or refer to the global object in non strict mode. The goal of `callPolyfill` is to simulate the behavior of `Function.prototype.call` without actually using the built in `call` method.

The method receives:

1. An object `obj`, which should become the `this` context for the function.
2. Any number of additional arguments, which should be passed normally to the target function.

For example:

```
function add(b) {
    return this.a + b;
}
```

If we execute:

```
add.callPolyfill({a: 5}, 7)
```

then inside the function:

```
this === {a: 5}
```

and the result becomes:

```
5 + 7 = 12
```

The constraints are fairly small, but this problem is not about algorithmic complexity. Instead, it tests understanding of JavaScript function invocation mechanics and prototype manipulation.

The important guarantees are:

- `args[0]` is always a non null object.
- The number of arguments is relatively small.
- We are forbidden from using the built in `Function.call`.

Several edge cases are important:

- The target object may already contain properties with names we choose internally.
- The function may receive zero or many extra arguments.
- The function may return any type of value.
- The implementation must preserve the correct `this` binding even for dynamically supplied objects.

A naive implementation that hardcodes property names could accidentally overwrite existing properties on the target object, creating subtle bugs.

## Approaches

### Brute Force Approach

The brute force idea is to temporarily attach the function directly onto the target object as a property, invoke it as a method of that object, then remove the temporary property afterward.

For example:

```
obj.temp = fn;
obj.temp(arg1, arg2);
delete obj.temp;
```

This works because when a function is called as a property of an object, JavaScript automatically sets `this` to that object.

The major problem with a naive brute force implementation is that the temporary property name may already exist on the object. If we use something common like `"temp"`, we could overwrite existing data.

Although this approach works conceptually, it is unsafe unless we ensure the temporary key is unique.

### Optimal Approach

The optimal solution uses a `Symbol` as the temporary property key.

A JavaScript `Symbol` is guaranteed to be unique, even if other symbols have the same description. This completely avoids accidental property collisions.

The process becomes:

1. Create a unique symbol.
2. Assign the function to `obj[symbol]`.
3. Invoke `obj[symbol](...args)`.
4. Remove the temporary property.
5. Return the function result.

This solution is both safe and clean. It reproduces how `call` behaves while avoiding naming conflicts.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(k) | O(1) | Uses a normal temporary property name, unsafe due to collisions |
| Optimal | O(k) | O(1) | Uses a unique `Symbol` key to safely bind `this` |

Here, `k` represents the number of arguments passed into the function.

## Algorithm Walkthrough

1. Extend `Function.prototype` with a method named `callPolyfill`.

This ensures every function automatically gains access to the new method.
2. Inside `callPolyfill`, create a unique `Symbol`.

A symbol guarantees that no existing property on the object can conflict with the temporary function reference.
3. Store the current function onto the object using the symbol as the key.

Since `this` inside `callPolyfill` refers to the function being invoked, we can write:

```
obj[symbol] = this;
```
4. Invoke the function through the object.

Calling:

```
obj[symbol](...args)
```

automatically binds `this` to `obj`.
5. Store the returned value.

The original function may return anything, so we must preserve and return that result.
6. Delete the temporary property.

This restores the object to its original state and prevents unnecessary mutations.
7. Return the stored result.

### Why it works

The key JavaScript rule is that when a function is invoked as `object.method()`, the `this` keyword inside the function becomes `object`.

By temporarily attaching the function to the provided object and invoking it as a method, we force JavaScript to bind `this` correctly. Using a unique `Symbol` guarantees we never overwrite existing properties, so the implementation is both correct and safe.

## Python Solution

LeetCode provides this problem in JavaScript, not Python. However, to mirror the requested format, the following Python code demonstrates the same conceptual behavior.

```python
from types import MethodType
from typing import Any, Callable

class Solution:
    def call_polyfill(
        self,
        fn: Callable,
        obj: object,
        *args: Any
    ) -> Any:
        """
        Simulates JavaScript-style call() behavior in Python.
        """

        bound_function = MethodType(fn, obj)
        return bound_function(*args)
```

The Python implementation uses `MethodType` from the `types` module to dynamically bind a function to an object instance. This mirrors the JavaScript concept of changing the `this` context.

The `call_polyfill` method accepts:

- A function `fn`
- An object `obj`
- Any additional positional arguments

`MethodType(fn, obj)` creates a bound method where `obj` becomes the equivalent of JavaScript's `this`. The resulting bound function is then executed with the supplied arguments.

Although Python and JavaScript handle method binding differently internally, the conceptual behavior is the same.

## Go Solution

Go does not support JavaScript style dynamic `this` binding. However, the following code demonstrates a structurally similar concept using closures.

```go
package main

import "fmt"

type Context struct {
	A int
}

func CallPolyfill(
	fn func(*Context, int) int,
	ctx *Context,
	arg int,
) int {
	return fn(ctx, arg)
}

func add(ctx *Context, b int) int {
	return ctx.A + b
}

func main() {
	ctx := &Context{A: 5}
	result := CallPolyfill(add, ctx, 7)

	fmt.Println(result)
}
```

Go does not have a runtime equivalent of JavaScript's `this` rebinding. Instead, context is passed explicitly as a parameter.

The primary difference from JavaScript is that Go functions do not implicitly capture a dynamic execution context. Because of this, we simulate the behavior by manually passing the context object into the function.

## Worked Examples

    return this.a + b;

## Edge Cases

One important edge case occurs when the target object already contains a property name identical to the temporary property used internally. A naive implementation using names like `"temp"` or `"fn"` could overwrite existing data. Using a `Symbol` completely avoids this issue because symbols are guaranteed unique.

Another important edge case is when the function accepts no additional arguments. Some implementations incorrectly assume there will always be parameters after the object. Using the rest operator `...args` allows the function to work correctly whether there are zero or many arguments.

A third edge case involves functions that return complex values such as objects, arrays, `null`, or `undefined`. The implementation must not alter or wrap the return value in any way. By storing the invocation result directly and returning it unchanged, the solution preserves exact JavaScript behavior.

A final subtle edge case is object cleanup. If the temporary property is not deleted after invocation, the object becomes permanently mutated. The optimal implementation removes the temporary symbol immediately after the call completes, restoring the original object state correctly.
