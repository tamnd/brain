## Solution

## Problem Understanding

The problem asks us to analyze the `Spotify` table and determine how many times each artist appears in the ranking list. Each row in the table represents a song with a unique `id`, a `track_name`, and an `artist`. The goal is to count the number of occurrences of each artist and return a table that lists each artist along with their occurrence count. The results must be sorted first by the occurrence count in descending order, so that the most frequent artists appear first, and then by the artist's name in ascending order for ties.

The input is a relational table with three columns, and there is no restriction on the number of songs or artists. The primary constraint is that the `id` is unique, which guarantees that no two rows are identical. Important edge cases include artists with only one song, multiple artists with the same number of songs, and an empty table. The output should always reflect the ordering rules and properly count all occurrences.

## Approaches

The brute-force approach would involve scanning the table multiple times to count occurrences for each artist individually. While this would produce correct results, it would be inefficient for large datasets since repeated scans are unnecessary. Specifically, this could lead to a quadratic number of operations if the table contains many rows and many artists.

The optimal approach leverages SQL's aggregation and sorting features. By grouping the rows by `artist` and counting the number of rows in each group, we efficiently determine the occurrence count for each artist. Sorting by the count in descending order and then by the artist name in ascending order produces the required output. This approach is direct, efficient, and scales well with larger datasets.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * m) | O(m) | Scan for each artist individually; n = number of rows, m = number of unique artists |
| Optimal | O(n log n) | O(m) | Group by artist, count occurrences, sort results; n = number of rows, m = number of unique artists |

## Algorithm Walkthrough

1. Begin by selecting the `artist` column from the `Spotify` table.
2. Use the `COUNT(*)` aggregation function to compute the total number of occurrences for each artist.
3. Group the results by the `artist` column to ensure each artist appears exactly once in the result.
4. Sort the grouped results by the count in descending order, ensuring that artists with higher occurrence counts come first.
5. Apply a secondary sort by the artist's name in ascending order to break ties between artists with the same number of occurrences.
6. Return the final table containing the `artist` and `occurrences` columns.

This algorithm works because SQL's `GROUP BY` guarantees that each artist is represented once, and `COUNT(*)` accurately counts the rows for each group. Sorting by multiple criteria ensures the output meets the specified ordering requirements.

## Python Solution

```python
# Python solution using SQLite syntax for LeetCode
import sqlite3
from typing import List, Tuple

def count_artist_occurrences(cursor: sqlite3.Cursor) -> List[Tuple[str, int]]:
    query = """
        SELECT artist, COUNT(*) AS occurrences
        FROM Spotify
        GROUP BY artist
        ORDER BY occurrences DESC, artist ASC
    """
    cursor.execute(query)
    return cursor.fetchall()
```

The Python implementation defines a function that executes a SQL query against a provided database cursor. The `GROUP BY` ensures each artist appears once, `COUNT(*)` computes occurrences, and `ORDER BY occurrences DESC, artist ASC` enforces the required ordering. Finally, `fetchall()` returns the results as a list of tuples.

## Go Solution

```go
package main

import (
    "database/sql"
    _ "github.com/mattn/go-sqlite3"
    "fmt"
)

func countArtistOccurrences(db *sql.DB) ([]struct {
    Artist      string
    Occurrences int
}, error) {
    query := `
        SELECT artist, COUNT(*) AS occurrences
        FROM Spotify
        GROUP BY artist
        ORDER BY occurrences DESC, artist ASC
    `
    rows, err := db.Query(query)
    if err != nil {
        return nil, err
    }
    defer rows.Close()

    var results []struct {
        Artist      string
        Occurrences int
    }
    for rows.Next() {
        var artist string
        var occurrences int
        if err := rows.Scan(&artist, &occurrences); err != nil {
            return nil, err
        }
        results = append(results, struct {
            Artist      string
            Occurrences int
        }{artist, occurrences})
    }
    return results, nil
}
```

In Go, we query the database and iterate over the returned rows. Each row is scanned into a struct, and the results are appended to a slice. Go requires explicit error handling and the use of slices rather than lists. The SQL query itself is identical to the Python version.

## Worked Examples

For the given example:

1. The table contains five songs with three unique artists: "DJ Khalid", "Ed Sheeran", and "Sia".
2. `COUNT(*)` for each artist produces: DJ Khalid → 2, Ed Sheeran → 2, Sia → 1.
3. Sorting by `occurrences DESC` gives 2 → 2 → 1. For the tie of 2, sorting by `artist ASC` gives "DJ Khalid" before "Ed Sheeran".
4. Final output:

| artist | occurrences |
| --- | --- |
| DJ Khalid | 2 |
| Ed Sheeran | 2 |
| Sia | 1 |

## Edge Cases

One important edge case is when the table is empty. The implementation correctly returns an empty list because the SQL aggregation on zero rows naturally yields no output. Another edge case occurs when multiple artists have the same occurrence count. Here, the secondary alphabetical sort guarantees deterministic ordering. A third edge case is artists with special characters or spaces in their names; SQL correctly handles strings in the `GROUP BY` and `ORDER BY` clauses, so the implementation remains correct regardless of the string content. These considerations ensure robustness across a wide variety of inputs.
