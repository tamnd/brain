# LeetCode 2661, First Completely Painted Row or Column

## Problem Understanding

The problem gives us two inputs:

1. An array `arr` of length `m * n`
2. A matrix `mat` with dimensions `m x n`

Both structures contain every integer from `1` to `m * n` exactly once. In other words, every value is unique, and both `arr` and `mat` are permutations of the same set of numbers.

We process the array `arr` from left to right. For each value `arr[i]`, we locate that value inside the matrix `mat` and mark its cell as painted.

The goal is to return the earliest index `i` such that after painting `arr[i]`, at least one entire row or one entire column of the matrix has been fully painted.

The key detail is that we are not asked to simulate arbitrary painting operations. Every number appears exactly once in the matrix, so each value directly identifies a single matrix cell.

For example, suppose:

```
mat = [
  [1, 4],
  [2, 3]
]
```

and:

```
arr = [1, 3, 4, 2]
```

At index `0`, number `1` is painted.

At index `1`, number `3` is painted.

At index `2`, number `4` is painted.

Now the first row `[1, 4]` is fully painted, so the answer is `2`.

The constraints are important:

- `m * n <= 100000`
- Every number is unique
- Values range from `1` to `m * n`

These constraints tell us that:

- We need something close to linear time.
- Repeatedly scanning rows or columns after every paint operation would be too slow.
- A hash map or indexed lookup structure will likely be necessary.

An important guarantee is that some row or column must eventually become fully painted, because all cells are eventually painted.

Some edge cases worth thinking about:

- A matrix with only one row
- A matrix with only one column
- The first completed structure being a column instead of a row
- The completion happening very early
- Very large matrices where inefficient rescanning would time out

## Approaches

### Brute Force Approach

A straightforward solution is to simulate the painting process directly.

For every value in `arr`:

1. Find its position inside the matrix
2. Mark that cell as painted
3. Scan the entire row to check if every cell is painted
4. Scan the entire column to check if every cell is painted

This approach is correct because after every operation we explicitly verify whether a row or column has become completely painted.

However, it is inefficient.

Finding the position of a number by scanning the matrix costs `O(m * n)`. Even if we optimize lookup with a hash map, repeatedly scanning rows and columns still becomes expensive.

In the worst case:

- There are `m * n` painting operations
- Each operation may scan up to `m + n` cells

This leads to roughly `O(m * n * (m + n))`, which is too slow for `100000` total cells.

### Optimal Approach

The key observation is that we do not need to repeatedly scan rows and columns.

Instead, we can maintain:

- A counter for how many painted cells each row has
- A counter for how many painted cells each column has

We also create a mapping:

```
number -> (row, column)
```

Then, when we paint a value:

1. Instantly locate its coordinates
2. Increment the corresponding row count
3. Increment the corresponding column count
4. Check whether:

- `row_count[row] == n`
- or `col_count[col] == m`

As soon as one condition becomes true, we return the current index.

This works because a row is fully painted exactly when its painted-cell count equals the number of columns. The same logic applies to columns.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | `O(m * n * (m + n))` | `O(m * n)` | Repeatedly rescans rows and columns |
| Optimal | `O(m * n)` | `O(m * n)` | Uses coordinate mapping and counters |

## Algorithm Walkthrough

1. Create a hash map that stores the coordinates of every number in the matrix.

For every cell `(r, c)` in `mat`, store:

```
position[mat[r][c]] = (r, c)
```

This allows constant-time lookup of a number's location.
2. Create two arrays:

- `row_count` of size `m`
- `col_count` of size `n`

These arrays track how many painted cells exist in each row and column.
3. Iterate through `arr` from left to right.

At each index `i`:

- Retrieve the coordinates `(r, c)` of `arr[i]`
- Increment `row_count[r]`
- Increment `col_count[c]`
4. After updating the counts, check whether:

```
row_count[r] == n
```

If true, the entire row is painted.
5. Also check whether:

```
col_count[c] == m
```

If true, the entire column is painted.
6. As soon as either condition becomes true, return the current index `i`.
7. Since all cells are eventually painted, at least one row or column must eventually complete.

### Why it works

The algorithm maintains the invariant that:

- `row_count[r]` always equals the number of painted cells in row `r`
- `col_count[c]` always equals the number of painted cells in column `c`

Because every value is unique and painted exactly once, the counters never overcount.

A row is fully painted if and only if all `n` cells in that row have been painted, which is exactly when `row_count[r] == n`.

Similarly, a column is fully painted if and only if `col_count[c] == m`.

Since we process `arr` in order and return immediately upon completion, the returned index is the smallest valid index.

## Python Solution

```python
from typing import List

class Solution:
    def firstCompleteIndex(self, arr: List[int], mat: List[List[int]]) -> int:
        rows = len(mat)
        cols = len(mat[0])

        position = {}

        for r in range(rows):
            for c in range(cols):
                position[mat[r][c]] = (r, c)

        row_count = [0] * rows
        col_count = [0] * cols

        for index, value in enumerate(arr):
            r, c = position[value]

            row_count[r] += 1
            col_count[c] += 1

            if row_count[r] == cols:
                return index

            if col_count[c] == rows:
                return index

        return -1
```

The implementation begins by determining the matrix dimensions.

Next, we build the `position` dictionary. This preprocessing step is crucial because it allows us to instantly locate any number from `arr` inside the matrix.

The `row_count` and `col_count` arrays track painting progress. Every time a value is processed:

- We find its matrix coordinates
- Increment the corresponding row and column counters
- Check whether either structure has become fully painted

The moment a row or column reaches its full size, we return the current index immediately because the problem asks for the earliest such index.

The final `return -1` is technically unnecessary under the problem guarantees, but including it keeps the function structurally complete.

## Go Solution

```go
func firstCompleteIndex(arr []int, mat [][]int) int {
	rows := len(mat)
	cols := len(mat[0])

	position := make(map[int][2]int)

	for r := 0; r < rows; r++ {
		for c := 0; c < cols; c++ {
			position[mat[r][c]] = [2]int{r, c}
		}
	}

	rowCount := make([]int, rows)
	colCount := make([]int, cols)

	for i, value := range arr {
		coords := position[value]
		r := coords[0]
		c := coords[1]

		rowCount[r]++
		colCount[c]++

		if rowCount[r] == cols {
			return i
		}

		if colCount[c] == rows {
			return i
		}
	}

	return -1
}
```

The Go implementation follows exactly the same logic as the Python version.

One difference is that Go does not have tuples, so the coordinate mapping uses:

```go
map[int][2]int
```

where the two-element array stores `(row, column)`.

Slices are used for `rowCount` and `colCount`, and integer overflow is not a concern because the maximum counts are at most `100000`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | `O(m * n)` | Each matrix cell and each array element is processed once |
| Space | `O(m * n)` | The position map stores every matrix value |

The preprocessing step scans the matrix once, which costs `O(m * n)`.

The main loop processes each value in `arr` exactly once, also costing `O(m * n)` because `arr` contains exactly `m * n` elements.

All operations inside the loop are constant time due to hash map lookups and counter updates.

## Edge Cases

A very important edge case is a matrix with only one row. In this situation, the first row becomes complete only after all its elements are painted. A buggy implementation might incorrectly assume square matrices or mishandle row and column dimensions. This solution correctly handles the case because it independently tracks row and column sizes using `rows` and `cols`.

Another important edge case is a matrix with only one column. In that scenario, every painted cell immediately contributes toward completing the only column. The implementation correctly checks:

```
col_count[c] == rows
```

so the column completion is detected exactly when all rows in that column have been painted.

A subtle edge case occurs when a row and a column become complete at the same step. This can happen in small matrices such as:

```
[[1,4],
 [2,3]]
```

In the example, painting `4` completes both the first row and the second column simultaneously. The implementation handles this naturally because it checks both completion conditions immediately after updating the counters.

Another possible source of bugs is inefficient repeated scanning. With up to `100000` cells, rescanning entire rows and columns after every paint operation would be too slow. The counter-based approach avoids this completely by maintaining incremental state updates in constant time.
