## Solution

## Problem Understanding

In this problem, we are given two sets of positions on a one dimensional number line:

- `hens[i]` represents the starting position of the `i-th` hen.
- `grains[j]` represents the position of the `j-th` grain.

A hen can move left or right by exactly one unit per second. Whenever a hen reaches the same position as a grain, it instantly eats that grain. Multiple hens move simultaneously, and a single hen may eat multiple grains.

The goal is to determine the minimum amount of time required so that every grain can be eaten if all hens move optimally.

The important detail is that we are minimizing the maximum time taken by any hen. Since all hens move simultaneously, we are effectively searching for the smallest time `T` such that there exists some assignment of grains to hens where every assigned grain can be reached within `T` seconds.

The constraints are large:

- Up to `2 * 10^4` hens
- Up to `2 * 10^4` grains
- Positions up to `10^9`

These limits immediately rule out exponential assignment approaches or any solution that tries all pairings between hens and grains. We need something close to `O((n + m) log M)`, where `M` is the coordinate range.

Several edge cases are especially important:

- Multiple grains may lie on only one side of a hen.
- A hen may need to move left first, then right.
- There may be many more grains than hens.
- Hens and grains can already overlap.
- Coordinates may be very large, so coordinate compression or direct simulation on the line is impossible.

The key challenge is determining whether all grains can be covered within a candidate time limit.

## Approaches

### Brute Force Approach

A brute force solution would attempt to assign grains to hens in every possible way. For each assignment, we could compute the maximum travel time required and choose the minimum over all assignments.

This approach is correct because it explicitly explores every valid configuration. However, it is completely infeasible. Even assigning each grain independently to one of `n` hens produces `n^m` possibilities, which is astronomically large for `m = 2 * 10^4`.

Another naive variant would simulate all movements or perform dynamic programming over subsets, but those approaches also explode combinatorially.

The core issue is that the search space is enormous.

### Key Insight

The critical observation is that the answer is monotonic.

Suppose all grains can be eaten within time `T`. Then they can also be eaten within any larger time `T + 1`, because hens can simply wait or move less aggressively.

This monotonic property suggests binary search on the answer.

The remaining problem becomes:

> Given a time limit `T`, can all grains be eaten?

After sorting both arrays, we can greedily process grains from left to right. For each hen, we determine the farthest consecutive range of grains it can cover within time `T`.

The tricky part is determining how far a single hen can travel if it may need to visit grains on both sides.

Suppose a hen starts at position `h` and the leftmost uncovered grain is at `g <= h`.

If the hen first moves left to `g`, then moves right afterward, there are two possible movement strategies:

1. Move left first, then fully sweep right.
2. Move right partially first, then come back.

The optimal reachable right endpoint under time `T` is:

$$\max\left(
h + \frac{T - 2(h-g)}{1},
h + \frac{T - (h-g)}{2}
\right)$$

This can be simplified into the standard formula used in accepted solutions:

$$\max(
T - 2d,
(T-d)/2
)$$

where:

$$d = h - g$$

This gives the maximum extra distance we can still move right after covering the left side.

Using this observation, we can greedily assign as many consecutive grains as possible to each hen.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |

## Algorithm Walkthrough

1. Sort both `hens` and `grains`.

Sorting is essential because movement happens on a line. Once sorted, optimal assignments become contiguous. A hen will always consume a consecutive block of grains rather than scattered positions.
2. Binary search the answer.

The minimum possible answer is `0`.

The maximum possible answer can safely be `10^9`, since coordinates are bounded by `10^9`.
3. For each candidate time `mid`, run a feasibility check.

The feasibility function answers:

> Can all grains be eaten within `mid` seconds?
4. Maintain a pointer `j` to the leftmost uncovered grain.

We process hens from left to right.
5. For the current hen at position `h`, determine how many grains it can cover.

There are two cases:

- Grain is already to the right of the hen.
- Grain lies to the left of the hen.
6. If the grain is to the right, the hen can simply walk right up to distance `mid`.

So all grains with:

$$grain \le h + mid$$

are coverable.
7. If the grain lies to the left, define:

$$d = h - grain$$

The hen must first spend `d` time reaching that grain.

After that, the best possible rightward reach is:

$$h + \max(T - 2d, (T-d)/2)$$

This formula captures the two optimal movement patterns.
8. Advance `j` while grains remain reachable.

Every grain consumed by this hen is permanently assigned.
9. After processing all hens, check whether every grain was covered.

If yes, the candidate time is feasible.
10. Use standard binary search logic.

- If feasible, try smaller time.
- Otherwise, try larger time.

1. Return the smallest feasible time.

### Why it works

The correctness relies on two properties.

First, after sorting, an optimal assignment always partitions grains into contiguous segments assigned to hens in order. Crossing assignments would only increase unnecessary travel.

Second, the feasibility condition is monotonic. If time `T` works, every larger value also works. Therefore binary search correctly finds the minimum feasible time.

The greedy feasibility check is optimal because each hen always consumes the maximum possible prefix of remaining grains. Delaying a grain to a later hen can never help if the current hen can already cover it.

## Python Solution

```python
from typing import List

class Solution:
    def minimumTime(self, hens: List[int], grains: List[int]) -> int:
        hens.sort()
        grains.sort()

        def can_finish(time: int) -> bool:
            grain_index = 0
            total_grains = len(grains)

            for hen in hens:
                if grain_index >= total_grains:
                    return True

                # If the leftmost uncovered grain is already to the right
                if grains[grain_index] >= hen:
                    reach = hen + time
                else:
                    distance_left = hen - grains[grain_index]

                    if distance_left > time:
                        continue

                    option1 = time - 2 * distance_left
                    option2 = (time - distance_left) // 2

                    reach = hen + max(option1, option2)

                while (
                    grain_index < total_grains
                    and grains[grain_index] <= reach
                ):
                    grain_index += 1

            return grain_index == total_grains

        left = 0
        right = 10**9

        while left < right:
            mid = (left + right) // 2

            if can_finish(mid):
                right = mid
            else:
                left = mid + 1

        return left
```

The implementation begins by sorting both arrays. This guarantees that we process hens and grains in spatial order, which is necessary for the greedy assignment strategy.

The nested `can_finish` function performs the feasibility check for a candidate time value. It maintains a pointer `grain_index` that tracks the leftmost uncovered grain.

For each hen, we compute the farthest reachable position under the current time limit. If the current grain lies to the right of the hen, the logic is simple because the hen only moves right.

If the grain lies to the left, we compute the travel cost needed to reach it first. The formula then determines the maximum additional distance the hen can still travel rightward.

After computing the reachable boundary, we greedily consume every grain within reach by advancing the pointer.

The outer binary search repeatedly tests candidate times until it converges on the smallest feasible value.

## Go Solution

```go
package main

import (
	"sort"
)

func minimumTime(hens []int, grains []int) int {
	sort.Ints(hens)
	sort.Ints(grains)

	canFinish := func(time int) bool {
		grainIndex := 0
		totalGrains := len(grains)

		for _, hen := range hens {
			if grainIndex >= totalGrains {
				return true
			}

			reach := 0

			if grains[grainIndex] >= hen {
				reach = hen + time
			} else {
				distanceLeft := hen - grains[grainIndex]

				if distanceLeft > time {
					continue
				}

				option1 := time - 2*distanceLeft
				option2 := (time - distanceLeft) / 2

				if option1 > option2 {
					reach = hen + option1
				} else {
					reach = hen + option2
				}
			}

			for grainIndex < totalGrains && grains[grainIndex] <= reach {
				grainIndex++
			}
		}

		return grainIndex == totalGrains
	}

	left := 0
	right := 1_000_000_000

	for left < right {
		mid := left + (right-left)/2

		if canFinish(mid) {
			right = mid
		} else {
			left = mid + 1
		}
	}

	return left
}
```

The Go implementation mirrors the Python solution closely. The main difference is explicit integer handling.

Go integer division automatically truncates toward zero, which matches the desired behavior for this problem. The binary search midpoint is written as:

```go
mid := left + (right-left)/2
```

to avoid potential overflow in general practice, although overflow is not actually possible under these constraints.

Slices are used directly after sorting with `sort.Ints`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O((n + m) log M) | Binary search over answer range, each feasibility check is linear |
| Space | O(1) excluding sorting | Only pointers and scalar variables are used |

The binary search performs roughly `log2(10^9)` iterations, which is about 31.

Each feasibility check scans through hens and grains at most once, giving `O(n + m)` per check.

Therefore the total complexity is:

$$O((n+m)\log M)$$

where `M = 10^9`.

## Edge Cases

One important edge case occurs when a hen must move left before moving right. This is the hardest part of the problem because naive implementations often assume movement in only one direction. For example, if a hen starts at `5` and grains are at `2` and `8`, the hen may need to visit the left grain first and then move right. The implementation handles this correctly using the two-strategy reach formula.

Another critical edge case is when all grains lie entirely on one side of every hen. In this scenario, the problem simplifies to straightforward linear movement. The implementation detects this naturally because the current grain will already be greater than or equal to the hen position, allowing direct rightward movement calculation.

Large coordinate values are also important. Since positions can reach `10^9`, any algorithm that attempts to simulate movement second by second would fail. The implementation never iterates over coordinates themselves. It only performs arithmetic on positions, which keeps the solution efficient even at maximum limits.

A final subtle edge case occurs when there are significantly more grains than hens. In such cases, individual hens may need to consume long consecutive segments of grains. The greedy scanning logic correctly allows each hen to absorb as many reachable grains as possible before passing responsibility to the next hen.
