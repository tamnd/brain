## Solution

## Problem Understanding

The problem is asking us to implement a **counter generator** that starts from a given integer `n` and increments by one each time it is called. Essentially, we need to return a **function** that maintains internal state between calls. The first call should return the initial value `n`, the second call returns `n + 1`, the third call returns `n + 2`, and so on.

The input `n` represents the starting point of the counter. The output is a sequence of integers generated by repeatedly calling the returned function. The problem constraints are quite relaxed: `n` can range from `-1000` to `1000`, and there can be up to `1000` calls. This means performance is not a major concern, and a straightforward solution that stores an internal counter is sufficient.

Important edge cases include starting from negative numbers, zero, or a positive number, as well as calling the counter zero times. The problem guarantees that the input is always a valid integer, so we do not need to handle invalid inputs.

## Approaches

The brute-force approach would be to store a **list of all values** generated so far and return the next one on each call. While this works, it is unnecessary because we only need to remember the **current value** and increment it. Maintaining the full list consumes extra memory without adding any benefit.

The optimal approach is to use a **closure** or a **function object** that maintains an internal integer variable representing the current value of the counter. Each call to the function returns this value and then increments it by one. This approach is efficient in both time and space because we only track a single integer instead of storing all generated values.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(1) per call | O(calls.length) | Stores all previous values unnecessarily |
| Optimal | O(1) per call | O(1) | Uses a closure to maintain current value |

## Algorithm Walkthrough

1. Define a function `createCounter` that accepts the starting integer `n`.
2. Inside `createCounter`, initialize a local variable `current` to `n`. This variable will track the current value of the counter.
3. Define a nested function `counter` which will be returned. This function will be called repeatedly to generate the next number.
4. In `counter`, first store the value of `current` in a temporary variable `result`.
5. Increment `current` by 1 to prepare for the next call.
6. Return `result` from the `counter` function.
7. Return the `counter` function from `createCounter`.

**Why it works:** The nested function forms a closure over the `current` variable. This means `current` retains its state between calls to `counter`. Every time `counter` is invoked, it returns the current value and then increments it, guaranteeing that each subsequent call produces the next integer in sequence.

## Python Solution

```python
from typing import Callable

def createCounter(n: int) -> Callable[[], int]:
    current = n

    def counter() -> int:
        nonlocal current
        result = current
        current += 1
        return result

    return counter
```

In this Python implementation, `current` is captured in the closure of the `counter` function. The `nonlocal` keyword allows the inner function to modify the variable `current` defined in the outer scope. Each call returns the current value before incrementing it, exactly matching the problem requirements.

## Go Solution

```go
package main

func CreateCounter(n int) func() int {
    current := n

    return func() int {
        result := current
        current++
        return result
    }
}
```

In Go, closures work similarly to Python. The anonymous function returned by `CreateCounter` captures the `current` variable and increments it with each call. Go does not require a keyword like `nonlocal` because variables captured in closures are automatically mutable within the scope of the closure.

## Edge Cases

1. **Negative start**: Starting from a negative integer ensures the counter increments correctly through zero. Implementation works because the variable `current` is simply incremented; no special handling is required.
2. **Zero calls**: If the counter is never called, no values are returned. This is naturally handled because the function is only invoked when called.
3. **Single call**: Starting from any number and calling the counter once should return the starting number. The closure guarantees the initial value is returned before any increment.

This approach handles all edge cases elegantly using simple closure semantics without requiring additional state management or conditional logic.
