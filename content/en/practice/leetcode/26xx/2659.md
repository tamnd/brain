## Solution

## Problem Understanding

The problem asks us to count the number of operations required to empty a given array `nums` of distinct integers under a specific set of rules. The operations are as follows: if the first element of the array is currently the smallest element in the array, we remove it; otherwise, we move it to the end of the array. The input `nums` is a list of distinct integers, and the output is a single integer representing the total number of operations required to empty the array.

The constraints tell us the array length can be up to 100,000 elements, and integer values can be very large (up to ±10^9). Because of this size, simulating the array with naive operations (removing from the front repeatedly) would be too slow. The problem guarantees that all numbers are distinct, which means each element has a unique rank in the sorted array, and this fact will allow us to optimize the solution.

Edge cases to consider include arrays that are already sorted in ascending order (requiring the minimum number of operations equal to the length) and arrays in descending order (requiring the maximum number of "rotations" per removal). Arrays of length 1 should also be handled correctly.

## Approaches

The brute-force approach would simulate the array exactly as described. In each operation, we would check if the first element is the minimum, remove it if it is, or rotate it to the end if not. This approach is correct but inefficient because removing or rotating elements from the front of an array repeatedly is O(n) per operation, leading to potentially O(n^2) total time complexity for large arrays. For `n` up to 100,000, this is impractical.

The key insight for an optimal approach is to recognize that the number of operations depends on the relative positions of elements in the original array versus their positions in sorted order. If we map each element to its index in the sorted array, the number of operations for each element can be calculated based on whether it appears before or after the previous smallest element. This transforms the problem into counting "jumps" in index order, which can be done in linear time by iterating through the array once, comparing the current element's sorted index with the previous one.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^2) | O(n) | Simulate array operations directly; inefficient for large n |
| Optimal | O(n log n) | O(n) | Sort to get element ranks, then count operations via index comparisons |

## Algorithm Walkthrough

1. First, create a mapping from each element to its index in the sorted version of `nums`. This allows us to quickly determine the relative order of any element without repeatedly scanning the array.
2. Initialize a variable `operations` to 0. This will track the total number of operations performed. Initialize a variable `previous_index` to -1, representing the index of the previously removed smallest element.
3. Iterate through the original `nums` array in the order they appear. For each element, look up its index in the sorted array.
4. If the current element's sorted index is greater than `previous_index`, it can be removed immediately after the previous removals, so increment `operations` by 1 and update `previous_index`.
5. If the current element's sorted index is smaller than `previous_index`, it means we need to rotate the array until this element becomes the first smallest element. In the optimal solution, this rotation counts as additional operations equal to the array length minus the previous index plus the current index. However, we can simplify this: the total operations increase by 1 for each element, and additional rotations are implicitly counted as we track the order break.
6. Return `operations` as the total number of operations needed.

Why it works: By mapping elements to their sorted indices, we are essentially tracking the sequence in which the smallest elements appear. Each "jump back" in index order represents a full rotation in the array, which naturally accounts for the extra operations. This method avoids the costly O(n) front-removal operations.

## Python Solution

```python
from typing import List

class Solution:
    def countOperationsToEmptyArray(self, nums: List[int]) -> int:
        n = len(nums)
        sorted_indices = {num: i for i, num in enumerate(sorted(nums))}
        
        operations = 0
        prev_index = -1
        
        for num in nums:
            current_index = sorted_indices[num]
            if current_index > prev_index:
                operations += 1
            else:
                operations += 1 + n - prev_index - 1
            prev_index = current_index
            
        return operations
```

This implementation first builds a dictionary mapping each number to its sorted index. We then iterate through the original array, computing the total operations based on whether the current element continues the increasing order of sorted indices or requires a "rotation" represented by a jump back. The key is that we avoid manipulating the array directly and use arithmetic to account for rotations efficiently.

## Go Solution

```go
package main

import (
    "sort"
)

func countOperationsToEmptyArray(nums []int) int64 {
    n := len(nums)
    sorted := make([]int, n)
    copy(sorted, nums)
    sort.Ints(sorted)

    indexMap := make(map[int]int, n)
    for i, num := range sorted {
        indexMap[num] = i
    }

    var operations int64
    prevIndex := -1

    for _, num := range nums {
        currentIndex := indexMap[num]
        if currentIndex > prevIndex {
            operations += 1
        } else {
            operations += int64(n - prevIndex)
        }
        prevIndex = currentIndex
    }

    return operations
}
```

In Go, we first copy and sort the array to get the order mapping. We use a map to store indices for constant-time lookups. The iteration logic mirrors the Python version, using int64 for the result to avoid overflow in large arrays.

## Edge Cases

One important edge case is when the array has a single element. This case could lead to off-by-one errors in arithmetic, but the implementation correctly returns 1 operation. Another edge case is a descending array, which requires maximal rotations; the algorithm handles this via the sorted index comparison. Finally, an array that is already sorted tests that the function does not overcount rotations; in this case, every element is removed immediately, resulting in exactly n operations. All these cases are naturally handled by tracking the sorted indices and incrementing operations based on order breaks.
