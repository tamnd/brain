## Solution

## Problem Understanding

The problem asks us to examine every integer in the inclusive range from `1` to `n` and determine whether each number is divisible by `3`, `5`, or `7`. If a number satisfies at least one of these divisibility conditions, we add it to a running total. At the end, we return the final sum.

More formally, for every integer `x` such that `1 <= x <= n`, we check whether any of the following are true:

- `x % 3 == 0`
- `x % 5 == 0`
- `x % 7 == 0`

If at least one condition is true, the number contributes to the answer.

The input consists of a single positive integer `n`. The output is a single integer representing the sum of all qualifying numbers.

The constraints are very small:

```
1 <= n <= 1000
```

This tells us several important things:

- We only need to process at most 1000 numbers.
- Even a straightforward iteration over the entire range is easily fast enough.
- There is no need for advanced mathematical optimizations unless we specifically want a more elegant solution.

One subtle detail is that numbers divisible by more than one divisor should only be counted once. For example, `15` is divisible by both `3` and `5`, but it should only be added a single time. Using a logical `or` condition naturally handles this requirement.

Important edge cases include:

- Very small inputs such as `n = 1`, where no numbers satisfy the condition.
- Inputs where `n` itself is divisible by `3`, `5`, or `7`.
- Numbers divisible by multiple values simultaneously, such as `15`, `21`, or `35`.
- Boundary values near the upper constraint, such as `n = 1000`.

## Approaches

### Brute Force Approach

The brute force approach checks every integer from `1` through `n`. For each number, we test whether it is divisible by `3`, `5`, or `7`. If it is, we add it to the answer.

This approach is correct because it explicitly evaluates every candidate number and includes exactly those that satisfy the required condition.

A naive implementation is already efficient enough because the maximum value of `n` is only `1000`. Even though we call this a brute force solution, it performs extremely well within the given constraints.

### Key Insight for the Optimal Approach

The main observation is that we do not need any complicated data structures or algorithms. Since the range is small, a single pass through the numbers is sufficient.

The optimal solution is essentially the same as the brute force approach because the input size is tiny. We simply maintain a running sum and add numbers that satisfy the divisibility condition.

This leads to a clean and efficient linear-time algorithm.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Checks every number individually |
| Optimal | O(n) | O(1) | Single pass with constant extra space |

## Algorithm Walkthrough

1. Initialize a variable called `total_sum` to `0`.

This variable stores the running total of all valid numbers.
2. Iterate through every integer from `1` to `n` inclusive.

We must inspect every number because any number in the range could potentially be divisible by `3`, `5`, or `7`.
3. For each number, check whether it is divisible by at least one of the three values.

We use the modulo operator `%`:

- `number % 3 == 0`
- `number % 5 == 0`
- `number % 7 == 0`

If any condition is true, the number qualifies.
4. Add qualifying numbers to `total_sum`.

Since we use a logical `or`, numbers divisible by multiple divisors are only added once.
5. After the loop finishes, return `total_sum`.

At this point, the sum contains exactly the numbers required by the problem.

### Why it works

The algorithm examines every integer in the required range exactly once. A number is added to the result if and only if it satisfies the divisibility condition. Since every valid number is included and every invalid number is excluded, the final sum is correct.

## Python Solution

```python
class Solution:
    def sumOfMultiples(self, n: int) -> int:
        total_sum = 0

        for number in range(1, n + 1):
            if (
                number % 3 == 0
                or number % 5 == 0
                or number % 7 == 0
            ):
                total_sum += number

        return total_sum
```

The implementation starts by initializing `total_sum` to zero. This variable accumulates the answer throughout the iteration.

The `for` loop iterates from `1` through `n` inclusive using `range(1, n + 1)`. For each number, we evaluate the divisibility conditions using the modulo operator.

If the current number is divisible by `3`, `5`, or `7`, we add it to `total_sum`.

Finally, after processing all numbers, we return the accumulated sum.

The code directly mirrors the algorithm described earlier, making it easy to understand and verify.

## Go Solution

```go
func sumOfMultiples(n int) int {
    totalSum := 0

    for number := 1; number <= n; number++ {
        if number%3 == 0 || number%5 == 0 || number%7 == 0 {
            totalSum += number
        }
    }

    return totalSum
}
```

The Go implementation follows the same logic as the Python version. The main syntactic differences are:

- Go uses `:=` for variable initialization.
- The loop uses Go's standard `for` syntax.
- Logical OR is written as `||`.

Integer overflow is not a concern because the constraints are very small. Even the sum of all numbers from `1` to `1000` comfortably fits inside a standard Go `int`.

## Edge Cases

One important edge case is when `n` is smaller than all divisors except possibly `3`. For example, when `n = 1` or `n = 2`, there are no numbers divisible by `3`, `5`, or `7`. A buggy implementation might accidentally include invalid numbers or mishandle empty sums. Our implementation correctly returns `0` because no numbers satisfy the condition.

Another important case involves numbers divisible by multiple divisors simultaneously. For example, `15` is divisible by both `3` and `5`. A flawed implementation using separate summation passes could accidentally add `15` twice. Our implementation avoids this issue because each number is evaluated exactly once with a combined logical `or` condition.

The upper boundary case, `n = 1000`, is also important. Although the constraints are small, it still verifies that the algorithm scales correctly across the full allowed range. Since the algorithm performs only a single linear scan with constant extra space, it handles the maximum input comfortably.
