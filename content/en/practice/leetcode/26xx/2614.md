# LeetCode 2614 - Prime In Diagonal

## Problem Understanding

This problem gives us a square matrix named `nums`, where the number of rows equals the number of columns. We need to examine the values that lie on either of the two diagonals of the matrix and return the largest prime number among them.

The primary diagonal consists of positions where the row index and column index are the same. For a matrix of size `n`, these are:

```
nums[0][0], nums[1][1], nums[2][2], ...
```

The secondary diagonal consists of positions where the column index is `n - i - 1` for row `i`. These are:

```
nums[0][n-1], nums[1][n-2], nums[2][n-3], ...
```

A number may appear on both diagonals. For example, in a matrix with odd dimensions, the center element belongs to both diagonals. We only care about whether a value is prime and whether it is the largest such value.

The input is a square matrix with dimensions between `1` and `300`. Each value can be as large as `4 * 10^6`. This constraint is important because checking whether a number is prime cannot be done with naive divisor checking up to `n`. Instead, we should only test divisors up to the square root of the number.

The matrix size is relatively small. Even at the maximum size of `300 x 300`, there are only `600` diagonal positions to inspect, because each diagonal contains `n` elements. This means the bottleneck is not matrix traversal, but primality testing.

Several edge cases are important:

- A matrix may contain no prime numbers on either diagonal, in which case we must return `0`.
- The matrix may be of size `1 x 1`.
- The center element of an odd-sized matrix belongs to both diagonals, so we may inspect the same value twice.
- Values such as `1` are not prime and must be handled correctly.
- Large numbers near `4 * 10^6` require an efficient primality check.

## Approaches

### Brute Force Approach

A brute force solution would first collect all diagonal elements, then determine whether each number is prime by checking divisibility from `2` all the way up to `n - 1`.

This approach is correct because a number is prime if no smaller positive integer other than `1` divides it evenly. However, this method is unnecessarily slow for large values. If a diagonal value is close to `4 * 10^6`, checking all possible divisors would take millions of operations for a single number.

Although the matrix itself is small, the value range makes naive primality testing inefficient.

### Optimal Approach

The key observation is that a number only needs to be tested for divisibility up to its square root.

If a number `n` has a divisor larger than `sqrt(n)`, then it must also have a corresponding divisor smaller than `sqrt(n)`. Therefore, once we have checked all integers up to the square root, we know whether the number is prime.

The optimal algorithm iterates through both diagonals, checks whether each value is prime using square root optimization, and keeps track of the largest prime encountered.

Since there are only `2n` diagonal positions and each primality test takes at most `O(sqrt(val))`, this solution is efficient enough for the given constraints.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × val) | O(1) | Checks divisibility up to n-1 for each diagonal value |
| Optimal | O(n × sqrt(val)) | O(1) | Uses square root optimization for primality testing |

## Algorithm Walkthrough

1. Initialize a variable named `largest_prime` with value `0`. This variable stores the best answer found so far.
2. Let `n` be the size of the matrix.
3. Iterate through every row index `i` from `0` to `n - 1`.
4. For each row:

- Extract the primary diagonal value:

```
nums[i][i]
```

- Extract the secondary diagonal value:

```
nums[i][n - i - 1]
```
5. For each extracted value, determine whether it is prime.

- If the value is less than or equal to `1`, it is not prime.
- Otherwise, test divisibility from `2` up to `sqrt(value)`.
- If any divisor divides evenly, the number is not prime.
- If no divisor is found, the number is prime.
6. Whenever a prime number is found, compare it with `largest_prime`.

- If it is larger, update `largest_prime`.
7. After processing all diagonal elements, return `largest_prime`.

### Why it works

The algorithm examines every value that belongs to either diagonal exactly as required by the problem statement. For each such value, it correctly determines primality using the mathematical property that non-prime numbers must have a factor less than or equal to their square root. Since every eligible diagonal element is checked and the maximum prime value is tracked throughout the process, the final result is guaranteed to be the largest prime appearing on either diagonal.

## Python Solution

```python
from typing import List
import math

class Solution:
    def diagonalPrime(self, nums: List[List[int]]) -> int:
        def is_prime(value: int) -> bool:
            if value <= 1:
                return False

            limit = int(math.sqrt(value))

            for divisor in range(2, limit + 1):
                if value % divisor == 0:
                    return False

            return True

        n = len(nums)
        largest_prime = 0

        for i in range(n):
            primary = nums[i][i]
            secondary = nums[i][n - i - 1]

            if is_prime(primary):
                largest_prime = max(largest_prime, primary)

            if is_prime(secondary):
                largest_prime = max(largest_prime, secondary)

        return largest_prime
```

The implementation begins by defining a helper function named `is_prime`. This function handles all primality checking logic. It first eliminates invalid cases such as `0` and `1`, then checks divisibility only up to the square root of the number.

The main function calculates the matrix size and initializes the answer variable.

The loop iterates through all rows exactly once. During each iteration, the algorithm extracts the corresponding value from both diagonals. Each value is checked independently for primality.

Whenever a prime number is found, the algorithm updates the maximum value seen so far. At the end of traversal, the function returns the largest discovered prime, or `0` if none were found.

## Go Solution

```go
package main

import "math"

func diagonalPrime(nums [][]int) int {
	isPrime := func(value int) bool {
		if value <= 1 {
			return false
		}

		limit := int(math.Sqrt(float64(value)))

		for divisor := 2; divisor <= limit; divisor++ {
			if value%divisor == 0 {
				return false
			}
		}

		return true
	}

	n := len(nums)
	largestPrime := 0

	for i := 0; i < n; i++ {
		primary := nums[i][i]
		secondary := nums[i][n-i-1]

		if isPrime(primary) && primary > largestPrime {
			largestPrime = primary
		}

		if isPrime(secondary) && secondary > largestPrime {
			largestPrime = secondary
		}
	}

	return largestPrime
}
```

The Go implementation follows the same algorithmic structure as the Python version. A local helper function handles primality testing.

Go requires explicit type conversion when using `math.Sqrt`, because the function operates on `float64`. After computing the square root, the result is converted back to `int`.

Slices are used naturally for matrix representation. No special handling for nil matrices is necessary because the problem guarantees valid input dimensions.

## Edge Cases

### Matrix With No Prime Numbers

A common mistake is assuming at least one prime always exists. In reality, both diagonals may contain only composite numbers or `1`. For example:

```
[
    [4, 6],
    [8, 9]
]
```

Since no diagonal value is prime, the algorithm correctly returns `0`. The implementation handles this naturally because `largest_prime` starts at `0` and is updated only when a prime is found.

### Single Element Matrix

The smallest possible input is a `1 x 1` matrix. The single value belongs to both diagonals simultaneously.

For example:

```
[[2]]
```

The algorithm checks the same value twice, but this causes no issues because taking the maximum repeatedly is harmless.

### Center Element Appearing On Both Diagonals

In odd-sized matrices, the center cell belongs to both diagonals. A naive implementation might accidentally mishandle duplicate processing.

For example:

```
[
    [1, 2, 3],
    [4, 5, 6],
    [7, 8, 9]
]
```

The value `5` is checked twice. The implementation remains correct because repeated primality checks do not affect correctness, and the maximum operation is idempotent.

### Large Prime Values

Diagonal elements may be as large as `4 * 10^6`. A naive primality test that checks every divisor up to `n - 1` would be inefficient.

The implementation avoids this issue by testing divisibility only up to the square root of the number, which keeps the runtime efficient even for large inputs.
