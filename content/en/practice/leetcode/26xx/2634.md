# LeetCode 2634 - Filter Elements from Array

## Problem Understanding

The problem asks us to implement our own version of array filtering without using JavaScript's built in `Array.filter()` method.

We are given two inputs:

- `arr`, an array of integers
- `fn`, a filtering function

For every element in the array, we call the function `fn(arr[i], i)`. The function may use:

- the value itself, `arr[i]`
- the index of the value, `i`

If the return value of the function is truthy, we keep the element in the final result. If the return value is falsy, we discard it.

In JavaScript, values such as `false`, `0`, `""`, `null`, `undefined`, and `NaN` are considered falsy. Everything else is truthy. This means the filtering decision is not restricted to boolean values. Any return value can be interpreted according to JavaScript truthiness rules.

The important detail is that we do not return the function results themselves. We return the original array elements whose corresponding function calls evaluate to truthy.

For example:

- If `fn(20)` returns `true`, we keep `20`
- If `fn(0)` returns `0`, we remove `0`
- If `fn(-2)` returns `-1`, we keep `-2` because `-1` is truthy

The constraints are small:

- `0 <= arr.length <= 1000`
- Values can range from `-10^9` to `10^9`

Since the array size is at most 1000, performance is not a major concern. A simple linear scan is more than sufficient.

Several edge cases are important:

- The array may be empty
- Every element may be filtered out
- Every element may be kept
- The callback may depend on indices instead of values
- The callback may return non boolean values
- The array may contain zeros or negative numbers, which can interact with truthiness rules in subtle ways

A correct implementation must evaluate the callback exactly once per element and preserve the original order of retained elements.

## Approaches

### Brute Force Approach

The most direct solution is to iterate through the array one element at a time. For each element:

1. Call the filtering function with the current value and index
2. Check whether the result is truthy
3. If it is truthy, append the original element to a result array

After processing all elements, return the result array.

This approach is already optimal because every element must be examined at least once. There is no way to determine whether an element should remain in the filtered array without calling the callback function for that element.

### Key Insight

The key observation is that filtering inherently requires checking each element independently. There is no shortcut or preprocessing technique that avoids visiting elements individually.

Because the output depends on evaluating the callback for every array position, a single linear traversal is both necessary and sufficient.

The optimal solution is therefore a straightforward one pass scan using an output array to store retained elements.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Iterates through every element and stores matching values |
| Optimal | O(n) | O(n) | Same linear scan, this is already the best possible solution |

## Algorithm Walkthrough

1. Create an empty result array that will store the filtered elements.
2. Traverse the input array from index `0` to `arr.length - 1`.
3. For each position:

- Retrieve the current value
- Call the filtering function using the current value and current index
4. Evaluate the return value of the callback using truthiness rules.

- If the result is truthy, append the original element to the result array
- Otherwise, skip it
5. Continue until all elements have been processed.
6. Return the result array.

### Why it works

The algorithm examines every array element exactly once and applies the filtering condition exactly as specified by the problem statement. An element is included if and only if the callback returns a truthy value for that element and index. Since every decision is made independently and elements are appended in traversal order, the final array contains exactly the correct filtered elements in the original order.

## Python Solution

```python
from typing import List, Callable, Any

class Solution:
    def filter(self, arr: List[int], fn: Callable[[int, int], Any]) -> List[int]:
        filtered_arr = []

        for index, value in enumerate(arr):
            if fn(value, index):
                filtered_arr.append(value)

        return filtered_arr
```

The implementation starts by creating an empty list called `filtered_arr`. This list will hold the elements that pass the filtering condition.

The `enumerate()` function is used so that both the index and value are available during iteration. This matches the problem requirement because the callback may depend on either the value or the index.

For every element, the callback function `fn(value, index)` is executed. Python naturally evaluates values in conditional statements according to truthiness rules, similar to JavaScript.

If the callback result is truthy, the original value is appended to the result list.

Finally, after all elements have been processed, the filtered list is returned.

## Go Solution

```go
package main

func filter(arr []int, fn func(int, int) bool) []int {
	filteredArr := make([]int, 0)

	for index, value := range arr {
		if fn(value, index) {
			filteredArr = append(filteredArr, value)
		}
	}

	return filteredArr
}
```

The Go solution follows the same overall algorithm as the Python version. The main difference is that Go requires explicit slice management.

The result slice is initialized using `make([]int, 0)`. During iteration, elements that satisfy the callback condition are appended to the slice.

Unlike JavaScript, Go does not have generalized truthy and falsy semantics for arbitrary values. Therefore, the callback explicitly returns a `bool`.

Go slices automatically grow as needed when using `append`, so no manual resizing logic is necessary.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every element is processed exactly once |
| Space | O(n) | In the worst case, all elements are stored in the output array |

The algorithm performs a single pass over the input array. Since each element requires one callback invocation and possibly one append operation, the total runtime grows linearly with the size of the array.

The extra space usage comes from the output array. In the worst case, every element satisfies the filter condition, so the result array contains all `n` elements.

## Edge Cases

One important edge case is an empty input array. A naive implementation might assume at least one element exists and accidentally access invalid indices. In this solution, the loop simply executes zero times, and the empty result array is returned immediately.

Another important case is when the callback returns non boolean values. Many implementations incorrectly assume the callback always returns `true` or `false`. However, the problem explicitly relies on truthiness semantics. The implementation correctly uses the callback result directly inside the conditional statement, allowing values like `0`, `""`, or `None` to behave as falsy while nonzero numbers and nonempty strings behave as truthy.

A third important edge case occurs when every element is filtered out. Some incorrect solutions accidentally return `None` or an uninitialized array in this scenario. Here, the result array is always initialized before iteration, so even if no elements are appended, the function correctly returns an empty array.

Another subtle case is preserving the original order of elements. Filtering should not rearrange the array. Since elements are appended to the result array during a left to right traversal, the original relative ordering is maintained automatically.
