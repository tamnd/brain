**Difficulty:** 🔴 Hard  
**Topics:** Breadth-First Search, Graph Theory

## Solution

## Problem Understanding

The problem gives us an undirected graph with `n` vertices labeled from `0` to `n - 1`. Each edge connects two different vertices, and there are no duplicate edges. The graph is not guaranteed to be connected, which means there may be several disconnected components.

Our task is to find the length of the shortest cycle in the graph.

A cycle is a path that starts and ends at the same node, without reusing edges. Since the graph is undirected, cycles can be traversed in either direction, but the cycle length is simply the number of edges involved.

For example, if we have edges:

```
0 - 1
1 - 2
2 - 0
```

then we have a cycle of length `3`.

If the graph contains multiple cycles, we must return the smallest cycle length among all of them. If the graph contains no cycles at all, we return `-1`.

The constraints are important:

- `n <= 1000`
- `edges.length <= 1000`

These limits are relatively moderate. An `O(n^3)` algorithm would likely be too slow in practice, but `O(n * (n + m))` is completely acceptable here, where `m` is the number of edges.

A few important observations and edge cases:

- The graph may be disconnected.
- Some components may contain cycles while others do not.
- A tree component contains no cycles.
- The shortest possible cycle in a simple undirected graph is `3`, because there are no self loops or duplicate edges.
- A naive DFS implementation can accidentally count parent edges as cycles unless handled carefully.
- Since the graph is undirected, we must distinguish between revisiting the parent and discovering a true cycle.

## Approaches

### Brute Force Approach

A brute force idea is to try removing each edge one at a time and then compute the shortest path between the two endpoints of that edge.

Suppose we remove edge `(u, v)`. If there is still a path from `u` to `v`, then that path together with edge `(u, v)` forms a cycle. The cycle length becomes:

```
shortest_path(u, v) + 1
```

We repeat this for every edge and take the minimum cycle length found.

This works because every cycle contains at least one edge, and removing that edge breaks the cycle into a path between its endpoints.

However, this approach repeatedly performs graph traversals for each edge. With up to `1000` edges, repeatedly running BFS becomes relatively expensive.

### Optimal Approach

A more efficient observation is that we can run BFS from every node.

In an unweighted graph, BFS naturally discovers shortest paths. While performing BFS from a source node, if we encounter a previously visited node that is not the parent of the current node, then we have discovered a cycle.

More importantly, the BFS distance information lets us compute the cycle length directly:

```
cycle_length = dist[current] + dist[neighbor] + 1
```

This works because:

- `dist[current]` is the shortest distance from the source to `current`
- `dist[neighbor]` is the shortest distance from the source to `neighbor`
- the edge `(current, neighbor)` closes the cycle

Running BFS from every node guarantees that we eventually discover the shortest cycle in the graph.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m × (n + m)) | O(n + m) | Remove each edge and run BFS between endpoints |
| Optimal | O(n × (n + m)) | O(n + m) | Run BFS from every node and detect cycles during traversal |

Since `m <= 1000`, the BFS-from-every-node solution is efficient enough.

## Algorithm Walkthrough

### Step 1: Build the adjacency list

We first convert the edge list into an adjacency list.

An adjacency list is ideal because:

- the graph is sparse
- we need efficient neighbor traversal
- BFS naturally works with adjacency lists

For every edge `(u, v)`:

- add `v` to `u`'s neighbor list
- add `u` to `v`'s neighbor list

### Step 2: Initialize the answer

We maintain a variable:

```
shortest_cycle = infinity
```

This stores the minimum cycle length discovered so far.

If it never changes, we return `-1`.

### Step 3: Run BFS from every node

For each node `start`:

- initialize a distance array with `-1`
- initialize a parent array with `-1`
- start BFS from `start`

The distance array tracks the shortest distance from the source.

The parent array helps us avoid falsely treating the immediate back edge to the parent as a cycle.

### Step 4: Process BFS levels

During BFS:

1. Pop a node from the queue.
2. Traverse all neighbors.

For each neighbor:

- If the neighbor has not been visited:

- assign its distance
- record its parent
- push it into the queue
- Otherwise:

- if the neighbor is not the parent of the current node, then we found a cycle

The cycle length becomes:

```
dist[node] + dist[neighbor] + 1
```

We update the global minimum.

### Step 5: Continue until all BFS traversals finish

Because BFS computes shortest distances in unweighted graphs, every detected cycle length is valid.

After trying all start nodes, the minimum discovered cycle is the answer.

### Step 6: Return the result

If no cycle was found:

```
return -1
```

Otherwise return the shortest cycle length.

### Why it works

BFS guarantees shortest path distances from the source node. When we encounter an already visited neighbor that is not the parent, we have identified two distinct paths from the source that merge through the current edge, forming a cycle.

The formula:

```
dist[u] + dist[v] + 1
```

correctly measures the cycle length because it combines:

- the shortest path from source to `u`
- the shortest path from source to `v`
- the connecting edge `(u, v)`

Running BFS from every node ensures that every possible cycle configuration is eventually examined, so the global minimum cycle length is correctly found.

## Python Solution

```python
from collections import deque
from typing import List

class Solution:
    def findShortestCycle(self, n: int, edges: List[List[int]]) -> int:
        graph = [[] for _ in range(n)]

        for u, v in edges:
            graph[u].append(v)
            graph[v].append(u)

        shortest_cycle = float('inf')

        for start in range(n):
            dist = [-1] * n
            parent = [-1] * n

            queue = deque([start])
            dist[start] = 0

            while queue:
                node = queue.popleft()

                for neighbor in graph[node]:
                    if dist[neighbor] == -1:
                        dist[neighbor] = dist[node] + 1
                        parent[neighbor] = node
                        queue.append(neighbor)

                    elif parent[node] != neighbor:
                        cycle_length = dist[node] + dist[neighbor] + 1
                        shortest_cycle = min(shortest_cycle, cycle_length)

        return shortest_cycle if shortest_cycle != float('inf') else -1
```

The implementation begins by constructing the adjacency list representation of the graph. Since the graph is undirected, each edge is added in both directions.

The outer loop runs BFS from every possible starting node. For each BFS traversal:

- `dist` stores the shortest distance from the start node
- `parent` tracks the BFS tree structure

When we encounter an unvisited neighbor, we assign its distance and parent, then enqueue it.

When we encounter a previously visited neighbor that is not the parent, we have detected a cycle. The cycle length formula uses the already computed BFS distances.

The global minimum is updated whenever a shorter cycle is found.

Finally, if no cycle was discovered, we return `-1`.

## Go Solution

```go
package main

import "container/list"

func findShortestCycle(n int, edges [][]int) int {
	graph := make([][]int, n)

	for _, edge := range edges {
		u, v := edge[0], edge[1]
		graph[u] = append(graph[u], v)
		graph[v] = append(graph[v], u)
	}

	const INF = int(1e9)
	shortestCycle := INF

	for start := 0; start < n; start++ {
		dist := make([]int, n)
		parent := make([]int, n)

		for i := 0; i < n; i++ {
			dist[i] = -1
			parent[i] = -1
		}

		queue := list.New()
		queue.PushBack(start)
		dist[start] = 0

		for queue.Len() > 0 {
			front := queue.Front()
			queue.Remove(front)

			node := front.Value.(int)

			for _, neighbor := range graph[node] {
				if dist[neighbor] == -1 {
					dist[neighbor] = dist[node] + 1
					parent[neighbor] = node
					queue.PushBack(neighbor)
				} else if parent[node] != neighbor {
					cycleLength := dist[node] + dist[neighbor] + 1

					if cycleLength < shortestCycle {
						shortestCycle = cycleLength
					}
				}
			}
		}
	}

	if shortestCycle == INF {
		return -1
	}

	return shortestCycle
}
```

The Go implementation closely mirrors the Python solution. The primary difference is queue handling.

Go does not have a built in deque structure like Python's `collections.deque`, so we use `container/list` to implement BFS efficiently.

The distance and parent arrays are initialized with `-1` values to indicate unvisited nodes.

Since the graph size is relatively small, standard integer arithmetic is completely safe and overflow is not a concern.

Neighbors: `0`, `2`

- `0` is parent, ignore
- `2` already visited and not parent

Cycle detected:

```
dist[1] + dist[2] + 1
= 1 + 1 + 1
= 3
```

Shortest cycle becomes `3`.

No smaller cycle can exist in a simple graph, so this is optimal.

Each BFS traversal visits every node and edge at most once, giving `O(n + m)` per traversal.

Given the constraints `n <= 1000` and `m <= 1000`, this is efficient enough.

## Edge Cases

### Graph With No Cycles

A graph may be completely acyclic, such as a tree or forest.

This is important because some naive implementations accidentally detect false cycles in undirected graphs due to revisiting parent nodes.

The implementation avoids this by checking:

```
elif parent[node] != neighbor:
```

Only non-parent revisits count as cycles.

If no valid cycle is ever found, the answer remains infinity and the algorithm returns `-1`.

### Multiple Connected Components

The graph is not guaranteed to be connected.

A cycle may exist in only one component while others are trees.

The implementation handles this naturally because BFS is started from every node. Even isolated or disconnected components are fully explored.

### Overlapping Cycles

A graph may contain many cycles sharing vertices or edges.

For example:

```
0 - 1 - 2
|       |
4 - 3 -/
```

A naive DFS might detect a longer cycle first and stop early.

This implementation never stops after finding a cycle. Instead, it continuously updates the minimum cycle length across all BFS traversals, guaranteeing the shortest cycle is returned.
