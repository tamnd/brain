## Solution

## Problem Understanding

This problem asks us to implement a generator function that produces the Fibonacci sequence one value at a time. Instead of returning an array containing all Fibonacci numbers, we must return a generator object. Each time the caller invokes `next()` on that generator, it should yield the next Fibonacci number in the sequence.

The Fibonacci sequence is defined recursively. Each number is the sum of the previous two numbers:

$$F(n) = F(n - 1) + F(n - 2)$$

The sequence begins with:

$$0, 1, 1, 2, 3, 5, 8, 13, \dots$$

The input shown in the examples is `callCount`, which represents how many times the caller invokes `next()` on the generator. The generator itself does not receive this value as an argument. Instead, it must be capable of producing Fibonacci numbers indefinitely.

For example, if the generator is called five times, the outputs should be:

```
0, 1, 1, 2, 3
```

The constraints are very small, with at most 50 calls to the generator. Because of this, efficiency is not a major concern. However, the problem is designed to test understanding of JavaScript generators and state management across successive yields.

An important detail is that the generator must preserve its internal state between calls. Each invocation of `next()` should continue from where the previous one stopped.

There are several edge cases worth considering:

- If `callCount = 0`, the generator is never used, so no output is produced.
- The first two Fibonacci numbers are special cases, `0` and `1`.
- The generator should continue indefinitely without running out of values.
- A naive recursive Fibonacci computation could repeatedly recompute values unnecessarily.

## Approaches

### Brute Force Approach

A straightforward solution is to compute the Fibonacci number from scratch every time the generator is asked for the next value. For the $n$-th call, we could recursively compute the $n$-th Fibonacci number using the mathematical definition.

For example:

$$F(n) = F(n - 1) + F(n - 2)$$

with base cases:

$$F(0) = 0,\quad F(1) = 1$$

This approach is correct because it directly follows the Fibonacci recurrence relation. However, it is inefficient because recursive Fibonacci computation performs massive repeated work. Computing `F(40)` recursively already involves millions of repeated calculations.

Even though the constraints are small, this approach is conceptually inefficient and unnecessary.

### Optimal Approach

The key observation is that each Fibonacci number depends only on the previous two numbers. We do not need to recompute earlier values repeatedly.

Instead, we can maintain two variables:

- `a`, representing the current Fibonacci number
- `b`, representing the next Fibonacci number

At each step:

1. Yield `a`
2. Update the pair:

- new `a = b`
- new `b = a + b`

This allows the generator to produce values one at a time in constant time per operation while storing only two integers.

Because generators naturally preserve execution state between yields, this approach fits perfectly.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) per generated value | O(n) | Recursive recomputation of Fibonacci numbers |
| Optimal | O(1) per generated value | O(1) | Maintains only the previous two numbers |

## Algorithm Walkthrough

1. Initialize two variables:

- `current = 0`
- `next_value = 1`

These represent the first two Fibonacci numbers.
2. Start an infinite loop.

The generator should always be capable of producing another Fibonacci number.
3. Yield `current`.

This returns the current Fibonacci value to the caller while pausing execution.
4. Update the two variables simultaneously:

- The next current value becomes the old next value.
- The next value becomes the sum of the previous two values.
5. Resume execution when the caller invokes `next()` again.
6. Repeat the process indefinitely.

### Why it works

The algorithm maintains an important invariant:

- `current` always stores the next Fibonacci number to output.
- `next_value` always stores the Fibonacci number immediately after it.

After each update:

$$(current, next\_value) \rightarrow (next\_value, current + next\_value)$$

This transformation exactly matches the Fibonacci recurrence relation, so every yielded value is correct.

## Python Solution

```python
from typing import Generator

class Solution:
    def fibGenerator(self) -> Generator[int, None, None]:
        current = 0
        next_value = 1

        while True:
            yield current
            current, next_value = next_value, current + next_value
```

The implementation begins by initializing the first two Fibonacci numbers. The variable `current` stores the value that should be yielded next, while `next_value` stores the following Fibonacci number.

The `while True` loop ensures the generator can continue indefinitely. Each iteration yields the current Fibonacci number and then updates both variables simultaneously using tuple assignment.

Python generators automatically preserve local state between calls to `next()`. After yielding a value, execution pauses. When resumed, the generator continues immediately after the `yield` statement.

This makes generators ideal for sequence generation problems like Fibonacci.

## Go Solution

```go
package main

func fibGenerator() func() int {
	current := 0
	nextValue := 1

	return func() int {
		value := current
		current, nextValue = nextValue, current+nextValue
		return value
	}
}
```

Go does not have native generator syntax like Python or JavaScript. Instead, we simulate generator behavior using closures.

The returned anonymous function captures the variables `current` and `nextValue`. Because closures preserve state between function calls, each invocation returns the next Fibonacci number in the sequence.

Unlike Python generators, there is no `yield` keyword in Go. The closure pattern is the standard way to emulate generator-like behavior.

## Edge Cases

One important edge case is when the generator is never called. In this situation, no Fibonacci numbers are produced at all. Some implementations incorrectly attempt to precompute values immediately when the generator is created. This implementation avoids that issue because execution does not begin until the first `next()` call.

Another important case is the handling of the first two Fibonacci numbers. Many incorrect implementations start directly with the recurrence relation and accidentally skip `0` or duplicate values incorrectly. This solution explicitly initializes the sequence with `0` and `1`, guaranteeing the correct starting state.

A third edge case involves long sequences. Recursive solutions can become extremely slow because they repeatedly recompute earlier Fibonacci values. This implementation avoids that entirely by storing only the previous two numbers and updating them iteratively. Even after many generator calls, performance remains constant per operation.

A final subtle edge case is maintaining generator state independently. If multiple generators are created, they should not interfere with each other. Because all state is stored locally inside the generator instance, each generator progresses independently through the sequence.
