# LeetCode 2642 - Design Graph With Shortest Path Calculator

## Problem Understanding

This problem asks us to design a mutable directed weighted graph that supports two operations efficiently:

1. Adding a new directed edge with a weight.
2. Querying the shortest path cost between two nodes.

The graph contains `n` nodes labeled from `0` to `n - 1`. Every edge has a direction and a positive weight. Since the graph is directed, an edge from node `u` to node `v` does not imply the existence of an edge from `v` to `u`.

The class must support three methods:

- The constructor initializes the graph using the provided edges.
- `addEdge(edge)` inserts a new edge into the graph.
- `shortestPath(node1, node2)` returns the minimum total cost required to travel from `node1` to `node2`.

If no path exists, the method must return `-1`.

An important detail is that the graph changes over time because new edges can be added after initialization. Therefore, the shortest paths may change dynamically.

The constraints are relatively small:

- `n <= 100`
- At most `100` edge additions
- At most `100` shortest path queries

These limits strongly influence the best design choice. Since the graph is small, we can afford algorithms that would normally be too expensive for large graphs.

Another key observation is that all edge weights are positive. This means Dijkstra’s algorithm is valid because it relies on the absence of negative-weight edges.

Several edge cases are important:

- The graph may initially contain no edges.
- A queried destination may be unreachable.
- Multiple calls to `addEdge` may create shorter alternative routes.
- Queries may ask for the shortest path from a node to itself.
- The graph is directed, so reachability is asymmetric.

The problem guarantees there are no duplicate edges and no self-loops when adding edges.

## Approaches

### Brute Force Approach

A brute-force solution would try to enumerate all possible paths between two nodes and compute their costs. We could use Depth First Search to explore every reachable route from `node1` to `node2`.

This approach is correct because it eventually checks every valid path and keeps the minimum cost encountered.

However, this becomes extremely inefficient because the number of possible paths in a directed graph can grow exponentially. Even with only 100 nodes, the number of routes can explode combinatorially.

Therefore, while brute force guarantees correctness, it is computationally impractical.

### Optimal Approach

The optimal solution uses Dijkstra’s algorithm with an adjacency list representation of the graph.

The key insight is that all edge weights are positive. Dijkstra’s algorithm efficiently computes the shortest distance from one source node to all reachable nodes by repeatedly expanding the currently cheapest known node.

Since the graph is mutable, we maintain the adjacency list and simply append new edges whenever `addEdge` is called.

For every `shortestPath` query:

- Run Dijkstra starting from `node1`
- Stop early once `node2` is reached
- Return the computed shortest distance

Because the graph size is small, running Dijkstra per query is entirely acceptable.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | O(V) | Explores every possible path |
| Optimal | O((V + E) log V) per query | O(V + E) | Uses adjacency list and Dijkstra’s algorithm |

## Algorithm Walkthrough

### Step 1: Store the Graph as an Adjacency List

We maintain an adjacency list where:

- `graph[u]` contains all outgoing edges from node `u`
- Each edge is stored as `(neighbor, weight)`

This structure allows efficient traversal of outgoing edges during shortest path computation.

### Step 2: Initialize the Graph

During construction:

1. Create an empty adjacency list for all nodes.
2. Iterate through the initial edge list.
3. Add every edge into the adjacency structure.

This gives us a complete directed graph representation.

### Step 3: Add New Edges Dynamically

When `addEdge(edge)` is called:

1. Extract `from`, `to`, and `cost`
2. Append the edge to the adjacency list

Since there are no duplicate edges, we can safely append directly.

### Step 4: Run Dijkstra’s Algorithm for Queries

To answer `shortestPath(node1, node2)`:

1. Create a distance array initialized to infinity.
2. Set the source distance to `0`.
3. Use a min-heap priority queue storing `(distance, node)`.
4. Repeatedly extract the node with the smallest known distance.
5. Relax all outgoing edges:

- If a shorter path is found, update the distance.
- Push the updated value into the heap.
6. If we reach `node2`, return the shortest distance immediately.
7. If the heap becomes empty without reaching the target, return `-1`.

### Step 5: Handle Outdated Heap Entries

A node may appear multiple times in the heap with different distances.

Before processing a node:

- If the popped distance is larger than the recorded shortest distance, skip it.

This ensures correctness and avoids unnecessary work.

### Why it works

Dijkstra’s algorithm maintains the invariant that once a node is removed from the priority queue with its minimum distance, that distance is final.

Because all edge weights are positive, any future path reaching that node would necessarily be equal or more expensive.

Therefore, when the destination node is popped from the heap, the algorithm has already found the globally shortest path.

## Python Solution

```python
from typing import List
import heapq

class Graph:

    def __init__(self, n: int, edges: List[List[int]]):
        self.graph = [[] for _ in range(n)]

        for from_node, to_node, cost in edges:
            self.graph[from_node].append((to_node, cost))

    def addEdge(self, edge: List[int]) -> None:
        from_node, to_node, cost = edge
        self.graph[from_node].append((to_node, cost))

    def shortestPath(self, node1: int, node2: int) -> int:
        n = len(self.graph)

        distances = [float("inf")] * n
        distances[node1] = 0

        min_heap = [(0, node1)]

        while min_heap:
            current_distance, current_node = heapq.heappop(min_heap)

            if current_node == node2:
                return current_distance

            if current_distance > distances[current_node]:
                continue

            for neighbor, weight in self.graph[current_node]:
                new_distance = current_distance + weight

                if new_distance < distances[neighbor]:
                    distances[neighbor] = new_distance
                    heapq.heappush(
                        min_heap,
                        (new_distance, neighbor)
                    )

        return -1

# Your Graph object will be instantiated and called as such:
# obj = Graph(n, edges)
# obj.addEdge(edge)
# param_2 = obj.shortestPath(node1, node2)
```

The constructor initializes the adjacency list using the given edges. Each node stores all outgoing edges together with their costs.

The `addEdge` method simply appends a new edge into the adjacency list. Since the graph is directed, the edge is only added in one direction.

The `shortestPath` method implements Dijkstra’s algorithm. The `distances` array stores the currently known shortest distance to each node. The priority queue always expands the node with the minimum tentative distance.

When a better route to a neighbor is discovered, the distance is updated and the new state is pushed into the heap.

The early return when `current_node == node2` is valid because Dijkstra guarantees that the first time we pop the destination node, we have already found its optimal distance.

If the destination is never reached, the method returns `-1`.

## Go Solution

```go
package main

import (
	"container/heap"
	"math"
)

type Edge struct {
	to     int
	weight int
}

type State struct {
	node     int
	distance int
}

type PriorityQueue []State

func (pq PriorityQueue) Len() int {
	return len(pq)
}

func (pq PriorityQueue) Less(i, j int) bool {
	return pq[i].distance < pq[j].distance
}

func (pq PriorityQueue) Swap(i, j int) {
	pq[i], pq[j] = pq[j], pq[i]
}

func (pq *PriorityQueue) Push(x interface{}) {
	*pq = append(*pq, x.(State))
}

func (pq *PriorityQueue) Pop() interface{} {
	old := *pq
	n := len(old)

	item := old[n-1]
	*pq = old[:n-1]

	return item
}

type Graph struct {
	graph [][]Edge
}

func Constructor(n int, edges [][]int) Graph {
	graph := make([][]Edge, n)

	for _, edge := range edges {
		fromNode := edge[0]
		toNode := edge[1]
		cost := edge[2]

		graph[fromNode] = append(
			graph[fromNode],
			Edge{to: toNode, weight: cost},
		)
	}

	return Graph{graph: graph}
}

func (this *Graph) AddEdge(edge []int) {
	fromNode := edge[0]
	toNode := edge[1]
	cost := edge[2]

	this.graph[fromNode] = append(
		this.graph[fromNode],
		Edge{to: toNode, weight: cost},
	)
}

func (this *Graph) ShortestPath(node1 int, node2 int) int {
	n := len(this.graph)

	distances := make([]int, n)

	for i := 0; i < n; i++ {
		distances[i] = math.MaxInt
	}

	distances[node1] = 0

	pq := &PriorityQueue{}
	heap.Init(pq)

	heap.Push(pq, State{
		node:     node1,
		distance: 0,
	})

	for pq.Len() > 0 {
		current := heap.Pop(pq).(State)

		currentNode := current.node
		currentDistance := current.distance

		if currentNode == node2 {
			return currentDistance
		}

		if currentDistance > distances[currentNode] {
			continue
		}

		for _, edge := range this.graph[currentNode] {
			newDistance := currentDistance + edge.weight

			if newDistance < distances[edge.to] {
				distances[edge.to] = newDistance

				heap.Push(pq, State{
					node:     edge.to,
					distance: newDistance,
				})
			}
		}
	}

	return -1
}
```

The Go implementation follows the same logic as the Python solution, but Go requires explicit priority queue implementation using the `container/heap` package.

The adjacency list is stored as a slice of slices. Each edge is represented using an `Edge` struct.

Go does not provide a built-in min-heap, so we implement the heap interface manually using `Len`, `Less`, `Swap`, `Push`, and `Pop`.

The `math.MaxInt` value is used as infinity for initial distances.

Since edge weights and path lengths remain within safe integer bounds under the constraints, integer overflow is not an issue here.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O((V + E) log V) per query | Dijkstra processes vertices and edges using a heap |
| Space | O(V + E) | Adjacency list, heap, and distance array |

The adjacency list stores every edge exactly once, so storage is proportional to the graph size.

For each shortest path query, Dijkstra may process every node and edge. Heap operations contribute the logarithmic factor.

Given that `n <= 100`, this solution easily fits within the limits.

## Edge Cases

One important edge case occurs when the source and destination are the same node. The shortest path from a node to itself should be `0` because no travel is required. The implementation handles this naturally because the source distance is initialized to zero, and Dijkstra immediately returns zero when the source node is popped from the heap.

Another critical edge case is when no path exists between two nodes. A common bug is accidentally returning infinity or an uninitialized value. In this implementation, if the priority queue becomes empty before reaching the destination, the method explicitly returns `-1`.

A third important case involves dynamically added edges creating new shorter paths. Some implementations incorrectly cache previous shortest path results and fail to update them after graph mutations. This solution avoids the problem entirely because Dijkstra runs fresh for every query using the latest graph structure.

A fourth subtle edge case is handling outdated heap entries in Dijkstra’s algorithm. Since nodes may be pushed multiple times with different distances, processing stale entries can cause unnecessary work or incorrect logic in flawed implementations. The condition:

```
if current_distance > distances[current_node]:
    continue
```

ensures only the best known state for each node is processed.
