## Solution

## Problem Understanding

The problem is asking us to determine whether two JSON-parsed values, `o1` and `o2`, are **deeply equal**. This means we cannot simply compare the references or use shallow equality checks. Instead, we need to recursively inspect the structure and content of the values.

If the values are **primitive types** like numbers, strings, booleans, or `null`, they are deeply equal only if they are strictly equal. If the values are **arrays**, they are deeply equal if they have the same length and corresponding elements are deeply equal in order. If the values are **objects**, they are deeply equal if they have the same keys and the values for each key are also deeply equal.

The input is guaranteed to be valid JSON, so we do not need to handle malformed objects, functions, or other non-JSON types. The constraints specify that the string representation of the objects can be up to `10^5` characters long, and nesting can go up to 1000 levels, which implies we should be careful with recursion depth but that typical recursion in Python or Go will suffice.

Edge cases to consider include empty objects `{}`, empty arrays `[]`, `null` values, arrays with mixed types, and objects with keys in different orders.

## Approaches

The simplest approach is a brute-force method using type checks and recursion. We first compare primitive values directly. For arrays, we iterate element by element and recursively check equality. For objects, we compare sets of keys and recursively compare corresponding values.

A key optimization is that we do not need to sort object keys; we can compare them as sets and directly access values. Recursion depth is limited by Python or Go stack, but the problem guarantees a maximum nesting of 1000, which is acceptable.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(N) | O(D) | Recursively traverse all elements; N is total number of primitive elements, D is max nesting depth |
| Optimal | O(N) | O(D) | Same as brute force, but avoids unnecessary copying or key sorting; uses direct set comparisons for object keys |

## Algorithm Walkthrough

1. Check if `o1` and `o2` are strictly equal as primitives. If yes, return `True`.
2. Determine the type of `o1` and `o2`. If the types differ, return `False`.
3. If both are arrays:

1. Check if their lengths are equal; if not, return `False`.
2. Iterate over each pair of elements and recursively call the deep equality check. If any pair fails, return `False`.
4. If both are objects:

1. Compare the set of keys; if they differ, return `False`.
2. For each key, recursively check if the values are deeply equal. If any check fails, return `False`.
5. After all checks, if none failed, return `True`.

**Why it works:** The algorithm directly mirrors the definition of deep equality. By recursively checking each element in arrays and each key-value pair in objects, it ensures that equality is verified at every level of nesting. The recursion terminates either at primitive values or at empty structures, ensuring correctness.

## Python Solution

```python
from typing import Any

class Solution:
    def isDeepEqual(self, o1: Any, o2: Any) -> bool:
        if o1 is o2:
            return True
        if type(o1) != type(o2):
            return False
        if isinstance(o1, dict):
            if o1.keys() != o2.keys():
                return False
            for key in o1:
                if not self.isDeepEqual(o1[key], o2[key]):
                    return False
            return True
        if isinstance(o1, list):
            if len(o1) != len(o2):
                return False
            for v1, v2 in zip(o1, o2):
                if not self.isDeepEqual(v1, v2):
                    return False
            return True
        return o1 == o2
```

This implementation starts with identity and type checks for efficiency. Dictionaries are compared by key sets first to avoid unnecessary recursion. Arrays are compared element-wise recursively, and primitive types are compared using equality. This ensures a correct, recursive deep equality check.

## Go Solution

```go
package main

import "reflect"

func isDeepEqual(o1, o2 any) bool {
    if reflect.TypeOf(o1) != reflect.TypeOf(o2) {
        return false
    }
    switch v1 := o1.(type) {
    case map[string]any:
        v2 := o2.(map[string]any)
        if len(v1) != len(v2) {
            return false
        }
        for key := range v1 {
            val2, ok := v2[key]
            if !ok || !isDeepEqual(v1[key], val2) {
                return false
            }
        }
        return true
    case []any:
        v2 := o2.([]any)
        if len(v1) != len(v2) {
            return false
        }
        for i := range v1 {
            if !isDeepEqual(v1[i], v2[i]) {
                return false
            }
        }
        return true
    default:
        return reflect.DeepEqual(o1, o2)
    }
}
```

In Go, `reflect.DeepEqual` could be used directly for primitives, but we manually handle arrays and maps to maintain recursive consistency. `nil` vs empty checks are automatically handled by Go's type assertions.

- Compare keys: both have `{"x", "y"}`
- Compare values recursively:

- key `x`: 1 == 1 → True
- key `y`: 2 == 2 → True
- All checks pass → return True

**Example 2:**

Input: `[1,2,3]`, `[1,2,"3"]`

- Check type: both arrays
- Compare lengths: 3 == 3 → True
- Compare elements recursively:

- 1 == 1 → True
- 2 == 2 → True
- 3 == "3" → False
- Return False

## Edge Cases

**Empty structures:** Empty objects `{}` and empty arrays `[]` can cause subtle bugs if the algorithm assumes at least one key or element. The implementation explicitly handles these by comparing lengths or key sets, returning True when both are empty.

**Nested structures:** Deeply nested arrays or objects could exceed recursion limits or fail to compare inner elements correctly. The recursive algorithm works because it checks type and structure at each level, ensuring correctness even for maximum nesting of 1000.

**Type mismatches:** Arrays vs objects, numbers vs strings, booleans vs numbers are all potential pitfalls. The initial type check prevents comparing incompatible structures, immediately returning False if types differ, which safeguards against incorrect equality results.
