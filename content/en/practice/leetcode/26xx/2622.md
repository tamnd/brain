# LeetCode 2622 - Cache With Time Limit

## Problem Understanding

This problem asks us to design a cache system where each key-value pair automatically expires after a certain amount of time. Unlike a normal hash map where values remain until explicitly removed, this cache must make keys inaccessible once their duration has elapsed.

We need to implement a class called `TimeLimitedCache` with three operations:

- `set(key, value, duration)`
- `get(key)`
- `count()`

The `set` operation inserts or updates a key with a value and an expiration duration in milliseconds. The key becomes invalid after that duration passes. The method returns `true` if the key already existed and had not yet expired, otherwise it returns `false`.

The `get` operation retrieves the value associated with a key, but only if the key has not expired. If the key is expired or does not exist, the method returns `-1`.

The `count` operation returns how many keys are currently active and un-expired.

The input examples simulate the passing of time using `timeDelays`. Each operation occurs at a specific timestamp relative to the start of execution. Internally, we must rely on actual timers or expiration timestamps to determine whether entries are still valid.

The constraints are very small:

- At most 100 operations
- Duration is at most 1000 milliseconds
- Time delays are at most 1450 milliseconds

These small limits mean performance is not especially demanding, but we still want a clean and efficient implementation.

Several edge cases are important:

- A key may be overwritten before it expires
- A duration can be zero, meaning the key expires immediately
- Calling `get` on an expired key must return `-1`
- `count()` should include only currently active keys
- Multiple keys may expire independently
- Updating a key should reset both its value and expiration time

The most important implementation challenge is ensuring that old timers do not accidentally remove newer values after a key has been overwritten.

## Approaches

### Brute Force Approach

A brute force implementation could store every key together with its expiration timestamp. Whenever `get` or `count` is called, we would scan through the entire cache and manually remove expired entries.

For example:

- `set` stores `(value, expiration_time)`
- `get` checks whether the current time exceeds the expiration time
- `count` iterates through every key and counts only valid ones

This approach is correct because every operation explicitly checks expiration validity before returning results.

However, `count()` becomes inefficient because it requires scanning every key each time. With a larger input size, repeated full scans would become expensive.

### Optimal Approach

The better solution uses:

- A hash map for O(1) key access
- A timer for automatic expiration

When we insert a key, we schedule a timer that removes the key after the specified duration. If the key is updated before expiration, we cancel the old timer and create a new one.

This ensures:

- `set` works in constant time
- `get` works in constant time
- `count` becomes constant time because expired keys are removed automatically

The key insight is that expiration should happen proactively instead of repeatedly checking all entries.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) for `count` | O(n) | Requires scanning cache repeatedly |
| Optimal | O(1) average per operation | O(n) | Uses timers and hash map for automatic expiration |

## Algorithm Walkthrough

### Optimal Algorithm

1. Create a hash map called `cache`.

Each key in the map stores:

- The associated value
- A timer reference used for expiration
2. When `set(key, value, duration)` is called, first check whether the key already exists.

If it exists, the key is currently active because expired keys are automatically removed by timers.
3. If the key already exists, cancel its old timer.

This step is critical because the previous expiration callback should no longer remove the updated value.
4. Create a new timer using `setTimeout`.

The timer will execute after `duration` milliseconds and remove the key from the hash map.
5. Store the new value and timer inside the hash map.
6. Return `true` if the key previously existed, otherwise return `false`.
7. When `get(key)` is called, check whether the key exists in the hash map.

- If yes, return the stored value
- Otherwise return `-1`
8. When `count()` is called, simply return the size of the hash map.

Since expired keys are automatically deleted, the map size always equals the number of active entries.

### Why it works

The algorithm maintains the invariant that the hash map contains only un-expired keys. Every insertion schedules automatic removal at the correct expiration time, and overwriting a key cancels its previous expiration timer. Therefore, the cache state is always accurate, making `get` and `count` straightforward constant-time operations.

## Python Solution

```python
from threading import Timer
from typing import Dict, Tuple

class TimeLimitedCache:

    def __init__(self):
        self.cache: Dict[int, Tuple[int, Timer]] = {}

    def set(self, key: int, value: int, duration: int) -> bool:
        key_exists = key in self.cache

        if key_exists:
            _, old_timer = self.cache[key]
            old_timer.cancel()

        def expire():
            if key in self.cache:
                del self.cache[key]

        timer = Timer(duration / 1000, expire)
        timer.start()

        self.cache[key] = (value, timer)

        return key_exists

    def get(self, key: int) -> int:
        if key not in self.cache:
            return -1

        return self.cache[key][0]

    def count(self) -> int:
        return len(self.cache)
```

The implementation uses Python's `threading.Timer` class to schedule automatic expiration.

The constructor initializes a dictionary named `cache`. Each entry maps a key to a tuple containing:

- The stored value
- The timer responsible for expiration

Inside `set`, we first determine whether the key already exists. If it does, we cancel the previous timer because the expiration time is being reset.

Next, we define an `expire` function that removes the key from the cache once the timer fires.

A new timer is created with `duration / 1000` because Python timers use seconds rather than milliseconds.

The updated `(value, timer)` pair is stored in the dictionary, and we return whether the key previously existed.

The `get` method simply checks whether the key is present. Since expired keys are automatically removed, existence alone determines validity.

The `count` method returns the dictionary size, which always equals the number of active entries.

## Go Solution

```go
package main

import (
	"time"
)

type CacheEntry struct {
	value int
	timer *time.Timer
}

type TimeLimitedCache struct {
	cache map[int]CacheEntry
}

func Constructor() TimeLimitedCache {
	return TimeLimitedCache{
		cache: make(map[int]CacheEntry),
	}
}

func (this *TimeLimitedCache) Set(key int, value int, duration int) bool {
	_, exists := this.cache[key]

	if exists {
		this.cache[key].timer.Stop()
	}

	timer := time.AfterFunc(time.Duration(duration)*time.Millisecond, func() {
		delete(this.cache, key)
	})

	this.cache[key] = CacheEntry{
		value: value,
		timer: timer,
	}

	return exists
}

func (this *TimeLimitedCache) Get(key int) int {
	entry, exists := this.cache[key]

	if !exists {
		return -1
	}

	return entry.value
}

func (this *TimeLimitedCache) Count() int {
	return len(this.cache)
}
```

The Go implementation follows the same logic as the Python version but uses Go's `time.AfterFunc` for scheduling expiration callbacks.

A small struct named `CacheEntry` stores both the value and the timer reference.

When overwriting a key, the old timer is stopped using `Stop()` before creating a new expiration timer.

Go maps naturally support constant-time lookup and deletion, making them ideal for implementing the cache.

## Edge Cases

### Overwriting an Existing Key Before Expiration

One subtle bug source occurs when a key is updated before its previous timer expires. If the old timer is not cancelled, it may later remove the newly updated value incorrectly.

The implementation avoids this by explicitly cancelling the previous timer before creating a new one.

### Duration Equal to Zero

A duration of zero means the key should expire immediately. Some implementations accidentally leave such keys accessible briefly.

Here, the timer executes immediately, removing the key right away. Even if there is a tiny scheduling delay internally, subsequent operations correctly observe expiration.

### Calling `get` on Missing or Expired Keys

A naive implementation might distinguish incorrectly between never-inserted keys and expired keys.

This implementation treats both cases identically because expired keys are physically removed from the cache. Therefore, `get` simply checks whether the key exists in the dictionary and returns `-1` otherwise.
