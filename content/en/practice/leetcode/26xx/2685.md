# LeetCode 2685 - Count the Number of Complete Components

## Problem Understanding

This problem gives us an undirected graph with `n` vertices labeled from `0` to `n - 1`. The graph is represented using an edge list, where each edge connects two vertices bidirectionally.

Our goal is to count how many connected components in the graph are complete graphs.

A connected component is a group of nodes where every node can reach every other node through some path, and there are no edges connecting nodes inside the component to nodes outside it.

A complete component is a connected component where every pair of distinct vertices has a direct edge between them. In graph theory, this is called a complete graph.

For example:

- A single isolated node is considered complete because there are no missing edges.
- Two connected nodes form a complete component because the only possible edge exists.
- Three nodes form a complete component only if all three pairwise edges exist.

The input consists of:

- `n`, the number of vertices
- `edges`, the list of undirected edges

The output is a single integer representing how many connected components are complete.

The constraints are small:

- `n <= 50`
- Maximum possible edges are `n * (n - 1) / 2`

Because the graph is small, even relatively expensive graph traversals are acceptable. However, we still want a clean and efficient solution.

Several edge cases are important:

- A graph with no edges at all
- Multiple disconnected components
- Components containing a single node
- Components that are connected but missing exactly one edge
- Fully connected graphs
- Sparse graphs with many isolated vertices

The problem guarantees:

- No self-loops
- No duplicate edges
- Valid vertex indices

These guarantees simplify implementation because we do not need to handle malformed input.

## Approaches

### Brute Force Approach

The brute-force idea is to first identify every connected component, then explicitly verify whether every pair of vertices inside the component has an edge.

For each component:

1. Collect all vertices in the component.
2. For every pair `(u, v)` in the component:

- Check whether an edge exists between them.
3. If every pair is connected, count the component as complete.

To make pair checking efficient, we can store edges in a hash set.

This approach is correct because a graph is complete exactly when every pair of distinct vertices has an edge.

However, the pairwise verification step can become expensive. If a component contains `k` vertices, we must check `k * (k - 1) / 2` pairs.

Although the constraints are small enough for this to pass, we can do better conceptually.

### Optimal Approach

The key observation is that a complete graph with `k` vertices must contain exactly:

$\frac{k(k-1)}{2}$

edges.

Instead of checking every pair manually, we can:

1. Find each connected component using DFS or BFS.
2. Count:

- Number of vertices in the component
- Total degree sum of all vertices
3. Compute the number of edges using:

$\text{edges} = \frac{\sum \text{degree}}{2}$

because every undirected edge contributes to the degree count twice.

Then compare:

- Actual edges in the component
- Expected edges for a complete graph

If they match, the component is complete.

This avoids pairwise edge checking and gives a clean graph-theory-based solution.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n²) | Checks every vertex pair inside each component |
| Optimal | O(n + e) | O(n + e) | Uses DFS/BFS and edge counting |

Since `n <= 50`, both are feasible, but the optimal solution is cleaner and scales better conceptually.

## Algorithm Walkthrough

### Step 1: Build the Adjacency List

Create an adjacency list representation of the graph.

For every edge `[u, v]`:

- Add `v` to `u`'s neighbor list
- Add `u` to `v`'s neighbor list

An adjacency list is ideal because:

- Traversing neighbors is efficient
- DFS and BFS naturally operate on it
- Space usage is proportional to the number of edges

### Step 2: Track Visited Vertices

Create a `visited` array of size `n`.

This prevents revisiting nodes during DFS and ensures each connected component is processed exactly once.

### Step 3: Traverse Each Connected Component

Loop through all vertices from `0` to `n - 1`.

If a vertex has not been visited:

- Start a DFS from that vertex
- Collect information about the component

During DFS, maintain:

- `node_count`, the number of vertices in the component
- `degree_sum`, the total sum of degrees of all nodes

For each visited node:

- Increment `node_count`
- Add `len(graph[node])` to `degree_sum`

### Step 4: Compute the Actual Edge Count

In an undirected graph, every edge contributes twice to the degree total.

So the actual number of edges is:

$\text{actual edges} = \frac{\text{degree sum}}{2}$

### Step 5: Compute the Expected Edge Count

A complete graph with `k` nodes must contain:

$\text{expected edges} = \frac{k(k-1)}{2}$

where `k` is the number of nodes in the component.

### Step 6: Compare Counts

If:

- `actual_edges == expected_edges`

then the component is complete.

Increment the answer counter.

### Step 7: Return the Result

After processing all components, return the total count of complete components.

### Why it works

The algorithm works because a connected component is complete if and only if every possible edge between its vertices exists.

For a component with `k` vertices, there are exactly:

$\frac{k(k-1)}{2}$

possible undirected edges.

By counting the actual number of edges in the component and comparing it to this required value, we can determine whether the component is complete.

DFS guarantees that every node in a connected component is visited exactly once, so every component is processed correctly and independently.

## Python Solution

```python
from typing import List

class Solution:
    def countCompleteComponents(self, n: int, edges: List[List[int]]) -> int:
        graph = [[] for _ in range(n)]

        for u, v in edges:
            graph[u].append(v)
            graph[v].append(u)

        visited = [False] * n

        def dfs(node: int) -> tuple[int, int]:
            visited[node] = True

            node_count = 1
            degree_sum = len(graph[node])

            for neighbor in graph[node]:
                if not visited[neighbor]:
                    child_nodes, child_degree_sum = dfs(neighbor)
                    node_count += child_nodes
                    degree_sum += child_degree_sum

            return node_count, degree_sum

        complete_components = 0

        for node in range(n):
            if not visited[node]:
                node_count, degree_sum = dfs(node)

                actual_edges = degree_sum // 2
                expected_edges = node_count * (node_count - 1) // 2

                if actual_edges == expected_edges:
                    complete_components += 1

        return complete_components
```

The solution begins by constructing the adjacency list for the undirected graph. Each edge is added in both directions because the graph is undirected.

The `visited` array ensures each node is explored only once.

The DFS function returns two values:

- Number of nodes in the current component
- Sum of degrees inside the component

Every recursive DFS call explores all reachable nodes, which means the entire connected component is processed together.

After DFS finishes for a component:

- `degree_sum // 2` gives the actual number of edges
- `node_count * (node_count - 1) // 2` gives the expected edge count for a complete graph

If these values match, the component is complete.

Finally, the algorithm returns the total number of complete components.

## Go Solution

```go
func countCompleteComponents(n int, edges [][]int) int {
	graph := make([][]int, n)

	for _, edge := range edges {
		u := edge[0]
		v := edge[1]

		graph[u] = append(graph[u], v)
		graph[v] = append(graph[v], u)
	}

	visited := make([]bool, n)

	var dfs func(int) (int, int)

	dfs = func(node int) (int, int) {
		visited[node] = true

		nodeCount := 1
		degreeSum := len(graph[node])

		for _, neighbor := range graph[node] {
			if !visited[neighbor] {
				childNodes, childDegreeSum := dfs(neighbor)

				nodeCount += childNodes
				degreeSum += childDegreeSum
			}
		}

		return nodeCount, degreeSum
	}

	completeComponents := 0

	for node := 0; node < n; node++ {
		if !visited[node] {
			nodeCount, degreeSum := dfs(node)

			actualEdges := degreeSum / 2
			expectedEdges := nodeCount * (nodeCount - 1) / 2

			if actualEdges == expectedEdges {
				completeComponents++
			}
		}
	}

	return completeComponents
}
```

The Go implementation closely mirrors the Python solution.

The adjacency list is implemented using a slice of slices. DFS is written as a recursive closure function.

Go does not support tuple returns directly like Python, so multiple values are returned using Go's native multi-value return syntax.

Integer division is safe because the degree sum of an undirected graph is always even.

Since the constraints are small, recursion depth is not a concern.

## Edge Cases

### Isolated Vertices

A single node with no edges is still considered a complete graph. This can be unintuitive because there are no edges at all.

Mathematically, a complete graph requires every pair of vertices to have an edge. A single-node graph has zero pairs, so the condition is vacuously true.

The implementation handles this naturally:

- `node_count = 1`
- `expected_edges = 0`
- `actual_edges = 0`

so the component is counted correctly.

### Connected Components Missing One Edge

A common bug is assuming that a highly connected component is complete.

For example:

```
0 - 1
 \ /
  2

3 - 4
|
5
```

The second component has three nodes and two edges, but a complete graph with three nodes requires three edges.

The implementation avoids this mistake by using the exact complete graph formula instead of heuristic checks.

### Multiple Disconnected Components

Graphs may contain many disconnected groups of nodes.

A naive implementation might accidentally merge component statistics together or revisit nodes multiple times.

The `visited` array ensures:

- Every node belongs to exactly one DFS traversal
- Each component is processed independently
- Edge counts and node counts remain accurate for each component

This guarantees correctness even when the graph has many disconnected pieces.
