# LeetCode 2640 - Find the Score of All Prefixes of an Array

## Problem Understanding

The problem defines a special transformation called the conversion array. For every index `i`, the converted value is:

- `nums[i] + max(nums[0..i])`

In other words, each element is added to the maximum value seen so far in the array up to that position.

The score of an array is then defined as the sum of all values in this conversion array.

The task is not to compute the score only once for the full array. Instead, for every prefix `nums[0..i]`, we must compute the score of that prefix and store it in the answer array.

This means:

- `ans[0]` is the score of the prefix containing only the first element
- `ans[1]` is the score of the first two elements
- `ans[2]` is the score of the first three elements
- and so on

For example, if:

```
nums = [2,3,7]
```

Then:

- Prefix `[2]`

- max so far = 2
- conversion array = `[2 + 2] = [4]`
- score = 4
- Prefix `[2,3]`

- running maximums = `[2,3]`
- conversion array = `[4,6]`
- score = 10
- Prefix `[2,3,7]`

- running maximums = `[2,3,7]`
- conversion array = `[4,6,14]`
- score = 24

So the output becomes:

```
[4,10,24]
```

The constraints are important:

- `nums.length` can be up to `10^5`
- `nums[i]` can be up to `10^9`

These constraints immediately tell us that an `O(n^2)` solution is too slow. Recomputing the maximum and score separately for every prefix would perform billions of operations in the worst case.

We therefore need an efficient linear-time approach.

Several edge cases are important:

- Arrays with only one element
- Strictly increasing arrays, where the maximum changes every step
- Arrays where the maximum stays constant for long periods
- Very large values, which require careful handling of integer overflow in languages like Go
- Long arrays, where repeated recomputation becomes prohibitively expensive

The problem guarantees that all numbers are positive integers, which simplifies handling because we never deal with negative running maximums.

## Approaches

### Brute Force Approach

A straightforward solution is to process every prefix independently.

For each prefix ending at index `i`:

1. Iterate through all elements from `0` to `i`
2. Compute the running maximum for every position
3. Build the conversion array
4. Sum all converted values
5. Store the score in `ans[i]`

This approach is correct because it directly follows the problem definition.

However, it is inefficient because each prefix recomputes information already processed for earlier prefixes. For an array of length `n`, this leads to roughly:

```
1 + 2 + 3 + ... + n = O(n^2)
```

operations.

With `n = 10^5`, this becomes far too slow.

### Optimal Approach

The key observation is that we do not need to recompute everything for every prefix.

While scanning the array from left to right, we can maintain:

- The current maximum value seen so far
- The current prefix score

At index `i`:

1. Update the running maximum
2. Compute the converted value:

- `nums[i] + current_max`
3. Add this converted value to the running score
4. Store the running score into the answer array

This works because every prefix score is simply the previous prefix score plus one new converted value.

Instead of recalculating old work, we incrementally build the answer in one pass.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) excluding output | Recomputes maximums and scores for every prefix |
| Optimal | O(n) | O(n) | Uses running maximum and prefix accumulation |

## Algorithm Walkthrough

1. Initialize an empty answer array of length `n`.
2. Create a variable `current_max` to store the maximum value seen so far. Initially, this can be `0`.
3. Create another variable `running_score` to store the cumulative score of the current prefix.
4. Iterate through the array from left to right.
5. For each element `nums[i]`:

- Update the running maximum:

- `current_max = max(current_max, nums[i])`
- Compute the converted value:

- `converted = nums[i] + current_max`
- Add this converted value to the running score:

- `running_score += converted`
- Store the running score into `ans[i]`
6. After processing all elements, return the answer array.

### Why it works

At every index `i`, the algorithm maintains two correct invariants:

- `current_max` always equals the maximum element in `nums[0..i]`
- `running_score` always equals the sum of all converted values for the current prefix

Because each prefix differs from the previous prefix by exactly one new element, we only need to compute one additional converted value each step. This guarantees correctness while avoiding redundant computation.

## Python Solution

```python
from typing import List

class Solution:
    def findPrefixScore(self, nums: List[int]) -> List[int]:
        answer = []
        
        current_max = 0
        running_score = 0
        
        for value in nums:
            current_max = max(current_max, value)
            
            converted_value = value + current_max
            
            running_score += converted_value
            
            answer.append(running_score)
        
        return answer
```

The implementation follows the algorithm directly.

The variable `current_max` tracks the largest number encountered so far while traversing the array.

For each element, we compute its converted value by adding the current running maximum. That converted value contributes to the total score of the current prefix.

The variable `running_score` avoids recomputing prefix sums repeatedly. Instead of recalculating the score from scratch for every prefix, we simply extend the previous score with one additional contribution.

Finally, every updated prefix score is appended to the answer array.

## Go Solution

```go
func findPrefixScore(nums []int) []int64 {
    n := len(nums)
    
    answer := make([]int64, n)
    
    currentMax := 0
    var runningScore int64 = 0
    
    for i, value := range nums {
        if value > currentMax {
            currentMax = value
        }
        
        convertedValue := int64(value + currentMax)
        
        runningScore += convertedValue
        
        answer[i] = runningScore
    }
    
    return answer
}
```

The Go implementation is nearly identical to the Python solution.

The most important difference is integer handling. Since:

- `nums[i]` can be up to `10^9`
- `n` can be up to `10^5`

the total score can exceed the range of a 32-bit integer.

For this reason:

- `runningScore`
- `convertedValue`
- `answer`

all use `int64`.

Go slices are dynamically allocated using `make`, and the result slice is filled in place during traversal.

The algorithm performs a single left-to-right traversal of the input array. Every operation inside the loop takes constant time, so the total runtime is linear.

The only additional memory used beyond scalar variables is the output array itself.

## Edge Cases

### Single Element Array

When the array contains only one element, the conversion array contains exactly one value:

```
nums[0] + nums[0]
```

A buggy implementation might incorrectly initialize the running maximum or forget to append the first score. The current implementation handles this naturally because the loop processes every element uniformly, including the first one.

### Maximum Never Changes

Consider:

```
[10,1,1,1]
```

The running maximum remains `10` for the entire traversal after the first element. Some incorrect solutions recompute maximums inefficiently or accidentally reset the maximum during iteration.

This implementation preserves the invariant that `current_max` always stores the largest value seen so far.

### Very Large Values

The constraints allow values up to `10^9`, and the total accumulated score can become much larger than a 32-bit integer.

For example:

```
nums = [10^9, 10^9, 10^9, ...]
```

The Go implementation explicitly uses `int64` to avoid overflow. Python integers automatically expand, so no special handling is required there.

### Repeated Equal Maximums

Arrays such as:

```
[5,5,5,5]
```

can expose bugs where implementations incorrectly update the maximum only on strictly larger values but mishandle equal values elsewhere in the logic.

This solution works correctly because the converted value is always computed using the current maximum, regardless of whether the maximum changed during the current iteration.
