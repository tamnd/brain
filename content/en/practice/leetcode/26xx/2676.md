# LeetCode 2676 - Throttle

## Problem Understanding

This problem asks us to implement a throttling mechanism for a function. A throttled function limits how frequently another function can execute. The specific throttling behavior in this problem is slightly different from a basic throttle because it must remember the most recent call that occurred during the cooldown period.

We are given a function `fn` and a delay duration `t` in milliseconds. The returned throttled version of `fn` behaves according to these rules:

1. The first call executes immediately.
2. After execution, the function enters a blocked period lasting `t` milliseconds.
3. During this blocked period, incoming calls do not execute immediately.
4. Instead, only the latest call arguments are stored.
5. Once the blocked period ends:

- if stored arguments exist, `fn` executes once using the most recent stored arguments,
- this execution starts another blocked period of length `t`.
6. This process continues until no stored calls remain.

The important detail is that only the latest pending arguments matter. Earlier pending calls are discarded if newer calls arrive before the cooldown expires.

The input examples describe events occurring at specific timestamps. The output shows exactly when the original function `fn` actually executes and with which arguments.

The constraints are very small. There are at most 10 calls in the examples, and timestamps are bounded by 1000 milliseconds. Even though the constraints are tiny, the goal is to correctly implement the throttling behavior, not merely simulate small inputs inefficiently.

One subtle aspect that can cause bugs is handling calls that arrive exactly when the cooldown ends. Such calls should execute immediately because the blocked period has already expired.

Another important edge case occurs when multiple calls happen during the same cooldown interval. Only the latest arguments should survive.

The problem also guarantees that timestamps are non-negative integers, which simplifies scheduling logic.

## Approaches

### Brute Force Approach

A brute-force simulation could explicitly track every pending call and repeatedly scan the timeline millisecond by millisecond. At each millisecond, we would check whether the throttle window has expired and whether a queued call should execute.

This approach works because it models the system exactly as described. However, it is unnecessarily inefficient because most milliseconds contain no events. If timestamps were large, iterating through every millisecond would waste significant time.

The brute-force method also complicates implementation because we must maintain a detailed event queue and synchronize simulated time progression with scheduled executions.

### Optimal Approach

The key observation is that we do not need to simulate every millisecond. We only care about two kinds of moments:

1. Incoming function calls
2. Scheduled throttle releases

Instead of tracking all intermediate time values, we can maintain:

- the next time when execution is allowed,
- the latest pending arguments,
- and whether a delayed execution is currently scheduled.

When a call arrives:

- if the current time is outside the cooldown window, execute immediately,
- otherwise overwrite the stored pending arguments.

When the cooldown expires:

- if pending arguments exist, execute them immediately and start a new cooldown.

This event-driven approach processes only meaningful timestamps and directly models the throttling rules.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(maxTime) | O(1) | Simulates every millisecond |
| Optimal | O(n) | O(1) | Processes only actual events |

## Algorithm Walkthrough

1. Initialize variables to track the throttle state.

We maintain:

- `next_available_time`, the earliest timestamp when execution is allowed,
- `pending_args`, the latest arguments received during cooldown,
- `pending_time`, the scheduled execution time for pending arguments.
2. Process calls in chronological order.

Each incoming call has:

- a timestamp,
- and input arguments.
3. Before handling the current call, process any scheduled executions that should occur first.

If the pending execution time is less than or equal to the current call time, we execute the stored pending arguments.

This matters because delayed executions may happen before the next incoming call arrives.
4. When executing pending arguments:

- append the execution to the result,
- reset the cooldown window,
- clear the stored pending arguments.
5. After processing scheduled executions, handle the current incoming call.

There are two cases:

- If the current time is greater than or equal to `next_available_time`, execute immediately.
- Otherwise, overwrite `pending_args` with the current arguments.
6. If arguments are stored during cooldown, schedule their execution at `next_available_time`.

We always keep only the latest pending arguments.
7. After all calls are processed, continue executing remaining pending calls until none remain.

This ensures delayed executions after the final input are included.

### Why it works

The algorithm maintains an important invariant:

At any moment, the system stores at most one pending execution, and that pending execution always represents the most recent call received during the current cooldown window.

Because each execution creates a new cooldown interval, and because later calls overwrite earlier pending calls, the algorithm exactly matches the throttling behavior described in the problem statement.

## Python Solution

```python
from typing import Any, Callable, List
from threading import Timer

class Throttle:
    def throttle(self, fn: Callable, t: int) -> Callable:
        next_allowed = 0
        timer = None
        pending_args = None

        def execute_pending():
            nonlocal next_allowed, timer, pending_args

            if pending_args is None:
                timer = None
                return

            args = pending_args
            pending_args = None

            fn(*args)

            next_allowed = int(__import__("time").time() * 1000) + t

            timer = Timer(t / 1000, execute_pending)
            timer.start()

        def throttled(*args):
            nonlocal next_allowed, timer, pending_args

            current_time = int(__import__("time").time() * 1000)

            if current_time >= next_allowed:
                fn(*args)

                next_allowed = current_time + t

                if timer is not None:
                    timer.cancel()

                timer = Timer(t / 1000, execute_pending)
                timer.start()
            else:
                pending_args = args

        return throttled
```

The implementation maintains the throttle state using closures. The variable `next_allowed` tracks when immediate execution becomes possible again.

Whenever the throttled function is called, we compare the current timestamp with `next_allowed`.

If execution is permitted, the function runs immediately and a timer is started for the cooldown period.

If execution is blocked, the arguments are stored in `pending_args`. Since every blocked call overwrites the previous one, only the most recent arguments survive.

The helper function `execute_pending` runs when the cooldown expires. If pending arguments exist, it executes them and starts another cooldown period automatically.

This recursive scheduling behavior exactly matches the required throttling semantics.

## Go Solution

```go
package main

import (
	"time"
)

func throttle(fn func(...interface{}), t int) func(...interface{}) {
	var nextAllowed int64 = 0
	var pendingArgs []interface{}
	var timer *time.Timer

	var executePending func()

	executePending = func() {
		if pendingArgs == nil {
			timer = nil
			return
		}

		args := pendingArgs
		pendingArgs = nil

		fn(args...)

		nextAllowed = time.Now().UnixMilli() + int64(t)

		timer = time.AfterFunc(
			time.Duration(t)*time.Millisecond,
			executePending,
		)
	}

	return func(args ...interface{}) {
		currentTime := time.Now().UnixMilli()

		if currentTime >= nextAllowed {
			fn(args...)

			nextAllowed = currentTime + int64(t)

			if timer != nil {
				timer.Stop()
			}

			timer = time.AfterFunc(
				time.Duration(t)*time.Millisecond,
				executePending,
			)
		} else {
			pendingArgs = args
		}
	}
}
```

The Go implementation follows the same logic as the Python version but uses `time.AfterFunc` for delayed execution.

Go slices are used to store pending arguments. Since slices are reference types, assigning `pendingArgs = args` is efficient.

The implementation uses `int64` timestamps because `UnixMilli()` returns `int64`.

Unlike Python closures, Go requires explicit declaration of the recursive function variable before assigning the function body.

## Edge Cases

One important edge case occurs when `t = 0`. In this scenario, there is effectively no cooldown period, so every call should execute immediately. A buggy implementation might still queue delayed calls unnecessarily. The implementation handles this correctly because the current time is always greater than or equal to `next_allowed`.

Another tricky case happens when many calls arrive during a single cooldown interval. Only the latest call should survive. If an implementation accidentally stores all calls instead of overwriting them, it would behave more like debouncing with a queue rather than true throttling. The solution avoids this by replacing `pending_args` every time a new blocked call arrives.

A third subtle edge case occurs when a call arrives exactly when the cooldown ends. For example, if the previous execution occurred at `50ms` and `t = 50`, a call at `100ms` should execute immediately. Using a strict greater-than comparison instead of greater-than-or-equal would incorrectly delay the call. The implementation explicitly uses `>=` to handle this correctly.

Another edge case involves delayed execution chaining. When a pending call executes after a cooldown, it creates a brand new cooldown period. If the implementation forgets to restart the cooldown timer after executing pending arguments, subsequent calls would incorrectly execute immediately. The recursive scheduling logic ensures the throttle state continues correctly after delayed executions.
