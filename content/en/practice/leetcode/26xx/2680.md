## Solution

## Problem Understanding

The problem gives us an integer array `nums` and an integer `k`. In one operation, we may choose any element in the array and multiply it by `2`. Since multiplying by `2` is equivalent to performing a left bit shift by one position, applying the operation `k` times to a number is equivalent to shifting it left by `k` bits.

Our goal is to maximize the value of:

$$nums[0] \;|\; nums[1] \;|\; \dots \;|\; nums[n-1]$$

where `|` is the bitwise OR operator.

The important detail is that we may perform at most `k` operations total, and we can distribute them however we want among the elements. For example, we could apply all operations to one element, or split them across multiple elements.

The array length can be as large as `10^5`, which means any solution worse than roughly linear or near-linear time will likely time out. Also, `nums[i]` can be up to `10^9`, so bit manipulation is clearly relevant.

A naive implementation might try all ways of distributing `k` operations across the array. However, even for small `k`, the number of distributions grows rapidly and becomes infeasible for `n = 10^5`.

An important observation is that left shifting a number can introduce higher bits that dominate the OR result. Since OR preserves any set bit once it appears, concentrating all `k` shifts onto a single element is always optimal. Splitting shifts across multiple elements wastes the opportunity to create the largest possible high-order bit contribution.

Several edge cases are worth noting:

- Arrays with only one element. In this case, the answer is simply that element shifted left by `k`.
- Arrays where some elements already contain many high bits. We still need to test every candidate because shifting different numbers affects different bit positions.
- Large values after shifting. In Python this is automatic, but in Go we must use `int64` carefully because shifting can exceed 32-bit integer range.

## Approaches

### Brute Force Approach

The brute force idea is to try every possible way to distribute the `k` operations across all elements.

Suppose we assign `x_i` operations to `nums[i]`, where:

$$x_0 + x_1 + \dots + x_{n-1} \le k$$

For each distribution, we compute:

$$(nums[0] << x_0) \;|\; (nums[1] << x_1) \;|\; \dots$$

and keep the maximum result.

This approach is correct because it explicitly checks every valid configuration. However, it is far too slow. The number of ways to distribute `k` operations among `n` elements grows combinatorially. Even though `k <= 15`, the array size can reach `10^5`, making enumeration impossible.

### Key Insight

The crucial observation is that applying multiple shifts to the same number is more valuable than splitting shifts across several numbers.

Bitwise OR rewards the presence of high-order bits. When we repeatedly shift one number left, we create increasingly significant bits. Those high bits dominate the OR result much more effectively than several smaller shifts on different numbers.

Therefore, the optimal strategy is:

- Choose exactly one index `i`
- Apply all `k` operations to `nums[i]`
- OR the shifted value with all remaining numbers unchanged

Now the problem becomes:

$$\max_i \left( (nums[i] << k) \;|\; \text{OR of all other elements} \right)$$

To compute the OR of all elements except one efficiently, we use prefix OR and suffix OR arrays.

| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Tries all distributions of operations |
| Optimal | O(n) | O(n) | Uses prefix and suffix OR arrays |

## Algorithm Walkthrough

1. Create a prefix OR array.

The prefix array stores the cumulative OR from the beginning up to each index.

$$prefix[i] = nums[0] \;|\; nums[1] \;|\; \dots \;|\; nums[i]$$

This allows us to quickly retrieve the OR of everything before index `i`.
2. Create a suffix OR array.

The suffix array stores the cumulative OR from the end backward.

$$suffix[i] = nums[i] \;|\; nums[i+1] \;|\; \dots \;|\; nums[n-1]$$

This allows us to quickly retrieve the OR of everything after index `i`.
3. Iterate through every index as the candidate element to receive all `k` operations.

For each index `i`:

- Compute the OR of elements before `i`
- Compute the OR of elements after `i`
- Shift `nums[i]` left by `k`
- Combine all parts with OR
4. Track the maximum value.

Since we test every possible target element, the maximum encountered value is the answer.
5. Return the maximum result.

### Why it works

The correctness relies on the fact that repeated left shifts on a single number generate larger high-order bits than distributing shifts across multiple numbers. Since OR keeps all set bits, creating the largest possible high bit contribution is always optimal. The prefix and suffix arrays ensure we can efficiently compute the OR of all elements except the chosen index in constant time per candidate.

## Python Solution

```python
from typing import List

class Solution:
    def maximumOr(self, nums: List[int], k: int) -> int:
        n = len(nums)

        prefix = [0] * n
        suffix = [0] * n

        prefix[0] = nums[0]
        for i in range(1, n):
            prefix[i] = prefix[i - 1] | nums[i]

        suffix[n - 1] = nums[n - 1]
        for i in range(n - 2, -1, -1):
            suffix[i] = suffix[i + 1] | nums[i]

        answer = 0

        for i in range(n):
            left_or = prefix[i - 1] if i > 0 else 0
            right_or = suffix[i + 1] if i < n - 1 else 0

            shifted = nums[i] << k

            current = left_or | shifted | right_or

            answer = max(answer, current)

        return answer
```

The implementation begins by building the prefix OR array. Each position stores the OR of all values up to that index. This allows constant-time lookup of the OR of elements before any position.

Next, the suffix OR array is constructed in reverse order. Each position stores the OR of all elements from that index to the end.

The main loop evaluates each element as the candidate to receive all `k` shifts. The element is shifted left by `k`, while the OR of all remaining elements is assembled using the prefix and suffix arrays.

Finally, the algorithm tracks the largest OR value found and returns it.

## Go Solution

```go
func maximumOr(nums []int, k int) int64 {
	n := len(nums)

	prefix := make([]int64, n)
	suffix := make([]int64, n)

	prefix[0] = int64(nums[0])
	for i := 1; i < n; i++ {
		prefix[i] = prefix[i-1] | int64(nums[i])
	}

	suffix[n-1] = int64(nums[n-1])
	for i := n - 2; i >= 0; i-- {
		suffix[i] = suffix[i+1] | int64(nums[i])
	}

	var answer int64 = 0

	for i := 0; i < n; i++ {
		var leftOr int64 = 0
		var rightOr int64 = 0

		if i > 0 {
			leftOr = prefix[i-1]
		}

		if i < n-1 {
			rightOr = suffix[i+1]
		}

		shifted := int64(nums[i]) << k

		current := leftOr | shifted | rightOr

		if current > answer {
			answer = current
		}
	}

	return answer
}
```

The Go solution follows the same logic as the Python version. The main difference is explicit use of `int64` to avoid overflow issues during left shifts. Since shifting can produce values larger than 32-bit integers, using `int64` ensures correctness.

Slices are used for the prefix and suffix arrays, and the implementation carefully handles boundary conditions when computing left and right OR values.

## Edge Cases

### Single Element Array

If the array contains only one element, all operations must be applied to that element. A buggy implementation might incorrectly attempt to access nonexistent prefix or suffix neighbors. This solution handles the case correctly because the left and right OR values default to `0`.

### Very Large Left Shifts

Shifting numbers left by up to `15` positions can produce values much larger than the original input range. In Go especially, using ordinary `int` carelessly could lead to overflow on some systems. The implementation explicitly uses `int64` to safely store shifted results.

### Choosing the Wrong Element to Shift

A naive greedy strategy might always shift the numerically largest element. However, the optimal choice depends on bit patterns, not just magnitude. The algorithm correctly evaluates every index independently, guaranteeing the maximum possible OR value is found.
