## Solution

## Problem Understanding

This problem asks us to construct a complete knight's tour on an `m x n` chessboard. A knight starts at position `(r, c)` and must move according to standard chess knight rules until every cell on the board has been visited exactly once.

The output is not a sequence of moves, but rather a matrix where each cell stores the order in which it was visited. The starting position must contain `0`, the next visited position contains `1`, and so on until all `m * n` cells are filled.

A knight move always changes position by:

- 2 rows and 1 column, or
- 1 row and 2 columns

For example, from `(x, y)` a knight may move to:

- `(x + 2, y + 1)`
- `(x + 2, y - 1)`
- `(x - 2, y + 1)`
- `(x - 2, y - 1)`
- `(x + 1, y + 2)`
- `(x + 1, y - 2)`
- `(x - 1, y + 2)`
- `(x - 1, y - 2)`

The board dimensions are very small:

- `1 <= m, n <= 5`

This is extremely important because it tells us exhaustive search is feasible. Even though the Knight's Tour problem is difficult on large boards, the search space here is constrained enough that a backtracking solution can work comfortably.

The problem also guarantees that a valid tour exists for the given inputs. That means we do not need to handle impossible cases or return failure states. Our algorithm only needs to find one valid arrangement.

Several edge cases are important:

- A `1 x 1` board contains only the starting cell.
- Very narrow boards may severely restrict knight movement.
- The knight must never revisit a cell.
- Every cell must be visited exactly once, no more and no less.

Because the board is tiny and a solution is guaranteed, a recursive backtracking search with intelligent move ordering is the ideal approach.

## Approaches

### Brute Force Approach

The brute force solution tries every possible sequence of knight moves recursively.

Starting from `(r, c)`, we attempt all valid knight moves that lead to unvisited cells. From each candidate position, we continue recursively until either:

- all cells are visited, or
- no valid moves remain

If we reach a dead end before visiting all cells, we backtrack by undoing the move and trying another option.

This approach is guaranteed to find a valid tour because it explores every possible path systematically.

However, the branching factor can become very large. At each position the knight may have up to 8 possible moves, leading to exponential growth in the number of explored states.

Although the board size is small enough for brute force to sometimes succeed, naive exploration can still waste significant time exploring bad paths.

### Optimal Approach, Backtracking with Warnsdorff-Style Heuristic

The key observation is that some knight moves are much more promising than others.

If we always move into positions that have the fewest future move options, we reduce the chance of getting trapped later. This idea is known as Warnsdorff's heuristic.

Instead of trying moves in arbitrary order, we:

1. Generate all valid next moves.
2. Count how many onward moves each candidate has.
3. Explore candidates with the fewest onward moves first.

This dramatically reduces unnecessary exploration because constrained positions are handled early rather than postponed until they become impossible.

The algorithm still uses backtracking, but the heuristic makes it highly efficient for the small board sizes in this problem.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(8^(m*n)) | O(m*n) | Explores all possible knight move sequences |
| Optimal | O(8^(m*n)) worst case, much faster in practice | O(m*n) | Uses backtracking with move-ordering heuristic |

## Algorithm Walkthrough

### Step 1: Initialize the Board

Create an `m x n` matrix filled with `-1`.

A value of `-1` means the cell has not been visited yet.

Mark the starting position `(r, c)` with `0` because it is visited first.

### Step 2: Store All Knight Move Directions

A knight has exactly 8 possible movement patterns.

We store them in a list so we can iterate through all candidate moves efficiently.

```
(2,1), (2,-1), (-2,1), (-2,-1),
(1,2), (1,-2), (-1,2), (-1,-2)
```

### Step 3: Define a Function to Count Future Moves

For a given position `(x, y)`, count how many valid unvisited positions can be reached next.

This helps us apply Warnsdorff's heuristic.

Cells with fewer future options are explored first.

### Step 4: Generate Candidate Moves

From the current cell:

1. Try all 8 knight directions.
2. Keep only positions inside the board.
3. Ignore already visited cells.

For each valid candidate, compute its onward degree using the helper function from Step 3.

### Step 5: Sort Candidates

Sort candidate moves by the number of onward moves.

The move with the fewest future options is explored first.

This reduces the probability of entering dead-end configurations later.

### Step 6: Recursive Backtracking

For each sorted candidate move:

1. Mark the cell with the current step number.
2. Recursively continue the search.
3. If recursion succeeds, return success immediately.
4. Otherwise undo the move by restoring `-1`.

This is the core backtracking mechanism.

### Step 7: Detect Completion

If the current step number equals `m * n`, then every cell has been visited successfully.

Return `True` to terminate the recursion.

### Step 8: Return the Final Board

Once recursion succeeds, the board contains the visiting order for the complete knight's tour.

Return the board.

### Why it works

The algorithm maintains the invariant that every visited cell contains a unique visit number and every unvisited cell remains `-1`.

Backtracking guarantees completeness because every valid sequence of knight moves can eventually be explored if necessary.

Warnsdorff's heuristic improves efficiency by prioritizing constrained positions early, which greatly reduces the number of dead-end branches explored.

Because the problem guarantees at least one valid solution, the recursive search will eventually find a complete tour.

## Python Solution

```python
from typing import List

class Solution:
    def tourOfKnight(self, m: int, n: int, r: int, c: int) -> List[List[int]]:
        board = [[-1] * n for _ in range(m)]

        directions = [
            (2, 1), (2, -1),
            (-2, 1), (-2, -1),
            (1, 2), (1, -2),
            (-1, 2), (-1, -2)
        ]

        def is_valid(x: int, y: int) -> bool:
            return 0 <= x < m and 0 <= y < n and board[x][y] == -1

        def count_onward_moves(x: int, y: int) -> int:
            count = 0

            for dx, dy in directions:
                nx = x + dx
                ny = y + dy

                if is_valid(nx, ny):
                    count += 1

            return count

        def backtrack(x: int, y: int, step: int) -> bool:
            if step == m * n:
                return True

            candidates = []

            for dx, dy in directions:
                nx = x + dx
                ny = y + dy

                if is_valid(nx, ny):
                    degree = count_onward_moves(nx, ny)
                    candidates.append((degree, nx, ny))

            candidates.sort()

            for _, nx, ny in candidates:
                board[nx][ny] = step

                if backtrack(nx, ny, step + 1):
                    return True

                board[nx][ny] = -1

            return False

        board[r][c] = 0
        backtrack(r, c, 1)

        return board
```

The implementation begins by initializing the board with `-1` values, indicating all cells are initially unvisited.

The `directions` array stores the eight possible knight moves. This avoids repeated hardcoding and keeps the move generation logic concise.

The `is_valid` helper checks two conditions simultaneously:

- the position is inside board boundaries
- the cell has not been visited

The `count_onward_moves` function implements Warnsdorff's heuristic. For a candidate cell, it counts how many valid next moves remain available.

The recursive `backtrack` function performs the depth-first search. The parameter `step` represents how many cells have already been assigned visit numbers.

When `step == m * n`, every cell has been visited successfully, so recursion terminates.

Candidate moves are collected together with their onward degree values, then sorted so the most constrained positions are explored first.

If a recursive path fails, the algorithm restores the board cell to `-1`, undoing the move and allowing other branches to be explored correctly.

## Go Solution

```go
func tourOfKnight(m int, n int, r int, c int) [][]int {
	board := make([][]int, m)

	for i := 0; i < m; i++ {
		board[i] = make([]int, n)

		for j := 0; j < n; j++ {
			board[i][j] = -1
		}
	}

	directions := [][]int{
		{2, 1}, {2, -1},
		{-2, 1}, {-2, -1},
		{1, 2}, {1, -2},
		{-1, 2}, {-1, -2},
	}

	isValid := func(x int, y int) bool {
		return x >= 0 && x < m && y >= 0 && y < n && board[x][y] == -1
	}

	var countOnwardMoves func(int, int) int

	countOnwardMoves = func(x int, y int) int {
		count := 0

		for _, d := range directions {
			nx := x + d[0]
			ny := y + d[1]

			if isValid(nx, ny) {
				count++
			}
		}

		return count
	}

	type Move struct {
		degree int
		x      int
		y      int
	}

	var backtrack func(int, int, int) bool

	backtrack = func(x int, y int, step int) bool {
		if step == m*n {
			return true
		}

		candidates := []Move{}

		for _, d := range directions {
			nx := x + d[0]
			ny := y + d[1]

			if isValid(nx, ny) {
				degree := countOnwardMoves(nx, ny)

				candidates = append(candidates, Move{
					degree: degree,
					x:      nx,
					y:      ny,
				})
			}
		}

		for i := 0; i < len(candidates); i++ {
			for j := i + 1; j < len(candidates); j++ {
				if candidates[j].degree < candidates[i].degree {
					candidates[i], candidates[j] = candidates[j], candidates[i]
				}
			}
		}

		for _, move := range candidates {
			board[move.x][move.y] = step

			if backtrack(move.x, move.y, step+1) {
				return true
			}

			board[move.x][move.y] = -1
		}

		return false
	}

	board[r][c] = 0
	backtrack(r, c, 1)

	return board
}
```

The Go implementation follows the same algorithmic structure as the Python solution.

One difference is that Go does not provide a built-in tuple type, so a small `Move` struct is used to store candidate move information.

The board is initialized manually because Go slices default to `0`, while we require `-1` to represent unvisited cells.

Sorting is implemented with a simple nested loop because the candidate list is extremely small, at most 8 elements. This keeps the implementation lightweight without requiring additional imports.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(8^(m*n)) worst case | Recursive search may explore many move combinations |
| Space | O(m*n) | Board storage plus recursion depth |

Although the theoretical worst-case complexity is exponential, the board dimensions are capped at `5 x 5`, which keeps the total search manageable.

Warnsdorff's heuristic dramatically reduces practical runtime because it prioritizes constrained cells early and avoids many dead-end paths.

The recursion depth never exceeds `m * n`, so auxiliary space usage is linear in the number of board cells.

## Edge Cases

### Single Cell Board

A `1 x 1` board is the smallest possible input. There are no legal knight moves because the knight already occupies the only cell.

A buggy implementation might incorrectly attempt recursion or fail because no moves exist. This solution handles it naturally because the completion condition `step == m * n` becomes true immediately after placing `0`.

### Starting Near the Boundary

When the knight begins in a corner or along an edge, many of the 8 theoretical knight moves fall outside the board.

Without careful boundary checks, the algorithm could access invalid indices and crash. The `is_valid` helper ensures every candidate move stays inside board boundaries before it is explored.

### Dead-End Paths During Search

Some move sequences can trap the knight before all cells are visited.

A naive greedy solution without backtracking might get stuck permanently and fail even though a valid solution exists.

This implementation avoids that issue through recursive backtracking. Whenever a path fails, the algorithm undoes the move and tries another candidate, guaranteeing eventual success because the problem guarantees at least one valid tour exists.
