## Solution

## Problem Understanding

This problem asks us to build a utility that can control the execution of a JavaScript generator which yields promises. The generator represents a sequence of asynchronous operations, and our task is to manage the communication between the generator and those promises while also supporting cancellation.

The `cancellable` function receives a generator object and must return two values:

1. A `cancel` function
2. A `promise`

The returned promise represents the eventual completion of the generator.

The generator only yields promises. Whenever a yielded promise resolves, its resolved value must be passed back into the generator using `generator.next(value)`. Whenever a yielded promise rejects, the rejection reason must be thrown back into the generator using `generator.throw(error)`.

The challenging part is cancellation support.

If the user calls `cancel()` before the generator completes, we must inject the string `"Cancelled"` into the generator using `generator.throw("Cancelled")`.

At that point, two outcomes are possible:

- If the generator catches the error using a `try/catch`, execution continues normally.
- If the generator does not catch the error, the entire returned promise rejects with `"Cancelled"`.

The problem also guarantees that after cancellation, no further work should continue unless the generator explicitly handles the cancellation and continues execution.

This problem is fundamentally about simulating the JavaScript runtime behavior around generators and asynchronous control flow.

The input is a generator object. The output is an array containing:

- A function used to cancel execution
- A promise representing the final result

The constraints are very small. The cancellation time is at most 1000 milliseconds, and the generator yields promises. Because the input size is tiny, efficiency is not the primary challenge. The real difficulty is correctness and asynchronous control flow management.

Several edge cases are important:

- The generator may immediately return without yielding anything.
- A yielded promise may reject.
- Cancellation may happen while waiting on a promise.
- Cancellation may happen after completion, which should do nothing.
- The generator may catch cancellation and continue execution.
- The generator may throw uncaught exceptions.

A naive implementation can easily mishandle promise rejection propagation, generator completion detection, or cancellation timing.

## Approaches

### Brute Force Approach

A brute force approach would repeatedly poll the generator state using timers or manually track every yielded promise with nested callbacks. One could attempt to maintain a queue of pending operations and continuously check whether cancellation occurred.

This works conceptually because each promise resolution or rejection can eventually be forwarded back into the generator. However, such an implementation quickly becomes messy and difficult to reason about. Nested callbacks create deeply coupled control flow, and cancellation handling becomes error prone.

Additionally, repeatedly polling generator state is unnecessary and inefficient.

### Optimal Approach

The key insight is that generators already provide a clean interface for asynchronous orchestration.

At every step:

- The generator yields a promise.
- We wait for that promise to settle.
- We feed the result back into the generator.
- The generator either yields another promise or finishes.

This naturally forms a recursive asynchronous loop.

Cancellation becomes manageable by introducing a shared cancellation flag. Whenever `cancel()` is called, we mark the operation as cancelled. The next time execution resumes, instead of calling `next(value)`, we call `throw("Cancelled")`.

This approach mirrors how asynchronous runtimes internally coordinate generators and promises.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Uses polling or deeply nested callbacks |
| Optimal | O(n) | O(n) | Clean recursive promise-driven execution |

Here, `n` represents the number of yielded promises.

## Algorithm Walkthrough

1. Create a boolean flag named `cancelled` and initialize it to `False`.

This flag tracks whether the user has requested cancellation.
2. Define the `cancel` function.

When called, this function simply sets `cancelled = True`.
3. Create a recursive helper function that advances the generator.

This helper receives:

- The method to invoke (`next` or `throw`)
- The value or error to send into the generator
4. Inside the helper, execute the generator method inside a `try/catch`.

This is necessary because:

- `generator.throw(...)` may itself throw
- User code inside the generator may throw exceptions
5. If the generator finishes (`done == true`), resolve the outer promise with the returned value.
6. Otherwise, the generator yielded a promise.

Wait for the promise to settle.
7. If the promise resolves:

- If cancellation has been requested, resume the generator using `throw("Cancelled")`
- Otherwise, resume normally using `next(resolvedValue)`
8. If the promise rejects:

- If cancellation has been requested, inject `"Cancelled"`
- Otherwise, propagate the rejection back into the generator using `throw(rejectionReason)`
9. Continue recursively until the generator completes or throws an uncaught error.

### Why it works

The algorithm maintains a simple invariant:

After every asynchronous operation completes, control is correctly transferred back into the generator using either `next()` or `throw()`.

This exactly matches JavaScript generator semantics. Because every promise settlement resumes the generator appropriately, the generator behaves exactly as if it were running under a standard asynchronous runtime. Cancellation works because injecting `"Cancelled"` through `throw()` gives the generator an opportunity to handle the error naturally using its own `try/catch` blocks.

## Python Solution

```python
from typing import Any, Callable, Generator, List
import asyncio

class Solution:
    def cancellable(
        self,
        generator: Generator
    ) -> List[Callable]:
        
        cancelled = False

        def cancel() -> None:
            nonlocal cancelled
            cancelled = True

        async def run() -> Any:
            try:
                current = next(generator)

                while True:
                    try:
                        result = await current

                        if cancelled:
                            current = generator.throw("Cancelled")
                        else:
                            current = generator.send(result)

                    except Exception as error:
                        if cancelled:
                            current = generator.throw("Cancelled")
                        else:
                            current = generator.throw(error)

            except StopIteration as stop:
                return stop.value

        return [cancel, asyncio.create_task(run())]
```

### Implementation Explanation

The implementation starts by defining a `cancelled` flag. This shared state allows both the asynchronous execution loop and the cancel function to coordinate.

The `cancel()` function only updates the flag. It does not directly interrupt execution because generators can only be resumed safely at suspension points.

The `run()` coroutine performs the main orchestration logic.

First, execution begins using `next(generator)`, which starts the generator until its first `yield`.

The yielded value is always a promise-like awaitable. The code waits for it using `await`.

After the promise resolves:

- If cancellation has occurred, the generator receives `"Cancelled"` through `generator.throw(...)`
- Otherwise, the resolved value is sent back using `generator.send(...)`

If awaiting the promise raises an exception, the error is thrown back into the generator so the generator can decide whether to handle it.

The outer `try/except StopIteration` block detects when the generator completes. In Python generators, the final returned value is stored inside `StopIteration.value`.

The returned array contains:

- The cancellation callback
- The running task representing the asynchronous execution

## Go Solution

```go
package main

import (
	"errors"
)

type Generator interface {
	Next(value interface{}) (interface{}, bool, error)
	Throw(err interface{}) (interface{}, bool, error)
}

type Promise interface {
	Then(
		resolve func(interface{}),
		reject func(interface{}),
	)
}

func cancellable(generator Generator) (func(), chan interface{}) {
	cancelled := false

	resultChan := make(chan interface{}, 1)

	cancel := func() {
		cancelled = true
	}

	var run func(method string, value interface{})

	run = func(method string, value interface{}) {
		var (
			yielded interface{}
			done    bool
			err     error
		)

		if method == "next" {
			yielded, done, err = generator.Next(value)
		} else {
			yielded, done, err = generator.Throw(value)
		}

		if err != nil {
			resultChan <- err
			return
		}

		if done {
			resultChan <- yielded
			return
		}

		promise := yielded.(Promise)

		promise.Then(
			func(res interface{}) {
				if cancelled {
					run("throw", "Cancelled")
				} else {
					run("next", res)
				}
			},
			func(rej interface{}) {
				if cancelled {
					run("throw", "Cancelled")
				} else {
					run("throw", rej)
				}
			},
		)
	}

	go run("next", nil)

	return cancel, resultChan
}

func main() {
	_ = errors.New
}
```

### Go-Specific Notes

Go does not have native JavaScript-style generators or promises, so this implementation uses interfaces to simulate the behavior.

The Python solution relies on native async/await semantics, while the Go version manually wires callbacks together.

Go also lacks exception handling like JavaScript generators, so explicit error values are returned from the simulated generator interface.

Channels are used to communicate the final result asynchronously.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each yielded promise is processed once |
| Space | O(n) | Recursive promise chaining and generator state |

The algorithm processes every yielded promise exactly one time. Each promise resolution or rejection triggers one generator resumption. Therefore the total work grows linearly with the number of yielded promises.

The space complexity comes from asynchronous recursion and generator execution state.

## Test Cases

```python
import asyncio

async def run_test(generator_func, cancel_at=None):
    solution = Solution()
    cancel, task = solution.cancellable(generator_func())

    if cancel_at is not None:
        asyncio.get_event_loop().call_later(
            cancel_at / 1000,
            cancel
        )

    try:
        result = await task
        return ("resolved", result)
    except Exception as e:
        return ("rejected", str(e))

async def tests():

    # Immediate return
    result = await run_test(lambda: (x for x in [42]))
    assert result[0] == "resolved"  # generator completes immediately

    # Promise rejection propagation
    def gen1():
        msg = yield asyncio.sleep(0, result="Hello")
        raise Exception(f"Error: {msg}")

    result = await run_test(gen1)
    assert result[0] == "rejected"  # uncaught exception propagates

    # Cancellation before completion
    def gen2():
        yield asyncio.sleep(0.2)
        return "Success"

    result = await run_test(gen2, 100)
    assert result == ("rejected", "Cancelled")  # cancellation works

    # Multiple successful yields
    def gen3():
        result = 0
        yield asyncio.sleep(0.1)
        result += yield asyncio.sleep(0, result=1)
        yield asyncio.sleep(0.1)
        result += yield asyncio.sleep(0, result=1)
        return result

    result = await run_test(gen3)
    assert result == ("resolved", 2)  # normal async workflow

    # Cancellation caught inside generator
    def gen4():
        result = 0
        try:
            yield asyncio.sleep(0.1)
            result += yield asyncio.sleep(0, result=1)
            yield asyncio.sleep(0.1)
            result += yield asyncio.sleep(0, result=1)
        except:
            return result

    result = await run_test(gen4, 150)
    assert result == ("resolved", 1)  # cancellation handled internally

    # Rejected promise caught inside generator
    def gen5():
        try:
            yield asyncio.Future()
        except:
            a = yield asyncio.sleep(0, result=2)
            b = yield asyncio.sleep(0, result=2)
            return a + b

    result = await run_test(gen5)
    assert result == ("resolved", 4)  # recovery after rejection

asyncio.run(tests())
```

| --- | --- |
| Immediate return | Verifies generators that never yield |
| Promise rejection propagation | Ensures uncaught errors reject correctly |
| Cancellation before completion | Validates cancellation injection |
| Multiple successful yields | Tests normal sequential execution |

| Rejected promise caught internally | Verifies rejection recovery behavior |

## Edge Cases

### Generator Completes Immediately

A generator may immediately return without yielding any promises. This is easy to mishandle if the implementation assumes at least one asynchronous step exists.

The implementation correctly handles this because the very first `next(generator)` call may immediately raise `StopIteration`, whose value becomes the resolved result.

### Cancellation After Completion

The user may call `cancel()` after the generator has already completed. A buggy implementation might incorrectly reject an already resolved promise.

The solution avoids this naturally because once the asynchronous task finishes, no additional generator operations occur. The cancellation flag becomes irrelevant.

### Promise Rejects While Cancelled

A subtle race condition occurs if a promise rejects after cancellation has already been requested.

The implementation prioritizes cancellation. Instead of forwarding the promise rejection into the generator, it injects `"Cancelled"`.

This guarantees deterministic cancellation semantics and prevents unexpected post-cancellation execution paths.

### Generator Catches Cancellation

A generator may intentionally catch `"Cancelled"` and continue or return partial results.

Many naive implementations immediately terminate execution upon cancellation. That would violate the problem requirements.

This implementation correctly throws `"Cancelled"` into the generator itself, allowing the generator's own `try/catch` logic to decide what should happen next.
