# LeetCode 2643, Row With Maximum Ones

## Problem Understanding

This problem gives us a binary matrix `mat` of size `m x n`, where every element is either `0` or `1`. Our goal is to determine which row contains the largest number of `1` values.

The output should be an array with exactly two elements:

1. The index of the row containing the maximum number of ones
2. The number of ones in that row

The matrix uses `0-indexing`, meaning the first row has index `0`, the second row has index `1`, and so on.

An important detail is the tie-breaking rule. If multiple rows contain the same maximum number of ones, we must return the row with the smallest index. In other words, if two rows both contain three ones, we choose the earlier row.

For example, consider this matrix:

```
[[0,1],
 [1,0]]
```

Both rows contain exactly one `1`. Since row `0` appears before row `1`, the correct answer is `[0,1]`.

The constraints are relatively small:

- `1 <= m, n <= 100`
- Every value is either `0` or `1`

Since the matrix dimensions are at most `100 x 100`, the total number of elements is at most `10,000`. This is small enough that even a straightforward traversal of the entire matrix is completely acceptable.

There are several important edge cases to keep in mind:

- Multiple rows may contain the same number of ones
- A matrix may contain all zeros
- A matrix may contain all ones
- The matrix may have only one row
- The matrix may have only one column

The problem guarantees that the matrix is non-empty, so we never need to handle an empty input.

## Approaches

### Brute Force Approach

The most direct solution is to examine every row independently and count how many ones it contains.

For each row:

1. Traverse every element
2. Count the number of `1` values
3. Compare that count with the best count seen so far
4. Update the answer if the current row has more ones

This approach is correct because every cell is examined exactly once, so we never miss any ones.

Although this is called a brute-force solution, it is actually efficient enough for the problem constraints. Since the matrix contains at most `10,000` elements, scanning all of them is inexpensive.

### Key Insight

The key observation is that the problem only asks for the row with the largest count of ones. There is no need for advanced data structures or preprocessing.

Because the matrix is small and unsorted, the simplest solution is already optimal. We can process each row independently, compute its sum, and keep track of the best row encountered so far.

Using the row sum directly is particularly convenient because the matrix contains only `0` and `1`. The sum of a row is exactly the number of ones in that row.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m × n) | O(1) | Manually count ones in every row |
| Optimal | O(m × n) | O(1) | Use row sums while tracking best row |

In this problem, the brute-force and optimal approaches are effectively the same because every element must be inspected to determine the counts accurately.

## Algorithm Walkthrough

1. Initialize two variables:

- `max_ones` to store the largest number of ones found so far
- `best_row` to store the index of the corresponding row
2. Start iterating through the matrix row by row using the row index.
3. For the current row, count how many ones it contains.

Since the row only contains `0` and `1`, the sum of the row equals the number of ones.
4. Compare the current row's count with `max_ones`.
5. If the current row contains strictly more ones than the previous best:

- Update `max_ones`
- Update `best_row`
6. If the current row has the same number of ones as the current maximum, do nothing.

This automatically preserves the smaller row index because we only update when the count is strictly larger.
7. After processing all rows, return `[best_row, max_ones]`.

### Why it works

The algorithm maintains the invariant that after processing row `i`, `best_row` stores the smallest-indexed row among all rows `0...i` that contains the maximum number of ones.

Whenever we find a row with a strictly larger count, we update the answer because it is now the best candidate. When counts are equal, we intentionally keep the earlier row index, which satisfies the tie-breaking requirement.

Since every row is processed exactly once and all counts are evaluated correctly, the final answer is guaranteed to be correct.

## Python Solution

```python
from typing import List

class Solution:
    def rowAndMaximumOnes(self, mat: List[List[int]]) -> List[int]:
        best_row = 0
        max_ones = 0

        for row_index, row in enumerate(mat):
            ones_count = sum(row)

            if ones_count > max_ones:
                max_ones = ones_count
                best_row = row_index

        return [best_row, max_ones]
```

The solution starts by initializing `best_row` and `max_ones`. Initially, we assume the first row is the answer with zero ones found so far.

The `enumerate` function is used so we can access both the row index and the row contents during iteration.

For each row, `sum(row)` computes the number of ones because the row contains only binary values. This avoids writing a manual counting loop.

If the current row contains more ones than any previous row, we update both tracking variables.

Notice that we only update when `ones_count > max_ones`. We do not update when the counts are equal. This preserves the earlier row index automatically and correctly handles ties.

Finally, the function returns the row index together with the maximum number of ones.

## Go Solution

```go
func rowAndMaximumOnes(mat [][]int) []int {
    bestRow := 0
    maxOnes := 0

    for i, row := range mat {
        onesCount := 0

        for _, value := range row {
            onesCount += value
        }

        if onesCount > maxOnes {
            maxOnes = onesCount
            bestRow = i
        }
    }

    return []int{bestRow, maxOnes}
}
```

The Go implementation follows the same logic as the Python version. Since Go does not provide a built-in `sum()` function for slices, we manually iterate through each row and accumulate the count.

Slices are used naturally for the matrix representation. Integer overflow is not a concern because the maximum row length is only `100`.

The implementation also preserves the smallest row index in tie situations by updating the answer only when a strictly larger count is found.

## Edge Cases

### Multiple Rows With the Same Maximum Count

One of the most important edge cases is when several rows contain the same number of ones. A careless implementation might overwrite the answer whenever it sees an equal count, incorrectly returning the last matching row instead of the first.

This implementation avoids that bug by updating the answer only when a row has a strictly larger count. Equal counts are ignored, preserving the earliest row index.

### Matrix Containing Only Zeros

If every value in the matrix is `0`, then every row has zero ones. The correct result should be the first row with count `0`.

Since `best_row` starts at `0` and `max_ones` starts at `0`, no updates occur during traversal, and the algorithm correctly returns `[0,0]`.

### Single Row or Single Column

Very small matrix dimensions often expose indexing mistakes or incorrect assumptions about input size.

For a single-row matrix, the algorithm still works because it processes exactly one row and returns its count.

For a single-column matrix, each row contains either `0` or `1`, and the algorithm still correctly tracks the maximum count and applies tie-breaking rules.
