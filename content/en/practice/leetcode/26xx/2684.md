# LeetCode 2684 - Maximum Number of Moves in a Grid

## Problem Understanding

This problem gives us a 2D grid of positive integers. We may start from any cell in the first column, and from each position we can move only toward the next column. More specifically, from position `(row, col)` we may move to:

- `(row - 1, col + 1)`, the upper-right diagonal
- `(row, col + 1)`, directly right
- `(row + 1, col + 1)`, the lower-right diagonal

However, a move is valid only if the destination cell contains a strictly larger value than the current cell.

The goal is to determine the maximum number of moves possible starting from any cell in the first column.

A move counts as a transition between two cells. If we start at a cell and cannot move anywhere, the answer for that starting point is `0`.

The input consists of:

- `m` rows
- `n` columns
- Positive integer values in every cell

The constraints are important:

- `m, n <= 1000`
- `m * n <= 100000`

This means the total number of cells is at most `100000`, which is large enough that exponential recursion or repeated recomputation will be too slow. We need an efficient dynamic programming solution.

Several edge cases are important:

- A grid where no move is possible from any starting cell
- A grid where every move is possible
- Single valid paths versus multiple branching paths
- Boundary rows, where moving upward or downward may go out of bounds
- Equal neighboring values, since moves require strictly greater values

The problem guarantees all numbers are positive integers, so we never need to handle missing or invalid values.

## Approaches

### Brute Force Approach

A straightforward solution is to try every possible path starting from every row in the first column.

For each starting position, we recursively explore all valid moves:

- upper-right
- right
- lower-right

At every step we branch into all valid next positions. We continue until no valid move exists, then record the number of moves made.

This approach is correct because it explicitly explores every possible path and therefore cannot miss the optimal answer.

The problem is that many subproblems repeat. For example, if multiple paths reach the same cell, we recompute the maximum remaining moves from that cell multiple times. In the worst case this creates exponential behavior.

With up to `100000` cells, brute force recursion is far too slow.

### Optimal Dynamic Programming Approach

The key observation is that the answer for a cell depends only on the cells to its right.

Suppose we define:

`dp[row][col] = maximum moves starting from this cell`

From `(row, col)` we can move only to column `col + 1`, so we can compute answers from right to left.

For each cell, we check up to three possible next positions:

- upper-right
- right
- lower-right

If the destination value is strictly greater, then:

`dp[row][col] = 1 + dp[next_row][col + 1]`

We take the maximum among all valid moves.

Since each cell is processed once and each processing step checks only three neighbors, the solution becomes very efficient.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(3^n) | O(n) recursion stack | Explores all possible paths recursively |
| Optimal Dynamic Programming | O(m × n) | O(m × n) | Computes each cell once using DP |

## Algorithm Walkthrough

1. Create a DP table with the same dimensions as the grid.

Each entry `dp[row][col]` stores the maximum number of moves possible starting from that cell.
2. Initialize the last column.

Any cell in the final column cannot move further because there is no column to the right. Therefore:

```
dp[row][last_column] = 0
```
3. Process columns from right to left.

Since moves only go toward the right, every future state is already computed when we process a cell.
4. For each cell, examine the three possible directions.

The candidate next positions are:

- `(row - 1, col + 1)`
- `(row, col + 1)`
- `(row + 1, col + 1)`
5. Check whether each move is valid.

A move is valid if:

- the next row stays inside the grid
- the destination value is strictly greater
6. Update the DP value.

If a move is valid:

```
dp[row][col] =
    max(dp[row][col],
        1 + dp[next_row][col + 1])
```
7. After filling the table, compute the answer.

Since we may start from any row in the first column, the final result is:

```
max(dp[row][0] for all rows)
```

### Why it works

The algorithm works because every valid move always goes to the next column. This creates a directed acyclic structure, meaning there are no cycles.

When processing a cell, all reachable future states have already been computed. Therefore, the optimal answer for the current cell is simply:

- one move to a valid neighbor
- plus the best possible continuation from that neighbor

Dynamic programming guarantees each subproblem is solved exactly once, and the recurrence correctly captures all valid paths.

## Python Solution

```python
from typing import List

class Solution:
    def maxMoves(self, grid: List[List[int]]) -> int:
        rows = len(grid)
        cols = len(grid[0])

        dp = [[0] * cols for _ in range(rows)]

        for col in range(cols - 2, -1, -1):
            for row in range(rows):
                current_value = grid[row][col]

                for next_row in (row - 1, row, row + 1):
                    if 0 <= next_row < rows:
                        if grid[next_row][col + 1] > current_value:
                            dp[row][col] = max(
                                dp[row][col],
                                1 + dp[next_row][col + 1]
                            )

        answer = 0

        for row in range(rows):
            answer = max(answer, dp[row][0])

        return answer
```

### Implementation Explanation

The solution begins by determining the grid dimensions.

We then create a DP table initialized with zeros. Every cell in the last column naturally remains zero because no moves are possible from there.

The outer loop processes columns from right to left. This order is essential because the value of a cell depends on cells in the next column.

For each cell, we check the three allowed movement directions. Boundary checks ensure we do not access invalid row indices.

Whenever a neighboring cell contains a strictly larger value, we update the current DP value using:

```
1 + dp[next_row][col + 1]
```

This represents making one move plus the best continuation afterward.

Finally, we scan the first column because the starting position may be any row.

## Go Solution

```go
func maxMoves(grid [][]int) int {
    rows := len(grid)
    cols := len(grid[0])

    dp := make([][]int, rows)

    for i := 0; i < rows; i++ {
        dp[i] = make([]int, cols)
    }

    directions := []int{-1, 0, 1}

    for col := cols - 2; col >= 0; col-- {
        for row := 0; row < rows; row++ {
            currentValue := grid[row][col]

            for _, d := range directions {
                nextRow := row + d

                if nextRow >= 0 && nextRow < rows {
                    if grid[nextRow][col+1] > currentValue {
                        candidate := 1 + dp[nextRow][col+1]

                        if candidate > dp[row][col] {
                            dp[row][col] = candidate
                        }
                    }
                }
            }
        }
    }

    answer := 0

    for row := 0; row < rows; row++ {
        if dp[row][0] > answer {
            answer = dp[row][0]
        }
    }

    return answer
}
```

### Go-Specific Notes

The Go implementation follows the same dynamic programming logic as the Python version.

Slices are used for the DP table. Since Go does not provide a built-in `max` function for integers, comparisons are handled manually with `if` statements.

Integer overflow is not a concern because the maximum number of moves is at most `n - 1`, which is at most `999`.

## Edge Cases

### Single Column Grid

If the grid contains only one column, movement is impossible because every move requires advancing to the next column.

This case can easily cause bugs if an implementation assumes at least one move exists. Our DP initialization handles this naturally because every cell in the last column has value `0`, and there are no earlier columns to process.

### Equal Neighbor Values

Moves require the destination value to be strictly greater. Equal values are not valid moves.

A careless implementation might accidentally use `>=` instead of `>`, allowing invalid transitions. Our implementation explicitly checks:

```
grid[next_row][col + 1] > grid[row][col]
```

which correctly enforces the problem rule.

### Boundary Rows

Cells in the top row cannot move upward, and cells in the bottom row cannot move downward.

Without proper boundary checks, the algorithm could access invalid indices and crash. The implementation safely validates:

```
0 <= next_row < rows
```

before attempting any transition.

### Multiple Possible Paths

Some cells may have several valid next moves, and the locally largest value is not always part of the optimal path.

The dynamic programming solution correctly evaluates all valid continuations and stores the maximum possible moves, guaranteeing the globally optimal answer rather than a greedy local choice.
