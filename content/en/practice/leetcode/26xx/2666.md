## Solution

## Problem Understanding

The problem asks us to create a higher-order function. A higher-order function is a function that either takes another function as input or returns a function as output. In this problem, we are given a function `fn`, and we must return a new function that behaves almost exactly like `fn`, with one important restriction: the original function may only be executed once.

The returned function should follow these rules:

- On the very first invocation, it should call `fn` normally and return the result produced by `fn`.
- On every later invocation, it should not call `fn` again. Instead, it should immediately return `undefined`.

The input examples demonstrate this behavior clearly. Suppose we wrap a function that adds numbers together. The first call returns the correct sum because the original function executes. Any later call returns `undefined`, even if different arguments are passed.

The problem is fundamentally about maintaining state across multiple function calls. The returned function must somehow remember whether `fn` has already been executed before.

The constraints are very small. The number of calls is at most 10, and the arguments are limited in size. This means performance is not a concern here. The real focus of the problem is understanding closures and persistent state in JavaScript.

Several edge cases are important to think about:

- The wrapped function may take any number of arguments.
- The wrapped function may return different types of values.
- The wrapped function may return `undefined` naturally on the first call, so we cannot use the return value itself to determine whether the function was already executed.
- The returned wrapper must preserve the argument list exactly when forwarding arguments to `fn`.

The problem guarantees valid input, so we do not need to worry about malformed function definitions or invalid calls.

## Approaches

### Brute Force Approach

A naive approach would be to store all previous calls and check whether the function has already been executed before making another call.

For example, we could maintain a list of invocation records. Every time the wrapper function is called, we check whether the list is empty:

- If it is empty, execute `fn`, store a record, and return the result.
- Otherwise, return `undefined`.

This works correctly because the list acts as evidence that the function has already been called at least once.

However, storing full call history is unnecessary. We only care whether the function has been called before, not how many times or with which arguments. Maintaining an entire list wastes memory and adds unnecessary complexity.

### Optimal Approach

The key observation is that we only need a single boolean variable to track whether the function has already been executed.

We can use a closure variable such as `hasBeenCalled`:

- Initially, it is `False`.
- On the first invocation:

- Execute `fn`
- Set `hasBeenCalled = True`
- Return the result
- On later invocations:

- Return `undefined`

This is the cleanest and most efficient solution because the state persists inside the closure created by the returned function.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(1) | O(n) | Stores unnecessary call history |
| Optimal | O(1) | O(1) | Uses a single boolean flag |

## Algorithm Walkthrough

1. Define a boolean variable named `has_been_called` and initialize it to `False`.
2. Return a new inner function that accepts any number of arguments using variadic parameters.
3. Inside the returned function, first check whether `has_been_called` is already `True`.
4. If the function has already been executed before, immediately return `undefined`.
5. Otherwise, mark the function as called by setting `has_been_called = True`.
6. Invoke the original function `fn` with the provided arguments.
7. Return the result produced by `fn`.

The reason this works is because the inner function forms a closure over `has_been_called`. Even after the outer function finishes execution, the returned function still remembers and updates that variable.

### Why it works

The algorithm maintains a simple invariant:

- Before the first successful execution of `fn`, `has_been_called` is `False`.
- After the first successful execution, `has_been_called` becomes `True` permanently.

Because every invocation checks this flag before calling `fn`, the original function can only execute once.

## Python Solution

```python
from typing import Any, Callable

class Solution:
    def once(self, fn: Callable) -> Callable:
        has_been_called = False

        def wrapper(*args: Any) -> Any:
            nonlocal has_been_called

            if has_been_called:
                return None

            has_been_called = True
            return fn(*args)

        return wrapper
```

The implementation starts by creating a closure variable named `has_been_called`. This variable exists in the outer function scope and persists across calls to the returned wrapper function.

The `wrapper` function accepts arbitrary arguments using `*args`, allowing it to support any function signature.

The `nonlocal` keyword is necessary because we modify the outer variable inside the nested function. Without it, Python would treat `has_been_called` as a new local variable.

The first conditional checks whether the original function has already been executed. If so, the wrapper returns `None`, which corresponds to JavaScript's `undefined` concept in this translated implementation.

If the function has not yet been called, we update the flag and execute `fn` with the supplied arguments.

## Go Solution

```go
package main

func once(fn func(...interface{}) interface{}) func(...interface{}) interface{} {
	hasBeenCalled := false

	return func(args ...interface{}) interface{} {
		if hasBeenCalled {
			return nil
		}

		hasBeenCalled = true
		return fn(args...)
	}
}
```

The Go implementation follows the same closure-based design as the Python version.

Go closures naturally capture variables from the surrounding scope, so `hasBeenCalled` remains available between function calls.

Instead of returning `undefined`, Go returns `nil`, which is the closest equivalent for an absent value.

Go uses `...interface{}` to support arbitrary argument lists and return values.

## Edge Cases

One important edge case is when the wrapped function naturally returns `undefined` or `None` on the first call. A buggy implementation might incorrectly use the return value itself to determine whether the function has already executed. Our implementation avoids this problem entirely because it tracks execution state using a dedicated boolean variable.

Another important case is functions with varying numbers of arguments. Some functions may take zero arguments, while others may take many. A rigid wrapper implementation could fail if it expects a fixed parameter count. Using variadic arguments, `*args` in Python and `...interface{}` in Go, ensures the wrapper transparently forwards all arguments.

A third edge case involves multiple independently wrapped functions. Each call to `once()` should create a completely separate closure state. If the state variable were global instead of local to the closure, different wrapped functions would interfere with one another. Because our implementation stores the flag inside the closure created during each `once()` invocation, every wrapped function maintains independent behavior correctly.
