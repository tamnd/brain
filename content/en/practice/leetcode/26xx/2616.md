## Solution

## Problem Understanding

The problem asks us to select exactly `p` disjoint pairs from the array `nums` such that the largest difference among all chosen pairs is as small as possible.

For any pair of indices `(i, j)`, the pair difference is:

$|nums[i]-nums[j]|$

The goal is not to minimize the sum of differences, nor to minimize each pair independently. Instead, we want to minimize the worst pair difference among the selected pairs.

The array is not initially sorted, and pairs may use any indices as long as no index appears in more than one pair. The answer is the smallest possible value `x` such that we can form at least `p` disjoint pairs where every pair has difference at most `x`.

The constraints are important:

- `nums.length` can be as large as `10^5`
- Values inside `nums` can be as large as `10^9`

These constraints immediately rule out exponential or quadratic pairing strategies. Since the number of possible pairings grows extremely quickly, we need something close to `O(n log n)`.

An important observation is that the answer itself lies within a numeric range. The minimum possible answer is `0`, and the maximum possible answer is:

$\max(nums)-\min(nums)$

This strongly suggests binary search on the answer.

There are several edge cases worth identifying early:

- If `p = 0`, we do not need any pairs, so the answer is `0`.
- Arrays containing duplicate values may allow zero-difference pairs.
- Large gaps between numbers can force the optimal answer upward.
- Greedy pairing choices can accidentally waste indices if implemented incorrectly.
- Because indices cannot overlap, we must carefully ensure disjoint pair construction.

## Approaches

### Brute Force Approach

A brute force solution would attempt to generate all possible ways to choose `p` disjoint pairs and compute the maximum pair difference for each configuration.

For every valid pairing configuration:

1. Compute the difference for every pair.
2. Take the maximum difference within that configuration.
3. Track the minimum among all configurations.

This works because it explicitly checks every possible answer.

However, the number of ways to form disjoint pairs grows combinatorially. Even for relatively small arrays, the total number of pair combinations becomes enormous. With `n = 10^5`, this approach is completely infeasible.

The brute force solution is useful conceptually because it clarifies what we are optimizing, but it cannot satisfy the constraints.

### Optimal Approach

The key insight is that instead of directly constructing the optimal pairing, we can ask a simpler decision question:

> Can we form at least `p` disjoint pairs such that every pair difference is at most `x`?

This transforms the problem into a binary search problem.

After sorting the array, nearby values have the smallest differences. If a valid solution exists for some threshold `x`, then any larger threshold is also valid.

This creates a monotonic property:

- Small `x` values may fail.
- Once a value succeeds, all larger values succeed too.

That monotonicity allows binary search over the answer space.

The second insight is that after sorting, the optimal way to maximize the number of valid pairs for a given threshold is greedy:

- Scan left to right.
- If adjacent numbers differ by at most `x`, pair them immediately.
- Otherwise skip the current number.

Pairing adjacent elements is optimal because adjacent sorted values produce the smallest possible differences.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Tries all possible pair combinations |
| Optimal | O(n log n + n log M) | O(1) or O(log n) depending on sort | Binary search with greedy feasibility check |

Here, `M` is the numeric range of possible answers.

## Algorithm Walkthrough

1. Sort the array.

Sorting is crucial because nearby elements in sorted order produce the smallest absolute differences. Once sorted, we can greedily pair adjacent elements.
2. Define the binary search range.

The smallest possible answer is `0`.

The largest possible answer is:

$\max(nums)-\min(nums)$
3. Create a helper function `can_form_pairs(limit)`.

This function determines whether we can form at least `p` disjoint pairs where every pair difference is at most `limit`.
4. Inside the helper function, greedily scan the sorted array.

Use an index `i` starting from `0`.

At each step:

- If `nums[i + 1] - nums[i] <= limit`, form a pair and increment the pair count.
- Skip both elements because indices cannot overlap.
- Otherwise skip only the current element.
5. Return whether the pair count reached `p`.

If yes, the threshold is feasible.
6. Perform binary search.

For each midpoint `mid`:

- If `can_form_pairs(mid)` is true, try smaller answers.
- Otherwise try larger answers.
7. Continue until the search converges.

The final left boundary is the minimum achievable maximum difference.

### Why it works

The correctness depends on two properties.

First, after sorting, pairing adjacent elements is always optimal for maximizing the number of pairs under a threshold. Any non-adjacent pairing would produce an equal or larger difference and could only reduce future pairing opportunities.

Second, feasibility is monotonic. If we can form `p` pairs using threshold `x`, then we can also form them using any threshold larger than `x`. This monotonicity guarantees binary search correctness.

## Python Solution

```python
from typing import List

class Solution:
    def minimizeMax(self, nums: List[int], p: int) -> int:
        if p == 0:
            return 0

        nums.sort()

        def can_form_pairs(limit: int) -> bool:
            pair_count = 0
            index = 0

            while index < len(nums) - 1:
                if nums[index + 1] - nums[index] <= limit:
                    pair_count += 1
                    index += 2
                else:
                    index += 1

                if pair_count >= p:
                    return True

            return False

        left = 0
        right = nums[-1] - nums[0]

        while left < right:
            mid = (left + right) // 2

            if can_form_pairs(mid):
                right = mid
            else:
                left = mid + 1

        return left
```

The implementation begins by handling the simplest edge case. If `p` is zero, no pairs are needed, so the answer is immediately `0`.

Next, the array is sorted. This enables the greedy pairing strategy because adjacent elements now represent the smallest available differences.

The helper function `can_form_pairs` performs the feasibility check. It greedily scans the array and pairs adjacent elements whenever their difference is within the given limit.

If a pair is formed, both elements are skipped because indices cannot be reused. Otherwise, only the current element is skipped.

The binary search repeatedly tests candidate maximum differences. Whenever a candidate succeeds, the algorithm searches for a smaller valid answer. Otherwise, it searches for a larger threshold.

Eventually, the search converges to the minimum feasible maximum difference.

## Go Solution

```go
package main

import (
	"sort"
)

func minimizeMax(nums []int, p int) int {
	if p == 0 {
		return 0
	}

	sort.Ints(nums)

	canFormPairs := func(limit int) bool {
		pairCount := 0
		index := 0

		for index < len(nums)-1 {
			if nums[index+1]-nums[index] <= limit {
				pairCount++
				index += 2
			} else {
				index++
			}

			if pairCount >= p {
				return true
			}
		}

		return false
	}

	left := 0
	right := nums[len(nums)-1] - nums[0]

	for left < right {
		mid := left + (right-left)/2

		if canFormPairs(mid) {
			right = mid
		} else {
			left = mid + 1
		}
	}

	return left
}
```

The Go implementation closely mirrors the Python solution.

The primary difference is the use of `sort.Ints(nums)` for sorting. The binary search midpoint is computed as:

```go
mid := left + (right-left)/2
```

This avoids potential integer overflow, although the constraints here would still fit safely inside Go's integer range.

Go slices are naturally dynamic references to arrays, so no additional copying is necessary.

| Space | O(1) or O(log n) | Extra algorithmic space is constant, sorting may use stack space |

The sorting step costs `O(n log n)`.

The binary search runs over the numeric answer range. Each feasibility check scans the array once, costing `O(n)`.

Since the answer range is at most `10^9`, the binary search performs roughly `log2(10^9)` iterations, which is about 31. Therefore the total complexity is:

```
O(n log n + n log M)
```

where `M` is the maximum numeric difference.

## Edge Cases

One important edge case occurs when `p = 0`. Since no pairs are required, the maximum difference over an empty set is defined as zero. A careless implementation might still perform binary search unnecessarily or even fail because no pairs are formed. The implementation handles this immediately with an early return.

Another important case involves duplicate numbers. Arrays like `[5,5,5,5]` allow zero-difference pairs. Some implementations incorrectly assume the minimum answer must be positive whenever pairs exist. Because the algorithm explicitly checks adjacent sorted differences, it correctly detects zero as a valid answer.

A third tricky case occurs when the array length is odd. Since one element may remain unused, the greedy scan must correctly skip elements without accidentally reusing indices. The implementation carefully advances the pointer by two only when a pair is formed, otherwise by one. This guarantees all chosen pairs remain disjoint.

A fourth subtle case involves large gaps between numbers. Consider `[1,2,100,101]` with `p = 2`. Pairing adjacent elements gives differences `1` and `1`, while careless non-adjacent pairing could produce huge differences unnecessarily. Sorting combined with greedy adjacent pairing ensures the smallest possible maximum difference is always found.
