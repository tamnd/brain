## Solution

## Problem Understanding

The problem gives us a 2D matrix `nums`, where each row contains integers. We repeatedly perform operations until every element has been removed from the matrix.

In each operation, we do two things:

1. From every row, we remove the largest remaining value.
2. Among all removed values, we find the maximum and add it to our score.

The process continues column by column until all elements are removed.

The important detail is that every row loses exactly one element during each operation. Since all rows have the same number of columns in the examples and constraints, after enough operations the matrix becomes empty.

The task is to compute the final accumulated score.

To understand the process more clearly, imagine each row being sorted from smallest to largest. During each round, we remove the current largest element from every row. The score contribution for that round becomes the largest among those removed elements.

The constraints are relatively small:

- Up to `300` rows
- Up to `500` columns
- Values up to `1000`

This tells us that sorting each row independently is completely feasible. Even an `O(m * n log n)` solution works comfortably, where:

- `m` is the number of rows
- `n` is the number of columns

Some important edge cases include matrices with only one element, rows containing duplicate values, and rows where all numbers are identical. The problem guarantees valid integer inputs, so we do not need to handle malformed matrices or empty rows.

## Approaches

### Brute Force Approach

A direct simulation approach would repeatedly scan every row to locate the current maximum element, remove it, then compute the maximum among all removed values.

For each operation:

- Traverse each row to find its largest remaining element.
- Remove that element.
- Track the largest removed value globally.
- Add it to the answer.

This produces the correct result because it exactly follows the problem statement.

However, removal from arbitrary positions inside arrays is expensive. Additionally, repeatedly searching for maximum values in every row creates unnecessary repeated work.

If each row contains `n` elements, and we perform `n` rounds, then each round may require scanning up to `n` elements per row. This leads to roughly `O(m * n^2)` time complexity.

Although this may still pass under current constraints, it is inefficient and misses an important observation.

### Key Insight

The order in which elements are removed from a row is completely determined by sorting.

If we sort each row in ascending order:

- The last element is removed first.
- The second last element is removed second.
- And so on.

Instead of simulating removals, we can think column-wise after sorting.

For example:

```
Original row:
[7,2,1]

Sorted row:
[1,2,7]
```

The removal order becomes:

```
7 -> 2 -> 1
```

Now consider all rows sorted. During each round, elements removed from rows correspond to the same column index when traversing from right to left.

That means:

- Sort every row.
- For each column index:

- Find the maximum value among all rows at that index.
- Add it to the answer.

This avoids repeated searches and removals entirely.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m * n²) | O(1) or O(n) | Repeatedly searches and removes maximum elements |
| Optimal | O(m * n log n) | O(1) extra, ignoring sort space | Sort rows once, then process column-wise |

## Algorithm Walkthrough

1. Sort every row in ascending order.

Sorting transforms the repeated removal process into a predictable structure. After sorting, the largest element of a row is always at the end, the second largest is one position before it, and so on.
2. Initialize a variable `score = 0`.

This variable stores the total score accumulated across all operations.
3. Iterate through each column index from left to right.

Since rows are sorted ascending, elements at the same column index represent values removed during the same operation when viewed from largest to smallest.
4. For the current column, compute the maximum value across all rows.

This simulates the rule where, after removing one element from each row, we select the largest among those removed values.
5. Add this maximum value to `score`.

Each column contributes exactly one value to the final answer.
6. After processing all columns, return `score`.

### Why it works

Sorting each row preserves the relative removal order of elements. The largest values in every row are aligned in the final columns after sorting. During each operation, the removed elements correspond exactly to one column across all rows. Taking the maximum value from that column therefore matches the problem's scoring rule. Since every operation is represented once and only once, the computed score is correct.

## Python Solution

```python
from typing import List

class Solution:
    def matrixSum(self, nums: List[List[int]]) -> int:
        # Sort every row
        for row in nums:
            row.sort()

        rows = len(nums)
        cols = len(nums[0])

        score = 0

        # Process column by column
        for col in range(cols):
            current_max = 0

            for row in range(rows):
                current_max = max(current_max, nums[row][col])

            score += current_max

        return score
```

The implementation begins by sorting every row in ascending order. This preprocessing step converts the repeated removal simulation into a simple positional comparison problem.

Next, we determine the number of rows and columns. The algorithm then iterates through each column index.

For every column, we scan all rows and compute the maximum value appearing in that column. This corresponds to the largest value removed during that operation.

Finally, we add the column maximum to the running score and return the total result after all columns are processed.

## Go Solution

```go
package main

import "sort"

func matrixSum(nums [][]int) int {
	for _, row := range nums {
		sort.Ints(row)
	}

	rows := len(nums)
	cols := len(nums[0])

	score := 0

	for col := 0; col < cols; col++ {
		currentMax := 0

		for row := 0; row < rows; row++ {
			if nums[row][col] > currentMax {
				currentMax = nums[row][col]
			}
		}

		score += currentMax
	}

	return score
}
```

The Go implementation follows the same logic as the Python version.

The main difference is that Go uses `sort.Ints()` for sorting slices in place. Since Go does not provide a built in `max()` function for integers, we manually compare values while scanning each column.

There are no integer overflow concerns because the maximum possible score is well within Go's `int` range under the given constraints.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m * n log n) | Each row is sorted independently |
| Space | O(1) extra | Sorting is done in place, excluding sorting internals |

The dominant cost comes from sorting each row. If there are `m` rows and each row contains `n` elements, sorting one row costs `O(n log n)`, so sorting all rows costs `O(m * n log n)`.

The column scanning phase only requires `O(m * n)` time, which is smaller than the sorting cost.

No additional data structures proportional to input size are required.

## Edge Cases

One important edge case is a matrix containing only a single element. In this scenario, there is only one operation, and that element must be returned directly. The implementation handles this naturally because sorting a single value changes nothing, and the column iteration runs exactly once.

Another important case is when rows contain duplicate values. A naive implementation might incorrectly assume removals must be unique or depend on element positions. The problem explicitly states that ties do not matter, so sorting works perfectly even when many identical values exist. The algorithm simply compares column values without caring which duplicate instance was removed.

A third edge case is a matrix with only one row. In this situation, every removed value is automatically the maximum among removed elements because there are no competing rows. The algorithm still works because each column maximum becomes the single value in that row.

A fourth subtle case is when all values are zero. Some implementations incorrectly initialize maxima using negative infinity or fail to accumulate properly. Here, `current_max` starts at `0`, which is safe because all inputs are guaranteed to be non negative.

Finally, matrices with already sorted or reverse sorted rows are also important. Since the algorithm explicitly sorts every row before processing, the original ordering does not matter and cannot affect correctness.
