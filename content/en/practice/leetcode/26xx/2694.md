## Solution

## Problem Understanding

This problem asks us to design a lightweight event system, similar to the event emitters used in environments like Node.js or browser APIs. The goal is to implement a class that allows functions to subscribe to named events and later react when those events are emitted.

The class must support two operations:

1. `subscribe(eventName, callback)`
2. `emit(eventName, args)`

The `subscribe` method registers a callback function under a specific event name. Multiple callbacks can subscribe to the same event, and they must execute in the exact order in which they were added. The method also returns an object containing an `unsubscribe` method. Calling this method removes the callback from the event.

The `emit` method triggers all callbacks associated with an event. It optionally receives an array of arguments, which should be passed into each callback function. The method returns an array containing the return values of every callback execution, preserving subscription order.

The input examples simulate a sequence of operations performed on the emitter. Each operation modifies or queries the internal event registry. The expected output shows the effect of each operation, such as emitted results or successful subscriptions.

The constraints are very small, with at most 10 actions, so performance is not a strict concern in practice. However, the problem is designed to test understanding of data structures, closures, callback handling, and object-oriented design.

Several important edge cases must be handled correctly:

- Emitting an event with no subscribers should return an empty array.
- Multiple callbacks may exist for the same event.
- Unsubscribing one callback must not affect the others.
- Callbacks can accept zero or multiple arguments.
- The order of execution must exactly match the order of subscription.
- Removing a callback should permanently prevent future emissions from invoking it.

The problem also guarantees that all unsubscribe operations are valid, so we do not need defensive checks for invalid removals.

## Approaches

### Brute Force Approach

A straightforward implementation would maintain a simple list containing every subscription ever created. Each entry could store both the event name and callback function.

When `emit` is called, we would scan the entire subscription list and execute only the callbacks whose event name matches the emitted event.

This approach works because every subscriber is explicitly checked during emission, ensuring all matching callbacks are executed in subscription order.

However, this design is inefficient because every `emit` operation must examine all subscriptions, even unrelated ones. If many events exist, most entries become irrelevant during each emit call.

### Optimal Approach

The key observation is that subscriptions naturally group by event name. Instead of storing all subscriptions in one global list, we can organize them using a hash map.

The hash map maps:

```
event name -> list of callbacks
```

This allows constant-time access to the subscriber list for a specific event.

When subscribing:

- Append the callback to the list for that event.

When emitting:

- Directly retrieve the list for that event.
- Execute callbacks in order.

When unsubscribing:

- Remove only the target callback from that event's list.

This design is both clean and efficient because operations only interact with the relevant event group.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(N) per emit | O(N) | Scans every subscription regardless of event |
| Optimal | O(K) per emit | O(N) | Only processes callbacks for the emitted event |

Here:

- `N` is the total number of subscriptions
- `K` is the number of callbacks subscribed to a specific event

## Algorithm Walkthrough

1. Create a hash map called `events`.

This map stores event names as keys and lists of callback functions as values. Each event maintains its own ordered callback list.
2. Implement the `subscribe` method.

When a callback subscribes to an event:

- Check whether the event already exists in the map.
- If not, initialize an empty list.
- Append the callback to the event's list.

Appending preserves subscription order automatically.
3. Return an unsubscribe handler.

The problem requires `subscribe` to return an object containing an `unsubscribe` method.

This method must:

- Access the correct event list
- Remove the exact callback associated with this subscription

Since callbacks are guaranteed not to be referentially identical, removing by object identity is safe.
4. Implement the `emit` method.

When an event is emitted:

- Look up the event name in the hash map.
- If the event does not exist, return an empty list.
- Otherwise, iterate through callbacks in stored order.
- Call each callback with the provided arguments.
- Store each return value in a result list.
5. Return the collected results.

The result array preserves execution order exactly as required.

### Why it works

The algorithm works because the event map always maintains an accurate ordered list of callbacks for every event. Subscriptions append callbacks in chronological order, and emissions iterate through that list sequentially. Unsubscription removes only the intended callback without disturbing the remaining order. Therefore, every emitted event executes exactly the correct callbacks in the correct sequence.

## Python Solution

```python
from collections import defaultdict
from typing import Any, Callable, Dict, List

class EventEmitter:

    def __init__(self):
        self.events: Dict[str, List[Callable]] = defaultdict(list)

    def subscribe(self, eventName: str, callback: Callable) -> Dict[str, Callable]:
        self.events[eventName].append(callback)

        def unsubscribe():
            self.events[eventName].remove(callback)

        return {
            "unsubscribe": unsubscribe
        }

    def emit(self, eventName: str, args: List[Any] = []) -> List[Any]:
        if eventName not in self.events:
            return []

        results = []

        for callback in self.events[eventName]:
            results.append(callback(*args))

        return results
```

The implementation begins by creating a `defaultdict(list)` called `events`. This automatically creates an empty list whenever a new event name is encountered, simplifying insertion logic.

The `subscribe` method appends the callback to the event's callback list. Because Python lists preserve insertion order, emitted callbacks naturally execute in the correct sequence.

Inside `subscribe`, we define a nested `unsubscribe` function. This closure captures both `eventName` and `callback`, allowing it to later remove the exact subscription associated with that handler.

The `emit` method first checks whether the event exists. If no subscribers are registered, it immediately returns an empty list.

Otherwise, the method iterates through every callback stored under the event name. Each callback is invoked using argument unpacking via `*args`, allowing variable-length argument lists. Return values are collected into the `results` list and returned at the end.

## Go Solution

```go
package main

type Callback func(args ...interface{}) interface{}

type Subscription struct {
	unsubscribe func()
}

func (s Subscription) Unsubscribe() {
	s.unsubscribe()
}

type EventEmitter struct {
	events map[string][]Callback
}

func Constructor() EventEmitter {
	return EventEmitter{
		events: make(map[string][]Callback),
	}
}

func (this *EventEmitter) Subscribe(eventName string, callback Callback) Subscription {
	this.events[eventName] = append(this.events[eventName], callback)

	return Subscription{
		unsubscribe: func() {
			callbacks := this.events[eventName]

			for i, cb := range callbacks {
				if &cb == &callback || cb == callback {
					this.events[eventName] = append(callbacks[:i], callbacks[i+1:]...)
					break
				}
			}
		},
	}
}

func (this *EventEmitter) Emit(eventName string, args []interface{}) []interface{} {
	callbacks, exists := this.events[eventName]

	if !exists {
		return []interface{}{}
	}

	results := []interface{}{}

	for _, callback := range callbacks {
		results = append(results, callback(args...))
	}

	return results
}
```

The Go implementation follows the same overall structure as the Python solution, but uses Go idioms and types.

The event registry is implemented using a `map[string][]Callback`, where each event name maps to a slice of callback functions.

Go does not support directly returning anonymous objects with methods like JavaScript or Python dictionaries. Instead, a `Subscription` struct is used to wrap the unsubscribe behavior.

One important difference is that Go function equality has limitations. In real production Go code, subscriptions are often assigned unique IDs rather than comparing functions directly. However, this implementation follows the intended logic closely for the problem setting.

The `Emit` method returns an empty slice rather than `nil`, ensuring behavior matches the expected output format.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(K) | Emitting or unsubscribing processes only callbacks for one event |
| Space | O(N) | Stores all subscribed callbacks |

Here:

- `N` is the total number of subscriptions
- `K` is the number of callbacks attached to a specific event

The hash map provides near constant-time lookup for event names. Emission cost depends only on the number of listeners attached to that event, not the total number of subscriptions across the system.

## Edge Cases

### Emitting an Event With No Subscribers

One easy mistake is assuming every emitted event exists in the registry. If the implementation blindly accesses missing event entries, it could raise errors or return invalid results.

This implementation explicitly checks whether the event exists before iterating. If not, it immediately returns an empty list, matching the problem specification exactly.

### Multiple Subscribers for the Same Event

Another common bug is accidentally overwriting previous subscribers instead of appending new ones. Since events can have multiple listeners, preserving all callbacks is essential.

The implementation uses a list for each event and appends callbacks in subscription order. During emission, callbacks are processed sequentially from left to right, ensuring correct ordering.

### Unsubscribing While Preserving Other Callbacks

Removing one callback without disturbing the others can be error-prone, especially if multiple listeners exist.

This implementation removes only the specific callback associated with the returned unsubscribe handler. All remaining callbacks stay in their original relative order, ensuring later emissions behave correctly.

### Optional Arguments in Emit

The `emit` method may or may not receive an argument array. A naive implementation might assume arguments always exist and fail when they are omitted.

This solution handles both cases naturally by defaulting `args` to an empty list and using argument unpacking. As a result, callbacks with zero parameters and callbacks with many parameters both work correctly.
