## Solution

## Problem Understanding

This problem asks us to build a higher order asynchronous function. We are given an async function `fn` and a time limit `t` in milliseconds. Our goal is to return a new function that behaves exactly like `fn`, except that it enforces the specified timeout.

The returned function receives arbitrary arguments and forwards them to `fn`. The important behavior is how the wrapper handles execution time:

- If `fn` finishes before the timeout expires, the wrapper should resolve with the same result.
- If `fn` takes longer than `t` milliseconds, the wrapper should reject with the string `"Time Limit Exceeded"`.

The key detail is that `fn` itself already returns a promise. This means our wrapper must also work asynchronously and coordinate between two competing events:

1. The original function finishing
2. The timeout expiring

Whichever happens first determines the outcome.

The input consists of:

- `fn`, an asynchronous function
- `t`, the maximum allowed execution time in milliseconds
- arbitrary arguments passed later into the returned function

The output is another asynchronous function that enforces the timeout rule.

The constraints are small:

- At most 10 input arguments
- Timeout is between `0` and `1000`
- `fn` always returns a promise

These constraints tell us that performance is not really about processing large data. Instead, this is a problem about correctly handling asynchronous control flow.

Several edge cases are important:

- `fn` may reject immediately
- `fn` may resolve immediately
- `t` may be `0`
- `fn` may never resolve
- `fn` may resolve exactly at the timeout boundary

A naive implementation can easily introduce race conditions or fail to reject properly when the timeout expires first.

## Approaches

### Brute Force Approach

A straightforward idea is to execute `fn`, then manually track elapsed time in a loop until either:

- the promise resolves, or
- enough time has passed

For example, we could repeatedly check the current time using `Date.now()` or `performance.now()` while polling the promise state.

This approach is technically possible but highly inefficient and impractical. JavaScript promises are event driven, not designed for busy waiting. Polling wastes CPU cycles and complicates synchronization logic.

Additionally, JavaScript does not expose direct promise state inspection in standard APIs, so implementing manual polling becomes awkward and unreliable.

### Optimal Approach

The key observation is that this problem is naturally modeled as a race between two promises:

- the original function promise
- a timeout promise

JavaScript already provides exactly the tool we need: `Promise.race()`.

`Promise.race()` resolves or rejects as soon as the first promise settles. This perfectly matches the problem statement.

We create:

1. A promise from `fn(...args)`
2. A timeout promise that rejects after `t` milliseconds

Then we race them together.

If `fn` finishes first, its result becomes the final result.

If the timer finishes first, the wrapper rejects with `"Time Limit Exceeded"`.

This solution is concise, efficient, and directly aligned with JavaScript's asynchronous model.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(t) | O(1) | Continuously polls time and promise state |
| Optimal | O(1) | O(1) | Uses `Promise.race()` to efficiently coordinate timeout behavior |

## Algorithm Walkthrough

1. Create and return a wrapper function.

The problem requires returning a new time limited version of `fn`. This wrapper accepts arbitrary arguments and forwards them to the original function.
2. Inside the wrapper, create the original function promise.

Call `fn(...args)` and store the resulting promise. This represents the actual asynchronous work.
3. Create a timeout promise.

Use `setTimeout()` to schedule a rejection after `t` milliseconds. The rejection value must be exactly `"Time Limit Exceeded"`.
4. Use `Promise.race()`.

Pass both promises into `Promise.race()`.

The first promise to settle determines the final outcome:

- If `fn` resolves first, the wrapper resolves.
- If `fn` rejects first, the wrapper rejects with the same error.
- If the timeout fires first, the wrapper rejects with `"Time Limit Exceeded"`.
5. Return the raced promise.

Since the wrapper itself is asynchronous, it returns the result of `Promise.race()` directly.

### Why it works

`Promise.race()` guarantees that the earliest settled promise determines the final result. Since the timeout promise rejects exactly after `t` milliseconds, and the function promise settles whenever `fn` completes, the wrapper always behaves according to the problem requirements.

The correctness relies on the invariant that only the earliest completed asynchronous event matters.

## Python Solution

LeetCode provides this problem in JavaScript, but below is an equivalent Python implementation using `asyncio`.

```python
from typing import Any, Callable
import asyncio

class Solution:
    def timeLimit(self, fn: Callable, t: int) -> Callable:

        async def limited_function(*args) -> Any:
            try:
                return await asyncio.wait_for(fn(*args), timeout=t / 1000)
            except asyncio.TimeoutError:
                raise Exception("Time Limit Exceeded")

        return limited_function
```

The implementation uses Python's `asyncio.wait_for()`, which is the direct equivalent of JavaScript timeout enforcement.

The `timeLimit` method returns a new async function named `limited_function`.

Inside `limited_function`, we call the original async function `fn(*args)` and wrap it with `asyncio.wait_for()`.

The timeout must be converted from milliseconds to seconds because Python's asyncio APIs use seconds.

If the function finishes in time, `wait_for()` returns the result normally.

If the timeout expires first, Python raises `asyncio.TimeoutError`. We catch that exception and replace it with `"Time Limit Exceeded"` to match the problem statement.

## Go Solution

Go does not use promises directly, but we can model the same behavior using goroutines, channels, and `time.After`.

```go
package main

import (
	"errors"
	"time"
)

type AsyncFunc func(args ...interface{}) (interface{}, error)

func timeLimit(fn AsyncFunc, t int) AsyncFunc {
	return func(args ...interface{}) (interface{}, error) {

		resultChan := make(chan interface{})
		errorChan := make(chan error)

		go func() {
			result, err := fn(args...)

			if err != nil {
				errorChan <- err
				return
			}

			resultChan <- result
		}()

		select {
		case result := <-resultChan:
			return result, nil

		case err := <-errorChan:
			return nil, err

		case <-time.After(time.Duration(t) * time.Millisecond):
			return nil, errors.New("Time Limit Exceeded")
		}
	}
}
```

The Go implementation uses goroutines instead of promises.

We launch the original function in a separate goroutine so it can execute concurrently.

Two channels are used:

- `resultChan` for successful completion
- `errorChan` for failures

The `select` statement behaves similarly to `Promise.race()` because it waits for whichever channel operation completes first.

`time.After()` creates a timeout channel that becomes ready after `t` milliseconds.

Unlike Python or JavaScript, Go requires explicit concurrency primitives instead of built in promise utilities.

## Edge Cases

### Zero Timeout

When `t = 0`, the timeout should occur immediately unless the function resolves synchronously before the timer check. This case is easy to mishandle because developers may assume every function gets at least some execution time.

The implementation handles this correctly because the timeout promise is scheduled immediately. If the function cannot settle before the timeout event, the wrapper rejects with `"Time Limit Exceeded"`.

### Function Throws Immediately

An async function may reject instantly without waiting. A naive implementation could accidentally overwrite this rejection with the timeout error.

Using `Promise.race()` or equivalent concurrency primitives guarantees the earliest settled result wins. Therefore immediate errors propagate correctly.

### Function Never Resolves

Some asynchronous operations may hang forever because of network failures or unresolved promises.

Without timeout protection, the caller would wait indefinitely. The timeout promise guarantees that the wrapper eventually rejects after `t` milliseconds, preventing permanent blocking.
