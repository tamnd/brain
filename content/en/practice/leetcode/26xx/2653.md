## Solution

## Problem Understanding

The problem asks us to process every contiguous subarray of length `k` and compute a value called the "beauty" of that subarray.

For each window of size `k`, we look at all the numbers inside it and sort them conceptually in ascending order. We then identify the `x`th smallest integer. However, the beauty is only meaningful if that `x`th smallest integer is negative. If there are fewer than `x` negative numbers in the window, the beauty is defined as `0`.

The input consists of:

- `nums`, an integer array of length `n`
- `k`, the fixed size of every sliding window
- `x`, the rank of the negative number we care about

The output should contain exactly `n - k + 1` integers because that is the number of contiguous subarrays of length `k`.

The constraints are extremely important:

- `n` can be as large as `10^5`
- Values in `nums` are limited to the range `[-50, 50]`

The large value of `n` tells us that repeatedly sorting every window would be too slow. However, the tiny numeric range is a major clue. Since there are only 101 possible values, we can use frequency counting instead of sorting entire windows repeatedly.

An important observation is that only negative numbers matter for the answer. Positive numbers and zero never contribute directly to the beauty.

Several edge cases are worth identifying immediately:

- A window may contain no negative numbers at all
- A window may contain fewer than `x` negative numbers
- Multiple identical negative numbers may appear
- The `x`th smallest negative number might be repeated many times
- `k` may equal `n`, meaning there is only one window
- Every value may already be negative, which stresses the ranking logic

## Approaches

### Brute Force Approach

The most straightforward solution is to examine every window independently.

For each subarray of length `k`, we can:

1. Extract the subarray
2. Sort it
3. Find the `x`th smallest negative number
4. Return `0` if there are fewer than `x` negative values

This works because sorting gives us the elements in ascending order, making it easy to identify the required rank.

However, this becomes too slow for large inputs. There are `O(n)` windows, and sorting each window costs `O(k log k)`. In the worst case, `k` may be close to `n`, making the total complexity approximately `O(n^2 log n)`.

With `n = 10^5`, this is not practical.

### Optimal Sliding Window with Frequency Counting

The key observation is that the values are restricted to `[-50, 50]`.

Instead of sorting each window, we can maintain a frequency array for the current sliding window. Since we only care about negative numbers, we only need counts for `[-50, -1]`.

As the window slides:

- Remove the outgoing element from the frequency table
- Add the incoming element
- Scan the negative values from smallest to largest
- Find the `x`th negative number using cumulative frequencies

Because there are only 50 negative values, each scan takes constant time.

This transforms the problem into an efficient sliding window solution.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × k log k) | O(k) | Sort every window independently |
| Optimal | O(n) | O(1) | Sliding window with fixed-size frequency counting |

## Algorithm Walkthrough

1. Create a frequency array large enough to count numbers from `-50` to `50`.

Since array indices cannot be negative, we shift values by `+50`. For example:

- `-50` maps to index `0`
- `0` maps to index `50`
- `50` maps to index `100`
2. Build the initial window of size `k`.

Iterate through the first `k` elements and increment their frequencies.
3. Compute the beauty for the current window.

To find the `x`th smallest negative number, scan values from `-50` up to `-1`.

Maintain a running total of how many negative numbers have been seen so far.

Once the cumulative count reaches or exceeds `x`, that current number is the answer.

If the scan finishes without reaching `x`, return `0`.
4. Slide the window one step to the right.

Remove the leftmost outgoing value from the frequency array.

Add the new incoming value.
5. Recompute the beauty for the updated window.

Because there are only 50 negative numbers to inspect, this remains constant time.
6. Repeat until all windows are processed.

### Why it works

The frequency array always represents the exact contents of the current window. By scanning negative values in ascending order and accumulating frequencies, we effectively simulate a sorted order without explicitly sorting.

The first value whose cumulative count reaches `x` must be the `x`th smallest negative integer in the window. If the cumulative count never reaches `x`, then the window contains fewer than `x` negative numbers, so the beauty is correctly `0`.

## Python Solution

```python
from typing import List

class Solution:
    def getSubarrayBeauty(self, nums: List[int], k: int, x: int) -> List[int]:
        frequency = [0] * 101
        
        def get_beauty() -> int:
            negative_count = 0
            
            for value in range(-50, 0):
                negative_count += frequency[value + 50]
                
                if negative_count >= x:
                    return value
            
            return 0
        
        # Build first window
        for i in range(k):
            frequency[nums[i] + 50] += 1
        
        result = [get_beauty()]
        
        # Slide the window
        for right in range(k, len(nums)):
            left = right - k
            
            frequency[nums[left] + 50] -= 1
            frequency[nums[right] + 50] += 1
            
            result.append(get_beauty())
        
        return result
```

The implementation begins by creating a frequency array of size 101 to represent all possible values from `-50` to `50`.

The helper function `get_beauty()` scans only the negative values. It keeps a cumulative count of how many negative numbers have appeared so far. As soon as this count reaches `x`, the current value is the `x`th smallest negative number.

The first window is built explicitly before the sliding process begins.

After that, the window slides one step at a time:

- The outgoing value is removed from the frequency table
- The incoming value is added
- The beauty of the updated window is computed

Because the scan checks at most 50 negative values, each beauty computation is constant time.

## Go Solution

```go
package main

func getSubarrayBeauty(nums []int, k int, x int) []int {
	frequency := make([]int, 101)

	getBeauty := func() int {
		negativeCount := 0

		for value := -50; value < 0; value++ {
			negativeCount += frequency[value+50]

			if negativeCount >= x {
				return value
			}
		}

		return 0
	}

	// Build first window
	for i := 0; i < k; i++ {
		frequency[nums[i]+50]++
	}

	result := []int{getBeauty()}

	// Slide window
	for right := k; right < len(nums); right++ {
		left := right - k

		frequency[nums[left]+50]--
		frequency[nums[right]+50]++

		result = append(result, getBeauty())
	}

	return result
}
```

The Go implementation follows exactly the same logic as the Python version.

One small difference is that Go uses slices instead of Python lists. The frequency array is created with `make([]int, 101)`.

Go also uses explicit integer arithmetic and does not support nested named functions in quite the same way as Python, so the helper is implemented as an anonymous function assigned to `getBeauty`.

No integer overflow concerns exist because all counts are bounded by `k`, which is at most `10^5`.

## Edge Cases

One important edge case occurs when a window contains no negative numbers. A naive implementation might incorrectly return the smallest element overall, even if it is non-negative. The problem explicitly requires returning `0` in this situation. The frequency scan correctly handles this because the cumulative negative count never reaches `x`.

Another tricky case occurs when there are fewer than `x` negative numbers in the window. For example, if `x = 3` but the window contains only two negatives, the algorithm must again return `0`. The implementation naturally handles this because the scan finishes without satisfying the required count.

Duplicate negative values are another common source of bugs. Consider a window like `[-2, -2, -2]`. The 2nd smallest negative number is still `-2`. Because the algorithm uses frequency counting instead of distinct values, duplicates are counted correctly in the cumulative total.

A final important edge case is when `k == n`. In this situation there is only one window covering the entire array. Some sliding window implementations accidentally assume at least one slide operation occurs. This implementation handles the case correctly because it computes the first window result before entering the sliding loop.
