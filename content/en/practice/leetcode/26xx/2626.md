## Solution

## Problem Understanding

The problem is asking us to simulate the behavior of a "reduce" function for arrays without using the built-in `Array.reduce` method. In essence, we are given an array of integers `nums`, a binary function `fn`, and an initial value `init`. We need to sequentially apply the function `fn` to each element of the array and the accumulated result, starting with `init`.

For example, if `nums = [1,2,3]`, `fn` is a sum function, and `init = 0`, we compute the accumulated value as follows: first `val = fn(init, nums[0])`, which gives `val = 0 + 1 = 1`, then `val = fn(val, nums[1])`, giving `val = 1 + 2 = 3`, and finally `val = fn(val, nums[2])`, giving `val = 3 + 3 = 6`. This final value `6` is the result.

The input guarantees that `nums.length` is at most 1000, and each element, as well as `init`, is between 0 and 1000. This implies the algorithm can be straightforward and does not need optimization for extremely large input sizes. Important edge cases include an empty array, in which case the function should return `init` immediately, and cases where `fn` may produce large numbers (though within the constraints, overflow is not a concern in most languages).

## Approaches

The brute-force approach is actually the straightforward approach in this problem. We can iterate through the array and keep updating a variable `val` by applying the function `fn` to the current accumulated value and the current element. This approach works correctly because it directly follows the problem description. Given the constraints, this approach is efficient enough. There is no faster algorithm in terms of time complexity because each element must be processed.

The key observation is that this is inherently a sequential process; each result depends on the previous computation. Therefore, the optimal approach is the same as the brute-force approach, iterating through the array sequentially and updating the accumulated value.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Iterate through the array, apply the function sequentially, update accumulator |
| Optimal | O(n) | O(1) | Same as brute-force because each element must be processed |

## Algorithm Walkthrough

1. Initialize a variable `accumulator` with the value `init`. This variable will store the result of applying the reducer function sequentially.
2. Iterate through each element `num` in the array `nums`.
3. For each element, update `accumulator` by calling `fn(accumulator, num)`. This step applies the reducer function to the accumulated value so far and the current element.
4. After iterating through all elements, return the final value of `accumulator`.

Why it works: The algorithm works because it mirrors the definition of a reduce operation. The accumulator is initialized to `init` and sequentially updated using each element in the array. This guarantees that the final accumulator value is exactly what the problem expects.

## Python Solution

```python
from typing import List, Callable

class Solution:
    def arrayReduce(self, nums: List[int], fn: Callable[[int, int], int], init: int) -> int:
        accumulator = init
        for num in nums:
            accumulator = fn(accumulator, num)
        return accumulator
```

In this implementation, we first set `accumulator` to `init`. Then, we loop over every element in `nums`, updating `accumulator` by calling the provided function `fn` with the current accumulated value and the current element. Finally, we return `accumulator` as the result. This directly follows the algorithm walkthrough described above.

## Go Solution

```go
package main

func arrayReduce(nums []int, fn func(int, int) int, init int) int {
    accumulator := init
    for _, num := range nums {
        accumulator = fn(accumulator, num)
    }
    return accumulator
}
```

In Go, the implementation is similar. We initialize `accumulator` with `init`. Then we iterate over the `nums` slice using the range-based loop and update `accumulator` using the provided function `fn`. Finally, the result is returned. Go handles slices and function types cleanly, and the code mirrors the Python version closely.

## Edge Cases

One important edge case is an empty array. If the array has no elements, the function should immediately return `init`. This prevents errors from attempting to access array elements.

Another edge case is when the reducer function is non-additive, such as multiplication or subtraction. The implementation must correctly pass the accumulator and current element to `fn` in order, preserving the sequential dependency.

A third edge case is when `init` is non-zero or large. The function must correctly incorporate `init` into the first computation, ensuring the initial value is part of the final accumulated result. This is handled by initializing `accumulator` to `init` before processing any array elements.
