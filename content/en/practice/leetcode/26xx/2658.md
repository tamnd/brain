## Solution

## Problem Understanding

The problem gives us a 2D grid where each cell represents either land or water.

A value of `0` means the cell is land, and the fisher cannot move onto it. A positive value means the cell is water, and the number represents how many fish are available in that cell.

The fisher may start from any water cell and can repeatedly move up, down, left, or right, but only through water cells. Whenever the fisher visits a water cell, all fish in that cell can be collected.

The important observation is that once the fisher starts inside a connected group of water cells, he can eventually visit every water cell in that connected component. Since movement between adjacent water cells is unrestricted, the total number of fish collectable from a starting position is simply the sum of all fish values inside that connected water region.

Therefore, the task becomes:

1. Find every connected component of water cells.
2. Compute the sum of fish in each component.
3. Return the maximum component sum.

The constraints are very small:

- `1 <= m, n <= 10`
- At most `100` cells exist in the grid.

This tells us that even relatively inefficient graph traversal solutions are acceptable. Still, we should design a clean and scalable solution.

Several edge cases are important:

- The entire grid may contain only land cells.
- There may be multiple disconnected water regions.
- A region may consist of only one cell.
- The maximum answer may come from a large connected component, not from the cell with the largest individual fish count.
- Traversal must avoid revisiting cells, otherwise infinite recursion or repeated counting can occur.

## Approaches

### Brute Force Approach

A naive approach would be to start a traversal from every water cell independently.

For each starting cell, we could perform a DFS or BFS to compute the total fish reachable from that cell. However, if we do this without remembering previously explored connected components, the same region gets traversed repeatedly.

Suppose a connected component contains `k` water cells. Starting a full traversal from each of those `k` cells means the same component is explored `k` times.

Although the grid size is small enough that this would still pass, it is unnecessarily repetitive.

The brute-force method works because every traversal correctly computes the total fish reachable from that starting position. The inefficiency comes from recomputing the same connected-component sums over and over.

### Optimal Approach

The key insight is that all cells inside the same connected water component produce the exact same total fish count.

If two water cells are connected, starting from either one allows the fisher to visit the entire component. Therefore, we only need to compute the component sum once.

This naturally becomes a graph traversal problem:

- Treat every water cell as a graph node.
- Connect adjacent water cells.
- Use DFS or BFS to explore each connected component exactly once.
- Sum the fish values during traversal.
- Track the maximum component sum.

A `visited` structure prevents revisiting cells and ensures each cell contributes to the answer only once.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O((m × n)²) | O(m × n) | Repeats traversal for the same connected components |
| Optimal | O(m × n) | O(m × n) | Each cell is visited at most once |

## Algorithm Walkthrough

### Optimal DFS Algorithm

1. Initialize variables for the grid dimensions and the maximum fish count found so far.
2. Create a `visited` matrix of the same size as the grid.

This matrix keeps track of which cells have already been explored. Without it, the traversal could revisit cells repeatedly and double count fish.
3. Iterate through every cell in the grid.

For each position `(row, col)`:

- Skip the cell if it is land (`0`).
- Skip the cell if it has already been visited.
- Otherwise, this cell starts a new connected water component.
4. Launch a DFS from the current water cell.

The DFS:

- Marks the current cell as visited.
- Adds the current cell's fish count to the component total.
- Explores the four neighboring directions:

- up
- down
- left
- right
5. During DFS neighbor exploration:

For each neighboring cell:

- Ignore it if it is outside grid bounds.
- Ignore it if it is land.
- Ignore it if it was already visited.
- Otherwise recursively continue DFS.
6. The DFS returns the total fish count for the entire connected component.
7. Update the global maximum answer using the returned component sum.
8. Continue scanning the grid until all cells have been processed.
9. Return the maximum fish total found.

### Why it works

The algorithm works because DFS completely explores one connected water component before moving to another. Every reachable water cell from the starting position is visited exactly once, ensuring the component sum is computed correctly.

The `visited` matrix guarantees that:

- No cell is counted multiple times.
- No connected component is processed more than once.

Since every water cell belongs to exactly one connected component, and every component sum is evaluated, the maximum computed sum is the correct answer.

## Python Solution

```python
from typing import List

class Solution:
    def findMaxFish(self, grid: List[List[int]]) -> int:
        rows = len(grid)
        cols = len(grid[0])

        visited = [[False] * cols for _ in range(rows)]

        directions = [(1, 0), (-1, 0), (0, 1), (0, -1)]

        def dfs(row: int, col: int) -> int:
            visited[row][col] = True

            total_fish = grid[row][col]

            for dr, dc in directions:
                next_row = row + dr
                next_col = col + dc

                if (
                    0 <= next_row < rows
                    and 0 <= next_col < cols
                    and not visited[next_row][next_col]
                    and grid[next_row][next_col] > 0
                ):
                    total_fish += dfs(next_row, next_col)

            return total_fish

        max_fish = 0

        for row in range(rows):
            for col in range(cols):
                if grid[row][col] > 0 and not visited[row][col]:
                    component_fish = dfs(row, col)
                    max_fish = max(max_fish, component_fish)

        return max_fish
```

The implementation begins by determining the grid dimensions and allocating a `visited` matrix. This matrix mirrors the grid structure and tracks whether a cell has already been explored.

The `directions` array stores the four valid movement directions. Using a reusable direction list keeps neighbor traversal concise and avoids duplicated logic.

The nested `dfs` function performs the depth-first traversal. As soon as a cell is visited, it is marked in the `visited` matrix so it cannot be processed again later.

The DFS initializes `total_fish` with the current cell's fish count. It then checks all four neighboring cells. A neighbor is explored only if:

- it stays inside grid boundaries,
- it has not already been visited,
- and it contains water.

The recursive call returns the fish sum from that neighboring connected region, which is added into the current component total.

The outer nested loops scan every cell in the grid. Whenever an unvisited water cell is discovered, a new DFS begins, computing the total fish inside that connected component.

Finally, the algorithm keeps the largest component sum seen so far and returns it.

## Go Solution

```go
func findMaxFish(grid [][]int) int {
	rows := len(grid)
	cols := len(grid[0])

	visited := make([][]bool, rows)
	for i := range visited {
		visited[i] = make([]bool, cols)
	}

	directions := [][]int{
		{1, 0},
		{-1, 0},
		{0, 1},
		{0, -1},
	}

	var dfs func(int, int) int

	dfs = func(row int, col int) int {
		visited[row][col] = true

		totalFish := grid[row][col]

		for _, dir := range directions {
			nextRow := row + dir[0]
			nextCol := col + dir[1]

			if nextRow >= 0 &&
				nextRow < rows &&
				nextCol >= 0 &&
				nextCol < cols &&
				!visited[nextRow][nextCol] &&
				grid[nextRow][nextCol] > 0 {

				totalFish += dfs(nextRow, nextCol)
			}
		}

		return totalFish
	}

	maxFish := 0

	for row := 0; row < rows; row++ {
		for col := 0; col < cols; col++ {
			if grid[row][col] > 0 && !visited[row][col] {
				componentFish := dfs(row, col)

				if componentFish > maxFish {
					maxFish = componentFish
				}
			}
		}
	}

	return maxFish
}
```

The Go implementation closely follows the Python version. The main difference is explicit allocation of the `visited` 2D slice using `make`.

Go does not support nested named functions in the same way as Python, so the DFS is assigned to a function variable before use.

Integer overflow is not a concern because the maximum possible fish total is very small:

- at most `100` cells,
- each cell contains at most `10` fish,
- maximum sum is `1000`.

This safely fits inside Go's `int` type.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m × n) | Every cell is visited at most once |
| Space | O(m × n) | Visited matrix plus recursion stack |

The DFS traversal processes each cell exactly one time. Each neighbor check is constant work, so the total runtime grows linearly with the number of cells in the grid.

The auxiliary space comes from two sources:

- the `visited` matrix,
- the recursion stack in the worst case where the entire grid forms one connected component.

Since the grid contains at most `100` cells, recursion depth is safe for both languages.

## Edge Cases

### Entire Grid Contains Only Land

A common mistake is assuming at least one water cell exists. If the entire grid is filled with zeros, no DFS should ever start.

The implementation handles this naturally because the outer loops only begin DFS when `grid[row][col] > 0`. Since this never happens, `max_fish` remains `0`, which is correctly returned.

### Diagonal Water Cells

Some implementations accidentally treat diagonal cells as connected, especially when using generalized neighbor loops.

For example:

```
[
    [1,0],
    [0,2]
]
```

These two water cells are not adjacent because only up, down, left, and right movement is allowed.

The implementation explicitly uses only four directional offsets, ensuring diagonal connectivity is never considered.

### One Massive Connected Component

If every cell contains water, the DFS will recursively traverse the entire grid.

A buggy implementation might revisit cells repeatedly and either double count fish or recurse infinitely.

The `visited` matrix prevents this issue. Each cell is marked visited immediately upon entering DFS, guaranteeing every cell is processed exactly once.
