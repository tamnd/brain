# LeetCode 2696, Minimum String Length After Removing Substrings

## Problem Understanding

The problem gives us a string `s` that contains only uppercase English letters. We are allowed to repeatedly remove occurrences of the substrings `"AB"` and `"CD"`.

When a substring is removed, the remaining parts of the string are joined together immediately. This is important because new removable substrings may appear after the concatenation.

The goal is to determine the smallest possible length of the string after applying these operations as many times as possible.

For example, consider the string `"ABFCACDB"`.

We can remove `"AB"` from the beginning, which leaves `"FCACDB"`. Then we can remove `"CD"` from `"FCACDB"` to get `"FCAB"`. Finally, we remove `"AB"` again to get `"FC"`. No more removals are possible, so the answer is `2`.

The constraints are small:

```
1 <= s.length <= 100
```

This means even relatively inefficient approaches could work, but the problem is designed to test whether we can recognize a pattern that naturally fits a stack-based simulation.

Several edge cases are important to think about early:

A string may already contain no removable substrings at all, such as `"ACBBD"`.

A removal may create a new removable substring across the boundary formed after concatenation. For example, removing characters in the middle may cause two previously separated characters to become adjacent and form `"AB"` or `"CD"`.

The entire string may disappear completely, such as `"ABCD"`.

The string length is small, but we still want a clean and reliable implementation that correctly handles chained removals.

## Approaches

### Brute Force Approach

The most direct solution is to repeatedly search the string for occurrences of `"AB"` or `"CD"`.

Whenever one is found, remove it and restart the search from the beginning. Continue until neither substring exists anymore.

This approach works because every valid operation strictly reduces the string length by `2`, so the process must eventually terminate.

However, repeatedly rebuilding strings is inefficient. Every removal requires creating a new string, and each scan may traverse most of the current string again.

Even though the constraint is only `100`, this approach is not elegant and performs unnecessary repeated work.

### Optimal Stack-Based Approach

The key observation is that removals only depend on adjacent character pairs.

Whenever we process a new character, we only need to know whether it forms `"AB"` or `"CD"` with the immediately previous character.

This naturally suggests using a stack.

We iterate through the string character by character:

- Push characters onto the stack.
- Before pushing, check whether the current character together with the stack top forms `"AB"` or `"CD"`.
- If it does, pop the stack instead of pushing the current character, effectively removing the substring.

This works because the stack always represents the current reduced version of the string after all possible removals so far.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeatedly scans and rebuilds the string |
| Optimal | O(n) | O(n) | Uses a stack to simulate removals in one pass |

## Algorithm Walkthrough

1. Initialize an empty stack.

The stack will store the characters that remain after processing removals so far.
2. Iterate through each character in the string.

We process the string from left to right because removals only depend on adjacent characters.
3. For the current character, check the stack top.

If the stack is not empty, inspect the top character together with the current character.
4. Determine whether they form a removable pair.

If the top character is `'A'` and the current character is `'B'`, remove the pair.

If the top character is `'C'` and the current character is `'D'`, remove the pair.
5. Remove the pair when applicable.

We do this by popping the stack and skipping the push of the current character.
6. Otherwise, push the current character onto the stack.

This means no removal is possible at this moment.
7. After processing all characters, return the stack size.

The stack contains the fully reduced string after all possible operations.

### Why it works

The stack always maintains the reduced form of the processed prefix of the string.

Whenever a new character arrives, the only possible new removable substring must involve that character and the immediately preceding surviving character. The stack top represents exactly that preceding character.

By removing pairs immediately when they appear, we guarantee that every possible removal is performed. Since every operation only affects adjacent characters, no valid removal can be missed.

## Python Solution

```python
class Solution:
    def minLength(self, s: str) -> int:
        stack: list[str] = []

        for char in s:
            if stack:
                top = stack[-1]

                if (top == 'A' and char == 'B') or \
                   (top == 'C' and char == 'D'):
                    stack.pop()
                    continue

            stack.append(char)

        return len(stack)
```

The implementation directly follows the stack simulation described earlier.

We begin with an empty stack. As we iterate through the string, we check whether the current character forms either `"AB"` or `"CD"` with the character at the top of the stack.

If a removable pair is found, we pop the top character and skip pushing the current one. This simulates deleting the substring from the string.

If no removable pair exists, we push the current character onto the stack because it remains part of the reduced string.

At the end, the stack contains exactly the characters that could not be removed, so its length is the answer.

## Go Solution

```go
func minLength(s string) int {
    stack := []byte{}

    for i := 0; i < len(s); i++ {
        ch := s[i]

        if len(stack) > 0 {
            top := stack[len(stack)-1]

            if (top == 'A' && ch == 'B') ||
                (top == 'C' && ch == 'D') {
                stack = stack[:len(stack)-1]
                continue
            }
        }

        stack = append(stack, ch)
    }

    return len(stack)
}
```

The Go implementation uses a `[]byte` slice as the stack because the input contains only ASCII uppercase letters.

Popping from the stack is done using slice truncation:

```go
stack = stack[:len(stack)-1]
```

Appending behaves similarly to pushing in other languages.

Unlike Python, Go does not have a built-in stack type, so slices are the idiomatic choice.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each character is pushed and popped at most once |
| Space | O(n) | The stack may store all characters in the worst case |

The algorithm is linear because every character participates in at most one push and one pop operation.

The auxiliary space is linear in the worst case when no removals occur and the entire string remains in the stack.

## Edge Cases

One important edge case is when removals create new removable substrings after concatenation. For example, `"CABD"` initially contains `"AB"` in the middle. Removing it leaves `"CD"`, which is also removable. A naive implementation that only scans once would fail here. The stack approach handles this naturally because the stack always reflects the current reduced string.

Another important edge case is when no removals are possible at all. Strings like `"ACBBD"` or `"AAAA"` never form `"AB"` or `"CD"`. The algorithm correctly keeps pushing characters onto the stack, and the final stack size equals the original string length.

A third important case is complete elimination of the string. Inputs like `"ABCD"` or `"ABAB"` can be reduced to an empty string. The implementation correctly returns `0` because every removable pair causes a pop operation, eventually leaving the stack empty.
