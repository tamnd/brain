# LeetCode 2639 - Find the Width of Columns of a Grid

## Problem Understanding

This problem gives us a two dimensional integer matrix called `grid`, with `m` rows and `n` columns. Our task is to compute the width of every column and return those widths as an array.

The width of a column is defined as the maximum number of characters needed to represent any integer in that column. This includes the negative sign for negative numbers.

For example:

- The integer `333` has length `3`
- The integer `12` has length `2`
- The integer `-15` has length `3` because the minus sign counts as a character

We need to process every column independently and determine the largest string length among all values in that column.

The input matrix is guaranteed to be rectangular because the constraints state that every row has exactly `n` columns. The matrix dimensions are relatively small:

- `1 <= m, n <= 100`
- Each integer value is between `-10^9` and `10^9`

Since the grid size is at most `100 x 100`, the total number of elements is at most `10,000`, which is small enough for a straightforward traversal solution.

An important detail is that negative numbers require special handling because their string representation includes the `-` character. Another subtle point is that `0` has length `1`, not `0`.

Some edge cases that could cause bugs include:

- Columns containing negative numbers
- Columns containing both positive and negative values
- Single row or single column matrices
- Values with very different digit lengths
- The value `0`

The problem guarantees valid integer inputs and a non-empty matrix, so we do not need to handle malformed input.

## Approaches

### Brute Force Approach

A brute force solution would examine each column separately. For every column, we would iterate through all rows, convert each integer into a string, compute its length, and keep track of the maximum length seen so far.

This approach is correct because the definition of column width is literally the maximum length among all values in that column. By checking every element, we ensure that no candidate is missed.

Although this approach is called brute force, it is already efficient enough for the given constraints because the matrix is small. Every element must be inspected at least once, so this is effectively optimal.

### Optimal Approach

The key observation is that each cell contributes independently to its column's width. There is no need for complicated preprocessing or additional data structures.

We can maintain an array `answer` of size `n`, where `answer[col]` stores the largest width encountered for that column so far.

As we traverse the matrix:

1. Convert the current integer into a string
2. Compute its length
3. Update the maximum width for that column

This solution processes each element exactly once and uses only a small amount of extra space.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m × n) | O(n) | Check every value column by column |
| Optimal | O(m × n) | O(n) | Single traversal while tracking column maximums |

## Algorithm Walkthrough

1. Determine the number of columns using `len(grid[0])`.
2. Create a result array called `column_widths` of size `n`, initialized with zeros. Each position represents the maximum width found for that column so far.
3. Iterate through every row in the matrix.
4. For each row, iterate through every column index.
5. Convert the current integer into a string using `str(value)` in Python or `strconv.Itoa(value)` in Go.
6. Compute the length of that string. This automatically handles:

- Positive numbers
- Negative numbers
- Zero
7. Compare this length with the current stored width for the column.
8. If the current length is larger, update the column width.
9. After processing all rows and columns, return the `column_widths` array.

### Why it works

The algorithm maintains the invariant that after processing each element, `column_widths[col]` stores the maximum length seen so far in that column. Since every element in the matrix is examined exactly once, by the end of the traversal each column width must equal the maximum length among all values in that column. Therefore, the returned array is correct.

## Python Solution

```python
from typing import List

class Solution:
    def findColumnWidth(self, grid: List[List[int]]) -> List[int]:
        rows = len(grid)
        cols = len(grid[0])

        column_widths = [0] * cols

        for row in range(rows):
            for col in range(cols):
                current_length = len(str(grid[row][col]))
                column_widths[col] = max(column_widths[col], current_length)

        return column_widths
```

The implementation begins by determining the number of rows and columns in the matrix. We then initialize the `column_widths` array with zeros because no values have been processed yet.

The nested loops traverse every cell in the matrix. For each value, the code converts the integer into a string and measures its length. This is the simplest and safest way to correctly handle negative numbers because the minus sign becomes part of the string automatically.

The `max` function updates the stored width for the current column whenever a larger width is found.

Finally, the completed `column_widths` array is returned.

## Go Solution

```go
package main

import "strconv"

func findColumnWidth(grid [][]int) []int {
    rows := len(grid)
    cols := len(grid[0])

    columnWidths := make([]int, cols)

    for row := 0; row < rows; row++ {
        for col := 0; col < cols; col++ {
            currentLength := len(strconv.Itoa(grid[row][col]))

            if currentLength > columnWidths[col] {
                columnWidths[col] = currentLength
            }
        }
    }

    return columnWidths
}
```

The Go implementation follows the same logic as the Python version. The primary difference is that Go requires explicit conversion from integer to string using `strconv.Itoa`.

Slices are used instead of Python lists. Since the constraints guarantee a non-empty matrix, accessing `grid[0]` is safe.

Go integers are large enough to safely store values within the problem constraints, so overflow is not a concern.

## Edge Cases

### Single Element Matrix

A matrix containing only one value is the smallest valid input. A buggy implementation might incorrectly assume multiple rows or columns exist. This solution handles the case naturally because the loops still execute exactly once and compute the correct width.

### Negative Numbers

Negative integers are an important edge case because the minus sign contributes to the total width. For example, `-15` has width `3`, not `2`. By converting integers directly into strings, the implementation automatically includes the minus sign in the length calculation.

### Columns with Mixed Digit Lengths

A column may contain numbers with very different lengths, such as `1`, `22`, and `3333`. A flawed implementation might overwrite the width incorrectly instead of maintaining the maximum. This solution avoids that problem by always taking the maximum between the current stored width and the newly computed length.

### Zero Values

The value `0` is easy to mishandle if digit counting logic is implemented manually. Some digit counting approaches accidentally treat zero as having zero digits. Using string conversion guarantees that `"0"` has length `1`, which is correct.
