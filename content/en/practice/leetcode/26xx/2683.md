## Solution

## Problem Understanding

The problem gives us a binary array called `derived`, where each element was created using the XOR operation between neighboring elements of another binary array called `original`.

For every index:

- `derived[i] = original[i] XOR original[i + 1]`
- For the last element, the array wraps around:

`derived[n - 1] = original[n - 1] XOR original[0]`

Our task is to determine whether at least one valid binary array `original` could produce the given `derived` array.

The key detail is that `original` contains only `0` and `1`, and XOR also produces only `0` or `1`.

The constraints allow `n` up to `10^5`, which immediately tells us that exponential or quadratic brute-force solutions are not appropriate. We need a linear-time solution.

The input represents the XOR differences between neighboring values in a circular binary array. The output should be:

- `true` if some binary array `original` exists
- `false` otherwise

An important observation is that the array is circular. The last XOR relation connects back to the first element, which creates a consistency condition across the entire array.

Several edge cases are important:

- Arrays of length `1`
- Arrays where all values are `0`
- Arrays where the XOR relationships contradict each other
- Arrays with an odd or even number of `1`s

These cases can expose incorrect assumptions about how XOR behaves in cyclic structures.

## Approaches

### Brute Force Approach

A straightforward approach is to try every possible binary array `original` of length `n`.

Since each position can be either `0` or `1`, there are:

$2^n$

possible arrays.

For each candidate array:

1. Compute the derived XOR values.
2. Compare them with the input `derived`.
3. Return `true` if a match is found.

This works because it exhaustively checks every possible valid binary array. However, it is far too slow for `n = 100000`.

Even for moderate values of `n`, exponential growth makes this infeasible.

### Optimal Observation

The XOR operation has a very useful property.

If we XOR all values in `derived` together:

$derived[0] \oplus derived[1] \oplus \cdots \oplus derived[n-1]$

then every element in `original` appears exactly twice.

For example:

$(a_0 \oplus a_1) \oplus (a_1 \oplus a_2) \oplus \cdots \oplus (a_{n-1} \oplus a_0)$

Because XOR has these properties:

- `x XOR x = 0`
- `x XOR 0 = x`
- XOR is associative and commutative

every duplicated value cancels out.

So the total XOR becomes:

$0$

Therefore, a valid `original` array exists if and only if the XOR of all elements in `derived` equals `0`.

This gives us a very efficient linear-time solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × 2^n) | O(n) | Tries every possible binary array |
| Optimal | O(n) | O(1) | Uses the global XOR cancellation property |

## Algorithm Walkthrough

1. Initialize a variable called `xor_total` to `0`.

This variable will store the cumulative XOR of all elements in `derived`.
2. Iterate through every value in `derived`.

For each value:

- Update `xor_total` using XOR:

`xor_total ^= value`

XOR accumulation works because XOR is associative and commutative, so the order does not matter.
3. After processing all elements, check whether `xor_total` equals `0`.

If the total XOR is `0`, then the XOR equations are globally consistent and some valid `original` array exists.

Otherwise, the equations contradict each other, so no valid binary array can satisfy all conditions simultaneously.
4. Return the result of the comparison.

### Why it works

Each element of `original` participates in exactly two XOR equations because the array is circular.

When all `derived` values are XORed together, every `original[i]` appears twice:

$x \oplus x = 0$

so all elements cancel out.

Therefore, the total XOR of `derived` must equal `0` for the system to be valid. This condition is both necessary and sufficient.

## Python Solution

```python
from typing import List

class Solution:
    def doesValidArrayExist(self, derived: List[int]) -> bool:
        xor_total = 0

        for value in derived:
            xor_total ^= value

        return xor_total == 0
```

The implementation follows the exact mathematical observation derived earlier.

The variable `xor_total` starts at `0`, which is the identity element for XOR operations.

We iterate through the array and continuously XOR each value into `xor_total`. Since XOR accumulation automatically cancels duplicate contributions, the final value tells us whether the XOR constraints are globally consistent.

Finally, the function returns whether the accumulated XOR equals `0`.

The implementation uses constant extra space and only scans the array once.

## Go Solution

```go
func doesValidArrayExist(derived []int) bool {
    xorTotal := 0

    for _, value := range derived {
        xorTotal ^= value
    }

    return xorTotal == 0
}
```

The Go implementation is nearly identical to the Python version.

The solution uses Go's `range` loop to iterate through the slice. Since all values are either `0` or `1`, there are no integer overflow concerns.

Go slices naturally handle arrays of any valid size within the constraints, and no additional memory allocation is required.

## Edge Cases

### Single Element Array

A length-one array is easy to mishandle because the element XORs with itself.

The equation becomes:

$derived[0] = original[0] \oplus original[0]$

Since any value XOR itself equals `0`, the only valid derived array is `[0]`.

If the input is `[1]`, no solution exists. The implementation handles this automatically because the total XOR becomes `1`, so it returns `false`.

### Arrays With All Zeroes

If every value in `derived` is `0`, then every adjacent pair in `original` must be equal.

This means the entire original array can consist entirely of `0`s or entirely of `1`s.

The cumulative XOR is also `0`, so the implementation correctly returns `true`.

### Arrays With Odd Number of Ones

A subtle but important case occurs when the total XOR becomes `1`.

For binary values, XOR behaves like parity. An odd number of `1`s produces total XOR `1`, which violates the cancellation property required by the cyclic structure.

Many naive implementations attempt reconstruction and may incorrectly conclude a valid array exists. The mathematical XOR check avoids this issue entirely and guarantees correctness.
