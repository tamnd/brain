# LeetCode 2615, Sum of Distances

## Problem Understanding

The problem gives us a `0-indexed` integer array `nums`. For every position `i`, we must compute the total distance from index `i` to every other index `j` where the value is the same.

Formally, for each index `i`, we compute:

$$\sum |i - j|$$

for all indices `j` such that:

- `nums[j] == nums[i]`
- `j != i`

The result for every index is stored in an array `arr`, and we return that array.

For example, suppose:

```
nums = [1,3,1,1,2]
```

At index `0`, the value is `1`. Other positions containing `1` are indices `2` and `3`.

So:

```
arr[0] = |0 - 2| + |0 - 3|
       = 2 + 3
       = 5
```

The same process is repeated for every index.

The constraints are important:

- `nums.length` can be as large as `10^5`
- Values can be as large as `10^9`

The large array size immediately tells us that an `O(n^2)` solution will be too slow. We need something close to linear time.

Several edge cases are important:

- Arrays where all values are unique, every answer should be `0`
- Arrays where all values are identical, every index contributes to many others
- Very large groups of repeated values, where repeated distance computation becomes expensive
- Values can be large integers, so we must use hash maps instead of arrays indexed by value

The problem guarantees valid input and at least one element in the array.

## Approaches

### Brute Force Approach

The most direct solution is to compute the answer independently for every index.

For each position `i`, we scan the entire array again and check every `j`.

If `nums[j] == nums[i]` and `j != i`, we add:

```
|i - j|
```

to the answer.

This approach is correct because it explicitly follows the problem definition. However, it is far too slow.

If `n = 10^5`, then checking every pair requires:

```
O(n^2) = 10^10
```

operations in the worst case, which is not feasible.

### Optimal Approach

The key observation is that indices with the same value can be processed together.

Suppose a value appears at indices:

```
[p0, p1, p2, ..., pk]
```

For any position `pi`, we want:

```
(pi - p0) + (pi - p1) + ...
```

Instead of recomputing these distances from scratch, we can use prefix sums.

If we know:

- how many identical indices are to the left
- the sum of their positions
- how many identical indices are to the right
- the sum of their positions

then we can compute the total contribution in constant time.

For an index `pos`:

Left contribution:

$$pos \times leftCount - leftSum$$

Right contribution:

$$rightSum - pos \times rightCount$$

Adding them gives the total distance.

We group indices by value using a hash map, then process each group independently.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Compare every pair of indices |
| Optimal | O(n) | O(n) | Group indices and use prefix sums |

## Algorithm Walkthrough

1. Create a hash map where the key is a number from `nums` and the value is a list of indices where that number appears.

For example:

```
nums = [1,3,1,1,2]
```

becomes:

```
{
    1: [0,2,3],
    3: [1],
    2: [4]
}
```
2. Initialize the result array with zeros.

Every position eventually receives its total distance.
3. Process each group of indices independently.

Suppose the current group is:

```
positions = [0,2,3]
```
4. Compute the total sum of all positions in the group.

For the example:

```
totalSum = 0 + 2 + 3 = 5
```
5. Traverse the positions from left to right.

Maintain:

- `leftSum`, the sum of processed positions
- `leftCount`, how many positions are already processed
6. For the current position `pos`, compute the left contribution.

Every previous index contributes:

```
pos - previousIndex
```

Summed together:

$$pos \times leftCount - leftSum$$
7. Compute the right contribution.

Let:

```
rightCount = totalCount - leftCount - 1
rightSum = totalSum - leftSum - pos
```

Then:

$$rightSum - pos \times rightCount$$
8. Add left and right contributions together and store the result.
9. Update `leftSum` and continue to the next index.
10. After all groups are processed, return the result array.

### Why it works

For each index, every identical element lies either to the left or to the right.

For left-side elements:

$$|pos - x| = pos - x$$

because `x < pos`.

For right-side elements:

$$|pos - x| = x - pos$$

because `x > pos`.

By storing counts and sums of indices, we can aggregate all these distances mathematically instead of computing them individually. Every contribution is counted exactly once, so the result is correct.

## Python Solution

```python
from collections import defaultdict
from typing import List

class Solution:
    def distance(self, nums: List[int]) -> List[int]:
        positions_map = defaultdict(list)

        for index, value in enumerate(nums):
            positions_map[value].append(index)

        result = [0] * len(nums)

        for positions in positions_map.values():
            total_sum = sum(positions)

            left_sum = 0
            left_count = 0
            total_count = len(positions)

            for pos in positions:
                right_count = total_count - left_count - 1
                right_sum = total_sum - left_sum - pos

                left_distance = pos * left_count - left_sum
                right_distance = right_sum - pos * right_count

                result[pos] = left_distance + right_distance

                left_sum += pos
                left_count += 1

        return result
```

The implementation begins by grouping indices with the same value using a `defaultdict(list)`.

Each value maps to all positions where it appears. This preprocessing allows us to process identical elements together.

The `result` array stores the final answers.

For every group of positions, we first compute `total_sum`, which is the sum of all indices in that group. This allows constant-time computation of the right-side contribution.

During iteration, `left_sum` tracks the sum of indices already processed, and `left_count` tracks how many positions are on the left side.

For each position:

- `left_distance` computes distances to earlier identical indices
- `right_distance` computes distances to later identical indices

The formula avoids repeated subtraction for every pair.

Finally, we store the total distance for that position and update the running prefix information.

## Go Solution

```go
func distance(nums []int) []int64 {
    positionsMap := make(map[int][]int)

    for index, value := range nums {
        positionsMap[value] = append(positionsMap[value], index)
    }

    result := make([]int64, len(nums))

    for _, positions := range positionsMap {
        var totalSum int64 = 0

        for _, pos := range positions {
            totalSum += int64(pos)
        }

        var leftSum int64 = 0
        leftCount := 0
        totalCount := len(positions)

        for _, pos := range positions {
            rightCount := totalCount - leftCount - 1
            rightSum := totalSum - leftSum - int64(pos)

            leftDistance := int64(pos*leftCount) - leftSum
            rightDistance := rightSum - int64(pos*rightCount)

            result[pos] = leftDistance + rightDistance

            leftSum += int64(pos)
            leftCount++
        }
    }

    return result
}
```

The Go solution follows exactly the same algorithmic structure as the Python version.

The main implementation difference is integer handling. Since the sum of distances can exceed the range of a 32-bit integer, the result array uses `int64`.

Go also requires explicit type conversion between `int` and `int64`, especially during arithmetic involving sums and multiplication.

Slices are used to store positions for each value, and the hash map is implemented with `map[int][]int`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every index is processed a constant number of times |
| Space | O(n) | Hash map stores all indices |

The grouping phase processes each element once.

Later, each stored index is visited exactly once again while computing contributions. No nested iteration over the entire array occurs.

The total work across all groups is therefore linear in the size of the array.

The extra memory comes from storing the index lists inside the hash map.

## Edge Cases

One important edge case occurs when every element is unique. In this situation, every group contains exactly one index, so no valid `j` exists for any position. A buggy implementation might accidentally compute negative values or perform invalid prefix calculations. This implementation handles the case naturally because both left and right counts become zero.

Another important case is when all elements are identical. Here, every index contributes to every other index, creating the maximum amount of work for a naive solution. The prefix sum method handles this efficiently because each position is processed once, regardless of group size.

A subtle edge case involves very large answers. Suppose the array length is `10^5` and all elements are equal. The total distances can become very large, exceeding 32-bit integer limits. The Go solution explicitly uses `int64` to prevent overflow, and Python naturally supports arbitrary-sized integers.

A final edge case is repeated values appearing in non-contiguous positions, such as:

```
[1,2,1,2,1]
```

The algorithm does not rely on adjacency. It groups indices globally using the hash map, ensuring correct distance computation even when identical values are spread throughout the array.
