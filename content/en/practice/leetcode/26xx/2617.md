# LeetCode 2617 - Minimum Number of Visited Cells in a Grid

## Problem Understanding

The problem gives us a 2D grid where each cell contains a non-negative integer. We begin at the top-left corner `(0, 0)` and want to reach the bottom-right corner `(m - 1, n - 1)` while visiting as few cells as possible.

From a cell `(i, j)`, the value `grid[i][j]` determines how far we can move. We may move:

- Right to any cell `(i, k)` where `j < k <= j + grid[i][j]`
- Down to any cell `(k, j)` where `i < k <= i + grid[i][j]`

Each movement counts as visiting another cell. The goal is to compute the minimum number of visited cells required to reach the destination. If the destination cannot be reached, we return `-1`.

This is fundamentally a shortest path problem on a directed graph. Each cell represents a node, and valid movements represent directed edges. However, explicitly building all edges is impossible because a single cell may connect to many other cells, and the total number of edges can become quadratic.

The constraints are the key observation:

- `m * n <= 100000`
- A naive BFS exploring every reachable edge would be too slow
- We need something close to `O(m * n log(m * n))`

An important detail is that movements are only rightward or downward. There are no backward moves. This means the graph is acyclic and strongly structured, which enables more efficient processing.

Several edge cases are important:

- A grid may contain many zeros, blocking movement entirely
- The destination cell itself is always zero, meaning once we arrive there, no further movement is needed
- The start cell could be zero in a `1 x 1` grid, which is still reachable because we already begin there
- Multiple paths may exist with different path lengths
- Large jump values can create many possible transitions, so we must avoid enumerating all reachable neighbors directly

## Approaches

### Brute Force Approach

The most direct solution is to model the grid as a graph and run Breadth-First Search.

For every cell `(i, j)`, we attempt all valid rightward and downward moves:

- Right moves: from `j + 1` to `j + grid[i][j]`
- Down moves: from `i + 1` to `i + grid[i][j]`

Since BFS explores states level by level, the first time we reach the destination gives the minimum number of visited cells.

This approach is correct because BFS guarantees shortest path length in an unweighted graph.

However, the problem is performance. A single cell can generate `O(m + n)` outgoing edges. Across all cells, the total number of transitions can become `O(m * n * (m + n))`, which is far too large for `100000` cells.

### Optimal Approach

The key optimization is to avoid repeatedly scanning already processed cells.

Suppose we are at `(i, j)` and can move right up to column `r`. Instead of iterating over every column repeatedly from many different cells, we want a way to quickly skip positions that have already been visited.

We can achieve this using ordered sets or disjoint-set style "next unvisited" structures.

The idea is:

- For every row, maintain the set of unvisited columns
- For every column, maintain the set of unvisited rows
- When BFS reaches a cell, efficiently retrieve only the still-unvisited reachable cells
- Once a cell is processed, remove it from the corresponding row and column structures

This turns repeated scans into nearly linear work overall.

The result is an efficient BFS with pruning, running in approximately `O(m * n log(m * n))`.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m * n * (m + n)) | O(m * n) | Explicitly explores all possible moves |
| Optimal | O(m * n * log(m * n)) | O(m * n) | Uses BFS with ordered skipping of processed cells |

## Algorithm Walkthrough

1. Treat every cell as a graph node and perform BFS starting from `(0, 0)`.
2. Create a queue storing `(row, col, distance)` where `distance` represents how many cells have been visited so far.
3. For every row, maintain a sorted structure containing all currently unvisited columns. Similarly, for every column, maintain all currently unvisited rows.
4. Initially, every position is unvisited except `(0, 0)`. Remove `(0, 0)` from the row and column structures because BFS starts there immediately.
5. While processing cell `(i, j)`:

- Compute the furthest reachable column:

`max_col = j + grid[i][j]`
- Compute the furthest reachable row:

`max_row = i + grid[i][j]`
6. Using the row structure for row `i`, efficiently retrieve all unvisited columns in the interval `(j, max_col]`.

- Every such cell becomes reachable in one additional step
- Add them to the BFS queue
- Remove them from both row and column structures immediately so they are never processed twice
7. Similarly, use the column structure for column `j` to retrieve all unvisited rows in `(i, max_row]`.
8. Continue BFS layer by layer until:

- The destination is reached, return the distance
- The queue becomes empty, return `-1`

### Why it works

BFS guarantees that the first time a node is reached, it is reached using the minimum number of steps. The optimization structures do not alter reachability or traversal order. They only prevent repeated scanning of already visited nodes. Since each cell enters the queue exactly once, the computed distance remains optimal.

## Python Solution

```python
from typing import List
from collections import deque
from sortedcontainers import SortedSet

class Solution:
    def minimumVisitedCells(self, grid: List[List[int]]) -> int:
        m = len(grid)
        n = len(grid[0])

        if m == 1 and n == 1:
            return 1

        row_sets = [SortedSet(range(n)) for _ in range(m)]
        col_sets = [SortedSet(range(m)) for _ in range(n)]

        queue = deque([(0, 0, 1)])

        row_sets[0].remove(0)
        col_sets[0].remove(0)

        while queue:
            row, col, dist = queue.popleft()

            jump = grid[row][col]

            max_col = min(n - 1, col + jump)

            idx = row_sets[row].bisect_right(col)

            while idx < len(row_sets[row]):
                next_col = row_sets[row][idx]

                if next_col > max_col:
                    break

                if row == m - 1 and next_col == n - 1:
                    return dist + 1

                queue.append((row, next_col, dist + 1))

                row_sets[row].remove(next_col)
                col_sets[next_col].remove(row)

            max_row = min(m - 1, row + jump)

            idx = col_sets[col].bisect_right(row)

            while idx < len(col_sets[col]):
                next_row = col_sets[col][idx]

                if next_row > max_row:
                    break

                if next_row == m - 1 and col == n - 1:
                    return dist + 1

                queue.append((next_row, col, dist + 1))

                col_sets[col].remove(next_row)
                row_sets[next_row].remove(col)

        return -1
```

The implementation begins by constructing two collections of sorted sets.

- `row_sets[i]` stores all unvisited columns in row `i`
- `col_sets[j]` stores all unvisited rows in column `j`

These structures allow efficient interval queries and deletions.

The BFS queue stores the current cell and the number of visited cells required to reach it.

When processing a cell, the algorithm computes the maximum reachable row and column. Using binary search operations on the sorted sets, it efficiently jumps directly to reachable unvisited cells.

Every newly discovered cell is immediately removed from both tracking structures. This guarantees that no cell is inserted into the BFS queue more than once.

The algorithm terminates as soon as the destination cell is discovered because BFS guarantees optimality.

## Go Solution

```go
package main

import (
	"container/list"
)

type State struct {
	row  int
	col  int
	dist int
}

func minimumVisitedCells(grid [][]int) int {
	m := len(grid)
	n := len(grid[0])

	if m == 1 && n == 1 {
		return 1
	}

	rowNext := make([][]int, m)
	colNext := make([][]int, n)

	for i := 0; i < m; i++ {
		rowNext[i] = make([]int, n+1)
		for j := 0; j <= n; j++ {
			rowNext[i][j] = j
		}
	}

	for j := 0; j < n; j++ {
		colNext[j] = make([]int, m+1)
		for i := 0; i <= m; i++ {
			colNext[j][i] = i
		}
	}

	var findRow func(int, int) int
	findRow = func(r, x int) int {
		if rowNext[r][x] != x {
			rowNext[r][x] = findRow(r, rowNext[r][x])
		}
		return rowNext[r][x]
	}

	var findCol func(int, int) int
	findCol = func(c, x int) int {
		if colNext[c][x] != x {
			colNext[c][x] = findCol(c, colNext[c][x])
		}
		return colNext[c][x]
	}

	removeRow := func(r, x int) {
		rowNext[r][x] = findRow(r, x+1)
	}

	removeCol := func(c, x int) {
		colNext[c][x] = findCol(c, x+1)
	}

	queue := list.New()
	queue.PushBack(State{0, 0, 1})

	removeRow(0, 0)
	removeCol(0, 0)

	for queue.Len() > 0 {
		front := queue.Front()
		queue.Remove(front)

		cur := front.Value.(State)

		row := cur.row
		col := cur.col
		dist := cur.dist

		jump := grid[row][col]

		maxCol := col + jump
		if maxCol >= n {
			maxCol = n - 1
		}

		nextCol := findRow(row, col+1)

		for nextCol <= maxCol {
			if row == m-1 && nextCol == n-1 {
				return dist + 1
			}

			queue.PushBack(State{row, nextCol, dist + 1})

			removeRow(row, nextCol)
			removeCol(nextCol, row)

			nextCol = findRow(row, nextCol)
		}

		maxRow := row + jump
		if maxRow >= m {
			maxRow = m - 1
		}

		nextRow := findCol(col, row+1)

		for nextRow <= maxRow {
			if nextRow == m-1 && col == n-1 {
				return dist + 1
			}

			queue.PushBack(State{nextRow, col, dist + 1})

			removeCol(col, nextRow)
			removeRow(nextRow, col)

			nextRow = findCol(col, nextRow)
		}
	}

	return -1
}
```

The Go implementation uses a union-find style skipping structure instead of balanced trees.

Each row and column maintains a "next available index" pointer. Once a position is processed, it is unioned with the next index, allowing future searches to skip it immediately.

This achieves nearly constant amortized skipping performance while avoiding the need for external ordered set libraries.

Go slices are used for all structures, and recursion with path compression provides efficient lookups.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m * n * log(m * n)) | Each cell is processed once with efficient ordered operations |
| Space | O(m * n) | Queue and row/column tracking structures store all cells |

Each cell enters the BFS queue at most once because it is removed from the tracking structures immediately after discovery. Ordered set operations such as binary search and deletion require logarithmic time, leading to the final complexity.

The Go union-find optimization can achieve nearly linear practical performance because path compression aggressively skips processed positions.

## Edge Cases

One important edge case is a `1 x 1` grid. Since the start and destination are the same cell, the correct answer is `1`, not `0`. It is easy to accidentally treat this as requiring no movement and return the wrong value. The implementation handles this with an explicit early return.

Another important edge case is when many cells contain `0`. These cells cannot move anywhere, so BFS can quickly become trapped. A naive implementation might repeatedly revisit dead-end cells, but this solution removes cells from the tracking structures immediately after processing, preventing redundant exploration.

A third important edge case involves extremely large jump values. A cell may theoretically reach thousands of positions. Explicitly iterating over already processed cells repeatedly would cause time limit exceeded errors. The ordered skipping structures ensure that once a cell has been visited, it is never scanned again, preserving efficiency even in dense movement scenarios.
