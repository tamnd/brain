## Solution

## Problem Understanding

This problem gives us an undirected, connected, weighted graph with `n` nodes and a list of edges. Some edges already have fixed positive weights, while others have a weight of `-1`, meaning their values are unknown and can be assigned later.

Our task is to replace every `-1` edge with some positive integer between `1` and `2 * 10^9` so that the shortest path distance from `source` to `destination` becomes exactly equal to `target`.

We are allowed to choose any valid assignments for the unknown edges, as long as:

1. Every `-1` edge becomes a positive integer.
2. The shortest path from `source` to `destination` equals `target`.
3. Existing positive weights cannot be modified.

If no assignment can satisfy the target shortest path, we must return an empty array.

The graph is relatively small, since `n <= 100`, but the number of edges can still be dense, up to `n * (n - 1) / 2`. This strongly suggests that shortest path algorithms such as Dijkstra are appropriate. Since edge weights are non negative after modification, Dijkstra works perfectly.

The most important challenge is that modifying one edge can change the entire shortest path structure of the graph. We cannot greedily assign values without considering global effects.

There are several important edge cases:

- The existing graph without any `-1` edges may already produce a shortest path smaller than `target`. Since we can only increase unknown edges, we may not be able to force the shortest path upward if a fixed shorter path already exists.
- The graph may require multiple `-1` edges to participate in the final shortest path.
- Assigning small values to unknown edges may accidentally create an even shorter alternative path.
- Some `-1` edges may never need to be used, and can safely be assigned extremely large values to prevent them from affecting shortest paths.
- The shortest path may already equal `target` before modifying unknown edges.

The problem guarantees:

- The graph is connected.
- There are no duplicate edges.
- There are no self loops.

These guarantees simplify graph construction and shortest path computation.

## Approaches

### Brute Force Approach

A brute force solution would try every possible assignment for the `-1` edges and then run a shortest path algorithm to check whether the resulting graph has shortest distance exactly equal to `target`.

Suppose there are `k` unknown edges. Even if we restricted each edge to only a small range of candidate values, the number of combinations would grow exponentially.

For every assignment:

1. Replace all `-1` edges with candidate weights.
2. Run Dijkstra from `source`.
3. Check whether the shortest distance to `destination` equals `target`.

This approach is correct because it exhaustively checks all possibilities, but it is computationally infeasible.

The search space is enormous because each unknown edge can take values up to `2 * 10^9`.

### Key Insight

The critical observation is that we do not need to search all assignments.

Instead, we can reason about shortest paths strategically:

1. First, compute the shortest distance using only existing positive edges.
2. If this distance is already smaller than `target`, the answer is impossible because fixed edges already create a path that is too short.
3. Otherwise, we can gradually introduce unknown edges with carefully chosen values.

The optimal strategy is:

- Treat unknown edges as very large initially so they do not interfere.
- Then try assigning weight `1` to unknown edges one at a time.
- As soon as the shortest path becomes less than or equal to `target`, we know this edge participates in a potentially useful path.
- We can then increase that edge just enough so the shortest path becomes exactly `target`.
- All remaining unknown edges are assigned huge values so they never create shorter alternative paths.

This works because shortest paths behave monotonically with respect to edge weights. Lowering an edge weight can only decrease or preserve shortest path distances.

We repeatedly use Dijkstra because the graph is small enough.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | O(V + E) | Tries all assignments of unknown edges |
| Optimal | O(E * (V + E) log V) | O(V + E) | Repeated Dijkstra with greedy edge adjustment |

## Algorithm Walkthrough

1. Build a Dijkstra helper function that computes the shortest path from `source` to `destination`.

We use an adjacency list because it supports efficient traversal of neighbors. A priority queue is used to always process the currently closest node first.
2. Ignore all `-1` edges initially and compute the shortest path using only fixed positive edges.

This tells us whether a permanently short path already exists.
3. If the shortest distance is less than `target`, return an empty array.

This means there already exists a fixed path shorter than `target`, and since fixed edges cannot be modified, we can never force the shortest path to become exactly `target`.
4. If the shortest distance already equals `target`, assign all `-1` edges a huge value such as `2 * 10^9`.

This prevents unknown edges from creating shorter paths while preserving the existing shortest distance.
5. Otherwise, iterate through every edge with weight `-1`.
6. Temporarily set the current unknown edge weight to `1`.

We choose the minimum possible value because we want to determine whether this edge can help reduce the shortest path toward `target`.
7. Run Dijkstra again.
8. If the new shortest distance is still greater than `target`, keep this edge as `1` and continue.

This edge alone is not enough to reduce the path sufficiently.
9. If the new shortest distance becomes less than or equal to `target`, this edge is part of a useful shortest path.
10. Increase this edge weight by `target - current_distance`.

This adjustment precisely raises the shortest path to equal `target`.
11. Assign all remaining `-1` edges a huge value.

This prevents future edges from accidentally introducing shorter paths.
12. Return the modified edge list.
13. If no edge configuration achieves the target, return an empty array.

### Why it works

The algorithm relies on the monotonic nature of shortest paths.

Initially, unknown edges are effectively unusable. Then we gradually lower selected edges to see whether they can help reduce the shortest path.

Once a shortest path becomes less than or equal to `target`, increasing one participating edge by the exact difference raises the shortest path to exactly `target`.

Assigning all unused unknown edges huge values guarantees no alternative shorter paths appear later.

Because Dijkstra always computes the true shortest path, the final graph satisfies the required condition exactly.

## Python Solution

```python
from typing import List
import heapq

class Solution:
    def modifiedGraphEdges(
        self,
        n: int,
        edges: List[List[int]],
        source: int,
        destination: int,
        target: int
    ) -> List[List[int]]:

        INF = 2 * 10**9

        def dijkstra() -> int:
            graph = [[] for _ in range(n)]

            for u, v, w in edges:
                if w == -1:
                    continue

                graph[u].append((v, w))
                graph[v].append((u, w))

            distances = [float("inf")] * n
            distances[source] = 0

            min_heap = [(0, source)]

            while min_heap:
                current_dist, node = heapq.heappop(min_heap)

                if current_dist > distances[node]:
                    continue

                for neighbor, weight in graph[node]:
                    new_dist = current_dist + weight

                    if new_dist < distances[neighbor]:
                        distances[neighbor] = new_dist
                        heapq.heappush(min_heap, (new_dist, neighbor))

            return distances[destination]

        initial_distance = dijkstra()

        if initial_distance < target:
            return []

        if initial_distance == target:
            for edge in edges:
                if edge[2] == -1:
                    edge[2] = INF

            return edges

        for edge in edges:
            if edge[2] != -1:
                continue

            edge[2] = 1

            current_distance = dijkstra()

            if current_distance <= target:
                edge[2] += target - current_distance

                for remaining_edge in edges:
                    if remaining_edge[2] == -1:
                        remaining_edge[2] = INF

                return edges

        return []
```

The implementation begins by defining a reusable Dijkstra helper function. This function rebuilds the adjacency list every time using only edges whose weights are currently known.

Unknown edges with weight `-1` are skipped entirely during shortest path computation.

The first Dijkstra run determines whether the graph already contains a path shorter than `target`. If it does, the problem is impossible because fixed edges cannot be changed.

If the shortest path already equals `target`, all unknown edges receive extremely large values. This ensures they never affect the shortest path.

Otherwise, the algorithm iterates through every unknown edge. Each candidate edge is temporarily assigned weight `1`, which is the smallest legal value.

After each assignment, Dijkstra is run again. If the shortest path becomes less than or equal to `target`, we know this edge can help construct the desired path.

We then increase that edge weight by exactly the remaining difference so the shortest path becomes exactly `target`.

Finally, all remaining unknown edges are assigned huge values to prevent accidental shorter paths.

## Go Solution

```go
package main

import (
	"container/heap"
	"math"
)

type Pair struct {
	dist int
	node int
}

type MinHeap []Pair

func (h MinHeap) Len() int {
	return len(h)
}

func (h MinHeap) Less(i, j int) bool {
	return h[i].dist < h[j].dist
}

func (h MinHeap) Swap(i, j int) {
	h[i], h[j] = h[j], h[i]
}

func (h *MinHeap) Push(x interface{}) {
	*h = append(*h, x.(Pair))
}

func (h *MinHeap) Pop() interface{} {
	old := *h
	n := len(old)
	item := old[n-1]
	*h = old[:n-1]
	return item
}

func modifiedGraphEdges(
	n int,
	edges [][]int,
	source int,
	destination int,
	target int,
) [][]int {

	const INF = 2000000000

	dijkstra := func() int {
		graph := make([][][2]int, n)

		for _, edge := range edges {
			u, v, w := edge[0], edge[1], edge[2]

			if w == -1 {
				continue
			}

			graph[u] = append(graph[u], [2]int{v, w})
			graph[v] = append(graph[v], [2]int{u, w})
		}

		dist := make([]int, n)

		for i := range dist {
			dist[i] = math.MaxInt64
		}

		dist[source] = 0

		pq := &MinHeap{}
		heap.Init(pq)

		heap.Push(pq, Pair{0, source})

		for pq.Len() > 0 {
			current := heap.Pop(pq).(Pair)

			if current.dist > dist[current.node] {
				continue
			}

			for _, neighbor := range graph[current.node] {
				nextNode := neighbor[0]
				weight := neighbor[1]

				newDist := current.dist + weight

				if newDist < dist[nextNode] {
					dist[nextNode] = newDist
					heap.Push(pq, Pair{newDist, nextNode})
				}
			}
		}

		return dist[destination]
	}

	initialDistance := dijkstra()

	if initialDistance < target {
		return [][]int{}
	}

	if initialDistance == target {
		for i := range edges {
			if edges[i][2] == -1 {
				edges[i][2] = INF
			}
		}

		return edges
	}

	for i := range edges {
		if edges[i][2] != -1 {
			continue
		}

		edges[i][2] = 1

		currentDistance := dijkstra()

		if currentDistance <= target {
			edges[i][2] += target - currentDistance

			for j := range edges {
				if edges[j][2] == -1 {
					edges[j][2] = INF
				}
			}

			return edges
		}
	}

	return [][]int{}
}
```

The Go implementation follows the same logic as the Python version, but several language specific details are important.

Go does not provide a built in priority queue, so we implement one using the `container/heap` package.

The adjacency list uses slices of fixed size arrays `[2]int`, where each entry stores `(neighbor, weight)`.

Go integers are large enough for this problem on 64 bit systems, but using `math.MaxInt64` as the initial distance sentinel ensures safe comparisons.

Unlike Python, Go distinguishes between `nil` slices and empty slices. Returning `[][]int{}` explicitly returns an empty result rather than `nil`.

## Worked Examples

### Example 1

Input:

```
n = 5
edges = [[4,1,-1],[2,0,-1],[0,3,-1],[4,3,-1]]
source = 0
destination = 1
target = 5
```

Initially, all unknown edges are ignored.

The graph has no usable path from `0` to `1`.

| Step | Edge Modified | Shortest Distance |
| --- | --- | --- |
| Initial | none | infinity |
| 1 | [4,1] = 1 | infinity |
| 2 | [2,0] = 1 | infinity |
| 3 | [0,3] = 1 | infinity |
| 4 | [4,3] = 1 | 3 |

Now the shortest path is:

```
0 -> 3 -> 4 -> 1
1 + 1 + 1 = 3
```

We need distance `5`.

Difference:

```
5 - 3 = 2
```

Increase the current edge weight:

```
[4,3] becomes 3
```

Final path:

```
0 -> 3 -> 4 -> 1
1 + 3 + 1 = 5
```

Valid output:

```
[[4,1,1],[2,0,1],[0,3,1],[4,3,3]]
```

target = 6
```

Ignoring unknown edges:

```
0 -> 2 = 5
```

No modification can remove or increase this fixed path.

### Example 3

Input:

```
n = 4
edges = [[1,0,4],[1,2,3],[2,3,5],[0,3,-1]]
source = 0
destination = 2
target = 6
```

Ignoring unknown edges:

```
0 -> 1 -> 2

```

Since `7 > 6`, we need a shorter path.

Set edge `[0,3] = 1`.

Now shortest path becomes:

```
0 -> 3 -> 2
1 + 5 = 6
```

Target achieved exactly.

Final edges:

```
[[1,0,4],[1,2,3],[2,3,5],[0,3,1]]

| Measure | Complexity | Explanation |

| Space | O(V + E) | Adjacency list, heap, and distance array |

The graph contains at most about 5000 edges when `n = 100`, so repeated Dijkstra executions remain practical.

Each Dijkstra run costs:

```
O((V + E) log V)
```

In the worst case, we may run Dijkstra once for every edge.

The space usage comes from:

- The adjacency list
- The priority queue
- The distance array

All are linear in graph size.

## Edge Cases

One important edge case occurs when the graph already contains a fixed shortest path smaller than `target`. This is dangerous because no amount of modifying unknown edges can remove or lengthen that existing path. A naive implementation might still attempt modifications and incorrectly return a graph. The solution prevents this by running Dijkstra before any modifications and immediately returning an empty array if the shortest distance is already below `target`.

Another critical case occurs when the shortest path already equals `target` before using any unknown edges. In this scenario, unknown edges must not accidentally introduce a shorter alternative path. The implementation handles this by assigning all remaining unknown edges the maximum allowed value, effectively removing them from consideration in shortest path calculations.

A third tricky situation happens when adding one unknown edge with weight `1` suddenly creates a path much shorter than `target`. The algorithm correctly handles this by increasing that exact edge weight by the difference between `target` and the current shortest distance. This guarantees the final shortest distance becomes exactly equal to `target`.

A subtle edge case involves disconnected shortest path computations during intermediate steps. Since unknown edges are initially ignored, the graph used by Dijkstra may temporarily disconnect `source` and `destination`. The implementation naturally handles this because unreachable nodes retain infinite distance values, allowing the algorithm to continue testing additional unknown edges safely.
