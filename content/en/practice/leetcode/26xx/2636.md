## Solution

## Problem Understanding

This problem asks us to implement a concurrency-controlled executor for asynchronous functions. We are given an array called `functions`, where every element is itself a function that returns a Promise when called. We are also given an integer `n`, which represents the maximum number of promises that may be running simultaneously.

The goal is to create a function called `promisePool` that starts executing these asynchronous functions while respecting the concurrency limit. At no point should more than `n` promises remain pending.

The execution order matters. The functions must be started in the same order they appear in the input array. This means:

- `functions[0]` starts before `functions[1]`
- `functions[1]` starts before `functions[2]`
- and so on

However, completion order does not matter because promises may resolve at different times.

The important behavior is that whenever one promise finishes, another waiting function should immediately begin execution, as long as there are still unstarted functions remaining.

The returned promise should resolve only after every asynchronous task has completed.

The constraints are small:

- `functions.length <= 10`
- `n <= 10`

Because the input size is tiny, almost any reasonable solution will pass performance-wise. However, the problem is fundamentally about understanding asynchronous control flow and concurrency management rather than raw efficiency.

Several edge cases are important:

- The input array may be empty
- `n` may be larger than the number of functions
- `n` may equal `1`, forcing purely sequential execution
- Multiple promises may resolve nearly simultaneously
- The implementation must ensure that no more than `n` promises are active at any time

The problem also guarantees that promises never reject. This simplifies the implementation because we do not need explicit error handling or rejection propagation logic.

## Approaches

### Brute Force Approach

The most straightforward solution is to execute every function one after another using `await`.

In this approach, we loop through the array and wait for each promise to finish before starting the next one.

This guarantees correctness because:

- Functions execute in order
- At most one promise is active at a time
- The returned promise resolves after all functions finish

However, this ignores the concurrency limit. If `n > 1`, we are wasting potential parallelism.

For example, if three tasks take 300ms, 400ms, and 200ms respectively:

- Sequential execution takes `300 + 400 + 200 = 900ms`
- Proper pooling with `n = 2` finishes in `500ms`

So while the brute-force solution is simple and correct, it fails to fully utilize allowed concurrency.

### Optimal Approach

The key insight is that we should always keep the pool as full as possible.

Instead of waiting for all current promises to finish before starting more work, we immediately launch a new task whenever one finishes.

The algorithm maintains:

- An index pointing to the next function to start
- A count of currently active promises

Initially, we start up to `n` functions.

Whenever a promise resolves:

1. The active count decreases
2. Another waiting function is started immediately if available

This creates a dynamic worker pool where completed tasks continuously free slots for new tasks.

The solution works because:

- Functions are started strictly in array order
- The number of active promises never exceeds `n`
- Every function eventually gets executed
- The returned promise resolves only after all tasks complete

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m) | O(1) | Executes promises sequentially, ignores concurrency |
| Optimal | O(m) | O(n) | Maintains up to `n` concurrent promises |

Here, `m` represents the number of functions.

## Algorithm Walkthrough

1. Create a variable called `next_index` to track the next function that has not yet been started.
2. Create a variable called `active_tasks` to track how many promises are currently running.
3. Return a new Promise that resolves when all tasks complete.
4. Inside the Promise, define a helper function called `run_next`.
5. The helper first checks whether all functions have already been started and whether there are no active tasks remaining. If both conditions are true, every promise has completed, so we resolve the outer Promise.
6. While:

- there are still functions left to execute, and
- the number of active tasks is less than `n`

start more tasks.
7. Each time a task starts:

- increment `active_tasks`
- store the current function index
- increment `next_index`
8. Execute the asynchronous function immediately.
9. Attach a `.then()` handler to the promise. When it resolves:

- decrement `active_tasks`
- call `run_next()` again so another waiting task can begin
10. Start the initial batch by calling `run_next()` once.

### Why it works

The core invariant is that `active_tasks` never exceeds `n`.

Whenever a task finishes, exactly one slot becomes available in the pool. The algorithm immediately attempts to fill that slot with the next unstarted function.

Because `next_index` only increases, functions are always started in the correct order. Because each function is started exactly once, every task eventually executes. Finally, the outer Promise resolves only when all functions have been started and no active tasks remain, guaranteeing that all asynchronous work has completed.

## Python Solution

```python
from typing import List, Callable, Awaitable
import asyncio

class Solution:
    async def promisePool(
        self,
        functions: List[Callable[[], Awaitable[None]]],
        n: int
    ) -> None:
        
        next_index = 0
        active_tasks = 0
        
        loop = asyncio.get_event_loop()
        finished = loop.create_future()

        def run_next():
            nonlocal next_index, active_tasks

            if next_index == len(functions) and active_tasks == 0:
                if not finished.done():
                    finished.set_result(None)
                return

            while next_index < len(functions) and active_tasks < n:
                current_index = next_index
                next_index += 1
                active_tasks += 1

                async def execute(index=current_index):
                    nonlocal active_tasks

                    await functions[index]()
                    active_tasks -= 1
                    run_next()

                asyncio.create_task(execute())

        run_next()
        await finished
```

The implementation follows the worker-pool strategy directly.

The variable `next_index` tracks which function should start next. The variable `active_tasks` tracks how many promises are currently running.

A Future named `finished` is used to signal when all asynchronous work has completed.

The helper function `run_next()` is responsible for keeping the pool full. It repeatedly launches new tasks while:

- more functions remain
- the pool still has capacity

Each task is wrapped inside the `execute()` coroutine. After the promise resolves, the task count decreases and `run_next()` is called again so another waiting function can start immediately.

Finally, the main function waits for the `finished` Future, which resolves only after all tasks complete.

## Go Solution

```go
package main

import (
	"sync"
)

type AsyncFunc func() chan struct{}

func promisePool(functions []AsyncFunc, n int) chan struct{} {
	done := make(chan struct{})

	go func() {
		var wg sync.WaitGroup
		semaphore := make(chan struct{}, n)

		for _, fn := range functions {
			wg.Add(1)

			semaphore <- struct{}{}

			go func(f AsyncFunc) {
				defer wg.Done()

				<-f()

				<-semaphore
			}(fn)
		}

		wg.Wait()
		close(done)
	}()

	return done
}
```

The Go version uses goroutines and channels instead of promises.

A buffered channel called `semaphore` acts as the concurrency limiter. Before starting a goroutine, the code inserts a value into the semaphore channel. If the channel is already full, execution blocks until another goroutine finishes and removes a value.

A `sync.WaitGroup` waits for all goroutines to complete before closing the `done` channel.

Unlike Python's async/await model, Go uses lightweight goroutines and synchronization primitives for concurrency management.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m) | Each function is started and completed exactly once |
| Space | O(n) | At most `n` active asynchronous tasks exist simultaneously |

The time complexity is linear because every function is processed once. The algorithm does not repeatedly scan the array or perform nested work.

The space complexity is proportional to the concurrency limit because at most `n` promises may remain active simultaneously.

## Edge Cases

### Empty Function Array

If `functions` is empty, there are no asynchronous tasks to execute. A naive implementation might wait forever because no completion callback ever triggers.

The implementation handles this correctly by checking whether:

- all functions have been started, and
- there are zero active tasks

immediately when `run_next()` begins. In the empty-array case, both conditions are already true, so the promise resolves immediately.

### Pool Size Greater Than Number of Functions

When `n` exceeds the number of functions, every task should begin immediately. Some incorrect implementations accidentally wait for unused pool slots or overcomplicate scheduling logic.

The current solution naturally handles this because the startup loop continues until either:

- all functions are started, or
- the pool limit is reached

If fewer functions exist than pool slots, the loop simply starts them all.

### Sequential Execution with `n = 1`

When the concurrency limit is exactly one, the algorithm becomes fully sequential. This is an important stress case because it verifies that new tasks start only after the current task finishes.

The implementation correctly handles this because `active_tasks < n` prevents another task from starting until the current promise resolves and decrements the active count.
