## Solution

## Problem Understanding

We are given an undirected tree with `n` nodes. A tree is a connected graph with exactly `n - 1` edges and no cycles. Each node may or may not contain a coin. The `coins` array tells us whether a node contains a coin:

- `coins[i] == 1` means node `i` has a coin
- `coins[i] == 0` means node `i` has no coin

We may start from any node in the tree. During the traversal, we can repeatedly perform two operations:

1. Collect all coins within distance at most `2` from the current node
2. Move along an edge to an adjacent node

The goal is to collect every coin and return to the starting node while minimizing the total number of edge traversals.

An important detail is that collecting coins does not require visiting the exact node containing the coin. If a coin is within distance `2` from our current location, it is collected automatically. This changes the problem dramatically because we do not need to physically traverse every branch containing a coin.

The constraints are large:

- `n` can be as large as `3 * 10^4`
- The graph is guaranteed to be a valid tree

Because of the input size, any algorithm worse than linear or near-linear time will likely time out. We need an `O(n)` solution.

Several edge cases are important:

- No nodes contain coins
- Only one node contains a coin
- All nodes contain coins
- A long chain-shaped tree
- Coins located only on leaves
- Trees where many branches become unnecessary because coins can be collected from distance `2`

The tree property is extremely important because it guarantees unique paths between nodes, which enables pruning techniques and topological processing.

## Approaches

### Brute Force Approach

A brute-force strategy would try every possible starting node and determine the minimum traversal needed to collect all coins and return to the start.

One possible implementation would be:

- For each starting node:

- Simulate traversal across the tree
- Determine which nodes must be visited
- Compute the shortest closed walk that collects all coins

This quickly becomes extremely expensive because the number of traversal possibilities grows exponentially. Even if we optimize using BFS or DFS, repeatedly exploring large parts of the tree for every starting position is infeasible for `n = 30000`.

The brute-force method is correct because it explicitly explores all possibilities, but it is far too slow.

### Key Insight

The key observation is that many parts of the tree are completely irrelevant.

First, any leaf node without a coin can never help collect coins. Since it contributes nothing, we can safely remove it.

After repeatedly removing useless leaves, every remaining leaf must contain a coin or be necessary to connect coins.

Next comes the most important insight:

Because coins can be collected from distance `2`, we do not actually need to travel all the way to the remaining leaves containing coins.

If we remove the outermost two layers of leaves from the remaining tree, every remaining edge must still be traversed twice, once going out and once coming back.

This transforms the problem into a pruning problem on trees.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | O(n) | Tries many traversal possibilities |
| Optimal | O(n) | O(n) | Uses topological pruning on the tree |

## Algorithm Walkthrough

### Step 1: Build the Graph

We construct an adjacency list representation of the tree and maintain a degree array.

The adjacency list allows efficient neighbor traversal, while the degree array helps identify leaves quickly.

### Step 2: Remove Leaf Nodes Without Coins

We initialize a queue with every leaf node whose coin value is `0`.

These leaves are useless because:

- They contain no coin
- They only connect to one other node
- Visiting them cannot help collect additional coins

We repeatedly remove such leaves using a BFS-style pruning process.

Whenever removing a node causes another node to become a leaf with no coin, we also remove that node.

This step leaves only the minimal subtree necessary to connect all coins.

### Step 3: Remove Two Layers of Leaves

After the first pruning phase, all remaining leaves are important coin-related leaves.

However, because coins can be collected within distance `2`, we do not need to physically travel to those leaves.

We therefore remove leaves twice:

- First layer removal corresponds to collecting coins from distance `1`
- Second layer removal corresponds to collecting coins from distance `2`

We process all current leaves simultaneously for exactly two rounds.

### Step 4: Count Remaining Edges

After pruning:

- Any remaining edge must be traversed twice
- Once to enter that subtree
- Once to return

If `remaining_nodes` is the number of nodes left, then the remaining tree contains:

```
remaining_nodes - 1
```

edges.

Therefore, the answer becomes:

```
(remaining_nodes - 1) * 2
```

If no nodes remain, the answer is `0`.

### Why it works

The algorithm works because each pruning phase removes nodes that are provably unnecessary.

The first pruning phase removes subtrees that contain no coins at all. Traversing into those regions can never help.

The second pruning phase uses the distance-2 collection rule. Since coins can be collected without directly visiting the leaf nodes, the outer two layers of the coin-containing subtree do not need traversal.

After all pruning is complete, every remaining edge belongs to the essential traversal core. Since the traversal must return to the starting node, every remaining edge must be used exactly twice.

## Python Solution

```python
from collections import deque
from typing import List

class Solution:
    def collectTheCoins(self, coins: List[int], edges: List[List[int]]) -> int:
        n = len(coins)

        if n == 1:
            return 0

        graph = [[] for _ in range(n)]
        degree = [0] * n

        for u, v in edges:
            graph[u].append(v)
            graph[v].append(u)
            degree[u] += 1
            degree[v] += 1

        # Step 1:
        # Remove leaves without coins
        queue = deque()

        for node in range(n):
            if degree[node] == 1 and coins[node] == 0:
                queue.append(node)

        remaining_nodes = n

        while queue:
            node = queue.popleft()
            remaining_nodes -= 1

            for neighbor in graph[node]:
                degree[neighbor] -= 1

                if degree[neighbor] == 1 and coins[neighbor] == 0:
                    queue.append(neighbor)

            degree[node] = 0

        # Step 2:
        # Remove two layers of leaves
        queue = deque()

        for node in range(n):
            if degree[node] == 1:
                queue.append(node)

        for _ in range(2):
            size = len(queue)

            for _ in range(size):
                node = queue.popleft()
                remaining_nodes -= 1
                degree[node] = 0

                for neighbor in graph[node]:
                    if degree[neighbor] == 0:
                        continue

                    degree[neighbor] -= 1

                    if degree[neighbor] == 1:
                        queue.append(neighbor)

        if remaining_nodes <= 0:
            return 0

        return (remaining_nodes - 1) * 2
```

The implementation starts by building the adjacency list and degree array. The adjacency list enables efficient neighbor traversal, while the degree array helps identify leaves in constant time.

The first queue processes leaf nodes without coins. This repeatedly trims useless branches from the tree. Every time a node is removed, the degrees of its neighbors decrease, potentially creating new removable leaves.

After that pruning finishes, the second queue gathers all remaining leaves. These leaves correspond to coin-related boundary nodes. Because coins are collectible within distance `2`, we remove exactly two layers of leaves.

Finally, the remaining graph represents the essential traversal structure. Since every remaining edge must be traversed twice, the result is `(remaining_nodes - 1) * 2`.

## Go Solution

```go
package main

import "container/list"

func collectTheCoins(coins []int, edges [][]int) int {
	n := len(coins)

	if n == 1 {
		return 0
	}

	graph := make([][]int, n)
	degree := make([]int, n)

	for _, edge := range edges {
		u, v := edge[0], edge[1]

		graph[u] = append(graph[u], v)
		graph[v] = append(graph[v], u)

		degree[u]++
		degree[v]++
	}

	queue := list.New()

	// Step 1:
	// Remove leaves without coins
	for i := 0; i < n; i++ {
		if degree[i] == 1 && coins[i] == 0 {
			queue.PushBack(i)
		}
	}

	remainingNodes := n

	for queue.Len() > 0 {
		front := queue.Front()
		queue.Remove(front)

		node := front.Value.(int)
		remainingNodes--

		for _, neighbor := range graph[node] {
			degree[neighbor]--

			if degree[neighbor] == 1 && coins[neighbor] == 0 {
				queue.PushBack(neighbor)
			}
		}

		degree[node] = 0
	}

	// Step 2:
	// Remove two layers of leaves
	for i := 0; i < n; i++ {
		if degree[i] == 1 {
			queue.PushBack(i)
		}
	}

	for round := 0; round < 2; round++ {
		size := queue.Len()

		for i := 0; i < size; i++ {
			front := queue.Front()
			queue.Remove(front)

			node := front.Value.(int)
			remainingNodes--

			degree[node] = 0

			for _, neighbor := range graph[node] {
				if degree[neighbor] == 0 {
					continue
				}

				degree[neighbor]--

				if degree[neighbor] == 1 {
					queue.PushBack(neighbor)
				}
			}
		}
	}

	if remainingNodes <= 0 {
		return 0
	}

	return (remainingNodes - 1) * 2
}
```

The Go implementation follows the same logic as the Python version. The primary difference is queue handling. Go does not provide a built-in deque structure, so we use `container/list` as a queue.

Slices are used for adjacency lists, and integer overflow is not a concern because the answer is at most `2 * (n - 1)`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each node and edge is processed a constant number of times |
| Space | O(n) | Adjacency list, degree array, and queues require linear space |

Each node enters and leaves the queue at most once during each pruning phase. Every edge is examined only a constant number of times while updating degrees. Since a tree has exactly `n - 1` edges, the total work remains linear.

## Edge Cases

One important edge case occurs when the tree contains no coins at all. A naive implementation might still attempt traversal or count edges incorrectly. In this situation, the first pruning phase removes every node because all leaves are coinless, and the process cascades inward until nothing remains. The implementation correctly returns `0` because there is nothing to collect.

Another important case is a single-node tree. Since there are no edges, traversal cost must always be `0`, regardless of whether the node contains a coin. The implementation explicitly handles `n == 1` at the beginning to avoid unnecessary processing and edge-counting issues.

A subtle edge case appears when all coins are already collectible from the starting position without movement. For example, in a star graph, starting at the center collects every leaf coin immediately because all leaves are within distance `1`. The second pruning phase correctly removes all outer layers, leaving no remaining traversal core and producing answer `0`.

A more difficult scenario involves long chain-shaped trees with coins far apart. In such cases, some internal edges genuinely must be traversed. The pruning process removes only unnecessary outer layers while preserving the essential middle segment. This guarantees the algorithm does not over-prune and underestimate the required traversal distance.
