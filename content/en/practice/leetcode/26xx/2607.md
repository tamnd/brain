# LeetCode 2607 - Make K-Subarray Sums Equal

## Problem Understanding

You are given a circular array `arr` and an integer `k`. Because the array is circular, index `0` comes after the last index, and the last index comes before index `0`.

The goal is to make every subarray of length `k` have the same sum. You are allowed to increase or decrease any element by `1` in a single operation, and you may perform as many operations as needed.

The task is to compute the minimum number of operations required.

To understand the condition more clearly, suppose the array length is `n`. Every contiguous subarray of length `k` must have identical sums:

- Sum of indices `[0, 1, ..., k-1]`
- Sum of indices `[1, 2, ..., k]`
- Sum of indices `[2, 3, ..., k+1]`
- and so on, wrapping around because the array is circular

The constraints are important:

- `1 <= n <= 10^5`
- `1 <= arr[i] <= 10^9`

These limits immediately rule out any brute-force simulation that repeatedly adjusts elements and recomputes sums. An efficient mathematical observation is required.

There are several important edge cases to keep in mind:

- When `k = n`, every subarray of length `k` is the whole array, so all sums are already equal and the answer is `0`.
- When `k = 1`, every individual element must become equal.
- When `gcd(n, k) > 1`, the array splits into independent cycles, which is the key insight behind the optimal solution.
- Large values up to `10^9` mean we should avoid approaches that depend on value ranges.

## Approaches

### Brute Force Approach

A naive approach would repeatedly compare all circular subarrays of length `k` and attempt to modify elements until all sums become equal.

For example, we could:

1. Compute every subarray sum.
2. Detect differences between neighboring subarray sums.
3. Adjust elements incrementally.
4. Repeat until all sums match.

This is impractical because each operation may affect multiple subarrays, and the number of operations could become extremely large. Even recomputing all subarray sums after every adjustment would already be too expensive.

Another brute-force variation would attempt all target configurations, but the search space is exponential.

Therefore, brute force is not feasible for `n = 10^5`.

### Key Insight

The crucial observation comes from comparing consecutive subarrays.

Let:

- `S(i)` be the sum of the circular subarray of length `k` starting at index `i`

If all subarray sums are equal, then:

$$S(i) = S(i+1)$$

Subtracting the two equations gives:

$$arr[i] = arr[(i+k) \bmod n]$$

This means every element must equal the element `k` positions ahead in the circular array.

That creates groups of indices connected by repeated jumps of size `k`.

These groups form cycles determined by:

$$g = \gcd(n, k)$$

All indices inside the same cycle must become equal.

Now the problem becomes:

- Partition the array into cycles.
- Make every value inside each cycle equal.
- Minimize total adjustment cost.

For minimizing absolute differences, the optimal target value is the median of the cycle.

## Algorithm Walkthrough

### Optimal Algorithm

1. Compute `g = gcd(n, k)`.

The array splits into exactly `g` independent cycles. Each cycle contains indices connected by repeatedly adding `k` modulo `n`.
2. For each starting position `0` through `g-1`, build one cycle.

Starting from index `start`, repeatedly move to:

$$(current + k) \bmod n$$

until returning to the starting index.
3. Collect all values in the current cycle.

Every value in this cycle must become equal in the final valid array.
4. Sort the cycle values.

To minimize the total number of increment/decrement operations, we need the value minimizing:

$$\sum |x_i - target|$$

The median minimizes this expression.
5. Choose the median value.

If the cycle has length `m`, use:

$$median = values[m // 2]$$
6. Compute the cost for this cycle.

Sum:

$$|value - median|$$

for every element in the cycle.
7. Add all cycle costs together.

Since cycles are independent, the total minimum cost is the sum of the minimum costs of each cycle.

### Why it works

The equality of all length-`k` subarray sums forces:

$$arr[i] = arr[(i+k) \bmod n]$$

This creates equivalence classes of indices. Every index inside the same class must share one final value.

For minimizing total absolute adjustment cost within a group, the median is mathematically optimal. Since the groups are independent, optimizing each one separately yields the global optimum.

## Python Solution

```python
from typing import List
from math import gcd

class Solution:
    def makeSubKSumEqual(self, arr: List[int], k: int) -> int:
        n = len(arr)
        group_count = gcd(n, k)

        total_operations = 0

        for start in range(group_count):
            group = []

            index = start
            while True:
                group.append(arr[index])
                index = (index + k) % n

                if index == start:
                    break

            group.sort()

            median = group[len(group) // 2]

            for value in group:
                total_operations += abs(value - median)

        return total_operations
```

The implementation begins by computing `gcd(n, k)`, which determines how many independent cycles exist.

For each cycle, the code repeatedly jumps forward by `k` modulo `n`. This traversal guarantees that every index belonging to the same equivalence class is collected exactly once.

The group values are sorted so the median can be selected efficiently. Once the median is chosen, the total adjustment cost is computed using absolute differences.

Finally, all cycle costs are added together and returned.

## Go Solution

```go
package main

import (
	"math"
	"sort"
)

func gcd(a, b int) int {
	for b != 0 {
		a, b = b, a%b
	}
	return a
}

func makeSubKSumEqual(arr []int, k int) int64 {
	n := len(arr)
	groupCount := gcd(n, k)

	var totalOperations int64 = 0

	for start := 0; start < groupCount; start++ {
		group := []int{}

		index := start

		for {
			group = append(group, arr[index])

			index = (index + k) % n

			if index == start {
				break
			}
		}

		sort.Ints(group)

		median := group[len(group)/2]

		for _, value := range group {
			totalOperations += int64(math.Abs(float64(value - median)))
		}
	}

	return totalOperations
}
```

The Go implementation follows the same logic as the Python version.

One important difference is integer handling. Since the answer may exceed the range of a 32-bit integer, the return type is `int64`.

Go also requires an explicit `gcd` helper function and uses `sort.Ints` for sorting cycle values.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | Sorting dominates total runtime |
| Space | O(n) | Storage for cycle groups |

The array is partitioned into disjoint cycles, and every element appears in exactly one cycle. Across all cycles, the total number of elements processed is `n`.

Sorting each cycle contributes the logarithmic factor. In the worst case, one cycle may contain all `n` elements, producing `O(n log n)` total complexity.

## Edge Cases

### Case 1: `k = n`

When `k` equals the array length, every circular subarray of length `k` contains the entire array. Therefore all subarray sums are automatically equal, regardless of the values.

A buggy implementation might still perform unnecessary grouping work or accidentally modify values. This implementation handles the case naturally because every cycle has length `1`, producing zero cost.

### Case 2: `k = 1`

When `k = 1`, every subarray consists of a single element. Therefore all array elements must become identical.

This is effectively the classic problem of minimizing absolute differences by converting everything to the median. The implementation correctly forms one large cycle and computes the optimal median cost.

### Case 3: Multiple Independent Cycles

When `gcd(n, k) > 1`, the array splits into several disconnected groups.

For example:

```
n = 6
k = 2
```

creates two separate cycles:

```
0 → 2 → 4
1 → 3 → 5
```

A common bug is assuming the entire array must converge to one value. That produces incorrect answers because different cycles may end at different target values. The implementation processes each cycle independently, ensuring correctness.

### Case 4: Even-Length Groups

For even-sized groups, there are two middle elements after sorting.

Any value between them minimizes the total absolute deviation. The implementation simply selects:

```
group[len(group) // 2]
```

which is always valid and optimal.
