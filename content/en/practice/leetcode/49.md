## Solution

## Problem Understanding

The problem asks us to take an array of strings and group together words that are anagrams of each other.

An anagram is a word or phrase formed by rearranging the letters of another word while using exactly the same characters and the same number of occurrences of each character. For example, `"eat"`, `"tea"`, and `"ate"` are anagrams because they all contain the same letters: `a`, `e`, and `t`.

The input is an array of strings called `strs`. Each element is a lowercase English string, and the goal is to return a list of groups, where each group contains strings that are anagrams of one another.

The order of the output does not matter. Similarly, the order of words inside each group does not matter. This flexibility simplifies implementation because we do not need to sort the final result.

The constraints provide important clues about performance requirements:

- The number of strings can be as large as `10^4`
- Each string can be up to length `100`
- Strings only contain lowercase English letters

A naive comparison between every pair of strings would become too expensive when `strs.length` grows large. We need an approach that can efficiently identify whether words belong to the same anagram group.

Several edge cases are important to recognize upfront. Empty strings are valid inputs, so `[""]` should correctly produce `[[""]]`. Single character strings should be grouped correctly. Duplicate strings may appear and should remain in the same group. Strings with no anagrams, such as `"bat"` in Example 1, should still appear as a group containing only themselves.

## Approaches

### Brute Force Approach

A straightforward solution is to compare every string with every other string and determine whether they are anagrams.

To check whether two strings are anagrams, we can sort both strings and compare the results. For example:

- `"eat"` → `"aet"`
- `"tea"` → `"aet"`

Since the sorted forms match, the strings are anagrams.

The brute force method works as follows:

1. Start with an ungrouped string.
2. Compare it against every remaining string.
3. If another string is an anagram, place it in the same group.
4. Mark grouped strings as visited.
5. Continue until every string has been processed.

This approach is correct because every pair of strings is explicitly checked. However, it is inefficient because it repeatedly compares many strings and repeatedly sorts them.

If there are `n` strings and each comparison involves sorting strings of average length `k`, the runtime becomes too large for `10^4` strings.

### Optimal Approach, Hash Map with Canonical Representation

The key insight is that all anagrams share the same canonical representation.

If we sort the characters of a word, every anagram produces the same result:

- `"eat"` → `"aet"`
- `"tea"` → `"aet"`
- `"ate"` → `"aet"`

This sorted string can act as a unique identifier, or key, for a group of anagrams.

We use a hash map where:

- The key is the sorted version of the string
- The value is a list of words belonging to that anagram group

As we iterate through the input:

- Compute the sorted version of the current word
- Insert the word into the corresponding hash map entry
- If the key does not exist yet, create a new group

At the end, all values in the hash map represent the grouped anagrams.

This works efficiently because hash map insertion and lookup are approximately `O(1)` operations.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n² × k log k) | O(n) | Compares each string with many others and repeatedly sorts |
| Optimal | O(n × k log k) | O(n × k) | Uses a hash map keyed by sorted strings |

Where:

- `n` = number of strings
- `k` = average length of a string

## Algorithm Walkthrough

1. Create an empty hash map where the key represents an anagram signature and the value stores all strings sharing that signature.
2. Iterate through every string in `strs`. Each word must be assigned to exactly one anagram group.
3. For the current string, sort its characters alphabetically. This sorted result becomes the canonical representation of the word. For example, `"tan"` becomes `"ant"`.
4. Use the sorted string as a hash map key. If the key already exists, append the current word to the existing list. If not, create a new list.
5. Continue until all words have been processed. Every anagram will naturally end up under the same key because sorting identical character sets produces identical strings.
6. Return all hash map values as the final grouped result.

The hash map is a natural choice because it provides fast insertion and lookup. Instead of repeatedly comparing strings to determine group membership, we immediately know where a word belongs by computing its signature.

## Python Solution

```
from typing import List
from collections import defaultdict

class Solution:
    def groupAnagrams(self, strs: List[str]) -> List[List[str]]:
        anagram_groups = defaultdict(list)

        for word in strs:
            sorted_word = ''.join(sorted(word))
            anagram_groups[sorted_word].append(word)

        return list(anagram_groups.values())
```

The implementation starts by creating a `defaultdict(list)`. This is useful because it automatically creates an empty list whenever a new key is encountered, which avoids manual existence checks.

For each word in the input array, the code sorts the characters and joins them back into a string. This sorted string becomes the hash map key.

The current word is appended to the appropriate list in `anagram_groups`. Since all anagrams produce the same sorted representation, they naturally accumulate together.

Finally, the function returns all grouped lists using `list(anagram_groups.values())`.

## Go Solution

```
import "sort"

func groupAnagrams(strs []string) [][]string {
	anagramGroups := make(map[string][]string)

	for _, word := range strs {
		characters := []rune(word)

		sort.Slice(characters, func(i, j int) bool {
			return characters[i] < characters[j]
		})

		sortedWord := string(characters)

		anagramGroups[sortedWord] = append(
			anagramGroups[sortedWord],
			word,
		)
	}

	result := make([][]string, 0, len(anagramGroups))

	for _, group := range anagramGroups {
		result = append(result, group)
	}

	return result
}
```

The Go implementation follows the same logic as the Python version, but there are language-specific differences.

Go strings are immutable, so we convert each string into a `[]rune` slice before sorting. We then use `sort.Slice` to sort the characters in ascending order.

Go maps require explicit initialization using `make`, unlike Python's `defaultdict`. Appending to a nil slice is safe in Go, so `append` works even when the key does not already exist.

Finally, because Go maps do not provide direct access to values as a list, we explicitly iterate through the map and build the result slice.

## Worked Examples

### Example 1

**Input:**

```
["eat","tea","tan","ate","nat","bat"]
```

We maintain a hash map called `anagram_groups`.

| Current Word | Sorted Key | Hash Map State |
| --- | --- | --- |
| `"eat"` | `"aet"` | `{ "aet": ["eat"] }` |
| `"tea"` | `"aet"` | `{ "aet": ["eat", "tea"] }` |
| `"tan"` | `"ant"` | `{ "aet": ["eat", "tea"], "ant": ["tan"] }` |
| `"ate"` | `"aet"` | `{ "aet": ["eat", "tea", "ate"], "ant": ["tan"] }` |
| `"nat"` | `"ant"` | `{ "aet": ["eat", "tea", "ate"], "ant": ["tan", "nat"] }` |
| `"bat"` | `"abt"` | `{ "aet": ["eat", "tea", "ate"], "ant": ["tan", "nat"], "abt": ["bat"] }` |

Final result:

```
[["eat","tea","ate"],["tan","nat"],["bat"]]
```

The order may vary.

### Example 2

```
[""]
```

| Current Word | Sorted Key | Hash Map State |
| --- | --- | --- |
| `""` | `""` | `{ "": [""] }` |

Final result:

```
[[""]]
```

The empty string is still a valid anagram group.

### Example 3

**Input:**

```
["a"]
```

| Current Word | Sorted Key | Hash Map State |
| --- | --- | --- |
| `"a"` | `"a"` | `{ "a": ["a"] }` |

Final result:


A single character string forms its own group.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n × k log k) | Each string is sorted independently |
| Space | O(n × k) | Hash map stores all input strings and sorted keys |

The dominant cost comes from sorting each string. If a string has length `k`, sorting takes `O(k log k)`. Since we process `n` strings, the total runtime becomes `O(n × k log k)`.

The space complexity is `O(n × k)` because the hash map stores all strings grouped together, and the sorted keys also require additional memory proportional to the total input size.

## Edge Cases

### Empty String

An empty string may seem unusual because it contains no characters. A buggy implementation might fail when sorting or constructing a key.

For input:

```
[""]
```

Sorting produces another empty string, which becomes a valid hash map key. The implementation correctly groups it as:

```
[[""]]
```

### Single Character Strings

Strings containing only one character cannot be rearranged into anything different, but multiple identical letters should still group together correctly.

For example:

```
["a", "b", "a"]
```

The sorted keys are simply `"a"` and `"b"`, so the result becomes:

```
[["a", "a"], ["b"]]
```

### No Anagrams Exist

Some inputs contain no matching anagrams at all.

For example:

```
["abc", "def", "ghi"]
```

Each string generates a unique sorted key, meaning every word becomes its own group.

The implementation handles this naturally because the hash map simply creates one entry per unique key.

### Duplicate Strings

Duplicate words should remain grouped together.

For example:

```
["abc", "bca", "abc"]
```

All strings produce the key `"abc"`, so duplicates are appended into the same list:

```
[["abc", "bca", "abc"]]
```

The implementation preserves all occurrences correctly without removing duplicates.
