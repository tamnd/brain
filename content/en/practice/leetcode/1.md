## Solution

## Problem Understanding

The problem asks us to find two distinct numbers in an integer array whose sum equals a given `target` value, and return their indices. Importantly, the question is not asking for the numbers themselves, but rather their positions in the array.

The input consists of two parts. The first input, `nums`, is an array of integers that may contain positive numbers, negative numbers, or duplicates. The second input, `target`, is the integer sum we are trying to form using exactly two different elements from `nums`.

The expected output is a list or slice containing the indices of the two elements whose values add up to `target`. The order of the indices does not matter, meaning `[0,1]` and `[1,0]` are both valid answers if they refer to the correct pair.

A key guarantee provided by the problem is that there is **exactly one valid solution**. This simplifies implementation because we do not need to handle multiple possible answers or cases where no solution exists. Additionally, the same element cannot be used twice, which means we cannot reuse the same index even if doubling a value would equal the target.

The constraints are also important to analyze:

- `2 <= nums.length <= 10^4`
- `-10^9 <= nums[i] <= 10^9`
- `-10^9 <= target <= 10^9`

An array size of up to `10^4` is relatively moderate, but it is large enough that a quadratic solution may become inefficient. A naive `O(n²)` approach would require up to roughly 100 million comparisons in the worst case, which is unnecessarily expensive for such a straightforward problem.

There are several edge cases worth identifying early. Duplicate values can cause mistakes if an implementation incorrectly overwrites or reuses indices. For example, `[3,3]` with target `6` requires using two separate occurrences of the same number. Negative numbers can also matter because complements may be negative or cross zero. Finally, a naive implementation may accidentally use the same element twice if it checks for a complement incorrectly before accounting for index uniqueness. Fortunately, the problem guarantees exactly one valid answer, which allows us to stop searching immediately once the correct pair is found.

## Approaches

### Brute Force Approach

The most straightforward solution is to check every possible pair of numbers in the array and determine whether their sum equals the target.

We can accomplish this using two nested loops. The outer loop selects one number, and the inner loop checks every number that comes after it. For each pair `(i, j)`, we calculate:

```
nums[i] + nums[j]
```

If the sum equals `target`, we immediately return the indices `[i, j]`.

This approach is correct because it exhaustively checks all possible pairs, guaranteeing that the valid solution will eventually be found. Since the problem guarantees exactly one valid answer, we can safely stop once we encounter it.

The downside is efficiency. For an array of size `n`, the algorithm checks roughly:

$$\frac{n(n-1)}{2}$$

pairs in the worst case, which gives a time complexity of `O(n²)`. This becomes inefficient as `n` grows.

### Optimal Approach, Hash Map

The key observation is that for any number `x`, we already know exactly what value we need to complete the target:

```
complement = target - x
```

Instead of searching the entire array for this complement every time, we can store previously seen numbers in a hash map.

The hash map will store:

```
number -> index
```

As we iterate through the array:

1. Compute the complement needed.
2. Check if the complement already exists in the hash map.
3. If it exists, we have found the answer.
4. Otherwise, store the current number and its index for future lookups.

Hash maps provide average `O(1)` lookup time, allowing us to solve the problem in a single pass through the array.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | `O(n²)` | `O(1)` | Checks every possible pair using nested loops |
| Optimal | `O(n)` | `O(n)` | Uses a hash map for constant-time complement lookup |

## Algorithm Walkthrough

The optimal algorithm proceeds as follows:

1. Create an empty hash map that will store numbers we have already seen and their corresponding indices. This allows constant-time lookups when searching for complements.
2. Iterate through the array one element at a time. At each index, extract the current number.
3. Compute the complement using the formula:

```
complement = target - current_number
```

This tells us exactly what number we need in order to reach the target sum.

1. Check whether the complement already exists in the hash map. If it does, that means we have previously encountered a number that pairs with the current number to form the target.
2. Return the stored index of the complement along with the current index. Since the problem guarantees exactly one valid solution, we can immediately stop.
3. If the complement is not found, insert the current number and its index into the hash map. This ensures future elements can use it as a complement.
4. Continue iterating until the answer is found.

The reason we check the hash map **before inserting the current element** is to avoid using the same element twice. Consider:

```
nums = [3,3]
target = 6
```

If we inserted first, we might accidentally match an element with itself. By checking first, we ensure we only pair with previously encountered indices.

## Python Solution

```
from typing import List

class Solution:
    def twoSum(self, nums: List[int], target: int) -> List[int]:
        seen_numbers = {}

        for index, current_number in enumerate(nums):
            complement = target - current_number

            if complement in seen_numbers:
                return [seen_numbers[complement], index]

            seen_numbers[current_number] = index

        return []
```

The implementation begins by initializing an empty dictionary named `seen_numbers`. This dictionary acts as our hash map and stores numbers as keys and their indices as values.

We then iterate through the array using `enumerate`, which gives us both the current index and current number simultaneously.

For each element, we calculate the complement needed to reach the target. Before storing the current number, we check whether the complement already exists in `seen_numbers`.

If the complement exists, we immediately return the stored index along with the current index. This works because the earlier number and the current number together produce the target.

If the complement does not exist, we insert the current number into the dictionary so future iterations can reference it.

The final `return []` is technically unnecessary because the problem guarantees one valid solution, but it makes the function syntactically complete and safe.

## Go Solution

```
func twoSum(nums []int, target int) []int {
	seenNumbers := make(map[int]int)

	for index, currentNumber := range nums {
		complement := target - currentNumber

		if complementIndex, exists := seenNumbers[complement]; exists {
			return []int{complementIndex, index}
		}

		seenNumbers[currentNumber] = index
	}

	return []int{}
}
```

The Go implementation follows the exact same algorithmic idea as the Python version, but uses Go's built-in `map[int]int` type for hash map storage.

One small Go-specific detail is the use of the two-value map lookup syntax:

```
value, exists := map[key]
```

This allows us to determine whether a complement exists in the map without ambiguity.

Another difference is that Go returns a slice (`[]int`) instead of a Python list. Returning an empty slice at the end is included for completeness, even though the problem guarantees that execution will always return earlier.

Integer overflow is not a concern here because the constraints fit safely within Go's `int` range on LeetCode systems.

## Worked Examples

### Example 1

**Input:**

```
nums = [2,7,11,15]
target = 9
```

| Iteration | Index | Current Number | Complement Needed | Hash Map Before | Action |
| --- | --- | --- | --- | --- | --- |
| 1 | 0 | 2 | 7 | `{}` | Store `2:0` |
| 2 | 1 | 7 | 2 | `{2:0}` | Found complement, return `[0,1]` |

At index `0`, the number `2` requires a complement of `7`, but nothing exists in the hash map yet, so we store `2`.

At index `1`, the number `7` requires a complement of `2`. Since `2` already exists in the hash map at index `0`, we return `[0,1]`.

### Example 2

**Input:**

```
nums = [3,2,4]

```

| Iteration | Index | Current Number | Complement Needed | Hash Map Before | Action |
| --- | --- | --- | --- | --- | --- |
| 1 | 0 | 3 | 3 | `{}` | Store `3:0` |
| 2 | 1 | 2 | 4 | `{3:0}` | Store `2:1` |
| 3 | 2 | 4 | 2 | `{3:0, 2:1}` | Found complement, return `[1,2]` |

The algorithm first stores `3`, then stores `2`. When it reaches `4`, the required complement is `2`, which already exists at index `1`.

### Example 3

**Input:**


| Iteration | Index | Current Number | Complement Needed | Hash Map Before | Action |
| --- | --- | --- | --- | --- | --- |
| 1 | 0 | 3 | 3 | `{}` | Store `3:0` |
| 2 | 1 | 3 | 3 | `{3:0}` | Found complement, return `[0,1]` |

This example demonstrates why checking before inserting is important. The second `3` correctly matches the first `3`, rather than accidentally matching itself.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | `O(n)` | We traverse the array once, and each hash map lookup is `O(1)` on average |
| Space | `O(n)` | In the worst case, we may store nearly every element in the hash map |

The time complexity is linear because every element is processed exactly once. Hash map insertion and lookup operations are constant time on average, so the total work scales proportionally with the size of the input.

The space complexity is linear because, in the worst case, the algorithm may need to store almost all elements before finding the solution.

## Edge Cases

### Duplicate Numbers

An important edge case occurs when the valid pair contains duplicate values, such as:

```
nums = [3,3]
target = 6
```

A careless implementation may accidentally reuse the same index twice. Our solution avoids this by checking for the complement **before** inserting the current number into the hash map. This guarantees that only previously seen elements can form a pair.

### Negative Numbers

The array may contain negative values:

```
nums = [-1,-2,-3,-4,-5]
target = -8
```

The correct answer is `[-3, -5]`, corresponding to indices `[2,4]`. Since the algorithm relies only on subtraction and hash map lookups, negative values are handled naturally without requiring any special logic.

### Complement Appears Later

Sometimes the complement does not appear until later in the array:

```
nums = [1,5,8,2]
target = 10
```

When the algorithm encounters `8`, the required complement `2` has not yet appeared, so `8` is stored. Later, when `2` is encountered, the complement `8` is found in the hash map, and the correct indices are returned. This demonstrates why storing previously seen elements is effective.

### Large Inputs Near Constraint Limits

With arrays approaching `10^4` elements, a brute-force solution becomes increasingly expensive because it performs quadratic comparisons. The hash map solution remains efficient because it only performs one pass through the array, making it practical even near the upper constraint limit.
