## Solution

## Problem Understanding

The problem asks us to multiply two non-negative integers where each integer is given as a string instead of a numeric type. The result must also be returned as a string.

At first glance, this may seem straightforward because most programming languages already support integer multiplication. However, the problem explicitly forbids converting the entire string into an integer directly or using any built-in big integer library. This restriction is important because the input length can be as large as 200 digits, which may exceed the range of standard integer types in many languages.

The input consists of two strings:

- `num1`, representing a non-negative integer
- `num2`, representing another non-negative integer

Each character in the strings is guaranteed to be a digit from `'0'` to `'9'`.

The output should be a string representation of the product of these two numbers.

For example:

```
"123" × "456" = "56088"
```

The challenge is to simulate manual multiplication, the same way multiplication is performed on paper.

The constraints are important:

- Each string can contain up to 200 digits
- Leading zeros do not exist unless the entire number is `"0"`

These constraints immediately suggest that direct integer conversion is unsafe or disallowed. Instead, we must process digits individually.

Several edge cases are important:

- One or both numbers may be `"0"`
- Multiplication may generate carries across many positions
- The result can be much longer than either input
- Digits must align correctly according to place value

A naive implementation often fails because of incorrect carry handling or incorrect index placement when storing partial multiplication results.

## Approaches

### Brute Force Approach

The brute force idea is to simulate elementary school multiplication literally.

For every digit in `num1`, we multiply it with every digit in `num2`, generate a partial product string, shift it according to place value, and then repeatedly add all partial results together using string addition.

This approach is correct because it mirrors the mathematical definition of multiplication. Every digit pair contributes to the final result according to its positional value.

However, this approach is inefficient because:

- We create many intermediate strings
- We repeatedly perform string addition
- Each addition may require traversing long strings

As the input size grows, repeated string construction becomes expensive.

### Optimal Approach

The key insight is that we do not actually need to build intermediate strings.

When multiplying two numbers manually:

```
      123
   ×  456
```

each digit multiplication contributes directly to a position in the final result.

If:

```
num1[i] * num2[j]
```

is multiplied, its contribution belongs to:

```
i + j + 1
```

in the result array.

This observation allows us to:

- Preallocate an array of size `m + n`
- Store all partial sums directly
- Handle carries afterward

This avoids repeated string additions and reduces overhead significantly.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m × n + repeated additions) | O(m + n) | Simulates manual multiplication with intermediate strings |
| Optimal | O(m × n) | O(m + n) | Stores products directly into a result array |

## Algorithm Walkthrough

1. Handle the special case where either input is `"0"`.

If one of the numbers is zero, the result is immediately `"0"`.
2. Create a result array of length `len(num1) + len(num2)`.

The largest possible product of an `m` digit number and an `n` digit number has at most `m + n` digits.
3. Traverse both strings from right to left.

We process digits from least significant to most significant because multiplication naturally starts from the units place.
4. Convert characters into integer digits.

For each pair of digits:

```
digit1 = int(num1[i])
digit2 = int(num2[j])
```
5. Multiply the digits.

Compute:

```
product = digit1 * digit2
```
6. Determine where this product belongs.

The multiplication contributes to two positions:

```
position_low = i + j + 1
position_high = i + j
```

The lower position stores the current digit.

The higher position stores the carry.
7. Add the multiplication result into the result array.

We add:

```
total = product + result[position_low]
```

This includes any previously accumulated value.
8. Store the carry and remainder.

```
result[position_low] = total % 10
result[position_high] += total // 10
```
9. Remove leading zeros.

Since the result array was preallocated, it may contain leading zeros.
10. Convert the result array into a string.

Join the digits together and return the final string.

## Python Solution

```
class Solution:
    def multiply(self, num1: str, num2: str) -> str:
        if num1 == "0" or num2 == "0":
            return "0"

        m = len(num1)
        n = len(num2)

        result = [0] * (m + n)

        for i in range(m - 1, -1, -1):
            digit1 = ord(num1[i]) - ord('0')

            for j in range(n - 1, -1, -1):
                digit2 = ord(num2[j]) - ord('0')

                product = digit1 * digit2

                position_low = i + j + 1
                position_high = i + j

                total = product + result[position_low]

                result[position_low] = total % 10
                result[position_high] += total // 10

        start = 0

        while start < len(result) and result[start] == 0:
            start += 1

        return ''.join(str(digit) for digit in result[start:])
```

The implementation begins by handling the zero case immediately. This prevents unnecessary computation and avoids returning strings like `"0000"`.

The `result` array is preallocated with length `m + n`. Each index represents a decimal position in the final answer.

The nested loops process digits from right to left, exactly like manual multiplication. For every digit pair, we compute the product and place it into the correct positions.

The lower index stores the current digit after modulus operation, while the higher index accumulates the carry.

After all multiplications are complete, leading zeros are skipped. Finally, the digits are joined into a single string.

Using `ord()` instead of `int()` for digit conversion is a small optimization because character arithmetic is slightly faster.

## Go Solution

```
func multiply(num1 string, num2 string) string {
    if num1 == "0" || num2 == "0" {
        return "0"
    }

    m := len(num1)
    n := len(num2)

    result := make([]int, m+n)

    for i := m - 1; i >= 0; i-- {
        digit1 := int(num1[i] - '0')

        for j := n - 1; j >= 0; j-- {
            digit2 := int(num2[j] - '0')

            product := digit1 * digit2

            positionLow := i + j + 1
            positionHigh := i + j

            total := product + result[positionLow]

            result[positionLow] = total % 10
            result[positionHigh] += total / 10
        }
    }

    start := 0

    for start < len(result) && result[start] == 0 {
        start++
    }

    answer := make([]byte, 0)

    for i := start; i < len(result); i++ {
        answer = append(answer, byte(result[i]+'0'))
    }

    return string(answer)
}
```

The Go solution follows exactly the same logic as the Python implementation.

One difference is digit extraction. In Go, characters are bytes, so digit conversion uses:

```
int(num1[i] - '0')
```

instead of `ord()` or `int()`.

Another difference is string construction. Since strings are immutable in Go, we build a byte slice and convert it into a string at the end for efficiency.

Go integers are large enough for intermediate digit multiplication because the largest multiplication is:

```
9 × 9 + carry
```

which easily fits within standard integer types.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m × n) | Every digit in `num1` multiplies every digit in `num2` |
| Space | O(m + n) | Result array stores at most `m + n` digits |

The algorithm uses two nested loops, one for each digit of the input strings. Since every pair of digits is processed exactly once, the runtime is proportional to the product of the lengths of the strings.

The extra space comes from the result array, which stores the digits of the final multiplication result. No additional large intermediate structures are created.

## Edge Cases

### One Input Is Zero

If either input is `"0"`, the result must be `"0"`.

This is important because the result array would otherwise contain only zeros, and removing leading zeros could accidentally produce an empty string.

The implementation handles this immediately at the beginning:

```
if num1 == "0" or num2 == "0":
    return "0"
```

### Carry Propagation Across Multiple Digits

Cases like:

```
99 × 99
```

produce many carries.

A buggy implementation may overwrite carries incorrectly or fail to accumulate previous values already stored in the result array.

The algorithm avoids this by always adding:

```
product + existing_value
```

before computing carry and remainder.

### Different Length Inputs

Inputs may have very different lengths:

```
num1 = "1"
num2 = "999999999"
```

The indexing logic must still place digits correctly.

Using:

```
position_low = i + j + 1
position_high = i + j
```

ensures proper positional alignment regardless of input length.

### Leading Zero Removal

The preallocated result array may begin with zero even for valid products.

For example:

```
123 × 456 = 056088
```

inside the array representation.

The implementation carefully skips only leading zeros while preserving meaningful zeros inside the number.
