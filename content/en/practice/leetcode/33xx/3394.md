## Solution

## Problem Understanding

We are given an `n x n` grid and a list of non-overlapping rectangles placed inside that grid. Each rectangle is represented by four integers:

- `startx`, `starty`, the bottom-left corner
- `endx`, `endy`, the top-right corner

The rectangles never overlap, although they may touch at edges.

We need to determine whether it is possible to make exactly two cuts of the same orientation:

- either two horizontal cuts,
- or two vertical cuts,

such that the grid becomes divided into three sections, and all of the following conditions hold:

1. Every section contains at least one rectangle.
2. Every rectangle belongs completely to exactly one section.
3. No rectangle is split by a cut.

The cuts themselves lie along grid lines. For horizontal cuts, we cut at specific `y` coordinates. For vertical cuts, we cut at specific `x` coordinates.

The important detail is that a rectangle cannot cross a cut line. That means if we make a vertical cut at `x = c`, then every rectangle must either satisfy:

- `endx <= c`, entirely on the left side,
- or `startx >= c`, entirely on the right side.

The same logic applies for horizontal cuts using `y` coordinates.

The constraints are large:

- Up to `10^5` rectangles
- Coordinates up to `10^9`

This immediately tells us that any grid-based simulation is impossible. We cannot build the actual grid. Instead, the solution must operate only on rectangle coordinates.

Another key observation is that cuts only matter relative to rectangle intervals. For vertical cuts, only the `x` intervals matter. For horizontal cuts, only the `y` intervals matter.

Important edge cases include:

- Rectangles that touch exactly at boundaries
- Multiple rectangles sharing the same start or end coordinate
- Cases where only one orientation works
- Cases where gaps do not actually create three non-empty groups
- Rectangles arranged so that any cut would split at least one rectangle

The problem guarantees that rectangles do not overlap, which greatly simplifies interval reasoning.

## Approaches

### Brute Force Approach

A brute force solution would try all possible pairs of vertical cuts and all possible pairs of horizontal cuts.

For vertical cuts:

- Try every pair `(x1, x2)` where `0 < x1 < x2 < n`
- For every rectangle, determine which section it belongs to
- Verify:

- no rectangle crosses a cut,
- each of the three sections contains at least one rectangle

The same process would then be repeated for horizontal cuts.

This approach is correct because it exhaustively checks every possible valid cut configuration.

However, it is far too slow. There are up to `10^9` possible cut positions, so even iterating through all cut pairs is impossible. Even if we compressed coordinates, checking every pair would still be too expensive.

We need a smarter observation.

### Key Insight

For a cut to be valid, it must lie in a gap between rectangle intervals.

Consider vertical cuts. Every rectangle projects onto the x-axis as an interval:

$$[startx, endx]$$

If we sort rectangles by their starting coordinate and merge overlapping intervals, then every gap between merged intervals represents a legal place to cut.

If there are at least three separated groups of intervals, then we can place two cuts between them.

For example:

- Group 1 occupies x-range `[0,2]`
- Group 2 occupies x-range `[4,5]`
- Group 3 occupies x-range `[7,9]`

Then we can cut between:

- `2` and `4`
- `5` and `7`

Every rectangle stays entirely inside one section.

So the problem reduces to:

- Project rectangles onto one axis
- Merge overlapping intervals
- Count how many disconnected interval groups exist

If there are at least three groups horizontally or vertically, the answer is `true`.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n³) or worse | O(1) | Tries all possible cut pairs |
| Optimal | O(n log n) | O(n) | Merge intervals and count separated groups |

## Algorithm Walkthrough

### Step 1: Build intervals for one axis

For vertical cuts, extract:

```
[startx, endx]
```

from every rectangle.

For horizontal cuts, extract:

```
[starty, endy]
```

These intervals represent the occupied ranges along that axis.

### Step 2: Sort intervals

Sort intervals by:

1. starting coordinate
2. ending coordinate

Sorting allows us to merge overlapping intervals efficiently from left to right.

### Step 3: Merge overlapping intervals

Traverse the sorted intervals while maintaining the current merged interval.

If the next interval starts before the current interval ends, they overlap and should be merged.

Otherwise, a new disconnected group begins.

For example:

```
[0,2], [1,4], [6,8]
```

becomes:

```
[0,4], [6,8]
```

This indicates two separated groups.

### Step 4: Count disconnected groups

Each merged interval corresponds to one continuous occupied region.

If we obtain at least three merged groups, then two cuts can be placed between them.

That creates three sections, each containing at least one rectangle.

### Step 5: Check both orientations

Run the same process:

- once for x-intervals,
- once for y-intervals.

If either orientation produces at least three groups, return `true`.

Otherwise return `false`.

### Why it works

A cut is valid only if it lies entirely between occupied regions. Merging intervals identifies all continuous occupied regions on an axis.

If there are at least three disconnected groups, then there exist at least two gaps between them where cuts can be placed safely. Because rectangles never overlap and cuts lie in empty gaps, every rectangle remains entirely inside exactly one section.

Conversely, if fewer than three merged groups exist, then no two cuts can divide the rectangles into three non-empty sections without cutting through some rectangle.

## Python Solution

```python
from typing import List

class Solution:
    def checkValidCuts(self, n: int, rectangles: List[List[int]]) -> bool:
        
        def count_groups(intervals: List[List[int]]) -> int:
            intervals.sort()
            
            groups = 0
            current_end = -1
            
            for start, end in intervals:
                if start >= current_end:
                    groups += 1
                    current_end = end
                else:
                    current_end = max(current_end, end)
            
            return groups
        
        x_intervals = []
        y_intervals = []
        
        for startx, starty, endx, endy in rectangles:
            x_intervals.append([startx, endx])
            y_intervals.append([starty, endy])
        
        return (
            count_groups(x_intervals) >= 3 or
            count_groups(y_intervals) >= 3
        )
```

The solution defines a helper function called `count_groups` that computes how many disconnected interval groups exist along one axis.

The intervals are first sorted. We then scan them from left to right while tracking the end of the current merged interval.

If the next interval starts before the current merged interval ends, the intervals overlap and must belong to the same group. We merge them by extending the current end.

Otherwise, we found a new disconnected group, so we increment the group counter.

The main function constructs two interval lists:

- x-axis intervals for vertical cuts
- y-axis intervals for horizontal cuts

Finally, we check whether either axis contains at least three disconnected groups.

## Go Solution

```go
package main

import "sort"

func checkValidCuts(n int, rectangles [][]int) bool {

	countGroups := func(intervals [][]int) int {
		sort.Slice(intervals, func(i, j int) bool {
			if intervals[i][0] == intervals[j][0] {
				return intervals[i][1] < intervals[j][1]
			}
			return intervals[i][0] < intervals[j][0]
		})

		groups := 0
		currentEnd := -1

		for _, interval := range intervals {
			start := interval[0]
			end := interval[1]

			if start >= currentEnd {
				groups++
				currentEnd = end
			} else if end > currentEnd {
				currentEnd = end
			}
		}

		return groups
	}

	xIntervals := make([][]int, 0, len(rectangles))
	yIntervals := make([][]int, 0, len(rectangles))

	for _, rect := range rectangles {
		xIntervals = append(xIntervals, []int{rect[0], rect[2]})
		yIntervals = append(yIntervals, []int{rect[1], rect[3]})
	}

	return countGroups(xIntervals) >= 3 ||
		countGroups(yIntervals) >= 3
}
```

The Go implementation follows the same logic as the Python version.

The main difference is that Go requires explicit sorting using `sort.Slice`. We also use slices of integer pairs to represent intervals.

There are no integer overflow concerns because coordinates are at most `10^9`, well within Go's `int` range on LeetCode platforms.

The helper function performs a sort followed by a linear scan. Sorting takes `O(n log n)` time, while merging intervals takes `O(n)` time.

## Edge Cases

### Rectangles Touching at Boundaries

One subtle case occurs when one interval ends exactly where another begins.

For example:

```
[0,2], [2,4]
```

A cut at `x = 2` is still valid because rectangles are not split. One rectangle ends exactly at the cut, and the next begins there.

That is why the implementation starts a new group when:

```
start >= current_end
```

rather than:

```
start > current_end
```

This distinction is essential for correctness.

### All Rectangles Form One Connected Region

Another important case is when intervals chain together through overlaps.

For example:

```
[0,3], [2,5], [4,7]
```

Even though the first and third intervals do not directly overlap, the middle interval connects them into one continuous region.

A naive solution might mistakenly count multiple groups. The merge process correctly collapses them into one merged interval.

### Exactly Two Groups

Having two disconnected groups is not enough.

We need two cuts, which create three sections. Therefore, we must have at least three disconnected interval groups.

For example:

```
[0,1], [5,6]
```

Only one cut can separate them. No second cut can create a third non-empty section.

The implementation explicitly checks:

```
groups >= 3
```

which guarantees all three sections contain at least one rectangle.
