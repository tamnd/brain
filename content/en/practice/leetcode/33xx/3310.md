**Difficulty:** 🟡 Medium  
**Topics:** Depth-First Search, Breadth-First Search, Graph Theory

## Solution

## Problem Understanding

The problem models a project as a directed graph of method calls. Each method is represented as a node, and each invocation `[a, b]` represents a directed edge from method `a` to method `b`, meaning method `a` calls method `b`.

We are told that method `k` contains a bug. Because bugs can propagate through method calls, every method reachable from `k` is considered suspicious. This includes both direct calls and indirect chains of calls.

The first task is therefore to identify the full set of suspicious methods. In graph terms, this means finding every node reachable from node `k`.

However, we are not always allowed to remove them.

The important restriction is:

> A group of methods can only be removed if no method outside the group invokes any method inside the group.

This means that after identifying the suspicious set, we must verify whether there exists any incoming edge from a non suspicious node into a suspicious node.

If such an edge exists, then the suspicious methods are still being used by safe methods, so removing them would break the project. In that case, we are required to return all methods unchanged.

If no outside method invokes any suspicious method, then the suspicious component is completely isolated from the rest of the project and can safely be removed.

The output should therefore be:

- All non suspicious methods, if removal is valid
- All methods, if removal is invalid

The constraints are large:

- `n` can be up to `100000`
- `invocations.length` can be up to `200000`

These constraints immediately rule out expensive graph traversals repeated multiple times. Any acceptable solution should run in approximately linear time relative to the number of nodes and edges.

Several edge cases are important:

- The suspicious set may contain every node in the graph
- The suspicious set may contain only `k`
- The graph may contain cycles
- The graph may contain disconnected components
- Multiple methods outside the suspicious set may invoke suspicious methods
- There may be no invocations at all

A naive implementation can easily fail on cycles if it does not track visited nodes properly. Another common mistake is checking outgoing edges instead of incoming edges when validating whether removal is allowed.

## Approaches

### Brute Force Approach

A straightforward approach would be:

1. Start from method `k`
2. Find all reachable suspicious methods using DFS or BFS
3. For every suspicious method:

- Scan every invocation edge
- Check whether some non suspicious method invokes it

This works correctly because:

- Reachability correctly identifies suspicious methods
- Exhaustively checking every edge guarantees we detect illegal incoming references

However, this becomes inefficient because for every suspicious node we may scan the entire edge list again.

In the worst case:

- There are `O(n)` suspicious nodes
- There are `O(m)` edges

This leads to `O(n * m)` complexity, which is far too slow when `n = 100000` and `m = 200000`.

### Optimal Approach

The key insight is that we do not need to repeatedly scan the graph.

We can solve the problem in two linear passes:

1. Perform DFS or BFS from `k` to mark all suspicious methods
2. Iterate through every invocation edge exactly once:

- If an edge goes from a non suspicious node to a suspicious node, removal is impossible

This works because the removal condition depends only on whether such an incoming boundary edge exists.

Instead of repeatedly checking each suspicious node individually, we process every edge once and verify whether it crosses from outside the suspicious set into it.

This reduces the complexity to linear time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * m) | O(n + m) | Repeatedly scans all edges for each suspicious node |
| Optimal | O(n + m) | O(n + m) | Uses one graph traversal and one edge validation pass |

Here:

- `n` is the number of methods
- `m` is the number of invocation edges

## Algorithm Walkthrough

1. Build an adjacency list for the graph.

We need efficient access to all methods invoked by a given method. An adjacency list allows us to traverse outgoing edges in linear time overall.
2. Perform DFS or BFS starting from method `k`.

Every method reachable from `k` becomes suspicious.

We maintain a boolean array `suspicious` where:

- `suspicious[i] = True` means method `i` belongs to the suspicious set
- `False` means it does not

A stack or queue prevents repeated visits and handles cycles safely.
3. Traverse every invocation edge once.

For each edge `[a, b]`:

- If `a` is not suspicious
- But `b` is suspicious

then a safe method depends on a suspicious method.

This violates the removal rule.
4. If such an invalid edge exists, return all methods.

Since removal is impossible, the project must remain unchanged.
5. Otherwise, return only the non suspicious methods.

These are the methods that remain after successfully removing the suspicious component.

### Why it works

The algorithm works because the suspicious set is exactly the set of nodes reachable from `k`.

The only condition preventing removal is the existence of an incoming edge from outside the set into the set. By checking every edge once, we guarantee that we detect any such dependency.

If no such edge exists, then removing all suspicious methods cannot break any remaining method, because no remaining method references them.

Therefore the algorithm always returns the correct result.

## Python Solution

```python
from typing import List

class Solution:
    def remainingMethods(self, n: int, k: int, invocations: List[List[int]]) -> List[int]:
        graph = [[] for _ in range(n)]

        for source, target in invocations:
            graph[source].append(target)

        suspicious = [False] * n

        stack = [k]
        suspicious[k] = True

        while stack:
            current = stack.pop()

            for neighbor in graph[current]:
                if not suspicious[neighbor]:
                    suspicious[neighbor] = True
                    stack.append(neighbor)

        for source, target in invocations:
            if not suspicious[source] and suspicious[target]:
                return list(range(n))

        remaining = []

        for method in range(n):
            if not suspicious[method]:
                remaining.append(method)

        return remaining
```

The implementation begins by constructing the adjacency list. Each method stores the list of methods it invokes.

Next, we perform iterative DFS using a stack. Iterative traversal is preferred here because recursive DFS may exceed Python's recursion depth limit when the graph is large.

The `suspicious` array tracks which methods are reachable from `k`. Whenever we discover a new reachable node, we mark it immediately before pushing it onto the stack. This prevents duplicate processing and safely handles cycles.

After the traversal completes, the suspicious set is fully known.

We then validate whether removal is allowed. Every edge is checked once. If an edge crosses from a non suspicious node into a suspicious node, we immediately return all methods because removal would violate the dependency rule.

If no invalid edge exists, we collect all non suspicious methods into the result list and return them.

## Go Solution

```go
func remainingMethods(n int, k int, invocations [][]int) []int {
	graph := make([][]int, n)

	for _, edge := range invocations {
		source := edge[0]
		target := edge[1]
		graph[source] = append(graph[source], target)
	}

	suspicious := make([]bool, n)

	stack := []int{k}
	suspicious[k] = true

	for len(stack) > 0 {
		current := stack[len(stack)-1]
		stack = stack[:len(stack)-1]

		for _, neighbor := range graph[current] {
			if !suspicious[neighbor] {
				suspicious[neighbor] = true
				stack = append(stack, neighbor)
			}
		}
	}

	for _, edge := range invocations {
		source := edge[0]
		target := edge[1]

		if !suspicious[source] && suspicious[target] {
			result := make([]int, n)

			for i := 0; i < n; i++ {
				result[i] = i
			}

			return result
		}
	}

	result := []int{}

	for method := 0; method < n; method++ {
		if !suspicious[method] {
			result = append(result, method)
		}
	}

	return result
}
```

The Go implementation follows the same logic as the Python version.

A slice of slices is used for the adjacency list. DFS is implemented iteratively using a slice as a stack.

Unlike Python, Go does not provide a built in `range(n)` equivalent, so when removal is impossible we explicitly construct the result slice containing all methods.

Go slices naturally handle dynamic resizing when appending remaining methods.

Since all values fit comfortably within standard integer ranges, integer overflow is not a concern.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n + m) | Each node and edge is processed at most once |
| Space | O(n + m) | Adjacency list and visitation tracking require linear space |

The DFS traversal visits every suspicious node once and examines each outgoing edge once. The validation pass scans all invocation edges once more.

Because both operations are linear, the total runtime remains `O(n + m)`.

The adjacency list stores every edge exactly once, and the `suspicious` array stores one boolean per node, giving linear auxiliary space usage.

## Edge Cases

### Cycles in the Invocation Graph

The graph may contain cycles, such as:

```
0 → 1 → 2 → 0
```

A naive DFS without a visited array would loop forever. The implementation avoids this by marking methods as suspicious immediately when discovered. This guarantees each node is processed at most once.

### Incoming Edge From a Safe Method

This is the most important correctness condition.

For example:

```
0 → 1
```

If `1` is suspicious but `0` is not, then removing `1` would break method `0`.

A common bug is checking outgoing edges from suspicious methods instead of incoming edges into suspicious methods. The implementation correctly checks whether:

```
source is safe AND target is suspicious
```

### Entire Graph Becomes Suspicious

If every method is reachable from `k`, then all methods are suspicious.

In this case, removal is valid because there are no outside methods remaining that could invoke suspicious ones.

The implementation handles this naturally. The validation pass never finds an edge from a non suspicious node because no such nodes exist. The returned result is therefore an empty list.
