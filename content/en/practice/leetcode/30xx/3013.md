**Difficulty:** 🔴 Hard  
**Topics:** Array, Hash Table, Sliding Window, Heap (Priority Queue)

## Solution

## Problem Understanding

We are given an array `nums`, along with two integers `k` and `dist`.

We must divide the array into exactly `k` contiguous, non-overlapping subarrays that together cover the entire array. The cost of each subarray is simply its first element. The total cost is the sum of those starting elements.

If the subarrays begin at indices:

$$0,\ i_1,\ i_2,\ \dots,\ i_{k-1}$$

then the total cost becomes:

$$nums[0] + nums[i_1] + nums[i_2] + \dots + nums[i_{k-1}]$$

The important restriction is:

$$i_{k-1} - i_1 \le dist$$

This means all starting positions except the very first one at index `0` must lie inside a window of width at most `dist`.

The task is to minimize the sum of the starting values of the `k` subarrays.

A very important observation is that choosing the subarray boundaries is equivalent to choosing the starting indices of the last `k - 1` subarrays. Since the first subarray always starts at index `0`, we only need to choose `k - 1` additional starting positions.

Suppose we choose indices:

$$i_1 < i_2 < \dots < i_{k-1}$$

Then the answer becomes:

$$nums[0] + nums[i_1] + nums[i_2] + \dots + nums[i_{k-1}]$$

subject to:

$$i_{k-1} - i_1 \le dist$$

So the problem transforms into:

> Find `k - 1` elements inside a sliding window of length at most `dist + 1` whose sum is minimum.

The constraints are large:

- `n` can be up to `10^5`
- `nums[i]` can be up to `10^9`

This immediately rules out exponential or quadratic enumeration approaches. We need something around `O(n log n)`.

There are several important edge cases:

- `dist = k - 2`, which forces the chosen indices to occupy a very tight range.
- `dist = n - 2`, which effectively removes most restrictions.
- Duplicate values, which require careful handling in ordered data structures.
- Large numbers, which require 64-bit integer arithmetic in Go.
- Windows where elements continuously enter and leave, requiring efficient updates.

## Approaches

### Brute Force

The brute-force idea is to try every possible choice of `k - 1` starting indices.

For every increasing sequence:

$$i_1 < i_2 < \dots < i_{k-1}$$

we check whether:

$$i_{k-1} - i_1 \le dist$$

If valid, we compute:

$$nums[0] + \sum nums[i_j]$$

and keep the minimum.

This is correct because it exhaustively checks all valid divisions.

However, the number of ways to choose `k - 1` indices from `n - 1` positions is:

$$\binom{n-1}{k-1}$$

which is astronomically large for `n = 10^5`.

Even dynamic programming approaches with naive transitions become too slow because they often require `O(n^2)` state transitions.

### Key Insight

The restriction:

$$i_{k-1} - i_1 \le dist$$

means all chosen indices must lie inside some window of size `dist + 1`.

Suppose we fix the smallest chosen index `left = i_1`.

Then every other chosen index must lie in:

$$[left,\ left + dist]$$

Inside that window, we simply need to choose the smallest `k - 1` values.

Therefore, for every sliding window of size `dist + 1`, we need to efficiently maintain:

- the sum of the smallest `k - 1` elements
- while elements enter and leave the window

This is a classic two-multiset problem.

We maintain:

- a set `small` containing the smallest `k - 1` values
- a set `large` containing the remaining values

We also maintain the running sum of elements in `small`.

As the window slides:

- insert one element
- remove one element
- rebalance the two sets

This gives an `O(n log n)` solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | O(k) | Enumerates all possible partitions |
| Optimal | O(n log n) | O(n) | Sliding window with balanced ordered sets |

## Algorithm Walkthrough

### Step 1: Reformulate the problem

The first subarray always starts at index `0`, so `nums[0]` is always included in the answer.

We only need to choose the starting indices of the remaining `k - 1` subarrays.

If those indices are:

$$i_1 < i_2 < \dots < i_{k-1}$$

then:

$$i_{k-1} - i_1 \le dist$$

means all selected indices lie inside a contiguous window of size `dist + 1`.

### Step 2: Build sliding windows

We slide a window over indices:

$$[1,\ dist + 1]$$

then:

$$[2,\ dist + 2]$$

and so on.

Each window represents all possible valid positions for the chosen indices.

Inside each window we must choose exactly `k - 1` values with minimum sum.

### Step 3: Maintain the smallest `k - 1` elements

We split window elements into two groups:

1. `small`

- contains the smallest `k - 1` elements
- contributes to the answer
2. `large`

- contains all remaining elements

We maintain:

- `small_sum`, the sum of values in `small`

The answer for the current window is:

$$nums[0] + small\_sum$$

### Step 4: Handle duplicates correctly

Values may repeat, so we cannot store only the number.

Instead, we store pairs:

$$(value,\ index)$$

This guarantees uniqueness.

### Step 5: Insert elements

When a new element enters the window:

1. Insert it into `small`
2. Add its value to `small_sum`
3. If `small` becomes too large:

- move the largest element from `small` into `large`

This preserves the invariant that `small` contains exactly the smallest `k - 1` elements.

### Step 6: Remove elements

When an element leaves the window:

- If it belongs to `small`

- remove it from `small`
- subtract from `small_sum`
- Otherwise remove it from `large`

After removal, if `small` has fewer than `k - 1` elements:

- move the smallest element from `large` into `small`

### Step 7: Track the minimum answer

For every valid window:

$$answer = \min(answer,\ nums[0] + small\_sum)$$

### Why it works

At every step, `small` contains exactly the `k - 1` smallest elements inside the current window.

Since every valid solution must choose all indices from some window of length `dist + 1`, considering every such window guarantees we examine all valid possibilities.

For each window, selecting the smallest `k - 1` values is obviously optimal. Therefore, the global minimum over all windows is the correct answer.

## Python Solution

```python
from typing import List
from sortedcontainers import SortedList

class Solution:
    def minimumCost(self, nums: List[int], k: int, dist: int) -> int:
        need = k - 1
        
        small = SortedList()
        large = SortedList()
        
        small_sum = 0
        
        def add(item):
            nonlocal small_sum
            
            small.add(item)
            small_sum += item[0]
            
            if len(small) > need:
                moved = small.pop(-1)
                small_sum -= moved[0]
                large.add(moved)
        
        def remove(item):
            nonlocal small_sum
            
            if item in small:
                small.remove(item)
                small_sum -= item[0]
            else:
                large.remove(item)
            
            if len(small) < need and large:
                moved = large.pop(0)
                small.add(moved)
                small_sum += moved[0]
        
        for i in range(1, dist + 2):
            add((nums[i], i))
        
        answer = nums[0] + small_sum
        
        left = 1
        
        for right in range(dist + 2, len(nums)):
            add((nums[right], right))
            remove((nums[left], left))
            left += 1
            
            answer = min(answer, nums[0] + small_sum)
        
        return answer
```

The implementation closely follows the sliding window strategy described earlier.

The variable `need` stores how many additional subarray starts we must choose, which is `k - 1`.

We maintain two `SortedList` structures:

- `small` contains the smallest required elements
- `large` contains the remaining window elements

Each element is stored as `(value, index)` so duplicates are handled correctly.

The `add` function first inserts into `small`. If `small` grows too large, we move its largest element into `large`.

The `remove` function deletes the outgoing element from whichever structure contains it. After removal, if `small` has too few elements, we refill it using the smallest element from `large`.

The initial window is:

$$[1,\ dist + 1]$$

because index `0` is fixed as the start of the first subarray and is not part of the selectable indices.

As the window slides, we continuously maintain the sum of the smallest `k - 1` values.

## Go Solution

```go
package main

import (
	"container/heap"
)

type MinHeap []Pair
type MaxHeap []Pair

type Pair struct {
	val int
	idx int
}

func (h MinHeap) Len() int {
	return len(h)
}

func (h MinHeap) Less(i, j int) bool {
	if h[i].val == h[j].val {
		return h[i].idx < h[j].idx
	}
	return h[i].val < h[j].val
}

func (h MinHeap) Swap(i, j int) {
	h[i], h[j] = h[j], h[i]
}

func (h *MinHeap) Push(x interface{}) {
	*h = append(*h, x.(Pair))
}

func (h *MinHeap) Pop() interface{} {
	old := *h
	n := len(old)
	item := old[n-1]
	*h = old[:n-1]
	return item
}

func (h MaxHeap) Len() int {
	return len(h)
}

func (h MaxHeap) Less(i, j int) bool {
	if h[i].val == h[j].val {
		return h[i].idx > h[j].idx
	}
	return h[i].val > h[j].val
}

func (h MaxHeap) Swap(i, j int) {
	h[i], h[j] = h[j], h[i]
}

func (h *MaxHeap) Push(x interface{}) {
	*h = append(*h, x.(Pair))
}

func (h *MaxHeap) Pop() interface{} {
	old := *h
	n := len(old)
	item := old[n-1]
	*h = old[:n-1]
	return item
}

func minimumCost(nums []int, k int, dist int) int64 {
	need := k - 1

	small := &MaxHeap{}
	large := &MinHeap{}

	heap.Init(small)
	heap.Init(large)

	inSmall := map[int]bool{}
	removed := map[int]bool{}

	var smallSum int64 = 0

	pruneSmall := func() {
		for small.Len() > 0 {
			top := (*small)[0]
			if !removed[top.idx] {
				break
			}
			heap.Pop(small)
		}
	}

	pruneLarge := func() {
		for large.Len() > 0 {
			top := (*large)[0]
			if !removed[top.idx] {
				break
			}
			heap.Pop(large)
		}
	}

	balance := func() {
		pruneSmall()
		pruneLarge()

		for len(inSmall) > need {
			pruneSmall()
			top := heap.Pop(small).(Pair)

			if removed[top.idx] {
				continue
			}

			delete(inSmall, top.idx)
			smallSum -= int64(top.val)
			heap.Push(large, top)
		}

		for len(inSmall) < need {
			pruneLarge()
			top := heap.Pop(large).(Pair)

			if removed[top.idx] {
				continue
			}

			inSmall[top.idx] = true
			smallSum += int64(top.val)
			heap.Push(small, top)
		}
	}

	add := func(p Pair) {
		if len(inSmall) < need {
			inSmall[p.idx] = true
			smallSum += int64(p.val)
			heap.Push(small, p)
		} else {
			pruneSmall()

			top := (*small)[0]

			if p.val < top.val || (p.val == top.val && p.idx < top.idx) {
				moved := heap.Pop(small).(Pair)

				delete(inSmall, moved.idx)
				smallSum -= int64(moved.val)

				heap.Push(large, moved)

				inSmall[p.idx] = true
				smallSum += int64(p.val)
				heap.Push(small, p)
			} else {
				heap.Push(large, p)
			}
		}

		balance()
	}

	remove := func(p Pair) {
		removed[p.idx] = true

		if inSmall[p.idx] {
			delete(inSmall, p.idx)
			smallSum -= int64(p.val)
		}

		balance()
	}

	for i := 1; i <= dist+1; i++ {
		add(Pair{nums[i], i})
	}

	answer := int64(nums[0]) + smallSum

	left := 1

	for right := dist + 2; right < len(nums); right++ {
		add(Pair{nums[right], right})
		remove(Pair{nums[left], left})
		left++

		current := int64(nums[0]) + smallSum

		if current < answer {
			answer = current
		}
	}

	return answer
}
```

The Go implementation uses heaps because Go does not provide a built-in balanced ordered set like Python's `SortedList`.

A max-heap stores the selected smallest elements, while a min-heap stores the remaining elements.

Because heaps do not support arbitrary deletion efficiently, lazy deletion is used through the `removed` map.

All sums use `int64` to avoid overflow since values may reach `10^9`.

| Space | O(n) | The balanced structures store window elements |

Each element enters and leaves the sliding window exactly once. Every operation on the ordered structures costs `O(log n)`, leading to an overall complexity of `O(n log n)`.

## Edge Cases

### Minimum Possible Distance

When:

```
dist = k - 2
```

the chosen indices are forced into a very narrow range.

This is easy to mishandle because there may be only one valid window size capable of holding all required indices. The implementation handles this naturally because every window is evaluated independently and must always contain exactly `k - 1` selected elements.

### Duplicate Values

If multiple indices contain the same value, storing only numbers can lead to ambiguous deletions.

For example:

```
[5,5,5,5]
```

When the window slides, we must remove the correct occurrence.

The solution stores `(value, index)` pairs, ensuring every element is unique even when values repeat.

### Very Large Numbers

Since:

```
nums[i] <= 10^9
```

and up to `10^5` elements may contribute to sums, 32-bit integers can overflow.

The Go implementation uses `int64` for all accumulated sums to guarantee correctness.
