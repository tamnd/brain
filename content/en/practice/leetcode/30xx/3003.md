# LeetCode 3003 - Maximize the Number of Partitions After Operations

## Problem Understanding

The problem gives us a string `s` and an integer `k`. We repeatedly partition the string using a very specific greedy rule:

- At every step, we take the longest possible prefix containing at most `k` distinct characters.
- We remove that prefix and count one partition.
- We continue until the string becomes empty.

Before this partitioning process begins, we are allowed to modify at most one character in the string into any lowercase English letter.

Our goal is to maximize the number of partitions produced by the greedy process.

The important detail is that the partitioning rule is fixed. We are not free to split the string wherever we want. At each step, the prefix must be the longest valid prefix with at most `k` distinct characters. This means the partition boundaries are completely determined once the final string is chosen.

The constraints are significant:

- `s.length` can be as large as `10^4`
- There are 26 lowercase letters
- We may change one character into any other lowercase letter

A brute force approach that tries every modification and recomputes partitions from scratch would become too slow because there are up to:

- `10^4` positions
- `26` replacement characters
- `O(n)` work to recompute partitions

That leads to roughly `O(26 * n^2)` complexity, which is too expensive for the largest inputs.

Several edge cases are important:

- When `k = 26`, the entire string always forms one partition regardless of changes.
- When `k = 1`, partitions become extremely sensitive to character changes.
- Changing a character may affect not only one partition but also all later partitions because greedy boundaries shift.
- Sometimes the best choice is not changing anything.
- Very short strings require careful handling because one modification can completely alter the answer.

## Approaches

### Brute Force

The brute force strategy is straightforward:

1. Try every possible index to modify.
2. Try every replacement character from `'a'` to `'z'`.
3. Build the modified string.
4. Simulate the greedy partitioning process.
5. Track the maximum number of partitions.

To simulate partitioning, we scan left to right while maintaining the set of distinct characters in the current partition. Whenever adding another character would exceed `k` distinct characters, we start a new partition.

This method is correct because it explicitly evaluates every possible valid modification and computes the exact partition count for each resulting string.

However, it is too slow for `n = 10^4`.

If each simulation takes `O(n)`, then:

- `n` positions
- `26` replacement choices
- `O(n)` simulation

gives `O(26 * n^2)` time complexity.

### Optimal Approach

The key observation is that the partitioning process depends only on:

- The current index
- The set of distinct characters in the current partition
- Whether we have already used our one modification

This naturally leads to dynamic programming with memoization.

We process the string from left to right. At every position, we maintain:

- Current index `i`
- Bitmask representing characters inside the current partition
- Count of distinct characters
- Whether we already used the modification

At each position we have two possibilities:

1. Keep the current character.
2. Replace it with another character, if modification has not been used yet.

Whenever adding a character would exceed `k` distinct characters, we must start a new partition.

Since there are only 26 lowercase letters, we can represent the character set using a bitmask. This allows efficient state transitions.

Memoization ensures each state is computed only once.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(26 * n²) | O(1) | Tries every modification and rescans string |
| Optimal | O(n * 26 * states) | O(states) | DP with memoization and bitmask compression |

The actual number of DP states remains manageable because each partition can contain at most `k` distinct characters.

## Algorithm Walkthrough

1. Define a recursive DP function `dfs(index, mask, distinctCount, usedChange)`.

This function returns the maximum additional partitions obtainable starting from `index`.
2. The `mask` stores which characters currently belong to the active partition.

Since there are only 26 lowercase letters, we use a 26-bit integer.
3. The `distinctCount` stores how many distinct characters currently exist in the active partition.

We store this separately because recomputing bit counts repeatedly is expensive.
4. The `usedChange` boolean indicates whether the one allowed modification has already been consumed.
5. At each index, first consider keeping the current character unchanged.

We compute:

- The bit corresponding to the character
- Whether adding it increases the distinct count
6. If adding the character keeps the partition within `k` distinct characters, we continue extending the current partition.
7. Otherwise, we must start a new partition.

The current character becomes the first character of the next partition, and the partition count increases by one.
8. If we have not used our modification yet, we also try replacing the current character with every possible letter from `'a'` to `'z'`.
9. For every replacement character:

- Compute the updated mask
- Determine whether the partition overflows
- Either continue the current partition or start a new one
10. Memoize the result for the current DP state.
11. The base case occurs when `index == len(s)`.

At that point, the current partition contributes one final partition.

### Why it works

The algorithm works because the greedy partitioning process is deterministic once the final string is fixed. The DP explores every possible valid modification while preserving exactly the information needed to determine future partition boundaries:

- Current position
- Current partition character set
- Whether modification remains available

Since every future decision depends only on these values, the DP state is complete. Memoization guarantees we avoid recomputing identical subproblems.

## Python Solution

```python
from functools import lru_cache

class Solution:
    def maxPartitionsAfterOperations(self, s: str, k: int) -> int:
        n = len(s)
        nums = [ord(c) - ord('a') for c in s]

        @lru_cache(None)
        def dfs(index: int, mask: int, distinct: int, used_change: bool) -> int:
            if index == n:
                return 1

            best = 0

            # Keep original character
            char = nums[index]
            bit = 1 << char

            new_mask = mask
            new_distinct = distinct

            if (mask & bit) == 0:
                new_mask |= bit
                new_distinct += 1

            if new_distinct > k:
                best = max(
                    best,
                    1 + dfs(index + 1, bit, 1, used_change)
                )
            else:
                best = max(
                    best,
                    dfs(index + 1, new_mask, new_distinct, used_change)
                )

            # Try changing character
            if not used_change:
                for replacement in range(26):
                    if replacement == char:
                        continue

                    rep_bit = 1 << replacement

                    temp_mask = mask
                    temp_distinct = distinct

                    if (temp_mask & rep_bit) == 0:
                        temp_mask |= rep_bit
                        temp_distinct += 1

                    if temp_distinct > k:
                        best = max(
                            best,
                            1 + dfs(index + 1, rep_bit, 1, True)
                        )
                    else:
                        best = max(
                            best,
                            dfs(index + 1, temp_mask, temp_distinct, True)
                        )

            return best

        return dfs(0, 0, 0, False)
```

The implementation starts by converting the string into integer character indices from `0` to `25`. This simplifies bitmask operations.

The recursive `dfs` function is memoized using `lru_cache`. Each state is uniquely identified by:

- Current index
- Current bitmask
- Current distinct count
- Whether modification has already been used

For every character, we first evaluate the case where the character remains unchanged. We update the mask and distinct count accordingly.

If the distinct count exceeds `k`, then the current partition must end before this character. Therefore, we add one partition and start a new partition containing only this character.

If modification is still available, we additionally try replacing the current character with every other lowercase letter. Each replacement follows the same logic.

The recursion returns the maximum number of partitions obtainable from the current state onward.

The base case returns `1` because the active partition itself contributes one partition when the string ends.

## Go Solution

```go
package main

func maxPartitionsAfterOperations(s string, k int) int {
	n := len(s)

	type State struct {
		index      int
		mask       int
		distinct   int
		usedChange int
	}

	memo := make(map[State]int)

	var dfs func(int, int, int, int) int

	dfs = func(index, mask, distinct, usedChange int) int {
		if index == n {
			return 1
		}

		state := State{
			index:      index,
			mask:       mask,
			distinct:   distinct,
			usedChange: usedChange,
		}

		if val, exists := memo[state]; exists {
			return val
		}

		best := 0

		// Keep original character
		char := int(s[index] - 'a')
		bit := 1 << char

		newMask := mask
		newDistinct := distinct

		if (mask & bit) == 0 {
			newMask |= bit
			newDistinct++
		}

		if newDistinct > k {
			candidate := 1 + dfs(index+1, bit, 1, usedChange)
			if candidate > best {
				best = candidate
			}
		} else {
			candidate := dfs(index+1, newMask, newDistinct, usedChange)
			if candidate > best {
				best = candidate
			}
		}

		// Try changing character
		if usedChange == 0 {
			for replacement := 0; replacement < 26; replacement++ {
				if replacement == char {
					continue
				}

				repBit := 1 << replacement

				tempMask := mask
				tempDistinct := distinct

				if (tempMask & repBit) == 0 {
					tempMask |= repBit
					tempDistinct++
				}

				if tempDistinct > k {
					candidate := 1 + dfs(index+1, repBit, 1, 1)
					if candidate > best {
						best = candidate
					}
				} else {
					candidate := dfs(index+1, tempMask, tempDistinct, 1)
					if candidate > best {
						best = candidate
					}
				}
			}
		}

		memo[state] = best
		return best
	}

	return dfs(0, 0, 0, 0)
}
```

The Go solution follows the same DP structure as the Python implementation.

Because Go does not provide built in memoization decorators, we explicitly use a `map[State]int` where `State` is a struct containing all DP parameters.

Bitmask operations work efficiently using Go integers. Since there are only 26 letters, standard integer types are sufficient.

Unlike Python, Go does not support nested function memoization automatically, so recursion and caching are implemented manually.

Now perform greedy partitioning.

| Step | Remaining String | Longest Valid Prefix | Distinct Characters | Partitions |

| 3 | a | a | {a} | 3 |

Final answer is `3`.

No modification can force additional partitions.

| Step | Remaining String | Longest Valid Prefix | Partitions |
| --- | --- | --- | --- |
| 1 | aabaab | aabaab | 1 |

Final answer is `1`.

| Step | Remaining String | Longest Valid Prefix | Partitions |

| 2 | xyz | x | 2 |

| 4 | z | z | 4 |

Final answer is `4`.

## Edge Cases

One important edge case occurs when `k = 26`. Since there are only 26 lowercase letters, every possible substring automatically satisfies the distinct character limit. Therefore, the entire string always becomes a single partition regardless of modifications. The implementation handles this naturally because the distinct count can never exceed `k`.

Another important case occurs when `k = 1`. In this situation, every partition may contain only one distinct character. Small character changes can dramatically increase the number of partitions. The DP correctly explores all replacement possibilities and evaluates how they affect greedy partition boundaries.

A third subtle case involves strings where changing a character causes partition boundaries to shift globally. For example, modifying one early character can alter the longest valid prefix and therefore change all subsequent partitions. A greedy local strategy would fail here, but the recursive DP correctly models future consequences because every state fully captures the current partition configuration.

A final edge case involves very short strings such as length one or two. Recursive implementations sometimes mishandle base conditions and accidentally overcount partitions. Here, the base case explicitly returns `1` when reaching the end of the string, correctly counting the final active partition.
