## Solution

## Problem Understanding

The problem gives us a string `s` representing keys typed by a user. Every character is an English alphabet letter, either uppercase or lowercase. We must count how many times the user changes from one key to a different key while typing the string from left to right.

The important detail is that uppercase and lowercase versions of the same letter are considered the same key. For example, typing `'a'` followed by `'A'` does not count as a key change because both use the same alphabetic key. Similarly, `'B'` followed by `'b'` is also not a change.

What actually matters is whether the alphabetic letter changes after ignoring case. If two consecutive characters represent different letters after converting both to lowercase, then we increment the answer.

The input consists of:

- A single string `s`
- Length between `1` and `100`
- Only uppercase and lowercase English letters

The output is a single integer representing the number of times the user switched to a different key.

Because the string length is very small, performance constraints are not strict. However, the problem is fundamentally a linear scan problem, so an efficient solution is straightforward.

There are several edge cases worth identifying early:

- A string of length `1` should always return `0` because there are no adjacent characters to compare.
- A string containing only case variations of the same letter, such as `"AaAaA"`, should return `0`.
- Consecutive repeated letters in mixed casing, such as `"bBBb"`, should also produce `0`.
- Alternating different letters, such as `"aBcD"`, should count every actual alphabetic transition.

## Approaches

### Brute Force Approach

The brute force idea is to compare every adjacent pair of characters while handling case sensitivity manually. For each pair, we could check whether the characters are equal in lowercase form. If they differ, we increment the counter.

One inefficient variation of brute force would repeatedly perform expensive conversions or use additional temporary strings for every comparison. Another overly complicated version might use nested loops or auxiliary structures even though only adjacent comparisons matter.

This approach is correct because the definition of a key change depends only on consecutive characters. However, unnecessary repeated processing would make the implementation more complicated than needed.

### Optimal Approach

The key observation is that only adjacent characters matter, and uppercase/lowercase differences should be ignored. Therefore, we can normalize the string by converting characters to lowercase during comparison.

We scan the string once from left to right:

- Compare `s[i - 1].lower()` with `s[i].lower()`
- If they differ, increment the answer

This works because converting both characters to the same case removes all capitalization differences while preserving the actual letter identity.

Since every character is processed once, the algorithm runs in linear time with constant extra space.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | May create extra normalized strings or temporary structures |
| Optimal | O(n) | O(1) | Single pass with direct adjacent comparisons |

## Algorithm Walkthrough

1. Initialize a variable `changes` to `0`. This variable will store the total number of key changes found while scanning the string.
2. Traverse the string starting from index `1`. We begin at index `1` because every comparison involves the current character and the previous character.
3. For each position `i`, convert both `s[i]` and `s[i - 1]` to lowercase. This ensures uppercase and lowercase versions of the same letter are treated identically.
4. Compare the lowercase versions of the two characters.

- If they are different, increment `changes` because the user switched to a different key.
- If they are the same, do nothing because capitalization changes do not count.
5. Continue until the end of the string.
6. Return `changes` as the final answer.

### Why it works

The algorithm maintains the invariant that after processing index `i`, the variable `changes` equals the number of valid key transitions among all adjacent pairs from index `0` through `i`.

Every possible key change occurs between two consecutive characters, and the lowercase conversion guarantees that capitalization differences are ignored. Since each adjacent pair is examined exactly once, every valid transition is counted exactly once.

## Python Solution

```python
class Solution:
    def countKeyChanges(self, s: str) -> int:
        changes = 0

        for i in range(1, len(s)):
            if s[i].lower() != s[i - 1].lower():
                changes += 1

        return changes
```

The implementation begins by initializing a counter named `changes`. This variable tracks how many times the key changes throughout the string.

The loop starts at index `1` because each character must be compared with the previous one. Inside the loop, both characters are converted to lowercase using `.lower()`. This normalization step ensures that letters like `'A'` and `'a'` are treated as identical.

Whenever the lowercase versions differ, the algorithm increments the counter. After all adjacent pairs are processed, the final count is returned.

The implementation directly mirrors the algorithm described earlier and uses only constant extra memory.

## Go Solution

```go
func countKeyChanges(s string) int {
    changes := 0

    for i := 1; i < len(s); i++ {
        current := s[i]
        previous := s[i-1]

        if current >= 'A' && current <= 'Z' {
            current = current - 'A' + 'a'
        }

        if previous >= 'A' && previous <= 'Z' {
            previous = previous - 'A' + 'a'
        }

        if current != previous {
            changes++
        }
    }

    return changes
}
```

The Go solution follows the same logic as the Python version but handles lowercase conversion manually using ASCII arithmetic.

Since Go characters are bytes in this problem, uppercase letters can be converted to lowercase by subtracting `'A'` and adding `'a'`. This avoids importing additional packages like `strings` or `unicode`.

The algorithm still performs a single linear scan with constant extra space.

## Edge Cases

One important edge case is a string with only one character, such as `"a"`. Since there are no adjacent characters to compare, the correct answer must be `0`. A buggy implementation might accidentally access invalid indices or incorrectly initialize the counter. The provided solution avoids this because the loop starts at index `1`, so it never executes for a single-character string.

Another important case is repeated letters with different capitalization, such as `"bBBb"`. A naive implementation that directly compares characters without normalization would incorrectly count changes between `'b'` and `'B'`. The solution handles this correctly by converting both characters to lowercase before comparison.

A third important edge case is a string where every adjacent character represents a different letter, such as `"abcd"`. In this scenario, every comparison should count as a key change. The implementation correctly increments the counter for every differing lowercase comparison, resulting in `len(s) - 1` changes.
