# LeetCode 3050 - Pizza Toppings Cost Analysis

## Problem Understanding

This problem asks us to generate every valid pizza combination that contains exactly three different toppings from the `Toppings` table. For each valid combination, we must calculate the total cost by summing the costs of the three toppings.

The important detail is that toppings cannot repeat. A combination such as `"Pepperoni, Pepperoni, Onion"` is invalid because the same topping appears more than once.

Another important requirement is ordering. Inside each pizza combination string, toppings must appear in alphabetical order. This means `"Chicken,Onions,Sausage"` is valid, while `"Sausage,Chicken,Onions"` is not.

The final result table must also be sorted. The primary sorting rule is total cost in descending order. If multiple combinations have the same total cost, then the pizza combination string should be sorted in ascending lexicographical order.

The input consists of a single table:

| Column | Description |
| --- | --- |
| topping_name | Unique topping name |
| cost | Cost of the topping |

Since `topping_name` is the primary key, we know there are no duplicate topping rows.

Conceptually, this is a combinations problem implemented in SQL. We need to select all unique groups of three toppings without repetition and without generating duplicate permutations.

A naive solution might accidentally generate:

- `(Chicken, Pepperoni, Sausage)`
- `(Chicken, Sausage, Pepperoni)`
- `(Pepperoni, Chicken, Sausage)`

All of these represent the same pizza. The problem requires us to generate only one canonical version, with toppings ordered alphabetically.

The main challenge is therefore not the arithmetic, but correctly generating unique combinations.

## Approaches

### Brute Force Approach

The brute force idea is to generate every possible triple of rows from the table. We could self join the table three times and then filter invalid combinations afterward.

For example, we might produce all triples `(t1, t2, t3)` and then discard rows where toppings repeat.

However, this still produces many duplicate permutations of the same pizza combination. We would then need additional logic to eliminate duplicates.

While this works logically, it performs unnecessary work because it generates permutations instead of combinations.

### Optimal Approach

The key insight is that we can avoid duplicates entirely by enforcing an ordering condition during the self join.

If we require:

```
t1.topping_name < t2.topping_name
AND t2.topping_name < t3.topping_name
```

then:

- Every topping is guaranteed to be distinct.
- Every combination appears exactly once.
- The toppings automatically appear in alphabetical order.

This transforms the problem into a clean combinations-generation query using three self joins.

We then calculate the total cost using simple addition and sort the final output according to the required ordering rules.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n³) | O(n³) | Generates all permutations and filters duplicates later |
| Optimal | O(n³) | O(n³) | Generates only valid ordered combinations directly |

Even though both approaches are cubic in theory, the optimal solution avoids redundant permutations and is much cleaner and more efficient in practice.

## Algorithm Walkthrough

1. Create three aliases of the `Toppings` table, typically named `t1`, `t2`, and `t3`.
2. Perform a self join between these aliases to simulate choosing three toppings.
3. Enforce alphabetical ordering using:

```
t1.topping_name < t2.topping_name
AND t2.topping_name < t3.topping_name
```

This guarantees:

- No repeated toppings
- No duplicate permutations
- Alphabetical ordering inside the pizza string
4. Build the pizza description using string concatenation:

```
CONCAT(t1.topping_name, ',', t2.topping_name, ',', t3.topping_name)
```
5. Compute total cost:

```
ROUND(t1.cost + t2.cost + t3.cost, 2)
```
6. Sort the result:

- First by `total_cost DESC`
- Then by `pizza ASC`

### Why it works

The ordering constraint is the core correctness guarantee. By forcing:

```
t1.topping_name < t2.topping_name < t3.topping_name
```

we ensure every valid set of three toppings is generated exactly once. No repeated toppings are possible because strict inequality is enforced. No duplicate permutations appear because only the alphabetically ordered arrangement is allowed.

Therefore, every row in the output corresponds to one unique valid pizza combination.

## SQL Solution

```sql
SELECT
    CONCAT(
        t1.topping_name, ',',
        t2.topping_name, ',',
        t3.topping_name
    ) AS pizza,
    ROUND(t1.cost + t2.cost + t3.cost, 2) AS total_cost
FROM Toppings t1
JOIN Toppings t2
    ON t1.topping_name < t2.topping_name
JOIN Toppings t3
    ON t2.topping_name < t3.topping_name
ORDER BY total_cost DESC, pizza ASC;
```

## Implementation Explanation

The query uses three copies of the same table to represent the three toppings selected for a pizza.

The first join condition:

```
t1.topping_name < t2.topping_name
```

ensures that the second topping comes alphabetically after the first.

The second join condition:

```
t2.topping_name < t3.topping_name
```

ensures the third topping comes alphabetically after the second.

Together, these constraints automatically enforce uniqueness and ordering.

The `CONCAT` function builds the pizza description string exactly in the required format.

The `ROUND(..., 2)` call guarantees the total cost is displayed with two decimal places.

Finally, the `ORDER BY` clause implements the required sorting rules.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n³) | Three-way self join over toppings |
| Space | O(n³) | Result set may contain all 3-combinations |

The dominant factor is generating all possible valid triples. If there are `n` toppings, the number of unique 3-topping combinations is:

$$\binom{n}{3}$$

which grows proportionally to `O(n³)`.

## Edge Cases

### Exactly Three Toppings

If the table contains exactly three toppings, then there is only one possible pizza combination. A buggy implementation might accidentally exclude it due to incorrect join conditions. The strict ordering joins still correctly produce exactly one row.

### Multiple Combinations With Same Cost

Two different pizza combinations may have identical total costs. In that case, the problem requires sorting by pizza name in ascending order. The query explicitly handles this with:

```
ORDER BY total_cost DESC, pizza ASC
```

which guarantees deterministic ordering.

### Input Table Not Alphabetically Sorted

The rows inside the database table may appear in arbitrary order. A naive implementation might incorrectly assume input order matters. This solution does not rely on table ordering at all. The alphabetical constraints are enforced directly in the join conditions.

### Decimal Precision Issues

Floating point arithmetic can sometimes produce values like `1.749999`. The `ROUND(..., 2)` function ensures the result always matches the required monetary format with exactly two decimal places.
