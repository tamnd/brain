# LeetCode 3037 - Find Pattern in Infinite Stream II

## Problem Understanding

This problem gives us a binary pattern and access to an infinite binary stream. Unlike a normal array, we cannot directly access arbitrary positions in the stream. The only operation available is `next()`, which returns the next bit and advances the stream by one position.

Our task is to determine the first index in the stream where the given pattern appears as a contiguous subsequence.

For example, if the pattern is `[0, 1]` and the stream begins as:

```
[1, 1, 1, 0, 1, 1, ...]
```

then the first occurrence starts at index `3`, because:

```
stream[3:5] = [0, 1]
```

The pattern length can be as large as `10^4`, and the matching position is guaranteed to occur within the first `10^5` bits of the stream.

These constraints are extremely important. A naive solution that repeatedly compares the pattern against every possible starting position would perform too many redundant comparisons. Since the stream can only be read sequentially, we need an algorithm that processes each incoming bit efficiently while maintaining information about possible matches.

Several edge cases are worth noting:

- The pattern may have length `1`.
- The pattern may contain repeated prefixes and suffixes, such as `[1,0,1,0,1]`.
- The stream may contain long runs of identical bits.
- The first occurrence might start at index `0`.
- The first valid occurrence is guaranteed to exist within the first `10^5` bits, so we never need to handle the case where no match exists.

Because we are matching a pattern against a sequential stream and need linear-time processing, this is a classic application of the Knuth-Morris-Pratt (KMP) string matching algorithm.

## Approaches

### Brute Force

A straightforward approach is to consider every possible starting position in the stream.

As bits arrive from the stream, we could maintain the last `m` bits, where `m` is the pattern length. For every potential starting position, we compare the pattern against the corresponding window bit by bit.

This works because eventually we will encounter the first matching window and return its starting index.

However, the same bits may be compared many times. In the worst case, every position requires almost a full pattern comparison before failing.

For a pattern of length `m` and up to `n = 10^5` stream bits examined, the worst-case complexity becomes `O(nm)`, which is too expensive when `m = 10^4`.

### Optimal Approach: KMP on the Stream

The key observation is that when a mismatch occurs, we do not need to restart matching from scratch.

Suppose we have already matched a prefix of the pattern. If the next bit causes a mismatch, some suffix of the matched portion may still be a valid prefix of the pattern. KMP precomputes this information in the failure function (also called the LPS array, longest proper prefix which is also a suffix).

Using the LPS array, every stream bit is processed exactly once. Instead of backing up and rechecking previously matched characters, we jump directly to the next viable matching state.

This is particularly well suited for an infinite stream because KMP naturally operates in an online fashion. We only need the current matching state and the next incoming bit.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(nm) | O(m) | Repeatedly compares pattern against candidate positions |
| Optimal | O(n + m) | O(m) | KMP processes each stream bit once |

Here, `m` is the pattern length and `n` is the number of stream bits read until the first match is found.

## Algorithm Walkthrough

### Step 1: Build the LPS Array

For every position in the pattern, compute the length of the longest proper prefix that is also a suffix ending at that position.

For example:

```
Pattern: 1 0 1 0 1
LPS:     0 0 1 2 3
```

The value `3` at the last position means that after matching the entire pattern, the suffix `[1,0,1]` is also a prefix of length `3`.

### Step 2: Initialize Matching State

Maintain:

- `matched`, the current number of pattern characters matched.
- `index`, the current stream position.

Initially:

```
matched = 0
index = 0
```

### Step 3: Read Bits from the Stream

Repeatedly call:

```
bit = stream.next()
```

Each bit is processed exactly once.

### Step 4: Handle Mismatches Using LPS

While we have a mismatch and some prefix has already been matched:

```
while matched > 0 and bit != pattern[matched]:
    matched = lps[matched - 1]
```

Instead of restarting from zero, we jump to the longest valid prefix that could still continue matching.

### Step 5: Extend the Match

If the current bit matches the next required pattern element:

```
if bit == pattern[matched]:
    matched += 1
```

### Step 6: Check for Full Pattern Match

When:

```
matched == len(pattern)
```

the entire pattern has been matched.

The ending position is:

```
index
```

and the starting position is:

```
index - len(pattern) + 1
```

Return that value immediately.

### Step 7: Advance the Stream Index

After processing the current bit:

```
index += 1
```

Continue until a match is found.

### Why it works

The KMP invariant is that `matched` always represents the length of the longest prefix of the pattern that matches a suffix of the stream processed so far.

Whenever a mismatch occurs, the LPS array tells us exactly how much of the previously matched information remains useful. Therefore, no stream bit is reconsidered unnecessarily. Once `matched` reaches the pattern length, we know the pattern appears ending at the current stream position, so the computed starting index is the first occurrence.

## Python Solution

```python
from typing import List, Optional

# Definition for an infinite stream.
# class InfiniteStream:
#     def next(self) -> int:
#         pass

class Solution:
    def findPattern(
        self,
        stream: Optional['InfiniteStream'],
        pattern: List[int]
    ) -> int:
        m = len(pattern)

        lps = [0] * m
        length = 0

        for i in range(1, m):
            while length > 0 and pattern[i] != pattern[length]:
                length = lps[length - 1]

            if pattern[i] == pattern[length]:
                length += 1

            lps[i] = length

        matched = 0
        index = 0

        while True:
            bit = stream.next()

            while matched > 0 and bit != pattern[matched]:
                matched = lps[matched - 1]

            if bit == pattern[matched]:
                matched += 1

            if matched == m:
                return index - m + 1

            index += 1
```

The implementation consists of two phases.

The first phase constructs the LPS array. For every position in the pattern, it computes the length of the longest prefix that can still be reused after a mismatch. This preprocessing takes linear time.

The second phase performs online matching against the stream. Each call to `stream.next()` produces one new bit. The variable `matched` tracks how many pattern characters currently match the suffix of the processed stream.

When a mismatch occurs, the LPS array determines the next candidate prefix length. This avoids repeating comparisons. Once `matched` reaches the pattern length, the pattern has been fully matched and the starting index is returned.

## Go Solution

```go
/**
 * Definition for an infinite stream.
 * type InfiniteStream interface {
 *     Next() int
 * }
 */
func findPattern(stream InfiniteStream, pattern []int) int {
	m := len(pattern)

	lps := make([]int, m)

	length := 0
	for i := 1; i < m; i++ {
		for length > 0 && pattern[i] != pattern[length] {
			length = lps[length-1]
		}

		if pattern[i] == pattern[length] {
			length++
		}

		lps[i] = length
	}

	matched := 0
	index := 0

	for {
		bit := stream.Next()

		for matched > 0 && bit != pattern[matched] {
			matched = lps[matched-1]
		}

		if bit == pattern[matched] {
			matched++
		}

		if matched == m {
			return index - m + 1
		}

		index++
	}
}
```

The Go implementation follows exactly the same KMP logic. Slices are used for the pattern and LPS array. Since the pattern length is at least one, accessing `pattern[matched]` is always safe. No special handling for integer overflow is needed because indices never exceed the guaranteed search range of approximately `10^5`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n + m) | LPS construction is O(m), stream matching is O(n) |
| Space | O(m) | Only the LPS array is stored |

The key property of KMP is that every stream bit is processed at most a constant number of times. Although mismatches may trigger several LPS jumps, the total number of such jumps over the entire execution is linear. Therefore the matching phase runs in `O(n)` time, yielding an overall complexity of `O(n + m)`.

## Edge Cases

### Pattern Length Equals One

A common source of bugs is assuming the pattern has at least two elements. When the pattern length is one, the first matching bit should immediately produce an answer. The implementation handles this naturally because the KMP state reaches `matched == 1` as soon as the required bit is read.

### Highly Repetitive Patterns

Patterns such as `[1,0,1,0,1,0,1]` contain large overlapping prefixes and suffixes. A naive algorithm repeatedly rechecks the same positions after mismatches, leading to poor performance. The LPS array allows the algorithm to jump directly to the next valid state, preserving linear complexity.

### Match Starting at Index Zero

The very first bits of the stream may already equal the pattern. Some implementations accidentally advance the index before checking for completion and return an off-by-one result. This solution checks for a completed match immediately after updating `matched`, ensuring that a pattern beginning at index `0` correctly returns `0`.

### Long Runs of Identical Bits

Streams like `[0,0,0,0,0,...]` can cause many failed comparisons for naive approaches. KMP efficiently handles these situations because each mismatch uses the LPS table to determine the next valid matching state instead of restarting completely.

### Overlapping Matches

Patterns may overlap with themselves. For example, `[1,1,1]` inside a stream of many ones creates multiple overlapping candidate matches. The KMP invariant guarantees that the longest valid prefix is always maintained, ensuring the earliest occurrence is found without missing any potential starting positions.
