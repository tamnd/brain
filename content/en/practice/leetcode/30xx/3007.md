# LeetCode 3007 - Maximum Number That Sum of the Prices Is Less Than or Equal to K

## Problem Understanding

This problem defines a special "price" for every integer based on its binary representation.

For a given integer `x`, we only care about bit positions:

- `x`
- `2x`
- `3x`
- and so on

The positions are counted from the least significant bit, using 1-based indexing.

For example:

- Position 1 is the rightmost bit
- Position 2 is the next bit
- Position 3 is the next bit after that

The price of a number is the count of set bits (`1` bits) that appear at those positions.

Suppose `x = 2`.

Then we only check positions:

- 2
- 4
- 6
- ...

If a number has `1`s in those positions, each contributes to the price.

The accumulated price of a number `num` is:

$$\sum_{i=1}^{num} \text{price}(i)$$

A number is considered cheap if its accumulated price is less than or equal to `k`.

The task is to find the largest cheap number.

The constraints are extremely important:

- `k` can be as large as $10^{15}$
- `x` is at most 8

A brute-force simulation that computes prices one number at a time would be far too slow. Even iterating up to $10^{15}$ is impossible.

The problem therefore requires a mathematical counting technique combined with binary search.

Several edge cases are important:

- Very small `k`, where only a few numbers are valid
- `x = 1`, where every set bit contributes to the price
- Large values of `k`, where the answer may become extremely large
- Bit positions beyond the current number length, which should contribute nothing
- Powers of two, because they create sharp transitions in bit-count patterns

## Approaches

### Brute Force Approach

The most direct solution is to iterate through every number starting from 1.

For each number:

1. Convert it to binary
2. Count the set bits at positions `x`, `2x`, `3x`, and so on
3. Add this price to a running accumulated total
4. Stop once the accumulated price exceeds `k`

The previous number would then be the answer.

This approach is correct because it literally follows the problem definition exactly.

However, it is completely impractical for the given constraints. The answer can become extremely large, potentially around $10^{15}$ or higher. Computing prices one number at a time would take far too long.

### Key Insight

The important observation is that we do not need to evaluate numbers individually.

Instead, for a candidate number `n`, we can efficiently compute:

$$\text{accumulatedPrice}(n)$$

by counting how many times each relevant bit position is set among all numbers from `1` to `n`.

This transforms the problem into:

- Given `n`, determine whether accumulatedPrice(`n`) ≤ `k`
- Use binary search to find the largest valid `n`

The remaining challenge is efficiently counting how many numbers from `0` to `n` have a particular bit set.

This is a classic bit-counting pattern problem.

For bit position `b`:

- Bits repeat in cycles of length $2^b$
- Half the cycle contains zeros
- Half the cycle contains ones

Using this repeating structure, we can count set bits in logarithmic time.

## Algorithm Walkthrough

### Step 1: Define a helper function for counting set bits

We create a function that counts how many numbers from `0` to `n` have a set bit at a specific position.

Suppose we want to count how many numbers have the 3rd bit set.

Bits follow repeating patterns:

- 4 zeros
- 4 ones
- repeated forever

More generally:

- Cycle length = $2^b$
- Ones appear for half the cycle

We compute:

- Number of complete cycles
- Contribution from complete cycles
- Contribution from the remaining partial cycle

This gives the count in constant time.

### Step 2: Compute accumulated price for a candidate number

Now we evaluate a candidate `n`.

We iterate through bit positions:

- `x`
- `2x`
- `3x`
- ...

For each such position:

1. Count how many numbers from `0` to `n` have that bit set
2. Add that count to the total accumulated price

This works because every set bit at those positions contributes exactly 1 to the price.

### Step 3: Binary search for the largest valid number

The accumulated price is monotonic:

- Larger numbers can only increase the accumulated price
- It never decreases

Therefore binary search applies perfectly.

We search over a large range:

- Low = 0
- High = $10^{18}$

For each midpoint:

- Compute accumulated price
- If price ≤ `k`, the midpoint is valid
- Otherwise it is too large

We continue until we find the largest valid number.

### Why It Works

The algorithm works because the accumulated price function is monotonic.

If a number `n` is cheap, then every smaller number must also be cheap.

If a number `n` is not cheap, then every larger number must also be not cheap.

This monotonic property guarantees binary search correctness.

The bit-counting formula is correct because binary digits follow fixed repeating cycles. Every bit alternates between blocks of zeros and ones with predictable lengths, allowing exact counting without iterating through all numbers.

## Python Solution

```python
from typing import List

class Solution:
    def findMaximumNumber(self, k: int, x: int) -> int:
        def count_set_bits(n: int, bit: int) -> int:
            cycle = 1 << bit
            half = 1 << (bit - 1)

            full_cycles = (n + 1) // cycle
            result = full_cycles * half

            remainder = (n + 1) % cycle
            result += max(0, remainder - half)

            return result

        def accumulated_price(n: int) -> int:
            total = 0
            bit = x

            while bit <= 60:
                total += count_set_bits(n, bit)
                bit += x

            return total

        left = 0
        right = 10**18

        while left < right:
            mid = (left + right + 1) // 2

            if accumulated_price(mid) <= k:
                left = mid
            else:
                right = mid - 1

        return left
```

### Implementation Explanation

The `count_set_bits` function computes how many integers from `0` to `n` contain a set bit at a given position.

The cycle logic is the key mathematical observation:

- For bit position `b`
- The pattern repeats every $2^b$
- Exactly half of each cycle contains ones

We first count contributions from full cycles, then handle the remaining partial cycle separately.

The `accumulated_price` function evaluates the total accumulated price for all numbers from `1` to `n`.

It only checks positions that are multiples of `x`, exactly matching the problem statement.

The binary search then finds the largest valid number whose accumulated price does not exceed `k`.

The upper bound `10**18` is safely large enough for all constraints.

## Go Solution

```go
package main

func findMaximumNumber(k int64, x int) int64 {
	countSetBits := func(n int64, bit int) int64 {
		cycle := int64(1 << bit)
		half := int64(1 << (bit - 1))

		fullCycles := (n + 1) / cycle
		result := fullCycles * half

		remainder := (n + 1) % cycle

		if remainder > half {
			result += remainder - half
		}

		return result
	}

	accumulatedPrice := func(n int64) int64 {
		var total int64 = 0

		for bit := x; bit <= 60; bit += x {
			total += countSetBits(n, bit)
		}

		return total
	}

	var left int64 = 0
	var right int64 = 1_000_000_000_000_000_000

	for left < right {
		mid := (left + right + 1) / 2

		if accumulatedPrice(mid) <= k {
			left = mid
		} else {
			right = mid - 1
		}
	}

	return left
}
```

### Go-Specific Notes

The Go solution closely mirrors the Python implementation.

Several implementation details are important:

- `int64` is required because values may exceed standard 32-bit integer ranges
- Bit shifts use integer literals carefully to avoid overflow
- Anonymous helper functions are used to keep the logic organized
- The binary search uses `mid := (left + right + 1) / 2` to avoid infinite loops when searching for the upper bound

Unlike Python, Go integers do not automatically expand in size, so explicit `int64` usage is necessary throughout the implementation.

## Edge Cases

One important edge case occurs when `k = 0`. In this situation, no accumulated price is allowed at all. The only valid answer is 0, because even the number 1 contributes a positive price whenever its relevant bit position is counted. The binary search naturally handles this because accumulated prices for positive numbers exceed the limit immediately.

Another important edge case occurs when `x = 1`. In this configuration, every set bit contributes to the price. This creates the largest possible accumulated prices and stresses the counting logic heavily. The implementation handles this correctly because it iterates through every bit position and sums all contributions.

A third important edge case involves very large values of `k`, close to $10^{15}$. In such cases, the answer may become extremely large. A naive solution would overflow or time out. The implementation avoids this problem by using logarithmic binary search and constant-time bit counting formulas. Both the Python and Go versions use integer types capable of handling very large values safely.

Another subtle edge case involves bit positions larger than the binary length of the current number. For example, checking bit position 20 for small numbers should contribute zero. The counting formula naturally handles this because incomplete cycles contribute nothing when the range is too small.

Finally, powers of two are important edge cases because bit patterns change sharply at those boundaries. The cycle-based counting logic correctly handles these transitions because it is based on exact binary repetition patterns rather than approximation or simulation.
