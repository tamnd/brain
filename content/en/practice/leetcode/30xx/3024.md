## Solution

## Problem Understanding

The problem gives us an integer array `nums` of length `3`, where each element represents the length of one side of a potential triangle. Our task is to determine what type of triangle these three sides form.

There are four possible outputs:

- `"equilateral"` if all three sides are equal
- `"isosceles"` if exactly two sides are equal
- `"scalene"` if all three sides are different
- `"none"` if the three sides cannot form a valid triangle

The important detail is that not every set of three positive numbers can form a triangle. A valid triangle must satisfy the triangle inequality rule:

- The sum of any two sides must be strictly greater than the third side

For three sides `a`, `b`, and `c`, the following must all be true:

- `a + b > c`
- `a + c > b`
- `b + c > a`

Since the array size is always exactly `3`, the input scale is extremely small. The constraints tell us:

- `nums.length == 3`
- `1 <= nums[i] <= 100`

This means efficiency is not a concern in practice, because we only ever process three numbers. However, we still want a clean and logically correct solution.

Several edge cases are important:

- Three equal sides, such as `[5,5,5]`
- Two equal sides that still form a valid triangle, such as `[5,5,8]`
- Inputs that fail the triangle inequality, such as `[1,2,3]`
- Inputs with distinct sides that form a triangle, such as `[3,4,5]`

A naive implementation may incorrectly classify invalid triangles if it checks equality before validating the triangle inequality. For example, `[1,1,2]` has two equal sides, but it is not a valid triangle because `1 + 1` is not greater than `2`.

## Approaches

### Brute Force Approach

The brute force idea is to explicitly check all three triangle inequality conditions individually, then determine the triangle type by comparing every pair of sides.

The algorithm works like this:

1. Extract the three side lengths.
2. Verify:

- `a + b > c`
- `a + c > b`
- `b + c > a`
3. If any condition fails, return `"none"`.
4. Otherwise:

- If all three sides are equal, return `"equilateral"`.
- If exactly two sides are equal, return `"isosceles"`.
- Otherwise return `"scalene"`.

This approach is fully correct because it directly implements the mathematical definition of a triangle and the triangle classifications.

Even though the problem size is tiny, this solution performs several redundant comparisons. We can simplify the logic using sorting.

### Optimal Approach

The key observation is that after sorting the three side lengths in nondecreasing order:

```
a <= b <= c
```

we only need to check a single triangle condition:

```
a + b > c
```

Why does this work?

Because `c` is the largest side. If the two smaller sides together are greater than the largest side, then the other two triangle inequalities are automatically satisfied.

For example:

```
a + c > b
```

is always true because `c >= b` and `a > 0`.

Similarly:

```
b + c > a
```

is also automatically true.

Sorting therefore simplifies the validation step and makes the implementation cleaner.

After validating the triangle, we classify it:

- All equal → `"equilateral"`
- Exactly two equal → `"isosceles"`
- All distinct → `"scalene"`

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |

| Optimal | O(1) | O(1) | Sorts three values and reduces validation to one condition |

Even though sorting is technically `O(n log n)`, here `n = 3`, which is constant. Therefore the runtime is still effectively constant.

## Algorithm Walkthrough

1. Sort the array in nondecreasing order.

After sorting, the sides become:

```
nums[0] <= nums[1] <= nums[2]
```

This arrangement makes the triangle inequality check simpler because the largest side is always at index `2`.
2. Check whether the sides can form a triangle.

A valid triangle must satisfy:

```
nums[0] + nums[1] > nums[2]
```

If this condition fails, return `"none"` immediately.
3. Check whether all three sides are equal.

If:

```
nums[0] == nums[1] == nums[2]
```

then the triangle is equilateral.
4. Check whether exactly two sides are equal.

Since the equilateral case has already been handled, we only need to check whether any pair matches:

```
nums[0] == nums[1]
or
nums[1] == nums[2]
```

If true, return `"isosceles"`.
5. Otherwise, all three sides are different.

Return `"scalene"`.

### Why it works

Sorting guarantees that the largest side is placed last. For three sorted sides `a <= b <= c`, the only necessary triangle inequality check is `a + b > c`. If this condition holds, the other two inequalities are automatically true.

After confirming the triangle is valid, the side equality relationships uniquely determine the triangle type:

- Three equal sides means equilateral
- Exactly two equal sides means isosceles
- No equal sides means scalene

Therefore the algorithm always returns the correct classification.

## Python Solution

```python
from typing import List

class Solution:
    def triangleType(self, nums: List[int]) -> str:
        nums.sort()

        # Check triangle validity
        if nums[0] + nums[1] <= nums[2]:
            return "none"

        # All sides equal
        if nums[0] == nums[1] == nums[2]:
            return "equilateral"

        # Exactly two sides equal
        if nums[0] == nums[1] or nums[1] == nums[2]:
            return "isosceles"

        # All sides different
        return "scalene"
```

The implementation begins by sorting the array so the largest side is always at the end. This allows the triangle validity check to be reduced to a single condition.

The first conditional checks whether the sum of the two smaller sides is less than or equal to the largest side. If so, the sides cannot form a valid triangle, and the function immediately returns `"none"`.

Next, the code checks whether all three sides are equal. If they are, the triangle is equilateral.

If the triangle is not equilateral, the implementation checks whether either adjacent pair is equal. Since the array is sorted, any equal values must appear next to each other. If exactly one pair matches, the triangle is isosceles.

Finally, if no sides are equal, the triangle must be scalene.

## Go Solution

```go
package main

import "sort"

func triangleType(nums []int) string {
	sort.Ints(nums)

	// Check triangle validity
	if nums[0]+nums[1] <= nums[2] {
		return "none"
	}

	// All sides equal
	if nums[0] == nums[1] && nums[1] == nums[2] {
		return "equilateral"
	}

	// Exactly two sides equal
	if nums[0] == nums[1] || nums[1] == nums[2] {
		return "isosceles"
	}

	// All sides different
	return "scalene"
}
```

The Go solution follows the same logic as the Python implementation. The main difference is that Go uses `sort.Ints(nums)` to sort the slice in place.

Integer overflow is not a concern because the maximum side length is only `100`, so adding two sides always remains safely within Go's integer range.

Go slices are mutable, so sorting directly modifies the input slice. This behavior is acceptable for this problem.

## Edge Cases

One important edge case occurs when the sum of the two smaller sides is exactly equal to the largest side, such as `[1,2,3]`. A naive implementation might incorrectly treat this as a valid triangle. However, the triangle inequality requires the sum to be strictly greater, not greater than or equal. The implementation correctly handles this using:

```
if nums[0] + nums[1] <= nums[2]:
```

Another important edge case is a triangle with repeated sides that is still invalid, such as `[1,1,2]`. A careless solution might immediately classify this as isosceles because two sides match. The implementation avoids this bug by always validating the triangle before checking the triangle type.

Unsorted input is another common source of mistakes. For example, `[8,5,5]` represents a valid isosceles triangle, but without sorting, identifying the largest side becomes less straightforward. Sorting guarantees that the largest side is always at the final index, simplifying both correctness and implementation.

A final edge case involves the smallest and largest allowed values. Inputs such as `[1,1,1]` and `[100,100,100]` verify that the implementation correctly handles the full constraint range without overflow or comparison issues.
