## Solution

## Problem Understanding

The problem gives us an array of positive integers. Before selecting elements, we are allowed to independently increase any element by at most `1`. This means every number `x` has two possible final values:

- keep it as `x`
- change it to `x + 1`

After performing these optional modifications, we want to choose a subset of elements such that, when sorted, they form a sequence of consecutive integers with no gaps and no duplicates.

The goal is to maximize the number of selected elements.

A very important detail is that the selected numbers must become consecutive after sorting. That means if we select `k` numbers, they must look exactly like:

```
a, a+1, a+2, ..., a+k-1
```

No repeated values are allowed inside the chosen sequence because duplicates break consecutiveness.

The constraints are large:

- `nums.length` can be up to `10^5`
- `nums[i]` can be up to `10^6`

These limits immediately tell us that exponential search or quadratic dynamic programming will not work. We need something around `O(n log n)` or better.

Several edge cases are important:

- Arrays with many duplicates, because duplicates can sometimes be shifted upward to create longer chains.
- Arrays with large gaps, where no consecutive sequence can be formed.
- Arrays already containing long consecutive runs.
- Arrays where incrementing the wrong occurrence destroys a better future chain.
- Single-element arrays, where the answer is always `1`.

The core challenge is deciding, for each value, whether to keep it unchanged or increment it by one so that we maximize the length of a consecutive chain.

## Approaches

### Brute Force Approach

A brute force solution would try every possible modification pattern.

Since each element has two choices:

- keep it unchanged
- increment it by one

there are `2^n` possible transformed arrays.

For each transformed array, we could:

1. Sort the resulting numbers.
2. Find the largest subset forming consecutive integers.
3. Track the maximum length.

This approach is correct because it explicitly checks every possible modification configuration. However, it is completely infeasible for `n = 10^5`.

Even for `n = 30`, `2^30` possibilities are already too large.

A slightly better brute force idea would involve dynamic programming over subsets or trying all possible consecutive intervals, but these still become too slow under the constraints.

### Key Insight

The crucial observation is that each number only affects at most two possible values:

```
x or x + 1
```

After sorting the array, we can process numbers in increasing order and build the longest possible consecutive chain ending at a particular value.

Suppose we are currently processing number `x`.

We have two options:

- use it as `x`
- use it as `x + 1`

If we want a consecutive sequence ending at `x`, then we should extend a sequence ending at `x - 1`.

If we want a consecutive sequence ending at `x + 1`, then we should extend a sequence ending at `x`.

This naturally leads to dynamic programming with a hash map:

```
dp[v] = longest consecutive sequence ending at value v
```

For each number `x`:

```
dp[x + 1] = dp[x] + 1
dp[x] = dp[x - 1] + 1
```

The order matters, which is why sorting is required.

This transforms the problem into an efficient `O(n log n)` solution.

## Algorithm Walkthrough

1. Sort the array in nondecreasing order.

Sorting is essential because consecutive sequences depend on numeric order. Processing smaller values first ensures that when we try to extend a chain ending at `x - 1`, that information has already been computed.
2. Create a hash map called `dp`.

The meaning of the map is:

```
dp[v] = maximum length of a consecutive sequence ending at value v
```
3. Iterate through every number `x` in sorted order.

Each number can contribute in two ways.
4. First, try using the number as `x + 1`.

If we place this element at value `x + 1`, then it can extend a sequence ending at `x`.

So:

```
dp[x + 1] = max(dp[x + 1], dp[x] + 1)
```
5. Then, try using the number as `x`.

If we keep the number unchanged, it can extend a sequence ending at `x - 1`.

So:

```
dp[x] = max(dp[x], dp[x - 1] + 1)
```
6. Track the global maximum answer during updates.
7. Return the maximum value stored in the dynamic programming table.

### Why it works

The dynamic programming invariant is:

```
dp[v] always stores the longest valid consecutive sequence ending at value v
```

When processing a number `x`, the only valid ways to extend a consecutive sequence are:

- append `x` after a sequence ending at `x - 1`
- append `x + 1` after a sequence ending at `x`

Because the array is processed in sorted order, all smaller-ending sequences have already been finalized when we need them. Therefore every transition is valid, and the algorithm explores all optimal possibilities.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Solution:
    def maxSelectedElements(self, nums: List[int]) -> int:
        nums.sort()

        dp = defaultdict(int)
        answer = 1

        for value in nums:
            dp[value + 1] = max(dp[value + 1], dp[value] + 1)
            dp[value] = max(dp[value], dp[value - 1] + 1)

            answer = max(answer, dp[value], dp[value + 1])

        return answer
```

The implementation starts by sorting the array so that smaller numbers are processed before larger ones.

The `dp` hash map stores the best consecutive sequence ending at each possible value.

For every number:

- `dp[value + 1]` represents using the current element after incrementing it
- `dp[value]` represents using the current element unchanged

Each transition extends an already-existing consecutive chain.

The answer is updated continuously because the best sequence can end at many different values.

The use of `defaultdict(int)` automatically treats missing states as `0`, which simplifies the transitions.

## Go Solution

```go
package main

import (
	"sort"
)

func maxSelectedElements(nums []int) int {
	sort.Ints(nums)

	dp := make(map[int]int)
	answer := 1

	for _, value := range nums {
		if dp[value]+1 > dp[value+1] {
			dp[value+1] = dp[value] + 1
		}

		if dp[value-1]+1 > dp[value] {
			dp[value] = dp[value-1] + 1
		}

		if dp[value] > answer {
			answer = dp[value]
		}

		if dp[value+1] > answer {
			answer = dp[value+1]
		}
	}

	return answer
}
```

The Go implementation follows the same logic as the Python version.

A `map[int]int` replaces Python's `defaultdict(int)`. Missing keys automatically evaluate to `0` when accessed, which matches the desired behavior.

The array is sorted using `sort.Ints`.

Since the problem constraints fit comfortably within 32-bit integers, normal Go `int` arithmetic is sufficient.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | Sorting dominates the runtime |
| Space | O(n) | Hash map stores dynamic programming states |

The dynamic programming loop itself is linear because every element performs only constant-time hash map updates.

Sorting costs `O(n log n)`, which becomes the overall complexity.

The hash map can contain at most `O(n)` distinct states because every element contributes to at most two values.

## Edge Cases

One important edge case is an array consisting entirely of duplicates, such as `[1,1,1,1]`. A naive implementation might incorrectly count all four elements, but duplicates are not allowed inside a consecutive sequence. Since each value can increase by at most one, the best achievable sequence is only `[1,2]`, with length `2`. The dynamic programming approach handles this naturally because sequences are indexed by their ending value.

Another tricky case is when values are already consecutive, such as `[1,2,3,4,5]`. In this situation, incrementing elements unnecessarily could break the chain. The algorithm safely considers both choices independently and preserves the optimal unchanged chain.

A third important edge case involves many overlapping duplicate decisions, such as `[1,1,2,2,3,3]`. Greedy local choices can fail here because incrementing one duplicate may interfere with another future extension. The dynamic programming map avoids this issue by storing the best possible chain length for every ending value, ensuring all valid possibilities are preserved.

A final edge case is arrays with large numeric gaps, such as `[1,100,1000]`. No modifications can bridge these gaps because each number changes by at most one. The algorithm correctly produces `1` because every element forms only an isolated consecutive sequence.
