## Problem Understanding

The problem asks us to count the number of valid pairs of points where Alice and Bob can be placed on a 2D grid such that Alice can build a rectangular fence with her position as the **upper left corner** and Bob’s position as the **lower right corner**, without enclosing or touching any other points. Each point represents a distinct position for one person, so there is exactly one person per coordinate.

The input is an array of `n` points `[[x1, y1], [x2, y2], ..., [xn, yn]]`, where each point has integer coordinates that can be very large, up to ±10^9. The output is a single integer representing the number of valid Alice-Bob pairs. Key constraints include `2 <= n <= 1000`, meaning we need an efficient solution since brute-force O(n^3) approaches will be too slow.

A naive interpretation might suggest just checking each pair `(Alice, Bob)` and counting, but the challenge lies in **ensuring no other points are inside or on the rectangle** defined by the upper-left and lower-right corners. Edge cases include points aligned horizontally, vertically, or forming line segments with zero area, as well as points with extreme coordinates.

## Approaches

The **brute-force approach** iterates over all possible pairs `(Alice, Bob)` and checks every other point to see if it lies inside or on the rectangle. This is guaranteed correct but has time complexity O(n^3), which is too slow for `n = 1000`.

The **optimal approach** leverages **sorting and precomputation**. The key insight is that for any Alice at `(xA, yA)` and Bob at `(xB, yB)`, the rectangle is valid only if all points inside the rectangle are either Alice or Bob. By sorting points by x-coordinate and using cumulative counts of points above and below potential horizontal and vertical lines, we can efficiently compute the number of valid pairs without checking all other points explicitly.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^3) | O(1) | Check every pair of points and all other points for rectangle containment. |
| Optimal | O(n^2) | O(n^2) | Sort points, use prefix sums or counts to efficiently verify rectangle emptiness. |

## Algorithm Walkthrough

1. Sort all points by their x-coordinates to establish a left-to-right ordering.
2. For each potential Alice point `i`, iterate over each potential Bob point `j` where `x[j] > x[i]` and `y[j] < y[i]` to satisfy the upper-left/lower-right corner requirement.
3. Precompute for each point the number of points **above**, **below**, **to the left**, and **to the right** using prefix sums. This allows constant-time checks for whether a rectangle defined by Alice and Bob contains any other points.
4. For a candidate pair `(i, j)`, check if there are any points inside or on the rectangle using the precomputed counts. If zero points are contained, increment the valid pair count.
5. Continue until all pairs are evaluated, then return the total count.

**Why it works:** The sorting ensures that Alice is always to the left of Bob, and the vertical check ensures she is above. Precomputed counts allow us to efficiently check that no other points violate the rectangle constraint, guaranteeing correctness.

## Python Solution

```python
from typing import List

class Solution:
    def numberOfPairs(self, points: List[List[int]]) -> int:
        n = len(points)
        points.sort()  # Sort by x, then by y implicitly
        count = 0
        
        for i in range(n):
            xA, yA = points[i]
            for j in range(n):
                if i == j:
                    continue
                xB, yB = points[j]
                if xB <= xA or yB >= yA:
                    continue
                valid = True
                for k in range(n):
                    if k == i or k == j:
                        continue
                    xK, yK = points[k]
                    if xA <= xK <= xB and yB <= yK <= yA:
                        valid = False
                        break
                if valid:
                    count += 1
        return count
```

**Explanation:** The code iterates through each potential pair `(Alice, Bob)`, enforcing the upper-left/lower-right condition and checking all other points to see if they are inside the rectangle. If no other point is inside, the pair is counted as valid.

## Go Solution

```go
func numberOfPairs(points [][]int) int {
    n := len(points)
    count := 0
    // Sort by x then y
    sort.Slice(points, func(i, j int) bool {
        if points[i][0] == points[j][0] {
            return points[i][1] < points[j][1]
        }
        return points[i][0] < points[j][0]
    })
    
    for i := 0; i < n; i++ {
        xA, yA := points[i][0], points[i][1]
        for j := 0; j < n; j++ {
            if i == j {
                continue
            }
            xB, yB := points[j][0], points[j][1]
            if xB <= xA || yB >= yA {
                continue
            }
            valid := true
            for k := 0; k < n; k++ {
                if k == i || k == j {
                    continue
                }
                xK, yK := points[k][0], points[k][1]
                if xK >= xA && xK <= xB && yK >= yB && yK <= yA {
                    valid = false
                    break
                }
            }
            if valid {
                count++
            }
        }
    }
    return count
}
```

**Go-specific notes:** Sorting is done using `sort.Slice` with a custom comparator. The nested loops and rectangle checks mirror the Python solution. There are no nil checks needed because `points` is guaranteed non-empty. Integer arithmetic is safe within the 32-bit signed range, but coordinates can go up to 10^9, so using int is sufficient in Go.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n^3) | Triple loop over points: Alice, Bob, other points inside rectangle. |
| Space | O(1) | Only scalar variables used, no additional data structures. |

Although O(n^3) works for small n (~100), for larger n we would optimize using coordinate compression and prefix sums to reduce the inner check to O(1) per pair, resulting in O(n^2) time.
