# LeetCode 3034 - Number of Subarrays That Match a Pattern

## Problem Understanding

The problem gives us two arrays:

- `nums`, an integer array of length `n`
- `pattern`, an integer array of length `m`, where every value is one of `-1`, `0`, or `1`

We must count how many subarrays of `nums` with length `m + 1` satisfy the relationship pattern described by `pattern`.

The key observation is that the pattern does not care about the exact values inside the subarray. Instead, it only cares about the relationship between consecutive elements.

For every index `k` in `pattern`:

- `pattern[k] == 1` means the next number must be larger
- `pattern[k] == 0` means the next number must be equal
- `pattern[k] == -1` means the next number must be smaller

Suppose we are checking a subarray starting at index `i`. Then:

- compare `nums[i]` and `nums[i + 1]`
- compare `nums[i + 1]` and `nums[i + 2]`
- continue until all `m` comparisons are checked

If every comparison matches the corresponding value in `pattern`, then this subarray is counted.

The constraints are relatively small:

- `n <= 100`
- `m < n`

Since the array size is tiny, even quadratic solutions are acceptable. However, the problem is still useful for practicing pattern transformation and matching techniques.

An important conceptual simplification is that the problem can be transformed into matching two arrays of comparisons:

- Convert consecutive relationships in `nums` into an array of `-1`, `0`, and `1`
- Then search for occurrences of `pattern`

Several edge cases are important:

- All elements equal
- Strictly increasing or decreasing arrays
- Patterns of length `1`
- Multiple overlapping matches
- Arrays containing repeated numbers

The problem guarantees that `pattern.length < nums.length`, so every candidate subarray always has valid bounds.

## Approaches

### Brute Force Approach

The simplest solution is to examine every subarray of size `m + 1`.

For each starting position:

1. Compare consecutive elements inside the subarray.
2. Check whether each comparison matches the corresponding value in `pattern`.
3. If every comparison matches, increment the answer.

This approach is correct because it directly implements the problem definition. Every possible candidate subarray is checked exactly once.

The time complexity is `O((n - m) * m)` because:

- There are `n - m` candidate subarrays.
- Each subarray requires up to `m` comparisons.

With `n <= 100`, this solution is already fast enough.

### Optimal Approach

A cleaner and more scalable solution is to transform the problem into array pattern matching.

Instead of repeatedly comparing raw numbers, we first build a derived array called `relations`.

For every adjacent pair in `nums`:

- store `1` if increasing
- store `0` if equal
- store `-1` if decreasing

Example:

```
nums = [1,4,4,1]

relations = [1,0,-1]
```

Now the problem becomes:

"How many subarrays inside `relations` are exactly equal to `pattern`?"

This avoids repeatedly recomputing comparisons and separates the transformation step from the matching step.

Although the asymptotic complexity remains similar for these constraints, the logic becomes cleaner and closer to classic string matching problems.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O((n - m) * m) | O(1) | Directly checks every candidate subarray |
| Optimal | O(n + (n - m) * m) | O(n) | Converts comparisons into a relation array, then performs pattern matching |

## Algorithm Walkthrough

### Optimal Algorithm

1. Create an empty array called `relations`.
2. Traverse the `nums` array from left to right and compare adjacent elements.
3. For every pair `nums[i]` and `nums[i + 1]`:

- append `1` if increasing
- append `0` if equal
- append `-1` if decreasing
4. After this step, `relations` has length `n - 1`.
5. Traverse all possible starting positions inside `relations` where a subarray of length `m` can fit.
6. For each starting position:

- compare every element of the window with `pattern`
- if all elements match, increment the answer
7. Return the final count.

### Why it works

Each value in `pattern` describes the relationship between two consecutive numbers. The `relations` array stores exactly these relationships for the entire `nums` array.

A subarray of length `m + 1` in `nums` produces exactly `m` consecutive comparisons. Therefore, a subarray matches the pattern if and only if the corresponding segment inside `relations` is identical to `pattern`.

Because the algorithm checks every possible segment of length `m` in `relations`, every valid matching subarray is counted exactly once.

## Python Solution

```python
from typing import List

class Solution:
    def countMatchingSubarrays(self, nums: List[int], pattern: List[int]) -> int:
        relations = []

        for i in range(len(nums) - 1):
            if nums[i + 1] > nums[i]:
                relations.append(1)
            elif nums[i + 1] == nums[i]:
                relations.append(0)
            else:
                relations.append(-1)

        pattern_length = len(pattern)
        count = 0

        for start in range(len(relations) - pattern_length + 1):
            match = True

            for j in range(pattern_length):
                if relations[start + j] != pattern[j]:
                    match = False
                    break

            if match:
                count += 1

        return count
```

The implementation begins by constructing the `relations` array. This preprocessing step converts adjacent numerical comparisons into symbolic values `1`, `0`, and `-1`.

Next, the algorithm slides a window of size `len(pattern)` across the `relations` array. For every starting position, it compares the window element by element against `pattern`.

If every value matches, the counter is incremented.

The implementation uses clear variable names and early termination with `break` when a mismatch is found, which avoids unnecessary comparisons.

## Go Solution

```go
func countMatchingSubarrays(nums []int, pattern []int) int {
    relations := make([]int, 0, len(nums)-1)

    for i := 0; i < len(nums)-1; i++ {
        if nums[i+1] > nums[i] {
            relations = append(relations, 1)
        } else if nums[i+1] == nums[i] {
            relations = append(relations, 0)
        } else {
            relations = append(relations, -1)
        }
    }

    patternLength := len(pattern)
    count := 0

    for start := 0; start <= len(relations)-patternLength; start++ {
        match := true

        for j := 0; j < patternLength; j++ {
            if relations[start+j] != pattern[j] {
                match = false
                break
            }
        }

        if match {
            count++
        }
    }

    return count
}
```

The Go implementation follows the same logic as the Python version.

A slice named `relations` is created to store adjacent comparisons. Go slices are dynamic, so `append` is used while building the array.

The nested loops perform the window matching process. Since all values fit comfortably inside standard integer ranges, there are no overflow concerns.

Go does not distinguish between lists and arrays in the same way as Python, so slices are the natural structure for dynamic storage.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n + (n - m) * m) | Building the relations array takes O(n), matching each window takes O(m) |
| Space | O(n) | The relations array stores n - 1 elements |

The preprocessing step scans the input once, producing the `relations` array in linear time.

The matching phase checks every possible window against the pattern. Since there are `n - m` windows and each comparison may examine up to `m` elements, the matching cost is `O((n - m) * m)`.

Because the constraints are small, this implementation performs comfortably within limits.

## Edge Cases

One important edge case occurs when all numbers are equal. In this scenario, every adjacent comparison produces `0`. A buggy implementation might accidentally treat equality as increasing or decreasing if the comparison logic is not written carefully. This implementation explicitly checks equality separately, ensuring correct behavior.

Another important case is overlapping matches. For example, in `[1,2,1,2,1]` with pattern `[1,-1]`, multiple valid subarrays share elements. Some incorrect solutions accidentally skip indices after finding a match. This implementation checks every possible starting position independently, so overlapping matches are counted correctly.

A third important edge case involves the smallest valid input sizes. When `nums` has length `2` and `pattern` has length `1`, there is exactly one possible subarray to test. Off by one mistakes are common in such situations. The loop boundaries in this solution are carefully constructed so that all valid windows are checked exactly once.

Another subtle edge case occurs when no matches exist at all. The algorithm must correctly return `0` without errors or incorrect increments. Because the implementation only increments the counter after a full successful comparison, non-matching windows are safely ignored.
