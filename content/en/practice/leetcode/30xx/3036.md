## Solution

## Problem Understanding

The problem gives us two arrays:

- `nums`, an integer array of length `n`
- `pattern`, an array of length `m` containing only `-1`, `0`, and `1`

We must count how many subarrays of `nums` with length `m + 1` satisfy the relationship sequence described by `pattern`.

The important detail is that `pattern` does not describe the actual values inside the subarray. Instead, it describes the relationship between adjacent elements.

For every index `k` in `pattern`:

- `pattern[k] == 1` means the next number must be larger
- `pattern[k] == 0` means the next number must be equal
- `pattern[k] == -1` means the next number must be smaller

Suppose we take a subarray starting at index `i`. Then:

- `pattern[0]` compares `nums[i]` and `nums[i+1]`
- `pattern[1]` compares `nums[i+1]` and `nums[i+2]`
- and so on

So a subarray of size `m + 1` generates exactly `m` comparisons, which must match the entire pattern.

For example:

```
nums = [1,4,4,1]
pattern = [1,0,-1]
```

The comparisons are:

```
1 -> 4  : increasing  => 1
4 -> 4  : equal       => 0
4 -> 1  : decreasing  => -1
```

This exactly matches the pattern.

The constraints are extremely important:

- `n` can be as large as `10^6`
- `pattern.length` can also be close to `10^6`

This immediately tells us that quadratic solutions are impossible. Even an `O(n * m)` approach would time out badly at this scale.

The problem guarantees:

- `m < n`
- `pattern` contains only `-1`, `0`, and `1`

Important edge cases include:

- All elements equal
- Strictly increasing or strictly decreasing arrays
- Pattern length very close to array length
- Repeated values causing many `0` comparisons
- Very large inputs where inefficient substring checking becomes too expensive

The key challenge is to efficiently compare the comparison pattern across the entire array.

## Approaches

### Brute Force Approach

The most direct solution is to check every possible subarray of length `m + 1`.

For each starting position `i`:

1. Compare every adjacent pair inside the subarray
2. Generate the relationship:

- `1` if increasing
- `0` if equal
- `-1` if decreasing
3. Compare the generated sequence with `pattern`
4. If all positions match, increment the answer

This approach is correct because it directly implements the problem definition.

However, the complexity is too large.

There are `O(n)` possible starting positions, and each requires checking `O(m)` comparisons.

That gives:

```
O(n * m)
```

With both values potentially near `10^6`, this becomes completely infeasible.

### Optimal Approach

The key observation is that the problem is really a pattern matching problem.

Instead of repeatedly comparing subarrays, we can transform `nums` into a comparison array.

Define:

```
diff[i] =
    1  if nums[i+1] > nums[i]
    0  if nums[i+1] == nums[i]
   -1  if nums[i+1] < nums[i]
```

Now the task becomes:

> Count how many times `pattern` appears inside `diff`.

This is a classic string matching problem.

For example:

```
nums    = [1,4,4,1,3]
diff    = [1,0,-1,1]
pattern = [1,0,-1]
```

Now we simply search for occurrences of `pattern` inside `diff`.

To do this efficiently, we can use the Knuth-Morris-Pratt algorithm, commonly called KMP.

KMP allows pattern matching in linear time by avoiding unnecessary re-comparisons after mismatches.

The total complexity becomes:

```
O(n + m)
```

which is efficient enough for the constraints.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * m) | O(1) | Checks every subarray independently |
| Optimal | O(n + m) | O(m) | Converts to pattern matching using KMP |

## Algorithm Walkthrough

### Step 1: Build the comparison array

We transform `nums` into an array describing adjacent relationships.

For every adjacent pair:

- append `1` if increasing
- append `0` if equal
- append `-1` if decreasing

Example:

```
nums = [1,4,4,1,3]

diff:
1 -> 4 : 1
4 -> 4 : 0
4 -> 1 : -1
1 -> 3 : 1

diff = [1,0,-1,1]
```

This transformation is important because matching subarrays now becomes equivalent to matching a sequence inside another sequence.

### Step 2: Build the KMP prefix table

KMP preprocessing computes the longest proper prefix that is also a suffix for every prefix of `pattern`.

This table is usually called:

- `lps`
- prefix function
- failure function

The purpose is to efficiently continue matching after a mismatch.

Suppose:

```
pattern = [1,0,1,0]
```

If we mismatch after partially matching `[1,0,1]`, we should not restart from scratch because some prefix may already match.

The `lps` table tells us exactly where to continue.

### Step 3: Run KMP matching on the comparison array

We scan through `diff` while maintaining:

- `i` = current index in `diff`
- `j` = current matched length in `pattern`

If characters match:

```
diff[i] == pattern[j]
```

we advance both pointers.

When `j` reaches `m`, we found a full match.

We increment the answer and continue searching using the prefix table.

If a mismatch occurs:

- if `j > 0`, jump using `lps[j - 1]`
- otherwise move `i` forward

This guarantees every element is processed only a constant number of times.

### Why it works

The transformation step is correct because every subarray of length `m + 1` uniquely corresponds to exactly `m` adjacent comparisons.

Therefore, a subarray matches the pattern if and only if the generated comparison sequence equals `pattern`.

KMP correctly counts all occurrences of a pattern inside a sequence in linear time. Since `diff` represents all adjacent comparisons in `nums`, every valid subarray corresponds exactly to one occurrence of `pattern` in `diff`.

Thus the algorithm counts precisely all matching subarrays.

## Python Solution

```python
from typing import List

class Solution:
    def countMatchingSubarrays(self, nums: List[int], pattern: List[int]) -> int:
        # Build comparison array
        diff = []

        for i in range(len(nums) - 1):
            if nums[i + 1] > nums[i]:
                diff.append(1)
            elif nums[i + 1] == nums[i]:
                diff.append(0)
            else:
                diff.append(-1)

        m = len(pattern)

        # Build LPS array for KMP
        lps = [0] * m
        length = 0
        i = 1

        while i < m:
            if pattern[i] == pattern[length]:
                length += 1
                lps[i] = length
                i += 1
            else:
                if length != 0:
                    length = lps[length - 1]
                else:
                    lps[i] = 0
                    i += 1

        # KMP pattern matching
        answer = 0
        i = 0  # index for diff
        j = 0  # index for pattern

        while i < len(diff):
            if diff[i] == pattern[j]:
                i += 1
                j += 1

                if j == m:
                    answer += 1
                    j = lps[j - 1]
            else:
                if j != 0:
                    j = lps[j - 1]
                else:
                    i += 1

        return answer
```

The implementation begins by converting the original array into the comparison array `diff`. This removes the need to repeatedly compare adjacent elements for every candidate subarray.

Next, the code constructs the KMP prefix table `lps`. Each entry stores the length of the longest proper prefix that is also a suffix for the current prefix of the pattern. This preprocessing enables efficient mismatch recovery during matching.

Finally, the algorithm performs KMP matching between `diff` and `pattern`. Whenever a complete match is found, the answer is incremented. Instead of restarting matching from scratch, the algorithm uses the prefix table to continue efficiently.

Because both preprocessing and matching are linear, the overall solution handles arrays up to size `10^6` comfortably.

## Go Solution

```go
func countMatchingSubarrays(nums []int, pattern []int) int {
	// Build comparison array
	diff := make([]int, 0, len(nums)-1)

	for i := 0; i < len(nums)-1; i++ {
		if nums[i+1] > nums[i] {
			diff = append(diff, 1)
		} else if nums[i+1] == nums[i] {
			diff = append(diff, 0)
		} else {
			diff = append(diff, -1)
		}
	}

	m := len(pattern)

	// Build LPS array
	lps := make([]int, m)

	length := 0
	i := 1

	for i < m {
		if pattern[i] == pattern[length] {
			length++
			lps[i] = length
			i++
		} else {
			if length != 0 {
				length = lps[length-1]
			} else {
				lps[i] = 0
				i++
			}
		}
	}

	// KMP matching
	answer := 0
	i = 0
	j := 0

	for i < len(diff) {
		if diff[i] == pattern[j] {
			i++
			j++

			if j == m {
				answer++
				j = lps[j-1]
			}
		} else {
			if j != 0 {
				j = lps[j-1]
			} else {
				i++
			}
		}
	}

	return answer
}
```

The Go implementation closely mirrors the Python version. Slices are used for both the comparison array and the prefix table.

Go does not require special handling for integer overflow here because all values are very small and the answer fits within standard integer limits under the problem constraints.

The implementation carefully reuses indices during KMP matching to maintain linear complexity.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n + m) | Building `diff`, constructing `lps`, and KMP matching are all linear |
| Space | O(n + m) | `diff` uses O(n), `lps` uses O(m) |

The comparison array contains `n - 1` elements, so constructing it requires linear time and space. The KMP prefix table requires linear preprocessing relative to the pattern size.

During matching, each index moves forward without backtracking excessively, which guarantees total linear runtime.

## Edge Cases

One important edge case occurs when all numbers are equal. In this situation, every comparison becomes `0`. A buggy implementation might incorrectly treat equality as either increasing or decreasing. The solution handles this explicitly by generating `0` whenever adjacent values are equal.

Another important edge case is when the pattern length is `1`. In this case, every valid subarray has length `2`, and the algorithm effectively becomes a scan of adjacent pairs. The KMP implementation still works correctly because single-element patterns are fully supported.

Overlapping matches are another subtle case. For example:

```
diff = [1,-1,1,-1,1]
pattern = [1,-1,1]
```

There are overlapping valid matches. A naive implementation that restarts after each match might miss some occurrences. KMP avoids this issue by using the prefix table to continue matching efficiently after each successful match.

A final edge case involves extremely large inputs near `10^6`. Any solution that repeatedly constructs subarrays or rescans previous elements would time out. This implementation processes every element only a constant number of times, ensuring it remains efficient even at maximum scale.
