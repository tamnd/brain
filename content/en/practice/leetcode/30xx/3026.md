# LeetCode 3026 - Maximum Good Subarray Sum

## Problem Understanding

We are given an integer array `nums` and a positive integer `k`.

A subarray `nums[i..j]` is considered **good** if the absolute difference between its first and last element is exactly `k`:

$$|nums[i] - nums[j]| = k$$

Among all such good subarrays, we must find the one with the largest sum and return that sum.

The important detail is that the condition depends only on the **first** and **last** element of the subarray. The values in the middle do not affect whether the subarray is good. However, they do contribute to the subarray's total sum, which is what we want to maximize.

The array length can be as large as:

$$n \le 10^5$$

A quadratic solution that checks all subarrays would require examining roughly $10^{10}$ pairs in the worst case, which is far too slow. Therefore, we need a solution that runs in approximately linear time.

The values of `nums[i]` can be both positive and negative, ranging from $-10^9$ to $10^9$. This means several important things:

- The maximum good subarray sum may be negative.
- We cannot assume that extending a subarray always improves its sum.
- We need 64-bit integer arithmetic because cumulative sums can exceed 32-bit limits.

Some important edge cases include arrays containing only negative numbers, situations where no good subarray exists, repeated values, and cases where many possible starting positions can produce the same ending value. A correct solution must handle all of these efficiently.

## Approaches

### Brute Force

The most direct approach is to examine every possible subarray.

For every pair of indices `(i, j)` with `i ≤ j`, we check whether:

$$|nums[i] - nums[j]| = k$$

If the condition holds, we compute the sum of the subarray and update the answer.

Using prefix sums, each subarray sum can be computed in constant time, but we still need to inspect all possible pairs.

There are:

$$O(n^2)$$

possible subarrays, so the overall complexity remains quadratic.

This approach is correct because it explicitly checks every candidate subarray, but it is too slow for $n = 10^5$.

### Key Insight

Suppose we are currently processing index `j`.

If `nums[j] = x`, then any good subarray ending at `j` must start at some earlier index `i` where:

$$nums[i] = x-k$$

or

$$nums[i] = x+k$$

because the absolute difference between the first and last elements must equal `k`.

Now consider the sum of a subarray from `i` to `j`.

Let `prefix[t]` denote the sum of the first `t` elements.

Then:

$$\text{subarraySum}(i,j)
=
\text{prefix}[j+1] - \text{prefix}[i]$$

For a fixed ending position `j`, maximizing the subarray sum means minimizing:

$$\text{prefix}[i]$$

among all valid starting positions.

Therefore, for every value `v`, we only need to remember the **smallest prefix sum that appeared before an occurrence of `v`**.

When we reach position `j`, we can immediately determine the best possible starting position whose value is `nums[j]-k` or `nums[j]+k`.

This transforms the problem into a linear scan with a hash map.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Checks every subarray using prefix sums |
| Optimal | O(n) | O(n) | Hash map stores minimum prefix sum for each value |

## Algorithm Walkthrough

### Optimal Algorithm

1. Initialize a running prefix sum `prefix = 0`.
2. Create a hash map `min_prefix` where `min_prefix[value]` stores the smallest prefix sum observed before any occurrence of `value`.
3. Initialize `answer` as negative infinity and maintain a flag indicating whether any good subarray has been found.
4. Iterate through the array from left to right.
5. At the start of processing `nums[i]`, the current `prefix` represents the sum of all elements before index `i`.
6. Let the current value be `x = nums[i]`.
7. Any good subarray ending at index `i` must begin at a position whose value is either:

$$x-k$$

or

$$x+k$$
8. For each of these two candidate values:

- Check whether it exists in `min_prefix`.
- If it does, compute the subarray sum:

$$(\text{prefix} + x) - \text{min\_prefix}[candidate]$$

because `prefix + x` equals the prefix sum after including the current element.
- Update the answer if this sum is larger.
9. Before advancing the prefix sum, record the current value `x` in the map.

The prefix sum before index `i` is exactly the quantity needed if some future subarray starts at this index.

Therefore:

- If `x` is not in the map, store the current prefix.
- If `x` already exists, keep the smaller of the existing value and the current prefix.
10. Add `x` to the running prefix sum.
11. After processing all elements, return `0` if no good subarray was found. Otherwise return the maximum sum discovered.

### Why it works

For every value `v`, the hash map stores the minimum possible prefix sum before an occurrence of `v`. When processing an ending position `j`, every valid starting position must have value `nums[j]-k` or `nums[j]+k`. Among all such starting positions, the maximum subarray sum is obtained by choosing the one with the smallest prefix sum before it. Since the map always stores exactly this minimum prefix sum, every good subarray ending at `j` is considered optimally. Therefore the algorithm examines the best candidate for every possible ending position and returns the global maximum.

## Python Solution

```python
from typing import List
from math import inf

class Solution:
    def maximumSubarraySum(self, nums: List[int], k: int) -> int:
        min_prefix = {}
        
        prefix = 0
        answer = -inf
        found = False
        
        for x in nums:
            current_prefix = prefix + x
            
            for target in (x - k, x + k):
                if target in min_prefix:
                    answer = max(
                        answer,
                        current_prefix - min_prefix[target]
                    )
                    found = True
            
            if x in min_prefix:
                min_prefix[x] = min(min_prefix[x], prefix)
            else:
                min_prefix[x] = prefix
            
            prefix = current_prefix
        
        return 0 if not found else int(answer)
```

The implementation follows the algorithm exactly.

The variable `prefix` stores the sum of all elements processed so far. Before updating it with the current element, its value represents the prefix sum immediately before the current index, which is exactly what a future subarray starting at this position requires.

The dictionary `min_prefix` maps each array value to the smallest prefix sum seen before any occurrence of that value. This allows us to instantly retrieve the best possible starting position for a future subarray.

For every element `x`, we check both `x - k` and `x + k`, since either value could satisfy the absolute difference requirement. If either exists in the map, we compute the best subarray sum ending at the current position.

Finally, if no valid subarray was ever found, we return `0` as required by the problem statement.

## Go Solution

```go
func maximumSubarraySum(nums []int, k int) int64 {
	minPrefix := make(map[int]int64)

	var prefix int64 = 0
	var answer int64
	found := false

	for _, num := range nums {
		x := int64(num)
		currentPrefix := prefix + x

		candidates := []int{num - k, num + k}

		for _, target := range candidates {
			if minVal, ok := minPrefix[target]; ok {
				sum := currentPrefix - minVal

				if !found || sum > answer {
					answer = sum
				}
				found = true
			}
		}

		if old, ok := minPrefix[num]; ok {
			if prefix < old {
				minPrefix[num] = prefix
			}
		} else {
			minPrefix[num] = prefix
		}

		prefix = currentPrefix
	}

	if !found {
		return 0
	}

	return answer
}
```

The Go implementation mirrors the Python solution. The main difference is the explicit use of `int64` for all prefix sums and answer calculations. This is necessary because the sum of up to `10^5` values, each potentially as large as `10^9`, can exceed the range of a 32-bit integer.

Go maps return a value and a boolean indicating whether the key exists, which is used when checking candidate starting values and when maintaining the minimum prefix sum for each number.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each element is processed once and performs only constant-time hash map operations |
| Space | O(n) | In the worst case every value is distinct and stored in the hash map |

The algorithm performs a single pass through the array. For each element, only two hash map lookups and one update are required. Since hash map operations are expected O(1), the total running time is O(n). The map may contain one entry for every distinct value, resulting in O(n) auxiliary space.

## Edge Cases

### No Good Subarray Exists

An array may contain no pair of endpoints whose values differ by exactly `k`. In that situation, the problem requires returning `0`, not negative infinity and not some default value. The implementation tracks whether any valid subarray has been found using the `found` flag and returns `0` only when no valid candidate ever appears.

### All Valid Subarrays Have Negative Sums

Because array values can be negative, the best valid subarray sum may also be negative. A common mistake is initializing the answer to `0`, which would incorrectly reject valid negative results. The implementation keeps the true maximum among all valid subarrays and only returns `0` when no valid subarray exists at all.

### Multiple Occurrences of the Same Value

A value may appear many times in the array. Different occurrences correspond to different starting positions and therefore different prefix sums. To maximize a future subarray sum, we only care about the smallest prefix sum associated with that value. The hash map stores exactly this minimum, ensuring that every future ending position automatically uses the most profitable starting point.

### Large Integer Sums

The sum of a subarray can reach approximately:

$$10^5 \times 10^9 = 10^{14}$$

which exceeds 32-bit integer limits. The Go solution therefore uses `int64` throughout, and Python naturally handles arbitrarily large integers. This guarantees correct behavior for the largest inputs.
