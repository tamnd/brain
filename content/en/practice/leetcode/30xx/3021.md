# LeetCode 3021 - Alice and Bob Playing Flower Game

## Problem Understanding

The problem describes a turn-based game involving two lanes of flowers. One lane contains `x` flowers and the other contains `y` flowers. Alice moves first, and during each turn, the current player removes exactly one flower from either lane.

The game continues until all flowers from both lanes have been removed. The important rule is that the player who removes the final flower wins immediately.

We are given two integers, `n` and `m`. The value `x` can be any integer in the range `[1, n]`, and the value `y` can be any integer in the range `[1, m]`.

Our task is to count how many pairs `(x, y)` allow Alice, the first player, to win the game assuming optimal play.

The key observation is that the players do not actually have any strategic branching decisions that affect the game length. Every turn removes exactly one flower total, regardless of which lane is chosen. Therefore, the total number of turns in the game is always exactly:

$x+y$

If the total number of flowers is odd, then Alice makes the final move because she moves first. If the total number is even, then Bob makes the final move.

So the entire problem reduces to counting how many pairs `(x, y)` satisfy:

$x+y\text{ is odd}$

The constraints are up to `10^5`, which means an `O(n*m)` brute-force solution could become too slow because it may require up to `10^10` iterations in the worst case.

An important edge case is when both dimensions are very small, such as `n = 1` and `m = 1`. Another important case is when one dimension is even and the other is odd, because the parity distribution affects the final count.

## Approaches

### Brute Force Approach

The brute-force method checks every possible pair `(x, y)` where:

- `1 <= x <= n`
- `1 <= y <= m`

For each pair, we compute whether `x + y` is odd. If it is odd, Alice wins and we increment the answer.

This approach is correct because the winner depends entirely on the parity of the total number of flowers.

However, the time complexity is too large for the constraints. If both `n` and `m` are `10^5`, then we would need to examine `10^10` pairs.

### Optimal Approach

The key insight is that a sum is odd only when one number is odd and the other is even.

That means valid pairs come from two categories:

1. `x` is odd and `y` is even
2. `x` is even and `y` is odd

So instead of iterating through every pair, we only need to count:

- Number of odd integers in `[1, n]`
- Number of even integers in `[1, n]`
- Number of odd integers in `[1, m]`
- Number of even integers in `[1, m]`

Then the answer becomes:

$(\text{odd}_n \times \text{even}_m)+(\text{even}_n \times \text{odd}_m)$

This gives an `O(1)` solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × m) | O(1) | Checks every possible pair |
| Optimal | O(1) | O(1) | Uses parity counting |

## Algorithm Walkthrough

1. Count how many odd numbers exist from `1` to `n`.

The count of odd integers from `1` to `n` is:

$\left\lfloor\frac{n+1}{2}\right\rfloor$

This works because odd and even numbers alternate.
2. Count how many even numbers exist from `1` to `n`.

This is simply:

$n-\text{odd}_n$
3. Repeat the same calculations for `m`.
4. Compute the number of winning pairs.

Alice wins when the total number of flowers is odd. This happens in exactly two situations:

- odd `x` with even `y`
- even `x` with odd `y`

So we compute:

$(\text{odd}_n \times \text{even}_m)+(\text{even}_n \times \text{odd}_m)$
5. Return the computed result.

### Why it works

Each move removes exactly one flower, so the total number of turns equals `x + y`. Since Alice always moves first, she wins exactly when the total number of turns is odd. An odd total occurs only when one number is odd and the other is even. Therefore, counting those parity combinations gives the exact number of winning pairs.

## Python Solution

```python
class Solution:
    def flowerGame(self, n: int, m: int) -> int:
        odd_n = (n + 1) // 2
        even_n = n // 2

        odd_m = (m + 1) // 2
        even_m = m // 2

        return odd_n * even_m + even_n * odd_m
```

The implementation directly follows the mathematical observation developed earlier.

First, we compute the number of odd and even integers in both ranges. Integer division makes this calculation very compact.

For example:

- `n // 2` gives the count of even numbers
- `(n + 1) // 2` gives the count of odd numbers

After obtaining the parity counts, we compute the number of valid pairs using the two parity combinations that produce an odd sum.

The final expression returns the total number of winning configurations for Alice.

## Go Solution

```go
func flowerGame(n int, m int) int64 {
    oddN := (n + 1) / 2
    evenN := n / 2

    oddM := (m + 1) / 2
    evenM := m / 2

    result := oddN*evenM + evenN*oddM

    return int64(result)
}
```

The Go implementation is almost identical to the Python version. The main difference is the explicit conversion to `int64` before returning, because the function signature requires `int64`.

Go integer division behaves exactly as needed here because division between integers truncates toward zero.

## Edge Cases

One important edge case occurs when both `n` and `m` equal `1`. In this situation, the only pair is `(1,1)`, and the total number of flowers is `2`, which is even. Bob wins, so the answer must be `0`. Implementations that incorrectly assume the first player always wins when there is at least one move would fail here.

Another important edge case happens when one range contains only odd numbers or only one valid value. For example, when `n = 1`, every pair depends entirely on the parity of `y`. The implementation handles this correctly because the odd and even counts are computed mathematically rather than through iteration.

A third edge case involves the largest constraints, such as `n = 100000` and `m = 100000`. A brute-force solution would require checking ten billion pairs, which is computationally infeasible. The optimized solution avoids iteration entirely and computes the answer in constant time using parity counting.
