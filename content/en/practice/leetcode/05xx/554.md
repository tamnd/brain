## Solution

## Problem Understanding

The problem gives us a wall represented as a list of rows, where each row contains integers representing brick widths. Every brick has height `1`, but widths may differ. All rows together form a rectangular wall, meaning the total width of every row is identical.

We want to draw a vertical line from the top of the wall to the bottom such that the line crosses the fewest bricks possible.

A critical detail is that if the vertical line passes exactly along the boundary between two bricks, then it does not count as crossing a brick. This means we are really trying to align our vertical line with as many brick edges as possible across different rows.

However, there is one restriction: we cannot draw the line along the far left or far right edge of the wall. Those positions would trivially cross zero bricks, so they are excluded.

The input `wall` is a 2D array where:

- `wall[i]` represents the `i-th` row
- `wall[i][j]` is the width of the `j-th` brick in that row

The output is the minimum number of bricks crossed by the optimal vertical line.

The constraints are important:

- There can be up to `10^4` rows
- The total number of bricks across all rows is at most `2 * 10^4`

This tells us that a solution roughly linear in the total number of bricks is expected. A quadratic solution would likely be too slow.

Several edge cases are important:

- Rows containing only one brick, there are no internal edges available
- Walls where no edges align between rows
- Walls where many rows share the same internal edge position
- Very large brick widths, which means we should rely on prefix sums rather than expanding the wall into unit cells

The problem guarantees that all rows have the same total width, so we never need to validate wall consistency ourselves.

## Approaches

### Brute Force Approach

A naive solution would consider every possible vertical position where a line could be drawn. For each position, we would scan every row and determine whether the line cuts through a brick or lands on a brick edge.

To do this, we could compute prefix sums for each row and check whether the current position matches an edge. If it does not, then the line crosses a brick in that row.

This approach is correct because it explicitly evaluates every candidate line position and computes exactly how many bricks are crossed.

However, it is inefficient because there may be many possible positions to test. If we repeatedly scan all rows for every position, the work becomes unnecessarily repetitive.

The key inefficiency is that we only care about positions where brick edges occur. Recomputing crossings independently for every position wastes time.

### Optimal Approach

The key insight is that crossing the fewest bricks is equivalent to passing through the most brick edges.

Suppose there are `n` rows:

- If a vertical line passes through an edge in `k` rows
- Then it crosses bricks in the remaining `n - k` rows

So instead of minimizing crossings directly, we maximize the number of aligned brick edges.

We compute prefix sums for each row to determine all internal brick edge positions. Every time an edge occurs at some position `x`, we increment a counter for `x`.

The position with the highest frequency is the best place to draw the vertical line.

We exclude the final edge of each row because that represents the right boundary of the wall, which the problem forbids.

This transforms the problem into a frequency counting problem, which can be solved efficiently with a hash map.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(W × B) | O(B) | Tests every possible vertical position against all bricks |
| Optimal | O(B) | O(B) | Counts shared edge positions using a hash map |

Here:

- `W` is the wall width
- `B` is the total number of bricks across all rows

## Algorithm Walkthrough

1. Initialize a hash map called `edge_count`.

The keys will represent edge positions within the wall, and the values will store how many rows contain an edge at that position.
2. Iterate through every row in the wall.

We process rows independently because brick edges are defined row by row.
3. For each row, compute running prefix sums.

The prefix sum represents the position of a brick edge measured from the left side of the wall.

For example, for `[1,2,2,1]`:

- After first brick: edge at `1`
- After second brick: edge at `3`
- After third brick: edge at `5`
4. Do not include the final edge of the row.

The final edge equals the full wall width, which corresponds to the outer boundary. The problem explicitly forbids drawing the line there.
5. Update the hash map frequency for every internal edge.

If multiple rows share the same edge position, that position becomes increasingly attractive because drawing a line there avoids crossing bricks in all those rows.
6. Track the maximum edge frequency encountered.

This tells us the largest number of rows where a line can pass through edges instead of bricks.
7. Return:

```
number_of_rows - maximum_edge_frequency
```

This works because every row not contributing an edge at the chosen position must have a brick crossed.

### Why it works

Every vertical line either:

- Passes through a brick edge in a row, crossing zero bricks there
- Or passes through the interior of a brick, crossing one brick there

Therefore, minimizing crossed bricks is exactly equivalent to maximizing aligned edges across rows.

The hash map correctly counts how many rows share each edge position, and selecting the maximum frequency guarantees the minimum number of crossed bricks.

## Python Solution

```
from typing import List
from collections import defaultdict

class Solution:
    def leastBricks(self, wall: List[List[int]]) -> int:
        edge_count = defaultdict(int)
        max_edges = 0

        for row in wall:
            position = 0

            # Exclude the last brick to avoid using the wall boundary
            for brick_width in row[:-1]:
                position += brick_width
                edge_count[position] += 1

                max_edges = max(max_edges, edge_count[position])

        return len(wall) - max_edges
```

The implementation starts by creating a hash map called `edge_count`. This stores how many rows contain a brick edge at a particular position.

For each row, we maintain a running prefix sum called `position`. As we move through the bricks, the prefix sum identifies where the current brick ends.

We iterate only through `row[:-1]`, excluding the last brick. This ensures we never count the wall's outer boundary as a valid edge.

Each internal edge position is added to the hash map, and we continuously update `max_edges`, which tracks the most commonly shared edge.

Finally, the answer is computed as:

```
total_rows - maximum_shared_edges
```

because every row not sharing that edge must have a brick crossed.

## Go Solution

```
func leastBricks(wall [][]int) int {
    edgeCount := make(map[int]int)
    maxEdges := 0

    for _, row := range wall {
        position := 0

        // Exclude the last brick
        for i := 0; i < len(row)-1; i++ {
            position += row[i]
            edgeCount[position]++

            if edgeCount[position] > maxEdges {
                maxEdges = edgeCount[position]
            }
        }
    }

    return len(wall) - maxEdges
}
```

The Go implementation follows the same logic as the Python version.

A `map[int]int` is used instead of Python's `defaultdict`. Since Go maps return the zero value for missing keys, incrementing counts works naturally.

Slices are used for iterating rows, and we explicitly loop only until `len(row)-1` to exclude the final boundary edge.

Integer overflow is not an issue because Go's `int` type comfortably handles the problem constraints on modern platforms.

wall = [
    [1,2,2,1],
    [3,1,2],
    [1,3,2],

We compute internal edge positions row by row.

| [3,1,2] | 3, 4 |
| [1,3,2] | 1, 4 |
| [2,4] | 2 |
| [3,1,2] | 3, 4 |
| [1,3,1,1] | 1, 4, 5 |

Now count frequencies:

| Edge Position | Frequency |
| --- | --- |
| 1 | 3 |
| 2 | 1 |
| 3 | 3 |
| 4 | 4 |
| 5 | 2 |

The best edge position is `4`, appearing in `4` rows.

Total rows = `6`

Result:

```
6 - 4 = 2
```

So the minimum crossed bricks is `2`.

### Example 2

Input:

```
wall = [[1],[1],[1]]
```

Each row contains only one brick.

There are no internal edges because the only edge is the wall boundary, which is forbidden.

So the hash map remains empty.

| Edge Position | Frequency |
| --- | --- |
| none | 0 |

Maximum shared edges = `0`

Total rows = `3`

Result:

```
3 - 0 = 3
```

The vertical line must cross every row.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(B) | Each brick is processed once |
| Space | O(B) | Hash map may store every unique edge position |

Here, `B` is the total number of bricks across all rows.

The algorithm is efficient because each brick contributes to at most one prefix sum operation and one hash map update. Since the total number of bricks is bounded by `2 * 10^4`, the solution easily fits within the problem constraints.

The space complexity comes from storing edge frequencies. In the worst case, every edge position is unique.

## Edge Cases

One important edge case occurs when every row contains exactly one brick. In this situation, there are no internal brick boundaries at all. A naive implementation might accidentally count the wall boundary as a valid edge and incorrectly return `0`. The solution avoids this by explicitly excluding the last brick in every row when computing prefix sums.

Another important case is when all rows share the same internal edge positions. In that scenario, the optimal line can pass through edges in every row, producing an answer of `0`. Some implementations incorrectly assume at least one brick must always be crossed. This algorithm correctly handles the case because the maximum edge frequency can equal the total number of rows.

A more subtle edge case involves very large brick widths. Since widths may be as large as `2^31 - 1`, expanding the wall into unit cells would be infeasible in both time and memory. The solution avoids this entirely by working only with cumulative edge positions using prefix sums, making the actual brick sizes irrelevant to performance.

Another potential source of bugs is accidentally including the rightmost boundary edge. Every row ends at the same total width, so including that edge would falsely suggest a perfect alignment and incorrectly produce `0` for many inputs. By iterating through `row[:-1]` in Python and `len(row)-1` in Go, the implementation safely excludes the forbidden boundary.
