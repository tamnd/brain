## Solution

## Problem Understanding

This problem asks us to compute the diameter of a binary tree. The diameter is defined as the length of the longest path between any two nodes in the tree, measured in the number of edges.

A binary tree consists of nodes where each node may have a left child and a right child. The input is given as the root node of the tree. We need to examine all possible paths between nodes and determine which path contains the greatest number of edges.

An important detail is that the longest path does not necessarily pass through the root. The diameter could exist entirely inside the left subtree or entirely inside the right subtree.

For example, consider this tree:

```
        1
       / \
      2   3
     / \
    4   5
```

The longest path is:

```
4 -> 2 -> 1 -> 3
```

This path contains 3 edges, so the diameter is 3.

The constraints tell us that the tree can contain up to `10^4` nodes. This means we need an efficient solution. An algorithm with quadratic time complexity could become too slow for large trees.

The problem guarantees that at least one node exists. Node values themselves are irrelevant because the diameter depends only on the tree structure.

Several edge cases are important:

- A tree with only one node has diameter `0` because there are no edges.
- A tree with two nodes has diameter `1`.
- A completely skewed tree behaves like a linked list, which can expose inefficient recursive solutions.
- The diameter may exist entirely within one subtree and not involve the root at all.

Understanding these cases helps avoid incorrect assumptions during implementation.

## Approaches

### Brute Force Approach

A straightforward approach is to compute the diameter through every node individually.

For each node, we can:

1. Compute the height of its left subtree.
2. Compute the height of its right subtree.
3. Treat `leftHeight + rightHeight` as the diameter passing through that node.
4. Recursively compute the diameter of the left subtree.
5. Recursively compute the diameter of the right subtree.
6. Return the maximum of all these values.

This works because every possible longest path must pass through some node. By evaluating each node as a potential center of the longest path, we eventually discover the true diameter.

However, the issue is efficiency. Computing height requires traversing subtrees repeatedly. For every node, we may recompute heights for many descendants multiple times.

In the worst case, such as a skewed tree, height computation costs `O(N)` and we do it for every node, producing `O(N^2)` time complexity.

With up to `10^4` nodes, this becomes unnecessarily expensive.

### Optimal Approach

The key observation is that while computing the height of a node, we already have enough information to compute the diameter passing through that node.

For any node:

```
diameter_through_node = left_height + right_height
```

At the same time:

```
height_of_node = 1 + max(left_height, right_height)
```

This means a single postorder traversal can compute both:

- Heights of subtrees
- Maximum diameter encountered so far

We use Depth-First Search recursively:

- First compute the heights of left and right children.
- Use them to update the global maximum diameter.
- Return the current node's height to the parent.

Since each node is visited exactly once, the algorithm runs in linear time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(N²) | O(H) | Recomputes subtree heights repeatedly |
| Optimal | O(N) | O(H) | Single DFS traversal computes height and diameter together |

Here, `N` is the number of nodes and `H` is the tree height.

## Algorithm Walkthrough

1. Create a variable called `diameter` to store the maximum diameter found so far. Initialize it to `0`.
2. Define a recursive helper function called `height(node)` that returns the height of the subtree rooted at `node`.
3. If the current node is `None`, return `0`. An empty subtree has height `0`.
4. Recursively compute the height of the left subtree:

```
left_height = height(node.left)
```
5. Recursively compute the height of the right subtree:

```
right_height = height(node.right)
```
6. The longest path passing through the current node uses:

- the deepest path in the left subtree
- the deepest path in the right subtree

Therefore:

```
left_height + right_height
```

is the diameter through this node.
7. Update the global maximum diameter if this value is larger than the current best answer.
8. Return the height of the current subtree:

```
1 + max(left_height, right_height)
```

The `1` accounts for the current node itself.
9. Start DFS from the root.
10. After traversal completes, return the stored maximum diameter.

### Why it works

The algorithm works because every longest path in a binary tree has some highest node where the path changes direction. At that node, the path consists of:

- the deepest branch in the left subtree
- the deepest branch in the right subtree

By computing subtree heights for every node, we automatically evaluate every possible diameter candidate. Since DFS visits every node exactly once and updates the maximum diameter whenever a longer path is found, the final answer is guaranteed to be correct.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import Optional

class Solution:
    def diameterOfBinaryTree(self, root: Optional[TreeNode]) -> int:
        diameter = 0

        def height(node: Optional[TreeNode]) -> int:
            nonlocal diameter

            if node is None:
                return 0

            left_height = height(node.left)
            right_height = height(node.right)

            diameter = max(diameter, left_height + right_height)

            return 1 + max(left_height, right_height)

        height(root)

        return diameter
```

The solution begins by initializing a variable named `diameter`. This variable stores the largest diameter encountered during traversal.

The nested `height` function performs the DFS traversal. It returns the height of the subtree rooted at the current node.

The base case handles `None` nodes by returning `0`. This simplifies height calculations because leaf nodes naturally receive child heights of `0`.

The recursive calls compute heights for the left and right subtrees before processing the current node. This is a postorder traversal because children are processed before the parent.

After obtaining both heights, the algorithm computes the diameter passing through the current node using:

```
left_height + right_height
```

If this value exceeds the current maximum diameter, the global answer is updated.

Finally, the function returns the current subtree height using:

```
1 + max(left_height, right_height)
```

The outer function triggers DFS from the root and returns the final diameter.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func diameterOfBinaryTree(root *TreeNode) int {
    diameter := 0

    var height func(node *TreeNode) int

    height = func(node *TreeNode) int {
        if node == nil {
            return 0
        }

        leftHeight := height(node.Left)
        rightHeight := height(node.Right)

        if leftHeight+rightHeight > diameter {
            diameter = leftHeight + rightHeight
        }

        if leftHeight > rightHeight {
            return 1 + leftHeight
        }

        return 1 + rightHeight
    }

    height(root)

    return diameter
}
```

The Go implementation follows the same recursive DFS strategy as the Python version.

Instead of Python's `nonlocal`, Go captures the `diameter` variable inside the closure naturally.

Go uses `nil` instead of `None` for empty nodes. Since tree heights and diameters remain well within integer limits for the given constraints, standard `int` values are sufficient.

The logic for updating the diameter and computing subtree heights remains identical.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(N) | Each node is visited exactly once |
| Space | O(H) | Recursive call stack depth equals tree height |

The algorithm performs one DFS traversal. Every node contributes a constant amount of work, so the total runtime is linear in the number of nodes.

The space complexity comes from recursion. In a balanced tree, the height is `O(log N)`. In the worst case, a completely skewed tree has height `O(N)`.

## Edge Cases

### Single Node Tree

A tree containing only one node has no edges, so its diameter must be `0`.

This case can cause bugs if the implementation mistakenly counts nodes instead of edges. Our solution avoids this problem because null children return height `0`, and the diameter calculation uses:

```
left_height + right_height
```

For a leaf node, both values are `0`, producing the correct result.

### Completely Skewed Tree

A skewed tree behaves like a linked list:

```
1 -> 2 -> 3 -> 4
```

The diameter equals the number of edges from top to bottom.

Naive solutions may repeatedly recompute subtree heights and become quadratic in runtime. Our DFS solution computes each height exactly once, maintaining linear complexity even for highly unbalanced trees.

### Diameter Does Not Pass Through Root

Consider this structure:

```
        1
       /
      2
     / \
    3   4
   /
  5
```

The longest path is:

```
5 -> 3 -> 2 -> 4
```

This path never reaches the root.

An incorrect implementation might only compute:

```
height(left subtree) + height(right subtree)
```

for the root node. Our solution evaluates this quantity for every node, ensuring that subtree diameters are considered properly.
