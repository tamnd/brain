## Solution

## Problem Understanding

In this problem, we are given an array called `boxes`, where each integer represents the color of a box. The boxes are arranged in a fixed order, and during each move we may remove a contiguous group of boxes that all share the same color.

If we remove a group containing `k` equal-colored boxes, we earn `k * k` points. After removing those boxes, the remaining boxes shift together automatically, which can create new adjacent groups that were previously separated.

The goal is to determine the maximum total score obtainable after removing all boxes.

The key challenge is that removing boxes earlier can affect future opportunities. Sometimes it is beneficial to remove a small group immediately, while other times it is better to delay removal so that separated boxes of the same color can later merge into a larger group worth significantly more points.

For example:

```
[1, 3, 2, 2, 2, 3, 4, 3, 1]
```

If we immediately remove the `3`s separately, we lose the opportunity to merge them later. Instead, removing the `2`s first allows multiple `3`s to become adjacent, producing a larger square score.

The constraints are important:

- `1 <= boxes.length <= 100`
- `1 <= boxes[i] <= 100`

A length of 100 is relatively small for cubic or quartic dynamic programming, but far too large for naive brute force recursion that explores every removal ordering. This strongly suggests that the intended solution uses memoization or dynamic programming.

Several edge cases are important:

- A single box, where the answer is always `1`
- All boxes having the same color, where removing everything at once gives `n²`
- Completely distinct boxes, where every removal gives only `1`
- Repeated colors separated by other colors, which is the central difficulty of the problem
- Long chains where optimal decisions require postponing removals

The most important observation is that the score function is quadratic. Combining same-colored boxes into larger groups is disproportionately valuable. For example:

- Removing two separate groups of size 1 gives `1 + 1 = 2`
- Removing one merged group of size 2 gives `4`

This means the algorithm must consider future merging opportunities.

## Approaches

### Brute Force Approach

A brute force solution tries every possible valid removal sequence.

At each step:

1. Find every contiguous block of equal-colored boxes.
2. Remove one possible block.
3. Recursively solve the remaining array.
4. Return the maximum score among all choices.

This approach is correct because it explores every possible sequence of removals. Eventually, one explored path must correspond to the optimal strategy.

However, the number of possible states grows explosively. Different removal orders produce different intermediate arrays, and many equivalent subproblems are recomputed repeatedly. The total number of recursive states becomes exponential.

With up to 100 boxes, brute force is completely infeasible.

### Key Insight

The central insight is that the value of removing a box depends on how many same-colored boxes can eventually be grouped with it.

This means a subproblem cannot be described only by a range `[left, right]`.

We also need to know:

- How many boxes of the same color as `boxes[right]` are already attached to the right side.

This leads to the classic DP definition:

```
dp(left, right, k)
```

where:

- `left` and `right` define the current interval
- `k` represents how many extra boxes equal to `boxes[right]` are attached to the right

The interpretation is:

```
Maximum score obtainable from boxes[left:right+1]
assuming there are already k extra boxes equal to boxes[right]
connected to the right side.
```

This extra parameter is the breakthrough that makes the problem solvable.

The algorithm works by considering two possibilities:

1. Remove `boxes[right]` immediately together with its attached group.
2. Delay removing `boxes[right]` so it can merge with another same-colored box earlier in the array.

The second case is what enables optimal solutions.

| --- | --- | --- | --- |

## Algorithm Walkthrough

### Step 1: Define the DP State

We define:

```
dp(left, right, k)
```

This means:

```
Maximum points obtainable from boxes[left:right+1]
if there are already k extra boxes equal to boxes[right]
attached to the right.
```

The parameter `k` is essential because future merges affect scoring.

### Step 2: Compress Consecutive Equal Boxes

Before solving the state, we compress consecutive equal boxes at the right boundary.

Suppose:

```
[2, 2, 2]
```

appears at the end of the interval.

Instead of handling them individually, we absorb them into `k`.

Example:

```
dp(left, right, k)
```

becomes:

```
dp(left, right-1, k+1)
```

while the colors match.

This reduces duplicate work and simplifies transitions.

### Step 3: Remove the Right Group Immediately

After compression, one option is:

- Remove the group containing `boxes[right]` now.

If the group size is `k + 1`, the score gained is:

```
(k + 1)^2
```

The remaining problem becomes:

```
dp(left, right-1, 0)
```

So:

```
result = dp(left, right-1, 0) + (k+1)^2
```

### Step 4: Try Merging with Earlier Equal Colors

Now comes the important optimization.

Suppose there exists some index `mid` such that:

```
boxes[mid] == boxes[right]
```

Instead of removing `boxes[right]` immediately, we can:

1. Remove everything between `mid` and `right`
2. Allow `boxes[mid]` to merge with `boxes[right]`

This creates a larger group and potentially a better score.

The recurrence becomes:

```
dp(left, mid, k+1) + dp(mid+1, right-1, 0)
```

This is the heart of the algorithm.

### Step 5: Memoize Results

Many states repeat during recursion.

We store computed results in a memoization table so each state is solved only once.

Without memoization, the recursion would still be exponential.

### Step 6: Return the Final Answer

The final problem is:

```
dp(0, n-1, 0)
```

since initially no extra boxes are attached.

### Why it works

The algorithm works because every optimal solution must eventually decide one of two things for the rightmost box:

- Remove it immediately
- Merge it with an earlier box of the same color

The DP explores both possibilities exhaustively while memoization prevents recomputation. The parameter `k` preserves all information needed about future merge potential, making the state fully sufficient for optimal substructure.

## Python Solution

```
from typing import List
from functools import lru_cache

class Solution:
    def removeBoxes(self, boxes: List[int]) -> int:
        n = len(boxes)

        @lru_cache(None)
        def dp(left: int, right: int, k: int) -> int:
            if left > right:
                return 0

            # Compress consecutive equal boxes at the right end
            while left < right and boxes[right] == boxes[right - 1]:
                right -= 1
                k += 1

            # Remove the current group immediately
            best = dp(left, right - 1, 0) + (k + 1) * (k + 1)

            # Try merging with earlier equal-colored boxes
            for mid in range(left, right):
                if boxes[mid] == boxes[right]:
                    best = max(
                        best,
                        dp(left, mid, k + 1) +
                        dp(mid + 1, right - 1, 0)
                    )

            return best

        return dp(0, n - 1, 0)
```

The implementation closely follows the recursive definition of the dynamic programming state.

The `dp(left, right, k)` function computes the optimal score for a subarray while tracking how many additional same-colored boxes are attached to the right side.

The first base case handles empty intervals. If `left > right`, there are no boxes remaining, so the score is `0`.

Next, the implementation compresses consecutive equal-colored boxes at the right boundary. This is an important optimization because consecutive equal boxes should always be treated as one larger removable group.

The variable `best` initially stores the strategy of removing the right group immediately. The score earned is `(k + 1)^2`, plus the optimal solution for the remaining left interval.

The loop over `mid` explores delayed merging opportunities. Whenever `boxes[mid]` matches `boxes[right]`, we attempt to remove the middle section first so the matching colors become adjacent.

Memoization through `@lru_cache` ensures every state is computed once.

## Go Solution

```
func removeBoxes(boxes []int) int {
	n := len(boxes)

	type State struct {
		left  int
		right int
		k     int
	}

	memo := make(map[State]int)

	var dp func(left, right, k int) int

	dp = func(left, right, k int) int {
		if left > right {
			return 0
		}

		state := State{left, right, k}

		if val, exists := memo[state]; exists {
			return val
		}

		originalRight := right
		originalK := k

		// Compress consecutive equal boxes at the right end
		for left < right && boxes[right] == boxes[right-1] {
			right--
			k++
		}

		// Remove the current group immediately
		best := dp(left, right-1, 0) + (k+1)*(k+1)

		// Try merging with earlier equal-colored boxes
		for mid := left; mid < right; mid++ {
			if boxes[mid] == boxes[right] {
				candidate := dp(left, mid, k+1) +
					dp(mid+1, right-1, 0)

				if candidate > best {
					best = candidate
				}
			}
		}

		memo[State{left, originalRight, originalK}] = best

		return best
	}

	return dp(0, n-1, 0)
}
```

The Go implementation uses an explicit `State` struct as the memoization key because Go does not provide built-in function caching like Python's `lru_cache`.

The recursive structure is otherwise identical to the Python version.

Go slices are reference-based, so the algorithm does not copy arrays during recursion. This keeps memory usage manageable.

Integer overflow is not a concern because the maximum score is bounded by `100 * 100 = 10000`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n^4) | There are O(n^3) states, each may scan O(n) positions |
| Space | O(n^3) | Memoization stores all DP states |

There are three DP dimensions:

- `left`
- `right`
- `k`

Each ranges up to `n`, producing at most `O(n^3)` unique states.

For each state, the algorithm may iterate through the interval searching for matching colors, which costs `O(n)` time.

Therefore the total complexity becomes `O(n^4)`.

With `n <= 100`, this complexity is acceptable.

## Edge Cases

### Single Box

A single box is the smallest valid input. This case can expose incorrect recursion base conditions.

For example:

```
[5]
```

The correct answer is `1`.

The implementation handles this naturally because removing one box yields:

```
(0 + 1)^2 = 1
```

and the remaining interval becomes empty.

### All Boxes Have the Same Color

Inputs like:

```
[2,2,2,2,2]
```

are important because the optimal solution removes everything at once.

A buggy implementation might split the group unnecessarily and lose points.

The compression loop correctly combines consecutive equal boxes into the `k` parameter, ensuring the entire group is removed together for:

```
5^2 = 25
```

### Same Colors Separated by Other Colors

This is the hardest category of cases.

Example:

```
[1,2,1]
```

If we remove either `1` immediately, the total becomes:

```
1 + 1 + 1 = 3
```

But the optimal strategy is:

1. Remove `2`
2. Merge the two `1`s
3. Score `1 + 4 = 5`

This is exactly why the extra DP parameter `k` is required. It preserves information about future merge opportunities.

### Completely Distinct Boxes

Inputs such as:

```
[1,2,3,4,5]
```

contain no merge opportunities.

The optimal strategy simply removes every box individually.

This case verifies that the algorithm still behaves correctly when no beneficial merges exist.
