## Solution

## Problem Understanding

The problem gives a list of time points represented as strings in 24-hour clock format, where each time is written as `"HH:MM"`. We must determine the minimum difference in minutes between any two time points in the list.

A time such as `"23:59"` represents 11:59 PM, while `"00:00"` represents midnight. The difference between two times is measured in minutes. One important detail is that the clock wraps around after midnight. For example, the difference between `"23:59"` and `"00:00"` is only 1 minute, not 1439 minutes.

The input size can be as large as `2 * 10^4`, which means we need an efficient solution. A naive comparison of every pair may still technically pass in some languages, but it is not ideal and does unnecessary work.

The key challenge is handling the circular nature of time. If we sort the times normally, the minimum difference might occur between two adjacent times in sorted order, or between the last and first times when wrapping around midnight.

Several edge cases are important:

- Duplicate times immediately produce an answer of `0`, because two identical times have zero difference.
- Times near midnight can create the smallest difference through wraparound.
- The list can contain many values, so repeated expensive operations should be avoided.
- The problem guarantees that every string is a valid `"HH:MM"` time.

A useful observation is that there are only `24 * 60 = 1440` unique minute values in a day. Since the input can exceed 1440 elements, duplicates are guaranteed by the pigeonhole principle whenever `len(timePoints) > 1440`.

## Approaches

### Brute Force Approach

The brute force solution converts every time into total minutes from midnight, then compares every possible pair of times.

For example:

- `"01:30"` becomes `90`
- `"23:59"` becomes `1439`

For each pair `(i, j)`, we compute the absolute difference between the two minute values. Because time wraps around midnight, we must also consider the circular difference:

```
min(diff, 1440 - diff)
```

We keep track of the smallest value across all pairs.

This approach is correct because it explicitly checks every possible pair, so the minimum cannot be missed.

However, its time complexity is `O(n^2)`, which becomes inefficient for large inputs.

### Optimal Approach

The key insight is that after converting times into minutes and sorting them, the minimum difference must occur between adjacent times in sorted order.

Why is this true?

Suppose three times are ordered like this:

```
a < b < c
```

The difference between `a` and `c` must be at least as large as one of the adjacent gaps:

```
(c - a) >= min(b - a, c - b)
```

Therefore, checking non-adjacent pairs is unnecessary.

The only extra case we must handle is the circular wraparound between the last and first elements. For example:

```
23:59 -> 1439
00:00 -> 0
```

The wraparound difference is:

```
1440 - 1439 + 0 = 1
```

This leads to a much more efficient algorithm:

1. Convert all times to minutes.
2. Sort the minutes.
3. Compare adjacent values.
4. Compare the last and first values with wraparound.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Compares every pair of times |
| Optimal | O(n log n) | O(n) | Sorts times and checks adjacent differences |

## Algorithm Walkthrough

1. Convert each time string into total minutes from midnight.

Split the string using `":"` to obtain hours and minutes. Convert the values into integers and compute:

```
total_minutes = hours * 60 + minutes
```

This transforms the problem from string comparison into simple integer arithmetic.
2. Store all minute values in a list.

Working with integers makes sorting and difference calculations straightforward.
3. Sort the list of minutes.

After sorting, the closest times must appear next to each other in the sorted order.
4. Initialize the minimum difference.

Start with a very large number so that any valid difference will replace it.
5. Compare every adjacent pair.

For each index `i`, compute:

```
minutes[i] - minutes[i - 1]
```

Update the minimum difference if the current gap is smaller.
6. Handle the circular wraparound case.

The day wraps after 1440 minutes, so we must compare the last time with the first time across midnight:

```
wrap_diff = 1440 - last + first
```
7. Return the smallest difference found.

### Why it works

After sorting, any non-adjacent pair must have a difference greater than or equal to at least one adjacent pair. Therefore, checking adjacent differences is sufficient to find the minimum linear gap. The only missing possibility is the circular transition across midnight, which is handled separately by comparing the first and last elements with wraparound arithmetic.

## Python Solution

```
from typing import List

class Solution:
    def findMinDifference(self, timePoints: List[str]) -> int:
        minutes_list = []

        for time_point in timePoints:
            hours, minutes = map(int, time_point.split(":"))
            total_minutes = hours * 60 + minutes
            minutes_list.append(total_minutes)

        minutes_list.sort()

        minimum_difference = float("inf")

        for i in range(1, len(minutes_list)):
            current_difference = minutes_list[i] - minutes_list[i - 1]
            minimum_difference = min(minimum_difference, current_difference)

        wraparound_difference = (
            1440 - minutes_list[-1] + minutes_list[0]
        )

        minimum_difference = min(
            minimum_difference,
            wraparound_difference
        )

        return minimum_difference
```

The implementation begins by converting each time string into an integer representing minutes from midnight. This avoids dealing with string comparisons later.

The list of minute values is then sorted. Once sorted, the minimum difference must occur between neighboring elements or across the midnight boundary.

The loop checks every adjacent pair and continuously updates the smallest difference seen so far.

Finally, the code computes the wraparound difference between the last and first elements. This step is essential because the clock is circular.

The function returns the smallest value among all computed differences.

## Go Solution

```
package main

import (
	"sort"
	"strconv"
	"strings"
)

func findMinDifference(timePoints []string) int {
	minutesList := make([]int, 0, len(timePoints))

	for _, timePoint := range timePoints {
		parts := strings.Split(timePoint, ":")

		hours, _ := strconv.Atoi(parts[0])
		minutes, _ := strconv.Atoi(parts[1])

		totalMinutes := hours*60 + minutes
		minutesList = append(minutesList, totalMinutes)
	}

	sort.Ints(minutesList)

	minimumDifference := 1 << 30

	for i := 1; i < len(minutesList); i++ {
		currentDifference := minutesList[i] - minutesList[i-1]

		if currentDifference < minimumDifference {
			minimumDifference = currentDifference
		}
	}

	wraparoundDifference := 1440 - minutesList[len(minutesList)-1] + minutesList[0]

	if wraparoundDifference < minimumDifference {
		minimumDifference = wraparoundDifference
	}

	return minimumDifference
}
```

The Go implementation follows the same algorithmic structure as the Python solution.

The main language-specific difference is parsing the time strings. Go uses `strings.Split` and `strconv.Atoi` to convert substrings into integers.

Sorting is performed with `sort.Ints`.

Go does not provide an infinity constant for integers, so a large initial value such as `1 << 30` is used for initialization.

Since all minute differences are at most 1440, integer overflow is never a concern.

## Edge Cases

One important edge case is duplicate times. If the input contains two identical time strings, the minimum difference must be zero. A naive implementation that only compares distinct values or incorrectly removes duplicates could fail here. In this solution, sorting naturally places duplicates next to each other, so the adjacent comparison immediately detects a difference of zero.

Another important edge case is the midnight wraparound. For example, `"23:59"` and `"00:00"` are only one minute apart, even though their direct numerical difference is 1439. A solution that only checks adjacent sorted values without handling circular wraparound would incorrectly return 1439. The implementation explicitly computes the wraparound difference between the last and first sorted times.

A third edge case involves very large inputs. Since the input size can reach `2 * 10^4`, an `O(n^2)` solution performs too many pairwise comparisons. The optimized algorithm avoids this by sorting once and checking only neighboring elements, reducing the complexity to `O(n log n)`.

Another subtle edge case occurs when all times are evenly spread apart. For example:

```
["00:00", "12:00"]
```

The minimum difference is 720 minutes in both directions. The implementation handles this correctly because the wraparound calculation and adjacent comparison both produce valid results.
