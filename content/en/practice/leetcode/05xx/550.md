## Solution

## Problem Understanding

The problem asks us to calculate the fraction of players who returned to play on the day immediately after their first login. The input is a database table `Activity` that tracks each player’s daily activity on a game. Each row contains a `player_id`, a `device_id`, the `event_date`, and the number of games played. The combination of `player_id` and `event_date` is unique, so each player has at most one entry per day.

The output should be a single value, the fraction of players who logged in the day after their first login, rounded to two decimal places. For example, if three players exist and only one returned on the next day, the output should be `0.33`.

Key constraints and implications include: the table may contain many players over multiple years, a player may use different devices, and some players may only have one login day. The solution must account for first login dates, consecutive day activity, and rounding. Edge cases that could trip a naive solution include players who only logged in once, multiple entries for the same player on non-consecutive days, or players whose first login is the last date in the dataset.

## Approaches

A brute-force approach would retrieve the first login date for each player, then check if a record exists for that player on the next day. While this would yield the correct result, it could require joining the table with itself or scanning the table multiple times, which may be inefficient for large datasets.

The key insight for an optimal solution is that we can use a self-join to compare each player’s first login date to their next-day login. By filtering the first login dates and using a left join or inner join on the table with `event_date = first_login_date + 1`, we can efficiently find players who returned the next day. Using aggregation functions like `MIN` and simple joins ensures the solution scales with the number of players without scanning each player multiple times.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(N^2) | O(N) | Retrieve first login, then scan the table for each player’s next day login |
| Optimal | O(N) | O(N) | Use aggregation and self-join to find next-day logins efficiently |

## Algorithm Walkthrough

1. **Find first login dates**: For each `player_id`, determine the earliest `event_date` using an aggregation function like `MIN(event_date)`. This gives us a table of `player_id` and `first_login_date`.
2. **Check for next-day login**: Perform a join of this table with `Activity` itself, where the join condition matches `player_id` and `event_date = first_login_date + 1`. This will identify players who logged in the day after their first login.
3. **Count returning players**: Count the number of players who have a matching next-day login from step 2.
4. **Count total players**: Count the total number of distinct players in the table.
5. **Calculate fraction**: Divide the count of returning players by the total number of players, round the result to two decimal places, and return it as a single-row output.

**Why it works:** By using the first login date and checking only for the next-day login, we are guaranteed to identify all players who returned immediately after their initial login. The aggregation ensures each player is considered once, and the join ensures we only count valid consecutive-day logins.

## Python Solution

```
# Since this is a database problem, the solution is expressed in SQL.
# In Python, we would typically use an ORM or execute raw SQL queries.
# Here is a Python-style representation using raw SQL execution.

import sqlite3

def fraction(players_db: sqlite3.Connection) -> float:
    query = """
        WITH first_login AS (
            SELECT player_id, MIN(event_date) AS first_date
            FROM Activity
            GROUP BY player_id
        )
        SELECT ROUND(
            CAST(COUNT(DISTINCT a.player_id) AS FLOAT) /
            (SELECT COUNT(DISTINCT player_id) FROM Activity), 2
        ) AS fraction
        FROM first_login f
        JOIN Activity a
          ON f.player_id = a.player_id
         AND a.event_date = DATE(f.first_date, '+1 day');
    """
    cursor = players_db.cursor()
    cursor.execute(query)
    result = cursor.fetchone()
    return result[0]
```

This code first creates a temporary table of the first login dates for each player. It then joins this with the original `Activity` table to find players who logged in the next day. Finally, it computes the fraction, rounds it, and returns the value. The join ensures only the relevant next-day logins are counted.

## Go Solution

package main

import (
	"database/sql"
	"fmt"
	_ "github.com/mattn/go-sqlite3"

func Fraction(db *sql.DB) (float64, error) {
	query := `
		WITH first_login AS (
			SELECT player_id, MIN(event_date) AS first_date
			FROM Activity
			GROUP BY player_id
		)
		SELECT ROUND(
			CAST(COUNT(DISTINCT a.player_id) AS FLOAT) /
			(SELECT COUNT(DISTINCT player_id) FROM Activity), 2
		) AS fraction
		FROM first_login f
		JOIN Activity a
		  ON f.player_id = a.player_id
		 AND a.event_date = DATE(f.first_date, '+1 day');
	`
	var fraction float64
	err := db.QueryRow(query).Scan(&fraction)
	if err != nil {
		return 0, err
	}
	return fraction, nil
}
