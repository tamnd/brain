## Solution

## Problem Understanding

This problem asks us to compute the boundary traversal of a binary tree in a very specific order. The boundary is not simply the outermost nodes visually, instead it is defined as a concatenation of four parts:

1. The root node
2. The left boundary
3. All leaf nodes from left to right
4. The reversed right boundary

The challenge comes from carefully avoiding duplicate nodes while respecting the exact traversal rules.

The input is the root of a binary tree. Each node contains an integer value and pointers to its left and right children. The output should be a list of integers representing the boundary traversal.

The left boundary starts from the root's left child and follows a path downward. At each step, we prefer the left child. If no left child exists, we move to the right child. Importantly, leaf nodes are excluded from the left boundary because leaves are handled separately.

The right boundary is symmetric. It starts from the root's right child and follows downward, preferring the right child first. If no right child exists, it uses the left child. Again, leaf nodes are excluded. Finally, the right boundary must appear in reverse order in the final answer.

The leaves are all nodes without children, collected from left to right across the entire tree. The root itself is not considered a leaf for this purpose because it already appears as the first element of the boundary.

The constraints allow up to `10^4` nodes, which means we need an efficient traversal. An `O(n)` solution is ideal because every node may need to be examined at least once.

Several edge cases can easily cause bugs:

- A tree with only one node
- Trees that only have left children
- Trees that only have right children
- Trees where boundary nodes are also leaves
- Skewed trees where the fallback rules for left or right boundary traversal matter
- Duplicate insertion of leaves or root nodes

A correct implementation must ensure every boundary node appears exactly once.

## Approaches

### Brute Force Approach

A brute force strategy would first compute the full set of nodes that visually belong to the outer edge of the tree. One way to do this is to track every node with its depth and horizontal position, then reconstruct which nodes belong to the left edge, bottom leaves, and right edge.

This approach becomes unnecessarily complicated because the problem already defines explicit traversal rules. Managing coordinates or level boundaries introduces extra bookkeeping and increases implementation complexity.

Another brute force variant could repeatedly traverse the tree to locate:

- The leftmost node at each depth
- All leaves
- The rightmost node at each depth

While this produces the correct answer, it may revisit many nodes multiple times and requires additional structures to avoid duplicates.

The brute force method works because it eventually identifies all required boundary nodes, but it is inefficient and harder to reason about.

### Optimal Approach

The key observation is that the boundary naturally decomposes into three independent traversals:

1. Left boundary traversal
2. Leaf traversal
3. Right boundary traversal

Each traversal can be implemented with a simple DFS or iterative walk.

The most important insight is that leaf nodes should only be added during the leaf traversal. This avoids duplicates between boundaries and leaves.

The optimal algorithm works as follows:

- Add the root first
- Traverse the left boundary while excluding leaves
- Traverse all leaves from left to right
- Traverse the right boundary while excluding leaves, then reverse the collected nodes

Since each node is visited at most once during DFS operations, the total complexity is linear.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeated scans or excessive bookkeeping |
| Optimal | O(n) | O(h) | Separate boundary traversals with DFS |

Here, `n` is the number of nodes and `h` is the tree height.

## Algorithm Walkthrough

1. Start by handling the empty tree case.

If the root is `None`, return an empty list immediately.
2. Add the root value to the result.

The root is always part of the boundary. However, if the root is the only node, we should return immediately to avoid duplication during leaf traversal.
3. Traverse the left boundary.

Begin from `root.left`.

While traversing:

- Add the current node if it is not a leaf
- Prefer moving to the left child
- If no left child exists, move to the right child

This follows the exact problem definition for the left boundary.
4. Collect all leaf nodes using DFS.

Perform a depth first traversal over the entire tree.

For every node:

- If it is a leaf, add it to the result
- Otherwise continue DFS on left and right children

Since leaves are collected left-to-right naturally through DFS order, no extra sorting is needed.
5. Traverse the right boundary.

Begin from `root.right`.

While traversing:

- Add non-leaf nodes into a temporary list
- Prefer moving to the right child
- If no right child exists, move to the left child

After traversal completes, reverse the temporary list and append it to the result.
6. Return the final boundary list.

### Why it works

The algorithm partitions the boundary into three disjoint groups:

- Left boundary excluding leaves
- All leaves
- Right boundary excluding leaves

Because leaf nodes are handled separately, no node is added twice. The traversal rules exactly mirror the formal boundary definitions from the problem statement. Since every required node belongs to exactly one of these groups, the final result is correct.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import List, Optional

class Solution:
    def boundaryOfBinaryTree(self, root: Optional[TreeNode]) -> List[int]:
        if not root:
            return []

        def is_leaf(node: TreeNode) -> bool:
            return not node.left and not node.right

        if is_leaf(root):
            return [root.val]

        boundary = [root.val]

        # Add left boundary
        current = root.left
        while current:
            if not is_leaf(current):
                boundary.append(current.val)

            if current.left:
                current = current.left
            else:
                current = current.right

        # Add leaves
        def add_leaves(node: Optional[TreeNode]) -> None:
            if not node:
                return

            if is_leaf(node):
                boundary.append(node.val)
                return

            add_leaves(node.left)
            add_leaves(node.right)

        add_leaves(root)

        # Add right boundary
        stack = []

        current = root.right
        while current:
            if not is_leaf(current):
                stack.append(current.val)

            if current.right:
                current = current.right
            else:
                current = current.left

        while stack:
            boundary.append(stack.pop())

        return boundary
```

The implementation begins with an early return for empty trees. A helper function called `is_leaf` simplifies repeated leaf checks throughout the code.

If the root itself is a leaf, the function immediately returns `[root.val]`. This prevents the root from being added multiple times later.

The left boundary traversal iteratively walks downward from `root.left`. Non-leaf nodes are appended directly into the boundary list.

The leaf collection step uses DFS recursion. Every leaf node encountered is added exactly once. Since recursion visits the left subtree before the right subtree, leaves naturally appear in left-to-right order.

The right boundary traversal mirrors the left traversal, except collected nodes are stored temporarily in a stack list. Popping from the stack reverses their order before appending to the final answer.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */

func boundaryOfBinaryTree(root *TreeNode) []int {
    if root == nil {
        return []int{}
    }

    isLeaf := func(node *TreeNode) bool {
        return node.Left == nil && node.Right == nil
    }

    if isLeaf(root) {
        return []int{root.Val}
    }

    boundary := []int{root.Val}

    // Add left boundary
    current := root.Left

    for current != nil {
        if !isLeaf(current) {
            boundary = append(boundary, current.Val)
        }

        if current.Left != nil {
            current = current.Left
        } else {
            current = current.Right
        }
    }

    // Add leaves
    var addLeaves func(node *TreeNode)

    addLeaves = func(node *TreeNode) {
        if node == nil {
            return
        }

        if isLeaf(node) {
            boundary = append(boundary, node.Val)
            return
        }

        addLeaves(node.Left)
        addLeaves(node.Right)
    }

    addLeaves(root)

    // Add right boundary
    stack := []int{}

    current = root.Right

    for current != nil {
        if !isLeaf(current) {
            stack = append(stack, current.Val)
        }

        if current.Right != nil {
            current = current.Right
        } else {
            current = current.Left
        }
    }

    for len(stack) > 0 {
        boundary = append(boundary, stack[len(stack)-1])
        stack = stack[:len(stack)-1]
    }

    return boundary
}
```

The Go implementation closely mirrors the Python solution. The primary difference is explicit pointer handling using `nil`.

Go slices are used for both the final boundary list and the temporary stack. Removing the last element from a slice simulates stack popping efficiently.

The recursive `addLeaves` helper is declared using a function variable because Go requires named recursive closures.

No integer overflow concerns exist because node values remain within small bounds.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each node is visited at most once |
| Space | O(h) | Recursive DFS stack depth equals tree height |

The left boundary traversal and right boundary traversal each process only a single root-to-leaf path. The DFS leaf traversal visits every node exactly once. Therefore the total runtime is linear in the number of nodes.

The extra space usage comes primarily from recursion depth during DFS. In the worst case of a skewed tree, the height becomes `O(n)`.

## Edge Cases

### Single Node Tree

A tree containing only the root is one of the most common sources of duplication bugs. If the implementation adds the root first and later adds all leaves, the root may appear twice. The solution prevents this by returning immediately when the root itself is a leaf.

### Completely Left-Skewed Tree

In a tree where every node only has a left child, every internal node belongs to the left boundary. The final leaf node should appear only once as part of the leaf traversal. The implementation correctly skips leaf nodes during left boundary collection.

### Completely Right-Skewed Tree

This case is tricky because the right boundary must appear in reverse order. A naive top-down append would produce the wrong order. The implementation solves this by storing right boundary nodes in a temporary stack and appending them in reverse.

### Boundary Fallback Traversal

The problem definition specifies fallback behavior when the preferred child does not exist. For example, a left boundary node without a left child should continue using the right child. Missing this rule leads to incomplete boundaries. The implementation explicitly checks the preferred child first, then falls back to the alternate child.
