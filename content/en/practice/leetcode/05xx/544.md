# LeetCode 544 - Output Contest Matches

## Problem Understanding

This problem asks us to simulate the structure of a tournament bracket where the strongest team always plays the weakest remaining team in each round.

The input is a single integer `n`, representing the total number of teams in the tournament. Teams are labeled from `1` to `n`, where:

- Team `1` is the strongest
- Team `n` is the weakest

The value of `n` is guaranteed to be a power of two. This guarantee is important because it means every round can cleanly divide the teams into pairs until exactly one final winner remains.

The output must be a string representing the complete tournament bracket. Each matchup is written in the form:

```
(a,b)
```

where `a` and `b` are either team numbers or previous match expressions.

The tournament proceeds round by round:

- In every round, the strongest remaining participant is paired with the weakest remaining participant.
- Winners advance as grouped expressions.
- The process continues until only one final expression remains.

For example, when `n = 8`:

- First round:

`(1,8),(2,7),(3,6),(4,5)`
- Second round:

`((1,8),(4,5)),((2,7),(3,6))`
- Third round:

`(((1,8),(4,5)),((2,7),(3,6)))`

The final remaining expression is the answer.

The constraints are small:

- `n == 2^x`
- `1 <= x <= 12`

This means:

- Minimum `n = 2`
- Maximum `n = 4096`

Even though the constraints are not very large, we still want a clean and efficient construction method.

There are several important observations and edge cases:

- Since `n` is always a power of two, we never need to handle odd numbers of teams.
- The smallest possible input is `n = 2`, which should directly produce `(1,2)`.
- Each round reduces the number of participants by half.
- Match expressions become nested strings, so string construction efficiency matters somewhat for larger values like `4096`.

## Approaches

### Brute Force Approach

A brute force solution would explicitly simulate every round of the tournament using nested recursive structures or binary trees.

One possible implementation would:

1. Create a tree node for every team.
2. Pair strongest and weakest nodes.
3. Build parent nodes representing matches.
4. Continue until only one root node remains.
5. Traverse the tree to generate the final string.

This works because it directly models the tournament structure exactly as described in the problem statement.

However, this approach introduces unnecessary complexity:

- Extra tree node structures
- Recursive traversal logic
- Additional memory overhead

Although the constraints are small enough that this would still pass, it is more complicated than necessary.

### Optimal Approach

The key insight is that we never actually need a tree structure.

At every round, we only care about the current list of participants. Each participant can already be represented as a string.

Initially:

```
["1", "2", "3", ..., "n"]
```

In each round:

- Pair the first and last entries
- Pair the second and second-last entries
- Continue inward
- Replace them with combined match strings

For example:

```
["1","2","3","4"]
```

becomes:

```
["(1,4)", "(2,3)"]
```

Then:

```
["((1,4),(2,3))"]
```

This iterative simulation is simple, efficient, and directly matches the tournament rules.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) | O(n) | Builds explicit tournament tree |
| Optimal | O(n log n) | O(n) | Iteratively combines match strings |

## Algorithm Walkthrough

1. Initialize a list of team strings.

Create an array containing all team labels as strings:

```
["1", "2", "3", ..., "n"]
```

We use strings because every future round produces string expressions.
2. Repeat until only one expression remains.

Each iteration represents one tournament round.
3. Pair strongest with weakest.

Use two pointers:

- One starting from the left
- One starting from the right

Combine them as:

```
"(" + left + "," + right + ")"
```

This preserves the required bracket structure.
4. Build the next round.

Store all newly created match expressions into a new list.
5. Replace the current list.

After finishing one round, the new list becomes the active participants for the next round.
6. Continue until a single expression remains.

The remaining string represents the complete tournament bracket.

### Why it works

The invariant is that after every round, the list contains valid tournament expressions representing the surviving participants in correct tournament order.

Since each round pairs the strongest remaining participant with the weakest remaining participant, and since the algorithm always pairs the leftmost and rightmost elements, the generated bracket exactly matches the required tournament structure.

Eventually only one fully nested expression remains, which is the final tournament representation.

## Python Solution

```
class Solution:
    def findContestMatch(self, n: int) -> str:
        matches = [str(i) for i in range(1, n + 1)]

        while len(matches) > 1:
            next_round = []

            left = 0
            right = len(matches) - 1

            while left < right:
                paired = f"({matches[left]},{matches[right]})"
                next_round.append(paired)

                left += 1
                right -= 1

            matches = next_round

        return matches[0]
```

The implementation begins by converting all team numbers into strings. This allows every future tournament expression to be constructed uniformly without repeated integer conversions.

The outer `while` loop represents tournament rounds. Each round cuts the number of participants in half.

Inside the loop, two pointers are used:

- `left` starts at the beginning
- `right` starts at the end

This directly models the strongest-versus-weakest pairing rule.

For every pair, we create a new expression:

```
f"({matches[left]},{matches[right]})"
```

These expressions are collected into `next_round`.

After all pairs are processed, `matches` is replaced with `next_round`, simulating advancement to the next tournament round.

When only one expression remains, it represents the entire tournament bracket and is returned.

## Go Solution

```
package main

import "strconv"

func findContestMatch(n int) string {
    matches := make([]string, n)

    for i := 0; i < n; i++ {
        matches[i] = strconv.Itoa(i + 1)
    }

    for len(matches) > 1 {
        nextRound := []string{}

        left := 0
        right := len(matches) - 1

        for left < right {
            paired := "(" + matches[left] + "," + matches[right] + ")"
            nextRound = append(nextRound, paired)

            left++
            right--
        }

        matches = nextRound
    }

    return matches[0]
}
```

The Go implementation follows the same logic as the Python version.

The primary difference is string conversion. Go requires `strconv.Itoa()` to convert integers into strings.

Slices are used to represent tournament rounds dynamically. Each round constructs a new slice containing the combined match expressions.

Since the problem constraints are small, repeated string concatenation is perfectly acceptable and does not create performance issues.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | There are log n rounds, and each round processes O(n) total string content |
| Space | O(n) | We store the current round's match expressions |

The number of rounds is `log n` because the number of participants halves each time.

Across all rounds, every participant contributes to increasingly nested strings. Since the final output itself has linear size relative to the number of teams and nesting operations occur across multiple rounds, the overall runtime is effectively `O(n log n)`.

The space usage is `O(n)` because we maintain lists of current tournament expressions.

## Edge Cases

### Smallest Tournament

The smallest valid input is `n = 2`.

A buggy implementation might incorrectly add extra nesting or fail because there is only one pairing round. The implementation handles this naturally because the first round immediately produces a single combined expression:

```
(1,2)
```

The loop then terminates correctly.

### Deeply Nested Large Inputs

For large inputs like `n = 4096`, the resulting expression becomes deeply nested and quite large.

Recursive implementations could risk excessive recursion depth or unnecessary overhead. The iterative approach avoids recursion entirely, making it safe and efficient even for the maximum constraint.

### Correct Pairing Order

One subtle bug source is pairing adjacent teams instead of strongest versus weakest remaining teams.

For example, incorrectly pairing:

```
(1,2),(3,4)
```

would violate the problem rules.

The implementation avoids this by always using two pointers:

- Leftmost participant
- Rightmost participant

This guarantees correct tournament seeding in every round.
