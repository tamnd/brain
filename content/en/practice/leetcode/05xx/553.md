## Solution

## Problem Understanding

In this problem, we are given an array of positive integers `nums`. The array represents a sequence of division operations performed from left to right.

For example, if:

```
nums = [2,3,4]
```

then the default expression is:

```
2/3/4
```

which evaluates as:

```
(2/3)/4
```

because division is left associative.

The key twist is that we are allowed to insert parentheses anywhere in the expression to change the order of evaluation. Our goal is to produce the expression string that yields the maximum possible numerical result.

We are not asked to compute the maximum value itself. Instead, we must return the exact expression string that produces that value.

The problem also specifies that the returned expression must not contain redundant parentheses. This means we should only include parentheses when they actually affect evaluation order.

The constraints are small:

- `1 <= nums.length <= 10`
- `2 <= nums[i] <= 1000`

Since the array length is at most 10, even exponential solutions could technically work. However, the problem is designed around discovering a mathematical observation that leads to a very elegant optimal solution.

A few important edge cases immediately stand out:

- If the array contains only one number, the answer is simply that number as a string.
- If the array contains exactly two numbers, there is no meaningful way to add parentheses, so the answer is just `"a/b"`.
- Parentheses become useful only when there are at least three numbers.
- Because all numbers are positive and greater than 1, maximizing the expression means making the denominator as small as possible.

The problem guarantees that there is exactly one optimal division arrangement, which simplifies reasoning about ties.

## Approaches

### Brute Force Approach

The brute force solution tries every possible way to parenthesize the expression.

This is similar to generating all possible binary expression trees for the sequence of divisions. For every subarray, we recursively split the expression at every possible division point and compute all possible values.

For example, for:

```
[1000,100,10,2]
```

we would recursively explore expressions like:

```
((1000/100)/10)/2
(1000/(100/10))/2
1000/((100/10)/2)
1000/(100/(10/2))
```

and many others.

For each generated expression, we compute:

- its numeric value
- the corresponding expression string

Then we choose the expression with the maximum value.

This approach is correct because it exhaustively checks every valid parenthesization. However, the number of possible parenthesizations grows exponentially, following Catalan-number behavior.

Even though `n <= 10` makes this feasible in practice, it is unnecessarily complicated for a problem with a much simpler mathematical insight.

### Key Insight

Division behaves differently from addition or multiplication because changing parentheses can drastically alter the denominator.

Suppose we have:

```
a / b / c / d
```

Without parentheses, this becomes:

```
((a / b) / c) / d
```

But if we write:

```
a / (b / c / d)
```

then the denominator becomes:

```
((b / c) / d)
```

which is usually much smaller than `b * c * d`.

Since all numbers are greater than 1:

- We want the numerator to stay as large as possible.
- We want the denominator to become as small as possible.

To minimize the denominator, we should divide it repeatedly rather than multiply it through left association.

Therefore, the optimal structure is always:

```
nums[0] / (nums[1] / nums[2] / ... / nums[n-1])
```

for arrays of length at least 3.

This observation completely eliminates the need for dynamic programming or recursion.

## Algorithm Walkthrough

1. Check whether the array contains only one number.

If `nums.length == 1`, no division occurs. The answer is simply the number converted to a string.
2. Check whether the array contains exactly two numbers.

If `nums.length == 2`, there is only one valid expression:

```
a/b
```

Adding parentheses would not change evaluation order.
3. Handle arrays with length at least 3.

For larger arrays, the optimal structure is always:

```
nums[0]/(nums[1]/nums[2]/.../nums[n-1])
```
4. Build the expression string.

Convert each integer to a string and concatenate them using `/`.
5. Add parentheses around everything after the first number.

The denominator should be grouped together so that the repeated divisions minimize its value.
6. Return the final expression.

### Why it works

The first number is the numerator of the entire expression, so maximizing the result means minimizing everything after the first division.

For positive numbers greater than 1:

```
b / c / d
```

evaluates to:

```
(b / c) / d
```

which is smaller than:

```
b / (c / d)
```

Therefore, wrapping all remaining numbers into one grouped denominator produces the smallest possible denominator and thus the maximum overall value.

## Python Solution

```
from typing import List

class Solution:
    def optimalDivision(self, nums: List[int]) -> str:
        n = len(nums)

        if n == 1:
            return str(nums[0])

        if n == 2:
            return f"{nums[0]}/{nums[1]}"

        middle = "/".join(str(num) for num in nums[1:])

        return f"{nums[0]}/({middle})"
```

The implementation directly follows the mathematical insight.

First, we handle the small edge cases separately. Arrays of length 1 and 2 do not benefit from parentheses, so we return straightforward expressions.

For arrays of length at least 3, we build the denominator by joining all elements after the first with division operators.

For example:

```
nums = [1000, 100, 10, 2]
```

The joined denominator becomes:

```
100/10/2
```

Then we wrap it in parentheses:

```
1000/(100/10/2)
```

The algorithm performs only a single pass through the array while constructing the string.

## Go Solution

```
package main

import (
	"strconv"
	"strings"
)

func optimalDivision(nums []int) string {
	n := len(nums)

	if n == 1 {
		return strconv.Itoa(nums[0])
	}

	if n == 2 {
		return strconv.Itoa(nums[0]) + "/" + strconv.Itoa(nums[1])
	}

	parts := make([]string, 0)

	for i := 1; i < n; i++ {
		parts = append(parts, strconv.Itoa(nums[i]))
	}

	denominator := strings.Join(parts, "/")

	return strconv.Itoa(nums[0]) + "/(" + denominator + ")"
}
```

The Go implementation mirrors the Python logic closely.

Because Go does not have Python style string interpolation, we use:

- `strconv.Itoa()` to convert integers to strings
- `strings.Join()` to concatenate the denominator components

Slices are used to collect denominator strings efficiently before joining them.

Since the constraints are very small, integer overflow and memory usage are not concerns.

## Worked Examples

### Example 1

Input:

```
nums = [1000,100,10,2]
```

The algorithm proceeds as follows.

| Step | Action | Result |
| --- | --- | --- |
| 1 | Length is 4 | Use general case |
| 2 | Take first number | `1000` |
| 3 | Join remaining numbers | `"100/10/2"` |
| 4 | Wrap denominator in parentheses | `"(100/10/2)"` |
| 5 | Combine final expression | `"1000/(100/10/2)"` |

Evaluation:

```
1000 / ((100 / 10) / 2)
= 1000 / (10 / 2)
= 1000 / 5
= 200
```

This is larger than every alternative arrangement.

### Example 2

Input:

```
nums = [2,3,4]
```

| Step | Action | Result |
| --- | --- | --- |
| 1 | Length is 3 | Use general case |
| 2 | First number | `2` |
| 3 | Join remaining numbers | `"3/4"` |
| 4 | Add parentheses | `"(3/4)"` |
| 5 | Final expression | `"2/(3/4)"` |

Evaluation:

```
2 / (3 / 4)
= 2 * (4 / 3)
= 8 / 3
≈ 2.667
```

Without parentheses:

```
(2 / 3) / 4
= 1 / 6
≈ 0.167
```

The parenthesized version is clearly larger.

## Edge Cases

### Single Element Array

If the array contains only one number, there is no division operation at all. A naive implementation that always inserts division operators or parentheses could generate invalid output.

For example:

```
[5]
```

should return:

```
"5"
```

The implementation handles this immediately with a dedicated condition for `n == 1`.

### Two Element Array

With exactly two numbers, parentheses are unnecessary because there is only one possible division operation.

For example:

```
[8,2]
```

should return:

```
"8/2"
```

A careless implementation might incorrectly produce:

```
"8/(2)"
```

which contains redundant parentheses and violates the problem requirements.

The implementation avoids this by treating `n == 2` separately.

### Longer Arrays Requiring Minimal Denominator

The core mathematical challenge appears when there are at least three numbers.

For example:

```
[1000,100,10,2]
```

A naive left associative expression gives:

```
((1000/100)/10)/2 = 0.5
```

But grouping the denominator:

```
1000/(100/10/2)
```

produces:

```
200
```

The implementation always wraps every element after the first into a single parenthesized denominator, guaranteeing the maximum possible result.
