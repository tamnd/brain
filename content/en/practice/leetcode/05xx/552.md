## Solution

## Problem Understanding

The problem asks us to count how many attendance records of length `n` satisfy two rules for receiving an award.

Each character in the attendance record represents one day:

- `'P'` means the student was present.
- `'A'` means the student was absent.
- `'L'` means the student was late.

A valid attendance record must satisfy both of these conditions:

1. The total number of absences must be strictly less than 2, which means the record may contain either 0 or 1 `'A'`.
2. The student can never be late for 3 or more consecutive days, which means `"LLL"` may never appear as a substring.

We are given an integer `n`, representing the length of the attendance record, and we must return the number of valid strings of length `n`.

The answer can become extremely large because the number of possible strings grows exponentially with `n`. Therefore, we must return the result modulo `10^9 + 7`.

The constraint `1 <= n <= 10^5` is the most important part of the problem. A naive solution that generates all possible strings would require exploring `3^n` combinations, which is completely infeasible for `n = 100000`. This strongly suggests that we need a dynamic programming solution that reuses previously computed states instead of recomputing subproblems repeatedly.

Several edge cases are important:

- `n = 1`, because this is the smallest valid input.
- Records that already contain one `'A'`, because adding another `'A'` invalidates the sequence.
- Records ending with two consecutive `'L'`, because another `'L'` would create an invalid `"LLL"` sequence.
- Very large values of `n`, where efficiency and modulo arithmetic become critical.

The problem guarantees that `n` is always at least 1, so we do not need to handle empty input.

## Approaches

### Brute Force Approach

The brute force approach generates every possible attendance record of length `n`. Since each position can contain `'P'`, `'A'`, or `'L'`, there are `3^n` total strings.

For each generated string, we check:

- Whether it contains fewer than 2 `'A'` characters.
- Whether it contains `"LLL"` as a substring.

If both conditions are satisfied, we count the string as valid.

This approach is correct because it exhaustively checks every possible attendance record and filters out invalid ones. However, it is far too slow for the given constraints.

For example:

- When `n = 20`, there are already over 3 billion possible strings.
- When `n = 100000`, brute force is completely impossible.

The key issue is that many subproblems overlap. For example, when building strings character by character, the future validity only depends on:

- How many absences have been used so far.
- How many consecutive late days currently exist.

We do not need the entire string history.

### Optimal Dynamic Programming Approach

The key insight is that the validity of future choices depends only on a small amount of state information:

1. How many `'A'` characters have been used so far, either 0 or 1.
2. How many consecutive trailing `'L'` characters exist, which can only be 0, 1, or 2.

This means we can define a DP state:

`dp[absences][late_streak]`

where:

- `absences` is either 0 or 1.
- `late_streak` is 0, 1, or 2.

Each state stores the number of valid sequences that end with that configuration.

For each day, we transition by adding:

- `'P'`, which resets the late streak to 0.
- `'A'`, if we have not already used an absence.
- `'L'`, if the late streak is currently less than 2.

Because there are only `2 * 3 = 6` possible states, each iteration is constant time. We process `n` days, giving an overall complexity of `O(n)`.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(3^n) | O(n) | Generates all possible attendance records |
| Optimal Dynamic Programming | O(n) | O(1) | Tracks only absence count and trailing late streak |

## Algorithm Walkthrough

1. Define a DP table where `dp[a][l]` represents the number of valid attendance records for the current length:

- `a` is the number of absences used, either 0 or 1.
- `l` is the current consecutive late streak, 0 to 2.
2. Initialize the DP table for length 0:

- `dp[0][0] = 1`
- This represents the empty sequence.
3. Iterate from day 1 to day `n`.
4. For each existing state `(a, l)`, consider adding `'P'`:

- Present resets the late streak.
- Transition to `(a, 0)`.
5. Consider adding `'A'`:

- Only allowed if `a == 0`.
- Transition to `(1, 0)` because absence count increases and late streak resets.
6. Consider adding `'L'`:

- Only allowed if `l < 2`.
- Transition to `(a, l + 1)`.
7. Apply modulo `10^9 + 7` during every update to avoid overflow.
8. After processing all `n` days, sum all six DP states because every remaining state represents a valid attendance record.

### Why it works

The algorithm works because every valid attendance record can be uniquely described by two properties:

- The number of absences used so far.
- The current consecutive late streak.

These two values completely determine which characters may legally be added next. Since every transition preserves validity and every valid sequence is counted exactly once, the DP correctly computes the total number of valid attendance records.

## Python Solution

```
class Solution:
    def checkRecord(self, n: int) -> int:
        MOD = 10**9 + 7

        # dp[a][l]
        # a = number of absences used (0 or 1)
        # l = consecutive late count (0, 1, 2)
        dp = [[0] * 3 for _ in range(2)]
        dp[0][0] = 1

        for _ in range(n):
            next_dp = [[0] * 3 for _ in range(2)]

            for absences in range(2):
                for late_streak in range(3):
                    current = dp[absences][late_streak]

                    if current == 0:
                        continue

                    # Add 'P'
                    next_dp[absences][0] = (
                        next_dp[absences][0] + current
                    ) % MOD

                    # Add 'A'
                    if absences == 0:
                        next_dp[1][0] = (
                            next_dp[1][0] + current
                        ) % MOD

                    # Add 'L'
                    if late_streak < 2:
                        next_dp[absences][late_streak + 1] = (
                            next_dp[absences][late_streak + 1] + current
                        ) % MOD

            dp = next_dp

        total = 0

        for absences in range(2):
            for late_streak in range(3):
                total = (total + dp[absences][late_streak]) % MOD

        return total
```

The implementation follows the DP formulation directly.

The table `dp[a][l]` stores how many valid sequences currently exist with a particular combination of absence count and trailing late streak.

At each iteration, we build a new table called `next_dp`. This represents all valid sequences after adding one more character.

For every state, we try all three possible character additions:

- `'P'` resets the late streak to 0.
- `'A'` is only allowed if no absence has been used yet.
- `'L'` is only allowed when the late streak is less than 2.

After processing all transitions for the current day, we replace `dp` with `next_dp`.

Finally, we sum all valid states because every remaining state corresponds to a legal attendance record of length `n`.

## Go Solution

```
func checkRecord(n int) int {
	const MOD int = 1000000007

	dp := [2][3]int{}
	dp[0][0] = 1

	for day := 0; day < n; day++ {
		nextDP := [2][3]int{}

		for absences := 0; absences < 2; absences++ {
			for lateStreak := 0; lateStreak < 3; lateStreak++ {
				current := dp[absences][lateStreak]

				if current == 0 {
					continue
				}

				// Add 'P'
				nextDP[absences][0] =
					(nextDP[absences][0] + current) % MOD

				// Add 'A'
				if absences == 0 {
					nextDP[1][0] =
						(nextDP[1][0] + current) % MOD
				}

				// Add 'L'
				if lateStreak < 2 {
					nextDP[absences][lateStreak+1] =
						(nextDP[absences][lateStreak+1] + current) % MOD
				}
			}
		}

		dp = nextDP
	}

	result := 0

	for absences := 0; absences < 2; absences++ {
		for lateStreak := 0; lateStreak < 3; lateStreak++ {
			result = (result + dp[absences][lateStreak]) % MOD
		}
	}

	return result
}
```

The Go implementation mirrors the Python solution closely. Since Go supports fixed-size arrays efficiently, we use `[2][3]int` arrays instead of slices. This keeps memory usage minimal and avoids unnecessary allocations.

Modulo arithmetic is especially important in Go because integer overflow can occur if values become too large. Every update applies `% MOD` immediately.

## Edge Cases

One important edge case is `n = 1`. This is the smallest possible input, and the valid records are simply `"P"`, `"A"`, and `"L"`. A buggy implementation might incorrectly initialize the DP table or mishandle the base state. Our implementation correctly starts with the empty sequence and builds valid length-1 sequences through transitions.

Another critical edge case involves consecutive late days. Sequences ending with `"LL"` are still valid, but adding another `'L'` would create `"LLL"`, which is forbidden. A common mistake is allowing unlimited late streaks or resetting the streak incorrectly. Our DP explicitly tracks late streaks only up to length 2 and prevents transitions beyond that.

A third important edge case is handling one absence correctly. Sequences may contain exactly one `'A'`, but never two. An incorrect implementation might accidentally allow multiple absences when transitioning states. Our solution prevents this by only allowing an `'A'` transition when the current absence count is 0.

Finally, very large inputs such as `n = 100000` are a major stress case. Recursive or exponential solutions would fail due to time or memory limits. Our implementation uses iterative DP with only constant space, making it efficient even at the maximum constraint.
