## Solution

## Problem Understanding

This problem asks us to determine whether an entire input string `s` matches a wildcard pattern `p`. Unlike substring matching problems, partial matches are not allowed. Every character in `s` must be consumed by the pattern, and every relevant character in the pattern must participate in the match.

The pattern supports two special wildcard characters:

- `'?'` matches exactly one arbitrary character.
- `'*'` matches any sequence of characters, including an empty sequence.

All other characters are treated as literal characters and must match exactly.

For example:

- `"a"` does not match `"aa"` because the pattern only covers one character.
- `"*"` matches `"aa"` because `'*'` can represent the entire string.
- `"?a"` does not match `"cb"` because although `'?'` matches `'c'`, the literal `'a'` does not match `'b'`.

The constraints are important:

- `0 <= s.length, p.length <= 2000`

A brute force recursive solution would explode exponentially because every `'*'` can branch into many possibilities. Since both strings can be length 2000, we need an algorithm that is close to linear or quadratic time.

Several edge cases immediately stand out:

- Empty string with empty pattern.
- Empty string with patterns like `"*"`, `"**"`, or `"***"`.
- Consecutive `'*'` characters.
- Patterns longer than the string but containing `'*'`.
- Strings requiring backtracking when a previous `'*'` consumed too many or too few characters.

A naive greedy approach without proper backtracking can fail on these tricky cases.

## Approaches

### Brute Force Recursive Backtracking

The most direct approach is recursive exploration.

At every position `(i, j)`:

- If `p[j]` is a normal character, it must equal `s[i]`.
- If `p[j] == '?'`, we consume one character from both strings.
- If `p[j] == '*'`, we have many possibilities:

- Match zero characters.
- Match one character.
- Match two characters.
- Continue until the end of the string.

This guarantees correctness because it explores every possible interpretation of `'*'`.

However, the same states are recomputed repeatedly. Consider patterns with multiple stars:

```
s = "aaaaaaaaaa"
p = "*a*a*a*a*"
```

The recursion tree becomes enormous because each `'*'` creates branching choices.

Without memoization, the complexity becomes exponential.

### Dynamic Programming and Greedy Optimization

A standard optimization is dynamic programming, where we define:

```
dp[i][j] = whether s[0:i] matches p[0:j]
```

This reduces repeated computation and guarantees correctness in `O(m * n)` time.

However, there is an even better practical approach using greedy matching with controlled backtracking.

The key insight is that `'*'` is the only wildcard that introduces ambiguity. When a mismatch occurs, the only meaningful recovery option is to revisit the most recent `'*'` and allow it to consume one additional character.

This observation allows us to process the strings with two pointers while remembering:

- The position of the last `'*'`
- The position in `s` where that `'*'` started matching

Instead of exploring all possibilities recursively, we incrementally expand the match range of the most recent `'*'`.

This gives a linear-time solution.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | O(m + n) | Explores every possible interpretation of `'*'` |
| Optimal Greedy | O(m + n) | O(1) | Uses two pointers with controlled backtracking |

## Algorithm Walkthrough

### Greedy Two Pointer Algorithm

1. Initialize two pointers:

- `s_index` for traversing the input string.
- `p_index` for traversing the pattern.
2. Maintain two additional variables:

- `star_index`, the most recent position of `'*'` in the pattern.
- `match_index`, the position in `s` corresponding to the start of the `'*'` match.
3. If the current pattern character matches:

- Either characters are equal,
- Or pattern contains `'?'`,

then advance both pointers.
4. If the current pattern character is `'*'`:

- Record its position in `star_index`.
- Record the current string position in `match_index`.
- Advance the pattern pointer.

This initially assumes the `'*'` matches an empty sequence.
5. If characters do not match:

- If we previously encountered a `'*'`,

then backtrack:

- Reset pattern pointer to the character after `'*'`.
- Increase the number of characters consumed by `'*'`.
- Move string pointer accordingly.

This effectively allows `'*'` to absorb one more character.
6. If no previous `'*'` exists and characters mismatch:

- Matching fails immediately.
7. After processing the string:

- Remaining pattern characters must all be `'*'`.
- Any other remaining character causes failure.
8. If all conditions succeed:

- Return `True`.

This works because `'*'` is the only character capable of flexibly adjusting match length.

## Python Solution

```
class Solution:
    def isMatch(self, s: str, p: str) -> bool:
        s_index = 0
        p_index = 0

        star_index = -1
        match_index = -1

        while s_index < len(s):
            if (
                p_index < len(p)
                and (p[p_index] == s[s_index] or p[p_index] == '?')
            ):
                s_index += 1
                p_index += 1

            elif p_index < len(p) and p[p_index] == '*':
                star_index = p_index
                match_index = s_index
                p_index += 1

            elif star_index != -1:
                p_index = star_index + 1
                match_index += 1
                s_index = match_index

            else:
                return False

        while p_index < len(p) and p[p_index] == '*':
            p_index += 1

        return p_index == len(p)
```

The implementation uses four integer variables instead of recursion or dynamic programming tables.

The main loop processes the string from left to right. When characters match normally, both pointers advance.

When a `'*'` appears, its position is stored. The algorithm initially assumes the `'*'` matches nothing.

If a later mismatch occurs, the algorithm revisits the most recent `'*'` and expands its coverage by one additional character. This is the core greedy insight.

The final loop handles trailing `'*'` characters because they can legally represent empty sequences.

If any non-star pattern characters remain, the match fails.

## Go Solution

```
func isMatch(s string, p string) bool {
	sIndex := 0
	pIndex := 0

	starIndex := -1
	matchIndex := -1

	for sIndex < len(s) {
		if pIndex < len(p) &&
			(p[pIndex] == s[sIndex] || p[pIndex] == '?') {

			sIndex++
			pIndex++

		} else if pIndex < len(p) && p[pIndex] == '*' {

			starIndex = pIndex
			matchIndex = sIndex
			pIndex++

		} else if starIndex != -1 {

			pIndex = starIndex + 1
			matchIndex++
			sIndex = matchIndex

		} else {
			return false
		}
	}

	for pIndex < len(p) && p[pIndex] == '*' {
		pIndex++
	}

	return pIndex == len(p)
}
```

The Go implementation closely mirrors the Python solution.

Strings in Go are byte indexed, which works correctly here because the problem guarantees lowercase English letters and wildcard symbols, all of which are single byte ASCII characters.

The algorithm uses integer indices instead of slices or recursion, keeping memory usage constant.

No overflow concerns exist because indices are bounded by length 2000.

s = "aa"
p = "a"

| --- | --- | --- | --- | --- |
| 1 | 0 | 0 | `'a' == 'a'` | Advance both |
| 2 | 1 | 1 | Pattern ended | Mismatch |

The string still contains one character, but the pattern is exhausted.

```

```
s = "aa"

```

| Step | s_index | p_index | Action |
| --- | --- | --- | --- |
| 1 | 0 | 0 | Record `'*'` |
| 2 | 0 | 1 | Pattern exhausted |
| 3 | 1 | 1 | Backtrack, `'*'` consumes one char |
| 4 | 2 | 1 | Backtrack again |

Now the string is fully consumed.

Remaining pattern contains nothing.

Result:

```
True
```

```

| Step | s_index | p_index | Current Match | Action |
| --- | --- | --- | --- | --- |
| 1 | 0 | 0 | `'?'` matches `'c'` | Advance |
| 2 | 1 | 1 | `'a' != 'b'` | Fail |

No previous `'*'` exists for recovery.

Result:

```
False
```

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m + n) | Each pointer moves forward at most linearly |
| Space | O(1) | Only a few integer variables are stored |

Although backtracking occurs, the algorithm never revisits pattern positions arbitrarily many times. Each character in the string is processed at most a constant number of times, giving linear overall complexity.

This is significantly better than dynamic programming for large inputs.

## Edge Cases

### Empty String and Empty Pattern

If both `s` and `p` are empty, the result should be `True` because an empty pattern fully matches an empty string.

The implementation handles this naturally because the main loop never executes, and the final condition verifies that the pattern has been fully consumed.

### Pattern Contains Only Stars

Patterns like `"*"`, `"**"`, or `"*****"` should match any string, including the empty string.

This can be a source of bugs because some implementations incorrectly require stars to consume at least one character.

The solution correctly handles this because:

- `'*'` initially matches an empty sequence.
- The final cleanup loop skips trailing stars.

### Mismatch After Star Expansion

Consider:

```
s = "adceb"
p = "*a*b"
```

A naive greedy solution might let the first `'*'` consume too many characters and become stuck.

The implemented algorithm avoids this problem by storing the most recent star position and incrementally expanding its match only when necessary.

This controlled backtracking guarantees correctness without exponential recursion.

### Consecutive Stars

Patterns like:

```
"a**b***c"
```

are logically equivalent to:

```
"a*b*c"
```

Some implementations mishandle repeated stars and create redundant recursion.

This algorithm works correctly because each `'*'` is simply recorded as the latest wildcard position. Consecutive stars behave naturally without requiring special preprocessing.
