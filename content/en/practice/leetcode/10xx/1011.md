## Solution

## Problem Understanding

This problem asks us to determine the minimum ship capacity required to transport all packages within a fixed number of days.

We are given an array `weights`, where `weights[i]` represents the weight of the `i`th package on a conveyor belt. Packages must be shipped in the exact order they appear in the array. This ordering constraint is extremely important because it prevents us from arbitrarily rearranging packages to better balance daily loads.

Each day, we load packages onto the ship until adding another package would exceed the ship's capacity. At that point, we stop for the day and continue with the remaining packages on the next day.

The goal is to return the smallest ship capacity that allows all packages to be shipped within exactly `days` days or fewer.

The constraints provide important insight into the expected solution complexity:

- `weights.length` can be as large as `5 * 10^4`
- Each weight can be up to `500`
- A naive simulation over all possible capacities could become too slow

Since the input size is large, especially the number of packages, we need something significantly better than quadratic time.

Several edge cases are important to recognize immediately:

- If `days == 1`, the ship must carry all packages in a single day, so the answer is the sum of all weights.
- If `days == len(weights)`, we can ship one package per day, so the answer is simply the maximum package weight.
- A ship capacity smaller than the heaviest package is impossible because packages cannot be split.
- The ordering constraint means greedy packing must preserve the original sequence.

## Approaches

### Brute Force Approach

A brute force solution would try every possible ship capacity starting from the heaviest package weight and going up to the sum of all weights.

For each candidate capacity, we simulate the shipping process:

- Start loading packages for the current day
- If adding a package exceeds capacity, move to the next day
- Count how many days are required
- The first capacity that ships everything within `days` is the answer

This approach is correct because it explicitly tests every feasible capacity in increasing order. The first valid one must therefore be the minimum.

However, this method is too slow for the constraints. The range of capacities can be very large:

- Minimum possible capacity = `max(weights)`
- Maximum possible capacity = `sum(weights)`

In the worst case:

- `sum(weights)` can be `5 * 10^4 * 500 = 25,000,000`

Testing millions of capacities individually would be far too expensive.

### Optimal Approach, Binary Search on Answer

The key insight is that the problem has a monotonic property.

If a ship capacity `C` is sufficient to ship all packages within `days`, then any capacity larger than `C` will also work.

Similarly:

- Capacities that are too small form one region
- Capacities that work form another region

This creates a perfect binary search condition.

Instead of testing every capacity one by one, we binary search over the range:

- Left bound = `max(weights)`
- Right bound = `sum(weights)`

For each midpoint capacity:

- Simulate shipping greedily
- Determine how many days are needed
- If we can finish within `days`, try a smaller capacity
- Otherwise, increase the capacity

This reduces the search space dramatically.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * (sum(weights) - max(weights))) | O(1) | Tests every possible capacity individually |
| Optimal | O(n * log(sum(weights))) | O(1) | Uses binary search on the answer space |

## Algorithm Walkthrough

1. Determine the search boundaries.

The minimum possible capacity must be at least the heaviest package because a package cannot be split across days. Therefore:

```
left = max(weights)
```

The maximum possible capacity is the sum of all package weights because that would allow shipping everything in one day:

```
right = sum(weights)
```
2. Perform binary search on the capacity range.

We repeatedly test the midpoint capacity:

```
mid = (left + right) // 2
```

This midpoint represents a candidate ship capacity.
3. Simulate shipping with the candidate capacity.

We iterate through the packages in order.

- Maintain the current load for the day
- If adding the next package exceeds capacity, start a new day
- Otherwise, continue loading packages

This greedy strategy is optimal because we always pack as many packages as possible before moving to the next day.
4. Count the number of days required.

During simulation:

- Start with `days_needed = 1`
- Every time capacity is exceeded, increment the day count

At the end, compare `days_needed` with the target `days`.
5. Adjust the binary search boundaries.

If the candidate capacity works:

```
days_needed <= days
```

then we try to find an even smaller valid capacity:

```
right = mid
```

Otherwise, the capacity is too small:

```
left = mid + 1
```
6. Continue until the search converges.

The binary search stops when:

```
left == right
```

At this point, we have found the minimum valid capacity.

### Why it works

The correctness relies on the monotonic nature of the search space.

If a capacity can ship all packages within the required number of days, then every larger capacity can also do so. This means the valid capacities form a contiguous range.

Binary search efficiently locates the first valid capacity in that range. The greedy simulation is also correct because shipping more packages earlier can never increase the total number of days required.

## Python Solution

```
from typing import List

class Solution:
    def shipWithinDays(self, weights: List[int], days: int) -> int:
        def can_ship(capacity: int) -> bool:
            days_needed = 1
            current_load = 0

            for weight in weights:
                if current_load + weight > capacity:
                    days_needed += 1
                    current_load = 0

                current_load += weight

            return days_needed <= days

        left = max(weights)
        right = sum(weights)

        while left < right:
            mid = (left + right) // 2

            if can_ship(mid):
                right = mid
            else:
                left = mid + 1

        return left
```

The implementation begins by defining a helper function called `can_ship`. This function determines whether a given ship capacity is sufficient.

Inside the helper function:

- `days_needed` starts at `1` because shipping always begins on the first day
- `current_load` tracks the total weight loaded for the current day

As we iterate through the packages:

- If adding the next package would exceed the candidate capacity, we start a new day
- Otherwise, we continue filling the current day

At the end of the simulation, we return whether the required number of days is within the allowed limit.

The main algorithm initializes the binary search bounds:

- `left` is the maximum package weight
- `right` is the sum of all package weights

The binary search repeatedly tests the midpoint capacity.

If the midpoint works, we move the right boundary leftward because we want the minimum valid capacity.

If the midpoint fails, we move the left boundary upward because larger capacities are required.

Eventually, both boundaries converge to the smallest feasible capacity.

## Go Solution

```
func shipWithinDays(weights []int, days int) int {
	canShip := func(capacity int) bool {
		daysNeeded := 1
		currentLoad := 0

		for _, weight := range weights {
			if currentLoad+weight > capacity {
				daysNeeded++
				currentLoad = 0
			}

			currentLoad += weight
		}

		return daysNeeded <= days
	}

	left := 0
	right := 0

	for _, weight := range weights {
		if weight > left {
			left = weight
		}
		right += weight
	}

	for left < right {
		mid := left + (right-left)/2

		if canShip(mid) {
			right = mid
		} else {
			left = mid + 1
		}
	}

	return left
}
```

The Go implementation follows the same logic as the Python solution.

One small difference is that Go does not provide built-in `max()` and `sum()` helpers for slices, so we compute both values manually in a loop.

The binary search midpoint is calculated as:

```
mid := left + (right-left)/2
```

This form avoids potential integer overflow, although overflow is not actually a concern under this problem's constraints.

Go slices are used directly without any additional memory allocation, so the space complexity remains constant.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n * log(sum(weights))) | Binary search performs O(log(sum(weights))) iterations, each scanning all packages |
| Space | O(1) | Only a few variables are used |

The binary search reduces the candidate capacity range exponentially. For each midpoint capacity, we perform a linear pass through the weights array to simulate shipping.

Since the search space ranges from `max(weights)` to `sum(weights)`, the number of binary search iterations is logarithmic in the total weight sum.

No auxiliary data structures proportional to input size are used, so the space complexity remains constant.

## Edge Cases

One important edge case occurs when `days == len(weights)`. In this scenario, we can ship exactly one package per day. The minimum valid capacity is therefore the heaviest package. A buggy implementation might incorrectly return a smaller value if it does not properly enforce that every package must fit individually. Our implementation avoids this because the binary search lower bound is initialized to `max(weights)`.

Another critical edge case happens when `days == 1`. Here, all packages must be shipped together in a single day. The required capacity becomes the sum of all weights. Some incorrect solutions underestimate the capacity by splitting packages across multiple simulated days even though only one day is allowed. Our simulation correctly counts days and forces the binary search to eventually converge to the total sum.

A subtle edge case involves package ordering. Since packages must remain in order, we cannot rearrange them to improve balancing. For example:

```
weights = [10, 1, 10]
days = 2
```

A capacity of `11` works because we can ship `[10,1]` and `[10]`. However, a capacity smaller than `10` is impossible, and rearranging packages is forbidden. Our greedy simulation strictly processes packages sequentially, preserving the required order.

Another edge case involves repeated identical weights. Arrays like:

```
[5,5,5,5,5]
```

can expose off-by-one mistakes in day counting. If the implementation incorrectly increments the day count too early or too late, it may accept invalid capacities. Our approach only increments the day count when adding the next package would exceed capacity, which precisely matches the problem definition.
