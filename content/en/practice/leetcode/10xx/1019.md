## Solution

## Problem Understanding

The problem asks us to process a singly linked list and determine, for every node, the value of the first node that appears later in the list and has a strictly greater value. If no such node exists, the result for that position should be `0`.

More formally, suppose we are given a linked list of length `n`. For each node at index `i`, we want to find the first node at some index `j > i` where:

```
node[j].val > node[i].val
```

If such a node exists, we record `node[j].val` as the answer for position `i`. Otherwise, we record `0`.

The input is provided as the `head` of a singly linked list. Since linked lists only allow sequential traversal and do not support direct indexing, we cannot immediately jump forward or backward between elements like we could with an array.

The output is an integer array of length `n`, where each element corresponds to the next greater node value for the node at the same position in the linked list.

For example, consider:

```
[2,1,5]
```

For the node with value `2`, the next greater node is `5`.

For the node with value `1`, the next greater node is also `5`.

For the node with value `5`, there is no greater value afterward, so the answer is `0`.

Thus the result becomes:

```
[5,5,0]
```

The constraints provide important hints about the expected efficiency:

- `1 <= n <= 10^4`
- `1 <= Node.val <= 10^9`

A linked list of up to `10^4` nodes is large enough that an `O(n^2)` brute force solution may become inefficient in worst case scenarios. Since values can be very large, we cannot rely on counting sort or frequency based optimizations. This strongly suggests we need a linear or near-linear time solution.

Several edge cases are important to consider upfront. A linked list containing only one node should always return `[0]`, since there are no future nodes. A strictly decreasing list, such as `[5,4,3,2]`, should produce all zeros because no node ever finds a larger value. A strictly increasing list, such as `[1,2,3,4]`, should always return the next immediate value except for the final node. Repeated values also matter because the problem requires a **strictly larger** node, meaning equal values do not count.

## Approaches

### Brute Force Approach

The most direct solution is to examine every node individually and search forward until we find a larger value.

For each node, we start from the next node and continue traversing until one of two things happens:

1. We find a node with a strictly larger value.
2. We reach the end of the list.

If we find a larger value, we record it. Otherwise, we record `0`.

This method is straightforward and guarantees correctness because it explicitly checks every future possibility for every node. However, it performs unnecessary repeated work. If we repeatedly scan forward from each node, many nodes are revisited multiple times.

Consider a decreasing list:

```
[5,4,3,2,1]
```

For the first node, we scan the entire remaining list. For the second node, we scan nearly the entire remainder again. This repeated scanning creates quadratic behavior.

Since `n` can be `10^4`, an `O(n^2)` solution may perform roughly:

```
10^8 operations
```

in the worst case, which is inefficient.

### Optimal Approach, Monotonic Stack

The key observation is that we do not actually need to repeatedly scan forward from every node.

Instead, we can process the values once and maintain a **monotonic decreasing stack** of unresolved nodes. The stack stores nodes whose next greater value has not yet been found.

However, linked lists do not support indexing, while stacks work naturally with indices. Therefore, the first step is converting the linked list into an array of values.

Once we have the array, we iterate through it from left to right.

At each value:

- If the current value is greater than values waiting on the stack, we resolve them immediately.
- We continue popping until the stack becomes empty or the top value is greater than or equal to the current value.
- Then we push the current index onto the stack.

The stack remains **monotonically decreasing**, meaning values represented in the stack decrease from bottom to top.

This works because the first time we encounter a larger value for a node, it must be the correct answer.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Scan forward from every node |
| Optimal | O(n) | O(n) | Convert to array and use monotonic decreasing stack |

## Algorithm Walkthrough

### Optimal Algorithm

1. Traverse the linked list and copy all node values into an array.

We do this because linked lists do not allow random access by index. Since a monotonic stack works best when we can revisit positions efficiently, converting to an array simplifies the problem.
2. Initialize an answer array filled with zeros.

Every node defaults to `0`, since if no greater node exists, the answer should remain zero.
3. Create an empty stack.

The stack will store indices of nodes whose next greater value has not yet been found.
4. Iterate through the array from left to right.

For each position `i`, examine `values[i]`.
5. While the stack is not empty and the current value is greater than the value at the top index of the stack:

This means we have found the next greater node for the unresolved node at the stack top.

Update:

```
answer[stack_top] = current_value
```

Then remove that index from the stack.
6. Push the current index onto the stack.

This node may need a greater value later, so we keep it unresolved for now.
7. Continue until all values are processed.
8. Return the answer array.

Any indices left in the stack never found a greater node, so their values remain `0`.

### Why it works

The algorithm maintains an important invariant: the stack always contains indices of nodes whose next greater value has not yet been found, and their corresponding values form a decreasing sequence.

Whenever a larger value appears, it resolves every smaller unresolved node immediately. Since we process elements from left to right, the first larger value encountered must be the correct next greater node. No earlier valid candidate could have been skipped, because it would already have been processed.

This guarantees correctness while ensuring every index enters and leaves the stack at most once.

## Python Solution

```
# Definition for singly-linked list.
# class ListNode:
#     def __init__(self, val=0, next=None):
#         self.val = val
#         self.next = next

from typing import Optional, List

class Solution:
    def nextLargerNodes(self, head: Optional[ListNode]) -> List[int]:
        values = []

        current = head
        while current:
            values.append(current.val)
            current = current.next

        result = [0] * len(values)
        stack = []

        for index, value in enumerate(values):
            while stack and values[stack[-1]] < value:
                previous_index = stack.pop()
                result[previous_index] = value

            stack.append(index)

        return result
```

The implementation begins by traversing the linked list and storing every node value in an array called `values`. This transformation is necessary because linked lists do not support indexing, while the monotonic stack algorithm relies on quickly accessing previous elements.

Next, we initialize the `result` array with zeros. This automatically handles nodes that never find a larger value.

The `stack` stores unresolved indices. During iteration, whenever the current value exceeds the value at the top stack index, we know we have discovered the next greater node for that earlier element. We repeatedly pop smaller elements and assign the current value as their answer.

Finally, we push the current index because its next greater value may appear later.

Each index is pushed and popped at most once, which keeps the algorithm linear.

## Go Solution

```
/**
 * Definition for singly-linked list.
 * type ListNode struct {
 *     Val int
 *     Next *ListNode
 * }
 */
func nextLargerNodes(head *ListNode) []int {
	values := []int{}

	current := head
	for current != nil {
		values = append(values, current.Val)
		current = current.Next
	}

	result := make([]int, len(values))
	stack := []int{}

	for index, value := range values {
		for len(stack) > 0 && values[stack[len(stack)-1]] < value {
			previousIndex := stack[len(stack)-1]
			stack = stack[:len(stack)-1]

			result[previousIndex] = value
		}

		stack = append(stack, index)
	}

	return result
}
```

The Go implementation follows the same logic as the Python version. Since Go slices dynamically resize, they naturally replace Python lists.

The linked list traversal uses `nil` checks rather than Python's truthy behavior. Stack operations are implemented using slices, where removing the last element is done with:

```
stack = stack[:len(stack)-1]
```

Integer overflow is not a concern here because Go's `int` comfortably handles values up to `10^9`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each node index is pushed and popped at most once |
| Space | O(n) | Extra space for values array, result array, and stack |

Although there is a nested `while` loop, the total number of stack pops across the entire execution is at most `n`. Each element enters the stack once and leaves once, which makes the total work linear rather than quadratic.

The space complexity is `O(n)` because we store the linked list values in an array, maintain a result array, and use a stack that can grow to size `n` in the worst case.

## Edge Cases

### Single Node Linked List

A linked list containing only one node is an important edge case because there are no later nodes to compare against. A buggy implementation might accidentally access invalid indices or assume another node exists.

For example:

```
[7]
```

The correct output is:

```
[0]
```

Our implementation handles this naturally because the stack simply stores the single index and no larger value is ever found.

### Strictly Decreasing Sequence

A decreasing sequence can expose inefficiencies or logic errors.

For example:

```
[9,8,7,6]
```

No node has a larger value afterward, so the answer must be:

```
[0,0,0,0]
```

In the monotonic stack approach, every element is pushed onto the stack but never popped. Since the result array is initialized with zeros, unresolved nodes automatically receive the correct output.

### Duplicate Values

Equal values are tricky because the problem requires a **strictly greater** value.

For example:

```
[2,2,3]
```

The first `2` should not consider the second `2` as valid. Both `2`s should instead resolve to `3`.

The implementation handles this correctly by using:

```
values[stack[-1]] < value
```

instead of:

```
values[stack[-1]] <= value
```

This ensures only strictly larger values trigger stack pops.

### Large Increasing Sequence

A fully increasing sequence stresses whether the stack resolves multiple elements correctly.

For example:

```
[1,2,3,4,5]
```

Every element except the last should resolve immediately to the next value.

The monotonic stack efficiently processes this case in linear time, popping previously unresolved elements as soon as a larger value appears.
