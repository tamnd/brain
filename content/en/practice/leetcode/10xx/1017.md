## Solution

## Problem Understanding

This problem asks us to convert a non-negative integer `n` into its representation using **base -2**, instead of the standard positive base systems such as base 2 or base 10.

In a normal binary representation, each digit position represents a power of `2`:

$$2^0, 2^1, 2^2, 2^3, \dots$$

For example:

$$110_2 = 1 \times 2^2 + 1 \times 2^1 + 0 \times 2^0 = 6$$

In this problem, the base is **-2**, meaning each digit position represents powers of `-2`:

$$(-2)^0, (-2)^1, (-2)^2, (-2)^3, \dots$$

The signs alternate naturally:

$$1, -2, 4, -8, 16, -32, \dots$$

For example:

$$110_{-2} = 1 \times (-2)^2 + 1 \times (-2)^1 + 0 \times (-2)^0$$

$$= 4 + (-2) + 0 = 2$$

The task is to return a **string** containing the base `-2` representation of `n`.

There are two important formatting requirements:

- The representation must contain only binary digits, `0` and `1`.
- The result must not contain leading zeros unless the answer is exactly `"0"`.

The constraints tell us that:

$$0 \le n \le 10^9$$

This is small enough that logarithmic algorithms are more than sufficient. Since every division by `-2` reduces the number's magnitude significantly, the number of digits in the representation will be proportional to:

$$O(\log n)$$

An important detail is that `n` can be `0`. This is a special case because repeated division logic would otherwise produce an empty string. The correct answer for `0` is simply `"0"`.

Another subtle issue comes from dividing by a **negative base**. In ordinary base conversion, we repeatedly divide by the base and take remainders. However, negative division introduces complications because remainders can become negative. Since valid digits must only be `0` or `1`, we must carefully adjust the quotient and remainder when needed.

## Approaches

### Brute Force Approach

A brute-force approach would attempt to generate candidate binary strings and check whether they evaluate to `n` in base `-2`.

For example, we could try all binary strings of increasing length, compute their decimal value under powers of `-2`, and stop once we find a match.

To verify a candidate string, we would compute:

$$\sum digit_i \times (-2)^i$$

This method is correct because it eventually enumerates every valid representation and checks whether it equals `n`.

However, this approach is highly inefficient. The number of candidate binary strings grows exponentially with the length of the representation. Even though the answer length is only logarithmic in `n`, the search space becomes prohibitively large.

A smarter solution is needed.

### Optimal Approach, Repeated Division by Base -2

The key insight is that standard base conversion still works for a negative base, with one adjustment.

In ordinary base conversion, we repeatedly:

1. Take the remainder.
2. Divide by the base.
3. Continue until the quotient becomes zero.

For base `-2`, we still divide repeatedly, but we must ensure the remainder is always either `0` or `1`.

Suppose:

$$n = q \times (-2) + r$$

where `r` must be one of:

$$0 \text{ or } 1$$

If the remainder becomes negative, we can fix it by:

- Adding `2` to the remainder
- Incrementing the quotient

This preserves correctness because:

$$q(-2) + r
=
(q+1)(-2) + (r+2)$$

After adjustment, the remainder becomes valid again.

Since each division reduces the number significantly, the algorithm runs efficiently in logarithmic time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^log n) = O(n) | O(log n) | Enumerates candidate representations until finding a match |
| Optimal | O(log n) | O(log n) | Repeated division by `-2` with remainder correction |

## Algorithm Walkthrough

1. Handle the special case where `n == 0`.

Since the representation of zero in any positional system is simply `"0"`, we return it immediately. Without this step, the loop would produce an empty result.
2. Create a list to store digits.

We build the representation from least significant digit to most significant digit, exactly like ordinary base conversion. A list allows efficient appending.
3. Repeatedly divide by `-2`.

While `n` is not zero, compute:

```
remainder = n % -2
quotient = n // -2
```

However, because the divisor is negative, Python may produce a negative remainder.
4. Fix negative remainders.

If the remainder becomes `-1`, it is invalid because base `-2` digits must only be `0` or `1`.

We correct it by:

```
remainder += 2
quotient += 1
```

This preserves the equation:

```
n = quotient × (-2) + remainder
```
5. Append the digit.

Add the remainder (`0` or `1`) to the result list.
6. Continue using the new quotient.

Set `n = quotient` and repeat.
7. Reverse the digits.

Since digits were generated from least significant to most significant, reverse them before returning.
8. Join the digits into a string.

Convert the digit list into the final binary string.

### Why it works

The algorithm maintains the invariant:

$$n = q \times (-2) + r$$

at every step, where `r` is always constrained to be `0` or `1`. Each iteration extracts one valid base `-2` digit while reducing the magnitude of `n`. Eventually, `n` becomes zero, meaning all positional contributions have been extracted. Reversing the collected digits reconstructs the unique valid base `-2` representation.

## Python Solution

```
class Solution:
    def baseNeg2(self, n: int) -> str:
        if n == 0:
            return "0"

        digits: list[str] = []

        while n != 0:
            remainder = n % -2
            n //= -2

            if remainder < 0:
                remainder += 2
                n += 1

            digits.append(str(remainder))

        return "".join(reversed(digits))
```

The implementation begins by handling the special case where `n` equals zero. This avoids returning an empty string.

We then create a `digits` list to store the extracted base `-2` digits. Since the least significant digit is computed first, appending to a list is efficient.

Inside the loop, we compute both the quotient and remainder using division by `-2`. Because Python allows negative remainders when dividing by a negative number, we check whether the remainder is negative. If it is, we adjust both the remainder and quotient to restore a valid digit.

Each valid remainder is appended to `digits`, and the process continues until `n` becomes zero.

Finally, since digits were collected in reverse order, we reverse the list and join it into the final string.

## Go Solution

```
func baseNeg2(n int) string {
	if n == 0 {
		return "0"
	}

	digits := make([]byte, 0)

	for n != 0 {
		remainder := n % -2
		n /= -2

		if remainder < 0 {
			remainder += 2
			n += 1
		}

		digits = append(digits, byte(remainder)+'0')
	}

	// Reverse digits
	for left, right := 0, len(digits)-1; left < right; left, right = left+1, right-1 {
		digits[left], digits[right] = digits[right], digits[left]
	}

	return string(digits)
}
```

The Go implementation follows the same logic as the Python solution. Since Go strings are immutable, we build the answer using a `[]byte` slice for efficiency.

Instead of converting integers to strings repeatedly, we directly convert digits into characters using:

```
byte(remainder) + '0'
```

After collecting digits, we reverse the slice in place using a two-pointer approach and return the resulting string.

Go integer arithmetic safely handles the given constraints because `n ≤ 10^9`, which is far below integer overflow limits.

## Edge Cases

### Edge Case 1, `n = 0`

This is the most important special case. Since the algorithm works by repeatedly dividing and collecting digits, a value of zero would never enter the loop, producing an empty string. That would be incorrect because the representation of zero must be `"0"`.

The implementation handles this explicitly with an early return.

### Edge Case 2, Negative Remainders During Division

When dividing by a negative base, some languages produce negative remainders. For example:

```
-1 % -2 = -1
```

A negative remainder is invalid because base `-2` digits can only be `0` or `1`.

The implementation fixes this by adding `2` to the remainder and incrementing the quotient. This preserves the mathematical value while ensuring the digit stays valid.

### Edge Case 3, Powers of Four

Numbers such as `4`, `16`, and `64` are interesting because they correspond directly to powers of `(-2)^2`, `(-2)^4`, and so on.

For example:

```
4 → "100"
```

A buggy implementation might accidentally introduce leading zeros or unnecessary digits. Because this solution terminates exactly when `n` becomes zero and only reverses collected digits, it naturally avoids leading zeros.

### Edge Case 4, Large Values Near `10^9`

Large numbers stress both runtime and correctness. A naive brute-force search would become impractical, but repeated division by `-2` still finishes quickly because the value shrinks every iteration.

Since the algorithm runs in logarithmic time, even the maximum allowed input is processed efficiently.
