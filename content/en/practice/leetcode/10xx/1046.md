## Solution

## Problem Understanding

The problem gives us an array called `stones`, where each value represents the weight of a stone. We repeatedly perform an operation where we select the two heaviest stones and smash them together.

If the two stones have equal weight, both are destroyed. If they have different weights, the lighter stone disappears and the heavier stone is reduced by the lighter stone's weight. The process continues until either one stone remains or all stones are destroyed.

The goal is to return the weight of the final remaining stone. If every stone gets destroyed, we return `0`.

For example, if we have stones `[2,7,4,1,8,1]`, the two heaviest stones are `8` and `7`. Smashing them leaves a new stone of weight `1`. The collection becomes `[2,4,1,1,1]`. We continue until only one stone remains.

The constraints are relatively small:

- The number of stones is between `1` and `30`
- Each stone weight is between `1` and `1000`

These constraints tell us that even less efficient solutions could pass, but the problem is clearly designed to test understanding of heaps or priority queues. The repeated requirement of finding the two largest elements strongly hints toward a max heap.

Several edge cases are important:

- A single stone, such as `[1]`, should immediately return that value.
- Two equal stones, such as `[5,5]`, should return `0`.
- Cases where repeated smashing produces new stones that must be reinserted correctly into the structure.
- Inputs where many stones have identical values.
- Inputs where every stone eventually cancels out.

A naive implementation can easily become inefficient if it repeatedly sorts the array after every smash operation.

## Approaches

### Brute Force Approach

The brute force solution repeatedly sorts the array so the two largest stones can be removed easily.

The process works like this:

1. Sort the stones array.
2. Remove the two largest elements.
3. Compute their difference.
4. If the difference is nonzero, insert it back into the array.
5. Repeat until at most one stone remains.

This approach is correct because every iteration explicitly follows the rules of the game. However, it is inefficient because sorting is expensive and must be repeated many times.

If there are `n` stones, sorting costs `O(n log n)` each iteration, and we may perform nearly `n` iterations. That leads to approximately `O(n^2 log n)` time complexity.

### Optimal Approach

The key observation is that we repeatedly need access to the largest elements. A heap is designed exactly for this type of operation.

Python's `heapq` module implements a min heap, but we can simulate a max heap by storing negative values.

With a max heap:

- Extracting the largest stone takes `O(log n)`
- Reinserting a new stone takes `O(log n)`
- Building the heap initially takes `O(n)`

This dramatically improves efficiency because we avoid repeatedly sorting the entire collection.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n² log n) | O(1) or O(n) | Repeatedly sorts the array after each smash |
| Optimal | O(n log n) | O(n) | Uses a max heap to efficiently retrieve largest stones |

## Algorithm Walkthrough

1. Convert all stone weights into negative values and build a heap from them.

Python's `heapq` is a min heap, meaning the smallest value is removed first. By storing negative numbers, the stone with the largest original weight becomes the smallest negative number, effectively simulating a max heap.
2. While the heap contains more than one stone, remove the two heaviest stones.

We use `heappop()` twice. Since values are stored as negatives, we negate them again to recover the original positive weights.
3. Compare the two stones.

If the weights are equal, both stones are destroyed and nothing is added back into the heap.
4. If the stones are different, compute the remaining weight.

The remaining stone has weight `largest - second_largest`.
5. Insert the remaining stone back into the heap.

Since we are simulating a max heap using negatives, we push the negative of the difference.
6. Continue until at most one stone remains.
7. If the heap is empty, return `0`. Otherwise, return the remaining stone's positive value.

### Why it works

The heap always maintains quick access to the current heaviest stones. At every iteration, the algorithm exactly follows the game rules by selecting the two largest available stones. Reinserting the remaining difference preserves the correct state for future operations. Because every smash operation is simulated faithfully, the final value produced by the algorithm must match the actual game outcome.

## Python Solution

```
from typing import List
import heapq

class Solution:
    def lastStoneWeight(self, stones: List[int]) -> int:
        # Convert to negative values to simulate a max heap
        max_heap = [-stone for stone in stones]
        heapq.heapify(max_heap)

        # Continue smashing stones until at most one remains
        while len(max_heap) > 1:
            heaviest = -heapq.heappop(max_heap)
            second_heaviest = -heapq.heappop(max_heap)

            # If the stones are different, push the remainder back
            if heaviest != second_heaviest:
                remaining = heaviest - second_heaviest
                heapq.heappush(max_heap, -remaining)

        # Return the remaining stone or 0 if none remain
        return -max_heap[0] if max_heap else 0
```

The implementation begins by converting every stone weight into a negative value. This allows Python's min heap to behave like a max heap.

`heapq.heapify()` transforms the list into a valid heap in linear time.

The main loop continues while at least two stones exist. The two largest stones are removed using `heappop()`. Since values are stored negatively, we negate them again to recover the original weights.

If the stones differ in weight, the remaining difference is pushed back into the heap. Equal stones are simply discarded.

At the end, if the heap still contains one element, we return its positive value. Otherwise, we return `0`.

## Go Solution

```
package main

import (
	"container/heap"
)

type MaxHeap []int

func (h MaxHeap) Len() int {
	return len(h)
}

func (h MaxHeap) Less(i, j int) bool {
	return h[i] > h[j]
}

func (h MaxHeap) Swap(i, j int) {
	h[i], h[j] = h[j], h[i]
}

func (h *MaxHeap) Push(x interface{}) {
	*h = append(*h, x.(int))
}

func (h *MaxHeap) Pop() interface{} {
	old := *h
	n := len(old)
	item := old[n-1]
	*h = old[:n-1]
	return item
}

func lastStoneWeight(stones []int) int {
	maxHeap := &MaxHeap{}

	for _, stone := range stones {
		*maxHeap = append(*maxHeap, stone)
	}

	heap.Init(maxHeap)

	for maxHeap.Len() > 1 {
		first := heap.Pop(maxHeap).(int)
		second := heap.Pop(maxHeap).(int)

		if first != second {
			heap.Push(maxHeap, first-second)
		}
	}

	if maxHeap.Len() == 0 {
		return 0
	}

	return heap.Pop(maxHeap).(int)
}
```

The Go implementation uses the `container/heap` package, which requires implementing the `heap.Interface`.

Unlike Python, Go allows us to define a true max heap directly by customizing the `Less()` method. Returning `h[i] > h[j]` ensures larger values have higher priority.

Go does not have Python's convenient built-in heap functions, so we must explicitly define `Push()` and `Pop()` methods for the custom heap type.

Integer overflow is not a concern because stone weights are very small relative to Go's integer limits.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | Each heap operation costs O(log n), and we perform at most n smash operations |
| Space | O(n) | The heap stores up to n stones |

The heap is built in `O(n)` time initially. Each iteration removes two stones and may insert one new stone. Heap insertion and removal both take `O(log n)` time. Since the number of stones decreases over time, the total number of heap operations is proportional to `n`, resulting in overall `O(n log n)` complexity.

## Edge Cases

A very important edge case is when the input contains only one stone. In this scenario, no smashing can occur because at least two stones are required for an operation. A careless implementation might still attempt to pop two values from the heap and crash. The provided solution handles this naturally because the loop only runs while the heap contains more than one element.

Another important case occurs when two stones have identical weights. According to the rules, both stones are destroyed and nothing should be inserted back into the heap. A buggy implementation might incorrectly insert a `0`, which would change the future behavior of the algorithm. The implementation avoids this by only reinserting when the weights differ.

A third edge case involves repeated creation of new stones. For example, `[10,4]` becomes `[6]` after one smash. The algorithm must correctly insert the new stone back into the heap so it can participate in future operations. Forgetting to reinsert the remainder would produce incorrect results whenever multiple rounds are required.

Another subtle edge case is when all stones eventually cancel out completely, such as `[3,3,3,3]`. The heap eventually becomes empty, and the algorithm must return `0` instead of attempting to access a nonexistent heap element. The final return statement explicitly checks whether the heap is empty before accessing the remaining value.
