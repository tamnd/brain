## Solution

## Problem Understanding

The problem asks us to find the length of the longest arithmetic subsequence in a given integer array.

A subsequence is formed by deleting zero or more elements from the array while preserving the original order of the remaining elements. Unlike a subarray, the elements do not need to be contiguous.

An arithmetic sequence is a sequence where the difference between every pair of consecutive elements is the same. For example:

- `[3, 6, 9, 12]` has a common difference of `3`
- `[20, 15, 10, 5]` has a common difference of `-5`

We are given an array `nums`, and we must determine the maximum possible length of any arithmetic subsequence that can be formed from it.

The constraints are important:

- `2 <= nums.length <= 1000`
- `0 <= nums[i] <= 500`

The array length of `1000` is large enough that brute force enumeration of all subsequences is infeasible. The value range is relatively small, but the main challenge is efficiently tracking arithmetic progressions across many index pairs.

An important detail is that arithmetic subsequences are defined by both:

1. Their ending position
2. Their common difference

This means we cannot simply track lengths globally. Two subsequences ending at the same index may have different differences and therefore represent completely different arithmetic progressions.

Several edge cases are worth identifying early:

- Arrays where all numbers are equal, such as `[5,5,5,5]`
- Arrays with negative differences, such as `[9,7,5,3]`
- Arrays with multiple competing arithmetic subsequences
- Very short arrays of length `2`, where the answer is always `2`
- Arrays where the best subsequence skips many elements

The problem guarantees at least two elements, so the minimum valid answer is always `2`.

## Approaches

### Brute Force Approach

A brute force solution would attempt to generate subsequences and check whether each one forms an arithmetic sequence.

One possible implementation is:

1. Choose any subset of elements while preserving order
2. Check whether consecutive differences are identical
3. Track the maximum valid length

This approach is correct because it explicitly examines every possible subsequence.

However, the number of subsequences in an array of length `n` is `2^n`. With `n = 1000`, this becomes astronomically large and completely infeasible.

Even a more optimized brute force strategy that fixes pairs of starting elements and greedily searches for continuations still becomes extremely expensive because many overlapping states are recomputed repeatedly.

### Key Insight

The key observation is that an arithmetic subsequence is fully characterized by:

- Its ending index
- Its common difference

Suppose we already know the longest arithmetic subsequence ending at index `j` with difference `d`.

If:

```
nums[i] - nums[j] = d
```

then we can extend that subsequence by appending `nums[i]`.

This naturally leads to dynamic programming.

We define:

```
dp[i][d]
```

as the length of the longest arithmetic subsequence ending at index `i` with common difference `d`.

For every pair `(j, i)` where `j < i`:

```
difference = nums[i] - nums[j]
```

We can extend the sequence ending at `j`:

```
dp[i][difference] = dp[j][difference] + 1
```

If no previous sequence exists for that difference, then the pair itself forms a sequence of length `2`.

This avoids recomputation and efficiently builds solutions incrementally.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(n) | Enumerates subsequences explicitly |
| Optimal Dynamic Programming | O(n^2) | O(n^2) | Tracks arithmetic subsequences by ending index and difference |

## Algorithm Walkthrough

1. Create a dynamic programming array called `dp`.

Each `dp[i]` is a hash map.

The key represents a difference value, and the value represents the length of the longest arithmetic subsequence ending at index `i` with that difference.
2. Initialize the answer to `2`.

Since any two elements form an arithmetic subsequence, the minimum valid answer is always `2`.
3. Iterate through all pairs of indices `(j, i)` where `j < i`.

This ensures we only build subsequences that preserve order.
4. Compute the difference between the two numbers.

```
difference = nums[i] - nums[j]
```
5. Check whether there is already an arithmetic subsequence ending at `j` with this difference.

If it exists, we can extend it:

```
new_length = dp[j][difference] + 1
```

Otherwise, the pair `(nums[j], nums[i])` itself forms a new arithmetic subsequence of length `2`.
6. Store the best length for this difference at index `i`.

```
dp[i][difference] = max(existing_value, new_length)
```
7. Continuously update the global maximum answer.
8. After processing all pairs, return the maximum length found.

### Why it works

The dynamic programming state captures all information necessary to extend arithmetic subsequences correctly.

For any arithmetic subsequence ending at index `j` with difference `d`, appending `nums[i]` where:

```
nums[i] - nums[j] = d
```

preserves the arithmetic property.

Because every valid subsequence must end at some index with some difference, and because every pair `(j, i)` is processed exactly once, the algorithm explores all possible arithmetic subsequences and computes the optimal length.

## Python Solution

```
from collections import defaultdict
from typing import List

class Solution:
    def longestArithSeqLength(self, nums: List[int]) -> int:
        n = len(nums)

        dp = [defaultdict(int) for _ in range(n)]

        longest = 2

        for i in range(n):
            for j in range(i):
                difference = nums[i] - nums[j]

                if difference in dp[j]:
                    dp[i][difference] = dp[j][difference] + 1
                else:
                    dp[i][difference] = 2

                longest = max(longest, dp[i][difference])

        return longest
```

The implementation begins by creating a list of hash maps called `dp`. Each map stores arithmetic subsequences ending at a specific index.

The outer loop selects the current ending position `i`. The inner loop examines all previous indices `j`.

For every pair `(j, i)`, the difference is computed. If an arithmetic subsequence with that difference already ends at `j`, it can be extended by including `nums[i]`.

If no such subsequence exists, the pair itself starts a new arithmetic subsequence of length `2`.

The variable `longest` keeps track of the best answer encountered during processing.

The use of `defaultdict(int)` simplifies storage because missing entries default to `0`.

## Go Solution

```
func longestArithSeqLength(nums []int) int {
	n := len(nums)

	dp := make([]map[int]int, n)

	for i := 0; i < n; i++ {
		dp[i] = make(map[int]int)
	}

	longest := 2

	for i := 0; i < n; i++ {
		for j := 0; j < i; j++ {
			difference := nums[i] - nums[j]

			if length, exists := dp[j][difference]; exists {
				dp[i][difference] = length + 1
			} else {
				dp[i][difference] = 2
			}

			if dp[i][difference] > longest {
				longest = dp[i][difference]
			}
		}
	}

	return longest
}
```

The Go implementation mirrors the Python logic closely.

Instead of Python dictionaries, Go uses `map[int]int`.

Since Go maps return zero values for missing keys, we explicitly check whether a key exists using:

```
length, exists := dp[j][difference]
```

The rest of the logic remains identical.

Integer overflow is not a concern because:

- `nums[i] <= 500`
- Differences range from `-500` to `500`

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n^2) | Every pair of indices is processed once |
| Space | O(n^2) | Each index may store many difference values |

The algorithm uses nested loops over the array, producing `O(n^2)` pair evaluations.

For each pair, hash map operations are expected `O(1)` time.

The space complexity is also `O(n^2)` in the worst case because each index may store a distinct difference for many previous elements.

## Edge Cases

One important edge case is when all numbers are identical, such as:

```
[5,5,5,5]
```

The common difference is `0`. Some implementations accidentally assume differences are positive or nonzero. This solution handles the case naturally because the difference value is simply used as a hash map key, including `0`.

Another important edge case is decreasing arithmetic sequences, such as:

```
[10,7,4,1,-2]
```

The common difference is negative. Implementations that only consider positive differences would fail here. Since the solution stores differences as signed integers, negative differences work correctly without any special handling.

A third edge case is when the optimal subsequence is highly non-contiguous, such as:

```
[1,100,2,99,3,98,4]
```

The best arithmetic subsequence is `[1,2,3,4]`, which skips many elements. Greedy contiguous approaches would fail because arithmetic subsequences are not required to be adjacent. The dynamic programming formulation correctly evaluates every ordered pair, ensuring non-contiguous subsequences are discovered.

A final subtle case occurs with arrays of length `2`. Since any two numbers form an arithmetic sequence, the correct answer must always be `2`. The implementation initializes the answer to `2`, guaranteeing correct behavior even when no longer sequence exists.
