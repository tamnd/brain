## Solution

## Problem Understanding

The problem asks us to construct the lexicographically smallest subsequence of a given string such that every distinct character appears exactly once.

A subsequence is formed by deleting zero or more characters from the original string without changing the relative order of the remaining characters. This is important because we are not allowed to rearrange characters arbitrarily. We can only decide which characters to keep and which to discard.

The phrase "lexicographically smallest" means that among all valid subsequences containing every unique character exactly once, we must return the one that would appear first in dictionary order.

For example, given the string `"bcabc"`, the distinct characters are `{'a', 'b', 'c'}`. Multiple valid subsequences exist, such as `"bca"` and `"abc"`, but `"abc"` is lexicographically smaller, so it is the correct answer.

The constraints are relatively small, with `s.length <= 1000`, but the number of possible subsequences grows exponentially. A brute-force approach that generates every subsequence is therefore infeasible.

There are several important observations and edge cases:

- Characters can appear multiple times, but the final result must contain each distinct character exactly once.
- We cannot simply sort the distinct characters because the relative ordering from the original string must still be respected.
- Greedily taking the first occurrence of each character can fail because a smaller character may appear later.
- Some characters may appear only once. Those characters cannot be removed once encountered at the wrong time.
- Strings with already optimal ordering, such as `"abc"`, should remain unchanged.
- Strings with descending order, such as `"cba"`, require careful handling because no reordering is allowed.

The key challenge is balancing two goals simultaneously:

1. Keep the result lexicographically small.
2. Ensure every distinct character still appears exactly once.

## Approaches

### Brute Force Approach

A brute-force solution would generate every possible subsequence of the string, then filter only those subsequences that:

- Contain every distinct character exactly once.
- Preserve relative order.
- Are lexicographically smallest among all valid candidates.

This works because it exhaustively checks all possibilities, guaranteeing correctness.

However, a string of length `n` has `2^n` subsequences. Even for moderate values of `n`, this becomes computationally impossible. With `n = 1000`, brute force is completely infeasible.

### Optimal Greedy + Monotonic Stack Approach

The key insight is that we should build the answer greedily while maintaining lexicographical minimality.

Suppose we are processing characters from left to right. If the current character is smaller than the character at the top of our current result, then replacing the larger character with the smaller one could improve the final answer lexicographically.

However, we can only remove the larger character if it appears again later in the string. Otherwise, removing it would make it impossible to include all distinct characters exactly once.

This naturally leads to a monotonic stack strategy:

- Maintain a stack representing the current best subsequence.
- Track whether a character is already in the stack.
- Track how many future occurrences remain for each character.
- While the current character is smaller than the top of the stack and the top character appears later again, pop the stack.
- Then insert the current character if it is not already included.

This guarantees that the resulting subsequence is both valid and lexicographically smallest.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(2^n) | Generates all subsequences and filters valid ones |
| Optimal | O(n) | O(1) or O(k) | Greedy monotonic stack with character tracking |

Here, `k` is the number of distinct characters, at most 26 because the input contains lowercase English letters.

## Algorithm Walkthrough

1. Count the frequency of every character in the string.

We need to know whether a character will appear again later. This allows us to decide whether it is safe to remove a character from the current subsequence.
2. Create an empty stack.

The stack stores the characters of the current best subsequence in order.
3. Create a set to track characters already in the stack.

Since each character must appear exactly once, we must avoid duplicates.
4. Iterate through each character in the string.

For every character:

- Decrease its remaining frequency count because we are now processing it.
- If the character is already in the stack, skip it because we already included it.
5. Maintain lexicographical order using the stack.

While all three conditions are true:

- The stack is not empty.
- The current character is smaller than the top character of the stack.
- The top character appears again later in the string.

Pop the top character from the stack.

This step improves lexicographical order while preserving validity.
6. Add the current character to the stack.

Push it onto the stack and mark it as present.
7. Build the final answer.

Join all characters in the stack into a string.

### Why it works

The algorithm maintains an important invariant:

- The stack always represents the lexicographically smallest valid subsequence that can still be completed using the remaining characters.

Whenever we remove a character from the stack, we know it appears later again, so we are not losing it permanently. By always preferring smaller characters earlier when safe, we greedily construct the smallest possible valid subsequence.

## Python Solution

```
class Solution:
    def smallestSubsequence(self, s: str) -> str:
        remaining_count = {}

        for char in s:
            remaining_count[char] = remaining_count.get(char, 0) + 1

        stack = []
        in_stack = set()

        for char in s:
            remaining_count[char] -= 1

            if char in in_stack:
                continue

            while (
                stack
                and char < stack[-1]
                and remaining_count[stack[-1]] > 0
            ):
                removed = stack.pop()
                in_stack.remove(removed)

            stack.append(char)
            in_stack.add(char)

        return "".join(stack)
```

The implementation begins by counting how many times each character appears. This information is essential because we must know whether a character can safely be removed and reinserted later.

The `stack` stores the current subsequence being constructed. The `in_stack` set ensures that each character appears at most once.

As we iterate through the string, we first decrement the remaining count for the current character. This reflects that we have consumed one occurrence.

If the character already exists in the stack, we skip it immediately because duplicates are not allowed.

The critical logic is the `while` loop. We remove characters from the stack when:

- The current character is lexicographically smaller.
- The top character will appear again later.

This ensures we improve the final ordering without losing required characters.

Finally, we append the current character and continue processing.

At the end, joining the stack produces the lexicographically smallest valid subsequence.

## Go Solution

```
func smallestSubsequence(s string) string {
	remainingCount := make(map[byte]int)

	for i := 0; i < len(s); i++ {
		remainingCount[s[i]]++
	}

	stack := make([]byte, 0)
	inStack := make(map[byte]bool)

	for i := 0; i < len(s); i++ {
		char := s[i]

		remainingCount[char]--

		if inStack[char] {
			continue
		}

		for len(stack) > 0 &&
			char < stack[len(stack)-1] &&
			remainingCount[stack[len(stack)-1]] > 0 {

			removed := stack[len(stack)-1]
			stack = stack[:len(stack)-1]
			delete(inStack, removed)
		}

		stack = append(stack, char)
		inStack[char] = true
	}

	return string(stack)
}
```

The Go implementation closely mirrors the Python version. Since strings in Go are immutable, we use a `[]byte` slice as the stack for efficient append and pop operations.

The `map[byte]int` structure stores remaining frequencies, while `map[byte]bool` tracks whether a character already exists in the stack.

Unlike Python, Go does not have a built-in set type, so a boolean map is used instead.

No integer overflow concerns exist here because the maximum string length is only 1000.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each character is pushed and popped at most once |
| Space | O(k) | Stack, set, and frequency map store at most distinct characters |

The algorithm is linear because every character enters the stack once and can be removed at most once. Even though there is a nested `while` loop, the total number of stack pops across the entire execution is bounded by the number of pushes.

The auxiliary space depends on the number of distinct characters. Since the input only contains lowercase English letters, this is effectively bounded by 26.

## Edge Cases

### Single Character Input

A string containing only one character, such as `"a"`, is the simplest possible case. A naive implementation could still attempt unnecessary stack operations or duplicate checks. This implementation handles it naturally because the stack starts empty, the character is added once, and the final result is returned immediately.

### All Characters Identical

Consider the input `"aaaaaa"`. The result must contain exactly one `"a"`. Without proper duplicate tracking, an implementation could incorrectly add multiple copies. The `in_stack` set prevents duplicates by skipping characters already present in the stack.

### Characters That Cannot Be Removed

An important subtle case occurs when the top character of the stack will not appear again later. For example, in `"cbad"`, once `'c'` is processed, it cannot safely be removed because no future `'c'` exists. The condition:

```
remaining_count[stack[-1]] > 0
```

ensures we only pop characters that can be recovered later.

### Descending Lexicographical Order

Inputs like `"cba"` can be deceptive. Although `"abc"` is lexicographically smaller, it is not a valid subsequence because subsequences must preserve relative order. The algorithm correctly returns `"cba"` because none of the earlier characters appear again later, so they cannot be removed.

### Complex Duplicate Patterns

Inputs such as `"cbacdcbc"` contain repeated characters interleaved in complicated ways. A naive greedy approach could remove characters too aggressively and lose required letters permanently. The frequency tracking guarantees that characters are only removed when they will reappear later, preserving correctness.
