## Solution

## Problem Understanding

The problem gives us a database table named `ActorDirector`. Each row represents one cooperation between an actor and a director at a specific timestamp. The table has three columns:

- `actor_id`, identifies the actor
- `director_id`, identifies the director
- `timestamp`, uniquely identifies the cooperation event

The `timestamp` column is guaranteed to be unique, which means every row represents a distinct collaboration.

The task is to find every `(actor_id, director_id)` pair where the actor and director have worked together at least three times. The result should contain only the actor and director identifiers, not the timestamps or the number of collaborations.

In other words, we need to group all rows by actor and director, count how many rows belong to each pair, and return the pairs whose count is greater than or equal to 3.

The output order does not matter, so we do not need to sort the result unless we specifically want deterministic ordering.

Since this is a database problem, the expected solution is an SQL query. The core operation here is aggregation. We need to combine rows with the same `(actor_id, director_id)` values and compute how many times each pair appears.

An important observation is that multiple timestamps can exist for the same actor and director pair. Those repeated rows are exactly what we want to count.

There are several edge cases worth considering:

- A pair may cooperate exactly three times. This pair must be included because the condition says "at least three times".
- A pair may cooperate fewer than three times. These pairs must not appear in the output.
- Multiple different actor-director pairs may independently satisfy the condition.
- The table may contain only one or two rows total, in which case the result is empty.
- Since `timestamp` is unique, we do not need to worry about duplicate records representing the same collaboration event.

Because the problem only asks for counting grouped rows, the solution scales very efficiently using SQL aggregation functions.

## Approaches

### Brute Force Approach

A brute-force solution would manually compare every row with every other row and count how many times each `(actor_id, director_id)` pair appears.

Conceptually, this approach would work like this:

1. Iterate through every row in the table.
2. For each row, scan the entire table again to count how many rows have the same actor and director.
3. If the count is at least 3, add the pair to the result.

This produces the correct answer because every possible collaboration pair is checked exhaustively. However, it is extremely inefficient because each row triggers another full scan of the table.

If there are `n` rows, this leads to `O(n²)` time complexity. Database systems are optimized for aggregation, so manually rescanning the table repeatedly is unnecessary and inefficient.

### Optimal Approach

The key insight is that this problem is fundamentally a grouping and counting problem.

SQL provides the `GROUP BY` clause specifically for this purpose. By grouping rows using `(actor_id, director_id)`, all collaborations between the same actor and director become part of the same group.

Once grouped, we can use the `COUNT(*)` aggregate function to determine how many collaborations each pair has.

Finally, the `HAVING` clause filters groups based on aggregate conditions. Unlike `WHERE`, which filters individual rows, `HAVING` filters grouped results.

So the optimal process is:

1. Group rows by actor and director.
2. Count rows inside each group.
3. Keep only groups where the count is at least 3.

This approach is both simple and highly efficient.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Repeatedly scans the table for each row |
| Optimal | O(n) | O(n) | Uses SQL aggregation with `GROUP BY` and `HAVING` |

## Algorithm Walkthrough

1. Start with the `ActorDirector` table, where each row represents one collaboration event between an actor and a director.
2. Group all rows using both `actor_id` and `director_id`. This creates one group for every unique actor-director pair.
3. For each group, compute the number of rows using `COUNT(*)`. Since each row represents one cooperation, this count directly tells us how many times the actor and director worked together.
4. Use the `HAVING` clause to keep only the groups whose collaboration count is greater than or equal to 3.
5. Return the `actor_id` and `director_id` columns for all qualifying groups.

### Why it works

The algorithm works because every collaboration between the same actor and director is placed into the same group by `GROUP BY actor_id, director_id`.

The `COUNT(*)` aggregate correctly measures the total number of collaborations in each group. The `HAVING COUNT(*) >= 3` condition guarantees that only pairs with at least three collaborations are returned.

Since every row belongs to exactly one group, and every group is counted exactly once, the result is complete and correct.

## Python Solution

Even though this is a database problem, LeetCode database problems are typically solved using SQL. Below is the correct SQL query represented inside a Python multiline string for completeness.

```
class Solution:
    def actorsAndDirectors(self):
        return """
        SELECT actor_id, director_id
        FROM ActorDirector
        GROUP BY actor_id, director_id
        HAVING COUNT(*) >= 3
        """
```

The solution uses SQL aggregation to solve the problem efficiently.

The `GROUP BY actor_id, director_id` clause creates groups for every unique actor-director pair. All collaboration rows for the same pair are collected together.

The `COUNT(*)` function computes how many rows belong to each group. Since every row represents one cooperation, this count equals the number of collaborations.

The `HAVING COUNT(*) >= 3` clause filters the grouped results and keeps only those pairs that collaborated at least three times.

Finally, the query returns only the required columns: `actor_id` and `director_id`.

## Edge Cases

One important edge case is when a pair cooperates exactly three times. A common bug is using `COUNT(*) > 3` instead of `COUNT(*) >= 3`. The implementation handles this correctly by explicitly checking for greater than or equal to three.

Another important case is when multiple actor-director pairs satisfy the condition simultaneously. Some incorrect solutions accidentally aggregate only by actor or only by director, which merges unrelated collaborations together. This implementation groups by both columns, ensuring every pair is counted independently.

A third edge case occurs when the table contains very few rows, such as zero, one, or two rows total. In these situations, no pair can possibly satisfy the requirement. The query naturally handles this because no group can reach a count of three, so the result is simply empty.

A fourth edge case involves repeated collaborations with different timestamps. Since timestamps are unique, each row represents a distinct event. The query counts rows directly, which correctly measures the total number of collaborations without needing extra deduplication logic.
