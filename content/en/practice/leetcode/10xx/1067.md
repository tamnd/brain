**Difficulty:** 🔴 Hard  
**Topics:** Math, Dynamic Programming

## Solution

## Problem Understanding

The problem asks us to count how many times a specific digit `d` appears in every number within the inclusive range `[low, high]`.

For example, if `d = 1` and the range is `[1, 13]`, we examine every number:

- `1` contains one `1`
- `10` contains one `1`
- `11` contains two `1`s
- `12` contains one `1`
- `13` contains one `1`

The total count is `6`.

A very important detail is that we are counting occurrences of the digit, not the number of integers containing the digit. If a number contains the digit multiple times, every occurrence must be counted separately.

The input consists of:

- A digit `d` between `0` and `9`
- Two integers `low` and `high`
- The range is inclusive

The constraints are large:

- `high` can be as large as `2 * 10^8`
- The range size itself can therefore also be extremely large

This immediately tells us that iterating through every number and converting it to a string is not efficient enough in the worst case.

The most important edge case is when `d = 0`. Leading zeros are not considered part of a number. For example:

- The number `7` is not treated as `007`
- Therefore those leading zeros must not be counted

Handling zero correctly is the main complication in this problem.

Another important edge case is when the range starts at `1`. Since the standard technique computes counts from `0` to `n`, we must carefully compute:

```
count(high) - count(low - 1)
```

We also need to correctly handle numbers with repeated digits such as `1111`, where multiple occurrences appear in the same number.

## Approaches

### Brute Force Approach

The simplest solution is to iterate through every integer in the range `[low, high]`, convert each number into a string, and count how many times digit `d` appears.

For example:

- If the current number is `131`
- And `d = 1`
- Then we count two occurrences

This approach is correct because it directly follows the problem definition.

However, it is too slow for large ranges. In the worst case, the range size can approach `2 * 10^8`, meaning hundreds of millions of numbers must be processed.

Even if digit counting per number is efficient, the total runtime becomes impractical.

### Optimal Approach

The key insight is that we should count digit occurrences mathematically instead of enumerating every number.

We define a helper function:

```
f(n) = number of times digit d appears in all integers from 0 to n
```

Then the answer becomes:

```
f(high) - f(low - 1)
```

The main challenge is computing `f(n)` efficiently.

Instead of examining every number individually, we analyze each digit position independently:

- ones place
- tens place
- hundreds place
- and so on

For every position, we determine how many complete cycles of digits occur and how many partial cycles contribute additional occurrences.

This technique is commonly called digit DP or positional digit counting.

The only tricky part is handling digit `0`, because leading zeros cannot be counted.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O((high - low + 1) * log(high)) | O(1) | Iterates through every number and counts digits directly |
| Optimal | O(log(high)) | O(1) | Counts contributions position by position mathematically |

## Algorithm Walkthrough

### Step 1: Define a helper function

We first define a helper function:

```
count(n)
```

which returns the number of times digit `d` appears in all integers from `0` to `n`.

If `n < 0`, the result is `0`.

### Step 2: Process one digit position at a time

We iterate through every decimal position:

- `1` for ones place
- `10` for tens place
- `100` for hundreds place
- etc.

For each position `factor`, we split the number into three parts:

```
higher = n // (factor * 10)
current = (n // factor) % 10
lower = n % factor
```

These represent:

- digits left of the current position
- the digit at the current position
- digits right of the current position

For example, if:

```
n = 314159
factor = 100
```

then:

```
higher = 314
current = 1
lower = 59
```

We are examining the hundreds digit.

### Step 3: Count full cycles

For a nonzero digit `d`, every complete cycle contributes:

```
higher * factor
```

occurrences.

Why?

Because each digit repeats uniformly every `factor * 10` numbers.

For example, in the ones place:

- `0-9`
- `10-19`
- `20-29`

Each full block contributes exactly one occurrence of each digit.

### Step 4: Handle the partial cycle

After counting full cycles, we examine the current digit.

If:

```
current > d
```

then we gain one extra full block:

```
factor
```

If:

```
current == d
```

then we gain:

```
lower + 1
```

because the remaining suffix determines how many numbers are included.

### Step 5: Special handling for zero

Zero is special because leading zeros are invalid.

For example:

- We should not count the tens digit of `7` as zero
- We should not treat `7` as `07`

To avoid counting leading zeros:

- We subtract one complete cycle whenever processing digit `0`

This adjustment removes invalid leading-zero contributions.

### Step 6: Compute the final answer

Once we can compute:

```
count(n)
```

the final result is:

```
count(high) - count(low - 1)
```

This isolates the inclusive range `[low, high]`.

### Why it works

The algorithm works because decimal digits follow a predictable repeating structure.

For any position:

- digits repeat in blocks of size `factor * 10`
- every digit appears exactly `factor` times per block

By separating the number into higher, current, and lower sections, we can precisely count:

- complete cycles
- partial cycles

without enumerating individual numbers.

The zero adjustment guarantees that only valid non-leading zeros are counted.

## Python Solution

```
class Solution:
    def digitsCount(self, d: int, low: int, high: int) -> int:
        def count(n: int) -> int:
            if n <= 0:
                return 0

            total = 0
            factor = 1

            while factor <= n:
                lower = n % factor
                current = (n // factor) % 10
                higher = n // (factor * 10)

                if d != 0:
                    total += higher * factor

                    if current > d:
                        total += factor
                    elif current == d:
                        total += lower + 1
                else:
                    if higher != 0:
                        total += (higher - 1) * factor

                        if current > 0:
                            total += factor
                        elif current == 0:
                            total += lower + 1

                factor *= 10

            return total

        return count(high) - count(low - 1)
```

The implementation follows the mathematical counting strategy exactly.

The `count()` helper computes how many times digit `d` appears from `0` to `n`.

The variable `factor` represents the current digit position:

- `1` for ones
- `10` for tens
- `100` for hundreds

For every position, we split the number into:

- `higher`
- `current`
- `lower`

These values allow us to determine how many full digit cycles have occurred and how many additional occurrences come from the partial cycle.

When `d != 0`, the counting logic is straightforward because every digit appears uniformly in each cycle.

When `d == 0`, we must avoid counting leading zeros. The expression:

```
(higher - 1) * factor
```

removes invalid leading-zero contributions.

Finally, we compute:

```
count(high) - count(low - 1)
```

to isolate the target range.

## Go Solution

```
func digitsCount(d int, low int, high int) int {
	count := func(n int) int {
		if n <= 0 {
			return 0
		}

		total := 0
		factor := 1

		for factor <= n {
			lower := n % factor
			current := (n / factor) % 10
			higher := n / (factor * 10)

			if d != 0 {
				total += higher * factor

				if current > d {
					total += factor
				} else if current == d {
					total += lower + 1
				}
			} else {
				if higher != 0 {
					total += (higher - 1) * factor

					if current > 0 {
						total += factor
					} else if current == 0 {
						total += lower + 1
					}
				}
			}

			factor *= 10
		}

		return total
	}

	return count(high) - count(low-1)
}
```

The Go implementation mirrors the Python logic closely.

Go requires explicit integer arithmetic and does not support nested named functions in the same flexible way as Python, so the helper is implemented as a closure assigned to `count`.

All arithmetic remains safely within 32-bit integer range because the maximum input is only `2 * 10^8`.

## Edge Cases

### Edge Case 1: Counting Zero

The most error-prone case is when `d = 0`.

A naive positional counting approach accidentally counts leading zeros. For example, it may incorrectly treat the number `7` as `07` or `007`.

Our implementation avoids this by subtracting one full cycle when processing zero:

```
(higher - 1) * factor
```

This removes invalid leading-zero contributions while preserving legitimate zeros inside numbers like `10` or `205`.

### Edge Case 2: Single Number Range

The range may contain only one number, such as:

```
low = high = 111
```

A brute-force solution naturally handles this, but mathematical range formulas can sometimes introduce off-by-one mistakes.

Using:

```
count(high) - count(low - 1)
```

correctly isolates the inclusive interval, even when both bounds are identical.

### Edge Case 3: Repeated Digits

Numbers like:

```
1111
```

contain multiple copies of the same digit.

A flawed implementation might count the number once instead of counting every occurrence.

Our algorithm processes each digit position independently, so every occurrence contributes separately. The ones, tens, hundreds, and thousands positions are all counted individually.

### Edge Case 4: Large Inputs

The upper bound can reach:

```
2 * 10^8
```

A brute-force approach would require processing hundreds of millions of integers.

The optimal solution scales logarithmically because it only processes one iteration per digit position. Even the maximum input requires fewer than ten iterations.
