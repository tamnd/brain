## Solution

## Problem Understanding

This problem provides two database tables, `Sales` and `Product`, and asks us to generate a report containing three pieces of information for every sale:

- `product_name`
- `year`
- `price`

The important detail is that the `Sales` table does not directly store the product name. Instead, it stores a `product_id`, which references the `Product` table. The `Product` table contains the mapping between a `product_id` and its corresponding `product_name`.

In other words, we need to combine information from two related tables. For every row in `Sales`, we must find the matching row in `Product` using `product_id`, then output:

- the product's name from `Product`
- the year from `Sales`
- the price from `Sales`

This is a classic SQL join problem.

The schema tells us that:

- `(sale_id, year)` uniquely identifies rows in `Sales`
- `product_id` uniquely identifies rows in `Product`
- `Sales.product_id` is a foreign key referencing `Product.product_id`

Because of the foreign key guarantee, every `product_id` in `Sales` is guaranteed to exist in `Product`. That means we do not need to worry about missing matches.

The output can be returned in any order, so sorting is unnecessary unless explicitly requested.

An important observation is that the problem only asks for `product_name`, `year`, and `price`. We do not need to return `sale_id`, `quantity`, or `product_id`.

Potential edge cases include:

- Multiple sales for the same product across different years
- Multiple products existing in `Product` that never appear in `Sales`
- A single product appearing many times in `Sales`
- Large datasets where inefficient nested scans could become expensive

The relational structure and foreign key guarantee make this problem ideal for a SQL `JOIN`.

## Approaches

### Brute Force Approach

A brute force approach would manually compare every row in `Sales` with every row in `Product`.

For each sale:

1. Iterate through all rows in `Product`
2. Find the row whose `product_id` matches
3. Extract the product name
4. Output the required fields

This works because eventually the matching product row will be found for every sale row.

However, this approach is inefficient because for every sale record we may scan the entire `Product` table. If there are `n` sales and `m` products, the total work becomes `O(n × m)`.

In SQL terms, this resembles performing a Cartesian comparison before filtering.

### Optimal Approach

The optimal approach is to use an SQL `INNER JOIN`.

The key insight is that both tables share a common field, `product_id`. Since `Sales.product_id` references `Product.product_id`, we can directly combine related rows using a join operation.

An `INNER JOIN` efficiently matches rows from both tables where the `product_id` values are equal.

After joining:

- `product_name` comes from `Product`
- `year` comes from `Sales`
- `price` comes from `Sales`

This avoids unnecessary repeated scanning and leverages the database engine's optimized join implementation.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × m) | O(1) | Compare every sales row with every product row |
| Optimal | O(n + m) | O(1) extra | Use an INNER JOIN on `product_id` |

## Algorithm Walkthrough

1. Start with the `Sales` table because each row represents one sale that must appear in the final result.
2. For every row in `Sales`, locate the matching row in the `Product` table using the shared `product_id` column.
3. Use an `INNER JOIN` so that only rows with matching `product_id` values are combined.
4. From the joined result:

- Select `Product.product_name`
- Select `Sales.year`
- Select `Sales.price`
5. Return the resulting rows in any order.

### Why it works

The correctness comes from the foreign key relationship between `Sales.product_id` and `Product.product_id`.

Every sales record references exactly one valid product. The `INNER JOIN` pairs each sales row with its corresponding product row, ensuring that the correct product name is attached to each sale. Since we select exactly the requested columns from the joined rows, the output is guaranteed to be correct.

## Python Solution

```
# Write your MySQL query statement below

SELECT
    Product.product_name,
    Sales.year,
    Sales.price
FROM Sales
INNER JOIN Product
    ON Sales.product_id = Product.product_id;
```

This solution directly implements the optimal join strategy.

The `FROM Sales` clause starts with the sales records because each sale should appear in the output.

The `INNER JOIN Product` clause combines rows from both tables whenever their `product_id` values match.

The `ON` condition specifies the relationship between the two tables:

```
Sales.product_id = Product.product_id
```

Finally, the `SELECT` statement extracts only the required columns:

- `product_name` from `Product`
- `year` from `Sales`
- `price` from `Sales`

Since the problem states that output order does not matter, no `ORDER BY` clause is necessary.

## Edge Cases

### Multiple sales referencing the same product

A common source of mistakes is accidentally collapsing duplicate product entries. For example, one product may appear in many sales rows across different years. The implementation handles this correctly because the join operates row by row, preserving every sale independently.

### Products that never appear in Sales

The `Product` table may contain products that were never sold. A naive query using the wrong join direction could accidentally include these unused products. Using an `INNER JOIN` ensures only products that actually appear in `Sales` are returned.

### Large datasets

A brute force nested comparison between tables could become inefficient when both tables are large. The join-based solution avoids this issue by using the database engine's optimized relational operations, which are designed specifically for matching rows across tables efficiently.
