# LeetCode 1090 - Largest Values From Labels

## Problem Understanding

This problem gives us two arrays, `values` and `labels`, where each index represents a single item. The value at `values[i]` is the score or profit of the item, and `labels[i]` represents the category or group that item belongs to.

We are also given two constraints:

- We can select at most `numWanted` items total.
- For any single label, we can select at most `useLimit` items with that label.

Our goal is to choose a subset of items that maximizes the total sum of selected values while respecting both constraints.

The important detail is that the restriction is based on labels, not indices or values. Multiple items may share the same label, and we must carefully track how many items from each label have already been chosen.

The constraints are important:

- `n` can be as large as `2 * 10^4`
- Values and labels can also be fairly large
- A brute-force subset search would be completely infeasible

Since we only care about maximizing the sum, larger values are always preferable when possible. This strongly suggests that sorting and greedy selection may be useful.

There are several important edge cases to keep in mind:

- Multiple large values may share the same label, forcing us to skip some of them because of `useLimit`.
- `numWanted` may be larger than the number of selectable items after label restrictions are applied.
- Some values may be zero.
- All items may share the same label.
- Every item may have a distinct label.

The problem guarantees valid input sizes and that both arrays have the same length.

## Approaches

### Brute Force Approach

A brute-force solution would try every possible subset of items.

For each subset, we would:

1. Count how many items were selected.
2. Count how many times each label appears.
3. Verify the subset satisfies:

- selected count `<= numWanted`
- each label count `<= useLimit`
4. Compute the total value.
5. Track the maximum valid sum.

This approach is correct because it explicitly examines every possible selection and therefore cannot miss the optimal answer.

However, the number of subsets of `n` items is `2^n`. With `n` up to `20,000`, this is astronomically large and impossible to compute within time limits.

### Optimal Greedy Approach

The key observation is that higher values are always better than lower values.

Suppose we process items in descending order of value. Whenever we encounter an item, we should take it if:

- We still have remaining capacity for total items.
- Its label has not exceeded `useLimit`.

Why is this greedy strategy correct?

Because if an item has a larger value than another unused item, selecting the larger one can never be worse. Sorting by descending value ensures we always consider the most profitable items first.

The label restriction only limits how many items from a category we may take, so we maintain a frequency map to track how many times each label has already been used.

This transforms the problem into a straightforward greedy selection process.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n * n) | O(n) | Enumerates every subset |
| Optimal Greedy | O(n log n) | O(n) | Sorts by value, then greedily selects valid items |

## Algorithm Walkthrough

### Optimal Greedy Algorithm

1. Combine each item's value and label into a single structure.

This makes it easy to sort and process items together. Each item becomes a pair like `(value, label)`.
2. Sort all items in descending order by value.

Since we want the maximum possible sum, we should always try larger values before smaller ones.
3. Create a hash map to track label usage counts.

The map stores how many items of each label have already been selected.
4. Initialize:

- `total_sum = 0`
- `selected_count = 0`
5. Iterate through the sorted items.

For each item:

- Check whether we already selected `numWanted` items.
- Check whether the item's label has already been used `useLimit` times.
6. If both constraints allow selection:

- Add the item's value to the answer.
- Increment the label count.
- Increment the selected item count.
7. Continue until:

- We processed all items, or
- We selected `numWanted` items.
8. Return the accumulated sum.

### Why it works

The algorithm works because items are processed from highest value to lowest value. Whenever we choose an item, there is no better remaining item available.

If a higher-value item is legal to take, skipping it in favor of a lower-value item could never improve the final sum. Therefore, greedily selecting the largest currently valid item always preserves optimality.

The label count map guarantees that no label exceeds `useLimit`, and the selection counter guarantees we never exceed `numWanted`.

## Python Solution

```
from typing import List
from collections import defaultdict

class Solution:
    def largestValsFromLabels(
        self,
        values: List[int],
        labels: List[int],
        numWanted: int,
        useLimit: int
    ) -> int:
        
        items = list(zip(values, labels))
        
        # Sort by value descending
        items.sort(reverse=True)
        
        label_count = defaultdict(int)
        
        total_sum = 0
        selected_count = 0
        
        for value, label in items:
            if selected_count == numWanted:
                break
            
            if label_count[label] < useLimit:
                total_sum += value
                label_count[label] += 1
                selected_count += 1
        
        return total_sum
```

The implementation begins by pairing each value with its corresponding label using `zip`. This creates a list of `(value, label)` tuples that can be sorted together.

The list is then sorted in descending order. Python tuple sorting automatically prioritizes the first element, which is the value.

A `defaultdict(int)` is used to track how many times each label has already been selected. This avoids manual existence checks.

The loop processes items from largest value to smallest value. Before selecting an item, the algorithm checks whether:

- the total item limit has already been reached
- the label limit for that item has already been reached

If the item is valid, its value is added to the running sum and both counters are updated.

The loop stops early once `numWanted` items have been selected.

## Go Solution

```
package main

import "sort"

func largestValsFromLabels(values []int, labels []int, numWanted int, useLimit int) int {
	type Item struct {
		value int
		label int
	}

	n := len(values)
	items := make([]Item, n)

	for i := 0; i < n; i++ {
		items[i] = Item{
			value: values[i],
			label: labels[i],
		}
	}

	sort.Slice(items, func(i, j int) bool {
		return items[i].value > items[j].value
	})

	labelCount := make(map[int]int)

	totalSum := 0
	selectedCount := 0

	for _, item := range items {
		if selectedCount == numWanted {
			break
		}

		if labelCount[item.label] < useLimit {
			totalSum += item.value
			labelCount[item.label]++
			selectedCount++
		}
	}

	return totalSum
}
```

The Go implementation follows the same logic as the Python version.

Since Go does not have tuples, we define a small `Item` struct containing `value` and `label`.

Sorting is handled using `sort.Slice`, which allows a custom comparison function for descending order.

The label frequency map is implemented using `map[int]int`.

Go integers are sufficient for this problem because the maximum possible sum easily fits within a 32-bit signed integer, though Go's `int` type is typically 64-bit on modern systems anyway.

## Edge Cases

### All Items Share the Same Label

If every item has the same label, the algorithm must enforce the `useLimit` restriction correctly. A buggy implementation might accidentally select too many items because it only tracks total count.

For example:

```
values = [9,8,7]
labels = [1,1,1]
useLimit = 1
```

Only the value `9` may be selected.

The implementation handles this using the `label_count` hash map, which strictly prevents selecting more than `useLimit` items for any label.

### numWanted Is Larger Than the Number of Valid Selections

Sometimes the total number of items allowed by label constraints is smaller than `numWanted`.

For example:

```
values = [5,4,3]
labels = [1,1,1]
numWanted = 10
useLimit = 1
```

Only one item can actually be selected.

The algorithm correctly processes all items and simply returns the best achievable sum without requiring exactly `numWanted` selections.

### High-Value Items Must Be Skipped

A common bug occurs when the algorithm incorrectly assumes the largest values should always be taken.

Example:

```
values = [9,8,7]
labels = [1,1,2]
useLimit = 1
```

After taking `9`, the value `8` must be skipped because label `1` already reached its limit. The algorithm must continue scanning for other valid items instead of stopping early.

The implementation handles this correctly because it checks label usage independently for every item and continues iterating through the sorted list.
