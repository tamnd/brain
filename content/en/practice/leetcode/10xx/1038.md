## Solution

## Problem Understanding

The problem gives us the root of a Binary Search Tree (BST) and asks us to transform it into a Greater Sum Tree. In the transformed tree, every node’s value should become:

- its original value
- plus the sum of all node values greater than it in the BST

Because the input tree is a BST, it has an important ordering property:

- every value in the left subtree is smaller than the current node
- every value in the right subtree is larger than the current node

This ordering is the key observation that allows an efficient solution.

For example, consider this BST:

```
      4
    /   \
   1     6
```

The node `4` should become:

```
4 + 6 = 10
```

because `6` is the only value greater than `4`.

The node `1` should become:

```
1 + 4 + 6 = 11
```

because all larger values are included.

The node `6` remains `6` because there are no greater values.

The input is given as the root of a binary tree structure, and we must modify the tree in place and return the root of the updated tree.

The constraints are relatively small:

- at most 100 nodes
- node values are between 0 and 100
- all values are unique

Even though the constraints are small enough for less efficient approaches to pass, the problem is fundamentally about recognizing and exploiting BST traversal order.

Several edge cases are important:

- A tree with only one node, the value should remain unchanged.
- A tree where every node only has a right child, values accumulate from largest to smallest.
- A tree where every node only has a left child, each node receives the sum of all larger ancestors.
- The minimum value can be `0`, which must still be updated correctly.
- Since values are unique, we never need to handle duplicate ordering behavior.

## Approaches

### Brute Force Approach

A straightforward solution is to process every node independently.

For each node:

1. Traverse the entire tree.
2. Compute the sum of all values greater than the current node.
3. Update the node with:

```
node.val + greater_sum
```

This works because we explicitly calculate the exact values needed for every node.

However, this approach is inefficient. If the tree contains `n` nodes, then for every node we may traverse all `n` nodes again to compute the greater sum.

That gives a total time complexity of:

```
O(n^2)
```

Although `n <= 100` is small enough to pass, the solution ignores the BST structure entirely.

### Key Insight for the Optimal Solution

The important BST property is:

```
Inorder traversal of a BST visits nodes in sorted ascending order.
```

Normally, inorder traversal processes nodes like this:

```
Left -> Node -> Right
```

which gives values from smallest to largest.

For this problem, we need the opposite direction because each node depends on larger values. Therefore, we use a reverse inorder traversal:

```
Right -> Node -> Left
```

This visits nodes from largest to smallest.

As we traverse:

- maintain a running cumulative sum
- when visiting a node, add its value to the running sum
- replace the node’s value with the updated sum

Because larger nodes are processed first, the running sum always contains exactly the values greater than or equal to the current node.

This produces an elegant linear-time solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(h) | For each node, traverse the tree again to compute greater sums |
| Optimal | O(n) | O(h) | Reverse inorder traversal processes nodes from largest to smallest |

Here, `h` is the height of the tree.

## Algorithm Walkthrough

### Optimal Reverse Inorder Traversal

1. Initialize a variable called `running_sum` to `0`.

This variable stores the sum of all previously visited nodes. Since we traverse from largest to smallest, it always represents the sum of greater values.
2. Start a reverse inorder traversal.

Instead of the normal BST traversal order:

```
Left -> Node -> Right
```

we use:

```
Right -> Node -> Left
```

This ensures we visit larger values first.
3. Recursively traverse the right subtree.

The right subtree contains all values greater than the current node, so these must be processed before updating the current node.
4. Update the current node.

Add the current node’s original value to `running_sum`:

```
running_sum += node.val
```

Then replace the node’s value:

```
node.val = running_sum
```
5. Recursively traverse the left subtree.

After updating the current node, continue processing smaller values.
6. Continue until all nodes are processed.

Because every node is visited exactly once, the traversal completes in linear time.

### Why it works

The correctness comes from the traversal order invariant.

When visiting a node during reverse inorder traversal, all nodes with greater values have already been processed. Therefore, `running_sum` already contains the sum of every greater value in the BST.

After adding the current node’s original value, the updated value becomes:

```
current value + sum of all greater values
```

which is exactly the required transformation.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import Optional

class Solution:
    def bstToGst(self, root: Optional[TreeNode]) -> Optional[TreeNode]:
        running_sum = 0

        def reverse_inorder(node: Optional[TreeNode]) -> None:
            nonlocal running_sum

            if not node:
                return

            # Process larger values first
            reverse_inorder(node.right)

            # Update running sum and node value
            running_sum += node.val
            node.val = running_sum

            # Process smaller values
            reverse_inorder(node.left)

        reverse_inorder(root)
        return root
```

The implementation follows the reverse inorder traversal strategy directly.

The variable `running_sum` stores the cumulative sum of all previously visited nodes. Since traversal occurs from largest to smallest, this variable always contains the correct greater-value total.

The helper function `reverse_inorder` performs the recursive traversal.

The traversal order is:

```
Right -> Node -> Left
```

The right subtree is processed first because it contains larger values. After returning from the right subtree, the current node is updated using the accumulated sum. Finally, the left subtree is processed.

The `nonlocal` keyword allows the nested function to modify `running_sum` defined in the outer scope.

The tree is modified in place, and the original root reference is returned.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func bstToGst(root *TreeNode) *TreeNode {
    runningSum := 0

    var reverseInorder func(node *TreeNode)

    reverseInorder = func(node *TreeNode) {
        if node == nil {
            return
        }

        // Process larger values first
        reverseInorder(node.Right)

        // Update running sum and node value
        runningSum += node.Val
        node.Val = runningSum

        // Process smaller values
        reverseInorder(node.Left)
    }

    reverseInorder(root)
    return root
}
```

The Go implementation mirrors the Python solution closely.

Instead of Python’s `nonlocal`, Go closures naturally capture variables from the surrounding scope, so `runningSum` can be updated directly inside the recursive function.

Go uses `nil` for empty pointers instead of Python’s `None`.

Because node values are small and the maximum number of nodes is only 100, integer overflow is not a concern in Go.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each node is visited exactly once |
| Space | O(h) | Recursive call stack depth equals tree height |

The traversal touches every node exactly one time, giving linear time complexity.

The auxiliary space comes only from recursion. In the worst case of a completely skewed tree, the recursion depth becomes `O(n)`. In a balanced BST, the recursion depth is `O(log n)`.

## Edge Cases

### Single Node Tree

A tree with only one node is the simplest possible input. Since there are no greater values in the BST, the node should remain unchanged.

A buggy implementation might accidentally double-count the node or incorrectly initialize the running sum. This solution handles the case naturally because the traversal visits the node once, adds its own value to `running_sum`, and assigns the same value back.

### Completely Left-Skewed Tree

In a left-skewed BST, every node only has a left child. This creates the maximum recursion depth for left-heavy trees.

This case is important because each node depends on all ancestor values. The reverse inorder traversal correctly processes larger ancestor nodes before smaller descendants, ensuring the running sum already contains all greater values.

### Completely Right-Skewed Tree

In a right-skewed BST, traversal effectively becomes a descending linked-list traversal.

This case verifies that the algorithm correctly accumulates values in descending order without skipping nodes or processing them in the wrong direction.

Because the traversal starts at the rightmost node and moves leftward through recursion, every node receives the proper cumulative sum.

### Minimum Value of Zero

The problem allows node values equal to `0`.

Some implementations incorrectly treat `0` as a falsy or special case during updates. This solution never relies on truthiness of node values, only whether the node itself is `None`, so zero-valued nodes are processed correctly.
