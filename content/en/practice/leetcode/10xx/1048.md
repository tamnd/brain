## Solution

## Problem Understanding

The problem gives us a list of lowercase English words and asks for the length of the longest possible string chain.

A word `A` is considered a predecessor of another word `B` if we can insert exactly one character anywhere inside `A` and obtain `B`, while preserving the order of all existing characters. The inserted character can appear at the beginning, middle, or end.

For example:

- `"abc"` → `"abac"` is valid because inserting `'a'` after `'ab'` creates `"abac"`.
- `"abc"` → `"acbd"` is invalid because the relative ordering of characters changes.

A chain is formed when each word is a predecessor of the next word. The task is to determine the maximum possible chain length.

The input size is relatively small:

- At most `1000` words
- Each word length is at most `16`

The short maximum word length is extremely important. It suggests that generating modified versions of a word is cheap, because each word only has up to 16 characters. This observation enables an efficient dynamic programming solution.

Several edge cases are important:

- A list with only one word should return `1`.
- Multiple unrelated words may produce no valid chain.
- Words may branch into multiple possible chains, so greedy approaches are unsafe.
- Words are not guaranteed to be sorted.
- Chains can begin from any word, not necessarily the shortest one.
- Duplicate predecessor possibilities may exist, requiring careful dynamic programming updates.

## Approaches

### Brute Force Approach

A straightforward solution would try every possible ordering of words and recursively determine whether one word can follow another.

For every pair of words, we can check whether one is a predecessor of the other using a two-pointer comparison. Then we could build chains recursively using depth-first search.

The problem with this approach is that the number of possible chains grows exponentially. Even with memoization, comparing every word against every other word repeatedly becomes expensive.

The brute-force strategy works because it eventually explores all valid chains and therefore cannot miss the optimal answer. However, it performs a large amount of redundant work.

### Key Insight

The critical observation is that valid chains always increase word length by exactly one.

This means if `"abc"` is a predecessor of `"abdc"`, then:

- `len("abdc") = len("abc") + 1`

Therefore, if we process words in increasing order of length, we can build solutions incrementally.

For each word, we can generate all possible predecessor candidates by removing one character at every position. If the resulting smaller word already exists, then we can extend its chain.

For example:

- Current word: `"bdca"`
- Remove each character:

- `"dca"`
- `"bca"`
- `"bda"`
- `"bdc"`

If `"bca"` already has chain length `3`, then `"bdca"` can achieve chain length `4`.

This transforms the problem into dynamic programming over word lengths.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(N² × L) to O(2^N) depending on implementation | O(N) | Repeatedly checks many redundant chains |
| Optimal Dynamic Programming | O(N × L²) | O(N) | Builds chains incrementally using hash map lookups |

Where:

- `N` = number of words
- `L` = maximum word length, at most 16

## Algorithm Walkthrough

1. Sort all words by length in ascending order.

This guarantees that when processing a word, all possible predecessors have already been processed. Since a predecessor must be exactly one character shorter, this ordering naturally supports dynamic programming.
2. Create a hash map called `dp`.

The map stores:

```
dp[word] = longest chain length ending at this word
```

A hash map is ideal because predecessor lookups become O(1).
3. Initialize a variable `longest_chain = 1`.

Every individual word forms a valid chain of length 1 by itself.
4. Process each word in sorted order.

For every word:

- Start with chain length `1`
- Generate all possible predecessors by deleting one character at every position
5. Generate predecessor candidates.

For a word like `"abcd"`:

- Remove index 0 → `"bcd"`
- Remove index 1 → `"acd"`
- Remove index 2 → `"abd"`
- Remove index 3 → `"abc"`

Each generated string is a possible predecessor.
6. Check whether the predecessor exists in `dp`.

If it exists, then we already know the best chain ending at that predecessor.

We can extend that chain:

```
current_chain = dp[predecessor] + 1
```
7. Update the best chain length for the current word.

Store:

```
dp[word] = maximum chain length found
```
8. Update the global answer.

Keep track of the largest chain seen so far.
9. Return the final maximum chain length.

### Why it works

The algorithm works because chains must increase in word length by exactly one character. Sorting by length ensures all smaller valid predecessors are processed before larger words.

For every word, the algorithm examines every possible predecessor obtainable by deleting one character. Since every valid predecessor relationship must satisfy this condition, no valid chain is missed.

Dynamic programming guarantees optimality because each word stores the best possible chain ending at that word, and future words build upon those optimal subproblems.

## Python Solution

```
from typing import List

class Solution:
    def longestStrChain(self, words: List[str]) -> int:
        words.sort(key=len)

        dp = {}
        longest_chain = 1

        for word in words:
            current_best = 1

            for i in range(len(word)):
                predecessor = word[:i] + word[i + 1:]

                if predecessor in dp:
                    current_best = max(current_best, dp[predecessor] + 1)

            dp[word] = current_best
            longest_chain = max(longest_chain, current_best)

        return longest_chain
```

The implementation begins by sorting the words according to length. This guarantees that every possible predecessor appears earlier in the iteration order.

The `dp` dictionary stores the best chain length ending at each word. When processing a new word, the algorithm tries removing every character position to generate all possible predecessor candidates.

For each predecessor candidate, the dictionary is checked. If the predecessor exists, then its chain can be extended by one.

The variable `current_best` tracks the maximum chain length ending at the current word. After evaluating all predecessor possibilities, the value is stored in `dp`.

Finally, `longest_chain` maintains the global maximum across all words.

## Go Solution

```
package main

import (
	"sort"
)

func longestStrChain(words []string) int {
	sort.Slice(words, func(i, j int) bool {
		return len(words[i]) < len(words[j])
	})

	dp := make(map[string]int)
	longestChain := 1

	for _, word := range words {
		currentBest := 1

		for i := 0; i < len(word); i++ {
			predecessor := word[:i] + word[i+1:]

			if chainLength, exists := dp[predecessor]; exists {
				if chainLength+1 > currentBest {
					currentBest = chainLength + 1
				}
			}
		}

		dp[word] = currentBest

		if currentBest > longestChain {
			longestChain = currentBest
		}
	}

	return longestChain
}
```

The Go implementation follows the same dynamic programming strategy as the Python solution.

A `map[string]int` is used instead of a Python dictionary. Go string slicing is efficient for this problem because word lengths are extremely small.

Unlike Python, Go requires explicit sorting with `sort.Slice`, and hash map lookups return both the value and a boolean indicating whether the key exists.

No integer overflow concerns exist because the maximum chain length is bounded by the maximum word length, which is only 16.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(N × L²) | Each word generates L predecessors, each creation costs O(L) |
| Space | O(N) | Hash map stores one entry per word |

The sorting step costs `O(N log N)`, but the dynamic programming work dominates for the given constraints.

For every word of length `L`, we generate `L` predecessor strings. Constructing each predecessor requires slicing and concatenation, which costs `O(L)`. Therefore, each word costs `O(L²)` total.

Since `L ≤ 16`, the algorithm is extremely efficient in practice.

## Edge Cases

One important edge case occurs when no two words can form a valid predecessor relationship. A naive implementation might incorrectly assume at least one chain extension exists and return `0`. The correct answer must still be `1`, because every individual word forms a valid chain by itself. The implementation handles this by initializing each word's chain length to `1`.

Another subtle edge case involves multiple possible predecessors for the same word. For example, `"bdca"` can be reached from both `"bca"` and `"bda"`. A greedy approach that chooses the first predecessor found could miss the optimal chain. The dynamic programming solution correctly checks all predecessor possibilities and stores the maximum chain length.

A third important edge case is unordered input. The words may appear in any arbitrary order, and predecessor words are not guaranteed to come before their successors. Without sorting, the algorithm could attempt to process a longer word before its predecessor chain has been computed. Sorting by word length ensures every valid predecessor has already been processed.

Another common source of bugs is handling character removal correctly. The predecessor must preserve the relative order of the remaining characters. The implementation guarantees this by generating predecessors only through deletion, which naturally preserves ordering.
