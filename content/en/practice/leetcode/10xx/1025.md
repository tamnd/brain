# LeetCode 1025 - Divisor Game

## Problem Understanding

The problem describes a two player game between Alice and Bob. The game starts with a single integer `n` written on a chalkboard. Alice always takes the first turn. On each turn, the current player must choose a positive integer `x` such that:

- `x` is smaller than `n`
- `x` divides `n` evenly, meaning `n % x == 0`

After choosing `x`, the player replaces `n` with `n - x`. The game continues with the new value of `n`.

A player loses if they cannot make a valid move. Since both players are assumed to play optimally, each player always chooses the move that maximizes their own chance of winning.

The task is to determine whether Alice wins the game for a given starting value of `n`.

The input consists of a single integer `n`, and the output should be a boolean value:

- `true` if Alice can force a win
- `false` if Bob can force a win

The constraint `1 <= n <= 1000` is relatively small. This means even a dynamic programming solution would work comfortably within limits. However, the problem contains a mathematical pattern that allows for an even simpler solution.

An important edge case is `n = 1`. In this situation, Alice cannot choose any valid divisor because there is no positive integer `x` such that `0 < x < 1`. Therefore, Alice immediately loses.

Another important observation is that the game alternates between even and odd numbers. This turns out to be the key insight behind the optimal solution.

## Approaches

### Brute Force Dynamic Programming

A straightforward approach is to model the game using recursion or dynamic programming.

For each value of `n`, we try every valid divisor `x`. If there exists at least one move such that the opponent loses after moving to `n - x`, then the current state is winning.

This works because the game has optimal play. A player wins if they can move the game into a losing state for the opponent.

The recursive relation is:

- `dp[n] = true` if there exists a divisor `x` where `dp[n - x] == false`
- otherwise `dp[n] = false`

Although this approach is correct, it requires checking divisors for every number up to `n`.

### Optimal Mathematical Observation

The key insight is that Alice wins if and only if `n` is even.

To understand why, observe these properties:

- If `n` is even, Alice can always subtract `1`
- Since `1` divides every integer, this move is always valid
- Subtracting `1` from an even number produces an odd number

Now consider what happens with odd numbers:

- Every divisor of an odd number is also odd
- Subtracting an odd divisor from an odd number always produces an even number

This means:

- From an even number, a player can force the opponent into an odd number
- From an odd number, every move gives the opponent an even number

Eventually, the player who receives `1` loses because no moves are possible.

Since the sequence alternates between even and odd numbers, the player who starts with an even number always has a winning strategy.

## Algorithm Walkthrough

1. Check whether `n` is even.
2. If `n` is even, return `true` because Alice can always force Bob into an odd number after every move.
3. If `n` is odd, return `false` because every valid move leads to an even number, allowing Bob to maintain the winning pattern.

### Why it works

The invariant is that even numbers are winning states and odd numbers are losing states.

- From an even number, the current player can always move to an odd number by subtracting `1`
- From an odd number, every valid divisor is odd, so every move results in an even number

This creates a perfect alternating structure where the player receiving an odd number is always at a disadvantage. Since Alice starts first, she wins exactly when the initial number is even.

## Python Solution

```
class Solution:
    def divisorGame(self, n: int) -> bool:
        return n % 2 == 0
```

The implementation directly applies the mathematical observation.

The expression `n % 2 == 0` checks whether `n` is even. If it is even, the function returns `True`, meaning Alice can force a win. Otherwise, it returns `False`.

This solution avoids recursion, dynamic programming tables, and divisor enumeration because the game outcome depends entirely on the parity of `n`.

## Edge Cases

### Edge Case 1, `n = 1`

This is the smallest possible input and an important base case. Since there are no positive integers smaller than `1`, Alice cannot make any move and immediately loses.

The implementation handles this naturally because `1` is odd, so the expression `n % 2 == 0` evaluates to `False`.

### Edge Case 2, Large Even Numbers

Large even values such as `1000` could potentially cause performance concerns in recursive or simulation based approaches. However, the mathematical solution only checks parity, so the runtime remains constant regardless of input size.

The implementation correctly returns `True` for every even number.

### Edge Case 3, Odd Prime Numbers

Odd prime numbers only have one valid divisor, which is `1`. A naive implementation might incorrectly assume limited choices change the outcome.

However, the parity argument still applies. Subtracting `1` from an odd number produces an even number, which is a winning state for the opponent.

The implementation correctly returns `False` for all odd primes such as `3`, `5`, `7`, and `997`.
