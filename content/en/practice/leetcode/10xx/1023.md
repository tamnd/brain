## Solution

## Problem Understanding

The problem asks us to determine, for each string in the `queries` array, whether it can be derived from a given `pattern` by inserting lowercase letters at any position. Importantly, we cannot remove or reorder any characters from the pattern, and uppercase letters in the pattern must appear in the same order in the query. The queries and pattern are composed of English letters, with both lowercase and uppercase letters relevant for matching.

The input consists of an array of strings `queries` and a single string `pattern`. The expected output is a boolean array `answer` where each element corresponds to whether the respective query matches the pattern under the insertion rules described. Constraints indicate that the maximum length for `pattern` and `queries` is 100, and the number of queries is also capped at 100, making a quadratic-time solution feasible.

Key points to notice include that only uppercase letters in the pattern strictly define required positions, while lowercase letters in the query can fill gaps. Edge cases that could trip a naive implementation include queries that contain extra uppercase letters not in the pattern, queries with missing uppercase letters, or patterns that are empty strings.

## Approaches

The brute-force approach would attempt to generate all possible strings from the pattern by inserting lowercase letters at every possible position and then check if the query matches any of these generated strings. While correct, this approach is highly inefficient because the number of possible insertions grows exponentially with the length of the query and pattern.

The optimal approach leverages a two-pointer technique. We traverse both the query and the pattern simultaneously. We advance the pattern pointer only when a character in the query matches the current character in the pattern. Any extra lowercase letters in the query are ignored. If the query contains an uppercase letter not in the pattern or the pattern pointer does not reach the end after processing the query, the match fails. This approach is efficient because it performs a single pass through each query.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(26^(n+m)) | O(1) | Generate all possible strings from the pattern by inserting lowercase letters and compare |
| Optimal | O(Q * L) | O(1) | Two-pointer traversal of query and pattern, ignoring extra lowercase letters |

Here, `Q` is the number of queries and `L` is the maximum length of a query or the pattern.

## Algorithm Walkthrough

1. Initialize an empty result list `answer`.
2. For each query in `queries`, initialize two pointers: `i` for the query and `j` for the pattern, both starting at 0.
3. Traverse the query string with `i`. If the current character of the query matches the current character of the pattern, increment both `i` and `j`.
4. If the query character is lowercase and does not match the pattern, increment only `i` to skip it.
5. If the query character is uppercase and does not match the pattern, immediately mark the query as not matching and break.
6. After traversing the query, if the pattern pointer `j` has reached the end of the pattern, the query matches. Otherwise, it does not.
7. Append the boolean result for the query to `answer`.
8. Return the complete `answer` list.

**Why it works**: The algorithm ensures that all uppercase letters in the query must match the pattern in order. Extra lowercase letters are permissible anywhere, which aligns with the rules for inserting lowercase letters. The two-pointer invariant guarantees that the sequence of pattern characters is followed exactly in the query.

## Python Solution

```
from typing import List

class Solution:
    def camelMatch(self, queries: List[str], pattern: str) -> List[bool]:
        answer = []
        
        for query in queries:
            i, j = 0, 0
            match = True
            
            while i < len(query):
                if j < len(pattern) and query[i] == pattern[j]:
                    i += 1
                    j += 1
                elif query[i].islower():
                    i += 1
                else:
                    match = False
                    break
            
            if j < len(pattern):
                match = False
                
            answer.append(match)
        
        return answer
```

**Explanation**: The `camelMatch` method iterates over each query. Two pointers `i` and `j` track positions in the query and pattern. Matching characters advance both pointers. Extra lowercase characters in the query are skipped, while unexpected uppercase characters trigger a mismatch. After processing, if the pattern pointer has not reached the end, the query does not match. The final boolean for each query is appended to `answer`.

## Go Solution

```
func camelMatch(queries []string, pattern string) []bool {
    answer := make([]bool, len(queries))
    
    for idx, query := range queries {
        i, j := 0, 0
        match := true
        
        for i < len(query) {
            if j < len(pattern) && query[i] == pattern[j] {
                i++
                j++
            } else if query[i] >= 'a' && query[i] <= 'z' {
                i++
            } else {
                match = false
                break
            }
        }
        
        if j < len(pattern) {
            match = false
        }
        
        answer[idx] = match
    }
    
    return answer
}
```

**Explanation**: The Go version mirrors the Python implementation. The main difference is that we check lowercase letters using ASCII comparison. Slices are used to store the result. Go does not require `List` or `islower()` methods, so comparisons are done directly on rune values.

## Edge Cases

**Case 1: Query has extra uppercase letters not in pattern**

If a query contains an uppercase letter that does not appear in the pattern at the expected order, it should immediately fail. Our implementation checks for this during traversal and returns `False` without completing the rest of the query.

**Case 2: Pattern is empty**

An empty pattern can only match queries consisting entirely of lowercase letters. Any uppercase character in the query will cause a mismatch. Our two-pointer check naturally handles this by failing if any character in the query is uppercase while the pattern pointer is already at the end.

**Case 3: Queries shorter than pattern**

If a query ends before the pattern is fully matched, it cannot satisfy the insertion condition.
