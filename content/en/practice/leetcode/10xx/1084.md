## Solution

## Problem Understanding

The problem gives us two database tables, `Product` and `Sales`.

The `Product` table stores information about products. Each row contains a unique `product_id`, the product's name, and its unit price.

The `Sales` table stores transaction records. Every row represents a sale event and contains the product sold, the date of the sale, the quantity, and other related information. A single product may appear many times in this table because the same product can be sold multiple times on different dates.

The task is to return all products that were sold only during the first quarter of 2019. The first quarter includes all dates from `2019-01-01` through `2019-03-31`, inclusive.

The important detail is the word "only". A product qualifies if:

- It has at least one sale during the first quarter of 2019.
- It has no sales outside that range.

This means that if a product was sold both inside and outside the quarter, it must be excluded.

The expected output contains:

- `product_id`
- `product_name`

The result may be returned in any order.

Even though the problem statement labels this as an Easy database problem, it tests an important SQL concept: filtering groups based on aggregate conditions across multiple rows.

There are several edge cases to consider carefully.

A product may have multiple sales within the first quarter. That product should still qualify as long as every sale falls within the allowed range.

A product may have one valid sale in the quarter and another sale later in the year. That product must be excluded because it was not sold exclusively during the first quarter.

A product may never appear in the `Sales` table at all. Such a product should not be included because the problem asks for products that were sold in the first quarter, not products that simply exist.

The `Sales` table may also contain duplicate rows. Our logic must still work correctly even if identical sale records appear multiple times.

## Approaches

### Brute Force Approach

A brute force approach would examine every product individually and scan all sales rows associated with that product.

For each product, we would:

1. Gather all sales rows for that product.
2. Check whether every sale date lies within the first quarter of 2019.
3. Verify that at least one qualifying sale exists.

This approach is logically correct because it explicitly validates every sale record for every product. However, it can become inefficient because we repeatedly scan the sales data for each product.

If there are `P` products and `S` sales rows, repeatedly filtering the sales table for each product can lead to a time complexity near `O(P × S)`.

### Optimal Approach

The key observation is that we do not need to inspect each product separately with repeated scans.

Instead, we can group sales by `product_id` and use aggregate conditions.

For a product to qualify:

- The minimum sale date must be at least `2019-01-01`.
- The maximum sale date must be at most `2019-03-31`.

If both conditions are true, then every sale for that product falls inside the first quarter.

This is much more efficient because we process the sales table once, aggregate by product, and then filter using `HAVING`.

We then join the result with the `Product` table to retrieve the product name.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(P × S) | O(1) | Repeatedly scans sales rows for each product |
| Optimal | O(S) | O(S) | Uses grouping and aggregate filtering |

## Algorithm Walkthrough

1. Start with the `Sales` table because the qualification depends entirely on sale dates.
2. Group all rows by `product_id`. This allows us to analyze all sales belonging to the same product together instead of processing them individually.
3. For each product group, compute:

- The earliest sale date using `MIN(sale_date)`
- The latest sale date using `MAX(sale_date)`
4. Check whether:

- The earliest sale date is greater than or equal to `2019-01-01`
- The latest sale date is less than or equal to `2019-03-31`
5. If both conditions hold, then every sale for that product occurred inside the first quarter of 2019.
6. Join the filtered product IDs with the `Product` table so we can retrieve the corresponding `product_name`.
7. Return the final result set containing `product_id` and `product_name`.

### Why it works

The algorithm works because the minimum and maximum sale dates completely define the range of dates for a product's sales history.

If the earliest sale is inside the allowed interval and the latest sale is also inside the interval, then every intermediate sale must also be inside that interval.

Therefore, checking only the minimum and maximum dates is sufficient to guarantee correctness.

## Python Solution

```
# Write your MySQL query statement below

SELECT
    p.product_id,
    p.product_name
FROM Product p
JOIN Sales s
    ON p.product_id = s.product_id
GROUP BY p.product_id, p.product_name
HAVING MIN(s.sale_date) >= '2019-01-01'
   AND MAX(s.sale_date) <= '2019-03-31';
```

This solution begins by joining the `Product` and `Sales` tables using `product_id`.

After the join, all sales belonging to the same product are grouped together using `GROUP BY`.

The `HAVING` clause is critical because aggregate functions such as `MIN()` and `MAX()` operate after grouping.

`MIN(s.sale_date)` identifies the earliest sale date for the product, while `MAX(s.sale_date)` identifies the latest sale date.

If the earliest date is still within the first quarter and the latest date is also within the first quarter, then every sale for that product must lie within the allowed range.

The query finally returns only the required columns:

- `product_id`
- `product_name`

## Edge Cases

One important edge case is a product with sales both inside and outside the first quarter. A naive solution might check only whether a product has at least one sale during Q1, which would incorrectly include such products. The aggregate date range approach avoids this mistake because the maximum or minimum sale date would fall outside the allowed interval.

Another important edge case is boundary dates. The problem explicitly states that the range is inclusive, meaning sales on `2019-01-01` and `2019-03-31` are valid. Using `>=` and `<=` ensures those products are handled correctly.

A third edge case is duplicate sales rows. Since the query uses aggregate functions over grouped rows, duplicate records do not affect correctness. The minimum and maximum sale dates remain the same regardless of duplication.

A final edge case involves products that exist in the `Product` table but never appear in `Sales`. Because the solution performs an inner join with `Sales`, products without any sales are automatically excluded, which matches the problem requirements.
