## Solution

## Problem Understanding

The problem requires designing a system that can process a stream of characters, one character at a time, and at each step check whether the most recent characters form any word from a given list `words`. Essentially, after each new character is added to the stream, the algorithm must efficiently check if any suffix of the accumulated stream matches one of the words in the input list.

The input consists of a list of words `words`, where each word contains only lowercase English letters, and a stream of individual letters queried sequentially. The expected output is a boolean for each query: `true` if the current stream’s suffix matches a word in `words`, and `false` otherwise.

Constraints indicate that `words` can contain up to 2000 words, each of length up to 200, and the total number of queries can be up to 40,000. This combination of constraints implies that a naive approach that scans all words on every query will likely be too slow. Important edge cases include: a stream containing letters that do not appear in any word, repeated letters that could form multiple words, and words that are prefixes of others.

## Approaches

The brute-force approach would involve storing the entire character stream and checking, on every query, every possible suffix against all words. This guarantees correctness because it explicitly tests all possible suffixes, but it is extremely inefficient. For each query, we could potentially check up to `length_of_stream × number_of_words` comparisons, and each comparison could take up to the length of the word. With 40,000 queries and long words, this becomes computationally expensive.

The optimal approach leverages a **reversed Trie**. The key insight is that we only need to check the most recent characters for matches. By storing all words in reverse order in a Trie, each query can start from the newest character and traverse backward through the stream, checking if any sequence forms a valid word. This avoids repeatedly scanning the whole word list and reduces the number of comparisons significantly.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(Q × W × L) | O(Q) | Check all suffixes of the stream against each word, where Q is the number of queries, W is the number of words, L is the average word length |
| Optimal | O(Q × L) | O(W × L) | Use a reversed Trie to check only relevant suffixes; L is the maximum length of a word |

## Algorithm Walkthrough

1. Construct a Trie where each word is inserted in reversed order. Each node in the Trie represents a letter and contains child nodes for the next letters, as well as a boolean flag indicating if the node completes a word.
2. Maintain a deque (or list) to store the most recent characters of the stream. The deque length is bounded by the length of the longest word in `words`.
3. On each query, append the new letter to the deque. If the deque exceeds the maximum word length, remove the oldest character to maintain a fixed window.
4. Starting from the most recent character, traverse the Trie backward through the deque. If the traversal reaches a node marked as a word, return `true`.
5. If the traversal finishes without finding a word, return `false`.

Why it works: The reversed Trie ensures that only relevant suffixes of the stream are considered. By limiting the stored stream to the length of the longest word, we avoid unnecessary checks, and traversing the Trie guarantees that every possible word suffix is correctly detected.

## Python Solution

```
from collections import deque
from typing import List

class TrieNode:
    def __init__(self):
        self.children = {}
        self.is_word = False

class StreamChecker:

    def __init__(self, words: List[str]):
        self.root = TrieNode()
        self.stream = deque()
        self.max_len = 0
        
        for word in words:
            node = self.root
            self.max_len = max(self.max_len, len(word))
            for char in reversed(word):
                if char not in node.children:
                    node.children[char] = TrieNode()
                node = node.children[char]
            node.is_word = True

    def query(self, letter: str) -> bool:
        self.stream.appendleft(letter)
        if len(self.stream) > self.max_len:
            self.stream.pop()
        
        node = self.root
        for char in self.stream:
            if char not in node.children:
                return False
            node = node.children[char]
            if node.is_word:
                return True
        return False
```

The Python implementation constructs a reversed Trie for efficient suffix checking. The `deque` is used to manage the recent characters efficiently, and each query checks only relevant characters up to the length of the longest word. This keeps each query efficient.

## Go Solution

```
type TrieNode struct {
    children map[byte]*TrieNode
    isWord   bool
}

type StreamChecker struct {
    root    *TrieNode
    stream  []byte
    maxLen  int
}

func Constructor(words []string) StreamChecker {
    root := &TrieNode{children: make(map[byte]*TrieNode)}
    maxLen := 0
    for _, word := range words {
        maxLen = max(maxLen, len(word))
        node := root
        for i := len(word) - 1; i >= 0; i-- {
            c := word[i]
            if node.children[c] == nil {
                node.children[c] = &TrieNode{children: make(map[byte]*TrieNode)}
            }
            node = node.children[c]
        }
        node.isWord = true
    }
    return StreamChecker{root: root, stream: []byte{}, maxLen: maxLen}
}

func (this *StreamChecker) Query(letter byte) bool {
    this.stream = append([]byte{letter}, this.stream...)
    if len(this.stream) > this.maxLen {
        this.stream = this.stream[:this.maxLen]
    }
    
    node := this.root
    for _, c := range this.stream {
        if node.children[c] == nil {
            return false
        }
        node = node.children[c]
        if node.isWord {
            return true
        }
    }
    return false
}

func max(a, b int) int {
    if a > b {
        return a
    }
    return b
}
```

In Go, a slice is used to simulate the deque behavior. The Trie is implemented with a map of byte to `TrieNode`, and the traversal logic is similar to Python. Go requires explicit slice management, so we truncate the slice when it exceeds the maximum word length.
