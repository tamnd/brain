## Solution

## Problem Understanding

The problem asks us to merge an array of stone piles into a single pile at minimum cost, where each pile has a certain number of stones. The key constraint is that in each move, you can only merge exactly `k` **consecutive** piles, and the cost of merging is equal to the total number of stones in those piles. The input is a list of integers `stones`, representing the number of stones in each pile, and an integer `k`, representing how many consecutive piles can be merged at a time. The output is the minimum cost to merge all piles into one, or `-1` if it is impossible.

Constraints such as `n <= 30` and `k <= 30` indicate that brute force may not scale, but dynamic programming is feasible. The problem also introduces an important feasibility condition: for it to be possible to merge all piles into one, `(n - 1) % (k - 1) == 0` must hold. Otherwise, merging cannot reduce the number of piles exactly to one.

Edge cases that can cause naive implementations to fail include cases where `k > n` (impossible to merge), or when the piles cannot reduce to exactly one pile due to the `(n - 1) % (k - 1)` condition. Another edge case is when all piles have the same number of stones or very small arrays (length 1 or 2).

## Approaches

The brute-force approach is to recursively try all possible sequences of merging `k` consecutive piles. At each step, we would sum the selected piles for the cost and recursively compute the cost for the remaining piles. This guarantees correctness because it explores all sequences, but it is extremely slow because the number of ways to merge piles grows combinatorially with `n`, making it infeasible for `n = 30`.

The key insight for a better solution is dynamic programming with memoization. We observe that merging subarrays has overlapping subproblems: the cost to merge a subarray into one or multiple piles can be reused. We define a DP state `dp[i][j][m]` as the minimum cost to merge the subarray `stones[i..j]` into `m` piles. Using prefix sums allows efficient computation of the sum of stones in any range, which is required for cost calculation. The DP recurrence considers splitting the interval into smaller intervals that can eventually be merged.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^n) | O(n) | Recursively try all merge sequences, too slow for n=30 |
| Optimal | O(n^3 * k) | O(n^2 * k) | DP with state dp[i][j][m] and prefix sums for range sums |

## Algorithm Walkthrough

1. First, check if `(n - 1) % (k - 1) != 0`. If true, return `-1` immediately because merging into one pile is impossible.
2. Compute a prefix sum array `prefix` where `prefix[i+1] = prefix[i] + stones[i]`. This allows O(1) sum queries for any subarray `[i, j]` as `prefix[j+1] - prefix[i]`.
3. Initialize a 3D DP array `dp[i][j][m]` representing the minimum cost to merge stones from index `i` to `j` into `m` piles. Initially, all values are infinity, except `dp[i][i][1] = 0` because a single pile requires no merge.
4. Iterate over all subarray lengths from 2 to `n`. For each subarray `[i, j]` and number of piles `m` from 2 to `k`, compute `dp[i][j][m]` by splitting the subarray into two parts `[i, mid]` and `[mid+1, j]` at positions `mid` that satisfy `(mid-i+1) % (k-1) == m-1` for merging into smaller sub-piles. Take the minimum sum of costs from these splits.
5. If `m == 1`, after merging into `k` piles internally, add the sum of stones in `[i, j]` to the DP value: `dp[i][j][1] = dp[i][j][k] + sum(i, j)`.
6. Finally, `dp[0][n-1][1]` contains the minimum cost to merge the entire array into one pile.

Why it works: The DP recurrence guarantees that every subarray is considered optimally, and by only considering splits that respect the `(k-1)` merging structure, we ensure that the moves are valid. Memoization prevents recalculation of overlapping subproblems.

## Python Solution

```
from typing import List

class Solution:
    def mergeStones(self, stones: List[int], k: int) -> int:
        n = len(stones)
        if (n - 1) % (k - 1) != 0:
            return -1

        prefix = [0] * (n + 1)
        for i in range(n):
            prefix[i + 1] = prefix[i] + stones[i]

        dp = [[[float('inf')] * (k + 1) for _ in range(n)] for _ in range(n)]
        for i in range(n):
            dp[i][i][1] = 0

        for length in range(2, n + 1):
            for i in range(n - length + 1):
                j = i + length - 1
                for m in range(2, k + 1):
                    for mid in range(i, j, k - 1):
                        dp[i][j][m] = min(dp[i][j][m], dp[i][mid][1] + dp[mid + 1][j][m - 1])
                if (length - 1) % (k - 1) == 0:
                    dp[i][j][1] = dp[i][j][k] + prefix[j + 1] - prefix[i]

        return dp[0][n - 1][1]
```

The implementation first checks feasibility. The prefix sums allow fast range queries. The 3D DP array keeps track of costs for merging subarrays into different numbers of piles. The nested loops iterate over all subarray lengths, positions, and target pile counts. When merging into one pile is valid, the total cost of the subarray is added. The final result is read from `dp[0][n-1][1]`.

## Go Solution

```
func mergeStones(stones []int, k int) int {
    n := len(stones)
    if (n-1)%(k-1) != 0 {
        return -1
    }

    prefix := make([]int, n+1)
    for i := 0; i < n; i++ {
        prefix[i+1] = prefix[i] + stones[i]
    }

    dp := make([][][]int, n)
    for i := 0; i < n; i++ {
        dp[i] = make([][]int, n)
        for j := 0; j < n; j++ {
            dp[i][j] = make([]int, k+1)
            for m := 0; m <= k; m++ {
                dp[i][j][m] = 1 << 60
            }
        }
        dp[i][i][1] = 0
    }

    for length := 2; length <= n; length++ {
        for i := 0; i <= n-length; i++ {
            j := i + length - 1
            for m := 2; m <= k; m++ {
                for mid := i; mid < j; mid += k - 1 {
                    dp[i][j][m] = min(dp[i][j][m], dp[i][mid][1]+dp[mid+1][j][m-1])
                }
            }
            if (length-1)%(k-1) == 0 {
                dp[i][j][1] = dp[i][j][k] + prefix[j+1] - prefix[i]
            }
        }
    }

    return dp[0][n-1][1]
}

func min(a, b int) int {
    if a < b {
        return a
    }
    return b
}
```

Go implementation is similar to Python but uses slices and an explicit `min` function. A large number (`1 << 60`) represents infinity. Prefix sums are computed similarly, and the DP array uses slices of slices. The approach avoids recursion entirely.
