## Problem Understanding

The problem asks us to find all customers who have purchased every single product listed in the `Product` table. We are given two tables: `Customer` and `Product`. The `Customer` table contains a `customer_id` and a `product_key` representing a purchase, and the `Product` table lists all possible products with unique `product_key`s.

Our goal is to return all `customer_id`s such that each customer in the result has bought every product from the `Product` table. The output can be in any order.

Important points to note include the possibility of duplicate rows in the `Customer` table. This means a customer might appear multiple times for the same product, but we should count unique products purchased. Additionally, `customer_id` is guaranteed to be non-null, and the `Product` table has unique keys.

Key edge cases include:

1. A customer has not purchased any product.
2. The `Customer` table is empty.
3. There is only one product in the `Product` table.
4. Customers purchase the same product multiple times.

These considerations guide us to a solution that handles uniqueness and completeness efficiently.

## Approaches

The brute-force approach is to check, for each customer, whether the set of products they bought matches the full set of products in the `Product` table. This works because the comparison ensures all required products are bought. However, this approach requires fetching all products for each customer and comparing sets, which becomes inefficient as the number of customers and products grows.

The key insight for a more optimal solution is that instead of comparing sets, we can leverage SQL aggregation. If we count the number of unique products each customer bought and compare it to the total number of products, we can immediately identify customers who bought all products. This removes the need for set comparisons and reduces data processing overhead.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(C * P) | O(P) | For each customer (C), build and compare sets of products against all products (P) |
| Optimal | O(C + P) | O(1) | Aggregate unique product purchases per customer and compare to total product count |

## Algorithm Walkthrough

1. Compute the total number of products in the `Product` table. This represents the threshold for a customer to have bought all products.
2. Group the `Customer` table by `customer_id` and count distinct `product_key`s for each customer. This gives the number of unique products each customer has purchased.
3. Compare each customer’s count to the total number of products. Include the customer in the result if the counts match exactly.
4. Return all customers that meet this condition.

Why it works: Counting distinct products ensures we ignore duplicate purchases. Comparing this count to the total product count guarantees the customer has bought every product exactly once or more. Aggregation efficiently handles all customers in a single query without iterative set comparisons.

## Python Solution

```
import pandas as pd

class Solution:
    def customersWhoBoughtAllProducts(self, customer: pd.DataFrame, product: pd.DataFrame) -> pd.DataFrame:
        # Step 1: Total number of unique products
        total_products = product['product_key'].nunique()
        
        # Step 2: Count distinct products per customer
        customer_counts = customer.groupby('customer_id')['product_key'].nunique().reset_index()
        customer_counts.columns = ['customer_id', 'product_count']
        
        # Step 3: Filter customers who bought all products
        result = customer_counts[customer_counts['product_count'] == total_products][['customer_id']]
        
        return result
```

The implementation first calculates the total number of products. Then it groups purchases by `customer_id` and counts unique products. Finally, it filters out customers whose product count matches the total number of products. This matches our algorithm walkthrough.

## Go Solution

```
package main

import (
    "database/sql"
    _ "github.com/go-sql-driver/mysql"
    "fmt"
)

func customersWhoBoughtAllProducts(db *sql.DB) ([]int, error) {
    query := `
        SELECT customer_id
        FROM Customer
        GROUP BY customer_id
        HAVING COUNT(DISTINCT product_key) = (SELECT COUNT(*) FROM Product)
    `
    rows, err := db.Query(query)
    if err != nil {
        return nil, err
    }
    defer rows.Close()
    
    var result []int
    for rows.Next() {
        var customerID int
        if err := rows.Scan(&customerID); err != nil {
            return nil, err
        }
        result = append(result, customerID)
    }
    return result, nil
}
```

In Go, the solution directly uses SQL aggregation. The `COUNT(DISTINCT product_key)` ensures unique product purchases are counted. SQL handles grouping and filtering, so Go just fetches results into a slice. Differences from Python include explicit query execution and result scanning.

## Edge Cases

1. **Customer purchases repeated products**. This could overcount products if we use naive counting. Using `COUNT(DISTINCT product_key)` prevents duplicates from inflating counts.
2. **Empty Customer table**. Some implementations may fail if no rows exist. The solution correctly returns an empty result using aggregation.
3. **Only one product in Product table**. Every customer who purchased it should be returned. This is handled by comparing against the total product count, which could be 1, and no special case is required.

This solution is robust, handles duplicates, empty inputs, and minimal product lists efficiently.
