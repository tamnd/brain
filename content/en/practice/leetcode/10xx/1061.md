## Solution

## Problem Understanding

This problem asks us to build groups of equivalent characters and then transform a given string into the lexicographically smallest possible equivalent string.

We are given three strings:

- `s1`
- `s2`
- `baseStr`

The strings `s1` and `s2` have the same length. For every index `i`, the characters `s1[i]` and `s2[i]` are considered equivalent. These equivalence relationships behave exactly like mathematical equivalence relations:

- A character is always equivalent to itself.
- If `a` is equivalent to `b`, then `b` is equivalent to `a`.
- If `a` is equivalent to `b`, and `b` is equivalent to `c`, then `a` is equivalent to `c`.

This means equivalence relationships form connected groups of characters.

For example:

- If `a == c`
- And `c == e`

Then `a`, `c`, and `e` all belong to the same equivalence class.

The task is to transform every character in `baseStr` into the smallest lexicographical character from its equivalence group. If a character has no equivalence relationship with anything else, it remains unchanged.

The important observation is that the equivalence relation creates connected components among the 26 lowercase English letters. Every letter in the same component can be replaced by the lexicographically smallest letter in that component.

The constraints are relatively small:

- String lengths are at most `1000`
- Only lowercase English letters are used

Since there are only 26 possible characters, we can use a very efficient Union-Find, also called Disjoint Set Union or DSU, structure.

Several edge cases are important:

- Characters may already be identical, such as `a == a`
- Equivalence chains may span multiple relationships
- Some characters in `baseStr` may never appear in `s1` or `s2`
- Multiple merge operations may connect already connected groups
- The smallest character in a group may appear late in the input

A naive implementation that repeatedly propagates equivalence information can become inefficient or error-prone. We need a structure that efficiently maintains connected groups while always preserving the lexicographically smallest representative.

## Approaches

### Brute Force Approach

A brute force solution would explicitly build equivalence groups by repeatedly propagating relationships until no more updates occur.

One way to do this is to use a graph:

- Each character is a node
- Every pair `(s1[i], s2[i])` creates an undirected edge

For every character in `baseStr`, we could run a DFS or BFS to find all reachable characters and then select the smallest one lexicographically.

This works because equivalence relationships form connected components, and graph traversal correctly finds all equivalent characters.

However, this approach is inefficient because we repeatedly perform graph traversals for characters in `baseStr`. Even though there are only 26 letters, repeatedly exploring the graph is unnecessary work.

### Key Insight

The important observation is that equivalence relationships form disjoint sets.

If two characters are connected through any sequence of equivalence relations, they belong to the same component.

This is exactly the kind of problem Union-Find is designed to solve.

Union-Find efficiently supports:

- Merging two sets
- Finding the representative of a set

The additional twist in this problem is that we want the lexicographically smallest character to become the representative of every group.

Instead of arbitrary unions, whenever we merge two groups:

- The smaller character becomes the parent
- The larger character points to it

This guarantees that every connected component is always represented by its smallest character.

Then transforming `baseStr` becomes straightforward:

- Find the representative of each character
- Replace the character with that representative

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(N + M × 26) | O(26) | Repeated graph traversals for each character |
| Optimal | O(N + M) | O(26) | Uses Union-Find with smallest-character representative |

Where:

- `N = len(s1)`
- `M = len(baseStr)`

## Algorithm Walkthrough

1. Initialize a Union-Find structure for all 26 lowercase letters.

Each character initially belongs to its own set. We can represent characters as integers from `0` to `25`.

For example:

- `'a' -> 0`
- `'b' -> 1`
- `'z' -> 25`
2. Create the parent array.

Initially:

```
parent[i] = i
```

This means every character is its own representative at the beginning.
3. Process all equivalence relationships from `s1` and `s2`.

For every index `i`:

- Convert `s1[i]` and `s2[i]` into integer indices
- Find their root representatives
- Merge the two sets
4. During union operations, always keep the lexicographically smaller root as the parent.

Suppose:

- Root of `'r'` is `'r'`
- Root of `'k'` is `'k'`

Since `'k' < 'r'`, we make:

```
parent['r'] = 'k'
```

This guarantees the smallest character becomes the representative for the entire group.
5. Use path compression in the `find` operation.

Path compression flattens the structure of the tree during searches, making future operations faster.

When finding a root:

- Recursively climb to the root
- Update all visited nodes to point directly to the root
6. Build the result string.

For every character in `baseStr`:

- Find its representative
- Convert the representative back to a character
- Append it to the answer
7. Return the final transformed string.

### Why it works

Union-Find guarantees that all equivalent characters belong to the same connected component.

By always attaching the larger root under the smaller root, the smallest lexicographical character becomes the representative of the entire component.

Therefore, when we replace each character in `baseStr` with its representative, we always obtain the lexicographically smallest possible equivalent character.

## Python Solution

```
class Solution:
    def smallestEquivalentString(self, s1: str, s2: str, baseStr: str) -> str:
        parent = list(range(26))

        def find(x: int) -> int:
            if parent[x] != x:
                parent[x] = find(parent[x])
            return parent[x]

        def union(a: int, b: int) -> None:
            root_a = find(a)
            root_b = find(b)

            if root_a == root_b:
                return

            if root_a < root_b:
                parent[root_b] = root_a
            else:
                parent[root_a] = root_b

        for ch1, ch2 in zip(s1, s2):
            idx1 = ord(ch1) - ord('a')
            idx2 = ord(ch2) - ord('a')
            union(idx1, idx2)

        result = []

        for ch in baseStr:
            idx = ord(ch) - ord('a')
            smallest = find(idx)
            result.append(chr(smallest + ord('a')))

        return "".join(result)
```

The implementation starts by initializing the `parent` array for all 26 lowercase letters. Every character initially points to itself, meaning each character forms its own equivalence class.

The `find` function locates the representative of a set. It also performs path compression, which improves efficiency by shortening future lookup paths.

The `union` function merges two equivalence groups. The key detail is that the smaller root always becomes the parent. This ensures that the root of every connected component is always the lexicographically smallest character in that component.

Next, the code processes every pair of equivalent characters from `s1` and `s2`, merging their groups.

Finally, the algorithm iterates through `baseStr`. For each character, it finds the smallest equivalent representative and appends it to the result.

The result list is joined into the final string and returned.

## Go Solution

```
func smallestEquivalentString(s1 string, s2 string, baseStr string) string {
	parent := make([]int, 26)

	for i := 0; i < 26; i++ {
		parent[i] = i
	}

	var find func(int) int

	find = func(x int) int {
		if parent[x] != x {
			parent[x] = find(parent[x])
		}
		return parent[x]
	}

	union := func(a int, b int) {
		rootA := find(a)
		rootB := find(b)

		if rootA == rootB {
			return
		}

		if rootA < rootB {
			parent[rootB] = rootA
		} else {
			parent[rootA] = rootB
		}
	}

	for i := 0; i < len(s1); i++ {
		idx1 := int(s1[i] - 'a')
		idx2 := int(s2[i] - 'a')
		union(idx1, idx2)
	}

	result := make([]byte, 0, len(baseStr))

	for i := 0; i < len(baseStr); i++ {
		idx := int(baseStr[i] - 'a')
		smallest := find(idx)
		result = append(result, byte(smallest)+'a')
	}

	return string(result)
}
```

The Go implementation follows the same algorithmic structure as the Python version.

A slice of integers is used for the parent array. The `find` function uses recursion and path compression exactly like the Python solution.

Since Go strings are byte sequences, character handling is done using byte arithmetic:

```
int(s[i] - 'a')
```

The result is constructed using a byte slice for efficiency, then converted back into a string at the end.

There are no concerns about integer overflow because all values remain within the range `0` to `25`.

## Edge Cases

One important edge case occurs when equivalence relationships form long transitive chains. For example:

```
a == b
b == c
c == d
```

A naive implementation might only track direct relationships and fail to realize that `a` and `d` are also equivalent. Union-Find handles this naturally because all connected characters eventually share the same root representative.

Another critical edge case is repeated or redundant unions. Inputs may repeatedly connect characters that are already in the same group. Without checking whether two roots are already identical, an implementation could accidentally corrupt the parent structure or perform unnecessary work. The `union` function safely returns early when both characters already belong to the same component.

A subtle edge case occurs when the lexicographically smallest character is introduced late in the merge sequence. For example:

```
d == e
e == f
a == d
```

If unions are not carefully implemented, the representative could incorrectly remain `d` instead of `a`. This implementation always attaches the larger root beneath the smaller root, guaranteeing that the smallest character eventually becomes the representative of the entire equivalence class.

Another important case involves characters in `baseStr` that never appear in `s1` or `s2`. These characters should remain unchanged because they are only equivalent to themselves. Since every character initially points to itself in the parent array, the implementation naturally handles this situation correctly.
