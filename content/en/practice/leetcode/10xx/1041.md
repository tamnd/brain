## Solution

## Problem Understanding

The problem asks us to determine whether a robot that follows a set of movement instructions indefinitely will remain confined within a finite circle on the plane. The robot starts at coordinates `(0, 0)` facing north. The instructions consist of the characters `"G"` (move forward 1 unit), `"L"` (turn left 90 degrees), and `"R"` (turn right 90 degrees). The robot executes these instructions repeatedly forever.

The input is a string of instructions of length up to 100 characters. The output is a boolean: `true` if the robot never escapes some circle, and `false` otherwise. The constraints tell us that the instruction set is small enough to simulate multiple iterations if needed, but since the instructions repeat infinitely, we need a reasoning-based approach rather than simulating forever.

Important edge cases include when the robot moves straight without turning (`"G"`) or turns in such a way that it forms cycles like `"GL"` or `"GGLLGG"`.

The key is to determine if after executing the instructions once, the robot either returns to the starting point or is facing a direction other than north. If either condition is true, repeating the instructions infinitely will keep it bounded.

## Approaches

The **brute-force approach** simulates the robot moving step by step for a large number of iterations, tracking its coordinates. This works because if the robot is bounded, it will eventually repeat positions or directions. However, simulating indefinitely is impossible, and even simulating hundreds of cycles is inefficient.

The **optimal approach** leverages a mathematical insight: the robot will be bounded if, after executing the instructions once, it either ends up at `(0, 0)` or its facing direction changes from north. The reasoning is that if the direction is not north, after at most four repetitions of the instructions, the robot will complete a loop and return to its starting area. Thus, we only need to simulate **one pass of the instructions** and check the position and direction.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(N * K) | O(1) | Simulate K repetitions (potentially large) of instructions to detect cycles |
| Optimal | O(N) | O(1) | Simulate a single instruction pass and check position and direction |

## Algorithm Walkthrough

1. Define four cardinal directions as vectors in the plane: north `(0, 1)`, east `(1, 0)`, south `(0, -1)`, west `(-1, 0)`. Use an index to track the current facing direction.
2. Initialize the robot's position at `(x, y) = (0, 0)` and facing north.
3. Iterate through the instruction string once.
4. For each `"G"`, move the robot 1 unit in the current direction by updating `(x, y)`.
5. For each `"L"`, turn left by updating the direction index counterclockwise.
6. For each `"R"`, turn right by updating the direction index clockwise.
7. After completing one pass of the instructions, check two conditions:

- If `(x, y)` is `(0, 0)`, the robot returned to the origin and is bounded.
- If the direction is not north, the robot's path will eventually form a cycle after repeating instructions, so it is bounded.
8. If neither condition is true, the robot moves indefinitely in the initial direction and is unbounded.

**Why it works:** The robot is guaranteed to either return to its starting position or change direction. If it returns to `(0, 0)` or changes direction, repeated execution will form a closed loop or cycle, keeping the robot bounded. If it moves straight and remains facing north, it will continue infinitely, making it unbounded.

## Python Solution

```
class Solution:
    def isRobotBounded(self, instructions: str) -> bool:
        # Directions: North, East, South, West
        directions = [(0, 1), (1, 0), (0, -1), (-1, 0)]
        x = y = 0
        dir_idx = 0  # Start facing North

        for instr in instructions:
            if instr == "G":
                dx, dy = directions[dir_idx]
                x += dx
                y += dy
            elif instr == "L":
                dir_idx = (dir_idx - 1) % 4
            elif instr == "R":
                dir_idx = (dir_idx + 1) % 4

        # Robot is bounded if it returns to origin or doesn't face North
        return (x == 0 and y == 0) or dir_idx != 0
```

The code defines the cardinal directions as vectors and uses `dir_idx` to track which direction the robot faces. Moving forward updates the position, and turning updates the direction index. After one pass, the return statement checks if the robot is bounded by the criteria discussed.

## Go Solution

```
func isRobotBounded(instructions string) bool {
    // Directions: North, East, South, West
    directions := [][2]int{{0, 1}, {1, 0}, {0, -1}, {-1, 0}}
    x, y := 0, 0
    dirIdx := 0 // Start facing North

    for _, instr := range instructions {
        switch instr {
        case 'G':
            x += directions[dirIdx][0]
            y += directions[dirIdx][1]
        case 'L':
            dirIdx = (dirIdx + 3) % 4 // Turning left
        case 'R':
            dirIdx = (dirIdx + 1) % 4 // Turning right
        }
    }

    return (x == 0 && y == 0) || dirIdx != 0
}
```

In Go, the logic mirrors Python. `dirIdx` keeps track of direction. The left turn uses `+3 % 4` to cycle counterclockwise, and right turn uses `+1 % 4` for clockwise. Position updates use array indexing for the direction vector.

## Worked Examples

### Example 1: `"GGLLGG"`

| Step | Instruction | Position (x,y) | Direction |
| --- | --- | --- | --- |
| 0 | Start | (0,0) | North |
| 1 | G | (0,1) | North |
| 2 | G | (0,2) | North |
| 3 | L | (0,2) | West |
| 4 | L | (0,2) | South |
| 5 | G | (0,1) | South |
| 6 | G | (0,0) | South |

Return `(x, y) == (0,0)` → bounded.

### Example 2: `"GG"`

| Step | Instruction | Position (x,y) | Direction |
| --- | --- | --- | --- |
| 0 | Start | (0,0) | North |
| 1 | G | (0,1) | North |
| 2 | G | (0,2) | North |

Return `(x, y) != (0,0)` and facing North → unbounded.

### Example 3: `"GL"`

| Step | Instruction | Position (x,y) | Direction |
| --- | --- | --- | --- |
| 0 | Start | (0,0) | North |
| 1 | G | (0,1) | North |
| 2 | L | (0,1) | West |

Facing not North → bounded.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(N) | Each instruction is processed once, N = length of instructions |
| Space | O(1) | Only fixed-size variables for position and direction are used |

The complexity is linear in the number of instructions because we only simulate one pass. Space is constant since we do not store past positions or directions.
