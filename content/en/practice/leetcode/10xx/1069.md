## Solution

## Problem Understanding

The problem asks us to compute the total quantity sold for each product across all years using data from two tables: `Sales` and `Product`. The `Sales` table records individual sales transactions, including the `product_id`, the `year`, the `quantity` sold, and the price per unit. The `Product` table maps `product_id` to a `product_name`.

The expected output is a table containing `product_id` and the aggregated `total_quantity` sold for that product. Importantly, if a product has no recorded sales, it is not required in the output. Each sale is uniquely identified by `(sale_id, year)` so there are no duplicate rows for the same sale, but multiple sales for the same product across different years or sales records are possible.

Edge cases to consider include products with zero sales (they should not appear in the output), products with multiple sales across years (quantities must be summed correctly), and handling of large numbers of sales efficiently. The problem guarantees valid `product_id` references, so no orphan sales exist.

## Approaches

A brute-force approach would involve manually iterating over all `Sales` records, grouping them by `product_id`, and summing the quantities using application code. This would work but is inefficient for large datasets, as it requires reading all rows into memory and processing them in code, rather than leveraging database aggregation.

The key insight for an optimal solution is that SQL provides a `GROUP BY` operation with an aggregate function. We can group the `Sales` table by `product_id` and use `SUM(quantity)` to compute the total quantity sold for each product directly in the database. This approach avoids manual iteration and is highly efficient because the database engine is optimized for such aggregation queries.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Iterate through all sales in application code and accumulate sums in a map/dictionary |
| Optimal | O(n) | O(1) | Use SQL `GROUP BY` with `SUM(quantity)` to compute totals directly in the database |

## Algorithm Walkthrough

1. Start by selecting the `product_id` column from the `Sales` table. This column identifies which product each sale belongs to.
2. Use the SQL `SUM()` function to calculate the total quantity sold per `product_id`. The `SUM()` function will automatically aggregate all rows with the same `product_id`.
3. Apply a `GROUP BY` clause on the `product_id` to ensure that the aggregation happens separately for each product.
4. Return the results with two columns: `product_id` and the calculated `total_quantity`.
5. There is no need to join with the `Product` table unless product names are needed. Products with zero sales are naturally excluded because they do not appear in the `Sales` table.

**Why it works:** The `GROUP BY` operation in SQL ensures that all sales belonging to the same product are aggregated together. Using `SUM(quantity)` guarantees that the total quantity across all years and sales entries is calculated correctly. Because we only rely on SQL aggregation, there is no risk of missing or double-counting sales.

## Python Solution

For LeetCode's database problems, Python code typically executes the SQL query via a database engine, but here we can provide a conceptual solution using `sqlite3` or `pandas` for demonstration.

```
# Python solution using SQL execution
def total_sales_query() -> str:
    return """
    SELECT product_id, SUM(quantity) AS total_quantity
    FROM Sales
    GROUP BY product_id
    """
```

**Explanation:** The query directly implements the steps from the algorithm. `SUM(quantity)` calculates the total sales for each product. `GROUP BY product_id` ensures that aggregation happens for each distinct product. This approach handles all products efficiently, including those with multiple sales entries.

## Edge Cases

1. **Product with zero sales:** A product present in the `Product` table but not in `Sales` will not appear in the output. This is correct because aggregation only occurs on existing sales. No join with `Product` is required unless explicitly asked.
2. **Multiple sales per year:** If a product has multiple sales in the same year, all quantities are summed. The `GROUP BY product_id` ensures year distinctions do not affect the total.
3. **Large quantity values:** Quantities can be large integers. SQL engines handle large sums natively, but if implementing in application code, use a data type capable of storing large integers to avoid overflow.

This solution is fully compliant with LeetCode's database expectations and handles all edge cases efficiently.
