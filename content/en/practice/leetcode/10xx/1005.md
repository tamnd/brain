## Solution

## Problem Understanding

The problem gives us an integer array `nums` and an integer `k`. In one operation, we may select any index `i` and replace `nums[i]` with its negated value, meaning a positive number becomes negative and a negative number becomes positive.

We must perform exactly `k` negations. The important detail is that the same index may be selected multiple times. This means we can flip a value back and forth if necessary.

Our goal is to maximize the final sum of the array after all `k` operations have been completed.

The input array may contain positive numbers, negative numbers, and zeros. Since negating a negative number increases the total sum, while negating a positive number decreases it, the main challenge is deciding which elements to flip and in what order.

The constraints are relatively small:

- `nums.length <= 10^4`
- `k <= 10^4`
- `nums[i]` is between `-100` and `100`

These constraints tell us that an `O(n log n)` sorting solution is perfectly acceptable. They also suggest that a greedy strategy is likely intended because we only need the maximum possible sum, not the sequence of operations itself.

Several edge cases are important:

- Arrays containing zeros, because flipping zero changes nothing and can absorb leftover operations.
- Situations where `k` is larger than the number of negative values.
- Cases where `k` remains odd after all negatives have been converted to positives.
- Arrays where all numbers are already positive.
- Arrays with repeated values or only one element.

A naive implementation can easily mishandle the leftover parity of `k`, especially when deciding whether an additional flip is required after all negative values are processed.

## Approaches

### Brute Force Approach

A brute force solution would try every possible sequence of `k` negations and compute the resulting array sum. For each operation, we can choose any of the `n` indices, so the total number of possible operation sequences is `n^k`.

This guarantees correctness because every possible combination is explored, but it becomes computationally impossible even for modest input sizes. With `n = 10^4` and `k = 10^4`, exhaustive search is completely infeasible.

Even a slightly optimized brute force approach that repeatedly searches for the best local flip can still become inefficient if implemented poorly.

### Key Insight

The key observation is that negating a negative number always increases the total array sum.

For example:

- Flipping `-5` to `5` increases the sum by `10`
- Flipping `-1` to `1` increases the sum by `2`

This means we should prioritize converting the most negative values first.

Sorting the array allows us to process negative numbers in increasing order. We flip each negative number while we still have remaining operations.

After all negatives are handled, two cases remain:

- If `k` is even, additional flips cancel each other out and the sum remains unchanged.
- If `k` is odd, we must flip one element one final time. To minimize damage, we flip the element with the smallest absolute value.

This greedy strategy works because every decision is locally optimal and does not interfere with future optimal choices.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^k) | O(k) | Tries every possible sequence of negations |
| Optimal | O(n log n) | O(1) or O(log n) | Sorts array and greedily flips negatives |

## Algorithm Walkthrough

1. Sort the array in ascending order.

Sorting places the most negative values at the beginning. Since negating a negative value increases the sum, we want to process these values first.
2. Iterate through the array from left to right.

For each number:

- If the number is negative and we still have remaining operations (`k > 0`), negate it.
- Decrement `k` after each negation.

This greedily converts as many negative values as possible into positive values.
3. Track the smallest absolute value in the array.

After all useful negative flips are complete, we may still have leftover operations. If the remaining `k` is odd, one extra flip must occur.

Flipping the smallest absolute value minimizes the reduction in the total sum.
4. Compute the sum of the array.

Once all required operations are accounted for, compute the final total.
5. Handle leftover odd `k`.

If `k % 2 == 1`, subtract twice the smallest absolute value from the sum.

This simulates flipping the smallest magnitude element exactly once.

### Why it works

The algorithm works because negating a negative number always increases the sum more than negating any less negative number. Therefore, greedily converting negatives to positives is always optimal.

After all negatives are processed, any additional pair of negations cancels out. Only the parity of the remaining `k` matters. If one unavoidable extra flip remains, minimizing the damage requires flipping the smallest absolute value.

This greedy invariant guarantees the maximum possible final sum.

## Python Solution

```
from typing import List

class Solution:
    def largestSumAfterKNegations(self, nums: List[int], k: int) -> int:
        nums.sort()

        for i in range(len(nums)):
            if nums[i] < 0 and k > 0:
                nums[i] = -nums[i]
                k -= 1

        total_sum = sum(nums)
        smallest_abs = min(abs(num) for num in nums)

        if k % 2 == 1:
            total_sum -= 2 * smallest_abs

        return total_sum
```

The implementation begins by sorting the array so that negative numbers appear first. This allows the algorithm to greedily process the most beneficial negations immediately.

The loop traverses the sorted array and flips every negative number while operations remain. Each negation consumes one operation by decrementing `k`.

After the loop completes, the array contains as many converted positive values as possible. The code then computes the total array sum and identifies the smallest absolute value.

If `k` is still odd, one additional flip is unavoidable. Rather than physically modifying the array again, the code directly adjusts the sum by subtracting twice the smallest absolute value. This works because flipping a value `x` changes the total by `-2x`.

Finally, the computed maximum sum is returned.

## Go Solution

```
package main

import (
	"sort"
)

func largestSumAfterKNegations(nums []int, k int) int {
	sort.Ints(nums)

	for i := 0; i < len(nums); i++ {
		if nums[i] < 0 && k > 0 {
			nums[i] = -nums[i]
			k--
		}
	}

	totalSum := 0
	smallestAbs := abs(nums[0])

	for _, num := range nums {
		totalSum += num

		if abs(num) < smallestAbs {
			smallestAbs = abs(num)
		}
	}

	if k%2 == 1 {
		totalSum -= 2 * smallestAbs
	}

	return totalSum
}

func abs(x int) int {
	if x < 0 {
		return -x
	}
	return x
}
```

The Go implementation follows the same algorithmic structure as the Python version.

The array is sorted using `sort.Ints`. Since Go does not provide a built in absolute value function for integers, a helper `abs` function is implemented manually.

Unlike Python, Go does not have generator expressions, so the total sum and smallest absolute value are computed using an explicit loop.

There are no special concerns regarding integer overflow because the constraints are small enough to fit comfortably within Go's standard integer range.

## Edge Cases

One important edge case occurs when the array contains a zero. Since negating zero has no effect, any leftover operations can effectively be wasted on that element. This is especially important when `k` remains odd after all negative numbers have been processed. Without special handling, a naive implementation might unnecessarily flip a positive value and reduce the sum. The current implementation naturally handles this because the smallest absolute value becomes zero.

Another critical edge case happens when all numbers are already positive. In this scenario, flipping any value decreases the total sum. If `k` is even, repeated flips can cancel out and the original sum remains optimal. If `k` is odd, exactly one element must ultimately remain negated, so the algorithm flips the smallest absolute value to minimize the loss.

Arrays containing more negative numbers than available operations are also important. In such cases, we cannot flip every negative value. The greedy strategy ensures we flip the most negative numbers first because those provide the greatest increase in the final sum. Sorting guarantees these values are processed in the correct order.

A final subtle case appears when the smallest absolute value changes after negations. For example, flipping `-1` to `1` preserves the same absolute value. The implementation correctly computes the smallest absolute value after all primary negations are complete, ensuring the final parity adjustment is always optimal.
