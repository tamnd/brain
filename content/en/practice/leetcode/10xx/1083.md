## Solution

## Problem Understanding

This problem is asking us to identify **buyers who purchased a specific product (`S8`) but never purchased another specific product (`iPhone`)**. We are given two tables: `Product` and `Sales`. The `Product` table contains `product_id`, `product_name`, and `unit_price`. The `Sales` table records individual sales with `seller_id`, `product_id`, `buyer_id`, `sale_date`, `quantity`, and `price`.

The input represents all products and all sales made. Each row in `Sales` indicates that a buyer purchased a certain quantity of a product at a certain price on a particular date.

The expected output is a table of `buyer_id` values who bought `S8` but **never bought `iPhone`**. The output order does not matter.

Constraints to note are: every `buyer_id` is non-null, every `sale_date` is non-null, `product_id` in `Sales` references a valid product in `Product`, and there could be repeated rows in `Sales`. Edge cases include buyers who bought neither `S8` nor `iPhone`, buyers who bought only `iPhone`, and buyers who bought both products.

## Approaches

The brute-force approach involves first finding all buyers of `S8` and all buyers of `iPhone` separately, then performing a set difference to find buyers in the first set but not the second. This works correctly but involves multiple full scans and can be slow if tables are very large.

The key observation for an optimal solution is that we can **use SQL joins and `NOT IN` or `EXCEPT` queries** to perform this in a single query efficiently. We join `Sales` to `Product` to identify the correct products, then use a subquery to exclude any buyers who purchased `iPhone`.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(N + M) | O(B) | Find S8 buyers and iPhone buyers separately, then compute difference. N is Sales size, M is Product size, B is distinct buyers. |
| Optimal | O(N) | O(B) | Single pass query with JOINs and NOT IN subquery. Efficient and concise. |

## Algorithm Walkthrough

1. Identify the `product_id` values for `S8` and `iPhone` from the `Product` table. This allows us to target these products without hardcoding IDs.
2. Select all distinct `buyer_id` values from the `Sales` table where `product_id` matches the `S8` ID. This gives the set of buyers who bought `S8`.
3. Select all distinct `buyer_id` values from the `Sales` table where `product_id` matches the `iPhone` ID. This gives the set of buyers who bought `iPhone`.
4. Perform a set difference or `NOT IN` operation to exclude any buyers who are in the iPhone buyers set from the S8 buyers set.
5. Return the resulting `buyer_id` values as the answer.

**Why it works:** The algorithm guarantees correctness because we explicitly select only buyers of `S8` and remove any buyer who appears in the iPhone buyers set. This ensures that only buyers who purchased S8 and never purchased iPhone are returned.

## Python Solution

```
# Since this is a SQL problem, we use Python with a SQL execution framework.
# On LeetCode, you would directly write SQL, but here is a Python representation.

import sqlite3
from typing import List, Tuple

def buyers_s8_not_iphone(conn: sqlite3.Connection) -> List[Tuple[int]]:
    query = """
    SELECT DISTINCT s1.buyer_id
    FROM Sales s1
    JOIN Product p1 ON s1.product_id = p1.product_id
    WHERE p1.product_name = 'S8'
      AND s1.buyer_id NOT IN (
          SELECT s2.buyer_id
          FROM Sales s2
          JOIN Product p2 ON s2.product_id = p2.product_id
          WHERE p2.product_name = 'iPhone'
      )
    """
    cursor = conn.cursor()
    cursor.execute(query)
    return cursor.fetchall()
```

The code joins `Sales` with `Product` to identify the correct product names. We select buyers of `S8` and exclude any buyers who appear in the subquery selecting iPhone buyers. The `DISTINCT` keyword ensures we do not return duplicates.

## Go Solution

```
package main

import (
    "database/sql"
    _ "github.com/mattn/go-sqlite3"
)

func BuyersS8NotIphone(db *sql.DB) ([]int, error) {
    query := `
    SELECT DISTINCT s1.buyer_id
    FROM Sales s1
    JOIN Product p1 ON s1.product_id = p1.product_id
    WHERE p1.product_name = 'S8'
      AND s1.buyer_id NOT IN (
          SELECT s2.buyer_id
          FROM Sales s2
          JOIN Product p2 ON s2.product_id = p2.product_id
          WHERE p2.product_name = 'iPhone'
      )
    `
    rows, err := db.Query(query)
    if err != nil {
        return nil, err
    }
    defer rows.Close()

    var buyers []int
    for rows.Next() {
        var buyerID int
        if err := rows.Scan(&buyerID); err != nil {
            return nil, err
        }
        buyers = append(buyers, buyerID)
    }
    return buyers, nil
}
```

The Go implementation is very similar to Python, but we iterate through `rows` to append buyer IDs to a slice. Proper error handling and `defer rows.Close()` ensure resources are released.

| --- | --- | --- |

The time complexity is linear in the size of the `Sales` table, which is optimal for this problem. Space usage is proportional to the number of distinct buyers, which is usually smaller than the total number of sales.
