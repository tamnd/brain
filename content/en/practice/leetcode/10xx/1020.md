**Difficulty:** 🟡 Medium  
**Topics:** Array, Depth-First Search, Breadth-First Search, Union-Find, Matrix

## Solution

## Problem Understanding

The problem gives us a binary matrix called `grid`, where:

- `0` represents water
- `1` represents land

We can move only in the four cardinal directions, up, down, left, and right. A land cell is considered an enclave if there is no path from that cell to any boundary of the grid through connected land cells.

In other words, we want to count how many land cells are completely trapped inside the grid and cannot eventually "escape" by walking off the boundary.

The key observation is that any land cell connected to a boundary land cell is not an enclave. Even if the cell itself is not on the edge, as long as there exists a path of land cells leading to the boundary, it can escape.

The input constraints are important:

- `m` and `n` can each be as large as `500`
- The total number of cells can therefore reach `250,000`

This immediately tells us that expensive repeated traversals are dangerous. An `O((m*n)^2)` solution would likely time out. We need an algorithm close to linear in the number of cells.

Several edge cases are important:

- A grid with no land at all
- A grid where every land cell touches the boundary
- A single-cell grid
- Large connected components that partially touch the boundary
- Islands fully enclosed by water

A naive implementation can easily double count cells, revisit the same region many times, or fail to properly distinguish enclosed regions from boundary-connected regions.

## Approaches

### Brute Force Approach

A straightforward approach is to examine every land cell independently.

For each land cell:

1. Start a DFS or BFS.
2. Explore all connected land cells.
3. Determine whether any cell in that connected component touches the boundary.
4. If the component never touches the boundary, count all cells in it as enclaves.

This works because each connected component of land is either:

- connected to the boundary, or
- fully enclosed

However, the brute-force approach becomes inefficient if we repeat searches for many cells. Without careful memoization, the same connected component may be traversed multiple times.

For example, if a large island contains 10,000 cells, we may repeatedly explore the same 10,000-cell component from many starting points.

This leads to extremely poor performance in the worst case.

### Optimal Approach

Instead of trying to identify enclosed land directly, we reverse the perspective.

A land cell is not an enclave if it can reach the boundary.

Therefore:

1. Start from all boundary land cells.
2. Flood-fill all land connected to the boundary.
3. Mark those cells as visited or convert them into water.
4. At the end, every remaining land cell must be enclosed.

This is much more efficient because every cell is processed at most once.

The key insight is that it is easier to identify cells that can escape than cells that cannot.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O((m*n)^2) | O(m*n) | Repeatedly traverses the same connected components |
| Optimal | O(m*n) | O(m*n) | Flood-fill from the boundary once |

## Algorithm Walkthrough

### Optimal Boundary Flood-Fill Algorithm

1. Traverse the boundary of the grid and locate every land cell.

Boundary cells are the only cells that can directly escape the grid. Any land connected to these cells is also not enclosed.
2. For each boundary land cell, perform DFS or BFS.

During traversal:

- Visit all connected land cells
- Mark them as visited
- Optionally convert them to water

This removes every land cell that can reach the boundary.
3. Continue until all reachable boundary-connected land has been removed.

After this process, the only remaining land cells are those trapped inside the grid.
4. Traverse the entire grid one final time.

Count all remaining land cells.
5. Return the count.

These remaining cells are exactly the enclaves.

### Why it works

The algorithm relies on a simple invariant:

- Any land cell connected to the boundary is not an enclave.
- Any land cell not connected to the boundary must be enclosed.

By removing all boundary-reachable land first, the remaining land cells are guaranteed to be enclosed. Since every cell is visited at most once, the algorithm is both correct and efficient.

## Python Solution

```
from typing import List

class Solution:
    def numEnclaves(self, grid: List[List[int]]) -> int:
        rows = len(grid)
        cols = len(grid[0])

        def dfs(r: int, c: int) -> None:
            if (
                r < 0 or r >= rows or
                c < 0 or c >= cols or
                grid[r][c] == 0
            ):
                return

            grid[r][c] = 0

            dfs(r + 1, c)
            dfs(r - 1, c)
            dfs(r, c + 1)
            dfs(r, c - 1)

        for r in range(rows):
            if grid[r][0] == 1:
                dfs(r, 0)

            if grid[r][cols - 1] == 1:
                dfs(r, cols - 1)

        for c in range(cols):
            if grid[0][c] == 1:
                dfs(0, c)

            if grid[rows - 1][c] == 1:
                dfs(rows - 1, c)

        enclave_count = 0

        for r in range(rows):
            for c in range(cols):
                if grid[r][c] == 1:
                    enclave_count += 1

        return enclave_count
```

The implementation begins by defining the grid dimensions. A nested `dfs` function is used to remove all land connected to the boundary.

The DFS immediately returns if:

- the coordinates are outside the grid
- the cell is water
- the cell has already been visited

Instead of maintaining a separate visited matrix, the algorithm modifies the input grid directly by changing visited land cells from `1` to `0`. This reduces extra memory usage and simplifies the logic.

Next, the algorithm scans all four boundaries:

- left column
- right column
- top row
- bottom row

Whenever a boundary land cell is found, DFS removes the entire connected component.

After all boundary-connected land has been erased, the algorithm performs one final traversal of the grid and counts the remaining `1`s.

Those remaining land cells are enclaves.

## Go Solution

```
func numEnclaves(grid [][]int) int {
    rows := len(grid)
    cols := len(grid[0])

    var dfs func(int, int)

    dfs = func(r int, c int) {
        if r < 0 || r >= rows || c < 0 || c >= cols || grid[r][c] == 0 {
            return
        }

        grid[r][c] = 0

        dfs(r+1, c)
        dfs(r-1, c)
        dfs(r, c+1)
        dfs(r, c-1)
    }

    for r := 0; r < rows; r++ {
        if grid[r][0] == 1 {
            dfs(r, 0)
        }

        if grid[r][cols-1] == 1 {
            dfs(r, cols-1)
        }
    }

    for c := 0; c < cols; c++ {
        if grid[0][c] == 1 {
            dfs(0, c)
        }

        if grid[rows-1][c] == 1 {
            dfs(rows-1, c)
        }
    }

    enclaveCount := 0

    for r := 0; r < rows; r++ {
        for c := 0; c < cols; c++ {
            if grid[r][c] == 1 {
                enclaveCount++
            }
        }
    }

    return enclaveCount
}
```

The Go implementation closely mirrors the Python version.

A recursive closure is used for DFS. Since Go does not support nested named functions in the same way as Python, we declare a function variable first and then assign the recursive function to it.

Like the Python solution, the grid itself is modified in place to mark visited cells.

There are no integer overflow concerns because the maximum number of cells is only `250,000`, well within Go's `int` range.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m*n) | Every cell is visited at most once |
| Space | O(m*n) | Recursive DFS stack can reach all cells in worst case |

The algorithm processes each cell only once during DFS and once during the final counting pass. Therefore, the total work is linear in the size of the grid.

The recursion stack may grow to `O(m*n)` in the worst case if the entire grid is land and forms one connected component. An iterative BFS or DFS could reduce recursion risk but would still require linear auxiliary space in the worst case.

## Edge Cases

### Single Cell Grid

A `1 x 1` grid is an important edge case because the only cell is automatically on the boundary. If that cell contains land, it is not an enclave because you can immediately walk off the grid.

The implementation handles this naturally because boundary scanning includes every cell in the grid.

### Entire Grid Filled with Land

Consider a grid where every cell is `1`.

Although many cells are far from the edge, every land cell is connected to boundary land through adjacent paths. Therefore, no enclaves exist.

A buggy implementation might incorrectly count interior cells without considering connectivity. The flood-fill approach avoids this by removing the entire connected component starting from the boundary.

### Isolated Interior Islands

A grid may contain land completely surrounded by water and disconnected from every boundary.

For example:

```
[
  [0,0,0],
  [0,1,0],
  [0,0,0]
]
```

This is the smallest true enclave.

The implementation correctly preserves this cell because no DFS ever reaches it from the boundary. It remains `1` during the final counting phase.

### Multiple Disconnected Components

The grid may contain several separate islands, some boundary-connected and some enclosed.

The algorithm correctly processes all of them because boundary DFS removes only reachable land. Independent enclosed components remain untouched and are counted at the end.
