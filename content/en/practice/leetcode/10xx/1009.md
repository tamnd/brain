## Solution

## Problem Understanding

The problem asks us to compute the complement of a non-negative integer. The complement is formed by flipping every bit in the binary representation of the number. Every `0` becomes `1`, and every `1` becomes `0`.

For example, the number `5` in binary is `101`. Flipping each bit produces `010`, which equals `2` in decimal form.

A very important detail is that we only flip the bits that appear in the number's actual binary representation. We do not consider leading zeroes. For example:

- `5` is represented as `101`, not `00000101`
- Its complement is therefore `010`, not `11111010`

The input consists of a single integer `n`, where:

```
0 <= n < 10^9
```

This constraint tells us several useful things:

- The input easily fits within standard 32-bit integer ranges
- The binary representation has at most about 30 bits
- We can safely use bit manipulation operations efficiently
- Even a linear scan over all bits is extremely fast

One important edge case is `n = 0`.

Normally, `0` in binary is represented as `"0"`. Flipping that single bit gives `"1"`, so the correct answer is `1`.

This special case can easily break implementations that rely on finding the highest set bit, because `0` has no set bits at all.

Another subtle issue is that using the bitwise NOT operator directly, such as `~n`, does not work by itself in most programming languages. This is because integers are stored with fixed widths using two's complement representation, so `~5` does not simply produce `2`. We must carefully limit the inversion to only the significant bits of the number.

## Approaches

### Brute Force Approach

A straightforward way to solve the problem is to manually process the binary representation bit by bit.

We can repeatedly extract the least significant bit using modulo or bit operations, flip it, and build the resulting number. Specifically:

- Read each bit from right to left
- If the bit is `0`, place `1` in the result
- If the bit is `1`, place `0` in the result
- Continue until all bits are processed

This works because every bit is independently inverted.

The brute force approach is correct, but it involves explicit bit-by-bit reconstruction of the answer. While the input size is small enough that this still runs quickly, it is more verbose and less elegant than necessary.

### Optimal Approach

The key insight is that we can create a bitmask containing only `1`s for the length of the number's binary representation.

For example:

- `5 = 101`
- The corresponding mask is `111`

Once we have this mask, we can XOR the number with it:

```
101
111
---
010
```

This works because XOR with `1` flips a bit:

- `0 ^ 1 = 1`
- `1 ^ 1 = 0`

The only challenge is constructing the correct mask.

If the number has `k` bits, then the mask should be:

```
(1 << k) - 1
```

For example:

- `5` has 3 bits
- `(1 << 3) - 1 = 8 - 1 = 7`
- `7` in binary is `111`

This produces an elegant and efficient solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(log n) | O(1) | Processes each bit individually |
| Optimal | O(log n) | O(1) | Uses a bitmask and XOR to flip all bits at once |

## Algorithm Walkthrough

1. Handle the special case where `n == 0`.

The binary representation of `0` is `"0"`. Flipping that single bit gives `"1"`, so we immediately return `1`.
2. Determine how many bits are present in the binary representation of `n`.

We repeatedly shift the number right until it becomes zero. Each shift removes one bit, so the number of shifts tells us the bit length.
3. Construct a mask containing only `1`s for that bit length.

If the number has `k` bits, the mask is:

$(1 \ll k) - 1$

This produces a binary value like:

- `1 bit  -> 1`
- `3 bits -> 111`
- `5 bits -> 11111`
4. XOR the original number with the mask.

XOR flips every bit wherever the mask contains `1`.

Since the mask contains `1`s only in the significant bit positions, only those bits are inverted.
5. Return the resulting value.

### Why it works

The algorithm works because the mask contains `1`s exactly where the original number has meaningful binary digits. XOR with `1` flips a bit, and XOR with `0` leaves a bit unchanged. Since the mask only spans the significant bits, the algorithm flips exactly the required bits and ignores all leading zeroes.

## Python Solution

```
class Solution:
    def bitwiseComplement(self, n: int) -> int:
        if n == 0:
            return 1

        bit_length = 0
        temp = n

        while temp > 0:
            bit_length += 1
            temp >>= 1

        mask = (1 << bit_length) - 1

        return n ^ mask
```

The implementation begins by handling the special case `n == 0`. Without this check, the bit-length calculation would produce zero bits, which would create an invalid mask.

Next, the code computes the number of bits in `n`. The variable `temp` is repeatedly shifted right until it becomes zero. Each shift removes one binary digit from the right side, and `bit_length` counts how many digits existed.

After determining the bit length, the code constructs the mask using:

```
(1 << bit_length) - 1
```

If `bit_length` is `3`, then:

```
1 << 3 = 1000
1000 - 1 = 111
```

Finally, the code XORs the original number with the mask. Since XOR with `1` flips a bit, this operation produces the complement directly.

## Go Solution

```
func bitwiseComplement(n int) int {
    if n == 0 {
        return 1
    }

    bitLength := 0
    temp := n

    for temp > 0 {
        bitLength++
        temp >>= 1
    }

    mask := (1 << bitLength) - 1

    return n ^ mask
}
```

The Go implementation follows the same logic as the Python version.

One small difference is variable declaration syntax. Go uses explicit type inference with `:=`.

Go integers are fixed-width values internally, but since the problem constraint is less than `10^9`, integer overflow is not a concern here.

The right-shift operator `>>` behaves exactly as needed for positive integers.

## Edge Cases

### Edge Case 1, `n = 0`

This is the most important edge case in the problem.

The number `0` has binary representation `"0"`. Flipping that bit should produce `"1"`.

Many implementations fail here because they attempt to determine the highest set bit, but `0` has no set bits. That can lead to a mask of `0`, producing the incorrect result `0`.

The implementation handles this by explicitly checking:

```
if n == 0:
    return 1
```

### Edge Case 2, Numbers Consisting Entirely of `1`s

Inputs like:

```
1  -> 1
3  -> 11
7  -> 111
15 -> 1111
```

all become zero after inversion.

A buggy implementation might accidentally flip extra leading bits and produce negative numbers or large unexpected values.

The mask-based approach avoids this because it limits inversion strictly to the significant bits.

### Edge Case 3, Powers of Two

Numbers like:

```
2  -> 10
4  -> 100
8  -> 1000
16 -> 10000
```

contain exactly one set bit.

These cases are useful because the complement becomes a sequence of all `1`s beneath the leading bit:

```
1000 -> 0111
```

The algorithm handles this correctly because the mask spans the exact bit width of the number.
