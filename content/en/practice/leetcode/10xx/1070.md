## Solution

## Problem Understanding

The problem presents a `Sales` table, where each row represents a sale of a specific product in a given year, including the quantity sold and the price per unit. Each sale is uniquely identified by the combination of `sale_id` and `year`.

The goal is to determine, for each product, which year was the first year it was sold, and then return all sales entries that occurred in that first year. The output table must include the `product_id`, the `first_year`, and the `quantity` and `price` from the sales records of that year.

In other words, for each product, we are filtering its sales to only those that match its earliest year of sale.

Important constraints and implications include that a product may have multiple sales in the same year, but we only want the sales from the earliest year. The table is expected to be moderately sized (typical SQL tables), so the solution should avoid unnecessary full-table scans or subqueries that compute minimums repeatedly.

Edge cases include products with a single sale, products with multiple sales in the same first year, and non-consecutive years of sales for a product.

## Approaches

A brute-force approach would first identify all distinct `product_id`s, then for each product, scan all its sales to find the minimum year, and finally filter sales by that minimum year. This is correct because it explicitly finds the first year and then selects the relevant sales. However, this approach is inefficient because it requires multiple passes over the data and nested operations, which can be costly for large datasets.

The optimal approach leverages SQL's grouping and join features. We first compute the first year for each product using a `GROUP BY` and `MIN(year)` aggregation. Then we join this result back with the original `Sales` table to filter only the rows matching each product's first year. This method avoids repeated scans and takes advantage of SQL's set-based operations, which are more efficient.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n*m) | O(n) | Iterate per product and per sale, multiple scans |
| Optimal | O(n) | O(n) | Single aggregation and join, leverages SQL optimizations |

## Algorithm Walkthrough

1. Aggregate the `Sales` table by `product_id` to find the minimum `year` for each product. This identifies the first year each product was sold.
2. Create a derived table or temporary table that contains `product_id` and `first_year` as a result of the aggregation.
3. Join the original `Sales` table with this aggregated table on `product_id` and `year = first_year`. This ensures only sales from the first year are included.
4. Select the required columns: `product_id`, `first_year`, `quantity`, and `price`.
5. Return the result in any order.

Why it works: the join ensures that only rows from the `Sales` table that match the computed first year for each product are included. Since the first year is guaranteed to be correct via the `MIN(year)` aggregation, all included rows satisfy the problem's condition.

## Python Solution

```
# Python does not directly interact with SQL tables on LeetCode,
# but we can provide a SQL execution string or pandas equivalent.
# For LeetCode, you would submit SQL directly:

def product_sales_analysis():
    query = """
    SELECT s.product_id, f.first_year, s.quantity, s.price
    FROM Sales s
    JOIN (
        SELECT product_id, MIN(year) AS first_year
        FROM Sales
        GROUP BY product_id
    ) f
    ON s.product_id = f.product_id AND s.year = f.first_year
    """
    return query
```

This Python function returns the SQL query string that implements the optimal approach. The inner query aggregates by `product_id` to compute `first_year`, and the outer query joins it back with the `Sales` table to select only relevant rows.

## Edge Cases

The first edge case is when a product has only one sale record. This ensures that the aggregation and join still correctly return that row. The implementation works because `MIN(year)` for a single-row product equals that row's year, and the join matches it directly.

A second edge case is when multiple sales occur in the same first year for a product. Naive implementations might return only one sale, but the join ensures all matching rows are included.

A third edge case is when a product has non-consecutive sales years, e.g., 2008, 2010, 2012. The algorithm correctly identifies 2008 as the first year and only returns rows with that year, ignoring later sales. This guarantees correctness regardless of gaps in sales data.
