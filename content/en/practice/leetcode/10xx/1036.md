## Solution

## Problem Understanding

LeetCode 1036, Escape a Large Maze, asks whether it is possible to travel from a starting position to a target position on an extremely large grid while avoiding blocked cells.

The grid dimensions are `1,000,000 x 1,000,000`, meaning coordinates range from `0` to `999999` inclusive in both directions. Movement is allowed only in the four cardinal directions: up, down, left, and right. Diagonal movement is not allowed.

The input consists of three parts:

- `blocked`, a list of blocked coordinates that cannot be entered
- `source`, the starting coordinate
- `target`, the destination coordinate

The goal is to determine whether there exists at least one valid path from `source` to `target`.

At first glance, this looks like a standard graph traversal problem where each grid cell is a node and adjacent cells are connected. A natural idea would be to run Breadth-First Search (BFS) or Depth-First Search (DFS) from the source until the target is found.

However, the constraints completely change the nature of the problem.

The grid contains up to:

$10^6 \times 10^6 = 10^{12}$

cells, which is far too large for any traversal that explores the entire space.

Fortunately, the number of blocked cells is extremely small:

$|blocked| \leq 200$

This constraint is the key insight. With only 200 blocked cells, it is impossible to wall off a large portion of the grid. The only realistic danger is that the source or target becomes enclosed inside a relatively small trapped region.

Important edge cases include situations where:

- The source is immediately surrounded by blocked cells
- The target is trapped even if the source is not
- There are no blocked cells at all
- The blocked cells form a partial enclosure but leave an escape path
- The source and target are very far apart, making naive traversal infeasible

The problem also guarantees that:

- `source != target`
- Neither source nor target is blocked
- All coordinates are valid grid positions

These guarantees simplify implementation because we never need to handle invalid or degenerate starting conditions.

## Approaches

### Brute Force Approach

The most straightforward solution is to perform a full BFS or DFS starting from the source and continue until either:

- The target is reached
- All reachable cells are exhausted

This works because BFS and DFS correctly explore all reachable states in an unweighted graph.

Unfortunately, the grid is enormous. In the worst case, the traversal could potentially explore millions or billions of cells before determining reachability.

Even though only 200 cells are blocked, the empty space is huge. Traversing large open regions is computationally impossible within reasonable limits.

A brute-force BFS therefore has unacceptable time and memory requirements.

### Key Insight

The crucial observation is that with at most 200 blocked cells, the blocked region can only enclose a limited area.

A finite number of walls cannot isolate a massive portion of a 1,000,000 by 1,000,000 grid. The maximum enclosed area created by `n` blocked cells is bounded.

For this problem, the important mathematical bound is:

$\frac{n(n-1)}{2}$

where `n = len(blocked)`.

If during BFS we manage to explore more than this many cells, then we know we are not trapped inside an enclosure. Once we escape any potential enclosure, the huge open grid guarantees eventual reachability unless the target itself is trapped.

This leads to the optimal strategy:

- Run a limited BFS from the source
- Run a limited BFS from the target
- If neither side is enclosed, then a path exists

The second BFS is necessary because the source may escape its enclosure while the target remains trapped inside another enclosure.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(10^12) | O(10^12) | Potentially explores enormous portions of the grid |
| Optimal | O(n²) | O(n²) | Explores only bounded regions around blocked cells |

## Algorithm Walkthrough

1. Convert the blocked coordinates into a hash set.

We need constant-time lookup to determine whether a cell is blocked. A hash set allows `O(1)` membership checks during BFS.
2. Compute the maximum possible enclosed area.

If there are `n` blocked cells, the largest region they can completely surround is:

$\frac{n(n-1)}{2}$

This becomes our BFS exploration limit.
3. Create a helper BFS function.

The BFS starts from one point and attempts to reach another point. During traversal:

- If the target is reached, return `True`
- If the number of visited cells exceeds the enclosure limit, return `True`
- If BFS terminates before exceeding the limit, return `False`

The reasoning is important here. If we can expand beyond the maximum enclosed area, we have escaped any possible trap.
4. Use a queue for BFS traversal.

BFS naturally explores all reachable nearby cells level by level. A queue efficiently supports this process.
5. Track visited cells.

Without a visited set, BFS would revisit the same positions repeatedly and potentially loop forever.
6. Generate neighboring cells.

For every current cell, try moving:

- Up
- Down
- Left
- Right

Ignore moves that:

- Leave the grid
- Enter blocked cells
- Revisit previously seen cells
7. Run BFS from source to target.

This checks whether the source is trapped.
8. Run BFS from target to source.

This checks whether the target is trapped.
9. Return `True` only if both searches succeed.

If either side is enclosed, escape is impossible.

### Why it works

The correctness relies on the fact that 200 blocked cells cannot isolate a large region. If BFS expands beyond the theoretical maximum enclosed area, the search must have escaped any enclosure. Once outside, the vast open grid ensures connectivity unless the opposite endpoint is itself trapped. Checking from both directions guarantees neither endpoint is enclosed.

## Python Solution

```
from collections import deque
from typing import List, Set, Tuple

class Solution:
    def isEscapePossible(
        self,
        blocked: List[List[int]],
        source: List[int],
        target: List[int]
    ) -> bool:

        BLOCK_LIMIT = len(blocked) * (len(blocked) - 1) // 2
        GRID_SIZE = 10**6

        blocked_set: Set[Tuple[int, int]] = {
            (x, y) for x, y in blocked
        }

        def bfs(start: List[int], end: List[int]) -> bool:
            queue = deque([tuple(start)])
            visited = {tuple(start)}

            directions = [
                (1, 0),
                (-1, 0),
                (0, 1),
                (0, -1)
            ]

            while queue and len(visited) <= BLOCK_LIMIT:
                x, y = queue.popleft()

                if [x, y] == end:
                    return True

                for dx, dy in directions:
                    nx = x + dx
                    ny = y + dy

                    if not (0 <= nx < GRID_SIZE and 0 <= ny < GRID_SIZE):
                        continue

                    if (nx, ny) in blocked_set:
                        continue

                    if (nx, ny) in visited:
                        continue

                    visited.add((nx, ny))
                    queue.append((nx, ny))

            return len(visited) > BLOCK_LIMIT

        return bfs(source, target) and bfs(target, source)
```

The implementation begins by converting all blocked cells into a set for efficient membership testing.

The variable `BLOCK_LIMIT` stores the maximum possible enclosed area. This value determines when BFS can safely conclude that it has escaped confinement.

The helper function `bfs()` performs a bounded breadth-first search. The traversal stops under two possible success conditions:

- The destination is reached directly
- The visited region grows beyond the enclosure threshold

The `while` loop condition is important. Once the search exceeds the theoretical enclosure size, there is no need to continue exploring.

The algorithm uses a queue because BFS systematically explores nearby positions and avoids deep recursive calls that could occur with DFS.

Finally, the algorithm performs the search twice, once from the source and once from the target, ensuring that neither endpoint is trapped.

## Go Solution

```
package main

func isEscapePossible(blocked [][]int, source []int, target []int) bool {
	blockLimit := len(blocked) * (len(blocked) - 1) / 2
	gridSize := 1000000

	blockedSet := make(map[[2]int]bool)

	for _, cell := range blocked {
		blockedSet[[2]int{cell[0], cell[1]}] = true
	}

	bfs := func(start []int, end []int) bool {
		queue := [][2]int{{start[0], start[1]}}
		visited := make(map[[2]int]bool)
		visited[[2]int{start[0], start[1]}] = true

		directions := [][2]int{
			{1, 0},
			{-1, 0},
			{0, 1},
			{0, -1},
		}

		for len(queue) > 0 && len(visited) <= blockLimit {
			current := queue[0]
			queue = queue[1:]

			x, y := current[0], current[1]

			if x == end[0] && y == end[1] {
				return true
			}

			for _, d := range directions {
				nx := x + d[0]
				ny := y + d[1]

				if nx < 0 || nx >= gridSize || ny < 0 || ny >= gridSize {
					continue
				}

				next := [2]int{nx, ny}

				if blockedSet[next] {
					continue
				}

				if visited[next] {
					continue
				}

				visited[next] = true
				queue = append(queue, next)
			}
		}

		return len(visited) > blockLimit
	}

	return bfs(source, target) && bfs(target, source)
}
```

The Go implementation closely mirrors the Python solution, but there are some language-specific differences.

Go does not support tuple hashing directly like Python, so fixed-size arrays of type `[2]int` are used as map keys.

The BFS queue is implemented using slices. Removing the front element uses `queue = queue[1:]`.

Go maps return zero values for missing keys, making membership checks straightforward with boolean maps.

Integer overflow is not a concern because all coordinate values and intermediate computations fit comfortably within Go's standard integer range.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n²) | BFS explores at most the bounded enclosure area |
| Space | O(n²) | Visited set stores at most the explored bounded region |

Here, `n = len(blocked)`.

The key observation is that BFS never explores the full grid. Instead, it explores at most the maximum enclosed region created by blocked cells.

Since:

$\frac{n(n-1)}{2}$

and `n <= 200`, the search remains extremely manageable.

The maximum explored region is roughly 20,000 cells, which is easily feasible.

## Edge Cases

One important edge case occurs when the source is trapped immediately at the boundary of the grid. Since movement outside the grid is illegal, even two blocked cells can fully trap a corner position. A careless implementation might only consider blocked neighbors and forget boundary restrictions. This implementation explicitly validates coordinate bounds before exploring neighbors, correctly handling corner cases.

Another critical edge case occurs when the source is free but the target is enclosed. A one-directional BFS from the source may incorrectly conclude that escape is possible simply because the source can expand into open space. The solution avoids this bug by performing the bounded BFS from both the source and the target.

A third subtle case occurs when there are no blocked cells at all. In this situation, the enclosure limit becomes zero. The BFS correctly interprets any successful expansion beyond zero visited cells as proof that the starting point is not enclosed. This allows the algorithm to terminate almost immediately instead of traversing the enormous grid.

A final edge case involves blocked configurations that appear to form a wall but actually contain a small opening. Incorrect implementations may accidentally treat diagonal adjacency as blocking or mishandle neighbor generation. This implementation only considers the four valid movement directions and therefore correctly identifies reachable paths through narrow gaps.
