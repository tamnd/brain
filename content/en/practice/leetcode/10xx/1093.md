# LeetCode 1093 - Statistics from a Large Sample

## Problem Understanding

This problem gives us a compressed representation of a very large dataset. Instead of listing every number explicitly, we receive an array called `count` of length `256`, where `count[k]` tells us how many times the integer `k` appears in the sample.

For example, if:

```
count[2] = 3
```

then the value `2` appears three times in the dataset.

The dataset only contains integers in the range `[0, 255]`, which is important because it means the number of distinct values is always fixed and very small.

We must compute five statistical values:

1. `minimum`, the smallest number that appears at least once.
2. `maximum`, the largest number that appears at least once.
3. `mean`, the arithmetic average.
4. `median`, the middle value after sorting the sample.
5. `mode`, the value that appears most frequently.

The result must be returned as floating point numbers.

The key challenge is that the actual sample size can be extremely large. The constraint:

```
sum(count) <= 10^9
```

means there may be up to one billion numbers in the dataset. Constructing the full array explicitly would be completely impractical.

However, the number of possible values is only `256`, which strongly suggests we should process the frequency table directly instead of rebuilding the sample.

There are several important edge cases to consider.

The sample may contain only one distinct value. In that case, minimum, maximum, median, mean, and mode are all the same number.

The total number of elements may be odd or even. Median calculation differs between these two cases.

The median may require averaging two different values, for example `2` and `3`, producing `2.5`.

Some frequencies may be extremely large, so we must avoid approaches that iterate once per occurrence.

The problem guarantees that the mode is unique, so we never need to handle ties for the most frequent value.

## Approaches

### Brute Force Approach

A straightforward solution would reconstruct the full sample array from the frequency counts.

For every value `k` from `0` to `255`, we append `k` into a new array exactly `count[k]` times. Once the full sorted array is built, computing the statistics becomes simple.

We can directly access:

- the first element for minimum,
- the last element for maximum,
- the middle element(s) for median,
- the average using the total sum,
- the most frequent value using the counts.

This approach is correct because the reconstructed array exactly matches the original sample.

However, it is far too expensive for large inputs. If the sample contains up to `10^9` elements, building the explicit array would require enormous memory and time.

### Optimal Approach

The key observation is that there are only `256` possible values.

Instead of reconstructing the sample, we can process the frequency table directly.

For minimum and maximum, we scan for the first and last indices whose count is positive.

For mean, we compute:

```
(sum of value * frequency) / total frequency
```

For mode, we track the value with the largest frequency.

For median, we use cumulative frequencies. Since the data is already implicitly sorted by index order, we can determine where the middle position lies without expanding the sample.

This reduces the problem to a simple linear scan over the fixed-size array.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(N) | O(N) | Reconstructs the entire sample explicitly |
| Optimal | O(256) | O(1) | Processes frequencies directly |

Here, `N` represents the total number of elements in the sample.

Since `256` is constant, the optimal solution effectively runs in constant time.

## Algorithm Walkthrough

1. Initialize variables for minimum, maximum, total count, total sum, and mode.

We need these values while scanning the frequency table. The total count helps compute the mean and determine median positions.
2. Scan through all values from `0` to `255`.

For every value `i` where `count[i] > 0`, update:

- minimum if this is the first seen value,
- maximum to the current value,
- total count by adding `count[i]`,
- total sum by adding `i * count[i]`,
- mode if `count[i]` is larger than the current maximum frequency.
3. Compute the mean.

The mean is simply:

```
total_sum / total_count
```
4. Determine the median positions.

If the sample size is odd, there is one middle position:

```
total_count // 2 + 1
```

If the sample size is even, there are two middle positions:

```
total_count // 2
total_count // 2 + 1
```
5. Perform another scan using cumulative frequencies.

Maintain a running count of how many elements have been passed so far.

Once the cumulative count reaches a median position, record the corresponding value.
6. Compute the median.

If the sample size is odd, the median is the single middle value.

If even, the median is the average of the two middle values.
7. Return all statistics as floating point numbers.

### Why it works

The frequency array represents a sorted multiset implicitly. Every index appears in sorted order according to its frequency.

By accumulating frequencies, we can determine exactly where each position in the sorted sample lies without constructing the sample itself.

The cumulative frequency invariant guarantees that once the running total reaches a target median position, the current index is the value occupying that position in sorted order.

Therefore, all computed statistics match the true dataset.

## Python Solution

```
from typing import List

class Solution:
    def sampleStats(self, count: List[int]) -> List[float]:
        minimum = -1
        maximum = -1
        
        total_count = 0
        total_sum = 0
        
        mode = 0
        mode_frequency = 0

        # First pass, compute min, max, mean components, and mode
        for value in range(256):
            frequency = count[value]

            if frequency == 0:
                continue

            if minimum == -1:
                minimum = value

            maximum = value

            total_count += frequency
            total_sum += value * frequency

            if frequency > mode_frequency:
                mode_frequency = frequency
                mode = value

        mean = total_sum / total_count

        # Determine median positions
        if total_count % 2 == 1:
            left_position = right_position = total_count // 2 + 1
        else:
            left_position = total_count // 2
            right_position = total_count // 2 + 1

        cumulative = 0
        left_value = 0
        right_value = 0

        # Second pass, find median values
        for value in range(256):
            cumulative += count[value]

            if cumulative >= left_position and left_value == 0:
                left_value = value

            if cumulative >= right_position:
                right_value = value
                break

        median = (left_value + right_value) / 2.0

        return [
            float(minimum),
            float(maximum),
            mean,
            median,
            float(mode),
        ]
```

The implementation uses two passes over the fixed-size frequency array.

The first pass gathers all statistics except the median. During this scan, we compute the total sum and total count needed for the mean. We also track the first and last seen values for minimum and maximum.

The mode is updated whenever a larger frequency is found.

The second pass computes the median using cumulative frequencies. Instead of reconstructing the sorted sample, the algorithm determines where the middle positions fall within the cumulative counts.

For even-length samples, the algorithm identifies two middle values and averages them.

Because the frequency array always has length `256`, the algorithm remains extremely efficient regardless of the total sample size.

## Go Solution

```
func sampleStats(count []int) []float64 {
	minimum := -1
	maximum := -1

	totalCount := 0
	totalSum := 0

	mode := 0
	modeFrequency := 0

	// First pass
	for value := 0; value < 256; value++ {
		frequency := count[value]

		if frequency == 0 {
			continue
		}

		if minimum == -1 {
			minimum = value
		}

		maximum = value

		totalCount += frequency
		totalSum += value * frequency

		if frequency > modeFrequency {
			modeFrequency = frequency
			mode = value
		}
	}

	mean := float64(totalSum) / float64(totalCount)

	var leftPosition, rightPosition int

	if totalCount%2 == 1 {
		leftPosition = totalCount/2 + 1
		rightPosition = leftPosition
	} else {
		leftPosition = totalCount / 2
		rightPosition = totalCount/2 + 1
	}

	cumulative := 0
	leftValue := -1
	rightValue := -1

	// Second pass for median
	for value := 0; value < 256; value++ {
		cumulative += count[value]

		if cumulative >= leftPosition && leftValue == -1 {
			leftValue = value
		}

		if cumulative >= rightPosition {
			rightValue = value
			break
		}
	}

	median := float64(leftValue+rightValue) / 2.0

	return []float64{
		float64(minimum),
		float64(maximum),
		mean,
		median,
		float64(mode),
	}
}
```

The Go implementation follows the same logic as the Python version, but there are a few language-specific details.

Go requires explicit floating point conversion when computing the mean and median, otherwise integer division would occur.

The solution uses integer sentinel values like `-1` to indicate uninitialized variables for minimum and median tracking.

Since Go slices are reference types, no additional copying is required when reading the input array.

Integer overflow is not a concern here because the maximum possible sum is:

```
255 * 10^9
```

which safely fits inside Go's 64-bit integer handling on LeetCode environments.

| Time | O(256) | Two scans over a fixed-size array |
| Space | O(1) | Only a constant amount of extra memory is used |

Even though the sample itself may conceptually contain up to one billion elements, the algorithm never reconstructs it. All operations are performed directly on the compressed frequency representation.

Because the frequency array length is always exactly `256`, the runtime is effectively constant.

## Edge Cases

One important edge case occurs when the dataset contains only a single distinct value. A naive implementation might incorrectly compute the median or fail to initialize minimum and maximum properly. In this implementation, the first nonzero frequency initializes both minimum and maximum, and the cumulative scan correctly identifies the same value for both median positions.

Another important case is an even-length dataset where the two middle elements are different. This is a common source of off-by-one errors. The algorithm carefully computes the two median positions separately and averages the corresponding values. For example, the sorted sample `[1,2]` correctly produces a median of `1.5`.

A third edge case involves extremely large frequencies. Since `count[i]` may be as large as `10^9`, reconstructing the dataset would be impossible. This implementation never expands the sample and instead relies entirely on frequency accumulation, which keeps both memory usage and runtime efficient.

A subtle implementation issue appears when the median value itself is `0`. If sentinel values are not chosen carefully, the algorithm may confuse a valid median value with an uninitialized variable. Using `-1` as the uninitialized marker avoids this problem and ensures that `0` remains a valid statistical value.
