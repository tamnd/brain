## Solution

## Problem Understanding

The problem gives us two database tables, `Project` and `Employee`.

The `Project` table represents assignments between employees and projects. Each row means that a specific employee is working on a specific project. A single project can have multiple employees, and a single employee can work on multiple projects.

The `Employee` table stores information about employees, including their years of experience.

The goal is to find, for every project, the employee or employees who have the highest number of experience years within that project.

This is important because ties are allowed. If multiple employees in the same project share the maximum experience value, all of them must appear in the result.

The output should contain:

- `project_id`
- `employee_id`

for every employee who has the maximum experience in their project.

The result can be returned in any order, so we do not need to worry about sorting unless explicitly required.

In the example:

- Project `1` has employees `1`, `2`, and `3`
- Their experience values are `3`, `2`, and `3`
- The maximum is `3`
- Both employees `1` and `3` must be returned

The problem guarantees that:

- `(project_id, employee_id)` is unique in the `Project` table
- Every `employee_id` in `Project` exists in `Employee`

This means we do not need to handle invalid foreign key references.

The main challenge is correctly handling ties while efficiently grouping employees by project.

Important edge cases include:

- A project with only one employee
- Multiple employees tied for maximum experience
- Employees working on multiple projects
- All employees in a project having the same experience value

A naive solution might incorrectly return only one employee instead of all tied employees.

## Approaches

### Brute Force Approach

A straightforward solution is to process each project independently.

For every project:

1. Find all employees assigned to that project
2. Look up each employee's experience
3. Compute the maximum experience value
4. Iterate again through the employees and collect everyone matching that maximum

This works correctly because every project is handled independently, and we explicitly identify the maximum experience before selecting employees.

However, this approach can become inefficient if we repeatedly scan the same tables for every project. If there are many projects and many employees, repeated filtering and searching introduce unnecessary overhead.

### Optimal Approach

The key observation is that this is fundamentally a grouping and aggregation problem.

We first join the `Project` and `Employee` tables so that each project assignment includes the employee's experience years.

Then, for each project, we compute the maximum experience value.

Finally, we select only the rows where an employee's experience equals the maximum for that project.

In SQL, this can be done efficiently using either:

- a subquery with `GROUP BY`
- a window function such as `RANK()` or `MAX() OVER()`

The most common solution uses:

1. A grouped subquery to compute maximum experience per project
2. A join back to the detailed rows

This avoids repeated scans and leverages database aggregation efficiently.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(P × E) | O(1) extra | Repeatedly scans employees for each project |
| Optimal | O(N) | O(N) | Uses joins and aggregation efficiently |

Here:

- `P` is the number of projects
- `E` is the number of employees
- `N` is the number of rows in the joined dataset

## Algorithm Walkthrough

### Optimal SQL Strategy

1. Join the `Project` and `Employee` tables using `employee_id`.

This gives us access to both the project assignment and the employee's experience years in the same row.
2. Group the joined rows by `project_id`.

For each project, compute the maximum `experience_years`.
3. Store the result of step 2 as a derived table or subquery.

This subquery contains:

- `project_id`
- `max_experience`
4. Join the original joined dataset with the subquery.

Match rows where:

- the project IDs are equal
- the employee's experience equals the project's maximum experience
5. Return the matching `project_id` and `employee_id`.

Since ties are allowed, all employees matching the maximum experience are returned.

### Why it works

The algorithm works because the subquery computes the exact maximum experience for every project. The final join filters employees so that only those whose experience equals that maximum remain. Since equality is used rather than selecting a single row, all ties are preserved automatically.

## Python Solution

Although this is a SQL database problem on LeetCode, the platform still expects the solution in SQL syntax. Below is the correct SQL query wrapped in a Python multiline string for presentation purposes.

```
class Solution:
    def projectEmployeesIII(self):
        return """
        WITH project_experience AS (
            SELECT
                p.project_id,
                p.employee_id,
                e.experience_years
            FROM Project p
            JOIN Employee e
                ON p.employee_id = e.employee_id
        ),
        max_experience AS (
            SELECT
                project_id,
                MAX(experience_years) AS max_years
            FROM project_experience
            GROUP BY project_id
        )
        SELECT
            pe.project_id,
            pe.employee_id
        FROM project_experience pe
        JOIN max_experience me
            ON pe.project_id = me.project_id
           AND pe.experience_years = me.max_years
        """
```

The solution begins by creating a common table expression named `project_experience`. This joins the `Project` and `Employee` tables so each assignment row contains the employee's experience years.

The second common table expression, `max_experience`, groups rows by project and computes the maximum experience value for each project.

Finally, the query joins these two intermediate results. Only employees whose experience matches the maximum value for their project are returned.

Because the comparison uses equality rather than limiting to a single row, ties are naturally included.

## Edge Cases

One important edge case is when multiple employees in the same project share the exact same maximum experience value. A flawed solution might use `LIMIT 1`, `TOP 1`, or a ranking function incorrectly, returning only one employee. This implementation avoids that issue by matching all employees whose experience equals the computed maximum.

Another important case is a project containing only one employee. In this scenario, that employee is automatically the most experienced employee in the project. Since the grouping query computes the maximum correctly even for single-row groups, the employee is returned normally.

A third edge case occurs when the same employee works on multiple projects. The implementation handles this naturally because the aggregation is grouped by `project_id`, not by employee. The same employee can therefore appear in the result multiple times, once for each project where they have maximum experience.

A final subtle case is when all employees in a project have identical experience values. In that situation, every employee should be returned. Because the query filters using equality against the maximum value, all rows satisfy the condition and are included in the output.
