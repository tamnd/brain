## Solution

## Problem Understanding

The problem requires adding two numbers that are represented in **negabinary**, a base of -2, instead of the usual positive bases like 2 or 10. In negabinary, the weight of each digit alternates between positive and negative powers of 2: the least significant bit has a weight of `(-2)^0 = 1`, the next `(-2)^1 = -2`, then `(-2)^2 = 4`, and so on. This makes arithmetic a bit non-intuitive compared to standard binary because carrying and borrowing behave differently.

The inputs `arr1` and `arr2` are arrays of 0s and 1s representing numbers in negabinary from **most significant bit to least significant bit**, with no leading zeros unless the number is zero. The output should also be an array in the same format representing the sum of the two numbers.

The constraints indicate that the arrays can be up to length 1000. This means a naive approach that converts the arrays into integers could work in Python due to arbitrary-precision integers, but it could be risky in languages with fixed integer sizes. The main challenge lies in performing addition **directly in negabinary** without converting to decimal.

Important edge cases include arrays representing zero, numbers of different lengths, and situations where multiple carries propagate due to the negabinary base.

## Approaches

The **brute-force approach** is straightforward: convert both negabinary numbers to decimal integers, add them, and then convert the result back into negabinary. This guarantees correctness because integer addition is well-defined, but converting large numbers to decimal could be costly in terms of space and performance if done naïvely.

The **optimal approach** performs **addition directly in negabinary** by simulating bitwise addition with a carry. The main insight is that in negabinary, each position can carry differently: a sum of 2 at a position leads to a carry of -1 in the next higher position, and a sum of -1 leads to a carry of 1. By iterating from the least significant bit upwards, adjusting for carries, we can construct the resulting negabinary number without converting to decimal.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n + m) | O(n + m) | Convert arrays to decimal integers, sum, then convert back to negabinary |
| Optimal | O(max(n, m)) | O(max(n, m)) | Direct negabinary addition with carry handling, no conversions |

## Algorithm Walkthrough

1. Reverse both input arrays so we can process from least significant bit to most significant bit. This simplifies addition similar to standard binary addition.
2. Initialize an empty list `result` and a `carry` variable set to 0.
3. Iterate index `i` from 0 up to the maximum length of the two arrays.

a. Compute the sum of `carry`, `arr1[i]` if it exists, and `arr2[i]` if it exists.

b. Append `sum % 2` to `result`. In negabinary, modulo 2 gives the correct bit at the current position.

c. Update `carry` to `-(sum // 2)`. This handles the negabinary carry rule.
4. After finishing the loop, continue processing `carry` while it is not zero. Append `carry % 2` and update `carry` similarly.
5. Remove any leading zeros from the final result. Since we processed in reverse, this means popping zeros from the end until the last element is 1.
6. Reverse `result` to restore the most significant bit first order.
7. Return `result`.

**Why it works:** The algorithm correctly maintains the invariant that after each iteration, the partial result plus the propagated carry equals the true sum of the inputs in negabinary form. Using `carry = -(sum // 2)` ensures proper propagation according to the negabinary rules.

## Python Solution

```
from typing import List

class Solution:
    def addNegabinary(self, arr1: List[int], arr2: List[int]) -> List[int]:
        i, j = len(arr1) - 1, len(arr2) - 1
        result = []
        carry = 0
        
        while i >= 0 or j >= 0 or carry != 0:
            total = carry
            if i >= 0:
                total += arr1[i]
                i -= 1
            if j >= 0:
                total += arr2[j]
                j -= 1
            
            result.append(total & 1)
            carry = -(total >> 1)
        
        while len(result) > 1 and result[-1] == 0:
            result.pop()
        
        return result[::-1]
```

**Implementation walkthrough:** We initialize pointers to the end of both arrays to simulate addition from the least significant bit. The `total` includes the current digits and the carry. Using `total & 1` gives the correct bit for the current position, and `carry = -(total >> 1)` propagates negabinary carry rules. Leading zeros are removed at the end, and the list is reversed for proper order.

## Go Solution

```
func addNegabinary(arr1 []int, arr2 []int) []int {
    i, j := len(arr1)-1, len(arr2)-1
    result := []int{}
    carry := 0

    for i >= 0 || j >= 0 || carry != 0 {
        total := carry
        if i >= 0 {
            total += arr1[i]
            i--
        }
        if j >= 0 {
            total += arr2[j]
            j--
        }
        
        result = append(result, total & 1)
        carry = -(total >> 1)
    }

    for len(result) > 1 && result[len(result)-1] == 0 {
        result = result[:len(result)-1]
    }

    // Reverse result
    for k := 0; k < len(result)/2; k++ {
        result[k], result[len(result)-1-k] = result[len(result)-1-k], result[k]
    }

    return result
}
```

**Go-specific notes:** Go lacks native slicing reverse, so we manually swap elements to reverse the `result`. Integer overflow is not an issue due to the carry propagation being small and the problem constraints. Empty arrays are handled naturally with the loop condition.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(max(n, m)) | We iterate once over the maximum length of the input arrays plus any additional carry propagation steps. |
| Space | O(max(n, m)) | The output array can at most be one element longer than the longer input array due to carry propagation. |

The time complexity is linear because we only process each input element once, and the carry propagation loop cannot exceed O(max(n, m)) iterations.
