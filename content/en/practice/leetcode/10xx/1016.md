## Solution

## Problem Understanding

The problem gives us a binary string `s` and an integer `n`. We must determine whether every integer from `1` through `n` appears somewhere inside `s` as a binary substring.

For example, if `s = "0110"` and `n = 3`, then:

- `1` in binary is `"1"`
- `2` in binary is `"10"`
- `3` in binary is `"11"`

All three substrings appear in `"0110"`, so the answer is `true`.

However, if `n = 4`, then:

- `4` in binary is `"100"`

The substring `"100"` does not appear in `"0110"`, so the answer is `false`.

The important detail is that substrings must be contiguous. We are not checking whether the bits appear in order with gaps, they must appear consecutively.

The constraints are interesting:

- `s.length <= 1000`
- `n <= 10^9`

The string itself is relatively short, but `n` can be extremely large. This immediately tells us that we cannot afford to do anything proportional to `n` in the worst case unless we can terminate early.

A key observation comes from substring capacity. A binary string of length `1000` cannot possibly contain the binary representations of all numbers up to very large values. This naturally limits the practical search space.

There are several edge cases worth considering:

- Very small strings like `"1"` or `"0"`
- Large values of `n` that are impossible to satisfy
- Binary representations with leading bits such as `"1000"`
- Duplicate substrings appearing many times
- Numbers whose binary representation length exceeds `s.length`

The problem guarantees that `s` contains only `'0'` and `'1'`, so we never need to validate the input format.

## Approaches

### Brute Force Approach

The most direct solution is to iterate through every integer from `1` to `n`, convert each number to its binary representation, and check whether that binary string exists inside `s`.

For each number:

1. Convert it to binary
2. Search for the substring inside `s`
3. If any binary string is missing, return `false`
4. Otherwise return `true`

This works because the problem definition directly asks whether every binary representation appears as a substring.

However, this approach becomes inefficient when `n` is large. Even though `s` is short, repeatedly converting numbers and performing substring searches creates unnecessary overhead.

A more serious issue is that many values of `n` are impossible from the start. A string of length `1000` cannot contain all binary representations up to billions of numbers.

### Optimal Approach

The key insight is that we only need to check a reduced range of numbers.

There is a mathematical observation behind the optimization:

If all numbers in the range `[1, n]` appear, then all numbers in the upper half `[n/2 + 1, n]` must appear. Any smaller number can be obtained by removing the leading bit from a larger binary representation.

For example:

- `5` is `"101"`
- Removing the leading `'1'` gives `"01"`
- `"01"` contains `"1"`

More generally, if all larger numbers exist, the smaller patterns naturally appear as substrings.

This means we only need to explicitly verify numbers from `n` down to `n // 2 + 1`.

Because `s.length <= 1000`, this drastically reduces the practical workload.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × m) | O(1) | Checks every number individually |
| Optimal | O((n/2) × m) | O(1) | Only checks the upper half of numbers |

Here, `m` represents the average substring search cost, bounded by the length of `s`.

## Algorithm Walkthrough

### Optimal Algorithm

1. Iterate through integers from `n` down to `n // 2 + 1`.

We only check the upper half because smaller binary representations are guaranteed to appear as substrings of larger ones if the larger values already exist.
2. Convert each integer into its binary representation.

In Python, this is done with `bin(value)[2:]`. In Go, we use `strconv.FormatInt`.
3. Check whether the binary string exists inside `s`.

We use the language's built in substring search because the string length is small enough that advanced pattern matching algorithms are unnecessary.
4. If any binary representation is missing, immediately return `false`.

Early termination is important because failing cases are common.
5. If all required binary representations are found, return `true`.

### Why it works

The algorithm relies on the property that every smaller binary number appears as a suffix of some larger binary number. If all numbers in the upper half exist as substrings, then the lower half naturally appears inside those substrings as well.

Therefore, verifying only the upper half is sufficient to guarantee the entire range `[1, n]`.

## Python Solution

```
class Solution:
    def queryString(self, s: str, n: int) -> bool:
        for value in range(n, n // 2, -1):
            binary = bin(value)[2:]

            if binary not in s:
                return False

        return True
```

The implementation follows the algorithm directly.

The loop starts from `n` and moves downward to `n // 2 + 1`. This avoids unnecessary checks for smaller numbers.

For each value, `bin(value)` converts the integer into a binary string prefixed with `"0b"`. The slice `[2:]` removes that prefix.

The substring test `binary not in s` performs the required containment check. Since the maximum length of `s` is only `1000`, this operation is efficient enough.

The function immediately returns `False` when any required binary representation is missing. Otherwise, after all checks succeed, it returns `True`.

## Go Solution

```
package main

import (
	"strconv"
	"strings"
)

func queryString(s string, n int) bool {
	for value := n; value > n/2; value-- {
		binary := strconv.FormatInt(int64(value), 2)

		if !strings.Contains(s, binary) {
			return false
		}
	}

	return true
}
```

The Go implementation mirrors the Python logic closely.

`strconv.FormatInt(int64(value), 2)` converts the integer into a base 2 string representation.

`strings.Contains` performs the substring check.

Go requires explicit integer conversion to `int64` for `FormatInt`, unlike Python where integer handling is automatic.

## Edge Cases

One important edge case is when the string does not contain `'1'`. Since every positive integer in binary begins with `'1'`, the absence of this character guarantees failure immediately. For example, `"0000"` can never satisfy even `n = 1`. The implementation handles this naturally because substring checks fail instantly.

Another critical edge case involves very large values of `n`. Since the string length is capped at `1000`, many large values are impossible to satisfy. The algorithm still behaves efficiently because it only checks the upper half of the range and terminates as soon as a missing substring is found.

A third edge case involves overlapping binary substrings. For example, in `"0110"`, the substring `"11"` overlaps with surrounding bits. A naive implementation that incorrectly tokenizes substrings could fail here. Using direct substring search ensures overlapping matches are handled correctly.

A final edge case occurs when binary representations are longer than the input string itself. For example, if `s` has length `3`, no binary string of length `4` can possibly appear. The implementation automatically handles this because substring search immediately fails for longer patterns.
