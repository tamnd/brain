## Solution

## Problem Understanding

This problem asks us to compute the average number of experience years for employees assigned to each project.

We are given two database tables:

The `Project` table represents which employees are assigned to which projects. Each row contains a `project_id` and an `employee_id`. Since `(project_id, employee_id)` is the primary key, we know that the same employee cannot appear twice for the same project.

The `Employee` table contains information about employees, including their number of experience years.

The task is to produce a result table where each row contains:

- A `project_id`
- The average experience of employees working on that project
- The average rounded to exactly 2 decimal places

To solve this problem, we need to combine information from both tables. The `Project` table tells us which employees belong to each project, while the `Employee` table tells us how many years of experience each employee has. Since the connection between the tables is `employee_id`, we need to join the two tables together.

After joining, we group rows by `project_id` and compute the average of `experience_years`.

The problem guarantees that every `employee_id` in `Project` exists in `Employee`, because `employee_id` is a foreign key. This means we do not need to worry about missing employee records during the join.

An important detail is rounding. SQL averages may produce many decimal places, but the output must contain exactly 2 digits after the decimal point. Therefore, we must explicitly use a rounding function.

There are several edge cases worth considering:

- A project may contain only one employee. In that case, the average is simply that employee's experience.
- Multiple projects may share the same employee. Each project's average must still be computed independently.
- Experience values may vary widely, but averaging still works normally.
- Since `experience_years` is guaranteed not to be `NULL`, we do not need special handling for missing values.

## Approaches

### Brute Force Approach

A brute force approach would manually process each project separately.

For every distinct `project_id`, we could:

1. Find all employees assigned to that project.
2. For each employee, search the `Employee` table to locate their experience years.
3. Sum all experience values.
4. Divide by the number of employees.
5. Round the result.

This approach is correct because it explicitly computes the average for every project. However, it is inefficient because it repeatedly scans the `Employee` table for employee information.

If there are `P` project assignments and `E` employees, repeatedly searching through employees for every project assignment can lead to an `O(P × E)` solution.

### Optimal Approach

The better solution uses a SQL `JOIN` together with `GROUP BY`.

The key observation is that relational databases are specifically optimized for combining related tables and aggregating data.

We can:

1. Join `Project` and `Employee` using `employee_id`
2. Group all matching rows by `project_id`
3. Compute `AVG(experience_years)` for each group
4. Round the result using `ROUND(..., 2)`

This solution is efficient because databases internally optimize joins and aggregation operations.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(P × E) | O(1) | Repeatedly scans employee records for each project |
| Optimal | O(P + E) | O(1) extra | Uses SQL JOIN and GROUP BY efficiently |

## Algorithm Walkthrough

1. Start with the `Project` table because it contains the mapping between projects and employees.
2. Join the `Project` table with the `Employee` table using the shared `employee_id` column. This gives us rows containing:

- project ID
- employee ID
- employee experience years
3. Group the joined rows by `project_id`. Grouping ensures that all employees belonging to the same project are processed together.
4. For each project group, compute the average of `experience_years` using SQL's `AVG()` aggregate function.
5. Use the `ROUND()` function to round the computed average to 2 decimal places.
6. Return the resulting table containing:

- `project_id`
- `average_years`

### Why it works

The algorithm works because every employee assignment is represented exactly once in the joined table. Grouping by `project_id` collects all employees for the same project together, and averaging their `experience_years` produces the correct project experience average. Since SQL aggregation functions operate over complete groups, every project's result is computed independently and correctly.

## Python Solution

LeetCode database problems are solved using SQL queries rather than executable Python code. However, the platform still expects the solution to be presented in a code block.

```
# Write your MySQL query statement below

SELECT
    p.project_id,
    ROUND(AVG(e.experience_years), 2) AS average_years
FROM Project p
JOIN Employee e
    ON p.employee_id = e.employee_id
GROUP BY p.project_id;
```

The query begins by selecting from the `Project` table using alias `p`. It joins the `Employee` table using alias `e` on the shared `employee_id` column.

After joining, each row contains both project information and employee experience information.

The `GROUP BY p.project_id` clause groups all employees belonging to the same project together. Inside each group, `AVG(e.experience_years)` computes the average experience.

Finally, `ROUND(..., 2)` ensures the output is formatted with exactly two decimal places, matching the problem requirements.

## Edge Cases

### Project With Only One Employee

A project may contain exactly one employee. In this case, the average should simply equal that employee's experience years. Some incorrect implementations may accidentally divide incorrectly or mishandle grouping. Using SQL's `AVG()` function automatically handles single-row groups correctly.

### Employees Assigned to Multiple Projects

The same employee can appear in multiple projects. A naive implementation might accidentally deduplicate employees globally rather than per project. Grouping strictly by `project_id` ensures that each project's average is computed independently, even if employees are shared.

### Averages Producing Long Decimals

Some averages may produce repeating or long decimal values. For example, averaging `1`, `2`, and `2` gives `1.666...`. Without explicit rounding, the output format would not match the problem requirements. Using `ROUND(AVG(...), 2)` guarantees exactly two decimal places.

### Large Numbers of Projects and Employees

In larger datasets, repeated scanning of employee records becomes inefficient. The join-and-group approach avoids redundant work and allows the database engine to optimize execution internally. This ensures the solution scales efficiently even for larger inputs.
