## Solution

## Problem Understanding

The problem presents a two-player game with pieces in a line, each piece colored either `'A'` or `'B'`. Alice and Bob take turns removing pieces according to strict rules. Alice can remove an `'A'` only if both neighbors are `'A'`, while Bob can remove a `'B'` only if both neighbors are `'B'`. Players cannot remove edge pieces, and the game ends when a player cannot make a valid move. The goal is to determine whether Alice, who moves first, can win if both players play optimally.

The input is a string `colors` of length `n` (up to 100,000) representing the sequence of colored pieces. The output is a boolean: `true` if Alice can force a win, `false` if Bob can force a win. Important constraints include handling short strings where no removal is possible, strings with only one color, and large strings requiring an efficient solution. Edge cases include strings shorter than three (where no moves are possible) and strings where only one player has potential moves.

The problem essentially reduces to counting the maximum number of potential removals for Alice and Bob. Since players cannot remove edge pieces and can only remove interior pieces with two neighbors of the same color, each run of consecutive same-color pieces of length ≥ 3 contributes `length - 2` possible removals.

## Approaches

### Brute Force

A naive approach simulates the game turn by turn. On each player's turn, the algorithm scans the string for removable pieces, removes one, and repeats until one player has no moves. This guarantees correctness because it follows the game rules precisely. However, this approach is far too slow for large strings because each removal requires scanning the string, and there can be up to `O(n)` removals, leading to an `O(n^2)` or worse time complexity.

### Optimal Approach

The key observation is that the order of removals does not affect the total number of potential moves for each player. Alice can only remove pieces from runs of `'A'` of length ≥ 3, and Bob can only remove pieces from runs of `'B'` of length ≥ 3. Therefore, we can precompute the maximum number of moves available to each player by counting `length - 2` for each run of length ≥ 3. If Alice has more potential moves than Bob, she can always win by playing optimally; otherwise, Bob wins.

This approach is linear in time because it requires a single pass through the string to count runs, and it uses constant additional space.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^2) | O(n) | Simulate each move step by step; too slow for large n |
| Optimal | O(n) | O(1) | Count potential moves for Alice and Bob by run lengths; very efficient |

## Algorithm Walkthrough

1. Initialize counters `alice_moves` and `bob_moves` to zero.
2. Traverse the string from left to right, identifying consecutive runs of the same color.
3. For each run, if the color is `'A'` and the run length is ≥ 3, increment `alice_moves` by `run_length - 2`.
4. Similarly, if the color is `'B'` and the run length is ≥ 3, increment `bob_moves` by `run_length - 2`.
5. After processing the entire string, compare `alice_moves` and `bob_moves`.
6. Return `true` if `alice_moves > bob_moves`, otherwise return `false`.

Why it works: This algorithm works because the number of removable pieces within each run determines the maximum number of turns a player can take. Since both players play optimally, having more potential moves guarantees the ability to force a win.

## Python Solution

```python
class Solution:
    def winnerOfGame(self, colors: str) -> bool:
        alice_moves = 0
        bob_moves = 0
        n = len(colors)
        i = 0

        while i < n:
            j = i
            while j < n and colors[j] == colors[i]:
                j += 1
            run_length = j - i
            if colors[i] == 'A' and run_length >= 3:
                alice_moves += run_length - 2
            elif colors[i] == 'B' and run_length >= 3:
                bob_moves += run_length - 2
            i = j

        return alice_moves > bob_moves
```

The code iterates through `colors` while grouping consecutive same-color pieces into runs. Each run contributes `length - 2` potential removals if its length is ≥ 3. The comparison at the end determines the winner based on the number of moves.

## Go Solution

```go
func winnerOfGame(colors string) bool {
    aliceMoves, bobMoves := 0, 0
    n := len(colors)
    i := 0

    for i < n {
        j := i
        for j < n && colors[j] == colors[i] {
            j++
        }
        runLength := j - i
        if colors[i] == 'A' && runLength >= 3 {
            aliceMoves += runLength - 2
        } else if colors[i] == 'B' && runLength >= 3 {
            bobMoves += runLength - 2
        }
        i = j
    }

    return aliceMoves > bobMoves
}
```

The Go solution mirrors the Python implementation. `aliceMoves` and `bobMoves` track the maximum potential moves. The loop scans the string for consecutive runs, calculating `runLength - 2` for runs of length ≥ 3. The final return compares the counters.

## Edge Cases

The first edge case is a string shorter than three characters, like `"AA"` or `"B"`. Since no piece can have two neighbors, neither player can make a move. The implementation correctly handles this because no run will satisfy `length >= 3`, leaving both move counters at zero.
