## Solution

## Problem Understanding

The problem gives us an array called `digits`, where every element is a single decimal digit from `0` to `9`. We must build all possible unique three digit even numbers using exactly three elements from this array.

There are several important conditions:

1. The number must contain exactly three digits.
2. The first digit cannot be `0`, because numbers like `012` are not valid three digit integers.
3. The final digit must be even, because the resulting number itself must be even.
4. Each digit can only be used as many times as it appears in the input array.

The output must contain every valid number exactly once, and the result must be sorted in ascending order.

For example, if the input is `[2,1,3,0]`, we can build numbers such as `102`, `132`, and `320`. We cannot build `123` because it is odd, and we cannot build `032` because it has a leading zero.

The constraints are small:

- `3 <= digits.length <= 100`
- `0 <= digits[i] <= 9`

Even though the array can contain up to 100 elements, there are only 10 possible digit values. This is a very important observation because it means we can efficiently test combinations without worrying about extremely large search spaces.

Several edge cases are important:

- Inputs containing many duplicate digits, such as `[2,2,2]`
- Inputs containing zeros, which can accidentally create leading zero problems
- Inputs where no even number can be formed, such as `[3,7,5]`
- Inputs where a digit is required more times than it appears in the array

A correct solution must carefully track digit frequencies and avoid duplicate outputs.

## Approaches

### Brute Force Approach

The most direct approach is to generate every possible ordered triple of indices `(i, j, k)` such that all indices are distinct. For every triple, we form the three digit number:

```
number = digits[i] * 100 + digits[j] * 10 + digits[k]
```

We then verify:

- The hundreds digit is not zero
- The number is even
- The same index is not reused

Since duplicate digits may exist, we also need a set to avoid duplicate numbers in the output.

This approach is correct because it explicitly checks every possible valid construction. Any legal three digit number must come from some ordered choice of three distinct indices.

However, the method performs a triple nested loop over the array. If `n` is the length of `digits`, the complexity becomes `O(n^3)`. With `n <= 100`, this is still manageable, but it is inefficient and unnecessary because the actual number of distinct digits is very small.

### Optimal Approach

A better observation is that every valid answer must be a three digit even number between `100` and `998`.

There are only 450 possible even three digit numbers:

```
100, 102, 104, ..., 998
```

Instead of generating combinations from the input, we can generate all possible candidate even numbers and check whether each one can be formed from the available digits.

To do this:

1. Count the frequency of every digit in the input array.
2. Iterate through all even numbers from `100` to `998`.
3. Extract the three digits of the candidate number.
4. Count how many times each digit is required.
5. Verify that the input contains enough copies of every digit.

This works efficiently because the candidate space is extremely small and fixed.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n³) | O(k) | Generates all index permutations and removes duplicates with a set |
| Optimal | O(1) | O(1) | Checks all possible three digit even numbers using digit frequencies |

The optimal approach is effectively constant time because the range of candidate numbers is fixed.

## Algorithm Walkthrough

### Optimal Algorithm

1. Create a frequency array of size 10.

Since digits range only from `0` to `9`, we store how many times each digit appears. For example, if the input is `[2,1,3,0]`, the frequency array becomes:

```
freq[0] = 1
freq[1] = 1
freq[2] = 1
freq[3] = 1
```
2. Iterate through every three digit even number.

Start from `100` and move in steps of `2` until `998`. Stepping by `2` guarantees the number is even.
3. Extract the digits of the current number.

For a number like `312`:

- Hundreds digit = `3`
- Tens digit = `1`
- Units digit = `2`
4. Count how many copies of each digit the number requires.

For `288`:

```
need[2] = 1
need[8] = 2
```
5. Compare required counts against available counts.

If the candidate needs more copies of any digit than exist in the input, the number is invalid.

For example:

- Input: `[2,8,8]`
- Candidate: `288`

This works because:

- One `2` exists
- Two `8`s exist
6. Add valid numbers to the result list.

Since we iterate in increasing order, the result is automatically sorted.

### Why it works

The algorithm checks every possible three digit even number exactly once. A number is added only if the input contains enough copies of every required digit. Therefore:

- Every returned number is valid.
- Every valid number is eventually checked and included.
- No duplicates occur because each candidate number is tested once.

This guarantees correctness.

## Python Solution

```python
from typing import List

class Solution:
    def findEvenNumbers(self, digits: List[int]) -> List[int]:
        frequency = [0] * 10
        
        for digit in digits:
            frequency[digit] += 1
        
        result = []
        
        for number in range(100, 1000, 2):
            hundreds = number // 100
            tens = (number // 10) % 10
            ones = number % 10
            
            needed = [0] * 10
            needed[hundreds] += 1
            needed[tens] += 1
            needed[ones] += 1
            
            valid = True
            
            for digit in range(10):
                if needed[digit] > frequency[digit]:
                    valid = False
                    break
            
            if valid:
                result.append(number)
        
        return result
```

The implementation begins by constructing a frequency array that records how many times each digit appears in the input. This allows constant time lookup when verifying whether a candidate number can be formed.

The loop iterates through every even number from `100` to `998`. Since the increment is `2`, every number checked is automatically even, eliminating the need for a separate parity check.

For each candidate number, the code extracts the hundreds, tens, and ones digits using integer arithmetic. A second array called `needed` records how many copies of each digit the current number requires.

The validation loop compares the required digit counts against the available counts from the input. If any required count exceeds availability, the candidate is rejected.

If all checks pass, the number is appended to the result list. Because candidates are processed in ascending order, the output is already sorted.

## Go Solution

```go
func findEvenNumbers(digits []int) []int {
    frequency := make([]int, 10)

    for _, digit := range digits {
        frequency[digit]++
    }

    result := []int{}

    for number := 100; number < 1000; number += 2 {
        hundreds := number / 100
        tens := (number / 10) % 10
        ones := number % 10

        needed := make([]int, 10)
        needed[hundreds]++
        needed[tens]++
        needed[ones]++

        valid := true

        for digit := 0; digit < 10; digit++ {
            if needed[digit] > frequency[digit] {
                valid = false
                break
            }
        }

        if valid {
            result = append(result, number)
        }
    }

    return result
}
```

The Go implementation follows the same logic as the Python version. Slices are used instead of Python lists for both the frequency array and the result array.

Go does not have built in dynamic list append semantics identical to Python lists, but the `append` function provides equivalent behavior for slices.

There are no integer overflow concerns because all numbers remain below `1000`.

An empty slice is returned naturally if no valid numbers exist, which matches LeetCode expectations.

## Edge Cases

One important edge case occurs when the input contains zeros. Numbers like `012` or `002` are not valid because three digit numbers cannot start with zero. A naive implementation that simply concatenates digits may accidentally include such values. This implementation avoids the problem automatically because it only checks numbers from `100` through `998`.

Another important case involves duplicate digits. Consider the input `[2,2,8]`. The number `228` is valid because there are two copies of digit `2`. However, `222` is invalid because only two copies of `2` exist. The frequency comparison logic handles this correctly by checking exact required counts.

A third edge case appears when no even digits exist in the input. Since every even number must end with `0`, `2`, `4`, `6`, or `8`, arrays like `[1,3,5]` can never produce valid answers. The algorithm naturally rejects every candidate because the required units digit frequency check fails.

Another subtle edge case is `[0,0,0]`. Although `000` is technically even, it is not a valid three digit number because leading zeros are forbidden. Since the algorithm starts checking from `100`, such invalid values are never considered.
