## Solution

## Problem Understanding

The problem gives us a singly linked list and asks us to identify all of its critical points. A node is considered a critical point if it is either a local maximum or a local minimum.

A local maximum is a node whose value is strictly greater than both its previous node and its next node. A local minimum is a node whose value is strictly smaller than both its previous node and its next node.

Because a node must have both a previous and next neighbor, the first and last nodes of the list can never be critical points.

Once all critical points are identified, we must compute two values:

1. The minimum distance between any two distinct critical points
2. The maximum distance between any two distinct critical points

The distance between two critical points is measured using their positions in the linked list. Positions are 1-indexed in the examples, but internally we can use either 0-indexed or 1-indexed positions consistently.

If there are fewer than two critical points, then it is impossible to form a pair, so the answer must be `[-1, -1]`.

The constraints allow up to `10^5` nodes, which immediately tells us that an efficient linear solution is preferred. Any solution with quadratic behavior would likely be too slow for the worst case.

There are several important edge cases to keep in mind:

- A list with fewer than three nodes can never contain a critical point.
- A list may contain exactly one critical point, which still requires returning `[-1, -1]`.
- Adjacent equal values are not considered maxima or minima because the comparisons must be strictly greater or strictly smaller.
- The minimum distance may come from neighboring critical points, while the maximum distance always comes from the first and last critical points.

## Approaches

### Brute Force Approach

A brute force solution would first traverse the linked list and store all critical point positions in an array. After collecting them, we could compare every pair of critical points to compute the minimum and maximum distances.

This works because checking every pair guarantees that we eventually find the smallest and largest distance.

However, if there are `k` critical points, comparing all pairs requires `O(k^2)` time. In the worst case, nearly every internal node could be a critical point, making `k` close to `n`. That produces quadratic complexity, which is too expensive for `n = 10^5`.

### Optimal Approach

The key observation is that we do not need to compare every pair of critical points.

For the maximum distance, the answer is always:

```
lastCriticalPosition - firstCriticalPosition
```

This works because the farthest two critical points must be the first and last ones encountered.

For the minimum distance, we only need to compare consecutive critical points. Any non-consecutive pair will always have a distance greater than or equal to one of the consecutive gaps between them.

Therefore, while traversing the linked list once, we can:

- Detect critical points
- Track the first critical point
- Track the previous critical point
- Continuously update the minimum distance

This gives a clean linear-time solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Stores all critical points and compares every pair |
| Optimal | O(n) | O(1) | Single traversal with running distance calculations |

## Algorithm Walkthrough

1. Start traversing the linked list from the second node because the first node cannot be a critical point.
2. Maintain three pointers:

- `prev`, the previous node
- `curr`, the current node
- `curr.next`, the next node

This allows us to compare the current value against both neighbors.
3. Maintain a position counter to track the index of the current node. Using 1-indexed positions matches the examples naturally.
4. For each current node, determine whether it is a critical point:

- Local maximum:

```
curr.val > prev.val AND curr.val > curr.next.val
```
- Local minimum:

```
curr.val < prev.val AND curr.val < curr.next.val
```
5. When the first critical point is found:

- Store its position in `first_critical`
- Also store it as `previous_critical`
6. For every later critical point:

- Compute the distance from the previous critical point
- Update the minimum distance if this gap is smaller
- Update the maximum distance using:

```
current_position - first_critical
```
- Move `previous_critical` to the current position
7. Continue until reaching the end of the list.
8. If fewer than two critical points were found, return `[-1, -1]`.
9. Otherwise, return `[min_distance, max_distance]`.

### Why it works

The algorithm works because every critical point is processed exactly once in order.

The minimum distance between any two critical points must occur between two consecutive critical points in sorted order. If there were a smaller distance between non-consecutive points, then at least one consecutive pair inside that interval would be even smaller, which is impossible.

The maximum distance is always achieved by the first and last critical points because positions increase monotonically during traversal.

Therefore, maintaining only the first and previous critical positions is sufficient to compute both answers correctly.

## Python Solution

```python
# Definition for singly-linked list.
# class ListNode:
#     def __init__(self, val=0, next=None):
#         self.val = val
#         self.next = next

from typing import Optional, List

class Solution:
    def nodesBetweenCriticalPoints(self, head: Optional[ListNode]) -> List[int]:
        prev = head
        curr = head.next

        position = 2

        first_critical = -1
        previous_critical = -1

        min_distance = float("inf")
        max_distance = -1

        while curr and curr.next:
            is_local_max = (
                curr.val > prev.val and curr.val > curr.next.val
            )

            is_local_min = (
                curr.val < prev.val and curr.val < curr.next.val
            )

            if is_local_max or is_local_min:
                if first_critical == -1:
                    first_critical = position
                else:
                    min_distance = min(
                        min_distance,
                        position - previous_critical
                    )

                    max_distance = position - first_critical

                previous_critical = position

            prev = curr
            curr = curr.next
            position += 1

        if max_distance == -1:
            return [-1, -1]

        return [min_distance, max_distance]
```

The implementation begins by setting up two pointers, `prev` and `curr`, so that each node can be compared against both neighbors. Since the first node cannot be critical, traversal starts from the second node.

The variable `position` tracks the current node index using 1-indexed numbering.

Two variables, `first_critical` and `previous_critical`, store the first and most recent critical point positions. These are sufficient because:

- The maximum distance always uses the first critical point
- The minimum distance only requires comparing consecutive critical points

For every node, the code checks whether it is a local maximum or local minimum using strict comparisons.

When the first critical point is found, its position is recorded. For every later critical point:

- The gap from the previous critical point updates the minimum distance
- The gap from the first critical point updates the maximum distance

Finally, if no valid pair of critical points exists, the method returns `[-1, -1]`.

## Go Solution

```go
/**
 * Definition for singly-linked list.
 * type ListNode struct {
 *     Val int
 *     Next *ListNode
 * }
 */
func nodesBetweenCriticalPoints(head *ListNode) []int {
    prev := head
    curr := head.Next

    position := 2

    firstCritical := -1
    previousCritical := -1

    minDistance := int(^uint(0) >> 1)
    maxDistance := -1

    for curr != nil && curr.Next != nil {
        isLocalMax := curr.Val > prev.Val && curr.Val > curr.Next.Val
        isLocalMin := curr.Val < prev.Val && curr.Val < curr.Next.Val

        if isLocalMax || isLocalMin {
            if firstCritical == -1 {
                firstCritical = position
            } else {
                distance := position - previousCritical

                if distance < minDistance {
                    minDistance = distance
                }

                maxDistance = position - firstCritical
            }

            previousCritical = position
        }

        prev = curr
        curr = curr.Next
        position++
    }

    if maxDistance == -1 {
        return []int{-1, -1}
    }

    return []int{minDistance, maxDistance}
}
```

The Go implementation follows the same logic as the Python version. Since Go does not provide infinity values for integers directly, the code initializes `minDistance` using the maximum possible integer value.

Go uses `nil` instead of Python's `None`, and slices are returned instead of Python lists.

The traversal logic and critical point detection remain identical.

## Edge Cases

One important edge case is when the linked list has fewer than three nodes. Since a critical point must have both a previous and next node, no node in such a list can qualify. A careless implementation might attempt to access nonexistent neighbors and cause runtime errors. The solution avoids this by starting traversal at the second node and requiring `curr.next` to exist in the loop condition.

Another important edge case occurs when the list contains exactly one critical point. In this situation, there is no pair of critical points from which to compute distances. The implementation correctly detects this because `max_distance` remains `-1`, causing the function to return `[-1, -1]`.

Equal neighboring values are another subtle source of bugs. The problem requires strict inequalities for maxima and minima. For example, in `[1,2,2,1]`, neither `2` is a critical point because neither is strictly greater than both neighbors. The implementation uses `>` and `<` comparisons only, ensuring duplicates are handled correctly.

A final edge case involves consecutive critical points. In a pattern like `[1,3,1,3,1]`, critical points appear very close together. The minimum distance becomes `1`, which must be detected properly. Because the algorithm compares every new critical point with the immediately previous one, it naturally captures the smallest possible gap.
