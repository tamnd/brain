## Solution

## Problem Understanding

The problem asks us to find the **next numerically balanced number** greater than a given integer `n`. A number is **numerically balanced** if for every digit `d` present in the number, it appears **exactly `d` times**. For example, the number `1333` is numerically balanced because the digit `1` occurs once and the digit `3` occurs three times. The input `n` is an integer between `0` and `10^6`, and the output should be the smallest numerically balanced number strictly greater than `n`.

Key points to note include that `0` cannot appear in a numerically balanced number because `0` would have to appear exactly zero times. The number of digits in the solution is limited since the input itself is at most `10^6`. Edge cases include small inputs like `0` or `1`, numbers where multiple digits are candidates, and numbers that require the next numerically balanced number to have more digits than `n`.

## Approaches

The **brute-force approach** would be to start at `n + 1` and check each integer sequentially to see if it is numerically balanced. To check if a number is numerically balanced, we would count the occurrences of each digit and compare it to the digit value itself. While this guarantees correctness, it is inefficient because there could be many numbers to check between `n` and the next balanced number.

The **key observation** is that numerically balanced numbers are **rare and limited in size**. For digits `1` to `9`, the maximum times they can appear is equal to their value. Therefore, the total number of valid numerically balanced numbers below a certain threshold is small. This allows us to **precompute all numerically balanced numbers up to a certain limit**, sort them, and then perform a simple search to find the smallest number greater than `n`. This approach is efficient and avoids unnecessary checks.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(K * log K) per check | O(1) | Check every number from n+1 upwards, counting digits |
| Optimal | O(1) | O(1) | Precompute all valid numerically balanced numbers up to 10^7, then select the smallest greater than n |

## Algorithm Walkthrough

1. **Precompute candidates**: Generate all numerically balanced numbers using backtracking or enumeration. Limit digit counts to their value and combine digits in all valid ways. Stop once the number exceeds `10^7`, since `n <= 10^6`.
2. **Sort the list**: After generating all numerically balanced numbers, sort them in ascending order to facilitate searching.
3. **Find next number**: Iterate through the sorted list and return the first number strictly greater than `n`.
4. **Return result**: This is guaranteed to be the smallest numerically balanced number greater than `n`.

**Why it works**: Numerically balanced numbers are rare, and we explicitly generate all of them within the limit. Sorting ensures that the first number greater than `n` is indeed the smallest one. Precomputation guarantees correctness and efficiency.

## Python Solution

```python
class Solution:
    def nextBeautifulNumber(self, n: int) -> int:
        from itertools import combinations_with_replacement, permutations
        nums = set()

        def backtrack(curr, counts):
            if curr:
                number = int("".join(curr))
                if number > 0:
                    nums.add(number)
            for d in range(1, 10):
                if counts.get(d, 0) < d:
                    counts[d] = counts.get(d, 0) + 1
                    backtrack(curr + [str(d)], counts)
                    counts[d] -= 1
                    if counts[d] == 0:
                        del counts[d]

        backtrack([], {})
        candidates = sorted(nums)
        for num in candidates:
            if num > n:
                return num
```

This implementation uses **backtracking** to generate all valid numerically balanced numbers. The `counts` dictionary tracks how many times each digit appears. For each recursive step, it tries to add digits `1` to `9` only if their current count is less than the digit value. After generating all numbers, they are sorted and the first number greater than `n` is returned.

## Go Solution

```go
package main

import (
    "sort"
    "strconv"
)

func nextBeautifulNumber(n int) int {
    nums := []int{}
    var backtrack func(curr []int, counts map[int]int)
    backtrack = func(curr []int, counts map[int]int) {
        if len(curr) > 0 {
            numStr := ""
            for _, d := range curr {
                numStr += strconv.Itoa(d)
            }
            num, _ := strconv.Atoi(numStr)
            if num > 0 {
                nums = append(nums, num)
            }
        }
        for d := 1; d <= 9; d++ {
            if counts[d] < d {
                counts[d]++
                backtrack(append(curr, d), counts)
                counts[d]--
            }
        }
    }
    backtrack([]int{}, make(map[int]int))
    sort.Ints(nums)
    for _, num := range nums {
        if num > n {
            return num
        }
    }
    return -1
}
```

The Go version mirrors the Python solution closely. It uses recursion and a map to track digit counts. Slices are used to build numbers, and after generating all candidates, the slice is sorted to find the first number greater than `n`.

## Edge Cases

**Case 1: Input is zero**

Zero is tricky because numerically balanced numbers cannot contain `0`. The implementation handles this correctly by generating numbers starting from `1`.

**Case 2: Input equal to a numerically balanced number**

For example, `22`. The function must return the next greater number, not the input itself. The sorted candidate list and `num > n` comparison ensures correctness.

**Case 3: Input near upper constraint**

Numbers close to `10^6` require generating balanced numbers with more digits. The backtracking approach ensures all valid combinations are generated, so even `n = 999999` correctly returns `122333`.
