[LeetCode Problem 2055](https://leetcode.com/problems/plates-between-candles/)

**Difficulty:** 🟡 Medium  
**Topics:** Array, String, Binary Search, Prefix Sum

## Solution

## Problem Understanding

The problem gives us a string `s` containing only two characters:

- `'*'` represents a plate
- `'|'` represents a candle

We are also given multiple queries, where each query specifies a substring range `[left, right]`.

For every query, we must count how many plates are enclosed between candles inside that substring.

A plate counts only if:

- there is at least one candle to its left inside the substring
- there is at least one candle to its right inside the substring

This means plates outside the outermost candles of the query do not count.

Consider the substring:

```
*||**|
```

The valid enclosed region is between the first and last candle:

```
||**|
```

The two `'*'` characters inside are counted.

The constraints are important:

- `s.length` can be up to `100000`
- `queries.length` can also be up to `100000`

A naive approach that scans every substring independently would be far too slow because it could require up to:

```
100000 × 100000 = 10^10
```

operations in the worst case.

This immediately suggests we need preprocessing so each query can be answered efficiently.

Several edge cases are important:

- A query may contain no candles at all.
- A query may contain only one candle.
- Plates before the first candle or after the last candle must not be counted.
- The closest usable candles may not lie exactly at `left` or `right`.
- Very large inputs require an `O(n + q)` or `O((n + q) log n)` solution.

## Approaches

### Brute Force Approach

The straightforward solution is to process every query independently.

For each query `[left, right]`:

1. Scan the substring from `left` to `right`
2. Find the first candle
3. Find the last candle
4. Count plates between them

This works correctly because we explicitly identify the valid candle boundaries and count only plates enclosed inside them.

However, the time complexity is too expensive.

If each query scans up to `O(n)` characters and there are `q` queries, the total complexity becomes:

```
O(n × q)
```

With both values potentially equal to `100000`, this approach is not feasible.

### Optimal Approach

The key observation is that queries are independent but operate on the same string.

Instead of rescanning the string every time, we can preprocess useful information:

1. A prefix sum array storing how many plates appear up to each position.
2. An array storing the nearest candle to the left of every position.
3. An array storing the nearest candle to the right of every position.

Using these structures:

- we can quickly locate the valid candle boundaries for a query
- we can count plates between them in constant time using prefix sums

This transforms query processing from linear time to constant time after preprocessing.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × q) | O(1) | Scans each query substring independently |
| Optimal | O(n + q) | O(n) | Uses prefix sums and nearest candle preprocessing |

## Algorithm Walkthrough

### Step 1: Build a Prefix Sum Array

We create an array `prefixPlates` where:

```
prefixPlates[i]
```

stores the number of plates from index `0` through `i - 1`.

This allows us to count plates in any range `[l, r]` using:

```
prefixPlates[r + 1] - prefixPlates[l]
```

We use a prefix sum because repeated range counting becomes constant time.

### Step 2: Compute the Nearest Candle on the Left

We create an array `leftCandle`.

For every index `i`, `leftCandle[i]` stores the index of the closest candle at or before `i`.

We scan from left to right:

- whenever we encounter a candle, we update the latest candle position
- otherwise we reuse the previous candle index

This helps us quickly locate the right boundary candle for a query.

### Step 3: Compute the Nearest Candle on the Right

We create an array `rightCandle`.

For every index `i`, `rightCandle[i]` stores the index of the closest candle at or after `i`.

We scan from right to left:

- whenever we encounter a candle, we update the latest candle position
- otherwise we reuse the next candle index

This helps us quickly locate the left boundary candle for a query.

### Step 4: Process Each Query

For a query `[left, right]`:

1. Find the first candle inside the range:

```
start = rightCandle[left]
```
2. Find the last candle inside the range:

```
end = leftCandle[right]
```
3. If either candle does not exist, or `start >= end`, then no enclosed plates exist.
4. Otherwise compute:

```
prefixPlates[end] - prefixPlates[start]
```

This gives the number of plates strictly between the two candles.

### Why it works

The algorithm works because every valid plate must lie between two candles within the query range.

The `rightCandle[left]` value guarantees we choose the first usable candle from the left side.

The `leftCandle[right]` value guarantees we choose the last usable candle from the right side.

All plates between these two candle indices are valid enclosed plates, and the prefix sum array counts them efficiently and correctly.

## Python Solution

```python
from typing import List

class Solution:
    def platesBetweenCandles(self, s: str, queries: List[List[int]]) -> List[int]:
        n = len(s)

        # Prefix sum of plates
        prefix_plates = [0] * (n + 1)

        for i in range(n):
            prefix_plates[i + 1] = prefix_plates[i]
            if s[i] == '*':
                prefix_plates[i + 1] += 1

        # Nearest candle to the left
        left_candle = [-1] * n
        last_candle = -1

        for i in range(n):
            if s[i] == '|':
                last_candle = i
            left_candle[i] = last_candle

        # Nearest candle to the right
        right_candle = [-1] * n
        last_candle = -1

        for i in range(n - 1, -1, -1):
            if s[i] == '|':
                last_candle = i
            right_candle[i] = last_candle

        # Process queries
        result = []

        for left, right in queries:
            start = right_candle[left]
            end = left_candle[right]

            if start == -1 or end == -1 or start >= end:
                result.append(0)
            else:
                plates = prefix_plates[end] - prefix_plates[start]
                result.append(plates)

        return result
```

The implementation closely follows the preprocessing strategy described earlier.

The prefix sum array allows constant time plate counting across ranges. The array is built using cumulative counts so that every position knows how many plates have appeared before it.

The `left_candle` array is constructed with a left to right scan. At every index, it remembers the most recent candle seen so far.

The `right_candle` array is constructed with a right to left scan. At every index, it remembers the nearest candle to the right.

During query processing, we determine the valid candle boundaries immediately. If the boundaries are invalid or missing, the answer is zero. Otherwise, the prefix sum array computes the number of enclosed plates in constant time.

## Go Solution

```go
func platesBetweenCandles(s string, queries [][]int) []int {
	n := len(s)

	// Prefix sum of plates
	prefixPlates := make([]int, n+1)

	for i := 0; i < n; i++ {
		prefixPlates[i+1] = prefixPlates[i]

		if s[i] == '*' {
			prefixPlates[i+1]++
		}
	}

	// Nearest candle to the left
	leftCandle := make([]int, n)
	lastCandle := -1

	for i := 0; i < n; i++ {
		if s[i] == '|' {
			lastCandle = i
		}
		leftCandle[i] = lastCandle
	}

	// Nearest candle to the right
	rightCandle := make([]int, n)
	lastCandle = -1

	for i := n - 1; i >= 0; i-- {
		if s[i] == '|' {
			lastCandle = i
		}
		rightCandle[i] = lastCandle
	}

	// Process queries
	result := make([]int, 0, len(queries))

	for _, query := range queries {
		left := query[0]
		right := query[1]

		start := rightCandle[left]
		end := leftCandle[right]

		if start == -1 || end == -1 || start >= end {
			result = append(result, 0)
		} else {
			plates := prefixPlates[end] - prefixPlates[start]
			result = append(result, plates)
		}
	}

	return result
}
```

The Go implementation mirrors the Python logic almost exactly.

Slices are used for all preprocessing arrays. Since the constraints fit comfortably inside standard integer ranges, regular `int` values are sufficient.

Unlike Python lists, Go slices require explicit allocation with `make`. The result slice is initialized with capacity equal to the number of queries to reduce reallocations during appends.

## Edge Cases

### Queries With No Candles

A substring may contain only plates and no candles at all.

For example:

```
s = "*****"
query = [0,4]
```

A naive implementation might accidentally count all plates, but plates are only valid if enclosed by candles on both sides.

The implementation handles this by checking whether valid boundary candles exist. If either boundary is `-1`, the answer is immediately zero.

### Queries With Only One Candle

A substring containing a single candle still cannot enclose any plates.

For example:

```
s = "***|***"
query = [0,6]
```

Some implementations incorrectly count plates on one side of the candle.

The condition:

```
start >= end
```

prevents this mistake because valid enclosing regions require two distinct candles.

### Plates Outside the Outer Candles

A common bug is accidentally counting plates that appear before the first candle or after the last candle.

For example:

```
s = "**|**|**"
query = [0,7]
```

Only the two middle plates count.

The implementation avoids this issue by always shrinking the query boundaries inward to the nearest valid candles before computing the prefix sum difference.

### Consecutive Candles

Adjacent candles create valid boundaries with zero plates between them.

For example:

```
s = "||"
```

The implementation correctly returns zero because the prefix sum difference between adjacent candle indices is zero.

### Very Large Inputs

The constraints allow both the string length and number of queries to reach `100000`.

Any repeated substring scanning would time out.

The preprocessing strategy guarantees that every query is answered in constant time, making the solution scalable for the maximum input size.
