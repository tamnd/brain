## Solution

## Problem Understanding

The problem gives us an integer array `nums` and an integer `k`. We must choose exactly `k` elements from the array such that their sum is even, and among all such valid subsequences, we want the largest possible sum.

A subsequence means we can pick elements from the array without changing their relative order. However, because we only care about the sum and not the actual ordering of the chosen elements, this problem effectively becomes a selection problem rather than a traditional subsequence dynamic programming problem.

The key requirement is that the final sum must be even. Recall an important parity rule:

- Even + Even = Even
- Odd + Odd = Even
- Even + Odd = Odd

This means the parity of the final sum depends entirely on how many odd numbers are selected. If the count of odd numbers in the subsequence is even, the total sum is even. Otherwise, the total sum is odd.

The constraints are large:

- `nums.length` can be up to `10^5`
- Each value can also be up to `10^5`

These limits immediately tell us that exponential or quadratic enumeration approaches are impossible. We need something close to `O(n log n)` or better.

There are several important edge cases to consider:

- The largest possible sum of `k` elements may be odd, requiring an adjustment.
- There may be no valid even sum at all.
- All numbers may be odd.
- `k` may equal the array length.
- The array can contain zeros, which are even and can help fix parity issues.
- There may be only one available swap candidate when trying to correct parity.

The problem guarantees:

- `1 <= k <= nums.length`
- All numbers are non-negative

The non-negative property is especially useful because it means taking the largest values greedily is always beneficial for maximizing the sum.

## Approaches

### Brute Force Approach

The most straightforward solution is to generate every possible subsequence of size `k`, compute its sum, and track the largest even sum encountered.

For each combination of `k` elements:

1. Compute the sum.
2. Check whether the sum is even.
3. Update the answer if it is larger than the current best.

This approach is correct because it exhaustively examines every valid subsequence of length `k`.

However, the number of combinations is:

$$\binom{n}{k}$$

This grows exponentially and becomes completely infeasible for `n = 100000`.

Even for moderate values such as `n = 30`, this approach is already too slow.

### Optimal Greedy + Sorting Approach

The key observation is that to maximize the sum, we should first take the `k` largest elements.

If their sum is already even, then we are done because no other subsequence of size `k` can have a larger sum.

If the sum is odd, then we must adjust the parity with the smallest possible loss in total value.

To change an odd sum into an even sum, we must swap:

- one odd number with one even number, or
- one even number with one odd number

Why? Because changing the parity of exactly one selected element flips the parity of the total sum.

The optimal strategy is:

- Start with the `k` largest numbers.
- If the sum is odd:

- Remove the smallest odd selected number and replace it with the largest even unselected number.
- Or remove the smallest even selected number and replace it with the largest odd unselected number.
- Choose the better valid replacement.

This works because we minimize the amount lost during the swap while still fixing the parity.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(C(n, k) × k) | O(k) | Enumerates every subsequence of size `k` |
| Optimal | O(n log n) | O(1) or O(n) depending on sorting | Sorts numbers and performs at most two parity-fixing swaps |

## Algorithm Walkthrough

1. Sort the array in descending order.

We want the largest possible sum, so the natural starting point is the `k` largest numbers.
2. Take the first `k` elements.

These are the currently selected elements. Compute their total sum.
3. Check whether the sum is even.

If the sum is already even, return it immediately because no larger subsequence exists.
4. Identify critical candidates inside the selected group.

We need:

- the smallest odd number among the selected elements
- the smallest even number among the selected elements

We choose the smallest because removing a smaller number causes less damage to the total sum.
5. Identify critical candidates outside the selected group.

We need:

- the largest odd number among the unselected elements
- the largest even number among the unselected elements

We choose the largest because replacing with a larger number preserves more of the total sum.
6. Try both valid parity-fixing swaps.

Since the current sum is odd:

- Option 1:

Remove the smallest odd selected number and add the largest even unselected number.
- Option 2:

Remove the smallest even selected number and add the largest odd unselected number.
7. Compute the resulting sums for all valid options.

Some swaps may not be possible if the required parity element does not exist.
8. Return the maximum valid even sum.

If neither swap is possible, return `-1`.

### Why it works

The algorithm works because the `k` largest elements produce the maximum possible sum regardless of parity. If that sum is odd, any valid even-sum subsequence must differ by at least one element.

Changing exactly one selected element is sufficient to flip parity. Among all such parity-fixing changes, the optimal one minimizes the value lost from the selected set and maximizes the value gained from the unselected set. Therefore, considering only the two swap patterns described above guarantees the best possible even sum.

## Python Solution

```python
from typing import List

class Solution:
    def largestEvenSum(self, nums: List[int], k: int) -> int:
        nums.sort(reverse=True)

        selected = nums[:k]
        remaining = nums[k:]

        total_sum = sum(selected)

        if total_sum % 2 == 0:
            return total_sum

        smallest_selected_odd = float("inf")
        smallest_selected_even = float("inf")

        for value in selected:
            if value % 2 == 0:
                smallest_selected_even = min(smallest_selected_even, value)
            else:
                smallest_selected_odd = min(smallest_selected_odd, value)

        largest_remaining_odd = -1
        largest_remaining_even = -1

        for value in remaining:
            if value % 2 == 0:
                largest_remaining_even = max(largest_remaining_even, value)
            else:
                largest_remaining_odd = max(largest_remaining_odd, value)

        best = -1

        # Replace selected odd with remaining even
        if (
            smallest_selected_odd != float("inf")
            and largest_remaining_even != -1
        ):
            candidate = (
                total_sum
                - smallest_selected_odd
                + largest_remaining_even
            )
            best = max(best, candidate)

        # Replace selected even with remaining odd
        if (
            smallest_selected_even != float("inf")
            and largest_remaining_odd != -1
        ):
            candidate = (
                total_sum
                - smallest_selected_even
                + largest_remaining_odd
            )
            best = max(best, candidate)

        return best
```

The implementation begins by sorting the array in descending order so that the first `k` elements represent the maximum possible sum.

The algorithm then separates the array into two logical groups:

- `selected`, the current chosen subsequence
- `remaining`, the unused numbers

After computing the total sum of the selected elements, the code immediately checks whether the sum is even. If it is, we can safely return it because no better sum exists.

If the sum is odd, the implementation searches for the smallest odd and smallest even numbers inside the selected set. These are the best candidates for removal because they minimize the loss.

Next, the code scans the remaining elements to find the largest odd and largest even values available for replacement. These maximize the gain.

The algorithm then evaluates the two possible parity-correcting swaps and keeps the larger valid result.

If no valid swap exists, the answer remains `-1`.

## Go Solution

```go
package main

import (
	"math"
	"sort"
)

func largestEvenSum(nums []int, k int) int64 {
	sort.Slice(nums, func(i, j int) bool {
		return nums[i] > nums[j]
	})

	var totalSum int64 = 0

	for i := 0; i < k; i++ {
		totalSum += int64(nums[i])
	}

	if totalSum%2 == 0 {
		return totalSum
	}

	smallestSelectedOdd := math.MaxInt32
	smallestSelectedEven := math.MaxInt32

	for i := 0; i < k; i++ {
		value := nums[i]

		if value%2 == 0 {
			if value < smallestSelectedEven {
				smallestSelectedEven = value
			}
		} else {
			if value < smallestSelectedOdd {
				smallestSelectedOdd = value
			}
		}
	}

	largestRemainingOdd := -1
	largestRemainingEven := -1

	for i := k; i < len(nums); i++ {
		value := nums[i]

		if value%2 == 0 {
			if value > largestRemainingEven {
				largestRemainingEven = value
			}
		} else {
			if value > largestRemainingOdd {
				largestRemainingOdd = value
			}
		}
	}

	var best int64 = -1

	// Replace selected odd with remaining even
	if smallestSelectedOdd != math.MaxInt32 && largestRemainingEven != -1 {
		candidate := totalSum -
			int64(smallestSelectedOdd) +
			int64(largestRemainingEven)

		if candidate > best {
			best = candidate
		}
	}

	// Replace selected even with remaining odd
	if smallestSelectedEven != math.MaxInt32 && largestRemainingOdd != -1 {
		candidate := totalSum -
			int64(smallestSelectedEven) +
			int64(largestRemainingOdd)

		if candidate > best {
			best = candidate
		}
	}

	return best
}
```

The Go implementation follows the same logic as the Python version, but there are a few language-specific differences.

The function signature requires returning `int64`, so all sum calculations use `int64` to avoid overflow concerns.

Go does not have built-in descending sort for slices, so `sort.Slice` is used with a custom comparator.

Instead of Python's `float("inf")`, the code uses `math.MaxInt32` as a sentinel value for tracking minimum candidates.

The rest of the implementation mirrors the same greedy parity-fixing logic.

## Edge Cases

One important edge case occurs when all numbers are odd and `k` is odd. In this situation, every possible subsequence of size `k` will also have an odd sum because the sum of an odd number of odd integers is always odd. A naive implementation might incorrectly assume some replacement can fix the parity. The algorithm correctly detects that no valid parity-changing swap exists and returns `-1`.

Another tricky case is when the initial top `k` elements already produce the optimal even sum. Some implementations unnecessarily attempt swaps anyway, which can accidentally reduce the answer. This solution immediately returns once it detects the sum is even, guaranteeing the maximum possible sum is preserved.

A third important edge case happens when only one parity-fixing swap type is available. For example, there may be no even number outside the selected group, making one replacement impossible. The implementation independently checks both swap possibilities and only evaluates valid candidates, preventing invalid computations or incorrect answers.

A fourth subtle case involves zeros. Since zero is even, it can participate in parity-fixing operations. Some buggy implementations mishandle zero because it is neither positive nor negative. This algorithm uses modulo parity checks, so zero is naturally treated as an even number and works correctly in all cases.
