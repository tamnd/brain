## Solution

## Problem Understanding

This problem asks us to maximize the total money earned while driving a taxi along a one directional road. The road contains points numbered from `1` to `n`, and the taxi always moves forward, never backward.

Each passenger request is represented as:

```
[start, end, tip]
```

If we accept that ride, we earn:

```
(end - start) + tip
```

This means every ride gives a base fare equal to the distance traveled, plus an additional tip.

The important restriction is that we can carry at most one passenger at a time. Therefore, two rides cannot overlap in time. However, we are allowed to drop off one passenger and immediately pick up another at the same point.

The goal is to choose a subset of non overlapping rides such that the total earnings are maximized.

The input consists of:

- `n`, the total number of road points
- `rides`, the list of passenger requests

The output is a single integer representing the maximum total earnings possible.

The constraints are important:

- `n` can be as large as `10^5`
- `rides.length` can be as large as `3 * 10^4`

A brute force search over all subsets of rides would be completely infeasible because the number of combinations grows exponentially.

The problem structure strongly suggests an interval scheduling style dynamic programming solution because:

- Each ride is an interval `[start, end]`
- We need a maximum value subset of non overlapping intervals
- We must efficiently find the best ride that ends before another begins

Several edge cases are important:

- Multiple rides may end at the same location
- Multiple rides may start at the same location
- A ride can begin exactly where another ends, this is allowed
- The optimal solution may skip many rides in favor of a few highly profitable ones
- The best answer may involve taking no ride from the beginning of the road until later points

The problem guarantees:

- `start < end`
- All rides move forward
- Tips are positive

These guarantees simplify the logic because we never need to handle invalid intervals or backward movement.

## Approaches

### Brute Force Approach

A brute force solution would try every possible subset of rides and check whether the rides overlap.

For every subset:

1. Sort rides by ending time
2. Verify that no two selected rides overlap
3. Compute the total earnings
4. Track the maximum value

This approach is correct because it explicitly examines every valid combination of rides. Eventually, it will discover the optimal subset.

However, this is far too slow. With up to `3 * 10^4` rides, the number of subsets is:

```
2^(30000)
```

which is astronomically large.

Even a recursive backtracking approach that branches between taking and skipping rides would still be exponential in the worst case.

### Key Insight

The important observation is that this problem is actually a weighted interval scheduling problem.

Each ride behaves like an interval:

```
[start, end]
```

with a weight:

```
(end - start + tip)
```

If we sort rides by ending position, then for each ride we only need to know:

```
What is the best earnings value achievable before this ride starts?
```

This transforms the problem into dynamic programming.

The challenge becomes efficiently finding the latest non overlapping ride.

We can solve this efficiently using:

- Sorting
- Dynamic programming
- Binary search

The binary search allows us to quickly locate the last compatible ride for each current ride.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^m) | O(m) | Tries every subset of rides |
| Optimal | O(m log m) | O(m) | Dynamic programming with binary search |

Here, `m = len(rides)`.

## Algorithm Walkthrough

### Step 1: Compute Ride Profit

For each ride:

```
[start, end, tip]
```

the earned money is:

```
(end - start) + tip
```

We will use this as the ride's profit.

### Step 2: Sort Rides by Ending Position

We sort all rides by their ending point.

This ordering is important because dynamic programming will process rides in chronological order of completion.

After sorting, when considering a ride, all earlier rides have already been processed.

### Step 3: Create Arrays for Binary Search

We extract all ending positions into a separate array:

```
ends = [ride_end values]
```

This array is sorted because rides were sorted by end time.

We use this array for binary searching the latest ride that finishes before the current ride starts.

### Step 4: Define Dynamic Programming State

Let:

```
dp[i]
```

represent the maximum earnings obtainable using the first `i` rides in sorted order.

This means:

- `dp[0] = 0`
- `dp[i]` stores the best answer considering rides `0` through `i-1`

### Step 5: Process Each Ride

For each ride:

```
[start, end, tip]
```

we compute:

```
profit = end - start + tip
```

Then we binary search to find the last ride whose end position is less than or equal to `start`.

This gives us the latest compatible ride.

Suppose that compatible ride index corresponds to DP state `j`.

Then:

```
take = dp[j] + profit
```

We also have the option to skip the current ride:

```
skip = dp[i - 1]
```

So:

```
dp[i] = max(skip, take)
```

### Step 6: Return Final Answer

The answer is:

```
dp[m]
```

where `m` is the number of rides.

### Why it works

The algorithm works because every ride decision is independent once we know the best achievable earnings before its start time.

Sorting by ending position guarantees that all compatible earlier rides have already been computed.

For each ride, the optimal solution must either:

- include the current ride, combined with the best compatible earlier solution
- exclude the current ride

Taking the maximum of these two possibilities guarantees optimality through dynamic programming.

## Python Solution

```python
from typing import List
from bisect import bisect_right

class Solution:
    def maxTaxiEarnings(self, n: int, rides: List[List[int]]) -> int:
        rides.sort(key=lambda ride: ride[1])

        ends = [ride[1] for ride in rides]

        m = len(rides)

        dp = [0] * (m + 1)

        for i in range(1, m + 1):
            start, end, tip = rides[i - 1]

            profit = end - start + tip

            compatible_index = bisect_right(ends, start, hi=i - 1)

            take = dp[compatible_index] + profit

            skip = dp[i - 1]

            dp[i] = max(take, skip)

        return dp[m]
```

The implementation begins by sorting rides according to their ending positions. This is necessary because the dynamic programming transition depends on previously completed rides.

The `ends` array stores only the ride end positions. Since it is sorted, Python's `bisect_right` function can efficiently locate the latest compatible ride.

The DP array uses one based indexing semantics:

- `dp[0]` means no rides considered
- `dp[i]` means the best result using the first `i` rides

For each ride, we compute the ride profit and use binary search to locate the last ride whose ending point does not exceed the current ride's start.

The variable `compatible_index` directly corresponds to the number of rides that can safely precede the current ride.

We then compare:

- taking the ride
- skipping the ride

and store the better option.

Finally, `dp[m]` contains the optimal answer.

## Go Solution

```go
package main

import (
	"sort"
)

func maxTaxiEarnings(n int, rides [][]int) int64 {
	sort.Slice(rides, func(i, j int) bool {
		return rides[i][1] < rides[j][1]
	})

	m := len(rides)

	ends := make([]int, m)

	for i := 0; i < m; i++ {
		ends[i] = rides[i][1]
	}

	dp := make([]int64, m+1)

	for i := 1; i <= m; i++ {
		start := rides[i-1][0]
		end := rides[i-1][1]
		tip := rides[i-1][2]

		profit := int64(end - start + tip)

		compatibleIndex := sort.Search(i-1, func(j int) bool {
			return ends[j] > start
		})

		take := dp[compatibleIndex] + profit

		skip := dp[i-1]

		if take > skip {
			dp[i] = take
		} else {
			dp[i] = skip
		}
	}

	return dp[m]
}
```

The Go implementation follows the same logic as the Python solution.

One important difference is integer handling. Since total earnings can exceed the range of a 32 bit integer, the DP array uses `int64`.

Go does not provide a direct equivalent of Python's `bisect_right`, so we use `sort.Search` to find the first ending position greater than `start`. The returned index naturally represents the number of compatible rides.

Slices are used throughout instead of arrays because the number of rides is dynamic.

## Edge Cases

### Rides That Touch at Endpoints

A very important detail is that a ride ending at point `x` is compatible with another ride starting at point `x`.

Many incorrect implementations accidentally require:

```
previous_end < current_start
```

instead of:

```
previous_end <= current_start
```

Our implementation correctly handles this by using `bisect_right`, which allows equality.

### Multiple Overlapping Rides

Consider a situation where many rides overlap heavily:

```
[[1,10,1], [2,9,100], [3,8,200]]
```

A greedy strategy based on earliest ending ride or longest distance would fail.

The dynamic programming approach evaluates both taking and skipping every ride, guaranteeing that the globally optimal combination is chosen.

### Very Large Earnings

Tips can be as large as `10^5`, and there can be many rides.

The total earnings may exceed 32 bit integer limits.

Python handles large integers automatically, but in Go we explicitly use `int64` for the DP array and final result to avoid overflow bugs.
