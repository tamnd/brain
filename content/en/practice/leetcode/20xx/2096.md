## Solution

## Problem Understanding

This problem gives us a binary tree where every node has a unique value. We are also given two specific node values:

- `startValue`, representing the node where we begin
- `destValue`, representing the node where we want to end

The goal is to produce the shortest sequence of directions needed to travel from the start node to the destination node.

The output string may contain three characters:

- `'L'` means move from a node to its left child
- `'R'` means move from a node to its right child
- `'U'` means move from a node to its parent

The tree is not a graph with explicit parent pointers, so we need to derive the path using tree traversal.

The important observation is that movement between two nodes in a tree always goes through their Lowest Common Ancestor, often abbreviated as LCA. If we can determine:

1. How to move from the root to the start node
2. How to move from the root to the destination node

then we can reconstruct the shortest route between them.

The constraints are important:

- The tree may contain up to `10^5` nodes
- Node values are unique
- `startValue != destValue`

A solution with quadratic complexity would be too slow for the largest inputs. We need an algorithm that runs in linear time.

There are several edge cases worth considering:

- The start node may be an ancestor of the destination node
- The destination node may be an ancestor of the start node
- The two nodes may lie in completely different subtrees
- The root itself could be either the start or destination node
- The tree may be highly unbalanced, essentially behaving like a linked list

The uniqueness of node values guarantees that every target node can be identified unambiguously.

## Approaches

### Brute Force Approach

A brute force solution would treat the tree like an undirected graph.

We could first traverse the tree and build a graph representation where each node connects to:

- Its left child
- Its right child
- Its parent

After constructing this graph, we could run Breadth-First Search from `startValue` to `destValue`. BFS guarantees the shortest path in an unweighted graph.

While this works correctly, it requires:

- Building a parent map or adjacency list
- Performing BFS
- Storing path reconstruction information

Although this approach is still technically linear in complexity, it uses more auxiliary structures than necessary and does not fully exploit the properties of a tree.

### Optimal Approach

The key insight is that every path between two tree nodes passes through their Lowest Common Ancestor.

Instead of converting the tree into a graph, we can:

1. Find the path from the root to `startValue`
2. Find the path from the root to `destValue`
3. Remove the shared prefix from both paths

The shared prefix represents the path from the root to the LCA.

After removing the common portion:

- Every remaining step in the start path becomes `'U'`
- The remaining destination path already consists of `'L'` and `'R'`

This directly constructs the shortest path.

For example:

- Root to start: `"LLR"`
- Root to destination: `"LRR"`

Shared prefix: `"L"`

Remaining:

- Start suffix: `"LR"`
- Destination suffix: `"RR"`

Result:

- `"UU"` for leaving the start node upward
- `"RR"` for descending to the destination

Final answer: `"UURR"`

This approach is elegant because trees naturally encode unique paths from the root to every node.

## Comparison of Approaches

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Build graph with parent edges, then run BFS |
| Optimal | O(n) | O(n) | Find root-to-node paths and remove common prefix |

## Algorithm Walkthrough

1. Perform a DFS traversal to find the path from the root to `startValue`.

During traversal:

- Append `'L'` before exploring the left subtree
- Append `'R'` before exploring the right subtree
- Backtrack if the target is not found

This produces the exact sequence needed to move from the root to the start node.
2. Perform another DFS traversal to find the path from the root to `destValue`.

The traversal logic is identical. At the end, we obtain another path string representing directions from the root to the destination node.
3. Compare the two paths character by character from the beginning.

As long as characters match, both paths are moving through the same nodes. This common prefix represents the route from the root to the Lowest Common Ancestor.
4. Stop when the paths diverge.

Suppose:

- `startPath = "LLR"`
- `destPath = "LRR"`

The common prefix is `"L"`.
5. Compute how many upward moves are required.

Everything remaining in the start path after the common prefix must be traversed upward toward the LCA.

If the remaining suffix length is `k`, we need `"U" * k`.
6. Append the remaining destination suffix.

The destination suffix already tells us how to descend from the LCA to the destination node.
7. Return the concatenated result.

### Why it works

A binary tree has exactly one simple path between any two nodes. The shared prefix of the root-to-node paths identifies the Lowest Common Ancestor. To move from the start node to the destination node, we first move upward from the start node to the LCA, then downward from the LCA to the destination node. Because the tree path is unique, this construction always produces the shortest valid route.

## Python Solution

```python
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import Optional, List

class Solution:
    def getDirections(
        self,
        root: Optional[TreeNode],
        startValue: int,
        destValue: int
    ) -> str:

        def find_path(
            node: Optional[TreeNode],
            target: int,
            path: List[str]
        ) -> bool:
            if not node:
                return False

            if node.val == target:
                return True

            path.append("L")
            if find_path(node.left, target, path):
                return True
            path.pop()

            path.append("R")
            if find_path(node.right, target, path):
                return True
            path.pop()

            return False

        start_path = []
        dest_path = []

        find_path(root, startValue, start_path)
        find_path(root, destValue, dest_path)

        common_length = 0

        while (
            common_length < len(start_path)
            and common_length < len(dest_path)
            and start_path[common_length] == dest_path[common_length]
        ):
            common_length += 1

        upward_moves = "U" * (len(start_path) - common_length)
        downward_moves = "".join(dest_path[common_length:])

        return upward_moves + downward_moves
```

The implementation begins with a helper DFS function named `find_path`. This function searches for a target node while simultaneously recording the path taken from the root.

The `path` list acts like a mutable stack. Before traversing left, the algorithm appends `'L'`. Before traversing right, it appends `'R'`. If a subtree does not contain the target, the algorithm backtracks by removing the last direction.

Once both paths are collected, the algorithm compares them from left to right to determine the longest shared prefix. This shared portion corresponds to the path from the root to the Lowest Common Ancestor.

The remaining directions in `start_path` indicate how many upward moves are necessary. Each remaining edge becomes a `'U'`.

Finally, the remaining portion of `dest_path` is appended directly, producing the shortest route from start to destination.

## Go Solution

```go
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */

func getDirections(root *TreeNode, startValue int, destValue int) string {

	var findPath func(node *TreeNode, target int, path *[]byte) bool

	findPath = func(node *TreeNode, target int, path *[]byte) bool {
		if node == nil {
			return false
		}

		if node.Val == target {
			return true
		}

		*path = append(*path, 'L')
		if findPath(node.Left, target, path) {
			return true
		}
		*path = (*path)[:len(*path)-1]

		*path = append(*path, 'R')
		if findPath(node.Right, target, path) {
			return true
		}
		*path = (*path)[:len(*path)-1]

		return false
	}

	startPath := []byte{}
	destPath := []byte{}

	findPath(root, startValue, &startPath)
	findPath(root, destValue, &destPath)

	commonLength := 0

	for commonLength < len(startPath) &&
		commonLength < len(destPath) &&
		startPath[commonLength] == destPath[commonLength] {

		commonLength++
	}

	result := make([]byte, 0)

	for i := commonLength; i < len(startPath); i++ {
		result = append(result, 'U')
	}

	result = append(result, destPath[commonLength:]...)

	return string(result)
}
```

The Go implementation follows the same algorithmic structure as the Python version, but uses byte slices for efficient mutable string construction.

Instead of Python lists, Go uses `[]byte` to store path directions. Backtracking is implemented by slicing away the last element:

```go
*path = (*path)[:len(*path)-1]
```

Go does not have Python's built in string multiplication, so upward moves are appended one at a time using a loop.

Nil checking is explicit through pointer comparisons, which is standard in Go tree problems.

## Worked Examples

### Example 1

Input:

```
root = [5,1,2,3,null,6,4]
startValue = 3
destValue = 6
```

Tree structure:

```
        5
       / \
      1   2
     /   / \
    3   6   4
```

### Step 1: Find path to start node

We search for node `3`.

| Current Node | Action | Path |
| --- | --- | --- |
| 5 | Go left | L |
| 1 | Go left | LL |
| 3 | Found target | LL |

So:

```
startPath = "LL"
```

### Step 2: Find path to destination node

We search for node `6`.

| Current Node | Action | Path |
| --- | --- | --- |
| 5 | Go right | R |
| 2 | Go left | RL |
| 6 | Found target | RL |

So:

```
destPath = "RL"
```

### Step 3: Remove common prefix

Compare:

```
startPath = LL
destPath  = RL
```

The first characters already differ, so the common prefix length is `0`.

### Step 4: Build final answer

Remaining start suffix:

```
LL
```

This requires:

```
UU
```

Remaining destination suffix:

```
RL
```

Final answer:

```
UURL
```

### Example 2

Input:

```
root = [2,1]
startValue = 2
destValue = 1
```

Tree:

```
    2
   /
  1
```

### Step 1: Find start path

The root itself is the start node.

```
startPath = ""
```

### Step 2: Find destination path

Move left from root.

```
destPath = "L"
```

### Step 3: Remove common prefix

The start path is empty, so the common prefix length is `0`.

### Step 4: Build answer

No upward moves are needed.

Append destination suffix:

```
"L"
```

Final answer:

```
L
```

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each DFS traversal may visit every node once |
| Space | O(n) | Recursive call stack and stored paths |

The algorithm performs two DFS traversals, one for the start node and one for the destination node. In the worst case, each traversal touches every node in the tree, producing linear time complexity.

The space complexity comes from recursion depth and path storage. In the worst case of a highly skewed tree, recursion depth can reach `n`.

## Edge Cases

One important edge case occurs when the start node is an ancestor of the destination node. In this situation, the start path is a prefix of the destination path. After removing the shared prefix, there are no upward moves required. The algorithm correctly returns only the remaining downward directions.

Another important case happens when the destination node is an ancestor of the start node. Here, the destination path becomes the shared prefix. After removing it, only upward moves remain. The implementation handles this naturally by generating the required number of `'U'` characters.

A highly unbalanced tree is also important to consider. If the tree resembles a linked list, recursion depth may become very large. The algorithm still remains correct because DFS continues to operate on a single branch, and the complexity remains linear. The stored paths also grow linearly with tree depth, which is acceptable within the problem constraints.

A final subtle case occurs when the Lowest Common Ancestor is the root itself. In this scenario, the two root-to-node paths diverge immediately. The shared prefix length becomes zero, and the algorithm correctly constructs a path consisting of upward moves followed by downward moves through the opposite subtree.
