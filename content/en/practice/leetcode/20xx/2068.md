## Solution

## Problem Understanding

The problem asks us to determine whether two strings are "almost equivalent" based on the frequency of each lowercase English letter.

For every character from `'a'` to `'z'`, we compare how many times that character appears in `word1` and `word2`. If the absolute difference in frequency for every letter is at most `3`, then the two strings are considered almost equivalent. Otherwise, they are not.

For example, if `'a'` appears `4` times in `word1` and `1` time in `word2`, the difference is `3`, which is acceptable. However, if the difference becomes `4` or greater, the answer must be `false`.

The input consists of two strings of equal length. Each string contains only lowercase English letters. The output is a boolean value:

- `true` if all frequency differences are at most `3`
- `false` if at least one character differs by more than `3`

The constraints are relatively small:

- `1 <= n <= 100`
- only lowercase English letters exist

These constraints immediately suggest that counting frequencies is both simple and efficient. Since there are only 26 possible letters, we can store counts in a fixed-size array instead of using more complicated data structures.

Several edge cases are important:

- One letter may appear many times in one string and not at all in the other.
- The strings may already be identical.
- The maximum allowed difference is exactly `3`, so the implementation must use `>` rather than `>=`.
- Some letters may not appear in either string, and those letters should not affect the result.

## Approaches

### Brute Force Approach

A brute-force solution would examine every lowercase letter individually and count its occurrences in both strings each time.

For every character from `'a'` to `'z'`:

1. Traverse `word1` and count occurrences.
2. Traverse `word2` and count occurrences.
3. Compute the absolute difference.
4. Return `false` if the difference exceeds `3`.

This approach is correct because it explicitly computes the required frequency comparison for every character.

However, it repeatedly scans the strings many times. Since there are 26 letters and each scan takes `O(n)` time, the total complexity becomes `O(26 * n)`. While this is still acceptable for the given constraints, it is inefficient because the same characters are processed repeatedly.

### Optimal Approach

The key observation is that we only need the final frequency counts of each character.

Instead of rescanning the strings for every letter, we can process both strings once and maintain frequency arrays of size `26`.

We count how many times each character appears in both strings, then compare the corresponding counts. If any difference exceeds `3`, we immediately return `false`. Otherwise, we return `true`.

Because the alphabet size is fixed at 26, this solution runs in linear time relative to the string length and uses constant extra space.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(26 × n) | O(1) | Recounts frequencies for every letter |
| Optimal | O(n) | O(1) | Counts frequencies once using fixed arrays |

## Algorithm Walkthrough

1. Create two arrays of size `26`, initialized with zeros.

Each index represents one lowercase English letter:

- index `0` corresponds to `'a'`
- index `1` corresponds to `'b'`
- and so on

These arrays will store character frequencies for `word1` and `word2`.
2. Traverse both strings.

For every character in `word1`, increment its corresponding count in the first frequency array.

For every character in `word2`, increment its corresponding count in the second frequency array.
3. Compare the frequencies for all 26 letters.

For each index from `0` to `25`:

- compute the absolute difference
- if the difference exceeds `3`, return `false`
4. If all letters satisfy the condition, return `true`.

### Why it works

The algorithm works because it directly verifies the exact condition given in the problem statement. The frequency arrays accurately store how many times each letter appears in each string. Comparing all 26 positions guarantees that every lowercase letter is checked exactly once. If no difference exceeds `3`, then the strings satisfy the definition of almost equivalent.

## Python Solution

```python
class Solution:
    def checkAlmostEquivalent(self, word1: str, word2: str) -> bool:
        freq1 = [0] * 26
        freq2 = [0] * 26

        for ch in word1:
            freq1[ord(ch) - ord('a')] += 1

        for ch in word2:
            freq2[ord(ch) - ord('a')] += 1

        for i in range(26):
            if abs(freq1[i] - freq2[i]) > 3:
                return False

        return True
```

The implementation begins by creating two fixed-size arrays to store letter frequencies. Since the problem guarantees lowercase English letters only, a length of 26 is sufficient.

The expression `ord(ch) - ord('a')` converts a character into its array index. For example:

- `'a'` becomes `0`
- `'b'` becomes `1`
- `'z'` becomes `25`

The first loop counts frequencies for `word1`, while the second loop counts frequencies for `word2`.

After both counts are complete, the final loop compares corresponding frequencies. The function immediately returns `False` if any difference exceeds `3`. If all checks pass, the strings are almost equivalent, so the function returns `True`.

## Go Solution

```go
func checkAlmostEquivalent(word1 string, word2 string) bool {
    freq1 := make([]int, 26)
    freq2 := make([]int, 26)

    for _, ch := range word1 {
        freq1[ch-'a']++
    }

    for _, ch := range word2 {
        freq2[ch-'a']++
    }

    for i := 0; i < 26; i++ {
        diff := freq1[i] - freq2[i]

        if diff < 0 {
            diff = -diff
        }

        if diff > 3 {
            return false
        }
    }

    return true
}
```

The Go implementation follows the same logic as the Python version. The main difference is that Go does not provide a built-in absolute value function for integers in the standard language syntax, so the code manually converts negative differences into positive values.

The frequency arrays are created using slices of length 26. Characters are processed as runes, and subtracting `'a'` gives the correct array index.

Because the maximum string length is only 100, integer overflow is not a concern.

## Edge Cases

One important edge case occurs when the frequency difference is exactly `3`. Since the problem states that differences are allowed to be "at most 3", this case should return `true`. A common bug is accidentally using `>= 3` instead of `> 3`. The implementation correctly checks `> 3`, so exact boundary values are accepted.

Another important case is when one letter appears many times in one string and not at all in the other. For example, `"aaaa"` and `"bbbb"` produce a difference of `4` for both `'a'` and `'b'`. Some incorrect implementations only compare letters that appear in both strings. This solution checks all 26 letters, ensuring missing characters are handled correctly.

A third edge case involves very small strings, such as length `1`. Even though the strings are tiny, the logic must still work correctly. The implementation does not rely on any minimum frequency or repeated characters, so single-character inputs are processed naturally by the same counting mechanism.
