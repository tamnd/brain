## Problem Understanding

The problem gives us a database table named `Store`, where each row represents a bill issued to a customer. Every row contains three columns:

- `bill_id`, a unique identifier for the bill
- `customer_id`, the customer who owns the bill
- `amount`, the bill amount

We need to determine how many **distinct customers** have **at least one bill whose amount is strictly greater than 500**.

In simpler terms, we are looking for customers who qualify as "rich" according to the problem's rule. A customer is considered rich if they have at least one transaction above `500`. Once a customer qualifies, they should only be counted once, even if they have multiple qualifying bills.

The output should be a single value named `rich_count`, representing the number of qualifying customers.

The problem is classified as a database problem, which means the intended solution is an SQL query rather than an algorithm in Python or Go. Since `bill_id` is guaranteed to be unique, we do not need to worry about duplicate bill entries. However, multiple bills may belong to the same customer, so handling duplicate `customer_id` values correctly is important.

An important detail is the phrase **"strictly greater than 500"**. Bills equal to `500` do not count. Another important observation is that customers can have many bills, but each qualifying customer should be counted exactly once.

## Approaches

### Brute Force Approach

A brute-force way to solve this problem would be to iterate through every row in the table and maintain a set of customers whose bill amount exceeds `500`. Every time a qualifying bill is encountered, we insert the customer into the set. After processing all rows, the size of the set becomes the answer.

This works because a set naturally removes duplicates, ensuring each customer is counted only once regardless of how many qualifying bills they have.

Although this approach is straightforward conceptually, in SQL we usually want to express the logic declaratively rather than manually simulating iteration.

### Optimal Approach

The key observation is that SQL already provides tools for filtering and deduplication.

We can first filter rows where `amount > 500`, since only these bills matter. Then we count the number of distinct `customer_id` values among those filtered rows.

The `COUNT(DISTINCT customer_id)` operation is ideal here because it automatically prevents counting the same customer multiple times.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Iterate through rows and track qualifying customers in a set |
| Optimal | O(n) | O(1) or O(k) | Filter qualifying rows and count distinct customers |

Here, `n` is the number of rows in the `Store` table, and `k` is the number of distinct qualifying customers.

## Algorithm Walkthrough

1. Examine every row in the `Store` table because every bill could potentially qualify a customer.
2. Filter the rows so that only bills with `amount > 500` remain. Bills with amount `500` or below cannot make a customer rich.
3. Extract the `customer_id` values from the filtered rows. Since one customer may have multiple qualifying bills, duplicates may exist.
4. Count only distinct customer IDs using `COUNT(DISTINCT customer_id)` so that each customer contributes at most one to the result.
5. Return the count as `rich_count`.

### Why it works

The algorithm works because the definition of a rich customer is precisely "a customer with at least one bill greater than 500." By filtering rows using `amount > 500`, we isolate all qualifying bills. Then, counting distinct `customer_id` values guarantees that each qualifying customer is counted exactly once, which matches the problem definition.

## SQL Solution

```sql
SELECT COUNT(DISTINCT customer_id) AS rich_count
FROM Store
WHERE amount > 500;
```

After filtering rows where the amount exceeds `500`, the query counts distinct customer IDs. The alias `rich_count` ensures the output column matches the required format.

The use of `DISTINCT` is critical because a customer may have multiple qualifying bills. Without it, the same customer could be counted more than once.

## Edge Cases

One important edge case occurs when a customer has multiple bills greater than `500`. A naive implementation might count every qualifying bill instead of every qualifying customer. Using `COUNT(DISTINCT customer_id)` ensures each customer contributes only once to the final answer.

Another important case is when a bill amount is exactly `500`. Since the requirement says **strictly greater than 500**, these rows must not qualify. The query correctly handles this by using `WHERE amount > 500` rather than `>= 500`.

A third edge case is when no customer has a qualifying bill. In this situation, the filtered dataset becomes empty. SQL handles this naturally, and `COUNT(DISTINCT customer_id)` correctly returns `0`.

Finally, a customer may have both qualifying and non-qualifying bills. As long as at least one bill exceeds `500`, the customer should be included. Because the filtering step retains all qualifying rows before deduplication, such customers are handled correctly.
