## Solution

## Problem Understanding

This problem asks us to classify each store member into one of four categories based on how often their visits result in purchases. The data is spread across three tables, and we need to combine information from all of them to compute a conversion rate for every member.

The `Members` table contains the list of all registered members. Each member has a unique `member_id` and a `name`.

The `Visits` table records every store visit made by a member. Each row represents a single visit and contains a unique `visit_id`, the `member_id` of the visitor, and the visit date.

The `Purchases` table records which visits resulted in purchases. A purchase is associated with exactly one `visit_id`, and each visit can appear at most once in this table.

The key metric is the conversion rate:

$$\text{conversion rate} = \frac{100 \times \text{number of purchases}}{\text{number of visits}}$$

The category rules are:

- `"Diamond"` if conversion rate ≥ 80
- `"Gold"` if conversion rate ≥ 50 and < 80
- `"Silver"` if conversion rate < 50
- `"Bronze"` if the member never visited the store

The output should contain one row for every member, including:

- `member_id`
- `name`
- `category`

The order of the result does not matter.

A subtle but important detail is that purchases are counted by counting rows in the `Purchases` table, not by summing `charged_amount`. Even a very large purchase still counts as only one successful conversion.

Another important edge case is members with zero visits. Their conversion rate would mathematically involve division by zero, so the problem defines them separately as `"Bronze"` members.

The constraints are not explicitly stated, but this is a database aggregation problem. That strongly suggests we should avoid repeated scans or nested queries that recompute counts for every member individually. Instead, we should aggregate visit counts and purchase counts efficiently using joins and grouping.

Some important edge cases include:

- Members who never visited the store
- Members who visited but never purchased
- Members whose conversion rate falls exactly on category boundaries such as 50 or 80
- Members with many visits but only a few purchases
- Visits that do not appear in the `Purchases` table

A correct solution must handle all of these cases cleanly.

## Approaches

### Brute Force Approach

A brute force solution would process each member individually.

For every member, we could:

1. Scan the entire `Visits` table to count how many visits belong to that member.
2. For each of those visits, scan the `Purchases` table to determine whether a purchase exists.
3. Compute the conversion rate.
4. Assign the category.

This approach works because it directly follows the problem definition. For each member, we explicitly count visits and purchases before computing the classification.

However, the approach is inefficient because it repeatedly scans the same tables for every member. If there are `M` members, `V` visits, and `P` purchases, then the repeated table scans become expensive.

### Optimal Approach

The key insight is that this problem is fundamentally an aggregation problem.

Instead of recomputing counts for every member separately, we can:

1. Join visits with purchases once.
2. Group results by member.
3. Count visits and purchases simultaneously.
4. Use a `CASE` expression to classify members.

This avoids redundant work and leverages SQL's aggregation capabilities efficiently.

The optimal solution uses:

- `LEFT JOIN` so members without visits are preserved
- `COUNT(v.visit_id)` to count visits
- `COUNT(p.visit_id)` to count successful purchases
- `CASE` logic to assign categories

Because every table is processed only once through joins and grouping, the solution scales much better.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(M × (V + P)) | O(1) | Repeatedly scans tables for every member |
| Optimal | O(M + V + P) | O(M) | Uses joins and aggregation efficiently |

## Algorithm Walkthrough

1. Start with the `Members` table because every member must appear in the final result, even if they never visited the store.
2. Perform a `LEFT JOIN` from `Members` to `Visits` using `member_id`. This ensures that members with zero visits are still included in the output.
3. Perform another `LEFT JOIN` from `Visits` to `Purchases` using `visit_id`. This allows us to determine which visits resulted in purchases.
4. Group the combined rows by:

- `member_id`
- `name`

Grouping lets us compute statistics for each member independently.
5. Count the total visits using:

```
COUNT(v.visit_id)
```

This gives the denominator for the conversion rate.
6. Count the purchases using:

```
COUNT(p.visit_id)
```

Since only visits with purchases appear in the `Purchases` table, this gives the numerator.
7. Use a `CASE` expression to determine the category:

- If visits = 0 → `"Bronze"`
- Else if conversion rate ≥ 80 → `"Diamond"`
- Else if conversion rate ≥ 50 → `"Gold"`
- Otherwise → `"Silver"`
8. Return the final result with:

- `member_id`
- `name`
- `category`

### Why it works

The solution works because grouping guarantees that all visit and purchase information for a member is aggregated together. The `LEFT JOIN` ensures no member is lost, including those without visits. The counts accurately compute the conversion rate because every purchase corresponds to exactly one visit. The `CASE` conditions are mutually exclusive and fully cover all possibilities, so every member receives exactly one valid category.

## Python Solution

Although this is a database problem, LeetCode database solutions are written in SQL. The following SQL query is the correct submission.

```python
SELECT
    m.member_id,
    m.name,
    CASE
        WHEN COUNT(v.visit_id) = 0 THEN 'Bronze'
        WHEN 100 * COUNT(p.visit_id) / COUNT(v.visit_id) >= 80 THEN 'Diamond'
        WHEN 100 * COUNT(p.visit_id) / COUNT(v.visit_id) >= 50 THEN 'Gold'
        ELSE 'Silver'
    END AS category
FROM Members m
LEFT JOIN Visits v
    ON m.member_id = v.member_id
LEFT JOIN Purchases p
    ON v.visit_id = p.visit_id
GROUP BY m.member_id, m.name;
```

The query begins with the `Members` table because every member must appear in the output. A `LEFT JOIN` to `Visits` ensures that members with no visits are still included.

The second `LEFT JOIN` connects visits to purchases. If a visit does not result in a purchase, the purchase columns become `NULL`.

The `GROUP BY` clause creates one aggregated row per member. Inside each group:

- `COUNT(v.visit_id)` computes total visits
- `COUNT(p.visit_id)` computes total purchases

The `CASE` statement applies the category rules in order. The `"Bronze"` condition must come first because members with zero visits would otherwise cause division by zero.

The multiplication by `100` converts the purchase ratio into a percentage before comparison.

## Edge Cases

### Members With No Visits

This is the most important edge case because it can cause division by zero errors. A naive implementation might attempt to compute:

$$\frac{0}{0}$$

which is undefined.

The solution avoids this by checking:

```
WHEN COUNT(v.visit_id) = 0 THEN 'Bronze'
```

before performing any division.

### Members With Visits But No Purchases

A member may have many visits but never buy anything. In this case, the purchase count becomes zero while the visit count remains positive.

For example:

$$\frac{100 \times 0}{5} = 0$$

The query correctly classifies these members as `"Silver"`.

### Boundary Conversion Rates

Values exactly equal to 50 or 80 are easy places to make mistakes if comparisons are written incorrectly.

For example:

- 50 must be `"Gold"`
- 80 must be `"Diamond"`

The implementation handles this correctly using `>=` comparisons:

```
>= 80
>= 50
```

This ensures boundary values belong to the higher category as required.

### Members Without Matching Purchase Rows

Some visits never appear in the `Purchases` table. If an inner join were used instead of a left join, those visits would disappear entirely, causing incorrect visit counts.

Using:

```
LEFT JOIN Purchases
```

preserves all visits while only counting purchases when they exist.
