## Solution

## Problem Understanding

The problem describes a robot moving on the boundary and interior edges of a rectangular grid. The grid has dimensions `width x height`, where valid coordinates range from `(0, 0)` in the bottom-left corner to `(width - 1, height - 1)` in the top-right corner.

The robot starts at position `(0, 0)` facing `"East"`. When instructed to move a certain number of steps, it repeatedly tries to move forward one cell at a time. If the next cell would go outside the grid boundaries, the robot does not move immediately. Instead, it rotates 90 degrees counterclockwise and retries the same step in the new direction.

The important detail is that turning does not consume a step. A step is only consumed when the robot successfully moves into a valid cell.

The class must support four operations:

- Initialization of the robot and grid.
- Moving the robot forward by `num` steps.
- Returning the robot's current position.
- Returning the robot's current direction.

The constraints are especially important:

- `width` and `height` are at most `100`.
- Each `step(num)` call can contain up to `10^5` steps.
- There can be up to `10^4` total method calls.

A naive simulation that processes every single movement individually could become too slow in the worst case. If every call performs `10^5` operations and there are `10^4` calls, the total work could reach `10^9` operations.

The key observation is that the robot's movement eventually repeats in a cycle. Once the robot reaches the outer perimeter, it continuously traverses the boundary in a fixed loop. This allows us to reduce large step counts using modular arithmetic.

There are several edge cases that can easily cause bugs:

- When the robot completes an exact full cycle around the perimeter.
- The special handling of direction after full rotations.
- Very thin grids such as `2 x N` or `N x 2`.
- Large step counts that require modulo reduction.
- Correctly updating direction when the robot reaches corners.

The most subtle case is when `num % perimeter == 0`. In that situation, the robot returns to the same position, but the direction may not remain unchanged. The problem's movement rules require careful handling of this scenario.

## Approaches

### Brute Force Simulation

The most straightforward solution is to simulate the robot exactly as described.

For every step:

1. Compute the next cell in the current direction.
2. Check whether that cell is inside the grid.
3. If it is outside, rotate counterclockwise.
4. Retry movement until a valid move is found.
5. Move into the valid cell.

This approach is conceptually simple and directly follows the problem statement, so correctness is easy to reason about.

However, the time complexity is too high. Each `step(num)` call may require processing up to `10^5` individual movements. Across many calls, this becomes inefficient.

### Optimal Observation, Perimeter Cycle

The robot eventually moves only along the outer perimeter of the rectangle. The total number of unique perimeter positions is:

```
2 * (width + height) - 4
```

This is the perimeter cycle length.

Instead of simulating every movement individually, we can:

1. Reduce the number of steps using modulo arithmetic.
2. Treat the robot's path as movement along a circular sequence.
3. Compute the final perimeter position directly.

The main challenge is preserving the correct direction, especially after exact perimeter cycles.

We can model the robot's motion as movement along a one-dimensional perimeter index. Each index corresponds to a unique `(x, y)` coordinate and direction.

This reduces each operation to constant time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(num) per call | O(1) | Simulates every movement individually |
| Optimal | O(1) per call | O(1) | Uses perimeter cycle and modular arithmetic |

## Algorithm Walkthrough

1. Compute the perimeter cycle length during initialization.

The robot continuously traverses the outer boundary. The number of cells on this cycle is:

```
perimeter = 2 * (width + height) - 4
```

This value is reused for every movement operation.
2. Represent the robot's state as a perimeter index.

Instead of tracking complicated movement transitions, we track a single integer representing the robot's position along the perimeter traversal order.

The traversal order is:

- Bottom edge, left to right
- Right edge, bottom to top
- Top edge, right to left
- Left edge, top to bottom
3. Reduce large step counts using modulo arithmetic.

Since the movement repeats every full perimeter cycle, moving `num` steps is equivalent to moving:

```
num % perimeter
```

steps.
4. Handle exact full-cycle movement carefully.

If `num % perimeter == 0`, the robot still completes a full loop. This affects its direction.

For example, returning to `(0, 0)` after a complete cycle means the robot faces `"South"` rather than `"East"`.

To preserve this behavior, we convert zero remainder into one full perimeter traversal when `num > 0`.
5. Update the perimeter index.

Add the effective movement amount and wrap around using modulo arithmetic.
6. Convert the perimeter index back into coordinates and direction.

Depending on which segment of the perimeter the index lies in, we compute:

- `(x, y)` position
- current facing direction

Each edge corresponds to a fixed direction.

### Why it works

The robot always follows the same cyclic perimeter traversal. Every movement sequence repeats after exactly `perimeter` successful moves. By mapping the robot's motion to a circular index, we preserve the exact order of positions and turns that the original simulation would produce. Modular arithmetic guarantees that large movement counts produce the same final state as explicit simulation.

## Python Solution

```python
from typing import List

class Robot:

    def __init__(self, width: int, height: int):
        self.width = width
        self.height = height
        self.perimeter = 2 * (width + height) - 4

        # Position along perimeter traversal
        self.position = 0

    def step(self, num: int) -> None:
        move = num % self.perimeter

        # Exact cycle still changes direction
        if move == 0 and num > 0:
            move = self.perimeter

        self.position = (self.position + move) % self.perimeter

    def getPos(self) -> List[int]:
        pos = self.position
        w = self.width
        h = self.height

        # Bottom edge
        if pos < w:
            return [pos, 0]

        pos -= (w - 1)

        # Right edge
        if pos < h:
            return [w - 1, pos]

        pos -= (h - 1)

        # Top edge
        if pos < w:
            return [w - 1 - pos, h - 1]

        pos -= (w - 1)

        # Left edge
        return [0, h - 1 - pos]

    def getDir(self) -> str:
        pos = self.position
        w = self.width
        h = self.height

        if pos == 0:
            return "South"

        # Bottom edge
        if pos < w - 1:
            return "East"

        pos -= (w - 1)

        # Right edge
        if pos < h - 1:
            return "North"

        pos -= (h - 1)

        # Top edge
        if pos < w - 1:
            return "West"

        return "South"
```

The implementation stores only the robot's perimeter index rather than simulating individual coordinates and turns. This dramatically simplifies the logic.

During initialization, the perimeter length is precomputed because it never changes.

The `step()` method performs modulo reduction immediately. This ensures that extremely large movement counts remain efficient. The special case for exact perimeter cycles preserves the correct facing direction.

The `getPos()` method converts the perimeter index into Cartesian coordinates. The perimeter is divided into four segments corresponding to the four edges of the rectangle.

The `getDir()` method determines which edge the robot currently occupies. Each edge implies a fixed direction. The only special case is position `0`, where the robot faces `"South"` after completing a cycle.

## Go Solution

	height    int
	perimeter int
	position  int
}

	return Robot{
		width:     width,
		height:    height,
		perimeter: 2*(width+height) - 4,
		position:  0,
	}

func (this *Robot) Step(num int) {
	move := num % this.perimeter

	if move == 0 && num > 0 {
		move = this.perimeter
	}

	this.position = (this.position + move) % this.perimeter
}

	pos := this.position
	w := this.width
	h := this.height

	// Bottom edge
	if pos < w {
		return []int{pos, 0}
	}

	pos -= (w - 1)

	// Right edge
	if pos < h {
		return []int{w - 1, pos}
	}

	pos -= (h - 1)

	// Top edge
	if pos < w {
		return []int{w - 1 - pos, h - 1}
	}

	pos -= (w - 1)

	// Left edge
	return []int{0, h - 1 - pos}
}

	h := this.height

	if pos == 0 {
		return "South"

		return "East"

## Edge Cases

### Exact Full Perimeter Traversal

One of the trickiest cases occurs when the robot moves exactly a multiple of the perimeter length. A naive modulo operation would reduce the movement to zero and incorrectly preserve the old direction.

For example, after a full cycle, the robot returns to `(0, 0)` but faces `"South"` rather than `"East"`.

The implementation handles this by converting zero modulo results into one complete perimeter traversal whenever `num > 0`.

### Very Thin Grids

Grids such as `2 x N` or `N x 2` contain very short edges. These configurations are prone to off-by-one indexing bugs because some perimeter segments contain only one movable cell.

The implementation carefully subtracts segment lengths using `w - 1` and `h - 1`, ensuring every perimeter position maps correctly.

### Extremely Large Step Counts

The problem allows movement counts up to `10^5` per call, and many total calls. A naive simulation would repeatedly iterate through every step and become inefficient.

The optimized solution avoids this entirely by reducing movement using:

```
num % perimeter
```

This guarantees constant-time processing regardless of movement size.
