## Solution

## Problem Understanding

The problem gives us the head of a singly linked list and asks us to delete the middle node. The middle node is defined using 0-based indexing, meaning that for a list of length `n`, the node at index `floor(n / 2)` must be removed.

For example, if the list contains 7 nodes, the middle index is `7 // 2 = 3`, so the fourth node is removed. If the list contains 4 nodes, the middle index is `4 // 2 = 2`, so the third node is removed.

The input is a standard singly linked list, where each node contains a value and a pointer to the next node. The output should be the head of the modified linked list after deleting the required node.

An important detail is that the linked list is singly linked, meaning we can only move forward through the list. We cannot directly access previous nodes unless we explicitly track them.

The constraints state that the number of nodes can be as large as `10^5`. This tells us that the solution should ideally run in linear time, because anything quadratic could become too slow for large inputs.

There are several edge cases that are especially important:

- A list with only one node. Since that single node is also the middle node, deleting it should return an empty list, represented by `None`.
- A list with two nodes. The second node is considered the middle because `2 // 2 = 1`.
- Even-length lists, where the "right-middle" node is removed because of the floor division definition.
- Very large lists, where efficiency matters.

A naive implementation could easily fail if it does not properly handle the single-node case or if it loses track of the node before the middle node.

## Approaches

### Brute Force Approach

A straightforward way to solve this problem is to first compute the length of the linked list. Once we know the length `n`, we can determine the middle index as `n // 2`.

After finding the middle index, we traverse the linked list again until we reach the node immediately before the middle node. Then we adjust its `next` pointer to skip the middle node.

This approach is correct because the first traversal accurately determines the list size, and the second traversal precisely reaches the node before the target node.

Although this method works efficiently enough for the constraints, it requires two full passes through the linked list.

### Optimal Approach

The better approach uses the classic fast and slow pointer technique.

The key insight is that if one pointer moves twice as fast as another pointer, then when the fast pointer reaches the end of the list, the slow pointer will be positioned at the middle.

However, because we need to delete the middle node, we also need access to the node before the slow pointer. We therefore maintain three pointers:

- `slow`, which moves one step at a time
- `fast`, which moves two steps at a time
- `prev`, which tracks the node before `slow`

When the traversal finishes:

- `slow` points to the middle node
- `prev` points to the node before the middle

We can then remove the middle node by setting:

```
prev.next = slow.next
```

This solution only requires one traversal of the linked list.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Two passes through the linked list |
| Optimal | O(n) | O(1) | Single-pass fast and slow pointer solution |

## Algorithm Walkthrough

1. First, handle the special case where the linked list contains only one node. Since that node is the middle node, deleting it means returning `None`.
2. Initialize three pointers:

- `slow = head`
- `fast = head`
- `prev = None`

The `slow` pointer will eventually reach the middle node. The `fast` pointer helps determine when the middle has been reached. The `prev` pointer remembers the node before `slow`.
3. Traverse the linked list while `fast` and `fast.next` are not `None`.
4. Inside each iteration:

- Move `prev` to `slow`
- Move `slow` forward by one node
- Move `fast` forward by two nodes

Since `fast` moves twice as quickly, `slow` progresses toward the middle at half the speed.
5. When the loop ends:

- `slow` points to the middle node
- `prev` points to the node before the middle
6. Delete the middle node by changing:

```
prev.next = slow.next
```

This skips the middle node and reconnects the list.
7. Return `head`, which still points to the start of the modified linked list.

### Why it works

The algorithm relies on the invariant that `fast` moves twice as quickly as `slow`. Every time `fast` advances two nodes, `slow` advances one node. Therefore, when `fast` reaches the end of the linked list, `slow` must be exactly halfway through the list. Because `prev` always trails one node behind `slow`, it correctly identifies the node before the middle, allowing the middle node to be removed safely.

## Python Solution

```python
# Definition for singly-linked list.
# class ListNode:
#     def __init__(self, val=0, next=None):
#         self.val = val
#         self.next = next

from typing import Optional

class Solution:
    def deleteMiddle(self, head: Optional[ListNode]) -> Optional[ListNode]:
        # Edge case: single node list
        if head is None or head.next is None:
            return None

        slow = head
        fast = head
        prev = None

        # Find the middle node
        while fast and fast.next:
            prev = slow
            slow = slow.next
            fast = fast.next.next

        # Delete the middle node
        prev.next = slow.next

        return head
```

The implementation begins by handling the smallest possible linked list. If the list contains only one node, removing the middle node leaves an empty list.

Next, three pointers are initialized. The `slow` pointer advances one node per iteration, while the `fast` pointer advances two nodes per iteration. The `prev` pointer trails behind `slow` so that we can later reconnect the list after removing the middle node.

The loop continues until the `fast` pointer reaches the end of the linked list. At that moment, the `slow` pointer is positioned at the middle node.

Finally, the node is removed by updating `prev.next` to bypass `slow`. Since singly linked lists only support forward traversal, this pointer reassignment is the standard way to delete a node.

## Go Solution

```go
/**
 * Definition for singly-linked list.
 * type ListNode struct {
 *     Val int
 *     Next *ListNode
 * }
 */
func deleteMiddle(head *ListNode) *ListNode {
    // Edge case: single node list
    if head == nil || head.Next == nil {
        return nil
    }

    slow := head
    fast := head
    var prev *ListNode = nil

    // Find the middle node
    for fast != nil && fast.Next != nil {
        prev = slow
        slow = slow.Next
        fast = fast.Next.Next
    }

    // Delete the middle node
    prev.Next = slow.Next

    return head
}
```

The Go implementation follows the same logic as the Python version. The primary difference is the explicit use of pointers and `nil` handling.

Go does not support Python-style optional typing, so linked list nodes are represented using pointers. The `prev` pointer is initialized as `nil`, and the traversal loop updates it before advancing `slow`.

Since the linked list size is at most `10^5`, integer overflow is not a concern in this problem.

## Edge Cases

### Single Node List

A linked list containing only one node is the most important edge case. Since that single node is also the middle node, deleting it should result in an empty list. A common bug is forgetting to handle this case before running the fast and slow pointer loop, which can lead to dereferencing `None` or `nil`. The implementation explicitly checks whether `head.next` is `None` and immediately returns `None`.

### Two Node List

For a list with two nodes, the middle index is `2 // 2 = 1`, meaning the second node must be removed. Some incorrect implementations mistakenly remove the first node because they assume the left-middle should be deleted in even-sized lists. The fast and slow pointer technique used here correctly positions `slow` on the second node.

### Even-Length Lists

Even-length lists are tricky because there are technically two central nodes. The problem specifically defines the middle using floor division, which selects the right-middle node. The pointer movement strategy naturally satisfies this requirement because the slow pointer advances after each fast pointer movement. By the time the fast pointer reaches the end, the slow pointer lands on the correct node to delete.

### Large Linked Lists

The input size can reach `10^5` nodes, so inefficient approaches could become problematic. Recursive solutions may risk stack overflow, while solutions that repeatedly traverse the list could become unnecessarily expensive. The iterative fast and slow pointer approach processes the list in linear time with constant extra space, making it suitable for large inputs.
