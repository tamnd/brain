## Solution

## Problem Understanding

This problem gives us an undirected graph representing a maze. Each room is a node, and each corridor is an undirected edge between two rooms. The task is to count how many distinct cycles of length 3 exist in the graph.

A cycle of length 3 is simply a triangle. If room `a` connects to room `b`, room `b` connects to room `c`, and room `c` connects back to room `a`, then those three rooms form one valid cycle.

The graph is undirected, which means every corridor works in both directions. Therefore, the cycle:

```
1 → 2 → 3 → 1
```

is identical to:

```
2 → 3 → 1 → 2
```

and also identical to:

```
3 → 1 → 2 → 3
```

All of these represent the same set of three rooms, so they should only be counted once.

The input consists of:

- `n`, the number of rooms
- `corridors`, a list of undirected edges

The output should be the number of distinct triangles in the graph.

The constraints are important:

- `n <= 1000`
- `corridors.length <= 50000`

A naive cubic solution that checks every triple of rooms would potentially require:

```
1000^3 = 1,000,000,000
```

operations, which is too slow.

The graph is also guaranteed to have:

- no self loops
- no duplicate edges

These guarantees simplify the implementation because we do not need to filter repeated corridors or handle edges like `[1,1]`.

Several edge cases are important:

- A graph with no cycles at all
- A graph with disconnected components
- Multiple triangles sharing edges
- Dense graphs where many triangles exist
- Sparse graphs where adjacency checks must still remain efficient

The key challenge is counting each triangle exactly once while remaining efficient enough for the input size.

## Approaches

### Brute Force Approach

The most direct approach is to try every possible combination of three rooms.

For every triple `(a, b, c)`:

1. Check whether edge `(a,b)` exists
2. Check whether edge `(b,c)` exists
3. Check whether edge `(c,a)` exists

If all three edges exist, then the three rooms form a triangle.

This approach is correct because every possible set of three rooms is examined exactly once. If those three rooms are mutually connected in a cycle, we count them.

However, this solution is too slow.

There are `O(n^3)` triples of rooms. With `n = 1000`, this becomes roughly one billion combinations before even considering adjacency checks.

Even with constant time edge lookups using hash sets, the runtime is not practical.

### Optimal Approach

The key observation is that a triangle exists whenever two connected rooms share a common neighbor.

Suppose we process an edge `(u,v)`.

If some room `w` is connected to both `u` and `v`, then:

```
u ↔ v
u ↔ w
v ↔ w
```

forms a triangle.

So instead of checking every triple of nodes, we can:

1. Build adjacency sets for all nodes
2. For each edge `(u,v)`
3. Count how many neighbors are common to both `u` and `v`

Every shared neighbor corresponds to one triangle involving edge `(u,v)`.

However, each triangle is discovered three times:

- once from each of its three edges

Therefore, after counting all shared neighbors across all edges, we divide the total by 3.

Using adjacency sets allows efficient membership checks.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n³) | O(E) | Checks every triple of rooms |
| Optimal | O(E × min(deg(u), deg(v))) | O(E) | Counts shared neighbors for each edge |

## Algorithm Walkthrough

1. Create an adjacency list using hash sets.

Since the graph is undirected, every corridor `[u,v]` is added in both directions:

```
u -> v
v -> u
```

Hash sets are chosen because they provide average `O(1)` membership checks.
2. Initialize a variable `triangle_count = 0`.

This variable stores the total number of triangle discoveries before correction.
3. Iterate through every corridor `(u,v)`.

Each edge may participate in one or more triangles.
4. For efficiency, iterate through the smaller adjacency set.

Suppose:

```
degree(u) < degree(v)
```

Then we iterate through neighbors of `u` and check whether each neighbor also exists in `v`'s adjacency set.

This optimization reduces unnecessary work in dense graphs.
5. For every shared neighbor `w`, increment the count.

If `w` connects to both `u` and `v`, then:

```
u-v-w
```

forms a triangle.
6. After processing all edges, divide the result by 3.

Each triangle is counted once per edge, and every triangle has exactly 3 edges.
7. Return the final result.

### Why it works

A triangle exists if and only if three nodes are pairwise connected. For any edge `(u,v)`, every common neighbor `w` completes exactly one triangle involving that edge.

Because every triangle has exactly three edges, the algorithm counts each triangle exactly three times. Dividing by 3 therefore produces the correct number of distinct cycles of length 3.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Solution:
    def numberOfPaths(self, n: int, corridors: List[List[int]]) -> int:
        adjacency = defaultdict(set)

        for u, v in corridors:
            adjacency[u].add(v)
            adjacency[v].add(u)

        triangle_count = 0

        for u, v in corridors:
            if len(adjacency[u]) > len(adjacency[v]):
                u, v = v, u

            for neighbor in adjacency[u]:
                if neighbor in adjacency[v]:
                    triangle_count += 1

        return triangle_count // 3
```

The implementation begins by constructing an adjacency structure using `defaultdict(set)`. Each node maps to a set containing all directly connected neighbors.

The algorithm then iterates through every corridor. For each edge, it searches for shared neighbors between the two endpoints.

To improve performance, the smaller adjacency set is always iterated first. This reduces the number of membership checks needed.

Whenever a node appears in both adjacency sets, a triangle is detected and counted.

Finally, the total is divided by 3 because each triangle is discovered once from each of its three edges.

## Go Solution

```go
func numberOfPaths(n int, corridors [][]int) int {
	adjacency := make([]map[int]bool, n+1)

	for i := 0; i <= n; i++ {
		adjacency[i] = make(map[int]bool)
	}

	for _, edge := range corridors {
		u := edge[0]
		v := edge[1]

		adjacency[u][v] = true
		adjacency[v][u] = true
	}

	triangleCount := 0

	for _, edge := range corridors {
		u := edge[0]
		v := edge[1]

		if len(adjacency[u]) > len(adjacency[v]) {
			u, v = v, u
		}

		for neighbor := range adjacency[u] {
			if adjacency[v][neighbor] {
				triangleCount++
			}
		}
	}

	return triangleCount / 3
}
```

The Go implementation mirrors the Python logic closely. Since Go does not have a built in hash set type, `map[int]bool` is used to represent adjacency sets.

The adjacency structure is initialized as a slice of maps indexed by room number.

Integer overflow is not a concern because the maximum number of triangles fits comfortably within Go's standard integer range for the given constraints.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(E × min(deg(u), deg(v))) | For each edge, iterate through the smaller adjacency set |
| Space | O(E) | Adjacency sets store all undirected edges |

The graph contains `E` corridors. For each corridor `(u,v)`, the algorithm iterates through the smaller degree neighborhood and performs constant time hash lookups in the larger neighborhood.

In sparse graphs, this performs very efficiently. Even in relatively dense graphs, the optimization of iterating through the smaller adjacency set keeps the runtime manageable for the problem constraints.

The space complexity is linear in the number of edges because every undirected edge is stored twice in the adjacency structure.

## Edge Cases

One important edge case is a graph with no corridors at all. In this situation, the adjacency sets remain empty, and the algorithm never finds any shared neighbors. The implementation correctly returns `0` because no triangles can exist without edges.

Another important case is a graph containing cycles longer than 3. For example:

```
1 → 2 → 3 → 4 → 1
```

This is a valid cycle, but it is not a triangle. A naive implementation that only checks connectivity loosely could accidentally count it. This solution avoids that problem because it only counts situations where two endpoints of an edge share a direct common neighbor.

Dense graphs are another potential source of performance problems. In a nearly complete graph, many nodes have large adjacency sets. Iterating through both neighbor lists for every edge could become expensive. The implementation handles this efficiently by always iterating through the smaller adjacency set, reducing unnecessary checks.

A final subtle edge case involves duplicate counting. Every triangle is reachable from all three of its edges, so failing to correct for overcounting would produce incorrect answers. The implementation guarantees correctness by dividing the total count by exactly 3 at the end.
