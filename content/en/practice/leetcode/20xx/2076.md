**Difficulty:** 🔴 Hard  
**Topics:** Union-Find, Graph Theory

## Solution

## Problem Understanding

This problem models a dynamic friendship network with restrictions on which groups of people are allowed to become connected.

We are given `n` people labeled from `0` to `n - 1`. Initially, nobody is connected to anyone else. Over time, friendship requests arrive one by one. When a request between two people is accepted, those two people become directly connected, and more importantly, their entire friendship groups become merged.

The important detail is that friendships are transitive. If `0` is friends with `2`, and `2` is friends with `5`, then `0` and `5` are considered indirectly connected. This means each accepted request potentially merges two entire connected components.

We are also given a list of restrictions. Each restriction `[x, y]` means that `x` and `y` are never allowed to end up in the same connected component. They cannot become directly or indirectly connected through chains of friendships.

For every request `[u, v]`, we must determine whether merging the groups containing `u` and `v` would violate any restriction. If it would, we reject the request. Otherwise, we accept it and permanently merge their friendship groups.

The output is a boolean array where:

- `true` means the request was accepted
- `false` means the request was rejected

The requests must be processed sequentially, because earlier accepted requests affect the structure used for later requests.

The constraints are relatively small:

- `n <= 1000`
- `restrictions.length <= 1000`
- `requests.length <= 1000`

These limits suggest that an `O(R * Q)` style solution is acceptable, where:

- `R` is the number of restrictions
- `Q` is the number of requests

However, repeatedly rebuilding graph connectivity from scratch would still be inefficient and unnecessarily complicated.

Several edge cases are important:

- Two people may already belong to the same connected component. In that case, the request is automatically valid because it changes nothing.
- Restrictions apply indirectly, not just directly.
- A request may look harmless locally but still violate a restriction after component merging.
- Restrictions never change, but connected components do.
- There may be no restrictions at all.
- There may be many repeated requests involving already connected users.

## Approaches

### Brute Force Approach

A straightforward solution is to explicitly maintain the friendship graph and, for every request, temporarily add the edge and check whether any restriction pair becomes connected.

One possible implementation is:

1. Add the requested friendship edge temporarily.
2. For every restriction `[x, y]`, run DFS or BFS to determine whether `x` can now reach `y`.
3. If any restriction is violated, remove the temporary edge and reject the request.
4. Otherwise, keep the edge and accept the request.

This works because the graph traversal accurately checks whether forbidden users became connected.

However, this approach is inefficient. Each request may require traversing a large portion of the graph for many restriction pairs. With up to 1000 requests and 1000 restrictions, repeated graph searches become expensive.

A better approach is to track connected components efficiently using Union-Find.

### Optimal Approach, Union-Find

The key observation is that we only care about connected components, not the exact graph structure.

Union-Find, also called Disjoint Set Union (DSU), is ideal for dynamically merging groups and checking whether two people already belong to the same component.

For each request `[u, v]`:

- Find the roots of `u` and `v`
- If they are already in the same component, the request is automatically valid
- Otherwise, before merging them, check every restriction `[x, y]`
- If merging the two components would place `x` and `y` into the same component, reject the request
- Otherwise, safely union the two components

The important insight is that restrictions only matter at the component level. We do not need to inspect individual friendship paths.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(Q * R * (V + E)) | O(V + E) | Repeated graph traversal for every restriction |
| Optimal | O(Q * R * α(N)) | O(N) | Uses Union-Find to track connected components efficiently |

Here:

- `Q` = number of requests
- `R` = number of restrictions
- `α(N)` = inverse Ackermann function, effectively constant

## Algorithm Walkthrough

### Step 1, Initialize Union-Find

Create two arrays:

- `parent[i]`, the representative parent of node `i`
- `rank[i]` or `size[i]`, used for efficient union

Initially, every person is in their own component.

### Step 2, Process Requests Sequentially

Requests must be handled in order because accepted friendships permanently affect future connectivity.

For each request `[u, v]`:

- Find the root of `u`
- Find the root of `v`

These roots represent the current friendship groups.

### Step 3, Handle Already Connected Users

If both users already belong to the same component, the request is automatically successful.

This is because accepting the request changes nothing in the graph structure.

### Step 4, Check Restrictions Before Merging

Before unioning the two components, inspect every restriction `[x, y]`.

For each restriction:

- Find the current roots of `x` and `y`
- Suppose:

- `rootU` is the root of `u`
- `rootV` is the root of `v`

The merge would violate the restriction if:

- `x` belongs to `rootU` and `y` belongs to `rootV`, or
- `x` belongs to `rootV` and `y` belongs to `rootU`

Because after the union, those two restricted users would become connected.

If any restriction is violated:

- Reject the request
- Do not perform the union

### Step 5, Merge Components

If no restriction blocks the merge:

- Union the two components
- Record `true` in the result array

### Step 6, Continue Until All Requests Are Processed

Repeat the same logic for every request.

### Why it works

The Union-Find structure always maintains the exact connected components formed by accepted requests.

Before accepting a new request, we verify that no restriction pair would end up inside the same merged component. Since restrictions only depend on connectivity, not on specific paths, checking component representatives is sufficient.

Thus, every accepted request preserves all restrictions, and every rejected request correctly identifies a forbidden merge.

## Python Solution

```python
from typing import List

class Solution:
    def friendRequests(
        self,
        n: int,
        restrictions: List[List[int]],
        requests: List[List[int]]
    ) -> List[bool]:

        parent = list(range(n))
        rank = [0] * n

        def find(x: int) -> int:
            if parent[x] != x:
                parent[x] = find(parent[x])
            return parent[x]

        def union(x: int, y: int) -> None:
            root_x = find(x)
            root_y = find(y)

            if root_x == root_y:
                return

            if rank[root_x] < rank[root_y]:
                parent[root_x] = root_y
            elif rank[root_x] > rank[root_y]:
                parent[root_y] = root_x
            else:
                parent[root_y] = root_x
                rank[root_x] += 1

        result = []

        for u, v in requests:
            root_u = find(u)
            root_v = find(v)

            # Already connected
            if root_u == root_v:
                result.append(True)
                continue

            valid = True

            for x, y in restrictions:
                root_x = find(x)
                root_y = find(y)

                if (
                    (root_x == root_u and root_y == root_v)
                    or
                    (root_x == root_v and root_y == root_u)
                ):
                    valid = False
                    break

            if valid:
                union(root_u, root_v)
                result.append(True)
            else:
                result.append(False)

        return result
```

The implementation begins by initializing the Union-Find structure. Each user starts as their own parent, meaning everyone is initially isolated.

The `find` function uses path compression. Whenever we search for a component representative, we flatten the structure to make future queries faster.

The `union` function uses union by rank. This keeps the trees shallow and ensures nearly constant time operations.

For every request, the algorithm first checks whether the two users are already connected. If they are, the request succeeds immediately.

Otherwise, the code iterates through all restrictions and checks whether the pending merge would place a restricted pair into the same connected component.

If no restriction is violated, the two components are merged using Union-Find. If a restriction would be broken, the request is rejected and the structure remains unchanged.

## Go Solution

```go
package main

func friendRequests(n int, restrictions [][]int, requests [][]int) []bool {
	parent := make([]int, n)
	rank := make([]int, n)

	for i := 0; i < n; i++ {
		parent[i] = i
	}

	var find func(int) int

	find = func(x int) int {
		if parent[x] != x {
			parent[x] = find(parent[x])
		}
		return parent[x]
	}

	union := func(x int, y int) {
		rootX := find(x)
		rootY := find(y)

		if rootX == rootY {
			return
		}

		if rank[rootX] < rank[rootY] {
			parent[rootX] = rootY
		} else if rank[rootX] > rank[rootY] {
			parent[rootY] = rootX
		} else {
			parent[rootY] = rootX
			rank[rootX]++
		}
	}

	result := make([]bool, 0, len(requests))

	for _, request := range requests {
		u := request[0]
		v := request[1]

		rootU := find(u)
		rootV := find(v)

		// Already connected
		if rootU == rootV {
			result = append(result, true)
			continue
		}

		valid := true

		for _, restriction := range restrictions {
			x := restriction[0]
			y := restriction[1]

			rootX := find(x)
			rootY := find(y)

			if (rootX == rootU && rootY == rootV) ||
				(rootX == rootV && rootY == rootU) {
				valid = false
				break
			}
		}

		if valid {
			union(rootU, rootV)
			result = append(result, true)
		} else {
			result = append(result, false)
		}
	}

	return result
}
```

The Go implementation follows the same logic as the Python version. Slices are used for the `parent`, `rank`, and result arrays.

Go does not support nested named functions as cleanly as Python, so the `find` function is declared as a variable holding a recursive closure.

The solution uses path compression and union by rank exactly as in Python. Since all values are small integers within problem constraints, integer overflow is not a concern.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(Q * R * α(N)) | Each request scans all restrictions and performs near-constant DSU operations |
| Space | O(N) | Parent and rank arrays store DSU state |

The inverse Ackermann function `α(N)` grows so slowly that it is effectively constant for all practical input sizes.

For each request, we potentially examine every restriction once. Each restriction check performs several `find` operations, which are amortized nearly constant time because of path compression and union by rank.

## Edge Cases

### Already Connected Users

A request may involve two users who are already in the same connected component. This can happen because of previous accepted requests.

A buggy implementation might unnecessarily re-check restrictions or even reject the request. However, accepting such a request changes nothing structurally, so it must always succeed.

The implementation handles this by immediately returning `true` whenever the two users already share the same root.

### Restrictions Through Indirect Connections

The restriction does not only prohibit direct friendships. It also prohibits indirect connectivity through chains of friendships.

For example:

```
0 restricted with 3
0 connected to 1
1 connected to 2
2 connected to 3
```

Even though `0` and `3` were never directly connected, the final request must be rejected because they become indirectly connected.

The DSU structure naturally models connected components, so indirect connectivity is handled automatically.

### Rejected Requests Must Not Modify State

One subtle bug is accidentally performing a partial union before finishing all restriction checks.

If the structure changes after a rejected request, future requests will operate on corrupted connectivity information.

This implementation first checks all restrictions completely. Only if the merge is proven safe does it perform the union operation. This guarantees rejected requests leave the DSU unchanged.
