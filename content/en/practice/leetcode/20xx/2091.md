## Solution

## Problem Understanding

The problem gives us a 0-indexed array of distinct integers. Among all the numbers in the array, one value is the minimum element and one value is the maximum element. Our task is to remove both of these elements using the fewest deletions possible.

A deletion operation is very restricted. We are not allowed to remove an element from the middle of the array. We may only remove an element from either:

- the front of the array
- the back of the array

Every deletion shortens the array, which changes the positions of the remaining elements.

The goal is to compute the minimum number of deletions required so that both the minimum and maximum elements are removed.

The array contains distinct integers, which guarantees that:

- there is exactly one minimum element
- there is exactly one maximum element

The constraints allow up to `10^5` elements, which means the solution must be efficient. An `O(n^2)` approach would likely be too slow, while an `O(n)` solution is ideal.

Several important edge cases appear immediately:

- The array may contain only one element. In that case, the single element is both the minimum and maximum, so the answer is `1`.
- The minimum and maximum may both be near the front.
- The minimum and maximum may both be near the back.
- One may be near the front while the other is near the back.
- The minimum index may be larger than the maximum index, so we must normalize their order before computing deletion counts.

The key challenge is determining the best combination of front deletions and back deletions.

## Approaches

### Brute Force Approach

A brute-force solution would try every possible combination of deletions from the front and back.

Suppose we remove `x` elements from the front and `y` elements from the back. We could simulate whether both the minimum and maximum elements are removed after those deletions. Among all valid combinations, we would choose the smallest total value of `x + y`.

Since there are `n` possible choices for front deletions and `n` possible choices for back deletions, this produces `O(n^2)` combinations. Each combination may also require checking whether the minimum and maximum were removed.

Although this approach is conceptually simple, it is too slow for arrays of size `10^5`.

### Optimal Greedy Observation

The important observation is that we only care about the positions of two elements:

- the minimum element
- the maximum element

Once we know their indices, there are only three meaningful strategies:

1. Remove both from the front.
2. Remove both from the back.
3. Remove one from the front and the other from the back.

There is no benefit to any other pattern because deletions only affect the ends of the array.

Suppose:

- `left` is the smaller index
- `right` is the larger index

Then:

- Removing both from the front requires `right + 1` deletions.
- Removing both from the back requires `n - left` deletions.
- Removing one from each side requires:

- `(left + 1)` deletions from the front
- `(n - right)` deletions from the back

We simply compute all three possibilities and take the minimum.

This gives a clean `O(n)` solution.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Tries every possible front/back deletion combination |
| Optimal | O(n) | O(1) | Uses indices of min and max to evaluate only three cases |

## Algorithm Walkthrough

1. Traverse the array once to locate the indices of the minimum and maximum elements.

We can use Python's `min()` and `max()` or track them manually during traversal. The important result is knowing where these two elements are located.
2. Store the smaller index in `left` and the larger index in `right`.

This normalization simplifies the formulas later because we always know:

- `left <= right`
3. Compute the cost of removing both elements from the front.

If we delete from the front until reaching the farther element, both elements will be removed automatically.

The number of deletions is:

```
right + 1
```
4. Compute the cost of removing both elements from the back.

If we delete from the back until reaching the earlier element, both elements disappear.

The number of deletions is:

```
n - left
```
5. Compute the mixed strategy.

Remove the earlier element from the front and the later element from the back.

This requires:

```
(left + 1) + (n - right)
```
6. Return the minimum among the three computed values.

### Why it works

Any valid sequence of deletions must eventually remove both the minimum and maximum elements. Since deletions are only allowed from the two ends, every optimal strategy must fall into one of three categories:

- both removed from the front
- both removed from the back
- one removed from each side

There are no other meaningful possibilities. By explicitly computing the cost of all three cases and taking the minimum, we guarantee the optimal answer.

## Python Solution

```python
from typing import List

class Solution:
    def minimumDeletions(self, nums: List[int]) -> int:
        n = len(nums)

        min_index = nums.index(min(nums))
        max_index = nums.index(max(nums))

        left = min(min_index, max_index)
        right = max(min_index, max_index)

        remove_from_front = right + 1
        remove_from_back = n - left
        remove_from_both = (left + 1) + (n - right)

        return min(
            remove_from_front,
            remove_from_back,
            remove_from_both
        )
```

The implementation begins by finding the indices of the minimum and maximum elements. Because all values are distinct, each index is unique.

Next, we normalize the indices by assigning the smaller one to `left` and the larger one to `right`. This simplifies the later calculations.

The code then evaluates the three valid deletion strategies:

- deleting only from the front
- deleting only from the back
- deleting from both ends

Finally, the minimum value among those strategies is returned.

The solution uses constant extra space and scans the array only a few times, making it efficient enough for the largest constraint values.

## Go Solution

```go
package main

func minimumDeletions(nums []int) int {
	n := len(nums)

	minIndex := 0
	maxIndex := 0

	for i := 1; i < n; i++ {
		if nums[i] < nums[minIndex] {
			minIndex = i
		}

		if nums[i] > nums[maxIndex] {
			maxIndex = i
		}
	}

	left := min(minIndex, maxIndex)
	right := max(minIndex, maxIndex)

	removeFromFront := right + 1
	removeFromBack := n - left
	removeFromBoth := (left + 1) + (n - right)

	return min(
		removeFromFront,
		min(removeFromBack, removeFromBoth),
	)
}

func min(a, b int) int {
	if a < b {
		return a
	}
	return b
}

func max(a, b int) int {
	if a > b {
		return a
	}
	return b
}
```

The Go version follows the same logic as the Python implementation. Since Go does not provide built-in `min()` and `max()` helpers for integers, helper functions are defined manually.

Instead of using methods like Python's `index()`, the Go implementation performs a single traversal to locate both indices efficiently.

There are no integer overflow concerns because all calculations stay within the range of standard Go integers.

## Edge Cases

### Single Element Array

If the array contains only one element, that element is simultaneously the minimum and maximum. A naive implementation might incorrectly count two removals because it treats the minimum and maximum independently. The implementation handles this naturally because both indices become `0`, and the minimum among the computed strategies is correctly `1`.

### Minimum and Maximum at Opposite Ends

Consider:

```
[1, 2, 3, 4, 5]
```

The minimum is already at the front, and the maximum is already at the back. A naive strategy that removes from only one side would require removing the entire array. The optimal strategy removes one element from the front and one from the back for a total of `2` deletions. The mixed strategy calculation handles this correctly.

### Minimum and Maximum Adjacent Near One Side

Consider:

```
[0, -4, 19, 1, 8]
```

The minimum and maximum are adjacent near the front. The optimal strategy is deleting only from the front. Some incorrect implementations overcomplicate the logic and fail to recognize this case efficiently. Our implementation explicitly evaluates the "front only" strategy, guaranteeing the correct result.

### Minimum Index Greater Than Maximum Index

Consider:

```
[2, 9, 5, 1]
```

The maximum appears before the minimum. If we directly apply formulas without normalizing the indices, the calculations become error-prone. The implementation solves this by assigning:

```
left = min(min_index, max_index)
right = max(min_index, max_index)
```

This guarantees consistent and correct formulas for all cases.
