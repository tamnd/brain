[LeetCode Problem 2059](https://leetcode.com/problems/minimum-operations-to-convert-number/)

**Difficulty:** 🟡 Medium  
**Topics:** Array, Breadth-First Search

## Solution

## Problem Understanding

The problem asks us to transform a starting number `start` into a target number `goal` by repeatedly applying operations using elements from the array `nums`. The operations allowed are addition, subtraction, and bitwise XOR. Importantly, we can apply these operations any number of times, and each element of `nums` can be used multiple times in any order. The transformation may take `x` outside the range `[0,1000]`, which terminates further operations.

The inputs are:

- `nums`, an array of distinct integers of length up to 1000.
- `start`, an integer in the range `[0,1000]`.
- `goal`, an integer that may be large, positive or negative.

The output is the **minimum number of operations** needed to reach `goal` from `start`, or `-1` if it is impossible.

Key constraints:

- `nums` elements and `goal` can be very large (up to ±10^9), but `start` is small (0-1000).
- We must consider numbers outside the `[0,1000]` range because they can directly produce `goal`.
- The array elements are distinct, so we do not have repeated operations caused by duplicates.

Important edge cases:

- `goal` can be negative or far outside `[0,1000]`.
- Some operations may immediately take `x` out of range.
- A solution may require mixing additions, subtractions, and XORs.

This problem naturally models a **graph traversal**, where each number `x` represents a node, and edges correspond to the allowed operations. Our task is to find the shortest path from `start` to `goal`.

## Approaches

The brute-force approach would be to explore all sequences of operations recursively. For each number, we try every `nums[i]` with every operation, counting steps until we hit `goal`. While correct in principle, this approach is **exponentially slow** due to repeated exploration of the same numbers.

The key insight is that we only need to explore numbers reachable from `start` **without revisiting the same number multiple times**. Numbers in `[0,1000]` form a manageable search space, and numbers outside can be treated as terminal nodes. Therefore, a **Breadth-First Search (BFS)** is optimal. BFS guarantees that the first time we reach `goal`, it is with the minimum number of operations. We maintain a visited set to avoid cycles and redundant exploration.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(3^n * ? ) | O(?) | Recursively explore all sequences; exponential and infeasible |
| BFS Optimal | O(1001 * n) | O(1001) | Each number in range [0,1000] is visited at most once; operations per number = 3*n |

## Algorithm Walkthrough

1. Initialize a queue for BFS and a visited set. Start by adding `start` to the queue with `0` operations.
2. While the queue is not empty, remove the front element `(current, steps)`.
3. If `current` equals `goal`, return `steps` because BFS guarantees minimum steps.
4. For each number `num` in `nums`, compute the results of `current + num`, `current - num`, and `current ^ num`.
5. If a result equals `goal`, return `steps + 1`.
6. If a result is in the range `[0,1000]` and not visited, add it to the queue and mark it as visited. If a result is outside `[0,1000]`, do not enqueue it, but if it equals `goal` we already handle it.
7. If BFS completes without finding `goal`, return `-1`.

**Why it works:** BFS explores numbers in order of increasing operation count. By marking visited numbers, we prevent revisiting the same node, ensuring correctness and efficiency. Because all operations are allowed repeatedly, BFS guarantees that the first time we reach `goal`, it is the minimum number of operations.

## Python Solution

```python
from collections import deque
from typing import List

class Solution:
    def minimumOperations(self, nums: List[int], start: int, goal: int) -> int:
        if start == goal:
            return 0
        
        visited = set()
        queue = deque([(start, 0)])
        visited.add(start)
        
        while queue:
            current, steps = queue.popleft()
            for num in nums:
                for next_val in (current + num, current - num, current ^ num):
                    if next_val == goal:
                        return steps + 1
                    if 0 <= next_val <= 1000 and next_val not in visited:
                        visited.add(next_val)
                        queue.append((next_val, steps + 1))
        return -1
```

**Implementation explanation:** We start BFS from `start`. For each number, we generate all possible next numbers using the three operations with each `nums[i]`. If a number equals `goal`, we return immediately. Numbers in `[0,1000]` are enqueued to continue exploration, while numbers outside this range are ignored for further operations. The visited set prevents cycles.

## Go Solution

```go
package main

func minimumOperations(nums []int, start int, goal int) int {
	if start == goal {
		return 0
	}

	type Node struct {
		val   int
		steps int
	}
	queue := []Node{{start, 0}}
	visited := make(map[int]bool)
	visited[start] = true

	for len(queue) > 0 {
		current := queue[0]
		queue = queue[1:]
		for _, num := range nums {
			nextVals := []int{current.val + num, current.val - num, current.val ^ num}
			for _, next := range nextVals {
				if next == goal {
					return current.steps + 1
				}
				if next >= 0 && next <= 1000 && !visited[next] {
					visited[next] = true
					queue = append(queue, Node{next, current.steps + 1})
				}
			}
		}
	}
	return -1
}
```

**Go-specific notes:** We use a slice for BFS and a map to track visited numbers. The `Node` struct stores the current number and step count. Slice append operations are equivalent to enqueue. Handling integer bounds explicitly ensures no overflow within `[0,1000]`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(1001 * n) | Each number in [0,1000] is visited at most once, and for each we try n numbers with 3 operations. |
| Space | O(1001) | Visited set and BFS queue store numbers in [0,1000]. |

Because `start` is small and the BFS space is limited to `[0,1000]`, the algorithm is efficient despite potentially large `goal` or `nums[i]`.
