## Solution

## Problem Understanding

This problem models course scheduling as a directed acyclic graph, also called a DAG. Each course is a node, and each prerequisite relationship is a directed edge from one course to another.

You are given:

- `n`, the number of courses
- `relations`, where `[a, b]` means course `a` must be completed before course `b`
- `time[i]`, the number of months required to finish course `i + 1`

The important detail is that multiple courses can be taken simultaneously as long as their prerequisites are already satisfied. This means we are not trying to find a sequential ordering of courses. Instead, we want to determine the earliest possible completion time for every course while allowing parallel execution whenever possible.

The final answer is the minimum total number of months needed to complete all courses.

A useful way to think about the problem is this:

- Each course has a duration.
- A course can only begin after all prerequisite courses are completed.
- Since courses can run in parallel, the start time of a course depends only on the slowest prerequisite chain leading into it.

The graph is guaranteed to be a DAG. This guarantee is critical because it means there are no cycles, so a valid ordering of courses always exists.

The constraints are large:

- Up to `5 * 10^4` courses
- Up to `5 * 10^4` prerequisite relations

These constraints immediately rule out expensive graph algorithms with quadratic or exponential complexity. We need a solution that processes the graph efficiently, ideally in linear time relative to the number of nodes and edges.

Several edge cases are important:

- Some courses may have no prerequisites and can start immediately.
- Multiple prerequisite chains may converge into one course.
- The graph may consist of several disconnected components.
- There may be no relations at all, meaning all courses can be taken simultaneously.
- A course may depend on a long chain of prerequisites, creating a critical path that determines the final answer.

The key challenge is computing the earliest completion time for every course while respecting dependency ordering.

## Approaches

### Brute Force Approach

A brute-force solution would attempt to simulate all possible valid schedules.

One idea is to repeatedly identify courses whose prerequisites are satisfied, try different combinations of parallel execution, and compute the resulting completion time. Since many valid execution orders exist, this becomes a combinatorial scheduling problem.

Another brute-force interpretation is to recursively compute the completion time for every course by exploring all prerequisite paths repeatedly. Without memoization, the same subproblems would be recalculated many times.

Although these approaches eventually produce the correct answer, they are far too slow for the given constraints. The number of valid schedules grows exponentially with the number of courses, and repeated traversal of prerequisite chains causes severe inefficiency.

### Optimal Approach, Topological Sort with Dynamic Programming

The key observation is that the graph is a DAG.

In a DAG, we can process nodes in topological order, meaning every prerequisite is processed before the course that depends on it.

For each course, we want to know:

- the earliest time we can finish that course

If a course has no prerequisites, its completion time is simply its own duration.

If a course depends on several prerequisites, it can only begin after all prerequisites are complete. Therefore:

- earliest completion time of current course =

maximum completion time among prerequisites +

current course duration

This becomes a classic dynamic programming problem on a DAG.

We use:

- an adjacency list to represent the graph
- an indegree array to track prerequisite counts
- a queue for Kahn's topological sort
- a DP array where `dp[i]` stores the earliest completion time for course `i`

By processing courses in topological order, we ensure prerequisite information is always available before computing dependent courses.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Explores many possible schedules or recomputes prerequisite chains repeatedly |
| Optimal | O(n + m) | O(n + m) | Uses topological sort and dynamic programming on a DAG |

Here, `m` is the number of relations.

## Algorithm Walkthrough

1. Build the graph representation.

Create an adjacency list where `graph[u]` contains all courses that depend on course `u`.

Also create an `indegree` array where `indegree[v]` stores how many prerequisites course `v` still has.
2. Initialize the DP array.

Create a `dp` array where `dp[i]` represents the earliest month when course `i` can be completed.

Initially, for courses with no prerequisites, the completion time is simply their own duration.
3. Initialize the queue.

Any course with indegree `0` has no prerequisites and can start immediately.

Add all such courses to the queue.
4. Process courses using topological sort.

While the queue is not empty:

- Remove a course from the queue.
- For every dependent course:

- Update its completion time using:

```
dp[next] = max(dp[next], dp[current] + time[next])
```
- Decrease its indegree.
- If indegree becomes `0`, add it to the queue.
5. Track the maximum completion time.

Since courses may belong to different disconnected components, the total answer is the maximum value in the DP array.

### Why it works

The algorithm works because topological ordering guarantees that every prerequisite is processed before the dependent course.

For every course, the earliest start time is determined by the prerequisite chain that finishes latest. Therefore, taking the maximum completion time among prerequisites correctly captures the dependency constraints.

The DP array maintains the invariant that once a course is processed, `dp[i]` equals the earliest possible completion time for that course. Since the graph is acyclic, every dependency chain is processed exactly once in valid order.

## Python Solution

```python
from collections import deque
from typing import List

class Solution:
    def minimumTime(self, n: int, relations: List[List[int]], time: List[int]) -> int:
        graph = [[] for _ in range(n)]
        indegree = [0] * n

        for prev_course, next_course in relations:
            prev_course -= 1
            next_course -= 1

            graph[prev_course].append(next_course)
            indegree[next_course] += 1

        dp = time[:]

        queue = deque()

        for course in range(n):
            if indegree[course] == 0:
                queue.append(course)

        while queue:
            current = queue.popleft()

            for neighbor in graph[current]:
                dp[neighbor] = max(
                    dp[neighbor],
                    dp[current] + time[neighbor]
                )

                indegree[neighbor] -= 1

                if indegree[neighbor] == 0:
                    queue.append(neighbor)

        return max(dp)
```

The implementation begins by constructing the adjacency list and indegree array. Since the input uses 1-based indexing but Python lists use 0-based indexing, each course number is converted by subtracting one.

The `dp` array is initialized as a copy of `time`. This is important because a course with no prerequisites finishes after exactly its own duration.

The queue stores all courses whose indegree is zero. These courses can start immediately because they have no unmet prerequisites.

During topological traversal, each processed course updates the earliest completion time of its dependent courses. The update uses `max` because a course may have multiple prerequisites, and it must wait for the slowest prerequisite chain.

Finally, the answer is the maximum value in `dp`, representing the latest finishing course among all components.

## Go Solution

```go
package main

import "container/list"

func minimumTime(n int, relations [][]int, time []int) int {
	graph := make([][]int, n)
	indegree := make([]int, n)

	for _, relation := range relations {
		prev := relation[0] - 1
		next := relation[1] - 1

		graph[prev] = append(graph[prev], next)
		indegree[next]++
	}

	dp := make([]int, n)
	copy(dp, time)

	queue := list.New()

	for i := 0; i < n; i++ {
		if indegree[i] == 0 {
			queue.PushBack(i)
		}
	}

	for queue.Len() > 0 {
		front := queue.Front()
		queue.Remove(front)

		current := front.Value.(int)

		for _, neighbor := range graph[current] {
			if dp[current]+time[neighbor] > dp[neighbor] {
				dp[neighbor] = dp[current] + time[neighbor]
			}

			indegree[neighbor]--

			if indegree[neighbor] == 0 {
				queue.PushBack(neighbor)
			}
		}
	}

	answer := 0

	for _, value := range dp {
		if value > answer {
			answer = value
		}
	}

	return answer
}
```

The Go implementation follows the same algorithmic structure as the Python version.

The queue is implemented using Go's `container/list` package because Go does not have a built-in deque structure.

Slices are used for the adjacency list and DP array. The `copy` function duplicates the `time` slice into `dp`, ensuring that initial completion times are correctly initialized.

Integer overflow is not a concern because the maximum possible completion time remains well within Go's `int` range under the problem constraints.

The topological traversal processes every node once and every edge once, resulting in linear complexity relative to the graph size.

## Edge Cases

### No Prerequisites

When `relations` is empty, every course can begin immediately. A common bug is accidentally summing all course durations instead of recognizing that all courses can run in parallel.

The implementation handles this correctly because every course starts with indegree `0`, so all are added to the queue immediately. The answer becomes the maximum course duration.

### Multiple Prerequisites

A course may depend on several prerequisite chains with different completion times. A buggy implementation might overwrite completion times instead of taking the maximum.

The algorithm correctly uses:

```
dp[neighbor] = max(dp[neighbor], dp[current] + time[neighbor])
```

This guarantees the course waits for the slowest prerequisite chain.

### Disconnected Graph Components

The graph may contain several independent groups of courses. A naive implementation might only process one connected component.

This implementation initializes the queue with every indegree `0` node across the entire graph. As a result, all components are processed correctly, and the final answer is the maximum completion time across all components.

### Long Dependency Chains

Some inputs may form a very deep chain of prerequisites. Recursive DFS solutions risk stack overflow in languages with limited recursion depth.

This implementation uses iterative topological sorting with a queue, so it safely handles large chains within the problem constraints.
