**Topics:** Breadth-First Search, Graph Theory, Shortest Path

## Solution

## Problem Understanding

This problem gives us an undirected connected graph with `n` nodes. Every edge takes the same amount of travel time, represented by `time`. However, the graph traversal is complicated by synchronized traffic signals at every node.

Each traffic signal alternates between green and red every `change` minutes:

- From time `0` to `change - 1`, signals are green.
- From time `change` to `2 * change - 1`, signals are red.
- Then the pattern repeats forever.

A traveler may enter a node at any moment, but can only leave a node while its signal is green. An important restriction is that if the signal is green, the traveler must leave immediately and cannot wait voluntarily.

The goal is not to find the shortest path in terms of edges. Instead, we must find the second minimum travel time from node `1` to node `n`.

The phrase "second minimum" means the smallest value strictly larger than the minimum. If multiple routes produce the same minimum travel time, they do not count as the second minimum.

The constraints are large:

- Up to `10^4` nodes
- Up to `2 * 10^4` edges

This immediately rules out exhaustive path enumeration because the number of possible paths can grow exponentially due to revisiting nodes.

A subtle detail is that revisiting nodes is allowed, including the start and destination nodes. This matters because the second minimum path may involve cycles.

Another important observation is that every edge has the same traversal time. That simplifies the shortest path structure significantly and allows a BFS-based solution rather than a full weighted shortest path algorithm.

Several edge cases are important:

- The second minimum path may revisit nodes.
- Multiple shortest paths can exist with identical arrival times.
- Waiting occurs automatically only when arriving during a red phase.
- The second minimum time may require intentionally taking a longer route even if a shorter edge-count path exists.
- Since all signals are synchronized, the waiting behavior depends only on the current global time.

## Approaches

### Brute Force Approach

A brute force solution would attempt to enumerate all possible paths from node `1` to node `n`, simulate the travel time for each path, and then sort the resulting times to find the second minimum.

For every path:

1. Start at node `1` at time `0`.
2. Traverse edges one by one.
3. Before leaving a node, check whether the signal is green or red.
4. If red, wait until the next green interval.
5. Continue until reaching node `n`.

This approach is correct because it evaluates every possible route and therefore cannot miss the second minimum time.

However, it is computationally infeasible. The graph allows revisiting nodes, which means the number of possible paths is effectively unbounded unless we artificially limit path length. Even with pruning, the search space grows exponentially.

With up to `10^4` nodes and `2 * 10^4` edges, brute force exploration is far too slow.

### Key Insight

The crucial observation is that all edges have identical travel time.

That means the only thing determining the order of possible arrival times is:

1. The number of edges traversed.
2. The waiting time imposed by traffic lights.

Instead of exploring arbitrary paths, we can compute the first and second shortest distances in terms of edge count using BFS.

Why does this work?

Because BFS naturally discovers paths in increasing order of edge count. Since all edge traversal costs are equal, the first time we reach a node gives the minimum number of edges, and the second distinct reach gives the second minimum number of edges.

After obtaining the second shortest edge-count path to node `n`, we simulate the actual travel time while accounting for traffic lights.

The important idea is that we do not need to track every possible arrival time. We only need the two smallest distinct distances for every node.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Explores all possible paths |
| Optimal | O(V + E) | O(V + E) | BFS stores only two shortest distances per node |

## Algorithm Walkthrough

### Step 1: Build the Graph

Construct an adjacency list from the edge list.

Since the graph is undirected, each edge `[u, v]` is added in both directions.

We use an adjacency list because it provides efficient traversal for BFS.

### Step 2: Store Two Best Distances Per Node

For every node, maintain:

- `dist[node][0]`, the shortest number of edges from node `1`
- `dist[node][1]`, the second shortest number of edges from node `1`

Initialize all distances to infinity.

Set:

```
dist[1][0] = 0
```

because we start at node `1` with zero edges traveled.

### Step 3: Run BFS

Use a queue storing:

```
(node, distance_in_edges)
```

For each neighboring node:

```
new_distance = current_distance + 1
```

Then compare it against the stored shortest and second shortest distances.

There are three cases:

1. `new_distance` is smaller than the shortest distance.

- Shift the old shortest into second shortest.
- Update shortest.
2. `new_distance` is larger than shortest but smaller than second shortest.

- Update second shortest.
3. Otherwise ignore it.

This guarantees that each node stores only its two smallest distinct edge counts.

### Step 4: Stop When Second Shortest to n Is Found

Once `dist[n][1]` is determined, we know the second minimum number of edges needed to reach node `n`.

Call this value:

```
steps
```

### Step 5: Simulate the Actual Time

Now simulate traveling exactly `steps` edges.

For each edge traversal:

1. Check whether the current signal is red.
2. If red, wait until the next green interval.
3. Traverse the edge, adding `time`.

The signal state is determined by:

```
(current_time // change) % 2
```

- `0` means green
- `1` means red

If red:

```
wait_until = (current_time // change + 1) * change
```

Then continue.

### Why It Works

BFS explores paths in increasing order of edge count. Since all edges have equal traversal cost, the first and second distinct times a node is reached correspond to its shortest and second shortest path lengths in edges.

The traffic waiting behavior depends only on the global time and not on which nodes are visited. Therefore, once the second shortest edge count is known, simulating that many edge traversals produces the true second minimum arrival time.

Because we preserve the two smallest distinct path lengths for every node, we never miss the valid second minimum route.

## Python Solution

```python
from collections import deque, defaultdict
from typing import List

class Solution:
    def secondMinimum(self, n: int, edges: List[List[int]], time: int, change: int) -> int:
        graph = defaultdict(list)

        for u, v in edges:
            graph[u].append(v)
            graph[v].append(u)

        INF = float('inf')

        # dist[node][0] = shortest edge count
        # dist[node][1] = second shortest edge count
        dist = [[INF, INF] for _ in range(n + 1)]

        queue = deque()
        queue.append((1, 0))

        dist[1][0] = 0

        while queue:
            node, steps = queue.popleft()

            for neighbor in graph[node]:
                new_steps = steps + 1

                if new_steps < dist[neighbor][0]:
                    dist[neighbor][1] = dist[neighbor][0]
                    dist[neighbor][0] = new_steps
                    queue.append((neighbor, new_steps))

                elif dist[neighbor][0] < new_steps < dist[neighbor][1]:
                    dist[neighbor][1] = new_steps
                    queue.append((neighbor, new_steps))

        second_steps = dist[n][1]

        current_time = 0

        for step in range(second_steps):
            phase = (current_time // change) % 2

            # Red signal
            if phase == 1:
                current_time = (current_time // change + 1) * change

            current_time += time

        return current_time
```

The implementation begins by constructing the adjacency list representation of the graph. Since the graph is undirected, every edge is inserted in both directions.

The `dist` array stores two values for every node. The first represents the shortest number of edges needed to reach the node, and the second represents the second shortest distinct value.

The BFS queue stores the current node along with the number of edges used to reach it. Whenever a neighbor is explored, the algorithm checks whether the new path length improves either the shortest or second shortest value for that neighbor.

Only two distances are ever stored for each node, which keeps the BFS efficient even for large graphs.

After BFS completes, `dist[n][1]` contains the second minimum number of edge traversals required to reach the destination.

The final phase simulates the actual travel time. Before every edge traversal, the code checks whether the signal is currently red. If so, it waits until the next green interval before continuing.

The resulting accumulated time is returned.

## Go Solution

```go
package main

import (
	"container/list"
)

func secondMinimum(n int, edges [][]int, time int, change int) int {
	graph := make([][]int, n+1)

	for _, edge := range edges {
		u := edge[0]
		v := edge[1]

		graph[u] = append(graph[u], v)
		graph[v] = append(graph[v], u)
	}

	const INF = int(1e9)

	dist := make([][2]int, n+1)

	for i := 0; i <= n; i++ {
		dist[i][0] = INF
		dist[i][1] = INF
	}

	type State struct {
		node  int
		steps int
	}

	queue := list.New()
	queue.PushBack(State{1, 0})

	dist[1][0] = 0

	for queue.Len() > 0 {
		front := queue.Front()
		queue.Remove(front)

		cur := front.Value.(State)

		for _, neighbor := range graph[cur.node] {
			newSteps := cur.steps + 1

			if newSteps < dist[neighbor][0] {
				dist[neighbor][1] = dist[neighbor][0]
				dist[neighbor][0] = newSteps

				queue.PushBack(State{neighbor, newSteps})
			} else if dist[neighbor][0] < newSteps && newSteps < dist[neighbor][1] {
				dist[neighbor][1] = newSteps

				queue.PushBack(State{neighbor, newSteps})
			}
		}
	}

	secondSteps := dist[n][1]

	currentTime := 0

	for i := 0; i < secondSteps; i++ {
		phase := (currentTime / change) % 2

		// Red signal
		if phase == 1 {
			currentTime = (currentTime/change + 1) * change
		}

		currentTime += time
	}

	return currentTime
}
```

The Go implementation follows the same logic as the Python version.

A slice of slices is used for the adjacency list. The BFS queue is implemented using `container/list`, since Go does not provide a built in deque structure.

The `dist` structure uses fixed size arrays `[2]int` for efficiency because only two distances are needed per node.

Go integers are sufficient because the maximum possible answer is well within 32 bit integer range under the given constraints.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(V + E) | BFS processes each node state at most twice |
| Space | O(V + E) | Adjacency list plus distance tracking |

The BFS stores at most two distances per node, so each node enters the queue only a limited number of times. Every edge is explored during those BFS traversals, resulting in linear complexity relative to the graph size.

The final time simulation runs for the number of edges in the second shortest path, which is bounded by the BFS exploration and does not affect the asymptotic complexity.

## Edge Cases

One important edge case occurs when multiple shortest paths exist with the same travel time. A naive implementation might incorrectly treat another identical shortest path as the second minimum. The problem specifically requires the second value to be strictly larger. The implementation handles this by storing only distinct path lengths using the condition:

```
dist[neighbor][0] < new_steps < dist[neighbor][1]
```

Another tricky case occurs when the second minimum route requires revisiting nodes. In many shortest path problems, revisiting is unnecessary and can be pruned aggressively. Here, revisiting is explicitly allowed and sometimes required. For example, with only one edge between nodes `1` and `2`, the second minimum route becomes:

```
1 -> 2 -> 1 -> 2
```

The BFS correctly supports this because it allows nodes to be processed multiple times as long as a new distinct distance is discovered.

A third subtle edge case involves traffic signal timing. If a traveler arrives exactly when the signal changes from green to red, they must wait. The implementation correctly determines signal state using:

```
(current_time // change) % 2
```

When this value equals `1`, the signal is red, so the traveler waits until the next multiple of `change`. This ensures correct handling of boundary times where off by one errors are common.
