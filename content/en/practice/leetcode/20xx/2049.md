# LeetCode 2049 - Count Nodes With the Highest Score

## Problem Understanding

This problem gives us a rooted binary tree represented by a `parents` array. Each index represents a node, and `parents[i]` tells us the parent of node `i`. The root node is always node `0`, and its parent is `-1`.

The task is to compute a special score for every node in the tree. To calculate the score of a node, we imagine removing that node and all edges connected to it. This breaks the original tree into several disconnected subtrees. The score is the product of the sizes of all resulting non-empty subtrees.

We then need to determine the maximum score among all nodes and return how many nodes achieve that maximum score.

Because the tree is binary, every node can have at most two children. However, the algorithm should not rely too heavily on that fact, because the same subtree-size logic works for any rooted tree structure.

The constraints are important:

- `n` can be as large as `10^5`
- A naive solution that recomputes subtree sizes repeatedly would be too slow
- The input is guaranteed to form a valid binary tree
- Every node except the root has exactly one parent

The large input size strongly suggests that we need a linear or near-linear solution.

Several edge cases are important:

- The root node has no parent component after removal
- Leaf nodes produce only one remaining component
- Very unbalanced trees can create deep recursion
- The score can become very large, so integer multiplication must be handled carefully
- Some subtree sizes may be `0`, but only non-empty subtrees contribute to the score

## Approaches

### Brute Force Approach

A brute-force solution would compute the score of every node independently.

For each node:

1. Remove the node conceptually
2. Determine all connected components created
3. Compute the size of each component
4. Multiply the component sizes together

To determine component sizes, we could run DFS or BFS repeatedly after each removal.

This approach is correct because it directly follows the problem definition. However, it is far too slow. If we perform a traversal for every node, the total complexity becomes approximately `O(n^2)`.

With `n = 10^5`, this would be infeasible.

### Optimal Approach

The key observation is that removing a node divides the tree into at most three parts:

1. The left subtree
2. The right subtree
3. The remaining part of the tree above the node

If we already know the size of every subtree, we can compute the score of a node in constant time.

Suppose:

- `left_size` is the size of the left subtree
- `right_size` is the size of the right subtree
- `remaining = n - 1 - left_size - right_size`

Then:

- Each non-zero component contributes to the product
- The score is the product of all non-empty component sizes

So the real task becomes:

1. Build the tree
2. Compute subtree sizes using DFS
3. Compute scores while traversing
4. Track the maximum score and frequency

This leads to a linear-time solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Recomputes component sizes for every node |
| Optimal | O(n) | O(n) | Uses one DFS to compute subtree sizes and scores |

## Algorithm Walkthrough

1. Build an adjacency list representing the children of every node.

Since the input gives parent relationships, we convert it into a tree structure where each node stores its children. This makes DFS traversal straightforward.
2. Initialize global tracking variables.

We maintain:

- `max_score` to store the highest score seen so far
- `count` to store how many nodes achieve that score
3. Perform a depth-first search starting from the root.

The DFS returns the size of the subtree rooted at the current node.
4. For each node, recursively compute child subtree sizes.

During DFS:

- Visit every child
- Compute each child's subtree size
- Sum them together
5. Compute the remaining tree size.

If the current node's subtree size is `subtree_size`, then the remaining nodes outside this subtree equal:

```
remaining = n - subtree_size
```
6. Compute the node's score.

Start with score `1`.

Multiply by:

- Each child subtree size
- The remaining component size, if non-zero

We ignore zero-sized components because the problem specifies only non-empty subtrees contribute.
7. Update the global maximum.

- If the current score exceeds `max_score`, replace it and reset the count to `1`
- If the score equals `max_score`, increment the count
8. Return the subtree size to the parent.

The subtree size equals:

```
1 + sum(child subtree sizes)
```

### Why it works

The algorithm works because every possible connected component after removing a node corresponds exactly to either:

- One of its child subtrees
- The remaining nodes above the node

DFS guarantees that child subtree sizes are computed before the parent node processes its own score. Therefore, every score is computed correctly using already-known subtree sizes.

Since every node is processed exactly once, the algorithm runs in linear time.

## Python Solution

```python
from typing import List

class Solution:
    def countHighestScoreNodes(self, parents: List[int]) -> int:
        n = len(parents)

        children = [[] for _ in range(n)]

        for node in range(1, n):
            parent = parents[node]
            children[parent].append(node)

        max_score = 0
        count = 0

        def dfs(node: int) -> int:
            nonlocal max_score, count

            subtree_size = 1
            score = 1

            for child in children[node]:
                child_size = dfs(child)
                subtree_size += child_size
                score *= child_size

            remaining_size = n - subtree_size

            if remaining_size > 0:
                score *= remaining_size

            if score > max_score:
                max_score = score
                count = 1
            elif score == max_score:
                count += 1

            return subtree_size

        dfs(0)

        return count
```

The implementation begins by constructing the `children` adjacency list. Since the input only tells us each node's parent, we need to build downward connections to make DFS traversal efficient.

The DFS function returns the subtree size rooted at the current node. This is essential because parent nodes depend on child subtree sizes when computing their scores.

Inside DFS:

- `subtree_size` starts at `1` because the node itself belongs to its subtree
- `score` starts at `1` because we will multiply component sizes into it

For every child:

- Recursively compute the child's subtree size
- Add it to the current subtree size
- Multiply it into the score

After processing all children, we compute the size of the remaining component outside the current subtree. If that component exists, it also contributes to the score.

Finally, the code updates the global maximum score tracking variables.

The DFS visits each node exactly once, making the solution efficient enough for the constraints.

## Go Solution

```go
package main

func countHighestScoreNodes(parents []int) int {
	n := len(parents)

	children := make([][]int, n)

	for node := 1; node < n; node++ {
		parent := parents[node]
		children[parent] = append(children[parent], node)
	}

	var maxScore int64 = 0
	count := 0

	var dfs func(int) int

	dfs = func(node int) int {
		subtreeSize := 1
		score := int64(1)

		for _, child := range children[node] {
			childSize := dfs(child)
			subtreeSize += childSize
			score *= int64(childSize)
		}

		remainingSize := n - subtreeSize

		if remainingSize > 0 {
			score *= int64(remainingSize)
		}

		if score > maxScore {
			maxScore = score
			count = 1
		} else if score == maxScore {
			count++
		}

		return subtreeSize
	}

	dfs(0)

	return count
}
```

The Go implementation follows the same logic as the Python version but uses slices instead of Python lists.

One important difference is integer handling. Scores can become very large, so the implementation uses `int64` for score calculations to avoid overflow issues on systems where `int` may not be large enough.

The DFS function is implemented as a recursive closure, which allows it to access outer variables like `maxScore` and `count`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each node is visited exactly once |
| Space | O(n) | Adjacency list and recursion stack require linear space |

The DFS traversal processes every node once and performs constant work per node. Building the adjacency list also takes linear time.

The space complexity comes primarily from:

- The adjacency list storing all edges
- The recursion stack in the worst case of a skewed tree

In the worst-case unbalanced tree, recursion depth can reach `O(n)`.

## Edge Cases

One important edge case is the root node. When removing the root, there is no "remaining tree" above it. A buggy implementation might incorrectly multiply by zero and force the entire score to zero. This implementation avoids that problem by only multiplying non-zero component sizes.

Another important case is leaf nodes. Removing a leaf leaves the rest of the tree connected as one large component. Some implementations incorrectly treat missing child subtrees as size zero and multiply by zero. Here, we ignore empty components entirely, which matches the problem statement.

A highly skewed tree is another critical edge case. For example, a chain of `10^5` nodes creates maximum recursion depth. The algorithm itself still runs in linear time, but recursive implementations in some languages may need stack considerations. The logic remains correct because subtree sizes propagate upward one node at a time.

Balanced trees are also important because nodes may produce identical scores. A buggy implementation might only track the maximum score itself and forget to count how many nodes achieve it. This solution correctly increments the count whenever another node matches the current maximum.
