## Solution

## Problem Understanding

The problem asks us to calculate the running balance of each account in a banking system after every transaction. The input is a table named `Transactions` with columns `account_id`, `day`, `type`, and `amount`. Each row represents a single transaction for a particular account on a given day, where `type` is either `'Deposit'` or `'Withdraw'`. All accounts start with a balance of 0, and no account will ever go negative.

The output should be a table with `account_id`, `day`, and `balance`, showing the updated balance for each transaction. The results must be sorted first by `account_id` and then by `day`.

Important constraints include the guarantee that no balance will go negative and that `(account_id, day)` uniquely identifies each transaction. Edge cases include accounts with only deposits, only withdrawals (if any), or multiple transactions on the same day, as well as accounts that never transact.

## Approaches

The brute-force approach would involve iterating over every transaction for each account in chronological order and updating a running balance. While this would produce correct results, it requires sorting all transactions and manually accumulating balances, which can be inefficient if the dataset is large.

The optimal approach leverages SQL window functions, specifically `SUM() OVER (PARTITION BY account_id ORDER BY day)`, which allows computing a running total directly in the query. The key insight is that we can treat deposits as positive amounts and withdrawals as negative amounts and then compute a cumulative sum for each account chronologically.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) | O(n) | Sort transactions and accumulate balances per account in application code |
| Optimal | O(n log n) | O(n) | SQL window function computes running total efficiently in-database |

## Algorithm Walkthrough

1. Start by transforming each transaction into a numeric effect on the balance: deposits are positive, withdrawals are negative. This allows us to treat all transactions uniformly.
2. Sort all transactions by `account_id` and `day` to ensure chronological processing for each account.
3. Use a window function `SUM(amount) OVER (PARTITION BY account_id ORDER BY day)` to compute the cumulative balance for each account at each transaction.
4. Return the result table with `account_id`, `day`, and the computed `balance`.

**Why it works:**

By converting withdrawals to negative numbers, the cumulative sum across a sorted sequence of transactions for each account naturally produces the correct balance after each transaction. Partitioning by `account_id` ensures balances are computed independently for each account.

## Python Solution

```python
import pandas as pd

def get_account_balance(transactions: pd.DataFrame) -> pd.DataFrame:
    # Convert transaction type to numeric effect
    transactions['amount_signed'] = transactions.apply(
        lambda row: row['amount'] if row['type'] == 'Deposit' else -row['amount'],
        axis=1
    )
    
    # Sort by account_id and day
    transactions = transactions.sort_values(by=['account_id', 'day'])
    
    # Compute running balance using cumulative sum per account
    transactions['balance'] = transactions.groupby('account_id')['amount_signed'].cumsum()
    
    # Select and return the required columns
    return transactions[['account_id', 'day', 'balance']]
```

The Python solution reads the transactions into a DataFrame, converts withdrawals to negative values, sorts the data, and uses `groupby().cumsum()` to calculate the running balance for each account. Finally, it returns the required columns.

## Go Solution

```go
package main

import (
    "sort"
    "time"
)

type Transaction struct {
    AccountID int
    Day       string
    Type      string
    Amount    int
}

type AccountBalance struct {
    AccountID int
    Day       string
    Balance   int
}

func GetAccountBalance(transactions []Transaction) []AccountBalance {
    // Sort transactions by account_id and day
    sort.Slice(transactions, func(i, j int) bool {
        if transactions[i].AccountID == transactions[j].AccountID {
            return transactions[i].Day < transactions[j].Day
        }
        return transactions[i].AccountID < transactions[j].AccountID
    })

    balances := make([]AccountBalance, 0, len(transactions))
    accountRunning := make(map[int]int)

    for _, tx := range transactions {
        amt := tx.Amount
        if tx.Type == "Withdraw" {
            amt = -amt
        }
        accountRunning[tx.AccountID] += amt
        balances = append(balances, AccountBalance{
            AccountID: tx.AccountID,
            Day:       tx.Day,
            Balance:   accountRunning[tx.AccountID],
        })
    }

    return balances
}
```

The Go solution mirrors the Python logic: sort transactions, track running balances in a map keyed by account_id, apply signed amounts, and append results.

## Edge Cases

**1. No transactions for an account:** Some accounts may not have any transactions. The implementation ignores such accounts, producing no rows, which is correct because there is no balance to report.

**2. Multiple transactions on the same day:** Sorting by day ensures chronological order. If multiple transactions occur on the same day, the implementation will compute balances in the order they appear in the sorted list, preserving correctness.

**3. Large transaction amounts:** The balance could potentially exceed typical integer limits. Python handles large integers natively, whereas in Go, careful attention to integer type (`int64` if necessary) is needed to prevent overflow. The algorithm itself remains correct as long as types support the values.
