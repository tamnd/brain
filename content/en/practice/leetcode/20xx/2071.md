## Solution

## Problem Understanding

The problem asks us to determine the **maximum number of tasks that can be assigned** to a set of workers given their strength levels, optional magical pills, and task requirements. Each task has a minimum strength requirement and each worker has a certain strength. A worker can complete a task if their strength is **greater than or equal** to the task's requirement. We also have a limited number of magical pills that can increase a worker's strength by a fixed amount, but each worker can take **at most one pill**.

The input consists of two integer arrays: `tasks` and `workers`, representing the strength required for each task and the strength of each worker, respectively. Two integers, `pills` and `strength`, represent the number of available magical pills and the strength bonus each pill provides. The output is a single integer indicating the maximum number of tasks that can be completed under these constraints.

The constraints suggest that both arrays can be large (`1 <= n, m <= 5 * 10^4`) and individual values can also be large (`<= 10^9`). This implies that a naive brute-force approach of trying every possible assignment is computationally infeasible. Edge cases include scenarios where no worker is strong enough, all workers are strong enough, or the pills must be strategically assigned to achieve the maximum completion count.

## Approaches

The brute-force approach would involve iterating over every permutation of workers and tasks, trying all combinations of pill assignments. This would ensure correctness but would have exponential time complexity due to the factorial growth in the number of permutations, which is impossible for input sizes up to `5 * 10^4`.

The key observation to optimize is that **tasks and workers can be sorted** and that the problem allows for a **binary search over the number of tasks we can assign**. For a given number of tasks `k`, we can check if it is possible to assign the `k` weakest tasks to some subset of workers using a **greedy approach with a two-pointer method and a double-ended queue** to decide which workers receive pills. This reduces the problem to **logarithmic binary search combined with linear feasibility checking**, which is efficient enough for the given constraints.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n! * m!) | O(n) | Try every worker-task assignment with all pill options; infeasible for large input |
| Optimal | O(n log n + m log m + log(min(n,m)) * k) | O(k) | Sort arrays, binary search on number of tasks, use deque for efficient pill allocation |

## Algorithm Walkthrough

1. **Sort both arrays**: Sort `tasks` in ascending order and `workers` in ascending order. This allows matching weakest tasks with weakest capable workers first.
2. **Binary search on task count**: Set `low = 0` and `high = min(n, m)`. While `low <= high`, compute `mid = (low + high) // 2`. `mid` represents the candidate number of tasks we try to assign.
3. **Check feasibility for `mid` tasks**: Use a helper function to determine if we can assign the `mid` weakest tasks to workers using at most the available pills. Initialize two pointers: one for the weakest available worker and one for the strongest available worker. Maintain a deque for tasks that may require pills.
4. **Greedy assignment**:

- For each task starting from the weakest, check if the weakest available worker can complete it without a pill.
- If yes, assign the worker to the task.
- If no, check if the strongest available worker can complete it **with a pill**. If yes, decrement the pill count and assign. If not, it is impossible to complete `mid` tasks.
5. **Adjust binary search**:

- If the tasks are feasible, update `low = mid + 1`.
- Otherwise, update `high = mid - 1`.
6. **Return the maximum feasible task count**: After the binary search terminates, `high` will contain the maximum number of tasks that can be completed.

**Why it works**: Sorting ensures that we always try to match the weakest tasks first, which maximizes the number of tasks that can be assigned. The binary search guarantees that we find the largest feasible number of tasks efficiently, while the greedy deque-based assignment ensures pills are used optimally.

## Python Solution

```python
from collections import deque
from typing import List

class Solution:
    def maxTaskAssign(self, tasks: List[int], workers: List[int], pills: int, strength: int) -> int:
        tasks.sort()
        workers.sort()

        def can_assign(k: int) -> bool:
            task_deque = deque()
            i, j = 0, len(workers) - k
            available_pills = pills
            for t in tasks[:k]:
                while i < j and workers[i] < t:
                    task_deque.append(t)
                    i += 1
                if i < j:
                    i += 1
                elif available_pills > 0 and workers[j] + strength >= t:
                    available_pills -= 1
                    j += 1
                else:
                    return False
            return True

        low, high = 0, min(len(tasks), len(workers))
        result = 0
        while low <= high:
            mid = (low + high) // 2
            if can_assign(mid):
                result = mid
                low = mid + 1
            else:
                high = mid - 1
        return result
```

The Python implementation first sorts `tasks` and `workers`. The `can_assign` function tests if a certain number of tasks `k` can be assigned, greedily using pills on the strongest necessary workers. The main function uses binary search on the number of tasks and updates the result accordingly.

## Go Solution

```go
package main

import (
    "sort"
)

func maxTaskAssign(tasks []int, workers []int, pills int, strength int) int {
    sort.Ints(tasks)
    sort.Ints(workers)

    canAssign := func(k int) bool {
        left, right := 0, len(workers)-1
        availablePills := pills
        for i := 0; i < k; i++ {
            t := tasks[i]
            if workers[left] >= t {
                left++
            } else if workers[right]+strength >= t && availablePills > 0 {
                right--
                availablePills--
            } else {
                return false
            }
        }
        return true
    }

    low, high, result := 0, min(len(tasks), len(workers)), 0
    for low <= high {
        mid := (low + high) / 2
        if canAssign(mid) {
            result = mid
            low = mid + 1
        } else {
            high = mid - 1
        }
    }
    return result
}

func min(a, b int) int {
    if a < b {
        return a
    }
    return b
}
```

In Go, slices are sorted and the helper function `canAssign` mirrors the Python greedy logic. Edge cases such as empty slices or no available pills are handled by default logic without special cases.

## Worked Examples

Binary search tries `mid=3`: feasible with pill given to worker 0 for task 1

Sorting: `tasks = [4,5]`, `workers = [0,0,0]`

`mid=2` is infeasible, `mid=1` feasible by giving pill to one worker

Maximum tasks assigned = 1

**Example 3:** `tasks = [10,15,30]`, `workers = [0,10,10,10,10]`, `pills = 3`, `strength = 10`

`mid=2` feasible by using pills for first two tasks

`mid=3` infeasible because last task requires 30

Maximum tasks assigned = 2

Sorting is required to enable the greedy assignment. Binary search reduces the number of feasibility checks. Greedy assignment ensures each task is assigned optimally using the weakest worker possible and pills efficiently.
