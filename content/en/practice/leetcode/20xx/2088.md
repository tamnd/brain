**Difficulty:** 🔴 Hard  
**Topics:** Array, Dynamic Programming, Matrix

## Solution

## Problem Understanding

This problem gives us a binary matrix where each cell represents a piece of land. A value of `1` means the land is fertile, while `0` means the land is barren. We must count how many valid pyramidal and inverse pyramidal plots exist inside the grid.

A normal pyramid grows downward from a top apex. If the pyramid has height `h`, then each lower row expands by one cell to both the left and right. For example:

```
  1
 111
11111
```

An inverse pyramid is the same shape flipped vertically. It grows upward from a bottom apex.

The important requirement is that every cell inside the pyramid must be fertile. Also, the pyramid must contain more than one cell, meaning height must be at least `2`.

The input size is important:

- `m, n <= 1000`
- `m * n <= 100000`

This means the total number of cells is at most `100000`, so we need something close to linear time. Any brute-force solution that repeatedly scans large triangular regions will become too slow.

A few important observations immediately stand out:

- A single fertile cell does not count as a pyramid.
- A pyramid of height `h` automatically contains all smaller pyramids inside it.
- Boundary cells cannot form large pyramids because expansion would go outside the grid.
- A barren cell completely breaks any pyramid passing through it.

The challenge is efficiently determining, for every cell, the largest pyramid height possible.

## Approaches

### Brute Force Approach

A direct approach is to treat every fertile cell as a possible apex and attempt to grow pyramids layer by layer.

For a downward pyramid:

1. Start from cell `(r, c)`.
2. Try height `2`, then height `3`, and so on.
3. For each new layer, verify that every required cell is fertile.
4. Stop once any required cell is barren or outside the grid.

We repeat the same process for inverse pyramids.

This approach is correct because it explicitly validates every possible pyramid shape. However, it is inefficient. In the worst case, for each cell we may try many heights, and for each height we scan an entire row segment.

For a dense `1000 x 1000` grid of all `1`s, this becomes far too expensive.

### Key Insight for the Optimal Solution

The key observation is that a pyramid can be built recursively from smaller pyramids.

Suppose we want to know the largest downward pyramid rooted at `(r, c)`.

That pyramid can only have height greater than `1` if:

- the current cell is fertile,
- the cell directly below can support a pyramid,
- the lower-left cell can support a pyramid,
- the lower-right cell can support a pyramid.

More specifically:

```
dp[r][c] = 1 + min(
    dp[r+1][c-1],
    dp[r+1][c],
    dp[r+1][c+1]
)
```

This works because a pyramid expands symmetrically. The smallest supporting child pyramid determines the maximum achievable height.

This transforms the problem into dynamic programming over the matrix.

We compute:

- downward pyramids from bottom to top,
- upward pyramids from top to bottom.

Each cell contributes:

```
height - 1
```

because a height of `1` is just a single cell and does not count.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m * n * min(m, n)^2) | O(1) | Explicitly checks every pyramid layer |
| Optimal Dynamic Programming | O(m * n) | O(m * n) | Reuses previously computed pyramid heights |

## Algorithm Walkthrough

### Step 1: Define the DP Meaning

We create a DP table where:

```
dp[r][c]
```

stores the maximum pyramid height with apex at `(r, c)`.

For downward pyramids:

- height extends downward,
- computation proceeds bottom-up.

For upward pyramids:

- height extends upward,
- computation proceeds top-down.

### Step 2: Compute Downward Pyramids

We process rows from bottom to top.

If `grid[r][c] == 0`, then:

```
dp[r][c] = 0
```

because barren cells cannot belong to any pyramid.

Otherwise:

- the minimum pyramid height is `1`,
- if the three supporting cells below exist, we extend the pyramid.

Transition:

```
dp[r][c] = 1 + min(
    dp[r+1][c-1],
    dp[r+1][c],
    dp[r+1][c+1]
)
```

Every cell contributes:

```
dp[r][c] - 1
```

valid downward pyramids.

For example:

- height `1` contributes `0`,
- height `2` contributes `1`,
- height `4` contributes `3`.

This is because all smaller heights are also valid pyramids.

### Step 3: Compute Upward Pyramids

We repeat the exact same logic but reverse direction.

Now we process from top to bottom:

```
dp[r][c] = 1 + min(
    dp[r-1][c-1],
    dp[r-1][c],
    dp[r-1][c+1]
)
```

Again, each cell contributes:

```
dp[r][c] - 1
```

inverse pyramids.

### Step 4: Return Total Count

The final answer is:

```
downward pyramids + upward pyramids
```

### Why it works

The DP recurrence works because a pyramid of height `h` requires three supporting pyramids of height at least `h-1` directly beneath it. The smallest of those supporting heights determines how large the current pyramid can become.

This creates an inductive structure:

- height `1` pyramids are trivially valid for fertile cells,
- larger pyramids are built only from smaller valid pyramids.

Thus every computed height is correct, and every valid pyramid is counted exactly once.

## Python Solution

```python
from typing import List

class Solution:
    def countPyramids(self, grid: List[List[int]]) -> int:
        rows = len(grid)
        cols = len(grid[0])

        def count_downward() -> int:
            dp = [[0] * cols for _ in range(rows)]
            total = 0

            for r in range(rows - 1, -1, -1):
                for c in range(cols):
                    if grid[r][c] == 0:
                        continue

                    dp[r][c] = 1

                    if (
                        r + 1 < rows
                        and c - 1 >= 0
                        and c + 1 < cols
                    ):
                        dp[r][c] += min(
                            dp[r + 1][c - 1],
                            dp[r + 1][c],
                            dp[r + 1][c + 1]
                        )

                    total += dp[r][c] - 1

            return total

        def count_upward() -> int:
            dp = [[0] * cols for _ in range(rows)]
            total = 0

            for r in range(rows):
                for c in range(cols):
                    if grid[r][c] == 0:
                        continue

                    dp[r][c] = 1

                    if (
                        r - 1 >= 0
                        and c - 1 >= 0
                        and c + 1 < cols
                    ):
                        dp[r][c] += min(
                            dp[r - 1][c - 1],
                            dp[r - 1][c],
                            dp[r - 1][c + 1]
                        )

                    total += dp[r][c] - 1

            return total

        return count_downward() + count_upward()
```

The implementation separates the problem into two independent passes.

The `count_downward` function computes pyramids that grow downward. The traversal order is critical. We process from bottom to top so that when evaluating a cell, all supporting cells below have already been computed.

Each fertile cell starts with height `1`. If the three supporting positions exist, we extend the pyramid using the minimum child height.

The expression:

```python
dp[r][c] - 1
```

counts how many valid pyramids originate from this apex.

The `count_upward` function mirrors the exact same logic but processes rows from top to bottom. This handles inverse pyramids.

Finally, we sum both counts.

## Go Solution

```go
func countPyramids(grid [][]int) int {
	rows := len(grid)
	cols := len(grid[0])

	countDownward := func() int {
		dp := make([][]int, rows)

		for i := range dp {
			dp[i] = make([]int, cols)
		}

		total := 0

		for r := rows - 1; r >= 0; r-- {
			for c := 0; c < cols; c++ {
				if grid[r][c] == 0 {
					continue
				}

				dp[r][c] = 1

				if r+1 < rows && c-1 >= 0 && c+1 < cols {
					minVal := dp[r+1][c-1]

					if dp[r+1][c] < minVal {
						minVal = dp[r+1][c]
					}

					if dp[r+1][c+1] < minVal {
						minVal = dp[r+1][c+1]
					}

					dp[r][c] += minVal
				}

				total += dp[r][c] - 1
			}
		}

		return total
	}

	countUpward := func() int {
		dp := make([][]int, rows)

		for i := range dp {
			dp[i] = make([]int, cols)
		}

		total := 0

		for r := 0; r < rows; r++ {
			for c := 0; c < cols; c++ {
				if grid[r][c] == 0 {
					continue
				}

				dp[r][c] = 1

				if r-1 >= 0 && c-1 >= 0 && c+1 < cols {
					minVal := dp[r-1][c-1]

					if dp[r-1][c] < minVal {
						minVal = dp[r-1][c]
					}

					if dp[r-1][c+1] < minVal {
						minVal = dp[r-1][c+1]
					}

					dp[r][c] += minVal
				}

				total += dp[r][c] - 1
			}
		}

		return total
	}

	return countDownward() + countUpward()
}
```

The Go implementation follows the same DP structure as the Python solution.

One difference is that Go does not provide a built-in `min` function for integers, so we manually compute the minimum among the three neighboring values.

Slices are initialized explicitly using `make`, and integer arithmetic is safe because the maximum number of pyramids fits comfortably within Go's `int` range for the given constraints.

Final answer:


## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m * n) | Each cell is processed a constant number of times |
| Space | O(m * n) | DP tables store one integer per cell |

The algorithm performs two full matrix traversals, one for downward pyramids and one for upward pyramids. Each traversal touches every cell once and performs only constant-time operations.

The DP tables require storage proportional to the grid size.

## Edge Cases

### Single Row or Single Column

If the grid has only one row or one column, no pyramid can exist because pyramids require horizontal expansion and at least two rows. A naive implementation might accidentally count single cells as height-1 pyramids. This solution correctly subtracts `1` from every DP height, so height `1` contributes nothing.

### Boundary Cells

Cells near the left or right border cannot support large pyramids because expansion would move outside the grid. Without proper boundary checks, an implementation could access invalid indices. The algorithm explicitly verifies:

```
c - 1 >= 0 and c + 1 < cols
```

before extending a pyramid.

### Internal Zeroes Breaking Large Pyramids

A pyramid may appear visually valid near the top but fail because of a single barren cell deep inside. Brute-force approaches sometimes miss this if they only inspect borders. The DP recurrence avoids this issue entirely because every larger pyramid depends on smaller valid pyramids beneath it. A single zero propagates upward and prevents invalid expansion automatically.
