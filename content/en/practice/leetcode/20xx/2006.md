# LeetCode 2006 - Count Number of Pairs With Absolute Difference K

## Problem Understanding

The problem gives us an integer array `nums` and an integer `k`. We must count how many index pairs `(i, j)` satisfy two conditions:

1. `i < j`
2. `|nums[i] - nums[j]| == k`

The notation `|x|` means the absolute value of `x`, so the difference between two numbers can be either positive or negative, but its magnitude must equal `k`.

In simpler terms, we need to examine every pair of elements in the array and determine whether their difference is exactly `k`. If it is, we count that pair.

The input array contains integers in the range `1` to `100`, and the array length is at most `200`. These constraints are relatively small, which means even a quadratic solution is acceptable. However, the problem is still a good opportunity to practice frequency counting and hash map usage.

The output is a single integer representing the total number of valid pairs.

A few important observations come directly from the constraints:

- The array length is small, so an `O(n^2)` brute-force solution will pass.
- The values are also small, which makes counting-based approaches very efficient.
- Since `k >= 1`, we never need to count pairs of equal numbers.
- The condition `i < j` prevents double counting the same pair.

Several edge cases are worth considering upfront:

- Arrays with no valid pairs.
- Arrays where many duplicates exist.
- Arrays where every pair contributes to the answer.
- Very small arrays such as length `1`.
- Cases where the same value appears multiple times and contributes to multiple pair combinations.

For example, in `nums = [1,2,2,1]` and `k = 1`, each `1` can pair with each `2`, creating four total valid pairs.

## Approaches

### Brute-Force Approach

The most direct solution is to check every possible pair `(i, j)` where `i < j`.

For each pair:

1. Compute `abs(nums[i] - nums[j])`
2. If the result equals `k`, increment the answer.

This approach is correct because it explicitly examines every valid pair exactly once. No pair can be missed, and no pair can be counted twice.

The downside is the time complexity. Since we compare every pair, the algorithm requires nested loops, leading to `O(n^2)` time complexity.

Although the constraints are small enough for this to pass, we can still design a cleaner and more efficient solution.

### Optimal Hash Map Approach

The key insight is that for each number `x`, the only numbers that can form a valid pair with it are:

- `x - k`
- `x + k`

Instead of checking every previous element manually, we can maintain a frequency map of numbers we have already seen.

As we iterate through the array:

- If we previously saw `num - k`, then each occurrence forms a valid pair with the current number.
- If we previously saw `num + k`, then each occurrence also forms a valid pair.

We add both counts to the answer, then store the current number in the frequency map.

This works because the map always contains only elements from earlier indices, automatically enforcing `i < j`.

| Optimal | O(n) | O(n) | Uses a hash map to count previously seen values |

## Algorithm Walkthrough

### Optimal Hash Map Algorithm

1. Initialize a variable `pair_count` to `0`.

This variable stores the total number of valid pairs found so far.
2. Create an empty hash map called `frequency`.

The hash map stores how many times each number has appeared earlier in the array.
3. Traverse the array from left to right.

At each step, the current element becomes the second element of a pair, while the hash map contains all possible first elements.
4. For the current number `num`, check whether `num - k` exists in the map.

Every occurrence of `num - k` forms a valid pair because:

```
|num - (num - k)| = k
```

Add its frequency to `pair_count`.
5. Check whether `num + k` exists in the map.

Every occurrence of `num + k` also forms a valid pair because:

```
|num - (num + k)| = k
```

Add this frequency as well.
6. Insert the current number into the frequency map.

Increment its count because future elements may pair with it.
7. Continue until all elements are processed.
8. Return `pair_count`.

### Why it works

The algorithm maintains the invariant that the frequency map contains exactly the elements that appear before the current index. Therefore, every counted pair automatically satisfies `i < j`.

For each current number `num`, the only possible values that can produce an absolute difference of `k` are `num - k` and `num + k`. By counting how many times those values have already appeared, we count every valid pair exactly once.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Solution:
    def countKDifference(self, nums: List[int], k: int) -> int:
        frequency = defaultdict(int)
        pair_count = 0

        for num in nums:
            pair_count += frequency[num - k]
            pair_count += frequency[num + k]

            frequency[num] += 1

        return pair_count
```

The solution uses a `defaultdict(int)` so missing keys automatically return `0`. This removes the need for explicit existence checks.

The variable `frequency` tracks how many times each number has appeared earlier in the traversal.

For every number:

- `frequency[num - k]` counts valid smaller partners.
- `frequency[num + k]` counts valid larger partners.

After counting these pairs, the current number is added to the map so it can help future elements form pairs.

The order is important. We must count pairs before inserting the current element into the map. Otherwise, we could accidentally count the current element with itself.

## Go Solution

```go
func countKDifference(nums []int, k int) int {
    frequency := make(map[int]int)
    pairCount := 0

    for _, num := range nums {
        pairCount += frequency[num-k]
        pairCount += frequency[num+k]

        frequency[num]++
    }

    return pairCount
}
```

The Go implementation follows the same logic as the Python version.

Go maps return the zero value for missing keys, so accessing `frequency[num-k]` is safe even if the key does not exist.

Since the constraints are very small, integer overflow is not a concern here. Standard `int` values are more than sufficient.

The algorithm performs a single pass through the array. Each hash map lookup and insertion is `O(1)` on average, so the total running time is linear in the size of the input.

The extra memory comes from storing frequencies of previously seen numbers. In the worst case, every number is distinct, so the map stores `n` entries.

## Edge Cases

### Single Element Array

If the array contains only one element, there are no possible pairs because a pair requires two distinct indices. A careless implementation might assume at least one pair exists or accidentally access invalid indices. The current implementation handles this naturally because the loop never finds any matching previous values.

### Large Number of Duplicates

Arrays with repeated values can create many valid combinations. For example:

```
nums = [1,2,1,2,1,2], k = 1
```

Every `1` can pair with every `2`, resulting in nine valid pairs. A buggy implementation might undercount or double count these combinations. The frequency map correctly tracks how many matching values have already appeared, ensuring every valid pair contributes exactly once.

### No Valid Pairs

Some arrays contain no pairs whose absolute difference equals `k`. For example:

```
nums = [1,3], k = 3
```

The implementation safely returns `0` because missing keys in the frequency map contribute zero additional pairs.

### Descending or Unsorted Arrays

The array is not guaranteed to be sorted. A solution that assumes sorted order could fail on inputs like:

```
nums = [10,8,6,4]
```

The hash map approach works regardless of ordering because it only depends on previously seen values, not sorted positions.
