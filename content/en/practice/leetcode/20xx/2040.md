## Solution

## Problem Understanding

The problem asks us to find the `k`th smallest value among all possible products formed by taking one element from `nums1` and one element from `nums2`.

More formally, for every valid pair `(i, j)`, we compute:

```
nums1[i] * nums2[j]
```

We then conceptually place all these products into one large sorted list and return the element at position `k` using 1-based indexing.

The arrays are already sorted, which is the key property that allows us to design an efficient solution.

The constraints are extremely important here:

- Each array can contain up to `5 * 10^4` elements
- The total number of products can therefore reach:

```
5 * 10^4 * 5 * 10^4 = 2.5 * 10^9
```

This immediately tells us that explicitly generating all products is impossible in both time and memory.

Another important detail is that the arrays may contain:

- Negative numbers
- Positive numbers
- Zero

This makes the ordering of products much more complicated than in standard multiplication problems.

For example:

- Negative × Positive = Negative
- Negative × Negative = Positive
- Zero × Anything = Zero

Because of this, the product space is not monotonic in a simple left-to-right way unless we carefully analyze signs.

Some important edge cases include:

- Arrays containing only negative values
- Arrays containing only positive values
- Arrays containing many zeros
- Duplicate products
- Extremely large positive and negative products
- Very small or very large `k`

The product values can also become large:

```
10^5 * 10^5 = 10^10
```

So 32-bit integers are not sufficient. We must use 64-bit integer arithmetic.

## Approaches

### Brute Force Approach

The most straightforward solution is to generate every possible product.

For each element in `nums1`, we multiply it with every element in `nums2`, store the results in a list, sort the list, and return the `(k - 1)`th element.

This approach is correct because it explicitly constructs the exact set of all products and sorts them into the required order.

However, the complexity is disastrous.

If both arrays have length `5 * 10^4`, then the total number of products is:

```
2.5 * 10^9
```

Generating billions of products is infeasible in both memory and runtime.

Therefore, we need a fundamentally different strategy.

### Optimal Approach

The key insight is that we do not actually need to generate all products.

Instead, we can binary search on the answer itself.

Suppose we guess a value `mid`.

We can ask:

```
How many products are <= mid?
```

If that count is at least `k`, then the `k`th smallest product must be `<= mid`.

Otherwise, it must be larger.

This creates a monotonic condition:

- Smaller `mid` values produce smaller counts
- Larger `mid` values produce larger counts

That monotonicity allows binary search.

The remaining challenge is efficiently counting how many products are `<= mid`.

Because the arrays are sorted, we can use binary search inside the counting process.

The handling differs depending on whether the current number is:

- Positive
- Negative
- Zero

This sign-based reasoning is the core difficulty of the problem.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * m * log(n * m)) | O(n * m) | Generates and sorts all products |
| Optimal | O((n + m) * log(MAX) * log(min(n,m))) | O(1) | Binary search on answer with efficient counting |

## Algorithm Walkthrough

### Step 1: Define the Search Space

The smallest possible product is:

```
-10^5 * 10^5 = -10^10
```

The largest possible product is:

```
10^5 * 10^5 = 10^10
```

So we binary search in the range:

```
[-10^10, 10^10]
```

### Step 2: Binary Search on the Answer

For each midpoint `mid`, we determine:

```
count = number of products <= mid
```

If:

```
count >= k
```

then `mid` may already be the answer, so we continue searching left.

Otherwise, we search right.

### Step 3: Count Products Less Than or Equal to `mid`

We iterate through every value `x` in `nums1`.

For each `x`, we count how many values `y` in `nums2` satisfy:

```
x * y <= mid
```

The logic depends on the sign of `x`.

### Step 4: Handle Positive `x`

If:

```
x > 0
```

then dividing preserves inequality direction:

```
y <= mid // x
```

Because `nums2` is sorted, we can use binary search to find how many elements satisfy this condition.

### Step 5: Handle Negative `x`

If:

```
x < 0
```

then dividing reverses inequality direction:

```
y >= ceil(mid / x)
```

We again use binary search to find the first valid position.

### Step 6: Handle Zero

If:

```
x == 0
```

then:

```
x * y = 0
```

So:

- If `mid >= 0`, every element in `nums2` contributes
- Otherwise, none contribute

### Step 7: Continue Binary Search

We repeat until:

```
left == right
```

At that point, we have found the smallest value whose count is at least `k`, which is exactly the `k`th smallest product.

### Why it works

The binary search works because the predicate:

```
count(products <= mid) >= k
```

is monotonic.

If it is true for some `mid`, it will remain true for all larger values.

The counting procedure is correct because, for each fixed `x`, the sorted order of `nums2` allows binary searching the valid range of `y` values satisfying:

```
x * y <= mid
```

Combining counts across all elements gives the exact number of products less than or equal to `mid`.

## Python Solution

```python
from typing import List
import bisect

class Solution:
    def kthSmallestProduct(self, nums1: List[int], nums2: List[int], k: int) -> int:
        
        def count_less_equal(target: int) -> int:
            count = 0
            
            for x in nums1:
                if x > 0:
                    limit = target // x
                    count += bisect.bisect_right(nums2, limit)
                
                elif x < 0:
                    # Need y >= ceil(target / x)
                    limit = -(-target // x)
                    idx = bisect.bisect_left(nums2, limit)
                    count += len(nums2) - idx
                
                else:
                    if target >= 0:
                        count += len(nums2)
            
            return count
        
        left = -10**10
        right = 10**10
        
        while left < right:
            mid = (left + right) // 2
            
            if count_less_equal(mid) >= k:
                right = mid
            else:
                left = mid + 1
        
        return left
```

The implementation follows the exact structure described in the algorithm walkthrough.

The helper function `count_less_equal()` computes how many products are less than or equal to a given target value.

For positive values of `x`, we compute the maximum valid `y` and use `bisect_right()` to count all valid elements.

For negative values of `x`, the inequality reverses direction after division, so we compute the lower bound and use `bisect_left()`.

For zero, all products equal zero, so every element contributes when the target is nonnegative.

The outer binary search repeatedly narrows the answer range until only one value remains.

Because the search range is finite and monotonic, the algorithm always converges correctly.

## Go Solution

```go
package main

import "sort"

func kthSmallestProduct(nums1 []int, nums2 []int, k int64) int64 {

	countLessEqual := func(target int64) int64 {
		var count int64 = 0

		for _, val := range nums1 {
			x := int64(val)

			if x > 0 {
				limit := target / x

				cnt := sort.Search(len(nums2), func(i int) bool {
					return int64(nums2[i]) > limit
				})

				count += int64(cnt)

			} else if x < 0 {

				limit := ceilDiv(target, x)

				idx := sort.Search(len(nums2), func(i int) bool {
					return int64(nums2[i]) >= limit
				})

				count += int64(len(nums2) - idx)

			} else {

				if target >= 0 {
					count += int64(len(nums2))
				}
			}
		}

		return count
	}

	var left int64 = -10000000000
	var right int64 = 10000000000

	for left < right {
		mid := left + (right-left)/2

		if countLessEqual(mid) >= k {
			right = mid
		} else {
			left = mid + 1
		}
	}

	return left
}

func ceilDiv(a, b int64) int64 {
	q := a / b
	r := a % b

	if r != 0 && ((r > 0) == (b > 0)) {
		q++
	}

	return q
}
```

The Go version follows the same overall strategy as the Python implementation.

The biggest difference is handling integer division carefully. Go truncates division toward zero, so we implement a dedicated `ceilDiv()` helper to correctly compute ceiling division for negative values.

The solution also uses `int64` everywhere because product values can exceed 32-bit integer limits.

The `sort.Search()` function is used to perform binary searches on slices efficiently.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n * log(m) * log(MAX)) | Binary search over answer space, each step performs binary searches |
| Space | O(1) | Only constant extra memory is used |

Here:

- `n = len(nums1)`
- `m = len(nums2)`
- `MAX` is the numeric search range, approximately `2 * 10^10`

The outer binary search performs about 45 iterations because:

```
log2(2 * 10^10) ≈ 45
```

Inside each iteration, we process every element of `nums1` and perform a binary search on `nums2`.

This is efficient enough for the given constraints.

## Edge Cases

### Arrays Containing Zero

Zero is particularly tricky because:

```
0 * anything = 0
```

This creates a large block of identical products.

A naive implementation can easily miscount these values, especially when `mid` is negative.

The implementation handles this explicitly:

- If `x == 0` and `mid >= 0`, all products qualify
- Otherwise, none qualify

This guarantees accurate counting around zero boundaries.

### Negative Numbers Reversing Inequalities

When dividing inequalities involving negative numbers, the inequality direction reverses.

For example:

```
x * y <= mid
```

with negative `x` becomes:

```
y >= ceil(mid / x)
```

Failing to reverse the inequality correctly leads to incorrect counts.

The implementation carefully handles this using ceiling division and lower-bound binary search.

### Extremely Large Products

Products can reach:

```
10^10
```

which exceeds 32-bit integer limits.

Using standard integer arithmetic would overflow in many languages.

The implementation avoids this by:

- Using Python's arbitrary precision integers
- Using `int64` in Go

This guarantees correctness for all valid inputs.
