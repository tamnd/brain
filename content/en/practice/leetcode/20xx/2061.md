## Solution

## Problem Understanding

In this problem, we are given a 2D binary matrix called `room`. Each cell contains either:

- `0`, meaning the cell is empty and the robot may move onto it
- `1`, meaning the cell contains an obstacle

The robot always starts at the top-left corner `(0, 0)`, and the problem guarantees that this starting cell is empty.

The robot initially faces right. It repeatedly follows the same movement rules forever:

1. Move forward in the current direction as long as the next cell is valid and empty.
2. If the next cell is outside the grid or contains an obstacle, rotate 90 degrees clockwise.
3. Continue indefinitely.

Every cell the robot visits becomes cleaned. Since the robot never stops moving, we must determine how many distinct cells are eventually cleaned.

The important detail is that the robot may revisit cells many times. We only count unique cleaned cells.

The grid dimensions can be as large as `300 x 300`, which means there may be up to `90,000` cells. A naive infinite simulation would obviously not work, because the robot can move forever. We therefore need a way to detect when the robot's behavior starts repeating.

A critical observation is that the robot's future behavior depends not only on its position, but also on its direction. If the robot reaches the same `(row, column, direction)` state again, then the entire movement pattern from that point onward will repeat forever. At that moment, we can safely terminate the simulation.

Several edge cases are important:

- A room with only one cell
- A robot immediately trapped by obstacles
- A room with no obstacles, where the robot loops around the perimeter
- Situations where the robot revisits cells from different directions
- Large grids where efficiency matters

The problem guarantees that the starting cell is always empty, so we never need to handle an invalid starting position.

## Approaches

### Brute Force Approach

The most straightforward idea is to directly simulate the robot's movement forever while recording cleaned cells.

At every step, we attempt to move forward. If movement is impossible, we rotate clockwise. Every visited empty cell is added to a set of cleaned cells.

The issue is that the robot may enter a cycle and continue moving forever. Without cycle detection, the simulation never terminates.

One possible brute-force variation is to simulate for a very large number of steps and hope a cycle appears, but this is unreliable and incorrect. Since the robot can revisit cells many times, we need a mathematically guaranteed stopping condition.

### Optimal Approach

The key insight is that the robot's behavior is fully determined by:

- Its current row
- Its current column
- Its current direction

There are only:

$$m \times n \times 4$$

possible states.

Once a state repeats, the robot has entered a cycle. From that point onward, it will repeat the exact same sequence of movements forever. Therefore, no new cells can ever be cleaned after the first repeated state.

We simulate the robot while storing visited states `(row, column, direction)` in a set. We also separately track cleaned cells.

As soon as a state repeats, we stop the simulation and return the number of unique cleaned cells.

This works efficiently because each state is processed at most once.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Infinite / Unbounded | O(m × n) | Cannot reliably terminate because the robot may loop forever |
| Optimal | O(m × n) | O(m × n) | Detects repeated states to terminate safely |

## Algorithm Walkthrough

1. Initialize the robot position at `(0, 0)` and direction `0`, representing "right".
2. Create a direction array to represent movement:

- Right: `(0, 1)`
- Down: `(1, 0)`
- Left: `(0, -1)`
- Up: `(-1, 0)`
3. Create a set called `visited_states` to store tuples of:

- row
- column
- direction

This allows us to detect cycles.
4. Create another set called `cleaned_cells` to store all unique cells visited by the robot.
5. Start a simulation loop:

- Add the current state `(row, col, direction)` to `visited_states`
- Add `(row, col)` to `cleaned_cells`
6. Compute the next position using the current direction.
7. Check whether the next position is:

- outside the room, or
- an obstacle

If either condition is true, rotate clockwise:

- `direction = (direction + 1) % 4`
8. Otherwise, move the robot to the next cell.
9. Before processing the next iteration, check whether the new `(row, col, direction)` state already exists in `visited_states`.
10. If the state already exists, terminate the simulation because the robot has entered a repeating cycle.
11. Return the size of `cleaned_cells`.

### Why it works

The algorithm works because the robot is deterministic. If the robot ever reaches the same position with the same direction again, every future movement will exactly repeat the previous sequence from that point onward.

Since there are only finitely many possible states, the robot must eventually revisit a state. Therefore, the simulation always terminates, and every cleaned cell is counted exactly once.

## Python Solution

```python
from typing import List, Set, Tuple

class Solution:
    def numberOfCleanRooms(self, room: List[List[int]]) -> int:
        rows = len(room)
        cols = len(room[0])

        # Right, Down, Left, Up
        directions = [
            (0, 1),
            (1, 0),
            (0, -1),
            (-1, 0)
        ]

        row = 0
        col = 0
        direction = 0

        visited_states: Set[Tuple[int, int, int]] = set()
        cleaned_cells: Set[Tuple[int, int]] = set()

        while (row, col, direction) not in visited_states:
            visited_states.add((row, col, direction))
            cleaned_cells.add((row, col))

            dr, dc = directions[direction]
            next_row = row + dr
            next_col = col + dc

            if (
                next_row < 0
                or next_row >= rows
                or next_col < 0
                or next_col >= cols
                or room[next_row][next_col] == 1
            ):
                direction = (direction + 1) % 4
            else:
                row = next_row
                col = next_col

        return len(cleaned_cells)
```

The implementation closely follows the simulation described in the algorithm walkthrough.

The `directions` array encodes the four possible movement directions in clockwise order. This makes rotation simple because we can move to the next direction using modulo arithmetic.

The `visited_states` set is the key optimization. Each entry contains the robot's position and orientation. If the same state appears again, we know the robot's behavior will repeat forever.

The `cleaned_cells` set stores unique visited coordinates. Even if the robot revisits a location multiple times, the set guarantees it is counted only once.

Inside the loop, we first record the current state and cleaned cell. Then we attempt to move forward. If the move is blocked by either a wall or obstacle, we rotate clockwise. Otherwise, we advance into the next cell.

The loop condition ensures the simulation stops immediately when a repeated state is detected.

## Go Solution

```go
package main

func numberOfCleanRooms(room [][]int) int {
	rows := len(room)
	cols := len(room[0])

	// Right, Down, Left, Up
	directions := [][]int{
		{0, 1},
		{1, 0},
		{0, -1},
		{-1, 0},
	}

	row, col, direction := 0, 0, 0

	visitedStates := make(map[[3]int]bool)
	cleanedCells := make(map[[2]int]bool)

	for !visitedStates[[3]int{row, col, direction}] {
		visitedStates[[3]int{row, col, direction}] = true
		cleanedCells[[2]int{row, col}] = true

		dr := directions[direction][0]
		dc := directions[direction][1]

		nextRow := row + dr
		nextCol := col + dc

		if nextRow < 0 ||
			nextRow >= rows ||
			nextCol < 0 ||
			nextCol >= cols ||
			room[nextRow][nextCol] == 1 {

			direction = (direction + 1) % 4
		} else {
			row = nextRow
			col = nextCol
		}
	}

	return len(cleanedCells)
}
```

The Go solution uses maps instead of Python sets. Since Go does not have a built-in set type, we represent sets with maps whose values are booleans.

We use fixed-size arrays as map keys:

- `[3]int` for `(row, col, direction)`
- `[2]int` for cleaned cells

This works well because arrays in Go are comparable and can be used directly as map keys.

Otherwise, the logic is identical to the Python implementation.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m × n) | Each state `(row, col, direction)` is processed at most once |
| Space | O(m × n) | The visited state set and cleaned cell set store bounded states |

There are at most `4 × m × n` possible robot states because each cell can be visited facing four directions. Once a state repeats, the simulation terminates. Therefore, the algorithm performs at most `O(m × n)` iterations.

The space usage is also linear because we store visited states and cleaned cells.

## Edge Cases

### Single Cell Room

A `1 x 1` grid is the smallest possible input. The robot starts at `(0,0)` and cannot move in any direction because every move would leave the grid.

A buggy implementation might accidentally enter an infinite loop if it only tracks positions and not directions. Our implementation correctly tracks `(row, col, direction)`, so after four rotations the state repeats and the simulation terminates.

### Robot Immediately Blocked

Consider:

```
[
  [0,1],
  [1,0]
]
```

The robot cannot move right or down, and left/up are outside the grid. It simply rotates forever in place.

This case is important because the robot never changes position. If we only tracked positions, we would incorrectly terminate too early or miss cycles. Tracking full states ensures correctness.

### Large Open Room

In a large room without obstacles, the robot repeatedly traverses the perimeter and never enters many interior cells.

A naive assumption might be that all empty cells become cleaned eventually. This example proves otherwise. Our simulation accurately follows the robot's deterministic movement rules and only counts truly visited cells.
