## Solution

## Problem Understanding

This problem asks us to calculate the **minimum total cost** for a robot to move from its starting position to its home in a 2D grid, where moving into a row or column incurs a specified cost. We are given a `startPos` and a `homePos` which indicate the robot's initial and target cells, respectively. The robot can move **up**, **down**, **left**, or **right**, but cannot leave the grid. The cost of moving depends on the destination row or column: moving vertically costs `rowCosts[r]` for row `r`, and moving horizontally costs `colCosts[c]` for column `c`.

The inputs represent:

- `startPos` - the initial cell coordinates `[startrow, startcol]`
- `homePos` - the target cell coordinates `[homerow, homecol]`
- `rowCosts` - an array specifying the cost of moving into each row
- `colCosts` - an array specifying the cost of moving into each column

The output is the minimum total cost to reach `homePos` from `startPos`.

Constraints indicate that the grid can be **very large** (`m, n ≤ 10^5`) but each movement is deterministic and independent. This allows a **greedy approach** rather than full grid exploration. Key edge cases include the robot already being at home (cost 0), movement only along a row or column, and cost arrays containing zeros or maximum values.

## Approaches

### Brute Force Approach

A brute-force approach would involve simulating all possible paths from the starting position to the home position and summing the costs for each. This could be implemented using BFS or DFS with memoization to explore the grid. While this guarantees correctness, it is **too slow** for large grids because the number of paths grows exponentially with distance, and the constraints allow grids of size up to 10^5 in either dimension.

### Optimal Approach

The key observation is that moving **directly along the rows first or the columns first** (or a combination of the two in order) will always yield the minimum cost. Because costs are associated with entering a specific row or column and there are no obstacles, we can **sum the row costs for each row traversed and the column costs for each column traversed** between the start and home positions. This reduces the problem to simply iterating over the row and column ranges and summing the corresponding costs.

| --- | --- | --- | --- |
| Brute Force | O(2^(m+n)) | O(m*n) | Explore all paths using BFS/DFS |
| Optimal | O( | homerow - startrow | + |

## Algorithm Walkthrough

1. Extract the starting and home coordinates as `(sr, sc)` and `(hr, hc)`.
2. Initialize a variable `totalCost` to 0.
3. Determine the direction to move along rows. If `hr > sr`, iterate from `sr + 1` to `hr` inclusive; otherwise iterate from `sr - 1` down to `hr` inclusive. For each row `r` in this range, add `rowCosts[r]` to `totalCost`.
4. Determine the direction to move along columns. If `hc > sc`, iterate from `sc + 1` to `hc` inclusive; otherwise iterate from `sc - 1` down to `hc` inclusive. For each column `c` in this range, add `colCosts[c]` to `totalCost`.
5. Return `totalCost`.

**Why it works:** This works because the cost of moving into a row or column is independent of the path taken. Therefore, moving straight along the rows and columns between the start and home positions is always optimal. The greedy choice at each step (move directly towards the target) guarantees minimum total cost.

## Python Solution

```python
from typing import List

class Solution:
    def minCost(self, startPos: List[int], homePos: List[int], rowCosts: List[int], colCosts: List[int]) -> int:
        sr, sc = startPos
        hr, hc = homePos
        totalCost = 0
        
        # Move vertically
        if hr > sr:
            for r in range(sr + 1, hr + 1):
                totalCost += rowCosts[r]
        else:
            for r in range(sr - 1, hr - 1, -1):
                totalCost += rowCosts[r]
        
        # Move horizontally
        if hc > sc:
            for c in range(sc + 1, hc + 1):
                totalCost += colCosts[c]
        else:
            for c in range(sc - 1, hc - 1, -1):
                totalCost += colCosts[c]
        
        return totalCost
```

**Implementation Explanation:** We separate vertical and horizontal moves because their costs are handled independently. We iterate only over the rows and columns that need to be traversed, avoiding unnecessary computations. Edge cases like starting at home automatically result in 0 because the loops do not execute.

## Go Solution

```go
func minCost(startPos []int, homePos []int, rowCosts []int, colCosts []int) int {
    sr, sc := startPos[0], startPos[1]
    hr, hc := homePos[0], homePos[1]
    totalCost := 0

    // Move vertically
    if hr > sr {
        for r := sr + 1; r <= hr; r++ {
            totalCost += rowCosts[r]
        }
    } else {
        for r := sr - 1; r >= hr; r-- {
            totalCost += rowCosts[r]
        }
    }

    // Move horizontally
    if hc > sc {
        for c := sc + 1; c <= hc; c++ {
            totalCost += colCosts[c]
        }
    } else {
        for c := sc - 1; c >= hc; c-- {
            totalCost += colCosts[c]
        }
    }

    return totalCost
}
```

**Go Implementation Notes:** The logic mirrors the Python version. We use slices for inputs and integer iteration over ranges. Go handles index bounds safely if inputs are guaranteed by constraints. There is no need for additional space beyond a few variables.

## Worked Examples

**Example 1:** `startPos = [1, 0]`, `homePos = [2, 3]`, `rowCosts = [5, 4, 3]`, `colCosts = [8, 2, 6, 7]`

1. Move down from row 1 → row 2: `totalCost += 3` → `totalCost = 3`
2. Move right from col 0 → col 1: `totalCost += 2` → `totalCost = 5`
3. Move right from col 1 → col 2: `totalCost += 6` → `totalCost = 11`
4. Move right from col 2 → col 3: `totalCost += 7` → `totalCost = 18`

Result: 18

**Example 2:** `startPos = [0, 0]`, `homePos = [0, 0]`, `rowCosts = [5]`, `colCosts = [26]`

- No movement occurs; loops do not execute. `totalCost = 0`

Result: 0
