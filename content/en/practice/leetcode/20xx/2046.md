## Solution

## Problem Understanding

The problem gives us a singly linked list where the nodes are sorted according to the absolute values of their contents, not according to the actual signed values.

This distinction is extremely important.

For example, consider the list:

```
[0, 2, -5, 5, 10, -10]
```

If we look at the absolute values, we get:

```
[0, 2, 5, 5, 10, 10]
```

These are in non-decreasing order, which satisfies the problem condition.

However, the actual signed values are not sorted:

```
0, 2, -5, 5, 10, -10
```

The correct sorted order using the real values should be:

```
-10, -5, 0, 2, 5, 10
```

The task is to rearrange the linked list so that the nodes become sorted by their true values while preserving all nodes.

The input is the head of a singly linked list. We are expected to return the head of the reordered list.

The constraints are important:

- Up to `10^5` nodes
- Node values range from `-5000` to `5000`
- The list is already sorted by absolute value

The large input size means we should avoid algorithms worse than `O(n log n)` if possible. The follow-up explicitly asks whether we can achieve `O(n)` time.

The crucial observation is that because the list is sorted by absolute value, all negative numbers appear in reverse order relative to where they belong in a properly sorted list.

For example:

```
Absolute-value order:
1 -> -2 -> -3 -> 4

Actual sorted order:
-3 -> -2 -> 1 -> 4
```

Notice that the negative values naturally appear in decreasing order as we traverse the list. If we reverse the sequence of negative nodes, the entire list becomes correctly sorted.

Several edge cases are important:

- A list with only positive numbers, already correctly sorted
- A list with only negative numbers
- A single-node list
- Multiple negative numbers interleaved among positive numbers because of absolute-value ordering
- Duplicate absolute values such as `-5` and `5`

The problem guarantees the absolute-value ordering property, which makes the linear-time solution possible.

## Approaches

### Brute Force Approach

The most straightforward solution is to extract all node values into an array, sort the array normally, and then rebuild or overwrite the linked list.

The algorithm works as follows:

1. Traverse the linked list and store all values in an array.
2. Sort the array using a standard sorting algorithm.
3. Traverse the linked list again and overwrite node values using the sorted array.

This works because sorting the actual values directly guarantees the correct order.

However, this approach ignores the special structure of the input. It uses an `O(n log n)` sorting step even though the list already contains useful ordering information.

For `10^5` nodes, this is acceptable but not optimal.

### Key Insight for the Optimal Approach

The list is already sorted by absolute value.

That means:

- Positive numbers already appear in correct relative order.
- Negative numbers appear in reverse order relative to where they belong.

Consider this example:

```
0 -> 2 -> -5 -> 5 -> 10 -> -10
```

The negative numbers encountered during traversal are:

```
-5 -> -10
```

But in a correctly sorted list, they should appear as:

```
-10 -> -5
```

This means we only need to reverse the order of the negative nodes.

A very elegant way to do this is:

- Traverse the list once.
- Whenever we encounter a negative node, remove it from its current position and insert it at the front of the list.

Because negative values appear in decreasing order of magnitude during traversal, inserting each negative node at the front automatically reverses them into the correct ascending order.

This produces an `O(n)` time solution with `O(1)` extra space.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) | O(n) | Extract values, sort array, rebuild list |
| Optimal | O(n) | O(1) | Move negative nodes to front during traversal |

## Algorithm Walkthrough

### Optimal In-Place Reordering Algorithm

1. Start with two pointers:

- `prev`, initially pointing to the head
- `curr`, initially pointing to the second node

We keep `prev` because we may need to remove `curr` from the middle of the list.
2. Traverse the list while `curr` is not `None`.
3. If `curr.val` is non-negative, leave it where it is.

Positive values are already in correct order because the list is sorted by absolute value.
4. If `curr.val` is negative:

- Remove `curr` from its current position by setting:

```
prev.next = curr.next
```

- Insert `curr` at the front of the list:

```
curr.next = head
head = curr
```

This effectively reverses the order of negative nodes.
5. After moving a negative node:

- `curr` should become `prev.next`

because the current node has been removed.
6. If the node was not moved:

- Advance both pointers normally.
7. Continue until the end of the list.
8. Return the updated `head`.

### Why it works

Because the list is sorted by absolute value, every newly encountered negative number has magnitude greater than or equal to all previously seen negative numbers.

For example:

```
-1, -3, -5
```

appear in that order during traversal.

Their correct sorted order should be:

```
-5, -3, -1
```

By repeatedly inserting negative nodes at the front, we reverse their order exactly once. The positive nodes remain untouched and already sorted. Therefore, the final list becomes fully sorted in non-decreasing order.

## Python Solution

```python
# Definition for singly-linked list.
# class ListNode:
#     def __init__(self, val=0, next=None):
#         self.val = val
#         self.next = next

from typing import Optional

class Solution:
    def sortLinkedList(self, head: Optional[ListNode]) -> Optional[ListNode]:
        if not head or not head.next:
            return head

        prev = head
        curr = head.next

        while curr:
            if curr.val < 0:
                # Remove curr from its current position
                prev.next = curr.next

                # Move curr to the front
                curr.next = head
                head = curr

                # Continue traversal
                curr = prev.next
            else:
                prev = curr
                curr = curr.next

        return head
```

The implementation begins by handling trivial cases. If the list is empty or contains only one node, it is already sorted.

The algorithm then uses two pointers:

- `prev` tracks the node before `curr`
- `curr` is the node currently being inspected

When a negative node is found, it is detached from its current location and inserted at the head of the list.

The key detail is that after removing `curr`, we do not advance `prev`. This is because `prev.next` now points to the next unprocessed node.

Positive nodes remain untouched because their ordering is already valid.

The algorithm modifies the linked list in place and does not allocate additional data structures.

## Go Solution

```go
/**
 * Definition for singly-linked list.
 * type ListNode struct {
 *     Val int
 *     Next *ListNode
 * }
 */
func sortLinkedList(head *ListNode) *ListNode {
    if head == nil || head.Next == nil {
        return head
    }

    prev := head
    curr := head.Next

    for curr != nil {
        if curr.Val < 0 {
            // Remove curr from current position
            prev.Next = curr.Next

            // Move curr to front
            curr.Next = head
            head = curr

            // Continue traversal
            curr = prev.Next
        } else {
            prev = curr
            curr = curr.Next
        }
    }

    return head
}
```

The Go implementation follows the exact same logic as the Python version.

Go uses `nil` instead of `None`, and pointer manipulation is explicit through struct fields.

No additional slices or arrays are created, so the solution maintains constant extra space.

Because node values are small and no arithmetic beyond comparisons occurs, integer overflow is not a concern.

## Edge Cases

### Single-Node List

A list containing only one node is already sorted regardless of whether the value is positive or negative.

This case can easily cause pointer-related bugs if the implementation assumes the existence of `head.next`.

The solution explicitly checks:

```
if not head or not head.next:
    return head
```

which safely handles both empty and single-node lists.

### All Negative Values

Consider:

```
-1 -> -2 -> -3 -> -4
```

This is sorted by absolute value but completely reversed relative to actual values.

A naive implementation might incorrectly assume the list is nearly sorted and fail to reposition all nodes.

The front-insertion technique naturally reverses the order of all negative nodes, producing:

```
-4 -> -3 -> -2 -> -1
```

which is correct.

### No Negative Values

Consider:

```
1 -> 2 -> 3 -> 4
```

The list is already correctly sorted.

Some implementations might unnecessarily rearrange nodes or accidentally corrupt pointers.

This solution leaves positive nodes untouched and simply advances traversal pointers normally, preserving the original structure exactly.
