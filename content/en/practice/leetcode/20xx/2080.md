## Solution

## Problem Understanding

The problem asks us to design a data structure that can efficiently answer frequency queries on subarrays.

We are given an integer array `arr`, and we must support repeated queries of the form:

- Given `left`, `right`, and `value`
- Return how many times `value` appears in the subarray `arr[left:right+1]`

The key detail is that there can be up to `10^5` queries, and the array itself can also contain up to `10^5` elements. Because of these large constraints, repeatedly scanning portions of the array for every query would become too slow.

For example, suppose:

```
arr = [12, 33, 4, 56, 22, 2, 34, 33, 22, 12, 34, 56]
```

If we query:

```
query(0, 11, 33)
```

we need to count how many times `33` appears between indices `0` and `11`, inclusive. The answer is `2`.

The constraints tell us several important things:

- The array size can reach `100000`
- The number of queries can also reach `100000`
- Query performance must therefore be much faster than linear time
- The values themselves are relatively small, up to `10^4`, which suggests grouping or indexing values could be useful

A naive solution that scans every element between `left` and `right` for each query would take:

```
O(right - left + 1)
```

per query, which becomes too expensive in the worst case:

```
100000 queries × 100000 elements = 10^10 operations
```

That is far beyond acceptable limits.

Several edge cases are important:

- The queried value may not exist in the array at all
- The queried range may contain exactly one element
- The entire array may consist of the same value
- The queried value may appear many times throughout the array but only a few times inside the requested range
- The range may cover the entire array

The problem guarantees that:

- `left <= right`
- All indices are valid
- The array is non-empty

This means we do not need to handle invalid input cases.

## Approaches

### Brute Force Approach

The most direct solution is to process each query independently by scanning the subarray from `left` to `right`.

For every index in that range:

- Check whether `arr[i] == value`
- If yes, increment a counter

At the end of the scan, return the counter.

This approach is correct because it explicitly examines every element in the requested subarray and counts exactly how many match the target value.

However, it is too slow for the constraints. In the worst case, each query scans nearly the entire array, leading to `O(n)` work per query. With up to `10^5` queries, the total runtime becomes `O(n × q)`, which can reach `10^10`.

### Optimal Approach

The key observation is that we do not actually need to scan subarrays repeatedly.

Instead, we can preprocess the array by storing all positions where each value appears.

For example:

```
arr = [12, 33, 4, 56, 22, 2, 34, 33]
```

We can build:

```
12 -> [0]
33 -> [1, 7]
4  -> [2]
56 -> [3]
22 -> [4]
2  -> [5]
34 -> [6]
```

Now consider the query:

```
query(1, 7, 33)
```

Instead of scanning the array, we only examine the list:

```
[1, 7]
```

We need to determine how many indices fall within `[1, 7]`.

Because the indices are naturally stored in sorted order, we can use binary search:

- Find the first occurrence >= `left`
- Find the first occurrence > `right`

The difference between these two positions gives the count.

This reduces query time to logarithmic complexity.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) per query | O(1) | Scans the subarray directly |
| Optimal | O(log n) per query | O(n) | Uses index lists and binary search |

## Algorithm Walkthrough

### Step 1: Build a Mapping from Value to Indices

During initialization, create a hash map where:

- The key is a number from the array
- The value is a sorted list of indices where that number appears

For example:

```
arr = [1, 2, 1, 3, 1]
```

becomes:

```
1 -> [0, 2, 4]
2 -> [1]
3 -> [3]
```

We use a hash map because it allows fast lookup of a value's index list.

### Step 2: Process a Query

Suppose we receive:

```
query(left, right, value)
```

First, check whether `value` exists in the map.

If it does not exist, return `0` immediately.

### Step 3: Retrieve the Sorted Index List

Get the list of indices associated with `value`.

Example:

```
value = 1
indices = [0, 2, 4]
```

### Step 4: Use Binary Search to Find the Left Boundary

We want the first position in the list where the index is greater than or equal to `left`.

This is done using `bisect_left`.

Example:

```
indices = [0, 2, 4]
left = 1
```

The first valid position is index `1` in the list, corresponding to actual array index `2`.

### Step 5: Use Binary Search to Find the Right Boundary

Now find the first position where the index is strictly greater than `right`.

This is done using `bisect_right`.

Example:

```
right = 3
```

The insertion point becomes position `2`.

### Step 6: Compute the Frequency

The number of valid occurrences is:

```
right_position - left_position
```

In the example:

```
2 - 1 = 1
```

So the frequency is `1`.

### Why it works

The preprocessing step stores all occurrences of each value in sorted order. Since indices are inserted in increasing order while traversing the array, the lists remain sorted automatically.

Binary search allows us to efficiently locate:

- The first occurrence inside the range
- The first occurrence outside the range

The difference between these positions exactly equals the number of occurrences within the query interval.

## Python Solution

```python
from collections import defaultdict
from bisect import bisect_left, bisect_right
from typing import List

class RangeFreqQuery:

    def __init__(self, arr: List[int]):
        self.positions = defaultdict(list)

        for index, value in enumerate(arr):
            self.positions[value].append(index)

    def query(self, left: int, right: int, value: int) -> int:
        if value not in self.positions:
            return 0

        indices = self.positions[value]

        left_pos = bisect_left(indices, left)
        right_pos = bisect_right(indices, right)

        return right_pos - left_pos

# Your RangeFreqQuery object will be instantiated and called as such:
# obj = RangeFreqQuery(arr)
# param_1 = obj.query(left, right, value)
```

The implementation begins by creating a `defaultdict(list)` called `positions`. This structure maps each value to all indices where it appears.

During initialization, we iterate through the array once using `enumerate`. Each index is appended to the corresponding value's list.

The `query` method first checks whether the target value exists in the map. If not, the answer is immediately `0`.

Otherwise, we retrieve the sorted index list for that value.

The `bisect_left` call finds the first index position that is greater than or equal to `left`.

The `bisect_right` call finds the first index position strictly greater than `right`.

Subtracting these two insertion points gives the total number of valid occurrences inside the requested range.

## Go Solution

```go
package main

import "sort"

type RangeFreqQuery struct {
	positions map[int][]int
}

func Constructor(arr []int) RangeFreqQuery {
	positions := make(map[int][]int)

	for index, value := range arr {
		positions[value] = append(positions[value], index)
	}

	return RangeFreqQuery{
		positions: positions,
	}
}

func (this *RangeFreqQuery) Query(left int, right int, value int) int {
	indices, exists := this.positions[value]

	if !exists {
		return 0
	}

	leftPos := sort.SearchInts(indices, left)
	rightPos := sort.Search(len(indices), func(i int) bool {
		return indices[i] > right
	})

	return rightPos - leftPos
}

/**
 * Your RangeFreqQuery object will be instantiated and called as such:
 * obj := Constructor(arr);
 * param_1 := obj.Query(left,right,value);
 */
```

The Go implementation follows the same logic as the Python version.

Instead of Python's `bisect` module, Go uses:

- `sort.SearchInts` for the lower bound
- `sort.Search` with a custom predicate for the upper bound

Go slices naturally maintain insertion order, so the index lists remain sorted automatically during preprocessing.

Unlike Python dictionaries, Go maps require checking whether a key exists using the two-value assignment syntax:

```go
indices, exists := this.positions[value]
```

If the value does not exist, the function returns `0`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(log n) per query | Binary search on occurrence lists |
| Space | O(n) | Stores every array index once |

The preprocessing step takes `O(n)` time because we traverse the array once.

Each query performs two binary searches on the occurrence list for the target value. In the worst case, a value may appear `n` times, so each binary search costs `O(log n)`.

The space complexity is `O(n)` because every array index is stored exactly once across all lists.

## Edge Cases

### Value Does Not Exist in the Array

A common source of bugs occurs when the queried value is completely absent from the array.

For example:

```
arr = [1, 2, 3]
query(0, 2, 5)
```

If the implementation directly attempts binary search on a missing list, it may raise an error or behave incorrectly.

The solution handles this safely by checking:

```
if value not in self.positions:
    return 0
```

This guarantees correct behavior for nonexistent values.

### Query Range Contains Only One Element

Single-element ranges are easy to mishandle because the left and right boundaries are identical.

Example:

```
query(3, 3, value)
```

The implementation still works because binary search correctly identifies whether the single index exists in the occurrence list.

The subtraction:

```
right_pos - left_pos
```

naturally becomes either `1` or `0`.

### All Elements Are the Same

Consider:

```
arr = [7, 7, 7, 7, 7]
```

Every query targets a very large occurrence list.

A naive implementation could degrade badly here if it scanned ranges directly.

The binary search approach remains efficient because it only performs logarithmic operations, regardless of how large the query range becomes.

The sorted occurrence list:

```
[0, 1, 2, 3, 4]
```

allows immediate calculation of how many indices lie inside any interval.
