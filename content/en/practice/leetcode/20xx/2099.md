## Solution

## Problem Understanding

The problem gives us an integer array `nums` and an integer `k`. We must select exactly `k` elements from the array such that their total sum is as large as possible. However, there is one important restriction: the chosen elements must form a subsequence.

A subsequence means we are allowed to remove elements from the original array, but the relative order of the remaining elements must stay the same. This is different from simply choosing the largest `k` values and returning them in sorted order. The returned array must preserve the original ordering from `nums`.

For example, if `nums = [3,4,3,3]` and `k = 2`, the two largest values are `4` and `3`. The valid subsequences are `[3,4]` or `[4,3]`, depending on which `3` we choose. Both preserve the original ordering of the selected indices.

The input constraints are relatively small:

- `1 <= nums.length <= 1000`
- `-10^5 <= nums[i] <= 10^5`
- `1 <= k <= nums.length`

Since the array size is only up to 1000, even an `O(n log n)` solution is easily fast enough. The constraints also tell us that negative numbers are allowed, so we cannot assume that taking more elements always improves the sum. However, because we must select exactly `k` elements, we simply want the `k` largest values overall, regardless of sign.

Several edge cases are important:

- Arrays containing duplicate values
- Arrays with all negative numbers
- Cases where `k = 1`
- Cases where `k = nums.length`
- Situations where multiple optimal subsequences exist

The problem guarantees that at least one valid answer always exists because `k` is always between `1` and `nums.length`.

## Approaches

### Brute Force Approach

The brute force solution is to generate every possible subsequence of length `k`, compute the sum of each subsequence, and keep the one with the maximum sum.

This works because it exhaustively checks all valid possibilities. Since every subsequence of length `k` is considered, the optimal one must eventually be found.

However, this approach becomes very expensive. The number of subsequences of size `k` is:

$$\binom{n}{k}$$

In the worst case, this grows exponentially. Even with `n = 1000`, generating all combinations is completely impractical.

### Optimal Approach

The key observation is that the subsequence with the maximum sum must contain the `k` largest values from the array.

The only complication is preserving the original order of those selected elements.

The strategy is:

1. Pair each number with its original index.
2. Sort the pairs by value in descending order.
3. Take the first `k` pairs, since these correspond to the `k` largest numbers.
4. Sort those selected pairs by their original indices.
5. Extract the values in index order to produce a valid subsequence.

This works because maximizing the sum only depends on selecting the largest values, while the subsequence requirement only affects the final ordering.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(C(n,k) × k) | O(k) | Generates every subsequence of length `k` |
| Optimal | O(n log n) | O(n) | Sorts values and restores original order |

## Algorithm Walkthrough

1. Create a list of pairs where each pair contains:

- the value from `nums`
- its original index

For example:

```
nums = [2,1,3,3]
pairs = [(2,0), (1,1), (3,2), (3,3)]
```

Keeping the index is important because we eventually need to reconstruct a subsequence in original order.
2. Sort the pairs by value in descending order.

After sorting:

```
[(3,2), (3,3), (2,0), (1,1)]
```

Now the largest elements appear first.
3. Take the first `k` pairs.

If `k = 2`:

```
[(3,2), (3,3)]
```

These are the `k` largest values, so they produce the maximum possible sum.
4. Sort the selected pairs by their original indices.

This restores subsequence order.

```
[(3,2), (3,3)]
```

In other examples, the order may change here.
5. Extract only the values from the selected pairs.

```
[3,3]
```

This is the final subsequence.

### Why it works

The algorithm is correct because the maximum possible sum is achieved by selecting the `k` largest values in the array. Any other choice would replace at least one selected value with a smaller one, reducing the total sum.

After choosing the correct values, sorting by original index guarantees the resulting sequence preserves the order from the original array, which satisfies the definition of a subsequence.

## Python Solution

```python
from typing import List

class Solution:
    def maxSubsequence(self, nums: List[int], k: int) -> List[int]:
        indexed_nums = [(value, index) for index, value in enumerate(nums)]

        indexed_nums.sort(key=lambda x: x[0], reverse=True)

        selected = indexed_nums[:k]

        selected.sort(key=lambda x: x[1])

        return [value for value, index in selected]
```

The implementation begins by pairing every number with its original index using `enumerate`. This allows us to track where each value came from in the original array.

Next, the list is sorted in descending order based on the values. The first `k` elements after sorting are therefore the `k` largest numbers.

However, selecting elements this way destroys the original ordering. To restore subsequence order, the chosen pairs are sorted again, this time using their original indices.

Finally, the values are extracted and returned as the result.

## Go Solution

```go
package main

import "sort"

func maxSubsequence(nums []int, k int) []int {
	type Pair struct {
		value int
		index int
	}

	pairs := make([]Pair, 0, len(nums))

	for i, value := range nums {
		pairs = append(pairs, Pair{value, i})
	}

	sort.Slice(pairs, func(i, j int) bool {
		return pairs[i].value > pairs[j].value
	})

	selected := pairs[:k]

	sort.Slice(selected, func(i, j int) bool {
		return selected[i].index < selected[j].index
	})

	result := make([]int, 0, k)

	for _, pair := range selected {
		result = append(result, pair.value)
	}

	return result
}
```

The Go implementation follows the same logic as the Python version. A custom `Pair` struct stores both the value and original index.

The `sort.Slice` function is used twice:

- First to sort by value descending
- Then to sort selected elements by index ascending

Slices in Go are references to underlying arrays, so `selected := pairs[:k]` does not create a copy. This is perfectly fine here because we only need to reorder the selected section.

Integer overflow is not a concern because we never compute large arithmetic values beyond the problem constraints.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | Sorting dominates the runtime |
| Space | O(n) | Extra storage for indexed pairs |

The algorithm performs two sorting operations. The first sort processes all `n` elements, which costs `O(n log n)`. The second sort only processes `k` elements, which costs `O(k log k)`. Since `k <= n`, the overall complexity remains `O(n log n)`.

The extra space comes from storing value-index pairs and the selected subsequence.

## Edge Cases

One important edge case occurs when all numbers are negative. A common mistake is assuming that negative values should be avoided. However, the problem requires selecting exactly `k` elements. The algorithm correctly handles this by still choosing the largest values numerically, even if they are negative.

Another important case is when duplicate values exist. If multiple elements have the same value, we must still preserve subsequence order. Because the algorithm stores original indices and reorders selected elements by index at the end, duplicates are handled correctly and consistently.

A third edge case is when `k` equals the length of the array. In this situation, every element must be selected. The algorithm naturally handles this because taking the top `k` values simply includes all elements, and sorting by original index reconstructs the original array unchanged.

A final edge case is when `k = 1`. Here we only need the single maximum value. The algorithm still works because sorting places the largest value first, and selecting one element produces the correct subsequence.
