**Difficulty:** 🟡 Medium  
**Topics:** Array, Binary Search, Sorting

## Solution

## Problem Understanding

The problem gives us a list of items, where each item has two values:

- `price`
- `beauty`

Each item is represented as:

```
[price, beauty]
```

We are also given a list of queries. For every query value `q`, we must determine:

> What is the maximum beauty among all items whose price is less than or equal to `q`?

If no item satisfies the price constraint, the answer for that query is `0`.

The result must be returned as an array where the `jth` position corresponds to the answer for `queries[j]`.

For example, suppose we have:

```
items = [[1,2],[2,4],[5,6]]
query = 3
```

The valid items are:

```
[1,2]
[2,4]
```

because their prices are less than or equal to `3`.

Among those, the maximum beauty is `4`, so the answer for that query is `4`.

The important detail is that queries are independent. We are not removing items after answering a query. Every query examines the full set of items again.

The constraints are very important:

- Up to `10^5` items
- Up to `10^5` queries
- Prices and beauties can be as large as `10^9`

A naive solution that scans every item for every query would require:

```
10^5 * 10^5 = 10^10
```

operations in the worst case, which is far too slow.

The constraints strongly suggest we need:

- sorting
- binary search
- prefix preprocessing

There are several edge cases we must handle carefully:

- Multiple items may have the same price
- Multiple items may have the same beauty
- A query may be smaller than every item price
- A query may be larger than every item price
- Items are not initially sorted
- Queries are not initially sorted

The problem guarantees that every item contains exactly two integers and that all values are positive integers.

## Approaches

### Brute Force Approach

The most direct solution is to process each query independently.

For every query:

1. Iterate through all items
2. Check whether `price <= query`
3. Track the maximum beauty among valid items
4. Store the result

This works because it explicitly checks every candidate item for every query, so it cannot miss the correct answer.

However, the time complexity is too large.

If there are `n` items and `m` queries:

- Each query scans all `n` items
- Total work becomes `O(n * m)`

With both values up to `10^5`, this becomes `10^10` operations in the worst case, which is not feasible.

### Optimal Approach

The key observation is:

> Queries only care about the maximum beauty among items up to a certain price.

If we sort items by price, then every query becomes:

> Find the rightmost item whose price is less than or equal to the query.

After sorting, we can preprocess a prefix maximum array where:

```
prefixMax[i] =
maximum beauty among items[0...i]
```

This transforms the problem into two steps:

1. Use binary search to find the last valid item
2. Read the precomputed maximum beauty directly

Binary search reduces query processing from `O(n)` to `O(log n)`.

Since sorting costs `O(n log n)` and each query costs `O(log n)`, the overall solution becomes efficient enough for the constraints.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * m) | O(1) | Scan all items for every query |
| Optimal | O(n log n + m log n) | O(n) | Sort items, build prefix maximums, answer queries with binary search |

## Algorithm Walkthrough

1. Sort the items by price.

Sorting ensures that prices appear in increasing order. This allows binary search to work correctly later.

Example:

```
items = [[3,2],[1,2],[2,4]]
```

becomes:

```
[[1,2],[2,4],[3,2]]
```
2. Build two arrays:

- `prices`
- `max_beauty`

`prices[i]` stores the sorted item price.

`max_beauty[i]` stores the maximum beauty seen from index `0` through `i`.
3. Compute prefix maximum beauties.

Suppose the sorted beauties are:

```
[2,4,2,6]
```

The prefix maximum array becomes:

```
[2,4,4,6]
```

This means:

- Up to price index `0`, best beauty is `2`
- Up to price index `1`, best beauty is `4`
- Up to price index `2`, best beauty is still `4`
- Up to price index `3`, best beauty is `6`
4. Process each query independently.

For a query `q`, use binary search to find the rightmost index where:

```
price <= q
```
5. If no valid index exists, return `0`.

This happens when the query is smaller than the smallest item price.
6. Otherwise, return `max_beauty[index]`.

Since the prefix array already stores the maximum beauty among all earlier items, this instantly gives the correct answer.

### Why it works

After sorting, all valid items for a query form a continuous prefix of the array.

Binary search correctly finds the boundary of that prefix, namely the last item whose price does not exceed the query.

The prefix maximum array guarantees that `max_beauty[i]` equals the best beauty among all valid items up to index `i`.

Therefore, once binary search finds the boundary index, the corresponding prefix maximum is exactly the answer.

## Python Solution

```python
from typing import List
import bisect

class Solution:
    def maximumBeauty(self, items: List[List[int]], queries: List[int]) -> List[int]:
        # Sort items by price
        items.sort()

        prices = []
        prefix_max_beauty = []

        current_max = 0

        # Build price array and prefix maximum beauty array
        for price, beauty in items:
            prices.append(price)

            current_max = max(current_max, beauty)
            prefix_max_beauty.append(current_max)

        answers = []

        # Process each query
        for query in queries:
            # Find rightmost index where price <= query
            index = bisect.bisect_right(prices, query) - 1

            # No valid item
            if index < 0:
                answers.append(0)
            else:
                answers.append(prefix_max_beauty[index])

        return answers
```

The implementation starts by sorting the items in ascending order of price. Python automatically sorts lists of pairs lexicographically, so sorting directly works correctly.

Next, the code builds two arrays:

- `prices`
- `prefix_max_beauty`

The `prices` array is used for binary search. The `prefix_max_beauty` array stores the maximum beauty encountered so far while iterating through the sorted items.

The variable `current_max` maintains the running maximum beauty. At each item:

```python
current_max = max(current_max, beauty)
```

This ensures the prefix array always contains the best beauty up to that position.

For each query, the solution uses:

```python
bisect.bisect_right(prices, query)
```

This returns the insertion position to keep the array sorted while placing the query value to the right of equal elements.

Subtracting `1` gives the index of the last price less than or equal to the query.

If the index becomes negative, then no valid item exists, so the answer is `0`.

Otherwise, the prefix maximum array directly provides the correct answer.

## Go Solution

```go
package main

import (
	"sort"
)

func maximumBeauty(items [][]int, queries []int) []int {
	// Sort items by price
	sort.Slice(items, func(i, j int) bool {
		return items[i][0] < items[j][0]
	})

	n := len(items)

	prices := make([]int, n)
	prefixMaxBeauty := make([]int, n)

	currentMax := 0

	// Build arrays
	for i, item := range items {
		price := item[0]
		beauty := item[1]

		prices[i] = price

		if beauty > currentMax {
			currentMax = beauty
		}

		prefixMaxBeauty[i] = currentMax
	}

	answer := make([]int, len(queries))

	// Process queries
	for i, query := range queries {
		// First index where prices[idx] > query
		index := sort.Search(len(prices), func(j int) bool {
			return prices[j] > query
		}) - 1

		if index >= 0 {
			answer[i] = prefixMaxBeauty[index]
		} else {
			answer[i] = 0
		}
	}

	return answer
}
```

The Go solution follows exactly the same algorithmic structure as the Python implementation.

The main difference is how binary search is implemented. Go uses:

```go
sort.Search
```

which finds the first index satisfying a condition. Here, we search for the first price greater than the query, then subtract one to obtain the last valid price less than or equal to the query.

Go slices are dynamic views over arrays, so no additional array handling is necessary. Integer overflow is not a concern because all values are within standard 32 bit integer limits, and Go's `int` type comfortably handles them on LeetCode environments.

## Edge Cases

One important edge case occurs when every query is smaller than the minimum item price. In this situation, binary search returns an invalid position because no item satisfies the condition `price <= query`. A buggy implementation might accidentally access index `-1` incorrectly or return an unrelated value. This implementation explicitly checks:

```
if index < 0:
    answers.append(0)
```

which correctly handles the case.

Another critical edge case involves duplicate prices. Multiple items may share the same price but have different beauties. If we only stored the latest beauty for a price instead of the maximum prefix beauty, we could return incorrect answers. The prefix maximum array guarantees that every position stores the best beauty seen so far, regardless of duplicates.

A third important edge case is when the maximum beauty appears at a smaller price than the query boundary. For example:

```
items = [[1,100],[10,1]]
query = 10
```

The correct answer is `100`, not `1`. A naive implementation that only checked the rightmost item would fail. The prefix maximum preprocessing ensures that all earlier items are considered automatically.

Another subtle edge case is unsorted input. Since the problem does not guarantee ordering, binary search would produce invalid results on the raw input. Sorting the items before preprocessing is essential for correctness.

Finally, very large input sizes require an efficient algorithm. The solution avoids quadratic behavior entirely by using sorting and binary search, making it scalable for the full constraint range of `10^5` items and queries.
