## Solution

## Problem Understanding

The problem gives us an integer array `nums` and an integer `target`. We are asked to sort the array in non-decreasing order and then return all indices where the value equals `target`.

A target index is simply a position `i` such that:

```
sortedNums[i] == target
```

The important detail is that the indices must correspond to the array _after sorting_, not the original array.

For example, if:

```
nums = [1,2,5,2,3]
target = 2
```

then after sorting we get:

```
[1,2,2,3,5]
```

The value `2` appears at indices `1` and `2`, so the answer is:

```
[1,2]
```

The input size is very small:

```
1 <= nums.length <= 100
```

This means even relatively inefficient approaches would still run comfortably within limits. However, the goal is to understand the logic clearly and write an efficient, clean solution.

The constraints also tell us that:

- Every element is positive
- The maximum array size is only 100
- Values are limited to the range `[1, 100]`

Because the input is small, sorting is perfectly acceptable.

There are several important edge cases to consider:

- The target may not exist in the array at all, in which case we must return an empty list.
- Every element could equal the target, meaning every index in the sorted array is part of the answer.
- The array may already be sorted.
- The array may contain only one element.
- Multiple duplicate values may appear before and after the target values.

A correct solution must handle all of these scenarios consistently.

## Approaches

### Brute Force Approach

The most direct solution is to first sort the array, then scan through the sorted array and collect every index where the value equals `target`.

The algorithm works like this:

1. Sort the array.
2. Iterate through every index.
3. If the current value equals `target`, append the index to the answer list.
4. Return the collected indices.

This approach is straightforward and guarantees correctness because sorting places equal values together in their final positions.

Although this is already efficient enough for the problem constraints, we can think about an alternative observation.

### Better Observation

Instead of fully relying on the sorted array traversal, we can observe an important property:

- The first target index after sorting equals the number of elements strictly smaller than `target`.
- The number of target indices equals the count of elements equal to `target`.

For example:

```
nums = [1,2,5,2,3]
target = 2
```

Elements smaller than `2`:

```
[1]
```

Count = 1

Elements equal to `2`:

```
[2,2]
```

Count = 2

Therefore, target indices must be:

```
[1,2]
```

because the targets start at index `1` and occupy `2` consecutive positions.

This avoids explicitly sorting the array and achieves linear time complexity.

| Optimal | O(n) | O(1) | Count smaller elements and occurrences directly |

## Algorithm Walkthrough

The optimal approach avoids sorting entirely.

### Step 1: Initialize Counters

Create two variables:

- `smaller_count`, counts how many elements are strictly smaller than `target`
- `target_count`, counts how many elements equal `target`

These two values completely determine the final answer.

### Step 2: Traverse the Array

Iterate through each number in `nums`.

For each value:

- If the value is smaller than `target`, increment `smaller_count`
- If the value equals `target`, increment `target_count`

We do not care about numbers greater than `target` because they do not affect the target indices.

### Step 3: Build the Result

After processing the entire array:

- The first target index equals `smaller_count`
- The targets occupy `target_count` consecutive positions

So the answer becomes:

```
[smaller_count, smaller_count + 1, ..., smaller_count + target_count - 1]
```

### Step 4: Return the Result

If `target_count` is zero, the resulting range is empty, so we correctly return an empty list.

### Why it works

After sorting, every value smaller than `target` must appear before every occurrence of `target`. Therefore, the number of smaller elements exactly determines where the first target appears.

Similarly, all occurrences of `target` become consecutive in the sorted array. If there are `k` occurrences, they occupy exactly `k` consecutive indices.

Because the algorithm computes both quantities directly, it reconstructs the correct indices without needing to sort.

## Python Solution

```python
from typing import List

class Solution:
    def targetIndices(self, nums: List[int], target: int) -> List[int]:
        smaller_count = 0
        target_count = 0

        for num in nums:
            if num < target:
                smaller_count += 1
            elif num == target:
                target_count += 1

        return list(range(smaller_count, smaller_count + target_count))
```

The implementation begins by initializing two counters.

```python
smaller_count = 0
target_count = 0
```

These track the number of elements smaller than the target and equal to the target.

Next, we iterate through the array once:

```python
for num in nums:
```

If a number is smaller than the target, it must appear before the target after sorting, so we increment `smaller_count`.

```python
if num < target:
    smaller_count += 1
```

If the number equals the target, we increment `target_count`.

```python
elif num == target:
    target_count += 1
```

Finally, we generate the target indices using Python's `range` function.

```python
return list(range(smaller_count, smaller_count + target_count))
```

If `target_count` is zero, the range is empty, which correctly returns an empty list.

## Go Solution

```go
func targetIndices(nums []int, target int) []int {
    smallerCount := 0
    targetCount := 0

    for _, num := range nums {
        if num < target {
            smallerCount++
        } else if num == target {
            targetCount++
        }
    }

    result := make([]int, 0, targetCount)

    for i := 0; i < targetCount; i++ {
        result = append(result, smallerCount+i)
    }

    return result
}
```

The Go implementation follows the same logic as the Python version.

Since Go does not have a built-in equivalent to Python's `range` list generation, we manually append each target index into the result slice.

The slice is initialized with capacity `targetCount`:

```go
result := make([]int, 0, targetCount)
```

This avoids unnecessary reallocations while appending.

Go slices naturally handle empty results, so if `targetCount` is zero, the returned slice is simply empty.

## Complexity Analysis

| --- | --- | --- |
| Time | O(n) | We scan the array exactly once |
| Space | O(1) | Only a few counters are used, excluding output |

The algorithm performs a single pass through the input array. Every operation inside the loop is constant time, so the total runtime grows linearly with the size of the array.

The extra memory usage is constant because we only store two integer counters. The returned list itself is not counted as auxiliary space.

## Edge Cases

One important edge case occurs when the target does not exist in the array. A buggy implementation might accidentally generate indices even though there are no matching elements. In this solution, `target_count` remains zero, so the generated range is empty and the algorithm correctly returns `[]`.

Another critical case is when every element equals the target. After sorting, every index should be included in the answer. The implementation handles this naturally because `smaller_count` stays zero while `target_count` becomes the full array length. The generated indices become `[0, 1, ..., n - 1]`.

A third important edge case is a single-element array. Small inputs often reveal off-by-one errors in index calculations. If the lone element equals the target, the algorithm returns `[0]`. Otherwise, it returns an empty list. The range generation logic handles both cases correctly without requiring special conditions.
