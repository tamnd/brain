# LeetCode 2047 - Number of Valid Words in a Sentence

## Problem Understanding

This problem asks us to analyze a sentence and determine how many of its tokens qualify as valid words according to a strict set of grammar-like rules.

The input is a string called `sentence`. The string may contain lowercase English letters, digits, spaces, hyphens, and punctuation characters. Tokens are separated by one or more spaces. A token is simply a contiguous sequence of non-space characters.

Our task is to examine every token independently and decide whether it is valid. A token is considered valid only if all of the following conditions hold:

1. The token cannot contain any digits.
2. The token may contain at most one hyphen.
3. If a hyphen exists, it must appear between two lowercase letters.
4. The token may contain at most one punctuation character.
5. If punctuation exists, it must appear at the end of the token.

The punctuation characters allowed are `'!'`, `'.'`, and `','`.

The output is the total number of valid tokens in the sentence.

The constraints are relatively small. The sentence length is at most 1000 characters, which means even a straightforward linear scan is more than fast enough. Since the character set is limited and the validation rules are local to each token, this problem is primarily about careful condition checking rather than advanced algorithms.

Several edge cases are important to recognize early:

- Multiple consecutive spaces can produce empty splits if handled incorrectly.
- A token containing any digit is immediately invalid.
- A hyphen at the beginning or end of a token is invalid.
- A token containing multiple hyphens is invalid.
- Punctuation in the middle of a token is invalid.
- A token consisting solely of punctuation, such as `"!"`, is valid.
- Tokens like `"a-b"` are valid because the hyphen is surrounded by lowercase letters.

A naive implementation often fails because it checks characters independently without enforcing relationships between neighboring characters, especially around hyphens and punctuation.

## Approaches

### Brute Force Approach

A brute force solution would tokenize the sentence and then test every possible rule independently using multiple scans over each token.

For every token, we could:

- Scan once to detect digits.
- Scan again to count hyphens.
- Scan again to validate hyphen placement.
- Scan again to count punctuation.
- Scan again to verify punctuation position.

This approach is correct because every rule is checked exhaustively. However, it performs unnecessary repeated passes over the same token.

Although the constraints are small enough that this would still pass, it is inefficient and more error-prone because the validation logic becomes fragmented across multiple loops.

### Optimal Approach

The key observation is that all validation rules can be checked during a single left-to-right scan of each token.

Instead of performing separate passes, we maintain simple counters and conditions while iterating through the token once:

- Reject immediately if a digit is encountered.
- Track whether a hyphen has already appeared.
- Validate that any hyphen is surrounded by lowercase letters.
- Track whether punctuation has already appeared.
- Ensure punctuation only appears at the last position.

This produces a clean linear solution with constant extra space.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Multiple scans per token |
| Optimal | O(n) | O(1) | Single pass validation per token |

Here, `n` represents the total length of the sentence.

## Algorithm Walkthrough

1. Split the sentence into tokens using spaces.
2. Ignore empty tokens, because multiple consecutive spaces may produce empty strings.
3. For each token, validate it character by character.
4. Maintain a boolean variable `has_hyphen` to ensure only one hyphen appears.
5. Maintain a boolean variable `has_punctuation` to ensure only one punctuation mark appears.
6. Iterate through every character in the token.
7. If the character is a digit, immediately mark the token invalid because digits are never allowed.
8. If the character is a hyphen:

- Reject the token if a hyphen already appeared earlier.
- Reject the token if the hyphen is at the beginning or end.
- Reject the token if the characters before and after it are not lowercase letters.
- Otherwise, mark `has_hyphen = True`.
9. If the character is punctuation:

- Reject the token if punctuation already appeared earlier.
- Reject the token if the punctuation is not the final character of the token.
- Otherwise, mark `has_punctuation = True`.
10. If the character is a lowercase letter, continue scanning.
11. If the token survives the entire scan without violating any rule, count it as valid.
12. Return the total number of valid tokens.

### Why it works

The algorithm works because every validity rule is enforced exactly when the relevant character is encountered. Digits are rejected immediately, hyphen placement is validated using neighboring characters, and punctuation positioning is checked using the current index.

At every step, the algorithm maintains the invariant that all previously scanned characters satisfy the token rules. If any rule is violated, the token is rejected immediately. Therefore, any token accepted at the end of the scan must satisfy all constraints.

## Python Solution

```python
class Solution:
    def countValidWords(self, sentence: str) -> int:
        def is_valid(token: str) -> bool:
            has_hyphen = False
            has_punctuation = False
            punctuation_marks = {"!", ".", ","}

            for index, char in enumerate(token):
                if char.isdigit():
                    return False

                if char == "-":
                    if has_hyphen:
                        return False

                    if index == 0 or index == len(token) - 1:
                        return False

                    if not token[index - 1].islower():
                        return False

                    if not token[index + 1].islower():
                        return False

                    has_hyphen = True

                elif char in punctuation_marks:
                    if has_punctuation:
                        return False

                    if index != len(token) - 1:
                        return False

                    has_punctuation = True

            return True

        valid_count = 0

        for token in sentence.split():
            if is_valid(token):
                valid_count += 1

        return valid_count
```

The implementation begins by defining a helper function called `is_valid`. This function encapsulates all token validation logic, which keeps the main method clean and easy to read.

Inside the helper function, two boolean variables track whether a hyphen or punctuation mark has already appeared. This ensures the "at most one" constraints are enforced naturally during the scan.

The loop iterates through each character along with its index. The index is important because both hyphen and punctuation rules depend on character position.

If a digit appears, the function immediately returns `False`.

When a hyphen is encountered, the implementation checks four conditions:

- Only one hyphen may exist.
- The hyphen cannot be first.
- The hyphen cannot be last.
- The surrounding characters must both be lowercase letters.

Punctuation handling is similar. Only one punctuation mark is allowed, and it must be located at the end of the token.

Finally, the main function iterates through all tokens generated by `sentence.split()`. Python's `split()` automatically handles multiple spaces correctly, which simplifies tokenization considerably.

## Go Solution

```go
func countValidWords(sentence string) int {
	isValid := func(token string) bool {
		hasHyphen := false
		hasPunctuation := false

		for i := 0; i < len(token); i++ {
			ch := token[i]

			if ch >= '0' && ch <= '9' {
				return false
			}

			if ch == '-' {
				if hasHyphen {
					return false
				}

				if i == 0 || i == len(token)-1 {
					return false
				}

				if !(token[i-1] >= 'a' && token[i-1] <= 'z') {
					return false
				}

				if !(token[i+1] >= 'a' && token[i+1] <= 'z') {
					return false
				}

				hasHyphen = true
			} else if ch == '!' || ch == '.' || ch == ',' {
				if hasPunctuation {
					return false
				}

				if i != len(token)-1 {
					return false
				}

				hasPunctuation = true
			}
		}

		return true
	}

	validCount := 0
	current := ""

	for i := 0; i < len(sentence); i++ {
		if sentence[i] == ' ' {
			if len(current) > 0 {
				if isValid(current) {
					validCount++
				}
				current = ""
			}
		} else {
			current += string(sentence[i])
		}
	}

	if len(current) > 0 {
		if isValid(current) {
			validCount++
		}
	}

	return validCount
}
```

The Go implementation follows the same logic as the Python version but handles tokenization manually. Unlike Python's built-in `split()`, Go does not automatically collapse multiple spaces unless additional utilities are used.

To avoid unnecessary imports, the solution builds tokens incrementally while scanning the sentence.

Character classification is done using direct ASCII comparisons rather than helper methods like `isdigit()` or `islower()`. This is common in Go solutions because it is efficient and straightforward.

No integer overflow concerns exist because the input size is very small.

## Edge Cases

One important edge case is tokens containing punctuation at invalid positions. For example, `"a!b"` should be rejected because punctuation may only appear at the end of the token. A careless implementation might only count punctuation occurrences without validating its location. This solution explicitly checks that punctuation appears at the final index.

Another important edge case is hyphen placement. Tokens such as `"-ab"` or `"ab-"` are invalid because a hyphen must have lowercase letters on both sides. Many incorrect solutions only count hyphens without verifying neighboring characters. This implementation checks both adjacent positions carefully before accepting the hyphen.

Multiple consecutive spaces are also a common source of bugs. If tokenization is implemented incorrectly, empty strings may accidentally be counted as valid tokens. The Python solution avoids this naturally by using `split()`, while the Go solution explicitly ignores empty tokens during parsing.

A subtle edge case involves tokens consisting entirely of punctuation, such as `"!"`. These are valid according to the problem rules because there is one punctuation mark and it appears at the end. The implementation correctly accepts these tokens because punctuation-only tokens never violate any rule.
