## Solution

## Problem Understanding

The problem gives us an array that was originally sorted in ascending order, but then rotated some number of times. A rotation shifts elements from the end of the array to the front. For example, the sorted array `[1,2,3,4,5]` becomes `[3,4,5,1,2]` after being rotated.

Our task is to find the minimum element in this rotated array.

The important detail is that the original array was sorted before rotation, and all elements are unique. Because of this, the rotated array still contains two sorted portions. The minimum value is exactly the point where the rotation occurred.

For example:

- `[4,5,6,7,0,1,2]`

- Left portion: `[4,5,6,7]`
- Right portion: `[0,1,2]`
- The smallest value is `0`

The constraints tell us that:

- The array length can be as large as 5000
- All values are unique
- We must solve the problem in `O(log n)` time

The `O(log n)` requirement immediately suggests binary search. A linear scan would work functionally, but it would take `O(n)` time, which does not satisfy the requirement.

Several edge cases are important:

- The array may not actually appear rotated, such as `[11,13,15,17]`
- The array may contain only one element
- The minimum element may appear at the very end or near the beginning
- Since all values are unique, we never need to handle equal comparisons

The uniqueness guarantee simplifies the binary search logic significantly.

## Approaches

### Brute Force Approach

The simplest solution is to scan through the entire array and keep track of the smallest element seen so far.

For example, in `[4,5,6,7,0,1,2]`, we would:

- Start with minimum = 4
- Compare each remaining value
- Update minimum when we encounter `0`

This approach is correct because every element is examined exactly once, guaranteeing that the smallest value will eventually be found.

However, this solution requires traversing the whole array, resulting in `O(n)` time complexity. Since the problem explicitly requires `O(log n)` time, we need something more efficient.

### Optimal Binary Search Approach

The key observation is that the array is still mostly sorted.

In a rotated sorted array:

- One half of the array is always properly sorted
- The minimum element lies in the unsorted portion, or at the boundary between the two portions

Consider:

```
[4,5,6,7,0,1,2]
```

If we examine the middle element:

```
mid = 7
right = 2
```

Since `7 > 2`, the minimum must lie to the right of `mid`.

Now consider:

```
[0,1,2,4,5,6,7]
```

If:

```
mid = 4
right = 7
```

Since `4 < 7`, the right half is sorted, meaning the minimum must lie at `mid` or to the left.

This allows us to repeatedly discard half the search space, exactly like standard binary search.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Scan every element and track the minimum |
| Optimal | O(log n) | O(1) | Binary search using sorted half properties |

## Algorithm Walkthrough

1. Initialize two pointers, `left` and `right`, representing the current search range.
2. While `left < right`, compute the middle index:

```
mid = (left + right) // 2
```

This divides the search space into two halves.
3. Compare `nums[mid]` with `nums[right]`.

This comparison tells us which side contains the minimum.
4. If `nums[mid] > nums[right]`, the minimum must be to the right of `mid`.

For example:

```
[4,5,6,7,0,1,2]
         ^
```

Here:

```
nums[mid] = 7
nums[right] = 2
```

Since `7 > 2`, the rotation point lies after `mid`.

Move:

```
left = mid + 1
```
5. Otherwise, the minimum is at `mid` or to the left of it.

Example:

```
[0,1,2,4,5,6,7]
     ^
```

Since the right side is sorted and larger, the minimum cannot be farther right.

Move:

```
right = mid
```
6. Continue shrinking the search space until `left == right`.

At this point, both pointers identify the minimum element.
7. Return `nums[left]`.

The algorithm works because every iteration removes half of the remaining search space while never discarding the minimum element.

## Python Solution

```
from typing import List

class Solution:
    def findMin(self, nums: List[int]) -> int:
        left = 0
        right = len(nums) - 1

        while left < right:
            mid = (left + right) // 2

            if nums[mid] > nums[right]:
                left = mid + 1
            else:
                right = mid

        return nums[left]
```

The implementation starts by initializing the binary search boundaries. The search space initially includes the entire array.

Inside the loop, the middle index is calculated. The algorithm compares the middle value with the rightmost value because the right side gives us information about whether the rotation point lies to the left or right.

If the middle value is larger than the rightmost value, the minimum must be somewhere after the middle index. Therefore, the left boundary moves to `mid + 1`.

Otherwise, the minimum could still be the middle element itself, so the right boundary becomes `mid`.

Eventually, the search space shrinks to a single index, which contains the minimum value.

## Go Solution

```
func findMin(nums []int) int {
    left := 0
    right := len(nums) - 1

    for left < right {
        mid := left + (right-left)/2

        if nums[mid] > nums[right] {
            left = mid + 1
        } else {
            right = mid
        }
    }

    return nums[left]
}
```

The Go implementation follows the same logic as the Python solution.

One small implementation detail is the midpoint calculation:

```
mid := left + (right-left)/2
```

This form avoids potential integer overflow in languages where integer limits matter. While Go integers are typically large enough for this problem, this is considered best practice for binary search implementations.

Go slices are used naturally as arrays, and no extra memory allocation is required.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(log n) | Each iteration removes half of the remaining search space |
| Space | O(1) | Only a few pointer variables are used |

The binary search repeatedly halves the size of the search interval. Starting with `n` elements, the number of iterations required is logarithmic.

No auxiliary data structures are used, so the algorithm requires constant extra memory.

## Edge Cases

### Single Element Array

Consider:

```
[5]
```

A naive implementation might incorrectly enter the loop or access invalid indices.

In this implementation:

```
left = 0
right = 0
```

Since `left < right` is false immediately, the loop never runs, and the function correctly returns `nums[0]`.

### Array Not Rotated

Consider:

```
[1,2,3,4,5]
```

Some solutions incorrectly assume there must always be a rotation point.

Here, every middle value is smaller than the rightmost value, so the search continually moves leftward until reaching index `0`, which correctly contains the minimum.

### Minimum Near the End

Consider:

```
[2,3,4,5,1]
```

This configuration can expose off by one mistakes.

When:

```
nums[mid] > nums[right]
```

the implementation uses:

```
left = mid + 1
```

This guarantees that the already checked `mid` index is excluded safely while still preserving the minimum element.

### Minimum Near the Beginning

Consider:

```
[5,1,2,3,4]
```

The minimum appears immediately after the first element.

The binary search still correctly identifies the unsorted side and narrows the range until only the minimum index remains.
