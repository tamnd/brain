## Solution

## Problem Understanding

This problem asks us to determine whether a singly linked list contains a cycle. A cycle exists when traversing the list through the `next` pointers eventually leads us back to a previously visited node instead of reaching the end of the list.

In a normal singly linked list, the last node points to `None` (or `nil` in Go), which terminates traversal. However, in a cyclic linked list, some node points back to an earlier node, creating a loop. If we continuously follow `next` pointers, we would keep revisiting the same nodes forever.

The input consists of the `head` node of a linked list. Although the problem description mentions `pos`, it is only used internally by LeetCode to describe where the cycle connects. We are not given `pos` as input, so our algorithm must infer whether a cycle exists solely by examining the linked structure.

The expected output is a boolean value:

- Return `true` if there is a cycle.
- Return `false` if the list terminates normally.

The constraints tell us that the number of nodes ranges from `0` to `10^4`. This is small enough that an `O(n)` solution is perfectly acceptable, but the follow-up explicitly asks whether we can solve the problem using constant extra memory. This strongly suggests that we should look for a pointer-based technique rather than storing visited nodes.

Several edge cases are important to consider upfront. The list may be empty (`head = None`), meaning there are no nodes at all. A list with a single node may either point to `None` or point back to itself, creating a cycle. Very short lists of one or two nodes often expose pointer handling mistakes. We also need to safely handle pointer movement without dereferencing `None`, which is a common source of runtime errors in linked list problems.

## Approaches

### Brute Force Approach, Track Visited Nodes

A straightforward way to detect a cycle is to keep track of every node we have already visited. As we traverse the linked list, we store each node reference in a hash set.

For every node:

- If the node has already been seen, we know we have looped back to a previously visited location, so a cycle exists.
- Otherwise, we add the node to the set and continue moving forward.

This approach is correct because a cycle necessarily revisits nodes. If traversal reaches `None`, we know the list terminated normally and therefore contains no cycle.

The main drawback is memory usage. In the worst case, we may store every node in the hash set, requiring `O(n)` extra space. While this works within the constraints, it does not satisfy the follow-up requirement for constant memory.

### Optimal Approach, Floyd's Cycle Detection Algorithm

The key observation is that if a cycle exists, two pointers moving at different speeds will eventually meet.

We use two pointers:

- A **slow pointer** moves one step at a time.
- A **fast pointer** moves two steps at a time.

If there is no cycle, the fast pointer eventually reaches `None`, meaning the list terminates.

If there is a cycle, the fast pointer will eventually "lap" the slow pointer and land on the same node. This happens because both pointers are trapped inside the loop, and the faster pointer continuously closes the gap.

This technique, commonly called **Floyd's Tortoise and Hare Algorithm**, is ideal here because it runs in linear time while using only constant extra memory.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Uses a hash set to track visited nodes |
| Optimal | O(n) | O(1) | Uses fast and slow pointers to detect cycles |

## Algorithm Walkthrough

1. Initialize two pointers, `slow` and `fast`, both starting at the `head` node. We use two pointers because their differing speeds allow us to detect loops without extra memory.
2. Enter a loop that continues while `fast` and `fast.next` are not `None`. This check is necessary because the fast pointer moves two steps at a time, and attempting to access `fast.next.next` without validation would cause errors.
3. Move the slow pointer forward by one step using `slow = slow.next`. This pointer progresses steadily through the list.
4. Move the fast pointer forward by two steps using `fast = fast.next.next`. This creates the speed difference necessary for cycle detection.
5. After moving both pointers, check whether `slow == fast`. If they point to the same node, a cycle exists. Since the fast pointer moves faster, the only way they can meet is inside a loop.
6. If the loop ends because `fast` becomes `None` or `fast.next` becomes `None`, return `False`. This means the fast pointer reached the end of the list, proving there is no cycle.

## Python Solution

```
# Definition for singly-linked list.
# class ListNode:
#     def __init__(self, x):
#         self.val = x
#         self.next = None

from typing import Optional

class Solution:
    def hasCycle(self, head: Optional[ListNode]) -> bool:
        slow = head
        fast = head

        while fast and fast.next:
            slow = slow.next
            fast = fast.next.next

            if slow == fast:
                return True

        return False
```

The implementation starts by initializing both `slow` and `fast` to the head of the list. Starting from the same node ensures both pointers begin traversal consistently.

The `while fast and fast.next` condition protects against invalid pointer access. Since the fast pointer moves two steps, we must guarantee both the current node and the next node exist.

Inside the loop, the slow pointer advances by one node while the fast pointer advances by two nodes. This speed difference is the core idea of Floyd's algorithm.

After movement, the code checks whether the two pointers point to the same node. If they do, a cycle exists because the fast pointer has caught up to the slow pointer inside the loop.

If traversal finishes without the pointers meeting, the function returns `False`, indicating that the linked list terminates normally.

## Go Solution

```
/**
 * Definition for singly-linked list.
 * type ListNode struct {
 *     Val int
 *     Next *ListNode
 * }
 */
func hasCycle(head *ListNode) bool {
    slow := head
    fast := head

    for fast != nil && fast.Next != nil {
        slow = slow.Next
        fast = fast.Next.Next

        if slow == fast {
            return true
        }
    }

    return false
}
```

The Go implementation follows the same logic as the Python version, but pointer handling uses Go's `nil` semantics. The condition `fast != nil && fast.Next != nil` prevents dereferencing a `nil` pointer when moving the fast pointer two steps.

Unlike Python, Go does not require type hints because the function signature already specifies pointer types. Equality comparison between nodes works naturally because linked list nodes are pointers, so `slow == fast` checks whether both pointers reference the exact same node.

Integer overflow is not relevant here because we are only moving pointers, not performing arithmetic on values.

## Worked Examples

### Example 1

**Input:** `head = [3,2,0,-4], pos = 1`

The list structure looks like this:

`3 → 2 → 0 → -4`

↑             ↓

└─────────────┘

The tail connects back to the node with value `2`.

| Iteration | Slow Position | Fast Position | Match? |
| --- | --- | --- | --- |
| Start | 3 | 3 | No |
| 1 | 2 | 0 | No |
| 2 | 0 | 2 | No |
| 3 | -4 | -4 | Yes |

At iteration 3, both pointers point to the same node. Since they meet, we return `true`.

### Example 2

**Input:** `head = [1,2], pos = 0`

The structure is:

`1 → 2`

↑      ↓

└──────┘

The tail points back to the first node.

| Iteration | Slow Position | Fast Position | Match? |
| --- | --- | --- | --- |
| Start | 1 | 1 | No |
| 1 | 2 | 1 | No |
| 2 | 1 | 1 | Yes |

The fast pointer eventually catches the slow pointer inside the cycle, so we return `true`.

### Example 3

**Input:** `head = [1], pos = -1`

The structure is:

`1 → None`

| Iteration | Slow Position | Fast Position | Match? |
| --- | --- | --- | --- |
| Start | 1 | 1 | No |

The loop never executes because `fast.next` is `None`. Since the fast pointer reaches the end immediately, we return `false`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each pointer traverses at most a linear number of nodes |
| Space | O(1) | Only two pointers are used regardless of input size |

The time complexity is `O(n)` because each node is visited at most a constant number of times. In the absence of a cycle, the fast pointer reaches the end after traversing the list. In the presence of a cycle, the pointers meet after a bounded number of steps proportional to the number of nodes.

The space complexity is `O(1)` because we only maintain two pointers, `slow` and `fast`, without allocating any additional data structures that grow with input size.

## Edge Cases

### Empty Linked List

The input may be an empty list where `head = None` (or `nil` in Go). This case can easily cause errors if the implementation assumes at least one node exists. Our solution handles this naturally because the `while fast and fast.next` condition immediately fails, returning `False`.

### Single Node Without a Cycle

A list containing one node that points to `None` is another important edge case. A naive implementation might incorrectly assume a single node forms a cycle. Since `fast.next` is `None`, the loop never starts, and the algorithm correctly returns `False`.

### Single Node With a Self Cycle

A single node may point back to itself. This is a valid cycle and often exposes bugs in implementations that assume cycles require multiple nodes. In our approach, both pointers move and quickly meet again on the same node, causing the function to correctly return `True`.

### Very Small Lists

Lists with only two nodes can be tricky because the fast pointer jumps two steps at once. Without proper null checks, accessing `fast.next.next` may crash the program. The condition `fast and fast.next` in Python, or `fast != nil && fast.Next != nil` in Go, guarantees safe traversal.
