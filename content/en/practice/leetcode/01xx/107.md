## Solution

## Problem Understanding

The problem asks us to perform a level order traversal of a binary tree, but with one important difference from the standard traversal order. Instead of returning the levels from top to bottom, we must return them from bottom to top.

In a normal level order traversal, also called Breadth-First Search (BFS), we visit nodes level by level starting from the root. For the tree:

```
        3
       / \
      9  20
         / \
        15  7
```

a standard level order traversal would produce:

```
[[3], [9,20], [15,7]]
```

However, this problem requires the reverse ordering:

```
[[15,7], [9,20], [3]]
```

The input is the root node of a binary tree. Each node contains a value and references to its left and right child nodes. The output must be a list of lists, where each inner list contains all node values at a particular depth in the tree.

The constraints tell us that the tree can contain up to 2000 nodes. This is small enough that an O(n) traversal is completely acceptable, where n is the number of nodes in the tree. Since every node must be visited at least once to produce the traversal, O(n) is also the best possible time complexity.

Several edge cases are important to consider:

- The tree may be empty, meaning `root = None` in Python or `root == nil` in Go. In this case, the answer should be an empty list.
- The tree may contain only one node. The traversal should still return a nested list structure.
- The tree may be highly unbalanced, such as a linked-list-shaped tree. The algorithm must still correctly process levels.
- Some nodes may have only one child. The traversal logic must avoid assuming both children exist.

## Approaches

### Brute Force Approach

A straightforward brute-force solution is to first compute the height of the tree, then repeatedly collect nodes level by level using Depth-First Search (DFS).

The idea works as follows:

1. Compute the height of the tree.
2. For every level from the bottommost level up to the root:

- Run a DFS traversal to collect all nodes at that level.
3. Append the collected values into the result.

This approach is correct because every level is explicitly traversed and gathered in bottom-up order. However, it is inefficient because each DFS may revisit many nodes multiple times.

For a tree with height `h`, traversing each level separately can lead to O(n × h) complexity. In the worst case, a skewed tree has height `n`, producing O(n²) time complexity.

This becomes unnecessarily expensive when a single traversal can already gather all levels.

### Optimal Approach

The optimal solution uses Breadth-First Search with a queue.

BFS naturally processes a tree level by level. During traversal:

- We process all nodes currently in the queue, which represents one level.
- We collect their values into a temporary list.
- We enqueue their children for the next level.

This gives us the standard top-down level order traversal in O(n) time.

To convert it into bottom-up order, we can either:

- Insert each level at the beginning of the result list, or
- Build the traversal normally and reverse the result at the end.

Reversing at the end is usually cleaner and more efficient because inserting at the front of a list repeatedly can be expensive in some languages.

Since every node is visited exactly once, this solution runs in linear time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) worst case | O(h) | Repeated DFS for every level |
| Optimal | O(n) | O(n) | Single BFS traversal with reversal |

## Algorithm Walkthrough

1. First, check whether the tree is empty. If `root` is `None` or `nil`, return an empty list immediately. There are no levels to process.
2. Initialize a queue and place the root node inside it. The queue is essential because BFS processes nodes in first-in-first-out order, which naturally matches level traversal.
3. Create an empty result list that will store levels in top-down order initially.
4. Start a loop that continues while the queue is not empty.
5. At the beginning of each iteration, determine the current queue size. This size represents exactly how many nodes belong to the current level.
6. Create an empty list called `current_level` to store node values for this level.
7. Process exactly `level_size` nodes:

- Remove a node from the front of the queue.
- Append its value to `current_level`.
- If the node has a left child, enqueue it.
- If the node has a right child, enqueue it.
8. After processing all nodes for the level, append `current_level` to the result list.
9. Continue until all levels have been processed and the queue becomes empty.
10. Reverse the result list so that the deepest level appears first and the root level appears last.
11. Return the reversed result.

The queue is the key data structure here because it preserves the exact order needed for BFS traversal. Each loop iteration cleanly separates one level from the next.

## Python Solution

```
from collections import deque
from typing import List, Optional

# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

class Solution:
    def levelOrderBottom(self, root: Optional[TreeNode]) -> List[List[int]]:
        if not root:
            return []

        queue = deque([root])
        result = []

        while queue:
            level_size = len(queue)
            current_level = []

            for _ in range(level_size):
                node = queue.popleft()
                current_level.append(node.val)

                if node.left:
                    queue.append(node.left)

                if node.right:
                    queue.append(node.right)

            result.append(current_level)

        result.reverse()
        return result
```

The implementation begins by handling the empty tree case. Returning immediately avoids unnecessary processing and prevents queue operations on invalid input.

A `deque` is used because it provides efficient O(1) removal from the front with `popleft()`. A normal Python list would make front removals O(n), which would degrade performance.

The outer `while` loop processes one level at a time. Before processing a level, the algorithm records the queue length in `level_size`. This is important because children added during processing should belong to the next level, not the current one.

Inside the loop, each node is removed, its value is stored, and its children are added to the queue if they exist.

After all levels are collected in top-down order, `result.reverse()` transforms the traversal into the required bottom-up ordering.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func levelOrderBottom(root *TreeNode) [][]int {
    if root == nil {
        return [][]int{}
    }

    queue := []*TreeNode{root}
    result := [][]int{}

    for len(queue) > 0 {
        levelSize := len(queue)
        currentLevel := []int{}

        for i := 0; i < levelSize; i++ {
            node := queue[0]
            queue = queue[1:]

            currentLevel = append(currentLevel, node.Val)

            if node.Left != nil {
                queue = append(queue, node.Left)
            }

            if node.Right != nil {
                queue = append(queue, node.Right)
            }
        }

        result = append(result, currentLevel)
    }

    for left, right := 0, len(result)-1; left < right; left, right = left+1, right-1 {
        result[left], result[right] = result[right], result[left]
    }

    return result
}
```

The Go solution follows the same BFS strategy as the Python version.

Instead of Python's `deque`, Go uses a slice to simulate a queue. Removing the first element is done with:

```
queue = queue[1:]
```

Go uses `nil` checks instead of Python truthiness checks. The result reversal is implemented manually because Go does not provide a built-in reverse function for slices.

The algorithmic behavior and complexity remain identical between the two implementations.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every node is visited exactly once |
| Space | O(n) | Queue and result storage can hold all nodes |

The BFS traversal processes each node a single time, making the runtime linear relative to the number of nodes in the tree.

The space complexity is also O(n) because, in the worst case, the queue may contain an entire level of the tree. For a balanced tree, the last level can contain approximately n/2 nodes. The result list also stores all node values.

## Edge Cases

### Empty Tree

An empty tree is represented by `root = None` in Python or `root == nil` in Go. This case can easily cause errors if the algorithm assumes the root exists and immediately attempts queue operations.

The implementation handles this safely by checking the root at the beginning and returning an empty list immediately.

### Single Node Tree

A tree containing only one node is a minimal valid input. Some incorrect implementations accidentally return a flat list like `[1]` instead of the required nested structure `[[1]]`.

This solution always stores values level by level, ensuring the result format remains correct even for one node.

### Highly Unbalanced Tree

A tree may resemble a linked list:

```
1
 \
  2
   \
    3
```

This can expose flaws in traversal logic if levels are not separated properly.

The BFS implementation avoids this issue because each iteration explicitly processes only the nodes currently in the queue. Even in a skewed tree, each node forms its own level correctly.

### Nodes With Missing Children

Many binary trees are incomplete, meaning some nodes have only one child.

For example:

```
    1
   /
  2
```

An implementation that blindly accesses both children could crash or enqueue invalid values.

This solution checks whether each child exists before enqueueing it, preventing invalid operations while still preserving correct traversal order.
