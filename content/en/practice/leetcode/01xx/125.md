## Solution

## Problem Understanding

The problem asks us to determine whether a given string is a palindrome after applying two transformations:

1. Convert all uppercase letters to lowercase.
2. Remove every character that is not alphanumeric.

An alphanumeric character means either:

- A lowercase or uppercase English letter (`a-z`, `A-Z`)
- A digit (`0-9`)

After cleaning the string according to these rules, we must check whether the resulting sequence reads the same forward and backward.

For example, the input:

```
"A man, a plan, a canal: Panama"
```

becomes:

```
"amanaplanacanalpanama"
```

Since this cleaned string is identical when reversed, the answer is `true`.

The input is a single string `s`, and the output is a boolean:

- `true` if the processed string is a palindrome
- `false` otherwise

The constraint:

```
1 <= s.length <= 2 * 10^5
```

is important because it tells us the string can be very large. Any unnecessarily expensive operation could become problematic. In particular:

- Nested loops would be inefficient.
- Repeated string concatenation could become costly.
- We should aim for a linear time solution.

The problem also guarantees that the input contains only printable ASCII characters. This simplifies character handling because we only need to deal with standard English letters, digits, spaces, and punctuation.

Several edge cases are important:

- A string containing only punctuation or spaces becomes an empty string after filtering, and an empty string is considered a palindrome.
- Case differences should not matter, meaning `"Aa"` should return `true`.
- Digits are valid alphanumeric characters, so `"1a1"` is a palindrome.
- Mixed punctuation and spaces can easily break naive implementations if filtering is not handled correctly.

## Approaches

### Brute Force Approach

A straightforward solution is to explicitly construct a cleaned version of the string.

We iterate through every character in the input:

- If the character is alphanumeric, we convert it to lowercase and append it to a new string.
- Otherwise, we ignore it.

Once the cleaned string is built, we reverse it and compare the reversed version with the original cleaned string.

This works because a palindrome is defined precisely as a string that reads the same forward and backward. By normalizing the input first, we ensure the comparison follows the problem rules exactly.

However, this approach uses additional memory proportional to the size of the string because:

- We store the cleaned string.
- We often create another reversed copy.

Although the time complexity is still linear, the extra memory usage is unnecessary.

### Optimal Two Pointer Approach

The key observation is that we do not actually need to build the cleaned string.

Instead, we can compare characters directly from both ends of the original string using two pointers:

- One pointer starts at the left side.
- One pointer starts at the right side.

At each step:

- Skip characters that are not alphanumeric.
- Convert valid characters to lowercase.
- Compare the two characters.

If they differ, the string cannot be a palindrome.

If they match, move both pointers inward and continue.

This approach works because palindrome validation fundamentally compares mirrored positions. Non-alphanumeric characters do not participate in the comparison, so we can simply skip them during traversal instead of physically removing them.

This avoids constructing intermediate strings and reduces space complexity to constant extra space.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Builds a cleaned string and compares it with its reverse |
| Optimal | O(n) | O(1) | Uses two pointers directly on the original string |

## Algorithm Walkthrough

1. Initialize two pointers:

- `left` at index `0`
- `right` at index `len(s) - 1`
2. Continue processing while `left < right`.

This ensures we compare mirrored positions moving inward from both ends.
3. Check whether the character at `left` is alphanumeric.

- If it is not, increment `left`.
- We skip it because punctuation and spaces are ignored by the problem definition.
4. Check whether the character at `right` is alphanumeric.

- If it is not, decrement `right`.
- Again, ignored characters do not participate in palindrome checking.
5. Once both pointers reference valid alphanumeric characters:

- Convert both characters to lowercase.
- Compare them.
6. If the characters are different:

- Return `false` immediately.
- A palindrome must match at every mirrored position.
7. If the characters match:

- Increment `left`
- Decrement `right`
- Continue checking the next pair.
8. If the loop finishes without finding a mismatch:

- Return `true`
- Every relevant character matched successfully.

The reason the two pointer technique is ideal here is that palindrome checking naturally compares symmetric positions. By skipping invalid characters during traversal, we avoid extra memory allocation entirely.

## Python Solution

```
class Solution:
    def isPalindrome(self, s: str) -> bool:
        left = 0
        right = len(s) - 1

        while left < right:
            while left < right and not s[left].isalnum():
                left += 1

            while left < right and not s[right].isalnum():
                right -= 1

            if s[left].lower() != s[right].lower():
                return False

            left += 1
            right -= 1

        return True
```

The implementation begins by initializing two pointers at opposite ends of the string.

The outer `while` loop continues until the pointers cross. This guarantees that every mirrored pair is checked exactly once.

Inside the loop, the first inner `while` loop advances the left pointer until it reaches an alphanumeric character. The second inner loop performs the same operation from the right side.

Once both pointers reference valid characters, the implementation converts them to lowercase using `.lower()` before comparison. This ensures uppercase and lowercase letters are treated identically.

If the characters differ, the function immediately returns `False` because a palindrome condition has been violated.

Otherwise, both pointers move inward and the process repeats.

If the traversal completes successfully, the function returns `True`, meaning every valid mirrored pair matched correctly.

## Go Solution

```
import "unicode"

func isPalindrome(s string) bool {
    left := 0
    right := len(s) - 1

    for left < right {
        for left < right && !unicode.IsLetter(rune(s[left])) && !unicode.IsDigit(rune(s[left])) {
            left++
        }

        for left < right && !unicode.IsLetter(rune(s[right])) && !unicode.IsDigit(rune(s[right])) {
            right--
        }

        leftChar := unicode.ToLower(rune(s[left]))
        rightChar := unicode.ToLower(rune(s[right]))

        if leftChar != rightChar {
            return false
        }

        left++
        right--
    }

    return true
}
```

The Go implementation follows the same two pointer logic as the Python solution.

One notable difference is character handling. Go strings are byte sequences, so character operations require conversion to `rune` when using functions from the `unicode` package.

Instead of `.isalnum()`, Go uses:

- `unicode.IsLetter`
- `unicode.IsDigit`

Similarly, lowercase conversion is handled with `unicode.ToLower`.

The algorithm still runs in linear time and constant extra space.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each character is processed at most once by either pointer |
| Space | O(1) | Only a few pointer variables are used |

The time complexity is linear because both pointers move inward across the string without revisiting characters. Even though there are nested `while` loops, each character is skipped or processed only once overall.

The space complexity is constant because no additional data structures proportional to input size are allocated. The algorithm works directly on the original string.

## Edge Cases

One important edge case is a string containing only non-alphanumeric characters, such as:

```
"!!!"
```

After filtering, this becomes an empty string. A naive implementation might incorrectly return `false` or encounter index errors. The two pointer solution handles this correctly because both pointers simply skip invalid characters until the loop terminates naturally.

Another important case is mixed uppercase and lowercase letters:

```
"Aa"
```

Without case normalization, this would incorrectly appear unequal. The implementation explicitly converts both characters to lowercase before comparison, ensuring case-insensitive matching.

A third edge case involves digits mixed with letters:

```
"1a1"
```

Digits are valid alphanumeric characters and must participate in palindrome checking. Some incorrect solutions only process letters and accidentally ignore numbers. The implementation correctly treats both letters and digits as valid comparison characters.

A final subtle case is very large inputs near the constraint limit. Constructing multiple intermediate strings could waste memory unnecessarily. The two pointer approach avoids this by processing the original string in place, keeping space usage constant even for extremely large inputs.
