## Solution

## Problem Understanding

This problem asks us to determine the **minimum initial health** a knight must start with to successfully rescue a princess trapped in the bottom-right corner of a dungeon grid.

The dungeon is represented as a two-dimensional matrix of integers, where each cell affects the knight's health in one of three ways:

- A **negative number** represents damage from demons, reducing health.
- A **positive number** represents healing from magic orbs, increasing health.
- A value of **0** means the room has no effect.

The knight begins in the **top-left corner** and must reach the **bottom-right corner**, moving only **right** or **down**. The important constraint is that the knight **dies immediately if health becomes 0 or negative at any point**, not just at the end.

The challenge is not simply finding a path with the maximum total gain or minimum damage. Instead, we must ensure that the knight's health **never drops below 1 at any intermediate step**.

The input consists of an `m x n` grid called `dungeon`, where:

- `m` is the number of rows
- `n` is the number of columns
- Each cell contains an integer between `-1000` and `1000`

The output is a single integer representing the **minimum initial health required** for the knight to survive the journey and rescue the princess.

The constraints are significant:

- `1 <= m, n <= 200`
- The grid may contain up to `40,000` cells

A brute-force exploration of every path becomes impractical because the number of possible paths grows exponentially. Since the knight can only move right or down, the number of unique paths in a large grid can become enormous.

Several edge cases make this problem tricky:

A dungeon consisting of a single cell must still ensure survival. If the only cell contains damage, the knight must begin with enough health to survive that room.

A path with high rewards later may still be invalid if the knight dies early. This means maximizing total health gain does not necessarily produce the correct answer.

Large negative values near the destination may require significantly more health before entering the final room.

Positive cells can reduce the required starting health, but the knight must still always maintain at least `1` health.

The problem guarantees that the grid is non-empty and rectangular, so we do not need to handle malformed input.

## Approaches

### Brute Force Approach

A natural first idea is to try **every possible path** from the top-left corner to the bottom-right corner.

For each path, we could simulate the knight's health throughout the journey. Since the knight must never reach `0` or below, we would compute the minimum initial health necessary for that specific path. After evaluating all paths, we would choose the smallest valid starting value.

This approach is correct because it explicitly checks every legal route and guarantees the optimal answer will be found.

However, it is computationally infeasible.

At every cell, the knight can move either right or down. This creates an exponential number of paths. In the worst case, the number of paths is approximately:

inom{m+n-2}{m-1}

For a `200 x 200` grid, this number becomes astronomically large. Even memoizing partial results is not straightforward because survival depends on future cells, not just current accumulated health.

### Key Insight for an Optimal Solution

The main insight is that this problem is easier to solve **backward instead of forward**.

If we try to move from the start to the destination, we do not know how much health we should preserve for future rooms.

Instead, consider the reverse question:

**How much health must the knight have when entering a given cell to guarantee survival to the princess?**

Suppose we already know the minimum health required for the cell to the right and the cell below. Then for the current cell, we only need enough health to safely transition into the cheaper of those two options.

If a room provides healing, we may need less health entering it. If it inflicts damage, we need more.

This naturally leads to **dynamic programming**.

We define:

`dp[row][col]` = minimum health required **when entering this cell** so the knight can eventually rescue the princess.

We compute values starting from the bottom-right corner and move backward toward the top-left.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^(m+n)) | O(m+n) | Explores every possible path recursively |
| Optimal Dynamic Programming | O(m × n) | O(m × n) | Computes minimum required health backward |

## Algorithm Walkthrough

### Step 1: Create a DP Table

We create a `dp` table of size `(m + 1) x (n + 1)`.

The extra row and column simplify boundary handling and eliminate repeated edge-case checks.

We initialize every value with infinity because we want to find minimum required health.

### Step 2: Set Base Conditions

The princess is located at `(m - 1, n - 1)`.

To survive after rescuing the princess, the knight must still have at least `1` health.

We initialize:

- `dp[m][n - 1] = 1`
- `dp[m - 1][n] = 1`

These artificial boundary cells make the bottom-right transition work naturally.

### Step 3: Process the Grid Backward

We iterate from bottom-right toward top-left.

For each cell `(row, col)`:

1. Determine the minimum health needed from the next move:

`min(dp[row + 1][col], dp[row][col + 1])`
2. Subtract the current room's effect because entering this room may increase or decrease health.
3. Ensure health never falls below `1`.

The formula becomes:

$dp[i][j]=\max(1,\min(dp[i+1][j],dp[i][j+1])-dungeon[i][j])$

This formula ensures the knight enters each room with just enough health to survive.

### Step 4: Return the Starting Requirement

After filling the table, the answer is:

`dp[0][0]`

This represents the minimum health required when entering the starting cell.

## Python Solution

```
from typing import List

class Solution:
    def calculateMinimumHP(self, dungeon: List[List[int]]) -> int:
        rows = len(dungeon)
        cols = len(dungeon[0])

        dp = [[float("inf")] * (cols + 1) for _ in range(rows + 1)]

        dp[rows][cols - 1] = 1
        dp[rows - 1][cols] = 1

        for row in range(rows - 1, -1, -1):
            for col in range(cols - 1, -1, -1):
                minimum_next_health = min(
                    dp[row + 1][col],
                    dp[row][col + 1]
                )

                required_health = (
                    minimum_next_health - dungeon[row][col]
                )

                dp[row][col] = max(1, required_health)

        return dp[0][0]
```

### Implementation Walkthrough

We first determine the dimensions of the dungeon grid.

The `dp` table is created with one extra row and column filled with infinity. This prevents invalid paths from being selected during minimization.

The two boundary cells near the princess are initialized to `1`. These act as virtual exits that guarantee the knight only needs `1` health after rescuing the princess.

We then process the grid in reverse order, starting from the bottom-right corner. At each position, we inspect the right and down neighbors to determine the cheaper survival path.

The room's value is subtracted from the required future health. A healing room reduces the requirement, while a damaging room increases it.

Finally, we use `max(1, required_health)` to enforce the rule that health can never drop below `1`.

The result stored in `dp[0][0]` gives the minimum starting health.

## Go Solution

```
func calculateMinimumHP(dungeon [][]int) int {
    rows := len(dungeon)
    cols := len(dungeon[0])

    const INF = int(1e9)

    dp := make([][]int, rows+1)
    for i := range dp {
        dp[i] = make([]int, cols+1)
        for j := range dp[i] {
            dp[i][j] = INF
        }
    }

    dp[rows][cols-1] = 1
    dp[rows-1][cols] = 1

    for row := rows - 1; row >= 0; row-- {
        for col := cols - 1; col >= 0; col-- {
            minimumNextHealth := dp[row+1][col]
            if dp[row][col+1] < minimumNextHealth {
                minimumNextHealth = dp[row][col+1]
            }

            requiredHealth := minimumNextHealth - dungeon[row][col]

            if requiredHealth < 1 {
                dp[row][col] = 1
            } else {
                dp[row][col] = requiredHealth
            }
        }
    }

    return dp[0][0]
}
```

### Go-Specific Notes

The Go implementation follows the same dynamic programming logic as Python.

Instead of using `float("inf")`, Go uses a large sentinel value (`1e9`) to represent infinity. Since the maximum possible required health is well below this number, it safely behaves as an unreachable state.

Go does not provide built-in `min` and `max` functions for integers in older versions, so comparisons are implemented manually with `if` statements.

Slices are used for the DP matrix, which behaves similarly to Python lists.

Since the problem guarantees a valid non-empty grid, no additional nil or empty checks are necessary.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m × n) | Every cell is processed exactly once |
| Space | O(m × n) | The DP table stores one value per cell |

The algorithm visits each cell exactly one time and performs only constant-time operations for each visit. Since the dungeon contains `m × n` cells, the total runtime is `O(m × n)`.

The space complexity is also `O(m × n)` because we allocate a DP table of the same size as the dungeon, plus one extra row and column for boundary simplification.

## Edge Cases

### Single Cell Dungeon

A dungeon with only one room can be tricky because that room may contain damage or healing.

For example:

```
[[-10]]
```

The knight immediately loses `10` health upon entering, so the minimum starting health is `11`.

The implementation handles this naturally because the bottom-right cell is processed using the same recurrence relation as every other cell.

### Extremely Negative Final Cell

Consider:

```
[[1, 2],
 [3, -100]]
```

Even though earlier rooms provide healing, the knight still needs enough health to survive the final room.

A naive greedy strategy might underestimate the requirement. The backward DP correctly propagates the large health requirement from the princess cell to earlier cells.

### Highly Positive Rooms

Consider:

```
[[100]]
```

A naive implementation might return `0` or a negative value because the knight gains so much health.

However, the problem requires the knight to start with at least `1` health. The `max(1, required_health)` condition guarantees this invariant is always maintained.

### Paths With Early Death

Consider:

```
[
    [-100, 1000],
    [1, 1]
]
```

A greedy algorithm might choose the path with a large reward, but the knight may die before reaching it.

The dynamic programming solution avoids this mistake because it computes the exact health needed to safely traverse every future state, ensuring survival at every step rather than maximizing total reward.
