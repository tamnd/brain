## Solution

## Problem Understanding

This problem asks us to maximize profit from stock trading under a strict constraint: we are allowed to complete **at most two transactions**.

A transaction consists of exactly one buy operation followed later by one sell operation. Importantly, we cannot hold multiple stocks simultaneously. This means we must sell any currently held stock before starting another transaction.

The input is an integer array `prices`, where `prices[i]` represents the stock price on the `i-th` day. We may choose whether to buy, sell, or do nothing on each day. The objective is to return the **maximum possible profit** after making at most two valid transactions.

The phrase **"at most two transactions"** is important. We are not required to make exactly two trades. In some cases, the best answer may involve only one profitable transaction or even no transaction at all.

For example:

- If prices continually rise, a single long transaction may be better than splitting into two.
- If prices continually fall, no transaction should be made, producing a profit of `0`.
- If the stock price fluctuates multiple times, carefully choosing two separate buy-sell intervals may yield maximum profit.

The constraints tell us something important about performance requirements:

- `1 <= prices.length <= 10^5`
- `0 <= prices[i] <= 10^5`

Since the input size can reach **100,000 elements**, any quadratic solution such as checking all buy/sell combinations will be too slow. We need a solution with roughly **O(n)** or **O(n log n)** time complexity.

Several edge cases deserve attention:

A single-day array cannot produce profit because selling requires buying on an earlier day.

Strictly decreasing prices should return `0`, since every trade loses money.

Repeated values and flat price regions should be handled correctly, since buying and selling at equal prices produces zero profit.

Cases where only one profitable transaction exists must not force a second unnecessary trade.

## Approaches

### Brute Force Approach

The brute-force idea is to try every possible pair of transactions.

We can imagine selecting:

1. A first buy day and first sell day
2. A second buy day and second sell day

For every possible valid arrangement, we calculate the total profit and track the maximum.

This guarantees correctness because every legal transaction combination is evaluated.

However, the number of possibilities grows extremely quickly. Even a single transaction requires checking all `(buy, sell)` pairs, which already costs `O(n²)`. Adding a second transaction introduces another nested search over remaining days, pushing complexity toward `O(n⁴)` in the naive form.

Even with pruning or memoization, brute force becomes impractical for `n = 10^5`.

### Key Insight for an Optimal Solution

The important observation is that the problem only allows **two transactions**.

Instead of explicitly searching for buy/sell ranges, we can model the process as **state transitions**.

At any point, we care about only four meaningful states:

1. Maximum balance after first buy
2. Maximum profit after first sell
3. Maximum balance after second buy
4. Maximum profit after second sell

These states summarize everything important from previous days.

For example:

- After the first buy, we want the lowest purchase cost.
- After the first sell, we want maximum profit so far.
- After the second buy, we effectively spend profit from the first transaction.
- After the second sell, we maximize total accumulated profit.

Since each day updates only these four states, we can process the array in one pass while using constant extra memory.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n⁴) | O(1) | Tries every possible pair of transactions |
| Optimal State DP | O(n) | O(1) | Tracks four transaction states in one pass |

## Algorithm Walkthrough

We use four variables to represent the best possible outcome at each trading stage.

- `first_buy`
- `first_sell`
- `second_buy`
- `second_sell`

The meaning of each variable is crucial.

`first_buy` represents the maximum remaining money after buying the first stock. Since buying costs money, this value is negative.

`first_sell` represents the maximum profit after selling the first stock.

`second_buy` represents the maximum remaining money after buying the second stock, after accounting for profit from the first transaction.

`second_sell` represents the maximum total profit after selling the second stock.

We process prices one by one.

1. Initialize states.

We begin assuming no transaction has happened.

`first_buy` and `second_buy` start at negative infinity because no purchase has occurred yet.

`first_sell` and `second_sell` start at `0` because profit begins at zero.
2. Update `first_buy`.

On each day, we either keep our previous purchase or buy today.

We want the best balance after buying:

`max(previous_first_buy, -price)`

Since buying costs money, the value becomes negative.
3. Update `first_sell`.

We either keep previous profit or sell today.

Selling today gives:

`first_buy + price`

We take the maximum.
4. Update `second_buy`.

We either keep the previous second purchase or buy again today using profit from the first sell.

Buying the second stock costs money:

`first_sell - price`
5. Update `second_sell`.

We either keep prior best profit or sell the second stock today.

Profit becomes:

`second_buy + price`
6. Continue through all days.

By the end of the array, `second_sell` contains the best possible profit using at most two transactions.

The reason this works is that each state always stores the best possible value seen so far. Every update uses only previously computed optimal states.

## Python Solution

```
from typing import List

class Solution:
    def maxProfit(self, prices: List[int]) -> int:
        first_buy = float("-inf")
        first_sell = 0
        second_buy = float("-inf")
        second_sell = 0

        for price in prices:
            first_buy = max(first_buy, -price)
            first_sell = max(first_sell, first_buy + price)
            second_buy = max(second_buy, first_sell - price)
            second_sell = max(second_sell, second_buy + price)

        return second_sell
```

The implementation directly mirrors the algorithm.

We initialize the four transaction states before processing prices.

Inside the loop, every price updates the states in order.

`first_buy` determines the cheapest first purchase opportunity.

`first_sell` tracks the best profit after completing one transaction.

`second_buy` calculates the best effective balance after reinvesting first-transaction profit into another stock purchase.

`second_sell` tracks the best final profit after completing two transactions.

The update order matters. Each step depends on the most recently updated previous state.

For example, `second_buy` depends on the newest value of `first_sell`, not the previous iteration's value.

At the end, `second_sell` naturally represents the answer.

Since we are allowed **at most two transactions**, returning `second_sell` also correctly covers cases where only one transaction is optimal, because the second transaction may simply never improve profit.

## Go Solution

```
func maxProfit(prices []int) int {
	firstBuy := -1 << 60
	firstSell := 0
	secondBuy := -1 << 60
	secondSell := 0

	for _, price := range prices {
		if -price > firstBuy {
			firstBuy = -price
		}

		if firstBuy+price > firstSell {
			firstSell = firstBuy + price
		}

		if firstSell-price > secondBuy {
			secondBuy = firstSell - price
		}

		if secondBuy+price > secondSell {
			secondSell = secondBuy + price
		}
	}

	return secondSell
}
```

The Go solution follows the same logic as the Python version.

Go does not have built-in infinity values for integers, so we initialize `firstBuy` and `secondBuy` with a very small negative number.

We use conditional comparisons instead of Python's `max()` function.

There are no integer overflow concerns here because constraints are at most `10^5`, and profits remain comfortably within Go's integer range.

An empty slice is not possible due to constraints, but the implementation would still safely return `0`.

## Edge Cases

### Single Day Input

An input such as:

```
[5]
```

cannot produce profit because buying and selling require at least two days.

A naive implementation might attempt invalid transactions or produce unexpected values. Our solution handles this naturally because no sell state ever becomes profitable, so the result remains `0`.

### Strictly Decreasing Prices

Consider:

```
[9,7,5,3,1]
```

Every future price is lower than the previous one.

A buggy implementation may force transactions and accidentally produce negative profit. Our algorithm never accepts worse outcomes because every state update uses `max()`, ensuring profit never drops below `0`.

### Only One Profitable Transaction

Consider:

```
[1,10]
```

Only one transaction is meaningful.

Some implementations incorrectly assume exactly two transactions must occur. Our solution handles this properly because the second transaction is optional. `second_sell` simply inherits the best achievable result from one transaction.

### Repeated Prices and Flat Regions

Consider:

```
[3,3,3,3]
```

No profitable opportunity exists.

Repeated prices can sometimes expose logic bugs involving duplicate values or incorrect transaction ordering. Since our state transitions compare maximum values at every step, the algorithm safely returns `0`.

### Multiple Peaks and Valleys

Consider:

```
[1,5,2,10]
```

The optimal answer is not a single long trade.

The best strategy is:

- Buy at `1`, sell at `5`
- Buy at `2`, sell at `10`

Total profit = `12`.

The state-based DP naturally captures these opportunities without explicitly searching all intervals.
