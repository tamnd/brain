# LeetCode 156 - Binary Tree Upside Down

## Problem Understanding

This problem asks us to transform a binary tree into a new structure by flipping it upside down according to a very specific rule.

For every node in the original tree:

- The original left child becomes the new parent.
- The original root becomes the new right child.
- The original right child becomes the new left child.

This transformation is applied recursively from top to bottom along the left spine of the tree.

The important observation is that the tree has a special structure guaranteed by the problem:

- Every right child always has a corresponding left sibling.
- Right children never have children of their own.

Because of this guarantee, the tree behaves almost like a linked list along the left side, with each node optionally carrying a right sibling that must be repositioned during the transformation.

For example, consider this tree:

```
        1
       / \
      2   3
     / \
    4   5
```

After flipping:

```
        4
       / \
      5   2
         / \
        3   1
```

The original leftmost node becomes the new root because every transformation promotes the left child upward.

The input is the root node of a binary tree. The output should be the root node of the transformed tree.

The constraints are very small, only up to 10 nodes, so performance is not a major issue. However, the problem is primarily about correctly manipulating pointers without losing access to parts of the tree.

Several edge cases are important:

- An empty tree should return `None` or `nil`.
- A single-node tree should remain unchanged.
- Trees with multiple levels require careful pointer rewiring to avoid cycles or disconnected nodes.
- Since right children never have children, we never need to recursively process a right subtree independently.

## Approaches

### Brute Force Approach

A straightforward brute-force solution is to reconstruct the tree from scratch.

We could traverse the original tree, record parent-child relationships, and then explicitly build a completely new tree with reversed relationships. This works because the transformation rules are deterministic.

For every node:

- The left child becomes the parent.
- The right child becomes the left child of the promoted node.
- The current node becomes the right child.

Although this produces the correct answer, it requires allocating entirely new nodes or maintaining extensive auxiliary mappings between old and new relationships.

This approach is unnecessarily complicated because the transformation can actually be done in-place by reusing the existing nodes.

### Optimal Approach

The key insight is that the transformation only depends on the path formed by repeatedly following left children.

If we recursively travel to the leftmost node first, that node naturally becomes the new root of the flipped tree.

Then, while returning from recursion, we can rewire pointers:

- The original right child becomes the new left child.
- The original parent becomes the new right child.
- The old connections are cleared to avoid cycles.

Because every node is visited exactly once and modified in-place, this produces an efficient linear-time solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Rebuilds the tree using extra storage |
| Optimal | O(n) | O(h) | In-place recursive pointer rewiring |

Here, `n` is the number of nodes and `h` is the height of the tree.

## Algorithm Walkthrough

1. Start from the root and recursively move to the left child until reaching the leftmost node.

This node becomes the new root because every parent will eventually rotate beneath it.
2. Once recursion reaches the base case, return the leftmost node upward.

The recursive calls now begin unwinding from bottom to top.
3. For each node during unwinding, rewire the pointers.

Suppose the current node is `root` and its left child is `root.left`.

We perform:

- `root.left.left = root.right`
- `root.left.right = root`

This implements the upside-down transformation rules.
4. Remove the original connections.

Set:

- `root.left = None`
- `root.right = None`

This prevents cycles and disconnects obsolete links.
5. Continue returning the new root upward unchanged.

Every recursive call ultimately returns the leftmost node discovered initially.

Because the transformation only affects parent-child relationships along the left chain, recursion is a natural fit for this problem.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

class Solution:
    def upsideDownBinaryTree(self, root: Optional[TreeNode]) -> Optional[TreeNode]:
        if not root or not root.left:
            return root

        new_root = self.upsideDownBinaryTree(root.left)

        root.left.left = root.right
        root.left.right = root

        root.left = None
        root.right = None

        return new_root
```

The implementation begins with the base case. If the tree is empty or the current node has no left child, that node is already the leftmost node and therefore becomes the new root.

The recursive call continues descending leftward until reaching that node.

After recursion returns, the algorithm rewires the tree structure. The current node's original right child becomes the left child of the promoted node, and the current node itself becomes the right child.

Finally, the original links are removed to avoid cycles in the transformed structure.

The function always returns `new_root`, which remains the leftmost node discovered during recursion.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func upsideDownBinaryTree(root *TreeNode) *TreeNode {
    if root == nil || root.Left == nil {
        return root
    }

    newRoot := upsideDownBinaryTree(root.Left)

    root.Left.Left = root.Right
    root.Left.Right = root

    root.Left = nil
    root.Right = nil

    return newRoot
}
```

The Go implementation follows the exact same recursive logic as the Python version.

The primary language-specific difference is the use of `nil` instead of `None`.

Go pointers directly represent tree node references, so pointer rewiring works naturally without any additional memory allocation.

Since node values are very small, integer overflow is not a concern.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every node is visited exactly once |
| Space | O(h) | Recursive call stack stores at most tree height |

The algorithm performs a single DFS traversal of the tree. Each node undergoes constant-time pointer updates, so the total runtime is linear in the number of nodes.

The only extra space comes from recursion. In the worst case, the recursion depth equals the height of the tree.

## Edge Cases

### Empty Tree

An empty tree is represented by `None` or `nil`. A naive implementation might attempt to access child pointers immediately and crash.

The solution safely handles this with the base case:

```
if not root:
    return root
```

This immediately returns an empty tree unchanged.

### Single Node Tree

A tree containing only one node has no left child. Since there is nothing to rotate, the node should remain the root.

The condition:

```
if not root.left:
    return root
```

correctly stops recursion and returns the original node.

### Preventing Cycles

One of the most common bugs in pointer manipulation problems is forgetting to disconnect obsolete links.

Suppose we do not execute:

```
root.left = None
root.right = None
```

The old parent-child relationships would still exist, creating cycles in the tree structure.

The implementation explicitly clears the old pointers after rewiring, ensuring the final structure is a valid tree.

### Deep Left-Skewed Trees

A tree where every node only has a left child still works correctly.

Example:

```
    1
   /
  2
 /
3
```

After transformation:

```
3
 \
  2
   \
    1
```

The recursive approach naturally handles this because each node simply becomes the right child of its former left child.
