## Solution

## Problem Understanding

The problem asks us to generate the first `numRows` rows of **Pascal's Triangle**.

Pascal's Triangle is a triangular arrangement of numbers where:

- The first and last number of every row is always `1`
- Every number between them is the sum of the two numbers directly above it from the previous row

For example, the first few rows look like this:

```
        [1]
       [1,1]
      [1,2,1]
     [1,3,3,1]
    [1,4,6,4,1]
```

The input is a single integer, `numRows`, which represents how many rows of the triangle we need to generate. The output should be a two-dimensional list, where each inner list represents one row of Pascal's Triangle.

For example:

- If `numRows = 1`, we only return the first row: `[[1]]`
- If `numRows = 5`, we return the first five rows.

The constraint is:

```
1 <= numRows <= 30
```

This tells us that the input size is very small. Since there are at most only 30 rows, performance is not a major concern, and even an `O(n²)` solution is completely acceptable. However, we should still aim for a clean and logically efficient approach.

An important observation is that each row depends only on the row immediately before it. This means we can build the triangle incrementally, one row at a time.

Some important edge cases include:

- `numRows = 1`, where we return only `[[1]]`
- Small inputs like `2`, where the triangle contains only boundary values (`1`s)
- Larger valid inputs like `30`, where many inner values must be computed correctly

The problem guarantees that `numRows` is always at least `1`, so we never need to handle invalid or empty input.

## Approaches

### Brute Force Approach

A brute-force way to solve this problem would be to compute each element using the mathematical definition of Pascal's Triangle through combinations.

Each value in Pascal's Triangle corresponds to:

$$C(n, k) = \frac{n!}{k!(n-k)!}$$

This means for every row and every position inside that row, we could calculate the binomial coefficient directly using factorials.

This approach produces the correct answer because Pascal's Triangle is mathematically defined by combinations. However, repeatedly computing factorials introduces unnecessary repeated computation. Even though the constraints are small, this solution is less intuitive and less efficient than necessary.

The major drawback is that computing factorials repeatedly for every element adds overhead and makes the implementation more complicated.

### Optimal Dynamic Programming Approach

The better observation is that Pascal's Triangle has a natural recurrence relation.

Every row can be generated from the previous row:

- The first element is always `1`
- The last element is always `1`
- Every middle element is the sum of the two numbers directly above it

If the previous row is:

```
[1,3,3,1]
```

The next row becomes:

```
[1, 1+3, 3+3, 3+1, 1]
= [1,4,6,4,1]
```

This works well because each row depends only on previously computed values. We build the answer incrementally, storing already-generated rows and using them to compute the next one.

Since there are roughly `1 + 2 + 3 + ... + numRows` elements, the total work is quadratic.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n³) | O(n²) | Computes every value using repeated factorial calculations or combinations |
| Optimal | O(n²) | O(n²) | Builds each row from the previous row using Pascal's recurrence |

## Algorithm Walkthrough

1. Create an empty result list that will store all rows of Pascal's Triangle.
2. Iterate from row `0` to `numRows - 1`. Each iteration constructs one row.
3. Start the current row with the value `1` because every row begins with `1`.
4. If this is not the first row, look at the previous row and compute the middle elements. For each position, add the two adjacent values from the previous row.
5. Append another `1` at the end of the row if the row index is greater than `0`, because every row ends with `1`.
6. Add the completed row to the result list.
7. After all rows are generated, return the result.

This approach naturally mirrors the structure of Pascal's Triangle. We reuse previously computed rows rather than recomputing values from scratch.

## Python Solution

```
from typing import List

class Solution:
    def generate(self, numRows: int) -> List[List[int]]:
        triangle = []

        for row_index in range(numRows):
            current_row = [1]

            if row_index > 0:
                previous_row = triangle[-1]

                for i in range(1, row_index):
                    current_row.append(
                        previous_row[i - 1] + previous_row[i]
                    )

                current_row.append(1)

            triangle.append(current_row)

        return triangle
```

The implementation begins by creating an empty list called `triangle`, which stores all rows.

We iterate `numRows` times, where each iteration corresponds to one row in Pascal's Triangle.

Each row starts with `[1]` because the first value is always `1`.

For rows beyond the first row, we retrieve the previous row using `triangle[-1]`. We then calculate the middle elements by summing adjacent values from the previous row:

```
previous_row[i - 1] + previous_row[i]
```

This directly follows Pascal's Triangle's definition.

After computing the middle elements, we append another `1` to close the row. Finally, we add the row to `triangle`.

Once all rows are generated, we return the completed triangle.

## Go Solution

```
func generate(numRows int) [][]int {
    triangle := make([][]int, 0, numRows)

    for rowIndex := 0; rowIndex < numRows; rowIndex++ {
        currentRow := []int{1}

        if rowIndex > 0 {
            previousRow := triangle[len(triangle)-1]

            for i := 1; i < rowIndex; i++ {
                currentRow = append(
                    currentRow,
                    previousRow[i-1]+previousRow[i],
                )
            }

            currentRow = append(currentRow, 1)
        }

        triangle = append(triangle, currentRow)
    }

    return triangle
}
```

The Go implementation follows the same logic as the Python version, but uses slices instead of lists.

The `triangle` variable is initialized with a capacity of `numRows` to reduce reallocations. Since Go slices grow dynamically, `append()` is used whenever new elements are added.

Unlike Python, Go does not have built-in list typing, so we explicitly declare the result as `[][]int`.

Integer overflow is not a concern because the problem constraint is small (`numRows <= 30`), and Pascal's Triangle values remain safely within Go's integer range.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n²) | We generate roughly `1 + 2 + ... + n` elements across all rows |
| Space | O(n²) | The output itself stores all rows of Pascal's Triangle |

The time complexity is quadratic because every element in the triangle is computed exactly once. Since row `i` contains `i + 1` elements, the total number of operations is proportional to the sum of the first `n` integers.

The space complexity is also quadratic because we store every generated number in the final output. This storage is unavoidable because the problem explicitly asks us to return the entire triangle.

## Edge Cases

### Edge Case 1, `numRows = 1`

This is the smallest valid input. A buggy implementation might incorrectly assume there is always a previous row to reference.

Our implementation handles this safely because the first row starts as `[1]`, and the previous-row logic only executes when `row_index > 0`.

Result:

```
[[1]]
```

### Edge Case 2, Small Inputs Like `numRows = 2`

When `numRows = 2`, there are no middle elements to calculate.

The second row is simply:

```
[1,1]
```

A buggy implementation might accidentally enter the middle-element loop and produce incorrect indexing behavior. Our implementation avoids this because:

```
for i in range(1, row_index)
```

For `row_index = 1`, the loop does not execute.

### Edge Case 3, Maximum Constraint `numRows = 30`

Larger inputs create many rows with larger values.

A poor implementation using repeated factorial calculations may introduce unnecessary computational overhead or complexity. Our dynamic programming solution remains efficient because every value is computed exactly once from already-existing rows.

Since `numRows` is capped at `30`, the algorithm runs comfortably within limits while remaining easy to understand and maintain.
