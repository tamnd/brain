## Solution

## Problem Understanding

The problem asks us to return a specific row from Pascal's Triangle, using zero-based indexing. Pascal's Triangle is a triangular arrangement of numbers where every element is the sum of the two elements directly above it. The first and last element of every row are always `1`.

For example, the first few rows are:

```
Row 0: [1]
Row 1: [1, 1]
Row 2: [1, 2, 1]
Row 3: [1, 3, 3, 1]
Row 4: [1, 4, 6, 4, 1]
```

The input `rowIndex` tells us which row we need to generate. Since indexing starts at `0`, an input of `3` means we must return the fourth row:

```
[1, 3, 3, 1]
```

The output must be a list or slice containing all values in that row.

The constraint is:

```
0 <= rowIndex <= 33
```

This tells us the input size is quite small. Even relatively inefficient solutions would still run comfortably within limits. However, the follow up specifically asks whether we can solve the problem using only `O(rowIndex)` extra space, so the intended solution focuses on space optimization.

There are several important edge cases to consider:

- `rowIndex = 0`, where the answer is simply `[1]`
- `rowIndex = 1`, where the result becomes `[1, 1]`
- Updating values in place, because careless updates can overwrite values that are still needed later
- Correctly handling the boundaries of each row, since the first and last elements must always remain `1`

A naive implementation can easily corrupt values if it updates the current row from left to right. Understanding why update order matters is central to the optimal solution.

## Approaches

### Brute Force Approach

The most straightforward way to solve this problem is to generate the entire Pascal's Triangle up to the requested row, then return the final row.

We start with the first row `[1]`. For every new row, we create a fresh array where:

- The first element is `1`
- The last element is `1`
- Every middle element is the sum of the two elements above it from the previous row

For example:

```
Previous row: [1, 3, 3, 1]

New row:
1
1+3 = 4
3+3 = 6
3+1 = 4
1

Result: [1, 4, 6, 4, 1]
```

This approach is correct because it directly follows the mathematical definition of Pascal's Triangle.

However, it stores all intermediate rows even though we only need the final one. That wastes memory.

### Optimal Approach

The key observation is that each row only depends on the row immediately before it. We do not need to store the entire triangle.

We can instead maintain a single array and update it in place.

The challenge is avoiding overwriting values too early. Suppose we update from left to right:

```
row[j] = row[j] + row[j - 1]
```

The moment we modify `row[j]`, we lose its original value, which may still be needed later.

To avoid this problem, we update from right to left. This preserves the previous row's values while computing the new row.

For example:

```
Current row: [1, 2, 1]

Update right to left:
index 2 -> 1 + 2 = 3
index 1 -> 2 + 1 = 3

Result: [1, 3, 3, 1]
```

This achieves the follow up requirement because we use only one array of size `rowIndex + 1`.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(rowIndex²) | O(rowIndex²) | Builds the entire triangle |
| Optimal | O(rowIndex²) | O(rowIndex) | Uses one array updated in place |

## Algorithm Walkthrough

1. Initialize the result array with a single element `[1]`.

This represents row `0` of Pascal's Triangle.
2. Iterate from row `1` up to `rowIndex`.

Each iteration constructs the next row from the current one.
3. Append a `1` to the end of the array.

Every row in Pascal's Triangle begins and ends with `1`, so the new row must gain one additional element.
4. Update the middle elements from right to left.

For every index `j` from `i - 1` down to `1`:

```
row[j] = row[j] + row[j - 1]
```

Updating backward is critical because it preserves the old values needed for future calculations in the same iteration.
5. After finishing all iterations, return the array.

The array now contains exactly the requested row.

## Python Solution

```
from typing import List

class Solution:
    def getRow(self, rowIndex: int) -> List[int]:
        row = [1]

        for i in range(1, rowIndex + 1):
            row.append(1)

            for j in range(i - 1, 0, -1):
                row[j] = row[j] + row[j - 1]

        return row
```

The implementation begins by initializing the first row as `[1]`.

The outer loop runs once for every new row we need to construct. During each iteration, we append a trailing `1` because every Pascal row ends with `1`.

The inner loop performs the in place updates. It runs backward from `i - 1` down to `1`. This direction is important because it prevents overwriting values that are still needed later in the computation.

For example, suppose the current row is:

```
[1, 3, 3, 1]
```

When building the next row:

```
[1, 4, 6, 4, 1]
```

Updating from right to left ensures that the original `3` values remain available until they are fully used.

Finally, after all rows are generated, the function returns the resulting array.

## Go Solution

```
func getRow(rowIndex int) []int {
    row := []int{1}

    for i := 1; i <= rowIndex; i++ {
        row = append(row, 1)

        for j := i - 1; j >= 1; j-- {
            row[j] = row[j] + row[j-1]
        }
    }

    return row
}
```

The Go solution follows exactly the same algorithm as the Python version.

Slices in Go behave similarly to dynamic arrays, so we use `append` to grow the row by one element during each iteration.

Unlike Python, Go does not have built in list comprehensions or automatic resizing semantics beyond slices, but the overall logic remains identical.

There are no overflow concerns because the problem guarantees `rowIndex <= 33`, and all resulting values comfortably fit within Go's `int` type.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(rowIndex²) | Each row update processes up to `i` elements |
| Space | O(rowIndex) | Only one row array is stored |

The outer loop runs `rowIndex` times. Inside it, the inner loop may process nearly the entire row. The total number of operations becomes:

```
1 + 2 + 3 + ... + rowIndex
```

This arithmetic series evaluates to `O(rowIndex²)`.

The space usage is optimal because we only maintain a single array containing the current row, whose size is at most `rowIndex + 1`.

## Edge Cases

### Edge Case 1: `rowIndex = 0`

This is the smallest possible input. A buggy implementation might accidentally return an empty list or attempt unnecessary processing.

Our implementation handles this naturally because the initial array starts as `[1]`, and the loop does not execute when `rowIndex` is `0`.

### Edge Case 2: Small Rows Like `rowIndex = 1`

Rows with very few elements can expose off by one errors in loops.

For `rowIndex = 1`, the inner loop:

```
for j in range(i - 1, 0, -1)
```

does not execute, which is correct because there are no middle elements to update.

The final result correctly becomes:

```
[1, 1]
```

### Edge Case 3: In Place Updates

The most subtle issue is overwriting values too early.

Suppose we updated from left to right:

```
row[j] = row[j] + row[j - 1]
```

Then newly updated values would incorrectly affect later calculations.

For example:

```
Current row: [1, 3, 3, 1]
```

A left to right update would corrupt the row because modified values get reused immediately.

Updating from right to left avoids this problem entirely because every calculation still uses untouched values from the previous row state.
