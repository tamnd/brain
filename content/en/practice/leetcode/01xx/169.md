## Solution

## Problem Understanding

This problem asks us to find the **majority element** in an integer array. A majority element is defined as the value that appears **more than** `⌊n / 2⌋` times, where `n` is the size of the array.

In simpler terms, one particular number appears strictly more than half of the total array length, and we need to return that number.

The input consists of an integer array `nums`, and the output is a single integer representing the majority element. The problem guarantees that such an element **always exists**, which is a very important detail because it simplifies the solution significantly. We do not need to handle situations where no majority element exists.

The constraints tell us that:

- The array length can be as large as `5 * 10^4`
- Values may be negative or positive
- A valid majority element is guaranteed

Since `n` can grow to 50,000, inefficient approaches such as nested loops may become too slow. The follow-up explicitly asks for a solution in **linear time** and **constant extra space**, which strongly hints that a clever observation is needed rather than straightforward counting with extra memory.

Several edge cases are worth considering:

A single-element array such as `[7]` is an important case. Since one element appears once and `⌊1 / 2⌋ = 0`, it is automatically the majority element.

Arrays where the majority element appears interleaved with other values, such as `[2,1,2,1,2]`, can be tricky for naive logic because the majority is not clustered together.

Arrays with negative values, such as `[-1,-1,-1,2,3]`, remind us that the algorithm should work regardless of integer sign.

Fortunately, the guarantee that a majority element always exists eliminates ambiguity and allows us to rely on stronger mathematical reasoning.

## Approaches

### Brute Force Approach

A straightforward way to solve this problem is to count how many times each number appears.

For every element in the array, we can scan the entire array again and count its frequency. If a number appears more than `n / 2` times, we return it.

This approach is correct because it explicitly verifies the occurrence count of every candidate. Eventually, the majority element will be checked and identified.

However, the issue is efficiency. For each of the `n` elements, we potentially scan another `n` elements to count occurrences. This leads to **O(n²)** time complexity, which becomes inefficient as input size grows.

A slightly better counting solution would use a hash map to track frequencies in one pass, achieving **O(n)** time and **O(n)** space. While effective, it does not satisfy the follow-up requirement of constant extra space.

### Key Insight for the Optimal Solution

The crucial observation is that the majority element appears **more than half** the time. This means that if we repeatedly "cancel out" one occurrence of the majority element with one occurrence of a non-majority element, the majority element will still remain.

This idea leads to the **Boyer-Moore Voting Algorithm**.

Imagine pairing different elements together and removing them. Since the majority element appears more than all other elements combined, it cannot be fully canceled out.

The algorithm maintains:

- A **candidate** for the majority element
- A **count** representing confidence in that candidate

As we iterate:

- If the count becomes zero, we choose the current number as a new candidate.
- If the current number matches the candidate, we increment the count.
- Otherwise, we decrement the count because it cancels out one occurrence.

By the end of the traversal, the remaining candidate must be the majority element due to the problem guarantee.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Count occurrences for each element using nested loops |
| Optimal | O(n) | O(1) | Boyer-Moore Voting Algorithm, cancels non-majority elements |

## Algorithm Walkthrough

1. Initialize two variables: `candidate` and `count`. Set `count = 0`. The `candidate` will represent the current majority candidate.
2. Traverse the array from left to right.
3. If `count` becomes `0`, select the current number as the new `candidate`. This means the previous candidate has been fully canceled out.
4. Compare the current number with the `candidate`.

- If they match, increment `count` because this strengthens confidence in the candidate.
- If they differ, decrement `count` because this element cancels out one occurrence of the candidate.
5. Continue this process until the entire array has been processed.
6. Return the final `candidate`.

The reason this works is mathematical. Since the majority element appears more than half the time, it can never be completely canceled by all other elements combined. Eventually, it survives as the final candidate.

## Python Solution

```
from typing import List

class Solution:
    def majorityElement(self, nums: List[int]) -> int:
        candidate = 0
        count = 0

        for number in nums:
            if count == 0:
                candidate = number

            if number == candidate:
                count += 1
            else:
                count -= 1

        return candidate
```

The implementation follows the Boyer-Moore Voting Algorithm exactly.

We begin by initializing `candidate` and `count`. The `candidate` stores the currently suspected majority element, while `count` tracks the balance between matching and non-matching values.

During iteration, if `count` reaches zero, the current number becomes the new candidate. This happens because previous elements have effectively canceled each other out.

Next, we compare the current number with the candidate. Matching values increase confidence, so `count` increases. Different values reduce confidence, so `count` decreases.

At the end of the loop, the surviving candidate is returned. Since the problem guarantees that a majority element always exists, no verification pass is necessary.

## Go Solution

```
func majorityElement(nums []int) int {
    candidate := 0
    count := 0

    for _, num := range nums {
        if count == 0 {
            candidate = num
        }

        if num == candidate {
            count++
        } else {
            count--
        }
    }

    return candidate
}
```

The Go implementation mirrors the Python solution closely. Go uses a `range` loop to iterate through the slice, and integer variables are initialized using short variable declarations.

There are no concerns about integer overflow because the `count` variable never exceeds `n`, which is at most `5 * 10^4`. Since the problem guarantees at least one element exists, we do not need additional handling for empty slices.

Unlike Python lists, Go slices may technically be `nil`, but LeetCode guarantees valid input with at least one element, so no special nil handling is required.

## Worked Examples

### Example 1

**Input:** `nums = [3,2,3]`

We track the values of `candidate` and `count` at each step.

| Index | Number | Candidate | Count | Explanation |
| --- | --- | --- | --- | --- |
| 0 | 3 | 3 | 1 | Count was 0, choose 3 as candidate |
| 1 | 2 | 3 | 0 | Different number cancels candidate |
| 2 | 3 | 3 | 1 | Count was 0, choose 3 again |

Final result: **3**

### Example 2

**Input:** `nums = [2,2,1,1,1,2,2]`

| Index | Number | Candidate | Count | Explanation |
| --- | --- | --- | --- | --- |
| 0 | 2 | 2 | 1 | Count was 0, choose 2 |
| 1 | 2 | 2 | 2 | Same as candidate |
| 2 | 1 | 2 | 1 | Different number cancels |
| 3 | 1 | 2 | 0 | Another cancellation |
| 4 | 1 | 1 | 1 | Count reset, choose 1 |
| 5 | 2 | 1 | 0 | Cancellation |
| 6 | 2 | 2 | 1 | Count reset, choose 2 |

Final result: **2**

Even though the candidate temporarily changes during execution, the true majority element ultimately survives.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | We traverse the array exactly once |
| Space | O(1) | Only two extra variables are used |

The algorithm runs in linear time because every element is processed once. No nested loops or repeated scans are needed.

The space complexity is constant because the solution only maintains two integer variables, `candidate` and `count`, regardless of input size. No auxiliary data structures such as hash maps are required.

## Edge Cases

### Single Element Array

Consider an input like `[10]`. Since there is only one element, it automatically appears more than `⌊1 / 2⌋ = 0` times. The algorithm correctly assigns it as the candidate when `count == 0`, increments the count, and returns it.

### Majority Element Appears Interleaved

An input such as `[1,2,1,2,1]` can cause issues for naive approaches that rely on contiguous sequences. Here, the majority element is distributed throughout the array rather than grouped together.

The Boyer-Moore algorithm handles this naturally because it relies on cancellation rather than position. Matching values eventually outnumber opposing values.

### Majority Element Barely Exceeds Half

Consider `[3,3,4,2,4,4,2,4,4]`.

The majority element appears only slightly more than half the time, making it easy to lose track using simplistic logic. However, the cancellation principle guarantees correctness because the majority element still outnumbers all non-majority elements combined.

### Negative Numbers

Inputs such as `[-1,-1,-1,2,3]` work without modification. The algorithm only compares equality and never depends on ordering or positivity, so negative values are handled identically to positive ones.
