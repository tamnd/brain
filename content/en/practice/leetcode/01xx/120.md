## Solution

## Problem Understanding

The problem gives us a triangular array of integers and asks for the **minimum path sum from the top row to the bottom row**. Starting from the top element, at each row we may move only to one of the two adjacent positions directly below. If we are currently at index `i` in a row, the next move can go to either index `i` or index `i + 1` in the following row.

In simpler terms, we need to find the cheapest path from the top of the triangle to the bottom while following the movement rule. The answer should be the smallest possible sum of all numbers visited along such a path.

For example, in:

```
   2
  3 4
 6 5 7
4 1 8 3
```

Starting at `2`, we can move to either `3` or `4`. If we choose `3`, we may then move to `6` or `5`, and so on. Among all valid paths, the minimum path is:

```
2 → 3 → 5 → 1
```

which gives a sum of `11`.

The input is a list of lists where:

- `triangle[i]` represents the `i-th` row.
- Each row has exactly one more element than the previous row.
- Values may be positive, zero, or negative.

The constraints tell us important information about the input size:

- The triangle contains at most `200` rows.
- Each value can range from `-10^4` to `10^4`.

Although `200` rows is not enormous, the number of possible paths grows exponentially because every position offers two choices. This means a brute-force search would quickly become too slow.

There are also several important edge cases to consider. The triangle may contain only one element, meaning the answer is simply that value. Values can be negative, so greedy decisions such as always taking the smaller immediate number will not necessarily work. The minimum path may involve temporarily taking a larger number to enable a better future path.

## Approaches

### Brute Force Approach

A straightforward approach is to recursively explore **every possible path** from the top to the bottom.

At each position `(row, col)`, we have two choices:

1. Move to `(row + 1, col)`
2. Move to `(row + 1, col + 1)`

We recursively calculate the minimum sum for both choices and return the smaller result plus the current value.

This works because it evaluates all valid paths, guaranteeing that the minimum one will eventually be found. However, it is extremely inefficient because the same subproblems are recomputed many times.

For example, while computing paths from row `0`, multiple recursive branches may repeatedly calculate the minimum path from the same position such as `(2,1)`.

Since every row roughly doubles the number of possible paths, the time complexity becomes exponential, which is too slow for `200` rows.

### Key Insight for an Optimal Solution

The crucial observation is that **the minimum path from a cell depends only on the minimum paths of the two cells below it**.

Suppose we know the minimum path sum from every position in the last row upward. Then for any cell:

```
minimum_path(row, col) =
triangle[row][col] +
min(below, diagonal_below)
```

This naturally suggests **dynamic programming**.

Instead of recomputing results repeatedly, we can compute them once and reuse them.

A particularly elegant optimization is to work **bottom-up**. We start from the last row and gradually collapse the triangle upward.

We maintain a `dp` array where:

```
dp[j]
```

represents the minimum path sum from the current row to the bottom.

Initially, `dp` is simply the last row because the minimum path from the last row to itself is the value itself.

Then, moving upward row by row:

```
dp[j] = triangle[i][j] + min(dp[j], dp[j + 1])
```

At the end, `dp[0]` contains the answer.

This achieves the follow-up requirement of using only `O(n)` extra space, where `n` is the number of rows.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | `O(2^n)` | `O(n)` | Explores every possible path recursively |
| Optimal Dynamic Programming | `O(n²)` | `O(n)` | Bottom-up DP using a single array |

## Algorithm Walkthrough

1. **Initialize a DP array with the last row of the triangle**

Since the last row has no children beneath it, the minimum path sum from each position is simply the value itself. We copy the last row into a dynamic programming array called `dp`.
2. **Process rows from bottom to top**

Starting from the second-to-last row, move upward one row at a time.

We process from bottom to top because each cell depends on information from the row below. By the time we process a row, the required values are already computed.
3. **Update each position in the current row**

For every index `j` in the row:

```
dp[j] =
triangle[i][j] + min(dp[j], dp[j + 1])
```

Here:

- `dp[j]` represents the minimum path from the cell directly below.
- `dp[j + 1]` represents the minimum path from the diagonal-right cell below.

We choose the smaller of the two because we want the minimum possible total path.
4. **Reuse the same DP array**

Instead of creating a new array for every row, we overwrite the existing `dp` array.

This works because each update only depends on values from the row below, which have not yet been overwritten when processing left to right.
5. **Return the top result**

After processing all rows, `dp[0]` stores the minimum path sum from the top to the bottom.

## Python Solution

```
from typing import List

class Solution:
    def minimumTotal(self, triangle: List[List[int]]) -> int:
        dp = triangle[-1][:]

        for row in range(len(triangle) - 2, -1, -1):
            for col in range(len(triangle[row])):
                dp[col] = triangle[row][col] + min(
                    dp[col],
                    dp[col + 1]
                )

        return dp[0]
```

The implementation begins by copying the final row into `dp`. We create a copy using slicing (`[:]`) so that we do not accidentally modify the original triangle.

Next, we iterate upward from the second-to-last row to the top. For every element in the current row, we compute the minimum path sum by adding the current value to the smaller of the two reachable child values already stored in `dp`.

Because `dp` always represents the minimum path sums for the row below, overwriting it in place is safe and memory efficient.

After all rows are processed, `dp[0]` contains the minimum total path sum from the top of the triangle.

## Go Solution

```
func minimumTotal(triangle [][]int) int {
    rows := len(triangle)

    dp := make([]int, len(triangle[rows-1]))
    copy(dp, triangle[rows-1])

    for row := rows - 2; row >= 0; row-- {
        for col := 0; col < len(triangle[row]); col++ {
            if dp[col] < dp[col+1] {
                dp[col] = triangle[row][col] + dp[col]
            } else {
                dp[col] = triangle[row][col] + dp[col+1]
            }
        }
    }

    return dp[0]
}
```

The Go solution follows the same bottom-up dynamic programming approach.

Since Go does not provide a built-in `min()` function for integers, we manually compare `dp[col]` and `dp[col+1]`.

We also explicitly allocate a slice for `dp` and use `copy()` to duplicate the last row. This avoids mutating the original input.

Integer overflow is not a concern here because the constraints are small enough that the total path sum comfortably fits inside Go's `int` type.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | `O(n²)` | Every element in the triangle is processed exactly once |
| Space | `O(n)` | The DP array stores one row, where `n` is the number of rows |

The time complexity is quadratic because a triangle with `n` rows contains approximately `n(n + 1) / 2` elements. Since each element is visited exactly once, the total work is proportional to the number of cells.

The space complexity is `O(n)` because we only keep a single dynamic programming array whose size matches the bottom row, which is at most the number of rows.

## Edge Cases

### Single Element Triangle

A triangle such as:

```
[[-10]]
```

can easily break implementations that assume at least two rows exist. Our solution handles this naturally because `dp` becomes `[-10]`, and the loop never executes. We simply return `dp[0]`.

### Negative Numbers

The triangle may contain negative values:

```
[[2],
 [-3, 4],
 [6, -5, 7]]
```

A greedy approach that always picks the smaller immediate number may fail because future choices matter. Dynamic programming works correctly because it evaluates the full minimum path from every possible position.

### Large Triangle Depth

A triangle with `200` rows contains many possible paths. A recursive brute-force solution would become exponentially slow and potentially hit recursion limits. The bottom-up DP approach avoids recursion entirely and processes each element only once, remaining efficient even at maximum constraints.
