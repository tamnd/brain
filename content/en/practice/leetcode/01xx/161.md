## Solution

## Problem Understanding

This problem asks us to determine whether two strings are **exactly one edit apart**.

An edit can be one of three operations:

1. Insert exactly one character into `s` to make it equal to `t`
2. Delete exactly one character from `s` to make it equal to `t`
3. Replace exactly one character in `s` with a different character to make it equal to `t`

The key word here is **exactly**. This means:

- If the strings are already identical, the answer is `false`, because zero edits are required.
- If transforming one string into another requires two or more edits, the answer is also `false`.
- Only a single valid modification is allowed.

The input consists of two strings, `s` and `t`, whose lengths range from `0` to `10^4`. Since the strings can be fairly large, we should aim for an efficient linear time solution instead of repeatedly reconstructing strings or testing many possibilities.

There are several important observations and edge cases to notice immediately.

If the length difference between the strings is greater than `1`, then it is impossible for them to be one edit apart. A single insertion or deletion can only change length by one, and replacement keeps length unchanged.

If the strings are identical, the answer must be `false` because no edit is needed.

Another tricky scenario occurs when the strings differ only at the very end. For example:

```
s = "abc"
t = "abcd"
```

A naive mismatch-based implementation may fail to account for this final extra character. We must explicitly handle it.

We must also distinguish between the three edit types:

- Same length → replacement only
- Length differs by one → insertion/deletion only
- Length difference greater than one → impossible

These cases guide the design of an efficient algorithm.

## Approaches

### Brute Force Approach

A straightforward approach is to explicitly try every possible edit operation and check whether it transforms one string into the other.

If the strings have equal length, we can attempt replacing each character in `s` with every possible alternative character and see whether it becomes `t`.

If the lengths differ by one, we can try inserting or deleting a character at every possible position and compare the resulting string.

This works because it exhaustively explores every valid one-edit transformation. If any transformation produces the target string, then the strings are one edit apart.

However, this approach is inefficient. For every position, we may construct new strings, and each construction can take linear time. Since we potentially try many positions, the overall runtime becomes quadratic in the worst case.

### Optimal Approach

The key insight is that we do not need to simulate all edits.

Since only **one edit** is allowed, the first mismatch tells us almost everything we need to know.

We compare the strings character by character:

- If lengths are equal, a mismatch means we must perform exactly one replacement, so the remaining suffixes must match.
- If one string is longer by one character, a mismatch means we skip one character in the longer string and continue checking.

This naturally leads to a **two pointer** solution.

We scan both strings from left to right and allow at most one mismatch. Once a mismatch occurs, we adjust pointers depending on the string lengths.

Because each character is visited at most once, this gives us a linear time solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Try every insertion, deletion, or replacement explicitly |
| Optimal | O(n) | O(1) | Single pass with two pointers and mismatch handling |

## Algorithm Walkthrough

1. First, compute the lengths of `s` and `t`.

If their length difference is greater than `1`, immediately return `false`. One edit cannot bridge a gap larger than one character.
2. Ensure `s` is the shorter string.

If `s` is longer than `t`, swap them. This simplifies the logic because we only need to handle insertion/deletion in one direction.
3. Initialize two pointers.

One pointer tracks the current index in `s`, and another tracks the current index in `t`.
4. Traverse both strings simultaneously.

While both pointers are within bounds, compare the current characters.
5. Handle matching characters.

If the characters are equal, advance both pointers because no edit is needed at this position.
6. Handle the first mismatch.

If a mismatch occurs, we must use our single allowed edit.

- If lengths are equal, this must be a replacement. Compare the remaining substrings after skipping one character in both strings.
- If lengths differ by one, this must be an insertion or deletion. Compare the remainder after skipping one character only in the longer string.
7. If no mismatch occurs during traversal.

This means every overlapping character matched. The only possible valid edit is an extra trailing character in the longer string.

Return `true` only if the lengths differ by exactly one.

This approach works because a valid one-edit transformation can only create at most one structural deviation between the strings.

## Python Solution

```
class Solution:
    def isOneEditDistance(self, s: str, t: str) -> bool:
        len_s = len(s)
        len_t = len(t)

        if abs(len_s - len_t) > 1:
            return False

        # Ensure s is the shorter string
        if len_s > len_t:
            return self.isOneEditDistance(t, s)

        for index in range(len_s):
            if s[index] != t[index]:
                # Same length: replacement case
                if len_s == len_t:
                    return s[index + 1:] == t[index + 1:]

                # Different length: insertion/deletion case
                return s[index:] == t[index + 1:]

        # No mismatch found
        return len_s + 1 == len_t
```

### Code Explanation

The solution starts by calculating the lengths of both strings. If their difference exceeds one, we immediately know they cannot be transformed using a single edit.

Next, we ensure `s` is always the shorter string. This reduces branching logic because we only need to handle one insertion/deletion direction.

The main loop scans through the shorter string character by character.

When characters match, we continue scanning.

At the first mismatch, we decide the edit type based on string lengths.

If the lengths are equal, we simulate a replacement by checking whether the remaining portions of both strings match after skipping one character in each string.

If the lengths differ by one, we simulate insertion/deletion by skipping one character only in the longer string.

If the loop finishes without finding any mismatch, then the strings match entirely across their shared length. The only way they can still be one edit apart is if the longer string contains exactly one additional trailing character.

## Go Solution

```
func isOneEditDistance(s string, t string) bool {
	lenS := len(s)
	lenT := len(t)

	if abs(lenS-lenT) > 1 {
		return false
	}

	// Ensure s is the shorter string
	if lenS > lenT {
		return isOneEditDistance(t, s)
	}

	for i := 0; i < lenS; i++ {
		if s[i] != t[i] {
			// Replacement case
			if lenS == lenT {
				return s[i+1:] == t[i+1:]
			}

			// Insertion/deletion case
			return s[i:] == t[i+1:]
		}
	}

	// No mismatch found
	return lenS+1 == lenT
}

func abs(x int) int {
	if x < 0 {
		return -x
	}
	return x
}
```

### Go-Specific Implementation Notes

The Go implementation closely mirrors the Python version.

Unlike Python, Go does not provide a built-in `abs()` function for integers, so we implement a small helper function.

String slicing in Go is efficient because slices reference the original underlying data rather than copying the full string. This keeps the implementation efficient while still making the suffix comparison clean and readable.

Go strings are byte-based. Since the problem guarantees only letters and digits, byte indexing is safe and avoids Unicode complications.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | We scan the strings at most once |
| Space | O(1) | Only a few variables are used |

The runtime is linear because we only traverse the strings once until the first mismatch or the end of the shorter string. Although substring comparisons appear in the implementation, each character is still processed at most once overall, keeping the complexity linear.

The space complexity is constant because no auxiliary data structures proportional to input size are created.

## Edge Cases

### Identical Strings

Consider:

```
s = "abc"
t = "abc"
```

A common mistake is returning `true` because the strings are "close." However, the problem requires **exactly one** edit, not zero edits.

Our implementation correctly returns `false` because no mismatch occurs and the final length check fails.

### Length Difference Greater Than One

Consider:

```
s = "abc"
t = "abcde"
```

The length difference is `2`, meaning at least two insertions are required.

The algorithm immediately rejects this case using:

```
abs(len(s) - len(t)) > 1
```

This avoids unnecessary processing.

### Extra Character at the End

Consider:

```
s = "abc"
t = "abcd"
```

This is a subtle case because all compared characters match.

A naive mismatch-only approach may incorrectly return `false`.

Our implementation handles this using the final condition:

```
len_s + 1 == len_t
```

Since `t` contains exactly one additional trailing character, the result is `true`.

### Multiple Differences

Consider:

```
s = "abc"
t = "axy"
```

The first mismatch occurs at index `1`.

We compare the remaining suffixes:

```
"c" == "y"
```

This fails, correctly identifying that more than one edit is required.

The implementation immediately returns `false` without continuing unnecessary comparisons.
