## Solution

## Problem Understanding

This problem asks us to convert a fraction, represented by a `numerator` and `denominator`, into its decimal string representation. If the decimal expansion repeats infinitely, we must detect the repeating portion and enclose it in parentheses.

For example:

- `1 / 2 = 0.5`, which terminates, so the result is simply `"0.5"`.
- `4 / 333 = 0.012012012...`, where `"012"` repeats forever, so the result becomes `"0.(012)"`.

The input consists of two integers:

- `numerator`, the top part of the fraction
- `denominator`, the bottom part of the fraction

The output must be a string representation of the decimal value.

The main challenge is identifying when the fractional portion begins repeating. In ordinary long division, repetition starts when the same remainder appears again. Once a remainder repeats, the decimal digits generated after its first appearance will repeat in the exact same pattern.

The constraints are important:

- Both values can be as small as `-2^31` and as large as `2^31 - 1`
- `denominator != 0`
- The answer length is guaranteed to be less than `10^4`

These constraints tell us that we must carefully handle integer overflow, especially for edge cases like `-2^31`, whose absolute value cannot fit in a signed 32-bit integer. We also need to correctly process negative fractions and distinguish between terminating and repeating decimals.

Several edge cases can easily break a naive implementation:

- Negative numbers, such as `-1 / 2`
- Both numbers negative, such as `-1 / -2`
- Zero numerator, such as `0 / 5`
- Repeating decimals, such as `1 / 3`
- Large values near integer limits, especially `-2^31`

A correct implementation must account for all of these.

## Approaches

### Brute Force Approach

A brute force approach would simulate decimal expansion digit by digit without tracking previously seen remainders. We could repeatedly multiply the remainder by `10`, compute the next digit, and append it to the result.

This works for terminating decimals because the remainder eventually becomes `0`. However, for repeating decimals such as `1 / 3`, the remainder sequence repeats forever:

```
1 → 10 → remainder 1 → 10 → remainder 1 ...
```

Without detecting cycles, the algorithm would never stop. One could artificially cap the number of digits, but that would not produce the exact required format with parentheses.

Therefore, brute force is not practical for repeating decimals.

### Optimal Approach, Long Division with Remainder Tracking

The key observation is that repeating decimals occur when a remainder repeats.

During long division:

1. Divide the current remainder by the denominator.
2. Record the quotient digit.
3. Update the remainder.

If we ever encounter the same remainder again, we know the decimal digits from its first occurrence onward are repeating.

To detect this efficiently, we use a hash map:

```
remainder → index in result string
```

When a remainder repeats:

- Insert `"("` at the stored index
- Append `")"` at the end

This allows us to construct the exact repeating decimal representation in linear time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Infinite in worst case | O(1) | Fails on repeating decimals because cycles are not detected |
| Optimal | O(n) | O(n) | Uses a hash map to track previously seen remainders |

Here, `n` represents the number of generated decimal digits.

## Algorithm Walkthrough

1. **Handle the zero numerator case**

If `numerator == 0`, the fraction evaluates to `"0"` regardless of the denominator.
2. **Determine the sign of the result**

The result is negative only if exactly one of `numerator` or `denominator` is negative.

We can use XOR logic:

```
numerator < 0 XOR denominator < 0
```
3. **Convert numbers to positive values**

To simplify division logic, convert both values to absolute values.

This also prevents overflow issues in languages like Go when using larger integer types.
4. **Compute the integer portion**

Perform integer division:

```
integer_part = numerator // denominator
remainder = numerator % denominator
```

Add the integer portion to the result.
5. **Return early if there is no fractional part**

If `remainder == 0`, the decimal terminates, so we return immediately.
6. **Start processing the fractional part**

Append `"."` to the result.
7. **Track remainders using a hash map**

Create a map:

```
remainder → position in result
```

Before generating each decimal digit, check whether the current remainder has appeared before.
8. **Generate decimal digits using long division**

Repeat until remainder becomes `0`:

1. If remainder already exists in the map, a cycle is found.
2. Insert `"("` at the stored position.
3. Append `")"` and stop.
4. Otherwise, record the remainder position.
5. Multiply remainder by `10`.
6. Append the next quotient digit.
7. Update the remainder.
9. **Return the final string**

The result now contains either a terminating decimal or a repeating section enclosed in parentheses.

## Python Solution

```
class Solution:
    def fractionToDecimal(self, numerator: int, denominator: int) -> str:
        if numerator == 0:
            return "0"

        result = []

        # Determine sign
        if (numerator < 0) ^ (denominator < 0):
            result.append("-")

        numerator = abs(numerator)
        denominator = abs(denominator)

        # Integer part
        integer_part = numerator // denominator
        result.append(str(integer_part))

        remainder = numerator % denominator

        # No fractional part
        if remainder == 0:
            return "".join(result)

        result.append(".")

        # remainder -> position in result
        seen_remainders = {}

        while remainder != 0:
            if remainder in seen_remainders:
                repeat_index = seen_remainders[remainder]
                result.insert(repeat_index, "(")
                result.append(")")
                break

            seen_remainders[remainder] = len(result)

            remainder *= 10
            digit = remainder // denominator
            result.append(str(digit))
            remainder %= denominator

        return "".join(result)
```

The implementation begins by handling the special case where the numerator is `0`. Since any fraction with numerator `0` evaluates to zero, we immediately return `"0"`.

Next, we determine whether the result should be negative. A fraction is negative only when exactly one operand is negative, which is efficiently checked using XOR.

We then convert both values to their absolute values to simplify long division. The integer part is computed using integer division and immediately appended to the result.

The remainder determines whether a fractional part exists. If the remainder is `0`, the decimal terminates and we can return the result immediately.

For repeating decimals, we append `"."` and start long division. The `seen_remainders` dictionary records where each remainder first appeared in the output. Before generating a new digit, we check whether the current remainder has already been seen. If it has, repetition begins at the stored position, so we insert `"("` there and append `")"` at the end.

Using a list for `result` is important because string concatenation inside a loop would be inefficient. At the end, `"".join(result)` converts the pieces into the final string.

## Go Solution

```
package main

import (
	"strconv"
	"strings"
)

func fractionToDecimal(numerator int, denominator int) string {
	if numerator == 0 {
		return "0"
	}

	var result strings.Builder

	// Determine sign
	if (numerator < 0) != (denominator < 0) {
		result.WriteString("-")
	}

	// Use int64 to avoid overflow
	num := int64(numerator)
	den := int64(denominator)

	if num < 0 {
		num = -num
	}
	if den < 0 {
		den = -den
	}

	// Integer part
	integerPart := num / den
	result.WriteString(strconv.FormatInt(integerPart, 10))

	remainder := num % den

	// No fractional part
	if remainder == 0 {
		return result.String()
	}

	result.WriteString(".")

	// Since we may need insertion for parentheses,
	// build fractional part separately
	fraction := []byte{}
	seenRemainders := make(map[int64]int)

	for remainder != 0 {
		if pos, exists := seenRemainders[remainder]; exists {
			fraction = append(fraction[:pos], append([]byte{'('}, fraction[pos:]...)...)
			fraction = append(fraction, ')')
			break
		}

		seenRemainders[remainder] = len(fraction)

		remainder *= 10
		digit := remainder / den
		fraction = append(fraction, byte('0'+digit))
		remainder %= den
	}

	result.Write(fraction)

	return result.String()
}
```

The Go implementation follows the same algorithm but introduces a few language-specific adjustments.

The most important difference is integer overflow handling. Since the input range includes `-2^31`, taking the absolute value of a 32-bit integer can overflow. To avoid this, the implementation converts values to `int64` before processing.

Instead of building the entire result using string concatenation, the solution uses `strings.Builder`, which is more memory efficient for repeated appends.

Because Go strings are immutable and inserting characters into the middle is cumbersome, the fractional portion is stored separately as a `[]byte`. When a repeating cycle is found, `"("` can be inserted at the correct index before appending `")"`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each unique remainder is processed at most once |
| Space | O(n) | The hash map stores previously seen remainders |

The key observation is that there are only finitely many possible remainders, specifically fewer than the denominator. Each remainder is processed at most once before either reaching `0` or repeating. Therefore, the runtime is linear in the number of generated digits. The hash map storing remainders also grows proportionally to the number of digits produced.

## Edge Cases

### Zero Numerator

If `numerator = 0`, the mathematical result is always `0`, regardless of the denominator. This case can easily be overlooked, causing unnecessary processing or malformed output such as `"0."`. The implementation immediately returns `"0"`.

### Negative Numbers

Fractions involving negative values must produce the correct sign. Examples include:

```
-1 / 2 = "-0.5"
1 / -2 = "-0.5"
-1 / -2 = "0.5"
```

A common bug is incorrectly handling double negatives. The implementation uses XOR logic to determine negativity, ensuring the result is negative only when exactly one input is negative.

### Repeating Decimals

Fractions like `1 / 3` produce infinitely repeating decimals:

```
0.333333...
```

Without remainder tracking, the algorithm would loop forever. By storing the first occurrence of each remainder, the implementation detects cycles and inserts parentheses correctly:

```
"0.(3)"
```

### Large Integer Limits

Values near `-2^31` can cause overflow when taking absolute values in some languages. For example:

```
numerator = -2147483648
```

In Go, converting to `int64` before taking the absolute value prevents overflow and ensures correctness for all valid inputs.
