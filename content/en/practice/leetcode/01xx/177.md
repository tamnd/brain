## Solution

## Problem Understanding

This problem asks us to write a SQL query that returns the `nth` highest distinct salary from the `Employee` table.

The table contains two columns:

| Column | Meaning |
| --- | --- |
| `id` | Unique employee identifier |
| `salary` | Employee salary |

The important detail is that the salary ranking must be based on distinct salary values, not individual employees. If multiple employees share the same salary, that salary should only count once in the ranking.

For example, if the salaries are:

```
100, 200, 200, 300
```

the distinct salaries are:

```
100, 200, 300
```

The second highest distinct salary is therefore `200`.

The query must return:

```
getNthHighestSalary(n)
```

as the column name.

If the table contains fewer than `n` distinct salaries, the query must return `null`.

This is important because a naive solution might accidentally return no rows instead of explicitly returning `null`.

The input consists of:

1. The `Employee` table
2. An integer `n`

The output is a single value representing the `nth` highest distinct salary.

Since this is a database problem, the primary challenge is not computational complexity in the traditional algorithmic sense, but writing a correct SQL query that handles duplicates and missing rankings properly.

Several edge cases are important:

- Multiple employees may share the same salary
- The table may contain only one employee
- `n` may be larger than the number of distinct salaries
- Salaries may not be sorted
- There may be gaps between salary values

A correct solution must handle all of these cases cleanly.

## Approaches

### Brute Force Approach

A brute force approach would first retrieve all salaries, sort them in descending order, remove duplicates manually, and then select the `nth` value.

Conceptually, the process looks like this:

1. Read every salary from the table
2. Sort salaries descending
3. Remove duplicate salary values
4. Count positions until reaching the `nth` distinct salary
5. Return the result

This approach is logically correct because sorting guarantees ranking order, and removing duplicates ensures distinct salary handling.

However, this method is inefficient and unnecessarily verbose in SQL. It often requires temporary tables, procedural logic, or multiple nested queries.

In large datasets, sorting all rows and manually processing duplicates becomes more expensive than necessary.

### Optimal Approach

The key observation is that SQL already provides powerful tools for ranking and limiting results.

We can:

1. Select distinct salaries
2. Sort them in descending order
3. Skip the first `n - 1` salaries
4. Return the next salary

The `LIMIT` and `OFFSET` clauses make this straightforward.

The query:

```
SELECT DISTINCT salary
FROM Employee
ORDER BY salary DESC
LIMIT 1 OFFSET n - 1
```

returns exactly the `nth` highest distinct salary.

The remaining challenge is handling cases where the requested rank does not exist. We solve this by wrapping the query inside a function and returning `NULL` automatically when no row is found.

This approach is concise, efficient, and leverages SQL's built in sorting and filtering mechanisms.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) | O(n) | Sort all salaries and manually remove duplicates |
| Optimal | O(n log n) | O(n) | Uses DISTINCT + ORDER BY + LIMIT/OFFSET efficiently |

## Algorithm Walkthrough

1. First, select only distinct salary values from the `Employee` table. This ensures duplicate salaries are counted only once.
2. Sort the distinct salaries in descending order so the highest salary appears first.
3. Use `OFFSET n - 1` to skip the first `n - 1` salaries. Since indexing starts from zero in SQL offsets, this positions the query exactly at the `nth` salary.
4. Use `LIMIT 1` to return only one row, which is the desired salary.
5. Wrap the query inside the required SQL function format so LeetCode can execute it correctly.
6. If the offset exceeds the number of available distinct salaries, SQL returns no row, which becomes `NULL` automatically.

## Python Solution

Although this is a SQL database problem, LeetCode expects a SQL function rather than executable Python code. The following is the correct MySQL solution.

```
CREATE FUNCTION getNthHighestSalary(N INT) RETURNS INT
BEGIN
  RETURN (
      SELECT DISTINCT salary
      FROM Employee
      ORDER BY salary DESC
      LIMIT 1 OFFSET N - 1
  );
END
```

The implementation begins by defining a SQL function named `getNthHighestSalary`.

Inside the function, we execute a query against the `Employee` table.

The `DISTINCT` keyword removes duplicate salary values. This is essential because the problem specifically asks for distinct salary rankings.

The `ORDER BY salary DESC` clause sorts salaries from highest to lowest.

The `OFFSET N - 1` skips the first `N - 1` salaries. For example:

- If `N = 1`, offset becomes `0`, so we return the highest salary.
- If `N = 2`, offset becomes `1`, so we skip the highest salary and return the second highest.

Finally, `LIMIT 1` ensures only one salary is returned.

If the query cannot find a matching row, the function returns `NULL`, which satisfies the problem requirements.

## Edge Cases

### Duplicate Salaries

A common mistake is counting duplicate salaries multiple times.

Consider:

```
100, 200, 200, 300
```

The second highest distinct salary is `200`, not the second occurrence of `200`.

Using `DISTINCT` ensures duplicates are removed before ranking occurs.

### Fewer Than N Distinct Salaries

If the table contains fewer distinct salaries than requested, the query must return `NULL`.

For example:

```
Salaries: 100
n = 2
```

After applying `OFFSET 1`, no rows remain.

The SQL function correctly returns `NULL` automatically.

### Unsorted Input Data

The salaries in the table may appear in any order.

For example:

```
300, 100, 200
```

Without sorting, selecting the `nth` salary would produce incorrect results.

Using:

```
ORDER BY salary DESC
```

guarantees proper ranking regardless of insertion order.
