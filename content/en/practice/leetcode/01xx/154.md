# LeetCode 154, Find Minimum in Rotated Sorted Array II

## Problem Understanding

This problem gives us an array that was originally sorted in ascending order, then rotated some number of times. A rotation shifts elements from the end of the array to the beginning. Unlike the simpler version of this problem, the array may contain duplicate values.

The task is to return the minimum element in the array.

For example, consider the sorted array:

```
[0,1,4,4,5,6,7]
```

After rotation, it could become:

```
[4,5,6,7,0,1,4]
```

The minimum value is still `0`, but it is no longer at the beginning.

The important detail is that duplicates are allowed. This changes the behavior of binary search because equal values can make it impossible to determine which side is properly sorted.

The input consists of:

- `nums`, a rotated sorted array
- The array length is between `1` and `5000`
- Values may be negative
- Duplicates are allowed

The expected output is a single integer, the smallest value in the array.

The constraints are relatively small, so even an `O(n)` solution would pass. However, the problem explicitly asks us to reduce operation count as much as possible, which strongly suggests using binary search.

Several edge cases are important:

- The array may not actually appear rotated, such as `[1,2,3,4]`
- All elements may be identical, such as `[2,2,2,2]`
- Duplicates near the pivot may make the sorted side ambiguous
- The array may contain only one element
- The minimum may appear multiple times

These cases can break naive binary search implementations if duplicates are not handled carefully.

## Approaches

### Brute Force Approach

The simplest solution is to scan the entire array and track the smallest value encountered.

We initialize the minimum with the first element, then iterate through every element and update the minimum whenever we find a smaller value.

This works because every element is examined exactly once, so the true minimum cannot be missed.

Although this solution is straightforward and correct, it ignores the partially sorted structure of the rotated array. Its runtime is linear, which is less efficient than binary search for large inputs.

### Optimal Approach, Modified Binary Search

The key insight is that the array still contains sorted structure despite rotation.

In a rotated sorted array without duplicates, binary search can determine which half contains the minimum by comparing the middle element with the rightmost element.

With duplicates, however, equal values can hide the pivot location.

For example:

```
[2,2,2,0,1,2]
```

If `nums[mid] == nums[right]`, we cannot determine whether the minimum lies on the left or right side.

To handle this, we shrink the search space conservatively by decrementing `right` by one when equality occurs.

The logic becomes:

- If `nums[mid] < nums[right]`, the minimum is in the left half, including `mid`
- If `nums[mid] > nums[right]`, the minimum is in the right half, excluding `mid`
- If `nums[mid] == nums[right]`, we cannot determine the side, so reduce `right`

This preserves correctness while still often achieving logarithmic behavior.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Scan every element and track minimum |
| Optimal | O(log n) average, O(n) worst case | O(1) | Modified binary search that handles duplicates |

## Algorithm Walkthrough

1. Initialize two pointers, `left` and `right`, representing the current search range.
2. Continue searching while `left < right`. Once both pointers meet, that position must contain the minimum.
3. Compute the middle index:

```
mid = (left + right) // 2
```
4. Compare `nums[mid]` with `nums[right]`.
5. If `nums[mid] < nums[right]`, the right half is sorted and the minimum must lie at `mid` or to its left. Set:

```
right = mid
```
6. If `nums[mid] > nums[right]`, the minimum must lie strictly to the right of `mid`. Set:

```
left = mid + 1
```
7. If `nums[mid] == nums[right]`, duplicates prevent us from identifying the correct half. Since `nums[right]` is not uniquely useful, safely shrink the range:

```
right -= 1
```
8. When the loop finishes, `left` points to the minimum element.
9. Return `nums[left]`.

The algorithm uses only constant extra space because all work is done using index pointers.

## Python Solution

```
from typing import List

class Solution:
    def findMin(self, nums: List[int]) -> int:
        left = 0
        right = len(nums) - 1

        while left < right:
            mid = (left + right) // 2

            if nums[mid] < nums[right]:
                right = mid
            elif nums[mid] > nums[right]:
                left = mid + 1
            else:
                right -= 1

        return nums[left]
```

The implementation begins by defining the search boundaries using `left` and `right`.

Inside the loop, the midpoint is calculated. The algorithm then compares the middle element with the rightmost element to determine which side may contain the minimum.

When the middle value is smaller than the right value, the minimum must lie in the left portion, including `mid`, so `right` moves to `mid`.

When the middle value is greater than the right value, the rotation pivot must lie to the right of `mid`, so `left` becomes `mid + 1`.

The tricky case occurs when both values are equal. In that situation, duplicates prevent us from deciding which half contains the minimum. Reducing `right` by one safely removes a duplicate without losing the minimum.

Eventually, the search range collapses to a single index, and that index contains the smallest element.

## Go Solution

```
func findMin(nums []int) int {
    left := 0
    right := len(nums) - 1

    for left < right {
        mid := left + (right-left)/2

        if nums[mid] < nums[right] {
            right = mid
        } else if nums[mid] > nums[right] {
            left = mid + 1
        } else {
            right--
        }
    }

    return nums[left]
}
```

The Go implementation follows the same logic as the Python solution.

One small difference is the midpoint calculation:

```
mid := left + (right-left)/2
```

This style avoids potential integer overflow in languages with fixed integer sizes, although overflow is not an issue under this problem's constraints.

Go slices are used directly, and no extra memory allocations are needed.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(log n) average, O(n) worst case | Binary search usually halves the search space, but duplicates may force one-by-one shrinking |
| Space | O(1) | Only a few pointer variables are used |

In most cases, the algorithm behaves like standard binary search and eliminates half of the remaining elements at each step.

However, when many duplicates exist, especially when `nums[mid] == nums[right]` repeatedly, the algorithm cannot determine which half to discard. In that scenario, it may reduce the search range by only one element at a time, resulting in worst case `O(n)` behavior.

The space complexity remains constant because no additional data structures are allocated.

## Edge Cases

### Single Element Array

An input such as:

```
[7]
```

contains only one value. The loop condition `left < right` is immediately false, so the algorithm directly returns the only element.

This avoids unnecessary comparisons and correctly handles the smallest possible input size.

### Array With No Visible Rotation

Consider:

```
[1,2,3,4,5]
```

Even though the problem states the array was rotated, rotating by `n` positions produces the same sorted array.

The algorithm still works because every comparison shows the left side contains the minimum, gradually shrinking the range toward index `0`.

### All Elements Identical

Consider:

```
[2,2,2,2,2]
```

This is the hardest case for the binary search logic because comparisons provide no directional information.

The implementation handles this safely using:

```
right -= 1
```

This guarantees progress without accidentally discarding the minimum value.

### Duplicates Around the Pivot

Consider:

```
[10,1,10,10,10]
```

A naive binary search might incorrectly discard the left side because many values are equal.

The duplicate handling logic prevents this mistake by shrinking cautiously whenever equality prevents a clear decision.

Eventually, the algorithm isolates the true minimum value `1`.
