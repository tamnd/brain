## Solution

## Problem Understanding

This problem asks us to maximize profit from stock trading over a sequence of days. The input array `prices` represents the stock price on each day, where `prices[i]` is the price on day `i`.

Unlike the simpler stock trading problem where only one transaction is allowed, this version allows multiple transactions. You may buy and sell as many times as you want, with one important restriction: you can only hold at most one share of stock at a time. This means you must sell your current stock before buying again.

The statement also mentions that buying and selling can happen on the same day. In practice, this means transactions do not require a cooldown period, and consecutive profitable movements can all be captured.

The goal is to compute the maximum total profit achievable by making optimal buy and sell decisions throughout the timeline.

The constraints are important:

- `1 <= prices.length <= 3 * 10^4`
- `0 <= prices[i] <= 10^4`

The array can contain up to 30,000 elements, so exponential or quadratic approaches become impractical. This strongly suggests that we need a linear or near linear solution.

Several edge cases are important to identify early:

- If prices are always decreasing, the best strategy is to make no transactions at all.
- If prices are always increasing, the best strategy is effectively to capture the entire rise.
- If the array contains only one day, no transaction is possible.
- Consecutive equal prices should not create profit opportunities.
- Multiple alternating rises and falls require combining several smaller profitable trades.

Understanding why multiple small profits can equal one large profit is the key insight behind the optimal solution.

## Approaches

### Brute Force Approach

A brute force solution would try every possible sequence of transactions. At every day, we could decide:

- buy,
- sell,
- or do nothing.

This naturally forms a recursive decision tree. The algorithm would track whether we currently hold a stock and recursively explore every valid action for the next day.

For example:

- If we are not holding a stock, we may either buy or skip.
- If we are holding a stock, we may either sell or hold.

Eventually, the recursion explores every legal trading sequence and returns the maximum profit.

This approach is correct because it exhaustively checks all possible transaction combinations. However, it becomes extremely slow because the number of possible states grows exponentially with the number of days.

With up to 30,000 days, exponential complexity is completely infeasible.

### Optimal Greedy Approach

The key observation is that every increasing price movement contributes profit independently.

Suppose prices increase like this:

```
1 -> 2 -> 3 -> 4 -> 5
```

One strategy is:

- buy at 1
- sell at 5
- profit = 4

Another strategy is:

- buy at 1, sell at 2
- buy at 2, sell at 3
- buy at 3, sell at 4
- buy at 4, sell at 5

Total profit:

```
(2 - 1) + (3 - 2) + (4 - 3) + (5 - 4) = 4
```

The total profit is identical.

This leads to a simple greedy insight:

Whenever today's price is higher than yesterday's price, we should capture that increase.

In other words:

```
profit += prices[i] - prices[i - 1]
```

whenever:

```
prices[i] > prices[i - 1]
```

This works because every upward movement contributes positively to the final achievable profit.

The greedy solution processes the array once and accumulates all positive differences.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(n) | Explores all possible buy/sell decisions recursively |
| Optimal Greedy | O(n) | O(1) | Adds every positive price increase |

## Algorithm Walkthrough

1. Initialize a variable called `profit` to `0`.

This variable will store the total accumulated profit from all successful transactions.
2. Iterate through the array starting from index `1`.

We start from the second day because we compare each day's price with the previous day's price.
3. For each day, compare `prices[i]` with `prices[i - 1]`.

If today's price is greater than yesterday's price, there is a profit opportunity.
4. Add the positive difference to `profit`.

The amount added is:

```
prices[i] - prices[i - 1]
```

This represents buying yesterday and selling today.
5. Ignore non profitable movements.

If today's price is less than or equal to yesterday's price, no profit is possible from that transition, so we skip it.
6. Continue until the end of the array.

By summing all positive increases, we effectively capture every profitable upward movement.
7. Return `profit`.

This final value represents the maximum achievable profit.

## Python Solution

```
from typing import List

class Solution:
    def maxProfit(self, prices: List[int]) -> int:
        total_profit = 0

        for i in range(1, len(prices)):
            if prices[i] > prices[i - 1]:
                total_profit += prices[i] - prices[i - 1]

        return total_profit
```

The implementation begins by initializing `total_profit` to zero. This variable accumulates the profit gained from every profitable price increase.

The loop starts at index `1` because each iteration compares the current day's price with the previous day's price.

Inside the loop, the algorithm checks whether the current price is greater than the previous price. If so, that increase represents guaranteed profit that can be safely captured.

The profit from that upward movement is added directly to `total_profit`.

If the price decreases or stays the same, the algorithm does nothing because no profitable transaction exists for that transition.

Finally, the method returns the accumulated profit.

## Go Solution

```
func maxProfit(prices []int) int {
    totalProfit := 0

    for i := 1; i < len(prices); i++ {
        if prices[i] > prices[i-1] {
            totalProfit += prices[i] - prices[i-1]
        }
    }

    return totalProfit
}
```

The Go implementation follows the same logic as the Python version.

Go slices already provide dynamic length handling, so no additional data structures are necessary.

The problem constraints guarantee that the result fits comfortably within Go's `int` type because the maximum total profit is bounded by the input size and price range.

Unlike Python, Go does not require explicit type hints because variable types are inferred automatically.

## Edge Cases

One important edge case is an array containing only one price, such as:

```
[5]
```

Since at least two days are required to complete a buy and sell transaction, the algorithm correctly returns `0`. The loop never executes because there is no previous day to compare against.

Another important edge case is a strictly decreasing sequence:

```
[9,7,5,3,1]
```

A naive implementation might incorrectly attempt transactions that produce negative profit. This implementation avoids that issue because it only adds positive differences. Every comparison fails the condition, so the final result remains `0`.

A third important edge case is repeated equal prices:

```
[3,3,3,3]
```

Equal prices do not produce profit. The condition:

```
prices[i] > prices[i - 1]
```

ensures that zero differences are ignored. The algorithm therefore correctly returns `0`.

Another subtle edge case involves alternating increases and decreases:

```
[1,5,2,8]
```

The optimal strategy requires multiple separate transactions:

- buy at 1, sell at 5
- buy at 2, sell at 8

The greedy approach naturally captures both increases independently and produces the correct total profit of `10`.
