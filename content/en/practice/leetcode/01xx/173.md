## Solution

## Problem Understanding

This problem asks us to design an iterator for a Binary Search Tree, abbreviated as BST. The iterator should return the nodes of the tree in ascending order, which corresponds to an in-order traversal of a BST.

In a BST, every node satisfies the property that:

- All values in the left subtree are smaller than the node value.
- All values in the right subtree are larger than the node value.

Because of this property, an in-order traversal visits nodes in sorted order:

1. Traverse the left subtree
2. Visit the current node
3. Traverse the right subtree

The iterator must support three operations:

- `BSTIterator(root)` initializes the iterator using the tree root.
- `next()` moves the iterator forward and returns the next smallest value.
- `hasNext()` returns whether another value still exists.

The important detail is that the iterator starts before the smallest element. That means the very first call to `next()` should return the minimum value in the tree.

The constraints are large:

- Up to `10^5` nodes
- Up to `10^5` iterator operations

These constraints immediately tell us that repeatedly traversing the tree from scratch would be far too expensive. The follow-up requirement is even more important:

- Average `O(1)` time per operation
- `O(h)` memory usage, where `h` is the height of the tree

This strongly suggests using an iterative traversal with a stack, because recursion could use `O(h)` call stack space, and a stack allows us to pause and resume traversal efficiently.

Several edge cases are important:

- A tree with only one node
- A completely skewed tree, where height becomes `O(n)`
- Calling `hasNext()` after consuming all nodes
- Trees where nodes only have left children or only right children

The problem guarantees that `next()` is always valid when called, so we never need to handle invalid iterator access.

## Approaches

### Brute Force Approach

The most straightforward solution is to perform a complete in-order traversal during initialization and store all node values in an array.

Since in-order traversal of a BST produces sorted values, the iterator can simply maintain an index into this array.

For example:

- Traverse the entire tree
- Store values like `[3, 7, 9, 15, 20]`
- `next()` returns `values[index]` and increments the index
- `hasNext()` checks whether `index < len(values)`

This approach is easy to implement and clearly correct because the traversal order is precomputed exactly once.

However, it uses `O(n)` memory because all values are stored. The follow-up specifically asks for `O(h)` memory, so this solution does not satisfy the optimal requirements.

### Optimal Approach

The key observation is that we do not need to store the entire traversal result ahead of time.

Instead, we can simulate the recursive in-order traversal iteratively using a stack.

In recursive in-order traversal:

1. Keep moving left
2. Process current node
3. Move right

The stack allows us to pause traversal at exactly the right location.

The main idea is:

- During initialization, push all left descendants onto the stack.
- The top of the stack is always the next smallest node.
- When a node is consumed:

- Pop it
- If it has a right child, push that right child and all its left descendants

This ensures nodes are visited lazily, only when needed.

Each node is pushed and popped exactly once, which gives amortized `O(1)` time per operation.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | `O(n)` initialization, `O(1)` per query | `O(n)` | Stores full in-order traversal |
| Optimal | `O(1)` amortized per operation | `O(h)` | Uses stack to simulate traversal lazily |

## Algorithm Walkthrough

1. Create an empty stack inside the iterator.

The stack will store nodes that still need to be processed. It represents the traversal state.
2. During initialization, start from the root and repeatedly move left.

Every node encountered is pushed onto the stack.

This is done because, in an in-order traversal, the leftmost node is always visited first.
3. After initialization completes, the top of the stack contains the smallest node in the BST.
4. When `next()` is called:

- Pop the top node from the stack.
- This node is the next smallest value.
- Store its value to return later.
5. If the popped node has a right child:

- Move to that right child.
- Push the right child and all its left descendants onto the stack.

This step is essential because after visiting a node in in-order traversal, the traversal continues into its right subtree.
6. Return the popped node's value.
7. When `hasNext()` is called:

- Check whether the stack is non-empty.
- If nodes remain in the stack, another value exists.

The reason this works efficiently is that each node enters and leaves the stack exactly once.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import Optional

class BSTIterator:

    def __init__(self, root: Optional[TreeNode]):
        self.stack = []
        self._push_left_path(root)

    def _push_left_path(self, node: Optional[TreeNode]) -> None:
        while node:
            self.stack.append(node)
            node = node.left

    def next(self) -> int:
        current_node = self.stack.pop()

        if current_node.right:
            self._push_left_path(current_node.right)

        return current_node.val

    def hasNext(self) -> bool:
        return len(self.stack) > 0

# Your BSTIterator object will be instantiated and called as such:
# obj = BSTIterator(root)
# param_1 = obj.next()
# param_2 = obj.hasNext()
```

The implementation uses a helper method called `_push_left_path`.

This helper repeatedly pushes nodes while moving left. It encapsulates the repeated logic needed both during initialization and after visiting a node with a right subtree.

Inside the constructor:

- The stack is initialized.
- The leftmost path from the root is pushed immediately.

This guarantees that the smallest node is ready at the top of the stack.

Inside `next()`:

- The top node is popped because it represents the next in-order element.
- If the node has a right subtree, we process that subtree's leftmost path next.
- The node value is returned.

Inside `hasNext()`:

- We simply check whether the stack still contains nodes.

The implementation satisfies the follow-up requirements because:

- Each node is processed once
- The stack size never exceeds the tree height

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */

type BSTIterator struct {
	stack []*TreeNode
}

func Constructor(root *TreeNode) BSTIterator {
	iterator := BSTIterator{}
	iterator.pushLeftPath(root)
	return iterator
}

func (this *BSTIterator) pushLeftPath(node *TreeNode) {
	for node != nil {
		this.stack = append(this.stack, node)
		node = node.Left
	}
}

func (this *BSTIterator) Next() int {
	lastIndex := len(this.stack) - 1

	currentNode := this.stack[lastIndex]
	this.stack = this.stack[:lastIndex]

	if currentNode.Right != nil {
		this.pushLeftPath(currentNode.Right)
	}

	return currentNode.Val
}

func (this *BSTIterator) HasNext() bool {
	return len(this.stack) > 0
}

/**
 * Your BSTIterator object will be instantiated and called as such:
 * obj := Constructor(root);
 * param_1 := obj.Next();
 * param_2 := obj.HasNext();
 */
```

The Go implementation follows the same logic as the Python version but uses slices to simulate a stack.

A few Go-specific details are worth noting:

- `nil` is used instead of Python's `None`
- Slices are dynamically resized and work naturally as stacks
- Popping from the stack is done by slicing:

`this.stack = this.stack[:lastIndex]`
- Methods that modify the iterator use pointer receivers

No integer overflow concerns exist because node values are safely within Go's integer range.

## Worked Examples

### Example 1

Tree:

```
        7
       / \
      3   15
         /  \
        9   20
```

### Initialization

We push the leftmost path starting from root `7`.

| Step | Node | Stack |
| --- | --- | --- |
| Push 7 | 7 | [7] |
| Move left to 3 | 3 | [7] |
| Push 3 | 3 | [7, 3] |
| Move left to null | null | [7, 3] |

Top of stack is `3`.

### First `next()`

Pop `3`.

| Action | Stack |
| --- | --- |
| Pop 3 | [7] |

`3` has no right child.

Return `3`.

### Second `next()`

Pop `7`.

| Action | Stack |
| --- | --- |
| Pop 7 | [] |

`7` has right child `15`.

Push left path from `15`.

| Step | Stack |
| --- | --- |
| Push 15 | [15] |
| Move left to 9 | [15] |
| Push 9 | [15, 9] |

Return `7`.

### `hasNext()`

Stack contains `[15, 9]`.

Return `true`.

### Third `next()`

Pop `9`.

| Action | Stack |
| --- | --- |
| Pop 9 | [15] |

`9` has no right child.

Return `9`.

### Fourth `next()`

Pop `15`.

| Action | Stack |
| --- | --- |
| Pop 15 | [] |

`15` has right child `20`.

Push left path from `20`.

| Step | Stack |
| --- | --- |
| Push 20 | [20] |

### Fifth `next()`

Pop `20`.

| Action | Stack |
| --- | --- |

Return `false`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | `O(1)` amortized | Each node is pushed and popped exactly once |
| Space | `O(h)` | Stack stores at most one root-to-leaf path |

Although a single `next()` call may push multiple nodes onto the stack, every node can only be pushed once during the entire execution. Across all operations, total work is proportional to the number of nodes, giving amortized constant time.

The stack depth never exceeds the tree height, because it only stores the active traversal path.

## Edge Cases

### Single Node Tree

A tree containing only one node is the smallest valid input.

Example:

```
1
```

Initialization pushes the single node onto the stack. The first `next()` returns `1`, and afterward the stack becomes empty. `hasNext()` correctly returns `false`.

This case is important because incorrect stack handling can cause empty-stack access errors.

### Completely Left-Skewed Tree

Example:

```
    5
   /
  4
 /
3
/
2
```

Initialization pushes every node immediately because the traversal keeps moving left.

The stack size becomes equal to the tree height.

This tests whether the implementation properly handles worst-case height and confirms the `O(h)` memory bound.

### Completely Right-Skewed Tree

Example:

```
1
 \
  2
   \
    3
```

Initialization pushes only the root initially.

Every call to `next()` discovers another right subtree and pushes its left path.

This case is important because it verifies that right subtree processing is implemented correctly.

### Calling `hasNext()` After Full Traversal

After all nodes are consumed, the stack becomes empty.

The implementation safely returns:

```
False
```

or

```
false
```

without attempting invalid operations.

This edge case ensures the iterator properly tracks traversal completion.
