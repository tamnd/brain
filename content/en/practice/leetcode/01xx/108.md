## Solution

## Problem Understanding

This problem asks us to convert a **sorted array in ascending order** into a **height balanced Binary Search Tree (BST)**.

A **Binary Search Tree** is a binary tree where:

- Every node's left subtree contains values smaller than the node.
- Every node's right subtree contains values greater than the node.
- Both left and right subtrees must themselves also satisfy BST properties.

A **height balanced** binary tree means that for every node, the height difference between the left and right subtree is at most 1. In simpler terms, the tree should not become heavily skewed to one side.

The input is an integer array `nums`, which is already sorted in **strictly increasing order**. Because the array is sorted, we already know the relative ordering of all values. The challenge is to organize these values into a BST while keeping the tree balanced.

The output should be the root node of a valid height balanced BST constructed from the array. Importantly, the problem does **not require a unique answer**. Multiple balanced BST structures may be valid. For example, when there are two possible middle elements, either choice can still produce a valid balanced tree.

The constraints tell us:

- `1 <= nums.length <= 10^4`, meaning the input can be fairly large.
- Values range between `-10^4` and `10^4`, which is small enough that integer overflow is not a concern.
- The array is **strictly increasing**, meaning there are no duplicate values.

Because the input is already sorted, we should exploit that structure instead of trying to insert elements one by one.

Several edge cases are important:

- A single-element array such as `[5]` should produce a tree with only one node.
- Arrays with two elements such as `[1,3]` have multiple valid balanced BST outputs.
- Very large arrays require an efficient solution because repeated insertions into a BST could become inefficient if implemented poorly.
- Since the array is guaranteed to be sorted and non-empty, we do not need to validate ordering or handle an empty input outside recursive base cases.

## Approaches

### Brute Force Approach

A naive idea is to insert each element from the sorted array into a BST one at a time using standard BST insertion rules.

Since the numbers are already sorted, inserting them sequentially causes a serious problem. Each new value is larger than the previous one, so every insertion goes to the right subtree. Instead of building a balanced BST, this creates a highly skewed tree that resembles a linked list.

For example:

```
[1,2,3,4,5]
```

Would become:

```
1
 \
  2
   \
    3
     \
      4
       \
        5
```

This violates the requirement that the BST must be height balanced.

Even if we tried to rebalance afterward, the approach becomes unnecessarily expensive.

### Key Insight for the Optimal Solution

The crucial observation is that because the array is already sorted, the **middle element naturally belongs at the root**.

Why?

- Everything to the left of the middle is smaller.
- Everything to the right of the middle is larger.

This perfectly matches BST ordering rules.

Choosing the middle element also keeps the tree balanced because it splits the array into two roughly equal halves.

We can then apply the same idea recursively:

1. Pick the middle element as the root.
2. Recursively build the left subtree from the left half.
3. Recursively build the right subtree from the right half.

This is a classic **Divide and Conquer** strategy.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Sequential BST insertion creates a skewed tree for sorted input |
| Optimal | O(n) | O(log n) | Recursively choose middle element for balanced construction |

## Algorithm Walkthrough

1. Define a recursive helper function that builds a BST for a subarray range `[left, right]`. Instead of slicing arrays repeatedly, we pass indices to avoid extra memory overhead.
2. Check the base case. If `left > right`, there are no elements left to process, so return `None` (or `nil` in Go). This represents an empty subtree.
3. Compute the middle index using:

```
middle = (left + right) // 2
```

The middle element becomes the current root node because it best preserves balance.
4. Create a new tree node using `nums[middle]`.
5. Recursively construct the left subtree using:

```
left to middle - 1
```

Since all these values are smaller than the middle element, they belong in the left subtree.
6. Recursively construct the right subtree using:

```
middle + 1 to right
```

Since these values are larger than the middle element, they belong in the right subtree.
7. Attach the returned left and right subtrees to the current node.
8. Return the constructed node.

This recursive process continues until every element has been placed into the BST exactly once.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import List, Optional

class Solution:
    def sortedArrayToBST(self, nums: List[int]) -> Optional[TreeNode]:
        def build_tree(left: int, right: int) -> Optional[TreeNode]:
            if left > right:
                return None

            middle = (left + right) // 2

            root = TreeNode(nums[middle])

            root.left = build_tree(left, middle - 1)
            root.right = build_tree(middle + 1, right)

            return root

        return build_tree(0, len(nums) - 1)
```

The implementation follows the divide and conquer strategy directly.

The `build_tree()` helper function works on a subarray defined by `left` and `right` indices. This is more efficient than slicing arrays because slicing creates copies, increasing memory usage.

The base case handles empty ranges. Whenever `left > right`, there are no values available to form a node, so we return `None`.

The middle index is computed to guarantee balance. The chosen middle element becomes the current root node.

After creating the node, recursion builds the left and right children independently. Since the input is sorted, all elements to the left are guaranteed to be smaller, and all elements to the right are guaranteed to be larger, preserving BST correctness automatically.

Finally, the root node is returned to the caller, eventually constructing the entire tree.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func sortedArrayToBST(nums []int) *TreeNode {
	var buildTree func(left, right int) *TreeNode

	buildTree = func(left, right int) *TreeNode {
		if left > right {
			return nil
		}

		middle := (left + right) / 2

		root := &TreeNode{
			Val: nums[middle],
		}

		root.Left = buildTree(left, middle-1)
		root.Right = buildTree(middle+1, right)

		return root
	}

	return buildTree(0, len(nums)-1)
}
```

The Go implementation follows the same recursive structure as Python but uses pointers to `TreeNode`.

Instead of returning `None`, Go returns `nil` for empty subtrees. Tree nodes are allocated using `&TreeNode{}` syntax.

Go slices are passed efficiently by reference-like semantics, so we only pass index boundaries into the recursive helper function rather than copying portions of the slice.

Integer overflow is not a concern here because the array length is capped at `10^4`, but using index boundaries still keeps the implementation clean and efficient.

## Worked Examples

### Example 1

**Input:**

```
nums = [-10,-3,0,5,9]
```

We recursively select the middle element.

| Step | Left | Right | Middle | Value Chosen | Result |
| --- | --- | --- | --- | --- | --- |
| 1 | 0 | 4 | 2 | 0 | Root = 0 |
| 2 | 0 | 1 | 0 | -10 | Left child of 0 |
| 3 | 1 | 1 | 1 | -3 | Right child of -10 |
| 4 | 3 | 4 | 3 | 5 | Right child of 0 |
| 5 | 4 | 4 | 4 | 9 | Right child of 5 |

Resulting BST:

```
        0
       / \
    -10   5
       \    \
       -3    9
```

This is balanced because subtree heights differ by at most 1.

Another valid solution is possible if a different middle index is chosen in some recursive calls.

### Example 2

**Input:**

```
nums = [1,3]
```

| Step | Left | Right | Middle | Value Chosen | Result |
| --- | --- | --- | --- | --- | --- |
| 1 | 0 | 1 | 0 | 1 | Root = 1 |
| 2 | 1 | 1 | 1 | 3 | Right child |

Resulting BST:

```
1
 \
  3
```

This tree is balanced.

Another valid answer could be:

```
  3
 /
1
```

Both satisfy BST ordering and height balance requirements.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every element is visited exactly once to create one tree node |
| Space | O(log n) | Recursive call stack depth equals tree height for a balanced tree |

The time complexity is **O(n)** because each number in the array becomes exactly one tree node, and each recursive call performs constant work outside recursion.

The space complexity is **O(log n)** due to recursion depth. Since the tree remains balanced, the maximum depth of recursive calls is proportional to the height of the tree, which is logarithmic in the number of nodes.

## Edge Cases

### Single Element Array

Consider:

```
nums = [7]
```

This is the smallest valid input. A naive implementation might recurse incorrectly or fail to stop. Our implementation handles this naturally because the middle index points to `7`, and subsequent recursive calls immediately hit the `left > right` base case.

### Two Elements

Consider:

```
nums = [1,3]
```

There are two valid balanced BSTs. Some implementations incorrectly assume only one exact structure is valid. Our recursive middle-selection strategy automatically generates one valid answer while satisfying the height balance condition.

### Large Sorted Input

Consider an array near the constraint limit:

```
nums = [1,2,3,...,10000]
```

A sequential insertion approach would create a skewed tree and potentially lead to poor performance. Our divide and conquer method keeps the tree balanced throughout construction, ensuring efficient recursion depth and linear overall runtime.

### Negative Numbers

Consider:

```
nums = [-10,-5,-2,0,8]
```

Negative values can sometimes reveal comparison mistakes in BST logic. Since we rely solely on sorted ordering and index positions, negative numbers require no special handling. The BST property is preserved automatically.
