## Solution

## Problem Understanding

The problem provides a database table named `Weather`, where each row represents the temperature recorded on a particular date. Each row contains three columns:

| Column | Meaning |
| --- | --- |
| `id` | Unique identifier for the row |
| `recordDate` | The date of the temperature reading |
| `temperature` | The temperature recorded on that date |

The goal is to find all rows where the temperature is strictly higher than the temperature recorded on the previous calendar day.

The important detail is that we are comparing temperatures between consecutive dates, not consecutive rows by `id`. The `id` column is only a unique identifier and does not imply chronological order. We must compare rows using the `recordDate` column.

For example, if one row contains:

```
2015-01-02 -> 25
```

and the previous calendar day contains:

```
2015-01-01 -> 10
```

then the row for `2015-01-02` should be included because `25 > 10`.

The output should contain only the `id` values of rows that satisfy this condition.

The problem also guarantees that:

- Every `recordDate` is unique.
- There are no duplicate dates.
- The table may not contain every calendar day consecutively.

That last guarantee is especially important. A naive implementation might incorrectly compare adjacent rows even if the dates are not consecutive. For example:

```
2015-01-01
2015-01-03
```

These dates are two days apart, so they must not be compared.

An important edge case is missing dates. If yesterday's record does not exist, the current row cannot qualify because there is no valid previous day comparison.

Another edge case is equal temperatures. The problem specifically asks for temperatures that are higher, so equal values should not be included.

Finally, since this is a database problem, the expected solution is written in SQL rather than a traditional algorithmic language.

## Approaches

### Brute Force Approach

A straightforward solution is to compare every row with every other row.

For each weather record, we can scan the entire table searching for another row whose `recordDate` is exactly one day earlier. Once found, we compare the temperatures.

Conceptually, the algorithm works like this:

1. Pick a row `w1`.
2. Scan every row `w2`.
3. Check whether:

- `w2.recordDate = w1.recordDate - 1 day`
- `w1.temperature > w2.temperature`
4. If both conditions are true, output `w1.id`.

This approach is correct because every possible pair is examined, ensuring we never miss a valid comparison.

However, it is inefficient because for every row we potentially scan the entire table again. If there are `n` rows, the complexity becomes `O(n²)`.

For small datasets this is acceptable, but for larger tables it becomes unnecessarily expensive.

### Optimal Approach

The key observation is that we only need to compare each row with the row from the previous day.

SQL databases are very good at joining tables based on relationships between rows. Instead of manually scanning every pair, we can use a self join.

A self join means joining the `Weather` table with itself:

- One instance represents the current day.
- The other instance represents the previous day.

We then connect rows where:

```
DATEDIFF(current.recordDate, previous.recordDate) = 1
```

This guarantees that the dates are exactly one day apart.

After establishing the relationship, we simply check whether:

```
current.temperature > previous.temperature
```

If true, we return the current row's `id`.

This approach avoids unnecessary comparisons and leverages database indexing and join optimizations.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Compares every row against every other row |
| Optimal | O(n log n) typically | O(1) extra | Uses a self join to compare only relevant rows |

## Algorithm Walkthrough

1. Create two aliases of the `Weather` table.

One alias represents the current day's weather record, and the other represents the previous day's weather record. This allows us to compare rows within the same table.
2. Match rows that are exactly one day apart.

Use the SQL `DATEDIFF` function to ensure the current row's date is exactly one day after the previous row's date.
3. Compare temperatures.

After identifying valid consecutive dates, check whether the current day's temperature is greater than the previous day's temperature.
4. Return the matching IDs.

If both conditions are satisfied, output the `id` of the current day's row.

This method is efficient because it only evaluates meaningful row pairs instead of all possible combinations.

## Python Solution

LeetCode database problems are submitted as SQL queries. The Python section below shows the complete SQL solution embedded as a Python multiline string for consistency with the requested format.

```
class Solution:
    def risingTemperature(self) -> str:
        return """
        SELECT w1.id
        FROM Weather w1
        JOIN Weather w2
            ON DATEDIFF(w1.recordDate, w2.recordDate) = 1
        WHERE w1.temperature > w2.temperature;
        """
```

The solution uses a self join between two instances of the `Weather` table.

The alias `w1` represents the current day's weather record, while `w2` represents the previous day's weather record.

The `DATEDIFF` condition ensures that only rows exactly one day apart are compared:

```
DATEDIFF(w1.recordDate, w2.recordDate) = 1
```

This prevents incorrect comparisons between non consecutive dates.

After establishing valid date pairs, the `WHERE` clause filters rows where the current temperature is greater than the previous day's temperature:

```
w1.temperature > w2.temperature
```

Finally, the query returns the `id` values from the current day's rows.

## Edge Cases

### Missing Dates

Consider the following table:

| id | recordDate | temperature |
| --- | --- | --- |
| 1 | 2015-01-01 | 10 |
| 2 | 2015-01-03 | 20 |

A naive solution might compare these rows because they appear adjacent in sorted order. However, the dates are two days apart, so they must not be compared.

The implementation handles this correctly using:

```
DATEDIFF(w1.recordDate, w2.recordDate) = 1
```

Since the difference is not exactly one day, the rows are excluded.

### Equal Temperatures

Consider:

| id | recordDate | temperature |
| --- | --- | --- |
| 1 | 2015-01-01 | 20 |
| 2 | 2015-01-02 | 20 |

The problem requires the current temperature to be strictly higher.

The condition:

```
w1.temperature > w2.temperature
```

ensures equal temperatures are not included.

### Single Row Table

If the table contains only one row, there is no previous day available for comparison.

In this situation, the self join produces no matching rows, and the query correctly returns an empty result set.

### Unordered IDs

The `id` values may not correspond to chronological order.

For example:

| id | recordDate | temperature |
| --- | --- | --- |
| 100 | 2015-01-01 | 10 |
| 3 | 2015-01-02 | 20 |

A buggy implementation might compare rows using `id - 1`.

This solution avoids that mistake entirely because comparisons are based only on `recordDate`, which is the true chronological field.
