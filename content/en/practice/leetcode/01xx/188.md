## Solution

## Problem Understanding

This problem asks us to maximize profit from buying and selling a stock over several days, with an important restriction: we are allowed to complete **at most `k` transactions**.

A transaction consists of:

1. Buying one stock
2. Selling that stock later

We cannot hold multiple stocks at the same time, meaning we must sell before buying again.

The input consists of:

- `k`, the maximum number of allowed transactions.
- `prices`, where `prices[i]` represents the stock price on day `i`.

The goal is to compute the **maximum total profit** achievable while respecting the transaction limit.

For example, if prices are `[3,2,6,5,0,3]` and `k = 2`, we can:

- Buy at `2`, sell at `6`, profit = `4`
- Buy at `0`, sell at `3`, profit = `3`

Total profit = `7`.

The challenge comes from balancing when to buy and sell while also respecting the limited number of transactions. A greedy approach that always takes local profit opportunities may fail because using one transaction early might prevent a better opportunity later.

The constraints are important:

- `k <= 100`
- `prices.length <= 1000`

A brute-force search over all buy/sell combinations becomes infeasible very quickly because the number of possible transaction sequences grows exponentially.

A useful observation is that the number of transactions can exceed what is realistically possible. Since each transaction requires at least two days, the maximum meaningful number of transactions is `n // 2`, where `n` is the number of days. If `k >= n // 2`, the problem effectively becomes the unlimited transactions version, where we simply capture every profitable upward movement.

Several edge cases matter:

- If `prices` has only one day, no transaction is possible.
- If `k = 0`, profit is always `0`.
- If prices continually decrease, the answer is `0`.
- If `k` is very large relative to the number of days, we should optimize for the unlimited-transactions case.

## Approaches

### Brute Force Approach

A brute-force solution tries every possible sequence of buy and sell operations.

At each day, we can make decisions such as:

- Do nothing
- Buy stock
- Sell stock

We also need to track:

- How many transactions remain
- Whether we currently hold a stock

This naturally forms a decision tree. For every day, we branch into multiple possibilities and recursively explore all valid transaction sequences.

This approach guarantees correctness because it exhaustively examines every possible legal strategy and returns the maximum profit among them.

However, it is far too slow. Since each day introduces branching choices, the total number of possibilities grows exponentially. Even memoization would still require careful state compression to become practical.

### Key Insight for an Optimal Solution

The core observation is that this is a **dynamic programming** problem.

At any day, our future profit depends on only a few things:

1. The current day
2. How many transactions we still have available
3. Whether we currently hold a stock

Instead of recomputing overlapping subproblems repeatedly, we can store the best possible profit for each state.

A particularly efficient formulation tracks two arrays:

- `buy[t]` = maximum profit after buying during transaction `t`
- `sell[t]` = maximum profit after selling during transaction `t`

The intuition is:

- Buying reduces profit because we spend money.
- Selling increases profit because we gain money.

For each stock price, we update these states incrementally.

We also optimize an important special case:

If `k >= n // 2`, we can perform effectively unlimited transactions. In that case, the optimal strategy is simply to collect every positive price increase.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) or worse | O(n) recursion | Tries every buy/sell possibility |
| Optimal Dynamic Programming | O(n × k) | O(k) | Tracks best buy/sell states efficiently |

## Algorithm Walkthrough

### Optimal Dynamic Programming Strategy

We maintain two arrays of size `k + 1`:

- `buy[t]`
- `sell[t]`

Where:

- `buy[t]` represents the best profit after completing the buy operation for transaction `t`.
- `sell[t]` represents the best profit after completing the sell operation for transaction `t`.

Initially:

- Buying should start as very negative because we spend money.
- Selling starts at `0` because no profit has been made yet.

The algorithm proceeds as follows:

1. **Handle trivial cases**

If the prices array is empty or `k == 0`, return `0` immediately because no profit can be made.
2. **Check for unlimited transactions optimization**

If `k >= len(prices) // 2`, treat the problem as unlimited transactions.

Since we can buy and sell as often as profitable, we simply add every positive difference:

```
prices[i] - prices[i - 1]
```

whenever the price increases.
3. **Initialize DP arrays**

Create:

- `buy = [-infinity] * (k + 1)`
- `sell = [0] * (k + 1)`

`buy[t]` starts at negative infinity because we have not bought anything yet.
4. **Process each stock price**

For every price:

Update transaction states from `1` to `k`.
5. **Update buy state**

For transaction `t`:

We either:

- Keep previous buy state
- Buy today after finishing transaction `t - 1`

Formula:

```
buy[t] = max(buy[t], sell[t - 1] - price)
```
6. **Update sell state**

We either:

- Keep previous sell state
- Sell today after previously buying

Formula:

```
sell[t] = max(sell[t], buy[t] + price)
```
7. **Return final result**

The answer is:

```
sell[k]
```

because maximum profit must end after selling, not holding stock.

## Python Solution

```
from typing import List

class Solution:
    def maxProfit(self, k: int, prices: List[int]) -> int:
        if not prices or k == 0:
            return 0

        n = len(prices)

        # Unlimited transactions optimization
        if k >= n // 2:
            profit = 0

            for i in range(1, n):
                if prices[i] > prices[i - 1]:
                    profit += prices[i] - prices[i - 1]

            return profit

        buy = [float("-inf")] * (k + 1)
        sell = [0] * (k + 1)

        for price in prices:
            for transaction in range(1, k + 1):
                buy[transaction] = max(
                    buy[transaction],
                    sell[transaction - 1] - price
                )

                sell[transaction] = max(
                    sell[transaction],
                    buy[transaction] + price
                )

        return sell[k]
```

The implementation begins by handling trivial cases where profit is impossible, such as an empty list or zero allowed transactions.

Next, it applies the unlimited-transactions optimization. When `k` is large enough, we no longer need dynamic programming because every profitable increase can be taken greedily.

For the constrained case, two arrays are created:

- `buy` tracks the best profit after buying
- `sell` tracks the best profit after selling

For every stock price, we update all transaction states.

The buy transition:

```
sell[transaction - 1] - price
```

represents buying stock today after finishing the previous transaction.

The sell transition:

```
buy[transaction] + price
```

represents selling stock today after already owning stock.

At the end, `sell[k]` stores the best achievable profit after at most `k` transactions.

## Go Solution

```
func maxProfit(k int, prices []int) int {
	if len(prices) == 0 || k == 0 {
		return 0
	}

	n := len(prices)

	// Unlimited transactions optimization
	if k >= n/2 {
		profit := 0

		for i := 1; i < n; i++ {
			if prices[i] > prices[i-1] {
				profit += prices[i] - prices[i-1]
			}
		}

		return profit
	}

	buy := make([]int, k+1)
	sell := make([]int, k+1)

	// Initialize buy with very small values
	for i := 0; i <= k; i++ {
		buy[i] = -1 << 30
	}

	for _, price := range prices {
		for transaction := 1; transaction <= k; transaction++ {
			if sell[transaction-1]-price > buy[transaction] {
				buy[transaction] = sell[transaction-1] - price
			}

			if buy[transaction]+price > sell[transaction] {
				sell[transaction] = buy[transaction] + price
			}
		}
	}

	return sell[k]
}
```

The Go implementation follows exactly the same dynamic programming logic as the Python version.

One implementation difference is initialization. Go does not have `float("-inf")`, so we initialize `buy` using a very small negative integer (`-1 << 30`) to simulate negative infinity.

Go slices are used instead of Python lists, but the update logic remains identical. Since the constraints are small (`prices[i] <= 1000`), integer overflow is not a concern.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n × k) | For every day, we process all `k` transactions |
| Space | O(k) | Only two arrays of size `k + 1` are maintained |

The dynamic programming approach is efficient because it avoids recomputing overlapping subproblems. Instead of storing results for every day and transaction in a large table, it compresses the state into two one-dimensional arrays. Since `k <= 100` and `n <= 1000`, this solution performs comfortably within limits.

## Edge Cases

One important edge case occurs when the input contains only one day, such as `prices = [5]`. Since a transaction requires both a buy and a sell, no profit can be made. The implementation correctly returns `0` because the loops never produce a profitable sell.

Another important case is when prices continuously decrease, for example `[7,6,4,3,1]`. A naive implementation might accidentally force transactions and produce negative profit. Our algorithm avoids this because every state update uses `max()`, meaning we never accept a worse outcome than simply doing nothing. The final result remains `0`.

A third edge case happens when `k` is much larger than the number of days. For example:

```
k = 100
prices = [1,2,3,4]
```

In reality, only `n // 2` transactions are possible. Instead of wasting time on unnecessary dynamic programming, the implementation switches to the unlimited-transactions optimization and greedily captures all positive gains.

Another subtle case involves repeated prices, such as `[5,5,5,5]`. Since there are no profitable price differences, buying and selling produces no benefit. The algorithm naturally returns `0` because sell states never improve.
