## Solution

## Problem Understanding

This problem asks us to reorder a singly linked list in a very specific alternating pattern. We start with a list in its original sequential order:

```
L0 → L1 → L2 → ... → Ln
```

We must rearrange the node connections so that the list becomes:

```
L0 → Ln → L1 → Ln-1 → L2 → Ln-2 → ...
```

The important detail is that we are **not allowed to modify node values**. We cannot simply copy values into an array and rewrite them into nodes. Instead, we must physically rearrange the `next` pointers between existing nodes.

The input is the head of a singly linked list. Since it is singly linked, each node only points to the next node, meaning we cannot move backward efficiently. The function modifies the linked list **in place**, meaning we do not return a new list. We simply alter the existing structure.

The constraints are important:

- The list contains between `1` and `5 × 10^4` nodes.
- Node values range from `1` to `1000`.

The large upper bound of `50,000` nodes tells us that inefficient approaches, especially quadratic solutions, may become too slow. Since linked list operations often involve pointer traversal, we should aim for a linear-time algorithm.

Several edge cases are worth considering upfront. A list with only one node does not need reordering. A list with two nodes is already effectively reordered. Odd-length lists introduce a middle node that should remain at the end of the reordered sequence. Even-length lists split cleanly into two equal halves. Another subtle issue is ensuring we terminate the reordered list correctly to avoid cycles, since improper pointer manipulation can accidentally reconnect nodes indefinitely.

## Approaches

### Brute Force Approach

A straightforward solution is to first place every node into an array or stack. Since arrays allow random access, we can alternate between taking nodes from the beginning and the end.

For example:

```
[1,2,3,4,5]
```

Store nodes in an array:

```
[1,2,3,4,5]
```

Then reconstruct:

```
1 → 5 → 2 → 4 → 3
```

We can maintain two pointers, one at the front and one at the back of the array, reconnecting nodes alternately.

This approach is correct because the array gives direct access to both ends of the list, exactly matching the required reordering pattern. However, it requires additional memory proportional to the number of nodes. Since the problem encourages in-place modification, we should seek a solution with constant extra space.

### Optimal Approach

The key insight is that the desired ordering resembles **merging two lists**, where the second half is traversed backward.

For example:

```
1 → 2 → 3 → 4 → 5
```

Can be viewed as:

```
First half:  1 → 2 → 3
Second half: 4 → 5
```

If we reverse the second half:

```
5 → 4
```

We can merge alternately:

```
1 → 5 → 2 → 4 → 3
```

This observation leads to a three-step strategy:

1. Find the middle of the linked list.
2. Reverse the second half.
3. Merge the two halves alternately.

Since each step takes linear time and uses only pointer manipulation, we achieve an efficient in-place solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Store all nodes in an array and reorder using two pointers |
| Optimal | O(n) | O(1) | Find middle, reverse second half, merge alternately |

## Algorithm Walkthrough

1. **Handle trivial cases**

If the list has `0`, `1`, or `2` nodes, there is nothing meaningful to reorder. We can immediately return.
2. **Find the middle of the list**

Use the classic slow and fast pointer technique.

- `slow` moves one step at a time.
- `fast` moves two steps at a time.

When `fast` reaches the end, `slow` will be near the middle.

For example:

```
1 → 2 → 3 → 4 → 5
```

`slow` stops at `3`.

This allows us to split the list into two parts efficiently without counting nodes.
3. **Split the list**

Disconnect the first half from the second half:

```
First half:  1 → 2 → 3
Second half: 4 → 5
```

Breaking the connection is important to avoid cycles during merging.
4. **Reverse the second half**

Reverse the second half in place using iterative pointer reversal.

Example:

```
4 → 5
```

becomes:

```
5 → 4
```

Reversal is necessary because we need nodes in the order:

```
Ln, Ln-1, Ln-2...
```

A singly linked list cannot move backward naturally, so reversing simulates backward traversal.
5. **Merge the two halves alternately**

Merge nodes one by one:

```
First:   1 → 2 → 3
Second:  5 → 4
```

Merge process:

```
1 → 5 → 2 → 4 → 3
```

At every step:

- Save next pointers.
- Link one node from first half.
- Link one node from reversed second half.
- Advance both pointers.
6. **Finish the list properly**

Since the first half may contain one extra node for odd-length lists, ensure the final node points to `None`.

## Python Solution

```
# Definition for singly-linked list.
# class ListNode:
#     def __init__(self, val=0, next=None):
#         self.val = val
#         self.next = next

from typing import Optional

class Solution:
    def reorderList(self, head: Optional[ListNode]) -> None:
        """
        Do not return anything, modify head in-place instead.
        """
        if not head or not head.next or not head.next.next:
            return

        # Step 1: Find the middle
        slow = head
        fast = head

        while fast and fast.next:
            slow = slow.next
            fast = fast.next.next

        # Step 2: Reverse second half
        second_half = slow.next
        slow.next = None

        previous = None
        current = second_half

        while current:
            next_node = current.next
            current.next = previous
            previous = current
            current = next_node

        reversed_second_half = previous

        # Step 3: Merge two halves
        first = head
        second = reversed_second_half

        while second:
            first_next = first.next
            second_next = second.next

            first.next = second
            second.next = first_next

            first = first_next
            second = second_next
```

The implementation closely follows the algorithm.

The first section finds the midpoint using the slow and fast pointer technique. This works in a single traversal and avoids explicitly counting nodes.

After locating the middle, the list is split into two halves by setting `slow.next = None`. This prevents accidental cycles when reconnecting nodes later.

The reversal logic uses three pointers: `previous`, `current`, and `next_node`. This is the standard in-place linked list reversal pattern. By the end of the loop, `previous` points to the new head of the reversed second half.

Finally, the merge step alternates between nodes from the first half and reversed second half. Temporary variables preserve the next nodes before rewiring pointers, ensuring we never lose access to the remaining list.

## Go Solution

```
/**
 * Definition for singly-linked list.
 * type ListNode struct {
 *     Val int
 *     Next *ListNode
 * }
 */
func reorderList(head *ListNode) {
	if head == nil || head.Next == nil || head.Next.Next == nil {
		return
	}

	// Step 1: Find the middle
	slow, fast := head, head

	for fast != nil && fast.Next != nil {
		slow = slow.Next
		fast = fast.Next.Next
	}

	// Step 2: Reverse second half
	secondHalf := slow.Next
	slow.Next = nil

	var prev *ListNode
	current := secondHalf

	for current != nil {
		nextNode := current.Next
		current.Next = prev
		prev = current
		current = nextNode
	}

	reversedSecondHalf := prev

	// Step 3: Merge two halves
	first := head
	second := reversedSecondHalf

	for second != nil {
		firstNext := first.Next
		secondNext := second.Next

		first.Next = second
		second.Next = firstNext

		first = firstNext
		second = secondNext
	}
}
```

The Go implementation follows the exact same logic as the Python solution. The main differences come from language syntax and memory handling.

Go uses `nil` instead of `None`, and linked list pointers are explicitly typed as `*ListNode`. Variable declarations use short assignment syntax (`:=`) where appropriate. Since Go manages pointers safely, we can manipulate linked list references similarly to Python without worrying about manual memory allocation.

Integer overflow is not a concern here because we are not performing arithmetic operations on values, only pointer rearrangement.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | We traverse the list a constant number of times, finding middle, reversing, and merging |
| Space | O(1) | Only a few pointer variables are used, no extra data structures proportional to input size |

The runtime is linear because each node is visited at most a few times. The middle-finding pass scans the list once, reversal scans the second half once, and merging scans both halves once. Since these operations happen sequentially, total work remains proportional to `n`.

The space complexity is constant because we only maintain a small fixed number of pointers regardless of input size. No arrays, stacks, or recursive calls are used.

## Edge Cases

### Single Node List

Consider:

```
[1]
```

A naive implementation might attempt to split or reverse nonexistent nodes, causing null pointer errors. Our implementation immediately returns for lists shorter than three nodes, preventing unnecessary operations.

### Two Node List

Input:

```
[1,2]
```

The reordered form is effectively identical:

```
1 → 2
```

Without careful handling, merging logic could accidentally duplicate nodes or create cycles. The early return condition handles this safely.

### Odd Length List

Example:

```
[1,2,3,4,5]
```

The middle node `3` has no partner during merging and should remain at the end:

```
1 → 5 → 2 → 4 → 3
```

Our implementation naturally handles this because the first half contains one extra node, and merging stops when the reversed second half is exhausted.

### Even Length List

Example:

```
[1,2,3,4]
```

The split is balanced:

```
1 → 2
3 → 4
```

After reversal and merging:

```
1 → 4 → 2 → 3
```

The implementation works correctly because both halves merge evenly without leftover nodes.

### Avoiding Cycles

A subtle bug in linked list problems is accidentally creating a cycle. If we forget to disconnect the first half using:

```
slow.next = None
```

the merged list could reconnect into itself indefinitely. Explicitly severing the connection guarantees the final structure terminates properly.
