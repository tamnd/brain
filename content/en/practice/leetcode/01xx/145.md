## Solution

## Problem Understanding

In this problem, we are given the root node of a binary tree, and we must return the values of the nodes in postorder traversal order.

A binary tree is a hierarchical data structure where every node can have at most two children, commonly called the left child and the right child. The input `root` represents the starting node of this tree. If the tree is empty, the root is `null` in Python or `nil` in Go.

Postorder traversal follows a very specific visiting order:

1. Traverse the left subtree
2. Traverse the right subtree
3. Visit the current node

This ordering is important because the current node is processed only after both of its children have already been processed.

For example, consider this tree:

```
    1
     \
      2
     /
    3
```

The traversal order is:

- Visit left subtree of 1, which is empty
- Visit right subtree rooted at 2

- Visit left subtree rooted at 3
- Visit right subtree of 2, which is empty
- Visit node 2
- Visit node 1

So the final traversal becomes:

```
[3, 2, 1]
```

The constraints tell us that the tree contains at most 100 nodes. This is a relatively small input size, which means even recursive solutions are completely safe from recursion depth issues in practice. However, the follow up explicitly asks for an iterative solution, so we should understand both recursive and stack based approaches.

Several important edge cases must be considered:

- The tree may be empty, so we should return an empty list.
- The tree may contain only one node.
- The tree may be completely skewed to the left or right.
- Some nodes may have only one child.
- Negative values are allowed, but traversal logic does not depend on node values.

The problem guarantees that the structure is a valid binary tree, so we do not need to handle cycles or malformed input.

## Approaches

### Brute Force Approach

The most straightforward solution is recursive depth first traversal.

For every node:

1. Recursively traverse the left subtree
2. Recursively traverse the right subtree
3. Add the current node's value to the result

This directly matches the definition of postorder traversal, which makes the recursive approach extremely natural and easy to understand.

The recursive call stack automatically remembers which nodes still need processing, so we do not need to manually manage traversal state.

This approach is correct because every node is processed only after both of its subtrees have been completely traversed.

Although this solution is already efficient enough for the problem constraints, the follow up asks whether we can do the traversal iteratively. Recursive solutions rely on the system call stack, while iterative solutions explicitly manage traversal state using a stack data structure.

### Optimal Approach

The iterative solution uses a stack to simulate recursion.

One common trick for iterative postorder traversal is:

- Process nodes in modified preorder order:

- Root
- Right
- Left
- Reverse the result at the end

Why does this work?

Normal preorder traversal is:

```
Root -> Left -> Right
```

If we instead traverse:

```
Root -> Right -> Left
```

and reverse the final order, we get:

```
Left -> Right -> Root
```

which is exactly postorder traversal.

This approach is efficient, avoids recursion, and cleanly handles all tree shapes.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(h) | Recursive DFS, where h is tree height |
| Optimal | O(n) | O(n) | Iterative stack based traversal |

## Algorithm Walkthrough

### Iterative Postorder Traversal

1. Create an empty result list that will store traversal values.
2. If the tree is empty, immediately return the empty result list. This prevents unnecessary processing.
3. Initialize a stack with the root node. The stack will simulate the recursive traversal process.
4. While the stack is not empty:

1. Pop the top node from the stack.
2. Add the node's value to the result list.
3. Push the left child onto the stack if it exists.
4. Push the right child onto the stack if it exists.
5. The traversal order generated so far is:

```
Root -> Right -> Left
```
6. Reverse the result list. After reversing, the order becomes:

```
Left -> Right -> Root
```

which is the required postorder traversal.
7. Return the reversed result list.

The stack is chosen because it naturally supports depth first traversal. By controlling the order in which children are pushed onto the stack, we can precisely control traversal order.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import List, Optional

class Solution:
    def postorderTraversal(self, root: Optional[TreeNode]) -> List[int]:
        if not root:
            return []

        result = []
        stack = [root]

        while stack:
            node = stack.pop()
            result.append(node.val)

            if node.left:
                stack.append(node.left)

            if node.right:
                stack.append(node.right)

        return result[::-1]
```

The implementation begins by handling the empty tree case. If `root` is `None`, the function immediately returns an empty list.

The `result` list stores traversal values, while the `stack` manages nodes that still need processing. Initially, the stack contains only the root node.

Inside the loop, the top node is removed from the stack and its value is appended to `result`. This creates a modified preorder traversal.

The left child is pushed before the right child. Because stacks are Last In First Out structures, the right child is processed first. This produces the traversal order:

```
Root -> Right -> Left
```

Finally, the result list is reversed to obtain true postorder traversal:

```
Left -> Right -> Root
```

The slicing operation `result[::-1]` creates the reversed list efficiently and concisely.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func postorderTraversal(root *TreeNode) []int {
    if root == nil {
        return []int{}
    }

    result := []int{}
    stack := []*TreeNode{root}

    for len(stack) > 0 {
        n := len(stack) - 1
        node := stack[n]
        stack = stack[:n]

        result = append(result, node.Val)

        if node.Left != nil {
            stack = append(stack, node.Left)
        }

        if node.Right != nil {
            stack = append(stack, node.Right)
        }
    }

    left := 0
    right := len(result) - 1

    for left < right {
        result[left], result[right] = result[right], result[left]
        left++
        right--
    }

    return result
}
```

The Go implementation follows the same algorithm as the Python version.

Since Go does not provide a built in stack type, a slice is used as a stack. The last element of the slice represents the top of the stack.

Go uses `nil` instead of Python's `None` for empty pointers. The empty tree case therefore checks whether `root == nil`.

Unlike Python slicing reversal syntax, Go requires manual in place reversal using two pointers. One pointer starts at the beginning of the slice and the other starts at the end. The values are swapped until the pointers meet.

No integer overflow concerns exist here because node values are constrained between `-100` and `100`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every node is visited exactly once |
| Space | O(n) | Stack and result list can store up to n nodes |

The algorithm processes each node a single time, so the runtime grows linearly with the number of nodes in the tree.

The stack may contain many nodes at once in skewed trees, and the result list stores every node value, so the total auxiliary space usage is linear in the number of nodes.

## Edge Cases

### Empty Tree

An empty tree is represented by `root = null` or `root = nil`. This is an important edge case because attempting to access children of a nonexistent node would cause runtime errors.

The implementation handles this immediately with an early return:

```
if not root:
    return []
```

This guarantees safe execution.

### Single Node Tree

A tree containing only one node is the smallest nonempty valid input.

For example:

```
[1]
```

The traversal should simply return `[1]`.

The algorithm correctly handles this because the stack initially contains the root, the root is processed once, and reversing a single element list leaves it unchanged.

### Completely Skewed Tree

A tree may be entirely left skewed or right skewed, behaving similarly to a linked list.

Example:

```
1
 \
  2
   \
    3
```

Naive implementations sometimes mishandle child ordering in these cases.

The iterative solution works correctly because the stack always processes nodes in controlled Last In First Out order, and the final reversal restores the exact postorder sequence.

### Nodes With Only One Child

Some nodes may have only a left child or only a right child.

The implementation carefully checks whether children exist before pushing them onto the stack:

```
if node.left:
    stack.append(node.left)
```

This prevents invalid accesses and ensures traversal remains correct regardless of tree shape.
