## Solution

## Problem Understanding

The problem asks us to create a **deep copy** of a linked list where each node contains two pointers: a standard `next` pointer and an additional `random` pointer.

In a normal singly linked list, each node only points to the next node. Here, every node also has a `random` pointer that may point to **any node in the list**, including itself, or it may be `null`.

A **deep copy** means we must create an entirely new linked list using brand new nodes. The copied nodes must preserve the exact same structure as the original list:

- The `next` relationships must remain identical.
- The `random` relationships must also remain identical.
- No copied node may point back to an original node.

For example, suppose node `A` in the original list points to node `B` through its `random` pointer. Then in the copied list, the copied version of `A` must point to the copied version of `B`, not to the original `B`.

The input is represented conceptually as a list of pairs:

```
[val, random_index]
```

Here:

- `val` is the node value.
- `random_index` identifies which node the random pointer targets.
- `null` means there is no random pointer.

However, an important detail is that our function does **not** receive this array representation. Instead, we only receive the `head` of the linked list. Therefore, we must reconstruct the copied structure directly from pointer relationships.

The constraints tell us:

- The list size can be at most `1000` nodes.
- Node values range between `-10^4` and `10^4`.
- A random pointer can point anywhere within the list or be `null`.

Since `n ≤ 1000`, an `O(n²)` solution may still technically pass in some environments, but it is unnecessary and inefficient. We should aim for an `O(n)` solution.

Several edge cases immediately stand out:

An empty list (`head = null`) must return `null`.

A node may have a `random` pointer set to `null`, which means we must safely handle missing references.

Multiple nodes may share the same random target, so we cannot assume a one-to-one mapping between nodes and random references.

A node may point to itself through `random`, which means self references must be preserved correctly.

Node values are not guaranteed to be unique, so we cannot identify nodes by value alone. We must track nodes by their actual object references.

## Approaches

### Brute Force Approach

A straightforward idea is to first copy all nodes into a new list, then repeatedly search for the matching random target.

The process would work like this:

1. Traverse the original list and create copied nodes connected by `next`.
2. For each original node, inspect its `random` pointer.
3. Search through the original list to determine the index or identity of the random target.
4. Traverse the copied list again to find the equivalent copied node.
5. Assign the copied random pointer.

This works because every original node eventually gets a corresponding copied node, and we explicitly reconstruct both `next` and `random` relationships.

However, the problem is efficiency. For every node, we may scan the list again to locate its random target. This leads to repeated traversals.

If there are `n` nodes:

- Creating nodes takes `O(n)`.
- Searching for random targets for every node takes `O(n²)` in total.

This becomes unnecessarily expensive.

### Key Insight for an Optimal Solution

The key observation is that the difficulty comes from reconstructing `random` pointers efficiently.

We need a fast way to answer:

> Given an original node, what is its copied version?

A **hash map** solves this perfectly.

We can maintain:

```
original node -> copied node
```

mapping.

Then:

- Creating copied nodes becomes easy.
- Looking up the copied version of a random target becomes `O(1)`.

The algorithm becomes:

1. First pass: create all copied nodes and store mappings.
2. Second pass: connect `next` and `random` pointers using the hash map.

Because every lookup is constant time, the entire algorithm runs in linear time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeatedly searches for random pointer targets |
| Optimal | O(n) | O(n) | Uses a hash map to map original nodes to copied nodes |

## Algorithm Walkthrough

### Optimal Hash Map Approach

1. **Handle the empty list case**

If `head` is `None`, there is nothing to copy. Return `None` immediately.
2. **Create a hash map for node correspondence**

We create a dictionary:

```
original node → copied node
```

This allows us to instantly find the copied version of any original node.
3. **First traversal, create copied nodes**

Traverse the original list from head to tail.

For each original node:

- Create a new node with the same value.
- Store it in the hash map.

At this stage, we only copy values. We do not connect pointers yet because some referenced nodes may not exist yet.
4. **Second traversal, assign pointers**

Traverse the original list again.

For each node:

- Use the hash map to find its copied version.
- Set `copy.next` using:

```
map[original.next]
```

- Set `copy.random` using:

```
map[original.random]
```

Since every original node already has a copied version in the map, these assignments are immediate.
5. **Return the copied head**

The copied head is simply:

```
node_map[head]
```

This two pass strategy is clean and reliable because pointer assignment only happens after every node exists.

## Python Solution

```
"""
# Definition for a Node.
class Node:
    def __init__(self, x: int, next: 'Node' = None, random: 'Node' = None):
        self.val = int(x)
        self.next = next
        self.random = random
"""

from typing import Optional

class Solution:
    def copyRandomList(self, head: 'Optional[Node]') -> 'Optional[Node]':
        if not head:
            return None

        node_map: dict[Node, Node] = {}

        current = head

        # First pass: create copied nodes
        while current:
            node_map[current] = Node(current.val)
            current = current.next

        current = head

        # Second pass: assign next and random pointers
        while current:
            copied_node = node_map[current]

            copied_node.next = (
                node_map[current.next]
                if current.next
                else None
            )

            copied_node.random = (
                node_map[current.random]
                if current.random
                else None
            )

            current = current.next

        return node_map[head]
```

The implementation begins with an early return for an empty list. This prevents unnecessary processing and correctly handles the `null` input case.

The dictionary `node_map` stores relationships between original nodes and copied nodes. Using actual node objects as keys is important because values are not unique.

During the first traversal, we only create new nodes and store mappings. We intentionally delay pointer assignments because some referenced nodes may not yet exist.

In the second traversal, we reconstruct the graph structure of the list. Since every copied node already exists in `node_map`, assigning both `next` and `random` pointers becomes a constant time lookup.

Finally, we return the copied version of the original head.

## Go Solution

```
/**
 * Definition for a Node.
 * type Node struct {
 *     Val int
 *     Next *Node
 *     Random *Node
 * }
 */

func copyRandomList(head *Node) *Node {
	if head == nil {
		return nil
	}

	nodeMap := make(map[*Node]*Node)

	current := head

	// First pass: create copied nodes
	for current != nil {
		nodeMap[current] = &Node{
			Val: current.Val,
		}
		current = current.Next
	}

	current = head

	// Second pass: assign next and random pointers
	for current != nil {
		copiedNode := nodeMap[current]

		if current.Next != nil {
			copiedNode.Next = nodeMap[current.Next]
		}

		if current.Random != nil {
			copiedNode.Random = nodeMap[current.Random]
		}

		current = current.Next
	}

	return nodeMap[head]
}
```

The Go implementation follows the exact same algorithmic structure as the Python solution.

The biggest difference is pointer handling. In Go, `nil` is used instead of `None`, and node references are represented using pointers (`*Node`).

The hash map is declared as:

```
map[*Node]*Node
```

This means:

- Key = original node pointer
- Value = copied node pointer

This is critical because node values are not unique, so using integers as keys would be incorrect.

Unlike Python, Go requires explicit conditional checks before assigning `Next` and `Random` pointers.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Two linear traversals of the linked list |
| Space | O(n) | Hash map stores one copied node per original node |

The time complexity is linear because we traverse the list twice, once to create nodes and once to assign pointers. Each node is processed a constant number of times.

The space complexity is also linear because the hash map stores exactly one entry for every node in the original list.

## Edge Cases

### Empty List

If `head` is `null`, there are no nodes to copy.

This case is easy to overlook and could cause runtime errors if we immediately begin traversing the list. The implementation safely handles this by returning `None` in Python or `nil` in Go before any processing begins.

### Random Pointer Is Null

Some nodes may not have a random target.

A naive implementation might try to dereference a null pointer and fail. Our implementation explicitly checks whether `current.random` exists before looking it up in the hash map. If it does not exist, we assign `None` or `nil`.

### Self Referencing Random Pointer

A node may point to itself:

```
node.random = node
```

This can cause subtle bugs if references are copied incorrectly.

Because the hash map stores a one-to-one mapping between original and copied nodes, self references naturally work:

```
copied.random = node_map[original.random]
```

If `original.random` points to itself, the copied node will correctly point to itself as well.

### Duplicate Node Values

Multiple nodes may contain identical values.

A buggy solution might try to identify nodes by value, which would fail when duplicates exist.

Our implementation uses actual node references as keys in the hash map, guaranteeing correctness even when every node has the same value.
