## Solution

## Problem Understanding

The problem asks us to count how many distinct subsequences of a string `s` are equal to another string `t`.

A subsequence is formed by deleting zero or more characters from a string without changing the relative order of the remaining characters. This means characters do not need to be contiguous, but they must appear in the same order as in the original string.

For example, if `s = "babgbag"` and `t = "bag"`, there are multiple ways to choose characters from `s` so that the resulting subsequence spells `"bag"`.

The important detail is that we are counting distinct ways to form `t`, not simply checking whether `t` exists as a subsequence.

The input consists of two strings:

- `s`, the source string from which we can delete characters
- `t`, the target string we want to form

The output is a single integer representing the total number of distinct subsequences of `s` equal to `t`.

The constraints are important:

- `1 <= s.length, t.length <= 1000`
- Both strings contain only English letters

Since both strings can be length 1000, exponential solutions are completely infeasible. Any recursive brute-force approach without optimization would time out. This strongly suggests a dynamic programming solution.

Several edge cases are important:

- If `t` is longer than `s`, the answer must be `0`
- If `s` and `t` are identical, the answer is `1`
- Repeated characters can create many overlapping subsequences
- Large counts must still fit within a 32-bit signed integer, which the problem guarantees

## Approaches

### Brute Force Approach

A brute-force solution would recursively explore every possible subsequence of `s`.

At each character in `s`, we have two choices:

1. Include the character in the subsequence
2. Skip the character

This creates a binary recursion tree with up to `2^n` possibilities, where `n` is the length of `s`.

Whenever a generated subsequence matches `t`, we increment the answer.

This approach is correct because it explicitly checks every possible subsequence. However, it is far too slow. With `s.length = 1000`, the number of subsequences becomes astronomically large.

Additionally, many subproblems repeat. For example, the number of ways to match the suffix `"bit"` from a certain position in `s` may be computed many times.

### Dynamic Programming Insight

The key observation is that the problem contains overlapping subproblems.

Suppose we are trying to match:

- `s[i:]`, the suffix of `s` starting at index `i`
- `t[j:]`, the suffix of `t` starting at index `j`

At every step:

- If `s[i] != t[j]`, we cannot use this character, so we skip it
- If `s[i] == t[j]`, we have two choices:

- Use this character to match `t[j]`
- Skip this character and try matching later

This naturally leads to a recurrence relation.

Let `dp[i][j]` represent the number of ways to form `t[j:]` from `s[i:]`.

Then:

- If characters match:

`dp[i][j] = dp[i+1][j+1] + dp[i+1][j]`
- Otherwise:

`dp[i][j] = dp[i+1][j]`

This transforms an exponential problem into a polynomial one.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(n) | Recursively generates all subsequences |
| Optimal Dynamic Programming | O(m × n) | O(m × n) or O(n) | Reuses overlapping subproblems efficiently |

Here:

- `m = len(s)`
- `n = len(t)`

## Algorithm Walkthrough

### Dynamic Programming State Definition

We define:

`dp[i][j]`

as the number of distinct ways to form `t[j:]` using `s[i:]`.

This means:

- We are currently at position `i` in `s`
- We are currently trying to match position `j` in `t`

### Step-by-Step Algorithm

1. Create a 2D DP table with dimensions `(m + 1) × (n + 1)`.

We add one extra row and column to handle empty suffixes cleanly.
2. Initialize the base case for matching an empty target string.

If `t[j:]` is empty, there is exactly one way to form it, by choosing nothing.

Therefore:

```
dp[i][n] = 1
```

for all `i`.
3. Fill the table from bottom to top.

We process from the end because each state depends on future states like `dp[i+1][j]`.
4. Compare characters `s[i]` and `t[j]`.

If they match, we have two choices:

- Use the character
- Skip the character

So:

```
dp[i][j] = dp[i+1][j+1] + dp[i+1][j]
```
5. If the characters do not match, we must skip `s[i]`.

So:

```
dp[i][j] = dp[i+1][j]
```
6. After filling the table, return `dp[0][0]`.

This represents the number of ways to form the entire string `t` from the entire string `s`.

## Python Solution

```
class Solution:
    def numDistinct(self, s: str, t: str) -> int:
        m = len(s)
        n = len(t)

        dp = [[0] * (n + 1) for _ in range(m + 1)]

        # Empty target can always be formed
        for i in range(m + 1):
            dp[i][n] = 1

        for i in range(m - 1, -1, -1):
            for j in range(n - 1, -1, -1):
                if s[i] == t[j]:
                    dp[i][j] = dp[i + 1][j + 1] + dp[i + 1][j]
                else:
                    dp[i][j] = dp[i + 1][j]

        return dp[0][0]
```

The implementation starts by determining the lengths of both strings. The DP table is then created with one extra row and column so that empty suffixes can be represented naturally.

The initialization step sets `dp[i][n] = 1` for all `i`. This encodes the fact that an empty target string can always be formed by deleting all remaining characters.

The nested loops iterate backward through both strings. Processing backward is essential because each state depends on states below and to the right.

When characters match, the algorithm explores both possibilities:

- Use the current character
- Skip the current character

When characters do not match, skipping is the only valid option.

Finally, `dp[0][0]` contains the number of distinct subsequences for the full strings.

## Go Solution

```
func numDistinct(s string, t string) int {
    m := len(s)
    n := len(t)

    dp := make([][]int, m+1)
    for i := range dp {
        dp[i] = make([]int, n+1)
    }

    // Empty target can always be formed
    for i := 0; i <= m; i++ {
        dp[i][n] = 1
    }

    for i := m - 1; i >= 0; i-- {
        for j := n - 1; j >= 0; j-- {
            if s[i] == t[j] {
                dp[i][j] = dp[i+1][j+1] + dp[i+1][j]
            } else {
                dp[i][j] = dp[i+1][j]
            }
        }
    }

    return dp[0][0]
}
```

The Go implementation follows the same logic as the Python solution.

The primary difference is explicit slice allocation. Go does not provide dynamic multidimensional arrays automatically, so each row must be allocated individually.

Both implementations use integers safely because the problem guarantees the result fits within a 32-bit signed integer.

Strings in Go are byte indexed, which works correctly here because the input contains only English letters.

We build the DP table from the bottom upward.

Let:

- Rows represent positions in `s`
- Columns represent positions in `t`

The important transitions occur around the repeated `'b'` characters.

At each matching `'b'`, the algorithm branches into:

- Use this `'b'`
- Skip this `'b'`

This creates multiple valid combinations.

Key states:

| --- | --- | --- |
| dp[6][5] | Match `"t"` from `"t"` | 1 |

| dp[3][2] | Match `"bbit"` from `"bbit"` | 2 |

dp[0][0] = 3

There are exactly 3 distinct ways to form `"rabbit"`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m × n) | Every DP state is computed exactly once |
| Space | O(m × n) | The DP table stores all subproblem results |

The nested loops iterate through every pair of indices `(i, j)` exactly once. Since there are `m × n` states, the runtime is quadratic.

The DP table also contains `m × n` entries, giving the same space complexity.

An optimized 1D DP version can reduce space complexity to `O(n)`, but the 2D version is easier to understand and debug.

## Edge Cases

### Target Longer Than Source

Consider:

```
s = "abc"
t = "abcd"
```

It is impossible to form a longer string from a shorter one using deletions alone.

The DP table naturally handles this because no valid transitions can complete the match, so the result becomes `0`.

### Identical Strings

Consider:

```
s = "abc"
t = "abc"
```

There is exactly one valid subsequence, using every character.

The algorithm correctly computes this because each character match contributes exactly one valid path through the DP table.

### Repeated Characters

Consider:

```
s = "aaaaa"
t = "aa"
```

This case is tricky because many overlapping subsequences exist.

The correct answer is:

```
10
```

since we are choosing any 2 positions from 5.

Naive recursive solutions often recompute the same states repeatedly. The DP solution avoids this by storing intermediate results.

### Empty Remaining Target

During computation, the algorithm frequently reaches states where all characters of `t` have already been matched.

For example:

```
t[j:] = ""
```

There is exactly one way to complete the subsequence from this point, by choosing nothing else.

This is why the initialization:

```
dp[i][n] = 1
```

is essential for correctness.
