## Solution

## Problem Understanding

The problem describes a row of houses, where each house contains some amount of money. You are acting as a robber who wants to maximize the amount of money stolen, but there is a strict restriction: you cannot rob two adjacent houses. If you rob neighboring houses on the same night, the alarm system is triggered and the police are contacted.

The input is an integer array `nums`, where each element represents the amount of money stored in a specific house. For example, if `nums = [2,7,9,3,1]`, the first house contains 2 dollars, the second contains 7, and so on.

The expected output is a single integer, representing the maximum amount of money that can be robbed without ever robbing two adjacent houses.

At first glance, this may seem like a greedy problem where we simply pick houses with the largest values. However, that approach fails because choosing one house directly affects which neighboring houses are available. For example, robbing a large-value house might prevent robbing two moderately valuable neighbors whose combined total is larger.

The constraints are important:

- `1 <= nums.length <= 100`
- `0 <= nums[i] <= 400`

The number of houses is relatively small, only up to 100, which means even an `O(n²)` solution would technically work. However, the structure of the problem strongly suggests there is a more elegant dynamic programming solution that runs in linear time.

Several edge cases should immediately stand out:

A street containing only one house, such as `[5]`, must return `5` because there are no adjacency conflicts.

A street with two houses, such as `[2,7]`, requires choosing the larger amount since both cannot be robbed.

Houses with zero money, such as `[0,0,0]`, should still work correctly and return `0`.

Cases where skipping a large immediate reward leads to a better total later, such as `[2,1,1,2]`, are especially important because they expose flaws in greedy thinking.

## Approaches

### Brute Force Approach

A straightforward way to solve this problem is to recursively try every possible decision at each house.

For every house, there are two choices:

1. Rob the current house and skip the next one.
2. Skip the current house and move to the next one.

We recursively compute the best possible result for both choices and take the maximum.

For example, if we are at index `i`, we can calculate:

- Rob current house: `nums[i] + solve(i + 2)`
- Skip current house: `solve(i + 1)`

The answer becomes:

```
max(nums[i] + solve(i + 2), solve(i + 1))
```

This brute-force approach is correct because it explores every valid robbery combination and guarantees the maximum result is found. However, it performs excessive repeated work.

For instance, when calculating `solve(0)`, we may repeatedly recompute `solve(2)`, `solve(3)`, and so on from multiple recursive paths. This overlap causes exponential growth in runtime.

The time complexity becomes `O(2ⁿ)`, which is inefficient as input size grows.

### Optimal Approach, Dynamic Programming

The key observation is that the problem contains **overlapping subproblems**.

At any house `i`, the maximum money we can rob depends only on earlier decisions. Specifically, we only care about two states:

- The best amount if we skip the current house.
- The best amount if we rob the current house.

For each house, we decide:

- Skip it, keeping the previous maximum.
- Rob it, adding its money to the maximum from two houses ago.

This leads to the recurrence relation:

```
dp[i] = max(dp[i - 1], dp[i - 2] + nums[i])
```

Where:

- `dp[i - 1]` means skipping the current house.
- `dp[i - 2] + nums[i]` means robbing the current house.

Since we only ever need the previous two values, we can optimize space and avoid storing the entire `dp` array.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2ⁿ) | O(n) | Recursively explores every valid robbery combination, repeated work causes exponential growth |
| Optimal | O(n) | O(1) | Dynamic programming with two running variables, processes each house once |

## Algorithm Walkthrough

1. Handle the simplest case implicitly by initializing two tracking variables. We do not need separate special handling for one house because the logic naturally works for all valid inputs.
2. Maintain two variables:

- `previous_two`, representing the maximum money that could be robbed up to two houses ago.
- `previous_one`, representing the maximum money that could be robbed up to the previous house.
3. Iterate through each house amount in `nums`.
4. For the current house, calculate two choices:

- Skip the house, which keeps the value `previous_one`.
- Rob the house, which gives `previous_two + current_money`.
5. Take the maximum of those two choices because we always want the best possible profit.
6. Update the variables:

- `previous_two` becomes the old `previous_one`.
- `previous_one` becomes the newly computed maximum.
7. After processing all houses, return `previous_one`, because it stores the maximum amount obtainable from the entire street.

This approach works because every decision depends only on the best results from the previous one or two houses. Since adjacent houses cannot both be robbed, the optimal decision at each step naturally builds from earlier optimal decisions.

## Python Solution

```
from typing import List

class Solution:
    def rob(self, nums: List[int]) -> int:
        previous_two = 0
        previous_one = 0

        for money in nums:
            current_max = max(previous_one, previous_two + money)
            previous_two = previous_one
            previous_one = current_max

        return previous_one
```

The implementation uses two variables instead of a dynamic programming array.

`previous_two` stores the best amount achievable up to two houses before the current one. `previous_one` stores the best amount achievable up to the immediately previous house.

For each house, we compute the best possible amount by comparing two options:

- Skip the house, keeping `previous_one`.
- Rob the house, adding the current money to `previous_two`.

The result is stored in `current_max`.

After computing the best value for the current position, we shift the state variables forward. The old `previous_one` becomes the new `previous_two`, and the newly computed maximum becomes `previous_one`.

At the end of the loop, `previous_one` contains the best achievable amount across all houses.

## Go Solution

```
func rob(nums []int) int {
	previousTwo := 0
	previousOne := 0

	for _, money := range nums {
		currentMax := max(previousOne, previousTwo+money)
		previousTwo = previousOne
		previousOne = currentMax
	}

	return previousOne
}

func max(a, b int) int {
	if a > b {
		return a
	}
	return b
}
```

The Go implementation follows exactly the same logic as the Python solution, but Go does not include a built-in `max` function for integers, so we define one manually.

Go slices naturally handle varying input lengths, and since the problem guarantees at least one house exists, we do not need special handling for empty input.

Integer overflow is not a concern here because the maximum possible value is small. At worst, there are 100 houses each containing 400 dollars, producing a maximum sum of 40,000, which easily fits within Go's `int` type.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | We process each house exactly once |
| Space | O(1) | Only a constant number of variables are used |

The time complexity is `O(n)` because the algorithm performs a single pass through the input array. Each house contributes a constant amount of work, specifically one maximum comparison and a few assignments.

The space complexity is `O(1)` because we avoid storing an entire dynamic programming table. Instead, only two running variables are maintained regardless of input size.

## Edge Cases

### Single House

If the input contains only one house, such as:

```
[10]
```

the algorithm correctly returns `10`.

This case can easily cause bugs in solutions that assume at least two elements exist. Our implementation avoids this problem because the loop naturally processes one element and updates the running maximum.

### Two Houses

For an input like:

```
[2,7]
```

the robber must choose only one house because robbing both would violate the adjacency rule.

The algorithm correctly compares both possibilities and returns `7`.

This prevents mistakes where implementations accidentally sum both houses.

### All Zero Values

For input such as:

```
[0,0,0]
```

the correct answer is `0`.

Even though robbing houses is technically allowed, there is no money to gain. The algorithm still behaves correctly because the maximum comparisons always remain zero.

### Non Greedy Cases

Consider:

```
[2,1,1,2]
```

A naive greedy strategy might pick the first `2`, then incorrectly choose only one of the middle houses, missing the best answer.

The optimal result is:

```
2 + 2 = 4
```

Our dynamic programming approach handles this correctly because every decision considers both robbing and skipping, ensuring future opportunities are not ignored.
