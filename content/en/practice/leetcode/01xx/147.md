## Solution

## Problem Understanding

The problem asks us to sort a singly linked list using the insertion sort algorithm. Instead of sorting an array, we must rearrange the nodes of a linked list so that the final list is in ascending order.

Insertion sort works by maintaining a sorted portion of the data and inserting one new element into its correct position during each iteration. In the context of a linked list, this means we repeatedly take the next node from the original list and insert it into the correct location within an already sorted linked list.

The input is the head node of a singly linked list. Each node contains an integer value and a pointer to the next node. The output should be the head of the newly sorted linked list. We are expected to rearrange the existing nodes rather than create a completely new collection of values.

The constraints tell us that the list can contain between 1 and 5000 nodes, and node values range from -5000 to 5000. A list of 5000 elements is small enough that an O(n²) algorithm is acceptable. This is important because insertion sort naturally runs in quadratic time in the worst case.

Several edge cases are important to consider:

- A list with only one node is already sorted.
- A list that is already sorted should remain unchanged.
- A list sorted in descending order represents the worst case for insertion sort because every node must be inserted at the beginning.
- Duplicate values must be handled correctly while preserving valid list structure.
- Negative numbers must be sorted correctly alongside positive numbers.

Because this is a singly linked list, we cannot move backward through the list. That limitation strongly influences how we implement insertion sort efficiently.

## Approaches

### Brute Force Approach

One straightforward approach is to extract all node values into an array, sort the array using a built-in sorting algorithm, and then rebuild the linked list using the sorted values.

This works because arrays are much easier to sort efficiently than linked lists. After sorting the values, we can either modify the existing nodes or construct a new sorted linked list.

The approach is correct because sorting the extracted values guarantees the final ordering is correct. However, it does not truly use insertion sort on the linked list itself, which is what the problem is specifically asking for.

This method also uses additional memory proportional to the size of the list because all values must be copied into an array.

### Optimal Approach

The intended solution is to perform insertion sort directly on the linked list.

The key observation is that insertion sort is actually very natural for linked lists. In arrays, inserting an element requires shifting many values to make room. In linked lists, insertion only requires pointer manipulation.

We maintain a separate sorted list and iterate through the original list one node at a time. For each node:

1. Remove it from the input list.
2. Find the correct insertion position in the sorted list.
3. Insert the node by adjusting pointers.

A dummy head node simplifies insertion logic, especially when inserting at the beginning of the sorted list.

Although this approach still has O(n²) time complexity in the worst case, it uses only O(1) extra space because nodes are rearranged in place.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) | O(n) | Copy values to array, sort array, rebuild list |
| Optimal | O(n²) | O(1) | Perform insertion sort directly on linked list |

## Algorithm Walkthrough

1. Create a dummy node to serve as the head of the sorted list. This simplifies insertion operations because every insertion can be handled uniformly, even when inserting at the beginning.
2. Start iterating through the original linked list using a pointer called `current`. This pointer represents the node we want to insert into the sorted portion.
3. Before modifying pointers, store `current.next` in a temporary variable. This prevents losing access to the remaining unsorted portion of the list.
4. Starting from the dummy node, search through the sorted list to find the correct insertion position. Continue moving forward while the next node's value is smaller than the current node's value.
5. Insert the current node into the sorted list:

- Point `current.next` to the node after the insertion position.
- Point the insertion position's `next` to `current`.
6. Move to the next node from the original list using the temporary variable saved earlier.
7. Repeat this process until all nodes have been inserted into the sorted list.
8. Return `dummy.next`, which points to the actual head of the sorted linked list.

The reason this works is that after each iteration, the sorted list remains correctly ordered. Each new node is inserted into exactly the right position relative to the nodes already processed.

## Python Solution

```
# Definition for singly-linked list.
# class ListNode:
#     def __init__(self, val=0, next=None):
#         self.val = val
#         self.next = next

from typing import Optional

class Solution:
    def insertionSortList(self, head: Optional[ListNode]) -> Optional[ListNode]:
        dummy = ListNode(0)
        current = head

        while current:
            next_node = current.next

            prev = dummy
            while prev.next and prev.next.val < current.val:
                prev = prev.next

            current.next = prev.next
            prev.next = current

            current = next_node

        return dummy.next
```

The implementation begins by creating a dummy node. This node acts as a stable starting point for the sorted list and avoids special handling when inserting at the head.

The `current` pointer traverses the original list. During each iteration, the algorithm stores `current.next` in `next_node` before changing any pointers. This is necessary because insertion operations will modify `current.next`.

The variable `prev` searches for the insertion position within the sorted list. It starts from the dummy node every time because the correct insertion position could be anywhere in the sorted portion.

Once the correct position is found, the node is inserted using two pointer updates:

```
current.next = prev.next
prev.next = current
```

The first line connects the current node to the remainder of the sorted list. The second line inserts the node after `prev`.

Finally, the algorithm continues with the next unsorted node.

## Go Solution

```
/**
 * Definition for singly-linked list.
 * type ListNode struct {
 *     Val int
 *     Next *ListNode
 * }
 */
func insertionSortList(head *ListNode) *ListNode {
    dummy := &ListNode{}
    current := head

    for current != nil {
        nextNode := current.Next

        prev := dummy
        for prev.Next != nil && prev.Next.Val < current.Val {
            prev = prev.Next
        }

        current.Next = prev.Next
        prev.Next = current

        current = nextNode
    }

    return dummy.Next
}
```

The Go implementation follows exactly the same logic as the Python version.

The primary difference is pointer syntax. Go uses explicit pointer types, so node creation uses `&ListNode{}` and field access uses dot notation on pointers automatically.

The loop condition checks `current != nil` instead of Python's truthy check.

Because Go handles pointers explicitly, careful ordering of assignments is important to avoid losing references to nodes. The logic remains identical to the Python implementation.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n²) | Each insertion may scan the sorted portion of the list |
| Space | O(1) | Sorting is performed in place using existing nodes |

The time complexity is quadratic because for every node, we may need to scan through nearly the entire sorted portion to find the insertion position. In the worst case, such as a reverse sorted list, the total work becomes:

```
1 + 2 + 3 + ... + (n - 1)
```

This sums to O(n²).

The space complexity is constant because the algorithm only uses a few pointer variables. No extra arrays or auxiliary data structures proportional to input size are created.

## Edge Cases

### Single Node List

A list containing only one node is already sorted. This case can sometimes expose pointer handling bugs because there is no next node to process. In this implementation, the loop runs exactly once, inserts the node into the sorted list, and terminates correctly.

### Already Sorted List

For an already sorted list such as:

```
1 -> 2 -> 3 -> 4
```

each node is inserted at the end of the sorted portion. The algorithm still works correctly because it scans until it finds the insertion point after all smaller elements. Although no structural changes are really needed, the pointer updates remain valid.

### Reverse Sorted List

A reverse sorted list such as:

```
4 -> 3 -> 2 -> 1
```

is the worst case for insertion sort. Every new node must be inserted at the beginning of the sorted list. This stresses the insertion logic heavily, especially insertion immediately after the dummy node. The dummy node ensures this case is handled cleanly without special conditions.

### Duplicate Values

A list with duplicate values such as:

```
2 -> 1 -> 2
```

must preserve correct ordering while allowing equal values. The comparison:

```
prev.next.val < current.val
```

stops when equal values are encountered, so duplicates are inserted before larger elements and remain properly grouped together.

### Negative Values

Negative numbers can expose incorrect comparison logic if assumptions are made about positivity. Since the algorithm relies only on standard integer comparison, values like:

```
-3 -> -1 -> -2
```

are handled naturally and sorted correctly.
