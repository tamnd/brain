## Solution

## Problem Understanding

The problem gives us a sorted integer array called `numbers` and a target integer called `target`. The array is already sorted in non-decreasing order, which means values can stay the same or increase as we move from left to right.

We need to find exactly two different elements whose sum equals the target value. Instead of returning the values themselves, we must return their indices. One important detail is that the array is 1-indexed for the output, even though programming languages typically use 0-based indexing internally.

The problem guarantees that there is exactly one valid answer. This guarantee is very important because it simplifies the implementation. We do not need to handle cases where multiple answers exist or where no answer exists.

The constraint that the solution must use only constant extra space rules out approaches that rely on auxiliary data structures such as hash maps. In the classic "Two Sum" problem, a hash map is usually the best approach, but here the array is already sorted, which allows a more space-efficient strategy.

The input size can be as large as `3 * 10^4`, which means an `O(n^2)` brute-force solution may become too slow. We should aim for a linear or near-linear solution.

Several edge cases are important to consider:

- The array may contain negative numbers.
- The valid pair may appear at the beginning or end of the array.
- Duplicate values may exist.
- The target may be negative.
- The array contains exactly one solution, so we never need to handle ambiguity.

Because the array is sorted, we can take advantage of ordering information to efficiently narrow down the search space.

## Approaches

### Brute Force Approach

The most straightforward solution is to check every possible pair of numbers.

We can use two nested loops. The outer loop selects the first number, and the inner loop checks every number after it to see whether the pair sums to the target.

For each pair `(i, j)`:

- Compute `numbers[i] + numbers[j]`
- If the sum equals `target`, return `[i + 1, j + 1]`

This approach is correct because it exhaustively examines every possible pair, so the valid pair will eventually be found.

However, this method is inefficient. If the array contains `n` elements, then in the worst case we examine approximately `n^2 / 2` pairs. With `n` up to `30,000`, this becomes too slow.

### Optimal Two Pointer Approach

The key observation is that the array is already sorted.

Because the numbers are ordered, we can intelligently adjust pointers instead of checking every pair.

We place:

- One pointer at the beginning of the array, called `left`
- One pointer at the end of the array, called `right`

Then:

- Compute the sum of `numbers[left] + numbers[right]`
- If the sum is too small, move `left` forward to increase the sum
- If the sum is too large, move `right` backward to decrease the sum
- If the sum equals the target, we found the answer

This works because sorting gives us predictable behavior:

- Moving the left pointer right increases the value
- Moving the right pointer left decreases the value

Instead of exploring all pairs, we eliminate large portions of the search space with each step.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Checks every possible pair |
| Optimal Two Pointers | O(n) | O(1) | Uses sorted order to eliminate impossible pairs efficiently |

## Algorithm Walkthrough

1. Initialize two pointers.

Set `left = 0` and `right = len(numbers) - 1`.

The left pointer starts at the smallest value, and the right pointer starts at the largest value.
2. Compute the current sum.

Calculate:

```
current_sum = numbers[left] + numbers[right]
```

This tells us whether we need a larger or smaller total.
3. Compare the sum with the target.

- If `current_sum == target`, we found the correct pair.
- If `current_sum < target`, the sum is too small, so move `left` one step to the right.
- If `current_sum > target`, the sum is too large, so move `right` one step to the left.
4. Continue shrinking the search space.

Each pointer movement removes impossible combinations while preserving all potentially valid pairs.
5. Return 1-indexed positions.

Since the problem expects 1-indexed answers, return:

```
[left + 1, right + 1]
```

The algorithm works because the sorted property guarantees that moving pointers changes the sum in a predictable direction.

## Python Solution

```
from typing import List

class Solution:
    def twoSum(self, numbers: List[int], target: int) -> List[int]:
        left = 0
        right = len(numbers) - 1

        while left < right:
            current_sum = numbers[left] + numbers[right]

            if current_sum == target:
                return [left + 1, right + 1]

            if current_sum < target:
                left += 1
            else:
                right -= 1

        return []
```

The implementation begins by initializing two pointers. The `left` pointer starts at the beginning of the array, while the `right` pointer starts at the end.

Inside the loop, we compute the sum of the two pointed values. If the sum matches the target, we immediately return the 1-indexed positions.

If the sum is too small, we increase the sum by moving `left` forward. Because the array is sorted, every value to the right is greater than or equal to the current value.

If the sum is too large, we decrease the sum by moving `right` backward. Since values decrease as we move leftward, this reduces the total.

The loop continues until the correct pair is found. The problem guarantees exactly one solution, so the function will always return before the loop ends.

## Go Solution

```
func twoSum(numbers []int, target int) []int {
    left := 0
    right := len(numbers) - 1

    for left < right {
        currentSum := numbers[left] + numbers[right]

        if currentSum == target {
            return []int{left + 1, right + 1}
        }

        if currentSum < target {
            left++
        } else {
            right--
        }
    }

    return []int{}
}
```

The Go implementation follows the same logic as the Python version.

Slices in Go behave similarly to dynamic arrays, so accessing elements by index is straightforward. Since the constraints are small, integer overflow is not a concern here.

The function returns an empty slice at the end only for completeness. According to the problem guarantees, execution will always find a valid pair before reaching that line.

Initial state:

| --- | --- | --- | --- | --- |

The sum is greater than 9, so move `right` leftward.

| left | right | numbers[left] | numbers[right] | sum |
| --- | --- | --- | --- | --- |
| 0 | 2 | 2 | 11 | 13 |

The sum is still greater than 9, so move `right` again.

| left | right | numbers[left] | numbers[right] | sum |
| --- | --- | --- | --- | --- |
| 0 | 1 | 2 | 7 | 9 |

The sum equals the target.

Return:

```
[1, 2]
```

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each pointer moves at most once across the array |
| Space | O(1) | Only a few variables are used |

The time complexity is linear because every iteration moves either the left pointer or the right pointer. Since each pointer can move at most `n` times, the total number of operations is proportional to the array length.

The space complexity is constant because the algorithm uses only a few integer variables regardless of input size. No auxiliary data structures are allocated.

## Edge Cases

### Arrays Containing Negative Numbers

An input such as:

```
numbers = [-10, -3, 0, 5, 9]
target = -13
```

can cause problems for implementations that incorrectly assume all numbers are positive.

The two-pointer approach still works perfectly because it relies only on sorted order, not positivity. The sum comparisons remain valid regardless of whether values are negative or positive.

### Duplicate Values

Consider:

```
numbers = [1, 2, 3, 3, 4]
target = 6
```

The correct answer uses the duplicate `3` values.

A buggy implementation might accidentally reuse the same element twice or skip duplicates incorrectly. The two-pointer solution avoids this issue because it always maintains `left < right`, guaranteeing distinct indices.

### Smallest Possible Input Size

The minimum input length is 2:

```
numbers = [-1, 0]
target = -1
```

Some implementations fail due to incorrect loop conditions or pointer initialization.

This implementation handles the case correctly because the pointers are initialized to opposite ends of the array and checked immediately.

### Pair Located at the Extremes

Consider:

```
numbers = [1, 2, 3, 10]
target = 11
```

The answer uses the first and last elements.

The two-pointer method is especially effective here because it begins by testing the extreme values immediately. No extra work is required.
