# LeetCode 124, Binary Tree Maximum Path Sum

## Problem Understanding

This problem asks us to compute the maximum possible sum of values along any valid path in a binary tree.

A path is defined as a sequence of connected nodes where every adjacent pair has an edge between them. The path does not need to start at the root, and it does not need to end at a leaf. The only restriction is that a node cannot appear more than once in the same path.

This definition is important because it means the optimal path can exist entirely inside a subtree and may never touch the root node.

The input is the root node of a binary tree. Each node contains an integer value, which may be positive, zero, or negative.

The output is a single integer representing the largest possible path sum among all non-empty paths in the tree.

A key observation is that a valid path can move:

- from parent to child
- from child to parent
- through a node connecting both left and right children

However, the path cannot branch multiple times. A valid path is always a single continuous chain.

The constraints are large enough that brute force enumeration of all possible paths is not feasible:

- Up to `3 * 10^4` nodes
- Node values may be negative

The presence of negative values creates several tricky cases. Sometimes including a subtree actually decreases the total sum, so the algorithm must be able to discard harmful branches.

Some important edge cases include:

- A tree containing only negative numbers
- A tree with only one node
- A skewed tree that behaves like a linked list
- A path that does not include the root
- Large trees where recursion depth and efficiency matter

The problem guarantees the tree is non-empty, so there is always at least one valid path.

## Approaches

### Brute Force Approach

A brute force solution would attempt to evaluate every possible path in the tree.

One way to think about this is:

1. Start from every node
2. Explore every possible connected path
3. Compute the sum of each path
4. Keep track of the maximum

This works because eventually every valid path will be considered.

However, the number of possible paths in a tree grows very quickly. For each node, many combinations of descendant paths are possible. Recomputing sums repeatedly across overlapping subtrees becomes extremely expensive.

In the worst case, this approach can approach quadratic or even worse complexity depending on implementation details.

Given that the tree may contain up to `30,000` nodes, brute force is far too slow.

### Optimal Depth-First Search Approach

The key insight is that every path has a highest point, sometimes called the "turning node".

For any node, there are two different quantities we care about:

1. The best path that can be extended upward to the parent
2. The best complete path that passes through the node

This distinction is the core of the problem.

Suppose we are currently processing a node.

A path extending upward can only choose one side:

- node + left branch
- node + right branch
- node alone

It cannot take both children because the parent could not continue such a branched structure.

However, when computing the global answer at the current node, we _can_ combine:

- left contribution
- current node
- right contribution

This creates a complete path that may represent the maximum answer.

Depth-first search works perfectly here because child information must be computed before the parent can evaluate its own best path.

At each node:

- recursively compute the best extendable gain from the left subtree
- recursively compute the best extendable gain from the right subtree
- ignore negative gains by replacing them with zero
- update the global maximum using:

current node + left gain + right gain
- return the best single-side path upward

This produces an `O(n)` solution because every node is processed exactly once.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) or worse | O(h) | Repeatedly explores overlapping paths |
| Optimal DFS | O(n) | O(h) | Single traversal using postorder DFS |

Here, `h` represents the height of the tree.

## Algorithm Walkthrough

1. Initialize a global variable to store the maximum path sum found so far.

This variable is necessary because the best path may occur anywhere in the tree, not necessarily at the root.
2. Perform a depth-first traversal of the tree.

We use postorder traversal because a node depends on results from both children before computing its own value.
3. For each node, recursively compute the maximum gain from the left subtree.

This gain represents the best path starting from the left child and extending downward.
4. Recursively compute the maximum gain from the right subtree.

This is handled exactly the same way as the left side.
5. Ignore negative contributions.

If a subtree contributes a negative sum, including it would reduce the overall path sum. Therefore we replace negative gains with zero.

Mathematically:

```
left_gain = max(0, left_result)
right_gain = max(0, right_result)
```
6. Compute the best complete path passing through the current node.

This path may include:

- left subtree
- current node
- right subtree

The value becomes:

```
current_path_sum = node.val + left_gain + right_gain
```
7. Update the global maximum.

If the current path sum is larger than the previously recorded maximum, store it.
8. Return the best extendable path upward.

Since a parent cannot continue through both children simultaneously, we return only one branch:

```
node.val + max(left_gain, right_gain)
```
9. Continue recursively until all nodes are processed.
10. Return the global maximum.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import Optional

class Solution:
    def maxPathSum(self, root: Optional[TreeNode]) -> int:
        self.maximum_sum = float("-inf")

        def dfs(node: Optional[TreeNode]) -> int:
            if not node:
                return 0

            left_gain = max(dfs(node.left), 0)
            right_gain = max(dfs(node.right), 0)

            current_path_sum = (
                node.val + left_gain + right_gain
            )

            self.maximum_sum = max(
                self.maximum_sum,
                current_path_sum
            )

            return node.val + max(left_gain, right_gain)

        dfs(root)

        return self.maximum_sum
```

The implementation uses a nested recursive helper function named `dfs`.

The helper function returns the maximum gain that can be extended upward from the current node. This is important because a parent node can only continue along one branch.

The first base case checks whether the node is `None`. In that situation, the contribution is zero because there is no path.

Next, the algorithm recursively computes the best gains from the left and right subtrees. Negative values are discarded using `max(..., 0)` because including them would only reduce the total path sum.

The variable `current_path_sum` represents a complete path that passes through the current node and possibly connects both subtrees.

The global variable `self.maximum_sum` stores the best answer found anywhere in the tree.

Finally, the function returns the best single-branch path that can continue upward toward the parent.

The outer function starts the DFS traversal and returns the final maximum path sum.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */

func maxPathSum(root *TreeNode) int {
    maxSum := -1 << 31

    var dfs func(node *TreeNode) int

    dfs = func(node *TreeNode) int {
        if node == nil {
            return 0
        }

        leftGain := max(dfs(node.Left), 0)
        rightGain := max(dfs(node.Right), 0)

        currentPathSum := node.Val + leftGain + rightGain

        if currentPathSum > maxSum {
            maxSum = currentPathSum
        }

        return node.Val + max(leftGain, rightGain)
    }

    dfs(root)

    return maxSum
}

func max(a int, b int) int {
    if a > b {
        return a
    }
    return b
}
```

The Go solution follows exactly the same algorithmic structure as the Python implementation.

Since Go does not provide a built-in `max` function for integers, a helper function is defined manually.

The recursive DFS function is declared as a closure so it can access and update `maxSum`.

Go uses `nil` instead of Python's `None` for missing child pointers.

The initialization value `-1 << 31` acts as a sufficiently small integer so that trees containing all negative values are handled correctly.

## Worked Examples

Input:

root = [1,2,3]
```

Tree structure:

     / \

### DFS Traversal

| Node | Left Gain | Right Gain | Current Path Sum | Global Maximum | Return Value |
| --- | --- | --- | --- | --- | --- |
| 2 | 0 | 0 | 2 | 2 | 2 |
| 3 | 0 | 0 | 3 | 3 | 3 |
| 1 | 2 | 3 | 6 | 6 | 4 |

At node `1`, the algorithm combines both child gains:

```
2 + 1 + 3 = 6
```

This becomes the final answer.

Output:

```
6

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every node is visited exactly once |
| Space | O(h) | Recursive call stack uses tree height |

The algorithm performs a single DFS traversal across the entire tree. Each node performs only constant work after receiving results from its children.

The auxiliary space comes from recursion depth. In a balanced tree, the height is approximately `O(log n)`. In the worst case, a completely skewed tree behaves like a linked list, producing `O(n)` recursion depth.

## Edge Cases

### Tree With All Negative Values

Consider a tree like:

```
[-3, -2, -5]
```

A common bug is incorrectly forcing the answer to be at least zero. However, the problem requires a non-empty path, so the correct answer is the least negative node.

The implementation handles this correctly because the global maximum is initialized to negative infinity rather than zero.

### Single Node Tree

For a tree containing only one node:

```
[5]
```

the maximum path is simply the node itself.

The DFS computes zero gains from both missing children and correctly returns:

```
5
```

This verifies the algorithm works even at minimum input size.

### Highly Skewed Tree

Consider a tree shaped like a linked list:

```
1
 \
  2
   \
    3
```

In this situation there is no branching path. The algorithm naturally reduces to selecting the best linear chain because each node can only extend through one child.

The recursion still works correctly because every node returns the best single extendable path upward.

### Negative Child Contributions

Consider:

```
    10
   /  \
 -5    20
```

Including `-5` would reduce the total path sum.

The implementation avoids this issue with:

```
max(dfs(node.left), 0)
```

Negative gains are discarded entirely, ensuring only beneficial branches contribute to the final answer.
