## Solution

## Problem Understanding

This problem asks us to determine whether a binary tree is **height-balanced**.

A binary tree is considered height-balanced if, for **every node in the tree**, the height difference between its left subtree and right subtree is **at most 1**. In other words, no node can have one subtree that is significantly deeper than the other.

The input is the root node of a binary tree. Each node contains a value and references to its left and right child nodes. The values themselves are irrelevant to the balancing condition, only the structure of the tree matters.

The expected output is a boolean value:

- Return `true` if the tree is balanced.
- Return `false` if any node violates the balance rule.

To better understand the condition, consider what "height" means in a binary tree. The height of a node is the number of levels beneath it, including itself. For a node to be balanced, the heights of its left and right subtrees must differ by no more than one.

The constraints tell us that the tree contains between `0` and `5000` nodes. This is small enough that several recursive approaches are feasible, but inefficient repeated computations can still become problematic. Since binary tree operations often involve recursion, we should aim for a solution that visits each node as few times as possible.

There are several important edge cases to keep in mind:

A completely empty tree should return `true`. Since there are no nodes that violate the balance condition, an empty tree is considered balanced.

A tree containing only one node is also balanced because both left and right subtree heights are `0`.

Highly skewed trees, such as a linked-list shaped tree where every node only has a left child, are important because they expose inefficient implementations and frequently violate the balance requirement.

Another subtle case is when the root itself appears balanced, but a deeper subtree is not. A naive implementation that only checks the root would fail here. The algorithm must validate every node.

## Approaches

### Brute Force Approach

A straightforward approach is to recursively check every node and compute the height of its left and right subtree independently.

For each node:

1. Compute the height of the left subtree.
2. Compute the height of the right subtree.
3. Check whether their difference exceeds `1`.
4. Recursively validate the left and right subtrees.

This approach is correct because every node is explicitly verified against the balance condition.

However, the issue is inefficiency. The height of the same subtree gets recalculated many times. For example, when checking a node near the top of the tree, we compute subtree heights. Then later, when recursively checking child nodes, we recompute many of those same heights again.

In the worst case, such as a skewed tree, height computation becomes repeated work, producing a time complexity of `O(n²)`.

### Optimal Approach, Bottom-Up DFS

The key observation is that while checking balance, we also need subtree heights. Instead of computing heights repeatedly, we can compute height and balance information in a **single traversal**.

We use a **postorder depth-first search**:

- First process the left subtree.
- Then process the right subtree.
- Finally evaluate the current node.

At each node:

- If either subtree is already unbalanced, immediately propagate failure upward.
- Otherwise, compute the current node's height.
- If the height difference exceeds `1`, mark the subtree as unbalanced.

A common trick is to return a special value, such as `-1`, to indicate that a subtree is unbalanced. This lets us terminate unnecessary computation early.

This works efficiently because every node is visited exactly once, and each node performs constant work.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | `O(n²)` | `O(h)` | Repeatedly recalculates subtree heights |
| Optimal | `O(n)` | `O(h)` | Computes height and balance in one DFS traversal |

Here, `n` is the number of nodes and `h` is the height of the tree.

## Algorithm Walkthrough

1. Create a recursive helper function that computes the height of a subtree.
2. If the current node is `None`, return height `0`. This serves as the base case because an empty subtree has height zero.
3. Recursively compute the height of the left subtree.
4. If the left subtree returned `-1`, immediately return `-1`. This means the subtree is already unbalanced, so there is no reason to continue further work.
5. Recursively compute the height of the right subtree.
6. If the right subtree returned `-1`, immediately return `-1` for the same reason.
7. Compare the heights of the left and right subtree. If their absolute difference exceeds `1`, return `-1` to indicate imbalance.
8. Otherwise, return the height of the current subtree using:

`1 + max(leftHeight, rightHeight)`

We add one because the current node contributes one level to the subtree height.
9. After the recursion completes, check whether the helper function returned `-1`. If not, the tree is balanced.

The reason this approach is effective is that each subtree's height is calculated exactly once, and balance checking happens naturally during the height computation.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

class Solution:
    def isBalanced(self, root: Optional[TreeNode]) -> bool:
        def get_height(node: Optional[TreeNode]) -> int:
            if not node:
                return 0

            left_height = get_height(node.left)
            if left_height == -1:
                return -1

            right_height = get_height(node.right)
            if right_height == -1:
                return -1

            if abs(left_height - right_height) > 1:
                return -1

            return 1 + max(left_height, right_height)

        return get_height(root) != -1
```

The implementation follows the bottom-up depth-first search strategy.

The `get_height` helper function serves two purposes simultaneously. It computes subtree height while also detecting imbalance. Returning `-1` acts as a signal that the subtree is invalid.

The base case handles `None` nodes by returning `0`, since empty trees contribute no height.

The recursive calls process the left and right subtrees before evaluating the current node. This postorder traversal is essential because the balance condition depends on subtree heights being known first.

If either subtree returns `-1`, the function exits immediately and propagates the failure upward. This avoids unnecessary work.

Finally, if the current node is balanced, the function computes and returns its height. The main function simply checks whether the root evaluation produced `-1`.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func isBalanced(root *TreeNode) bool {
    var getHeight func(node *TreeNode) int

    getHeight = func(node *TreeNode) int {
        if node == nil {
            return 0
        }

        leftHeight := getHeight(node.Left)
        if leftHeight == -1 {
            return -1
        }

        rightHeight := getHeight(node.Right)
        if rightHeight == -1 {
            return -1
        }

        difference := leftHeight - rightHeight
        if difference < 0 {
            difference = -difference
        }

        if difference > 1 {
            return -1
        }

        if leftHeight > rightHeight {
            return leftHeight + 1
        }

        return rightHeight + 1
    }

    return getHeight(root) != -1
}
```

The Go implementation closely mirrors the Python solution. Since Go does not have nested named functions in the same way as Python, we declare a function variable using `var getHeight func(...)`.

Go uses `nil` instead of `None` for missing child nodes. We also manually compute the absolute difference because Go does not provide an integer `abs()` function in the standard library without importing additional packages.

Integer overflow is not a concern because the maximum height of the tree is bounded by the number of nodes, which is at most `5000`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | `O(n)` | Every node is visited exactly once |
| Space | `O(h)` | Recursive call stack depends on tree height |

The time complexity is `O(n)` because each node is processed exactly one time during the depth-first traversal. No subtree height is recomputed.

The space complexity is `O(h)`, where `h` is the height of the tree. This comes from the recursion stack. In a balanced tree, the height is approximately `O(log n)`. In the worst case, such as a skewed tree, the height becomes `O(n)`.

## Edge Cases

### Empty Tree

An empty tree can easily be overlooked because there are no nodes to process. Some implementations mistakenly treat it as invalid input or return `false`.

This implementation handles it correctly because the helper function immediately returns `0` when `root` is `None` or `nil`, meaning the tree is balanced.

### Single Node Tree

A tree containing only one node has no children, so both subtree heights are `0`.

This case is important because incorrect height calculations sometimes misclassify leaf nodes. Here, the algorithm computes:

```
abs(0 - 0) = 0
```

Since the difference is within the limit, the node is balanced.

### Deeply Skewed Tree

Consider a tree where every node only has a left child:

```
1
|
2
|
3
|
4
```

This is a common stress case because it exposes inefficiencies in naive implementations and quickly violates the balance rule.

The algorithm detects imbalance early using the `-1` sentinel value. Once an unbalanced subtree is found, the result propagates upward immediately without unnecessary recalculation.

### Balanced Root but Unbalanced Subtree

A subtle bug occurs when the root itself appears balanced while a deeper subtree is not.

For example:

```
        1
       / \
      2   2
     /
    3
   /
  4
```

The root might initially seem balanced, but node `2` violates the height condition.

Because the algorithm recursively validates every subtree before computing the parent height, this case is handled correctly and returns `false`.
