# LeetCode 162 - Find Peak Element

## Problem Understanding

This problem asks us to find the index of any "peak" element in an array. A peak element is defined as an element that is strictly greater than its immediate neighbors.

For an element `nums[i]` to be considered a peak:

- `nums[i] > nums[i - 1]`
- `nums[i] > nums[i + 1]`

The problem also defines imaginary boundary values:

- `nums[-1] = -∞`
- `nums[n] = -∞`

This means the first or last element can also be a peak if it is greater than its only real neighbor.

The input is a 0-indexed integer array `nums`. We must return the index of any peak element. If multiple peaks exist, returning any valid peak index is acceptable.

The key requirement is that the algorithm must run in `O(log n)` time, which strongly suggests that a binary search based solution is expected. A simple linear scan would work logically, but it would not satisfy the required complexity.

The constraints also provide important guarantees:

- `1 <= nums.length <= 1000`
- Adjacent elements are never equal:

```
nums[i] != nums[i + 1]
```

This guarantee is extremely important because it ensures the array always has a strictly increasing or decreasing relationship between neighboring elements. That property enables binary search reasoning.

Some important edge cases include:

- Arrays with only one element
- Arrays that are strictly increasing
- Arrays that are strictly decreasing
- Peaks located at the boundaries
- Arrays containing multiple valid peaks

The problem guarantees at least one peak always exists because the virtual `-∞` boundaries force a rise and eventual fall somewhere in the array.

## Approaches

### Brute Force Approach

The simplest solution is to scan the array linearly and check every element.

For each index `i`, we compare the current value against its left and right neighbors. Since the boundaries are treated as `-∞`, the first and last elements require special handling.

If we find an element greater than both neighbors, we immediately return its index.

This approach is correct because it explicitly verifies the peak condition for every element. Since at least one peak must exist, we are guaranteed to eventually find one.

However, this approach requires scanning the array sequentially, resulting in `O(n)` time complexity. The problem explicitly requires `O(log n)` time, so we need a more efficient strategy.

### Optimal Binary Search Approach

The key insight comes from observing the shape of the array.

At any index `mid`, we compare:

```
nums[mid] and nums[mid + 1]
```

There are two possible situations:

1. `nums[mid] < nums[mid + 1]`

This means the array is ascending at `mid`. Since values are increasing, there must be a peak somewhere on the right side.
2. `nums[mid] > nums[mid + 1]`

This means the array is descending at `mid`. A peak must exist on the left side, including `mid` itself.

This observation allows us to eliminate half of the search space each iteration, exactly like binary search.

The reason this works is subtle but important:

- If we are climbing upward, eventually we must either:

- reach a peak, or
- reach the end, which itself becomes a peak because of the virtual `-∞`
- If we are descending, then a peak already exists on the left side.

By repeatedly narrowing the search interval, we eventually converge on a peak index.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Scan every element and check neighbors |
| Optimal | O(log n) | O(1) | Binary search using slope direction |

## Algorithm Walkthrough

1. Initialize two pointers:

- `left = 0`
- `right = len(nums) - 1`

These pointers define the current search interval.
2. Continue searching while `left < right`.

This condition ensures we keep narrowing the range until only one candidate remains.
3. Compute the middle index:

```
mid = (left + right) // 2
```
4. Compare `nums[mid]` and `nums[mid + 1]`.

This comparison tells us whether we are currently moving upward or downward in the array.
5. If `nums[mid] < nums[mid + 1]`:

- The slope is increasing.
- A peak must exist somewhere to the right.
- Move the search space rightward:

```
left = mid + 1
```
6. Otherwise:

- The slope is decreasing.
- A peak exists at `mid` or somewhere to the left.
- Keep `mid` in the search space:

```
right = mid
```
7. Eventually, `left == right`.

At this point only one element remains in the search interval, and that element must be a peak.
8. Return `left` or `right`.

## Python Solution

```
from typing import List

class Solution:
    def findPeakElement(self, nums: List[int]) -> int:
        left = 0
        right = len(nums) - 1

        while left < right:
            mid = (left + right) // 2

            if nums[mid] < nums[mid + 1]:
                left = mid + 1
            else:
                right = mid

        return left
```

The implementation begins by initializing the binary search boundaries. The search interval always contains at least one valid peak.

Inside the loop, we compute the middle index and compare the current element with the next element. This comparison determines the slope direction.

If the sequence is increasing at `mid`, the algorithm discards the left half because a peak must exist to the right. Otherwise, the algorithm keeps the left side, including `mid`, because the peak could already be there.

The search space continuously shrinks until both pointers converge to the same index. That index is returned as the peak.

An important detail is that we compare `nums[mid]` with `nums[mid + 1]`. Because the loop condition ensures `left < right`, `mid + 1` is always a valid index.

## Go Solution

```
func findPeakElement(nums []int) int {
    left := 0
    right := len(nums) - 1

    for left < right {
        mid := (left + right) / 2

        if nums[mid] < nums[mid+1] {
            left = mid + 1
        } else {
            right = mid
        }
    }

    return left
}
```

The Go implementation follows exactly the same binary search logic as the Python solution.

Go slices already provide dynamic indexing behavior similar to Python lists, so no special handling is required. Integer division in Go automatically truncates toward zero, which correctly computes the midpoint.

There are no integer overflow concerns here because the array size is at most 1000, but in larger systems, some developers prefer:

```
mid := left + (right-left)/2
```

to avoid overflow.

The algorithm uses only constant extra memory and modifies no input data.

## Worked Examples

### Example 1

Input:

```
nums = [1,2,3,1]
```

We trace the binary search step by step.

| left | right | mid | nums[mid] | nums[mid+1] | Action |
| --- | --- | --- | --- | --- | --- |
| 0 | 3 | 1 | 2 | 3 | Increasing slope, move right |
| 2 | 3 | 2 | 3 | 1 | Decreasing slope, keep left half |
| 2 | 2 | - | - | - | Stop |

Final answer:

```
2
```

The peak element is `3`.

### Example 2

Input:

```
nums = [1,2,1,3,5,6,4]
```

| left | right | mid | nums[mid] | nums[mid+1] | Action |
| --- | --- | --- | --- | --- | --- |
| 0 | 6 | 3 | 3 | 5 | Increasing slope, move right |
| 4 | 6 | 5 | 6 | 4 | Decreasing slope, keep left half |
| 4 | 5 | 4 | 5 | 6 | Increasing slope, move right |
| 5 | 5 | - | - | - | Stop |

Final answer:

```
5
```

The peak element is `6`.

Notice that index `1` would also have been a valid answer because `2` is another peak. The problem allows returning any valid peak.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(log n) | Binary search halves the search space each iteration |
| Space | O(1) | Only a few variables are used |

The algorithm achieves logarithmic time complexity because each iteration removes half of the remaining elements from consideration. This is the defining characteristic of binary search.

The space complexity remains constant because no additional data structures are allocated. Only pointer variables and the midpoint variable are maintained throughout execution.

## Edge Cases

### Single Element Array

Consider:

```
nums = [7]
```

Since the virtual neighbors are both `-∞`, the single element is automatically a peak.

The algorithm handles this naturally because:

```
left = 0
right = 0
```

The loop never executes, and the function immediately returns index `0`.

### Strictly Increasing Array

Consider:

```
nums = [1,2,3,4,5]
```

The peak is the final element because the right boundary is treated as `-∞`.

The binary search repeatedly moves right because every comparison sees an increasing slope. Eventually, the search converges on the last index.

### Strictly Decreasing Array

Consider:

```
nums = [5,4,3,2,1]
```

The peak is the first element because the left boundary is treated as `-∞`.

The binary search repeatedly keeps the left half because every comparison sees a decreasing slope. Eventually, the search converges on index `0`.

### Multiple Peaks

Consider:

```
nums = [1,3,2,5,4]
```

Both `3` and `5` are valid peaks.

The problem allows returning any peak, so the algorithm does not need to search for all peaks or the maximum peak. Binary search naturally converges toward one valid solution depending on the search path.

### Boundary Peaks

Consider:

```
nums = [10,5,3]
```

The first element is a peak because:

```
10 > -∞
10 > 5
```

Similarly:

```
nums = [1,2,8]
```

The last element is a peak because:

```
8 > 2
8 > -∞
```

The imaginary boundary definition ensures these cases work cleanly without requiring special conditional logic in the implementation.
