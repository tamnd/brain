## Solution

## Problem Understanding

The problem asks us to perform a modified level order traversal on a binary tree. In a normal level order traversal, also called Breadth-First Search (BFS), we visit nodes level by level from left to right. In this problem, the traversal direction alternates at every level.

At the first level, we traverse from left to right.

At the second level, we traverse from right to left.

At the third level, we go back to left to right.

This alternating pattern continues until all levels of the tree have been processed.

The input is the root node of a binary tree. Each node contains an integer value and pointers to its left and right child nodes. The output should be a list of lists, where each inner list contains the values of one tree level in the required zigzag order.

For example, given the tree:

```
        3
       / \
      9  20
         / \
        15  7
```

The traversal proceeds like this:

- Level 0, left to right → `[3]`
- Level 1, right to left → `[20, 9]`
- Level 2, left to right → `[15, 7]`

So the final answer becomes:

```
[[3], [20, 9], [15, 7]]
```

The constraints tell us that the number of nodes can be as large as 2000. This is small enough that an O(n) traversal is completely acceptable, but large enough that repeatedly reversing or rebuilding large structures inefficiently could become unnecessary overhead.

Several edge cases are important to recognize immediately. The tree may be empty, meaning the root is `None` or `nil`, in which case we should return an empty list. The tree may contain only one node, where the result is simply a single-element nested list. The tree may also be highly unbalanced, resembling a linked list, so the solution should not rely on the tree being complete or balanced.

## Approaches

A brute-force solution would first perform a standard level order traversal and store every level in left-to-right order. After collecting all levels, we could iterate through the result and reverse every odd-indexed level.

This works because BFS naturally groups nodes level by level. Reversing every second level transforms the standard traversal into the required zigzag ordering.

However, this approach performs additional reversal operations on many levels. Although reversing a list is linear in the size of that level, the total complexity still remains O(n). The real drawback is that it performs unnecessary extra work and requires additional processing after traversal.

The key insight for the optimal solution is that we can build each level directly in the correct order while performing BFS. Instead of collecting values and reversing later, we track the current traversal direction with a boolean flag.

When traversing left to right, we append values normally.

When traversing right to left, we insert values at the beginning of the current level structure.

A queue is the ideal data structure for BFS because it naturally processes nodes level by level in the order they are discovered.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Perform normal BFS, then reverse every alternate level |
| Optimal | O(n) | O(n) | Build each level directly in zigzag order during BFS |

## Algorithm Walkthrough

1. Start by handling the empty tree case. If the root is `None`, return an empty list immediately because there are no levels to traverse.
2. Create a queue and place the root node inside it. The queue is used for Breadth-First Search because it processes nodes level by level in first-in, first-out order.
3. Create a result list that will store the final zigzag traversal.
4. Maintain a boolean variable such as `left_to_right`. Initially, this should be `True` because the first level is traversed from left to right.
5. While the queue is not empty, process one complete level at a time. The current queue length tells us exactly how many nodes belong to the current level.
6. Create an empty structure to store values for the current level.
7. Remove nodes from the queue one by one for the current level:

1. Extract the node from the front of the queue.
2. If the direction is left to right, append the value to the end of the current level list.
3. Otherwise, insert the value at the beginning of the current level list.
4. Add the node's left child to the queue if it exists.
5. Add the node's right child to the queue if it exists.
8. After processing all nodes in the current level, append the completed level list to the result.
9. Flip the traversal direction by changing `left_to_right` to its opposite value.
10. Continue until the queue becomes empty.
11. Return the final result list.

## Python Solution

```
from collections import deque
from typing import List, Optional

# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

class Solution:
    def zigzagLevelOrder(self, root: Optional[TreeNode]) -> List[List[int]]:
        if not root:
            return []

        result = []
        queue = deque([root])
        left_to_right = True

        while queue:
            level_size = len(queue)
            current_level = deque()

            for _ in range(level_size):
                node = queue.popleft()

                if left_to_right:
                    current_level.append(node.val)
                else:
                    current_level.appendleft(node.val)

                if node.left:
                    queue.append(node.left)

                if node.right:
                    queue.append(node.right)

            result.append(list(current_level))
            left_to_right = not left_to_right

        return result
```

The implementation begins with an empty tree check. Returning immediately avoids unnecessary processing and correctly handles the edge case where the tree has no nodes.

The queue is implemented using `collections.deque` because it supports efficient insertion and removal from both ends in O(1) time. This makes it ideal for BFS traversal.

The `left_to_right` flag controls the ordering for each level. When the flag is `True`, values are appended normally. When it is `False`, values are inserted at the front using `appendleft`.

The loop processes one full level at a time by recording the queue size before iteration begins. This is important because children from the next level are added during traversal, and we only want to process nodes belonging to the current level.

After completing a level, we convert the deque into a list and append it to the final result. Then we flip the traversal direction for the next level.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func zigzagLevelOrder(root *TreeNode) [][]int {
    if root == nil {
        return [][]int{}
    }

    result := [][]int{}
    queue := []*TreeNode{root}
    leftToRight := true

    for len(queue) > 0 {
        levelSize := len(queue)
        currentLevel := make([]int, levelSize)

        for i := 0; i < levelSize; i++ {
            node := queue[0]
            queue = queue[1:]

            index := i
            if !leftToRight {
                index = levelSize - 1 - i
            }

            currentLevel[index] = node.Val

            if node.Left != nil {
                queue = append(queue, node.Left)
            }

            if node.Right != nil {
                queue = append(queue, node.Right)
            }
        }

        result = append(result, currentLevel)
        leftToRight = !leftToRight
    }

    return result
}
```

The Go implementation follows the same BFS logic as the Python version, but there are a few language-specific differences.

Go does not provide a built-in deque structure like Python's `collections.deque`, so the queue is implemented using slices. Nodes are removed from the front using `queue = queue[1:]`.

Instead of using a deque for the current level, the Go solution preallocates a slice of the correct size and computes the insertion index directly. When traversing left to right, values are placed normally. When traversing right to left, values are placed using the mirrored index:

```
index = levelSize - 1 - i
```

This avoids expensive insertions at the front of a slice.

Go uses `nil` instead of `None`, and pointers are explicitly checked before appending child nodes to the queue.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every node is visited exactly once |
| Space | O(n) | Queue and result storage can contain up to all nodes |

The BFS traversal processes each node exactly one time, so the total time complexity is linear in the number of nodes.

The space complexity is also O(n). In the worst case, the queue may contain an entire tree level at once. For a balanced tree, this can approach roughly half the nodes in the tree. The result structure itself also stores all node values.

## Edge Cases

An empty tree is the most important edge case. If the root is `None` or `nil`, attempting BFS traversal would cause errors because there are no nodes to process. The implementation handles this immediately by returning an empty list before initializing the queue.

A single-node tree is another important case because there is only one level. Some incorrect implementations may still try to alternate directions unnecessarily or mishandle queue processing. In this solution, the BFS loop naturally processes the single node and returns `[[value]]`.

Highly unbalanced trees can also expose bugs. Consider a tree shaped like a linked list:

```
1
 \
  2
   \
    3
```

Each level contains only one node, so reversing directions should not change the output. The implementation correctly processes one node per level and alternates directions harmlessly.

Another subtle edge case is levels with many nodes. Some implementations repeatedly reverse large arrays after traversal, which introduces unnecessary overhead. This solution avoids that by constructing each level directly in the correct order during traversal.
