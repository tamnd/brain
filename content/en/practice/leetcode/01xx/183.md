## Solution

## Problem Understanding

This problem provides two database tables, `Customers` and `Orders`. The `Customers` table stores information about customers, specifically their unique `id` and their `name`. The `Orders` table stores information about customer orders, where each order has its own `id` and a `customerId` that references the customer who placed the order.

The task is to find all customers who have never placed an order. In other words, we need to identify every customer whose `id` does not appear anywhere in the `Orders.customerId` column.

The output should contain a single column named `Customers`, which lists the names of customers with no associated orders. The problem states that the result can be returned in any order, so there is no need to sort the output unless explicitly required.

The input represents a classic relational database setup with a one to many relationship:

- One customer can place many orders.
- Each order belongs to exactly one customer.

The key observation is that customers who never ordered anything will not have any matching record in the `Orders` table.

Since this is labeled as an Easy database problem, the expected solution is straightforward SQL using joins or subqueries. The data size constraints are not explicitly stated, but the problem is designed to teach efficient relational filtering rather than brute-force row scanning in application code.

Several edge cases are important to consider:

- A customer may appear in `Customers` but never appear in `Orders`.
- Multiple orders may belong to the same customer.
- The `Orders` table may be completely empty, meaning every customer should be returned.
- Every customer may have at least one order, meaning the result should be empty.
- Customer names are not guaranteed to be unique, so the solution must rely on customer IDs rather than names.

## Approaches

### Brute Force Approach

A brute-force approach would conceptually examine every customer and manually search through all orders to determine whether that customer has placed an order.

For each customer:

1. Scan the entire `Orders` table.
2. Check whether any `customerId` matches the current customer's `id`.
3. If no match is found, include the customer in the result.

This approach works because it explicitly verifies whether an order exists for every customer. However, it is inefficient because for every customer we potentially scan the entire orders table.

If there are `n` customers and `m` orders, the total work becomes `O(n × m)`.

Although acceptable for very small datasets, this approach does not scale well.

### Optimal Approach

The key insight is that relational databases are designed specifically for this type of lookup operation.

Instead of repeatedly scanning the `Orders` table, we can use a `LEFT JOIN` between `Customers` and `Orders`.

A `LEFT JOIN` keeps all rows from the `Customers` table. If a matching order exists, the joined row contains order information. If no matching order exists, the joined columns become `NULL`.

This means customers with no orders can be identified by checking where the joined order columns are `NULL`.

Another equivalent approach is using `NOT IN` or `NOT EXISTS`, but `LEFT JOIN ... IS NULL` is one of the clearest and most commonly used solutions.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × m) | O(1) | Scans all orders for every customer |
| Optimal | O(n + m) | O(1) | Uses SQL join operations efficiently |

## Algorithm Walkthrough

1. Start with the `Customers` table because every customer must be considered, even those without orders.
2. Perform a `LEFT JOIN` between `Customers` and `Orders` using the condition:

```
Customers.id = Orders.customerId
```

This ensures that every customer remains in the result set.
3. For customers who have matching orders, the joined row will contain values from the `Orders` table.
4. For customers who never placed an order, the columns coming from `Orders` will contain `NULL`.
5. Filter the joined result using:

```
WHERE Orders.customerId IS NULL
```

This keeps only customers without matching orders.
6. Return the customer names using the required column alias:

```
Customers
```

This approach leverages the database engine's optimized join operations instead of manually searching row by row.

## Python Solution

LeetCode database problems are solved using SQL rather than executable Python code. However, following the requested structure, the solution can be represented as a string containing the SQL query.

```
class Solution:
    def customersWhoNeverOrder(self) -> str:
        return """
        SELECT
            Customers.name AS Customers
        FROM Customers
        LEFT JOIN Orders
            ON Customers.id = Orders.customerId
        WHERE Orders.customerId IS NULL;
        """
```

The implementation performs a `LEFT JOIN` between the `Customers` and `Orders` tables. The join condition matches customer IDs with order customer IDs.

After the join, customers who do not have matching orders will produce rows where the order-related columns are `NULL`. The `WHERE Orders.customerId IS NULL` condition filters the result to keep only those customers.

Finally, the query returns the customer names using the alias `Customers`, which matches the required output format.

## Edge Cases

### Orders Table Is Empty

If the `Orders` table contains no rows, every customer has never placed an order.

Example:

`Customers`

| id | name |
| --- | --- |
| 1 | Alice |
| 2 | Bob |

`Orders`

| id | customerId |
| --- | --- |

After the `LEFT JOIN`, all order columns become `NULL`, so every customer is returned correctly.

### Every Customer Has At Least One Order

If all customers appear in the `Orders` table, no rows will satisfy the `IS NULL` condition.

Example:

| Customers.id | name |
| --- | --- |
| 1 | Alice |

| customerId |
| --- |
| 1 |

The final result becomes an empty table, which is correct.

### Multiple Orders For One Customer

A customer may place several orders. A naive implementation could accidentally return duplicate rows if not handled carefully.

Example:

| customerId |
| --- |
| 1 |
| 1 |
| 1 |

In this problem, duplicate rows are not an issue because customers with orders are filtered out entirely. Only customers without matching orders survive the `IS NULL` condition.

### Duplicate Customer Names

Two customers may share the same name but have different IDs.

Example:

| id | name |
| --- | --- |
| 1 | Alex |
| 2 | Alex |

The solution correctly relies on customer IDs for matching rather than names, preventing incorrect filtering behavior.
