# LeetCode 148 - Sort List

## Problem Understanding

The problem gives us the head of a singly linked list and asks us to sort the list in ascending order. The returned result must also be a linked list, not an array or another data structure.

A linked list is fundamentally different from an array because nodes are connected through pointers rather than stored contiguously in memory. This means we cannot directly access the middle element in constant time, and operations like random indexing are inefficient. On the other hand, splitting and reconnecting lists can be done efficiently with pointer manipulation.

The input consists of the head node of a singly linked list. Each node contains an integer value and a pointer to the next node. The output should be the head of a new ordering of the same nodes such that their values appear in non decreasing order.

The constraints are important:

- The list can contain up to `5 * 10^4` nodes.
- Node values range from `-10^5` to `10^5`.

Because the list may contain tens of thousands of nodes, an `O(n^2)` sorting algorithm such as Bubble Sort or Insertion Sort will likely be too slow in the worst case. The follow up specifically asks for `O(n log n)` time and constant auxiliary memory, which strongly suggests using Merge Sort.

Several edge cases are important:

- The list may be empty.
- The list may contain only one node.
- The list may already be sorted.
- The list may contain duplicate values.
- The list may contain negative numbers.
- The list may be sorted in reverse order.

A correct implementation must handle all of these cases safely without losing nodes or creating cycles in the linked list.

## Approaches

### Brute Force Approach

A straightforward solution is to copy all node values into an array, sort the array, and then rebuild the linked list values in sorted order.

The process works like this:

1. Traverse the linked list and store all values in an array.
2. Sort the array using the language's built in sorting function.
3. Traverse the linked list again and overwrite each node's value with the sorted values.

This produces the correct answer because sorting the array gives the values in ascending order, and rewriting the nodes preserves the linked list structure.

However, this approach uses extra memory proportional to the number of nodes because we store all values in an additional array. While the time complexity is acceptable at `O(n log n)`, the space complexity is `O(n)`.

The follow up asks for constant auxiliary space, so we need a more optimal approach.

### Optimal Approach

The best solution is Merge Sort on the linked list itself.

Merge Sort is especially well suited for linked lists because:

- Linked lists can be split efficiently using slow and fast pointers.
- Two sorted linked lists can be merged efficiently without additional arrays.
- Merge Sort naturally achieves `O(n log n)` time complexity.

The key insight is that we can recursively divide the linked list into halves until each sublist contains one node. A single node is already sorted. Then we merge the sorted halves back together in sorted order.

Unlike arrays, linked lists do not require expensive shifting operations during merges. We simply reconnect pointers.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) | O(n) | Copy values into array, sort array, rewrite list |
| Optimal | O(n log n) | O(log n) | Merge Sort using linked list splitting and merging |

## Algorithm Walkthrough

### Merge Sort on Linked List

1. Check the base case.

If the list is empty or contains only one node, it is already sorted. Return the head immediately.
2. Find the middle of the linked list.

Use two pointers:

- `slow` moves one step at a time.
- `fast` moves two steps at a time.

When `fast` reaches the end, `slow` will be near the middle.
3. Split the list into two halves.

Keep track of the node before `slow` using a `prev` pointer. Once the middle is found, disconnect the list:

- First half starts at `head`
- Second half starts at `slow`

Set `prev.next = None` to break the list into two independent lists.
4. Recursively sort both halves.

Apply Merge Sort recursively to:

- Left half
- Right half
5. Merge the two sorted halves.

Create a dummy node to simplify merging logic.

Compare nodes from both lists:

- Attach the smaller node to the merged list
- Move the corresponding pointer forward

After one list is exhausted, append the remaining nodes from the other list.
6. Return the merged result.

The merged list starting from `dummy.next` is the sorted linked list.

## Python Solution

```
# Definition for singly-linked list.
# class ListNode:
#     def __init__(self, val=0, next=None):
#         self.val = val
#         self.next = next

from typing import Optional

class Solution:
    def sortList(self, head: Optional[ListNode]) -> Optional[ListNode]:
        # Base case
        if not head or not head.next:
            return head

        # Find middle using slow and fast pointers
        prev = None
        slow = head
        fast = head

        while fast and fast.next:
            prev = slow
            slow = slow.next
            fast = fast.next.next

        # Split the list into two halves
        prev.next = None

        # Recursively sort both halves
        left_sorted = self.sortList(head)
        right_sorted = self.sortList(slow)

        # Merge sorted halves
        return self.merge(left_sorted, right_sorted)

    def merge(
        self,
        left: Optional[ListNode],
        right: Optional[ListNode]
    ) -> Optional[ListNode]:

        dummy = ListNode(0)
        current = dummy

        while left and right:
            if left.val <= right.val:
                current.next = left
                left = left.next
            else:
                current.next = right
                right = right.next

            current = current.next

        # Attach remaining nodes
        if left:
            current.next = left
        else:
            current.next = right

        return dummy.next
```

The implementation begins with the base case. If the list is empty or contains only one node, the function returns immediately because such a list is already sorted.

The middle of the list is found using the classic slow and fast pointer technique. The `fast` pointer advances twice as quickly as the `slow` pointer, allowing `slow` to stop near the midpoint when traversal ends.

The `prev` pointer is essential because we must physically split the linked list into two independent parts. Without disconnecting the first half from the second half, recursive calls would continue traversing the entire list and never terminate correctly.

After splitting, both halves are recursively sorted. This repeatedly divides the problem until each recursive call operates on a single node.

The `merge` helper function combines two sorted linked lists into one sorted list. A dummy node simplifies edge cases because we never need separate logic for the head insertion.

The merging process repeatedly chooses the smaller node from the two lists and advances the corresponding pointer. Once one list becomes empty, the remainder of the other list is appended directly.

Finally, `dummy.next` is returned because the dummy node itself is only a placeholder.

## Go Solution

```
/**
 * Definition for singly-linked list.
 * type ListNode struct {
 *     Val int
 *     Next *ListNode
 * }
 */

func sortList(head *ListNode) *ListNode {
    // Base case
    if head == nil || head.Next == nil {
        return head
    }

    // Find middle
    var prev *ListNode
    slow := head
    fast := head

    for fast != nil && fast.Next != nil {
        prev = slow
        slow = slow.Next
        fast = fast.Next.Next
    }

    // Split the list
    prev.Next = nil

    // Recursively sort both halves
    left := sortList(head)
    right := sortList(slow)

    // Merge sorted halves
    return merge(left, right)
}

func merge(left *ListNode, right *ListNode) *ListNode {
    dummy := &ListNode{}
    current := dummy

    for left != nil && right != nil {
        if left.Val <= right.Val {
            current.Next = left
            left = left.Next
        } else {
            current.Next = right
            right = right.Next
        }

        current = current.Next
    }

    if left != nil {
        current.Next = left
    } else {
        current.Next = right
    }

    return dummy.Next
}
```

The Go implementation closely mirrors the Python solution. The main difference is explicit pointer handling using `nil` rather than Python's `None`.

Go does not support default constructors in the same way as Python, so the dummy node is created with `&ListNode{}`.

Recursive behavior and merging logic remain identical. Pointer manipulation is particularly important in Go because linked list operations directly mutate node connections.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | Each level processes all nodes, and there are log n levels |
| Space | O(log n) | Recursive call stack depth from Merge Sort |

At every recursion level, all nodes are visited during splitting and merging. Since the list is repeatedly divided in half, there are `log n` levels of recursion. Multiplying these together gives `O(n log n)` time complexity.

The merge operation itself does not allocate additional arrays or lists. However, recursive function calls consume stack space. Because the recursion depth is logarithmic, the auxiliary space complexity is `O(log n)`.

## Edge Cases

### Empty List

An empty list is represented by `head = None` in Python or `head == nil` in Go. This case is important because attempting to access `head.next` without checking for null would cause runtime errors.

The implementation handles this immediately in the base case:

```
if not head or not head.next:
    return head
```

This ensures safe behavior before any pointer traversal occurs.

### Single Node List

A list containing only one node is already sorted. Recursive splitting must stop here, otherwise infinite recursion would occur because the list cannot be divided further.

The same base case safely handles this scenario by returning the node directly.

### Duplicate Values

Lists may contain repeated values such as:

```
4 -> 2 -> 2 -> 1
```

A sorting algorithm must preserve all nodes and correctly order duplicates.

The merge step uses:

```
if left.val <= right.val:
```

Using `<=` instead of `<` ensures stable merging behavior and prevents nodes from being skipped.

### Reverse Sorted List

A worst case ordering like:

```
5 -> 4 -> 3 -> 2 -> 1
```

can expose inefficiencies in poor sorting algorithms.

Merge Sort handles this efficiently because its runtime does not depend on initial ordering. The algorithm still performs in `O(n log n)` time.

### Negative Numbers

The list may contain negative values, such as:

```
-3 -> -1 -> -7 -> 2
```

Since comparisons are performed directly on integer values, negative numbers naturally integrate into the merge process without any special handling.
