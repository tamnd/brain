# LeetCode 155 - Min Stack

## Problem Understanding

This problem asks us to design a stack data structure that supports four operations efficiently:

- `push(val)` inserts a value onto the stack
- `pop()` removes the most recently added value
- `top()` returns the most recently added value
- `getMin()` returns the smallest value currently present in the stack

The important requirement is that **every operation must run in constant time, O(1)**.

In a normal stack, `push`, `pop`, and `top` are naturally `O(1)` because they only interact with the top element. However, finding the minimum value is more difficult. If we simply scan the stack each time `getMin()` is called, that would take `O(n)` time, which violates the problem constraint.

The input in LeetCode is represented as a sequence of method calls on the `MinStack` object. For example:

```
["MinStack","push","push","push","getMin","pop","top","getMin"]
[[],[-2],[0],[-3],[],[],[],[]]
```

This means:

1. Create a `MinStack`
2. Push `-2`
3. Push `0`
4. Push `-3`
5. Retrieve minimum
6. Pop top element
7. Retrieve top
8. Retrieve minimum again

The expected output corresponds to the result of each operation. Constructor and mutation methods (`push`, `pop`) return `null`, while query methods (`top`, `getMin`) return integers.

The constraints tell us several useful things:

- Values can range from `-2³¹` to `2³¹ - 1`, so negative values and large integers must be handled correctly.
- `pop`, `top`, and `getMin` are always called on a non-empty stack, meaning we do not need defensive checks for empty stack access.
- Up to `3 × 10⁴` operations may occur, which is large enough that inefficient repeated scans would become problematic.

Several edge cases are important to think about up front. The stack may contain duplicate minimum values, which means removing one minimum should not necessarily change the current minimum. The stack may also contain negative numbers, meaning comparisons must work correctly for values below zero. Another important case occurs when the minimum element itself gets popped, since the data structure must immediately know the next smallest element without recomputing it.

## Approaches

### Brute Force Approach

A straightforward implementation would use a normal stack and compute the minimum whenever `getMin()` is called.

In this design:

- `push(val)` appends to the stack
- `pop()` removes the last element
- `top()` returns the last element
- `getMin()` scans the entire stack and returns the smallest value

This approach is correct because scanning the stack guarantees that the minimum element will be found. However, the major issue is efficiency. Every `getMin()` call would require traversing all elements in the stack, taking `O(n)` time.

Since the problem explicitly requires constant time operations, this approach does not satisfy the constraints.

### Optimal Approach, Two Stacks

The key insight is that we should **store minimum information as we build the stack**, instead of recalculating it later.

We maintain two stacks:

1. **Main stack** stores all values normally.
2. **Minimum stack** tracks the minimum value seen so far at each point.

Whenever we push a value:

- Add it to the main stack.
- Compare it with the current minimum.
- Push the smaller value onto the minimum stack.

This means each position in the minimum stack represents the minimum element corresponding to the same position in the main stack.

For example:

```
Push sequence: [5, 3, 7, 2]

Main stack: [5, 3, 7, 2]
Min stack:  [5, 3, 3, 2]
```

At every depth, the minimum stack stores the minimum up to that point.

When popping:

- Remove the top from both stacks.

When retrieving the minimum:

- Return the top of the minimum stack.

Since stack access at the top is constant time, all operations become `O(1)`.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | `push/pop/top: O(1)`, `getMin: O(n)` | `O(n)` | Uses one stack and scans to find minimum |
| Optimal | `O(1)` for all operations | `O(n)` | Uses an additional stack to track minimum values |

## Algorithm Walkthrough

1. Initialize two stacks, one for storing actual values and one for tracking minimum values. The main stack behaves like a normal stack, while the minimum stack stores the smallest value seen so far at every position.
2. When `push(val)` is called, insert the value into the main stack. Then determine the new minimum. If the minimum stack is empty, the pushed value becomes the minimum. Otherwise, compare the new value with the current minimum and push the smaller one onto the minimum stack.
3. When `pop()` is called, remove the top element from both stacks. This ensures both stacks remain perfectly synchronized.
4. When `top()` is called, return the last element in the main stack. Since stack access at the end of an array or list is constant time, this operation is efficient.
5. When `getMin()` is called, return the last element of the minimum stack. Because this stack always stores the minimum at the current depth, the answer is immediately available without scanning.

The reason for using a second stack instead of recalculating values is to preserve constant-time access. Every push records enough information so future queries can be answered instantly.

## Python Solution

```
class MinStack:

    def __init__(self):
        self.stack: list[int] = []
        self.min_stack: list[int] = []

    def push(self, val: int) -> None:
        self.stack.append(val)

        if not self.min_stack:
            self.min_stack.append(val)
        else:
            current_min = self.min_stack[-1]
            self.min_stack.append(min(val, current_min))

    def pop(self) -> None:
        self.stack.pop()
        self.min_stack.pop()

    def top(self) -> int:
        return self.stack[-1]

    def getMin(self) -> int:
        return self.min_stack[-1]

# Your MinStack object will be instantiated and called as such:
# obj = MinStack()
# obj.push(val)
# obj.pop()
# param_3 = obj.top()
# param_4 = obj.getMin()
```

The implementation maintains two lists:

- `stack` stores the actual stack elements.
- `min_stack` stores the minimum value corresponding to each position.

During `push`, the value is appended to the main stack. The minimum stack receives either the pushed value itself, if it becomes the new minimum, or the previous minimum if the new value is larger.

The `pop` method removes elements from both stacks simultaneously. This synchronization is essential because the minimum stack mirrors the structure of the main stack.

The `top` method simply returns the last element of the main stack, while `getMin` returns the last element of the minimum stack. Since both operations only access the end of a list, they are constant time.

## Go Solution

```
type MinStack struct {
	stack    []int
	minStack []int
}

func Constructor() MinStack {
	return MinStack{
		stack:    []int{},
		minStack: []int{},
	}
}

func (this *MinStack) Push(val int) {
	this.stack = append(this.stack, val)

	if len(this.minStack) == 0 {
		this.minStack = append(this.minStack, val)
	} else {
		currentMin := this.minStack[len(this.minStack)-1]

		if val < currentMin {
			this.minStack = append(this.minStack, val)
		} else {
			this.minStack = append(this.minStack, currentMin)
		}
	}
}

func (this *MinStack) Pop() {
	this.stack = this.stack[:len(this.stack)-1]
	this.minStack = this.minStack[:len(this.minStack)-1]
}

func (this *MinStack) Top() int {
	return this.stack[len(this.stack)-1]
}

func (this *MinStack) GetMin() int {
	return this.minStack[len(this.minStack)-1]
}

/**
 * Your MinStack object will be instantiated and called as such:
 * obj := Constructor();
 * obj.Push(val);
 * obj.Pop();
 * param_3 := obj.Top();
 * param_4 := obj.GetMin();
 */
```

The Go implementation follows exactly the same logic as the Python version but uses slices instead of lists.

Since Go slices support efficient append and end access, they naturally model stacks. Removing the top element is done using slice truncation:

```
stack = stack[:len(stack)-1]
```

Unlike Python, Go does not have built in negative indexing, so the last element must be accessed using:

```
stack[len(stack)-1]
```

Integer overflow is not an issue here because the problem guarantees values fit within Go's `int` constraints for LeetCode environments.

## Worked Examples

### Example 1

#### Input

```
["MinStack","push","push","push","getMin","pop","top","getMin"]
[[],[-2],[0],[-3],[],[],[],[]]
```

We trace both stacks after every operation.

| Operation | Main Stack | Min Stack | Return Value |
| --- | --- | --- | --- |
| `MinStack()` | `[]` | `[]` | `null` |
| `push(-2)` | `[-2]` | `[-2]` | `null` |
| `push(0)` | `[-2, 0]` | `[-2, -2]` | `null` |
| `push(-3)` | `[-2, 0, -3]` | `[-2, -2, -3]` | `null` |
| `getMin()` | `[-2, 0, -3]` | `[-2, -2, -3]` | `-3` |
| `pop()` | `[-2, 0]` | `[-2, -2]` | `null` |
| `top()` | `[-2, 0]` | `[-2, -2]` | `0` |
| `getMin()` | `[-2, 0]` | `[-2, -2]` | `-2` |

Notice what happens during the push of `0`. Even though `0` is larger than `-2`, we still push `-2` into the minimum stack because it remains the minimum value at that depth.

When `-3` is pushed, it becomes the new minimum, so `-3` is added to the minimum stack.

After popping `-3`, the minimum automatically becomes `-2` again because the previous minimum is already stored beneath it in `min_stack`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | `O(1)` | Every operation only interacts with the top of one or two stacks |
| Space | `O(n)` | Both stacks together store information proportional to the number of elements |

The time complexity is constant for all operations because no traversal occurs. Every method performs only a fixed number of operations regardless of stack size.

The space complexity is linear because the auxiliary minimum stack stores one value for every element in the main stack. While this doubles the storage requirement, it preserves the required `O(1)` query performance.

## Edge Cases

### Duplicate Minimum Values

A subtle bug can occur if the minimum value appears multiple times.

Example:

```
push(2)
push(2)
pop()
getMin()
```

If only one copy of the minimum were tracked, popping one `2` could incorrectly remove the minimum. This implementation avoids that issue because every stack position stores its own minimum snapshot.

### Negative Numbers

The stack may contain negative values, such as:

```
push(-10)
push(-20)
push(-5)
```

A naive implementation might mishandle comparisons if assumptions about positivity are made. Here, the minimum is always determined using standard integer comparison, so negative values work naturally.

### Popping the Current Minimum

Consider:

```
push(5)
push(2)
push(8)
pop()
pop()
getMin()
```

When `2` is removed, the stack must immediately know that `5` is now the minimum. Since the previous minimum was stored underneath in `min_stack`, no recomputation is needed.

### Single Element Stack

A stack containing only one element can sometimes expose synchronization bugs.

Example:

```
push(10)
getMin()
pop()
```

Both stacks contain exactly one element, and both remove their top element during `pop`. This guarantees consistent behavior even at minimum size.
