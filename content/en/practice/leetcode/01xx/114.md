## Solution

## Problem Understanding

The problem gives us the root of a binary tree and asks us to transform the tree into a flattened structure that behaves like a singly linked list. The linked list must reuse the existing tree nodes, which means we are not supposed to create a new list structure. Instead, we modify the pointers inside the original tree.

The flattened structure must follow preorder traversal order. In preorder traversal, we visit nodes in this sequence:

1. Current node
2. Left subtree
3. Right subtree

After flattening, every node's `left` pointer must become `null`, and the `right` pointer must point to the next node in preorder sequence.

For example, consider this tree:

```
        1
       / \
      2   5
     / \   \
    3   4   6
```

The preorder traversal is:

```
1 -> 2 -> 3 -> 4 -> 5 -> 6
```

So the flattened tree should look like:

```
1
 \
  2
   \
    3
     \
      4
       \
        5
         \
          6
```

The important detail is that the transformation must happen in-place. We are modifying the existing tree rather than constructing a new data structure.

The constraints tell us that the tree can contain up to 2000 nodes. This is small enough that even recursive solutions are acceptable, but the follow-up question asks whether we can achieve `O(1)` extra space. That hint strongly suggests there is a pointer manipulation technique that avoids recursion stacks or auxiliary data structures.

Several edge cases are important:

- The tree may be empty, so `root` can be `null`.
- The tree may contain only one node.
- The tree may already be flattened.
- The tree may be completely left-skewed or right-skewed.
- Nodes can contain duplicate values, so we cannot rely on values being unique.

A naive implementation can easily lose references to subtrees while rearranging pointers, so careful pointer handling is critical.

## Approaches

### Brute-Force Approach

A straightforward solution is to first perform a preorder traversal and store all visited nodes in an array or list. After collecting the nodes in preorder order, we iterate through the list and reconnect the pointers so that:

- `left = None`
- `right = next node in preorder sequence`

This approach works because preorder traversal already produces the exact order required for the flattened linked list.

The algorithm is conceptually simple:

1. Traverse the tree in preorder.
2. Store nodes in a list.
3. Reconnect consecutive nodes.

The main downside is the additional memory usage. We store all nodes in an auxiliary array, which requires `O(n)` extra space.

### Optimal Approach

The optimal solution modifies the tree directly during traversal using pointer rewiring. The key insight is that for every node:

- If the node has a left subtree, that left subtree should appear immediately after the current node in preorder traversal.
- The original right subtree should come after the entire left subtree.

Therefore:

1. Find the rightmost node of the left subtree.
2. Connect that node's `right` pointer to the original right subtree.
3. Move the left subtree to the right side.
4. Set the left pointer to `null`.

This process preserves preorder order while transforming the tree in-place.

The idea resembles Morris traversal because we reuse existing pointers instead of allocating extra memory.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Store preorder traversal in an array, then reconnect nodes |
| Optimal | O(n) | O(1) | Rewire pointers directly inside the tree |

## Algorithm Walkthrough

### Optimal In-Place Algorithm

1. Start from the root node and iterate through the tree using a pointer called `current`.
2. For each node, check whether it has a left child. If there is no left child, we simply move to `current.right` because the node is already correctly positioned in the flattened structure.
3. If the node has a left subtree, we must insert that subtree between the current node and the current node's original right subtree. This matches preorder traversal order because preorder always visits the left subtree before the right subtree.
4. To do this safely, find the rightmost node inside the left subtree. This node represents the final node that would appear when traversing the left subtree in preorder.
5. Attach the original right subtree to the rightmost node of the left subtree. This preserves the correct traversal order.
6. Move the entire left subtree to the right side by setting:

- `current.right = current.left`
7. Remove the old left pointer by setting:

- `current.left = None`
8. Move to the next node using:

- `current = current.right`
9. Continue until all nodes are processed.

Because every node is visited only a constant number of times, the total running time remains linear.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import Optional

class Solution:
    def flatten(self, root: Optional[TreeNode]) -> None:
        """
        Do not return anything, modify root in-place instead.
        """
        
        current = root
        
        while current:
            if current.left:
                predecessor = current.left
                
                while predecessor.right:
                    predecessor = predecessor.right
                
                predecessor.right = current.right
                current.right = current.left
                current.left = None
            
            current = current.right
```

The implementation begins with a pointer called `current` that traverses the tree from top to bottom.

Whenever a node has a left subtree, we locate the rightmost node inside that subtree using the `predecessor` pointer. This rightmost node becomes the connection point for the original right subtree.

After attaching the original right subtree, we shift the left subtree into the right pointer position. Finally, we clear the left pointer because the flattened structure must not contain any left children.

The traversal then continues along the right side, which now represents the partially flattened linked list.

The algorithm modifies the tree directly and does not allocate additional data structures.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func flatten(root *TreeNode) {
	current := root

	for current != nil {
		if current.Left != nil {
			predecessor := current.Left

			for predecessor.Right != nil {
				predecessor = predecessor.Right
			}

			predecessor.Right = current.Right
			current.Right = current.Left
			current.Left = nil
		}

		current = current.Right
	}
}
```

The Go implementation follows the same logic as the Python version. The primary difference is Go's explicit pointer syntax and use of `nil` instead of `None`.

Because Go passes pointers directly, all modifications occur in-place automatically. No special return value is needed.

Go does not require type hints, but pointer handling is explicit through `*TreeNode`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each node is processed a constant number of times |
| Space | O(1) | No auxiliary data structures or recursion stack are used |

The time complexity is linear because every node participates in pointer rewiring at most once. Although we search for rightmost nodes inside left subtrees, the total traversal work across the entire algorithm remains proportional to the number of nodes.

The space complexity is constant because the algorithm only uses a few pointers regardless of tree size.

## Edge Cases

### Empty Tree

An empty tree is represented by `root = null`. This is important because attempting to access child pointers on a null node would cause runtime errors.

The implementation handles this naturally because the `while current:` loop never begins when `root` is `None`.

### Single Node Tree

A tree with one node has no left or right children. A buggy implementation might incorrectly modify pointers or enter unnecessary loops.

The algorithm correctly detects that there is no left subtree and simply moves to `current.right`, which is `None`.

### Completely Left-Skewed Tree

Consider this tree:

```
    1
   /
  2
 /
3
```

This is a common source of pointer bugs because every node must be rearranged.

The algorithm repeatedly moves the left subtree to the right side while preserving preorder order:

```
1 -> 2 -> 3
```

All left pointers become `null`, satisfying the problem requirements.

### Already Flattened Tree

Consider:

```
1
 \
  2
   \
    3
```

A poor implementation might unnecessarily rewire pointers or damage the structure.

In this algorithm, every node already has `left = null`, so the traversal simply walks down the right side without modifications.

### Nodes With Both Left and Right Subtrees

This is the most important general case because incorrect pointer ordering can lose entire subtrees.

The algorithm carefully stores the original right subtree by attaching it to the rightmost node of the left subtree before overwriting `current.right`. This guarantees that no nodes are lost during restructuring.
