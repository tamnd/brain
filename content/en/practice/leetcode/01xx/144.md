## Solution

## Problem Understanding

This problem asks us to perform a preorder traversal on a binary tree and return the values of the visited nodes in the order they are processed.

A binary tree is a hierarchical data structure where each node may have a left child and a right child. The input `root` represents the root node of the tree. If the tree is empty, `root` is `None` in Python or `nil` in Go.

In a preorder traversal, nodes are visited in this exact order:

1. Visit the current node
2. Traverse the left subtree
3. Traverse the right subtree

For example, consider this tree:

```
    1
     \
      2
     /
    3
```

The preorder traversal is:

```
1 -> 2 -> 3
```

because we first visit `1`, then move to its left subtree, which does not exist, then its right subtree rooted at `2`. Inside the subtree rooted at `2`, we visit `2` first, then its left child `3`.

The constraints are relatively small, with at most 100 nodes. This means almost any correct traversal solution will run efficiently enough. However, the follow up specifically asks for an iterative solution, which means we should understand both recursive and stack based traversal techniques.

Several edge cases are important:

- The tree may be empty
- The tree may contain only one node
- The tree may be completely skewed to the left or right
- Some nodes may have only one child
- The traversal order must strictly follow root, left, right

A common mistake is accidentally implementing inorder or postorder traversal instead of preorder traversal.

## Approaches

### Brute Force Approach

The most straightforward approach is recursive depth first traversal.

At every node:

1. Add the node's value to the result
2. Recursively traverse the left subtree
3. Recursively traverse the right subtree

This naturally matches the preorder definition itself, so it is easy to reason about and implement.

The recursive call stack implicitly keeps track of which nodes still need processing. Because each node is visited exactly once, the solution is efficient enough for the given constraints.

Although recursion is often considered the simplest solution, the follow up asks for an iterative approach. Recursive solutions also rely on the system call stack, which can become problematic for very deep trees in larger inputs.

### Optimal Approach

The optimal iterative approach uses an explicit stack to simulate recursion.

The key insight is that preorder traversal processes nodes in this order:

```
root -> left -> right
```

Since a stack is Last In First Out, we must push the right child before the left child. That way, the left child gets processed first when popped from the stack.

The algorithm works like this:

1. Push the root onto the stack
2. Repeatedly pop a node
3. Record its value
4. Push its right child if it exists
5. Push its left child if it exists

This guarantees preorder traversal order while avoiding recursion.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(h) | Recursive DFS using call stack |
| Optimal | O(n) | O(h) | Iterative DFS using explicit stack |

Here:

- `n` is the number of nodes
- `h` is the height of the tree

## Algorithm Walkthrough

### Iterative Preorder Traversal

1. Create an empty result list that will store the preorder traversal.

We need this list because the problem requires returning the traversal order as an array.
2. Handle the empty tree case.

If the root is `None` or `nil`, return an empty list immediately since there are no nodes to traverse.
3. Create a stack and push the root node onto it.

The stack simulates the recursive DFS process. We begin traversal from the root.
4. Start a loop that continues while the stack is not empty.

Each iteration processes one node from the tree.
5. Pop the top node from the stack.

This node is the next node in preorder traversal order.
6. Add the node's value to the result list.

Preorder traversal visits the current node before its children.
7. Push the node's right child onto the stack if it exists.

We push the right child first because stacks are LIFO. This ensures the left child gets processed before the right child.
8. Push the node's left child onto the stack if it exists.

Since it is pushed last, it is popped first, preserving preorder order.
9. Continue until the stack becomes empty.

At this point, every node has been visited exactly once.
10. Return the result list.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import List, Optional

class Solution:
    def preorderTraversal(self, root: Optional[TreeNode]) -> List[int]:
        if not root:
            return []

        result = []
        stack = [root]

        while stack:
            current = stack.pop()

            result.append(current.val)

            if current.right:
                stack.append(current.right)

            if current.left:
                stack.append(current.left)

        return result
```

This implementation begins by checking whether the tree is empty. If the root does not exist, an empty list is returned immediately.

The `result` list stores the preorder traversal output. The `stack` keeps track of nodes that still need processing.

Inside the loop, the current node is popped from the stack and its value is appended to the result list. Since preorder traversal visits the current node first, this operation happens immediately after popping.

The right child is pushed before the left child. This detail is extremely important. Because stacks process the most recently added element first, pushing the left child last guarantees that it will be visited before the right child.

The loop continues until every node has been processed and the stack becomes empty.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func preorderTraversal(root *TreeNode) []int {
    if root == nil {
        return []int{}
    }

    result := []int{}
    stack := []*TreeNode{root}

    for len(stack) > 0 {
        current := stack[len(stack)-1]
        stack = stack[:len(stack)-1]

        result = append(result, current.Val)

        if current.Right != nil {
            stack = append(stack, current.Right)
        }

        if current.Left != nil {
            stack = append(stack, current.Left)
        }
    }

    return result
}
```

The Go implementation follows the same logic as the Python solution.

Instead of Python lists, Go uses slices for both the result and the stack. Removing the top stack element is done by taking the last element and shrinking the slice.

Go uses `nil` instead of `None` to represent missing child pointers. Other than syntax differences, the traversal logic is identical.

Since node values are constrained between `-100` and `100`, integer overflow is not a concern.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every node is visited exactly once |
| Space | O(h) | Stack stores at most the height of the tree |

The traversal processes each node one time, so the runtime is linear in the number of nodes.

The space complexity depends on the maximum number of nodes stored in the stack simultaneously. In a balanced tree, this is proportional to the tree height. In the worst case of a completely skewed tree, the height becomes `n`, so the worst case auxiliary space is `O(n)`.

## Edge Cases

### Empty Tree

An empty tree is represented by `root = None` in Python or `root = nil` in Go. This case can easily cause errors if the implementation assumes the root always exists.

The solution handles this immediately with an early return, producing an empty traversal list.

### Single Node Tree

A tree with only one node tests whether the implementation correctly handles trees without children.

The algorithm pushes the root onto the stack, processes it once, and terminates correctly without attempting to access nonexistent children.

### Completely Skewed Tree

A tree may resemble a linked list, with every node having only one child.

For example:

```
1
 \
  2
   \
    3
```

This case is important because the stack may grow to size `n`. The iterative solution still works correctly because each node is processed independently and child existence is checked before pushing.

### Nodes With Only One Child

Some nodes may have only a left child or only a right child.

A buggy implementation might incorrectly assume both children exist or push them in the wrong order.

The solution explicitly checks whether each child exists before pushing it onto the stack, ensuring correct traversal behavior for all tree shapes.
