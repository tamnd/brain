## Solution

## Problem Understanding

This problem asks us to determine the **maximum number of points that lie on the same straight line** in a 2D coordinate plane.

The input is an array `points`, where each element is a coordinate pair `[x, y]`. Each coordinate represents a unique point in the X-Y plane. Our goal is to examine all possible straight lines that can be formed using these points and return the largest number of points that lie on any single line.

In simpler terms, imagine drawing a line through any two points. Some other points may also fall perfectly on that same line. We want to find the line that passes through the most points.

For example:

```
points = [[1,1],[2,2],[3,3]]
```

All three points lie on the line:

```
y = x
```

So the answer is `3`.

The problem guarantees that all points are unique, meaning we never need to handle exact duplicate coordinates. However, multiple lines can share some points, and vertical lines must also be handled carefully.

The constraints are important:

- `1 <= points.length <= 300`
- Coordinates range from `-10^4` to `10^4`

Since there can be up to 300 points, an extremely expensive brute-force solution may still work if carefully optimized, but cubic or quartic solutions become risky. We should aim for approximately `O(n²)` or `O(n² log n)`.

Several edge cases are worth identifying upfront.

A single point should return `1`, because one point alone trivially lies on a line.

Vertical lines such as:

```
[[1,1],[1,2],[1,3]]
```

must be handled correctly, since their slope is undefined.

Horizontal lines such as:

```
[[1,2],[3,2],[5,2]]
```

should also work.

Floating point precision can introduce subtle bugs when comparing slopes. For example:

```
[[0,0],[94911151,94911150],[94911152,94911151]]
```

These points are almost collinear, and floating point arithmetic may incorrectly group slopes together. We need an exact representation instead of decimals.

## Approaches

### Brute Force Approach

The brute-force idea is to examine every possible line formed by every pair of points, then count how many points lie on that line.

For every pair `(i, j)`:

1. Construct the line passing through those two points.
2. Check every remaining point `k`.
3. Determine whether point `k` lies on the same line.
4. Track the maximum count found.

To test whether three points are collinear without floating point issues, we can use the cross product condition:

```
(y2 - y1)(x3 - x1) = (y3 - y1)(x2 - x1)
```

This guarantees exact integer arithmetic.

The brute-force approach is correct because it explicitly checks every possible line and counts all points on it. However, it is inefficient.

There are `O(n²)` candidate lines, and for each line we scan all `n` points, resulting in `O(n³)` time complexity.

With `n = 300`, this means approximately:

```
300³ = 27,000,000
```

operations in the worst case. While borderline acceptable in some languages, it is unnecessarily expensive.

### Optimal Approach, Hash Map of Normalized Slopes

The key insight is that **a line can be uniquely identified by its slope relative to a fixed anchor point**.

Instead of considering all triples, we choose one point as an anchor and compute the slope to every other point.

Suppose point `i` is fixed.

Every point that lies on the same line through `i` will produce the same slope.

Example:

```
anchor = [1,1]

[2,2] -> slope = 1
[3,3] -> slope = 1
[4,4] -> slope = 1
```

All of these belong to the same line.

We can use a hash map:

```
slope -> count
```

For each anchor point:

- Compute normalized slopes to all later points.
- Count how often each slope appears.
- The largest slope frequency plus the anchor itself gives the maximum line through that point.

The most important detail is how we represent slopes.

Using floating point numbers is dangerous due to precision errors. Instead, we normalize the slope using a reduced fraction.

For two points:

```
dx = x2 - x1
dy = y2 - y1
```

We divide both by:

```
gcd(dx, dy)
```

Example:

```
dx = 4, dy = 2
```

becomes:

```
(2,1)
```

Likewise:

```
dx = 8, dy = 4
```

also becomes:

```
(2,1)
```

This ensures equivalent slopes map to the same key.

We also normalize signs so that:

```
(-2,-1)
```

becomes:

```
(2,1)
```

This prevents duplicate representations of the same slope.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | `O(n³)` | `O(1)` | Check every line and test every point for collinearity |
| Optimal | `O(n²)` | `O(n)` | Use slope frequency counting for each anchor point |

## Algorithm Walkthrough

1. Handle the trivial case where there is only one point. Return `1`, since a single point forms a valid line by itself.
2. Initialize a variable `max_points` to track the best answer found so far.
3. Iterate through each point and treat it as an **anchor point**.
4. For the current anchor point, create a hash map that stores:

```
(normalized_dx, normalized_dy) -> frequency
```

This map counts how many points share the same slope relative to the anchor.
5. Compare the anchor point with every later point.

Compute:

```
dx = x2 - x1
dy = y2 - y1
```
6. Normalize the slope using `gcd(dx, dy)`.

Divide both values:

```
dx //= gcd
dy //= gcd
```

Then standardize the sign:

- If `dx < 0`, multiply both by `-1`
- If `dx == 0`, force vertical lines to `(0,1)`
- If `dy == 0`, force horizontal lines to `(1,0)`

This guarantees one canonical representation for each slope.
7. Store the normalized slope in the hash map and increment its count.
8. Update the local maximum for this anchor point.
9. After processing all points relative to the anchor, update the global answer.
10. Continue until every point has been used as an anchor.
11. Return the maximum number of collinear points found.

The reason this works is that every line must pass through at least one anchor point. When that anchor is processed, all points on the same line generate identical normalized slopes and are counted together.

## Python Solution

```
from typing import List
from collections import defaultdict
from math import gcd

class Solution:
    def maxPoints(self, points: List[List[int]]) -> int:
        n = len(points)

        if n <= 2:
            return n

        max_points = 1

        for i in range(n):
            slopes = defaultdict(int)
            local_max = 0

            x1, y1 = points[i]

            for j in range(i + 1, n):
                x2, y2 = points[j]

                dx = x2 - x1
                dy = y2 - y1

                divisor = gcd(dx, dy)

                dx //= divisor
                dy //= divisor

                # Normalize sign
                if dx < 0:
                    dx *= -1
                    dy *= -1

                # Vertical line
                if dx == 0:
                    dy = 1

                # Horizontal line
                if dy == 0:
                    dx = 1

                slope = (dx, dy)

                slopes[slope] += 1
                local_max = max(local_max, slopes[slope])

            max_points = max(max_points, local_max + 1)

        return max_points
```

The implementation begins by handling the small edge case where the number of points is at most two. In that situation, every point lies on the same line trivially.

We then iterate through every point and treat it as an anchor point. For each anchor, we create a `defaultdict(int)` to count how many times each normalized slope appears.

The nested loop compares the anchor with every later point. We compute `dx` and `dy`, then reduce them using `gcd`. This reduction ensures equivalent slopes receive identical representations.

The normalization logic is important. A slope of:

```
(-2, -1)
```

should be treated the same as:

```
(2, 1)
```

so we force the sign into a consistent form.

Vertical and horizontal lines receive canonical representations:

```
vertical -> (0, 1)
horizontal -> (1, 0)
```

Without this normalization, equivalent lines could end up in separate hash map buckets.

The largest slope count for an anchor represents how many _other_ points share a line with it, so we add `1` for the anchor itself.

## Go Solution

```
package main

import "math"

func gcd(a, b int) int {
	a = int(math.Abs(float64(a)))
	b = int(math.Abs(float64(b)))

	for b != 0 {
		a, b = b, a%b
	}

	return a
}

func maxPoints(points [][]int) int {
	n := len(points)

	if n <= 2 {
		return n
	}

	maxPointsCount := 1

	for i := 0; i < n; i++ {
		slopes := make(map[[2]int]int)
		localMax := 0

		x1, y1 := points[i][0], points[i][1]

		for j := i + 1; j < n; j++ {
			x2, y2 := points[j][0], points[j][1]

			dx := x2 - x1
			dy := y2 - y1

			divisor := gcd(dx, dy)

			dx /= divisor
			dy /= divisor

			// Normalize sign
			if dx < 0 {
				dx *= -1
				dy *= -1
			}

			// Vertical line
			if dx == 0 {
				dy = 1
			}

			// Horizontal line
			if dy == 0 {
				dx = 1
			}

			slope := [2]int{dx, dy}

			slopes[slope]++
			if slopes[slope] > localMax {
				localMax = slopes[slope]
			}
		}

		if localMax+1 > maxPointsCount {
			maxPointsCount = localMax + 1
		}
	}

	return maxPointsCount
}
```

The Go implementation follows the same logic as the Python version. Since Go does not have a built-in `gcd` for integers, we implement it manually using the Euclidean algorithm.

Instead of tuples, Go uses a fixed-size array:

```
[2]int
```

as the map key for slopes.

Maps in Go default to zero values for missing keys, so incrementing counts works naturally.

Integer overflow is not a concern here because coordinates are bounded by `10^4`, making slope differences safely fit within standard integer ranges.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | `O(n²)` | For every point, we compare against all later points |
| Space | `O(n)` | The slope hash map stores at most one entry per other point |

The nested loop processes every pair of points exactly once, resulting in quadratic complexity:

```
O(n²)
```

For each anchor point, the hash map stores slope counts for at most `n - 1` other points, giving:

```
O(n)
```

extra space.

Since `n <= 300`, this solution comfortably fits within the constraints.

## Edge Cases

### Single Point

If the input contains only one point:

```
[[5,7]]
```

the answer should be `1`.

This can easily be overlooked if the algorithm assumes every line requires at least two points. The implementation handles this immediately with:

```
if n <= 2:
    return n
```

### Vertical Lines

Points like:

```
[[2,1],[2,3],[2,8]]
```

all lie on a vertical line.

The slope is technically undefined because division by zero occurs. A naive implementation using floating point division may crash or produce invalid values.

We avoid this by storing vertical lines in canonical form:

```
(0,1)
```

which safely groups all vertical lines together.

### Negative Slopes and Sign Ambiguity

Consider:

```
[[1,1],[2,2],[0,0]]
```

The same line can produce:

```
(1,1)
(-1,-1)
```

depending on direction.

Without normalization, these would incorrectly appear as separate slopes in the hash map.

The implementation fixes this by forcing the sign to a consistent representation:

```
if dx < 0:
    dx *= -1
    dy *= -1
```

This guarantees equivalent slopes always map to the same key.

### Nearly Equal Floating Point Slopes

Coordinates with large values can produce precision problems if decimal slopes are used.

Example:

```
[[0,0],[94911151,94911150],[94911152,94911151]]
```

Using floating point arithmetic may incorrectly treat these slopes as identical.

By reducing slopes with `gcd` and storing exact integer pairs:

```
(dx, dy)
```

the implementation avoids precision errors entirely and remains mathematically exact.
