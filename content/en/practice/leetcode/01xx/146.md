## Solution

## Problem Understanding

This problem asks us to design an efficient cache system that follows the **Least Recently Used (LRU)** eviction policy. A cache is a temporary storage structure that keeps recently accessed data so it can be retrieved quickly. However, the cache has a fixed capacity, meaning it cannot grow indefinitely. When it becomes full and a new item must be inserted, we need a strategy to decide which existing item to remove.

The required strategy here is **Least Recently Used**. This means the cache should remove the item that has gone the longest without being accessed. Importantly, both reading (`get`) and writing (`put`) count as usage. Whenever a key is accessed or updated, it becomes the most recently used item.

The class must support three operations:

- `LRUCache(capacity)` initializes the cache with a maximum number of entries.
- `get(key)` returns the value associated with the key if it exists, otherwise `-1`.
- `put(key, value)` inserts or updates a key-value pair. If inserting causes the cache to exceed capacity, the least recently used entry must be removed.

The major challenge is the time complexity requirement. Both `get` and `put` must run in **O(1)** average time complexity. This requirement immediately rules out many straightforward approaches.

The input represents a sequence of operations applied to the cache. For example:

```
["LRUCache", "put", "put", "get"]
[[2], [1,1], [2,2], [1]]
```

means:

1. Create an `LRUCache` with capacity `2`
2. Insert `(1,1)`
3. Insert `(2,2)`
4. Retrieve key `1`

The output corresponds to each operation. Constructor and `put` return `null`, while `get` returns either a stored value or `-1`.

The constraints provide important guidance:

- `capacity <= 3000`, which is relatively small, but the number of operations can be as high as `2 * 10^5`.
- Since there may be hundreds of thousands of operations, even an `O(n)` operation would become too slow in the worst case.
- We need consistent constant-time performance.

Several edge cases are important:

- Accessing a non-existent key should return `-1`.
- Updating an existing key should not increase cache size.
- Accessing a key makes it recently used.
- When capacity is exceeded, exactly one least recently used key must be evicted.
- A cache with capacity `1` is particularly error-prone because every new insertion may trigger eviction.

## Approaches

### Brute Force Approach

A straightforward solution is to store key-value pairs in a hash map and separately maintain usage order using a list or array.

For `get(key)`:

- Look up the key in the map.
- If found, move it to the end of the list to mark it as recently used.

For `put(key, value)`:

- If the key exists, update the value and move it to the most recent position.
- If it does not exist, append it.
- If capacity is exceeded, remove the least recently used key from the front.

The problem is that moving elements inside an array or list is expensive. Removing an item from the middle or front requires shifting elements, which costs `O(n)` time.

With up to `200,000` operations, this becomes too slow.

### Key Insight

We need **constant-time lookup** and **constant-time ordering updates**.

A hash map provides `O(1)` key lookup, but it does not maintain ordering in a way that supports constant-time reordering.

A doubly linked list provides:

- `O(1)` insertion
- `O(1)` deletion
- `O(1)` movement of nodes

The key observation is that if we combine these two structures:

1. A **hash map** maps keys to linked list nodes for instant access.
2. A **doubly linked list** keeps items ordered by recency.

We place the **most recently used item near the head** and the **least recently used item near the tail**.

Whenever:

- `get(key)` succeeds, move that node to the front.
- `put(key, value)` updates an existing key, move it to the front.
- Capacity is exceeded, remove the tail node.

To simplify edge cases, we use **dummy head and tail nodes**. These sentinel nodes eliminate special-case logic when inserting or deleting at boundaries.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) per operation | O(n) | Uses a list to track recency, shifting makes updates slow |
| Optimal | O(1) per operation | O(capacity) | Combines hash map and doubly linked list |

## Algorithm Walkthrough

1. **Initialize the cache structures**

Create:

- A hash map `cache` that maps `key -> node`
- A doubly linked list for recency tracking
- Dummy `head` and `tail` nodes

The list structure looks like:

```
head <-> most recent ... least recent <-> tail
```

Dummy nodes simplify insertions and removals because every real node always has neighbors.
2. **Create helper methods for linked list operations**

We define:

- `remove(node)` to disconnect a node from the list.
- `insert(node)` to place a node immediately after the head.

These operations run in constant time.
3. **Handle `get(key)`**

First check whether the key exists in the hash map.

- If not found, return `-1`.
- If found:

- Remove the node from its current position.
- Insert it near the head.
- Return its value.

Moving it to the front marks it as recently used.
4. **Handle `put(key, value)` when key exists**

If the key already exists:

- Update the stored value.
- Remove the node from its current position.
- Reinsert it near the head.

This refreshes its usage status.
5. **Handle `put(key, value)` for a new key**

Create a new node and insert it after the head.

Add it to the hash map.
6. **Evict least recently used item if necessary**

After insertion, if cache size exceeds capacity:

- Remove the node before `tail`
- Delete its key from the hash map

Since the least recently used item is always near the tail, eviction is constant time.

## Python Solution

```
class Node:
    def __init__(self, key: int = 0, value: int = 0):
        self.key = key
        self.value = value
        self.prev = None
        self.next = None

class LRUCache:

    def __init__(self, capacity: int):
        self.capacity = capacity
        self.cache: dict[int, Node] = {}

        self.head = Node()
        self.tail = Node()

        self.head.next = self.tail
        self.tail.prev = self.head

    def _remove(self, node: Node) -> None:
        prev_node = node.prev
        next_node = node.next

        prev_node.next = next_node
        next_node.prev = prev_node

    def _insert(self, node: Node) -> None:
        first_real_node = self.head.next

        self.head.next = node
        node.prev = self.head

        node.next = first_real_node
        first_real_node.prev = node

    def get(self, key: int) -> int:
        if key not in self.cache:
            return -1

        node = self.cache[key]

        self._remove(node)
        self._insert(node)

        return node.value

    def put(self, key: int, value: int) -> None:
        if key in self.cache:
            existing_node = self.cache[key]

            self._remove(existing_node)

        new_node = Node(key, value)
        self.cache[key] = new_node
        self._insert(new_node)

        if len(self.cache) > self.capacity:
            lru_node = self.tail.prev

            self._remove(lru_node)
            del self.cache[lru_node.key]
```

The implementation begins by defining a `Node` class for the doubly linked list. Each node stores:

- `key`, needed for removal from the hash map during eviction
- `value`, the cached value
- `prev` and `next` pointers for linked list navigation

The constructor initializes:

- `capacity`
- `cache`, a hash map
- Dummy `head` and `tail` nodes

The helper method `_remove()` disconnects a node from the list by reconnecting its neighboring nodes. Because it directly manipulates pointers, this operation is `O(1)`.

The helper method `_insert()` places a node immediately after the head, marking it as most recently used.

In `get()`, we first check whether the key exists. If it does, the node is moved to the front because reading counts as usage.

In `put()`, existing keys are removed before reinsertion. Instead of updating the node in place, we create a new node and replace the mapping. This keeps the implementation simple and still maintains `O(1)` performance.

After insertion, if capacity is exceeded, we evict the node immediately before the tail, which always represents the least recently used item.

## Go Solution

```
type Node struct {
	key   int
	value int
	prev  *Node
	next  *Node
}

type LRUCache struct {
	capacity int
	cache    map[int]*Node
	head     *Node
	tail     *Node
}

func Constructor(capacity int) LRUCache {
	head := &Node{}
	tail := &Node{}

	head.next = tail
	tail.prev = head

	return LRUCache{
		capacity: capacity,
		cache:    make(map[int]*Node),
		head:     head,
		tail:     tail,
	}
}

func (this *LRUCache) remove(node *Node) {
	prevNode := node.prev
	nextNode := node.next

	prevNode.next = nextNode
	nextNode.prev = prevNode
}

func (this *LRUCache) insert(node *Node) {
	firstRealNode := this.head.next

	this.head.next = node
	node.prev = this.head

	node.next = firstRealNode
	firstRealNode.prev = node
}

func (this *LRUCache) Get(key int) int {
	node, exists := this.cache[key]

	if !exists {
		return -1
	}

	this.remove(node)
	this.insert(node)

	return node.value
}

func (this *LRUCache) Put(key int, value int) {
	if existingNode, exists := this.cache[key]; exists {
		this.remove(existingNode)
	}

	newNode := &Node{
		key:   key,
		value: value,
	}

	this.cache[key] = newNode
	this.insert(newNode)

	if len(this.cache) > this.capacity {
		lruNode := this.tail.prev

		this.remove(lruNode)
		delete(this.cache, lruNode.key)
	}
}
```

The Go implementation mirrors the Python logic closely. Since Go does not have built-in doubly linked list support suited for this problem, we manually implement the node structure using pointers.

The biggest language-specific difference is pointer handling. Nodes are referenced through `*Node`, and linked list updates manipulate pointers directly.

Go maps return two values during lookup: the value and a boolean indicating existence. This is used in `Get()` and `Put()` to determine whether a key exists.

There are no concerns about integer overflow here because constraints remain well within Go's integer range.

## Worked Examples

### Example 1

Input:

```
LRUCache(2)
put(1,1)
put(2,2)
get(1)
put(3,3)
get(2)
put(4,4)
get(1)
get(3)
get(4)
```

We represent cache order as:

```
[Most Recent ... Least Recent]
```

| Operation | Result | Cache State |
| --- | --- | --- |
| `put(1,1)` | null | `[1]` |
| `put(2,2)` | null | `[2,1]` |
| `get(1)` | `1` | `[1,2]` |
| `put(3,3)` | null | `[3,1]` (evict 2) |
| `get(2)` | `-1` | `[3,1]` |
| `put(4,4)` | null | `[4,3]` (evict 1) |
| `get(1)` | `-1` | `[4,3]` |
| `get(3)` | `3` | `[3,4]` |
| `get(4)` | `4` | `[4,3]` |

Detailed walkthrough:

1. `put(1,1)` inserts key `1`.

```
head <-> 1 <-> tail
```
2. `put(2,2)` inserts key `2` at front.

```
head <-> 2 <-> 1 <-> tail
```
3. `get(1)` returns `1` and moves key `1` to front.

```
head <-> 1 <-> 2 <-> tail
```
4. `put(3,3)` inserts `3`.

Cache exceeds capacity.

Least recently used node is `2`, remove it.

```
head <-> 3 <-> 1 <-> tail
```
5. `get(2)` fails.

Return `-1`.
6. `put(4,4)` inserts `4`.

Cache exceeds capacity.

Remove LRU node `1`.

```
head <-> 4 <-> 3 <-> tail
```
7. `get(1)` returns `-1`.
8. `get(3)` returns `3`.

```
head <-> 3 <-> 4 <-> tail
```
9. `get(4)` returns `4`.

```
head <-> 4 <-> 3 <-> tail
```

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(1) | Every `get`, insertion, removal, and eviction is constant time |
| Space | O(capacity) | Hash map and linked list store at most `capacity` nodes |

The time complexity is `O(1)` because every operation consists only of hash map lookups and pointer updates in the doubly linked list. There are no loops or shifting operations.

The space complexity is `O(capacity)` because we store at most one node per cache entry in both the hash map and linked list.

## Edge Cases

### Capacity Equals One

A cache with capacity `1` is easy to mishandle because every insertion after the first must evict the existing element.

Example:

```
put(1,1)
put(2,2)
```

The second insertion must immediately remove key `1`.

The implementation handles this naturally because capacity checks occur after every insertion, and the least recently used node is always removed from the tail side.

### Updating an Existing Key

If `put(key, value)` is called for an existing key, we should update the value rather than create a duplicate entry.

Example:

```
put(1,1)
put(1,10)
```

After the second operation, key `1` should contain value `10` and become the most recently used item.

The implementation removes the old node, creates a replacement node with the updated value, reinserts it, and updates the hash map.

### Accessing a Missing Key

Calling `get()` on a key that does not exist must return `-1` without changing cache order.

Example:

```
get(99)
```

A naive implementation might accidentally alter state or throw an error.

The hash map lookup ensures this case is handled immediately by returning `-1`.

### Repeated Accesses to the Same Key

A frequently accessed key should remain in the cache and continuously move to the most recently used position.

Example:

```
put(1,1)
put(2,2)
get(1)
get(1)
get(1)
```

Repeated access should never duplicate nodes or corrupt ordering.

Because the implementation removes and reinserts the same node in constant time, the ordering remains correct throughout.
