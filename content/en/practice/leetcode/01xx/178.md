## Solution

## Problem Understanding

This problem asks us to assign rankings to game scores stored in a SQL table. The rankings must follow a specific rule set known as **dense ranking**. The important detail is that equal scores receive the same rank, and rank numbers remain consecutive without gaps.

We are given a table called `Scores`:

| Column | Meaning |
| --- | --- |
| `id` | Unique identifier for each row |
| `score` | A decimal score value |

The goal is to return a result table containing:

- The `score`
- Its computed `rank`

The result must be sorted by `score` in **descending order**, meaning the highest score appears first.

The ranking rules are the most important part of the problem:

1. Higher scores get smaller rank numbers.
2. Equal scores share the same rank.
3. Rankings must remain consecutive after ties.

This third rule is where many implementations go wrong. Consider the following scores:

`100, 90, 90, 80`

The ranks should be:

`1, 2, 2, 3`

Notice that after the tied `90`s, the next rank is `3`, not `4`. This differs from standard competition ranking, where skipped numbers can occur.

The input represents a database table where duplicate scores are allowed. Since `id` is unique but irrelevant to ranking, we only care about the `score` values.

The output should preserve duplicates because every row in the table represents an actual score entry. Even if two players have the same score, both rows should appear in the result.

There are a few important constraints implied by the problem:

- Scores are decimal values with two decimal places.
- Duplicate scores are common and must be handled correctly.
- We must order results by score descending.
- The solution should work efficiently on large datasets.

An important observation is that this problem is fundamentally about **ranking unique score values**, then mapping those rankings back to every row.

### Important Edge Cases

A naive implementation can easily fail on tied scores. If we simply count row positions after sorting, duplicate scores would incorrectly receive different ranks.

Another tricky case occurs when **all scores are identical**. Every row should receive rank `1`.

A third edge case is when **all scores are unique**. In that scenario, rankings simply become sequential integers starting from `1`.

Finally, decimal comparison matters. Since scores are floating-point decimals, we must compare exact score values rather than relying on integer assumptions.

## Approaches

### Brute Force Approach

A straightforward way to solve this problem is to compute each score's rank independently.

For every row in the table, we could count how many **distinct scores are strictly greater** than the current score. The rank would then be:

`rank = count(distinct higher scores) + 1`

For example, if a score is `3.65` and there are two distinct scores greater than it (`4.00` and `3.85`), then its rank is `3`.

This works because ranking depends entirely on how many unique scores appear above the current score.

In SQL, this can be implemented using a correlated subquery:

```
SELECT
    s1.score,
    (
        SELECT COUNT(DISTINCT s2.score)
        FROM Scores s2
        WHERE s2.score > s1.score
    ) + 1 AS rank
FROM Scores s1
ORDER BY s1.score DESC;
```

The problem with this approach is efficiency. For every row, we scan the table again to count higher scores. If there are `N` rows, this can lead to roughly `O(N²)` work.

### Optimal Approach

The key insight is that SQL databases already provide ranking functionality through **window functions**.

This problem exactly matches the behavior of the `DENSE_RANK()` function.

`DENSE_RANK()` assigns:

- The same rank to equal values
- Consecutive ranks without gaps

That is precisely what the problem requires.

We can simply rank rows based on `score` in descending order:

```
DENSE_RANK() OVER (ORDER BY score DESC)
```

This automatically handles duplicates and consecutive ranking.

Because the database engine handles sorting and ranking efficiently, this solution is significantly better than repeatedly scanning the table.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(N²) | O(1) | Correlated subquery scans the table repeatedly |
| Optimal | O(N log N) | O(N) | Uses `DENSE_RANK()` after sorting scores |

## Algorithm Walkthrough

### Optimal Algorithm Using `DENSE_RANK()`

1. Sort all scores in descending order because the highest score must receive rank `1`.
2. Use the SQL window function `DENSE_RANK()` to assign rankings. This function automatically groups equal scores together and ensures ranks remain consecutive.
3. Specify the ordering rule inside the window function using:

```
ORDER BY score DESC
```

This guarantees larger scores receive smaller rank values.
4. Select the `score` column alongside the computed rank so the output matches the required format.
5. Order the final result by score descending to satisfy the problem statement.

The reason `DENSE_RANK()` is ideal here is that it directly models the problem requirements. We do not need additional data structures, manual counting, or custom ranking logic.

## Python Solution

Since this is a Database problem, LeetCode expects an SQL query rather than executable Python code. The following is the complete LeetCode-submittable SQL solution.

```
# This problem requires SQL, not Python code.
```

The core SQL query is:

```
SELECT
    score,
    DENSE_RANK() OVER (ORDER BY score DESC) AS `rank`
FROM Scores
ORDER BY score DESC;
```

The implementation works by first sorting scores in descending order. The `DENSE_RANK()` window function then traverses the sorted values and assigns rankings. Identical scores automatically receive the same rank, while new distinct scores increment the rank by exactly one. Finally, the outer `ORDER BY` ensures the result appears from highest to lowest score.

## Edge Cases

### All Scores Are Identical

Consider:

| score |
| --- |
| 5.00 |
| 5.00 |
| 5.00 |

This case can break naive implementations that increment rank for every row. Since every score is equal, all rows must receive rank `1`. `DENSE_RANK()` handles this naturally because identical values share the same rank.

### All Scores Are Unique

Consider:

| score |
| --- |
| 10.00 |
| 8.50 |
| 7.20 |

Here, every score is different, so ranks should simply be:

`1, 2, 3`

The implementation correctly increments the rank for each new distinct value.

### Multiple Ties

Consider:

| score |
| --- |
| 100 |
| 95 |
| 95 |
| 90 |
| 90 |
| 90 |

A buggy implementation might produce:

`1, 2, 2, 4, 4, 4`

which incorrectly skips rank `3`.

The correct output is:

`1, 2, 2, 3, 3, 3`

`DENSE_RANK()` guarantees consecutive rankings with no holes, making it ideal for this requirement.

### Single Row Table

If the table contains only one score, that score should always receive rank `1`.

The implementation handles this automatically because the first row in a dense ranking sequence always starts at `1`.
