# LeetCode 135 - Candy

## Problem Understanding

The problem gives us an array called `ratings`, where each element represents the rating of a child standing in a line. We need to distribute candies to these children while satisfying two rules:

1. Every child must receive at least one candy.
2. If a child has a higher rating than one of their immediate neighbors, that child must receive more candies than that neighbor.

The goal is to minimize the total number of candies distributed while still satisfying both conditions.

The input is a one dimensional integer array. The position of each element matters because each child only compares with adjacent neighbors, not with every child in the line. The output is a single integer representing the minimum total candies required.

The constraints tell us that the number of children can be as large as `2 * 10^4`. This is large enough that inefficient repeated adjustment approaches can become too slow. A quadratic time solution may still pass in some environments, but the problem is designed to encourage a linear time greedy strategy.

Several edge cases are important:

- A single child should always receive exactly one candy.
- Multiple children with the same rating do not require different candy counts.
- Strictly increasing sequences require continuously increasing candy counts.
- Strictly decreasing sequences are trickier because earlier assignments may need adjustment.
- Peaks and valleys in the ratings array can create conflicting neighbor requirements.

The main difficulty comes from satisfying both left neighbor and right neighbor constraints simultaneously while still minimizing the total.

## Approaches

### Brute Force Approach

A straightforward idea is to initialize every child with one candy, then repeatedly scan the array and adjust candy counts whenever a rule is violated.

For example:

- If `ratings[i] > ratings[i - 1]`, ensure `candies[i] > candies[i - 1]`
- If `ratings[i] > ratings[i + 1]`, ensure `candies[i] > candies[i + 1]`

We continue repeating passes until no changes are needed.

This approach eventually reaches a valid distribution because every update moves the system closer to satisfying the constraints. However, the same positions may need to be updated many times, especially in long decreasing sequences.

In the worst case, such as a strictly decreasing array, this repeated adjustment process can take quadratic time.

### Optimal Greedy Approach

The key insight is that the problem can be decomposed into two directional constraints:

- Left to right constraints
- Right to left constraints

If a child has a higher rating than the left neighbor, they need more candies than the left neighbor. This can be handled with a left to right pass.

Similarly, if a child has a higher rating than the right neighbor, they need more candies than the right neighbor. This can be handled with a right to left pass.

The elegant observation is that both constraints can be satisfied independently, then combined by taking the maximum candy requirement at each position.

We begin by assigning every child one candy. Then:

1. Traverse left to right to satisfy increasing relationships from the left.
2. Traverse right to left to satisfy increasing relationships from the right.
3. For each child, keep the larger requirement.

This greedy strategy works because each local comparison only depends on adjacent children, and taking the maximum preserves both directional constraints.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeatedly adjusts candies until all constraints are satisfied |
| Optimal | O(n) | O(n) | Uses two greedy passes to satisfy both neighbor directions |

## Algorithm Walkthrough

1. Create an array called `candies` with size `n`, initialized to `1`.

Every child must receive at least one candy, so this establishes the minimum valid starting state.
2. Perform a left to right traversal.

For each index `i` from `1` to `n - 1`:

- If `ratings[i] > ratings[i - 1]`, then child `i` must receive more candies than the left neighbor.
- Set:

```
candies[i] = candies[i - 1] + 1
```

This pass ensures all left neighbor constraints are satisfied.
3. Perform a right to left traversal.

For each index `i` from `n - 2` down to `0`:

- If `ratings[i] > ratings[i + 1]`, then child `i` must receive more candies than the right neighbor.
- Update:

```
candies[i] = max(candies[i], candies[i + 1] + 1)
```

The `max` is critical because the first pass may already have assigned a larger value that must be preserved.
4. Sum all values in the `candies` array.

This produces the minimum total candies satisfying all constraints.

The algorithm uses a simple array because we only need direct indexed access and sequential traversal. No advanced data structure is necessary.

## Python Solution

```
from typing import List

class Solution:
    def candy(self, ratings: List[int]) -> int:
        n = len(ratings)

        candies = [1] * n

        # Left to right pass
        for i in range(1, n):
            if ratings[i] > ratings[i - 1]:
                candies[i] = candies[i - 1] + 1

        # Right to left pass
        for i in range(n - 2, -1, -1):
            if ratings[i] > ratings[i + 1]:
                candies[i] = max(candies[i], candies[i + 1] + 1)

        return sum(candies)
```

The implementation begins by initializing every child with one candy. This guarantees the minimum valid distribution.

The first loop processes the array from left to right. Whenever the current child's rating is greater than the previous child's rating, the current child receives one more candy than the previous child.

However, this only handles left side relationships. A decreasing sequence such as `[5,4,3,2,1]` would still violate the right neighbor rule after the first pass.

The second loop processes the array from right to left. Whenever the current child's rating is greater than the next child's rating, we ensure the current child has at least one more candy than the next child.

The `max` operation is important because the left to right pass may already have assigned a larger candy count that should not be overwritten.

Finally, summing the `candies` array gives the minimum total candies needed.

## Go Solution

```
func candy(ratings []int) int {
	n := len(ratings)

	candies := make([]int, n)

	for i := 0; i < n; i++ {
		candies[i] = 1
	}

	// Left to right pass
	for i := 1; i < n; i++ {
		if ratings[i] > ratings[i-1] {
			candies[i] = candies[i-1] + 1
		}
	}

	// Right to left pass
	for i := n - 2; i >= 0; i-- {
		if ratings[i] > ratings[i+1] {
			if candies[i] < candies[i+1]+1 {
				candies[i] = candies[i+1] + 1
			}
		}
	}

	total := 0

	for _, candy := range candies {
		total += candy
	}

	return total
}
```

The Go implementation follows the exact same logic as the Python version. Since Go does not provide automatic list initialization with a repeated value like Python does, we explicitly fill the `candies` slice with `1`.

The right to left pass uses a manual comparison instead of a built in `max` function because Go does not include one for integers in the standard library.

No special handling for `nil` slices is needed because the problem guarantees at least one child.

### Left to Right Pass

At `i = 1`:

- `0 > 1` is false
- No change

- `2 > 0` is true
- `candies[2] = candies[1] + 1 = 2`

State after first pass:

| Index | Rating | Candies |
| --- | --- | --- |
| 0 | 1 | 1 |
| 1 | 0 | 1 |

### Right to Left Pass

At `i = 1`:

- `0 > 2` is false
- No change

At `i = 0`:

- `1 > 0` is true
- `candies[0] = max(1, 1 + 1) = 2`

Final state:

| Index | Rating | Candies |
| --- | --- | --- |
| 0 | 1 | 2 |
| 1 | 0 | 1 |
| 2 | 2 | 2 |

Total:

```
2 + 1 + 2 = 5
```

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Two linear passes through the array |
| Space | O(n) | Additional candies array of size n |

The algorithm performs exactly two traversals of the ratings array, making the runtime linear with respect to the number of children.

The extra memory comes from storing the candy assignments in a separate array. Each child requires one integer entry, so the space usage is linear.

## Edge Cases

### Single Child

If the input contains only one child, such as:

```
ratings = [5]
```

The child must receive at least one candy, and there are no neighbors to compare against. The algorithm correctly initializes the candies array to `[1]` and returns `1`.

### Strictly Increasing Ratings

Consider:

```
ratings = [1,2,3,4]
```

Each child must receive more candies than the previous child. The left to right pass produces:

```
[1,2,3,4]
```

The right to left pass makes no changes. The implementation correctly handles long increasing chains.

### Strictly Decreasing Ratings

Consider:

```
ratings = [4,3,2,1]
```

The left to right pass initially gives every child one candy because no increasing relationship exists from the left.

The right to left pass then adjusts values to:

```
[4,3,2,1]
```

This case is important because it demonstrates why a single directional pass is insufficient.

### Equal Adjacent Ratings

Consider:

```
ratings = [1,2,2]
```

Children with equal ratings do not need different candy counts. The algorithm uses strict greater than comparisons, not greater than or equal comparisons, which prevents unnecessary candy increases.

### Peaks and Valleys

Consider:

```
ratings = [1,3,2,2,1]
```

The child with rating `3` must have more candies than both neighbors. The two pass approach correctly combines left and right constraints to handle these local peaks and valleys without over allocating candies.
