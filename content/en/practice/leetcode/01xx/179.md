## Solution

## Problem Understanding

The problem gives an array of non-negative integers and asks us to rearrange the numbers so that their concatenation forms the largest possible value.

The important detail is that we are not performing arithmetic addition or multiplication. Instead, we are deciding the order in which numbers should appear when converted to strings and joined together.

For example, if the input is:

```
[10, 2]
```

there are only two possible concatenations:

```
"102"
"210"
```

Since `"210"` is larger, the correct output is:

```
"210"
```

The challenge comes from determining the correct ordering rule. A normal descending numeric sort does not work. Consider:

```
[3, 30]
```

A descending numeric sort would place `30` before `3`, producing:

```
"303"
```

However:

```
"330" > "303"
```

so `3` must come before `30`.

The input constraints are relatively small:

- Up to 100 numbers
- Each number can be as large as `10^9`

Because the result may contain many digits, the problem requires returning a string instead of an integer.

Several edge cases are especially important:

- Arrays containing only zeros, such as `[0,0]`
- Numbers where one is the prefix of another, such as `3` and `30`
- Large combinations where local ordering decisions affect the global result

The key difficulty is defining the correct comparison rule for sorting.

## Approaches

### Brute Force Approach

The brute force solution tries every possible permutation of the array.

For each permutation:

1. Convert all numbers to strings
2. Concatenate them
3. Compare the result against the current maximum
4. Keep the largest string found

This works because it exhaustively checks every possible arrangement, guaranteeing the optimal answer.

However, the number of permutations grows factorially:

```
n!
```

For even moderately sized inputs, this becomes infeasible. With `n = 100`, the number of permutations is astronomically large.

Therefore, brute force is not practical.

### Optimal Approach, Custom Sorting

The key insight is that we can determine the correct order between any two numbers independently.

Suppose we have two numbers:

```
a = "3"
b = "30"
```

We need to decide whether `a` should come before `b`, or vice versa.

Instead of comparing the numbers directly, we compare:

```
a + b
b + a
```

For this example:

```
"330"
"303"
```

Since `"330"` is larger, `"3"` should come before `"30"`.

This comparison rule completely determines the correct ordering.

Why does this work?

Because the final answer is built through concatenation. The contribution of two elements depends entirely on which combined arrangement creates a larger lexicographical value.

Thus, we sort all numbers using the comparator:

```
a comes before b if (a+b) > (b+a)
```

After sorting:

1. Concatenate all strings
2. Handle the special case where the result starts with `'0'`
3. Return the final string

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n! × n) | O(n) | Generates every permutation |
| Optimal | O(n log n × k) | O(n) | Uses custom sorting comparator |

Here, `k` represents the average number of digits involved in string comparisons.

## Algorithm Walkthrough

1. Convert every integer into a string.

String concatenation is the core operation in this problem, so working directly with strings simplifies the logic.
2. Define a custom comparison rule.

For two strings `a` and `b`, compare:

```
a + b
b + a
```

If `a + b` is larger, then `a` should appear before `b`.
3. Sort the array using this comparator.

The sorting process ensures every pair of adjacent elements is ordered optimally for the final concatenated result.
4. Concatenate the sorted strings.

After sorting, joining all elements together produces the maximum possible number.
5. Handle the all-zero case.

Consider:

```
[0,0,0]
```

After sorting and concatenation, we get:

```
"000"
```

The correct answer should be:

```
"0"
```

Therefore, if the first character of the final string is `'0'`, return `"0"`.
6. Return the final result string.

## Python Solution

```
from typing import List
from functools import cmp_to_key

class Solution:
    def largestNumber(self, nums: List[int]) -> str:
        numbers = [str(num) for num in nums]

        def compare(a: str, b: str) -> int:
            if a + b > b + a:
                return -1
            elif a + b < b + a:
                return 1
            return 0

        numbers.sort(key=cmp_to_key(compare))

        result = "".join(numbers)

        if result[0] == "0":
            return "0"

        return result
```

The implementation begins by converting every integer into a string because concatenation comparisons are easier with strings than integers.

Python's built-in sorting does not directly support a custom comparator function, so we use `cmp_to_key` from the `functools` module. This converts the comparator into a sortable key object.

The comparator checks:

```
a + b
b + a
```

If `a + b` is larger, then `a` should appear earlier in the sorted order.

After sorting, all strings are joined into one final string.

The last check handles the special case where all numbers are zero. If the largest element is `"0"`, then the entire number must also be zero.

## Go Solution

```
package main

import (
	"sort"
	"strconv"
	"strings"
)

func largestNumber(nums []int) string {
	strs := make([]string, len(nums))

	for i, num := range nums {
		strs[i] = strconv.Itoa(num)
	}

	sort.Slice(strs, func(i, j int) bool {
		return strs[i]+strs[j] > strs[j]+strs[i]
	})

	result := strings.Join(strs, "")

	if result[0] == '0' {
		return "0"
	}

	return result
}
```

The Go solution follows the same logic as the Python implementation.

Instead of `cmp_to_key`, Go provides `sort.Slice`, which accepts a custom comparison function directly.

Strings are built using `strconv.Itoa`, and concatenation uses `strings.Join`.

The all-zero edge case is handled identically by checking whether the first character is `'0'`.

Because Go strings are immutable, repeated concatenation inside loops can be expensive, so `strings.Join` is the preferred approach.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n × k) | Sorting dominates, each comparison may examine up to `k` digits |
| Space | O(n) | Stores string representations of the numbers |

The dominant operation is sorting. A comparison between two numbers requires concatenating and comparing strings, which costs proportional to the number of digits involved.

Since sorting requires `O(n log n)` comparisons, the total complexity becomes:

```
O(n log n × k)
```

The space complexity is `O(n)` because we store the string versions of the input numbers.

## Edge Cases

### All Numbers Are Zero

Consider:

```
[0,0,0]
```

A naive implementation may return:

```
"000"
```

which is incorrect.

After sorting, if the first element is `"0"`, then every element must also be zero. The implementation detects this case and returns exactly:

```
"0"
```

### Prefix Relationships Between Numbers

Inputs like:

```
[3,30]
```

are tricky because standard numeric sorting fails.

A descending numeric sort produces:

```
"303"
```

but the correct answer is:

```
"330"
```

The custom comparator solves this by comparing concatenated combinations instead of raw numeric values.

### Very Large Concatenated Result

The resulting number may contain hundreds of digits.

For example:

```
100 numbers × up to 10 digits each
```

can produce a string with around 1000 digits.

This exceeds normal integer ranges in many languages. Returning a string avoids overflow problems entirely.

### Single Element Array

If the input contains only one number:

```
[7]
```

the algorithm still works correctly.

The sorting step is trivial, and the final concatenated result is simply:

```
"7"
```

No special handling is required.
