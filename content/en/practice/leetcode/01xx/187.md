## Solution

## Problem Understanding

The problem gives us a DNA string consisting only of the four characters `'A'`, `'C'`, `'G'`, and `'T'`. We must find every substring of length `10` that appears more than once in the string.

A substring is a contiguous sequence of characters. For example, in the string `"AAAAACCCCCAAAAA"`, the substring `"AAAAACCCCC"` appears starting at index `0`. If the same 10-character substring appears again somewhere else in the string, it should be included in the output.

The important detail is that we only care about substrings of exactly length `10`. We are not looking for longer repeated regions unless they contain repeated 10-character windows.

The input size can be as large as `10^5`, which means an inefficient solution can easily become too slow. Since every possible starting position of a 10-character substring must be checked, there are approximately `n` windows to process. The challenge is to detect duplicates efficiently.

The output may be returned in any order, which means we do not need to preserve the order in which repeated sequences are discovered.

Several edge cases are important:

- If the string length is less than `10`, there are no valid substrings of length `10`, so the answer is empty.
- A substring may appear many times, but it should appear only once in the output.
- Overlapping substrings count as separate occurrences. For example, `"AAAAAAAAAAAAA"` contains multiple overlapping `"AAAAAAAAAA"` windows.
- Because the alphabet contains only four characters, the problem is well suited for hashing or bit encoding optimizations.

## Approaches

### Brute Force Approach

The most direct solution is to examine every substring of length `10` and count how many times it appears.

We iterate through the string from index `0` to `n - 10`. For each position, we extract the substring of length `10`. We store the frequency of each substring in a hash map. After processing all substrings, we collect the ones whose frequency is greater than `1`.

This works because every valid 10-character sequence is examined exactly once, and the frequency map correctly records how many times each sequence appears.

Although this approach is already acceptable for the given constraints because substring length is fixed at `10`, it still repeatedly creates substring objects. In some languages or implementations, substring creation may add overhead.

### Key Insight for an Optimal Solution

The key observation is that:

- Every substring has fixed length `10`
- The alphabet size is only `4`

Since there are only four possible characters, each character can be encoded using only `2` bits:

| Character | Binary |
| --- | --- |
| A | 00 |
| C | 01 |
| G | 10 |
| T | 11 |

A 10-character sequence therefore requires only `20` bits total.

Instead of storing full strings repeatedly, we can maintain a rolling integer hash representing the current 10-character window. As the window slides forward:

- Remove the leftmost character contribution
- Shift bits left
- Add the new character

This creates a compact rolling hash solution with efficient updates and reduced memory overhead.

## Algorithm Walkthrough

### Optimal Rolling Hash Algorithm

1. First, check whether the string length is smaller than `10`.

If the length is less than `10`, no valid substring exists, so return an empty list immediately.
2. Create a mapping from DNA characters to their 2-bit representations.

This allows every 10-character sequence to be stored compactly as an integer.
3. Maintain a rolling hash integer representing the current 10-character window.

As we process characters:

- Shift the current hash left by `2` bits
- Add the encoded value of the new character
- Keep only the last `20` bits using a bitmask
4. Use a hash set called `seen` to store hashes that have already appeared.

If a hash is encountered for the first time, add it to `seen`.
5. Use another hash set called `repeated` to track hashes already added to the answer.

This prevents duplicate entries in the final result.
6. Once the window size reaches `10`, check whether the current hash has been seen before.

- If yes, add the substring to the answer if it has not already been recorded.
- Otherwise, insert the hash into `seen`.
7. Continue sliding the window until the end of the string.
8. Return the collected repeated sequences.

The rolling hash works efficiently because each new window differs from the previous one by only one character. Instead of rebuilding the substring representation from scratch, we update the hash incrementally.

## Python Solution

```
from typing import List

class Solution:
    def findRepeatedDnaSequences(self, s: str) -> List[str]:
        sequence_length = 10
        
        if len(s) < sequence_length:
            return []

        encoding = {
            'A': 0,
            'C': 1,
            'G': 2,
            'T': 3
        }

        seen = set()
        repeated = set()
        result = []

        rolling_hash = 0
        mask = (1 << (2 * sequence_length)) - 1

        for index, char in enumerate(s):
            rolling_hash = (
                (rolling_hash << 2) | encoding[char]
            ) & mask

            if index >= sequence_length - 1:
                if rolling_hash in seen:
                    substring = s[index - sequence_length + 1:index + 1]

                    if substring not in repeated:
                        repeated.add(substring)
                        result.append(substring)
                else:
                    seen.add(rolling_hash)

        return result
```

The implementation begins by handling the smallest edge case. If the string length is smaller than `10`, the method immediately returns an empty list.

The `encoding` dictionary converts each DNA character into a compact 2-bit integer representation. This enables the rolling hash technique.

The variable `rolling_hash` stores the encoded representation of the current 10-character window. Each iteration shifts the existing bits left by `2` positions and appends the new character bits.

The bitmask ensures that only the most recent `20` bits are retained. Since each character uses `2` bits and the window length is `10`, the current window always fits inside `20` bits.

The `seen` set tracks hashes already encountered. When a hash appears again, the corresponding substring is repeated. The `repeated` set ensures that duplicate answers are not inserted multiple times.

Finally, the collected repeated sequences are returned.

## Go Solution

```
func findRepeatedDnaSequences(s string) []string {
	sequenceLength := 10

	if len(s) < sequenceLength {
		return []string{}
	}

	encoding := map[byte]int{
		'A': 0,
		'C': 1,
		'G': 2,
		'T': 3,
	}

	seen := make(map[int]bool)
	repeated := make(map[string]bool)

	result := []string{}

	rollingHash := 0
	mask := (1 << (2 * sequenceLength)) - 1

	for i := 0; i < len(s); i++ {
		rollingHash = ((rollingHash << 2) | encoding[s[i]]) & mask

		if i >= sequenceLength-1 {
			if seen[rollingHash] {
				substring := s[i-sequenceLength+1 : i+1]

				if !repeated[substring] {
					repeated[substring] = true
					result = append(result, substring)
				}
			} else {
				seen[rollingHash] = true
			}
		}
	}

	return result
}
```

The Go implementation follows the same rolling hash logic as the Python version.

A `map[int]bool` is used instead of a Python set for tracking seen hashes. Another map tracks substrings already included in the result.

Go string slicing is efficient because slices reference the underlying string data instead of copying entire substrings immediately.

The hash fits safely within a standard integer because only `20` bits are required.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each character is processed once |
| Space | O(n) | Hash sets may store many unique sequences |

The algorithm processes the string in a single pass. Each iteration performs constant-time operations, including bit shifting, masking, and hash set lookups.

The space complexity is linear because we may store up to `n` distinct rolling hashes and repeated substrings in the worst case.

## Edge Cases

### Input Length Smaller Than 10

If the string length is less than `10`, there cannot possibly be any valid substring of length `10`.

For example:

```
s = "ACGT"
```

A naive implementation might still attempt substring extraction and produce indexing bugs. The solution avoids this by immediately returning an empty list when `len(s) < 10`.

### Multiple Overlapping Repetitions

Consider:

```
s = "AAAAAAAAAAAAA"
```

The repeated substring `"AAAAAAAAAA"` overlaps heavily with itself. Some implementations mistakenly assume repeated substrings must be disjoint.

The sliding window correctly processes overlapping windows independently, so the repeated sequence is detected properly.

### Same Sequence Appearing Many Times

Consider:

```
s = "AAAAAAAAAAAAAAA"
```

The substring `"AAAAAAAAAA"` appears multiple times. A buggy solution might add the same substring repeatedly to the result.

The implementation uses the `repeated` set to ensure each repeated sequence appears only once in the final output.

### Large Input Size

The input can contain up to `100000` characters. A quadratic solution would be too slow.

The rolling hash solution processes each character once and performs only constant-time operations per iteration, making it efficient even for the largest valid inputs.
