## Solution

## Problem Understanding

This problem asks us to determine whether a singly linked list contains a cycle, and if it does, return the exact node where the cycle begins.

A linked list normally ends when a node’s `next` pointer becomes `null`. In this problem, however, the tail node may point back to an earlier node in the list, creating a loop. Once traversal enters this loop, continuing to follow `next` pointers will never terminate.

The input is the `head` pointer of a singly linked list. The variable `pos` mentioned in the problem statement is only used internally by LeetCode to construct the test case. It tells us which node the tail connects to, but it is not available to our function. Our implementation must discover the cycle entirely by examining the linked list structure itself.

The expected output is:

- The node where the cycle starts, if a cycle exists
- `null` (or `nil` in Go) if there is no cycle

An important detail is that we must return the actual node reference, not the index of the node and not just a boolean indicating whether a cycle exists.

The constraints tell us that the list can contain up to `10^4` nodes. This size is small enough for an `O(n)` solution to run comfortably, but large enough that inefficient quadratic approaches should be avoided.

The problem also explicitly asks for an `O(1)` memory solution in the follow up. This strongly hints that the intended optimal approach is Floyd’s Cycle Detection Algorithm, also called the tortoise and hare algorithm.

Several edge cases are important:

- The list may be empty
- The list may contain only one node
- A single node may point to itself, forming a cycle
- The cycle may begin at the head
- The cycle may begin somewhere in the middle
- There may be no cycle at all

A correct solution must handle all of these cases safely without modifying the linked list.

## Approaches

### Brute Force Approach

The most direct approach is to keep track of every node we visit using a hash set.

As we traverse the linked list, we check whether the current node has already been seen before. If it has, then we know we have re-entered the cycle, and that node is the beginning of the cycle. If we reach `null`, then the list has no cycle.

This works because node references are unique. The first repeated node encountered during traversal must be the cycle entry point.

The algorithm is straightforward and easy to reason about, but it uses additional memory proportional to the number of visited nodes. Since the follow up specifically asks for constant memory, this approach is not optimal.

### Optimal Approach, Floyd’s Cycle Detection Algorithm

The key insight is that two pointers moving at different speeds can detect a cycle without extra memory.

We use:

- A slow pointer that moves one step at a time
- A fast pointer that moves two steps at a time

If there is no cycle, the fast pointer eventually reaches `null`.

If there is a cycle, the fast pointer eventually laps the slow pointer and they meet somewhere inside the cycle.

The most interesting part of the algorithm is determining the cycle entry point after the meeting occurs.

Suppose:

- `a` is the distance from the head to the cycle entry
- `b` is the distance from the cycle entry to the meeting point
- `c` is the remaining distance around the cycle

When the pointers meet:

- Slow pointer traveled `a + b`
- Fast pointer traveled `a + b + k(b + c)` for some integer `k`

Since the fast pointer moves twice as fast:

`2(a + b) = a + b + k(b + c)`

Rearranging:

`a = k(b + c) - b`

This means that if we place one pointer at the head and another at the meeting point, then move both one step at a time, they will meet exactly at the cycle entry.

This gives us an `O(n)` time and `O(1)` space solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Uses a hash set to track visited nodes |
| Optimal | O(n) | O(1) | Uses Floyd’s two pointer cycle detection algorithm |

## Algorithm Walkthrough

1. Initialize two pointers, `slow` and `fast`, both starting at the head of the list.
2. Move `slow` one step at a time and `fast` two steps at a time. This phase determines whether a cycle exists.
3. Continue until either:

- `fast` reaches `null`, meaning no cycle exists
- `slow` and `fast` point to the same node, meaning a cycle exists
4. If no cycle exists, return `null`.
5. Once the pointers meet inside the cycle, initialize a new pointer called `entry` at the head of the list.
6. Move both `entry` and `slow` one step at a time. The important insight is that these two pointers will meet at the cycle entry node.
7. Return the node where they meet.

The algorithm works because the mathematical relationship between the pointer distances guarantees that the distance from the head to the cycle start equals the distance from the meeting point to the cycle start.

## Python Solution

```
# Definition for singly-linked list.
# class ListNode:
#     def __init__(self, x):
#         self.val = x
#         self.next = None

class Solution:
    def detectCycle(self, head: Optional[ListNode]) -> Optional[ListNode]:
        slow = head
        fast = head

        # Phase 1: Detect whether a cycle exists
        while fast and fast.next:
            slow = slow.next
            fast = fast.next.next

            if slow == fast:
                # Phase 2: Find the cycle entry
                entry = head

                while entry != slow:
                    entry = entry.next
                    slow = slow.next

                return entry

        return None
```

The implementation is divided naturally into two phases.

The first phase detects whether a cycle exists. The `slow` pointer advances one node at a time, while the `fast` pointer advances two nodes at a time. If the list contains a cycle, the faster pointer must eventually catch up to the slower one.

The condition `while fast and fast.next` ensures we never dereference a `None` pointer. If either `fast` or `fast.next` becomes `None`, then the list terminates normally and no cycle exists.

Once `slow` and `fast` meet, we know we are somewhere inside the cycle. At this point, we create a new pointer called `entry` starting at the head.

Both `entry` and `slow` move one step at a time. Their meeting point is guaranteed to be the cycle entry node because of the mathematical relationship derived from the pointer distances.

Finally, we return the cycle entry node. If no cycle exists, the function returns `None`.

## Go Solution

```
/**
 * Definition for singly-linked list.
 * type ListNode struct {
 *     Val int
 *     Next *ListNode
 * }
 */
func detectCycle(head *ListNode) *ListNode {
    slow := head
    fast := head

    // Phase 1: Detect cycle
    for fast != nil && fast.Next != nil {
        slow = slow.Next
        fast = fast.Next.Next

        if slow == fast {
            // Phase 2: Find cycle entry
            entry := head

            for entry != slow {
                entry = entry.Next
                slow = slow.Next
            }

            return entry
        }
    }

    return nil
}
```

The Go implementation follows the same logic as the Python version.

The primary language specific difference is the use of `nil` instead of `None`. Go also requires explicit pointer handling, but linked list node comparison works naturally because pointer equality checks whether two variables reference the same node in memory.

No special overflow handling is needed because we are only manipulating pointers, not performing arithmetic on large integers.

## Worked Examples

### Example 1

Input:

```
3 -> 2 -> 0 -> -4
     ^         |
     |_________|
```

Cycle begins at node with value `2`.

### Phase 1, Detect Cycle

| Iteration | slow | fast |
| --- | --- | --- |
| Start | 3 | 3 |
| 1 | 2 | 0 |
| 2 | 0 | 2 |
| 3 | -4 | -4 |

At iteration 3, `slow` and `fast` meet.

### Phase 2, Find Entry

Initialize:

| Pointer | Node |
| --- | --- |
| entry | 3 |
| slow | -4 |

Move both one step at a time:

| Step | entry | slow |
| --- | --- | --- |
| 1 | 2 | 2 |

They meet at node `2`, which is the cycle entry.

### Example 2

Input:

```
1 -> 2
^    |
|____|
```

Cycle begins at node `1`.

### Phase 1, Detect Cycle

| Iteration | slow | fast |
| --- | --- | --- |
| Start | 1 | 1 |
| 1 | 2 | 1 |
| 2 | 1 | 1 |

Pointers meet at node `1`.

### Phase 2, Find Entry

| Pointer | Node |
| --- | --- |
| entry | 1 |
| slow | 1 |

They already point to the same node, so the cycle entry is immediately found.

### Example 3

Input:

```
1 -> null
```

### Phase 1, Detect Cycle

| Iteration | slow | fast |
| --- | --- | --- |
| Start | 1 | 1 |

The loop condition fails because `fast.next` is `null`.

Therefore, no cycle exists, and the function returns `null`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each pointer traverses at most a linear number of nodes |
| Space | O(1) | Only a few pointers are used regardless of input size |

The time complexity is linear because each pointer moves through the list a limited number of times. Even in the presence of a cycle, the total number of pointer movements remains proportional to the number of nodes.

The space complexity is constant because the algorithm uses only a fixed number of pointer variables and does not allocate additional data structures.

## Edge Cases

### Empty List

If `head` is `None` or `nil`, there are no nodes to traverse and therefore no cycle.

A careless implementation might attempt to access `head.next` immediately and crash. This implementation avoids that problem because the loop condition checks `fast` and `fast.next` before advancing pointers.

### Single Node Without Cycle

Consider:

```
1 -> null
```

In this case, `fast.next` is already `null`, so the loop terminates immediately and the algorithm correctly returns `null`.

This case is important because some incorrect implementations assume at least two nodes exist.

### Single Node With Self Cycle

Consider:

```
1
^
|
|
```

The node points to itself.

After one iteration:

- `slow` moves to the same node
- `fast` also moves to the same node

The pointers meet immediately, and the second phase correctly identifies the node itself as the cycle entry.

### Cycle Starts at Head

Consider:

```
1 -> 2 -> 3
^         |
|_________|
```

Some implementations incorrectly assume the cycle starts later in the list. Floyd’s algorithm handles this naturally because the second phase works regardless of where the cycle begins.

### No Cycle In Long List

A long acyclic list tests whether the implementation safely handles termination conditions.

If the code fails to check `fast` and `fast.next` properly, it may dereference a null pointer. The loop guard prevents this issue entirely.
