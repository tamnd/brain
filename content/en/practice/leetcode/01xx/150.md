# LeetCode 150, Evaluate Reverse Polish Notation

## Problem Understanding

This problem asks us to evaluate an arithmetic expression written in Reverse Polish Notation, often abbreviated as RPN. In standard arithmetic notation, operators appear between operands, such as `2 + 3`. In Reverse Polish Notation, the operator appears after the operands, so the same expression becomes `2 3 +`.

The input is given as an array of strings called `tokens`. Each token is either:

- An integer value
- One of the four arithmetic operators: `+`, `-`, `*`, `/`

The expression is guaranteed to be valid. That means every operator will always have enough operands available before it appears, and the overall expression will produce exactly one final result.

For example:

```
["2","1","+","3","*"]
```

represents:

```
((2 + 1) * 3)
```

The evaluation order is naturally determined by the structure of the notation, so parentheses are unnecessary.

A very important detail in this problem is integer division. Division must truncate toward zero. This matters especially for negative numbers. For example:

```
6 / -132 = 0
```

because truncating toward zero removes the fractional part.

The constraints tell us that:

- The number of tokens can be as large as `10^4`
- Each token is small, but intermediate computations can still grow
- All intermediate and final results fit within a 32-bit signed integer

Since the input size can be large, we need an efficient linear-time solution.

There are several edge cases worth considering:

- Expressions containing only one number
- Negative numbers, such as `"-11"`
- Division involving negative operands
- Long expressions with many nested operations
- Correct operand ordering for subtraction and division

Operand ordering is especially important. For subtraction and division, the first popped value is actually the second operand.

For example:

```
["4","2","-"]
```

means:

```
4 - 2
```

not:

```
2 - 4
```

so careful stack handling is required.

## Approaches

### Brute Force Approach

A brute force approach would repeatedly scan the token list looking for patterns of:

```
[number, number, operator]
```

Whenever such a pattern is found, we compute the result and replace those three tokens with the computed value.

For example:

```
["2","1","+","3","*"]
```

would become:

```
["3","3","*"]
```

and then:

```
["9"]
```

This approach works because every valid RPN expression eventually reduces into a single value after repeatedly evaluating valid subexpressions.

However, this method is inefficient because each reduction may require scanning the list again, and removing or inserting elements inside arrays is costly. In the worst case, the total work becomes quadratic.

### Optimal Approach, Stack

The key observation is that Reverse Polish Notation is naturally evaluated using a stack.

Whenever we encounter a number, we store it because it may be used later by an operator.

Whenever we encounter an operator:

1. The two most recent numbers are the operands
2. We pop them from the stack
3. Apply the operator
4. Push the result back onto the stack

This works because RPN always places operators after their operands. The stack maintains exactly the values that are still waiting to participate in future computations.

Each token is processed exactly once, making the solution linear in time complexity.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeatedly scans and modifies the token list |
| Optimal | O(n) | O(n) | Uses a stack to evaluate expression in one pass |

## Algorithm Walkthrough

1. Create an empty stack to store intermediate numbers.
2. Iterate through each token in the input array.
3. If the token is a number, convert it to an integer and push it onto the stack. We store numbers because future operators may need them.
4. If the token is an operator, pop the top two values from the stack. The first popped value is the right operand, and the second popped value is the left operand.
5. Apply the operation:

- `+` performs addition
- `-` performs subtraction
- `*` performs multiplication
- `/` performs division with truncation toward zero
6. Push the computed result back onto the stack because it may participate in later operations.
7. Continue processing until all tokens are consumed.
8. After processing the entire expression, the stack contains exactly one value, which is the final answer.

The stack is the ideal data structure here because the most recently seen operands are always the ones needed next. This follows the Last In, First Out behavior of stacks.

## Python Solution

```
from typing import List

class Solution:
    def evalRPN(self, tokens: List[str]) -> int:
        stack = []

        for token in tokens:
            if token not in {"+", "-", "*", "/"}:
                stack.append(int(token))
            else:
                right = stack.pop()
                left = stack.pop()

                if token == "+":
                    stack.append(left + right)
                elif token == "-":
                    stack.append(left - right)
                elif token == "*":
                    stack.append(left * right)
                else:
                    stack.append(int(left / right))

        return stack[-1]
```

The implementation begins by initializing an empty stack.

As we iterate through the tokens, we first determine whether the current token is an operator or a number. If it is a number, we convert it into an integer and push it onto the stack.

When we encounter an operator, we remove the top two values from the stack. The order is important. The first popped value becomes the right operand, while the second popped value becomes the left operand.

We then apply the operator and push the result back onto the stack.

For division, Python's `//` operator is not used because it rounds downward for negative numbers. The problem specifically requires truncation toward zero. Using:

```
int(left / right)
```

ensures the correct behavior.

At the end of processing, the stack contains exactly one value, which is returned as the final answer.

## Go Solution

```
import "strconv"

func evalRPN(tokens []string) int {
    stack := []int{}

    for _, token := range tokens {
        switch token {
        case "+", "-", "*", "/":
            right := stack[len(stack)-1]
            stack = stack[:len(stack)-1]

            left := stack[len(stack)-1]
            stack = stack[:len(stack)-1]

            var result int

            switch token {
            case "+":
                result = left + right
            case "-":
                result = left - right
            case "*":
                result = left * right
            case "/":
                result = left / right
            }

            stack = append(stack, result)

        default:
            number, _ := strconv.Atoi(token)
            stack = append(stack, number)
        }
    }

    return stack[len(stack)-1]
}
```

The Go solution follows the same stack-based logic as the Python version.

Go integer division already truncates toward zero, so no extra conversion logic is needed for division.

Slices are used to implement the stack. Pushing is done with `append`, and popping is done by shortening the slice:

```
stack = stack[:len(stack)-1]
```

The problem guarantees valid input, so the conversion error from `strconv.Atoi` can safely be ignored.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each token is processed exactly once |
| Space | O(n) | The stack may store up to n operands |

The algorithm performs a single pass through the input array. Every token results in constant-time work, either pushing onto the stack or performing one arithmetic operation.

The stack can grow proportionally to the number of operands in the expression, so the worst-case space complexity is linear.

## Edge Cases

### Single Number Input

An expression may contain only one number:

```
["42"]
```

In this case, no operations are performed. The algorithm simply pushes the number onto the stack and returns it. The implementation naturally handles this without requiring special logic.

### Negative Numbers

Tokens may contain negative integers such as:

```
["4","-2","/"]
```

A common bug is confusing negative numbers with the subtraction operator. The implementation avoids this issue by checking whether the token exactly matches one of the four operators. Any other token is treated as a number and converted with `int()` or `strconv.Atoi`.

### Division with Negative Values

Division behavior is particularly important:

```
["7","-3","/"]
```

The correct result is:

```
-2
```

because division truncates toward zero.

In Python, using floor division:

```
7 // -3
```

would incorrectly produce:

```
-3
```

The implementation correctly uses:

```
int(left / right)
```

to satisfy the problem requirements.

### Operand Order for Subtraction and Division

Subtraction and division are not commutative. The order of operands matters.

For example:

```
["5","2","-"]
```

must evaluate to:

```
3
```

not:

```
-3
```

The implementation correctly pops the right operand first, then the left operand, preserving the proper evaluation order.
