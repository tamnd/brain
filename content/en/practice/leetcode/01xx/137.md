## Solution

## Problem Understanding

This problem asks us to find the one unique number in an array where every other number appears **exactly three times**.

In simpler terms, the input array contains many repeated values, but there is one special value that appears only once. Every other value appears three times, no more and no less. Our goal is to return that unique value.

For example, in `nums = [2,2,3,2]`, the number `2` appears three times, while `3` appears once. Therefore, the answer is `3`.

The key challenge comes from the constraints. The problem explicitly requires:

1. **Linear runtime complexity**, meaning the algorithm should ideally run in `O(n)` time.
2. **Constant extra space**, meaning we cannot use additional memory proportional to the input size.

A straightforward frequency counting solution using a hash map would easily identify the unique number, but it requires `O(n)` extra space, violating the problem requirement. This constraint strongly suggests that we need to exploit some mathematical or bit manipulation property of the numbers.

The constraints also tell us useful information about the input:

- The array length can be as large as `3 * 10^4`, so inefficient approaches such as nested loops may become expensive.
- Values may be negative, ranging from `-2^31` to `2^31 - 1`, which means the implementation must correctly handle signed integers.
- The problem guarantees there is exactly one valid answer, meaning we do not need to handle ambiguous cases.

Several edge cases are important to keep in mind. The unique number could be negative, which creates complications when using bit operations due to two's complement representation. The array could contain only one element, in which case that element must be returned immediately. The unique number may also be `0`, which can expose bugs in bit counting logic if not handled carefully.

## Approaches

### Brute Force Approach, Frequency Counting

The most intuitive approach is to count how many times each number appears using a hash map.

We iterate through the array and maintain a frequency count for every number. After building the map, we scan through it to find the element whose count is `1`.

This works because the problem guarantees that every number except one appears exactly three times. Once all frequencies are known, identifying the unique value becomes trivial.

However, this solution violates the constant space requirement because the hash map may grow proportionally to the number of distinct elements. In the worst case, this requires `O(n)` additional memory.

### Optimal Approach, Bit Manipulation with Bit Counting

The key observation is that numbers appearing three times create a predictable pattern in their binary representation.

Consider examining each bit position independently. Since every repeated number appears exactly three times, the contribution of repeated elements to any bit position will always be a multiple of `3`.

Suppose we count how many numbers have the `i-th` bit set:

- If the total count is divisible by `3`, then the unique number does **not** contribute a `1` at that position.
- If the remainder is `1`, then the unique number **does** contribute a `1`.

By reconstructing the result bit by bit, we can recover the unique number.

For example:

```
nums = [2,2,3,2]

Binary:

2 = 0010
2 = 0010
3 = 0011
2 = 0010

Bit counts:
bit 0: 1
bit 1: 4

Modulo 3:
bit 0: 1 % 3 = 1
bit 1: 4 % 3 = 1

Result = 0011 = 3
```

Because integers are represented using 32 bits, we only need to examine all 32 bit positions. This ensures linear runtime and constant extra memory.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | `O(n)` | `O(n)` | Uses a hash map to count frequencies |
| Optimal | `O(32n)` = `O(n)` | `O(1)` | Counts set bits and reconstructs the answer |

## Algorithm Walkthrough

1. Initialize a variable called `result` to `0`. This variable will store the reconstructed unique number.
2. Iterate through all `32` bit positions from `0` to `31`. We use `32` because integers are bounded by 32-bit signed representation.
3. For each bit position, count how many numbers in the array have that bit set.

To check whether a bit is set, use:

```
(num >> bit) & 1
```

This shifts the desired bit to the least significant position and extracts it.
4. Compute the bit count modulo `3`.

Since every repeated number contributes bits exactly three times, their effect disappears after `% 3`.

If:

```
bit_count % 3 == 1
```

then the unique number must contain that bit.
5. Set the corresponding bit in `result`.

This is done using bitwise OR:

```
result |= (1 << bit)
```
6. Handle negative numbers.

Since Python integers are not fixed to 32 bits, reconstructing a negative number produces a large unsigned integer instead of a negative value.

If the sign bit, bit `31`, is set, convert the number back into signed representation:

```
if result >= 2**31:
    result -= 2**32
```
7. Return the final reconstructed number.

## Python Solution

```
from typing import List

class Solution:
    def singleNumber(self, nums: List[int]) -> int:
        result = 0

        for bit in range(32):
            bit_count = 0

            for num in nums:
                bit_count += (num >> bit) & 1

            if bit_count % 3:
                result |= (1 << bit)

        # Handle negative numbers
        if result >= (1 << 31):
            result -= (1 << 32)

        return result
```

The implementation follows the bit counting strategy directly.

We begin by initializing `result` to `0`. Then we iterate through all `32` bit positions. For every bit position, we count how many numbers contain a `1` in that position.

The expression:

```
(num >> bit) & 1
```

extracts the bit value at the current position. We accumulate these values into `bit_count`.

After counting, we compute `bit_count % 3`. If the remainder is non-zero, it means the unique number contributes a set bit at that position. We update `result` using bitwise OR.

Finally, we correct negative numbers. Because Python integers do not have a fixed bit width, numbers reconstructed from 32-bit patterns may appear as large positive integers. We convert them back into signed form by subtracting `2^32` when necessary.

## Go Solution

```
func singleNumber(nums []int) int {
	result := 0

	for bit := 0; bit < 32; bit++ {
		bitCount := 0

		for _, num := range nums {
			bitCount += (num >> bit) & 1
		}

		if bitCount%3 != 0 {
			result |= (1 << bit)
		}
	}

	// Handle negative numbers
	if result >= (1 << 31) {
		result -= (1 << 32)
	}

	return result
}
```

The Go implementation closely mirrors the Python solution.

One important difference is integer behavior. Go integers are fixed-width depending on the architecture, unlike Python's arbitrary precision integers. We still explicitly handle the sign bit to ensure correct reconstruction of negative values.

Since slices in Go naturally handle empty-like behavior, no special handling is needed. The problem guarantees at least one element exists, so we never encounter an invalid input case.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | `O(n)` | We iterate through 32 bit positions, and for each bit scan the array once. Since 32 is constant, this simplifies to `O(n)` |
| Space | `O(1)` | Only a few integer variables are used, regardless of input size |

Although the algorithm technically performs `32 × n` operations, the number `32` is fixed and independent of input size. Therefore, it simplifies to linear time complexity. The extra memory usage remains constant because we do not allocate any data structures proportional to the input size.

## Edge Cases

### Single Element Array

An input such as:

```
[7]
```

contains only one number, meaning that number must be the unique value. A naive implementation might overcomplicate this case, but our bit counting approach naturally handles it because the single number contributes its bits directly to the result.

### Negative Unique Number

Consider:

```
[-4, 1, 1, 1]
```

Negative integers are stored using two's complement representation. Without explicit sign handling, reconstructing the bits would produce a very large positive integer instead of `-4`.

Our implementation fixes this by checking whether the sign bit, bit `31`, is set. If it is, we subtract `2^32` to convert back into signed representation.

### Unique Number is Zero

Consider:

```
[5,5,5,0]
```

Zero has no set bits, which can expose mistakes in bit reconstruction logic. If the modulo logic is incorrect, the algorithm might accidentally reconstruct an invalid number.

Our implementation correctly returns `0` because every bit count becomes divisible by `3`, leaving `result` unchanged.

### Large Integer Values

The input may contain values close to:

```
2^31 - 1
```

or

```
-2^31
```

These boundary values can cause overflow concerns in some languages. By explicitly processing only 32 bits and carefully handling the sign bit, both the Python and Go implementations correctly reconstruct extreme integer values.
