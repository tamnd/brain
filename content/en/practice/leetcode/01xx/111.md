## Solution

## Problem Understanding

The problem asks us to compute the minimum depth of a binary tree. A binary tree consists of nodes where each node may have a left child, a right child, both children, or no children at all. The minimum depth is defined as the number of nodes along the shortest path from the root node to the nearest leaf node.

A leaf node is very important in this definition. A node is considered a leaf only if it has no left child and no right child. This detail is the source of many incorrect solutions. A node with only one child is not a leaf, so we cannot stop counting depth at that node.

The input is given as the root node of a binary tree. Each node contains a value and pointers to its left and right children. The output is a single integer representing the minimum depth.

The constraints tell us that the tree can contain up to `10^5` nodes. This means the solution must be efficient and ideally run in linear time. Since the tree can be very large, recursive depth-first traversal may also risk stack overflow in some languages if the tree becomes highly skewed.

There are several important edge cases to consider:

- The tree may be empty. In that case, the minimum depth is `0`.
- The tree may contain only one node. Since the root is also a leaf, the answer is `1`.
- The tree may be heavily skewed, such as a linked-list-shaped tree. In that case, the minimum depth equals the number of nodes.
- A node with only one child must not be treated as having a depth contribution from the missing child. This is the most common bug in naive implementations.

For example, consider this tree:

```
    1
     \
      2
```

The minimum depth is `2`, not `1`. Even though the left child is missing, the root is not a leaf because it still has a right child.

## Approaches

### Brute Force Approach

A brute-force approach would recursively compute the depth of every possible root-to-leaf path and then return the smallest value. Conceptually, this means traversing the entire tree while tracking all depths.

One naive implementation mistake is to use:

```
1 + min(leftDepth, rightDepth)
```

without handling missing children correctly. If one child is `null`, its depth becomes `0`, which incorrectly causes the algorithm to choose the missing path.

To fix this, the brute-force recursive solution must explicitly distinguish between:

- nodes with two children,
- nodes with one child,
- and leaf nodes.

This recursive traversal correctly explores all root-to-leaf paths, but it still visits every node in the tree. While this is acceptable for the problem constraints, recursion depth may become an issue for highly skewed trees.

### Key Insight for the Optimal Solution

The key observation is that we only care about the first leaf node encountered at the shallowest level.

Breadth-First Search, BFS, is naturally suited for this problem because BFS explores the tree level by level. The first time we encounter a leaf node, we know it must be the nearest leaf. There is no need to continue searching deeper levels.

This gives us an efficient and elegant solution:

- Traverse the tree level by level using a queue.
- Track the current depth.
- As soon as a leaf node is found, return its depth immediately.

This approach guarantees correctness because BFS processes nodes in increasing order of depth.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force Recursive DFS | O(n) | O(h) | Traverses entire tree recursively, where `h` is tree height |
| Optimal BFS | O(n) | O(w) | Stops at first leaf node, where `w` is maximum tree width |

## Algorithm Walkthrough

### Optimal BFS Algorithm

1. First, check whether the root is `null`. If the tree is empty, return `0` immediately because there are no nodes.
2. Create a queue to perform Breadth-First Search. Each queue entry will store:

- the current node,
- and its depth from the root.
3. Insert the root node into the queue with depth `1`. The root itself counts as one level.
4. Start processing nodes level by level:

- Remove the front node from the queue.
- Check whether it is a leaf node.
5. A node is a leaf if:

- `node.left is None`
- and `node.right is None`

If both conditions are true, immediately return the current depth. Because BFS explores shallower levels first, this is guaranteed to be the minimum depth.
6. If the current node has a left child:

- add the left child to the queue,
- with depth increased by `1`.
7. If the current node has a right child:

- add the right child to the queue,
- with depth increased by `1`.
8. Continue until a leaf node is found.

The queue is the ideal data structure here because it naturally supports first-in-first-out processing, which is exactly what level-order traversal requires.

## Python Solution

```
from collections import deque
from typing import Optional

# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

class Solution:
    def minDepth(self, root: Optional[TreeNode]) -> int:
        if root is None:
            return 0

        queue = deque([(root, 1)])

        while queue:
            current_node, depth = queue.popleft()

            if current_node.left is None and current_node.right is None:
                return depth

            if current_node.left:
                queue.append((current_node.left, depth + 1))

            if current_node.right:
                queue.append((current_node.right, depth + 1))

        return 0
```

The implementation begins by handling the empty tree case. If the root is `None`, the function returns `0`.

A `deque` is used as the BFS queue because it provides efficient insertion and removal from both ends. Each queue entry stores both the node and its current depth.

The algorithm repeatedly removes nodes from the front of the queue. For each node, it checks whether the node is a leaf. If so, the current depth is returned immediately because BFS guarantees that this is the shallowest leaf.

If the node has children, they are inserted into the queue with an incremented depth value.

The final `return 0` is technically unreachable for valid inputs, but it satisfies Python function completeness.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */

type QueueNode struct {
    node  *TreeNode
    depth int
}

func minDepth(root *TreeNode) int {
    if root == nil {
        return 0
    }

    queue := []QueueNode{{root, 1}}

    for len(queue) > 0 {
        current := queue[0]
        queue = queue[1:]

        node := current.node
        depth := current.depth

        if node.Left == nil && node.Right == nil {
            return depth
        }

        if node.Left != nil {
            queue = append(queue, QueueNode{node.Left, depth + 1})
        }

        if node.Right != nil {
            queue = append(queue, QueueNode{node.Right, depth + 1})
        }
    }

    return 0
}
```

The Go implementation follows the same BFS strategy as the Python version.

Since Go does not have tuples, a custom `QueueNode` struct is created to store both the tree node and its associated depth.

The queue is implemented using a slice. Removing the front element is done with:

```
queue = queue[1:]
```

Go uses `nil` instead of `None` when checking for missing children or an empty root.

Integer overflow is not a concern here because the maximum depth is bounded by the number of nodes, which is at most `10^5`.

## Worked Examples

### Example 1

Input:

```
        3
       / \
      9  20
         / \
        15  7
```

Expected output:

```
2
```

### Step-by-Step BFS Traversal

| Queue State | Current Node | Depth | Leaf? | Action |
| --- | --- | --- | --- | --- |
| `[(3,1)]` | 3 | 1 | No | Add 9 and 20 |
| `[(9,2),(20,2)]` | 9 | 2 | Yes | Return 2 |

The algorithm stops immediately after reaching node `9` because it is the first leaf encountered.

### Example 2

Input:

```
    2
     \
      3
       \
        4
         \
          5
           \
            6
```

Expected output:

```
5
```

### Step-by-Step BFS Traversal

| Queue State | Current Node | Depth | Leaf? | Action |
| --- | --- | --- | --- | --- |
| `[(2,1)]` | 2 | 1 | No | Add 3 |
| `[(3,2)]` | 3 | 2 | No | Add 4 |
| `[(4,3)]` | 4 | 3 | No | Add 5 |
| `[(5,4)]` | 5 | 4 | No | Add 6 |
| `[(6,5)]` | 6 | 5 | Yes | Return 5 |

Because the tree is completely skewed, the minimum depth equals the total number of nodes.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each node is visited at most once |
| Space | O(w) | Queue stores at most one tree level at a time |

The time complexity is linear because every node may need to be examined once in the worst case.

The space complexity depends on the maximum width of the tree. In a balanced binary tree, the widest level may contain roughly `n / 2` nodes, so the worst-case auxiliary space is `O(n)`. In skewed trees, the queue size remains small.

## Edge Cases

### Empty Tree

If the input root is `None` or `nil`, the tree contains no nodes. The correct minimum depth is `0`.

This case is important because attempting BFS on a null root would cause runtime errors. The implementation handles this immediately with an early return.

### Single Node Tree

Consider:

```
1
```

The root node is also a leaf node because it has no children. The correct answer is `1`.

Some implementations accidentally return `0` if they incorrectly initialize depth tracking. The BFS solution correctly starts the root at depth `1`.

### Tree With One Missing Child

Consider:

```
    1
     \
      2
```

The minimum depth is `2`, not `1`.

This is the most common source of bugs in recursive solutions. A naive formula like:

```
1 + min(leftDepth, rightDepth)
```

fails because the missing child contributes a depth of `0`.

The BFS implementation avoids this issue naturally because it only terminates when it reaches a true leaf node.

### Highly Skewed Tree

A tree shaped like a linked list may contain up to `10^5` nodes.

Recursive DFS solutions risk stack overflow in languages with limited recursion depth. The iterative BFS approach avoids recursion entirely, making it safer for extremely deep trees.
