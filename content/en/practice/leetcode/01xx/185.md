## Solution

## Problem Understanding

This problem asks us to identify the employees who belong to the top three unique salary levels within each department.

The database contains two tables:

- The `Employee` table stores employee information, including their salary and department ID.
- The `Department` table stores department names.

The output should contain:

- The department name
- The employee name
- The employee salary

The important detail is that the ranking is based on **unique salary values**, not employee count.

For example, suppose a department has salaries:

```
90000, 85000, 85000, 70000, 69000
```

The unique salary rankings are:

```
1st: 90000
2nd: 85000
3rd: 70000
4th: 69000
```

Even though two employees earn `85000`, that salary only occupies one ranking position. Therefore:

- Both employees earning `85000` should be included
- The employee earning `69000` should not be included because that salary is the fourth unique salary

The result may be returned in any order, so sorting is not required unless explicitly requested.

The constraints tell us several important things:

- Multiple employees can share the same salary within a department
- Departments may contain fewer than three distinct salary values
- Employee names are unique within the context of salary and department combinations
- We must handle ties correctly

A naive implementation can easily fail if it ranks employees individually instead of ranking unique salary values. That distinction is the core challenge of the problem.

Important edge cases include:

- Departments with fewer than three unique salaries
- Multiple employees sharing the same top salary
- Multiple employees sharing the third-highest salary
- Departments containing only one employee

Our solution must correctly include all employees whose salary belongs to the top three distinct salary values in their department.

## Approaches

### Brute Force Approach

A brute force solution would process each employee independently and determine how many distinct salaries in the same department are greater than that employee's salary.

For every employee:

1. Scan all employees in the same department
2. Collect distinct salaries greater than the current employee's salary
3. Count how many unique greater salaries exist
4. If fewer than three unique greater salaries exist, include the employee in the result

This works because:

- If there are zero greater unique salaries, the employee has the highest salary
- If there is one greater unique salary, the employee has the second-highest salary
- If there are two greater unique salaries, the employee has the third-highest salary
- If there are three or more greater unique salaries, the employee is outside the top three

Although correct, this approach repeatedly scans the same department data for every employee, making it inefficient.

### Optimal Approach

The key insight is that this is fundamentally a ranking problem.

SQL window functions are designed specifically for ranking rows within partitions. In this problem:

- Each department forms a partition
- Salaries must be ranked within each department
- Equal salaries must share the same rank

The correct window function is `DENSE_RANK()`.

`DENSE_RANK()` assigns:

- The same rank to equal salaries
- Consecutive rank numbers without gaps

For example:

```
90000 -> rank 1
85000 -> rank 2
85000 -> rank 2
70000 -> rank 3
69000 -> rank 4
```

Once the salaries are ranked, we simply keep employees whose rank is less than or equal to 3.

This avoids repeated scans and expresses the problem directly in SQL.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | For each employee, scan others in the same department |
| Optimal | O(n log n) | O(n) | Uses SQL window functions for efficient ranking |

## Algorithm Walkthrough

1. Join the `Employee` and `Department` tables so that each employee row contains the department name instead of only the department ID.
2. Partition employees by department using a window function. This ensures salary rankings are calculated independently for each department.
3. Apply `DENSE_RANK()` ordered by salary descending. Higher salaries receive smaller rank values:

- Highest salary gets rank 1
- Second distinct salary gets rank 2
- Third distinct salary gets rank 3
4. Employees sharing the same salary receive the same rank. This is important because the problem specifies top three unique salaries, not top three employees.
5. Store the ranking result in a subquery or common table expression so it can be filtered cleanly.
6. Select only rows where the rank is less than or equal to 3.
7. Return the department name, employee name, and salary.

The reason `DENSE_RANK()` is preferred over `ROW_NUMBER()` is that `ROW_NUMBER()` would incorrectly assign different rankings to employees with identical salaries.

For example:

```
90000 -> row_number 1
85000 -> row_number 2
85000 -> row_number 3
70000 -> row_number 4
```

This would incorrectly exclude the employee earning `70000`, even though it is the third unique salary.

## Python Solution

Even though this is a database problem, LeetCode still expects the SQL query inside a Python code block for consistency with the platform format.

```
SELECT
    d.name AS Department,
    ranked.employee_name AS Employee,
    ranked.salary AS Salary
FROM (
    SELECT
        e.name AS employee_name,
        e.salary,
        e.departmentId,
        DENSE_RANK() OVER (
            PARTITION BY e.departmentId
            ORDER BY e.salary DESC
        ) AS salary_rank
    FROM Employee e
) ranked
JOIN Department d
    ON ranked.departmentId = d.id
WHERE ranked.salary_rank <= 3;
```

The query begins by creating a derived table named `ranked`.

Inside this subquery:

- Employees are grouped logically by `departmentId`
- Salaries are ordered from highest to lowest
- `DENSE_RANK()` assigns rankings based on distinct salary values

The result contains:

- Employee name
- Salary
- Department ID
- Salary rank

Next, the query joins the derived table with `Department` so the department name can be displayed.

Finally, the `WHERE` clause filters the results to include only employees whose salary rank is 1, 2, or 3.

This directly matches the requirement of selecting employees with top three unique salaries per department.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | Ranking requires sorting salaries within partitions |
| Space | O(n) | Window function processing stores ranking information |

The dominant cost comes from sorting employee salaries inside each department partition when computing `DENSE_RANK()`.

The database engine internally manages partitioning and ranking structures, which require additional memory proportional to the number of processed rows.

Compared to the brute force approach, this solution is significantly more efficient because rankings are computed once per partition rather than recomputed for every employee.

## Edge Cases

### Departments With Fewer Than Three Unique Salaries

A department may contain only one or two distinct salary values. For example:

```
80000, 60000
```

In this case, both salary values should still be included because they belong to the top three unique salaries available.

`DENSE_RANK()` naturally handles this situation because no employee receives a rank greater than 2.

### Multiple Employees Sharing the Same Salary

This is the most important edge case in the problem.

Consider:

```
90000, 85000, 85000, 70000
```

Both employees earning `85000` must appear in the output because `85000` is the second-highest unique salary.

Using `DENSE_RANK()` ensures tied salaries receive the same rank, which correctly includes all matching employees.

### Multiple Employees Sharing the Third-Highest Salary

Suppose salaries are:

```
100000, 90000, 80000, 80000
```

The salary `80000` is the third-highest unique salary.

Both employees earning `80000` must be included.

Again, `DENSE_RANK()` correctly assigns both employees rank 3.

### Departments With Only One Employee

A department containing a single employee should still return that employee.

For example:

```
Alice, 75000
```

The employee receives rank 1 and passes the filter condition `salary_rank <= 3`.

The implementation handles this automatically without any special-case logic.
