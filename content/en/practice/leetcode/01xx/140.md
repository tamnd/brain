# Word Break II

## Problem Understanding

The problem gives us a string `s` and a list of valid dictionary words called `wordDict`. Our goal is to insert spaces into the string so that every resulting segment is a valid dictionary word.

Unlike the simpler "Word Break" problem, this version requires returning all possible valid sentences, not just determining whether one exists.

For example, if:

```
s = "catsanddog"
wordDict = ["cat","cats","and","sand","dog"]
```

there are two valid ways to split the string:

```
"cats and dog"
"cat sand dog"
```

Both must be returned.

The important detail is that dictionary words may be reused multiple times. A word appearing once in `wordDict` can still be used repeatedly in the final sentence construction.

The constraints are relatively small:

- `s.length <= 20`
- `wordDict.length <= 1000`
- Each dictionary word has length at most `10`

Even though the input string is short, the number of possible sentence combinations can grow exponentially. This means a naive recursive search can become extremely expensive due to repeated computations.

The problem also guarantees that the total output size does not exceed `10^5`, which ensures the returned answer is manageable.

Several edge cases are important:

- The string may not be breakable at all.
- Multiple overlapping dictionary choices may exist.
- Words can share prefixes, such as `"cat"` and `"cats"`.
- The same suffix may need to be solved many times during recursion.
- The entire string itself may already be a dictionary word.

These cases make memoization extremely important.

## Approaches

### Brute Force Backtracking

The most straightforward solution is recursive backtracking.

Starting from index `0`, we try every possible substring beginning at the current position. If a substring exists in the dictionary, we recursively solve the remaining suffix.

For example:

```
catsanddog
```

At index `0`, we try:

- `"c"`
- `"ca"`
- `"cat"` -> valid
- `"cats"` -> valid

For every valid word, we recursively continue solving the remainder.

This eventually generates every possible sentence, so the algorithm is correct.

However, the same suffix gets recomputed repeatedly.

For instance, both:

```
"cat" + solve("sanddog")
"cats" + solve("anddog")
```

may later recurse into the same substring `"dog"` many times.

This repeated work causes exponential blowup.

### Optimal Approach, DFS + Memoization

The key insight is that many recursive calls solve the exact same suffix repeatedly.

If we define:

```
dfs(start)
```

as:

> all valid sentences that can be formed from `s[start:]`

then every starting index only needs to be solved once.

We use memoization to cache results for each index.

When recursion revisits the same index, we immediately return the cached answer instead of recomputing it.

This transforms the solution from repeated exponential recomputation into a much more efficient top down dynamic programming approach.

We also use a hash set for the dictionary so substring lookups are `O(1)` on average.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n * n) | O(n) recursion depth | Recomputes the same suffixes repeatedly |
| Optimal | O(n * 2^n) worst case | O(n * 2^n) | Memoization avoids repeated subproblems |

The output itself can be exponential in size, so no algorithm can avoid that worst case completely.

## Algorithm Walkthrough

1. Convert `wordDict` into a hash set for fast lookup.

Checking whether a substring is a valid word will happen frequently. A hash set reduces lookup time to approximately `O(1)`.
2. Create a memoization map.

The key is the starting index in the string.

The value is a list of all valid sentences that can be formed from that suffix.
3. Define a recursive DFS function.

```
dfs(start)
```

returns every valid sentence that can be built from:

```
s[start:]
```
4. Handle the base case.

If `start == len(s)`, we have successfully segmented the entire string.

Return:

```
[""]
```

The empty string acts as a sentinel that helps sentence concatenation work cleanly.
5. Check memoization.

Before doing any computation, see whether this index has already been solved.

If so, return the cached result immediately.
6. Try every possible ending position.

For every index `end` from `start + 1` to `len(s)`:

- Extract:

```
s[start:end]
```
- If the substring exists in the dictionary, recursively solve the remaining suffix.
7. Combine current word with recursive results.

Suppose:

```
current_word = "cat"
```

and recursive results are:

```
["sand dog"]
```

Then construct:

```
"cat sand dog"
```

If the recursive sentence is empty, simply use the current word itself.
8. Store results in memoization.

Cache the generated sentences for the current index.
9. Return the final result for index `0`.

This represents all valid sentences for the entire string.

## Python Solution

```
from typing import List

class Solution:
    def wordBreak(self, s: str, wordDict: List[str]) -> List[str]:
        word_set = set(wordDict)
        memo = {}

        def dfs(start: int) -> List[str]:
            if start == len(s):
                return [""]

            if start in memo:
                return memo[start]

            sentences = []

            for end in range(start + 1, len(s) + 1):
                word = s[start:end]

                if word in word_set:
                    suffix_sentences = dfs(end)

                    for sentence in suffix_sentences:
                        if sentence:
                            sentences.append(word + " " + sentence)
                        else:
                            sentences.append(word)

            memo[start] = sentences
            return sentences

        return dfs(0)
```

The implementation begins by converting the dictionary into a set for fast membership checks.

The `memo` dictionary stores previously computed results. Each key is a starting index, and each value is the list of all valid sentences from that position onward.

The recursive `dfs` function performs the main search.

When the recursion reaches the end of the string, it returns `[""]`. This simplifies sentence construction because it avoids special handling during concatenation.

For every possible substring beginning at `start`, we check whether it exists in the dictionary. If it does, we recursively compute all valid suffix sentences.

The combination logic is important:

- If the recursive result is non empty, append:

```
current_word + " " + suffix
```
- Otherwise, append only the current word.

Finally, the computed results are cached in `memo` before returning.

## Go Solution

```
func wordBreak(s string, wordDict []string) []string {
	wordSet := make(map[string]bool)

	for _, word := range wordDict {
		wordSet[word] = true
	}

	memo := make(map[int][]string)

	var dfs func(int) []string

	dfs = func(start int) []string {
		if start == len(s) {
			return []string{""}
		}

		if value, exists := memo[start]; exists {
			return value
		}

		var sentences []string

		for end := start + 1; end <= len(s); end++ {
			word := s[start:end]

			if wordSet[word] {
				suffixSentences := dfs(end)

				for _, sentence := range suffixSentences {
					if sentence != "" {
						sentences = append(sentences, word+" "+sentence)
					} else {
						sentences = append(sentences, word)
					}
				}
			}
		}

		memo[start] = sentences
		return sentences
	}

	return dfs(0)
}
```

The Go implementation follows the same recursive memoized structure as the Python solution.

A `map[string]bool` is used for constant time dictionary lookup.

Memoization uses:

```
map[int][]string
```

where each index maps to all valid sentences starting from that position.

Go slices are naturally dynamic, so appending sentences works efficiently.

Unlike Python, Go does not have built in memo decorators or convenient list comprehensions, so sentence construction is handled explicitly with loops.

The function returns an empty slice if no valid segmentation exists.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n * 2^n) worst case | The number of valid sentences can be exponential |
| Space | O(n * 2^n) | Memoization and output storage can grow exponentially |

The recursion explores possible sentence constructions, and in the worst case the number of valid outputs itself becomes exponential.

Memoization ensures each starting index is solved only once, eliminating redundant recursive computation.

The dominant cost comes from constructing and storing all valid sentences.

## Edge Cases

### No Valid Segmentation

Consider:

```
s = "catsandog"
```

where no complete sentence exists.

A buggy implementation might partially construct sentences and accidentally return incomplete results.

This solution correctly returns an empty list because recursive branches only contribute results when the entire suffix can be segmented successfully.

### Multiple Overlapping Prefixes

Consider:

```
s = "catsanddog"
wordDict = ["cat", "cats", "sand", "and", "dog"]
```

Both `"cat"` and `"cats"` are valid prefixes.

A greedy algorithm could incorrectly choose only one path and miss valid answers.

This implementation tries every possible valid prefix and therefore returns all valid sentences.

### Entire String Is a Dictionary Word

Consider:

```
s = "apple"
wordDict = ["apple"]
```

The algorithm must recognize that the whole string itself forms a valid sentence.

When recursion reaches the end of the string, it returns `[""]`, which allows the final word to be added correctly without trailing spaces.

### Repeated Subproblem Computation

Consider:

```
s = "aaaaaaa"
wordDict = ["a", "aa", "aaa"]
```

Many recursive paths revisit the same suffix repeatedly.

Without memoization, runtime becomes extremely slow.

This implementation caches results for each starting index, ensuring every suffix is solved only once.
