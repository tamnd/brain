## Solution

## Problem Understanding

The problem asks us to convert a positive integer into the column naming system used by spreadsheet applications like Microsoft Excel. In Excel, columns are labeled alphabetically instead of numerically. The sequence begins as:

```
1  -> A
2  -> B
...
26 -> Z
27 -> AA
28 -> AB
...
```

The input, `columnNumber`, represents a 1-based column index. Our task is to return the corresponding column title as a string.

At first glance, this looks similar to converting a number into base 26. However, there is an important difference. Standard positional numeral systems use digits ranging from `0` to `base - 1`. Excel columns do not contain a zero character. Instead, the alphabet maps like this:

```
A -> 1
B -> 2
...
Z -> 26
```

This means the system is effectively a modified base-26 representation with digits `1` through `26` instead of `0` through `25`.

The constraint is:

```
1 <= columnNumber <= 2^31 - 1
```

This tells us several things:

- The input is always positive, so we never need to handle zero or negative values.
- The maximum value is large, so the algorithm should be efficient.
- Since each division by 26 reduces the number significantly, the number of iterations will remain very small, even for the maximum input.

There are several edge cases that can easily cause bugs:

- Values exactly divisible by 26, such as `26`, `52`, or `702`. These correspond to titles ending in `Z`.
- Transitions between digit lengths, such as `26 -> Z` and `27 -> AA`.
- Very large numbers near the upper constraint.

A naive implementation using ordinary base conversion logic will fail because Excel column titles are not zero-indexed.

## Approaches

### Brute Force Approach

A brute force solution would simulate Excel column generation sequentially. We could start from `"A"`, then `"B"`, then `"C"`, and continue generating column names one by one until reaching the desired index.

For example:

```
1  -> A
2  -> B
...
26 -> Z
27 -> AA
...
```

This approach works because eventually we would generate the exact target column title. However, it is extremely inefficient for large inputs. If `columnNumber` is near `2^31 - 1`, generating every previous column title would be computationally infeasible.

The brute force approach also requires complex string increment logic similar to incrementing numbers with carry propagation.

### Optimal Approach

The key observation is that this problem is essentially a modified base-26 conversion problem.

In a normal base-26 system:

```
0 -> A
1 -> B
...
25 -> Z
```

But Excel uses:

```
1 -> A
2 -> B
...
26 -> Z
```

Because there is no zero digit, we must adjust the number before taking the remainder. The standard trick is:

1. Subtract 1 from the current number.
2. Compute the remainder using `% 26`.
3. Convert the remainder into a letter.
4. Divide by 26 and continue.

This adjustment converts the problem into a clean zero-indexed representation internally.

For example, for `28`:

```
28 - 1 = 27
27 % 26 = 1 -> B
27 / 26 = 1

1 - 1 = 0
0 % 26 = 0 -> A
```

Reading characters in reverse order gives `"AB"`.

This approach is efficient because each iteration divides the number by 26.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) or worse | O(n) | Simulates every column title sequentially |
| Optimal | O(log26 n) | O(log26 n) | Repeated division by 26 |

## Algorithm Walkthrough

1. Initialize an empty list or string builder to store the characters of the column title.
2. While `columnNumber` is greater than zero, continue processing digits from right to left. Each iteration determines one character of the final title.
3. Subtract 1 from `columnNumber`. This adjustment is necessary because Excel columns are 1-indexed instead of 0-indexed. Without this step, values divisible by 26 would produce incorrect results.
4. Compute the remainder using `% 26`. The remainder will now fall in the range `[0, 25]`, which maps cleanly to letters `'A'` through `'Z'`.
5. Convert the remainder into a character:

- `0 -> A`
- `1 -> B`
- ...
- `25 -> Z`

This is done by adding the remainder to the ASCII value of `'A'`.
6. Append the character to the result container.
7. Update `columnNumber` using integer division by 26.
8. Since characters are generated from least significant digit to most significant digit, reverse the collected characters before returning the final string.

## Python Solution

```
class Solution:
    def convertToTitle(self, columnNumber: int) -> str:
        characters = []

        while columnNumber > 0:
            columnNumber -= 1

            remainder = columnNumber % 26
            characters.append(chr(ord('A') + remainder))

            columnNumber //= 26

        return "".join(reversed(characters))
```

The implementation follows the mathematical conversion process directly.

The `characters` list stores the generated letters. We use a list because repeatedly appending to a Python string is inefficient.

Inside the loop, we first subtract 1 from `columnNumber`. This is the most important part of the solution because Excel column numbering starts at 1 rather than 0.

Next, we compute the remainder using `% 26`. The remainder identifies the current character. We convert it into a letter using:

```
chr(ord('A') + remainder)
```

This maps numeric values `0` through `25` to letters `'A'` through `'Z'`.

After extracting the current character, we reduce the number using integer division by 26.

The characters are produced in reverse order because we process the least significant digit first. Therefore, we reverse the list before joining it into the final string.

## Go Solution

```
func convertToTitle(columnNumber int) string {
    characters := []byte{}

    for columnNumber > 0 {
        columnNumber--

        remainder := columnNumber % 26
        characters = append(characters, byte('A'+remainder))

        columnNumber /= 26
    }

    // Reverse the slice
    left, right := 0, len(characters)-1
    for left < right {
        characters[left], characters[right] = characters[right], characters[left]
        left++
        right--
    }

    return string(characters)
}
```

The Go implementation follows the same logic as the Python version.

Instead of using a list of strings, we use a `[]byte` slice because characters in Go can be efficiently stored as bytes.

Go does not provide a built-in reverse function for slices, so we reverse the slice manually using two pointers.

The integer arithmetic works safely within Go's `int` type because the problem constraints fit comfortably within standard integer ranges.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(log26 n) | Each iteration divides the number by 26 |
| Space | O(log26 n) | The output string stores one character per digit |

The algorithm repeatedly divides the input number by 26, so the number of iterations equals the number of base-26 digits in the result. Since logarithms grow very slowly, the algorithm is extremely efficient even for the maximum constraint value.

The space complexity comes from storing the output characters. The number of characters is proportional to the number of digits in the modified base-26 representation.

## Edge Cases

One important edge case occurs when the number is exactly divisible by 26. For example:

```
26 -> Z
52 -> AZ
702 -> ZZ
```

A naive base conversion would incorrectly map remainder `0` to `'A'`. The subtraction step before modulo fixes this issue by shifting the numbering into a zero-based range internally.

Another important edge case is the transition between different title lengths. For example:

```
26 -> Z
27 -> AA
```

This is similar to how decimal numbers transition from `9` to `10`. Incorrect carry handling can easily produce invalid outputs like `"BA"` or `"AAA"` if the algorithm is not implemented carefully.

A third important edge case is very large input values near `2^31 - 1`. A brute force approach would be far too slow, but the optimal solution handles large numbers efficiently because each iteration reduces the value by a factor of 26.
