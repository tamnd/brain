# LeetCode 180 - Consecutive Numbers

## Problem Understanding

This problem provides a database table named `Logs` with two columns:

| Column | Description |
| --- | --- |
| `id` | A unique auto incrementing integer |
| `num` | A string or numeric value stored as varchar |

The task is to find every value in the `num` column that appears at least three times consecutively when ordered by `id`.

The important detail is the meaning of "consecutively". Two rows are considered consecutive only if their `id` values are adjacent in sequence and the corresponding `num` values are identical. We are not counting total frequency across the entire table. We are specifically searching for runs of repeated values.

For example, suppose the table contains:

| id | num |
| --- | --- |
| 1 | 1 |
| 2 | 1 |
| 3 | 1 |

The value `1` appears three times in a row, so it should be included in the result.

However, consider:

| id | num |
| --- | --- |
| 1 | 1 |
| 2 | 2 |
| 3 | 1 |
| 4 | 1 |

Although `1` appears three times overall, it never appears three times consecutively, so it should not be returned.

The output should contain a single column named `ConsecutiveNums`. The rows may be returned in any order.

The constraints are relatively small and straightforward because this is a SQL database problem rather than an algorithmic data structure problem. The key challenge is recognizing patterns between neighboring rows.

Several edge cases are important:

- A number may appear more than three times consecutively. It should still appear only once in the output.
- Multiple different numbers may satisfy the condition.
- The table may contain fewer than three rows, in which case the result is empty.
- Consecutive values must occur in neighboring rows by `id`, not merely close together.
- Duplicate answers must be avoided.

## Approaches

### Brute Force Approach

A brute force solution would examine every row and manually check the next two rows to see whether all three `num` values are equal.

Conceptually, for each row:

1. Compare the current row's `num` with the next row.
2. Compare it again with the row after that.
3. If all three match, add the value to the result.

This approach works because every possible consecutive sequence of length three is checked explicitly.

In SQL, this could be implemented using multiple self joins or correlated subqueries. While correct, a naive implementation may become inefficient because repeated scans of the table are performed.

### Optimal Approach

The key insight is that consecutive rows can be compared efficiently using self joins on neighboring `id` values.

If we join the table three times:

- `l1.id + 1 = l2.id`
- `l2.id + 1 = l3.id`

then the three joined rows represent consecutive positions in the table.

We then simply check whether:

```
l1.num = l2.num AND l2.num = l3.num
```

If this condition holds, we have found a value appearing at least three times consecutively.

Using `DISTINCT` ensures that values appearing in longer runs are returned only once.

This solution is elegant because SQL joins naturally model neighboring relationships between rows.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Repeated scans or comparisons for each row |
| Optimal | O(n) | O(1) | Uses sequential self joins on adjacent ids |

## Algorithm Walkthrough

1. Start with the `Logs` table and create three aliases of it, typically named `l1`, `l2`, and `l3`.
2. Join `l1` with `l2` such that:

```
l1.id + 1 = l2.id
```

This guarantees that `l2` represents the row immediately after `l1`.
3. Join `l2` with `l3` such that:

```
l2.id + 1 = l3.id
```

This guarantees that `l3` represents the row immediately after `l2`.
4. Now each joined row group represents three consecutive rows from the original table.
5. Check whether:

```
l1.num = l2.num AND l2.num = l3.num
```

If true, all three consecutive rows contain the same value.
6. Select the matching number and rename the column to `ConsecutiveNums`.
7. Use `DISTINCT` to avoid duplicate outputs when a number appears more than three times consecutively.

## Python Solution

Although this is fundamentally a SQL problem, LeetCode database problems are usually solved using raw SQL. Below is the correct SQL solution presented inside a Python string for completeness.

```
class Solution:
    def consecutiveNumbers(self) -> str:
        return """
        SELECT DISTINCT l1.num AS ConsecutiveNums
        FROM Logs l1
        JOIN Logs l2
            ON l1.id + 1 = l2.id
        JOIN Logs l3
            ON l2.id + 1 = l3.id
        WHERE l1.num = l2.num
          AND l2.num = l3.num
        """
```

The implementation creates three logical views of the same table using aliases. The first join guarantees adjacency between the first and second rows. The second join guarantees adjacency between the second and third rows.

After establishing consecutive ordering, the `WHERE` clause verifies that all three `num` values are identical.

Finally, `DISTINCT` removes duplicate outputs. This is necessary because a sequence longer than three produces overlapping matches. For example:

| id | num |
| --- | --- |
| 1 | 7 |
| 2 | 7 |
| 3 | 7 |
| 4 | 7 |

This generates two valid triples:

- `(1,2,3)`
- `(2,3,4)`

Without `DISTINCT`, the value `7` would appear twice.

The database engine performs joins across neighboring rows. Since each row is examined only a constant number of times, the overall runtime is linear relative to the number of rows in the table.

The query itself does not allocate additional auxiliary storage proportional to input size, so the extra space usage is constant.

## Edge Cases

### Table Contains Fewer Than Three Rows

If the table has one or two rows, there is no possible sequence of three consecutive entries. The joins naturally fail because matching neighboring rows do not exist. The query correctly returns an empty result set.

### A Number Appears More Than Three Times Consecutively

Consider:

| id | num |
| --- | --- |
| 1 | 5 |
| 2 | 5 |
| 3 | 5 |
| 4 | 5 |

There are two valid overlapping triples:

- `(1,2,3)`
- `(2,3,4)`

Without `DISTINCT`, the output would contain duplicate `5` values. The query handles this correctly by selecting distinct results only.

### Repeated Values That Are Not Consecutive

Consider:

| id | num |
| --- | --- |
| 1 | 8 |
| 2 | 8 |
| 3 | 9 |
| 4 | 8 |
| 5 | 8 |

The value `8` appears four times overall, but never three times consecutively. Since the joins require adjacent ids and the equality checks require all three values to match, the query correctly excludes `8`.

### Multiple Valid Consecutive Numbers

Consider:

| id | num |
| --- | --- |
| 1 | 1 |
| 2 | 1 |
| 3 | 1 |
| 4 | 2 |
| 5 | 2 |
| 6 | 2 |

Both `1` and `2` satisfy the condition. The query correctly returns both values because every qualifying triple is independently detected.
