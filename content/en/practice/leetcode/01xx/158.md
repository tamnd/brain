## Solution

## Problem Understanding

This problem asks us to implement a custom `read(buf, n)` function using only a restricted API called `read4(buf4)`. The challenge is not simply reading characters once, but handling the fact that `read` may be called multiple times.

The `read4` API reads up to four consecutive characters from the file and places them into a temporary buffer, `buf4`. It returns the number of characters actually read, which can be fewer than four if the end of the file is reached.

Our goal is to implement a `read(buf, n)` method that reads at most `n` characters into `buf` and returns the actual number of characters read.

The key complication is that `read4` always reads in chunks of four, while the caller may request any number of characters. Since `read` can be called multiple times, we cannot discard unused characters from a previous `read4` call. If we read extra characters, we must preserve them for future calls.

For example, suppose the file is `"abcdef"` and the first call is `read(buf, 1)`.

If we call `read4`, we receive `"abcd"`. We only need `"a"` for the current request, but `"bcd"` must remain available for later calls. A naive implementation that discards them would permanently lose data.

The input consists of:

- A hidden file that can only be accessed through `read4`
- Multiple calls to `read(buf, n)`
- A destination buffer `buf` with enough capacity
- An integer `n`, the maximum number of characters to read

The output of each call is the number of actual characters copied into `buf`.

The constraints are relatively small, with file length and query sizes up to `500`, but the difficulty comes from correctness rather than efficiency. Since the problem is interactive in nature, a wrong state-management strategy will fail even if the asymptotic complexity looks acceptable.

Several edge cases are important:

- A request may ask for fewer than four characters, causing leftover data that must be preserved.
- A request may ask for more characters than remain in the file.
- Multiple consecutive calls may eventually exhaust the file.
- A call after reaching end-of-file should return `0`.
- The same `buf` object is reused across calls, so our implementation must carefully track internal state.

## Approaches

### Brute Force Approach

A naive approach would call `read4` repeatedly every time `read` is invoked and directly copy characters into `buf` until `n` characters are collected.

The problem with this strategy is that it ignores extra characters returned by `read4`.

Suppose the file is `"abcde"` and the first request is `read(buf, 1)`.

We call `read4` and receive `"abcd"`. We return `"a"` and discard `"bcd"`.

The next request `read(buf, 2)` should return `"bc"`, but we already lost those characters. The file pointer has advanced to `"e"`.

This approach is incorrect because it violates the requirement that `read` may be called multiple times.

### Optimal Approach

The key observation is that we must maintain a persistent internal buffer across calls.

Whenever `read4` returns more characters than we immediately need, we store the unused characters in a class-level buffer. Future calls to `read` will consume those stored characters before invoking `read4` again.

This transforms the problem into a buffered stream reader.

We maintain:

- A temporary internal buffer of size `4`
- A pointer indicating the next unread character in that buffer
- A count of how many valid characters currently exist in the buffer

Whenever the internal buffer becomes empty, we refill it using `read4`.

This ensures:

1. No characters are lost.
2. Characters are returned in correct order.
3. Multiple calls behave consistently.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Incorrect for multiple calls because leftover characters are discarded |
| Optimal | O(n) | O(1) | Uses persistent buffer to preserve unread characters |

## Algorithm Walkthrough

1. Initialize persistent state in the constructor.

We maintain three class variables:

- `buffer4`, an internal buffer of size four
- `buffer_ptr`, the next unread position inside `buffer4`
- `buffer_count`, the number of valid characters currently in the buffer

These variables persist across multiple `read` calls.
2. Start reading into the destination buffer.

We keep a variable `total_read` to track how many characters have been copied into `buf`.
3. Check whether the internal buffer is empty.

If `buffer_ptr == buffer_count`, it means all buffered characters have been consumed.

In this case, call `read4(buffer4)` to refill it.
4. Detect end-of-file.

If `read4` returns `0`, there are no more characters left in the file, so stop reading.
5. Copy characters from the internal buffer.

While there are characters available and we still need more characters, copy from `buffer4[buffer_ptr]` into `buf[total_read]`.

Increment:

- `buffer_ptr`
- `total_read`
6. Repeat until either:

- We read `n` characters, or
- We reach end-of-file
7. Return the number of actual characters read.

This design works because leftover characters remain in the internal buffer for future calls.

## Python Solution

```
from typing import List

# The read4 API is already defined for you.
# def read4(buf4: List[str]) -> int:

class Solution:
    def __init__(self):
        self.buffer4 = [""] * 4
        self.buffer_ptr = 0
        self.buffer_count = 0

    def read(self, buf: List[str], n: int) -> int:
        total_read = 0

        while total_read < n:
            # Refill internal buffer if empty
            if self.buffer_ptr == self.buffer_count:
                self.buffer_count = read4(self.buffer4)
                self.buffer_ptr = 0

                # End of file
                if self.buffer_count == 0:
                    break

            # Copy from internal buffer to output
            while (
                total_read < n
                and self.buffer_ptr < self.buffer_count
            ):
                buf[total_read] = self.buffer4[self.buffer_ptr]
                total_read += 1
                self.buffer_ptr += 1

        return total_read
```

The implementation begins by defining persistent instance variables in `__init__`. This is critical because LeetCode may call `read` multiple times on the same `Solution` object. Any leftover characters from one invocation must remain available for future invocations.

Inside `read`, we maintain `total_read`, which tracks how many characters have been copied into the destination buffer.

The outer loop continues until we either satisfy the request or run out of file contents.

Before copying, we check whether our internal buffer is exhausted. If so, we refill it using `read4`. Resetting `buffer_ptr` to `0` ensures we start consuming the fresh buffer from the beginning.

If `read4` returns `0`, we know we have reached the end of the file and terminate early.

The inner loop copies characters one by one from the internal buffer into the destination buffer until either:

- We reach the requested `n` characters
- The internal buffer becomes exhausted

Unused characters remain in `buffer4`, and `buffer_ptr` preserves where reading should resume in the next call.

## Go Solution

```
/**
 * The read4 API is already defined for you.
 *
 *     read4 := func(buf4 []byte) int
 */

var solution = func(read4 func([]byte) int) func([]byte, int) int {
	buffer4 := make([]byte, 4)
	bufferPtr := 0
	bufferCount := 0

	return func(buf []byte, n int) int {
		totalRead := 0

		for totalRead < n {
			// Refill internal buffer if empty
			if bufferPtr == bufferCount {
				bufferCount = read4(buffer4)
				bufferPtr = 0

				// End of file
				if bufferCount == 0 {
					break
				}
			}

			// Copy from internal buffer
			for totalRead < n && bufferPtr < bufferCount {
				buf[totalRead] = buffer4[bufferPtr]
				totalRead++
				bufferPtr++
			}
		}

		return totalRead
	}
}
```

The Go implementation follows the same logic as Python, but state persistence is achieved differently.

Instead of instance variables, we use closure variables inside the `solution` function. The variables `buffer4`, `bufferPtr`, and `bufferCount` live in the enclosing scope and persist across multiple invocations of the returned function.

Since Go slices are mutable references, we can directly copy bytes into `buf`. No special handling for `nil` is needed because the problem guarantees enough capacity.

There are also no integer overflow concerns because all values are bounded by `500`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each character is processed at most once |
| Space | O(1) | Internal buffer size is fixed at 4 |

The time complexity is `O(n)` because each requested character is copied exactly once into the output buffer. Although `read4` is called multiple times, each call only contributes toward reading new characters, so the total work remains proportional to the number of characters read.

The space complexity is `O(1)` because we only maintain a fixed-size internal buffer of length `4`, regardless of file size or request size.

## Edge Cases

### Reading Fewer Than Four Characters

A common bug occurs when the request size is smaller than the amount returned by `read4`.

For example, if the file is `"abcd"` and `read(buf, 1)` is called, `read4` returns all four characters. A naive implementation may discard `"bcd"`.

Our implementation avoids this issue by storing unread characters inside `buffer4` and tracking the current position with `buffer_ptr`.

### Reading More Characters Than Remain

Suppose the file contains `"abc"` and the request is `read(buf, 10)`.

The implementation continues calling `read4` until end-of-file is reached. Since `read4` eventually returns `0`, the algorithm stops naturally and returns `3`, the number of available characters.

### Multiple Consecutive Reads

The hardest edge case is when leftover characters must persist across multiple calls.

For example:

```
file = "abcdef"
read(1) -> "a"
read(2) -> "bc"
read(2) -> "de"
```

Without persistent state, characters would be lost after each call.

Our solution preserves unread characters in `buffer4`, ensuring subsequent calls resume exactly where the previous one stopped.

### Calls After End-of-File

Once the file is fully consumed, future calls should immediately return `0`.

Our implementation handles this correctly because `read4` returns `0` after EOF, causing the function to stop without copying any additional characters.
