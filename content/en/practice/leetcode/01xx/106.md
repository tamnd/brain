## Solution

## Problem Understanding

The problem gives two traversal orders of the same binary tree:

- `inorder`, which visits nodes in the order:

left subtree → root → right subtree
- `postorder`, which visits nodes in the order:

left subtree → right subtree → root

The task is to reconstruct the original binary tree and return its root node.

The important detail is that all node values are unique. This guarantee is critical because it allows us to uniquely identify the position of every node inside the inorder traversal. Without uniqueness, reconstruction would become ambiguous.

To understand the reconstruction process, recall the properties of the traversals:

- In a postorder traversal, the last element is always the root of the current subtree.
- In an inorder traversal, elements to the left of the root belong to the left subtree, and elements to the right belong to the right subtree.

For example:

```
inorder   = [9,3,15,20,7]
postorder = [9,15,7,20,3]
```

The last value in postorder is `3`, so `3` must be the root.

In the inorder traversal:

```
[9, 3, 15,20,7]
    ^
```

Everything left of `3` belongs to the left subtree, and everything right belongs to the right subtree.

The constraints tell us that the input size can be as large as 3000 nodes. This is large enough that inefficient repeated scans can become problematic. A quadratic solution may still pass in some languages, but the intended solution is linear time.

There are several important edge cases:

- A tree containing only one node.
- A completely skewed tree, where every node only has a left child or only a right child.
- Empty subtree ranges during recursion.
- Large inputs where repeated linear searches would become expensive.

The problem guarantees that:

- Both traversals are valid.
- Every value is unique.
- Both arrays describe the same binary tree.

These guarantees simplify the implementation significantly.

## Approaches

### Brute Force Approach

The straightforward recursive approach works as follows:

1. Take the last value from the current postorder range as the root.
2. Search linearly through the inorder array to find the root position.
3. Determine the sizes of the left and right subtrees.
4. Recursively build the left subtree.
5. Recursively build the right subtree.

This approach is correct because the traversal properties uniquely determine the structure of the tree.

However, the expensive part is repeatedly scanning the inorder array to locate the root index. In the worst case, each recursive call performs an `O(n)` search, and there are `O(n)` recursive calls overall.

This leads to `O(n²)` time complexity.

For a skewed tree, the recursion depth can also become `O(n)`.

### Optimal Approach

The key optimization is to avoid repeatedly searching the inorder array.

Since all node values are unique, we can preprocess the inorder traversal into a hash map:

```
value -> index in inorder
```

This allows us to locate any root position in constant time.

The recursive structure remains the same:

- The last value in postorder is the root.
- The inorder position splits the tree into left and right subtrees.
- Subtree sizes determine how to partition the traversal ranges.

Using the hash map eliminates the repeated linear scans, reducing the total complexity to `O(n)`.

## Algorithm Walkthrough

### Optimal Recursive Construction

1. Create a hash map from the inorder traversal.

The map stores each value and its index inside the inorder array. This allows constant-time lookup of subtree boundaries.
2. Define a recursive helper function.

The helper receives the current boundaries of both traversals:

- `in_left`, `in_right`
- `post_left`, `post_right`

These boundaries represent the subtree currently being constructed.
3. Handle the base case.

If the inorder range becomes invalid:

```
in_left > in_right
```

then the subtree is empty, so return `None`.
4. Identify the root node.

In postorder traversal, the last element is always the root of the current subtree:

```
root_value = postorder[post_right]
```
5. Create the root node.

Construct a `TreeNode` using the root value.
6. Find the root position in inorder.

Use the hash map to find the root index instantly.

This divides the inorder traversal into:

- Left subtree
- Right subtree
7. Compute the left subtree size.

The number of nodes in the left subtree is:

```
left_size = root_index - in_left
```

This value is necessary to partition the postorder traversal correctly.
8. Recursively build the left subtree.

The left subtree occupies:

- Left portion of inorder
- First portion of postorder
9. Recursively build the right subtree.

The right subtree occupies:

- Right portion of inorder
- Remaining portion of postorder before the root
10. Return the constructed root node.

The recursion gradually rebuilds the entire tree from the bottom up.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

class Solution:
    def buildTree(self, inorder: List[int], postorder: List[int]) -> Optional[TreeNode]:
        inorder_index = {}

        for index, value in enumerate(inorder):
            inorder_index[value] = index

        def build(
            in_left: int,
            in_right: int,
            post_left: int,
            post_right: int
        ) -> Optional[TreeNode]:

            if in_left > in_right:
                return None

            root_value = postorder[post_right]
            root = TreeNode(root_value)

            root_index = inorder_index[root_value]

            left_size = root_index - in_left

            root.left = build(
                in_left,
                root_index - 1,
                post_left,
                post_left + left_size - 1
            )

            root.right = build(
                root_index + 1,
                in_right,
                post_left + left_size,
                post_right - 1
            )

            return root

        return build(
            0,
            len(inorder) - 1,
            0,
            len(postorder) - 1
        )
```

The implementation begins by building the `inorder_index` hash map. This preprocessing step allows constant-time lookup of any node position in the inorder traversal.

The recursive `build` function reconstructs a subtree using index boundaries rather than slicing arrays. Avoiding slicing is important because array slicing creates copies and would increase the complexity.

The base case checks whether the inorder range is invalid. An invalid range means the subtree contains no nodes.

The root value is always taken from the end of the current postorder range. A new `TreeNode` is created from that value.

Using the inorder index map, the algorithm determines how many nodes belong to the left subtree. That subtree size is then used to correctly partition the postorder traversal.

The left subtree is built first, followed by the right subtree. Finally, the root node is returned upward through the recursion stack.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func buildTree(inorder []int, postorder []int) *TreeNode {
	inorderIndex := make(map[int]int)

	for i, value := range inorder {
		inorderIndex[value] = i
	}

	var build func(
		inLeft int,
		inRight int,
		postLeft int,
		postRight int,
	) *TreeNode

	build = func(
		inLeft int,
		inRight int,
		postLeft int,
		postRight int,
	) *TreeNode {

		if inLeft > inRight {
			return nil
		}

		rootValue := postorder[postRight]
		root := &TreeNode{Val: rootValue}

		rootIndex := inorderIndex[rootValue]

		leftSize := rootIndex - inLeft

		root.Left = build(
			inLeft,
			rootIndex-1,
			postLeft,
			postLeft+leftSize-1,
		)

		root.Right = build(
			rootIndex+1,
			inRight,
			postLeft+leftSize,
			postRight-1,
		)

		return root
	}

	return build(
		0,
		len(inorder)-1,
		0,
		len(postorder)-1,
	)
}
```

The Go implementation closely mirrors the Python version. The primary difference is memory handling:

- Go uses pointers to `TreeNode`.
- Empty subtrees are represented with `nil`.
- A map is used for constant-time inorder index lookup.
- Recursive boundaries are passed as integers to avoid slice copying.

Since the constraints are relatively small, integer overflow is not a concern.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every node is processed once |
| Space | O(n) | Hash map and recursion stack |

The algorithm visits each node exactly one time. Hash map lookups are constant time, so the total running time is linear.

The extra space comes from two sources:

- The inorder index hash map
- The recursion stack

In the worst case of a skewed tree, recursion depth becomes `O(n)`.

## Edge Cases

### Single Node Tree

A tree containing only one node is the smallest valid input:

```
inorder = [-1]
postorder = [-1]
```

This case can expose bugs in recursive boundary handling. If index calculations are incorrect, recursion may continue unnecessarily or access invalid indices.

The implementation handles this correctly because the recursive ranges immediately become invalid after constructing the single node.

### Completely Left-Skewed Tree

Consider:

```
inorder   = [4,3,2,1]
postorder = [4,3,2,1]
```

Every node only has a left child.

This creates maximum recursion depth and tests whether subtree boundaries are computed correctly. Incorrect calculations can easily swap left and right subtree ranges.

The implementation correctly computes subtree sizes using inorder positions, so the recursive partitions remain valid.

### Completely Right-Skewed Tree

Consider:

```
inorder   = [1,2,3,4]
postorder = [4,3,2,1]
```

Every node only has a right child.

This case is especially useful for testing empty left subtree handling. The algorithm must correctly return `None` when a subtree range becomes invalid.

The base case:

```
if in_left > in_right:
    return None
```

ensures empty subtrees are handled safely.

### Large Input Size

The input size can reach 3000 nodes. A naive implementation that repeatedly scans the inorder array for root positions would become inefficient.

The hash map optimization guarantees constant-time lookup, allowing the algorithm to remain efficient even for the maximum input size.
