## Solution

## Problem Understanding

The problem gives two binary tree roots, `p` and `q`, and asks whether the two trees are exactly the same.

Two trees are considered the same only if both of the following conditions are true:

1. Their structure is identical.
2. Every corresponding node contains the same value.

This means we must compare both the arrangement of nodes and the values stored inside them. If even one node differs in value, or one tree contains a node where the other tree has `null`, the trees are not the same.

The input is provided as references to the roots of two binary trees. Each node contains:

- An integer value
- A left child pointer
- A right child pointer

The output is a boolean:

- `true` if the trees are identical
- `false` otherwise

The constraints state that each tree contains at most 100 nodes. This is a relatively small input size, which means even recursive traversal is completely safe here. The trees can also be empty, since the number of nodes may be `0`.

Several edge cases are important:

- Both trees may be empty. In that case, they are considered the same.
- One tree may be empty while the other is not.
- Trees may contain the same values but different structures.
- Trees may contain identical structures but different values.
- Trees may be highly unbalanced, behaving more like linked lists.

A naive implementation that only compares node values without checking structure would fail on cases such as `[1,2]` versus `[1,null,2]`.

## Approaches

### Brute Force Approach

A brute-force strategy is to serialize both trees into arrays or strings using a traversal such as preorder traversal, including explicit markers for `null` nodes. After serialization, compare the resulting sequences.

For example:

- Tree A: `[1,2,null,null,3]`
- Tree B: `[1,2,null,null,3]`

If the serialized forms are identical, then the trees are identical.

This approach works because including `null` markers preserves tree structure. Without them, different trees could produce the same traversal order.

However, this method introduces unnecessary extra storage because we build full serialized representations before comparing them. We can do better by comparing the trees directly during traversal.

### Optimal Approach

The key observation is that two trees are the same if and only if:

- Their current nodes have the same value
- Their left subtrees are the same
- Their right subtrees are the same

This recursive definition maps perfectly to a depth-first traversal.

At every pair of nodes:

- If both are `null`, they match.
- If one is `null` and the other is not, they differ.
- If their values differ, they differ.
- Otherwise, recursively compare their left and right children.

This avoids extra serialization storage and stops immediately once a mismatch is found.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Serialize both trees and compare serialized output |
| Optimal | O(n) | O(h) | Direct recursive DFS comparison |

Here:

- `n` is the number of nodes
- `h` is the height of the tree

## Algorithm Walkthrough

1. Start at the roots of both trees.

We begin by comparing the current nodes from both trees simultaneously. The entire problem reduces to determining whether these two nodes represent identical subtrees.
2. Check whether both nodes are `null`.

If both nodes are `null`, they match because both trees ended at the same position. Return `True`.
3. Check whether only one node is `null`.

If one node exists and the other does not, the structures differ. Return `False`.
4. Compare the node values.

If the values stored in the two nodes are different, the trees cannot be identical. Return `False`.
5. Recursively compare the left subtrees.

Call the same function on `p.left` and `q.left`. This verifies that the left portions of both trees match structurally and by value.
6. Recursively compare the right subtrees.

Call the function on `p.right` and `q.right`.
7. Combine the results.

The current trees are identical only if:

- The current node values match
- The left subtrees match
- The right subtrees match

Therefore, return the logical AND of the recursive results.

This algorithm naturally performs a synchronized depth-first traversal across both trees.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import Optional

class Solution:
    def isSameTree(self, p: Optional[TreeNode], q: Optional[TreeNode]) -> bool:
        # Both nodes are empty
        if not p and not q:
            return True
        
        # One node is empty and the other is not
        if not p or not q:
            return False
        
        # Values differ
        if p.val != q.val:
            return False
        
        # Recursively compare left and right subtrees
        return (
            self.isSameTree(p.left, q.left) and
            self.isSameTree(p.right, q.right)
        )
```

The implementation directly follows the recursive definition of identical trees.

The first condition checks whether both nodes are `None`. If so, the subtrees match because both ended simultaneously.

The second condition checks whether exactly one node is missing. This detects structural differences immediately.

The third condition compares node values. Even if the structure matches, differing values make the trees unequal.

Finally, the function recursively compares both left subtrees and both right subtrees. The logical AND ensures that every corresponding part of the trees must match for the final result to be `True`.

Because recursion naturally traverses the trees depth first, no explicit stack or queue is needed.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func isSameTree(p *TreeNode, q *TreeNode) bool {
    // Both nodes are nil
    if p == nil && q == nil {
        return true
    }

    // One node is nil
    if p == nil || q == nil {
        return false
    }

    // Values differ
    if p.Val != q.Val {
        return false
    }

    // Compare left and right subtrees
    return isSameTree(p.Left, q.Left) &&
        isSameTree(p.Right, q.Right)
}
```

The Go implementation is nearly identical to the Python version, but it uses `nil` instead of `None`.

Because Go passes pointers to tree nodes, checking whether a node exists is done with pointer comparisons.

There are no integer overflow concerns because the algorithm only compares values and never performs arithmetic operations.

The recursion depth is safe given the maximum constraint of 100 nodes.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every node is visited once |
| Space | O(h) | Recursive call stack depends on tree height |

The algorithm visits each pair of corresponding nodes exactly once, resulting in linear time complexity.

The space complexity comes from recursion depth. In a balanced tree, the height is approximately `log n`. In the worst case, when the tree is completely skewed, the height becomes `n`.

## Edge Cases

### Both Trees Are Empty

If both `p` and `q` are `null`, the trees are considered identical. This case is important because recursion eventually reaches leaf boundaries. The implementation correctly returns `True` when both nodes are absent.

### One Tree Is Empty

If one root is `null` and the other contains nodes, the trees cannot be identical because their structures differ. The implementation explicitly checks this condition before attempting value comparisons.

### Same Values but Different Structure

Consider:

```
p = [1,2]
q = [1,null,2]
```

A naive solution that only compares traversal values without preserving structure could incorrectly return `true`. The recursive implementation avoids this by checking node existence at every corresponding position.

### Deeply Skewed Trees

A tree may resemble a linked list:

```
1
 \
  2
   \
    3
```

The recursive solution still works correctly because it compares corresponding child pointers at every level. Although recursion depth increases, the constraints are small enough that stack overflow is not a concern.

### Duplicate Values

Trees may contain repeated values, such as:

```
[1,1,1]
```

The algorithm does not rely on uniqueness of values. It compares every corresponding node individually, ensuring correctness even when many nodes share the same value.
