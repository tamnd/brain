# Single Number

## Problem Understanding

The problem gives us a non-empty integer array `nums`. Every value in the array appears exactly twice, except for one value that appears only once. Our task is to find and return that unique value.

In other words, the array mostly consists of pairs. Each duplicated number has a matching copy somewhere else in the array. However, there is exactly one number that does not have a duplicate. We need to identify that number efficiently.

The input is an array of integers, and the output is a single integer representing the element that appears only once.

The constraints are important:

- The array length can be as large as `3 * 10^4`
- Values can be negative or positive
- Every duplicated number appears exactly twice
- Exactly one number appears once

The problem also imposes two strict performance requirements:

- Linear time complexity, `O(n)`
- Constant extra space, `O(1)`

These requirements eliminate many straightforward approaches. For example, using a hash map would require additional memory proportional to the input size, which violates the constant space requirement.

There are several important edge cases to consider:

- The array may contain only one element, such as `[1]`
- The unique number may appear at the beginning, middle, or end
- Numbers may be negative
- The array is not guaranteed to be sorted

The problem guarantees that there will always be exactly one valid answer, so we never need to handle invalid input scenarios like multiple unique values or missing unique values.

## Approaches

### Brute Force Approach

A straightforward solution is to check each number and count how many times it appears in the array.

For every element, we scan the entire array and count occurrences. If the count is `1`, we return that number.

This works because the problem guarantees that exactly one number appears once.

However, this approach is inefficient. For each of the `n` elements, we potentially scan all `n` elements again. This results in `O(n^2)` time complexity, which is unnecessarily slow for larger inputs.

The space complexity is `O(1)` because we only use a few variables.

### Optimal Approach, Bit Manipulation with XOR

The key observation is the behavior of the XOR (`^`) operation.

XOR has several useful properties:

- `a ^ a = 0`
- `a ^ 0 = a`
- XOR is commutative and associative

This means the order of operations does not matter.

If we XOR all numbers together:

- Every duplicated number cancels itself out
- Only the unique number remains

For example:

```
2 ^ 2 ^ 1
= 0 ^ 1
= 1
```

Since every duplicated value appears exactly twice, all pairs disappear during XOR operations.

This approach satisfies both constraints:

- Linear time, because we scan the array once
- Constant extra space, because we only store one running result

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Count occurrences for every element |
| Optimal | O(n) | O(1) | XOR removes duplicate pairs automatically |

## Algorithm Walkthrough

1. Initialize a variable called `result` to `0`.

We start with `0` because XOR with `0` does not change a value. This makes it the ideal starting point for accumulation.
2. Traverse through every number in the array.

We process each element exactly once to maintain linear runtime complexity.
3. XOR the current number with `result`.

Each duplicate pair cancels itself out because `a ^ a = 0`.
4. Continue until all elements have been processed.

By the end of the traversal, every duplicated number has disappeared from the accumulated XOR result.
5. Return `result`.

The only remaining value is the number that appeared once.

## Python Solution

```
from typing import List

class Solution:
    def singleNumber(self, nums: List[int]) -> int:
        result = 0
        
        for number in nums:
            result ^= number
        
        return result
```

The implementation is extremely compact because XOR handles the duplicate removal automatically.

The variable `result` stores the running XOR of all processed elements. Initially it is `0`.

During each iteration, we XOR the current number into `result`.

If a number appears twice, its contributions cancel out:

```
result ^ x ^ x = result ^ 0 = result
```

After processing the entire array, only the unique number remains.

This directly matches the algorithm described earlier and satisfies the required `O(n)` time and `O(1)` space constraints.

## Go Solution

```
func singleNumber(nums []int) int {
    result := 0

    for _, number := range nums {
        result ^= number
    }

    return result
}
```

The Go implementation follows the exact same logic as the Python version.

The `range` loop iterates through the slice, and `^=` performs the XOR assignment operation.

There are no special concerns regarding integer overflow because XOR is a bitwise operation and does not increase numeric magnitude.

The problem guarantees a non-empty array, so we do not need additional checks for empty slices.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | We traverse the array exactly once |
| Space | O(1) | Only one extra variable is used |

The algorithm performs one XOR operation per element, so the runtime grows linearly with the size of the input array.

The space usage remains constant regardless of input size because we only store a single accumulator variable.

## Edge Cases

### Array with One Element

An input like:

```
[7]
```

is the smallest possible valid input.

A naive implementation might incorrectly assume that duplicates always exist. Our XOR approach handles this naturally because:

```
0 ^ 7 = 7
```

The single element is returned immediately.

### Negative Numbers

The array may contain negative integers, such as:

```
[-1, -1, -5]
```

Bitwise XOR works correctly with signed integers as well. Duplicate negative values still cancel each other out, leaving the unique number.

Our implementation does not rely on sorting or numeric magnitude, so negative values pose no issue.

### Unique Element at Any Position

The unique element may appear at the beginning, middle, or end of the array.

Examples:

```
[3, 1, 1]
[1, 3, 1]
[1, 1, 3]
```

Because XOR is commutative and associative, processing order does not matter. Duplicate pairs always cancel out regardless of position.

### Large Input Size

The array may contain up to `30,000` elements.

A quadratic solution could become unnecessarily slow because it repeatedly scans the array. The XOR solution processes each element once, making it efficient even at the maximum input size.
