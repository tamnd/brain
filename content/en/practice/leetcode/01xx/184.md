## Solution

## Problem Understanding

This problem asks us to identify the employee or employees who earn the **highest salary within each department**. The result should include the department name, employee name, and salary for every employee whose salary matches the maximum salary in their department.

We are given two tables:

The `Employee` table stores information about employees. Each row contains an employee ID, employee name, salary, and a `departmentId` that references the department where the employee works.

The `Department` table stores department information. Each row contains a department ID and department name.

The task is to combine information from both tables and determine which employees earn the highest salary in their department. A very important detail is that **multiple employees may share the highest salary in the same department**. In such cases, we must include **all tied employees** in the result.

For example, if both Jim and Max earn `90000` in the IT department, and no one earns more than that, both employees must appear in the output.

The expected output contains three columns:

| Column | Meaning |
| --- | --- |
| `Department` | Name of the department |
| `Employee` | Name of the employee |
| `Salary` | Employee's salary |

The result can be returned in **any order**, which means sorting is unnecessary unless explicitly desired.

The problem statement guarantees that:

- `Employee.id` and `Department.id` are unique.
- `departmentId` in `Employee` references a valid department.
- Department names are not `NULL`.

Because this is a database problem, the main challenge is not computational complexity in the traditional algorithmic sense, but rather expressing the logic efficiently in SQL.

There are several edge cases that could easily cause bugs in a naive solution:

- **Multiple employees with the same highest salary in a department**. A solution that uses `LIMIT 1` or assumes uniqueness would fail.
- **Departments with only one employee**. That employee should automatically be considered the highest paid.
- **Departments with many employees and repeated salary values**. We must correctly compute the maximum salary and compare employees against it.

The problem guarantees valid foreign keys, so we do not need to worry about employees referencing nonexistent departments.

## Approaches

### Brute Force Approach

A brute force strategy would be to process each department independently.

For every department, we could first scan the `Employee` table to determine the maximum salary for that department. Then, we would scan the employee list again to find all employees whose salary matches that maximum value.

Conceptually, this works because every department is handled independently, and the highest salary is explicitly computed before filtering employees.

However, this method is inefficient because it repeatedly scans the employee table for every department. If there are `D` departments and `E` employees, we may end up traversing employees multiple times, producing a time complexity near **O(D × E)**.

In SQL, repeatedly querying the employee table for each department is also undesirable because databases are optimized for set-based operations rather than repeated scans.

### Optimal Approach

The key observation is that we can compute the **maximum salary per department once**, and then match employees whose salaries equal that maximum.

We can do this efficiently using a **subquery with `GROUP BY`**.

The idea works in two phases:

1. Group employees by `departmentId` and compute the maximum salary for each department.
2. Join this result back to the `Employee` table so that we keep only employees whose salary equals the department maximum.

Finally, we join with the `Department` table to retrieve department names.

This approach is efficient because each table is processed in a set-oriented way, allowing the database optimizer to execute the query effectively.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(D × E) | O(1) | Repeatedly scans employees for every department |
| Optimal | O(E + D) | O(E) | Computes max salary once and joins results |

## Algorithm Walkthrough

1. First, compute the maximum salary for every department using `GROUP BY departmentId`. This gives us a mapping from department ID to the highest salary in that department.
2. Join the resulting table with the `Employee` table. The join condition checks two things: the department ID must match, and the employee salary must equal the department maximum salary. This ensures that only top earners remain.
3. Join the filtered employees with the `Department` table so we can replace department IDs with readable department names.
4. Select the required columns and rename them according to the expected output format:

- `Department`
- `Employee`
- `Salary`
5. Return the results in any order, since ordering is not required.

The important design choice here is using a grouped subquery. Instead of repeatedly searching for the maximum salary, we compute it once and reuse it through joins.

## Python Solution

In database LeetCode problems, the "Python solution" corresponds to the SQL query submitted in the platform.

```
# Write your MySQL query statement below

SELECT
    d.name AS Department,
    e.name AS Employee,
    e.salary AS Salary
FROM Employee e
JOIN (
    SELECT
        departmentId,
        MAX(salary) AS max_salary
    FROM Employee
    GROUP BY departmentId
) max_salaries
    ON e.departmentId = max_salaries.departmentId
    AND e.salary = max_salaries.max_salary
JOIN Department d
    ON e.departmentId = d.id;
```

The implementation begins by creating a derived table called `max_salaries`. This subquery groups employees by department and computes the maximum salary using `MAX(salary)`.

Next, we join this derived table back with `Employee`. The join condition ensures that only employees whose salary matches the department maximum are selected.

Finally, we join with `Department` to retrieve the department name instead of only the department ID.

Because we match on salary equality rather than selecting only one employee, ties are naturally handled. If multiple employees share the highest salary, each matching row appears in the final result.

The query primarily depends on traversing the `Employee` table to compute maximum salaries and then joining against it. Database engines optimize joins heavily, so the practical performance is typically very efficient. Since we only store grouped salary information by department, auxiliary storage remains modest.

## Edge Cases

### Multiple Employees Sharing the Highest Salary

This is the most important edge case. A naive implementation might accidentally return only one employee per department by using `LIMIT 1` or ranking improperly. Our solution avoids this problem by matching employees whose salary equals the maximum salary. Therefore, all tied top earners are included.

### Department With a Single Employee

If a department contains only one employee, that employee is automatically the highest-paid employee. Since `MAX(salary)` returns that employee's salary, the join condition succeeds and the employee appears in the result correctly.

### Duplicate Salary Values Below the Maximum

A department may contain repeated salaries that are not the maximum. For example, two employees might earn `70000`, while another earns `90000`. Our query filters strictly on equality with the department maximum salary, ensuring only the highest-paid employees are returned.

### Large Numbers of Employees

In a department with many employees, repeatedly scanning for the maximum salary would be inefficient. The grouped subquery avoids redundant work by computing each department's maximum salary exactly once, making the query scale efficiently with larger datasets.
