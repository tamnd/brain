## Solution

## Problem Understanding

The problem asks us to find **all root-to-leaf paths** in a binary tree where the sum of the node values equals a given `targetSum`.

A binary tree is provided through its `root`, where each node contains an integer value and pointers to left and right children. We need to explore every possible path that starts at the root and ends at a **leaf node**, meaning a node that has no children. For every valid path, if the sum of all node values along that path equals `targetSum`, we include that path in the final answer.

An important detail is that the result must contain the **values of the nodes**, not references to the nodes themselves. For example, if a valid path traverses nodes with values `5 → 4 → 11 → 2`, the result should include `[5, 4, 11, 2]`.

The constraints tell us several useful things about the problem:

- The tree may contain up to **5000 nodes**, which is large enough that repeated unnecessary traversals could become inefficient.
- Node values and `targetSum` can be negative, which means we cannot assume path sums always increase as we go deeper.
- The tree can be empty (`root = null`), so we must handle that case correctly.

Since the problem specifically asks for **root-to-leaf** paths, partial paths do not count. Even if a path sum equals `targetSum` before reaching a leaf, we cannot include it unless the current node is a leaf.

There are several edge cases worth identifying immediately:

- **Empty tree**: If `root` is `None`, there are no paths to explore, so the answer must be an empty list.
- **Single node tree**: If the root itself is a leaf, we must check whether its value equals `targetSum`.
- **Negative values**: Because node values may be negative, greedy pruning strategies are unsafe. A path sum that exceeds the target may still later become valid.
- **No valid path**: Even after exploring the entire tree, we may find no valid root-to-leaf path, in which case we return an empty list.

## Approaches

### Brute Force Approach

A straightforward but inefficient solution would be to first generate **all root-to-leaf paths**, then calculate the sum of each path afterward.

In this method, we perform a traversal of the tree and store every complete path in a list. Once all paths are collected, we iterate through them again, summing each path to determine whether it equals `targetSum`.

This approach is correct because it exhaustively checks every root-to-leaf path. Since every valid path must appear among those collected, no solution is missed.

However, this method performs unnecessary repeated work. Every path must be traversed again to compute its sum, even though we already visited those nodes during tree traversal. Additionally, storing all paths before filtering increases memory usage.

### Optimal Approach, Depth-First Search with Backtracking

The key observation is that we can evaluate path sums **while traversing the tree**, rather than after traversal.

Since we naturally move from root to leaf in a recursive tree traversal, **Depth-First Search (DFS)** is an ideal fit. At each node, we:

1. Add the current node's value to the path.
2. Reduce the remaining target sum.
3. Check whether we reached a leaf.
4. If the leaf satisfies the target sum, save a copy of the path.
5. Backtrack by removing the current node before returning.

The important idea is **backtracking**. Because recursion reuses the same path list while exploring different branches, we must remove the current node when returning to the parent. Otherwise, values from one branch would incorrectly remain in another branch.

This approach avoids recomputing sums and only traverses the tree once.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(N²) | O(N²) | Generate all paths, then compute sums separately |
| Optimal | O(N²) | O(N) | DFS with backtracking, avoids repeated sum computation |

The `O(N²)` complexity in the worst case comes from copying paths into the result list. In a skewed tree with many valid paths, copying paths of length `O(N)` may happen multiple times.

## Algorithm Walkthrough

1. **Initialize the result container**

Create an empty list called `result` to store all valid root-to-leaf paths.
2. **Define a recursive DFS function**

The recursive function keeps track of:

- The current node
- The remaining sum needed to reach `targetSum`
- The current path of node values

This structure allows us to explore paths incrementally.
3. **Handle the base case**

If the current node is `None`, there is nothing to process, so we immediately return.
4. **Add the current node to the path**

Append the node's value to the current path.

We do this before exploring children because the current node is part of the root-to-current traversal.
5. **Update the remaining target**

Subtract the current node's value from the remaining sum.

Instead of recalculating the total path sum repeatedly, we track the remaining amount needed. This is more efficient and easier to reason about.
6. **Check if the current node is a leaf**

A node is a leaf if both `left` and `right` are `None`.

If we are at a leaf and the remaining sum becomes zero, this path is valid.
7. **Store a copy of the path**

Since `path` is mutable and reused during recursion, append a **copy** of the path to `result`.

Without copying, later modifications would alter previously stored answers.
8. **Explore child nodes**

Recursively call DFS on the left and right children using the updated remaining sum.
9. **Backtrack**

Remove the current node from the path using `pop()`.

This restores the path to its previous state before returning to the parent node.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

class Solution:
    def pathSum(self, root: Optional[TreeNode], targetSum: int) -> List[List[int]]:
        result: List[List[int]] = []

        def dfs(node: Optional[TreeNode], remaining_sum: int, path: List[int]) -> None:
            if not node:
                return

            path.append(node.val)
            remaining_sum -= node.val

            if not node.left and not node.right and remaining_sum == 0:
                result.append(path[:])

            dfs(node.left, remaining_sum, path)
            dfs(node.right, remaining_sum, path)

            path.pop()

        dfs(root, targetSum, [])

        return result
```

The implementation begins by creating a `result` list that will hold all valid paths.

The recursive `dfs()` function drives the traversal. At each node, we add the node value to the current `path` and subtract that value from `remaining_sum`.

When we reach a leaf node, we check whether the remaining sum equals zero. If it does, the current path is valid, so we append a copy using `path[:]`.

The recursive calls explore the left and right subtrees. After exploration finishes, `path.pop()` performs the backtracking step so sibling branches begin with the correct path state.

Finally, the traversal starts from the root using the full `targetSum`, and the accumulated result is returned.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func pathSum(root *TreeNode, targetSum int) [][]int {
	result := [][]int{}

	var dfs func(node *TreeNode, remainingSum int, path []int)

	dfs = func(node *TreeNode, remainingSum int, path []int) {
		if node == nil {
			return
		}

		path = append(path, node.Val)
		remainingSum -= node.Val

		if node.Left == nil && node.Right == nil && remainingSum == 0 {
			pathCopy := make([]int, len(path))
			copy(pathCopy, path)
			result = append(result, pathCopy)
		}

		dfs(node.Left, remainingSum, path)
		dfs(node.Right, remainingSum, path)
	}

	dfs(root, targetSum, [])

	return result
}
```

The Go implementation follows the same recursive DFS strategy, but there are a few language-specific differences.

Go slices are reference-like structures, so when saving a valid path we must explicitly create a new slice using `make()` and `copy()`. If we directly append `path`, future recursive modifications would corrupt stored results.

Unlike Python, Go handles missing nodes using `nil` rather than `None`.

One subtle difference is that explicit backtracking with `pop()` is unnecessary here because slices are passed by value. Each recursive call receives its own slice header, so appending creates the desired isolation between branches.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(N²) | Each node is visited once, but path copying may cost O(N) per valid path |
| Space | O(N) | Recursion stack and current path storage |

The DFS traversal itself visits every node exactly once, which suggests `O(N)` time. However, whenever a valid path is found, we must copy the current path. In the worst case, a skewed tree can contain paths of length `O(N)`, leading to `O(N²)` total work.

Space complexity is `O(N)` because the recursion stack and current path may grow to the height of the tree. In the worst case of a skewed tree, the height becomes `N`.

## Edge Cases

### Empty Tree

If `root` is `None`, there are no nodes to traverse and therefore no root-to-leaf paths. A naive implementation might forget this and attempt recursion on a null node. Our solution handles this naturally because the first DFS call immediately returns when encountering `None`.

### Single Node Tree

A tree containing only one node is also a root-to-leaf path. This case is easy to overlook because there are no child nodes to traverse. Our implementation explicitly checks for leaf nodes and validates the remaining sum, so a single-node tree works correctly.

For example:

```
root = [5]
targetSum = 5
```

The result becomes:

```
[[5]]
```

### Negative Values in Nodes

Since node values may be negative, it is incorrect to stop traversal early when the running sum becomes larger than `targetSum`. A later negative value might bring the sum back to the target.

For example:

```
root = [5,4,-3]
targetSum = 6
```

Even if the running sum temporarily exceeds the target, the `-3` can still make the final path valid. Our DFS explores every root-to-leaf path without unsafe pruning, guaranteeing correctness.

### Path Sum Reached Before a Leaf

A common bug is to stop once the sum matches `targetSum`, even if the node is not a leaf.

For example:

```
root = [5,4,8]
targetSum = 9
```

The partial path `[5,4]` sums to `9`, but node `4` is not necessarily a leaf. Since the problem specifically requires **root-to-leaf** paths, our implementation only accepts paths when both child pointers are `None`.
