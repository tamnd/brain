## Solution

## Problem Understanding

This problem asks us to identify employees whose salary is greater than the salary of their direct manager. The input consists of a single database table named `Employee`, where each row represents an employee and contains four pieces of information: their unique employee ID (`id`), their name (`name`), their salary (`salary`), and the ID of their manager (`managerId`).

The important detail is that `managerId` refers to another employee's `id` in the same table. This means the table is self-referential. In other words, an employee and their manager are both stored in the same dataset, and the relationship is established through IDs.

The goal is to return the names of employees who earn strictly more than their managers. The output should contain a single column named `Employee`, which stores the names of qualifying employees. The order of results does not matter.

Looking at the example:

| id | name | salary | managerId |
| --- | --- | --- | --- |
| 1 | Joe | 70000 | 3 |
| 2 | Henry | 80000 | 4 |
| 3 | Sam | 60000 | NULL |
| 4 | Max | 90000 | NULL |

Joe reports to Sam (`managerId = 3`). Joe earns `70000`, while Sam earns `60000`, so Joe qualifies. Henry reports to Max, but Henry earns less than Max, so Henry does not qualify.

The problem is relatively small in scale because it is labeled as an Easy database problem, but efficiency and correctness still matter. Since managers are stored in the same table, the main challenge is comparing each employee with another row in that same table.

An important constraint is that some employees may not have managers. These employees will have `managerId = NULL`, and they should not be considered because there is no manager salary to compare against.

Several edge cases are important to consider. Employees without managers should be ignored. Employees whose salary is exactly equal to their manager's salary should not be included because the comparison is strictly greater than. There is also the possibility that no employees satisfy the condition, in which case the query should return an empty result set.

## Approaches

### Brute Force Approach

A brute force approach would compare every employee against every other employee in the table to determine whether one is their manager. For each employee, we could scan the entire table to find the row where `employee.managerId = manager.id`. Once the manager is found, we compare salaries and include the employee if their salary is greater.

This approach is correct because it explicitly checks every possible manager relationship. However, it is inefficient because for each employee we potentially scan the whole table again.

If there are `n` employees, the worst case time complexity becomes `O(n²)` since each employee may require searching through all employees to find their manager.

### Optimal Approach

The key observation is that the manager already exists in the same table. SQL provides a natural way to compare rows within the same table using a **self join**.

A self join allows us to treat the `Employee` table as two separate logical tables:

- One alias represents the employee.
- Another alias represents the manager.

We join the table where:

```
employee.managerId = manager.id
```

This directly links each employee to their manager. Once the relationship is established, we simply filter for employees whose salary exceeds their manager's salary.

This approach is more efficient and much cleaner than repeatedly scanning the table manually.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Compare every employee against every other employee to locate managers |
| Optimal | O(n) | O(1) | Use a self join to directly connect employees with managers |

## Algorithm Walkthrough

1. Start with the `Employee` table and conceptually create two aliases of it, one representing employees and one representing managers.
2. Join the employee alias with the manager alias using the condition:

```
employee.managerId = manager.id
```

This step is necessary because managers are stored in the same table. The join establishes the reporting relationship.
3. Compare the salaries of the employee and manager:

```
employee.salary > manager.salary
```

We use a strict greater-than comparison because employees earning exactly the same salary should not be included.
4. Select only the employee's name and rename the output column to `Employee`, matching the required output format.
5. Return the result in any order.

The resulting SQL query is concise because the relational structure of the table maps naturally to a self join.

## Python Solution

Since this is a Database problem, LeetCode expects an SQL query rather than executable Python code. Below is the complete, LeetCode-submittable SQL solution.

```
# This problem requires an SQL query, not Python code.

SELECT
    e.name AS Employee
FROM Employee e
JOIN Employee m
    ON e.managerId = m.id
WHERE e.salary > m.salary;
```

The implementation works by creating two aliases of the `Employee` table. The alias `e` represents employees, while `m` represents managers. The `JOIN` condition connects an employee to their manager using `managerId = id`.

After establishing the relationship, the `WHERE` clause filters rows where the employee's salary exceeds the manager's salary. Finally, we select only the employee name and rename the output column to `Employee`, which matches the required result format exactly.

## Edge Cases

One important edge case is employees without managers. These employees have `managerId = NULL`, which means there is no manager row to compare against. A naive implementation might accidentally include them or cause invalid comparisons. The self join naturally avoids this issue because rows with `NULL` manager IDs do not match any manager record.

Another important case is when an employee earns exactly the same salary as their manager. Since the problem specifies employees who earn **more than** their managers, equality should not count. The query correctly handles this by using the strict comparison operator `>` rather than `>=`.

A third edge case occurs when no employee earns more than their manager. In this situation, the query should return an empty result set rather than an error or incorrect data. The `WHERE` clause naturally filters out all rows, producing an empty table.

A final edge case is when multiple employees report to the same manager. The query handles this correctly because each employee-manager relationship is evaluated independently through the join, allowing multiple qualifying employees to appear in the result.
