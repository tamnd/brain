## Solution

## Problem Understanding

This problem gives us two arrays representing different traversal orders of the same binary tree:

- `preorder`, which contains the preorder traversal of the tree
- `inorder`, which contains the inorder traversal of the same tree

Our task is to reconstruct the original binary tree and return its root node.

To solve this problem correctly, we first need to understand what these traversals mean.

In a **preorder traversal**, nodes are visited in the following order:

1. Visit the root node
2. Traverse the left subtree
3. Traverse the right subtree

This means that the very first element in `preorder` is always the root of the current subtree.

In an **inorder traversal**, nodes are visited in this order:

1. Traverse the left subtree
2. Visit the root node
3. Traverse the right subtree

This ordering is important because it tells us how the tree is split into left and right subtrees.

The input arrays contain unique values, which is a very important guarantee. Since values are unique, every node value appears exactly once in `inorder`, meaning we can unambiguously determine where the left subtree ends and the right subtree begins.

The constraints tell us that the input size can be as large as `3000`. This means an inefficient `O(n┬▓)` solution may still pass in some cases, but it is not ideal. Since tree reconstruction naturally suggests recursive subdivision, we should aim for an `O(n)` solution.

There are several important observations and edge cases:

A tree may contain only a single node, such as `preorder = [-1]` and `inorder = [-1]`. In this case, we simply create one node and return it.

The tree may be highly skewed, meaning every node only has a left child or only a right child. Such cases can cause inefficient implementations to repeatedly scan large portions of arrays.

A naive implementation that repeatedly slices arrays or linearly searches for node positions can become unnecessarily expensive.

The problem guarantees that the input is always valid. We do not need to handle malformed traversals or mismatched trees because `preorder` and `inorder` are guaranteed to represent the same valid binary tree.

## Approaches

### Brute Force Approach

The brute-force approach follows the recursive structure of tree construction directly.

We know the first element in `preorder` is the root of the current subtree. Once we identify this root, we search for it inside `inorder`. Everything to the left of the root belongs to the left subtree, and everything to the right belongs to the right subtree.

After splitting the tree into left and right sections, we recursively repeat the process for both sides.

The problem with this approach is that each recursive call performs a linear search through `inorder` to find the root position. Since this search costs `O(n)` and occurs for many nodes, the total runtime can degrade to `O(n┬▓)` in skewed trees.

Additionally, repeatedly creating sliced arrays introduces extra memory overhead.

Although this method is conceptually simple and correct, it is inefficient for larger inputs.

### Optimal Approach

The key insight is that we repeatedly search for node positions inside `inorder`. Since node values are unique, we can preprocess `inorder` into a hash map that stores:

```
value -> index
```

This allows us to find the inorder position of any node in constant time.

Instead of slicing arrays, we use index boundaries to represent subtrees. This avoids repeatedly allocating new arrays.

The recursive idea remains the same:

- The current root always comes from preorder
- The inorder position splits the tree into left and right subtrees
- Recursively construct both subtrees using index ranges

We maintain a pointer into `preorder` so each node is processed exactly once.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n┬▓) | O(n) | Repeated linear search in inorder, may also incur slicing overhead |
| Optimal | O(n) | O(n) | Hash map enables constant-time lookups, each node processed once |

## Algorithm Walkthrough

1. Create a hash map that stores the index of every value in `inorder`.

This allows us to instantly determine where a node appears in the inorder traversal. Since values are unique, every lookup is unambiguous.
2. Maintain a `preorder_index` pointer.

Since preorder always visits the root first, we process nodes in preorder order. Instead of modifying the array, we use a pointer to track which node should be constructed next.
3. Define a recursive helper function.

The helper receives two boundaries, `left` and `right`, representing the current inorder segment.

If `left > right`, it means the subtree is empty, so return `None`.
4. Read the next root value from preorder.

Use `preorder[preorder_index]` as the root value, then increment `preorder_index`.

Create a new tree node using this value.
5. Find the root position in inorder.

Use the hash map to get the root index in constant time.

Everything to the left belongs to the left subtree.

Everything to the right belongs to the right subtree.
6. Recursively build the left subtree.

Since preorder visits left children immediately after the root, we build the left subtree first.

The inorder range becomes:

```
left to root_index - 1
```
7. Recursively build the right subtree.

After finishing the left subtree, preorder naturally continues into the right subtree.

The inorder range becomes:

```
root_index + 1 to right
```
8. Return the constructed node.

Each recursive call returns the root of the subtree it built.
9. Start recursion using the full inorder range.

Begin with:

```
0 to len(inorder) - 1
```

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

class Solution:
    def buildTree(self, preorder: List[int], inorder: List[int]) -> Optional[TreeNode]:
        inorder_index_map = {
            value: index for index, value in enumerate(inorder)
        }

        preorder_index = 0

        def build_subtree(left: int, right: int) -> Optional[TreeNode]:
            nonlocal preorder_index

            if left > right:
                return None

            root_value = preorder[preorder_index]
            preorder_index += 1

            root = TreeNode(root_value)

            inorder_root_index = inorder_index_map[root_value]

            root.left = build_subtree(
                left,
                inorder_root_index - 1
            )

            root.right = build_subtree(
                inorder_root_index + 1,
                right
            )

            return root

        return build_subtree(0, len(inorder) - 1)
```

The implementation starts by preprocessing `inorder` into a hash map. This is the optimization that removes repeated linear scans and makes root lookup constant time.

The `preorder_index` variable tracks which node should be processed next. Since preorder traversal always visits roots first, we can sequentially consume values from the preorder array.

The recursive helper `build_subtree(left, right)` represents the subtree whose inorder traversal lies between the indices `left` and `right`.

The base case occurs when `left > right`. This means there are no nodes in the subtree, so we return `None`.

For valid ranges, we take the next preorder value as the root, create a node, and locate its inorder position using the hash map.

The inorder position naturally partitions the tree. We recursively build the left subtree first because preorder traversal visits left descendants immediately after the root. Then we build the right subtree.

Finally, the constructed root node is returned upward through recursion.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func buildTree(preorder []int, inorder []int) *TreeNode {
	inorderIndexMap := make(map[int]int)

	for index, value := range inorder {
		inorderIndexMap[value] = index
	}

	preorderIndex := 0

	var buildSubtree func(left int, right int) *TreeNode

	buildSubtree = func(left int, right int) *TreeNode {
		if left > right {
			return nil
		}

		rootValue := preorder[preorderIndex]
		preorderIndex++

		root := &TreeNode{
			Val: rootValue,
		}

		inorderRootIndex := inorderIndexMap[rootValue]

		root.Left = buildSubtree(
			left,
			inorderRootIndex-1,
		)

		root.Right = buildSubtree(
			inorderRootIndex+1,
			right,
		)

		return root
	}

	return buildSubtree(0, len(inorder)-1)
}
```

The Go implementation follows exactly the same algorithmic structure as the Python version. A `map[int]int` is used for constant-time inorder index lookup.

Since Go does not support nested functions with `nonlocal`, the recursive helper closes over the `preorderIndex` variable directly.

For empty subtrees, Go returns `nil` instead of Python's `None`.

No integer overflow concerns exist here because values are constrained between `-3000` and `3000`, well within Go's integer range.

Slices are passed by reference-like semantics, so we avoid copying arrays and maintain efficient memory usage.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every node is visited once, and hash map lookup is O(1) |
| Space | O(n) | Hash map stores all nodes, recursion stack may also grow to O(n) |

The algorithm runs in linear time because every node is processed exactly once. Each node creation, hash map lookup, and recursive operation contributes constant work.

The space complexity is `O(n)` due to two factors. First, the hash map stores an entry for every node. Second, the recursive call stack may grow to `O(n)` in the worst case for a skewed tree.

## Edge Cases

### Single Node Tree

A tree containing only one node is the smallest valid input.

```
preorder = [1]
inorder = [1]
```

This case can expose bugs in recursive boundaries if the implementation incorrectly assumes children exist. Our implementation handles it naturally because recursive calls immediately encounter invalid ranges and return `None`.

### Completely Left-Skewed Tree

Consider:

```
preorder = [3,2,1]
inorder = [1,2,3]
```

Every node only has a left child. This creates the worst-case recursion depth.

A naive implementation that repeatedly scans `inorder` becomes inefficient here, degrading toward `O(n┬▓)`. Our hash map optimization ensures constant-time lookups, preserving `O(n)` runtime.

### Completely Right-Skewed Tree

Consider:

```
preorder = [1,2,3]
inorder = [1,2,3]
```

Every node only has a right child.

This case is important because preorder traversal continuously consumes nodes while left subtree ranges remain empty. Incorrect recursion order could accidentally assign nodes to the wrong subtree. Since our implementation always constructs the left subtree before the right subtree, following preorder semantics, the tree is reconstructed correctly.

### Negative Values

The constraints allow values between `-3000` and `3000`.

```
preorder = [-1,-2,-3]
inorder = [-2,-1,-3]
```

Negative numbers can sometimes reveal assumptions about indexing or positivity. Since we only use values as hash map keys and never as array indices, negative values are handled safely and correctly.
