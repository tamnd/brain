## Solution

## Problem Understanding

The problem describes a circular route of gas stations. Each station provides some amount of fuel, represented by the `gas` array, and traveling from one station to the next consumes fuel, represented by the `cost` array.

At station `i`:

- You can collect `gas[i]` units of fuel.
- It costs `cost[i]` units of fuel to travel to station `(i + 1) % n`.

Your car starts with an empty tank, but the tank has unlimited capacity. The goal is to determine whether there exists a station from which you can start, travel around the entire circle exactly once, and return to the starting point without the fuel in the tank ever becoming negative.

If such a station exists, you return its index. Otherwise, you return `-1`.

The problem guarantees that if a valid solution exists, it is unique.

The constraints are important:

- `n` can be as large as `10^5`
- Each gas and cost value can be up to `10^4`

These limits immediately suggest that an `O(n^2)` solution may be too slow in the worst case. A more efficient linear solution is required.

There are several important edge cases to think about:

- The total amount of gas may be less than the total travel cost. In this case, completing the circuit is impossible regardless of the starting point.
- The valid starting station might appear near the end of the array, requiring wraparound behavior.
- Some stations may provide zero gas.
- Some intermediate paths may temporarily fail even though a later station works.
- Since the answer is guaranteed to be unique, once we identify a valid start, we do not need to handle multiple candidates.

## Approaches

### Brute Force Approach

The most straightforward approach is to try every station as a starting point.

For each station:

1. Start with an empty tank.
2. Simulate traveling around the entire circuit.
3. At every station:

- Add gas from the current station.
- Subtract the travel cost to the next station.
4. If the tank ever becomes negative, this starting point fails.
5. If we successfully return to the starting station, return that index.

This approach is correct because it explicitly checks every possible starting position and directly simulates the journey.

However, it is inefficient. In the worst case, for every starting station we may traverse nearly the entire circuit before failing. With `n` stations, this leads to `O(n^2)` time complexity, which is too slow for `n = 10^5`.

### Optimal Greedy Approach

The key insight comes from understanding when a starting station can be eliminated.

Suppose we start at station `start` and eventually fail at station `i`. This means the tank becomes negative somewhere between `start` and `i`.

An important observation is:

If starting from `start` cannot reach `i + 1`, then no station between `start` and `i` can reach `i + 1` either.

Why?

Because any station between `start` and `i` would begin with even less accumulated fuel than starting at `start`. Therefore, all those stations are guaranteed failures.

This allows us to skip many unnecessary checks.

Another critical condition is:

- If total gas < total cost, completing the circuit is impossible.

So the algorithm becomes:

1. Compute total gas and total cost.
2. If total gas < total cost, return `-1`.
3. Traverse the stations once while maintaining a running tank balance.
4. Whenever the tank becomes negative:

- The current starting point is invalid.
- Reset the tank to `0`.
- Set the next station as the new candidate start.

This greedy approach works because every failed segment eliminates an entire range of impossible starting positions.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Simulates the journey starting from every station |
| Optimal | O(n) | O(1) | Greedy elimination of impossible starting positions |

## Algorithm Walkthrough

1. Initialize three variables:

- `total_tank`, which tracks the total gas surplus across the entire circuit.
- `current_tank`, which tracks the gas balance for the current candidate starting station.
- `start_index`, which stores the current candidate starting station.
2. Traverse the arrays from left to right.
3. At each station:

- Compute `net_gain = gas[i] - cost[i]`.
- Add `net_gain` to both `total_tank` and `current_tank`.
4. If `current_tank` becomes negative:

- The current candidate start cannot complete the circuit.
- Any station between `start_index` and `i` also cannot work.
- Set `start_index = i + 1`.
- Reset `current_tank = 0`.
5. After finishing the traversal:

- If `total_tank < 0`, return `-1`.
- Otherwise, return `start_index`.

The algorithm only requires a single pass through the arrays, making it extremely efficient.

## Python Solution

```
from typing import List

class Solution:
    def canCompleteCircuit(self, gas: List[int], cost: List[int]) -> int:
        total_tank = 0
        current_tank = 0
        start_index = 0

        for i in range(len(gas)):
            net_gain = gas[i] - cost[i]

            total_tank += net_gain
            current_tank += net_gain

            if current_tank < 0:
                start_index = i + 1
                current_tank = 0

        if total_tank < 0:
            return -1

        return start_index
```

The implementation follows the greedy strategy directly.

The variable `total_tank` determines whether completing the circuit is globally possible. If the total gas available is less than the total travel cost, there is no valid answer.

The variable `current_tank` tracks whether the current candidate starting station remains feasible while traversing the route.

Whenever `current_tank` becomes negative, the algorithm immediately discards the current starting station and all stations before the failure point. This is the central greedy insight that reduces the complexity from quadratic to linear.

The final answer is returned only if the total fuel balance is nonnegative.

## Go Solution

```
func canCompleteCircuit(gas []int, cost []int) int {
    totalTank := 0
    currentTank := 0
    startIndex := 0

    for i := 0; i < len(gas); i++ {
        netGain := gas[i] - cost[i]

        totalTank += netGain
        currentTank += netGain

        if currentTank < 0 {
            startIndex = i + 1
            currentTank = 0
        }
    }

    if totalTank < 0 {
        return -1
    }

    return startIndex
}
```

The Go implementation is almost identical to the Python version.

Go slices are used directly for the input arrays. Since the constraints guarantee values remain well within 32 bit integer limits, standard `int` arithmetic is sufficient.

Unlike Python, Go does not require explicit type hints or imports for slices in this example.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | The arrays are traversed exactly once |
| Space | O(1) | Only a few integer variables are used |

The greedy algorithm achieves linear time complexity because each station is processed once, and failed ranges are skipped entirely instead of being rechecked.

The space complexity is constant because no additional data structures proportional to the input size are allocated.

## Edge Cases

### Total Gas Less Than Total Cost

Consider:

```
gas  = [1,2,3]
cost = [4,4,4]
```

The total gas is `6`, while the total cost is `12`.

No matter where the journey starts, the car will eventually run out of fuel. The implementation correctly detects this using `total_tank < 0` and immediately returns `-1`.

### Valid Starting Station Near the End

Consider:

```
gas  = [1,1,5]
cost = [2,2,1]
```

The correct starting station is index `2`.

A naive implementation that mishandles circular traversal could fail here. The greedy solution naturally handles wraparound behavior because the mathematical reasoning guarantees the identified starting station works for the entire cycle.

### Multiple Early Failures Before Success

Consider:

```
gas  = [1,2,3,10]
cost = [2,3,4,1]
```

The algorithm repeatedly fails at early stations and resets the candidate start.

This is important because a naive implementation might repeatedly retry stations that are already proven invalid. The greedy observation ensures those stations are skipped permanently, preserving linear performance.

### Single Station Input

Consider:

```
gas  = [5]
cost = [4]
```

There is only one station, and it provides enough fuel to return to itself.

The algorithm handles this naturally because the loop runs once, `total_tank` remains nonnegative, and the starting index `0` is returned correctly.
