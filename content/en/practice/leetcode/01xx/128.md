## Solution

## Problem Understanding

The problem gives us an unsorted array of integers and asks for the length of the longest sequence of consecutive numbers.

A consecutive sequence means numbers that increase by exactly 1. The numbers do not need to appear next to each other in the original array, because the array is unsorted. For example, in the array `[100,4,200,1,3,2]`, the sequence `[1,2,3,4]` exists even though those values are scattered across the input.

The task is not to return the sequence itself, only its length.

One very important requirement is that the algorithm must run in `O(n)` time. This immediately rules out straightforward sorting approaches, because sorting usually costs `O(n log n)` time.

The input constraints tell us several important things:

- The array can contain up to `10^5` elements, so inefficient nested loop solutions will time out.
- Numbers can be negative, so we cannot rely on array indexing tricks based on value ranges.
- Duplicates may exist, which means we must handle repeated numbers carefully.
- The array may be empty, in which case the answer should be `0`.

Some important edge cases include:

- An empty array, where no sequence exists.
- Arrays with duplicates, such as `[1,2,2,3]`.
- Arrays with negative numbers, such as `[-3,-2,-1,0]`.
- Arrays where every number is isolated, such as `[10,30,50]`.
- Arrays where the longest sequence spans the entire input.

Understanding these cases upfront helps avoid subtle bugs later in the implementation.

## Approaches

### Brute Force Approach

A naive solution is to start from every number and repeatedly check whether the next consecutive number exists.

For example, if we start at `1`, we check whether `2` exists, then whether `3` exists, then whether `4` exists, and continue until the sequence ends.

The problem with this approach is that checking whether a number exists in a plain array requires scanning the array, which costs `O(n)` time. Since we may do this for every element, the overall complexity can become `O(n^2)`.

Another possible brute-force approach is sorting the array first and then scanning linearly to count consecutive streaks. That works correctly, but sorting costs `O(n log n)`, which violates the strict `O(n)` requirement.

### Key Insight for the Optimal Solution

The key observation is that we only need to start counting from numbers that are the beginning of a sequence.

Suppose we have the numbers `[1,2,3,4]`.

If we start from `1`, we can discover the whole sequence.

If we also start from `2`, `3`, and `4`, we would repeat unnecessary work.

So how do we identify the beginning of a sequence?

A number is the start of a sequence if `num - 1` does not exist in the set.

For example:

- `1` is a sequence start because `0` is absent.
- `2` is not a sequence start because `1` exists.
- `3` is not a sequence start because `2` exists.

Using a hash set allows constant average time lookups. This lets us:

1. Insert all numbers into a set.
2. Only begin sequence expansion from valid starting points.
3. Expand each sequence exactly once.

This produces an overall `O(n)` solution.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Repeatedly scans the array for consecutive values |
| Sorting Based | O(n log n) | O(1) or O(n) | Easier to implement, but violates required time complexity |
| Optimal Hash Set | O(n) | O(n) | Uses a set for constant-time lookups and avoids duplicate work |

## Algorithm Walkthrough

1. Create a hash set containing all numbers from the input array.

The set allows average `O(1)` lookup time. This is critical because we need to quickly determine whether a consecutive number exists.
2. Initialize a variable to store the maximum sequence length found so far.

This variable will be updated whenever we discover a longer consecutive sequence.
3. Iterate through every number in the set.

We iterate through the set instead of the original array because duplicates are automatically removed.
4. Check whether the current number is the start of a sequence.

A number is a sequence start if `num - 1` is not present in the set.

This step is the key optimization. It prevents us from repeatedly traversing the same sequence.
5. If the number is a sequence start, begin expanding the sequence.

Start with:

- `current_number = num`
- `current_length = 1`
6. Repeatedly check whether `current_number + 1` exists in the set.

If it exists:

- Increment `current_number`
- Increment `current_length`

Continue until the next consecutive number is missing.
7. Update the maximum sequence length.

Compare `current_length` with the best answer seen so far.
8. After processing all numbers, return the maximum length.

## Python Solution

```
from typing import List

class Solution:
    def longestConsecutive(self, nums: List[int]) -> int:
        number_set = set(nums)
        longest_streak = 0

        for num in number_set:
            # Only start counting if this is the beginning
            # of a sequence
            if num - 1 not in number_set:
                current_number = num
                current_streak = 1

                while current_number + 1 in number_set:
                    current_number += 1
                    current_streak += 1

                longest_streak = max(longest_streak, current_streak)

        return longest_streak
```

The implementation begins by converting the input list into a set. This removes duplicates and enables constant average time membership checks.

The variable `longest_streak` stores the best answer found so far.

The loop iterates through every unique number in the set. For each number, the code first checks whether `num - 1` exists. If it does, then the current number belongs to a sequence that was already processed earlier, so we skip it.

If `num - 1` does not exist, then the current number is the beginning of a new sequence. The algorithm then expands forward using a `while` loop, checking for consecutive values one by one.

As the sequence grows, `current_streak` tracks its length. After the sequence ends, the algorithm updates `longest_streak` if necessary.

Finally, the function returns the maximum sequence length discovered.

## Go Solution

```
func longestConsecutive(nums []int) int {
	numberSet := make(map[int]bool)

	for _, num := range nums {
		numberSet[num] = true
	}

	longestStreak := 0

	for num := range numberSet {
		// Only start counting if this is the
		// beginning of a sequence
		if !numberSet[num-1] {
			currentNumber := num
			currentStreak := 1

			for numberSet[currentNumber+1] {
				currentNumber++
				currentStreak++
			}

			if currentStreak > longestStreak {
				longestStreak = currentStreak
			}
		}
	}

	return longestStreak
}
```

The Go implementation follows the same logic as the Python version, but uses a `map[int]bool` instead of a Python set.

Go does not have a built-in set type, so a map is the standard substitute. Each number is inserted into the map with value `true`.

The iteration happens over the map keys, which naturally removes duplicates.

Unlike Python, Go distinguishes between nil slices and empty slices, but both are handled correctly here because iterating over an empty slice simply performs zero iterations.

Integer overflow is not a concern under the given constraints because values remain well within Go's `int` range.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each number is inserted and processed at most once |
| Space | O(n) | The hash set stores all unique numbers |

Although there is a nested `while` loop, the total work across all expansions is still linear. Each number participates in sequence expansion only once because sequences are only started from valid starting points. This guarantees that the total number of membership checks remains proportional to `n`.

## Edge Cases

### Empty Array

If the input array is empty, the set is also empty. The loop never executes, and `longest_streak` remains `0`.

This case is important because some implementations incorrectly assume at least one element exists and may initialize the answer to `1`, which would produce the wrong result.

### Duplicate Values

Consider the input:

```
[1,2,2,3]
```

Duplicates could cause repeated processing or incorrect sequence lengths if handled poorly.

The implementation avoids this issue by converting the array into a set first. The set becomes:

```
{1,2,3}
```

The duplicate `2` disappears automatically, ensuring correct counting and preventing unnecessary work.

### Negative Numbers

Consider:

```
[-3,-2,-1,0]
```

Some solutions incorrectly assume numbers are non-negative and attempt array indexing tricks.

The hash set approach works naturally with negative values because hash tables support arbitrary integer keys. The algorithm correctly identifies `-3` as the sequence start and expands through the rest of the sequence.

### Single Element Array

For input:

```
[42]
```

The number `42` is a valid sequence of length `1`.

The implementation correctly handles this because `41` does not exist, so the algorithm starts a new sequence and returns `1`.

### Disconnected Numbers

Consider:

```
[10,30,50]
```

Each number forms its own sequence of length `1`.

The algorithm treats each number as a sequence start because no predecessor exists for any of them. The maximum sequence length remains `1`, which is the correct answer.
