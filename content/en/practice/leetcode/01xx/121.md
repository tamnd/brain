# Best Time to Buy and Sell Stock

## Problem Understanding

The problem gives us an integer array called `prices`, where each element represents the stock price on a specific day. The value at index `i` corresponds to the stock price on the `i`th day.

We are allowed to complete exactly one transaction, which means:

1. Buy the stock on one day
2. Sell the stock on a later day

The important restriction is that the sell day must come after the buy day. We cannot sell before we buy.

Our goal is to compute the maximum possible profit from a single transaction. Profit is calculated as:

```
sell_price - buy_price
```

If no profitable transaction exists, we return `0`.

For example:

- If prices rise after a low point, we want to buy at the lowest earlier price and sell at the highest later price.
- If prices continuously decrease, no profitable transaction is possible, so the answer is `0`.

The constraints are important:

- `1 <= prices.length <= 10^5`
- `0 <= prices[i] <= 10^4`

The array can contain up to 100,000 elements, which means inefficient algorithms such as nested loops may become too slow. This strongly suggests that we need a linear time solution.

Several edge cases are important to recognize early:

- Arrays with only one element, where selling is impossible
- Strictly decreasing prices, where profit should remain `0`
- Multiple local minimums and maximums
- Repeated values
- The maximum profit occurring near the end of the array

A correct solution must always ensure that buying happens before selling.

## Approaches

### Brute Force Approach

The most straightforward idea is to try every possible pair of days.

For each day `i`, we treat it as the buying day. Then we check every later day `j` and calculate the profit:

```
prices[j] - prices[i]
```

We keep track of the maximum profit found across all pairs.

This approach is correct because it exhaustively checks every valid transaction. Since every possible buy-sell combination is evaluated, the maximum profit cannot be missed.

However, the problem is efficiency. With two nested loops:

- The outer loop runs `n` times
- The inner loop also runs up to `n` times

This leads to `O(n^2)` time complexity.

For `n = 100000`, this becomes far too slow.

### Optimal Approach

The key observation is that when we are at a particular day, we only need two pieces of information:

1. The lowest stock price seen so far
2. The best profit seen so far

As we scan from left to right:

- We continuously update the minimum price
- At each day, we compute the profit if we sold today
- We compare that profit against the best answer so far

This works because the optimal transaction for a selling day depends only on the cheapest earlier buying day.

Instead of comparing every pair, we maintain the best candidate buy price while traversing the array once.

This reduces the complexity to linear time.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Checks every possible buy-sell pair |
| Optimal | O(n) | O(1) | Tracks minimum price and maximum profit in one pass |

## Algorithm Walkthrough

1. Initialize a variable `minimum_price` with the first stock price. This represents the cheapest stock price we have seen so far.
2. Initialize another variable `maximum_profit` to `0`. This stores the best profit found during traversal.
3. Traverse the array starting from the second day.
4. For each current price:

- Compute the profit if we sold today:

```
current_price - minimum_price
```

- Compare this profit with `maximum_profit` and update if larger.
5. After calculating profit, check whether the current price is lower than `minimum_price`.

- If yes, update `minimum_price`
- This means we found a better day to buy
6. Continue until the end of the array.
7. Return `maximum_profit`.

This algorithm works because at every position we know the best possible buying price from all previous days. Therefore, the profit calculation is always valid and respects the rule that buying must happen before selling.

## Python Solution

```
from typing import List

class Solution:
    def maxProfit(self, prices: List[int]) -> int:
        minimum_price = prices[0]
        maximum_profit = 0

        for current_price in prices[1:]:
            potential_profit = current_price - minimum_price

            if potential_profit > maximum_profit:
                maximum_profit = potential_profit

            if current_price < minimum_price:
                minimum_price = current_price

        return maximum_profit
```

The implementation begins by storing the first price as the initial minimum price. Since we must buy before selling, the earliest day is the first valid candidate.

The variable `maximum_profit` starts at `0` because the problem states that if no profitable transaction exists, we should return `0`.

The loop iterates through all remaining prices. For each price:

- We first compute the profit obtained by selling on the current day
- We update the best answer if this profit is larger
- Then we check whether the current price becomes the new minimum buying price

The order is important because we must calculate profit using a previous buying day, not the current day itself.

The solution uses constant extra space and processes the array only once.

## Go Solution

```
func maxProfit(prices []int) int {
    minimumPrice := prices[0]
    maximumProfit := 0

    for i := 1; i < len(prices); i++ {
        currentPrice := prices[i]

        potentialProfit := currentPrice - minimumPrice

        if potentialProfit > maximumProfit {
            maximumProfit = potentialProfit
        }

        if currentPrice < minimumPrice {
            minimumPrice = currentPrice
        }
    }

    return maximumProfit
}
```

The Go implementation follows the same logic as the Python solution.

Slices in Go are dynamically sized views over arrays, so iterating through `prices` is efficient.

No special handling for empty input is needed because the constraints guarantee that the array contains at least one element.

Integer overflow is not a concern because the maximum price is only `10^4`, well within Go's integer range.

Input:

```
prices = [7,1,5,3,6,4]
```

Initial state:

```
minimum_price = 7
maximum_profit = 0
```

| Day | Current Price | Minimum Price So Far | Potential Profit | Maximum Profit |
| --- | --- | --- | --- | --- |

| 2 | 5 | 1 | 5 - 1 = 4 | 4 |
| 3 | 3 | 1 | 3 - 1 = 2 | 4 |
| 4 | 6 | 1 | 6 - 1 = 5 | 5 |
| 5 | 4 | 1 | 4 - 1 = 3 | 5 |

Final answer:

```
5
```

The best transaction is:

- Buy at price `1`
- Sell at price `6`

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | The array is traversed exactly once |

The algorithm processes each element a single time and performs constant work per iteration. No auxiliary data structures proportional to the input size are created, so the extra memory usage remains constant.

## Edge Cases

### Single Element Array

Consider:

```
prices = [5]
```

There is no future day available for selling, so no transaction can occur.

The implementation handles this naturally because the loop over `prices[1:]` executes zero times, and `maximum_profit` remains `0`.

### Strictly Decreasing Prices

Consider:

```
prices = [9,7,5,3,1]
```

Every possible transaction results in a loss.

A common bug is returning a negative value. Our implementation avoids this because `maximum_profit` starts at `0` and is only updated when a larger positive profit appears.

### Repeated Prices

Consider:

```
prices = [3,3,3,3]
```

Buying and selling at equal prices produces zero profit.

The algorithm correctly computes zero throughout traversal and returns `0`.

### Profit Appears Late

Consider:

```
prices = [8,6,5,2,10]
```

The minimum price occurs near the end, and the maximum profit is achieved afterward.

The algorithm continuously updates `minimum_price`, ensuring that late opportunities are not missed.

### Minimum Price After Large Profit

Consider:

```
prices = [1,10,0,5]
```

The best transaction is still buying at `1` and selling at `10`, even though a smaller price appears later.

The algorithm correctly preserves the earlier larger profit in `maximum_profit` while continuing to scan future prices.
