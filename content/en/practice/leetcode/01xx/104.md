## Solution

## Problem Understanding

The problem gives us the root node of a binary tree and asks us to compute the tree's maximum depth.

A binary tree's depth is defined as the number of nodes along a path from the root down to a leaf node. A leaf node is a node with no children. The maximum depth is therefore the length of the longest root-to-leaf path in the tree.

For example, consider the tree:

```
        3
       / \
      9  20
         / \
        15  7
```

The longest path is either:

```
3 -> 20 -> 15
```

or

```
3 -> 20 -> 7
```

Both paths contain 3 nodes, so the maximum depth is 3.

The input is provided as the `root` node of a binary tree. Each node may have a left child, a right child, both, or neither. The output is a single integer representing the deepest level reachable in the tree.

The constraints tell us that the number of nodes is between `0` and `10^4`. This means the tree may be empty, and it may also be fairly large. Because the tree can contain up to ten thousand nodes, we want an algorithm that visits each node efficiently, ideally only once.

Several important edge cases appear immediately:

An empty tree is possible. If `root` is `None` in Python or `nil` in Go, the depth should be `0`.

A tree with only one node should return `1`, because the root itself counts as a level.

A highly unbalanced tree is also possible, such as a linked-list-shaped tree where every node has only one child. In that case, the depth equals the number of nodes. Recursive solutions must correctly handle this structure.

The problem guarantees that the input is a valid binary tree, so we do not need to detect cycles or malformed structures.

## Approaches

### Brute Force Approach

A brute-force style solution would repeatedly compute subtree depths multiple times for different nodes. For example, one could separately explore every possible root-to-leaf path and compare their lengths.

This works because every path from the root to a leaf can be examined, and the longest one is by definition the maximum depth. However, if implemented poorly, many subtree calculations may be repeated again and again.

For instance, if we recursively calculate depths for overlapping subtrees multiple times, the same nodes may be revisited unnecessarily. While this still eventually produces the correct answer, it wastes computation.

### Optimal Approach

The key observation is that the maximum depth of a tree depends entirely on the maximum depths of its left and right subtrees.

For any node:

```
depth(node) = 1 + max(depth(left), depth(right))
```

The `1` counts the current node itself.

This naturally suggests a Depth-First Search approach using recursion. We recursively compute the depth of the left subtree and the depth of the right subtree, then take the larger value and add one.

This works efficiently because each node is visited exactly once. Every recursive call solves the same smaller subproblem: "What is the maximum depth of this subtree?"

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(N²) | O(H) | Recomputes subtree depths repeatedly |
| Optimal | O(N) | O(H) | Visits each node once using DFS recursion |

Here:

- `N` is the number of nodes in the tree.
- `H` is the height of the tree.

## Algorithm Walkthrough

### Recursive DFS Algorithm

1. Start at the root node of the tree.
2. If the current node is `None` or `nil`, return `0`.

This is the base case. An empty subtree contributes no depth.
3. Recursively compute the maximum depth of the left subtree.

This gives the longest path starting from the left child.
4. Recursively compute the maximum depth of the right subtree.

This gives the longest path starting from the right child.
5. Compare the two depths and take the larger one.

The overall tree depth must follow the deeper branch.
6. Add `1` to include the current node itself.

Every recursive layer represents one additional level in the tree.
7. Return the computed value upward to the caller.

Eventually, the root call returns the maximum depth of the entire tree.

This recursive structure is particularly elegant because the tree itself is recursive in nature. Each subtree is simply another binary tree, so the same logic applies repeatedly.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import Optional

class Solution:
    def maxDepth(self, root: Optional[TreeNode]) -> int:
        if root is None:
            return 0

        left_depth = self.maxDepth(root.left)
        right_depth = self.maxDepth(root.right)

        return 1 + max(left_depth, right_depth)
```

The implementation directly follows the recursive definition of tree depth.

The first condition checks whether the current node is `None`. This handles empty subtrees and serves as the recursion stopping condition.

Next, the function recursively computes the depths of the left and right subtrees. Each recursive call returns the maximum depth beneath that child node.

Finally, the function returns:

```
1 + max(left_depth, right_depth)
```

The `1` represents the current node, while `max(...)` ensures we follow the deeper subtree.

Because recursion automatically manages traversal state through the call stack, we do not need any explicit stack or queue data structure.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */

func maxDepth(root *TreeNode) int {
    if root == nil {
        return 0
    }

    leftDepth := maxDepth(root.Left)
    rightDepth := maxDepth(root.Right)

    if leftDepth > rightDepth {
        return 1 + leftDepth
    }

    return 1 + rightDepth
}
```

The Go implementation mirrors the Python solution closely.

Instead of `None`, Go uses `nil` to represent an empty node reference.

Go does not provide a built-in `max` function for integers, so we explicitly compare `leftDepth` and `rightDepth` with an `if` statement.

The recursive structure and logic remain identical to the Python version.

## Worked Examples

### Example 1

Input:

```
root = [3,9,20,null,null,15,7]
```

Tree structure:

```
        3
       / \
      9  20
         / \
        15  7
```

We recursively compute depths bottom-up.

### Recursive Trace

| Node | Left Depth | Right Depth | Result |
| --- | --- | --- | --- |
| 9 | 0 | 0 | 1 |
| 15 | 0 | 0 | 1 |
| 7 | 0 | 0 | 1 |
| 20 | 1 | 1 | 2 |
| 3 | 1 | 2 | 3 |

Detailed flow:

1. Start at node `3`.
2. Explore left subtree rooted at `9`.

Node `9` has no children, so:

```
depth(9) = 1
```
3. Explore right subtree rooted at `20`.
4. Node `20` explores children `15` and `7`.
5. Both `15` and `7` are leaves:

```
depth(15) = 1
depth(7) = 1
```
6. Node `20` computes:

```
1 + max(1, 1) = 2
```
7. Root node `3` computes:

```
1 + max(1, 2) = 3
```

Final answer:

```
3
```

### Example 2

Input:

```
root = [1,null,2]
```

Tree structure:

```
    1
     \
      2
```

### Recursive Trace

| Node | Left Depth | Right Depth | Result |
| --- | --- | --- | --- |
| 2 | 0 | 0 | 1 |
| 1 | 0 | 1 | 2 |

Detailed flow:

1. Start at node `1`.
2. Left child is `null`, so left depth is `0`.
3. Explore right child `2`.
4. Node `2` is a leaf:

```
depth(2) = 1
```
5. Node `1` computes:

```
1 + max(0, 1) = 2
```

Final answer:

```
2
```

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(N) | Every node is visited exactly once |
| Space | O(H) | Recursive call stack stores one frame per tree level |

The algorithm performs a single DFS traversal of the tree. Each node contributes constant work, consisting mainly of two recursive calls and one `max` comparison. Therefore, the total runtime is linear in the number of nodes.

The space complexity comes from recursion depth. In the best case of a balanced tree, the height is approximately `log N`. In the worst case of a completely skewed tree, the height becomes `N`, so the recursion stack may grow to linear size.

## Edge Cases

### Empty Tree

An empty tree occurs when the input root is `None` or `nil`.

```
root = []
```

This case can easily cause bugs if recursion assumes every node exists. Our implementation handles it immediately with the base case:

```
if root is None:
    return 0
```

This correctly returns depth `0`.

### Single Node Tree

A tree containing only the root node should return depth `1`.

```
    1
```

A common mistake is forgetting to count the current node when returning subtree depths. Our implementation avoids this by always adding `1`:

```
1 + max(left_depth, right_depth)
```

Thus the root contributes one level even when both children are empty.

### Highly Skewed Tree

Consider a tree shaped like a linked list:

```
1
 \
  2
   \
    3
     \
      4
```

The maximum depth here is `4`.

This case is important because it produces the maximum recursion depth and tests whether the algorithm correctly follows only one branch. Our recursive DFS handles this naturally because each call continues deeper until reaching the leaf node.

### Uneven Subtrees

A tree may have one shallow subtree and one deep subtree:

```
      1
     / \
    2   3
         \
          4
           \
            5
```

The algorithm must choose the larger subtree depth rather than averaging or summing them.

Using:

```
max(left_depth, right_depth)
```

ensures the deeper branch determines the final answer.
