## Solution

## Problem Understanding

This problem provides a database table named `Person` with two columns:

| Column | Description |
| --- | --- |
| `id` | A unique integer identifier for each row |
| `email` | An email address stored as a string |

The task is to find all email addresses that appear more than once in the table. In other words, we need to identify duplicate email values.

The result should contain only the duplicated email addresses, with each duplicate email appearing exactly once in the output table. The order of the result does not matter.

For example, if the table contains:

| id | email |
| --- | --- |
| 1 | a@b.com |
| 2 | c@d.com |
| 3 | a@b.com |

then `a@b.com` appears twice, so it should be returned.

The input represents relational database rows, so the solution is expected to be written in SQL rather than a traditional programming language algorithm. Since the `email` column is guaranteed to be non-null, we do not need to handle missing values. The emails are also lowercase only, meaning we do not need to worry about case-insensitive matching such as treating `A@B.COM` and `a@b.com` as equivalent.

An important detail is that the output should contain unique duplicate emails, not every repeated row. If an email appears five times, it should still appear only once in the result.

Several edge cases are important:

- A table with no duplicate emails should return an empty result.
- A table where every row contains the same email should still return only one row for that email.
- Multiple different duplicated emails must all be included.
- Since `email` is guaranteed non-null, we do not need special handling for `NULL` values.

## Approaches

### Brute Force Approach

A brute-force solution would compare every email against every other email in the table. For each row, we could scan the entire table again to count how many times that email appears.

Conceptually, this works because every possible pair of rows is checked. If two rows share the same email, then that email is identified as duplicated.

However, this approach is inefficient because it performs repeated comparisons. If the table contains `n` rows, then each row may compare itself against all other rows, leading to approximately `n × n` operations.

In SQL, a brute-force implementation could use a self join:

```
SELECT DISTINCT p1.email
FROM Person p1
JOIN Person p2
ON p1.email = p2.email
AND p1.id != p2.id;
```

Although correct, this approach is unnecessarily expensive for large datasets.

### Optimal Approach

The key observation is that we do not actually need to compare every pair of rows. Instead, we only need to know how many times each email appears.

SQL provides aggregation tools specifically for this purpose. By grouping rows by `email` and counting how many rows belong to each group, we can directly identify duplicates.

The `GROUP BY` clause creates one group per email address, and the `HAVING` clause filters groups whose count is greater than 1.

This approach is both cleaner and more efficient.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) or O(n) | Compares rows against each other using self join |
| Optimal | O(n) | O(n) | Uses grouping and counting to identify duplicates efficiently |

## Algorithm Walkthrough

1. Start by grouping all rows in the `Person` table according to the `email` column. All identical emails are collected into the same group.
2. For each email group, count how many rows belong to that group. This tells us how many times each email appears in the table.
3. Use a `HAVING` clause to keep only the groups where the count is greater than 1. These are the duplicated emails.
4. Return the email values from those filtered groups.

The reason `GROUP BY` is ideal here is that databases are optimized for aggregation operations. Instead of manually scanning rows multiple times, the database engine efficiently organizes rows into groups and computes counts internally.

## Python Solution

Even though this is a database problem, LeetCode expects a SQL query. The following is the correct SQL solution.

```
SELECT email AS Email
FROM Person
GROUP BY email
HAVING COUNT(email) > 1;
```

This query has three main parts.

First, `GROUP BY email` collects all rows with the same email into a single group.

Second, `COUNT(email)` computes how many rows exist in each group.

Third, `HAVING COUNT(email) > 1` filters out groups that occur only once, leaving only duplicate emails.

Finally, `SELECT email AS Email` returns the result using the exact column name expected by the problem statement.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each row is processed once during grouping and counting |
| Space | O(n) | The database may store grouped aggregation results internally |

The database engine scans the table and organizes rows into groups by email. Each row contributes to exactly one group, making the operation effectively linear with respect to the number of rows.

The space complexity depends on how many distinct email groups exist. In the worst case, every email could be unique, requiring storage proportional to the number of rows.

## Edge Cases

### No Duplicate Emails

Consider a table where every email appears exactly once:

| id | email |
| --- | --- |
| 1 | a@b.com |
| 2 | c@d.com |
| 3 | e@f.com |

Each group has a count of 1, so the `HAVING COUNT(email) > 1` condition removes all rows. The query correctly returns an empty result set.

### All Emails Are the Same

Consider:

| id | email |
| --- | --- |
| 1 | a@b.com |
| 2 | a@b.com |
| 3 | a@b.com |

The grouping step creates a single group with count 3. The query returns exactly one row containing `a@b.com`, which is correct because the problem wants distinct duplicate emails, not repeated rows.

### Multiple Different Duplicate Emails

Consider:

| id | email |
| --- | --- |
| 1 | a@b.com |
| 2 | c@d.com |
| 3 | a@b.com |
| 4 | c@d.com |

The grouping operation creates two groups with counts greater than 1. Both emails are returned correctly.

### Large Input Size

A naive self-join solution becomes inefficient as the table grows because it compares rows repeatedly. The aggregation-based solution scales much better because databases optimize grouping and counting operations internally.
