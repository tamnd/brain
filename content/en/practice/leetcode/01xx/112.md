## Solution

## Problem Understanding

This problem asks us to determine whether a binary tree contains at least one valid root-to-leaf path whose node values add up to a given target sum.

A binary tree is made of nodes, where each node contains an integer value and references to a left and right child. A root-to-leaf path starts at the root node and moves downward through child nodes until reaching a leaf node. A leaf node is a node with no children.

The input consists of two parts:

- `root`, which represents the root node of the binary tree
- `targetSum`, which is the integer sum we want to match

The expected output is a boolean value:

- Return `true` if there exists at least one root-to-leaf path whose values sum exactly to `targetSum`
- Return `false` otherwise

The important detail is that the path must end at a leaf node. A partial path that reaches the correct sum before reaching a leaf does not count.

For example, consider this path:

```
5 -> 4 -> 11 -> 2
```

The sum is:

```
5 + 4 + 11 + 2 = 22
```

If `2` is a leaf node and the target sum is `22`, the answer is `true`.

The constraints tell us several important things:

- The tree can contain up to 5000 nodes
- Node values can be negative
- The target sum can also be negative

Since the tree size is moderate, an `O(n)` traversal is efficient enough. Because node values may be negative, we cannot use pruning strategies that stop searching once the running sum exceeds the target.

Several edge cases are important:

- The tree may be empty
- The target sum may be zero
- Node values may be negative
- A path may temporarily exceed the target and later decrease because of negative values
- A node that matches the target sum is only valid if it is also a leaf node

A naive implementation may incorrectly return `true` when the running sum matches the target before reaching a leaf.

## Approaches

### Brute Force Approach

A brute-force approach would explicitly generate every root-to-leaf path in the tree, compute the sum of each path, and then check whether any path equals `targetSum`.

This method works because every possible root-to-leaf path is examined. If a valid path exists, it will eventually be found.

One way to implement this is:

1. Use DFS to collect all root-to-leaf paths
2. Store each path in a list
3. Compute the sum for every stored path
4. Compare each sum with `targetSum`

The main problem with this approach is unnecessary memory usage. Storing entire paths is expensive, especially when the tree contains many root-to-leaf paths.

We do not actually need to remember the whole path. We only need the current remaining sum while traversing the tree.

### Optimal Approach

The key observation is that while traversing from the root toward a leaf, we can continuously subtract node values from the target sum.

Suppose the original target is:

```
targetSum = 22
```

If we visit node `5`, the remaining required sum becomes:

```
22 - 5 = 17
```

Then if we visit node `4`:

```
17 - 4 = 13
```

Eventually, if we reach a leaf node and the remaining sum equals the leaf's value, then the entire path matches the target.

This naturally leads to a Depth-First Search traversal.

At every node:

- Subtract the node value from the remaining target
- Recursively search the left and right subtrees
- If we reach a leaf and the remaining sum equals the leaf value, return `true`

This avoids storing paths entirely and processes each node only once.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) in worst case | O(n²) | Stores all root-to-leaf paths explicitly |
| Optimal | O(n) | O(h) | DFS traversal with running remaining sum |

Here:

- `n` is the number of nodes
- `h` is the height of the tree

## Algorithm Walkthrough

1. Start at the root node with the original `targetSum`.
2. If the current node is `None`, return `false` because there is no valid path through an empty subtree.
3. Subtract the current node's value from the remaining target sum. This represents how much more sum we still need from the rest of the path.
4. Check whether the current node is a leaf node. A node is a leaf if both `left` and `right` are `None`.
5. If the node is a leaf, compare its value with the remaining target sum before subtraction, or equivalently check whether the updated remaining sum is zero. If the remaining sum becomes zero exactly at a leaf, return `true`.
6. Otherwise, recursively search the left subtree and the right subtree using the updated remaining sum.
7. If either subtree returns `true`, then a valid root-to-leaf path exists.
8. If neither subtree contains a valid path, return `false`.

This approach works efficiently because each node is visited exactly once, and the recursion naturally tracks the current path.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import Optional

class Solution:
    def hasPathSum(self, root: Optional[TreeNode], targetSum: int) -> bool:
        if root is None:
            return False

        remaining_sum = targetSum - root.val

        # Check if current node is a leaf
        if root.left is None and root.right is None:
            return remaining_sum == 0

        return (
            self.hasPathSum(root.left, remaining_sum) or
            self.hasPathSum(root.right, remaining_sum)
        )
```

The implementation begins by handling the empty tree case. If the root is `None`, there cannot be any root-to-leaf path, so the function immediately returns `False`.

Next, the algorithm subtracts the current node's value from `targetSum`. This creates a new value called `remaining_sum`, representing how much more total we still need from the remaining path.

The next step checks whether the current node is a leaf node. This is done by verifying that both children are `None`.

If the node is a leaf, the path is complete. At this point, the path is valid only if the remaining sum becomes zero.

If the node is not a leaf, the algorithm recursively explores both subtrees. The recursive calls use the updated remaining sum so that every deeper node contributes toward the total path sum.

The final result is the logical OR of the left and right subtree searches. If either subtree contains a valid path, the answer is `True`.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func hasPathSum(root *TreeNode, targetSum int) bool {
    if root == nil {
        return false
    }

    remainingSum := targetSum - root.Val

    // Check if current node is a leaf
    if root.Left == nil && root.Right == nil {
        return remainingSum == 0
    }

    return hasPathSum(root.Left, remainingSum) ||
        hasPathSum(root.Right, remainingSum)
}
```

The Go implementation follows exactly the same recursive logic as the Python solution.

The primary language-specific difference is the use of `nil` instead of `None`.

Go passes pointers to tree nodes using `*TreeNode`, so checking for an empty node requires:

```
if root == nil
```

Go also uses explicit variable declarations such as:

```
remainingSum := targetSum - root.Val
```

There are no integer overflow concerns here because the problem constraints are small enough to fit comfortably within Go's standard integer type.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every node is visited once |
| Space | O(h) | Recursive call stack uses tree height |

The time complexity is `O(n)` because each node in the tree is processed exactly one time.

The space complexity is `O(h)`, where `h` is the height of the tree. This space comes from the recursion stack during DFS traversal.

In the worst case, when the tree is completely skewed, the height becomes `n`, producing `O(n)` space usage.

In a balanced tree, the height is approximately `log n`, giving `O(log n)` recursive stack space.

## Edge Cases

### Empty Tree

If the root is `None`, there are no root-to-leaf paths available. A common bug is incorrectly returning `True` when `targetSum` is zero. The problem definition requires an actual path, so the correct result is always `False` for an empty tree.

The implementation handles this immediately with:

```
if root is None:
    return False
```

### Single Node Tree

A tree containing only one node is both the root and a leaf. This case is important because the algorithm must correctly recognize that a single-node path is valid.

For example:

```
root = [5]
targetSum = 5
```

The algorithm subtracts `5` from the target, producing `0`, and since the node is a leaf, it correctly returns `True`.

### Negative Values in the Tree

Since node values may be negative, the running sum can increase or decrease throughout traversal.

For example:

```
1 -> -2 -> 3
```

A naive optimization that stops searching when the running sum becomes negative or exceeds the target would fail on such cases.

The implementation never performs pruning based on value magnitude. It explores every root-to-leaf path correctly.

### Matching Sum Before Reaching a Leaf

A subtle bug occurs when the running sum becomes zero before reaching a leaf node.

For example:

```
5 -> 4 -> 13
```

Suppose the target is `9`.

After visiting `5 -> 4`, the remaining sum becomes zero, but node `4` is not a leaf. The path is invalid because the traversal has not ended at a leaf node.

The implementation prevents this bug by checking:

```
if root.left is None and root.right is None:
```

Only leaf nodes are allowed to complete a valid path.
