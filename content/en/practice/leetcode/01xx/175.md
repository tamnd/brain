# LeetCode 175, Combine Two Tables

## Problem Understanding

This problem provides two database tables, `Person` and `Address`, and asks us to combine information from both tables into a single result.

The `Person` table contains a person's ID along with their first and last names. The `Address` table contains address information, specifically the city and state associated with a `personId`.

The goal is to return a result containing:

- `firstName`
- `lastName`
- `city`
- `state`

for every person in the `Person` table.

The important requirement is that every person must appear in the output, even if they do not have a corresponding address entry. In cases where no matching address exists, the `city` and `state` columns should contain `NULL`.

This is fundamentally a table joining problem. We need to combine rows from two tables using the shared column `personId`.

The input represents relational database tables. The expected output is another table produced through a SQL query.

The problem guarantees that:

- `personId` is the primary key in the `Person` table.
- `addressId` is the primary key in the `Address` table.
- The `Address` table may contain entries whose `personId` does not exist in `Person`.
- Some people in `Person` may not have any address entry.

The scale of the problem is small because this is categorized as an Easy database problem, but the solution should still use the correct relational operation.

One important edge case is when a person has no address. A naive inner join would incorrectly exclude such people from the result. Another edge case is when the `Address` table contains extra records for nonexistent people. Those rows should not appear because the output is driven by the `Person` table. Finally, an empty `Address` table should still return all people with `NULL` values for city and state.

## Approaches

### Brute Force Approach

A brute force approach would conceptually iterate through every person in the `Person` table and then scan the entire `Address` table to find a matching `personId`.

For each person:

1. Search every row in `Address`
2. If a matching `personId` is found, output the combined data
3. If no match is found, output the person with `NULL` values

This approach is correct because every possible address row is checked for every person. However, it is inefficient because it repeatedly scans the entire `Address` table.

If there are `P` rows in `Person` and `A` rows in `Address`, the total complexity becomes `O(P × A)`.

### Optimal Approach

The key insight is that SQL already provides efficient join operations specifically designed for this kind of task.

Since we must include every row from the `Person` table regardless of whether an address exists, we should use a `LEFT JOIN`.

A `LEFT JOIN` works as follows:

- Keep all rows from the left table, which is `Person`
- Match rows from the right table, which is `Address`
- If no match exists, fill the right-side columns with `NULL`

This exactly matches the problem requirements.

The join condition is:

```
Person.personId = Address.personId
```

Using a database join is significantly more efficient because relational databases optimize join operations internally using indexes and query planning.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(P × A) | O(1) | Scans the entire Address table for every person |
| Optimal | O(P + A) | O(1) extra | Uses SQL LEFT JOIN to efficiently combine tables |

## Algorithm Walkthrough

1. Start with the `Person` table because every person must appear in the final result.
2. Perform a `LEFT JOIN` with the `Address` table using the shared `personId` column. This ensures that all rows from `Person` are preserved.
3. For each row in `Person`, the database searches for a matching `personId` in `Address`.
4. If a match exists, retrieve the corresponding `city` and `state`.
5. If no match exists, SQL automatically fills `city` and `state` with `NULL`.
6. Select only the required columns:

- `firstName`
- `lastName`
- `city`
- `state`
7. Return the resulting table in any order.

The reason a `LEFT JOIN` is chosen instead of an `INNER JOIN` is that an inner join would exclude people without addresses, which violates the problem requirements.

## Python Solution

Although this is a database problem and LeetCode expects a SQL query, the platform still asks for a language section in many reference guides. Below is the SQL solution represented inside a Python string for completeness.

```
class Solution:
    def combineTables(self) -> str:
        return """
        SELECT
            Person.firstName,
            Person.lastName,
            Address.city,
            Address.state
        FROM Person
        LEFT JOIN Address
        ON Person.personId = Address.personId;
        """
```

The implementation uses a `LEFT JOIN` between the `Person` and `Address` tables.

The query begins with the `Person` table because every person must appear in the output. The `LEFT JOIN` operation attempts to find matching rows in `Address` using the condition:

```
Person.personId = Address.personId
```

If a matching address exists, the query returns the associated city and state. If no matching address exists, SQL automatically returns `NULL` values for those columns.

Finally, the query selects only the required output columns instead of returning all table data.

## Edge Cases

### Person Without an Address

A person may exist in the `Person` table without any corresponding row in `Address`.

This is the most important edge case because using an `INNER JOIN` would accidentally remove such people from the output. The `LEFT JOIN` handles this correctly by preserving the person row and filling `city` and `state` with `NULL`.

### Address Without a Matching Person

The `Address` table may contain rows whose `personId` does not exist in `Person`.

This could happen because the tables are independent datasets. The solution correctly ignores these rows because the query starts from the `Person` table and only includes matching address information.

### Empty Address Table

If the `Address` table is completely empty, the result should still include every person.

The `LEFT JOIN` naturally handles this scenario. Every person appears exactly once, and all address-related columns become `NULL`.

### Empty Person Table

If the `Person` table is empty, the result should also be empty.

Since the query begins with `Person`, there are no rows to preserve, so the output contains zero rows. This behavior is correct and requires no special handling.
