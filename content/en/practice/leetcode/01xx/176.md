## Solution

## Problem Understanding

This problem asks us to retrieve the second highest distinct salary from the `Employee` table. The table contains two columns, `id` and `salary`, where `id` uniquely identifies each employee and `salary` stores that employee's salary value.

The key detail in the problem statement is the word distinct. We are not looking for the second row when salaries are sorted. Instead, we are looking for the second unique salary value. For example, if multiple employees have a salary of `300`, that salary should only count once when determining rank.

The input is a relational database table. Each row represents one employee and their salary. The expected output is a single-column result named `SecondHighestSalary`. If a second distinct salary exists, we return it. Otherwise, we return `null`.

For example:

- Salaries `[100, 200, 300]` produce `200`
- Salaries `[100, 100]` produce `null`
- Salaries `[300, 300, 200]` produce `200`

The problem does not specify a maximum table size, but database interview problems generally expect efficient SQL queries that avoid unnecessary computation. We should therefore think carefully about handling duplicates and missing values correctly.

Several edge cases are important:

- The table may contain only one distinct salary
- Multiple employees may share the same salary
- The highest salary may appear many times
- The table may contain only a single row
- Salaries are integers, but the output may need to be `null`

A naive implementation that simply sorts rows and takes the second row would fail whenever duplicate salaries exist.

## Approaches

### Brute Force Approach

A straightforward solution is to retrieve all distinct salaries, sort them in descending order, and then manually pick the second value.

The process works like this:

1. Select all distinct salary values
2. Sort them from largest to smallest
3. Skip the first value
4. Return the next value if it exists

This approach is correct because sorting distinct salaries guarantees that the second element corresponds to the second highest unique salary.

However, this solution can become inefficient if the table is very large, because sorting requires additional computation. In SQL databases, sorting operations are generally more expensive than targeted aggregation queries.

### Optimal Approach

The key insight is that we do not actually need to sort the entire dataset ourselves. SQL provides aggregation functions and subqueries that allow us to directly identify the second highest distinct value.

A common optimal solution is:

- Find the maximum salary
- Then find the maximum salary smaller than that maximum

This works because:

- The first query identifies the highest distinct salary
- The second query searches only among smaller salaries
- The largest remaining value must therefore be the second highest distinct salary

If no such smaller salary exists, SQL automatically returns `NULL`, which matches the required output.

This approach is concise, efficient, and easy to reason about.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) | O(n) | Sorts all distinct salaries |
| Optimal | O(n) | O(1) | Uses nested aggregation queries |

## Algorithm Walkthrough

1. First, compute the maximum salary in the table. This gives us the highest distinct salary value.
2. Next, search for salaries that are strictly smaller than the maximum salary. This removes all occurrences of the highest salary from consideration.
3. Among the remaining salaries, compute the maximum again. Since all higher salaries were excluded, this value becomes the second highest distinct salary.
4. If no salary remains after filtering, SQL returns `NULL`. This correctly handles cases where only one distinct salary exists.
5. Rename the result column to `SecondHighestSalary` so the output matches the required format.

The reason this approach works is that the maximum function naturally identifies the largest value in a set. By removing the highest salary first, the next maximum becomes the answer we need.

## Python Solution

LeetCode database problems that use Python generally expect a Pandas implementation.

```
import pandas as pd

def second_highest_salary(employee: pd.DataFrame) -> pd.DataFrame:
    distinct_salaries = employee["salary"].drop_duplicates()

    if len(distinct_salaries) < 2:
        return pd.DataFrame({"SecondHighestSalary": [None]})

    second_highest = distinct_salaries.nlargest(2).iloc[-1]

    return pd.DataFrame({"SecondHighestSalary": [second_highest]})
```

The implementation begins by extracting the `salary` column and removing duplicate values. This step is essential because the problem requires distinct salaries rather than employee rows.

Next, the code checks whether at least two unique salary values exist. If not, the function returns `None`, matching the problem specification.

The `nlargest(2)` operation retrieves the two highest distinct salaries in descending order. The second value in that result is the second highest distinct salary.

Finally, the result is wrapped inside a DataFrame with the exact required column name, `SecondHighestSalary`.

## Go Solution

For LeetCode database problems, the Go solution is typically represented as an SQL query.

```
SELECT
    MAX(salary) AS SecondHighestSalary
FROM Employee
WHERE salary < (
    SELECT MAX(salary)
    FROM Employee
);
```

The query works in two stages. The inner query computes the highest salary in the table. The outer query filters out all rows with that maximum salary and then computes the maximum among the remaining rows.

If no salary remains after filtering, `MAX()` returns `NULL`, which satisfies the problem requirements automatically.

Unlike the Python solution, SQL handles null propagation naturally through aggregate functions, so no explicit conditional logic is needed.

## Edge Cases

One important edge case occurs when the table contains only one distinct salary. For example, `[100]` or `[100, 100]`. A naive solution might incorrectly return `100` as both the highest and second highest salary. The correct behavior is to return `NULL`, because a second distinct value does not exist. The implementation handles this correctly by filtering salaries smaller than the maximum salary. If none remain, `MAX()` returns `NULL`.

Another important case involves duplicate highest salaries. Consider salaries `[300, 300, 200]`. A row-based approach might mistakenly return `300` as the second highest value because the second row after sorting is still `300`. The solution avoids this issue by comparing salary values rather than row positions. All occurrences of the maximum salary are excluded before computing the next maximum.

A third edge case occurs when multiple duplicate lower salaries exist, such as `[500, 400, 400, 300]`. The correct answer is still `400`. Since the query operates on salary comparisons rather than counting duplicates, repeated values do not affect correctness.

Another subtle case is an empty result after filtering. SQL aggregate functions are useful here because `MAX()` automatically returns `NULL` when applied to an empty set. This removes the need for additional conditional logic in the SQL solution.
