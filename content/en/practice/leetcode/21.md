## Solution

## Problem Understanding

This problem asks us to merge two already sorted singly linked lists into one sorted linked list. The important detail is that we are **not creating a completely new list of values**, instead, we are expected to reuse the existing nodes by rearranging their `next` pointers. The phrase "splicing together the nodes" indicates that we should connect the original nodes from `list1` and `list2` into a single ordered sequence.

The input consists of two linked list heads, `list1` and `list2`. Each linked list is sorted in **non-decreasing order**, meaning values can stay the same or increase as we traverse the list. A linked list node contains a value and a pointer to the next node.

The goal is to return the head of a new linked list that contains all nodes from both input lists while maintaining sorted order.

For example:

- `list1 = [1,2,4]`
- `list2 = [1,3,4]`

Since both lists are already sorted, we repeatedly take the smaller current node from either list, producing:

`[1,1,2,3,4,4]`

The constraints are relatively small:

- Each list can contain between `0` and `50` nodes.
- Node values range from `-100` to `100`.
- Both input lists are guaranteed to be sorted.

Even though the input size is small, this problem is fundamentally about understanding linked list manipulation and pointer handling. A correct solution should work efficiently regardless of size.

Several edge cases are important to recognize upfront. One or both lists may be empty, which means the answer is simply the non-empty list or an empty list. Duplicate values can appear, so equal values must be handled correctly. One list may finish before the other, meaning the remaining nodes from the unfinished list should be attached directly to the result.

## Approaches

### Brute Force Approach

A straightforward but less elegant approach is to first collect all node values from both linked lists into an array, sort the array, and then construct a brand-new linked list from the sorted values.

This works because sorting guarantees the final order is correct. We traverse both lists, insert every value into a temporary array, sort the array, then rebuild a linked list node by node.

While this produces the correct answer, it ignores an important property of the problem: **both input lists are already sorted**. Sorting again wastes work that has already been done. It also requires additional memory to store all values and construct new nodes, rather than reusing the existing ones.

If there are `n` nodes in `list1` and `m` nodes in `list2`, sorting requires `O((n+m) log(n+m))` time.

### Optimal Approach

The key insight is that the two lists are **already sorted**.

This means we can merge them using the same idea as the merge step in merge sort. At every moment, we compare the current node of `list1` and `list2`. The smaller node must come next in the merged list because both remaining portions are sorted.

We maintain a pointer to the end of the merged list and repeatedly attach the smaller node. Once one list becomes empty, we simply connect the rest of the remaining list since it is already sorted.

This avoids unnecessary sorting and only requires a single pass through all nodes.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O((n+m) log(n+m)) | O(n+m) | Store all values, sort them, rebuild list |
| Optimal | O(n+m) | O(1) | Merge directly using pointers |

## Algorithm Walkthrough

1. Create a **dummy node** to simplify linked list construction. This avoids special handling for the first inserted node because every merged node will attach to `dummy.next`.
2. Create a pointer called `current` that initially points to the dummy node. This pointer always represents the tail of the merged list.
3. Traverse both lists while neither is empty.
4. Compare the current values of `list1` and `list2`.
5. If `list1.val` is smaller or equal, connect `current.next` to `list1`, then move `list1` forward to its next node.
6. Otherwise, connect `current.next` to `list2`, then move `list2` forward.
7. Move `current` forward because the merged list has grown by one node.
8. Continue until one list becomes empty.
9. Once the loop ends, attach whichever list still has remaining nodes. Since the lists are sorted, the remaining portion is already in valid order.
10. Return `dummy.next`, because the dummy node itself is only a helper node.

Using a dummy node is particularly helpful because it removes edge cases around initializing the head of the merged list. Without it, we would need extra logic to determine the first node.

## Python Solution

```
# Definition for singly-linked list.
# class ListNode:
#     def __init__(self, val=0, next=None):
#         self.val = val
#         self.next = next

from typing import Optional

class Solution:
    def mergeTwoLists(
        self,
        list1: Optional[ListNode],
        list2: Optional[ListNode]
    ) -> Optional[ListNode]:
        dummy = ListNode()
        current = dummy

        while list1 and list2:
            if list1.val <= list2.val:
                current.next = list1
                list1 = list1.next
            else:
                current.next = list2
                list2 = list2.next

            current = current.next

        current.next = list1 if list1 else list2

        return dummy.next
```

The implementation starts by creating a dummy node. This node acts as a placeholder before the actual head of the merged linked list.

The `current` pointer tracks the tail of the merged list as we build it. Inside the `while` loop, we compare the values at the current nodes of `list1` and `list2`.

If `list1.val` is smaller or equal, we attach that node to the merged list and move `list1` forward. Otherwise, we attach the node from `list2`.

After attaching a node, we move `current` forward because the merged list has expanded.

Once either list becomes empty, we connect the remaining nodes directly. Since the remaining nodes are already sorted, no further comparisons are necessary.

Finally, we return `dummy.next`, which skips the helper node and gives us the real head of the merged linked list.

## Go Solution

```
/**
 * Definition for singly-linked list.
 * type ListNode struct {
 *     Val int
 *     Next *ListNode
 * }
 */
func mergeTwoLists(list1 *ListNode, list2 *ListNode) *ListNode {
	dummy := &ListNode{}
	current := dummy

	for list1 != nil && list2 != nil {
		if list1.Val <= list2.Val {
			current.Next = list1
			list1 = list1.Next
		} else {
			current.Next = list2
			list2 = list2.Next
		}

		current = current.Next
	}

	if list1 != nil {
		current.Next = list1
	} else {
		current.Next = list2
	}

	return dummy.Next
}
```

The Go implementation follows the same logic as the Python solution. The main difference is how linked list pointers are handled.

Go uses `nil` instead of Python's `None` for empty pointers. The dummy node is created with `&ListNode{}` and pointers are manipulated directly.

No integer overflow concerns exist here because node values remain within a very small range. Since linked lists are pointer-based, slices or arrays are unnecessary.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n+m) | Every node from both lists is visited exactly once |
| Space | O(1) | Only a few pointer variables are used |

The time complexity is `O(n+m)` because each node in both linked lists is processed exactly once. No nested loops or repeated traversals occur.

The space complexity is `O(1)` because we reuse the original linked list nodes rather than creating extra data structures proportional to input size. Only a few pointers, such as `dummy` and `current`, are maintained.

## Edge Cases

### One or Both Lists Are Empty

If `list1` or `list2` is empty, a naive implementation may fail due to null pointer access. For example, attempting to compare values when one list is `None` or `nil` would cause an error.

This implementation handles the case safely because the loop only executes while both lists are non-empty. Afterward, the remaining non-empty list is attached directly.

### Duplicate Values

The lists may contain duplicate values, such as:

`list1 = [1,1,2]`

`list2 = [1,3]`

An incorrect implementation could mishandle ordering or accidentally skip duplicates.

Here, the condition `<=` ensures duplicates are included properly and appear in valid sorted order.

### One List Finishes Earlier

Sometimes one list is much shorter:

`list1 = [1]`

`list2 = [2,3,4,5]`

A buggy implementation might forget to append the remaining nodes after comparisons stop.

This solution correctly attaches the unfinished list using:

```
current.next = list1 if list1 else list2
```

This guarantees no nodes are lost and preserves sorted order.
