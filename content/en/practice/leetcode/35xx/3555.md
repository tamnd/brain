## Solution

## Problem Understanding

This problem asks us to analyze every contiguous subarray (window) of length `k` in a given array `nums` and determine the **minimum length of a continuous segment** within that window that must be sorted so that the entire window becomes non-decreasing. In simpler terms, for each window, we need to find the smallest subarray which, if sorted, would make the whole window sorted.

The input consists of an array `nums` of integers and an integer `k` representing the window size. The output is an array of length `n - k + 1`, where each element represents the answer for the corresponding window.

Constraints tell us that `nums` can have up to 1000 elements and each element is between 1 and 1,000,000. These constraints are small enough to allow algorithms with time complexity up to roughly O(n²), but we should aim for an optimal O(n) or O(n log n) solution if possible.

Important edge cases include windows that are already sorted (length should be 0), windows that are fully reversed (length should be k), and windows with repeated numbers where sorting might only require adjusting a small segment.

## Approaches

### Brute Force

The brute-force approach is straightforward: for each window of length `k`, we check the minimal unsorted subarray by comparing the window with its sorted version. Specifically, we slide the window, sort it, and then find the leftmost and rightmost positions where the sorted window differs from the original. The length of this segment is the answer. This approach is correct but inefficient because sorting each window individually has O(k log k) complexity, and there are O(n) windows, resulting in O(n k log k) overall.

### Optimal Approach

The key insight for a better solution is to recognize that the minimum subarray that must be sorted in a window can be found using a monotonic stack or a greedy approach without explicitly sorting each window. We can track the **boundaries of unsorted elements** by scanning from left to right and right to left, maintaining the smallest and largest values seen so far. This allows us to determine, in O(k) time per window, the segment that violates the non-decreasing order. By combining this with a sliding window technique, we can compute the result efficiently for all windows.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * k log k) | O(k) | Sort each window and compare with original |
| Optimal | O(n * k) | O(k) | Track unsorted segment using monotonic stacks within each window |

## Algorithm Walkthrough

1. Initialize an empty result array of length `n - k + 1` to store answers for each window.
2. For each window starting at index `i` from 0 to `n - k`:

1. Extract the window `nums[i:i+k]`.
2. Initialize `left` and `right` boundaries of the unsorted segment to -1.
3. Scan from left to right within the window:

- Track the maximum value seen so far.
- If the current element is smaller than this maximum, update the right boundary to the current index.
4. Scan from right to left within the window:

- Track the minimum value seen so far.
- If the current element is larger than this minimum, update the left boundary to the current index.
5. If `left` and `right` are not -1, the minimal segment length is `right - left + 1`; otherwise, it is 0.
6. Append the length to the result array.
3. Return the result array.

**Why it works**: By keeping track of the maximum from the left and the minimum from the right, we can identify exactly where elements violate the sorted order. The leftmost and rightmost violations form the minimal subarray that must be sorted. This ensures we capture the smallest necessary segment without sorting the entire window.

## Python Solution

```python
from typing import List

class Solution:
    def minSubarraySort(self, nums: List[int], k: int) -> List[int]:
        n = len(nums)
        result = []
        
        for i in range(n - k + 1):
            window = nums[i:i+k]
            
            max_seen = window[0]
            right = -1
            for j in range(1, k):
                if window[j] < max_seen:
                    right = j
                else:
                    max_seen = window[j]
            
            min_seen = window[-1]
            left = -1
            for j in range(k-2, -1, -1):
                if window[j] > min_seen:
                    left = j
                else:
                    min_seen = window[j]
            
            if left == -1 and right == -1:
                result.append(0)
            else:
                result.append(right - left + 1)
        
        return result
```

**Explanation**: The outer loop iterates over all windows of size `k`. The first inner loop tracks the right boundary of disorder by maintaining the maximum seen value and comparing each element. The second inner loop tracks the left boundary using the minimum seen value. The segment length is computed and appended to the result array. Edge cases where the window is already sorted are handled by checking if `left` and `right` are still -1.

## Go Solution

```go
func minSubarraySort(nums []int, k int) []int {
    n := len(nums)
    result := make([]int, 0, n-k+1)

    for i := 0; i <= n-k; i++ {
        window := nums[i : i+k]

        maxSeen := window[0]
        right := -1
        for j := 1; j < k; j++ {
            if window[j] < maxSeen {
                right = j
            } else {
                maxSeen = window[j]
            }
        }

        minSeen := window[k-1]
        left := -1
        for j := k-2; j >= 0; j-- {
            if window[j] > minSeen {
                left = j
            } else {
                minSeen = window[j]
            }
        }

        if left == -1 && right == -1 {
            result = append(result, 0)
        } else {
            result = append(result, right-left+1)
        }
    }

    return result
}
```

**Go-specific notes**: Go slices handle window extraction naturally. The logic mirrors Python, with `append` used for the result. Edge cases like empty or fully sorted windows are handled identically.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n * k) | Each window of size k is scanned twice, once left-to-right and once right-to-left. There are n - k + 1 windows. |
| Space | O(n - k + 1 + k) | Result array of length n - k + 1 and temporary window slice of size k for each iteration. |

The complexity is acceptable given the constraints (`n ≤ 1000`, `k ≤ n`).
