## Solution

## Problem Understanding

This problem asks us to maintain an array while processing a sequence of online operations. Each operation modifies or queries the current state of the array, and all operations must behave as if previous changes have already been applied.

There are three possible query types:

1. Update a single position:

`nums[index] = value`
2. Compute the XOR of a contiguous subarray:

`nums[left...right]`
3. Reverse a contiguous subarray in place:

`nums[left...right]`

For every XOR query, we must record the result and return all collected answers in order.

The difficulty comes from supporting all three operations efficiently at the same time.

Point updates are easy with segment trees.

Range XOR queries are also easy with segment trees because XOR is associative.

The challenge is subarray reversal.

A reversal physically changes positions of elements. If implemented naively, reversing a segment requires moving every element inside that range, which becomes prohibitively expensive for arrays and query counts up to `10^5`.

The constraints strongly suggest that each operation should run in approximately `O(log n)` time.

This combination of requirements points toward an implicit balanced binary tree structure where:

- Elements are stored according to position rather than explicit indices.
- Subarrays can be split and merged efficiently.
- Reversal can be represented lazily.
- Aggregate XOR values can be maintained for range queries.

An implicit Treap satisfies these requirements.

Important edge cases include arrays of length one, reversing a single element, reversing the whole array repeatedly, performing updates after reversals, and querying ranges immediately after modifications.

## Approaches

### Brute Force

The most direct solution stores the array normally and executes operations literally.

For updates, assign the value directly.

For XOR queries, iterate through the requested subarray and compute XOR.

For reversals, reverse the selected slice in place.

This approach is correct because every operation exactly follows the problem definition.

However, the complexity becomes unacceptable.

Each XOR query may scan `O(n)` elements.

Each reversal may move `O(n)` elements.

With `10^5` operations and `10^5` elements, worst case complexity reaches `O(nq)` which is approximately `10^10`.

### Key Insight

The expensive operation is reversal.

Instead of physically moving elements, we represent the array as an implicit Treap.

Each node stores:

- One array value
- Subtree size
- XOR of its subtree
- A lazy reverse flag

The tree order represents array order.

To operate on a range `[l, r]`, split the tree into:

- prefix `[0, l-1]`
- middle `[l, r]`
- suffix `[r+1, n-1]`

Now:

- Update modifies one isolated node.
- XOR reads the stored aggregate.
- Reverse toggles a lazy flag.

Then merge the parts back.

All operations become logarithmic.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(nq) | O(1) | Direct simulation of array operations |
| Optimal | O((n + q) log n) | O(n) | Implicit Treap with lazy reversal and subtree XOR |

## Algorithm Walkthrough

### Data Structure

Each Treap node stores:

- `value`, current element
- `priority`, randomized heap key
- `size`, subtree size
- `xor_value`, XOR of subtree
- `rev`, lazy reversal flag
- `left`, `right`

### Operations

#### 1. Maintain subtree information

After any structural change:

- recompute subtree size
- recompute subtree XOR

This allows range answers in constant time.

#### 2. Push lazy reversal

If a node is marked reversed:

- swap left and right children
- propagate reverse flag downward

No elements are moved.

#### 3. Split by position

Split a tree into:

- first `k` elements
- remaining elements

This allows extraction of arbitrary ranges.

#### 4. Merge trees

Merge two Treaps assuming all positions in the first come before the second.

Heap priorities maintain balance.

#### 5. Build initial structure

Insert array values one by one.

Each insertion merges a new node onto the right.

#### 6. Handle update query

Split:

- `[0, index-1]`
- `[index]`
- `[index+1, end]`

Change isolated node value.

Recompute.

Merge back.

#### 7. Handle XOR query

Split:

- prefix
- target range
- suffix

Read subtree XOR.

Store answer.

Merge back.

#### 8. Handle reverse query

Split:

- prefix
- target range
- suffix

Toggle reverse flag.

Merge back.

### Why it works

The Treap invariant guarantees that in-order traversal always equals the current array order.

Splitting isolates exactly the requested positions.

Subtree XOR always equals XOR of the represented segment because every update recomputes aggregates and reversal preserves values while only changing order. Since XOR is associative and independent of ordering, lazy reversal does not invalidate subtree XOR.

## Python Solution

```python
from typing import List, Optional
import random

class Node:
    __slots__ = (
        "value",
        "priority",
        "left",
        "right",
        "size",
        "xor_value",
        "rev",
    )

    def __init__(self, value: int):
        self.value = value
        self.priority = random.randint(1, 1 << 30)
        self.left = None
        self.right = None
        self.size = 1
        self.xor_value = value
        self.rev = False

def size(node: Optional[Node]) -> int:
    return node.size if node else 0

def subtree_xor(node: Optional[Node]) -> int:
    return node.xor_value if node else 0

def push(node: Optional[Node]) -> None:
    if not node or not node.rev:
        return

    node.left, node.right = node.right, node.left

    if node.left:
        node.left.rev ^= True

    if node.right:
        node.right.rev ^= True

    node.rev = False

def pull(node: Optional[Node]) -> None:
    if not node:
        return

    node.size = 1 + size(node.left) + size(node.right)

    node.xor_value = (
        subtree_xor(node.left)
        ^ node.value
        ^ subtree_xor(node.right)
    )

def split(root: Optional[Node], k: int):
    if not root:
        return None, None

    push(root)

    if size(root.left) >= k:
        left, root.left = split(root.left, k)
        pull(root)
        return left, root

    root.right, right = split(
        root.right,
        k - size(root.left) - 1,
    )
    pull(root)
    return root, right

def merge(a: Optional[Node], b: Optional[Node]):
    if not a or not b:
        return a or b

    push(a)
    push(b)

    if a.priority < b.priority:
        a.right = merge(a.right, b)
        pull(a)
        return a

    b.left = merge(a, b.left)
    pull(b)
    return b

class Solution:
    def getResults(
        self,
        nums: List[int],
        queries: List[List[int]],
    ) -> List[int]:

        root = None

        for value in nums:
            root = merge(root, Node(value))

        results = []

        for op, x, y in queries:

            if op == 1:
                left, temp = split(root, x)
                mid, right = split(temp, 1)

                mid.value = y
                pull(mid)

                root = merge(
                    left,
                    merge(mid, right),
                )

            elif op == 2:
                left, temp = split(root, x)
                mid, right = split(
                    temp,
                    y - x + 1,
                )

                results.append(
                    subtree_xor(mid)
                )

                root = merge(
                    left,
                    merge(mid, right),
                )

            else:
                left, temp = split(root, x)
                mid, right = split(
                    temp,
                    y - x + 1,
                )

                if mid:
                    mid.rev ^= True

                root = merge(
                    left,
                    merge(mid, right),
                )

        return results
```

The implementation mirrors the algorithm directly.

`split()` isolates positional ranges.

`merge()` restores the array.

`push()` applies deferred reversals only when traversal reaches a node.

`pull()` keeps subtree metadata correct.

Updates modify one node, XOR reads aggregate information, and reversal only flips a flag.

## Go Solution

```go
package main

import (
	"math/rand"
)

type Node struct {
	value    int
	priority int
	size     int
	xorValue int
	rev      bool
	left     *Node
	right    *Node
}

func newNode(v int) *Node {
	return &Node{
		value:    v,
		priority: rand.Int(),
		size:     1,
		xorValue: v,
	}
}

func size(n *Node) int {
	if n == nil {
		return 0
	}
	return n.size
}

func xorVal(n *Node) int {
	if n == nil {
		return 0
	}
	return n.xorValue
}

func push(n *Node) {
	if n == nil || !n.rev {
		return
	}

	n.left, n.right = n.right, n.left

	if n.left != nil {
		n.left.rev = !n.left.rev
	}

	if n.right != nil {
		n.right.rev = !n.right.rev
	}

	n.rev = false
}

func pull(n *Node) {
	if n == nil {
		return
	}

	n.size = 1 + size(n.left) + size(n.right)
	n.xorValue = xorVal(n.left) ^ n.value ^ xorVal(n.right)
}

func split(root *Node, k int) (*Node, *Node) {
	if root == nil {
		return nil, nil
	}

	push(root)

	if size(root.left) >= k {
		l, r := split(root.left, k)
		root.left = r
		pull(root)
		return l, root
	}

	l, r := split(
		root.right,
		k-size(root.left)-1,
	)

	root.right = l
	pull(root)

	return root, r
}

func merge(a, b *Node) *Node {
	if a == nil {
		return b
	}

	if b == nil {
		return a
	}

	push(a)
	push(b)

	if a.priority < b.priority {
		a.right = merge(a.right, b)
		pull(a)
		return a
	}

	b.left = merge(a, b.left)
	pull(b)

	return b
}

func getResults(
	nums []int,
	queries [][]int,
) []int {

	var root *Node

	for _, v := range nums {
		root = merge(root, newNode(v))
	}

	var ans []int

	for _, q := range queries {

		op := q[0]
		x := q[1]
		y := q[2]

		if op == 1 {

			l, t := split(root, x)
			m, r := split(t, 1)

			m.value = y
			pull(m)

			root = merge(
				l,
				merge(m, r),
			)

		} else if op == 2 {

			l, t := split(root, x)
			m, r := split(
				t,
				y-x+1,
			)

			ans = append(
				ans,
				xorVal(m),
			)

			root = merge(
				l,
				merge(m, r),
			)

		} else {

			l, t := split(root, x)
			m, r := split(
				t,
				y-x+1,
			)

			if m != nil {
				m.rev = !m.rev
			}

			root = merge(
				l,
				merge(m, r),
			)
		}
	}

	return ans
}
```

The Go version follows the same structure. Go uses pointers for tree nodes and relies on `nil` rather than optional references. Integer overflow is not an issue because XOR values remain within standard integer limits.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O((n + q) log n) | Each split and merge is logarithmic |
| Space | O(n) | One Treap node per array element |

Each query performs a constant number of split and merge operations. Because randomized priorities keep the Treap balanced in expectation, height remains `O(log n)`.

## Edge Cases

### Reversing a Single Element

A reversal where `left == right` should not change the array. Implementations that physically swap elements may accidentally modify structure. The Treap handles this naturally because toggling a reverse flag on a one-node subtree leaves the order unchanged.

### Updating After Multiple Reversals

After reversals, logical positions no longer match original indices. Array-based shortcuts often update the wrong element. Splitting by current position guarantees the update targets the correct node regardless of previous reversals.

### Querying the Entire Array

Range `[0, n-1]` isolates the whole Treap as the middle segment. The subtree XOR already stores the aggregate answer, so no traversal is required.

### Consecutive Reversals on Overlapping Ranges

Overlapping reversals frequently break implementations that eagerly reorder data. Lazy propagation composes correctly because reverse flags accumulate and cancel naturally when applied twice.
