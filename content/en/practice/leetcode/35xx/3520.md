## Solution

## Problem Understanding

We are given an array `nums` and an integer `k`.

For a pair of indices `(i, j)` to be considered a valid inversion pair under threshold `x`, three conditions must hold:

1. `i < j`
2. `nums[i] > nums[j]`
3. `nums[i] - nums[j] <= x`

The first two conditions are the standard inversion definition. The third condition adds an upper bound on the value difference between the two elements.

The goal is not to count inversion pairs directly. Instead, we must find the smallest integer threshold `x` such that the number of inversion pairs satisfying the above rules is at least `k`.

If even the largest possible threshold cannot produce `k` inversion pairs, we return `-1`.

The constraints are important:

- `n = nums.length <= 10^4`
- `nums[i] <= 10^9`
- `k <= 10^9`

A naive solution that checks all pairs requires `O(n²)` time. Since `n` can be `10^4`, that would require up to 100 million pair checks, which is too expensive, especially when combined with a binary search over thresholds.

The key observation is that the number of valid inversion pairs is monotonic with respect to the threshold:

- Increasing `x` can only add more valid pairs.
- It can never remove an existing valid pair.

This monotonic property immediately suggests binary search on the answer.

Some important edge cases are:

- Arrays with no inversions at all.
- Arrays where all elements are equal.
- Very large `k` values that exceed the total number of inversions.
- Threshold `0`, which only allows inversions with equal values, but equal values cannot satisfy `nums[i] > nums[j]`, so the count is always zero.
- Arrays containing large values up to `10^9`, which prevents using value-indexed frequency arrays directly.

## Approaches

### Brute Force

The most direct approach is to try every possible threshold and count all valid inversion pairs.

For a fixed threshold `x`, we examine every pair `(i, j)` with `i < j` and check whether:

```
nums[i] > nums[j]
nums[i] - nums[j] <= x
```

This correctly counts the valid pairs.

Since the answer is monotonic, we could binary search on `x`, but each count operation would still require `O(n²)` work.

With `n = 10^4`, a single count requires approximately 50 million pair checks. Repeating that inside a binary search is far too slow.

### Key Insight

For a fixed threshold `x`, consider processing the array from left to right.

When we are at position `j` with value `v = nums[j]`, we need to count previous elements satisfying:

```
v < nums[i] <= v + x
```

The first inequality guarantees an inversion (`nums[i] > v`).

The second inequality enforces the threshold condition.

Therefore, for every position, we need a dynamic data structure that supports:

- Inserting previous values.
- Counting how many inserted values lie inside a value range.

A Binary Indexed Tree (Fenwick Tree) after coordinate compression supports exactly these operations in `O(log n)` time.

This allows us to count valid pairs for a fixed threshold in `O(n log n)` time.

Since the count is monotonic in `x`, we combine this with binary search on the threshold.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n² log V) | O(1) | Binary search answer, but counting pairs is quadratic |
| Optimal | O(n log n log V) | O(n) | Binary search + Fenwick Tree + coordinate compression |

Here `V` denotes the value range of the answer, which is at most `max(nums) - min(nums)`.

## Algorithm Walkthrough

### 1. Coordinate compress the values

The array values may be as large as `10^9`.

A Fenwick Tree requires compact indices, so we sort the distinct values and map them to positions:

```
sorted_values = sorted(set(nums))
```

### 2. Build a counting function

Create a function `countPairs(x)` that returns the number of inversion pairs whose difference is at most `x`.

While scanning from left to right:

- The Fenwick Tree stores frequencies of all previously seen values.
- For the current value `v`, we need previous values in:

```
(v, v + x]
```

The count is:

```
count(<= v + x) - count(<= v)
```

We add this quantity to the answer.

Then we insert `v` into the Fenwick Tree.

### 3. Count all inversions

Before binary searching, determine whether a solution exists.

If we choose:

```
x = max(nums) - min(nums)
```

then every inversion automatically satisfies the threshold condition.

Therefore:

```
total_inversions = countPairs(max(nums) - min(nums))
```

If:

```
total_inversions < k
```

then no threshold can ever reach `k`, and we return `-1`.

### 4. Binary search the answer

The threshold lies in:

```
[0, max(nums) - min(nums)]
```

For each midpoint:

```
mid = (left + right) // 2
```

compute:

```
countPairs(mid)
```

If the count is at least `k`, the answer may be smaller, so search left.

Otherwise, search right.

### 5. Return the smallest valid threshold

The binary search maintains the invariant that the answer remains inside the current range.

When it finishes, `left` is the minimum threshold producing at least `k` valid inversion pairs.

### Why it works

For a fixed threshold `x`, `countPairs(x)` correctly counts every pair `(i, j)` such that:

```
nums[j] < nums[i] <= nums[j] + x
```

which is exactly equivalent to:

```
nums[i] > nums[j]
nums[i] - nums[j] <= x
```

The count is monotonic because increasing `x` only enlarges the allowable value range. Therefore, once a threshold produces at least `k` pairs, every larger threshold does as well. Binary search on this monotonic predicate finds the smallest valid threshold.

## Python Solution

```python
from typing import List
from bisect import bisect_right

class FenwickTree:
    def __init__(self, n: int):
        self.n = n
        self.bit = [0] * (n + 1)

    def add(self, idx: int, delta: int) -> None:
        while idx <= self.n:
            self.bit[idx] += delta
            idx += idx & -idx

    def query(self, idx: int) -> int:
        result = 0
        while idx > 0:
            result += self.bit[idx]
            idx -= idx & -idx
        return result

class Solution:
    def minThreshold(self, nums: List[int], k: int) -> int:
        n = len(nums)

        values = sorted(set(nums))
        m = len(values)

        def count_pairs(threshold: int) -> int:
            bit = FenwickTree(m)
            total = 0

            for value in nums:
                right_idx = bisect_right(values, value + threshold)
                value_idx = bisect_right(values, value)

                total += bit.query(right_idx) - bit.query(value_idx)

                insert_idx = bisect_right(values, value)
                bit.add(insert_idx, 1)

            return total

        max_threshold = max(nums) - min(nums)

        if count_pairs(max_threshold) < k:
            return -1

        left = 0
        right = max_threshold

        while left < right:
            mid = (left + right) // 2

            if count_pairs(mid) >= k:
                right = mid
            else:
                left = mid + 1

        return left
```

The implementation begins by coordinate-compressing the values so that a Fenwick Tree can be used efficiently.

The helper function `count_pairs(threshold)` processes the array from left to right. At each position, the Fenwick Tree contains exactly the values that appeared earlier in the array. The query computes how many previous values fall in the interval `(value, value + threshold]`, which is precisely the set of values that form valid inversion pairs with the current element.

After verifying that enough inversions exist at all, the code binary searches the threshold. The monotonicity of the count function guarantees correctness.

## Go Solution

```go
package main

import "sort"

type Fenwick struct {
	n   int
	bit []int64
}

func NewFenwick(n int) *Fenwick {
	return &Fenwick{
		n:   n,
		bit: make([]int64, n+1),
	}
}

func (f *Fenwick) Add(idx int, delta int64) {
	for idx <= f.n {
		f.bit[idx] += delta
		idx += idx & -idx
	}
}

func (f *Fenwick) Query(idx int) int64 {
	var res int64
	for idx > 0 {
		res += f.bit[idx]
		idx -= idx & -idx
	}
	return res
}

func minThreshold(nums []int, k int) int {
	values := append([]int(nil), nums...)
	sort.Ints(values)

	unique := make([]int, 0, len(values))
	for _, v := range values {
		if len(unique) == 0 || unique[len(unique)-1] != v {
			unique = append(unique, v)
		}
	}

	countPairs := func(threshold int) int64 {
		bit := NewFenwick(len(unique))
		var total int64

		for _, v := range nums {
			rightIdx := sort.Search(len(unique), func(i int) bool {
				return unique[i] > v+threshold
			})

			valueIdx := sort.Search(len(unique), func(i int) bool {
				return unique[i] > v
			})

			total += bit.Query(rightIdx) - bit.Query(valueIdx)

			bit.Add(valueIdx+1, 1)
		}

		return total
	}

	minVal := nums[0]
	maxVal := nums[0]

	for _, v := range nums {
		if v < minVal {
			minVal = v
		}
		if v > maxVal {
			maxVal = v
		}
	}

	maxThreshold := maxVal - minVal

	if countPairs(maxThreshold) < int64(k) {
		return -1
	}

	left, right := 0, maxThreshold

	for left < right {
		mid := left + (right-left)/2

		if countPairs(mid) >= int64(k) {
			right = mid
		} else {
			left = mid + 1
		}
	}

	return left
}
```

The Go implementation follows the same logic as the Python version. The primary difference is that the Fenwick Tree stores `int64` values to avoid overflow during counting. Since the number of inversion pairs can reach approximately `n(n-1)/2`, using 64-bit integers is safer than relying on the platform-dependent size of `int`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n log V) | Binary search over threshold, each check uses Fenwick Tree counting |
| Space | O(n) | Coordinate compression and Fenwick Tree |

The coordinate compression requires `O(n log n)` preprocessing. Each threshold check performs `n` Fenwick Tree operations, each taking `O(log n)`. Binary search requires `O(log V)` iterations, where `V = max(nums) - min(nums)`.

Therefore the total complexity is:

```
O(n log n log V)
```

with linear auxiliary space.

## Edge Cases

### No inversion pairs exist

An array such as:

```
[1, 2, 3, 4]
```

contains zero inversions.

A common mistake is to binary search anyway and accidentally return some threshold. The implementation first computes the total number of inversions obtainable with the maximum threshold. If that count is still smaller than `k`, it immediately returns `-1`.

### All values are equal

Consider:

```
[7, 7, 7, 7]
```

Although many pairs exist, none satisfy:

```
nums[i] > nums[j]
```

because all values are identical.

The Fenwick query counts only values strictly greater than the current value, so equal values are automatically excluded.

### Very large value differences

Consider:

```
[1000000000, 1]
```

The required threshold is:

```
999999999
```

Using coordinate compression avoids allocating arrays based on value magnitude. The algorithm depends only on the number of distinct values, not on their absolute size.

### Extremely large k

The constraints allow:

```
k = 10^9
```

which may exceed the total number of inversion pairs in the array.

The feasibility check using the maximum threshold correctly detects this situation and returns `-1` without performing unnecessary binary search work.
