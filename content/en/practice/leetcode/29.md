## Solution

## Problem Understanding

This problem asks us to implement integer division manually, without using the multiplication (`*`), division (`/`), or modulo (`%`) operators. Given two integers, `dividend` and `divisor`, we must compute how many times the divisor fits into the dividend and return the integer quotient.

The result must truncate toward zero. This means we discard the fractional part instead of rounding. For example:

- `10 / 3 = 3.333...`, truncated to `3`
- `7 / -3 = -2.333...`, truncated to `-2`

The input consists of two signed 32-bit integers. The divisor is guaranteed to be non-zero, so we never need to handle division by zero. However, the problem introduces an important overflow condition because the valid 32-bit signed integer range is:

$$[-2^{31}, 2^{31} - 1]$$

which corresponds to:

- Minimum: `-2147483648`
- Maximum: `2147483647`

One special case can overflow:

```
-2147483648 / -1 = 2147483648
```

This value is larger than the maximum allowed 32-bit signed integer, so the problem explicitly requires returning `2147483647` in this situation.

The constraints tell us several important things:

- Inputs may be very large in magnitude
- A naive repeated subtraction solution may become extremely slow
- We must carefully manage overflow behavior
- Bit manipulation is likely intended because the problem explicitly forbids built-in division operations

Several edge cases can easily break incorrect implementations:

- Dividing the minimum 32-bit integer
- Negative numbers and sign handling
- Dividing by `1` or `-1`
- Very large dividends with small divisors
- Cases where the quotient should be `0`

A correct solution must handle all of these carefully while staying efficient.

## Approaches

### Brute Force Approach

The most straightforward idea is repeated subtraction.

We repeatedly subtract the divisor from the dividend until the remaining value becomes smaller than the divisor. The number of successful subtractions is the quotient.

For example:

```
10 / 3

10 - 3 = 7
7 - 3 = 4
4 - 3 = 1

Performed subtraction 3 times, answer = 3
```

This works because division fundamentally represents repeated subtraction.

However, this approach is far too slow for large values. Consider:

```
2147483647 / 1
```

We would need over two billion subtraction operations, which is completely impractical.

The brute-force method is correct mathematically, but its time complexity is too large for the problem constraints.

### Optimal Approach, Bit Manipulation with Exponential Subtraction

The key insight is that repeated subtraction can be accelerated dramatically by subtracting large multiples of the divisor at once.

Instead of subtracting:

```
3 + 3 + 3 + 3 + ...
```

we can double the divisor repeatedly:

```
3
6
12
24
48
...
```

This is similar to how long division works manually.

Using left shifts, we can efficiently compute powers of two multiples:

```
divisor << k
```

which is equivalent to:

```
divisor * (2^k)
```

without using multiplication.

At each step:

- Find the largest doubled divisor that still fits into the remaining dividend
- Subtract it
- Add the corresponding power of two to the quotient

This reduces the number of operations dramatically, from linear time to logarithmic time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Repeatedly subtracts divisor one step at a time |
| Optimal | O((log n)^2) | O(1) | Uses exponential doubling with bit shifts |

## Algorithm Walkthrough

1. First, handle the overflow edge case.

If the dividend is `-2147483648` and the divisor is `-1`, the mathematical result would be `2147483648`, which exceeds the maximum 32-bit signed integer. In this case, immediately return `2147483647`.
2. Determine whether the final answer should be negative.

Division rules say:

- Positive ÷ Positive = Positive
- Negative ÷ Negative = Positive
- Positive ÷ Negative = Negative
- Negative ÷ Positive = Negative

We can detect this using:

```
negative = (dividend < 0) != (divisor < 0)
```

The result is negative only when exactly one number is negative.
3. Convert both numbers to positive values.

Working with positive numbers simplifies the subtraction and shifting logic considerably.
4. Initialize the quotient to `0`.

This variable accumulates the final result.
5. Repeatedly subtract the largest doubled divisor.

While the dividend is still large enough:

- Start with the original divisor
- Double it repeatedly using left shift
- Track how many times it has been doubled
- Stop when doubling further would exceed the remaining dividend
6. Subtract the largest valid multiple from the dividend.

This removes a large chunk at once instead of subtracting one divisor at a time.
7. Add the corresponding power of two to the quotient.

Since each doubling represents multiplication by `2`, we add the matching amount to the answer.
8. Apply the sign at the end.

If the result should be negative, negate the quotient.
9. Return the final quotient.

## Python Solution

```
class Solution:
    def divide(self, dividend: int, divisor: int) -> int:
        INT_MIN = -(2 ** 31)
        INT_MAX = (2 ** 31) - 1

        # Handle overflow case
        if dividend == INT_MIN and divisor == -1:
            return INT_MAX

        # Determine sign of result
        negative = (dividend < 0) != (divisor < 0)

        # Work with positive values
        dividend = abs(dividend)
        divisor = abs(divisor)

        quotient = 0

        while dividend >= divisor:
            temp_divisor = divisor
            multiple = 1

            # Double divisor until it exceeds dividend
            while dividend >= (temp_divisor << 1):
                temp_divisor <<= 1
                multiple <<= 1

            dividend -= temp_divisor
            quotient += multiple

        return -quotient if negative else quotient
```

The implementation begins by defining the 32-bit integer boundaries. This is necessary because Python integers can grow arbitrarily large, but the problem requires explicit 32-bit overflow handling.

Next, the overflow condition is checked. The only problematic case is dividing the minimum integer by `-1`.

The code then determines whether the final answer should be negative. Using XOR-style logic through `!=` makes the sign detection concise and correct.

Both values are converted to positive numbers because bit shifting and subtraction become easier to reason about when working only with non-negative integers.

The main loop continues while the remaining dividend is at least as large as the divisor. Inside this loop, the algorithm repeatedly doubles the divisor using left shifts. Each left shift multiplies the value by two.

The largest doubled divisor that still fits into the remaining dividend is selected. That value is subtracted, and the matching multiple is added to the quotient.

Finally, the correct sign is applied before returning the answer.

## Go Solution

```
func divide(dividend int, divisor int) int {
	const INT_MIN = -2147483648
	const INT_MAX = 2147483647

	// Handle overflow case
	if dividend == INT_MIN && divisor == -1 {
		return INT_MAX
	}

	// Determine sign
	negative := (dividend < 0) != (divisor < 0)

	// Convert to positive values
	a := dividend
	if a < 0 {
		a = -a
	}

	b := divisor
	if b < 0 {
		b = -b
	}

	quotient := 0

	for a >= b {
		tempDivisor := b
		multiple := 1

		// Double divisor until too large
		for a >= (tempDivisor << 1) {
			tempDivisor <<= 1
			multiple <<= 1
		}

		a -= tempDivisor
		quotient += multiple
	}

	if negative {
		return -quotient
	}

	return quotient
}
```

The Go implementation follows the same algorithmic structure as the Python version. One important difference is that Go integers have fixed sizes depending on the platform, so overflow behavior must be considered more carefully.

The overflow check is still necessary because dividing `INT_MIN` by `-1` exceeds the 32-bit signed integer limit.

Unlike Python, Go does not provide a built-in arbitrary precision integer type for normal integers, so the implementation relies on standard integer arithmetic and explicit constants.

The shifting and subtraction logic remains identical because Go supports bitwise left shifts directly.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O((log n)^2) | Outer loop runs logarithmically, inner doubling loop is also logarithmic |
| Space | O(1) | Only a constant number of variables are used |

The time complexity requires some careful reasoning. Each iteration of the outer loop removes a large power-of-two chunk from the dividend, so the number of iterations is logarithmic. Inside each iteration, we repeatedly double the divisor, which also takes logarithmic time. Multiplying these together gives:

$$O((\log n)^2)$$

The algorithm uses constant extra space because it stores only a few integer variables regardless of input size.

## Edge Cases

### Minimum Integer Divided by Negative One

The most important edge case is:

```
-2147483648 / -1
```

Mathematically this equals:

```
2147483648
```

but this exceeds the maximum 32-bit signed integer. Without special handling, many languages overflow here. The implementation explicitly checks this condition and returns `2147483647`.

### Negative Numbers

Sign handling is a common source of bugs in division problems. If the implementation mixes signed arithmetic carelessly, incorrect rounding or incorrect signs can appear.

The solution avoids this issue by:

- Computing the final sign separately
- Converting both numbers to positive values
- Performing all arithmetic using non-negative integers
- Applying the sign only once at the end

This keeps the logic simple and reliable.

### Dividend Smaller Than Divisor

Consider:

```
3 / 5
```

Since the divisor is larger than the dividend, the quotient should be `0`.

The algorithm handles this naturally because the outer loop condition:

```
while dividend >= divisor:
```

fails immediately, so no subtraction occurs and the quotient remains `0`.

### Dividing by One

When dividing by `1`:

```
12345 / 1
```

the answer should simply be the original number.

The algorithm still works correctly because repeated doubling quickly subtracts large chunks until the dividend becomes zero.

### Dividing Zero

If:

```
0 / 5
```

the result should be `0`.

The loop condition fails immediately because:

```
0 < 5
```

so the quotient stays `0` and the algorithm returns the correct result.
