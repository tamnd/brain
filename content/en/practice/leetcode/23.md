## Solution

## Problem Understanding

The problem gives us an array named `lists`, where each element is the head of a singly linked list. Every linked list is already sorted in ascending order. The task is to merge all of these sorted linked lists into one final sorted linked list.

This is essentially a generalization of the classic "merge two sorted lists" problem. Instead of only two lists, we may have anywhere from `0` to `10^4` lists. The total number of nodes across all lists is also bounded by `10^4`.

The input structure is important to understand clearly. Each element in `lists` is either:

- A pointer/reference to the head node of a sorted linked list
- `None` or `nil`, representing an empty list

The output should be the head of a single linked list containing every node from every input list, still sorted in ascending order.

For example:

```
lists = [
  1 -> 4 -> 5,
  1 -> 3 -> 4,
  2 -> 6
]
```

must become:

```
1 -> 1 -> 2 -> 3 -> 4 -> 4 -> 5 -> 6
```

The constraints tell us several important things about the problem scale:

- `k` can be very large, up to `10^4`
- Each individual list can contain up to `500` nodes
- The total number of nodes across all lists will not exceed `10^4`

This means that algorithms with excessive repeated scanning or repeated full sorting may become inefficient. We need a solution that efficiently combines already sorted structures.

Several edge cases are especially important:

- `lists` may be empty
- Some lists inside `lists` may themselves be empty
- All lists may be empty
- Only one list may exist
- Many lists may contain duplicate values
- Negative values are allowed
- Lists may have dramatically different lengths

A correct solution must gracefully handle all of these situations.

## Approaches

### Brute Force Approach

The most straightforward solution is to collect every value from every linked list into a single array, sort the array, then rebuild a new linked list from the sorted values.

The process works like this:

1. Traverse all linked lists
2. Store every node value in an array
3. Sort the array
4. Create a new linked list using the sorted values

This approach is correct because sorting all values globally guarantees ascending order in the final linked list.

However, this solution ignores an important property of the input: each individual list is already sorted. By flattening everything into an array and sorting again, we lose the advantage of the existing order.

If the total number of nodes is `N`, sorting requires `O(N log N)` time.

While this is acceptable for the given constraints, we can do better conceptually by merging sorted lists directly.

### Optimal Approach, Divide and Conquer

The key observation is that merging two sorted linked lists can be done efficiently in linear time.

Instead of merging all lists at once, we repeatedly merge pairs of lists:

- Merge list 0 and list 1
- Merge list 2 and list 3
- Continue pairwise
- Repeat until only one merged list remains

This is essentially the same idea used in merge sort.

At every round, the number of lists is cut roughly in half. Since there are `k` lists initially, we perform approximately `log k` rounds of merging.

Each node participates in merging work once per round, leading to total complexity:

```
O(N log k)
```

where:

- `N` is the total number of nodes
- `k` is the number of lists

This approach is significantly more efficient and fully leverages the fact that every list is already sorted.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(N log N) | O(N) | Collect all values, sort them, rebuild list |
| Optimal | O(N log k) | O(1) auxiliary, excluding recursion/merged structure | Repeatedly merge sorted lists pairwise |

## Algorithm Walkthrough

1. First, handle the edge case where `lists` is empty. If there are no linked lists, return `None` immediately.
2. Define a helper function that merges two sorted linked lists. This helper works similarly to the classic merge step from merge sort.
3. Inside the merge helper, create a dummy node. The dummy node simplifies pointer handling because we always append nodes after the current tail.
4. Maintain a pointer called `tail` that tracks the end of the merged list being built.
5. Compare the current nodes from both lists:

- If the first list's node value is smaller, attach that node to `tail`
- Otherwise attach the second list's node
6. Move the chosen list pointer forward and also move `tail` forward.
7. Continue until one list becomes empty.
8. Attach the remaining nodes from the non-empty list directly to the merged list. Since the remaining portion is already sorted, no additional comparisons are needed.
9. Return `dummy.next`, which is the real head of the merged list.
10. Now perform pairwise merging on the entire `lists` array.
11. While more than one list remains:

- Create a new array called `merged_lists`
- Iterate through `lists` two at a time
- Merge adjacent pairs
- Append the merged result to `merged_lists`

1. Replace `lists` with `merged_lists`.
2. Eventually only one merged list remains. Return that final list.

This divide and conquer structure is efficient because every merging round reduces the number of active lists by approximately half.

## Python Solution

```
# Definition for singly-linked list.
# class ListNode:
#     def __init__(self, val=0, next=None):
#         self.val = val
#         self.next = next

from typing import List, Optional

class Solution:
    def mergeKLists(
        self,
        lists: List[Optional[ListNode]]
    ) -> Optional[ListNode]:

        if not lists:
            return None

        def merge_two_lists(
            list1: Optional[ListNode],
            list2: Optional[ListNode]
        ) -> Optional[ListNode]:

            dummy = ListNode(0)
            tail = dummy

            while list1 and list2:
                if list1.val <= list2.val:
                    tail.next = list1
                    list1 = list1.next
                else:
                    tail.next = list2
                    list2 = list2.next

                tail = tail.next

            if list1:
                tail.next = list1
            else:
                tail.next = list2

            return dummy.next

        while len(lists) > 1:
            merged_lists = []

            for i in range(0, len(lists), 2):
                list1 = lists[i]

                if i + 1 < len(lists):
                    list2 = lists[i + 1]
                else:
                    list2 = None

                merged_lists.append(
                    merge_two_lists(list1, list2)
                )

            lists = merged_lists

        return lists[0]
```

The implementation begins by checking whether the input array itself is empty. If so, there is nothing to merge.

The `merge_two_lists` helper performs the standard linear merge operation for two sorted linked lists. A dummy node is used so we do not need special logic for the first inserted node.

The main loop repeatedly merges adjacent pairs of lists. Every iteration reduces the number of lists approximately by half.

For example:

```
8 lists -> 4 lists -> 2 lists -> 1 list
```

This structure mirrors merge sort and gives the logarithmic number of merge rounds that produces the optimal complexity.

An important detail is that we reuse existing nodes instead of allocating entirely new nodes. This keeps auxiliary memory usage low.

## Go Solution

```
/**
 * Definition for singly-linked list.
 * type ListNode struct {
 *     Val int
 *     Next *ListNode
 * }
 */

func mergeKLists(lists []*ListNode) *ListNode {
    if len(lists) == 0 {
        return nil
    }

    var mergeTwoLists func(list1, list2 *ListNode) *ListNode

    mergeTwoLists = func(list1, list2 *ListNode) *ListNode {
        dummy := &ListNode{}
        tail := dummy

        for list1 != nil && list2 != nil {
            if list1.Val <= list2.Val {
                tail.Next = list1
                list1 = list1.Next
            } else {
                tail.Next = list2
                list2 = list2.Next
            }

            tail = tail.Next
        }

        if list1 != nil {
            tail.Next = list1
        } else {
            tail.Next = list2
        }

        return dummy.Next
    }

    for len(lists) > 1 {
        mergedLists := []*ListNode{}

        for i := 0; i < len(lists); i += 2 {
            list1 := lists[i]

            var list2 *ListNode

            if i+1 < len(lists) {
                list2 = lists[i+1]
            }

            mergedLists = append(
                mergedLists,
                mergeTwoLists(list1, list2),
            )
        }

        lists = mergedLists
    }

    return lists[0]
}
```

The Go solution follows the same algorithmic structure as the Python implementation.

One important Go-specific detail is the handling of `nil` pointers. Go requires explicit `nil` checks before dereferencing linked list nodes.

Slices are dynamic, so `mergedLists` is built using `append`.

Unlike Python, Go does not support nested named functions as naturally, so we define `mergeTwoLists` as a function variable assigned to an anonymous function.

Integer overflow is not a concern because node values remain within small bounds.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(N log k) | Each node participates in merging across log k rounds |
| Space | O(1) auxiliary | Only pointer manipulation is used, excluding output structure |

The divide and conquer process creates approximately `log k` rounds of merging because the number of lists is halved every round.

During each round, every node is processed exactly once. Since there are `N` total nodes, each round costs `O(N)`.

Combining these gives:

```
O(N log k)
```

The algorithm reuses existing linked list nodes instead of allocating additional arrays or copying values, so the auxiliary space usage remains constant aside from temporary bookkeeping structures.

## Edge Cases

### Empty Input Array

The input may be:

```
lists = []
```

A naive implementation might attempt to access `lists[0]`, causing an index error.

The solution avoids this by checking:

```
if not lists:
    return None
```

immediately at the start.

### Lists Containing Empty Lists

The input may contain many empty linked lists:

```
lists = [[], [], []]
```

During merging, these appear as `None` references.

The merge helper naturally handles this because if one list is `None`, it simply attaches the other list.

If both are `None`, the result is also `None`.

### Odd Number of Lists

When the number of lists is odd, one list will not have a partner during a merge round.

For example:

```
[ A, B, C ]
```

After merging `A` and `B`, list `C` must still survive into the next round.

The implementation handles this by setting:

```
list2 = None
```

when no partner exists.

This allows the final unpaired list to pass through unchanged.

### Duplicate Values

Many linked lists may contain identical values.

For example:

```
[1->1->1]
[1->1]
```

The merge comparison uses `<=`, ensuring duplicates are merged correctly while preserving sorted order.

### Single List Input

If only one list exists:

```
lists = [1->2->3]
```

the outer merging loop never runs because `len(lists) == 1`.

The algorithm simply returns the original list directly, which is already sorted.
