# LeetCode 1429 - First Unique Number

## Problem Understanding

This problem asks us to design a data structure that continuously tracks the first unique number in a queue of integers.

We are given an initial list of integers, which represents the starting queue. After initialization, we must support two operations efficiently:

1. `showFirstUnique()`

This returns the first number in queue order that appears exactly once.
2. `add(value)`

This appends a new number to the queue.

The important detail is that uniqueness depends on the entire queue at the current moment. A number that was previously unique may stop being unique after another occurrence is added later.

For example, if the queue is:

```
[2, 3, 5]
```

then `2` is the first unique number because it appears once and comes earliest.

After adding another `2`:

```
[2, 3, 5, 2]
```

the number `2` is no longer unique, so the first unique number becomes `3`.

The constraints are important:

- The queue can contain up to `10^5` numbers initially.
- There can be up to `50000` operations.
- Values can be large, up to `10^8`.

These constraints immediately tell us that repeatedly scanning the entire queue to find the first unique number would be too slow. We need an efficient design that supports both insertion and retrieval quickly.

Several edge cases are important:

- The queue may contain no unique numbers at all.
- A number may start unique and later become duplicated.
- The initial array itself may already contain many duplicates.
- A single element queue should work correctly.
- Multiple duplicate insertions should not corrupt the tracking logic.

The problem guarantees that all inputs are valid integers and that operations follow the specified constraints.

## Approaches

### Brute Force Approach

The most straightforward solution is to store the queue and maintain frequency counts for every number.

Whenever `showFirstUnique()` is called, we scan the queue from front to back and return the first number whose frequency is exactly `1`.

When `add(value)` is called:

- Append the value to the queue
- Increase its frequency count

This approach is correct because scanning from left to right guarantees that we find the earliest unique value.

However, it becomes inefficient because every `showFirstUnique()` call may require scanning the entire queue.

If the queue grows to size `N`, each query costs `O(N)`. With many operations, this becomes too slow.

### Optimal Approach

The key insight is that once a number becomes non-unique, it will never become unique again.

We can combine:

- A queue to preserve insertion order
- A hash map to count frequencies

Instead of removing invalid elements immediately, we use a technique called lazy removal.

Whenever `showFirstUnique()` is called:

- Look at the front of the queue
- If its frequency is greater than `1`, remove it
- Continue until the front is unique or the queue becomes empty

This works efficiently because each element is removed from the queue at most once.

The hash map gives constant-time frequency updates, and the queue preserves the correct ordering.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(N) per query | O(N) | Scan queue every time to find first unique |
| Optimal | O(1) amortized per operation | O(N) | Queue plus frequency map with lazy cleanup |

## Algorithm Walkthrough

1. Create a frequency map.

The frequency map stores how many times each number appears. This allows constant-time uniqueness checks.
2. Create a queue.

The queue preserves insertion order. Every number added to the stream is appended to this queue.
3. During initialization, process every number in the input array.

For each number:

- Increment its frequency count
- Append it to the queue
4. When `add(value)` is called:

- Increment the value's frequency count
- Append the value to the queue

We always append because future operations may need to inspect this value in order.
5. When `showFirstUnique()` is called:

- While the queue is not empty and the front element has frequency greater than `1`, remove it from the front
- After cleanup:

- If the queue is empty, return `-1`
- Otherwise return the front element
6. The cleanup process is lazy.

We do not immediately remove duplicates when they become invalid because removing arbitrary elements from the middle of a queue is expensive.

Instead, we only discard invalid elements when they reach the front.

### Why it works

The queue always maintains insertion order, and the frequency map always reflects the current number of occurrences.

Any non-unique number will eventually be removed once it reaches the front of the queue. Since elements are processed in insertion order, the first remaining valid element is guaranteed to be the first unique number.

Each element enters and leaves the queue at most once, which gives efficient amortized performance.

## Python Solution

```python
from collections import deque
from typing import List

class FirstUnique:

    def __init__(self, nums: List[int]):
        self.queue = deque()
        self.frequency = {}

        for num in nums:
            self.add(num)

    def showFirstUnique(self) -> int:
        while self.queue and self.frequency[self.queue[0]] > 1:
            self.queue.popleft()

        if not self.queue:
            return -1

        return self.queue[0]

    def add(self, value: int) -> None:
        self.frequency[value] = self.frequency.get(value, 0) + 1
        self.queue.append(value)

# Your FirstUnique object will be instantiated and called as such:
# obj = FirstUnique(nums)
# param_1 = obj.showFirstUnique()
# obj.add(value)
```

The implementation uses two core data structures.

The `frequency` dictionary tracks how many times each number appears. This allows constant-time uniqueness checks.

The `queue` stores all inserted numbers in arrival order. We use Python's `deque` because it supports efficient insertion at the back and removal from the front.

During initialization, the constructor processes every number using the `add()` method. This avoids duplicating logic.

The `add()` method increments the frequency count and appends the value to the queue.

The `showFirstUnique()` method performs lazy cleanup. While the front element is duplicated, it is removed from the queue. After cleanup, the remaining front element is guaranteed to be the first unique number.

If the queue becomes empty, there are no unique numbers left, so the method returns `-1`.

## Go Solution

```go
package main

type FirstUnique struct {
	queue     []int
	frequency map[int]int
	head      int
}

func Constructor(nums []int) FirstUnique {
	fu := FirstUnique{
		queue:     []int{},
		frequency: make(map[int]int),
		head:      0,
	}

	for _, num := range nums {
		fu.Add(num)
	}

	return fu
}

func (this *FirstUnique) ShowFirstUnique() int {
	for this.head < len(this.queue) &&
		this.frequency[this.queue[this.head]] > 1 {
		this.head++
	}

	if this.head >= len(this.queue) {
		return -1
	}

	return this.queue[this.head]
}

func (this *FirstUnique) Add(value int) {
	this.frequency[value]++
	this.queue = append(this.queue, value)
}

/**
 * Your FirstUnique object will be instantiated and called as such:
 * obj := Constructor(nums);
 * param_1 := obj.ShowFirstUnique();
 * obj.Add(value);
 */
```

The Go implementation uses a slice to simulate a queue.

Instead of physically removing elements from the front, we maintain a `head` index. This avoids expensive shifting operations that would occur if we repeatedly removed the first slice element.

The `frequency` map works the same way as the Python dictionary.

Go integers are fully sufficient for the problem constraints because values only go up to `10^8`, well within the standard integer range.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(1) amortized per operation | Each element is inserted once and removed once |
| Space | O(N) | Queue and frequency map store up to N elements |

Although `showFirstUnique()` contains a loop, each element can only be removed from the queue one time. Across all operations combined, the total number of removals is linear. Therefore the amortized cost per operation remains constant.

The space usage is linear because we store the queue contents and the frequency map.

## Edge Cases

### All Numbers Are Duplicates

A common bug occurs when every number appears more than once. In this case, the queue eventually becomes empty after cleanup.

For example:

```
[7,7,7]
```

A naive implementation might incorrectly return the front element without verifying uniqueness.

This implementation safely removes all duplicated elements and returns `-1` when the queue becomes empty.

### A Number Starts Unique but Later Becomes Duplicated

This is the central challenge of the problem.

For example:

```
[2,3,5]
```

Initially `2` is unique. After adding another `2`, it is no longer valid.

The lazy cleanup approach handles this naturally because frequency counts always reflect the latest state. Once the old `2` reaches the front again, it is discarded.

### Large Numbers of Duplicate Insertions

Repeated duplicate insertions can expose inefficient queue operations.

For example:

```
add(5) called thousands of times
```

If front removals required shifting the entire structure repeatedly, performance would degrade significantly.

The Python solution uses `deque`, which supports efficient front removal. The Go solution uses a head pointer instead of physically removing elements, preventing repeated shifting costs.
