## Solution

## Problem Understanding

The problem asks us to generate every fraction strictly between `0` and `1` whose denominator is less than or equal to `n`, but only if the fraction is already in its simplest form.

A fraction is considered simplified if the numerator and denominator share no common divisor greater than `1`. In mathematical terms, a fraction `a/b` is simplified when:

$\gcd(a,b)=1$

For example:

- `1/2` is simplified because `gcd(1, 2) = 1`
- `2/4` is not simplified because `gcd(2, 4) = 2`

The input consists of a single integer `n`. This value represents the maximum allowed denominator. We must consider every denominator from `2` through `n`, and for each denominator, test all numerators smaller than it.

The output should be a list of strings representing valid simplified fractions. The order does not matter.

The constraints are small:

- `1 <= n <= 100`

This tells us several important things:

- The total number of candidate fractions is manageable
- A quadratic style enumeration approach is acceptable
- Expensive optimizations are unnecessary
- Simplicity and correctness are more important than micro-optimization

There are several edge cases worth identifying early:

- When `n = 1`, there are no valid fractions because no denominator greater than `1` exists
- Fractions like `2/4`, `3/6`, and `4/8` must be excluded because they are reducible
- Fractions like `1/99` are always valid because `1` is coprime with every positive integer
- We must exclude `0/x` and `x/x` because the fraction must be strictly between `0` and `1`

## Approaches

### Brute Force Approach

The most direct solution is to generate every possible fraction `numerator/denominator` where:

- `1 <= numerator < denominator <= n`

For each generated fraction, we repeatedly attempt to simplify it or compare it against previously seen fractions.

One brute force method would be:

1. Generate every fraction
2. Convert each fraction to its decimal value
3. Store decimal representations in a set to avoid duplicates

This approach is problematic because floating point precision can create inaccuracies. Different fractions may map to imprecise decimal representations, and comparing floating point numbers is unreliable.

Another brute force variant would repeatedly divide numerator and denominator by common factors. While correct, it performs unnecessary repeated work.

Although the constraints are small enough that even inefficient methods might pass, these approaches are not mathematically elegant and are less reliable.

### Optimal Approach

The key observation is that a fraction is simplified exactly when the numerator and denominator are coprime.

Instead of trying to simplify fractions afterward, we can directly test whether:

$\gcd(numerator, denominator)=1$

If the greatest common divisor equals `1`, then the fraction is already simplified and should be included.

This works because any common divisor greater than `1` means the fraction can be reduced.

The Euclidean algorithm computes the greatest common divisor very efficiently, making this approach straightforward and clean.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n³) | O(n²) | Generates all fractions and repeatedly simplifies or compares them |
| Optimal | O(n² log n) | O(1), excluding output | Enumerates all candidate fractions and uses gcd to test validity |

## Algorithm Walkthrough

1. Initialize an empty result list that will store all valid simplified fractions as strings.
2. Iterate through every possible denominator from `2` to `n`. We start at `2` because denominator `1` cannot produce a fraction strictly between `0` and `1`.
3. For each denominator, iterate through all possible numerators from `1` to `denominator - 1`. This guarantees:

- The numerator is positive
- The fraction is less than `1`
- We never generate invalid fractions like `0/x` or `x/x`
4. Compute the greatest common divisor of the numerator and denominator using the Euclidean algorithm.
5. If the gcd equals `1`, the fraction is already simplified. Convert it into the format `"numerator/denominator"` and append it to the result list.
6. Continue until every denominator and numerator pair has been checked.
7. Return the final result list.

### Why it works

The algorithm works because every valid fraction between `0` and `1` with denominator at most `n` appears exactly once during enumeration.

The gcd test guarantees correctness:

- If `gcd(a, b) = 1`, then no common divisor exists, so the fraction is simplified
- If `gcd(a, b) > 1`, then the fraction can be reduced and must be excluded

Therefore, the algorithm includes all and only valid simplified fractions.

## Python Solution

```python
from typing import List
from math import gcd

class Solution:
    def simplifiedFractions(self, n: int) -> List[str]:
        result = []

        for denominator in range(2, n + 1):
            for numerator in range(1, denominator):
                if gcd(numerator, denominator) == 1:
                    result.append(f"{numerator}/{denominator}")

        return result
```

The implementation begins by importing `gcd` from Python's `math` module. Python already provides an efficient Euclidean algorithm implementation, so there is no need to write one manually.

The outer loop iterates through all denominators from `2` through `n`.

The inner loop iterates through every numerator smaller than the denominator. This guarantees the generated fraction lies strictly between `0` and `1`.

For each pair, the code computes `gcd(numerator, denominator)`. If the gcd equals `1`, the fraction is simplified and gets appended to the result list using formatted string syntax.

Finally, the method returns the complete list of simplified fractions.

## Go Solution

```go
import "strconv"

func gcd(a, b int) int {
	for b != 0 {
		a, b = b, a%b
	}
	return a
}

func simplifiedFractions(n int) []string {
	result := []string{}

	for denominator := 2; denominator <= n; denominator++ {
		for numerator := 1; numerator < denominator; numerator++ {
			if gcd(numerator, denominator) == 1 {
				fraction := strconv.Itoa(numerator) + "/" + strconv.Itoa(denominator)
				result = append(result, fraction)
			}
		}
	}

	return result
}
```

The Go solution follows the same logic as the Python version, but Go does not provide a built in gcd function in the standard integer utilities, so we implement the Euclidean algorithm manually.

String construction uses `strconv.Itoa` to convert integers into strings before concatenation.

Slices in Go dynamically expand as elements are appended, making them a natural replacement for Python lists.

Integer overflow is not a concern because the constraints are extremely small.

| --- | --- | --- |
| Time | O(n² log n) | There are O(n²) candidate fractions, each gcd computation costs O(log n) |
| Space | O(1), excluding output | Only a few variables are used besides the result list |

The nested loops generate approximately:

$\frac{n(n-1)}{2}$

candidate fractions.

For each candidate, we compute a gcd using the Euclidean algorithm, which runs in logarithmic time relative to the numbers involved.

The algorithm itself uses constant auxiliary space. The output list is not counted in auxiliary space complexity because it is required by the problem.

## Edge Cases

### Edge Case 1, `n = 1`

When `n = 1`, there are no denominators greater than `1`, so no fractions can exist strictly between `0` and `1`.

A buggy implementation might accidentally include invalid fractions or fail due to empty iteration ranges.

This implementation handles the case naturally because the denominator loop begins at `2`. The loop never executes, and the method correctly returns an empty list.

### Edge Case 2, Reducible Fractions

Fractions such as `2/4`, `3/6`, and `4/8` are mathematically equivalent to smaller fractions and must not appear in the output.

A naive implementation that simply generates every numerator and denominator pair would incorrectly include duplicates in reduced form.

The gcd check prevents this. Any fraction with a gcd greater than `1` is excluded immediately.

### Edge Case 3, Numerator Equal to 1

Fractions like `1/97` are always simplified because `1` shares no divisor greater than `1` with any denominator.

Some implementations attempt unnecessary reduction logic or special handling.

This algorithm handles such cases automatically because:

$\gcd(1,x)=1$

Therefore every fraction with numerator `1` is correctly included.

### Edge Case 4, Large Constraint Values

At the upper limit `n = 100`, the number of candidate fractions becomes significantly larger than in small examples.

An inefficient repeated simplification approach could become unnecessarily slow.

The Euclidean algorithm keeps gcd computations efficient, allowing the solution to comfortably handle the maximum constraint.
