# LeetCode 1427 - Perform String Shifts

## Problem Understanding

This problem asks us to simulate a sequence of shift operations on a string. Each operation tells us whether to shift the string left or right, and by how many positions.

A left shift removes characters from the beginning of the string and appends them to the end. For example, shifting `"abcde"` left by 2 produces `"cdeab"`.

A right shift removes characters from the end of the string and inserts them at the beginning. For example, shifting `"abcde"` right by 2 produces `"deabc"`.

The input consists of:

- A string `s`
- A list called `shift`
- Each element in `shift` is a pair `[direction, amount]`

The meaning of each pair is:

- `direction == 0` means perform a left shift
- `direction == 1` means perform a right shift
- `amount` specifies how many positions to shift

The task is to apply all operations in order and return the final resulting string.

The constraints are small:

- The string length is at most 100
- There are at most 100 operations

Even though the constraints are small enough for brute force simulation, the problem contains an important mathematical observation that leads to a cleaner and more efficient solution.

One key detail is that repeated shifts can cancel each other out. For example:

- Left shift by 3
- Right shift by 3

Together these produce no change at all.

Another important observation is that shifting by the string length produces the original string again. For example, shifting a length 5 string by 5 positions changes nothing.

Important edge cases include:

- A single-character string, where every shift leaves the string unchanged
- Shift amounts larger than the string length
- Operations that completely cancel each other
- Empty net movement after combining all shifts
- Multiple consecutive shifts in the same direction

The problem guarantees that all inputs are valid, so we do not need to handle malformed operations or invalid directions.

## Approaches

### Brute Force Approach

The most direct solution is to simulate every operation exactly as described.

For each shift operation:

- If it is a left shift, repeatedly move the first character to the end
- If it is a right shift, repeatedly move the last character to the front

This approach is easy to understand and always produces the correct answer because it follows the problem statement literally.

However, it performs unnecessary work. If we repeatedly shift the string many times, we may keep rebuilding intermediate strings even though many operations cancel each other out.

For example:

- Left by 1
- Left by 2
- Right by 3

These operations together produce no net change.

The brute force approach ignores this simplification.

### Optimal Approach

The key insight is that all shifts can be combined into one final net shift.

We can think of:

- Left shifts as negative movement
- Right shifts as positive movement

We accumulate all operations into a single integer:

- Add `amount` for right shifts
- Subtract `amount` for left shifts

After processing all operations, we reduce the final shift using modulo arithmetic:

```
net_shift %= len(s)
```

This works because rotating by the string length returns the string to its original form.

Finally, we perform exactly one rotation:

- A right shift by `k` means:

- Take the last `k` characters
- Place them before the remaining prefix

This gives a very compact and efficient solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m × n) | O(n) | Simulates every shift operation individually |
| Optimal | O(m + n) | O(n) | Combines all shifts into one final rotation |

Here:

- `n` is the length of the string
- `m` is the number of shift operations

## Algorithm Walkthrough

1. Initialize a variable called `net_shift` to `0`.

This variable stores the combined effect of all operations.
2. Iterate through every operation in `shift`.

Each operation contains:

- `direction`
- `amount`
3. Update `net_shift` based on the direction.

- If the operation is a right shift (`direction == 1`), add `amount`
- If the operation is a left shift (`direction == 0`), subtract `amount`
4. Reduce the final shift using modulo arithmetic.

```
net_shift %= len(s)
```

This prevents unnecessary full rotations.
5. Construct the final rotated string.

A right shift by `k` means:

- Take the last `k` characters
- Append the remaining prefix after them

In Python slicing notation:

```
s[-k:] + s[:-k]
```
6. Return the resulting string.

### Why it works

Every left or right shift is simply a rotation of the string. Rotations are additive, meaning multiple shifts can be combined into one equivalent rotation.

If we treat:

- Right shifts as positive
- Left shifts as negative

then the total effect is the sum of all operations.

Modulo arithmetic guarantees that rotations larger than the string length wrap around correctly. Since string rotation is cyclic, performing one final rotation with the combined shift produces exactly the same result as executing all operations individually.

## Python Solution

```python
from typing import List

class Solution:
    def stringShift(self, s: str, shift: List[List[int]]) -> str:
        net_shift = 0
        n = len(s)

        for direction, amount in shift:
            if direction == 1:
                net_shift += amount
            else:
                net_shift -= amount

        net_shift %= n

        return s[-net_shift:] + s[:-net_shift]
```

The implementation begins by storing the string length in `n`, since we will use it for modulo operations.

The loop processes every shift operation and accumulates the combined movement into `net_shift`. Right shifts increase the value, while left shifts decrease it.

After all operations are processed, modulo arithmetic reduces the shift into the valid range `[0, n - 1]`.

Finally, the string is rotated using slicing:

- `s[-net_shift:]` extracts the suffix that moves to the front
- `s[:-net_shift]` extracts the remaining prefix

Concatenating them produces the final shifted string.

One subtle detail is the behavior when `net_shift == 0`. In Python:

```python
s[-0:] == s
s[:-0] == ""
```

So the expression correctly returns the original string.

## Go Solution

```go
func stringShift(s string, shift [][]int) string {
    netShift := 0
    n := len(s)

    for _, operation := range shift {
        direction := operation[0]
        amount := operation[1]

        if direction == 1 {
            netShift += amount
        } else {
            netShift -= amount
        }
    }

    netShift = ((netShift % n) + n) % n

    return s[n-netShift:] + s[:n-netShift]
}
```

The Go implementation follows the same logic as the Python version.

One important difference is modulo handling. In Go, negative modulo values remain negative, so we normalize the result using:

```go
((netShift % n) + n) % n
```

This guarantees the shift is always within the valid range.

String slicing in Go also works slightly differently from Python, so we compute the split index explicitly using:

```go
n - netShift
```

The rest of the algorithm is identical.

| --- | --- | --- |

| Space | O(n) | The returned string requires linear space |

The algorithm performs one pass through the `shift` array, which costs `O(m)`, where `m` is the number of operations.

Constructing the rotated string requires copying characters into a new string, which costs `O(n)`.

No additional data structures proportional to the number of operations are used.

## Edge Cases

### Single Character String

A string with length 1 is an important edge case because every possible rotation should return the same string. A buggy implementation might accidentally produce invalid slices or indexing errors.

This implementation handles the case naturally because modulo reduction converts every shift into `0`, and the slicing operation still returns the original string correctly.

### Shift Amount Larger Than String Length

A shift amount may be much larger than the string itself. For example, shifting a length 5 string by 12 positions is equivalent to shifting by 2 positions.

Without modulo arithmetic, a solution might waste time performing unnecessary rotations or even produce incorrect indexing.

This implementation fixes the issue using:

```
net_shift %= len(s)
```

which guarantees the effective shift always stays within bounds.

### Operations That Cancel Each Other

Some sequences of operations produce no net movement at all. For example:

```
left by 3
right by 3
```

A brute force solution still performs all rotations individually.

The optimized approach correctly combines them into a net shift of zero. The final slicing operation then returns the original string unchanged.
