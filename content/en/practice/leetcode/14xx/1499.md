## Solution

## Problem Understanding

The problem gives us a list of 2D points sorted by their x-coordinates. Each point is represented as `[xi, yi]`, and the x-values are strictly increasing. We must find the maximum possible value of the equation:

$$yi + yj + |xi - xj|$$

subject to the condition:

$$|xi - xj| \le k$$

and only pairs where `i < j` are allowed.

Since the points are already sorted by x-coordinate and `xi < xj` for all `i < j`, the absolute value simplifies naturally:

$$|xi - xj| = xj - xi$$

because `xj` is always larger.

That transforms the equation into:

$$yi + yj + xj - xi$$

which can be rearranged as:

$$(yi - xi) + (yj + xj)$$

This rearrangement is the key observation that enables an efficient solution.

The input size can be as large as `10^5`, which immediately tells us that an `O(n^2)` solution will not work. A quadratic solution would require checking roughly `10^10` pairs in the worst case, which is far too slow.

The problem guarantees that at least one valid pair exists, so we never need to handle a scenario where no answer can be formed.

Several edge cases are important:

- Multiple points may fall outside the `k` window, requiring careful removal of outdated candidates.
- Coordinates and equation values can be negative, so we cannot assume positive intermediate values.
- Large coordinate values up to `10^8` mean arithmetic must be handled carefully, especially in Go where integer overflow can matter on smaller integer types.
- The best pair may not involve nearby y-values, it depends on the transformed expression `(yi - xi)`.

## Approaches

### Brute Force

The most straightforward solution is to check every possible pair `(i, j)` where `i < j`.

For each pair:

1. Compute `|xi - xj|`.
2. If the distance is at most `k`, evaluate the equation.
3. Track the maximum value found.

This works because it explicitly examines every valid pair, guaranteeing correctness.

However, there are `O(n^2)` possible pairs. With `n = 10^5`, this becomes computationally infeasible.

### Key Insight

Because the points are sorted by x-coordinate, we know:

$$|xi - xj| = xj - xi$$

So the equation becomes:

$$yi + yj + xj - xi$$

Rearranging:

$$(yi - xi) + (yj + xj)$$

When processing point `j`, the term `(yj + xj)` is fixed for that iteration. Therefore, we only need to find the previous point `i` within distance `k` that maximizes:

$$yi - xi$$

This transforms the problem into a sliding window maximum problem.

We maintain candidate points whose x-distance from the current point is at most `k`, and among them we want the maximum `(yi - xi)` value.

A monotonic deque is perfect for this:

- The deque stores candidate points in decreasing order of `(yi - xi)`.
- The front always contains the best candidate.
- Old points outside the `k` range are removed from the front.
- Inferior candidates are removed from the back.

This gives an `O(n)` solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Checks every pair |
| Optimal Monotonic Queue | O(n) | O(n) | Maintains best `(yi - xi)` candidates efficiently |

## Algorithm Walkthrough

1. Initialize a deque to store candidate points.

Each entry in the deque stores:

- `x`
- `y - x`

The deque is maintained in decreasing order of `(y - x)`.
2. Initialize the answer to negative infinity.

Since coordinates may be negative, we cannot initialize with zero.
3. Iterate through the points from left to right.

Let the current point be `(xj, yj)`.
4. Remove points from the front of the deque if they are too far away.

If:

$$xj - xi > k$$

then that point can never again participate in a valid pair with the current or future points, because x-values only increase.
5. If the deque is not empty, compute a candidate answer.

The front of the deque contains the maximum `(yi - xi)` among all valid previous points.

So:

$$candidate = (yj + xj) + (yi - xi)$$

Update the global maximum.
6. Compute the current point's contribution:

$$current = yj - xj$$
7. Remove weaker candidates from the back of the deque.

While the back has a value less than or equal to `current`, remove it.

Those points are useless because:

- They are older.
- They have a smaller or equal `(y - x)` value.

The current point dominates them completely.
8. Add the current point to the back of the deque.
9. Continue until all points are processed.

### Why it works

The deque always maintains all valid candidate points in decreasing order of `(yi - xi)`. Therefore, the front always gives the optimal previous point for the current `j`.

Because each point is inserted once and removed at most once, the algorithm remains linear.

The correctness relies on two invariants:

- Every point in the deque satisfies the distance constraint for future processing.
- The deque is monotonic decreasing in `(y - x)`, so the best candidate is always at the front.

## Python Solution

```python
from collections import deque
from typing import List

class Solution:
    def findMaxValueOfEquation(self, points: List[List[int]], k: int) -> int:
        # deque stores (x, y - x)
        candidates = deque()

        best = float("-inf")

        for x, y in points:
            # Remove points that are too far away
            while candidates and x - candidates[0][0] > k:
                candidates.popleft()

            # Use the best valid candidate
            if candidates:
                best = max(best, x + y + candidates[0][1])

            current_value = y - x

            # Maintain decreasing order of (y - x)
            while candidates and candidates[-1][1] <= current_value:
                candidates.pop()

            candidates.append((x, current_value))

        return best
```

The implementation follows the exact algorithm described earlier.

The deque stores tuples `(x, y - x)`. This lets us efficiently:

- Check whether a point is still within distance `k`.
- Retrieve the best `(y - x)` value immediately from the front.

The first `while` loop removes outdated points whose x-distance exceeds `k`.

The candidate equation value is computed using:

```python
x + y + candidates[0][1]
```

because `candidates[0][1]` already represents `(yi - xi)`.

The second `while` loop maintains monotonicity. Any point with a smaller or equal `(y - x)` value becomes useless once a stronger and newer point appears.

Each point enters and leaves the deque at most once, guaranteeing linear complexity.

## Go Solution

```go
func findMaxValueOfEquation(points [][]int, k int) int {
	type Pair struct {
		x     int
		value int // y - x
	}

	deque := []Pair{}
	best := -(1 << 60)

	for _, point := range points {
		x := point[0]
		y := point[1]

		// Remove points outside the valid range
		for len(deque) > 0 && x-deque[0].x > k {
			deque = deque[1:]
		}

		// Compute candidate answer
		if len(deque) > 0 {
			candidate := x + y + deque[0].value
			if candidate > best {
				best = candidate
			}
		}

		currentValue := y - x

		// Maintain decreasing order
		for len(deque) > 0 && deque[len(deque)-1].value <= currentValue {
			deque = deque[:len(deque)-1]
		}

		deque = append(deque, Pair{
			x:     x,
			value: currentValue,
		})
	}

	return best
}
```

The Go implementation mirrors the Python logic closely.

Instead of Python's `deque`, Go uses a slice to simulate a double-ended queue.

The `Pair` struct stores:

- `x`
- `y - x`

The value `-(1 << 60)` is used as a safely small initial value for `best`.

Slice operations:

- `deque = deque[1:]` removes from the front.
- `deque = deque[:len(deque)-1]` removes from the back.

Since Go slices share underlying arrays, this approach remains efficient for this problem size.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each point enters and leaves the deque at most once |
| Space | O(n) | The deque may store up to all points |

The linear runtime comes from amortized analysis. Although there are nested `while` loops, each point can only be inserted once and removed once. Therefore, the total number of deque operations across the entire algorithm is proportional to `n`.

The deque can grow to size `n` in the worst case, giving linear auxiliary space usage.

## Edge Cases

One important edge case occurs when points rapidly leave the valid `k` window. If outdated points are not removed correctly, the algorithm may use invalid pairs and produce an incorrect maximum. The implementation handles this by continuously removing points from the front while `xj - xi > k`.

Another tricky case involves negative coordinate values. Since both x and y values may be negative, the expression `(y - x)` can also be negative. Initializing the answer to zero would fail for all-negative valid results. The implementation correctly initializes the answer to negative infinity.

A third important case happens when a newly processed point dominates older candidates. Suppose two points are both within range, but one has a larger `(y - x)` value and is newer. The older point can never become optimal again. If we keep it in the deque, the structure grows unnecessarily and performance degrades. The monotonic deque logic removes these dominated points immediately.

Another subtle edge case occurs when `k = 0`. Since x-values are strictly increasing, no pair can satisfy `|xi - xj| <= 0`. The original problem guarantees at least one valid pair exists, which implicitly prevents such invalid input scenarios. The implementation assumes the guarantee holds.
