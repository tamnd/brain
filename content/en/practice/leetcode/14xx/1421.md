## Solution

## Problem Understanding

This problem gives us two database tables, `NPV` and `Queries`.

The `NPV` table stores the net present value, abbreviated as `npv`, for a specific inventory `id` in a specific `year`. The combination `(id, year)` is unique, which means there can never be two rows with the same pair of values.

The `Queries` table contains requests asking for the `npv` value associated with particular `(id, year)` combinations.

The task is to return a result table containing:

- the `id`
- the `year`
- the matching `npv`

for every row in the `Queries` table.

If a queried `(id, year)` pair does not exist in the `NPV` table, the answer for `npv` must be `0`.

In other words, we need to perform a lookup from `Queries` into `NPV`. Since some queries may not exist in the `NPV` table, we must still include those rows in the result and substitute `0` when no match is found.

This is the key detail that determines the type of SQL join we need. A regular inner join would remove unmatched rows, which would be incorrect. We instead need a left join so every query remains in the output.

The input size is small because this is labeled as an Easy database problem, but the solution should still be efficient and logically correct for all possible valid inputs.

Important edge cases include:

- Queries that do not exist in the `NPV` table
- `npv` values equal to `0`, which are valid stored values and should not be confused with missing rows
- Multiple different years for the same `id`
- Empty intersections between the tables

The problem guarantees that `(id, year)` is unique in both tables, so each query can match at most one row in `NPV`.

## Approaches

### Brute Force Approach

A brute force solution would examine every row in `Queries` and then scan the entire `NPV` table looking for a matching `(id, year)` pair.

For each query:

1. Iterate through all rows in `NPV`
2. Compare both `id` and `year`
3. If a match is found, return the associated `npv`
4. If no match exists after scanning the entire table, return `0`

This approach works because it explicitly checks every possible candidate row and guarantees that the correct match is found if it exists.

However, it is inefficient because every query may require scanning the entire `NPV` table. If there are `Q` query rows and `N` NPV rows, the total complexity becomes `O(Q × N)`.

In SQL terms, manually simulating this type of nested search is unnecessary because relational databases are designed to perform joins efficiently.

### Optimal Approach

The key insight is that this problem is fundamentally a table join operation.

We want:

- every row from `Queries`
- matching `npv` values from `NPV`
- default value `0` when no match exists

This is exactly what a `LEFT JOIN` provides.

We join the tables using both `id` and `year` because together they form the unique key identifying a record.

After the join:

- matched rows contain the real `npv`
- unmatched rows produce `NULL`

We then replace `NULL` with `0` using `IFNULL()` or `COALESCE()`.

This solution is concise, efficient, and directly models the problem statement.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(Q × N) | O(1) | Scans the entire NPV table for every query |
| Optimal | O(Q + N) approximately | O(1) extra | Uses SQL LEFT JOIN with indexed matching |

## Algorithm Walkthrough

1. Start with the `Queries` table because every query must appear in the final result.
2. Perform a `LEFT JOIN` from `Queries` to `NPV`.

The join condition uses:

- `Queries.id = NPV.id`
- `Queries.year = NPV.year`

Both columns are required because the pair uniquely identifies a row.
3. For rows where a matching record exists in `NPV`, retrieve the corresponding `npv`.
4. For rows where no match exists, SQL produces `NULL` for `npv`.
5. Use `IFNULL(NPV.npv, 0)` or `COALESCE(NPV.npv, 0)` to replace missing values with `0`.
6. Return the columns:

- `Queries.id`
- `Queries.year`
- computed `npv`

### Why it works

The correctness comes from the behavior of a `LEFT JOIN`.

A left join guarantees that every row from the left table, which is `Queries`, appears in the result exactly once. Matching rows from `NPV` contribute their `npv` values. When no match exists, SQL returns `NULL`, which we explicitly convert to `0`.

Because `(id, year)` is unique in `NPV`, each query can match at most one row, ensuring the output remains correct and unambiguous.

## Python Solution

```python
# Write your MySQL query statement below

SELECT
    q.id,
    q.year,
    IFNULL(n.npv, 0) AS npv
FROM Queries q
LEFT JOIN NPV n
ON q.id = n.id AND q.year = n.year;
```

This solution begins with the `Queries` table because the problem requires every query to appear in the output, even when no matching NPV record exists.

The `LEFT JOIN` attempts to match rows from `NPV` using both `id` and `year`. Since these two columns together form the primary key, the join uniquely identifies records.

When a matching row exists, `n.npv` contains the correct value. When no matching row exists, SQL produces `NULL`. The `IFNULL()` function converts those missing values into `0`, which matches the problem requirements.

The final result contains exactly the three required columns.

## Edge Cases

### Query Row Missing from NPV

One of the most important edge cases occurs when a queried `(id, year)` pair does not exist in the `NPV` table.

A naive inner join would completely remove such rows from the output, producing an incorrect result. The implementation avoids this issue by using a `LEFT JOIN`, which preserves all query rows and substitutes `0` for missing matches using `IFNULL()`.

### Stored NPV Value Equals Zero

A row may legitimately contain `npv = 0`.

This is different from a missing row, even though both eventually display `0` in the output. A careless implementation might incorrectly treat stored zeros as missing values.

The query handles this correctly because `IFNULL()` only replaces actual `NULL` values, not numeric zeros.

### Same ID Appearing in Multiple Years

An inventory `id` can appear across many years with different `npv` values.

If the join only used `id` and ignored `year`, incorrect matches would occur. The implementation correctly joins on both columns together:

```
ON q.id = n.id AND q.year = n.year
```

This guarantees that each query retrieves the exact intended record.
