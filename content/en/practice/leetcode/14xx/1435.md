# LeetCode 1435 - Create a Session Bar Chart

## Problem Understanding

This problem gives us a database table named `Sessions` with two columns:

| Column | Meaning |
| --- | --- |
| `session_id` | Unique identifier for a session |
| `duration` | Session duration in seconds |

The goal is to group session durations into predefined time ranges, called bins, and count how many sessions belong to each bin.

The bins are:

| Bin | Condition |
| --- | --- |
| `[0-5>` | duration is at least 0 minutes and less than 5 minutes |
| `[5-10>` | duration is at least 5 minutes and less than 10 minutes |
| `[10-15>` | duration is at least 10 minutes and less than 15 minutes |
| `15 or more` | duration is at least 15 minutes |

The durations are stored in seconds, so we must carefully convert the minute boundaries into seconds:

| Minutes | Seconds |
| --- | --- |
| 5 minutes | 300 seconds |
| 10 minutes | 600 seconds |
| 15 minutes | 900 seconds |

The output should contain two columns:

| Column | Meaning |
| --- | --- |
| `bin` | The name of the interval |
| `total` | Number of sessions inside that interval |

One important detail is that every bin must appear in the output, even if its count is zero. For example, if there are no sessions between 10 and 15 minutes, the result must still include `[10-15>` with a total of `0`.

The problem is categorized as an Easy SQL problem because it mainly tests conditional aggregation and grouping logic.

Several edge cases are important:

- Sessions with duration exactly equal to a boundary value such as `300`, `600`, or `900` seconds.
- Empty bins that must still appear in the result.
- Very small durations such as `0`.
- Large durations that belong in the final `"15 or more"` bucket.

The interval notation `[a-b>` means inclusive on the left side and exclusive on the right side. For example:

- `300` seconds belongs to `[5-10>`, not `[0-5>`
- `600` seconds belongs to `[10-15>`
- `900` seconds belongs to `15 or more`

Understanding these boundaries correctly is the most important part of the problem.

## Approaches

### Brute Force Approach

A brute force approach would iterate through all sessions multiple times, once for each bin.

For example:

- First pass counts sessions between `0` and `300`
- Second pass counts sessions between `300` and `600`
- Third pass counts sessions between `600` and `900`
- Fourth pass counts sessions greater than or equal to `900`

This works because every session is checked against each range condition independently.

However, this method is inefficient because the table is scanned repeatedly. Even though the dataset size in this problem is small, repeated scans become expensive on large database tables.

### Optimal Approach

The better approach is to scan the table once and classify every row into a bin using conditional aggregation.

The key insight is that each session belongs to exactly one bucket. We can use SQL conditional logic such as `CASE WHEN` to map every row into a category, then count how many rows fall into each category.

Another important observation is that bins with zero counts must still appear. To guarantee this, we can explicitly create the list of bins and left join the aggregated counts onto it.

This gives a clean and scalable solution with only one pass over the sessions table.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(4N) | O(1) | Scans the table once per bin |
| Optimal | O(N) | O(1) | Single pass classification using aggregation |

## Algorithm Walkthrough

1. Create a fixed list of all required bins.

We explicitly define the four labels:

- `[0-5>`
- `[5-10>`
- `[10-15>`
- `15 or more`

This guarantees every bin appears in the final output.
2. Traverse every session row exactly once.

For each session, examine the `duration` column.
3. Determine which interval the duration belongs to.

Since durations are stored in seconds, compare against:

- `< 300`
- `< 600`
- `< 900`
- otherwise `>= 900`
4. Assign the corresponding label.

Each session maps to exactly one bin because the intervals are mutually exclusive.
5. Count how many sessions belong to each label.

Use `GROUP BY` with `COUNT(*)` to aggregate totals.
6. Merge the aggregated counts with the predefined list of bins.

This ensures that bins with zero matching sessions still appear in the output.
7. Replace missing counts with zero.

A `LEFT JOIN` may produce `NULL` for empty bins, so use `COALESCE` to convert them to `0`.

### Why it works

The algorithm works because every possible duration falls into exactly one of the four mutually exclusive ranges. The predefined bin table guarantees all categories exist in the output, while the aggregation correctly counts the number of sessions assigned to each category.

## Python Solution

```python
# Write your MySQL query statement below

WITH bins AS (
    SELECT '[0-5>' AS bin
    UNION ALL
    SELECT '[5-10>'
    UNION ALL
    SELECT '[10-15>'
    UNION ALL
    SELECT '15 or more'
),
session_bins AS (
    SELECT
        CASE
            WHEN duration < 300 THEN '[0-5>'
            WHEN duration < 600 THEN '[5-10>'
            WHEN duration < 900 THEN '[10-15>'
            ELSE '15 or more'
        END AS bin
    FROM Sessions
)
SELECT
    b.bin,
    COALESCE(COUNT(sb.bin), 0) AS total
FROM bins b
LEFT JOIN session_bins sb
    ON b.bin = sb.bin
GROUP BY b.bin;
```

The solution begins by constructing a temporary table named `bins` that contains all required labels. This step is critical because the final answer must include empty categories.

Next, the `session_bins` common table expression classifies every session into exactly one interval using a `CASE` expression. The conditions are evaluated from smallest to largest threshold.

The main query then performs a `LEFT JOIN` from the predefined bins onto the aggregated session categories. Using a left join guarantees every bin remains in the result even if no rows match it.

Finally, `COUNT` computes the number of matching sessions, and `COALESCE` ensures empty bins display `0` instead of `NULL`.

## Edge Cases

### Boundary Durations

Values exactly equal to `300`, `600`, and `900` seconds are common sources of bugs because interval boundaries are inclusive on the left side and exclusive on the right side. A careless implementation using `<=` instead of `<` could place sessions into the wrong bucket.

This solution handles boundaries correctly by checking conditions in ascending order with strict `<` comparisons.

### Empty Categories

Some bins may contain no sessions at all. A normal `GROUP BY` query would omit those rows entirely, producing an incomplete result.

The implementation avoids this issue by explicitly creating the full list of bins first and performing a `LEFT JOIN`. This guarantees every category appears.

### Extremely Large Durations

A session duration may be much larger than `900` seconds. A naive implementation could accidentally ignore large values if it only checks finite ranges.

The final `ELSE '15 or more'` branch ensures every duration greater than or equal to `900` seconds is classified correctly.

### Very Small Durations

Sessions with durations close to zero must still belong to the first bucket. Since the first condition checks `duration < 300`, all valid small nonnegative values are handled automatically.
