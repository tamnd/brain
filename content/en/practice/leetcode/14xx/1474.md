## Solution

## Problem Understanding

The problem gives us the head of a singly linked list and two integers, `m` and `n`. We must repeatedly keep `m` nodes, then delete the next `n` nodes, continuing this pattern until we reach the end of the list.

In other words, we process the linked list in blocks. For each cycle:

- Preserve the first `m` nodes
- Remove the next `n` nodes
- Repeat the same process starting from the next remaining node

The important detail is that the deletions must happen directly within the linked list structure. We are not asked to create a new list containing only the kept nodes. Instead, we should reconnect pointers so the unwanted nodes are skipped.

The input consists of:

- `head`, which points to the first node of a singly linked list
- `m`, the number of consecutive nodes to keep
- `n`, the number of consecutive nodes to remove

The output is the modified linked list after all required deletions have been performed.

The constraints tell us several important things:

- The list can contain up to `10^4` nodes, which is large enough that inefficient repeated traversals should be avoided
- `m` and `n` can each be as large as `1000`
- The list is guaranteed to contain at least one node
- Since this is a linked list problem, pointer manipulation is the intended focus

A naive implementation could easily make mistakes with pointer updates, especially near the end of the list where fewer than `m` or `n` nodes may remain.

Several edge cases are especially important:

- The list may end while we are in the "keep" phase
- The list may end while we are in the "delete" phase
- `m = 1`, meaning we only preserve one node per cycle
- `n` may be larger than the number of remaining nodes
- The list may contain only a single node

The problem guarantees valid positive integers for both `m` and `n`, so we never need to handle zero-length keep or delete operations.

## Approaches

### Brute Force Approach

A brute-force strategy would first copy all linked list values into an array. Once the values are in an array, we could iterate through the array while applying the keep/delete pattern. After determining which values remain, we could build a brand new linked list from the surviving elements.

This approach works because arrays make indexed operations straightforward. We can easily determine whether a position belongs to a keep section or a delete section.

However, this solution is inefficient from a space perspective because it duplicates the entire linked list into an auxiliary array and then creates a second linked list. The follow-up specifically asks whether the problem can be solved in-place, which this method does not satisfy.

### Optimal In-Place Linked List Approach

The key observation is that we never actually need random access to the nodes. We only move through the list sequentially.

Because linked lists naturally support pointer rewiring, we can process the list directly in-place:

- Move forward `m - 1` times to keep the required nodes
- Skip the next `n` nodes by reconnecting pointers
- Continue the process from the next remaining node

The crucial insight is that deleting nodes in a singly linked list does not require physically removing memory. We only need to bypass the unwanted nodes by updating the `next` pointer of the last kept node.

This allows us to solve the problem in a single traversal with constant extra space.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Copies values into an array and rebuilds the list |
| Optimal | O(n) | O(1) | Modifies the linked list directly in-place |

## Algorithm Walkthrough

1. Start with a pointer called `current` at the head of the linked list.
2. Traverse the next `m` nodes because these nodes must be preserved. We move the pointer forward `m - 1` times so that `current` ends on the last node we want to keep.
3. If the list ends while keeping nodes, we stop immediately because there is nothing left to delete.
4. Store another pointer called `temp` at `current.next`. This marks the beginning of the nodes that should be deleted.
5. Move `temp` forward `n` times. These are the nodes we want to remove from the list.
6. Reconnect the list by setting `current.next = temp`. This skips all deleted nodes and connects the kept portion to the next surviving node.
7. Move `current` to `temp` and repeat the same process until the end of the list is reached.

### Why it works

The algorithm maintains an important invariant: before each cycle begins, `current` points to the first node of the next section to process. After preserving `m` nodes, the algorithm reconnects the final kept node directly to the node after the deleted section. Since every node is visited at most once and all unwanted nodes are bypassed correctly, the resulting linked list exactly follows the required keep/delete pattern.

## Python Solution

```python
# Definition for singly-linked list.
# class ListNode:
#     def __init__(self, val=0, next=None):
#         self.val = val
#         self.next = next

from typing import Optional

class Solution:
    def deleteNodes(
        self,
        head: Optional[ListNode],
        m: int,
        n: int
    ) -> Optional[ListNode]:

        current = head

        while current:

            # Keep m nodes
            for _ in range(1, m):
                if current is None:
                    return head
                current = current.next

            if current is None:
                return head

            # Start deleting from current.next
            temp = current.next

            # Skip n nodes
            for _ in range(n):
                if temp is None:
                    break
                temp = temp.next

            # Connect kept nodes to remaining list
            current.next = temp

            # Continue from next remaining node
            current = temp

        return head
```

The implementation begins by placing `current` at the head of the list. The outer `while` loop continues processing until the list is exhausted.

Inside each cycle, the first `for` loop advances through the nodes that should be preserved. We move `m - 1` times because the current node itself already counts as one preserved node.

After preserving the required nodes, `temp` is initialized to `current.next`, which marks the start of the deletion section.

The second `for` loop advances `temp` through the nodes that should be removed. Once this traversal finishes, `temp` points to the first node that should remain after deletion.

The line:

```python
current.next = temp
```

is the core pointer update. It skips all deleted nodes and reconnects the linked list.

Finally, `current` moves to `temp` so the next cycle begins at the correct location.

## Go Solution

```go
/**
 * Definition for singly-linked list.
 * type ListNode struct {
 *     Val int
 *     Next *ListNode
 * }
 */
func deleteNodes(head *ListNode, m int, n int) *ListNode {

    current := head

    for current != nil {

        // Keep m nodes
        for i := 1; i < m && current != nil; i++ {
            current = current.Next
        }

        if current == nil {
            return head
        }

        // Start deleting from current.Next
        temp := current.Next

        // Skip n nodes
        for i := 0; i < n && temp != nil; i++ {
            temp = temp.Next
        }

        // Reconnect list
        current.Next = temp

        // Continue processing
        current = temp
    }

    return head
}
```

The Go implementation follows the same logic as the Python version, but uses Go pointer syntax and `nil` checks instead of Python's `None`.

Because Go does not have optional typing syntax like Python, pointers naturally represent nullable references. The algorithm still works entirely in-place and does not allocate additional linked list nodes.

Integer overflow is not a concern because the constraints are small and loop counters remain within safe bounds.

The algorithm performs a single linear traversal of the linked list. Even though there are nested loops, each node is processed only once overall, either during the keep phase or the delete phase.

No additional data structures proportional to input size are allocated, so the auxiliary space complexity remains constant.

## Edge Cases

One important edge case occurs when the linked list ends during the keep phase. For example, if we need to keep five nodes but only three remain, a careless implementation might attempt to access `current.next` after `current` becomes `None`. The implementation prevents this by checking whether `current` becomes `None` during traversal and returning immediately.

Another critical case happens when the delete count extends beyond the remaining list length. Suppose we must delete ten nodes but only three remain. The algorithm safely handles this because the deletion loop stops as soon as `temp` becomes `None`. The last kept node is then connected to `None`, correctly terminating the list.

A third tricky case is when `m = 1`. In this situation, every cycle preserves exactly one node before deleting nodes. Off-by-one errors are common here because the keep loop runs `m - 1` times, which becomes zero iterations. The implementation handles this naturally because the current node itself already counts as the kept node.

Another subtle case involves very small lists, especially a single-node list. Since there may be no nodes available for deletion after preserving the first node, the algorithm must avoid dereferencing null pointers. The existing boundary checks ensure that no invalid pointer access occurs.
