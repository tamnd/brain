## Solution

## Problem Understanding

The problem gives us a grid with `n` rows and exactly `3` columns. Every cell in the grid must be painted using one of three colors:

- Red
- Yellow
- Green

The important restriction is that no two adjacent cells may have the same color. Adjacency is defined both horizontally and vertically.

This means:

- Two neighboring cells in the same row cannot share the same color.
- Two cells directly above and below each other cannot share the same color.

We are asked to compute the total number of valid ways to paint the entire `n x 3` grid.

Since the number of valid colorings grows very quickly as `n` increases, the answer must be returned modulo:

$$10^9 + 7$$

The constraints are important:

- `1 <= n <= 5000`

A brute force solution would attempt to try all color assignments. Since each cell has 3 choices and there are `3n` cells total, the search space is:

$$3^{3n}$$

This becomes astronomically large even for moderate values of `n`, so a direct enumeration approach is impossible.

The key observation is that the number of columns is fixed at exactly `3`. Because the width is so small, we can classify valid row patterns and build a dynamic programming transition between rows.

Important edge cases include:

- `n = 1`, where only a single row exists and we only need to satisfy horizontal adjacency constraints.
- Very large values such as `n = 5000`, where efficiency becomes critical.
- Repeated transitions between rows, where vertical conflicts must be carefully checked.

The problem guarantees valid input, so we do not need to handle invalid grid sizes or malformed values.

## Approaches

### Brute Force Approach

A brute force solution would recursively assign one of the three colors to every cell in the grid. During recursion, we would check whether the current color violates horizontal or vertical adjacency constraints.

This approach is correct because it explores every possible coloring and counts only the valid ones.

However, the total number of possible assignments is:

$$3^{3n}$$

For `n = 5000`, this is completely infeasible.

Even with pruning, the search space remains far too large.

### Key Insight

The grid width is fixed at `3`, which is very small.

Instead of thinking about individual cells, we can think about entire rows as states.

A row contains exactly 3 cells. Each cell may use one of 3 colors, subject to horizontal constraints. The number of valid row configurations is therefore very small.

There are only two structural types of valid rows:

1. Rows using exactly 2 colors in an ABA pattern.
2. Rows using 3 distinct colors in an ABC pattern.

Examples:

- ABA type:

- RYR
- YGY
- GRG
- ABC type:

- RYG
- YGR
- GRY

We can derive transitions between these row types and use dynamic programming to count valid grids efficiently.

The important observation is that every valid row belongs to one of these two categories, and the number of transitions between categories is fixed.

Let:

- `same` = number of ABA type rows
- `diff` = number of ABC type rows

Initially:

- `same = 6`
- `diff = 6`

because:

- There are 6 ABA patterns.
- There are 6 ABC patterns.

Transition rules:

- A new ABA row can follow:

- 3 ABA rows
- 2 ABC rows
- A new ABC row can follow:

- 2 ABA rows
- 2 ABC rows

This leads to the recurrence:

$$newSame = 3 \times same + 2 \times diff$$

$$newDiff = 2 \times same + 2 \times diff$$

We iterate this recurrence `n - 1` times.

## Algorithm Walkthrough

1. Observe that each row has exactly three cells, and adjacent cells in a row cannot share the same color.
2. Enumerate the possible valid row structures:

- ABA pattern, where first and third cells are equal.
- ABC pattern, where all three cells differ.
3. Count how many rows belong to each category:

- ABA type has 6 possibilities.
- ABC type has 6 possibilities.
4. Create two dynamic programming variables:

- `same` for ABA rows.
- `diff` for ABC rows.
5. Initialize:

- `same = 6`
- `diff = 6`
6. For each additional row from `2` to `n`, compute the number of new valid rows:

- An ABA row can be formed from:

- 3 compatible ABA rows
- 2 compatible ABC rows
- An ABC row can be formed from:

- 2 compatible ABA rows
- 2 compatible ABC rows
7. Apply the recurrence:

$$newSame = 3 \times same + 2 \times diff$$

$$newDiff = 2 \times same + 2 \times diff$$
8. Replace the old values with the new ones after every iteration.
9. Return:

$$(same + diff) \bmod (10^9 + 7)$$

### Why it works

Every valid row must belong to either the ABA category or the ABC category. The recurrence precisely counts how many compatible next rows can follow each category while preserving vertical adjacency constraints. Since every row transition is accounted for exactly once, the dynamic programming process counts every valid grid configuration without duplication or omission.

## Python Solution

```python
class Solution:
    def numOfWays(self, n: int) -> int:
        MOD = 10**9 + 7

        # For one row:
        # same = ABA patterns
        # diff = ABC patterns
        same = 6
        diff = 6

        for _ in range(1, n):
            new_same = (3 * same + 2 * diff) % MOD
            new_diff = (2 * same + 2 * diff) % MOD

            same = new_same
            diff = new_diff

        return (same + diff) % MOD
```

The implementation follows the mathematical recurrence directly.

We first initialize the number of valid single-row configurations. There are six ABA patterns and six ABC patterns.

The loop processes each additional row. At every iteration, we compute the number of new ABA and ABC configurations that can legally follow the previous row types.

Modulo arithmetic is applied during every update to prevent integer overflow and satisfy the problem requirement.

Finally, we return the sum of both categories because every valid grid must end in either an ABA row or an ABC row.

## Go Solution

```go
func numOfWays(n int) int {
    const MOD int = 1000000007

    // same = ABA patterns
    // diff = ABC patterns
    same := 6
    diff := 6

    for i := 1; i < n; i++ {
        newSame := (3*same + 2*diff) % MOD
        newDiff := (2*same + 2*diff) % MOD

        same = newSame
        diff = newDiff
    }

    return (same + diff) % MOD
}
```

The Go implementation mirrors the Python logic closely.

Go integers are large enough for intermediate computations here because we apply modulo reduction during every iteration. No special handling for overflow is required beyond standard modular arithmetic.

Unlike Python, Go does not support arbitrary precision integers automatically, so using modulo consistently is important.

## Edge Cases

### Single Row Grid

When `n = 1`, there are no vertical adjacency constraints because only one row exists. A buggy implementation might incorrectly attempt transitions or overcount configurations.

This implementation handles the case naturally because the loop starts at row `2`. The initial values already represent all valid one-row configurations.

### Very Large `n`

For `n = 5000`, brute force enumeration is impossible. Recursive solutions may also risk excessive overhead.

This implementation uses an iterative dynamic programming approach with constant memory, so it scales efficiently even for the largest allowed input.

### Modulo Overflow

The number of valid grids grows exponentially. Without modulo arithmetic, integer values would become extremely large.

The implementation applies modulo reduction during every iteration, ensuring values remain within safe bounds while satisfying the problem requirement.
