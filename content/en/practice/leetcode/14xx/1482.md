## Solution

## Problem Understanding

The problem gives us an array `bloomDay` where `bloomDay[i]` represents the day the `i-th` flower blooms. We are also given two integers:

- `m`, the number of bouquets we need to make
- `k`, the number of adjacent flowers required for one bouquet

A flower can only be used once, and the flowers used in a bouquet must be adjacent in the array.

The task is to determine the minimum number of days we must wait so that it becomes possible to create exactly `m` bouquets using the blooming flowers. If it is impossible, we return `-1`.

The important detail is that a flower becomes permanently available once its bloom day has passed. For example, if a flower blooms on day `5`, then it is usable on day `5` and every day afterward.

The constraints are large:

- `n` can be up to `100,000`
- `bloomDay[i]` can be as large as `1,000,000,000`

These limits immediately tell us that we cannot simulate every day one by one. A naive day-by-day approach would be far too slow.

Another important observation is that adjacency matters. Even if enough flowers have bloomed overall, they must appear in contiguous groups of size `k`.

There are several edge cases that matter:

- If `m * k > n`, then there are not enough flowers in total, so the answer is immediately `-1`
- If `k == 1`, every bloomed flower can form a bouquet independently
- If all flowers bloom on the same day, the answer is simply that day if enough flowers exist
- Large bloom days require an efficient search strategy rather than linear simulation

## Approaches

### Brute Force Approach

The brute force idea is straightforward. We could try every possible day starting from the minimum bloom day up to the maximum bloom day.

For each day:

1. Mark all flowers whose bloom day is less than or equal to the current day as available.
2. Scan the array and count how many bouquets can be formed using adjacent bloomed flowers.
3. Return the first day where at least `m` bouquets can be made.

This approach is correct because it directly simulates the problem definition. However, it is far too slow.

The range of days can be as large as `10^9`, which makes checking every day infeasible. Even if each scan takes `O(n)`, iterating through billions of days is impossible within time limits.

### Optimal Approach

The key insight is that the answer has a monotonic property.

If we can make `m` bouquets on day `d`, then we can also make `m` bouquets on every day after `d`.

Why? Because more flowers bloom over time, never fewer.

This monotonic behavior makes the problem ideal for binary search.

Instead of checking every day sequentially, we binary search the minimum valid day.

For any candidate day:

- We determine whether it is possible to form at least `m` bouquets.
- If it is possible, we try smaller days.
- If it is not possible, we try larger days.

The feasibility check itself is linear:

- Traverse the array
- Count consecutive bloomed flowers
- Every time we reach `k` consecutive flowers, we form one bouquet and reset the counter

This gives us an efficient solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × range) | O(1) | Checks every possible day individually |
| Optimal | O(n log range) | O(1) | Uses binary search on the answer |

## Algorithm Walkthrough

1. First, check whether it is even possible to create the required bouquets.

We need exactly `m * k` flowers in total. If the array contains fewer flowers than that, we immediately return `-1`.
2. Define the binary search boundaries.

The smallest possible answer is the minimum bloom day in the array.

The largest possible answer is the maximum bloom day.
3. Perform binary search over the day range.

For each midpoint day `mid`, we ask:

"Can we form at least `m` bouquets by day `mid`?"
4. To answer this feasibility question, scan the array from left to right.

- If `bloomDay[i] <= mid`, the flower has bloomed, so increase the consecutive flower count.
- Otherwise, reset the consecutive count because adjacency is broken.
5. Whenever the consecutive count reaches `k`, we form one bouquet.

- Increment the bouquet counter
- Reset consecutive count to zero because flowers cannot be reused
6. After scanning the array:

- If we formed at least `m` bouquets, the current day is valid.
- Otherwise, the current day is invalid.
7. Update the binary search range.

- If the day is valid, store it as a candidate answer and search the left half for a smaller valid day.
- If the day is invalid, search the right half.
8. Continue until the binary search finishes.
9. Return the smallest valid day found.

### Why it works

The correctness comes from the monotonic nature of the problem.

If a certain day allows us to form enough bouquets, every later day will also work because flowers never become unavailable. Binary search is therefore guaranteed to find the smallest valid day.

The feasibility check is also correct because scanning consecutive bloomed flowers exactly models the adjacency requirement. Resetting after forming a bouquet ensures flowers are never reused.

## Python Solution

```python
from typing import List

class Solution:
    def minDays(self, bloomDay: List[int], m: int, k: int) -> int:
        total_needed = m * k

        if total_needed > len(bloomDay):
            return -1

        def can_make(day: int) -> bool:
            bouquets = 0
            consecutive = 0

            for bloom in bloomDay:
                if bloom <= day:
                    consecutive += 1

                    if consecutive == k:
                        bouquets += 1
                        consecutive = 0
                else:
                    consecutive = 0

            return bouquets >= m

        left = min(bloomDay)
        right = max(bloomDay)

        answer = right

        while left <= right:
            mid = (left + right) // 2

            if can_make(mid):
                answer = mid
                right = mid - 1
            else:
                left = mid + 1

        return answer
```

The implementation begins with the impossibility check. If the total number of flowers required exceeds the number available, we immediately return `-1`.

The helper function `can_make(day)` determines whether enough bouquets can be formed by a given day. It scans the array once while maintaining a count of consecutive bloomed flowers.

When a flower has bloomed by the current day, we increase the consecutive counter. If the counter reaches `k`, we form one bouquet and reset the counter to avoid reusing flowers.

If we encounter a flower that has not yet bloomed, adjacency breaks, so we reset the consecutive counter to zero.

The binary search then operates over the possible day range. Whenever a valid day is found, we continue searching toward smaller days to find the minimum valid answer.

## Go Solution

```go
func minDays(bloomDay []int, m int, k int) int {
	totalNeeded := m * k

	if totalNeeded > len(bloomDay) {
		return -1
	}

	canMake := func(day int) bool {
		bouquets := 0
		consecutive := 0

		for _, bloom := range bloomDay {
			if bloom <= day {
				consecutive++

				if consecutive == k {
					bouquets++
					consecutive = 0
				}
			} else {
				consecutive = 0
			}
		}

		return bouquets >= m
	}

	left := bloomDay[0]
	right := bloomDay[0]

	for _, day := range bloomDay {
		if day < left {
			left = day
		}

		if day > right {
			right = day
		}
	}

	answer := right

	for left <= right {
		mid := left + (right-left)/2

		if canMake(mid) {
			answer = mid
			right = mid - 1
		} else {
			left = mid + 1
		}
	}

	return answer
}
```

The Go implementation follows the same logic as the Python solution. One difference is that Go does not provide built-in `min()` and `max()` operations for slices, so we manually compute the minimum and maximum bloom days.

The midpoint calculation uses:

```go
mid := left + (right-left)/2
```

This avoids potential integer overflow, which is a common best practice in Go and other statically typed languages.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log range) | Binary search over days, each feasibility check scans the array once |
| Space | O(1) | Only a few variables are used |

The binary search runs over the range between the minimum and maximum bloom days. Since bloom days can be up to `10^9`, the number of binary search iterations is approximately `log2(10^9)`, which is about 31.

Each iteration performs a linear scan of the array, giving overall complexity:

```
O(n × log(range))
```

The algorithm uses constant extra space because it only stores counters and pointers.

## Edge Cases

One important edge case occurs when there are not enough flowers overall. Even before considering adjacency or bloom timing, if `m * k > n`, the task is mathematically impossible. Many incorrect solutions forget this early termination and waste time performing unnecessary binary search operations. The implementation handles this immediately with a direct check at the start.

Another tricky edge case happens when adjacency breaks frequently. For example:

```
[1, 10, 1, 10, 1]
```

with `k = 2`.

Even though three flowers bloom early, none are adjacent. A buggy implementation might incorrectly count total bloomed flowers instead of consecutive groups. The solution correctly resets the consecutive counter whenever it encounters an unbloomed flower.

A third important edge case involves large bloom values such as `10^9`. A naive simulation that iterates through every day would time out immediately. The binary search approach avoids this by checking only logarithmically many candidate days.

Another subtle case is when flowers bloom in long consecutive runs. After forming one bouquet, the flowers used cannot be reused. Resetting the consecutive counter after reaching `k` ensures bouquets remain disjoint and prevents accidental reuse of flowers.
