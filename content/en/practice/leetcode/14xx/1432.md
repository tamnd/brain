## Solution

## Problem Understanding

The problem gives us an integer `num`, and we are allowed to perform a digit replacement operation twice, independently.

In each operation:

- We choose a digit `x`
- We choose another digit `y`
- Every occurrence of `x` in the decimal representation of the number is replaced with `y`

The two operations are completely independent. One produces a number `a`, and the other produces a number `b`. Our goal is to maximize:

$$a - b$$

There are two important restrictions:

- Neither resulting number may contain leading zeros
- Neither resulting number may become `0`

This means some replacements that appear attractive are actually invalid. For example, turning the leading digit into `0` is not allowed because it creates a leading zero.

The input size is very small:

$$1 \le num \le 10^8$$

Since the number contains at most 9 digits, even brute force approaches are feasible. However, the problem is really testing whether we can identify the greedy structure of the optimal transformation.

The key idea is that we want:

- `a` to be as large as possible
- `b` to be as small as possible

Because the operations are independent, we can optimize them separately.

Several edge cases are important:

- A single digit number like `9`
- Numbers where all digits are already `9`
- Numbers where minimizing requires careful handling of the leading digit
- Numbers containing many repeated digits
- Cases where replacing a digit with itself is the best option

For example, if the number is already `999`, the best way to maximize it is to leave it unchanged.

## Approaches

### Brute Force Approach

The brute force solution tries every possible transformation for `a` and every possible transformation for `b`.

A number contains digits `0` through `9`, so for each operation:

- Choose `x` from `0` to `9`
- Choose `y` from `0` to `9`

That gives 100 possible transformations for one operation.

Since `a` and `b` are independent, we can generate all valid transformed numbers for both operations and compute the maximum difference.

This works because the search space is tiny:

$$100 \times 100 = 10{,}000$$

Each transformation takes at most `O(d)` time where `d` is the number of digits.

Although this is acceptable for the constraints, it is not elegant and does unnecessary work. The problem has a strong greedy structure that allows us to directly construct the optimal maximum and minimum values.

### Greedy Insight

To maximize the difference:

$$a - b$$

we should independently:

- Make `a` as large as possible
- Make `b` as small as possible

For maximizing:

- The best replacement is always turning some digit into `9`
- We should replace the first digit from left to right that is not already `9`
- Replacing earlier digits gives larger gains because higher place values matter more

For minimizing:

There are two cases:

- If the first digit is not `1`, replace all occurrences of that leading digit with `1`
- Otherwise, keep the leading digit unchanged and find the first digit that is neither `0` nor `1`, then replace it with `0`

This strategy minimizes the number while avoiding leading zeros.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(100 × 100 × d) | O(d) | Tries every valid transformation pair |
| Optimal | O(d) | O(d) | Greedily constructs maximum and minimum numbers |

## Algorithm Walkthrough

### Optimal Greedy Algorithm

1. Convert the number into a string so digit replacement becomes easy.

String manipulation is much simpler than repeatedly extracting digits mathematically.
2. Construct the maximum possible value `a`.

Scan the digits from left to right and find the first digit that is not `9`.

Why the first one? Because changing a more significant digit produces a larger increase than changing a less significant digit.
3. Replace every occurrence of that digit with `9`.

Since the operation replaces all occurrences globally, we must replace every matching digit.
4. Construct the minimum possible value `b`.

There are two separate cases.
5. If the leading digit is not `1`, replace all occurrences of that leading digit with `1`.

This produces the smallest valid number because:

- The leading digit contributes the most to the value
- We cannot replace it with `0` due to the leading zero restriction
6. Otherwise, the leading digit is already `1`.

In this case, we preserve the leading digit and search from left to right for the first digit that is neither `0` nor `1`.
7. Replace every occurrence of that digit with `0`.

This minimizes the number while still keeping the number valid.
8. Convert both transformed strings back into integers and return their difference.

### Why it works

The algorithm relies on positional value greediness.

For maximizing, improving the leftmost non-`9` digit gives the largest possible increase because digits in higher positions contribute more to the final value.

For minimizing, reducing the leftmost possible digit gives the largest decrease. However, the leading digit cannot become `0`, so the best valid choice is `1`. If the leading digit is already `1`, the next best strategy is minimizing later digits to `0`.

Because the maximize and minimize operations are independent, optimizing them separately guarantees the largest possible difference.

## Python Solution

```python
class Solution:
    def maxDiff(self, num: int) -> int:
        num_str = str(num)

        # Build the maximum number
        max_str = num_str
        for ch in num_str:
            if ch != '9':
                max_str = num_str.replace(ch, '9')
                break

        # Build the minimum number
        min_str = num_str

        if num_str[0] != '1':
            min_str = num_str.replace(num_str[0], '1')
        else:
            for ch in num_str[1:]:
                if ch != '0' and ch != '1':
                    min_str = num_str.replace(ch, '0')
                    break

        return int(max_str) - int(min_str)
```

The implementation directly follows the greedy strategy.

First, the number is converted to a string so we can easily perform global digit replacements using Python's `replace()` method.

To build the maximum number:

- We scan from left to right
- The first digit that is not `9` is selected
- All occurrences of that digit become `9`

If all digits are already `9`, the number remains unchanged.

To build the minimum number:

- If the leading digit is not `1`, we replace all occurrences of it with `1`
- Otherwise, we search the remaining digits for the first digit that is neither `0` nor `1`
- That digit is replaced with `0`

Finally, both strings are converted back into integers and their difference is returned.

## Go Solution

```go
package main

import (
	"strconv"
	"strings"
)

func maxDiff(num int) int {
	numStr := strconv.Itoa(num)

	// Build maximum number
	maxStr := numStr
	for _, ch := range numStr {
		if ch != '9' {
			maxStr = strings.ReplaceAll(numStr, string(ch), "9")
			break
		}
	}

	// Build minimum number
	minStr := numStr

	if numStr[0] != '1' {
		minStr = strings.ReplaceAll(numStr, string(numStr[0]), "1")
	} else {
		for i := 1; i < len(numStr); i++ {
			if numStr[i] != '0' && numStr[i] != '1' {
				minStr = strings.ReplaceAll(numStr, string(numStr[i]), "0")
				break
			}
		}
	}

	maxVal, _ := strconv.Atoi(maxStr)
	minVal, _ := strconv.Atoi(minStr)

	return maxVal - minVal
}
```

The Go implementation mirrors the Python logic closely.

Go does not have a built in string replacement method directly on the string object, so we use `strings.ReplaceAll`.

Character handling is also slightly different:

- Python iterates directly over characters
- Go strings are byte slices, so indexing returns bytes

Since the input only contains decimal digits, byte indexing is perfectly safe here.

Integer conversion is handled using `strconv.Itoa` and `strconv.Atoi`.

## Edge Cases

### Single Digit Numbers

A single digit number can only become another single digit number. This case is easy to mishandle because replacing the digit with `0` is forbidden.

For example:

```
num = 9
```

The maximum remains `9`, while the minimum becomes `1`. The implementation correctly avoids invalid zero results by always minimizing the leading digit to `1` instead of `0`.

### Numbers Already Full of Nines

Consider:

```
num = 9999
```

There is no digit available to improve the maximum number. A naive implementation might accidentally perform unnecessary replacements.

The algorithm handles this naturally because it searches for the first non-`9` digit. If none exists, the original number is preserved.

### Leading Zero Risk

Consider:

```
num = 10023
```

A buggy implementation might attempt to minimize by replacing the leading `1` with `0`, producing an invalid number with leading zeros.

The algorithm explicitly prevents this:

- If the leading digit is not `1`, it becomes `1`
- Otherwise, only later digits may become `0`

This guarantees all generated numbers remain valid.
