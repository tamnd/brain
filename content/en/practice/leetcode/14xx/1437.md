## Solution

## Problem Understanding

The problem gives us a binary array `nums`, where each element is either `0` or `1`, and an integer `k`. We must determine whether every pair of `1`s in the array is separated by at least `k` positions containing other elements.

More specifically, if two `1`s appear at indices `i` and `j`, then the number of elements between them must be at least `k`. Since the number of elements between two indices is `j - i - 1`, the condition becomes:

$$j - i - 1 \ge k$$

If this condition holds for every pair of consecutive `1`s, we return `true`. Otherwise, we return `false`.

The input array can contain up to `10^5` elements, which means the solution must be efficient. A quadratic solution that compares every pair of positions would become too slow for the upper bound.

The constraints also tell us several important things:

- The array always contains at least one element.
- `nums[i]` is guaranteed to be either `0` or `1`.
- `k` can be `0`, which means no spacing requirement exists.
- The array may contain no `1`s or only one `1`, both of which should automatically satisfy the condition because there are no conflicting pairs.

Several edge cases are important:

- Arrays with no `1`s should return `true`.
- Arrays with exactly one `1` should also return `true`.
- When `k = 0`, every arrangement is valid.
- Consecutive `1`s with insufficient spacing must immediately return `false`.
- Large arrays require a linear-time approach.

## Approaches

### Brute Force Approach

A straightforward brute-force solution is to first collect all indices containing `1`. Then, for every pair of consecutive indices, compute the distance between them and verify whether the number of positions between them is at least `k`.

For example, if the indices are `[0, 4, 7]`, then:

- Between `0` and `4`, there are `4 - 0 - 1 = 3` elements.
- Between `4` and `7`, there are `7 - 4 - 1 = 2` elements.

If any computed gap is smaller than `k`, we return `false`.

This approach is correct because every pair of consecutive `1`s is explicitly checked. However, a less careful brute-force implementation might compare every pair of elements, leading to unnecessary quadratic work.

### Optimal Approach

The key observation is that we only care about the distance between consecutive `1`s. We do not need to store all indices or compare all pairs.

Instead, we can scan the array once while remembering the index of the most recent `1`.

Whenever we encounter another `1`, we compute the gap between the current index and the previous `1`. If the gap is smaller than required, we immediately return `false`.

This works because only adjacent `1`s can violate the spacing condition. If every consecutive pair satisfies the requirement, then all `1`s satisfy it.

This gives us a clean linear-time solution with constant extra space.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Compare every pair of positions |
| Optimal | O(n) | O(1) | Single pass while tracking previous `1` |

## Algorithm Walkthrough

1. Initialize a variable `previous_one_index` to `-1`.

This variable stores the index of the last encountered `1`. A value of `-1` means we have not seen any `1` yet.
2. Traverse the array from left to right using an index `i`.

We inspect each element exactly once.
3. If the current element is `0`, continue scanning.

Only positions containing `1` matter for the spacing requirement.
4. If the current element is `1` and this is the first `1` encountered, store its index.

Since there is no earlier `1`, no distance check is needed yet.
5. If the current element is `1` and we already saw a previous `1`, compute the number of elements between them.

The gap is:

$$i - previous\_one\_index - 1$$
6. If the gap is smaller than `k`, immediately return `false`.

This means two `1`s are too close together.
7. Otherwise, update `previous_one_index` to the current index.

This current `1` becomes the reference point for future checks.
8. If the loop finishes without finding any violation, return `true`.

### Why it works

The algorithm maintains the invariant that every pair of consecutive `1`s processed so far satisfies the required spacing condition. Since any violation must occur between consecutive `1`s, checking only adjacent occurrences is sufficient. If the algorithm finishes without detecting a violation, then all `1`s are at least `k` positions apart.

## Python Solution

```python
from typing import List

class Solution:
    def kLengthApart(self, nums: List[int], k: int) -> bool:
        previous_one_index = -1

        for current_index, value in enumerate(nums):
            if value == 1:
                if previous_one_index != -1:
                    gap = current_index - previous_one_index - 1

                    if gap < k:
                        return False

                previous_one_index = current_index

        return True
```

The implementation begins by initializing `previous_one_index` to `-1`, which indicates that no `1` has been encountered yet.

The loop iterates through the array using `enumerate`, giving both the current index and value. Whenever the value is `1`, the algorithm checks whether a previous `1` exists.

If a previous `1` was already seen, the code computes the number of elements between the two positions. If this gap is smaller than `k`, the function immediately returns `False`.

If the spacing is valid, the current index becomes the new `previous_one_index`, and the scan continues.

If the loop completes without detecting any invalid spacing, the function returns `True`.

## Go Solution

```go
func kLengthApart(nums []int, k int) bool {
	previousOneIndex := -1

	for currentIndex, value := range nums {
		if value == 1 {
			if previousOneIndex != -1 {
				gap := currentIndex - previousOneIndex - 1

				if gap < k {
					return false
				}
			}

			previousOneIndex = currentIndex
		}
	}

	return true
}
```

The Go implementation follows the same logic as the Python version. The `range` loop provides both the index and value for each element in the slice.

Go slices naturally handle empty or non-empty arrays without requiring special cases. Integer overflow is not a concern here because the indices are bounded by `10^5`.

The implementation uses constant extra space and performs a single linear scan through the input slice.

## Edge Cases

One important edge case occurs when the array contains no `1`s at all. In this situation, there are no pairs of `1`s that could violate the condition, so the correct result is `true`. A buggy implementation might incorrectly assume at least one `1` exists. The provided solution handles this naturally because `previous_one_index` remains `-1` throughout the traversal, and no invalid condition is ever triggered.

Another important edge case is when the array contains exactly one `1`. Since spacing constraints only apply between pairs of `1`s, a single `1` is always valid regardless of the value of `k`. The implementation correctly handles this because the first `1` only updates `previous_one_index` and never performs a gap comparison.

A third critical edge case occurs when `k = 0`. In this case, no spacing is required, meaning even adjacent `1`s are acceptable. Some implementations mistakenly require at least one zero between `1`s. The current solution correctly evaluates the condition because any computed gap will always satisfy `gap >= 0`.

A final edge case involves consecutive `1`s such as `[1,1]` when `k > 0`. Here, the computed gap becomes `0`, which should immediately fail the condition. The implementation correctly detects this and returns `false` as soon as the invalid pair is encountered.
