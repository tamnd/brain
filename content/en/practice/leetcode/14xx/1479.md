## Solution

## Problem Understanding

This problem asks us to generate a pivot-style sales report from two database tables, `Orders` and `Items`. The goal is to calculate the total number of units sold for each item category on every day of the week.

The `Orders` table stores transactional information. Each row represents a purchase of a specific item on a particular date, along with the quantity purchased. The `Items` table maps each `item_id` to its category. Since the report is grouped by category rather than by individual item, we must join these two tables together.

The final output should contain one row per category and seven columns corresponding to the days of the week: Monday through Sunday. Each column should contain the total quantity sold for that category on that weekday. If no sales occurred for a particular category on a particular day, the result must show `0` instead of `NULL`.

An important detail is that every category from the `Items` table must appear in the output, even if that category never appears in the `Orders` table. This means we cannot simply aggregate over `Orders`, because categories with no sales would disappear from the result. Instead, we need a `LEFT JOIN` from `Items` to `Orders`.

The output must also be ordered alphabetically by category.

The main challenge in this problem is transforming row-based sales data into a column-based weekly summary. This is commonly known as conditional aggregation or pivoting.

The constraints are not explicitly provided, but since this is a SQL database problem, we should assume the tables may contain many rows. A naive solution that repeatedly scans the data for each category and weekday combination would be inefficient. Instead, we should aggregate all required information in a single grouped query.

Several edge cases are important:

- Categories with no sales at all must still appear with all zeroes.
- Multiple orders may occur on the same weekday for the same category, so quantities must be summed correctly.
- Different items can belong to the same category, meaning aggregation must happen at the category level rather than the item level.
- Some weekdays may have no sales globally, requiring explicit zero handling.
- SQL weekday functions vary slightly across database systems, so we must use the correct interpretation for LeetCode's MySQL environment.

## Approaches

### Brute Force Approach

A brute-force solution would first determine all categories and then, for every category, separately compute sales totals for Monday, Tuesday, Wednesday, and so on.

Conceptually, this means:

1. Iterate through every category.
2. For each category, scan all orders.
3. Determine the weekday of each order.
4. Accumulate quantities into the correct weekday bucket.

In SQL terms, this could be implemented using multiple subqueries, one for each weekday and category combination. While this approach is logically correct because it explicitly computes every required total, it is inefficient because it repeatedly scans the same data.

If there are `C` categories and `N` orders, repeatedly scanning the orders table for each category results in roughly `O(C × N)` work.

### Optimal Approach

The key insight is that all required aggregates can be computed in a single grouped query using conditional aggregation.

Instead of scanning the orders table multiple times, we:

1. Join `Items` and `Orders`.
2. Group rows by category.
3. Use conditional `SUM` expressions to accumulate quantities for each weekday.

For example:

```
SUM(CASE WHEN DAYNAME(order_date) = 'Monday' THEN quantity ELSE 0 END)
```

This expression contributes the quantity only when the order occurred on Monday. Similar expressions are used for the other weekdays.

This approach is efficient because every row is processed only once, and all weekday totals are computed simultaneously during aggregation.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(C × N) | O(C) | Repeatedly scans orders for each category |
| Optimal | O(N) | O(C) | Single pass aggregation using conditional sums |

## Algorithm Walkthrough

1. Start from the `Items` table because every category must appear in the output, even if there are no matching orders. Using `Items` as the base table guarantees that categories without sales are preserved.
2. Perform a `LEFT JOIN` between `Items` and `Orders` on `item_id`. A left join ensures unmatched categories still remain in the result set with `NULL` order information.
3. Group the joined rows by `item_category`. This allows us to compute one final output row per category.
4. For each weekday, compute a conditional sum using a `CASE WHEN` expression. For example, the Monday column adds `quantity` only when `DAYNAME(order_date)` equals `'Monday'`. Otherwise, it contributes `0`.
5. Repeat the conditional aggregation for all seven weekdays. Each expression independently computes the total sales for that day.
6. Since categories without sales produce `NULL` quantities after the left join, the `ELSE 0` clause ensures the aggregation remains numeric and produces zero instead of null.
7. Order the final result alphabetically by category to satisfy the output requirement.

### Why it works

The correctness comes from the fact that every order belongs to exactly one weekday and exactly one category. The grouping step ensures all rows for the same category are combined together. The conditional aggregation ensures each order contributes only to the correct weekday column. Since all categories originate from the `Items` table and a left join is used, categories without matching orders are still preserved with zero totals.

## Python Solution

Although this is fundamentally a SQL problem, LeetCode database problems are traditionally answered with SQL rather than executable Python code. The following solution is the correct LeetCode submission.

```python
# Write your MySQL query statement below

SELECT
    i.item_category AS Category,
    SUM(CASE WHEN DAYNAME(o.order_date) = 'Monday' THEN o.quantity ELSE 0 END) AS Monday,
    SUM(CASE WHEN DAYNAME(o.order_date) = 'Tuesday' THEN o.quantity ELSE 0 END) AS Tuesday,
    SUM(CASE WHEN DAYNAME(o.order_date) = 'Wednesday' THEN o.quantity ELSE 0 END) AS Wednesday,
    SUM(CASE WHEN DAYNAME(o.order_date) = 'Thursday' THEN o.quantity ELSE 0 END) AS Thursday,
    SUM(CASE WHEN DAYNAME(o.order_date) = 'Friday' THEN o.quantity ELSE 0 END) AS Friday,
    SUM(CASE WHEN DAYNAME(o.order_date) = 'Saturday' THEN o.quantity ELSE 0 END) AS Saturday,
    SUM(CASE WHEN DAYNAME(o.order_date) = 'Sunday' THEN o.quantity ELSE 0 END) AS Sunday
FROM Items i
LEFT JOIN Orders o
    ON i.item_id = o.item_id
GROUP BY i.item_category
ORDER BY i.item_category;
```

The query begins by selecting from the `Items` table because every category must appear in the output. The `LEFT JOIN` ensures categories without sales are retained.

Each weekday column uses conditional aggregation. The `CASE WHEN` expression checks the weekday of each order and contributes the quantity only when the weekday matches the target column.

For example:

```python
SUM(CASE WHEN DAYNAME(o.order_date) = 'Monday'
         THEN o.quantity
         ELSE 0
    END)
```

This expression guarantees that only Monday sales contribute to the Monday total.

The `GROUP BY` clause combines all rows belonging to the same category into a single output row. Finally, the result is ordered alphabetically by category.

## Edge Cases

One important edge case occurs when a category has no associated orders. A naive inner join would completely remove that category from the output. This implementation avoids the issue by using a `LEFT JOIN` from `Items` to `Orders`, guaranteeing that every category remains visible. The conditional sums then evaluate to zero.

Another subtle case occurs when multiple items belong to the same category. If the query grouped by `item_id` instead of `item_category`, the result would incorrectly produce multiple rows for the same category. Grouping explicitly by category ensures all relevant item sales are merged together.

A third important edge case is when no sales occur on a particular weekday. Without the `ELSE 0` portion of the conditional aggregation, the result could become `NULL` instead of `0`. By explicitly contributing zero for nonmatching rows, the query always produces valid numeric totals.

A final edge case involves multiple orders occurring on the same weekday for the same category. The use of `SUM` guarantees that all matching quantities accumulate correctly rather than overwriting one another.
