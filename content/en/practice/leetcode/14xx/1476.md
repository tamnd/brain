## Solution

## Problem Understanding

The problem asks us to design a mutable 2D matrix data structure that supports two operations efficiently.

We are given an initial rectangular matrix of integers. The class must support updating every value inside a specified subrectangle and querying the value at any single cell.

The `updateSubrectangle` operation receives the coordinates of the top left corner and bottom right corner of a rectangle, along with a new value. Every element inside that rectangular region must be replaced with the new value.

The `getValue` operation simply returns the current value stored at a specific coordinate.

The important detail is that updates permanently modify the matrix. Future queries and future updates must operate on the latest state of the rectangle.

For example, suppose the matrix is:

```
1 2 1
4 3 4
3 2 1
1 1 1
```

If we update the entire matrix to `5`, every cell becomes `5`. Later, if we update only the last row to `10`, only that row changes while the rest remain `5`.

The constraints are relatively small:

- Maximum matrix size is `100 x 100`
- At most `500` total operations
- Values can be as large as `10^9`

These constraints are extremely important because they tell us we do not need sophisticated optimization techniques like segment trees, Fenwick trees, or lazy propagation. Even a direct simulation approach is fast enough.

A full matrix contains at most:

```
100 * 100 = 10,000 cells
```

If every one of the `500` operations updates the entire matrix, the total work is:

```
500 * 10,000 = 5,000,000 operations
```

This is completely acceptable in modern programming environments.

Several edge cases are important:

- Updating a single cell, where `row1 == row2` and `col1 == col2`
- Updating the entire matrix
- Multiple overlapping updates
- Queries immediately after updates
- Queries on cells that were never affected by later updates

The problem guarantees all coordinates are valid, so we do not need bounds checking.

## Approaches

### Brute Force Approach

The most straightforward solution is to directly store the matrix and modify every affected cell during each update.

When `updateSubrectangle` is called, we iterate through every row from `row1` to `row2` and every column from `col1` to `col2`, assigning `newValue` to each position.

When `getValue` is called, we simply return the value at the requested coordinates.

This approach works because the matrix always reflects the latest state after every update. Every query therefore returns the correct value immediately.

Even though this is technically brute force, the constraints are small enough that it easily passes.

### Better Observation

In many matrix update problems, repeatedly modifying large regions would be too expensive. However, this problem has unusually small limits.

The key observation is that the matrix can contain at most `10,000` cells, and there are only `500` operations total. That means even repeatedly updating entire subrectangles remains computationally cheap.

Because of this, the simplest direct-update solution is already optimal for the given constraints. Attempting more advanced structures would only add unnecessary complexity without meaningful performance improvement.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Update: O(rows × cols), Query: O(1) | O(rows × cols) | Directly modify every affected cell |
| Optimal | Update: O(rows × cols), Query: O(1) | O(rows × cols) | Same approach, constraints make it optimal |

## Algorithm Walkthrough

1. Store the input matrix inside the class.

We keep the rectangle as an instance variable so all updates permanently modify the same matrix state.
2. Process update operations by iterating through the target subrectangle.

For every row between `row1` and `row2`, and every column between `col1` and `col2`, replace the current value with `newValue`.

We use nested loops because every cell inside the rectangle must change.
3. Process query operations by directly accessing the matrix.

Since the matrix always stores the latest values, retrieving a cell is simply:

```
rectangle[row][col]
```
4. Return the requested value.

No extra computation is necessary because updates are applied eagerly.

### Why it works

The algorithm maintains a simple invariant:

> The stored matrix always exactly matches the current state of the rectangle after all updates.

Every update explicitly changes all affected cells, so the invariant remains true after each operation. Since queries directly read from this matrix, they always return the correct current value.

## Python Solution

```python
from typing import List

class SubrectangleQueries:

    def __init__(self, rectangle: List[List[int]]):
        self.rectangle = rectangle

    def updateSubrectangle(
        self,
        row1: int,
        col1: int,
        row2: int,
        col2: int,
        newValue: int
    ) -> None:

        for row in range(row1, row2 + 1):
            for col in range(col1, col2 + 1):
                self.rectangle[row][col] = newValue

    def getValue(self, row: int, col: int) -> int:
        return self.rectangle[row][col]

# Your SubrectangleQueries object will be instantiated and called as such:
# obj = SubrectangleQueries(rectangle)
# obj.updateSubrectangle(row1,col1,row2,col2,newValue)
# param_2 = obj.getValue(row,col)
```

The constructor stores the original matrix reference in `self.rectangle`.

The `updateSubrectangle` method uses two nested loops. The outer loop iterates through the rows of the target rectangle, and the inner loop iterates through the columns. Every visited cell is overwritten with `newValue`.

The `getValue` method simply returns the current value stored at the requested coordinates.

Because updates are applied immediately, queries require no additional processing.

## Go Solution

```go
type SubrectangleQueries struct {
	rectangle [][]int
}

func Constructor(rectangle [][]int) SubrectangleQueries {
	return SubrectangleQueries{
		rectangle: rectangle,
	}
}

func (this *SubrectangleQueries) UpdateSubrectangle(
	row1 int,
	col1 int,
	row2 int,
	col2 int,
	newValue int,
) {

	for row := row1; row <= row2; row++ {
		for col := col1; col <= col2; col++ {
			this.rectangle[row][col] = newValue
		}
	}
}

func (this *SubrectangleQueries) GetValue(row int, col int) int {
	return this.rectangle[row][col]
}

/**
 * Your SubrectangleQueries object will be instantiated and called as such:
 * obj := Constructor(rectangle);
 * obj.UpdateSubrectangle(row1,col1,row2,col2,newValue);
 * param_2 := obj.GetValue(row,col);
 */
```

The Go implementation mirrors the Python solution closely.

The matrix is stored as a slice of slices, `[][]int`. Since slices in Go are reference-like structures, updates directly modify the underlying matrix.

No special handling for overflow is needed because Go's `int` type easily accommodates the constraint limit of `10^9`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | Update: O((row2 - row1 + 1) × (col2 - col1 + 1)), Query: O(1) | Updates visit every cell in the subrectangle |
| Space | O(rows × cols) | Storage for the matrix |

The update operation touches every cell inside the specified rectangle exactly once. In the worst case, the entire matrix is updated, resulting in `O(rows × cols)` time.

The query operation is constant time because it directly indexes into the matrix.

The space complexity comes from storing the matrix itself.

## Edge Cases

One important edge case is updating a single cell. This happens when `row1 == row2` and `col1 == col2`. A buggy implementation might incorrectly skip the loops or mishandle inclusive boundaries. Our implementation uses `range(row1, row2 + 1)` and `range(col1, col2 + 1)`, so even single-cell ranges are processed correctly.

Another important case is updating the entire matrix. This is the largest possible update region and stresses the nested loop logic. Since the algorithm explicitly visits every cell in the target range, the entire matrix is correctly overwritten.

Overlapping updates are also critical. A later update must overwrite values from earlier updates only within the new target region. Because the matrix is modified directly during every update, the most recent update naturally takes precedence.

A final subtle case is querying cells outside recently updated regions. Some incorrect implementations accidentally modify unrelated cells due to indexing mistakes. Our nested loops only iterate between the specified row and column boundaries, ensuring untouched cells preserve their previous values correctly.
