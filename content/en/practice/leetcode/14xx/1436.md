## Solution

## Problem Understanding

The problem gives us a list of directed paths between cities. Each entry in `paths` contains two strings, where `paths[i] = [cityA, cityB]` means there is a one way path from `cityA` to `cityB`.

The graph formed by these paths is guaranteed to be a straight line without cycles. This guarantee is extremely important because it means there is exactly one city that has no outgoing path. That city is the final destination.

In simpler terms, every city except one points to another city. The destination city is the only city that never appears as a starting city.

For example, if we have:

```
London -> New York
New York -> Lima
Lima -> Sao Paulo
```

then `"Sao Paulo"` is the destination because it never appears on the left side of any path.

The input size is small, with at most 100 paths, so efficiency is not a major concern. However, we should still aim for a clean and optimal solution.

The constraints also guarantee:

- The graph forms a valid line
- There are no loops
- There is exactly one destination city
- Every path has exactly two cities

These guarantees eliminate many complicated graph cases such as cycles, disconnected components, or multiple valid answers.

One important edge case is when there is only one path, such as `["A", "Z"]`. In that case, `"Z"` is immediately the destination city.

Another important consideration is that city names may contain spaces and mixed capitalization, so we must treat them as exact strings.

## Approaches

### Brute Force Approach

The brute force method checks every city to determine whether it has an outgoing edge.

The idea is straightforward. For every destination city candidate, scan through all paths again and verify whether that city ever appears as a starting city. If it never appears as a source, then it must be the destination city.

This works because the destination city is uniquely defined as the city with zero outgoing paths.

However, this approach repeatedly scans the list of paths, which leads to unnecessary repeated work.

### Optimal Approach

The key observation is that every source city appears as the first element of some path, while the destination city never appears as a source.

Therefore, we can store all source cities in a hash set. Then we iterate through the destination cities and return the one that is not present in the set.

A hash set is ideal because it provides average O(1) lookup time.

This transforms the problem into:

- Build a set of all starting cities
- Find the city that never starts a path

Since the graph guarantee ensures exactly one valid answer, the first such city we find is the correct destination.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | For each city, scan all paths again |
| Optimal | O(n) | O(n) | Use a hash set for constant time lookups |

## Algorithm Walkthrough

1. Create an empty hash set called `starting_cities`.

This set will store every city that appears as the source of a path. A hash set is chosen because membership checks are extremely fast.
2. Iterate through every path in `paths`.

For each path `[source, destination]`, insert `source` into the hash set.
3. Iterate through the paths again.

For each path `[source, destination]`, check whether `destination` exists inside `starting_cities`.
4. If the destination city is not in the set, return it immediately.

A city that never appears as a source has no outgoing path, which matches the definition of the destination city.
5. The problem guarantees exactly one valid answer, so the algorithm will always return a city.

### Why it works

The algorithm relies on a simple invariant:

Every non destination city appears as a source at least once.

Because the graph forms a line, all intermediate cities must have outgoing edges. The only city without an outgoing edge is the final destination. Therefore, the destination city is precisely the city absent from the set of starting cities.

## Python Solution

```python
from typing import List

class Solution:
    def destCity(self, paths: List[List[str]]) -> str:
        starting_cities = set()

        # Store all cities that have outgoing paths
        for source, destination in paths:
            starting_cities.add(source)

        # Find the city with no outgoing path
        for source, destination in paths:
            if destination not in starting_cities:
                return destination

        return ""
```

The implementation closely follows the algorithm described earlier.

The first loop builds the `starting_cities` set. Every source city is inserted so we can later perform fast membership checks.

The second loop examines each destination city. If a destination city does not exist in `starting_cities`, then it never appears as a source. That means it has no outgoing edges and must be the final destination.

The final `return ""` is technically unnecessary because the problem guarantees a valid answer, but it satisfies Python's function return requirements.

## Go Solution

```go
func destCity(paths [][]string) string {
    startingCities := make(map[string]bool)

    // Store all source cities
    for _, path := range paths {
        source := path[0]
        startingCities[source] = true
    }

    // Find the destination city
    for _, path := range paths {
        destination := path[1]

        if !startingCities[destination] {
            return destination
        }
    }

    return ""
}
```

The Go implementation uses a `map[string]bool` to simulate a hash set because Go does not have a built in set type.

The logic is identical to the Python solution:

- Store all source cities
- Return the destination city not present in the map

There are no concerns about integer overflow because the problem only involves strings. The solution also naturally handles empty versus nil slices because the constraints guarantee at least one path.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Two linear passes through the paths array |
| Space | O(n) | The hash set stores up to n source cities |

The algorithm performs two separate traversals of the input array. Each traversal takes linear time. Hash set insertion and lookup operations are O(1) on average, so the total runtime remains O(n).

The extra space comes from storing all source cities in the hash set. In the worst case, every path contributes a unique source city.

## Edge Cases

One important edge case is when the input contains only a single path. In this situation, the destination city is simply the second city in that path. Some implementations accidentally overcomplicate this scenario, but the hash set approach handles it naturally because the destination city never appears as a source.

Another important edge case involves paths appearing in arbitrary order. The paths are not guaranteed to be arranged sequentially from start to finish. For example, the input may contain `["D","B"]` before `["B","C"]`. A naive traversal based on order could fail, but the hash set approach is independent of ordering because it only checks source membership.

A third edge case is city names containing spaces or mixed capitalization. Since city names are plain strings, `"New York"` and `"new york"` must be treated as different values. The implementation uses exact string matching, which correctly preserves case sensitivity and spacing.

A final subtle edge case is ensuring there is exactly one destination city. The problem guarantees this property, so the algorithm can safely return the first destination city not found in the source set. Without this guarantee, additional validation logic would be required.
