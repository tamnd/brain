**Difficulty:** 🟡 Medium  
**Topics:** Array, Hash Table, String, Sorting, Ordered Set

## Solution

## Problem Understanding

This problem asks us to build a formatted summary table from a list of restaurant orders. Each order contains three pieces of information:

- The customer name
- The table number
- The food item ordered

The customer name is irrelevant for the final result. What matters is which table ordered which food items, and how many times each item was ordered.

The output must be a two dimensional table represented as a list of lists of strings. The structure of the output is very important:

- The first row is the header row.
- The first column is always `"Table"`.
- Every remaining column corresponds to a food item.
- Food item columns must appear in alphabetical order.
- Every subsequent row represents one table.
- Table rows must appear in increasing numerical order.
- Each cell contains the count of how many times that table ordered that food item.

For example, if table 3 ordered `"Ceviche"` twice and `"Water"` once, then the row for table 3 should contain those counts in the appropriate columns.

The constraints are important:

- There can be up to `5 * 10^4` orders.
- Table numbers range from `1` to `500`.
- Food names can contain spaces and mixed casing.

These constraints immediately suggest that we need an efficient counting solution. A quadratic solution that repeatedly scans all orders for every table and every food item would become unnecessarily expensive.

There are several edge cases worth noticing:

- Multiple customers at the same table may order the same food item.
- Different tables may order completely different sets of foods.
- Food names contain spaces, so we must treat them as full strings.
- Table numbers are provided as strings, but sorting must be numerical, not lexicographical. For example, `"10"` should come after `"5"`, not before `"2"` as string sorting would produce.
- A table may order only one food item repeatedly.
- There may be only one order in the entire input.

## Approaches

### Brute Force Approach

A straightforward approach would be:

1. Extract all unique food items.
2. Extract all unique tables.
3. Sort foods alphabetically.
4. Sort tables numerically.
5. For every table and every food item, scan the entire orders array and count matches.

This approach works because it directly computes every cell in the output table. However, it is inefficient because for each `(table, food)` pair, we repeatedly iterate through all orders.

If:

- `T` is the number of tables
- `F` is the number of food items
- `N` is the number of orders

Then the counting phase costs `O(T * F * N)`.

With up to `50,000` orders, repeated rescanning becomes wasteful.

### Optimal Approach

The key insight is that we should count orders while processing the input only once.

Instead of repeatedly scanning all orders, we can maintain:

- A set of all food items
- A nested hash map that stores counts for each table and food combination

The structure looks like this:

```
table_orders[table][food] = count
```

As we process each order:

- Add the food item to the food set
- Increment the corresponding table-food count

After processing all orders:

1. Sort food items alphabetically
2. Sort tables numerically
3. Build the final output table directly from the stored counts

This avoids repeated rescanning and makes the solution much more efficient.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(T × F × N) | O(T + F) | Repeatedly scans all orders for every table-food pair |
| Optimal | O(N + F log F + T log T + T × F) | O(T × F) | Counts everything in one pass using hash maps |

## Algorithm Walkthrough

1. Create a set to store all unique food items.

We need all food names because they become the columns of the final table. A set allows fast insertion and automatically removes duplicates.
2. Create a nested hash map for counts.

The structure is:

```
table_orders[table][food] = count
```

This lets us quickly increment the count for any `(table, food)` pair.
3. Process every order once.

For each order:

- Ignore the customer name
- Extract the table number
- Extract the food item
- Add the food item to the food set
- Increment the corresponding count in the nested hash map
4. Sort all food items alphabetically.

The problem explicitly requires food columns to appear in alphabetical order.
5. Sort all table numbers numerically.

Table numbers are strings in the input, but they must be ordered numerically. Therefore, we convert them to integers during sorting.
6. Build the header row.

The first row starts with `"Table"` followed by the sorted food items.
7. Build each table row.

For every sorted table:

- Start the row with the table number
- For every food item:

- Retrieve the count from the hash map
- Use `"0"` if the food was never ordered at that table
- Convert counts to strings because the output format requires strings
8. Append all rows to the result and return it.

### Why it works

The algorithm works because every order contributes exactly once to the correct `(table, food)` count. The nested hash map maintains the invariant that:

```
table_orders[t][f]
```

always equals the total number of times table `t` ordered food `f`.

After processing all orders, the data structure contains complete frequency information. Sorting food names and table numbers ensures the final output satisfies the required ordering rules.

## Python Solution

```python
from collections import defaultdict
from typing import List

class Solution:
    def displayTable(self, orders: List[List[str]]) -> List[List[str]]:
        food_items = set()
        table_orders = defaultdict(lambda: defaultdict(int))

        for _, table, food in orders:
            food_items.add(food)
            table_orders[table][food] += 1

        sorted_foods = sorted(food_items)
        sorted_tables = sorted(table_orders.keys(), key=int)

        result = []

        header = ["Table"] + sorted_foods
        result.append(header)

        for table in sorted_tables:
            row = [table]

            for food in sorted_foods:
                row.append(str(table_orders[table].get(food, 0)))

            result.append(row)

        return result
```

The implementation begins by creating two core data structures.

The `food_items` set stores all distinct food names. This is necessary because the final table must contain one column for every unique food item.

The `table_orders` nested dictionary stores counts for each `(table, food)` pair. The outer dictionary maps table numbers to another dictionary, and the inner dictionary maps food names to counts.

During the single pass through `orders`, each food item is added to the set, and its count for the corresponding table is incremented.

After processing:

- `sorted_foods` contains alphabetically ordered food names.
- `sorted_tables` contains numerically ordered table numbers.

The result table is then constructed row by row.

The header row starts with `"Table"` followed by the sorted food names.

For every table, we build a row containing:

- The table number
- The count for each food item

The `.get(food, 0)` call ensures that missing foods default to zero.

Finally, all rows are returned.

## Go Solution

```go
package main

import (
	"sort"
	"strconv"
)

func displayTable(orders [][]string) [][]string {
	foodSet := make(map[string]bool)
	tableOrders := make(map[string]map[string]int)

	for _, order := range orders {
		table := order[1]
		food := order[2]

		foodSet[food] = true

		if _, exists := tableOrders[table]; !exists {
			tableOrders[table] = make(map[string]int)
		}

		tableOrders[table][food]++
	}

	var foods []string
	for food := range foodSet {
		foods = append(foods, food)
	}

	sort.Strings(foods)

	var tables []string
	for table := range tableOrders {
		tables = append(tables, table)
	}

	sort.Slice(tables, func(i, j int) bool {
		a, _ := strconv.Atoi(tables[i])
		b, _ := strconv.Atoi(tables[j])
		return a < b
	})

	result := [][]string{}

	header := []string{"Table"}
	header = append(header, foods...)
	result = append(result, header)

	for _, table := range tables {
		row := []string{table}

		for _, food := range foods {
			count := tableOrders[table][food]
			row = append(row, strconv.Itoa(count))
		}

		result = append(result, row)
	}

	return result
}
```

The Go implementation follows the same logic as the Python solution, but there are several language specific details.

Go does not have a built in `defaultdict`, so we manually initialize inner maps when needed.

Food items are collected using a `map[string]bool` to simulate a set.

Table sorting requires converting string table numbers into integers using `strconv.Atoi`.

When accessing nonexistent map keys in Go, the zero value is returned automatically. Therefore:

```go
tableOrders[table][food]
```

returns `0` if the food does not exist for that table.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(N + F log F + T log T + T × F) | Processing orders is linear, sorting foods and tables costs logarithmic sorting time, building output requires visiting every table-food pair |
| Space | O(T × F) | The nested hash map may store counts for every table-food combination |

The dominant work comes from two phases.

First, we process all `N` orders exactly once.

Second, while constructing the output, we generate every cell in the final display table. Since there are `T` tables and `F` food items, this phase costs `O(T × F)`.

The sorting operations are comparatively smaller unless the number of distinct foods or tables becomes large.

## Edge Cases

### Tables Must Be Sorted Numerically

One of the easiest mistakes is sorting table numbers as strings instead of integers.

For example:

```
["1", "10", "2"]
```

would appear in incorrect order if lexicographical sorting is used.

The implementation avoids this by sorting with:

```
sorted(table_orders.keys(), key=int)
```

This guarantees correct numeric ordering.

### Missing Food Items for a Table

Most tables will not order every food item. A naive implementation might raise errors when trying to access nonexistent entries.

For example:

```
table_orders["3"]["Water"]
```

may not exist.

The implementation safely handles this using:

```
table_orders[table].get(food, 0)
```

which correctly inserts `"0"` into the display table.

### Multiple Orders of the Same Food

Several customers at the same table may order the same food item repeatedly. A buggy implementation might overwrite counts instead of accumulating them.

For example:

```
[
    ["A","1","Pizza"],
    ["B","1","Pizza"]
]
```

should produce a count of `2`.

The implementation correctly increments counts using:

```
table_orders[table][food] += 1
```

which accumulates all occurrences.

### Food Names With Spaces

Food names such as `"Beef Burrito"` or `"Fried Chicken"` contain spaces. Any implementation attempting to tokenize or split food names would fail.

The solution treats food names as complete strings throughout processing, sorting, and lookup, so spaces cause no issues.
