## Solution

## Problem Understanding

The problem is asking us to maximize the score obtained by taking exactly `k` cards from a row of cards represented by the array `cardPoints`. Each step allows you to pick a card from **either the start or the end** of the array. The goal is to maximize the sum of the points from the chosen cards.

The input consists of the array `cardPoints` of length `n` (up to 105), where each element represents the point value of a card, and an integer `k` representing the number of cards to take. The output is a single integer representing the maximum score achievable.

Key points about the constraints:

- `1 <= cardPoints.length <= 10^5` indicates we need an efficient solution; brute-force approaches iterating over all combinations of start/end choices will be too slow.
- `1 <= k <= cardPoints.length` ensures we always take at least one card and never more than available.
- Each `cardPoints[i]` is positive, which guarantees that adding more cards never decreases the sum.

Important edge cases to consider upfront include:

- `k` equals the length of the array, where the solution is simply the sum of all elements.
- `k` is 1, where only the maximum of the first or last element matters.
- Arrays where all elements are equal, to confirm the solution correctly handles ties.
- Arrays where optimal picks require a mix of start and end elements rather than just one side.

## Approaches

The brute-force approach involves trying all combinations of picking `i` cards from the start and `k-i` cards from the end, iterating `i` from `0` to `k`. For each combination, we sum the selected elements. This approach is correct because it exhaustively considers all possibilities. However, it is too slow, requiring `O(k^2)` time if summing each combination naively, which is infeasible for `k` up to 10^5.

The key observation for an optimal solution is that taking `k` cards from the start or end is equivalent to leaving a **contiguous subarray of length `n-k` unchosen** in the middle. Instead of maximizing the sum of chosen cards, we can equivalently **minimize the sum of the `n-k` contiguous cards that are left**. This insight allows us to use a sliding window approach to find the minimal sum subarray of length `n-k` in `O(n)` time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(k^2) | O(1) | Try all combinations of taking i from start and k-i from end |
| Optimal (Sliding Window) | O(n) | O(1) | Transform problem to minimizing unchosen subarray sum and use sliding window |

## Algorithm Walkthrough

1. Compute the total sum of all elements in `cardPoints`. This represents the maximum possible score if you could take all cards.
2. Determine the length of the subarray to leave unchosen, which is `n - k`.
3. Use a sliding window of size `n - k` to find the subarray with the minimum sum. Start with the first `n - k` elements as the initial window.
4. Slide the window one element at a time from left to right, updating the sum by adding the new element entering the window and subtracting the element leaving the window.
5. Keep track of the minimum window sum encountered during the sliding process.
6. Subtract the minimum window sum from the total sum of all elements. This gives the maximum points obtainable by taking `k` cards from the ends.

**Why it works:** The algorithm works because taking `k` cards from the ends is equivalent to leaving a contiguous block of `n - k` cards in the middle unchosen. By minimizing the sum of this unchosen block, we maximize the sum of the chosen cards. Sliding window guarantees we efficiently check all possible blocks of length `n-k`.

## Python Solution

```python
from typing import List

class Solution:
    def maxScore(self, cardPoints: List[int], k: int) -> int:
        n = len(cardPoints)
        total_sum = sum(cardPoints)
        
        if k == n:
            return total_sum
        
        window_size = n - k
        current_window_sum = sum(cardPoints[:window_size])
        min_window_sum = current_window_sum
        
        for i in range(window_size, n):
            current_window_sum += cardPoints[i] - cardPoints[i - window_size]
            min_window_sum = min(min_window_sum, current_window_sum)
        
        return total_sum - min_window_sum
```

The Python implementation first calculates the total sum. If `k` equals `n`, we return the total sum directly. Otherwise, we initialize the sliding window with the sum of the first `n-k` elements. We then slide the window through the array, updating the sum and tracking the minimal window sum. Finally, subtracting the minimal window sum from the total sum gives the maximum obtainable score.

## Go Solution

```go
func maxScore(cardPoints []int, k int) int {
    n := len(cardPoints)
    totalSum := 0
    for _, val := range cardPoints {
        totalSum += val
    }
    
    if k == n {
        return totalSum
    }
    
    windowSize := n - k
    currentWindowSum := 0
    for i := 0; i < windowSize; i++ {
        currentWindowSum += cardPoints[i]
    }
    
    minWindowSum := currentWindowSum
    for i := windowSize; i < n; i++ {
        currentWindowSum += cardPoints[i] - cardPoints[i - windowSize]
        if currentWindowSum < minWindowSum {
            minWindowSum = currentWindowSum
        }
    }
    
    return totalSum - minWindowSum
}
```

The Go solution mirrors the Python logic. Go handles integer arithmetic naturally, slices replace array indexing, and the sliding window calculation is done explicitly. We check the `k == n` edge case before the main loop.

## Worked Examples

- Total sum = 22
- Window size = 7 - 3 = 4
- Initial window sum = 1+2+3+4 = 10
- Slide window:

- Next sum = 2+3+4+5 = 14 → min = 10
- Next sum = 3+4+5+6 = 18 → min = 10
- Next sum = 4+5+6+1 = 16 → min = 10
- Max score = 22 - 10 = 12

**Example 2: `cardPoints = [2,2,2], k = 2`**

- Window size = 3 - 2 = 1
- Initial window sum = 2 → min = 2
- Slide:

- Next sum = 2 → min = 2
- Next sum = 2 → min = 2
- Max score = 6 - 2 = 4

- Max score = 55 - 0 = 55
