## Solution

## Problem Understanding

The problem presents a rooted tree with `n` nodes numbered from `0` to `n - 1`. The tree is described using a `parent` array, where `parent[i]` indicates the parent of node `i`. The root node is always node `0`, and its parent is represented as `-1`. You are asked to implement a data structure, `TreeAncestor`, which allows efficient queries for the `kth` ancestor of any node. The `kth` ancestor of a node is the node located `k` steps up along the path from that node toward the root. If the `kth` ancestor does not exist, the method should return `-1`.

The input constraints are significant: `1 <= k <= n <= 5 * 10^4` and there can be up to `5 * 10^4` queries. This means that a naive approach that traverses the parent chain for each query could take `O(n * q)` time in the worst case, which is too slow for the given limits. The solution must handle large trees and numerous queries efficiently. Key edge cases include querying for ancestors beyond the root and querying for a node itself.

## Approaches

The brute-force approach simply iterates up the parent chain for each query until either `k` steps are taken or the root is reached. This method is simple and correct but has a worst-case time complexity of `O(n)` per query, which becomes prohibitive when `n` and the number of queries are large.

The optimal solution uses **binary lifting**, a technique that precomputes a table where `up[node][j]` stores the `2^j`-th ancestor of `node`. Using this table, the `kth` ancestor can be found by decomposing `k` into powers of two and jumping up in logarithmic steps. This reduces query time to `O(log n)` per query, while the preprocessing step takes `O(n log n)` time and space.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) per query | O(1) | Traverses parent chain directly for each query |
| Optimal (Binary Lifting) | O(n log n) preprocessing, O(log n) per query | O(n log n) | Uses precomputed ancestor table to jump powers of two |

## Algorithm Walkthrough

1. **Initialization**: Create a 2D array `up` where `up[node][j]` stores the `2^j`-th ancestor of `node`. Initialize `up[node][0] = parent[node]`.
2. **Precompute Ancestors**: For each node and each `j > 0`, set `up[node][j] = up[up[node][j - 1]][j - 1]` if `up[node][j - 1] != -1`. This computes higher powers of ancestors recursively.
3. **Querying `kth` Ancestor**: To find the `kth` ancestor of a node, decompose `k` into binary. For each set bit in the binary representation of `k`, jump to the corresponding `2^j`-th ancestor using the precomputed table.
4. **Termination Condition**: If at any step the ancestor becomes `-1`, immediately return `-1` as no further ancestor exists.

**Why it works**: The binary lifting table ensures that any `k`-step ancestor query can be answered using a combination of `2^j` jumps. By precomputing all powers of two for each node, we guarantee correctness for all valid `k` values, and the logarithmic decomposition ensures efficiency.

## Python Solution

```python
from typing import List

class TreeAncestor:

    def __init__(self, n: int, parent: List[int]):
        self.LOG = 16  # Since n <= 5*10^4, log2(5*10^4) < 16
        self.up = [[-1] * self.LOG for _ in range(n)]
        
        for node in range(n):
            self.up[node][0] = parent[node]
        
        for j in range(1, self.LOG):
            for node in range(n):
                prev = self.up[node][j - 1]
                self.up[node][j] = self.up[prev][j - 1] if prev != -1 else -1

    def getKthAncestor(self, node: int, k: int) -> int:
        for j in range(self.LOG):
            if k & (1 << j):
                node = self.up[node][j]
                if node == -1:
                    return -1
        return node
```

The Python implementation initializes a binary lifting table with enough columns to cover the maximum possible `k`. Each row corresponds to a node, and each column stores the `2^j`-th ancestor. Queries iterate over the bits of `k` and perform jumps using the table, returning `-1` immediately if an ancestor does not exist.

## Go Solution

```go
type TreeAncestor struct {
	up [][]int
	LOG int
}

func Constructor(n int, parent []int) TreeAncestor {
	LOG := 16
	up := make([][]int, n)
	for i := 0; i < n; i++ {
		up[i] = make([]int, LOG)
		up[i][0] = parent[i]
	}
	for j := 1; j < LOG; j++ {
		for i := 0; i < n; i++ {
			prev := up[i][j-1]
			if prev != -1 {
				up[i][j] = up[prev][j-1]
			} else {
				up[i][j] = -1
			}
		}
	}
	return TreeAncestor{up: up, LOG: LOG}
}

func (this *TreeAncestor) GetKthAncestor(node int, k int) int {
	for j := 0; j < this.LOG; j++ {
		if k&(1<<j) != 0 {
			node = this.up[node][j]
			if node == -1 {
				return -1
			}
		}
	}
	return node
}
```

The Go implementation mirrors the Python approach, using slices for the binary lifting table and bitwise operations for querying. Slice initialization and bounds handling ensure safety, and the logic is identical in essence.

## Worked Examples

**Example 1**: `TreeAncestor(7, [-1,0,0,1,1,2,2])`

Query `getKthAncestor(3, 1)`:

- Binary of `k = 1` is `1` → jump 2^0 = 1 step.
- Node 3 → `up[3][0] = 1`.
- Answer is `1`.

Query `getKthAncestor(5, 2)`:

- Binary of `k = 2` is `10` → jump 2^1 = 2 steps.
- Node 5 → `up[5][1] = up[up[5][0]][0] = up[2][0] = 0`.
- Answer is `0`.

Query `getKthAncestor(6, 3)`:

- Binary of `k = 3` is `11`.
- Jump 2^0 = 1: Node 6 → `up[6][0] = 2`.
- Jump 2^1 = 2: Node 2 → `up[2][1] = up[0][0] = -1`.
- Answer is `-1`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) preprocessing, O(log n) per query | Precomputing table for n nodes with log n levels, query uses log n jumps |
| Space | O(n log n) | Binary lifting table stores n nodes × log n ancestors |

The preprocessing ensures that all possible ancestor queries can be answered efficiently, and the space used is proportional to the number of nodes times the logarithm of the tree size.
