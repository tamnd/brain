## Solution

## Problem Understanding

The problem gives us a database table named `Activities` with two columns:

| Column | Meaning |
| --- | --- |
| `sell_date` | The date a product was sold |
| `product` | The name of the product sold |

The table does not contain a primary key, which means duplicate rows may appear. For example, the same product could appear multiple times on the same date.

The task is to generate a summary for each distinct `sell_date`. For every date, we must compute two things:

1. The number of unique products sold on that date.
2. A comma-separated string containing the unique product names sorted lexicographically.

The final result must be ordered by `sell_date`.

The important detail is that duplicates must not affect the result. If `"Mask"` appears twice on the same day, it should only be counted once and should only appear once in the product list.

The expected output contains three columns:

| Column | Meaning |
| --- | --- |
| `sell_date` | The grouped date |
| `num_sold` | Number of distinct products sold that day |
| `products` | Sorted comma-separated list of unique product names |

The database guarantees that every row contains a valid date and product name. Since duplicates are allowed, any correct solution must explicitly remove duplicates before counting and concatenating products.

A naive implementation can easily make mistakes in two places. First, it may count duplicate products multiple times. Second, it may concatenate product names without sorting them lexicographically. Both requirements are essential for correctness.

## Approaches

### Brute Force Approach

A brute-force solution would process each distinct date independently. For every date, we could scan the entire table again to collect matching products. Then we could manually remove duplicates, sort the products, count them, and build the comma-separated string.

This approach works because every date is handled separately, and every row is examined to determine whether it belongs to that date.

However, this is inefficient because the table is repeatedly scanned for every distinct date. If there are `n` rows and `d` distinct dates, the complexity can grow close to `O(n * d)`. In the worst case, when every row has a unique date, this becomes `O(n²)`.

### Optimal Approach

The key insight is that SQL databases already provide efficient grouping and aggregation operations.

We can group rows by `sell_date` using `GROUP BY`. Then:

- `COUNT(DISTINCT product)` gives the number of unique products sold on that date.
- `GROUP_CONCAT(DISTINCT product ORDER BY product SEPARATOR ',')` builds the required sorted comma-separated string.

This approach avoids repeatedly scanning the table for each date because the database engine performs grouping internally in a single aggregation pass.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeatedly scans the table for each date |
| Optimal | O(n log n) | O(n) | Uses SQL aggregation and sorting efficiently |

## Algorithm Walkthrough

1. Group all rows by `sell_date`.

We want one output row per date, so grouping is the natural first step. Every aggregation operation will then operate only on rows belonging to the same date.
2. Count distinct products for each group.

Since duplicate rows may exist, we cannot use a normal `COUNT(product)`. Instead, we use `COUNT(DISTINCT product)` so that repeated product names only contribute once.
3. Generate the sorted product list.

We use `GROUP_CONCAT` to combine product names into a single string. The `DISTINCT` keyword removes duplicates, and `ORDER BY product` ensures lexicographical ordering.
4. Separate product names with commas.

The problem requires a comma-separated string, so we specify `SEPARATOR ','`.
5. Order the final result by `sell_date`.

The output specification explicitly requires dates to appear in chronological order.

### Why it works

The algorithm works because grouping partitions the table into independent date-based subsets. Within each subset, `DISTINCT` guarantees uniqueness, `COUNT` correctly measures the number of unique products, and `GROUP_CONCAT` constructs the required ordered string representation. Since every row belongs to exactly one date group, the final result is complete and correct.

## Python Solution

```python
# Write your MySQL query statement below

SELECT
    sell_date,
    COUNT(DISTINCT product) AS num_sold,
    GROUP_CONCAT(
        DISTINCT product
        ORDER BY product
        SEPARATOR ','
    ) AS products
FROM Activities
GROUP BY sell_date
ORDER BY sell_date;
```

This solution uses SQL aggregation functions directly.

The `GROUP BY sell_date` clause creates one group for each distinct date. Inside each group, `COUNT(DISTINCT product)` computes the number of unique products sold on that day.

The `GROUP_CONCAT` function combines product names into a single string. Using `DISTINCT` inside the function removes duplicate product names, and `ORDER BY product` sorts them lexicographically before concatenation.

Finally, `ORDER BY sell_date` ensures the output rows appear in chronological order.

## Go Solution

```go

SELECT
    sell_date,
    COUNT(DISTINCT product) AS num_sold,
    GROUP_CONCAT(
        DISTINCT product
        ORDER BY product
        SEPARATOR ','
    ) AS products
FROM Activities
GROUP BY sell_date
ORDER BY sell_date;
```

Unlike algorithmic coding problems, this database problem is solved entirely with SQL. Therefore, LeetCode does not require a traditional Go function implementation.

## Edge Cases

### Duplicate product entries on the same date

This is the most important edge case because the table explicitly allows duplicates. A naive query using `COUNT(product)` would incorrectly count repeated products multiple times. Similarly, a plain `GROUP_CONCAT(product)` would repeat product names in the output string. The implementation avoids this issue by using `DISTINCT` in both aggregation functions.

### Only one product sold on a date

Some dates may contain exactly one unique product. In this situation, the output should still contain a valid comma-separated string, even though no commas are needed. `GROUP_CONCAT` naturally handles this case by returning the single product name directly.

### Products appearing on multiple dates

The same product may appear across different dates. The grouping logic must ensure that uniqueness is applied only within each individual date group, not globally across the table. Because the query groups rows by `sell_date` first, each date is processed independently and correctly.
