## Solution

## Problem Understanding

This problem gives us an undirected tree with `n` nodes numbered from `0` to `n - 1`. Some nodes contain apples, represented by the boolean array `hasApple`, where `hasApple[i]` is `true` if node `i` has an apple.

We start at node `0`, which is the root for traversal purposes. Moving across a single edge costs `1` second. The goal is to collect every apple in the tree and finally return back to node `0`. We need to compute the minimum total travel time required.

A tree is an acyclic connected graph, which is important because it guarantees there is exactly one path between any two nodes. This property allows us to avoid worrying about multiple routes or shortest path algorithms.

The input consists of:

- `n`, the number of vertices
- `edges`, a list describing the undirected connections between nodes
- `hasApple`, indicating which nodes contain apples

The output is a single integer representing the minimum number of seconds needed to collect all apples and return to node `0`.

The constraints are large:

- `n` can be up to `10^5`
- The number of edges is `n - 1`

This immediately tells us that inefficient graph traversal approaches will not work. We need a solution close to linear time, ideally `O(n)`.

Several edge cases are important:

- No apples exist anywhere in the tree, the answer should be `0`
- Only the root node has an apple, the answer is also `0` because we already start there
- Apples may exist deep in the tree, requiring traversal through many intermediate nodes
- Multiple apples may share parts of their paths, so we should avoid counting shared edges multiple times

The key challenge is determining which edges actually need to be traversed.

## Approaches

### Brute Force Approach

A brute force approach would attempt to explicitly generate traversal paths for collecting apples. One possibility is to compute paths from node `0` to every apple node, combine them, and count all traversed edges.

For every apple node, we could perform a graph traversal such as DFS or BFS to reconstruct the path from the root. Then we could merge all paths and count unique edges.

This approach works because every apple must eventually be visited, and every required edge must belong to at least one root-to-apple path.

However, this approach is inefficient because repeatedly traversing the tree for every apple can lead to `O(n^2)` complexity in the worst case. Consider a skewed tree where almost every node contains an apple. Recomputing paths repeatedly becomes too expensive for `10^5` nodes.

### Optimal Approach

The important observation is that we only need to traverse edges that lead to at least one apple somewhere in their subtree.

If a subtree contains no apples, we never need to visit it. That entire branch can be ignored.

This naturally suggests using Depth-First Search. During DFS, each node asks its children whether their subtree contains any apples. If a child subtree contains at least one apple, then we must travel to that child and return back, costing `2` seconds for that edge.

This creates a very elegant recursive solution:

- Recursively compute the cost for each child subtree
- If the subtree contains apples, add `2` for the round trip edge
- Otherwise, contribute nothing

Because every edge is processed exactly once, the algorithm runs in linear time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeatedly reconstructs paths to apples |
| Optimal | O(n) | O(n) | DFS visits each node and edge once |

## Algorithm Walkthrough

1. Build an adjacency list from the `edges` array.

Since the graph is undirected, each edge `[u, v]` must be added in both directions. An adjacency list is ideal because trees are sparse graphs with exactly `n - 1` edges.
2. Start a DFS traversal from node `0`.

We recursively explore the tree while keeping track of the parent node. Tracking the parent prevents us from traversing backward and creating infinite recursion.
3. For each node, recursively process all children.

Every recursive call returns the amount of time needed to collect all apples in that child subtree and return back to the child node.
4. Determine whether the current subtree contains apples.

A subtree is useful if:

- The current node itself has an apple
- At least one child subtree contains an apple
5. Add traversal cost when needed.

If a child subtree contains apples, then we must:

- Travel from the current node to the child, costing `1`
- Return back to the current node, costing another `1`

Therefore, we add `2` seconds for every useful child subtree.
6. Return the total cost upward.

Each recursive call returns the total traversal cost required for that subtree.
7. The final DFS result from node `0` is the answer.

Since the DFS already accounts for returning back to the root, the returned value is exactly the minimum required time.

### Why it works

The algorithm works because every necessary edge in the final traversal must connect to a subtree containing at least one apple. DFS correctly identifies those subtrees recursively.

Each required edge is traversed exactly twice:

- Once going down
- Once returning back

Unnecessary branches are skipped entirely because their recursive calls contribute zero cost. Since trees have unique paths between nodes, this guarantees the minimum traversal cost.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Solution:
    def minTime(self, n: int, edges: List[List[int]], hasApple: List[bool]) -> int:
        graph = defaultdict(list)

        for u, v in edges:
            graph[u].append(v)
            graph[v].append(u)

        def dfs(node: int, parent: int) -> int:
            total_time = 0

            for neighbor in graph[node]:
                if neighbor == parent:
                    continue

                child_time = dfs(neighbor, node)

                if child_time > 0 or hasApple[neighbor]:
                    total_time += child_time + 2

            return total_time

        return dfs(0, -1)
```

The implementation begins by constructing an adjacency list called `graph`. Since the tree is undirected, each edge is inserted in both directions.

The recursive `dfs` function computes the minimum time required to collect all apples in the subtree rooted at `node` and return back to `node`.

Inside the DFS:

- We initialize `total_time` to accumulate traversal costs
- We iterate through all neighboring nodes
- We skip the parent node to avoid traversing backward
- We recursively compute the cost of the child subtree

The important logic is:

```python
if child_time > 0 or hasApple[neighbor]:
```

This condition determines whether the child subtree is useful.

If the child subtree already requires traversal time, then apples exist somewhere below it. Alternatively, the child node itself may contain an apple.

In either case, we must travel to that child and back, so we add:

```python
child_time + 2
```

Finally, the DFS result from node `0` is returned as the answer.

## Go Solution

```go
func minTime(n int, edges [][]int, hasApple []bool) int {
	graph := make([][]int, n)

	for _, edge := range edges {
		u := edge[0]
		v := edge[1]

		graph[u] = append(graph[u], v)
		graph[v] = append(graph[v], u)
	}

	var dfs func(node int, parent int) int

	dfs = func(node int, parent int) int {
		totalTime := 0

		for _, neighbor := range graph[node] {
			if neighbor == parent {
				continue
			}

			childTime := dfs(neighbor, node)

			if childTime > 0 || hasApple[neighbor] {
				totalTime += childTime + 2
			}
		}

		return totalTime
	}

	return dfs(0, -1)
}
```

The Go solution follows the same logic as the Python implementation.

Instead of `defaultdict`, Go uses a slice of slices:

```go
graph := make([][]int, n)
```

This creates an adjacency list for all nodes.

The DFS function is declared as a closure so it can recursively reference itself. Integer overflow is not a concern because the maximum traversal cost is bounded by `2 * (n - 1)`.

Go slices automatically grow as elements are appended, making adjacency list construction straightforward.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every node and edge is processed once |
| Space | O(n) | Adjacency list and recursion stack |

The adjacency list stores exactly `2 * (n - 1)` entries because each undirected edge is stored twice.

The DFS traversal visits every node exactly once and processes every edge exactly once, resulting in linear time complexity.

The recursion stack can reach depth `n` in a skewed tree, so the worst case auxiliary space is also linear.

## Edge Cases

One important edge case occurs when no apples exist anywhere in the tree. A naive implementation might still traverse parts of the tree unnecessarily. In this solution, every DFS call returns `0` because neither the child subtree nor the child node contains apples. As a result, the final answer correctly becomes `0`.

Another critical edge case is when the root node itself contains an apple. Since we already start at node `0`, no movement is required. Some incorrect implementations mistakenly add traversal cost simply because an apple exists. This solution avoids that issue because it only adds cost when traversing to child subtrees.

A skewed tree is another important case. Consider a chain-like tree with depth close to `10^5`. Recursive traversal must correctly process long paths without incorrectly revisiting nodes. The parent-tracking mechanism ensures that DFS moves strictly downward through the tree and never loops backward.

A final subtle edge case involves multiple apples sharing part of the same path. For example, if two apples exist in the same subtree, the shared edges should only be traversed once in each direction. The DFS accumulation naturally handles this because subtree costs are aggregated together before returning upward.
