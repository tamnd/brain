## Solution

## Problem Understanding

The problem asks us to return the `k` strongest values from an integer array. The definition of "strongest" is based on a special value called the centre, also referred to as the median-like middle element of the sorted array.

To determine strength, we first sort the array and compute:

$$m = \text{sorted\_arr}[(n - 1) / 2]$$

where `n` is the array length and integer division is used.

An element `x` is considered stronger than an element `y` if:

1. `|x - m| > |y - m|`
2. If the distances are equal, then the larger value is stronger

This means strength is determined primarily by how far a number is from the centre, and secondarily by the number itself.

The output can be in any order, which is important because we do not need to preserve original ordering or produce a uniquely sorted answer.

The constraints are large:

- `arr.length` can be up to `100000`
- Values can range from `-100000` to `100000`

These limits strongly suggest that quadratic solutions are too slow. Any efficient solution should ideally be around `O(n log n)` because sorting is almost certainly required.

Several edge cases are important:

- Arrays with many duplicate values
- Arrays where all elements are identical
- Arrays containing negative numbers
- Cases where multiple elements have identical strength
- Cases where `k == arr.length`
- Even-length arrays, where the centre uses index `(n - 1) / 2`

The tie-breaking rule is especially important because many naive implementations incorrectly treat equal distances as equal strength.

## Approaches

### Brute Force Approach

The brute-force method begins by sorting the array to determine the centre value. Then, for every element, we compute its strength using the pair:

$$(|arr[i] - m|, arr[i])$$

We can then sort all elements again using a custom comparator:

- Larger absolute difference comes first
- If equal, larger value comes first

After sorting by strength, we simply take the first `k` elements.

This approach is correct because it explicitly follows the exact comparison rule given in the problem statement. Every element is ranked according to the definition of strength.

However, this solution performs a full custom sort over all elements after already sorting the array once. While still acceptable for the constraints, it does more work than necessary.

### Key Insight for the Optimal Solution

After sorting the array, the strongest values must come from the ends of the sorted array.

Why?

- Elements farthest from the centre have the largest absolute difference
- In a sorted array, those values naturally appear near the leftmost and rightmost ends

This observation allows us to use a two-pointer approach:

- One pointer starts at the beginning
- One pointer starts at the end

At each step:

- Compare the strength of both candidates
- Take the stronger one
- Move the corresponding pointer inward

This avoids an additional full strength sort and directly extracts the strongest values greedily.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) | O(n) | Sorts entire array again using custom strength ordering |
| Optimal | O(n log n) | O(k) | Uses sorting once, then two pointers to greedily select strongest values |

## Algorithm Walkthrough

1. Sort the array in non-decreasing order.

Sorting is necessary because the centre depends on the ordered array. It also enables the two-pointer strategy since the strongest elements will lie near the extremes.
2. Compute the centre value.

The centre index is:

$$(n - 1) // 2$$

where `n` is the array length.

The centre value is:

$$m = arr[(n - 1) // 2]$$
3. Initialize two pointers.

- `left = 0`
- `right = n - 1`

These pointers represent the smallest and largest remaining values.
4. Repeatedly choose the stronger value.

Compare:

$$|arr[left] - m|$$

and

$$|arr[right] - m|$$

If the right value has greater or equal strength, select it. Otherwise, select the left value.

The tie-breaking rule naturally favors the larger number, which is always the right side in a sorted array.
5. Append the selected value to the result.

Move the corresponding pointer inward.
6. Continue until `k` elements have been selected.
7. Return the result.

### Why it works

After sorting, values farther from the centre must appear closer to the array boundaries. At every step, the strongest remaining element must therefore be either the current leftmost or rightmost element.

The algorithm always compares exactly those two candidates and chooses the stronger one according to the problem definition. Since each selected value is globally strongest among the remaining elements, greedily taking it preserves correctness throughout the process.

## Python Solution

```python
from typing import List

class Solution:
    def getStrongest(self, arr: List[int], k: int) -> List[int]:
        arr.sort()

        n = len(arr)
        median = arr[(n - 1) // 2]

        left = 0
        right = n - 1

        result = []

        while len(result) < k:
            left_strength = abs(arr[left] - median)
            right_strength = abs(arr[right] - median)

            if right_strength > left_strength:
                result.append(arr[right])
                right -= 1
            elif right_strength < left_strength:
                result.append(arr[left])
                left += 1
            else:
                if arr[right] >= arr[left]:
                    result.append(arr[right])
                    right -= 1
                else:
                    result.append(arr[left])
                    left += 1

        return result
```

The implementation starts by sorting the array because both the centre computation and the two-pointer strategy depend on ordered values.

The median is computed using index `(n - 1) // 2`, exactly matching the problem statement.

Two pointers are then initialized at the extremes of the array. During each iteration, the code computes the strength of both candidates relative to the median.

The comparison follows the exact problem rules:

- Larger absolute difference wins
- If equal, larger value wins

Since the array is sorted, the right pointer always contains the larger value when strengths tie.

Each selected value is appended to `result`, and the appropriate pointer moves inward. The loop continues until exactly `k` elements have been collected.

## Go Solution

```go
package main

import (
	"math"
	"sort"
)

func getStrongest(arr []int, k int) []int {
	sort.Ints(arr)

	n := len(arr)
	median := arr[(n-1)/2]

	left := 0
	right := n - 1

	result := make([]int, 0, k)

	for len(result) < k {
		leftStrength := int(math.Abs(float64(arr[left] - median)))
		rightStrength := int(math.Abs(float64(arr[right] - median)))

		if rightStrength > leftStrength {
			result = append(result, arr[right])
			right--
		} else if rightStrength < leftStrength {
			result = append(result, arr[left])
			left++
		} else {
			if arr[right] >= arr[left] {
				result = append(result, arr[right])
				right--
			} else {
				result = append(result, arr[left])
				left++
			}
		}
	}

	return result
}
```

The Go implementation follows the same logic as the Python version.

The primary difference is that Go does not have a built-in absolute value function for integers, so `math.Abs` is used with float conversion.

Slices are dynamically appended using `append`, and the result slice is preallocated with capacity `k` for efficiency.

No special handling for nil slices is required because the constraints guarantee at least one element.

## Edge Cases

One important edge case occurs when all elements are identical. In this situation, every value has the same distance from the centre and the same numeric value. A buggy implementation might incorrectly move pointers or mishandle ties. This implementation works correctly because both sides are treated equally and the algorithm still consistently selects valid elements until `k` values are collected.

Another important case is an even-length array. The problem does not use the mathematical median. Instead, it specifically defines the centre as the element at index `(n - 1) / 2` after sorting. Many implementations incorrectly average the two middle values. This solution follows the exact definition from the problem statement, ensuring correctness.

A third tricky case involves equal strengths with different values. For example, in `[1,5]`, both values are equally distant from the centre `1`, but `5` is considered stronger because it is larger. The implementation explicitly handles this tie-breaking rule by preferring the larger value whenever absolute differences are equal.

Another subtle case occurs with negative numbers. Since strength depends on absolute difference from the centre, negative values must be handled carefully. Using `abs(arr[i] - median)` ensures that both positive and negative distances are compared correctly.
