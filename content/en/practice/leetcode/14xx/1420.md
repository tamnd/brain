## Solution

## Problem Understanding

LeetCode 1420, LeetCode "Build Array Where You Can Find The Maximum Exactly K Comparisons", asks us to count how many arrays satisfy a very specific property related to how the maximum value changes while scanning the array from left to right.

The problem gives us three integers:

- `n`, the length of the array
- `m`, the maximum allowed value in the array
- `k`, the exact number of times the running maximum must increase

Each element of the array must satisfy:

- `1 <= arr[i] <= m`

We then simulate the following process:

- Start with `current_max = -1`
- Scan the array from left to right
- Whenever we encounter a value strictly larger than `current_max`, we update the maximum and increment `search_cost`

The final `search_cost` must equal exactly `k`.

In other words, we are counting arrays where the maximum value changes exactly `k` times while traversing the array.

For example, consider:

```
[2, 1, 4, 3]
```

The running maximum evolves like this:

| Index | Value | Running Max | Cost Increase |
| --- | --- | --- | --- |
| 0 | 2 | 2 | yes |
| 1 | 1 | 2 | no |
| 2 | 4 | 4 | yes |
| 3 | 3 | 4 | no |

The total search cost is `2`.

The constraints are important:

- `n <= 50`
- `m <= 100`
- `k <= n`

A brute force enumeration would require checking all possible arrays. Since there are `m^n` arrays, this becomes astronomically large even for moderate inputs. For example:

```
100^50
```

is completely infeasible.

The constraints strongly suggest a dynamic programming solution. The state space is small enough that we can track:

- how many elements we have placed,
- what the current maximum is,
- how many times the maximum has changed so far.

Several edge cases are important:

- `k = 0` is impossible unless `n = 0`, which never occurs in this problem, so the answer is always `0`
- `k > m` is impossible because the maximum can increase at most `m` distinct times
- `m = 1` means every element must be `1`, so the search cost is always exactly `1`
- Arrays where every new element becomes a new maximum produce the largest possible cost, which is `n`

## Approaches

### Brute Force

The brute force approach generates every possible array of length `n`, where each element ranges from `1` to `m`.

For each generated array, we simulate the search process:

- Track the current maximum
- Count how many times the maximum changes
- If the count equals `k`, include the array in the answer

This approach is correct because it explicitly checks every valid array and counts exactly those satisfying the condition.

However, the runtime is completely impractical. There are:

```
m^n
```

possible arrays.

With the maximum constraints:

```
100^50
```

the search space is impossibly large.

### Optimal Dynamic Programming with Prefix Sums

The key observation is that the only information that matters while building the array is:

- how many elements we have already placed,
- what the current maximum value is,
- how many maximum updates have occurred.

This naturally leads to dynamic programming.

Suppose we already built a prefix of the array.

When adding a new value:

- If the value is less than or equal to the current maximum, the search cost does not change.
- If the value is greater than the current maximum, the search cost increases by one and the new value becomes the new maximum.

This gives us clean transitions between states.

A naive DP transition would still be too slow because computing transitions for all previous maximums introduces another factor of `m`.

To optimize this, we use prefix sums.

Instead of recomputing:

```
sum(dp[previous_max])
```

repeatedly, we maintain cumulative sums so transitions can be computed in constant time.

This reduces the complexity enough to fit comfortably within the constraints.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m^n * n) | O(n) | Generates every possible array |
| Optimal | O(n * m * k) | O(m * k) | Dynamic programming with prefix sum optimization |

## Algorithm Walkthrough

1. Define the DP state.

Let:

```
dp[max_value][cost]
```

represent the number of arrays constructed so far where:

- the current maximum equals `max_value`
- the search cost equals `cost`

We process the array one position at a time.
2. Initialize the first position.

For arrays of length `1`, choosing value `x` creates:

- maximum = `x`
- search cost = `1`

Therefore:

```
dp[x][1] = 1
```

for all `1 <= x <= m`.
3. Process additional positions.

For every new element we append, there are two possibilities.
4. Case 1, append a value that does not create a new maximum.

Suppose the current maximum is `max_value`.

We may append any value from:

```
1..max_value
```

There are exactly `max_value` choices.

The search cost remains unchanged.

Transition:

```
next_dp[max_value][cost] +=
    dp[max_value][cost] * max_value
```
5. Case 2, append a value that becomes a new maximum.

Suppose the new maximum becomes `new_max`.

Then the previous maximum must be smaller:

```
previous_max < new_max
```

The search cost increases by one.

Transition:

```
next_dp[new_max][cost + 1] +=
    sum(dp[previous_max][cost])
```

over all:

```
previous_max < new_max
```
6. Optimize using prefix sums.

Directly summing all previous maximums would cost `O(m)` per transition.

Instead, for each cost value we maintain:

```
prefix[new_max]
```

which stores cumulative sums up to `new_max`.

Then:

```
sum(previous_max < new_max)
```

becomes an `O(1)` lookup.
7. Repeat for all positions.

We process positions from length `2` through `n`.
8. Compute the final answer.

The answer is:

```
sum(dp[max_value][k])
```

over all maximum values.

### Why it works

The DP state captures exactly the information needed to continue constructing the array correctly:

- the current maximum,
- the current search cost.

Every possible appended value falls into exactly one of two categories:

- it does not change the maximum,
- it creates a new maximum.

The transitions exhaust all valid possibilities without overlap, so every valid array is counted exactly once.

## Python Solution

```python
class Solution:
    def numOfArrays(self, n: int, m: int, k: int) -> int:
        MOD = 10**9 + 7

        if k == 0:
            return 0

        dp = [[0] * (k + 1) for _ in range(m + 1)]

        for max_value in range(1, m + 1):
            dp[max_value][1] = 1

        for _ in range(2, n + 1):
            next_dp = [[0] * (k + 1) for _ in range(m + 1)]

            for cost in range(1, k + 1):
                prefix = [0] * (m + 1)

                for max_value in range(1, m + 1):
                    prefix[max_value] = (
                        prefix[max_value - 1]
                        + dp[max_value][cost - 1]
                    ) % MOD

                for max_value in range(1, m + 1):
                    # Append value <= current max
                    next_dp[max_value][cost] = (
                        next_dp[max_value][cost]
                        + dp[max_value][cost] * max_value
                    ) % MOD

                    # Append value that creates new max
                    next_dp[max_value][cost] = (
                        next_dp[max_value][cost]
                        + prefix[max_value - 1]
                    ) % MOD

            dp = next_dp

        answer = 0

        for max_value in range(1, m + 1):
            answer = (answer + dp[max_value][k]) % MOD

        return answer
```

The implementation follows the DP formulation directly.

We first initialize all arrays of length `1`. Each possible starting value contributes exactly one valid array with search cost `1`.

For every additional position, we create a fresh `next_dp` table representing arrays one element longer.

The prefix sum array is rebuilt for every search cost. This allows us to efficiently compute how many previous states can transition into a larger maximum.

The first transition handles values that do not increase the maximum. Since there are `max_value` possible choices, we multiply the number of existing arrays by `max_value`.

The second transition handles creating a new maximum. The prefix sum provides the total number of smaller previous maximums in constant time.

Finally, we sum all states whose search cost equals `k`.

## Go Solution

```go
func numOfArrays(n int, m int, k int) int {
	const MOD int = 1e9 + 7

	if k == 0 {
		return 0
	}

	dp := make([][]int, m+1)

	for i := range dp {
		dp[i] = make([]int, k+1)
	}

	for maxValue := 1; maxValue <= m; maxValue++ {
		dp[maxValue][1] = 1
	}

	for length := 2; length <= n; length++ {
		nextDP := make([][]int, m+1)

		for i := range nextDP {
			nextDP[i] = make([]int, k+1)
		}

		for cost := 1; cost <= k; cost++ {
			prefix := make([]int, m+1)

			for maxValue := 1; maxValue <= m; maxValue++ {
				prefix[maxValue] = (
					prefix[maxValue-1] +
						dp[maxValue][cost-1]
				) % MOD
			}

			for maxValue := 1; maxValue <= m; maxValue++ {
				// Append value <= current max
				nextDP[maxValue][cost] = (
					nextDP[maxValue][cost] +
						dp[maxValue][cost]*maxValue
				) % MOD

				// Append value that creates new max
				nextDP[maxValue][cost] = (
					nextDP[maxValue][cost] +
						prefix[maxValue-1]
				) % MOD
			}
		}

		dp = nextDP
	}

	answer := 0

	for maxValue := 1; maxValue <= m; maxValue++ {
		answer = (answer + dp[maxValue][k]) % MOD
	}

	return answer
}
```

The Go implementation mirrors the Python logic closely.

Slices are used instead of nested Python lists. Since Go integers are fixed-width, we carefully apply modulo operations after every update to avoid overflow issues.

Unlike Python, Go does not automatically initialize nested slices, so each inner slice must be explicitly allocated.

For maximum `1`:

- We can append only `1`
- Count becomes:

- Existing arrays with max `2` can append `1` or `2`
- Contribution:

```
1 * 2 = 2
```

- Existing arrays with max `3` can append `1`, `2`, or `3`
- Contribution:

```
1 * 3 = 3

```
1 + 2 + 3 = 6

[1,1]

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n * m * k) | Each DP state is processed once with O(1) transitions |
| Space | O(m * k) | Only current and next DP layers are stored |

The optimization comes from prefix sums. Without them, transitions involving new maximums would require iterating over all smaller maximums, increasing complexity to `O(n * m^2 * k)`.

Using cumulative sums reduces that inner computation to constant time.

## Edge Cases

One important edge case is `k = 0`. Since the first element always creates a new maximum, every non-empty array has at least search cost `1`. A naive implementation might accidentally allow zero-cost states to propagate through the DP. The implementation explicitly returns `0` immediately when `k == 0`.

Another important edge case occurs when `k > m`. The running maximum can only increase to distinct values between `1` and `m`. Once the maximum reaches `m`, it cannot increase further. Therefore, achieving more than `m` maximum updates is impossible. The DP naturally handles this because no valid transitions exist for such states.

The case `m = 1` is also easy to mishandle. When all elements must equal `1`, there is exactly one possible array regardless of `n`, and its search cost is always `1`. Some incorrect implementations accidentally count multiple transitions or fail to initialize the DP correctly. Our initialization guarantees only one valid path exists.

Another subtle case occurs when `k = n`. This means every new element must create a new maximum. Therefore, the array must be strictly increasing. The DP handles this naturally because the only valid transitions are those that move to larger maximum values at every step.
