## Solution

## Problem Understanding

The problem gives an integer array `nums` and an integer `limit`. We must find the length of the longest contiguous subarray where the absolute difference between any two elements in that subarray is at most `limit`.

Another way to think about the condition is this:

A subarray is valid if:

$$\max(subarray) - \min(subarray) \le limit$$

This reformulation is extremely important because instead of checking every pair of elements, we only need to know the maximum and minimum values inside the current window.

The input array can contain up to:

$$10^5$$

elements, which immediately tells us that quadratic algorithms will not work. Any approach worse than roughly $O(n \log n)$ is likely too slow.

The output is a single integer representing the size of the longest valid contiguous subarray.

Several edge cases are important:

- Arrays of length 1 must return 1 because every single element subarray is valid.
- A `limit` of 0 means all elements inside the valid window must be identical.
- Strictly increasing or decreasing arrays can cause frequent window shrinking.
- Large values up to $10^9$ mean we should avoid unnecessary arithmetic tricks or counting arrays.

The problem guarantees the array is non-empty, so we never need to handle an empty input.

## Approaches

### Brute Force Approach

The brute force solution examines every possible subarray. For each starting index, we extend the ending index one step at a time while tracking the minimum and maximum values in the current subarray.

Whenever:

$$maxVal - minVal \le limit$$

the subarray is valid, and we update the answer.

This approach is correct because it explicitly checks every contiguous subarray, so the longest valid one cannot be missed.

However, there are $O(n^2)$ possible subarrays. Even if we maintain running minimum and maximum values efficiently during extension, the total complexity remains quadratic.

With $n = 10^5$, this becomes far too slow.

### Optimal Sliding Window + Monotonic Queue Approach

The key observation is that we only need to maintain the current window's minimum and maximum values.

A sliding window works naturally because:

- If a window is valid, extending it may or may not remain valid.
- If a window becomes invalid, removing elements from the left can restore validity.

The challenge is efficiently tracking the minimum and maximum values inside the current window.

A monotonic deque solves this perfectly:

- One deque stores elements in decreasing order, its front is the maximum.
- Another deque stores elements in increasing order, its front is the minimum.

As the right pointer expands:

- We insert the new value while maintaining monotonic order.
- We shrink the left side whenever the difference exceeds `limit`.

Each element enters and leaves each deque at most once, giving linear time complexity.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Checks every subarray explicitly |
| Optimal | O(n) | O(n) | Sliding window with monotonic deques |

## Algorithm Walkthrough

1. Initialize two deques.

One deque maintains decreasing order for maximum values. The front always contains the largest value in the current window.

The second deque maintains increasing order for minimum values. The front always contains the smallest value in the current window.
2. Initialize two pointers.

The `left` pointer represents the start of the sliding window, and the `right` pointer expands the window across the array.
3. Process each element from left to right.

For each `nums[right]`:

- Remove smaller elements from the back of the max deque because they can never become the maximum again while the current element exists.
- Append the current value to the max deque.
4. Maintain the min deque similarly.

- Remove larger elements from the back of the min deque because they can never become the minimum again.
- Append the current value.
5. Check whether the window is valid.

The current window maximum is:

$$maxDeque[0]$$

The current window minimum is:

$$minDeque[0]$$

If:

$$maxDeque[0] - minDeque[0] > limit$$

then the window is invalid.
6. Shrink the window from the left until it becomes valid again.

Before moving `left`:

- If `nums[left]` equals the front of the max deque, remove it.
- If `nums[left]` equals the front of the min deque, remove it.

Then increment `left`.
7. Update the answer.

After restoring validity:

$$answer = \max(answer, right - left + 1)$$
8. Continue until the array is fully processed.

### Why it works

The algorithm maintains the invariant that the current window always satisfies:

$$max(window) - min(window) \le limit$$

The deques always contain candidates for the minimum and maximum values in sorted order. Since invalid windows are immediately shrunk, every valid window is considered exactly once. Because each element is inserted and removed at most once from each deque, the algorithm runs in linear time.

## Python Solution

```python
from collections import deque
from typing import List

class Solution:
    def longestSubarray(self, nums: List[int], limit: int) -> int:
        max_deque = deque()
        min_deque = deque()

        left = 0
        longest = 0

        for right, value in enumerate(nums):

            while max_deque and value > max_deque[-1]:
                max_deque.pop()

            max_deque.append(value)

            while min_deque and value < min_deque[-1]:
                min_deque.pop()

            min_deque.append(value)

            while max_deque[0] - min_deque[0] > limit:

                if nums[left] == max_deque[0]:
                    max_deque.popleft()

                if nums[left] == min_deque[0]:
                    min_deque.popleft()

                left += 1

            longest = max(longest, right - left + 1)

        return longest
```

The implementation closely follows the sliding window algorithm.

The `max_deque` stores values in decreasing order. The largest value always stays at the front. Before inserting a new value, smaller values are removed because they can never become the maximum again.

The `min_deque` works symmetrically. It stores values in increasing order so the smallest value remains at the front.

The `right` pointer expands the window one element at a time. After inserting the new value into both deques, we check whether the window violates the limit condition.

If the window becomes invalid, the `left` pointer moves forward until the window becomes valid again. Whenever the outgoing value equals the front of either deque, that front element is removed because it is no longer inside the window.

Finally, the algorithm updates the maximum valid window length found so far.

## Go Solution

```go
func longestSubarray(nums []int, limit int) int {
    maxDeque := []int{}
    minDeque := []int{}

    left := 0
    longest := 0

    for right, value := range nums {

        for len(maxDeque) > 0 && value > maxDeque[len(maxDeque)-1] {
            maxDeque = maxDeque[:len(maxDeque)-1]
        }

        maxDeque = append(maxDeque, value)

        for len(minDeque) > 0 && value < minDeque[len(minDeque)-1] {
            minDeque = minDeque[:len(minDeque)-1]
        }

        minDeque = append(minDeque, value)

        for maxDeque[0]-minDeque[0] > limit {

            if nums[left] == maxDeque[0] {
                maxDeque = maxDeque[1:]
            }

            if nums[left] == minDeque[0] {
                minDeque = minDeque[1:]
            }

            left++
        }

        windowLength := right - left + 1

        if windowLength > longest {
            longest = windowLength
        }
    }

    return longest
}
```

The Go implementation mirrors the Python solution closely. Since Go does not provide a built-in deque structure, slices are used to simulate deques.

Removing from the back is done with slicing:

```go
deque = deque[:len(deque)-1]
```

Removing from the front is done with:

```go
deque = deque[1:]
```

Go integers are sufficient for this problem because all values fit comfortably inside standard 64-bit integer ranges used by modern Go implementations.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each element enters and leaves each deque at most once |
| Space | O(n) | Deques can store up to all elements in worst case |

The linear time complexity comes from amortized analysis. Even though there are nested loops, every array element is inserted into and removed from the deques only once. Therefore, the total number of deque operations is proportional to `n`.

The space complexity is linear because, in the worst case, the deques may contain all elements of the current window.

## Edge Cases

### Limit Equals Zero

When `limit = 0`, every valid subarray must contain only identical values. This is a common source of bugs because any difference immediately invalidates the window.

The monotonic deque solution handles this naturally because the condition:

$$max - min \le 0$$

only holds when `max == min`.

### Strictly Increasing or Decreasing Arrays

Arrays such as:

```
[1,2,3,4,5]
```

or:

```
[5,4,3,2,1]
```

cause continuous updates to the deques. A poorly implemented deque structure could become inefficient.

The monotonic deque guarantees amortized constant-time operations because every element is pushed and popped at most once.

### Arrays With Large Integer Values

The constraints allow values up to:

$$10^9$$

Naive implementations using unsafe arithmetic or custom difference logic could overflow in some languages.

This implementation only computes:

```
max - min
```

which safely fits inside standard integer ranges in both Python and Go.
