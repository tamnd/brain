## Solution

## Problem Understanding

The problem gives us two integer arrays, `target` and `arr`, both having the same length. In a single operation, we are allowed to select any non-empty contiguous subarray of `arr` and reverse its elements. We may perform this operation as many times as we want.

The goal is to determine whether it is possible to transform `arr` into `target` using these reversal operations.

At first glance, the problem may appear to require simulation of array reversals. However, the important observation is that reversing subarrays is an extremely flexible operation. By repeatedly reversing carefully chosen segments, we can effectively rearrange elements into almost any order we want.

This means the real question is not about the order of elements initially present in `arr`, but whether `arr` contains exactly the same elements as `target`, including duplicates.

The input consists of:

- `target`, the desired final arrangement
- `arr`, the array we are allowed to manipulate

The expected output is:

- `true` if `arr` can be transformed into `target`
- `false` otherwise

The constraints are relatively small:

- Array length is at most 1000
- Element values range from 1 to 1000

These constraints tell us that even an `O(n log n)` sorting solution is perfectly acceptable. We do not need advanced optimizations.

Several edge cases are important to recognize early:

- Arrays of length 1
- Arrays containing duplicate values
- Arrays containing different frequencies of the same value
- Arrays already equal
- Arrays with one missing or extra element value

A naive implementation that only checks whether both arrays contain the same unique values would fail for duplicates. For example:

```
target = [1,1,2]
arr = [1,2,2]
```

Both arrays contain `1` and `2`, but the frequencies differ, so transformation is impossible.

## Approaches

### Brute Force Approach

A brute force solution would attempt to simulate all possible reversal operations and search for a sequence of reversals that transforms `arr` into `target`.

This can be modeled as a graph search problem:

- Each array configuration is a state
- Reversing a subarray creates a new state
- We search until we either find `target` or exhaust all possibilities

This approach is correct because it explicitly explores all reachable configurations. However, it is computationally infeasible.

For an array of length `n`, there are:

```
n * (n + 1) / 2
```

possible subarrays to reverse at each step. The number of reachable permutations grows factorially, making the search exponentially expensive.

Even for relatively small arrays, this approach becomes impractical.

### Optimal Approach

The key insight is that reversing subarrays allows arbitrary permutations of the array.

Any adjacent swap can be achieved through reversals, and adjacent swaps are sufficient to generate any permutation. Therefore, the only thing that matters is whether both arrays contain the same multiset of values.

A multiset means:

- Same elements
- Same frequency of each element

There are two common ways to verify this:

- Sort both arrays and compare
- Count frequencies using a hash map

Both are valid. The hash map approach achieves linear time complexity.

We count how many times each value appears in `target`, then subtract counts using `arr`. If every count ends at zero, the arrays contain identical frequencies and transformation is possible.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n! * n²) | O(n!) | Explores many reversal sequences and array states |
| Optimal | O(n) | O(n) | Uses frequency counting to verify both arrays contain identical elements |

## Algorithm Walkthrough

### Frequency Counting Algorithm

1. Create an empty hash map called `frequency`.

The hash map will store how many times each value appears in `target`.
2. Iterate through `target`.

For each number:

- Increment its count in the hash map

After this step, the map represents the required element frequencies.
3. Iterate through `arr`.

For each number:

- Decrement its count in the hash map

This removes matching elements from the required frequencies.
4. Check the final counts in the hash map.

If every count is zero, both arrays contain exactly the same elements with identical frequencies.

Otherwise, transformation is impossible.
5. Return the result.

### Why it works

Reversing subarrays is powerful enough to rearrange elements into any permutation. Therefore, order does not matter, only the collection of elements matters.

The frequency map acts as an invariant checker. If every value appears the same number of times in both arrays, then `arr` can always be rearranged into `target`. If even one frequency differs, no sequence of reversals can create or remove elements, so transformation is impossible.

## Python Solution

```python
from typing import List
from collections import Counter

class Solution:
    def canBeEqual(self, target: List[int], arr: List[int]) -> bool:
        return Counter(target) == Counter(arr)
```

The implementation uses Python's `Counter` class from the `collections` module. A `Counter` is essentially a hash map specialized for counting frequencies.

First, `Counter(target)` builds a frequency map for all values in `target`.

Then, `Counter(arr)` builds the frequency map for `arr`.

Finally, the two counters are compared directly. Python automatically checks whether both maps contain identical keys and identical counts.

This implementation is concise because `Counter` abstracts away the manual increment and decrement logic while still implementing the same frequency-counting algorithm described earlier.

## Go Solution

```go
func canBeEqual(target []int, arr []int) bool {
    frequency := make(map[int]int)

    for _, value := range target {
        frequency[value]++
    }

    for _, value := range arr {
        frequency[value]--
    }

    for _, count := range frequency {
        if count != 0 {
            return false
        }
    }

    return true
}
```

The Go solution uses a `map[int]int` as the frequency table.

Unlike Python's `Counter`, Go requires explicit management of counts. We increment counts while scanning `target`, decrement counts while scanning `arr`, and finally verify that all counts are zero.

Go slices naturally handle empty and non-empty arrays uniformly, so no special handling is required. Integer overflow is not a concern because counts are bounded by the array length, which is at most 1000.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each array is traversed once |
| Space | O(n) | The hash map may store up to `n` distinct elements |

The algorithm performs two linear passes through the arrays. Hash map insertion and lookup operations are `O(1)` on average, so the total runtime is linear.

The extra space comes from storing frequencies in the hash map. In the worst case, every element is distinct, requiring `O(n)` space.

## Edge Cases

### Arrays of Length One

The smallest possible input size is length 1. A buggy implementation might overcomplicate reversal logic even though no meaningful rearrangement exists.

For example:

```
target = [7]
arr = [7]
```

The implementation handles this naturally because the frequency counts match exactly. If the single values differ, the counts will not match and the function correctly returns `false`.

### Duplicate Values with Different Frequencies

This is the most common source of logical bugs.

For example:

```
target = [1,1,2]
arr = [1,2,2]
```

A naive set-based solution would incorrectly return `true` because both arrays contain `1` and `2`.

The frequency-map solution avoids this problem by tracking exact counts for every value.

### Arrays Already Equal

Some implementations may attempt unnecessary transformations even when the arrays already match.

For example:

```
target = [1,2,3]
arr = [1,2,3]
```

The algorithm simply compares frequencies and immediately determines the arrays are compatible. No special-case logic is needed.

### Completely Different Elements

If one array contains elements not present in the other, transformation is impossible.

Example:

```
target = [1,2,3]
arr = [4,5,6]
```

The frequency counts become non-zero after processing both arrays, correctly causing the algorithm to return `false`.
