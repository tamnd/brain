## Solution

## Problem Understanding

This problem asks us to maximize the **like-time coefficient** of dishes a chef prepares. Each dish has a satisfaction level, represented as an integer, which can be negative, zero, or positive. The chef can choose to cook some, all, or none of the dishes in any order. Each dish takes exactly one unit of time to prepare, and the like-time coefficient for a dish is the product of the time at which it is cooked (1-indexed) and its satisfaction level. The goal is to determine the maximum sum of like-time coefficients achievable.

The input `satisfaction` is an array of integers, where `satisfaction[i]` is the satisfaction level of the i-th dish. The output is a single integer, the maximum total like-time coefficient. Constraints indicate that the array length `n` is between 1 and 500, and satisfaction levels are in the range [-1000, 1000].

Key edge cases include arrays where all values are negative (where the optimal choice is to cook nothing), arrays with all positive values (where cooking all dishes in ascending order of satisfaction maximizes the sum), and arrays with mixed negative and positive values (where careful selection and ordering of dishes matter).

## Approaches

The brute-force approach involves generating all possible subsets of dishes and calculating the like-time coefficient sum for all possible permutations of each subset. While correct, this is computationally infeasible because for `n` dishes there are `2^n` subsets, and each subset requires `k!` permutations to consider all orders, resulting in exponential time complexity.

The key insight for an optimal solution is that **sorting the dishes by satisfaction level in ascending order and then considering which dishes to include from the end yields a greedy dynamic approach**. If we start from the highest satisfaction and iteratively add dishes to the beginning of the cooking sequence, we can track if adding the next dish increases the total sum. This works because adding a dish with negative satisfaction to the front only increases the sum if the cumulative total still improves.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n! * 2^n) | O(n) | Generate all subsets and permutations, then compute sum. Infeasible for n=500. |
| Optimal | O(n log n) | O(1) | Sort array, iteratively include dishes from the back, sum like-time coefficients greedily. |

## Algorithm Walkthrough

1. **Sort the array** of satisfaction in ascending order. This ensures that high satisfaction dishes are considered last, maximizing their multiplier effect if included later.
2. **Initialize two variables**, `total` and `current_sum`. `total` tracks the total like-time coefficient, and `current_sum` tracks the cumulative sum of selected dishes.
3. **Iterate from the end of the array backwards**. For each dish, calculate the potential new total if we include this dish at the beginning of the cooking sequence. Specifically, `new_sum = current_sum + dish_satisfaction`.
4. **Check if adding the dish improves the total**. If `new_sum` is greater than zero, prepend the dish to the cooking sequence and update `current_sum` and `total`.
5. **Stop adding dishes** when including the next dish would decrease the total like-time coefficient. This ensures we never include dishes that reduce the total sum.
6. **Return the total sum** as the maximum like-time coefficient.

Why it works: This greedy approach works because the sorted order guarantees that higher satisfaction dishes are multiplied by higher time indices. The running total sum ensures that adding a dish only occurs if it increases the cumulative like-time coefficient.

## Python Solution

```python
from typing import List

class Solution:
    def maxSatisfaction(self, satisfaction: List[int]) -> int:
        satisfaction.sort()
        total = 0
        current_sum = 0
        
        for s in reversed(satisfaction):
            if current_sum + s > 0:
                current_sum += s
                total += current_sum
            else:
                break
        
        return total
```

The implementation first sorts the satisfaction array. Then it iterates from the highest satisfaction backward, keeping track of a running sum. If including a dish increases the total, it updates `current_sum` and `total`. This matches the greedy algorithm explained above and avoids unnecessary computations.

## Go Solution

```go
import "sort"

func maxSatisfaction(satisfaction []int) int {
    sort.Ints(satisfaction)
    total := 0
    currentSum := 0

    for i := len(satisfaction) - 1; i >= 0; i-- {
        if currentSum + satisfaction[i] > 0 {
            currentSum += satisfaction[i]
            total += currentSum
        } else {
            break
        }
    }

    return total
}
```

In Go, we use `sort.Ints` to sort the array. Integer sums are safe because the maximum constraints guarantee values stay within the integer range. The algorithm mirrors the Python version exactly, iterating backward and updating totals based on the running sum.

## Worked Examples

### Example 1: `[-1, -8, 0, 5, -9]`

Sorted: `[-9, -8, -1, 0, 5]`

Iterating from the back:

| Dish | current_sum | total | Include? |
| --- | --- | --- | --- |
| 5 | 5 | 5 | Yes |
| 0 | 5 | 10 | Yes |
| -1 | 4 | 14 | Yes |
| -8 | -4 | 14 | No |
| -9 | -13 | 14 | No |

Maximum total = 14

### Example 2: `[4, 3, 2]`

| Dish | current_sum | total | Include? |
| --- | --- | --- | --- |
| 4 | 4 | 4 | Yes |
| 3 | 7 | 11 | Yes |
| 2 | 9 | 20 | Yes |

Maximum total = 20

### Example 3: `[-1, -4, -5]`

Sorted: `[-5, -4, -1]`

Adding any dish decreases the total, so total = 0

## Edge Cases

One important edge case is an array with all negative values. A naive algorithm might still attempt to include these and reduce the total, but our greedy algorithm breaks out of the loop when adding any dish decreases the cumulative sum.

A second edge case is a single-element array. Depending on whether the element is negative, zero, or positive, the output should be 0, 0, or the element itself. The algorithm handles this naturally because the initial `current_sum` is zero and the conditional check prevents negative contributions.

A third edge case is an array where the optimal solution is to include only some of the elements. For instance, adding a small negative value in front of positive values may reduce the overall like-time coefficient. Iterating from the back ensures that we only prepend a dish when it increases the total sum. This guarantees correctness even when negative dishes exist alongside positive ones.
