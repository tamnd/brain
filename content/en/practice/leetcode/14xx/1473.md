**Difficulty:** 🔴 Hard  
**Topics:** Array, Dynamic Programming

## Solution

## Problem Understanding

The problem asks us to paint a row of `m` houses with `n` possible colors such that the total number of neighborhoods equals `target`. A neighborhood is a contiguous sequence of houses with the same color. Some houses may already be painted, represented by a positive integer in the `houses` array, and these cannot be repainted. Unpainted houses are marked as `0`, and painting them incurs a cost from the `cost` matrix.

The input consists of:

- `houses`: an array where `houses[i]` is either `0` (unpainted) or a color from `1` to `n`.
- `cost`: an `m x n` matrix where `cost[i][j]` is the cost to paint house `i` with color `j+1`.
- `m`: number of houses.
- `n`: number of colors.
- `target`: the required number of neighborhoods.

The output is the minimum total painting cost that achieves exactly `target` neighborhoods. If it is impossible, we return `-1`.

Constraints tell us that `m` can go up to `100`, `n` up to `20`, and `cost[i][j]` up to `10^4`. The problem scale suggests that brute-force exploration of all possible colorings will be infeasible because the number of ways to color `m` houses is `n^m` in the worst case. Therefore, we need a dynamic programming approach.

Important edge cases include:

- All houses are already painted but the number of neighborhoods differs from `target`.
- Only one house or one color exists.
- `target` is larger than `m` (impossible) or smaller than the number of prepainted neighborhoods (impossible).

## Approaches

### Brute Force

The brute-force approach would be to try every possible coloring of unpainted houses and count the neighborhoods for each combination. For each valid coloring with exactly `target` neighborhoods, we compute the painting cost and keep track of the minimum.

This method is correct because it exhaustively evaluates all possibilities, but it is computationally infeasible. With `m` houses and `n` colors, the worst-case complexity is `O(n^m)`, which exceeds acceptable limits even for moderate values of `m`.

### Optimal Approach

The key insight is to use dynamic programming. We can define a state `dp[i][k][c]` representing the minimum cost to paint the first `i` houses, forming exactly `k` neighborhoods, with the `i-th` house painted with color `c`.

We transition based on whether the previous house had the same color or a different color. If the previous house's color is the same, the neighborhood count does not increase; otherwise, it increases by one. For already painted houses, we enforce that the color is fixed and the cost is `0`.

This reduces the problem complexity drastically by avoiding redundant calculations, allowing us to solve it in polynomial time relative to `m`, `n`, and `target`.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^m) | O(m) | Explore every possible painting combination |
| Dynamic Programming | O(m * target * n^2) | O(m * target * n) | Keeps track of minimum cost for each house, neighborhood count, and last color |

## Algorithm Walkthrough

1. Initialize a 3D DP array `dp[i][k][c]` with infinity to represent the minimum cost for painting the first `i` houses, forming `k` neighborhoods, ending with color `c`.
2. If the first house is unpainted, iterate over all colors to set the initial cost `dp[0][1][c] = cost[0][c]`. If already painted, set `dp[0][1][houses[0]] = 0`.
3. For each subsequent house `i` from `1` to `m-1`, consider each possible current color `cur_color`.
4. For each valid previous color `prev_color` and neighborhood count `k`, update the DP state:

- If `cur_color == prev_color`, `dp[i][k][cur_color] = min(dp[i][k][cur_color], dp[i-1][k][prev_color] + paint_cost)`
- If `cur_color != prev_color`, `dp[i][k+1][cur_color] = min(dp[i][k+1][cur_color], dp[i-1][k][prev_color] + paint_cost)`
5. After processing all houses, find the minimum cost among `dp[m-1][target][c]` for all colors `c`.
6. Return the minimum cost if it is finite; otherwise, return `-1`.

This works because at each house, the DP tracks all possible ways to form `k` neighborhoods ending with each color, so the minimum cost is always carried forward.

## Python Solution

```python
from typing import List

class Solution:
    def minCost(self, houses: List[int], cost: List[List[int]], m: int, n: int, target: int) -> int:
        INF = float('inf')
        dp = [[[INF] * n for _ in range(target + 1)] for _ in range(m)]

        # Initialize the first house
        if houses[0] != 0:
            dp[0][1][houses[0] - 1] = 0
        else:
            for c in range(n):
                dp[0][1][c] = cost[0][c]

        # Fill DP table
        for i in range(1, m):
            for k in range(1, target + 1):
                for c in range(n):
                    if houses[i] != 0 and houses[i] - 1 != c:
                        continue
                    paint_cost = 0 if houses[i] != 0 else cost[i][c]
                    for prev_c in range(n):
                        if prev_c == c:
                            if dp[i-1][k][prev_c] != INF:
                                dp[i][k][c] = min(dp[i][k][c], dp[i-1][k][prev_c] + paint_cost)
                        else:
                            if k > 1 and dp[i-1][k-1][prev_c] != INF:
                                dp[i][k][c] = min(dp[i][k][c], dp[i-1][k-1][prev_c] + paint_cost)

        ans = min(dp[m-1][target])
        return ans if ans != INF else -1
```

The DP table `dp[i][k][c]` tracks the minimum cost for every house, neighborhood count, and last color. We carefully handle pre-painted houses and only update relevant DP states. The final answer is the minimum cost for the last house achieving `target` neighborhoods.

## Go Solution

```go
import "math"

func minCost(houses []int, cost [][]int, m int, n int, target int) int {
    INF := math.MaxInt32
    dp := make([][][]int, m)
    for i := 0; i < m; i++ {
        dp[i] = make([][]int, target+1)
        for k := 0; k <= target; k++ {
            dp[i][k] = make([]int, n)
            for c := 0; c < n; c++ {
                dp[i][k][c] = INF
            }
        }
    }

    // Initialize first house
    if houses[0] != 0 {
        dp[0][1][houses[0]-1] = 0
    } else {
        for c := 0; c < n; c++ {
            dp[0][1][c] = cost[0][c]
        }
    }

    for i := 1; i < m; i++ {
        for k := 1; k <= target; k++ {
            for c := 0; c < n; c++ {
                if houses[i] != 0 && houses[i]-1 != c {
                    continue
                }
                paintCost := 0
                if houses[i] == 0 {
                    paintCost = cost[i][c]
                }
                for prevC := 0; prevC < n; prevC++ {
                    if prevC == c {
                        if dp[i-1][k][prevC] != INF {
                            if dp[i][k][c] > dp[i-1][k][prevC]+paintCost {
                                dp[i][k][c] = dp[i-1][k][prevC] + paintCost
                            }
                        }
                    } else {
                        if k > 1 && dp[i-1][k-1][prevC] != INF {
                            if dp[i][k][c] > dp[i-1][k-1][prevC]+paintCost {
                                dp[i][k][c] = dp[i-1][k-1][prevC] + paintCost
                            }
                        }
                    }
                }
            }
        }
    }

    ans := INF
    for c := 0; c < n; c++ {
        if dp[m-1][target][c] < ans {
            ans = dp[m-1][target][c]
        }
    }
    if ans == INF {
        return -1
    }
    return ans
}
```

In Go, integer overflow is handled by using `math.MaxInt32` as infinity. The logic mirrors the Python solution, with nested slices for DP storage.
