## Solution

## Problem Understanding

The problem gives us an integer array `arr` and asks us to count the number of triplets `(i, j, k)` such that:

- `0 <= i < j <= k < n`
- The XOR of the subarray from `i` to `j - 1` is equal to the XOR of the subarray from `j` to `k`

More formally:

- `a = arr[i] ^ arr[i + 1] ^ ... ^ arr[j - 1]`
- `b = arr[j] ^ arr[j + 1] ^ ... ^ arr[k]`

We need to count how many valid triplets satisfy `a == b`.

The key detail is that the split point `j` divides the range `[i, k]` into two consecutive parts. The first part ends at `j - 1`, and the second part starts at `j`.

The array length is at most `300`, which is relatively small. This means even quadratic or cubic algorithms may be acceptable, but we should still look for a cleaner and more efficient solution.

An important observation comes from XOR properties. If:

```
a == b
```

then:

```
a ^ b == 0
```

Since XORing identical values cancels them out.

That means:

```
(arr[i] ^ ... ^ arr[j - 1]) ^ (arr[j] ^ ... ^ arr[k]) == 0
```

which simplifies to:

```
arr[i] ^ arr[i + 1] ^ ... ^ arr[k] == 0
```

This is the crucial transformation. Instead of checking every possible split `j`, we only need to know whether the XOR of the entire range `[i, k]` is zero.

If the XOR from `i` to `k` is zero, then every possible split point `j` between them produces a valid triplet.

For a fixed pair `(i, k)`:

- valid `j` values are `i + 1` through `k`
- number of choices is `k - i`

This dramatically simplifies the problem.

Important edge cases include:

- Arrays of length `1`, where no triplets are possible
- Arrays where every element is the same
- Arrays with no zero XOR ranges
- Cases where many overlapping ranges produce zero XOR

The problem guarantees all numbers are positive integers, but XOR logic works regardless of sign.

## Approaches

### Brute Force Approach

The most direct approach is to try every possible triplet `(i, j, k)`.

For each triplet:

1. Compute XOR from `i` to `j - 1`
2. Compute XOR from `j` to `k`
3. Compare the two values
4. Increment the answer if they are equal

This works because it explicitly follows the problem definition.

However, there are three nested loops for choosing `i`, `j`, and `k`, and computing XOR repeatedly inside those loops can add another linear factor. Even with prefix XOR optimization, the brute-force structure still requires checking all triplets.

This leads to `O(n^3)` time complexity.

Although `n = 300` is small enough that cubic time may pass, we can do much better with XOR properties.

### Optimal Approach

The key observation is:

```
a == b
```

if and only if:

```
arr[i] ^ arr[i + 1] ^ ... ^ arr[k] == 0
```

Using prefix XOR:

```
prefix[x] = arr[0] ^ arr[1] ^ ... ^ arr[x - 1]
```

then XOR of subarray `[i, k]` becomes:

```
prefix[k + 1] ^ prefix[i]
```

This equals zero when:

```
prefix[k + 1] == prefix[i]
```

So instead of checking every `j`, we only need to find pairs `(i, k)` where the prefix XOR values match.

For every such pair:

```
number of valid j values = k - i
```

This reduces the problem to a quadratic scan over all `(i, k)` pairs.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n³) | O(1) | Checks every triplet directly |
| Optimal | O(n²) | O(n) | Uses prefix XOR and XOR properties |

## Algorithm Walkthrough

1. Create a prefix XOR array.

Define:

```
prefix[0] = 0
```

Then:

```
prefix[i + 1] = prefix[i] ^ arr[i]
```

This allows constant-time XOR queries for any subarray.
2. Initialize the answer variable.

We will accumulate the total number of valid triplets.
3. Iterate through all possible starting indices `i`.

For each `i`, we will examine every possible ending index `k`.
4. For each pair `(i, k)`, compute the XOR of the subarray `[i, k]`.

Using prefix XOR:

```
xor(i, k) = prefix[k + 1] ^ prefix[i]
```
5. If this XOR equals zero, then every split point between `i` and `k` is valid.

The valid `j` values are:

```
i + 1, i + 2, ..., k
```

Count of such values:

```
k - i
```
6. Add `k - i` to the answer.
7. Continue until all `(i, k)` pairs are processed.
8. Return the final answer.

### Why it works

The algorithm relies on the XOR identity:

```
x ^ x = 0
```

If:

```
a == b
```

then:

```
a ^ b = 0
```

Combining the two consecutive XOR ranges produces the XOR of the entire range `[i, k]`.

Therefore:

```
a == b
```

exactly when:

```
arr[i] ^ arr[i + 1] ^ ... ^ arr[k] == 0
```

Whenever this condition holds, every split position `j` between `i` and `k` creates valid equal XOR partitions, so adding `k - i` correctly counts all valid triplets.

## Python Solution

```python
from typing import List

class Solution:
    def countTriplets(self, arr: List[int]) -> int:
        n = len(arr)

        prefix = [0] * (n + 1)

        for i in range(n):
            prefix[i + 1] = prefix[i] ^ arr[i]

        result = 0

        for i in range(n):
            for k in range(i + 1, n):
                if prefix[i] == prefix[k + 1]:
                    result += (k - i)

        return result
```

The implementation starts by building the prefix XOR array. The value `prefix[i]` stores the XOR of all elements before index `i`.

This enables constant-time subarray XOR computation:

```python
xor(i, k) = prefix[k + 1] ^ prefix[i]
```

Next, the nested loops iterate over every possible pair `(i, k)`.

Instead of explicitly computing the XOR each time, the code uses the condition:

```python
prefix[i] == prefix[k + 1]
```

This means the XOR of the range `[i, k]` is zero.

Whenever this condition holds, all split positions between `i` and `k` are valid, so the code adds:

```python
k - i
```

to the result.

Finally, the total count is returned.

## Go Solution

```go
func countTriplets(arr []int) int {
	n := len(arr)

	prefix := make([]int, n+1)

	for i := 0; i < n; i++ {
		prefix[i+1] = prefix[i] ^ arr[i]
	}

	result := 0

	for i := 0; i < n; i++ {
		for k := i + 1; k < n; k++ {
			if prefix[i] == prefix[k+1] {
				result += (k - i)
			}
		}
	}

	return result
}
```

The Go implementation follows the same logic as the Python version.

The prefix XOR array is created using a slice of length `n + 1`. Nested loops examine every `(i, k)` pair, and matching prefix XOR values indicate a zero XOR subarray.

Go uses fixed integer types, but overflow is not a concern here because XOR operations stay within normal integer bounds and the maximum answer is relatively small for `n <= 300`.

An empty array is not possible because the constraints guarantee at least one element.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n²) | Two nested loops over all `(i, k)` pairs |
| Space | O(n) | Prefix XOR array stores `n + 1` values |

The algorithm processes every pair of indices exactly once. Since there are approximately `n² / 2` such pairs, the runtime is quadratic.

The extra memory comes from the prefix XOR array, which allows constant-time XOR range queries.

## Edge Cases

### Single Element Array

If the array contains only one element, no valid triplet can exist because we need indices satisfying:

```
i < j <= k
```

At least two elements are required to form two non-empty XOR ranges. The implementation naturally handles this because the inner loop never executes.

### Arrays With No Zero XOR Subarrays

Some arrays never produce a zero XOR across any range `[i, k]`.

For example:

```
[1, 2]
```

No pair satisfies:

```
prefix[i] == prefix[k + 1]
```

The algorithm correctly returns `0` because no contribution is added to the result.

### Arrays With Many Overlapping Valid Ranges

Arrays like:

```
[1,1,1,1,1]
```

produce many overlapping zero XOR subarrays.

A naive implementation can easily undercount or double count triplets in these situations. The formula:

```
k - i
```

ensures every valid split point `j` is counted exactly once for each valid range `[i, k]`.

### Arrays Containing Zero

If elements are zero, XOR behavior becomes especially important because:

```
0 ^ x = x
```

and:

```
0 ^ 0 = 0
```

Arrays like:

```
[0,0,0]
```

generate many valid triplets. The prefix XOR approach handles this naturally because repeated prefix XOR values immediately identify zero XOR ranges.
