## Solution

## Problem Understanding

LeetCode 1490, LeetCode Clone N-ary Tree, asks us to create a completely independent copy of an N-ary tree. A deep copy means that every node in the new tree must be newly allocated, and no node from the original tree can be reused in the cloned tree.

An N-ary tree is a tree where each node can have zero or more children. Unlike a binary tree, which has at most two children, an N-ary tree stores its children in a list. Each node contains two pieces of information:

- `val`, the integer value stored in the node
- `children`, a list containing references to child nodes

The input is the root node of the original tree. The output should be the root node of an entirely new tree that has the exact same structure and node values.

For example, if the original root has three children, then the cloned root must also have three cloned children in the same order. If a child node itself has descendants, those descendants must also be recursively cloned.

The constraints tell us several important things about the problem:

- The tree depth can be up to `1000`
- The total number of nodes can be up to `10^4`

This means the solution must scale linearly with the number of nodes. An algorithm worse than `O(n)` would become unnecessarily expensive for larger trees.

The problem guarantees that the input is a valid tree. This is important because trees do not contain cycles. Every node has exactly one parent except the root. Because of this guarantee, we do not need complex cycle detection logic for the tree version of the problem.

There are several edge cases that can easily cause bugs if not handled carefully:

- The input root may be `None`, meaning the tree is empty
- A node may have an empty `children` list
- The tree may be extremely deep, which increases recursive call depth
- A naive implementation might accidentally reuse child references from the original tree, creating a shallow copy instead of a deep copy

The key requirement is structural independence. Modifying the cloned tree after creation must not affect the original tree in any way.

## Approaches

### Brute Force Approach

A brute-force style solution would attempt to recreate the tree level by level while repeatedly searching for parent-child relationships in the original structure. For each node, we could scan the tree again to identify and rebuild all descendants.

This approach would eventually produce a correct clone because every node and edge would be reconstructed. However, repeatedly traversing portions of the tree introduces unnecessary repeated work.

For example, if we recursively search the tree again for every child relationship, the total work can grow toward `O(n^2)` in the worst case.

The brute-force strategy is therefore inefficient because tree cloning naturally lends itself to a direct traversal where each node only needs to be visited once.

### Optimal Approach

The key insight is that cloning a tree is fundamentally a traversal problem.

Each node only depends on the clones of its children. This means we can use Depth-First Search to recursively:

1. Create a clone of the current node
2. Recursively clone each child
3. Attach the cloned children to the cloned parent

Because every node in a tree is visited exactly once, we can build the cloned structure in linear time.

DFS works especially well here because the recursive structure of the algorithm naturally mirrors the recursive structure of the tree itself.

Even though the follow-up mentions graphs, the tree version is simpler because trees do not contain cycles. Therefore, we do not need a hash map to avoid revisiting nodes.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeatedly rescans parts of the tree while rebuilding |
| Optimal | O(n) | O(h) | DFS clones each node exactly once, where `h` is tree height |

## Algorithm Walkthrough

1. First, check whether the input root is `None`. If the tree is empty, there is nothing to clone, so we immediately return `None`.
2. Create a new node that copies the current node's value. This new node becomes the cloned version of the current original node.
3. Initialize the cloned node's `children` list as empty. We will gradually populate it with cloned child nodes.
4. Iterate through every child in the original node's `children` list. For each child:

1. Recursively call the cloning function on that child
2. Receive the cloned child subtree
3. Append the cloned child to the current cloned node's `children` list
5. After all children have been processed, return the cloned node upward to the caller.
6. The recursion continues until leaf nodes are reached. A leaf node has no children, so its cloned version simply returns immediately with an empty `children` list.
7. Eventually, the recursion reconstructs the entire tree from the bottom upward, producing a complete deep copy.

### Why it works

The algorithm works because every node is cloned exactly once, and every parent-child relationship in the original tree is recreated in the cloned tree.

The recursive invariant is:

- `cloneTree(node)` returns a fully independent deep copy of the subtree rooted at `node`

Since the invariant holds for leaf nodes and recursively holds for all child subtrees, it also holds for the entire tree.

## Python Solution

```python
"""
# Definition for a Node.
class Node:
    def __init__(self, val: Optional[int] = None, children: Optional[List['Node']] = None):
        self.val = val
        self.children = children if children is not None else []
"""

from typing import List, Optional

class Solution:
    def cloneTree(self, root: 'Node') -> 'Node':
        if root is None:
            return None

        cloned_node = Node(root.val)

        for child in root.children:
            cloned_child = self.cloneTree(child)
            cloned_node.children.append(cloned_child)

        return cloned_node
```

The implementation directly follows the recursive DFS strategy.

The first conditional handles the empty tree case. If `root` is `None`, we return `None` immediately because there is nothing to clone.

Next, we create a brand-new node using the current node's value. This guarantees that the cloned structure does not share references with the original tree.

The loop iterates through every child of the original node. For each child, the recursive call clones the entire subtree rooted at that child. The resulting cloned subtree root is appended to the cloned node's `children` list.

Finally, the cloned node is returned upward through the recursion stack.

Because recursion naturally mirrors tree structure, the implementation remains concise and easy to reason about.

## Go Solution

```go
/**
 * Definition for a Node.
 * type Node struct {
 *     Val int
 *     Children []*Node
 * }
 */

func cloneTree(root *Node) *Node {
	if root == nil {
		return nil
	}

	clonedNode := &Node{
		Val:      root.Val,
		Children: []*Node{},
	}

	for _, child := range root.Children {
		clonedChild := cloneTree(child)
		clonedNode.Children = append(clonedNode.Children, clonedChild)
	}

	return clonedNode
}
```

The Go implementation follows the exact same recursive logic as the Python solution.

The main language-specific difference is pointer handling. In Go, nodes are manipulated through pointers, so `nil` is used instead of Python's `None`.

The `Children` field is a slice of node pointers. We initialize it as an empty slice to ensure children can be appended safely.

Go does not require concerns about integer overflow here because node values are simply copied without arithmetic operations.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every node is visited and cloned exactly once |
| Space | O(h) | Recursive call stack stores at most one path from root to leaf |

The time complexity is linear because each node contributes constant work outside of recursive calls.

The space complexity depends on recursion depth. In the worst case of a highly skewed tree, the recursion stack may grow to `O(n)`. In a balanced tree, the recursion depth is much smaller.

## Edge Cases

### Empty Tree

The input may be `None`, meaning no tree exists at all. This is an important edge case because attempting to access `root.val` or iterate through `root.children` would immediately raise an error.

The implementation handles this safely with the initial base-case check:

```
if root is None:
    return None
```

This guarantees correct behavior for empty input.

### Nodes With No Children

Leaf nodes have empty `children` lists. A buggy implementation might accidentally leave `children` uninitialized or set it to `None`.

The implementation correctly creates a new node with an empty child list, then skips the loop because there are no children to process.

This preserves the exact structure of the original tree.

### Deeply Nested Tree

The tree depth may be as large as `1000`. Deep recursion can become problematic if the implementation is not carefully structured.

The DFS approach handles this naturally because each recursive call only stores information for one node on the current root-to-leaf path.

Although recursion depth is large, it remains within the intended constraints of the problem.

### Shared References Accidentally Creating a Shallow Copy

A common mistake is copying child references directly:

```
cloned.children = root.children
```

This does not create a deep copy. Both trees would share the same child nodes.

The implementation avoids this by recursively constructing entirely new child nodes before appending them to the cloned parent. As a result, the original and cloned trees are completely independent.
