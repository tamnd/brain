# LeetCode 1475 - Final Prices With a Special Discount in a Shop

## Problem Understanding

The problem gives us an array called `prices`, where each element represents the price of an item in a shop. For every item at index `i`, we must look to the right side of the array and find the first item whose price is less than or equal to the current item's price.

If such an item exists at index `j`, then the value `prices[j]` becomes the discount applied to `prices[i]`. The final price for that item becomes:

$$prices[i] - prices[j]$$

If there is no qualifying item to the right, then no discount is applied, and the final price remains unchanged.

The key detail is that we are not looking for the smallest discount or the best discount. We specifically need the first item to the right that satisfies:

$$prices[j] \le prices[i]$$

The output is another array where each position contains the final discounted price for the corresponding item.

For example, given:

```
prices = [8,4,6,2,3]
```

For the first item `8`, the first smaller or equal value to its right is `4`, so the final price becomes `8 - 4 = 4`.

For `6`, the first smaller or equal value to its right is `2`, so the final price becomes `6 - 2 = 4`.

The constraints are relatively small:

- `1 <= prices.length <= 500`
- `1 <= prices[i] <= 1000`

An array size of 500 means even an `O(n²)` solution would technically pass. However, this problem is designed to introduce the monotonic stack pattern, which improves the complexity to linear time.

Several edge cases are important:

- Arrays with only one element
- Arrays where no discounts exist
- Arrays where every item gets a discount
- Duplicate values, because equal values are allowed as discounts
- Strictly increasing arrays
- Strictly decreasing arrays

The problem guarantees that all prices are positive integers, so we never need to handle negative numbers or empty arrays.

## Approaches

### Brute Force Approach

The simplest solution is to process each item independently.

For every index `i`, we scan all indices to its right until we find the first value less than or equal to `prices[i]`.

If we find such a value, we subtract it from the current price and store the result. Otherwise, we keep the original price.

This approach is straightforward and easy to understand because it directly follows the problem statement.

For example:

```
prices = [8,4,6,2,3]
```

For index `0`:

- Check `4`
- Since `4 <= 8`, stop immediately
- Final price = `8 - 4`

For index `2`:

- Check `2`
- Since `2 <= 6`, stop
- Final price = `6 - 2`

Although this solution works correctly, it repeatedly scans the array. In the worst case, every element checks nearly all remaining elements.

For an array of size `n`, this leads to:

$$O(n^2)$$

time complexity.

### Optimal Approach, Monotonic Stack

The important observation is that we repeatedly search for the next smaller or equal element to the right.

This is a classic monotonic stack problem.

Instead of scanning forward repeatedly, we maintain a stack of indices whose discounts have not yet been found.

As we move from left to right:

- If the current price is smaller than or equal to the price represented by the stack top, then the current price becomes the discount for that earlier item.
- We pop that index from the stack and compute its final discounted price.
- We continue while the current price can serve as a discount for earlier items.

The stack remains monotonic increasing in terms of prices.

Each index is pushed once and popped once, which gives linear time complexity.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) extra space | Direct simulation of the problem statement |
| Optimal | O(n) | O(n) | Uses a monotonic stack to efficiently find next smaller or equal values |

## Algorithm Walkthrough

### Optimal Monotonic Stack Algorithm

1. Create a copy of the `prices` array called `answer`.

We use this array to store the final discounted prices. Initially, every item keeps its original price.
2. Initialize an empty stack.

The stack will store indices of items whose discounts have not yet been found.
3. Iterate through the array from left to right.

For every current index `i`, we compare `prices[i]` with the prices represented by indices in the stack.
4. While the stack is not empty and the current price is less than or equal to the price at the stack top:

- Let `topIndex` be the index at the top of the stack.
- The current price becomes the discount for that earlier item.
- Update:

```
answer[topIndex] = prices[topIndex] - prices[i]
```

- Remove the top index from the stack.
5. Push the current index onto the stack.

This means the current item is now waiting for its own future discount.
6. Continue until all elements are processed.
7. Return the `answer` array.

Any indices still remaining in the stack never found a valid discount, so their prices remain unchanged.

### Why it works

The stack always stores indices in increasing order of their prices relative to unresolved discounts.

When we encounter a current price that is smaller than or equal to the stack top price, we know it is the first qualifying discount for that earlier item because:

- We process elements from left to right
- Any earlier qualifying discount would already have been encountered

Thus, each item receives exactly the first valid discount to its right.

## Python Solution

```python
from typing import List

class Solution:
    def finalPrices(self, prices: List[int]) -> List[int]:
        answer = prices[:]
        stack = []

        for i, price in enumerate(prices):
            while stack and prices[stack[-1]] >= price:
                top_index = stack.pop()
                answer[top_index] = prices[top_index] - price

            stack.append(i)

        return answer
```

The implementation begins by creating a copy of the input array. This allows us to preserve the original prices while modifying the final answers.

The stack stores indices instead of values because we eventually need to update specific positions inside the result array.

As we iterate through the array, we compare the current price with the prices corresponding to indices stored in the stack.

Whenever the current price is less than or equal to the price at the top of the stack, we have found the first valid discount for that earlier item. We pop the index and update its discounted value.

After processing all applicable discounts, we push the current index onto the stack because it may receive a discount later.

At the end, any remaining indices in the stack did not find a smaller or equal value to their right, so their values remain unchanged.

## Go Solution

```go
func finalPrices(prices []int) []int {
    n := len(prices)

    answer := make([]int, n)
    copy(answer, prices)

    stack := []int{}

    for i, price := range prices {
        for len(stack) > 0 && prices[stack[len(stack)-1]] >= price {
            topIndex := stack[len(stack)-1]
            stack = stack[:len(stack)-1]

            answer[topIndex] = prices[topIndex] - price
        }

        stack = append(stack, i)
    }

    return answer
}
```

The Go implementation follows the same logic as the Python version.

Since slices in Go reference shared underlying arrays, we explicitly create a separate `answer` slice and copy the contents of `prices` into it.

The stack is implemented using a slice of integers. Push operations use `append`, and pop operations shrink the slice using slicing syntax.

Integer overflow is not a concern because the constraints are very small.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each index is pushed and popped from the stack at most once |
| Space | O(n) | The stack may contain up to all indices |

The linear time complexity comes from the fact that every element enters the stack once and leaves the stack once. Even though there is a nested `while` loop, the total number of pop operations across the entire algorithm is bounded by `n`.

The auxiliary space complexity is linear because the stack can grow to the size of the input array in the worst case, such as a strictly increasing array.

## Edge Cases

### Single Element Array

If the array contains only one item, there are no elements to the right, so no discount can ever be applied.

A buggy implementation might accidentally attempt stack operations or index access without checking bounds. The provided solution handles this naturally because the loop runs once, pushes the index, and finishes without modification.

### Strictly Increasing Prices

Consider:

```
[1,2,3,4,5]
```

No element has a smaller or equal value to its right.

This case is important because the stack grows to its maximum size. Incorrect implementations may accidentally apply discounts from larger future values instead of strictly smaller or equal values.

The monotonic stack implementation correctly leaves all values unchanged because the `while` condition never becomes true.

### Duplicate Values

Consider:

```
[2,2,2]
```

The problem explicitly allows equal values as discounts because the condition is:

```
prices[j] <= prices[i]
```

A common mistake is using a strict greater-than comparison instead of greater-than-or-equal.

The implementation uses:

```
prices[stack[-1]] >= price
```

which correctly handles equal values and produces:

```
[0,0,2]
```

### Strictly Decreasing Prices

Consider:

```
[5,4,3,2,1]
```

Every element immediately finds a discount from the next item.

This case stresses repeated stack popping. Incorrect stack management could lead to skipped updates or index errors.

The implementation correctly pops all qualifying indices and computes:

```
[1,1,1,1,1]
```
