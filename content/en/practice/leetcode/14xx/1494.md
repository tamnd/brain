**Difficulty:** 🔴 Hard  
**Topics:** Dynamic Programming, Bit Manipulation, Graph Theory, Bitmask

## Solution

## Problem Understanding

The problem describes a directed acyclic graph where each node represents a course and each directed edge represents a prerequisite dependency. If there is an edge from course `u` to course `v`, then `u` must be completed before `v` can be taken.

You are allowed to take multiple courses in the same semester, but there is an important restriction: you may take at most `k` courses per semester. Additionally, every course chosen in a semester must already have all its prerequisites completed in earlier semesters. You cannot satisfy a prerequisite and take the dependent course in the same semester.

The goal is to determine the minimum number of semesters required to finish all `n` courses.

The input consists of three parts:

- `n`, the total number of courses
- `relations`, a list of prerequisite pairs
- `k`, the maximum number of courses that may be taken in one semester

The output is a single integer representing the smallest number of semesters needed to complete every course.

The constraints are the key to understanding the intended solution:

- `n <= 15`
- The graph is guaranteed to be a DAG
- The graph can be dense
- We need an optimal minimum answer

The fact that `n` is at most `15` is extremely important. Normally, exponential algorithms are infeasible, but `2^15 = 32768`, which is manageable. This strongly suggests a bitmask dynamic programming solution where each subset of completed courses is represented as a bitmask.

Several edge cases are important:

- There may be no prerequisite relations at all
- `k` may equal `n`, allowing many courses per semester
- Some semesters may have fewer than `k` available courses
- Multiple independent prerequisite chains may exist
- A course may depend on many prerequisites
- The optimal solution may require choosing a subset of available courses strategically rather than greedily taking arbitrary courses

The problem guarantees the graph is acyclic and solvable, so we never need to handle impossible states.

## Approaches

### Brute Force Approach

A brute-force solution would simulate every possible sequence of semester choices.

At any state, we determine which courses are currently available, meaning all prerequisites have already been completed. From those available courses, we try every possible subset of size at most `k` and recursively continue to the next semester.

This approach is correct because it explores all valid schedules and therefore eventually finds the minimum number of semesters.

However, it is far too slow. At each semester, the number of possible subsets can become extremely large. Since the same completed-course states may be reached through different paths, the brute-force approach repeatedly recomputes identical subproblems.

The exponential branching factor makes this infeasible without memoization.

### Key Insight

The critical observation is that the only thing that matters at any point is which courses have already been completed.

We do not care about the exact sequence of semesters that produced that state. If two different paths lead to the same set of completed courses, the future decisions are identical.

This makes the problem a perfect candidate for dynamic programming over subsets.

We represent each completed-course configuration as a bitmask:

- Bit `i` is `1` if course `i` has already been completed
- Bit `i` is `0` otherwise

For each state:

1. Compute which courses are currently available
2. Generate all subsets of available courses of size at most `k`
3. Transition to the next state by marking those courses completed
4. Minimize the semester count

Because there are only `2^n` possible states and `n <= 15`, this becomes computationally feasible.

| --- | --- | --- | --- |

| Optimal Bitmask DP | `O(3^n)` | `O(2^n)` | Uses memoization over subsets and efficient subset iteration |

## Algorithm Walkthrough

### 1. Represent prerequisites using bitmasks

For every course, store a prerequisite mask.

If course `v` depends on course `u`, then set the bit corresponding to `u` inside `prereq[v]`.

For example:

```
Course 4 depends on 1 and 3

prereq[4] = 1010
```

This allows prerequisite checking using fast bit operations.

### 2. Define the DP state

We define:

```sql
dp(mask) = minimum semesters needed starting from completed courses = mask
```

Here:

- `mask` is a bitmask representing all completed courses
- If all courses are completed, the answer is `0`

The final answer is:

```
dp(0)
```

because initially no courses are completed.

### 3. Find all available courses

For every course not yet completed:

- Check whether all prerequisites are already satisfied
- This means:

```
(prereq[i] & mask) == prereq[i]
```

If true, the course is available this semester.

We collect all available courses into another bitmask called `available`.

### 4. Generate all valid course subsets

If the number of available courses is at most `k`, we take all of them immediately because there is no downside.

Otherwise, we must choose exactly `k` courses from the available set.

We iterate through all submasks of `available` using the classic subset enumeration trick:

```
subset = available
while subset:
    subset = (subset - 1) & available
```

For every subset whose bit count equals `k`, we consider taking those courses this semester.

### 5. Transition to the next state

If we choose subset `s`, then the next completed-course mask becomes:

```
next_mask = mask | s
```

The recurrence is:

```
dp(mask) = 1 + min(dp(next_mask))
```

over all valid subsets.

### 6. Memoize results

Since many paths can reach the same mask, we store computed results in a memoization cache.

This reduces the complexity dramatically.

### Why it works

The algorithm works because every DP state fully captures all information needed for future decisions. The only relevant information is which courses have already been completed.

For each state, the algorithm explores every valid semester choice and recursively computes the optimal continuation. Since all possibilities are considered and memoization prevents recomputation, the algorithm guarantees the minimum number of semesters.

## Python Solution

```python
from typing import List
from functools import lru_cache

class Solution:
    def minNumberOfSemesters(self, n: int, relations: List[List[int]], k: int) -> int:
        prereq = [0] * n

        for prev_course, next_course in relations:
            prereq[next_course - 1] |= (1 << (prev_course - 1))

        full_mask = (1 << n) - 1

        @lru_cache(None)
        def dp(mask: int) -> int:
            if mask == full_mask:
                return 0

            available = 0

            for course in range(n):
                if (mask & (1 << course)) == 0:
                    if (prereq[course] & mask) == prereq[course]:
                        available |= (1 << course)

            available_count = available.bit_count()

            if available_count <= k:
                return 1 + dp(mask | available)

            answer = float('inf')

            subset = available

            while subset:
                if subset.bit_count() == k:
                    answer = min(answer, 1 + dp(mask | subset))

                subset = (subset - 1) & available

            return answer

        return dp(0)
```

The implementation begins by constructing the prerequisite bitmask array. Each course stores a mask representing all courses that must be completed before it becomes available.

The variable `full_mask` represents the state where every course has been completed. Since each bit corresponds to a course, a mask of all `1`s indicates completion of all courses.

The recursive `dp` function computes the minimum semesters needed from the current completed-course state.

Inside the function, we first identify all available courses. A course is available if:

- It has not yet been completed
- All prerequisite bits are already present in the current mask

The available courses are stored in another bitmask.

If the number of available courses is less than or equal to `k`, we take all of them immediately because delaying any available course cannot improve the answer.

Otherwise, we enumerate all subsets of size exactly `k` and recursively compute the optimal future result.

Memoization ensures each state is solved only once.

## Go Solution

```go
package main

import (
	"math"
	"math/bits"
)

func minNumberOfSemesters(n int, relations [][]int, k int) int {
	prereq := make([]int, n)

	for _, relation := range relations {
		prev := relation[0] - 1
		next := relation[1] - 1

		prereq[next] |= (1 << prev)
	}

	fullMask := (1 << n) - 1

	memo := make(map[int]int)

	var dp func(mask int) int

	dp = func(mask int) int {
		if mask == fullMask {
			return 0
		}

		if value, exists := memo[mask]; exists {
			return value
		}

		available := 0

		for course := 0; course < n; course++ {
			if (mask&(1<<course)) == 0 &&
				(prereq[course]&mask) == prereq[course] {

				available |= (1 << course)
			}
		}

		if bits.OnesCount(uint(available)) <= k {
			result := 1 + dp(mask|available)
			memo[mask] = result
			return result
		}

		answer := math.MaxInt32

		for subset := available; subset > 0; subset = (subset - 1) & available {
			if bits.OnesCount(uint(subset)) == k {
				answer = min(answer, 1+dp(mask|subset))
			}
		}

		memo[mask] = answer
		return answer
	}

	return dp(0)
}

func min(a, b int) int {
	if a < b {
		return a
	}
	return b
}
```

The Go implementation follows the same logic as the Python version, but there are several language-specific differences.

Go does not provide built-in memoization decorators, so we use a `map[int]int` for caching DP results manually.

Bit counting is handled using the `math/bits` package through `bits.OnesCount`.

The recursive closure `dp` is declared using a function variable because Go requires recursive anonymous functions to be declared before assignment.

Go integers are fixed-width machine integers, but since `n <= 15`, bit operations are completely safe from overflow concerns.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | `O(3^n)` | Each state may enumerate subsets of available courses |
| Space | `O(2^n)` | Memoization stores one value per mask |

The complexity analysis comes from subset dynamic programming.

There are `2^n` possible masks. For each mask, we may iterate through subsets of the available set. Across all states, the total complexity becomes `O(3^n)`.

This is acceptable because `n <= 15`.

The space complexity is `O(2^n)` because we store one DP result for every possible completed-course state.

## Edge Cases

### No prerequisite relations

When `relations` is empty, every course is immediately available from the first semester.

A buggy implementation might still perform unnecessary dependency checks or fail to optimize the obvious case. The bitmask approach handles this naturally because every course has a prerequisite mask of `0`, meaning all courses are instantly available.

If `k >= n`, the algorithm correctly completes all courses in one semester.

### Long dependency chains

Consider:

```
1 -> 2 -> 3 -> 4 -> 5
```

Even if `k` is very large, only one course becomes available at a time.

A naive greedy implementation might incorrectly assume that larger `k` always reduces semesters dramatically. The DP correctly respects prerequisite ordering because a course only becomes available when all prerequisite bits are already present in the completed mask.

### More than k available courses

This is the most important edge case.

Suppose many courses are available simultaneously, but choosing the wrong subset delays future unlocks. A greedy algorithm that arbitrarily selects any `k` courses may produce a suboptimal answer.

The DP solution handles this correctly by exploring every subset of size `k` and choosing the minimum resulting semester count. This exhaustive exploration is exactly why the algorithm guarantees optimality.
