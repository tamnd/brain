## Solution

## Problem Understanding

This problem involves a network of `n` cities connected by `n - 1` roads, forming a tree structure. Each road has an initial direction, and the goal is to reorient some roads so that every city can reach city `0`, the capital. The roads are initially directed arbitrarily, and each connection is represented as `[a, b]`, meaning a one-way road from city `a` to city `b`. The task is to compute the **minimum number of roads that need to be reversed** to ensure all cities have a path to city `0`.

The input guarantees that it is always possible to reach city `0` after reorienting some roads, so we do not need to handle disconnected graphs or cycles. The constraints indicate that `n` can be as large as 50,000, so any solution must be linear or near-linear in time complexity. Key edge cases include cities already pointing toward city `0`, a single direct road to city `0`, and long chains that require multiple reversals.

## Approaches

A brute-force approach would attempt to explore all possible reorientations of edges and check connectivity to city `0`. This could involve recursively testing different directions for each road and counting the minimal changes required. While this approach is theoretically correct, it is completely infeasible for large inputs due to factorial growth in possibilities, making it unsuitable for `n` up to 50,000.

The key insight for an optimal solution is to treat the network as an undirected tree and perform a traversal starting from city `0`. For each edge, we can check if the existing direction is already correct (toward city `0`) or if it needs reversal. By marking edges as visited and recursively traversing, we only count edges that point away from city `0`. Using either Depth-First Search (DFS) or Breadth-First Search (BFS) ensures a linear traversal of all nodes and edges. This approach works efficiently because every city is connected, so we will touch each edge exactly once.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(n) | Check all possible edge reorientations and count reversals. Impractical for large n. |
| Optimal | O(n) | O(n) | DFS or BFS traversal from city 0; count edges that must be reversed. Linear due to tree structure. |

## Algorithm Walkthrough

1. Initialize an adjacency list to represent the graph. For each connection `[a, b]`, add `(b, 1)` to `a`'s neighbors and `(a, 0)` to `b`'s neighbors. Here, `1` indicates the edge goes **away** from the current node, and `0` indicates the edge goes **toward** the node. This encoding allows tracking which edges need reversal.
2. Create a visited set to track which cities have already been traversed. This prevents revisiting nodes and infinite loops.
3. Define a recursive DFS function (or a BFS loop) starting from city `0`. For each neighbor of the current city:

- Skip if the neighbor has been visited.
- Add the edge value (`1` or `0`) to the reversal count. If the edge points away from city `0` (value `1`), it counts as a required reversal.
- Recursively call DFS on the neighbor.
4. Sum the reversals across all traversed edges and return this count.

Why it works: The algorithm maintains the invariant that we traverse all cities exactly once starting from city `0`. By marking edges that point away from the traversal as needing reversal, we guarantee that after applying these reversals, every city will have a path to city `0`. Since the graph is a tree, every city has exactly one path to city `0`, so no edge is counted multiple times.

## Python Solution

```python
from typing import List, Tuple

class Solution:
    def minReorder(self, n: int, connections: List[List[int]]) -> int:
        from collections import defaultdict

        graph = defaultdict(list)
        for a, b in connections:
            graph[a].append((b, 1))  # edge a -> b, may need reversal
            graph[b].append((a, 0))  # edge b -> a, correct direction

        visited = set()

        def dfs(node: int) -> int:
            visited.add(node)
            reversals = 0
            for neighbor, needs_reversal in graph[node]:
                if neighbor not in visited:
                    reversals += needs_reversal
                    reversals += dfs(neighbor)
            return reversals

        return dfs(0)
```

The Python code constructs the adjacency list such that edges directed away from city `0` are marked with `1`. DFS traversal ensures each node is visited once, and the cumulative sum of `needs_reversal` flags gives the minimum number of edges to change. The `visited` set guarantees we do not traverse a node twice.

## Go Solution

```go
func minReorder(n int, connections [][]int) int {
    graph := make([][]struct {
        node int
        rev  int
    }, n)

    for _, conn := range connections {
        a, b := conn[0], conn[1]
        graph[a] = append(graph[a], struct {
            node int
            rev  int
        }{b, 1}) // edge a->b, may need reversal
        graph[b] = append(graph[b], struct {
            node int
            rev  int
        }{a, 0}) // edge b->a, correct
    }

    visited := make([]bool, n)

    var dfs func(int) int
    dfs = func(node int) int {
        visited[node] = true
        reversals := 0
        for _, neighbor := range graph[node] {
            if !visited[neighbor.node] {
                reversals += neighbor.rev
                reversals += dfs(neighbor.node)
            }
        }
        return reversals
    }

    return dfs(0)
}
```

The Go implementation mirrors the Python version. We define a struct for graph edges to store the neighbor and reversal flag. Boolean slices track visited nodes. Recursive DFS sums reversals for edges pointing away from the capital.
