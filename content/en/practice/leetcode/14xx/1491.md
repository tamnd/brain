## Solution

## Problem Understanding

The problem gives an array called `salary`, where each element represents the salary of an employee. Every salary value is unique, meaning no two employees have the same salary.

The task is to compute the average salary after removing the employee with the minimum salary and the employee with the maximum salary. In other words, we ignore the smallest and largest values in the array, then calculate the arithmetic mean of the remaining salaries.

For example, if the salaries are `[4000, 3000, 1000, 2000]`, the minimum salary is `1000` and the maximum salary is `4000`. After removing those two values, the remaining salaries are `[3000, 2000]`. Their average is `(3000 + 2000) / 2 = 2500`.

The constraints are small:

- The array length is between `3` and `100`
- Salaries range from `1000` to `10^6`
- All salaries are unique

These constraints tell us several important things. First, the array is guaranteed to contain at least three elements, which means that after removing the minimum and maximum salaries, there will always be at least one remaining salary to average. Second, the small input size means even less efficient solutions would still pass comfortably, but we should still aim for the cleanest and most optimal implementation.

A few edge cases are important to recognize upfront. The smallest valid input size is `3`, which means exactly one salary remains after excluding the minimum and maximum. Another important detail is that all salaries are unique, so there is never ambiguity about which values are the minimum and maximum. A naive implementation could accidentally divide by the wrong count or fail to exclude both extreme values correctly.

## Approaches

### Brute Force Approach

A straightforward way to solve this problem is to sort the array first. After sorting, the smallest salary will be at index `0`, and the largest salary will be at the last index. We can then iterate through the middle portion of the array, sum those values, and divide by the number of remaining employees.

This approach works because sorting places all salaries in ascending order, making it easy to identify and exclude the minimum and maximum values.

However, sorting requires `O(n log n)` time. While the constraints are small enough that this is perfectly acceptable, sorting is unnecessary because we only need the minimum value, maximum value, and total sum. We do not actually care about the relative ordering of the salaries.

### Optimal Approach

The key observation is that we can compute the answer in a single pass through the array.

Instead of sorting, we track three values while iterating:

- The total sum of all salaries
- The minimum salary seen so far
- The maximum salary seen so far

Once the traversal is complete, we subtract the minimum and maximum salaries from the total sum. The remaining value is the sum of all salaries that should be included in the average.

The number of remaining employees is `len(salary) - 2`, because exactly two salaries are excluded.

This avoids sorting entirely and reduces the time complexity to linear time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) | O(1) or O(n) | Sort the array, then average the middle elements |
| Optimal | O(n) | O(1) | Track sum, minimum, and maximum in one pass |

## Algorithm Walkthrough

1. Initialize three variables:

- `total_salary` to store the sum of all salaries
- `minimum_salary` to track the smallest salary
- `maximum_salary` to track the largest salary

These values allow us to compute the result without sorting the array.
2. Iterate through each salary in the array.

For every salary:

- Add it to `total_salary`
- Update `minimum_salary` if the current salary is smaller
- Update `maximum_salary` if the current salary is larger

By the end of the loop, we will know the total sum and the two salaries that must be excluded.
3. Compute the adjusted sum.

Subtract both the minimum and maximum salaries from the total:

`adjusted_sum = total_salary - minimum_salary - maximum_salary`
4. Compute how many salaries remain.

Since exactly two salaries are removed, the remaining count is:

`remaining_count = len(salary) - 2`
5. Return the average.

Divide the adjusted sum by the remaining count and return the result as a floating-point number.

### Why it works

The algorithm works because every salary contributes exactly once to the total sum. By subtracting the unique minimum and maximum salaries, we are left with precisely the salaries that should be averaged. Since the denominator correctly represents the number of remaining employees, the computed result is mathematically correct.

## Python Solution

```python
from typing import List

class Solution:
    def average(self, salary: List[int]) -> float:
        total_salary = 0
        minimum_salary = float("inf")
        maximum_salary = float("-inf")

        for current_salary in salary:
            total_salary += current_salary
            minimum_salary = min(minimum_salary, current_salary)
            maximum_salary = max(maximum_salary, current_salary)

        adjusted_sum = total_salary - minimum_salary - maximum_salary
        remaining_count = len(salary) - 2

        return adjusted_sum / remaining_count
```

The implementation follows the optimal algorithm directly.

The variable `total_salary` accumulates the sum of every salary in the array. The variables `minimum_salary` and `maximum_salary` track the smallest and largest values encountered during traversal.

The loop processes each salary exactly once. During each iteration, the code updates all three tracking variables.

After the loop completes, the code removes the minimum and maximum salaries from the total sum. The number of employees included in the average is `len(salary) - 2`, because exactly two salaries are excluded.

Finally, the function returns the floating-point division result.

## Go Solution

```go
func average(salary []int) float64 {
    totalSalary := 0
    minimumSalary := salary[0]
    maximumSalary := salary[0]

    for _, currentSalary := range salary {
        totalSalary += currentSalary

        if currentSalary < minimumSalary {
            minimumSalary = currentSalary
        }

        if currentSalary > maximumSalary {
            maximumSalary = currentSalary
        }
    }

    adjustedSum := totalSalary - minimumSalary - maximumSalary
    remainingCount := len(salary) - 2

    return float64(adjustedSum) / float64(remainingCount)
}
```

The Go implementation is nearly identical to the Python version, but there are a few language-specific differences.

Go does not have built-in infinity constants for integers, so the code initializes both `minimumSalary` and `maximumSalary` using the first element of the slice.

Go also requires explicit type conversion for floating-point division. Since integer division would truncate the result, both operands are converted to `float64` before division.

The problem guarantees at least three salaries, so accessing `salary[0]` is always safe.

## Edge Cases

One important edge case is the minimum valid input size of three salaries. In this scenario, removing the minimum and maximum salaries leaves exactly one salary remaining. A buggy implementation might accidentally divide by zero or incorrectly handle the remaining count. This implementation correctly computes `len(salary) - 2`, which becomes `1`, so the average is simply the remaining salary.

Another important case is when the salaries are already sorted in ascending or descending order. Some implementations mistakenly assume unsorted input or rely on positional logic without proper checks. Since this solution explicitly tracks the minimum and maximum values during traversal, the original ordering has no effect on correctness.

A third edge case involves very large salary values near the upper constraint limit of `10^6`. If the implementation used inappropriate numeric types in some languages, integer overflow could become a concern. In Python, integers automatically scale to arbitrary size. In Go, the constraints are small enough that standard integer types safely handle all calculations. The final division is also explicitly converted to floating-point arithmetic to avoid truncation.
