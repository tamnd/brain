## Solution

## Problem Understanding

In this problem, we are given a string called `path` that represents movements on a two dimensional grid. Each character in the string corresponds to moving one unit in one of the four cardinal directions:

- `'N'` means move north, increasing the y-coordinate
- `'S'` means move south, decreasing the y-coordinate
- `'E'` means move east, increasing the x-coordinate
- `'W'` means move west, decreasing the x-coordinate

The walk always starts at the origin, which is the coordinate `(0, 0)`.

The goal is to determine whether the path ever visits the same location more than once. If at any point during the walk we arrive at a coordinate that has already been visited earlier, the path is considered to cross itself, and we must return `true`. Otherwise, we return `false`.

The important detail is that revisiting _any_ previously visited coordinate counts as a crossing, including returning to the starting point.

For example, with the input `"NES"`:

- Start at `(0, 0)`
- Move north to `(0, 1)`
- Move east to `(1, 1)`
- Move south to `(1, 0)`

Every position is unique, so the answer is `false`.

For the input `"NESWW"`:

- Start at `(0, 0)`
- Move north to `(0, 1)`
- Move east to `(1, 1)`
- Move south to `(1, 0)`
- Move west to `(0, 0)`
- Move west to `(-1, 0)`

The coordinate `(0, 0)` is visited twice, so the answer is `true`.

The constraints tell us that the path length can be as large as `10^4`. This is small enough for linear traversal, but large enough that inefficient repeated scanning approaches could become unnecessarily slow.

Several edge cases are important:

- A path that immediately returns to the origin, such as `"NS"`
- A path that repeatedly revisits locations
- A path that never crosses itself
- Very short paths of length `1`
- Long paths where the crossing happens near the end

The problem guarantees that every character is valid, so we do not need to handle invalid directions.

## Approaches

### Brute Force Approach

A straightforward solution is to simulate the movement step by step while storing every visited coordinate in a list. After each move, we scan the entire list of previously visited positions to see whether the current coordinate already exists.

This works because every time we move, we explicitly compare the new position against all earlier positions. If we find a match, we know the path crosses itself.

However, this approach becomes inefficient as the path grows longer. If there are `n` moves, then for each move we may scan up to `n` previously visited positions. This leads to quadratic time complexity.

With `n = 10^4`, an `O(n^2)` solution is unnecessarily slow compared to what is possible.

### Optimal Approach

The key observation is that we only need fast membership checking for previously visited coordinates.

A hash set is ideal for this task because it provides average `O(1)` lookup time. Instead of storing coordinates in a list and scanning repeatedly, we store each visited position in a set.

The algorithm works as follows:

1. Start at `(0, 0)`
2. Add the starting coordinate to the visited set
3. Process each movement character
4. Update the current coordinate
5. Check whether the new coordinate already exists in the set
6. If it does, return `true`
7. Otherwise, add it to the set and continue

This reduces the total runtime to linear time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Stores visited positions in a list and scans repeatedly |
| Optimal | O(n) | O(n) | Uses a hash set for constant time lookup |

## Algorithm Walkthrough

1. Initialize the current position as `(0, 0)` because the walk always starts at the origin.
2. Create a hash set called `visited` and insert the starting coordinate into it. This is important because revisiting the origin later counts as a path crossing.
3. Iterate through each character in the input string `path`.
4. For every direction character:

- If the character is `'N'`, increase `y` by `1`
- If the character is `'S'`, decrease `y` by `1`
- If the character is `'E'`, increase `x` by `1`
- If the character is `'W'`, decrease `x` by `1`
5. After updating the coordinates, create a tuple or coordinate representation for the new position.
6. Check whether this coordinate already exists in the `visited` set.

- If it exists, immediately return `true` because the path has crossed itself.
- Otherwise, insert the coordinate into the set.
7. If the loop finishes without finding any repeated coordinate, return `false`.

The hash set is the critical data structure because it allows efficient detection of repeated positions without repeatedly scanning all previous coordinates.

### Why it works

The algorithm maintains the invariant that the `visited` set always contains every coordinate visited before the current step. After each move, we check whether the new coordinate is already present in the set. If it is, then the current location has been visited earlier, which exactly matches the definition of a path crossing. If the coordinate is not present, we safely add it and continue. Since every move is processed exactly once, the algorithm correctly determines whether the path crosses itself.

## Python Solution

```python
class Solution:
    def isPathCrossing(self, path: str) -> bool:
        x, y = 0, 0
        
        visited = {(0, 0)}
        
        for direction in path:
            if direction == 'N':
                y += 1
            elif direction == 'S':
                y -= 1
            elif direction == 'E':
                x += 1
            else:  # direction == 'W'
                x -= 1
            
            current_position = (x, y)
            
            if current_position in visited:
                return True
            
            visited.add(current_position)
        
        return False
```

The implementation starts by initializing the current coordinates `x` and `y` to zero, representing the origin.

A set called `visited` stores all previously visited coordinates. The origin is inserted immediately because the starting position counts as visited before any movement occurs.

The loop processes one direction at a time. Depending on the direction character, the code updates either the x-coordinate or y-coordinate.

After moving, the new coordinate is stored in `current_position`. The code checks whether this coordinate already exists in the set. If it does, the function immediately returns `True`.

If the coordinate has not been visited before, it is added to the set, and the traversal continues.

If the loop finishes without detecting a repeated coordinate, the function returns `False`.

## Go Solution

```go
func isPathCrossing(path string) bool {
    x, y := 0, 0

    visited := make(map[[2]int]bool)
    visited[[2]int{0, 0}] = true

    for _, direction := range path {
        switch direction {
        case 'N':
            y++
        case 'S':
            y--
        case 'E':
            x++
        case 'W':
            x--
        }

        position := [2]int{x, y}

        if visited[position] {
            return true
        }

        visited[position] = true
    }

    return false
}
```

The Go implementation follows the same logic as the Python solution, but uses a `map[[2]int]bool` to represent the visited set.

Since Go does not have a built in set type, the map keys store coordinate pairs as fixed size arrays. Each coordinate is represented as `[2]int{x, y}`.

The `switch` statement cleanly handles movement updates for each direction character.

There are no overflow concerns because the coordinates can grow at most by `10^4`, which easily fits inside Go's integer type.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each movement is processed once, and set operations are average O(1) |
| Space | O(n) | In the worst case, every visited coordinate is unique |

The runtime is linear because the algorithm iterates through the path exactly once. Each membership check and insertion into the hash set takes constant average time.

The space complexity is also linear because, in the worst case, the path never crosses itself and every coordinate must be stored in the visited set.

## Edge Cases

One important edge case is immediately returning to the origin, such as `"NS"` or `"EW"`. A common mistake is forgetting to mark the starting coordinate `(0, 0)` as visited before processing movements. If the origin is not added initially, the algorithm would fail to detect these crossings. The implementation correctly handles this by inserting `(0, 0)` into the set before the loop begins.

Another important case is a path that never crosses itself, such as `"NNNEEE"`. In this situation, every coordinate is unique. The algorithm correctly processes all movements and finishes the loop without finding a repeated position, ultimately returning `False`.

A third important edge case involves repeated crossings later in the traversal, such as `"NNNNSSSS"`. The path moves upward several times and then retraces its steps downward. The algorithm detects the first repeated coordinate immediately and exits early, avoiding unnecessary additional processing.

A final subtle edge case is a path of length one. Since a single movement can never revisit a coordinate, the answer must always be `False`. The implementation naturally handles this because only one new coordinate is visited, and it cannot already exist in the set.
