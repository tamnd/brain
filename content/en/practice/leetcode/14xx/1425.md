## Solution

## Problem Understanding

The problem asks us to find the maximum possible sum of a non-empty subsequence of the array `nums`, with one important restriction. If two elements are consecutive in the chosen subsequence, then their original indices in the array cannot differ by more than `k`.

This means we are not allowed to arbitrarily skip large portions of the array between selected elements. If we choose `nums[i]` and later choose `nums[j]`, where `i < j`, then we must satisfy:

$$j - i \le k$$

The subsequence does not need to be contiguous, but the gap between adjacent chosen elements is limited.

For example, with:

```
nums = [10,2,-10,5,20], k = 2
```

we can choose:

```
[10, 2, 5, 20]
```

because each jump is within distance `2`.

The sum becomes:

```
10 + 2 + 5 + 20 = 37
```

which is the maximum possible answer.

The array length can be as large as `10^5`, which immediately tells us that quadratic dynamic programming solutions will be too slow. We need something close to linear time.

Another important detail is that the subsequence must be non-empty. If every number is negative, we cannot return `0`. Instead, we must choose the largest single element.

The major edge cases include:

- Arrays containing only negative numbers
- Very large `k`, where almost every previous element becomes reachable
- Very small `k`, especially `k = 1`
- Situations where extending a subsequence actually decreases the total sum
- Arrays with alternating positive and negative values

These cases are important because a naive greedy strategy can easily fail.

## Approaches

### Brute Force Dynamic Programming

A straightforward dynamic programming solution defines:

$$dp[i]$$

as the maximum subsequence sum ending at index `i`.

To compute `dp[i]`, we look at every previous index `j` such that:

$$i - j \le k$$

and try extending the subsequence ending at `j`.

The recurrence becomes:

$$dp[i] = nums[i] + \max(0, dp[j])$$

for all valid `j`.

This works because:

- If a previous subsequence sum is positive, extending it helps
- If all previous sums are negative, we start a new subsequence at `i`

The issue is performance. For every index, we may scan up to `k` previous elements.

In the worst case:

```
n = 100000
k = 100000
```

the complexity becomes:

$$O(nk) = O(n^2)$$

which is far too slow.

### Optimal Monotonic Queue Solution

The key observation is that for each index `i`, we only need the maximum `dp` value from the previous `k` positions.

Instead of repeatedly scanning the window, we can maintain the maximum efficiently using a monotonic deque.

The deque stores indices in decreasing order of `dp` values. This gives us:

- The maximum valid `dp` value at the front
- Efficient insertion and removal in amortized `O(1)` time

This transforms the solution into a linear-time dynamic programming algorithm.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(nk) | O(n) | Checks every previous reachable state |
| Optimal | O(n) | O(n) | Uses DP with a monotonic deque |

## Algorithm Walkthrough

### Step 1: Define the DP State

Let:

$$dp[i]$$

represent the maximum constrained subsequence sum ending exactly at index `i`.

The final answer is:

$$\max(dp[i])$$

because the best subsequence may end anywhere.

### Step 2: Derive the Transition

At index `i`, we have two choices:

1. Start a new subsequence using only `nums[i]`
2. Extend the best valid subsequence from the previous `k` indices

The recurrence is:

$$dp[i] = nums[i] + \max(0, \max(dp[j]))$$

where:

$$i-k \le j < i$$

If the best previous sum is negative, we ignore it and start fresh.

### Step 3: Maintain the Sliding Maximum

The expensive part is repeatedly finding:

$$\max(dp[j])$$

inside the sliding window.

We use a deque that stores indices in decreasing order of `dp` values.

The deque satisfies two invariants:

1. Indices are within the last `k` positions
2. `dp` values are monotonically decreasing

This means:

- The front always contains the maximum `dp`
- Smaller values behind a larger value are useless and removed

### Step 4: Remove Expired Indices

Before processing index `i`, remove indices from the front if:

$$deque[0] < i-k$$

because they are no longer reachable.

### Step 5: Compute the Current DP Value

The maximum valid previous sum is at the front of the deque.

So:

```
best_previous = dp[deque[0]]
```

if the deque is non-empty.

Then:

```
dp[i] = nums[i] + max(0, best_previous)
```

### Step 6: Maintain Monotonic Order

While the current `dp[i]` is greater than or equal to the `dp` value at the back of the deque, remove the back element.

Those smaller values will never be useful again because:

- The current value is larger
- The current index is more recent

Then append the current index.

### Step 7: Track the Global Maximum

Update the answer after each iteration.

## Why it works

The deque always contains candidate indices with the largest `dp` values within the valid window of size `k`.

Because smaller `dp` values behind a larger one can never become optimal later, removing them preserves correctness while improving efficiency.

At every step, the front of the deque gives the optimal subsequence sum that can legally connect to the current index. Therefore, the DP recurrence always uses the best possible transition.

## Python Solution

```python
from collections import deque
from typing import List

class Solution:
    def constrainedSubsetSum(self, nums: List[int], k: int) -> int:
        n = len(nums)

        dp = [0] * n
        max_sum = nums[0]

        dq = deque()

        for i in range(n):
            # Remove indices outside the window
            while dq and dq[0] < i - k:
                dq.popleft()

            # Best previous subsequence sum
            best_previous = dp[dq[0]] if dq else 0

            dp[i] = nums[i] + max(0, best_previous)

            max_sum = max(max_sum, dp[i])

            # Maintain decreasing order in deque
            while dq and dp[dq[-1]] <= dp[i]:
                dq.pop()

            dq.append(i)

        return max_sum
```

The implementation follows the dynamic programming recurrence directly.

The `dp` array stores the best subsequence sum ending at each position. The deque stores indices whose `dp` values are useful candidates for future transitions.

The first `while` loop removes expired indices that are farther than `k` away.

The current best reachable subsequence is always at the front of the deque. If extending it would reduce the sum, we ignore it using `max(0, best_previous)`.

The second `while` loop maintains the deque in decreasing order of `dp` values. Any smaller value behind the current one becomes irrelevant because the current value is both larger and more recent.

The algorithm processes each index once, and each index enters and leaves the deque at most one time.

## Go Solution

```go
func constrainedSubsetSum(nums []int, k int) int {
	n := len(nums)

	dp := make([]int, n)
	maxSum := nums[0]

	deque := make([]int, 0)

	for i := 0; i < n; i++ {
		// Remove indices outside the window
		if len(deque) > 0 && deque[0] < i-k {
			deque = deque[1:]
		}

		bestPrevious := 0
		if len(deque) > 0 {
			bestPrevious = dp[deque[0]]
		}

		if bestPrevious > 0 {
			dp[i] = nums[i] + bestPrevious
		} else {
			dp[i] = nums[i]
		}

		if dp[i] > maxSum {
			maxSum = dp[i]
		}

		// Maintain decreasing order
		for len(deque) > 0 && dp[deque[len(deque)-1]] <= dp[i] {
			deque = deque[:len(deque)-1]
		}

		deque = append(deque, i)
	}

	return maxSum
}
```

The Go implementation uses a slice as a deque. Removing from the front is handled with:

```go
deque = deque[1:]
```

while removing from the back uses slice truncation.

Go integers are sufficient because the maximum possible sum fits safely within 32-bit signed integer range:

```go
100000 * 10000 = 10^9
```

which is below the Go `int` limit on modern platforms.

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each index enters and leaves the deque at most once |

Although there are nested `while` loops, each element is inserted and removed from the deque only one time. Therefore, the total deque work across the entire algorithm is linear.

## Edge Cases

### All Numbers Are Negative

This is one of the most important edge cases because many maximum-subarray-style algorithms incorrectly return `0`.

For example:

```
nums = [-5,-2,-8]
```

The answer must be:

```
-2
```

because the subsequence must be non-empty.

The implementation handles this correctly by initializing:

```
max_sum = nums[0]
```

and only extending previous subsequences if their contribution is positive.

### Very Large k

When:

```
k >= len(nums)
```

almost every previous index becomes reachable.

A naive implementation might still repeatedly scan the entire prefix, causing quadratic performance.

The monotonic deque avoids this issue because the maximum candidate is always available in constant time.

### Small k, Especially k = 1

When `k = 1`, the subsequence behaves almost like a contiguous subarray because we can only move one step forward at a time.

This case can expose bugs in window maintenance logic.

The implementation correctly removes expired indices using:

```
while dq and dq[0] < i - k:
```

which guarantees only valid transitions remain available.
