## Solution

## Problem Understanding

The problem asks us to implement a simplified HTML entity parser. In HTML, certain special characters cannot always appear directly in text, so they are represented using entity strings such as `&amp;` for `&` or `&quot;` for `"`. The task is to scan the input string and replace every valid HTML entity from a predefined set with its corresponding character.

The input is a single string `text`, and the output should be another string where every recognized entity has been converted into its actual symbol. Only six entities are considered valid in this problem:

| Entity | Replacement |
| --- | --- |
| `&quot;` | `"` |
| `&apos;` | `'` |
| `&amp;` | `&` |
| `&gt;` | `>` |
| `&lt;` | `<` |
| `&frasl;` | `/` |

An important detail is that not every substring beginning with `&` is a valid entity. For example, `&ambassador;` should remain unchanged because it is not one of the six recognized entities.

The constraint `1 <= text.length <= 10^5` tells us the input can be quite large. This immediately suggests that inefficient repeated scanning or repeated string rebuilding could become too slow. We should aim for a linear time solution, or something very close to it.

Several edge cases are important:

- A substring may begin with `&` but not form a valid entity.
- Entities may appear adjacent to one another.
- The string may contain ordinary characters mixed with entities.
- An incomplete entity such as `&amp` without the trailing semicolon should not be replaced.
- Nested replacement behavior does not occur. Once text is parsed left to right, only exact entity matches are replaced.

For example, `&amp;gt;` becomes `&gt;`, not `>`.

## Approaches

### Brute Force Approach

A straightforward solution is to repeatedly scan the string for every known entity and replace it with its corresponding character using built-in string replacement operations.

For example, we could execute:

- Replace all `&quot;` with `"`
- Replace all `&apos;` with `'`
- Replace all `&amp;` with `&`
- And so on for all six entities

This approach works because the number of entities is fixed and small. Every valid occurrence will eventually be replaced.

However, this method is not ideal for large inputs. Each replacement operation scans the entire string, which means we repeatedly traverse potentially very large text. Since there are six entities, the total runtime becomes proportional to roughly six full passes over the string. While still acceptable in practice for this constraint, it is less elegant and less efficient than a single-pass parser.

A more subtle issue is replacement ordering. If replacements are done in the wrong order, earlier substitutions could accidentally create new entity-like sequences.

### Optimal Approach

The better solution is to process the string in a single left-to-right pass.

The key observation is that every valid entity:

- Begins with `&`
- Ends with `;`
- Has limited maximum length
- Comes from a small predefined set

So whenever we encounter `&`, we can try to match one of the known entities starting at that position. If a match exists, we append the replacement character and skip past the entity. Otherwise, we treat the current character as normal text.

A hash map is ideal here because it provides constant-time lookup from entity string to replacement character.

This gives us a clean linear-time parser.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(6n) | O(n) | Multiple full-string replacement passes |
| Optimal | O(n) | O(n) | Single-pass parser with hash map lookup |

## Algorithm Walkthrough

1. Create a hash map that stores every valid HTML entity and its corresponding replacement character.

The map allows constant-time lookup when we detect a possible entity. Since there are only six valid entities, the map is very small and efficient.
2. Initialize an empty result list.

Building strings character by character directly is inefficient in many languages because strings are immutable. Using a list or slice lets us append efficiently and join at the end.
3. Traverse the input string using an index `i`.

We process the string one character at a time from left to right.
4. If the current character is not `&`, append it directly to the result.

Ordinary characters do not need any special handling.
5. If the current character is `&`, attempt to match a valid entity.

Since all entities end with `;`, search forward until either:

- A semicolon is found
- The maximum entity length is exceeded
6. Extract the candidate substring.

For example, if the text contains `&amp;`, the candidate becomes exactly `&amp;`.
7. Check whether the candidate exists in the hash map.

- If it exists, append the mapped replacement character and move the index past the entity.
- If it does not exist, append the current `&` character as normal text and continue scanning.
8. Continue until the entire string has been processed.
9. Join the accumulated characters into the final output string.

### Why it works

The algorithm always processes characters in left-to-right order and only replaces substrings that exactly match one of the six valid entities. Every character is either copied directly or replaced by its mapped symbol exactly once. Because valid entities are uniquely identifiable by their full string form, the parser cannot incorrectly transform unrelated text.

## Python Solution

```python
class Solution:
    def entityParser(self, text: str) -> str:
        entity_map = {
            "&quot;": '"',
            "&apos;": "'",
            "&amp;": "&",
            "&gt;": ">",
            "&lt;": "<",
            "&frasl;": "/"
        }

        result = []
        n = len(text)
        i = 0

        while i < n:
            if text[i] == '&':
                semicolon_index = text.find(';', i)

                if semicolon_index != -1:
                    candidate = text[i:semicolon_index + 1]

                    if candidate in entity_map:
                        result.append(entity_map[candidate])
                        i = semicolon_index + 1
                        continue

            result.append(text[i])
            i += 1

        return ''.join(result)
```

The implementation begins by constructing a dictionary that maps entity strings to their replacement characters. This gives constant-time lookup for every potential entity.

The `result` list stores the output incrementally. This avoids repeated string concatenation, which would otherwise create unnecessary copies.

The main loop processes the string using index `i`. Whenever the current character is not `&`, the character is appended directly.

When `&` is encountered, the code searches for the next semicolon using `find`. If a semicolon exists, the substring from `&` through `;` becomes the candidate entity.

The candidate is checked against the dictionary. If it exists, the replacement character is appended and the index jumps forward past the entity. Otherwise, the current character is treated normally.

Finally, all accumulated characters are joined into the final parsed string.

## Go Solution

```go
func entityParser(text string) string {
	entityMap := map[string]string{
		"&quot;":  "\"",
		"&apos;":  "'",
		"&amp;":   "&",
		"&gt;":    ">",
		"&lt;":    "<",
		"&frasl;": "/",
	}

	result := make([]byte, 0, len(text))

	for i := 0; i < len(text); {
		if text[i] == '&' {
			semicolonIndex := -1

			for j := i; j < len(text); j++ {
				if text[j] == ';' {
					semicolonIndex = j
					break
				}
			}

			if semicolonIndex != -1 {
				candidate := text[i : semicolonIndex+1]

				if replacement, exists := entityMap[candidate]; exists {
					result = append(result, replacement...)
					i = semicolonIndex + 1
					continue
				}
			}
		}

		result = append(result, text[i])
		i++
	}

	return string(result)
}
```

The Go implementation follows the same logic as the Python version, but uses Go-specific constructs.

Instead of a Python list of strings, we use a byte slice for efficient output construction. Since all characters involved are ASCII, byte-level processing is sufficient.

Go does not have a built-in equivalent to Python's `find` method on strings, so we manually scan forward to locate the semicolon.

Map lookup in Go returns both the value and a boolean indicating whether the key exists. This allows clean validation of candidate entities.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each character is processed at most a constant number of times |
| Space | O(n) | Output storage requires linear space |

The runtime is linear because the parser scans the string from left to right exactly once. Even when searching for semicolons, the total work remains proportional to the input size because indices only move forward.

The space complexity is linear because the output string may be as large as the input string.

## Edge Cases

### Invalid Entity Names

A common mistake is assuming every substring beginning with `&` and ending with `;` is valid. For example, `&ambassador;` resembles an entity but should remain unchanged. The implementation handles this safely by checking the candidate substring against the hash map before replacing anything.

### Missing Semicolon

An incomplete entity such as `&amp` does not qualify as a valid HTML entity in this problem. A naive parser might accidentally replace partial matches. The implementation explicitly searches for a semicolon before attempting any lookup, so incomplete entities remain untouched.

### Adjacent Entities

Strings such as `&lt;&gt;&amp;` contain multiple entities directly next to each other. Some parsers incorrectly skip characters after replacement. This implementation correctly advances the index to the character immediately after the matched entity, allowing the next entity to be processed normally.

### Recursive-Looking Replacements

The input `&amp;gt;` is especially tricky. After replacing `&amp;`, the intermediate result becomes `&gt;`, which itself is another valid entity. However, the problem expects only a single left-to-right parse pass. The implementation satisfies this requirement because it never reparses already-generated output.
