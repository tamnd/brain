# LeetCode 1413 - Minimum Value to Get Positive Step by Step Sum

## Problem Understanding

The problem gives an integer array `nums` and asks us to choose the smallest possible positive integer called `startValue`.

We process the array from left to right while maintaining a running sum. Initially, the running sum starts as `startValue`. At every step, we add the current array element to the running sum.

The key requirement is that the running sum must never drop below `1` at any point during the traversal.

We need to return the minimum positive starting value that guarantees this condition.

For example, if:

```
nums = [-3, 2, -3, 4, 2]
```

and we choose `startValue = 4`, the running sums become:

```
4 + (-3) = 1
1 + 2 = 3
3 + (-3) = 0
```

At this point the sum becomes `0`, which violates the requirement because the running sum must always stay at least `1`.

However, if we choose `startValue = 5`, every intermediate sum stays positive.

The constraints are small:

- `1 <= nums.length <= 100`
- `-100 <= nums[i] <= 100`

This means even less efficient approaches could technically pass, but the problem is designed to test understanding of prefix sums and cumulative minimum tracking.

An important observation is that we do not care about the final sum. We only care about the minimum value reached during the running process.

Several edge cases are important:

- Arrays containing only positive numbers, where the answer is simply `1`
- Arrays with large negative prefixes, which require a larger starting value
- Arrays where the minimum prefix sum occurs in the middle instead of the beginning
- Arrays whose running sum becomes exactly `0`, which is invalid because the sum must always be at least `1`

## Approaches

### Brute Force Approach

The brute force solution tries every possible positive starting value beginning from `1`.

For each candidate starting value:

1. Initialize the running sum to that value.
2. Traverse the array from left to right.
3. Continuously update the running sum.
4. If the running sum ever becomes less than `1`, reject this starting value.
5. Otherwise, return it as the answer.

This approach is correct because it explicitly checks all possible valid starting values in increasing order, so the first valid one must be the minimum.

However, it is inefficient because we may need to test many starting values before finding the correct answer. In the worst case, the required start value could be fairly large.

### Optimal Prefix Sum Approach

The key insight is that we only need to know the minimum prefix sum encountered while traversing the array.

Suppose the prefix sums are:

```
prefixSum = cumulative sum of nums
```

If the minimum prefix sum is `minPrefix`, then:

```
startValue + minPrefix >= 1
```

Rearranging:

```
startValue >= 1 - minPrefix
```

Therefore:

- If the minimum prefix sum is negative, we need enough starting value to offset it.
- If the minimum prefix sum is already nonnegative, `startValue = 1` is sufficient.

This allows us to solve the problem in a single pass.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(k × n) | O(1) | Tries increasing start values until one works |
| Optimal | O(n) | O(1) | Tracks the minimum prefix sum in one traversal |

Here, `k` represents the size of the answer being tested in the brute force method.

## Algorithm Walkthrough

### Optimal Algorithm

1. Initialize two variables:

- `current_sum = 0`
- `minimum_prefix_sum = 0`

The `current_sum` tracks the running prefix sum while traversing the array. The `minimum_prefix_sum` stores the smallest prefix sum seen so far.
2. Traverse the array from left to right.
3. For each number:

- Add it to `current_sum`
- Update `minimum_prefix_sum` if the current sum is smaller
4. After processing the entire array, determine the minimum required starting value.
5. The smallest valid starting value is:

```
1 - minimum_prefix_sum
```

1. If the minimum prefix sum was never negative, this formula naturally gives `1`.

### Why it works

The running sum during execution is:

```
startValue + prefixSum
```

To ensure this quantity never becomes less than `1`, we only need to guarantee that the smallest prefix sum still keeps the result positive.

If the minimum prefix sum is `minPrefix`, then:

```
startValue + minPrefix >= 1
```

Choosing:

```
startValue = 1 - minPrefix
```

makes the smallest possible running total exactly `1`, while all other running totals remain at least `1`.

## Python Solution

```python
from typing import List

class Solution:
    def minStartValue(self, nums: List[int]) -> int:
        current_sum = 0
        minimum_prefix_sum = 0

        for num in nums:
            current_sum += num
            minimum_prefix_sum = min(minimum_prefix_sum, current_sum)

        return 1 - minimum_prefix_sum
```

The implementation follows the optimal prefix sum strategy directly.

The variable `current_sum` stores the cumulative sum while iterating through the array. After each update, we compare it with `minimum_prefix_sum` to keep track of the smallest prefix encountered so far.

At the end of the traversal, the expression:

```python
1 - minimum_prefix_sum
```

computes the minimum positive starting value needed to ensure that every intermediate running sum remains at least `1`.

The solution uses constant extra space and processes the array only once.

## Go Solution

```go
func minStartValue(nums []int) int {
	currentSum := 0
	minimumPrefixSum := 0

	for _, num := range nums {
		currentSum += num

		if currentSum < minimumPrefixSum {
			minimumPrefixSum = currentSum
		}
	}

	return 1 - minimumPrefixSum
}
```

The Go implementation mirrors the Python solution closely.

Go does not provide a built in `min` function for integers, so the minimum comparison is written explicitly with an `if` statement.

The problem constraints are small, so integer overflow is not a concern. Standard Go `int` values are more than sufficient for the possible cumulative sums.

## Edge Cases

### Arrays With Only Positive Numbers

If every element in the array is positive, the running sum can never decrease below the starting value. Since the problem requires the starting value to be positive, the minimum valid answer is simply `1`.

For example:

```
nums = [2, 3, 4]
```

The prefix sums are always positive, so the algorithm keeps `minimum_prefix_sum = 0`, resulting in:

```
1 - 0 = 1
```

### Arrays With Large Negative Prefixes

A common bug is underestimating how much starting value is needed when the cumulative sum drops sharply early in the array.

For example:

```
nums = [-5, 1, 1]
```

The first prefix sum becomes `-5`, so the starting value must compensate for that drop:

```
1 - (-5) = 6
```

The implementation correctly handles this because it continuously tracks the smallest prefix sum encountered.

### Prefix Sum Reaching Exactly Zero

The condition requires the running total to stay at least `1`, not merely nonnegative.

For example:

```
nums = [-1]
```

If `startValue = 1`, then:

```
1 + (-1) = 0
```

This is invalid.

The implementation avoids this mistake by enforcing:

```
startValue + minimum_prefix_sum >= 1
```

instead of checking for nonnegative values.
