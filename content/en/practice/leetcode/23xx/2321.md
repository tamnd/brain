## Solution

## Problem Understanding

We are given two arrays, `nums1` and `nums2`, both with the same length `n`. We may choose exactly one contiguous subarray and swap that subarray between the two arrays. We are also allowed to skip the operation entirely.

After the optional swap, we compute:

- `sum(nums1)`
- `sum(nums2)`

The final score is the larger of those two sums. Our goal is to maximize that score.

The key detail is that the swap must involve the same index range in both arrays. If we choose indices `left` and `right`, then:

- `nums1[left:right+1]` is exchanged with
- `nums2[left:right+1]`

The rest of the arrays remain unchanged.

The constraints are important:

- `n` can be as large as `100000`
- Each element can be up to `10000`

This immediately tells us that any quadratic solution will be too slow. An `O(n^2)` or `O(n^3)` algorithm cannot handle the largest inputs within typical time limits.

Another important observation is that we only care about the total sums after the swap, not the exact final arrangement of the arrays. That strongly suggests that we should think in terms of how much a swap changes the sum of an array.

Several edge cases are important:

- The best answer may be obtained without performing any swap.
- The optimal subarray may contain only one element.
- Swapping may improve `nums1` but hurt `nums2`, or vice versa.
- All improvements may be negative, meaning every possible swap reduces the maximum score.
- Arrays may already have very different sums, so the optimal move may simply preserve the larger one.

Because the arrays are large, we need a linear time solution.

## Approaches

### Brute Force Approach

A direct brute force solution would try every possible subarray `[left, right]`.

For each candidate subarray:

1. Compute the effect of swapping that segment.
2. Calculate the new sums of both arrays.
3. Update the maximum score seen so far.

There are `O(n^2)` possible subarrays. If we recompute sums naively for each swap, the complexity becomes `O(n^3)`. Even with prefix sums to optimize range calculations, the solution would still require `O(n^2)` time.

With `n = 100000`, this is far too slow.

The brute force approach is correct because it explicitly evaluates every legal swap, but it is computationally infeasible.

### Key Insight for the Optimal Solution

Instead of thinking about the entire arrays after a swap, we should focus on how the swap changes the sum.

Suppose we want to maximize the final sum of `nums1`.

Let:

- `sum1 = sum(nums1)`
- `sum2 = sum(nums2)`

If we swap a subarray `[left, right]`, then `nums1` loses:

```
nums1[left] + nums1[left+1] + ... + nums1[right]
```

and gains:

```
nums2[left] + nums2[left+1] + ... + nums2[right]
```

So the change in `sum1` is:

```
(nums2[i] - nums1[i]) over the chosen range
```

Define:

```
diff[i] = nums2[i] - nums1[i]
```

Then maximizing the new sum of `nums1` becomes:

```
sum1 + maximum subarray sum of diff
```

This is exactly the classic maximum subarray problem, which can be solved with Kadane's algorithm in linear time.

Similarly, to maximize the final sum of `nums2`, we define:

```
diff[i] = nums1[i] - nums2[i]
```

and again compute the maximum subarray sum.

The final answer is the larger of:

- Best achievable sum for `nums1`
- Best achievable sum for `nums2`

## Algorithm Walkthrough

### Step 1: Compute the original sums

First compute:

```
sum1 = sum(nums1)
sum2 = sum(nums2)
```

These represent the scores before any swap occurs.

### Step 2: Determine how to maximize `nums1`

If we swap index `i`, then `nums1` gains:

```
nums2[i] - nums1[i]
```

Construct the conceptual gain array:

```
gain1[i] = nums2[i] - nums1[i]
```

Now the problem becomes:

> Find a contiguous subarray whose total gain is as large as possible.

This is exactly the maximum subarray problem.

### Step 3: Run Kadane's algorithm

Kadane's algorithm maintains:

- `current_gain`, the best subarray ending at the current position
- `best_gain`, the best subarray seen overall

At each position:

```
current_gain = max(value, current_gain + value)
best_gain = max(best_gain, current_gain)
```

This efficiently finds the maximum possible improvement.

Then:

```
best_nums1 = sum1 + best_gain
```

### Step 4: Repeat symmetrically for `nums2`

Now reverse the perspective.

If we want to maximize `nums2`, the gain from swapping index `i` becomes:

```
nums1[i] - nums2[i]
```

Again apply Kadane's algorithm.

Then:

```
best_nums2 = sum2 + best_gain
```

### Step 5: Return the larger result

The answer is:

```
max(best_nums1, best_nums2)
```

because the score is defined as the larger of the two final sums.

### Why it works

Every valid swap corresponds to a contiguous subarray. The effect of that swap on an array's total sum is exactly the sum of per-index gains over that range.

Therefore, maximizing the final sum reduces to finding the contiguous range with maximum total gain. Kadane's algorithm is proven to compute the maximum subarray sum in linear time, so it correctly identifies the optimal swap.

By evaluating both directions independently, we guarantee that we find the best achievable score overall.

## Python Solution

```python
from typing import List

class Solution:
    def maximumsSplicedArray(self, nums1: List[int], nums2: List[int]) -> int:
        def max_gain(source: List[int], target: List[int]) -> int:
            best_gain = 0
            current_gain = 0

            for s, t in zip(source, target):
                gain = t - s

                current_gain = max(gain, current_gain + gain)
                best_gain = max(best_gain, current_gain)

            return best_gain

        sum1 = sum(nums1)
        sum2 = sum(nums2)

        best_for_nums1 = sum1 + max_gain(nums1, nums2)
        best_for_nums2 = sum2 + max_gain(nums2, nums1)

        return max(best_for_nums1, best_for_nums2)
```

The implementation begins by defining a helper function, `max_gain`. This function computes the maximum improvement obtainable by swapping a contiguous subarray from `target` into `source`.

For every index, we calculate:

```python
gain = target[i] - source[i]
```

This represents how much the source array's sum would increase if that element were swapped.

Kadane's algorithm is then applied directly to these gains. The variable `current_gain` tracks the best gain ending at the current index, while `best_gain` stores the overall maximum.

The helper function returns the maximum possible improvement.

Next, the code computes the original sums of both arrays. Then it evaluates:

- The best possible final sum for `nums1`
- The best possible final sum for `nums2`

Finally, the larger of those two values is returned.

The solution uses only constant extra memory and processes the arrays in linear time.

## Go Solution

```go
package main

func maximumsSplicedArray(nums1 []int, nums2 []int) int {
	maxGain := func(source []int, target []int) int {
		bestGain := 0
		currentGain := 0

		for i := 0; i < len(source); i++ {
			gain := target[i] - source[i]

			currentGain = max(gain, currentGain+gain)
			bestGain = max(bestGain, currentGain)
		}

		return bestGain
	}

	sum1 := 0
	sum2 := 0

	for _, value := range nums1 {
		sum1 += value
	}

	for _, value := range nums2 {
		sum2 += value
	}

	bestForNums1 := sum1 + maxGain(nums1, nums2)
	bestForNums2 := sum2 + maxGain(nums2, nums1)

	return max(bestForNums1, bestForNums2)
}

func max(a int, b int) int {
	if a > b {
		return a
	}
	return b
}
```

The Go implementation mirrors the Python solution closely.

One small difference is that Go does not provide a built in `max` function for integers, so we define a helper manually.

Slices are used directly without additional allocations, and all computations fit safely within Go's `int` range because the maximum possible sum is:

```go
100000 * 10000 = 1e9
```

which is well within 32 bit signed integer limits.

## Edge Cases

One important edge case occurs when every possible swap decreases the score. In this situation, the algorithm must correctly recognize that no swap should be performed. This is why Kadane's algorithm is initialized with `best_gain = 0`. If all gains are negative, the best improvement remains zero, meaning the original array sum is preserved.

Another critical case is when the optimal swap consists of exactly one element. A naive implementation might incorrectly assume that longer subarrays are always preferable. Kadane's algorithm naturally handles this because it evaluates whether starting a new subarray at the current index is better than extending the previous one.

A third important edge case is swapping the entire array. Some implementations accidentally restrict the solution to proper subarrays or mishandle boundaries. Because Kadane's algorithm can select a range spanning all indices, the implementation correctly supports full-array swaps as well.

A subtle case involves identical arrays. In that situation, every gain value is zero, so every swap produces the same sums. The algorithm correctly returns the original total because the maximum achievable gain is zero.

Finally, arrays with very large sizes require careful complexity management. Since the constraints allow up to `100000` elements, any quadratic solution would time out. The linear time Kadane-based approach guarantees efficient execution even at maximum input size.
