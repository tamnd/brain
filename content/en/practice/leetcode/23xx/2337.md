## Solution

## Problem Understanding

In this problem, we are given two strings, `start` and `target`, of equal length. Each position in the strings contains one of three characters:

- `'L'`, representing a piece that can move only to the left
- `'R'`, representing a piece that can move only to the right
- `'_'`, representing an empty space

A move consists of sliding a piece into an adjacent blank space in the direction it is allowed to move. An `'L'` piece can never move right, and an `'R'` piece can never move left.

The goal is to determine whether it is possible to transform `start` into `target` using any number of valid moves.

The important detail is that pieces cannot jump over each other. Every move only swaps a piece with a neighboring underscore. Because of this, the relative order of all non-blank characters must remain unchanged throughout the process.

The constraints are large, up to `10^5` characters, which means we need a linear time solution. Any brute force simulation that repeatedly tries moves would become too slow.

Several edge cases are especially important:

- The sequence of `'L'` and `'R'` characters must match between the two strings after removing underscores.
- An `'L'` piece cannot end up to the right of its original position.
- An `'R'` piece cannot end up to the left of its original position.
- Strings containing only underscores should return `true`.
- Pieces blocked by other pieces may prevent a transformation even if counts match.

## Approaches

### Brute Force Approach

A brute force solution would attempt to simulate all possible valid moves from the `start` configuration until either the `target` configuration is found or all reachable states are exhausted.

One possible implementation would use breadth first search. At every state, we try all valid moves:

- Swap `"L_"` into `"_L"`
- Swap `"_R"` into `"R_"`

This guarantees correctness because BFS explores every reachable arrangement.

However, the number of possible configurations grows exponentially with the string length. Even moderate input sizes would produce far too many states to explore efficiently. Since `n` can be `100000`, brute force is completely infeasible.

### Optimal Two Pointer Approach

The key insight is that the actual movement process does not need to be simulated.

Two observations make the problem much simpler:

1. The relative order of pieces never changes.

Since pieces cannot pass through each other, removing all underscores from both strings must produce the same sequence.
2. Each piece has movement restrictions.

- `'L'` can only move left, so its final position must be less than or equal to its starting position.
- `'R'` can only move right, so its final position must be greater than or equal to its starting position.

We can scan both strings using two pointers that skip underscores and compare only the actual pieces.

If we ever find:

- Different piece types
- An `'L'` that moved right
- An `'R'` that moved left

then the transformation is impossible.

Otherwise, it is possible.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Explores all reachable states |
| Optimal | O(n) | O(1) | Uses two pointers and movement constraints |

## Algorithm Walkthrough

1. Initialize two pointers, `i` and `j`, both starting at `0`.

Pointer `i` scans the `start` string, and pointer `j` scans the `target` string.
2. Skip underscores in both strings.

Since underscores represent empty spaces, they are irrelevant when comparing the order of actual pieces.
3. After skipping underscores, check whether both pointers reached the end.

If both pointers are at the end simultaneously, every piece matched successfully, so return `true`.
4. If only one pointer reached the end, return `false`.

This means the number of non-underscore pieces differs.
5. Compare the current non-underscore characters.

If `start[i] != target[j]`, the piece order changed, which is impossible.
6. Check movement rules.

- If the piece is `'L'` and `i < j`, then the piece moved right, which is illegal.
- If the piece is `'R'` and `i > j`, then the piece moved left, which is illegal.
7. Move both pointers forward and continue scanning.
8. If the entire scan finishes without violating any rule, return `true`.

### Why it works

The algorithm relies on two invariants:

- Pieces never change relative order because they cannot pass through each other.
- Each piece type has a one directional movement constraint.

By checking both conditions directly, we completely characterize whether a valid sequence of moves exists. No explicit simulation is necessary.

## Python Solution

```python
class Solution:
    def canChange(self, start: str, target: str) -> bool:
        n = len(start)

        i = 0
        j = 0

        while i < n or j < n:
            while i < n and start[i] == '_':
                i += 1

            while j < n and target[j] == '_':
                j += 1

            if i == n and j == n:
                return True

            if i == n or j == n:
                return False

            if start[i] != target[j]:
                return False

            if start[i] == 'L' and i < j:
                return False

            if start[i] == 'R' and i > j:
                return False

            i += 1
            j += 1

        return True
```

The implementation follows the algorithm directly.

The two pointers scan through both strings simultaneously. The inner `while` loops skip all underscores so that we compare only actual pieces.

After skipping blanks, the code verifies that both strings still contain pieces at the same stage of traversal. If one string has more pieces than the other, the transformation is impossible.

Next, the piece types must match exactly. This guarantees that the order of non-underscore characters is preserved.

Finally, the directional constraints are checked:

- `'L'` cannot move to a larger index
- `'R'` cannot move to a smaller index

If all checks pass for every piece, the transformation is valid.

## Go Solution

```go
func canChange(start string, target string) bool {
	n := len(start)

	i := 0
	j := 0

	for i < n || j < n {
		for i < n && start[i] == '_' {
			i++
		}

		for j < n && target[j] == '_' {
			j++
		}

		if i == n && j == n {
			return true
		}

		if i == n || j == n {
			return false
		}

		if start[i] != target[j] {
			return false
		}

		if start[i] == 'L' && i < j {
			return false
		}

		if start[i] == 'R' && i > j {
			return false
		}

		i++
		j++
	}

	return true
}
```

The Go implementation is almost identical to the Python version.

Since Go strings are byte indexed, we can directly compare characters using `start[i]` and `target[j]`.

No additional memory allocation is required, and the algorithm still runs in linear time with constant extra space.

## Edge Cases

One important edge case occurs when the strings contain only underscores. In this situation, there are no pieces to move, so the transformation is trivially possible. A careless implementation might incorrectly assume at least one piece exists. The two pointer solution handles this naturally because both pointers immediately reach the end simultaneously.

Another critical edge case is when the non-underscore sequences differ. For example:

```
start  = "R_L_"
target = "__LR"
```

Even though the counts of `'L'` and `'R'` match, the relative ordering changes from `"RL"` to `"LR"`. Since pieces cannot pass through each other, this is impossible. The implementation detects this by directly comparing corresponding non-underscore characters.

A third important case is illegal directional movement. Consider:

```
start  = "_R"
target = "R_"
```

The piece order matches, but the `'R'` piece would need to move left. The algorithm compares indices and rejects any `'R'` where `start_index > target_index`. Similarly, it rejects any `'L'` where `start_index < target_index`.

Another subtle case involves long distance movement across many blanks. For example:

```
start  = "__L__"
target = "L____"
```

Although the piece moves several positions, every move is still legal because `'L'` only moves left. The implementation does not care about the number of moves, only whether the final position respects the directional constraints.
