## Solution

## Problem Understanding

In this problem, we are given an integer array `nums`, and we may repeatedly swap only adjacent elements. The goal is to transform the array into a valid array with the minimum number of adjacent swaps.

A valid array must satisfy two conditions:

- One occurrence of the minimum value must end up at index `0`
- One occurrence of the maximum value must end up at index `n - 1`

If there are multiple occurrences of the minimum or maximum values, we are allowed to choose whichever occurrence gives the fewest swaps.

The important detail is that swaps are restricted to adjacent elements. Moving an element from index `i` to index `j` therefore costs `|i - j|` swaps.

The constraints allow arrays up to `10^5` elements, which means we need a linear or near-linear solution. Any algorithm that simulates swaps repeatedly or explores many possibilities will be too slow.

Several edge cases are important:

- Arrays with only one element are already valid
- Arrays where the minimum and maximum are the same value
- Arrays with multiple minimums or maximums
- Cases where moving one element changes the position of the other
- Situations where the minimum element lies to the right of the chosen maximum element

That final case is especially important because moving the maximum to the end first shifts indices left by one for elements originally to its right.

## Approaches

### Brute Force Approach

A brute-force strategy would try all possible choices of minimum and maximum positions, simulate the adjacent swaps, and compute the total number of swaps required.

For every occurrence of the minimum value and every occurrence of the maximum value, we could:

- Move the chosen minimum to the front
- Move the chosen maximum to the back
- Count swaps carefully while accounting for shifting indices

This approach is correct because it evaluates every valid configuration and selects the minimum cost.

However, it becomes inefficient for large arrays. If there are many duplicate minimums and maximums, the number of combinations can grow significantly. Simulating swaps repeatedly also adds extra overhead.

### Optimal Greedy Approach

The key insight is that the optimal strategy is always:

- Use the leftmost occurrence of the minimum value
- Use the rightmost occurrence of the maximum value

Why?

- The leftmost minimum requires the fewest swaps to reach index `0`
- The rightmost maximum requires the fewest swaps to reach index `n - 1`

We can compute both costs independently:

- Swaps to move minimum to front = `minIndex`
- Swaps to move maximum to end = `(n - 1 - maxIndex)`

However, there is one subtle interaction.

If the minimum element lies to the right of the maximum element, then moving the maximum to the end first shifts the minimum one position left. This reduces the total swaps by `1`.

So:

- If `minIndex > maxIndex`, subtract `1` from the total

This gives a clean linear-time greedy solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Tries multiple min/max choices and simulates swaps |
| Optimal | O(n) | O(1) | Greedily selects best min and max positions |

## Algorithm Walkthrough

1. Traverse the array to identify:

- The leftmost occurrence of the minimum value
- The rightmost occurrence of the maximum value
2. To get the leftmost minimum:

- Update the stored index only when finding a strictly smaller value
3. To get the rightmost maximum:

- Update the stored index whenever finding a value greater than or equal to the current maximum
- Using `>=` ensures we keep the latest occurrence
4. Compute swaps needed for the minimum:

- Moving an element at index `i` to the front requires exactly `i` adjacent swaps
5. Compute swaps needed for the maximum:

- Moving an element at index `j` to the end requires `(n - 1 - j)` swaps
6. Add the two swap counts together
7. Check whether the minimum lies to the right of the maximum:

- If `minIndex > maxIndex`, subtract `1`
- This adjustment accounts for the shift caused when the maximum moves rightward
8. Return the final answer

### Why it works

The algorithm works because adjacent swaps move elements one position at a time, so the number of swaps needed depends entirely on distance.

Choosing the leftmost minimum minimizes the distance to the front. Choosing the rightmost maximum minimizes the distance to the end.

The only interaction between the two operations occurs when the minimum starts to the right of the maximum. Moving the maximum first shifts the minimum left by one position, reducing its required swaps by one. Adjusting for this overlap guarantees the optimal answer.

## Python Solution

```python
from typing import List

class Solution:
    def minimumSwaps(self, nums: List[int]) -> int:
        n = len(nums)

        min_index = 0
        max_index = 0

        for i in range(n):
            if nums[i] < nums[min_index]:
                min_index = i

            if nums[i] >= nums[max_index]:
                max_index = i

        swaps = min_index + (n - 1 - max_index)

        if min_index > max_index:
            swaps -= 1

        return swaps
```

The implementation begins by initializing indices for the minimum and maximum elements.

During a single traversal:

- We update `min_index` only when a strictly smaller value is found. This preserves the leftmost minimum.
- We update `max_index` whenever we see a value greater than or equal to the current maximum. This preserves the rightmost maximum.

After the traversal:

- `min_index` directly represents the swaps needed to move the minimum to the front
- `(n - 1 - max_index)` represents the swaps needed to move the maximum to the end

Finally, we check whether the minimum originally lies after the maximum. If so, one swap overlaps, so we subtract one from the total.

## Go Solution

```go
func minimumSwaps(nums []int) int {
    n := len(nums)

    minIndex := 0
    maxIndex := 0

    for i := 0; i < n; i++ {
        if nums[i] < nums[minIndex] {
            minIndex = i
        }

        if nums[i] >= nums[maxIndex] {
            maxIndex = i
        }
    }

    swaps := minIndex + (n - 1 - maxIndex)

    if minIndex > maxIndex {
        swaps--
    }

    return swaps
}
```

The Go implementation follows the same logic as the Python version.

A few Go-specific notes:

- Slices are used directly since the input is already a dynamic array type
- Integer overflow is not a concern because the maximum answer is bounded by `n`
- The solution uses constant extra space and only simple integer variables

## Edge Cases

One important edge case occurs when the array contains only one element. In that situation, the single value is simultaneously the minimum and maximum, and it already satisfies both conditions. A careless implementation might attempt unnecessary movement or produce negative values. This solution correctly returns `0` because both indices remain `0`.

Another tricky case involves duplicate minimum or maximum values. For example, in `[1,5,5,5]`, using the leftmost maximum would require unnecessary swaps. The optimal choice is the rightmost maximum because it is already at the end. The implementation handles this by updating `maxIndex` whenever it encounters a value greater than or equal to the current maximum.

A third critical edge case occurs when the minimum element appears to the right of the maximum element. For example:

```
[2,5,1,4,3]
```

If we independently count swaps for both elements, we overcount by one because moving the maximum affects the minimum's position. The implementation explicitly checks `minIndex > maxIndex` and subtracts one to correct the overlap.

A final subtle case occurs when all elements are equal. In that scenario, every element is both a minimum and maximum candidate. The algorithm naturally chooses:

- Leftmost minimum at index `0`
- Rightmost maximum at index `n - 1`

This produces zero swaps, which is optimal.
