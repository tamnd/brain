## Solution

## Problem Understanding

The problem describes a street with two independent sides. Each side contains `n` plots, and every plot can either contain a house or remain empty.

The restriction is very important: no two houses may be adjacent on the same side of the street. This means that if one plot contains a house, the neighboring plots on that same side cannot also contain houses.

However, the two sides are independent from each other. A house on plot `i` on the left side does not affect whether a house may exist on plot `i` on the right side.

The task is to count the total number of valid arrangements across both sides of the street.

For example, when `n = 1`, each side has only one plot. A single plot can either be empty or contain a house, giving 2 possibilities per side. Since the two sides are independent, the total becomes:

- `2 * 2 = 4`

The answer can become extremely large, so we must return it modulo:

$$10^9 + 7$$

The constraint:

$$1 \le n \le 10^4$$

tells us several important things:

- A brute force enumeration of all configurations is impossible.
- Each side alone has `2^n` possible subsets of plots.
- Across both sides, naive enumeration would involve `2^(2n)` states, which is astronomically large when `n = 10^4`.

This strongly suggests that the problem requires a dynamic programming solution.

Important edge cases include:

- `n = 1`, the smallest valid input.
- Very large values of `n`, where exponential algorithms will time out.
- Consecutive placement restrictions, which can easily lead to incorrect transitions in a DP implementation.
- Correct handling of modulo arithmetic to avoid overflow.

## Approaches

### Brute Force Approach

A brute force solution would try every possible placement of houses on both sides of the street.

For each side, every plot has two choices:

- Place a house
- Leave it empty

That produces:

$$2^n$$

possible arrangements per side.

For both sides together, the total number of raw configurations becomes:

$$2^n \times 2^n = 2^{2n}$$

For every configuration, we would check whether any two adjacent plots on the same side both contain houses.

This approach is correct because it exhaustively examines every possible arrangement and counts only valid ones.

However, it is far too slow. Even for `n = 30`, the number of configurations becomes enormous. Since `n` can reach `10^4`, brute force is completely infeasible.

### Optimal Dynamic Programming Approach

The key insight is that the two sides of the street are independent.

Instead of thinking about both sides simultaneously, we can first compute:

- The number of valid arrangements for one side

Then square the result.

Now consider one side only.

For each plot:

- If the previous plot is empty, the current plot may contain a house or remain empty.
- If the previous plot contains a house, the current plot must remain empty.

This creates a Fibonacci-style recurrence.

Let:

- `dp[i]` = number of valid arrangements for `i` plots on one side.

For the `i-th` plot:

- Leave it empty:

- contributes `dp[i-1]`
- Place a house:

- previous plot must be empty
- contributes `dp[i-2]`

So:

$$dp[i] = dp[i-1] + dp[i-2]$$

This is exactly the Fibonacci recurrence.

After computing the valid arrangements for one side, the final answer is:

$$(dp[n])^2$$

because the left and right sides are independent choices.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^(2n)) | O(1) | Enumerates all configurations |
| Optimal Dynamic Programming | O(n) | O(1) | Fibonacci-style DP for one side, then square result |

## Algorithm Walkthrough

1. Observe that the two sides of the street are independent.

If we know the number of valid arrangements for one side, then the total number of arrangements for both sides is simply the square of that value.
2. Define a DP recurrence for one side of the street.

Let `dp[i]` represent the number of valid ways to arrange houses on `i` plots.
3. Establish the base cases.

For one plot:

- Empty
- House

Therefore:

$$dp[1] = 2$$

For zero plots:

$$dp[0] = 1$$

because there is exactly one way to arrange nothing.
4. Build the recurrence.

For plot `i`:

- If it is empty, then the first `i-1` plots can be arranged in `dp[i-1]` ways.
- If it contains a house, then plot `i-1` must be empty, leaving `dp[i-2]` valid arrangements.

Therefore:

$$dp[i] = dp[i-1] + dp[i-2]$$
5. Iteratively compute the DP values.

Since each state depends only on the previous two states, we only need two variables instead of a full array.
6. Compute the final answer.

If `ways` is the number of valid arrangements for one side, then:

$$answer = ways^2 \bmod (10^9 + 7)$$

### Why it works

The recurrence correctly partitions all valid arrangements into two disjoint categories:

- Arrangements where the last plot is empty
- Arrangements where the last plot contains a house

Every valid arrangement belongs to exactly one category, and the adjacency restriction determines the allowed transitions. Since the two sides of the street are independent, multiplying the counts for each side gives the total number of valid global arrangements.

## Python Solution

```python
class Solution:
    def countHousePlacements(self, n: int) -> int:
        MOD = 10**9 + 7

        prev2 = 1  # dp[0]
        prev1 = 2  # dp[1]

        for _ in range(2, n + 1):
            current = (prev1 + prev2) % MOD
            prev2 = prev1
            prev1 = current

        return (prev1 * prev1) % MOD
```

The implementation uses two variables to simulate the Fibonacci recurrence efficiently.

`prev2` stores `dp[i-2]`, while `prev1` stores `dp[i-1]`.

For every new plot:

```python
current = prev1 + prev2
```

computes the number of valid arrangements for the current number of plots.

After computing the DP for one side, the result is squared because the left and right sides are independent.

Modulo arithmetic is applied throughout the computation to avoid overflow and satisfy the problem requirements.

## Go Solution

```go
func countHousePlacements(n int) int {
    const MOD int = 1_000_000_007

    prev2 := 1 // dp[0]
    prev1 := 2 // dp[1]

    for i := 2; i <= n; i++ {
        current := (prev1 + prev2) % MOD
        prev2 = prev1
        prev1 = current
    }

    return int((int64(prev1) * int64(prev1)) % int64(MOD))
}
```

The Go implementation follows the same logic as the Python solution.

One important difference is integer overflow handling. Since the final multiplication may exceed the range of a 32-bit integer, the multiplication is performed using `int64` before applying the modulo.

Go does not have Python's arbitrary precision integers, so explicit casting is necessary for safe arithmetic.

## Edge Cases

One important edge case is `n = 1`. This is the smallest possible input, and many DP implementations fail here because they incorrectly assume at least two states exist. The implementation handles this naturally because `prev1` is initialized to the correct value for one plot.

Another important case is very large values of `n`, such as `10000`. A recursive Fibonacci implementation without memoization would be far too slow and would likely exceed recursion depth limits. The iterative DP approach avoids recursion entirely and runs efficiently in linear time.

Modulo arithmetic is also a critical edge case. Fibonacci-style growth becomes extremely large very quickly. Without applying modulo operations during computation, integer overflow could occur in languages with fixed-size integer types. The implementation applies modulo at every step and uses `int64` multiplication in Go to guarantee correctness.
