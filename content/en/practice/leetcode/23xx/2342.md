## Solution

## Problem Understanding

The problem gives us an array of positive integers called `nums`. We must find two different indices `i` and `j` such that the sum of the digits of `nums[i]` is equal to the sum of the digits of `nums[j]`.

Among all such valid pairs, we want the maximum possible value of:

```
nums[i] + nums[j]
```

If no two numbers share the same digit sum, then the answer is `-1`.

A digit sum means adding all digits of a number together. For example:

- `18 -> 1 + 8 = 9`
- `36 -> 3 + 6 = 9`
- `43 -> 4 + 3 = 7`

So in the first example:

```
18 and 36 both have digit sum 9
43 and 7 both have digit sum 7
```

The pair `(18, 36)` gives the larger total, so the answer is `54`.

The constraints are important:

- `nums.length` can be as large as `100000`
- `nums[i]` can be as large as `10^9`

These limits immediately tell us that a quadratic solution using nested loops will likely be too slow. A solution with time complexity close to `O(n)` is preferred.

Another important observation is that the largest possible digit sum is small. Since:

```
10^9 has at most 10 digits
Maximum digit sum <= 9 * 10 = 90
```

So although the array may be very large, the number of possible digit sums is tiny.

There are several edge cases to consider:

- No two numbers share the same digit sum, in which case we return `-1`
- Multiple numbers may share the same digit sum
- The maximum answer may come from very large values near `10^9`
- Duplicate numbers are allowed and may form a valid pair if they appear at different indices
- The array may contain only one element, making a pair impossible

## Approaches

### Brute Force Approach

The most direct solution is to check every possible pair of indices `(i, j)` where `i < j`.

For each pair:

1. Compute the digit sum of `nums[i]`
2. Compute the digit sum of `nums[j]`
3. Compare them
4. If they are equal, update the maximum pair sum

This approach is correct because it explicitly examines every valid pair and therefore cannot miss the optimal answer.

However, the time complexity is too large. With up to `100000` elements, the number of pairs is approximately:

```
100000 * 99999 / 2
```

which is about five billion comparisons. That is far too slow.

### Optimal Approach

The key insight is that for each digit sum, we only care about the largest number we have seen so far.

Suppose several numbers have the same digit sum:

```
digit sum = 9
numbers = [18, 36, 54, 72]
```

To maximize the pair sum, we only need the two largest numbers.

Instead of comparing every pair, we can process the array once while maintaining:

```
digit_sum -> largest number seen with this digit sum
```

For each number:

1. Compute its digit sum
2. Check whether we have already seen another number with the same digit sum
3. If yes, compute a candidate answer
4. Update the stored maximum number for that digit sum

This transforms the problem into a hash map lookup problem.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n² × d) | O(1) | Checks every pair, too slow for large inputs |
| Optimal | O(n × d) | O(k) | Uses a hash map grouped by digit sum |

Here:

- `d` is the number of digits in a number, at most 10
- `k` is the number of distinct digit sums, at most about 90

Since `d` and `k` are effectively constants, the optimal solution behaves like `O(n)` in practice.

## Algorithm Walkthrough

1. Create a hash map called `best_for_sum`.

The key will be a digit sum, and the value will be the largest number encountered so far with that digit sum.
2. Initialize the answer as `-1`.

This handles the case where no valid pair exists.
3. Iterate through each number in `nums`.

For every number, compute its digit sum by repeatedly taking the last digit using modulo 10 and removing digits using integer division.
4. Check whether the digit sum already exists in the hash map.

If it does, then we have found another number with the same digit sum. That means we can form a valid pair.
5. Compute the candidate pair sum.

Add the current number to the previously stored maximum number for this digit sum.
6. Update the global answer.

If the candidate pair sum is larger than the current answer, replace it.
7. Update the stored maximum number for this digit sum.

We always want the largest number available for future pair calculations.
8. Continue until all numbers have been processed.
9. Return the final answer.

### Why it works

For each digit sum group, the optimal pair must contain the two largest numbers in that group.

The algorithm maintains the largest number seen so far for every digit sum. When a new number with the same digit sum appears, combining it with the stored maximum produces the best possible pair involving the current number.

Since every number is processed exactly once and every valid pair is considered implicitly through the stored maximum, the algorithm always finds the global maximum pair sum.

## Python Solution

```python
from typing import List

class Solution:
    def maximumSum(self, nums: List[int]) -> int:
        def digit_sum(num: int) -> int:
            total = 0

            while num > 0:
                total += num % 10
                num //= 10

            return total

        best_for_sum = {}
        answer = -1

        for num in nums:
            current_sum = digit_sum(num)

            if current_sum in best_for_sum:
                answer = max(answer, num + best_for_sum[current_sum])

            best_for_sum[current_sum] = max(
                best_for_sum.get(current_sum, 0),
                num
            )

        return answer
```

The implementation begins with a helper function called `digit_sum`. This function repeatedly extracts the last digit using `% 10` and removes it using `// 10`. The accumulated total becomes the digit sum.

The dictionary `best_for_sum` stores the largest number encountered for each digit sum.

As we iterate through `nums`, we compute the digit sum for the current number. If that digit sum already exists in the dictionary, then we can form a valid pair. We compute the candidate sum and update the answer if necessary.

Afterward, we update the dictionary so that it always stores the largest number seen for that digit sum. This ensures future pair calculations are optimal.

Finally, we return the maximum answer found, or `-1` if no valid pair exists.

## Go Solution

```go
func maximumSum(nums []int) int {
	digitSum := func(num int) int {
		sum := 0

		for num > 0 {
			sum += num % 10
			num /= 10
		}

		return sum
	}

	bestForSum := make(map[int]int)
	answer := -1

	for _, num := range nums {
		currentSum := digitSum(num)

		if value, exists := bestForSum[currentSum]; exists {
			if num+value > answer {
				answer = num + value
			}
		}

		if value, exists := bestForSum[currentSum]; !exists || num > value {
			bestForSum[currentSum] = num
		}
	}

	return answer
}
```

The Go implementation follows the same logic as the Python solution.

A map called `bestForSum` stores the maximum number seen for each digit sum. Go maps return a second boolean value indicating whether a key exists, so we use:

```go
value, exists := bestForSum[currentSum]
```

to safely check whether a matching digit sum has already appeared.

Integer overflow is not a concern here because the largest possible pair sum is:

```go
10^9 + 10^9 = 2 * 10^9
```

which fits within Go's standard `int` type on LeetCode platforms.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n × d) | Each number is processed once and digit sum computation takes O(d) |
| Space | O(k) | Hash map stores one value per digit sum |

The algorithm processes every number exactly once. Computing the digit sum requires iterating through the digits of the number, which takes at most 10 operations because `nums[i] <= 10^9`.

The hash map contains at most one entry for each possible digit sum. Since digit sums are bounded by a small constant, the space usage is effectively constant in practice.

## Edge Cases

One important edge case occurs when no valid pair exists. A naive implementation might accidentally return `0` or some uninitialized value. For example:

```
[10, 12, 19, 14]
```

Every number has a unique digit sum, so the correct result is `-1`. The implementation handles this by initializing `answer = -1` and only updating it when a valid pair is found.

Another important case is when multiple numbers share the same digit sum. For example:

```
[18, 36, 54, 72]
```

All numbers have digit sum `9`. A buggy solution might use the first matching pair it encounters, such as `18 + 36`, instead of finding the maximum possible pair. The implementation avoids this by always storing the largest number seen so far for each digit sum.

A third important edge case involves duplicate values:

```
[11, 11]
```

The problem only requires different indices, not different values. Some implementations mistakenly reject duplicates. This solution correctly treats the second `11` as a valid pairing partner because it appears at a different position in the array.

Another subtle case involves very large integers close to the constraint limit. Since values may be as large as `10^9`, careless implementations could worry about overflow. However, the maximum possible pair sum is still within safe integer range for both Python and Go implementations.
