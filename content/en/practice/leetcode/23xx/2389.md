**Difficulty:** 🟢 Easy  
**Topics:** Array, Binary Search, Greedy, Sorting, Prefix Sum

## Solution

## Problem Understanding

The problem gives us two arrays:

- `nums`, which contains positive integers
- `queries`, where each query represents a maximum allowed sum

For every query `queries[i]`, we must determine the largest possible size of a subsequence from `nums` such that the total sum of the chosen elements is less than or equal to the query value.

A subsequence means we can remove elements without changing the order of the remaining ones. However, an important observation is that the order does not actually matter for maximizing the number of elements. Since we only care about the size of the subsequence and all numbers are positive, the optimal strategy is always to take the smallest available numbers first.

For example, suppose:

```
nums = [4,5,2,1]
query = 10
```

If we sort the numbers:

```
[1,2,4,5]
```

Then the best way to maximize how many numbers we can take is:

```
1 + 2 + 4 = 7
```

We can take 3 elements. Adding the next number:

```
1 + 2 + 4 + 5 = 12
```

would exceed the limit.

The constraints are relatively small:

- `1 <= n, m <= 1000`
- `1 <= nums[i], queries[i] <= 10^6`

Even though these limits are not huge, the problem is designed to encourage an efficient combination of sorting, prefix sums, and binary search.

There are several important edge cases to consider:

- A query may be smaller than every number in `nums`, meaning the answer is `0`.
- A query may be large enough to include every element.
- The array may contain duplicate values.
- Since all numbers are positive, adding more elements always increases the sum, which is the key property that makes prefix sums and binary search valid.

## Approaches

### Brute Force Approach

A straightforward approach is to process every query independently.

For each query:

1. Generate all possible subsequences.
2. Compute the sum of each subsequence.
3. Find the largest subsequence whose sum is within the query limit.

This approach is correct because it explicitly checks every possible subsequence and therefore cannot miss the optimal answer.

However, this is extremely inefficient. An array of length `n` has `2^n` subsequences. Even for `n = 1000`, this is completely infeasible.

A slightly better brute-force strategy would be:

1. Sort `nums`
2. For each query, greedily take the smallest numbers until the sum exceeds the limit

This avoids generating all subsequences, but each query still requires a linear scan through the array.

### Key Insight

The important observation is:

To maximize the number of selected elements under a sum limit, we should always choose the smallest numbers first.

Because all numbers are positive, choosing a larger number before a smaller one can never help increase the subsequence length.

After sorting the array, we can build a prefix sum array:

```
prefix[i] = sum of first i+1 smallest numbers
```

Now each query becomes:

> Find the largest prefix sum that is less than or equal to the query.

Since prefix sums are sorted in increasing order, we can use binary search to answer each query efficiently.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m × n) after sorting | O(1) | Greedily scans for every query |
| Optimal | O(n log n + m log n) | O(n) | Uses sorting, prefix sums, and binary search |

## Algorithm Walkthrough

1. Sort the `nums` array in non-decreasing order.

We sort because taking smaller numbers first always maximizes how many elements can fit within a given sum limit.
2. Build a prefix sum array.

Each position stores the cumulative sum of the smallest elements up to that index.

For example:

```
nums = [1,2,4,5]

prefix = [1,3,7,12]
```

This means:

- 1 element costs 1
- 2 elements cost 3
- 3 elements cost 7
- 4 elements cost 12
3. Process each query independently.

For a query value `q`, we want to know how many smallest elements can fit within the limit.
4. Use binary search on the prefix sum array.

We search for the first prefix sum greater than `q`.

The index of that position tells us how many elements fit.

Example:

```
prefix = [1,3,7,12]
query = 10
```

The first value greater than 10 is 12, at index 3.

Therefore, 3 elements can be chosen.
5. Store the result for each query.
6. Return the final answer array.

### Why it works

The algorithm works because all numbers are positive. If we want to maximize the number of selected elements under a sum constraint, choosing smaller numbers is always better than choosing larger ones.

Sorting guarantees that the cheapest possible subsequence of length `k` is always the first `k` elements. Therefore, if the prefix sum of the first `k` elements exceeds the query, then no other subsequence of length `k` can satisfy the constraint.

This property makes binary search on prefix sums both correct and efficient.

## Python Solution

```python
from typing import List
import bisect

class Solution:
    def answerQueries(self, nums: List[int], queries: List[int]) -> List[int]:
        nums.sort()

        prefix_sums = []

        current_sum = 0

        for num in nums:
            current_sum += num
            prefix_sums.append(current_sum)

        answer = []

        for query in queries:
            count = bisect.bisect_right(prefix_sums, query)
            answer.append(count)

        return answer
```

The implementation begins by sorting `nums`, ensuring the smallest values appear first. This is essential because the optimal subsequence always uses the smallest available numbers.

Next, the code constructs the `prefix_sums` array. Each entry represents the cumulative cost of taking the first `k` smallest numbers.

For every query, the solution uses `bisect_right`, which performs binary search. This function returns the insertion position for the query while maintaining sorted order.

That insertion index is exactly the number of prefix sums less than or equal to the query, which corresponds to the maximum valid subsequence size.

The final answer list is returned after all queries are processed.

## Go Solution

```go
package main

import (
	"sort"
)

func answerQueries(nums []int, queries []int) []int {
	sort.Ints(nums)

	prefixSums := make([]int, len(nums))

	currentSum := 0

	for i, num := range nums {
		currentSum += num
		prefixSums[i] = currentSum
	}

	answer := make([]int, len(queries))

	for i, query := range queries {
		count := sort.Search(len(prefixSums), func(j int) bool {
			return prefixSums[j] > query
		})

		answer[i] = count
	}

	return answer
}
```

The Go solution follows the same logic as the Python implementation.

Instead of Python's `bisect_right`, Go uses `sort.Search` to perform binary search. The search finds the first index where the prefix sum becomes greater than the query.

Go slices are dynamically sized views over arrays, making them convenient for prefix sum storage.

Integer overflow is not an issue here because the maximum possible total sum is:

```go
1000 × 10^6 = 10^9
```

which safely fits within Go's standard `int` type on LeetCode environments.

## Edge Cases

### Query Smaller Than Every Number

If a query is smaller than the minimum value in `nums`, no elements can be selected.

Example:

```
nums = [5,6,7]
query = 4
```

The implementation handles this naturally because binary search returns index `0`, meaning zero prefix sums are valid.

### Query Large Enough For Entire Array

A query may exceed the total sum of all elements.

Example:

```
nums = [1,2,3]
query = 100
```

In this case, binary search reaches the end of the prefix sum array and returns `n`, meaning all elements can be included.

### Duplicate Values

Arrays may contain repeated numbers.

Example:

```
nums = [2,2,2,2]
```

The algorithm still works correctly because sorting preserves duplicates and prefix sums remain monotonically increasing.

Binary search correctly identifies how many repeated values can fit within the query limit.
