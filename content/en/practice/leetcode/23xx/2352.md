## Solution

## Problem Understanding

In this problem, we are given an `n x n` matrix called `grid`. Each row and each column contains integers. The task is to count how many `(row, column)` pairs are exactly equal.

Two arrays are considered equal only if they contain the same values in the same order. That means the comparison is positional, not just based on frequency.

For example, if a row is:

```
[2, 4, 2]
```

then it matches only a column that is also:

```
[2, 4, 2]
```

A column such as:

```
[2, 2, 4]
```

does not match because the order differs.

The input matrix is always square, meaning the number of rows equals the number of columns. If the matrix has size `n`, then there are:

- `n` rows
- `n` columns

The output should be the total number of matching `(row, column)` pairs.

The constraints are important:

- `1 <= n <= 200`
- `1 <= grid[i][j] <= 10^5`

Since `n` is at most `200`, the matrix contains at most `40,000` values. This is small enough for several approaches, including some quadratic solutions, but we still want to avoid unnecessary repeated comparisons.

A naive implementation might compare every row with every column element by element. That works, but it performs many repeated operations.

Some important edge cases include:

- A `1 x 1` matrix, where the single row and single column are always equal.
- Multiple identical rows matching the same column multiple times.
- Matrices where no row matches any column.
- Repeated values that could confuse implementations using sets instead of ordered structures.
- Rows and columns with the same values but different ordering.

The problem guarantees:

- The matrix is valid and square.
- Every row has exactly `n` elements.
- Values are positive integers.

Because of these guarantees, we do not need additional input validation.

## Approaches

### Brute Force Approach

The most direct solution is to compare every row with every column.

There are `n` rows and `n` columns, so there are `n²` possible pairs. For each pair, we compare all `n` elements one by one.

The algorithm works as follows:

1. Iterate through every row index.
2. Iterate through every column index.
3. Compare the row and column element by element.
4. If all elements match, increment the answer.

This approach is correct because it explicitly checks every possible pair and verifies equality exactly as defined.

However, the time complexity becomes expensive:

- `n²` row-column pairs
- each comparison costs `O(n)`

Total complexity:

```
O(n³)
```

Although `n = 200` is not extremely large, we can still improve the implementation significantly.

### Optimal Approach

The key observation is that rows and columns can be treated as immutable sequences.

If two sequences are equal, then their tuple representation is equal. This means we can:

1. Convert every row into a tuple.
2. Store the frequency of each row tuple in a hash map.
3. Convert each column into a tuple.
4. Look up how many identical rows exist.

This avoids repeatedly comparing rows and columns element by element.

For example:

```
Rows:
(3,1,2,2) -> appears 1 time
(2,4,2,2) -> appears 2 times
```

When we build a column tuple:

```
(2,4,2,2)
```

we immediately know there are `2` matching rows.

Hash maps are ideal here because tuple lookup is efficient and tuples are hashable.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n³) | O(1) | Compare every row with every column element by element |
| Optimal | O(n²) | O(n²) | Store row patterns in a hash map for fast matching |

## Algorithm Walkthrough

1. Create a hash map called `row_count`.

The keys will be tuples representing rows, and the values will be how many times each row appears.
2. Iterate through every row in the matrix.

Convert the row into a tuple because tuples are immutable and hashable in Python. Increment its frequency inside `row_count`.
3. Initialize a variable `pairs = 0`.

This variable stores the total number of equal row-column pairs.
4. Iterate through every column index.

For each column:

- Build a tuple containing all elements from top to bottom.
- This tuple represents the column.
5. Check whether this column tuple exists in `row_count`.

If it does, then every occurrence of that row forms a valid pair with the current column.
6. Add the stored frequency to `pairs`.

For example, if the column tuple appears as a row three times, then this column contributes three valid pairs.
7. After processing all columns, return `pairs`.

### Why it works

The algorithm works because rows and columns are considered equal exactly when their ordered sequences match.

By converting rows and columns into tuples, we create a canonical representation of each sequence. The hash map stores how many times each row sequence appears. When a column sequence matches a stored row sequence, every occurrence contributes one valid pair.

Since every row and every column is processed exactly once, and equality is determined precisely by tuple equality, the algorithm counts all valid pairs correctly.

## Python Solution

```python
from collections import Counter
from typing import List

class Solution:
    def equalPairs(self, grid: List[List[int]]) -> int:
        n = len(grid)

        # Count occurrences of each row
        row_count = Counter(tuple(row) for row in grid)

        pairs = 0

        # Build each column and check matching rows
        for col in range(n):
            column_tuple = tuple(grid[row][col] for row in range(n))
            pairs += row_count[column_tuple]

        return pairs
```

The implementation begins by importing `Counter`, which is a convenient hash map for counting frequencies.

The expression:

```python
Counter(tuple(row) for row in grid)
```

converts every row into a tuple and stores how many times it appears.

Next, the algorithm iterates through every column index. For each column, it constructs a tuple using:

```python
tuple(grid[row][col] for row in range(n))
```

This extracts the column from top to bottom.

The hash map lookup:

```python
row_count[column_tuple]
```

returns how many identical rows exist. That count is added directly to the final answer.

The implementation is concise because Python tuples are naturally hashable and work perfectly as dictionary keys.

## Go Solution

```go
package main

import (
	"fmt"
	"strings"
)

func equalPairs(grid [][]int) int {
	n := len(grid)

	rowCount := make(map[string]int)

	// Store all rows
	for _, row := range grid {
		key := buildKey(row)
		rowCount[key]++
	}

	pairs := 0

	// Build each column and compare
	for col := 0; col < n; col++ {
		column := make([]int, n)

		for row := 0; row < n; row++ {
			column[row] = grid[row][col]
		}

		key := buildKey(column)
		pairs += rowCount[key]
	}

	return pairs
}

func buildKey(arr []int) string {
	var sb strings.Builder

	for _, num := range arr {
		sb.WriteString(fmt.Sprintf("%d#", num))
	}

	return sb.String()
}
```

In Go, slices cannot be used directly as map keys because they are mutable and not comparable.

To solve this, the implementation converts each row and column into a unique string representation. The separator character `#` prevents ambiguity between values such as `[1, 23]` and `[12, 3]`.

The rest of the algorithm is identical to the Python version:

- Count row representations
- Build each column representation
- Add matching frequencies

Integer overflow is not a concern because the maximum number of pairs is `200 × 200 = 40,000`, which easily fits inside Go's `int`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n²) | Building all row tuples and all column tuples each requires visiting every cell once |
| Space | O(n²) | The hash map may store up to `n` tuples of length `n` |

The matrix contains `n²` elements total. Every element participates once while building row representations and once while building column representations. Hash map operations are average `O(1)`, so the overall runtime is quadratic.

The additional memory comes from storing row tuples inside the hash map. In the worst case, all rows are unique, requiring storage proportional to the size of the matrix.

## Edge Cases

One important edge case is a `1 x 1` matrix. In this situation, there is exactly one row and one column, and they necessarily contain the same single value. Some implementations accidentally mishandle tiny inputs because loops assume at least two rows or columns. This implementation handles the case naturally because both the row tuple and column tuple become identical one-element tuples.

Another important edge case involves duplicate rows. For example:

```
[
    [2,4,2,2],
    [2,4,2,2]
]
```

If a matching column exists, both rows should contribute separately to the final answer. A common bug is using a set instead of a frequency map, which would lose duplicate information. This implementation uses `Counter`, so every occurrence is counted correctly.

A third important edge case occurs when rows and columns contain the same values but in different orders. For example:

```
Row:    [1,2,3]
Column: [3,2,1]
```

These are not equal because order matters. Since the algorithm stores tuples exactly as ordered sequences, mismatched ordering correctly produces different hash keys.

Another subtle edge case involves ambiguous serialization in languages like Go. If values are concatenated without separators, arrays such as `[1,23]` and `[12,3]` could produce the same string. The Go implementation avoids this by inserting `#` separators between numbers when building keys.
