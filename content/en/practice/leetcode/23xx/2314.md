## Solution

## Problem Understanding

The problem provides a database table named `Weather` that stores temperature records for different cities throughout the year 2022. Each row contains three pieces of information:

- `city_id`, identifying the city
- `day`, representing the date of the record
- `degree`, representing the recorded temperature

The pair `(city_id, day)` is guaranteed to be unique, meaning a city cannot have multiple temperature records on the same day.

The goal is to find, for every city, the row corresponding to the highest recorded temperature. If multiple days in the same city share the same maximum temperature, we must choose the earliest day among them.

The final output should include:

- `city_id`
- `day`
- `degree`

The results must be sorted by `city_id` in ascending order.

Conceptually, the problem is asking us to perform two levels of selection within each city:

1. Find the maximum temperature for that city.
2. Among all rows with that maximum temperature, choose the earliest date.

The example demonstrates this clearly. For city `2`, the maximum degree `37` appears twice, on `2022-08-07` and `2022-08-17`. Since the problem requires the earliest occurrence, we return `2022-08-07`.

An important observation is that this is fundamentally a grouped ranking problem. We need to evaluate rows within each city independently and determine which row should rank first according to these rules:

1. Higher degree is better.
2. Earlier date is better when degrees are tied.

A naive implementation can easily fail in tie handling. For example, selecting only the maximum degree without considering the earliest date would produce incorrect results when duplicates exist.

The problem guarantees that each city-day combination is unique, so we never need to worry about duplicate rows for the same date. It also guarantees that all records belong to 2022, which simplifies the domain of dates but does not materially affect the algorithm.

## Approaches

### Brute Force Approach

The brute force solution evaluates each city independently.

For every distinct `city_id`, we scan all rows belonging to that city and determine:

- the maximum degree
- the earliest date among rows with that maximum degree

One possible implementation is:

1. Collect all distinct cities.
2. For each city:

- scan the entire table
- track the current best degree
- update the answer whenever:

- a larger degree is found
- or the same degree appears on an earlier date

This approach is correct because every row for every city is examined directly. By comparing all candidate rows, we are guaranteed to find the correct maximum temperature and earliest associated date.

However, this approach becomes inefficient because we repeatedly scan the same table for each city. If there are `n` rows and `k` cities, the worst case complexity becomes `O(n * k)`. In the worst scenario where every row belongs to a different city, this degrades toward `O(n²)`.

### Optimal Approach

The key insight is that SQL window functions are designed exactly for grouped ranking problems like this one.

We can partition the rows by `city_id`, then rank rows inside each city according to:

1. `degree DESC`, so higher temperatures come first
2. `day ASC`, so earlier dates win ties

Once the rows are ranked, the desired answer for each city is simply the row with rank `1`.

The `ROW_NUMBER()` window function is ideal because it produces exactly one winning row per partition.

This approach is efficient because the database engine performs the grouping and ordering internally in a single pass of ranking logic rather than repeated rescanning.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Repeatedly scans rows for each city |
| Optimal | O(n log n) | O(n) | Uses window ranking by city |

## Algorithm Walkthrough

1. Start by partitioning all rows using `city_id`.

This ensures each city's weather records are processed independently from other cities.
2. Within each city partition, sort rows by `degree` in descending order.

The row with the highest temperature should appear first because the problem asks for the maximum recorded degree.
3. For rows with equal degrees, sort by `day` in ascending order.

This guarantees that if the maximum degree appears multiple times, the earliest date will come first.
4. Apply the `ROW_NUMBER()` window function.

This assigns:

- `1` to the best row in each city
- `2` to the next best row
- and so on
5. Filter the result to keep only rows where `row_number = 1`.

Since the ranking already incorporates both maximum degree and earliest day logic, this row is the correct answer for each city.
6. Sort the final result by `city_id ASC`.

This satisfies the output ordering requirement.

### Why it works

The ranking order precisely encodes the problem requirements. Inside each city:

- larger degrees are prioritized first
- earlier dates break ties

Therefore, the row assigned `ROW_NUMBER() = 1` must be the correct answer for that city. Since every city is partitioned independently, selecting rank `1` from every partition produces the full correct result set.

## Python Solution

```python
# Write your MySQL query statement below

SELECT city_id, day, degree
FROM (
    SELECT
        city_id,
        day,
        degree,
        ROW_NUMBER() OVER (
            PARTITION BY city_id
            ORDER BY degree DESC, day ASC
        ) AS rn
    FROM Weather
) ranked_weather
WHERE rn = 1
ORDER BY city_id;
```

The solution uses a subquery to compute rankings for every row.

Inside the subquery, `ROW_NUMBER()` creates rankings independently for each city using `PARTITION BY city_id`.

The ordering logic is critical:

- `degree DESC` ensures the highest temperatures appear first
- `day ASC` ensures earlier dates win ties

The generated row number is stored in the alias `rn`.

After ranking all rows, the outer query filters the result to rows where `rn = 1`. Since only the top-ranked row is kept for each city, the final output contains exactly one answer per city.

Finally, the result is sorted by `city_id` in ascending order.

## Edge Cases

### Multiple maximum temperatures on different days

A city may record the same maximum temperature several times. This is the most important edge case because a naive solution that only computes `MAX(degree)` would return multiple rows instead of exactly one.

The implementation handles this correctly by ordering tied rows using `day ASC`. The earliest date automatically receives `ROW_NUMBER() = 1`.

### Cities with only negative temperatures

Some cities may never record positive temperatures. A buggy implementation might incorrectly assume temperatures are non-negative.

The query works correctly because SQL numeric ordering naturally handles negative values. The largest negative number is still selected properly.

For example:

| degree |
| --- |
| -20 |
| -5 |
| -10 |

The query correctly selects `-5`.

### A city with only one record

A city may contain exactly one weather entry. In this case, that row must automatically become the answer.

The window function handles this naturally. Since the partition contains only one row, it receives rank `1` and is selected by the outer query.

This avoids the need for any special-case logic.
