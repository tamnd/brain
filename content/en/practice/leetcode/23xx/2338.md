## Solution

## Problem Understanding

The problem asks us to count how many arrays of length `n` satisfy a divisibility condition. Every value in the array must be between `1` and `maxValue`, and each element must be divisible by the previous element.

Formally, for every index `i > 0`:

```
arr[i] % arr[i - 1] == 0
```

This means the sequence can stay the same or grow by multiplying with some integer factor. For example:

```
[1, 2, 4, 8]
```

is valid because:

```
2 % 1 == 0
4 % 2 == 0
8 % 4 == 0
```

But:

```
[2, 3, 6]
```

is invalid because:

```
3 % 2 != 0
```

The task is not to generate the arrays themselves, but to count how many distinct valid arrays exist.

The constraints are important:

- `n <= 10^4`
- `maxValue <= 10^4`

A brute force search over all possible arrays would require exploring:

$maxValue^n$

possible arrays, which is astronomically large. Even for small inputs, this becomes impossible.

The key observation is that divisibility relationships form multiplicative chains. Instead of constructing arrays directly, we can reason mathematically about how values evolve through prime factorizations.

Some important edge cases include:

- Arrays where every value is the same, these are always valid.
- `maxValue = 1`, where only one array exists regardless of `n`.
- Large `n` with small `maxValue`, where combinatorial counting becomes critical.
- Values with many prime factors, because they create many possible distributions across positions.

## Approaches

### Brute Force Approach

The most direct solution is to recursively build every possible array.

Starting from the first element, we try every number from `1` to `maxValue`. For each chosen value, the next element must be one of its multiples that still stays within `maxValue`.

This approach is correct because it explicitly enumerates every valid sequence and counts them.

However, it is far too slow. The branching factor can remain large for many steps, and the total number of valid arrays grows exponentially. Even aggressive pruning cannot handle the constraint sizes.

### Key Insight

The crucial observation is that divisibility chains can be analyzed through prime factorization.

Suppose the final value of an array is:

$x=p_1^{e_1}p_2^{e_2}\cdots p_k^{e_k}$

Each exponent must be accumulated gradually across the array because every next value must remain divisible by the previous one.

For one prime factor with exponent `e`, we need to distribute `e` multiplicative increases across `n` positions.

This becomes a classic stars and bars combinatorics problem:

$\binom{n+e-1}{e}$

If multiple prime factors exist, they are independent, so we multiply the combinations for each exponent.

Thus, instead of counting arrays directly, we:

1. Enumerate every ending value from `1` to `maxValue`
2. Prime factorize it
3. Compute how many arrays can end at that value
4. Sum all contributions

This transforms an exponential search into an efficient combinatorial computation.

| --- | --- | --- | --- |

| Optimal | O(maxValue × log(maxValue)) | O(maxValue + n) | Uses prime factorization and combinatorics |

## Algorithm Walkthrough

1. Precompute binomial coefficients.

We repeatedly need values of:

$\binom{n+e-1}{e}$

Since the maximum exponent of any prime factor is small, we can precompute combinations efficiently using Pascal's triangle.
2. Iterate through every possible final value.

For every integer `value` from `1` to `maxValue`, we determine how many ideal arrays can end with this value.
3. Prime factorize the current value.

We decompose the value into prime powers.

Example:

```
12 = 2^2 × 3^1
```
4. Count distributions for each exponent.

For exponent `e`, we compute:

$\binom{n+e-1}{e}$

This represents distributing the exponent increases across the array positions.
5. Multiply contributions of all prime factors.

Since prime exponents evolve independently, we multiply the number of ways for each exponent.

Example:

```
12 = 2^2 × 3^1
```

contributes:

```
C(n+1,2) × C(n,1)
```
6. Add the result to the global answer.

Every valid factorization contributes some number of arrays ending at that value.
7. Return the total modulo `10^9 + 7`.

### Why it works

Every ideal array corresponds to a monotonic accumulation of prime exponents.

Because divisibility only allows exponents to stay the same or increase, the evolution of each prime factor becomes independent. Counting how exponents are distributed across positions reduces to stars and bars combinatorics. Summing over all ending values therefore counts every valid ideal array exactly once.

## Python Solution

```python
class Solution:
    def idealArrays(self, n: int, maxValue: int) -> int:
        MOD = 10**9 + 7
        
        max_exponent = 14
        
        comb = [[0] * (max_exponent + 1) for _ in range(n + max_exponent + 1)]
        
        for i in range(n + max_exponent + 1):
            comb[i][0] = 1
            
            for j in range(1, min(i, max_exponent) + 1):
                if j == i:
                    comb[i][j] = 1
                else:
                    comb[i][j] = (
                        comb[i - 1][j - 1] + comb[i - 1][j]
                    ) % MOD
        
        answer = 0
        
        for value in range(1, maxValue + 1):
            current = value
            ways = 1
            
            factor = 2
            
            while factor * factor <= current:
                exponent = 0
                
                while current % factor == 0:
                    current //= factor
                    exponent += 1
                
                if exponent > 0:
                    ways = (
                        ways * comb[n + exponent - 1][exponent]
                    ) % MOD
                
                factor += 1
            
            if current > 1:
                ways = (ways * comb[n][1]) % MOD
            
            answer = (answer + ways) % MOD
        
        return answer
```

The implementation begins by precomputing binomial coefficients using Pascal's triangle. We only need small exponents because:

$2^{14}>10^4$

so no prime exponent can exceed `13`.

The outer loop iterates through every possible ending value. For each value, the code performs prime factorization by trial division.

Whenever a prime factor appears with exponent `e`, the number of distributions across array positions is:

$\binom{n+e-1}{e}$

The contributions are multiplied together because prime factors are independent.

Finally, all ending-value contributions are summed modulo `10^9 + 7`.

## Go Solution

```go
func idealArrays(n int, maxValue int) int {
	const MOD int = 1_000_000_007

	maxExponent := 14

	comb := make([][]int, n+maxExponent+1)

	for i := range comb {
		comb[i] = make([]int, maxExponent+1)
		comb[i][0] = 1

		limit := i
		if limit > maxExponent {
			limit = maxExponent
		}

		for j := 1; j <= limit; j++ {
			if j == i {
				comb[i][j] = 1
			} else {
				comb[i][j] = (comb[i-1][j-1] + comb[i-1][j]) % MOD
			}
		}
	}

	answer := 0

	for value := 1; value <= maxValue; value++ {
		current := value
		ways := 1

		for factor := 2; factor*factor <= current; factor++ {
			exponent := 0

			for current%factor == 0 {
				current /= factor
				exponent++
			}

			if exponent > 0 {
				ways = (ways * comb[n+exponent-1][exponent]) % MOD
			}
		}

		if current > 1 {
			ways = (ways * comb[n][1]) % MOD
		}

		answer = (answer + ways) % MOD
	}

	return answer
}
```

The Go implementation follows the same logic as the Python version. Since Go uses fixed-width integers, all operations are carefully reduced modulo `MOD` to avoid overflow issues. Slices are used instead of dynamic lists, and the combination table is explicitly allocated with nested slices.

## Edge Cases

One important edge case occurs when `maxValue = 1`. In this scenario, the only possible value in the array is `1`, so exactly one valid array exists regardless of `n`. A buggy implementation might accidentally overcount combinations or mishandle empty prime factorizations. This implementation handles it correctly because the factorization loop contributes multiplicative identity `1`.

Another important case is when `n = 1`. Since there are no divisibility constraints between adjacent elements, every number from `1` to `maxValue` forms a valid array. The combinatorial formula naturally produces one arrangement per ending value, giving the correct total.

Prime powers create another subtle edge case. Consider a value like:

```
8 = 2^3
```

The exponent `3` must be distributed across positions correctly. Incorrect implementations may treat each multiplication independently and double count arrays. The stars and bars formulation guarantees each exponent distribution is counted exactly once.

Large inputs also matter. With:

```
n = 10000
maxValue = 10000
```

a naive DFS would be impossible. This solution remains efficient because the maximum exponent remains very small, allowing compact combinatorial preprocessing and fast factorization.
