# LeetCode 2364, Count Number of Bad Pairs

## Problem Understanding

We are given a 0-indexed integer array `nums`. A pair of indices `(i, j)` is considered a bad pair if:

- `i < j`
- `j - i != nums[j] - nums[i]`

The task is to count how many such bad pairs exist in the array.

To understand the condition more clearly, observe that the pair is bad when the difference between the indices does not equal the difference between the values at those indices.

The input array can contain up to `10^5` elements, and each value can be as large as `10^9`. These constraints are important because they immediately rule out inefficient quadratic solutions for large inputs.

The total number of index pairs in an array of length `n` is:

$$\frac{n(n-1)}{2}$$

A naive approach would examine every possible pair and check whether it is bad. However, for `n = 10^5`, this would require about `10^{10}` comparisons, which is far too slow.

An important observation is that the condition can be rearranged algebraically:

$$j - i = nums[j] - nums[i]$$

Move terms around:

$$j - nums[j] = i - nums[i]$$

or equivalently:

$$nums[j] - j = nums[i] - i$$

This means a pair is good if the expression `nums[i] - i` is equal for both indices.

Therefore:

- Good pairs satisfy `nums[i] - i == nums[j] - j`
- Bad pairs are all remaining pairs

This transformation is the key insight that makes an efficient solution possible.

Some important edge cases include arrays of length `1`, arrays where every pair is good, arrays where every pair is bad, and arrays containing very large integers. The problem guarantees valid integer input and at least one element.

## Approaches

### Brute Force Approach

The most direct solution is to examine every possible pair `(i, j)` where `i < j`.

For each pair:

1. Compute `j - i`
2. Compute `nums[j] - nums[i]`
3. Compare them
4. Increment the answer if they are not equal

This approach is correct because it explicitly checks the definition of a bad pair for every possible pair.

However, the time complexity is `O(n^2)` because there are approximately `n(n-1)/2` pairs. With `n = 10^5`, this becomes computationally infeasible.

### Optimal Approach

The algebraic transformation is the critical insight:

$$j - i = nums[j] - nums[i]$$

Rearranging gives:

$$nums[i] - i = nums[j] - j$$

This means every good pair shares the same value of `nums[i] - i`.

Instead of counting bad pairs directly, we can:

1. Count total pairs
2. Count good pairs efficiently using a hash map
3. Subtract good pairs from total pairs

While iterating through the array, we compute:

$$key = nums[i] - i$$

If this key has appeared before `k` times, then the current index forms `k` new good pairs.

A hash map allows us to track frequencies in constant average time.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Checks every possible pair directly |
| Optimal | O(n) | O(n) | Uses hash map to count good pairs efficiently |

## Algorithm Walkthrough

1. Let `n` be the length of the array.
2. Compute the total number of index pairs using:

$$\frac{n(n-1)}{2}$$

This represents all possible `(i, j)` pairs where `i < j`.

1. Create a hash map called `frequency`.

The key will be:

$$nums[i] - i$$

The value will store how many times this key has appeared so far.

1. Initialize a variable `good_pairs = 0`.

This variable counts how many pairs satisfy the good pair condition.

1. Iterate through the array from left to right.

For each index `i`:

- Compute `key = nums[i] - i`
- If `key` already exists in the hash map with frequency `f`, then the current element forms `f` good pairs with previous indices
- Add `f` to `good_pairs`
- Increment the frequency of `key`

1. After processing the entire array, compute:

$$bad\_pairs = total\_pairs - good\_pairs$$

1. Return the result.

### Why it works

The algorithm works because every good pair must satisfy:

$$nums[i] - i = nums[j] - j$$

The hash map groups indices by this invariant value. Every time we encounter another index with the same key, it forms good pairs with all previous indices sharing that key.

Since every pair is either good or bad, subtracting good pairs from total pairs yields the exact number of bad pairs.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Solution:
    def countBadPairs(self, nums: List[int]) -> int:
        n = len(nums)

        total_pairs = n * (n - 1) // 2
        good_pairs = 0

        frequency = defaultdict(int)

        for index, value in enumerate(nums):
            key = value - index

            good_pairs += frequency[key]
            frequency[key] += 1

        return total_pairs - good_pairs
```

The implementation begins by computing the total number of possible pairs. Since every pair is either good or bad, this gives us the starting point for the final calculation.

A `defaultdict(int)` is used to store frequencies of the transformed value `nums[i] - i`. This avoids manual existence checks and simplifies the code.

As we iterate through the array, we compute the key for the current index. If the key has already appeared several times, then each previous occurrence forms a good pair with the current index. We add that count directly to `good_pairs`.

Finally, we subtract the number of good pairs from the total number of pairs to obtain the answer.

## Go Solution

```go
func countBadPairs(nums []int) int64 {
	n := len(nums)

	totalPairs := int64(n*(n-1)) / 2
	var goodPairs int64 = 0

	frequency := make(map[int]int64)

	for i, value := range nums {
		key := value - i

		goodPairs += frequency[key]
		frequency[key]++
	}

	return totalPairs - goodPairs
}
```

The Go implementation follows the same logic as the Python solution.

One important detail is the use of `int64` for pair counts. The number of pairs can exceed the range of a 32-bit integer when `n` is large. Using `int64` prevents overflow issues.

The hash map stores frequencies using `map[int]int64` so that accumulated counts remain consistent with the return type.

## Worked Examples

### Example 1

```
nums = [4,1,3,3]
```

First compute total pairs:

$$\frac{4 \times 3}{2} = 6$$

Now process each index.

| Index | Value | Key = nums[i] - i | Existing Frequency | New Good Pairs | Frequency Map After |
| --- | --- | --- | --- | --- | --- |
| 0 | 4 | 4 | 0 | 0 | {4:1} |
| 1 | 1 | 0 | 0 | 0 | {4:1, 0:1} |
| 2 | 3 | 1 | 0 | 0 | {4:1, 0:1, 1:1} |
| 3 | 3 | 0 | 1 | 1 | {4:1, 0:2, 1:1} |

Total good pairs = 1

Therefore:

$$bad\_pairs = 6 - 1 = 5$$

Final answer:

```
5
```

Process each index.

| 0 | 1 | 1 | 0 | 0 |
| 1 | 2 | 1 | 1 | 1 |
| 2 | 3 | 1 | 2 | 2 |
| 3 | 4 | 1 | 3 | 3 |

Total good pairs:

$$1 + 2 + 3 + 4 = 10$$

Therefore:

$$bad\_pairs = 10 - 10 = 0$$

| --- | --- | --- |
| Time | O(n) | Each element is processed once with constant-time hash map operations |
| Space | O(n) | The hash map may store up to `n` distinct keys |

The algorithm is linear because every array element is visited exactly once. Hash map insertion and lookup operations are constant time on average.

The space complexity becomes `O(n)` in the worst case when every `nums[i] - i` value is unique.

## Edge Cases

### Single Element Array

When the array contains only one element, there are no possible pairs because a pair requires two distinct indices with `i < j`.

A buggy implementation might incorrectly assume at least one pair exists or mishandle the total pair formula. The implementation correctly computes:

$$\frac{1 \times 0}{2} = 0$$

so the answer is zero.

### All Pairs Are Good

Arrays like `[1,2,3,4,5]` produce the same value for `nums[i] - i` at every index.

This means every pair is good. A common mistake is to count bad pairs directly and accidentally double count or miss pairs. The implementation avoids this by counting good pairs using frequencies and subtracting from the total.

### All Pairs Are Bad

Arrays like `[1,1,1,1]` produce different transformed keys at each index:

$$1-0,\ 1-1,\ 1-2,\ 1-3$$

Since no keys match, there are no good pairs. The implementation correctly leaves `good_pairs` at zero and returns the total number of pairs.

### Large Inputs and Integer Overflow

For `n = 100000`, the number of pairs can be extremely large:

$$\frac{100000 \times 99999}{2}$$

This exceeds 32-bit integer limits. The Go solution explicitly uses `int64` to prevent overflow. Python integers automatically expand as needed, so no special handling is required there.
