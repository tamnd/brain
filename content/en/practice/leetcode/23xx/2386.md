# LeetCode 2386 - Find the K-Sum of an Array

## Problem Understanding

The problem asks us to find the `kth` largest subsequence sum that can be formed from a given integer array.

A subsequence is formed by choosing any subset of elements while preserving their original order. Since order does not affect the sum itself, every element can either be included or excluded independently. For an array of size `n`, there are `2^n` possible subsequences.

Each subsequence produces a sum. If multiple subsequences produce the same sum, they are still counted separately when determining the `kth` largest value. The empty subsequence is valid and contributes a sum of `0`.

The challenge comes from the constraints:

- `n` can be as large as `10^5`
- `nums[i]` can be very large in magnitude
- `k` is at most `2000`

The key observation is that although the total number of subsequences is enormous, the requested rank `k` is relatively small. This suggests that we do not need to enumerate all subsequences.

An important detail is that the array may contain both positive and negative numbers. Positive numbers increase the subsequence sum, while negative numbers decrease it. Intuitively, the largest possible subsequence sum is obtained by taking all positive numbers and excluding all negative numbers.

Several edge cases are important:

- All numbers may be negative, in which case the maximum subsequence sum is `0`, produced by the empty subsequence.
- Duplicate subsequence sums are allowed and must be counted separately.
- Very large integer values require 64-bit arithmetic in Go.
- The array may contain only one element.
- `k` may be equal to the total number of subsequences when `n` is small.

## Approaches

### Brute Force Approach

A brute force solution would generate every possible subsequence, compute its sum, sort all sums in descending order, and return the `kth` element.

This approach is correct because it explicitly examines every possible subsequence. However, it is computationally infeasible because the number of subsequences is `2^n`.

For `n = 10^5`, even storing all subsequence sums is impossible.

### Key Insight for the Optimal Solution

Instead of directly generating the largest subsequence sums, we can think about the problem from the opposite direction.

First, compute the maximum possible subsequence sum:

- Include every positive number
- Exclude every negative number

Call this value `max_sum`.

Now observe what happens when we modify this optimal subsequence:

- Excluding a positive number decreases the sum
- Including a negative number also decreases the sum

If we convert every number into its absolute value, then every deviation from `max_sum` becomes a subtraction of some absolute values.

So the problem becomes:

> Find the `kth` smallest reduction from `max_sum`.

This transforms the original problem into generating the smallest subset sums from the sorted absolute values array.

We can efficiently generate these values in increasing order using a min-heap.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n * n) | O(2^n) | Enumerates all subsequences |
| Optimal | O(n log n + k log k) | O(k) | Uses heap-based generation of smallest reductions |

## Algorithm Walkthrough

1. Compute the maximum possible subsequence sum.

Traverse the array and add all positive numbers to `max_sum`. This represents the largest subsequence sum achievable.
2. Convert every number to its absolute value.

Every change from the optimal subsequence can now be represented as subtracting absolute values from `max_sum`.
3. Sort the absolute values array.

Sorting allows us to generate subset sums in increasing order efficiently.
4. Use a min-heap to generate the smallest reductions.

Each heap entry stores:

- Current reduction sum
- Current index in the sorted array

Initially, push:

- `(abs_nums[0], 0)`

This represents subtracting the first absolute value.
5. Repeatedly extract the smallest reduction.

The heap always gives the next smallest reduction.

After popping `(current_sum, index)`:

- This reduction corresponds to one subsequence sum
- Increment the count of generated reductions
6. Generate the next candidate states.

From position `index`, we can produce two new states:

- Add the next absolute value
- Replace the current absolute value with the next one

Specifically:

- `(current_sum + abs_nums[index + 1], index + 1)`
- `(current_sum - abs_nums[index] + abs_nums[index + 1], index + 1)`
7. Continue until the `kth` largest subsequence sum is determined.

Since the largest subsequence sum is `max_sum`, and reductions are generated in increasing order, the answer is:

`max_sum - kth_smallest_reduction`

### Why It Works

The algorithm works because every subsequence sum can be represented as:

`max_sum - reduction`

where `reduction` is a subset sum of the sorted absolute values.

The heap-based process generates these reductions in strictly increasing order, similar to Dijkstra-style state expansion. Since we only need the first `k` reductions and `k <= 2000`, the algorithm remains efficient.

## Python Solution

```python
from typing import List
import heapq

class Solution:
    def kSum(self, nums: List[int], k: int) -> int:
        max_sum = 0
        abs_nums = []

        for num in nums:
            if num > 0:
                max_sum += num
            abs_nums.append(abs(num))

        abs_nums.sort()

        if k == 1:
            return max_sum

        min_heap = [(abs_nums[0], 0)]

        reduction = 0

        for _ in range(k - 1):
            reduction, index = heapq.heappop(min_heap)

            if index + 1 < len(abs_nums):
                heapq.heappush(
                    min_heap,
                    (reduction + abs_nums[index + 1], index + 1)
                )

                heapq.heappush(
                    min_heap,
                    (
                        reduction - abs_nums[index] + abs_nums[index + 1],
                        index + 1
                    )
                )

        return max_sum - reduction
```

The implementation starts by computing the maximum subsequence sum using all positive numbers.

Next, every number is converted into its absolute value and sorted. This sorted order is essential because the heap-based generation process relies on incrementally extending subset sums.

The heap stores candidate reductions. Each reduction represents how much we subtract from `max_sum`.

The loop runs `k - 1` times because the first largest subsequence sum is simply `max_sum` itself, corresponding to zero reduction.

At each step, the smallest reduction is extracted, and two new states are generated:

- Extending the current subset
- Replacing the last chosen value with the next one

This ensures that all possible subset reductions are explored in increasing order without duplicates being missed.

Finally, the answer is obtained by subtracting the current reduction from `max_sum`.

## Go Solution

```go
package main

import (
	"container/heap"
	"sort"
)

type State struct {
	sum   int64
	index int
}

type MinHeap []State

func (h MinHeap) Len() int {
	return len(h)
}

func (h MinHeap) Less(i, j int) bool {
	return h[i].sum < h[j].sum
}

func (h MinHeap) Swap(i, j int) {
	h[i], h[j] = h[j], h[i]
}

func (h *MinHeap) Push(x interface{}) {
	*h = append(*h, x.(State))
}

func (h *MinHeap) Pop() interface{} {
	old := *h
	n := len(old)
	item := old[n-1]
	*h = old[:n-1]
	return item
}

func abs(x int) int64 {
	if x < 0 {
		return int64(-x)
	}
	return int64(x)
}

func kSum(nums []int, k int) int64 {
	var maxSum int64
	absNums := make([]int64, len(nums))

	for i, num := range nums {
		if num > 0 {
			maxSum += int64(num)
		}
		absNums[i] = abs(num)
	}

	sort.Slice(absNums, func(i, j int) bool {
		return absNums[i] < absNums[j]
	})

	if k == 1 {
		return maxSum
	}

	h := &MinHeap{}
	heap.Init(h)

	heap.Push(h, State{
		sum:   absNums[0],
		index: 0,
	})

	var reduction int64

	for i := 0; i < k-1; i++ {
		current := heap.Pop(h).(State)

		reduction = current.sum
		index := current.index

		if index+1 < len(absNums) {
			heap.Push(h, State{
				sum:   reduction + absNums[index+1],
				index: index + 1,
			})

			heap.Push(h, State{
				sum:   reduction - absNums[index] + absNums[index+1],
				index: index + 1,
			})
		}
	}

	return maxSum - reduction
}
```

The Go implementation follows the same logic as the Python version but requires explicit heap implementation using the `container/heap` package.

All arithmetic uses `int64` because subsequence sums can exceed 32-bit integer limits.

Unlike Python's built-in `heapq`, Go requires implementing the heap interface manually.

The sorting step requires `O(n log n)` time.

The heap performs at most `2k` insertions and `k` removals. Since `k <= 2000`, the heap operations are very manageable.

The space complexity comes primarily from the heap, which stores at most `O(k)` states simultaneously.

## Edge Cases

One important edge case occurs when all numbers are negative. In this situation, the maximum subsequence sum is produced by the empty subsequence, which has sum `0`. A common bug is incorrectly forcing at least one element into the subsequence. The implementation handles this naturally because it only adds positive numbers to `max_sum`.

Another important case is duplicate subsequence sums. Different subsequences may produce identical totals, and the problem explicitly states that duplicates count separately. The heap-based generation process preserves duplicates correctly because each state represents a distinct construction path.

Large integer values are also important. Since `nums[i]` may be as large as `10^9` and there may be up to `10^5` elements, sums can exceed 32-bit limits. The Go implementation uses `int64` throughout to prevent overflow issues.

A final edge case occurs when `k = 1`. In this case, the answer is simply the maximum subsequence sum itself. The implementation handles this immediately without entering the heap generation loop.
