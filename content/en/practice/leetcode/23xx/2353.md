## Solution

## Problem Understanding

This problem asks us to design a data structure that supports two operations efficiently:

1. Update the rating of a food item.
2. Retrieve the highest-rated food item for a given cuisine.

Each food belongs to exactly one cuisine and has a numeric rating. The challenge is that ratings can change dynamically, so the system must always be able to return the correct highest-rated food for each cuisine after updates.

The input consists of three parallel arrays:

- `foods[i]` represents the food name.
- `cuisines[i]` represents the cuisine type of that food.
- `ratings[i]` represents the initial rating of that food.

All food names are unique.

The `highestRated(cuisine)` operation must return:

- The food with the highest rating in that cuisine.
- If multiple foods share the same highest rating, return the lexicographically smallest food name.

The lexicographical tie-breaking rule is extremely important because it affects how we order foods internally.

The constraints are large enough that a naive solution becomes inefficient:

- Up to `2 * 10^4` foods
- Up to `2 * 10^4` operations total

A solution that scans all foods for every query could become quadratic in the worst case.

Several edge cases are important:

- Multiple foods can have the same rating.
- Ratings can increase or decrease.
- After updates, the highest-rated food may change.
- Tie-breaking must always use lexicographical order.
- The same food may be updated multiple times.
- Every cuisine is guaranteed to contain at least one food.

Because the operations are dynamic and frequent, we need a data structure that supports fast insertion, update, and retrieval of the maximum element with custom ordering.

## Approaches

### Brute Force Approach

The most straightforward solution is to store all food information in hash maps and, whenever `highestRated(cuisine)` is called, scan through every food belonging to that cuisine.

We can maintain:

- A map from food name to its current rating
- A map from cuisine to list of foods

For `changeRating`, we simply update the rating in the hash map.

For `highestRated`, we iterate through all foods in that cuisine and track:

- The largest rating seen so far
- The lexicographically smallest food among ties

This works correctly because we explicitly evaluate every candidate food.

However, this approach is too slow in the worst case. If one cuisine contains many foods and we repeatedly call `highestRated`, each query becomes `O(n)`.

With up to `2 * 10^4` operations, the worst-case complexity becomes too large.

### Optimal Approach

The key observation is that we repeatedly need the maximum-rated food for each cuisine. This is exactly the type of operation that priority queues, heaps, or ordered sets are designed for.

We can maintain:

- A hash map storing the current rating and cuisine for each food
- A max-heap for every cuisine

The heap entries will contain:

- Negative rating, because Python heaps are min-heaps
- Food name

The ordering becomes:

1. Higher rating first
2. Lexicographically smaller name first

This naturally satisfies the problem requirements.

The challenge is that heaps do not support efficient updates of arbitrary elements.

Instead of removing old entries, we use lazy deletion:

- Whenever a rating changes, insert a new heap entry
- Old entries remain in the heap temporarily
- During `highestRated`, repeatedly remove outdated entries until the top matches the current rating

This gives efficient amortized performance.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) per query | O(n) | Scans all foods in a cuisine |
| Optimal | O(log n) per update/query amortized | O(n) | Uses heaps with lazy deletion |

## Algorithm Walkthrough

1. Create a hash map called `food_info`.

This map stores the current state of every food:

```
food_info[food] = (cuisine, rating)
```

We need this because ratings change over time, and we must always know the latest value.
2. Create a heap for each cuisine.

Each cuisine maps to a priority queue containing:

```
(-rating, food_name)
```

We negate ratings because Python's `heapq` is a min-heap, but we need maximum ratings first.
3. Initialize the data structure.

For every food:

- Store its cuisine and rating in `food_info`
- Push `(-rating, food)` into that cuisine's heap
4. Handle `changeRating(food, newRating)`.

First retrieve the cuisine from `food_info`.

Then:

- Update the food's current rating in `food_info`
- Push a new heap entry with the updated rating

We do not remove the old heap entry because removing arbitrary heap elements efficiently is difficult.
5. Handle `highestRated(cuisine)`.

Look at the top of the cuisine heap.

The top may contain outdated information because older ratings remain in the heap.

Repeatedly:

- Check the heap top
- Compare it with the current rating stored in `food_info`
- If they do not match, pop the outdated entry
- Continue until the top entry is valid
6. Return the valid top food.

Once the top matches the current rating, it is guaranteed to be:

- Highest rated
- Lexicographically smallest among ties

### Why it works

The heap always contains every rating version ever inserted. Although outdated entries remain temporarily, the latest rating for each food is stored in `food_info`.

Whenever we query the heap top, we remove stale entries until the top corresponds to the current valid rating. Since heap ordering prioritizes higher ratings and lexicographically smaller names, the remaining top element is always the correct answer.

Each stale entry is removed at most once, so lazy deletion remains efficient overall.

## Python Solution

```python
from typing import List
from collections import defaultdict
import heapq

class FoodRatings:

    def __init__(self, foods: List[str], cuisines: List[str], ratings: List[int]):
        self.food_info = {}
        self.cuisine_heaps = defaultdict(list)

        for food, cuisine, rating in zip(foods, cuisines, ratings):
            self.food_info[food] = (cuisine, rating)
            heapq.heappush(
                self.cuisine_heaps[cuisine],
                (-rating, food)
            )

    def changeRating(self, food: str, newRating: int) -> None:
        cuisine, _ = self.food_info[food]

        self.food_info[food] = (cuisine, newRating)

        heapq.heappush(
            self.cuisine_heaps[cuisine],
            (-newRating, food)
        )

    def highestRated(self, cuisine: str) -> str:
        heap = self.cuisine_heaps[cuisine]

        while True:
            neg_rating, food = heap[0]
            current_cuisine, current_rating = self.food_info[food]

            if -neg_rating == current_rating:
                return food

            heapq.heappop(heap)

# Your FoodRatings object will be instantiated and called as such:
# obj = FoodRatings(foods, cuisines, ratings)
# obj.changeRating(food, newRating)
# param_2 = obj.highestRated(cuisine)
```

The implementation uses two primary data structures.

The `food_info` hash map stores the authoritative current state for every food. This allows us to verify whether a heap entry is still valid.

The `cuisine_heaps` structure stores one heap per cuisine. Each heap orders foods primarily by descending rating and secondarily by lexicographical order.

During initialization, every food is inserted into its cuisine heap.

When a rating changes, we update the current value in `food_info` and push a new heap entry. The old entry is intentionally left in the heap.

The `highestRated` method performs lazy deletion. It repeatedly checks whether the heap top matches the latest rating stored in `food_info`. If not, that heap entry is stale and removed.

This approach avoids expensive heap updates while still guaranteeing correctness.

## Go Solution

```go
package main

import (
	"container/heap"
)

type FoodEntry struct {
	rating int
	food   string
}

type FoodHeap []FoodEntry

func (h FoodHeap) Len() int {
	return len(h)
}

func (h FoodHeap) Less(i, j int) bool {
	if h[i].rating == h[j].rating {
		return h[i].food < h[j].food
	}

	return h[i].rating > h[j].rating
}

func (h FoodHeap) Swap(i, j int) {
	h[i], h[j] = h[j], h[i]
}

func (h *FoodHeap) Push(x interface{}) {
	*h = append(*h, x.(FoodEntry))
}

func (h *FoodHeap) Pop() interface{} {
	old := *h
	n := len(old)

	item := old[n-1]
	*h = old[:n-1]

	return item
}

type FoodData struct {
	cuisine string
	rating  int
}

type FoodRatings struct {
	foodInfo    map[string]FoodData
	cuisineHeap map[string]*FoodHeap
}

func Constructor(foods []string, cuisines []string, ratings []int) FoodRatings {
	fr := FoodRatings{
		foodInfo:    make(map[string]FoodData),
		cuisineHeap: make(map[string]*FoodHeap),
	}

	for i := 0; i < len(foods); i++ {
		food := foods[i]
		cuisine := cuisines[i]
		rating := ratings[i]

		fr.foodInfo[food] = FoodData{
			cuisine: cuisine,
			rating:  rating,
		}

		if _, exists := fr.cuisineHeap[cuisine]; !exists {
			h := &FoodHeap{}
			heap.Init(h)
			fr.cuisineHeap[cuisine] = h
		}

		heap.Push(fr.cuisineHeap[cuisine], FoodEntry{
			rating: rating,
			food:   food,
		})
	}

	return fr
}

func (this *FoodRatings) ChangeRating(food string, newRating int) {
	data := this.foodInfo[food]

	this.foodInfo[food] = FoodData{
		cuisine: data.cuisine,
		rating:  newRating,
	}

	heap.Push(this.cuisineHeap[data.cuisine], FoodEntry{
		rating: newRating,
		food:   food,
	})
}

func (this *FoodRatings) HighestRated(cuisine string) string {
	h := this.cuisineHeap[cuisine]

	for {
		top := (*h)[0]
		current := this.foodInfo[top.food]

		if current.rating == top.rating {
			return top.food
		}

		heap.Pop(h)
	}
}

/**
 * Your FoodRatings object will be instantiated and called as such:
 * obj := Constructor(foods, cuisines, ratings);
 * obj.ChangeRating(food, newRating);
 * param_2 := obj.HighestRated(cuisine);
 */
```

The Go implementation follows the same lazy deletion strategy as the Python version.

One important difference is that Go's `container/heap` package requires implementing the heap interface manually. We define:

- `Len`
- `Less`
- `Swap`
- `Push`
- `Pop`

The `Less` method is particularly important because it defines priority ordering. Higher ratings must appear first, and ties are resolved using lexicographical order.

Unlike Python tuples, Go requires explicit comparison logic.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(log n) amortized per operation | Heap insertion and lazy deletion |
| Space | O(n) | Maps and heap storage |

Each `changeRating` operation performs one heap insertion, which costs `O(log n)`.

The `highestRated` operation may remove stale heap entries. However, each stale entry is popped at most once during the entire execution, so the total cleanup work across all queries is bounded.

This makes the amortized complexity per operation `O(log n)`.

The space complexity is `O(n)` because:

- We store one current record per food
- Heaps may temporarily contain stale entries, but total entries remain proportional to the number of updates

## Edge Cases

### Multiple Foods With Identical Ratings

A common bug is forgetting the lexicographical tie-break rule. If two foods share the same rating, the result must be the alphabetically smaller name.

Our heap ordering handles this naturally because entries are stored as:

```
(-rating, food_name)
```

Python tuple comparison first compares rating and then food name automatically.

### Stale Heap Entries After Updates

When ratings change repeatedly, old heap entries remain inside the priority queue. If we simply returned the heap top without validation, we could return outdated results.

The implementation solves this using lazy deletion. Every time we inspect the heap top, we compare it against the authoritative value stored in `food_info`. Invalid entries are removed until a valid one remains.

### Repeated Updates To The Same Food

A food may be updated many times in succession. This creates many stale heap entries.

A naive implementation might try to remove old heap entries directly, which is inefficient in standard heaps.

Lazy deletion avoids this issue entirely. Even if multiple stale entries accumulate, each one is popped only once, preserving amortized efficiency.
