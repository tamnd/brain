## Solution

## Problem Understanding

The problem is asking us to determine how many nodes in a tree are reachable starting from node `0` without traversing any **restricted** nodes. The tree is represented as an undirected graph with `n` nodes labeled from `0` to `n-1` and `n-1` edges connecting them, guaranteeing that the graph is connected and acyclic. The input `edges` defines the tree structure, while the input `restricted` lists nodes that cannot be visited. The output is a single integer representing the count of nodes accessible from node `0` without entering restricted nodes.

Important points include that node `0` is never restricted, the tree is valid (so no cycles exist), and the input size can be large (`n` up to 10^5), which informs us that a naive or inefficient solution may not scale. Edge cases include trees where almost all nodes are restricted except for node `0`, or cases where restricted nodes are leaves versus internal nodes, as they may impact traversal paths differently.

## Approaches

The brute-force approach is to attempt all possible paths from node `0` using either Depth-First Search (DFS) or Breadth-First Search (BFS), marking visited nodes along the way and skipping any node in `restricted`. This will give the correct result but may involve unnecessary overhead if not optimized for large `n`, especially if the graph is traversed using recursive DFS without considering restricted nodes early, which could result in stack overflows or excessive recursion depth.

The key insight is that because the graph is a tree, any path from node `0` to another node is unique. This means we do not need to track complex path histories, and we can safely use a simple BFS or DFS that skips restricted nodes. Using a hash set to store restricted nodes allows for constant-time checks during traversal, significantly improving efficiency.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Standard DFS/BFS without optimization; correct but could be slow if recursion is deep |
| Optimal | O(n) | O(n) | BFS or DFS with early skipping of restricted nodes using a hash set; fully linear and safe for large n |

## Algorithm Walkthrough

1. Convert the `edges` list into an adjacency list representation of the tree. This allows constant-time access to neighbors of any node.
2. Convert the `restricted` list into a hash set for O(1) lookup during traversal.
3. Initialize a visited set to track nodes we have already counted, starting with node `0`.
4. Use a BFS queue (or DFS stack) initialized with node `0`.
5. While the queue is not empty, pop a node and examine its neighbors.
6. For each neighbor, if it is not in `restricted` and has not been visited, mark it as visited and add it to the queue.
7. Continue until all reachable nodes are processed.
8. Return the size of the visited set, which represents the total number of nodes reachable from node `0` without entering restricted nodes.

**Why it works:** The algorithm works because the tree structure guarantees a unique path to each node. By skipping restricted nodes and marking visited nodes, we avoid revisiting nodes and correctly count all nodes that can be reached starting from node `0`. The BFS ensures all nodes are explored in an orderly fashion without recursion depth issues.

## Python Solution

```python
from typing import List
from collections import deque, defaultdict

class Solution:
    def reachableNodes(self, n: int, edges: List[List[int]], restricted: List[int]) -> int:
        # Build adjacency list
        graph = defaultdict(list)
        for u, v in edges:
            graph[u].append(v)
            graph[v].append(u)
        
        restricted_set = set(restricted)
        visited = set([0])
        queue = deque([0])
        
        while queue:
            node = queue.popleft()
            for neighbor in graph[node]:
                if neighbor not in visited and neighbor not in restricted_set:
                    visited.add(neighbor)
                    queue.append(neighbor)
        
        return len(visited)
```

The code first converts the edge list to a graph for quick neighbor lookup. It converts the restricted list into a set for fast membership checks. Using BFS ensures that we traverse all nodes connected to node `0` while skipping restricted nodes. The visited set prevents counting a node multiple times. Finally, the number of visited nodes is returned.

## Go Solution

```go
package main

func reachableNodes(n int, edges [][]int, restricted []int) int {
    graph := make([][]int, n)
    for _, edge := range edges {
        u, v := edge[0], edge[1]
        graph[u] = append(graph[u], v)
        graph[v] = append(graph[v], u)
    }

    restrictedSet := make(map[int]bool)
    for _, r := range restricted {
        restrictedSet[r] = true
    }

    visited := make(map[int]bool)
    visited[0] = true
    queue := []int{0}

    for len(queue) > 0 {
        node := queue[0]
        queue = queue[1:]
        for _, neighbor := range graph[node] {
            if !visited[neighbor] && !restrictedSet[neighbor] {
                visited[neighbor] = true
                queue = append(queue, neighbor)
            }
        }
    }

    return len(visited)
}
```

The Go implementation mirrors the Python BFS approach. We use slices to store adjacency lists and a map to track restricted nodes and visited nodes. The BFS queue is implemented using a slice. Differences include explicit map initializations and the lack of built-in deque, but the logic is identical.

## Edge Cases

One important edge case is when the restricted nodes block all paths from node `0` except itself. This tests whether the algorithm correctly handles situations where traversal is immediately blocked. Another case is when restricted nodes are leaves. Although they do not block traversal to other nodes, the algorithm must not include them in the visited count. A third case involves consecutive restricted nodes in the middle of a branch. Here, BFS or DFS must properly stop traversal without attempting to explore beyond restricted nodes. In all cases, the visited set and restricted set ensure correct traversal and counting.
