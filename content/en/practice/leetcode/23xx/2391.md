## Solution

## Problem Understanding

This problem models a city garbage collection system with three separate garbage trucks. Each truck is responsible for exactly one type of garbage:

- `'M'` for metal
- `'P'` for paper
- `'G'` for glass

The input array `garbage` represents the garbage present at each house. For example, if `garbage[i] = "GMP"`, then house `i` contains one unit each of glass, metal, and paper garbage.

Each unit of garbage takes exactly `1` minute to collect, regardless of type. In addition to collection time, trucks also spend time traveling between houses. The array `travel` describes how long it takes to move from one house to the next:

- `travel[i]` is the time needed to go from house `i` to house `i + 1`

All three trucks start at house `0`. A truck only needs to travel as far as the last house containing the garbage type it collects. For example, if the last paper garbage appears at house `5`, then the paper truck never needs to travel beyond house `5`.

An important detail is that only one truck may operate at a time. However, this constraint does not actually change the final calculation because the total time is simply the sum of all collection and travel times performed by all trucks. Since no work can overlap, every minute spent by any truck contributes directly to the answer.

The goal is to compute the minimum total number of minutes required to collect all garbage.

The constraints are important:

- `garbage.length` can be as large as `10^5`
- Each garbage string has length at most `10`
- `travel.length == garbage.length - 1`

These limits tell us that the solution must be efficient, ideally linear time. A quadratic simulation would be too slow for the largest inputs.

There are several edge cases worth identifying immediately:

- A garbage type may not appear at all, meaning its truck never needs to travel.
- Garbage may appear only at house `0`, meaning no travel is required for that truck.
- The last occurrence of a garbage type may be very far into the array, requiring large travel accumulation.
- Multiple garbage types may exist in the same house.
- Since each garbage string length is small, processing each house individually is efficient.

## Approaches

### Brute Force Approach

A straightforward brute force solution would simulate each garbage truck independently.

For each truck type (`M`, `P`, `G`):

1. Traverse all houses from left to right.
2. At each house:

- Add collection time for matching garbage.
- Add travel time whenever the truck moves to the next house.
3. Stop after reaching the final house containing that garbage type.

To determine whether a future house contains the garbage type, the brute force solution may repeatedly scan ahead through the remaining houses. For every position, it checks whether more garbage of that type exists later.

This approach is correct because it explicitly models the truck movement and collection process exactly as described. However, repeated future scans create unnecessary work.

In the worst case, for every house we may scan almost the entire remaining array. With up to `10^5` houses, this becomes too slow.

### Optimal Approach

The key observation is that each truck only needs to travel up to the last house containing its garbage type.

This transforms the problem into two independent calculations:

1. Total garbage collection time
2. Total travel time needed by each truck

The collection time is easy:

- Every garbage character costs exactly `1` minute.
- Therefore, the total collection time is simply the total number of characters across all strings.

For travel time:

- Determine the last house index containing each garbage type.
- Compute prefix sums of the `travel` array.
- The travel time for a truck becomes the prefix travel distance to its final required house.

Using prefix sums allows constant time travel lookup after linear preprocessing.

This gives an efficient linear-time solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Repeatedly scans future houses to determine required travel |
| Optimal | O(n) | O(n) | Uses last occurrence tracking and prefix sums |

## Algorithm Walkthrough

1. Initialize variables to track:

- Total garbage collection time
- Last occurrence index for `M`, `P`, and `G`
2. Traverse the `garbage` array once.

- Add the length of each garbage string to the total answer because every character requires one minute to collect.
- For every character in the current house:

- Update the last seen index for that garbage type.
3. Build a prefix sum array for `travel`.

- `prefix[i]` represents total travel time needed to reach house `i`.
- This allows constant-time lookup of travel distance to any house.
4. For each garbage type:

- Find its last occurrence index.
- Add the corresponding prefix travel distance to the answer.
- If the type never appears, skip it.
5. Return the final accumulated answer.

### Why it works

The algorithm works because each truck only needs to visit houses up to the final house containing its garbage type. Any travel beyond that point would be unnecessary.

The total collection time is fixed because every garbage unit must be collected exactly once. The only optimization comes from minimizing travel distance, which is achieved automatically by stopping each truck at its last required house.

Prefix sums guarantee efficient computation of travel distances, and tracking last occurrences ensures that each truck travels only as far as necessary.

## Python Solution

```python
from typing import List

class Solution:
    def garbageCollection(self, garbage: List[str], travel: List[int]) -> int:
        total_time = 0

        last_position = {
            'M': 0,
            'P': 0,
            'G': 0
        }

        # Count garbage collection time
        # Track last occurrence of each garbage type
        for index, house_garbage in enumerate(garbage):
            total_time += len(house_garbage)

            for garbage_type in house_garbage:
                last_position[garbage_type] = index

        # Build prefix travel sums
        prefix_travel = [0] * len(garbage)

        for i in range(1, len(garbage)):
            prefix_travel[i] = prefix_travel[i - 1] + travel[i - 1]

        # Add required travel time for each truck
        for garbage_type in ['M', 'P', 'G']:
            total_time += prefix_travel[last_position[garbage_type]]

        return total_time
```

The implementation closely follows the algorithm described earlier.

The variable `total_time` stores the final answer. Initially, it accumulates only garbage collection time. Since each garbage character requires one minute, adding `len(house_garbage)` for every house gives the total pickup cost.

The dictionary `last_position` tracks the final house index containing each garbage type. While traversing the array, every occurrence updates the corresponding index.

The `prefix_travel` array is then constructed. Its purpose is to efficiently answer the question:

> How much travel time is needed to reach house `i` from house `0`?

This avoids repeatedly summing travel distances.

Finally, the algorithm adds the required travel distance for each truck. Since `prefix_travel[i]` already represents the total distance to house `i`, the lookup is constant time.

The result is returned after all collection and travel costs are accumulated.

## Go Solution

```go
func garbageCollection(garbage []string, travel []int) int {
	totalTime := 0

	lastPosition := map[byte]int{
		'M': 0,
		'P': 0,
		'G': 0,
	}

	// Count garbage collection time
	// Track last occurrence of each garbage type
	for i, houseGarbage := range garbage {
		totalTime += len(houseGarbage)

		for j := 0; j < len(houseGarbage); j++ {
			lastPosition[houseGarbage[j]] = i
		}
	}

	// Build prefix travel sums
	prefixTravel := make([]int, len(garbage))

	for i := 1; i < len(garbage); i++ {
		prefixTravel[i] = prefixTravel[i-1] + travel[i-1]
	}

	// Add required travel time for each truck
	totalTime += prefixTravel[lastPosition['M']]
	totalTime += prefixTravel[lastPosition['P']]
	totalTime += prefixTravel[lastPosition['G']]

	return totalTime
}
```

The Go implementation mirrors the Python approach almost exactly.

One difference is that Go strings are indexed as bytes, so garbage types are handled using `byte` keys rather than strings.

Slices are used for the prefix sum array. Since all values remain comfortably within standard integer limits, no special overflow handling is needed.

Unlike Python dictionaries, Go maps require explicit key types, so `map[byte]int` is used for tracking last positions.

## Edge Cases

One important edge case occurs when a garbage type never appears in the input. For example, if all houses contain only glass garbage, the metal and paper trucks should contribute zero travel time. A buggy implementation might incorrectly force every truck to traverse the route. This solution handles the case naturally because the last position for unused garbage types remains `0`, and `prefix_travel[0]` equals `0`.

Another tricky case is when all garbage is located at house `0`. In this situation, no truck needs to travel at all. The algorithm correctly computes only collection time because the prefix travel distance to house `0` is zero.

A third important case involves sparse garbage placement. For example, metal garbage might appear only in the first house while paper garbage appears only in the final house. An incorrect implementation could accidentally add unnecessary travel for the metal truck. By explicitly tracking the last occurrence of each garbage type, this solution ensures every truck travels only as far as required.

A final subtle edge case involves very large inputs. Since the number of houses can reach `10^5`, repeatedly summing travel distances inside loops would become inefficient. The prefix sum array prevents this issue by reducing every travel lookup to constant time.
