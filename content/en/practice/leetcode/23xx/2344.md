# LeetCode 2344 - Minimum Deletions to Make Array Divisible

## Problem Understanding

This problem gives us two integer arrays, `nums` and `numsDivide`. We are allowed to delete any number of elements from `nums`, and our goal is to make the smallest remaining element in `nums` divide every element in `numsDivide`.

The important detail is that only the smallest remaining element matters. Once deletions are complete, suppose the smallest remaining value is `x`. Then every value in `numsDivide` must satisfy:

$a_i \bmod x = 0$

for every element `a_i` in `numsDivide`.

We are asked to return the minimum number of deletions required to achieve this condition. If no such element exists in `nums`, we return `-1`.

The arrays can each contain up to `10^5` elements, and values can be as large as `10^9`. These constraints immediately rule out expensive quadratic algorithms. We need something close to linear or `O(n log n)`.

The key observation is that the smallest remaining element must divide every number in `numsDivide`. That means the value must be a common divisor of all elements in `numsDivide`.

Instead of checking divisibility against every element independently for every candidate, we can reduce the problem to computing the greatest common divisor of all values in `numsDivide`.

If a number divides every element in an array, then it must divide the array's overall GCD.

So the problem becomes:

1. Compute the GCD of all numbers in `numsDivide`
2. Find the smallest element in `nums` that divides this GCD
3. Count how many smaller elements must be deleted before that element becomes the minimum

Several edge cases are important:

- No element in `nums` divides the GCD of `numsDivide`, in this case the answer is `-1`
- Duplicate values appear frequently
- The valid divisor may not be unique
- The smallest value in `nums` may already work, giving answer `0`
- Large input sizes require efficient sorting and GCD computation

## Approaches

### Brute Force Approach

A brute force solution would try every possible element in `nums` as the candidate smallest remaining value.

For each candidate value:

1. Check whether it divides every element in `numsDivide`
2. If it works, determine how many deletions are needed so it becomes the smallest remaining element

To make it the smallest remaining element, we must delete every element smaller than it.

This approach is correct because it explicitly tests all valid possibilities. However, it is far too slow.

If `n = len(nums)` and `m = len(numsDivide)`, then for every element in `nums` we may scan all elements of `numsDivide`.

That leads to roughly:

$O(n \times m)$

With both arrays potentially having `10^5` elements, this could require `10^10` operations.

### Optimal Approach

The key mathematical insight is that a number divides all elements of an array if and only if it divides the GCD of the array.

Suppose:

$g = \gcd(numsDivide[0], numsDivide[1], \dots)$

Then any valid smallest element must divide `g`.

This reduces the problem dramatically:

1. Compute the GCD of all elements in `numsDivide`
2. Sort `nums`
3. Scan from smallest to largest
4. The first value that divides `g` is the optimal answer
5. Its index in the sorted array equals the number of deletions required

Why does the index equal the deletion count? Because every smaller element must be removed for this value to become the smallest remaining element.

This transforms the expensive repeated divisibility checks into a single GCD computation plus a sorted scan.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × m) | O(1) | Checks every candidate against all values |
| Optimal | O(n log n + m log V) | O(1) or O(log n) depending on sort | Uses GCD property and sorting |

Here, `V` represents the magnitude of the numbers involved in GCD calculations.

## Algorithm Walkthrough

1. Compute the GCD of all elements in `numsDivide`.

Start with the first value in `numsDivide`. Iteratively compute the GCD with each subsequent value. At the end, we obtain a single number `g` such that any valid answer must divide `g`.
2. Sort the `nums` array in ascending order.

Sorting allows us to consider candidate values from smallest to largest. This is important because we want the minimum deletions. The earlier a valid value appears in sorted order, the fewer elements must be deleted.
3. Scan through the sorted array.

For each value `nums[i]`, check whether:

$g \bmod nums[i] = 0$

If this condition is true, then `nums[i]` divides every element in `numsDivide`.
4. Return the index of the first valid element.

Since the array is sorted, every earlier element is smaller and must be deleted. Therefore the current index equals the minimum number of deletions.
5. If no valid divisor exists, return `-1`.

After scanning the entire sorted array, if no element divides `g`, then no solution exists.

### Why it works

The correctness comes from the fundamental property of the greatest common divisor.

A number divides every element in an array if and only if it divides the array's GCD. Therefore checking divisibility against the GCD is equivalent to checking divisibility against every element individually.

Sorting guarantees that the first valid divisor requires the fewest deletions, because all earlier elements are smaller and must be removed before this element can become the minimum remaining value.

## Python Solution

```python
from typing import List
from math import gcd

class Solution:
    def minOperations(self, nums: List[int], numsDivide: List[int]) -> int:
        common_gcd = numsDivide[0]

        for value in numsDivide[1:]:
            common_gcd = gcd(common_gcd, value)

        nums.sort()

        for index, value in enumerate(nums):
            if common_gcd % value == 0:
                return index

        return -1
```

The implementation begins by computing the GCD of all values in `numsDivide`. Python's built in `math.gcd` function efficiently handles this operation.

Next, the `nums` array is sorted in ascending order. This allows the algorithm to find the valid divisor requiring the fewest deletions.

The loop scans the sorted array from left to right. For each value, it checks whether the value divides the computed GCD. The first successful divisor immediately gives the answer because every earlier element would need to be deleted.

If the loop finishes without finding a valid divisor, the function returns `-1`.

## Go Solution

```go
package main

import "sort"

func gcd(a, b int) int {
	for b != 0 {
		a, b = b, a%b
	}
	return a
}

func minOperations(nums []int, numsDivide []int) int {
	commonGCD := numsDivide[0]

	for i := 1; i < len(numsDivide); i++ {
		commonGCD = gcd(commonGCD, numsDivide[i])
	}

	sort.Ints(nums)

	for i, value := range nums {
		if commonGCD%value == 0 {
			return i
		}
	}

	return -1
}
```

The Go implementation follows the same algorithmic structure as the Python version.

Since Go does not provide a built in GCD function in the standard integer utilities, we implement Euclid's algorithm manually. The rest of the solution uses `sort.Ints` to sort the slice and then scans for the first valid divisor.

Go slices are reference based, so sorting modifies the original slice in place. This is perfectly acceptable for the problem.

Integer overflow is not an issue because all values are within `10^9`, and GCD computations remain within integer bounds.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n + m log V) | Sorting dominates, GCD computation is linear |
| Space | O(1) or O(log n) | Depends on sorting implementation |

The GCD computation processes each element of `numsDivide` once. Each GCD operation takes logarithmic time relative to the number size.

Sorting `nums` costs `O(n log n)`, which dominates the total runtime.

The algorithm uses only a few extra variables beyond the input arrays. Depending on the language's sorting implementation, recursion stack usage may introduce logarithmic auxiliary space.

## Edge Cases

One important edge case occurs when no element in `nums` divides the GCD of `numsDivide`. A naive implementation might incorrectly return the number of deletions performed so far or assume some divisor must exist. The correct behavior is to scan the entire sorted array and return `-1` if no valid divisor is found.

Another important case is when the smallest element already satisfies the condition. In this scenario, no deletions are necessary. Because the algorithm scans the sorted array from smallest to largest, it immediately returns index `0`, which correctly represents zero deletions.

Duplicate values can also create subtle bugs. For example, if multiple copies of an invalid small number appear before the first valid divisor, every copy must be counted as a deletion. Sorting and returning the first valid index automatically handles duplicates correctly because each earlier occurrence contributes to the deletion count.

A final edge case involves very large integer values close to `10^9`. Some inefficient approaches repeatedly test divisibility against every value in `numsDivide`, which becomes expensive at scale. The GCD reduction avoids this issue entirely and keeps the algorithm efficient even for the maximum constraints.
