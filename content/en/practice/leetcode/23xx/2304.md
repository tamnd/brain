## Solution

## Problem Understanding

The problem gives us a matrix called `grid` with `m` rows and `n` columns. Every cell contains a unique integer from `0` to `m * n - 1`. We may start from any cell in the first row, and at every step we are allowed to move to any cell in the next row directly below.

The important detail is that moving between rows has an additional cost. This movement cost is defined by another matrix called `moveCost`.

If we are currently standing on a cell whose value is `v`, and we move to column `c` in the next row, then the movement cost is:

```
moveCost[v][c]
```

The total path cost consists of two parts:

1. The sum of all grid cell values visited.
2. The sum of all movement costs between rows.

We must compute the minimum possible total cost from any cell in the first row to any cell in the last row.

The constraints are relatively small:

- `2 <= m, n <= 50`
- At most `2500` cells exist in the grid.

Because every row can connect to every column in the next row, each cell has up to `n` outgoing transitions. A brute-force recursive exploration would quickly become expensive because the number of possible paths grows exponentially.

The problem guarantees:

- All grid values are distinct.
- Every value is valid as an index into `moveCost`.
- The movement matrix is properly sized.

These guarantees simplify implementation because we never need to validate indices dynamically.

Several edge cases are important:

- The minimum path may not start from the smallest number in the first row.
- A locally cheap movement may lead to a globally expensive path.
- Since movement costs depend on cell values instead of coordinates, we must always use the grid value as the lookup key.
- Multiple different paths may reach the same cell, so recomputation can become expensive without dynamic programming.

## Approaches

### Brute Force

A brute-force solution tries every possible path from the first row to the last row.

From each cell in row `r`, we recursively attempt transitions to all `n` cells in row `r + 1`. For every move, we add:

- The current cell value
- The movement cost
- The cost of the remaining path

This approach is correct because it explicitly evaluates every legal path and chooses the minimum.

However, it is too slow. At every row transition we have `n` choices, and there are `m - 1` transitions. Therefore the total number of paths is approximately:

```
n^(m-1)
```

With `m` and `n` up to `50`, this becomes completely infeasible.

### Optimal Dynamic Programming

The key observation is that many paths overlap.

If we already know the minimum cost to reach a particular cell in row `r`, then we do not need to recompute it again from scratch.

This naturally leads to dynamic programming.

We define:

```
dp[r][c]
```

as the minimum total cost to reach cell `(r, c)`.

To transition into a cell `(r, c)` from the previous row:

- Suppose we came from `(r - 1, prevCol)`
- Let the previous cell value be `prevValue`
- Then the transition cost is:

```
dp[r-1][prevCol]
+ moveCost[prevValue][c]
+ grid[r][c]
```

We compute the minimum over all possible previous columns.

Because each state depends only on the previous row, this becomes a classic layered dynamic programming problem.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^(m-1)) | O(m) | Explores every possible path recursively |
| Optimal Dynamic Programming | O(m * n²) | O(m * n) | Reuses overlapping subproblems |

## Algorithm Walkthrough

1. Create a DP table where `dp[r][c]` stores the minimum cost required to reach cell `(r, c)`.
2. Initialize the first row of the DP table. Since a path may start from any cell in the first row, the initial cost is simply the value of that cell itself.

```
dp[0][c] = grid[0][c]
```

1. Process the grid row by row starting from row `1`.
2. For each target cell `(r, c)` in the current row, consider every possible cell in the previous row.
3. Let:

- `prevValue = grid[r-1][prevCol]`
- Transition cost:

```
dp[r-1][prevCol]
+ moveCost[prevValue][c]
+ grid[r][c]
```

1. Take the minimum over all possible previous columns.
2. Store this minimum into `dp[r][c]`.
3. After processing all rows, the answer is the minimum value in the last DP row.

### Why it works

The DP state always stores the minimum possible cost to reach a specific cell. Every valid path to `(r, c)` must come from some cell in row `r - 1`, so checking all previous columns guarantees that we consider every legal path. Since we always keep only the minimum cost for each state, the final row contains the optimal costs for all possible ending positions. Taking the minimum among them produces the globally optimal path.

## Python Solution

```python
from typing import List

class Solution:
    def minPathCost(self, grid: List[List[int]], moveCost: List[List[int]]) -> int:
        rows = len(grid)
        cols = len(grid[0])

        dp = [[float("inf")] * cols for _ in range(rows)]

        # Initialize first row
        for col in range(cols):
            dp[0][col] = grid[0][col]

        # Fill DP table
        for row in range(1, rows):
            for col in range(cols):
                current_value = grid[row][col]

                for prev_col in range(cols):
                    prev_value = grid[row - 1][prev_col]

                    transition_cost = (
                        dp[row - 1][prev_col]
                        + moveCost[prev_value][col]
                        + current_value
                    )

                    dp[row][col] = min(dp[row][col], transition_cost)

        return min(dp[-1])
```

The implementation starts by determining the grid dimensions. We then allocate a DP table initialized with infinity because we are searching for minimum values.

The first row is initialized directly from the grid values because starting positions do not require any movement cost.

Next, we iterate through each remaining row. For every target cell, we try all possible previous columns. The previous cell's grid value determines which row in `moveCost` to use.

The transition formula combines:

- The best cost to reach the previous cell
- The movement cost
- The value of the current cell

We continuously update the minimum possible value for each DP state.

Finally, we return the smallest value in the last row because the path may end at any last-row column.

## Go Solution

```go
package main

import "math"

func minPathCost(grid [][]int, moveCost [][]int) int {
	rows := len(grid)
	cols := len(grid[0])

	dp := make([][]int, rows)
	for i := 0; i < rows; i++ {
		dp[i] = make([]int, cols)

		for j := 0; j < cols; j++ {
			dp[i][j] = math.MaxInt32
		}
	}

	// Initialize first row
	for col := 0; col < cols; col++ {
		dp[0][col] = grid[0][col]
	}

	// Fill DP table
	for row := 1; row < rows; row++ {
		for col := 0; col < cols; col++ {
			currentValue := grid[row][col]

			for prevCol := 0; prevCol < cols; prevCol++ {
				prevValue := grid[row-1][prevCol]

				transitionCost := dp[row-1][prevCol] +
					moveCost[prevValue][col] +
					currentValue

				if transitionCost < dp[row][col] {
					dp[row][col] = transitionCost
				}
			}
		}
	}

	answer := math.MaxInt32

	for col := 0; col < cols; col++ {
		if dp[rows-1][col] < answer {
			answer = dp[rows-1][col]
		}
	}

	return answer
}
```

The Go implementation follows the same dynamic programming logic as the Python solution.

One difference is initialization. Go does not provide an infinity integer value, so we use `math.MaxInt32` as a very large sentinel value.

Slices are used to construct the DP table dynamically. Since all values in the problem remain comfortably within 32-bit integer range, integer overflow is not a concern here.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m * n²) | For each cell, we examine all columns in the previous row |
| Space | O(m * n) | DP table stores one value per grid cell |

The algorithm processes every cell in the grid. For each cell, we iterate through all columns of the previous row to compute the best transition. Since there are `m * n` cells and each transition checks `n` possibilities, the total running time becomes `O(m * n²)`.

The DP table stores one minimum cost per cell, requiring `O(m * n)` extra memory.

## Edge Cases

One important edge case occurs when the smallest value in the first row does not belong to the optimal path. A greedy algorithm that always chooses the smallest immediate value would fail here because future movement costs can dominate the total. The DP solution handles this correctly because it evaluates every possible transition combination.

Another important edge case involves very large movement costs. Even if a destination cell contains a small value, the move into that cell may be extremely expensive. Since the transition formula explicitly includes both the movement cost and the destination value, the implementation always computes the true total path cost instead of optimizing only one component.

A third edge case is when multiple paths converge into the same cell. Without dynamic programming, the algorithm would repeatedly recompute the same subproblems, causing exponential runtime growth. The DP table solves this by storing the minimum cost for every state exactly once, ensuring efficient computation.

A final edge case involves the smallest allowed grid dimensions, specifically a `2 x 2` grid. In such cases there is only one row transition, but the algorithm still works correctly because the initialization and transition logic are uniform for all valid grid sizes.
