## Solution

## Problem Understanding

The problem asks us to maximize the revenue from cutting and selling a rectangular piece of wood with dimensions `m x n`. You are given a list of prices for specific rectangular subpieces of wood, where each price corresponds to a rectangle of height `hi` and width `wi`. You can cut the wood along horizontal or vertical lines, splitting it into smaller rectangles, and then sell any number of these subpieces according to the given prices. Importantly, you **cannot rotate pieces**, so a `2 x 3` piece cannot be sold as `3 x 2`. You can make as many cuts as needed, and selling multiple copies of the same shape is allowed.

The input represents the size of the wood and a list of available sale prices for particular rectangle sizes. The output is a single integer representing the maximum total revenue achievable.

The constraints indicate that the maximum dimensions for the wood are `200 x 200` and up to `20,000` price entries exist, which is important because it rules out any naive approach that tries every possible sequence of cuts in exponential time. Edge cases include the smallest wood (`1 x 1`), cases where no subpiece matches the available prices, or cases where the optimal solution involves multiple levels of recursive cuts.

## Approaches

The brute-force approach involves recursively trying all possible vertical and horizontal cuts for a given `m x n` rectangle, then recursively computing the maximum revenue for the resulting two rectangles and taking the sum. If a piece matches a price directly, we could also consider selling it without cutting. While this approach is correct, it is too slow due to overlapping subproblems and the exponential number of possible cuts.

The key observation that leads to an optimal solution is **dynamic programming with memoization**. We notice that the maximum revenue for a given rectangle depends only on its dimensions. Thus, we can store results for all rectangles we have already computed, avoiding redundant computations. We also only need to consider the possible cuts along each dimension separately, which reduces the problem to a manageable `O(m * n * (m + n))` complexity.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^(m+n)) | O(m*n) recursion | Tries every cut recursively without caching results |
| Optimal (DP + Memoization) | O(m * n * (m + n)) | O(m * n) | Stores max revenue for each rectangle to avoid recomputation |

## Algorithm Walkthrough

1. First, convert the `prices` list into a lookup dictionary `price_map` where the key is a tuple `(hi, wi)` and the value is `pricei`. This allows constant-time retrieval of the price of a rectangle if it exists.
2. Initialize a memoization dictionary `dp` to store the maximum revenue for each rectangle `(h, w)`.
3. Define a recursive helper function `maxRevenue(h, w)` that returns the maximum revenue for a rectangle of height `h` and width `w`.
4. In `maxRevenue(h, w)`, start with `res = price_map.get((h, w), 0)`, representing the revenue if we sell the piece without cutting.
5. Loop over all possible vertical cuts from `1` to `w - 1`. For each cut at position `i`, split the rectangle into `(h, i)` and `(h, w - i)` and update `res = max(res, maxRevenue(h, i) + maxRevenue(h, w - i))`.
6. Loop over all possible horizontal cuts from `1` to `h - 1`. For each cut at position `j`, split the rectangle into `(j, w)` and `(h - j, w)` and update `res = max(res, maxRevenue(j, w) + maxRevenue(h - j, w))`.
7. Store `res` in `dp[(h, w)]` and return it.
8. Finally, call `maxRevenue(m, n)` to get the answer.

Why it works: The recursive function explores all ways to cut the rectangle either horizontally or vertically, while memoization ensures that each subrectangle is computed only once. Since each rectangle is considered exactly once and all combinations of cuts are checked, this guarantees the global maximum revenue.

## Python Solution

```python
from typing import List

class Solution:
    def sellingWood(self, m: int, n: int, prices: List[List[int]]) -> int:
        price_map = {(h, w): p for h, w, p in prices}
        dp = {}
        
        def maxRevenue(h: int, w: int) -> int:
            if (h, w) in dp:
                return dp[(h, w)]
            
            res = price_map.get((h, w), 0)
            
            # Vertical cuts
            for i in range(1, w):
                res = max(res, maxRevenue(h, i) + maxRevenue(h, w - i))
            
            # Horizontal cuts
            for j in range(1, h):
                res = max(res, maxRevenue(j, w) + maxRevenue(h - j, w))
            
            dp[(h, w)] = res
            return res
        
        return maxRevenue(m, n)
```

The Python implementation follows the algorithm step by step. We first prepare the price dictionary for O(1) lookups. The recursive function handles both horizontal and vertical cuts and uses memoization to store results, ensuring efficiency.

## Go Solution

```go
func sellingWood(m int, n int, prices [][]int) int64 {
    priceMap := make(map[[2]int]int)
    for _, p := range prices {
        priceMap[[2]int{p[0], p[1]}] = p[2]
    }
    
    dp := make(map[[2]int]int)
    
    var maxRevenue func(int, int) int
    maxRevenue = func(h, w int) int {
        key := [2]int{h, w}
        if val, exists := dp[key]; exists {
            return val
        }
        
        res := priceMap[key]
        
        for i := 1; i < w; i++ {
            left := maxRevenue(h, i)
            right := maxRevenue(h, w - i)
            if left + right > res {
                res = left + right
            }
        }
        
        for j := 1; j < h; j++ {
            top := maxRevenue(j, w)
            bottom := maxRevenue(h - j, w)
            if top + bottom > res {
                res = top + bottom
            }
        }
        
        dp[key] = res
        return res
    }
    
    return int64(maxRevenue(m, n))
}
```

The Go version is similar to Python but uses maps with array keys and integer type conversions to handle `int64` for large sums. The recursion and memoization logic is identical.

## Worked Examples

**Example 1: m = 3, n = 5, prices = [[1,4,2],[2,2,7],[2,1,3]]**

1. Start with `maxRevenue(3, 5)`.
2. Check price map: `(3,5)` not found, start with `res = 0`.
3. Try vertical cuts: `(3,1)+(3,4)`, `(3,2)+(3,3)`, `(3,3)+(3,2)`, `(3,4)+(3,1)` recursively computing each.
4. Try horizontal cuts: `(1,5)+(2,5)`, `(2,5)+(1,5)`.
5. Each recursive call uses memoization to avoid recomputation.
6. Optimal cuts yield two `2x2` pieces (14), one `2x1` piece (3), one `1x4` piece (2). Total = 19.

**Example 2: m = 4, n = 6, prices = [[3,2,10],[1,4,2],[4,1,3]]**

1. Start with `maxRevenue(4,6)`.
2. Optimal solution involves three `3x2` pieces (30) and one `1x4` piece (2). Total = 32.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m * n * (m + n)) | For each rectangle `h x w`, we try all vertical and horizontal cuts. Memoization ensures each rectangle is processed once. |
| Space | O(m * n) | Memoization dictionary stores results for all subrectangles. |

The time complexity is dominated by iterating over all possible cuts for each rectangle. The space complexity is linear in the number of distinct rectangles.
