# LeetCode 2368 - Reachable Nodes With Restrictions

## Problem Understanding

This problem gives us an undirected tree with `n` nodes numbered from `0` to `n - 1`. A tree is a connected graph with exactly `n - 1` edges and no cycles. Each edge connects two nodes bidirectionally.

We are also given a list called `restricted`, which contains nodes that are forbidden to visit. We start from node `0`, and we want to determine how many nodes can be reached without ever stepping onto a restricted node.

The key detail is that restricted nodes completely block traversal. Even if a restricted node connects to many other unrestricted nodes, we are not allowed to move through it. Since the graph is a tree, there is exactly one path between any two nodes. That property simplifies the traversal significantly.

The input consists of:

- `n`, the number of nodes
- `edges`, the tree connections
- `restricted`, the nodes we cannot visit

The output is a single integer representing the total number of reachable nodes starting from node `0`.

The constraints are important:

- `n` can be as large as `10^5`
- The graph is guaranteed to be a valid tree
- Restricted nodes are unique
- Node `0` is never restricted

These constraints immediately rule out inefficient repeated traversals or quadratic approaches. We need a solution that runs in approximately linear time.

Several edge cases are important:

- A restricted node may be directly connected to node `0`
- Almost all nodes may be restricted
- The tree may be highly unbalanced, like a linked list
- The tree may be very wide, like a star centered at node `0`
- Some unrestricted nodes may become unreachable because every path to them passes through a restricted node

Because the graph is a tree, we do not need to worry about multiple paths to the same node. A standard graph traversal with proper visited tracking is sufficient.

## Approaches

### Brute Force Approach

A naive strategy would be to attempt to determine reachability for every node individually.

For each node from `0` to `n - 1`, we could perform a DFS or BFS starting from node `0` and check whether that node can be reached without entering a restricted node. Since each traversal may visit nearly all nodes, and we repeat this for every node, the total complexity becomes very large.

This approach is correct because each search independently determines whether a node is reachable. However, it is extremely inefficient.

If each traversal costs `O(n)` and we perform it for all nodes, the total runtime becomes `O(n^2)`, which is too slow for `n = 10^5`.

### Optimal Approach

The important observation is that we do not need to compute reachability separately for every node.

Instead, we can perform a single traversal starting from node `0`. During the traversal:

- If we encounter a restricted node, we stop exploring that branch
- Otherwise, we count the node and continue traversing its neighbors

Since the graph is a tree, every node is visited at most once. We can use either DFS or BFS. Both work equally well here.

To support efficient restricted-node lookup, we store the restricted nodes in a hash set. That allows constant-time checks while traversing.

We also build an adjacency list representation of the tree so we can efficiently access neighbors.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeated traversal for each node |
| Optimal | O(n) | O(n) | Single DFS/BFS traversal with restricted filtering |

## Algorithm Walkthrough

### Optimal DFS Algorithm

1. Create an adjacency list for the tree.

Since the graph is undirected, every edge `[u, v]` is added in both directions. This structure allows efficient neighbor traversal.
2. Convert the `restricted` array into a hash set.

We need fast membership checks while traversing. A hash set provides average `O(1)` lookup time.
3. Initialize a visited set.

Even though the graph is a tree, using a visited set prevents revisiting parent nodes during DFS traversal.
4. Start DFS from node `0`.

Node `0` is guaranteed not to be restricted, so traversal always begins there.
5. For each visited node:

- Mark it as visited
- Count it as reachable
- Explore all neighbors
6. Before visiting a neighbor, check:

- Has it already been visited?
- Is it restricted?

If either condition is true, skip that neighbor.
7. Continue recursively until all reachable unrestricted nodes have been explored.
8. Return the total count.

### Why it works

The algorithm explores exactly the connected component reachable from node `0` after removing all restricted nodes from the tree.

Because the graph is a tree, every node has exactly one unique path from node `0`. If that path passes through a restricted node, the node is unreachable and will never be visited. Otherwise, DFS eventually reaches it.

The traversal therefore counts precisely the set of valid reachable nodes.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Solution:
    def reachableNodes(self, n: int, edges: List[List[int]], restricted: List[int]) -> int:
        graph = defaultdict(list)

        for u, v in edges:
            graph[u].append(v)
            graph[v].append(u)

        restricted_set = set(restricted)
        visited = set()

        def dfs(node: int) -> int:
            visited.add(node)

            reachable_count = 1

            for neighbor in graph[node]:
                if neighbor in visited:
                    continue

                if neighbor in restricted_set:
                    continue

                reachable_count += dfs(neighbor)

            return reachable_count

        return dfs(0)
```

The implementation begins by constructing an adjacency list using a `defaultdict(list)`. Every edge is inserted in both directions because the tree is undirected.

The restricted nodes are converted into a set called `restricted_set`. This allows constant-time filtering during traversal.

The `visited` set ensures that we do not revisit nodes that were already processed. Although trees do not contain cycles, undirected edges would otherwise cause DFS to immediately move back to the parent node.

The recursive `dfs` function counts the current node, then recursively explores all valid neighbors. A neighbor is considered valid only if:

- It has not already been visited
- It is not restricted

Each recursive call returns the number of reachable nodes in that subtree, and the totals are accumulated.

Finally, the traversal starts from node `0`, and the total reachable count is returned.

## Go Solution

```go
func reachableNodes(n int, edges [][]int, restricted []int) int {
    graph := make([][]int, n)

    for _, edge := range edges {
        u, v := edge[0], edge[1]
        graph[u] = append(graph[u], v)
        graph[v] = append(graph[v], u)
    }

    restrictedSet := make(map[int]bool)

    for _, node := range restricted {
        restrictedSet[node] = true
    }

    visited := make([]bool, n)

    var dfs func(int) int

    dfs = func(node int) int {
        visited[node] = true

        count := 1

        for _, neighbor := range graph[node] {
            if visited[neighbor] {
                continue
            }

            if restrictedSet[neighbor] {
                continue
            }

            count += dfs(neighbor)
        }

        return count
    }

    return dfs(0)
}
```

The Go implementation follows the same algorithmic structure as the Python solution.

Instead of using dictionaries of lists, Go uses a slice of slices for the adjacency list. Since node labels range from `0` to `n - 1`, direct indexing is efficient.

The restricted nodes are stored in a `map[int]bool`, which functions similarly to a hash set.

Visited tracking uses a boolean slice rather than a hash set because the node range is fixed and contiguous.

The DFS function is implemented as a recursive closure. It returns the number of reachable nodes in the current subtree.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each node and edge is processed at most once |
| Space | O(n) | Adjacency list, visited set, and recursion stack |

The adjacency list contains `2 * (n - 1)` total entries because every undirected edge is stored twice.

The DFS traversal visits each reachable node exactly once. Since the graph is a tree, the total number of edges is linear in `n`, so traversal complexity is also linear.

The recursion stack may reach depth `O(n)` in the worst case for a highly skewed tree.

## Edge Cases

One important edge case occurs when a restricted node is directly connected to node `0`. A buggy implementation might accidentally enqueue or recurse into that node before checking restrictions. The provided solution checks whether a neighbor is restricted before traversal, so those nodes are never visited.

Another important case is a chain-like tree where a restricted node appears in the middle. Since trees have unique paths, every node beyond that restricted node becomes unreachable. The DFS naturally handles this because recursion stops at the restricted node, preventing traversal into deeper descendants.

A third important case is when there are no restricted nodes at all. Some implementations incorrectly assume the restricted list is always non-empty. This solution converts the list into a set regardless of size, and traversal simply proceeds through the entire tree without special handling.

A final edge case is a highly unbalanced tree with depth close to `n`. Recursive DFS works correctly logically, but in Python, extremely deep recursion could approach recursion limits. On LeetCode, recursive DFS is generally accepted for this problem, but an iterative DFS or BFS could also be used if recursion depth becomes a concern.
