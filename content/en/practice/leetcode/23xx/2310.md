# LeetCode 2310 - Sum of Numbers With Units Digit K

## Problem Understanding

The problem gives us two integers, `num` and `k`. We must construct a set of positive integers such that every number in the set ends with the digit `k`, and the total sum of all numbers equals `num`.

Our goal is not just to find any valid set, but to find the minimum possible number of elements in such a set. If no valid set exists, we return `-1`.

The important observation is that only the units digit matters for determining whether a combination can produce the required sum. Every valid number has the form:

$$10x + k$$

where `x >= 0`.

When several such numbers are added together, the units digit of the total depends only on how many numbers we used. Specifically, if we use `count` numbers, then the units digit of the sum becomes:

$$(count \times k) \bmod 10$$

So the task reduces to finding the smallest positive integer `count` such that:

$$(count \times k) \bmod 10 = num \bmod 10$$

and also:

$$count \times k \le num$$

because every chosen number must be positive.

The constraints are small:

- `0 <= num <= 3000`
- `0 <= k <= 9`

This immediately suggests that a brute-force search over small candidate counts is completely feasible.

There are several important edge cases:

- If `num == 0`, the answer is always `0`, because the empty set is allowed.
- If `k == 0`, every number must end in `0`, so every chosen number is divisible by `10`. In this case, only sums ending in `0` are possible.
- Some combinations of `num` and `k` can never match the required units digit. For example, using numbers ending in `2` can never produce a sum ending in `7`.

## Approaches

### Brute Force Approach

A direct brute-force approach would attempt to generate combinations of numbers ending in `k` and check whether their sum equals `num`.

For example, if `k = 9`, we could try numbers like:

- `9`
- `19`
- `29`
- `39`

and recursively search for combinations that add up to `num`.

This approach is correct because it explores all possible valid combinations. However, it becomes inefficient because the number of combinations grows rapidly as `num` increases. Even with memoization or dynamic programming, we would still be exploring many unnecessary states.

A slightly improved brute-force dynamic programming solution could define:

$$dp[i] = \text{minimum numbers needed to form sum } i$$

and transition using all valid numbers ending in `k`.

Although this works within the constraints, it is more complicated than necessary.

### Key Insight

The critical observation is that only the units digit matters.

Every valid number contributes a units digit of `k`. If we use `count` numbers, then the total units digit becomes:

$$(count \times k) \bmod 10$$

Therefore, we only need to find the smallest `count` satisfying:

$$(count \times k) \bmod 10 = num \bmod 10$$

Once this condition holds, we can always construct actual numbers whose total sum is `num`.

Another important mathematical property is that only the last digit cycles matter, and there are only 10 possible last digits. Therefore, checking counts from `1` to `10` is sufficient.

This transforms the problem into a tiny enumeration problem.

## Algorithm Walkthrough

### Optimal Algorithm

1. First, check whether `num` is `0`.

If `num == 0`, return `0` immediately because the empty set already satisfies the condition.
2. Enumerate possible set sizes from `1` to `10`.

We only need to check the first 10 counts because units digits repeat every 10 multiplications.
3. For each candidate count, compute:

$$(count \times k) \bmod 10$$

This tells us the units digit produced by using `count` numbers ending in `k`.

1. Compare this value with:

$$num \bmod 10$$

If they match, then the units digit requirement is satisfied.

1. Also verify that:

$$count \times k \le num$$

This ensures the total sum is large enough to actually build `num` using positive integers.

1. Return the first valid count.

Since we check counts in increasing order, the first valid one is automatically the minimum.
2. If no count from `1` to `10` works, return `-1`.

### Why it works

The algorithm works because every valid number contributes the same units digit `k`. Therefore, only the number of elements affects the final units digit.

Since decimal units digits repeat modulo `10`, checking more than 10 counts is unnecessary. If no valid count appears within one cycle, none will ever appear later.

Once the units digit matches and the sum is large enough, we can distribute the remaining value in multiples of `10`, which preserves the units digit of each number.

## Python Solution

```python
class Solution:
    def minimumNumbers(self, num: int, k: int) -> int:
        if num == 0:
            return 0

        for count in range(1, 11):
            if (count * k) % 10 == num % 10 and count * k <= num:
                return count

        return -1
```

The implementation begins with the special case where `num` is zero. Since the empty set is allowed, we immediately return `0`.

Next, we iterate through candidate set sizes from `1` to `10`. For each candidate count, we compute the resulting units digit using modular arithmetic.

The condition:

```python
(count * k) % 10 == num % 10
```

checks whether the units digit matches the target sum.

The additional condition:

```python
count * k <= num
```

ensures the total can actually reach `num`. Without this check, we might incorrectly accept impossible cases where the minimum achievable sum already exceeds `num`.

Because the loop runs in increasing order, the first valid answer is guaranteed to be minimal.

If the loop finishes without finding a solution, the function returns `-1`.

## Go Solution

```go
func minimumNumbers(num int, k int) int {
    if num == 0 {
        return 0
    }

    for count := 1; count <= 10; count++ {
        if (count*k)%10 == num%10 && count*k <= num {
            return count
        }
    }

    return -1
}
```

The Go implementation follows exactly the same logic as the Python solution.

Go uses a standard `for` loop instead of Python's `range`. Integer arithmetic behaves safely here because the constraints are very small, so overflow is not a concern.

No additional data structures are needed, so the implementation remains extremely compact and efficient.

## Edge Cases

### Edge Case 1, `num = 0`

This is the most important special case. The problem explicitly allows the empty set, whose sum is `0`.

A naive implementation might continue searching for numbers and incorrectly return `-1`. The correct behavior is to immediately return `0`.

The implementation handles this with:

```
if num == 0:
    return 0
```

### Edge Case 2, `k = 0`

When `k = 0`, every chosen number must end in `0`. This means every valid number is divisible by `10`.

As a result, only sums ending in `0` are possible. Inputs like:

```
num = 1, k = 0
```

must return `-1`.

The modular arithmetic check naturally handles this case correctly.

### Edge Case 3, Units Digit Never Appears

Some values of `k` generate only a subset of possible units digits.

For example, repeated addition of numbers ending in `2` produces:

```
2, 4, 6, 8, 0
```

The digit `7` can never appear.

A brute-force implementation might waste time searching large combinations. The optimal solution detects impossibility quickly by checking all 10 modular outcomes.

### Edge Case 4, Minimum Sum Exceeds `num`

Even if the units digits match, the sum may still be impossible because the smallest achievable total already exceeds `num`.

For example:

```
num = 5, k = 9
```

Using one number ending in `9` already gives a minimum value of `9`, which exceeds `5`.

The condition:

```
count * k <= num
```

prevents incorrect acceptance of such cases.
