# LeetCode 2393 - Count Strictly Increasing Subarrays

## Problem Understanding

The problem gives us an integer array `nums` and asks us to count how many contiguous subarrays are strictly increasing.

A subarray is any continuous segment of the array. For example, in `[1,2,3]`, the subarrays include:

- `[1]`
- `[2]`
- `[3]`
- `[1,2]`
- `[2,3]`
- `[1,2,3]`

A subarray is considered strictly increasing if every element is larger than the previous one. In other words, for every adjacent pair inside the subarray:

$$nums[i] < nums[i+1]$$

The task is to count all such subarrays.

The constraints are important:

- `nums.length` can be as large as `10^5`
- Each value can be up to `10^6`

The large input size tells us immediately that an algorithm worse than `O(n^2)` will likely time out. Since there can already be roughly `5 * 10^9` subarrays in the worst case, we cannot explicitly generate and test every one of them for large inputs.

There are several important edge cases to keep in mind:

- Arrays with only one element
- Arrays where no adjacent pair is increasing
- Arrays where every element is increasing
- Arrays containing equal adjacent values, since equality breaks strict increasing order
- Arrays that alternate between increasing and non-increasing sections

The problem guarantees that all numbers are positive integers, but that detail does not actually affect the core algorithm. What matters is the comparison between neighboring elements.

## Approaches

### Brute Force Approach

A straightforward solution is to generate every possible subarray and check whether it is strictly increasing.

For each starting index `i`, we can extend the subarray one element at a time toward the right. While extending, we check whether the newly added element is greater than the previous element. If it is, the subarray remains strictly increasing and we increment the answer. Otherwise, we stop extending because any longer subarray starting at `i` will also fail.

This approach is correct because it explicitly examines every candidate subarray and counts exactly those satisfying the condition.

However, the worst case occurs when the array is fully increasing. In that case, for each starting position we scan almost the entire remaining array. The total work becomes:

$$1 + 2 + 3 + \dots + n = O(n^2)$$

With `n = 10^5`, this is far too slow.

### Optimal Approach

The key observation is that we do not need to examine every subarray independently.

Suppose we know the length of the current strictly increasing streak ending at index `i`.

For example:

```
nums = [1,2,3,2]
```

At index:

- `0`, streak length = 1
- `1`, streak length = 2
- `2`, streak length = 3
- `3`, streak length = 1

Why does this help?

If the increasing streak ending at index `i` has length `L`, then exactly `L` strictly increasing subarrays end at `i`.

For example, if the streak length is `3`, the valid subarrays ending there are:

- length 1
- length 2
- length 3

So instead of checking all subarrays individually, we only need to maintain the current increasing streak length.

If:

$$nums[i] > nums[i-1]$$

then the streak extends:

$$current += 1$$

Otherwise, the streak resets to `1`.

At every index, we add the streak length to the answer.

This gives a linear `O(n)` solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Checks every possible increasing subarray explicitly |
| Optimal | O(n) | O(1) | Tracks length of increasing streak ending at each index |

## Algorithm Walkthrough

1. Initialize a variable `total` to store the final count of strictly increasing subarrays.
2. Initialize another variable `current_length` to `1`.

Every individual element forms a strictly increasing subarray of length `1`.
3. Add `1` to the answer for the first element.
4. Iterate through the array starting from index `1`.
5. For each index `i`, compare `nums[i]` with `nums[i-1]`.
6. If `nums[i] > nums[i-1]`, the increasing streak continues.

Increment:

```
current_length += 1
```
7. Otherwise, the increasing streak breaks.

Reset:

```
current_length = 1
```
8. Add `current_length` to `total`.

This works because every suffix of the current increasing streak ending at `i` is itself strictly increasing.
9. After processing the entire array, return `total`.

### Why it works

The algorithm maintains the invariant that `current_length` equals the number of consecutive increasing elements ending at the current index.

If the current streak length is `L`, then there are exactly `L` strictly increasing subarrays ending at that position, one for each possible starting point within the streak.

Since every strictly increasing subarray ends at exactly one index, summing these values across all indices counts every valid subarray exactly once.

## Python Solution

```python
from typing import List

class Solution:
    def countSubarrays(self, nums: List[int]) -> int:
        total = 1
        current_length = 1

        for i in range(1, len(nums)):
            if nums[i] > nums[i - 1]:
                current_length += 1
            else:
                current_length = 1

            total += current_length

        return total
```

The implementation starts by initializing `total` and `current_length` to `1` because the first element itself forms a valid strictly increasing subarray.

The loop begins from index `1` since each element is compared with its predecessor.

Whenever the current value is greater than the previous value, the increasing sequence continues, so the streak length increases by one.

If the condition fails, the streak breaks and resets to `1`, representing the single-element subarray containing only the current number.

At every iteration, the current streak length is added to the total answer because it represents the number of strictly increasing subarrays ending at the current index.

The solution uses only constant extra space and processes the array exactly once.

## Go Solution

```go
func countSubarrays(nums []int) int64 {
    var total int64 = 1
    var currentLength int64 = 1

    for i := 1; i < len(nums); i++ {
        if nums[i] > nums[i-1] {
            currentLength++
        } else {
            currentLength = 1
        }

        total += currentLength
    }

    return total
}
```

The Go implementation follows the same logic as the Python version.

One important detail is the use of `int64` for the answer. The number of subarrays can be as large as:

$$\frac{n(n+1)}{2}$$

For `n = 100000`, this exceeds the range of a 32-bit integer, so `int64` is necessary.

The algorithm still uses constant extra space and performs a single linear scan.

## Edge Cases

### Single Element Array

An array containing only one element is an important boundary case. A naive implementation might accidentally initialize counters incorrectly or skip processing entirely.

For example:

```
[7]
```

There is exactly one strictly increasing subarray, the element itself.

The implementation handles this correctly because `total` and `current_length` both start at `1`, and the loop does not execute.

### All Elements Equal

Equal adjacent elements do not satisfy strict inequality.

For example:

```
[2,2,2]
```

Only the single-element subarrays are valid.

A common bug is using `>=` instead of `>` when checking increasing order. The implementation correctly uses:

```
nums[i] > nums[i - 1]
```

so equal values reset the streak immediately.

### Fully Increasing Array

This is the worst-case scenario for the answer size.

For example:

```
[1,2,3,4,5]
```

Every possible subarray is strictly increasing.

The number of subarrays becomes:

$$\frac{n(n+1)}{2}$$

The implementation handles this efficiently in linear time by growing the streak length continuously instead of explicitly enumerating all subarrays.

### Fully Decreasing Array

In a strictly decreasing array:

```
[5,4,3,2]
```

No multi-element subarray is valid.

The algorithm correctly resets the streak length to `1` at every step, so the answer becomes exactly the number of elements.
