**Difficulty:** 🟡 Medium  
**Topics:** Array, Two Pointers, Binary Search, Sorting

## Solution

## Problem Understanding

The problem gives us two arrays:

- `spells`, where each value represents the strength of a spell
- `potions`, where each value represents the strength of a potion

A pair `(spell, potion)` is considered successful if:

$spell \times potion \ge success$

For every spell in the `spells` array, we must determine how many potions can form a successful pair with it.

The output should be an array `pairs` where:

- `pairs[i]` equals the number of potions that work successfully with `spells[i]`

The important detail is that every spell can pair with every potion independently. We are not removing potions after use, and multiple spells may count the same potion.

The constraints are large:

- `n, m <= 10^5`

A naive nested loop over all spell and potion combinations would require:

$O(n \times m)$

In the worst case, that becomes:

$10^5 \times 10^5 = 10^{10}$

which is far too slow.

Another important constraint is:

- `success <= 10^10`

Since multiplication can produce large numbers, integer overflow must be considered in languages like Go. Python handles large integers automatically, but Go requires explicit use of `int64`.

Several edge cases are important:

- A spell may be too weak to form a successful pair with any potion
- A spell may succeed with every potion
- Potion values may contain duplicates
- The arrays may already be sorted or completely unsorted
- Very large multiplication results must be handled safely

The problem guarantees:

- All numbers are positive integers
- Arrays are non-empty

The positivity guarantee is especially useful because it allows binary search logic based on monotonic ordering.

## Approaches

### Brute Force Approach

The most direct solution is to try every possible spell and potion combination.

For each spell:

1. Iterate through every potion
2. Compute the product
3. Check whether the product is at least `success`
4. Count successful pairs

This approach is correct because it explicitly evaluates every valid combination. Nothing is skipped, so the result is guaranteed to be accurate.

However, the runtime is too large. If both arrays contain `10^5` elements, we perform:

$10^5 \times 10^5 = 10^{10}$

operations, which is infeasible.

### Optimal Approach

The key observation is that after sorting the `potions` array, the successful potions for a given spell form a contiguous suffix.

Suppose the potions are sorted:

```
[1, 2, 3, 4, 5]
```

For spell `3` and success `7`:

```
3*1 = 3
3*2 = 6
3*3 = 9
3*4 = 12
3*5 = 15
```

Once we reach a potion that works, every larger potion also works because all values are positive and the array is sorted.

That means we only need to find the first valid potion index. Binary search is ideal for this.

For each spell:

1. Compute the minimum potion strength needed
2. Binary search for the first potion meeting that threshold
3. Count all potions from that index to the end

This reduces the complexity dramatically.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × m) | O(1) | Checks every spell-potion pair directly |
| Optimal | O(m log m + n log m) | O(1) excluding output | Sort potions once, then binary search for each spell |

## Algorithm Walkthrough

1. Sort the `potions` array.

Sorting allows us to use binary search. After sorting, potion strengths increase monotonically.
2. Create a result array.

This array will store the number of successful pairs for each spell.
3. Iterate through every spell.

For each spell, we want to determine the smallest potion value that satisfies:

$spell \times potion \ge success$
4. Rearrange the inequality.

We can rewrite it as:

$potion \ge \frac{success}{spell}$

This tells us the minimum potion strength needed.
5. Use binary search on the sorted `potions` array.

We search for the first index where the potion value is large enough.

Because the array is sorted, all potions after that index also satisfy the condition.
6. Compute the number of successful potions.

If the first successful potion is at index `idx`, then:

```
successful count = len(potions) - idx
```
7. Store the result and continue.

Repeat the process for every spell.

### Why it works

The correctness comes from the sorted order of the `potions` array.

If a potion at index `i` satisfies the success condition, then every potion after index `i` also satisfies it because potion strengths only increase.

Therefore, the successful potions always form a suffix of the sorted array. Binary search correctly identifies the first valid potion, and counting the remaining elements gives the exact number of successful pairs.

## Python Solution

```python
from bisect import bisect_left
from typing import List

class Solution:
    def successfulPairs(
        self,
        spells: List[int],
        potions: List[int],
        success: int
    ) -> List[int]:

        potions.sort()

        total_potions = len(potions)
        result = []

        for spell in spells:
            minimum_potion = (success + spell - 1) // spell

            first_valid_index = bisect_left(
                potions,
                minimum_potion
            )

            successful_count = total_potions - first_valid_index

            result.append(successful_count)

        return result
```

The implementation begins by sorting the `potions` array. This is the foundation of the binary search optimization.

For each spell, we compute the minimum potion value required. The expression:

```python
(success + spell - 1) // spell
```

performs ceiling division. We need ceiling division because the potion strength must satisfy:

```python
potion >= success / spell
```

If the division is not exact, we must round upward.

We then use `bisect_left` to locate the first potion whose value is at least `minimum_potion`.

Everything from that index onward forms a successful pair, so the count is:

```python
len(potions) - first_valid_index
```

The result for each spell is appended to the output array.

## Go Solution

```go
package main

import "sort"

func successfulPairs(spells []int, potions []int, success int64) []int {
	sort.Ints(potions)

	m := len(potions)
	result := make([]int, len(spells))

	for i, spell := range spells {
		required := (success + int64(spell) - 1) / int64(spell)

		index := sort.Search(m, func(j int) bool {
			return int64(potions[j]) >= required
		})

		result[i] = m - index
	}

	return result
}
```

The Go solution follows the same algorithmic structure as the Python version.

The main implementation difference is integer handling. Since `success` can be as large as `10^10`, multiplication and division must use `int64` to avoid overflow.

Go does not provide a built in `bisect_left` function, so we use `sort.Search`, which performs binary search over indices.

The result slice is preallocated with the correct size for efficiency.

## Edge Cases

### No Successful Pairs

A spell may be too weak to pair successfully with any potion. For example:

```
spell = 1
potions = [1,2,3]
success = 10
```

A common bug is mishandling the binary search result when no valid potion exists. In this implementation, `bisect_left` returns the length of the array, producing:

```
len(potions) - len(potions) = 0
```

which correctly reports zero successful pairs.

### All Potions Succeed

Another important case occurs when every potion works with a spell.

For example:

```
spell = 10
potions = [1,2,3]
success = 5
```

The first valid potion index becomes `0`, so:

```
len(potions) - 0
```

correctly counts the entire array.

### Large Integer Multiplication

The product of spell and potion strengths can be extremely large:

$10^5 \times 10^5 = 10^{10}$

In Go, using regular `int` arithmetic could overflow on some systems. The implementation explicitly converts values to `int64` before multiplication and division to guarantee correctness.

### Duplicate Potion Values

Potion strengths may repeat many times. A flawed implementation might accidentally count only unique values.

Binary search avoids this problem because it searches for the first valid index, not distinct values. Every duplicate potion after that index is naturally included in the count.
