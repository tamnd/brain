## Solution

## Problem Understanding

The problem asks us to count the number of distinct ordered pairs `(num1, num2)` such that both values appear in the input array and the following condition holds:

$$\text{popcount}(num1 \; OR \; num2) + \text{popcount}(num1 \; AND \; num2) \ge k$$

Here, `popcount(x)` means the number of set bits, also called `1` bits, in the binary representation of `x`.

A very important detail is that pairs are ordered. This means `(a, b)` and `(b, a)` are considered different unless `a == b`.

Another important detail is that the problem asks for distinct pairs based on values, not indices. If the array contains duplicates, they should not create extra pairs. For example, if `nums = [1,1,2]`, the value `1` is still treated as a single unique number when forming pairs.

The input size is large:

- `nums.length` can be as large as `10^5`
- Each number can be up to `10^9`
- `k` can be up to `60`

These constraints immediately rule out any solution that explicitly checks every possible pair of numbers. A naive `O(n^2)` solution would be far too slow for `10^5` elements.

The key challenge is recognizing a hidden bit manipulation identity that dramatically simplifies the condition.

There are several important edge cases to keep in mind:

- The array may contain many duplicates.
- `k` may be larger than any possible bit count sum, making the answer zero.
- Pairs where `num1 == num2` are allowed.
- Since pairs are ordered, counting logic must preserve directionality.
- Numbers may have very different bit counts, so grouping by popcount becomes useful.

## Approaches

### Brute Force Approach

The most direct solution is to examine every ordered pair `(a, b)` formed from the distinct values in the array.

For each pair, we compute:

- `a | b`
- `a & b`
- The number of set bits in both values

If their sum is at least `k`, we increment the answer.

This approach is correct because it directly follows the problem definition. However, it is computationally infeasible for large inputs.

If there are `m` distinct numbers, we must check `m^2` ordered pairs. In the worst case, `m` can be `10^5`, resulting in `10^{10}` comparisons, which is far beyond acceptable limits.

### Key Insight

The crucial observation is the following identity:

$$\text{popcount}(a \; OR \; b) + \text{popcount}(a \; AND \; b)
=
\text{popcount}(a) + \text{popcount}(b)$$

This identity works because for every bit position:

- If both bits are `0`, contribution is `0`
- If one bit is `1`, contribution is `1`
- If both bits are `1`, contribution is `2`

Thus, the total contribution per bit equals the sum of set bits from both numbers individually.

This completely removes the need to compute OR and AND operations for every pair.

Now the condition becomes:

$$\text{popcount}(a) + \text{popcount}(b) \ge k$$

This is much easier to handle.

Since integers up to `10^9` have at most 30 set bits, and the problem allows up to 60 because two numbers are combined, there are only a small number of possible bit counts.

We can:

1. Remove duplicates from `nums`
2. Compute the popcount of each unique number
3. Sort the popcounts
4. Use binary search to efficiently count valid partners for each value

This reduces the complexity dramatically.

## Algorithm Walkthrough

### Step 1: Remove Duplicate Values

Convert `nums` into a set.

This is necessary because the problem counts distinct value pairs, not index pairs. Duplicate occurrences should not create additional combinations.

For example:

```
nums = [1,2,3,1]
```

becomes:

```
{1,2,3}
```

### Step 2: Compute Set Bit Counts

For every unique number, compute its number of set bits.

For example:

| Number | Binary | Set Bits |
| --- | --- | --- |
| 1 | 001 | 1 |
| 2 | 010 | 1 |
| 3 | 011 | 2 |

Store all bit counts in a list.

### Step 3: Sort the Bit Counts

Sorting allows us to use binary search efficiently.

Suppose the sorted bit counts are:

```
[1,1,2]
```

### Step 4: Iterate Through Each Bit Count

For each bit count `x`, we need to count how many values `y` satisfy:

$$x + y \ge k$$

Rearranging:

$$y \ge k - x$$

Since the array is sorted, we can binary search for the first valid position.

### Step 5: Use Binary Search

If the first valid index is `idx`, then all elements from `idx` onward form valid ordered pairs with the current value.

The count contributed is:

```
len(bit_counts) - idx
```

We repeat this for every bit count.

### Step 6: Sum All Contributions

Because pairs are ordered, every iteration naturally counts ordered combinations correctly.

For example:

- When processing `a`, we count `(a,b)`
- When processing `b`, we count `(b,a)`

No extra handling is required.

### Why it works

The correctness relies on the identity:

$$\text{popcount}(a \; OR \; b) + \text{popcount}(a \; AND \; b)
=
\text{popcount}(a) + \text{popcount}(b)$$

Once this transformation is made, the problem reduces to counting ordered pairs whose bit counts sum to at least `k`.

Sorting enables efficient binary search for valid partners, and removing duplicates ensures only distinct values are considered.

Therefore, every excellent pair is counted exactly once in the correct order.

## Python Solution

```python
from bisect import bisect_left
from typing import List

class Solution:
    def countExcellentPairs(self, nums: List[int], k: int) -> int:
        unique_nums = set(nums)

        bit_counts = []

        for num in unique_nums:
            bit_counts.append(num.bit_count())

        bit_counts.sort()

        n = len(bit_counts)
        excellent_pairs = 0

        for count in bit_counts:
            required = k - count

            index = bisect_left(bit_counts, required)

            excellent_pairs += n - index

        return excellent_pairs
```

The implementation starts by removing duplicate numbers using a set. This guarantees that each value contributes only once when forming pairs.

Next, the code computes the set bit count for every unique number using Python's built in `bit_count()` method. These counts are stored in a list and sorted.

For every bit count `count`, the algorithm determines the minimum partner bit count needed to satisfy the condition:

```python
required = k - count
```

Using `bisect_left`, the code finds the first position where the required value could appear in the sorted list.

Every value after that index forms a valid ordered pair with the current number, so the contribution is:

```python
n - index
```

The total across all iterations is returned as the final answer.

## Go Solution

```go
package main

import (
	"math/bits"
	"sort"
)

func countExcellentPairs(nums []int, k int) int64 {
	unique := make(map[int]bool)

	for _, num := range nums {
		unique[num] = true
	}

	bitCounts := make([]int, 0, len(unique))

	for num := range unique {
		bitCounts = append(bitCounts, bits.OnesCount(uint(num)))
	}

	sort.Ints(bitCounts)

	n := len(bitCounts)
	var excellentPairs int64 = 0

	for _, count := range bitCounts {
		required := k - count

		index := sort.SearchInts(bitCounts, required)

		excellentPairs += int64(n - index)
	}

	return excellentPairs
}
```

The Go implementation follows the same logic as the Python version.

A `map[int]bool` is used to remove duplicates. The `math/bits` package provides `bits.OnesCount()` for efficiently computing set bits.

Go requires explicit integer conversions, so the final answer is stored as `int64` to avoid overflow when counting large numbers of pairs.

The binary search is implemented using `sort.SearchInts`.

The space usage comes primarily from storing the unique numbers and their corresponding bit counts.

## Edge Cases

One important edge case occurs when the array contains many duplicates. Since the problem counts distinct value pairs rather than index pairs, duplicates must not increase the answer. A naive implementation that iterates directly over the original array would overcount. The solution avoids this bug by converting the input into a set before any processing.

Another important case is when `k` is larger than any achievable bit count sum. Since numbers are at most 30 bits wide, the maximum possible sum is 60. If `k` exceeds the achievable sums for all pairs, the correct answer is zero. The binary search naturally handles this because the required partner count becomes larger than all available values, producing no matches.

Self pairs are also easy to mishandle. The problem explicitly allows `(x, x)` as long as the value exists in the array. Some pair counting strategies accidentally exclude identical pairs. This implementation handles them correctly because each bit count searches across the entire sorted list, including itself.

Another subtle edge case involves ordered versus unordered pairs. The problem considers `(a, b)` and `(b, a)` different. Some optimizations incorrectly count only unordered combinations. This implementation processes every element independently, so both directions are counted automatically whenever both satisfy the condition.
