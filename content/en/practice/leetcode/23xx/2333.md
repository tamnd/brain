## Solution

## Problem Understanding

The problem gives us two arrays, `nums1` and `nums2`, each of length `n`. For every index `i`, we care about the difference between the two values, specifically:

$$(nums1[i] - nums2[i])^2$$

The final score is the sum of these squared differences across all indices.

We are also given two operation budgets, `k1` and `k2`. Each operation allows us to increase or decrease any element in the corresponding array by exactly `1`. Since changing either array affects only the difference between the two values, the important observation is that an operation can reduce an absolute difference by at most `1`.

Instead of thinking about modifying the arrays themselves, we can think about modifying the array of absolute differences:

$$diff[i] = |nums1[i] - nums2[i]|$$

Every operation can reduce one positive difference by `1`. The total number of reductions available is:

$$k = k1 + k2$$

Our goal is to distribute these reductions in a way that minimizes the sum of squares:

$$\sum diff[i]^2$$

The constraints are large:

- `n` can be up to `10^5`
- Values can also reach `10^5`
- `k1` and `k2` can be as large as `10^9`

These constraints immediately rule out any simulation that performs operations one at a time.

An important mathematical property is that reducing a large number decreases the squared sum more than reducing a small number. For example:

- Reducing `10` to `9` changes the square from `100` to `81`, a decrease of `19`
- Reducing `2` to `1` changes the square from `4` to `1`, a decrease of `3`

This means the optimal strategy always prioritizes reducing the largest differences first.

Several edge cases are important:

- If the total operations exceed the sum of all differences, the answer becomes `0`
- Some differences may already be `0`
- `k1` or `k2` may individually be `0`, but only the total matters
- Large values require careful handling of integer overflow in Go

## Approaches

### Brute Force Approach

A straightforward approach is to repeatedly perform one operation at a time. At every step, we locate the largest current difference and reduce it by `1`.

This greedy strategy is correct because reducing the largest difference always gives the greatest decrease in the squared sum.

However, the implementation is far too slow if done naively. The total operations can reach `2 * 10^9`, which makes direct simulation impossible.

Even with a max heap, each operation would require:

- Extract max
- Decrease by `1`
- Push back

That leads to:

$$O(k \log n)$$

Since `k` may be billions, this approach cannot pass.

### Optimal Approach

The key insight is that we do not need to simulate each operation individually.

Instead, we can think about lowering all large differences together. The optimal final state resembles a "leveled" distribution where no value is unnecessarily larger than others.

We use binary search to determine the smallest threshold `T` such that all differences can be reduced to at most `T` using at most `k` operations.

For every difference `d`:

- If `d <= T`, leave it unchanged
- If `d > T`, reduce it down to `T`

The total operations needed are:

$$\sum \max(0, d - T)$$

We binary search the minimum feasible threshold.

After reducing all values above `T`, we may still have some leftover operations. These remaining operations are applied one by one to values equal to `T`, reducing some of them to `T-1`.

Finally, we compute the squared sum.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(k log n) | O(n) | Simulates each operation individually using a heap |
| Optimal | O(n log M) | O(n) | Uses binary search on the maximum difference value |

Here, `M` is the maximum initial difference.

## Algorithm Walkthrough

1. Compute the absolute difference array.

For every index:

```
diff[i] = abs(nums1[i] - nums2[i])
```

Since only the differences matter, we can ignore the original arrays afterward.
2. Compute the total number of operations.

```
k = k1 + k2
```
3. Check if all differences can be eliminated.

If:

```
sum(diff) <= k
```

then every difference can be reduced to zero, so the answer is immediately `0`.
4. Binary search the optimal threshold.

We search for the smallest value `T` such that reducing every value above `T` down to `T` requires at most `k` operations.

The search range is:

```
left = 0
right = max(diff)
```
5. For a candidate threshold `mid`, calculate required operations.

For every difference `d`:

```
operations += max(0, d - mid)
```

If the total required operations exceed `k`, then `mid` is too small.
6. After binary search, reduce all differences to at most `T`.

Create the adjusted array:

- If `d > T`, set it to `T`
- Otherwise keep it unchanged

Track how many operations were used.
7. Use any remaining operations.

Suppose we still have `remaining` operations left.

Since all remaining reducible values equal `T`, we reduce exactly `remaining` of them by `1`.
8. Compute the final squared sum.

Sum:

$$d^2$$

for every adjusted difference.

### Why it works

The squared function is convex, meaning reducing larger values always provides a larger benefit than reducing smaller values. Therefore, an optimal solution always prioritizes decreasing the largest differences first.

The binary search finds the balancing level where all excessively large differences are reduced uniformly. Any leftover operations are then optimally distributed among the remaining largest values. This guarantees the minimum possible sum of squared differences.

## Python Solution

```python
from typing import List

class Solution:
    def minSumSquareDiff(
        self,
        nums1: List[int],
        nums2: List[int],
        k1: int,
        k2: int
    ) -> int:

        diffs = [abs(a - b) for a, b in zip(nums1, nums2)]
        total_ops = k1 + k2

        total_diff = sum(diffs)

        if total_diff <= total_ops:
            return 0

        left = 0
        right = max(diffs)

        while left < right:
            mid = (left + right) // 2

            required = 0

            for d in diffs:
                if d > mid:
                    required += d - mid

            if required > total_ops:
                left = mid + 1
            else:
                right = mid

        threshold = left

        adjusted = []
        used_ops = 0

        for d in diffs:
            if d > threshold:
                used_ops += d - threshold
                adjusted.append(threshold)
            else:
                adjusted.append(d)

        remaining = total_ops - used_ops

        for i in range(len(adjusted)):
            if remaining == 0:
                break

            if adjusted[i] == threshold and adjusted[i] > 0:
                adjusted[i] -= 1
                remaining -= 1

        result = 0

        for d in adjusted:
            result += d * d

        return result
```

The implementation begins by converting the problem into an array of absolute differences. This simplifies the logic because modifying either original array affects only the magnitude of the difference.

The early return handles the important case where the total operations are enough to eliminate every difference entirely.

The binary search determines the optimal cap value for the differences. For each candidate threshold, we calculate how many operations would be required to bring all larger values down to that level.

Once the threshold is found, we construct the adjusted difference array. Any value above the threshold is reduced down to it.

There may still be leftover operations because the binary search finds the smallest feasible threshold. Those remaining operations are applied one at a time to values equal to the threshold.

Finally, the algorithm computes the sum of squared differences.

## Go Solution

```go
package main

func minSumSquareDiff(nums1 []int, nums2 []int, k1 int, k2 int) int64 {
	n := len(nums1)

	diffs := make([]int, n)

	totalDiff := int64(0)

	for i := 0; i < n; i++ {
		diff := nums1[i] - nums2[i]

		if diff < 0 {
			diff = -diff
		}

		diffs[i] = diff
		totalDiff += int64(diff)
	}

	totalOps := int64(k1 + k2)

	if totalDiff <= totalOps {
		return 0
	}

	maxDiff := 0

	for _, d := range diffs {
		if d > maxDiff {
			maxDiff = d
		}
	}

	left := 0
	right := maxDiff

	for left < right {
		mid := (left + right) / 2

		required := int64(0)

		for _, d := range diffs {
			if d > mid {
				required += int64(d - mid)
			}
		}

		if required > totalOps {
			left = mid + 1
		} else {
			right = mid
		}
	}

	threshold := left

	adjusted := make([]int, n)

	usedOps := int64(0)

	for i, d := range diffs {
		if d > threshold {
			usedOps += int64(d - threshold)
			adjusted[i] = threshold
		} else {
			adjusted[i] = d
		}
	}

	remaining := totalOps - usedOps

	for i := 0; i < n && remaining > 0; i++ {
		if adjusted[i] == threshold && adjusted[i] > 0 {
			adjusted[i]--
			remaining--
		}
	}

	var result int64 = 0

	for _, d := range adjusted {
		result += int64(d) * int64(d)
	}

	return result
}
```

The Go implementation follows the same algorithmic structure as the Python version. The most important difference is integer handling.

Since squared values can become very large, the final answer must use `int64`. Intermediate operation counts also use `int64` to avoid overflow when summing large values.

Slices are used for the difference and adjusted arrays. No special nil handling is needed because the constraints guarantee valid input arrays.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log M) | Binary search over the difference range |
| Space | O(n) | Stores the difference array |

Here, `M` is the maximum absolute difference between corresponding elements.

The binary search performs `log M` iterations. During each iteration, we scan all `n` differences to compute how many operations are required. Since `M <= 10^5`, the logarithmic factor is very small in practice.

The extra space comes from storing the difference and adjusted arrays.

## Edge Cases

One important edge case occurs when the total available operations exceed the total sum of differences. A naive implementation might continue attempting reductions and accidentally create negative differences or perform unnecessary work. The implementation handles this immediately with an early return of `0`.

Another critical case involves many differences already being zero. During the leftover operation phase, we must avoid reducing values below zero. The implementation explicitly checks that a value is greater than zero before decrementing it.

A third important case is extremely large operation counts and large differences. Since both can reach very large values, integer overflow becomes a concern, especially in Go. The implementation uses `int64` for all accumulated sums and squared computations to guarantee correctness.
