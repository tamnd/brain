# LeetCode 2317 - Maximum XOR After Operations

## Problem Understanding

The problem gives us an array `nums`, and we may perform a special operation any number of times on any element.

For an index `i` and any non-negative integer `x`, we replace:

$$nums[i] = nums[i] \ \text{AND} \ (nums[i] \ \text{XOR} \ x)$$

Our goal is to maximize the XOR of all elements in the array after performing as many operations as we want.

The important part of the problem is understanding what this operation actually allows us to do to the bits of a number.

Suppose a bit in `nums[i]` is:

- `0`
- `1`

We want to know whether the operation can change that bit.

Consider a single bit independently:

- If the original bit is `0`, then:

- `0 AND anything = 0`
- So a `0` bit can never become `1`
- If the original bit is `1`, then:

- We can choose `x` so that `(1 XOR x_bit)` becomes either `0` or `1`
- Therefore:

- `1 AND 1 = 1`
- `1 AND 0 = 0`
- So a `1` bit may stay `1` or become `0`

This means the operation can only turn bits off. It can never create a new `1` bit.

The XOR of the entire array is maximized when every bit position that exists anywhere in the array appears in the final XOR result. Since we may selectively keep or remove bits from elements, every bit that appears at least once can be arranged to contribute to the final XOR.

That leads to the key observation:

The maximum possible XOR is simply the bitwise OR of all numbers.

The constraints are:

- `1 <= nums.length <= 10^5`
- `0 <= nums[i] <= 10^8`

This tells us that:

- We need an algorithm close to linear time
- Exponential or simulation-based solutions are impossible
- Bit manipulation is likely intended

Important edge cases include:

- Arrays containing only zeros
- Arrays with a single element
- Repeated numbers that normally cancel under XOR
- Large values with many bits set

All of these are handled naturally by the OR-based solution.

## Approaches

### Brute Force Approach

A brute-force solution would attempt to simulate operations on array elements and search through possible values of `x`.

The difficulty is that:

- `x` may be any non-negative integer
- We may perform operations infinitely many times
- Each operation changes the state space

Even for a single number, there are many possible resulting values because each set bit may independently remain `1` or become `0`.

A brute-force search would therefore involve exploring exponentially many states. With up to `10^5` elements, this is completely infeasible.

The brute-force idea is useful only for discovering the bit behavior of the operation.

### Optimal Approach

The key insight is understanding what the operation allows.

The operation can:

- Turn any existing `1` bit into `0`
- Never turn a `0` bit into `1`

Therefore, every final number is a subset of the original number's bits.

Now consider a particular bit position:

- If no number originally contains that bit, it can never appear in the final XOR
- If at least one number contains that bit, we can preserve it in exactly one element so that it contributes to the final XOR

Thus, every bit that exists anywhere in the array can appear in the final answer.

The set of all bits appearing anywhere in the array is exactly the bitwise OR of all elements.

So the answer is:

$$nums[0] \ OR \ nums[1] \ OR \dots OR \ nums[n-1]$$

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Explores many possible operation sequences |
| Optimal | O(n) | O(1) | Compute bitwise OR of all elements |

## Algorithm Walkthrough

1. Initialize a variable `answer` to `0`.

This variable will accumulate all bits that appear in the array.
2. Iterate through every number in `nums`.

For each number, compute:

$$answer = answer \ OR \ num$$

This keeps every bit that appears in any element.
3. After processing all numbers, return `answer`.

This value represents the maximum possible XOR after any sequence of operations.

### Why it works

The operation can only remove bits, never create new ones. Therefore, any bit absent from all original numbers can never appear in the final XOR.

Conversely, any bit that exists somewhere in the array may be preserved in an odd number of elements, allowing it to appear in the final XOR result.

Thus, the maximum achievable XOR contains exactly all bits present anywhere in the input array, which is the bitwise OR of all elements.

## Python Solution

```python
from typing import List

class Solution:
    def maximumXOR(self, nums: List[int]) -> int:
        answer = 0

        for num in nums:
            answer |= num

        return answer
```

The implementation is extremely compact because the mathematical observation fully simplifies the problem.

The variable `answer` starts at `0`, which is the identity value for the OR operation.

We iterate through every element in `nums` and OR it into `answer`. Any bit that appears in at least one number becomes permanently set in `answer`.

At the end, `answer` contains all bits that can possibly appear in the final XOR, which is exactly the optimal result.

## Go Solution

```go
func maximumXOR(nums []int) int {
    answer := 0

    for _, num := range nums {
        answer |= num
    }

    return answer
}
```

The Go implementation follows the exact same logic as the Python version.

A `for range` loop iterates through the slice, and the bitwise OR operator `|=` accumulates all bits into `answer`.

There are no overflow concerns because the constraints fit comfortably inside Go's `int` type.

## Edge Cases

A very important edge case is an array containing only zeros. Since the operation can never create new `1` bits, the final XOR must remain zero. The implementation handles this naturally because OR-ing zeros together always produces zero.

Another important case is a single-element array. Since XOR of one number is simply the number itself, the maximum achievable value is just that element. The OR accumulation logic works correctly because the OR of one value is the value itself.

Arrays with repeated values can also be tricky conceptually. Normally, identical numbers may cancel out under XOR. For example:

```
7 XOR 7 = 0
```

However, the operation allows us to selectively remove bits from some elements, preventing cancellation. The OR-based solution correctly captures the fact that every existing bit can still appear in the final answer.

Large values near the constraint limit are another important consideration. Since the algorithm uses only standard bitwise OR operations and does not depend on recursion or large auxiliary structures, it handles maximum-sized integers efficiently and safely.
