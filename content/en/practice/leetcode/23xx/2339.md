## Solution

## Problem Understanding

The `Teams` table contains the names of all teams participating in a league. Each row represents exactly one team, and the `team_name` column is guaranteed to be unique.

The problem asks us to generate every possible match between teams under the following rule:

- Every pair of distinct teams plays twice.
- In one match, Team A is the home team and Team B is the away team.
- In the second match, Team B becomes the home team and Team A becomes the away team.

In other words, for every ordered pair of different teams `(A, B)`, we must output one row:

```
home_team = A
away_team = B
```

The only restriction is that a team cannot play against itself. Therefore, pairs like `(Real Madrid, Real Madrid)` must not appear.

The output table should contain two columns:

- `home_team`
- `away_team`

The order of the rows does not matter.

For the example input:

```
Leetcode FC
Ahly SC
Real Madrid
```

we must generate all directed match combinations:

```
Real Madrid -> Leetcode FC
Real Madrid -> Ahly SC
Leetcode FC -> Real Madrid
Leetcode FC -> Ahly SC
Ahly SC -> Real Madrid
Ahly SC -> Leetcode FC
```

This is effectively asking for all ordered pairs of distinct teams.

The constraints are small because this is an Easy database problem, and the table contains only one column. The challenge is not computational complexity, but understanding how to generate pairwise combinations correctly using SQL joins.

An important observation is that every team must be paired with every other team exactly once in each direction. A common mistake is to generate only one direction of the matchup, or to accidentally include self matches.

Important edge cases include:

- A league with only one team, which should produce no rows because a team cannot play itself.
- Ensuring that `(A, B)` and `(B, A)` are both included.
- Avoiding duplicate rows beyond those two intended directional matches.
- Preventing self matches such as `(A, A)`.

## Approaches

### Brute Force Approach

The brute force idea is to compare every team with every other team manually. Conceptually, we iterate through the table twice using nested loops.

For each team chosen as the home team, we examine every team again as a potential away team. If the two teams are different, we output the pair.

This approach works because it explicitly checks every possible ordered pair of teams. Since every pair is examined, no valid match can be missed.

In procedural programming terms, this would look like:

```
for each home_team:
    for each away_team:
        if home_team != away_team:
            output match
```

The drawback is that it performs comparisons between all pairs of teams, leading to quadratic time complexity. However, since the database problem naturally maps to pair generation, this complexity is unavoidable because the output itself contains O(n²) rows.

### Optimal SQL Approach

The key insight is that SQL already provides a natural mechanism for generating all pair combinations: the `CROSS JOIN`.

A `CROSS JOIN` combines every row from the first table with every row from the second table. Since we need every possible ordered pair of teams, this operation is exactly what we want.

After generating all combinations, we simply remove the invalid self matches where the two team names are equal.

This produces all valid home and away matchups directly and cleanly.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1), excluding output | Nested iteration through every pair of teams |
| Optimal | O(n²) | O(1), excluding output | Uses SQL CROSS JOIN to naturally generate all ordered pairs |

## Algorithm Walkthrough

1. Start with the `Teams` table as the first instance of the table, representing the `home_team`.
2. Create a second instance of the same `Teams` table, representing the `away_team`.
3. Perform a `CROSS JOIN` between the two table instances. This generates every possible pair of rows.

If there are `n` teams, the cross join creates `n × n` combinations.
4. Filter out rows where the home team and away team are the same.

This removes invalid matches such as:

```
Real Madrid vs Real Madrid
```
5. Select the remaining rows using aliases:

```
t1.team_name AS home_team
t2.team_name AS away_team
```
6. Return the resulting table.

### Why it works

The `CROSS JOIN` guarantees that every team is paired with every other team. Since the join is directional, `(A, B)` and `(B, A)` are treated as distinct rows, which matches the league requirement that teams play both home and away matches. Filtering out equal team names ensures that no team is matched against itself.

## Python Solution

Although this is a database problem, LeetCode database solutions are typically written directly in SQL. The following is the correct SQL query.

```python
SELECT
    t1.team_name AS home_team,
    t2.team_name AS away_team
FROM Teams t1
CROSS JOIN Teams t2
WHERE t1.team_name != t2.team_name;
```

The query uses two aliases of the `Teams` table:

- `t1` represents the home team
- `t2` represents the away team

The `CROSS JOIN` creates every possible ordered pair between the two tables. Since every row from `t1` is matched with every row from `t2`, all combinations are generated automatically.

The `WHERE` clause removes invalid self matches where both team names are identical.

Finally, the `SELECT` statement renames the columns to match the required output format.

## Edge Cases

### Only One Team

If the league contains only one team, the cross join initially produces one row where the team is paired with itself. The `WHERE` clause removes this row, resulting in an empty output table. This is correct because a team cannot play against itself.

### Exactly Two Teams

With two teams, many implementations accidentally produce only one matchup instead of two. The directional nature of the cross join ensures that both:

```
(A, B)
(B, A)
```

are included.

### Self Matches

A very common bug is forgetting to exclude self pairings. Without the filter condition:

```
WHERE t1.team_name != t2.team_name
```

the output would incorrectly include matches such as:

```
Real Madrid vs Real Madrid
```

The implementation explicitly removes these rows, guaranteeing correctness.

### Duplicate Team Names

The problem guarantees that `team_name` values are unique. This matters because duplicate names could otherwise produce ambiguous or repeated match rows. Since uniqueness is guaranteed, each generated pair corresponds to exactly one valid matchup.
