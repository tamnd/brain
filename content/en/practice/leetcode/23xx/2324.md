# LeetCode 2324 - Product Sales Analysis IV

## Problem Understanding

This problem gives us two database tables, `Sales` and `Product`.

The `Sales` table records individual purchase events. Each row tells us which user bought which product and in what quantity. A single user may buy the same product multiple times across different rows.

The `Product` table stores the price for each product.

The goal is to determine, for every user, which product or products they spent the most money on.

The important detail is that spending is cumulative per product. If a user buys the same product multiple times, all purchases of that product must be combined before comparing totals.

The total money spent on a product is:

$$\text{total spent} = \sum (\text{quantity} \times \text{price})$$

For each user:

1. Compute the total amount spent on every product.
2. Find the maximum spending value among those products.
3. Return every product whose spending equals that maximum.

The output should contain:

- `user_id`
- `product_id`

If multiple products tie for the maximum spending for a user, all tied products must appear in the result.

The problem guarantees that:

- `sale_id` is unique.
- Every `product_id` in `Sales` exists in `Product`.
- Product prices are available for all sales.

This is fundamentally a grouping and ranking problem.

Important edge cases include:

- A user purchasing the same product multiple times.
- Multiple products tying for maximum spending.
- Users with only one purchased product.
- Products with different quantities but identical final spending totals.

A naive implementation can easily fail if it compares individual sales rows instead of aggregating spending across all purchases of the same product.

## Approaches

### Brute Force Approach

A brute force solution would work in several phases.

First, join every sales row with the corresponding product price to compute the spending for each row.

Next, for every user and every product, repeatedly scan all sales rows to accumulate the total spending for that combination.

After that, for each user, scan all computed totals to determine the maximum spending value.

Finally, scan the totals again to collect every product that matches the maximum.

This approach is correct because it explicitly computes every required spending total and compares them accurately. However, it performs many repeated scans across the same data, making it inefficient for large datasets.

### Optimal Approach

The key insight is that SQL aggregation functions already solve most of the problem efficiently.

We can:

1. Join `Sales` with `Product`.
2. Group by `(user_id, product_id)` to compute total spending.
3. Use a window function such as `RANK()` or `DENSE_RANK()` partitioned by `user_id`.
4. Order each user's products by total spending descending.
5. Keep only the rows with rank `1`.

This works efficiently because the database engine performs grouping and ranking in optimized passes instead of repeatedly rescanning data.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(N²) | O(N) | Repeatedly rescans sales records for aggregation |
| Optimal | O(N log N) | O(N) | Uses grouping and window ranking efficiently |

Here, `N` represents the number of sales rows.

## Algorithm Walkthrough

1. Join the `Sales` and `Product` tables using `product_id`.

This step gives access to both the purchased quantity and the product price in the same row. Without the join, we cannot compute spending.
2. Compute spending for each sale row.

For every joined row, calculate:

$$\text{quantity} \times \text{price}$$
3. Aggregate spending by user and product.

Group rows using:

- `user_id`
- `product_id`

Then sum all spending values inside each group.

This correctly combines multiple purchases of the same product by the same user.
4. Rank products within each user.

Use a window function:

```
RANK() OVER (
    PARTITION BY user_id
    ORDER BY total_spent DESC
)
```

Partitioning ensures each user's products are ranked independently.
5. Select only rank `1`.

Products with rank `1` represent the maximum spending products for each user.

Since `RANK()` assigns the same rank to ties, all tied maximum products are included automatically.

### Why it works

The algorithm works because it first computes the exact total spending for every `(user_id, product_id)` pair. Once those totals are known, ranking within each user guarantees that the highest spending value receives rank `1`. Since ties receive the same rank, every maximum-spending product is preserved in the final output.

## Python Solution

```python
# Write your MySQL query statement below

WITH spending AS (
    SELECT
        s.user_id,
        s.product_id,
        SUM(s.quantity * p.price) AS total_spent
    FROM Sales s
    JOIN Product p
        ON s.product_id = p.product_id
    GROUP BY s.user_id, s.product_id
),
ranked AS (
    SELECT
        user_id,
        product_id,
        total_spent,
        RANK() OVER (
            PARTITION BY user_id
            ORDER BY total_spent DESC
        ) AS ranking
    FROM spending
)
SELECT
    user_id,
    product_id
FROM ranked
WHERE ranking = 1;
```

The solution starts by creating a common table expression named `spending`. This CTE joins `Sales` and `Product`, then groups by both `user_id` and `product_id`. The `SUM(quantity * price)` expression calculates the total money each user spent on each product.

The second CTE, `ranked`, applies the `RANK()` window function. The ranking is reset for each user because of `PARTITION BY user_id`. Products are ordered by `total_spent` descending so that the largest totals receive rank `1`.

Finally, the outer query filters to rows where `ranking = 1`. This returns all products tied for maximum spending.

The join operation processes all sales rows once. Grouping aggregates rows by `(user_id, product_id)`, and the window ranking sorts products within each user partition. Database engines typically implement these operations using sorting or hashing, leading to approximately `O(N log N)` runtime in practice.

## Edge Cases

One important edge case occurs when a user purchases the same product multiple times. A buggy implementation might compare individual sale rows instead of aggregating totals. In the example, user `102` buys product `1` twice. The solution handles this correctly because the `GROUP BY` clause combines all rows before ranking.

Another critical edge case involves ties. Multiple products may have identical maximum spending values for the same user. Using `ROW_NUMBER()` would incorrectly discard tied rows because it forces unique rankings. The implementation uses `RANK()` instead, which assigns the same rank to equal totals and preserves all tied products.

A third edge case is users with only one purchased product. In that situation, the grouped result contains only one row for the user, and the ranking operation naturally assigns rank `1`. The query therefore returns that product correctly without requiring any special handling.
