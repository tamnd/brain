# LeetCode 2384 - Largest Palindromic Number

## Problem Understanding

The problem gives us a string `num` that contains only digits from `0` to `9`. We may rearrange these digits in any order, and we are allowed to use only some of the digits if we want. Our task is to build the numerically largest palindrome possible.

A palindrome is a string that reads the same forward and backward. For example, `"7449447"` is a palindrome because reversing it produces the same string.

The result must satisfy two important conditions:

1. It must be a valid palindrome.
2. It must not contain leading zeroes, unless the entire palindrome itself is `"0"`.

The input size can be as large as `10^5`, which immediately tells us that expensive combinatorial approaches are impossible. We need a solution that runs in roughly linear time.

The key observations about palindromes are:

- Digits on the left side must match digits on the right side.
- Every digit used in pairs contributes symmetrically to the palindrome.
- At most one digit may appear an odd number of times, and that digit can be placed in the center.

Several edge cases are important:

- Inputs containing mostly or entirely zeroes.
- Inputs where only one non-zero digit exists.
- Inputs where many digits have odd frequencies.
- Cases where using all digits would force leading zeroes.
- Cases where the best answer is a single digit.

For example:

- `"0000"` should return `"0"`, not `"0000"`.
- `"00009"` should return `"9"`, because `"00000"` is invalid due to leading zeroes and numerically smaller anyway.
- `"54321"` should return the largest single digit, `"5"`.

## Approaches

### Brute Force Approach

A brute-force solution would attempt every possible subset of digits, generate every permutation of those digits, and check whether the resulting number is a palindrome. Among all valid palindromes, we would keep the numerically largest one.

This works because it exhaustively explores every possible arrangement. Eventually, it would discover the optimal palindrome.

However, this approach is completely infeasible. Even for a string of length 20, the number of permutations becomes astronomically large. The actual constraint is `10^5`, making brute force impossible.

### Key Insight

A palindrome has a highly structured form:

- The left half completely determines the right half.
- Digits should be placed greedily from largest to smallest.
- Larger digits should appear earlier in the palindrome because earlier positions have greater numerical significance.

The optimal strategy is therefore:

1. Count how many times each digit appears.
2. Use as many pairs of large digits as possible to construct the left half.
3. Mirror the left half to form the right half.
4. Choose the largest remaining digit as the center.

There is one subtle issue: leading zeroes.

If we blindly place zero pairs at the front of the palindrome, we may create invalid numbers like `"0000"` or `"0110"`. Therefore, we should avoid adding zero pairs unless the palindrome already contains a non-zero digit.

Because there are only 10 possible digits, counting and constructing the palindrome is extremely efficient.

## Algorithm Walkthrough

1. Count the frequency of each digit from `0` to `9`.

Since the input contains only digits, an array of size 10 is sufficient. This allows constant-time access for every digit frequency.
2. Build the left half of the palindrome greedily.

Iterate from digit `9` down to `0`.

For each digit:

- Compute how many pairs are available using `count[digit] // 2`.
- Append that many copies of the digit to the left half.
- Reduce the frequency accordingly.

We process digits from largest to smallest because earlier positions contribute more to the numerical value.
3. Prevent leading zeroes.

If the left half is currently empty, we should not add zero pairs. Otherwise, we might create invalid numbers like `"0000"`.

For example:

- Input `"00009"` should produce `"9"`.
- Input `"0000"` should produce `"0"`.
4. Choose the center digit.

After removing pairs, some digits may still have one remaining occurrence.

The largest remaining digit should be placed in the center because the center contributes to the value of the palindrome.
5. Construct the final palindrome.

The structure is:

```
left_half + middle_digit + reverse(left_half)
```
6. Handle the all-zero case.

If no non-zero digit was added and no middle digit exists except zero, return `"0"`.

### Why it works

The algorithm is correct because the most significant positions in the palindrome are the outermost digits. By greedily placing the largest available pairs first, we maximize the number lexicographically and numerically.

The center digit affects the value less than the outer digits, so after maximizing the outer structure, choosing the largest remaining center digit gives the best possible result.

Since every palindrome must be symmetric, using digit pairs is mandatory for the left and right halves. The algorithm uses the maximum possible number of beneficial pairs while respecting the no-leading-zero rule.

## Python Solution

```python
class Solution:
    def largestPalindromic(self, num: str) -> str:
        counts = [0] * 10
        
        for ch in num:
            counts[int(ch)] += 1
        
        left_half = []
        
        # Build the left half greedily from largest digit to smallest
        for digit in range(9, -1, -1):
            pair_count = counts[digit] // 2
            
            # Prevent leading zeroes
            if digit == 0 and not left_half:
                continue
            
            if pair_count > 0:
                left_half.append(str(digit) * pair_count)
                counts[digit] -= pair_count * 2
        
        left = "".join(left_half)
        
        # Choose the largest remaining digit for the center
        middle = ""
        for digit in range(9, -1, -1):
            if counts[digit] > 0:
                middle = str(digit)
                break
        
        result = left + middle + left[::-1]
        
        # Handle cases like "0000"
        return result if result else "0"
```

The implementation starts by counting digit frequencies using a fixed-size array of length 10. This is more efficient than a hash map because the possible keys are known in advance.

The `left_half` list stores chunks of repeated digits. Using a list and joining at the end is more efficient than repeated string concatenation.

The loop processes digits from `9` down to `0`, ensuring that larger digits occupy more significant positions in the palindrome.

The condition:

```python
if digit == 0 and not left_half:
```

prevents the palindrome from starting with zeroes. This is the key detail that handles tricky inputs correctly.

After pair construction, the algorithm scans again from `9` to `0` to find the largest remaining digit for the center.

Finally, the palindrome is assembled by mirroring the left half.

## Go Solution

```go
func largestPalindromic(num string) string {
	counts := make([]int, 10)

	for _, ch := range num {
		counts[ch-'0']++
	}

	leftHalf := make([]byte, 0)

	// Build left half greedily
	for digit := 9; digit >= 0; digit-- {
		pairCount := counts[digit] / 2

		// Prevent leading zeroes
		if digit == 0 && len(leftHalf) == 0 {
			continue
		}

		for i := 0; i < pairCount; i++ {
			leftHalf = append(leftHalf, byte('0'+digit))
		}

		counts[digit] -= pairCount * 2
	}

	left := string(leftHalf)

	// Choose largest remaining middle digit
	middle := ""

	for digit := 9; digit >= 0; digit-- {
		if counts[digit] > 0 {
			middle = string(byte('0' + digit))
			break
		}
	}

	// Reverse left half
	rightHalf := make([]byte, len(leftHalf))

	for i := 0; i < len(leftHalf); i++ {
		rightHalf[i] = leftHalf[len(leftHalf)-1-i]
	}

	result := left + middle + string(rightHalf)

	if result == "" {
		return "0"
	}

	return result
}
```

The Go version follows the same algorithmic structure as the Python implementation.

One notable difference is string handling. Since Go strings are immutable, the implementation uses byte slices for efficient construction of the left half and reversed half.

The reverse operation is implemented manually because Go does not provide built-in string reversal.

Integer overflow is not a concern because all counts are bounded by `10^5`.

## Edge Cases

### All Zeroes

An input like `"0000"` is tricky because a naive algorithm might return `"0000"`. Although technically symmetric, it violates the no-leading-zero rule.

The implementation handles this by refusing to place zero pairs into an empty left half. If no other digits exist, the constructed result remains empty, and the algorithm explicitly returns `"0"`.

### No Available Pairs

Inputs like `"54321"` contain only unique digits. Since no digit appears twice, no left-right symmetry can be formed.

In this situation, the best palindrome is simply the largest single digit. The algorithm naturally handles this because the left half stays empty and the largest remaining digit becomes the center.

### Zero Pairs With Non-Zero Digits

An input like `"10001"` can easily produce incorrect results if zeroes are handled carelessly.

A buggy solution might skip all zero pairs entirely, producing `"101"` instead of `"10001"`.

The implementation only blocks zero pairs when the left half is still empty. Once a non-zero digit exists in the left half, zeroes may safely appear afterward because they are no longer leading digits.
