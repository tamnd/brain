## Solution

## Problem Understanding

The problem gives us three inputs:

- A string `s`, which is the larger text.
- A string `sub`, which we want to transform so it can appear inside `s`.
- A list of character replacement rules called `mappings`.

Each mapping `[old, new]` means that a character `old` in `sub` may be replaced with `new`.

The goal is to determine whether we can transform `sub` into some string that appears as a contiguous substring of `s`.

The important detail is that replacements only apply to characters from `sub`, not characters in `s`. Also, a character may stay unchanged if we want. For example, if `sub[i] == s[j]`, that position already matches without any replacement.

Another subtle point is that replacements are positional, not global. If the mapping says `"e" -> "3"`, then any occurrence of `'e'` in `sub` may independently become `'3'`. We are not required to replace every `'e'`.

The constraints are:

- `1 <= sub.length <= s.length <= 5000`
- `mappings.length <= 1000`

Since both strings can be as large as 5000 characters, a very slow nested simulation approach could become expensive. However, the limits are still small enough for an `O(n * m)` matching algorithm, where:

- `n = len(s)`
- `m = len(sub)`

Important edge cases include:

- No mappings at all.
- Exact substring match without replacements.
- Multiple mappings for the same source character.
- Characters that cannot be replaced.
- Uppercase and lowercase characters being different.
- Digits mixed with letters.
- Replacement rules being directional only, `"a" -> "b"` does not imply `"b" -> "a"`.

## Approaches

### Brute Force Approach

The most direct solution is to generate every possible transformed version of `sub` using the replacement rules, then check whether any generated string is a substring of `s`.

For example, suppose:

```
sub = "ab"
mappings = [["a","x"], ["b","y"]]
```

Possible transformed strings are:

```
"ab"
"xb"
"ay"
"xy"
```

We could generate all possibilities and test each against `s`.

This works because it explicitly enumerates every valid transformation. If any transformed version appears inside `s`, the answer is `true`.

The problem is that this becomes exponentially expensive. If many characters have multiple replacement choices, the number of generated strings explodes combinatorially.

For example, if 20 positions each have 2 choices, that already creates:

$2^{20}$

possible strings.

That is completely impractical for strings of length up to 5000.

### Optimal Approach

The key observation is that we do not need to generate transformed strings at all.

Instead, we can directly compare `sub` against every possible substring window of `s`.

Suppose we align `sub` starting at position `i` in `s`. Then for every position `j`:

- The characters match directly, or
- `sub[j]` may legally transform into `s[i + j]`

If every position satisfies one of those conditions, then the transformation is possible.

To make lookup efficient, we store the mappings in a hash set.

For example:

```
('e', '3')
('t', '7')
```

Then checking whether a replacement is allowed becomes an `O(1)` operation.

This avoids generating transformed strings entirely and reduces the problem to substring checking with character compatibility.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Generates all transformed versions of `sub` |
| Optimal | O((n - m + 1) * m) | O(k) | Directly checks every substring window using hash lookup |

Where:

- `n = len(s)`
- `m = len(sub)`
- `k = number of mappings`

## Algorithm Walkthrough

1. Create a hash set to store all allowed replacements.

Each mapping `[old, new]` is inserted as a tuple:

```
(old, new)
```

This allows constant time replacement checks.
2. Iterate over every possible starting position in `s`.

If `sub` has length `m`, then the last valid starting position is:

$n - m$
3. For each starting position, compare characters one by one.

Let:

- `sub[j]` be the current character from `sub`
- `s[i + j]` be the aligned character in `s`
4. For each pair of characters, check whether:

- They are exactly equal, or
- The pair `(sub[j], s[i + j])` exists in the mapping set
5. If neither condition is true, the current window fails immediately.

We break early because this substring alignment cannot work.
6. If every position matches successfully, return `True`.

This means we found a valid transformation of `sub` that appears in `s`.
7. If all windows fail, return `False`.

### Why it works

The algorithm checks every possible alignment where `sub` could appear inside `s`.

For each aligned position, it verifies the exact condition required by the problem:

- Either the characters already match.
- Or the character from `sub` can legally transform into the target character in `s`.

Because every valid transformation must satisfy these conditions position by position, and because every substring location is examined, the algorithm is guaranteed to find a solution whenever one exists.

## Python Solution

```python
from typing import List, Set, Tuple

class Solution:
    def matchReplacement(
        self,
        s: str,
        sub: str,
        mappings: List[List[str]]
    ) -> bool:

        allowed: Set[Tuple[str, str]] = set()

        for old_char, new_char in mappings:
            allowed.add((old_char, new_char))

        n = len(s)
        m = len(sub)

        for start in range(n - m + 1):

            valid = True

            for j in range(m):

                source_char = sub[j]
                target_char = s[start + j]

                if source_char == target_char:
                    continue

                if (source_char, target_char) not in allowed:
                    valid = False
                    break

            if valid:
                return True

        return False
```

The implementation begins by converting the mappings into a hash set. This is important because membership testing in a set is constant time on average.

Next, the code iterates through every possible substring starting index in `s`. Since `sub` has length `m`, there are exactly `n - m + 1` valid windows to test.

Inside each window, the algorithm compares corresponding characters from `sub` and the aligned substring of `s`.

If characters are equal, no replacement is needed. Otherwise, the code checks whether the replacement pair exists in the set of allowed mappings.

If a mismatch cannot be resolved, the inner loop exits immediately because the current window cannot possibly work.

If an entire window succeeds, the method returns `True` immediately.

If no window matches after checking all possibilities, the method returns `False`.

## Go Solution

```go
func matchReplacement(s string, sub string, mappings [][]byte) bool {
	allowed := make(map[[2]byte]bool)

	for _, mapping := range mappings {
		oldChar := mapping[0]
		newChar := mapping[1]

		allowed[[2]byte{oldChar, newChar}] = true
	}

	n := len(s)
	m := len(sub)

	for start := 0; start <= n-m; start++ {

		valid := true

		for j := 0; j < m; j++ {

			sourceChar := sub[j]
			targetChar := s[start+j]

			if sourceChar == targetChar {
				continue
			}

			if !allowed[[2]byte{sourceChar, targetChar}] {
				valid = false
				break
			}
		}

		if valid {
			return true
		}
	}

	return false
}
```

The Go implementation follows the same logic as the Python version.

Instead of Python tuples, Go uses a fixed-size array `[2]byte` as the hash map key. This works efficiently because arrays in Go are comparable and can be used as map keys directly.

Strings in Go are byte indexed, which is safe here because the problem only uses ASCII letters and digits.

There are no integer overflow concerns because all indices remain within the bounds of the input strings.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O((n - m + 1) * m) | Each substring window is checked character by character |
| Space | O(k) | The mapping set stores all replacement rules |

Where:

- `n = len(s)`
- `m = len(sub)`
- `k = len(mappings)`

The algorithm checks every possible alignment of `sub` inside `s`. For each alignment, it may compare up to `m` characters. Since there are `n - m + 1` windows, the total complexity becomes:

$O((n-m+1)\cdot m)$

The additional memory usage comes only from storing the replacement mappings in a hash set.

## Edge Cases

One important edge case occurs when there are no mappings at all. In this situation, the problem reduces to ordinary substring matching. A buggy implementation might incorrectly assume replacements are always available. This solution handles the case naturally because the mapping set is simply empty, so only exact character matches succeed.

Another critical edge case is mapping directionality. If the mapping contains `"o" -> "0"`, that does not mean `"0" -> "o"` is allowed. Many incorrect implementations accidentally treat mappings as bidirectional. This algorithm explicitly checks ordered pairs `(source, target)`, which preserves the correct direction.

Case sensitivity is another common source of mistakes. The strings may contain uppercase and lowercase letters, and they must be treated as different characters. For example, `'a'` and `'A'` are not interchangeable unless a mapping explicitly allows it. Since the implementation performs direct character comparisons without normalization, it correctly preserves case sensitivity.

A subtle edge case involves repeated characters in `sub`. If `sub` contains multiple copies of the same character, each occurrence may independently choose whether to use a replacement. The algorithm handles this correctly because each character position is checked independently during the window comparison.
