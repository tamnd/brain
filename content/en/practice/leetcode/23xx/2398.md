**Difficulty:** 🔴 Hard  
**Topics:** Array, Binary Search, Queue, Sliding Window, Heap (Priority Queue), Prefix Sum, Monotonic Queue

## Solution

## Problem Understanding

The problem asks us to determine the maximum number of **consecutive robots** we can run without exceeding a given `budget`. Each robot has two costs: `chargeTimes[i]` (a one-time cost per robot) and `runningCosts[i]` (a per-unit cost). For any consecutive group of `k` robots, the total cost is defined as:

```
total_cost = max(chargeTimes in window) + k * sum(runningCosts in window)
```

Our goal is to find the largest `k` such that there exists at least one **consecutive subarray of length k** whose total cost does not exceed `budget`.

The input arrays `chargeTimes` and `runningCosts` have lengths up to 50,000. The constraints indicate that a naive solution that checks every subarray of every possible length would be too slow because the number of subarrays grows quadratically. We need an approach that efficiently calculates the maximum charge time and the sum of running costs for sliding windows of robots.

Important edge cases include:

- No robot fits within the budget, in which case the answer is `0`.
- All robots individually fit but only small consecutive groups fit.
- Large values of `budget` that could allow all robots to run.

## Approaches

### Brute Force

A brute force approach would try every possible subarray of every length. For each subarray, compute `max(chargeTimes)` and `sum(runningCosts)` to calculate the total cost. Keep track of the largest subarray length that is under the budget. This is correct but too slow because for `n = 50,000`, there are approximately `n*(n+1)/2` subarrays, and computing `max` and `sum` for each takes linear time.

### Optimal Approach

The key insight is that we can slide a window across the robots while efficiently maintaining the `max(chargeTimes)` and the `sum(runningCosts)` for that window. We can maintain `sum(runningCosts)` with a simple running total. To maintain `max(chargeTimes)` efficiently, we can use a **monotonic decreasing deque** which keeps the maximum at the front and allows us to remove elements that slide out of the window in O(1) time.

The algorithm works by expanding the window to the right until the total cost exceeds the budget. If it exceeds, we shrink the window from the left until the total cost is under the budget again. We track the maximum window size encountered that fits the budget.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^2) | O(1) | Check every subarray, compute max and sum |
| Sliding Window + Monotonic Queue | O(n) | O(n) | Maintain window sum and max efficiently |

## Algorithm Walkthrough

1. Initialize `left = 0`, `running_sum = 0`, and `max_len = 0`. Create an empty deque to maintain indices of `chargeTimes` in decreasing order.
2. Iterate `right` from `0` to `n-1`:

1. Add `runningCosts[right]` to `running_sum`.
2. Update the deque: remove elements from the back while `chargeTimes[deque[-1]] <= chargeTimes[right]` and append `right`.
3. Compute the current total cost as `chargeTimes[deque[0]] + (right - left + 1) * running_sum`.
4. While total cost exceeds `budget`, shrink the window from the left:

1. Subtract `runningCosts[left]` from `running_sum`.
2. Remove `left` from the front of the deque if it matches `deque[0]`.
3. Increment `left`.
5. Update `max_len` with the current window size `right - left + 1`.
3. Return `max_len`.

**Why it works:** The deque maintains the maximum `chargeTimes` for the current window. The running sum tracks the cumulative `runningCosts`. By expanding the window greedily and shrinking it whenever the budget is exceeded, we ensure that we examine every valid consecutive window efficiently without recomputing sums or maxima from scratch.

## Python Solution

```python
from collections import deque
from typing import List

class Solution:
    def maximumRobots(self, chargeTimes: List[int], runningCosts: List[int], budget: int) -> int:
        n = len(chargeTimes)
        max_len = 0
        running_sum = 0
        left = 0
        dq = deque()  # stores indices of chargeTimes in decreasing order
        
        for right in range(n):
            running_sum += runningCosts[right]
            
            while dq and chargeTimes[dq[-1]] <= chargeTimes[right]:
                dq.pop()
            dq.append(right)
            
            while dq and chargeTimes[dq[0]] + (right - left + 1) * running_sum > budget:
                running_sum -= runningCosts[left]
                if dq[0] == left:
                    dq.popleft()
                left += 1
            
            max_len = max(max_len, right - left + 1)
        
        return max_len
```

**Implementation Notes:** We use a deque to keep track of the maximum `chargeTimes` in O(1) time per step. `running_sum` accumulates `runningCosts`. The inner `while` ensures we shrink the window whenever the budget is exceeded. This ensures linear time complexity since each element is added and removed from the deque at most once.

## Go Solution

```go
package main

func maximumRobots(chargeTimes []int, runningCosts []int, budget int64) int {
    n := len(chargeTimes)
    maxLen := 0
    runningSum := int64(0)
    left := 0
    dq := []int{} // indices of chargeTimes in decreasing order

    for right := 0; right < n; right++ {
        runningSum += int64(runningCosts[right])
        
        for len(dq) > 0 && chargeTimes[dq[len(dq)-1]] <= chargeTimes[right] {
            dq = dq[:len(dq)-1]
        }
        dq = append(dq, right)
        
        for len(dq) > 0 && int64(chargeTimes[dq[0]]) + int64(right-left+1)*runningSum > budget {
            runningSum -= int64(runningCosts[left])
            if dq[0] == left {
                dq = dq[1:]
            }
            left++
        }
        
        if right-left+1 > maxLen {
            maxLen = right - left + 1
        }
    }
    
    return maxLen
}
```

**Go-specific Notes:** We convert all intermediate sums to `int64` to prevent overflow. The deque is implemented as a slice, with elements appended at the back and removed from front efficiently. The logic mirrors the Python version.

## Worked Examples

### Example 1

```
chargeTimes = [3,6,1,3,4]
runningCosts = [2,1,3,4,5]
budget = 25
```

Step-by-step:

| left | right | deque | running_sum | total_cost | max_len |
| --- | --- | --- | --- | --- | --- |
| 0 | 0 | [0] | 2 | 5 | 1 |
| 0 | 1 | [1] | 3 | 6 + 2*3=12 | 2 |
| 0 | 2 | [1,2] | 6 | 6 + 3*6=24 | 3 |
| 0 | 3 | [1,3] | 10 | 6 + 4*10=46 >25, shrink left=1 | 3 |
| 1 | 3 | [3] | 8 | 6 + 3*8=30 >25, shrink left=2 | 3 |
| 2 | 3 | [3] | 7 | 3 + 2*7=17 | 3 |
| 2 | 4 | [4] | 12 | 4 + 3*12=40>25, shrink left=3 | 3 |

Maximum length is `3`.

### Example 2

```
chargeTimes = [11,12,19]
runningCosts = [10,8,7]
budget = 19
```

All windows exceed budget. Maximum length is `0`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each index is added and removed from deque at most once; one pass through the array |
| Space | O(n) | Deque can hold at most all indices in worst case |

Linear time is efficient for `n <= 5*10^4`.
