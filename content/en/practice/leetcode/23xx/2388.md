## Solution

## Problem Understanding

The problem asks us to process a table called `CoffeeShop` that contains two columns: `id` and `drink`. Each row corresponds to an order at the coffee shop. Some `drink` entries are `null`, and our task is to replace every `null` with the most recent non-null `drink` value that appeared in the previous rows. The result must preserve the original row order of the input.

In simpler terms, we are performing a "forward fill" operation on the `drink` column: each `null` inherits the last known non-null value from above. The constraints give us some guarantees that simplify the problem. Specifically, the first row's `drink` is never null, which means there will always be a valid starting point for forward filling. There is no upper bound mentioned for the number of rows, but the problem is framed for standard SQL table sizes typically manageable in a single query scan.

Important edge cases include consecutive `null` values, `null` values appearing after a single non-null value, and a table with no `null` values at all. Since the first row is guaranteed to be non-null, we do not need to handle the case where the first value is `null`.

## Approaches

The most naive approach is to repeatedly scan each row and, for any null `drink`, look backwards to find the last non-null `drink`. This works correctly because it directly implements the definition of forward fill. However, this requires a nested scan for each null value, giving a quadratic time complexity, which would be inefficient for large tables.

The key insight for an optimal solution is that we do not need to scan backward repeatedly. Instead, we can maintain a variable that always stores the last non-null `drink`. As we iterate through the table in order, whenever we encounter a null, we replace it with the stored value. This way, each row is visited exactly once, resulting in linear time complexity.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^2) | O(1) | Scan backwards for each null to find previous non-null drink |
| Optimal | O(n) | O(1) | Keep a running last seen drink and fill nulls on the fly |

## Algorithm Walkthrough

1. Initialize a variable `last_drink` to store the most recent non-null drink. Start with `None`.
2. Iterate through the table rows in their given order.
3. For each row, check if `drink` is `null`.
4. If `drink` is not `null`, update `last_drink` to this value.
5. If `drink` is `null`, set it to the value of `last_drink`.
6. Continue until all rows are processed.
7. Return the table in the original order, with all nulls replaced.

Why it works: The invariant is that `last_drink` always holds the last non-null value encountered. Each null is replaced with exactly the value immediately preceding it that is not null, satisfying the problem requirement.

## Python Solution

```python
# Definition for a SQL table row simulation
from typing import List, Optional
from dataclasses import dataclass

@dataclass
class CoffeeShopRow:
    id: int
    drink: Optional[str]

def fill_null_drinks(table: List[CoffeeShopRow]) -> List[CoffeeShopRow]:
    last_drink = None
    for row in table:
        if row.drink is not None:
            last_drink = row.drink
        else:
            row.drink = last_drink
    return table
```

This Python implementation iterates through the list of `CoffeeShopRow` objects. `last_drink` tracks the last non-null `drink` encountered. Each null is replaced immediately, and the original row order is preserved. This is a direct translation of the optimal algorithm.

## Go Solution

```go
package main

type CoffeeShopRow struct {
    Id    int
    Drink *string
}

func fillNullDrinks(table []CoffeeShopRow) []CoffeeShopRow {
    var lastDrink *string
    for i := range table {
        if table[i].Drink != nil {
            lastDrink = table[i].Drink
        } else {
            table[i].Drink = lastDrink
        }
    }
    return table
}
```

In Go, we use pointers to represent nullable strings. `lastDrink` is a pointer to the last non-nil value. When a `nil` is encountered, we assign `lastDrink` to it. This preserves the original table order and handles nulls correctly.

## Edge Cases

One important edge case is when multiple consecutive null values appear. Without maintaining a `last_drink`, a naive backward scan would be required for each null, increasing time complexity unnecessarily. The current implementation correctly fills each null using the stored `last_drink`.

Another edge case is a table where all rows are non-null. The algorithm handles this without modification, since the `last_drink` is updated but never used to replace a null.

Finally, consider a table with only one row. Since the first row is guaranteed non-null, the function correctly returns it unchanged. This tests the lower bound of input size and ensures no index or null pointer errors occur.
