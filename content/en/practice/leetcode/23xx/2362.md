## Solution

## Problem Understanding

This problem asks us to identify the invoice with the highest total monetary value, then return the detailed line items for that invoice.

We are given two database tables:

The `Products` table stores the price of each product.

| product_id | price |
| --- | --- |
| Product identifier | Unit price |

The `Purchases` table stores which products appear in each invoice and how many units were purchased.

| invoice_id | product_id | quantity |
| --- | --- | --- |
| Invoice identifier | Product identifier | Quantity purchased |

To compute the total value of an invoice, we multiply:

```
quantity × product price
```

for every product in that invoice, then sum all those values together.

The problem requires us to:

1. Compute the total price of every invoice.
2. Find the invoice with the largest total value.
3. If multiple invoices share the same maximum total, choose the one with the smallest `invoice_id`.
4. Return all rows belonging to that selected invoice, including:

- `product_id`
- `quantity`
- `price`, where price means:

```
quantity × unit_price
```

The output can be returned in any order.

### Important Observations

The main challenge is that invoice totals are not directly stored. We must compute them by joining the `Purchases` and `Products` tables.

For example:

```
Invoice 2:
(4 × 100) + (3 × 200)
= 400 + 600
= 1000
```

Another important detail is tie breaking. If two invoices both total `1000`, we must select the smaller invoice ID.

### Constraints and Input Characteristics

Even though explicit numeric constraints are not shown in the statement, the problem is categorized as a database aggregation problem. This strongly suggests the intended solution uses:

- SQL joins
- aggregation with `SUM`
- grouping with `GROUP BY`
- ordering with `ORDER BY`

The data model guarantees:

- `product_id` is unique in `Products`
- `(invoice_id, product_id)` is unique in `Purchases`

This means:

- each product has exactly one price
- a product appears at most once per invoice

### Important Edge Cases

A naive implementation can fail in several situations.

If multiple invoices have the same total value, we must correctly choose the smallest `invoice_id`. Forgetting this tie breaking rule is a common mistake.

Another edge case occurs when an invoice contains multiple products. We must aggregate all rows belonging to the same invoice before comparing totals.

We must also correctly compute the per row output value:

```
quantity × unit_price
```

rather than returning the unit price itself.

Finally, invoices can contain only one product. The aggregation logic must still work correctly in that situation.

## Approaches

### Brute Force Approach

The brute force strategy is to process each invoice independently.

We could:

1. Iterate through every invoice.
2. For each invoice, iterate through all purchase rows.
3. Match every purchase row with its product price.
4. Compute the total invoice value manually.
5. Track the invoice with the maximum total.
6. Finally scan the purchase table again to return its rows.

This approach is logically correct because it explicitly computes every invoice total and compares them.

However, it is inefficient because repeated scans over the purchase data create unnecessary work. If there are many invoices and many purchase rows, repeatedly searching for matching rows becomes expensive.

### Optimal Approach

The key insight is that SQL databases are designed for grouped aggregation operations.

Instead of computing invoice totals one invoice at a time, we can:

1. Join `Purchases` with `Products`
2. Compute:

```
quantity * price
```
3. Group by `invoice_id`
4. Compute the total invoice value using `SUM`
5. Sort by:

- total descending
- invoice_id ascending
6. Select the top invoice
7. Return all rows for that invoice

This avoids repeated scans and leverages efficient relational operations.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(P × I) | O(1) | Repeatedly scans purchases for each invoice |
| Optimal | O(P log P) | O(P) | Uses aggregation and sorting efficiently |

Here:

- `P` = number of purchase rows
- `I` = number of invoices

## Algorithm Walkthrough

### Step 1, Join Purchases with Products

We first combine the two tables using `product_id`.

This gives us access to:

- invoice ID
- quantity
- unit price

for every purchase row.

The join result conceptually looks like:

| invoice_id | product_id | quantity | unit_price |
| --- | --- | --- | --- |
| 2 | 1 | 4 | 100 |

### Step 2, Compute Line Item Prices

For each joined row, compute:

```
quantity * price
```

This gives the monetary contribution of each product line.

Example:

```
4 × 100 = 400
```

### Step 3, Aggregate Invoice Totals

Group all rows by `invoice_id`.

For every invoice:

- sum all line item prices

This produces:

| invoice_id | total |
| --- | --- |
| 2 | 1000 |
| 4 | 1000 |

### Step 4, Resolve Ties

Sort by:

1. total descending
2. invoice_id ascending

This guarantees:

- the highest invoice total appears first
- ties are broken by smallest invoice ID

Then select the first row.

### Step 5, Return Invoice Details

Using the selected invoice ID:

- retrieve all rows belonging to that invoice
- compute:

```
quantity * price
```

for each row

Return:

- `product_id`
- `quantity`
- computed price

### Why it works

The algorithm works because every invoice total is computed exactly once using grouped aggregation.

Sorting by:

```
ORDER BY total DESC, invoice_id ASC
```

guarantees the first invoice is:

- the maximum total invoice
- and among ties, the smallest invoice ID

Finally, filtering by that invoice ID ensures we return exactly the required invoice details.

## Python Solution

Although this is a database problem, LeetCode database solutions are typically written directly in SQL. The following SQL query is the intended solution.

```python
SELECT
    p.product_id,
    p.quantity,
    p.quantity * pr.price AS price
FROM Purchases p
JOIN Products pr
    ON p.product_id = pr.product_id
WHERE p.invoice_id = (
    SELECT invoice_id
    FROM (
        SELECT
            p.invoice_id,
            SUM(p.quantity * pr.price) AS total_price
        FROM Purchases p
        JOIN Products pr
            ON p.product_id = pr.product_id
        GROUP BY p.invoice_id
        ORDER BY total_price DESC, p.invoice_id ASC
        LIMIT 1
    ) AS best_invoice
);
```

### Implementation Explanation

The solution consists of two major parts.

The inner query computes the total value of every invoice.

```
SUM(p.quantity * pr.price)
```

calculates the invoice total by summing all product contributions.

We then sort using:

```
ORDER BY total_price DESC, p.invoice_id ASC
```

This ensures:

- highest total first
- smallest invoice ID wins ties

`LIMIT 1` selects the desired invoice.

The outer query retrieves all rows belonging to that invoice and computes the final line item price:

```
p.quantity * pr.price
```

The join is required in both places because product prices live in the `Products` table.

## Edge Cases

### Multiple Invoices With the Same Total

This is the most important edge case in the problem.

Two invoices may produce identical total prices. A naive solution that only sorts by total descending may return the wrong invoice.

The implementation correctly handles this using:

```
ORDER BY total_price DESC, invoice_id ASC
```

This guarantees the smallest invoice ID is selected among ties.

### Invoice With Multiple Product Rows

An invoice can contain several different products.

A buggy implementation might accidentally compare individual line item totals instead of the full invoice total.

The solution avoids this by grouping on:

```
GROUP BY p.invoice_id
```

and summing all line items together.

### Invoice Containing a Single Product

Some invoices may contain only one row.

The aggregation logic must still work correctly even when there is only one purchase entry.

Since `SUM()` over a single row simply returns that row's value, the implementation naturally handles this case without special logic.
