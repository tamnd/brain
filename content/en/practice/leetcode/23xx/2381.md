# LeetCode 2381 - Shifting Letters II

## Problem Understanding

The problem gives us a lowercase English string `s` and a list of shift operations called `shifts`. Each operation is represented as:

```
[start, end, direction]
```

This means that every character from index `start` to index `end`, inclusive, must be shifted either forward or backward in the alphabet.

If `direction == 1`, we shift forward:

```
a -> b
b -> c
...
z -> a
```

If `direction == 0`, we shift backward:

```
b -> a
a -> z
```

After applying all operations, we must return the final transformed string.

The important detail is that multiple operations can overlap. A single character may be shifted many times, both forward and backward. The final character depends on the net total shift applied to that index.

The constraints are very important:

- `s.length <= 5 * 10^4`
- `shifts.length <= 5 * 10^4`

A naive implementation that updates every character for every operation could potentially perform:

```
5 * 10^4 * 5 * 10^4 = 2.5 * 10^9
```

operations, which is far too slow.

The problem guarantees:

- All indices are valid
- The string contains only lowercase English letters
- Each shift operation is well formed

Some important edge cases include:

- A shift range covering the entire string
- Multiple overlapping ranges
- Many forward and backward operations canceling each other out
- Large accumulated shifts, requiring modulo arithmetic
- Wraparound cases like `'z' -> 'a'` and `'a' -> 'z'`
- A string of length 1

## Approaches

### Brute Force Approach

The most straightforward solution is to simulate every shift operation directly.

For each operation:

1. Read `start`, `end`, and `direction`
2. Iterate from `start` to `end`
3. Shift each character one step forward or backward

This works correctly because it directly follows the problem statement.

However, it is inefficient because each operation may touch up to `n` characters. If we have `m` operations, the total complexity becomes:

```
O(n * m)
```

With both values up to `5 * 10^4`, this is too slow.

### Key Insight

The crucial observation is that we do not need to update every character immediately.

Instead, we can determine the total net shift that applies to each index.

This is a classic range update problem. When many operations modify intervals, a difference array or prefix sum technique allows us to process all updates efficiently.

For a forward shift:

```
[start, end, 1]
```

we can record:

```
+1 starting at start
-1 after end
```

For a backward shift:

```
[start, end, 0]
```

we record:

```
-1 starting at start
+1 after end
```

After processing all operations, a prefix sum reconstructs the actual shift value for every position.

This transforms the problem from repeated range updates into a single linear scan.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * m) | O(1) | Applies every shift directly |
| Optimal | O(n + m) | O(n) | Uses difference array and prefix sums |

Where:

- `n = len(s)`
- `m = len(shifts)`

## Algorithm Walkthrough

### Optimal Algorithm Using Difference Array

1. Create a difference array of size `n + 1`.

We use this array to record how shift values change across ranges, instead of updating every character individually.
2. Process each shift operation.

For each operation:

- If direction is forward (`1`), treat it as `+1`
- If direction is backward (`0`), treat it as `-1`

Then:

```
diff[start] += value
diff[end + 1] -= value
```

if `end + 1` is within bounds.

This marks where the effect begins and where it stops.
3. Build prefix sums.

Traverse the difference array from left to right.

The running sum at index `i` gives the total net shift that should be applied to `s[i]`.
4. Convert characters.

For each character:

- Convert it into a number from `0` to `25`
- Add the accumulated shift
- Use modulo `26` to handle wraparound
- Convert back into a character
5. Construct the final string.

Append each transformed character into a result array and join everything at the end.

### Why it works

The difference array guarantees that every range update contributes exactly once to every covered index.

The prefix sum reconstructs the cumulative effect of all overlapping operations. Since addition is associative, the final accumulated shift at each position is exactly the same as applying all operations one by one.

Modulo `26` correctly handles alphabet wraparound, ensuring characters remain lowercase English letters.

## Python Solution

```python
from typing import List

class Solution:
    def shiftingLetters(self, s: str, shifts: List[List[int]]) -> str:
        n = len(s)

        diff = [0] * (n + 1)

        for start, end, direction in shifts:
            value = 1 if direction == 1 else -1

            diff[start] += value

            if end + 1 < len(diff):
                diff[end + 1] -= value

        result = []
        current_shift = 0

        for i in range(n):
            current_shift += diff[i]

            original_value = ord(s[i]) - ord('a')

            shifted_value = (original_value + current_shift) % 26

            result.append(chr(shifted_value + ord('a')))

        return "".join(result)
```

The implementation begins by creating a difference array called `diff`. This array stores how the shift value changes across positions.

Each operation contributes either `+1` or `-1` depending on the direction. Instead of updating every index inside the range, we only mark where the effect starts and ends.

After processing all operations, the variable `current_shift` maintains the running prefix sum. At every index, it represents the total net shift affecting that character.

Characters are converted into numeric positions using:

```python
ord(character) - ord('a')
```

After applying the shift, modulo `26` ensures proper wraparound behavior.

Finally, the transformed characters are joined into the final string.

## Go Solution

```go
func shiftingLetters(s string, shifts [][]int) string {
	n := len(s)

	diff := make([]int, n+1)

	for _, shift := range shifts {
		start := shift[0]
		end := shift[1]
		direction := shift[2]

		value := -1
		if direction == 1 {
			value = 1
		}

		diff[start] += value

		if end+1 < len(diff) {
			diff[end+1] -= value
		}
	}

	result := make([]byte, n)

	currentShift := 0

	for i := 0; i < n; i++ {
		currentShift += diff[i]

		originalValue := int(s[i] - 'a')

		shiftedValue := (originalValue + currentShift) % 26

		if shiftedValue < 0 {
			shiftedValue += 26
		}

		result[i] = byte(shiftedValue) + 'a'
	}

	return string(result)
}
```

The Go implementation follows the same logic as the Python solution.

One important Go-specific detail is handling negative modulo values. In Go, `%` can produce negative results, so we explicitly add `26` when needed.

The final string is built using a `[]byte` slice for efficiency, since strings in Go are immutable.

## Edge Cases

One important edge case is alphabet wraparound. Characters shifted beyond `'z'` must return to `'a'`, and characters shifted before `'a'` must return to `'z'`. This is handled using modulo `26`, which keeps every transformed character inside the lowercase alphabet range.

Another important case is many overlapping operations. A naive implementation may repeatedly update the same indices and become extremely slow. The difference array avoids this issue by recording only boundary changes, ensuring efficient handling even when thousands of ranges overlap.

A third edge case is large accumulated shifts. A character might receive hundreds or thousands of total forward and backward operations. Without modulo arithmetic, these values could grow unnecessarily large. The implementation safely reduces everything modulo `26`, since shifting by 26 positions produces the original character.

A final subtle case involves negative shifts in languages like Go. Negative modulo behavior differs between languages. The Go implementation explicitly adjusts negative values by adding `26`, ensuring valid alphabet indices at all times.
