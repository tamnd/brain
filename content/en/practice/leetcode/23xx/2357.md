# LeetCode 2357 - Make Array Zero by Subtracting Equal Amounts

## Problem Understanding

The problem gives us an array of non-negative integers called `nums`. Our goal is to make every value in the array equal to `0` using the minimum number of operations.

In a single operation, we must choose a positive integer `x` that is less than or equal to the smallest non-zero element currently in the array. After choosing `x`, we subtract it from every positive element in the array. Elements that are already `0` remain unchanged.

The key detail is that subtraction only applies to positive values. This means zeros stay zero permanently, and once an element becomes zero, it is no longer affected by future operations.

The output should be the minimum number of operations required to turn the entire array into zeros.

For example, if the array is:

```
[1,5,0,3,5]
```

We can repeatedly subtract the smallest non-zero value:

1. Subtract `1` → `[0,4,0,2,4]`
2. Subtract `2` → `[0,2,0,0,2]`
3. Subtract `2` → `[0,0,0,0,0]`

So the answer is `3`.

The constraints are very small:

- `1 <= nums.length <= 100`
- `0 <= nums[i] <= 100`

This tells us that even less efficient approaches would still work in practice. However, the problem is designed to test observation and reasoning rather than performance optimization.

Several edge cases are important:

- Arrays containing only zeros should return `0`.
- Duplicate positive values should not increase the number of operations unnecessarily.
- Arrays with a single positive value always require exactly one operation.
- The smallest value can change after each operation, so simulation must be handled carefully if using a brute-force approach.

## Approaches

### Brute Force Approach

A straightforward solution is to literally simulate the operations described in the problem.

At each step:

1. Find the smallest non-zero element.
2. Subtract it from every positive element.
3. Count one operation.
4. Repeat until the array becomes all zeros.

This approach works because it directly follows the rules given in the problem statement. Eventually, at least one positive number becomes zero during each operation, so the process always terminates.

However, this solution repeatedly scans and modifies the array. While the constraints are small enough that this is acceptable, it is more work than necessary.

The deeper observation is that every operation eliminates one distinct positive value from the array.

### Key Insight for the Optimal Solution

Suppose the array contains the positive values:

```
1, 3, 5
```

No matter how we perform the operations, we will need:

- One operation to remove all `1`s
- One operation to remove all remaining `3`s
- One operation to remove all remaining `5`s

Duplicate values do not require extra operations because the same subtraction affects all matching numbers simultaneously.

This means the answer is simply:

> The number of distinct positive integers in the array.

So instead of simulating operations, we can:

1. Ignore zeros.
2. Put all positive numbers into a set.
3. Return the size of the set.

## Algorithm Walkthrough

### Optimal Algorithm

1. Create an empty set called `distinct_positive`.

A set automatically stores only unique values, which is exactly what we need because duplicate positive numbers do not require additional operations.
2. Iterate through every number in `nums`.

We examine each element one time.
3. Ignore any value equal to `0`.

Zero values already satisfy the final condition and never require processing.
4. Add every positive number to the set.

Since sets only keep unique elements, duplicates are automatically removed.
5. Return the size of the set.

Each distinct positive value corresponds to one required operation.

### Why it works

Each operation removes the current smallest positive value from the array. After that subtraction, all elements equal to that minimum become zero. Therefore, every operation eliminates exactly one distinct positive number.

Since duplicate values disappear together in the same operation, the total number of operations equals the number of distinct positive integers in the array.

## Python Solution

```python
from typing import List

class Solution:
    def minimumOperations(self, nums: List[int]) -> int:
        distinct_positive = set()

        for num in nums:
            if num > 0:
                distinct_positive.add(num)

        return len(distinct_positive)
```

The implementation starts by creating a set named `distinct_positive`. This set stores only unique positive integers.

We then iterate through the array one element at a time. If the current number is greater than zero, we add it to the set. Zeros are ignored because they already satisfy the target condition.

Finally, we return the size of the set using `len(distinct_positive)`. The size represents the number of distinct positive values, which equals the minimum number of operations required.

## Go Solution

```go
func minimumOperations(nums []int) int {
    distinctPositive := make(map[int]bool)

    for _, num := range nums {
        if num > 0 {
            distinctPositive[num] = true
        }
    }

    return len(distinctPositive)
}
```

The Go implementation uses a map instead of a set because Go does not provide a built-in set type. The map keys represent unique positive numbers, and the boolean values are simply placeholders.

The logic is otherwise identical to the Python solution:

- Ignore zeros
- Store positive numbers uniquely
- Return the number of unique positive values

There are no integer overflow concerns because the constraints are very small. Go slices naturally handle the input array efficiently.

## Edge Cases

One important edge case is an array containing only zeros. A naive simulation might accidentally attempt to find the smallest non-zero value and fail because none exists. The optimal solution handles this naturally because zeros are ignored, leaving the set empty and producing an answer of `0`.

Another important case is when all positive numbers are identical, such as `[4,4,4]`. A buggy implementation might incorrectly count each element separately and return `3`. The set-based approach correctly stores only one unique value, producing the correct answer of `1`.

A third edge case occurs when zeros are mixed between positive values, such as `[0,1,0,2]`. Since operations only affect positive numbers, zeros must remain unchanged throughout the process. The implementation handles this correctly by explicitly ignoring zeros during iteration.

Finally, arrays where every positive number is distinct, such as `[1,2,3,4,5]`, represent the maximum possible number of operations. The algorithm correctly returns the number of unique positive values, which is `5` in this case.
