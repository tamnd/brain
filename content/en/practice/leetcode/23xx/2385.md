## Solution

## Problem Understanding

This problem models the spread of an infection through a binary tree. The infection starts at a specific node, identified by the value `start`, and spreads every minute to all adjacent nodes.

In a binary tree, adjacency means:

- A parent is adjacent to its children.
- A child is adjacent to its parent.

At minute `0`, only the starting node is infected. At minute `1`, all nodes directly connected to it become infected. At minute `2`, infection spreads one more level outward, and so on. The task is to determine how many minutes it takes until every node in the tree becomes infected.

The input consists of:

- `root`, which is the root node of the binary tree.
- `start`, the value of the node where the infection begins.

The output is a single integer representing the total number of minutes required to infect the entire tree.

The constraints are important:

- The tree can contain up to `10^5` nodes.
- Node values are unique.
- The start node is guaranteed to exist.

These constraints immediately tell us that inefficient repeated traversals will likely time out. Any solution worse than linear time becomes risky at this scale. Since values are unique, we can safely use node values as identifiers inside hash maps or adjacency lists.

There are several important edge cases:

- A tree with only one node. Infection starts there, so the answer is `0`.
- A highly skewed tree, which behaves like a linked list. Recursive depth must be considered carefully.
- The start node could be a leaf node, forcing the infection to travel upward first before spreading elsewhere.
- The start node could already be near the center of the tree, reducing total infection time.

The core difficulty is that binary trees normally only allow downward traversal. Infection, however, spreads in all directions, including upward toward parents. That means we must somehow model the tree as an undirected graph.

## Approaches

### Brute Force Approach

A brute force strategy would simulate the infection minute by minute directly on the tree.

At each minute:

1. Find all currently infected nodes.
2. Traverse the tree again to locate all uninfected adjacent nodes.
3. Mark those nodes infected for the next minute.
4. Repeat until every node becomes infected.

This works because it accurately models the infection process. However, it is extremely inefficient. Every infection wave may require scanning large portions of the tree repeatedly.

In the worst case, if the tree has `n` nodes arranged like a chain, the infection spreads one node per minute. That leads to roughly:

- `n` rounds of infection
- Each round potentially scanning `n` nodes

This results in `O(n^2)` time complexity, which is too slow for `10^5` nodes.

### Optimal Approach

The key observation is that infection spreading through a tree is equivalent to breadth-first search on an undirected graph.

A binary tree normally only stores child pointers, but infection can travel both downward and upward. Therefore, we first transform the tree into a graph representation where every edge is bidirectional.

For example:

- Parent connects to child
- Child connects back to parent

Once the graph is built:

1. Start BFS from the `start` node.
2. Each BFS level corresponds to one minute of infection spread.
3. The number of BFS layers required to visit all nodes is the answer.

This works efficiently because BFS naturally models shortest-path expansion in waves, which exactly matches the infection process.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeatedly scans the tree during infection simulation |
| Optimal | O(n) | O(n) | Converts tree to graph and uses BFS infection spreading |

## Algorithm Walkthrough

1. Create an adjacency list to represent the tree as an undirected graph.

A binary tree only provides child references, but infection spreads both upward and downward. We therefore need a structure that allows traversal in all directions.

For every parent-child relationship:

- Add child to parent's neighbor list
- Add parent to child's neighbor list
2. Traverse the tree to build the graph.

Use DFS or BFS starting from the root.

Whenever we visit a node:

- If it has a left child, connect them bidirectionally
- If it has a right child, connect them bidirectionally

After this step, every node can reach all adjacent nodes in constant time.
3. Initialize BFS from the start node.

Use a queue for BFS because infection spreads level by level.

Also maintain a `visited` set so we do not revisit already infected nodes.
4. Process BFS level by level.

Each BFS layer represents one minute.

For every node currently infected:

- Visit all neighbors
- If a neighbor has not yet been infected, add it to the queue
5. Increment the minute counter after processing each BFS level.

This matches the infection timing model:

- Nodes in the current level infect their neighbors
- Those neighbors become infected in the next minute
6. Continue until the queue becomes empty.

Once BFS finishes, every reachable node has been infected. Since the graph is connected, that means the entire tree is infected.
7. Return the total number of minutes.

### Why it works

BFS explores nodes in increasing order of distance from the starting node. Every edge traversal corresponds to one minute of infection spread. Therefore, the farthest node from the start determines the total infection time. Since BFS guarantees shortest-path distances in an unweighted graph, the algorithm correctly computes the minimum number of minutes required to infect the entire tree.

## Python Solution

```python
from collections import defaultdict, deque
from typing import Optional

# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

class Solution:
    def amountOfTime(self, root: Optional[TreeNode], start: int) -> int:
        graph = defaultdict(list)

        def build_graph(node: Optional[TreeNode]) -> None:
            if not node:
                return

            if node.left:
                graph[node.val].append(node.left.val)
                graph[node.left.val].append(node.val)
                build_graph(node.left)

            if node.right:
                graph[node.val].append(node.right.val)
                graph[node.right.val].append(node.val)
                build_graph(node.right)

        build_graph(root)

        queue = deque([start])
        visited = {start}
        minutes = -1

        while queue:
            level_size = len(queue)

            for _ in range(level_size):
                current = queue.popleft()

                for neighbor in graph[current]:
                    if neighbor not in visited:
                        visited.add(neighbor)
                        queue.append(neighbor)

            minutes += 1

        return minutes
```

The implementation begins by constructing an adjacency list called `graph`. A `defaultdict(list)` is used because each node may connect to multiple neighbors.

The `build_graph` function traverses the binary tree recursively. Whenever it encounters a parent-child relationship, it adds edges in both directions. This transforms the tree into an undirected graph.

After the graph is built, BFS begins from the `start` node. The queue stores currently infected nodes, and the `visited` set ensures nodes are processed only once.

The BFS loop processes one layer at a time. Every iteration of the outer loop represents one minute of infection spread. All neighbors discovered during the current minute become infected in the next minute.

The variable `minutes` starts at `-1` because the first BFS layer corresponds to minute `0`. After processing all layers, the final value equals the total infection time.

## Go Solution

```go
package main

/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */

func amountOfTime(root *TreeNode, start int) int {
    graph := make(map[int][]int)

    var buildGraph func(node *TreeNode)

    buildGraph = func(node *TreeNode) {
        if node == nil {
            return
        }

        if node.Left != nil {
            graph[node.Val] = append(graph[node.Val], node.Left.Val)
            graph[node.Left.Val] = append(graph[node.Left.Val], node.Val)
            buildGraph(node.Left)
        }

        if node.Right != nil {
            graph[node.Val] = append(graph[node.Val], node.Right.Val)
            graph[node.Right.Val] = append(graph[node.Right.Val], node.Val)
            buildGraph(node.Right)
        }
    }

    buildGraph(root)

    queue := []int{start}
    visited := make(map[int]bool)
    visited[start] = true

    minutes := -1

    for len(queue) > 0 {
        levelSize := len(queue)

        for i := 0; i < levelSize; i++ {
            current := queue[0]
            queue = queue[1:]

            for _, neighbor := range graph[current] {
                if !visited[neighbor] {
                    visited[neighbor] = true
                    queue = append(queue, neighbor)
                }
            }
        }

        minutes++
    }

    return minutes
}
```

The Go solution follows the same logic as the Python version.

Instead of Python dictionaries and sets, Go uses:

- `map[int][]int` for the adjacency list
- `map[int]bool` for visited tracking

The BFS queue is implemented using a slice. Nodes are removed from the front using `queue = queue[1:]`.

Go requires explicit nil checks for tree nodes, while Python uses truthiness checks such as `if not node`.

Since all values remain within standard integer ranges, integer overflow is not a concern.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each node and edge is processed at most once |
| Space | O(n) | Graph, queue, and visited set all store up to n nodes |

The graph construction visits every node exactly once. Since a tree with `n` nodes has `n - 1` edges, the adjacency list also requires linear space.

The BFS traversal processes every node once and every edge at most twice, once from each direction. Therefore, the total runtime remains linear.

## Edge Cases

One important edge case is a tree containing only a single node. In this scenario, the infection starts at the only existing node, so the entire tree is already infected at minute `0`. A buggy implementation might incorrectly return `1` if it increments the timer too early. This solution handles it correctly because BFS processes exactly one layer and returns `0`.

Another important edge case is a highly skewed tree. Such a tree behaves like a linked list, meaning the infection spreads one node per minute. This can expose inefficient implementations that repeatedly scan the tree. The graph plus BFS approach remains linear because each node is visited exactly once.

A third critical edge case occurs when the starting node is a leaf. Infection must first travel upward through parent connections before reaching other branches. A solution that only traverses downward from children would fail completely. By converting the tree into an undirected graph, the implementation naturally supports movement in both directions.

A fourth subtle edge case is when the start node is near the center of the tree. In that situation, multiple branches become infected simultaneously. BFS correctly models this parallel spreading because all nodes in the same layer are processed together before advancing the minute counter.
