## Solution

## Problem Understanding

The problem asks us to compute the minimum cost to reach each stop along a train line that has two parallel routes: a regular route and an express route. We start at stop 0 on the regular route. At each stop, we can either continue on the current route or transfer from the regular to the express route, incurring a fixed `expressCost`. There is no additional cost to transfer back from the express route to the regular route, nor to continue on the express route.

The input consists of two arrays, `regular` and `express`, which are **1-indexed**, where `regular[i]` represents the cost to go from stop `i-1` to `i` via the regular route, and `express[i]` is the cost for the express route. The `expressCost` is the fixed fee for switching from the regular to the express route. We are to return a **1-indexed array** `costs`, where `costs[i]` represents the minimum total cost to reach stop `i` from stop `0`.

Constraints indicate `1 <= n <= 10^5` and all costs are positive integers up to `10^5`. This ensures that we must find an efficient `O(n)` approach, as a naive method that evaluates all possible paths would be too slow.

Key edge cases include `n=1`, where there is only one stop to compute, and cases where transferring to the express route is either always beneficial or never beneficial. Additionally, all transfers must respect the expressCost rule.

## Approaches

### Brute Force

A brute-force solution would enumerate every possible sequence of moves, computing the total cost for every possible combination of regular and express choices at each stop. For stop `i`, we could either continue on the same route or switch routes if allowed. This produces a **combinatorial explosion** as the number of sequences grows exponentially with `n`. While correct, this method is computationally infeasible for `n` up to 10^5.

### Optimal Approach

The key observation is that we can maintain two running totals while iterating through stops: the minimum cost to reach the current stop **on the regular route** and the minimum cost **on the express route**. At each stop, the cost to reach the next stop on the regular route is simply the current regular cost plus `regular[i]`. The cost to reach the next stop on the express route is the minimum of:

1. Continuing on the express route: `express_cost + express[i]` from the previous express stop.
2. Transferring from regular to express: `regular_cost + expressCost + express[i]`.

By iteratively updating these two running totals, we can compute the minimum cost at each stop in **O(n)** time without backtracking.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(n) | Evaluates all sequences of route choices |
| Optimal | O(n) | O(n) | Maintains running min cost for regular and express routes |

## Algorithm Walkthrough

1. Initialize two variables: `regCost` as 0 (cost to reach stop 0 on the regular route) and `expCost` as `expressCost` (cost to transfer immediately if desired).
2. Initialize an empty list `costs` to store the minimum cost to reach each stop.
3. Iterate over each stop `i` from 0 to n-1:

a. Compute the new cost to reach stop `i+1` via the regular route as `regCost + regular[i]`.

b. Compute the new cost to reach stop `i+1` via the express route as `min(expCost + express[i], regCost + expressCost + express[i])`.

c. Append `min(newRegCost, newExpCost)` to `costs` for the minimum cost to reach stop `i+1`.

d. Update `regCost` and `expCost` with the newly computed values for the next iteration.
4. Return the `costs` list.

Why it works: At every stop, we maintain the invariant that `regCost` and `expCost` represent the minimum cost to reach that stop using the regular and express routes respectively. This guarantees that `min(regCost, expCost)` always represents the overall minimum cost.

## Python Solution

```python
from typing import List

class Solution:
    def minimumCosts(self, regular: List[int], express: List[int], expressCost: int) -> List[int]:
        n = len(regular)
        regCost, expCost = 0, expressCost
        costs = []
        
        for i in range(n):
            newRegCost = regCost + regular[i]
            newExpCost = min(expCost + express[i], regCost + expressCost + express[i])
            costs.append(min(newRegCost, newExpCost))
            regCost, expCost = newRegCost, newExpCost
        
        return costs
```

The Python implementation directly follows the algorithm. We initialize the running costs for both routes and iteratively compute the cost to reach the next stop, updating the running totals. The `costs` list stores the minimum cost for each stop.

## Go Solution

```go
func minimumCosts(regular []int, express []int, expressCost int) []int64 {
    n := len(regular)
    costs := make([]int64, n)
    
    var regCost int64 = 0
    var expCost int64 = int64(expressCost)
    
    for i := 0; i < n; i++ {
        newRegCost := regCost + int64(regular[i])
        newExpCost := minInt64(expCost+int64(express[i]), regCost+int64(expressCost)+int64(express[i]))
        costs[i] = minInt64(newRegCost, newExpCost)
        regCost, expCost = newRegCost, newExpCost
    }
    
    return costs
}

func minInt64(a, b int64) int64 {
    if a < b {
        return a
    }
    return b
}
```

The Go version uses `int64` to prevent potential integer overflow, as the sum of costs could exceed `int` for large inputs. A helper function `minInt64` is used to compute the minimum of two `int64` values. Otherwise, the algorithm mirrors the Python solution.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | We iterate once over all stops and perform constant-time calculations at each stop |
| Space | O(n) | The output list `costs` stores one entry per stop |

The reasoning is that the algorithm only maintains a constant number of running totals and computes the cost for each stop in one pass, which is linear in the number of stops.
