## Solution

## Problem Understanding

The problem gives us an array `tasks` where each value represents a task type. The tasks must be completed strictly in the given order. We are not allowed to reorder them.

We are also given an integer `space`, which defines a cooldown period between two executions of the same task type. If we perform task type `x` on day `d`, then the next time we perform task type `x` must be at least `space + 1` days later.

Every day we have exactly two choices:

- Execute the next task in the array
- Take a break

Our goal is to minimize the total number of days required to finish all tasks.

The important detail is that tasks must be processed sequentially. We cannot skip ahead and perform later tasks first. This makes the problem much simpler than the classic Task Scheduler problem where reordering is allowed.

For example, suppose:

```
tasks = [1,2,1]
space = 2
```

If we execute task `1` on day 1 and task `2` on day 2, then task `1` cannot be executed again until day 4 because two full days must separate executions of the same type.

So the schedule becomes:

```
Day 1 -> 1
Day 2 -> 2
Day 3 -> break
Day 4 -> 1
```

The answer is 4 days.

The constraints are important:

- `tasks.length` can be up to `10^5`
- `tasks[i]` can be as large as `10^9`

Because task values are large, we cannot use an array indexed by task type. A hash map is the natural choice.

Because the input size is large, we need an algorithm close to linear time. Any simulation that advances one day at a time during long idle periods could become too slow.

Several edge cases are important:

- A task may appear only once, meaning no cooldown matters.
- Consecutive identical tasks may force many idle days.
- `space` may be very large relative to the number of unique tasks.
- All tasks may be distinct, meaning the answer is simply `len(tasks)`.

## Approaches

### Brute Force Approach

A straightforward simulation would process the schedule one day at a time.

We maintain:

- The current day
- The next task index
- The last execution day for each task type

For every day:

- Check whether the next task can be executed
- If enough cooldown time has passed, execute it
- Otherwise, insert a break day

This approach is correct because it literally simulates the scheduling process according to the problem rules.

However, the problem is efficiency. Suppose:

```
tasks = [1,1]
space = 100000
```

The algorithm would simulate almost 100000 idle days individually. In worse cases, the total number of simulated days can become extremely large.

That makes this approach too slow.

### Optimal Approach

The key observation is that we never actually need to simulate idle days one by one.

If a task cannot be executed yet, we already know the exact future day when it becomes valid again. Instead of incrementing the day repeatedly, we can jump directly to that day.

We maintain:

- `current_day`
- A hash map storing the last day each task type was executed

When processing a task:

- If the task has never appeared before, we execute it on the next day.
- Otherwise, we compute the earliest valid day:

```
last_execution_day + space + 1
```

If our current day is earlier than that, we jump forward directly.

This greedy strategy is optimal because:

- Tasks must remain in order
- Executing a task earlier is always better
- If cooldown prevents execution, waiting exactly until the first legal day is optimal

The result is a linear time solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(total simulated days) | O(n) | Simulates every day individually |
| Optimal | O(n) | O(n) | Jumps directly to the next valid execution day |

## Algorithm Walkthrough

1. Create a hash map called `last_seen`.

This map stores the last day each task type was executed.

Example:

```
last_seen[5] = 12
```

means task type `5` was last executed on day 12.
2. Initialize `current_day = 0`.

We use this variable to track the current schedule day.
3. Iterate through each task in `tasks`.

Since tasks must be completed in order, we process them sequentially.
4. Increment `current_day` by 1.

This represents attempting to execute the current task on the next available day.
5. Check whether the current task has appeared before.

If it has never appeared, we can execute it immediately.
6. If the task has appeared before, compute the earliest valid execution day.

Suppose:

```
last_seen[task] = d
```

Then the next valid day is:

$d + \text{space} + 1$

because we need `space` full days between executions.
7. If `current_day` is earlier than this required day, jump directly forward.

Instead of simulating breaks one by one, we simply set:

```
current_day = required_day
```
8. Record the execution day for this task.

Update:

```
last_seen[task] = current_day
```
9. Continue until all tasks are processed.
10. Return `current_day`.

This represents the minimum total days required.

### Why it works

The algorithm always executes each task at the earliest legally possible day.

Because tasks must remain in order, there is never any advantage in delaying a task longer than necessary. If a cooldown prevents execution, the earliest valid future day is uniquely determined. Jumping directly to that day preserves correctness while avoiding unnecessary simulation.

Thus, the algorithm constructs the optimal schedule greedily.

## Python Solution

```python
from typing import List

class Solution:
    def taskSchedulerII(self, tasks: List[int], space: int) -> int:
        last_seen = {}
        current_day = 0

        for task in tasks:
            current_day += 1

            if task in last_seen:
                earliest_day = last_seen[task] + space + 1

                if current_day < earliest_day:
                    current_day = earliest_day

            last_seen[task] = current_day

        return current_day
```

The solution begins by creating a dictionary called `last_seen`. This dictionary maps each task type to the last day it was executed.

The variable `current_day` tracks the schedule timeline.

For every task, we first advance to the next day because each task execution consumes one day.

If the task has been seen before, we compute the earliest day when it may legally execute again. That value is:

```python
last_execution_day + space + 1
```

If the current day is too early, we jump directly to the required day.

After determining the correct execution day, we update the hash map with the new execution time.

At the end, `current_day` contains the minimum total days needed.

## Go Solution

```go
func taskSchedulerII(tasks []int, space int) int64 {
	lastSeen := make(map[int]int64)
	var currentDay int64 = 0

	for _, task := range tasks {
		currentDay++

		if lastDay, exists := lastSeen[task]; exists {
			earliestDay := lastDay + int64(space) + 1

			if currentDay < earliestDay {
				currentDay = earliestDay
			}
		}

		lastSeen[task] = currentDay
	}

	return currentDay
}
```

The Go implementation follows exactly the same logic as the Python version.

One important difference is integer handling. The return type is `int64`, and the day counter is also stored as `int64` because the total number of days can exceed the range of a standard 32 bit integer.

The hash map is implemented using:

```go
map[int]int64
```

which maps task types to their last execution days.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each task is processed exactly once |
| Space | O(n) | The hash map may store every distinct task |

The algorithm performs a constant amount of work per task. Every lookup and update in the hash map is average-case O(1), so the total runtime is linear in the number of tasks.

The extra memory usage comes from storing the most recent execution day for each distinct task type.

## Edge Cases

One important edge case occurs when all tasks are unique. In this situation, cooldown restrictions never matter because no task type repeats. A buggy implementation might still try to insert unnecessary idle days. This implementation handles the case naturally because tasks not present in `last_seen` execute immediately.

Another important case is consecutive repeated tasks such as:

```
[1,1,1]
```

These inputs force large idle gaps. A naive day by day simulation could become inefficient for large cooldown values. This solution avoids that issue by jumping directly to the earliest legal day instead of simulating every break individually.

A third edge case involves very large task identifiers such as `10^9`. Since task values are not bounded by a small range, using an indexed array would be impossible or wasteful. The implementation correctly uses a hash map, allowing arbitrary task values while maintaining efficient lookups.

A final edge case is when natural spacing already satisfies the cooldown. For example:

```
tasks = [1,2,3,1]
space = 2
```

By the time the second `1` appears, enough days have already passed. The algorithm correctly recognizes this because the current day is already greater than or equal to the earliest allowed day, so no jump is needed.
