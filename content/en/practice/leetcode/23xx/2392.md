## Solution

## Problem Understanding

The problem asks us to construct a `k x k` matrix containing integers from `1` to `k` exactly once each, such that specific **row** and **column** ordering constraints are satisfied. The `rowConditions` array specifies pairs `[abovei, belowi]`, meaning `abovei` must appear in a row strictly above `belowi`. Similarly, `colConditions` specifies pairs `[lefti, righti]` meaning `lefti` must appear in a column strictly left of `righti`. All other cells in the matrix should contain `0`.

The input guarantees that all numbers are between `1` and `k`, that each number appears only once, and that the conditions are strictly ordering constraints (no number equals another in the pair). The output must return a valid matrix if possible, or an empty matrix if no valid arrangement exists.

The key challenge is that the ordering constraints form **partial orders**, and if the constraints form a cycle, no valid matrix exists. The problem size is moderate, with `k` up to 400 and conditions up to 10^4, making brute force infeasible. The problem reduces to performing **topological sorts** independently on rows and columns and then placing numbers in the matrix according to these orders.

Important edge cases include situations where cycles exist in the constraints, multiple valid topological orders are possible, or when some numbers are unconstrained in either rows or columns.

## Approaches

A naive brute-force approach would try all permutations of numbers from `1` to `k` and check if each permutation satisfies all row and column constraints. This approach is guaranteed to be correct, but it has a factorial time complexity, `O(k!)`, which is far too slow for `k` up to 400. It also requires generating and checking each permutation, which is combinatorially explosive.

The optimal approach leverages the observation that the constraints define **directed acyclic graphs (DAGs)** for rows and columns. We can perform a **topological sort** on these DAGs to determine a valid order of numbers in rows and columns. Once we have the row and column positions, placing the numbers in the matrix is straightforward. If a cycle is detected during the topological sort, we return an empty matrix immediately, because no solution exists.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(k!) | O(k) | Try all permutations of 1..k and validate constraints |
| Optimal | O(k + n + m) | O(k + n + m) | Topological sort on row and column constraints, then construct matrix |

Here, `n` is the number of row constraints and `m` is the number of column constraints. Since `n, m <= 10^4` and `k <= 400`, this is efficient.

## Algorithm Walkthrough

1. **Construct the DAGs**: For both `rowConditions` and `colConditions`, build adjacency lists representing directed edges. For rows, an edge `abovei -> belowi` indicates `abovei` must appear in a higher row. For columns, `lefti -> righti` indicates `lefti` must appear in a column to the left of `righti`.
2. **Compute indegrees**: Maintain an array of indegrees for each number from `1` to `k`. Each indegree counts how many constraints point to a number, i.e., how many numbers must come before it.
3. **Perform topological sort**: Use Kahn's algorithm. Start with numbers having indegree zero, place them in the order, and decrement indegrees of their neighbors. If at the end some numbers remain with nonzero indegree, a cycle exists and the sort fails.
4. **Map numbers to positions**: The resulting topological order provides the exact row index for each number from the row sort and the column index from the column sort.
5. **Construct the matrix**: Initialize a `k x k` matrix of zeros. For each number, place it at the `[row_index][col_index]` determined by the topological sorts.
6. **Return the matrix**: If both topological sorts succeeded, the matrix satisfies all constraints. Otherwise, return an empty matrix.

This approach works because topological sorting guarantees that the numbers appear in an order that respects all constraints. Each DAG is independent, so the row and column positions are determined separately and then combined.

## Python Solution

```python
from collections import deque
from typing import List

class Solution:
    def buildMatrix(self, k: int, rowConditions: List[List[int]], colConditions: List[List[int]]) -> List[List[int]]:
        def topo_sort(conditions):
            adj = {i: [] for i in range(1, k+1)}
            indegree = {i: 0 for i in range(1, k+1)}
            
            for u, v in conditions:
                adj[u].append(v)
                indegree[v] += 1
            
            queue = deque([node for node in range(1, k+1) if indegree[node] == 0])
            order = []
            
            while queue:
                node = queue.popleft()
                order.append(node)
                for neighbor in adj[node]:
                    indegree[neighbor] -= 1
                    if indegree[neighbor] == 0:
                        queue.append(neighbor)
            
            if len(order) != k:
                return []  # cycle detected
            return order
        
        row_order = topo_sort(rowConditions)
        if not row_order:
            return []
        col_order = topo_sort(colConditions)
        if not col_order:
            return []
        
        pos_in_row = {num: i for i, num in enumerate(row_order)}
        pos_in_col = {num: i for i, num in enumerate(col_order)}
        
        matrix = [[0]*k for _ in range(k)]
        for num in range(1, k+1):
            r, c = pos_in_row[num], pos_in_col[num]
            matrix[r][c] = num
        
        return matrix
```

The Python implementation uses a helper function `topo_sort` that performs Kahn's algorithm. It constructs adjacency lists and indegrees, collects nodes with zero indegree, and iteratively builds the topological order. If the final order does not include all numbers, a cycle exists, and an empty list is returned. The row and column orders are then mapped to matrix indices.

## Go Solution

```go
package main

func buildMatrix(k int, rowConditions [][]int, colConditions [][]int) [][]int {
    topoSort := func(conditions [][]int) []int {
        adj := make([][]int, k+1)
        indegree := make([]int, k+1)
        for _, pair := range conditions {
            u, v := pair[0], pair[1]
            adj[u] = append(adj[u], v)
            indegree[v]++
        }
        queue := []int{}
        for i := 1; i <= k; i++ {
            if indegree[i] == 0 {
                queue = append(queue, i)
            }
        }
        order := []int{}
        for len(queue) > 0 {
            node := queue[0]
            queue = queue[1:]
            order = append(order, node)
            for _, neigh := range adj[node] {
                indegree[neigh]--
                if indegree[neigh] == 0 {
                    queue = append(queue, neigh)
                }
            }
        }
        if len(order) != k {
            return nil
        }
        return order
    }

    rowOrder := topoSort(rowConditions)
    if rowOrder == nil {
        return [][]int{}
    }
    colOrder := topoSort(colConditions)
    if colOrder == nil {
        return [][]int{}
    }

    posRow := make([]int, k+1)
    posCol := make([]int, k+1)
    for i, num := range rowOrder {
        posRow[num] = i
    }
    for i, num := range colOrder {
        posCol[num] = i
    }

    matrix := make([][]int, k)
    for i := range matrix {
        matrix[i] = make([]int, k)
    }
    for num := 1; num <= k; num++ {
        r, c := posRow[num], posCol[num]
        matrix[r][c] = num
    }
    return matrix
}
```

The Go implementation mirrors Python, using slices and maps. We must explicitly handle nil slices when cycles are detected. Indices are managed carefully because Go arrays are zero-based.
