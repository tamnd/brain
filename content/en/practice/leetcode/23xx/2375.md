# LeetCode 2375 - Construct Smallest Number From DI String

## Problem Understanding

The problem gives us a string called `pattern` consisting only of the characters `'I'` and `'D'`. The length of this string is `n`, and we must construct a numeric string of length `n + 1`.

Each character in the pattern describes the relationship between two adjacent digits in the resulting number:

- `'I'` means the next digit must be larger than the current digit.
- `'D'` means the next digit must be smaller than the current digit.

The constructed number must satisfy several important conditions:

- Only digits from `'1'` to `'9'` may be used.
- Each digit may be used at most once.
- The resulting number must be lexicographically smallest among all valid possibilities.

Lexicographically smallest means that when comparing two candidate strings from left to right, the first smaller digit determines the better answer. This requirement strongly influences the algorithm design because we want smaller digits to appear as early as possible whenever valid.

For example, if the pattern is `"ID"`, then:

- The first relation requires `num[0] < num[1]`
- The second relation requires `num[1] > num[2]`

One valid answer is `"132"` because:

- `1 < 3`
- `3 > 2`

The constraints are small:

- `1 <= pattern.length <= 8`

This means the final number length is at most `9`, which matches the number of unique digits available from `1` through `9`.

Because the input size is tiny, even exponential solutions are technically feasible. However, the problem is designed to test whether we can discover the greedy observation that leads to a much cleaner and more efficient solution.

Several edge cases are important:

- A pattern consisting entirely of `'I'`, such as `"III"`, should produce the strictly increasing smallest sequence, `"1234"`.
- A pattern consisting entirely of `'D'`, such as `"DDD"`, should produce the smallest descending arrangement, `"4321"`.
- Alternating patterns such as `"DIDI"` can easily break naive implementations because the direction changes frequently.
- Consecutive `'D'` runs are especially tricky because they require temporary reversal behavior to maintain lexicographical minimality.

## Approaches

### Brute Force Approach

A straightforward brute force approach is to generate every possible permutation of digits `'1'` through `'9'` with length `n + 1`. For each permutation, we verify whether it satisfies the pattern rules.

Since the pattern length is at most `8`, the maximum number length is `9`. In the worst case, we would check all `9! = 362880` permutations.

The brute force method works because it exhaustively checks every valid arrangement, guaranteeing that the first lexicographically valid permutation encountered is the correct answer.

However, this approach is inefficient and unnecessarily complicated for such a structured problem. Even though the constraints are small enough to allow brute force, the factorial growth is still expensive compared to the optimal linear solution.

### Optimal Greedy Stack Approach

The key observation is that increasing sequences naturally work well with ascending digits, while decreasing sequences require temporary reversal.

Suppose we process digits in increasing order:

- Push digits onto a stack.
- Whenever we encounter an `'I'`, we flush the stack into the result.

Why does this work?

A consecutive run of `'D'` characters means we need digits in descending order. If we push increasing digits onto a stack and later pop them, they naturally come out in reverse order, exactly matching the decreasing requirement.

This approach guarantees lexicographical minimality because:

- We always use the smallest available digits first.
- We delay output only when forced by a decreasing sequence.
- Stack reversal creates the minimal valid descending arrangement.

The algorithm becomes both elegant and efficient.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O((n+1)!) | O((n+1)!) | Generates all permutations and checks validity |
| Optimal | O(n) | O(n) | Greedy stack-based construction |

## Algorithm Walkthrough

1. Initialize an empty stack and an empty result list.
2. Iterate through numbers from `1` to `n + 1`. At each step, push the current number onto the stack.
3. After pushing the number, decide whether to flush the stack into the result.
4. We flush the stack in two situations:

- We have reached the end of the pattern.
- The current pattern character is `'I'`.
5. When flushing:

- Pop all elements from the stack.
- Append them to the result in popped order.
6. The stack reversal automatically converts increasing insertion order into decreasing output order, which satisfies runs of `'D'`.
7. Continue until all digits are processed.
8. Join the result list into a string and return it.

### Why it works

The algorithm maintains an important invariant: digits are always pushed in increasing order, ensuring minimal lexicographical usage of available digits.

For increasing relationships, we immediately flush the stack so smaller digits appear earlier. For decreasing relationships, we delay output until the decreasing segment ends. The stack reversal then creates the smallest valid descending arrangement possible.

Because every digit is used exactly once and every pattern constraint is satisfied locally, the entire constructed string is globally valid and lexicographically minimal.

## Python Solution

```python
class Solution:
    def smallestNumber(self, pattern: str) -> str:
        stack = []
        result = []

        for i in range(len(pattern) + 1):
            stack.append(str(i + 1))

            if i == len(pattern) or pattern[i] == 'I':
                while stack:
                    result.append(stack.pop())

        return "".join(result)
```

The implementation follows the greedy stack strategy directly.

The `stack` temporarily stores digits that belong to the current segment. Digits are pushed in increasing order, which guarantees we always prioritize smaller values first.

The loop runs `len(pattern) + 1` times because the final number always contains one more digit than the pattern length.

Whenever we encounter an `'I'`, or when we reach the end of the pattern, we flush the stack. Popping from the stack reverses the insertion order, producing the descending sequence needed for preceding `'D'` characters.

Finally, we join the accumulated digits into the answer string.

## Go Solution

```go
func smallestNumber(pattern string) string {
    stack := []byte{}
    result := []byte{}

    for i := 0; i <= len(pattern); i++ {
        stack = append(stack, byte('1'+i))

        if i == len(pattern) || pattern[i] == 'I' {
            for len(stack) > 0 {
                result = append(result, stack[len(stack)-1])
                stack = stack[:len(stack)-1]
            }
        }
    }

    return string(result)
}
```

The Go implementation mirrors the Python logic closely.

Instead of storing strings, Go uses byte slices for efficiency. Characters are generated using `byte('1' + i)`, which converts numeric offsets into digit characters.

Slices naturally support stack behavior through append and slicing operations. Since the maximum input size is tiny, there are no concerns about integer overflow or memory pressure.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each digit is pushed and popped at most once |
| Space | O(n) | The stack and result store at most n+1 digits |

Each digit enters the stack exactly once and leaves exactly once. Therefore, the total amount of stack work is linear.

The extra memory usage is also linear because the stack may temporarily hold up to `n + 1` digits in the worst case.

## Edge Cases

A pattern consisting entirely of `'I'` is an important edge case because the smallest lexicographical result should simply be ascending digits. A flawed implementation might unnecessarily reverse segments or delay flushing. This implementation handles the case naturally because the stack is flushed immediately after every digit.

A pattern consisting entirely of `'D'` is another critical case. Here, the algorithm must delay output until the very end so that the entire sequence can be reversed. The stack structure handles this perfectly because all digits accumulate before being popped in descending order.

Alternating patterns such as `"DIDI"` are tricky because the algorithm repeatedly switches between delaying output and flushing immediately. Incorrect implementations may accidentally reuse digits or flush at the wrong moment. This implementation correctly flushes only when encountering `'I'` or reaching the end, preserving the required relationships throughout the sequence.
