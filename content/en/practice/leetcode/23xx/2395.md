# LeetCode 2395 - Find Subarrays With Equal Sum

## Problem Understanding

The problem gives us a 0-indexed integer array `nums` and asks whether there exist two different subarrays of length `2` that have the same sum.

A subarray of length `2` simply means any pair of adjacent elements. For an array of length `n`, there are exactly `n - 1` such subarrays:

- `nums[0] + nums[1]`
- `nums[1] + nums[2]`
- `nums[2] + nums[3]`
- and so on

The key requirement is that the two subarrays must start at different indices. Overlapping subarrays are completely valid, as long as they are different positions in the array.

For example, in the array `[4,2,4]`:

- Subarray starting at index `0` is `[4,2]`, sum = `6`
- Subarray starting at index `1` is `[2,4]`, sum = `6`

Since both sums are equal and the subarrays begin at different indices, the answer is `true`.

The constraints are relatively small:

- `2 <= nums.length <= 1000`
- Values can range from `-10^9` to `10^9`

The small array size means even a quadratic solution would technically pass, but the problem naturally lends itself to a more efficient linear-time approach.

Several edge cases are important to think about upfront. Arrays containing negative numbers can still produce equal sums, so we cannot rely on positivity assumptions. Arrays with repeated values may create duplicate sums quickly. Very short arrays of length `2` contain only one subarray, so the answer must always be `false`. Overlapping subarrays must also be handled correctly, since the problem explicitly allows them.

## Approaches

### Brute Force Approach

The brute-force solution considers every possible subarray of length `2`, computes its sum, and compares it against every other subarray of length `2`.

For each starting index `i`, we calculate:

```
nums[i] + nums[i + 1]
```

Then we compare that sum with all later subarrays:

```
nums[j] + nums[j + 1]
```

If any pair matches, we return `true`. Otherwise, after checking all pairs, we return `false`.

This approach is correct because it explicitly checks every valid pair of subarrays. However, it performs unnecessary repeated comparisons. Since there are `n - 1` subarrays of size `2`, comparing every pair leads to quadratic time complexity.

### Optimal Approach

The key observation is that we only need to know whether a sum has appeared before.

As we iterate through the array, we compute each adjacent-pair sum exactly once. We store previously seen sums in a hash set. If the current sum already exists in the set, then we have found two different subarrays with equal sums, so we can immediately return `true`.

Hash sets provide average `O(1)` lookup time, making the entire algorithm linear.

This works especially well because each subarray is represented only by its sum, and we only care about whether duplicates exist.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Compare every pair of adjacent subarray sums |
| Optimal | O(n) | O(n) | Use a hash set to detect duplicate sums efficiently |

## Algorithm Walkthrough

1. Create an empty hash set called `seen_sums`.

This set will store sums of previously encountered subarrays of length `2`.
2. Iterate through the array from index `0` to `len(nums) - 2`.

We stop at `len(nums) - 2` because each subarray needs two elements, `nums[i]` and `nums[i + 1]`.
3. For each index `i`, compute the current subarray sum:

```
current_sum = nums[i] + nums[i + 1]
```
4. Check whether `current_sum` already exists in `seen_sums`.

- If it does, then another subarray with the same sum has already appeared, so return `true`.
- If it does not, insert the sum into the set.
5. If the loop finishes without finding duplicates, return `false`.

### Why it works

The algorithm maintains the invariant that `seen_sums` contains exactly the sums of all previously processed subarrays of length `2`.

When processing a new subarray, if its sum already exists in the set, then there must be an earlier subarray with the same sum. Since the current subarray starts at a different index, the problem condition is satisfied.

If no duplicate sum is ever found, then no two subarrays of length `2` share the same sum.

## Python Solution

```python
from typing import List

class Solution:
    def findSubarrays(self, nums: List[int]) -> bool:
        seen_sums = set()

        for i in range(len(nums) - 1):
            current_sum = nums[i] + nums[i + 1]

            if current_sum in seen_sums:
                return True

            seen_sums.add(current_sum)

        return False
```

The implementation begins by creating an empty set named `seen_sums`. This set tracks all adjacent-pair sums encountered so far.

The loop iterates through every valid starting position for a subarray of length `2`. At each step, the algorithm computes the sum of the current adjacent pair.

The membership check:

```python
if current_sum in seen_sums:
```

is the key operation. If the sum already exists, the function immediately returns `True`, because another subarray with the same sum has already been processed.

If the sum is new, it gets added to the set so future subarrays can compare against it.

If the loop completes without finding duplicates, the function returns `False`.

## Go Solution

```go
func findSubarrays(nums []int) bool {
    seenSums := make(map[int]bool)

    for i := 0; i < len(nums)-1; i++ {
        currentSum := nums[i] + nums[i+1]

        if seenSums[currentSum] {
            return true
        }

        seenSums[currentSum] = true
    }

    return false
}
```

The Go solution uses a `map[int]bool` to simulate a hash set, since Go does not provide a built-in set type.

The logic is otherwise identical to the Python implementation. Each adjacent-pair sum is computed once, checked against the map, and stored if it has not appeared before.

There are no integer overflow concerns under Go's standard integer handling for this problem's constraints, because even the largest possible sum remains safely within 32-bit signed integer range.

## Edge Cases

One important edge case is the minimum valid array length of `2`. In this situation, only one subarray of length `2` exists, so it is impossible to have two different subarrays with equal sums. The implementation handles this naturally because the loop runs exactly once, adds the sum to the set, and then returns `False`.

Another important edge case involves overlapping subarrays. For example, in `[0,0,0]`, the two valid subarrays are `[0,0]` starting at index `0` and `[0,0]` starting at index `1`. These subarrays overlap, but they are still considered different because they begin at different indices. The algorithm correctly handles this because it processes every adjacent pair independently.

Negative numbers can also be a source of bugs if an implementation incorrectly assumes sums are always positive or increasing. For example, `[5,-5,5,-5]` produces sums of `0`, `0`, and `0`. The hash set approach works regardless of sign, because sums are treated simply as hashable integer values.

A final edge case involves very large values near the constraint boundaries. Since values can be as large as `10^9`, implementations must ensure addition remains safe. Python integers automatically handle arbitrary precision, while Go integers comfortably handle these sums within standard integer limits.
