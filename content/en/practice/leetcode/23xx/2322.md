[LeetCode Problem 2322](https://leetcode.com/problems/minimum-score-after-removals-on-a-tree/)

**Difficulty:** 🔴 Hard  
**Topics:** Array, Bit Manipulation, Tree, Depth-First Search

## Solution

## Problem Understanding

The problem gives us an undirected tree with `n` nodes. Each node contains an integer value from the array `nums`. Since the graph is a tree, it is connected and contains exactly `n - 1` edges.

We must remove exactly two distinct edges. Removing two edges from a tree always splits it into exactly three connected components. For each component, we compute the XOR of all node values inside that component.

If the three XOR values are:

- `x1`
- `x2`
- `x3`

then the score for that pair of removed edges is:

```
max(x1, x2, x3) - min(x1, x2, x3)
```

Our goal is to try every valid pair of removed edges and return the minimum possible score.

The constraints are important:

- `n <= 1000`
- The graph is guaranteed to be a valid tree
- Node values can be large, but XOR operations are constant time

The small upper bound of `1000` suggests that an `O(n^2)` solution is acceptable, while `O(n^3)` or worse may become risky.

The major challenge is efficiently determining the XOR values of the three resulting components after removing two edges.

Several edge cases are important:

- The two removed edges may lie on completely different branches.
- One removed edge may lie inside the subtree separated by the other edge.
- Very small trees, especially `n = 3`, produce exactly three isolated nodes after removing both edges.
- Multiple different edge pairs may produce the same score.
- The minimum possible score can be `0`.

Understanding the relationship between subtrees is the key to solving the problem efficiently.

## Approaches

### Brute Force Approach

A straightforward solution is to try every pair of edges.

For each pair:

1. Remove both edges from the tree.
2. Run DFS or BFS to identify the three connected components.
3. Compute the XOR value of each component.
4. Compute the score.
5. Keep track of the minimum score.

This approach is correct because it explicitly simulates every valid way to remove two edges.

However, it is inefficient.

There are `O(n^2)` pairs of edges. For each pair, we may perform graph traversals taking `O(n)` time. Therefore, the total complexity becomes `O(n^3)`.

With `n = 1000`, this can become too slow.

### Key Insight for the Optimal Solution

The important observation is that removing an edge in a rooted tree separates one subtree from the rest of the tree.

If we root the tree at node `0`, then every edge can be viewed as:

```
parent -> child
```

Removing that edge isolates the child's subtree.

This means:

- Every removed edge corresponds to a subtree XOR.
- We can precompute subtree XOR values using DFS.
- We can determine relationships between subtrees using entry and exit times from DFS traversal.

The critical cases are:

1. The two removed subtrees are disjoint.
2. One removed subtree is contained inside the other.

Using subtree XORs and ancestor checks, we can compute the three component XOR values in constant time for each edge pair.

This reduces the total complexity to `O(n^2)`.

## Algorithm Walkthrough

### Step 1: Build the Adjacency List

Since the input graph is a tree, we store it using an adjacency list.

This allows efficient DFS traversal.

### Step 2: Root the Tree at Node 0

We perform DFS starting from node `0`.

During DFS, we compute:

- `subtree_xor[node]`, XOR of all values in that subtree
- `tin[node]`, DFS entry time
- `tout[node]`, DFS exit time
- `parent[node]`

The subtree XOR is computed as:

$subtree\_xor(u)=nums[u]\oplus\bigoplus subtree\_xor(child)$

The entry and exit times allow ancestor checks.

A node `u` is an ancestor of `v` if:

$tin[u]\le tin[v]\le tout[u]$

### Step 3: Represent Each Edge by Its Child Node

For every edge `(u, v)`:

- If `parent[u] == v`, then removing the edge isolates subtree `u`
- Otherwise, removing the edge isolates subtree `v`

We store the child node for every removable edge.

### Step 4: Iterate Through Every Pair of Removed Edges

Suppose the two removed edges isolate subtrees rooted at:

- `a`
- `b`

There are two structural possibilities.

### Step 5: Handle Nested Subtrees

If `a` is an ancestor of `b`, then subtree `b` lies completely inside subtree `a`.

The three component XORs become:

```
x1 = subtree_xor[b]
x2 = subtree_xor[a] ^ subtree_xor[b]
x3 = total_xor ^ subtree_xor[a]
```

Similarly, if `b` is an ancestor of `a`, we swap the roles.

### Step 6: Handle Disjoint Subtrees

If the subtrees are disjoint:

```
x1 = subtree_xor[a]
x2 = subtree_xor[b]
x3 = total_xor ^ subtree_xor[a] ^ subtree_xor[b]
```

### Step 7: Compute the Score

For each pair:

```
score = max(x1, x2, x3) - min(x1, x2, x3)
```

Update the minimum answer.

### Why it works

Each removed edge disconnects exactly one subtree from the rooted tree. The subtree XOR values therefore fully determine the XOR of every resulting component.

The ancestor relationship determines whether the two removed subtrees overlap or are disjoint. Since XOR is associative and self-canceling, we can derive all three component XORs directly from precomputed subtree XORs.

Because every pair of removed edges is examined exactly once, the algorithm always finds the minimum possible score.

## Python Solution

```python
from typing import List

class Solution:
    def minimumScore(self, nums: List[int], edges: List[List[int]]) -> int:
        n = len(nums)

        graph = [[] for _ in range(n)]

        for u, v in edges:
            graph[u].append(v)
            graph[v].append(u)

        subtree_xor = [0] * n
        parent = [-1] * n
        tin = [0] * n
        tout = [0] * n

        timer = 0

        def dfs(node: int, par: int) -> None:
            nonlocal timer

            parent[node] = par
            tin[node] = timer
            timer += 1

            curr_xor = nums[node]

            for nei in graph[node]:
                if nei == par:
                    continue

                dfs(nei, node)
                curr_xor ^= subtree_xor[nei]

            subtree_xor[node] = curr_xor

            tout[node] = timer
            timer += 1

        dfs(0, -1)

        total_xor = subtree_xor[0]

        def is_ancestor(u: int, v: int) -> bool:
            return tin[u] <= tin[v] and tout[v] <= tout[u]

        children = []

        for u, v in edges:
            if parent[u] == v:
                children.append(u)
            else:
                children.append(v)

        answer = float("inf")

        m = len(children)

        for i in range(m):
            for j in range(i + 1, m):
                a = children[i]
                b = children[j]

                if is_ancestor(a, b):
                    x1 = subtree_xor[b]
                    x2 = subtree_xor[a] ^ subtree_xor[b]
                    x3 = total_xor ^ subtree_xor[a]

                elif is_ancestor(b, a):
                    x1 = subtree_xor[a]
                    x2 = subtree_xor[b] ^ subtree_xor[a]
                    x3 = total_xor ^ subtree_xor[b]

                else:
                    x1 = subtree_xor[a]
                    x2 = subtree_xor[b]
                    x3 = total_xor ^ subtree_xor[a] ^ subtree_xor[b]

                score = max(x1, x2, x3) - min(x1, x2, x3)

                answer = min(answer, score)

        return answer
```

The implementation begins by building the adjacency list representation of the tree.

The DFS traversal computes four important arrays:

- `subtree_xor`
- `parent`
- `tin`
- `tout`

The `subtree_xor` array stores the XOR of every subtree. The DFS timestamps allow constant time ancestor checks.

Next, every edge is converted into its deeper endpoint, meaning the child node whose subtree becomes separated if that edge is removed.

The nested loops iterate through every pair of removable edges. Depending on whether one subtree contains the other, the algorithm computes the three component XORs using different formulas.

Finally, the score is computed and the minimum value is returned.

## Go Solution

```go
package main

func minimumScore(nums []int, edges [][]int) int {
	n := len(nums)

	graph := make([][]int, n)

	for _, edge := range edges {
		u, v := edge[0], edge[1]
		graph[u] = append(graph[u], v)
		graph[v] = append(graph[v], u)
	}

	subtreeXor := make([]int, n)
	parent := make([]int, n)
	tin := make([]int, n)
	tout := make([]int, n)

	for i := 0; i < n; i++ {
		parent[i] = -1
	}

	timer := 0

	var dfs func(int, int)

	dfs = func(node int, par int) {
		parent[node] = par

		tin[node] = timer
		timer++

		currXor := nums[node]

		for _, nei := range graph[node] {
			if nei == par {
				continue
			}

			dfs(nei, node)
			currXor ^= subtreeXor[nei]
		}

		subtreeXor[node] = currXor

		tout[node] = timer
		timer++
	}

	dfs(0, -1)

	totalXor := subtreeXor[0]

	isAncestor := func(u, v int) bool {
		return tin[u] <= tin[v] && tout[v] <= tout[u]
	}

	children := []int{}

	for _, edge := range edges {
		u, v := edge[0], edge[1]

		if parent[u] == v {
			children = append(children, u)
		} else {
			children = append(children, v)
		}
	}

	answer := 1 << 30

	m := len(children)

	for i := 0; i < m; i++ {
		for j := i + 1; j < m; j++ {
			a := children[i]
			b := children[j]

			x1, x2, x3 := 0, 0, 0

			if isAncestor(a, b) {
				x1 = subtreeXor[b]
				x2 = subtreeXor[a] ^ subtreeXor[b]
				x3 = totalXor ^ subtreeXor[a]

			} else if isAncestor(b, a) {
				x1 = subtreeXor[a]
				x2 = subtreeXor[b] ^ subtreeXor[a]
				x3 = totalXor ^ subtreeXor[b]

			} else {
				x1 = subtreeXor[a]
				x2 = subtreeXor[b]
				x3 = totalXor ^ subtreeXor[a] ^ subtreeXor[b]
			}

			maxVal := max(x1, max(x2, x3))
			minVal := min(x1, min(x2, x3))

			score := maxVal - minVal

			if score < answer {
				answer = score
			}
		}
	}

	return answer
}

func max(a, b int) int {
	if a > b {
		return a
	}
	return b
}

func min(a, b int) int {
	if a < b {
		return a
	}
	return b
}
```

The Go implementation follows the same structure as the Python solution.

Some Go-specific details are worth noting:

- Slices are used for adjacency lists and auxiliary arrays.
- Go does not provide built-in `max` and `min` functions for integers, so helper functions are implemented.
- The DFS is written as a recursive closure.
- Integer overflow is not an issue because XOR values remain within standard integer limits for the given constraints.

| Node | Subtree Nodes | XOR |
| --- | --- | --- |
| 4 | [4] | 11 |
| 3 | [3,4] | 4 ^ 11 = 15 |
| 2 | [2] | 5 |
| 1 | [1,2,3,4] | 5 ^ 5 ^ 15 = 15 |
| 0 | [0,1,2,3,4] | 1 ^ 15 = 14 |

Total XOR:

```
14
```

### Step 3: Try Edge Pairs

Suppose we remove:

```
(1,2) and (1,3)
```

Separated subtrees:

- subtree rooted at 2, XOR = 5
- subtree rooted at 3, XOR = 15

Remaining component:

```
14 ^ 5 ^ 15 = 4
```

Three XORs:

```
[5, 15, 4]
```

Score:

```
15 - 4 = 11
```

Now try:

```
(0,1) and (1,2)
```

Since subtree `2` lies inside subtree `1`:

```
x1 = 5
x2 = 15 ^ 5 = 10
x3 = 14 ^ 15 = 1
```

Values:

```
[5, 10, 1]
```

Score:

```
10 - 1 = 9
```

This is the minimum answer.

The DFS preprocessing takes `O(n)` time.

There are `n - 1` edges, so the number of edge pairs is:

$\binom{n-1}{2}=O(n^2)$

Each pair requires only constant time calculations because subtree XORs and ancestor checks are already precomputed.

Therefore, the total complexity is `O(n^2)`.

## Edge Cases

### Nested Removed Subtrees

One removed edge may lie inside the subtree disconnected by the other removed edge. This is the trickiest case because the resulting components are not independent.

A naive implementation may double count nodes or compute incorrect XOR values.

The solution handles this using ancestor checks with DFS timestamps. When one subtree contains another, the formulas correctly subtract the inner subtree XOR from the outer subtree XOR.

### Completely Disjoint Subtrees

The removed edges may separate two independent branches of the tree.

In this case, the third component is simply the remaining nodes outside both subtrees.

The implementation computes:

```
total_xor ^ subtree_xor[a] ^ subtree_xor[b]
```

This works because XOR removes both disconnected subtree contributions.

### Smallest Possible Tree

When `n = 3`, there are only two edges, so both must be removed.

This creates three single-node components.

Small trees often expose indexing or parent-handling bugs. The DFS initialization and edge-to-child conversion work correctly even in this minimal configuration.
