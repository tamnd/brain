## Solution

## Problem Understanding

The problem gives us a binary string `s` and an integer `k`. We must find the longest subsequence of `s` whose binary value is less than or equal to `k`.

A subsequence means we can delete characters from the string while preserving the relative order of the remaining characters. We are not allowed to rearrange bits.

The selected subsequence represents a binary number. For example:

- `"101"` represents decimal `5`
- `"00101"` also represents decimal `5`, because leading zeroes are allowed
- The empty subsequence represents `0`

Our goal is to maximize the length of the chosen subsequence while ensuring its binary value does not exceed `k`.

A very important detail is that leading zeroes are free from a value perspective. Adding extra leading zeroes increases the subsequence length without increasing the numeric value. This observation becomes central to the optimal solution.

The constraints are:

- `1 <= s.length <= 1000`
- `1 <= k <= 10^9`

A string length of `1000` is large enough that brute force enumeration of all subsequences is impossible. Since a string of length `n` has `2^n` subsequences, even `n = 50` would already be far too large.

Another important observation comes from the size of `k`. Since `k <= 10^9`, its binary representation contains at most about `30` bits. This means any valid subsequence containing many significant `1` bits will quickly exceed `k`.

Several edge cases are important:

- Strings containing many zeroes, since zeroes can always be included
- Very small `k`, especially `k = 1`
- Strings consisting entirely of ones
- Cases where adding one more bit suddenly exceeds `k`
- Leading zeroes in the subsequence, which do not affect the value

## Approaches

### Brute Force Approach

The most direct approach is to generate every possible subsequence of `s`.

For each subsequence:

1. Construct the binary string
2. Convert it into a decimal number
3. Check whether it is less than or equal to `k`
4. Track the maximum valid length

This works because it explicitly examines every possible candidate subsequence, guaranteeing that the optimal answer will eventually be found.

However, the number of subsequences is `2^n`. With `n = 1000`, this is completely infeasible.

Even if conversion and validation were efficient, enumerating all subsequences is impossible within time limits.

### Key Insight for the Optimal Approach

The crucial observation is that zeroes are always beneficial.

Adding a zero to the left side of a binary number does not increase its value, but it does increase the subsequence length. Therefore, every zero in the string should always be included in the answer.

The only difficult part is deciding which ones to include.

Another key insight is that lower binary positions contribute less to the total value. Therefore, when choosing ones, we should prioritize bits from the right side of the string because they become less significant bits in the resulting subsequence.

We greedily process the string from right to left:

- Every zero is included
- A one is included only if adding its binary contribution keeps the total value `<= k`

Since binary weights double as we move left, processing from right to left naturally considers smaller contributions first.

This gives an efficient greedy solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n * n) | O(n) | Enumerates all subsequences |
| Optimal | O(n) | O(1) | Greedy selection from right to left |

## Algorithm Walkthrough

1. Initialize the answer with the count of zeroes.

Every zero can always be included because zeroes do not increase the numeric value of the binary number. Since we want the longest subsequence possible, including all zeroes is always optimal.
2. Traverse the string from right to left.

The rightmost bits correspond to the least significant binary positions. These bits contribute smaller values, making them cheaper to include.
3. Maintain the current binary value contributed by selected ones.

We keep a running total called `current_value`.
4. Maintain the current binary place value.

The first position from the right contributes `1`, the next contributes `2`, then `4`, then `8`, and so on.
5. When encountering a `'1'`, decide whether it can be included.

If:

```
current_value + place_value <= k
```

then we include this bit:

- Increase the answer length
- Add the place value to the current numeric value
6. Double the place value after each step.

Moving one position left doubles the binary weight.
7. Stop increasing place values once they exceed `k`.

Extremely large place values can never be included because they alone would exceed `k`.

### Why it works

The greedy strategy works because lower significance bits are always cheaper than higher significance bits.

If we can only afford a limited number of ones, selecting them from the right side maximizes how many we can include. Any solution using a more significant one instead of a less significant one would produce a larger numeric value without increasing the subsequence length.

Since zeroes are always free, including all zeroes is always optimal.

Therefore, the algorithm constructs the longest valid subsequence.

## Python Solution

```python
class Solution:
    def longestSubsequence(self, s: str, k: int) -> int:
        length = s.count('0')

        current_value = 0
        place_value = 1

        for i in range(len(s) - 1, -1, -1):
            if s[i] == '1':
                if current_value + place_value <= k:
                    current_value += place_value
                    length += 1

            if place_value <= k:
                place_value <<= 1

        return length
```

The implementation begins by counting all zeroes in the string. Since zeroes never increase the binary value, every zero can safely be included.

Next, the algorithm scans the string from right to left. This direction is important because the rightmost positions correspond to smaller binary weights.

The variable `current_value` stores the decimal value of the selected subsequence so far. The variable `place_value` represents the binary contribution of the current position.

Whenever the algorithm encounters a `'1'`, it checks whether adding that bit would keep the value within `k`. If so, the bit is included.

After processing each position, the binary place value doubles using a left shift.

The condition:

```python
if place_value <= k:
```

prevents unnecessary growth of extremely large powers of two.

## Go Solution

```go
func longestSubsequence(s string, k int) int {
	length := 0

	for _, ch := range s {
		if ch == '0' {
			length++
		}
	}

	currentValue := 0
	placeValue := 1

	for i := len(s) - 1; i >= 0; i-- {
		if s[i] == '1' {
			if currentValue+placeValue <= k {
				currentValue += placeValue
				length++
			}
		}

		if placeValue <= k {
			placeValue <<= 1
		}
	}

	return length
}
```

The Go implementation follows exactly the same logic as the Python version.

One small difference is that Go strings are indexed as byte arrays, so accessing `s[i]` directly returns a byte representing either `'0'` or `'1'`.

Integer overflow is not an issue because `k <= 10^9`, so the place values remain well within Go's integer range.

## Edge Cases

One important edge case is when the string contains only zeroes. Since zeroes do not contribute to the numeric value, every character can always be included. A buggy implementation might incorrectly attempt to interpret the entire binary string numerically and accidentally reject large sequences of zeroes. This implementation avoids that issue because it counts all zeroes immediately.

Another important case is when `k` is very small, especially `k = 1`. In this situation, only the smallest possible one bit can be included. Higher significance bits quickly exceed the limit. The right to left greedy traversal ensures that the algorithm always tries the cheapest bits first.

A third important edge case occurs when the string contains many ones. Binary place values grow exponentially, so including several high significance bits can overflow the allowed value very quickly. The algorithm safely handles this by checking whether adding the current place value would exceed `k` before including the bit.

Another subtle case involves leading zeroes in the subsequence. Since leading zeroes do not change the binary value, they should always be included whenever possible. Some incorrect solutions accidentally discard them when constructing the numeric value. This implementation correctly treats every zero as free length.
