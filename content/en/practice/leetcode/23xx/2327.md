# LeetCode 2327 - Number of People Aware of a Secret

## Problem Understanding

The problem models how a secret spreads through a population over a sequence of days.

Initially, on day `1`, exactly one person knows the secret. After discovering the secret, a person must wait `delay` days before they can begin sharing it with others. Once they become eligible to share, they will share the secret with one new person every day until they forget the secret.

A person forgets the secret exactly `forget` days after learning it. On the day they forget, they can no longer share the secret. This means their sharing period is:

- Start sharing on day `discover_day + delay`
- Stop sharing before day `discover_day + forget`

The task is to determine how many people still know the secret at the end of day `n`.

The input consists of:

- `n`, the total number of days to simulate
- `delay`, the number of days before someone starts sharing
- `forget`, the number of days before someone forgets

The output is the number of people who still remember the secret on day `n`, modulo `10^9 + 7`.

The constraints are relatively small:

- `2 <= n <= 1000`
- `1 <= delay < forget <= n`

Since `n` is at most `1000`, an `O(n^2)` solution is acceptable. However, we still want a clean and efficient dynamic programming approach.

Several edge cases are important:

- People may forget immediately after only a short sharing window.
- When `delay = 1`, sharing begins the next day, causing rapid exponential growth.
- When `forget` is close to `delay`, people have very limited time to spread the secret.
- Large counts require modulo arithmetic to prevent overflow.

## Approaches

### Brute Force Approach

A straightforward simulation would track every individual person separately.

For each person, we would record:

- The day they learned the secret
- Whether they are currently able to share
- Whether they have forgotten

Then, for each day:

1. Iterate through all people.
2. Determine who can still share.
3. Add newly informed people.
4. Remove or deactivate people who forgot.

This works because it directly follows the rules of the problem. However, the number of people can grow exponentially, especially when `delay = 1`. Explicitly storing every individual quickly becomes inefficient.

Even though the constraints are small enough that a careful simulation might pass, this approach is unnecessarily complicated and potentially expensive.

### Optimal Dynamic Programming Approach

The key observation is that we do not need to track individuals separately. Instead, we only need to know:

- How many people discover the secret on each day.

Suppose:

- `dp[i]` = number of people who first learn the secret on day `i`

If someone learns the secret on day `i`, then:

- They begin sharing on day `i + delay`
- They stop sharing before day `i + forget`

So every person who learned the secret on some earlier day contributes to future days within a predictable interval.

We can maintain a running count of people who are currently able to share the secret. This avoids repeatedly scanning earlier days.

The process becomes:

- Add people who become eligible to share today.
- Remove people who forget today.
- The remaining active sharers create new people today.

This produces an efficient dynamic programming simulation.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(total people generated) | O(total people generated) | Explicitly simulates every individual |
| Optimal | O(n) | O(n) | Uses dynamic programming with rolling sharer count |

## Algorithm Walkthrough

1. Create a dynamic programming array `dp` of size `n + 1`.

`dp[i]` represents the number of people who discover the secret on day `i`.
2. Initialize the starting condition.

On day `1`, exactly one person knows the secret:

```
dp[1] = 1
```
3. Maintain a variable called `sharing`.

This stores the number of people who are currently eligible to share the secret on the current day.
4. Iterate through days from `2` to `n`.

For each day:

- People who learned the secret on day `day - delay` become eligible to share today.
- People who learned the secret on day `day - forget` forget the secret today and stop sharing.
5. Update the `sharing` count.

Add newly eligible sharers:

```
sharing += dp[day - delay]
```

Remove people who forgot:

```
sharing -= dp[day - forget]
```
6. The number of new people learning the secret today equals the number of active sharers.

```
dp[day] = sharing
```
7. Apply modulo arithmetic after every update.

Since counts grow rapidly, all calculations must use:

```
MOD = 10^9 + 7
```
8. After processing all days, compute the final answer.

Anyone who learned the secret during the last `forget - 1` days still remembers it on day `n`.

Therefore, sum:

```
dp[n - forget + 1] through dp[n]
```

### Why it works

The algorithm maintains the invariant that `sharing` always equals the number of people who are currently able to spread the secret on the current day.

Each person contributes to the sharing population exactly during the interval:

```
[discover_day + delay, discover_day + forget - 1]
```

By adding people when they become eligible and removing them when they forget, we correctly model the active sharers for every day. Since every new person is created only by active sharers, the recurrence exactly matches the problem rules.

## Python Solution

```python
class Solution:
    def peopleAwareOfSecret(self, n: int, delay: int, forget: int) -> int:
        MOD = 10**9 + 7

        dp = [0] * (n + 1)
        dp[1] = 1

        sharing = 0

        for day in range(2, n + 1):
            if day - delay >= 1:
                sharing += dp[day - delay]

            if day - forget >= 1:
                sharing -= dp[day - forget]

            sharing %= MOD
            dp[day] = sharing

        result = 0

        for day in range(max(1, n - forget + 1), n + 1):
            result = (result + dp[day]) % MOD

        return result
```

The implementation follows the dynamic programming strategy directly.

The `dp` array stores how many people first learn the secret on each day. Initially, only one person learns the secret on day `1`.

The `sharing` variable tracks the number of people who can actively share on the current day. As the simulation advances day by day:

- People reaching the delay threshold are added to `sharing`
- People reaching the forget threshold are removed from `sharing`

After updating `sharing`, the value represents exactly how many new people learn the secret today, so we assign it to `dp[day]`.

Finally, we sum all people who still remember the secret on day `n`. Anyone who learned the secret before `n - forget + 1` has already forgotten it.

## Go Solution

```go
func peopleAwareOfSecret(n int, delay int, forget int) int {
    const MOD int = 1_000_000_007

    dp := make([]int, n+1)
    dp[1] = 1

    sharing := 0

    for day := 2; day <= n; day++ {
        if day-delay >= 1 {
            sharing = (sharing + dp[day-delay]) % MOD
        }

        if day-forget >= 1 {
            sharing = (sharing - dp[day-forget]) % MOD
        }

        if sharing < 0 {
            sharing += MOD
        }

        dp[day] = sharing
    }

    result := 0

    start := n - forget + 1
    if start < 1 {
        start = 1
    }

    for day := start; day <= n; day++ {
        result = (result + dp[day]) % MOD
    }

    return result
}
```

The Go implementation mirrors the Python solution closely.

One important difference is handling negative modulo results. In Go, the `%` operator can produce negative values, so after subtracting forgotten sharers we explicitly add `MOD` if needed.

The `dp` slice stores daily discoveries, and integer overflow is avoided because all operations are reduced modulo `1_000_000_007`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each day is processed exactly once |
| Space | O(n) | The DP array stores values for all days |

The algorithm performs constant work per day. We only maintain a rolling sharer count and a single dynamic programming array, so the runtime scales linearly with `n`.

The space usage is also linear because we store one value for each day from `1` through `n`.

## Edge Cases

One important edge case occurs when `delay = 1`. In this scenario, every newly informed person can start sharing the very next day. The number of people grows extremely quickly, and a naive simulation can become inefficient. The dynamic programming solution handles this efficiently because it tracks counts per day instead of individual people.

Another tricky case is when `forget = delay + 1`. Here, people only have one day during which they can share the secret before forgetting it. Off by one errors are very common in this situation. The implementation avoids mistakes by carefully defining the sharing interval as:

```
[discover_day + delay, discover_day + forget - 1]
```

People are removed from the sharing pool exactly on the day they forget.

A third important case is when nobody forgets before the final day, such as `forget = n`. In this situation, all discovered people still remember the secret at the end. The final summation logic correctly handles this because it sums all `dp` entries from day `1` onward when necessary.

Finally, modulo handling is essential because the number of people can become extremely large. The implementation applies modulo arithmetic after every update, preventing overflow and ensuring the result remains within valid bounds.
