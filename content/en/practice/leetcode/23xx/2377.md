## Solution

## Problem Understanding

The problem gives us a database table named `Olympic` that stores medal counts for different countries. Each row contains the country name along with the number of gold, silver, and bronze medals that country won.

The task is not to calculate anything new, but rather to return the rows in a very specific sorted order. The ranking rules are hierarchical:

1. Countries with more gold medals should appear first.
2. If two countries have the same number of gold medals, compare silver medals.
3. If silver medals are also tied, compare bronze medals.
4. If all medal counts are tied, sort the country names lexicographically in ascending order.

The input is simply the existing `Olympic` table. The expected output is the same rows, reordered according to the ranking rules.

The important detail is that medal counts must be sorted in descending order, while country names must be sorted in ascending alphabetical order. Mixing ascending and descending ordering correctly is the key part of the problem.

Because this is a database problem, the constraints are generally small enough that efficiency is not a concern. SQL database engines are highly optimized for sorting operations, so the focus here is understanding the ordering logic and expressing it correctly using `ORDER BY`.

Several edge cases are important:

- Multiple countries may tie in all medal categories, requiring lexicographical ordering.
- Some countries may have zero medals in every category.
- Countries may tie in gold and silver medals but differ in bronze medals.
- The table guarantees that `country` is unique because it is the primary key.

## Approaches

### Brute Force Approach

A brute force approach would manually simulate the ranking process. One possible strategy would be:

- Compare every pair of countries.
- Determine which one should come first using the medal rules.
- Repeatedly swap rows until the entire table becomes sorted.

Conceptually, this behaves similarly to a simple comparison-based sorting algorithm such as bubble sort. For every comparison, we would check gold medals first, then silver, then bronze, then country name.

This approach works because it eventually enforces the ranking rules across all rows. However, implementing manual sorting logic is unnecessary in SQL because relational databases already provide optimized sorting functionality through `ORDER BY`.

### Optimal Approach

The key insight is that SQL supports multi-column sorting directly.

We can express the exact ranking rules using an `ORDER BY` clause:

- `gold_medals DESC`
- `silver_medals DESC`
- `bronze_medals DESC`
- `country ASC`

SQL automatically applies these sorting conditions in order. It first sorts by gold medals. If ties occur, it moves to silver medals, then bronze medals, and finally country names.

This solution is concise, readable, and leverages the database engine's built-in sorting optimizations.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Manual comparison and swapping of rows |
| Optimal | O(n log n) | O(1) or O(n) depending on DB engine | Uses SQL `ORDER BY` with multiple sorting keys |

## Algorithm Walkthrough

1. Start with the `Olympic` table containing all countries and their medal counts.
2. Sort the rows by `gold_medals` in descending order because countries with more gold medals rank higher.
3. For countries with equal gold medal counts, sort by `silver_medals` in descending order. This resolves ties among countries with identical gold totals.
4. If countries are still tied after comparing silver medals, sort by `bronze_medals` in descending order.
5. If all medal counts are identical, sort by `country` in ascending lexicographical order. This guarantees deterministic ordering.
6. Return the sorted result set.

### Why it works

The algorithm works because SQL applies multi-column sorting sequentially. Each later sorting condition acts only when earlier conditions produce a tie. This exactly matches the ranking rules described in the problem statement, ensuring the output order is correct.

## Python Solution

Although this is fundamentally a SQL problem, LeetCode database problems are typically solved using SQL directly. Below is the required SQL query represented inside a Python string for completeness.

```python
class Solution:
    def sortOlympicTable(self) -> str:
        return """
        SELECT
            country,
            gold_medals,
            silver_medals,
            bronze_medals
        FROM Olympic
        ORDER BY
            gold_medals DESC,
            silver_medals DESC,
            bronze_medals DESC,
            country ASC;
        """
```

The query selects all columns from the `Olympic` table and applies the required ordering conditions.

The medal columns use `DESC` because higher medal counts should appear first. The `country` column uses `ASC` because lexicographically smaller names should come earlier when medal counts are tied.

The order of the sorting clauses is extremely important. SQL evaluates them sequentially, which mirrors the ranking hierarchy described in the problem.

## Edge Cases

### Countries With Identical Medal Counts

A common source of bugs is forgetting the final lexicographical ordering requirement. If two countries have exactly the same gold, silver, and bronze medal counts, the output must still be deterministic. The implementation handles this by adding `country ASC` as the final sorting condition.

### Countries With Zero Medals

Some countries may have zero medals in every category. A naive implementation might accidentally exclude them or sort them incorrectly. Since the query sorts all rows regardless of values, countries with zero medals are naturally included and correctly ordered.

### Partial Ties Across Medal Categories

Another tricky situation occurs when countries tie in gold medals but differ in silver or bronze medals. The ordering rules must be applied sequentially, not independently. The multi-column `ORDER BY` clause guarantees that ties are resolved in the correct priority order.
