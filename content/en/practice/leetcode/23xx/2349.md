# LeetCode 2349 - Design a Number Container System

## Problem Understanding

This problem asks us to design a data structure that supports two operations efficiently:

1. Assign a number to an index.
2. Find the smallest index currently assigned to a given number.

The important detail is that assignments can overwrite previous values. If an index already contains a number and we assign a different number to it, the old association disappears and the new one becomes active.

The `change(index, number)` operation means:

- Store `number` at `index`.
- If the index already had another number, replace it.

The `find(number)` operation means:

- Return the smallest index currently storing `number`.
- If no index stores that number, return `-1`.

The constraints are extremely important:

- Up to `10^5` total operations
- Index and number values can be as large as `10^9`

Because indices and numbers are huge, we cannot use arrays indexed directly by value. We need hash maps or similar sparse structures.

The total number of operations is also large enough that a naive linear scan for every `find()` would be too slow. A solution with `O(n)` per query could degrade to `O(10^10)` work in the worst case, which is infeasible.

There are several edge cases that matter:

- Calling `find()` for a number that does not exist.
- Replacing a number at an index multiple times.
- Reassigning the same number to the same index.
- Old indices becoming stale after replacement.
- Multiple indices storing the same number, where we must always return the minimum one.

These edge cases make careful bookkeeping essential.

## Approaches

### Brute Force Approach

A straightforward solution is to maintain a hash map:

```
index -> number
```

For every `change(index, number)` operation, we simply update the map.

For every `find(number)` operation, we scan through all stored indices and locate the smallest index whose value equals `number`.

This approach is correct because it directly checks the current state of every stored index.

However, it is too slow. If we have `n` stored indices, each `find()` operation costs `O(n)`. Since there can be up to `10^5` operations, the total complexity can become quadratic.

### Optimal Approach

The key insight is that we need fast access to:

- The current number stored at an index
- The minimum index for a given number

A heap is ideal for retrieving minimum values efficiently.

We maintain:

1. A hash map from index to current number.
2. A hash map from number to a min heap of indices.

When we assign a number to an index:

- Update the current value in the index map.
- Push the index into the heap for that number.

When we search for the minimum index of a number:

- Look at the top of that number's heap.
- Some indices in the heap may be stale because the index was later reassigned to another number.
- Repeatedly remove stale indices until the top is valid.

This technique is called lazy deletion.

We never explicitly remove old indices during `change()`, because removing arbitrary elements from a heap is expensive. Instead, we clean them up only when needed during `find()`.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | `O(n)` per `find()` | `O(n)` | Scans all indices for every query |
| Optimal | `O(log n)` amortized | `O(n)` | Uses hash maps and min heaps with lazy deletion |

## Algorithm Walkthrough

1. Create a hash map called `index_to_number`.

This stores the current number assigned to each index.

Example:

```
{
    1: 20,
    2: 10
}
```
2. Create another hash map called `number_to_heap`.

Each number maps to a min heap containing indices that have ever been assigned that number.

Example:

```
{
    10: [2, 3, 5],
    20: [1]
}
```
3. During `change(index, number)`:

- Update `index_to_number[index] = number`
- Push `index` into the heap for `number`

We do not remove the index from its previous number's heap.
4. During `find(number)`:

- If the number has no heap, return `-1`.
- Otherwise, repeatedly inspect the top of the heap.
- Let the smallest candidate index be `idx`.
- Check whether:

```
index_to_number[idx] == number
```
- If true, this index is valid, so return it.
- Otherwise, the index is stale because it was reassigned later. Remove it from the heap and continue.
5. If the heap becomes empty after removing stale entries, return `-1`.

### Why it works

The invariant is:

```
index_to_number always stores the current assignment.
```

The heaps may contain outdated indices, but every valid index for a number is guaranteed to appear in that number's heap.

Since the heap always exposes the smallest candidate index first, and stale indices are removed lazily, the first valid index encountered must be the smallest correct answer.

## Python Solution

```python
import heapq
from collections import defaultdict
from typing import Dict, List

class NumberContainers:

    def __init__(self):
        self.index_to_number: Dict[int, int] = {}
        self.number_to_heap: Dict[int, List[int]] = defaultdict(list)

    def change(self, index: int, number: int) -> None:
        self.index_to_number[index] = number
        heapq.heappush(self.number_to_heap[number], index)

    def find(self, number: int) -> int:
        if number not in self.number_to_heap:
            return -1

        heap = self.number_to_heap[number]

        while heap:
            smallest_index = heap[0]

            if self.index_to_number.get(smallest_index) == number:
                return smallest_index

            heapq.heappop(heap)

        return -1

# Your NumberContainers object will be instantiated and called as such:
# obj = NumberContainers()
# obj.change(index, number)
# param_2 = obj.find(number)
```

The implementation uses two dictionaries.

`index_to_number` stores the current value for every index. This is the authoritative source of truth.

`number_to_heap` maps each number to a min heap of indices. The heap lets us retrieve the smallest candidate index efficiently.

In `change()`, we simply overwrite the current mapping and push the index into the appropriate heap.

In `find()`, we repeatedly check the smallest heap element. If the index still maps to the requested number, we return it immediately. Otherwise, the index is stale and gets removed from the heap.

This lazy deletion strategy avoids expensive arbitrary removals from heaps.

## Go Solution

```go
package main

import (
	"container/heap"
)

type IntHeap []int

func (h IntHeap) Len() int {
	return len(h)
}

func (h IntHeap) Less(i, j int) bool {
	return h[i] < h[j]
}

func (h IntHeap) Swap(i, j int) {
	h[i], h[j] = h[j], h[i]
}

func (h *IntHeap) Push(x interface{}) {
	*h = append(*h, x.(int))
}

func (h *IntHeap) Pop() interface{} {
	old := *h
	n := len(old)
	value := old[n-1]
	*h = old[:n-1]
	return value
}

type NumberContainers struct {
	indexToNumber map[int]int
	numberToHeap  map[int]*IntHeap
}

func Constructor() NumberContainers {
	return NumberContainers{
		indexToNumber: make(map[int]int),
		numberToHeap:  make(map[int]*IntHeap),
	}
}

func (this *NumberContainers) Change(index int, number int) {
	this.indexToNumber[index] = number

	if _, exists := this.numberToHeap[number]; !exists {
		h := &IntHeap{}
		heap.Init(h)
		this.numberToHeap[number] = h
	}

	heap.Push(this.numberToHeap[number], index)
}

func (this *NumberContainers) Find(number int) int {
	h, exists := this.numberToHeap[number]

	if !exists {
		return -1
	}

	for h.Len() > 0 {
		smallestIndex := (*h)[0]

		if this.indexToNumber[smallestIndex] == number {
			return smallestIndex
		}

		heap.Pop(h)
	}

	return -1
}

/**
 * Your NumberContainers object will be instantiated and called as such:
 * obj := Constructor();
 * obj.Change(index,number);
 * param_2 := obj.Find(number);
 */
```

The Go implementation follows the same algorithmic structure as the Python version.

The main difference is that Go does not provide a built in heap implementation for integers, so we implement the `heap.Interface` manually using `container/heap`.

Go maps return zero values for missing keys, so we explicitly check existence with the `exists` boolean when necessary.

### Step by Step Trace

| Operation | index_to_number | number_to_heap | Result |
| --- | --- | --- | --- |
| initialize | `{}` | `{}` |  |
| find(10) | `{}` | `{}` | `-1` |
| change(2, 10) | `{2: 10}` | `{10: [2]}` |  |
| change(1, 10) | `{2: 10, 1: 10}` | `{10: [1, 2]}` |  |
| change(3, 10) | `{2: 10, 1: 10, 3: 10}` | `{10: [1, 2, 3]}` |  |
| change(5, 10) | `{2: 10, 1: 10, 3: 10, 5: 10}` | `{10: [1, 2, 3, 5]}` |  |
| find(10) | unchanged | top of heap is `1` | `1` |
| change(1, 20) | `{2: 10, 1: 20, 3: 10, 5: 10}` | `{10: [1, 2, 3, 5], 20: [1]}` |  |
| find(10) | index `1` stale | remove `1` from heap |  |
| continue find(10) | top now `2` | valid | `2` |

Notice that the old index `1` remained inside the heap for `10`. Lazy deletion removes it only when encountered.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | `O(log n)` amortized per operation | Heap insertions and removals take logarithmic time |
| Space | `O(n)` | We store mappings and heap entries |

The complexity requires a bit of justification because stale indices remain in heaps.

Each stale entry is inserted once and removed once. Therefore, although an individual `find()` call might remove multiple stale entries, the total number of removals across all operations is bounded by the number of insertions.

This gives amortized `O(log n)` complexity.

## Edge Cases

### Reassigning an index to a different number

This is the most important edge case. Suppose index `1` initially stores `10`, then later stores `20`.

The heap for `10` still contains index `1`, which is now stale. A naive implementation might incorrectly return it.

The solution handles this by checking:

```
index_to_number[index] == requested_number
```

before returning a heap element. If the mapping no longer matches, the entry is discarded.

### Reassigning the same number repeatedly

Consider:

```
change(1, 5)
change(1, 5)
change(1, 5)
```

The heap for `5` will contain multiple copies of index `1`.

This does not break correctness because the top entry is still valid. Duplicate entries are harmless. They may slightly increase heap size, but the asymptotic complexity remains acceptable.

### Finding a number that does not exist

If a number was never inserted, or all its indices were reassigned away, `find()` must return `-1`.

The implementation handles both situations:

- If the number has no heap, return `-1`.
- If the heap exists but all entries become stale during cleanup, return `-1`.

This guarantees correct behavior even after extensive updates.
