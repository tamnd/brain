## Solution

## Problem Understanding

This problem asks us to design a binary expression tree that can evaluate arithmetic expressions written in postfix notation, also called Reverse Polish Notation.

In postfix notation, operands appear before operators. For example, the infix expression:

```
(3 + 4) * 2 / 7
```

becomes:

```
["3", "4", "+", "2", "*", "7", "/"]
```

The task is divided into two parts:

1. Build the binary expression tree from the postfix tokens.
2. Implement an `evaluate()` method that recursively computes the value represented by the tree.

The problem already provides an abstract `Node` interface with an `evaluate()` method. We must implement concrete node classes that represent either:

- numbers, which are leaf nodes
- operators, which are internal nodes with exactly two children

The `TreeBuilder.buildTree()` method receives the postfix token list and must return the root node of the expression tree.

The important observation is that postfix notation naturally maps to stack-based parsing. Whenever we encounter a number, we create a leaf node. Whenever we encounter an operator, we pop the two most recent expressions from the stack, combine them into a new operator node, and push the resulting subtree back.

The constraints are small:

- `s.length < 100`
- the expression is guaranteed valid
- no division by zero
- all intermediate values fit within bounds

Because the input size is small, performance is not difficult, but the design aspect matters. The follow-up specifically asks for a modular design that can support additional operators without rewriting the evaluation logic everywhere.

Several edge cases are important:

- A single number expression such as `["42"]`
- Nested operations with multiple operator levels
- Negative intermediate results
- Integer division behavior
- Correct operand ordering for subtraction and division

The operand ordering is especially important because postfix evaluation is order-sensitive. If we pop operands in the wrong order, subtraction and division will produce incorrect results.

## Approaches

### Brute Force Approach

A brute force approach would first reconstruct the full infix expression string from the postfix tokens, including parentheses, and then use a parser or interpreter to evaluate the generated expression.

For example:

```
["3","4","+","2","*"]
```

could become:

```
((3+4)*2)
```

After constructing the string, we could parse and evaluate it recursively.

This works because postfix notation uniquely defines the expression tree structure. However, this approach is unnecessarily complicated because it introduces an extra parsing phase after we already know the expression structure.

It also does not align well with the object-oriented design requirement. The problem specifically wants a tree of `Node` objects with an `evaluate()` method.

### Optimal Approach

The optimal approach directly constructs the expression tree using a stack.

The key insight is that postfix notation already describes the tree construction order:

- operands become leaf nodes
- operators combine the two most recent subexpressions

For every token:

- If the token is a number:

- create a number node
- push it onto the stack
- If the token is an operator:

- pop the right subtree
- pop the left subtree
- create an operator node with those children
- push the new subtree back

At the end, the stack contains exactly one node, the root of the full expression tree.

Evaluation becomes naturally recursive:

- number nodes return their value
- operator nodes recursively evaluate children and apply the operator

This design is modular because each node type encapsulates its own behavior.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Reconstructs expression string and reparses it |
| Optimal | O(n) | O(n) | Builds tree directly using stack |

## Algorithm Walkthrough

1. Create an empty stack that will store expression tree nodes.
2. Iterate through each token in the postfix array.
3. If the token is a number:

- Create a leaf node containing that integer value.
- Push the node onto the stack.

We push operands because future operators may use them as subtrees.
4. If the token is an operator:

- Pop the top node from the stack as the right child.
- Pop the next node as the left child.

The order matters because postfix notation preserves left-to-right operand ordering.
5. Create an operator node containing:

- the operator symbol
- the left subtree
- the right subtree
6. Push the newly created operator node back onto the stack.

This subtree now represents a complete expression that can participate in larger expressions.
7. Continue until all tokens are processed.
8. The stack will contain exactly one node, which is the root of the complete expression tree.
9. Return the root node.
10. To evaluate:

- number nodes return their integer value
- operator nodes recursively evaluate left and right children and apply the operator

### Why it works

The algorithm works because postfix notation guarantees that every operator appears after its operands. Therefore, when an operator is encountered, the stack already contains the complete left and right subtrees needed to build that operator node.

Each stack element always represents a valid expression subtree. By repeatedly combining the most recent two subtrees with an operator, we reconstruct the exact binary expression tree represented by the postfix expression.

## Python Solution

```python
from typing import List
from abc import ABC, abstractmethod

class Node(ABC):
    @abstractmethod
    def evaluate(self) -> int:
        pass

class NumberNode(Node):
    def __init__(self, value: int):
        self.value = value

    def evaluate(self) -> int:
        return self.value

class OperatorNode(Node):
    def __init__(self, operator: str, left: Node, right: Node):
        self.operator = operator
        self.left = left
        self.right = right

    def evaluate(self) -> int:
        left_value = self.left.evaluate()
        right_value = self.right.evaluate()

        if self.operator == "+":
            return left_value + right_value

        if self.operator == "-":
            return left_value - right_value

        if self.operator == "*":
            return left_value * right_value

        return int(left_value / right_value)

class TreeBuilder(object):
    def buildTree(self, postfix: List[str]) -> 'Node':
        stack = []

        operators = {"+", "-", "*", "/"}

        for token in postfix:
            if token not in operators:
                stack.append(NumberNode(int(token)))
            else:
                right = stack.pop()
                left = stack.pop()

                node = OperatorNode(token, left, right)
                stack.append(node)

        return stack[-1]
```

The implementation begins with the abstract `Node` base class. This defines the required `evaluate()` interface and ensures all node types behave consistently.

`NumberNode` represents leaf nodes in the expression tree. Its evaluation is simple because it directly returns its stored integer value.

`OperatorNode` represents internal nodes. During evaluation, it recursively evaluates both children and then applies its stored operator. This recursive structure mirrors the recursive nature of arithmetic expressions themselves.

Inside `TreeBuilder.buildTree()`, we use a stack to construct the tree from postfix notation. Numbers become leaf nodes immediately. Operators combine the two most recent subtrees into a larger subtree.

The division operation uses:

```python
int(left_value / right_value)
```

instead of floor division (`//`) because LeetCode expects truncation toward zero, not toward negative infinity.

## Go Solution

```go
package main

import (
	"strconv"
)

type Node interface {
	evaluate() int
}

type NumberNode struct {
	value int
}

func (n *NumberNode) evaluate() int {
	return n.value
}

type OperatorNode struct {
	operator string
	left     Node
	right    Node
}

func (o *OperatorNode) evaluate() int {
	leftValue := o.left.evaluate()
	rightValue := o.right.evaluate()

	switch o.operator {
	case "+":
		return leftValue + rightValue
	case "-":
		return leftValue - rightValue
	case "*":
		return leftValue * rightValue
	default:
		return leftValue / rightValue
	}
}

type TreeBuilder struct {
}

func (tb *TreeBuilder) buildTree(postfix []string) Node {
	stack := []Node{}
	operators := map[string]bool{
		"+": true,
		"-": true,
		"*": true,
		"/": true,
	}

	for _, token := range postfix {
		if !operators[token] {
			value, _ := strconv.Atoi(token)
			stack = append(stack, &NumberNode{value: value})
		} else {
			right := stack[len(stack)-1]
			stack = stack[:len(stack)-1]

			left := stack[len(stack)-1]
			stack = stack[:len(stack)-1]

			node := &OperatorNode{
				operator: token,
				left:     left,
				right:    right,
			}

			stack = append(stack, node)
		}
	}

	return stack[len(stack)-1]
}
```

The Go solution follows the same structure as the Python version, but uses interfaces and structs instead of abstract base classes.

One important difference is division behavior. Go integer division already truncates toward zero, which matches the problem requirements. Therefore, normal integer division is sufficient.

Slices are used as stacks by appending and removing elements from the end.

## Worked Examples

### Example 1

Input:

```
["3","4","+","2","*","7","/"]
```

### Step-by-step construction

| Token | Action | Stack State |
| --- | --- | --- |
| `"3"` | Push NumberNode(3) | [3] |
| `"4"` | Push NumberNode(4) | [3, 4] |
| `"+"` | Pop 4 and 3, create (3+4) | [(3+4)] |
| `"2"` | Push NumberNode(2) | [(3+4), 2] |
| `"*"` | Pop 2 and (3+4), create ((3+4)*2) | [((3+4)*2)] |
| `"7"` | Push NumberNode(7) | [((3+4)*2), 7] |
| `"/"` | Create (((3+4)*2)/7) | [(((3+4)*2)/7)] |

### Evaluation

```
3 + 4 = 7
7 * 2 = 14
14 / 7 = 2
```

Final answer:

```
2
```

### Example 2

Input:

```
["4","5","2","7","+","-","*"]
```

### Step-by-step construction

| Token | Action | Stack State |
| --- | --- | --- |
| `"4"` | Push 4 | [4] |
| `"5"` | Push 5 | [4, 5] |
| `"2"` | Push 2 | [4, 5, 2] |
| `"7"` | Push 7 | [4, 5, 2, 7] |
| `"+"` | Create (2+7) | [4, 5, (2+7)] |
| `"-"` | Create (5-(2+7)) | [4, (5-(2+7))] |
| `"*"` | Create 4*(5-(2+7)) | [(4*(5-(2+7)))] |

### Evaluation

```
2 + 7 = 9
5 - 9 = -4
4 * -4 = -16
```

Final answer:

```
-16
```

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each token is processed exactly once |
| Space | O(n) | Stack and expression tree both store up to n nodes |

The time complexity is linear because every token causes a constant amount of work. Each node is pushed and popped at most once.

The space complexity is also linear because the stack may temporarily contain many partial subtrees, and the final expression tree itself contains one node per token.

## Edge Cases

### Single Operand Expression

An expression may contain only one number, such as:

```
["42"]
```

A buggy implementation might assume every expression contains operators and attempt to pop operands from the stack. Our implementation handles this naturally because the stack simply ends with one `NumberNode`, which becomes the root.

### Operand Ordering in Subtraction and Division

Subtraction and division are not commutative. The expressions:

```
10 - 3
```

and

```
3 - 10
```

produce different results.

When processing postfix notation, the first popped value is the right operand, not the left operand. Many incorrect implementations reverse this order accidentally.

Our implementation correctly does:

```
right = stack.pop()
left = stack.pop()
```

before constructing the operator node.

### Division Truncation Behavior

Python's `//` operator performs floor division, which differs from truncation toward zero for negative numbers.

For example:

```
-7 // 2 == -4
```

but the problem expects:

```
-3
```

Therefore, we use:

```
int(left_value / right_value)
```

which truncates toward zero and matches the required behavior.
