## Solution

## Problem Understanding

This problem asks us to maximize the total profit earned by selling colored balls whose values decrease over time.

Each color has some number of balls in the inventory. The value of a ball is equal to how many balls of that color currently remain. If a color has `5` balls, then selling one ball from that color earns `5`. After selling it, the remaining count becomes `4`, so the next ball from the same color is worth `4`.

The customer wants exactly `orders` balls, and we may choose any colors in any order. Our goal is to maximize the total revenue.

The important observation is that the highest valued balls should always be sold first. If one color currently has more balls than another, then selling from the larger pile produces more profit at that moment.

The input consists of:

- `inventory[i]`, the number of balls of color `i`
- `orders`, the total number of balls that must be sold

The output is the maximum total profit modulo `10^9 + 7`.

The constraints are very important:

- There can be up to `10^5` colors
- Each inventory count can be as large as `10^9`
- `orders` can also be as large as `10^9`

These limits immediately rule out any simulation that sells one ball at a time. Even an `O(orders)` algorithm would be far too slow in the worst case.

Several edge cases are important:

- Multiple colors may have the same inventory count
- `orders` may require partially reducing a level instead of completely exhausting it
- Large values can overflow standard integer arithmetic unless handled carefully
- The final answer must be returned modulo `10^9 + 7`
- A single color may dominate the inventory

## Approaches

### Brute Force Approach

The most straightforward solution is to always sell the currently most valuable ball.

A max heap can be used for this:

1. Insert all inventory counts into a max heap
2. Repeatedly:

- Extract the largest count
- Add it to the answer
- Decrease it by one
- Push it back if it remains positive
3. Continue until `orders` balls have been sold

This works because greedily taking the largest available value at every step always maximizes profit.

However, this approach is too slow. Since we may need to process up to `10^9` orders, even logarithmic heap operations become infeasible.

### Key Insight for the Optimal Solution

Instead of selling balls one at a time, we can sell entire value ranges in batches.

Suppose we have inventories:

```
[8, 8, 5]
```

The two piles of `8` can both be reduced down to `5` together.

Instead of selling:

```
8, 8, 7, 7, 6, 6
```

one at a time, we compute the sum mathematically.

This transforms the problem into processing inventory levels in descending order.

The arithmetic series formula allows us to compute profits efficiently:

```
sum(a to b) = (a + b) * count / 2
```

By grouping identical heights together and reducing levels in chunks, we avoid iterating over individual balls.

## Algorithm Walkthrough

### Optimal Greedy + Sorting Algorithm

1. Sort the inventory in descending order.

This allows us to process the highest valued balls first. We append a trailing `0` to simplify handling the final level transition.
2. Maintain a variable `colors`.

This represents how many colors currently share the same inventory height.

For example:

```
[8, 8, 5]
```

When processing the first level, `colors = 2` because two piles have height `8`.
3. For each inventory level, compute the difference to the next level.

Suppose:

```
current = 8
next = 5
```

Then the number of balls available in this layer is:

```
(8 - 5) * colors
```
4. Decide whether we can fully consume this layer.

If:

```
orders >= layer_size
```

then we completely reduce all involved colors from `current` down to `next`.
5. Compute the profit using arithmetic progression formulas.

Selling values:

```
8 + 7 + 6
```

has sum:

```
(8 + 6) * 3 / 2
```

Multiply this by `colors` because several piles are processed together.
6. If we cannot fully consume the layer, process only a partial reduction.

Suppose we only have enough orders to reduce from:

```
8 -> 6
```

with some remainder.

We compute:

- Full rows reduced
- Remaining balls at the final value
7. Return the answer modulo `10^9 + 7`.

### Why it works

At every step, the algorithm sells the highest valued balls currently available. Because values decrease monotonically, any optimal solution must prioritize larger values before smaller ones.

Sorting lets us process equal-height groups together, and arithmetic progression formulas correctly compute the total contribution of entire ranges without iterating through individual sales.

Therefore, the greedy batching strategy always produces the maximum possible profit.

## Python Solution

```python
from typing import List

class Solution:
    def maxProfit(self, inventory: List[int], orders: int) -> int:
        MOD = 10**9 + 7
        
        inventory.sort(reverse=True)
        inventory.append(0)
        
        answer = 0
        colors = 1
        
        for i in range(len(inventory) - 1):
            current = inventory[i]
            next_value = inventory[i + 1]
            
            if current > next_value:
                count = colors * (current - next_value)
                
                if orders >= count:
                    total = (
                        (current + next_value + 1)
                        * (current - next_value)
                        // 2
                    )
                    
                    answer += total * colors
                    orders -= count
                else:
                    full_rows = orders // colors
                    remainder = orders % colors
                    
                    low = current - full_rows
                    
                    total = (
                        (current + low + 1)
                        * full_rows
                        // 2
                    )
                    
                    answer += total * colors
                    answer += remainder * low
                    
                    return answer % MOD
            
            colors += 1
        
        return answer % MOD
```

The implementation begins by sorting the inventory in descending order so that the largest values are processed first. A trailing zero is appended to avoid special handling for the final group.

The variable `colors` tracks how many inventory piles are currently participating in the active level reduction. As we move through the sorted array, more colors become part of the group.

For each level transition, the code computes how many balls exist between the current height and the next height. If enough orders remain, the entire range is sold using an arithmetic progression formula.

If only part of the layer can be sold, the algorithm calculates how many complete rows can be removed and how many additional balls remain. This avoids any per-ball simulation.

Finally, the answer is returned modulo `10^9 + 7`.

## Go Solution

```go
package main

import (
	"sort"
)

func maxProfit(inventory []int, orders int) int {
	const MOD int64 = 1e9 + 7

	sort.Sort(sort.Reverse(sort.IntSlice(inventory)))
	inventory = append(inventory, 0)

	var answer int64 = 0
	colors := int64(1)

	for i := 0; i < len(inventory)-1; i++ {
		current := int64(inventory[i])
		nextValue := int64(inventory[i+1])

		if current > nextValue {
			count := colors * (current - nextValue)

			if int64(orders) >= count {
				total := (current + nextValue + 1) * (current - nextValue) / 2

				answer += total * colors
				orders -= int(count)
			} else {
				fullRows := int64(orders) / colors
				remainder := int64(orders) % colors

				low := current - fullRows

				total := (current + low + 1) * fullRows / 2

				answer += total * colors
				answer += remainder * low

				return int(answer % MOD)
			}
		}

		colors++
	}

	return int(answer % MOD)
}
```

The Go implementation follows the same logic as the Python version, but uses `int64` extensively to avoid integer overflow. Since inventory values and order counts can reach `10^9`, intermediate arithmetic progression calculations may exceed 32 bit integer limits.

Go slices are dynamically extended with `append`, which is used to add the trailing zero sentinel value.

Unlike Python, Go requires explicit integer conversions between `int` and `int64`.

## Edge Cases

One important edge case occurs when all inventory counts are identical. In this situation, several colors are reduced simultaneously from the start. A buggy implementation may incorrectly process each pile independently and double count or miss profits. This solution handles the case naturally through the `colors` variable, which tracks how many piles share the same level.

Another critical edge case is when orders stop in the middle of a level reduction. For example, suppose several piles are being reduced together, but there are not enough remaining orders to fully reach the next inventory level. The implementation carefully computes how many complete rows can be removed and how many extra balls remain afterward. This avoids overselling and ensures the maximum profit is preserved.

Large integer arithmetic is also a common source of bugs. Arithmetic progression formulas can produce values far larger than 32 bit integer limits. The Go implementation uses `int64`, and the Python version relies on Python's arbitrary precision integers. Both implementations also apply modulo arithmetic before returning the final answer.

A final edge case involves a single very large inventory pile. A naive simulation would require billions of iterations. This solution never processes individual balls and instead computes entire ranges mathematically, allowing it to handle even the largest constraints efficiently.
