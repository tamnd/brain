## Solution

## Problem Understanding

The problem asks us to compute the average value of an array after removing the smallest `5%` and the largest `5%` of its elements.

More specifically, we are given an integer array `arr`. The array length is guaranteed to be a multiple of `20`, which is important because `5%` of the array size will always be an integer number of elements.

The task can be broken into three operations:

1. Sort the array so the smallest and largest elements become easy to identify.
2. Remove the lowest `5%` of elements and the highest `5%` of elements.
3. Compute the arithmetic mean of the remaining numbers.

For example, if the array length is `20`, then `5%` of `20` is `1`, so we remove exactly one smallest element and one largest element. If the array length is `40`, then `5%` is `2`, so we remove the two smallest and two largest elements.

The output should be a floating point number representing the mean of the remaining elements. The problem allows a small floating point error tolerance of `10^-5`, which means standard floating point arithmetic is sufficient.

The constraints are relatively small:

- `20 <= arr.length <= 1000`
- `arr.length` is a multiple of `20`
- `0 <= arr[i] <= 10^5`

These constraints tell us that efficiency is not a major concern. Even an `O(n log n)` sorting solution is easily fast enough for arrays of size at most `1000`.

Several edge cases are important:

- Arrays may contain many duplicate values.
- The smallest and largest values may appear multiple times.
- All remaining values after trimming may become identical.
- The array size is always valid for removing `5%` from both ends, because the length is guaranteed to be divisible by `20`.

The problem guarantees that after removing elements, at least some elements remain, so we never divide by zero.

## Approaches

### Brute Force Approach

A brute force solution would repeatedly search for the smallest and largest remaining values, remove them one by one, and finally compute the average of the leftover elements.

For an array of size `n`, we would need to remove `n / 20` smallest elements and `n / 20` largest elements. If each removal requires scanning the array to locate the minimum or maximum value, the repeated searches become inefficient.

Although this approach would still work for the given constraints, it is unnecessarily complicated and slower than needed.

### Optimal Approach

The key observation is that sorting the array immediately places all elements in order. Once sorted:

- The first `5%` of elements are the smallest values.
- The last `5%` of elements are the largest values.
- Everything in the middle should be included in the average.

After sorting, we simply compute:

- `remove_count = len(arr) // 20`
- Sum all elements from index `remove_count` to index `n - remove_count - 1`
- Divide by the number of remaining elements

This approach is simple, clean, and efficient.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Repeatedly searches and removes min/max elements |
| Optimal | O(n log n) | O(1) or O(n) depending on sorting implementation | Sort once, then average the middle section |

## Algorithm Walkthrough

1. Determine how many elements must be removed from each side.

Since the problem asks us to remove `5%` of the elements from both the beginning and the end, we compute:

```
remove_count = len(arr) // 20
```

The array length is guaranteed to be divisible by `20`, so this always produces an integer.
2. Sort the array.

Sorting arranges all values in ascending order. After sorting:

- The smallest values appear first.
- The largest values appear last.
- The remaining valid values are grouped in the middle.
3. Identify the valid range.

After removing the smallest and largest `5%`, the remaining elements lie between:

```
arr[remove_count : len(arr) - remove_count]
```
4. Compute the sum of the remaining elements.

We add together all numbers in the valid middle section.
5. Compute the mean.

Divide the sum by the number of remaining elements:

```
remaining_count = len(arr) - 2 * remove_count
answer = remaining_sum / remaining_count
```
6. Return the result as a floating point number.

### Why it works

Sorting guarantees that all smallest values are grouped at the front and all largest values are grouped at the back. By removing exactly `5%` from each side, we exclude precisely the required elements. The remaining subarray contains every value that should contribute to the mean, so averaging that subarray produces the correct answer.

## Python Solution

```python
from typing import List

class Solution:
    def trimMean(self, arr: List[int]) -> float:
        arr.sort()

        remove_count = len(arr) // 20

        remaining_elements = arr[remove_count:len(arr) - remove_count]

        return sum(remaining_elements) / len(remaining_elements)
```

The implementation begins by sorting the array in ascending order. This ensures the smallest values are positioned at the start and the largest values at the end.

Next, we compute how many elements must be removed from each side. Because the array size is always divisible by `20`, integer division correctly calculates `5%`.

The slice:

```python
arr[remove_count:len(arr) - remove_count]
```

extracts only the middle portion of the sorted array, excluding the trimmed elements.

Finally, we compute the sum of the remaining values and divide by the number of remaining elements to produce the mean.

## Go Solution

```go
package main

import "sort"

func trimMean(arr []int) float64 {
	sort.Ints(arr)

	removeCount := len(arr) / 20

	sum := 0

	for i := removeCount; i < len(arr)-removeCount; i++ {
		sum += arr[i]
	}

	remainingCount := len(arr) - 2*removeCount

	return float64(sum) / float64(remainingCount)
}
```

The Go solution follows the same logic as the Python version. The main difference is that Go does not provide built in slicing aggregation functions like `sum`, so we manually iterate through the remaining elements and accumulate the total.

The final division explicitly converts integers to `float64` to avoid integer division.

## Edge Cases

One important edge case occurs when the array contains many duplicate values at the boundaries. For example, several copies of the smallest or largest number may exist. A buggy implementation might incorrectly remove all duplicates instead of removing exactly `5%` of the elements. This solution avoids that issue by removing elements strictly by index after sorting.

Another important edge case is the minimum valid array size of `20`. Since `5%` of `20` equals `1`, we remove exactly one element from each side. Some implementations may accidentally remove too many elements due to incorrect percentage calculations or rounding. Using integer division with the guaranteed constraints ensures the correct count is always computed.

A third edge case is when all elements are identical. In this scenario, trimming the smallest and largest elements changes nothing conceptually because every value is the same. The implementation still works correctly because sorting preserves the identical values, and averaging the remaining elements yields the same number.

A fourth edge case involves very large numbers near the upper constraint of `10^5`. Summation could theoretically overflow in some languages if inappropriate data types were used. Python handles arbitrary size integers automatically, while the Go solution safely uses `int` because the maximum possible total remains well within 64 bit integer limits.
