## Solution

## Problem Understanding

The problem asks us to **reroot a binary tree** so that a given `leaf` node becomes the new root. Unlike a standard tree inversion or traversal problem, we are not simply rearranging left and right children arbitrarily. Instead, we must carefully reverse the parent-child relationships along the path from the given `leaf` up to the original `root`.

Each node contains a `parent` pointer in addition to `left` and `right` child pointers. This is important because the rerooting operation depends on traversing upward through the tree.

The transformation follows very specific rules for every node `cur` on the path from the `leaf` to the `root`, excluding the original root:

1. If `cur` already has a left child, that child must become `cur.right`.
2. The original parent of `cur` becomes `cur.left`.
3. The original parent's reference to `cur` must be removed by setting the corresponding child pointer to `null`.

The challenge is that modifying pointers while traversing the tree can easily corrupt the structure if done in the wrong order. Since parent pointers must also remain correct, every reassignment must update both child references and `parent` links consistently.

The input consists of:

- `root`, the original root of the binary tree.
- `leaf`, a node guaranteed to exist in the tree and guaranteed to be a leaf node.

The output should be the new root after rerooting, which will always be the provided `leaf`.

The constraints are relatively small:

- The tree contains between `2` and `100` nodes.
- Node values are unique.
- The `leaf` always exists.

Although the input size is small enough for many approaches to pass, the real difficulty lies in correctly maintaining tree structure and parent pointers.

Several edge cases are important to consider:

- The `leaf` may already be a direct child of the root, meaning only one reversal is needed.
- The path from `leaf` to `root` may contain nodes with existing left children, which must be shifted to the right side before reassignment.
- Parent pointers must be updated correctly, otherwise the solution will fail even if the visible tree structure appears correct.
- The original root must end with `parent = None`.

## Approaches

### Brute Force Approach

A brute force way to solve this problem is to first reconstruct the entire tree as an undirected graph using parent-child connections. After that, we could perform a DFS or BFS starting from the `leaf` and rebuild a completely new tree rooted at that node.

The process would work like this:

1. Traverse the original tree and create adjacency relationships between each node and its neighbors.
2. Start DFS from `leaf`.
3. Rebuild the tree while preventing cycles using a visited set.
4. Reassign left/right children according to traversal order.
5. Fix parent pointers afterward.

This approach works because rerooting is essentially reversing the path between the old root and the new root while preserving connectivity.

However, this solution is unnecessarily complicated and inefficient. It rebuilds the entire tree despite the fact that only one root-to-leaf path changes. It also introduces additional memory overhead for graph construction and visited tracking.

### Optimal Approach

The key insight is that **only the nodes on the path from `leaf` to `root` need modification**.

Since every node already contains a `parent` pointer, we can simply walk upward from the `leaf` and reverse the relationships in place.

At each node:

- Preserve the next parent before making modifications.
- Move the current left child to the right if necessary.
- Make the original parent the new left child.
- Remove the old child reference from the parent.
- Update parent pointers.

Because each node on the path is processed exactly once, the algorithm is both efficient and elegant.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Rebuilds the entire tree using graph traversal |
| Optimal | O(h) | O(1) | Only modifies the path from leaf to root |

Here, `n` is the number of nodes and `h` is the height of the tree.

## Algorithm Walkthrough

### Optimal In-Place Rerooting

1. Start from the `leaf` node and initialize a variable `current` to point to it.

We process nodes upward because parent pointers already exist. The `leaf` will eventually become the new root.
2. Store the current node's parent before making modifications.

Since we will overwrite child relationships, we must save the original parent reference first.
3. If the current node has a left child, move it to the right child position.

According to the problem statement, any existing left child must become the right child before assigning the original parent as the new left child.
4. Assign the original parent as the left child of the current node.

This reverses the direction of the edge between the current node and its parent.
5. Remove the original parent's pointer to the current node.

Since the edge direction is changing, the parent must stop referencing the current node.

If `parent.left == current`, set `parent.left = None`.

If `parent.right == current`, set `parent.right = None`.
6. Update parent pointers.

The original parent becomes a child of the current node, so its `parent` pointer must be updated accordingly.
7. Move upward.

Continue processing with the saved parent node.
8. Finish when the original root is reached.

The original root becomes a descendant, and the new root (`leaf`) must have `parent = None`.

### Why it works

The algorithm works because every edge on the path from `leaf` to `root` is reversed exactly once. At every step, we preserve the tree structure by first saving references before mutation. The invariant is that after processing a node, the subtree rooted at that node already has the correct rerooted structure. Since we eventually process every node on the root path, the entire tree becomes correctly rerooted with the `leaf` as the new root.

## Python Solution

```python
"""
# Definition for a Node.
class Node:
    def __init__(self, val):
        self.val = val
        self.left = None
        self.right = None
        self.parent = None
"""

from typing import Optional

class Solution:
    def flipBinaryTree(self, root: 'Node', leaf: 'Node') -> 'Node':
        current = leaf
        previous = None

        while current:
            parent = current.parent

            # Move left child to right if needed
            if current.left:
                current.right = current.left

            # Reverse parent-child relationship
            current.left = parent

            # Remove old reference from parent
            if parent:
                if parent.left == current:
                    parent.left = None
                elif parent.right == current:
                    parent.right = None

                parent.parent = current

            # Update current node's parent
            current.parent = previous

            previous = current
            current = parent

        leaf.parent = None
        return leaf
```

### Implementation Explanation

The algorithm begins at the `leaf` node and iteratively walks upward using parent pointers.

The variable `current` tracks the node currently being processed. The variable `previous` represents the node that should become the parent after reversal.

Inside the loop, we first save `current.parent` into `parent`. This prevents losing access to the upward path after pointer modifications.

If the current node has a left child, we move it to the right side. This directly follows the problem requirement that an existing left child becomes the right child.

We then assign the original parent to `current.left`, reversing the relationship.

The parent's old pointer to the current node is removed to prevent cycles. Since the current node could be either a left or right child, both cases are checked.

The parent's `parent` pointer is updated because the relationship direction has changed.

Finally, we set `current.parent = previous`, ensuring parent links remain correct.

Once traversal completes, the `leaf` becomes the new root and its parent pointer is explicitly set to `None`.

## Go Solution

```go
/**
 * Definition for a Node.
 * type Node struct {
 *     Val int
 *     Left *Node
 *     Right *Node
 *     Parent *Node
 * }
 */

func flipBinaryTree(root *Node, leaf *Node) *Node {
	current := leaf
	var previous *Node

	for current != nil {
		parent := current.Parent

		// Move left child to right if needed
		if current.Left != nil {
			current.Right = current.Left
		}

		// Reverse relationship
		current.Left = parent

		// Remove old reference from parent
		if parent != nil {
			if parent.Left == current {
				parent.Left = nil
			} else if parent.Right == current {
				parent.Right = nil
			}

			parent.Parent = current
		}

		current.Parent = previous

		previous = current
		current = parent
	}

	leaf.Parent = nil
	return leaf
}
```

### Go-Specific Implementation Differences

The Go implementation closely mirrors the Python solution. The main difference is explicit use of `nil` instead of `None`.

Pointers are naturally represented with `*Node`, making parent-child reassignment straightforward.

No integer overflow concerns exist because the algorithm only manipulates pointers and never performs arithmetic on node values.

Go requires explicit declaration of `previous` as `var previous *Node`, which initializes it to `nil`.

## Edge Cases

### Leaf Is Directly Connected to the Root

When the target `leaf` is an immediate child of the root, the algorithm performs only one reversal. This case is easy to mishandle because loop termination conditions can accidentally skip processing the root relationship. The implementation handles this naturally because the upward traversal continues until `current` becomes `None`.

### Existing Left Child Must Be Preserved

A node on the rerooting path may already have a left child. According to the rules, this child must move to the right side before the parent becomes the new left child. A naive implementation may overwrite the left child and lose part of the tree. The solution explicitly moves `current.left` to `current.right` before modifying the left pointer.

### Parent Pointer Consistency

The most common source of wrong answers is forgetting to update `parent` pointers. Even if the visible tree structure looks correct, stale parent references will fail hidden test cases. The implementation carefully updates `parent.parent = current` after every reversal and explicitly sets the final root's parent to `None`.

### Avoiding Cycles

When reversing edges, the old parent must stop referencing the child. Otherwise, cycles appear in the tree structure, causing infinite traversals or invalid trees. The implementation removes the parent's old child pointer immediately after reassignment, ensuring the tree remains acyclic throughout execution.
