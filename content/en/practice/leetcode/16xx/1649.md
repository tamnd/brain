**Difficulty:** 🔴 Hard  
**Topics:** Array, Binary Search, Divide and Conquer, Binary Indexed Tree, Segment Tree, Merge Sort, Ordered Set

## Solution

## Problem Understanding

The problem asks us to process the `instructions` array from left to right while maintaining a dynamically growing sorted container called `nums`.

For every value `instructions[i]`, we must determine two quantities before inserting it into `nums`:

1. How many existing elements in `nums` are strictly smaller than `instructions[i]`
2. How many existing elements in `nums` are strictly greater than `instructions[i]`

The insertion cost is the smaller of those two counts.

After computing the cost, we insert the current value into the sorted structure and continue processing the next instruction.

The final answer is the sum of all insertion costs modulo `10^9 + 7`.

The important detail is that the array evolves over time. Each insertion changes the future state of `nums`, so we cannot process elements independently.

For example, suppose the current sorted container is:

```
nums = [1,2,3,5]
```

If we insert `3`:

- Strictly smaller elements: `1,2` → count = 2
- Strictly greater elements: `5` → count = 1

So the insertion cost is:

```
min(2,1) = 1
```

The constraints are large:

```
1 <= instructions.length <= 100000
1 <= instructions[i] <= 100000
```

A solution that repeatedly inserts into a list and scans linearly would be far too slow. Since `n` can reach `100000`, we need approximately `O(n log n)` complexity.

Several edge cases are important:

- Repeated values, because equal values are neither strictly smaller nor strictly greater
- Strictly increasing arrays, where every insertion cost becomes `0`
- Strictly decreasing arrays, where every insertion cost also becomes `0`
- Large numbers of duplicates intermixed with distinct values
- Maximum input size, which rules out quadratic solutions

## Approaches

### Brute Force Approach

The most direct solution is to explicitly maintain the sorted array `nums`.

For each instruction:

1. Find the insertion position
2. Count how many elements are smaller
3. Count how many elements are greater
4. Insert the element into the correct position

We could use binary search to find the insertion point, but insertion into a Python list or Go slice still costs `O(n)` because elements must be shifted.

Therefore:

- Binary search costs `O(log n)`
- Insertion costs `O(n)`

Doing this for all `n` elements leads to `O(n^2)` time complexity in the worst case.

With `100000` elements, this approach is too slow.

### Optimal Approach, Binary Indexed Tree (Fenwick Tree)

The key observation is that we do not actually need to store the entire sorted array explicitly.

For each value `x`, we only need:

- How many previously inserted numbers are less than `x`
- How many previously inserted numbers are greater than `x`

This is fundamentally a frequency counting problem.

Since the values are bounded:

```
1 <= instructions[i] <= 100000
```

we can maintain frequencies using a Binary Indexed Tree, also called a Fenwick Tree.

A Fenwick Tree efficiently supports:

- Updating the frequency of a value
- Querying prefix sums

Both operations run in `O(log M)` time, where `M` is the maximum value.

For each number `x`:

- Numbers smaller than `x`:

```
prefixSum(x - 1)
```
- Numbers greater than `x`:

```
insertedCount - prefixSum(x)
```

Then:

```
cost += min(smaller, greater)
```

Finally, we update the tree to record the insertion of `x`.

This reduces the total complexity to `O(n log M)`.

Here, `M` is the maximum value in `instructions`, at most `100000`.

## Algorithm Walkthrough

### Step 1: Initialize the Fenwick Tree

We create a Fenwick Tree large enough to store frequencies for all possible values.

Each index stores how many times that number has already been inserted.

### Step 2: Process Instructions Left to Right

We iterate through the `instructions` array in order because the problem defines insertion sequentially.

At every step, the tree represents all previously inserted numbers.

### Step 3: Count Smaller Elements

For the current value `x`, we query:

```
query(x - 1)
```

This gives the number of inserted elements strictly smaller than `x`.

The Fenwick Tree stores prefix sums, so this operation is efficient.

### Step 4: Count Greater Elements

We also compute:

```
inserted_so_far - query(x)
```

`query(x)` counts all elements less than or equal to `x`.

Subtracting from the total inserted count leaves only strictly greater elements.

### Step 5: Add the Cost

The insertion cost is:

```
min(smaller, greater)
```

We add this to the running answer modulo `10^9 + 7`.

### Step 6: Insert the Current Value

We update the Fenwick Tree:

```
update(x, 1)
```

This records that value `x` has now appeared once more.

### Step 7: Continue Until All Elements Are Processed

After all insertions, the accumulated cost is returned.

### Why it works

The Fenwick Tree always stores the exact frequency distribution of previously inserted numbers.

For every value `x`:

- `query(x - 1)` correctly counts strictly smaller elements
- `query(x)` correctly counts elements less than or equal to `x`
- Subtracting from the total inserted count correctly isolates strictly greater elements

Because we update the tree immediately after each insertion, the data structure always reflects the current state of `nums`. Therefore every insertion cost is computed correctly.

## Python Solution

```python
from typing import List

class FenwickTree:
    def __init__(self, size: int):
        self.size = size
        self.tree = [0] * (size + 1)

    def update(self, index: int, delta: int) -> None:
        while index <= self.size:
            self.tree[index] += delta
            index += index & -index

    def query(self, index: int) -> int:
        total = 0
        while index > 0:
            total += self.tree[index]
            index -= index & -index
        return total

class Solution:
    def createSortedArray(self, instructions: List[int]) -> int:
        MOD = 10**9 + 7
        max_value = max(instructions)

        fenwick = FenwickTree(max_value)

        total_cost = 0

        for i, value in enumerate(instructions):
            smaller = fenwick.query(value - 1)
            greater = i - fenwick.query(value)

            total_cost += min(smaller, greater)
            total_cost %= MOD

            fenwick.update(value, 1)

        return total_cost
```

The implementation begins by defining a Fenwick Tree class. The tree internally stores cumulative frequencies.

The `update` method increases the frequency of a value. The expression:

```python
index += index & -index
```

moves upward through the Fenwick Tree structure.

The `query` method computes a prefix sum from `1` through `index`. This allows us to quickly count how many inserted numbers fall within a range.

Inside `createSortedArray`, we first determine the maximum value so the tree can be sized appropriately.

For every instruction:

- `smaller` counts values strictly less than the current number
- `greater` counts values strictly greater than the current number
- We add the minimum of those counts to the answer
- We insert the current number into the tree

The modulo operation is applied during accumulation to avoid overflow.

## Go Solution

```go
package main

func createSortedArray(instructions []int) int {
	const MOD = 1000000007

	maxValue := 0
	for _, value := range instructions {
		if value > maxValue {
			maxValue = value
		}
	}

	fenwick := make([]int, maxValue+1)

	update := func(index int, delta int) {
		for index <= maxValue {
			fenwick[index] += delta
			index += index & -index
		}
	}

	query := func(index int) int {
		sum := 0
		for index > 0 {
			sum += fenwick[index]
			index -= index & -index
		}
		return sum
	}

	totalCost := 0

	for i, value := range instructions {
		smaller := query(value - 1)
		greater := i - query(value)

		if smaller < greater {
			totalCost = (totalCost + smaller) % MOD
		} else {
			totalCost = (totalCost + greater) % MOD
		}

		update(value, 1)
	}

	return totalCost
}
```

The Go implementation follows the same logic as the Python version.

Instead of defining a separate class, the Fenwick Tree is implemented directly using a slice and closure functions.

Go uses explicit integer arithmetic, so we apply modulo operations during accumulation to keep values bounded.

Unlike Python lists, Go slices require explicit sizing up front, so we allocate the Fenwick Tree using the maximum instruction value.

## Edge Cases

One important edge case is when all values are identical. Since the problem asks for strictly smaller and strictly greater counts, equal values should never contribute to the insertion cost. A buggy implementation might accidentally include equal elements in one of the counts. The Fenwick Tree solution avoids this by carefully using `query(value - 1)` for smaller elements and `query(value)` when computing greater elements.

Another important case is a strictly increasing sequence. Every new element becomes the largest element seen so far, so there are no greater elements during insertion. The correct cost for every insertion is therefore zero. The algorithm handles this naturally because `greater = inserted_count - query(value)` becomes zero.

A third important case is a strictly decreasing sequence. Every new element becomes the smallest element seen so far, so there are no smaller elements. Again, the insertion cost should remain zero throughout the process. The Fenwick Tree correctly computes this because `query(value - 1)` becomes zero for each insertion.

Large inputs are also critical. With up to `100000` instructions, any approach using linear insertion or scanning would time out. The Fenwick Tree guarantees logarithmic operations, allowing the algorithm to scale efficiently even at the upper constraint limits.
