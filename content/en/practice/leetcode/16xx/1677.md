# LeetCode 1677 - Product's Worth Over Invoices

## Problem Understanding

The problem provides two database tables, `Product` and `Invoice`.

The `Product` table stores information about products. Each row contains a unique `product_id` and the product `name`.

The `Invoice` table stores invoice information associated with products. Every invoice belongs to exactly one product through the `product_id` column. Each invoice tracks four monetary fields:

- `rest`, the remaining amount still unpaid
- `paid`, the amount already paid
- `canceled`, the amount canceled
- `refunded`, the amount refunded

The goal is to compute aggregated totals for every product. For each product, we must calculate:

- the total `rest`
- the total `paid`
- the total `canceled`
- the total `refunded`

across all invoices associated with that product.

The output should contain one row per product and must be ordered alphabetically by the product name.

This is fundamentally a database aggregation problem. We need to combine rows from two tables using the shared `product_id`, then group all invoices belonging to the same product, and finally sum the monetary columns.

The problem guarantees that:

- `product_id` is unique in `Product`
- `invoice_id` is unique in `Invoice`
- product names are unique
- every invoice references a product

Because the task is purely aggregation-based, the solution naturally fits SQL operations such as:

- `JOIN`
- `GROUP BY`
- aggregate functions like `SUM()`

An important edge case is products with multiple invoices, because we must correctly accumulate values across all matching rows. Another important consideration is products with only one invoice, where aggregation still must work correctly. Depending on SQL dialect behavior, products with no invoices could also matter, although the problem examples imply invoices exist for referenced products.

## Approaches

### Brute Force Approach

A brute-force approach would manually process each product one at a time and scan the entire `Invoice` table for matching invoices.

For every product:

1. Initialize totals for `rest`, `paid`, `canceled`, and `refunded`
2. Iterate through every invoice
3. If the invoice belongs to the current product, add its values to the running totals
4. Store the final totals for output

This approach works because every invoice is checked against every product, guaranteeing that all matching invoices are included.

However, this method is inefficient because it repeatedly scans the entire invoice table for every product. If there are `P` products and `I` invoices, the total work becomes `O(P × I)`.

### Optimal Approach

The better solution uses SQL aggregation.

The key insight is that SQL databases are optimized for grouping and aggregation operations. Instead of repeatedly scanning invoices for every product, we can:

1. Join `Product` and `Invoice` using `product_id`
2. Group rows by product name
3. Use `SUM()` to aggregate each monetary column

This processes the data in a single grouped operation, which is significantly more efficient and cleaner.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(P × I) | O(1) extra | Scans all invoices for every product |
| Optimal | O(I + P) | O(P) | Uses SQL JOIN and GROUP BY aggregation |

## Algorithm Walkthrough

1. Start with the `Product` table because we need one output row per product.
2. Join the `Product` table with the `Invoice` table using the shared `product_id` column. This associates every invoice with its corresponding product name.
3. Group the resulting rows by product name. Grouping ensures that all invoices belonging to the same product are collected together.
4. For each group, calculate:

- `SUM(rest)`
- `SUM(paid)`
- `SUM(canceled)`
- `SUM(refunded)`

These aggregate functions compute the total values across all invoices for that product.
5. Select the product name and aggregated totals as the final output columns.
6. Order the results alphabetically by product name using `ORDER BY name`.

### Why it works

The solution works because every invoice row is joined with exactly one product through `product_id`. Grouping by product name guarantees that all invoices belonging to the same product are processed together. Applying `SUM()` to each monetary column correctly accumulates the totals required by the problem statement.

## Python Solution

Although this is a SQL problem on LeetCode, the platform still expects the query implementation. The following is the complete SQL solution written inside a Python-style code block for consistency with the requested format.

```python
SELECT
    p.name,
    SUM(i.rest) AS rest,
    SUM(i.paid) AS paid,
    SUM(i.canceled) AS canceled,
    SUM(i.refunded) AS refunded
FROM Product p
JOIN Invoice i
    ON p.product_id = i.product_id
GROUP BY p.name
ORDER BY p.name;
```

The query begins by joining `Product` and `Invoice` using `product_id`. This allows access to both the product name and invoice values in the same result set.

Next, the query groups rows by `p.name`. Each group represents all invoices for a single product.

The `SUM()` functions aggregate each monetary field across all invoices in the group.

Finally, `ORDER BY p.name` sorts the output alphabetically as required.

## Go Solution

LeetCode database problems are solved using SQL rather than Go code. The equivalent SQL solution is shown below in a Go-tagged block to satisfy the requested structure.

```go
SELECT
    p.name,
    SUM(i.rest) AS rest,
    SUM(i.paid) AS paid,
    SUM(i.canceled) AS canceled,
    SUM(i.refunded) AS refunded
FROM Product p
JOIN Invoice i
    ON p.product_id = i.product_id
GROUP BY p.name
ORDER BY p.name;
```

There are no Go-specific implementation differences because this problem is solved entirely with SQL. The database engine handles grouping, aggregation, and sorting internally.

## Edge Cases

### Products With Multiple Invoices

A common source of bugs is failing to aggregate across every invoice belonging to the same product. If grouping is omitted or done incorrectly, the query may return multiple rows per product instead of a single summarized row.

This implementation handles the issue correctly by grouping with:

```
GROUP BY p.name
```

which ensures all invoices for the same product are combined into one result row.

### Products With Zero Values

Some invoices may contain zeros in one or more monetary fields. Incorrect implementations sometimes ignore these rows or accidentally treat them as missing values.

Using `SUM()` correctly includes zero values in the aggregation, so totals remain accurate.

### Alphabetical Output Ordering

Without explicit sorting, SQL databases may return rows in arbitrary order. Since the problem requires ordering by product name, forgetting the `ORDER BY` clause would produce incorrect results even if aggregation is correct.

This solution explicitly uses:

```
ORDER BY p.name
```

to guarantee deterministic alphabetical ordering.
