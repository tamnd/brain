## Solution

## Problem Understanding

This problem asks us to design a dynamic inheritance system for a royal family. The family starts with a king, and over time people are born or die. We must maintain the inheritance order according to a specific recursive rule.

The inheritance order behaves exactly like a preorder traversal of a family tree:

1. Visit a person first.
2. Then visit their children from oldest to youngest.
3. Recursively continue for each child.

The important detail is that dead people are skipped only when generating the final inheritance list. Their position in the family tree still matters because their descendants must remain in the correct order.

For example, if Bob dies but Bob has children, those children still inherit immediately after where Bob would have appeared.

The operations are:

- `birth(parentName, childName)` adds a new child to a parent.
- `death(name)` marks a person as dead.
- `getInheritanceOrder()` returns the current inheritance order excluding dead people.

The constraints are very important:

- Up to `10^5` birth and death operations.
- Only up to `10` calls to `getInheritanceOrder`.

This tells us something critical about optimization strategy. Since inheritance order is requested very infrequently, we do not need to maintain the exact order after every update. Instead, we can store the family structure efficiently and compute the order on demand.

The natural structure here is a tree:

- Each person is a node.
- Children are stored in birth order.
- The king is the root.

Some important edge cases are guaranteed by the problem:

- Every child name is unique.
- Parents always exist before birth operations.
- A dead person may still have descendants.
- The king can die.
- The inheritance order must preserve birth order among siblings.

A naive implementation can easily fail if it physically removes dead people from the tree, because descendants would lose their correct inheritance position.

## Approaches

### Brute Force Approach

A brute force approach would try to maintain the entire inheritance order as a continuously updated list.

Whenever a birth occurs, we would need to determine the exact insertion position of the child in the inheritance list. This is difficult because the child must appear after all descendants of older siblings but before younger siblings. Finding this insertion point may require scanning a large portion of the current order.

Similarly, handling deaths might require removing or marking entries inside the list.

Although this approach can produce the correct result, maintaining the list dynamically becomes expensive because each insertion can take linear time.

With up to `10^5` operations, repeated list insertions become too slow.

### Optimal Approach

The key observation is that the inheritance order is simply a preorder traversal of the family tree.

Instead of maintaining the full order continuously, we only need:

- A mapping from parent to children.
- A set of dead people.
- The king's name.

When `getInheritanceOrder()` is called, we perform a DFS preorder traversal:

1. Visit current person.
2. Add them if alive.
3. Recursively visit children in birth order.

This works perfectly because the recursive successor definition from the problem statement is exactly equivalent to preorder traversal.

Since `getInheritanceOrder()` is called at most 10 times, recomputing the entire order each time is efficient enough.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) per insertion | O(n) | Maintaining exact order dynamically is expensive |
| Optimal | Birth: O(1), Death: O(1), Query: O(n) | O(n) | Store family tree and generate order with DFS |

## Algorithm Walkthrough

### Step 1: Store the Family Tree

We use a hash map where:

- Key = parent name
- Value = list of children in birth order

This preserves the exact sibling ordering required by inheritance rules.

For example:

```
king -> [andy, bob, catherine]
bob  -> [alex, asha]
```

### Step 2: Track Dead People

We use a hash set called `dead`.

When someone dies:

- We do not remove them from the tree.
- We simply add their name to the set.

This is important because their descendants still inherit in the same location.

### Step 3: Store the King

We store the king's name separately because DFS traversal always begins from the king.

### Step 4: Handle Birth Operations

When a child is born:

1. Find the parent's child list.
2. Append the child to the end.

Appending preserves oldest-to-youngest order automatically.

### Step 5: Handle Death Operations

When someone dies:

1. Add their name to the dead set.

No structural modifications are needed.

### Step 6: Generate the Inheritance Order

When `getInheritanceOrder()` is called:

1. Start DFS from the king.
2. During DFS:

- If current person is alive, append to result.
- Recursively process children in stored order.

This traversal exactly matches inheritance rules.

### Step 7: Return the Result

After DFS finishes, return the constructed list.

### Why it works

The inheritance rule described in the problem is identical to preorder traversal of the family tree. A person always appears before their descendants, and siblings are processed from oldest to youngest.

By storing children in birth order and traversing recursively, we guarantee the correct inheritance sequence. Dead people are skipped only in the output, which preserves the inheritance structure for their descendants.

## Python Solution

```python
from collections import defaultdict
from typing import List

class ThroneInheritance:

    def __init__(self, kingName: str):
        self.king = kingName
        self.children = defaultdict(list)
        self.dead = set()

    def birth(self, parentName: str, childName: str) -> None:
        self.children[parentName].append(childName)

    def death(self, name: str) -> None:
        self.dead.add(name)

    def getInheritanceOrder(self) -> List[str]:
        inheritance_order = []

        def dfs(person: str) -> None:
            if person not in self.dead:
                inheritance_order.append(person)

            for child in self.children[person]:
                dfs(child)

        dfs(self.king)

        return inheritance_order

# Your ThroneInheritance object will be instantiated and called as such:
# obj = ThroneInheritance(kingName)
# obj.birth(parentName, childName)
# obj.death(name)
# param_3 = obj.getInheritanceOrder()
```

The implementation closely follows the tree-based design discussed earlier.

The constructor initializes three pieces of state:

- `self.king` stores the root of the family tree.
- `self.children` maps parents to ordered child lists.
- `self.dead` tracks deceased people.

The `birth()` method simply appends the child to the parent's child list. Since births occur chronologically, appending naturally preserves age order.

The `death()` method only marks the person as dead. The tree structure remains unchanged because descendants still inherit through that branch.

The `getInheritanceOrder()` method performs preorder DFS traversal. The helper function first processes the current person, then recursively processes children in birth order.

If a person is dead, they are skipped from the result list, but recursion still continues into their descendants.

## Go Solution

```go
package main

type ThroneInheritance struct {
	king     string
	children map[string][]string
	dead     map[string]bool
}

func Constructor(kingName string) ThroneInheritance {
	return ThroneInheritance{
		king:     kingName,
		children: make(map[string][]string),
		dead:     make(map[string]bool),
	}
}

func (this *ThroneInheritance) Birth(parentName string, childName string) {
	this.children[parentName] = append(this.children[parentName], childName)
}

func (this *ThroneInheritance) Death(name string) {
	this.dead[name] = true
}

func (this *ThroneInheritance) GetInheritanceOrder() []string {
	result := []string{}

	var dfs func(string)

	dfs = func(person string) {
		if !this.dead[person] {
			result = append(result, person)
		}

		for _, child := range this.children[person] {
			dfs(child)
		}
	}

	dfs(this.king)

	return result
}

/**
 * Your ThroneInheritance object will be instantiated and called as such:
 * obj := Constructor(kingName);
 * obj.Birth(parentName, childName);
 * obj.Death(name);
 * param_3 := obj.GetInheritanceOrder();
 */
```

The Go implementation mirrors the Python logic very closely.

The primary difference is that Go requires explicit map initialization using `make`.

The DFS helper is implemented as a recursive closure. Slices are used to store children because slices preserve insertion order naturally.

The dead set is implemented using `map[string]bool`, which is the standard Go pattern for hash sets.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) for `getInheritanceOrder()` | DFS visits every person once |
| Space | O(n) | Tree storage, dead set, and recursion stack |

Birth and death operations are both `O(1)` because they only append to a list or insert into a hash set.

The expensive operation is `getInheritanceOrder()`, which performs DFS over the entire family tree. Since every person is visited exactly once, the complexity is linear in the number of people.

This is efficient because the problem guarantees at most 10 inheritance queries.

## Edge Cases

### Dead Person with Living Descendants

One major source of bugs is incorrectly removing dead people from the tree. If Bob dies and we remove Bob entirely, then Bob's children would disappear from inheritance order as well.

The implementation avoids this by only storing dead names in a set. DFS still traverses through dead individuals to reach their descendants.

### Death of the King

The king can die, but the inheritance system must still work correctly. A naive implementation might assume the root always appears in the answer.

Our DFS simply skips adding dead people to the result while still traversing children. If the king dies, the traversal still begins at the king node and correctly processes all descendants.

### Preserving Birth Order Among Siblings

Inheritance order depends strictly on age order among siblings. A common mistake would be using unordered structures like sets for children.

The implementation stores children in lists and appends births in chronological order. DFS then processes children sequentially, preserving the correct inheritance order exactly.
