## Solution

## Problem Understanding

This problem asks us to generate monthly statistics for the year 2020 using data from three database tables: `Drivers`, `Rides`, and `AcceptedRides`.

The first metric we must compute is `active_drivers`. For every month from January through December 2020, we need to determine how many drivers had joined the company on or before the end of that month. A driver remains active permanently after joining, so this value is cumulative over time.

The second metric is `accepted_rides`. For each month of 2020, we need to count how many rides were successfully accepted during that month. A ride is considered accepted if it appears in the `AcceptedRides` table. Since `AcceptedRides` only stores accepted rides, we must join it with the `Rides` table to determine when the ride was requested, and therefore which month it belongs to.

The output must contain exactly twelve rows, one for each month of 2020, ordered from January to December.

The important detail is that months with no drivers joining or no accepted rides must still appear in the output with a count of `0` where appropriate. This means we cannot simply aggregate existing data, we must explicitly generate all twelve months.

The input tables represent the following:

- `Drivers` stores the date each driver joined Hopper.
- `Rides` stores ride requests and their request dates.
- `AcceptedRides` stores only rides that were accepted by a driver.

The problem guarantees that every accepted ride exists in the `Rides` table, so joining these tables is always valid.

Several edge cases are important:

- Drivers who joined before 2020 must still count toward all months in 2020.
- Drivers who joined after 2020 must not affect any month in 2020.
- Some months may contain no accepted rides at all.
- Some rides in `Rides` may never appear in `AcceptedRides`.
- We must still return all twelve months even if a month has zero activity.

## Approaches

### Brute Force Approach

A straightforward solution would iterate through each month from 1 to 12 and independently compute both required metrics.

For `active_drivers`, we could count all drivers whose `join_date` is less than or equal to the last day of the current month.

For `accepted_rides`, we could join `AcceptedRides` with `Rides`, then count rides whose `requested_at` falls within the current month.

This works because each month is processed independently and all filtering conditions directly match the problem statement.

However, this approach repeatedly scans large tables for every month. If there are many drivers and rides, repeatedly reprocessing the same rows becomes inefficient.

### Optimal Approach

The key observation is that both metrics can be pre-aggregated by month.

For drivers, instead of recomputing counts for every month, we first calculate how many drivers joined in each month. Then we compute a running cumulative sum across months to determine active drivers by the end of each month.

For accepted rides, we can aggregate all accepted rides by request month in a single grouped query.

Finally, we generate all twelve months and left join the aggregated results so that missing months appear with zeros.

This approach avoids repeated scans and transforms the problem into efficient grouped aggregations and cumulative calculations.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(12 × (D + R)) | O(1) | Repeatedly scans tables for every month |
| Optimal | O(D + R) | O(12) | Uses monthly aggregation and cumulative counts |

Here, `D` is the number of drivers and `R` is the number of rides.

## Algorithm Walkthrough

1. Generate a list of all months from 1 through 12.

This guarantees the final result always contains every month, even if no data exists for that month.
2. Aggregate driver joins by month.

We examine the `Drivers` table and count how many drivers joined in each month of 2020. Drivers who joined before 2020 are handled separately because they contribute to every month of 2020.
3. Count drivers who joined before 2020.

These drivers are already active when 2020 begins, so they form the initial cumulative total.
4. Compute cumulative active drivers.

Starting with the number of pre-2020 drivers, we iterate through months 1 to 12 and continuously add drivers who joined in the current month.

This running total gives the number of active drivers by the end of each month.
5. Aggregate accepted rides by month.

We join `AcceptedRides` with `Rides` using `ride_id`. Then we group by the month of `requested_at` and count accepted rides.
6. Combine results.

For each month from 1 through 12, we output:

- The cumulative active driver count
- The accepted ride count for that month, defaulting to zero if none exist
7. Sort by month.

Since we iterate in ascending order from 1 to 12, the result is naturally sorted.

### Why it works

The algorithm works because both required metrics are cumulative or grouped by month.

For active drivers, once a driver joins, they remain active forever. Therefore, the number of active drivers at month `m` equals the number of drivers active before month `m` plus all drivers who joined during month `m`.

For accepted rides, each accepted ride belongs to exactly one request month, so grouping by request month produces the correct monthly totals.

By explicitly generating all months, we also guarantee that missing months appear correctly with zero counts.

## Python Solution

```python
SELECT
    m.month,
    (
        SELECT COUNT(*)
        FROM Drivers d
        WHERE d.join_date <= LAST_DAY(CONCAT('2020-', m.month, '-01'))
    ) AS active_drivers,
    COALESCE(ar.accepted_rides, 0) AS accepted_rides
FROM
(
    SELECT 1 AS month
    UNION SELECT 2
    UNION SELECT 3
    UNION SELECT 4
    UNION SELECT 5
    UNION SELECT 6
    UNION SELECT 7
    UNION SELECT 8
    UNION SELECT 9
    UNION SELECT 10
    UNION SELECT 11
    UNION SELECT 12
) AS m
LEFT JOIN
(
    SELECT
        MONTH(r.requested_at) AS month,
        COUNT(*) AS accepted_rides
    FROM AcceptedRides ar
    JOIN Rides r
        ON ar.ride_id = r.ride_id
    WHERE YEAR(r.requested_at) = 2020
    GROUP BY MONTH(r.requested_at)
) AS ar
ON m.month = ar.month
ORDER BY m.month;
```

The query begins by generating a derived table containing all twelve months. This ensures the final result always contains every month even when no rides or driver joins occur.

The `active_drivers` value is computed using a correlated subquery. For each month, we count all drivers whose `join_date` is less than or equal to the last day of that month. Using `LAST_DAY` correctly handles varying month lengths.

The accepted ride counts are pre-aggregated in a separate subquery. We join `AcceptedRides` with `Rides` to determine the request date of each accepted ride, then group by the request month.

Finally, a `LEFT JOIN` ensures months without accepted rides still appear with a count of zero.

## Edge Cases

One important edge case occurs when there are no accepted rides in a month. A naive inner join would completely omit those months from the result. The implementation avoids this problem by generating all months explicitly and using a `LEFT JOIN` with `COALESCE` to replace missing counts with zero.

Another important case involves drivers who joined before 2020. These drivers must count toward every month in 2020. The correlated subquery handles this naturally because any driver whose join date is before the month's last day is included.

A third edge case occurs when rides exist in the `Rides` table but not in `AcceptedRides`. These rides should not contribute to the accepted ride count. By aggregating only from `AcceptedRides` joined with `Rides`, the query correctly excludes unaccepted rides.

A final subtle case involves months with no data at all. Without explicitly generating months 1 through 12, such months would disappear entirely from the result set. The derived month table guarantees every month appears exactly once.
