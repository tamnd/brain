## Solution

## Problem Understanding

In this problem, we are given two inputs:

- `arr`, a one dimensional array of distinct integers.
- `pieces`, a list of smaller integer arrays, where every integer across all pieces is also distinct.

The goal is to determine whether we can recreate the exact sequence in `arr` by concatenating the arrays inside `pieces` in any order.

The important restriction is that we are allowed to rearrange the order of the pieces themselves, but we are not allowed to rearrange the integers inside any individual piece.

For example, if a piece is `[4, 64]`, then it must appear exactly in that order when forming `arr`. We cannot use it as `[64, 4]`.

The output should be:

- `true` if `arr` can be formed exactly.
- `false` otherwise.

The constraints are very small:

- `arr.length <= 100`
- All integers are distinct.
- The total number of integers across all pieces equals the length of `arr`.

These constraints tell us several important things:

- We do not need advanced optimization techniques.
- Since values are distinct, each number can belong to only one piece.
- We can safely use a hash map for fast lookups.
- We never need to worry about duplicate matching ambiguity.

An important observation is that because every integer is unique, the first value of a piece uniquely identifies where that piece must begin in `arr`.

Some edge cases that can trip up naive implementations include:

- A piece matches partially but not completely.
- The correct numbers exist, but their order inside a piece conflicts with `arr`.
- A value in `arr` does not start any piece.
- Pieces of length one.
- A single large piece that must match the entire array exactly.

The problem guarantees that all integers are distinct, which simplifies matching significantly because we never need to choose between multiple candidate pieces for the same value.

## Approaches

### Brute Force Approach

A brute force strategy would try every possible ordering of the pieces, concatenate them, and compare the result to `arr`.

Suppose there are `k` pieces. We could:

1. Generate all permutations of the pieces.
2. Concatenate each permutation.
3. Compare the resulting array with `arr`.

This works because if any ordering recreates `arr`, we can return `true`.

However, this approach is inefficient because the number of permutations is `k!`. Even with relatively small inputs, factorial growth becomes expensive quickly.

For example, if there are 10 pieces, we may need to check:

$$10! = 3,628,800$$

possible orderings.

Although the constraints here are small enough that brute force might still pass in practice, it is unnecessarily expensive and ignores the important structure of the problem.

### Optimal Approach

The key insight is that every number is unique.

This means:

- When we see a value in `arr`, there can only be one piece that starts with that value.
- Therefore, instead of trying permutations, we can directly determine which piece must appear next.

We create a hash map:

- Key = first element of a piece
- Value = the entire piece

Then we scan `arr` from left to right:

- At each position, look up the current value in the map.
- If no piece starts with that value, return `false`.
- Otherwise, verify that the entire piece matches the next segment of `arr`.

If every piece matches successfully and we reach the end of `arr`, return `true`.

This avoids permutations entirely and processes each element only once.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(k! * n) | O(n) | Tries every ordering of pieces |
| Optimal | O(n) | O(n) | Uses hash map for direct matching |

Here:

- `n` is the length of `arr`
- `k` is the number of pieces

## Algorithm Walkthrough

1. Create a hash map where the key is the first element of each piece and the value is the entire piece.

This works because all integers are distinct, so no two pieces can start with the same value.
2. Initialize an index `i = 0` to traverse `arr`.
3. While `i` is less than the length of `arr`:

- Look at `arr[i]`.
- Use it to find the matching piece in the hash map.
4. If no piece starts with `arr[i]`, immediately return `false`.

This means we cannot continue forming `arr`.
5. Otherwise, retrieve the corresponding piece.
6. Compare every element of the piece with the corresponding elements in `arr`.

For each position `j` in the piece:

- Check whether:

$$arr[i + j] == piece[j]$$
- If any value differs, return `false`.
7. After successfully matching the piece, move `i` forward by the length of the piece.
8. Continue until all elements in `arr` are processed.
9. If the entire traversal succeeds, return `true`.

### Why it works

The algorithm works because each integer is unique across all pieces and in `arr`.

Therefore:

- Every value in `arr` can correspond to at most one piece.
- The first element of a piece uniquely determines where that piece must appear.
- If a piece matches at its starting position, there is no alternative valid arrangement to consider.

By greedily matching the only possible piece at each step, we correctly determine whether `arr` can be formed.

## Python Solution

```python
from typing import List

class Solution:
    def canFormArray(self, arr: List[int], pieces: List[List[int]]) -> bool:
        piece_map = {}

        for piece in pieces:
            piece_map[piece[0]] = piece

        index = 0

        while index < len(arr):
            current_value = arr[index]

            if current_value not in piece_map:
                return False

            piece = piece_map[current_value]

            for offset in range(len(piece)):
                if index + offset >= len(arr):
                    return False

                if arr[index + offset] != piece[offset]:
                    return False

            index += len(piece)

        return True
```

The implementation begins by building a dictionary called `piece_map`. Each entry maps the first element of a piece to the piece itself.

For example:

```python
pieces = [[78], [4, 64], [91]]
```

becomes:

```python
{
    78: [78],
    4: [4, 64],
    91: [91]
}
```

Next, the algorithm scans through `arr` using the variable `index`.

At each step:

- It checks whether the current value starts a valid piece.
- If not, the answer is immediately `False`.

If a matching piece exists, the code validates every element of that piece against the corresponding segment in `arr`.

The loop:

```python
for offset in range(len(piece)):
```

ensures that the piece appears in exactly the required order.

After a successful match, the index advances by the size of the matched piece.

If the entire array is processed without mismatches, the function returns `True`.

## Go Solution

```go
func canFormArray(arr []int, pieces [][]int) bool {
    pieceMap := make(map[int][]int)

    for _, piece := range pieces {
        pieceMap[piece[0]] = piece
    }

    index := 0

    for index < len(arr) {
        currentValue := arr[index]

        piece, exists := pieceMap[currentValue]

        if !exists {
            return false
        }

        for offset := 0; offset < len(piece); offset++ {
            if index+offset >= len(arr) {
                return false
            }

            if arr[index+offset] != piece[offset] {
                return false
            }
        }

        index += len(piece)
    }

    return true
}
```

The Go implementation follows the same logic as the Python version.

A `map[int][]int` is used instead of a Python dictionary. The `exists` boolean from map lookup cleanly handles the case where no piece starts with the current value.

Go slices naturally represent the subarrays in `pieces`, so no additional data structures are required.

Integer overflow is not a concern because the values are very small.

## Edge Cases

### Missing Starting Value

A common bug occurs when the current value in `arr` does not start any piece.

Example:

```
arr = [1,2,3]
pieces = [[2],[3]]
```

Here, no piece begins with `1`.

The implementation handles this safely with:

```
if current_value not in piece_map:
    return False
```

This guarantees we never attempt invalid matching.

### Correct Numbers but Wrong Order

Another tricky case is when all numbers exist, but a piece appears in the wrong order.

Example:

```
arr = [1,2,3]
pieces = [[1,3],[2]]
```

Although the numbers are all present, `[1,3]` does not match the segment `[1,2]` in `arr`.

The element-by-element verification loop correctly detects this mismatch.

### Single Large Piece

Sometimes there is only one piece containing the entire array.

Example:

```
arr = [1,2,3]
pieces = [[1,2,3]]
```

A poorly designed implementation might incorrectly expect multiple pieces or mishandle index advancement.

This solution works correctly because after matching the entire piece, the index advances directly to the end of `arr`.
