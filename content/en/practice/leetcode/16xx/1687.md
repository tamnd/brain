## Solution

## Problem Understanding

This problem asks us to determine the minimum number of ship trips required to deliver a sequence of boxes while respecting two capacity constraints:

- The ship can carry at most `maxBoxes` boxes at once.
- The total weight of the carried boxes cannot exceed `maxWeight`.

Each box is represented as:

```
boxes[i] = [port_i, weight_i]
```

where:

- `port_i` is the destination port of the box
- `weight_i` is the box weight

The boxes must be delivered strictly in the given order. We are not allowed to reorder them.

A single delivery batch works like this:

1. Load a contiguous sequence of boxes from the front of the queue.
2. Sail through ports in the order required by those boxes.
3. Deliver boxes as the ship reaches their ports.
4. Return to storage.

The cost of a delivery batch is:

```
(number of port changes inside the batch) + 2
```

Why `+2`?

- One trip from storage to the first port
- One trip returning to storage

Additionally, every time consecutive boxes require different ports, the ship must travel to another port, which increases the trip count.

The goal is to partition the boxes into valid batches such that the total number of trips is minimized.

The constraints are very large:

- `boxes.length <= 100000`

This immediately tells us that:

- Quadratic dynamic programming is too slow
- We need an approximately linear time solution

Several tricky details make this problem difficult:

- Boxes must remain in order
- A batch must satisfy both count and weight constraints
- The cost of a batch depends on how many port transitions occur inside it
- Choosing locally optimal batches does not always produce a globally optimal answer

For example, taking as many boxes as possible is not always best, because combining boxes with many port switches can increase trip count unnecessarily.

Important edge cases include:

- All boxes going to the same port
- Every box going to a different port
- Very small `maxBoxes`
- Weight limits forcing extremely small batches
- Long stretches of identical ports
- Cases where the optimal partition is not greedy

## Approaches

### Brute Force Dynamic Programming

A natural first idea is dynamic programming.

Define:

```
dp[i] = minimum trips needed to deliver first i boxes
```

Then for every `i`, we try all possible previous split points `j`.

If boxes `j ... i-1` form a valid shipment, then:

```
dp[i] = min(dp[i], dp[j] + trip_cost(j, i-1))
```

To compute the shipment cost:

- Count port transitions inside the batch
- Add 2 for departure and return

This approach is correct because it examines every valid partitioning of the boxes.

However, it is far too slow.

There are `O(n^2)` possible intervals, and even with prefix sums for optimization, the runtime remains quadratic.

With `n = 100000`, this is infeasible.

### Optimal Dynamic Programming with Monotonic Queue

The key insight is that the transition formula can be rewritten into a form suitable for monotonic queue optimization.

Suppose:

```
cost(j, i)
```

represents the trip cost for delivering boxes from `j` to `i`.

Then:

```
dp[i+1] = min(dp[j] + cost(j, i))
```

We can preprocess port transitions using prefix sums.

Define:

```
diff[i]
```

as the number of port changes up to index `i`.

Then the shipment cost becomes:

```
(diff[i] - diff[j]) + 2
```

So:

```
dp[i+1] = min(dp[j] - diff[j]) + diff[i] + 2
```

Now the only changing part for each candidate `j` is:

```
dp[j] - diff[j]
```

We maintain candidate starting indices in a monotonic deque while enforcing:

- Box count constraint
- Weight constraint

This reduces the complexity from `O(n^2)` to `O(n)`.

| Optimal DP + Monotonic Queue | O(n) | O(n) | Maintains optimal transitions efficiently |

## Algorithm Walkthrough

### Step 1: Precompute Prefix Weights

We create a prefix sum array for weights.

```
prefixWeight[i]
```

stores the total weight of the first `i` boxes.

This allows constant time weight calculation for any interval.

### Step 2: Precompute Port Transition Prefixes

We define:

```
diff[i]
```

as the number of port changes among boxes `0 ... i`.

For example:

```
ports: 1 1 2 2 3
diff : 0 0 1 1 2
```

This lets us compute the number of port switches inside any batch in constant time.

### Step 3: Define Dynamic Programming State

We define:

```
dp[i]
```

as the minimum trips needed to deliver the first `i` boxes.

Initially:

```
dp[0] = 0
```

because no boxes require no trips.

### Step 4: Rewrite the Transition

Suppose we deliver boxes `j ... i-1` together.

The shipment cost is:

```
(diff[i-1] - diff[j]) + 2
```

So:

```
dp[i] = dp[j] + diff[i-1] - diff[j] + 2
```

Rearrange:

```
dp[i] = (dp[j] - diff[j]) + diff[i-1] + 2
```

For fixed `i`, we want the minimum value of:

```
dp[j] - diff[j]
```

among valid `j`.

### Step 5: Maintain Valid Shipment Window

A shipment is valid only if:

```
i - j <= maxBoxes
```

and

```
prefixWeight[i] - prefixWeight[j] <= maxWeight
```

We remove invalid indices from the front of the deque.

### Step 6: Maintain Monotonic Queue

For each index `j`, define:

```
value(j) = dp[j] - diff[j]
```

We maintain the deque in increasing order of this value.

This guarantees the front always contains the best candidate.

### Step 7: Compute DP

For each `i`:

1. Remove invalid candidates
2. Use the front candidate to compute `dp[i]`
3. Insert current index while preserving monotonicity

This gives linear complexity because every index enters and leaves the deque at most once.

### Why it works

The dynamic programming recurrence correctly considers every valid last shipment.

The monotonic queue optimization does not change the recurrence, it only accelerates selecting the best previous split point.

Because the deque always stores candidates in increasing order of:

```
dp[j] - diff[j]
```

the optimal transition is always available at the front.

The validity constraints ensure every chosen shipment respects both capacity limits.

Therefore the algorithm computes the true minimum trip count.

## Python Solution

```python
from typing import List
from collections import deque

class Solution:
    def boxDelivering(
        self,
        boxes: List[List[int]],
        portsCount: int,
        maxBoxes: int,
        maxWeight: int
    ) -> int:
        
        n = len(boxes)

        prefix_weight = [0] * (n + 1)
        diff = [0] * (n + 1)

        for i in range(n):
            port, weight = boxes[i]

            prefix_weight[i + 1] = prefix_weight[i] + weight

            if i > 0:
                diff[i] = diff[i - 1]
                if boxes[i][0] != boxes[i - 1][0]:
                    diff[i] += 1

        dp = [0] * (n + 1)

        dq = deque([0])

        for i in range(1, n + 1):

            while dq and (
                i - dq[0] > maxBoxes or
                prefix_weight[i] - prefix_weight[dq[0]] > maxWeight
            ):
                dq.popleft()

            j = dq[0]

            dp[i] = dp[j] + diff[i - 1] - diff[j] + 2

            if i < n:
                current_value = dp[i] - diff[i]

                while dq and (
                    dp[dq[-1]] - diff[dq[-1]]
                    >= current_value
                ):
                    dq.pop()

                dq.append(i)

        return dp[n]
```

The implementation begins by preprocessing two prefix-style arrays.

`prefix_weight` allows constant time calculation of shipment weights. Without it, checking shipment validity would require repeatedly summing intervals, which would be too expensive.

The `diff` array tracks how many port transitions have occurred up to each position. This converts shipment trip calculation into an `O(1)` operation.

The dynamic programming array `dp` stores the minimum cost for delivering the first `i` boxes.

The deque stores candidate split points. Its purpose is twofold:

- Keep only valid shipment starts
- Keep candidates ordered by optimal DP contribution

At each iteration:

1. Invalid shipment starts are removed from the front.
2. The best valid candidate is used to compute `dp[i]`.
3. The current index is inserted while preserving monotonic order.

Each index enters and leaves the deque once, which guarantees linear complexity.

## Go Solution

```go
package main

import "container/list"

func boxDelivering(boxes [][]int, portsCount int, maxBoxes int, maxWeight int) int {
	n := len(boxes)

	prefixWeight := make([]int, n+1)
	diff := make([]int, n+1)

	for i := 0; i < n; i++ {
		port := boxes[i][0]
		weight := boxes[i][1]

		prefixWeight[i+1] = prefixWeight[i] + weight

		if i > 0 {
			diff[i] = diff[i-1]

			if port != boxes[i-1][0] {
				diff[i]++
			}
		}
	}

	dp := make([]int, n+1)

	deque := list.New()
	deque.PushBack(0)

	for i := 1; i <= n; i++ {

		for deque.Len() > 0 {
			front := deque.Front().Value.(int)

			if i-front > maxBoxes ||
				prefixWeight[i]-prefixWeight[front] > maxWeight {
				deque.Remove(deque.Front())
			} else {
				break
			}
		}

		j := deque.Front().Value.(int)

		dp[i] = dp[j] + diff[i-1] - diff[j] + 2

		if i < n {
			currentValue := dp[i] - diff[i]

			for deque.Len() > 0 {
				back := deque.Back().Value.(int)

				if dp[back]-diff[back] >= currentValue {
					deque.Remove(deque.Back())
				} else {
					break
				}
			}

			deque.PushBack(i)
		}
	}

	return dp[n]
}
```

The Go implementation closely mirrors the Python version.

A `container/list` doubly linked list is used to implement the deque efficiently.

Go slices are used for prefix arrays and DP storage.

Integer overflow is not an issue because the maximum number of trips is bounded by a small multiple of `n`, which safely fits inside Go's `int`.

Unlike Python, Go requires explicit type assertions when retrieving values from the linked list.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each index enters and leaves the deque at most once |
| Space | O(n) | Prefix arrays, DP array, and deque storage |

The algorithm is linear because all operations on the deque are amortized constant time. Every index is pushed once and popped once.

Without the monotonic queue optimization, the DP would require checking many previous states repeatedly, leading to quadratic complexity.

## Edge Cases

### All Boxes Have the Same Port

If every box goes to the same port, the shipment cost becomes minimal because there are no internal port transitions.

A buggy implementation might accidentally count unnecessary port changes or split shipments inefficiently.

This implementation handles the case correctly because `diff` never increases, so the shipment cost remains:

```
2
```

for every fully packed batch.

### Weight Constraint Forces Single-Box Shipments

When every box weight equals `maxWeight`, each shipment can contain only one box.

Naive implementations sometimes fail to remove invalid states correctly when the weight window becomes too large.

This solution continuously removes invalid candidates from the deque front before computing transitions, ensuring only feasible shipment starts are considered.

### Alternating Ports

Inputs like:

```
1,2,1,2,1,2
```

maximize port transitions.

Greedy strategies often fail here because packing more boxes into one shipment can increase the number of trips significantly.

The DP formulation evaluates all valid split points and always chooses the partition with the minimum total cost.
