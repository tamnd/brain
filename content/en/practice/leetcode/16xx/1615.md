## Solution

## Problem Understanding

The problem describes a network of `n` cities connected by bidirectional roads. Each road connects exactly two distinct cities, and there can be at most one road between any pair of cities.

For any pair of different cities `(u, v)`, the **network rank** is defined as:

- The number of roads directly connected to city `u`
- Plus the number of roads directly connected to city `v`
- Minus one if there is a direct road between `u` and `v`, because that road would otherwise be counted twice

The goal is to compute the **maximum network rank** among all possible pairs of cities.

In graph terminology, the cities form the vertices of an undirected graph, and the roads form the edges. The network rank of two cities is essentially:

```
degree(u) + degree(v) - shared_edge
```

where `shared_edge` is `1` if the cities are directly connected, otherwise `0`.

The input consists of:

- `n`, the number of cities labeled from `0` to `n - 1`
- `roads`, an array of bidirectional edges

The output is a single integer representing the largest possible network rank among every pair of cities.

The constraints are relatively small:

- `n <= 100`
- Maximum number of roads is `n * (n - 1) / 2`

Since the number of cities is only 100, an `O(n²)` solution is completely acceptable. This is an important observation because we can safely examine every pair of cities without performance concerns.

Several edge cases are important:

- No roads at all, every degree is zero
- Fully connected graphs
- Disconnected components
- Cities with identical degrees
- A maximal pair that is not directly connected
- A maximal pair that is directly connected and therefore requires subtracting one

The problem guarantees:

- No duplicate roads
- No self-loops
- Roads are bidirectional

These guarantees simplify the implementation significantly.

## Approaches

### Brute Force Approach

The brute-force idea is to compute the network rank separately for every pair of cities by scanning the entire `roads` array each time.

For every pair `(u, v)`:

1. Count how many roads touch `u`
2. Count how many roads touch `v`
3. Check whether a road directly connects them
4. Compute the rank

This approach is correct because it explicitly evaluates the definition of network rank for every pair.

However, it is inefficient because for each of the `O(n²)` pairs, we may scan all roads. If there are `m` roads, the total complexity becomes `O(n² * m)`.

Although the constraints here are small enough that even this could pass, it is unnecessarily repetitive because the same degree information is recomputed many times.

### Optimal Approach

The key observation is that the degree of each city never changes. Instead of recalculating it repeatedly, we can preprocess it once.

We store:

- The degree of every city
- Whether two cities are directly connected

Then for every pair `(u, v)`:

```
network_rank = degree[u] + degree[v]
```

If they share a direct road, subtract one.

This reduces the repeated work dramatically.

The optimal solution works efficiently because:

- Degree lookup becomes `O(1)`
- Connectivity check becomes `O(1)`
- We only iterate through all city pairs once

Since there are at most `n²` pairs, the total complexity becomes `O(n²)`.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n² * m) | O(1) | Recomputes degrees repeatedly |
| Optimal | O(n² + m) | O(n²) | Uses preprocessing for constant-time lookups |

## Algorithm Walkthrough

1. Create an array called `degree` of size `n`.

Each entry stores how many roads are directly connected to that city. This allows constant-time degree lookup later.
2. Create a connectivity structure.

Since `n <= 100`, an adjacency matrix is simple and efficient. We use a 2D boolean matrix where:

```
connected[u][v] = True
```

means there is a direct road between `u` and `v`.
3. Process every road.

For each road `[u, v]`:

- Increment `degree[u]`
- Increment `degree[v]`
- Mark both `connected[u][v]` and `connected[v][u]` as `True`

This preprocessing step gives us all the information needed for fast queries.
4. Initialize a variable `max_rank = 0`.

This stores the best network rank seen so far.
5. Iterate through every pair of cities.

Use two nested loops:

- Outer loop for city `u`
- Inner loop for city `v > u`

We only consider each pair once.
6. Compute the pair's network rank.

Start with:

```
rank = degree[u] + degree[v]
```

If the cities are directly connected:

```
rank -= 1
```

because the shared road was counted twice.
7. Update the maximum.

Compare the current rank with `max_rank` and keep the larger value.
8. Return `max_rank`.

### Why it works

The algorithm works because every network rank depends only on two pieces of information:

- The degree of each city
- Whether the two cities share a direct road

The preprocessing phase stores both pieces exactly. Then every pair is evaluated according to the precise definition of network rank. Since all pairs are checked, the maximum found must be the true maximal network rank.

## Python Solution

```python
from typing import List

class Solution:
    def maximalNetworkRank(self, n: int, roads: List[List[int]]) -> int:
        degree = [0] * n
        connected = [[False] * n for _ in range(n)]

        for u, v in roads:
            degree[u] += 1
            degree[v] += 1
            connected[u][v] = True
            connected[v][u] = True

        max_rank = 0

        for u in range(n):
            for v in range(u + 1, n):
                rank = degree[u] + degree[v]

                if connected[u][v]:
                    rank -= 1

                max_rank = max(max_rank, rank)

        return max_rank
```

The implementation begins by initializing the `degree` array and adjacency matrix. The `degree` array tracks how many roads connect to each city, while the adjacency matrix enables constant-time checks for direct connections.

The preprocessing loop updates both structures simultaneously. Every road increases the degree count for both endpoints because the graph is undirected.

The nested loops iterate through every unique pair of cities. For each pair, the algorithm computes the total degree count and subtracts one if the cities are directly connected.

Finally, the algorithm continuously updates the largest network rank encountered and returns it at the end.

## Go Solution

```go
func maximalNetworkRank(n int, roads [][]int) int {
	degree := make([]int, n)

	connected := make([][]bool, n)
	for i := 0; i < n; i++ {
		connected[i] = make([]bool, n)
	}

	for _, road := range roads {
		u := road[0]
		v := road[1]

		degree[u]++
		degree[v]++

		connected[u][v] = true
		connected[v][u] = true
	}

	maxRank := 0

	for u := 0; u < n; u++ {
		for v := u + 1; v < n; v++ {
			rank := degree[u] + degree[v]

			if connected[u][v] {
				rank--
			}

			if rank > maxRank {
				maxRank = rank
			}
		}
	}

	return maxRank
}
```

The Go implementation follows the same logic as the Python solution. The primary difference is explicit slice initialization.

The adjacency matrix is represented as a slice of boolean slices. Since Go does not have Python's list comprehensions, each row must be allocated individually.

Integer overflow is not a concern because the maximum possible network rank is less than `2 * 99 = 198`.

## Edge Cases

### No Roads

When `roads` is empty, every city has degree zero. A careless implementation might assume at least one road exists and fail during processing.

This implementation handles the case naturally because:

- The degree array remains all zeros
- The adjacency matrix remains all `False`
- Every pair produces rank `0`

The final answer is therefore correctly `0`.

### Fully Connected Graph

In a complete graph, every city connects to every other city. This is important because every pair shares a direct edge and therefore requires subtracting one.

For `n` cities:

```
degree = n - 1
```

For any pair:

```
(n - 1) + (n - 1) - 1
= 2n - 3
```

The implementation correctly subtracts one using the adjacency matrix check.

### Best Pair Not Directly Connected

A common bug is always subtracting one regardless of whether the cities share a road.

Consider:

```
0 - 1 - 2 - 3
```

Cities `1` and `3` are not directly connected, so their network rank should be:

```
2 + 1 = 3
```

not `2`.

The adjacency matrix ensures subtraction occurs only when a direct edge exists.

### Disconnected Components

The graph does not need to be fully connected. Some cities may belong to entirely separate components.

The algorithm still works because network rank depends only on local degree counts and direct connectivity, not on graph reachability. The implementation never assumes the graph is connected.
