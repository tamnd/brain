# LeetCode 1630 - Arithmetic Subarrays

## Problem Understanding

The problem gives us an integer array `nums` and multiple queries represented by arrays `l` and `r`. For each query `i`, we examine the subarray:

```
nums[l[i]] ... nums[r[i]]
```

The question is whether the elements in that subarray can be rearranged into an arithmetic sequence.

An arithmetic sequence is a sequence where the difference between every pair of consecutive elements is identical. For example:

```
2, 4, 6, 8
```

has a common difference of `2`, while:

```
7, 7, 7, 7
```

has a common difference of `0`.

The important detail is that the subarray does not need to already be arithmetic in its current order. We are allowed to rearrange the elements in any order we want. This means the actual ordering inside the original array does not matter for a query. What matters is whether some permutation of the elements forms an arithmetic progression.

The constraints are relatively small:

- `n <= 500`
- `m <= 500`

This tells us that an `O(m * k log k)` solution, where `k` is the subarray size, is perfectly acceptable because even the worst case remains manageable.

The key observation is that once a sequence is sorted, checking whether it is arithmetic becomes very easy. If the sorted numbers have the same consecutive difference everywhere, then the sequence can be rearranged into an arithmetic progression.

Several edge cases are important:

- All numbers identical, such as `[5,5,5,5]`
- Negative numbers
- Very short subarrays of length `2`, which are always arithmetic
- Duplicate values that break the required difference pattern
- Unsorted inputs where only the rearranged version becomes arithmetic

For example:

```
[4,6,5]
```

is not arithmetic in its current order, but after sorting:

```
[4,5,6]
```

it becomes arithmetic with difference `1`.

## Approaches

### Brute Force Approach

The most direct solution is to generate every possible permutation of each queried subarray and test whether any permutation forms an arithmetic sequence.

For a subarray of size `k`, there are `k!` permutations. For each permutation, we would compare consecutive differences.

This approach is correct because it explicitly checks every possible arrangement. If any arrangement is arithmetic, we return `true`.

However, factorial complexity becomes infeasible very quickly. Even for modest subarray sizes, the number of permutations explodes.

For example:

```
10! = 3,628,800
```

Checking millions of permutations per query is far too slow.

### Optimal Approach

The key insight is that if a sequence can be rearranged into an arithmetic progression, then its sorted order must also form an arithmetic progression.

Why?

In an arithmetic sequence, values increase by a fixed amount. Sorting naturally places the elements into that exact increasing order.

Therefore, for each query:

1. Extract the subarray
2. Sort it
3. Compute the difference between the first two elements
4. Verify that every consecutive pair has the same difference

If all differences match, the subarray can be rearranged into an arithmetic sequence.

This dramatically reduces complexity because sorting dominates the work.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m * k!) | O(k) | Generates all permutations |
| Optimal | O(m * k log k) | O(k) | Sorts each queried subarray and checks differences |

Here:

- `m` = number of queries
- `k` = average subarray length

## Algorithm Walkthrough

### Optimal Algorithm

1. Initialize an empty result list.

We will store one boolean result for each query.
2. Iterate through every query index `i`.

Each query defines a subarray range:

```
[l[i], r[i]]
```
3. Extract the subarray from `nums`.

Example:

```
subarray = nums[l[i]:r[i]+1]
```

We include `r[i] + 1` because Python slicing excludes the end index.
4. Sort the subarray.

Sorting is important because any arithmetic progression must have equal spacing in sorted order.
5. Compute the expected common difference.

After sorting:

```
diff = sorted_subarray[1] - sorted_subarray[0]
```
6. Traverse the sorted subarray and compare consecutive differences.

For every index:

```
sorted_subarray[j] - sorted_subarray[j-1]
```

must equal `diff`.
7. If any difference mismatches, mark the query as `false`.

This means no arithmetic rearrangement exists.
8. Otherwise, mark the query as `true`.

The sorted order itself forms a valid arithmetic progression.
9. Return the final result list.

### Why it works

Sorting transforms the subarray into the only possible increasing arrangement. Any arithmetic progression must have equal gaps between consecutive sorted elements. Therefore, if the sorted sequence does not have constant differences, no rearrangement can possibly satisfy the arithmetic condition. Conversely, if the sorted sequence does have constant differences, then that sorted order itself is a valid arithmetic sequence.

## Python Solution

```python
from typing import List

class Solution:
    def checkArithmeticSubarrays(
        self,
        nums: List[int],
        l: List[int],
        r: List[int]
    ) -> List[bool]:

        def is_arithmetic(subarray: List[int]) -> bool:
            subarray.sort()

            difference = subarray[1] - subarray[0]

            for i in range(2, len(subarray)):
                if subarray[i] - subarray[i - 1] != difference:
                    return False

            return True

        result = []

        for start, end in zip(l, r):
            current_subarray = nums[start:end + 1]
            result.append(is_arithmetic(current_subarray))

        return result
```

The implementation begins by defining a helper function called `is_arithmetic`. This function receives a subarray, sorts it, and checks whether all consecutive differences are identical.

The sorting step is the foundation of the algorithm. Once sorted, any valid arithmetic progression must appear with equal spacing between neighbors.

The variable `difference` stores the expected common difference using the first two elements.

The loop begins at index `2` because the first difference has already been established. Each new difference is compared against the expected value. If any mismatch occurs, the function immediately returns `False`.

The main function processes every query using `zip(l, r)`. For each query, it extracts the corresponding subarray and passes it to the helper function.

Finally, the boolean result for each query is collected and returned.

## Go Solution

```go
package main

import "sort"

func checkArithmeticSubarrays(nums []int, l []int, r []int) []bool {

	isArithmetic := func(subarray []int) bool {
		sort.Ints(subarray)

		diff := subarray[1] - subarray[0]

		for i := 2; i < len(subarray); i++ {
			if subarray[i]-subarray[i-1] != diff {
				return false
			}
		}

		return true
	}

	result := make([]bool, 0, len(l))

	for i := 0; i < len(l); i++ {
		start := l[i]
		end := r[i]

		subarray := make([]int, end-start+1)
		copy(subarray, nums[start:end+1])

		result = append(result, isArithmetic(subarray))
	}

	return result
}
```

The Go solution follows exactly the same algorithmic structure as the Python version.

One important implementation detail is that Go slices reference underlying arrays. Since `sort.Ints()` modifies the slice in place, we create a copy of the subarray before sorting. Otherwise, the original `nums` array would be unintentionally modified.

The `copy()` function ensures that each query operates independently.

Go integers are fully sufficient for the given constraints, so no overflow concerns exist.

## Edge Cases

### All Elements Equal

Consider a subarray like:

```
[5,5,5,5]
```

The common difference is `0`. Some implementations mistakenly assume differences must be positive or nonzero. Our implementation correctly computes:

```
5 - 5 = 0
```

for every consecutive pair, so the sequence is recognized as arithmetic.

### Two-Element Subarrays

Any sequence with exactly two elements is automatically arithmetic because there is only one difference to compare.

For example:

```
[100, -50]
```

There are no additional pairs that could violate the condition. Our implementation handles this naturally because the verification loop starts at index `2`, meaning it never executes for length `2`.

### Unsorted but Valid Rearrangements

A common mistake is checking the subarray in its original order rather than after rearrangement.

For example:

```
[9,3,7,5]
```

is not arithmetic in its current arrangement. However, after sorting:

```
[3,5,7,9]
```

it becomes arithmetic.

Our implementation always sorts first, guaranteeing that valid rearrangements are correctly detected.

### Duplicate Values That Break the Pattern

Consider:

```
[1,1,2,5]
```

After sorting:

```
[1,1,2,5]
```

Differences become:

```
0, 1, 3
```

Since the differences are inconsistent, the implementation correctly returns `False`.

Without explicitly checking every consecutive difference, an incorrect solution might mistakenly accept this case.
