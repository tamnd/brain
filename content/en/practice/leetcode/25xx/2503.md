# LeetCode 2503 - Maximum Number of Points From Grid Queries

## Problem Understanding

This problem gives us a two dimensional grid of positive integers and a list of query values. For every query, we begin at the top left corner of the grid, which is position `(0, 0)`.

The movement rules are important:

- We may move in four directions, up, down, left, and right.
- We may revisit cells multiple times.
- We only gain a point for a cell the first time we visit it.
- We may continue moving only while the current cell's value is strictly smaller than the query value.

The goal for each query is to determine the maximum number of distinct cells we can visit starting from the top left cell.

The key observation is that a query value acts like a threshold. For a query `q`, we may traverse only cells whose value is strictly less than `q`. Therefore, the answer for a query is simply the number of cells reachable from `(0, 0)` using only cells with value `< q`.

The constraints are large:

- The grid can contain up to `10^5` cells.
- There can be up to `10^4` queries.

A naive solution that runs a full BFS or DFS for every query would be far too slow because traversing the grid repeatedly would lead to excessive work.

There are several important edge cases to consider:

- If `grid[0][0] >= query`, the answer is immediately `0` because we cannot even start.
- Queries are not sorted, so we must preserve original ordering in the output.
- Multiple queries may have the same value.
- Large grids require an efficient traversal strategy that avoids repeated processing of the same cells.
- Since movement is based on connectivity, cells with small values are not automatically reachable if surrounded by larger blocked cells.

## Approaches

### Brute Force Approach

The most direct solution is to process each query independently.

For every query:

1. Start a BFS or DFS from `(0, 0)`.
2. Only move into cells whose value is strictly less than the query.
3. Count the number of reachable cells.

This works because BFS or DFS correctly explores all connected cells satisfying the threshold condition.

However, the time complexity is too large. In the worst case:

- Each query explores the entire grid.
- The grid contains up to `10^5` cells.
- There are up to `10^4` queries.

This leads to a worst case complexity of:

`O(k * m * n)`

which can reach `10^9` operations.

### Optimal Approach

The key insight is that queries can be processed in sorted order.

Suppose we process queries from smallest to largest:

- If a cell is reachable for query `q1`,
- then it is also reachable for every larger query `q2 > q1`, assuming connectivity remains valid.

Instead of recomputing BFS repeatedly, we expand the reachable region incrementally.

We use a min heap to always process the smallest grid value currently reachable from the explored frontier.

The algorithm works like Dijkstra's traversal:

- Start with `(0,0)` in the heap.
- For each query threshold:

- Pop all cells whose value is smaller than the query.
- Add their neighbors into the heap.
- Count how many cells have become reachable.

Because every cell is inserted and removed at most once, the total work across all queries is efficient.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(k × m × n) | O(m × n) | Runs BFS or DFS independently for each query |
| Optimal | O(m × n × log(m × n) + k log k) | O(m × n) | Incremental expansion using sorted queries and a min heap |

## Algorithm Walkthrough

### Step 1: Sort the Queries

We first pair each query with its original index:

```
indexed_queries = [(query, index)]
```

This is necessary because we will sort the queries for efficient processing, but the final answers must match the original query order.

### Step 2: Initialize the Min Heap

We use a min heap storing:

```
(cell_value, row, col)
```

Initially, the heap contains only the top left cell.

The heap ensures we always process the smallest available cell first.

### Step 3: Maintain a Visited Matrix

We need to ensure every cell is added to the heap only once.

A boolean `visited` matrix prevents duplicate processing.

### Step 4: Process Queries in Increasing Order

For each query `q`:

- While the heap is not empty and the smallest cell value is less than `q`:

- Remove that cell from the heap.
- Increase the reachable cell count.
- Add all unvisited neighbors into the heap.

At the moment the loop stops:

- Every reachable cell with value `< q` has been processed.
- Every remaining heap element has value `>= q`.

Therefore, the current count is the answer for this query.

### Step 5: Store the Answer

Since queries were sorted, we place the result into the correct original position using the saved query index.

### Step 6: Return the Final Array

After all queries are processed, return the completed answer array.

### Why it works

The heap always exposes the smallest frontier cell that could become reachable next. Because queries are processed in increasing order, once a cell becomes reachable for a smaller query, it remains reachable for all larger queries.

Each cell is processed exactly once, and when a query `q` is handled, all reachable cells with value `< q` have already been counted. Therefore, the accumulated count is exactly the number of cells reachable under that threshold.

## Python Solution

```python
from typing import List
import heapq

class Solution:
    def maxPoints(self, grid: List[List[int]], queries: List[int]) -> List[int]:
        rows = len(grid)
        cols = len(grid[0])

        indexed_queries = sorted(
            [(query, index) for index, query in enumerate(queries)]
        )

        answers = [0] * len(queries)

        visited = [[False] * cols for _ in range(rows)]

        min_heap = [(grid[0][0], 0, 0)]
        visited[0][0] = True

        points = 0

        directions = [(1, 0), (-1, 0), (0, 1), (0, -1)]

        for query, original_index in indexed_queries:

            while min_heap and min_heap[0][0] < query:
                value, row, col = heapq.heappop(min_heap)

                points += 1

                for dr, dc in directions:
                    new_row = row + dr
                    new_col = col + dc

                    if (
                        0 <= new_row < rows
                        and 0 <= new_col < cols
                        and not visited[new_row][new_col]
                    ):
                        visited[new_row][new_col] = True
                        heapq.heappush(
                            min_heap,
                            (grid[new_row][new_col], new_row, new_col)
                        )

            answers[original_index] = points

        return answers
```

The implementation begins by sorting the queries while preserving their original indices. This allows us to process thresholds incrementally while still returning answers in the correct order.

The `visited` matrix guarantees that every cell enters the heap only once. This is important because multiple neighboring cells may attempt to push the same location.

The min heap stores frontier cells ordered by their grid value. During processing of a query, we repeatedly pop cells whose values are strictly smaller than the query threshold.

Every popped cell contributes one reachable point. After processing that cell, we examine its four neighbors and add unvisited neighbors into the heap.

The variable `points` maintains the cumulative number of reachable cells across all processed queries. Since queries are sorted, we never need to reset or recompute traversal state.

Finally, each answer is stored using the original query index so that the returned array matches the input ordering.

## Go Solution

```go
package main

import (
	"container/heap"
	"sort"
)

type Cell struct {
	value int
	row   int
	col   int
}

type MinHeap []Cell

func (h MinHeap) Len() int {
	return len(h)
}

func (h MinHeap) Less(i, j int) bool {
	return h[i].value < h[j].value
}

func (h MinHeap) Swap(i, j int) {
	h[i], h[j] = h[j], h[i]
}

func (h *MinHeap) Push(x interface{}) {
	*h = append(*h, x.(Cell))
}

func (h *MinHeap) Pop() interface{} {
	old := *h
	n := len(old)
	item := old[n-1]
	*h = old[:n-1]
	return item
}

func maxPoints(grid [][]int, queries []int) []int {
	rows := len(grid)
	cols := len(grid[0])

	type Query struct {
		value int
		index int
	}

	indexedQueries := make([]Query, len(queries))

	for i, q := range queries {
		indexedQueries[i] = Query{q, i}
	}

	sort.Slice(indexedQueries, func(i, j int) bool {
		return indexedQueries[i].value < indexedQueries[j].value
	})

	answers := make([]int, len(queries))

	visited := make([][]bool, rows)
	for i := range visited {
		visited[i] = make([]bool, cols)
	}

	minHeap := &MinHeap{}
	heap.Init(minHeap)

	heap.Push(minHeap, Cell{grid[0][0], 0, 0})
	visited[0][0] = true

	points := 0

	directions := [][]int{
		{1, 0},
		{-1, 0},
		{0, 1},
		{0, -1},
	}

	for _, query := range indexedQueries {

		for minHeap.Len() > 0 && (*minHeap)[0].value < query.value {

			current := heap.Pop(minHeap).(Cell)

			points++

			for _, dir := range directions {
				newRow := current.row + dir[0]
				newCol := current.col + dir[1]

				if newRow >= 0 &&
					newRow < rows &&
					newCol >= 0 &&
					newCol < cols &&
					!visited[newRow][newCol] {

					visited[newRow][newCol] = true

					heap.Push(
						minHeap,
						Cell{
							grid[newRow][newCol],
							newRow,
							newCol,
						},
					)
				}
			}
		}

		answers[query.index] = points
	}

	return answers
}
```

The Go implementation follows the same logic as the Python version, but Go requires a custom heap implementation using the `container/heap` package.

The `MinHeap` type implements the required heap interface methods:

- `Len`
- `Less`
- `Swap`
- `Push`
- `Pop`

Go slices are used for dynamic arrays, and the visited matrix is initialized using nested slices of booleans.

There are no integer overflow concerns because the maximum answer is at most `10^5`, which easily fits within Go's `int` type.

## Edge Cases

### Top Left Cell Is Already Too Large

If `grid[0][0] >= query`, traversal cannot even begin because movement rules require the current cell value to be strictly smaller than the query.

This is a common source of off-by-one bugs because many implementations accidentally use `<=` instead of `<`.

The implementation handles this naturally because the heap processing loop only pops cells when:

```
heap[0][0] < query
```

If the starting cell fails this condition, no processing occurs and the answer remains `0`.

### Queries Are Not Sorted

The input queries may appear in arbitrary order. Processing them directly would prevent incremental expansion.

For example:

```
queries = [100, 2, 50]
```

The optimal strategy requires sorted processing, but the output must still match the original ordering.

The implementation solves this by storing:

```
(query_value, original_index)
```

After processing sorted queries, answers are written back using the saved indices.

### High Value Cells Blocking Reachability

A cell having a small value does not automatically make it reachable.

Consider:

```
[
  [1,100,1]
]
```

For query `50`, the last cell with value `1` is unreachable because the `100` blocks the path.

A naive counting strategy based purely on value comparisons would incorrectly count all small cells.

The heap-based traversal correctly models connectivity because neighbors are only explored from already reachable cells.
