## Solution

## Problem Understanding

The problem gives us an integer array `nums` and defines two values based on the array:

- The low score is the minimum absolute difference between any pair of elements.
- The high score is the maximum absolute difference between any pair of elements.
- The total score is the sum of these two values.

We are allowed to change exactly two elements in the array to any values we want. After performing these changes, we must return the minimum possible score.

To understand the problem more clearly, notice an important observation about the score:

- The minimum absolute difference can always be made `0` after changing two elements, because we can set at least two numbers to the same value.
- Therefore, minimizing the total score mainly becomes equivalent to minimizing the maximum difference between elements.

The maximum absolute difference in an array is simply:

```
max(nums) - min(nums)
```

So the real task becomes:

> After changing at most two elements, minimize the difference between the largest and smallest remaining values.

The constraints are important:

- `3 <= nums.length <= 10^5`
- `1 <= nums[i] <= 10^9`

Since the array can contain up to one hundred thousand elements, any brute-force approach that tries many combinations of changes will be too slow. This strongly suggests that we need a sorting-based greedy observation.

There are several edge cases worth noticing immediately:

- Arrays of length `3` can always become all equal after changing two elements, so the answer is always `0`.
- Duplicate values already produce a low score of `0`.
- Extremely large values require careful handling, but standard integer arithmetic is sufficient in both Python and Go.
- Since only two elements can be changed, the optimal strategy must involve modifying extreme values rather than middle values.

## Approaches

### Brute Force Approach

A brute-force solution would try every possible pair of indices to modify, then attempt every meaningful replacement strategy and compute the resulting score.

After selecting two positions, we would need to determine what values to assign them. Since values can theoretically be any integer, the search space becomes enormous. Even if we restrict ourselves to choosing values from existing numbers, we still end up with a very large number of possibilities.

For each configuration, we would compute:

- The minimum pairwise absolute difference
- The maximum pairwise absolute difference

The brute-force method is correct because it exhaustively checks all valid modifications and therefore cannot miss the optimal answer. However, it is computationally infeasible for arrays of size up to `10^5`.

### Optimal Greedy + Sorting Approach

The key insight is that the low score can always be reduced to `0` by making two elements equal. Therefore, the only quantity that matters is minimizing the high score.

After sorting the array, the high score becomes:

```
nums[n - 1] - nums[0]
```

Since we can change two elements, we can effectively remove the influence of two extreme values.

There are only three meaningful possibilities:

1. Change the two smallest elements.
2. Change the two largest elements.
3. Change one smallest and one largest element.

After sorting, these correspond to:

1. Keep `nums[2]` through `nums[n-1]`

Result: `nums[n-1] - nums[2]`
2. Keep `nums[0]` through `nums[n-3]`

Result: `nums[n-3] - nums[0]`
3. Keep `nums[1]` through `nums[n-2]`

Result: `nums[n-2] - nums[1]`

The minimum of these three values is the answer.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential / infeasible | Very large | Tries all modification combinations |
| Optimal | O(n log n) | O(1) or O(log n) depending on sort | Sorts array and checks only three cases |

## Algorithm Walkthrough

1. Sort the array in non-decreasing order.

Sorting allows us to reason directly about the minimum and maximum values. Since the score depends on extreme values, having the array ordered makes the problem much simpler.
2. Consider changing the two smallest elements.

If we modify the first two elements, then the smallest remaining unchanged value becomes `nums[2]`. The maximum remains `nums[n-1]`.

The resulting range is:

```
nums[n-1] - nums[2]
```
3. Consider changing the two largest elements.

If we modify the last two elements, then the largest remaining unchanged value becomes `nums[n-3]`. The minimum remains `nums[0]`.

The resulting range is:

```
nums[n-3] - nums[0]
```
4. Consider changing one smallest and one largest element.

If we modify one value from each end, then the remaining range becomes:

```
nums[n-2] - nums[1]
```
5. Return the minimum among these three possibilities.

Since every optimal solution must eliminate at most two extreme values, one of these three cases must produce the best answer.

### Why it works

After sorting, only the smallest and largest elements affect the maximum absolute difference. Since changing internal elements does not reduce the range, any optimal strategy must modify elements near the boundaries.

With two allowed changes, there are exactly three ways to remove the influence of extremes:

- Remove two from the left
- Remove two from the right
- Remove one from each side

Checking all three guarantees that we find the minimum possible range, which also minimizes the total score because the low score can always be made `0`.

## Python Solution

```python
from typing import List

class Solution:
    def minimizeSum(self, nums: List[int]) -> int:
        nums.sort()
        n = len(nums)

        option1 = nums[n - 1] - nums[2]
        option2 = nums[n - 2] - nums[1]
        option3 = nums[n - 3] - nums[0]

        return min(option1, option2, option3)
```

The implementation begins by sorting the array so that the smallest and largest elements become easy to access.

The variable `option1` represents the case where the first two elements are changed. The smallest unchanged value is `nums[2]`, while the largest remains `nums[n - 1]`.

The variable `option2` represents changing one smallest and one largest value. The remaining range is between `nums[1]` and `nums[n - 2]`.

The variable `option3` represents changing the two largest elements. The largest unchanged value becomes `nums[n - 3]`.

Finally, the algorithm returns the minimum among the three candidate ranges.

## Go Solution

```go
package main

import "sort"

func minimizeSum(nums []int) int {
	sort.Ints(nums)

	n := len(nums)

	option1 := nums[n-1] - nums[2]
	option2 := nums[n-2] - nums[1]
	option3 := nums[n-3] - nums[0]

	return min(option1, min(option2, option3))
}

func min(a, b int) int {
	if a < b {
		return a
	}
	return b
}
```

The Go implementation follows the same logic as the Python solution.

Go does not provide a built-in `min` function for integers, so we define a helper function manually. The array is sorted using `sort.Ints`.

Since the constraints fit comfortably within Go's `int` range on modern platforms, no special overflow handling is required.

## Edge Cases

One important edge case is when the array length is exactly `3`. Since we are allowed to change two elements, we can always make all three values identical. This guarantees that both the low score and high score become `0`. The implementation handles this naturally because all three computed options evaluate to zero after sorting.

Another important edge case involves arrays with duplicate values. If duplicates already exist, then the low score is already `0`. A naive implementation might incorrectly try to optimize both the low and high scores independently. The current solution correctly recognizes that only the range matters and therefore focuses entirely on minimizing the maximum difference.

A third important edge case is arrays with extremely large integer values, such as values near `10^9`. Some languages can overflow during subtraction if care is not taken. Both Python and Go handle the required arithmetic safely under the given constraints, so the implementation remains correct even for the largest possible inputs.

A final subtle edge case occurs when the optimal strategy changes one smallest and one largest element instead of changing two elements from the same side. Some incorrect greedy approaches only consider removing two minimums or two maximums. By explicitly evaluating all three valid configurations, the implementation guarantees correctness for these balanced cases.
