## Solution

## Problem Understanding

The problem gives us an `n x n` matrix that initially contains only zeroes. We are then given a list of rectangular update operations called `queries`. Each query specifies the coordinates of the top left and bottom right corners of a submatrix. For every query, we must add `1` to every cell inside that rectangle.

More formally, for a query `[row1, col1, row2, col2]`, every position `(r, c)` satisfying:

- `row1 <= r <= row2`
- `col1 <= c <= col2`

must be incremented by `1`.

After processing all queries, we return the final matrix.

The important detail is that each query affects an entire rectangular region, not just a single cell. A naive implementation would iterate through every cell in every rectangle and increment it directly.

The constraints are extremely important here:

- `n <= 500`
- `queries.length <= 10^4`

A single query can potentially affect all `500 x 500 = 250000` cells. If we processed every query cell by cell, the worst case complexity would become:

$$10^4 \times 500 \times 500 = 2.5 \times 10^9$$

That is far too slow for a typical LeetCode time limit.

This immediately suggests that we need a more efficient way to represent range updates.

Several edge cases are worth noting:

- A query may affect only one cell, where `row1 == row2` and `col1 == col2`.
- A query may affect the entire matrix.
- Queries may heavily overlap, meaning some cells receive many increments.
- Updates can touch the last column, so boundary handling must be careful when using prefix techniques.

The problem guarantees that all coordinates are valid and within bounds, which simplifies implementation because we do not need additional validation logic.

## Approaches

### Brute Force Approach

The most direct solution is to simulate each query exactly as described.

For every query:

1. Iterate through all rows from `row1` to `row2`
2. Iterate through all columns from `col1` to `col2`
3. Increment each corresponding matrix cell by `1`

This approach is straightforward and obviously correct because it directly performs the required operation.

However, its performance is poor. In the worst case, every query updates the entire matrix. Since there can be `10^4` queries and each query may touch `250000` cells, the total work becomes enormous.

This solution will likely exceed time limits.

### Optimal Approach, Difference Array with Prefix Sums

The key insight is that repeatedly updating large contiguous ranges is inefficient if we modify every cell individually.

Instead, we can record only where updates start and stop.

For each row independently, we use a one dimensional difference array technique:

- Add `1` at the starting column
- Subtract `1` immediately after the ending column

Later, when we compute prefix sums across the row, the increments automatically propagate through the intended range.

This transforms each query from updating many cells into updating only a few positions.

The idea works because prefix sums reconstruct the accumulated effect of all range updates.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(q * n²) | O(n²) | Directly increments every affected cell |
| Optimal | O(q * n + n²) | O(n²) | Uses row-wise difference arrays and prefix sums |

Where:

- `q = len(queries)`

## Algorithm Walkthrough

### Step 1: Create the Matrix

We initialize an `n x n` matrix filled with zeroes.

This matrix will temporarily store difference values rather than final answers.

```
matrix = [[0] * n for _ in range(n)]
```

### Step 2: Process Each Query Using Difference Updates

For every query `[row1, col1, row2, col2]`:

We iterate through all rows from `row1` to `row2`.

For each row:

1. Add `1` at `col1`
2. Subtract `1` at `col2 + 1`, if it exists

This marks the beginning and end of the increment range.

For example, suppose a row initially looks like this:

```
[0, 0, 0, 0, 0]
```

If we want to increment columns `1` through `3`, we do:

```
[0, +1, 0, 0, -1]
```

After taking prefix sums:

```
[0, 1, 1, 1, 0]
```

The increment has automatically spread across the desired range.

### Step 3: Compute Prefix Sums Row by Row

After all queries are processed, every row contains difference values.

We now convert each row into its actual values by taking prefix sums from left to right.

For each row:

```
matrix[row][col] += matrix[row][col - 1]
```

This accumulates all active increments.

### Step 4: Return the Final Matrix

After prefix processing, the matrix contains the fully updated values.

### Why it works

The difference array stores changes in increment state rather than explicit cell values.

Adding `1` at the left boundary starts an active increment region. Subtracting `1` after the right boundary ends that region. Prefix sums reconstruct how many active updates affect each cell.

Because every query contributes exactly once to its intended interval, every cell receives the correct total number of increments.

## Python Solution

```python
from typing import List

class Solution:
    def rangeAddQueries(self, n: int, queries: List[List[int]]) -> List[List[int]]:
        matrix = [[0] * n for _ in range(n)]

        for row1, col1, row2, col2 in queries:
            for row in range(row1, row2 + 1):
                matrix[row][col1] += 1

                if col2 + 1 < n:
                    matrix[row][col2 + 1] -= 1

        for row in range(n):
            for col in range(1, n):
                matrix[row][col] += matrix[row][col - 1]

        return matrix
```

The implementation begins by creating an `n x n` matrix filled with zeroes. Instead of directly storing final values, this matrix temporarily stores difference updates.

For every query, we iterate through the affected rows. On each row, we mark the start of the increment range by adding `1` at `col1`. We then mark the end of the increment range by subtracting `1` at `col2 + 1`, if that position exists within bounds.

At this stage, the matrix does not yet contain actual answers. It only records where increments begin and end.

The second nested loop computes prefix sums across every row. Each cell accumulates the value from its left neighbor, which reconstructs the true increment counts.

Finally, the fully reconstructed matrix is returned.

## Go Solution

```go
func rangeAddQueries(n int, queries [][]int) [][]int {
	matrix := make([][]int, n)

	for i := 0; i < n; i++ {
		matrix[i] = make([]int, n)
	}

	for _, query := range queries {
		row1 := query[0]
		col1 := query[1]
		row2 := query[2]
		col2 := query[3]

		for row := row1; row <= row2; row++ {
			matrix[row][col1]++

			if col2+1 < n {
				matrix[row][col2+1]--
			}
		}
	}

	for row := 0; row < n; row++ {
		for col := 1; col < n; col++ {
			matrix[row][col] += matrix[row][col-1]
		}
	}

	return matrix
}
```

The Go implementation closely mirrors the Python solution.

The main difference is explicit slice allocation. In Go, we first allocate the outer slice and then allocate each row individually.

Go integers are sufficiently large for this problem because the maximum value in any cell cannot exceed `10^4`, which easily fits within a standard `int`.

Unlike Python lists, Go slices have fixed lengths after allocation here, which improves clarity and avoids accidental resizing.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(q * n + n²) | Each query may touch up to `n` rows, then we compute prefix sums for all `n²` cells |
| Space | O(n²) | The matrix itself stores all intermediate and final values |

The optimized solution avoids touching every cell for every query. Instead, each query performs only two updates per affected row. This dramatically reduces the total amount of work compared to the brute force method.

The prefix sum reconstruction step requires scanning every cell once, which contributes the `O(n²)` term.

## Edge Cases

One important edge case occurs when the update rectangle touches the final column of the matrix. In the difference array technique, we normally subtract `1` at `col2 + 1` to mark the end of the increment region. However, if `col2` is already the last column, then `col2 + 1` would be out of bounds. This is why the implementation explicitly checks:

```
if col2 + 1 < n:
```

Without this condition, the solution would crash with an index error.

Another important edge case is a query that updates only a single cell. In this situation, `row1 == row2` and `col1 == col2`. Some implementations accidentally assume the rectangle spans multiple cells and incorrectly process ranges. This solution handles the case naturally because the loops still execute exactly once, and the difference update correctly affects only that position after prefix reconstruction.

A third critical edge case is heavy overlap between many queries. Multiple updates may affect the same cells repeatedly. A naive implementation could become extremely slow because it repeatedly touches the same large regions. The difference array method handles overlap efficiently because each query only contributes constant work per affected row, and the final prefix sums automatically accumulate all contributions correctly.
