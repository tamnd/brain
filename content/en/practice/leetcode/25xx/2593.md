## Solution

## Problem Understanding

In this problem, we are given an integer array `nums`, where every value is positive. We repeatedly perform a marking process until every element in the array has been marked.

At each step, we must choose the smallest unmarked value in the array. If multiple unmarked elements have the same value, we must choose the one with the smaller index. Once an element is chosen, we add its value to the total score. Then we mark three possible positions:

- The chosen element itself
- Its left neighbor, if it exists
- Its right neighbor, if it exists

The process continues until every index has been marked at least once. The final answer is the total score accumulated from all chosen elements.

The important detail is that marking neighbors can eliminate elements from future consideration before they are ever selected directly. This means the order in which elements are chosen matters heavily, and the problem specifically enforces that we always pick the globally smallest currently unmarked element.

The constraints are large:

- `nums.length` can be up to `10^5`
- `nums[i]` can be up to `10^6`

These constraints immediately rule out inefficient repeated scanning solutions. Any algorithm worse than roughly `O(n log n)` will likely time out for the largest inputs.

Several edge cases are important:

- Arrays of length `1`
- Arrays where all values are equal
- Arrays already sorted
- Arrays with many duplicate minimum values
- Situations where marking one element removes large portions of the remaining candidates

A naive implementation can easily become inefficient because repeatedly searching for the smallest unmarked element requires scanning the entire array many times.

## Approaches

### Brute Force Approach

The most direct approach is to simulate the process exactly as described.

We maintain a boolean array `marked` to track which indices are already marked. At every iteration, we scan the entire array from left to right to find the smallest unmarked value. Once found, we add its value to the score and mark the chosen index along with its adjacent neighbors.

This approach is correct because it faithfully follows the problem statement step by step.

However, its performance is poor. In the worst case, we may perform up to `O(n)` iterations, and each iteration scans the entire array to locate the next smallest unmarked element. This produces a worst-case complexity of `O(n^2)`.

With `n = 100000`, quadratic time is far too slow.

### Optimal Approach

The key observation is that the selection order is entirely determined by two properties:

- Smaller values must be processed first
- For equal values, smaller indices must be processed first

Instead of repeatedly searching for the next candidate, we can preprocess all elements by sorting them once.

We create pairs of `(value, index)` and sort them lexicographically. Python tuple sorting and Go custom sorting naturally handle this correctly because pairs are sorted first by value and then by index.

After sorting, we iterate through the sorted list. For each pair:

- If the index is already marked, we skip it
- Otherwise, we add its value to the score
- Then we mark the index and its neighbors

This works because the sorted order exactly matches the order in which the algorithm would choose elements.

The result is an `O(n log n)` solution dominated by sorting.

## Algorithm Walkthrough

1. Create a list of pairs where each pair contains `(value, index)`.

This allows us to preserve both the numeric value and the original position in the array.
2. Sort the list of pairs.

The sorting order is first by value, then by index. This exactly matches the problem's selection rule.
3. Create a boolean array called `marked`.

This array tracks which indices are already marked and therefore unavailable for future selection.
4. Initialize `score = 0`.

This variable stores the running total.
5. Iterate through the sorted pairs.

For each `(value, index)`:

- If `marked[index]` is already `True`, skip this element because it was already removed by a neighboring selection.
- Otherwise:

- Add `value` to `score`
- Mark `index`
- Mark `index - 1` if it exists
- Mark `index + 1` if it exists
6. Continue until all pairs have been processed.

Since every possible candidate is considered in sorted order, the final score will match the required simulation.

### Why it works

The algorithm works because sorting by `(value, index)` produces exactly the same order that the problem statement requires when selecting elements. At every step, the smallest unmarked element with the smallest index among ties is processed first. The `marked` array ensures that elements removed by neighboring selections are skipped later. Therefore, the simulation performed by the sorted traversal is equivalent to the original process.

## Python Solution

```python
from typing import List

class Solution:
    def findScore(self, nums: List[int]) -> int:
        n = len(nums)

        elements = sorted((value, index) for index, value in enumerate(nums))

        marked = [False] * n
        score = 0

        for value, index in elements:
            if marked[index]:
                continue

            score += value

            marked[index] = True

            if index > 0:
                marked[index - 1] = True

            if index < n - 1:
                marked[index + 1] = True

        return score
```

The implementation begins by constructing and sorting `(value, index)` pairs. This guarantees that iteration follows the exact priority rules defined in the problem.

The `marked` array tracks which positions are no longer eligible for selection. Whenever we encounter an already marked index, we skip it immediately because it was removed earlier as a neighbor.

If the current index is still unmarked, we add its value to the total score and mark the current position along with its adjacent neighbors.

Boundary checks are important before accessing neighbors. The conditions `index > 0` and `index < n - 1` prevent out-of-bounds access at the edges of the array.

Finally, the accumulated score is returned.

## Go Solution

```go
package main

import (
	"sort"
)

func findScore(nums []int) int64 {
	n := len(nums)

	type Pair struct {
		value int
		index int
	}

	elements := make([]Pair, n)

	for i, v := range nums {
		elements[i] = Pair{value: v, index: i}
	}

	sort.Slice(elements, func(i, j int) bool {
		if elements[i].value == elements[j].value {
			return elements[i].index < elements[j].index
		}
		return elements[i].value < elements[j].value
	})

	marked := make([]bool, n)

	var score int64 = 0

	for _, element := range elements {
		index := element.index

		if marked[index] {
			continue
		}

		score += int64(element.value)

		marked[index] = true

		if index > 0 {
			marked[index-1] = true
		}

		if index < n-1 {
			marked[index+1] = true
		}
	}

	return score
}
```

The Go solution follows the same logic as the Python implementation. A custom `Pair` struct stores values and indices together.

Go requires an explicit comparator for sorting. The comparator first compares values, then indices to preserve the correct tie-breaking behavior.

The score uses `int64` because the sum can exceed the range of a 32-bit integer when many large values are added together.

The boolean slice `marked` functions exactly like the Python version.

## Edge Cases

One important edge case is an array with only one element. In this situation, the chosen element has no neighbors, so the algorithm must correctly avoid accessing invalid indices. The implementation handles this safely through boundary checks before marking adjacent elements.

Another important case is when many elements have identical values. Since the problem requires choosing the leftmost element among ties, incorrect sorting behavior can produce the wrong answer. The implementation solves this by sorting pairs `(value, index)`, which naturally applies the required tie-breaking rule.

A third critical edge case occurs when selecting a single element marks most or all of the array. For example, in `[10,1,10]`, choosing the middle element marks every position immediately. A buggy implementation might continue processing already marked indices incorrectly. The `marked` array prevents this by skipping any index that has already been removed from consideration.

Another subtle case involves arrays with alternating small and large values. Neighbor marking can remove large values before they are selected, drastically changing the score. The algorithm correctly simulates this behavior because every selection immediately updates the marked state before continuing through the sorted order.
