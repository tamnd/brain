# LeetCode 2526 - Find Consecutive Integers from a Data Stream

## Problem Understanding

The problem asks us to design a data structure that processes a stream of integers one value at a time. After each new integer is added, we must determine whether the last `k` integers in the stream are all equal to a specific target value.

The class is initialized with two integers:

- `value`, the target integer we care about
- `k`, the number of consecutive occurrences required

Every time the `consec(num)` method is called, a new integer is appended to the stream. The method should return:

- `true` if the most recent `k` integers are all equal to `value`
- `false` otherwise

An important detail is that if fewer than `k` numbers have been seen so far, the answer must always be `false`, because there are not enough elements to satisfy the requirement.

For example, if `value = 4` and `k = 3`, then the stream evolves like this:

| Stream | Last 3 Numbers | Result |
| --- | --- | --- |
| [4] | not enough numbers | false |
| [4,4] | not enough numbers | false |
| [4,4,4] | [4,4,4] | true |
| [4,4,4,3] | [4,4,3] | false |

The constraints are important:

- Up to `10^5` calls to `consec`
- `k` can also be as large as `10^5`

These constraints immediately suggest that repeatedly scanning the last `k` elements after every insertion would be inefficient in the worst case. We need a solution that processes each new number in constant time.

There are several important edge cases to keep in mind:

- The stream may contain many non-target values intermixed with valid values.
- `k` may equal `1`, meaning every occurrence of `value` should immediately return `true`.
- A long sequence of matching values can suddenly be broken by a different number.
- The stream may never reach length `k`.

A correct implementation must efficiently maintain enough information to answer each query without rescanning the stream repeatedly.

## Approaches

### Brute Force Approach

The most straightforward solution is to store every number in the stream inside a list or queue. Whenever `consec(num)` is called:

1. Append `num` to the stream.
2. If the stream length is smaller than `k`, return `false`.
3. Otherwise, examine the last `k` elements one by one.
4. If all `k` elements equal `value`, return `true`.
5. Otherwise, return `false`.

This approach is correct because it directly checks the exact condition described in the problem.

However, it is inefficient. Each call may require scanning up to `k` elements. Since there can be up to `10^5` calls, the worst case becomes `O(n * k)`, which can degrade to `O(10^10)` operations.

That is far too slow.

### Optimal Approach

The key observation is that we do not actually need to store the entire stream or repeatedly inspect the last `k` elements.

Instead, we only need to know how many consecutive occurrences of `value` appear at the end of the stream.

We maintain a counter:

- If the incoming number equals `value`, increment the counter.
- Otherwise, reset the counter to `0`.

At any moment:

- If the counter is at least `k`, then the last `k` numbers are all equal to `value`.
- Otherwise, they are not.

This works because the counter always represents the current streak of consecutive target values ending at the most recent element.

Each operation now takes constant time and constant space.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(k) per call | O(n) | Stores entire stream and scans last k elements repeatedly |
| Optimal | O(1) per call | O(1) | Tracks only the current streak length |

## Algorithm Walkthrough

1. Initialize the data structure with:

- the target `value`
- the required streak length `k`
- a counter called `count`, initially `0`
2. Whenever `consec(num)` is called, compare `num` with `value`.
3. If `num == value`, increment `count` by `1`.

- This means the consecutive streak continues.
4. Otherwise, reset `count` to `0`.

- A different number breaks the streak completely.
5. After updating the counter, check whether `count >= k`.

- If true, the last `k` numbers are all equal to `value`.
- Otherwise, they are not.
6. Return the result of that comparison.

### Why it works

The algorithm maintains an invariant:

> `count` always equals the number of consecutive occurrences of `value` ending at the current position in the stream.

Whenever a non-matching number appears, the streak becomes invalid, so resetting to `0` is correct. Whenever a matching number appears, the streak extends by one.

Therefore, if `count >= k`, there must exist at least `k` consecutive occurrences of `value` at the end of the stream, exactly matching the problem requirement.

## Python Solution

```python
class DataStream:

    def __init__(self, value: int, k: int):
        self.value = value
        self.k = k
        self.count = 0

    def consec(self, num: int) -> bool:
        if num == self.value:
            self.count += 1
        else:
            self.count = 0

        return self.count >= self.k

# Your DataStream object will be instantiated and called as such:
# obj = DataStream(value, k)
# param_1 = obj.consec(num)
```

The implementation closely follows the algorithm described earlier.

The constructor stores the target value and required streak length. It also initializes a counter named `count`, which tracks how many consecutive matching values have appeared most recently.

Inside `consec`, we compare the incoming number against the target value.

If they match, the streak extends, so we increment `count`.

If they do not match, the streak is broken completely, so we reset `count` to zero.

Finally, we return whether the streak length has reached at least `k`.

The implementation is extremely efficient because it avoids storing the stream entirely and only keeps the minimum necessary state.

## Go Solution

```go
type DataStream struct {
	value int
	k     int
	count int
}

func Constructor(value int, k int) DataStream {
	return DataStream{
		value: value,
		k:     k,
		count: 0,
	}
}

func (this *DataStream) Consec(num int) bool {
	if num == this.value {
		this.count++
	} else {
		this.count = 0
	}

	return this.count >= this.k
}

/**
 * Your DataStream object will be instantiated and called as such:
 * obj := Constructor(value, k);
 * param_1 := obj.Consec(num);
 */
```

The Go implementation mirrors the Python solution very closely.

The struct stores the target value, required streak length, and current streak counter.

Go uses explicit struct fields instead of dynamic instance attributes. The `Consec` method uses a pointer receiver so updates to `count` persist across method calls.

There are no integer overflow concerns because the counter never exceeds the number of method calls, which is bounded by `10^5`.

## Edge Cases

One important edge case occurs when `k = 1`. In this scenario, a single occurrence of `value` should immediately return `true`. A buggy implementation might incorrectly require more than one matching value before succeeding. Our implementation handles this naturally because after the first matching number, `count` becomes `1`, satisfying `count >= k`.

Another important case is when a streak is broken after many successful matches. For example, with `value = 4` and `k = 3`, the stream `[4,4,4,3]` should return `false` after the final `3`. A naive implementation that only tracks whether a valid streak was previously achieved could fail here. Our solution resets `count` to `0` immediately when a non-matching number appears.

A third important edge case is when the streak length exceeds `k`. For example, if `k = 2` and the stream is `[7,7,7,7]`, every call after the second should still return `true`. Our implementation correctly supports this because it checks `count >= k` rather than `count == k`.

Another subtle case occurs when the stream never reaches length `k`. The algorithm must continue returning `false` until enough consecutive matches exist. Since the counter only increases on matching values and the condition requires `count >= k`, this behavior is handled automatically without any special-case logic.
