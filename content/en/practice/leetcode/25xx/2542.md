# LeetCode 2542 - Maximum Subsequence Score

## Problem Understanding

This problem asks us to select exactly `k` indices from two arrays, `nums1` and `nums2`, both of length `n`. For the chosen indices, we compute a score using two components:

1. The sum of the selected values from `nums1`
2. The minimum selected value from `nums2`

The final score is:

$$(\text{sum of chosen nums1 values}) \times (\text{minimum chosen nums2 value})$$

Our goal is to maximize this score.

The important detail is that the chosen indices must be the same for both arrays. If we select index `i`, then both `nums1[i]` and `nums2[i]` participate in the calculation.

For example, suppose we choose indices `{0, 2, 3}`:

- The contribution to the sum comes from `nums1`
- The minimum comes from `nums2`

This creates an interesting interaction between the arrays. A large `nums1` value increases the sum, but a small `nums2` value can drastically reduce the multiplier because the minimum determines the entire multiplication factor.

The constraints are very important:

- `n` can be as large as `100000`
- A brute force search over all combinations is impossible
- We need an algorithm close to `O(n log n)`

The problem guarantees:

- Both arrays have the same length
- `k` is always valid, `1 <= k <= n`
- Values are non-negative

Several edge cases are important:

- `k = 1`, where the answer becomes simply `nums1[i] * nums2[i]`
- `k = n`, where all elements must be chosen
- Arrays containing many zeroes
- Large values requiring 64-bit integer handling
- Situations where the optimal solution does not use the largest values from either array individually

## Approaches

### Brute Force Approach

The most direct solution is to try every possible subsequence of size `k`.

For each combination of indices:

1. Compute the sum of selected `nums1` values
2. Compute the minimum selected `nums2` value
3. Calculate the score
4. Track the maximum score

This approach is correct because it exhaustively checks every valid subsequence.

However, the number of subsequences is:

$$\binom{n}{k}$$

This becomes astronomically large when `n = 100000`. Even for much smaller inputs, this approach is computationally infeasible.

### Key Insight for the Optimal Solution

The score formula is:

$$(\text{sum of nums1}) \times (\text{minimum nums2})$$

The crucial observation is that if we decide which element is the minimum `nums2` value in the subsequence, then all other selected elements must have `nums2` values greater than or equal to it.

This suggests sorting pairs `(nums1[i], nums2[i])` by `nums2` in descending order.

While iterating through the sorted list:

- The current `nums2` value becomes the minimum for any subsequence formed using elements seen so far
- We want the largest possible sum of `k` values from `nums1`

To efficiently maintain the best `k` values from `nums1`, we use a min heap.

The heap stores the currently selected `nums1` values:

- If the heap size exceeds `k`, remove the smallest value
- This ensures we always keep the largest possible sum for the current minimum `nums2`

This transforms the problem into a greedy + heap optimization problem.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(C(n, k) * k) | O(k) | Enumerates every subsequence |
| Optimal | O(n log n) | O(k) | Sort + min heap maintains best sum |

## Algorithm Walkthrough

1. Combine the arrays into pairs `(nums2[i], nums1[i])`.

We intentionally place `nums2` first because sorting by `nums2` is the core idea of the algorithm.
2. Sort all pairs in descending order of `nums2`.

After sorting, when we process an element with value `current_nums2`, every previously processed element has `nums2 >= current_nums2`.

This means if we build a subsequence using the current element and previous elements, the minimum `nums2` is guaranteed to be `current_nums2`.
3. Create a min heap to store selected `nums1` values.

The heap allows us to efficiently:

- Add new candidates
- Remove the smallest value when we exceed size `k`

This ensures we always keep the largest possible `k` values from `nums1`.
4. Maintain a running sum of heap values.

Every time we add a value to the heap, add it to `current_sum`.

Every time we remove a value from the heap, subtract it from `current_sum`.
5. Iterate through the sorted pairs.

For each pair:

- Add `nums1` to the heap
- Update the running sum
6. If the heap size exceeds `k`, remove the smallest `nums1`.

This is the greedy step.

We only care about maximizing the sum, so smaller values should be discarded first.
7. Once the heap size becomes exactly `k`, compute a candidate score.

The score is:

$$current\_sum \times current\_nums2$$

Since we process `nums2` in descending order, `current_nums2` is the minimum value among the chosen elements.
8. Track the maximum score across all iterations.

### Why it works

Sorting by descending `nums2` guarantees that during iteration, the current `nums2` value is the smallest possible multiplier for any subsequence formed from processed elements.

For each possible minimum value, the optimal subsequence must maximize the sum of selected `nums1` values. The min heap maintains exactly the largest `k` values available so far.

Therefore, every valid candidate minimum is evaluated together with the best achievable sum for that minimum, guaranteeing the global optimum.

## Python Solution

```python
from typing import List
import heapq

class Solution:
    def maxScore(self, nums1: List[int], nums2: List[int], k: int) -> int:
        pairs = list(zip(nums2, nums1))
        pairs.sort(reverse=True)

        min_heap = []
        current_sum = 0
        best_score = 0

        for current_nums2, current_nums1 in pairs:
            heapq.heappush(min_heap, current_nums1)
            current_sum += current_nums1

            if len(min_heap) > k:
                removed = heapq.heappop(min_heap)
                current_sum -= removed

            if len(min_heap) == k:
                score = current_sum * current_nums2
                best_score = max(best_score, score)

        return best_score
```

The implementation begins by pairing values from the two arrays together. Each pair is stored as `(nums2, nums1)` because sorting by `nums2` is the central operation.

After sorting in descending order, the algorithm iterates through the pairs. Each `nums1` value is pushed into a min heap, and its contribution is added to `current_sum`.

The heap is allowed to temporarily exceed size `k`. When this happens, the smallest `nums1` value is removed. This greedy removal keeps only the largest contributors to the sum.

Whenever the heap size becomes exactly `k`, the current `nums2` value acts as the minimum multiplier. The algorithm computes the score and updates the answer if necessary.

The heap operations each cost `O(log k)`, making the solution efficient enough for the problem constraints.

## Go Solution

```go
package main

import (
	"container/heap"
	"sort"
)

type MinHeap []int

func (h MinHeap) Len() int {
	return len(h)
}

func (h MinHeap) Less(i, j int) bool {
	return h[i] < h[j]
}

func (h MinHeap) Swap(i, j int) {
	h[i], h[j] = h[j], h[i]
}

func (h *MinHeap) Push(x interface{}) {
	*h = append(*h, x.(int))
}

func (h *MinHeap) Pop() interface{} {
	old := *h
	n := len(old)
	value := old[n-1]
	*h = old[:n-1]
	return value
}

func maxScore(nums1 []int, nums2 []int, k int) int64 {
	n := len(nums1)

	pairs := make([][2]int, n)

	for i := 0; i < n; i++ {
		pairs[i] = [2]int{nums2[i], nums1[i]}
	}

	sort.Slice(pairs, func(i, j int) bool {
		return pairs[i][0] > pairs[j][0]
	})

	minHeap := &MinHeap{}
	heap.Init(minHeap)

	var currentSum int64 = 0
	var bestScore int64 = 0

	for _, pair := range pairs {
		currentNums2 := pair[0]
		currentNums1 := pair[1]

		heap.Push(minHeap, currentNums1)
		currentSum += int64(currentNums1)

		if minHeap.Len() > k {
			removed := heap.Pop(minHeap).(int)
			currentSum -= int64(removed)
		}

		if minHeap.Len() == k {
			score := currentSum * int64(currentNums2)

			if score > bestScore {
				bestScore = score
			}
		}
	}

	return bestScore
}
```

The Go implementation follows the same logic as the Python version, but Go requires explicit heap implementation using the `container/heap` package.

One important difference is integer handling. Since the score can become very large, the implementation uses `int64` for `currentSum` and `bestScore` to avoid overflow.

The heap stores integers directly, and Go's interface-based heap API requires type assertions during push and pop operations.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | Sorting dominates, heap operations are O(log k) |
| Space | O(k) | Heap stores at most k elements |

The sorting step costs `O(n log n)`.

During iteration, each element is inserted into the heap once and possibly removed once. Each heap operation costs `O(log k)`, so total heap work is `O(n log k)`.

Since `k <= n`, the overall complexity becomes:

$$O(n \log n)$$

The heap never stores more than `k` elements, so auxiliary space usage is `O(k)`.

## Edge Cases

One important edge case occurs when `k = 1`. In this scenario, the problem reduces to finding the maximum product `nums1[i] * nums2[i]`. A buggy implementation might still try to combine multiple elements or mishandle heap removals. The current implementation handles this naturally because the heap size immediately reaches `k`, allowing every single element to be evaluated independently.

Another important case is when `k = n`. Here, every element must be selected, so there is no freedom in choosing the subsequence. The answer becomes:

$$(\text{sum of all nums1}) \times (\text{minimum of all nums2})$$

Some greedy implementations incorrectly remove elements even when all must remain. The current solution avoids this because the heap only removes elements when its size exceeds `k`.

Large integer values are also significant. Since each array value can reach `100000`, the score can exceed the range of a 32-bit integer. For example:

$$(100000 \times 100000) \times 100000$$

This requires 64-bit arithmetic. The Go solution explicitly uses `int64`, and Python automatically supports arbitrary precision integers.

A subtler edge case appears when the optimal solution does not use the globally largest `nums1` values. Sometimes choosing slightly smaller `nums1` values allows a much larger minimum `nums2`, leading to a better final score. The sorted traversal ensures every possible minimum `nums2` candidate is considered, preventing greedy mistakes based only on maximizing the sum.
