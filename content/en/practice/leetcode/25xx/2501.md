## Solution

## Problem Understanding

The problem gives us an integer array `nums`, and we need to find the length of the longest subsequence that forms a valid square streak.

A subsequence does not need to be contiguous. We may delete elements while preserving the relative order of the remaining elements. However, the definition of a square streak depends on the subsequence after sorting it.

A subsequence is considered a square streak if:

1. Its length is at least `2`
2. After sorting, every number is the square of the previous number

For example, if the sorted subsequence is:

```
[2, 4, 16]
```

then:

```
4 = 2²
16 = 4²
```

so this is a valid square streak of length `3`.

The goal is to return the maximum possible length of such a subsequence. If no valid square streak of length at least `2` exists, we return `-1`.

The constraints are important:

- `nums.length <= 10^5`
- `nums[i] <= 10^5`

These constraints immediately rule out expensive brute-force approaches that examine all subsequences. The number of subsequences grows exponentially, which is completely infeasible for an input size of `100000`.

Another important observation is that repeated squaring grows extremely fast:

```
2 -> 4 -> 16 -> 256 -> 65536
```

After only a few squaring operations, values exceed `10^5`. This property significantly limits the maximum possible chain length.

Several edge cases are important:

- Arrays with no valid square relationship at all
- Duplicate values
- Values whose square exceeds the maximum number present
- Chains of exactly length `2`, which are still valid
- Large inputs where efficiency matters

The problem guarantees all values are positive integers greater than or equal to `2`, so we never need to handle zero, negative numbers, or fractional square roots.

## Approaches

### Brute Force Approach

A brute-force solution would attempt to generate every possible subsequence and check whether it becomes a valid square streak after sorting.

For each subsequence:

1. Sort the subsequence
2. Verify that every element equals the square of the previous one
3. Track the maximum valid length

This approach is correct because it explicitly examines every possible subsequence, so it cannot miss the optimal answer.

However, it is far too slow.

An array of length `n` has `2^n` subsequences. Even for `n = 30`, this is already over one billion possibilities. Since the constraint allows `n = 100000`, brute force is completely impossible.

### Optimal Approach

The key insight is that the square relationship is deterministic.

If we start from a number `x`, the next valid number in the chain must be:

```
x²
```

Then the next one must be:

```
(x²)² = x⁴
```

and so on.

This means every starting number defines at most one possible chain.

Instead of searching all subsequences, we can:

1. Store all numbers in a hash set for O(1) lookup
2. Start from every number
3. Repeatedly square it while the result exists in the set
4. Count the chain length

Because squaring grows very quickly, each chain is very short.

This transforms an exponential problem into a highly efficient linear-style solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n * n log n) | O(n) | Generates all subsequences and validates each one |
| Optimal | O(n log log M) approximately | O(n) | Uses hash set lookups and repeated squaring |

Here, `M` represents the maximum value in the array.

## Algorithm Walkthrough

### Step 1: Store all numbers in a hash set

We place every number into a hash set.

```
number_set = set(nums)
```

This allows constant-time membership checks when determining whether a squared value exists.

Without a hash set, each lookup would require linear search or binary search, making the solution slower.

### Step 2: Initialize the answer

We keep track of the maximum square streak length found so far.

```
longest = -1
```

We initialize with `-1` because the problem requires returning `-1` if no valid streak of length at least `2` exists.

### Step 3: Try every number as a starting point

For every value in the set:

```
for num in number_set:
```

we attempt to build the longest possible square chain beginning from that number.

### Step 4: Repeatedly square the current value

We maintain:

- `current`, the current number in the chain
- `length`, the current chain length

At each step:

```
current = current * current
```

If the squared value exists in the set, we continue the chain.

Otherwise, the chain ends.

### Step 5: Update the maximum answer

If the chain length is at least `2`, it is a valid square streak.

We update the global maximum:

```
longest = max(longest, length)
```

### Step 6: Return the final answer

After processing all possible starting values, return the maximum streak length found.

## Why it works

The algorithm works because a valid square streak is completely determined by its first element.

Once a starting value `x` is chosen, every later value must be:

```
x², x⁴, x⁸, ...
```

There is no branching or ambiguity.

By trying every number as a starting point and repeatedly checking whether the required next square exists, we enumerate every possible valid square streak exactly once. Since we track the maximum valid length, the final answer is correct.

## Python Solution

```python
from typing import List

class Solution:
    def longestSquareStreak(self, nums: List[int]) -> int:
        number_set = set(nums)
        longest = -1

        for num in number_set:
            current = num
            length = 1

            while current * current in number_set:
                current = current * current
                length += 1

            if length >= 2:
                longest = max(longest, length)

        return longest
```

The implementation begins by converting the input array into a hash set. This removes duplicates and provides O(1) average lookup time.

The outer loop considers every distinct number as a possible beginning of a square streak.

Inside the loop, `current` stores the current value in the chain, while `length` tracks how many elements have been included so far.

The `while` loop repeatedly checks whether the square of the current value exists in the set. If it does, we extend the chain and continue.

Once the next square no longer exists, the chain terminates.

If the chain length is at least `2`, we update the best answer found so far.

Finally, we return the longest valid streak length, or `-1` if none exists.

## Go Solution

```go
package main

func longestSquareStreak(nums []int) int {
	numberSet := make(map[int]bool)

	for _, num := range nums {
		numberSet[num] = true
	}

	longest := -1

	for num := range numberSet {
		current := num
		length := 1

		for numberSet[current*current] {
			current = current * current
			length++
		}

		if length >= 2 && length > longest {
			longest = length
		}
	}

	return longest
}
```

The Go implementation follows the same logic as the Python version.

Since Go does not have a built-in set type, we simulate one using a `map[int]bool`.

The algorithm uses integer multiplication directly. Overflow is not an issue here because values exceed the valid range very quickly, and the constraints keep numbers relatively small.

The solution iterates over map keys to process each distinct value exactly once.

Therefore, although the worst-case theoretical analysis could look larger, the practical runtime is extremely efficient.

## Edge Cases

One important edge case occurs when no valid square streak exists. For example:

```
[2,3,5,7]
```

Every number lacks its square in the array, so all chains have length `1`. A buggy implementation might incorrectly return `1`, but the problem requires a minimum streak length of `2`. The solution handles this by initializing the answer to `-1` and only updating it when a chain length reaches at least `2`.

Another important edge case involves duplicate numbers. Consider:

```
[2,2,4,4,16]
```

Duplicates do not create additional square relationships. A naive implementation might repeatedly process identical values or accidentally count duplicates multiple times. By converting the array into a set, the solution naturally eliminates duplicates and processes each distinct value exactly once.

A third important edge case involves extremely rapid growth during squaring. For example:

```
2 -> 4 -> 16 -> 256 -> 65536
```

The next square becomes enormous and exceeds the problem constraints. Some implementations might worry about very deep recursion or long chains, but repeated squaring grows so quickly that chains remain extremely short. The iterative solution safely handles this without recursion or performance problems.
