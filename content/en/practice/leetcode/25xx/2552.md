## Solution

## Problem Understanding

The problem asks us to count the number of increasing quadruplets in a **permutation** array `nums` of length `n`. A quadruplet `(i, j, k, l)` is considered increasing if the indices satisfy `0 <= i < j < k < l < n` and the values satisfy `nums[i] < nums[k] < nums[j] < nums[l]`.

Restated, we are looking for four indices such that the first number is smaller than the third, which is smaller than the second, which is smaller than the fourth. This is not the standard "fully increasing" quadruplet but a specific ordering constraint.

The input `nums` is guaranteed to be a permutation of integers `1` to `n`, meaning every integer appears exactly once. This simplifies certain considerations, such as duplicate counting or sorting. The array size can go up to 4000, so a brute-force approach with four nested loops (`O(n^4)`) would be infeasible.

Edge cases to consider include arrays of minimal size (length 4), arrays where no quadruplet exists because the ordering cannot be satisfied, and arrays with strictly increasing sequences where `nums[j] < nums[k]` will prevent valid quadruplets.

## Approaches

### Brute Force Approach

The brute force solution would iterate over every combination of four indices `(i, j, k, l)` in nested loops. For each quadruplet, we would check whether `nums[i] < nums[k] < nums[j] < nums[l]` holds. While this guarantees correctness, its time complexity is `O(n^4)`, which is unacceptable for `n = 4000`. The space complexity is minimal (`O(1)`), but the approach is impractical due to the large number of iterations.

### Optimal Approach

The key insight is to fix the middle pair `(j, k)` and count how many valid `i` and `l` indices exist for each such pair. Specifically, for each `k`, we can precompute how many values before `k` are smaller than `nums[k]` (potential `i`), and for each `j` after `k`, we count how many values after `j` are greater than `nums[j]` (potential `l`).

Using **prefix counts** or **Binary Indexed Trees (Fenwick Trees)**, we can efficiently compute these counts in `O(n^2)` time rather than `O(n^4)`. Essentially, for each `j` and `k` pair where `k < j`, the number of valid quadruplets contributed is the product of counts of valid `i` before `k` and valid `l` after `j`.

This drastically reduces the time complexity while keeping space usage reasonable.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^4) | O(1) | Four nested loops checking the quadruplet condition |
| Optimal | O(n^2) | O(n) | Precompute prefix counts and suffix counts for efficient counting |

## Algorithm Walkthrough

1. Initialize an array `prefix_smaller` of size `n` where `prefix_smaller[k]` counts how many indices `i < k` have `nums[i] < nums[k]`. This allows fast lookup of valid `i` for a given `k`.
2. Initialize an array `suffix_larger` of size `n` where `suffix_larger[j]` counts how many indices `l > j` have `nums[l] > nums[j]`. This allows fast lookup of valid `l` for a given `j`.
3. Iterate over all possible `k` from `1` to `n-2`. For each `k`, iterate over all `j` such that `k < j < n`.
4. For each pair `(k, j)`, if `nums[k] < nums[j]`, compute the number of quadruplets contributed by multiplying `prefix_smaller[k]` (valid `i`) with `suffix_larger[j]` (valid `l`). Accumulate this into a running total.
5. Return the total count after processing all pairs.

**Why it works:** By precomputing counts of smaller elements before `k` and larger elements after `j`, we ensure that every quadruplet `(i, j, k, l)` satisfies the constraints. Since every `i` is counted only if it is valid for `k`, and every `l` is counted only if valid for `j`, the product gives all possible quadruplets for that pair.

## Python Solution

```python
from typing import List

class Solution:
    def countQuadruplets(self, nums: List[int]) -> int:
        n = len(nums)
        # prefix_smaller[k] counts how many nums[i] < nums[k] for i < k
        prefix_smaller = [0] * n
        for k in range(n):
            count = 0
            for i in range(k):
                if nums[i] < nums[k]:
                    count += 1
            prefix_smaller[k] = count
        
        # suffix_larger[j] counts how many nums[l] > nums[j] for l > j
        suffix_larger = [0] * n
        for j in range(n):
            count = 0
            for l in range(j + 1, n):
                if nums[l] > nums[j]:
                    count += 1
            suffix_larger[j] = count
        
        total = 0
        # iterate over middle pair (k, j) with k < j
        for j in range(1, n - 1):
            for k in range(j):
                if nums[k] < nums[j]:
                    total += prefix_smaller[k] * suffix_larger[j]
        
        return total
```

The Python implementation follows the algorithm closely. The nested loops for `prefix_smaller` and `suffix_larger` precompute valid counts efficiently. Then, iterating over each middle pair `(k, j)` ensures that every valid quadruplet is counted without missing any cases.

## Go Solution

```go
func countQuadruplets(nums []int) int64 {
    n := len(nums)
    prefixSmaller := make([]int, n)
    for k := 0; k < n; k++ {
        count := 0
        for i := 0; i < k; i++ {
            if nums[i] < nums[k] {
                count++
            }
        }
        prefixSmaller[k] = count
    }

    suffixLarger := make([]int, n)
    for j := 0; j < n; j++ {
        count := 0
        for l := j + 1; l < n; l++ {
            if nums[l] > nums[j] {
                count++
            }
        }
        suffixLarger[j] = count
    }

    var total int64 = 0
    for j := 1; j < n-1; j++ {
        for k := 0; k < j; k++ {
            if nums[k] < nums[j] {
                total += int64(prefixSmaller[k] * suffixLarger[j])
            }
        }
    }

    return total
}
```

The Go version mirrors the Python logic. We use slices for `prefixSmaller` and `suffixLarger` and accumulate the total in `int64` to avoid overflow. Go requires explicit type conversions when multiplying integers and summing into `int64`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n^2) | Two nested loops over `k` and `j`, plus precomputation of prefix and suffix counts. |
| Space | O(n) | Arrays `prefix_smaller` and `suffix_larger` of length `n`. |

The algorithm is efficient because it avoids the naive `O(n^4)` search by using precomputed counts, reducing the problem to `O(n^2)`.
