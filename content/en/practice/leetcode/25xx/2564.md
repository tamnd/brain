## Solution

## Problem Understanding

The problem gives us a binary string `s` and a list of queries. Each query contains two integers, `first` and `second`. For every query, we must find a substring of `s` whose decimal value `val` satisfies:

$val \oplus first = second$

If such a substring exists, we return its starting and ending indices. If multiple valid substrings exist, we must choose the shortest one. If there are still multiple answers with the same minimum length, we choose the one with the smallest left index.

The important observation is that the equation can be rearranged as:

$val = first \oplus second$

This means that each query is really asking:

> Does the binary string `s` contain a substring whose decimal value equals `first ^ second`?

If yes, we return the shortest such substring.

The input size is large:

- `s.length <= 10^4`
- `queries.length <= 10^5`

A naive solution that scans all substrings for every query would be far too slow because a string of length `10^4` contains roughly `5 * 10^7` substrings.

Another important detail is that query values can be as large as `10^9`, which means their binary representation contains at most 30 bits. This becomes the key optimization opportunity.

There are several edge cases that can easily cause bugs:

- The target value may be `0`, which corresponds to substring `"0"`.
- Leading zeros matter for substring length. For example, `"001"` and `"1"` both represent decimal `1`, but `"1"` is shorter and therefore preferred.
- Multiple substrings may represent the same value. We must keep the shortest one, and among equal lengths, the leftmost one.
- Some queries may have no matching substring at all.

The constraints strongly suggest preprocessing all relevant substring values once, then answering each query in constant time.

## Approaches

### Brute Force Approach

A direct solution would enumerate every possible substring of `s`, convert it into its decimal value, and then check each query to see whether that value matches `first ^ second`.

There are `O(n^2)` substrings in a string of length `n`. Converting each substring into an integer may take another `O(n)` time in the worst case, although this can be optimized incrementally.

Even with incremental binary conversion, we would still need to compare against all queries or store all substring values. The total runtime becomes prohibitively expensive for `n = 10^4`.

The brute force method is correct because it explicitly checks every substring, so it cannot miss any valid answer. However, the quadratic number of substrings makes it infeasible.

### Optimal Approach

The key insight is that query targets are bounded by `10^9`.

A number up to `10^9` requires at most 30 binary bits. Therefore, any useful substring has length at most 30.

This dramatically reduces the search space.

Instead of generating all `O(n^2)` substrings, we only generate substrings of length up to 30 starting from every position. That gives at most:

$30n$

possible substrings.

For each substring, we compute its decimal value incrementally and store the shortest occurrence of that value in a hash map.

Then each query becomes:

1. Compute `target = first ^ second`
2. Look up `target` in the hash map
3. Return the stored indices if present

This preprocessing transforms expensive repeated query work into fast hash lookups.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n² × q) or worse | O(1) | Enumerates all substrings for every query |
| Optimal | O(30n + q) | O(number of distinct values) | Precomputes all relevant substring values |

## Algorithm Walkthrough

1. Create a hash map called `value_to_indices`.

This map stores the best substring for every decimal value encountered. The key is the decimal value, and the value is the pair `[left, right]`.
2. Iterate through every starting position in the string.

Suppose the current starting index is `i`.
3. Build substring values incrementally.

Initialize `current_value = 0`.

Then extend the substring one character at a time from position `i` up to at most 30 characters.

For each new character:

$current\_value = current\_value \times 2 + bit$

where `bit` is either `0` or `1`.
4. Stop after length 30.

Any longer binary substring would exceed the maximum possible query target size, so it can never be useful.
5. Store the shortest substring for each value.

If the value has not been seen before, store `[i, j]`.

If it already exists, compare lengths:

- Replace only if the new substring is shorter
- If lengths are equal, keep the earlier one

Since we iterate left to right and lengths grow naturally, simply storing the first occurrence is sufficient.
6. Handle zero correctly.

The substring `"0"` always represents decimal value `0`.

Longer substrings like `"00"` are never preferable because `"0"` is shorter.
7. Process each query independently.

For query `[first, second]`, compute:

$target = first \oplus second$
8. Look up the target value.

- If the target exists in the hash map, append its indices
- Otherwise append `[-1, -1]`
9. Return the final answers array.

### Why it works

Every valid answer corresponds to a substring value equal to `first ^ second`. Since every useful value fits within 30 bits, enumerating all substrings up to length 30 guarantees that every candidate substring is considered.

The preprocessing step stores the shortest occurrence for each value. Therefore, when a query asks for a target value, the stored substring is guaranteed to satisfy both correctness and minimality requirements.

## Python Solution

```python
from typing import List

class Solution:
    def substringXorQueries(self, s: str, queries: List[List[int]]) -> List[List[int]]:
        value_to_indices = {}

        n = len(s)

        for left in range(n):
            current_value = 0

            # Only need up to 30 bits
            for right in range(left, min(left + 30, n)):
                current_value = (current_value << 1) | int(s[right])

                if current_value not in value_to_indices:
                    value_to_indices[current_value] = [left, right]

                # Optimization:
                # If current value is 0, extending further only creates longer
                # representations of 0, which are never optimal.
                if current_value == 0:
                    break

        answer = []

        for first, second in queries:
            target = first ^ second

            if target in value_to_indices:
                answer.append(value_to_indices[target])
            else:
                answer.append([-1, -1])

        return answer
```

The implementation begins by preprocessing all relevant substring values. The dictionary `value_to_indices` maps a decimal value to the best substring producing that value.

For every starting position, we incrementally construct binary values using left shifts. This avoids repeatedly converting substrings into integers from scratch.

The inner loop only explores up to 30 characters because larger binary numbers cannot match any query target under the constraints.

When a value appears for the first time, we store its substring boundaries. Since substrings are explored in increasing length order and from left to right, the first occurrence automatically satisfies the shortest and leftmost requirements.

The special optimization for `0` is important. Once we encounter a substring value of zero, extending it further only creates longer zero-valued substrings, which can never improve the answer.

Finally, each query becomes a simple XOR computation followed by a dictionary lookup.

## Go Solution

```go
func substringXorQueries(s string, queries [][]int) [][]int {
	valueToIndices := make(map[int][]int)

	n := len(s)

	for left := 0; left < n; left++ {
		currentValue := 0

		for right := left; right < n && right < left+30; right++ {
			currentValue = (currentValue << 1) | int(s[right]-'0')

			if _, exists := valueToIndices[currentValue]; !exists {
				valueToIndices[currentValue] = []int{left, right}
			}

			// Longer zero substrings are never optimal
			if currentValue == 0 {
				break
			}
		}
	}

	answer := make([][]int, 0, len(queries))

	for _, query := range queries {
		target := query[0] ^ query[1]

		if indices, exists := valueToIndices[target]; exists {
			answer = append(answer, indices)
		} else {
			answer = append(answer, []int{-1, -1})
		}
	}

	return answer
}
```

The Go implementation follows the same logic as the Python version.

The hash map uses `map[int][]int` to associate values with substring boundaries. Binary values are built incrementally using bit shifts.

Go strings are byte arrays, so characters are converted with:

```go
int(s[right] - '0')
```

Slices are used for storing the answer pairs. Since Go does not have Python's automatic tuple handling, we explicitly create slices like `[]int{left, right}`.

Integer overflow is not a concern because all values remain below `2^30`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(30n + q) | At most 30 substrings explored per starting index, plus constant time query lookups |
| Space | O(30n) in worst case | Hash map stores substring positions for distinct values |

The key optimization is limiting substring length to 30. Without that restriction, the algorithm would become quadratic.

For each index in the string, we examine at most 30 substrings, making preprocessing effectively linear. Each query is answered in constant expected time using a hash map lookup.

## Edge Cases

One important edge case is handling the value zero. Binary strings like `"0"`, `"00"`, and `"000"` all represent decimal zero. A naive implementation might incorrectly store a longer substring even though the problem requires the shortest one. The implementation handles this correctly by immediately breaking when the current value becomes zero, ensuring only the shortest `"0"` substring is stored.

Another subtle case involves leading zeros. For example, the substrings `"001"` and `"1"` both represent decimal value `1`. The correct answer must be the shorter substring `"1"`. Since the algorithm processes substrings in increasing length order and only stores the first occurrence of a value, the shortest representation is automatically preserved.

A third important edge case occurs when multiple identical shortest substrings exist. Suppose two different substrings both represent the same value with equal length. The problem requires choosing the one with the smallest left index. Because the algorithm scans from left to right and never overwrites an existing entry, the earliest valid substring is retained automatically.

Finally, queries may request values that do not appear anywhere in the string. Instead of attempting complicated recovery logic, the implementation simply performs a hash map lookup and returns `[-1, -1]` if the target is absent. This guarantees clean handling of impossible cases.
