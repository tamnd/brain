## Solution

## Problem Understanding

The problem gives us an undirected graph with `n` nodes and a list of edges. Every edge connects two distinct nodes, and there are no duplicate edges in the input.

Our task is to determine whether we can make the degree of every node even by adding at most two new edges. The new edges must satisfy two restrictions:

1. We cannot add a self-loop, meaning an edge from a node to itself.
2. We cannot add an edge that already exists in the graph.

The graph may already have all even degrees, in which case the answer is immediately `true` because we are allowed to add zero edges.

The key observation is that adding an edge changes the parity of exactly two nodes. If a node currently has odd degree, adding one edge connected to it makes its degree even. Likewise, connecting to an even-degree node makes it odd.

This means we only care about nodes with odd degree.

From graph theory, the number of odd-degree nodes in an undirected graph is always even. Therefore, the number of odd-degree nodes can only be:

- `0`
- `2`
- `4`
- `6`, `8`, and so on

Since we may add at most two edges, and each edge can fix at most two odd nodes, we can only handle cases where the number of odd-degree nodes is at most `4`.

The constraints are large:

- `n <= 10^5`
- `edges.length <= 10^5`

This immediately rules out expensive graph operations such as trying all possible pairs of nodes naively with adjacency matrices or exhaustive search over all edge combinations.

Efficient adjacency lookup is important because we frequently need to answer questions like:

"Does an edge already exist between node `u` and node `v`?"

A hash-set based adjacency representation is therefore ideal.

Several edge cases are important:

- The graph may already satisfy the condition.
- Two odd nodes may already be connected, preventing a direct fix.
- Four odd nodes may require careful pairing.
- There may be no valid intermediate node available for connecting two odd nodes through two edges.
- The graph can be disconnected, but parity reasoning still works independently of connectivity.

## Approaches

### Brute Force Approach

A brute force solution would try every possible way to add zero, one, or two edges and then check whether all node degrees become even.

For each candidate set of added edges, we would:

1. Verify that the edges are not duplicates and not self-loops.
2. Temporarily update node degrees.
3. Check whether every node has even degree.

The number of possible edges in a graph is roughly `O(n^2)`. Trying all combinations of up to two edges would require examining approximately `O(n^4)` possibilities in the worst case.

With `n = 10^5`, this is completely infeasible.

The brute force approach is correct because it explicitly tests all legal possibilities, but it is far too slow for the given constraints.

### Optimal Approach

The crucial observation is that only odd-degree nodes matter.

Adding one edge flips the parity of exactly two nodes. Therefore:

- `0` odd nodes → already valid
- `2` odd nodes → possibly solvable with one or two edges
- `4` odd nodes → possibly solvable with exactly two edges
- More than `4` odd nodes → impossible

This dramatically reduces the search space.

For the `2` odd-node case:

- If the two odd nodes are not connected, add one edge directly.
- Otherwise, try to connect each odd node to some intermediate node that is connected to neither.

For the `4` odd-node case:

- There are only three possible pairings of four nodes into two edges.
- We simply test whether either pairing uses only non-existing edges.

Efficient edge existence checks are handled using adjacency hash sets.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^4) | O(n^2) | Tries all possible edge additions |
| Optimal | O(n + m) | O(n + m) | Uses odd-degree parity observations |

Here, `m` is the number of edges.

## Algorithm Walkthrough

1. Build the graph representation.

We create an adjacency set for every node. This allows constant average time lookup to determine whether an edge already exists.

We also maintain a degree array to count how many edges touch each node.
2. Compute all odd-degree nodes.

We iterate through nodes `1` to `n` and collect nodes whose degree is odd.

Since every added edge changes parity for exactly two nodes, only these nodes matter.
3. Handle the case where there are no odd-degree nodes.

If the list of odd nodes is empty, the graph already satisfies the requirement. We return `true`.
4. Handle impossible cases.

If there are more than four odd-degree nodes, return `false`.

Two added edges can fix at most four odd nodes because each edge changes parity for exactly two endpoints.
5. Handle the case with exactly two odd-degree nodes.

Let the nodes be `a` and `b`.

First, check whether edge `(a, b)` already exists.

- If it does not exist, we can add it directly and both degrees become even.
- If it already exists, we must use two edges.

To use two edges, we search for some intermediate node `x` such that:

- `x` is not connected to `a`
- `x` is not connected to `b`

Then we can add edges `(a, x)` and `(b, x)`.

Both odd nodes become even, and node `x` changes parity twice, so it remains even.
6. Handle the case with exactly four odd-degree nodes.

Let the nodes be `a`, `b`, `c`, and `d`.

There are only three possible pairings:

- `(a, b)` and `(c, d)`
- `(a, c)` and `(b, d)`
- `(a, d)` and `(b, c)`

For each pairing, verify that neither edge already exists.

If any pairing is valid, return `true`.
7. If none of the valid constructions work, return `false`.

### Why it works

The correctness comes from parity properties in undirected graphs.

Each added edge flips the parity of exactly two nodes. Therefore, only odd-degree nodes need modification. Since we may add at most two edges, we can affect at most four odd nodes.

The algorithm exhaustively checks every possible valid parity correction for the only feasible cases, namely `0`, `2`, and `4` odd nodes. Because every legal construction must fall into one of these patterns, the algorithm is complete and correct.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Solution:
    def isPossible(self, n: int, edges: List[List[int]]) -> bool:
        graph = defaultdict(set)
        degree = [0] * (n + 1)

        for u, v in edges:
            graph[u].add(v)
            graph[v].add(u)
            degree[u] += 1
            degree[v] += 1

        odd_nodes = []

        for node in range(1, n + 1):
            if degree[node] % 2 == 1:
                odd_nodes.append(node)

        if len(odd_nodes) == 0:
            return True

        if len(odd_nodes) > 4:
            return False

        def can_connect(u: int, v: int) -> bool:
            return v not in graph[u]

        if len(odd_nodes) == 2:
            a, b = odd_nodes

            if can_connect(a, b):
                return True

            for node in range(1, n + 1):
                if (
                    node != a
                    and node != b
                    and can_connect(a, node)
                    and can_connect(b, node)
                ):
                    return True

            return False

        a, b, c, d = odd_nodes

        pairings = [
            ((a, b), (c, d)),
            ((a, c), (b, d)),
            ((a, d), (b, c)),
        ]

        for (u1, v1), (u2, v2) in pairings:
            if can_connect(u1, v1) and can_connect(u2, v2):
                return True

        return False
```

The implementation starts by constructing an adjacency set representation of the graph. This gives efficient edge existence checks, which are central to the solution.

The `degree` array tracks how many edges touch each node. After processing all edges, we collect all nodes whose degree is odd.

The helper function `can_connect(u, v)` checks whether an edge can legally be added between two nodes. Since self-loops are never attempted in this algorithm, we only need to ensure the edge does not already exist.

The logic then branches based on the number of odd-degree nodes.

For two odd nodes, we first attempt the simplest fix by connecting them directly. If that edge already exists, we search for an intermediate node that can connect to both.

For four odd nodes, we test all three valid pairings. Since there are only three possibilities, exhaustive checking is constant time.

If no valid construction exists, the function returns `False`.

## Go Solution

```go
package main

func isPossible(n int, edges [][]int) bool {
	graph := make([]map[int]bool, n+1)
	degree := make([]int, n+1)

	for i := 1; i <= n; i++ {
		graph[i] = make(map[int]bool)
	}

	for _, edge := range edges {
		u, v := edge[0], edge[1]

		graph[u][v] = true
		graph[v][u] = true

		degree[u]++
		degree[v]++
	}

	oddNodes := []int{}

	for i := 1; i <= n; i++ {
		if degree[i]%2 == 1 {
			oddNodes = append(oddNodes, i)
		}
	}

	if len(oddNodes) == 0 {
		return true
	}

	if len(oddNodes) > 4 {
		return false
	}

	canConnect := func(u, v int) bool {
		return !graph[u][v]
	}

	if len(oddNodes) == 2 {
		a, b := oddNodes[0], oddNodes[1]

		if canConnect(a, b) {
			return true
		}

		for node := 1; node <= n; node++ {
			if node != a &&
				node != b &&
				canConnect(a, node) &&
				canConnect(b, node) {
				return true
			}
		}

		return false
	}

	a, b, c, d := oddNodes[0], oddNodes[1], oddNodes[2], oddNodes[3]

	pairings := [][4]int{
		{a, b, c, d},
		{a, c, b, d},
		{a, d, b, c},
	}

	for _, p := range pairings {
		u1, v1 := p[0], p[1]
		u2, v2 := p[2], p[3]

		if canConnect(u1, v1) && canConnect(u2, v2) {
			return true
		}
	}

	return false
}
```

The Go solution follows exactly the same algorithmic structure as the Python version.

The adjacency representation uses `map[int]bool` for each node because Go does not have a built-in hash-set type. Presence in the map indicates an existing edge.

Go slices are used for the degree array and odd-node collection. Since node indices are 1-based, arrays are allocated with size `n + 1`.

No integer overflow concerns exist because all counts remain well within Go's `int` range under the problem constraints.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n + m) | Build graph and scan nodes once |
| Space | O(n + m) | Adjacency sets and degree storage |

The graph construction processes each edge once, giving `O(m)` time. Collecting odd-degree nodes requires scanning all nodes once, giving `O(n)` time.

The remaining logic operates on at most four odd nodes, so it is constant time relative to input size.

The adjacency representation stores every undirected edge twice, once for each endpoint, so total storage remains linear in the graph size.

## Edge Cases

One important edge case occurs when the graph already has all even degrees. A naive implementation might still attempt to add edges unnecessarily. The correct behavior is to immediately return `true` because adding zero edges is allowed. The implementation handles this by checking whether the odd-node list is empty.

Another subtle case occurs when there are exactly two odd nodes, but they are already connected by an edge. In that situation, we cannot directly connect them again because duplicate edges are forbidden. A buggy solution might incorrectly return `false` immediately. The correct approach is to search for an intermediate node that can connect to both odd nodes using two new edges. The implementation explicitly performs this search.

A third important case involves exactly four odd nodes. Some implementations incorrectly assume that any four odd nodes are solvable with two edges. This is not always true because existing edges may block all valid pairings. The implementation correctly tests all three possible pairings and only returns `true` if at least one pairing uses non-existing edges.

A final edge case appears in disconnected graphs. Connectivity does not matter for this problem because only node degree parity matters. The implementation never assumes the graph is connected, so disconnected components are naturally handled correctly.
