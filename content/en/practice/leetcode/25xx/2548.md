# LeetCode 2548 - Maximum Price to Fill a Bag

## Problem Understanding

This problem describes a variation of the classic fractional knapsack problem. We are given a collection of items, where each item has two properties:

- `price[i]`, the value we gain by taking the whole item
- `weight[i]`, the amount of capacity it consumes

We are also given a bag with a fixed capacity.

Unlike the standard 0/1 knapsack problem, every item can be divided into fractional parts. If we split an item into two portions, both the weight and the price are divided proportionally. For example, taking half of an item means taking half its weight and gaining half its price.

The goal is to exactly fill the bag with total weight equal to `capacity` while maximizing the total price collected.

If the total available weight from all items is smaller than `capacity`, then it is impossible to fill the bag completely, so we must return `-1`.

The constraints are large:

- Up to `10^5` items
- Item values and weights up to `10^4`
- Capacity up to `10^9`

These limits immediately tell us that dynamic programming approaches based on capacity are impossible because the capacity can be extremely large. We need an efficient greedy strategy.

Several edge cases are important:

- The total weight may be less than the bag capacity, making the answer impossible.
- A single item may need to be partially taken.
- Multiple items can have identical value-to-weight ratios.
- The optimal solution may require taking only a fraction of the final item.
- The bag must be filled exactly, not approximately.

Because fractional splitting is allowed, the problem behaves very differently from the classic integer knapsack problem.

## Approaches

### Brute Force Approach

A brute force approach would try all possible ways to distribute portions of items into the bag.

Since each item can be split continuously, the number of possible combinations is effectively infinite. Even if we discretized fractions into smaller units, the state space would explode exponentially.

One possible brute force interpretation is:

1. Generate all subsets and fractional allocations.
2. Compute their total weight and total value.
3. Keep the best valid configuration.

This guarantees correctness because every possible allocation is explored. However, it is computationally infeasible.

Even a dynamic programming approach based on weight does not work here because the capacity can reach `10^9`.

### Key Insight

The crucial observation is that this is exactly a fractional knapsack problem.

For fractional knapsack, the optimal strategy is always:

- Take items with the highest value-per-weight ratio first.
- Continue until the bag is full.
- If the next item does not fully fit, take only the fraction needed.

Why does this greedy strategy work?

Suppose we ever took a lower ratio item before a higher ratio item. Swapping some weight from the lower ratio item with the higher ratio item would strictly increase total value. Therefore, any optimal solution must prioritize higher ratios first.

So the algorithm becomes:

1. Compute `price / weight` for every item.
2. Sort items by this ratio in descending order.
3. Greedily take as much as possible from each item.

Before doing this, we should verify that the total available weight is at least `capacity`. Otherwise, filling the bag exactly is impossible.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Explores all possible allocations |
| Optimal | O(n log n) | O(n) | Greedy fractional knapsack using sorting |

## Algorithm Walkthrough

1. Compute the total available weight across all items.

This step determines whether it is even possible to fill the bag. If the sum of all weights is smaller than `capacity`, we immediately return `-1`.
2. For every item, compute its value density.

The value density is:

$$\text{ratio} = \frac{\text{price}}{\text{weight}}$$

This tells us how much value we gain per unit of weight. Items with larger ratios are always more desirable.
3. Sort all items in descending order of ratio.

After sorting, the best items appear first. This allows the greedy strategy to always make the locally optimal choice.
4. Iterate through the sorted items.

For each item:

- If the entire item fits into the remaining capacity, take it completely.
- Otherwise, take only the fraction needed to fill the remaining capacity.
5. Update the answer and remaining capacity.

Each time we take some amount of weight:

- Add the corresponding value contribution.
- Reduce the remaining capacity.
6. Stop once the bag becomes full.

Since we process items in descending ratio order, the first moment we fill the bag gives the maximum possible total value.

### Why it works

The correctness comes from the exchange argument used in fractional knapsack problems.

If a solution ever uses weight from a lower value-density item while some higher value-density item remains unused, we can swap those weights and increase the total price. Therefore, any optimal solution must prioritize higher ratios first.

Because items are divisible, this greedy ordering is always globally optimal.

## Python Solution

```python
from typing import List

class Solution:
    def maxPrice(self, items: List[List[int]], capacity: int) -> float:
        total_weight = sum(weight for _, weight in items)

        if total_weight < capacity:
            return -1.0

        # Sort by value density in descending order
        items.sort(key=lambda item: item[0] / item[1], reverse=True)

        remaining_capacity = capacity
        total_price = 0.0

        for price, weight in items:
            if remaining_capacity == 0:
                break

            if weight <= remaining_capacity:
                total_price += price
                remaining_capacity -= weight
            else:
                fraction = remaining_capacity / weight
                total_price += price * fraction
                remaining_capacity = 0

        return total_price
```

The implementation begins by computing the total available weight. This is an important feasibility check because the bag must be filled exactly.

Next, the items are sorted according to their price-to-weight ratio. Python's `sort()` method efficiently handles this in `O(n log n)` time.

The greedy traversal then processes items from highest ratio to lowest ratio. If an item completely fits into the remaining capacity, we take it entirely. Otherwise, we take only the exact fraction needed to fill the bag.

The answer is stored as a floating-point value because fractional items can produce non-integer results.

## Go Solution

```go
package main

import (
	"sort"
)

func maxPrice(items [][]int, capacity int) float64 {
	totalWeight := 0

	for _, item := range items {
		totalWeight += item[1]
	}

	if totalWeight < capacity {
		return -1.0
	}

	sort.Slice(items, func(i, j int) bool {
		return float64(items[i][0])/float64(items[i][1]) >
			float64(items[j][0])/float64(items[j][1])
	})

	remainingCapacity := capacity
	totalPrice := 0.0

	for _, item := range items {
		price := item[0]
		weight := item[1]

		if remainingCapacity == 0 {
			break
		}

		if weight <= remainingCapacity {
			totalPrice += float64(price)
			remainingCapacity -= weight
		} else {
			fraction := float64(remainingCapacity) / float64(weight)
			totalPrice += float64(price) * fraction
			remainingCapacity = 0
		}
	}

	return totalPrice
}
```

The Go implementation follows the same logic as the Python version.

The main difference is the use of `sort.Slice` for custom sorting. Since Go performs integer division by default, all ratio computations must explicitly convert values to `float64`.

The final answer is also stored as `float64` to correctly handle fractional contributions.

The extra space usage is minimal outside the sorting process itself. Python's sorting implementation may use additional auxiliary memory internally.

## Edge Cases

### Total Available Weight Is Too Small

One of the most important edge cases occurs when the combined weight of all items is smaller than the required capacity.

A naive implementation might greedily take everything and return the accumulated value, even though the bag is not completely full. That would violate the problem requirements.

This implementation explicitly checks total weight before starting the greedy process. If the total is insufficient, it immediately returns `-1.0`.

### Final Item Must Be Partially Taken

Many bugs occur when handling the last partially selected item.

For example, if only `3` units of capacity remain but the current item weighs `10`, we must correctly compute the proportional contribution:

$$\text{fraction} = \frac{3}{10}$$

The implementation carefully computes the fraction using floating-point division and adds only the proportional value.

### Multiple Items Have the Same Ratio

If two items have identical value densities, either order produces the same optimal result.

Some incorrect solutions assume a unique ordering is required. In reality, the greedy proof depends only on ratio ordering, not on tie-breaking.

The implementation handles this naturally because sorting stability does not affect correctness when ratios are equal.
