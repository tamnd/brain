## Solution

## Problem Understanding

The problem presents a **0-indexed integer array `nums`** and allows you to **rearrange its elements in any order**. After rearranging, we define the **prefix sum array** `prefix` such that `prefix[i]` is the sum of the first `i+1` elements of the rearranged array. The **score** of the array is defined as the **count of positive integers** in the `prefix` array. The task is to **maximize this score**.

In simpler terms, we want to reorder the array so that as many **prefix sums as possible are positive**. The input array can be large, up to 105 elements, and elements themselves can range from -10^6 to 10^6. These constraints suggest that any solution that attempts to generate all permutations is infeasible. Edge cases include arrays with all negative numbers, all zeros, or combinations of large positives and large negatives, which can affect prefix sums in subtle ways.

## Approaches

The **brute-force approach** would involve generating **all permutations of `nums`**, computing the prefix sums for each, and counting the number of positive sums. While this guarantees correctness, it is **exponentially slow**, O(n!) time complexity, which is impractical for n up to 10^5.

The **key insight for an optimal solution** is greedy: **larger numbers should appear earlier in the array**. By placing large positive numbers at the start, the prefix sum grows quickly and can absorb later negative values without turning negative. Sorting the array in **descending order** and then iterating while maintaining a running sum ensures that we **maximize the number of positive prefix sums**. If the running sum becomes non-positive, we stop counting because any subsequent additions cannot increase the prefix sum beyond zero without additional positives. This strategy leverages the fact that the prefix sum is cumulative.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n!) | O(n) | Generate all permutations, compute prefix sums |
| Optimal | O(n log n) | O(1) | Sort descending, count positive cumulative sums greedily |

## Algorithm Walkthrough

1. **Sort the array in descending order**. This ensures the largest positive elements contribute to the earliest prefix sums, maximizing the chance that each prefix sum remains positive.
2. **Initialize a running sum variable** to zero. This will keep track of the cumulative sum of elements as we iterate.
3. **Initialize a score counter** to zero. This will count how many prefix sums are positive.
4. **Iterate through the sorted array**:

- Add the current element to the running sum.
- If the running sum is positive, increment the score counter.
- If the running sum becomes non-positive, we stop counting because adding subsequent elements in descending order will not increase the prefix sum to positive again.
5. **Return the score counter** as the maximum achievable score.

**Why it works**: By sorting the array in descending order, we guarantee that each prefix sum is as large as possible at each step. Since the prefix sum is cumulative, any negative number appearing later will reduce the running sum, but the greedy approach ensures it only reduces it after maximizing the number of positives.

## Python Solution

```python
from typing import List

class Solution:
    def maxScore(self, nums: List[int]) -> int:
        nums.sort(reverse=True)
        score = 0
        running_sum = 0
        for num in nums:
            running_sum += num
            if running_sum > 0:
                score += 1
            else:
                break
        return score
```

In this implementation, the array is sorted in descending order so that the largest numbers contribute to early prefix sums. The `running_sum` tracks the cumulative sum, and the `score` counts how many prefix sums remain positive. The loop breaks as soon as the sum is no longer positive, since further additions will not improve the score.

## Go Solution

```go
import "sort"

func maxScore(nums []int) int {
    sort.Slice(nums, func(i, j int) bool {
        return nums[i] > nums[j]
    })
    score := 0
    runningSum := 0
    for _, num := range nums {
        runningSum += num
        if runningSum > 0 {
            score++
        } else {
            break
        }
    }
    return score
}
```

In Go, the `sort.Slice` function is used to sort the slice in descending order. The rest of the logic mirrors the Python implementation, maintaining a running sum and counting positive prefix sums.

## Worked Examples

**Example 1**: `nums = [2,-1,0,1,-3,3,-3]`

1. Sort descending: `[3,2,1,0,-1,-3,-3]`
2. Running sum and score:

- 3 → running sum 3 → score 1
- 2 → running sum 5 → score 2
- 1 → running sum 6 → score 3
- 0 → running sum 6 → score 4
- -1 → running sum 5 → score 5
- -3 → running sum 2 → score 6
- -3 → running sum -1 → stop

Result: score = 6

**Example 2**: `nums = [-2,-3,0]`

1. Sort descending: `[0,-2,-3]`
2. Running sum and score:

- 0 → running sum 0 → not positive → stop

Result: score = 0

## Edge Cases

One important edge case is an array with **all negative numbers**. In this scenario, any prefix sum is non-positive, so the score must be zero. The algorithm handles this by stopping immediately when the running sum is non-positive.

A second edge case is an array of **all zeros**. Here, the prefix sums are zero at each step, and the score should also be zero. The algorithm treats zero as non-positive and breaks the loop immediately, producing the correct score.

A third edge case involves a **large positive number followed by multiple negatives**. The algorithm ensures that the initial positive dominates the prefix sum and maximizes the number of positive sums. If negatives appear later, they may reduce the running sum but do not reduce the already counted positive prefix sums. This greedy ordering guarantees optimality.
