## Solution

## Problem Understanding

The problem asks us to take an input array of positive integers and transform it into a single flattened array of digits, preserving the original order of both the numbers and the digits inside each number.

In other words, for each integer in the input list, we decompose it into its individual decimal digits and append those digits into a result array in the same sequence they appear when reading the number from left to right. The final output is a single contiguous list of digits formed by concatenating these per-number digit expansions.

The input constraints are relatively small, with at most 1000 integers and each integer up to 100,000. This implies that each number has at most 6 digits, so the total output size is bounded by about 6000 digits. This strongly suggests that any solution with linear complexity in the number of digits will be efficient enough.

The key guarantee is that all numbers are positive integers, meaning there is no need to handle zero or negative edge cases. There are no leading zero concerns except those naturally arising from digit extraction, which we do not need to preserve beyond the value itself.

Important edge cases include single-digit numbers, numbers with repeated digits, and the smallest possible input size of a single-element array. Another subtle point is ensuring that digit order is preserved correctly, since naive extraction using modulo arithmetic produces digits in reverse order unless corrected.

## Approaches

The brute-force approach conceptually treats each number independently, converts it into a string or repeatedly extracts digits using modulo operations, and appends each digit into a result list. This is already efficient enough given constraints, but we can still distinguish a less optimal conceptual version where we repeatedly rebuild intermediate structures or concatenate strings inefficiently inside loops.

A more optimal and standard approach is to process each number once, extract digits in correct order, and append them directly into a result list. The cleanest method is to convert each number into its string representation and iterate through characters, converting each back into an integer digit. Alternatively, we can use arithmetic extraction with division and modulo, but this requires reversing the digit order after extraction.

The string-based approach is preferred here because it is simpler, less error-prone, and naturally preserves order without additional manipulation.

### Comparison of Approaches

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force (repeated reconstruction or inefficient concatenation) | O(n * k^2) | O(n * k) | Converts numbers to strings or digits repeatedly with extra overhead |
| Optimal (direct digit extraction) | O(n * k) | O(n * k) | Single pass over each digit, clean append into result |

Here, n is the number of elements in `nums`, and k is the average number of digits per number.

## Algorithm Walkthrough

The optimal algorithm proceeds by iterating through each number in the input array and extracting its digits in order.

1. Initialize an empty result list that will store all digits from all numbers in sequence. This list is the final output container and grows dynamically as we process each number.
2. Iterate over each number in the input array. The goal at this step is to isolate each integer so we can decompose it independently without affecting others.
3. Convert the current number into its string representation. This step is used because string iteration preserves digit order naturally, avoiding the complexity of reversing digits when using arithmetic operations.
4. Iterate through each character in the string representation of the number. Each character corresponds to a single digit in the original number.
5. Convert each character back into an integer and append it to the result list. This maintains the correct numeric format expected in the output.
6. After processing all numbers, return the result list, which now contains all digits in the required order.

### Why it works

The correctness relies on the fact that converting an integer to a string preserves the exact left-to-right ordering of its decimal digits. By iterating over each character and appending it immediately to the output list, we preserve both intra-number order (digits within a number) and inter-number order (sequence of numbers in the input array). Since every digit is processed exactly once, no information is lost or reordered incorrectly.

## Python Solution

```python
from typing import List

class Solution:
    def separateDigits(self, nums: List[int]) -> List[int]:
        result = []
        
        for num in nums:
            for ch in str(num):
                result.append(int(ch))
        
        return result
```

The implementation is straightforward and directly follows the algorithm steps. We initialize a result list, iterate through each number, convert it to a string, and append each digit to the result after converting it back to an integer. The structure mirrors the conceptual model of flattening nested digit sequences into a single list.

## Go Solution

```go
func separateDigits(nums []int) []int {
    result := make([]int, 0)

    for _, num := range nums {
        s := strconv.Itoa(num)
        for i := 0; i < len(s); i++ {
            result = append(result, int(s[i]-'0'))
        }
    }

    return result
}
```

In Go, we use `strconv.Itoa` to convert integers into strings, which requires importing the `strconv` package. Unlike Python, Go does not allow direct character-to-integer conversion in the same expressive way, so we subtract `'0'` from the byte value to obtain the digit.

The slice `result` is dynamically appended to, which is idiomatic in Go. There are no concerns about integer overflow because each element is a single digit between 0 and 9.

## Worked Examples

For the input `[13,25,83,77]`, we process each number sequentially.

For `13`, string conversion yields `"13"`, and we append `1`, then `3`.

For `25`, we append `2`, then `5`.

For `83`, we append `8`, then `3`.

For `77`, we append `7`, then `7`.

The intermediate state evolves as follows:

| Step | Current Number | Digits Extracted | Result |
| --- | --- | --- | --- |
| 1 | 13 | 1, 3 | [1, 3] |
| 2 | 25 | 2, 5 | [1, 3, 2, 5] |
| 3 | 83 | 8, 3 | [1, 3, 2, 5, 8, 3] |
| 4 | 77 | 7, 7 | [1, 3, 2, 5, 8, 3, 7, 7] |

For `[7,1,3,9]`, each number is a single digit, so the output remains identical to the input since each string has length one.

## Edge Cases

One important edge case is when the input contains single-digit numbers only. In this scenario, the output should match the input exactly, since no decomposition changes the structure. The implementation handles this naturally because converting a single-digit number to a string yields a one-character string, which is appended directly.

Another edge case involves numbers containing zeros, such as 10 or 100. These can sometimes be mishandled if integer arithmetic is used incorrectly, since leading or trailing zeros may be lost during division-based extraction. The string-based approach avoids this entirely because every character is preserved exactly as it appears in the original number.

A third edge case is the smallest input size, where `nums` contains only one number. The algorithm still works correctly because it initializes an empty result list and processes exactly one iteration, ensuring no special handling is required for singleton inputs.
