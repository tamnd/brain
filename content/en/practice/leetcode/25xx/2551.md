# LeetCode 2551, Put Marbles in Bags

## Problem Understanding

The problem gives us an array `weights`, where each value represents the weight of a marble placed in a fixed order. We must divide these marbles into exactly `k` contiguous groups, where each group represents a bag.

The important restriction is that each bag must contain a contiguous segment of the array. This means we are not allowed to rearrange marbles or arbitrarily pick elements for a bag. Every marble must belong to exactly one bag, and together the bags must cover the entire array.

For a bag containing marbles from index `i` to index `j`, the bag's cost is:

```
weights[i] + weights[j]
```

Only the first and last marble in the segment matter for the cost calculation. The marbles in the middle do not directly affect the bag's cost.

The total score of a distribution is the sum of the costs of all `k` bags.

Our task is to compute:

```
maximum possible score - minimum possible score
```

among all valid ways to partition the array into `k` contiguous bags.

The constraints are large:

- `weights.length` can be up to `10^5`
- `weights[i]` can be up to `10^9`

These constraints immediately rule out any exponential or quadratic partitioning dynamic programming approach. We need something close to `O(n log n)` or better.

There are several important edge cases:

- `k = 1`, meaning all marbles must stay in one bag. In this case, the maximum and minimum scores are identical, so the answer is `0`.
- `k = n`, meaning every marble gets its own bag. Again, there is only one possible distribution.
- Very large values in `weights`, which means we must avoid integer overflow in languages like Go.
- Arrays with repeated values, where multiple partition choices may produce identical scores.

The problem guarantees that `1 <= k <= weights.length`, so every requested partitioning is valid.

## Approaches

### Brute Force Approach

A brute force solution would try every possible way to place `k - 1` partition boundaries between marbles.

For an array of length `n`, there are `n - 1` possible cut positions. We need to choose `k - 1` of them.

The total number of possibilities is:

```
C(n - 1, k - 1)
```

For each partition configuration, we could compute the resulting score and track the minimum and maximum.

This approach is correct because it exhaustively explores every valid distribution.

However, it is far too slow. Even for moderate values of `n`, the number of combinations becomes enormous. With `n = 10^5`, brute force is completely infeasible.

### Key Insight for the Optimal Solution

The critical observation is that the score can be rewritten in a very convenient form.

Suppose we partition the array into bags using cuts between indices.

For example:

```
[1,3] | [5,1]
```

The score becomes:

```
(1 + 3) + (5 + 1)
```

Notice something important:

- The very first marble always contributes once.
- The very last marble always contributes once.
- Every cut between `i` and `i+1` adds:

```
weights[i] + weights[i+1]
```

to the total score.

This means every partition configuration can be represented as:

```sql
weights[0] + weights[n-1]
+ sum of contributions from chosen cuts
```

So the problem reduces to:

- Compute all adjacent pair sums:

```
weights[i] + weights[i+1]
```

- Choose `k - 1` of them.
- Maximum score uses the largest `k - 1` pair sums.
- Minimum score uses the smallest `k - 1` pair sums.

This transforms the problem from partition DP into a simple greedy sorting problem.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | O(k) | Tries every partition configuration |
| Optimal | O(n log n) | O(n) | Uses adjacent pair contributions and sorting |

## Algorithm Walkthrough

1. Observe that every partition introduces a cut between two adjacent marbles.
2. For every adjacent pair `(i, i+1)`, compute:

```
weights[i] + weights[i+1]
```

These values represent the contribution added to the score if we place a partition between those two marbles.

1. Store all adjacent pair sums in an array called `pair_sums`.
2. Sort `pair_sums`.
3. Since we need exactly `k - 1` cuts:

- The minimum score uses the smallest `k - 1` pair sums.
- The maximum score uses the largest `k - 1` pair sums.
4. Compute:

- `min_sum` as the sum of the first `k - 1` elements.
- `max_sum` as the sum of the last `k - 1` elements.
5. Return:

```
max_sum - min_sum
```

The constant contribution from `weights[0] + weights[n-1]` appears in both the maximum and minimum scores, so it cancels out automatically.

### Why it works

Every valid partitioning into `k` bags corresponds to choosing exactly `k - 1` cut positions. Each cut contributes exactly one adjacent pair sum to the total score. Since the fixed endpoint contribution is identical for all configurations, maximizing or minimizing the total score depends entirely on which adjacent pair sums are selected. Therefore, choosing the largest `k - 1` pair sums yields the maximum score, and choosing the smallest `k - 1` pair sums yields the minimum score.

## Python Solution

```python
from typing import List

class Solution:
    def putMarbles(self, weights: List[int], k: int) -> int:
        n = len(weights)

        if k == 1:
            return 0

        pair_sums = []

        for i in range(n - 1):
            pair_sums.append(weights[i] + weights[i + 1])

        pair_sums.sort()

        min_score = sum(pair_sums[:k - 1])
        max_score = sum(pair_sums[-(k - 1):])

        return max_score - min_score
```

The implementation begins by handling the simple edge case where `k == 1`. In that situation, there is only one possible bag, so the maximum and minimum scores are identical.

Next, we compute all adjacent pair sums. Each value represents the contribution added when placing a partition between two neighboring marbles.

After collecting these values, we sort them. Sorting allows us to efficiently identify:

- The smallest `k - 1` contributions
- The largest `k - 1` contributions

We then compute the two sums and return their difference.

The implementation directly mirrors the mathematical observation derived in the algorithm discussion.

## Go Solution

```go
package main

import "sort"

func putMarbles(weights []int, k int) int64 {
	n := len(weights)

	if k == 1 {
		return 0
	}

	pairSums := make([]int, 0, n-1)

	for i := 0; i < n-1; i++ {
		pairSums = append(pairSums, weights[i]+weights[i+1])
	}

	sort.Ints(pairSums)

	var minScore int64 = 0
	var maxScore int64 = 0

	for i := 0; i < k-1; i++ {
		minScore += int64(pairSums[i])
		maxScore += int64(pairSums[len(pairSums)-1-i])
	}

	return maxScore - minScore
}
```

The Go implementation follows the same logic as the Python solution.

One important difference is integer handling. Since `weights[i]` can be as large as `10^9`, the total score may exceed the range of a 32 bit integer. For that reason, the final accumulation variables use `int64`.

Go slices are used for storing adjacent pair sums, and the standard library `sort.Ints` function performs the sorting.

The algorithm performs a single pass to build the adjacent pair sums array, which costs `O(n)`. The sorting step costs `O(n log n)`, which becomes the dominant operation. The extra memory usage comes from storing the `n - 1` pair sums.

## Edge Cases

### Edge Case 1: `k = 1`

When there is only one bag, all marbles must remain together. No cuts are possible, which means there is only one valid score.

A buggy implementation might still attempt to compute partition contributions or access invalid ranges while summing the smallest and largest cuts.

This implementation handles the case immediately with:

```
if k == 1:
    return 0
```

which avoids unnecessary computation and guarantees correctness.

### Edge Case 2: `k = n`

When every marble must be placed in its own bag, every possible cut position is forced to exist.

Since both the maximum and minimum configurations use exactly the same cuts, the resulting difference must be zero.

The implementation naturally handles this because:

- The smallest `k - 1` pair sums
- The largest `k - 1` pair sums

become the same full set of adjacent sums.

### Edge Case 3: Large Weight Values

The constraints allow marble weights up to `10^9`. Summing many such values can exceed 32 bit integer limits.

In Python this is not an issue because integers have arbitrary precision.

In Go, however, using `int` for the accumulated result could overflow on some systems. The implementation avoids this by explicitly using `int64` for `minScore`, `maxScore`, and the return type.

### Edge Case 4: Repeated Adjacent Pair Sums

Arrays with repeated values may produce many identical adjacent pair sums.

For example:

```
weights = [1,1,1,1]
```

produces:

```
[2,2,2]
```

A flawed greedy approach might incorrectly assume unique ordering matters.

The implementation works correctly because it only relies on selecting the smallest and largest values after sorting, regardless of duplicates.
