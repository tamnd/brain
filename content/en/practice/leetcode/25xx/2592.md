# LeetCode 2592 - Maximize Greatness of an Array

## Problem Understanding

The problem gives us an integer array `nums`, and we are allowed to rearrange its elements into any permutation `perm` we want. After rearranging, we compare the arrays index by index. For every position `i`, if `perm[i] > nums[i]`, that position contributes `1` to the greatness score.

Our goal is to maximize the number of such positions.

In other words, we want to rearrange the numbers so that as many elements as possible become strictly larger than the original element at the same index.

The key detail is that `perm` must contain exactly the same elements as `nums`, just reordered. We cannot add or remove values.

For example, suppose:

```
nums = [1,2,3,4]
```

One optimal permutation is:

```
perm = [2,3,4,1]
```

Comparing index by index:

| Index | nums[i] | perm[i] | perm[i] > nums[i]? |
| --- | --- | --- | --- |
| 0 | 1 | 2 | Yes |
| 1 | 2 | 3 | Yes |
| 2 | 3 | 4 | Yes |
| 3 | 4 | 1 | No |

The greatness is `3`.

The constraints are important:

- `1 <= nums.length <= 10^5`
- `0 <= nums[i] <= 10^9`

Since the array can contain up to one hundred thousand elements, exponential or factorial solutions are impossible. Even quadratic solutions may be too slow in the worst case. This strongly suggests that we need an `O(n log n)` or better approach.

There are several important edge cases to think about:

- Arrays where all values are equal, such as `[5,5,5]`. No element can ever become strictly larger than another equal element, so the answer must be `0`.
- Arrays with many duplicates, such as `[1,1,2,2]`. We must carefully match larger values without wasting them.
- Arrays already in increasing order. Rearrangement is still allowed, so the original order does not matter.
- Arrays of size `1`. A single element can never be greater than itself after permutation, so the answer is always `0`.

## Approaches

### Brute Force Approach

The most direct idea is to generate every possible permutation of the array and compute the greatness score for each permutation.

For every permutation:

1. Compare it against the original array.
2. Count how many indices satisfy `perm[i] > nums[i]`.
3. Keep the maximum score.

This works because checking all permutations guarantees we eventually examine the optimal arrangement.

However, the number of permutations of an array of size `n` is `n!`, which becomes impossibly large even for modest values of `n`.

For example:

- `10! = 3,628,800`
- `15!` is already astronomically large

Since `n` can be `100000`, brute force is completely infeasible.

### Optimal Greedy Approach

The key insight is that we should always use the smallest available number that can beat the current number.

This naturally leads to sorting.

Suppose we sort the array:

```
sorted_nums
```

We then use two pointers:

- One pointer tracks the current number we want to beat.
- Another pointer searches for the smallest number strictly greater than it.

Whenever we find a larger number, we form a successful match and increase the greatness count.

This greedy strategy works because using the smallest possible winning number preserves larger values for future matches. If we waste a very large number early, we may lose opportunities later.

This is very similar to greedy matching problems where we try to maximize successful pairings.

## Algorithm Walkthrough

### Optimal Greedy Algorithm

1. Sort the array in nondecreasing order.

Sorting allows us to efficiently match smaller numbers with the smallest larger numbers that can beat them.
2. Initialize two pointers.

Let:

- `left` point to the number we want to beat
- `right` point to the candidate number that may beat it

Initially:

```
left = 0
right = 0
```
3. Iterate while both pointers remain inside the array.

At every step, compare:

```
nums[right] > nums[left]
```
4. If the condition is true, we found a successful match.

This means:

- `nums[right]` can be placed against `nums[left]`
- greatness increases by `1`

Then move both pointers forward because:

- the smaller element has now been beaten
- the larger element has now been used
5. Otherwise, the candidate is not large enough.

If:

```
nums[right] <= nums[left]
```

then `nums[right]` cannot beat `nums[left]`.

Since the array is sorted, this candidate is useless for the current target, so we move only `right` forward to search for a larger value.
6. Continue until `right` reaches the end.

The total number of successful matches is the maximum greatness.

### Why it works

The greedy strategy always uses the smallest possible value that can beat the current target. This is optimal because larger values are more valuable and should be saved for harder future matches.

If we used a larger-than-necessary number early, we could reduce the number of future successful pairings. By always choosing the minimal winning value, we maximize the total number of matches.

This creates the maximum possible number of pairs `(a, b)` such that `b > a`.

## Python Solution

```python
from typing import List

class Solution:
    def maximizeGreatness(self, nums: List[int]) -> int:
        nums.sort()

        left = 0
        right = 0
        greatness = 0

        n = len(nums)

        while right < n:
            if nums[right] > nums[left]:
                greatness += 1
                left += 1

            right += 1

        return greatness
```

The implementation begins by sorting the array. This is the foundation of the greedy strategy because it allows us to process values in increasing order.

The `left` pointer represents the current element we want to beat. The `right` pointer scans forward searching for the smallest value that is strictly greater.

Whenever `nums[right] > nums[left]`, we form a successful match:

```python
greatness += 1
left += 1
```

Moving `left` forward means that target has now been successfully beaten.

The `right` pointer always advances because each candidate value can only be used once.

At the end, `greatness` stores the maximum number of successful pairings.

## Go Solution

```go
package main

import "sort"

func maximizeGreatness(nums []int) int {
	sort.Ints(nums)

	left := 0
	right := 0
	greatness := 0
	n := len(nums)

	for right < n {
		if nums[right] > nums[left] {
			greatness++
			left++
		}

		right++
	}

	return greatness
}
```

The Go implementation follows the same logic as the Python solution.

The main difference is that Go uses `sort.Ints(nums)` to sort the slice in place. Slices in Go behave similarly to dynamic arrays, so no additional copying is needed.

There are no integer overflow concerns because the maximum answer is at most `n`, which fits easily inside Go's `int` type.

The algorithm also naturally handles empty or single-element edge cases correctly.

## Edge Cases

One important edge case is when all elements are identical. Consider:

```
[7,7,7,7]
```

No element can ever be strictly greater than another because all values are equal. A naive implementation might accidentally count equal values as valid matches if it uses `>=` instead of `>`. The current implementation correctly requires strict inequality.

Another important edge case is an array with only one element. For example:

```
[42]
```

Since the permutation must contain the same single value, there is no way to create a strictly greater comparison. The algorithm correctly returns `0` because the condition `nums[right] > nums[left]` never becomes true.

A third tricky case involves many duplicates with only a few larger numbers, such as:

```
[1,1,1,1,5]
```

Only one of the `1`s can be matched with `5`. A careless greedy strategy might incorrectly try to reuse the same larger value multiple times. The two pointer approach avoids this because every successful match advances both pointers, ensuring each element is used at most once.

Another subtle case occurs when the array is already sorted in increasing order. Some implementations may incorrectly assume the answer is always `n - 1`, but duplicates can reduce the achievable greatness. The greedy matching process computes the true maximum regardless of the original ordering.
