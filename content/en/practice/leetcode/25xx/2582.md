## Solution

## Problem Understanding

The problem describes a simulation of passing a pillow along a line of `n` people. The first person starts holding the pillow, and every second, the pillow is passed to the next person in line. When the pillow reaches either end of the line, the direction reverses. Given `n` and a `time` in seconds, we are asked to determine which person is holding the pillow at that exact second.

The input consists of two integers: `n` representing the number of people and `time` representing the number of seconds elapsed. The output is a single integer representing the index of the person currently holding the pillow. The constraints tell us that `n` and `time` are reasonably small (`<= 1000`), so even a straightforward simulation would be feasible but not elegant.

Edge cases to consider include `time` being very small (1 second), `time` being exactly at the turning points (`n-1`, `2n-2`), and `n` being the minimum allowed value (2).

## Approaches

The brute-force approach is to simulate each second by moving the pillow left or right depending on the current direction. We maintain a pointer for the current holder and a flag for direction. This works for small inputs but becomes tedious for large `time` values.

A better approach is to notice that the pillow follows a predictable pattern. The pillow moves from the first to the last person in `n-1` seconds, then back from the last to the first in another `n-1` seconds. Therefore, the pattern repeats every `2*(n-1)` seconds. By using modulo arithmetic, we can determine the position without simulating every second.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(time) | O(1) | Simulate each second by moving the pillow along the line |
| Optimal | O(1) | O(1) | Use pattern repetition and modulo to calculate the position directly |

## Algorithm Walkthrough

1. Calculate the period of the full back-and-forth cycle as `cycle = 2 * (n - 1)`.
2. Reduce the given `time` modulo the cycle: `t = time % cycle`. This gives the effective seconds within a single back-and-forth period.
3. If `t < n`, the pillow is moving to the right, so the current holder is `1 + t`.
4. Otherwise, the pillow is moving back to the left, so the current holder is `n - (t - (n - 1))`.
5. Return the computed position.

This works because the pattern of the pillow passing repeats every `2*(n-1)` seconds. By reducing `time` modulo this period, we avoid unnecessary simulation and directly compute the position.

## Python Solution

```python
class Solution:
    def passThePillow(self, n: int, time: int) -> int:
        cycle = 2 * (n - 1)
        t = time % cycle
        if t < n:
            return 1 + t
        else:
            return n - (t - (n - 1))
```

In this Python implementation, we first calculate the total cycle length. We then reduce the time to find its position within the current cycle. If the pillow is still moving forward in the line, we add `t` to the starting position. If it has passed the last person, we subtract the excess seconds from the last position to simulate the backward motion.

## Go Solution

```go
func passThePillow(n int, time int) int {
    cycle := 2 * (n - 1)
    t := time % cycle
    if t < n {
        return 1 + t
    }
    return n - (t - (n - 1))
}
```

The Go implementation mirrors the Python logic exactly. Go uses integer arithmetic naturally, so modulo and subtraction behave as expected. There is no need for extra error handling because constraints guarantee valid input.

## Edge Cases

First, the minimal value for `n = 2` is interesting because the pillow alternates between two people every second. This is a simple pattern, but incorrect handling of modulo or direction could give wrong results. The implementation correctly handles it using the cycle formula.

Second, when `time` is exactly `n-1` or `2*(n-1)`, the pillow reaches an endpoint of the line. A naive solution might overshoot or incorrectly reverse the direction. The modulo-based calculation ensures exact alignment with the cycle boundaries.

Third, very large `time` values, potentially multiple cycles, can expose inefficiency in a simulation-based solution. Using modulo arithmetic, our solution avoids looping over every second, maintaining constant time complexity regardless of how large `time` is within constraints.
