# LeetCode 2591 - Distribute Money to Maximum Children

## Problem Understanding

This problem asks us to distribute a fixed amount of money among a fixed number of children while satisfying several strict rules.

We are given two integers:

- `money`, the total amount of dollars available
- `children`, the number of children who must each receive some money

The distribution must satisfy three conditions:

1. Every dollar must be distributed.
2. Every child must receive at least `1` dollar.
3. No child is allowed to receive exactly `4` dollars.

Our goal is to maximize the number of children who receive exactly `8` dollars.

The important detail is that we are not trying to find just any valid distribution. We specifically want the largest possible count of children whose final amount is exactly `8`.

The constraints are quite small:

- `1 <= money <= 200`
- `2 <= children <= 30`

Since the input size is tiny, even brute-force solutions are technically feasible. However, the structure of the problem allows for a direct greedy mathematical solution.

Several edge cases are especially important:

- If there is not enough money to give every child at least `1` dollar, the answer must be `-1`.
- A distribution that accidentally leaves a child with exactly `4` dollars is invalid.
- Sometimes greedily assigning as many `8`s as possible creates an impossible leftover distribution.
- A special case occurs when the remaining money becomes exactly `4` for the last child, because that child cannot receive `4`.

Understanding these corner cases is the key to building the correct greedy strategy.

## Approaches

### Brute Force Approach

A brute-force solution would try every possible way to distribute the money among the children while checking all constraints.

One possible implementation is recursive backtracking:

- Assign some amount of money to the current child.
- Continue recursively for the remaining children and remaining money.
- Reject distributions where:

- a child gets less than `1`
- a child gets exactly `4`
- total money does not match
- Track the maximum number of children receiving exactly `8`.

This approach is correct because it explores all valid distributions. Eventually, it will find the optimal one.

However, the number of possible distributions grows extremely quickly. Even though the constraints are small, brute force is unnecessary and inefficient compared to the much simpler greedy observation.

### Optimal Greedy Approach

The key observation is that every child must first receive at least `1` dollar.

If we give every child `1` dollar initially, then:

- We spend `children` dollars immediately.
- The remaining money becomes `money - children`.

Now consider what it takes to make a child receive exactly `8` dollars.

Since every child already has `1`, we need to add `7` more dollars to that child.

Therefore, every additional group of `7` dollars can create one child with exactly `8`.

This naturally suggests a greedy strategy:

- First, give everyone `1`.
- Then maximize how many groups of `7` we can distribute.

However, there are two tricky cases:

1. If the leftover money after forming maximum `8`s becomes exactly `3` while only one child remains, that child would end up with `4`, which is forbidden.
2. If all children already have `8` dollars but extra money still remains, we must reduce one of the `8`s because someone must absorb the extra money.

These observations lead to a compact mathematical greedy solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | O(children) | Tries all possible distributions recursively |
| Optimal | O(1) | O(1) | Uses greedy math observations and edge case handling |

## Algorithm Walkthrough

1. First, check whether it is even possible to give every child at least `1` dollar.

If `money < children`, then some child would receive `0`, which violates the rules. In this case, return `-1`.
2. Give every child `1` dollar conceptually.

This uses exactly `children` dollars.

Update:

```
remaining = money - children
```
3. Compute how many children can potentially receive exactly `8` dollars.

Since each child already has `1`, we need `7` more dollars per child.

Therefore:

```
eight_children = remaining // 7
leftover = remaining % 7
```
4. If the number of potential `8`-dollar children exceeds the number of children available, cap it.

We cannot have more than `children` children receiving `8`.
5. Handle the special case where every child has `8` dollars but extra money still remains.

If:

```
eight_children == children AND leftover > 0
```

then some child must receive extra money beyond `8`.

That means at least one child can no longer stay at exactly `8`.

Return:

```
children - 1
```
6. Handle the forbidden `4`-dollar scenario.

Suppose exactly one child does not receive `8`, and the leftover amount is `3`.

That remaining child already has the initial `1` dollar.

So they would end up with:

```
1 + 3 = 4
```

which is forbidden.

Therefore, we must reduce one previously assigned `8`.

The condition is:

```
eight_children == children - 1 AND leftover == 3
```

In this case, return:

```
children - 2
```
7. Otherwise, the greedy allocation works directly.

Return:

```
eight_children
```

### Why it works

The greedy strategy works because every child must receive at least `1` dollar, so the problem reduces to distributing extra money. Each additional `7` dollars uniquely creates one child with exactly `8`. Since our objective is to maximize the number of `8`s, spending extra money in chunks of `7` is always optimal.

The only situations where the greedy result becomes invalid are:

- an unavoidable `4`-dollar child
- extra money forcing an `8`-dollar child above `8`

By explicitly correcting these two cases, the algorithm always produces the optimal valid distribution.

## Python Solution

```python
class Solution:
    def distMoney(self, money: int, children: int) -> int:
        # Not enough money to give everyone at least 1 dollar
        if money < children:
            return -1

        # Give every child 1 dollar first
        remaining = money - children

        # Each additional 7 dollars creates one child with exactly 8
        eight_children = remaining // 7
        leftover = remaining % 7

        # Cannot exceed total number of children
        if eight_children > children:
            eight_children = children

        # If all children have 8 dollars but money is still left,
        # one child must take extra money
        if eight_children == children and leftover > 0:
            return children - 1

        # Avoid a child ending up with exactly 4 dollars
        if eight_children == children - 1 and leftover == 3:
            return children - 2

        return eight_children
```

The implementation follows the mathematical greedy reasoning directly.

The first condition checks whether a valid distribution is even possible. If there is not enough money for every child to receive at least one dollar, the function immediately returns `-1`.

Next, the code conceptually distributes one dollar to every child. This simplifies the rest of the logic because we only need to think about extra money beyond the mandatory minimum.

The variables `eight_children` and `leftover` determine how many complete groups of `7` dollars we can form and how much money remains afterward.

The remaining conditions handle the two important edge cases:

- all children having `8` but leftover money still existing
- a remaining child accidentally receiving exactly `4`

Finally, the function returns the maximum valid number of children receiving exactly `8`.

## Go Solution

```go
func distMoney(money int, children int) int {
    // Not enough money to give everyone at least 1 dollar
    if money < children {
        return -1
    }

    // Give every child 1 dollar first
    remaining := money - children

    // Each additional 7 dollars creates one child with exactly 8
    eightChildren := remaining / 7
    leftover := remaining % 7

    // Cannot exceed total number of children
    if eightChildren > children {
        eightChildren = children
    }

    // If all children have 8 dollars but money is still left
    if eightChildren == children && leftover > 0 {
        return children - 1
    }

    // Avoid a child ending up with exactly 4 dollars
    if eightChildren == children-1 && leftover == 3 {
        return children - 2
    }

    return eightChildren
}
```

The Go implementation mirrors the Python logic almost exactly.

Go uses integer division automatically for integers, so `/` and `%` behave exactly as needed here.

No special handling for overflow is necessary because the constraints are extremely small. The solution also uses only a constant amount of extra memory.

## Edge Cases

One important edge case occurs when there is not enough money to give every child at least one dollar. For example, `money = 1` and `children = 2`. Since the problem requires every child to receive at least one dollar, no valid distribution exists. The implementation handles this immediately with the condition `money < children`, returning `-1`.

Another tricky edge case occurs when the greedy allocation accidentally creates a child with exactly `4` dollars. For example, `money = 20` and `children = 3`. After assigning one dollar to each child, we have `17` remaining. Greedily assigning two groups of `7` leaves `3` extra dollars for the last child, causing them to receive `4` total. Since `4` is forbidden, the implementation detects this situation and reduces the answer by one.

A third important edge case happens when every child initially receives exactly `8` dollars but extra money still remains. For example, `money = 25` and `children = 3`. Three children can receive `8`, consuming `24` dollars, but one extra dollar remains. Someone must take that extra dollar, meaning one child can no longer stay at exactly `8`. The implementation detects this case and reduces the result from `3` to `2`.

A final edge case involves distributions where no child can receive exactly `8` while still satisfying the rules. For example, `money = 9` and `children = 2`. After giving each child one dollar, only `7` dollars remain, allowing exactly one child to reach `8`. However, some nearby cases produce zero valid `8`-dollar children. The implementation correctly handles these naturally through integer division and leftover calculations.
