## Solution

## Problem Understanding

The problem asks us to find the **minimum time** required to travel from the top-left cell `(0,0)` to the bottom-right cell `(m-1,n-1)` of a `m x n` grid, where each cell `(i,j)` has a **non-negative integer** `grid[i][j]` representing the earliest time that cell can be visited. You start at time `0` at `(0,0)` and can move in four directions: up, down, left, and right, with each move taking exactly `1` second. You can only enter a cell if the current time is **greater than or equal to** `grid[i][j]`. If it is impossible to reach the bottom-right cell, the function should return `-1`.

The constraints provide important information about the input size and guarantee that `grid[0][0] == 0`. The grid can be as large as `1000 x 1000`, so any algorithm that is worse than `O(m * n log(m * n))` will likely time out. Edge cases include cells that are completely blocked until a time that cannot be reached due to parity constraints or isolated high-value cells.

Key points to note:

- Every move takes exactly 1 second.
- Each cell may have a "wait time" until it is allowed to be visited.
- It is possible that the bottom-right is unreachable if no valid path exists.
- The grid is guaranteed to be at least 2x2.

## Approaches

A brute-force approach is to perform a standard Breadth-First Search (BFS), starting at `(0,0)` and trying all possible paths while keeping track of the current time. For each neighboring cell, we would check if the time is at least `grid[i][j]`. If not, we can "wait" until the earliest possible time. This approach is correct but inefficient because it can explore redundant paths repeatedly, especially for large grids, leading to an exponential number of states.

The key insight for an optimal solution is that this is a **shortest path problem on a weighted graph**, where the weight of moving into a cell depends on the earliest allowed time of that cell. Using a **priority queue (min-heap)** to always expand the node with the minimum arrival time ensures we process cells in the order of the earliest achievable time. For cells where the earliest allowed time is greater than the current time, we must also handle the "parity" of moves because moving every second alternates time, so in some cases we may need to wait an extra second to match the parity of the required time.

This reduces the problem to a Dijkstra-style algorithm on a grid.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force BFS | O(4^(m*n)) | O(m*n) | Explores all possible paths; infeasible for large grids |
| Dijkstra / Min-Heap BFS | O(m_n log(m_n)) | O(m*n) | Uses priority queue to process earliest reachable cells efficiently |

## Algorithm Walkthrough

1. Initialize a **min-heap (priority queue)** that stores tuples of `(time, row, col)`. Start by pushing `(0,0,0)` since we start at `(0,0)` at time `0`.
2. Maintain a **visited matrix** or **distance matrix** `dist` of the same size as the grid, initialized to infinity. Set `dist[0][0] = 0`.
3. While the priority queue is not empty, pop the cell `(current_time, r, c)` with the **minimum time**.
4. If `(r,c)` is the bottom-right cell `(m-1,n-1)`, return `current_time` as the minimum time.
5. For each neighbor `(nr,nc)` of `(r,c)` (up, down, left, right), compute the **earliest time you can arrive**. This is `max(current_time + 1, grid[nr][nc])`. However, if `(earliest_time - grid[nr][nc]) % 2 == 1`, increment by 1 to satisfy parity constraints because you can only move in integer seconds.
6. If this new time is less than `dist[nr][nc]`, update `dist[nr][nc]` and push `(new_time, nr, nc)` into the priority queue.
7. If the priority queue empties without reaching `(m-1,n-1)`, return `-1`.

**Why it works:** This algorithm always expands the cell that can be reached the earliest. By updating neighbors only when a smaller time is found, it ensures we find the global minimum time for each cell. The parity adjustment guarantees that the time constraints imposed by the grid values are respected.

## Python Solution

```python
from typing import List
import heapq

class Solution:
    def minimumTime(self, grid: List[List[int]]) -> int:
        m, n = len(grid), len(grid[0])
        if m == 1 and n == 1:
            return 0
        
        # Check trivial impossibility due to parity at the start
        if grid[0][1] > 1 and grid[1][0] > 1:
            return -1

        directions = [(0,1),(1,0),(0,-1),(-1,0)]
        dist = [[float('inf')] * n for _ in range(m)]
        dist[0][0] = 0
        heap = [(0,0,0)]  # (time, row, col)
        
        while heap:
            time, r, c = heapq.heappop(heap)
            if (r, c) == (m-1, n-1):
                return time
            if time > dist[r][c]:
                continue
            for dr, dc in directions:
                nr, nc = r + dr, c + dc
                if 0 <= nr < m and 0 <= nc < n:
                    new_time = max(time + 1, grid[nr][nc])
                    # Adjust for parity
                    if (new_time - grid[nr][nc]) % 2 == 1:
                        new_time += 1
                    if new_time < dist[nr][nc]:
                        dist[nr][nc] = new_time
                        heapq.heappush(heap, (new_time, nr, nc))
        return -1
```

This implementation initializes the distance matrix and the heap, ensures parity adjustments, and guarantees that we always propagate the earliest reachable times through the grid. The priority queue ensures we process cells in optimal order.

## Go Solution

```go
package main

import (
    "container/heap"
    "math"
)

type Item struct {
    time, r, c int
}

type MinHeap []Item

func (h MinHeap) Len() int { return len(h) }
func (h MinHeap) Less(i, j int) bool { return h[i].time < h[j].time }
func (h MinHeap) Swap(i, j int) { h[i], h[j] = h[j], h[i] }
func (h *MinHeap) Push(x any) { *h = append(*h, x.(Item)) }
func (h *MinHeap) Pop() any {
    old := *h
    n := len(old)
    x := old[n-1]
    *h = old[:n-1]
    return x
}

func minimumTime(grid [][]int) int {
    m, n := len(grid), len(grid[0])
    if m == 1 && n == 1 {
        return 0
    }
    if grid[0][1] > 1 && grid[1][0] > 1 {
        return -1
    }

    dist := make([][]int, m)
    for i := range dist {
        dist[i] = make([]int, n)
        for j := range dist[i] {
            dist[i][j] = math.MaxInt32
        }
    }
    dist[0][0] = 0

    h := &MinHeap{{0,0,0}}
    heap.Init(h)

    dirs := [][2]int{{0,1},{1,0},{0,-1},{-1,0}}

    for h.Len() > 0 {
        item := heap.Pop(h).(Item)
        time, r, c := item.time, item.r, item.c
        if r == m-1 && c == n-1 {
            return time
        }
        if time > dist[r][c] {
            continue
        }
        for _, d := range dirs {
            nr, nc := r + d[0], c + d[1]
            if nr >= 0 && nr < m && nc >= 0 && nc < n {
                newTime := max(time+1, grid[nr][nc])
                if (newTime - grid[nr][nc]) % 2 == 1 {
                    newTime++
                }
                if newTime < dist[nr][nc] {
                    dist[nr][nc] = newTime
                    heap.Push(h, Item{newTime, nr, nc})
                }
            }
        }
    }
    return -1
}

func max(a, b int) int {
    if a > b { return a }
    return b
}
```

The Go implementation follows the same logic as Python, with explicit heap implementation using the container/heap package. The parity adjustment and distance updates work identically.
