# LeetCode 2598 - Smallest Missing Non-negative Integer After Operations

## Problem Understanding

This problem asks us to maximize the MEX, the minimum excluded non-negative integer, after performing any number of operations on the array.

Each operation allows us to either add or subtract `value` from any element. Since we can repeat this operation indefinitely, each number can effectively move across all integers that share the same remainder modulo `value`.

For example, if `value = 5` and we start with the number `7`, then after repeated additions or subtractions of `5`, we can transform it into:

- `..., -8, -3, 2, 7, 12, 17, ...`

All of these numbers have the same remainder modulo `5`.

The core observation is that an element can only become numbers belonging to its modulo class. A number with remainder `2` modulo `5` can never become a number with remainder `1` modulo `5`.

The goal is to rearrange the numbers through these operations so that the resulting array contains the longest possible sequence starting from `0`.

If we can construct:

```
0, 1, 2, 3
```

then the MEX is `4`.

If we can only construct:

```
0, 1
```

then the MEX is `2`.

The input constraints are important:

- `nums.length` can be up to `100000`
- `value` can also be up to `100000`

This immediately tells us that exponential search or repeated simulation is impossible. We need an efficient linear or near-linear solution.

A few important edge cases stand out immediately:

- Negative numbers exist in the input.
- Multiple numbers may share the same modulo remainder.
- Some modulo classes may be completely absent.
- The optimal arrangement is not unique.
- Large values require careful modulo handling, especially in languages where negative modulo behaves differently.

The problem guarantees that operations can be applied infinitely many times, which means only the modulo remainder matters, not the original magnitude of the number.

## Approaches

### Brute Force Approach

A naive solution would attempt to explicitly simulate operations and search for achievable arrays.

One possible brute-force strategy is:

1. Generate many possible transformed values for each element.
2. Try to greedily build the sequence `0, 1, 2, ...`
3. Track which transformed numbers have already been used.

This approach is theoretically correct because every reachable configuration could eventually be explored.

However, it is completely impractical.

Each number can transform into infinitely many integers differing by multiples of `value`. Even if we artificially bound the search space, the number of possible combinations becomes enormous.

The brute-force method quickly becomes computationally infeasible for arrays of size `100000`.

### Key Insight

The crucial observation is that adding or subtracting `value` never changes a number's modulo remainder.

That means:

```
x can become y
iff
x % value == y % value
```

Suppose we want to construct the sequence:

```
0, 1, 2, 3, ...
```

Then:

- To create `0`, we need a number with remainder `0 % value`
- To create `1`, we need a number with remainder `1 % value`
- To create `2`, we need a number with remainder `2 % value`

and so on.

Therefore, the problem becomes:

> How many numbers do we have available in each modulo class?

We count how many elements belong to each remainder bucket.

Then we greedily attempt to construct integers starting from `0`.

For every integer `mex`:

```
needed remainder = mex % value
```

If we still have an unused number in that remainder bucket, we consume one and continue.

Otherwise, we cannot build `mex`, so it becomes the answer.

This greedy strategy works because constructing smaller integers is always more important than saving resources for larger integers.

## Algorithm Walkthrough

1. Create a frequency map that counts how many numbers belong to each modulo remainder.

For every number:

```
remainder = num % value
```

Store how many times each remainder appears.
2. Initialize `mex = 0`.

This represents the smallest non-negative integer we are currently trying to construct.
3. Determine which remainder bucket is needed for the current `mex`.

Since any number used to create `mex` must satisfy:

```
num % value == mex % value
```

we compute:

```
needed = mex % value
```
4. Check whether the required remainder bucket still contains unused elements.

If the count is positive:

- consume one element
- decrement the frequency
- increment `mex`
5. If the required remainder bucket is empty, stop immediately.

At this point, we cannot create the current `mex`, so it becomes the answer.
6. Return `mex`.

### Why it works

The algorithm maintains the invariant that every integer from `0` to `mex - 1` has already been successfully constructed.

To construct integer `k`, we must consume one unused element from remainder class `k % value`.

If such an element does not exist, then constructing `k` is impossible regardless of future decisions, because modulo classes are immutable under the allowed operations.

Therefore, the first unreachable integer is exactly the maximum achievable MEX.

## Python Solution

```python
from collections import Counter
from typing import List

class Solution:
    def findSmallestInteger(self, nums: List[int], value: int) -> int:
        remainder_count = Counter()

        for num in nums:
            remainder = num % value
            remainder_count[remainder] += 1

        mex = 0

        while True:
            needed_remainder = mex % value

            if remainder_count[needed_remainder] == 0:
                return mex

            remainder_count[needed_remainder] -= 1
            mex += 1
```

The implementation begins by counting how many numbers belong to each modulo remainder class.

Python's modulo operator already produces non-negative results for positive divisors, so:

```python
num % value
```

works correctly even for negative numbers.

The main loop incrementally attempts to build integers starting from `0`.

For every candidate MEX value, we compute the required remainder bucket. If that bucket still has available elements, we consume one and continue. Otherwise, we immediately return the current value.

The solution is concise because the mathematical observation eliminates the need for explicit transformations.

## Go Solution

```go
func findSmallestInteger(nums []int, value int) int {
	remainderCount := make(map[int]int)

	for _, num := range nums {
		remainder := ((num % value) + value) % value
		remainderCount[remainder]++
	}

	mex := 0

	for {
		neededRemainder := mex % value

		if remainderCount[neededRemainder] == 0 {
			return mex
		}

		remainderCount[neededRemainder]--
		mex++
	}
}
```

The Go implementation follows the same logic as the Python version.

One important difference is modulo handling for negative numbers. In Go, negative values can produce negative remainders, so we normalize using:

```go
((num % value) + value) % value
```

This guarantees a remainder in the range `[0, value - 1]`.

The frequency map is implemented using `map[int]int`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each element is processed once |
| Space | O(value) | Stores remainder frequencies |

The algorithm performs a single pass through the array to count remainders.

The second phase increments the MEX until failure. At most, this loop runs `n + 1` times because each successful step consumes one array element.

Therefore, the total runtime is linear.

The space complexity depends on how many distinct remainder classes exist. Since remainders range from `0` to `value - 1`, the maximum auxiliary space is `O(value)`.

## Edge Cases

One important edge case occurs when remainder `0` is completely absent. Since constructing the integer `0` requires a number whose modulo is `0`, the answer immediately becomes `0`. A naive implementation might incorrectly assume transformations can cross modulo classes, but this is impossible. The algorithm correctly detects this because the frequency for remainder `0` is zero during the first iteration.

Another tricky case involves negative numbers. Different languages handle negative modulo differently, which can easily introduce bugs. Python already normalizes modulo values into the positive range, but Go does not. The Go implementation explicitly normalizes remainders to ensure consistent behavior.

A third important edge case occurs when `value = 1`. In this situation, every integer belongs to the same modulo class, meaning every number can transform into any other integer. The maximum MEX therefore becomes exactly the array length. The greedy frequency-based solution naturally handles this because all counts accumulate into remainder bucket `0`.

A final subtle case involves many duplicates in the same remainder class. For example:

```
nums = [0, 5, 10]
value = 5
```

All elements belong to remainder `0`. This allows construction of:

```
0
```

but not `1`, because remainder `1` does not exist. The algorithm correctly consumes one remainder bucket entry per constructed integer, preventing overuse of the same element.
