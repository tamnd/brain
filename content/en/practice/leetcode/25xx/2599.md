## Solution

## Problem Understanding

The problem gives us an integer array `nums`, and we are allowed to repeatedly perform one operation: choose any element and move it to the end of the array.

Our goal is to make every prefix sum of the final array non-negative. A prefix sum at index `i` is the sum of all elements from index `0` through `i`.

For example, if the array is:

```
[3, -5, -2, 6]
```

then the prefix sums are:

```
[3, -2, -4, 2]
```

Since some prefix sums are negative, this array is invalid. We need to rearrange elements using the allowed operation so that every prefix sum is at least `0`.

The important detail is that moving an element to the end preserves the relative order of all remaining elements. Conceptually, when we remove an element from the current prefix, we postpone its contribution until later in the array.

The task asks for the minimum number of such operations.

The constraints are large:

- `1 <= nums.length <= 10^5`
- `-10^9 <= nums[i] <= 10^9`

This immediately rules out expensive simulations or brute-force rearrangements. Any solution worse than roughly `O(n log n)` will likely time out.

There is also an important guarantee: it is always possible to make all prefix sums non-negative. This means we do not need to handle impossible cases.

Several edge cases are important:

- Arrays that are already valid should return `0`
- Arrays containing many negative values may require multiple operations
- Very large positive or negative values require careful handling of integer overflow in some languages
- Prefix sums may temporarily become negative multiple times during traversal
- Consecutive negative numbers can create situations where choosing the wrong element to move leads to unnecessary operations

The key challenge is determining which elements should be postponed so that we minimize the number of moves.

## Approaches

### Brute-Force Approach

A brute-force solution would attempt to simulate all possible choices whenever the prefix sum becomes negative.

At each position, if the running prefix sum drops below zero, we could try moving one of the previously encountered elements to the end, then continue checking the array again. Since many different choices are possible at every step, this becomes a combinatorial search problem.

One possible brute-force strategy is:

1. Generate all valid sequences of moves
2. Simulate the resulting arrays
3. Compute prefix sums for each arrangement
4. Return the minimum number of operations

This approach is correct because it eventually explores every reachable arrangement.

However, it is completely impractical. The number of possible move sequences grows exponentially. Even for relatively small arrays, the search space becomes enormous.

A slightly improved brute-force method could greedily retry different negative elements, but repeated array modifications still lead to very poor performance.

### Key Insight

The crucial observation is that when the running prefix sum becomes negative, we must remove some previously encountered negative contribution from the prefix.

Suppose the current prefix sum is negative. To fix it with as few operations as possible, we should remove the most harmful negative number seen so far.

Why?

Because removing the most negative value increases the prefix sum by the largest possible amount. This gives us the maximum correction using a single operation.

This naturally suggests a greedy strategy:

- Traverse the array from left to right
- Maintain the current prefix sum
- Track all negative numbers encountered so far
- Whenever the prefix sum becomes negative:

- Remove the most negative value encountered
- Count one operation
- Restore the prefix sum

A min-heap is perfect for this because it allows efficient retrieval of the smallest, most negative value.

This leads to an `O(n log n)` solution, which is efficient enough for `10^5` elements.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Explores many possible rearrangements |
| Optimal Greedy + Heap | O(n log n) | O(n) | Removes the most harmful negative value whenever needed |

## Algorithm Walkthrough

1. Initialize a variable `prefix_sum` to `0`.

This variable tracks the running sum while traversing the array.
2. Initialize a counter `operations` to `0`.

This stores how many elements we move to the end.
3. Create a min-heap.

The heap stores all negative numbers encountered so far. Since Python and Go heaps are min-heaps by default, the smallest value in the heap is the most negative number.
4. Traverse the array from left to right.

For each number:

- Add it to `prefix_sum`
- If the number is negative, insert it into the heap
5. After updating the prefix sum, check whether it became negative.

If `prefix_sum < 0`, then the current prefix is invalid and must be fixed.
6. Remove the most negative number from the heap.

This simulates moving that element to the end of the array.

Since removing a negative number increases the prefix sum, update:

```
prefix_sum -= removed_negative
```

Because `removed_negative` is itself negative, this effectively adds its magnitude.
7. Increment the operation counter.

We performed one move operation.
8. Continue processing the remaining elements.
9. Return the total number of operations after the traversal finishes.

### Why it works

The greedy choice is optimal because whenever the prefix sum becomes negative, we must remove at least one previously seen negative contribution.

Choosing the most negative number provides the largest possible increase to the prefix sum with a single operation. Any smaller negative value would improve the prefix sum less and could force additional operations later.

Therefore, every operation is maximally effective, which guarantees the minimum number of moves.

## Python Solution

```python
from typing import List
import heapq

class Solution:
    def makePrefSumNonNegative(self, nums: List[int]) -> int:
        prefix_sum = 0
        operations = 0
        min_heap = []

        for num in nums:
            prefix_sum += num

            if num < 0:
                heapq.heappush(min_heap, num)

            if prefix_sum < 0:
                most_negative = heapq.heappop(min_heap)

                prefix_sum -= most_negative
                operations += 1

        return operations
```

The implementation directly follows the greedy algorithm.

The variable `prefix_sum` maintains the running total while iterating through the array.

Whenever we encounter a negative number, we push it into the min-heap. The heap allows us to efficiently retrieve the most negative value encountered so far.

If the running sum becomes negative, we immediately repair it by removing the smallest element from the heap. Since the heap stores negative values, the smallest element corresponds to the largest negative contribution.

Removing that value simulates moving it to the end of the array, so we increase the running sum accordingly and increment the operation count.

The algorithm processes each element once, and each heap operation costs `O(log n)`.

## Go Solution

```go
package main

import (
	"container/heap"
)

type MinHeap []int

func (h MinHeap) Len() int {
	return len(h)
}

func (h MinHeap) Less(i, j int) bool {
	return h[i] < h[j]
}

func (h MinHeap) Swap(i, j int) {
	h[i], h[j] = h[j], h[i]
}

func (h *MinHeap) Push(x interface{}) {
	*h = append(*h, x.(int))
}

func (h *MinHeap) Pop() interface{} {
	old := *h
	n := len(old)

	value := old[n-1]
	*h = old[:n-1]

	return value
}

func makePrefSumNonNegative(nums []int) int {
	var prefixSum int64 = 0
	operations := 0

	minHeap := &MinHeap{}
	heap.Init(minHeap)

	for _, num := range nums {
		prefixSum += int64(num)

		if num < 0 {
			heap.Push(minHeap, num)
		}

		if prefixSum < 0 {
			mostNegative := heap.Pop(minHeap).(int)

			prefixSum -= int64(mostNegative)
			operations++
		}
	}

	return operations
}
```

The Go implementation uses the `container/heap` package to implement a min-heap.

One important difference from Python is integer handling. Since prefix sums may exceed the range of a 32-bit integer, the implementation uses `int64` for `prefixSum`.

The heap stores ordinary `int` values because individual array elements fit within the problem constraints.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | Each element may be inserted and removed from the heap once |
| Space | O(n) | The heap may store all negative elements |

The traversal itself is linear, but heap insertions and removals each cost `O(log n)`.

In the worst case, every element is negative and gets inserted into the heap, so the total complexity becomes `O(n log n)`.

The heap may contain up to `n` negative elements simultaneously, giving `O(n)` auxiliary space usage.

## Edge Cases

One important edge case occurs when the array is already valid. For example:

```
[2, 3, -1, 4]
```

The prefix sums never drop below zero, so the correct answer is `0`. A buggy implementation might still move negative numbers unnecessarily simply because they exist. The current implementation only performs operations when the running prefix sum becomes negative.

Another critical edge case is when the first element itself is negative, such as:

```
[-5, 10]
```

The prefix sum immediately becomes invalid. The algorithm correctly pushes `-5` into the heap, detects the negative prefix sum, removes `-5`, and counts one operation. This effectively postpones the negative value to the end.

A third important edge case involves multiple negative values where choosing the wrong one leads to extra operations. Consider:

```
[4, -2, -8, 10]
```

When the prefix sum becomes negative, removing `-2` would not sufficiently repair the prefix. The greedy strategy correctly removes `-8`, the most harmful value, minimizing future operations.

Another subtle case involves very large values near the constraint limits. Since prefix sums may exceed the 32-bit integer range, the Go solution uses `int64` for accumulation. Without this precaution, integer overflow could produce incorrect results.
