## Solution

## Problem Understanding

The problem presents two baskets of fruits, each containing `n` fruits. Each fruit has an associated integer cost, given by the arrays `basket1` and `basket2`. The goal is to make both baskets equal in content, meaning that after possibly reordering the fruits, the two arrays become identical. To achieve equality, you are allowed to swap a fruit from `basket1` with a fruit from `basket2`. The cost of each swap is the minimum of the two fruits being exchanged.

The input consists of two integer arrays of equal length, and the output is the **minimum total cost** required to make the baskets equal or `-1` if it is impossible. The constraints tell us that the arrays can be quite large (up to 105 elements), and individual fruit costs can be very large (up to 10^9). This implies that any solution must be linearithmic (`O(n log n)`) or linear (`O(n)`) in time; a naive solution that tries all swaps is infeasible.

Edge cases that are important include when a fruit appears in one basket but not the other, when all fruits are identical, or when swaps can be minimized by exchanging very cheap fruits instead of more expensive ones. The problem guarantees non-empty arrays and valid integer costs.

## Approaches

A brute-force approach would attempt all possible swaps between the two baskets and check after each swap if the baskets can be made equal. While correct in principle, this method would require examining a combinatorial number of swap sequences and would be far too slow for the input constraints.

The key insight for an optimal solution is to count the frequency of each fruit in both baskets. For the baskets to be equal, the total count of each fruit across both baskets must be even, since each fruit will eventually be split evenly between the two baskets. Once we know the surplus fruits in each basket, we can pair the most expensive surplus in one basket with the cheapest surplus in the other basket to minimize the total cost. Additionally, we should consider using the smallest fruit globally as a mediator for double swaps, as swapping two expensive fruits via the smallest fruit can reduce the cost further.

This observation reduces the problem to counting frequencies, identifying mismatched fruits, and computing an optimal cost using a greedy strategy.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O((n!)^2) | O(n) | Tries all possible swaps exhaustively, infeasible for n up to 10^5 |
| Optimal | O(n log n) | O(n) | Count frequencies, identify surpluses, sort, and greedily pair for minimal cost |

## Algorithm Walkthrough

1. Compute the frequency of each fruit across both baskets using a hash map. This allows us to quickly determine if the total count of a fruit is even, which is necessary for the baskets to be equal.
2. Check if any fruit has an odd total count. If so, return `-1` immediately because it is impossible to make the baskets equal.
3. Identify the fruits that are in surplus in each basket. A surplus is a fruit that occurs more than half of its total count in one basket.
4. Separate these surplus fruits into two arrays: `surplus1` for `basket1` and `surplus2` for `basket2`. The length of these two arrays will be equal because each excess in one basket corresponds to a deficiency in the other.
5. Sort both surplus arrays: `surplus1` in ascending order and `surplus2` in descending order. This allows pairing the cheapest surplus in one basket with the most expensive in the other, minimizing cost.
6. Compute the minimum fruit cost across both baskets. This value is used to determine whether using a double-swap via the smallest fruit is cheaper than swapping directly.
7. Iterate over the paired surpluses. For each pair, compute the cost as the minimum of the smaller fruit and either the other fruit or twice the smallest fruit (if that reduces cost). Accumulate this total cost.
8. Return the accumulated cost as the minimum cost to equalize the baskets.

Why it works: The algorithm guarantees the baskets are equalized by ensuring that all surplus fruits are paired properly. Sorting ensures that cheaper swaps are used first, and considering the global minimum fruit as a mediator guarantees that we never miss a cheaper double-swap option. This greedy approach produces the minimum total cost.

## Python Solution

```python
from collections import Counter
from typing import List

class Solution:
    def minCost(self, basket1: List[int], basket2: List[int]) -> int:
        count = Counter(basket1) + Counter(basket2)
        
        # Check if any fruit has odd total count
        for val in count.values():
            if val % 2 != 0:
                return -1
        
        # Determine surplus fruits in each basket
        surplus1, surplus2 = [], []
        basket1_count, basket2_count = Counter(basket1), Counter(basket2)
        for fruit in count:
            total = count[fruit]
            half = total // 2
            if basket1_count[fruit] > half:
                surplus1.extend([fruit] * (basket1_count[fruit] - half))
            elif basket2_count[fruit] > half:
                surplus2.extend([fruit] * (basket2_count[fruit] - half))
        
        if not surplus1:
            return 0
        
        surplus1.sort()
        surplus2.sort(reverse=True)
        min_fruit = min(min(basket1), min(basket2))
        
        cost = 0
        for a, b in zip(surplus1, surplus2):
            cost += min(2 * min_fruit, min(a, b))
        return cost
```

In the Python code, we first count the total occurrences of each fruit. We immediately return `-1` if any fruit cannot be evenly distributed. Then, for each fruit, we determine which basket has surplus and build lists of excess fruits. Sorting and pairing the surpluses ensures minimal cost. The check `min(2 * min_fruit, min(a, b))` ensures we take advantage of the global minimum fruit for double swaps if beneficial.

## Go Solution

```go
package main

import (
    "sort"
)

func minCost(basket1 []int, basket2 []int) int64 {
    count := make(map[int]int)
    for _, v := range basket1 {
        count[v]++
    }
    for _, v := range basket2 {
        count[v]++
    }
    
    for _, v := range count {
        if v%2 != 0 {
            return -1
        }
    }
    
    basket1Count := make(map[int]int)
    basket2Count := make(map[int]int)
    for _, v := range basket1 {
        basket1Count[v]++
    }
    for _, v := range basket2 {
        basket2Count[v]++
    }
    
    var surplus1, surplus2 []int
    for fruit, total := range count {
        half := total / 2
        if basket1Count[fruit] > half {
            for i := 0; i < basket1Count[fruit]-half; i++ {
                surplus1 = append(surplus1, fruit)
            }
        } else if basket2Count[fruit] > half {
            for i := 0; i < basket2Count[fruit]-half; i++ {
                surplus2 = append(surplus2, fruit)
            }
        }
    }
    
    if len(surplus1) == 0 {
        return 0
    }
    
    sort.Ints(surplus1)
    sort.Sort(sort.Reverse(sort.IntSlice(surplus2)))
    
    minFruit := basket1[0]
    for _, v := range basket1 {
        if v < minFruit {
            minFruit = v
        }
    }
    for _, v := range basket2 {
        if v < minFruit {
            minFruit = v
        }
    }
    
    var cost int64
    for i := 0; i < len(surplus1); i++ {
        a, b := surplus1[i], surplus2[i]
        if 2*minFruit < min(a, b) {
            cost += int64(2 * minFruit)
        } else {
            cost += int64(min(a, b))
        }
    }
    
    return cost
}
```

In Go, we handle integer slices and maps carefully, ensuring we construct surplus arrays dynamically. We explicitly cast sums to `int64` to avoid overflow issues. Sorting is used similarly to Python. Nil slices in Go behave like empty arrays, so no additional checks are necessary.
