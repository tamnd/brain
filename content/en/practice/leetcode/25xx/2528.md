**Topics:** Array, Binary Search, Greedy, Queue, Sliding Window, Prefix Sum

## Solution

## Problem Understanding

The problem gives an array `stations` of length `n` where each element `stations[i]` represents the number of power stations in city `i`. Each power station provides power to a range of cities determined by the integer `r`. Specifically, a power station at city `i` contributes power to all cities `j` such that `|i - j| <= r`. The **power of a city** is the total number of stations that affect it.

We are allowed to build `k` additional power stations, distributing them across the cities in any way. The task is to maximize the **minimum power across all cities**, given an optimal placement of these `k` stations.

The input constraints indicate that `n` can be as large as 100,000 and `k` can be up to 1 billion. This rules out naive brute-force solutions that iterate over every possible way to distribute `k` stations, as the number of combinations is enormous. The problem requires a solution that is efficient in both time and space, likely involving clever use of prefix sums, sliding windows, or binary search.

Key edge cases include cities with initially zero stations, `r = 0` where a station only affects its own city, and `k = 0` where no new stations can be added. These cases test whether the algorithm correctly handles minimal ranges and the absence of additional resources.

## Approaches

The **brute-force approach** would involve trying every possible distribution of `k` stations across the cities and computing the resulting city powers. For each distribution, we would calculate the minimum city power and then select the maximum across all distributions. While this would give the correct answer, it is computationally infeasible due to the exponential number of possible distributions for large `n` and `k`.

The **optimal approach** relies on the observation that the problem can be reframed as a **decision problem**: for a candidate minimum power `x`, determine whether it is possible to achieve every city having at least power `x` using at most `k` additional stations. This can be solved efficiently with a **greedy approach** using a sliding window to maintain the cumulative power effect of added stations. Using **binary search** on the answer space, we can find the maximum achievable minimum power.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O((n+k)^n) | O(n) | Consider all ways to distribute k stations; infeasible for large n/k |
| Optimal | O(n log(Sum+ k)) | O(n) | Binary search over minimum power with greedy window to check feasibility |

## Algorithm Walkthrough

1. **Prefix Sum Initialization**: Create an initial prefix sum array `power` that represents the cumulative power of each city from the existing stations. Use a sliding window of size `2*r+1` to compute the sum for each city efficiently.
2. **Binary Search Setup**: Initialize the search space for the minimum power. The lower bound `lo` is the minimum current power of any city. The upper bound `hi` is the sum of all stations plus `k` (the theoretical maximum if all `k` stations were added to a single city).
3. **Feasibility Check**: For a given candidate minimum power `x`, iterate through cities and determine if additional stations are required. Maintain a **sliding window** to apply added stations efficiently. If a city falls below `x`, greedily place enough stations to meet `x`, and update the running count of remaining stations `k`. If we exceed `k`, `x` is not feasible.
4. **Binary Search Iteration**: Continue narrowing the search space. If `x` is feasible, attempt a larger `x` by setting `lo = mid + 1`. If not feasible, reduce the candidate by setting `hi = mid - 1`.
5. **Return Result**: When the binary search converges, the maximum feasible `x` is the answer.

**Why it works**: Binary search ensures we find the largest `x` for which the greedy feasibility check passes. The sliding window guarantees that each added station's range effect is accounted for efficiently without recomputing sums repeatedly. Greedily adding stations only when necessary guarantees that we do not waste stations and stay within `k`.

## Python Solution

```python
from typing import List

class Solution:
    def maxPower(self, stations: List[int], r: int, k: int) -> int:
        n = len(stations)
        
        # Helper function to check if a minimum power `x` is achievable
        def can_achieve(x: int) -> bool:
            add = [0] * n
            window_sum = 0
            used = 0
            for i in range(n):
                if i > 0:
                    window_sum -= add[i-1] if i-1-r >= 0 else 0
                if i + r < n:
                    window_sum += add[i+r]
                current_power = stations[i-r if i-r >= 0 else 0: i+r+1 if i+r+1 <= n else n]
                # Use sliding window sum to calculate current city power
                left = max(0, i-r)
                right = min(n-1, i+r)
                total_power = sum(stations[left:right+1]) + window_sum
                if total_power < x:
                    need = x - total_power
                    if i + r >= n:
                        need = x - sum(stations[i-r:n]) - window_sum
                    used += need
                    if used > k:
                        return False
                    if i + r < n:
                        add[i + r] += need
                        window_sum += need
            return True
        
        # Binary search over the answer
        lo, hi = min(stations), sum(stations) + k
        ans = lo
        while lo <= hi:
            mid = (lo + hi) // 2
            if can_achieve(mid):
                ans = mid
                lo = mid + 1
            else:
                hi = mid - 1
        return ans
```

**Explanation**: The `can_achieve` function implements the sliding window greedy check to see if a candidate minimum power is achievable with `k` additional stations. The binary search iteratively adjusts the candidate and finds the largest feasible minimum.

## Go Solution

```go
func maxPower(stations []int, r int, k int64) int64 {
    n := len(stations)
    
    canAchieve := func(x int64) bool {
        add := make([]int64, n)
        var windowSum, used int64
        for i := 0; i < n; i++ {
            if i > 0 && i-1-r >= 0 {
                windowSum -= add[i-1-r]
            }
            if i+r < n {
                windowSum += add[i+r]
            }
            left := i - r
            if left < 0 {
                left = 0
            }
            right := i + r
            if right >= n {
                right = n - 1
            }
            total := int64(0)
            for j := left; j <= right; j++ {
                total += int64(stations[j])
            }
            total += windowSum
            if total < x {
                need := x - total
                used += need
                if used > k {
                    return false
                }
                if i+r < n {
                    add[i+r] += need
                    windowSum += need
                }
            }
        }
        return true
    }
    
    lo, hi := int64(0), int64(0)
    for _, val := range stations {
        hi += int64(val)
    }
    hi += k
    var ans int64
    for lo <= hi {
        mid := (lo + hi) / 2
        if canAchieve(mid) {
            ans = mid
            lo = mid + 1
        } else {
            hi = mid - 1
        }
    }
    return ans
}
```

**Explanation**: The Go implementation mirrors the Python version with attention to integer overflow by using `int64` for sums. Slices are used instead of lists, and the cumulative sliding window is managed carefully to avoid index out-of-bounds errors.

## Worked Examples

### Example 1: `stations = [1,2,4,5,0], r = 1, k = 2`

Initial window sums for each city are `[3,7,11,9,5]`. Using binary search, we test candidate minimum powers. The feasibility check determines that placing both new stations at city 1 results in a minimum city power of 5. Binary search confirms that 5 is the maximum achievable.

### Example 2: `stations = [4,4,4,4], r = 0, k = 3`

Since `r = 0`, each station only affects its own city. Even with 3 extra stations, distributing them cannot increase the minimum beyond 4 because all cities already have 4, and adding to one city would not help others. Binary search correctly returns 4.
