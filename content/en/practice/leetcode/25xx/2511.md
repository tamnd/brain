## Solution

## Problem Understanding

The problem gives us an array called `forts`, where each position represents one of three possibilities:

- `1` means the fort belongs to us.
- `0` means the fort belongs to the enemy.
- `-1` means the position is empty.

We want to move an army from one of our forts to an empty position. However, the movement is only valid if every position between the starting fort and the destination consists entirely of enemy forts.

That means a valid path looks like this:

```
1, 0, 0, 0, -1
```

or the reverse:

```
-1, 0, 0, 1
```

In both cases, the army crosses only enemy forts, and every enemy fort crossed gets captured.

The goal is to determine the maximum number of enemy forts that can be captured during one valid move.

The constraints are small:

- `1 <= forts.length <= 1000`

This means even an `O(n^2)` solution would technically pass. However, the structure of the problem allows a much cleaner linear solution.

Several edge cases are important:

- There may be no fort under our command.
- There may be no empty position.
- A `1` and `-1` may exist, but not with only `0`s between them.
- Adjacent `1` and `-1` positions capture zero forts.
- Multiple candidate segments may exist, and we must choose the maximum.

The key observation is that a valid move only occurs between opposite non-zero endpoints, meaning one side is `1` and the other is `-1`, with only `0`s in between.

## Approaches

### Brute Force Approach

A straightforward solution is to try every possible pair of indices `(i, j)`.

For every pair:

- One endpoint must be `1`
- The other endpoint must be `-1`
- Every position between them must be `0`

If the segment is valid, we count how many `0`s are between the endpoints and update the answer.

This works because it explicitly checks every possible movement path and verifies whether it satisfies the rules.

However, for every pair we may scan the entire interval between them. With `n` positions, this leads to an `O(n^2)` or even `O(n^3)` style implementation depending on how validation is written.

Although acceptable for `n <= 1000`, it is unnecessarily inefficient.

### Optimal Approach

The important observation is that only consecutive non-zero positions matter.

Suppose we scan the array from left to right and remember the last non-zero position we saw.

Whenever we encounter another non-zero value:

- If the two values are different (`1` and `-1` or `-1` and `1`), then every element between them must be checked.
- Since we only update the previous non-zero position when encountering a non-zero value, all positions between them are guaranteed to be zeros.
- Therefore, the number of capturable forts is simply the distance minus one.

For example:

```
1 0 0 0 -1
```

Indices:

```
0       4
```

Captured forts:

```
4 - 0 - 1 = 3
```

This lets us solve the problem in a single linear pass.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Check every possible pair and validate the segment |
| Optimal | O(n) | O(1) | Track consecutive non-zero positions while scanning |

## Algorithm Walkthrough

1. Initialize a variable `max_captured` to `0`. This stores the best answer found so far.
2. Initialize another variable `previous_index` to `-1`. This will store the index of the most recent non-zero position.
3. Traverse the array from left to right using index `i`.
4. If `forts[i]` is `0`, skip it because enemy forts are only meaningful as intermediate positions.
5. If `forts[i]` is non-zero:

- If `previous_index` is not `-1`, we now have two non-zero endpoints.
- Check whether the values at these positions are different.
- If one is `1` and the other is `-1`, then the segment between them contains only zeros.
- The number of captured forts is:

```
i - previous_index - 1
```
- Update `max_captured` if this value is larger.
6. Update `previous_index = i`.
7. After the traversal finishes, return `max_captured`.

### Why it works

The algorithm maintains the invariant that `previous_index` always points to the most recent non-zero position.

Therefore, when another non-zero position is encountered, every element between the two indices must be `0`, because any non-zero value would have replaced `previous_index` earlier.

This guarantees that whenever the endpoints are different, the interval forms a valid movement path and the number of enemy forts captured is exactly the number of positions between them.

Since every valid segment is examined exactly once, the algorithm always finds the maximum possible answer.

## Python Solution

```python
from typing import List

class Solution:
    def captureForts(self, forts: List[int]) -> int:
        max_captured = 0
        previous_index = -1

        for current_index, value in enumerate(forts):
            if value == 0:
                continue

            if previous_index != -1:
                if forts[previous_index] != value:
                    captured = current_index - previous_index - 1
                    max_captured = max(max_captured, captured)

            previous_index = current_index

        return max_captured
```

The implementation follows the exact logic described in the algorithm walkthrough.

The variable `previous_index` stores the most recent non-zero position. During traversal, every `0` is ignored because enemy forts only matter as positions crossed during movement.

Whenever a new non-zero value is found, the code compares it with the previous non-zero value. If they differ, the interval between them forms a valid capture path.

The number of captured forts is computed using:

```python
current_index - previous_index - 1
```

This works because the endpoints themselves are not counted, only the enemy forts between them.

The answer is updated continuously and returned after the scan completes.

## Go Solution

```go
func captureForts(forts []int) int {
    maxCaptured := 0
    previousIndex := -1

    for currentIndex, value := range forts {
        if value == 0 {
            continue
        }

        if previousIndex != -1 {
            if forts[previousIndex] != value {
                captured := currentIndex - previousIndex - 1

                if captured > maxCaptured {
                    maxCaptured = captured
                }
            }
        }

        previousIndex = currentIndex
    }

    return maxCaptured
}
```

The Go implementation mirrors the Python solution closely.

Slices in Go already provide dynamic array behavior, so no additional data structures are necessary. Integer overflow is not a concern because the maximum array length is only `1000`.

The logic remains identical:

- Skip zeros
- Track the previous non-zero position
- Compare endpoint types
- Compute the number of zeros between them

## Edge Cases

One important edge case occurs when the array contains no `1` values at all. Since there is no fort under our command, movement is impossible. A naive implementation might incorrectly count segments between `-1` and `0` values, but this solution only considers pairs of non-zero endpoints with different values, so it correctly returns `0`.

Another tricky case happens when two valid endpoints are adjacent:

```
[1, -1]
```

There are no enemy forts between them, so the captured count should be `0`. The formula:

```
current_index - previous_index - 1
```

naturally handles this by producing zero.

A third important edge case is when multiple non-zero values appear consecutively:

```
[1, -1, 1, -1]
```

Although the endpoints differ, there are no zeros between them. The implementation still behaves correctly because the computed captured value remains zero, and no invalid counting occurs.

A final subtle case involves segments interrupted by another non-zero value:

```
[1, 0, -1, 0, 0, 1]
```

The segment from index `0` to index `5` is not valid because a `-1` appears in the middle. The algorithm avoids this issue automatically because it only compares consecutive non-zero positions. Therefore, interrupted segments are never mistakenly treated as valid capture paths.
