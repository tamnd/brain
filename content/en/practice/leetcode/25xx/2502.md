# LeetCode 2502 - Design Memory Allocator

## Problem Understanding

This problem asks us to simulate a very simple memory allocation system. We are given a memory array of size `n`, where each position represents one memory unit. Initially, every unit is free.

The allocator must support two operations:

1. `allocate(size, mID)`
2. `freeMemory(mID)`

The `allocate` operation must find the leftmost contiguous block of free memory units whose length is exactly at least `size`. Once found, all units in that block are marked with the provided `mID`. The function returns the starting index of the allocated block. If no suitable block exists, the function returns `-1`.

The `freeMemory` operation releases every memory unit currently assigned to the given `mID`. Since the same `mID` may have been allocated multiple times in different regions, we must scan the entire memory and free all matching positions. The function returns how many memory units were freed.

The input constraints are small:

- `n <= 1000`
- At most `1000` operations

These limits strongly suggest that a direct simulation approach is acceptable. We do not need advanced interval trees, segment trees, or balanced search structures. A straightforward array-based implementation is efficient enough.

There are several important edge cases:

- Allocation may fail if there is no sufficiently large contiguous free block.
- Multiple allocations can use the same `mID`.
- Freeing a non-existent `mID` should return `0`.
- Memory can become fragmented, meaning enough total free cells exist but not consecutively.
- Allocation must always choose the leftmost valid block.

The most important detail is the requirement for consecutive free memory. Simply counting total free cells is not enough.

## Approaches

### Brute Force Approach

The most direct solution is to explicitly store the entire memory array and simulate every operation.

For allocation, we scan the array from left to right and try to find `size` consecutive free cells. Once found, we mark all those positions with `mID` and return the starting index.

For freeing memory, we scan the entire array and reset every occurrence of `mID` back to `0`, counting how many positions were freed.

This approach is simple and naturally matches the problem statement. Since the constraints are very small, even repeated full-array scans are efficient enough.

### Key Insight for the Optimal Solution

The key observation is that the constraints are small enough that the simulation itself is already optimal in practice.

A more complicated data structure would add implementation complexity without improving performance meaningfully for `n <= 1000`.

The optimal solution therefore still uses a linear array simulation, but it performs the allocation scan carefully by tracking consecutive free positions while moving from left to right.

Instead of repeatedly checking subarrays from scratch, we maintain a running count of free cells. When the count reaches the required `size`, we immediately know the starting position of the valid block.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) per operation | O(n) | Direct simulation with repeated scanning |
| Optimal | O(n) per operation | O(n) | Linear scan with consecutive free counting |

Although both approaches share the same asymptotic complexity, the optimal version performs the allocation scan more efficiently and cleanly.

## Algorithm Walkthrough

1. Initialize a memory array of size `n`.

Every position starts as `0`, representing free memory. Any positive value represents an allocated memory unit belonging to an `mID`.
2. For the `allocate(size, mID)` operation, scan the memory from left to right.

We maintain a counter called `consecutive_free` that tracks how many free cells have appeared consecutively.
3. If the current cell is free, increment the counter.

If the current cell is occupied, reset the counter back to `0` because the consecutive sequence has been interrupted.
4. When `consecutive_free == size`, we have found a valid block.

The starting index is:

```
current_index - size + 1
```
5. Mark all positions in that block with `mID`.

This reserves the memory for the allocation request.
6. Return the starting index immediately.

Since we scan from left to right, the first valid block found is automatically the leftmost one.
7. If the scan finishes without finding a valid block, return `-1`.
8. For the `freeMemory(mID)` operation, scan the entire memory array.

Every time a cell equals `mID`, reset it to `0` and increment a counter.
9. Return the total number of freed cells.

### Why it works

The algorithm maintains the invariant that the memory array always accurately reflects the current allocation state.

During allocation, scanning left to right guarantees that the first valid contiguous free block found is the leftmost possible block. During deallocation, scanning the entire memory guarantees that every occurrence of the specified `mID` is freed, including allocations spread across multiple regions.

Because every operation directly updates the memory array consistently, all future operations operate on correct memory state.

## Python Solution

```python
class Allocator:

    def __init__(self, n: int):
        self.memory = [0] * n

    def allocate(self, size: int, mID: int) -> int:
        consecutive_free = 0

        for index in range(len(self.memory)):
            if self.memory[index] == 0:
                consecutive_free += 1
            else:
                consecutive_free = 0

            if consecutive_free == size:
                start = index - size + 1

                for position in range(start, start + size):
                    self.memory[position] = mID

                return start

        return -1

    def freeMemory(self, mID: int) -> int:
        freed_count = 0

        for index in range(len(self.memory)):
            if self.memory[index] == mID:
                self.memory[index] = 0
                freed_count += 1

        return freed_count

# Your Allocator object will be instantiated and called as such:
# obj = Allocator(n)
# param_1 = obj.allocate(size, mID)
# param_2 = obj.freeMemory(mID)
```

The implementation begins by creating a memory array filled with zeros. Each zero represents an available memory unit.

The `allocate` method performs a linear scan across the array. The variable `consecutive_free` tracks how many free cells appear consecutively. Whenever an occupied cell is encountered, the counter resets because the contiguous block requirement has been broken.

Once the counter reaches the required `size`, the algorithm computes the starting index of the block and fills those positions with `mID`.

The `freeMemory` method simply scans the entire memory array. Every occurrence of `mID` is reset back to `0`, and the number of freed cells is counted and returned.

The implementation directly mirrors the problem statement, making it easy to reason about and verify.

## Go Solution

```go
type Allocator struct {
	memory []int
}

func Constructor(n int) Allocator {
	return Allocator{
		memory: make([]int, n),
	}
}

func (this *Allocator) Allocate(size int, mID int) int {
	consecutiveFree := 0

	for i := 0; i < len(this.memory); i++ {
		if this.memory[i] == 0 {
			consecutiveFree++
		} else {
			consecutiveFree = 0
		}

		if consecutiveFree == size {
			start := i - size + 1

			for j := start; j < start+size; j++ {
				this.memory[j] = mID
			}

			return start
		}
	}

	return -1
}

func (this *Allocator) FreeMemory(mID int) int {
	freedCount := 0

	for i := 0; i < len(this.memory); i++ {
		if this.memory[i] == mID {
			this.memory[i] = 0
			freedCount++
		}
	}

	return freedCount
}

/**
 * Your Allocator object will be instantiated and called as such:
 * obj := Constructor(n);
 * param_1 := obj.Allocate(size, mID);
 * param_2 := obj.FreeMemory(mID);
 */
```

The Go implementation closely follows the Python version. The memory array is represented using a slice of integers.

One notable difference is that Go requires explicit struct definitions and method receivers. The allocator state is stored inside the `Allocator` struct as a slice named `memory`.

Since the problem constraints are small, integer overflow is not a concern. Go slices naturally support efficient indexed access for the linear scans used in allocation and freeing operations.

| Measure | Complexity | Explanation |

| Time | O(n) per operation | Both allocation and freeing may scan the entire memory array |
| Space | O(n) | The memory array stores all memory units |

Since `n <= 1000`, these linear operations are very efficient in practice.

## Edge Cases

### Fragmented Free Space

One subtle bug source is fragmented memory. The allocator requires consecutive free units, not merely enough total free space.

For example:

```
[0, 0, 1, 0, 0]
```

There are four free cells total, but no block of size `3`. A naive implementation that only counts total free memory would incorrectly allow the allocation.

The implementation correctly handles this by tracking consecutive free cells during the scan.

### Multiple Allocations with the Same mID

The same `mID` can appear in several separate regions of memory. A buggy implementation might stop after freeing the first matching block.

The solution scans the entire memory array during `freeMemory`, guaranteeing that every matching cell is released.

### Allocation at the End of Memory

Another common off-by-one issue occurs when the valid allocation block ends exactly at the final index.

For example:

```
[1, 1, 0, 0]
```

Allocating size `2` should succeed starting at index `2`.

The formula:

```
start = current_index - size + 1
```

correctly computes the block start even when the allocation reaches the array boundary.
