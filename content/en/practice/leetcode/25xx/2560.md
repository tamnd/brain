## Solution

## Problem Understanding

This problem is a variation of the classic House Robber problem, with an added twist. We have a street of consecutive houses, each containing some amount of money represented by the array `nums`. A robber wants to steal money from these houses, but with the restriction that **no two adjacent houses can be robbed**.

The robber has a "capability" defined as the **maximum amount stolen from a single house among the robbed houses**. The goal is **not to maximize the total stolen money**, but to **minimize the robber’s capability** while ensuring at least `k` houses are robbed.

The input is an integer array `nums` of length up to 105, with values up to 10^9. The integer `k` guarantees that it is always possible to steal from at least `k` non-adjacent houses.

Key points to note:

- We must rob **at least k houses**, not exactly k.
- The robber’s capability is dictated by the house with the **largest value among the robbed houses**.
- Values in `nums` can be very large, so naive enumeration is infeasible.

Important edge cases:

- `nums` has the smallest possible length, e.g., `nums = [1]` and `k = 1`.
- All values in `nums` are the same.
- `k` is at its maximum allowed value `(len(nums) + 1) / 2`.

## Approaches

### Brute-Force Approach

A brute-force approach would try **all combinations of non-adjacent houses**, compute the capability for each combination, and pick the minimum capability among combinations that rob at least `k` houses.

This is correct because it exhaustively evaluates all possibilities. However, the time complexity is exponential because for each house, we have the choice to rob or skip it while maintaining the non-adjacent constraint. With `n` up to 105, this is computationally infeasible.

### Optimal Approach

The key insight is that the problem can be reduced to a **decision problem**: for a given candidate capability `x`, can we rob at least `k` houses such that no robbed house exceeds `x`?

If we can answer this efficiently, we can use **binary search** over the values in `nums` to find the **minimum possible capability**.

To check if a capability `x` is feasible, we iterate through `nums` and greedily select a house if its value is ≤ `x` and the previous house was not robbed. We count how many houses can be robbed in this manner. If the count reaches `k`, `x` is feasible.

This reduces the problem from exponential time to **O(n log(max(nums)))**, where `n` is the length of `nums`. The binary search checks potential capability values, and the greedy check validates feasibility for each.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(n) | Enumerate all valid non-adjacent subsets |
| Optimal | O(n log M) | O(1) | Binary search capability + greedy selection; M = max(nums) |

## Algorithm Walkthrough

1. **Initialize Binary Search Bounds**: Set `low` to the minimum value in `nums` and `high` to the maximum value. These represent the possible range of capabilities.
2. **Binary Search Loop**: While `low < high`, compute `mid = (low + high) // 2`.
3. **Feasibility Check**: Iterate through `nums`. Keep a flag `rob_prev` to track if the previous house was robbed. For each house, if `nums[i] <= mid` and `rob_prev` is False, rob this house (increment count) and set `rob_prev = True`. Otherwise, skip this house and set `rob_prev = False`.
4. **Update Bounds**: After checking all houses, if we managed to rob at least `k` houses, `mid` is feasible, so set `high = mid`. Otherwise, set `low = mid + 1`.
5. **Return Result**: When binary search ends, `low` (or `high`) is the minimum capability that allows robbing at least `k` houses.

**Why it works**: The binary search ensures that we find the smallest possible `x` such that at least `k` houses can be robbed. The greedy selection is safe because choosing the earliest possible houses that do not exceed `x` maximizes the number of houses robbed for that capability.

## Python Solution

```python
from typing import List

class Solution:
    def minCapability(self, nums: List[int], k: int) -> int:
        low, high = min(nums), max(nums)
        
        def can_rob(cap: int) -> bool:
            count, rob_prev = 0, False
            for money in nums:
                if money <= cap and not rob_prev:
                    count += 1
                    rob_prev = True
                else:
                    rob_prev = False
            return count >= k
        
        while low < high:
            mid = (low + high) // 2
            if can_rob(mid):
                high = mid
            else:
                low = mid + 1
        return low
```

The `can_rob` function iterates over the houses and tries to rob as many as possible without exceeding the candidate capability. The binary search finds the minimum such capability.

## Go Solution

```go
func minCapability(nums []int, k int) int {
    low, high := nums[0], nums[0]
    for _, val := range nums {
        if val < low {
            low = val
        }
        if val > high {
            high = val
        }
    }

    canRob := func(cap int) bool {
        count, robPrev := 0, false
        for _, money := range nums {
            if money <= cap && !robPrev {
                count++
                robPrev = true
            } else {
                robPrev = false
            }
        }
        return count >= k
    }

    for low < high {
        mid := (low + high) / 2
        if canRob(mid) {
            high = mid
        } else {
            low = mid + 1
        }
    }
    return low
}
```

The Go implementation mirrors the Python version. It uses a closure `canRob` for the greedy feasibility check. The logic is identical, and integer division handles `mid` calculation safely.

## Worked Examples

**Example 1:** `nums = [2,3,5,9], k = 2`

Binary search over `low = 2`, `high = 9`.

1. `mid = 5`, check feasibility: rob houses `[2,5]`, count = 2 ≥ k → feasible → high = 5
2. `mid = 3`, check feasibility: rob `[2,5]` not possible (5>3), rob `[2,3]`, count = 1 < k → not feasible → low = 4
3. `mid = 4`, check feasibility: count = 1 < k → not feasible → low = 5

Result: `5`

**Example 2:** `nums = [2,7,9,3,1], k = 2`

1. `low = 1`, `high = 9` → `mid = 5`, rob `[2,3,1]`, count = 3 ≥ k → feasible → high = 5
2. `mid = 3`, rob `[2,3,1]`, count = 3 ≥ k → feasible → high = 3
3. `mid = 2`, rob `[2,1]`, count = 2 ≥ k → feasible → high = 2
4. `mid = 1`, rob `[1]`, count = 1 < k → not feasible → low = 2

Result: `2`
