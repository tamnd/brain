## Solution

## Problem Understanding

This problem is asking us to determine how many cells will be colored after `n` minutes if we follow a simple rule on an infinite 2D grid: we start by coloring a single cell, and every minute afterward, every uncolored cell adjacent to a blue cell becomes blue. The adjacency here includes only the four cardinal directions: up, down, left, and right.

The input `n` is a positive integer representing the number of minutes this process runs. The output is a single integer representing the total number of colored cells after `n` minutes.

Constraints `1 <= n <= 10^5` indicate that brute-force simulation of each cell on a grid will be too slow because the number of affected cells grows quadratically with `n`. We must find a mathematical pattern rather than simulating the entire grid.

Important edge cases include the smallest input `n = 1`, where only the initial cell is colored, and large inputs near `10^5`, where performance becomes critical. The problem guarantees `n` is positive, so we do not need to handle zero or negative values.

## Approaches

A brute-force approach would simulate the coloring process using a 2D grid. We could store colored cells in a set or hash map, and for each minute, iterate through all currently colored cells to color their neighbors. This would give the correct answer because it directly implements the problem's rules. However, it is extremely inefficient because the number of cells grows roughly as a diamond pattern, i.e., quadratically with `n`. For `n = 10^5`, there could be over `2*10^10` operations, which is infeasible.

The key insight is to recognize the pattern of growth. If we observe the coloring process, the shape forms a diamond. At minute `n`, the diamond extends `n - 1` units from the center in all four cardinal directions. The number of cells in such a diamond can be expressed as a simple formula: the total number of colored cells at minute `n` is `2 * n * (n - 1) + 1`. This formula comes from counting the number of cells in each layer added per minute, where each new layer contributes `4 * (n - 1)` new cells to the total.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^2) | O(n^2) | Simulate grid expansion for each minute; too slow for large `n` |
| Optimal | O(1) | O(1) | Use the derived formula `2 * n * (n - 1) + 1` to compute the total directly |

## Algorithm Walkthrough

1. Observe the pattern of growth. After 1 minute, there is 1 cell. After 2 minutes, there are 5 cells. After 3 minutes, there are 13 cells. We see a diamond pattern expanding in layers.
2. Recognize that each new layer of the diamond contributes `4 * (n - 1)` cells at minute `n`. The center cell contributes 1.
3. Derive a closed formula for the total number of cells after `n` minutes. The sum of all cells up to minute `n` forms an arithmetic series: `1 + 4*(1 + 2 + ... + (n-1))`.
4. Simplify the series formula: `1 + 4 * ((n-1)*n/2) = 1 + 2 * n * (n - 1)`.
5. Return this value as the result. No loops or additional data structures are needed.

**Why it works:** Each minute adds a new "layer" around the previous diamond. By counting the number of cells per layer using the formula, we capture the total without iterating explicitly. The invariant is that at minute `k`, the diamond's radius is `k-1`, which allows exact counting.

## Python Solution

```python
class Solution:
    def coloredCells(self, n: int) -> int:
        return 2 * n * (n - 1) + 1
```

This Python implementation directly computes the total number of colored cells using the derived formula. The multiplication and addition are simple arithmetic operations. No loops or conditional statements are required, which makes this O(1) time and space.

## Go Solution

```go
func coloredCells(n int) int64 {
    return int64(2*n*(n-1) + 1)
}
```

In Go, we explicitly cast the result to `int64` because `n` can be as large as `10^5`, and `2*n*(n-1) + 1` may exceed the default `int32` range. Other than type casting, the logic is identical to the Python version.

## Edge Cases

The first edge case is `n = 1`. This is the simplest scenario and ensures the formula correctly handles the starting cell without any layers.

The second edge case is the maximum input `n = 10^5`. Since the number grows very large, it is essential to use a data type that can hold large numbers, such as `int64` in Go. Python handles large integers natively.

The third edge case is small consecutive inputs like `n = 2, 3, 4`, which test that the arithmetic progression for the new layers is applied correctly and consistently. This ensures the formula generalizes to any `n` and the pattern of growth is captured accurately.
