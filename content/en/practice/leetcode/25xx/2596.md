# LeetCode 2596 - Check Knight Tour Configuration

## Problem Understanding

This problem asks us to verify whether a given matrix represents a valid knight's tour on a chessboard.

The input is an `n x n` matrix called `grid`. Every cell contains a unique integer from `0` to `n * n - 1`. The value inside each cell tells us the exact order in which the knight visited that position during its tour.

For example, if `grid[2][3] = 7`, that means the knight visited cell `(2, 3)` as the 7th move in the sequence.

A valid knight's tour must satisfy several conditions:

1. The knight must start at the top-left corner `(0, 0)`.
2. Every cell must be visited exactly once.
3. Each consecutive move must follow the legal movement rules of a chess knight.

A knight moves in an L-shape:

- Two squares vertically and one horizontally
- Or two squares horizontally and one vertically

This creates exactly eight possible move directions from any position.

The problem guarantees that:

- All numbers are unique
- All numbers are within the valid range
- The board size is between `3` and `7`

These constraints are important because they tell us the maximum board size is very small. Since `n <= 7`, the largest board contains only `49` cells. This means we do not need highly advanced optimization techniques. A direct simulation approach is more than efficient enough.

Several edge cases are important to recognize early.

The first critical case is when the starting cell is not `0`. Since the knight must begin at the top-left corner, any configuration where `grid[0][0] != 0` is immediately invalid.

Another important case is when all numbers are present and unique, but one transition between consecutive numbers is not a legal knight move. A naive implementation might incorrectly assume uniqueness alone is sufficient.

A third tricky case involves very small boards where knight movement possibilities are limited. Even though the constraints start at `n = 3`, many random arrangements will fail because legal knight movement is restrictive.

## Approaches

### Brute Force Approach

A brute force solution would attempt to reconstruct the knight's movement by repeatedly searching the entire matrix for the next move number.

For every value from `0` to `n * n - 1`, we could scan the board to locate its coordinates. Then we would compare consecutive positions to determine whether the move is valid.

This approach works because it explicitly reconstructs the movement sequence and checks every transition.

However, it is inefficient because each lookup requires scanning the entire board. Since there are `n^2` values and each search costs `O(n^2)`, the overall complexity becomes `O(n^4)`.

Even though the constraints are small enough that this might still pass, it is unnecessarily expensive and not the cleanest design.

### Optimal Approach

The key insight is that we should preprocess the board so we can instantly determine the coordinates associated with each move number.

Since every number from `0` to `n^2 - 1` appears exactly once, we can create an array where:

- `positions[k] = (row, col)` of move `k`

After building this mapping, validating the knight's tour becomes straightforward.

For every consecutive pair:

- Move `k`
- Move `k + 1`

we simply check whether their coordinate difference forms a valid knight move.

A knight move always satisfies one of these patterns:

- `(|dr|, |dc|) = (2, 1)`
- `(|dr|, |dc|) = (1, 2)`

This approach avoids repeated board scans and processes the matrix efficiently in linear time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^4) | O(1) | Repeatedly scans board to locate next move |
| Optimal | O(n^2) | O(n^2) | Stores coordinates for direct lookup |

## Algorithm Walkthrough

1. First, verify that the knight starts at the top-left corner.

Since the problem explicitly requires the knight to begin at `(0, 0)`, we immediately return `false` if `grid[0][0] != 0`.
2. Create a positions array of size `n * n`.

Each index represents a move number, and the stored value will be the coordinates where that move occurred.
3. Traverse the entire matrix.

For each cell `(row, col)`, read its move number:

```
move = grid[row][col]
```

Store:

```
positions[move] = (row, col)
```

After this step, we can instantly retrieve the coordinates of any move number.
4. Iterate through all consecutive move pairs.

For every move `k` from `0` to `n * n - 2`:

- Retrieve the coordinates of move `k`
- Retrieve the coordinates of move `k + 1`
5. Compute the movement difference.

Let:

```
dr = abs(r1 - r2)
dc = abs(c1 - c2)
```
6. Check whether the move is a legal knight move.

A move is valid only if:

```
(dr == 2 and dc == 1)
```

or

```
(dr == 1 and dc == 2)
```
7. If any move is invalid, immediately return `false`.

Since every transition must be legal, a single invalid move invalidates the entire configuration.
8. If all moves pass validation, return `true`.

### Why it works

The algorithm works because the matrix uniquely defines the knight's visit order. By reconstructing the exact sequence of visited positions, we can verify every required transition.

A valid knight's tour requires that every consecutive pair of moves follows legal knight movement rules. Since we check all consecutive transitions exactly once, the algorithm guarantees correctness.

If every move satisfies the knight movement constraints and the tour starts at `(0, 0)`, then the configuration is valid.

## Python Solution

```python
from typing import List

class Solution:
    def checkValidGrid(self, grid: List[List[int]]) -> bool:
        n = len(grid)

        # The knight must start at the top-left corner
        if grid[0][0] != 0:
            return False

        # positions[k] = (row, col) where move k occurred
        positions = [None] * (n * n)

        for row in range(n):
            for col in range(n):
                move = grid[row][col]
                positions[move] = (row, col)

        # Check every consecutive move
        for move in range(n * n - 1):
            row1, col1 = positions[move]
            row2, col2 = positions[move + 1]

            row_diff = abs(row1 - row2)
            col_diff = abs(col1 - col2)

            if not (
                (row_diff == 2 and col_diff == 1) or
                (row_diff == 1 and col_diff == 2)
            ):
                return False

        return True
```

The implementation begins by checking whether the starting position is valid. If the top-left cell does not contain `0`, the function immediately returns `False`.

Next, the algorithm constructs the `positions` array. This converts the board representation into a direct mapping from move number to coordinates. This preprocessing step is important because it allows constant time lookup for every move.

The final loop iterates through all consecutive move numbers. For each pair, the code computes the row and column differences and validates whether the movement matches one of the two legal knight movement patterns.

If any transition fails validation, the function immediately exits with `False`. Otherwise, after all transitions are checked successfully, the function returns `True`.

## Go Solution

```go
package main

import "math"

func checkValidGrid(grid [][]int) bool {
	n := len(grid)

	// The knight must start at (0, 0)
	if grid[0][0] != 0 {
		return false
	}

	// positions[k] = {row, col}
	positions := make([][2]int, n*n)

	for row := 0; row < n; row++ {
		for col := 0; col < n; col++ {
			move := grid[row][col]
			positions[move] = [2]int{row, col}
		}
	}

	for move := 0; move < n*n-1; move++ {
		row1, col1 := positions[move][0], positions[move][1]
		row2, col2 := positions[move+1][0], positions[move+1][1]

		rowDiff := int(math.Abs(float64(row1 - row2)))
		colDiff := int(math.Abs(float64(col1 - col2)))

		valid :=
			(rowDiff == 2 && colDiff == 1) ||
				(rowDiff == 1 && colDiff == 2)

		if !valid {
			return false
		}
	}

	return true
}
```

The Go implementation follows the same logic as the Python version.

One notable difference is that Go does not provide a built-in integer absolute value function in the standard library, so we use `math.Abs` with temporary conversion to `float64`.

The `positions` array is implemented as a slice of fixed-size arrays:

```go
[][2]int
```

This efficiently stores coordinate pairs without additional struct definitions.

Since the constraints are very small, integer overflow is not a concern.

The algorithm performs two linear passes over the `n x n` board. The first builds the move-to-coordinate mapping, and the second validates consecutive moves.

Since the board contains `n^2` cells, both operations together remain `O(n^2)`.

The auxiliary space comes from storing coordinates for every move number.

## Edge Cases

One important edge case occurs when the top-left corner does not contain `0`. Since the knight must start at `(0, 0)`, any configuration violating this condition is automatically invalid. The implementation handles this immediately with an early return before any additional processing.

Another critical edge case involves a board where most moves are valid but one transition fails near the end. This can easily expose bugs in implementations that stop checking too early or incorrectly validate movement rules. The algorithm correctly checks every consecutive pair from `0` through `n^2 - 1`, ensuring no invalid transition is missed.

A third important edge case involves small boards such as `3 x 3`. Knight movement options are very restricted on small boards, so many configurations may appear plausible but are actually impossible. Since the algorithm explicitly validates movement geometry rather than relying on assumptions about numbering patterns, it correctly handles these constrained layouts.
