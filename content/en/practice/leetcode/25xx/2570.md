## Solution

## Problem Understanding

The problem gives us two sorted 2D integer arrays, `nums1` and `nums2`. Each element inside these arrays has the form `[id, value]`.

The `id` represents a unique identifier, and the `value` represents the number associated with that identifier.

For example:

```
[4, 5]
```

means:

```
id = 4
value = 5
```

Both arrays satisfy two important guarantees:

1. Every `id` inside a single array is unique.
2. Each array is already sorted in ascending order by `id`.

The task is to merge these two arrays into one sorted array.

If the same `id` appears in both arrays, we combine them into a single entry whose value is the sum of the two values.

For example:

```
[4, 5] and [4, 1]
```

becomes:

```
[4, 6]
```

If an `id` exists in only one array, we simply include it unchanged in the result.

The output must also remain sorted in ascending order by `id`.

The constraints are relatively small:

- Each array length is at most 200.
- IDs and values are at most 1000.

Even though the input size is small enough for several approaches to pass, the problem structure strongly suggests using an efficient merge process because:

- Arrays are already sorted.
- IDs are unique within each array.

This is very similar to the merge step of merge sort.

Several edge cases are important:

- One array may contain IDs that never appear in the other array.
- All IDs may overlap completely.
- No IDs may overlap at all.
- One array may be much shorter than the other.
- Remaining elements from one array must still be appended after the other array is exhausted.

The problem guarantees that:

- IDs are unique within each array.
- Arrays are sorted.
- Each element always contains exactly two integers.

These guarantees simplify the implementation significantly.

## Approaches

### Brute Force Approach

A straightforward solution is to use a hash map.

We iterate through `nums1` and store every `(id, value)` pair inside a dictionary where:

```
key = id
value = accumulated sum
```

Then we iterate through `nums2` and add each value to the same dictionary.

After processing both arrays, the dictionary contains the correct summed values for every ID.

Finally, because dictionaries are not automatically sorted by key order for algorithmic purposes, we sort the keys and build the final result array.

This approach works because every ID is processed exactly once from each array, and summing values inside the map correctly merges duplicate IDs.

The downside is that we ignore the fact that the arrays are already sorted. We still need an additional sorting step at the end.

### Optimal Approach

The key observation is that both arrays are already sorted by ID.

Whenever two collections are sorted, we should think about the two pointer merge technique.

We maintain:

- One pointer for `nums1`
- One pointer for `nums2`

At each step:

- If the IDs are equal, we sum the values and move both pointers.
- If one ID is smaller, we add it directly and move only that pointer.

Because the arrays are sorted, once we process an ID, we never need to revisit it.

This allows us to merge the arrays in a single linear pass, exactly like the merge operation in merge sort.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O((n + m) log(n + m)) | O(n + m) | Uses hash map and sorting |
| Optimal | O(n + m) | O(n + m) | Uses two pointers and leverages sorted order |

## Algorithm Walkthrough

1. Initialize two pointers, `i` and `j`, both starting at `0`.

Pointer `i` tracks the current position in `nums1`, and pointer `j` tracks the current position in `nums2`.
2. Create an empty result array.

This array will store the merged output.
3. Compare the current IDs from both arrays.

Let:

```
id1 = nums1[i][0]
id2 = nums2[j][0]
```
4. If the IDs are equal:

- Add their values together.
- Append `[id, summed_value]` to the result.
- Move both pointers forward.

This works because both arrays contain the same ID exactly once.
5. If `id1 < id2`:

- Append the current element from `nums1`.
- Move pointer `i`.

Since arrays are sorted, `id1` cannot appear later in `nums2`.
6. If `id2 < id1`:

- Append the current element from `nums2`.
- Move pointer `j`.
7. Continue until one array is exhausted.
8. Append remaining elements.

After one pointer reaches the end, the remaining elements in the other array are already sorted and do not overlap with anything else.
9. Return the result array.

### Why it works

The algorithm always processes the smallest unprocessed ID among both arrays.

Because the arrays are sorted:

- Smaller IDs can never appear later.
- Each ID is processed exactly once.
- Equal IDs are merged immediately when encountered.

Therefore, the result remains sorted and contains exactly one entry per ID with the correct summed value.

## Python Solution

```python
from typing import List

class Solution:
    def mergeArrays(self, nums1: List[List[int]], nums2: List[List[int]]) -> List[List[int]]:
        i = 0
        j = 0
        
        merged = []
        
        while i < len(nums1) and j < len(nums2):
            id1, val1 = nums1[i]
            id2, val2 = nums2[j]
            
            if id1 == id2:
                merged.append([id1, val1 + val2])
                i += 1
                j += 1
            
            elif id1 < id2:
                merged.append([id1, val1])
                i += 1
            
            else:
                merged.append([id2, val2])
                j += 1
        
        while i < len(nums1):
            merged.append(nums1[i])
            i += 1
        
        while j < len(nums2):
            merged.append(nums2[j])
            j += 1
        
        return merged
```

The implementation follows the exact merge procedure described earlier.

We start by initializing two pointers, `i` and `j`, for traversing the arrays.

The main `while` loop continues as long as both arrays still contain unprocessed elements.

Inside the loop:

- If IDs match, we sum the values and advance both pointers.
- If one ID is smaller, we append that pair directly and advance only its pointer.

After the main loop ends, one array may still contain remaining elements. Since these remaining elements are already sorted and cannot overlap with anything else, we append them directly.

Finally, we return the merged result.

## Go Solution

```go
func mergeArrays(nums1 [][]int, nums2 [][]int) [][]int {
    i := 0
    j := 0

    merged := [][]int{}

    for i < len(nums1) && j < len(nums2) {
        id1, val1 := nums1[i][0], nums1[i][1]
        id2, val2 := nums2[j][0], nums2[j][1]

        if id1 == id2 {
            merged = append(merged, []int{id1, val1 + val2})
            i++
            j++
        } else if id1 < id2 {
            merged = append(merged, []int{id1, val1})
            i++
        } else {
            merged = append(merged, []int{id2, val2})
            j++
        }
    }

    for i < len(nums1) {
        merged = append(merged, nums1[i])
        i++
    }

    for j < len(nums2) {
        merged = append(merged, nums2[j])
        j++
    }

    return merged
}
```

The Go implementation mirrors the Python solution closely.

The primary difference is slice handling. In Go, we use `append()` to dynamically grow the result slice.

Go integers are fully sufficient for this problem because the maximum possible summed value is small:

```go
1000 + 1000 = 2000
```

so integer overflow is not a concern.

The implementation also handles empty remaining portions naturally through the cleanup loops.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n + m) | Each array element is processed once |
| Space | O(n + m) | Result array stores all merged elements |

The two pointers each move strictly forward and never revisit elements.

Since every element from both arrays is processed exactly once, the runtime is linear in the combined input size.

The output array may contain every element from both arrays if there are no overlapping IDs, so the space complexity is proportional to `n + m`.

## Edge Cases

### All IDs overlap

A common bug is incorrectly duplicating IDs instead of summing them.

For example:

```
nums1 = [[1,2]]
nums2 = [[1,3]]
```

The correct output is:

```
[[1,5]]
```

Our implementation explicitly checks for equal IDs first and merges them immediately, guaranteeing that each ID appears only once.

### No IDs overlap

Another tricky case occurs when the arrays share no IDs at all.

For example:

```
nums1 = [[1,1],[2,2]]
nums2 = [[3,3],[4,4]]
```

A flawed implementation might accidentally skip elements after pointer movement.

Our two pointer logic safely appends whichever ID is smaller and advances only that pointer, ensuring every element is preserved.

### One array finishes early

Suppose one array is exhausted before the other:

```
nums1 = [[1,1]]
nums2 = [[2,2],[3,3],[4,4]]
```

Without cleanup loops, the remaining elements would never be added.

Our implementation includes two final `while` loops that append all remaining elements from whichever array still contains unprocessed entries.

### Very small arrays

The smallest allowed input size is length `1`.

For example:

```
nums1 = [[5,10]]
nums2 = [[7,20]]
```

The algorithm still works correctly because:

- The main merge loop handles single comparisons naturally.
- Cleanup loops append remaining elements if necessary.
- No special case handling is required.
