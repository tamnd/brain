## Solution

## Problem Understanding

The problem gives us an array `nums`, and defines a special operation that removes the same power of two from two different elements at the same time.

More specifically, if both `nums[i]` and `nums[j]` contain the `k`th bit set to `1`, we may subtract `2^k` from both numbers. Since subtracting `2^k` simply clears that bit, every operation removes one occurrence of a particular bit from two different elements simultaneously.

A subarray is considered beautiful if we can repeatedly apply these operations until every value in the subarray becomes `0`.

The key question is therefore:

"When is it possible to eliminate every set bit in a subarray by removing bits in pairs?"

This immediately suggests parity. Since each operation removes two copies of the same bit, the count of every bit across the subarray must be even. If some bit appears an odd number of times, one copy will remain forever and the subarray cannot become all zeroes.

This transforms the problem into a bit parity problem.

The input size is large:

- `nums.length` can be up to `10^5`
- `nums[i]` can be up to `10^6`

Because of this, any solution that checks all subarrays explicitly in quadratic time will be too slow. We need something close to linear time.

Important edge cases include:

- Arrays containing only zeros. Every subarray is beautiful because no operations are needed.
- Arrays with a single element. A nonzero single element can never be beautiful because there is no second element to pair bits with.
- Large repeated patterns where many prefix states repeat. The optimal solution relies heavily on detecting repeated parity states efficiently.

## Approaches

### Brute Force Approach

A straightforward solution is to examine every possible subarray.

For each subarray, we could count how many times each bit position appears. If every bit count is even, then the subarray is beautiful.

One way to implement this is:

- Iterate over every starting index
- Extend the subarray one element at a time
- Maintain bit frequencies
- After each extension, check whether all bit counts are even

This works because the parity condition is both necessary and sufficient. Every operation removes two equal bits, so every bit count must end even, and if every bit count is even, we can always pair them away.

However, there are `O(n^2)` subarrays. Even with incremental updates, the solution becomes too slow for `n = 100000`.

### Key Insight

The important observation is that bit parity behaves exactly like XOR.

For a given bit:

- Even occurrences produce parity `0`
- Odd occurrences produce parity `1`

XOR naturally tracks this parity.

If the XOR of all numbers in a subarray equals `0`, then every bit appears an even number of times in that subarray.

So the problem becomes:

"Count subarrays whose XOR is zero."

This is a classic prefix XOR problem.

Define:

- `prefixXor[i]` = XOR of elements from index `0` through `i`

Then the XOR of subarray `[l...r]` is:

$\text{subarrayXor} = \text{prefixXor}[r] \oplus \text{prefixXor}[l-1]$

A subarray XOR equals zero when:

$\text{prefixXor}[r] = \text{prefixXor}[l-1]$

So we only need to count how many times each prefix XOR value has appeared before.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n² × B) | O(B) | Checks every subarray and tracks bit frequencies |
| Optimal | O(n) | O(n) | Uses prefix XOR frequencies with a hash map |

Here, `B` is the number of bits, at most about 20 for this problem.

## Algorithm Walkthrough

1. Initialize a hash map called `frequency`.

This map stores how many times each prefix XOR value has appeared so far.
2. Insert the value `0` into the map with frequency `1`.

This represents the empty prefix before the array starts. It allows subarrays starting at index `0` to be counted correctly.
3. Maintain a running variable called `prefix_xor`.

As we iterate through the array, we continuously XOR the current number into it.
4. For each element in `nums`, update:

```
prefix_xor ^= num
```

This gives the XOR of all elements from the beginning up to the current position.

1. Check how many times this same `prefix_xor` has appeared before.

If it appeared `k` times earlier, then there are `k` subarrays ending at the current index whose XOR is zero.

Why? Because identical prefix XOR values cancel each other out.
2. Add that count to the answer.
3. Increment the frequency of the current `prefix_xor`.

Future subarrays may use this prefix as their starting boundary.
4. Continue until the entire array has been processed.

### Why it works

The algorithm relies on the property:

$a \oplus a = 0$

If two prefix XOR values are equal, then the XOR of the elements between them must be zero.

A subarray is beautiful exactly when every bit appears an even number of times, which is equivalent to the subarray XOR being zero. Therefore, counting equal prefix XOR states counts exactly the beautiful subarrays.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Solution:
    def beautifulSubarrays(self, nums: List[int]) -> int:
        frequency = defaultdict(int)
        frequency[0] = 1

        prefix_xor = 0
        beautiful_count = 0

        for num in nums:
            prefix_xor ^= num

            beautiful_count += frequency[prefix_xor]

            frequency[prefix_xor] += 1

        return beautiful_count
```

The implementation begins by creating a hash map called `frequency`. This stores how many times each prefix XOR value has appeared.

We initialize `frequency[0] = 1` because an empty prefix has XOR zero. This is important for handling subarrays that start at index `0`.

The variable `prefix_xor` maintains the cumulative XOR as we traverse the array.

For each number:

- We update the running XOR
- We check how many identical prefix XOR values have already appeared
- Each identical occurrence corresponds to one beautiful subarray ending here
- We add that count to the answer
- Then we record the current prefix XOR for future matches

The entire algorithm works in one pass through the array.

## Go Solution

```go
func beautifulSubarrays(nums []int) int64 {
    frequency := make(map[int]int64)
    frequency[0] = 1

    prefixXor := 0
    var beautifulCount int64 = 0

    for _, num := range nums {
        prefixXor ^= num

        beautifulCount += frequency[prefixXor]

        frequency[prefixXor]++
    }

    return beautifulCount
}
```

The Go implementation follows the exact same logic as the Python version.

One important difference is the return type. The number of subarrays can be as large as approximately `n * (n + 1) / 2`, which exceeds the range of a 32 bit integer for `n = 100000`. Because of this, the solution uses `int64`.

Go maps return the zero value automatically for missing keys, which simplifies the frequency lookup logic.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each element is processed once |
| Space | O(n) | The hash map may store up to `n + 1` distinct prefix XOR values |

The algorithm performs constant time work for each element:

- One XOR operation
- One hash map lookup
- One hash map update

The frequency map can grow to contain one entry per prefix, giving linear space complexity.

## Edge Cases

One important edge case is arrays containing only zeros. Since every element is already zero, no operations are needed. Every possible subarray is therefore beautiful. This can expose bugs in implementations that incorrectly assume at least one operation must occur. The prefix XOR approach handles this naturally because the XOR remains zero throughout the traversal.

Another important case is a single nonzero element, such as `[5]`. A single element cannot pair its set bits with anything else, so it can never become zero. The algorithm correctly handles this because the prefix XOR never repeats, producing zero valid subarrays.

A third tricky case involves repeated prefix XOR states, such as `[1,1,1,1]`. Multiple overlapping subarrays may all have XOR zero. A naive implementation might only count distinct XOR values once, undercounting the answer. The frequency map approach correctly accumulates all previous occurrences, ensuring every valid subarray is counted.
