# LeetCode 2512 - Reward Top K Students

## Problem Understanding

This problem asks us to rank students based on the sentiment of words appearing in their feedback reports. Each student has exactly one report, and every report contains a sequence of lowercase words separated by spaces.

We are given two reference lists:

- `positive_feedback`, containing words that contribute positively to a student's score
- `negative_feedback`, containing words that reduce a student's score

The scoring rules are simple:

- Every positive word adds `3` points
- Every negative word subtracts `1` point
- Words not appearing in either list do not affect the score

We are also given:

- `report[i]`, the textual feedback for a student
- `student_id[i]`, the unique ID corresponding to that report

After computing all student scores, we must rank students:

1. Higher score comes first
2. If scores are equal, smaller student ID comes first

Finally, we return the IDs of the top `k` students in ranked order.

The constraints are important because they guide the algorithm choice. There can be up to `10^4` feedback words and `10^4` reports. Each report can contain many words, so repeatedly scanning the positive and negative lists linearly would be too expensive. This strongly suggests using hash-based lookup structures such as sets.

The problem guarantees that:

- No word exists in both positive and negative lists
- Student IDs are unique
- Reports contain only lowercase letters and spaces
- There is exactly one space between consecutive words

Several edge cases are worth identifying immediately. A report may contain no positive or negative words at all, resulting in a score of zero. Multiple students may end with identical scores, making tie-breaking by student ID essential. A report may contain the same positive or negative word multiple times, and every occurrence contributes independently to the score. Finally, `k` may equal `n`, meaning we may need to return every student after sorting.

## Approaches

### Brute Force Approach

A straightforward solution would process every word in every report and check whether the word exists in the positive or negative arrays by scanning those arrays linearly.

For each report:

1. Split the report into words
2. For each word:

- Scan the positive array to see whether it matches
- Scan the negative array to see whether it matches
3. Compute the score
4. Store `(score, student_id)`
5. Sort all students according to the ranking rules

This approach is correct because every word is checked explicitly against both feedback lists, guaranteeing accurate score calculation.

However, this solution is inefficient. If there are `P` positive words, `N` negative words, and `W` total words across all reports, then each word may require scanning up to `P + N` entries. Since each list may contain up to `10^4` words, the runtime becomes unnecessarily large.

### Optimal Approach

The key observation is that we only need fast membership testing for words.

Hash sets provide average-case `O(1)` lookup time. By converting both feedback lists into sets:

- Checking whether a word is positive becomes constant time
- Checking whether a word is negative also becomes constant time

The algorithm becomes:

1. Convert feedback arrays into hash sets
2. Process each report word-by-word
3. Compute scores efficiently
4. Store `(score, student_id)`
5. Sort using custom ranking rules
6. Return the first `k` student IDs

This dramatically improves performance because we eliminate repeated linear scans through the feedback lists.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(W × (P + N) + n log n) | O(n) | Linear scan for every word lookup |
| Optimal | O(W + n log n) | O(P + N + n) | Uses hash sets for constant-time lookup |

Here:

- `W` is the total number of words across all reports
- `P` is the number of positive words
- `N` is the number of negative words
- `n` is the number of students

## Algorithm Walkthrough

1. Create two hash sets, one for positive words and one for negative words.

Hash sets are ideal because they allow constant-time membership testing. We will repeatedly check whether a word belongs to either category, so fast lookup is critical.
2. Initialize a list to store each student's score and ID.

We need both values later because sorting depends on score first and student ID second.
3. Iterate through every report.

For each report:

- Start the student's score at `0`
- Split the report string into individual words
- For every word:

- If the word exists in the positive set, add `3`
- Else if the word exists in the negative set, subtract `1`
4. Store the computed result as `(score, student_id)`.

Keeping both values together simplifies sorting later.
5. Sort the list using a custom ordering rule.

The ranking rules require:

- Higher scores first
- Smaller IDs first when scores tie

In Python, this can be implemented with:

```
key=lambda x: (-x[0], x[1])
```

The negative score ensures descending order.
6. Extract the first `k` student IDs from the sorted list.

These represent the top-ranked students.

### Why it works

The algorithm works because every report is processed exactly according to the scoring rules. Hash sets ensure accurate and efficient classification of words as positive or negative. After all scores are computed, sorting with the required comparator guarantees that students are ranked first by descending score and then by ascending ID. Since the sorting order exactly matches the problem specification, the returned top `k` students are correct.

## Python Solution

```python
from typing import List

class Solution:
    def topStudents(
        self,
        positive_feedback: List[str],
        negative_feedback: List[str],
        report: List[str],
        student_id: List[int],
        k: int
    ) -> List[int]:

        positive_set = set(positive_feedback)
        negative_set = set(negative_feedback)

        students = []

        for i in range(len(report)):
            score = 0

            words = report[i].split()

            for word in words:
                if word in positive_set:
                    score += 3
                elif word in negative_set:
                    score -= 1

            students.append((score, student_id[i]))

        students.sort(key=lambda x: (-x[0], x[1]))

        return [student for _, student in students[:k]]
```

The implementation begins by converting the positive and negative feedback arrays into sets. This optimization is crucial because reports may contain many words, and repeated membership testing must remain efficient.

The `students` list stores tuples in the form `(score, student_id)`. Keeping both values together makes sorting straightforward.

Each report is processed independently. The report string is split into words using `split()`, which automatically separates words by spaces. Every word is checked against the positive and negative sets. Positive words contribute `+3`, while negative words contribute `-1`.

After all scores are computed, the list is sorted using a custom key. The expression `-x[0]` sorts scores in descending order, while `x[1]` sorts IDs in ascending order when scores are tied.

Finally, the first `k` student IDs are extracted and returned.

## Go Solution

```go
package main

import (
	"sort"
	"strings"
)

func topStudents(
	positive_feedback []string,
	negative_feedback []string,
	report []string,
	student_id []int,
	k int,
) []int {

	positiveSet := make(map[string]bool)
	negativeSet := make(map[string]bool)

	for _, word := range positive_feedback {
		positiveSet[word] = true
	}

	for _, word := range negative_feedback {
		negativeSet[word] = true
	}

	type Student struct {
		score int
		id    int
	}

	students := make([]Student, 0)

	for i := 0; i < len(report); i++ {
		score := 0

		words := strings.Split(report[i], " ")

		for _, word := range words {
			if positiveSet[word] {
				score += 3
			} else if negativeSet[word] {
				score -= 1
			}
		}

		students = append(students, Student{
			score: score,
			id:    student_id[i],
		})
	}

	sort.Slice(students, func(i, j int) bool {
		if students[i].score == students[j].score {
			return students[i].id < students[j].id
		}
		return students[i].score > students[j].score
	})

	result := make([]int, 0, k)

	for i := 0; i < k; i++ {
		result = append(result, students[i].id)
	}

	return result
}
```

The Go implementation follows the same algorithmic structure as the Python version. Since Go does not have a built-in set type, maps with boolean values are used to simulate sets.

A custom `Student` struct stores the score and ID together. Sorting is performed using `sort.Slice` with a comparator implementing the ranking rules directly.

The implementation avoids overflow concerns because the maximum possible score is well within Go's integer range.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(W + n log n) | Processing all words plus sorting students |
| Space | O(P + N + n) | Hash sets and score storage |

The dominant work comes from two phases. First, every word in every report is processed exactly once. Hash set lookups are constant time on average, so this phase costs `O(W)`.

Second, the students are sorted according to the ranking rules. Sorting `n` students requires `O(n log n)` time.

The space usage comes primarily from storing the positive and negative sets along with the list of student scores.

## Edge Cases

One important edge case occurs when multiple students receive identical scores. A careless implementation might sort only by score and ignore the secondary ordering rule. The problem explicitly requires lower student IDs to rank higher in ties. The implementation handles this correctly by sorting with `(−score, student_id)` in Python and equivalent logic in Go.

Another subtle edge case is repeated sentiment words inside the same report. For example, `"good good good"` should contribute `9` points, not just `3`. Some implementations mistakenly treat reports as sets of words instead of sequences of words. This solution processes every word occurrence independently, ensuring repeated words contribute multiple times.

A third important case occurs when reports contain no recognized sentiment words. In such situations, students receive a score of zero. The implementation correctly leaves the score unchanged because words not appearing in either hash set simply do not affect the total.

Finally, when `k = n`, the algorithm must return all students in sorted order. Since the implementation sorts the entire list before slicing the first `k` elements, it naturally handles this case without requiring any special logic.
