# LeetCode 2534 - Time Taken to Cross the Door

## Problem Understanding

This problem asks us to simulate how people pass through a single door over time while following a set of priority rules.

We are given two arrays:

- `arrival[i]` tells us the second when person `i` reaches the door.
- `state[i]` tells us whether that person wants to enter (`0`) or exit (`1`).

Only one person can use the door during any given second. If multiple people are waiting at the same time, the decision depends on how the door was used in the previous second.

The rules are:

- If the door was unused in the previous second, exiting gets priority.
- If the previous usage was entering, entering gets priority.
- If the previous usage was exiting, exiting gets priority.
- Within the same direction, the smaller index goes first.

The output is an array where `answer[i]` represents the exact second when person `i` crosses the door.

The constraints are large:

- Up to `10^5` people
- Arrival times up to `10^5`
- Input already sorted by arrival time

These constraints immediately tell us that an inefficient simulation is not acceptable. A naive approach that repeatedly scans all waiting people every second would become too slow.

Several edge cases are important:

- Multiple people may arrive at exactly the same second.
- The door may remain unused for some seconds.
- Everyone may want the same direction.
- The priority direction can change dynamically depending on previous usage.
- Arrival times are non-decreasing, which allows efficient processing with queues.

The problem guarantees valid inputs and sorted arrival times, which makes queue-based simulation possible.

## Approaches

### Brute Force Approach

A brute-force simulation would process time second by second. At every second, we would:

1. Add all newly arrived people to a waiting structure.
2. Scan every waiting person to determine who should use the door.
3. Apply the priority rules.
4. Advance time.

This approach is correct because it directly models the problem statement exactly as written.

However, the inefficiency comes from repeatedly scanning all waiting people. In the worst case, we could examine nearly all `n` people for each second, leading to quadratic complexity.

With `n = 10^5`, an `O(n^2)` solution would time out.

### Optimal Approach

The key observation is that the problem naturally separates people into two waiting groups:

- People waiting to enter
- People waiting to exit

Within each group, the smallest index must go first. Since people are processed in index order already, simple queues are perfect.

We can maintain:

- One queue for entering
- One queue for exiting
- A variable storing how the door was used in the previous second

At each second:

1. Add everyone who arrives at that time.
2. Decide which queue gets priority.
3. Pop one person from the chosen queue.
4. Record their crossing time.

Another important optimization is handling idle periods efficiently. If both queues are empty, instead of advancing time one second at a time, we can jump directly to the next arrival time.

This reduces the solution to linear time.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeatedly scans waiting people each second |
| Optimal | O(n) | O(n) | Uses two queues and direct simulation |

## Algorithm Walkthrough

1. Initialize two queues:

- `enter_queue` for people wanting to enter
- `exit_queue` for people wanting to exit
2. Create an answer array of size `n` to store crossing times.
3. Maintain:

- `time`, representing the current second
- `index`, pointing to the next unprocessed arrival
- `last_direction`, representing previous door usage:

- `0` for entering
- `1` for exiting
- `-1` if the door was unused last second
4. While there are still unprocessed people or waiting queues are non-empty:

- Add all people whose arrival time is less than or equal to the current time into their respective queues.
5. If both queues are empty:

- Jump `time` directly to the next arrival time.
- Reset `last_direction` to `-1` because the door was idle.
6. Otherwise, determine who gets priority:

- If `last_direction` was `1` or the door was unused, exiting gets priority.
- If `last_direction` was `0`, entering gets priority.
7. If the preferred queue is non-empty:

- Pop the front person.
- Record their crossing time.
- Update `last_direction`.
8. Otherwise:

- Use the other queue.
9. Increment `time` by one second after each crossing.
10. Continue until all people are processed.

### Why it works

The algorithm maintains the exact ordering rules required by the problem. Each queue preserves index ordering automatically because people are inserted in increasing index order. The `last_direction` variable captures the door state from the previous second, which fully determines the next priority rule. Since every second is simulated consistently with the problem statement, the produced crossing times are correct.

## Python Solution

```python
from collections import deque
from typing import List

class Solution:
    def timeTaken(self, arrival: List[int], state: List[int]) -> List[int]:
        n = len(arrival)

        enter_queue = deque()
        exit_queue = deque()

        answer = [0] * n

        time = 0
        index = 0

        # -1 = unused
        # 0 = entering
        # 1 = exiting
        last_direction = -1

        while index < n or enter_queue or exit_queue:

            if not enter_queue and not exit_queue and time < arrival[index]:
                time = arrival[index]
                last_direction = -1

            while index < n and arrival[index] <= time:
                if state[index] == 0:
                    enter_queue.append(index)
                else:
                    exit_queue.append(index)
                index += 1

            if exit_queue and (last_direction == 1 or last_direction == -1):
                person = exit_queue.popleft()
                answer[person] = time
                last_direction = 1

            elif enter_queue and last_direction == 0:
                person = enter_queue.popleft()
                answer[person] = time
                last_direction = 0

            elif exit_queue:
                person = exit_queue.popleft()
                answer[person] = time
                last_direction = 1

            else:
                person = enter_queue.popleft()
                answer[person] = time
                last_direction = 0

            time += 1

        return answer
```

The implementation follows the simulation directly.

The two `deque` structures efficiently support FIFO behavior, which is required because smaller indices must go first within each direction.

The `while` loop continues until all arrivals are processed and both queues are empty.

The idle-time optimization is important. When both queues are empty, the simulation jumps directly to the next arrival time instead of wasting time simulating empty seconds.

The priority logic exactly mirrors the problem statement. If the previous usage was exiting or the door was idle, exiting gets priority. Otherwise, entering gets priority.

Each processed person gets their crossing time recorded immediately.

## Go Solution

```go
package main

func timeTaken(arrival []int, state []int) []int {
	n := len(arrival)

	enterQueue := []int{}
	exitQueue := []int{}

	answer := make([]int, n)

	time := 0
	index := 0

	// -1 = unused
	// 0 = entering
	// 1 = exiting
	lastDirection := -1

	for index < n || len(enterQueue) > 0 || len(exitQueue) > 0 {

		if len(enterQueue) == 0 && len(exitQueue) == 0 && time < arrival[index] {
			time = arrival[index]
			lastDirection = -1
		}

		for index < n && arrival[index] <= time {
			if state[index] == 0 {
				enterQueue = append(enterQueue, index)
			} else {
				exitQueue = append(exitQueue, index)
			}
			index++
		}

		if len(exitQueue) > 0 && (lastDirection == 1 || lastDirection == -1) {
			person := exitQueue[0]
			exitQueue = exitQueue[1:]

			answer[person] = time
			lastDirection = 1

		} else if len(enterQueue) > 0 && lastDirection == 0 {
			person := enterQueue[0]
			enterQueue = enterQueue[1:]

			answer[person] = time
			lastDirection = 0

		} else if len(exitQueue) > 0 {
			person := exitQueue[0]
			exitQueue = exitQueue[1:]

			answer[person] = time
			lastDirection = 1

		} else {
			person := enterQueue[0]
			enterQueue = enterQueue[1:]

			answer[person] = time
			lastDirection = 0
		}

		time++
	}

	return answer
}
```

The Go implementation mirrors the Python logic closely.

Go does not provide a built-in deque, so slices are used as queues. Removing the front element is done with slicing syntax.

Unlike Python, Go requires explicit slice management and type declarations.

No integer overflow concerns exist because the maximum time remains within reasonable bounds for Go's `int` type.

## Edge Cases

### Idle Time Gaps

One subtle edge case occurs when no one is waiting at the door for several seconds. A naive simulation might continue incrementing time one second at a time, which can become inefficient.

The implementation handles this by jumping directly to the next arrival time whenever both queues are empty. It also resets `last_direction` to `-1`, correctly modeling that the door was unused in the previous second.

### Multiple Simultaneous Arrivals

Many people may arrive at exactly the same second while wanting different directions. This creates tie-breaking situations where the previous door usage determines priority.

The algorithm correctly processes all arrivals before deciding who crosses. Since people are inserted into queues in index order, the smallest index is automatically preserved within each direction.

### Continuous Usage in One Direction

Another important edge case is when the same direction continues for many consecutive seconds. For example, a long stream of exiting people should continuously block entering people if both are waiting.

The `last_direction` variable preserves this behavior exactly. If the previous second used exiting, then exiting keeps priority until no exiting people remain waiting.
