# LeetCode 2533 - Number of Good Binary Strings

## Problem Understanding

The problem asks us to count how many binary strings satisfy a very specific set of rules.

We are given four integers:

- `minLength`, the minimum allowed length of a valid binary string
- `maxLength`, the maximum allowed length of a valid binary string
- `oneGroup`, the required divisibility condition for groups of consecutive `1`s
- `zeroGroup`, the required divisibility condition for groups of consecutive `0`s

A binary string is considered good if every consecutive block of identical characters has a length that is divisible by the corresponding group size.

For example, suppose:

- `oneGroup = 2`
- `zeroGroup = 3`

Then every consecutive block of `1`s must have length `2`, `4`, `6`, and so on, while every consecutive block of `0`s must have length `3`, `6`, `9`, and so on.

The key observation is that each block can only appear in chunks whose sizes are exact multiples of the required group size. Since consecutive blocks of the same character merge together automatically, every valid string can be constructed by repeatedly appending:

- a block of exactly `oneGroup` ones
- a block of exactly `zeroGroup` zeros

Appending multiple valid blocks consecutively naturally creates larger valid groups.

The output should be the total number of good binary strings whose lengths fall within the inclusive range `[minLength, maxLength]`. Since the number of possibilities can become extremely large, we return the answer modulo:

```
10^9 + 7
```

The constraints are important:

- `maxLength` can be as large as `100000`
- Exponential generation of all binary strings is impossible
- Any acceptable solution must run in approximately linear time

A naive brute force search over all binary strings of length up to `100000` would require examining `2^100000` possibilities, which is completely infeasible.

Several edge cases are important:

- When `oneGroup` or `zeroGroup` is larger than the current length
- When `minLength == maxLength`
- When only one type of block can fit
- When both group sizes are `1`, every binary string becomes valid
- When no valid construction exists for certain lengths

These cases must all be handled correctly by the dynamic programming solution.

## Approaches

### Brute Force Approach

A straightforward idea is to generate every binary string whose length lies between `minLength` and `maxLength`, then check whether each string satisfies the grouping rules.

For every generated string, we would scan through it and compute the sizes of all consecutive groups. Each group of `1`s would need to have size divisible by `oneGroup`, and each group of `0`s would need to have size divisible by `zeroGroup`.

This approach is correct because it explicitly checks every possible candidate string. However, it is far too slow.

For a length `n`, there are `2^n` binary strings. Since `maxLength` can reach `100000`, exhaustive generation is completely impossible.

### Optimal Dynamic Programming Approach

The key insight is that we do not actually need to construct strings character by character.

Instead, we can think about how valid strings are built.

Any valid string can be formed by repeatedly appending:

- a valid block of `oneGroup` consecutive `1`s
- a valid block of `zeroGroup` consecutive `0`s

Suppose we already know how many valid strings exist with length `i`.

Then:

- appending `oneGroup` ones creates valid strings of length `i + oneGroup`
- appending `zeroGroup` zeros creates valid strings of length `i + zeroGroup`

This naturally leads to dynamic programming.

Define:

```
dp[i] = number of valid strings of length i
```

Then:

```
dp[i] += dp[i - oneGroup]
dp[i] += dp[i - zeroGroup]
```

whenever those indices are non-negative.

This works because every valid string of length `i` must end with either:

- a valid block of ones
- a valid block of zeros

The solution becomes a classic counting DP with linear complexity.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^maxLength × maxLength) | O(maxLength) | Generates every binary string and validates groups |
| Optimal Dynamic Programming | O(maxLength) | O(maxLength) | Builds counts incrementally using valid block transitions |

## Algorithm Walkthrough

1. Create a DP array of size `maxLength + 1`.

We define `dp[i]` as the number of good binary strings with exact length `i`.
2. Initialize the base case.

```
dp[0] = 1
```

The empty string counts as one valid construction state. This is important because valid strings are built by appending blocks to previously valid strings.
3. Iterate through all lengths from `1` to `maxLength`.

For each length `currentLength`, we compute how many valid strings can end at that length.
4. Check whether we can append a block of ones.

If:

```
currentLength >= oneGroup
```

then any valid string of length:

```
currentLength - oneGroup
```

can append `oneGroup` consecutive ones.

Therefore:

```
dp[currentLength] += dp[currentLength - oneGroup]
```
5. Check whether we can append a block of zeros.

Similarly, if:

```
currentLength >= zeroGroup
```

then:

```
dp[currentLength] += dp[currentLength - zeroGroup]
```
6. Apply modulo arithmetic after every update.

Since counts become very large, we always compute values modulo:

```
10^9 + 7
```
7. Accumulate the answer.

Whenever the current length lies within:

```
[minLength, maxLength]
```

we add `dp[currentLength]` to the final result.
8. Return the accumulated answer modulo `10^9 + 7`.

### Why it works

The dynamic programming recurrence is correct because every valid binary string must end with either:

- a valid block of `1`s of size `oneGroup`
- a valid block of `0`s of size `zeroGroup`

Removing that final block leaves another valid binary string. Therefore, every valid string of length `i` corresponds uniquely to:

- a valid string of length `i - oneGroup`
- or a valid string of length `i - zeroGroup`

The recurrence counts all possibilities exactly once.

## Python Solution

```python
class Solution:
    def goodBinaryStrings(
        self,
        minLength: int,
        maxLength: int,
        oneGroup: int,
        zeroGroup: int
    ) -> int:
        MOD = 10**9 + 7

        dp = [0] * (maxLength + 1)
        dp[0] = 1

        answer = 0

        for length in range(1, maxLength + 1):

            if length >= oneGroup:
                dp[length] += dp[length - oneGroup]

            if length >= zeroGroup:
                dp[length] += dp[length - zeroGroup]

            dp[length] %= MOD

            if minLength <= length <= maxLength:
                answer += dp[length]
                answer %= MOD

        return answer
```

The implementation follows the dynamic programming formulation directly.

The array `dp` stores the number of valid binary strings for every exact length. The base case `dp[0] = 1` represents the empty construction state.

For every length, we try two possible transitions:

- append a valid block of ones
- append a valid block of zeros

If appending a block is possible, we inherit all valid constructions from the smaller length.

The modulo operation is applied immediately after updates to avoid overflow and keep numbers bounded.

Finally, whenever the current length falls within the required interval, its count contributes to the answer.

## Go Solution

```go
func goodBinaryStrings(minLength int, maxLength int, oneGroup int, zeroGroup int) int {
    const MOD int = 1_000_000_007

    dp := make([]int, maxLength+1)
    dp[0] = 1

    answer := 0

    for length := 1; length <= maxLength; length++ {

        if length >= oneGroup {
            dp[length] += dp[length-oneGroup]
        }

        if length >= zeroGroup {
            dp[length] += dp[length-zeroGroup]
        }

        dp[length] %= MOD

        if length >= minLength {
            answer += dp[length]
            answer %= MOD
        }
    }

    return answer
}
```

The Go implementation mirrors the Python solution closely.

The primary difference is explicit integer handling. Since Go uses fixed-size integers, modulo reduction is especially important to prevent overflow during accumulation.

The DP array is implemented using a slice created with `make`. Unlike Python lists, Go slices require explicit sizing at initialization.

The algorithm performs constant work for every length from `1` to `maxLength`. Since the maximum allowed length is `100000`, linear time is efficient enough.

The DP array requires storing counts for every possible length, resulting in linear space usage.

## Edge Cases

One important edge case occurs when both `oneGroup` and `zeroGroup` are larger than the target length. In this situation, no valid block can be appended, so the DP value remains zero. A naive implementation might accidentally count invalid partial constructions, but the transition checks prevent this.

Another critical case happens when both group sizes equal `1`. Then every binary string is valid because every block length is automatically divisible by `1`. The number of valid strings grows exponentially, so modulo arithmetic becomes essential. The implementation applies modulo reduction after every update to keep values bounded.

A third important edge case is when `minLength == maxLength`. In this scenario, we only count strings of one exact length. The implementation handles this naturally because it accumulates results only for lengths inside the inclusive interval.

A subtle case involves consecutive additions of the same block type. For example, appending two `oneGroup` blocks consecutively creates a larger block whose length is still a multiple of `oneGroup`. The DP formulation handles this correctly because repeated transitions naturally merge into larger valid groups without requiring special logic.
