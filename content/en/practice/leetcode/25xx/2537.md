## Solution

## Problem Understanding

The problem asks us to count how many contiguous subarrays contain at least `k` equal-value pairs.

A pair is defined by two indices `(i, j)` where `i < j` and the values are equal:

$$arr[i] == arr[j]$$

For every possible subarray of `nums`, we need to determine how many equal pairs exist inside it. If the number of pairs is at least `k`, then that subarray is considered good.

The input consists of:

- `nums`, an integer array
- `k`, the minimum number of equal pairs required

The output is a single integer representing the total number of good subarrays.

The constraints are very important:

- `nums.length` can be as large as `10^5`
- `nums[i]` can also be large, up to `10^9`

Because the array length is large, any algorithm worse than roughly linear or near-linear time will likely time out. In particular, checking every subarray explicitly is impossible because there are already `O(n^2)` subarrays.

The values themselves are large, so we cannot use a fixed-size frequency array indexed by value. A hash map is required for counting frequencies efficiently.

Several edge cases are important:

- Arrays with all distinct values, where no pairs exist
- Arrays with many repeated values, where pair counts grow rapidly
- Very small arrays
- Cases where exactly one subarray satisfies the condition
- Cases where many overlapping subarrays are valid
- Large `k` values close to the maximum possible pair count

The key challenge is efficiently maintaining the number of equal pairs while dynamically expanding and shrinking subarrays.

## Approaches

### Brute Force Approach

A straightforward solution is to examine every possible subarray.

For each starting index, we extend the subarray one element at a time and maintain a frequency map. Whenever we add a new element with current frequency `f`, that element forms `f` new equal pairs with previous occurrences of the same value.

This allows us to compute the pair count incrementally for each subarray.

Although this avoids recomputing pair counts from scratch, there are still `O(n^2)` subarrays. With `n = 10^5`, this becomes far too slow.

The brute force method is correct because it explicitly evaluates every possible subarray and counts equal pairs accurately. However, its quadratic time complexity makes it infeasible.

### Optimal Sliding Window Approach

The key observation is that we only need to count subarrays with at least `k` pairs, not the exact pair count for every subarray independently.

This naturally suggests a sliding window approach.

Suppose we maintain a window `[left, right]` and track:

- Frequency of each number inside the window
- Current number of equal pairs inside the window

When we add a new value with frequency `f`, it creates exactly `f` additional pairs.

For example:

- Existing occurrences of `5`: `f`
- Adding another `5`
- New pairs formed: `f`

Similarly, when removing a value with frequency `f`, we remove `f - 1` pairs.

The crucial insight is this:

Once a window has at least `k` pairs, every larger window extending to the right will also satisfy the condition.

That means if window `[left, right]` is valid, then:

- `[left, right]`
- `[left, right + 1]`
- `[left, right + 2]`
- ...

are all valid.

This allows us to count many subarrays at once instead of individually checking each one.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Examines every subarray explicitly |
| Optimal | O(n) | O(n) | Sliding window with frequency counting |

## Algorithm Walkthrough

1. Initialize two pointers, `left` and `right`, representing the sliding window boundaries.
2. Maintain a hash map called `frequency` that stores how many times each number appears inside the current window.
3. Maintain a variable `pairs` that stores the current number of equal pairs in the window.
4. Iterate through the array using `right` as the expanding boundary.
5. When adding `nums[right]`:

- Suppose its current frequency is `f`
- The new occurrence forms `f` additional equal pairs
- Add `f` to `pairs`
- Increment the frequency in the hash map
6. After expanding the window, check whether `pairs >= k`.
7. While the window is valid:

- Every subarray starting at `left` and ending at any index from `right` to `n - 1` is valid
- There are `n - right` such subarrays
- Add this amount to the answer
8. Shrink the window from the left:

- Let the outgoing value have frequency `f`
- Removing it destroys `f - 1` pairs
- Subtract `f - 1` from `pairs`
- Decrement its frequency
- Move `left` forward
9. Continue until the entire array has been processed.

### Why it works

The algorithm maintains an invariant that `pairs` always equals the number of equal-value pairs inside the current window.

Whenever the window becomes valid with `pairs >= k`, all larger windows extending to the right must also remain valid because adding elements cannot reduce the number of equal pairs.

Therefore, once a valid window is found, we can count all future extensions in constant time, which is what allows the algorithm to run in linear time overall.

## Python Solution

```python
from collections import defaultdict
from typing import List

class Solution:
    def countGood(self, nums: List[int], k: int) -> int:
        frequency = defaultdict(int)

        left = 0
        pairs = 0
        answer = 0
        n = len(nums)

        for right in range(n):
            value = nums[right]

            pairs += frequency[value]
            frequency[value] += 1

            while pairs >= k:
                answer += n - right

                left_value = nums[left]

                frequency[left_value] -= 1
                pairs -= frequency[left_value]

                left += 1

        return answer
```

The implementation directly follows the sliding window strategy.

The `frequency` hash map stores how many times each number appears inside the current window. The variable `pairs` tracks the number of equal pairs currently present.

As the `right` pointer expands the window, we add the contribution of the new element. If the element already appeared `f` times, then the new occurrence forms `f` additional pairs.

The inner `while` loop executes whenever the current window already satisfies the requirement of at least `k` pairs.

At that point, every subarray extending further right is also valid, so we add `n - right` to the answer immediately.

Then we shrink the window from the left while maintaining the correct pair count.

The subtle detail is:

```python
pairs -= frequency[left_value]
```

This works because the frequency was already decremented first. After decrementing, the remaining occurrences equal the number of pairs removed.

## Go Solution

```go
func countGood(nums []int, k int) int64 {
    frequency := make(map[int]int)

    left := 0
    pairs := int64(0)
    answer := int64(0)
    n := len(nums)

    for right := 0; right < n; right++ {
        value := nums[right]

        pairs += int64(frequency[value])
        frequency[value]++

        for pairs >= int64(k) {
            answer += int64(n - right)

            leftValue := nums[left]

            frequency[leftValue]--
            pairs -= int64(frequency[leftValue])

            left++
        }
    }

    return answer
}
```

The Go implementation is nearly identical to the Python version.

The main difference is integer handling. Since the number of subarrays can be very large, the result must use `int64`.

The pair count is also stored as `int64` to avoid overflow during accumulation.

Go maps automatically return zero for missing keys, which makes the frequency logic concise and similar to Python's `defaultdict(int)`.

## Worked Examples

### Example 1

Input:

```
nums = [1,1,1,1,1]
k = 10
```

The maximum possible pairs in a length-5 array are:

$$\binom{5}{2} = 10$$

So only the full array works.

| Right | Added | Frequency of 1 | Pairs | Valid? | Answer |
| --- | --- | --- | --- | --- | --- |
| 0 | 1 | 1 | 0 | No | 0 |
| 1 | 1 | 2 | 1 | No | 0 |
| 2 | 1 | 3 | 3 | No | 0 |
| 3 | 1 | 4 | 6 | No | 0 |
| 4 | 1 | 5 | 10 | Yes | 1 |

When `right = 4`, the window has exactly 10 pairs.

We add:

```
n - right = 5 - 4 = 1
```

So the answer becomes `1`.

### Example 2

Input:

```
nums = [3,1,4,3,2,2,4]
k = 2
```

| Right | Value | Pairs After Add | Window | Action |
| --- | --- | --- | --- | --- |
| 0 | 3 | 0 | [3] | Not valid |
| 1 | 1 | 0 | [3,1] | Not valid |
| 2 | 4 | 0 | [3,1,4] | Not valid |
| 3 | 3 | 1 | [3,1,4,3] | Not valid |
| 4 | 2 | 1 | [3,1,4,3,2] | Not valid |
| 5 | 2 | 2 | [3,1,4,3,2,2] | Valid |
| 6 | 4 | 3 | [3,1,4,3,2,2,4] | Valid |

When `right = 5`:

- Window becomes valid
- Add `7 - 5 = 2`
- Valid subarrays:

- `[3,1,4,3,2,2]`
- `[3,1,4,3,2,2,4]`

Then shrink from the left.

Later, when `right = 6`, another valid window contributes additional subarrays.

Final answer:

```
4
```

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each element enters and leaves the sliding window at most once |
| Space | O(n) | Hash map may store up to all distinct elements |

The time complexity is linear because both pointers only move forward. Although there is a nested `while` loop, the total number of left-pointer movements across the entire algorithm is at most `n`.

The space complexity is linear in the worst case when all values are distinct.

## Edge Cases

One important edge case occurs when all elements are distinct. In this situation, no equal pairs can ever exist. A naive implementation might still spend excessive time checking every subarray. The sliding window solution handles this efficiently because the `pairs` count never reaches `k`, so the inner shrinking loop never executes.

Another critical edge case is when the entire array is the only valid subarray. This happens when `k` equals the maximum possible pair count. For example, `[1,1,1,1,1]` with `k=10`. The algorithm correctly waits until the full array has been processed before counting a valid window.

A subtle edge case involves many overlapping valid windows. For example, `[1,1,1]` with `k=1`. Multiple subarrays satisfy the condition simultaneously. The key optimization of adding `n - right` ensures that all valid extensions are counted correctly without enumerating them individually.

Another potential source of bugs is updating the pair count while shrinking the window. The implementation carefully decrements the frequency first, then subtracts the remaining frequency from `pairs`. This guarantees that the exact number of destroyed pairs is removed.
