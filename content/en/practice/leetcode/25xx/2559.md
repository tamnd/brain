## Solution

## Problem Understanding

This problem asks us to process multiple range queries over an array of strings and count how many strings in each queried range satisfy a specific condition: the string must start and end with a vowel. The vowels are defined as the characters `'a'`, `'e'`, `'i'`, `'o'`, and `'u'`.

We are given a list `words`, where each element is a lowercase string, and a list of queries, where each query is a pair `[li, ri]`. For each query, we must consider the subarray `words[li:ri+1]` and count how many strings in that subarray start and end with a vowel.

The output is an array where each element corresponds to the answer for one query.

The constraints are large, with up to 10^5 words and 10^5 queries. This immediately tells us that a naive solution that scans the range for every query will be too slow. Each word has length up to 40, but since we only inspect the first and last character, string length does not dominate complexity. The key challenge is efficient range query processing.

Important edge cases include single-character strings (which may both start and end with a vowel), ranges of length one, and cases where no strings satisfy the condition. The problem guarantees valid indices and lowercase English letters, so we do not need to validate input bounds or character types.

## Approaches

### Brute Force Approach

The brute-force approach directly answers each query by iterating through the range `[li, ri]` and checking each string individually. For each string, we verify whether its first and last characters are vowels. If so, we increment a counter.

This approach is correct because it literally computes the definition of the query. However, it is inefficient because each query may require scanning up to O(n) elements, and there are up to 10^5 queries. This leads to a worst-case complexity of O(n × q), which is too slow for the constraints.

### Optimal Prefix Sum Approach

The key insight is that each word can be preprocessed into a binary indicator: 1 if it is "vowel string" (starts and ends with a vowel), otherwise 0. Once we convert the array into a binary array, each query becomes a range sum query.

To answer range sum queries efficiently, we build a prefix sum array. The prefix sum at index `i` stores the total number of valid strings from index `0` to `i-1`. Then each query `[l, r]` can be answered in O(1) time using subtraction: `prefix[r+1] - prefix[l]`.

This reduces repeated computation and shifts all heavy work into a single preprocessing pass.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × q) | O(1) | Scan each query range directly |
| Prefix Sum | O(n + q) | O(n) | Precompute validity and answer queries in O(1) |

## Algorithm Walkthrough

The optimal algorithm proceeds in a preprocessing phase followed by query answering.

1. First, we define a helper function to determine whether a character is a vowel. This is used to check the first and last character of each word efficiently. The reason for isolating this logic is to keep the main loop clean and avoid repeated string logic.
2. Next, we iterate through the `words` array and construct a binary array `valid`, where each element is 1 if the corresponding word starts and ends with a vowel, otherwise 0. This step compresses the problem into a numeric representation that is easier to aggregate.
3. We then build a prefix sum array `prefix`, where `prefix[i+1] = prefix[i] + valid[i]`. This cumulative structure allows us to query any range sum in constant time.
4. After preprocessing, we iterate through each query `[l, r]`. For each query, we compute the result using the formula `prefix[r+1] - prefix[l]`, which gives the total number of valid strings in that range.
5. We append each computed result into an answer array and return it at the end.

Why it works: the prefix sum array maintains an invariant that each position represents the total number of valid entries up to that index. Because subtraction of prefix sums cancels out the left portion of the range, we are left exactly with the sum over the query interval.

## Python Solution

```python
from typing import List

class Solution:
    def vowelStrings(self, words: List[str], queries: List[List[int]]) -> List[int]:
        vowels = set("aeiou")
        
        def is_vowel(c: str) -> bool:
            return c in vowels
        
        n = len(words)
        prefix = [0] * (n + 1)
        
        for i, w in enumerate(words):
            valid = 1 if is_vowel(w[0]) and is_vowel(w[-1]) else 0
            prefix[i + 1] = prefix[i] + valid
        
        res = []
        for l, r in queries:
            res.append(prefix[r + 1] - prefix[l])
        
        return res
```

The implementation first defines a vowel set for O(1) membership checks. It builds a prefix sum array where each entry accumulates the count of valid words up to that index. Then it processes each query using the difference of prefix values, ensuring constant time per query.

## Go Solution

```go
func vowelStrings(words []string, queries [][]int) []int {
	vowels := map[byte]bool{
		'a': true,
		'e': true,
		'i': true,
		'o': true,
		'u': true,
	}

	n := len(words)
	prefix := make([]int, n+1)

	for i := 0; i < n; i++ {
		first := words[i][0]
		last := words[i][len(words[i])-1]

		valid := 0
		if vowels[first] && vowels[last] {
			valid = 1
		}
		prefix[i+1] = prefix[i] + valid
	}

	result := make([]int, len(queries))
	for i, q := range queries {
		l, r := q[0], q[1]
		result[i] = prefix[r+1] - prefix[l]
	}

	return result
}
```

In Go, we use a `map[byte]bool` for vowel lookup since strings are byte-indexed for ASCII characters. The prefix sum logic remains identical to Python. Slices are used instead of dynamic lists, and integer types are sufficient because counts remain within bounds of input size.

### Example 1

Input:

`queries = [[0,2],[1,4],[1,1]]`

First, we compute validity:

| word | first | last | valid |
| --- | --- | --- | --- |
| aba | a | a | 1 |
| bcb | b | b | 0 |
| ece | e | e | 1 |
| aa | a | a | 1 |
| e | e | e | 1 |

Valid array: `[1, 0, 1, 1, 1]`

Prefix sum:

| i | prefix |
| --- | --- |
| 0 | 0 |
| 1 | 1 |
| 2 | 1 |
| 3 | 2 |
| 4 | 3 |
| 5 | 4 |

Now queries:

Query `[0,2]`: `prefix[3] - prefix[0] = 2 - 0 = 2`

Query `[1,4]`: `prefix[5] - prefix[1] = 4 - 1 = 3`

Query `[1,1]`: `prefix[2] - prefix[1] = 1 - 1 = 0`

Output: `[2,3,0]`

Queries:

Output: `[3,2,1]`

## Edge Cases

One important edge case is a single-character word. Since the same character serves as both first and last, it must be checked carefully. If that character is a vowel, it should be counted as valid. The implementation handles this naturally by checking both `w[0]` and `w[-1]`, which are identical for length-1 strings.

Another edge case is when all words are invalid. In this scenario, the prefix array will remain all zeros, and every query result will correctly compute to zero via subtraction. This ensures the algorithm does not assume at least one valid entry exists.

A third edge case is overlapping large queries, such as multiple queries covering the full range. The prefix sum approach handles this efficiently because each query is independent and constant time, avoiding recomputation even when queries are highly redundant.
