## Solution

## Problem Understanding

This problem asks us to design a small task management system that supports multiple users. Each user can create tasks, retrieve pending tasks, filter tasks by tags, and mark tasks as completed.

The class must support four core operations:

1. Add a new task for a user.
2. Retrieve all incomplete tasks for a user, sorted by due date.
3. Retrieve incomplete tasks for a user that contain a specific tag, also sorted by due date.
4. Mark a task as completed.

Each task contains:

- A unique sequential task ID
- A user ID
- A task description
- A due date
- A list of tags
- A completion status

The important detail is that `dueDate` values are globally unique. This means no two tasks share the same due date. Because of this guarantee, sorting tasks by due date is straightforward and deterministic.

The returned lists should contain only task descriptions, not task IDs or full task objects.

The constraints are very small:

- At most 100 calls per method
- IDs and due dates are at most 100
- Tags per task are at most 100

These constraints tell us that we do not need highly sophisticated data structures such as balanced trees or heaps. A straightforward implementation with sorting during retrieval is perfectly acceptable.

There are several important edge cases:

- A user may not exist yet.
- A user may request tasks when all tasks are completed.
- `completeTask` may be called with the wrong user ID.
- `completeTask` may be called on an already completed task.
- A task may have no tags.
- A tag query may return no matching tasks.

The problem guarantees that due dates are unique, which removes ambiguity during sorting.

## Approaches

### Brute Force Approach

The most direct solution is to store every task in a global list. Whenever we need to retrieve tasks for a user or a tag, we iterate through all tasks and filter the relevant ones.

For `getAllTasks`, we scan every task and keep tasks that:

- Belong to the requested user
- Are not completed

We then sort the filtered tasks by due date.

For `getTasksForTag`, we similarly scan all tasks and additionally check whether the tag exists in the task's tag list.

This approach is correct because every query explicitly checks all tasks and applies the required filtering conditions.

However, this solution becomes inefficient as the number of tasks grows because every retrieval operation scans the entire task list.

### Optimal Approach

A better design stores tasks in a hash map keyed by task ID. Each task object contains all necessary information.

We also maintain:

- A sequential task ID counter
- A mapping from task ID to task data

When retrieving tasks, we still scan all tasks because constraints are small, but lookup and updates become cleaner and faster.

The key insight is that the constraints are tiny, so simplicity is more valuable than building complicated indexing structures. A hash map provides efficient task lookup for completion operations while keeping the implementation straightforward.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) per query | O(n) | Scans all tasks and sorts matching tasks |
| Optimal | O(n log n) per query | O(n) | Uses hash map for direct task lookup and simpler updates |

## Algorithm Walkthrough

1. Initialize the data structures.

We maintain a counter called `next_task_id` starting at 1. We also maintain a hash map called `tasks` where the key is the task ID and the value stores the task information.
2. Add a task.

When `addTask` is called:

- Assign the current `next_task_id`
- Store the task data in the hash map
- Mark the task as incomplete
- Increment `next_task_id`
- Return the assigned task ID

Using a sequential counter guarantees unique IDs.
3. Retrieve all incomplete tasks for a user.

For `getAllTasks`:

- Iterate through all tasks
- Keep only tasks that:

- Belong to the given user
- Are not completed
- Sort the filtered tasks by due date
- Return only the task descriptions
4. Retrieve tasks for a tag.

For `getTasksForTag`:

- Iterate through all tasks
- Keep only tasks that:

- Belong to the given user
- Are not completed
- Contain the requested tag
- Sort by due date
- Return task descriptions
5. Complete a task.

For `completeTask`:

- Check whether the task ID exists
- Verify the task belongs to the given user
- Verify the task is not already completed
- Mark the task as completed

### Why it works

The algorithm works because every operation directly enforces the problem requirements.

Tasks are uniquely identified using sequential IDs. Retrieval methods explicitly filter tasks by user, completion status, and optional tag constraints. Sorting by due date guarantees the required output order. Completion operations safely validate ownership before updating task state.

Because all operations manipulate the single authoritative task map, the system always remains consistent.

## Python Solution

```python
from typing import List

class TodoList:

    def __init__(self):
        self.next_task_id = 1
        self.tasks = {}

    def addTask(
        self,
        userId: int,
        taskDescription: str,
        dueDate: int,
        tags: List[str]
    ) -> int:

        task_id = self.next_task_id
        self.next_task_id += 1

        self.tasks[task_id] = {
            "userId": userId,
            "description": taskDescription,
            "dueDate": dueDate,
            "tags": set(tags),
            "completed": False
        }

        return task_id

    def getAllTasks(self, userId: int) -> List[str]:

        pending_tasks = []

        for task in self.tasks.values():
            if task["userId"] == userId and not task["completed"]:
                pending_tasks.append(
                    (task["dueDate"], task["description"])
                )

        pending_tasks.sort()

        return [description for _, description in pending_tasks]

    def getTasksForTag(self, userId: int, tag: str) -> List[str]:

        matching_tasks = []

        for task in self.tasks.values():
            if (
                task["userId"] == userId
                and not task["completed"]
                and tag in task["tags"]
            ):
                matching_tasks.append(
                    (task["dueDate"], task["description"])
                )

        matching_tasks.sort()

        return [description for _, description in matching_tasks]

    def completeTask(self, userId: int, taskId: int) -> None:

        if taskId not in self.tasks:
            return

        task = self.tasks[taskId]

        if task["userId"] != userId:
            return

        if task["completed"]:
            return

        task["completed"] = True
```

The implementation starts by creating two instance variables:

- `next_task_id` tracks the next available sequential ID.
- `tasks` stores all task information indexed by task ID.

The `addTask` method creates a task dictionary containing all required information. Tags are stored as a set because membership checking is efficient and clean.

The retrieval methods both follow the same pattern:

- Iterate through all stored tasks
- Filter tasks according to the rules
- Store `(dueDate, description)` pairs
- Sort by due date
- Return descriptions only

Using tuples makes sorting simple because Python automatically sorts tuples lexicographically by the first element.

The `completeTask` method carefully validates all required conditions before updating the task state. Invalid requests simply do nothing, exactly as the problem specifies.

## Go Solution

```go
package main

import "sort"

type Task struct {
	UserID     int
	Description string
	DueDate    int
	Tags       map[string]bool
	Completed  bool
}

type TodoList struct {
	nextTaskID int
	tasks      map[int]*Task
}

func Constructor() TodoList {
	return TodoList{
		nextTaskID: 1,
		tasks:      make(map[int]*Task),
	}
}

func (this *TodoList) AddTask(
	userId int,
	taskDescription string,
	dueDate int,
	tags []string,
) int {

	taskID := this.nextTaskID
	this.nextTaskID++

	tagSet := make(map[string]bool)

	for _, tag := range tags {
		tagSet[tag] = true
	}

	this.tasks[taskID] = &Task{
		UserID:      userId,
		Description: taskDescription,
		DueDate:     dueDate,
		Tags:        tagSet,
		Completed:   false,
	}

	return taskID
}

func (this *TodoList) GetAllTasks(userId int) []string {

	type Pair struct {
		dueDate    int
		description string
	}

	var tasks []Pair

	for _, task := range this.tasks {
		if task.UserID == userId && !task.Completed {
			tasks = append(tasks, Pair{
				dueDate:    task.DueDate,
				description: task.Description,
			})
		}
	}

	sort.Slice(tasks, func(i, j int) bool {
		return tasks[i].dueDate < tasks[j].dueDate
	})

	result := []string{}

	for _, task := range tasks {
		result = append(result, task.description)
	}

	return result
}

func (this *TodoList) GetTasksForTag(
	userId int,
	tag string,
) []string {

	type Pair struct {
		dueDate    int
		description string
	}

	var tasks []Pair

	for _, task := range this.tasks {
		if task.UserID == userId &&
			!task.Completed &&
			task.Tags[tag] {

			tasks = append(tasks, Pair{
				dueDate:    task.DueDate,
				description: task.Description,
			})
		}
	}

	sort.Slice(tasks, func(i, j int) bool {
		return tasks[i].dueDate < tasks[j].dueDate
	})

	result := []string{}

	for _, task := range tasks {
		result = append(result, task.description)
	}

	return result
}

func (this *TodoList) CompleteTask(
	userId int,
	taskId int,
) {

	task, exists := this.tasks[taskId]

	if !exists {
		return
	}

	if task.UserID != userId {
		return
	}

	if task.Completed {
		return
	}

	task.Completed = true
}
```

The Go implementation mirrors the Python solution closely but uses explicit structs and typed maps.

A `Task` struct stores all task information. Tags are represented using `map[string]bool` to simulate a hash set.

Because Go does not have built in tuple sorting like Python, temporary `Pair` structs are used during retrieval operations.

Go slices naturally handle empty results, so returning an empty slice satisfies the problem requirements.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | Retrieval scans tasks and sorts matching results |
| Space | O(n) | All tasks are stored in memory |

The dominant cost comes from sorting matching tasks by due date. If there are `n` tasks, filtering takes `O(n)` time and sorting takes `O(n log n)` in the worst case.

The space complexity is linear because every task must be stored permanently until the program ends.

## Edge Cases

One important edge case is attempting to complete a task using the wrong user ID. A naive implementation might mark the task completed without validating ownership. The solution explicitly checks whether the task belongs to the provided user before modifying state.

Another important edge case is querying tasks for a user who has no tasks at all. Some implementations may accidentally return `None` or produce an error when no entries exist. This implementation simply iterates through the task map and naturally returns an empty list.

A third important edge case is tasks without tags. If tags are not handled carefully, membership checks may fail or produce errors. By storing tags as a set in Python and a map in Go, the implementation safely supports empty tag collections.

A fourth subtle edge case is attempting to complete an already completed task. Without validation, repeated operations could corrupt state or trigger unnecessary updates. The implementation safely checks the completion flag and exits early if the task is already completed.
