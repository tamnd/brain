# LeetCode 2595, Number of Even and Odd Bits

## Problem Understanding

The problem gives us a positive integer `n` and asks us to analyze its binary representation. Every bit position in the binary number has an index, starting from the rightmost bit at index `0`. The indices increase as we move left.

We need to count how many bits with value `1` appear at even indices and how many appear at odd indices.

For example, consider the number `50`. Its binary representation is `110010`.

Reading from right to left:

| Index | Bit |
| --- | --- |
| 0 | 0 |
| 1 | 1 |
| 2 | 0 |
| 3 | 0 |
| 4 | 1 |
| 5 | 1 |

The bits with value `1` are at indices `1`, `4`, and `5`.

Index `4` is even, so it contributes to the even count.

Indices `1` and `5` are odd, so they contribute to the odd count.

The final answer is `[1, 2]`.

The input constraint is very small:

- `1 <= n <= 1000`

This means performance is not a major concern, since the binary representation of numbers up to `1000` contains at most 10 bits. Even a straightforward solution would be fast enough.

Still, the problem is designed to test understanding of bit manipulation and binary indexing.

An important detail is that indexing starts from the least significant bit, not from the leftmost bit. This is a common source of mistakes.

Another important guarantee is that `n` is always positive, so we never need to handle `0` as input.

## Approaches

### Brute Force Approach

A brute-force solution converts the number into a binary string using something like `bin(n)` in Python. Then we iterate through the string from right to left and check each character.

For every character equal to `'1'`, we determine whether its position index is even or odd and increment the appropriate counter.

This approach is correct because it explicitly examines every bit in the binary representation and classifies each set bit according to its index parity.

Although this solution is perfectly acceptable for the given constraints, it relies on string conversion instead of direct bit manipulation. It also introduces extra overhead from creating the binary string.

### Optimal Bit Manipulation Approach

The key observation is that we can process the bits directly using bitwise operations.

The least significant bit can be extracted using:

```
n & 1
```

This expression returns `1` if the current bit is set and `0` otherwise.

After processing the current bit, we shift the number right by one position:

```
n >>= 1
```

This removes the bit we just processed and exposes the next bit.

At the same time, we maintain a position counter to track whether the current bit index is even or odd.

This approach is efficient, avoids string conversion, and demonstrates direct binary manipulation.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(log n) | O(log n) | Converts number to binary string and scans it |
| Optimal | O(log n) | O(1) | Processes bits directly using bit operations |

## Algorithm Walkthrough

1. Initialize two counters, `even_count` and `odd_count`, both starting at `0`.
2. Initialize a variable `bit_index` to `0`. This tracks the current bit position while scanning from right to left.
3. While `n` is greater than `0`, repeatedly process the least significant bit.
4. Extract the current bit using `n & 1`. If the result is `1`, then the current bit is set.
5. If the current bit is set, check whether `bit_index` is even or odd.

- If `bit_index % 2 == 0`, increment `even_count`.
- Otherwise, increment `odd_count`.
6. Shift the number right by one bit using `n >>= 1`. This discards the bit we just processed.
7. Increment `bit_index` by `1` so the next iteration processes the next bit position.
8. Continue until all bits have been processed.
9. Return `[even_count, odd_count]`.

### Why it works

The algorithm processes every bit exactly once, starting from the least significant bit. At each iteration, `bit_index` correctly represents the position of the current bit because the number is shifted right after processing. Every set bit is classified according to whether its index is even or odd, so the final counts are correct.

## Python Solution

```python
from typing import List

class Solution:
    def evenOddBit(self, n: int) -> List[int]:
        even_count = 0
        odd_count = 0
        bit_index = 0

        while n > 0:
            current_bit = n & 1

            if current_bit == 1:
                if bit_index % 2 == 0:
                    even_count += 1
                else:
                    odd_count += 1

            n >>= 1
            bit_index += 1

        return [even_count, odd_count]
```

The implementation begins by initializing counters for even and odd indexed set bits. The variable `bit_index` tracks the current position in the binary representation.

Inside the loop, `n & 1` extracts the least significant bit. If that bit equals `1`, the algorithm checks whether the current index is even or odd and updates the corresponding counter.

The statement `n >>= 1` shifts the number right by one position, removing the processed bit. The index counter is then incremented.

Once all bits have been processed and `n` becomes `0`, the function returns the two counts as a list.

## Go Solution

```go
func evenOddBit(n int) []int {
	evenCount := 0
	oddCount := 0
	bitIndex := 0

	for n > 0 {
		currentBit := n & 1

		if currentBit == 1 {
			if bitIndex%2 == 0 {
				evenCount++
			} else {
				oddCount++
			}
		}

		n >>= 1
		bitIndex++
	}

	return []int{evenCount, oddCount}
}
```

The Go implementation follows the same logic as the Python version. Since Go does not have Python's list type, the result is returned as a slice of integers.

Go integers are large enough for the given constraints, so overflow is not a concern here. The right shift operator works directly on integers in the same way as in Python.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |

The loop runs once for every bit in `n`. A number with value `n` has approximately `log2(n)` bits, so the time complexity is `O(log n)`.

The algorithm uses a constant amount of extra memory regardless of input size, so the space complexity is `O(1)`.

## Edge Cases

One important edge case is the smallest valid input, `n = 1`. Its binary representation is simply `1`, which contains a set bit only at index `0`. Since index `0` is even, the correct answer is `[1, 0]`. Implementations that incorrectly start indexing from `1` would fail on this case.

Another important edge case is when all set bits appear only at even indices or only at odd indices. For example, `5` has binary representation `101`, where both set bits are at even positions. Similarly, `10` has binary representation `1010`, where both set bits are at odd positions. These cases verify that the implementation correctly separates counts by parity.

A third important edge case is a power of two, such as `512`. Powers of two contain exactly one set bit. This is useful for verifying that the bit indexing logic remains correct for larger positions. Since `512` equals `2^9`, the only set bit is at index `9`, which is odd, so the correct answer is `[0, 1]`.

A final edge case involves numbers with many consecutive set bits, such as `255`, whose binary representation is `11111111`. This checks that the loop processes every bit correctly and does not accidentally skip or double count positions during shifting.
