## Solution

## Problem Understanding

The problem gives us a target score and several question types. Each question type is represented as:

- `counti`, the maximum number of questions available for that type
- `marksi`, the number of points each question of that type gives

We must determine how many distinct ways exist to earn exactly `target` points.

A key detail is that questions of the same type are indistinguishable. This means we only care about how many questions of each type we solve, not which exact questions. For example, if there are three identical questions worth 2 points each, solving the first and second questions is considered the same as solving the second and third questions. Therefore, each type behaves like a bounded resource where we can choose between `0` and `counti` questions.

The task is essentially:

- For every question type, choose how many questions to solve
- The chosen counts must stay within the allowed limit
- The total score must equal `target`
- Return the total number of valid combinations modulo `10^9 + 7`

The constraints are important:

- `target <= 1000`
- `n <= 50`
- `counti <= 50`
- `marksi <= 50`

These constraints strongly suggest a dynamic programming solution. A brute-force enumeration of all combinations would explode exponentially because every type can contribute many choices.

Important edge cases include:

- When the target cannot be reached at all
- When only one exact combination exists
- When multiple types share the same score value
- When large counts create many possible combinations
- When the target is very small, such as `1`
- When using all questions exactly matches the target

Because the answer can become very large, every intermediate result must use modulo arithmetic.

## Approaches

### Brute Force Approach

The brute-force solution tries every possible number of solved questions for each type.

For each type `i`, we can choose:

- `0` questions
- `1` question
- `2` questions
- ...
- `counti` questions

We recursively explore every combination and compute the total score. Whenever the total becomes exactly `target`, we count one valid way.

This approach is correct because it exhaustively checks every legal configuration. However, it becomes extremely slow because the number of possibilities grows exponentially.

Suppose each type has roughly 50 choices and there are 50 types. The search space becomes enormous:

$$51^{50}$$

Even with pruning, this is infeasible.

### Optimal Dynamic Programming Approach

The key observation is that this problem resembles a bounded knapsack counting problem.

Instead of exploring every configuration independently, we can build solutions incrementally.

Define:

- `dp[s]` = number of ways to achieve score `s`

Initially:

- `dp[0] = 1`
- There is exactly one way to score zero points, solve nothing.

For each question type, we create a new DP array and try taking:

- `0` questions of this type
- `1` question
- ...
- `count` questions

If taking `k` questions contributes `k * marks` points, then:

$$new\_dp[current + k \cdot marks] += dp[current]$$

This works because every type is processed exactly once, preventing duplicate counting.

The constraints are small enough that iterating through all scores and counts is efficient.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | O(n) recursion stack | Explores every possible combination |
| Optimal Dynamic Programming | O(n × target × count) | O(target) | Efficient bounded knapsack counting |

## Algorithm Walkthrough

### Step 1: Initialize the DP Array

Create an array `dp` of size `target + 1`.

`dp[s]` represents the number of ways to earn exactly `s` points using the processed question types so far.

Initialize:

```
dp[0] = 1
```

This means there is one way to earn zero points before solving any questions.

### Step 2: Process Each Question Type

For every question type:

```
count, marks = question_type
```

we create a fresh array called `next_dp`.

This prevents overwriting states that are still needed during the current iteration.

### Step 3: Try Every Existing Score

For every score already reachable:

```
for current_score in range(target + 1):
```

if `dp[current_score] > 0`, then we already know how many ways exist to achieve that score.

### Step 4: Try Taking Different Numbers of Questions

For the current type, try solving:

```
0, 1, 2, ..., count
```

questions.

Each choice adds:

$$used \times marks$$

points.

The new score becomes:

$$new\_score = current\_score + used \times marks$$

If this does not exceed the target, add the number of ways:

```
next_dp[new_score] += dp[current_score]
```

Apply modulo after every addition.

### Step 5: Move to the Next Type

After processing all possibilities for the current type:

```
dp = next_dp
```

Now `dp` represents solutions using all processed types so far.

### Step 6: Return the Final Answer

At the end:

```
return dp[target]
```

This contains the number of ways to earn exactly the target score.

### Why it works

The dynamic programming state maintains an invariant:

- After processing the first `i` question types, `dp[s]` equals the number of ways to achieve score `s` using only those types.

For every type, we consider all legal quantities from `0` to `count`. Since each type is processed exactly once, every valid combination is counted exactly once. No invalid combinations are included because we never exceed the allowed count or target score.

Therefore, when all types are processed, `dp[target]` is precisely the number of valid ways to earn the target score.

## Python Solution

```python
from typing import List

class Solution:
    def waysToReachTarget(self, target: int, types: List[List[int]]) -> int:
        MOD = 10**9 + 7
        
        dp = [0] * (target + 1)
        dp[0] = 1
        
        for count, marks in types:
            next_dp = [0] * (target + 1)
            
            for current_score in range(target + 1):
                if dp[current_score] == 0:
                    continue
                
                for used in range(count + 1):
                    new_score = current_score + used * marks
                    
                    if new_score > target:
                        break
                    
                    next_dp[new_score] = (
                        next_dp[new_score] + dp[current_score]
                    ) % MOD
            
            dp = next_dp
        
        return dp[target]
```

The implementation begins by defining the modulo constant and initializing the DP array. The entry `dp[0] = 1` establishes the base case that there is one way to achieve zero points.

For each question type, a new DP array is created. This separation is important because it prevents updates from the current type from affecting other transitions within the same iteration.

The outer loop iterates through every currently reachable score. If no ways exist for a score, the algorithm skips it for efficiency.

The inner loop tries every valid number of questions from the current type. Each choice generates a new score, and the number of ways is added into the next DP state.

The break condition:

```python
if new_score > target:
    break
```

avoids unnecessary computation because additional questions would only increase the score further.

Finally, after all types are processed, the answer is stored in `dp[target]`.

## Go Solution

```go
func waysToReachTarget(target int, types [][]int) int {
    const MOD int = 1e9 + 7

    dp := make([]int, target+1)
    dp[0] = 1

    for _, t := range types {
        count := t[0]
        marks := t[1]

        nextDP := make([]int, target+1)

        for currentScore := 0; currentScore <= target; currentScore++ {
            if dp[currentScore] == 0 {
                continue
            }

            for used := 0; used <= count; used++ {
                newScore := currentScore + used*marks

                if newScore > target {
                    break
                }

                nextDP[newScore] =
                    (nextDP[newScore] + dp[currentScore]) % MOD
            }
        }

        dp = nextDP
    }

    return dp[target]
}
```

The Go implementation follows the same logic as the Python version.

Slices are used instead of dynamic lists. Since Go integers can safely hold values up to the modulo range here, standard `int` is sufficient.

A new slice `nextDP` is allocated for every question type to avoid overwriting states during iteration. This mirrors the Python implementation exactly.

The modulo operation is applied during every update to prevent overflow and satisfy the problem requirement.

## Edge Cases

One important edge case occurs when the target cannot be formed at all. For example:

```
target = 5
types = [[1,2],[1,4]]
```

The only possible totals are 0, 2, 4, and 6. A naive implementation might accidentally count invalid states or overflow beyond the target. The DP solution avoids this by only updating states whose score does not exceed the target.

Another important case is when a type has a very large count compared to the target. For example:

```
target = 3
types = [[50,1]]
```

Although 50 questions are available, only the first few counts matter because larger totals exceed the target. The implementation handles this efficiently using:

```
if new_score > target:
    break
```

This avoids unnecessary iterations.

A third important edge case involves multiple question types with the same point value. For example:

```
types = [[2,2],[2,2]]
```

Even though both types give the same score per question, they are distinct categories and must be processed separately. The DP formulation naturally handles this because each type contributes transitions independently.

Another subtle case is the zero-score base state. Without initializing:

```
dp[0] = 1
```

no transitions would ever occur. This base case represents the empty selection and is essential for building all future combinations correctly.
