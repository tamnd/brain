## Solution

## Problem Understanding

The problem gives us an integer array `nums` and a fixed window size `k`. For every contiguous subarray of length `k`, we must compute a special value called the frequency score.

The frequency score is defined as follows:

For every distinct value `x` in the subarray:

- Count how many times `x` appears, call this `freq`
- Compute `x^freq`
- Add all such values together
- Take the result modulo `10^9 + 7`

We must return the maximum frequency score among all subarrays of size `k`.

For example, suppose a window contains:

```
[5,4,5,7,4,4]
```

The frequencies are:

- `5` appears `2` times → `5^2 = 25`
- `4` appears `3` times → `4^3 = 64`
- `7` appears `1` time → `7^1 = 7`

The score becomes:

```
25 + 64 + 7 = 96
```

The important detail is that we maximize the value after modulo arithmetic, not the raw mathematical value.

The constraints are large:

- `nums.length` can be up to `10^5`
- `nums[i]` can be up to `10^6`

This immediately tells us that recomputing the frequency score from scratch for every window would likely be too slow. We need an approach close to linear time.

Several edge cases are important:

- All elements may be identical
- All elements may be distinct
- `k` may equal `1`
- `k` may equal `nums.length`
- A value's frequency may increase and decrease repeatedly as the window slides

The main implementation challenge is updating the frequency score efficiently when the sliding window changes by one element.

## Approaches

### Brute Force Approach

The most direct solution is to examine every subarray of size `k`.

For each window:

1. Build a frequency map
2. For every distinct value:

- Compute `value^frequency mod MOD`
3. Sum all contributions
4. Track the maximum score

This works because it directly follows the definition of the problem.

However, it is too slow.

There are `O(n)` windows. Constructing a frequency map for each window takes `O(k)` time, and computing the score also takes up to `O(k)` time in the worst case.

That gives a total complexity of approximately `O(n * k)`, which can become `10^10` operations when both `n` and `k` are large.

### Optimal Approach

The key observation is that adjacent windows differ by only two elements:

- One element leaves the window
- One element enters the window

Instead of recomputing the entire score from scratch, we can update the score incrementally.

Suppose a number `x` currently appears `f` times.

Its contribution to the score is:

```
x^f
```

If another `x` enters the window:

- Old contribution: `x^f`
- New contribution: `x^(f+1)`

So we can remove the old contribution and add the new one.

Similarly, when an element leaves:

- Remove `x^f`
- Add `x^(f-1)` if frequency remains positive

This makes sliding window updates efficient.

We maintain:

- A frequency hash map
- The current score modulo `MOD`

Each window transition becomes `O(1)` average time, aside from modular exponentiation. Since exponents are at most `k`, Python's built-in modular power is fast enough.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(nk) | O(k) | Rebuilds frequency map for every window |
| Optimal | O(n log k) | O(k) | Sliding window with incremental score updates |

## Algorithm Walkthrough

1. Initialize a hash map called `frequency` to store how many times each number appears in the current window.
2. Maintain a variable `current_score` representing the frequency score of the current window modulo `10^9 + 7`.
3. Process the first `k` elements to build the initial window.

For each number `x`:

- Let its old frequency be `f`
- Remove the old contribution `x^f`
- Increase the frequency
- Add the new contribution `x^(f+1)`

This keeps `current_score` correct at every step.
4. Store the initial window score as the current best answer.
5. Slide the window from left to right.

For every move:

- Remove the outgoing element
- Add the incoming element
6. When removing a value `x`:

- Current frequency is `f`
- Remove contribution `x^f`
- Decrease frequency
- If new frequency becomes positive, add contribution `x^(f-1)`
7. When adding a value `x`:

- Current frequency is `f`
- Remove contribution `x^f`
- Increase frequency
- Add contribution `x^(f+1)`
8. After each window update, compare `current_score` with the best answer and keep the larger value.
9. Return the maximum score found.

### Why it works

At every step, `current_score` exactly equals the sum of:

```
value^(frequency in current window)
```

for every distinct value in the window.

The sliding window only changes frequencies of two elements, so updating only those contributions preserves correctness. Since every window is processed exactly once, and we track the maximum score encountered, the final answer is correct.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Solution:
    def maxFrequencyScore(self, nums: List[int], k: int) -> int:
        MOD = 10**9 + 7
        
        frequency = defaultdict(int)
        current_score = 0
        
        def add_value(x: int) -> None:
            nonlocal current_score
            
            old_freq = frequency[x]
            
            if old_freq > 0:
                current_score -= pow(x, old_freq, MOD)
            
            frequency[x] += 1
            new_freq = frequency[x]
            
            current_score += pow(x, new_freq, MOD)
            current_score %= MOD
        
        def remove_value(x: int) -> None:
            nonlocal current_score
            
            old_freq = frequency[x]
            
            current_score -= pow(x, old_freq, MOD)
            
            frequency[x] -= 1
            new_freq = frequency[x]
            
            if new_freq > 0:
                current_score += pow(x, new_freq, MOD)
            
            current_score %= MOD
        
        for i in range(k):
            add_value(nums[i])
        
        answer = current_score
        
        for right in range(k, len(nums)):
            left = right - k
            
            remove_value(nums[left])
            add_value(nums[right])
            
            answer = max(answer, current_score)
        
        return answer
```

The implementation uses a sliding window and a frequency hash map.

The `add_value` helper function updates the score when a number enters the window. Before increasing the frequency, it removes the old contribution. After increasing the frequency, it adds the updated contribution.

The `remove_value` helper function performs the reverse operation. It removes the contribution corresponding to the old frequency, decreases the count, and restores the smaller contribution if the number still exists in the window.

Because only two values change when the window slides, each transition is efficient.

The modulo operation is applied after every update to keep values bounded and ensure correctness.

## Go Solution

```go
package main

func maxFrequencyScore(nums []int, k int) int {
	const MOD int64 = 1_000_000_007

	frequency := make(map[int]int)
	var currentScore int64

	modPow := func(base, exp int) int64 {
		result := int64(1)
		b := int64(base) % MOD
		
		for exp > 0 {
			if exp&1 == 1 {
				result = (result * b) % MOD
			}
			
			b = (b * b) % MOD
			exp >>= 1
		}
		
		return result
	}

	addValue := func(x int) {
		oldFreq := frequency[x]

		if oldFreq > 0 {
			currentScore -= modPow(x, oldFreq)
		}

		frequency[x]++
		newFreq := frequency[x]

		currentScore += modPow(x, newFreq)

		currentScore %= MOD

		if currentScore < 0 {
			currentScore += MOD
		}
	}

	removeValue := func(x int) {
		oldFreq := frequency[x]

		currentScore -= modPow(x, oldFreq)

		frequency[x]--
		newFreq := frequency[x]

		if newFreq > 0 {
			currentScore += modPow(x, newFreq)
		}

		currentScore %= MOD

		if currentScore < 0 {
			currentScore += MOD
		}
	}

	for i := 0; i < k; i++ {
		addValue(nums[i])
	}

	answer := currentScore

	for right := k; right < len(nums); right++ {
		left := right - k

		removeValue(nums[left])
		addValue(nums[right])

		if currentScore > answer {
			answer = currentScore
		}
	}

	return int(answer)
}
```

The Go version follows the same logic as the Python implementation.

Since Go does not provide built in modular exponentiation for integers, we implement binary exponentiation manually in `modPow`.

The score is stored as `int64` to avoid overflow during multiplication.

Go modulo operations can produce negative values after subtraction, so we explicitly normalize the score back into the range `[0, MOD)`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log k) | Each add/remove performs modular exponentiation |
| Space | O(k) | Frequency map stores at most k distinct values |

Each element enters the window once and leaves once.

For every update, we compute a few modular exponentiations. Fast exponentiation takes `O(log k)` time because the exponent never exceeds the window size.

The hash map stores only frequencies of elements currently inside the window, so its size is bounded by `k`.

## Edge Cases

### All Elements Are Identical

When every value in the array is the same, the frequency map always contains only one key. This can expose bugs where implementations incorrectly remove keys or mishandle frequency decrements.

For example:

```
nums = [5,5,5,5]
k = 3
```

The implementation correctly removes the old contribution and adds the updated one whenever the frequency changes.

### All Elements Are Distinct

If every element is unique, then every frequency remains `1`.

For example:

```
nums = [1,2,3,4]
k = 2
```

The score simply becomes the sum of the window elements. This validates that the implementation handles frequent insertions and removals without relying on repeated frequencies.

### Window Size Equals Array Length

When `k == len(nums)`, there is only one possible subarray.

Some sliding window implementations accidentally perform extra iterations or incorrect boundary handling in this scenario.

The implementation correctly initializes the first window and skips the sliding phase entirely.

### Window Size Is One

When `k = 1`, every subarray contains exactly one number.

The score for a window becomes:

```
x^1 = x
```

So the answer is simply the maximum element in the array. The implementation naturally handles this case without any special branching.
