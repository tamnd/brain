## Solution

## Problem Understanding

The problem gives us a positive integer `n` and asks us to repeatedly transform it using the following rule:

1. Find all prime factors of `n`
2. Add those prime factors together
3. Replace `n` with that sum
4. Continue until the value no longer decreases

The important detail is that repeated prime factors must be counted multiple times. For example:

- `12 = 2 × 2 × 3`
- The sum of prime factors is `2 + 2 + 3 = 7`

We are asked to return the smallest value reached during this process.

The transformation behaves differently depending on whether the current number is prime or composite:

- If the number is prime, its only prime factor is itself, so the sum remains unchanged.
- If the number is composite, the sum of its prime factors is usually smaller than the number itself.

The constraints are relatively small:

- `2 <= n <= 10^5`

This means we can safely use repeated prime factorization with trial division. We do not need advanced factorization algorithms.

A few edge cases are especially important:

- Prime numbers such as `3` or `97` should immediately return themselves.
- Powers of a prime such as `8 = 2 × 2 × 2` must count repeated factors correctly.
- Numbers may require multiple transformation rounds before stabilizing.
- The sequence eventually stops because the value never increases for composite numbers.

## Approaches

### Brute Force Approach

A straightforward solution is to repeatedly factorize the current number by checking every integer from `2` up to `n`.

For each transformation:

1. Try every possible divisor
2. Extract all prime factors
3. Compute their sum
4. Replace the number
5. Continue until the value stops changing

This approach is correct because it directly simulates the process described in the problem statement.

However, repeatedly checking divisibility all the way up to `n` is inefficient. Most divisors are unnecessary because factorization only requires checking up to the square root of the number.

### Optimal Approach

The key observation is that prime factorization can be performed efficiently using trial division up to `sqrt(n)`.

If a number has a factor larger than its square root, then it must also have a corresponding smaller factor. Therefore, once we test all divisors up to `sqrt(n)`, any remaining value greater than `1` must itself be prime.

The optimized algorithm repeatedly:

1. Computes the sum of prime factors using efficient trial division
2. Stops when the transformed value equals the current value

Since the number decreases quickly for composite numbers, the number of iterations is small.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) | O(1) | Factorization checks too many divisors |
| Optimal | O(log n × sqrt(n)) | O(1) | Efficient prime factorization with trial division |

## Algorithm Walkthrough

1. Start with the initial value `n`.
2. Create a helper function that computes the sum of prime factors of a number.
3. Inside the helper function, begin checking divisors from `2`.
4. While the divisor squared is less than or equal to the current number:

- If the divisor divides the number, repeatedly divide it out.
- Each time it divides evenly, add the divisor to the running sum.
- This correctly handles repeated prime factors.
5. Continue increasing the divisor until all smaller factors are processed.
6. After the loop finishes, if the remaining number is greater than `1`, then it is a prime factor itself. Add it to the sum.
7. Return the total sum of prime factors.
8. Back in the main algorithm:

- Compute the transformed value.
- If the transformed value equals the current number, the process has stabilized.
- Return the current number.
- Otherwise, replace `n` with the transformed value and continue.

### Why it works

For every composite number, the sum of its prime factors is less than or equal to the original number. Prime numbers remain unchanged because their only prime factor is themselves. Therefore, the sequence eventually stabilizes at a fixed value, and that stabilized value is the smallest value encountered during the process.

## Python Solution

```python
class Solution:
    def smallestValue(self, n: int) -> int:
        def sum_of_prime_factors(num: int) -> int:
            factor_sum = 0
            divisor = 2

            while divisor * divisor <= num:
                while num % divisor == 0:
                    factor_sum += divisor
                    num //= divisor
                divisor += 1

            if num > 1:
                factor_sum += num

            return factor_sum

        while True:
            new_value = sum_of_prime_factors(n)

            if new_value == n:
                return n

            n = new_value
```

The implementation begins with a helper function called `sum_of_prime_factors`. This function performs prime factorization using trial division.

The variable `divisor` starts at `2`, which is the smallest prime number. The outer loop continues while `divisor * divisor <= num`. This condition ensures we only test divisors up to the square root.

The inner loop repeatedly divides out the same prime factor. Each successful division adds that factor to `factor_sum`. This correctly counts repeated prime factors such as the three `2`s in `8`.

After all smaller factors are processed, any remaining value greater than `1` must itself be prime. That leftover prime is added to the sum.

The main loop repeatedly applies the transformation. Once the transformed value equals the current number, the process has stabilized and the answer is returned.

## Go Solution

```go
func smallestValue(n int) int {
	sumOfPrimeFactors := func(num int) int {
		factorSum := 0
		divisor := 2

		for divisor*divisor <= num {
			for num%divisor == 0 {
				factorSum += divisor
				num /= divisor
			}
			divisor++
		}

		if num > 1 {
			factorSum += num
		}

		return factorSum
	}

	for {
		newValue := sumOfPrimeFactors(n)

		if newValue == n {
			return n
		}

		n = newValue
	}
}
```

The Go implementation follows the same logic as the Python version.

One small difference is that Go uses explicit integer types and function literals. The helper function is implemented as an anonymous function assigned to `sumOfPrimeFactors`.

No special overflow handling is needed because the constraints are small and all intermediate values remain within standard integer limits.

## Edge Cases

A very important edge case is when the input is already prime. For a prime number such as `3` or `97`, the sum of prime factors equals the number itself. A buggy implementation might continue looping forever if it does not correctly detect stabilization. This implementation explicitly checks whether the transformed value equals the current value and returns immediately.

Another important edge case is repeated prime factors. Numbers like `8 = 2 × 2 × 2` or `27 = 3 × 3 × 3` require counting the same prime multiple times. A common mistake is to add each prime factor only once. The inner `while` loop correctly handles multiplicity by repeatedly dividing and repeatedly adding the divisor.

Large composite numbers near the upper constraint also deserve attention. Even though `n` can be as large as `100000`, the implementation remains efficient because it only checks divisors up to the square root. This avoids unnecessary work and ensures the solution runs comfortably within limits.

A subtle edge case occurs when the remaining value after trial division is itself prime. For example, when factorizing `14`, the divisor `2` is found first, leaving `7`. Since `7` is larger than `1` after the loop, it must be added separately. The final `if num > 1` condition correctly handles this situation.
