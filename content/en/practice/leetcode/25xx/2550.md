# LeetCode 2550 - Count Collisions of Monkeys on a Polygon

## Problem Understanding

The problem describes a regular convex polygon with `n` vertices. Each vertex initially contains exactly one monkey. Every monkey must move simultaneously to one of its two neighboring vertices, either clockwise or counterclockwise.

Since each monkey has exactly two choices, the total number of possible movement configurations is `2^n`.

A collision occurs in two situations:

1. Two or more monkeys end up on the same vertex after moving.
2. Two monkeys cross each other while traveling along the same edge in opposite directions.

The task is to count how many movement configurations produce at least one collision.

The answer can become extremely large because `n` can be as large as `10^9`, so the result must be returned modulo `10^9 + 7`.

The key observation from the constraints is that brute force enumeration is impossible. Even for relatively small values such as `n = 50`, the number of configurations is already astronomically large. Therefore, the solution must rely on mathematical reasoning rather than simulation.

An important guarantee is that the polygon is convex and regular, and every monkey moves simultaneously to an adjacent vertex. This structure makes the movement patterns highly predictable.

Several edge cases are important to consider:

- Very small polygons such as `n = 3`
- Extremely large values such as `n = 10^9`
- Correct modular arithmetic handling
- Avoiding integer overflow in languages like Go
- Correctly identifying the only non-collision configurations

## Approaches

### Brute Force Approach

A brute force solution would generate all `2^n` possible movement combinations. For each monkey, we choose either clockwise or counterclockwise movement. After generating a configuration, we simulate all movements and check whether any collision occurs.

This approach is correct because it explicitly examines every possible scenario and directly verifies whether collisions happen.

However, this method is computationally infeasible. The number of configurations doubles with every additional monkey. Since `n` can be as large as `10^9`, exhaustive enumeration is impossible.

### Optimal Mathematical Observation

The key insight is that almost every movement configuration produces a collision.

The only ways to avoid collisions are:

1. Every monkey moves clockwise
2. Every monkey moves counterclockwise

Why do these two configurations work?

If all monkeys move in the same direction, every monkey simply shifts to the neighboring vertex in a perfect rotation. No two monkeys meet at a vertex, and no edges are crossed in opposite directions.

Any mixed movement creates a conflict somewhere on the polygon. At least one pair of adjacent monkeys will move toward each other across the same edge, causing an edge intersection collision.

Therefore:

- Total movement configurations = `2^n`
- Non-collision configurations = `2`
- Collision configurations = `2^n - 2`

The only remaining challenge is computing `2^n mod (10^9 + 7)` efficiently for very large `n`.

We use fast modular exponentiation, also called binary exponentiation.

## Algorithm Walkthrough

1. Define the modulus value as `10^9 + 7`.
2. Compute `2^n mod MOD` using fast exponentiation. Direct exponentiation is impossible because `n` can be extremely large. Binary exponentiation reduces the complexity from linear time to logarithmic time.
3. Subtract `2` from the result because there are exactly two movement configurations without collisions:

- All clockwise
- All counterclockwise
4. Add `MOD` before taking the final modulo to avoid negative values after subtraction.
5. Return the final result.

### Why it works

Every monkey has exactly two movement choices, so the total number of configurations is `2^n`.

A configuration avoids collisions only if all monkeys move in the same direction. Any change in direction creates at least one adjacent pair moving toward each other, which produces an edge collision.

Therefore, exactly two configurations avoid collisions, and all others contain at least one collision. Hence the answer is:

$2^n - 2$

## Python Solution

```python
class Solution:
    def monkeyMove(self, n: int) -> int:
        MOD = 10**9 + 7
        
        return (pow(2, n, MOD) - 2 + MOD) % MOD
```

The implementation is extremely compact because Python provides a built-in modular exponentiation function.

The expression `pow(2, n, MOD)` computes:

$2^n \bmod (10^9+7)$

efficiently using binary exponentiation internally.

We then subtract `2` to remove the two non-collision configurations.

Finally, we add `MOD` before taking `% MOD` to guarantee a non-negative result.

The Go implementation explicitly implements binary exponentiation because Go does not provide a built-in modular power function like Python.

The solution uses `int64` to avoid integer overflow during multiplication. Every multiplication operation is immediately reduced modulo `MOD`, which keeps intermediate values within safe bounds.

The iterative exponentiation loop repeatedly squares the base and processes one bit of the exponent at a time.

## Edge Cases

### Minimum Valid Input

The smallest possible polygon has `n = 3`. Small inputs are often a source of off-by-one mistakes or incorrect assumptions about movement behavior.

The implementation handles this correctly because the mathematical formula applies universally:

$2^3 - 2 = 6$

### Extremely Large Values of `n`

The constraint allows `n` up to `10^9`. A naive exponentiation approach would take far too long.

The implementation avoids this issue using binary exponentiation, which reduces the complexity to `O(log n)`. Even very large exponents are processed efficiently.

### Negative Result After Subtraction

After computing `2^n mod MOD`, subtracting `2` could theoretically produce a negative intermediate value in modular arithmetic.

The implementation safely handles this with:

```
(pow(2, n, MOD) - 2 + MOD) % MOD
```

Adding `MOD` before the final modulo guarantees the result remains non-negative.

### Integer Overflow in Go

In Go, repeated multiplication during exponentiation can overflow standard integer types if not handled carefully.

The implementation uses `int64` and applies modulo reduction immediately after every multiplication, which keeps all intermediate values within safe limits.
