## Solution

## Problem Understanding

The problem asks us to analyze queries on a **complete binary tree** of `2n - 1` nodes, where each node `val` has a left child `2*val` and right child `2*val + 1`. We are given a set of queries where each query temporarily adds an edge between two nodes, creating exactly one cycle because the tree initially has no cycles. The goal is to determine the **length of that cycle** after adding the edge, then conceptually remove the edge to process the next query independently.

The input `n` defines the size of the tree and guarantees that node values range from `1` to `2n - 1`. Each query `[ai, bi]` satisfies `ai != bi`, ensuring the added edge creates a valid cycle. The tree is complete, so every node can be represented efficiently by its value without explicitly building the tree structure. This is crucial given that `n` can be up to `30` and `m` (number of queries) can be as large as `10^5`.

Important edge cases include queries involving the root node, nodes that are parent-child, and nodes on opposite sides of the tree, since these will affect how many steps it takes to reach the least common ancestor (LCA), which determines the cycle length.

## Approaches

The **brute-force approach** is to construct the entire tree as a graph, add the edge for each query, and run a cycle detection algorithm like BFS or DFS to find the cycle length. While correct, this approach is inefficient because constructing a graph of up to `2^30 - 1` nodes is infeasible and running a graph traversal for each query would be extremely slow for `m = 10^5`.

The **optimal approach** leverages the properties of the **complete binary tree**. Since each node has a unique parent (`val // 2`) and paths to the root are easily computed, the cycle length after adding an edge between nodes `u` and `v` can be determined by:

1. Finding the path from `u` to the root.
2. Finding the path from `v` to the root.
3. Identifying the **least common ancestor (LCA)**.
4. Calculating the cycle length as the sum of distances from `u` to LCA and `v` to LCA, plus one for the added edge.

This approach is efficient because traversing from a node to the root requires at most `O(log n)` steps, and no explicit tree structure is needed.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n + m * n) | O(n) | Build full tree, BFS/DFS per query, impractical for n ~ 2^30 |
| Optimal | O(m * log n) | O(1) | Use node values to traverse to root, compute cycle via LCA, no tree needed |

## Algorithm Walkthrough

1. Initialize an empty list `answer` to store the cycle lengths for each query.
2. For each query `[u, v]`:

a. Initialize counters `cu` and `cv` to zero.

b. While `u` is not equal to `v`:

i. If `u > v`, move `u` to its parent `u //= 2` and increment `cu`.

ii. Otherwise, move `v` to its parent `v //= 2` and increment `cv`.

c. After the loop, `u` equals `v`, which is the LCA. The cycle length is `cu + cv + 1` (the +1 accounts for the added edge between original `u` and `v`).

d. Append this length to `answer`.
3. Return `answer`.

**Why it works:** This algorithm works because the added edge creates a cycle consisting of the path from `u` to the LCA, the path from `v` to the LCA, and the edge connecting `u` and `v`. By moving `u` and `v` up toward the root until they meet, we are effectively tracing the paths to their LCA. The sum of steps plus the added edge gives the cycle length.

## Python Solution

```python
from typing import List

class Solution:
    def cycleLengthQueries(self, n: int, queries: List[List[int]]) -> List[int]:
        answer = []
        for u, v in queries:
            cu = cv = 0
            orig_u, orig_v = u, v
            while u != v:
                if u > v:
                    u //= 2
                    cu += 1
                else:
                    v //= 2
                    cv += 1
            answer.append(cu + cv + 1)
        return answer
```

The implementation iterates over each query, counts the steps needed for `u` and `v` to meet at their LCA, and adds one for the query edge. This approach is memory-efficient because it does not store the tree and only uses constant additional variables.

## Go Solution

```go
func cycleLengthQueries(n int, queries [][]int) []int {
    answer := make([]int, len(queries))
    for i, q := range queries {
        u, v := q[0], q[1]
        cu, cv := 0, 0
        for u != v {
            if u > v {
                u /= 2
                cu++
            } else {
                v /= 2
                cv++
            }
        }
        answer[i] = cu + cv + 1
    }
    return answer
}
```

The Go implementation mirrors the Python version, using slices for the answer array and integer division for parent traversal. No special handling for slices or nil values is needed because node values are guaranteed valid.

| --- | --- | --- | --- |

The time complexity is efficient because `log n` for `n <= 2^30` is only 30 steps per query, which is feasible for `m <= 10^5`. Space complexity is minimal, making this approach highly scalable.

## Edge Cases

1. **Root Involvement:** If either `u` or `v` is the root, the path to LCA will be shorter. The algorithm handles this naturally because the loop terminates when `u == v`.
2. **Leaf Nodes:** Adding an edge between two leaves on opposite sides of the tree creates a cycle that traverses up to the LCA. The algorithm counts steps correctly regardless of tree depth.
3. **Adjacent Parent-Child Nodes:** When `u` and `v` are direct parent-child, the cycle length is always 3 (child -> parent -> child). The implementation handles this because traversal still increments the counters correctly before meeting at the parent.
