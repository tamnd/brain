## Solution

## Problem Understanding

The problem gives us two integer arrays, `nums1` and `nums2`, and both arrays are already sorted in non-decreasing order. Our task is to find the smallest integer that appears in both arrays. If the arrays do not share any value, we must return `-1`.

In other words, we need to identify the minimum common element between the two sorted arrays.

For example, if:

```
nums1 = [1,2,3]
nums2 = [2,4]
```

then the answer is `2`, because `2` is the only value that appears in both arrays.

The fact that both arrays are sorted is the key detail of the problem. Sorted arrays allow us to compare values efficiently without checking every pair of elements.

The constraints are important:

- Each array can contain up to `10^5` elements.
- Values can be as large as `10^9`.
- Arrays are already sorted.

Because the arrays may be very large, an inefficient nested-loop solution would be too slow. We need an algorithm that scales well for up to one hundred thousand elements.

Several edge cases are worth noticing immediately:

- The arrays may have no common value at all, so we must correctly return `-1`.
- The smallest common value might appear at the beginning of both arrays.
- One array may contain duplicates.
- Arrays may have very different sizes.
- The first common value encountered while scanning must be returned immediately, because the arrays are sorted and that guarantees it is the minimum common value.

## Approaches

### Brute Force Approach

The most straightforward solution is to compare every element in `nums1` with every element in `nums2`.

For each number in `nums1`, we iterate through all numbers in `nums2` and check whether they are equal. Whenever we find a match, we keep track of the minimum common value.

This approach is correct because it exhaustively checks every possible pair of elements. If a common value exists, we will eventually find it.

However, this method is far too slow for the given constraints. If both arrays contain `10^5` elements, the total number of comparisons becomes:

```
10^5 × 10^5 = 10^10
```

Ten billion operations is not feasible within typical time limits.

### Optimal Approach, Two Pointers

The important observation is that both arrays are already sorted.

Because the arrays are sorted, we never need to move backward. We can maintain one pointer for each array:

- `i` for `nums1`
- `j` for `nums2`

At each step:

- If `nums1[i] == nums2[j]`, we found the smallest common value and can return it immediately.
- If `nums1[i] < nums2[j]`, then `nums1[i]` cannot appear later in `nums2`, because `nums2` is sorted. We move `i` forward.
- If `nums1[i] > nums2[j]`, then `nums2[j]` cannot appear later in `nums1`, so we move `j` forward.

This technique works efficiently because each pointer only moves forward once.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × m) | O(1) | Compares every pair of elements |
| Optimal | O(n + m) | O(1) | Uses two pointers on sorted arrays |

## Algorithm Walkthrough

1. Initialize two pointers, `i = 0` and `j = 0`.

The pointer `i` tracks the current position in `nums1`, while `j` tracks the current position in `nums2`.
2. Continue looping while both pointers remain within array bounds.

We stop once either array is exhausted, because no further common elements can exist after that point.
3. Compare `nums1[i]` and `nums2[j]`.

There are three possible cases.
4. If the two values are equal, return that value immediately.

Since both arrays are sorted, this is guaranteed to be the smallest common value.
5. If `nums1[i] < nums2[j]`, increment `i`.

The current value in `nums1` is too small to match the current value in `nums2`, and because `nums2` is sorted, it will never match any later value in `nums2`.
6. Otherwise, increment `j`.

This means `nums2[j]` is smaller, so it cannot match any future value in `nums1`.
7. If the loop finishes without finding a match, return `-1`.

This means the arrays have no common value.

### Why it works

The algorithm works because the arrays are sorted. At every step, we eliminate one impossible candidate without missing any valid match.

When one value is smaller than the other, that smaller value cannot possibly match any later element in the opposite array, because all future elements are greater than or equal to the current one. Therefore, advancing the pointer of the smaller value is always safe.

Since the first equal pair encountered must be the smallest shared value, returning immediately is correct.

## Python Solution

```python
from typing import List

class Solution:
    def getCommon(self, nums1: List[int], nums2: List[int]) -> int:
        i = 0
        j = 0

        while i < len(nums1) and j < len(nums2):
            if nums1[i] == nums2[j]:
                return nums1[i]

            if nums1[i] < nums2[j]:
                i += 1
            else:
                j += 1

        return -1
```

The implementation follows the two-pointer algorithm directly.

We begin by initializing two indices, `i` and `j`, both starting at `0`. These indices track our current position in each array.

The `while` loop continues as long as both pointers remain inside their respective arrays. Inside the loop, we compare the current elements.

If the values are equal, we immediately return that value because the arrays are sorted and this must be the smallest common element.

If the value in `nums1` is smaller, we increment `i`. Otherwise, we increment `j`.

Eventually, one of the pointers reaches the end of its array. At that point, no common value exists, so the function returns `-1`.

## Go Solution

```go
func getCommon(nums1 []int, nums2 []int) int {
    i := 0
    j := 0

    for i < len(nums1) && j < len(nums2) {
        if nums1[i] == nums2[j] {
            return nums1[i]
        }

        if nums1[i] < nums2[j] {
            i++
        } else {
            j++
        }
    }

    return -1
}
```

The Go implementation is almost identical to the Python version.

Go uses slices instead of Python lists, but the logic remains the same. We use integer indices `i` and `j` to traverse the arrays.

There are no special concerns about integer overflow because the problem constraints fit safely within Go's standard `int` type.

Unlike some languages, Go does not distinguish between empty slices and `nil` slices in this algorithm because both have length `0`, and the loop condition handles them naturally.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n + m) | Each pointer moves through its array at most once |
| Space | O(1) | Only a few variables are used |

The time complexity is linear because each iteration advances either `i` or `j`. Neither pointer ever moves backward, so the total number of operations is proportional to the combined size of the arrays.

The space complexity is constant because we only store two pointers and do not allocate any additional data structures.

## Edge Cases

One important edge case occurs when the arrays have no common value at all. A buggy implementation might accidentally return the last compared value or fail to terminate properly. In this implementation, the loop ends naturally when one pointer reaches the end of its array, and the function correctly returns `-1`.

Another important case involves duplicate values. For example:

```
nums1 = [1,1,2,2]
nums2 = [2,2,3]
```

A careless implementation could repeatedly compare duplicate values inefficiently or skip valid matches incorrectly. The two-pointer method handles duplicates naturally because pointers advance in sorted order and eventually align on the shared value.

A third important edge case is when the smallest common value appears immediately at the beginning of both arrays. Since the algorithm checks equality before advancing pointers, it correctly returns the answer without unnecessary work.

Finally, arrays with very different sizes can expose inefficient solutions. For example, one array may contain only one element while the other contains one hundred thousand elements. The two-pointer solution still performs efficiently because each pointer moves only as needed.
