# LeetCode 2594, Minimum Time to Repair Cars

## Problem Understanding

This problem gives us a list of mechanics, where each mechanic has a rank. The rank determines how quickly that mechanic can repair cars.

If a mechanic has rank `r`, then repairing `n` cars takes:

$t = r \cdot n^2$

The mechanics work simultaneously, which means all of them can repair cars in parallel. Our goal is to determine the minimum amount of time needed so that the total number of repaired cars across all mechanics is at least `cars`.

The input consists of:

- `ranks`, an array where `ranks[i]` is the rank of the `i`th mechanic
- `cars`, the total number of cars that must be repaired

The output is a single integer representing the smallest possible time required to repair all cars.

The constraints are extremely important here:

- Up to `10^5` mechanics
- Up to `10^6` cars
- Ranks are small, at most `100`

These constraints immediately tell us that we cannot simulate repair operations one car at a time. Any algorithm close to `O(cars * mechanics)` would be far too slow.

A few important observations emerge from the formula:

- Lower rank means faster mechanic
- Repair time grows quadratically with the number of cars
- If we know a time limit `T`, we can compute how many cars a mechanic can repair within that time

From:

$r \cdot n^2 \leq T$

we derive:

$n \leq \sqrt{\frac{T}{r}}$

So a mechanic with rank `r` can repair:

$\left\lfloor \sqrt{\frac{T}{r}} \right\rfloor$

cars within time `T`.

Important edge cases include:

- Only one mechanic exists
- Only one car exists
- All mechanics have the same rank
- One mechanic is dramatically faster than the others
- The answer becomes very large, requiring 64-bit integers

The problem guarantees valid positive inputs, so we never need to handle empty arrays or negative values.

## Approaches

### Brute Force Approach

A brute-force approach would try increasing time values one by one and check whether all cars can be repaired within that time.

For every candidate time `T`, we could compute how many cars each mechanic repairs and sum them together. The first time where the total reaches `cars` would be the answer.

This approach is correct because eventually we will encounter the minimum feasible time. However, it is far too slow.

The maximum possible answer can be enormous. Suppose the slowest scenario occurs with one mechanic of rank `100` repairing `10^6` cars:

$100 \cdot (10^6)^2 = 10^{14}$

Iterating through all times up to `10^14` is completely infeasible.

### Optimal Approach, Binary Search on Time

The key insight is that the feasibility condition is monotonic.

If all cars can be repaired within time `T`, then they can also be repaired within any larger time.

Similarly, if time `T` is insufficient, then every smaller time is also insufficient.

This monotonic behavior makes binary search possible.

Instead of checking every possible time, we binary search over the answer space.

For a given time `mid`, we compute how many cars all mechanics can repair combined. If the total is at least `cars`, then `mid` is feasible and we try smaller times. Otherwise, we need more time.

This reduces the search space from linear to logarithmic.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(answer × n) | O(1) | Checks every possible time |
| Optimal | O(n log(answer)) | O(1) | Binary search on feasible repair time |

## Algorithm Walkthrough

1. Initialize the binary search range.

The minimum possible time is `1`.

The maximum possible time occurs if the fastest mechanic repairs every car alone. If the minimum rank is `min_rank`, then:

$\text{maxTime} = \text{minRank} \cdot cars^2$

This guarantees the true answer lies within the search interval.
2. Start binary search.

Compute:

```
mid = (left + right) // 2
```

Here, `mid` represents a candidate repair time.
3. Compute how many cars can be repaired within `mid`.

For each mechanic with rank `r`, solve:

$r \cdot n^2 \leq mid$

Rearranging gives:

$n = \left\lfloor \sqrt{\frac{mid}{r}} \right\rfloor$

Add this value to a running total.
4. Check feasibility.

If the total repaired cars is at least `cars`, then `mid` works. Record it as a candidate answer and search for a smaller valid time.

Otherwise, increase the search range because more time is required.
5. Continue until the search interval collapses.

The final value of `left` will be the minimum feasible repair time.

### Why it works

The algorithm works because feasibility is monotonic.

If a time `T` allows repairing all cars, every larger time also allows it. This property guarantees that the feasible region forms a continuous suffix over the search space.

Binary search correctly finds the smallest feasible value in a monotonic search space.

The per-mechanic formula is mathematically derived from the repair-time equation, so the feasibility check is exact.

## Python Solution

```python
from typing import List
import math

class Solution:
    def repairCars(self, ranks: List[int], cars: int) -> int:
        left = 1
        right = min(ranks) * cars * cars

        while left < right:
            mid = (left + right) // 2

            repaired = 0

            for rank in ranks:
                repaired += math.isqrt(mid // rank)

                if repaired >= cars:
                    break

            if repaired >= cars:
                right = mid
            else:
                left = mid + 1

        return left
```

The implementation follows the binary-search strategy directly.

The search range is initialized using the fastest mechanic repairing all cars alone. This guarantees the answer exists inside the interval.

Inside the loop, `mid` represents a candidate time. For each mechanic, we compute how many cars can be repaired within that time using integer square root.

The use of `math.isqrt` is important because it avoids floating-point precision issues and is faster than converting from floating-point square roots.

The early break optimization improves performance slightly. Once the repaired count already reaches `cars`, we do not need further computation.

If the candidate time is feasible, we shrink the right boundary to continue searching for a smaller valid answer. Otherwise, we move the left boundary upward.

When the loop finishes, `left` holds the minimum feasible time.

## Go Solution

```go
package main

import (
	"math"
)

func repairCars(ranks []int, cars int) int64 {
	minRank := ranks[0]

	for _, rank := range ranks {
		if rank < minRank {
			minRank = rank
		}
	}

	left := int64(1)
	right := int64(minRank) * int64(cars) * int64(cars)

	for left < right {
		mid := left + (right-left)/2

		repaired := int64(0)

		for _, rank := range ranks {
			repaired += int64(math.Sqrt(float64(mid / int64(rank))))

			if repaired >= int64(cars) {
				break
			}
		}

		if repaired >= int64(cars) {
			right = mid
		} else {
			left = mid + 1
		}
	}

	return left
}
```

The Go implementation is structurally identical to the Python version, but there are several language-specific details worth noting.

Go requires explicit use of `int64` because the answer can grow up to roughly `10^14`, which may overflow 32-bit integers.

Unlike Python, Go does not provide a built-in integer square root function equivalent to `math.isqrt`, so we use `math.Sqrt` with floating-point conversion. This is safe here because the values remain within representable precision limits for double-precision floating-point numbers.

Slices in Go naturally handle dynamic-length arrays, so no special handling for empty inputs is required because the constraints guarantee at least one mechanic.

Binary search over this range requires roughly `log2(10^14) ≈ 47` iterations.

Each iteration scans all mechanics once, producing total complexity:

```
O(n log(answer))
```

This is efficient enough for `10^5` mechanics.

## Edge Cases

One important edge case occurs when there is only a single mechanic. In this situation, the mechanic must repair every car alone, so the answer becomes exactly:

$r \cdot cars^2$

A naive implementation might accidentally assume work distribution across multiple mechanics. Our implementation handles this naturally because the feasibility calculation works correctly even for arrays of length one.

Another important case is when one mechanic is dramatically faster than all others. For example, ranks like `[1, 100, 100]` mean the rank-1 mechanic performs nearly all the work. Some incorrect greedy approaches try to balance cars evenly among mechanics, which would fail badly here. The binary-search solution avoids this mistake because it only checks feasibility mathematically.

A third critical edge case involves extremely large answers. Since `cars` can reach `10^6`, the repair time may exceed `10^14`. Using 32-bit integers would overflow. The Python implementation avoids overflow automatically because Python integers are unbounded, while the Go implementation explicitly uses `int64`.

Another subtle issue is floating-point precision. Using normal square root calculations with floating-point rounding can occasionally produce off-by-one errors. The Python implementation avoids this completely with `math.isqrt`, which computes exact integer square roots safely.
