## Solution

## Problem Understanding

The problem gives us two strings, `s` and `t`. We are allowed to delete characters from `t`, and after deletions, the remaining characters of `t` must form a subsequence of `s`.

The important detail is how the score is defined. If we remove no characters, the score is `0`. Otherwise, suppose the removed characters occupy indices from `left` to `right` in `t`. The score becomes:

$$right - left + 1$$

This means the score is the length of the contiguous segment in `t` that covers all removed characters.

Even if we delete scattered characters, the score is still measured by the smallest interval covering them all. Therefore, deleting indices `1` and `5` gives score `5`, because the interval `[1,5]` has length `5`.

The key observation is that any valid deletion pattern is equivalent to deleting one contiguous substring from `t`. Characters outside that interval remain untouched and must appear as a subsequence of `s`.

So the real problem becomes:

> Find the smallest substring of `t` that can be removed so that the remaining prefix and suffix together form a subsequence of `s`.

The constraints are very important:

- `1 <= s.length, t.length <= 10^5`

A quadratic or cubic solution will not work. We need an algorithm close to linear time.

There are several edge cases worth thinking about immediately.

If `t` is already a subsequence of `s`, the answer is `0`.

If `s` and `t` share no matching characters, we may need to remove the entire string `t`, giving answer `len(t)`.

Another subtle case is when only a middle portion of `t` must be removed. The remaining prefix and suffix must still preserve order inside `s`.

For example:

```
s = "abacaba"
t = "bzaa"
```

Removing only `"z"` works because `"baa"` is still a subsequence of `s`.

The problem guarantees only lowercase English letters, which simplifies matching logic.

## Approaches

### Brute Force

A brute-force solution would try every possible substring removal from `t`.

Suppose we remove `t[l:r+1]`. The remaining string becomes:

```
t[:l] + t[r+1:]
```

We can then check whether this remaining string is a subsequence of `s` using two pointers.

There are $O(m^2)$ possible substrings in `t`, where $m = len(t)$. Each subsequence check costs $O(n)$, where $n = len(s)$.

That gives total complexity:

$$O(m^2 \cdot n)$$

With lengths up to $10^5$, this is completely infeasible.

### Optimal Insight

The key insight is that we do not need to rebuild strings or test every subsequence independently.

We can precompute:

- The earliest position in `s` where each prefix of `t` can match.
- The latest position in `s` where each suffix of `t` can match.

Then we try to connect a prefix and suffix while removing the middle segment.

Define:

- `left[i]` = earliest index in `s` where `t[i]` can be matched while building the prefix subsequence from left to right.
- `right[i]` = latest index in `s` where `t[i]` can be matched while building the suffix subsequence from right to left.

Now suppose we keep:

- prefix `t[0:i]`
- suffix `t[j:m]`

Then we remove substring:

```
t[i:j]
```

This removal is valid if the prefix match ends before the suffix match begins in `s`.

Formally:

```
left[i - 1] < right[j]
```

This allows us to use two pointers and find the minimum removable interval efficiently.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m² · n) | O(1) | Tries every removable substring |
| Optimal | O(n + m) | O(m) | Uses prefix and suffix matching arrays |

## Algorithm Walkthrough

### Step 1: Build the prefix match array

We scan through `s` from left to right while matching characters of `t`.

For every index `i` in `t`, we store the earliest index in `s` where `t[i]` can be matched.

For example:

```
s = "abacaba"
t = "bzaa"
```

We can match:

```
t[0] = 'b' at s[1]
t[2] = 'a' at s[2]
t[3] = 'a' at s[4]
```

The unmatched character gets no valid position.

This array tells us how far into `s` we must go to preserve a prefix of `t`.

### Step 2: Build the suffix match array

Now scan from right to left.

For every index `i` in `t`, store the latest index in `s` where `t[i]` can be matched while preserving suffix order.

This tells us how late we can start matching the suffix.

### Step 3: Handle full deletion answer initially

The worst possible answer is deleting all of `t`.

So initialize:

```
answer = len(t)
```

### Step 4: Try keeping only a prefix

If a prefix alone can be matched, then we can delete everything after it.

For example:

```
keep t[0:i]
remove t[i+1:]
```

This gives deletion length:

```
m - (i + 1)
```

### Step 5: Try keeping only a suffix

Similarly, if a suffix alone can be matched, we can remove everything before it.

### Step 6: Connect prefix and suffix

Now use two pointers.

Suppose:

- prefix ends at `i`
- suffix starts at `j`

We need:

```
left[i] < right[j]
```

This guarantees the subsequence ordering inside `s`.

If valid, then removed substring length is:

```
j - i - 1
```

We slide the pointers efficiently to minimize this value.

### Why it works

The algorithm works because every valid solution corresponds to deleting one contiguous substring of `t`.

The `left` array guarantees the earliest feasible placement of a prefix in `s`, while the `right` array guarantees the latest feasible placement of a suffix.

If:

```
left[i] < right[j]
```

then the prefix and suffix can coexist as a subsequence in correct order inside `s`.

Since we examine all possible prefix-suffix boundaries efficiently, the minimum removable interval is guaranteed to be found.

## Python Solution

```python
class Solution:
    def minimumScore(self, s: str, t: str) -> int:
        n = len(s)
        m = len(t)

        left = [-1] * m
        right = [-1] * m

        # Build prefix matches
        j = 0
        for i in range(n):
            if j < m and s[i] == t[j]:
                left[j] = i
                j += 1

        # Build suffix matches
        j = m - 1
        for i in range(n - 1, -1, -1):
            if j >= 0 and s[i] == t[j]:
                right[j] = i
                j -= 1

        # If already subsequence
        if left[m - 1] != -1:
            return 0

        answer = m

        # Keep only prefix
        for i in range(m):
            if left[i] != -1:
                answer = min(answer, m - i - 1)

        # Keep only suffix
        for i in range(m):
            if right[i] != -1:
                answer = min(answer, i)

        # Keep prefix + suffix
        j = 0

        for i in range(m):
            if left[i] == -1:
                break

            j = max(j, i + 1)

            while j < m and right[j] <= left[i]:
                j += 1

            if j < m:
                answer = min(answer, j - i - 1)

        return answer
```

The implementation begins by constructing the `left` array. The pointer `j` tracks how many characters from `t` have been matched so far. Whenever a matching character is found in `s`, we record its position.

The `right` array is built similarly, except we scan backward. This records the latest valid positions for suffix matching.

If the entire `t` can already be matched as a subsequence, the answer is immediately `0`.

Next, we consider cases where only a prefix or only a suffix is kept. These correspond to deleting a suffix or prefix of `t`.

Finally, the two-pointer phase attempts to combine a valid prefix and suffix. The condition:

```python
right[j] > left[i]
```

ensures that the suffix begins after the prefix ends inside `s`.

The minimum removable interval across all valid combinations is returned.

## Go Solution

```go
func minimumScore(s string, t string) int {
	n := len(s)
	m := len(t)

	left := make([]int, m)
	right := make([]int, m)

	for i := 0; i < m; i++ {
		left[i] = -1
		right[i] = -1
	}

	// Build prefix matches
	j := 0
	for i := 0; i < n; i++ {
		if j < m && s[i] == t[j] {
			left[j] = i
			j++
		}
	}

	// Build suffix matches
	j = m - 1
	for i := n - 1; i >= 0; i-- {
		if j >= 0 && s[i] == t[j] {
			right[j] = i
			j--
		}
	}

	// Already subsequence
	if left[m-1] != -1 {
		return 0
	}

	answer := m

	// Keep only prefix
	for i := 0; i < m; i++ {
		if left[i] != -1 {
			if m-i-1 < answer {
				answer = m - i - 1
			}
		}
	}

	// Keep only suffix
	for i := 0; i < m; i++ {
		if right[i] != -1 {
			if i < answer {
				answer = i
			}
		}
	}

	// Prefix + suffix
	j = 0

	for i := 0; i < m; i++ {
		if left[i] == -1 {
			break
		}

		if j < i+1 {
			j = i + 1
		}

		for j < m && right[j] <= left[i] {
			j++
		}

		if j < m {
			if j-i-1 < answer {
				answer = j - i - 1
			}
		}
	}

	return answer
}
```

The Go implementation follows the exact same logic as the Python version.

One small difference is explicit initialization of arrays with `-1`, because Go integer slices default to `0`. We must distinguish unmatched positions from valid index `0`.

Go strings are byte-indexable here because the problem guarantees lowercase English letters, so UTF-8 complications do not matter.

## Worked Examples

### Example 1

```
s = "abacaba"
t = "bzaa"
```

### Prefix Matches

| t Index | Character | Matched s Index |
| --- | --- | --- |
| 0 | b | 1 |
| 1 | z | -1 |
| 2 | a | -1 |
| 3 | a | -1 |

```
left = [1, -1, -1, -1]
```

### Suffix Matches

Scanning backward:

| t Index | Character | Matched s Index |
| --- | --- | --- |
| 3 | a | 6 |
| 2 | a | 5 |
| 1 | z | -1 |
| 0 | b | -1 |

```
right = [-1, -1, 5, 6]
```

### Try Combining

Keep prefix ending at `i = 0`:

```
left[0] = 1
```

Find smallest `j` such that:

```
right[j] > 1
```

Smallest valid `j` is `2`.

Removed substring:

```
t[1:2]
```

Length:

```
2 - 0 - 1 = 1
```

Answer becomes `1`.

### Example 2

```
s = "cde"
t = "xyz"
```

No characters match.

```
left = [-1, -1, -1]
right = [-1, -1, -1]
```

No prefix or suffix can be preserved.

We must delete the entire string:

```
answer = 3
```

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n + m) | Each pointer moves at most once |
| Space | O(m) | Two arrays of size `m` |

The prefix scan processes `s` once, and the suffix scan also processes `s` once. The final two-pointer merge advances monotonically, so every pointer moves at most `m` times total.

Therefore the overall complexity is linear.

## Edge Cases

One important edge case occurs when `t` is already a subsequence of `s`. A buggy implementation might still try removing characters and accidentally return a positive answer. This implementation explicitly checks whether the final entry in the `left` array was matched. If so, it immediately returns `0`.

Another important case is when no characters from `t` appear in `s`. In this situation, neither the prefix nor suffix arrays contain valid matches. The algorithm correctly leaves the answer initialized to `len(t)`, meaning the entire string must be removed.

Repeated characters are another common source of mistakes. For example:

```
s = "aaaaa"
t = "aaa"
```

There are many possible match positions, and greedy matching must preserve subsequence ordering correctly. The prefix array stores earliest matches, while the suffix array stores latest matches. This separation guarantees that all valid combinations are considered without conflicts.

A subtle edge case appears when the removable substring lies strictly in the middle. The implementation handles this through the condition:

```
right[j] > left[i]
```

This ensures the suffix begins after the prefix ends in `s`, preserving subsequence order correctly.
