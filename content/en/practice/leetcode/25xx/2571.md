# LeetCode 2569 - Handling Sum Queries After Update

## Problem Understanding

The problem gives us two arrays, `nums1` and `nums2`, along with a sequence of queries that modify or inspect these arrays. The challenge is to process all queries efficiently while respecting the large input constraints.

The first array, `nums1`, contains only binary values, either `0` or `1`. The second array, `nums2`, contains non-negative integers. The queries come in three different forms.

A type `1` query flips all bits in `nums1` between indices `l` and `r`, inclusive. Every `0` becomes `1`, and every `1` becomes `0`.

A type `2` query updates `nums2`. For every index `i`, we perform:

```
nums2[i] = nums2[i] + nums1[i] * p
```

Since `nums1[i]` is either `0` or `1`, this means only positions where `nums1[i] == 1` contribute to the increase.

A type `3` query asks for the total sum of all elements currently in `nums2`.

The goal is to return all answers produced by type `3` queries.

The constraints are extremely important here. Both arrays and the query list can contain up to `10^5` elements. A naive solution that updates large ranges element by element would become too slow. For example, repeatedly flipping large ranges in `nums1` using direct iteration could lead to `O(n * q)` complexity, which is too expensive for these limits.

Several observations simplify the problem significantly.

First, type `2` queries do not require us to explicitly modify every element of `nums2`. Since the query adds `p` for every position where `nums1[i] == 1`, the total increase in the sum of `nums2` is simply:

```
(number of ones in nums1) * p
```

This means we only need to maintain the current sum of `nums2`, not the entire array contents.

Second, the only difficult operation becomes efficiently tracking how many ones currently exist in `nums1` after many range flips.

This naturally leads to a segment tree with lazy propagation.

Important edge cases include flipping the same range multiple times, flipping the entire array, handling arrays of length `1`, and processing queries where `p = 0`. The problem guarantees all indices are valid and that `nums1` contains only binary digits.

## Approaches

### Brute Force Approach

The most direct solution is to simulate every operation literally.

For a type `1` query, iterate from `l` to `r` and flip each bit in `nums1`.

For a type `2` query, iterate through the entire array and update every element of `nums2` using the formula:

```
nums2[i] += nums1[i] * p
```

For a type `3` query, compute the sum of `nums2`.

This approach is correct because it exactly follows the problem statement. However, it is far too slow. Each query may require `O(n)` work, and with up to `10^5` queries, the total complexity can become `O(n * q)`, which is approximately `10^10` operations in the worst case.

### Optimal Approach

The key insight is that we never actually need the full contents of `nums2`.

A type `3` query only asks for the total sum of `nums2`, not individual elements. Therefore, instead of updating every element during a type `2` query, we can directly increase the total sum.

If `countOnes` is the number of `1`s currently present in `nums1`, then:

```
sum(nums2) += countOnes * p
```

Now the main challenge becomes maintaining `countOnes` efficiently while supporting range bit flips.

A segment tree with lazy propagation solves this perfectly.

Each node stores how many ones exist in its segment. When flipping a segment, the number of ones becomes:

```
segmentLength - currentOnes
```

Lazy propagation allows range flips in `O(log n)` time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * q) | O(1) | Directly simulates every operation |
| Optimal | O((n + q) log n) | O(n) | Uses segment tree with lazy propagation |

## Algorithm Walkthrough

1. Compute the initial sum of all elements in `nums2`.

This value will be maintained throughout the algorithm. We never need to explicitly modify the array again.
2. Build a segment tree over `nums1`.

Each node stores the number of ones in its segment. This allows efficient range queries and updates.
3. Add a lazy propagation flag for every node.

The lazy flag indicates whether a pending flip operation still needs to be pushed to child nodes.
4. For a type `1` query `[1, l, r]`, perform a range flip.

If a segment is fully covered, we flip the count directly:

```
ones = segmentLength - ones
```

Then toggle the lazy flag.
5. For partially overlapping segments, push pending lazy updates downward before continuing recursion.

This ensures all counts remain correct.
6. After processing the flip, the root node of the segment tree contains the current total number of ones in `nums1`.
7. For a type `2` query `[2, p, 0]`, update the running sum:

```
totalSum += countOnes * p
```

Since only indices with value `1` contribute, we do not need to touch `nums2`.
8. For a type `3` query `[3, 0, 0]`, append the current total sum to the answer list.
9. Return the collected answers after processing all queries.

### Why it works

The segment tree always maintains the exact number of ones in every segment of `nums1`. Lazy propagation guarantees that deferred flips are correctly applied before deeper operations occur. Since type `2` queries only depend on how many ones exist, maintaining the total count is sufficient. Therefore, every query result matches the true state of the arrays without explicitly updating every element.

## Python Solution

```python
from typing import List

class Solution:
    def handleQuery(
        self,
        nums1: List[int],
        nums2: List[int],
        queries: List[List[int]]
    ) -> List[int]:

        n = len(nums1)

        tree = [0] * (4 * n)
        lazy = [False] * (4 * n)

        def build(node: int, left: int, right: int) -> None:
            if left == right:
                tree[node] = nums1[left]
                return

            mid = (left + right) // 2

            build(node * 2, left, mid)
            build(node * 2 + 1, mid + 1, right)

            tree[node] = tree[node * 2] + tree[node * 2 + 1]

        def apply_flip(node: int, left: int, right: int) -> None:
            segment_length = right - left + 1
            tree[node] = segment_length - tree[node]
            lazy[node] = not lazy[node]

        def push(node: int, left: int, right: int) -> None:
            if not lazy[node] or left == right:
                return

            mid = (left + right) // 2

            apply_flip(node * 2, left, mid)
            apply_flip(node * 2 + 1, mid + 1, right)

            lazy[node] = False

        def update(
            node: int,
            left: int,
            right: int,
            query_left: int,
            query_right: int
        ) -> None:

            if query_left <= left and right <= query_right:
                apply_flip(node, left, right)
                return

            push(node, left, right)

            mid = (left + right) // 2

            if query_left <= mid:
                update(node * 2, left, mid, query_left, query_right)

            if query_right > mid:
                update(node * 2 + 1, mid + 1, right, query_left, query_right)

            tree[node] = tree[node * 2] + tree[node * 2 + 1]

        build(1, 0, n - 1)

        total_sum = sum(nums2)
        answers = []

        for query_type, x, y in queries:

            if query_type == 1:
                update(1, 0, n - 1, x, y)

            elif query_type == 2:
                total_sum += tree[1] * x

            else:
                answers.append(total_sum)

        return answers
```

The implementation begins by constructing a segment tree that stores the number of ones in each interval of `nums1`.

The `build` function initializes the tree recursively. Leaf nodes store the actual binary values from `nums1`, while internal nodes store the sum of their children.

The `apply_flip` helper performs the key transformation. If a segment contains `k` ones out of length `len`, then after flipping, it contains `len - k` ones.

The `push` function handles lazy propagation. Whenever a node has a pending flip operation, the flip is propagated to its children before further recursion occurs.

The `update` function performs range flips in logarithmic time. Fully covered segments are updated immediately, while partially overlapping segments recurse into their children.

The main query loop processes each operation. Type `1` performs a segment tree update. Type `2` increases the running sum using the current number of ones stored at the root of the segment tree. Type `3` records the current total sum.

## Go Solution

```go
package main

func handleQuery(nums1 []int, nums2 []int, queries [][]int) []int64 {
	n := len(nums1)

	tree := make([]int, 4*n)
	lazy := make([]bool, 4*n)

	var build func(node, left, right int)

	build = func(node, left, right int) {
		if left == right {
			tree[node] = nums1[left]
			return
		}

		mid := (left + right) / 2

		build(node*2, left, mid)
		build(node*2+1, mid+1, right)

		tree[node] = tree[node*2] + tree[node*2+1]
	}

	var applyFlip func(node, left, right int)

	applyFlip = func(node, left, right int) {
		length := right - left + 1
		tree[node] = length - tree[node]
		lazy[node] = !lazy[node]
	}

	var push func(node, left, right int)

	push = func(node, left, right int) {
		if !lazy[node] || left == right {
			return
		}

		mid := (left + right) / 2

		applyFlip(node*2, left, mid)
		applyFlip(node*2+1, mid+1, right)

		lazy[node] = false
	}

	var update func(node, left, right, ql, qr int)

	update = func(node, left, right, ql, qr int) {
		if ql <= left && right <= qr {
			applyFlip(node, left, right)
			return
		}

		push(node, left, right)

		mid := (left + right) / 2

		if ql <= mid {
			update(node*2, left, mid, ql, qr)
		}

		if qr > mid {
			update(node*2+1, mid+1, right, ql, qr)
		}

		tree[node] = tree[node*2] + tree[node*2+1]
	}

	build(1, 0, n-1)

	var totalSum int64

	for _, value := range nums2 {
		totalSum += int64(value)
	}

	result := []int64{}

	for _, query := range queries {
		queryType := query[0]

		if queryType == 1 {
			update(1, 0, n-1, query[1], query[2])

		} else if queryType == 2 {
			p := int64(query[1])
			totalSum += int64(tree[1]) * p

		} else {
			result = append(result, totalSum)
		}
	}

	return result
}
```

The Go implementation follows the same structure as the Python version. The main difference is the explicit use of `int64` for the running sum and result array. This is necessary because repeated additions can exceed the range of a standard 32-bit integer.

Slices are used for the segment tree and lazy arrays. Recursive helper functions are implemented as closures inside the main function.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O((n + q) log n) | Building takes O(n), each update takes O(log n) |
| Space | O(n) | Segment tree and lazy arrays |

The segment tree requires linear space proportional to the input size. Every type `1` query performs a logarithmic range update due to lazy propagation. Type `2` and type `3` queries run in constant time because the algorithm maintains the total sum incrementally.

## Edge Cases

One important edge case occurs when the array length is `1`. Segment trees often introduce off-by-one mistakes in tiny intervals. The implementation handles this safely because leaf nodes are treated separately, and lazy propagation stops recursion when `left == right`.

Another important case is flipping the same range multiple times. Since flipping twice restores the original values, incorrect lazy propagation logic can easily corrupt the segment counts. The implementation avoids this by toggling the lazy flag with boolean negation instead of overwriting it.

A third important edge case involves very large sums. Since `nums2[i]` can already be as large as `10^9`, repeated type `2` operations may produce values exceeding 32-bit integer limits. The Go implementation explicitly uses `int64` for all accumulated sums to prevent overflow issues.
