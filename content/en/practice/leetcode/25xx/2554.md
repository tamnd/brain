## Solution

## Problem Understanding

This problem asks us to construct the largest possible subset of integers under several constraints. We are given an array `banned`, a positive integer range `[1, n]`, and a maximum allowed sum `maxSum`. We must choose integers from `[1, n]` such that none of the chosen integers appear in `banned`, each integer can be used at most once, and the total sum of chosen integers does not exceed `maxSum`. The goal is to maximize the number of chosen integers.

In other words, we are trying to pick as many valid numbers as possible, prioritizing smaller values implicitly because smaller numbers allow us to fit more elements under the same sum constraint.

The input array `banned` identifies forbidden values. The integer `n` defines the upper bound of the selectable range. The `maxSum` constraint limits the cumulative sum of chosen numbers.

The constraints indicate that `n` and `banned.length` are up to 10^4, which suggests that an O(n log n) or O(n) solution is required. The `maxSum` can be large (up to 10^9), so we cannot rely on enumeration of subsets or DP over sums.

Key edge cases include when all numbers are banned, when `maxSum` is very small (forcing zero or one selection), when `banned` is empty, and when `maxSum` is large enough to take all valid numbers.

## Approaches

The brute-force idea is to consider all subsets of numbers from `[1, n]`, filter out subsets that contain banned numbers, and compute the sum of each subset. We would then track the subset with the maximum size whose sum does not exceed `maxSum`. This is correct because it directly explores all possibilities, but it is computationally infeasible since the number of subsets grows exponentially.

The key insight for optimization is that we should always prefer smaller numbers first. Since all numbers have equal "value" in terms of count contribution but different "cost" in terms of sum, choosing smaller numbers maximizes how many we can fit. This makes the problem a greedy selection problem: iterate numbers from 1 to n, skip banned numbers, and keep adding valid numbers until adding the next one would exceed `maxSum`.

We use a hash set for `banned` to allow O(1) membership checks, enabling efficient filtering during iteration.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n * n) | O(n) | Try all subsets and validate constraints |
| Optimal Greedy | O(n) | O(b) | Iterate 1..n, skip banned, accumulate until sum limit |

## Algorithm Walkthrough

The optimal solution proceeds as follows.

1. Convert the `banned` list into a hash set so that membership checks can be performed in constant time. This is important because we will repeatedly test whether a number is allowed.
2. Initialize a variable `current_sum` to track the sum of selected integers, and a counter `count` to track how many numbers have been selected.
3. Iterate through all integers `x` from 1 to `n` in increasing order. We process in ascending order because smaller numbers are always more efficient in terms of maximizing count under a sum constraint.
4. For each number `x`, first check whether it is in the banned set. If it is banned, we skip it immediately since it cannot be selected.
5. If `x` is not banned, check whether adding it to `current_sum` would exceed `maxSum`. If `current_sum + x` is still within the limit, we select it, increment `count`, and update `current_sum`.
6. If `current_sum + x` exceeds `maxSum`, we break the loop. This is safe because all subsequent numbers are larger than `x`, meaning they will only further increase the sum and cannot be included.

Why it works: the algorithm maintains the invariant that at every step we have selected the maximum possible number of valid integers from the prefix `[1..x]` under the sum constraint. Since we process in increasing order and always take a number if it fits, we ensure optimality through a greedy choice property where smaller values are always more beneficial than larger ones.

## Python Solution

```python
from typing import List

class Solution:
    def maxCount(self, banned: List[int], n: int, maxSum: int) -> int:
        banned_set = set(banned)
        
        current_sum = 0
        count = 0
        
        for x in range(1, n + 1):
            if x in banned_set:
                continue
            
            if current_sum + x > maxSum:
                break
            
            current_sum += x
            count += 1
        
        return count
```

The implementation first constructs a set from the banned list for efficient lookups. It then iterates through all candidates in ascending order. Each valid number is either skipped (if banned) or considered for inclusion. If adding the number would exceed `maxSum`, the loop terminates early since no further numbers can be added safely.

## Go Solution

```go
func maxCount(banned []int, n int, maxSum int) int {
    bannedSet := make(map[int]bool)
    for _, v := range banned {
        bannedSet[v] = true
    }

    currentSum := 0
    count := 0

    for x := 1; x <= n; x++ {
        if bannedSet[x] {
            continue
        }

        if currentSum + x > maxSum {
            break
        }

        currentSum += x
        count++
    }

    return count
}
```

In Go, we use a map of integers to booleans to simulate a hash set. The logic mirrors the Python version closely. One difference is explicit type handling for integers, but since Go integers are sufficient for the given constraints, no special overflow handling is required.

We build `banned_set = {1, 5, 6}`.

We iterate:

| --- | --- | --- | --- |
| 1 | yes | 0 | skip |

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | We iterate from 1 to n once, with O(1) set lookups |
| Space | O(b) | We store banned elements in a hash set |

The runtime is linear because each integer in the range is processed exactly once, and all operations inside the loop are constant time. Space usage is dominated by storing the banned set.

## Edge Cases

One important edge case occurs when all numbers in `[1, n]` are present in the banned list. In this scenario, the algorithm correctly builds a hash set containing all values and skips every candidate, resulting in a final count of zero.

Another edge case is when `maxSum` is extremely small, such as 1. Even if valid numbers exist, only the smallest possible number can be selected, and often even that may not fit depending on banning. The algorithm handles this naturally by checking `current_sum + x > maxSum` before selection, ensuring no invalid accumulation occurs.

A third edge case involves an empty `banned` list. In this case, the algorithm reduces to a simple greedy accumulation over the range `[1, n]`, stopping precisely when the sum constraint is violated. The correctness is preserved because the logic does not depend on the presence of banned elements.

A final subtle case is when banned numbers are large and outside `[1, n]`. These values are safely ignored because iteration only considers numbers within the valid range, and hash set lookups never affect correctness for out-of-range values.
