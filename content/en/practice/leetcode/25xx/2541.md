# LeetCode 2541 - Minimum Operations to Make Array Equal II

## Problem Understanding

The problem gives us two integer arrays, `nums1` and `nums2`, both of the same length, along with an integer `k`. We are allowed to repeatedly perform a very specific operation on `nums1`.

In one operation, we choose two indices `i` and `j`:

- Increase `nums1[i]` by `k`
- Decrease `nums1[j]` by `k`

This means that every operation transfers exactly `k` units from one position to another inside the same array. The total sum of the array never changes.

Our goal is to transform `nums1` so that it becomes exactly equal to `nums2`. We must return the minimum number of operations required. If the transformation is impossible, we return `-1`.

The constraints are important:

- `n` can be as large as `10^5`
- Array values can be as large as `10^9`

These limits immediately tell us that any simulation-heavy or quadratic solution will be too slow. We need a linear or near-linear algorithm.

There are several key observations hidden in the operation definition.

First, every operation changes elements only in multiples of `k`. Therefore, for every index `i`, the difference:

```
nums1[i] - nums2[i]
```

must be divisible by `k`, otherwise we can never reach the target value.

Second, because every operation adds and subtracts the same amount, the total sum of the array remains constant. Therefore:

```
sum(nums1) == sum(nums2)
```

must also hold.

Another important edge case is when `k == 0`. In that situation, no values can ever change. The only possible successful case is when the arrays are already equal.

These observations allow us to avoid any explicit simulation of operations.

## Approaches

### Brute Force Approach

A brute-force strategy would try to repeatedly find positions where `nums1[i]` is too small and positions where `nums1[j]` is too large, then perform operations until all values match `nums2`.

This approach is conceptually correct because every valid operation moves excess value from one index to another. Eventually, if the transformation is possible, the process will terminate.

However, this method becomes extremely inefficient for large arrays and large value differences. A single element may require billions of unit adjustments, and repeatedly scanning the array to find donors and receivers leads to very poor performance.

In the worst case, the complexity becomes far too large for `n = 10^5`.

### Optimal Greedy Observation

The key insight is that we do not need to simulate the operations themselves.

For each index:

```
diff = nums1[i] - nums2[i]
```

- If `diff > 0`, this index has excess value that must be transferred away.
- If `diff < 0`, this index needs additional value.
- Every transfer occurs in chunks of size `k`.

Therefore:

```
abs(diff) % k == 0
```

must hold for every index.

Suppose an index has:

```
diff = 6
k = 3
```

This means it needs exactly `2` outgoing operations.

Similarly:

```
diff = -9
k = 3
```

means it needs exactly `3` incoming operations.

The total number of outgoing operations must equal the total number of incoming operations, otherwise balancing is impossible.

The minimum number of operations is simply the total outgoing transfers.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) or worse | O(1) | Simulates transfers directly |
| Optimal | O(n) | O(1) | Uses difference balancing and divisibility checks |

## Algorithm Walkthrough

1. Handle the special case where `k == 0`.

If `k` is zero, no operation can modify the array. Therefore, the arrays must already be identical. If they are equal, return `0`. Otherwise return `-1`.
2. Initialize a variable to track required operations.

We only need to count the total number of positive transfers because every valid operation moves one chunk of size `k` from a surplus index to a deficit index.
3. Iterate through every index.

For each position, compute:

```
diff = nums1[i] - nums2[i]
```
4. Check divisibility.

If `diff % k != 0`, the transformation is impossible because operations only change values in multiples of `k`.
5. Count surplus operations.

If `diff > 0`, then this index must send out:

```
diff // k
```

operations.

Add this amount to the answer.
6. Verify total balance.

The total sum difference across all indices must equal zero. If it does not, return `-1`.
7. Return the total positive transfers.

This value is the minimum number of operations required.

### Why it works

Each operation transfers exactly one chunk of size `k` from one index to another. Therefore, every surplus chunk must eventually match with some deficit chunk. Since transfers are independent and interchangeable, the minimum number of operations equals the total number of surplus chunks.

The divisibility condition guarantees that all adjustments can be formed using multiples of `k`, while the zero-sum condition guarantees that the excess and deficits perfectly balance.

## Python Solution

```python
from typing import List

class Solution:
    def minOperations(self, nums1: List[int], nums2: List[int], k: int) -> int:
        if k == 0:
            return 0 if nums1 == nums2 else -1

        total_diff = 0
        operations = 0

        for a, b in zip(nums1, nums2):
            diff = a - b
            total_diff += diff

            if diff % k != 0:
                return -1

            if diff > 0:
                operations += diff // k

        if total_diff != 0:
            return -1

        return operations
```

The implementation begins by handling the special `k == 0` case. Since no modifications are possible, equality must already hold.

Next, the algorithm iterates through both arrays simultaneously using `zip`. For every index, it computes the difference between the current value and the target value.

The divisibility check ensures that every required adjustment can actually be achieved using operations of size `k`.

The algorithm accumulates only positive differences because these represent the number of outgoing transfers required. Negative differences correspond to incoming transfers and are automatically balanced if the total sum difference is zero.

Finally, the algorithm checks whether the total difference across the entire array equals zero. If not, the operation sequence cannot preserve the array sum.

```go
func minOperations(nums1 []int, nums2 []int, k int) int64 {

            if nums1[i] != nums2[i] {

        totalDiff += int64(diff)

        if diff%k != 0 {

        if diff > 0 {
            operations += int64(diff / k)
        }
    }

    if totalDiff != 0 {
        return -1
    }

    return operations
```

The Go implementation follows exactly the same logic as the Python version.

One important difference is integer handling. Since the number of operations may exceed the range of a 32-bit integer, the solution uses `int64` for accumulated values such as `totalDiff` and `operations`.

Go also requires explicit loops for array comparison when `k == 0`, unlike Python where list equality can be checked directly.

| 1 | 3 | 3 | 0 | 0 | 0 |
| 2 | 1 | 7 | -6 | 0 | 0 |
| 3 | 4 | 1 | 3 | 0 | 1 |

Total difference:

```
3 + 0 - 6 + 3 = 0
```

Total positive transfers:

```
1 + 1 = 2
```

## Edge Cases

### Case 1: `k == 0`

This is the most important special case. When `k` equals zero, every operation becomes meaningless because adding and subtracting zero changes nothing. A buggy implementation may accidentally divide by zero or incorrectly assume transformations are possible.

The implementation explicitly checks this case before any arithmetic. If the arrays are already equal, it returns `0`. Otherwise, it returns `-1`.

### Case 2: Differences Not Divisible by `k`

Suppose:

```
nums1 = [5]
nums2 = [2]
k = 2
```

The difference is `3`, which cannot be constructed using increments or decrements of size `2`.

Without the divisibility check, an implementation might incorrectly assume partial transfers are possible. The solution prevents this by verifying:

```
diff % k == 0
```

for every index.

### Case 3: Total Array Sums Differ

Operations only redistribute value inside the array. They never create or destroy value.

For example:

```
nums1 = [1,1]
nums2 = [2,5]
```

Even if every individual difference is divisible by `k`, the transformation is impossible because the target array requires more total value than exists.

The implementation correctly detects this using the accumulated `total_diff` check.
