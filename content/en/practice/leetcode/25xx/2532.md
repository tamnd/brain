## Solution

## Problem Understanding

The problem asks us to simulate the process of transporting `n` boxes across a bridge using `k` workers, each with different time costs for four sequential actions: crossing the bridge to the right, picking a box, crossing back to the left, and putting the box down. The input array `time[i]` specifies these four durations for the `i`th worker. The output should be the time at which the **last box reaches the left side** of the bridge, regardless of how long it takes to put the box down.

All workers start on the left side. Only one worker can be on the bridge at a time. Workers are prioritized based on efficiency: the **least efficient worker crosses first** when multiple are waiting on the same side. Efficiency is defined by the sum of `lefti + righti`, and in case of a tie, the worker with the larger index is considered less efficient. Additionally, no more workers than necessary are dispatched to retrieve boxes.

Constraints indicate that `n` and `k` can be as large as 10,000, and time values can be up to 1,000, meaning a naive simulation that checks every second would be too slow. This implies we need a **priority queue approach** that efficiently simulates events in chronological order rather than simulating each unit of time.

Important edge cases include: all workers having the same efficiency, all boxes being picked by a single worker, or `n` much larger than `k`, requiring workers to make multiple trips.

## Approaches

The brute-force approach is to simulate every single time unit, moving workers step by step across the bridge, picking, or putting boxes. This approach is correct, but with `n` and `k` up to 10,000 and times up to 1,000, it results in potentially tens of millions of iterations, which is too slow.

The key observation is that the problem can be handled using **event-driven simulation with priority queues**. We can maintain two priority queues for workers waiting on the left and right sides of the bridge. Workers currently performing tasks (crossing or picking/putting boxes) can be tracked in separate min-heaps based on when they become available. At each step, we advance time to the next available event and move workers according to the efficiency rules.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * k) | O(k) | Simulate every second, very slow for large `n` and `k` |
| Optimal | O(n log k) | O(k) | Use priority queues to track availability and worker efficiency |

## Algorithm Walkthrough

1. Compute a **worker efficiency metric** for all workers as `(lefti + righti, index)`. This will allow us to prioritize less efficient workers.
2. Initialize two priority queues: `left_queue` and `right_queue`. Push all workers into `left_queue` since all start on the left side. Sort the queues so that **less efficient workers have higher priority**.
3. Initialize two min-heaps: `working_left` and `working_right` to track when workers currently performing tasks will finish. These heaps store tuples of `(finish_time, worker_index)`.
4. Start a variable `current_time = 0`. Initialize a counter `boxes_remaining = n`.
5. While there are boxes remaining or workers on the right side:

1. Release any workers who have completed their tasks from `working_left` and `working_right` into the corresponding queue.
2. If `right_queue` is non-empty, pop the most prioritized worker and let them cross to the left. Update `current_time` to max of worker availability and current time. Track when they arrive.
3. Otherwise, if `boxes_remaining > 0` and `left_queue` is non-empty, pop the most prioritized worker and let them cross to the right, pick a box, and schedule return to the left.
4. Update `boxes_remaining` accordingly.
5. If no workers are immediately available, advance `current_time` to the next soonest finish time in `working_left` or `working_right`.
6. Continue until the last box reaches the left side. Return the arrival time of the last worker with a box.

Why it works: The algorithm maintains the invariant that at every time step, the least efficient worker available will cross the bridge, and no more workers than necessary are dispatched. Event-driven simulation ensures we never waste time simulating idle seconds and correctly tracks simultaneous events using heaps.

## Python Solution

```python
from typing import List
import heapq

class Solution:
    def findCrossingTime(self, n: int, k: int, time: List[List[int]]) -> int:
        # Compute efficiency: (total cross time, index)
        efficiency = [(time[i][0] + time[i][2], i) for i in range(k)]
        
        # Max-heap for left and right workers based on efficiency
        left_queue = [(-efficiency[i][0], -i) for i in range(k)]
        heapq.heapify(left_queue)
        right_queue = []
        
        # Min-heaps for workers currently working: (finish_time, worker_index)
        working_left = []
        working_right = []
        
        current_time = 0
        boxes_remaining = n
        last_arrival = 0
        
        while boxes_remaining > 0 or right_queue or working_right:
            # Release workers who finished tasks
            while working_left and working_left[0][0] <= current_time:
                _, idx = heapq.heappop(working_left)
                heapq.heappush(left_queue, (-efficiency[idx][0], -idx))
            while working_right and working_right[0][0] <= current_time:
                _, idx = heapq.heappop(working_right)
                heapq.heappush(right_queue, (-efficiency[idx][0], -idx))
            
            if right_queue:
                # Worker crosses from right to left
                _, neg_idx = heapq.heappop(right_queue)
                idx = -neg_idx
                current_time += time[idx][2]  # left cross
                last_arrival = current_time
                finish_put = current_time + time[idx][3]
                heapq.heappush(working_left, (finish_put, idx))
            elif left_queue and boxes_remaining > 0:
                # Worker crosses from left to right
                _, neg_idx = heapq.heappop(left_queue)
                idx = -neg_idx
                current_time += time[idx][0]  # right cross
                finish_pick = current_time + time[idx][1]
                heapq.heappush(working_right, (finish_pick, idx))
                boxes_remaining -= 1
            else:
                # Advance to the next available worker
                next_times = []
                if working_left:
                    next_times.append(working_left[0][0])
                if working_right:
                    next_times.append(working_right[0][0])
                current_time = max(current_time, min(next_times))
        
        return last_arrival
```

The Python code uses negative values in max-heaps to prioritize the least efficient worker. Workers performing tasks are tracked in min-heaps, and time advances efficiently to the next relevant event.
