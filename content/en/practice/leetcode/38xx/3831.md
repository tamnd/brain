## Solution

## Problem Understanding

The problem gives us the root of a Binary Search Tree and an integer `level`. The root is considered to be at level `0`, its children are at level `1`, their children are at level `2`, and so on.

Our task is to look only at the nodes that belong to the specified level and determine the median value among those nodes. Before taking the median, the values are considered in sorted order. If the number of values is odd, the median is the middle element. If the number of values is even, the problem asks for the upper median, which means the larger of the two middle values.

For example, if the values at a level are `[3, 8]`, then after sorting we still have `[3, 8]`, and the answer is `8`, because the upper middle element is chosen.

The number of nodes can be as large as `2 × 10^5`, so any algorithm that repeatedly sorts large collections or traverses the tree many times would be inefficient. We need a solution that processes the tree efficiently.

Several edge cases are important.

If the requested level does not exist, we must return `-1`.

If the level contains only one node, that node itself is the median.

If the number of nodes at the level is even, we must choose the larger middle element.

Although the input tree is a Binary Search Tree, the level order structure is independent of the BST ordering. However, the BST property provides an important observation that helps avoid sorting.

## Approaches

A straightforward approach is to perform a breadth first search and collect all node values that appear at the target level. After obtaining these values, we sort them and choose the median position. This works because sorting gives the required non decreasing order. However, sorting costs `O(k log k)` where `k` is the number of nodes at that level. In the worst case, `k` can be `O(n)`.

The key observation is that the tree is a Binary Search Tree. An inorder traversal of a BST produces values in sorted order. Therefore, if we perform an inorder traversal while keeping track of each node's depth, then every node encountered at the target level is automatically collected in sorted order. No additional sorting is needed.

This reduces the overall complexity to linear time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n + k log k) | O(k + w) | BFS to gather level values, then sort them |
| Optimal | O(n) | O(k + h) | Inorder traversal gives target level values already sorted |

Here, `k` denotes the number of nodes at the target level, `h` denotes tree height, and `w` denotes maximum width of the tree.

## Algorithm Walkthrough

1. Perform an inorder traversal of the BST while keeping track of the current depth.
2. Whenever a node's depth equals the target level, append its value to a list. Because inorder traversal visits BST nodes in sorted order, the values added to the list are already sorted.
3. Continue traversing the entire tree so that every node at the desired level is collected.
4. After traversal finishes, check whether the list is empty. If no nodes were found at that level, return `-1`.
5. Let `m` be the number of collected values. Compute the upper median index as `m // 2`.
6. Return the value stored at that index.

### Why it works

A Binary Search Tree has the property that an inorder traversal visits nodes in non decreasing order. Therefore, every value collected from the target level appears in sorted order without requiring an explicit sorting step. Selecting index `len(values) // 2` gives the middle element for odd lengths and the larger middle element for even lengths, exactly matching the problem definition.

## Python Solution

```python
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right
class Solution:
    def levelMedian(self, root: Optional[TreeNode], level: int) -> int:
        values = []

        def inorder(node: Optional[TreeNode], depth: int) -> None:
            if node is None:
                return

            inorder(node.left, depth + 1)

            if depth == level:
                values.append(node.val)

            inorder(node.right, depth + 1)

        inorder(root, 0)

        if not values:
            return -1

        return values[len(values) // 2]
```

The implementation creates a list named `values` to store all node values found at the requested level.

The nested `inorder` function performs a standard inorder traversal. The current depth is passed as an argument. Whenever the depth matches the target level, the node value is appended to the list.

Because inorder traversal of a BST already produces sorted values, the list remains sorted automatically.

After traversal completes, the code checks whether the list is empty. An empty list means the level does not exist, so `-1` is returned.

Otherwise, index `len(values) // 2` corresponds to the required upper median and its value is returned.

## Go Solution

```go
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func levelMedian(root *TreeNode, level int) int {
    values := []int{}

    var inorder func(*TreeNode, int)
    inorder = func(node *TreeNode, depth int) {
        if node == nil {
            return
        }

        inorder(node.Left, depth+1)

        if depth == level {
            values = append(values, node.Val)
        }

        inorder(node.Right, depth+1)
    }

    inorder(root, 0)

    if len(values) == 0 {
        return -1
    }

    return values[len(values)/2]
}
```

The Go implementation follows exactly the same logic. A slice stores values encountered at the target level. The recursive closure performs the inorder traversal.

Go uses `nil` instead of `None`, and slices automatically grow when values are appended. Integer division naturally computes the upper median index.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every node is visited exactly once |
| Space | O(k + h) | Stores target level values and recursion stack |

The traversal touches each node one time, giving linear running time. The list stores only nodes from the target level, while the recursion stack depth equals the tree height. In the worst case of a skewed tree, the height may become `O(n)`.

## Edge Cases

### Requested level does not exist

A common source of bugs is assuming that every level exists. For example, in a tree with height one, requesting level five should not cause an error. The implementation naturally handles this because no values are collected, and an empty list results in returning `-1`.

### Even number of nodes

Many median problems use the lower median or the average of the two middle values. This problem specifically requires the upper median. Using index `len(values) // 2` ensures that for an even number of elements, the larger middle element is chosen.

### Highly skewed trees

A BST may degenerate into a linked list. In such cases, each level contains at most one node. The algorithm still works correctly because the inorder traversal visits every node exactly once and collects the unique value at the requested level.

### Root level

When `level = 0`, only the root belongs to the target level. The traversal begins with depth zero, so the root value is collected immediately. The resulting list contains exactly one element, and that value is returned as the median.
