## Solution

## Problem Understanding

The problem gives us a `9 x 9` Sudoku board that may be partially filled. Each cell contains either a digit from `'1'` to `'9'` or the character `'.'`, which represents an empty cell.

We need to determine whether the current board configuration is valid according to Sudoku rules. Importantly, the problem does not ask us to solve the Sudoku. We only need to verify that the already-filled cells do not violate any constraints.

A valid Sudoku board must satisfy three conditions:

1. Every row must contain unique digits from `1` to `9`.
2. Every column must contain unique digits from `1` to `9`.
3. Every `3 x 3` sub-box must contain unique digits from `1` to `9`.

Since empty cells are represented by `'.'`, they are ignored during validation.

The input is always guaranteed to be a `9 x 9` grid, which simplifies implementation because we never need to handle irregular board sizes. The constraints are small and fixed, which means even less efficient solutions are technically acceptable from a performance perspective. However, the goal is to design a clean and optimal solution that scales conceptually well.

One important detail is that a board can be valid even if it is not solvable. The problem only checks whether the currently filled values violate Sudoku rules.

Several edge cases are important:

- Multiple duplicate values in the same row.
- Multiple duplicate values in the same column.
- Multiple duplicate values inside the same `3 x 3` sub-box.
- Boards with mostly empty cells.
- Digits repeated across different rows or columns where repetition is actually allowed because they are not in the same row, column, or box.

A naive implementation can easily make mistakes when identifying which `3 x 3` sub-box a cell belongs to, so careful indexing is important.

## Approaches

### Brute Force Approach

A straightforward approach is to validate every row, every column, and every `3 x 3` box separately.

For each row, we scan all nine cells and check whether any digit appears more than once. We repeat the same process for each column. Finally, we iterate through each `3 x 3` sub-box and again check for duplicates.

This method works because Sudoku validity depends entirely on these three independent conditions. If all rows, columns, and sub-boxes contain no duplicate digits, then the board is valid.

Although this approach is perfectly acceptable for a fixed `9 x 9` board, it involves multiple passes over the same cells. Conceptually, it is less elegant because the same information is processed repeatedly.

### Optimal Approach

The better approach is to process the board in a single traversal.

As we iterate through each cell, we immediately check whether the digit has already appeared in:

- The current row
- The current column
- The current `3 x 3` box

We use hash sets to track previously seen digits.

The key insight is that Sudoku validity is fundamentally a duplicate detection problem. Hash sets provide constant-time lookup and insertion, making them ideal for checking whether a value has already appeared.

For each non-empty cell:

- If the digit already exists in the corresponding row set, the board is invalid.
- If the digit already exists in the corresponding column set, the board is invalid.
- If the digit already exists in the corresponding box set, the board is invalid.

Otherwise, we insert the digit into all three tracking structures and continue scanning.

This allows us to validate the entire board in one pass.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(81) | O(9) | Separately validates rows, columns, and boxes |
| Optimal | O(81) | O(81) | Single traversal using hash sets for fast duplicate detection |

Even though both are effectively constant time because the board size is fixed, the optimal solution demonstrates cleaner algorithmic design and better scalability principles.

## Algorithm Walkthrough

1. Create three tracking data structures:

- One for rows
- One for columns
- One for `3 x 3` boxes

Each structure stores digits that have already appeared.
2. Traverse the board row by row and column by column.
3. For each cell:

- If the value is `'.'`, skip it because empty cells do not affect validity.
- Otherwise, continue validation.
4. Compute the box index for the current cell.

The formula is:

```
box_index = (row // 3) * 3 + (col // 3)
```

This maps every cell into one of the nine sub-boxes.
5. Check whether the current digit already exists in:

- The current row set
- The current column set
- The current box set
6. If the digit already exists in any of them:

- Return `False` immediately because the Sudoku rules are violated.
7. Otherwise:

- Add the digit to the row set
- Add the digit to the column set
- Add the digit to the box set
8. Continue until all cells are processed.
9. If no duplicates are found after scanning the entire board:

- Return `True`

The use of hash sets is important because membership checks are extremely fast, typically `O(1)`. This allows duplicate detection without nested searches.

## Python Solution

```
from typing import List

class Solution:
    def isValidSudoku(self, board: List[List[str]]) -> bool:
        rows = [set() for _ in range(9)]
        cols = [set() for _ in range(9)]
        boxes = [set() for _ in range(9)]

        for row in range(9):
            for col in range(9):
                value = board[row][col]

                if value == ".":
                    continue

                box_index = (row // 3) * 3 + (col // 3)

                if value in rows[row]:
                    return False

                if value in cols[col]:
                    return False

                if value in boxes[box_index]:
                    return False

                rows[row].add(value)
                cols[col].add(value)
                boxes[box_index].add(value)

        return True
```

The implementation begins by creating three arrays of sets. Each row set stores digits already seen in that row, each column set stores digits already seen in that column, and each box set stores digits already seen in that `3 x 3` box.

The nested loops iterate through every cell in the board.

Whenever the current cell contains `'.'`, the algorithm skips it because empty cells do not participate in Sudoku validation.

For non-empty cells, the algorithm computes the correct sub-box index using integer division. This groups the board into nine distinct regions.

Before inserting the digit, the algorithm checks whether it already exists in the corresponding row, column, or box set. If any check succeeds, the board is invalid and the function immediately returns `False`.

If no duplicates are found, the digit is inserted into all three tracking sets.

If the traversal completes without detecting violations, the board satisfies all Sudoku rules and the function returns `True`.

## Go Solution

```
func isValidSudoku(board [][]byte) bool {
	rows := make([]map[byte]bool, 9)
	cols := make([]map[byte]bool, 9)
	boxes := make([]map[byte]bool, 9)

	for i := 0; i < 9; i++ {
		rows[i] = make(map[byte]bool)
		cols[i] = make(map[byte]bool)
		boxes[i] = make(map[byte]bool)
	}

	for row := 0; row < 9; row++ {
		for col := 0; col < 9; col++ {
			value := board[row][col]

			if value == '.' {
				continue
			}

			boxIndex := (row/3)*3 + (col / 3)

			if rows[row][value] {
				return false
			}

			if cols[col][value] {
				return false
			}

			if boxes[boxIndex][value] {
				return false
			}

			rows[row][value] = true
			cols[col][value] = true
			boxes[boxIndex][value] = true
		}
	}

	return true
}
```

The Go implementation follows the same algorithmic structure as the Python version.

Instead of Python sets, Go uses `map[byte]bool` to simulate hash sets. The keys represent digits already encountered, while the boolean values are simply placeholders.

The board is represented as `[][]byte`, so each cell is accessed as a byte rather than a string.

Go does not provide built-in set types, so maps are the standard replacement for constant-time membership checking.

Since the input size is fixed and small, there are no concerns about integer overflow or memory growth.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(81) | Each cell is processed exactly once |
| Space | O(81) | Hash sets store at most 81 digits total |

Although the complexity is written as `O(81)`, the board size is fixed, so the runtime is effectively constant.

Each operation inside the nested loops performs constant-time hash set lookups and insertions. Since there are exactly 81 cells, the algorithm performs a bounded amount of work.

The space complexity comes from storing digits inside row, column, and box tracking sets. At most, each of the 81 filled cells contributes one entry to each logical tracking structure.

## Edge Cases

### Duplicate Values in the Same Row

A row may contain the same digit more than once, which immediately violates Sudoku rules.

For example:

```
["5","3","5",".",".",".",".",".","."]
```

A naive implementation might accidentally overlook this if it only validates columns or boxes. The current solution prevents this by maintaining a dedicated hash set for every row and checking membership before insertion.

### Duplicate Values in the Same Column

Two identical digits may appear in the same column but different rows.

For example:

```
[
["7",".","."],
["7",".","."],
[".",".","."]
]
```

This can be easy to miss if validation focuses only on rows. The algorithm handles this correctly using separate column tracking sets.

### Duplicate Values Inside a Sub-Box

The trickiest case is detecting duplicates inside a `3 x 3` box.

For example:

```
[
["8",".","."],
[".","8","."],
[".",".","."]
]
```

Even though the digits are in different rows and columns, they still violate Sudoku rules because they belong to the same sub-box.

The implementation correctly groups cells into boxes using:

```
(row // 3) * 3 + (col // 3)
```

This ensures every cell maps to the proper box index.

### Mostly Empty Boards

A board may contain very few filled cells.

For example:

```
[
[".",".",".",".",".",".",".",".","."],
[".",".",".",".",".",".",".",".","."],
...
]
```

The algorithm handles this naturally by skipping all `'.'` cells. Empty cells are never inserted into tracking structures.

### Valid Repetition Across Different Regions

The same digit is allowed to appear multiple times as long as the occurrences are in different rows, different columns, and different sub-boxes.

For example, two `5`s can exist on the board if they do not conflict according to Sudoku rules.

The implementation avoids false positives because it checks all three constraints independently before declaring a violation.
