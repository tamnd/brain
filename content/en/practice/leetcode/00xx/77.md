# LeetCode 77, Combinations

## Problem Understanding

The problem asks us to generate every possible combination of exactly `k` distinct numbers selected from the range `[1, n]`.

A combination is a selection where order does not matter. This is the key distinction between combinations and permutations. For example, `[1,2]` and `[2,1]` represent the same combination, so only one of them should appear in the result.

The input consists of two integers:

- `n`, which defines the range of available numbers from `1` through `n`
- `k`, which defines how many numbers each combination must contain

The output should be a list containing all valid combinations of size `k`.

For example, if `n = 4` and `k = 2`, the available numbers are `[1,2,3,4]`. We must generate every possible unordered pair:

```
[1,2]
[1,3]
[1,4]
[2,3]
[2,4]
[3,4]
```

The constraints are relatively small:

- `1 <= n <= 20`
- `1 <= k <= n`

Although `n` is only at most `20`, the number of combinations can still become large. For example:

$\binom{20}{10}$

This evaluates to `184756`, which means any valid solution must fundamentally deal with a potentially large output size.

An important observation is that the problem guarantees `k <= n`, so we never need to handle invalid requests such as choosing more numbers than exist.

Several edge cases are important:

- When `k = 1`, every number forms its own combination.
- When `k = n`, there is exactly one valid combination containing all numbers.
- When `n = 1` and `k = 1`, the result contains only `[[1]]`.
- A naive implementation could accidentally generate duplicate combinations in different orders, such as both `[1,2]` and `[2,1]`.

The challenge is to systematically generate all unique combinations without duplication.

## Approaches

### Brute Force Approach

The brute force idea is to generate every possible sequence of length `k` using the numbers `1` through `n`, then filter out invalid or duplicate results.

For example, if `n = 4` and `k = 2`, brute force could generate:

```
[1,2]
[2,1]
[1,3]
[3,1]
...
```

Then we would need to:

1. Remove sequences containing repeated elements
2. Sort or normalize each sequence
3. Eliminate duplicates

This works because every valid combination will eventually appear among the generated sequences. However, it is extremely inefficient because permutations grow much faster than combinations.

The brute force solution wastes significant work generating many orderings that represent the same combination.

### Optimal Backtracking Approach

The key insight is that combinations can be built incrementally in sorted order.

Instead of generating all possible arrangements, we only choose numbers in increasing order. Once we choose a number, future choices must come after it.

For example:

- Start with `1`

- Then choose from `[2,3,4]`
- Start with `2`

- Then choose from `[3,4]`

This naturally avoids duplicates because `[2,1]` is never considered after `[1,2]`.

Backtracking is ideal here because:

- We build combinations one decision at a time.
- We explore all valid possibilities recursively.
- After exploring one choice, we undo it and try another.
- The recursion tree directly models the structure of combinations.

The algorithm maintains:

- A current partial combination
- A starting number for the next choice

Whenever the current combination reaches size `k`, we add it to the answer.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^k × k log k) | O(n^k) | Generates many duplicate orderings and filters them later |
| Optimal Backtracking | O(C(n,k) × k) | O(k) excluding output | Generates only valid combinations directly |

## Algorithm Walkthrough

### Optimal Backtracking Algorithm

1. Initialize an empty result list that will store all valid combinations.
2. Create a recursive backtracking function that takes:

- The next number we are allowed to choose from
- The current partial combination
3. At each recursive call, check whether the current combination already contains `k` numbers.

- If it does, we have found a complete valid combination.
- Append a copy of the current combination to the result list.
- Return to explore other possibilities.
4. Otherwise, iterate through all possible next numbers starting from the current starting value.

- This guarantees numbers are chosen in increasing order.
- Increasing order prevents duplicate combinations.
5. For each candidate number:

- Add it to the current combination.
- Recursively continue building the combination using the next number as the new starting point.
- Remove the number afterward so the next iteration starts with a clean state.
6. Continue this process until every valid combination has been explored.

The recursive structure forms a decision tree where:

- Each level corresponds to selecting one more number.
- Each path from root to leaf represents one unique combination.

Because we never revisit smaller numbers, duplicate permutations are impossible.

## Python Solution

```
from typing import List

class Solution:
    def combine(self, n: int, k: int) -> List[List[int]]:
        result = []
        current_combination = []

        def backtrack(start: int) -> None:
            if len(current_combination) == k:
                result.append(current_combination[:])
                return

            for number in range(start, n + 1):
                current_combination.append(number)
                backtrack(number + 1)
                current_combination.pop()

        backtrack(1)

        return result
```

The solution begins by initializing two lists:

- `result` stores all completed combinations
- `current_combination` stores the combination currently being constructed

The nested `backtrack` function performs the recursive exploration.

The parameter `start` indicates the smallest number that can be chosen next. This ensures numbers are always added in increasing order, which prevents duplicates.

The base case occurs when the current combination length becomes exactly `k`. At this point, a copy of the combination is appended to the result list.

Copying is important because lists are mutable. Without creating a copy, future modifications would affect previously stored combinations.

The recursive loop explores every valid next number:

```
for number in range(start, n + 1):
```

For each candidate:

1. Add the number
2. Recurse using `number + 1`
3. Remove the number afterward

The removal step is the "backtracking" operation. It restores the state so other branches can be explored independently.

Finally, the recursion begins with:

```
backtrack(1)
```

This allows the first selection to be any number from `1` through `n`.

## Go Solution

```
func combine(n int, k int) [][]int {
    result := [][]int{}
    currentCombination := []int{}

    var backtrack func(start int)

    backtrack = func(start int) {
        if len(currentCombination) == k {
            combinationCopy := make([]int, k)
            copy(combinationCopy, currentCombination)
            result = append(result, combinationCopy)
            return
        }

        for number := start; number <= n; number++ {
            currentCombination = append(currentCombination, number)

            backtrack(number + 1)

            currentCombination = currentCombination[:len(currentCombination)-1]
        }
    }

    backtrack(1)

    return result
}
```

The Go implementation follows the same recursive structure as the Python version.

One important difference is that slices in Go share underlying memory. Because of this, we must explicitly create a new slice before appending a completed combination to the result:

```
combinationCopy := make([]int, k)
copy(combinationCopy, currentCombination)
```

Without this copy, later modifications during backtracking would corrupt previously stored results.

The backtracking removal step uses slice truncation:

```
currentCombination = currentCombination[:len(currentCombination)-1]
```

Go does not have automatic list copying like Python slicing, so explicit memory handling is necessary.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(C(n,k) × k) | There are C(n,k) combinations, each requiring up to k work to copy |
| Space | O(k) excluding output | Recursive depth and current combination store at most k elements |

The algorithm generates every valid combination exactly once, so the running time must be proportional to the number of combinations produced.

Each completed combination requires copying `k` elements into the result list, which explains the multiplicative `k` factor.

The recursion depth never exceeds `k`, because each recursive level adds one element to the current combination. Therefore, auxiliary space usage is linear in `k`.

The output itself may be very large, but output space is typically excluded from auxiliary space complexity analysis.

## Edge Cases

### Case 1: `k = 1`

When only one number must be selected, every number from `1` through `n` becomes its own combination.

Example:

```
n = 4, k = 1
```

Expected result:

```
[[1],[2],[3],[4]]
```

A buggy implementation might incorrectly continue recursion after selecting one number. This solution handles the case naturally because the base condition immediately triggers once the combination size reaches `1`.

### Case 2: `k = n`

When we must select all available numbers, there is exactly one valid combination.

Example:

```
n = 4, k = 4
```

Expected result:

```
[[1,2,3,4]]
```

Some implementations accidentally terminate early or fail to explore the full depth. This backtracking approach correctly continues recursion until exactly `k` numbers are chosen.

### Case 3: Smallest Possible Input

Example:

```
n = 1, k = 1
```

This is the smallest valid input allowed by the constraints.

A recursive implementation with incorrect base conditions could fail to produce any output. Here, the algorithm immediately selects `1`, reaches size `k`, and stores the combination correctly.

### Case 4: Duplicate Combination Prevention

A major source of bugs is accidentally generating both `[1,2]` and `[2,1]`.

This implementation avoids duplicates entirely by always choosing future numbers from:

```
range(start, n + 1)
```

Since recursive calls only consider larger numbers, combinations are always constructed in increasing order. Duplicate permutations can never occur.
