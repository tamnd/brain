## Solution

## Problem Understanding

The problem asks us to determine whether a given integer reads the same forward and backward. Such numbers are called palindromes. For example, `121` is a palindrome because reversing its digits still produces `121`. On the other hand, `123` is not a palindrome because reversing it gives `321`.

The input is a single integer `x`, and the output should be a boolean value. We return `true` if the integer is a palindrome and `false` otherwise.

One important detail is that negative numbers are never considered palindromes in this problem. A value like `-121` becomes `121-` when read backward, which is clearly different from the original representation. Another important detail is numbers ending in zero. For example, `10` reversed becomes `01`, which is simply `1`, not `10`. Therefore numbers ending in zero, except the number `0` itself, cannot be palindromes.

The constraints specify that the integer fits within the standard 32-bit signed integer range:

$-2^{31} \le x \le 2^{31}-1$

This tells us the input size is small enough that arithmetic operations on integers are safe and efficient. The follow-up question is especially important because it encourages solving the problem without converting the integer into a string. That means we should think carefully about how to reverse digits mathematically.

Several edge cases are important to identify early:

- Negative numbers should immediately return `false`
- Numbers ending in zero should return `false`, unless the number itself is `0`
- Single-digit numbers are always palindromes
- Very large integers should still work correctly without overflow issues

## Approaches

### Brute Force Approach

The simplest solution converts the integer into a string. Once converted, we can compare the string with its reversed version. If both are equal, the number is a palindrome.

For example:

- Convert `121` to `"121"`
- Reverse it to get `"121"`
- Compare the two strings

This works because palindrome checking naturally maps to character symmetry. Reversing the string gives us the backward representation directly.

Although this solution is easy to write and understand, it does not satisfy the follow-up requirement because it relies on string conversion. It also uses additional memory proportional to the number of digits.

### Optimal Mathematical Approach

The better solution avoids string conversion entirely. Instead, we reverse part of the integer mathematically using modulo and division operations.

The key insight is that we do not actually need to reverse the entire number. We only need to reverse half of it.

Consider the number `1221`:

- Original left half: `12`
- Reversed right half: `12`

Once the reversed half becomes greater than or equal to the remaining half, we know we have processed enough digits.

This approach is more efficient because:

- It avoids string allocation
- It uses only arithmetic operations
- It avoids reversing the full integer, reducing overflow concerns

The algorithm repeatedly:

1. Extracts the last digit using `% 10`
2. Appends it to a reversed number
3. Removes the last digit from the original number using integer division

Eventually, we compare the first half and reversed second half.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Converts integer to string and reverses it |
| Optimal | O(log n) | O(1) | Reverses only half the digits mathematically |

Here, `n` refers to the numeric value of the integer. Since the number of digits is proportional to `log n`, the optimal solution effectively processes each digit once.

## Algorithm Walkthrough

1. First, handle invalid palindrome cases immediately. If the number is negative, return `false` because negative numbers cannot be palindromes. If the number ends in zero but is not zero itself, also return `false`.
2. Create a variable called `reversed_half` and initialize it to `0`. This variable will store the reversed second half of the number.
3. Continue processing digits while the original number is greater than `reversed_half`. This condition ensures we only reverse half the digits.
4. Extract the last digit using `% 10`.
5. Append this digit to `reversed_half` by multiplying the current value by `10` and adding the extracted digit.
6. Remove the last digit from the original number using integer division by `10`.
7. Once the loop finishes, compare the two halves:

- For even digit counts, the two halves should be exactly equal.
- For odd digit counts, the middle digit does not matter, so remove it from `reversed_half` using integer division by `10`.
8. Return whether the halves match.

The reason this works is that palindromes are symmetric. By reversing only half the digits, we can directly compare corresponding positions without reconstructing the entire number.

## Python Solution

```
class Solution:
    def isPalindrome(self, x: int) -> bool:
        # Negative numbers are not palindromes
        # Numbers ending in 0 cannot be palindromes unless the number is 0
        if x < 0 or (x % 10 == 0 and x != 0):
            return False

        reversed_half = 0

        # Reverse only half of the number
        while x > reversed_half:
            digit = x % 10
            reversed_half = reversed_half * 10 + digit
            x //= 10

        # For even-length numbers:
        # x == reversed_half
        #
        # For odd-length numbers:
        # x == reversed_half // 10
        return x == reversed_half or x == reversed_half // 10
```

The implementation begins with two early rejection checks. Negative numbers are immediately rejected because their backward representation differs from their forward representation. Numbers ending in zero are also rejected unless the number itself is zero.

The variable `reversed_half` stores the reversed second half of the digits. During each iteration of the loop, we extract the last digit from `x` and append it to `reversed_half`.

The loop condition `x > reversed_half` is important because it ensures we stop once half the digits have been processed. This avoids reversing the entire number unnecessarily.

Finally, the return statement handles both even-length and odd-length palindromes. For odd-length numbers, the middle digit does not affect palindrome symmetry, so we remove it with `reversed_half // 10`.

## Go Solution

```
func isPalindrome(x int) bool {
    // Negative numbers are not palindromes
    // Numbers ending in 0 cannot be palindromes unless the number is 0
    if x < 0 || (x%10 == 0 && x != 0) {
        return false
    }

    reversedHalf := 0

    // Reverse only half of the number
    for x > reversedHalf {
        digit := x % 10
        reversedHalf = reversedHalf*10 + digit
        x /= 10
    }

    // For even-length numbers:
    // x == reversedHalf
    //
    // For odd-length numbers:
    // x == reversedHalf / 10
    return x == reversedHalf || x == reversedHalf/10
}
```

The Go implementation follows the same logic as the Python version, but there are a few language-specific differences.

Go uses explicit variable declarations with `:=` for initialization. Integer division is naturally supported with `/` on integer types, so `x /= 10` behaves similarly to Python's `x //= 10`.

Because we reverse only half the digits, overflow is not a concern even with 32-bit integers. The reversed value never grows beyond half the length of the original number.

The runtime depends on the number of digits in the integer. Since a number with `d` digits satisfies:

$d = \lfloor \log_{10}(n) \rfloor + 1$

the algorithm runs in logarithmic time relative to the numeric value of the input. The memory usage remains constant because no additional data structures are allocated.

## Edge Cases

### Negative Numbers

Negative integers are a common source of mistakes. A naive implementation might reverse the digits of `-121` into `-121` and incorrectly conclude it is a palindrome. However, the actual backward representation is `121-`, which is not equal to the original form. The implementation correctly handles this by immediately returning `false` when `x < 0`.

### Numbers Ending in Zero

Numbers like `10`, `100`, or `120` cannot be palindromes because reversing them would introduce leading zeros. For example, `10` reversed becomes `01`, which is simply `1`. The implementation detects this situation with the condition:

```
x % 10 == 0 and x != 0
```

The special case for `0` is necessary because `0` itself is a valid palindrome.

### Odd-Length Numbers

Odd-length palindromes contain a middle digit that does not affect symmetry. For example, in `121`, the center digit `2` does not need a matching counterpart. A common bug is comparing both halves directly without removing the middle digit. The implementation avoids this by comparing:

```
x == reversed_half // 10
```

which discards the center digit from the reversed half.

### Single-Digit Numbers

Any single-digit number is automatically a palindrome because it reads the same forward and backward. The algorithm handles this naturally. For example, if `x = 7`, the loop runs once and the comparison succeeds correctly.
