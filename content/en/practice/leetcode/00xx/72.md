## Solution

## Problem Understanding

This problem asks us to determine the **minimum number of edit operations** required to transform one string into another. The two input strings, `word1` and `word2`, may have different lengths and different characters. We are allowed to perform exactly three types of operations:

1. Insert a character
2. Delete a character
3. Replace one character with another

The goal is to compute the smallest number of operations necessary to convert `word1` into `word2`.

In other words, we are trying to measure how different the two strings are, where the cost of fixing differences is based on the number of insertions, deletions, and replacements needed. This value is commonly called the **edit distance** or **Levenshtein distance**.

For example, if `word1 = "horse"` and `word2 = "ros"`, one valid transformation sequence is:

- `"horse"` → `"rorse"` (replace `'h'` with `'r'`)
- `"rorse"` → `"rose"` (delete `'r'`)
- `"rose"` → `"ros"` (delete `'e'`)

This requires `3` operations, and no sequence can do it in fewer operations.

The input consists of two strings whose lengths range from `0` to `500`. Since the maximum length is relatively large, we need an algorithm that is significantly better than exponential time. A brute-force recursive solution would quickly become infeasible because the number of possible operation sequences grows explosively.

The constraints also tell us something important about the nature of the problem. Since strings can be empty, our solution must correctly handle cases where one string has length `0`. In those situations, the answer is simply the length of the other string, because we must insert or delete every character.

Several edge cases are important to think about upfront:

- If both strings are identical, the answer should be `0`.
- If one string is empty, the answer equals the length of the other string.
- If all characters differ, many replacements may be needed.
- If one string is significantly longer than the other, insertions or deletions become important.
- Repeated characters can create multiple possible transformation paths, so greedy strategies are unreliable.

Because every choice can affect future operations, we need a systematic way to compare all possibilities efficiently.

## Approaches

### Brute Force Approach

A brute-force solution tries every possible sequence of edit operations recursively.

At each position, if the current characters match, we simply move to the next characters in both strings. However, if the characters differ, we have three choices:

1. Insert a character
2. Delete a character
3. Replace a character

For each mismatch, we recursively compute the result for all three possibilities and take the minimum.

For example, if we compare `"horse"` and `"ros"` and encounter a mismatch, we might recursively explore:

- Insert into `word1`
- Delete from `word1`
- Replace a character

Each of these branches creates more recursive calls, and many overlapping subproblems appear repeatedly. The same substring combinations are recalculated over and over.

This approach is correct because it explores every possible edit sequence and chooses the minimum. However, it is far too slow for strings of length up to `500`. The time complexity becomes exponential because each mismatch can branch into three recursive paths.

### Optimal Approach, Dynamic Programming

The key observation is that this problem contains **overlapping subproblems**.

Suppose we already know the minimum edit distance between suffixes of the strings. Then we can reuse those answers instead of recomputing them.

Let `dp[i][j]` represent:

> The minimum number of operations needed to convert `word1[0:i]` into `word2[0:j]`.

This definition allows us to build the solution incrementally.

If the current characters match:

```
word1[i - 1] == word2[j - 1]
```

then no operation is required, and we simply reuse the previous result:

```
dp[i][j] = dp[i - 1][j - 1]
```

If the characters differ, we consider all three operations:

- **Insert**: match `word2[j - 1]` by inserting into `word1`
- **Delete**: remove a character from `word1`
- **Replace**: change one character into another

We take the minimum cost among these three possibilities:

```
dp[i][j] =
1 + min(
    dp[i][j - 1],     // insert
    dp[i - 1][j],     // delete
    dp[i - 1][j - 1]  // replace
)
```

This works efficiently because every subproblem is computed exactly once.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(3^(m+n)) | O(m+n) | Explores every possible edit sequence recursively |
| Optimal | O(m × n) | O(m × n) | Dynamic programming stores overlapping subproblems |

Where `m = len(word1)` and `n = len(word2)`.

## Algorithm Walkthrough

1. **Create a DP table**

We create a 2D table `dp` of size `(m + 1) × (n + 1)` where `m` is the length of `word1` and `n` is the length of `word2`.

The extra row and column handle empty string cases.
2. **Initialize base cases**

If `word1` is empty, we must insert every character of `word2`.

Therefore:

```
dp[0][j] = j
```

Similarly, if `word2` is empty, we must delete every character from `word1`.

```
dp[i][0] = i
```
3. **Iterate through both strings**

We loop through every pair of positions `(i, j)`.

Each cell represents the edit distance between prefixes of the two strings.
4. **Handle matching characters**

If:

```
word1[i - 1] == word2[j - 1]
```

then no edit operation is required.

We inherit the answer from:

```
dp[i - 1][j - 1]
```
5. **Handle mismatched characters**

If the characters differ, we compute the minimum among:

- Insert: `dp[i][j - 1]`
- Delete: `dp[i - 1][j]`
- Replace: `dp[i - 1][j - 1]`

Then add `1` for the current operation.
6. **Return the final answer**

The bottom-right cell:

```
dp[m][n]
```

contains the minimum edit distance for the complete strings.

## Python Solution

```
class Solution:
    def minDistance(self, word1: str, word2: str) -> int:
        length1 = len(word1)
        length2 = len(word2)

        dp = [[0] * (length2 + 1) for _ in range(length1 + 1)]

        for i in range(length1 + 1):
            dp[i][0] = i

        for j in range(length2 + 1):
            dp[0][j] = j

        for i in range(1, length1 + 1):
            for j in range(1, length2 + 1):
                if word1[i - 1] == word2[j - 1]:
                    dp[i][j] = dp[i - 1][j - 1]
                else:
                    insert_cost = dp[i][j - 1]
                    delete_cost = dp[i - 1][j]
                    replace_cost = dp[i - 1][j - 1]

                    dp[i][j] = 1 + min(
                        insert_cost,
                        delete_cost,
                        replace_cost
                    )

        return dp[length1][length2]
```

The implementation starts by computing the lengths of both strings and creating a 2D dynamic programming table. The table dimensions are increased by one because we also need to represent empty prefixes.

Next, the first row and first column are initialized. These represent transformations involving an empty string. Converting an empty string to a string of length `j` requires `j` insertions, while converting a string of length `i` to an empty string requires `i` deletions.

The nested loops then process each pair of characters. If the characters match, the value is copied diagonally from `dp[i - 1][j - 1]`, since no operation is necessary.

If the characters differ, the algorithm evaluates the cost of insertion, deletion, and replacement. Since each operation counts as one edit, we take the minimum previous cost and add `1`.

Finally, the answer is returned from the bottom-right corner of the table, which represents transforming the entire `word1` into the entire `word2`.

## Go Solution

```
func minDistance(word1 string, word2 string) int {
	length1 := len(word1)
	length2 := len(word2)

	dp := make([][]int, length1+1)

	for i := range dp {
		dp[i] = make([]int, length2+1)
	}

	for i := 0; i <= length1; i++ {
		dp[i][0] = i
	}

	for j := 0; j <= length2; j++ {
		dp[0][j] = j
	}

	for i := 1; i <= length1; i++ {
		for j := 1; j <= length2; j++ {
			if word1[i-1] == word2[j-1] {
				dp[i][j] = dp[i-1][j-1]
			} else {
				insertCost := dp[i][j-1]
				deleteCost := dp[i-1][j]
				replaceCost := dp[i-1][j-1]

				dp[i][j] = 1 + min(
					insertCost,
					deleteCost,
					replaceCost,
				)
			}
		}
	}

	return dp[length1][length2]
}

func min(a, b, c int) int {
	result := a

	if b < result {
		result = b
	}

	if c < result {
		result = c
	}

	return result
}
```

The Go implementation follows the same dynamic programming strategy as Python. Since Go does not provide a built-in `min` function for multiple integers, we define a helper function to compute the minimum among three values.

Strings in Go can be indexed directly using byte positions because the problem guarantees lowercase English letters, meaning ASCII characters only. This avoids complications involving Unicode rune handling.

There is no need to worry about integer overflow because the maximum edit distance cannot exceed `500`, which is far below Go's integer limits.

## Worked Examples

### Example 1

Input:

```
word1 = "horse"
word2 = "ros"
```

We create a DP table of size `6 × 4`.

|  | "" | r | o | s |
| --- | --- | --- | --- | --- |
| "" | 0 | 1 | 2 | 3 |
| h | 1 |  |  |  |
| o | 2 |  |  |  |
| r | 3 |  |  |  |
| s | 4 |  |  |  |
| e | 5 |  |  |  |

Now we fill the table.

|  | "" | r | o | s |
| --- | --- | --- | --- | --- |
| "" | 0 | 1 | 2 | 3 |
| h | 1 | 1 | 2 | 3 |
| o | 2 | 2 | 1 | 2 |
| r | 3 | 2 | 2 | 2 |
| s | 4 | 3 | 3 | 2 |
| e | 5 | 4 | 4 | 3 |

The final answer is:

```
dp[5][3] = 3
```

So the minimum edit distance is `3`.

### Example 2

Input:

```
word1 = "intention"
word2 = "execution"
```

We progressively compute the DP table while comparing prefixes of the strings.

At several positions:

- Matching characters inherit the diagonal value.
- Mismatched characters compute the minimum of insert, delete, and replace.

For example:

| Comparison | Action | Cost |
| --- | --- | --- |
| `i` vs `e` | Replace | 1 |
| `n` vs `x` | Replace | 1 |
| `t` vs `e` | Delete | 1 |
| `n` vs `c` | Replace | 1 |
| Insert `u` | Insert | 1 |

The final computed DP value becomes:

```
dp[9][9] = 5
```

Thus, the minimum edit distance is `5`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m × n) | Every DP cell is computed exactly once |
| Space | O(m × n) | The full DP table stores results for all prefix pairs |

The algorithm uses two nested loops over the lengths of the input strings. Since each cell in the dynamic programming table is processed exactly once and each computation takes constant time, the total runtime is `O(m × n)`.

The space complexity is also `O(m × n)` because we explicitly store the entire DP table. This storage allows constant-time access to previously computed subproblems, which is essential for efficient dynamic programming.

## Edge Cases

### One or Both Strings Are Empty

A common source of bugs occurs when one string is empty. For example:

```
word1 = ""
word2 = "abc"
```

The correct answer is `3`, because we must insert all three characters. Similarly:

```
word1 = "abc"
word2 = ""
```

requires `3` deletions.

The implementation handles this naturally through DP initialization, where the first row and first column are prefilled.

### Identical Strings

If both strings are already equal:

```
word1 = "abc"
word2 = "abc"
```

the answer should be `0`.

The implementation correctly handles this because matching characters always inherit the diagonal DP value without adding any operation cost.

### Completely Different Strings

Consider:

```
word1 = "abc"
word2 = "xyz"
```

Every character differs, so replacements become necessary.

The algorithm compares insertion, deletion, and replacement at every step and determines that replacing each character is optimal, producing a result of `3`.

### Different Length Strings

When one string is much longer:

```
word1 = "cat"
word2 = "catalog"
```

the best strategy often involves insertions.

A naive greedy solution might make poor local choices, but the DP approach evaluates all valid possibilities and guarantees the global minimum edit count.
