## Solution

## Problem Understanding

The problem asks us to find three numbers in the array whose sum is as close as possible to a given target value. Unlike the classic 3Sum problem, we are not looking for triplets that equal a target exactly. Instead, we want the triplet whose sum minimizes the absolute difference from the target.

The input consists of:

- An integer array `nums`
- An integer `target`

We must select three elements from the array using distinct indices, meaning the same array position cannot be reused. The output is not the triplet itself, but the sum of those three numbers.

For example, if:

```
nums = [-1,2,1,-4]
target = 1
```

The possible triplet sums include:

- `-1 + 2 + 1 = 2`
- `-1 + 2 + (-4) = -3`
- `-1 + 1 + (-4) = -4`
- `2 + 1 + (-4) = -1`

Among these, `2` is closest to `1`, so the answer is `2`.

The constraints are important:

- `nums.length` can be up to `500`
- Values range between `-1000` and `1000`

A brute-force solution that checks every possible triplet would require examining all combinations of three elements, which leads to `O(n^3)` time complexity. With `n = 500`, that becomes approximately 20 million triplets, which is inefficient for repeated computations.

The problem guarantees exactly one solution, which simplifies implementation because we do not need to handle ambiguity between equally close sums.

Several edge cases are important:

- Arrays containing all zeros
- Arrays containing negative numbers only
- Arrays where the exact target sum exists
- Arrays with duplicate values
- Very small arrays of length exactly `3`

A naive implementation can easily make mistakes with duplicate handling, pointer movement, or updating the closest sum incorrectly.

## Approaches

### Brute Force Approach

The most direct solution is to generate every possible combination of three distinct elements and compute their sum.

For every triplet:

1. Compute the sum
2. Measure how far it is from the target using absolute difference
3. Track the sum with the smallest difference seen so far

This approach is guaranteed to work because it exhaustively checks every possible triplet.

However, the number of triplets grows cubically with the array size. Since there are roughly:

```
n * (n - 1) * (n - 2) / 6
```

possible triplets, the time complexity becomes `O(n^3)`.

With `n = 500`, this is unnecessarily expensive.

### Optimal Approach, Sorting + Two Pointers

The key observation is that after sorting the array, we can use the structure of ordered numbers to intelligently move toward the target instead of checking every triplet.

Once the array is sorted:

- If the current sum is too small, we need a larger value
- If the current sum is too large, we need a smaller value

This naturally leads to the two-pointer technique.

We fix one number at index `i`, then use:

- A left pointer starting at `i + 1`
- A right pointer starting at the end of the array

For each fixed element:

- Compute the current triplet sum
- Update the closest answer if needed
- Move pointers strategically based on whether the sum is less than or greater than the target

Because each pointer only moves linearly for each fixed index, the total complexity becomes `O(n^2)`.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n³) | O(1) | Checks every possible triplet |
| Optimal | O(n²) | O(1) | Uses sorting and two pointers to reduce search space |

## Algorithm Walkthrough

1. Sort the array so that pointer movement becomes meaningful.

Sorting allows us to determine how changing a pointer affects the sum. Moving the left pointer right increases the sum, while moving the right pointer left decreases it.
2. Initialize the closest sum.

A common strategy is to use the sum of the first three elements after sorting. This guarantees we start with a valid triplet.
3. Iterate through the array using index `i`.

At each step, `nums[i]` becomes the first element of the triplet.
4. Initialize two pointers.

Set:

- `left = i + 1`
- `right = len(nums) - 1`

These pointers search the remaining portion of the array.
5. Compute the current sum.

For each pointer configuration:

```
current_sum = nums[i] + nums[left] + nums[right]
```
6. Update the closest sum if necessary.

Compare:

```
abs(current_sum - target)
```

against the best difference seen so far.
7. Decide how to move pointers.

- If `current_sum < target`, move `left` rightward to increase the sum.
- If `current_sum > target`, move `right` leftward to decrease the sum.
- If `current_sum == target`, return immediately because we cannot get closer than an exact match.
8. Continue until pointers cross.

Repeat the process while `left < right`.
9. Return the closest sum found.

## Python Solution

```
from typing import List

class Solution:
    def threeSumClosest(self, nums: List[int], target: int) -> int:
        nums.sort()

        closest_sum = nums[0] + nums[1] + nums[2]

        for i in range(len(nums) - 2):
            left = i + 1
            right = len(nums) - 1

            while left < right:
                current_sum = nums[i] + nums[left] + nums[right]

                if abs(current_sum - target) < abs(closest_sum - target):
                    closest_sum = current_sum

                if current_sum < target:
                    left += 1
                elif current_sum > target:
                    right -= 1
                else:
                    return current_sum

        return closest_sum
```

The implementation begins by sorting the array. This is essential because the two-pointer strategy depends on the numbers being ordered.

The variable `closest_sum` stores the best answer found so far. Initializing it with the first three elements ensures we always have a valid comparison baseline.

The outer loop fixes one element of the triplet at index `i`. For each fixed element, the algorithm searches the remaining subarray using two pointers.

Inside the `while left < right` loop, the current triplet sum is computed. If this sum is closer to the target than the previously recorded best sum, `closest_sum` is updated.

The pointer movement logic is the core optimization:

- If the sum is too small, moving `left` increases the sum because the array is sorted.
- If the sum is too large, moving `right` decreases the sum.
- If an exact match is found, the algorithm immediately returns because no better answer exists.

Finally, after all possibilities are explored, the closest sum is returned.

## Go Solution

```
package main

import (
	"math"
	"sort"
)

func threeSumClosest(nums []int, target int) int {
	sort.Ints(nums)

	closestSum := nums[0] + nums[1] + nums[2]

	for i := 0; i < len(nums)-2; i++ {
		left := i + 1
		right := len(nums) - 1

		for left < right {
			currentSum := nums[i] + nums[left] + nums[right]

			if math.Abs(float64(currentSum-target)) <
				math.Abs(float64(closestSum-target)) {
				closestSum = currentSum
			}

			if currentSum < target {
				left++
			} else if currentSum > target {
				right--
			} else {
				return currentSum
			}
		}
	}

	return closestSum
}
```

The Go implementation closely mirrors the Python solution. The main differences come from language syntax and type handling.

Go requires explicit sorting through `sort.Ints(nums)`. Absolute value comparisons use `math.Abs`, which operates on `float64`, so integer values must be converted temporarily.

Slices in Go behave similarly to dynamic arrays, so no special handling is required for indexing. Integer overflow is not a concern here because the maximum possible triplet sum is well within Go's integer range.

## Complexity Analysis

The sorting step costs `O(n log n)`, but the nested two-pointer traversal dominates overall complexity with `O(n²)` time.

The algorithm uses only a few extra variables regardless of input size, so auxiliary space complexity remains constant. The sorting operation itself may use implementation-dependent internal space, but the algorithmic extra space is considered `O(1)`.

## Edge Cases

### Array Length Exactly Three

When the array contains exactly three elements, there is only one possible triplet. A buggy implementation might still attempt unnecessary pointer movement or invalid indexing.

This solution handles the case naturally because:

- The initial `closest_sum` uses the only triplet available
- The loop executes correctly once
- The algorithm returns the proper result immediately

### Exact Match Exists

If a triplet sum equals the target exactly, that is mathematically the best possible answer because the difference becomes zero.

Some implementations continue searching unnecessarily after finding an exact match, wasting time.

This implementation immediately returns when:

```
current_sum == target
```

which is both correct and optimal.

### Duplicate Values

Arrays may contain repeated numbers such as:

```
[1,1,1,1]
```

A naive implementation might incorrectly skip duplicates or accidentally reuse indices.

This algorithm does not need duplicate elimination because the problem asks only for the closest sum, not unique triplets. Distinct indices are automatically guaranteed by the pointer setup:

```
i < left < right
```

### All Negative or All Positive Numbers

Arrays containing only negative numbers or only positive numbers can expose flaws in pointer logic.

For example:

```
nums = [-8,-6,-5,-3]
target = 2
```

Even though all sums are negative, the algorithm still works because pointer movement is based on comparison against the target, not assumptions about sign.

Sorting preserves the monotonic behavior needed for the two-pointer technique to remain valid.
