## Solution

## Problem Understanding

The problem gives us the root of a Binary Search Tree (BST) in which exactly two nodes have had their values swapped by mistake. Our goal is to restore the BST so that it once again satisfies BST ordering rules, without changing the structure of the tree itself.

In a valid BST, every node follows a strict ordering property. All values in the left subtree must be smaller than the current node, and all values in the right subtree must be larger. Because of this property, an inorder traversal of a BST always produces values in sorted ascending order.

The issue in this problem is that two node values were accidentally exchanged. This breaks the BST ordering. We must identify the two misplaced nodes and swap their values back so that the tree becomes valid again.

The input is the `root` node of the binary tree. Since the tree structure itself is still correct, we are not allowed to rearrange pointers or reconnect nodes. We only need to modify the values stored in the incorrect nodes.

The expected output is not a returned value. Instead, the tree must be modified in place. LeetCode will verify whether the tree structure now represents a valid BST.

The constraints tell us that the tree contains between `2` and `1000` nodes. This is small enough that even an `O(n)` space solution is acceptable, but the follow up explicitly asks whether we can solve it in constant `O(1)` extra space. Since `n ≤ 1000`, recursion depth is manageable, though recursive DFS still uses implicit stack space.

An important observation is that exactly two nodes are swapped, no more and no less. This guarantee simplifies the problem significantly because we know the tree is almost valid.

Several edge cases are worth considering upfront. The swapped nodes may be adjacent in inorder traversal, such as `[1,3,2,4]`, where only one ordering violation appears. They may also be far apart, producing two violations, such as `[1,5,3,4,2,6]`. The swapped nodes could include the root, a leaf, or nodes located in entirely different subtrees. A naive implementation that only fixes the first violation would fail for non-adjacent swaps.

## Approaches

### Brute Force Approach

A straightforward approach is to exploit the inorder property of BSTs.

We first perform an inorder traversal and store all visited nodes in an array. Since a valid BST produces a sorted sequence, we can compare the collected values against a sorted version of the array. The values that differ reveal the swapped nodes. After identifying them, we swap their values back.

This works because only two values are misplaced. By comparing the inorder traversal against its sorted version, we can precisely determine which values were swapped.

However, this approach requires additional memory to store the traversal and sorting incurs extra overhead. While acceptable for the given constraints, it does not satisfy the follow up asking for constant extra space.

### Key Insight for an Optimal Solution

The key insight comes from the inorder traversal property of BSTs.

In a valid BST, inorder traversal must always produce ascending values. If two nodes are swapped, this ordering breaks.

Suppose we keep track of the previously visited node during inorder traversal. Whenever we encounter a node whose value is smaller than the previous node, we have found an inversion.

There are two possible scenarios:

1. **Adjacent swapped nodes**, where only one inversion appears.
2. **Non-adjacent swapped nodes**, where two inversions appear.

For example:

Valid inorder:

`1, 2, 3, 4`

Adjacent swap:

`1, 3, 2, 4`

There is one inversion:

`3 > 2`

Non-adjacent swap:

`1, 4, 3, 2, 5`

There are two inversions:

`4 > 3` and `3 > 2`

The strategy is:

- The first node involved in the first inversion is the first misplaced node.
- The second node involved in the last inversion is the second misplaced node.

After traversal, swapping these two node values restores the BST.

We can implement this using DFS inorder traversal in `O(n)` time and `O(h)` recursive stack space, where `h` is the tree height.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) | O(n) | Store inorder traversal and compare with sorted version |
| Optimal | O(n) | O(h) | Detect misplaced nodes during inorder traversal |

## Algorithm Walkthrough

1. Initialize three variables: `first`, `second`, and `previous`.

`previous` keeps track of the previously visited node during inorder traversal. `first` and `second` will eventually store the two swapped nodes.
2. Perform an inorder traversal of the tree.

Since inorder traversal of a BST should be sorted, this traversal allows us to detect ordering violations naturally.
3. Visit the left subtree first.

This ensures values are processed in ascending order for a valid BST.
4. Compare the current node with `previous`.

If `previous.val > current.val`, the BST ordering has been violated.

- If this is the first violation, assign:

- `first = previous`
- `second = current`
- If another violation occurs later:

- Update `second = current`

We do this because non-adjacent swaps create two inversions, and the correct second misplaced node appears in the last inversion.
5. Update `previous`.

After processing the current node, store it as the new previous node before moving to the right subtree.
6. Visit the right subtree.

Continue inorder traversal to inspect the remaining nodes.
7. Swap the values of `first` and `second`.

After traversal finishes, the misplaced nodes have been identified. Swapping their values restores BST validity.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import Optional

class Solution:
    def recoverTree(self, root: Optional[TreeNode]) -> None:
        """
        Do not return anything, modify root in-place instead.
        """
        first = None
        second = None
        previous = None

        def inorder(node: Optional[TreeNode]) -> None:
            nonlocal first, second, previous

            if not node:
                return

            inorder(node.left)

            if previous and previous.val > node.val:
                if first is None:
                    first = previous
                second = node

            previous = node

            inorder(node.right)

        inorder(root)

        if first and second:
            first.val, second.val = second.val, first.val
```

The implementation closely follows the algorithm described earlier.

We begin by initializing `first`, `second`, and `previous` as `None`. These variables persist across recursive calls using `nonlocal`.

The nested `inorder()` function performs a standard inorder traversal. For each node, we recursively process the left subtree, examine the current node, and then process the right subtree.

The important logic occurs during the comparison between `previous` and the current node. If `previous.val > node.val`, we know BST ordering has been violated.

For the first inversion, we store `previous` as `first` because it is too large for its position. We always update `second` to the current node because it represents the smaller misplaced value. If another inversion appears later, only `second` changes.

After traversal, swapping the values of `first` and `second` repairs the tree.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func recoverTree(root *TreeNode) {
	var first *TreeNode
	var second *TreeNode
	var previous *TreeNode

	var inorder func(node *TreeNode)

	inorder = func(node *TreeNode) {
		if node == nil {
			return
		}

		inorder(node.Left)

		if previous != nil && previous.Val > node.Val {
			if first == nil {
				first = previous
			}
			second = node
		}

		previous = node

		inorder(node.Right)
	}

	inorder(root)

	if first != nil && second != nil {
		first.Val, second.Val = second.Val, first.Val
	}
}
```

The Go implementation mirrors the Python solution almost exactly. Since Go does not support nested function closures in the same way as Python's `nonlocal`, we declare variables in the outer function scope so the recursive closure can access and modify them.

Instead of `None`, Go uses `nil` to represent missing pointers. Pointer comparisons are used to check whether nodes exist before dereferencing them.

No integer overflow concerns exist here because we only compare values and swap them, never perform arithmetic.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every node is visited exactly once during inorder traversal |
| Space | O(h) | Recursive call stack stores at most tree height `h` |

The algorithm performs a single traversal through the tree, so the runtime is linear in the number of nodes. The extra space comes only from recursion depth. In a balanced tree this becomes `O(log n)`, while in the worst case of a skewed tree it becomes `O(n)`.

Although the follow up asks for constant space, this recursive DFS solution is typically accepted. A true `O(1)` extra space solution can be achieved using Morris Traversal, which temporarily modifies tree links during traversal and restores them afterward.
