## Solution

## Problem Understanding

The problem gives us a string of digits where each digit is between `2` and `9`. These digits correspond to letters on a traditional phone keypad. For example, digit `2` maps to `"abc"`, digit `3` maps to `"def"`, and so on.

We must generate every possible letter combination that can be formed by choosing one letter for each digit in the input string. The result can be returned in any order.

For example, if the input is `"23"`, then:

- `2` can produce `a`, `b`, or `c`
- `3` can produce `d`, `e`, or `f`

We combine every choice from the first digit with every choice from the second digit:

- `a` with `d`, `e`, `f`
- `b` with `d`, `e`, `f`
- `c` with `d`, `e`, `f`

This produces:

```
["ad","ae","af","bd","be","bf","cd","ce","cf"]
```

The input length is at most `4`, which means the total number of combinations is relatively small. Since each digit maps to at most `4` letters, the largest possible output size is:

```
4^4 = 256
```

This constraint tells us that generating all combinations directly is completely feasible.

One important edge case is an empty input string. Even though the constraints say the length is at least `1`, LeetCode often includes hidden tests for empty input. In that case, the correct output should be an empty list.

Another important detail is that digits `0` and `1` never appear, so we do not need to handle invalid mappings.

## Approaches

### Brute Force Approach

A brute force solution would try to manually generate every possible combination using nested loops.

For example:

- One loop for the first digit
- Another loop for the second digit
- Another loop for the third digit
- Another loop for the fourth digit

This works because the maximum length is only `4`. However, the implementation becomes awkward and inflexible because the number of loops depends on the input length.

For example:

- Length `1` needs one loop
- Length `2` needs two loops
- Length `3` needs three loops
- Length `4` needs four loops

This makes the solution difficult to generalize and maintain. The algorithm is also conceptually inefficient because it hardcodes the recursion depth instead of using a scalable traversal strategy.

### Optimal Approach

The key observation is that this problem naturally forms a decision tree.

For each digit, we have several possible letters to choose from. Every time we choose one letter, we move to the next digit and repeat the process.

This is exactly the kind of structure that backtracking handles well.

At each step:

- Choose one letter from the current digit
- Add it to the current partial combination
- Recurse to the next digit
- Remove the letter when returning so the next choice can be explored

This systematically explores every valid combination without hardcoding any number of loops.

Because the output itself contains all combinations, any correct solution must spend time proportional to the number of generated strings. Backtracking efficiently produces exactly those combinations.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(4^n) | O(4^n) | Requires manually handling different input lengths with nested loops |
| Optimal | O(4^n * n) | O(n) auxiliary, excluding output | Uses recursive backtracking to build combinations cleanly and systematically |

## Algorithm Walkthrough

1. Create a mapping from digits to letters.

We store the standard phone keypad mapping in a hash map or dictionary. This allows constant time lookup for the letters corresponding to each digit.

Example:

```
"2" -> "abc"
"3" -> "def"
```
2. Handle the empty input case.

If the input string is empty, there are no combinations to generate, so we immediately return an empty list.
3. Initialize the result container.

We create a list that will store all completed combinations.
4. Start backtracking from index `0`.

The recursive function keeps track of:

- The current digit index
- The current partial string being built
5. Check the base case.

If the current string length equals the number of digits, we have constructed a complete valid combination.

We append this combination to the result list.
6. Retrieve the letters for the current digit.

Using the digit-to-letter mapping, we get all possible character choices for the current position.
7. Try each possible letter.

For every letter:

- Add the letter to the current path
- Recurse to the next digit
- Remove the letter afterward

This removal step is what makes the process "backtracking". It restores the previous state so another choice can be explored.
8. Continue until all branches are explored.

The recursion systematically traverses the entire decision tree and generates every valid combination.

## Python Solution

```
from typing import List

class Solution:
    def letterCombinations(self, digits: str) -> List[str]:
        if not digits:
            return []

        digit_to_letters = {
            "2": "abc",
            "3": "def",
            "4": "ghi",
            "5": "jkl",
            "6": "mno",
            "7": "pqrs",
            "8": "tuv",
            "9": "wxyz",
        }

        result = []

        def backtrack(index: int, current_combination: str) -> None:
            if index == len(digits):
                result.append(current_combination)
                return

            current_digit = digits[index]
            possible_letters = digit_to_letters[current_digit]

            for letter in possible_letters:
                backtrack(index + 1, current_combination + letter)

        backtrack(0, "")

        return result
```

The implementation begins by checking whether the input string is empty. If it is, we return an empty list immediately.

Next, we define the phone keypad mapping using a dictionary. This provides fast lookup for the letters corresponding to each digit.

The `result` list stores all completed combinations.

The recursive `backtrack` function is the core of the algorithm. It receives the current digit index and the partial combination constructed so far.

The base case occurs when `index == len(digits)`. This means we have chosen one letter for every digit, so the combination is complete and can be added to the result list.

Otherwise, we retrieve the letters associated with the current digit and iterate through them. For each letter, we recursively continue building the combination.

The expression `current_combination + letter` creates the next partial string. Since Python strings are immutable, this automatically creates a new string for each recursive branch.

Finally, we start recursion with index `0` and an empty combination.

## Go Solution

```
func letterCombinations(digits string) []string {
	if len(digits) == 0 {
		return []string{}
	}

	digitToLetters := map[byte]string{
		'2': "abc",
		'3': "def",
		'4': "ghi",
		'5': "jkl",
		'6': "mno",
		'7': "pqrs",
		'8': "tuv",
		'9': "wxyz",
	}

	var result []string

	var backtrack func(index int, currentCombination string)

	backtrack = func(index int, currentCombination string) {
		if index == len(digits) {
			result = append(result, currentCombination)
			return
		}

		currentDigit := digits[index]
		possibleLetters := digitToLetters[currentDigit]

		for _, letter := range possibleLetters {
			backtrack(index+1, currentCombination+string(letter))
		}
	}

	backtrack(0, "")

	return result
}
```

The Go implementation follows the same recursive backtracking structure as the Python version.

One important difference is the use of `byte` keys in the map because Go strings are indexed as bytes. The recursive helper function is declared as a function variable so it can recursively reference itself.

Go strings are immutable just like Python strings, so concatenating `currentCombination + string(letter)` creates a new string for each recursive call.

The solution returns an empty slice when the input is empty, which matches LeetCode expectations.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(4^n * n) | There are at most 4 choices per digit, and constructing each string costs O(n) |
| Space | O(n) auxiliary, excluding output | Recursive call stack depth is at most n |

The recursion tree branches up to four times for each digit because digits like `7` and `9` map to four letters.

If the input length is `n`, the maximum number of combinations is `4^n`.

Each completed combination has length `n`, and creating those strings contributes an additional factor of `n` to the runtime.

The auxiliary space complexity comes from the recursion stack. Since recursion depth never exceeds the number of digits, the stack uses `O(n)` space.

The output list itself is not counted in auxiliary space complexity because it is required by the problem.

## Edge Cases

### Empty Input String

An empty string is a common source of bugs because recursion may incorrectly add an empty string to the result list.

For example:

```
Input: ""
```

The correct output is:

```
[]
```

Our implementation handles this by checking for empty input before starting recursion and immediately returning an empty list.

### Single Digit Input

A single digit tests whether the algorithm correctly handles the smallest non-empty case.

Example:

```
Input: "2"
```

The algorithm should simply return all letters mapped from that digit:

```
["a","b","c"]
```

The recursion handles this naturally because the base case is reached after selecting one character.

### Digits With Four Letters

Digits `7` and `9` map to four letters instead of three.

Example:

```
Input: "79"
```

A buggy implementation might incorrectly assume every digit has exactly three letters.

Our mapping explicitly stores the correct letter sets for every digit, so the algorithm automatically handles variable branching factors without special logic.

### Maximum Length Input

The maximum input length is `4`.

Example:

```
Input: "9999"
```

This produces:

```
4^4 = 256
```

possible combinations.

Although this is the largest possible search space, it is still small enough for backtracking to generate efficiently within time limits.
