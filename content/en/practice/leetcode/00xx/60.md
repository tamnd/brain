# LeetCode 60, Permutation Sequence

## Problem Understanding

This problem asks us to find the `kth` permutation of the numbers from `1` to `n`, when all permutations are ordered lexicographically.

For example, if `n = 3`, the numbers are:

```
[1, 2, 3]
```

All possible permutations in sorted order are:

```
123
132
213
231
312
321
```

If `k = 3`, the answer is `"213"` because it is the third permutation in lexicographical order.

The important detail is that we are not asked to generate all permutations and sort them ourselves. Instead, we need to directly determine which permutation appears in the `kth` position.

The input consists of:

- `n`, the number of digits from `1` through `n`
- `k`, the 1-indexed position of the desired permutation

The output is a string representing the `kth` permutation.

The constraints are relatively small:

```
1 <= n <= 9
1 <= k <= n!
```

Even though `n` is at most `9`, the number of permutations grows extremely quickly:

```
9! = 362880
```

Generating every permutation is possible for these limits, but it is unnecessarily expensive and not the intended solution. The problem is designed to test understanding of factorial number systems and combinatorial indexing.

Several edge cases are important:

- When `n = 1`, there is only one permutation.
- When `k = 1`, we must return the smallest lexicographical permutation.
- When `k = n!`, we must return the largest lexicographical permutation.
- Because `k` is 1-indexed, off-by-one mistakes are very common.
- Digits must not repeat, so once a number is used, it must be removed from future consideration.

## Approaches

### Brute Force Approach

The brute-force solution generates every possible permutation of the numbers `1` through `n`, sorts them lexicographically, and returns the `kth` one.

A recursive backtracking algorithm can generate all permutations. After generating them, we either:

1. Store all permutations and sort them, or
2. Generate them directly in lexicographical order

This approach is correct because it explicitly enumerates every valid arrangement.

However, the time complexity is very large because there are `n!` permutations, and each permutation has length `n`.

For `n = 9`, this means generating `362880` permutations, which is computationally expensive and unnecessary when we only need one specific permutation.

### Optimal Approach

The key insight is that permutations follow a predictable block structure based on factorials.

Consider `n = 4`:

```
1234
1243
1324
1342
1423
1432
2134
2143
...
```

Notice that:

- All permutations starting with `1` occupy the first `3! = 6` positions.
- All permutations starting with `2` occupy the next `6` positions.
- All permutations starting with `3` occupy the next `6` positions.
- All permutations starting with `4` occupy the final `6` positions.

This means we can determine the first digit directly using division.

More generally:

- Each digit fixes a block of `(n-1)!` permutations.
- We can determine which block contains the `kth` permutation.
- Then we recursively determine the remaining digits.

This is essentially representing the permutation index in the factorial number system.

Instead of generating permutations, we construct the answer digit by digit.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n! * n) | O(n! * n) | Generates every permutation |
| Optimal | O(n²) | O(n) | Uses factorial indexing to directly build answer |

## Algorithm Walkthrough

1. Create a list containing all available digits from `1` to `n`.

For example, if `n = 4`:

```
digits = [1, 2, 3, 4]
```

We will remove digits from this list as they are used.
2. Precompute factorial values.

We need factorials because each fixed prefix corresponds to a block of permutations.

For `n = 4`:

```
factorials:
0! = 1
1! = 1
2! = 2
3! = 6
4! = 24
```
3. Convert `k` from 1-indexed to 0-indexed.

This is extremely important.

Since array indexing starts at `0`, we subtract one:

```
k = k - 1
```
4. Determine the first digit.

The number of permutations per leading digit is:

```
block_size = (n - 1)!
```

The correct digit index is:

```
index = k // block_size
```

We select the digit at that index from the remaining digits.
5. Remove the chosen digit.

Once a digit is used, it cannot appear again.
6. Update `k`.

After choosing a block, we determine the offset inside that block:

```
k = k % block_size
```
7. Repeat for the remaining positions.

Each step reduces the number of remaining digits by one.
8. Continue until all digits are used.

The constructed sequence is the answer.

## Python Solution

```
class Solution:
    def getPermutation(self, n: int, k: int) -> str:
        # Precompute factorials
        factorials = [1] * (n + 1)

        for i in range(1, n + 1):
            factorials[i] = factorials[i - 1] * i

        # Available digits
        digits = [str(i) for i in range(1, n + 1)]

        # Convert k to 0-indexed
        k -= 1

        result = []

        # Build permutation one digit at a time
        for remaining in range(n, 0, -1):
            block_size = factorials[remaining - 1]

            index = k // block_size

            result.append(digits[index])

            # Remove used digit
            digits.pop(index)

            # Update k for next position
            k %= block_size

        return "".join(result)
```

The implementation begins by precomputing factorial values from `0!` through `n!`. These factorials allow constant-time lookup of permutation block sizes during the algorithm.

Next, we create the `digits` list containing all currently available numbers. These are stored as strings because the final answer must be returned as a string.

The conversion from 1-indexed to 0-indexed is critical. Without subtracting one from `k`, the block calculations would be incorrect.

The main loop processes positions from left to right. At each step:

- `block_size` determines how many permutations belong to each possible leading digit.
- Integer division identifies which digit block contains the target permutation.
- That digit is appended to the result.
- The digit is removed from future consideration.
- `k` is reduced to its position within the selected block.

Finally, the result characters are joined into the final permutation string.

## Go Solution

```
func getPermutation(n int, k int) string {
    // Precompute factorials
    factorials := make([]int, n+1)
    factorials[0] = 1

    for i := 1; i <= n; i++ {
        factorials[i] = factorials[i-1] * i
    }

    // Available digits
    digits := make([]byte, n)

    for i := 0; i < n; i++ {
        digits[i] = byte('1' + i)
    }

    // Convert k to 0-indexed
    k--

    result := make([]byte, 0, n)

    for remaining := n; remaining >= 1; remaining-- {
        blockSize := factorials[remaining-1]

        index := k / blockSize

        result = append(result, digits[index])

        // Remove used digit
        digits = append(digits[:index], digits[index+1:]...)

        // Update k
        k %= blockSize
    }

    return string(result)
}
```

The Go implementation follows the same algorithmic structure as the Python solution.

One notable difference is that Go uses byte slices instead of string lists for efficiency. Since all digits are single ASCII characters, byte manipulation is straightforward and memory-efficient.

Removing a used digit requires slice reconstruction:

```
digits = append(digits[:index], digits[index+1:]...)
```

Go does not provide Python-style `pop`, so this technique removes the selected element while preserving order.

Integer overflow is not a concern because `9! = 362880`, which easily fits within Go's `int` type.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n²) | Removing elements from a list or slice costs O(n) and happens n times |
| Space | O(n) | Storage for factorials, digits, and result |

The algorithm runs in quadratic time because each removal from the middle of the `digits` list requires shifting elements. Since this occurs once per digit, the total complexity becomes:

```
O(n + (n-1) + (n-2) + ...)
= O(n²)
```

The space usage is linear because we store factorials, remaining digits, and the output string.

## Edge Cases

### Edge Case 1, Smallest Input

When `n = 1`, there is only one possible permutation:

```
"1"
```

A naive implementation might incorrectly assume multiple factorial blocks exist, leading to division or indexing issues. This implementation handles the case naturally because the loop runs exactly once and selects the only available digit.

### Edge Case 2, k Equals 1

When `k = 1`, we must return the lexicographically smallest permutation.

For example:

```
n = 4
k = 1
```

The answer should be:

```
"1234"
```

After converting to zero-based indexing, `k` becomes `0`, which causes every index calculation to select the first remaining digit. This correctly produces the smallest permutation.

### Edge Case 3, k Equals n!

When `k = n!`, we must return the lexicographically largest permutation.

For example:

```
n = 3
k = 6
```

The answer is:

```
"321"
```

After subtracting one, `k` becomes the largest valid zero-based index. Each division selects the last remaining digit, correctly constructing the maximum permutation.

### Edge Case 4, Off-by-One Errors

This problem is especially prone to indexing mistakes because permutations are naturally described using 1-based indexing, while arrays use 0-based indexing.

If we forget:

```
k -= 1
```

the algorithm shifts every answer by one position and produces incorrect permutations.

The implementation explicitly converts `k` before any calculations, ensuring all division and modulo operations align correctly with array indexing.
