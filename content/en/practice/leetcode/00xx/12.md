## Solution

## Problem Understanding

The problem asks us to convert a positive integer into its Roman numeral representation. Roman numerals use specific symbols to represent values, and the final numeral is constructed according to a set of strict formatting rules.

The input is a single integer `num`, where `1 <= num <= 3999`. The output is a string containing the valid Roman numeral representation of that integer.

Roman numerals are not purely additive. In many cases, symbols are appended from largest to smallest value, but there are special subtractive cases that must be handled carefully. For example:

- `III` represents 3
- `VIII` represents 8
- `XXVII` represents 27

However, numbers like 4 and 9 use subtractive notation:

- `IV` represents 4
- `IX` represents 9

The same pattern applies to larger place values:

- `XL` represents 40
- `XC` represents 90
- `CD` represents 400
- `CM` represents 900

A key detail is that Roman numeral construction is based on decimal place decomposition, not arbitrary subtraction. For example:

- `49` is written as `XLIX`
- It is not written as `IL`

The constraints are small, since the maximum value is only 3999. This means performance is not a major concern, but correctness is extremely important because Roman numeral formatting rules are strict.

Several edge cases are important:

- Numbers involving subtractive notation such as `4`, `9`, `40`, `90`, `400`, and `900`
- Large values like `3999`, which produce long Roman numeral strings
- Numbers requiring repeated symbols such as `3 = III` or `3000 = MMM`
- Numbers mixing additive and subtractive patterns like `1994 = MCMXCIV`

The problem guarantees that the input is always within the valid Roman numeral conversion range, so we do not need to handle invalid or negative inputs.

## Approaches

### Brute Force Approach

A brute-force approach would simulate Roman numeral construction one value at a time. We could repeatedly subtract the largest possible Roman value from the number and append the corresponding symbol.

For example, for `58`:

- Subtract `50`, append `L`
- Subtract `5`, append `V`
- Subtract `1` three times, append `III`

This works because Roman numerals are fundamentally built greedily from largest to smallest values.

However, a naive brute-force implementation may repeatedly process individual units unnecessarily. For example, it might append `I` one character at a time for repeated values instead of handling structured mappings efficiently. It also becomes more cumbersome when handling subtractive cases separately.

Even though the constraints are small enough that this solution would still pass, it is not the cleanest or most maintainable approach.

### Optimal Approach

The key insight is that Roman numeral construction follows a fixed set of value-symbol mappings, including both standard and subtractive forms.

If we prepare all valid Roman numeral building blocks in descending order:

- `1000 -> M`
- `900 -> CM`
- `500 -> D`
- `400 -> CD`
- ...
- `1 -> I`

then we can greedily consume the number from largest to smallest.

At each step:

- Find the largest Roman value less than or equal to the remaining number
- Append its symbol
- Subtract its value
- Continue until the number becomes zero

This works because Roman numerals are designed around greedy decomposition using these canonical symbols.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Repeatedly subtracts values and appends symbols incrementally |
| Optimal | O(1) | O(1) | Uses predefined Roman mappings and greedy construction |

Although the optimal approach technically processes a constant-sized mapping table, the generated Roman numeral length is bounded by a small constant because `num <= 3999`.

## Algorithm Walkthrough

1. Create a list of Roman numeral mappings ordered from largest value to smallest value.

This ordering is critical because Roman numerals must always prioritize the largest valid symbol first. The mapping includes both normal and subtractive forms:

- `1000 -> M`
- `900 -> CM`
- `500 -> D`
- ...
- `1 -> I`
2. Initialize an empty result string.

This string will store the Roman numeral characters as we construct the answer.
3. Iterate through each `(value, symbol)` pair in descending order.

For each pair, determine how many times the current Roman value fits into the remaining number.
4. While the current value can still be subtracted from the number:

- Append the corresponding Roman symbol to the result
- Subtract the value from `num`

This greedy step ensures we always use the largest possible Roman symbol first.
5. Continue processing all mappings until `num` becomes zero.

At that point, the Roman numeral construction is complete.
6. Return the final result string.

The data structure used here is simply a list of tuples or pairs. A hash map is unnecessary because ordering matters more than lookup speed. The algorithm depends on processing values from largest to smallest in a deterministic sequence.

## Python Solution

```
class Solution:
    def intToRoman(self, num: int) -> str:
        roman_mappings = [
            (1000, "M"),
            (900, "CM"),
            (500, "D"),
            (400, "CD"),
            (100, "C"),
            (90, "XC"),
            (50, "L"),
            (40, "XL"),
            (10, "X"),
            (9, "IX"),
            (5, "V"),
            (4, "IV"),
            (1, "I")
        ]

        result = []

        for value, symbol in roman_mappings:
            while num >= value:
                result.append(symbol)
                num -= value

        return "".join(result)
```

The implementation begins by defining all Roman numeral mappings in descending order. Including subtractive forms like `CM`, `XC`, and `IV` directly in the table simplifies the logic significantly because we do not need separate conditional handling for those cases.

The `result` list is used instead of repeatedly concatenating strings. In Python, appending to a list and joining at the end is more efficient than repeated string concatenation.

The outer loop iterates through each Roman numeral mapping. For each value-symbol pair, the inner `while` loop repeatedly applies that symbol as long as the current value can still fit into the remaining number.

For example, if `num = 3749`:

- `1000` fits three times, so append `"MMM"`
- `500` fits once, append `"D"`
- `100` fits twice, append `"CC"`
- `40` fits once, append `"XL"`
- `9` fits once, append `"IX"`

Finally, all accumulated symbols are joined into a single string and returned.

## Go Solution

```
func intToRoman(num int) string {
    values := []int{
        1000, 900, 500, 400,
        100, 90, 50, 40,
        10, 9, 5, 4, 1,
    }

    symbols := []string{
        "M", "CM", "D", "CD",
        "C", "XC", "L", "XL",
        "X", "IX", "V", "IV", "I",
    }

    result := ""

    for i := 0; i < len(values); i++ {
        for num >= values[i] {
            result += symbols[i]
            num -= values[i]
        }
    }

    return result
}
```

The Go implementation follows the same greedy strategy as the Python solution. Since Go does not have tuple literals like Python, the values and symbols are stored in parallel slices.

String concatenation is acceptable here because the output length is very small and bounded by the problem constraints. For larger-scale string building, `strings.Builder` would typically be preferred.

Go integers comfortably handle the maximum value of `3999`, so there are no overflow concerns.

## Edge Cases

One important edge case is numbers that require subtractive notation, such as `4`, `9`, `40`, `90`, `400`, and `900`. A naive implementation that only appends symbols greedily without including subtractive mappings could incorrectly generate outputs like `IIII` instead of `IV`. The implementation handles this correctly by explicitly including subtractive combinations in the mapping table.

Another important edge case is the maximum valid input, `3999`. This value expands into `MMMCMXCIX`, which combines repeated symbols and subtractive forms. Some implementations fail because they do not properly limit repeated symbols or because they process place values incorrectly. The greedy ordered mapping guarantees the correct construction.

A third edge case is numbers composed mostly of repeated symbols, such as `3`, `30`, or `3000`. Roman numerals allow powers of ten to repeat at most three times consecutively. The implementation naturally respects this because the mapping table transitions to subtractive forms like `4 -> IV` before a fourth repetition could occur.

Another subtle case is mixed additive and subtractive decomposition, such as `49`. Incorrect implementations may attempt invalid constructions like `IL`. The algorithm avoids this entirely because it always processes valid Roman numeral building blocks in descending order, resulting in the correct output `XLIX`.
