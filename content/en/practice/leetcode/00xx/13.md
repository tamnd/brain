## Solution

## Problem Understanding

The problem asks us to convert a Roman numeral string into its corresponding integer value. Roman numerals use specific symbols to represent values, and the symbols are usually arranged from largest to smallest. The challenge comes from the special subtraction cases where a smaller numeral appears before a larger numeral.

The Roman numeral symbols and their values are:

| Symbol | Value |
| --- | --- |
| I | 1 |
| V | 5 |
| X | 10 |
| L | 50 |
| C | 100 |
| D | 500 |
| M | 1000 |

Normally, Roman numerals are additive. For example, `VIII` means `5 + 1 + 1 + 1 = 8`. However, there are specific subtraction patterns:

- `IV = 4`
- `IX = 9`
- `XL = 40`
- `XC = 90`
- `CD = 400`
- `CM = 900`

The input is a string `s` containing a valid Roman numeral. The output should be the integer value represented by that numeral.

The constraints tell us several important things:

- The input length is at most 15 characters, which is very small.
- The input is guaranteed to be valid.
- The value range is from 1 to 3999.

Because the input is guaranteed valid, we do not need to perform error checking for invalid Roman numeral sequences. This simplifies the implementation significantly.

Several edge cases are important to consider. A naive implementation that simply adds all values would fail for subtraction cases like `IV` or `IX`. Another potential issue is correctly handling multiple subtraction pairs within the same string, such as `MCMXCIV`. Single-character numerals like `I` or `V` must also work correctly without accessing out-of-bounds indices.

## Approaches

### Brute Force Approach

A straightforward brute-force solution is to explicitly check for every subtraction pair while iterating through the string. We could scan the string character by character and attempt to match two-character combinations such as `IV`, `IX`, `XL`, `XC`, `CD`, and `CM`.

If a subtraction pair is found, we add its special value and skip the next character. Otherwise, we add the normal value of the current symbol.

This approach works because Roman numerals only contain six valid subtraction combinations, so explicitly checking them guarantees correctness. However, the implementation becomes more verbose because we must manually handle many cases.

Although the constraints are small enough that this solution would pass comfortably, it is not the cleanest or most elegant approach.

### Optimal Approach

The key insight is that subtraction cases can be detected using adjacent values.

If a Roman numeral symbol has a smaller value than the symbol immediately after it, then it should be subtracted instead of added.

For example:

- In `IV`, `I < V`, so we subtract `1`
- In `IX`, `I < X`, so we subtract `1`
- In `MCM`, `C < M`, so we subtract `100`

Otherwise, we add the value normally.

This observation allows us to solve the problem with a single pass through the string using a hash map for value lookup.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Explicitly checks all subtraction pairs |
| Optimal | O(n) | O(1) | Uses adjacent-value comparison to determine add vs subtract |

## Algorithm Walkthrough

1. Create a hash map that stores the integer value for each Roman numeral symbol. This allows constant-time lookup for every character in the string.
2. Initialize a variable called `total` to store the final integer result.
3. Iterate through the string from left to right.
4. For each character, retrieve its numeric value from the hash map.
5. Check whether the current symbol forms a subtraction pair:

- If the current value is smaller than the next value, subtract the current value from `total`.
- Otherwise, add the current value to `total`.
6. Continue until all characters have been processed.
7. Return the final value stored in `total`.

The reason this works is that subtraction cases are precisely the situations where a smaller numeral appears before a larger numeral. Every other case is additive.

A hash map is the ideal data structure here because it provides very fast lookup while keeping the implementation simple and readable.

## Python Solution

```
class Solution:
    def romanToInt(self, s: str) -> int:
        roman_values = {
            "I": 1,
            "V": 5,
            "X": 10,
            "L": 50,
            "C": 100,
            "D": 500,
            "M": 1000,
        }

        total = 0

        for i in range(len(s)):
            current_value = roman_values[s[i]]

            if i + 1 < len(s) and current_value < roman_values[s[i + 1]]:
                total -= current_value
            else:
                total += current_value

        return total
```

The implementation begins by defining a dictionary that maps each Roman numeral symbol to its integer value. This allows quick access to values during iteration.

The variable `total` keeps track of the accumulated result.

The loop iterates through every character in the string. For each position, the algorithm retrieves the current numeral's value.

The condition checks whether there is a next character and whether the current value is smaller than the next value. If both are true, this indicates a subtraction case such as `IV` or `XC`, so the current value is subtracted.

Otherwise, the current value is added normally.

At the end of the loop, `total` contains the correct integer representation of the Roman numeral.

## Go Solution

```
func romanToInt(s string) int {
    romanValues := map[byte]int{
        'I': 1,
        'V': 5,
        'X': 10,
        'L': 50,
        'C': 100,
        'D': 500,
        'M': 1000,
    }

    total := 0

    for i := 0; i < len(s); i++ {
        currentValue := romanValues[s[i]]

        if i+1 < len(s) && currentValue < romanValues[s[i+1]] {
            total -= currentValue
        } else {
            total += currentValue
        }
    }

    return total
}
```

The Go implementation follows the same logic as the Python version. The primary difference is the use of a `map[byte]int` because Go strings are byte sequences.

Roman numeral characters are ASCII characters, so indexing the string by byte is perfectly safe.

There are no concerns about integer overflow because the maximum Roman numeral value is only 3999, which easily fits inside Go's `int` type.

## Complexity Analysis

The algorithm performs a single linear scan through the input string. Each iteration performs only constant-time operations, specifically hash map lookups and comparisons.

The extra space usage is constant because the Roman numeral mapping always contains exactly seven entries regardless of input size.

## Edge Cases

A very important edge case is a numeral containing subtraction notation, such as `IV` or `IX`. A naive solution that simply sums all values would incorrectly produce `6` for `IV` instead of `4`. The implementation handles this correctly by comparing the current symbol with the next symbol and subtracting when the current value is smaller.

Another important edge case is a numeral containing multiple subtraction pairs, such as `MCMXCIV`. This case is tricky because subtraction occurs multiple times throughout the string. The algorithm handles this naturally because each character independently checks whether it should be added or subtracted.

Single-character numerals such as `I`, `V`, or `M` are also important. These cases can cause index-out-of-bounds errors if the implementation blindly accesses the next character. The condition `i + 1 < len(s)` ensures that the algorithm safely checks whether a next character exists before comparing values.

A final edge case is the largest possible Roman numeral, `MMMCMXCIX`, which equals `3999`. The implementation correctly handles large values because all operations remain constant-time integer arithmetic, and the result easily fits within standard integer ranges.
