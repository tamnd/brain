## Solution

## Problem Understanding

This problem asks us to determine whether one string can be transformed into another through a recursive scrambling process.

We are given two strings, `s1` and `s2`, which are guaranteed to have the same length. A string is considered a **scramble** of another string if it can be generated through a recursive process where the string is repeatedly split into two non-empty parts, and at each split we may either keep the order of the parts or swap them.

The important detail is that scrambling is recursive. Even if two large segments are not swapped directly, smaller internal segments may be rearranged independently. This makes the problem significantly more difficult than simply checking whether the strings contain the same characters.

For example:

```
s1 = "great"
s2 = "rgeat"
```

The transformation works because `"great"` can first be split into `"gr"` and `"eat"`, then `"gr"` can be recursively scrambled into `"rg"` while `"eat"` remains unchanged.

The input represents:

- `s1`: the original string
- `s2`: the target string we want to verify as a possible scramble

The expected output is:

- `true` if `s2` can be formed by recursively scrambling `s1`
- `false` otherwise

The constraints are important:

- `1 <= s1.length <= 30`
- Both strings consist of lowercase English letters.

A maximum length of only `30` strongly suggests that exponential brute force might initially seem possible, but it will still be far too slow because the number of recursive split combinations grows extremely quickly. Since many overlapping subproblems occur, dynamic programming becomes a natural optimization.

Several edge cases are important to recognize upfront.

If the strings are already identical, the answer is immediately `true` because no scrambling is needed.

If the strings contain different character frequencies, the answer must be `false`. For example:

```
s1 = "abc"
s2 = "abd"
```

No scrambling can create or remove characters, so mismatched character counts immediately invalidate the possibility.

Another tricky case occurs when the strings contain repeated characters. For example:

```
s1 = "aabb"
s2 = "abab"
```

A naive implementation may incorrectly prune valid possibilities or repeat expensive recursive work. Memoization becomes especially important in these cases.

## Approaches

### Brute Force Approach

The brute force solution recursively tries every possible split position and every possible swap configuration.

At every recursive step, we split the string at every index `i` where `1 <= i < n`. For each split, we consider two possibilities:

1. No swap:

- Left part matches left part
- Right part matches right part
2. Swap:

- Left part matches right part
- Right part matches left part

We recursively test every combination until we either find a valid scrambling configuration or exhaust all possibilities.

This approach is correct because it exhaustively explores every legal scrambling operation allowed by the problem definition.

However, it becomes extremely slow because identical substring comparisons occur repeatedly. For example, the same pair of substrings might be checked dozens or hundreds of times through different recursive paths.

The recursion tree grows exponentially, making brute force infeasible even for relatively small input sizes.

### Key Insight for an Optimal Solution

The key observation is that the same substring pairs are evaluated repeatedly.

Suppose we want to determine:

```
"great" → "rgeat"
```

During recursion, we may repeatedly ask questions such as:

```
"is 'gr' a scramble of 'rg'?"
"is 'eat' a scramble of 'eat'?"
```

Instead of recomputing these repeatedly, we can cache the result.

This transforms the problem into **top-down dynamic programming with memoization**.

We define a recursive function:

```
dfs(a, b)
```

which answers:

```
Can substring a be scrambled into substring b?
```

To optimize further, we first check whether the substrings contain identical character frequencies. If not, we can immediately return `false`, preventing unnecessary recursion.

The algorithm then tries every split point and recursively verifies both the swapped and non-swapped configurations.

Memoization ensures that every unique substring pair is computed only once.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential, approximately O(2^n) or worse | O(n) recursion stack | Tries all recursive split combinations repeatedly |
| Optimal Memoized DP | O(n^5) | O(n^4) | Uses memoization to avoid repeated substring evaluations |

## Algorithm Walkthrough

### Step 1: Handle obvious invalid cases

Before doing expensive recursion, check whether the lengths are equal.

The problem guarantees equal length, but this is still good defensive programming.

If the strings are already identical, return `true` immediately because no scrambling is necessary.

### Step 2: Create a memoization cache

We use a hash map to store previously computed results.

The key is:

```
(substring1, substring2)
```

The value is:

```
true or false
```

This prevents repeated work.

For example:

```
("gr", "rg") → true
```

Once computed, future recursive calls can reuse it instantly.

### Step 3: Compare character frequencies

Before exploring recursive splits, verify that both substrings contain the same characters.

If:

```
sorted(a) != sorted(b)
```

or equivalently:

```
character counts differ
```

then scrambling is impossible.

This pruning dramatically reduces unnecessary recursion.

### Step 4: Try every split point

For a substring of length `n`, try every split:

```
1 to n - 1
```

At each split, test two possibilities.

#### Case 1: No swap

Check:

```
left(a) ↔ left(b)
right(a) ↔ right(b)
```

#### Case 2: Swap

Check:

```
left(a) ↔ right(b)
right(a) ↔ left(b)
```

If either succeeds, store `true` in the memo and return immediately.

### Step 5: Cache failed attempts

If all splits fail, store:

```
false
```

in the memo before returning.

This prevents recomputation of failed states.

## Python Solution

```
from collections import Counter
from functools import lru_cache

class Solution:
    def isScramble(self, s1: str, s2: str) -> bool:
        if len(s1) != len(s2):
            return False

        @lru_cache(maxsize=None)
        def dfs(a: str, b: str) -> bool:
            if a == b:
                return True

            if Counter(a) != Counter(b):
                return False

            n = len(a)

            for split_index in range(1, n):
                # No swap case
                if (
                    dfs(a[:split_index], b[:split_index])
                    and dfs(a[split_index:], b[split_index:])
                ):
                    return True

                # Swap case
                if (
                    dfs(a[:split_index], b[n - split_index:])
                    and dfs(a[split_index:], b[: n - split_index])
                ):
                    return True

            return False

        return dfs(s1, s2)
```

The implementation uses a recursive helper function named `dfs`, which determines whether one substring can scramble into another.

The `@lru_cache` decorator provides automatic memoization. This is important because the same substring pairs appear repeatedly during recursion.

The first condition checks whether the substrings are already identical. If they are, we immediately return `True`.

The second optimization compares character frequencies using `Counter`. If the characters differ, scrambling is impossible and recursion stops early.

The main recursive logic iterates through every split point. At each split, we check both valid scrambling scenarios: keeping the same order and swapping the two halves.

If either configuration works, we return `True`. Otherwise, after testing every split, we return `False`.

## Go Solution

```
package main

func isScramble(s1 string, s2 string) bool {
	if len(s1) != len(s2) {
		return false
	}

	memo := make(map[string]bool)
	visited := make(map[string]bool)

	var dfs func(string, string) bool

	dfs = func(a string, b string) bool {
		key := a + "#" + b

		if visited[key] {
			return memo[key]
	 }

		if a == b {
			return true
		}

		if !sameCharacterCount(a, b) {
			return false
		}

		n := len(a)

		for splitIndex := 1; splitIndex < n; splitIndex++ {
			// No swap case
			if dfs(a[:splitIndex], b[:splitIndex]) &&
				dfs(a[splitIndex:], b[splitIndex:]) {
				visited[key] = true
				memo[key] = true
				return true
			}

			// Swap case
			if dfs(a[:splitIndex], b[n-splitIndex:]) &&
				dfs(a[splitIndex:], b[:n-splitIndex]) {
				visited[key] = true
				memo[key] = true
				return true
			}
		}

		visited[key] = true
		memo[key] = false
		return false
	}

	return dfs(s1, s2)
}

func sameCharacterCount(a string, b string) bool {
	count := make([]int, 26)

	for i := 0; i < len(a); i++ {
		count[a[i]-'a']++
		count[b[i]-'a']--
	}

	for _, value := range count {
		if value != 0 {
			return false
		}
	}

	return true
}
```

The Go implementation follows the same recursive memoized strategy, but there are a few language-specific differences.

Unlike Python's `lru_cache`, Go does not provide built-in memoization, so we manually maintain two hash maps:

- `memo` stores computed results
- `visited` tracks whether a state has already been computed

This distinction matters because `false` is the zero value for booleans in Go, so we need `visited` to distinguish between "computed false" and "not computed yet".

Instead of using `Counter`, the Go solution uses a fixed-size integer array of length `26` to count lowercase English letters. This is more efficient because the problem guarantees only lowercase characters.

No integer overflow concerns exist here because string lengths are at most `30`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n^5) | There are O(n^4) substring states and each may try O(n) split positions |
| Space | O(n^4) | Memoization cache stores substring-pair states |

The time complexity comes from counting the number of unique recursive states. A state is defined by two substrings, and there are approximately `O(n^4)` such pairs. For each state, we attempt every possible split point, adding another factor of `O(n)`.

The space complexity comes primarily from memoization storage. Since each substring pair may be cached, we store up to `O(n^4)` states. The recursion depth itself is at most `O(n)`.

## Edge Cases

### Edge Case 1: Strings Are Already Equal

Consider:

```
s1 = "abc"
s2 = "abc"
```

A buggy implementation might continue recursively splitting even though no work is needed.

Our implementation handles this immediately:

```
if a == b:
    return True
```

This avoids unnecessary recursion and improves performance.

### Edge Case 2: Different Character Frequencies

Consider:

```
s1 = "abc"
s2 = "abd"
```

No scrambling operation can create or destroy characters.

Without character frequency pruning, recursion would waste time exploring impossible split combinations.

Our implementation stops immediately:

```
if Counter(a) != Counter(b):
    return False
```

The Go version performs the same optimization using a frequency array.

### Edge Case 3: Repeated Characters

Consider:

```
s1 = "aabb"
s2 = "abab"
```

Repeated characters create many overlapping recursive possibilities.

A naive recursion may recompute identical substring comparisons many times, causing severe performance degradation.

Memoization solves this problem by caching:

```
("ab", "ba") → true
("aa", "aa") → true
```

Once computed, these results are reused instantly.

### Edge Case 4: Single Character Strings

Consider:

```
s1 = "a"
s2 = "b"
```

Since length is `1`, no splitting is possible.

The algorithm correctly falls back to direct comparison and character count checking, returning `false`.

This prevents invalid recursive behavior on minimal input sizes.
