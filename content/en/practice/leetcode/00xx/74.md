## Solution

## Problem Understanding

The problem gives us a two dimensional matrix with two very important ordering properties:

1. Each row is sorted in non-decreasing order.
2. The first element of every row is greater than the last element of the previous row.

These two guarantees together mean that if we read the matrix row by row from left to right, the entire matrix behaves exactly like one fully sorted one dimensional array.

For example, this matrix:

```
[
  [1, 3, 5, 7],
  [10, 11, 16, 20],
  [23, 30, 34, 60]
]
```

can conceptually be viewed as:

```
[1, 3, 5, 7, 10, 11, 16, 20, 23, 30, 34, 60]
```

The task is to determine whether a given integer `target` exists in the matrix.

The problem explicitly requires an `O(log(m * n))` solution, where:

- `m` is the number of rows
- `n` is the number of columns

This time complexity requirement strongly suggests binary search, because binary search operates in logarithmic time on sorted data.

The constraints are relatively small:

- `1 <= m, n <= 100`

Even though a brute force solution would pass due to the small limits, the problem is testing whether we recognize the sorted structure and apply binary search correctly.

There are several important edge cases to consider:

- A matrix with only one row
- A matrix with only one column
- A target smaller than the smallest element
- A target larger than the largest element
- A target that falls between rows
- Negative numbers
- The smallest possible matrix, `1 x 1`

The problem guarantees that the matrix is non-empty and every row has the same length, so we do not need to handle malformed input.

## Approaches

### Brute Force Approach

The most straightforward solution is to scan every element in the matrix one by one.

We iterate through each row, then iterate through each value in that row. If we find the target, we return `true`. If we finish scanning the entire matrix without finding it, we return `false`.

This approach is correct because it checks every possible location where the target could exist.

However, this solution ignores the sorted properties of the matrix. In the worst case, we examine all `m * n` elements, resulting in linear time complexity.

Since the problem explicitly asks for `O(log(m * n))`, the brute force approach is not sufficient.

### Optimal Approach, Binary Search on a Virtual Sorted Array

The key insight is that the matrix behaves like one globally sorted array.

Because:

- Rows are individually sorted
- Each row starts with a value greater than the previous row's last value

we can flatten the matrix conceptually without actually creating a new array.

Instead of physically converting the matrix into a one dimensional array, we map one dimensional indices back into two dimensional coordinates.

Suppose the matrix has:

- `rows = m`
- `cols = n`

Then:

- Total elements = `m * n`
- Valid virtual indices range from `0` to `m * n - 1`

For any virtual index `mid`:

```
row = mid // cols
col = mid % cols
```

This lets us access the corresponding matrix element directly.

Because the virtual array is sorted, standard binary search works perfectly.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m * n) | O(1) | Checks every element sequentially |
| Optimal | O(log(m * n)) | O(1) | Uses binary search on a virtual flattened array |

## Algorithm Walkthrough

1. Determine the matrix dimensions.

We store:

- `rows = len(matrix)`
- `cols = len(matrix[0])`

These values help us map one dimensional indices back into matrix coordinates.
2. Treat the matrix as a virtual sorted array.

Instead of creating a new flattened array, we imagine the matrix as a sorted sequence of length:

```
rows * cols
```
3. Initialize binary search boundaries.

Set:

```
left = 0
right = rows * cols - 1
```

These represent the first and last virtual indices.
4. Perform binary search while `left <= right`.

At each iteration:

- Compute the middle index:

```
mid = (left + right) // 2
```
5. Convert the virtual index into matrix coordinates.

Since each row contains `cols` elements:

```
row = mid // cols
col = mid % cols
```

This mapping gives us the actual matrix position.
6. Compare the current value with the target.

Let:

```
value = matrix[row][col]
```

Then:

- If `value == target`, return `true`
- If `value < target`, search the right half
- If `value > target`, search the left half
7. Continue until the search space becomes empty.

If the loop finishes without finding the target, return `false`.

This algorithm works because binary search repeatedly cuts the search space in half, giving logarithmic time complexity.

## Python Solution

```
from typing import List

class Solution:
    def searchMatrix(self, matrix: List[List[int]], target: int) -> bool:
        rows = len(matrix)
        cols = len(matrix[0])

        left = 0
        right = rows * cols - 1

        while left <= right:
            mid = (left + right) // 2

            row = mid // cols
            col = mid % cols

            value = matrix[row][col]

            if value == target:
                return True
            elif value < target:
                left = mid + 1
            else:
                right = mid - 1

        return False
```

The implementation begins by determining the number of rows and columns. These dimensions are necessary because the matrix is treated as a virtual one dimensional array.

The variables `left` and `right` define the binary search boundaries over this virtual array.

Inside the loop, we compute the midpoint index and translate it into matrix coordinates using division and modulo operations.

The value at that position is compared with the target:

- Equal means the target exists
- Smaller means the target must be to the right
- Larger means the target must be to the left

If the loop finishes without returning, the target does not exist in the matrix.

## Go Solution

```
func searchMatrix(matrix [][]int, target int) bool {
    rows := len(matrix)
    cols := len(matrix[0])

    left := 0
    right := rows*cols - 1

    for left <= right {
        mid := left + (right-left)/2

        row := mid / cols
        col := mid % cols

        value := matrix[row][col]

        if value == target {
            return true
        } else if value < target {
            left = mid + 1
        } else {
            right = mid - 1
        }
    }

    return false
}
```

The Go implementation follows exactly the same logic as the Python solution.

One small difference is the midpoint calculation:

```
mid := left + (right-left)/2
```

This style avoids potential integer overflow in languages where integer ranges are fixed. While overflow is not a practical issue for this problem's constraints, this is considered good binary search practice in Go and many other languages.

Go slices are used naturally for the matrix representation, and no additional memory allocation is needed.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(log(m * n)) | Binary search halves the search space each iteration |
| Space | O(1) | Only constant extra variables are used |

The matrix contains `m * n` total elements. Binary search operates on this virtual array, reducing the search space by half each iteration. Therefore, the total number of operations is logarithmic in the number of elements.

No auxiliary data structures are created, so the algorithm uses constant extra memory.

## Edge Cases

### Single Element Matrix

A matrix like:

```
[[5]]
```

is the smallest valid input.

This case is important because indexing mistakes often appear when there is only one element. Our implementation handles it naturally because binary search still works correctly when `left == right == 0`.

### Target Smaller Than All Elements

Consider:

```
matrix = [[10, 20]]
target = 5
```

The binary search quickly determines that the target is smaller than the minimum value and moves the search boundary leftward until the search space becomes empty.

The algorithm correctly returns `false`.

### Target Larger Than All Elements

Consider:

```
matrix = [[1, 2, 3]]
target = 10
```

The search repeatedly moves right until `left > right`.

The implementation correctly handles this without any out of bounds access.

### Single Row or Single Column

Examples:

```
[[1, 3, 5, 7]]
```

or

```
[
 [1],
 [3],
 [5]
]
```

These cases are important because division and modulo logic must still correctly map indices.

The formulas:

```
row = mid // cols
col = mid % cols
```

continue to work correctly regardless of whether there is only one row or one column.

### Target Between Rows

Consider:

```
[
 [1, 2, 3],
 [10, 11, 12]
]
```

with:

```
target = 5
```

A naive row selection strategy might fail here if implemented incorrectly.

The virtual array approach avoids this issue entirely because the entire matrix is treated as one globally sorted sequence.
