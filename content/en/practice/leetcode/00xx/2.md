## Solution

## Problem Understanding

This problem asks us to add two non-negative integers, but instead of receiving the numbers directly, we are given them as linked lists. Each node in the linked list contains a single digit, and the digits are stored in **reverse order**.

For example, the linked list:

```
2 -> 4 -> 3
```

actually represents the number:

```
342
```

because the digits are reversed. Similarly:

```
5 -> 6 -> 4
```

represents:

```
465
```

The task is to add these two numbers and return the result as a new linked list, also in reverse order.

For the previous example:

```
342 + 465 = 807
```

Since the result must also be reversed, we return:

```
7 -> 0 -> 8
```

which corresponds to:

```
807
```

The input consists of two non-empty singly linked lists, `l1` and `l2`. Every node contains a digit between `0` and `9`. The linked lists may have different lengths, and there are no leading zeros except for the number `0` itself.

The constraints tell us that each linked list has at most `100` nodes. This is small enough that an `O(n)` or `O(n log n)` solution is more than sufficient, but we should still aim for an elegant and efficient implementation.

A few important edge cases stand out immediately:

- The linked lists may have different lengths, such as `[9,9,9]` and `[1]`.
- A carry may remain after processing the final digits, such as `999 + 1 = 1000`.
- One or both numbers may be zero, such as `[0]` and `[0]`.
- Digits must remain in reverse order, so we cannot accidentally reverse the final answer.

A naive implementation can easily fail if it ignores carry propagation or assumes both linked lists have the same length.

## Approaches

### Brute Force Approach

A straightforward way to solve this problem is to reconstruct the actual integers represented by the linked lists, add them together, and then convert the sum back into a linked list.

We could traverse each linked list, rebuild the integer digit by digit, perform integer addition, and then create a new linked list from the resulting number.

This works because the linked list representation is deterministic, and converting between representations preserves correctness.

However, there are important drawbacks. In some languages, very large integers may exceed built-in integer limits. Even though Python supports arbitrarily large integers, relying on integer conversion misses the spirit of the problem, which is fundamentally about linked list manipulation and arithmetic simulation.

Additionally, reconstructing numbers introduces unnecessary conversion overhead.

### Optimal Approach

The key insight is that the linked lists are already arranged in the exact order needed for manual addition.

Since digits are stored in reverse order, the least significant digit comes first. This means we can simulate elementary school addition directly:

1. Add the current digits from both linked lists.
2. Include any carry from the previous digit.
3. Store the result digit (`sum % 10`).
4. Carry forward (`sum // 10`).
5. Move to the next nodes.

This works perfectly because addition naturally proceeds from least significant digit to most significant digit, and the linked lists already present digits in that order.

We process both linked lists simultaneously until both are exhausted and no carry remains.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n + m) | O(n + m) | Convert linked lists to integers, add, then rebuild list |
| Optimal | O(max(n, m)) | O(max(n, m)) | Simulate digit-by-digit addition with carry |

Here, `n` and `m` are the lengths of the two linked lists.

## Algorithm Walkthrough

### Optimal Algorithm

1. Create a **dummy node** to simplify linked list construction.

A dummy node helps avoid special handling for the head of the result list. We maintain a pointer called `current` that always points to the last node in the result.
2. Initialize a `carry` variable to `0`.

During addition, sums greater than or equal to `10` produce a carry that must be added to the next digit.
3. Traverse both linked lists simultaneously.

Continue iterating while at least one linked list still has nodes or there is a remaining carry.
4. Read values from the current nodes.

If a linked list is exhausted, treat its value as `0`. This allows us to handle linked lists of unequal lengths naturally.
5. Compute the digit sum.

Add:

- current digit from `l1`
- current digit from `l2`
- carry from the previous step
6. Create the new digit node.

The resulting digit is:

```
total % 10
```

because only the ones place belongs in the current node.
7. Update the carry.

The carry becomes:

```
total // 10
```

since anything greater than `9` moves to the next digit.
8. Advance pointers.

Move `l1`, `l2`, and `current` forward.
9. Return the result list.

Since we used a dummy node, the actual answer begins at `dummy.next`.

## Python Solution

```
# Definition for singly-linked list.
# class ListNode:
#     def __init__(self, val=0, next=None):
#         self.val = val
#         self.next = next

from typing import Optional

class Solution:
    def addTwoNumbers(
        self,
        l1: Optional[ListNode],
        l2: Optional[ListNode]
    ) -> Optional[ListNode]:

        dummy = ListNode()
        current = dummy
        carry = 0

        while l1 or l2 or carry:
            value1 = l1.val if l1 else 0
            value2 = l2.val if l2 else 0

            total = value1 + value2 + carry

            carry = total // 10
            digit = total % 10

            current.next = ListNode(digit)
            current = current.next

            if l1:
                l1 = l1.next

            if l2:
                l2 = l2.next

        return dummy.next
```

The implementation begins by creating a dummy node. This node serves as a stable starting point for the result linked list and eliminates the need for special handling of the first node.

The `carry` variable stores overflow from digit addition. Initially, there is no carry, so it starts at `0`.

Inside the loop, we retrieve values from `l1` and `l2`. If either linked list has already ended, we substitute `0`. This ensures the addition logic remains uniform even when the linked lists have different lengths.

We compute the total for the current digit by adding both values and the carry. The new digit is determined using modulo (`% 10`), while integer division (`// 10`) computes the carry for the next iteration.

A new node is appended to the result linked list, and the pointers move forward.

The loop condition is especially important:

```
while l1 or l2 or carry:
```

This guarantees that we process every digit and also handle cases where a final carry remains after both linked lists are exhausted.

Finally, we return `dummy.next`, skipping the placeholder dummy node.

## Go Solution

```
/**
 * Definition for singly-linked list.
 * type ListNode struct {
 *     Val int
 *     Next *ListNode
 * }
 */
func addTwoNumbers(l1 *ListNode, l2 *ListNode) *ListNode {
	dummy := &ListNode{}
	current := dummy
	carry := 0

	for l1 != nil || l2 != nil || carry != 0 {
		value1 := 0
		value2 := 0

		if l1 != nil {
			value1 = l1.Val
		}

		if l2 != nil {
			value2 = l2.Val
		}

		total := value1 + value2 + carry

		carry = total / 10
		digit := total % 10

		current.Next = &ListNode{
			Val: digit,
		}

		current = current.Next

		if l1 != nil {
			l1 = l1.Next
		}

		if l2 != nil {
			l2 = l2.Next
		}
	}

	return dummy.Next
}
```

The Go implementation follows exactly the same logic as the Python version. The primary difference is explicit pointer handling. Since Go uses `nil` instead of Python's `None`, pointer checks are written as `l1 != nil`.

Node allocation is also explicit using:

```
&ListNode{}
```

Unlike Python, Go does not have arbitrary precision integers by default, but integer overflow is not a concern here because every digit is between `0` and `9`, meaning the largest possible intermediate sum is:

```
9 + 9 + 1 = 19
```

which easily fits inside an integer.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(max(n, m)) | We process each node exactly once |
| Space | O(max(n, m)) | The output linked list stores the resulting digits |

The time complexity is linear because each node from both linked lists is visited once. No nested iteration occurs, and each operation inside the loop takes constant time.

The space complexity is also linear because we must construct a new linked list to store the result. Ignoring the required output space, the auxiliary space complexity is `O(1)` since we only use a few pointer variables and a carry value.

## Edge Cases

### Different Length Linked Lists

One linked list may be longer than the other.

Example:

```
[9,9,9] + [1]
```

A buggy implementation may stop as soon as the shorter list ends. Our solution handles this by treating missing digits as `0`, allowing addition to continue correctly.

### Final Carry After Traversal

A carry may remain even after both linked lists are exhausted.

Example:

```
[9] + [1]
```

This produces:

```
10
```

Without explicitly checking for remaining carry, the result would incorrectly become `[0]`. The loop condition:

```
while l1 or l2 or carry:
```

ensures the extra node is created.

### Both Inputs Are Zero

The smallest valid input is:

```
[0] + [0]
```

Some implementations accidentally return an empty list if they assume at least one non-zero digit exists. Our implementation correctly produces:

```
[0]
```

because the loop executes once and creates the appropriate node.

### Continuous Carry Propagation

Long carry chains can occur across many digits.

Example:

```
[9,9,9,9] + [1]
```

Each addition generates another carry. Our implementation correctly propagates the carry through every iteration until no carry remains.
