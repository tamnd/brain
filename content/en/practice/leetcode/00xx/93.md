## Solution

## Problem Understanding

The problem asks us to take a string of digits and determine every possible way to insert exactly three dots so that the resulting string becomes a valid IPv4 address.

A valid IP address consists of four segments separated by dots. Each segment must satisfy two important conditions. First, it must represent an integer between `0` and `255`, inclusive. Second, it cannot contain leading zeros unless the segment itself is exactly `"0"`.

For example:

- `"192.168.1.1"` is valid because each segment is between `0` and `255`, and no segment has unnecessary leading zeros.
- `"01.2.3.4"` is invalid because `"01"` has a leading zero.
- `"256.1.1.1"` is invalid because `256` exceeds the allowed maximum of `255`.

The input is a string `s` containing only digits. We are not allowed to reorder digits, remove digits, or insert anything other than dots. Our task is to return all possible valid IP addresses that can be created by partitioning the string into exactly four valid segments.

The constraints are relatively small:

- `1 <= s.length <= 20`
- The string contains digits only.

Even though the maximum length is `20`, a valid IP address can contain at most `12` digits because there are exactly four segments and each segment can have at most three digits. This observation immediately tells us that many inputs can be rejected early. For example, any string longer than `12` or shorter than `4` cannot possibly form a valid IP address.

Several edge cases are important to identify before implementation. Strings containing many zeros can easily trigger leading-zero bugs. For example, `"0000"` should produce `"0.0.0.0"` and not invalid variants like `"00.0.0"`. Inputs longer than `12` digits must return an empty result because no valid segmentation exists. Another tricky case is segments near the numeric limit, such as `"25525511135"`, where values exactly equal to `255` are valid but anything larger is not.

## Approaches

### Brute Force Approach

A straightforward solution is to try every possible placement of three dots in the string and check whether the resulting four segments form a valid IP address.

Since an IP address contains exactly four parts, we can think of choosing three split positions in the string. For every possible combination of splits, we extract the four segments and validate each one.

Validation requires checking:

1. The segment is non-empty.
2. It does not contain leading zeros unless its length is exactly one.
3. Its integer value is between `0` and `255`.

This brute-force strategy is correct because it exhaustively explores every possible way to partition the string into four pieces. If a valid IP exists, we will eventually test that split configuration.

However, this approach performs unnecessary work. It may repeatedly examine impossible segment lengths or invalid paths that could be eliminated earlier.

### Optimal Approach, Backtracking with Pruning

The key observation is that an IP address has a very rigid structure:

- Exactly four segments
- Each segment length is between `1` and `3`
- Each segment must satisfy strict validation rules

Instead of generating every split and validating afterward, we can build the address incrementally using backtracking.

At each recursive step, we decide the next segment by taking `1`, `2`, or `3` digits from the current position. Before continuing recursion, we immediately validate the segment. If it is invalid, we stop exploring that branch.

This pruning is important because it prevents wasted work. For example:

- If a segment starts with `'0'`, we know only `"0"` is allowed, so longer choices are skipped.
- If a segment exceeds `255`, there is no reason to continue deeper.

Since there are only four segments and each segment has at most three choices, the search space is very small.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n³) | O(1) excluding output | Tries all possible dot placements and validates afterward |
| Optimal | O(1) | O(1) excluding output | Backtracking explores a bounded search space with pruning |

Although the backtracking solution appears recursive, the problem structure limits recursion depth to exactly four levels, making the complexity effectively constant.

## Algorithm Walkthrough

1. **Handle impossible input lengths early**

Before starting recursion, check whether the string length is between `4` and `12`.

Why? Because:

- Minimum valid length is `4` (`"1.1.1.1"`)
- Maximum valid length is `12` (`"255.255.255.255"`)

If the length is outside this range, immediately return an empty list.
2. **Initialize result storage**

Create a result list to store valid IP addresses.

We also maintain a temporary list representing the current path of chosen segments.
3. **Start backtracking from index `0`**

Define a recursive function with two parameters:

- `start_index`, indicating where we are in the string
- `segments`, containing chosen IP parts so far
4. **Check completion condition**

If we already have `4` segments:

- If all digits have been consumed, join the segments with `"."` and add the address to results.
- Otherwise, discard the branch because a valid IP must use every digit.
5. **Try segment lengths from `1` to `3`**

At every step, attempt to take:

- `1` digit
- `2` digits
- `3` digits

This reflects the IPv4 constraint that segments cannot exceed three digits.
6. **Stop if substring goes out of bounds**

If taking the current length exceeds the string size, stop exploring further.
7. **Validate the candidate segment**

For each substring:

- Reject if it has leading zeros and length greater than `1`
- Convert to integer
- Reject if greater than `255`

This pruning prevents invalid branches from expanding.
8. **Choose the segment and recurse**

Add the valid segment to the path and recursively process the next position.
9. **Backtrack after recursion**

Remove the last segment before trying the next possibility.

This restores state so other branches can reuse the same path list.

## Python Solution

```
from typing import List

class Solution:
    def restoreIpAddresses(self, s: str) -> List[str]:
        if len(s) < 4 or len(s) > 12:
            return []

        result = []

        def backtrack(start_index: int, segments: List[str]) -> None:
            if len(segments) == 4:
                if start_index == len(s):
                    result.append(".".join(segments))
                return

            for length in range(1, 4):
                end_index = start_index + length

                if end_index > len(s):
                    break

                segment = s[start_index:end_index]

                if len(segment) > 1 and segment[0] == "0":
                    continue

                if int(segment) > 255:
                    continue

                segments.append(segment)
                backtrack(end_index, segments)
                segments.pop()

        backtrack(0, [])
        return result
```

The implementation begins with an early length check because strings shorter than `4` or longer than `12` can never form a valid IP address.

The `backtrack` function drives the search. It keeps track of the current index in the string and the list of chosen segments. Once four segments are selected, it verifies that the entire input string has been consumed. If so, the segments are joined using dots and added to the result.

The main recursive loop tries segment lengths from `1` to `3`. Each candidate substring is validated immediately. Leading zeros are rejected unless the segment is exactly `"0"`, and values larger than `255` are discarded.

When a valid segment is found, it is appended to the current path, recursion continues, and afterward the segment is removed. This backtracking pattern allows efficient exploration without creating unnecessary copies of the segment list.

## Go Solution

```
func restoreIpAddresses(s string) []string {
	if len(s) < 4 || len(s) > 12 {
		return []string{}
	}

	var result []string
	var segments []string

	var backtrack func(startIndex int)

	backtrack = func(startIndex int) {
		if len(segments) == 4 {
			if startIndex == len(s) {
				ip := segments[0] + "." +
					segments[1] + "." +
					segments[2] + "." +
					segments[3]

				result = append(result, ip)
			}
			return
		}

		for length := 1; length <= 3; length++ {
			endIndex := startIndex + length

			if endIndex > len(s) {
				break
			}

			segment := s[startIndex:endIndex]

			if len(segment) > 1 && segment[0] == '0' {
				continue
			}

			value := 0
			for _, ch := range segment {
				value = value*10 + int(ch-'0')
			}

			if value > 255 {
				continue
			}

			segments = append(segments, segment)
			backtrack(endIndex)
			segments = segments[:len(segments)-1]
		}
	}

	backtrack(0)

	return result
}
```

The Go implementation follows the same recursive structure as the Python solution. Since Go does not provide automatic string-to-integer conversion as conveniently as Python, the numeric value is computed manually while iterating through the segment characters.

The result slice starts empty and grows dynamically as valid IP addresses are found. During backtracking, the current segment is appended to `segments`, recursion is executed, and then the slice is shortened to remove the most recent choice.

Unlike Python, Go does not use `None`, so an empty slice is returned when no valid addresses exist.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(1) | At most `3⁴ = 81` recursive branches are explored |
| Space | O(1) | Recursion depth is capped at `4`, excluding output storage |

The time complexity is effectively constant because the recursion tree has a fixed maximum size. Each of the four segments can branch into at most three choices, resulting in at most `3⁴ = 81` possibilities. Since this upper bound never grows with input size in practice, the runtime is constant.

The auxiliary space complexity is also constant. The recursion stack never exceeds four levels, and the temporary segment list stores at most four strings. The output list is not counted toward auxiliary space complexity.

## Edge Cases

### Input Shorter Than Four Digits

An input such as `"123"` cannot form four IP segments because each segment must contain at least one digit. A naive implementation might still attempt recursion and waste computation. This implementation prevents unnecessary work with an early length check and immediately returns an empty list.

### Input Longer Than Twelve Digits

An input such as `"1234567890123"` contains too many digits to fit into four valid segments because each segment can contain at most three digits. Without early pruning, recursion would still explore many invalid paths. The implementation avoids this entirely by rejecting lengths greater than `12`.

### Leading Zeros

Inputs like `"010010"` are particularly tricky because `"01"` and `"001"` are invalid segments. A common bug is accidentally accepting them after integer conversion. This implementation explicitly checks whether a segment has length greater than `1` and starts with `'0'`. If so, the segment is skipped immediately.

### Segment Greater Than 255

An input like `"256256256256"` may create segments larger than the allowed range. Since IPv4 segments must be at most `255`, any substring exceeding this limit is discarded immediately. This pruning ensures invalid branches never continue recursively.

### Valid Use of Zero

The input `"0000"` demonstrates that `"0"` itself is valid even though leading zeros are not. The implementation correctly allows single-character `"0"` segments while rejecting longer forms like `"00"` or `"000"`.
