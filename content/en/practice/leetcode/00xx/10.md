## Solution

## Problem Understanding

This problem asks us to implement a simplified regular expression engine. We are given two strings:

- `s`, the input string we want to match
- `p`, the pattern string containing lowercase letters, `.` and `*`

The matching must cover the entire input string. Partial matches are not allowed.

The pattern supports two special characters:

- `.` matches any single character
- `*` matches zero or more occurrences of the character immediately before it

For example:

- `"a*"` can match `""`, `"a"`, `"aa"`, `"aaa"`, and so on
- `".*"` can match any sequence of characters because `.` matches any character and `*` allows repetition

The key difficulty is that `*` introduces branching decisions. When we encounter something like `"a*"`, we must decide whether:

- to skip it completely and match zero `'a'` characters
- or consume one character and continue matching more `'a'` characters

This creates many overlapping subproblems, which makes the problem a strong candidate for dynamic programming.

The constraints are small:

- `1 <= s.length <= 20`
- `1 <= p.length <= 20`

Even though the strings are short, a naive recursive solution can still become exponentially slow because it repeatedly recomputes the same states.

There are several important edge cases:

- Patterns like `"a*"` can match an empty string
- Consecutive `*` behavior must always apply only to the preceding character
- `".*"` can consume arbitrary portions of the input
- The pattern may finish before the string does
- The string may finish while the pattern still contains skippable sections like `"x*"`

The problem guarantees that every `*` has a valid preceding character, so we never need to handle malformed patterns like `"*a"`.

## Approaches

### Brute Force Recursive Backtracking

The most direct approach is recursive backtracking.

At every position `(i, j)`:

- `i` represents the current position in `s`
- `j` represents the current position in `p`

We try to determine whether `s[i:]` matches `p[j:]`.

If the next character in the pattern is not `*`, we simply check whether the current characters match and recursively continue.

If the next character is `*`, we have two possibilities:

1. Skip the `x*` pattern entirely, representing zero occurrences
2. Use one occurrence if the current character matches, then continue matching against the same pattern position

This approach is correct because it explores every valid interpretation of the pattern.

The problem is performance. Many states are recomputed repeatedly. For example, the same `(i, j)` pair may be reached through different recursive paths. This causes exponential time complexity.

### Dynamic Programming with Memoization

The key observation is that the problem has overlapping subproblems.

The result of matching `s[i:]` with `p[j:]` is always the same regardless of how we reached that state.

This means we can cache results for each `(i, j)` pair.

Instead of recomputing the same recursive calls repeatedly, we store the answer the first time we compute it.

This transforms the exponential brute-force recursion into a polynomial-time solution.

The state space is small:

- `i` ranges from `0` to `len(s)`
- `j` ranges from `0` to `len(p)`

So there are at most `O(m * n)` unique states.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^(m+n)) | O(m+n) | Explores all recursive branches repeatedly |
| Optimal | O(m * n) | O(m * n) | Memoized recursion avoids recomputation |

Where:

- `m = len(s)`
- `n = len(p)`

## Algorithm Walkthrough

### Optimal Dynamic Programming Algorithm

1. Define a recursive function `dfs(i, j)`.

This function returns whether `s[i:]` matches `p[j:]`.

Using indices instead of substring slicing avoids repeatedly creating new strings and keeps the implementation efficient.
2. Handle the base case.

If `j == len(p)`, the pattern is exhausted.

At this point, the match is valid only if the string is also exhausted:

```
return i == len(s)
```
3. Determine whether the current characters match.

A match exists if:

- `i < len(s)`
- and either:

- `s[i] == p[j]`
- or `p[j] == '.'`
4. Check whether the next pattern character is `*`.

We look ahead safely using:

```
j + 1 < len(p) and p[j + 1] == '*'
```
5. If the next character is `*`, branch into two possibilities.

First possibility:

- Skip the entire `x*` segment
- Move pattern pointer forward by 2

```
dfs(i, j + 2)
```

Second possibility:

- If current characters match, consume one character from `s`
- Keep pattern pointer at the same location because `*` can repeat

```
first_match and dfs(i + 1, j)
```

If either branch succeeds, the match succeeds.
6. If the next character is not `*`, perform a normal single-character match.

We require:

- current characters to match
- recursive remainder to match

```
first_match and dfs(i + 1, j + 1)
```
7. Cache every computed `(i, j)` result.

Memoization prevents repeated work and guarantees polynomial time complexity.

### Why it works

The algorithm works because every recursive state represents a precise subproblem:

- whether the suffix `s[i:]` matches the suffix `p[j:]`

For each state, the algorithm exhaustively considers all valid interpretations of `*` while preserving exact matching semantics.

Memoization guarantees that each distinct state is solved only once. Since every recursive transition corresponds exactly to the formal rules of regular expression matching, the final result is correct.

## Python Solution

```
class Solution:
    def isMatch(self, s: str, p: str) -> bool:
        memo = {}

        def dfs(i: int, j: int) -> bool:
            if (i, j) in memo:
                return memo[(i, j)]

            if j == len(p):
                return i == len(s)

            first_match = (
                i < len(s) and
                (s[i] == p[j] or p[j] == '.')
            )

            if j + 1 < len(p) and p[j + 1] == '*':
                result = (
                    dfs(i, j + 2) or
                    (first_match and dfs(i + 1, j))
                )
            else:
                result = (
                    first_match and
                    dfs(i + 1, j + 1)
                )

            memo[(i, j)] = result
            return result

        return dfs(0, 0)
```

The implementation follows the recursive definition directly.

The `memo` dictionary stores results for previously computed `(i, j)` states. This avoids exponential recomputation.

The helper function `dfs(i, j)` answers the question: does `s[i:]` match `p[j:]`?

The variable `first_match` checks whether the current characters are compatible. This includes the wildcard `.` case.

The most important section is handling `*`. When we detect a `*`, we evaluate two possibilities:

- skipping the pattern entirely
- consuming one matching character and continuing

The recursive structure naturally models the semantics of regular expressions.

## Go Solution

```
func isMatch(s string, p string) bool {
    memo := make(map[[2]int]bool)
    visited := make(map[[2]int]bool)

    var dfs func(i int, j int) bool

    dfs = func(i int, j int) bool {
        key := [2]int{i, j}

        if visited[key] {
            return memo[key]
        }

        visited[key] = true

        if j == len(p) {
            return i == len(s)
        }

        firstMatch := (
            i < len(s) &&
                (s[i] == p[j] || p[j] == '.')
        )

        var result bool

        if j+1 < len(p) && p[j+1] == '*' {
            result = dfs(i, j+2) ||
                (firstMatch && dfs(i+1, j))
        } else {
            result = firstMatch &&
                dfs(i+1, j+1)
        }

        memo[key] = result
        return result
    }

    return dfs(0, 0)
}
```

The Go implementation closely mirrors the Python version.

Since Go maps return zero values for missing keys, we use a separate `visited` map to distinguish between:

- a cached `false`
- an uncomputed state

The memoization key is represented using a fixed-size array `[2]int{i, j}`, which works cleanly as a map key in Go.

String indexing in Go accesses bytes, which is safe here because the input only contains lowercase English letters and special ASCII characters.

## Worked Examples

### Example 1

Input:

```
s = "aa"
p = "a"
```

We begin with:

| State | Action |
| --- | --- |
| dfs(0,0) | compare `'a'` with `'a'` |
| dfs(1,1) | pattern exhausted but string remains |
| return false | incomplete match |

Result:

```
false
```

The pattern only matches one character, but the string contains two characters.

### Example 2

Input:

```
s = "aa"
p = "a*"
```

State transitions:

| State | Decision |
| --- | --- |
| dfs(0,0) | detect `a*` |
| dfs(0,2) | skip `a*`, fails because string remains |
| dfs(1,0) | consume one `'a'` |
| dfs(2,0) | consume second `'a'` |
| dfs(2,2) | both exhausted |

Detailed flow:

1. At `(0,0)`, current pattern is `"a*"`
2. First branch skips `"a*"` entirely, this fails
3. Second branch consumes one `'a'`
4. Continue recursively
5. Eventually both string and pattern finish simultaneously

Result:

```
true
```

### Example 3

Input:

```
s = "ab"
p = ".*"
```

State transitions:

| State | Decision |
| --- | --- |
| dfs(0,0) | detect `.*` |
| dfs(0,2) | skip `.*`, fails |
| dfs(1,0) | consume `'a'` |
| dfs(2,0) | consume `'b'` |
| dfs(2,2) | complete match |

Because `.` matches any character and `*` allows repetition, the pattern can consume the entire string.

Result:

```
true
```

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m * n) | Each `(i, j)` state is computed once |
| Space | O(m * n) | Memoization table stores all states |

The recursive function has at most `(m + 1) * (n + 1)` unique states because both indices move within bounded ranges.

Each state performs only constant work aside from recursive calls. Memoization guarantees that repeated states are reused instead of recomputed.

The recursion stack can also reach `O(m + n)` depth in the worst case, but the memo table dominates overall space usage.

## Edge Cases

### Empty String with Star Patterns

A common bug occurs when the string becomes empty but the pattern still contains sections like `"a*"` or `"b*c*"`.

These sections can legally represent zero occurrences, so the match may still succeed.

For example:

```
s = ""
p = "a*b*"
```

The implementation handles this correctly because the recursive branch:

```
dfs(i, j + 2)
```

allows the algorithm to skip entire `x*` groups.

### Greedy-Looking Patterns

Patterns like:

```
s = "aaa"
p = "a*a"
```

can easily confuse naive implementations.

If `a*` greedily consumes all three characters, the final `'a'` in the pattern cannot match anything.

Correct solutions must explore both possibilities:

- consuming more characters
- stopping repetition early

The recursive branching structure naturally handles this by exploring both paths.

### Wildcard and Star Combination

The pattern `".*"` is especially dangerous because it can match nearly anything.

Naive recursion may repeatedly revisit the same states and explode exponentially.

For example:

```
s = "aaaaaaaaaaaa"
p = ".*.*.*.*"
```

Without memoization, the recursion tree becomes enormous.

The memo table prevents repeated computation by caching every `(i, j)` result, keeping the algorithm efficient even with heavy branching.
