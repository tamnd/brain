# Regular Expression Matching

## Problem Understanding

This problem asks us to determine whether an entire input string matches a pattern that contains two special regular expression characters, `.` and `*`.

The pattern rules are intentionally limited compared to full regular expressions:

- `.` matches exactly one arbitrary character
- `*` matches zero or more occurrences of the character immediately before it

The important detail is that the match must cover the entire string. Partial matches are not acceptable. If any characters remain unmatched in either the string or the pattern, the result should be `false`.

For example:

- `"aa"` does not match `"a"` because the pattern only consumes one character
- `"aa"` matches `"a*"` because `*` allows repeating the preceding character
- `"ab"` matches `".*"` because `.` matches any character and `*` allows repeating it indefinitely

The input consists of two strings:

- `s`, the text we are trying to match
- `p`, the pattern containing lowercase letters plus `.` and `*`

The constraints are small:

- `1 <= s.length <= 20`
- `1 <= p.length <= 20`

These constraints strongly suggest that exponential recursive exploration may still pass, but the problem is designed to teach memoization or dynamic programming. Many recursive states repeat, so caching intermediate results is the key optimization.

There are several important edge cases:

- Patterns where `*` represents zero occurrences, such as `"a*b"` matching `"b"`
- Patterns where `*` consumes many characters, such as `"a*"` matching `"aaaa"`
- Mixed wildcard behavior, such as `".*"`
- Situations where the pattern ends before the string
- Situations where the string ends but the remaining pattern can still legally match an empty string
- Consecutive recursive branching caused by `*`

The problem guarantees that every `*` always has a valid preceding character, so invalid patterns like `"*a"` never appear.

## Approaches

### Brute Force Recursive Backtracking

The most direct approach is recursive backtracking.

At every position, we try to match the current character from `s` against the current character from `p`. The complicated part comes from `*`.

Suppose we encounter `"a*"`. We have two possibilities:

- Treat `"a*"` as matching zero characters
- Consume one matching character from the string and continue using the same pattern position

This naturally creates a branching recursion tree.

For example:

```
s = "aaa"
p = "a*"
```

We repeatedly decide whether to stop consuming `a` characters or continue.

This approach is correct because it explores every valid interpretation of `*`. If any branch successfully consumes the entire string and pattern simultaneously, the answer is `true`.

However, the same subproblems appear repeatedly. The recursive state is completely determined by:

- current index in `s`
- current index in `p`

Without caching, identical states are recomputed many times, leading to exponential complexity.

### Optimal Dynamic Programming With Memoization

The key insight is that the recursive state space is small.

Since `s.length <= 20` and `p.length <= 20`, there are at most:

```
21 * 21 = 441
```

distinct `(i, j)` states.

Instead of recomputing the same recursive branches repeatedly, we cache results for each pair of indices.

The recursive definition becomes:

```
dp(i, j) = whether s[i:] matches p[j:]
```

For each state:

- If the next pattern character is `*`, we branch into:

- skipping the `x*`
- consuming one matching character
- Otherwise, we perform a normal character match

Memoization transforms the exponential recursion tree into a polynomial number of states.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^(m+n)) | O(m+n) | Explores repeated recursive states without caching |
| Optimal | O(m × n) | O(m × n) | Memoized recursion stores each state once |

Here, `m = len(s)` and `n = len(p)`.

## Algorithm Walkthrough

1. Define a recursive function `dp(i, j)`.

This function answers the question:

```
Does s[i:] match p[j:]?
```

Using indices instead of slicing strings avoids unnecessary string allocations.
2. Handle the base case.

If `j == len(p)`, the pattern is exhausted.

The match succeeds only if the string is also exhausted:

```
return i == len(s)
```
3. Determine whether the current characters match.

A match exists if:

- `i` is still inside the string
- and either:

- `s[i] == p[j]`
- or `p[j] == '.'`
4. Check whether the next pattern character is `*`.

We look ahead one position:

```
j + 1 < len(p) and p[j + 1] == '*'
```
5. If a `*` exists, branch into two possibilities.

First possibility:

Skip the entire `x*` sequence:

```
dp(i, j + 2)
```

Second possibility:

If the current characters match, consume one character from the string while staying on the same pattern position:

```
first_match and dp(i + 1, j)
```

Staying at `j` allows `*` to continue matching more characters.
6. If there is no `*`, perform a normal one-character match.

The current characters must match, and the remainder must also match:

```
first_match and dp(i + 1, j + 1)
```
7. Cache every computed state.

Since many recursive calls revisit identical `(i, j)` pairs, memoization ensures each state is solved once.

## Python Solution

```
class Solution:
    def isMatch(self, s: str, p: str) -> bool:
        memo = {}

        def dp(i: int, j: int) -> bool:
            if (i, j) in memo:
                return memo[(i, j)]

            if j == len(p):
                return i == len(s)

            first_match = (
                i < len(s) and
                (s[i] == p[j] or p[j] == '.')
            )

            if j + 1 < len(p) and p[j + 1] == '*':
                result = (
                    dp(i, j + 2) or
                    (first_match and dp(i + 1, j))
                )
            else:
                result = (
                    first_match and
                    dp(i + 1, j + 1)
                )

            memo[(i, j)] = result
            return result

        return dp(0, 0)
```

The implementation directly mirrors the recursive definition of the problem.

The `memo` dictionary stores previously computed `(i, j)` states. This prevents exponential recomputation.

The helper function `dp(i, j)` evaluates whether the suffix `s[i:]` matches the suffix `p[j:]`.

The base case checks whether the pattern has been completely consumed. If so, the string must also be fully consumed.

The `first_match` variable determines whether the current characters are compatible. This includes exact character equality and wildcard `.` matching.

The most important section handles `*`. Since `*` modifies the preceding character, we inspect `p[j + 1]`.

When `*` exists, the algorithm tries both valid interpretations:

- skip the `x*` entirely
- consume one character and continue matching the same pattern position

If no `*` exists, the algorithm advances both indices normally.

Because every `(i, j)` state is solved once, the runtime becomes polynomial.

## Go Solution

```
func isMatch(s string, p string) bool {
	memo := make(map[[2]int]bool)
	visited := make(map[[2]int]bool)

	var dp func(i, j int) bool

	dp = func(i, j int) bool {
		key := [2]int{i, j}

		if visited[key] {
			return memo[key]
		}

		visited[key] = true

		if j == len(p) {
			memo[key] = (i == len(s))
			return memo[key]
		}

		firstMatch := i < len(s) &&
			(s[i] == p[j] || p[j] == '.')

		var result bool

		if j+1 < len(p) && p[j+1] == '*' {
			result = dp(i, j+2) ||
				(firstMatch && dp(i+1, j))
		} else {
			result = firstMatch &&
				dp(i+1, j+1)
		}

		memo[key] = result
		return result
	}

	return dp(0, 0)
}
```

The Go implementation follows the same recursive structure as the Python version.

Go does not support tuple keys directly in maps, so we use a fixed-size array `[2]int` as the memoization key.

Unlike Python dictionaries, Go maps return zero values automatically. Because `false` could mean either "computed false" or "not computed yet", we maintain a separate `visited` map to distinguish those cases.

String indexing in Go returns bytes, which is safe here because the problem guarantees lowercase English letters plus `.` and `*`, all of which are single-byte ASCII characters.

No integer overflow concerns exist because the input size is extremely small.

## Worked Examples

## Example 1

```
s = "aa"
p = "a"
```

We start with:

```
dp(0, 0)
```

| Step | i | j | s[i] | p[j] | first_match | Action |
| --- | --- | --- | --- | --- | --- | --- |
| 1 | 0 | 0 | a | a | true | advance both |
| 2 | 1 | 1 | a | end | false | pattern exhausted |

At step 2:

```
j == len(p)
```

But:

```
i != len(s)
```

So the result is:

```
false
```

## Example 2

```
s = "aa"
p = "a*"
```

Initial call:

```
dp(0, 0)
```

Pattern contains `*` at `p[1]`.

### Branch 1: Skip `a*`

```
dp(0, 2)
```

Pattern ends immediately while string still contains `"aa"`.

Result:

```
false
```

### Branch 2: Consume one `a`

Current characters match.

Call:

```
dp(1, 0)
```

Again we encounter `a*`.

#### Skip branch

```
dp(1, 2)
```

String still contains one character.

Result:

```
false
```

#### Consume branch

```
dp(2, 0)
```

Now the string is exhausted.

At this point:

- skip `a*`
- call `dp(2, 2)`

Both string and pattern are exhausted.

Result:

```
true
```

Therefore the entire expression evaluates to `true`.

## Example 3

```
s = "ab"
p = ".*"
```

Initial state:

```
dp(0, 0)
```

Current pattern character is `.` and next character is `*`.

`.*` can match any sequence.

### Skip branch

```
dp(0, 2)
```

Pattern exhausted while string remains.

Result:

```
false
```

### Consume branch

`.` matches `'a'`.

Call:

```
dp(1, 0)
```

Again:

- `.` matches `'b'`
- consume another character

Call:

```
dp(2, 0)
```

Now string is exhausted.

Skip `.*`:

```
dp(2, 2)
```

Both exhausted.

Result:

```
true
```

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m × n) | Each `(i, j)` state is computed once |
| Space | O(m × n) | Memoization table stores all states |

The recursive state is fully determined by two indices:

- current position in `s`
- current position in `p`

Since there are at most `(m + 1) × (n + 1)` unique states, the algorithm computes each state once and caches the result. Every transition performs constant work outside recursive calls, leading to overall polynomial complexity.

The recursion stack depth is at most `O(m + n)`.

## Edge Cases

One important edge case occurs when `*` represents zero occurrences of a character. For example:

```
s = "b"
p = "a*b"
```

A naive implementation might incorrectly require at least one `'a'`. Our algorithm correctly handles this by always trying the branch that skips the entire `x*` pattern.

Another tricky case occurs when `*` must consume many characters:

```
s = "aaaa"
p = "a*"
```

A greedy implementation could become difficult to reason about if backtracking is not handled carefully. Our recursive formulation naturally supports repeated consumption because the recursive call stays at the same pattern index after consuming one character.

Patterns involving `.` and `*` together are also error-prone:

```
s = "abcdef"
p = ".*"
```

Since `.` matches any single character and `*` allows repetition, the pattern can match arbitrary-length strings. The algorithm correctly processes this because `first_match` accepts `.` universally, while the `*` recursion repeatedly consumes characters.

A final subtle case occurs when the string becomes empty before the pattern:

```
s = ""
p = "a*b*c*"
```

Even though characters remain in the pattern, the match should succeed because every remaining section can represent zero occurrences. The recursive skip branch repeatedly advances by two positions until the pattern is exhausted.
