# LeetCode 4 - Median of Two Sorted Arrays

## Problem Understanding

The problem gives us two individually sorted integer arrays, `nums1` and `nums2`. Our task is to compute the median of the combined dataset formed by merging both arrays together, without actually performing a full merge unless necessary.

The median is the middle value of a sorted sequence. If the total number of elements is odd, the median is the single middle element. If the total number of elements is even, the median is the average of the two middle elements.

For example:

- Combining `[1,3]` and `[2]` produces `[1,2,3]`, whose median is `2`
- Combining `[1,2]` and `[3,4]` produces `[1,2,3,4]`, whose median is `(2 + 3) / 2 = 2.5`

The important detail is the required runtime complexity: `O(log(m+n))`. This immediately tells us that a straightforward merge approach is not the intended solution. A full merge takes linear time, which violates the problem requirement for large inputs.

The arrays are already sorted, and this property is the key to building a logarithmic-time solution. The constraints also guarantee that at least one array contains elements because `m + n >= 1`.

Several edge cases are important:

- One array may be empty
- Arrays may have very different sizes
- Arrays may contain duplicate values
- Arrays may contain negative numbers
- The total number of elements may be odd or even
- The correct partition between left and right halves may occur at the beginning or end of an array

A naive implementation often fails on boundary conditions where one side of a partition contains no elements. Proper handling of these situations is essential.

## Approaches

### Brute Force Approach

The most direct solution is to merge the two sorted arrays into a single sorted array, then compute the median from the merged result.

Because both arrays are sorted, we can use the standard two-pointer merge process from merge sort. We repeatedly compare the current elements from both arrays and append the smaller one into a new array.

Once the merged array is complete:

- If the total length is odd, return the middle element
- If the total length is even, return the average of the two middle elements

This approach is correct because the merged array is fully sorted, making median extraction straightforward.

However, the runtime is `O(m+n)`, which does not satisfy the required logarithmic complexity.

### Optimal Binary Search Approach

The key insight is that we do not actually need the fully merged array. We only need enough information to determine the middle position.

Instead of merging elements, we partition the arrays into left and right halves such that:

- Every element on the left side is less than or equal to every element on the right side
- The left half contains exactly half the total elements

If we can find such a partition, then:

- For odd total length, the median is the maximum value on the left
- For even total length, the median is the average of:

- maximum value on the left
- minimum value on the right

Because the arrays are sorted, we can use binary search on the smaller array to efficiently locate the correct partition.

The binary search works because adjusting the partition position changes the boundary values predictably.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(m+n) | O(m+n) | Merge arrays completely, then compute median |
| Optimal | O(log(min(m,n))) | O(1) | Binary search on partition positions |

## Algorithm Walkthrough

### Optimal Binary Search Algorithm

1. First, ensure that `nums1` is the smaller array.

This is important because the binary search will run on this array. Searching the smaller array guarantees the logarithmic complexity remains minimal and avoids partition overflow issues.
2. Compute the total number of elements and determine how many elements should belong to the left partition.

```
total = len(nums1) + len(nums2)
half = total // 2
```

The left partition should contain exactly half the elements.
3. Perform binary search on `nums1`.

We choose a partition index `i` in `nums1`. Based on `i`, we can derive the corresponding partition index `j` in `nums2`.

```
j = half - i
```
4. Determine the boundary values around the partitions.

We define:

- `Aleft`, the largest value on the left side of `nums1`
- `Aright`, the smallest value on the right side of `nums1`
- `Bleft`, the largest value on the left side of `nums2`
- `Bright`, the smallest value on the right side of `nums2`

If a partition reaches the edge of an array, we use sentinel values:

- negative infinity for missing left values
- positive infinity for missing right values
5. Check whether the partition is valid.

The partition is correct if:

$A_{left} \leq B_{right} \text{ and } B_{left} \leq A_{right}$

This condition guarantees that every element on the left side is less than or equal to every element on the right side.
6. If the partition is valid, compute the median.

For odd total length:

```
median = min(Aright, Bright)
```

For even total length:

```
median = (max(Aleft, Bleft) + min(Aright, Bright)) / 2
```
7. If the partition is not valid, adjust the binary search.

- If `Aleft > Bright`, move left
- Otherwise, move right

Because the arrays are sorted, binary search correctly converges to the valid partition.

### Why it works

The algorithm maintains the invariant that the combined left partition always contains exactly half the total elements. Binary search adjusts the partition until every left-side element is less than or equal to every right-side element.

At that point, the middle of the conceptual merged array lies exactly between the partition boundaries, which allows direct median computation without merging.

## Python Solution

```
from typing import List
import math

class Solution:
    def findMedianSortedArrays(self, nums1: List[int], nums2: List[int]) -> float:
        # Ensure nums1 is the smaller array
        if len(nums1) > len(nums2):
            nums1, nums2 = nums2, nums1

        m, n = len(nums1), len(nums2)
        total = m + n
        half = total // 2

        left = 0
        right = m

        while True:
            partitionA = (left + right) // 2
            partitionB = half - partitionA

            Aleft = nums1[partitionA - 1] if partitionA > 0 else -math.inf
            Aright = nums1[partitionA] if partitionA < m else math.inf

            Bleft = nums2[partitionB - 1] if partitionB > 0 else -math.inf
            Bright = nums2[partitionB] if partitionB < n else math.inf

            # Correct partition found
            if Aleft <= Bright and Bleft <= Aright:
                if total % 2 == 1:
                    return float(min(Aright, Bright))

                return (
                    max(Aleft, Bleft) + min(Aright, Bright)
                ) / 2.0

            # Move partition left
            elif Aleft > Bright:
                right = partitionA - 1

            # Move partition right
            else:
                left = partitionA + 1
```

The implementation begins by ensuring that `nums1` is the smaller array. This guarantees that binary search runs on the shorter array, minimizing runtime and simplifying edge-case handling.

The binary search range spans all possible partition positions inside `nums1`.

For every candidate partition:

- `partitionA` divides `nums1`
- `partitionB` automatically divides `nums2` so that the total number of left-side elements remains constant

The code then extracts the four boundary values around the partitions. Sentinel infinity values simplify edge handling when a partition lies at the beginning or end of an array.

The central condition checks whether the partition is valid. Once it is, the median is computed directly from the neighboring boundary values.

If the partition is invalid, binary search adjusts accordingly:

- Too many large values on the left side means move left
- Otherwise move right

Because each iteration halves the search space, the algorithm runs efficiently in logarithmic time.

## Go Solution

```
package main

import "math"

func findMedianSortedArrays(nums1 []int, nums2 []int) float64 {
    // Ensure nums1 is the smaller array
    if len(nums1) > len(nums2) {
        return findMedianSortedArrays(nums2, nums1)
    }

    m := len(nums1)
    n := len(nums2)

    total := m + n
    half := total / 2

    left := 0
    right := m

    for {
        partitionA := (left + right) / 2
        partitionB := half - partitionA

        Aleft := math.Inf(-1)
        if partitionA > 0 {
            Aleft = float64(nums1[partitionA-1])
        }

        Aright := math.Inf(1)
        if partitionA < m {
            Aright = float64(nums1[partitionA])
        }

        Bleft := math.Inf(-1)
        if partitionB > 0 {
            Bleft = float64(nums2[partitionB-1])
        }

        Bright := math.Inf(1)
        if partitionB < n {
            Bright = float64(nums2[partitionB])
        }

        // Correct partition
        if Aleft <= Bright && Bleft <= Aright {
            if total%2 == 1 {
                if Aright < Bright {
                    return Aright
                }
                return Bright
            }

            leftMax := math.Max(Aleft, Bleft)
            rightMin := math.Min(Aright, Bright)

            return (leftMax + rightMin) / 2.0
        }

        // Move left
        if Aleft > Bright {
            right = partitionA - 1
        } else {
            // Move right
            left = partitionA + 1
        }
    }
}
```

The Go implementation follows the same algorithmic structure as the Python solution. One notable difference is the explicit use of `float64` conversions because Go distinguishes integer and floating-point arithmetic more strictly than Python.

Go also uses `math.Inf(1)` and `math.Inf(-1)` for sentinel values instead of Python's `math.inf`.

The recursive swap at the beginning ensures that the first array is always the smaller one, which keeps the binary search bounds valid.

## Edge Cases

### One Array Is Empty

A very common source of bugs occurs when one array contains no elements. Many implementations accidentally access invalid indices when computing partition boundaries.

This implementation handles the situation cleanly using sentinel infinity values. If a partition lies entirely outside an array, the corresponding boundary becomes positive or negative infinity, which keeps comparisons valid without special-case branching.

### Arrays With Very Different Sizes

The arrays may differ greatly in length, such as `[1]` and `[2,3,4,5,6,7,8]`.

A naive partitioning strategy can produce invalid indices in the larger array. This implementation avoids that issue by always performing binary search on the smaller array. That guarantees the derived partition in the second array remains valid.

### Even Total Length

When the combined length is even, the median is not a single element. It is the average of the two middle values.

Many incorrect implementations accidentally return only one boundary value. This solution correctly computes:

```
(max(left side) + min(right side)) / 2
```

which precisely matches the mathematical definition of the median for even-sized datasets.
