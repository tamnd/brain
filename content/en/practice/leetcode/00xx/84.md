## Solution

## Problem Understanding

In this problem, we are given an array called `heights`, where each value represents the height of a bar in a histogram. Every bar has a fixed width of `1`, and the bars are adjacent to each other.

The task is to determine the **largest rectangular area** that can be formed using one or more consecutive histogram bars.

A rectangle in the histogram must be formed by choosing a continuous segment of bars. The height of the rectangle is limited by the **shortest bar** in that segment, because a rectangle cannot extend above any bar that is too short. The width of the rectangle is simply the number of consecutive bars included.

For example, in:

```
heights = [2,1,5,6,2,3]
```

We can choose bars `[5,6]`. Since the shorter bar has height `5`, the rectangle height becomes `5`, and the width is `2`. The area is:

```
5 × 2 = 10
```

This is the maximum possible rectangle.

The input constraints are important:

```
1 <= heights.length <= 10^5
0 <= heights[i] <= 10^4
```

The histogram can contain up to **100,000 bars**, meaning inefficient algorithms will time out. In particular, anything quadratic, `O(n²)`, becomes impractical for the largest inputs.

There are several important edge cases to think about:

- **Strictly increasing heights**, such as `[1,2,3,4,5]`, where the best rectangle may not involve the tallest bar.
- **Strictly decreasing heights**, such as `[5,4,3,2,1]`, where many rectangles terminate immediately.
- **Bars with height `0`**, which completely break a rectangle and force resets.
- **All equal heights**, such as `[4,4,4,4]`, where the best rectangle spans the entire histogram.
- **Single element arrays**, where the answer is simply the height of the only bar.

The key challenge is efficiently determining, for every bar, how far it can extend left and right while remaining the limiting height.

## Approaches

### Brute Force Approach

A straightforward way to solve this problem is to consider **every possible contiguous subarray** of bars.

For every starting index, we expand to every ending index and track the minimum height within that range. Since the rectangle height is constrained by the shortest bar, we compute:

```
area = minimum height × width
```

and update the maximum area found.

For example:

```
[2,1,5,6,2,3]
```

We would evaluate:

```
[2] → 2
[2,1] → 1×2 = 2
[2,1,5] → 1×3 = 3
...
[5,6] → 5×2 = 10
...
```

This brute-force approach is correct because it checks every possible rectangle. However, it is too slow.

There are `O(n²)` possible subarrays, and even if we maintain the minimum height efficiently while expanding, the total complexity remains `O(n²)`.

Given `n = 10^5`, this is far too expensive.

### Key Insight for an Optimal Solution

The important observation is this:

For every bar, imagine that **this bar is the shortest bar in a rectangle**.

If we know:

- the first smaller bar on the **left**
- the first smaller bar on the **right**

then we know exactly how far this bar can extend while remaining the minimum height.

Suppose we have:

```
[2,1,5,6,2,3]
```

For bar `5`:

```
Left smaller index = 1 (height 1)
Right smaller index = 4 (height 2)
```

This means the rectangle using height `5` can stretch from:

```
index 2 → index 3
```

Width:

```
4 - 1 - 1 = 2
```

Area:

```
5 × 2 = 10
```

The challenge becomes efficiently finding the nearest smaller elements.

This is where a **monotonic increasing stack** becomes useful.

The stack stores indices of bars in increasing height order. Whenever we encounter a shorter bar, we know the taller bars on the stack can no longer extend further, so we calculate their maximal rectangles immediately.

This gives us an `O(n)` solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Checks every possible rectangle |
| Optimal Monotonic Stack | O(n) | O(n) | Computes maximal rectangle for each bar efficiently |

## Algorithm Walkthrough

1. Create an empty stack that will store indices of bars. The stack will maintain heights in **increasing order**.
2. Append a virtual bar of height `0` conceptually by iterating one extra position. This guarantees every remaining bar in the stack gets processed at the end.
3. Traverse the histogram from left to right.
4. For the current position, determine the current height. If we are at the extra position, treat the height as `0`.
5. While the stack is not empty **and the current height is smaller than the height at the stack top**, pop the stack.
6. The popped index represents a bar whose largest possible rectangle has now been determined, because we found a smaller bar on the right.
7. Compute the rectangle height using the popped bar.
8. Determine the width:

- If the stack is empty after popping, the rectangle extends to the beginning of the histogram.
- Otherwise, the new stack top gives the nearest smaller bar on the left.
9. Calculate:

```
area = height × width
```

and update the maximum area.
10. Push the current index onto the stack.
11. Continue until the traversal finishes.

The reason this works is that every bar is pushed and popped exactly once. The stack efficiently tracks boundaries where rectangles can expand.

## Python Solution

```
from typing import List

class Solution:
    def largestRectangleArea(self, heights: List[int]) -> int:
        stack = []
        max_area = 0
        n = len(heights)

        for i in range(n + 1):
            current_height = 0 if i == n else heights[i]

            while stack and heights[stack[-1]] > current_height:
                height = heights[stack.pop()]

                if stack:
                    width = i - stack[-1] - 1
                else:
                    width = i

                max_area = max(max_area, height * width)

            stack.append(i)

        return max_area
```

The implementation begins by initializing an empty stack and a variable `max_area` to track the best answer.

The stack stores **indices**, not heights. Storing indices allows us to compute widths later.

We iterate through `n + 1` positions instead of just `n`. The extra iteration simulates a bar of height `0`, forcing all remaining bars in the stack to be processed.

Inside the loop, we compare the current bar against the top of the stack. If the current bar is shorter, taller bars can no longer expand to the right, so we repeatedly pop them.

For every popped bar:

- Its height is known.
- The current index `i` acts as the first smaller element on the right.
- The stack top after popping acts as the first smaller element on the left.

Using these boundaries, we compute width and area.

Finally, we push the current index into the stack to maintain increasing order.

Since every index enters and leaves the stack only once, the algorithm runs in linear time.

## Go Solution

```
func largestRectangleArea(heights []int) int {
	stack := []int{}
	maxArea := 0
	n := len(heights)

	for i := 0; i <= n; i++ {
		currentHeight := 0
		if i < n {
			currentHeight = heights[i]
		}

		for len(stack) > 0 &&
			heights[stack[len(stack)-1]] > currentHeight {

			topIndex := stack[len(stack)-1]
			stack = stack[:len(stack)-1]

			height := heights[topIndex]

			width := i
			if len(stack) > 0 {
				width = i - stack[len(stack)-1] - 1
			}

			area := height * width
			if area > maxArea {
				maxArea = area
			}
		}

		stack = append(stack, i)
	}

	return maxArea
}
```

The Go implementation follows the same logic as the Python version.

Slices are used to simulate a stack. Pushing is done with `append`, and popping is handled using slicing:

```
stack = stack[:len(stack)-1]
```

Unlike Python, Go does not have a built-in `max()` for integers, so the maximum area update is written manually using an `if` condition.

Integer overflow is not a concern here because the maximum area is:

```
10^5 × 10^4 = 10^9
```

which safely fits inside Go's `int` on modern systems.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every bar is pushed and popped from the stack at most once |
| Space | O(n) | The stack may hold all indices in the worst case |

The linear time complexity comes from the fact that each index participates in stack operations only once. Although there is a nested `while` loop, the total number of pops across the entire algorithm is at most `n`, making the amortized cost linear.

The auxiliary space complexity is `O(n)` because, in the worst case of strictly increasing heights, the stack may temporarily contain every index.

## Edge Cases

### Single Bar Histogram

Consider:

```
[7]
```

A naive implementation may fail to process the final bar if it relies only on encountering smaller elements. Our implementation avoids this problem by adding a virtual height `0` at the end, forcing the last rectangle to be computed.

### Strictly Increasing Heights

Consider:

```
[1,2,3,4,5]
```

Since every bar is taller than the previous one, nothing gets popped during the main traversal. A buggy solution might miss the best rectangle entirely. The extra iteration with height `0` ensures every remaining bar gets processed correctly.

### Strictly Decreasing Heights

Consider:

```
[5,4,3,2,1]
```

Every new bar immediately causes pops. This stresses whether left boundaries are computed correctly. Our stack logic correctly recalculates widths using the nearest smaller element on the left.

### Bars Containing Zero

Consider:

```
[2,0,3,4]
```

A height of `0` acts like a divider, because rectangles cannot span across it. The algorithm naturally handles this, since encountering `0` forces all taller bars to be popped and finalized.

### All Equal Heights

Consider:

```
[4,4,4,4]
```

The best rectangle spans the entire histogram:

```
4 × 4 = 16
```

The implementation correctly preserves equal-height bars in the stack and computes the full width when the virtual `0` is encountered at the end.
