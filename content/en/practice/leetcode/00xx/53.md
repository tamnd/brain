# LeetCode 53, Maximum Subarray

## Problem Understanding

The problem gives an integer array `nums` and asks us to find the contiguous subarray that has the largest possible sum. A contiguous subarray means the elements must appear next to each other in the original array. We are not allowed to skip elements or rearrange the array.

The output is not the subarray itself, but the sum of the subarray with the maximum total value.

For example, in the array `[-2,1,-3,4,-1,2,1,-5,4]`, the best contiguous subarray is `[4,-1,2,1]`. Adding these values gives `6`, which is larger than the sum of every other contiguous subarray.

The constraints are important:

- The array length can be as large as `10^5`
- Each value can range from `-10^4` to `10^4`

These limits immediately tell us that highly inefficient solutions will not work. Since the array can contain one hundred thousand elements, an `O(n^2)` or `O(n^3)` solution may become too slow in practice. This strongly suggests that we should aim for a linear `O(n)` solution.

There are several important edge cases to think about before designing the algorithm.

If the array contains only one element, that element must be the answer because every subarray consists of that single value.

If every number is negative, the answer is not zero. The problem requires choosing a non-empty subarray, so we must return the largest individual negative number.

If the array contains many positive values in sequence, the optimal subarray may include almost the entire array.

If there are alternating positive and negative numbers, we must carefully determine when continuing a subarray helps and when restarting produces a better result.

## Approaches

### Brute Force Approach

The most direct solution is to examine every possible contiguous subarray and compute its sum.

A subarray is determined by a starting index and an ending index. For every possible starting position, we can extend the ending position one step at a time and calculate the running sum. Each computed sum is compared against the best answer found so far.

This approach is correct because it explicitly checks all possible contiguous subarrays, guaranteeing that the maximum one will eventually be found.

However, this method is inefficient for large inputs. There are `O(n^2)` possible subarrays in an array of length `n`. Even with an optimized running sum calculation, we still need quadratic time.

With `n = 10^5`, an `O(n^2)` solution would require roughly `10^10` operations in the worst case, which is far too slow.

### Optimal Approach, Kadane's Algorithm

The key insight is that when building a subarray, a negative running sum actively hurts future totals.

Suppose we are scanning the array from left to right and maintaining the sum of the current subarray. If the current running sum becomes negative, continuing to include it in future calculations can only reduce future sums.

For example, if the current running sum is `-5` and the next number is `10`, then continuing gives `5`, while starting fresh at `10` gives a better result.

This observation leads to a greedy dynamic programming strategy:

- At each position, determine the best subarray ending at that index
- Either extend the previous subarray or start a new one
- Keep track of the global maximum seen so far

This method processes the array in a single pass, giving an `O(n)` solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Checks every possible contiguous subarray |
| Optimal | O(n) | O(1) | Uses Kadane's Algorithm with a running maximum |

## Algorithm Walkthrough

1. Initialize two variables, `current_sum` and `max_sum`, using the first element of the array.

We use the first element because the problem guarantees the array is non-empty. Initializing with the first value also correctly handles arrays containing only negative numbers.
2. Iterate through the array starting from index `1`.

At each position, we decide whether it is better to continue the existing subarray or begin a brand new subarray starting at the current element.
3. Update `current_sum`.

The recurrence relation is:

$\text{current\_sum} = \max(\text{nums}[i],\ \text{current\_sum} + \text{nums}[i])$

This works because:

- `nums[i]` means we start a new subarray
- `current_sum + nums[i]` means we extend the previous subarray

We choose whichever option gives the larger sum.
4. Update `max_sum`.

After computing the best subarray ending at the current position, compare it against the overall best answer found so far.

$\text{max\_sum} = \max(\text{max\_sum},\ \text{current\_sum})$
5. Continue until the end of the array.

By the time we finish scanning the array, `max_sum` contains the largest subarray sum in the entire input.

## Python Solution

```
from typing import List

class Solution:
    def maxSubArray(self, nums: List[int]) -> int:
        current_sum = nums[0]
        max_sum = nums[0]

        for i in range(1, len(nums)):
            current_sum = max(nums[i], current_sum + nums[i])
            max_sum = max(max_sum, current_sum)

        return max_sum
```

The implementation begins by initializing both `current_sum` and `max_sum` with the first element. This avoids incorrectly returning `0` for arrays containing only negative numbers.

The loop starts from index `1` because the first element has already been used for initialization.

Inside the loop, `current_sum` represents the best possible subarray ending at the current index. The expression:

```
max(nums[i], current_sum + nums[i])
```

decides whether continuing the previous subarray is beneficial or whether starting fresh gives a larger sum.

After updating `current_sum`, the algorithm compares it with `max_sum`. If the new value is larger, we update the global best answer.

Finally, after processing all elements, the function returns `max_sum`.

## Go Solution

```
func maxSubArray(nums []int) int {
    currentSum := nums[0]
    maxSum := nums[0]

    for i := 1; i < len(nums); i++ {
        currentSum = max(nums[i], currentSum+nums[i])
        maxSum = max(maxSum, currentSum)
    }

    return maxSum
}

func max(a, b int) int {
    if a > b {
        return a
    }
    return b
}
```

The Go implementation follows the exact same logic as the Python version. Since Go does not provide a built-in integer `max` function for this use case, we define a helper function manually.

The problem guarantees that the input array is non-empty, so directly accessing `nums[0]` is safe.

Go slices are reference-like structures, so no additional copying occurs during iteration. Integer overflow is not a concern here because the maximum possible sum remains well within Go's integer range.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | The array is scanned exactly once |
| Space | O(1) | Only a few variables are used regardless of input size |

The algorithm performs constant work for each array element. Since every element is visited exactly once, the running time grows linearly with the input size.

The memory usage remains constant because the algorithm stores only two integer variables, `current_sum` and `max_sum`. No extra arrays or data structures are created.

## Edge Cases

One important edge case is an array containing only negative numbers, such as `[-5, -2, -8]`. A common mistake is initializing the maximum sum to `0`, which would incorrectly return `0`. The correct answer is `-2`, since the subarray must be non-empty. This implementation avoids the issue by initializing both tracking variables with the first array element.

Another important case is a single-element array, such as `[7]` or `[-3]`. Since there is only one possible subarray, the algorithm should immediately return that value. The implementation handles this naturally because the loop begins at index `1`, so no further processing occurs.

A third important case is when the best subarray appears in the middle of the array rather than at the beginning or end. For example, in `[-10, 4, 5, -1, 2, -20]`, the correct answer is `10` from the subarray `[4,5,-1,2]`. The algorithm correctly tracks the best running sum while scanning through the array and does not assume the optimal subarray must extend to either boundary.

Another subtle edge case occurs when the running sum becomes negative. For example, in `[3, -10, 5]`, continuing the previous subarray after reaching `-7` would hurt future totals. The algorithm correctly discards the negative running sum and starts fresh at `5`, ensuring the best future subarray is considered.
