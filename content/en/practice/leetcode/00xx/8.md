## Solution

## Problem Understanding

The problem asks us to implement a simplified version of the C/C++ `atoi` (ASCII to integer) function. We are given a string `s`, and our task is to extract a valid integer from it according to a very specific set of parsing rules.

The input string may contain leading whitespace, an optional sign (`+` or `-`), digits, and arbitrary non-digit characters. The goal is to interpret the string from left to right and construct a valid **32-bit signed integer**.

The parsing process follows a strict order. First, we ignore any leading spaces. After that, we check whether the next character is a sign. If it is `'-'`, the final number should be negative. If it is `'+'`, the number remains positive. If neither sign exists, we assume the number is positive.

Once the sign is determined, we start reading consecutive digits. The moment we encounter a non-digit character, parsing stops immediately. Importantly, if no digits are read at all, the function must return `0`.

The final result must also respect the bounds of a 32-bit signed integer:

- Minimum value: `-2^31 = -2147483648`
- Maximum value: `2^31 - 1 = 2147483647`

If the parsed number exceeds these bounds, we clamp the result to the nearest valid boundary.

The constraints tell us that the input length is at most `200`, which is very small. This means performance is not a bottleneck, and even less efficient approaches would likely pass. However, the real challenge of this problem is correctly implementing all the parsing rules and handling tricky edge cases.

Several inputs can easily break a naive implementation. Strings with only spaces, strings with no digits, multiple signs, embedded characters, or extremely large numbers can all introduce bugs. Examples like `"words and 987"` should return `0` because parsing never begins with a valid number, while `"4193 with words"` should return `4193` because parsing stops once letters appear. Inputs such as `"2147483648"` require overflow handling and must return `2147483647`.

## Approaches

### Brute Force Approach

A straightforward solution is to first preprocess the string by trimming whitespace, then use additional logic to identify a sign and extract numeric characters into a separate string. After collecting the digits, we could attempt to convert the resulting substring into an integer using built-in conversion functions.

For example, after removing leading spaces, we might repeatedly append valid digit characters into a temporary string until a non-digit appears. Then, we convert this temporary string to an integer and manually clamp the value if it exceeds the allowed range.

This approach is correct because it follows the required parsing order. However, it creates unnecessary intermediate strings and may rely heavily on language-specific conversion behavior. Some languages may also raise overflow exceptions or require additional handling for invalid inputs.

Although the input size is small, repeatedly building strings introduces unnecessary overhead.

### Optimal Approach

The key insight is that we do not actually need to build a separate substring or perform an expensive conversion afterward. Instead, we can process the string in a **single left-to-right pass**, constructing the integer incrementally.

At each step, we read one digit and update the number using:

`result = result * 10 + digit`

This mimics how humans naturally build numbers. For example:

- Read `1` → `1`
- Read `2` → `1 × 10 + 2 = 12`
- Read `3` → `12 × 10 + 3 = 123`

The most important optimization is **early overflow detection**. Before appending a digit, we can determine whether adding it would exceed the valid 32-bit range. This prevents integer overflow and allows us to immediately return the correct clamped value.

This solution is optimal because it processes each character once, uses constant extra memory, and directly follows the required parsing logic.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Builds an intermediate digit string before conversion |
| Optimal | O(n) | O(1) | Single pass parsing with inline overflow detection |

## Algorithm Walkthrough

1. **Skip leading whitespace**

Start at index `0` and move forward while the current character is a space. We do this because the problem explicitly states that leading whitespace must be ignored.
2. **Determine the sign**

After whitespace is removed, check whether the current character is `'+'` or `'-'`.

If the character is `'-'`, set `sign = -1`.

If the character is `'+'`, keep `sign = 1`.

Then move to the next position because the sign character itself is not part of the numeric value.
3. **Initialize the number**

Create a variable `result = 0` to store the integer as we build it digit by digit.
4. **Read consecutive digits**

Continue processing characters while they are digits.

For each digit:

- Convert the character into an integer value.
- Check whether appending this digit would overflow the 32-bit range.
- If safe, update the number:

`result = result * 10 + digit`
5. **Handle overflow before it happens**

Before multiplying by `10`, verify:

- For positive numbers, the limit is `2147483647`
- For negative numbers, the limit is `2147483648`

If adding the next digit exceeds the limit, immediately return:

- `2147483647` for positive overflow
- `-2147483648` for negative overflow
6. **Apply the sign**

Multiply the accumulated result by the sign and return the final integer.

## Python Solution

```
class Solution:
    def myAtoi(self, s: str) -> int:
        index = 0
        length = len(s)

        # Skip leading whitespace
        while index < length and s[index] == " ":
            index += 1

        # Determine sign
        sign = 1
        if index < length and s[index] in ["+", "-"]:
            if s[index] == "-":
                sign = -1
            index += 1

        result = 0
        INT_MAX = 2147483647
        INT_MIN = -2147483648

        # Process digits
        while index < length and s[index].isdigit():
            digit = ord(s[index]) - ord("0")

            # Overflow check
            if (
                result > INT_MAX // 10
                or (
                    result == INT_MAX // 10
                    and digit > (7 if sign == 1 else 8)
                )
            ):
                return INT_MAX if sign == 1 else INT_MIN

            result = result * 10 + digit
            index += 1

        return sign * result
```

The implementation starts by skipping leading spaces using a pointer called `index`. This ensures parsing begins at the first meaningful character.

Next, we determine the sign. The default assumption is positive (`sign = 1`), but if a minus sign appears, we switch to `-1`.

The main logic happens inside the digit-processing loop. We repeatedly check whether the current character is a digit. Instead of building a string and converting later, we directly update `result` using multiplication by `10` and digit addition.

The overflow check happens before modifying `result`. This is important because waiting until after multiplication could already exceed the valid integer range in some languages. We compare against `INT_MAX // 10`, which represents the highest safe value before adding another digit.

Finally, we multiply by the sign and return the answer.

## Go Solution

```
func myAtoi(s string) int {
	index := 0
	n := len(s)

	// Skip leading whitespace
	for index < n && s[index] == ' ' {
		index++
	}

	// Determine sign
	sign := 1
	if index < n && (s[index] == '+' || s[index] == '-') {
		if s[index] == '-' {
			sign = -1
		}
		index++
	}

	result := 0
	const INT_MAX = 2147483647
	const INT_MIN = -2147483648

	// Process digits
	for index < n && s[index] >= '0' && s[index] <= '9' {
		digit := int(s[index] - '0')

		// Overflow check
		if result > INT_MAX/10 ||
			(result == INT_MAX/10 &&
				digit > map[bool]int{true: 7, false: 8}[sign == 1]) {
			if sign == 1 {
				return INT_MAX
			}
			return INT_MIN
		}

		result = result*10 + digit
		index++
	}

	return sign * result
}
```

The Go implementation follows the exact same logic as Python but uses byte comparisons instead of methods like `.isdigit()`.

Since Go strings are byte slices, checking whether a character is numeric is done with:

```
s[index] >= '0' && s[index] <= '9'
```

Overflow handling is particularly important in Go because integers have fixed sizes depending on the architecture. We proactively detect overflow before multiplication to guarantee correctness regardless of platform.

One small improvement over Python is that Go works directly with byte arithmetic, which avoids conversion overhead.

The algorithm performs a single pass through the string. Every character is visited at most one time, making the time complexity linear with respect to the string length. Since we only maintain a few integer variables and pointers, the memory usage remains constant.

## Edge Cases

One important edge case is an input containing only whitespace, such as `"     "`. A naive implementation may accidentally attempt to access characters beyond the string bounds after skipping spaces. Our implementation safely checks index boundaries at every step, so it returns `0`.

Another tricky case is multiple sign characters, such as `"+-12"` or `"--42"`. According to the problem rules, only one optional sign is allowed immediately before digits. After reading the first sign, if the next character is not a digit, parsing stops and the result becomes `0`. Our implementation naturally handles this because the digit loop never begins.

Overflow is one of the most error-prone cases. Consider `"91283472332"`, which exceeds the 32-bit signed range. Instead of allowing the number to grow indefinitely, we detect overflow before it occurs and immediately return `2147483647`. Similarly, for negative overflow such as `"-91283472332"`, we correctly return `-2147483648`.

Another subtle edge case is input with digits followed by symbols, such as `"4193 with words"` or `"3.14159"`. Parsing must stop at the first invalid character. Our digit loop terminates immediately when a non-digit appears, preserving the correct parsed prefix.

Finally, strings that begin with non-digit letters, such as `"abc123"`, should return `0`. Since parsing never starts with a valid sign or digit, our implementation correctly produces zero without attempting partial parsing later in the string.
