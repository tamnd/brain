# Jump Game

## Problem Understanding

The problem gives us an integer array `nums`, where each element represents the maximum number of steps we can jump forward from that position. We always start at index `0`, and the goal is to determine whether it is possible to eventually reach the last index of the array.

For example, if `nums[i] = 3`, then from index `i` we may jump forward by `1`, `2`, or `3` positions. We are not required to use the full jump length, we only need to stay within the allowed maximum.

The output is a boolean value:

- Return `true` if there exists at least one sequence of jumps that reaches the last index.
- Return `false` if every possible path gets stuck before the final position.

The constraints are important:

- `1 <= nums.length <= 10^4`
- `0 <= nums[i] <= 10^5`

The array can contain up to ten thousand elements, so extremely slow algorithms, especially exponential recursion, will not finish in time. The jump values themselves can be large, but the actual array size remains moderate.

Several edge cases are important to recognize early:

- A single-element array like `[0]` should return `true`, because we already start at the last index.
- Arrays containing zeros can block progress if no earlier jump can skip over them.
- Large jump values may allow reaching the end immediately.
- Consecutive small jumps may still succeed if they collectively maintain reachability.

The key challenge is deciding whether some reachable path exists, not necessarily finding the exact path itself.

## Approaches

### Brute Force Approach

A straightforward approach is to explore every possible jump recursively.

From each index, we try every jump length from `1` up to `nums[i]`. For each jump, we recursively check whether the destination index can eventually reach the end.

This approach is correct because it exhaustively explores all valid paths. If any recursive branch reaches the final index, we know the answer is `true`. If every branch fails, then reaching the end is impossible.

However, this method becomes extremely slow because many states are recomputed repeatedly. Consider an array where each position allows several jump options. The number of recursive calls grows exponentially, making the solution impractical for arrays of length up to `10^4`.

Dynamic programming can reduce repeated work, but there is an even simpler and more efficient observation.

### Key Insight for the Optimal Solution

Instead of exploring every possible path, we only need to track the farthest index we can currently reach.

Suppose we scan the array from left to right.

At each index:

- If the current index is beyond the farthest reachable position, then we are stuck and cannot proceed further.
- Otherwise, we update the farthest reachable index using:

`farthest = max(farthest, i + nums[i])`

If at any point the farthest reachable index reaches or exceeds the last index, then the answer is immediately `true`.

This works because the exact sequence of jumps does not matter. What matters is whether each position is reachable, and how far that position extends our reach.

This leads to a greedy algorithm with linear time complexity.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) in the worst case | O(n) recursion stack | Explores all possible jump combinations |
| Optimal Greedy | O(n) | O(1) | Tracks the farthest reachable index while scanning once |

## Algorithm Walkthrough

1. Initialize a variable called `farthest` to `0`.

This variable represents the farthest index we can currently reach using the jumps seen so far.
2. Iterate through the array from left to right.

For each index `i`, first check whether `i` is reachable.
3. If `i > farthest`, return `false`.

This means the current position cannot be reached from any earlier jump. Since we cannot even stand on this index, we cannot continue exploring further positions.
4. Update the farthest reachable position.

Compute:

```
farthest = max(farthest, i + nums[i])
```

This extends our reachable range using the jump available at the current index.
5. Check whether `farthest` has reached the last index.

If `farthest >= len(nums) - 1`, return `true`.

At this point, we already know the end is reachable, so there is no need to continue scanning.
6. If the loop finishes without getting stuck, return `true`.

This means every index up to the end remained reachable.

The algorithm uses only one integer variable and processes each element exactly once, making it highly efficient.

## Python Solution

```
from typing import List

class Solution:
    def canJump(self, nums: List[int]) -> bool:
        farthest = 0
        last_index = len(nums) - 1

        for index, jump_length in enumerate(nums):
            if index > farthest:
                return False

            farthest = max(farthest, index + jump_length)

            if farthest >= last_index:
                return True

        return True
```

The implementation begins by initializing `farthest` to `0`, because initially we can only stand at the starting position.

The variable `last_index` stores the target position to avoid repeatedly computing `len(nums) - 1`.

The loop iterates through every index and corresponding jump length. Before processing the jump, the algorithm checks whether the current index is reachable. If `index > farthest`, then the algorithm has encountered a gap that cannot be crossed, so it immediately returns `False`.

If the index is reachable, the algorithm updates the farthest reachable position using the current jump capacity.

After updating, the algorithm checks whether the last index is already reachable. If so, it returns `True` immediately for efficiency.

If the loop completes, it means no unreachable gap was found, so the function returns `True`.

## Go Solution

```
func canJump(nums []int) bool {
    farthest := 0
    lastIndex := len(nums) - 1

    for index, jumpLength := range nums {
        if index > farthest {
            return false
        }

        if index+jumpLength > farthest {
            farthest = index + jumpLength
        }

        if farthest >= lastIndex {
            return true
        }
    }

    return true
}
```

The Go implementation follows the same greedy logic as the Python version.

Go uses `range` to iterate through the slice, giving both the current index and jump value. Since Go does not provide a built in `max` function for integers, the code uses a simple conditional comparison before updating `farthest`.

There are no integer overflow concerns under the given constraints because the maximum possible value of `index + jumpLength` remains safely within Go's integer range.

An empty slice is not possible because the constraints guarantee at least one element.

## Worked Examples

### Example 1

Input:

```
nums = [2,3,1,1,4]
```

Target index:

```
4
```

| Index | nums[i] | Current farthest | Updated farthest | Reachable? |
| --- | --- | --- | --- | --- |
| 0 | 2 | 0 | 2 | Yes |
| 1 | 3 | 2 | 4 | Yes |

At index `1`, the updated farthest position becomes `4`, which is the last index.

Therefore, the algorithm immediately returns `true`.

### Example 2

Input:

```
nums = [3,2,1,0,4]
```

Target index:

```
4
```

| Index | nums[i] | Current farthest | Updated farthest | Reachable? |
| --- | --- | --- | --- | --- |
| 0 | 3 | 0 | 3 | Yes |
| 1 | 2 | 3 | 3 | Yes |
| 2 | 1 | 3 | 3 | Yes |
| 3 | 0 | 3 | 3 | Yes |
| 4 | 4 | 3 | Not processed | No |

When the algorithm reaches index `4`, it checks:

```
4 > farthest
4 > 3
```

This means index `4` is unreachable.

Therefore, the algorithm returns `false`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each index is processed exactly once |
| Space | O(1) | Only a few integer variables are used |

The greedy algorithm performs a single linear scan through the array. Every operation inside the loop is constant time, so the total runtime grows proportionally with the array size.

The space complexity is constant because the algorithm does not allocate additional data structures proportional to the input size.

## Edge Cases

### Single Element Array

Consider the input:

```
[0]
```

We start at index `0`, which is already the last index. A buggy implementation might incorrectly think the zero prevents movement and return `false`.

This implementation handles the case correctly because the farthest reachable index starts at `0`, which already satisfies the goal.

### Early Blocking Zero

Consider:

```
[0, 2]
```

At index `0`, the maximum jump length is `0`, so we cannot move anywhere.

When the loop reaches index `1`, the condition:

```
1 > farthest
```

becomes true, because `farthest` is still `0`.

The algorithm correctly returns `false`.

### Large Jump Reaches the End Immediately

Consider:

```
[10, 0, 0, 0]
```

From index `0`, we can directly jump beyond the last index.

The algorithm updates:

```
farthest = max(0, 0 + 10)
```

Since `farthest >= last_index`, the function immediately returns `true` without unnecessary processing.

### Reachable Zeros

Consider:

```
[2, 0, 1]
```

A naive implementation might incorrectly fail when encountering a zero.

However, index `1` is still reachable from index `0`, and index `2` is also reachable because the farthest reachable position remains sufficient.

The greedy strategy correctly tracks overall reachability rather than focusing only on the current jump value.
