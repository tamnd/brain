## Solution

## Problem Understanding

This problem asks us to perform an inorder traversal on a binary tree and return the sequence of node values visited during that traversal.

In a binary tree, each node can have up to two children, a left child and a right child. Inorder traversal follows a very specific visiting order:

1. Visit the left subtree
2. Visit the current node
3. Visit the right subtree

The input is the root node of a binary tree. The output should be a list of integers representing the values encountered in inorder sequence.

For example, consider the tree:

```
    1
     \
      2
     /
    3
```

The traversal proceeds like this:

- Start at node 1
- Visit its left subtree, which is empty
- Visit node 1
- Move to node 2
- Visit node 2's left subtree, node 3
- Visit node 3
- Return to node 2
- Visit node 2

The result becomes:

```
[1, 3, 2]
```

The constraints tell us that the tree contains between 0 and 100 nodes. Since the tree is relatively small, almost any reasonable traversal approach will be fast enough. However, the follow up specifically asks whether we can solve the problem iteratively instead of recursively, which means understanding stack based traversal is important.

Several edge cases are important:

- The tree may be empty, meaning `root = []`
- The tree may contain only one node
- The tree may be completely skewed to the left or right
- Some nodes may have only one child
- Recursive implementations must correctly handle `None` or `nil` children

A naive implementation can fail if it does not correctly process empty subtrees or if it visits nodes in the wrong order.

## Approaches

### Brute Force Approach

The most straightforward approach is recursive depth first traversal.

The recursive idea directly mirrors the inorder definition:

1. Traverse the left subtree
2. Add the current node's value
3. Traverse the right subtree

This works because recursion naturally keeps track of where we came from. Each recursive call processes one subtree independently, and the call stack automatically handles returning to parent nodes after finishing child subtrees.

This approach is correct because every node is processed exactly once in the required inorder sequence.

Although this solution is already efficient enough for the problem constraints, the follow up asks for an iterative version. Recursive solutions also rely on the system call stack, which can become problematic for very deep trees in larger scale problems.

### Optimal Approach

The iterative solution uses an explicit stack to simulate the recursive call stack.

The key observation is that inorder traversal always moves as far left as possible before visiting nodes. While moving left, we must remember the parent nodes so we can return to them later. A stack is perfect for this because it preserves nodes in last in, first out order.

The algorithm works like this:

- Keep moving left while pushing nodes onto the stack
- When no more left children exist, pop from the stack
- Visit the popped node
- Move to its right subtree
- Repeat until both the current node and stack are empty

This approach achieves the same traversal order as recursion while giving us explicit control over memory usage.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(h) | Recursive DFS using system call stack |
| Optimal | O(n) | O(h) | Iterative traversal using explicit stack |

Here:

- `n` is the number of nodes
- `h` is the height of the tree

## Algorithm Walkthrough

1. Create an empty stack and an empty result list.

The stack will help us remember parent nodes while traversing left. The result list stores the inorder traversal sequence.
2. Initialize a pointer called `current` to the root node.

This pointer tracks the node currently being explored.
3. Continue looping while either `current` is not `None` or the stack is not empty.

We need both conditions because:

- `current` represents nodes still being explored downward
- the stack represents nodes waiting to be processed
4. Move as far left as possible.

While `current` exists:

- push `current` onto the stack
- move `current` to `current.left`

This step delays visiting nodes until their entire left subtree has been processed.
5. Pop the top node from the stack.

Once no more left children exist, the top stack node is the next inorder node.
6. Visit the node.

Add the node's value to the result list.
7. Move to the right subtree.

Set `current` to `current.right`.

After processing a node, inorder traversal must continue into its right subtree.
8. Repeat until traversal is complete.

The algorithm ends when:

- there are no remaining nodes to explore
- the stack is empty

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

class Solution:
    def inorderTraversal(self, root: Optional[TreeNode]) -> List[int]:
        result = []
        stack = []
        current = root

        while current or stack:
            while current:
                stack.append(current)
                current = current.left

            current = stack.pop()
            result.append(current.val)

            current = current.right

        return result
```

The implementation starts by creating three variables:

- `result` stores the inorder traversal output
- `stack` simulates recursive function calls
- `current` tracks the node currently being explored

The outer `while` loop continues as long as there are nodes remaining either in the current traversal path or inside the stack.

The inner `while` loop repeatedly moves left. Every visited node is pushed onto the stack because we must return to it later after finishing its left subtree.

Once the leftmost node is reached, the algorithm pops the most recent node from the stack. That node is now safe to visit because its entire left subtree has already been processed.

After appending the node value to `result`, traversal shifts to the node's right subtree.

Eventually all nodes are processed exactly once, and the final inorder traversal list is returned.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func inorderTraversal(root *TreeNode) []int {
    result := []int{}
    stack := []*TreeNode{}
    current := root

    for current != nil || len(stack) > 0 {
        for current != nil {
            stack = append(stack, current)
            current = current.Left
        }

        current = stack[len(stack)-1]
        stack = stack[:len(stack)-1]

        result = append(result, current.Val)

        current = current.Right
    }

    return result
}
```

The Go implementation follows the exact same logic as the Python version.

The main difference is stack handling. Go does not have a built in stack type, so a slice of `*TreeNode` pointers is used instead. Pushing is done with `append`, and popping removes the last element using slicing.

Go uses `nil` instead of Python's `None`, and slices automatically expand as needed.

Since node values are small integers, integer overflow is not a concern in this problem.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every node is pushed and popped exactly once |
| Space | O(h) | Stack stores at most the height of the tree |

The time complexity is linear because each node is processed exactly once. Every node enters the stack once and leaves the stack once, resulting in constant work per node.

The space complexity depends on the tree height. In the worst case of a completely skewed tree, the stack may contain all `n` nodes, producing `O(n)` space usage. In a balanced tree, the height is approximately `log n`, so stack usage becomes `O(log n)`.

## Edge Cases

### Empty Tree

An empty tree is represented by `root = []`.

This case is important because attempting to access child pointers or node values without checking for `None` or `nil` can cause runtime errors.

The implementation handles this naturally because the outer loop condition immediately fails when both `current` and `stack` are empty. The function simply returns an empty result list.

### Single Node Tree

A tree with only one node is the smallest non empty valid input.

This case verifies that the algorithm correctly processes nodes even when both left and right children are missing.

The implementation pushes the node once, visits it, and terminates correctly.

### Completely Left Skewed Tree

Example:

```
    4
   /
  3
 /
2
/
1
```

This case stresses the repeated left traversal logic. A buggy implementation might forget to revisit parent nodes after reaching the bottom.

The stack correctly preserves all parent nodes while descending left. Nodes are then popped in reverse order, producing the correct inorder traversal:

```
[1,2,3,4]
```

### Completely Right Skewed Tree

Example:

```
1
 \
  2
   \
    3
```

This case verifies that the algorithm correctly transitions into right subtrees after visiting nodes.

The implementation visits each node and then shifts to the right child, producing:

```
[1,2,3]
```

### Nodes With Missing Children

Many trees are irregular, meaning some nodes have only one child.

A naive implementation may incorrectly assume both children always exist.

The iterative algorithm safely handles missing children because it always checks whether `current` is non null before accessing its fields.
