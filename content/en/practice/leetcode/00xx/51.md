## Solution

## Problem Understanding

The problem asks us to place `n` queens on an `n x n` chessboard such that no two queens can attack each other. In chess, a queen can attack any square in the same row, same column, or either diagonal. Our goal is to generate every valid board configuration that satisfies these constraints.

The input is a single integer `n`, representing both the dimensions of the chessboard and the number of queens that must be placed. Since each solution must place exactly `n` queens on the board, every row and every column will ultimately contain exactly one queen.

The output is a list of all valid board arrangements. Each arrangement is represented as a list of strings. Each string corresponds to one row of the board. A `'Q'` character indicates the position of a queen, while `'.'` represents an empty square.

For example, when `n = 4`, there are exactly two valid configurations. Each configuration is returned as a list of four strings.

The constraints specify that `1 <= n <= 9`. This is an important clue. A naive brute-force approach that tries every possible board arrangement would be extremely expensive because the search space grows exponentially. However, since `n` is relatively small, a well-pruned backtracking solution becomes practical and efficient.

Several edge cases are important:

- When `n = 1`, the only possible board is valid, so the answer is simply `[["Q"]]`.
- When `n = 2` or `n = 3`, no valid configurations exist because it is impossible to place queens without attacks.
- Larger boards require efficient conflict detection because repeatedly scanning the entire board would become unnecessarily expensive.

## Approaches

### Brute Force Approach

A brute-force solution would generate every possible way to place queens on the board and then check whether the configuration is valid.

One way to think about this is:

- Each row can place a queen in any column.
- Therefore, there are `n^n` total possible placements.
- For every generated board, we would scan all queens and verify whether any two attack each other.

This approach is correct because it eventually examines every possible arrangement, so no valid solution can be missed.

However, it is far too slow. The number of candidate boards grows exponentially, and validating each board also requires additional work. Even with `n = 9`, the number of possibilities becomes enormous.

### Optimal Backtracking Approach

The key observation is that we do not need to generate obviously invalid boards.

Instead of placing queens arbitrarily and validating later, we can build the board row by row while maintaining validity at every step.

At any row:

- We attempt to place a queen in each column.
- Before placing, we check whether the column or diagonals are already occupied.
- If the placement is safe, we continue recursively to the next row.
- If we reach a dead end, we backtrack and try another position.

This technique is called backtracking because we explore partial solutions and abandon paths as soon as they become invalid.

Efficient conflict checking is the most important optimization. Instead of scanning the board repeatedly, we maintain three sets:

- Occupied columns
- Occupied main diagonals (`row - col`)
- Occupied anti-diagonals (`row + col`)

These allow constant-time safety checks.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n^n * n^2) | O(n^2) | Generates every board and validates afterward |
| Optimal Backtracking | O(n!) | O(n) | Builds only valid partial configurations |

## Algorithm Walkthrough

1. Initialize data structures to track occupied columns and diagonals.

We use:

- A set for columns
- A set for main diagonals identified by `row - col`
- A set for anti-diagonals identified by `row + col`

These sets allow constant-time conflict detection.
2. Create a recursive backtracking function.

The function receives the current row index. Its responsibility is to place a queen somewhere in that row.
3. Check the base case.

If the current row equals `n`, it means queens have been successfully placed in all rows. At this point, the current board configuration is valid and should be added to the result list.
4. Iterate through every column in the current row.

For each column:

- Compute the diagonal identifiers:

- `diag = row - col`
- `anti_diag = row + col`
- Check whether:

- The column is occupied
- The main diagonal is occupied
- The anti-diagonal is occupied

If any are occupied, placing a queen here would create an attack, so we skip this position.
5. Place the queen if the position is safe.

We:

- Add the column to the column set
- Add the diagonals to their sets
- Update the board cell to `'Q'`
6. Recursively process the next row.

Since the current placement is valid, we continue building the solution.
7. Backtrack after recursion returns.

Once all deeper possibilities are explored:

- Remove the queen from the board
- Remove entries from the sets

This restores the board state so another column can be tested.
8. Continue until all possibilities are explored.

The recursion systematically explores every valid arrangement while pruning invalid branches early.

## Python Solution

```
from typing import List

class Solution:
    def solveNQueens(self, n: int) -> List[List[str]]:
        results = []

        board = [["." for _ in range(n)] for _ in range(n)]

        columns = set()
        diagonals = set()
        anti_diagonals = set()

        def backtrack(row: int) -> None:
            if row == n:
                configuration = ["".join(r) for r in board]
                results.append(configuration)
                return

            for col in range(n):
                diagonal = row - col
                anti_diagonal = row + col

                if (
                    col in columns
                    or diagonal in diagonals
                    or anti_diagonal in anti_diagonals
                ):
                    continue

                board[row][col] = "Q"
                columns.add(col)
                diagonals.add(diagonal)
                anti_diagonals.add(anti_diagonal)

                backtrack(row + 1)

                board[row][col] = "."
                columns.remove(col)
                diagonals.remove(diagonal)
                anti_diagonals.remove(anti_diagonal)

        backtrack(0)

        return results
```

The implementation begins by creating an empty board filled with `'.'` characters. This board is mutable, which makes queen placement and removal efficient during recursion.

Three sets are maintained for conflict detection:

- `columns`
- `diagonals`
- `anti_diagonals`

The recursive `backtrack()` function attempts to place a queen in the given row.

When `row == n`, all queens have been placed successfully. The board is converted into a list of strings and added to the final results.

Inside the loop, each column is tested. The algorithm computes both diagonal identifiers and checks whether they already contain queens. If so, the position is invalid and skipped.

When a safe position is found:

- The queen is placed
- The tracking sets are updated
- Recursion proceeds to the next row

After recursion returns, the queen is removed and the sets are restored. This is the essence of backtracking.

## Go Solution

```
func solveNQueens(n int) [][]string {
	var results [][]string

	board := make([][]byte, n)
	for i := 0; i < n; i++ {
		board[i] = make([]byte, n)
		for j := 0; j < n; j++ {
			board[i][j] = '.'
		}
	}

	columns := make(map[int]bool)
	diagonals := make(map[int]bool)
	antiDiagonals := make(map[int]bool)

	var backtrack func(row int)

	backtrack = func(row int) {
		if row == n {
			configuration := make([]string, n)

			for i := 0; i < n; i++ {
				configuration[i] = string(board[i])
			}

			results = append(results, configuration)
			return
		}

		for col := 0; col < n; col++ {
			diagonal := row - col
			antiDiagonal := row + col

			if columns[col] || diagonals[diagonal] || antiDiagonals[antiDiagonal] {
				continue
			}

			board[row][col] = 'Q'

			columns[col] = true
			diagonals[diagonal] = true
			antiDiagonals[antiDiagonal] = true

			backtrack(row + 1)

			board[row][col] = '.'

			delete(columns, col)
			delete(diagonals, diagonal)
			delete(antiDiagonals, antiDiagonal)
		}
	}

	backtrack(0)

	return results
}
```

The Go implementation follows the same logic as the Python version, but there are some language-specific details.

The board is represented as a 2D slice of bytes because bytes are mutable and efficient for character manipulation. When storing a solution, each row is converted into a string.

Go does not have a built-in set type, so maps with boolean values are used instead.

Backtracking state restoration uses `delete()` to remove entries from the maps after recursion completes.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n!) | Each row attempts queen placement with heavy pruning |
| Space | O(n) | Recursion stack and tracking sets |

The time complexity is commonly approximated as `O(n!)` because each row places one queen and the number of valid column choices shrinks as queens are added. Although pruning dramatically reduces the search space, the algorithm still explores many permutations in the worst case.

The space complexity is `O(n)` excluding the output. The recursion depth is at most `n`, and the tracking sets store at most `n` entries each. The board itself uses `O(n^2)` space, but this is generally considered part of the working representation rather than auxiliary recursion overhead.

## Edge Cases

### Case 1: `n = 1`

This is the smallest possible board. A single queen can always be placed safely because there are no other queens to attack it. Some implementations accidentally overcomplicate this case or incorrectly assume larger board behavior. The current implementation handles it naturally through the normal backtracking process.

### Case 2: `n = 2`

No valid arrangement exists for a `2 x 2` board. Every possible queen placement leads to either a row, column, or diagonal conflict. A naive implementation that assumes every board has a solution may fail here. The backtracking algorithm correctly explores all possibilities and returns an empty list.

### Case 3: `n = 3`

Like the `2 x 2` case, no valid solution exists. This case is particularly useful for verifying that the algorithm properly backtracks instead of prematurely terminating after partial placements.

### Case 4: Diagonal Collision Detection

Diagonal handling is one of the most common sources of bugs. Two queens share:

- A main diagonal if `row - col` matches
- An anti-diagonal if `row + col` matches

The implementation uses these mathematical properties directly, ensuring accurate and constant-time diagonal conflict detection.

### Case 5: Backtracking State Restoration

A common mistake in recursive algorithms is forgetting to undo state changes after recursion returns. If a queen placement is not removed properly, future recursive branches inherit incorrect state and produce invalid results. This implementation carefully removes queens and deletes set entries during backtracking, guaranteeing correctness.
