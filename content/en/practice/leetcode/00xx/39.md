## Solution

## Problem Understanding

The problem gives us an array of distinct positive integers called `candidates` and a target integer called `target`. Our task is to find every unique combination of numbers from `candidates` whose sum equals `target`.

A key detail is that each number can be used an unlimited number of times. This means if `2` exists in the array, we may use it once, twice, or many times in the same combination as long as the total sum does not exceed the target.

The output should contain all valid combinations, where each combination is represented as a list of integers. The order of combinations in the final answer does not matter, and the order of elements inside a combination also does not matter for uniqueness purposes. For example, `[2,2,3]` and `[2,3,2]` represent the same combination and should only appear once.

The constraints tell us several important things:

- The number of candidates is at most `30`
- Each candidate value is between `2` and `40`
- The target is at most `40`
- All candidate values are distinct

Since the target is relatively small, exhaustive search with pruning is feasible. However, a naive brute-force approach can still generate many duplicate combinations or explore unnecessary paths.

The fact that candidates are distinct simplifies the duplicate handling logic. We do not need to worry about repeated values in the input array itself.

Several edge cases are important:

- No combination exists, such as `candidates = [2]` and `target = 1`
- A candidate itself equals the target, such as `[7]` for target `7`
- Multiple repeated uses of the same value are required, such as `[2,2,2,2]`
- Small targets where many recursive paths terminate quickly
- Large branching possibilities where pruning becomes important

This problem is a classic backtracking problem because we need to explore combinations incrementally and abandon paths once they can no longer lead to a valid solution.

## Approaches

### Brute Force Approach

A brute-force approach would attempt to generate every possible sequence of candidate selections and then check whether the sum equals the target.

For each step, we could try adding every candidate again and again until the sum exceeds the target. After generating all possibilities, we would filter out invalid combinations and duplicates.

This approach is correct because it eventually explores every possible way to construct sums using the candidates. However, it is highly inefficient because:

- It explores many redundant paths
- It generates duplicate permutations such as `[2,3,2]` and `[3,2,2]`
- The number of recursive states grows exponentially

Without careful pruning and ordering constraints, the search space becomes unnecessarily large.

### Optimal Approach, Backtracking with Controlled Exploration

The key insight is that combinations do not care about order. Therefore, once we choose a candidate at index `i`, we should only continue exploring candidates starting from index `i` or later.

This prevents duplicate permutations automatically.

For example:

- If we already chose `2`, we may choose `2` again
- We may also choose `3`
- But after choosing `3`, we should not go back and choose `2`

This creates combinations in non-decreasing order and guarantees uniqueness.

We use backtracking because:

- We build combinations incrementally
- We stop exploring when the remaining target becomes negative
- We record a solution when the remaining target becomes zero
- After exploring one path, we undo the last choice and try another possibility

This approach efficiently prunes invalid paths and avoids duplicate work.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential, approximately O(2^target) or worse | O(target) | Explores all sequences, generates duplicates |
| Optimal | O(N^(target/minCandidate)) in worst case | O(target) | Backtracking with pruning and duplicate avoidance |

## Algorithm Walkthrough

1. Start with an empty combination and the full target value.
2. Define a recursive backtracking function that takes:

- The current starting index
- The remaining target
- The current combination being built
3. If the remaining target becomes exactly `0`, we have found a valid combination. Copy the current combination into the result list.
4. If the remaining target becomes negative, this path cannot produce a valid solution, so we stop exploring it.
5. Iterate through candidates starting from the current index.
6. Choose the current candidate by appending it to the current combination.
7. Recursively continue searching with:

- The same index, because the same number can be reused
- The reduced target value
8. After recursion finishes, remove the last number from the current combination. This is the "backtracking" step that restores the previous state before exploring another choice.
9. Continue exploring all remaining candidates.

The reason we pass the current index into recursion is to prevent duplicate permutations. By never moving backward in the candidate list, combinations remain ordered and unique.

## Python Solution

```
from typing import List

class Solution:
    def combinationSum(self, candidates: List[int], target: int) -> List[List[int]]:
        result = []

        def backtrack(start_index: int, remaining: int, current_combination: List[int]) -> None:
            if remaining == 0:
                result.append(current_combination[:])
                return

            if remaining < 0:
                return

            for index in range(start_index, len(candidates)):
                candidate = candidates[index]

                current_combination.append(candidate)

                backtrack(
                    index,
                    remaining - candidate,
                    current_combination
                )

                current_combination.pop()

        backtrack(0, target, [])

        return result
```

The implementation begins by creating a `result` list that will store all valid combinations.

The nested `backtrack` function performs the recursive search. It receives three parameters:

- `start_index`, which controls where iteration begins
- `remaining`, which tracks how much of the target is still needed
- `current_combination`, which stores the current path being explored

The first base case checks whether `remaining == 0`. If true, the current combination forms a valid solution, so we append a copy of it to the result list.

The second base case checks whether `remaining < 0`. In that situation, the current path has exceeded the target and should be abandoned immediately.

The `for` loop iterates from `start_index` onward. This guarantees combinations are generated in non-decreasing order and prevents duplicates.

When a candidate is chosen, it is appended to `current_combination`. The recursive call then continues using the same index because repeated use of the same number is allowed.

After recursion completes, `pop()` removes the most recently added element so the algorithm can explore alternative choices cleanly.

Finally, the algorithm starts recursion with index `0`, the original target, and an empty combination.

## Go Solution

```
func combinationSum(candidates []int, target int) [][]int {
    result := [][]int{}
    currentCombination := []int{}

    var backtrack func(startIndex int, remaining int)

    backtrack = func(startIndex int, remaining int) {
        if remaining == 0 {
            combinationCopy := make([]int, len(currentCombination))
            copy(combinationCopy, currentCombination)

            result = append(result, combinationCopy)
            return
        }

        if remaining < 0 {
            return
        }

        for index := startIndex; index < len(candidates); index++ {
            candidate := candidates[index]

            currentCombination = append(currentCombination, candidate)

            backtrack(index, remaining-candidate)

            currentCombination = currentCombination[:len(currentCombination)-1]
        }
    }

    backtrack(0, target)

    return result
}
```

The Go implementation follows the same recursive structure as the Python version.

One important difference is that slices in Go reference shared underlying memory. Therefore, when we add a valid combination to the result, we must create a separate copy using `make()` and `copy()`. Without this step, future modifications to the slice would affect previously stored results.

Another difference is the backtracking removal step. In Python we use `pop()`, while in Go we reslice using:

```
currentCombination = currentCombination[:len(currentCombination)-1]
```

Go does not distinguish between `nil` and empty slices in a way that affects this problem. Returning an empty `[][]int{}` is perfectly valid.

Integer overflow is not a concern because the constraints are very small.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(N^(target/minCandidate)) | Recursive branching explores many combinations |
| Space | O(target) | Maximum recursion depth occurs when repeatedly choosing the smallest candidate |

The exact runtime depends heavily on the input values and the number of valid combinations. In the worst case, each recursive level may branch into many possibilities.

The recursion depth is bounded by `target / min(candidates)` because each recursive step reduces the remaining target by at least the smallest candidate value.

The result storage itself is not included in auxiliary space complexity.

## Edge Cases

One important edge case occurs when no valid combination exists. For example, `candidates = [2]` and `target = 1`. A naive implementation might continue recursion indefinitely or accidentally add invalid combinations. This implementation avoids that issue because recursion stops immediately when the remaining target becomes negative.

Another important case occurs when a candidate directly equals the target. For example, `candidates = [2,3,7]` and `target = 7`. Some implementations accidentally skip single-element solutions if they only check combinations after multiple recursive calls. This solution correctly handles the case because the base condition checks `remaining == 0` after every selection.

A third important edge case involves repeated usage of the same number. For example, `candidates = [2,3,5]` and `target = 8` requires combinations like `[2,2,2,2]`. If recursion always advanced to the next index after selecting a number, repeated usage would become impossible. This implementation correctly allows reuse by recursively calling `backtrack(index, ...)` instead of `backtrack(index + 1, ...)`.

Another subtle edge case involves duplicate permutations. Without controlling the starting index, the algorithm could generate both `[2,3,2]` and `[3,2,2]`. By restricting recursion to continue only from the current index onward, the implementation guarantees unique combinations automatically.
