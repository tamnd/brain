## Solution

## Problem Understanding

The problem gives us the head of a singly linked list and an integer `n`. We must remove the node that is positioned `n` places from the end of the list, then return the modified list.

A singly linked list consists of nodes where each node stores a value and a pointer to the next node. Because the list is singly linked, we can only move forward through the list. This limitation is important because removing a node requires access to the node immediately before it.

For example, consider the list:

```
1 -> 2 -> 3 -> 4 -> 5
```

If `n = 2`, the second node from the end is `4`, so after removal the result becomes:

```
1 -> 2 -> 3 -> 5
```

The constraints tell us several important things:

- The list size is between 1 and 30 nodes
- `n` is always valid, meaning there will always be an nth node from the end
- The input size is small, so even less efficient solutions would technically work within limits
- However, the follow-up explicitly asks for a one-pass solution, which suggests the intended optimal approach uses two pointers

The most important edge cases involve situations where the removed node is difficult to handle with naive pointer manipulation.

If the head itself must be removed, such as `[1]` with `n = 1`, the resulting list becomes empty. Many incorrect implementations fail here because they assume a previous node always exists.

Another tricky case occurs when removing the first node in a longer list, such as `[1,2,3]` with `n = 3`. Since the node to remove is the head, special handling is required unless we use a dummy node.

Very short lists also expose pointer mistakes. For example, `[1,2]` with `n = 1` removes the tail node, while `[1,2]` with `n = 2` removes the head.

## Approaches

### Brute Force Approach

A straightforward solution is to make two passes through the linked list.

In the first pass, we compute the total length of the list. Once we know the length, we can determine the position of the node to remove from the beginning of the list.

If the list length is `L`, then the node to remove is at position:

```
L - n
```

from the start, using zero-based indexing.

We then make a second pass to reach the node immediately before the target node and adjust pointers to skip the target.

This approach is correct because the relationship between positions from the start and positions from the end is deterministic once the total length is known.

The main drawback is that the list must be traversed twice. Although the constraints are small enough that this is acceptable, the follow-up asks whether the problem can be solved in one pass.

### Optimal Two-Pointer Approach

The key insight is that we do not actually need the full length of the list.

Instead, we can maintain a fixed distance of `n` nodes between two pointers. When the leading pointer reaches the end of the list, the trailing pointer will automatically be positioned immediately before the node that must be removed.

This works because the distance between the two pointers remains constant throughout traversal.

To simplify edge cases, especially when removing the head node, we introduce a dummy node that points to the head. This ensures there is always a node before the one being removed.

The algorithm proceeds as follows:

- Move the `fast` pointer `n` steps ahead
- Move both `fast` and `slow` pointers together until `fast` reaches the last node
- At that moment, `slow.next` is the node to remove
- Skip that node by changing pointers

This achieves the follow-up requirement because the list is traversed only once.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(L) | O(1) | Two passes, first computes length, second removes node |
| Optimal | O(L) | O(1) | Single-pass two-pointer technique with dummy node |

## Algorithm Walkthrough

1. Create a dummy node and point it to the head of the linked list.

The dummy node is important because it eliminates special handling when the head node must be removed. After introducing the dummy node, every removable node always has a previous node.
2. Initialize two pointers, `fast` and `slow`, both starting at the dummy node.

These two pointers will maintain a gap of `n` nodes between them.
3. Move the `fast` pointer forward by `n` steps.

After this step, the distance between `fast` and `slow` is exactly `n` nodes.
4. Move both pointers forward together until `fast.next` becomes `None`.

This means `fast` is currently at the last node in the list.

Since the gap between the pointers has remained constant, `slow` is now positioned immediately before the node that must be removed.
5. Remove the target node by updating:

```
slow.next = slow.next.next
```

This skips the target node and reconnects the linked list.

1. Return `dummy.next`.

Since the head may have been removed, the real head of the updated list is whatever comes after the dummy node.

## Python Solution

```
# Definition for singly-linked list.
# class ListNode:
#     def __init__(self, val=0, next=None):
#         self.val = val
#         self.next = next

from typing import Optional

class Solution:
    def removeNthFromEnd(
        self,
        head: Optional[ListNode],
        n: int
    ) -> Optional[ListNode]:

        dummy = ListNode(0, head)

        fast = dummy
        slow = dummy

        # Move fast pointer n steps ahead
        for _ in range(n):
            fast = fast.next

        # Move both pointers until fast reaches the end
        while fast.next:
            fast = fast.next
            slow = slow.next

        # Remove the target node
        slow.next = slow.next.next

        return dummy.next
```

The implementation begins by creating a dummy node that points to the original head. This simplifies removal logic because every removable node now has a predecessor.

Both `fast` and `slow` pointers start at the dummy node. The `fast` pointer is advanced `n` steps first, establishing the required distance between the pointers.

The loop that moves both pointers together preserves this distance. When `fast` reaches the final node, `slow` is guaranteed to be immediately before the node that must be removed.

The removal itself is performed with a single pointer update:

```
slow.next = slow.next.next
```

This bypasses the target node entirely.

Finally, `dummy.next` is returned because the original head may no longer exist.

## Go Solution

```
/**
 * Definition for singly-linked list.
 * type ListNode struct {
 *     Val int
 *     Next *ListNode
 * }
 */
func removeNthFromEnd(head *ListNode, n int) *ListNode {

    dummy := &ListNode{
        Val:  0,
        Next: head,
    }

    fast := dummy
    slow := dummy

    // Move fast pointer n steps ahead
    for i := 0; i < n; i++ {
        fast = fast.Next
    }

    // Move both pointers together
    for fast.Next != nil {
        fast = fast.Next
        slow = slow.Next
    }

    // Remove target node
    slow.Next = slow.Next.Next

    return dummy.Next
}
```

The Go implementation follows the exact same logic as the Python solution.

The primary language-specific difference is the use of explicit pointers with `*ListNode`. Go uses `nil` instead of Python's `None`.

The dummy node is created using a struct literal, and pointer traversal uses `.Next` instead of `.next`.

Because the constraints are small and integer values are limited, there are no concerns about integer overflow.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(L) | Each node is visited at most once |
| Space | O(1) | Only a few pointers are used |

The algorithm performs a single traversal of the linked list. Even though there are two pointers, they move through the same list rather than creating nested iterations, so the total work remains linear in the number of nodes.

The space complexity is constant because no extra data structures proportional to input size are allocated. Only the dummy node and a few pointer variables are used.

## Edge Cases

One important edge case occurs when the head node must be removed. For example:

```
[1,2,3], n = 3
```

A naive implementation may fail because there is no previous node before the head. The dummy node solves this cleanly by providing an artificial predecessor to the head.

Another tricky case is a single-node list:

```
[1], n = 1
```

After removal, the result should be an empty list. Incorrect implementations sometimes attempt to access invalid pointers after deletion. In this solution, the dummy node safely updates its `next` pointer to `None`.

Removing the last node is also a common source of bugs:

```
[1,2], n = 1
```

The algorithm correctly positions `slow` before the tail node, allowing the tail to be skipped cleanly by setting:

```
slow.next = slow.next.next
```

which becomes:

```
1.next = None
```

Another subtle edge case involves very short lists where pointer movement can easily go out of bounds. Because the problem guarantees `1 <= n <= sz`, the algorithm can safely move the `fast` pointer exactly `n` steps without needing additional validation checks.
