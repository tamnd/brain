## Solution

## Problem Understanding

This problem asks us to insert a new interval into an already sorted list of non-overlapping intervals while preserving two important properties:

1. The intervals must remain sorted by starting value.
2. No intervals in the final result may overlap.

If inserting the new interval creates overlaps with existing intervals, those overlapping intervals must be merged into a single interval.

The input consists of:

- `intervals`, a sorted list of intervals where each interval is represented as `[start, end]`
- `newInterval`, another interval `[start, end]` that must be inserted into the list

Each interval represents a continuous range of numbers. Since the existing intervals are guaranteed to be non-overlapping and sorted, this structure gives us an important advantage, we can process the intervals from left to right without needing to repeatedly compare every pair.

For example:

```
intervals = [[1,3],[6,9]]
newInterval = [2,5]
```

The new interval `[2,5]` overlaps with `[1,3]`, so they merge into `[1,5]`. The interval `[6,9]` does not overlap, so it stays unchanged.

The expected output is the updated interval list after insertion and merging.

The constraints tell us several important things:

- `intervals.length <= 10^4`, meaning the solution should ideally be linear or near-linear
- Intervals are already sorted by start time
- Intervals do not overlap initially
- Start and end values are bounded by `10^5`, so integer overflow is not a concern

These guarantees are extremely useful because they eliminate the need for sorting after insertion and allow us to exploit ordering for an efficient solution.

Several edge cases can easily cause bugs in a naive implementation:

- The new interval may come entirely before all intervals.
- The new interval may come entirely after all intervals.
- The new interval may overlap multiple consecutive intervals.
- The input list may be empty.
- The new interval may fully contain existing intervals.
- The new interval may not overlap anything at all.

Because the intervals are sorted and non-overlapping, any overlap will happen in one continuous segment, which becomes the key observation for an optimal solution.

## Approaches

### Brute Force Approach

A straightforward approach is to insert `newInterval` into the list, sort the entire array again, and then perform a full interval merge process.

The process works as follows:

1. Append `newInterval` to `intervals`.
2. Sort all intervals by starting value.
3. Traverse the sorted intervals and merge overlapping intervals.

This approach is correct because sorting guarantees intervals appear in order, and the standard merge intervals algorithm correctly combines overlapping ranges.

However, this solution performs unnecessary work. The original array is already sorted, and only one interval is being added. Re-sorting the entire collection wastes computation.

Sorting dominates the runtime, giving a time complexity of `O(n log n)`.

### Optimal Approach

The key insight is that the input is already sorted and non-overlapping.

Since intervals are ordered, every interval falls into exactly one of three categories relative to `newInterval`:

1. Intervals completely before `newInterval`, meaning they cannot overlap.
2. Intervals that overlap with `newInterval`, which must be merged.
3. Intervals completely after `newInterval`, meaning they can be appended as-is.

Because overlapping intervals must appear consecutively in a sorted non-overlapping list, we can process everything in one linear scan.

We first add all intervals that end before `newInterval` starts. Then we merge all overlapping intervals by expanding the boundaries of `newInterval`. Finally, we append the remaining intervals.

This avoids sorting entirely and achieves linear time complexity.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) | O(n) | Insert, sort again, then merge |
| Optimal | O(n) | O(n) | Single linear scan using sorted property |

## Algorithm Walkthrough

### Optimal Algorithm

1. Create an empty result list to store the final intervals.

We build a new list because the problem explicitly states that modifying the original array in-place is unnecessary.
2. Traverse all intervals that come completely before `newInterval`.

If an interval ends before `newInterval` starts, there is no overlap:

```
interval[1] < newInterval[0]
```

These intervals can safely be added directly to the result because they will never merge with `newInterval`.
3. Merge all overlapping intervals.

While intervals overlap with `newInterval`, we expand the boundaries of `newInterval`:

- Update the start to the smaller value.
- Update the end to the larger value.

The overlap condition is:

```
interval.start <= newInterval.end
```

Since intervals are sorted, once overlap begins, all overlapping intervals appear consecutively.
4. Add the merged interval to the result.

After merging finishes, `newInterval` now represents the fully merged interval and can safely be appended.
5. Append all remaining intervals.

Any remaining intervals start after `newInterval` ends, meaning no overlap exists. They can be added directly.
6. Return the result list.

At this point:

- Order is preserved
- No intervals overlap
- `newInterval` has been inserted correctly

## Python Solution

```
from typing import List

class Solution:
    def insert(
        self,
        intervals: List[List[int]],
        newInterval: List[int]
    ) -> List[List[int]:
        result = []
        index = 0
        total_intervals = len(intervals)

        # Add intervals completely before newInterval
        while (
            index < total_intervals
            and intervals[index][1] < newInterval[0]
        ):
            result.append(intervals[index])
            index += 1

        # Merge overlapping intervals
        while (
            index < total_intervals
            and intervals[index][0] <= newInterval[1]
        ):
            newInterval[0] = min(
                newInterval[0],
                intervals[index][0]
            )
            newInterval[1] = max(
                newInterval[1],
                intervals[index][1]
            )
            index += 1

        # Add merged interval
        result.append(newInterval)

        # Add remaining intervals
        while index < total_intervals:
            result.append(intervals[index])
            index += 1

        return result
```

The implementation follows the exact structure of the algorithm.

The first `while` loop copies intervals that definitely cannot overlap because they end before the new interval starts.

The second `while` loop handles merging. Instead of creating a separate merged interval object, the implementation updates `newInterval` directly by expanding its boundaries whenever overlap is found. This keeps the code concise and avoids extra memory usage.

Once all overlapping intervals are consumed, the merged interval is appended to the result.

Finally, the remaining intervals are added without modification because they are guaranteed to come after the merged interval.

## Go Solution

```
func insert(intervals [][]int, newInterval []int) [][]int {
	result := [][]int{}
	index := 0
	totalIntervals := len(intervals)

	// Add intervals completely before newInterval
	for index < totalIntervals &&
		intervals[index][1] < newInterval[0] {

		result = append(result, intervals[index])
		index++
	}

	// Merge overlapping intervals
	for index < totalIntervals &&
		intervals[index][0] <= newInterval[1] {

		if intervals[index][0] < newInterval[0] {
			newInterval[0] = intervals[index][0]
		}

		if intervals[index][1] > newInterval[1] {
			newInterval[1] = intervals[index][1]
		}

		index++
	}

	// Add merged interval
	result = append(result, newInterval)

	// Add remaining intervals
	for index < totalIntervals {
		result = append(result, intervals[index])
		index++
	}

	return result
}
```

The Go implementation closely mirrors the Python version.

Since Go does not provide built-in `min()` and `max()` functions for integers, simple conditional comparisons are used to update interval boundaries.

Slices are used instead of dynamic lists. Appending to `result` works efficiently because Go slices automatically resize when needed.

There are no integer overflow concerns because interval values are bounded by `10^5`, which is far below Go's integer limits.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each interval is processed at most once |
| Space | O(n) | A result array is constructed |

The algorithm performs a single pass through the interval list. Every interval is visited at most once, either during the "before", "merge", or "after" phase. Since there are `n` intervals, the runtime is linear.

The extra space complexity is `O(n)` because a new result array is created to store the output. In the worst case, all intervals remain separate and must be copied into the result.

## Edge Cases

### Empty Input Array

If `intervals` is empty, a naive implementation might fail due to indexing errors.

Example:

```
intervals = []
newInterval = [5,7]
```

The implementation handles this naturally because both loops are skipped and `newInterval` is appended directly to the result.

Output:

```
[[5,7]]
```

### New Interval Before All Existing Intervals

If the new interval comes entirely before every existing interval, it should appear at the beginning.

Example:

```
intervals = [[5,7],[8,10]]
newInterval = [1,3]
```

The first loop adds nothing because no interval ends before `1`. No merging occurs either. The algorithm appends `newInterval` first and then adds all remaining intervals.

Output:

```
[[1,3],[5,7],[8,10]]
```

### New Interval After All Existing Intervals

If the new interval belongs at the end, it should simply be appended.

Example:

```
intervals = [[1,2],[3,5]]
newInterval = [7,9]
```

The first loop copies all intervals because they all end before `7`. No overlap occurs, and the new interval is appended last.

Output:

```
[[1,2],[3,5],[7,9]]
```

### New Interval Fully Covers Existing Intervals

Sometimes the new interval absorbs several existing intervals entirely.

Example:

```
intervals = [[3,4],[5,6],[7,8]]
newInterval = [1,10]
```

Every interval overlaps with `newInterval`, so the merge loop continuously expands boundaries until only one interval remains.

Output:

```
[[1,10]]
```

### No Overlap Anywhere

The new interval may fit cleanly between two intervals.

Example:

```
intervals = [[1,2],[6,8]]
newInterval = [3,5]
```

The algorithm copies intervals before, inserts the new interval, then appends remaining intervals without merging.

Output:

```
[[1,2],[3,5],[6,8]]
```
