# Minimum Path Sum

## Problem Understanding

The problem gives us a two dimensional grid of non negative integers. We start at the top left cell, located at position `(0, 0)`, and we want to reach the bottom right cell, located at `(m - 1, n - 1)`.

Every cell contains a cost, and the total cost of a path is the sum of all values along that path, including both the starting and ending cells.

The important restriction is movement. From any position, we may only move:

- Right
- Down

Moving left or up is not allowed.

The goal is to compute the minimum possible sum among all valid paths from the top left corner to the bottom right corner.

For example, in this grid:

```
1 3 1
1 5 1
4 2 1
```

one valid path is:

```
1 → 3 → 1 → 1 → 1
```

The total is:

```
1 + 3 + 1 + 1 + 1 = 7
```

and this is the minimum possible answer.

The constraints are important:

- `1 <= m, n <= 200`
- `0 <= grid[i][j] <= 200`

The grid can contain up to:

```
200 × 200 = 40,000 cells
```

This size is large enough that an exponential solution would be far too slow. We need an algorithm that runs efficiently, ideally in polynomial time.

The problem also guarantees:

- The grid is never empty
- All values are non negative
- Every row has the same number of columns

Several edge cases are important to consider:

- A grid with only one cell
- A single row grid
- A single column grid
- Large grids where recursive brute force would cause exponential explosion
- Cells containing zero, which can affect path choices

Because movement is restricted to only right and down, the structure of the problem strongly suggests dynamic programming.

## Approaches

### Brute Force Approach

The brute force solution explores every possible path from the top left corner to the bottom right corner.

At each position, there are at most two choices:

- Move right
- Move down

A recursive solution would try both possibilities and return the smaller total.

For example:

```
minPath(i, j) =
    grid[i][j] +
    min(
        minPath(i + 1, j),
        minPath(i, j + 1)
    )
```

This approach is correct because it exhaustively checks all valid paths and selects the minimum sum.

However, it is extremely inefficient.

Many subproblems are recalculated repeatedly. For example, the minimum path from `(2, 2)` to the destination may be computed dozens or hundreds of times from different recursive branches.

The number of paths grows exponentially, making brute force impractical for a `200 × 200` grid.

### Dynamic Programming Insight

The key observation is that the minimum path sum to reach a cell depends only on:

- The minimum path sum to the cell above
- The minimum path sum to the cell to the left

Since movement is restricted to right and down, these are the only two ways to enter a cell.

If we define:

```
dp[i][j]
```

as the minimum path sum needed to reach cell `(i, j)`, then:

```
dp[i][j] =
    grid[i][j] +
    min(dp[i - 1][j], dp[i][j - 1])
```

This works because any optimal path to `(i, j)` must come from the better of those two previous positions.

Dynamic programming avoids repeated computation by storing intermediate results.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^(m+n)) | O(m+n) | Recursively explores all possible paths |
| Optimal Dynamic Programming | O(m × n) | O(m × n) | Stores minimum cost for each cell |

## Algorithm Walkthrough

### Step 1: Create a DP Table

We create a two dimensional array called `dp` with the same dimensions as the grid.

Each cell in `dp[i][j]` represents the minimum path sum required to reach that position.

### Step 2: Initialize the Starting Cell

The top left cell is the starting point.

```
dp[0][0] = grid[0][0]
```

There is only one way to stand on the starting cell, so its minimum cost is simply its own value.

### Step 3: Fill the First Row

In the first row, we can only move right.

That means every cell depends only on the cell immediately to its left.

```
dp[0][j] = dp[0][j - 1] + grid[0][j]
```

### Step 4: Fill the First Column

In the first column, we can only move down.

That means every cell depends only on the cell directly above it.

```
dp[i][0] = dp[i - 1][0] + grid[i][0]
```

### Step 5: Fill the Remaining Cells

For every other cell, we choose the cheaper path between:

- Coming from above
- Coming from the left

```
dp[i][j] =
    grid[i][j] +
    min(dp[i - 1][j], dp[i][j - 1])
```

This guarantees that every cell stores the minimum possible cost to reach it.

### Step 6: Return the Final Answer

The bottom right corner contains the minimum total path sum.

```
return dp[m - 1][n - 1]
```

## Python Solution

```
from typing import List

class Solution:
    def minPathSum(self, grid: List[List[int]]) -> int:
        rows = len(grid)
        cols = len(grid[0])

        dp = [[0] * cols for _ in range(rows)]

        dp[0][0] = grid[0][0]

        # Fill first row
        for col in range(1, cols):
            dp[0][col] = dp[0][col - 1] + grid[0][col]

        # Fill first column
        for row in range(1, rows):
            dp[row][0] = dp[row - 1][0] + grid[row][0]

        # Fill remaining cells
        for row in range(1, rows):
            for col in range(1, cols):
                dp[row][col] = grid[row][col] + min(
                    dp[row - 1][col],
                    dp[row][col - 1]
                )

        return dp[rows - 1][cols - 1]
```

The implementation begins by determining the number of rows and columns in the grid.

A dynamic programming table named `dp` is then created with identical dimensions. Initially, every value is set to zero.

The starting position is initialized first because it acts as the base case for the entire computation.

Next, the algorithm fills the first row. Since movement is restricted to the right, each cell accumulates the path sum from its left neighbor.

The same logic applies to the first column, except movement is only downward.

After the borders are initialized, the algorithm processes every remaining cell. For each position, it computes the minimum of:

- The path coming from above
- The path coming from the left

Then it adds the current grid value.

Finally, the bottom right cell contains the minimum path sum and is returned.

## Go Solution

```
func minPathSum(grid [][]int) int {
    rows := len(grid)
    cols := len(grid[0])

    dp := make([][]int, rows)

    for i := 0; i < rows; i++ {
        dp[i] = make([]int, cols)
    }

    dp[0][0] = grid[0][0]

    // Fill first row
    for col := 1; col < cols; col++ {
        dp[0][col] = dp[0][col-1] + grid[0][col]
    }

    // Fill first column
    for row := 1; row < rows; row++ {
        dp[row][0] = dp[row-1][0] + grid[row][0]
    }

    // Fill remaining cells
    for row := 1; row < rows; row++ {
        for col := 1; col < cols; col++ {
            if dp[row-1][col] < dp[row][col-1] {
                dp[row][col] = grid[row][col] + dp[row-1][col]
            } else {
                dp[row][col] = grid[row][col] + dp[row][col-1]
            }
        }
    }

    return dp[rows-1][cols-1]
}
```

The Go implementation follows the same dynamic programming strategy as the Python version.

Go slices are used to build the `dp` table. Since Go does not provide a built in `min` function for integers, the implementation uses a simple conditional comparison.

Integer overflow is not a concern here because the maximum possible path sum is:

```
(200 + 200 - 1) × 200 = 79,800
```

which fits comfortably inside Go's `int` type.

The problem guarantees the grid is non empty, so accessing `grid[0]` is safe.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m × n) | Every cell is processed exactly once |
| Space | O(m × n) | The DP table stores one value per grid cell |

The algorithm iterates through the grid a constant number of times. Since there are `m × n` cells and each cell requires only constant work, the total runtime is linear relative to the number of cells.

The space complexity comes from the dynamic programming table. We store one integer for every position in the grid.

## Edge Cases

### Single Cell Grid

Consider:

```
[[5]]
```

The starting cell is also the ending cell.

A buggy implementation might incorrectly assume movement is required or fail when initializing rows and columns.

This implementation handles the case correctly because:

```
dp[0][0] = grid[0][0]
```

and no additional loops execute.

### Single Row Grid

Consider:

```
[[1,2,3,4]]
```

The only valid movement is to the right.

A common bug is incorrectly referencing cells above the first row.

This solution explicitly initializes the first row separately, ensuring correct cumulative sums.

### Single Column Grid

Consider:

```
[
  [1],
  [2],
  [3]
]
```

The only valid movement is downward.

Some implementations accidentally attempt left accesses that do not exist.

This solution avoids that problem by initializing the first column independently.

### Grids Containing Zero

Consider:

```
[
  [0,0],
  [0,0]
]
```

A naive greedy approach might fail if it assumes larger numbers are always unavoidable.

Dynamic programming correctly computes the true minimum regardless of zero values because every possible optimal subpath is considered.

### Large Grid Sizes

A `200 × 200` grid contains 40,000 cells.

A brute force recursive solution would become exponentially slow and likely time out.

The dynamic programming solution processes each cell once, making it efficient enough for the largest allowed inputs.
