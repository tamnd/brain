## Solution

## Problem Understanding

The problem gives an array of intervals, where each interval is represented as `[start, end]`. Each interval describes a continuous range of values from `start` to `end`, inclusive.

The goal is to merge all overlapping intervals and return a new list containing only non-overlapping intervals. Two intervals overlap if one interval starts before the other one ends. In this problem, intervals that touch at the boundary are also considered overlapping. For example, `[1,4]` and `[4,5]` should merge into `[1,5]`.

The input array can be in any order. This means intervals are not guaranteed to be sorted by their starting values. Because of this, one of the key challenges is identifying which intervals overlap efficiently.

The output should contain intervals that collectively cover exactly the same ranges as the original intervals, but without any overlaps between them.

The constraints provide important information about the expected solution:

- The number of intervals can be as large as `10^4`
- Each interval contains exactly two integers
- Interval values are between `0` and `10^4`

A quadratic solution may still pass for small inputs, but for larger inputs it becomes inefficient. Since sorting `10^4` elements is inexpensive, an `O(n log n)` solution is the expected optimal approach.

Several edge cases are important:

- Intervals may already be non-overlapping
- Every interval may overlap into one giant merged interval
- Intervals may appear in completely unsorted order
- Multiple intervals may share the same start value
- Boundary-touching intervals such as `[1,4]` and `[4,5]` must merge
- A single interval input must be returned unchanged

The problem guarantees valid intervals because `starti <= endi`.

## Approaches

### Brute Force Approach

A brute-force solution repeatedly compares intervals against one another to determine whether they overlap. If two intervals overlap, they are merged into a larger interval, and the process continues until no further merges are possible.

One possible implementation repeatedly scans the array looking for overlapping pairs. Whenever a merge occurs, the old intervals are removed and replaced with the merged interval. This continues until the array stabilizes.

This approach works because eventually all overlapping ranges collapse into maximal merged intervals. However, it is inefficient because intervals may be compared many times across repeated passes.

In the worst case, each interval may need to be checked against many others repeatedly, leading to quadratic or even worse behavior depending on implementation details.

### Optimal Approach

The key insight is that overlapping intervals become easy to detect once the intervals are sorted by their starting values.

After sorting:

- If the current interval starts after the previous merged interval ends, there is no overlap
- Otherwise, the intervals overlap and should be merged

Sorting transforms the problem from arbitrary overlap detection into a simple linear scan.

For example, after sorting:

```
[1,3], [2,6], [8,10]
```

When processing `[2,6]`, we only need to compare it with the most recent merged interval `[1,3]`. Because intervals are sorted, there is no need to compare against earlier intervals.

This observation reduces the merging step to a single pass after sorting.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeatedly compares intervals and merges overlaps |
| Optimal | O(n log n) | O(n) | Sort intervals first, then merge in one linear scan |

## Algorithm Walkthrough

1. Sort the intervals by their starting value.

Sorting ensures that intervals are processed from left to right. This makes overlap detection straightforward because any overlapping interval must appear adjacent after sorting.
2. Initialize the result list with the first interval.

The first interval always belongs in the output because there is nothing before it to merge with.
3. Iterate through the remaining intervals one by one.

For each interval, compare it with the most recently merged interval stored in the result list.
4. Check whether the current interval overlaps with the last merged interval.

Overlap occurs when:

$\text{currentStart} \leq \text{lastEnd}$

If this condition is true, the intervals overlap or touch at the boundary.
5. Merge overlapping intervals.

Update the end of the last merged interval to the larger ending value:

$\text{mergedEnd} = \max(\text{lastEnd}, \text{currentEnd})$

This expands the merged interval to fully cover both ranges.
6. Handle non-overlapping intervals.

If the current interval starts after the last merged interval ends, there is no overlap. Append the current interval directly to the result list.
7. Continue until all intervals are processed.

At the end of the traversal, the result list contains all merged non-overlapping intervals.

## Python Solution

```
from typing import List

class Solution:
    def merge(self, intervals: List[List[int]]) -> List[List[int]]:
        intervals.sort(key=lambda interval: interval[0])

        merged_intervals = [intervals[0]]

        for current_start, current_end in intervals[1:]:
            last_interval = merged_intervals[-1]
            last_end = last_interval[1]

            if current_start <= last_end:
                last_interval[1] = max(last_end, current_end)
            else:
                merged_intervals.append([current_start, current_end])

        return merged_intervals
```

The implementation begins by sorting the intervals based on their starting values. This guarantees that overlapping intervals appear next to each other.

The `merged_intervals` list stores the final merged output. The first interval is inserted immediately because it must appear in the answer.

The loop processes every remaining interval. For each interval, the algorithm compares it with the last merged interval in the result list.

If the current interval overlaps with the previous merged interval, the code updates the end value of the previous interval using `max()`. This ensures the merged interval fully covers both ranges.

If no overlap exists, the current interval begins a new independent range and is appended to the result list.

Finally, the merged intervals are returned.

## Go Solution

```
package main

import "sort"

func merge(intervals [][]int) [][]int {
	sort.Slice(intervals, func(i, j int) bool {
		return intervals[i][0] < intervals[j][0]
	})

	mergedIntervals := [][]int{intervals[0]}

	for i := 1; i < len(intervals); i++ {
		currentStart := intervals[i][0]
		currentEnd := intervals[i][1]

		lastInterval := mergedIntervals[len(mergedIntervals)-1]
		lastEnd := lastInterval[1]

		if currentStart <= lastEnd {
			if currentEnd > lastEnd {
				lastInterval[1] = currentEnd
			}
		} else {
			mergedIntervals = append(mergedIntervals, []int{currentStart, currentEnd})
		}
	}

	return mergedIntervals
}
```

The Go solution follows the same logic as the Python implementation. The main difference is that Go uses `sort.Slice` to sort the intervals.

Slices in Go are reference-like structures, so modifying `lastInterval[1]` directly updates the interval stored inside `mergedIntervals`.

Unlike Python, Go does not have built-in tuple unpacking or a built-in `max()` function for integers, so the comparison is written explicitly using an `if` statement.

The constraints are small enough that integer overflow is not a concern because interval values are at most `10^4`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log n) | Sorting dominates the runtime |
| Space | O(n) | Output list may contain all intervals |

The most expensive step is sorting the intervals, which requires `O(n log n)` time. After sorting, the algorithm performs a single linear scan through the intervals, which costs `O(n)`.

The output list may contain as many as `n` intervals when no overlaps exist, so the space complexity is `O(n)`. Aside from the output storage, the algorithm uses only constant extra space.

## Edge Cases

### Single Interval

An input such as:

```
[[5,7]]
```

contains only one interval. A buggy implementation might assume there are at least two intervals to compare.

This solution handles the case naturally because the result list is initialized with the first interval, and the loop over `intervals[1:]` simply does not execute.

### Completely Non-Overlapping Intervals

Consider:

```
[[1,2],[4,5],[7,8]]
```

No intervals overlap, so every interval should appear unchanged in the output.

A common bug is accidentally merging intervals that merely appear close together. The implementation avoids this by requiring:

```
current_start <= last_end
```

for merging.

Since each interval starts strictly after the previous interval ends, all intervals are appended independently.

### Fully Nested Intervals

Consider:

```
[[1,10],[2,3],[4,5],[6,7]]
```

All smaller intervals lie completely inside `[1,10]`.

A buggy implementation might incorrectly shrink the interval when processing smaller ranges. This implementation always updates the merged end using the maximum end value, ensuring the larger interval remains intact.

The final output correctly remains:

```
[[1,10]]
```

### Boundary-Touching Intervals

Consider:

```
[[1,4],[4,5]]
```

The problem explicitly states that touching intervals count as overlapping.

A common mistake is using:

```
current_start < last_end
```

instead of:

```
current_start <= last_end
```

This implementation uses `<=`, so boundary-touching intervals merge correctly into `[1,5]`.
