## Solution

## Problem Understanding

The problem asks us to rearrange a string into a zigzag pattern across a given number of rows, then read the characters row by row to produce a new string.

The zigzag pattern works by writing characters vertically downward through the rows, then diagonally upward back toward the first row, repeating this movement until all characters are placed. After constructing this visual arrangement, we concatenate the rows from top to bottom.

For example, with the string `"PAYPALISHIRING"` and `numRows = 3`, the pattern becomes:

```
P   A   H   N
A P L S I I G
Y   I   R
```

Reading row by row produces:

```
PAHNAPLSIIGYIR
```

The input consists of:

- A string `s`
- An integer `numRows`

The output must be the zigzag-converted string.

The constraints are relatively small:

- `s.length <= 1000`
- `numRows <= 1000`

Because the input size is modest, we do not need extremely advanced optimization techniques. However, we should still aim for a clean linear-time solution.

Several edge cases are important:

- If `numRows == 1`, there is no zigzag movement at all. The string should be returned unchanged.
- If `numRows >= len(s)`, every character effectively occupies its own row without any diagonal movement.
- Very short strings can easily expose indexing mistakes or incorrect direction changes.

A correct implementation must carefully simulate the movement between rows.

## Approaches

### Brute Force Approach

A straightforward way to solve the problem is to explicitly build a two-dimensional grid representing the zigzag layout.

We could simulate the writing process by placing characters into a matrix row by row and column by column. Vertical movement would fill one column downward, and diagonal movement would fill upward diagonally across multiple columns.

After constructing the full matrix, we would traverse the rows and collect all non-empty characters.

This approach works because it directly mirrors the visual zigzag structure shown in the problem statement. However, it is inefficient in both implementation complexity and memory usage. Most cells in the matrix remain empty, especially for large strings and row counts.

The extra bookkeeping for column movement also makes the implementation more error-prone.

### Optimal Approach

The key observation is that we do not actually need to build the visual grid.

The final answer only depends on the characters that belong to each row. Instead of storing empty spaces and columns, we can maintain one string builder for each row.

As we iterate through the input string:

- We append the current character to the current row.
- We move downward through the rows.
- Once we reach the bottom row, we reverse direction and move upward.
- Once we reach the top row again, we reverse direction once more.

This simulates the zigzag traversal directly without constructing the full matrix.

At the end, we concatenate all row strings together.

This approach is efficient because:

- Every character is processed exactly once.
- No unnecessary empty cells are stored.
- Direction changes are easy to track with a simple flag.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * numRows) | O(n * numRows) | Builds an explicit zigzag grid with many unused cells |
| Optimal | O(n) | O(n) | Simulates row traversal directly using row builders |

## Algorithm Walkthrough

1. Handle the special case where `numRows == 1`.

In this situation, the zigzag pattern degenerates into a straight line. No rearrangement occurs, so we immediately return the original string.
2. Create a collection of row builders.

We maintain one mutable string container for each row. In Python this can be a list of strings, and in Go it can be a slice of byte slices or string builders.
3. Initialize traversal state.

We keep track of:

- `current_row`, the row where the next character will be placed
- `direction`, which determines whether we are moving downward or upward
4. Iterate through each character in the string.

For every character:

- Append it to the current row.
- Check whether we reached the top or bottom row.
- If we reached either boundary, reverse the movement direction.
- Move to the next row according to the direction.
5. Concatenate all rows.

After all characters are distributed across rows, combine every row into a single string and return the result.

This algorithm works because the zigzag pattern is fundamentally just repeated movement up and down between rows.

## Python Solution

```
class Solution:
    def convert(self, s: str, numRows: int) -> str:
        if numRows == 1 or numRows >= len(s):
            return s

        rows = [""] * numRows

        current_row = 0
        going_down = False

        for char in s:
            rows[current_row] += char

            if current_row == 0 or current_row == numRows - 1:
                going_down = not going_down

            if going_down:
                current_row += 1
            else:
                current_row -= 1

        return "".join(rows)
```

The implementation begins by handling the two important trivial cases. If there is only one row, or if the number of rows exceeds the string length, no zigzag transformation is needed.

Next, we create an array called `rows`, where each element stores the characters belonging to that row.

The variable `current_row` tracks where the next character should be placed. The boolean variable `going_down` controls movement direction.

For every character in the input string:

- The character is appended to the current row.
- If we are at the top row or bottom row, the direction flips.
- The row index is then updated according to the current direction.

Finally, all row strings are concatenated together using `"".join(rows)`.

## Go Solution

```
func convert(s string, numRows int) string {
    if numRows == 1 || numRows >= len(s) {
        return s
    }

    rows := make([][]byte, numRows)

    currentRow := 0
    goingDown := false

    for i := 0; i < len(s); i++ {
        rows[currentRow] = append(rows[currentRow], s[i])

        if currentRow == 0 || currentRow == numRows-1 {
            goingDown = !goingDown
        }

        if goingDown {
            currentRow++
        } else {
            currentRow--
        }
    }

    result := make([]byte, 0, len(s))

    for _, row := range rows {
        result = append(result, row...)
    }

    return string(result)
}
```

The Go implementation follows the same logic as the Python version, but there are a few language-specific details worth noting.

Instead of using string concatenation repeatedly, we use slices of bytes for efficiency. Appending to byte slices avoids unnecessary string allocations during iteration.

The final result is assembled into a single byte slice and converted back to a string at the end.

Because Go strings are immutable, using byte slices is typically the most efficient approach for problems involving repeated character appends.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every character is processed exactly once |
| Space | O(n) | The row builders collectively store all characters |

The algorithm visits each character exactly one time and performs constant-time operations during each iteration. Therefore, the runtime grows linearly with the input size.

The auxiliary space usage is also linear because we store the rearranged characters across the row containers before constructing the final result.

## Edge Cases

### Single Row

If `numRows == 1`, the zigzag structure disappears entirely. A naive implementation that always tries to move upward and downward may accidentally produce index errors or infinite direction switching.

The implementation handles this immediately with an early return.

### Number of Rows Greater Than String Length

Consider:

```
s = "ABC"
numRows = 5
```

In this case, each character simply occupies a separate row, and no diagonal traversal occurs. The correct output remains `"ABC"`.

The implementation handles this naturally through the early return condition:

```
if numRows >= len(s):
    return s
```

### Very Small Strings

Short strings such as `"A"` or `"AB"` can expose incorrect direction logic because the traversal changes direction very quickly.

The implementation avoids these issues because direction changes only occur at the top or bottom row boundaries, and the row index is updated carefully after each character placement.

### Boundary Direction Changes

The most common bug in zigzag conversion problems is incorrectly reversing direction.

For example, when reaching the last row, the algorithm must begin moving upward immediately. Similarly, after reaching the first row, it must begin moving downward again.

The implementation handles this using:

```
if current_row == 0 or current_row == numRows - 1:
    going_down = not going_down
```

This guarantees correct oscillation between rows throughout the traversal.
