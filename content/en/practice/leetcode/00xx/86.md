## Solution

## Problem Understanding

This problem asks us to rearrange a singly linked list around a pivot value `x`. Every node whose value is less than `x` must appear before every node whose value is greater than or equal to `x`.

The important detail is that the relative ordering inside each partition must remain unchanged. This means the operation is stable. If two nodes both belong to the "less than x" group, they must appear in the same order as they originally appeared in the input list. The same rule applies to nodes greater than or equal to `x`.

For example, consider:

```
1 -> 4 -> 3 -> 2 -> 5 -> 2
x = 3
```

Nodes less than `3` are:

```
1 -> 2 -> 2
```

Nodes greater than or equal to `3` are:

```
4 -> 3 -> 5
```

The final merged list becomes:

```
1 -> 2 -> 2 -> 4 -> 3 -> 5
```

The input is the head of a singly linked list and an integer `x`. The output must be the head of the reordered linked list.

The constraints are relatively small, at most 200 nodes, but the problem is fundamentally about linked list manipulation rather than input size. Even though a less efficient solution would still pass due to the small constraints, the interview expectation is to solve this in linear time with careful pointer management.

Several edge cases are important:

- The list may be empty.
- All nodes may already be less than `x`.
- All nodes may already be greater than or equal to `x`.
- The pivot value `x` may not exist in the list.
- Duplicate values may appear many times.
- Maintaining stable ordering is essential, naive swapping approaches often break this requirement.

## Approaches

### Brute Force Approach

A straightforward approach is to traverse the linked list and copy node values into two separate arrays or lists:

- One list stores values less than `x`
- Another list stores values greater than or equal to `x`

After traversal, concatenate the two arrays and rebuild the linked list.

This works because we process nodes from left to right, so the original relative ordering inside each partition is naturally preserved.

However, this approach uses extra memory proportional to the number of nodes because we duplicate the values into auxiliary storage and potentially create new nodes. While acceptable for small constraints, it is not optimal because linked list problems usually expect in place pointer manipulation.

### Optimal Approach

The key insight is that we do not need to reorder nodes by swapping or sorting. We only need to separate nodes into two stable partitions.

We can build two linked lists while traversing the original list:

- A "small" list containing nodes with values less than `x`
- A "large" list containing nodes with values greater than or equal to `x`

As we iterate through the original list:

- Append each node to the appropriate partition
- Preserve insertion order by always appending to the tail

At the end:

- Connect the tail of the small list to the head of the large list
- Terminate the large list properly with `None`

This produces a stable partition in one pass with constant extra space.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Uses extra arrays or rebuilt nodes |
| Optimal | O(n) | O(1) | Reuses original nodes with pointer manipulation |

## Algorithm Walkthrough

1. Create two dummy nodes, one for the "less than x" partition and one for the "greater than or equal to x" partition.

Dummy nodes simplify edge case handling because we always have a stable starting point for both lists.
2. Maintain two tail pointers.

One pointer tracks the end of the smaller partition, and another tracks the end of the larger partition.
3. Traverse the original linked list node by node.

For each node:

- If `node.val < x`, append it to the smaller partition.
- Otherwise, append it to the larger partition.
4. Move the corresponding tail pointer forward.

This preserves the original ordering because nodes are appended in the exact order they are encountered.
5. Continue until the entire original list has been processed.
6. Terminate the larger partition.

This step is critical. Since we reuse original nodes, some nodes may still point to old connections. Setting the final large tail's `next` to `None` prevents accidental cycles or incorrect links.
7. Connect the smaller partition to the larger partition.

Specifically:

```
small_tail.next = large_dummy.next
```
8. Return the head of the new combined list.

The actual head starts after the dummy node:

```
small_dummy.next
```

## Python Solution

```
# Definition for singly-linked list.
# class ListNode:
#     def __init__(self, val=0, next=None):
#         self.val = val
#         self.next = next

class Solution:
    def partition(self, head: Optional[ListNode], x: int) -> Optional[ListNode]:
        small_dummy = ListNode(0)
        large_dummy = ListNode(0)

        small_tail = small_dummy
        large_tail = large_dummy

        current = head

        while current:
            next_node = current.next

            if current.val < x:
                small_tail.next = current
                small_tail = small_tail.next
            else:
                large_tail.next = current
                large_tail = large_tail.next

            current = next_node

        large_tail.next = None
        small_tail.next = large_dummy.next

        return small_dummy.next
```

The implementation starts by creating two dummy nodes. These serve as fixed starting points for the two partitions and eliminate special handling for empty lists.

The `small_tail` pointer always points to the last node in the smaller partition, while `large_tail` points to the last node in the larger partition.

During traversal, we store `current.next` in `next_node` before modifying pointers. This is important because once we attach the current node to another list, its original linkage may change.

If the current node value is less than `x`, we append it to the smaller list. Otherwise, we append it to the larger list.

After processing all nodes, we terminate the larger partition with `None`. Without this step, leftover links from the original list could produce incorrect results.

Finally, we connect the two partitions and return the head of the combined list.

## Go Solution

```
/**
 * Definition for singly-linked list.
 * type ListNode struct {
 *     Val int
 *     Next *ListNode
 * }
 */
func partition(head *ListNode, x int) *ListNode {
    smallDummy := &ListNode{}
    largeDummy := &ListNode{}

    smallTail := smallDummy
    largeTail := largeDummy

    current := head

    for current != nil {
        nextNode := current.Next

        if current.Val < x {
            smallTail.Next = current
            smallTail = smallTail.Next
        } else {
            largeTail.Next = current
            largeTail = largeTail.Next
        }

        current = nextNode
    }

    largeTail.Next = nil
    smallTail.Next = largeDummy.Next

    return smallDummy.Next
}
```

The Go implementation follows the same logic as the Python version. Since Go uses explicit pointers, we allocate dummy nodes using `&ListNode{}`.

Go uses `nil` instead of Python's `None`.

No special integer overflow handling is needed because the node values are very small according to the constraints.

The overall pointer manipulation strategy remains identical between the two languages.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each node is visited exactly once |
| Space | O(1) | Only a few pointer variables are used |

The algorithm performs a single traversal of the linked list. Every node is appended to exactly one partition, so the total work grows linearly with the number of nodes.

No additional data structures proportional to the input size are created. The algorithm only uses a constant number of pointers and dummy nodes, which gives constant auxiliary space complexity.

## Edge Cases

### Empty List

If the input list is empty, the traversal loop never executes. Both partitions remain empty, and the function correctly returns `None`.

This case is important because linked list code often fails when dereferencing null pointers. The dummy node approach avoids this issue cleanly.

### All Nodes Less Than `x`

Consider:

```
1 -> 1 -> 2
x = 5
```

Every node belongs to the smaller partition. The larger partition remains empty.

At the end, connecting the smaller partition to an empty larger partition still works correctly because `large_dummy.next` is `None`.

The original ordering is fully preserved.

### All Nodes Greater Than or Equal to `x`

Consider:

```
5 -> 6 -> 7
x = 3
```

Every node belongs to the larger partition. The smaller partition remains empty.

Because we return `small_dummy.next`, which gets connected to the large partition head, the original list is returned unchanged.

### Duplicate Values Around the Pivot

Consider:

```
3 -> 1 -> 3 -> 2 -> 3
x = 3
```

All `3` values must remain in their original relative order.

A naive swapping approach could accidentally reorder equal elements. This implementation preserves stability because nodes are appended sequentially in encounter order.

### Preventing Cycles

Since nodes are reused rather than recreated, leftover `next` pointers from the original list can accidentally create cycles or incorrect links.

The line:

```
large_tail.next = None
```

is essential because it cleanly terminates the reconstructed linked list.
