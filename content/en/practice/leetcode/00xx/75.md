## Solution

## Problem Understanding

This problem asks us to sort an array containing only three possible values: `0`, `1`, and `2`. These numbers represent colors, specifically red, white, and blue. The goal is to rearrange the array so that all `0`s appear first, followed by all `1`s, and then all `2`s.

The important detail is that the sorting must happen **in-place**. This means we cannot create a separate array to hold the sorted result. Instead, we must modify the original `nums` array directly. Additionally, the problem explicitly forbids using the language's built-in sorting function.

In other words, we are given an unsorted array of three categories, and we need to partition it into three contiguous sections:

- First section contains all `0`s
- Second section contains all `1`s
- Third section contains all `2`s

The input is an integer array `nums`, where every element is guaranteed to be one of `0`, `1`, or `2`. The output is not returned from the function. Instead, the array itself must be modified so that it becomes sorted in the required order.

The constraints tell us several useful things. The array length is between `1` and `300`, which is relatively small. Even an `O(n log n)` solution would easily pass within limits. However, the follow-up explicitly asks for a **one-pass algorithm using constant extra space**, which strongly suggests that the intended optimal solution is linear time with no auxiliary data structures.

The problem also guarantees that all values are valid, meaning we never need to handle unexpected numbers outside the set `{0,1,2}`. This simplifies implementation because we only need logic for three known categories.

Several edge cases are worth thinking about upfront. An array of length `1`, such as `[0]`, is already sorted and should remain unchanged. Arrays where all elements are identical, such as `[2,2,2]`, should also work correctly without unnecessary swaps. Another important case is when the array is already sorted or completely reversed, since some pointer-based algorithms can accidentally skip elements or over-swap if not implemented carefully.

## Approaches

### Brute Force Approach

A straightforward solution is to count how many times each color appears and then overwrite the array.

We can make one pass through the array and count the number of `0`s, `1`s, and `2`s. Once we know these frequencies, we can overwrite the array so that the first section contains the required number of `0`s, the second section contains the required number of `1`s, and the final section contains the required number of `2`s.

This approach is correct because the final arrangement depends only on the counts of each color. Since there are only three possible values, reconstructing the array guarantees the required ordering.

Although this method runs in linear time, it requires two passes through the array and conceptually behaves like a counting sort. It still uses constant extra space because we only store three counters, but it does not satisfy the spirit of the follow-up question asking for a **one-pass solution**.

### Key Insight for the Optimal Solution

The key observation is that since there are only three values, we can partition the array in a single traversal.

Instead of counting first and rebuilding later, we can maintain three regions while scanning:

- A left region for `0`s
- A middle region for `1`s
- A right region for `2`s

As we inspect elements, we place them into the correct region immediately. This idea leads to the **Dutch National Flag algorithm**, named after the three-color Dutch flag.

We maintain three pointers:

- `left`, where the next `0` should go
- `current`, which scans the array
- `right`, where the next `2` should go

The logic is simple:

- If we see a `0`, swap it toward the front.
- If we see a `1`, leave it in the middle and move on.
- If we see a `2`, swap it toward the back.

The subtle but important detail is that after swapping with the `right` pointer, we do **not** immediately advance `current`, because the swapped-in value still needs inspection.

This gives us a true one-pass, constant-space solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Count occurrences, then overwrite array |
| Optimal | O(n) | O(1) | One-pass Dutch National Flag algorithm |

## Algorithm Walkthrough

1. Initialize three pointers: `left = 0`, `current = 0`, and `right = len(nums) - 1`.

The `left` pointer tracks where the next `0` belongs. The `right` pointer tracks where the next `2` belongs. The `current` pointer scans through the array and decides what action to take for each element.
2. Continue processing while `current <= right`.

We stop when `current` crosses `right`, because at that point every element has been classified into the correct region.
3. If `nums[current] == 0`, swap it with `nums[left]`.

This places the `0` into the correct front region. Since both the swapped position and the current position are now resolved, increment both `left` and `current`.
4. If `nums[current] == 1`, simply increment `current`.

A `1` already belongs in the middle region, so no swapping is necessary.
5. If `nums[current] == 2`, swap it with `nums[right]`.

This moves the `2` into the correct back region. After the swap, decrement `right`.

Importantly, do **not** increment `current` yet. The value swapped in from the right side has not been examined, so we must process it in the next iteration.
6. Repeat until the scanning region is exhausted.

By the end, the array is partitioned into sorted sections: `0`s first, then `1`s, then `2`s.

## Python Solution

```
from typing import List

class Solution:
    def sortColors(self, nums: List[int]) -> None:
        """
        Do not return anything, modify nums in-place instead.
        """
        left = 0
        current = 0
        right = len(nums) - 1

        while current <= right:
            if nums[current] == 0:
                nums[left], nums[current] = nums[current], nums[left]
                left += 1
                current += 1

            elif nums[current] == 1:
                current += 1

            else:  # nums[current] == 2
                nums[current], nums[right] = nums[right], nums[current]
                right -= 1
```

The implementation closely follows the Dutch National Flag algorithm.

The three pointers define dynamic boundaries inside the array. The `left` pointer marks the end of the `0` region, while `right` marks the beginning of the `2` region. Everything between `left` and `current - 1` is guaranteed to already contain `1`s.

The `while current <= right` condition ensures every element is processed exactly once. When a `0` is found, we swap it into the front region and advance both pointers because the new position is finalized. When encountering a `1`, no rearrangement is needed, so only `current` advances.

The handling of `2`s is the trickiest part. After swapping with the `right` side, we intentionally do not increment `current`. This is because the new element brought into the current position could be `0`, `1`, or another `2`, and must still be processed.

Since every operation is performed directly on `nums`, the algorithm satisfies the in-place requirement.

## Go Solution

```
func sortColors(nums []int) {
	left := 0
	current := 0
	right := len(nums) - 1

	for current <= right {
		if nums[current] == 0 {
			nums[left], nums[current] = nums[current], nums[left]
			left++
			current++

		} else if nums[current] == 1 {
			current++

		} else {
			nums[current], nums[right] = nums[right], nums[current]
			right--
		}
	}
}
```

The Go implementation is almost identical to the Python version. Since slices in Go are reference-like structures, modifying `nums` directly updates the original data in-place, which matches the problem requirements.

Go does not require explicit type hints because the slice type `[]int` already defines the input type. There are no concerns about integer overflow here because the constraints are very small and pointer indices remain within valid bounds.

An empty slice is not possible due to the constraint `n >= 1`, but even if it were, the loop condition would safely prevent invalid access.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each element is processed at most once |
| Space | O(1) | Only three pointers are used |

The time complexity is `O(n)` because the `current` pointer traverses the array once, and each swap operation takes constant time. Even though some elements may be revisited after swapping with `right`, every position is still processed a bounded number of times, so the total work remains linear.

The space complexity is `O(1)` because no additional data structures proportional to input size are used. Only three integer pointers are maintained regardless of array length.

## Edge Cases

One important edge case is an array with only one element, such as `[1]`. A naive implementation might accidentally access out-of-range indices or perform unnecessary swaps. In this solution, the loop runs safely because `current` and `right` both start at `0`, and the algorithm immediately terminates after processing the single element.

Another edge case is when all elements are identical, such as `[0,0,0]` or `[2,2,2]`. Some partitioning algorithms can get stuck repeatedly swapping the same values. Here, the pointer movement guarantees progress. For all `0`s, both `left` and `current` steadily move forward. For all `2`s, `right` steadily shrinks.

A third important case is an already sorted array such as `[0,0,1,1,2,2]`. The algorithm should avoid breaking correct ordering. Since `0`s and `1`s are handled naturally, and `2`s remain in their correct region, the array stays valid without unnecessary disruption.

A final tricky case is a reverse-sorted array like `[2,2,1,1,0,0]`. This stresses the swapping logic, especially the rule that `current` must not advance after swapping with `right`. The implementation handles this correctly because every newly swapped-in element is re-evaluated before moving on.
