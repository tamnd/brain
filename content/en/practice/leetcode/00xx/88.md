# LeetCode 88, Merge Sorted Array

## Problem Understanding

This problem gives us two arrays, `nums1` and `nums2`, that are already sorted in non-decreasing order. The goal is to merge them into a single sorted array. However, there is an important constraint: the final result must be stored directly inside `nums1`.

The array `nums1` has a total size of `m + n`. The first `m` elements are valid sorted values, while the last `n` positions are placeholders filled with `0`. These extra positions exist specifically to make room for the elements from `nums2`.

The array `nums2` contains `n` valid sorted elements.

The problem does not ask us to return a new array. Instead, we must modify `nums1` in-place.

For example:

```
nums1 = [1,2,3,0,0,0]
m = 3

nums2 = [2,5,6]
n = 3
```

The valid portion of `nums1` is `[1,2,3]`. We merge this with `[2,5,6]` and store the final sorted result back into `nums1`:

```
[1,2,2,3,5,6]
```

The constraints are relatively small, with at most 200 total elements, so even slower solutions could pass. However, the follow-up explicitly asks for an `O(m + n)` solution, which strongly suggests that we should avoid unnecessary sorting or copying.

Several edge cases are important:

- `nums2` may be empty, meaning no merge is needed.
- `nums1` may initially contain no valid elements, meaning all values come from `nums2`.
- Arrays can contain duplicate values.
- Arrays can contain negative numbers.
- Because the merge must happen in-place, overwriting values too early can corrupt data if we are not careful.

The main challenge is performing the merge efficiently while preserving all existing values in `nums1`.

## Approaches

### Brute Force Approach

A straightforward solution is to copy all elements from `nums2` into the unused portion of `nums1`, then sort the entire array.

The process would look like this:

1. Insert every element of `nums2` into the trailing zero positions of `nums1`.
2. Sort `nums1`.

This works because after inserting all elements, `nums1` contains every value exactly once. Sorting restores the required non-decreasing order.

Although correct, this approach is not optimal. Sorting costs `O((m+n) log(m+n))` time, which is slower than necessary because the arrays are already individually sorted.

### Optimal Approach

The key insight is that we can merge the arrays directly in linear time without re-sorting everything.

Normally, merging two sorted arrays is done from left to right. However, doing that here would overwrite values in `nums1` before they are processed.

Instead, we work backward.

Since the end of `nums1` contains empty space, we can safely place the largest remaining element at the back of the array. By filling positions from right to left, we never overwrite unprocessed data.

We maintain three pointers:

- One pointer at the end of the valid portion of `nums1`
- One pointer at the end of `nums2`
- One pointer at the very end of `nums1`, where the next largest value should be placed

At each step, we compare the two current elements and place the larger one at the back.

This gives an `O(m + n)` solution with `O(1)` extra space.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O((m+n) log(m+n)) | O(1) or O(log(m+n)) depending on sort | Copy elements then sort entire array |
| Optimal | O(m+n) | O(1) | Merge from the back using three pointers |

## Algorithm Walkthrough

1. Initialize three pointers.

The pointer `i` starts at `m - 1`, which is the last valid element in `nums1`.

The pointer `j` starts at `n - 1`, which is the last element in `nums2`.

The pointer `k` starts at `m + n - 1`, which is the final index of `nums1`.
2. Compare elements from the back.

Compare `nums1[i]` and `nums2[j]`.

The larger value belongs at position `k` because we are building the array from largest to smallest.
3. Place the larger element.

If `nums1[i] > nums2[j]`, place `nums1[i]` into `nums1[k]`.

Otherwise, place `nums2[j]` into `nums1[k]`.
4. Move pointers.

After placing an element:

- Decrement the pointer for the array that supplied the element
- Decrement `k`
5. Continue until all elements from `nums2` are processed.

The loop continues while `j >= 0`.

We only need to ensure all elements from `nums2` are inserted because any remaining elements in `nums1` are already in correct position.
6. Finish with the merged array stored in `nums1`.

## Python Solution

```
from typing import List

class Solution:
    def merge(self, nums1: List[int], m: int, nums2: List[int], n: int) -> None:
        """
        Do not return anything, modify nums1 in-place instead.
        """

        i = m - 1
        j = n - 1
        k = m + n - 1

        while j >= 0:
            if i >= 0 and nums1[i] > nums2[j]:
                nums1[k] = nums1[i]
                i -= 1
            else:
                nums1[k] = nums2[j]
                j -= 1

            k -= 1
```

The implementation begins by creating three pointers.

The pointer `i` tracks the last valid element in `nums1`. The pointer `j` tracks the last element in `nums2`. The pointer `k` tracks the insertion position at the back of `nums1`.

The loop continues while there are still elements remaining in `nums2`. During each iteration, the algorithm compares the current largest remaining elements from both arrays.

If the current element in `nums1` is larger, it is copied to position `k`. Otherwise, the current element from `nums2` is copied.

After placing an element, the corresponding pointer moves left, and `k` also moves left.

The reason this works safely is that we fill the array from the end, so no unprocessed values are overwritten.

Once all elements from `nums2` are inserted, the merge is complete.

## Go Solution

```
func merge(nums1 []int, m int, nums2 []int, n int) {

	i := m - 1
	j := n - 1
	k := m + n - 1

	for j >= 0 {

		if i >= 0 && nums1[i] > nums2[j] {
			nums1[k] = nums1[i]
			i--
		} else {
			nums1[k] = nums2[j]
			j--
		}

		k--
	}
}
```

The Go implementation follows exactly the same logic as the Python version.

One important detail is that slices in Go are reference-like structures, so modifying `nums1` inside the function directly updates the caller's array. No explicit return value is needed.

There are no integer overflow concerns because the problem constraints are small and indices remain within valid slice bounds.

The condition `i >= 0` is important because `nums1` may initially contain zero valid elements.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m+n) | Each element is processed at most once |
| Space | O(1) | Only a few pointer variables are used |

The algorithm runs in linear time because every comparison moves one pointer leftward, and each array element is handled exactly once.

The solution uses constant extra space because it modifies `nums1` directly without allocating another array.

## Edge Cases

### Case 1: `nums2` Is Empty

If `n = 0`, there is nothing to merge. A careless implementation might still attempt comparisons and cause index errors.

This implementation handles the case naturally because the loop condition is `while j >= 0`. When `j` starts at `-1`, the loop never executes, and `nums1` remains unchanged.

### Case 2: `nums1` Has No Valid Elements

If `m = 0`, then all values must come from `nums2`.

This is a common source of bugs because `i` becomes `-1`. The implementation explicitly checks `i >= 0` before accessing `nums1[i]`, preventing invalid indexing.

The algorithm then copies all elements from `nums2` into `nums1`.

### Case 3: Duplicate Values

Arrays may contain repeated numbers, such as:

```
nums1 = [2,2,3,0,0]
nums2 = [2,5]
```

The algorithm correctly preserves duplicates because it compares values individually and inserts every element exactly once.

No special duplicate handling is required.

### Case 4: All Elements in `nums2` Are Smaller

Example:

```
nums1 = [4,5,6,0,0,0]
nums2 = [1,2,3]
```

In this situation, many implementations that merge from the front would overwrite values before processing them.

The backward merge strategy avoids this entirely. Larger elements are moved to the end first, leaving room for smaller elements to be inserted safely at the beginning.

### Case 5: Negative Numbers

The arrays may contain negative integers.

Because the algorithm only relies on standard numeric comparisons, negative values are handled exactly the same way as positive values. No special logic is needed.
