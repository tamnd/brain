## Solution

## Problem Understanding

The problem asks us to find the longest prefix shared by every string in a given array of strings.

A prefix is a sequence of characters that appears at the beginning of a string. For example, the prefixes of `"flower"` include `"f"`, `"fl"`, `"flo"`, and so on. The goal is to determine the longest such prefix that all strings in the array have in common.

The input is an array of strings named `strs`. Each element is a lowercase English string, and the array contains at least one string. The output should be a single string representing the longest common prefix among all strings. If no common prefix exists, the function should return an empty string `""`.

For example, given:

```
["flower", "flow", "flight"]
```

all strings begin with `"fl"`, but the third character differs, so the answer is `"fl"`.

The constraints are relatively small:

- At most 200 strings
- Each string length is at most 200

This means even less optimized solutions could pass comfortably, but the goal is to design a clean and efficient algorithm.

Several edge cases are important to recognize immediately. If the array contains only one string, that entire string is the answer because it trivially matches itself. If any string is empty, the answer must be an empty string because no non-empty prefix can exist. Another important case is when the very first character differs between strings, such as `["dog", "racecar", "car"]`. In that situation, the correct result is `""`.

## Approaches

### Brute Force Approach

A straightforward brute-force solution is to generate every possible prefix of the first string and check whether all other strings contain that prefix at the beginning.

Suppose the first string is `"flower"`. We could test:

- `"f"`
- `"fl"`
- `"flo"`
- `"flow"`
- `"flowe"`
- `"flower"`

For each candidate prefix, we compare it against every other string using substring checks.

This approach is correct because every common prefix must also be a prefix of the first string. By checking all prefixes in increasing order and keeping track of the longest valid one, we eventually find the correct answer.

However, this method performs unnecessary repeated substring operations. Every prefix comparison can involve creating temporary substrings and repeatedly scanning characters that were already checked before. While the constraints are small enough for this to pass, it is not the cleanest or most efficient solution.

### Optimal Approach

The key observation is that the common prefix can only extend as long as all strings share the same character at the same position.

Instead of generating prefixes explicitly, we can compare characters column by column:

- Compare all first characters
- Then compare all second characters
- Continue until a mismatch occurs

The moment we encounter a mismatch, or one string ends, we know the common prefix cannot continue further.

This works because prefixes are positional. If two strings differ at index `i`, then no valid common prefix can include index `i`.

This character-by-character comparison avoids repeated substring generation and performs only the necessary checks.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(S²) | O(1) | Repeatedly creates and checks prefixes |
| Optimal | O(S) | O(1) | Compares characters directly across strings |

Here, `S` represents the total number of characters across all strings.

## Algorithm Walkthrough

1. Start by selecting the first string as the reference string.

Any common prefix must also be a prefix of the first string, so we only need to examine its characters.
2. Iterate through each character index of the first string.

At every position `i`, retrieve the current character from the first string.
3. Compare this character with the character at the same position in every other string.

During this comparison, two problems can occur:

- Another string may already end before reaching index `i`
- Another string may contain a different character at index `i`

Either condition means the common prefix stops immediately before index `i`.
4. If a mismatch occurs, return the substring from the beginning up to index `i`.

This substring represents the longest prefix confirmed to match in all strings.
5. If the loop completes without mismatches, return the entire first string.

This means every character of the first string appeared in all other strings at the same positions.

This approach uses only direct character comparisons and requires no additional data structures.

## Python Solution

```
from typing import List

class Solution:
    def longestCommonPrefix(self, strs: List[str]) -> str:
        first_string = strs[0]

        for index in range(len(first_string)):
            current_char = first_string[index]

            for string in strs[1:]:
                if index >= len(string) or string[index] != current_char:
                    return first_string[:index]

        return first_string
```

The implementation begins by storing the first string in `first_string`. Since every valid common prefix must come from this string, it serves as the reference for all comparisons.

The outer loop iterates through each character index of the reference string. For every index, the code stores the current character in `current_char`.

The inner loop checks every remaining string. Two conditions are tested:

- If the current index exceeds the string length, that string ended early
- If the character at the current index differs from `current_char`, the prefix no longer matches

When either condition occurs, the function immediately returns the prefix up to the current index.

If the loops finish completely, then every character of the first string matched across all strings, so the entire first string is returned.

## Go Solution

```
func longestCommonPrefix(strs []string) string {
    firstString := strs[0]

    for i := 0; i < len(firstString); i++ {
        currentChar := firstString[i]

        for j := 1; j < len(strs); j++ {
            currentString := strs[j]

            if i >= len(currentString) || currentString[i] != currentChar {
                return firstString[:i]
            }
        }
    }

    return firstString
}
```

The Go implementation follows the exact same logic as the Python solution. The main difference is that Go strings are indexed as byte slices. Since the problem guarantees lowercase English letters only, byte indexing is completely safe.

Go also uses explicit index-based loops instead of Python's more concise iteration syntax. Slicing syntax remains straightforward with `firstString[:i]`.

No additional memory allocation beyond the returned substring is required.

## Worked Examples

| 1 | l | match | match | continue |
| 2 | o | match | mismatch with i | stop |

At index `2`, `"flight"` contains `'i'` instead of `'o'`.

Therefore, the algorithm returns:

```
"fl"
```

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(S) | Every character is compared at most once |
| Space | O(1) | Only a few variables are used |

The time complexity is linear in the total number of characters examined across all strings. In the worst case, every character of every string is checked exactly once before the algorithm finishes.

The space complexity is constant because the algorithm uses only a few variables and does not allocate additional data structures proportional to the input size.

## Edge Cases

One important edge case occurs when the input contains only a single string. For example:

```
["apple"]
```

In this situation, the entire string is automatically the longest common prefix because there are no other strings to compare against. The implementation handles this naturally because the inner loop never finds a mismatch, so the full first string is returned.

Another important case is when one string is empty:

```
["", "flower", "flow"]
```

An empty string cannot share any non-empty prefix with other strings. The algorithm correctly handles this because the outer loop runs zero times for the empty first string, causing the function to immediately return `""`.

A third edge case happens when one string is shorter but still fully matches the prefix:

```
["flow", "flower", "flight"]
```

The common prefix is `"fl"`. The implementation carefully checks whether the current index exceeds a string's length before accessing characters. This prevents out-of-bounds errors and correctly stops the prefix at the shortest valid boundary.

Another subtle case occurs when all strings are identical:

```
["test", "test", "test"]
```

Every character matches at every position, so the loops complete successfully and the full string `"test"` is returned.
