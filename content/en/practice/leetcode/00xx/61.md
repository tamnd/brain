## Solution

## Problem Understanding

This problem asks us to rotate a singly linked list to the **right** by `k` positions.

A right rotation means that every node shifts `k` places toward the end of the list, wrapping around when reaching the tail. More specifically, the last node becomes the new head after one rotation, and the previous nodes shift right accordingly.

For example:

```
1 -> 2 -> 3 -> 4 -> 5
```

After one right rotation:

```
5 -> 1 -> 2 -> 3 -> 4
```

After two right rotations:

```
4 -> 5 -> 1 -> 2 -> 3
```

The input consists of:

- `head`, the first node of a singly linked list
- `k`, the number of right rotations to perform

The output should be the head of the newly rotated linked list.

The constraints provide several important insights:

- The number of nodes is at most `500`, which is relatively small.
- However, `k` can be as large as `2 * 10^9`, which is enormous.
- This means repeatedly rotating one step at a time would be inefficient for very large `k`.

A crucial observation is that rotating a list by its own length results in the exact same list. For example, rotating a list of length `5` by `5`, `10`, or `15` changes nothing. Therefore, only `k % length` actually matters.

Several edge cases must be handled carefully:

- An empty list (`head = None`)
- A single-node list
- `k = 0`
- `k` larger than the list length
- `k` being an exact multiple of the list length

If these cases are not handled properly, pointer manipulations may break the linked list or produce incorrect rotations.

## Approaches

### Brute Force Approach

A straightforward solution is to rotate the list one step at a time, repeating the process `k` times.

For each rotation:

1. Traverse to the second-to-last node.
2. Remove the last node.
3. Move the last node to the front.
4. Repeat `k` times.

This works because each operation correctly performs one right rotation.

However, this approach is inefficient. Every single rotation requires traversing nearly the entire linked list, which costs `O(n)` time. Repeating this `k` times results in `O(n * k)` complexity.

Since `k` can be as large as `2 * 10^9`, this approach becomes impractical.

### Key Insight for an Optimal Solution

Instead of rotating one step at a time, we can determine the final structure directly.

The key observations are:

1. Rotating by the list length changes nothing.
2. We only need `k % n` effective rotations.
3. A rotation can be viewed as splitting the list into two parts and reconnecting them.

For example:

```
1 -> 2 -> 3 -> 4 -> 5
k = 2
```

After rotation:

```
4 -> 5 -> 1 -> 2 -> 3
```

Notice what happened:

- The last `2` nodes moved to the front.
- The remaining prefix stayed in order.

If the list length is `n`, then the new tail will be at:

```
n - (k % n) - 1
```

One elegant trick is to temporarily make the list **circular**:

```
1 -> 2 -> 3 -> 4 -> 5
^                   |
|___________________|
```

Then we simply locate the new tail, break the circle, and return the new head.

This avoids repeated rotations entirely.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × k) | O(1) | Rotate one step at a time |
| Optimal | O(n) | O(1) | Convert to circular list and reconnect |

## Algorithm Walkthrough

1. **Handle trivial cases immediately**

If the list is empty, has only one node, or `k = 0`, there is nothing to rotate. Return the original head immediately.
2. **Find the length of the list and the tail**

Traverse the linked list once to count the total number of nodes. While doing this, keep track of the last node.

This gives us:

- `length`
- `tail`
3. **Reduce unnecessary rotations**

Since rotating by the full length returns the same list:

```
effective_rotation = k % length
```

If the remainder becomes `0`, the list remains unchanged.
4. **Create a circular linked list**

Connect the tail back to the head:

```
tail.next = head
```

This temporarily turns the list into a cycle.
5. **Find the new tail**

The new tail should be:

```
length - effective_rotation - 1
```

positions from the original head.

Move a pointer this many steps.
6. **Identify the new head**

The node immediately after the new tail becomes the new head:

```
new_head = new_tail.next
```
7. **Break the cycle**

Disconnect the circle:

```
new_tail.next = None
```

This restores a proper singly linked list.
8. **Return the new head**

## Python Solution

```
# Definition for singly-linked list.
# class ListNode:
#     def __init__(self, val=0, next=None):
#         self.val = val
#         self.next = next

from typing import Optional

class Solution:
    def rotateRight(
        self,
        head: Optional[ListNode],
        k: int
    ) -> Optional[ListNode]:

        # Edge cases
        if not head or not head.next or k == 0:
            return head

        # Find length and tail
        length = 1
        tail = head

        while tail.next:
            tail = tail.next
            length += 1

        # Reduce unnecessary rotations
        k %= length

        # No rotation needed
        if k == 0:
            return head

        # Make list circular
        tail.next = head

        # Find new tail
        steps_to_new_tail = length - k - 1
        new_tail = head

        for _ in range(steps_to_new_tail):
            new_tail = new_tail.next

        # Set new head
        new_head = new_tail.next

        # Break the circle
        new_tail.next = None

        return new_head
```

The implementation begins by handling trivial edge cases. If the list is empty, contains only one node, or `k` is zero, rotation is unnecessary.

Next, we traverse the list once to determine its length and identify the tail node. This traversal is essential because we need the list size to compute the effective number of rotations.

After obtaining the length, we compute `k % length`. This prevents unnecessary work when `k` exceeds the list size.

The algorithm then temporarily converts the linked list into a circular structure by connecting the tail to the head. This simplifies pointer manipulation because we no longer need to manually detach and reattach nodes.

We compute the position of the new tail using:

```
length - k - 1
```

The node after this becomes the new head. Finally, we break the cycle by setting `new_tail.next = None`, restoring a proper singly linked list.

## Go Solution

```
/**
 * Definition for singly-linked list.
 * type ListNode struct {
 *     Val int
 *     Next *ListNode
 * }
 */
func rotateRight(head *ListNode, k int) *ListNode {

	if head == nil || head.Next == nil || k == 0 {
		return head
	}

	// Find length and tail
	length := 1
	tail := head

	for tail.Next != nil {
		tail = tail.Next
		length++
	}

	// Reduce unnecessary rotations
	k %= length

	// No rotation needed
	if k == 0 {
		return head
	}

	// Make circular list
	tail.Next = head

	// Find new tail
	stepsToNewTail := length - k - 1
	newTail := head

	for i := 0; i < stepsToNewTail; i++ {
		newTail = newTail.Next
	}

	// Set new head
	newHead := newTail.Next

	// Break cycle
	newTail.Next = nil

	return newHead
}
```

The Go implementation follows the exact same algorithmic structure as the Python version. The primary difference is how pointers are handled.

In Go, `nil` is used instead of Python's `None`, and linked list traversal uses pointer references directly. Since integers in Go are fixed-width, there is no concern about overflow here because `k ≤ 2 * 10^9`, which safely fits within Go's standard integer range on modern systems.

The logic for circular linking, locating the new tail, and breaking the cycle remains identical.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | One traversal for length, one traversal to locate new tail |
| Space | O(1) | Only a few pointer variables are used |

The algorithm runs in linear time because each node is visited a constant number of times. First, we traverse the list to compute the length and identify the tail. Then we traverse part of the list again to locate the new tail. Since both traversals together still scale linearly with `n`, the total complexity remains `O(n)`.

The space complexity is constant because we only maintain a few pointers and integer variables. No additional data structures proportional to input size are used.

## Edge Cases

### Empty List

If `head = None`, there are no nodes to rotate. A naive implementation might attempt pointer traversal and crash due to null access. Our implementation immediately returns the original `head` when the list is empty.

### Single Node List

For a list like:

```
[1]
```

any number of rotations produces the same result. A careless implementation may accidentally create a self-cycle and never break it correctly. We avoid this by returning early when `head.next` is `None`.

### Rotation Count Is a Multiple of List Length

Consider:

```
[1,2,3]
k = 6
```

Since:

```
6 % 3 = 0
```

the list should remain unchanged. Without the modulo optimization, unnecessary pointer manipulation could occur and introduce bugs. Our solution explicitly checks `k == 0` after modulo reduction and returns the original list.

### Very Large `k`

Since `k` can be as large as `2 * 10^9`, repeatedly rotating is infeasible. For example:

```
[1,2,3]
k = 2,000,000,000
```

Using modulo reduces the work dramatically:

```
2,000,000,000 % 3 = 2
```

Only two effective rotations are required, keeping the solution efficient.
