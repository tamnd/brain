## Solution

## Problem Understanding

This problem asks us to modify a sorted integer array **in-place** so that every unique value appears only once. Since the array is already sorted in **non-decreasing order**, duplicate values will always appear next to each other. That property is the key observation that makes an efficient solution possible.

We are given an integer array `nums`, and our task is to remove duplicates without allocating another array for the final answer. Instead of returning the modified array itself, we return an integer `k`, which represents the number of unique elements.

After the operation is complete, the first `k` positions in `nums` must contain all unique values in sorted order. Anything beyond index `k - 1` does not matter and will be ignored by the judge. This means we only care about correctly arranging the beginning portion of the array.

For example, if the input is `[1,1,2]`, we should transform the beginning of the array into `[1,2,...]` and return `2`. The remaining values do not matter because the judge ignores everything after index `k - 1`.

The constraints tell us several important things:

- `1 <= nums.length <= 3 * 10^4` means the array can contain up to 30,000 elements.
- The array is guaranteed to be sorted in non-decreasing order.
- Values range between `-100` and `100`, but the actual values are not especially important because the sorted order matters more than the numeric range.

Since the array may be relatively large, repeatedly shifting elements or creating expensive operations would be inefficient. We should aim for a linear-time solution.

There are several edge cases worth identifying early:

- An array with only one element, such as `[5]`, already contains only unique elements.
- An array where every value is the same, such as `[2,2,2,2]`, should return `1`.
- An array with no duplicates, such as `[1,2,3,4]`, should return the full length.
- Arrays containing negative numbers still work normally because sorting guarantees duplicates remain adjacent.

## Approaches

### Brute Force Approach

A straightforward solution is to create a separate data structure to store unique values while iterating through the array.

Because the array is sorted, we could scan from left to right and insert values into a temporary list only when they differ from the previously stored value. After collecting unique elements, we would copy them back into `nums` and return the count.

This approach works because duplicates are adjacent in a sorted array, so comparing against the previous value is sufficient to detect repetition.

However, this solution violates the spirit of the problem because it uses extra memory. The problem explicitly asks us to modify the array **in-place**, which strongly suggests we should avoid allocating another array.

### Optimal Approach, Two Pointers

The key insight is that because the array is sorted, duplicates always appear consecutively. That means we only need to compare each element with the most recent unique element.

We can use a **two-pointer technique**:

- One pointer tracks where the next unique value should be written.
- Another pointer scans through the array.

Whenever we find a new unique number, we place it into the next available position and move the write pointer forward.

This works because the left portion of the array gradually becomes the final answer. We never need additional memory, and every element is processed exactly once.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Uses an extra array to store unique values before copying back |
| Optimal | O(n) | O(1) | Uses two pointers and modifies the array in-place |

## Algorithm Walkthrough

1. Start by handling the simplest case. Since the array has at least one element according to the constraints, we can assume the first value is always unique.
2. Create a pointer called `write_index` and initialize it to `1`. This pointer tracks where the next unique element should be written.
3. Start iterating through the array from index `1` onward using another pointer.
4. At each step, compare the current number with the previous number.
5. If the current number is different from the previous number, it means we found a new unique value. Write this value into `nums[write_index]`.
6. Increment `write_index` because the next unique value should go into the next slot.
7. Continue scanning until the end of the array.
8. Return `write_index`, which represents the total number of unique elements.

The reason this works is that the sorted property guarantees all duplicates appear consecutively. If the current element differs from the previous one, it must be a new unique number.

For example:

`[0,0,1,1,1,2,2,3,3,4]`

We gradually overwrite duplicate positions:

- Keep `0`
- Skip duplicate `0`
- Keep `1`
- Skip duplicate `1`
- Skip duplicate `1`
- Keep `2`
- Keep `3`
- Keep `4`

Final useful portion becomes:

`[0,1,2,3,4,...]`

## Python Solution

```
from typing import List

class Solution:
    def removeDuplicates(self, nums: List[int]) -> int:
        write_index = 1

        for read_index in range(1, len(nums)):
            if nums[read_index] != nums[read_index - 1]:
                nums[write_index] = nums[read_index]
                write_index += 1

        return write_index
```

The implementation follows the exact algorithm described earlier. The variable `write_index` represents the position where the next unique value should be placed.

We begin iterating from index `1` because the first element is always unique in a non-empty sorted array. For each element, we compare it with the previous element.

If the current value differs from the previous one, we copy it into the next available unique slot and increment `write_index`.

By the time the loop finishes, the first `write_index` positions contain all unique values in sorted order. Returning `write_index` gives the correct number of unique elements.

## Go Solution

```
func removeDuplicates(nums []int) int {
    writeIndex := 1

    for readIndex := 1; readIndex < len(nums); readIndex++ {
        if nums[readIndex] != nums[readIndex-1] {
            nums[writeIndex] = nums[readIndex]
            writeIndex++
        }
    }

    return writeIndex
}
```

The Go implementation mirrors the Python approach almost exactly. Go slices are mutable, so modifying `nums` directly updates the original array in-place.

Unlike Python, Go does not require type hints because types are explicitly declared. We use `:=` for variable initialization and standard `for` loops for iteration.

There are no concerns about integer overflow here because the maximum array length is only `30,000`, which comfortably fits inside Go's integer type.

The problem guarantees at least one element exists, so we do not need special handling for an empty slice.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | We scan the array exactly once |
| Space | O(1) | Only a few variables are used, no extra data structure is allocated |

The time complexity is **O(n)** because every element is visited once during the scan. No nested loops or repeated shifting operations occur.

The space complexity is **O(1)** because we modify the array in-place and only use a small fixed number of variables, regardless of input size.

## Edge Cases

### Single Element Array

Consider an input like `[7]`. Since there is only one element, it is automatically unique. The loop never executes because iteration starts from index `1`, and `write_index` remains `1`. The function correctly returns `1`.

### All Elements Are Duplicates

For an input like `[2,2,2,2]`, only the first value should remain. Every later element matches the previous one, so `write_index` never increments. The algorithm correctly returns `1`, and the first element remains `2`.

### No Duplicates Exist

For an input like `[1,2,3,4]`, every element is unique. Each comparison succeeds, so every value gets written back into its own position. The returned value equals the original array length, which is correct.

### Negative Numbers and Mixed Values

An input such as `[-3,-3,-1,0,0,2]` still works correctly because the algorithm only depends on sorted order, not numeric magnitude. Duplicates remain adjacent, allowing the comparison logic to identify unique values without any special handling.
