# LeetCode 70, Climbing Stairs

## Problem Understanding

The problem describes a staircase with `n` total steps, and you want to determine how many distinct ways exist to reach the top. At every move, you are allowed to climb either `1` step or `2` steps.

The input is a single integer `n`, representing the total number of stairs. The output is another integer, representing the total number of unique combinations of moves that exactly reach the top step.

The important detail is that order matters. For example, climbing `1` step followed by `2` steps is considered different from climbing `2` steps followed by `1` step.

For `n = 3`, the valid ways are:

- `1 + 1 + 1`
- `1 + 2`
- `2 + 1`

So the answer is `3`.

The constraints are:

- `1 <= n <= 45`

This constraint is small enough that several approaches are possible, including recursion, memoization, and dynamic programming. However, naive recursion becomes inefficient because it repeatedly recomputes the same subproblems.

An important observation is that the problem naturally breaks into overlapping subproblems. To reach step `n`, the final move must come from either:

- step `n - 1` using a `1` step move
- step `n - 2` using a `2` step move

This leads directly to a Fibonacci-style recurrence relation.

Several edge cases are important:

- `n = 1`, there is only one possible way
- `n = 2`, there are exactly two ways
- Small values can expose incorrect base cases
- Recursive solutions without memoization can become extremely slow because the same calculations repeat many times

The problem guarantees that `n` is always positive, so we do not need to handle zero or negative inputs.

## Approaches

### Brute Force Recursive Approach

The most straightforward solution is to recursively try every possible move.

At any step, you can either:

- climb `1` stair
- climb `2` stairs

So for a staircase of size `n`, the total number of ways is:

- ways to climb `n - 1` stairs
- plus ways to climb `n - 2` stairs

This gives the recurrence:

$$f(n) = f(n-1) + f(n-2)$$

The recursive solution is correct because every valid path to step `n` must end with either a `1` step jump from `n - 1` or a `2` step jump from `n - 2`.

However, this approach is very inefficient because the recursion tree repeatedly recalculates the same values. For example, `f(4)` may be computed many times while evaluating larger inputs.

The time complexity becomes exponential.

### Optimal Dynamic Programming Approach

The key insight is that this problem is identical to generating Fibonacci numbers.

If:

- `dp[i]` represents the number of ways to reach step `i`

then:

$$dp[i] = dp[i-1] + dp[i-2]$$

Instead of recomputing values recursively, we can build the answer iteratively from smaller subproblems.

Even better, we only need the previous two values at any time, so we can optimize space usage to constant space.

This approach is efficient because:

- each subproblem is solved exactly once
- every computation uses already computed results
- no recursion overhead exists

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(n) | Recursive exploration of all possibilities, repeated calculations make it slow |
| Optimal | O(n) | O(1) | Iterative Fibonacci-style dynamic programming using two variables |

## Algorithm Walkthrough

1. Handle the base cases first.

If `n` is `1`, there is only one way to climb the stairs.

If `n` is `2`, there are two possible ways.
2. Store the number of ways for the two previous steps.

We use:

- `one_step_before` for ways to reach `n - 1`
- `two_steps_before` for ways to reach `n - 2`

Initially:

- step `1` has `1` way
- step `2` has `2` ways
3. Iterate from step `3` up to step `n`.

For each step:

- calculate current ways as the sum of previous two values
- update variables for the next iteration
4. Return the final computed value.

After the loop finishes, the variable holding the current step count contains the answer.

This works because every staircase configuration must end in either:

- a single step from the previous stair
- a double step from two stairs below

So combining those counts gives the total number of valid paths.

## Python Solution

```
class Solution:
    def climbStairs(self, n: int) -> int:
        if n == 1:
            return 1
        
        if n == 2:
            return 2
        
        two_steps_before = 1
        one_step_before = 2
        
        for _ in range(3, n + 1):
            current = one_step_before + two_steps_before
            two_steps_before = one_step_before
            one_step_before = current
        
        return one_step_before
```

The implementation starts by handling the smallest valid inputs directly. These base cases are necessary because the recurrence depends on the previous two states.

Next, two variables are initialized:

- `two_steps_before` stores ways to reach step `1`
- `one_step_before` stores ways to reach step `2`

The loop starts from step `3` because the first two values are already known.

At each iteration:

```
current = one_step_before + two_steps_before
```

This computes the number of ways to reach the current step.

Then the variables shift forward:

```
two_steps_before = one_step_before
one_step_before = current
```

This maintains only the two most recent states, which keeps space complexity constant.

Finally, `one_step_before` contains the answer for step `n`.

## Go Solution

```
func climbStairs(n int) int {
    if n == 1 {
        return 1
    }

    if n == 2 {
        return 2
    }

    twoStepsBefore := 1
    oneStepBefore := 2

    for step := 3; step <= n; step++ {
        current := oneStepBefore + twoStepsBefore
        twoStepsBefore = oneStepBefore
        oneStepBefore = current
    }

    return oneStepBefore
}
```

The Go implementation follows the same logic as the Python solution.

One difference is variable declaration syntax. Go uses explicit variable initialization with `:=`.

Go integers are large enough for this problem because the constraint `n <= 45` ensures the result fits safely within a standard integer type.

The solution also avoids recursion, so there is no risk of stack overflow or recursive overhead.

## Edge Cases

### Edge Case 1, Minimum Input

When `n = 1`, there is only one possible way to reach the top.

A common mistake is forgetting to handle this separately, which can cause incorrect indexing or invalid recurrence calculations.

The implementation explicitly checks:

```
if n == 1:
    return 1
```

so the smallest valid input works correctly.

### Edge Case 2, Small Fibonacci Base

When `n = 2`, there are exactly two ways:

- `1 + 1`
- `2`

If the base cases are incorrect, the entire recurrence becomes incorrect because all future values depend on these starting values.

The implementation correctly returns `2` immediately.

### Edge Case 3, Larger Inputs

For larger values like `n = 45`, a brute force recursive solution becomes extremely slow because it repeatedly recalculates identical subproblems.

The iterative dynamic programming solution avoids this issue completely by computing each state exactly once.

This guarantees efficient execution even at the upper constraint limit.
