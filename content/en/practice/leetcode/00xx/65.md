## Solution

## Problem Understanding

This problem asks us to determine whether a given string represents a **valid numeric value** according to a precise grammar. Unlike simply checking whether a string can be converted into a number using a programming language's built in parser, we must manually validate the string based on the rules described in the problem statement.

The input is a single string `s`, and the output is a boolean value. We return `true` if the string satisfies the definition of a valid number, otherwise `false`.

A valid number can be one of two broad categories:

1. An **integer**, optionally followed by an exponent.
2. A **decimal**, optionally followed by an exponent.

A valid integer may include an optional leading sign (`+` or `-`) followed by at least one digit. Examples include `"42"`, `"-7"`, and `"+100"`.

A valid decimal may include an optional sign and must follow one of these formats:

- Digits followed by a decimal point: `"4."`
- Digits, decimal point, digits: `"3.14"`
- Decimal point followed by digits: `".9"`

An exponent section starts with `'e'` or `'E'`, and must be followed by a valid integer. Examples include `"2e10"` and `"3e+7"`.

The constraints are relatively small, `1 <= s.length <= 20`, which means performance is not the main challenge. The real difficulty comes from correctly handling many subtle edge cases and combinations of signs, decimal points, and exponents.

Several tricky inputs can easily break a naive implementation:

- `"."` is invalid because a decimal point alone is not enough, at least one digit must exist.
- `"1e"` is invalid because an exponent requires a number after it.
- `"e3"` is invalid because the base number is missing.
- `"99e2.5"` is invalid because the exponent must be an integer, not a decimal.
- `"--6"` is invalid because multiple signs are not allowed.
- `"4."` is valid because digits before a decimal point are enough.
- `"-.9"` is valid because digits after the decimal point exist.

The problem guarantees that the string only contains letters, digits, signs, and dots, so we do not need to handle spaces or arbitrary special characters.

## Approaches

### Brute Force Approach

A brute force solution would attempt to generate every valid number pattern and compare the input string against them. Since numbers can contain optional signs, optional decimal points, and optional exponents, we could imagine recursively enumerating all valid grammar structures and checking whether the string matches.

Another brute force variation would be to split the problem into many conditional checks, manually validating every possible arrangement of symbols. For example, we could separately handle numbers with exponents, numbers with decimal points, signed numbers, unsigned numbers, and so on.

This approach becomes difficult to maintain and error prone because there are many overlapping cases. For example:

- `"4."` is valid
- `"."` is invalid
- `"3e+7"` is valid
- `"1e"` is invalid

Manually enumerating all cases quickly becomes messy and difficult to reason about.

### Key Insight for the Optimal Solution

The key observation is that this problem is essentially a **finite state validation problem**. Instead of hardcoding every valid pattern, we can scan the string once while tracking a few important conditions.

We only need to know:

- Have we seen at least one digit?
- Have we already seen a decimal point?
- Have we already seen an exponent?

While iterating through the string:

- Digits are always acceptable.
- A sign (`+` or `-`) is only valid at the beginning or immediately after an exponent.
- A decimal point is only valid once, and cannot appear after an exponent.
- An exponent is only valid once, and must follow a valid number.
- The exponent must itself contain digits.

This transforms the problem into a clean linear scan with a handful of boolean flags.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Requires many case splits or grammar matching |
| Optimal | O(n) | O(1) | Single pass using validation flags |

## Algorithm Walkthrough

1. Initialize three boolean variables:

- `seen_digit` tracks whether we have encountered at least one digit.
- `seen_dot` tracks whether a decimal point has appeared.
- `seen_exponent` tracks whether an exponent (`e` or `E`) has appeared.
2. Traverse the string character by character.
3. If the current character is a digit:

- Mark `seen_digit = True`.
- Digits are always allowed.
4. If the character is `'+'` or `'-'`:

- It is only valid if:

- It appears at the beginning of the string, or
- It immediately follows `'e'` or `'E'`.
- Otherwise, return `False`.
5. If the character is `'.'`:

- Reject it if:

- We already saw a decimal point, or
- We already saw an exponent.
- Otherwise, mark `seen_dot = True`.
6. If the character is `'e'` or `'E'`:

- Reject it if:

- We already saw an exponent, or
- No digit appeared before it.
- Otherwise:

- Mark `seen_exponent = True`.
- Reset `seen_digit = False`, because the exponent must contain digits afterward.
7. If any other character appears:

- Return `False`.
8. After processing the string, return `seen_digit`.

This final check is important because inputs like `"1e"` should fail. When we encounter `'e'`, we reset `seen_digit` and require digits afterward.

## Python Solution

```
class Solution:
    def isNumber(self, s: str) -> bool:
        seen_digit = False
        seen_dot = False
        seen_exponent = False

        for index, char in enumerate(s):
            if char.isdigit():
                seen_digit = True

            elif char in "+-":
                if index > 0 and s[index - 1] not in "eE":
                    return False

            elif char == ".":
                if seen_dot or seen_exponent:
                    return False
                seen_dot = True

            elif char in "eE":
                if seen_exponent or not seen_digit:
                    return False

                seen_exponent = True
                seen_digit = False

            else:
                return False

        return seen_digit
```

The implementation closely follows the algorithm described above.

We begin by initializing three flags. These flags summarize everything we need to know about the string as we process it.

The loop iterates through each character exactly once. Whenever we encounter a digit, we simply mark `seen_digit = True`.

For signs (`+` or `-`), we validate their position carefully. A sign is only legal at the beginning or immediately after an exponent marker.

When we encounter a decimal point, we ensure it has not already appeared and that we have not entered the exponent section. This prevents invalid formats such as `"1.2.3"` or `"1e2.5"`.

Exponent handling is the most subtle part. We only allow one exponent, and it must appear after at least one digit. Once an exponent is seen, we reset `seen_digit` to `False` because the exponent portion must itself contain digits. This elegantly handles invalid inputs such as `"1e"`.

Finally, we return `seen_digit`. If the string ends without valid digits in the required position, the result is `False`.

## Go Solution

```
func isNumber(s string) bool {
	seenDigit := false
	seenDot := false
	seenExponent := false

	for i, ch := range s {
		if ch >= '0' && ch <= '9' {
			seenDigit = true

		} else if ch == '+' || ch == '-' {
			if i > 0 && s[i-1] != 'e' && s[i-1] != 'E' {
				return false
			}

		} else if ch == '.' {
			if seenDot || seenExponent {
				return false
			}
			seenDot = true

		} else if ch == 'e' || ch == 'E' {
			if seenExponent || !seenDigit {
				return false
			}

			seenExponent = true
			seenDigit = false

		} else {
			return false
		}
	}

	return seenDigit
}
```

The Go implementation mirrors the Python logic almost exactly. Since Go does not provide Python's `isdigit()` helper, digit checking is performed manually using ASCII comparisons (`'0' <= ch <= '9'`).

One small implementation detail is indexing. We use `range` to iterate over the string, but since the input contains only ASCII characters, indexing with `s[i-1]` remains safe.

No integer overflow concerns exist because we are never parsing the numeric value itself, we only validate syntax.

The algorithm is linear because every character is processed once, and each operation inside the loop takes constant time. Space complexity is constant because no additional data structures grow with input size.

## Edge Cases

One important edge case is a standalone decimal point such as `"."`. A naive implementation might accept it because decimal numbers can contain a dot. However, the definition requires at least one digit somewhere in the number. Our implementation correctly rejects this case because `seen_digit` never becomes `True`.

Another tricky case is incomplete exponents such as `"1e"` or `"2E+"`. These can be mistakenly accepted if the exponent marker is validated but digits afterward are not enforced. By resetting `seen_digit = False` after encountering an exponent, we guarantee that digits must appear later for the string to be valid.

A third subtle case involves misplaced signs, such as `"--6"` or `"1+2"`. Signs are only legal at the beginning of the string or immediately after an exponent. Our implementation explicitly checks the preceding character and rejects any invalid placement.

Finally, numbers with trailing decimal points, such as `"4."`, can easily be mishandled. Some implementations incorrectly require digits after the decimal point. Our solution accepts `"4."` because digits before the decimal point are sufficient, matching the problem definition exactly.
