## Solution

## Problem Understanding

This problem asks us to generate a valid **n-bit Gray code sequence**. A Gray code is a sequence of binary numbers where every pair of adjacent values differs by **exactly one bit**. Additionally, the sequence must be **cyclic**, meaning the first and last values must also differ by exactly one bit.

The input is a single integer `n`, representing the number of bits in each binary number. Since each bit can either be `0` or `1`, there are exactly `2^n` unique binary values representable using `n` bits. Therefore, the output must contain exactly `2^n` integers, each corresponding to a unique `n-bit` value.

The sequence must satisfy several conditions:

1. The sequence must begin with `0`.
2. Every integer must lie in the range `[0, 2^n - 1]`.
3. No value may appear more than once.
4. Every adjacent pair must differ by exactly one bit.
5. The first and last numbers must also differ by exactly one bit.

The problem guarantees that a valid Gray code sequence always exists, and importantly, it asks for **any valid sequence**, not necessarily a unique one. This flexibility allows us to focus on generating one mathematically valid construction rather than exploring every possibility.

The constraints are relatively small:

- `1 <= n <= 16`

Since `n` is at most `16`, the maximum output size is `2^16 = 65,536` elements. This immediately tells us that any solution must at least spend `O(2^n)` time because we must generate every number in the sequence. Exponential output size is unavoidable here.

Several edge cases are important to consider. The smallest valid input is `n = 1`, where only two values exist, `[0, 1]`. Another important consideration is larger values like `n = 16`, where efficiency matters because we generate over sixty thousand entries. We also need to ensure that the cyclic property holds automatically, because a naive sequence of binary numbers such as `[0,1,2,3,...]` does not satisfy the one-bit difference requirement.

## Approaches

### Brute Force Approach

A brute-force strategy would attempt to construct the Gray code sequence using **backtracking**. We could begin with `0`, then repeatedly search for an unused number that differs by exactly one bit from the current number. At each step, we try all valid candidates and recursively continue until a sequence of length `2^n` is formed.

To verify whether two numbers differ by one bit, we can use XOR. If `a ^ b` contains exactly one set bit, then the numbers differ in exactly one position.

This approach is correct because it explicitly explores valid transitions while ensuring uniqueness and adjacency requirements. If a complete sequence is found, it is guaranteed to satisfy all constraints.

However, this method is computationally expensive. The search space grows extremely quickly because we may need to try many combinations before finding a valid arrangement. In the worst case, the runtime approaches factorial or exponential backtracking complexity, making it impractical for `n = 16`.

### Optimal Approach, Mathematical Gray Code Formula

The key insight is that Gray codes have a well-known mathematical construction.

For any integer `i`, its Gray code representation can be generated using the formula:

$G(i)=i\oplus(i\gg1)$

This formula works because shifting `i` right by one aligns neighboring bits, and XORing with the original number ensures that consecutive integers differ by only one bit.

For example:

| Decimal `i` | Binary `i` | `i >> 1` | Gray Code |
| --- | --- | --- | --- |
| 0 | 000 | 000 | 000 |
| 1 | 001 | 000 | 001 |
| 2 | 010 | 001 | 011 |
| 3 | 011 | 001 | 010 |

Notice that every consecutive Gray code differs by exactly one bit.

Since we only need to iterate from `0` to `2^n - 1` and compute the Gray code for each number, this gives an elegant and optimal solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O((2^n)!) | O(2^n) | Backtracking over possible sequences, extremely expensive |
| Optimal | O(2^n) | O(2^n) | Uses Gray code formula `i ^ (i >> 1)` |

## Algorithm Walkthrough

1. Compute the total number of values needed as `2^n`. Since an `n-bit` number has `2^n` possible combinations, our result must contain exactly this many integers.
2. Create an empty list to store the Gray code sequence. We will generate values in order and append them directly.
3. Iterate through every integer `i` from `0` to `2^n - 1`. Each integer serves as an index into the Gray code construction.
4. Compute the Gray code for `i` using the formula `i ^ (i >> 1)`. The right shift aligns neighboring bits, and XOR ensures only one bit changes between consecutive values.
5. Append the computed Gray code to the result list. Since every value generated by this formula is unique and within the valid range, no extra validation is needed.
6. Return the completed list after processing all values.

This algorithm is efficient because it avoids recursive search entirely. Instead of constructing a sequence through trial and error, it directly computes a mathematically valid Gray code ordering.

## Python Solution

```
from typing import List

class Solution:
    def grayCode(self, n: int) -> List[int]:
        total_numbers = 1 << n
        gray_sequence = []

        for number in range(total_numbers):
            gray_code = number ^ (number >> 1)
            gray_sequence.append(gray_code)

        return gray_sequence
```

The implementation begins by calculating `1 << n`, which efficiently computes `2^n`. This determines how many numbers must exist in the final sequence.

We then initialize an empty list called `gray_sequence` to store results.

The main loop iterates through every integer from `0` to `2^n - 1`. For each value, we compute its Gray code representation using `number ^ (number >> 1)`.

The XOR operation compares the original number with its right-shifted version, producing the Gray code value. Each computed number is appended to the result list.

Finally, the completed sequence is returned.

This implementation directly mirrors the mathematical construction and guarantees correctness without requiring additional checks for adjacency or uniqueness.

## Go Solution

```
func grayCode(n int) []int {
	totalNumbers := 1 << n
	graySequence := make([]int, 0, totalNumbers)

	for number := 0; number < totalNumbers; number++ {
		grayCode := number ^ (number >> 1)
		graySequence = append(graySequence, grayCode)
	}

	return graySequence
}
```

The Go implementation follows the exact same logic as the Python version. We compute `2^n` using bit shifting and preallocate the slice capacity with `make([]int, 0, totalNumbers)` to reduce reallocations during appends.

Go slices automatically expand when needed, but preallocating capacity improves efficiency. Since `n <= 16`, integer overflow is not a concern because all values remain comfortably within Go's standard integer range.

Unlike Python lists, Go slices require explicit initialization, but otherwise the structure of the algorithm remains identical.

We first compute:


So we generate numbers from `0` to `3`.

| Iteration | `i` (Binary) | `i >> 1` | Gray Code Calculation | Result |

Final sequence:

```
[0, 1, 3, 2]
```

Binary representation:

```
00 → 01 → 11 → 10
```

Each adjacent pair differs by exactly one bit:

- `00 → 01`, one bit changes
- `01 → 11`, one bit changes
- `11 → 10`, one bit changes
- `10 → 00`, one bit changes

Thus, the sequence is valid.

## Edge Cases

### Smallest Input, `n = 1`

This is the smallest valid input and can expose bugs in implementations that assume multiple bits exist. The expected output is simply `[0,1]`. Our implementation handles this naturally because the loop runs for `2^1 = 2` iterations and correctly computes Gray codes using the same formula.

### Maximum Input, `n = 16`

For the largest allowed value, the sequence contains `65,536` numbers. A brute-force or backtracking approach would become infeasible at this scale. Our implementation remains efficient because each value is generated independently in constant time, resulting in a manageable `O(2^n)` runtime.

### Cyclic Property of First and Last Element

A common source of mistakes is forgetting that the first and last numbers must differ by exactly one bit. Simply generating numbers in binary order, such as `[0,1,2,3]`, fails this condition. The Gray code formula inherently guarantees the cyclic property, ensuring that the final sequence always satisfies the problem requirements.

### Duplicate Values

A naive construction might accidentally revisit previously used numbers. Since the formula `i ^ (i >> 1)` generates a unique mapping for every `i` in `[0, 2^n - 1]`, duplicates cannot occur. Every generated value is distinct and within the valid range.
