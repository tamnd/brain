# LeetCode 35 - Search Insert Position

## Problem Understanding

This problem gives us a **sorted array of distinct integers** and a target value. We need to determine where the target belongs in the array.

If the target already exists in the array, we return its exact index. If the target does not exist, we return the position where it would need to be inserted so that the array remains sorted in ascending order.

In other words, we are not just searching for whether the target exists, we are also finding the correct insertion position if it does not.

For example, consider:

```
nums = [1,3,5,6]
target = 2
```

The target `2` is not present. Since `2` belongs between `1` and `3`, the correct insertion index is `1`.

The input consists of:

- A sorted integer array `nums`
- An integer `target`

The expected output is a single integer representing either:

- The index where `target` is found, or
- The position where `target` should be inserted

The constraints are important:

- `nums.length` can be as large as `10^4`
- The array is already sorted in ascending order
- All values are distinct
- The required runtime complexity is `O(log n)`

The runtime requirement immediately rules out linear scanning as the intended solution. Since the array is sorted and logarithmic time is required, this strongly suggests using **binary search**.

Several edge cases are worth identifying upfront. The target may be smaller than every element, meaning it should be inserted at index `0`. The target may be larger than every element, meaning it belongs at the end of the array. The array could also contain only one element, which makes boundary handling especially important. Since the problem guarantees distinct values and sorted order, we do not need to handle duplicates or unsorted input.

## Approaches

### Brute Force Approach

A straightforward solution is to iterate through the array from left to right and compare each value with the target.

At every position:

- If we find the target, return its index.
- If the current number becomes larger than the target, return the current index because the target should be inserted before it.

If we finish scanning the array without finding a suitable position, the target must belong at the end of the array.

This approach works correctly because the array is sorted. The first number greater than or equal to the target determines the answer.

However, this method takes `O(n)` time because, in the worst case, we may examine every element. Since the problem explicitly requires `O(log n)` runtime complexity, this approach is too slow for the intended solution.

### Optimal Approach

The key observation is that the array is already **sorted**, which makes **binary search** applicable.

Binary search works by repeatedly dividing the search space in half.

At each step:

- Compute the middle index.
- If the middle value equals the target, return the index immediately.
- If the middle value is smaller than the target, discard the left half because the target can only appear to the right.
- If the middle value is greater than the target, discard the right half because the target must be to the left.

An important insight is that if the target does not exist, the final value of the left pointer represents the correct insertion position.

Why does this work? By the time the search ends, `left` points to the first location where the target could be placed without violating sorted order.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Linear scan until target or insertion point is found |
| Optimal | O(log n) | O(1) | Binary search halves the search space each iteration |

## Algorithm Walkthrough

1. Initialize two pointers, `left` and `right`. Set `left = 0` and `right = len(nums) - 1`. These pointers define the current search range.
2. Continue searching while `left <= right`. This condition ensures that there are still elements left to examine.
3. Compute the middle index using:

```
mid = left + (right - left) // 2
```

This avoids potential integer overflow in some languages and safely finds the midpoint.
4. Compare `nums[mid]` with `target`.

- If they are equal, we have found the target, so return `mid`.
- If `nums[mid] < target`, the target must be in the right half, so update `left = mid + 1`.
- If `nums[mid] > target`, the target must be in the left half, so update `right = mid - 1`.
5. Eventually, the loop ends when `left > right`.

At this point, the target was not found. The correct insertion position is exactly `left`, because it points to the first valid location where the target could be inserted while maintaining sorted order.

## Python Solution

```
from typing import List

class Solution:
    def searchInsert(self, nums: List[int], target: int) -> int:
        left = 0
        right = len(nums) - 1

        while left <= right:
            mid = left + (right - left) // 2

            if nums[mid] == target:
                return mid
            elif nums[mid] < target:
                left = mid + 1
            else:
                right = mid - 1

        return left
```

The implementation begins by initializing the search boundaries with `left` and `right`. These pointers represent the portion of the array that may still contain the target.

Inside the `while` loop, we compute the middle index and compare its value with the target. If the target is found, we immediately return the index.

When the middle value is smaller than the target, we move the `left` pointer to `mid + 1` because all elements before `mid` are guaranteed to be too small. Conversely, when the middle value is larger, we move the `right` pointer to `mid - 1` because everything after `mid` is too large.

If the loop exits without finding the target, the correct insertion position is stored in `left`. Returning `left` works for all insertion cases, including inserting at the beginning or the end of the array.

## Go Solution

```
func searchInsert(nums []int, target int) int {
	left := 0
	right := len(nums) - 1

	for left <= right {
		mid := left + (right-left)/2

		if nums[mid] == target {
			return mid
		} else if nums[mid] < target {
			left = mid + 1
		} else {
			right = mid - 1
		}
	}

	return left
}
```

The Go implementation follows the same logic as the Python version. The midpoint calculation uses:

```
left + (right-left)/2
```

This avoids potential overflow, although overflow is not a practical issue under this problem's constraints.

Go slices are used instead of Python lists, but indexing works similarly. Since the problem guarantees at least one element, we do not need special handling for an empty slice. The algorithm still correctly handles insertion at the beginning or end of the array.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(log n) | Each iteration cuts the search space roughly in half |
| Space | O(1) | Only a few variables are used regardless of input size |

The time complexity is `O(log n)` because binary search repeatedly halves the search interval. Instead of checking every element, the algorithm eliminates half of the remaining candidates after each comparison.

The space complexity is `O(1)` because no additional data structures are created. The algorithm only uses a fixed number of variables such as `left`, `right`, and `mid`.

## Edge Cases

### Target Smaller Than All Elements

Consider:

```
nums = [3,5,7]
target = 1
```

A naive implementation might incorrectly return `-1` or mishandle boundary conditions. In this case, the target belongs before the first element. The algorithm handles this naturally because `left` never moves from `0`, so the final insertion index is `0`.

### Target Larger Than All Elements

Consider:

```
nums = [1,3,5]
target = 10
```

This case is easy to mishandle if the implementation assumes the target must exist somewhere inside the array. The binary search eventually exhausts the search space, leaving `left = len(nums)`. Returning `left` correctly places the target at the end.

### Single Element Array

Consider:

```
nums = [5]
target = 2
```

or

```
nums = [5]
target = 5
```

Single element arrays often expose off by one errors in pointer logic. The condition `left <= right` ensures the lone element is checked properly. If the target exists, its index is returned. Otherwise, `left` naturally becomes the correct insertion position.

### Target Between Two Existing Elements

Consider:

```
nums = [1,4,6,8]
target = 5
```

The target does not exist but belongs between `4` and `6`. A naive implementation could return the wrong neighboring index. Binary search guarantees correctness because `left` eventually settles at the first valid insertion location, which is index `2`.
