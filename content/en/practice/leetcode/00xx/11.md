## Solution

## Problem Understanding

The problem gives an array called `height`, where each element represents the height of a vertical line drawn on a coordinate plane. The position of each line is determined by its index in the array. For example, if `height[3] = 5`, then there is a vertical line at x-coordinate `3` with height `5`.

Any two lines can form a container together with the x-axis. The amount of water this container can hold depends on two factors:

- The width between the two lines
- The shorter of the two heights

The width is the horizontal distance between the indices. If the two chosen lines are at indices `i` and `j`, then the width is `j - i`.

The height of the container is limited by the shorter line because water would spill over the shorter boundary. Therefore, the container area is calculated as:

$\text{Area} = \min(height[i], height[j]) \times (j - i)$

The task is to find the pair of lines that produces the maximum possible area.

The constraints are important because they determine which algorithms are feasible. The array length can be as large as `10^5`, which means an `O(n^2)` solution may perform up to `10^10` operations in the worst case. That is far too slow for LeetCode time limits. This strongly suggests that we need a linear or near-linear solution.

The problem guarantees that:

- There are at least two lines
- Heights are non-negative integers
- The container walls are vertical only, no slanting is allowed

Several edge cases are important to recognize early:

- Arrays with only two elements
- Heights containing zeros
- Multiple equal heights
- The maximum area not necessarily involving the tallest line
- Cases where moving the wrong pointer could skip the optimal answer

Understanding why the shorter line determines the area is the key insight that leads to the optimal two-pointer solution.

## Approaches

### Brute Force Approach

The most direct solution is to check every possible pair of lines.

For every index `i`, we can pair it with every index `j > i`. For each pair, we calculate:

- Width as `j - i`
- Height as `min(height[i], height[j])`
- Area as `width * height`

We keep track of the maximum area seen so far.

This approach is guaranteed to work because it evaluates every possible container. Since the optimal pair must exist among all pairs, brute force will eventually find it.

However, the performance is poor. If there are `n` lines, then there are approximately:

$\frac{n(n-1)}{2}$

possible pairs to evaluate.

With `n = 10^5`, this becomes far too large.

### Optimal Two-Pointer Approach

The optimal solution uses the two-pointer technique.

We begin with two pointers:

- One at the leftmost index
- One at the rightmost index

This gives the maximum possible width initially.

At each step:

1. Compute the current container area
2. Update the maximum area if necessary
3. Move the pointer pointing to the shorter line inward

The critical insight is understanding why moving the shorter line is safe.

Suppose the left line is shorter. The current area is limited by that left height. Even if we keep the same left line and move the right pointer inward, the width decreases. Since the limiting height does not improve, the area cannot become larger.

Therefore, the only chance of finding a larger area is to move away from the shorter line and hope to find a taller boundary.

This greedy observation allows us to eliminate many impossible candidates without checking every pair.

Because each pointer moves at most `n` times, the algorithm runs in linear time.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Checks every possible pair of lines |
| Optimal Two Pointers | O(n) | O(1) | Uses greedy pointer movement to eliminate unnecessary checks |

## Algorithm Walkthrough

1. Initialize two pointers, `left` at index `0` and `right` at index `n - 1`.

These two lines form the widest possible container initially. Starting with the maximum width is important because width decreases as pointers move inward.
2. Initialize a variable called `max_area` to store the best result found so far.

This variable will be updated whenever we discover a larger valid container.
3. While `left < right`, calculate the current container area.

The width is:

$\text{width} = right - left$

The usable height is:

$\text{container height} = \min(height[left], height[right])$

Then compute the area.
4. Compare the current area with `max_area`.

If the current area is larger, update `max_area`.
5. Move the pointer corresponding to the shorter line.

If `height[left] < height[right]`, increment `left`.

Otherwise, decrement `right`.

This is the core greedy step. Since the shorter line limits the current area, keeping it while shrinking the width cannot improve the result.
6. Continue until the pointers meet.

Once `left == right`, there are no more valid containers to evaluate.
7. Return `max_area`.

This value represents the maximum water any container can hold.

## Python Solution

```
from typing import List

class Solution:
    def maxArea(self, height: List[int]) -> int:
        left = 0
        right = len(height) - 1

        max_area = 0

        while left < right:
            width = right - left
            current_height = min(height[left], height[right])
            current_area = width * current_height

            max_area = max(max_area, current_area)

            if height[left] < height[right]:
                left += 1
            else:
                right -= 1

        return max_area
```

The implementation begins by placing two pointers at opposite ends of the array. This ensures we start with the maximum possible width.

Inside the loop, the algorithm computes the current container width and the limiting height. The area is then calculated and compared against the running maximum.

The next step is the greedy decision. If the left height is smaller, the left pointer moves inward because the current left boundary cannot help produce a larger area with any smaller width. Similarly, if the right height is smaller or equal, the right pointer moves inward.

The loop continues until both pointers meet, meaning every useful candidate configuration has been considered.

The algorithm uses only a few integer variables, so the memory usage remains constant throughout execution.

## Go Solution

```
func maxArea(height []int) int {
    left := 0
    right := len(height) - 1

    maxArea := 0

    for left < right {
        width := right - left

        currentHeight := height[left]
        if height[right] < currentHeight {
            currentHeight = height[right]
        }

        currentArea := width * currentHeight

        if currentArea > maxArea {
            maxArea = currentArea
        }

        if height[left] < height[right] {
            left++
        } else {
            right--
        }
    }

    return maxArea
}
```

The Go implementation follows the exact same logic as the Python version. Since Go does not have built-in `min` and `max` functions for integers, the implementation uses explicit conditional checks instead.

Slices in Go behave similarly to dynamic arrays, so accessing elements by index works naturally for this problem.

Integer overflow is not a concern here because the maximum possible area is:

$10^5 \times 10^4 = 10^9$

which fits safely within Go's standard `int` type on LeetCode environments.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each pointer moves at most once across the array |
| Space | O(1) | Only a few variables are used |

The time complexity is linear because every iteration moves either the left pointer or the right pointer inward. Since each pointer can move at most `n` times, the total number of iterations is bounded by `n`.

The space complexity is constant because the algorithm does not allocate additional data structures proportional to the input size. Only a few integer variables are maintained regardless of array length.

## Edge Cases

### Minimum Input Size

The smallest valid input contains exactly two elements.

Example:

```
[1, 1]
```

A buggy implementation might accidentally skip computation when the array size is minimal. The two-pointer approach handles this naturally because the loop executes exactly once before the pointers meet.

### Heights Containing Zero

Example:

```
[0, 5, 0, 7]
```

A container involving a zero-height line holds zero water because the shorter boundary determines the container height.

The implementation correctly computes:

$\min(0, 7) = 0$

so no invalid positive area is produced.

### Multiple Equal Heights

Example:

```
[5, 5, 5, 5]
```

When both heights are equal, the implementation moves the right pointer inward. Either pointer would work correctly because the limiting height is the same on both sides.

This ensures progress toward termination while still preserving correctness.

### Tall Lines Close Together

Example:

```
[1, 100, 100, 1]
```

A naive strategy might always favor the tallest lines, but width matters just as much as height.

The algorithm correctly evaluates all meaningful possibilities and recognizes that the two middle lines produce:

$100 \times 1 = 100$

which is larger than many wider containers with smaller limiting heights.

### Strictly Increasing or Decreasing Heights

Examples:

```
[1,2,3,4,5]
```

```
[5,4,3,2,1]
```

These cases are useful for validating pointer movement logic. The algorithm consistently moves the shorter boundary inward, ensuring no optimal solution is skipped even in monotonic arrays.
