## Solution

## Problem Understanding

The problem asks us to count how many valid ways we can place `n` queens on an `n x n` chessboard such that no two queens can attack each other.

In chess, a queen can attack another piece in three directions:

1. Horizontally along the same row
2. Vertically along the same column
3. Diagonally in both diagonal directions

We are given a single integer `n`, representing the size of the board and also the number of queens we must place. The goal is not to return the actual board configurations, but only the total number of distinct valid arrangements.

For example, when `n = 4`, there are exactly two valid ways to arrange four queens so that none attack each other. Therefore, the output is `2`.

The constraints are relatively small:

```
1 <= n <= 9
```

This immediately tells us several important things:

- The board is small enough that exponential search is acceptable.
- We are expected to use some form of backtracking or depth first search.
- Generating all permutations is feasible with pruning.
- A brute force enumeration of all board states would still be too large.

A naive implementation could easily become extremely slow because the number of possible queen placements grows very rapidly. For example, if we tried every possible arrangement independently for every square, there would be `n^(n)` possibilities. Even for `n = 9`, this becomes enormous.

There are also several important edge cases:

- `n = 1` is the smallest possible board. A single queen is always valid, so the answer is `1`.
- `n = 2` and `n = 3` have no valid solutions. Many incorrect implementations accidentally count invalid diagonal placements.
- Larger boards require efficient pruning, otherwise the recursive search becomes too expensive.

The key challenge is efficiently checking whether placing a queen is safe.

## Approaches

### Brute Force Approach

The brute force idea is to try placing queens on every square of the board and then verify whether the final arrangement is valid.

One way to think about this is:

- Generate all possible ways to place `n` queens on the board.
- For each arrangement, check every pair of queens.
- If no queens attack each other, count the arrangement.

This approach is correct because it examines every possible configuration, so it cannot miss a valid solution.

However, it is extremely inefficient.

An `n x n` board contains `n²` cells. Choosing `n` queen positions among them leads to a huge number of combinations. Even after reducing the search space by placing exactly one queen per row, there are still `n!` possible permutations of column placements.

Additionally, validating each arrangement requires checking attacks between queens, which adds more computational cost.

Because of this, brute force becomes impractical as `n` increases.

### Optimal Backtracking Approach

The key observation is that we do not need to generate entire invalid boards.

Instead, we can build the solution row by row:

- Place one queen in the current row.
- Immediately check whether that placement is safe.
- If it is invalid, stop exploring that branch.
- If it is valid, continue recursively to the next row.

This technique is called backtracking.

The major optimization comes from maintaining fast lookup structures:

- A set for occupied columns
- A set for occupied main diagonals
- A set for occupied anti diagonals

This allows us to determine whether a position is safe in constant time.

For a queen at position `(row, col)`:

- Main diagonal index is `row - col`
- Anti diagonal index is `row + col`

If either diagonal already contains a queen, the current placement is invalid.

By pruning invalid states early, the search space becomes dramatically smaller.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n! * n) or worse | O(n²) | Generates many invalid configurations before checking |
| Optimal Backtracking | O(n!) | O(n) | Prunes invalid states immediately using sets |

## Algorithm Walkthrough

1. Start with row `0`.

We place queens row by row because every valid solution must contain exactly one queen in each row.
2. Try every column in the current row.

For each column, determine whether placing a queen there would conflict with any previously placed queens.
3. Check whether the column is already occupied.

We maintain a set called `columns`. If the column already exists in the set, another queen is already attacking vertically.
4. Check the main diagonal.

Cells on the same top left to bottom right diagonal share the same value of `row - col`.

If this value already exists in the `diagonals` set, the queen would be attacked diagonally.
5. Check the anti diagonal.

Cells on the same top right to bottom left diagonal share the same value of `row + col`.

If this value already exists in the `anti_diagonals` set, the queen would also be attacked diagonally.
6. If the position is safe, place the queen.

Add the column and both diagonal identifiers into their respective sets.
7. Recursively move to the next row.

Continue exploring deeper placements.
8. If we reach row `n`, we found a valid arrangement.

Increment the solution count.
9. Backtrack after recursion returns.

Remove the queen's column and diagonal entries from the sets so that other configurations can be explored.
10. Continue exploring all possibilities.

Eventually, all valid board configurations are counted.

The sets are important because they provide constant time membership checks, making safety validation very efficient.

## Python Solution

```
class Solution:
    def totalNQueens(self, n: int) -> int:
        columns = set()
        diagonals = set()
        anti_diagonals = set()

        solution_count = 0

        def backtrack(row: int) -> None:
            nonlocal solution_count

            if row == n:
                solution_count += 1
                return

            for col in range(n):
                diagonal = row - col
                anti_diagonal = row + col

                if (
                    col in columns
                    or diagonal in diagonals
                    or anti_diagonal in anti_diagonals
                ):
                    continue

                columns.add(col)
                diagonals.add(diagonal)
                anti_diagonals.add(anti_diagonal)

                backtrack(row + 1)

                columns.remove(col)
                diagonals.remove(diagonal)
                anti_diagonals.remove(anti_diagonal)

        backtrack(0)

        return solution_count
```

The implementation begins by creating three sets:

- `columns` tracks occupied columns
- `diagonals` tracks occupied main diagonals
- `anti_diagonals` tracks occupied anti diagonals

The recursive `backtrack` function attempts to place a queen in the current row.

The base case occurs when `row == n`. This means queens have been successfully placed in all rows without conflicts, so we increment the solution counter.

For each column in the row, we compute:

```
diagonal = row - col
anti_diagonal = row + col
```

These uniquely identify the diagonals.

If any of these values already exist in their corresponding sets, the position is invalid and skipped.

Otherwise, we place the queen by adding the identifiers into the sets, recurse into the next row, and then remove them afterward during backtracking.

This ensures that the recursive state remains correct for future explorations.

## Go Solution

```
func totalNQueens(n int) int {
    columns := make(map[int]bool)
    diagonals := make(map[int]bool)
    antiDiagonals := make(map[int]bool)

    solutionCount := 0

    var backtrack func(row int)

    backtrack = func(row int) {
        if row == n {
            solutionCount++
            return
        }

        for col := 0; col < n; col++ {
            diagonal := row - col
            antiDiagonal := row + col

            if columns[col] || diagonals[diagonal] || antiDiagonals[antiDiagonal] {
                continue
            }

            columns[col] = true
            diagonals[diagonal] = true
            antiDiagonals[antiDiagonal] = true

            backtrack(row + 1)

            delete(columns, col)
            delete(diagonals, diagonal)
            delete(antiDiagonals, antiDiagonal)
        }
    }

    backtrack(0)

    return solutionCount
}
```

The Go implementation follows the same logic as the Python solution.

Instead of Python sets, Go uses `map[int]bool` to simulate set behavior.

Go does not support nested named functions in the same way as Python, so we declare a function variable:

```
var backtrack func(row int)
```

Then assign the recursive closure to it.

Backtracking is handled by deleting entries from the maps after recursion returns.

Since the constraints are very small, integer overflow is not a concern here.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n!) | Each row attempts column placements with aggressive pruning |
| Space | O(n) | Recursive stack depth and tracking sets |

The time complexity is commonly approximated as `O(n!)` because the algorithm effectively explores permutations of queen placements across rows. However, pruning removes many invalid branches early, making the practical runtime much better than naive enumeration.

The space complexity is `O(n)` because:

- At most `n` recursive calls exist simultaneously
- Each tracking set stores at most `n` entries

The board itself is not explicitly stored.

## Edge Cases

### Smallest Board Size, `n = 1`

A single queen on a `1 x 1` board is always valid. Some recursive implementations accidentally mishandle the base case and return `0`. Our implementation correctly increments the solution count when `row == n`.

### No Solution Cases, `n = 2` and `n = 3`

These board sizes have no valid queen arrangements. Incorrect implementations often fail to check diagonal conflicts properly and mistakenly count invalid placements. Our use of both diagonal sets guarantees all attack directions are validated.

### Large Branching Factor for `n = 9`

Although `n = 9` is still manageable, the search tree becomes large. Without pruning, performance would degrade significantly. Our implementation immediately rejects invalid placements using constant time set lookups, which keeps execution efficient.

### Diagonal Collision Bugs

A common mistake is incorrectly identifying diagonals. Two queens share:

- The same main diagonal if `row - col` matches
- The same anti diagonal if `row + col` matches

Our implementation uses these exact formulas, ensuring diagonal attacks are detected correctly.
