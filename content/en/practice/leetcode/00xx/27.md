## Solution

## Problem Understanding

The problem asks us to remove all occurrences of a specific value, `val`, from an integer array `nums`, and to do so in-place. An in-place operation means we are not supposed to create another array to hold the result. Instead, we must modify the existing array directly.

The important detail is that the function does not need to physically shrink the array. Instead, it must return an integer `k`, representing how many elements remain after removing all occurrences of `val`. The first `k` positions in the array must contain the valid remaining elements. Anything beyond index `k - 1` does not matter.

For example, if the input array is:

```
nums = [3,2,2,3], val = 3
```

After removing all `3`s, the valid elements are:

```
[2,2]
```

The function should return `2`, because there are two valid elements left.

The order of the remaining elements does not need to stay the same. This is important because it gives us flexibility in how we modify the array.

The constraints are very small:

- `0 <= nums.length <= 100`
- `0 <= nums[i] <= 50`
- `0 <= val <= 100`

Since the array size is at most 100, even less efficient solutions would pass. However, the goal of this problem is to practice in-place array manipulation and the two-pointer technique.

Several edge cases are important to consider:

- The array may be empty.
- Every element may equal `val`.
- No element may equal `val`.
- The array may contain only one element.
- Multiple removable values may appear consecutively.

A naive implementation can easily make mistakes when shifting elements or overwriting values during iteration, especially when consecutive elements need to be removed.

## Approaches

### Brute Force Approach

A straightforward solution is to create a new temporary array and copy every element that is not equal to `val`.

For example:

```
nums = [0,1,2,2,3]
val = 2
```

We iterate through the array and build:

```
[0,1,3]
```

After that, we copy the temporary array back into `nums` and return its length.

This approach is easy to understand and guarantees correctness because every valid element is preserved exactly once. However, it uses additional memory proportional to the number of elements in the array. The problem specifically asks for an in-place solution, so using extra storage is not ideal.

### Optimal Approach

The key observation is that we only care about keeping the elements that are not equal to `val`. We do not need to preserve removed elements, and we do not care about the contents after the first `k` positions.

This naturally leads to the two-pointer technique.

We maintain:

- A read pointer that scans every element in the array
- A write pointer that tracks where the next valid element should be placed

Whenever we encounter an element that is not equal to `val`, we write it into the current write position and advance the write pointer.

This works because:

- Every valid element is copied exactly once
- Invalid elements are skipped
- The first `k` positions become the filtered array

Since all operations happen directly inside `nums`, the algorithm satisfies the in-place requirement.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Uses an extra array to store valid elements |
| Optimal | O(n) | O(1) | Uses two pointers and modifies the array in-place |

## Algorithm Walkthrough

1. Initialize a variable called `write_index` to `0`.

This pointer represents the next position where a valid element should be stored.
2. Iterate through the array using a loop.

Each element is examined exactly once.
3. For each element, check whether it is equal to `val`.

- If it equals `val`, skip it because we want to remove it.
- If it does not equal `val`, place it at `nums[write_index]`.
4. After writing a valid element, increment `write_index`.

This prepares the next insertion position.
5. Continue until every element has been processed.
6. Return `write_index`.

The value of `write_index` is exactly the number of valid elements remaining in the array.

The two-pointer strategy works well because the read pointer scans the input while the write pointer compacts valid elements toward the beginning of the array.

## Python Solution

```
from typing import List

class Solution:
    def removeElement(self, nums: List[int], val: int) -> int:
        write_index = 0

        for current_number in nums:
            if current_number != val:
                nums[write_index] = current_number
                write_index += 1

        return write_index
```

The implementation starts by initializing `write_index` to `0`. This variable tracks the next location where a valid element should be stored.

The loop iterates through every value in `nums`. For each value, the code checks whether it is different from `val`.

If the value should be kept, it is written into the current `write_index` position. Then `write_index` is incremented so the next valid element can be placed immediately after it.

If the value equals `val`, the loop simply skips it.

At the end of the traversal, `write_index` contains the number of valid elements. Since the first `write_index` positions contain all remaining values, returning this number satisfies the problem requirements.

## Go Solution

```
func removeElement(nums []int, val int) int {
    writeIndex := 0

    for _, currentNumber := range nums {
        if currentNumber != val {
            nums[writeIndex] = currentNumber
            writeIndex++
        }
    }

    return writeIndex
}
```

The Go implementation follows the same logic as the Python version.

Go slices are already reference-like structures, so modifications to `nums` inside the function affect the original slice directly. No special handling is required for in-place updates.

Unlike Python, Go does not require explicit type hints because the language is statically typed. The `range` loop provides direct access to each value in the slice.

There are no integer overflow concerns because the maximum array length is only 100.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each element is visited exactly once |
| Space | O(1) | Only a few variables are used |

The algorithm performs a single pass through the array. Every element is checked once, so the runtime grows linearly with the size of the input.

The solution modifies the array directly and does not allocate any additional data structures proportional to input size. Therefore, the extra space usage is constant.

## Edge Cases

### Empty Array

If `nums` is empty, the loop never executes. `write_index` remains `0`, and the function correctly returns `0`.

This case is important because some implementations accidentally assume the array contains at least one element.

### All Elements Equal to `val`

Consider:

```
nums = [2,2,2]
val = 2
```

Every element is skipped, so `write_index` never increases. The function correctly returns `0`.

This case can expose bugs where implementations incorrectly leave stale values or return the original array length.

### No Elements Equal to `val`

Consider:

```
nums = [1,3,5]
val = 2
```

Every element is copied back into its original position. The final array remains unchanged, and the returned value is `3`.

This verifies that the algorithm does not accidentally remove valid elements.

### Consecutive Removable Elements

Consider:

```
nums = [1,2,2,2,3]
val = 2
```

Multiple adjacent occurrences of `val` can be tricky for solutions that shift elements manually during iteration. The two-pointer approach handles this naturally because skipped elements are simply ignored while valid elements are compacted forward.

### Single Element Array

Consider:

```
nums = [5]
val = 5
```

or

```
nums = [5]
val = 2
```

The implementation correctly handles both possibilities because the loop logic works uniformly regardless of array size.
