# LeetCode 97, Interleaving String

## Problem Understanding

This problem asks us to determine whether a third string, `s3`, can be formed by interleaving two other strings, `s1` and `s2`.

An interleaving means we take characters from both `s1` and `s2` while preserving the original relative order of characters inside each string. We are allowed to switch back and forth between the two strings, but we cannot rearrange characters within either string.

For example, if:

```
s1 = "abc"
s2 = "def"
```

then:

```
"adbcef"
```

is valid because:

- Characters from `s1` appear in order: `a -> b -> c`
- Characters from `s2` appear in order: `d -> e -> f`

However:

```
"abedcf"
```

is invalid because the order of characters from `s2` changes from `d -> e -> f` into `e -> d -> f`.

The input consists of three strings:

- `s1`, the first source string
- `s2`, the second source string
- `s3`, the target string we want to construct

The expected output is:

- `true` if `s3` can be formed by interleaving `s1` and `s2`
- `false` otherwise

The constraints are relatively small:

- `s1.length` and `s2.length` are at most `100`
- `s3.length` is at most `200`

These limits strongly suggest that a dynamic programming solution is appropriate. A brute-force recursive solution would explore too many possibilities because at many positions we may choose either `s1` or `s2`.

One very important observation is that the lengths must match:

```
len(s1) + len(s2) == len(s3)
```

If this condition fails, the answer is immediately `false`.

Several edge cases are important:

- One or both input strings may be empty
- Characters may repeat many times, creating ambiguous branching
- Multiple valid interleavings may exist
- The target string may partially match both strings before eventually failing

These cases make greedy approaches unreliable and motivate dynamic programming.

## Approaches

### Brute Force Approach

The brute-force approach uses recursion to try every possible way to build `s3`.

At each position in `s3`, we decide whether the next character came from:

- the current character in `s1`
- the current character in `s2`

If both characters match the current character in `s3`, we branch into two recursive calls.

For example:

```
s1 = "aa"
s2 = "ab"
s3 = "aaab"
```

At several positions, both strings may contain the same matching character, creating multiple recursive branches.

The recursion continues until:

- we successfully consume all characters in `s3`
- or no valid move remains

This approach is correct because it explores every possible interleaving configuration. However, it is extremely inefficient because the same states are recomputed repeatedly.

In the worst case, the number of recursive calls grows exponentially.

### Key Insight

The important observation is that the problem state is fully determined by two indices:

- how many characters we have used from `s1`
- how many characters we have used from `s2`

Once those two values are known, the position in `s3` is automatically determined:

```
k = i + j
```

This means many recursive paths actually reach the same subproblem.

We can therefore use dynamic programming.

Define:

```
dp[i][j]
```

to mean:

```
Can s3[0:i+j] be formed using:
- s1[0:i]
- s2[0:j]
```

Each state depends only on previous states:

- from above, if we take a character from `s1`
- from left, if we take a character from `s2`

This converts an exponential search into a polynomial-time solution.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^(m+n)) | O(m+n) | Explores every possible interleaving recursively |
| Optimal Dynamic Programming | O(m × n) | O(n) | Reuses overlapping subproblems efficiently |

Here:

- `m = len(s1)`
- `n = len(s2)`

## Algorithm Walkthrough

### Dynamic Programming with Space Optimization

We use a one-dimensional DP array to achieve the follow-up requirement of `O(s2.length)` extra space.

1. First, check whether the lengths are compatible.

If:

```
len(s1) + len(s2) != len(s3)
```

then forming `s3` is impossible.
2. Create a DP array of size `n + 1`, where:

```
dp[j]
```

represents whether the current prefix combination is valid.
3. Initialize:

```
dp[0] = True
```

because empty strings can form an empty target.
4. Fill the first row.

This corresponds to using only characters from `s2`.

For each `j`:

```
dp[j] = dp[j-1] and s2[j-1] == s3[j-1]
```
5. Iterate through each character of `s1`.

For each row:

- Update `dp[0]` first, representing using only `s1`
- Then iterate through columns
6. For each position `(i, j)`:

We can reach this state in two ways:

From `s1`:

```
dp[j] and s1[i-1] == s3[i+j-1]
```

From `s2`:

```
dp[j-1] and s2[j-1] == s3[i+j-1]
```

If either is valid, the current state is valid.
7. After processing all states, the final answer is:

```
dp[n]
```

This works because every state only depends on:

- the current row
- the previous value in the same row

so a full 2D table is unnecessary.

## Python Solution

```
class Solution:
    def isInterleave(self, s1: str, s2: str, s3: str) -> bool:
        m = len(s1)
        n = len(s2)

        if m + n != len(s3):
            return False

        dp = [False] * (n + 1)
        dp[0] = True

        # Initialize first row
        for j in range(1, n + 1):
            dp[j] = dp[j - 1] and s2[j - 1] == s3[j - 1]

        for i in range(1, m + 1):
            # Update first column
            dp[0] = dp[0] and s1[i - 1] == s3[i - 1]

            for j in range(1, n + 1):
                take_s1 = (
                    dp[j]
                    and s1[i - 1] == s3[i + j - 1]
                )

                take_s2 = (
                    dp[j - 1]
                    and s2[j - 1] == s3[i + j - 1]
                )

                dp[j] = take_s1 or take_s2

        return dp[n]
```

The implementation begins by checking the length condition. If the total lengths do not match, we immediately return `False`.

The DP array stores whether each prefix combination is valid. Initially, only the empty prefix is valid.

The first row handles cases where only characters from `s2` are used.

For every character in `s1`, we update the DP array in place. The value of `dp[j]` before modification represents the previous row, while `dp[j - 1]` represents the current row's left neighbor.

At each step we check whether the current character of `s3` can come from either:

- `s1`
- `s2`

If either transition is valid, the current state becomes `True`.

Finally, `dp[n]` tells us whether the entire target string can be formed.

## Go Solution

```
func isInterleave(s1 string, s2 string, s3 string) bool {
    m := len(s1)
    n := len(s2)

    if m+n != len(s3) {
        return false
    }

    dp := make([]bool, n+1)
    dp[0] = true

    // Initialize first row
    for j := 1; j <= n; j++ {
        dp[j] = dp[j-1] && s2[j-1] == s3[j-1]
    }

    for i := 1; i <= m; i++ {
        // Update first column
        dp[0] = dp[0] && s1[i-1] == s3[i-1]

        for j := 1; j <= n; j++ {
            takeS1 := dp[j] &&
                s1[i-1] == s3[i+j-1]

            takeS2 := dp[j-1] &&
                s2[j-1] == s3[i+j-1]

            dp[j] = takeS1 || takeS2
        }
    }

    return dp[n]
}
```

The Go implementation follows exactly the same algorithmic structure as the Python version.

Go strings are byte-indexable, which is safe here because the problem guarantees lowercase English letters. If Unicode characters were possible, rune handling would be necessary.

The DP slice is updated in place exactly like the Python list.

Go does not distinguish between `nil` and empty strings in this problem context because `len("")` correctly returns `0`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m × n) | Every DP state is computed once |
| Space | O(n) | Only one row of DP is stored |

The algorithm processes every pair `(i, j)` exactly once. Since there are `m × n` possible states, the total running time is quadratic.

The space optimization works because each state depends only on:

- the current row
- the previous row value at the same column
- the current row value at the previous column

This allows us to compress the DP table into a single array.

## Edge Cases

### Empty Strings

One or more input strings may be empty.

For example:

```
s1 = ""
s2 = "abc"
s3 = "abc"
```

A buggy implementation may incorrectly assume both strings contain characters. Our implementation handles this naturally because the DP initialization correctly processes prefixes formed entirely from `s2`.

### Length Mismatch

If:

```
len(s1) + len(s2) != len(s3)
```

then forming `s3` is impossible regardless of character order.

For example:

```
s1 = "ab"
s2 = "cd"
s3 = "abc"
```

Some implementations forget this early validation and waste time running DP unnecessarily. Our solution immediately returns `False`.

### Repeated Characters

Repeated characters create ambiguous branching paths.

Example:

```
s1 = "aa"
s2 = "aa"
s3 = "aaaa"
```

A greedy approach may fail because multiple valid choices exist at each step. Dynamic programming correctly explores all possibilities implicitly by storing reachable states.

### Partial Match Failures

Some inputs appear valid initially but fail later.

Example:

```
s1 = "abc"
s2 = "def"
s3 = "abdecf"
```

The prefix matches successfully for several steps, but eventually the ordering constraints break.

The DP solution correctly propagates invalid states forward and returns `False` once no valid transitions remain.
