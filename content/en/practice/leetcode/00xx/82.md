## Solution

## Problem Understanding

This problem asks us to modify a **sorted singly linked list** so that **all values that appear more than once are completely removed**. Unlike the easier version of the problem, where duplicates are compressed into a single node, here we must delete every node that belongs to a duplicated value.

The important distinction is that if a number appears multiple times, **none of those nodes should remain** in the final linked list.

For example, consider:

```
1 -> 2 -> 3 -> 3 -> 4 -> 4 -> 5
```

The values `3` and `4` appear more than once, so every occurrence of those values must be removed. The result becomes:

```
1 -> 2 -> 5
```

The input is the `head` of a singly linked list where:

- The list is already sorted in ascending order
- Each node contains an integer value
- The list length ranges from `0` to `300`

The output should be the head of a new linked list that preserves the sorted order while containing only values that appeared exactly once in the original list.

The fact that the list is **sorted** is extremely important. Since duplicate values appear consecutively, we can detect duplicates by simply comparing adjacent nodes. Without sorting, we would likely need a hash map to count occurrences.

The constraints are small, only up to `300` nodes, meaning even less efficient solutions could pass. However, the goal is to design a clean and optimal linked list solution.

Several edge cases can easily break a naive implementation. If duplicates appear at the beginning of the list, the head itself may need to be removed. If duplicates appear at the end, pointer handling becomes tricky. Entire lists may consist only of duplicates, producing an empty result. Single element lists or already unique lists should remain unchanged.

## Approaches

### Brute Force Approach

A straightforward solution is to perform two passes over the linked list.

In the first pass, we traverse the linked list and count how many times each value appears using a hash map. Since the list is sorted, duplicates are adjacent, but the brute force method ignores that property and simply counts frequencies.

In the second pass, we rebuild the linked list by keeping only nodes whose values appear exactly once. Nodes with frequencies greater than `1` are skipped.

This approach is correct because it explicitly records how often every number appears, allowing us to determine which nodes should remain. However, it uses extra memory proportional to the number of unique values.

Even though the constraints are small, this solution does not fully exploit the sorted nature of the input.

### Optimal Approach

The key observation is that the list is already **sorted**, meaning duplicates always appear consecutively.

Instead of counting frequencies globally, we can detect duplicates locally while traversing the list once.

The main challenge is safely removing duplicate sequences, especially when duplicates include the head node. To simplify pointer manipulation, we introduce a **dummy node** before the head. This dummy node acts as a stable anchor and allows us to remove the first actual node without special-case logic.

We maintain two pointers:

- `prev`, pointing to the last confirmed unique node
- `current`, used to traverse the list

Whenever we encounter consecutive duplicate values, we skip the entire duplicate block and reconnect `prev.next` to the next distinct node. If the current node is unique, we simply advance both pointers.

Because every node is visited at most once, this solution runs in linear time while using constant extra space.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Uses a hash map to count frequencies, then rebuilds the list |
| Optimal | O(n) | O(1) | Uses two pointers and exploits sorted order |

## Algorithm Walkthrough

1. Create a dummy node whose `next` pointer points to `head`. This avoids special handling when the first few nodes are duplicates.
2. Initialize a pointer called `prev` to the dummy node. This pointer always represents the last confirmed node in the result list.
3. Initialize another pointer called `current` to `head`. This pointer traverses the linked list.
4. While `current` exists, check whether it starts a duplicate sequence by comparing `current.val` with `current.next.val`.
5. If duplicates are found, store the duplicate value and continue advancing `current` until all nodes with that value are skipped.
6. After the duplicate sequence ends, connect `prev.next` directly to the next valid node. This effectively removes all duplicate nodes.
7. If no duplicate exists, move `prev` forward because the current node is unique and should remain in the result.
8. Move `current` forward and continue processing until the list ends.
9. Return `dummy.next`, which points to the new head of the cleaned linked list.

The dummy node is especially important because it prevents complicated edge cases when duplicates appear at the beginning of the list. Without it, we would need additional logic to update the head pointer.

## Python Solution

```
# Definition for singly-linked list.
# class ListNode:
#     def __init__(self, val=0, next=None):
#         self.val = val
#         self.next = next

from typing import Optional

class Solution:
    def deleteDuplicates(self, head: Optional[ListNode]) -> Optional[ListNode]:
        dummy = ListNode(0, head)
        prev = dummy
        current = head

        while current:
            if current.next and current.val == current.next.val:
                duplicate_value = current.val

                while current and current.val == duplicate_value:
                    current = current.next

                prev.next = current
            else:
                prev = current
                current = current.next

        return dummy.next
```

The implementation starts by creating a dummy node that points to the original head. This simplifies removing duplicate nodes at the front of the list.

The `prev` pointer tracks the last valid node in the result list, while `current` scans through the linked list.

When consecutive nodes share the same value, we know a duplicate block has started. We store that value in `duplicate_value` and continue advancing `current` until all matching nodes are skipped.

After skipping duplicates, `prev.next` is updated to reconnect the remaining valid portion of the list.

If no duplicate is detected, the node is unique, so we move both pointers forward normally.

At the end, `dummy.next` contains the head of the filtered linked list.

## Go Solution

```
/**
 * Definition for singly-linked list.
 * type ListNode struct {
 *     Val int
 *     Next *ListNode
 * }
 */
func deleteDuplicates(head *ListNode) *ListNode {
	dummy := &ListNode{Next: head}
	prev := dummy
	current := head

	for current != nil {
		if current.Next != nil && current.Val == current.Next.Val {
			duplicateValue := current.Val

			for current != nil && current.Val == duplicateValue {
				current = current.Next
			}

			prev.Next = current
		} else {
			prev = current
			current = current.Next
		}
	}

	return dummy.Next
}
```

The Go implementation follows exactly the same logic as the Python solution. Since Go uses pointers explicitly, `nil` checks are required before accessing `Next`.

The dummy node is allocated using:

```
dummy := &ListNode{Next: head}
```

This provides a stable reference point even if the original head is removed.

There are no integer overflow concerns because node values are very small, between `-100` and `100`. Since linked lists are pointer-based structures, no slices or arrays are necessary.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each node is visited at most once |
| Space | O(1) | Only a few pointers are used |

The algorithm runs in linear time because every node is processed at most once. Even when skipping duplicate sequences, no node is revisited. Space complexity remains constant because we only maintain a small fixed number of pointers, regardless of input size.

## Edge Cases

### Empty List

If the input list is empty (`head = None` in Python or `nil` in Go), the algorithm immediately returns an empty list.

This case could cause bugs if pointer access is attempted without checking for null values. The `while current:` condition in Python and `for current != nil` in Go prevent invalid access.

### Duplicates at the Beginning

Consider:

```
1 -> 1 -> 2 -> 3
```

A naive implementation may struggle because the head itself must be removed. The dummy node solves this cleanly by allowing `prev.next` to skip the duplicate section and point directly to `2`.

Result:

```
2 -> 3
```

### Duplicates at the End

Consider:

```
1 -> 2 -> 3 -> 3
```

The duplicate sequence occurs at the tail of the list. After skipping all `3`s, `current` becomes `None`, and `prev.next` is updated to `None`.

Result:

```
1 -> 2
```

### Entire List Contains Duplicates

Consider:

```
1 -> 1 -> 2 -> 2
```

Every value appears more than once, so the entire list should be removed.

Without careful pointer handling, implementations may accidentally return stale nodes. Here, the dummy node ensures `dummy.next` correctly becomes `None`.

Result:

```
[]
```

### Already Unique List

Consider:

```
1 -> 2 -> 3 -> 4
```

No duplicates exist, so every node remains. The algorithm simply advances both pointers without modifying links.

Result:

```
1 -> 2 -> 3 -> 4
```
