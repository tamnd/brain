## Solution

## Problem Understanding

This problem asks us to determine how many **structurally unique Binary Search Trees (BSTs)** can be constructed using exactly `n` nodes with distinct values from `1` to `n`.

A Binary Search Tree follows a strict ordering rule:

- Every node in the left subtree must contain a smaller value than the root.
- Every node in the right subtree must contain a larger value than the root.

The important detail here is that we are not being asked to generate the actual trees. Instead, we only need to count how many **different structures** are possible.

For example, when `n = 3`, we use values `{1, 2, 3}`. Different choices of root produce different subtree arrangements. Even if the same values appear, a different tree structure counts as a unique BST.

To understand the problem better, consider what happens when choosing a root:

- If `1` is the root, all remaining values `{2,3}` must go to the right subtree.
- If `2` is the root, `{1}` becomes the left subtree and `{3}` becomes the right subtree.
- If `3` is the root, `{1,2}` must go to the left subtree.

The total number of valid BSTs comes from combining all valid left and right subtree possibilities.

The input `n` represents the number of unique values available, ranging from `1` to `n`. The output is a single integer representing the number of structurally unique BSTs.

The constraint `1 <= n <= 19` is relatively small, but the number of possible BSTs grows rapidly. A naive recursive solution would repeatedly solve the same subproblems and become inefficient. Since `n` is small, an efficient dynamic programming solution is ideal.

An important observation is that the actual values do not matter, only the **count of nodes** matters. For example, a subtree built from `{1,2,3}` has the same number of possible BST structures as a subtree built from `{10,11,12}`.

Some important edge cases include:

- `n = 1`, where only one node exists, meaning there is exactly one BST.
- Small values like `n = 2`, where different root placements matter.
- Larger values near `n = 19`, where exponential recursion becomes expensive without memoization or dynamic programming.

## Approaches

### Brute Force Approach

A brute-force solution would recursively try every possible root value from `1` to `n`.

For each root:

- Compute all possible left subtrees.
- Compute all possible right subtrees.
- Multiply the number of left subtree possibilities by the number of right subtree possibilities.

This multiplication works because every valid left subtree can pair with every valid right subtree.

For example, if the left side has `2` possibilities and the right side has `3`, then the root contributes:

`2 × 3 = 6`

unique BSTs.

The issue with this approach is repeated computation. The same subtree size gets recalculated many times. For example, the number of BSTs for `3` nodes may be recomputed repeatedly across recursive calls.

This leads to exponential time complexity.

### Optimal Approach, Dynamic Programming

The key insight is that the number of unique BSTs depends only on the **number of nodes**, not the actual node values.

Let `dp[i]` represent the number of unique BSTs possible using exactly `i` nodes.

For every possible root:

- The left subtree contains some number of nodes.
- The right subtree contains the remaining nodes.

If the root is at position `root`, then:

- Left subtree size = `root - 1`
- Right subtree size = `i - root`

The number of trees for this root becomes:

`dp[left] * dp[right]`

We sum this over all possible roots.

This recurrence is identical to the **Catalan Number** recurrence:

$$dp[i] = \sum_{root=1}^{i}
dp[root-1] \times dp[i-root]$$

We build the answer bottom up so that smaller subtree counts are already known before computing larger ones.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) to O(4^n) | O(n) | Recursively recomputes overlapping subproblems |
| Optimal | O(n²) | O(n) | Dynamic programming avoids repeated work |

## Algorithm Walkthrough

1. Create a dynamic programming array `dp` of size `n + 1`.

The value `dp[i]` will store the number of structurally unique BSTs possible with `i` nodes.
2. Initialize base cases.

Set `dp[0] = 1` because an empty subtree counts as one valid arrangement.

Set `dp[1] = 1` because one node can only form one BST.

The empty subtree case is especially important because subtree multiplication depends on it.
3. Iterate through tree sizes from `2` to `n`.

For every tree size `nodes`, we calculate how many BSTs can be formed.
4. Try every node as the root.

For each root choice:

- Compute the number of nodes in the left subtree.
- Compute the number of nodes in the right subtree.

Since subtree counts are already computed in `dp`, we can retrieve them instantly.
5. Multiply left and right subtree possibilities.

Every left subtree can pair with every right subtree.

So the contribution of a root becomes:

`dp[left_nodes] * dp[right_nodes]`
6. Add the contribution to `dp[nodes]`.

Repeat this for every possible root.
7. Return `dp[n]`.

After filling the table, the final answer represents the number of unique BSTs for `n` nodes.

## Python Solution

```
class Solution:
    def numTrees(self, n: int) -> int:
        dp = [0] * (n + 1)

        dp[0] = 1
        dp[1] = 1

        for nodes in range(2, n + 1):
            for root in range(1, nodes + 1):
                left_nodes = root - 1
                right_nodes = nodes - root

                dp[nodes] += dp[left_nodes] * dp[right_nodes]

        return dp[n]
```

The implementation begins by creating a dynamic programming array `dp`, where each index represents a subtree size and stores the number of unique BSTs possible for that size.

The base cases are initialized first. `dp[0] = 1` represents an empty subtree, which is necessary because subtree combinations rely on multiplication. `dp[1] = 1` represents the only possible BST with one node.

The outer loop iterates through every possible tree size from `2` to `n`. For each tree size, the inner loop tries every node as the root.

For every root choice, we determine how many nodes belong to the left and right subtrees. Since the dynamic programming table already stores answers for smaller subtree sizes, we simply multiply the number of left subtree possibilities by the number of right subtree possibilities.

Each root contributes some number of valid BSTs, and these are accumulated into `dp[nodes]`.

Finally, the method returns `dp[n]`, which contains the answer for the requested number of nodes.

## Go Solution

```
func numTrees(n int) int {
	dp := make([]int, n+1)

	dp[0] = 1
	dp[1] = 1

	for nodes := 2; nodes <= n; nodes++ {
		for root := 1; root <= nodes; root++ {
			leftNodes := root - 1
			rightNodes := nodes - root

			dp[nodes] += dp[leftNodes] * dp[rightNodes]
		}
	}

	return dp[n]
}
```

The Go implementation follows the same logic as the Python solution. A slice is used instead of a list to store dynamic programming values.

Since Go uses fixed integer types, it is worth noting that the result for `n <= 19` comfortably fits within a standard integer, so integer overflow is not a concern for this problem.

Unlike Python, variable naming in Go conventionally uses camelCase, such as `leftNodes` and `rightNodes`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n²) | For each tree size from `1` to `n`, we iterate through every possible root |
| Space | O(n) | The dynamic programming array stores results for sizes `0` through `n` |

The time complexity is quadratic because we compute values for all subtree sizes, and for each size we try every possible root position. Since `n <= 19`, this runs very efficiently.

The space complexity is linear due to the dynamic programming table. No additional recursive stack space is needed because the solution is iterative.

## Edge Cases

### Case 1: `n = 1`

This is the smallest valid input. A naive implementation might accidentally return `0` if it forgets to initialize the base case properly. Our solution explicitly sets `dp[1] = 1`, ensuring the correct answer is returned.

### Case 2: Empty Subtree Contributions

When choosing a root at the far left or far right, one subtree becomes empty. For example, when root `1` is chosen, there is no left subtree.

A common bug is treating an empty subtree as `0`. That would incorrectly eliminate valid combinations because multiplication by zero would invalidate entire tree counts. Our implementation handles this correctly by setting `dp[0] = 1`, treating the empty subtree as one valid possibility.

### Case 3: Larger Inputs Near `n = 19`

As `n` increases, the number of BST structures grows quickly. A naive recursive solution would repeatedly recompute the same subtree counts, leading to severe inefficiency.

Our dynamic programming solution avoids repeated work by storing results in `dp`, ensuring each subtree size is computed exactly once and reused whenever needed.
