## Solution

## Problem Understanding

The problem describes a robot moving on a rectangular `m x n` grid. The robot starts in the top-left corner at position `(0, 0)` and wants to reach the bottom-right corner at position `(m - 1, n - 1)`.

The robot has only two valid moves:

- Move one cell to the right
- Move one cell downward

The task is to determine how many distinct paths exist from the start cell to the destination cell under these movement restrictions.

The input consists of two integers:

- `m`, the number of rows in the grid
- `n`, the number of columns in the grid

The output is a single integer representing the total number of unique valid paths.

The constraints are important:

- `1 <= m, n <= 100`

These limits immediately tell us that brute force recursion without optimization will become extremely slow because the number of possible paths grows exponentially. However, the answer is guaranteed to fit within a 32-bit signed integer, which means we do not need special big integer handling.

There are several important edge cases to consider:

If either `m == 1` or `n == 1`, the robot has only one possible path because it can move in only one direction the entire time.

Very small grids such as `1 x 1` are also important. In that case, the robot is already at the destination, so the answer is `1`.

Larger grids can produce many overlapping subproblems. A naive recursive implementation would repeatedly recompute the same states, causing severe performance issues.

## Approaches

### Brute Force Approach

A straightforward way to solve the problem is to use recursion.

At every cell, the robot has at most two choices:

- Move right
- Move down

We can recursively explore both possibilities until we either:

- Reach the destination, which contributes one valid path
- Move outside the grid, which contributes zero paths

The recursive relation is:

- `paths(row, col) = paths(row + 1, col) + paths(row, col + 1)`

This works because every valid path from the current cell must begin with either a downward move or a rightward move.

Although this approach is logically correct, it is extremely inefficient. The same states are recomputed many times. For example, the number of paths from `(2, 3)` may be recalculated repeatedly from different recursive branches.

As grid sizes grow, the recursion tree expands exponentially, making this approach too slow for the problem constraints.

### Optimal Approach, Dynamic Programming

The key observation is that this problem has overlapping subproblems.

The number of ways to reach a cell depends only on:

- The number of ways to reach the cell above it
- The number of ways to reach the cell to its left

If we define:

- `dp[row][col]` as the number of unique paths to reach cell `(row, col)`

Then:

- `dp[row][col] = dp[row - 1][col] + dp[row][col - 1]`

This recurrence works because the robot can enter a cell only from above or from the left.

The first row and first column are special cases because there is only one way to reach those cells:

- First row, always move right
- First column, always move down

Dynamic programming avoids repeated calculations by storing intermediate results and building the solution bottom-up.

We can further optimize space because each row depends only on the current row and the previous row.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^(m+n)) | O(m+n) | Explores all possible paths recursively |
| Optimal | O(m × n) | O(n) | Uses dynamic programming with space optimization |

## Algorithm Walkthrough

1. Create a one-dimensional DP array of size `n`.

Each element represents the number of ways to reach the corresponding column in the current row. Using a 1D array reduces memory usage because we only need information from the current row and the previous row.
2. Initialize the DP array with all `1`s.

The first row has only one possible path to each cell because the robot can only move right.

Example for `n = 5`:

```
[1, 1, 1, 1, 1]
```
3. Iterate through the remaining rows starting from row `1`.

For every cell, update:

- `dp[col] = dp[col] + dp[col - 1]`
4. Understand why this update works.

At any cell:

- `dp[col]` before update represents paths from above
- `dp[col - 1]` represents paths from the left

Their sum gives the total number of paths to the current cell.
5. Continue updating the array row by row.

The DP array gradually accumulates the correct path counts for each position.
6. Return the last value in the array.

The final element represents the number of ways to reach the bottom-right corner.

## Python Solution

```
class Solution:
    def uniquePaths(self, m: int, n: int) -> int:
        dp = [1] * n

        for _ in range(1, m):
            for col in range(1, n):
                dp[col] += dp[col - 1]

        return dp[-1]
```

The implementation begins by creating a DP array filled with `1`s. This represents the first row of the grid, where every cell has exactly one path from the start.

The outer loop iterates through each remaining row. For every row, the inner loop updates each column starting from index `1`.

The update:

```
dp[col] += dp[col - 1]
```

adds:

- the number of paths from above, stored in `dp[col]`
- the number of paths from the left, stored in `dp[col - 1]`

As the algorithm progresses, the array evolves into the current row's path counts.

Finally, `dp[-1]` contains the number of paths to the bottom-right corner.

## Go Solution

```
func uniquePaths(m int, n int) int {
    dp := make([]int, n)

    for i := 0; i < n; i++ {
        dp[i] = 1
    }

    for row := 1; row < m; row++ {
        for col := 1; col < n; col++ {
            dp[col] += dp[col-1]
        }
    }

    return dp[n-1]
}
```

The Go implementation follows the same logic as the Python solution.

A slice is used instead of a fixed-size array because the grid width `n` is determined at runtime.

Go integers are sufficient because the problem guarantees the result fits within a 32-bit signed integer.

Unlike Python, Go requires explicit initialization of the DP slice values to `1`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m × n) | Every cell is processed exactly once |
| Space | O(n) | Only one row of DP values is stored |

The algorithm iterates through all `m × n` grid positions, so the runtime is proportional to the number of cells.

The space complexity is optimized to `O(n)` because we reuse a single array instead of storing the entire grid.

## Edge Cases

### Single Cell Grid

When `m = 1` and `n = 1`, the robot starts at the destination immediately.

A naive implementation might incorrectly return `0` if it assumes movement is required. This solution correctly returns `1` because the DP array starts initialized with one valid path.

### Single Row Grid

If `m = 1`, the robot can only move right.

For example:

```
m = 1, n = 5
```

There is exactly one possible path. The algorithm handles this naturally because the outer loop never executes, leaving the DP array unchanged with all values equal to `1`.

### Single Column Grid

If `n = 1`, the robot can only move downward.

For example:

```
m = 5, n = 1
```

Again, there is exactly one valid path. Since the inner loop starts from column `1`, it never runs, and the algorithm correctly returns `1`.

### Large Grid Sizes

Large inputs such as:

```
m = 100, n = 100
```

would cause brute force recursion to become infeasible due to exponential growth.

The dynamic programming solution efficiently computes the answer in polynomial time and uses only linear extra space, making it practical even at the upper constraint limits.
