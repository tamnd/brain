## Solution

## Problem Understanding

The problem gives us a string `s` containing only digits, where each number or pair of numbers may correspond to a letter using the mapping:

`1 -> A`, `2 -> B`, ..., `26 -> Z`

Our goal is to determine how many different valid ways the entire string can be decoded.

The important detail is that a digit can sometimes be interpreted in multiple ways. For example, `"12"` can be split into `"1"` and `"2"` (`"AB"`), or interpreted together as `"12"` (`"L"`). This branching behavior means the number of decoding possibilities can grow quickly.

However, not every grouping is valid. Since the mapping only includes values from `1` to `26`, we must reject:

- Numbers starting with `'0'`, because `"0"` alone is invalid
- Two digit values greater than `26`
- Invalid combinations such as `"06"` because leading zeros are not allowed

The input is a string of length between `1` and `100`, which immediately suggests that brute force recursion may become too expensive. Since each position may branch into one digit or two digit choices, the number of possible recursive paths can grow exponentially.

The output should be a single integer representing the total number of valid decoding methods. If the string cannot be decoded at all, we return `0`.

Several edge cases are important to identify early:

- Strings beginning with `'0'`, such as `"06"` or `"0"`, are immediately invalid.
- Strings containing zeros in the middle require careful handling. For example, `"10"` is valid because `"10"` maps to `'J'`, but `"30"` is invalid because `"30"` has no mapping.
- Two digit combinations must stay in the range `10` to `26`. For example, `"27"` cannot be interpreted as `"27"` and must instead be decoded as `"2"` and `"7"`.

## Approaches

### Brute Force Approach

A natural first idea is to try every possible decoding recursively.

At each position in the string, we make one or two choices:

1. Decode the current single digit, if valid.
2. Decode the next two digits together, if they form a number between `10` and `26`.

We recursively continue until we reach the end of the string. Every valid path contributes one decoding.

For example, for `"226"`:

- `"2"` → recurse on `"26"`
- `"22"` → recurse on `"6"`

From `"26"`:

- `"2"` → recurse on `"6"`
- `"26"` → recurse on empty string

This produces three valid decodings.

This approach is correct because it explores every possible legal decoding path. However, it repeatedly recomputes the same subproblems. For example, the number of ways to decode `"6"` may be recalculated many times from different branches.

In the worst case, such as `"111111111"`, the recursion tree grows exponentially because each position can branch into two choices.

### Key Insight for an Optimal Solution

The crucial observation is that this problem has **overlapping subproblems**.

Suppose we define:

`dp[i] = number of ways to decode the substring starting at index i`

If we already know the number of ways to decode later positions, we can reuse those answers instead of recomputing them.

At any index `i`:

- If `s[i] == '0'`, there are `0` valid decodings.
- Otherwise, we can always try taking one digit, contributing `dp[i + 1]`.
- If the next two digits form a valid number between `10` and `26`, we can also take two digits, contributing `dp[i + 2]`.

This creates a dynamic programming recurrence.

Instead of storing an entire DP array, we can optimize space because each state only depends on the next two states.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(n) | Recursively explores every possible split |
| Optimal Dynamic Programming | O(n) | O(1) | Uses rolling variables to avoid recomputation |

## Algorithm Walkthrough

The optimal solution uses bottom up dynamic programming with constant extra space.

1. Handle invalid starting conditions. If the string starts with `'0'`, return `0` immediately because no decoding is possible.
2. Define the DP meaning. Let `dp[i]` represent the number of ways to decode the substring starting at index `i`.
3. Observe the recurrence:

- If `s[i] == '0'`, then `dp[i] = 0`
- Otherwise, start with `dp[i + 1]` because we can decode one digit.
- If the next two digits form a valid number from `10` to `26`, add `dp[i + 2]`.
4. Instead of storing a full array, keep only the last two computed values:

- `next_one = dp[i + 1]`
- `next_two = dp[i + 2]`

Since the recurrence only depends on these two states, a full array is unnecessary.
5. Iterate backward through the string. Working from right to left ensures that when calculating the current position, all future states are already known.
6. At each index:

- If the current character is `'0'`, current ways become `0`.
- Otherwise, start with the one digit decoding count.
- Check whether a valid two digit number exists and add its contribution.
7. Shift variables for the next iteration:

- Move the current answer into `next_one`
- Move the previous `next_one` into `next_two`
8. After processing the entire string, `next_one` contains the total number of decoding ways.

## Python Solution

```
class Solution:
    def numDecodings(self, s: str) -> int:
        if not s or s[0] == '0':
            return 0

        n = len(s)

        # Base cases:
        # dp[n] = 1 (empty string)
        # dp[n + 1] is not needed explicitly
        next_one = 1
        next_two = 0

        for index in range(n - 1, -1, -1):
            if s[index] == '0':
                current = 0
            else:
                current = next_one

                if (
                    index + 1 < n
                    and (
                        s[index] == '1'
                        or (s[index] == '2' and s[index + 1] <= '6')
                    )
                ):
                    current += next_two if index + 2 <= n - 1 else 1

            next_two = next_one
            next_one = current

        return next_one
```

The implementation begins by checking whether decoding is impossible from the start. A leading zero immediately invalidates the string.

The algorithm processes the string from right to left because each position depends on future positions. `next_one` represents the number of ways to decode the substring beginning at `i + 1`, while `next_two` represents the number of ways beginning at `i + 2`.

When the current digit is `'0'`, no decoding is possible from that position, so `current` becomes `0`.

Otherwise, we first consider the one digit decoding option, which contributes `next_one`.

Then we check whether a valid two digit number exists. The condition:

```
s[index] == '1' or (
    s[index] == '2' and s[index + 1] <= '6'
)
```

ensures that only values from `10` to `26` are accepted. If valid, we add the number of ways from two positions ahead.

Finally, we update the rolling variables so they correctly represent the next DP states.

## Go Solution

```
func numDecodings(s string) int {
    if len(s) == 0 || s[0] == '0' {
        return 0
    }

    n := len(s)

    // dp[n] = 1
    nextOne := 1
    nextTwo := 0

    for i := n - 1; i >= 0; i-- {
        current := 0

        if s[i] != '0' {
            current = nextOne

            if i+1 < n &&
                (s[i] == '1' ||
                    (s[i] == '2' && s[i+1] <= '6')) {

                if i+2 <= n-1 {
                    current += nextTwo
                } else {
                    current += 1
                }
            }
        }

        nextTwo = nextOne
        nextOne = current
    }

    return nextOne
}
```

The Go implementation follows the same dynamic programming logic as the Python version. Since Go strings are byte indexed, accessing characters like `s[i]` works naturally for digit comparisons.

Go does not have Python's dynamic integer behavior, but the problem guarantees the answer fits inside a 32 bit integer, so `int` is sufficient.

Unlike Python, there is no `None` or empty string distinction to handle separately. We simply check `len(s) == 0`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each character is processed exactly once |
| Space | O(1) | Only a few variables are maintained regardless of input size |

The time complexity is linear because we scan through the string one time from right to left. Every iteration performs constant work, consisting only of character comparisons and integer additions.

The space complexity is constant because we avoid a full dynamic programming array. Since each state depends only on the next two states, storing two rolling variables is sufficient.

## Edge Cases

### Strings Starting With Zero

Inputs such as `"0"` or `"06"` are invalid because no letter maps to zero. A naive implementation may accidentally treat `"06"` as `"6"` and return an incorrect answer.

The implementation explicitly checks `s[0] == '0'` at the beginning and immediately returns `0`.

### Zeros Inside the String

Cases like `"10"` and `"20"` are valid, but `"30"` and `"100"` are problematic.

For `"10"`, only the pair `"10"` is valid because `"0"` cannot stand alone. For `"30"`, no valid mapping exists because `"30"` is outside the allowed range.

The implementation handles this by setting `current = 0` whenever a position contains `'0'`. The only way zero contributes is through a valid two digit combination.

### Numbers Larger Than 26

Inputs such as `"27"` may appear ambiguous, but `"27"` cannot be decoded as a pair because only `1` through `26` are valid.

The implementation carefully validates two digit combinations using:

```
s[index] == '1' or (
    s[index] == '2' and s[index + 1] <= '6'
)
```

This ensures values greater than `26` are rejected.

### Single Character Strings

Strings like `"1"` or `"7"` should return `1`, while `"0"` should return `0`.

The backward DP naturally handles this because the base case represents the empty string as one valid decoding path, allowing single valid digits to contribute correctly.
