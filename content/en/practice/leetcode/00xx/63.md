## Solution

## Problem Understanding

This problem asks us to count how many different valid ways a robot can travel from the top-left corner of a grid to the bottom-right corner. The robot has very limited movement rules, it may only move **right** or **down** at any point.

The complication comes from the presence of **obstacles**. The input grid contains `0`s and `1`s:

- `0` represents an empty cell the robot may step on.
- `1` represents an obstacle that blocks movement.

The robot cannot move through an obstacle. Therefore, any path that touches a blocked cell becomes invalid.

The input is an `m x n` matrix called `obstacleGrid`, where:

- `m` is the number of rows.
- `n` is the number of columns.
- `obstacleGrid[i][j]` tells us whether a position is blocked.

The output should be a single integer representing the total number of unique valid paths from `(0,0)` to `(m-1,n-1)`.

A useful way to think about this problem is as a counting problem. At every cell, we want to know:

> How many ways are there to arrive here?

Since movement is restricted to only right and down, every reachable cell can only be reached from:

- The cell directly above it
- The cell directly to the left of it

This observation naturally suggests **dynamic programming**.

The constraints are important:

- `1 <= m, n <= 100`
- Grid size can be as large as `100 x 100`

A brute-force search would become extremely expensive because the number of possible paths grows exponentially. Since the grid is relatively large, we need something closer to polynomial time.

Several edge cases are important to identify early:

- The starting cell may contain an obstacle. If `grid[0][0] == 1`, the robot cannot even begin moving, so the answer is `0`.
- The destination cell may contain an obstacle. If `grid[m-1][n-1] == 1`, no valid path exists.
- A single-cell grid such as `[[0]]` should return `1`, because the robot is already at the destination.
- Entire rows or columns may become blocked, cutting off all paths.
- Obstacles near the beginning can make many later cells unreachable.

## Approaches

### Brute Force Approach

A straightforward approach is to recursively try every possible path.

Starting from `(0,0)`, we attempt:

1. Move right
2. Move down

If we hit an obstacle or go out of bounds, that path is invalid. If we reach the bottom-right corner, we count one successful path.

This works because it systematically explores every possible legal movement sequence and counts only valid paths.

However, the problem is efficiency.

At most cells, we branch into two possibilities. This creates a recursion tree with exponential growth. Even worse, many states are recomputed repeatedly. For example, the number of paths from `(2,2)` may be recalculated many times from different recursive branches.

For a `100 x 100` grid, this approach is far too slow.

### Key Insight for an Optimal Solution

The important observation is that the number of ways to reach a cell depends only on the cells immediately above and to the left.

If a cell is not blocked:

$$dp[i][j] = dp[i-1][j] + dp[i][j-1]$$

A robot can only enter `(i,j)` from those two positions.

If a cell contains an obstacle:

$$dp[i][j] = 0$$

because it cannot be visited.

This naturally leads to **dynamic programming**, where we compute answers for smaller subproblems and reuse them.

Instead of repeatedly recalculating path counts, we build the answer incrementally across the grid.

We can optimize space by using a **1D DP array**, since each row depends only on the current row and the previous row.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^(m+n)) | O(m+n) | Recursively explores every possible path |
| Optimal | O(m × n) | O(n) | Dynamic programming with a 1D array |

## Algorithm Walkthrough

1. Create a DP array of length `n`, where `n` is the number of columns. Each entry represents the number of ways to reach that column in the current row.
2. Handle the starting position carefully. If the top-left cell contains an obstacle, immediately return `0`, because movement is impossible.
3. Initialize `dp[0] = 1`, representing one way to stand at the starting cell.
4. Traverse the grid row by row and column by column.
5. At each cell:

- If the cell contains an obstacle (`1`), set `dp[col] = 0`. This wipes out any path count because the robot cannot stand on an obstacle.
- Otherwise:

- If `col > 0`, add `dp[col - 1]` to `dp[col]`.
6. Why does this work?

- `dp[col]` already stores the number of ways to come from above.
- `dp[col - 1]` stores the number of ways to come from the left.
- Their sum gives the total number of ways to reach the current cell.
7. Continue processing the entire grid.
8. After processing all rows, `dp[n - 1]` contains the number of ways to reach the bottom-right corner.

The reason a 1D array works is because we only ever need:

- The current row’s left value
- The previous row’s value at the same column

The array naturally preserves this information.

## Python Solution

```
from typing import List

class Solution:
    def uniquePathsWithObstacles(self, obstacleGrid: List[List[int]]) -> int:
        rows = len(obstacleGrid)
        cols = len(obstacleGrid[0])

        # If the starting cell is blocked, no path exists
        if obstacleGrid[0][0] == 1:
            return 0

        dp = [0] * cols
        dp[0] = 1

        for row in range(rows):
            for col in range(cols):
                # Obstacle blocks all paths to this cell
                if obstacleGrid[row][col] == 1:
                    dp[col] = 0
                # Add paths from the left cell
                elif col > 0:
                    dp[col] += dp[col - 1]

        return dp[-1]
```

The implementation begins by determining the grid dimensions. We immediately check whether the starting position contains an obstacle. If it does, we return `0` because no movement is possible.

We then initialize a one-dimensional dynamic programming array called `dp`. Each position in `dp` represents the number of ways to reach that column for the current row being processed.

Initially, `dp[0] = 1` because there is exactly one way to stand at the starting cell, assuming it is not blocked.

As we iterate through the grid:

- If we encounter an obstacle, we set `dp[col] = 0`. This ensures no future path may pass through that position.
- Otherwise, we add the left neighbor’s path count (`dp[col - 1]`) to the current value (`dp[col]`), which already represents paths from above.

By the end of traversal, the final entry contains the total number of unique valid paths.

## Go Solution

```
func uniquePathsWithObstacles(obstacleGrid [][]int) int {
	rows := len(obstacleGrid)
	cols := len(obstacleGrid[0])

	// If the starting position is blocked
	if obstacleGrid[0][0] == 1 {
		return 0
	}

	dp := make([]int, cols)
	dp[0] = 1

	for row := 0; row < rows; row++ {
		for col := 0; col < cols; col++ {
			// Obstacle blocks all paths
			if obstacleGrid[row][col] == 1 {
				dp[col] = 0
			} else if col > 0 {
				// Add paths from the left
				dp[col] += dp[col-1]
			}
		}
	}

	return dp[cols-1]
}
```

The Go implementation follows the same algorithm as the Python version. A slice named `dp` is used instead of a Python list.

Since the problem guarantees the answer fits within `2 * 10^9`, Go's standard `int` type is sufficient and overflow is not a concern.

Go slices are reference-backed structures, which makes the one-dimensional DP solution memory efficient. No special handling for `nil` input is required because the problem guarantees valid dimensions.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m × n) | Every cell in the grid is processed exactly once |
| Space | O(n) | Only one row of DP values is stored |

The algorithm visits each grid cell once and performs constant work per cell, producing a time complexity of `O(m × n)`.

The memory optimization comes from using a single one-dimensional DP array instead of a full `m × n` matrix. Since only the current row and previous values are needed, the space complexity becomes `O(n)`.

## Edge Cases

### Starting Cell Is Blocked

If the input is:

```
[[1]]
```

the robot starts on an obstacle and cannot move at all. A naive implementation might incorrectly initialize the starting position as reachable. Our implementation prevents this by immediately returning `0` when `obstacleGrid[0][0] == 1`.

### Destination Cell Is Blocked

Consider:

```
[
  [0,0],
  [0,1]
]
```

The robot can move through the grid but can never stand on the destination. Since obstacle cells force `dp[col] = 0`, the final result correctly becomes `0`.

### Single Cell Grid

For:

```
[[0]]
```

the robot begins at the destination, so there is exactly one valid path. The algorithm correctly returns `1` because `dp[0]` starts at `1` and remains unchanged.

### Entire Path Gets Blocked

Consider:

```
[
  [0,1,0],
  [0,1,0],
  [0,1,0]
]
```

The obstacle column completely blocks progress to the right side of the grid. A naive recursive solution might waste enormous time exploring dead ends. Our DP solution naturally propagates zeros through unreachable cells and correctly returns `0`.

### First Row or First Column Contains Obstacles

For:

```
[
  [0,0,1],
  [0,0,0]
]
```

once an obstacle appears in the first row, every cell to its right becomes unreachable. The DP update logic automatically handles this because obstacles reset the path count to `0`, preventing invalid propagation.
