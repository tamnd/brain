## Solution

## Problem Understanding

The problem asks us to compute the **integer square root** of a non-negative integer `x`. More specifically, we need to return the largest integer `r` such that:

$$r^2 \le x$$

If the square root is not a whole number, we round it **down** to the nearest integer.

For example, if `x = 8`, the actual square root is approximately `2.828...`, but because the problem asks for the result rounded down, we return `2`.

The input is a single non-negative integer `x`, and the output is another non-negative integer representing the floor of the square root. The problem explicitly forbids using built-in exponentiation or square root functions such as `pow()` or `x ** 0.5`, which means we must compute the answer manually.

The constraints are important:

$$0 \le x \le 2^{31} - 1$$

This means `x` can be as large as `2,147,483,647`, which is over two billion. A naive solution that checks every number from `1` upward may still work in some cases, but it becomes inefficient for large values. We need an approach that scales efficiently.

Several edge cases deserve attention:

- `x = 0`, because the square root of zero is zero.
- `x = 1`, because the square root is exactly one.
- Large values near `2³¹ - 1`, because squaring integers can overflow in some languages.
- Numbers that are not perfect squares, where we must correctly return the floor value instead of rounding.

## Approaches

### Brute Force Approach

A straightforward solution is to test integers starting from `0` and continue increasing until the square exceeds `x`.

For every integer `i`, we compute:

$$i^2$$

If `i²` becomes greater than `x`, then the answer must be `i - 1`, because `i - 1` was the largest integer whose square did not exceed `x`.

This approach is correct because it checks all possible square roots in increasing order. The first value that exceeds `x` tells us exactly where the valid square root ends.

However, this method is inefficient for large values. In the worst case, we may need to check up to `√x` numbers. Since `x` can be extremely large, this becomes unnecessarily slow.

### Optimal Approach, Binary Search

The key observation is that the answer lies within a **sorted search space**.

The possible square root must exist between:

$$0 \text{ and } x$$

If we choose a midpoint `mid`, then:

- If `mid² == x`, we found the exact square root.
- If `mid² < x`, the answer must be larger, so we search the right half.
- If `mid² > x`, the answer must be smaller, so we search the left half.

This works because the function:

$$f(n) = n^2$$

is monotonically increasing for non-negative integers. Since larger numbers always produce larger squares, binary search applies naturally.

Instead of checking every number one by one, binary search repeatedly halves the search range, making the algorithm dramatically faster.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(√x) | O(1) | Checks integers sequentially until square exceeds `x` |
| Optimal | O(log x) | O(1) | Uses binary search on the answer space |

## Algorithm Walkthrough

1. Handle small edge cases naturally by setting the binary search range. We initialize `left = 0` and `right = x`, because the square root cannot be smaller than `0` or greater than `x`.
2. While `left <= right`, compute the midpoint:

$$mid = left + \frac{right - left}{2}$$

This avoids overflow in languages where large integer arithmetic can be problematic.
3. Compute `mid * mid`.
4. If `mid² == x`, return `mid` immediately because we found the exact square root.
5. If `mid² < x`, then `mid` is a valid candidate, but there might be a larger valid square root. Store it and continue searching the right half by moving `left = mid + 1`.
6. If `mid² > x`, then `mid` is too large. Move to the left half using `right = mid - 1`.
7. When the loop finishes, return the largest valid candidate found. This represents the floor of the square root.

The reason binary search works here is that every midpoint tells us whether the correct answer lies to the left or right, eliminating half of the remaining search space each time.

## Python Solution

```
class Solution:
    def mySqrt(self, x: int) -> int:
        left = 0
        right = x
        answer = 0

        while left <= right:
            mid = left + (right - left) // 2
            square = mid * mid

            if square == x:
                return mid
            elif square < x:
                answer = mid
                left = mid + 1
            else:
                right = mid - 1

        return answer
```

This implementation follows the binary search strategy directly.

We begin by defining the search range from `0` to `x`. The variable `answer` stores the best valid square root seen so far. This matters when `x` is not a perfect square because we need to return the floor value.

Inside the loop, we calculate the midpoint and square it. If the square exactly matches `x`, we immediately return `mid`.

If the square is smaller than `x`, we know `mid` is a valid square root candidate. However, there may be a larger valid integer, so we update `answer` and search the right half.

If the square is too large, we discard the right half and continue searching the left side.

Once the search finishes, `answer` contains the largest integer whose square does not exceed `x`.

## Go Solution

```
func mySqrt(x int) int {
	left := 0
	right := x
	answer := 0

	for left <= right {
		mid := left + (right-left)/2
		square := mid * mid

		if square == x {
			return mid
		} else if square < x {
			answer = mid
			left = mid + 1
		} else {
			right = mid - 1
		}
	}

	return answer
}
```

The Go implementation closely mirrors the Python version. Since Go integers are fixed-size, integer overflow is more relevant than in Python. Fortunately, the problem constraint stays within 32-bit signed integer range, and Go's `int` type on modern systems safely handles the multiplication.

The midpoint calculation uses:

```
left + (right-left)/2
```

instead of `(left + right) / 2`, because this avoids potential overflow in general binary search problems.

Unlike Python, Go does not support arbitrary precision integers by default, so careful arithmetic is important in many problems. However, for this constraint range, the implementation remains safe.

## Worked Examples

### Example 1

**Input:** `x = 4`

We search for the integer square root of `4`.

| Iteration | Left | Right | Mid | Mid² | Action |
| --- | --- | --- | --- | --- | --- |
| 1 | 0 | 4 | 2 | 4 | Exact match found |

Since `2² = 4`, we immediately return `2`.

**Output:** `2`

### Example 2

**Input:** `x = 8`

We search for the integer square root of `8`.

| Iteration | Left | Right | Mid | Mid² | Action |
| --- | --- | --- | --- | --- | --- |
| 1 | 0 | 8 | 4 | 16 | Too large, search left |
| 2 | 0 | 3 | 1 | 1 | Valid candidate, search right |
| 3 | 2 | 3 | 2 | 4 | Valid candidate, search right |
| 4 | 3 | 3 | 3 | 9 | Too large, search left |

At the end of the search, the largest valid candidate is `2`.

Since:

$$2^2 = 4 \le 8$$

and:

$$3^2 = 9 > 8$$

the correct answer is `2`.

**Output:** `2`

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(log x) | Binary search halves the search range every iteration |
| Space | O(1) | Only a few variables are used |

The time complexity is logarithmic because every iteration removes half of the remaining search space. Even for very large inputs, the number of operations remains small.

The space complexity is constant because we only maintain a fixed number of integer variables, regardless of input size.

## Edge Cases

### Edge Case 1: `x = 0`

This is the smallest possible input. A naive implementation may accidentally return an incorrect value or fail due to improper loop initialization.

Our implementation handles it correctly because the binary search starts with `left = 0` and `right = 0`. The midpoint becomes `0`, and since `0² = 0`, the function immediately returns `0`.

### Edge Case 2: `x = 1`

This is the smallest positive perfect square. Some implementations mistakenly overcomplicate handling small numbers or fail because of incorrect search boundaries.

Our algorithm correctly finds that `1² = 1` and immediately returns `1`.

### Edge Case 3: Non-perfect Squares

Inputs such as `8`, `15`, or `26` do not have integer square roots. A common mistake is rounding incorrectly instead of returning the floor.

Our implementation avoids this by maintaining the variable `answer`, which stores the largest valid midpoint whose square does not exceed `x`. This guarantees the correct floor result.

### Edge Case 4: Very Large Input

For values near `2³¹ - 1`, brute-force approaches become inefficient, and careless multiplication may risk overflow in some languages.

Binary search keeps the runtime efficient at `O(log x)`, and the midpoint calculation avoids overflow risks by using:

$$left + \frac{right - left}{2}$$

instead of directly adding `left + right`.
