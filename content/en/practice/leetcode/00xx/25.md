## Solution

## Problem Understanding

This problem asks us to reverse a linked list in groups of size `k`. Instead of reversing the entire list at once, we divide the list into consecutive chunks containing exactly `k` nodes. Every complete group must be reversed, while any remaining nodes at the end that do not form a complete group must stay in their original order.

The important detail is that we are not allowed to modify node values. We must physically rearrange the pointers between nodes. This means the problem is fundamentally about linked list pointer manipulation.

For example, consider the list:

```
1 -> 2 -> 3 -> 4 -> 5
```

If `k = 2`, we reverse every pair:

```
(1 -> 2) becomes (2 -> 1)
(3 -> 4) becomes (4 -> 3)
5 remains unchanged
```

Final result:

```
2 -> 1 -> 4 -> 3 -> 5
```

If `k = 3`, only the first three nodes form a complete group:

```
1 -> 2 -> 3 becomes 3 -> 2 -> 1
```

The remaining nodes `4 -> 5` are left untouched because their count is smaller than `k`.

The constraints are important because they guide the expected efficiency of the solution. The number of nodes can be as large as 5000, which is not extremely large, but still large enough that repeatedly traversing or rebuilding the list inefficiently could become problematic. The follow-up requirement asks for `O(1)` extra space, which strongly suggests that the intended solution should reverse nodes in-place rather than copying them into arrays or auxiliary data structures.

Several edge cases are especially important:

- If the list length is smaller than `k`, nothing should be reversed.
- If `k = 1`, reversing groups of one node changes nothing.
- If the list length is exactly a multiple of `k`, every node should participate in a reversal.
- If the final group contains fewer than `k` nodes, it must remain unchanged.
- Pointer handling must be extremely careful to avoid losing parts of the list during reversal.

## Approaches

### Brute Force Approach

A straightforward way to solve the problem is to first copy all linked list nodes into an array. Once the nodes are stored in an indexed structure, we can process the array in chunks of size `k`. Every complete chunk can be reversed easily using array operations, while incomplete chunks at the end remain unchanged.

After rearranging the array, we reconnect all node pointers according to the new order.

This approach works because arrays make indexed access and reversal simple. However, it uses additional memory proportional to the number of nodes. Since the follow-up explicitly asks for constant extra space, this approach is not ideal.

Its time complexity is still linear because every node is visited a constant number of times, but the auxiliary array requires `O(n)` extra memory.

### Optimal Approach

The key observation is that linked list reversal can be performed in-place using pointer manipulation. Instead of storing nodes externally, we can reverse each group directly inside the linked list.

The optimal strategy works as follows:

1. Check whether a complete group of `k` nodes exists.
2. Reverse those `k` nodes in-place.
3. Reconnect the reversed group back into the main list.
4. Continue processing the next group.

A dummy node is extremely helpful because reversing the first group changes the head of the list. The dummy node provides a stable node before the actual head, simplifying pointer updates.

The critical insight is that reversing a linked list segment can be done iteratively with three pointers:

- `prev`
- `curr`
- `next_node`

This allows us to reverse each group in `O(k)` time using constant extra memory.

| --- | --- | --- | --- |

| Optimal | O(n) | O(1) | Reverse each k-sized group directly in the linked list |

## Algorithm Walkthrough

1. Create a dummy node whose `next` pointer points to the original head. This simplifies handling reversals involving the head node.
2. Maintain a pointer called `group_prev`. This pointer always represents the node immediately before the current group we want to reverse.
3. Before reversing anything, verify that at least `k` nodes remain. Starting from `group_prev`, move forward `k` times to locate the group's last node. If fewer than `k` nodes exist, stop processing because the remaining nodes should stay unchanged.
4. Store two important pointers:

- `group_start`, which is the first node in the current group
- `group_next`, which is the node immediately after the group
5. Reverse the current group using the standard linked list reversal technique. Initialize:

- `prev = group_next`
- `curr = group_start`

Repeatedly:

- Save `curr.next`
- Point `curr.next` to `prev`
- Advance both pointers

This reverses exactly `k` nodes while keeping the remainder of the list connected.
6. After reversal:

- `group_prev.next` should point to the new group head
- `group_start` becomes the tail of the reversed group
7. Move `group_prev` to `group_start` because it is now the tail of the processed group.
8. Repeat until no complete group remains.

The reason this works is that every reversal reconnects cleanly to both:

- the previously processed section
- the unprocessed remainder of the list

No extra arrays or stacks are needed.

## Python Solution

```
# Definition for singly-linked list.
# class ListNode:
#     def __init__(self, val=0, next=None):
#         self.val = val
#         self.next = next

from typing import Optional

class Solution:
    def reverseKGroup(
        self,
        head: Optional[ListNode],
        k: int
    ) -> Optional[ListNode]:

        dummy = ListNode(0)
        dummy.next = head

        group_prev = dummy

        while True:
            kth = group_prev

            # Find the kth node from group_prev
            for _ in range(k):
                kth = kth.next
                if not kth:
                    return dummy.next

            group_next = kth.next

            # Reverse current group
            prev = group_next
            curr = group_prev.next

            while curr != group_next:
                next_node = curr.next
                curr.next = prev
                prev = curr
                curr = next_node

            # Reconnect reversed group
            temp = group_prev.next
            group_prev.next = kth
            group_prev = temp
```

The implementation starts by creating a dummy node to simplify edge-case handling. Without the dummy node, reversing the first group would require special logic because the head pointer itself changes.

The variable `group_prev` tracks the node before the current group. This is important because after reversing a group, we must reconnect the previous section of the list to the newly reversed group.

The loop that finds `kth` ensures that a complete group exists before attempting reversal. If fewer than `k` nodes remain, the algorithm immediately returns the result because the remaining nodes must stay unchanged.

The reversal itself uses the classic iterative linked list reversal pattern. Initializing `prev` to `group_next` is the subtle but critical trick. It ensures that after reversal, the tail of the group automatically points to the next section of the list.

Finally, the pointers are reconnected:

- `group_prev.next` points to the new head of the reversed group
- the original group head becomes the new tail

Then `group_prev` advances to prepare for the next iteration.

## Go Solution

```
/**
 * Definition for singly-linked list.
 * type ListNode struct {
 *     Val int
 *     Next *ListNode
 * }
 */
func reverseKGroup(head *ListNode, k int) *ListNode {
    dummy := &ListNode{Next: head}

    groupPrev := dummy

    for {
        kth := groupPrev

        // Find kth node
        for i := 0; i < k; i++ {
            kth = kth.Next

            if kth == nil {
                return dummy.Next
            }
        }

        groupNext := kth.Next

        // Reverse group
        prev := groupNext
        curr := groupPrev.Next

        for curr != groupNext {
            nextNode := curr.Next
            curr.Next = prev
            prev = curr
            curr = nextNode
        }

        // Reconnect group
        temp := groupPrev.Next
        groupPrev.Next = kth
        groupPrev = temp
    }
}
```

The Go implementation follows exactly the same logic as the Python version, but pointer handling is more explicit because Go does not provide Python-style references.

One important difference is the handling of `nil` pointers. In Go, every pointer access must be carefully checked to avoid dereferencing `nil`. The loop that searches for the `kth` node immediately returns if `kth` becomes `nil`.

Go also uses explicit struct pointers throughout the implementation. Since linked lists naturally rely on shared node references, pointer semantics are essential.

No slices or arrays are used, which preserves the required `O(1)` auxiliary memory usage.

The algorithm processes each node during:

- group validation

- reconnection

Although some nodes are touched multiple times, the total work across the entire list remains linear. No auxiliary data structures proportional to input size are allocated, so the extra space remains constant.

## Edge Cases

One important edge case occurs when `k = 1`. Reversing groups of one node should leave the list unchanged. A buggy implementation might still attempt unnecessary pointer manipulation and accidentally corrupt the list. In this solution, each group reversal still works correctly because reversing a single node simply reconnects it to itself and advances normally.

Another critical case happens when the list length is smaller than `k`. For example:

```
head = [1,2]
k = 3
```

Since no complete group exists, the list should remain unchanged. The implementation handles this safely during the `kth` node search. If fewer than `k` nodes are found, the function immediately returns the original structure without performing any reversal.

A third important case is when the final group is incomplete. Consider:

```
head = [1,2,3,4,5]
k = 3
```

The first three nodes are reversed, but the remaining two nodes must stay untouched. Many incorrect solutions accidentally reverse the partial group or disconnect it from the list. This implementation avoids that problem by checking group size before any reversal begins.

Another subtle edge case involves reversing the very first group. Since the head pointer changes after reversal, implementations without a dummy node often require special-case handling. Using a dummy node eliminates this complexity and guarantees consistent pointer updates for every group, including the first one.
