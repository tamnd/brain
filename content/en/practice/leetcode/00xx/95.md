## Solution

## Problem Understanding

The problem asks us to generate every structurally unique Binary Search Tree (BST) that can be formed using the integers from `1` to `n`, where each number must appear exactly once.

A Binary Search Tree has a very important property:

- Every value in the left subtree must be smaller than the root.
- Every value in the right subtree must be larger than the root.

Because the values are fixed as `1` through `n`, the only thing that changes between valid trees is the structure of the tree. Two BSTs are considered different if their arrangement of nodes differs, even if they contain the same values.

For example, when `n = 3`, we must generate all valid BST structures using the values `{1, 2, 3}`. One valid tree might use `2` as the root with `1` on the left and `3` on the right. Another valid tree might use `1` as the root and place both remaining nodes on the right side.

The input is a single integer `n`, and the output is a list containing the root node of every valid BST.

The constraint `1 <= n <= 8` is very important. Since the number of unique BSTs grows rapidly according to the Catalan numbers, brute force generation becomes expensive as `n` increases. However, because `n` is capped at `8`, a recursive backtracking solution with memoization is practical.

An important detail is that the answer may be returned in any order. This means we do not need to sort or organize the final trees in a specific sequence.

Several edge cases are important:

- `n = 1`, there is only one possible BST.
- Small ranges during recursion may become empty. For example, when constructing a left subtree for the smallest value, there are no smaller values available. We must treat this as a valid empty subtree.
- Multiple recursive calls will generate the same subtree ranges repeatedly, so failing to cache results can lead to excessive recomputation.

## Approaches

### Brute Force Approach

A brute force strategy would attempt to generate every possible binary tree arrangement using the values `1` through `n`, then validate whether each tree satisfies BST properties.

This approach works conceptually because every possible structure is explored, and valid BSTs are filtered afterward. However, the number of binary tree structures grows extremely quickly. Additionally, validating each generated tree requires traversing it and checking ordering constraints.

The major inefficiency is that most generated trees are invalid, meaning a large amount of work is wasted.

Another brute force variation is recursively trying every permutation of insertion orders into a BST. Different insertion orders may still produce identical BST structures, which creates redundant work and duplicate trees that must be filtered.

Because of this combinatorial explosion, brute force becomes impractical.

### Optimal Approach

The key observation is that BST structure is naturally recursive.

If we choose some value `i` as the root:

- All values smaller than `i` must belong to the left subtree.
- All values larger than `i` must belong to the right subtree.

This means:

- Left subtree possibilities come entirely from the range `[start, i - 1]`
- Right subtree possibilities come entirely from the range `[i + 1, end]`

We can recursively generate all possible left subtrees and all possible right subtrees, then combine every left subtree with every right subtree.

This transforms the problem into a divide-and-conquer recursion problem.

The recursive function becomes:

- "Generate all BSTs using values in range `[start, end]`"

For every possible root value:

1. Generate all left subtree possibilities.
2. Generate all right subtree possibilities.
3. Combine them pairwise.

Memoization further improves performance because the same ranges are computed repeatedly.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential, far worse than Catalan growth | Exponential | Generates many invalid or duplicate trees |
| Optimal Recursive BST Construction | O(Cn * n) | O(Cn * n) | Uses BST properties directly and builds only valid trees |

Here, `Cn` represents the `n`th Catalan number, which approximates the number of unique BSTs.

## Algorithm Walkthrough

1. Define a recursive helper function `build(start, end)`.

This function generates all unique BSTs that can be formed using values from `start` to `end`.
2. Handle the base case.

If `start > end`, there are no values available to form a tree. Instead of returning an empty list, we return `[None]`.

This is extremely important because an empty subtree is still a valid subtree possibility during combinations.
3. Check whether the current range has already been computed.

Since the same ranges appear repeatedly during recursion, memoization avoids recomputing identical subtree sets.
4. Iterate through every possible root value.

For every integer `root_value` in the range `[start, end]`, treat that value as the root of the BST.
5. Recursively generate all left subtrees.

All values smaller than `root_value` belong in the left subtree, so recursively compute:

`build(start, root_value - 1)`
6. Recursively generate all right subtrees.

All values larger than `root_value` belong in the right subtree, so recursively compute:

`build(root_value + 1, end)`
7. Combine all left and right subtree possibilities.

Every left subtree can pair with every right subtree.

For each pair:

- Create a new root node.
- Attach the left subtree.
- Attach the right subtree.
- Add the tree to the result list.
8. Store the computed result in memoization.

This ensures future calls for the same range return immediately.
9. Return all generated trees.
10. Start the recursion with the full range `[1, n]`.

## Python Solution

```
from typing import List, Optional

# Definition for a binary tree node.
class TreeNode:
    def __init__(self, val=0, left=None, right=None):
        self.val = val
        self.left = left
        self.right = right

class Solution:
    def generateTrees(self, n: int) -> List[Optional[TreeNode]]:
        memo = {}

        def build(start: int, end: int) -> List[Optional[TreeNode]]:
            if start > end:
                return [None]

            if (start, end) in memo:
                return memo[(start, end)]

            all_trees = []

            for root_value in range(start, end + 1):
                left_subtrees = build(start, root_value - 1)
                right_subtrees = build(root_value + 1, end)

                for left_tree in left_subtrees:
                    for right_tree in right_subtrees:
                        root = TreeNode(root_value)
                        root.left = left_tree
                        root.right = right_tree
                        all_trees.append(root)

            memo[(start, end)] = all_trees
            return all_trees

        return build(1, n)
```

The implementation begins by creating a memoization dictionary named `memo`. The keys are tuples representing ranges `(start, end)`, and the values are lists of all BSTs that can be formed from that range.

The recursive helper function `build` handles subtree generation.

The first condition checks whether the range is empty. If `start > end`, we return `[None]`. Returning `[None]` instead of `[]` allows the combination logic to correctly form trees with missing left or right children.

The memoization check prevents repeated recursive work. Since many recursive calls request identical ranges, caching significantly improves performance.

The main loop iterates through every possible root value. For each root:

- Recursive calls generate all valid left subtree possibilities.
- Recursive calls generate all valid right subtree possibilities.

Nested loops then combine every left subtree with every right subtree. A new `TreeNode` is created for each combination because each resulting BST must be structurally independent.

Finally, all generated trees are cached and returned.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */

func generateTrees(n int) []*TreeNode {
    memo := make(map[[2]int][]*TreeNode)

    var build func(int, int) []*TreeNode

    build = func(start int, end int) []*TreeNode {
        if start > end {
            return []*TreeNode{nil}
        }

        key := [2]int{start, end}

        if trees, exists := memo[key]; exists {
            return trees
        }

        var allTrees []*TreeNode

        for rootValue := start; rootValue <= end; rootValue++ {
            leftSubtrees := build(start, rootValue-1)
            rightSubtrees := build(rootValue+1, end)

            for _, leftTree := range leftSubtrees {
                for _, rightTree := range rightSubtrees {
                    root := &TreeNode{
                        Val:   rootValue,
                        Left:  leftTree,
                        Right: rightTree,
                    }

                    allTrees = append(allTrees, root)
                }
            }
        }

        memo[key] = allTrees
        return allTrees
    }

    return build(1, n)
}
```

The Go implementation follows the same recursive structure as the Python solution.

A Go map cannot directly use slices as keys, so a fixed-size array `[2]int` is used to represent the `(start, end)` range for memoization.

Go uses `nil` instead of `None`, so empty subtree cases return `[]*TreeNode{nil}`.

Slices are dynamically resized using `append`, which works naturally for accumulating generated trees.

Unlike Python, Go requires explicit pointer creation using `&TreeNode{}`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(Cn * n) | Each unique BST is constructed once, and there are Catalan-number many BSTs |
| Space | O(Cn * n) | Required for storing all generated trees and recursion stack |

The number of unique BSTs for `n` nodes is the `n`th Catalan number:

$C_n = \frac{1}{n+1}\binom{2n}{n}$

This grows approximately like:

```
4^n / (n^(3/2))
```

Since every generated tree contains up to `n` nodes, the total work becomes proportional to the number of trees multiplied by tree size.

Memoization avoids recomputing identical subtree ranges, which dramatically reduces redundant recursive calls.

## Edge Cases

### Edge Case 1: `n = 1`

This is the smallest valid input. A buggy implementation might incorrectly return an empty list or fail recursion setup.

The algorithm handles this naturally because the recursion selects `1` as the only possible root and combines two empty subtrees.

### Edge Case 2: Empty Subtree Ranges

During recursion, calls such as `build(1, 0)` occur frequently. This represents an empty subtree.

A common mistake is returning `[]` instead of `[None]`. Returning an empty list would prevent subtree combinations from forming correctly.

By returning `[None]`, the algorithm correctly models the absence of a child node.

### Edge Case 3: Duplicate Subproblem Computation

Many recursive ranges repeat. For example:

```
build(1, 2)
```

may be requested multiple times.

Without memoization, the recursion tree grows extremely large and performs unnecessary repeated work.

The memo dictionary ensures each range is computed only once.

### Edge Case 4: Deep Recursive Combination Trees

As `n` increases, some BSTs become highly unbalanced, resembling linked lists.

The recursive construction still works correctly because subtree generation depends only on value ranges, not balance properties. Every valid BST structure is generated exactly once.
