## Solution

## Problem Understanding

The problem gives us an array `nums` where each element represents the maximum distance we can jump forward from that position. We start at index `0`, and our goal is to reach the last index using the fewest number of jumps possible.

If we are currently at index `i`, we may jump to any index between `i + 1` and `i + nums[i]`, as long as the destination remains inside the array bounds.

The task is not to determine whether the last index is reachable, because the problem guarantees that it is always reachable. Instead, we must compute the minimum number of jumps needed to get there.

For example, in the array:

```
[2,3,1,1,4]
```

From index `0`, we can jump either to index `1` or index `2`.

- Jumping to index `1` is better because from there we can jump directly to the end.
- Therefore, the optimal sequence is:

- `0 -> 1 -> 4`
- Total jumps = `2`

The constraints are important:

- `nums.length` can be as large as `10^4`
- Each value can be up to `1000`

Because the array can be fairly large, an exponential or quadratic exploration of all paths may become too slow. This strongly suggests that we need a linear or near linear solution.

There are several important edge cases to consider:

- An array with only one element, such as `[0]`. We are already at the destination, so the answer is `0`.
- Arrays where many small jumps are possible, forcing careful selection of the optimal path.
- Arrays containing zeros in the middle, such as `[2,3,0,1,4]`. A naive greedy strategy could accidentally jump into a dead region if not implemented carefully.
- Arrays where one large jump can skip most of the array.

The guarantee that the last index is reachable simplifies the implementation because we never need to handle impossible states.

## Approaches

### Brute Force Approach

A straightforward solution is to try every possible jump recursively.

At each index, we can jump anywhere from:

```
i + 1
```

up to:

```
i + nums[i]
```

For each possible destination, we recursively compute the minimum jumps required to reach the end.

This approach is correct because it explores every possible path and chooses the minimum among them. However, it is extremely inefficient because the same subproblems are recomputed many times.

For example:

```
[2,3,1,1,4]
```

From index `0`, we explore index `1` and index `2`. Later, both recursive branches may again compute the minimum jumps from index `3`, causing repeated work.

The recursion tree grows exponentially in the worst case.

Dynamic programming can reduce the repeated computations, but even a DP solution may still require checking many transitions from each index, leading to `O(n^2)` time.

### Optimal Greedy Approach

The key insight is that we do not actually need to know the exact path at every step. Instead, we only need to know how far we can reach using the current number of jumps.

This idea is similar to breadth first search over ranges.

Suppose we are currently able to reach all indices within a certain range using a fixed number of jumps. While scanning that range, we compute the farthest position reachable with one additional jump.

Once we finish scanning the current range, we must make another jump, and the next reachable range becomes the farthest position we discovered.

This greedy strategy works because:

- Every index inside the current range is reachable with the same number of jumps.
- Among all those positions, we only care about the farthest future reach.
- Taking the jump only when we exhaust the current range guarantees the minimum number of jumps.

This produces an optimal linear time solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(n) | Recursively explores all possible jump sequences |
| Optimal Greedy | O(n) | O(1) | Expands reachable ranges level by level |

## Algorithm Walkthrough

1. Initialize three variables:

- `jumps` tracks how many jumps we have used.
- `current_end` marks the farthest index reachable with the current number of jumps.
- `farthest` stores the farthest index we can reach while scanning the current range.
2. Iterate through the array from index `0` to `n - 2`.

We stop at `n - 2` because once we reach the last index, no additional jump is needed.
3. For each index `i`, compute:

```
i + nums[i]
```

This represents the farthest position reachable from the current index.

1. Update `farthest` with the maximum reachable position seen so far.
2. If `i` reaches `current_end`, it means we have finished exploring all positions reachable with the current number of jumps.
3. At this moment:

- Increment `jumps`
- Set `current_end = farthest`

This effectively moves us to the next "layer" of reachable indices.
4. Continue until the loop finishes.
5. Return `jumps`.

The algorithm behaves similarly to level order traversal in a graph:

- One jump corresponds to one BFS layer.
- `current_end` defines the current layer boundary.
- `farthest` determines the next layer boundary.

## Python Solution

```
from typing import List

class Solution:
    def jump(self, nums: List[int]) -> int:
        n = len(nums)

        if n == 1:
            return 0

        jumps = 0
        current_end = 0
        farthest = 0

        for i in range(n - 1):
            farthest = max(farthest, i + nums[i])

            if i == current_end:
                jumps += 1
                current_end = farthest

        return jumps
```

The implementation begins by handling the simplest edge case. If the array contains only one element, we are already at the destination, so no jumps are required.

The variable `jumps` counts how many jumps have been used so far.

The variable `current_end` represents the boundary of the current reachable range. All indices up to this boundary are reachable using the current number of jumps.

The variable `farthest` keeps track of the farthest position we can reach while scanning the current range.

During each iteration, we compute:

```
i + nums[i]
```

This gives the farthest index reachable from the current position.

When the loop index reaches `current_end`, we have exhausted the current jump range. At that point, we must commit to another jump, so we increment `jumps` and update `current_end` to the farthest reachable position discovered so far.

Because the problem guarantees reachability, we never need to handle failure cases.

## Go Solution

```
func jump(nums []int) int {
	n := len(nums)

	if n == 1 {
		return 0
	}

	jumps := 0
	currentEnd := 0
	farthest := 0

	for i := 0; i < n-1; i++ {
		reach := i + nums[i]

		if reach > farthest {
			farthest = reach
		}

		if i == currentEnd {
			jumps++
			currentEnd = farthest
		}
	}

	return jumps
}
```

The Go implementation follows the same logic as the Python solution.

One small difference is that Go does not provide a built in `max` function for integers, so we update `farthest` manually using an `if` statement.

There are no integer overflow concerns because the constraints are small. Go slices are naturally passed by reference semantics, so no additional memory copying occurs.

The implementation also explicitly handles the single element array case.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each index is processed exactly once |
| Space | O(1) | Only a few integer variables are used |

The greedy algorithm performs a single pass through the array. Every iteration performs only constant time operations, so the total runtime is linear.

The space usage is constant because the algorithm stores only a few tracking variables regardless of input size.

## Edge Cases

### Single Element Array

Consider:

```
[0]
```

We already start at the last index, so the correct answer is `0`.

A buggy implementation might incorrectly count one jump immediately. The solution avoids this by explicitly returning `0` when the array length is `1`.

### Large Jump at the Beginning

Consider:

```
[10,1,1,1,1]
```

The optimal solution is a single jump directly to the end.

A naive algorithm that always jumps one step at a time would produce a non optimal answer. The greedy approach correctly tracks the farthest reachable position and recognizes that the end is already reachable within one jump.

### Arrays Containing Zeroes

Consider:

```
[2,3,0,1,4]
```

The zero at index `2` could trap naive implementations that greedily jump to the locally largest value without considering future reach.

The algorithm handles this correctly because it evaluates the entire reachable range before committing to the next jump. This ensures the best future boundary is always selected.

### Long Chains of Small Jumps

Consider:

```
[1,1,1,1,1]
```

Every move advances by only one index, so the answer becomes `n - 1`.

This case verifies that the algorithm properly increments jumps whenever the current reachable range is exhausted.
