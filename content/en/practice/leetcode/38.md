## Solution

## Problem Understanding

The problem asks us to generate the `n`th term in the "count-and-say" sequence. This sequence is defined recursively, meaning each term is created from the previous one.

The sequence begins with:

```
1
```

To generate the next term, we "read" the previous term by counting consecutive repeated digits and describing them as:

```
<count><digit>
```

For example:

```
"21"
```

is read as:

- one `2`
- one `1`

which becomes:

```
"1211"
```

The input is a single integer `n`, where `1 <= n <= 30`. The output is the string representing the `n`th term in the sequence.

The constraint is relatively small, which tells us that we do not need highly advanced optimizations. However, the strings grow in length as `n` increases, so repeatedly rebuilding strings inefficiently could still become expensive.

An important detail is that this is not numeric computation. Even though the sequence contains digits, we must treat each term as a string.

Several edge cases are important:

- `n = 1` is the base case and should immediately return `"1"`.
- Runs of repeated digits can have varying lengths, such as `"111"` becoming `"31"`.
- Adjacent groups must be processed independently. For example, `"3322251"` contains multiple separate runs.
- The final run in a string must always be appended after the loop ends, otherwise it gets lost.

## Approaches

### Brute Force Approach

The brute-force idea is to recursively generate every previous sequence until we reach `n`.

For each level:

1. Generate `countAndSay(n - 1)`
2. Run-length encode the resulting string
3. Return the encoded result

This works because the problem definition itself is recursive.

However, recursion introduces unnecessary function call overhead and repeated stack usage. Since each term only depends on the immediately previous term, recursion is not required. Deep recursion can also make the solution harder to follow and less memory efficient.

### Key Insight for the Optimal Solution

The most important observation is that each term depends only on the previous term.

That means we can iteratively build the sequence from:

```
"1"
```

up to the `n`th term.

For every iteration:

- Scan the current string from left to right
- Count consecutive identical digits
- Append:

- the count
- then the digit itself

This is essentially a standard run-length encoding process.

Using iteration avoids recursion overhead and keeps the implementation simple and efficient.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force Recursive | O(T) | O(T + n) | Recursively builds every sequence, additional recursion stack usage |
| Optimal Iterative | O(T) | O(T) | Iteratively generates each term using run-length encoding |

Here, `T` represents the total number of characters processed across all generated strings.

## Algorithm Walkthrough

1. Start with the base sequence `"1"` because `countAndSay(1) = "1"`.
2. Repeat the transformation process `n - 1` times because the first sequence already exists.
3. For each iteration, create an empty list or builder to construct the next sequence efficiently.
4. Scan the current sequence from left to right using an index.
5. For the current digit, count how many consecutive times it appears.
6. Append the count followed by the digit itself into the builder.
7. Move the index forward until the entire string has been processed.
8. Convert the builder into the next sequence string.
9. Replace the current sequence with the newly generated sequence.
10. After all iterations finish, return the final sequence.

The reason we use a list or string builder instead of repeated string concatenation is efficiency. Strings are immutable in both Python and Go, so repeatedly appending directly to strings would create many unnecessary intermediate strings.

## Python Solution

```
class Solution:
    def countAndSay(self, n: int) -> str:
        current = "1"

        for _ in range(n - 1):
            next_sequence = []
            index = 0

            while index < len(current):
                current_digit = current[index]
                count = 0

                while index < len(current) and current[index] == current_digit:
                    count += 1
                    index += 1

                next_sequence.append(str(count))
                next_sequence.append(current_digit)

            current = "".join(next_sequence)

        return current
```

The implementation begins with the base sequence `"1"`.

The outer loop runs `n - 1` times because the first sequence is already known. Each iteration transforms the current sequence into the next one.

Inside the loop, we use a `while` loop to scan the current string. For each group of consecutive identical digits, we count how many times the digit appears.

Once the run ends, we append two things into `next_sequence`:

- the count
- the digit itself

For example, if the current run is `"222"`, we append:

```
"3"
"2"
```

At the end of the scan, `"".join(next_sequence)` creates the next sequence string efficiently.

Finally, after all iterations complete, we return the generated sequence.

## Go Solution

```
import (
	"strconv"
	"strings"
)

func countAndSay(n int) string {
	current := "1"

	for i := 0; i < n-1; i++ {
		var builder strings.Builder
		index := 0

		for index < len(current) {
			currentDigit := current[index]
			count := 0

			for index < len(current) && current[index] == currentDigit {
				count++
				index++
			}

			builder.WriteString(strconv.Itoa(count))
			builder.WriteByte(currentDigit)
		}

		current = builder.String()
	}

	return current
}
```

The Go solution follows the same logic as the Python version, but there are a few implementation differences.

Go strings are immutable, so we use `strings.Builder` for efficient string construction. This avoids repeated allocations during concatenation.

The `strconv.Itoa()` function converts the integer count into a string before appending it to the builder.

Characters in Go strings are accessed as bytes using indexing like:

```
current[index]
```

Since the input only contains digits, byte-based indexing is completely safe.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(T) | Every character across all generated sequences is processed once |
| Space | O(T) | The generated strings themselves require storage |

The exact string lengths grow with each iteration, so the total work depends on the cumulative number of characters generated across all terms.

Within each iteration, every character is visited exactly once during run-length encoding. No nested rescanning occurs beyond processing consecutive groups.

The auxiliary space comes primarily from constructing the next sequence string.

## Edge Cases

### Edge Case 1, `n = 1`

This is the base case of the recursive definition. A common bug is accidentally entering the transformation loop and modifying the sequence unnecessarily.

The implementation handles this naturally because the outer loop runs:

```
for _ in range(n - 1):
```

When `n = 1`, the loop executes zero times and immediately returns `"1"`.

### Edge Case 2, Long Repeated Runs

A sequence may contain long runs such as:

```
"111"
```

which should become:

```
"31"
```

A naive implementation might incorrectly append `"111"` or fail to count properly.

The solution correctly counts consecutive digits using the inner `while` loop before appending the encoded result.

### Edge Case 3, Final Character Group

One of the most common mistakes in run-length encoding problems is forgetting to process the final run after iteration ends.

For example:

```
"21"
```

contains two separate runs:

- one `2`
- one `1`

The implementation avoids this issue because each run is processed immediately inside the loop before moving to the next group. No special post-processing step is needed.

### Edge Case 4, Alternating Digits

Inputs such as:

```
"1211"
```

contain both single-character runs and multi-character runs.

The algorithm correctly separates them into:

- one `1`
- one `2`
- two `1`s

producing:

```
"111221"
```

This works because the index advances exactly to the end of each consecutive group before processing the next one.
