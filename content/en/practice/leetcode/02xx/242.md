## Solution

## Problem Understanding

The problem asks us to determine whether two strings are anagrams of each other. Two strings are considered anagrams if they contain exactly the same characters with exactly the same frequencies, but possibly in a different order.

For example, the strings `"anagram"` and `"nagaram"` are anagrams because both contain the same letters:

- Three `'a'` characters
- One `'n'`
- One `'g'`
- One `'r'`
- One `'m'`

The order is different, but the character counts are identical.

On the other hand, `"rat"` and `"car"` are not anagrams because the characters differ. The first string contains `'t'`, while the second contains `'c'`.

The input consists of two strings, `s` and `t`. The output should be a boolean value:

- `true` if the two strings are anagrams
- `false` otherwise

The constraints tell us that the strings can be as large as `5 * 10^4` characters. This means we should avoid inefficient approaches such as repeatedly searching through strings or generating permutations. Since the strings only contain lowercase English letters, we can take advantage of the small fixed alphabet size of 26 characters.

Several important edge cases should be considered:

- The strings may have different lengths. In that case, they cannot possibly be anagrams.
- The strings may contain repeated characters many times.
- The strings may already be identical.
- Very small inputs such as single-character strings should still work correctly.
- A naive implementation might incorrectly ignore character frequency and only check whether the same characters appear at least once.

The follow-up question introduces Unicode characters. In that case, we cannot rely on a fixed-size array of 26 lowercase letters, because Unicode contains a very large set of possible characters. A hash map based counting approach becomes the preferred solution in that scenario.

## Approaches

### Brute-Force Approach

A straightforward way to solve the problem is to sort both strings and compare the results.

The reasoning is simple. If two strings are anagrams, then sorting their characters alphabetically will produce identical strings. For example:

- `"anagram"` becomes `"aaagmnr"`
- `"nagaram"` also becomes `"aaagmnr"`

If the sorted versions match, the strings are anagrams.

This approach is correct because sorting groups identical characters together, making frequency mismatches immediately visible.

However, sorting requires `O(n log n)` time, where `n` is the string length. While this is acceptable for the given constraints, it is not the most efficient possible solution.

### Optimal Approach

The key insight is that anagrams are fundamentally about character frequencies, not ordering.

Instead of sorting, we can count how many times each character appears in both strings. If every character appears the same number of times, then the strings are anagrams.

Since the problem guarantees lowercase English letters, we can use an array of size 26:

- Index `0` represents `'a'`
- Index `1` represents `'b'`
- ...
- Index `25` represents `'z'`

We increment counts for characters in `s` and decrement counts for characters in `t`. At the end:

- If every count is zero, the strings are anagrams.
- If any count is nonzero, they are not.

This avoids sorting entirely and achieves linear time complexity.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) | O(n) | Sort both strings and compare |
| Optimal | O(n) | O(1) | Count character frequencies using a fixed-size array |

## Algorithm Walkthrough

### Optimal Frequency Counting Algorithm

1. First, check whether the lengths of `s` and `t` are equal.

This is an important early optimization. If the strings have different lengths, they cannot possibly contain the same characters with the same frequencies.
2. Create a frequency array of size 26 initialized with zeros.

Each index corresponds to one lowercase English letter. This array will track the difference in character counts between the two strings.
3. Iterate through both strings simultaneously.

For each position:

- Increment the count for the character from `s`
- Decrement the count for the character from `t`

This efficiently combines both counting operations into a single pass.
4. After processing all characters, examine the frequency array.

If every value is zero, then every character appeared the same number of times in both strings.
5. If any value is nonzero, return `false`.

A positive value means a character appeared more times in `s`. A negative value means it appeared more times in `t`.
6. If all values are zero, return `true`.

### Why it works

The algorithm maintains the invariant that each entry in the frequency array represents the difference between the number of occurrences of a character in `s` and in `t`.

If the strings are anagrams, every character frequency difference must eventually become zero. Conversely, if even one character frequency differs, at least one array entry will remain nonzero. Therefore, the algorithm correctly determines whether the two strings are anagrams.

## Python Solution

```
class Solution:
    def isAnagram(self, s: str, t: str) -> bool:
        if len(s) != len(t):
            return False

        frequency = [0] * 26

        for i in range(len(s)):
            frequency[ord(s[i]) - ord('a')] += 1
            frequency[ord(t[i]) - ord('a')] -= 1

        for count in frequency:
            if count != 0:
                return False

        return True
```

The implementation begins with a length check because unequal lengths immediately rule out the possibility of an anagram.

The `frequency` array stores character count differences. The expression `ord(character) - ord('a')` converts a lowercase letter into an index from `0` to `25`.

The loop processes both strings simultaneously. Each character from `s` increases its corresponding count, while each character from `t` decreases it. This efficiently tracks frequency differences in one pass.

Finally, the algorithm scans the frequency array. If any entry is not zero, the character frequencies differ and the strings are not anagrams. Otherwise, all frequencies match exactly.

## Go Solution

```
func isAnagram(s string, t string) bool {
    if len(s) != len(t) {
        return false
    }

    frequency := make([]int, 26)

    for i := 0; i < len(s); i++ {
        frequency[s[i]-'a']++
        frequency[t[i]-'a']--
    }

    for _, count := range frequency {
        if count != 0 {
            return false
        }
    }

    return true
}
```

The Go implementation follows the same logic as the Python version. A slice of 26 integers is used for frequency counting.

One Go-specific detail is that string indexing returns bytes rather than Unicode characters. This works correctly here because the problem guarantees lowercase English letters, which are single-byte ASCII characters.

There are no concerns about integer overflow because the maximum string length is only `5 * 10^4`.

## Edge Cases

### Different String Lengths

If the strings have different lengths, they cannot be anagrams because anagrams must contain exactly the same number of characters.

A naive implementation might still attempt unnecessary processing. The solution avoids this by performing an immediate length check at the beginning and returning `false` early.

### Repeated Characters

Repeated characters are a common source of bugs. For example, `"aacc"` and `"ccac"` contain similar characters, but the frequencies differ.

An implementation that only checks whether characters exist would incorrectly return `true`. The frequency counting approach correctly tracks exact counts, ensuring mismatches are detected.

### Identical Strings

Two identical strings are always valid anagrams because they contain exactly the same characters and frequencies.

Some implementations overcomplicate this case, but the frequency counting logic naturally handles it correctly. Every increment from `s` is canceled by the corresponding decrement from `t`, leaving all counts at zero.

### Very Small Inputs

Single-character strings such as `"a"` and `"b"` test whether the implementation handles minimal input sizes correctly.

The algorithm works uniformly regardless of string length. Even for length one, the same counting logic applies without requiring any special handling.

### Unicode Characters

The current solution relies on a fixed-size array because the input only contains lowercase English letters.

If Unicode characters were allowed, the array approach would no longer work because Unicode contains far more than 26 possible characters. In that case, a hash map or dictionary should be used instead:

- Increment counts for characters in `s`
- Decrement counts for characters in `t`
- Verify all counts are zero

This generalized approach works for arbitrary character sets while maintaining linear time complexity.
