# LeetCode 248 - Strobogrammatic Number III

**Difficulty:** Hard

**Topics:** Array, String, Recursion

## Problem Understanding

This problem asks us to count how many **strobogrammatic numbers** exist within a given inclusive range `[low, high]`, where both boundaries are represented as strings.

A **strobogrammatic number** is a number that appears identical after being rotated `180` degrees. In other words, if we flip the number upside down, it still forms a valid number with the same appearance.

Only certain digit pairs remain valid after rotation:

| Digit | Rotates To |
| --- | --- |
| `0` | `0` |
| `1` | `1` |
| `6` | `9` |
| `8` | `8` |
| `9` | `6` |

For example:

- `"69"` becomes `"69"` after rotation because `6 → 9` and `9 → 6`
- `"88"` remains `"88"`
- `"818"` remains `"818"`
- `"2"` is not valid because `2` does not map to any digit after rotation

The input consists of two strings, `low` and `high`, representing the lower and upper bounds of the range. We must return the total number of strobogrammatic numbers such that:

```
low <= number <= high
```

Since the inputs are strings, the problem avoids integer overflow issues for very large numbers. The maximum length is `15`, meaning numbers may be too large for standard integer arithmetic in some languages. This strongly suggests that we should primarily work with strings rather than converting everything to integers.

The constraints also reveal an important insight. Since `high.length <= 15`, brute force iteration across the numeric range may become infeasible. For example:

```
low = "1"
high = "100000000000000"
```

Iterating over every number would require checking nearly `10^15` candidates, which is computationally impossible.

Instead, we need a way to **generate only valid strobogrammatic numbers** and count those within range.

Several edge cases deserve attention immediately. Numbers cannot contain leading zeros unless the number itself is `"0"`, meaning `"0110"` is invalid even if it looks strobogrammatic. The boundaries are inclusive, so values equal to `low` or `high` must be counted. Single digit numbers also require special treatment because only `"0"`, `"1"`, and `"8"` are valid single digit strobogrammatic numbers.

## Approaches

### Brute Force Approach

A straightforward solution would be to iterate through every number between `low` and `high`, convert each number to a string, and check whether it is strobogrammatic.

To determine if a number is strobogrammatic, we could use two pointers:

- One pointer starts at the beginning.
- Another starts at the end.
- At each step, verify whether the digit pair forms a valid rotational mapping.

For example:

```
"69"
6 maps to 9
9 maps to 6
Valid
```

This approach is correct because every number in the range is examined exactly once, and every candidate is validated accurately.

However, the performance is unacceptable. If the range spans up to `10^15`, brute force would require checking trillions of numbers.

### Optimal Approach, Recursive Generation

The key observation is that **the number of valid strobogrammatic numbers is extremely small compared to all possible numbers**.

Instead of checking every number, we can **generate only valid strobogrammatic numbers**.

A strobogrammatic number is symmetric under rotational mapping. This means we can construct valid numbers recursively from the center outward using valid digit pairs:

```
("0", "0")
("1", "1")
("6", "9")
("8", "8")
("9", "6")
```

For example:

Starting with an empty center:

```
""
```

We can build:

```
"11"
"69"
"88"
"96"
```

Then continue expanding:

```
"1691"
"6889"
"8008"
```

We generate numbers for every length between:

```
len(low) and len(high)
```

Then we filter:

1. Skip numbers with leading zeros.
2. Skip values smaller than `low`.
3. Skip values larger than `high`.

Because the maximum length is only `15`, the total number of generated candidates remains manageable.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O((high - low) × n) | O(n) | Checks every number individually |
| Optimal | O(5^(n/2)) | O(n) | Generates only valid strobogrammatic numbers |

Here, `n` is the maximum number of digits.

## Algorithm Walkthrough

### Step 1: Define Valid Rotation Pairs

We first define the valid strobogrammatic digit mappings:

```
(0,0)
(1,1)
(6,9)
(8,8)
(9,6)
```

These are the only digit combinations that remain valid after a `180` degree rotation.

### Step 2: Generate Numbers Recursively

We recursively generate strobogrammatic numbers of a specific length.

The recursion works from the inside outward.

For example, to generate length `4`:

1. Generate all valid length `2` middles.
2. Add valid digit pairs around them.

Example:

```
Middle = "88"

Add:
1 + 88 + 1 = "1881"
6 + 88 + 9 = "6889"
```

### Step 3: Handle Base Cases

We need two recursion base cases.

If remaining length is `0`:

```
[""]
```

This acts as an empty center.

If remaining length is `1`:

```
["0", "1", "8"]
```

Only these digits are valid in the middle position.

Digits like `6` and `9` cannot appear alone because rotation changes them.

### Step 4: Avoid Leading Zeros

When constructing outer layers, we must avoid numbers like:

```
"0110"
```

This is invalid due to leading zero.

We allow:

```
"0"
```

but disallow:

```
"00"
"0110"
```

We accomplish this by skipping:

```
("0","0")
```

when building the outermost layer.

### Step 5: Validate Against Range

For each generated number:

- If same length as `low`, ensure:

```
candidate >= low
```

- If same length as `high`, ensure:

```
candidate <= high
```

Since all numbers have equal length in those comparisons, lexicographical string comparison works correctly.

### Step 6: Count Valid Results

Every candidate satisfying all constraints increments the count.

Finally, return the total.

### Why it works

The algorithm works because every strobogrammatic number can be uniquely constructed by recursively surrounding a smaller strobogrammatic number with valid rotational digit pairs. No valid number is missed, and no invalid number is generated. Since we explicitly enforce range constraints and leading zero rules, the final count contains exactly the valid strobogrammatic numbers in `[low, high]`.

## Python Solution

```
class Solution:
    def strobogrammaticInRange(self, low: str, high: str) -> int:
        pairs = [
            ("0", "0"),
            ("1", "1"),
            ("6", "9"),
            ("8", "8"),
            ("9", "6")
        ]

        def generate(current_length: int, target_length: int) -> list[str]:
            if current_length == 0:
                return [""]

            if current_length == 1:
                return ["0", "1", "8"]

            middles = generate(current_length - 2, target_length)
            result = []

            for middle in middles:
                for left, right in pairs:
                    if (
                        current_length == target_length
                        and left == "0"
                    ):
                        continue

                    result.append(left + middle + right)

            return result

        count = 0
        low_length = len(low)
        high_length = len(high)

        for length in range(low_length, high_length + 1):
            candidates = generate(length, length)

            for candidate in candidates:
                if (
                    length == low_length
                    and candidate < low
                ):
                    continue

                if (
                    length == high_length
                    and candidate > high
                ):
                    continue

                count += 1

        return count
```

The implementation begins by defining the valid rotational digit pairs. These represent the only legal ways to build a strobogrammatic number.

The `generate()` function performs recursive construction. The parameter `current_length` tracks how many digits remain to build, while `target_length` helps detect whether we are constructing the outermost layer. This distinction matters because leading zeros are forbidden only at the outer boundary.

The base case for length `0` returns an empty string. This provides a center for even length numbers. The base case for length `1` returns `["0", "1", "8"]`, which are the only valid center digits.

For larger lengths, recursion generates all smaller middle sections, then wraps each with valid digit pairs.

After generation, the main loop iterates through every valid length between `low` and `high`. Each candidate undergoes range filtering using string comparison. Since equal length numeric strings compare lexicographically in numeric order, this avoids integer conversion entirely.

## Go Solution

```
func strobogrammaticInRange(low string, high string) int {
	pairs := [][2]string{
		{"0", "0"},
		{"1", "1"},
		{"6", "9"},
		{"8", "8"},
		{"9", "6"},
	}

	var generate func(int, int) []string

	generate = func(currentLength int, targetLength int) []string {
		if currentLength == 0 {
			return []string{""}
		}

		if currentLength == 1 {
			return []string{"0", "1", "8"}
		}

		middles := generate(currentLength-2, targetLength)
		result := []string{}

		for _, middle := range middles {
			for _, pair := range pairs {
				left := pair[0]
				right := pair[1]

				if currentLength == targetLength && left == "0" {
					continue
				}

				result = append(
					result,
					left+middle+right,
				)
			}
		}

		return result
	}

	count := 0
	lowLength := len(low)
	highLength := len(high)

	for length := lowLength; length <= highLength; length++ {
		candidates := generate(length, length)

		for _, candidate := range candidates {
			if length == lowLength && candidate < low {
				continue
			}

			if length == highLength && candidate > high {
				continue
			}

			count++
		}
	}

	return count
}
```

The Go solution follows the same recursive strategy as the Python implementation. The main difference is the use of slices instead of Python lists.

The recursive helper is declared as a function variable because Go requires explicit recursive function declarations.

String comparison behaves lexicographically in Go, just like Python, so we can compare equal length numeric strings directly without integer conversion. This avoids overflow concerns for numbers near the `15` digit constraint.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(5^(n/2)) | Recursive generation explores valid digit pair combinations |
| Space | O(n) | Recursion depth and temporary construction |

The dominant cost comes from generating strobogrammatic candidates. Each recursive level introduces up to `5` choices for digit pairs, and recursion depth is approximately `n / 2`. Since `n ≤ 15`, this remains practical. The algorithm is dramatically faster than brute force because it generates only valid candidates instead of scanning the entire numeric range.

## Edge Cases

### Edge Case 1: Single Digit Inputs

A common source of bugs is mishandling single digit numbers. Only `"0"`, `"1"`, and `"8"` are valid strobogrammatic digits by themselves. Digits like `"6"` and `"9"` rotate into each other, meaning they cannot stand alone.

The implementation explicitly handles this with the recursion base case:

```
["0", "1", "8"]
```

This guarantees correct behavior for ranges like:

```
low = "0"
high = "9"
```

### Edge Case 2: Leading Zeros

Numbers such as:

```
"0110"
```

are technically symmetric but invalid because the problem forbids leading zeros.

This can easily introduce incorrect counts if `"0"` is always added as an outer pair.

The implementation solves this by skipping:

```
("0","0")
```

when constructing the outermost layer.

As a result:

```
"00"
```

is never counted, while:

```
"0"
```

remains valid.

### Edge Case 3: Boundary Comparisons

The range is inclusive, meaning exact boundary matches must count.

For example:

```
low = "69"
high = "69"
```

should return:

```
1
```

A careless implementation might accidentally exclude equal values.

The algorithm uses:

```
candidate < low
candidate > high
```

instead of non inclusive comparisons, ensuring equal boundary values are preserved.

### Edge Case 4: Different Number Lengths

Ranges may span multiple digit lengths:

```
low = "8"
high = "1000"
```

A naive implementation might only generate numbers matching the boundary lengths.

The solution instead iterates through:

```
range(len(low), len(high) + 1)
```

ensuring all valid lengths are considered. This guarantees no strobogrammatic numbers are skipped.
