## Solution

## Problem Understanding

The problem asks us to take a binary tree and invert it. Inverting a binary tree means swapping the left and right child of every node in the tree. After all swaps are completed, we return the root of the modified tree.

A binary tree is a hierarchical data structure where each node can have at most two children, typically called `left` and `right`. The input is provided as the root node of the tree. The output should be the same tree structure, but mirrored across its center.

For example, consider this tree:

```
        4
      /   \
     2     7
    / \   / \
   1   3 6   9
```

After inversion, every node swaps its children:

```
        4
      /   \
     7     2
    / \   / \
   9   6 3   1
```

The constraints are small. The number of nodes is at most 100, which means almost any reasonable tree traversal approach will run efficiently enough. Even though the input size is small, the goal is to demonstrate proper understanding of tree traversal and recursion or iterative traversal techniques.

An important observation is that every node must eventually be visited because each node may have children that need swapping. This naturally suggests a tree traversal algorithm such as Depth-First Search (DFS) or Breadth-First Search (BFS).

There are several edge cases that must be handled correctly. An empty tree, where `root` is `None`, should immediately return `None`. A tree with only one node should remain unchanged because there are no children to swap. Skewed trees, where nodes only exist on one side, must also work correctly because the inversion changes the direction of the chain.

## Approaches

A brute-force style approach would first create a completely new mirrored tree instead of modifying the original tree in place. For every node in the original tree, we would allocate a new node and recursively assign its left child using the original right subtree, and its right child using the original left subtree.

This approach is correct because it explicitly constructs the mirrored structure from scratch. However, it requires additional memory proportional to the size of the tree because an entirely new tree is allocated. Since the problem does not require preserving the original tree, this extra allocation is unnecessary.

The key insight for a better solution is that inversion only requires swapping pointers. We do not need to create new nodes or store additional structures permanently. Every node already contains references to its left and right children, so we can simply exchange them in place.

A recursive Depth-First Search traversal is especially natural here. For each node:

1. Swap its left and right children.
2. Recursively invert the left subtree.
3. Recursively invert the right subtree.

Because every node is visited exactly once and only constant work is done per node, the algorithm is efficient and clean.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Builds an entirely new mirrored tree |
| Optimal | O(n) | O(h) | Swaps child pointers in place using DFS recursion |

Here, `n` is the number of nodes and `h` is the height of the tree.

## Algorithm Walkthrough

### Recursive DFS Inversion

1. Start at the root node of the tree. If the current node is `None`, return immediately because there is nothing to invert.
2. Swap the current node's left and right child pointers. This is the core operation that performs the inversion locally for the current node.
3. Recursively process the new left subtree. After the swap, the original right subtree is now on the left side, so we recursively invert it.
4. Recursively process the new right subtree. Similarly, the original left subtree is now on the right side and must also be inverted.
5. Continue this process until all nodes have been visited. Leaf nodes naturally terminate recursion because both child pointers are `None`.
6. Return the root node after all recursive calls complete. The entire tree will now be inverted in place.

### Why it works

The algorithm works because inversion is a local operation that can be independently applied at every node. Swapping a node's children correctly mirrors that node, and recursively applying the same logic to its subtrees guarantees that the entire structure becomes mirrored. Since recursion visits every node exactly once, every subtree is inverted correctly.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import Optional

class Solution:
    def invertTree(self, root: Optional[TreeNode]) -> Optional[TreeNode]:
        if root is None:
            return None

        root.left, root.right = root.right, root.left

        self.invertTree(root.left)
        self.invertTree(root.right)

        return root
```

The implementation begins with the base case. If the current node is `None`, the function immediately returns `None`. This prevents recursion from continuing past leaf nodes.

Next, the left and right child pointers are swapped using Python's tuple assignment syntax. This performs the inversion operation for the current node in a concise and safe way.

After swapping, recursion is applied to both subtrees. Because the swap already occurred, the recursive calls operate on the newly positioned children.

Finally, the root node is returned. Since the tree is modified in place, the returned root represents the fully inverted tree.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func invertTree(root *TreeNode) *TreeNode {
    if root == nil {
        return nil
    }

    root.Left, root.Right = root.Right, root.Left

    invertTree(root.Left)
    invertTree(root.Right)

    return root
}
```

The Go implementation follows the same recursive structure as the Python version. The main difference is the use of `nil` instead of `None`. Go also uses pointer semantics explicitly, so the function receives a `*TreeNode` pointer and modifies the tree directly through that pointer.

Go supports tuple-style assignment for swapping values, which makes the child exchange concise and readable.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every node is visited exactly once |
| Space | O(h) | Recursive call stack stores at most tree height frames |

The algorithm performs one swap operation per node and recursively visits each node once, so the total runtime is linear in the number of nodes.

The auxiliary space complexity comes from recursion. In the worst case of a completely skewed tree, the recursion depth becomes `n`. In a balanced tree, the recursion depth is approximately `log n`.

## Edge Cases

An empty tree is one of the most important edge cases. If `root` is `None`, attempting to access child pointers would raise errors in Python or cause invalid memory access in Go. The implementation avoids this by immediately returning when the current node is null.

A tree containing only a single node can also expose incorrect recursion handling. Since the node has no children, swapping should leave the tree unchanged. The recursive solution naturally handles this because swapping two `None` references has no effect.

Skewed trees are another important edge case. In a left-skewed tree, every node only has a left child. After inversion, the structure becomes right-skewed. Implementations that incorrectly process child pointers before swapping can accidentally lose subtree references. The current implementation safely swaps the pointers first, then recursively processes the updated children.

Another subtle edge case involves trees with missing children at intermediate levels. For example, a node may have only a right child but no left child. The algorithm still works because swapping correctly moves the existing child to the opposite side while preserving the `None` reference on the other side.
