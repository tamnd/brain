## Solution

## Problem Understanding

The problem asks us to determine whether an integer array contains any duplicate values. In other words, we need to check if at least one number appears more than once anywhere in the array.

The input is an array named `nums`, where each element is an integer. The output is a boolean value:

- Return `true` if any value appears at least twice
- Return `false` if every element in the array is unique

For example, in the array `[1,2,3,1]`, the value `1` appears two times, so the correct result is `true`. In the array `[1,2,3,4]`, every number is distinct, so the result is `false`.

The constraints are important:

- The array length can be as large as `10^5`
- Values can range from `-10^9` to `10^9`

These constraints immediately tell us that efficiency matters. A solution with quadratic time complexity, `O(n^2)`, would be too slow for arrays of size `100,000`. We need a solution closer to linear time.

There are several important edge cases to consider:

- Arrays with only one element should always return `false`, because a single value cannot have duplicates.
- Arrays where duplicates appear immediately next to each other should be handled correctly.
- Arrays where duplicates appear far apart should also be detected correctly.
- Negative numbers must be handled the same way as positive numbers.
- Large inputs with many unique values should still run efficiently.

The problem guarantees that the input array is valid and contains at least one element, so we do not need to handle null or missing input cases.

## Approaches

### Brute Force Approach

The most straightforward solution is to compare every element with every other element in the array.

For each index `i`, we scan all later indices `j` and check whether `nums[i] == nums[j]`. If we ever find two equal values, we return `true`. If all comparisons finish without finding a match, we return `false`.

This approach is correct because it exhaustively checks every possible pair of elements. If a duplicate exists, we are guaranteed to find it.

However, this method is too slow for the given constraints. With `n` elements, we may perform roughly `n * (n - 1) / 2` comparisons. For `n = 100,000`, this becomes billions of operations, which is not practical.

### Optimal Approach Using a Hash Set

The key insight is that we do not need to compare every pair of elements directly. Instead, we only need to know whether we have seen a value before.

A hash set is ideal for this because it provides average `O(1)` lookup and insertion time.

As we iterate through the array:

- If the current number is already in the set, then we have found a duplicate and can immediately return `true`.
- Otherwise, we add the number to the set and continue.

This reduces the overall runtime to linear time, `O(n)`.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Compares every pair of elements |
| Optimal | O(n) | O(n) | Uses a hash set to track seen values |

## Algorithm Walkthrough

### Optimal Hash Set Algorithm

1. Create an empty hash set called `seen`.

The set will store all numbers we have encountered so far while scanning the array. A set is chosen because it supports very fast membership checks.
2. Iterate through each number in the array.

We process the array from left to right, examining one value at a time.
3. For the current number, check whether it already exists in `seen`.

If the number is already present, that means we encountered the same value earlier in the array. Therefore, a duplicate exists, and we immediately return `true`.
4. If the number is not in the set, add it to `seen`.

This records that we have now encountered this value.
5. Continue until the iteration finishes.

If we process the entire array without finding any repeated value, then all elements are distinct.
6. Return `false`.

This indicates that no duplicates were found.

### Why it works

The algorithm maintains an important invariant: before processing each element, the set `seen` contains exactly all previously encountered values.

When we encounter a number:

- If it is already in the set, then it must have appeared earlier, proving a duplicate exists.
- If it is not in the set, we add it so future occurrences can be detected.

Because every element is checked exactly once, and every previously seen value is tracked, the algorithm correctly detects duplicates whenever they exist.

## Python Solution

```
from typing import List

class Solution:
    def containsDuplicate(self, nums: List[int]) -> bool:
        seen = set()

        for number in nums:
            if number in seen:
                return True

            seen.add(number)

        return False
```

The implementation begins by creating an empty set called `seen`. This set stores all values encountered while iterating through the array.

The loop processes each number one at a time. For every number, the code first checks whether it already exists in the set. If it does, the function immediately returns `True` because a duplicate has been found.

If the number has not been seen before, it is added to the set so future occurrences can be detected.

If the loop completes without returning early, then every value was unique, so the function returns `False`.

This implementation directly follows the algorithm described earlier and efficiently handles the large input constraints.

## Go Solution

```
func containsDuplicate(nums []int) bool {
    seen := make(map[int]bool)

    for _, number := range nums {
        if seen[number] {
            return true
        }

        seen[number] = true
    }

    return false
}
```

In Go, there is no built in set type, so we simulate a set using a map from `int` to `bool`.

The map tracks whether a number has already been encountered. If `seen[number]` is `true`, then the number has appeared before and we return `true`.

Unlike Python, Go distinguishes between slices and arrays. The function accepts a slice `[]int`, which is the standard representation used by LeetCode. Empty and nil slices are both handled naturally by the loop without requiring special logic.

Integer overflow is not a concern because we are only storing and comparing integers, not performing arithmetic operations on them.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each element is processed once, with average O(1) set operations |
| Space | O(n) | In the worst case, all elements are stored in the set |

The runtime is linear because each number is visited exactly once. Hash set membership checks and insertions are constant time on average.

The space complexity is also linear because, in the worst case where all elements are unique, every value must be stored in the set.

## Edge Cases

### Single Element Array

An array with only one element can never contain duplicates. This case is important because some implementations incorrectly assume at least two elements exist.

For example:

```
[7]
```

The algorithm handles this naturally. The number is added to the set, the loop finishes, and the function returns `false`.

### Duplicate Appears Late

A duplicate may appear only near the end of a very large array.

For example:

```
[1,2,3,4,5,...,9999,1]
```

A naive solution with nested loops becomes very expensive in this situation because it repeatedly compares many pairs. The hash set approach remains efficient because each lookup is constant time on average.

The implementation correctly detects the duplicate when the second `1` is encountered.

### Negative Numbers

Some incorrect implementations accidentally assume all values are positive or use array indexing tricks that fail for negative numbers.

For example:

```
[-5,-2,-8,-5]
```

The hash set solution works correctly because sets store arbitrary integers, including negative values. The second occurrence of `-5` is detected immediately.

### All Unique Values

An array may contain no duplicates at all.

For example:

```
[1,2,3,4,5]
```

This case is important because the algorithm must correctly finish scanning the entire array before returning `false`.

The implementation processes every value, adds each one to the set, and finally returns `false` after confirming no duplicates exist.
