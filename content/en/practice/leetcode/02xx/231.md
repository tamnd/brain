## Solution

## Problem Understanding

The problem asks us to determine whether a given integer `n` is a power of two.

A number is considered a power of two if it can be written in the form:

$n = 2^x$

where `x` is a non-negative integer.

This means valid powers of two include:

- `1 = 2^0`
- `2 = 2^1`
- `4 = 2^2`
- `8 = 2^3`
- `16 = 2^4`

and so on.

The input is a single integer `n`, and the output should be a boolean value:

- `true` if `n` is a power of two
- `false` otherwise

The constraints specify that the integer fits within the 32-bit signed integer range:

$-2^{31} \le n \le 2^{31}-1$

This tells us several important things about the problem:

- Negative values are possible
- Zero is possible
- The input size is extremely small, so efficiency is not difficult to achieve
- Bit manipulation solutions are especially appropriate because the numbers fit comfortably within standard integer types

There are several important edge cases to consider.

The first is `n <= 0`. Powers of two are always positive, so any non-positive number must immediately return `false`.

The second edge case is `n = 1`. Many implementations accidentally reject this case, but `1` is a valid power of two because:

$2^0 = 1$

Another important case is numbers that are very close to powers of two, such as `15` or `31`. A naive implementation that only checks divisibility carelessly could produce incorrect results.

Finally, we should handle the largest valid powers of two within 32-bit range correctly, such as:

$2^{30} = 1073741824$

## Approaches

### Brute Force Approach

The brute-force approach repeatedly divides the number by `2` while it remains even.

The algorithm works like this:

- If `n <= 0`, return `false`
- While `n` is divisible by `2`, divide it by `2`
- At the end, if the remaining value is exactly `1`, then the original number was a power of two

For example, for `n = 16`:

- `16 → 8 → 4 → 2 → 1`

Since the process ends at `1`, the number is a power of two.

For `n = 12`:

- `12 → 6 → 3`

The process stops at `3`, so the number is not a power of two.

This approach is correct because every power of two can be reduced to `1` by repeatedly dividing by `2`. Any number containing another prime factor will eventually stop before reaching `1`.

Although this solution is already efficient enough for the constraints, it still performs multiple iterations and does not satisfy the follow-up requirement asking for a solution without loops or recursion.

### Optimal Bit Manipulation Approach

The key observation is based on the binary representation of powers of two.

A power of two always contains exactly one set bit in binary.

For example:

| Number | Binary |
| --- | --- |
| 1 | `0001` |
| 2 | `0010` |
| 4 | `0100` |
| 8 | `1000` |

If we subtract `1` from a power of two, all bits after the single set bit become `1`.

Examples:

| Number | Binary |
| --- | --- |
| 8 | `1000` |
| 7 | `0111` |

Notice that these two numbers have no overlapping set bits.

This leads to the critical property:

$n \& (n-1) = 0$

This expression is true only when `n` has exactly one set bit.

We must also ensure `n > 0`, because `0` incorrectly satisfies the bitwise condition.

The final condition becomes:

$n > 0 \;\text{and}\; (n \& (n-1)) = 0$

This solution runs in constant time and uses constant space.

## Algorithm Walkthrough

### Optimal Bit Manipulation Algorithm

1. First, check whether `n` is less than or equal to zero. This is necessary because powers of two are always positive integers. If `n <= 0`, immediately return `false`.
2. Compute the expression:

$n \& (n-1)$

This operation removes the lowest set bit from `n`.
3. If the result equals `0`, then `n` had exactly one set bit in its binary representation. Numbers with exactly one set bit are powers of two.
4. Return whether the condition is true.

### Why it works

A power of two contains exactly one `1` bit in binary representation. Subtracting `1` flips that bit to `0` and turns all lower bits into `1`. Therefore, the original number and `n - 1` share no common set bits, making their bitwise AND equal to zero.

Any non-power-of-two number contains multiple set bits, so after subtracting `1`, at least one set bit remains shared, producing a non-zero AND result.

## Python Solution

```
class Solution:
    def isPowerOfTwo(self, n: int) -> bool:
        return n > 0 and (n & (n - 1)) == 0
```

The implementation directly follows the mathematical property discussed earlier.

The first condition checks whether the number is positive. This prevents invalid cases like `0` and negative numbers from incorrectly returning `true`.

The second condition uses the bitwise AND trick. The expression:

```
(n & (n - 1))
```

removes the lowest set bit from the number. If the result becomes `0`, then the original number had exactly one set bit, meaning it was a power of two.

The implementation is concise because Python supports bitwise operations directly on integers.

## Go Solution

```

```

The Go implementation is almost identical to the Python version because Go also supports standard bitwise operators.

One small difference is syntax:

- Go uses `&&` for logical AND

There are no overflow concerns here because the problem guarantees the input fits within 32-bit signed integer range, and the operation `n - 1` remains safe for all valid inputs.

## Edge Cases

One important edge case is `n = 0`. The binary trick alone would incorrectly treat zero as valid because:

$0 \& (-1) = 0$

Without the explicit positivity check, the algorithm would return `true`. The implementation avoids this by requiring `n > 0` before performing the bitwise test.

Another important edge case is negative numbers. In two's complement binary representation, negative values can produce unexpected bitwise behavior. Since powers of two are mathematically positive, the implementation immediately rejects all negative values using the same `n > 0` condition.

The case `n = 1` is also easy to mishandle. Some implementations incorrectly assume powers of two start at `2`. However:

$2^0 = 1$

so `1` must return `true`. The bit manipulation property handles this naturally because:

$1 \& 0 = 0$

Finally, numbers very close to powers of two, such as `15` or `31`, are useful stress cases. These numbers contain many set bits, so the AND operation remains non-zero. This confirms that the algorithm correctly distinguishes between exact powers of two and nearby values.
