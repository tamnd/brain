## Solution

## Problem Understanding

The problem asks us to implement a queue using only stack operations. A queue follows the FIFO, first in first out, principle. This means the first element inserted into the queue must be the first element removed. The required operations are:

- `push(x)`, insert an element at the back of the queue
- `pop()`, remove and return the front element
- `peek()`, return the front element without removing it
- `empty()`, determine whether the queue contains any elements

The challenge is that we are only allowed to use stacks internally. A stack follows the LIFO, last in first out, principle, which is the opposite ordering behavior of a queue. This mismatch is the core difficulty of the problem.

The input shown in the example represents a sequence of method calls made on the `MyQueue` object. The corresponding output array contains the return values of those operations. Operations like `push` return `null` because they do not produce a value, while `peek`, `pop`, and `empty` return actual results.

The constraints are very small, at most 100 operations, so even inefficient solutions would technically pass. However, the follow-up explicitly asks for an amortized `O(1)` solution, which is the real algorithmic objective. This means that while an individual operation might occasionally take linear time, the average cost across many operations should remain constant.

An important guarantee is that every `pop` and `peek` call is valid. This means the queue will never be empty when those operations are invoked, so we do not need to handle invalid removals.

Several edge cases are important to consider. The queue may contain only one element, which tests whether the implementation correctly transitions between empty and non-empty states. Another important case is performing many `push` operations before the first `pop`, since this triggers the internal transfer logic between stacks. Alternating between pushes and pops is also important because it verifies that the ordering remains correct throughout repeated state changes.

## Approaches

### Brute Force Approach

A straightforward way to simulate a queue with stacks is to use a single stack and completely rearrange elements during every insertion. Since stacks only allow access to the top element, we cannot directly place a new value at the back of the queue while preserving FIFO order.

To solve this, we can temporarily move all elements from the main stack into a helper stack, insert the new element into the now-empty main stack, and then move everything back. This guarantees that the oldest element always remains on top of the main stack, making `pop` and `peek` efficient.

This approach is correct because every insertion rebuilds the stack in exact queue order. However, each `push` operation requires moving every existing element twice, making insertion expensive.

### Optimal Approach

The key observation is that reversing the order of elements twice restores FIFO behavior. We can exploit this using two stacks:

- An input stack for incoming elements
- An output stack for removals and front access

When elements are pushed, they are simply appended to the input stack. When we need to pop or peek, we transfer elements from the input stack to the output stack only if the output stack is empty. This reversal places the oldest element on top of the output stack.

The important optimization is that each element moves from the input stack to the output stack at most once. Because of this, although some operations occasionally require `O(n)` work, the average cost across many operations is `O(1)`.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Push: O(n), Pop: O(1) | O(n) | Reorders all elements during every insertion |
| Optimal | Amortized O(1) per operation | O(n) | Uses lazy transfer between two stacks |

## Algorithm Walkthrough

1. Initialize two stacks, commonly called `input_stack` and `output_stack`.

The `input_stack` stores newly pushed elements. The `output_stack` stores elements in dequeue order. Separating responsibilities between the two stacks allows us to efficiently simulate FIFO behavior.
2. For the `push(x)` operation, append the value directly to `input_stack`.

This operation is simple because stack insertion at the top is naturally efficient. We do not attempt to maintain queue order immediately.
3. Before performing `pop()` or `peek()`, check whether `output_stack` is empty.

If `output_stack` already contains elements, then its top element represents the front of the queue, so no additional work is needed.
4. If `output_stack` is empty, transfer all elements from `input_stack` to `output_stack`.

During this transfer, the order of elements reverses. The oldest inserted element ends up at the top of `output_stack`, which correctly matches queue behavior.
5. For `pop()`, remove and return the top element from `output_stack`.

Since the oldest element is now on top, removing from `output_stack` correctly simulates queue removal.
6. For `peek()`, return the top element from `output_stack` without removing it.

The front element remains accessible at the top after the transfer process.
7. For `empty()`, return whether both stacks are empty.

If neither stack contains elements, then the queue itself is empty.

### Why it works

The core invariant is that `output_stack`, when non-empty, always stores elements in correct dequeue order. New elements are temporarily stored in `input_stack` until a transfer occurs. Each transfer reverses insertion order exactly once, converting the stack ordering into queue ordering. Because every element moves from `input_stack` to `output_stack` at most once, the implementation preserves FIFO semantics while maintaining amortized constant time operations.

## Python Solution

```
class MyQueue:

    def __init__(self):
        self.input_stack = []
        self.output_stack = []

    def push(self, x: int) -> None:
        self.input_stack.append(x)

    def _transfer(self) -> None:
        while self.input_stack:
            self.output_stack.append(self.input_stack.pop())

    def pop(self) -> int:
        if not self.output_stack:
            self._transfer()

        return self.output_stack.pop()

    def peek(self) -> int:
        if not self.output_stack:
            self._transfer()

        return self.output_stack[-1]

    def empty(self) -> bool:
        return not self.input_stack and not self.output_stack

# Your MyQueue object will be instantiated and called as such:
# obj = MyQueue()
# obj.push(x)
# param_2 = obj.pop()
# param_3 = obj.peek()
# param_4 = obj.empty()
```

The implementation maintains two internal stacks represented by Python lists. The `push` operation appends directly to `input_stack`, which is always efficient.

The helper method `_transfer()` moves elements from `input_stack` to `output_stack`. This method is called only when `output_stack` is empty and a front-access operation is needed. By centralizing the transfer logic into a helper function, the implementation avoids duplicated code between `pop` and `peek`.

The `pop` method first ensures that `output_stack` contains the current front element. If necessary, it performs the transfer and then removes the top element.

The `peek` method follows the same preparation logic but only reads the top value without removing it.

The `empty` method checks whether both stacks are empty because elements may exist in either stack depending on the operation history.

## Go Solution

```
type MyQueue struct {
	inputStack  []int
	outputStack []int
}

func Constructor() MyQueue {
	return MyQueue{
		inputStack:  []int{},
		outputStack: []int{},
	}
}

func (this *MyQueue) Push(x int) {
	this.inputStack = append(this.inputStack, x)
}

func (this *MyQueue) transfer() {
	for len(this.inputStack) > 0 {
		n := len(this.inputStack)

		value := this.inputStack[n-1]
		this.inputStack = this.inputStack[:n-1]

		this.outputStack = append(this.outputStack, value)
	}
}

func (this *MyQueue) Pop() int {
	if len(this.outputStack) == 0 {
		this.transfer()
	}

	n := len(this.outputStack)

	value := this.outputStack[n-1]
	this.outputStack = this.outputStack[:n-1]

	return value
}

func (this *MyQueue) Peek() int {
	if len(this.outputStack) == 0 {
		this.transfer()
	}

	return this.outputStack[len(this.outputStack)-1]
}

func (this *MyQueue) Empty() bool {
	return len(this.inputStack) == 0 && len(this.outputStack) == 0
}

/**
 * Your MyQueue object will be instantiated and called as such:
 * obj := Constructor();
 * obj.Push(x);
 * param_2 := obj.Pop();
 * param_3 := obj.Peek();
 * param_4 := obj.Empty();
 */
```

The Go implementation uses slices as stacks. Appending to a slice simulates stack push behavior, while accessing and removing the last element simulates stack pop behavior.

Unlike Python, Go does not provide a built-in stack abstraction, so stack operations are implemented manually using slice indexing and slicing. There are no integer overflow concerns because the problem constraints are extremely small. Empty checks are performed using `len(slice) == 0`.

### Example 1

Input:

```
["MyQueue", "push", "push", "peek", "pop", "empty"]
[[], [1], [2], [], [], []]
```

| Operation | input_stack | output_stack | Return Value |
| --- | --- | --- | --- |
| Initialize | [] | [] | null |
| push(1) | [1] | [] | null |
| push(2) | [1, 2] | [] | null |
| peek() transfer occurs | [] | [2, 1] | 1 |
| pop() | [] | [2] | 1 |
| empty() | [] | [2] | false |

### Detailed Transfer Process

When `peek()` is called:

1. `output_stack` is empty
2. Move `2` from `input_stack` to `output_stack`
3. Move `1` from `input_stack` to `output_stack`

After transfer:

```
input_stack  = []
output_stack = [2, 1]
```

The top of `output_stack` is `1`, which is the oldest inserted element and therefore the correct front of the queue.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | Amortized O(1) per operation | Each element is pushed and transferred at most once |
| Space | O(n) | All elements are stored across the two stacks |

The important detail in the time analysis is amortization. A single `pop` or `peek` operation may trigger a transfer of many elements, resulting in `O(n)` work for that specific call. However, each element only moves from `input_stack` to `output_stack` once during its lifetime. Across `n` total operations, the total work remains linear, giving an amortized constant time complexity.

## Edge Cases

One important edge case is a queue containing only a single element. Bugs often occur when removing the last remaining value because internal data structures may not correctly transition back to an empty state. In this implementation, after the element is removed, both stacks become empty, and `empty()` correctly returns `True`.

Another important edge case occurs when many elements are pushed before the first `pop` or `peek`. A naive implementation might fail to preserve ordering during the transfer process. Here, the transfer reverses the stack order exactly once, ensuring that the oldest inserted element appears at the top of `output_stack`.

A third important edge case is repeatedly alternating between pushes and pops. This pattern tests whether the implementation accidentally loses elements or corrupts ordering when both stacks are partially populated. Because the implementation only transfers elements when `output_stack` is empty, existing dequeue order is preserved correctly across all operation sequences.
