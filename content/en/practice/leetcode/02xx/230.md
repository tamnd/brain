## Solution

## Problem Understanding

This problem asks us to find the `kth` smallest element in a **Binary Search Tree (BST)**. The key detail is that the tree is specifically a BST, which gives us an important structural property that can be exploited for efficiency.

A binary search tree is a binary tree where, for every node:

- All values in the left subtree are smaller than the node's value.
- All values in the right subtree are larger than the node's value.

The input consists of two parts:

- `root`, which is the root node of the BST.
- `k`, an integer representing which smallest element we want to retrieve, using **1-based indexing**.

The output should be the value of the node that would appear in the `kth` position if all node values were sorted in ascending order.

For example, consider the tree:

```
    3
   / \
  1   4
   \
    2
```

If `k = 1`, the sorted order of values is:

```
[1, 2, 3, 4]
```

The first smallest element is `1`, so the function should return `1`.

The constraints tell us several important things about the input size and guarantees:

- The number of nodes `n` can be up to `10^4`, which is large enough that inefficient repeated sorting or excessive copying should be avoided.
- `1 <= k <= n`, meaning `k` is always valid. We never need to handle invalid values like `k = 0` or `k > n`.
- Node values range from `0` to `10^4`, but the actual values themselves are not especially important for complexity because BST ordering is what matters.

There are several edge cases worth recognizing upfront. A tree with only one node is the smallest valid input and should immediately return that value. Highly skewed trees, which resemble linked lists, may affect recursion depth or traversal behavior. Another important case is when `k = 1` or `k = n`, meaning we want the smallest or largest element respectively. Since the problem guarantees valid input and a proper BST, we do not need to handle malformed trees.

## Approaches

### Brute Force Approach

A straightforward solution is to traverse the entire tree, collect all node values into an array, sort the array, and then return the `(k - 1)` indexed element.

This works because sorting all values produces the exact ascending order of the BST elements. After sorting, retrieving the `kth` smallest value becomes trivial.

However, this approach ignores the special BST property. Instead of leveraging the already ordered structure, it treats the tree like an arbitrary binary tree and performs unnecessary work.

The algorithm would be:

1. Traverse every node and store values in a list.
2. Sort the list.
3. Return `values[k - 1]`.

Since sorting takes `O(n log n)` time, this becomes inefficient for large trees.

### Optimal Approach, Inorder Traversal

The key insight comes from a fundamental BST property:

> Performing an **inorder traversal** of a BST visits nodes in ascending sorted order.

An inorder traversal visits:

```
Left subtree → Current node → Right subtree
```

Because all left subtree values are smaller and all right subtree values are larger, this traversal naturally generates sorted values without needing an explicit sort.

Instead of storing every value, we can count visited nodes during inorder traversal. Once we reach the `kth` visited node, we immediately return its value.

This is efficient because we stop early as soon as the answer is found.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) | O(n) | Traverse tree, collect values, sort array |
| Optimal | O(h + k), worst case O(n) | O(h) | Inorder traversal with early stopping |

Here, `h` represents the height of the tree.

## Algorithm Walkthrough

We will use **iterative inorder traversal with a stack**.

### Step 1: Initialize a stack

We use a stack to simulate recursion manually.

The stack helps us keep track of nodes whose left subtree is still being explored.

We also create a pointer called `current`, starting at `root`.

### Step 2: Move as far left as possible

Since inorder traversal visits the left subtree first, we repeatedly move left:

- Push the current node onto the stack.
- Move to `current.left`.

We continue until `current` becomes `None`.

This ensures we reach the smallest remaining node first.

### Step 3: Process the top node

When no more left nodes exist:

- Pop the top node from the stack.
- Decrease `k` by `1`.

This node represents the next smallest element in sorted order.

If `k == 0`, we have reached the `kth` smallest element, so we return the node's value immediately.

### Step 4: Explore the right subtree

After processing a node:

- Move to its right child.

This follows inorder traversal rules:

```
Left → Node → Right
```

We repeat the process until the answer is found.

### Step 5: Return the answer

The problem guarantees valid input, so we are guaranteed to find the `kth` smallest element before traversal completes.

### Why it works

The correctness comes from the BST inorder traversal property. In a BST, inorder traversal always visits nodes in strictly ascending order. Therefore, the first visited node is the smallest, the second visited node is the second smallest, and so on. By decrementing `k` after each visited node, the moment `k` reaches zero corresponds exactly to the `kth` smallest element.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import Optional

class Solution:
    def kthSmallest(self, root: Optional[TreeNode], k: int) -> int:
        stack = []
        current = root

        while stack or current:
            while current:
                stack.append(current)
                current = current.left

            current = stack.pop()
            k -= 1

            if k == 0:
                return current.val

            current = current.right

        return -1
```

The implementation follows the inorder traversal strategy described earlier.

We begin by creating an empty `stack` and assigning `root` to `current`. The stack simulates recursive traversal and allows us to revisit nodes after exploring their left subtree.

The outer `while` loop continues as long as there are unexplored nodes, either in the stack or through the `current` pointer.

Inside the inner `while` loop, we repeatedly move left, pushing each node onto the stack. This guarantees that the smallest remaining node will be processed first.

After reaching the leftmost node, we pop the top node from the stack. Since inorder traversal visits nodes in ascending order, this popped node is the next smallest element. We decrement `k`, and if `k` becomes zero, we immediately return the node's value.

If the target has not yet been found, we move to the node's right child and continue traversal.

The final `return -1` statement is technically unnecessary because the problem guarantees valid input, but it satisfies function completeness.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func kthSmallest(root *TreeNode, k int) int {
    stack := []*TreeNode{}
    current := root

    for len(stack) > 0 || current != nil {
        for current != nil {
            stack = append(stack, current)
            current = current.Left
        }

        current = stack[len(stack)-1]
        stack = stack[:len(stack)-1]

        k--
        if k == 0 {
            return current.Val
        }

        current = current.Right
    }

    return -1
}
```

The Go implementation follows the exact same logic as the Python version but adapts it to Go syntax and memory handling.

Instead of Python lists, Go uses a slice of `*TreeNode` pointers as the stack. Pushing is done with `append`, while popping requires manually accessing the last element and shrinking the slice.

Go distinguishes between `nil` and non-`nil` pointers, so traversal checks use `current != nil`.

Integer overflow is not a concern here because node values and `k` are very small relative to Go's integer range.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(h + k), worst case O(n) | We traverse only the nodes needed to reach the `kth` smallest |
| Space | O(h) | Stack stores at most tree height |

The traversal only visits nodes necessary to reach the answer. In a balanced BST, the height `h` is `O(log n)`, so the algorithm can be very efficient. In the worst case of a skewed tree, the height becomes `O(n)`, and traversal may visit every node, resulting in `O(n)` time and space.

## Edge Cases

### Single Node Tree

A tree containing only one node is the smallest possible valid input. This case can expose bugs where traversal logic assumes children exist. Since inorder traversal immediately visits the root, the implementation correctly returns the root value when `k = 1`.

### Highly Skewed Tree

A BST may be completely unbalanced, behaving like a linked list. For example:

```
1
 \
  2
   \
    3
```

This can expose recursion depth problems in recursive solutions. The iterative stack-based approach avoids recursion limits and handles skewed trees safely.

### `k = 1` or `k = n`

When `k = 1`, we want the smallest element, which is the leftmost node in the BST. When `k = n`, we want the largest element, which is the rightmost node. Off-by-one mistakes are common here because `k` is 1-indexed. The implementation avoids this issue by decrementing `k` after each visited node and returning when `k == 0`, which precisely matches the desired indexing.

### Follow Up Optimization

If the BST changes frequently and `kth` smallest queries happen often, repeatedly performing inorder traversal becomes inefficient.

A better approach is to augment each node with the **size of its subtree**. Each node stores how many nodes exist in its left and right descendants.

With subtree sizes:

- If `left_size + 1 == k`, the current node is the answer.
- If `k <= left_size`, search left.
- Otherwise, search right with an adjusted `k`.

This reduces query time to `O(log n)` in balanced BSTs, while insertions and deletions update subtree sizes dynamically.
