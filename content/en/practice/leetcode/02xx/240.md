## Solution

## Problem Understanding

The problem gives us an `m x n` matrix of integers where two ordering rules are guaranteed:

- Every row is sorted in ascending order from left to right.
- Every column is sorted in ascending order from top to bottom.

We are also given an integer `target`, and we need to determine whether the target exists anywhere inside the matrix.

In other words, this is not a completely arbitrary 2D grid. The matrix has structure that we can exploit to search more efficiently than checking every cell one by one.

The input consists of:

- A two-dimensional list `matrix`
- An integer `target`

The output is:

- `true` if the target exists in the matrix
- `false` otherwise

The constraints tell us several important things:

- Both dimensions can be as large as 300.
- The matrix may therefore contain up to `300 * 300 = 90,000` elements.
- Values may be negative and can be as large as `10^9` in magnitude.

A brute-force scan across all elements would still technically work within these limits, but the problem explicitly asks for an efficient algorithm. The sorted properties strongly suggest that we should do better than linear scanning of all elements.

There are several edge cases worth identifying early:

- A matrix with only one row
- A matrix with only one column
- A target smaller than the smallest value
- A target larger than the largest value
- A target that appears near the edges of the matrix
- A target that does not exist but lies numerically between existing values

The problem guarantees that the matrix is non-empty because both `m` and `n` are at least 1. That means we do not need to handle completely empty input matrices.

## Approaches

### Brute-Force Approach

The most straightforward solution is to iterate through every element in the matrix and compare it with the target.

For each row:

- Visit each column
- Check whether the current value equals the target
- Return `true` immediately if found

If the entire matrix is scanned without finding the target, return `false`.

This approach is correct because every element is examined exactly once, so no possible match can be missed.

However, this method completely ignores the sorted structure of the matrix. Even though rows and columns are ordered, brute force still checks every cell in the worst case.

With up to 90,000 elements, this solution is acceptable but not optimal.

### Key Insight for the Optimal Solution

The critical observation is that the matrix ordering allows us to eliminate large portions of the search space after examining a single value.

A particularly powerful strategy is to start from the top-right corner.

Suppose we are at position `(row, col)`:

- Everything below the current value in the same column is larger because columns are sorted.
- Everything to the left in the same row is smaller because rows are sorted.

This gives us directional information:

- If the current value is larger than the target, move left.
- If the current value is smaller than the target, move down.
- If the current value equals the target, return `true`.

Each move eliminates either:

- One entire row, or
- One entire column

That means we never revisit cells, producing an efficient linear-time traversal relative to matrix dimensions.

| --- | --- | --- | --- |

| Optimal | O(m + n) | O(1) | Uses matrix ordering to eliminate rows or columns |

## Algorithm Walkthrough

### Optimal Staircase Search Algorithm

1. Start at the top-right corner of the matrix.

This position is special because:

- Everything to the left is smaller
- Everything below is larger

That means one comparison gives enough information to decide which direction to move.
2. Compare the current value with the target.

There are three possible outcomes:

- The value equals the target
- The value is larger than the target
- The value is smaller than the target
3. If the current value equals the target, return `true`.

We found the target, so the search is complete.
4. If the current value is larger than the target, move left.

Since the current column is sorted from top to bottom, every value below the current cell is even larger. Therefore, the entire current column can be discarded.
5. If the current value is smaller than the target, move down.

Since the current row is sorted from left to right, every value to the left is even smaller. Therefore, the entire current row can be discarded.
6. Continue until either:

- The target is found, or
- The search moves outside matrix boundaries
7. If the loop ends without finding the target, return `false`.

### Why it works

The algorithm maintains an invariant that the target, if it exists, must still lie inside the remaining unexplored submatrix.

At every step:

- Moving left removes a column that cannot contain the target because all its remaining values are too large.
- Moving down removes a row that cannot contain the target because all its remaining values are too small.

Because each move safely eliminates impossible regions without excluding any valid candidates, the algorithm is guaranteed to find the target if it exists.

## Python Solution

```
from typing import List

class Solution:
    def searchMatrix(self, matrix: List[List[int]], target: int) -> bool:
        rows = len(matrix)
        cols = len(matrix[0])

        row = 0
        col = cols - 1

        while row < rows and col >= 0:
            current = matrix[row][col]

            if current == target:
                return True

            if current > target:
                col -= 1
            else:
                row += 1

        return False
```

The implementation begins by determining the matrix dimensions. Since the matrix is guaranteed to contain at least one row and one column, accessing `matrix[0]` is safe.

The algorithm initializes:

- `row = 0`, representing the first row
- `col = cols - 1`, representing the last column

This places the search at the top-right corner.

Inside the loop:

- `current` stores the current matrix value
- If `current == target`, the function immediately returns `True`
- If `current > target`, the algorithm moves left by decrementing `col`
- Otherwise, the algorithm moves down by incrementing `row`

The loop stops when either:

- The target is found
- The search moves outside the matrix boundaries

If the loop exits naturally, the target does not exist in the matrix, so the function returns `False`.

## Go Solution

```
func searchMatrix(matrix [][]int, target int) bool {
    rows := len(matrix)
    cols := len(matrix[0])

    row := 0
    col := cols - 1

    for row < rows && col >= 0 {
        current := matrix[row][col]

        if current == target {
            return true
        }

        if current > target {
            col--
        } else {
            row++
        }
    }

    return false
}
```

The Go implementation follows exactly the same logic as the Python version.

A few Go-specific notes are worth mentioning:

- Slices are used instead of Python lists.
- Matrix dimensions are obtained using `len`.
- No special handling for integer overflow is required because comparisons are safe within Go's integer range for the given constraints.
- The problem guarantees a non-empty matrix, so accessing `matrix[0]` is safe.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m + n) | At most one downward move per row and one leftward move per column |
| Space | O(1) | Only a few variables are used |

The runtime is linear in the matrix dimensions because each iteration removes either one row or one column from consideration.

The algorithm never revisits cells. Starting from the top-right corner, the pointer can move left at most `n` times and down at most `m` times.

The space complexity is constant because no auxiliary data structures are allocated.

## Edge Cases

### Single Row or Single Column Matrices

A matrix with only one row or one column can expose indexing mistakes or incorrect loop termination conditions.

For example:

```
[[1, 3, 5, 7]]
```

or

```
[[1],
 [3],
 [5]]
```

The implementation handles these correctly because the same movement rules still apply. The algorithm naturally degenerates into a simple linear scan across the only available direction.

### Target Smaller Than the Smallest Element

If the target is smaller than every value in the matrix, the algorithm repeatedly moves left from the top-right corner until the column index becomes negative.

Example:

```
matrix = [[5, 6], [7, 8]]
target = 1
```

This case is important because some implementations may accidentally continue searching after leaving matrix bounds. The loop condition `col >= 0` safely terminates the search.

### Target Larger Than the Largest Element

If the target exceeds every value in the matrix, the algorithm repeatedly moves downward until the row index exceeds the last valid row.

Example:

```
matrix = [[1, 2], [3, 4]]
target = 10
```

The condition `row < rows` ensures that the algorithm exits cleanly without accessing invalid memory.

### Values Near Integer Limits

The matrix values may be as small as `-10^9` and as large as `10^9`.

A flawed solution could attempt arithmetic operations that risk overflow in some languages. This implementation only performs comparisons, so it safely handles all valid input values without overflow concerns.
