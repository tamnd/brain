## Solution

## Problem Understanding

The problem gives two inputs: a string called `pattern` and another string called `s`. The goal is to determine whether the sequence of words in `s` follows the same structural pattern described by `pattern`.

Each character in `pattern` represents a mapping to a single word in `s`. The relationship must be a bijection, which means the mapping works in both directions without conflicts.

A bijection requires two conditions:

1. A pattern character must always map to the same word.
2. A word must always map back to the same pattern character.

This means:

- One character cannot map to multiple different words.
- Multiple characters cannot map to the same word.

For example, if the pattern is `"abba"` and the string is `"dog cat cat dog"`:

- `'a'` maps to `"dog"`
- `'b'` maps to `"cat"`

The structure matches correctly because every occurrence of `'a'` corresponds to `"dog"` and every occurrence of `'b'` corresponds to `"cat"`.

However, if the string were `"dog cat cat fish"`, the last word breaks the mapping because `'a'` previously mapped to `"dog"`, not `"fish"`.

The input constraints are relatively small:

- `pattern.length <= 300`
- `s.length <= 3000`

These limits indicate that an efficient linear scan solution is more than sufficient. We do not need advanced optimization techniques.

One important detail is that words in `s` are separated by exactly one space, with no leading or trailing spaces. This guarantees that splitting the string by spaces produces clean words without extra preprocessing.

Several edge cases are important:

- The number of words may not equal the number of pattern characters.
- Different pattern characters may incorrectly map to the same word.
- The same pattern character may appear multiple times and must always map consistently.
- A single-character pattern with one word should return `true`.
- Repeated characters with differing words should return `false`.

Understanding the bijection requirement is the key to solving the problem correctly.

## Approaches

### Brute Force Approach

A brute force approach would attempt to validate every mapping relationship repeatedly during comparison. For each character and word pair, we could scan all previously seen pairs to determine whether the mapping is valid.

For example:

- When processing a new `(character, word)` pair, search through earlier pairs to check consistency.
- Ensure the character never mapped to another word.
- Ensure the word never mapped to another character.

This approach is correct because it explicitly verifies all constraints of the bijection. However, repeated rescanning of previous pairs makes the algorithm inefficient.

If there are `n` pattern characters, then each step may require scanning up to `n` earlier elements, leading to quadratic time complexity.

Although the constraints are small enough that this might still pass, it is unnecessarily slow and inefficient.

### Optimal Approach

The key observation is that we need constant-time lookup in both directions:

- Character → word
- Word → character

A hash map is ideal for this requirement.

We maintain:

- One hash map from pattern character to word.
- Another hash map from word to pattern character.

As we iterate through the pattern and words simultaneously:

- If a mapping already exists, verify consistency.
- If no mapping exists, create it in both maps.

Using two hash maps guarantees bijection enforcement efficiently.

This solution processes each character-word pair exactly once, resulting in linear time complexity.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeatedly scans earlier mappings to validate consistency |
| Optimal | O(n) | O(n) | Uses two hash maps for constant-time bidirectional lookup |

## Algorithm Walkthrough

1. Split the input string `s` into individual words using spaces.

This converts the sentence into an array of words so we can compare each word against the corresponding pattern character.
2. Check whether the number of words matches the length of the pattern.

If they differ, a complete one-to-one correspondence is impossible, so return `false` immediately.
3. Create two hash maps.

The first map stores:

- pattern character → word

The second map stores:

- word → pattern character

Two maps are necessary because the bijection must work in both directions.
4. Iterate through the pattern and words simultaneously.

At each index:

- Let the current pattern character be `char`
- Let the current word be `word`
5. Check whether `char` already exists in the character-to-word map.

If it does:

- Verify the mapped word matches the current word.
- If not, return `false`.
6. Check whether `word` already exists in the word-to-character map.

If it does:

- Verify the mapped character matches the current character.
- If not, return `false`.
7. If neither mapping exists yet, create both mappings.

This establishes the bidirectional relationship.
8. Continue until all pairs are processed.
9. If no conflicts were found, return `true`.

### Why it works

The algorithm maintains an invariant throughout execution:

- Every pattern character maps to exactly one word.
- Every word maps to exactly one pattern character.

At every step, the two hash maps guarantee consistency in both directions. Any violation of the bijection property is detected immediately. Since every pair is validated exactly once, the final result is correct.

## Python Solution

```
class Solution:
    def wordPattern(self, pattern: str, s: str) -> bool:
        words = s.split()

        if len(pattern) != len(words):
            return False

        char_to_word: dict[str, str] = {}
        word_to_char: dict[str, str] = {}

        for char, word in zip(pattern, words):
            if char in char_to_word:
                if char_to_word[char] != word:
                    return False
            else:
                char_to_word[char] = word

            if word in word_to_char:
                if word_to_char[word] != char:
                    return False
            else:
                word_to_char[word] = char

        return True
```

The implementation begins by splitting the input string into words. This allows direct index-based comparison between the pattern characters and the corresponding words.

The first conditional checks whether the pattern length matches the number of words. This is a necessary condition for a valid bijection.

Two dictionaries are then created:

- `char_to_word`
- `word_to_char`

During iteration, the algorithm validates both directions of the mapping:

- A character must consistently map to the same word.
- A word must consistently map to the same character.

If any inconsistency appears, the function immediately returns `False`.

If the entire loop completes without conflicts, the mapping satisfies all conditions, so the function returns `True`.

## Go Solution

```
package main

import "strings"

func wordPattern(pattern string, s string) bool {
	words := strings.Split(s, " ")

	if len(pattern) != len(words) {
		return false
	}

	charToWord := make(map[byte]string)
	wordToChar := make(map[string]byte)

	for i := 0; i < len(pattern); i++ {
		char := pattern[i]
		word := words[i]

		if mappedWord, exists := charToWord[char]; exists {
			if mappedWord != word {
				return false
			}
		} else {
			charToWord[char] = word
		}

		if mappedChar, exists := wordToChar[word]; exists {
			if mappedChar != char {
				return false
			}
		} else {
			wordToChar[word] = char
		}
	}

	return true
}
```

The Go implementation follows the same logic as the Python solution but uses Go maps instead of Python dictionaries.

A few Go-specific details are worth noting:

- Pattern characters are handled as `byte` values because the input contains only lowercase English letters.
- `strings.Split()` is used to separate words by spaces.
- Go maps return an additional boolean indicating whether a key exists, which simplifies lookup and validation.

No special handling for integer overflow or nil values is needed because the problem constraints are small and straightforward.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each pattern character and word pair is processed exactly once |
| Space | O(n) | Two hash maps may store up to `n` unique mappings |

The algorithm performs a single pass through the pattern and word list. Hash map lookups and insertions operate in average constant time, so the total runtime is linear.

The space usage is also linear because, in the worst case, every character and every word is unique and must be stored in the maps.

## Edge Cases

One important edge case occurs when the number of pattern characters differs from the number of words. A naive implementation might attempt to process pairs anyway, leading to incorrect results or index errors. The implementation avoids this by checking lengths immediately and returning `false` if they differ.

Another critical edge case happens when two different pattern characters map to the same word. For example, `"ab"` and `"dog dog"` should return `false`. A solution using only one hash map from character to word would incorrectly accept this input. The second reverse mapping ensures the bijection property is enforced properly.

A third important case involves repeated characters. For example, `"aaaa"` with `"dog dog dog dog"` is valid, but `"aaaa"` with `"dog cat dog cat"` is invalid. The algorithm correctly handles this by verifying that every future occurrence of a character matches its original mapped word.

A subtle edge case appears when all mappings are unique. For example, `"abc"` and `"dog cat fish"` should return `true`. Some incorrect implementations overcomplicate the logic and mistakenly reject valid unique mappings. Since the algorithm only rejects actual conflicts, valid unique mappings are accepted naturally.
