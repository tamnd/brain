## Solution

## Problem Understanding

This problem asks us to summarize consecutive numbers in a sorted array into compact range representations.

We are given an integer array `nums` that is already **sorted in ascending order** and contains **unique values**. Because the array is sorted and duplicates are not allowed, consecutive numbers naturally appear next to each other. Our goal is to group adjacent consecutive values into ranges and return the smallest possible list of those ranges.

A range is defined as all integers between two endpoints `[a, b]`, inclusive.

If a sequence contains multiple consecutive integers, such as `[4,5,6]`, it should be represented as `"4->6"`. If a number appears alone with no neighboring consecutive values, such as `[7]`, it should simply be represented as `"7"`.

For example:

- `[0,1,2]` becomes `"0->2"`
- `[7]` becomes `"7"`

The phrase **"smallest sorted list of ranges"** means we should merge consecutive numbers whenever possible. We should not split a valid consecutive segment into smaller pieces.

The input constraints are relatively small:

- `0 <= nums.length <= 20`
- Numbers may range from `-2^31` to `2^31 - 1`
- Values are unique
- The array is already sorted

Although the input size is tiny, we should still aim for a clean and efficient solution. The sorted and unique guarantees are especially important because they eliminate the need for sorting or duplicate handling. We can safely process the array from left to right.

There are several important edge cases to consider upfront.

An empty array should return an empty list because there are no ranges to summarize. A single-element array should return that one value as a string. Arrays where all numbers are consecutive should produce one large range. Arrays where no numbers are consecutive should produce individual values only. We should also carefully handle extreme integer values near `-2^31` and `2^31 - 1`, since naive arithmetic can sometimes introduce overflow concerns in some languages.

## Approaches

### Brute Force Approach

A brute-force approach would repeatedly try to expand every number into the longest possible consecutive sequence by scanning forward.

For each position `i`, we could start a new range and check subsequent elements one by one to determine how long the consecutive chain continues. After finding the end of the chain, we format the result and move to the next uncovered number.

This works because every consecutive sequence is explicitly discovered and converted into the required string format. However, if implemented naively, we may repeatedly scan portions of the array unnecessarily. In the worst case, this could become quadratic if each index repeatedly checks many future elements.

Even though `n <= 20` makes performance irrelevant here, this is not the cleanest or most scalable solution.

### Optimal Approach

The key observation is that because the array is **already sorted and unique**, consecutive numbers always appear next to one another.

This means we only need **one pass** through the array.

We can track the start of the current range and continue extending it as long as consecutive numbers appear. The moment we encounter a gap, meaning `nums[i] != nums[i - 1] + 1`, we know the previous range has ended.

At that point:

- If the range contains one number, we add `"a"`
- Otherwise, we add `"a->b"`

Then we start a new range.

This greedy, single-pass strategy works because sorted unique input guarantees that once a gap appears, the previous range can never be extended later.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) excluding output | Repeatedly scans ahead to find consecutive ranges |
| Optimal | O(n) | O(1) excluding output | Single pass, builds ranges as gaps appear |

## Algorithm Walkthrough

1. Initialize an empty result list to store formatted ranges.
2. Handle the empty array case immediately. If `nums` is empty, return an empty list because there are no ranges to summarize.
3. Start tracking the current range by setting a variable `range_start` to the first element in the array. This marks where the active range begins.
4. Iterate through the array starting from index `1`.
5. For each number, check whether it is consecutive with the previous number by testing:

`nums[i] == nums[i - 1] + 1`

If this condition is true, the current range continues, so we simply move forward.
6. If the condition is false, we found a gap. This means the current range has ended at `nums[i - 1]`.
7. Format the finished range:

- If `range_start == range_end`, add a single number string such as `"7"`
- Otherwise, add a range string such as `"4->5"`
8. Start a new range by updating `range_start = nums[i]`.
9. After the loop finishes, process the final range because it will not have been added yet.
10. Return the completed result list.

### Why it works

The correctness of this algorithm depends on one important invariant: at every step, `range_start` marks the beginning of the current uninterrupted consecutive sequence.

Because the array is sorted and unique, consecutive numbers must appear next to each other. Once a gap is encountered, the current range is guaranteed to be complete and cannot be extended later. Therefore, finalizing the range immediately is always correct. Since every element belongs to exactly one consecutive segment, the algorithm covers all numbers exactly once without overlap.

## Python Solution

```
from typing import List

class Solution:
    def summaryRanges(self, nums: List[int]) -> List[str]:
        if not nums:
            return []

        result = []
        range_start = nums[0]

        for i in range(1, len(nums)):
            # Check if the sequence breaks
            if nums[i] != nums[i - 1] + 1:
                range_end = nums[i - 1]

                if range_start == range_end:
                    result.append(str(range_start))
                else:
                    result.append(f"{range_start}->{range_end}")

                range_start = nums[i]

        # Process the final range
        range_end = nums[-1]

        if range_start == range_end:
            result.append(str(range_start))
        else:
            result.append(f"{range_start}->{range_end}")

        return result
```

This implementation begins by handling the empty input case, which avoids index errors when accessing `nums[0]`.

The variable `range_start` keeps track of where the current consecutive sequence begins. We iterate through the array starting from the second element because each number must be compared with its predecessor.

Whenever a gap appears, meaning the current number is not consecutive with the previous one, we finalize the current range. The formatting logic checks whether the range contains a single element or multiple values and appends the correct string representation.

Finally, after the loop ends, the last range must still be added manually because there is no later gap to trigger its insertion.

## Go Solution

```
package main

import "strconv"

func summaryRanges(nums []int) []string {
	if len(nums) == 0 {
		return []string{}
	}

	result := []string{}
	rangeStart := nums[0]

	for i := 1; i < len(nums); i++ {
		// Check if the sequence breaks
		if nums[i] != nums[i-1]+1 {
			rangeEnd := nums[i-1]

			if rangeStart == rangeEnd {
				result = append(result, strconv.Itoa(rangeStart))
			} else {
				result = append(
					result,
					strconv.Itoa(rangeStart)+"->"+strconv.Itoa(rangeEnd),
				)
			}

			rangeStart = nums[i]
		}
	}

	// Process the final range
	rangeEnd := nums[len(nums)-1]

	if rangeStart == rangeEnd {
		result = append(result, strconv.Itoa(rangeStart))
	} else {
		result = append(
			result,
			strconv.Itoa(rangeStart)+"->"+strconv.Itoa(rangeEnd),
		)
	}

	return result
}
```

The Go implementation closely mirrors the Python solution. Since Go does not support direct integer-to-string conversion through formatting syntax like Python f-strings, we use `strconv.Itoa()`.

For empty input, we return an empty slice. Go slices naturally handle dynamic growth using `append()`. Integer overflow is not a concern here because the problem constraints fit safely within Go's `int` type on LeetCode platforms.

The algorithm performs a single left-to-right pass through the input array. Each number is processed exactly once, resulting in linear time complexity.

The auxiliary space usage is constant because we only maintain a few variables such as `range_start` and `range_end`. The result list itself is excluded from complexity analysis because it is required output.

## Edge Cases

### Empty Array

An empty array is one of the easiest cases to overlook. A naive implementation might immediately access `nums[0]`, causing an index error. Our implementation explicitly checks `if not nums` in Python and `len(nums) == 0` in Go before processing, safely returning an empty list.

### Single Element Array

If the input contains only one number, there is no loop iteration to trigger range creation. Without careful handling, the result might incorrectly remain empty. Our implementation solves this by always processing the final range after the loop, ensuring single-element arrays return the correct output.

### Final Range Not Added

A common bug occurs when developers only add ranges after detecting a gap. Since the final range has no future number to force a gap, it may accidentally be skipped. We avoid this issue by explicitly processing the last range after iteration ends.

### Extreme Integer Values

The input may contain values near `-2^31` or `2^31 - 1`. Some implementations can run into overflow issues when performing arithmetic like `nums[i - 1] + 1`. Python handles large integers naturally, while Go's `int` type on LeetCode platforms safely supports the required range. The implementation therefore works correctly even at boundary values.
