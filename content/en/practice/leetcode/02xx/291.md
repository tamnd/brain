## Solution

## Problem Understanding

This problem asks us to determine whether a pattern string can be mapped to another string using a bijective mapping between pattern characters and non-empty substrings.

The pattern consists of lowercase English letters, where each character represents a symbolic placeholder. The string `s` is the actual text we want to match against the pattern. A valid match means:

- Every unique character in the pattern maps to exactly one non-empty substring.
- The same pattern character must always map to the same substring.
- Different pattern characters cannot map to the same substring.
- When all pattern characters are replaced with their mapped substrings, the resulting concatenated string must equal `s`.

For example, if:

```
pattern = "abab"
s = "redblueredblue"
```

then one valid mapping is:

```
'a' -> "red"
'b' -> "blue"
```

Replacing the pattern characters gives:

```
"red" + "blue" + "red" + "blue"
```

which exactly matches `s`.

The constraints are small:

- `1 <= pattern.length <= 20`
- `1 <= s.length <= 20`

These limits strongly suggest that exponential search or backtracking is acceptable. Since we may need to try many possible substring assignments, a brute-force search with pruning is the intended direction.

The most important aspect of the problem is the bijection requirement. Many incorrect solutions only ensure consistency in one direction. For example:

```
pattern = "ab"
s = "aa"
```

A naive implementation might map:

```
'a' -> "a"
'b' -> "a"
```

But this is invalid because two pattern characters map to the same substring.

Several edge cases are especially important:

- Repeated pattern characters must reuse the exact same substring.
- Different characters cannot share a substring.
- Substrings cannot be empty.
- The entire string must be consumed exactly.
- Backtracking must correctly undo failed mappings.

Because of these constraints, recursive backtracking with hash maps is the most natural and effective solution.

## Approaches

### Brute Force Approach

The brute-force idea is to try every possible way to split the string `s` into substrings and then check whether those substrings can form a valid mapping for the pattern.

For each character in the pattern:

- If the character already has a mapping, verify that the next portion of `s` matches it.
- Otherwise, try every possible non-empty substring as a candidate mapping.

This guarantees correctness because every possible mapping is explored.

However, the number of possible substring partitions grows exponentially. At every step we may branch into many choices, and many paths eventually fail only after deep recursion. Without pruning or careful bookkeeping, the search becomes inefficient.

Even though the constraints are small enough for exponential solutions, we still need pruning to avoid unnecessary exploration.

### Optimal Approach, Backtracking with Bidirectional Hash Maps

The key insight is that the problem naturally forms a recursive decision tree.

At any point, we are trying to match:

- a position in `pattern`
- a position in `s`

If the current pattern character already has a mapping, then there is only one valid continuation:

- the next substring in `s` must exactly match the stored mapping

If the character does not yet have a mapping, then we try every possible substring beginning at the current position.

To enforce bijection efficiently, we maintain:

- a map from pattern character to substring
- a set of already used substrings

This prevents two characters from sharing the same substring.

Backtracking works well because:

- constraints are small
- invalid paths can often be rejected early
- mappings can be incrementally built and undone

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential, approximately O(n^m) | O(m) | Tries all substring partitions with minimal pruning |
| Optimal | Exponential in worst case | O(m + n) | Backtracking with pruning and bijection enforcement |

Here:

- `m = len(pattern)`
- `n = len(s)`

## Algorithm Walkthrough

1. Start recursive backtracking from the beginning of both the pattern and the string.

We maintain:

- `pattern_index`, current position in the pattern
- `string_index`, current position in `s`
- a hash map from pattern character to substring
- a set of used substrings
2. Define the base case.

If both indices reach the end simultaneously, we found a valid mapping and return `True`.

If only one reaches the end while the other does not, the mapping is incomplete and we return `False`.
3. Read the current pattern character.

Suppose the current character is `c`.
4. Check whether `c` already has a mapping.

If it does:

- retrieve the mapped substring
- verify that `s` begins with this substring at `string_index`

If it does not match, return `False`.

Otherwise, continue recursion using the next positions.
5. If `c` does not yet have a mapping, try all possible substrings.

Iterate through every possible ending index:

```
s[string_index:end]
```

Each candidate substring must be non-empty.
6. Enforce bijection.

Before assigning a substring:

- check whether it already exists in the used-substrings set

If another character already uses it, skip this candidate.
7. Create a temporary mapping.

Add:

```
c -> candidate
```

and mark the substring as used.
8. Recursively attempt to match the remaining pattern and string.

If recursion succeeds, immediately return `True`.
9. Backtrack if recursion fails.

Remove the mapping and unmark the substring so other possibilities can be explored.
10. If every candidate fails, return `False`.

### Why it works

The algorithm systematically explores every possible bijective mapping between pattern characters and substrings of `s`.

At every recursive step, the algorithm maintains the invariant that:

- all previously assigned mappings are consistent
- no two characters share the same substring
- the consumed portion of `s` exactly matches the processed portion of the pattern

Because every valid mapping is eventually explored, and invalid mappings are rejected immediately when inconsistencies appear, the algorithm is guaranteed to return the correct result.

## Python Solution

```
class Solution:
    def wordPatternMatch(self, pattern: str, s: str) -> bool:
        char_to_word: dict[str, str] = {}
        used_words: set[str] = set()

        def backtrack(pattern_index: int, string_index: int) -> bool:
            if pattern_index == len(pattern) and string_index == len(s):
                return True

            if pattern_index == len(pattern) or string_index == len(s):
                return False

            current_char = pattern[pattern_index]

            if current_char in char_to_word:
                mapped_word = char_to_word[current_char]

                if not s.startswith(mapped_word, string_index):
                    return False

                return backtrack(
                    pattern_index + 1,
                    string_index + len(mapped_word)
                )

            for end_index in range(string_index + 1, len(s) + 1):
                candidate = s[string_index:end_index]

                if candidate in used_words:
                    continue

                char_to_word[current_char] = candidate
                used_words.add(candidate)

                if backtrack(pattern_index + 1, end_index):
                    return True

                del char_to_word[current_char]
                used_words.remove(candidate)

            return False

        return backtrack(0, 0)
```

The implementation begins by creating two data structures:

- `char_to_word`, which stores mappings from pattern characters to substrings
- `used_words`, which prevents multiple characters from sharing the same substring

The recursive `backtrack` function processes both strings simultaneously using indices instead of slicing entire prefixes repeatedly. This keeps the implementation cleaner and more efficient.

The first base case checks whether both the pattern and string are fully consumed. If so, we found a complete valid mapping.

The second base case handles mismatch situations where one sequence finishes before the other.

When a pattern character already has a mapping, the code verifies consistency using `startswith`. This ensures repeated pattern characters always map to the same substring.

If the character is new, the algorithm tries every possible substring beginning at the current position. Each candidate is skipped if another character already uses it.

When a candidate is selected:

- it is added to the mapping
- marked as used
- recursion continues

If recursion later fails, the algorithm removes the mapping and continues trying other possibilities. This is the essence of backtracking.

## Go Solution

```
func wordPatternMatch(pattern string, s string) bool {
	charToWord := make(map[byte]string)
	usedWords := make(map[string]bool)

	var backtrack func(int, int) bool

	backtrack = func(patternIndex int, stringIndex int) bool {
		if patternIndex == len(pattern) && stringIndex == len(s) {
			return true
		}

		if patternIndex == len(pattern) || stringIndex == len(s) {
			return false
		}

		currentChar := pattern[patternIndex]

		if mappedWord, exists := charToWord[currentChar]; exists {
			if stringIndex+len(mappedWord) > len(s) {
				return false
			}

			if s[stringIndex:stringIndex+len(mappedWord)] != mappedWord {
				return false
			}

			return backtrack(
				patternIndex+1,
				stringIndex+len(mappedWord),
			)
		}

		for endIndex := stringIndex + 1; endIndex <= len(s); endIndex++ {
			candidate := s[stringIndex:endIndex]

			if usedWords[candidate] {
				continue
			}

			charToWord[currentChar] = candidate
			usedWords[candidate] = true

			if backtrack(patternIndex+1, endIndex) {
				return true
			}

			delete(charToWord, currentChar)
			delete(usedWords, candidate)
		}

		return false
	}

	return backtrack(0, 0)
}
```

The Go solution closely mirrors the Python implementation, but there are several language-specific differences.

Go uses `map[byte]string` for character mappings because pattern characters are lowercase English letters and can safely be represented as bytes.

The `usedWords` structure is implemented as `map[string]bool` since Go does not have a built-in set type.

Unlike Python's `startswith`, Go manually checks substring equality by slicing:

```
s[stringIndex:stringIndex+len(mappedWord)]
```

Go also requires explicit deletion during backtracking using:

```
delete(map, key)
```

The recursive closure is declared using:

```
var backtrack func(int, int) bool
```

before assigning the function body, which allows recursive self-reference.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | Exponential, approximately O(n^m) | Backtracking explores many substring assignments |
| Space | O(m + n) | Maps, recursion stack, and stored substrings |

Here:

- `m = len(pattern)`
- `n = len(s)`

The time complexity is exponential because each unmapped pattern character may try many candidate substrings. In the worst case, the recursion tree branches heavily.

However, the constraints are very small, with maximum length only 20, so this approach is practical.

The space complexity comes from:

- the recursion stack
- the mapping dictionary
- the used-substring set

At most, each character and substring participates once in the current recursion path.

## Edge Cases

### Multiple Pattern Characters Mapping to the Same Substring

One of the most common bugs is forgetting the bijection requirement.

For example:

```
pattern = "ab"
s = "aa"
```

A naive implementation might allow:

```
a -> "a"
b -> "a"
```

This is invalid because two different pattern characters cannot share the same substring.

The implementation prevents this by maintaining a `used_words` set. Before assigning a new substring to a character, the algorithm checks whether that substring is already used.

### Repeated Pattern Characters Must Match Exactly

Another important case occurs when a pattern character appears multiple times.

Example:

```
pattern = "aa"
s = "abc"
```

If:

```
a -> "a"
```

then the second `a` must also match `"a"`.

But the remaining string begins with `"b"` instead, so the mapping fails.

The implementation handles this by checking existing mappings using exact substring matching before recursing further.

### Entire String Must Be Consumed

A mapping is only valid if both the pattern and string are fully consumed simultaneously.

Example:

```
pattern = "ab"
s = "redblueextra"
```

Even if:

```
a -> "red"
b -> "blue"
```

matches the beginning, the leftover `"extra"` means the solution is invalid.

The base case ensures correctness by requiring:

```
pattern_index == len(pattern)
and
string_index == len(s)
```

before returning `True`.

### Backtracking Cleanup

Backtracking algorithms frequently fail because temporary state is not properly undone.

Suppose we try:

```
a -> "red"
```

and later discover this leads to failure.

If the mapping is not removed before exploring another branch, subsequent attempts operate on corrupted state.

The implementation carefully removes both:

- the character mapping
- the used substring marker

after every failed recursive attempt. This guarantees each recursive branch starts with a clean and correct state.
