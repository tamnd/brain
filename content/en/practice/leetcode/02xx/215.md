## Solution

## Problem Understanding

The problem asks us to find the `kth` largest element in an unsorted integer array. The important detail is that we are looking for the element that would appear in the `kth` position if the array were sorted in descending order. We are not looking for the `kth` distinct value.

For example, if the array is `[5, 5, 4, 3]` and `k = 2`, the answer is `5`, because the sorted order is `[5, 5, 4, 3]` and the second largest element is still `5`.

The input consists of:

- An integer array `nums`
- An integer `k`

The output is a single integer representing the `kth` largest element.

The constraints are important:

- The array can contain up to `10^5` elements
- Values range from `-10^4` to `10^4`

These constraints immediately suggest that inefficient algorithms may struggle. A quadratic solution such as repeated scanning or bubble sorting would be too slow in the worst case.

The problem also explicitly asks whether we can solve it without sorting. A full sort takes `O(n log n)` time, which is acceptable for many cases, but the question hints that a more efficient selection algorithm exists.

Several edge cases are worth considering:

- Arrays containing duplicate values
- Arrays containing negative numbers
- `k = 1`, meaning we want the maximum element
- `k = len(nums)`, meaning we want the minimum element
- Arrays where all elements are identical
- Very large arrays where performance matters significantly

The problem guarantees that `k` is always valid, so we never need to handle invalid indices or empty arrays.

## Approaches

### Brute Force Approach

The most straightforward solution is to sort the array in descending order and return the element at index `k - 1`.

This works because sorting places every element in its exact relative position. Once the array is sorted from largest to smallest, the `kth` largest element is simply the element at the corresponding position.

For example:

- Original array: `[3,2,1,5,6,4]`
- Sorted descending: `[6,5,4,3,2,1]`
- `k = 2`
- Result: `5`

This approach is easy to implement and guaranteed to be correct. However, sorting the entire array is more work than necessary. We only need one element's final position, not the complete ordering of every element.

Sorting requires `O(n log n)` time, which is acceptable but not optimal.

### Optimal Approach, Quickselect

The key insight is that we do not need the entire array sorted. We only need to know which element would occupy a specific position in sorted order.

This is exactly what the Quickselect algorithm is designed for.

Quickselect is related to Quicksort. Instead of recursively sorting both partitions, Quickselect only recurses into the partition that contains the target index.

The algorithm works by:

1. Choosing a pivot
2. Partitioning the array so that:

- Elements smaller than the pivot go to one side
- Elements larger than the pivot go to the other side
3. Determining whether the pivot ended up at the target index
4. Continuing only in the relevant partition

This reduces unnecessary work dramatically.

On average, Quickselect runs in linear time, `O(n)`.

## Algorithm Walkthrough

We will use Quickselect to find the `kth` largest element.

Instead of searching for the `kth` largest directly, it is easier to convert the problem into finding the element at index `len(nums) - k` in ascending order.

For example:

- `nums = [3,2,1,5,6,4]`
- `k = 2`
- Sorted ascending: `[1,2,3,4,5,6]`
- Target index: `6 - 2 = 4`
- Element at index `4` is `5`

### Step-by-Step Process

1. Convert the problem into finding the target index in ascending order.

We compute:

```
target = len(nums) - k
```

This allows us to work naturally with ascending partition logic.
2. Choose a pivot element.

We typically use the last element in the current range.

The pivot helps partition the array into smaller and larger elements.
3. Partition the array.

Rearrange elements so that:

- All elements smaller than the pivot appear before it
- All elements greater than or equal to the pivot appear after it

After partitioning, the pivot lands in its final sorted position.
4. Compare the pivot index with the target index.

- If they match, we found the answer
- If the pivot index is too small, search the right partition
- If the pivot index is too large, search the left partition
5. Repeat until the target position is found.

Each iteration reduces the search space.

### Why it works

The partition operation guarantees that the pivot ends up exactly where it would appear in a fully sorted array.

If the pivot lands at index `p`:

- Every element before `p` is smaller
- Every element after `p` is larger or equal

Therefore:

- If `p == target`, the pivot is exactly the element we want
- Otherwise, the target must lie entirely within one partition

This invariant ensures correctness.

## Python Solution

```
from typing import List

class Solution:
    def findKthLargest(self, nums: List[int], k: int) -> int:
        target = len(nums) - k
        
        left = 0
        right = len(nums) - 1
        
        while True:
            pivot_index = self.partition(nums, left, right)
            
            if pivot_index == target:
                return nums[pivot_index]
            elif pivot_index < target:
                left = pivot_index + 1
            else:
                right = pivot_index - 1

    def partition(self, nums: List[int], left: int, right: int) -> int:
        pivot = nums[right]
        
        store_index = left
        
        for i in range(left, right):
            if nums[i] < pivot:
                nums[store_index], nums[i] = nums[i], nums[store_index]
                store_index += 1
        
        nums[store_index], nums[right] = nums[right], nums[store_index]
        
        return store_index
```

The implementation begins by converting the problem into finding the element at index `len(nums) - k` in ascending order.

The algorithm then repeatedly partitions the array.

The `partition` function performs the core Quickselect logic. It chooses the last element as the pivot and moves all smaller elements to the left side. The `store_index` variable tracks where the next smaller element should be placed.

After partitioning completes, the pivot is swapped into its final sorted position.

Back in the main loop:

- If the pivot lands exactly at the target index, we return it
- If the pivot index is too small, we discard the left side
- If the pivot index is too large, we discard the right side

Because each iteration eliminates part of the array, the average runtime becomes linear.

## Go Solution

```
func findKthLargest(nums []int, k int) int {
    target := len(nums) - k
    
    left := 0
    right := len(nums) - 1
    
    for {
        pivotIndex := partition(nums, left, right)
        
        if pivotIndex == target {
            return nums[pivotIndex]
        } else if pivotIndex < target {
            left = pivotIndex + 1
        } else {
            right = pivotIndex - 1
        }
    }
}

func partition(nums []int, left int, right int) int {
    pivot := nums[right]
    
    storeIndex := left
    
    for i := left; i < right; i++ {
        if nums[i] < pivot {
            nums[storeIndex], nums[i] = nums[i], nums[storeIndex]
            storeIndex++
        }
    }
    
    nums[storeIndex], nums[right] = nums[right], nums[storeIndex]
    
    return storeIndex
}
```

The Go implementation follows the same logic as the Python version.

One notable difference is that Go does not support tuple unpacking in exactly the same way as Python, but simultaneous assignment still makes swapping concise.

Slices in Go behave similarly to dynamic array views, so partitioning modifies the underlying array in place without extra memory allocation.

There are no integer overflow concerns here because the constraints are small relative to Go's integer capacity.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) average, O(n²) worst | Each partition usually cuts the search space significantly |
| Space | O(1) | Partitioning is performed in place |

The average-case performance is linear because each partition discards roughly half the remaining elements.

The worst case occurs when poor pivot choices repeatedly create extremely unbalanced partitions, such as always selecting the smallest or largest element. In that scenario, the runtime degrades to `O(n²)`.

In practice, Quickselect performs very efficiently, especially with randomized pivots.

## Edge Cases

One important edge case occurs when the array contains many duplicate values. A naive implementation might incorrectly treat duplicates as distinct rankings. For example, in `[5,5,4]`, the second largest element is still `5`. The Quickselect implementation handles this correctly because it works with actual array positions after partitioning rather than distinct values.

Another important case is when `k = 1` or `k = len(nums)`. These represent the largest and smallest elements respectively. Off-by-one mistakes are common here because the algorithm converts the problem into an ascending-order index search. Using `target = len(nums) - k` correctly maps both extremes.

Arrays containing negative values can also expose bugs in partition logic if comparisons are implemented incorrectly. Since the algorithm relies purely on numeric comparisons and does not assume positive values, negative integers are handled naturally.

A final important edge case involves arrays where all values are identical. In such cases, partitioning repeatedly places pivots among equal elements. The implementation still works correctly because every identical value is a valid answer for the requested rank.
