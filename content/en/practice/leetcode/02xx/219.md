## Solution

## Problem Understanding

The problem asks us to determine whether an array contains any duplicate values that appear close enough together. More specifically, we need to check if there exist **two distinct indices** `i` and `j` such that:

- `nums[i] == nums[j]`, meaning the values are the same
- `abs(i - j) <= k`, meaning the positions are at most `k` indices apart

In simpler terms, we want to know if the same number appears again within a distance of `k` elements.

The input consists of:

- `nums`, an integer array
- `k`, a non-negative integer representing the maximum allowed distance between duplicate elements

The expected output is a boolean:

- Return `true` if at least one valid duplicate pair exists
- Return `false` otherwise

For example, if `nums = [1,2,3,1]` and `k = 3`, the two `1`s appear at indices `0` and `3`. Since `|3 - 0| = 3`, which satisfies the condition, the answer is `true`.

The constraints are important:

- `nums.length` can be as large as `10^5`
- Values in `nums` range from `-10^9` to `10^9`
- `k` can also be as large as `10^5`

These constraints immediately tell us that a quadratic time solution, `O(n²)`, will likely be too slow for large inputs. Since `n = 100,000`, comparing every pair would result in billions of operations in the worst case.

Several edge cases are worth identifying early:

- When `k = 0`, duplicates cannot count because indices must be distinct and distance must be `<= 0`. Since `i != j`, the answer is always `false`.
- Arrays with no duplicates should return `false`.
- Arrays where duplicates exist but are farther apart than `k` should also return `false`.
- Multiple repeated values may exist, and we only need to find one valid pair to return `true`.

## Approaches

### Brute Force Approach

A straightforward way to solve the problem is to compare every element with the next `k` elements and check for duplicates.

For each index `i`, we can examine indices from `i + 1` up to `i + k` (bounded by array length). If we find any `nums[i] == nums[j]`, we return `true`. If all comparisons fail, we return `false`.

This works because it explicitly checks every possible pair that satisfies the distance requirement.

However, this approach becomes inefficient when the input size grows. In the worst case, for every element we may compare up to `k` other elements. Since `k` can be as large as `n`, the time complexity becomes `O(n²)`.

Given `n = 10^5`, this is too slow.

### Optimal Approach, Hash Map for Last Seen Index

The key observation is that we do not need to compare every pair.

If we know the **last index where each number appeared**, then when we encounter the same number again, we can immediately compute the distance between occurrences.

For example:

```
nums = [1,2,3,1]
```

When we see the second `1`, we already know the previous `1` was at index `0`. We calculate:

```
current_index - previous_index
```

If the difference is at most `k`, we return `true`.

A hash map is ideal here because it allows us to quickly store and retrieve the last seen index of each number in constant time.

This transforms the problem from repeated comparisons into a single linear scan.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × k), worst case O(n²) | O(1) | Compare each element with the next `k` elements |
| Optimal | O(n) | O(n) | Store the most recent index of each value in a hash map |

## Algorithm Walkthrough

### Optimal Algorithm

1. Create an empty hash map called `last_seen`.

This map will store each number as the key and its most recent index as the value. We use a hash map because lookups and updates happen in constant time on average.
2. Iterate through the array using both the index and value.

At each step, we process one element and determine whether it creates a valid duplicate pair.
3. Check if the current number already exists in `last_seen`.

If it does, retrieve the previous index where it appeared.
4. Compute the distance between indices.

Calculate:

```
current_index - previous_index
```

Since we iterate left to right, the current index is always larger, so `abs()` is unnecessary.
5. If the distance is less than or equal to `k`, return `true`.

We found two equal values close enough together, which satisfies the problem requirements.
6. Update the hash map.

Store the current index as the latest occurrence of the number.

This is important because the most recent occurrence gives the best chance of satisfying the distance requirement for future duplicates.
7. Finish the iteration.

If no valid duplicate pair is found after scanning the whole array, return `false`.

### Why it works

The algorithm maintains an invariant: for every number encountered, `last_seen[value]` always stores the most recent index where that value appeared.

When we encounter a duplicate, comparing against the most recent occurrence is sufficient. If the closest previous duplicate is farther than `k`, any earlier occurrence would be even farther away. Therefore, we never miss a valid pair by storing only the latest index.

Since every element is processed once and every lookup/update takes constant time on average, the algorithm is both correct and efficient.

## Python Solution

```
from typing import List

class Solution:
    def containsNearbyDuplicate(self, nums: List[int], k: int) -> bool:
        last_seen = {}

        for index, value in enumerate(nums):
            if value in last_seen:
                if index - last_seen[value] <= k:
                    return True

            last_seen[value] = index

        return False
```

The implementation begins by creating an empty dictionary called `last_seen`. This dictionary maps each number to the most recent index where it appeared.

We then iterate through `nums` using `enumerate`, which provides both the index and value simultaneously.

For each number, we first check whether it already exists in the dictionary. If it does, we calculate the distance between the current index and the stored index. If this distance is at most `k`, we immediately return `True`.

After the check, we update the dictionary with the current index. This ensures we always keep the latest occurrence of the number, which is the most useful one for future comparisons.

If the loop finishes without finding a valid duplicate pair, we return `False`.

## Go Solution

```
func containsNearbyDuplicate(nums []int, k int) bool {
	lastSeen := make(map[int]int)

	for index, value := range nums {
		if previousIndex, exists := lastSeen[value]; exists {
			if index-previousIndex <= k {
				return true
			}
		}

		lastSeen[value] = index
	}

	return false
}
```

The Go implementation follows the same logic as the Python version. A `map[int]int` stores the most recent index for each number.

Go maps return two values during lookup: the stored value and a boolean indicating whether the key exists. This allows us to safely distinguish between a missing key and a stored index of `0`.

There are no integer overflow concerns here because indices remain within problem constraints. Since slices in Go already include length information, no special handling for empty arrays is needed, although the problem guarantees at least one element.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each element is processed once, hash map operations are O(1) on average |
| Space | O(n) | In the worst case, all numbers are unique and stored in the hash map |

The algorithm performs a single pass through the array. Each lookup and insertion into the hash map takes constant time on average, so the total runtime scales linearly with the number of elements.

The space complexity is `O(n)` because, in the worst case, every number is distinct and must be stored in the dictionary or map.

## Edge Cases

### Case 1: `k = 0`

This is one of the easiest cases to mishandle. Since the problem requires **distinct indices**, and `abs(i - j) <= 0` implies `i == j`, no valid pair can ever exist.

For example:

```
nums = [1,1], k = 0
```

Even though duplicates exist, the answer must be `false`. The implementation handles this naturally because:

```
index - previousIndex
```

will always be at least `1` for distinct indices.

### Case 2: Duplicate Exists but Too Far Away

A naive implementation might incorrectly return `true` as soon as it sees any duplicate.

For example:

```
nums = [1,2,3,1], k = 2
```

The duplicate `1` exists, but the distance is:

```
3 - 0 = 3
```

Since `3 > 2`, the answer is `false`.

Our implementation explicitly checks the distance before returning `true`, preventing this bug.

### Case 3: Multiple Occurrences of the Same Number

A subtle bug occurs if we fail to update the latest index.

Consider:

```
nums = [1,2,1,1], k = 1
```

At index `2`, the distance to the first `1` is `2`, which is too large. However, at index `3`, the distance to index `2` is only `1`, which should return `true`.

By always updating `last_seen[value]` to the latest index, the algorithm correctly detects the valid nearby duplicate.

### Case 4: Negative Numbers

The problem allows values as small as `-10^9`, so the solution must not rely on positive indexing or bounded value ranges.

For example:

```
nums = [-1, -2, -3, -1]
```

Since hash maps work with arbitrary integer keys, negative numbers are handled naturally without any special logic.
