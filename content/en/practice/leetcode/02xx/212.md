## Solution

## Problem Understanding

The problem gives us a two dimensional grid of lowercase English characters called `board`, along with a list of candidate words called `words`. We must determine which words can actually be formed by traversing the board according to the adjacency rules.

A word is formed by starting at some cell and repeatedly moving one step in one of the four cardinal directions: up, down, left, or right. Diagonal movement is not allowed. Each cell can only be used once while constructing a single word, which means we cannot revisit the same position during the same search path.

The output should contain every word from the input list that exists somewhere on the board. The order of the returned words does not matter.

The constraints are the most important clue for choosing the correct algorithm. The board can be as large as `12 x 12`, which means there can be up to `144` cells. The number of words can be as large as `30,000`, and each word can have length up to `10`.

A naive solution that independently searches the board for every word becomes extremely expensive because many words share prefixes. For example, if the dictionary contains `"eat"`, `"ear"`, `"earn"`, and `"earth"`, then separate searches repeatedly explore the same starting paths. The problem strongly suggests that we should share prefix computations across words.

There are several important edge cases to keep in mind:

- The board may contain repeated letters, which creates many overlapping search paths.
- Some words may share long prefixes, making repeated DFS searches wasteful.
- A word may require revisiting a cell, which is illegal and must be prevented.
- Some words may not exist at all even though all their letters appear in the board.
- The board may be very small, such as `1 x 1`.
- The word list may be very large relative to the board size.

The problem guarantees that all words are unique, which simplifies result handling because we do not need to deduplicate the input.

## Approaches

### Brute Force Approach

The brute force solution searches for each word independently using depth first search and backtracking.

For every word in `words`, we iterate through every board position. Whenever the first character matches, we launch a DFS that attempts to build the word character by character. During the DFS, we recursively explore the four neighboring cells while marking visited cells so they cannot be reused in the same path.

This approach is correct because it explicitly explores every valid path that could form the target word. If a path exists, the DFS eventually finds it. If no path exists, every possible search branch fails.

The problem is performance. Suppose there are `W` words, each of length `L`, and the board contains `m * n` cells. For every word we potentially launch DFS searches from many positions, and each DFS can branch up to four directions. Even with pruning from mismatched characters, the repeated searches become extremely expensive when the word list is large.

The biggest inefficiency is duplicated work. Different words often share prefixes, but the brute force solution recomputes those prefix traversals again and again.

### Optimal Approach, Trie + Backtracking

The key insight is that many words share prefixes, so we should search for all words simultaneously instead of independently.

A Trie, also called a prefix tree, is ideal for this problem because it efficiently represents shared prefixes. Instead of searching separately for `"eat"` and `"ear"`, we store both under the shared prefix `"ea"`.

We then perform DFS from every board cell, but instead of searching for one specific word, we traverse the Trie while traversing the board. If the current board path is not a valid Trie prefix, we immediately stop exploring that branch.

This dramatically reduces unnecessary work because invalid paths are pruned early. The Trie allows the board traversal to simultaneously search for all candidate words in one pass.

Whenever the DFS reaches a Trie node marked as a complete word, we add that word to the result.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(W × m × n × 4^L) | O(L) | Searches every word independently |
| Optimal | O(m × n × 4^L + totalWordsLength) | O(totalWordsLength) | Trie shares prefixes and prunes invalid paths early |

## Algorithm Walkthrough

1. Build a Trie from all input words.

Each Trie node stores a mapping from character to child node. A node may also store a complete word when that node represents the end of a valid dictionary word.

This structure allows efficient prefix checking. If a DFS path does not exist in the Trie, we know no word can continue from that path.
2. Initialize an empty result list.

This list stores all words discovered during board traversal.
3. Start DFS from every cell in the board.

Every board position could potentially begin a valid word, so we launch a search from each coordinate.
4. During DFS, check whether the current board character exists in the current Trie node.

If the character is not present, the current path cannot form any valid word, so we stop immediately. This pruning is the main optimization.
5. Move into the matching Trie child node.

Advancing in the Trie means we have matched one more character in a potential word.
6. If the Trie node stores a completed word, add it to the result.

Reaching such a node means the current board path exactly forms one of the target words.

To avoid duplicates, remove the stored word after adding it once.
7. Mark the current board cell as visited.

We temporarily replace the character with a sentinel value such as `"#"` so recursive calls cannot revisit it.
8. Explore all four directions.

Recursively search the neighboring cells:

1. Up
2. Down
3. Left
4. Right

This explores every valid continuation of the current path.
9. Restore the original character after recursion finishes.

This is the backtracking step. It ensures other DFS paths can reuse the cell later.
10. Optionally prune empty Trie branches.

If a Trie node no longer contains children after exploration, we can remove it from its parent. This slightly improves performance by shrinking the Trie during execution.

### Why it works

The algorithm works because the DFS explores every valid board path while the Trie guarantees that only prefixes of actual words are explored. Every discovered word corresponds to a legal board traversal because cells are marked visited during the current recursion path. Conversely, every valid word in the board must correspond to some Trie path, so the DFS eventually reaches and records it.

## Python Solution

```
from typing import List

class TrieNode:
    def __init__(self):
        self.children = {}
        self.word = None

class Solution:
    def findWords(self, board: List[List[str]], words: List[str]) -> List[str]:
        root = TrieNode()

        # Build Trie
        for word in words:
            node = root

            for char in word:
                if char not in node.children:
                    node.children[char] = TrieNode()

                node = node.children[char]

            node.word = word

        rows = len(board)
        cols = len(board[0])

        result = []

        def dfs(row: int, col: int, node: TrieNode) -> None:
            char = board[row][col]

            if char not in node.children:
                return

            next_node = node.children[char]

            # Found a complete word
            if next_node.word:
                result.append(next_node.word)
                next_node.word = None

            # Mark visited
            board[row][col] = "#"

            directions = [
                (1, 0),
                (-1, 0),
                (0, 1),
                (0, -1)
            ]

            for dr, dc in directions:
                new_row = row + dr
                new_col = col + dc

                if (
                    0 <= new_row < rows
                    and 0 <= new_col < cols
                    and board[new_row][new_col] != "#"
                ):
                    dfs(new_row, new_col, next_node)

            # Restore cell
            board[row][col] = char

            # Trie pruning
            if not next_node.children:
                del node.children[char]

        for row in range(rows):
            for col in range(cols):
                dfs(row, col, root)

        return result
```

The implementation begins by defining a `TrieNode` class. Each node contains a dictionary of child nodes and an optional `word` field. The `word` field is only populated when the node represents the end of a valid dictionary word.

The Trie construction step inserts every word character by character. Shared prefixes naturally reuse existing nodes.

The DFS function is the core of the solution. It first checks whether the current board character exists in the Trie node. If not, the search immediately stops because no valid word can continue from this path.

When a complete word is found, it is appended to the result list. The code then sets `next_node.word = None` so the same word cannot be added multiple times through different traversal paths.

The board cell is temporarily replaced with `"#"` to mark it as visited. This prevents revisiting the same cell within the current search path.

After recursively exploring the four directions, the original character is restored. This is classic backtracking behavior that allows future searches to reuse the cell.

Finally, the Trie pruning optimization removes child nodes that are no longer needed. Once a subtree becomes empty, it can never contribute to another match, so deleting it reduces future search overhead.

## Go Solution

```
package main

type TrieNode struct {
	children map[byte]*TrieNode
	word     string
}

func newTrieNode() *TrieNode {
	return &TrieNode{
		children: make(map[byte]*TrieNode),
	}
}

func findWords(board [][]byte, words []string) []string {
	root := newTrieNode()

	// Build Trie
	for _, word := range words {
		node := root

		for i := 0; i < len(word); i++ {
			char := word[i]

			if _, exists := node.children[char]; !exists {
				node.children[char] = newTrieNode()
			}

			node = node.children[char]
		}

		node.word = word
	}

	rows := len(board)
	cols := len(board[0])

	result := []string{}

	var dfs func(int, int, *TrieNode)

	dfs = func(row int, col int, node *TrieNode) {
		char := board[row][col]

		nextNode, exists := node.children[char]

		if !exists {
			return
		}

		// Found a complete word
		if nextNode.word != "" {
			result = append(result, nextNode.word)
			nextNode.word = ""
		}

		// Mark visited
		board[row][col] = '#'

		directions := [][2]int{
			{1, 0},
			{-1, 0},
			{0, 1},
			{0, -1},
		}

		for _, dir := range directions {
			newRow := row + dir[0]
			newCol := col + dir[1]

			if newRow >= 0 &&
				newRow < rows &&
				newCol >= 0 &&
				newCol < cols &&
				board[newRow][newCol] != '#' {

				dfs(newRow, newCol, nextNode)
			}
		}

		// Restore cell
		board[row][col] = char

		// Trie pruning
		if len(nextNode.children) == 0 {
			delete(node.children, char)
		}
	}

	for row := 0; row < rows; row++ {
		for col := 0; col < cols; col++ {
			dfs(row, col, root)
		}
	}

	return result
}
```

The Go implementation closely mirrors the Python version, but there are some language specific details.

Go uses `map[byte]*TrieNode` for child storage because board characters are bytes. The DFS function is implemented as a closure so it can access shared variables like `result`, `rows`, and `cols`.

Instead of using `None` to mark a discovered word as consumed, Go uses an empty string `""`.

The board is modified in place using the sentinel character `'#'` during recursion, then restored afterward. Slices in Go are reference types, so these updates affect the shared board structure directly.

board =

  ["o","a","a","n"],

### Trie Structure

| Prefix | Complete Word |
| --- | --- |
| o -> a -> t -> h | oath |
| p -> e -> a | pea |
| e -> a -> t | eat |
| r -> a -> i -> n | rain |

### DFS Traversal

Start at `(0,0)` which contains `'o'`.

| Step | Position | Current Path | Trie Valid? |
| --- | --- | --- | --- |
| 1 | (0,0) | o | Yes |
| 2 | (0,1) | oa | Yes |
| 3 | (1,1) | oat | Yes |
| 4 | (2,1) | oath | Complete word found |

The algorithm adds `"oath"` to the result.

Next, another DFS path starts at `(1,3)`:

| Step | Position | Current Path | Trie Valid? |
| --- | --- | --- | --- |
| 1 | (1,3) | e | Yes |
| 2 | (1,2) | ea | Yes |
| 3 | (1,1) | eat | Complete word found |

The algorithm adds `"eat"` to the result.

The words `"pea"` and `"rain"` are never completed because no valid board path exists for them.

Final result:

```
["oath", "eat"]
```

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(m × n × 4^L + totalWordsLength) | DFS explores valid paths while Trie construction costs total character count |
| Space | O(totalWordsLength + L) | Trie storage plus recursion stack |

The Trie construction phase inserts every character from every word exactly once, so its cost is proportional to the total number of characters across all words.

The DFS explores paths on the board. In the worst case, every path of length `L` may be explored, where `L` is the maximum word length. However, the Trie significantly reduces exploration because invalid prefixes terminate early.

The recursion depth is at most the maximum word length, which is bounded by `10`.

## Edge Cases

### Words Requiring Cell Reuse

One subtle bug occurs when a DFS accidentally revisits the same board position during a single word search. For example, on a board containing `["a", "b"]`, the word `"aba"` should not be considered valid because the first `'a'` cell would need to be reused.

The implementation prevents this by temporarily replacing the current board cell with `"#"` during recursion. Since recursive calls skip cells marked `"#"`, revisiting becomes impossible.

### Multiple Words Sharing Prefixes

A large dictionary may contain many overlapping prefixes such as `"app"`, `"apple"`, `"apply"`, and `"application"`.

A naive implementation would repeatedly traverse the same board paths for every word. The Trie eliminates this duplication by storing shared prefixes once. The DFS simultaneously searches for all matching continuations from the current prefix.

### Duplicate Discovery Paths

A word may be discoverable through multiple board paths. Without careful handling, the result list could contain duplicates.

The implementation solves this by setting `next_node.word = None` immediately after finding a word. This guarantees each dictionary word is added only once even if another traversal later reaches the same Trie node.
