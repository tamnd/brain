## Solution

## Problem Understanding

The problem asks us to group strings that belong to the same cyclic shifting sequence. Two strings belong to the same group if one string can be transformed into the other by repeatedly shifting every character forward or backward in the alphabet, with wraparound between `'z'` and `'a'`.

For example, the string `"abc"` becomes `"bcd"` after one right shift, `"cde"` after another shift, and eventually `"xyz"`, `"yza"`, and `"zab"` as the sequence wraps around the alphabet. Because shifting applies uniformly to every character, the relative distances between adjacent characters never change.

The input is an array of lowercase English strings. The output should group together all strings that share the same shifting pattern. The order of groups does not matter, and the order of strings inside each group also does not matter.

An important detail is that strings can only belong to the same sequence if they have the same length. A string of length 3 can never shift into a string of length 2 because shifting changes characters but never changes length.

The constraints are relatively small:

- Up to 200 strings
- Each string length up to 50

These limits mean even moderately inefficient solutions could pass, but the problem is fundamentally about recognizing structural similarity between strings, so there is a cleaner and more scalable hashing-based solution.

Several edge cases are important:

- Single-character strings, such as `"a"` and `"z"`, all belong to the same shifting sequence because any single letter can shift into any other single letter.
- Wraparound cases like `"az"` and `"ba"` must be handled correctly.
- Strings with identical characters like `"aaa"` and `"bbb"` belong together because each adjacent difference is zero.
- Different-length strings should never be grouped together, even if their patterns appear similar.

## Approaches

### Brute Force Approach

A brute-force solution would compare every pair of strings and determine whether they belong to the same shifting sequence.

To check whether two strings are shift-equivalent, we can compute the shift difference between corresponding characters. If the difference is consistent across all positions, the strings belong to the same group.

For example:

- `"abc"` to `"bcd"` has a shift of `+1` for every character
- `"xyz"` to `"yza"` also has a shift of `+1` for every character

The brute-force algorithm would repeatedly scan ungrouped strings and compare them with all others.

This approach is correct because it explicitly verifies the defining property of shift-equivalence. However, it becomes inefficient because every string may need to be compared against many others, and each comparison takes up to `O(L)` time where `L` is the string length.

### Key Insight

The important observation is that strings in the same shifting sequence share the same pattern of relative character differences.

Instead of comparing strings pairwise, we can normalize every string into a canonical representation.

For example:

- `"abc"` → differences `(1,1)`
- `"bcd"` → differences `(1,1)`
- `"xyz"` → differences `(1,1)`

Similarly:

- `"az"` → difference `(25)`
- `"ba"` → difference `(25)`

If two strings produce the same difference pattern, they belong to the same group.

This transforms the problem into a standard hash grouping problem:

1. Compute a signature for each string
2. Use the signature as a hash map key
3. Collect all strings with the same signature together

This avoids pairwise comparisons entirely.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(N² × L) | O(N) | Compare every pair of strings |
| Optimal | O(N × L) | O(N × L) | Use normalized difference pattern as hash key |

Where:

- `N` = number of strings
- `L` = maximum string length

## Algorithm Walkthrough

### Optimal Algorithm

1. Create a hash map where:

- The key represents the normalized shifting pattern
- The value is the list of strings belonging to that pattern
2. For each string, compute its signature.

The signature is built from the differences between adjacent characters.

For example:

- `"abc"`:

- `b - a = 1`
- `c - b = 1`
- signature = `(1, 1)`
- `"xyz"`:

- `y - x = 1`
- `z - y = 1`
- signature = `(1, 1)`
3. Handle wraparound correctly using modular arithmetic.

For example:

- `"az"`:

- `z - a = 25`
- `"ba"`:

- `a - b = -1`
- Normalize:

$(-1 + 26) \bmod 26 = 25$
- signature = `(25)`
4. Use the signature as the hash map key and append the string to the corresponding group.
5. After processing all strings, return all grouped values from the hash map.

### Why it works

The key invariant is that shifting every character by the same amount preserves the differences between adjacent characters.

If two strings belong to the same shifting sequence, their adjacent-character differences must match exactly after modular normalization. Conversely, if the difference patterns match, one string can be transformed into the other through uniform cyclic shifting.

Therefore, the normalized difference pattern uniquely identifies a shifting sequence.

## Python Solution

```
from collections import defaultdict
from typing import List, Tuple

class Solution:
    def groupStrings(self, strings: List[str]) -> List[List[str]]:
        def build_key(word: str) -> Tuple[int, ...]:
            differences = []

            for i in range(1, len(word)):
                diff = (ord(word[i]) - ord(word[i - 1])) % 26
                differences.append(diff)

            return tuple(differences)

        groups = defaultdict(list)

        for word in strings:
            key = build_key(word)
            groups[key].append(word)

        return list(groups.values())
```

The implementation begins by defining a helper function called `build_key`. This function converts a string into its normalized signature.

The signature is computed by iterating through adjacent characters and calculating their modular difference. Using modulo `26` ensures wraparound cases are handled correctly.

The differences are stored in a tuple because tuples are immutable and hashable, making them suitable dictionary keys.

A `defaultdict(list)` is then used to collect strings by their signature. Every string is processed once, its key is computed, and it is appended to the correct group.

Finally, the dictionary values are returned as the grouped result.

## Go Solution

```
package main

import (
	"strconv"
	"strings"
)

func groupStrings(stringsInput []string) [][]string {
	groups := make(map[string][]string)

	buildKey := func(word string) string {
		if len(word) == 1 {
			return ""
		}

		var builder strings.Builder

		for i := 1; i < len(word); i++ {
			diff := (int(word[i]) - int(word[i-1]) + 26) % 26

			builder.WriteString(strconv.Itoa(diff))
			builder.WriteByte(',')
		}

		return builder.String()
	}

	for _, word := range stringsInput {
		key := buildKey(word)
		groups[key] = append(groups[key], word)
	}

	result := make([][]string, 0, len(groups))

	for _, group := range groups {
		result = append(result, group)
	}

	return result
}
```

The Go implementation follows the same logic as the Python version, but there are a few language-specific differences.

Go does not support tuples as map keys in the same flexible way Python does, so the signature is serialized into a string using a `strings.Builder`.

Each normalized difference is appended to the builder with commas as separators to avoid ambiguity.

The hash map uses `map[string][]string`, where the key is the serialized signature and the value is the slice of grouped strings.

Go slices naturally handle dynamic appending, so building the result is straightforward.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(N × L) | Each string is scanned once to compute its signature |
| Space | O(N × L) | Hash map stores all strings and their signatures |

The algorithm processes each character exactly once while building signatures. Since each string has length at most `L`, processing one string costs `O(L)`. Repeating this for `N` strings gives total time complexity `O(N × L)`.

The space usage comes from storing signatures and grouped strings inside the hash map.

## Edge Cases

### Single-Character Strings

Strings of length one are a special case because there are no adjacent character differences to compute. A naive implementation might accidentally treat each letter as unique.

However, every single-character string belongs to the same shifting sequence because any letter can shift into any other letter through repeated cyclic shifts.

The implementation handles this naturally because the generated signature is an empty tuple or empty string for all one-character inputs.

### Alphabet Wraparound

Cases like `"az"` and `"ba"` are easy sources of bugs because subtraction can produce negative values.

Without modular arithmetic:

- `'a' - 'b' = -1`

But cyclic shifting means this should actually represent a shift of `25`.

The implementation fixes this by applying modulo `26`:

$(current - previous + 26) \bmod 26$

This guarantees every difference falls within `[0, 25]`.

### Strings with Repeated Characters

Strings like `"aaa"` and `"bbb"` may appear different at first glance, but they belong to the same sequence because every adjacent difference is zero.

A flawed implementation that compares absolute characters instead of relative differences would fail here.

The current solution correctly computes:

- `"aaa"` → `(0,0)`
- `"bbb"` → `(0,0)`

so they are grouped together properly.
