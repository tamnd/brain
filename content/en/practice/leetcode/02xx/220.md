## Solution

## Problem Understanding

The problem asks us to determine whether there exist two different indices in the array such that two conditions are simultaneously satisfied.

The first condition is positional proximity. The indices must be close enough together:

$$|i - j| \leq indexDiff$$

This means we are only allowed to compare elements that are within a sliding window of size `indexDiff`.

The second condition is value proximity. The numbers themselves must also be close enough:

$$|nums[i] - nums[j]| \leq valueDiff$$

So the task is not simply about duplicates. Instead, we are searching for two numbers that are numerically close while also appearing near each other in the array.

The input consists of:

- `nums`, an integer array
- `indexDiff`, the maximum allowed distance between indices
- `valueDiff`, the maximum allowed difference between values

The output is a boolean:

- `true` if at least one valid pair exists
- `false` otherwise

The constraints are important because they rule out inefficient solutions.

- `nums.length` can be as large as `10^5`
- Values can range from `-10^9` to `10^9`

A naive comparison of every pair would require checking approximately:

$O(n^2)$

pairs in the worst case, which is far too slow for `10^5` elements.

Several edge cases are especially important:

- `valueDiff = 0`, which reduces the problem to checking for exact duplicates within a window
- Negative numbers, which can create bucket-indexing mistakes
- Very large integers, which can overflow in some languages if subtraction is done carelessly
- Windows larger than the array itself
- Repeated values appearing just outside the allowed index range

A correct solution must efficiently maintain information about only the recent elements that are still eligible under the `indexDiff` constraint.

## Approaches

### Brute Force

The simplest solution is to examine every possible pair of indices `(i, j)` and check whether both conditions are satisfied.

For every index `i`, we can compare it with all later indices `j` such that:

$$j - i \leq indexDiff$$

For each pair, we compute:

$$|nums[i] - nums[j]|$$

and return `true` if the value difference is small enough.

This approach is correct because it directly checks every valid candidate pair. However, its performance is too slow for large inputs.

In the worst case, each element may compare against nearly `indexDiff` other elements. If `indexDiff` is close to `n`, the runtime becomes quadratic.

### Optimal Approach, Bucket-Based Sliding Window

The key observation is that we do not need to compare every pair explicitly.

Suppose two numbers differ by at most `valueDiff`. If we divide the number line into buckets of width:

$$valueDiff + 1$$

then any two numbers satisfying the condition must either:

- fall into the same bucket, or
- fall into neighboring buckets

Why does this work?

If bucket width is `valueDiff + 1`, then two numbers inside the same bucket are guaranteed to differ by at most `valueDiff`.

For example, if `valueDiff = 3`, bucket width becomes `4`.

- Bucket 0 contains `[0,1,2,3]`
- Bucket 1 contains `[4,5,6,7]`

Any two numbers in the same bucket differ by at most `3`.

This lets us avoid scanning an entire sliding window. Instead, for each number:

1. Compute its bucket ID
2. Check:

- the same bucket
- the left neighboring bucket
- the right neighboring bucket
3. Maintain only the last `indexDiff` elements using a sliding window

This produces an efficient linear-time solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | $O(n \cdot indexDiff)$ | $O(1)$ | Directly checks all nearby pairs |
| Optimal | $O(n)$ | $O(indexDiff)$ | Uses buckets plus sliding window |

## Algorithm Walkthrough

1. Define the bucket width as `valueDiff + 1`.

We use `valueDiff + 1` instead of `valueDiff` because when `valueDiff = 0`, bucket width cannot be zero. This also guarantees that numbers in the same bucket differ by at most `valueDiff`.
2. Create a hash map called `buckets`.

The key is the bucket ID.

The value is the actual number stored in that bucket.

At any moment, the map contains only numbers inside the current sliding window.
3. Iterate through the array from left to right.

For each number `num`, compute its bucket ID.

Conceptually:

$$bucketId = \left\lfloor \frac{num}{valueDiff + 1} \right\rfloor$$

Care must be taken with negative numbers because integer division behaves differently across languages.
4. Check whether the current bucket already contains a number.

If it does, then two numbers fall into the same bucket.

Since bucket width is `valueDiff + 1`, their difference must be at most `valueDiff`.

We can immediately return `true`.
5. Check the neighboring buckets.

A valid pair may lie across a bucket boundary.

Therefore, check:

- `bucketId - 1`
- `bucketId + 1`

If either exists, compute the absolute difference explicitly.
6. Insert the current number into its bucket.

This makes it available for future comparisons.
7. Maintain the sliding window.

Once the window exceeds size `indexDiff`, remove the element that falls out of range.

Specifically, when processing index `i`, remove:

$$nums[i - indexDiff]$$

because it is no longer allowed under the index constraint.
8. If the loop finishes without finding a valid pair, return `false`.

### Why it works

The algorithm maintains the invariant that the hash map contains exactly the elements whose indices are within `indexDiff` of the current index.

Because bucket width equals `valueDiff + 1`, any two numbers satisfying the value constraint must appear either in the same bucket or adjacent buckets. Therefore, checking only those buckets is sufficient.

Since every valid candidate is examined and invalid distant elements are removed from the window, the algorithm correctly detects whether a valid pair exists.

## Python Solution

```
from typing import List

class Solution:
    def containsNearbyAlmostDuplicate(
        self,
        nums: List[int],
        indexDiff: int,
        valueDiff: int
    ) -> bool:

        if valueDiff < 0:
            return False

        bucket_size = valueDiff + 1
        buckets = {}

        for i, num in enumerate(nums):

            bucket_id = num // bucket_size

            # Same bucket
            if bucket_id in buckets:
                return True

            # Left neighbor
            if (
                bucket_id - 1 in buckets and
                abs(num - buckets[bucket_id - 1]) <= valueDiff
            ):
                return True

            # Right neighbor
            if (
                bucket_id + 1 in buckets and
                abs(num - buckets[bucket_id + 1]) <= valueDiff
            ):
                return True

            buckets[bucket_id] = num

            # Remove element outside sliding window
            if i >= indexDiff:
                old_num = nums[i - indexDiff]
                old_bucket_id = old_num // bucket_size
                del buckets[old_bucket_id]

        return False
```

The implementation begins by defining the bucket size as `valueDiff + 1`. This guarantees that any two numbers in the same bucket satisfy the required value constraint.

The `buckets` dictionary stores the current sliding window. Each bucket can contain at most one value because encountering a second value in the same bucket immediately proves the answer is `true`.

For each number, the algorithm computes its bucket ID using integer division. It first checks the current bucket, then the neighboring buckets, because close values may straddle a bucket boundary.

After all checks are complete, the current number is inserted into the appropriate bucket.

The final section maintains the sliding window. Once the current index exceeds `indexDiff`, the oldest eligible element is removed from the hash map. This ensures that every stored value always satisfies the index-distance constraint.

## Go Solution

```
package main

import "math"

func containsNearbyAlmostDuplicate(nums []int, indexDiff int, valueDiff int) bool {
	if valueDiff < 0 {
		return false
	}

	bucketSize := int64(valueDiff) + 1
	buckets := make(map[int64]int64)

	getBucketID := func(num int64) int64 {
		if num >= 0 {
			return num / bucketSize
		}
		return ((num + 1) / bucketSize) - 1
	}

	for i, num := range nums {
		n := int64(num)
		bucketID := getBucketID(n)

		// Same bucket
		if _, exists := buckets[bucketID]; exists {
			return true
		}

		// Left neighbor
		if val, exists := buckets[bucketID-1]; exists {
			if int64(math.Abs(float64(n-val))) <= int64(valueDiff) {
				return true
			}
		}

		// Right neighbor
		if val, exists := buckets[bucketID+1]; exists {
			if int64(math.Abs(float64(n-val))) <= int64(valueDiff) {
				return true
			}
		}

		buckets[bucketID] = n

		// Remove old element outside window
		if i >= indexDiff {
			oldNum := int64(nums[i-indexDiff])
			oldBucketID := getBucketID(oldNum)
			delete(buckets, oldBucketID)
		}
	}

	return false
}
```

The Go version follows the same algorithmic structure as the Python implementation, but several language-specific details are important.

The solution uses `int64` throughout bucket calculations to avoid overflow when dealing with values near `10^9`.

Go integer division truncates toward zero, which breaks bucket handling for negative numbers. Because of this, a custom `getBucketID` helper is required to correctly emulate mathematical floor division.

The map type is:

```
map[int64]int64
```

where the key is the bucket ID and the value is the stored number.

Unlike Python, Go does not provide a built-in absolute value function for integers, so the implementation uses `math.Abs` with temporary conversion to `float64`.

## Worked Examples

### Example 1

Input:

```
nums = [1,2,3,1]
indexDiff = 3
valueDiff = 0
```

Bucket size:

$$valueDiff + 1 = 1$$

| i | num | bucket | buckets before | Result |
| --- | --- | --- | --- | --- |
| 0 | 1 | 1 | {} | insert 1 |
| 1 | 2 | 2 | {1:1} | insert 2 |
| 2 | 3 | 3 | {1:1,2:2} | insert 3 |
| 3 | 1 | 1 | {1:1,2:2,3:3} | same bucket found |

At index `3`, bucket `1` already contains value `1`.

Therefore:

$$|1 - 1| = 0$$

and:

$$|3 - 0| = 3$$

Both constraints are satisfied, so the algorithm returns `true`.

### Example 2

Input:

```
nums = [1,5,9,1,5,9]
indexDiff = 2
valueDiff = 3
```

Bucket size:

$$3 + 1 = 4$$

| i | num | bucket | buckets before | Action |
| --- | --- | --- | --- | --- |
| 0 | 1 | 0 | {} | insert |
| 1 | 5 | 1 | {0:1} | neighbor diff = 4, reject |
| 2 | 9 | 2 | {0:1,1:5} | neighbor diff = 4, reject |
| 3 | 1 | 0 | {1:5,2:9} | no valid match |
| 4 | 5 | 1 | {0:1,2:9} | no valid match |
| 5 | 9 | 2 | {0:1,1:5} | no valid match |

Although repeated values exist, they are always more than `indexDiff` apart.

The sliding window removes old values before they become eligible matches again, so the algorithm correctly returns `false`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | $O(n)$ | Each element is inserted and removed at most once |
| Space | $O(indexDiff)$ | The sliding window stores at most `indexDiff` elements |

The runtime is linear because every operation on the hash map is expected constant time. Each array element participates in a fixed number of bucket checks and at most one insertion and removal.

The space complexity depends on the maximum size of the sliding window. At most `indexDiff` elements remain in memory simultaneously.

## Edge Cases

One important edge case occurs when `valueDiff = 0`. In this situation, the problem becomes equivalent to finding exact duplicates within the allowed index range. A common mistake is using bucket width equal to `valueDiff`, which would produce division by zero. The implementation avoids this by defining bucket width as `valueDiff + 1`.

Another subtle edge case involves negative numbers. Integer division behaves differently across programming languages, especially for negative operands. Incorrect bucket calculations can place nearby negative values into unrelated buckets, causing false negatives. The Go implementation explicitly corrects this with a custom bucket-ID function that emulates mathematical floor division.

A third critical edge case involves values that are numerically close but lie on opposite sides of a bucket boundary. For example, with `valueDiff = 3`, the values `3` and `4` belong to adjacent buckets even though their difference is only `1`. A flawed implementation that checks only the current bucket would miss this valid pair. The algorithm correctly handles this by always checking neighboring buckets.

Another important scenario occurs when repeated values exist but are farther apart than `indexDiff`. Without proper sliding window maintenance, stale elements could remain in the hash map and incorrectly produce `true`. The implementation removes outdated elements immediately once they leave the valid index range, guaranteeing correctness.
