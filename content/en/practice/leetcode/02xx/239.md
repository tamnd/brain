## Solution

## Problem Understanding

The problem gives an integer array `nums` and a window size `k`. A sliding window of length `k` moves across the array from left to right, one position at a time. For every position of the window, we must determine the maximum value currently visible inside that window.

If the array has length `n`, then there are exactly `n - k + 1` valid windows. For each of these windows, we compute one maximum value and append it to the result.

For example, if:

```
nums = [1,3,-1,-3,5,3,6,7]
k = 3
```

then the first window contains:

```
[1,3,-1]
```

whose maximum is `3`.

After shifting one step right, the next window becomes:

```
[3,-1,-3]
```

whose maximum is still `3`.

We continue until the window reaches the end of the array.

The constraints are important:

- `nums.length` can be as large as `100000`
- `k` can also be as large as `nums.length`

These limits immediately tell us that inefficient approaches will fail. In particular, recomputing the maximum from scratch for every window would be too slow because there can be up to `100000` windows, each of size up to `100000`.

The problem guarantees:

- `k >= 1`
- `k <= nums.length`

so every window is valid and non-empty.

Several edge cases are especially important:

- When `k = 1`, every element is its own window, so the answer is simply the original array.
- When `k = nums.length`, there is only one window, covering the entire array.
- Arrays with repeated values can cause issues if the implementation incorrectly removes duplicates.
- Strictly increasing or strictly decreasing arrays stress the behavior of the data structure used for tracking maxima.

## Approaches

### Brute Force Approach

The most direct solution is to examine every sliding window independently.

For each starting position `i`, we inspect the subarray:

```
nums[i : i + k]
```

and compute its maximum using a linear scan.

This approach is correct because it explicitly checks every element inside every window and therefore always finds the true maximum.

However, the performance is poor. There are `n - k + 1` windows, and each window requires up to `k` comparisons. This leads to a total time complexity of:

```
O((n - k + 1) * k)
```

which becomes `O(nk)` in the worst case.

With `n = 100000`, this can easily exceed acceptable runtime limits.

### Optimal Approach, Monotonic Deque

The key observation is that we do not need to recompute the maximum from scratch after every window shift.

When the window moves one step to the right:

- One element leaves the window
- One new element enters the window

Most elements remain unchanged.

We can maintain candidate maximum values using a deque that stores indices in decreasing order of their corresponding values.

The deque has two critical properties:

1. The front always contains the index of the current window maximum.
2. Elements smaller than a newly added value are removed from the back because they can never become maximum while the larger element remains in the window.

This structure is called a monotonic decreasing deque.

Each element is inserted once and removed at most once, which gives linear time complexity.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(nk) | O(1) | Recomputes maximum for every window |
| Optimal | O(n) | O(k) | Uses a monotonic deque to maintain candidates efficiently |

## Algorithm Walkthrough

1. Initialize an empty deque.

The deque will store indices, not values. Storing indices allows us to determine whether an element has moved out of the current window.
2. Iterate through the array using index `i`.

For every element `nums[i]`, we update the deque to maintain the monotonic decreasing property.
3. Remove indices that are outside the current window.

The current window spans:

```
[i - k + 1, i]
```

If the index at the front of the deque is smaller than `i - k + 1`, it is no longer inside the window and must be removed.
4. Remove smaller elements from the back of the deque.

While the deque is not empty and:

```
nums[deque[-1]] < nums[i]
```

remove the back element.

This step is crucial because any smaller value behind the current value can never become the maximum in any future window that also contains the current value.
5. Add the current index to the deque.

After removing irrelevant elements, append `i`.
6. Start recording answers once the first full window is formed.

The first valid window appears when:

```
i >= k - 1
```

At this point, the front of the deque always contains the index of the maximum element for the current window.
7. Append the maximum to the result.

The maximum value is:

```
nums[deque[0]]
```

### Why it works

The deque always maintains indices in decreasing order of their corresponding values. Because smaller elements are removed whenever a larger element enters, the front of the deque is always the largest value currently inside the window.

Additionally, indices that leave the window are removed immediately, ensuring every index in the deque always belongs to the current window.

Together, these invariants guarantee that the front of the deque always represents the correct maximum for the current sliding window.

## Python Solution

```
from collections import deque
from typing import List

class Solution:
    def maxSlidingWindow(self, nums: List[int], k: int) -> List[int]:
        window = deque()
        result = []

        for i in range(len(nums)):
            # Remove indices outside the current window
            while window and window[0] <= i - k:
                window.popleft()

            # Remove smaller values from the back
            while window and nums[window[-1]] < nums[i]:
                window.pop()

            # Add current index
            window.append(i)

            # Record result once the first window is complete
            if i >= k - 1:
                result.append(nums[window[0]])

        return result
```

The implementation begins by creating a deque called `window`. This deque stores indices of elements in decreasing order of value.

The first `while` loop removes indices that are no longer inside the current sliding window. Since the left boundary of the window is `i - k + 1`, any index less than or equal to `i - k` is outdated.

The second `while` loop maintains the monotonic decreasing property. If the current element is larger than elements at the back of the deque, those smaller elements are removed because they can never become future maxima.

After cleanup, the current index is appended.

Once the window size reaches `k`, the maximum value for the current window is always located at the front of the deque, so we append `nums[window[0]]` to the result list.

Because each index enters and leaves the deque at most once, the algorithm runs in linear time.

## Go Solution

```
func maxSlidingWindow(nums []int, k int) []int {
    deque := []int{}
    result := []int{}

    for i := 0; i < len(nums); i++ {
        // Remove indices outside the current window
        if len(deque) > 0 && deque[0] <= i-k {
            deque = deque[1:]
        }

        // Remove smaller values from the back
        for len(deque) > 0 && nums[deque[len(deque)-1]] < nums[i] {
            deque = deque[:len(deque)-1]
        }

        // Add current index
        deque = append(deque, i)

        // Record result once the first window is complete
        if i >= k-1 {
            result = append(result, nums[deque[0]])
        }
    }

    return result
}
```

The Go implementation follows the same algorithmic logic as the Python version, but slices are used to simulate deque operations.

Removing from the front is performed using:

```
deque = deque[1:]
```

and removing from the back uses slice truncation:

```
deque = deque[:len(deque)-1]
```

Go does not provide a built-in deque structure in the standard library, so slices are the simplest approach here.

No special integer overflow handling is necessary because the problem constraints fit comfortably within Go's `int` type.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each index is added and removed from the deque at most once |
| Space | O(k) | The deque stores at most `k` indices |

Although there are nested `while` loops, the total number of deque operations across the entire algorithm is linear. Every element can only be inserted once and removed once, so the amortized cost remains `O(n)`.

The deque never grows larger than the window size, so the auxiliary space complexity is `O(k)`.

## Edge Cases

One important edge case occurs when `k = 1`. In this situation, every window contains exactly one element, so the output should match the original array. A buggy implementation might still attempt unnecessary deque maintenance or incorrectly delay output generation. The current implementation handles this naturally because every index immediately forms a complete window.

Another critical edge case is a strictly decreasing array such as:

```
[5,4,3,2,1]
```

In this case, no elements are removed from the back of the deque because each new value is smaller than previous values. The deque grows to size `k`, and the maximum always remains at the front until it leaves the window. Incorrect implementations sometimes fail to remove expired indices properly, which would produce outdated maxima.

Arrays with duplicate values are also subtle. For example:

```
[7,7,7,7]
```

The implementation removes only strictly smaller values:

```
nums[window[-1]] < nums[i]
```

rather than smaller-or-equal values. This preserves multiple equal candidates correctly and ensures the maximum remains valid even after one duplicate leaves the window.

A final important edge case is when the window size equals the array length. In that scenario, there is only one valid window. The algorithm still works correctly because the deque processes the entire array before producing exactly one output value, the global maximum.
