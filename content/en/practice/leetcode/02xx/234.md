## Solution

## Problem Understanding

The problem gives us the head of a singly linked list and asks whether the sequence of node values forms a palindrome.

A palindrome is a sequence that reads the same forward and backward. For example, the list:

```
1 -> 2 -> 2 -> 1
```

is a palindrome because the values from left to right are identical to the values from right to left.

On the other hand:

```
1 -> 2
```

is not a palindrome because reversing the sequence changes the order.

The input is the head node of a singly linked list. Since the list is singly linked, we can only move forward through the nodes. We cannot move backward directly, which is the main challenge of the problem.

The output is a boolean value:

- `true` if the linked list is a palindrome
- `false` otherwise

The constraints are important:

- The number of nodes can be as large as `10^5`
- Node values range from `0` to `9`

The large input size tells us that inefficient approaches may not be acceptable. In particular, any algorithm worse than `O(n)` time could become too slow.

The follow-up requirement asks whether we can solve the problem in `O(n)` time and `O(1)` extra space. This strongly suggests that the intended optimal solution should avoid using auxiliary data structures proportional to the input size.

Several edge cases are important:

- A single-node list is always a palindrome
- A two-node list may or may not be a palindrome
- Odd-length lists have a middle element that does not need comparison
- Even-length lists split evenly into two halves
- Reversing pointers incorrectly can corrupt the linked list structure

The problem guarantees that at least one node exists, so we never receive an empty list.

## Approaches

### Brute-Force Approach

The most straightforward solution is to copy all linked list values into an array.

Once the values are stored in an array, checking whether the sequence is a palindrome becomes easy because arrays support random access from both ends. We can use two pointers, one starting at the beginning and one at the end, and compare corresponding values until the pointers meet.

This approach is correct because the array preserves the exact order of the linked list values.

However, the drawback is that it requires additional memory proportional to the size of the list. Since the follow-up explicitly asks for `O(1)` extra space, this solution does not satisfy the optimal requirement.

### Optimal Approach

The key observation is that we only need to compare the first half of the list with the reversed second half.

Because the list is singly linked, we cannot traverse backward naturally. Instead of storing values externally, we can reverse the second half of the linked list in place.

The algorithm works in three major phases:

1. Find the middle of the linked list using the fast and slow pointer technique
2. Reverse the second half of the list
3. Compare the first half with the reversed second half

This works because reversing the second half effectively lets us traverse the original list from both directions simultaneously while still using only constant extra memory.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Copy values into an array, then compare from both ends |
| Optimal | O(n) | O(1) | Reverse the second half of the list in place and compare halves |

## Algorithm Walkthrough

1. Initialize two pointers, `slow` and `fast`, both starting at the head of the list.

The purpose of these pointers is to locate the middle of the linked list efficiently. The `slow` pointer moves one step at a time, while the `fast` pointer moves two steps at a time.
2. Move the pointers forward until `fast` reaches the end.

Because `fast` moves twice as quickly, when it reaches the end of the list, `slow` will be positioned at the middle.

For odd-length lists, `slow` lands exactly on the middle node.

For even-length lists, `slow` lands at the start of the second half.
3. Reverse the second half of the linked list starting from `slow`.

Reversing the list allows us to compare values from the end of the original list without requiring backward traversal.

We use the standard linked list reversal technique:

- Store the next node temporarily
- Reverse the current node's pointer
- Advance forward
4. Initialize two pointers for comparison.

- One pointer starts at the original head
- The other starts at the head of the reversed second half
5. Compare corresponding node values one by one.

If any pair of values differs, the list is not a palindrome, so return `False`.
6. If all compared values match, return `True`.

Since the second half has been reversed, matching values imply that the sequence reads identically forward and backward.

### Why it works

The algorithm relies on the fact that a palindrome has symmetric values around its center.

Finding the middle divides the list into two halves. Reversing the second half transforms the comparison problem into a forward traversal problem. Comparing corresponding nodes from the front and reversed back ensures that every symmetric pair is checked exactly once.

If every pair matches, the linked list must be a palindrome.

## Python Solution

```
# Definition for singly-linked list.
# class ListNode:
#     def __init__(self, val=0, next=None):
#         self.val = val
#         self.next = next

from typing import Optional

class Solution:
    def isPalindrome(self, head: Optional[ListNode]) -> bool:
        slow = head
        fast = head

        # Find the middle of the list
        while fast and fast.next:
            slow = slow.next
            fast = fast.next.next

        # Reverse the second half
        prev = None
        current = slow

        while current:
            next_node = current.next
            current.next = prev
            prev = current
            current = next_node

        # Compare first half and reversed second half
        left = head
        right = prev

        while right:
            if left.val != right.val:
                return False

            left = left.next
            right = right.next

        return True
```

The implementation begins by locating the middle of the linked list using the fast and slow pointer technique. The `fast` pointer advances two nodes at a time while `slow` advances one node at a time. This guarantees that `slow` reaches the midpoint when traversal ends.

Next, the second half of the linked list is reversed in place. The variables `prev`, `current`, and `next_node` are used to safely reverse pointers without losing access to the remaining nodes.

After reversal, the code compares nodes from the beginning of the list and the reversed second half simultaneously. The pointer `left` traverses the first half, while `right` traverses the reversed second half.

If any values differ, the function immediately returns `False`. Otherwise, if all comparisons succeed, the list is confirmed to be a palindrome and the function returns `True`.

## Go Solution

```
/**
 * Definition for singly-linked list.
 * type ListNode struct {
 *     Val int
 *     Next *ListNode
 * }
 */
func isPalindrome(head *ListNode) bool {
    slow := head
    fast := head

    // Find the middle
    for fast != nil && fast.Next != nil {
        slow = slow.Next
        fast = fast.Next.Next
    }

    // Reverse second half
    var prev *ListNode
    current := slow

    for current != nil {
        nextNode := current.Next
        current.Next = prev
        prev = current
        current = nextNode
    }

    // Compare both halves
    left := head
    right := prev

    for right != nil {
        if left.Val != right.Val {
            return false
        }

        left = left.Next
        right = right.Next
    }

    return true
}
```

The Go implementation follows the same algorithmic structure as the Python version. One difference is that Go uses explicit pointer types such as `*ListNode`.

Go does not have `None`, so `nil` is used instead. Variable declarations are also more explicit, especially for pointers like `prev`.

Because linked list manipulation is pointer-based in Go, care must be taken during reversal to avoid losing references to remaining nodes. The temporary variable `nextNode` preserves access before changing pointers.

No integer overflow concerns exist here because node values are very small and no arithmetic operations are performed.

| --- | --- | --- |

| Space | O(1) | Only a few pointer variables are used |

The algorithm performs three linear passes over the linked list:

- One pass to find the middle
- One pass to reverse the second half
- One pass to compare values

Each pass is proportional to `n`, so the total runtime remains `O(n)`.

The space complexity is `O(1)` because no additional data structures proportional to the input size are allocated. Only a small fixed number of pointers are used.

## Edge Cases

A single-node list is an important edge case because reversing or splitting the list could accidentally introduce null pointer issues. A list containing only one value is always a palindrome because reading it forward and backward produces the same sequence. The implementation handles this naturally because the comparison phase succeeds immediately.

Odd-length lists require careful handling because the middle element does not need a matching counterpart. For example:

```
1 -> 2 -> 3 -> 2 -> 1
```

The value `3` sits in the center and can be ignored during comparison. The fast and slow pointer approach automatically positions the second half correctly so that comparisons remain valid.

Even-length lists are another common source of bugs because there is no single center node. For example:

```
1 -> 2 -> 2 -> 1
```

The algorithm correctly places `slow` at the beginning of the second half, ensuring both halves contain the same number of nodes.

Lists that are almost palindromes but contain one mismatch are especially important for correctness. For example:

```
1 -> 2 -> 3 -> 2 -> 4
```

A flawed implementation might stop comparisons too early or reverse pointers incorrectly. The comparison loop checks every required symmetric pair, guaranteeing that mismatches are detected properly.
