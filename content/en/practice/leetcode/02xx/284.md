## Solution

## Problem Understanding

This problem asks us to design a wrapper around an existing iterator. The original iterator already supports two operations:

- `next()`, which returns the next element and advances the iterator
- `hasNext()`, which tells us whether more elements remain

Our task is to extend this behavior by adding a new operation:

- `peek()`, which returns the next element without advancing the iterator

The important detail is that calling `peek()` multiple times should always return the same value until `next()` is called. In other words, `peek()` should allow us to inspect the upcoming element while leaving the iterator state unchanged.

The input is not given directly as an array during execution. Instead, the constructor receives an existing iterator object that internally traverses the array. We are expected to build a `PeekingIterator` class that delegates iteration behavior while also supporting peeking.

For example, if the iterator contains `[1, 2, 3]`:

- `next()` returns `1`
- `peek()` returns `2` but does not consume it
- another `peek()` still returns `2`
- `next()` now returns `2`

The constraints are small:

- At most 1000 elements
- At most 1000 method calls

This means efficiency is not extremely critical, but we should still aim for a clean constant time design.

The key challenge is preserving iterator state correctly. A naive implementation could accidentally advance the iterator during `peek()` and lose track of elements.

There are several important edge cases:

- Calling `peek()` repeatedly before `next()`
- Calling `hasNext()` after peeking at the last element
- Handling a single-element iterator
- Ensuring `next()` and `peek()` remain synchronized
- Avoiding consuming elements twice

The problem guarantees that all `next()` and `peek()` calls are valid, meaning we never need to handle invalid access beyond the iterator's range.

## Approaches

### Brute Force Approach

A brute force strategy would attempt to simulate peeking by storing all remaining elements separately. One possible implementation would fully exhaust the original iterator during initialization and copy all values into an array or queue.

Then:

- `peek()` would look at the front element
- `next()` would remove and return the front element
- `hasNext()` would check whether the structure is empty

This works because we completely control the remaining elements after preprocessing.

However, this approach is inefficient in terms of initialization cost and unnecessary memory usage. We lose the benefit of lazy iteration because we eagerly consume the entire iterator immediately.

### Optimal Approach

The key observation is that we only ever need to know one future value: the next element.

Instead of storing all remaining elements, we can cache exactly one value ahead of time.

The idea is:

- During initialization, preload the first element if one exists
- Store it in a variable such as `next_value`
- `peek()` simply returns this cached value
- `next()` returns the cached value and then refreshes the cache using the underlying iterator
- `hasNext()` checks whether a cached value exists

This design keeps all operations in constant time and uses only constant extra space.

The cache effectively acts as a one-element lookahead buffer.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) initialization, O(1) operations | O(n) | Stores all remaining elements eagerly |
| Optimal | O(1) per operation | O(1) | Uses a single cached next element |

## Algorithm Walkthrough

1. Store the original iterator inside the `PeekingIterator` object so we can continue using its `next()` and `hasNext()` methods.
2. During initialization, check whether the iterator has a next element.
3. If it does, immediately consume that element and store it in a cache variable such as `next_value`.
4. If the iterator is empty, mark the cache as empty using a boolean flag or `None`.
5. When `peek()` is called, simply return the cached value. We do not advance the iterator here because the whole purpose of peeking is to inspect without consuming.
6. When `next()` is called:

1. Save the current cached value as the answer.
2. Advance the underlying iterator once to refresh the cache.
3. If another element exists, store it as the new cached value.
4. Otherwise, mark the cache as empty.
5. Return the saved answer.
7. When `hasNext()` is called, return whether the cache currently contains a valid element.

### Why it works

The core invariant is that the cache always stores the next unconsumed element in the iterator.

Because `peek()` only reads the cache, it never changes iterator state. Because `next()` both returns the cached value and refreshes the cache afterward, the iterator remains perfectly synchronized with future operations.

At every moment:

- the cache equals the next element to be returned
- the underlying iterator is positioned immediately after the cache

This guarantees correctness.

## Python Solution

```
# Below is the interface for Iterator, which is already defined for you.
#
# class Iterator:
#     def __init__(self, nums):
#         """
#         Initializes an iterator object to the beginning of a list.
#         :type nums: List[int]
#         """
#
#     def hasNext(self):
#         """
#         Returns true if the iteration has more elements.
#         :rtype: bool
#         """
#
#     def next(self):
#         """
#         Returns the next element in the iteration.
#         :rtype: int
#         """

from typing import Optional

class PeekingIterator:
    def __init__(self, iterator):
        """
        Initialize your data structure here.
        :type iterator: Iterator
        """
        self.iterator = iterator
        self.next_value: Optional[int] = None
        self.has_next_value = False

        if self.iterator.hasNext():
            self.next_value = self.iterator.next()
            self.has_next_value = True

    def peek(self) -> int:
        """
        Returns the next element in the iteration without advancing the iterator.
        :rtype: int
        """
        return self.next_value

    def next(self) -> int:
        """
        :rtype: int
        """
        current_value = self.next_value

        if self.iterator.hasNext():
            self.next_value = self.iterator.next()
            self.has_next_value = True
        else:
            self.next_value = None
            self.has_next_value = False

        return current_value

    def hasNext(self) -> bool:
        """
        :rtype: bool
        """
        return self.has_next_value
```

The implementation begins by storing the original iterator reference. During construction, we preload the first value if it exists. This preload step is the entire foundation of the peeking mechanism.

The `peek()` method is very simple because the next element is always cached. It merely returns the stored value without touching the underlying iterator.

The `next()` method is responsible for maintaining the invariant. It first saves the current cached value because that is the correct element to return. Then it advances the underlying iterator to refresh the cache for future operations.

The `hasNext()` method simply checks whether the cache currently contains a valid value. This avoids needing to query the underlying iterator repeatedly.

## Go Solution

```
/*   Below is the interface for Iterator, which is already defined for you.
 *
 *   type Iterator struct {
 *
 *   }
 *
 *   func (this *Iterator) hasNext() bool {
 *      // Returns true if the iteration has more elements.
 *   }
 *
 *   func (this *Iterator) next() int {
 *      // Returns the next element in the iteration.
 *   }
 */

type PeekingIterator struct {
	iterator     *Iterator
	nextValue    int
	hasNextValue bool
}

func Constructor(iter *Iterator) *PeekingIterator {
	p := &PeekingIterator{
		iterator: iter,
	}

	if iter.hasNext() {
		p.nextValue = iter.next()
		p.hasNextValue = true
	}

	return p
}

func (this *PeekingIterator) hasNext() bool {
	return this.hasNextValue
}

func (this *PeekingIterator) next() int {
	currentValue := this.nextValue

	if this.iterator.hasNext() {
		this.nextValue = this.iterator.next()
		this.hasNextValue = true
	} else {
		this.hasNextValue = false
	}

	return currentValue
}

func (this *PeekingIterator) peek() int {
	return this.nextValue
}
```

The Go implementation follows the exact same logic as the Python version. Since Go integers cannot naturally represent an "empty" state like Python's `None`, we use a separate boolean field named `hasNextValue` to track whether the cached value is valid.

This avoids ambiguity because `0` could theoretically be a legitimate iterator value in a more general version of the problem.

## Edge Cases

One important edge case is a single-element iterator. This is easy to mishandle because after returning the first value, the iterator immediately becomes exhausted. If the cache is not updated correctly, `hasNext()` might incorrectly return `true`. Our implementation explicitly clears the cache state after consuming the final element.

Another important edge case is repeated calls to `peek()`. A flawed implementation might accidentally advance the iterator every time `peek()` is called, causing elements to be skipped. Because our solution stores the next value in a cache and never modifies it during `peek()`, repeated peeks always return the same value safely.

A third edge case occurs when `peek()` is called immediately before the final `next()`. The implementation must ensure that peeking does not consume the value, and that the subsequent `next()` still returns the exact same element. Our cache invariant guarantees this because `peek()` only reads the cached value while `next()` is the only operation that advances state.

A fourth subtle edge case is keeping `hasNext()` accurate after every operation. If we relied solely on the underlying iterator's `hasNext()`, we could get inconsistent behavior because the cached value itself represents a remaining element. By explicitly tracking whether the cache contains a valid value, our implementation keeps `hasNext()` perfectly synchronized with iterator state.
