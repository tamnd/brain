## Solution

## Problem Understanding

This problem asks us to find **all unique combinations of exactly `k` distinct numbers** that add up to a target sum `n`, while following two strict rules:

1. We may only use numbers from `1` to `9`.
2. Each number can be used **at most once**.

The input consists of two integers:

- `k`, the exact number of elements that each combination must contain.
- `n`, the target sum the chosen numbers must add up to.

The output should be a list of all valid combinations. Each combination itself is a list of integers, and the order of combinations does not matter. Additionally, combinations are considered the same regardless of ordering, meaning `[1,2,4]` and `[2,1,4]` represent the same combination and should only appear once.

The key challenge is generating combinations without duplicates while ensuring that every selected number is unique and within the range `[1,9]`.

The constraints are relatively small:

- `2 <= k <= 9`
- `1 <= n <= 60`

Since only numbers from `1` through `9` are allowed, the search space is inherently bounded. Even though brute force is technically possible, we can exploit the structure of the problem to prune invalid paths early and avoid unnecessary work.

Several edge cases are important:

- If `n` is too small to be formed by `k` distinct numbers, no answer exists. For example, `k = 4, n = 1`.
- If `n` is larger than the maximum possible sum using `k` distinct numbers, no solution exists.
- We must avoid duplicate combinations. For example, `[1,2,6]` and `[2,1,6]` should not both appear.
- We must enforce exactly `k` numbers, not fewer or more.

This immediately suggests that we are solving a **constrained combination generation problem**, which makes backtracking a natural fit.

## Approaches

### Brute Force Approach

A brute force solution would generate **every possible subset** of numbers from `1` to `9`, then filter the subsets based on two conditions:

1. The subset length equals `k`.
2. The subset sum equals `n`.

Since there are `9` numbers, every number can either be included or excluded, producing `2^9 = 512` subsets. For each subset, we would compute the size and sum.

This approach is correct because it exhaustively checks every possible valid subset. If a solution exists, it will eventually be found.

However, brute force performs unnecessary work. It explores combinations that are clearly invalid early on, such as:

- Paths already exceeding the target sum.
- Paths using too many numbers.
- Duplicate orderings.

Instead of exploring everything and filtering afterward, we can construct only valid combinations incrementally.

### Optimal Approach, Backtracking with Pruning

The key insight is that we can build combinations **incrementally** and stop exploring a path as soon as it becomes impossible to produce a valid result.

Since numbers must be distinct and chosen from `1` to `9`, we can recursively decide which number to include next.

At every recursive step:

- We maintain the current combination.
- We track the remaining sum needed.
- We track how many numbers have been selected.
- We ensure future choices always come from larger numbers to avoid duplicates.

Pruning makes the solution efficient:

- If the remaining sum becomes negative, we stop.
- If the combination size exceeds `k`, we stop.
- If we already have `k` numbers but the sum is not zero, we stop.

This avoids exploring large portions of the search tree.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^9 × 9) | O(9) | Generates all subsets, then filters |
| Optimal | O(C(9, k)) | O(k) | Backtracking explores only valid paths with pruning |

## Algorithm Walkthrough

We use a **depth first search (DFS) backtracking** approach.

1. **Initialize the result list**

We create a list called `result` to store all valid combinations. Each valid combination found during recursion will be copied into this list.
2. **Define a recursive backtracking function**

The recursive function keeps track of:

- `start`, the smallest number we are allowed to pick next.
- `remaining_sum`, how much is left to reach `n`.
- `current_combination`, the numbers selected so far.

The `start` variable is important because it prevents duplicate combinations. Once we choose `3`, we only consider numbers larger than `3` afterward.
3. **Check for a valid combination**

If:

- the current combination contains exactly `k` numbers, and
- the remaining sum equals `0`

then we have found a valid answer.

We append a copy of the current combination to the result list.
4. **Prune impossible paths**

If:

- the combination size already exceeds `k`, or
- the remaining sum becomes negative

then continuing further is pointless.

We immediately return.
5. **Try all candidate numbers**

We iterate from `start` to `9`.

For each candidate number:

- Add it to the current combination.
- Recursively search for the remaining target.
- Remove it afterward.

This removal step is the essence of backtracking. We undo the choice so the next iteration can explore a different possibility.
6. **Continue until all paths are explored**

Eventually, recursion finishes after trying every valid increasing sequence of numbers from `1` to `9`.

### Why it works

The algorithm works because it systematically explores every possible combination of distinct numbers in increasing order. Since numbers are only chosen once and future choices must always be larger, duplicate combinations are impossible.

Backtracking guarantees completeness because every valid path is explored, while pruning guarantees efficiency by stopping paths that can never produce a valid answer.

## Python Solution

```
from typing import List

class Solution:
    def combinationSum3(self, k: int, n: int) -> List[List[int]]:
        result = []
        current_combination = []

        def backtrack(start: int, remaining_sum: int) -> None:
            if len(current_combination) == k:
                if remaining_sum == 0:
                    result.append(current_combination[:])
                return

            if remaining_sum < 0:
                return

            for number in range(start, 10):
                current_combination.append(number)
                backtrack(number + 1, remaining_sum - number)
                current_combination.pop()

        backtrack(1, n)
        return result
```

The implementation follows the algorithm directly.

We first initialize `result`, which stores all valid combinations, and `current_combination`, which tracks the numbers chosen during the current recursive path.

The nested `backtrack()` function performs the recursive exploration. It accepts the current starting number and the remaining target sum.

The first base case checks whether we have selected exactly `k` numbers. If so, we only add the combination when the remaining sum is exactly zero. Otherwise, the recursion stops immediately because adding more numbers would violate the problem requirement.

The second pruning condition checks whether the remaining sum is negative. Once this happens, the path is impossible to recover because all future numbers are positive.

The loop iterates through candidate numbers from `start` to `9`. By always moving forward with `number + 1`, each number is used at most once and duplicate orderings are avoided.

The append and pop operations implement backtracking. We temporarily make a choice, explore the consequences, and then undo it before trying the next option.

## Go Solution

```
func combinationSum3(k int, n int) [][]int {
	var result [][]int
	var currentCombination []int

	var backtrack func(start int, remainingSum int)

	backtrack = func(start int, remainingSum int) {
		if len(currentCombination) == k {
			if remainingSum == 0 {
				combination := make([]int, len(currentCombination))
				copy(combination, currentCombination)
				result = append(result, combination)
			}
			return
		}

		if remainingSum < 0 {
			return
		}

		for number := start; number <= 9; number++ {
			currentCombination = append(currentCombination, number)

			backtrack(number+1, remainingSum-number)

			currentCombination = currentCombination[:len(currentCombination)-1]
		}
	}

	backtrack(1, n)

	return result
}
```

The Go implementation mirrors the Python version closely.

One important difference is slice handling. In Python, `current_combination[:]` creates a copy automatically. In Go, slices share underlying memory, so we must explicitly allocate a new slice and copy values before appending to `result`. Without this step, later modifications would overwrite previously stored combinations.

Another difference is recursion syntax. Go requires explicitly declaring the recursive function variable before assigning it.

There are no integer overflow concerns because the maximum sum is very small.

## Edge Cases

### Target Sum Too Small

A subtle edge case occurs when the target sum is smaller than the smallest possible sum for `k` distinct numbers.

For example, `k = 4, n = 1` cannot work because the smallest four distinct positive integers are `1 + 2 + 3 + 4 = 10`. A naive implementation might still waste time exploring impossible paths. Our implementation naturally handles this through pruning, once the remaining sum becomes negative, recursion stops immediately.

### Target Sum Too Large

Another edge case occurs when the target exceeds the largest possible sum achievable with `k` distinct numbers.

For example, `k = 3, n = 50` is impossible because even `7 + 8 + 9 = 24` is far below the target. The algorithm correctly returns an empty list after exhausting valid search paths.

### Duplicate Combination Prevention

Avoiding duplicates is a common source of bugs in combination problems. Without careful handling, `[1,2,6]` and `[2,1,6]` might both appear.

This implementation prevents duplicates by enforcing increasing order. After selecting a number, future recursive calls only consider larger numbers using `number + 1` as the next starting point. As a result, every unique combination is generated exactly once.

### Exact Size Requirement

Another important detail is ensuring combinations contain **exactly `k` numbers**. A naive implementation might accidentally include shorter combinations whose sum equals `n`.

For example, if `k = 3` and `n = 6`, `[1,5]` should not be accepted. Our implementation explicitly checks `len(current_combination) == k` before validating the sum, guaranteeing correctness.
