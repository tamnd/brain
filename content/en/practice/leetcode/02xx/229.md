## Solution

## Problem Understanding

The problem asks us to find every integer in an array that appears more than `⌊ n/3 ⌋` times, where `n` is the length of the array. The floor operation means we round down to the nearest integer. For example, if the array length is `8`, then `⌊ 8/3 ⌋ = 2`, so any number appearing more than `2` times must be included in the result.

The input is a list of integers, `nums`, and the output is a list containing all values whose frequency exceeds the threshold `n/3`. The order of the returned elements does not matter.

An important observation is that there can be at most two such elements. This comes directly from the frequency requirement. If three different numbers each appeared more than `n/3` times, then together they would occupy more than `n` positions, which is impossible.

The constraints tell us that the array can contain up to `5 * 10^4` elements, which is large enough that inefficient quadratic solutions may become too slow. The follow up explicitly asks for a linear time solution using constant extra space, which strongly hints that a frequency map alone is not the intended optimal solution.

Several edge cases are important to consider. Arrays of length `1` or `2` are special because every distinct element may qualify. Arrays with no repeated values still produce valid results because the threshold can be zero for very small arrays. Another important case is when there are exactly two majority elements, since the algorithm must correctly track both simultaneously.

## Approaches

### Brute Force Approach

A straightforward solution is to count how many times every number appears in the array. One way to do this is to iterate through the array and, for each element, scan the entire array again to compute its frequency.

If a number appears more than `n/3` times, we add it to the answer list if it is not already present.

This approach is correct because it explicitly computes the exact frequency of every value. However, it is inefficient because each frequency computation requires another full traversal of the array. With `n` elements, this results in `O(n^2)` time complexity.

A slightly better brute force variation uses a hash map to count frequencies in one pass, then scans the map to find qualifying elements. That improves time complexity to `O(n)`, but it requires `O(n)` extra space, which does not satisfy the follow up requirement.

### Optimal Approach

The optimal solution uses the Boyer-Moore Voting Algorithm extension for the `n/3` case.

The key insight is that there can be at most two elements appearing more than `n/3` times. That means we only need to keep track of two candidate values and their counts.

The algorithm works by canceling out different elements against each other. Whenever we encounter a new value that matches one of our candidates, we increment its count. If it does not match either candidate and both counters are nonzero, we decrement both counters. Conceptually, this removes one occurrence of three distinct values from consideration.

Any element occurring more than `n/3` times cannot be completely canceled out, so it must remain as one of the final candidates.

After the first pass, the candidates are only potential answers. We must perform a second pass to verify their actual frequencies.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Repeatedly scans the array to count frequencies |
| Optimal | O(n) | O(1) | Extended Boyer-Moore Voting Algorithm |

## Algorithm Walkthrough

1. Initialize two candidate variables and two counters.

Since there can be at most two majority elements, we maintain two possible candidates, `candidate1` and `candidate2`, along with their counts, `count1` and `count2`.
2. Traverse the array once to identify potential candidates.

For each number:

- If the number matches `candidate1`, increment `count1`.
- Else if the number matches `candidate2`, increment `count2`.
- Else if `count1` is zero, assign the current number to `candidate1` and set `count1 = 1`.
- Else if `count2` is zero, assign the current number to `candidate2` and set `count2 = 1`.
- Otherwise, decrement both `count1` and `count2`.

The decrement step is the core of the algorithm. It effectively removes one occurrence of three distinct elements from consideration.
3. Verify the candidates.

After the first pass, the candidates are not guaranteed to be valid majority elements. They are only the surviving possibilities.

Count how many times `candidate1` and `candidate2` actually appear in the array.
4. Build the result.

If either candidate appears more than `n // 3` times, include it in the answer list.

### Why it works

The algorithm relies on the cancellation principle. Every time we decrement both counters, we are effectively removing a group of three distinct elements. Since a true majority element appears more than `n/3` times, it cannot be entirely eliminated through these cancellations. Therefore, any valid majority element must survive as one of the final candidates.

## Python Solution

```
from typing import List

class Solution:
    def majorityElement(self, nums: List[int]) -> List[int]:
        candidate1 = None
        candidate2 = None
        count1 = 0
        count2 = 0

        # First pass: find potential candidates
        for num in nums:
            if candidate1 == num:
                count1 += 1
            elif candidate2 == num:
                count2 += 1
            elif count1 == 0:
                candidate1 = num
                count1 = 1
            elif count2 == 0:
                candidate2 = num
                count2 = 1
            else:
                count1 -= 1
                count2 -= 1

        # Second pass: verify actual counts
        count1 = 0
        count2 = 0

        for num in nums:
            if num == candidate1:
                count1 += 1
            elif num == candidate2:
                count2 += 1

        result = []
        threshold = len(nums) // 3

        if count1 > threshold:
            result.append(candidate1)

        if count2 > threshold:
            result.append(candidate2)

        return result
```

The implementation follows the Boyer-Moore extension exactly.

The first section initializes two candidate slots and their counters. Initially, both counts are zero because no candidates have been selected yet.

The first loop performs the elimination process. Matching candidates increase their counts, empty candidate slots are filled with new values, and conflicting values decrement both counters. This preserves only the most dominant elements.

The second loop resets the counters and computes the true frequencies of the surviving candidates. This verification step is necessary because surviving candidates are not automatically guaranteed to exceed the threshold.

Finally, the code compares each verified frequency against `len(nums) // 3` and returns all valid majority elements.

## Go Solution

```
func majorityElement(nums []int) []int {
    var candidate1, candidate2 int
    count1, count2 := 0, 0

    // First pass: find potential candidates
    for _, num := range nums {
        if candidate1 == num {
            count1++
        } else if candidate2 == num {
            count2++
        } else if count1 == 0 {
            candidate1 = num
            count1 = 1
        } else if count2 == 0 {
            candidate2 = num
            count2 = 1
        } else {
            count1--
            count2--
        }
    }

    // Second pass: verify actual counts
    count1, count2 = 0, 0

    for _, num := range nums {
        if num == candidate1 {
            count1++
        } else if num == candidate2 {
            count2++
        }
    }

    result := []int{}
    threshold := len(nums) / 3

    if count1 > threshold {
        result = append(result, candidate1)
    }

    if count2 > threshold {
        result = append(result, candidate2)
    }

    return result
}
```

The Go implementation closely mirrors the Python version. Since Go does not support `None`, the candidate variables are initialized to zero values, but this is safe because the counters determine whether a candidate is currently active.

Slices are used for the result collection, and integer division naturally performs floor division in Go, which matches the problem requirement.

## Edge Cases

One important edge case is an array with only one element. Since `⌊1/3⌋ = 0`, the single value automatically qualifies. Algorithms that incorrectly assume at least two or three elements may fail here. The implementation handles this naturally because the candidate selection and verification logic still works correctly for a single iteration.

Another important case is when there are exactly two valid majority elements. For example, `[1,1,1,2,2,2,3]` contains both `1` and `2` more than `n/3` times. A naive adaptation of the standard Boyer-Moore algorithm for majority element greater than `n/2` would fail because that version only tracks one candidate. This implementation explicitly maintains two candidate slots, ensuring both are preserved.

A third edge case occurs when no element exceeds the threshold. For example, `[1,2,3,4]` contains only unique values. During the elimination phase, candidates may still survive temporarily, but the second verification pass prevents incorrect answers from being returned. Without verification, the algorithm could mistakenly output invalid candidates.

Another subtle case involves repeated cancellations, such as `[1,2,3,1,2,3,1]`. The counters repeatedly increase and decrease as distinct values cancel each other out. Despite this, the true majority element `1` survives because it appears more than `n/3` times. This demonstrates why the cancellation principle is mathematically sound.
