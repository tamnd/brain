## Solution

## Problem Understanding

This problem asks us to implement a **stack** using only **queue operations**. A stack follows the **Last In, First Out (LIFO)** principle, meaning the most recently added element is removed first. In contrast, a queue follows **First In, First Out (FIFO)** ordering, meaning the earliest inserted element is removed first.

The challenge is to simulate stack behavior while only using queue operations such as:

- Push to the back
- Peek or pop from the front
- Check size
- Check if empty

We are required to implement a `MyStack` class with four operations:

- `push(x)` inserts an element onto the top of the stack.
- `pop()` removes and returns the top element.
- `top()` returns the top element without removing it.
- `empty()` checks whether the stack contains any elements.

The input shown in the example represents a sequence of method calls on the `MyStack` object. For example:

```
["MyStack", "push", "push", "top", "pop", "empty"]
[[], [1], [2], [], [], []]
```

means:

1. Create a new stack.
2. Push `1`.
3. Push `2`.
4. Retrieve the top element.
5. Pop the top element.
6. Check if the stack is empty.

The output reflects the return value of each operation. Methods like `push()` return `null`, while `top()`, `pop()`, and `empty()` return values.

The constraints are very small, at most **100 method calls**. This means efficiency is not critically important for correctness, but we should still aim for a clean and optimal design. Since the problem explicitly asks us to use queues, the main challenge is designing queue manipulations that preserve stack ordering.

An important guarantee is that all `pop()` and `top()` calls are valid, meaning we never need to handle popping or peeking from an empty stack. This removes the need for extra defensive checks.

Several edge cases are worth considering upfront. A stack containing only one element can expose bugs in queue transfer logic. Repeated push and pop operations can reveal ordering mistakes. Another subtle case occurs when calling `top()` repeatedly, since it must not remove the element from the stack.

## Approaches

### Brute Force Approach

A straightforward approach uses **two queues**, where `push()` is intentionally expensive.

Whenever we push a new element, we place it into an empty secondary queue first. Then we move every element from the main queue into the secondary queue. This guarantees the newest element always appears at the front of the main queue.

After reordering, we swap the queues.

For example:

```
Push(1): [1]
Push(2): [2,1]
Push(3): [3,2,1]
```

Now, `pop()` simply removes from the front of the queue, which corresponds to the stack top.

This works because we actively maintain stack order after every insertion. However, each `push()` operation requires moving all existing elements, making insertion expensive.

### Optimal Approach

A cleaner and more elegant approach uses **one queue** instead of two.

The key insight is that after inserting a new element into the queue, we can rotate the existing elements behind it.

Suppose the queue currently contains:

```
[1,2]
```

After `push(3)`:

```
[1,2,3]
```

We then rotate the first two elements to the back:

```
[2,3,1]
[3,1,2]
```

Now `3` sits at the front, making queue front equivalent to stack top.

This trick maintains stack behavior while using only queue operations.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Push: O(n), Pop: O(1) | O(n) | Uses two queues and rearranges on every push |
| Optimal | Push: O(n), Pop: O(1) | O(n) | Uses one queue and rotates elements |

## Algorithm Walkthrough

We use a single queue and maintain the invariant that **the front of the queue always represents the top of the stack**.

1. Initialize an empty queue.

We need only one queue to store all elements. The queue will be reordered after every insertion to simulate stack ordering.
2. Push an element.

Insert the new element at the back of the queue using the normal queue operation.

Since the newest stack element should become the top, we rotate all previously existing elements behind it.

For example:

```
Before push: [1,2]
Add 3: [1,2,3]
Rotate twice:
[2,3,1]
[3,1,2]
```

Now the front contains `3`, which correctly represents the stack top.
3. Pop an element.

Since the stack top is always maintained at the front of the queue, simply remove and return the front element.
4. Get the top element.

Return the front element without removing it.
5. Check emptiness.

Return whether the queue contains any elements.

### Why it works

The core invariant is that **the queue front always stores the stack top**.

Whenever we push a new element, we rotate all older elements behind it, ensuring the newest item moves to the front. Because of this invariant, `pop()` and `top()` become direct front operations on the queue and always return the correct LIFO element.

## Python Solution

```
from collections import deque

class MyStack:

    def __init__(self):
        self.queue = deque()

    def push(self, x: int) -> None:
        self.queue.append(x)

        # Rotate previous elements behind the new one
        for _ in range(len(self.queue) - 1):
            self.queue.append(self.queue.popleft())

    def pop(self) -> int:
        return self.queue.popleft()

    def top(self) -> int:
        return self.queue[0]

    def empty(self) -> bool:
        return len(self.queue) == 0

# Your MyStack object will be instantiated and called as such:
# obj = MyStack()
# obj.push(x)
# param_2 = obj.pop()
# param_3 = obj.top()
# param_4 = obj.empty()
```

The implementation uses Python's `deque` from the `collections` module because it supports efficient insertion at the back and removal from the front, exactly matching queue operations.

In `push()`, we first append the new element to the queue. Since the newest stack element must become the top, we rotate every older element to the back. This transformation ensures the newly inserted element becomes the front element.

The `pop()` operation simply removes from the front of the queue because our invariant guarantees that the front always represents the stack top.

The `top()` method returns the front element without removing it, while `empty()` checks whether the queue contains any elements.

## Go Solution

```
type MyStack struct {
	queue []int
}

func Constructor() MyStack {
	return MyStack{
		queue: []int{},
	}
}

func (this *MyStack) Push(x int) {
	this.queue = append(this.queue, x)

	// Rotate existing elements
	n := len(this.queue)
	for i := 0; i < n-1; i++ {
		front := this.queue[0]
		this.queue = this.queue[1:]
		this.queue = append(this.queue, front)
	}
}

func (this *MyStack) Pop() int {
	top := this.queue[0]
	this.queue = this.queue[1:]
	return top
}

func (this *MyStack) Top() int {
	return this.queue[0]
}

func (this *MyStack) Empty() bool {
	return len(this.queue) == 0
}

/**
 * Your MyStack object will be instantiated and called as such:
 * obj := Constructor();
 * obj.Push(x);
 * param_2 := obj.Pop();
 * param_3 := obj.Top();
 * param_4 := obj.Empty();
 */
```

The Go implementation uses a slice to simulate a queue. Appending to the slice corresponds to pushing at the back of the queue.

Since Go slices do not have a built in `popleft()` operation, removing the front element is implemented with slicing:

```
this.queue = this.queue[1:]
```

This effectively removes the first element while keeping queue behavior intact. Since the constraints are tiny, this implementation is completely acceptable.

Unlike Python, Go does not provide a built in deque structure, so slice manipulation is the simplest solution.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | Push: O(n), Pop: O(1), Top: O(1), Empty: O(1) | Push rotates all previous elements |
| Space | O(n) | Queue stores all stack elements |

The expensive operation is `push()`. If there are `n` elements already in the queue, we rotate `n - 1` items after insertion, producing linear time complexity.

The remaining operations are constant time because the queue front always represents the stack top.

Space complexity is `O(n)` because we store every inserted element in the queue.

## Edge Cases

### Single Element Stack

A stack with only one element is a common source of bugs because rotation logic can accidentally remove or misplace the element. In our implementation, the rotation loop executes `len(queue) - 1` times, which becomes zero when only one element exists. This means no unnecessary movement occurs.

### Repeated `top()` Calls

A naive implementation might accidentally remove the top element while retrieving it. Since `top()` must only inspect the value, our implementation simply returns the queue front without modification, ensuring repeated calls behave correctly.

### Alternating Push and Pop Operations

Sequences such as:

```
push(1)
pop()
push(2)
push(3)
pop()
```

can expose state inconsistencies if queue order is not maintained properly. Our implementation preserves the invariant that the queue front always represents the stack top after every `push()`, ensuring all later operations behave correctly.

### Empty Stack Check

A newly initialized stack should correctly report as empty. Similarly, after popping all elements, `empty()` should return `true`. Since we directly check the queue size, this behavior remains correct regardless of prior operations.
