# LeetCode 2764 - Is Array a Preorder of Some Binary Tree

## Problem Understanding

This problem gives us a 2D array `nodes`, where each entry has the form `[id, parentId]`.

The `id` uniquely identifies a node in a binary tree, and `parentId` identifies that node's parent. A value of `-1` indicates the root node because the root has no parent.

The important detail is that the array itself represents an ordering of the nodes. We must determine whether this ordering could be the preorder traversal of the binary tree defined by the parent relationships.

Recall that preorder traversal visits nodes in the following order:

1. Visit the current node.
2. Traverse the left subtree.
3. Traverse the right subtree.

The challenge is that the input does not explicitly tell us which child is left and which child is right. Since the tree is binary, each parent can have at most two children. We are free to assign one child as the left child and the other as the right child if doing so makes the traversal valid.

We must return `true` if there exists some valid left/right assignment that makes the given node ordering a preorder traversal of the tree. Otherwise, we return `false`.

The constraints are significant:

- Up to `10^5` nodes.
- The input is guaranteed to form a valid binary tree.
- We need an algorithm that runs in approximately linear time.

A naive simulation that repeatedly explores subtrees would be too slow for this input size.

Several edge cases are important:

- A tree containing only a single node is always a valid preorder traversal.
- A parent may have only one child. That child can be considered either the left child or the right child.
- A node's entire subtree must appear as one contiguous block in preorder traversal. If nodes from another subtree appear in the middle, the ordering is invalid.
- The root must appear first in any preorder traversal.

## Approaches

### Brute Force

A brute force approach would explicitly construct the tree and try every possible assignment of left and right children whenever a node has two children.

For each assignment, we could generate the preorder traversal and compare it with the input order.

This approach is correct because it explores all possible binary-tree configurations consistent with the parent relationships. If any configuration produces the given traversal, the answer is `true`.

However, this becomes infeasible because every node with two children introduces two possible assignments. In the worst case, the number of configurations grows exponentially.

### Key Insight

In a preorder traversal, every subtree occupies one contiguous segment of the traversal order.

Suppose we process nodes from left to right in the given order. When we encounter a node, we are entering its subtree. Eventually, after all descendants have been processed, we leave that subtree.

A stack naturally models this process.

For the current node:

- If its parent is the node on top of the stack, then it is the next child being visited.
- Otherwise, we must have already completed one or more subtrees, so we pop nodes from the stack until we find its parent.
- If the parent is never found, then the ordering cannot correspond to a preorder traversal.

We also need to ensure that no parent receives more than two children, although the problem guarantees the structure is a binary tree.

This leads to a linear-time solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^k · n) | O(n) | Tries all possible left/right assignments |
| Optimal | O(n) | O(n) | Uses a stack to validate preorder structure |

Here, `k` is the number of nodes with two children.

## Algorithm Walkthrough

1. Create a stack and place the root node, which is the first node in the array, onto it.
2. Maintain a count of how many children each node has received. This helps ensure no parent exceeds the binary tree limit.
3. Process the remaining nodes in preorder order.
4. For each node, extract its `parentId`.
5. While the stack is not empty and the node at the top of the stack is not the parent, pop from the stack. This represents finishing entire subtrees whose traversal has already been completed.
6. After popping, check whether the stack is empty. If it is, the required parent cannot be found, so return `false`.
7. The parent is now at the top of the stack. Increment that parent's child count.
8. If the parent now has more than two children, return `false`.
9. Push the current node onto the stack because its subtree may contain future descendants.
10. Continue until all nodes have been processed.
11. If every node is processed successfully, return `true`.

### Why it works

The stack always contains the current path from the root to the most recently visited node whose subtree has not yet been completely processed.

In a valid preorder traversal, whenever a new node appears, its parent must be somewhere on this active path. If the parent is not present, then the traversal would require re-entering a subtree that has already been completed, which is impossible in preorder traversal.

Therefore, finding every parent on the stack is both necessary and sufficient for validating the preorder ordering.

## Python Solution

```python
from typing import List
from collections import defaultdict

class Solution:
    def isPreorder(self, nodes: List[List[int]]) -> bool:
        stack = [nodes[0][0]]
        child_count = defaultdict(int)

        for i in range(1, len(nodes)):
            node_id, parent_id = nodes[i]

            while stack and stack[-1] != parent_id:
                stack.pop()

            if not stack:
                return False

            child_count[parent_id] += 1
            if child_count[parent_id] > 2:
                return False

            stack.append(node_id)

        return True
```

The algorithm begins by placing the root on the stack. The stack represents the chain of ancestors whose subtrees are still active.

For each subsequent node, we repeatedly pop completed subtrees until we either find the parent or exhaust the stack.

If the parent cannot be found, the preorder property has been violated and we immediately return `False`.

When the parent is found, we record that another child has been attached to it and verify that the binary tree limit of two children is not exceeded.

Finally, we push the current node onto the stack because future nodes may belong to its subtree.

Since every node is pushed once and popped at most once, the implementation runs in linear time.

## Go Solution

```go
package main

func isPreorder(nodes [][]int) bool {
	stack := []int{nodes[0][0]}
	childCount := make(map[int]int)

	for i := 1; i < len(nodes); i++ {
		nodeID := nodes[i][0]
		parentID := nodes[i][1]

		for len(stack) > 0 && stack[len(stack)-1] != parentID {
			stack = stack[:len(stack)-1]
		}

		if len(stack) == 0 {
			return false
		}

		childCount[parentID]++
		if childCount[parentID] > 2 {
			return false
		}

		stack = append(stack, nodeID)
	}

	return true
}
```

The Go implementation follows exactly the same logic as the Python version.

Slices are used to implement the stack efficiently. Appending pushes a node, and slicing off the last element performs a pop operation.

No special handling for integer overflow is necessary because node IDs are at most `10^5`, well within Go's integer range.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every node is pushed once and popped at most once |
| Space | O(n) | Stack and child-count map may store all nodes |

The key observation is that each node enters the stack exactly once and leaves the stack at most once. Therefore, even though the algorithm contains a nested while loop, the total number of pop operations across the entire execution is at most `n`. This gives an overall linear running time.

## Edge Cases

### Single Node Tree

A tree containing only the root is always a valid preorder traversal. There are no children and therefore no subtree ordering constraints. The implementation handles this naturally because the loop over remaining nodes never executes and the function returns `True`.

### Deep Chain of Nodes

Consider a tree where every node has exactly one child. The preorder traversal is simply the sequence from root to leaf. A recursive validation strategy could risk deep recursion, but the stack-based approach handles this iteratively and efficiently. Every node is pushed once and never causes excessive overhead.

### Subtree Interruption

A common source of bugs occurs when a subtree begins, another subtree starts before the first one finishes, and then the traversal attempts to return to the first subtree. Preorder traversal never allows this because each subtree must occupy a contiguous block. The stack invariant detects exactly this situation. If the required parent is no longer on the active ancestor path, the stack becomes empty and the algorithm correctly returns `False`.

### Returning to an Ancestor

Another subtle case occurs when traversal finishes several nested subtrees and then moves to a child of a higher ancestor. The repeated popping process models precisely this behavior. The algorithm keeps removing completed nodes until it reaches the correct ancestor, ensuring that valid preorder traversals are accepted.
