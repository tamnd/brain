## Solution

## Problem Understanding

The problem gives us an array of strings called `words`. We begin with the first string, `words[0]`, and then process the remaining strings one by one. For every next word, we have two choices:

1. Append the new word to the current string.
2. Prepend the new word to the current string.

The special rule is the `join(x, y)` operation. Normally, concatenating two strings would produce `xy`. However, if the last character of `x` is the same as the first character of `y`, one duplicate character is removed during concatenation.

For example:

- `join("ab", "bc") = "abc"` because both sides contain `b` at the boundary.
- `join("ab", "cd") = "abcd"` because there is no overlap.

Our goal is to choose the order of concatenation operations so that the final resulting string has the smallest possible length.

The important observation is that the exact middle portion of the string does not matter for future operations. Only the first and last characters matter, because overlap can occur only at the boundary between two joined strings. This is the key insight that enables an efficient dynamic programming solution.

The constraints are significant:

- `words.length <= 1000`
- `words[i].length <= 50`

A brute-force search would examine every possible sequence of prepend/append decisions. Since each word introduces two choices, there are `2^(n-1)` possibilities. That becomes completely infeasible when `n = 1000`.

Another important detail is that all strings contain only lowercase English letters. That means there are only 26 possible starting characters and 26 possible ending characters. This small alphabet size heavily limits the number of meaningful DP states.

Several edge cases are important:

- A single word input, where the answer is simply its length.
- Situations where every concatenation creates an overlap.
- Situations where no overlaps are possible.
- Cases where a locally optimal choice is not globally optimal. Sometimes sacrificing an overlap now enables more overlaps later.

## Approaches

### Brute Force

The brute-force solution recursively explores every possible way to combine the words.

At each step, for the current word `words[i]`, we have two options:

- Append it to the right side of the current string.
- Prepend it to the left side of the current string.

We would explicitly build the resulting string after every operation and continue recursively until all words are processed. The minimum final length among all possibilities would be the answer.

This approach is correct because it exhaustively checks every legal sequence of operations. However, it is far too slow.

For `n` words, there are `2^(n-1)` possible operation sequences. Even for moderate values of `n`, this becomes enormous. Additionally, constructing strings repeatedly introduces extra overhead.

### Key Insight

The important realization is that future overlaps depend only on:

- The first character of the current combined string.
- The last character of the current combined string.

The internal characters never affect future decisions.

This means we can define a dynamic programming state using:

- The current index `i`
- The first character of the current string
- The last character of the current string

Instead of storing the entire concatenated string, we store only the minimum achievable length for each boundary-character configuration.

Since there are only 26 lowercase letters, the total number of `(first_char, last_char)` pairs is at most `26 * 26 = 676`.

This dramatically reduces the state space and makes the problem tractable.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n * n * L) | O(n * L) | Explores every prepend/append combination |
| Optimal Dynamic Programming | O(n * 26 * 26) | O(26 * 26) | Tracks only boundary characters |

Here, `L` represents the average word length.

## Algorithm Walkthrough

### Step 1: Define the DP State

We maintain a DP table where:

`dp[first][last]`

stores the minimum length of a concatenated string whose:

- first character is `first`
- last character is `last`

after processing some prefix of the words array.

Initially, we process only `words[0]`.

If `words[0] = "abc"`:

- first character = `a`
- last character = `c`
- length = `3`

So:

```
dp['a']['c'] = 3
```

### Step 2: Process Each New Word

For every next word:

- Let:

- `new_first = word[0]`
- `new_last = word[-1]`
- `word_len = len(word)`

We create a new DP table for the updated states.

For every existing state `(first, last)`:

#### Option A: Append the New Word

We compute:

```
current + word
```

The resulting:

- first character remains `first`
- last character becomes `new_last`

The added length is normally `word_len`.

However, if:

```
last == new_first
```

then one character overlaps, so we add:

```
word_len - 1
```

instead.

#### Option B: Prepend the New Word

We compute:

```
word + current
```

The resulting:

- first character becomes `new_first`
- last character remains `last`

Again, if:

```
new_last == first
```

then one overlapping character is removed.

### Step 3: Keep the Minimum Length

Multiple paths may lead to the same `(first, last)` pair.

We always keep the minimum length among them.

### Step 4: Return the Minimum Value

After processing all words, the answer is the smallest value across all DP states.

## Why it works

The algorithm works because the only information relevant for future concatenations is the current string's first and last characters. Any two strings with the same boundary characters behave identically for all future operations. Therefore, keeping only the minimum length for each boundary-character pair preserves all optimal possibilities.

Dynamic programming guarantees correctness because every valid sequence of operations is represented through state transitions, and every state stores the best possible result achievable for those boundaries.

## Python Solution

```python
from typing import List
from math import inf

class Solution:
    def minimizeConcatenatedLength(self, words: List[str]) -> int:
        dp = [[inf] * 26 for _ in range(26)]

        first = ord(words[0][0]) - ord('a')
        last = ord(words[0][-1]) - ord('a')

        dp[first][last] = len(words[0])

        for word in words[1:]:
            new_dp = [[inf] * 26 for _ in range(26)]

            word_first = ord(word[0]) - ord('a')
            word_last = ord(word[-1]) - ord('a')
            word_len = len(word)

            for start in range(26):
                for end in range(26):
                    if dp[start][end] == inf:
                        continue

                    current_len = dp[start][end]

                    # Append word to the right
                    add_len = word_len
                    if end == word_first:
                        add_len -= 1

                    new_dp[start][word_last] = min(
                        new_dp[start][word_last],
                        current_len + add_len
                    )

                    # Prepend word to the left
                    add_len = word_len
                    if word_last == start:
                        add_len -= 1

                    new_dp[word_first][end] = min(
                        new_dp[word_first][end],
                        current_len + add_len
                    )

            dp = new_dp

        answer = inf

        for i in range(26):
            for j in range(26):
                answer = min(answer, dp[i][j])

        return answer
```

The implementation uses a `26 x 26` DP table because there are only 26 lowercase letters.

Each cell stores the minimum possible concatenated length for a specific starting and ending character combination.

We initialize the DP table using the first word. Then, for every remaining word, we build a fresh DP table called `new_dp`.

For each existing state, we simulate both valid operations:

- Appending the new word to the right
- Prepending the new word to the left

The overlap rule is handled by checking whether the touching boundary characters are equal. If they are, we reduce the added length by one.

At the end of processing, the minimum value in the DP table is the answer.

## Go Solution

```go
package main

import "math"

func minimizeConcatenatedLength(words []string) int {
	const INF = math.MaxInt32

	dp := make([][]int, 26)
	for i := 0; i < 26; i++ {
		dp[i] = make([]int, 26)
		for j := 0; j < 26; j++ {
			dp[i][j] = INF
		}
	}

	first := int(words[0][0] - 'a')
	last := int(words[0][len(words[0])-1] - 'a')

	dp[first][last] = len(words[0])

	for i := 1; i < len(words); i++ {
		word := words[i]

		newDP := make([][]int, 26)
		for x := 0; x < 26; x++ {
			newDP[x] = make([]int, 26)
			for y := 0; y < 26; y++ {
				newDP[x][y] = INF
			}
		}

		wordFirst := int(word[0] - 'a')
		wordLast := int(word[len(word)-1] - 'a')
		wordLen := len(word)

		for start := 0; start < 26; start++ {
			for end := 0; end < 26; end++ {
				if dp[start][end] == INF {
					continue
				}

				currentLen := dp[start][end]

				// Append word to the right
				addLen := wordLen
				if end == wordFirst {
					addLen--
				}

				if currentLen+addLen < newDP[start][wordLast] {
					newDP[start][wordLast] = currentLen + addLen
				}

				// Prepend word to the left
				addLen = wordLen
				if wordLast == start {
					addLen--
				}

				if currentLen+addLen < newDP[wordFirst][end] {
					newDP[wordFirst][end] = currentLen + addLen
				}
			}
		}

		dp = newDP
	}

	answer := INF

	for i := 0; i < 26; i++ {
		for j := 0; j < 26; j++ {
			if dp[i][j] < answer {
				answer = dp[i][j]
			}
		}
	}

	return answer
}
```

The Go implementation mirrors the Python logic closely. Since Go does not provide a built-in infinity value for integers, we use `math.MaxInt32` as a large sentinel value.

Two-dimensional slices are used for the DP tables. Every state is initialized to `INF`, and transitions update the minimum achievable length.

Because the maximum possible final length is at most `1000 * 50 = 50000`, integer overflow is never a concern with Go's standard `int` type.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n * 26 * 26) | Each word processes all boundary-character states |
| Space | O(26 * 26) | DP table stores only character-pair states |

The DP table contains at most `26 * 26 = 676` states. For every word, we iterate through all possible states and perform constant-time transitions.

This makes the solution extremely efficient even for the maximum constraint of 1000 words.

The space complexity remains constant with respect to `n` because the DP table size never grows beyond the fixed alphabet size.

## Edge Cases

One important edge case is when the input contains only a single word. Since no join operations are performed, the answer is simply the length of that word. A buggy implementation might incorrectly initialize the DP table or assume at least one transition occurs. This solution handles the case naturally during initialization.

Another important edge case occurs when every concatenation overlaps. For example:

```
["a", "a", "a", "a"]
```

Every join removes one duplicated boundary character, so the final length remains `1`. Incorrect implementations may accidentally double-count overlaps or fail to apply the overlap rule consistently. The DP transition logic explicitly subtracts one whenever adjacent boundary characters match.

A third tricky case is when greedy decisions fail. Consider:

```
["ab", "ba", "ac"]
```

A locally optimal overlap may prevent a larger future overlap. Greedy algorithms that always maximize the immediate overlap can therefore produce suboptimal answers. The dynamic programming solution avoids this issue by preserving all optimal boundary-character states simultaneously.

Another subtle edge case involves different strings producing identical boundary characters. For example, multiple concatenation paths may all lead to strings starting with `a` and ending with `b`, but with different lengths. The implementation correctly keeps only the minimum length for each `(start, end)` pair, ensuring redundant worse states are discarded.
