## Solution

## Problem Understanding

The problem asks us to implement a throttled batching system for asynchronous queries. We are given a class named `QueryBatcher` that receives two constructor arguments:

- `queryMultiple`, an asynchronous function that accepts an array of string keys and returns an array of corresponding string values.
- `t`, a throttle interval in milliseconds.

The class exposes a single method:

```
async getValue(key)
```

This method accepts one key at a time, but internally the implementation should intelligently batch multiple requests together in order to reduce the number of calls to `queryMultiple`.

The important detail is the throttling rule. The very first call to `getValue` should immediately trigger a request using `queryMultiple([key])`. After that, another invocation of `queryMultiple` cannot happen until at least `t` milliseconds have passed since the previous invocation started.

Any requests arriving during the cooldown window must be queued. Once the throttle period expires, all queued keys should be sent together in a single batched call.

Each `getValue(key)` must still return a promise that resolves to the correct single value associated with that key.

The input examples simulate calls happening at different timestamps. The output records when each promise resolves. The returned timing depends both on the throttle window and on how long `queryMultiple` itself takes to resolve.

The constraints are intentionally small:

- `0 <= t <= 1000`
- At most 10 calls
- Keys are unique

Even though the input size is tiny, this problem is testing asynchronous coordination and system design logic rather than raw algorithmic complexity.

Several edge cases are important:

- `t = 0`, meaning batching windows disappear and every request can execute immediately.
- Long-running `queryMultiple` calls, where another batch may start while a previous one is still running.
- Requests arriving exactly when the throttle window expires.
- Multiple queued requests that must preserve ordering between keys and returned values.
- Ensuring every promise resolves exactly once.

A naive implementation can easily lose requests, resolve promises incorrectly, or accidentally violate the throttle timing requirement.

## Approaches

### Brute Force Approach

The simplest possible implementation is to call:

```
await queryMultiple([key])
```

every time `getValue` is invoked.

This guarantees correctness because each request immediately maps to a single query result. Every caller receives the right value.

However, this completely ignores the batching requirement. Consecutive calls may happen within the throttle interval `t`, violating the problem constraints.

For example, if requests arrive at times 10ms, 20ms, and 30ms with `t = 100`, the brute-force implementation would trigger three independent queries within 20 milliseconds, which is forbidden.

Even if we added delays manually, we still would not batch requests together efficiently.

### Optimal Approach

The key observation is that requests naturally fall into time windows.

Whenever a query is executed, we start a cooldown period of length `t`. During this period:

- New requests cannot trigger another immediate query.
- Instead, they are queued.

Once the cooldown expires, all queued requests are flushed together in a single batched call.

This means we need:

- A queue to store pending requests.
- A timer that schedules the next batch execution.
- A mechanism to resolve each individual promise after the batched query completes.

Each queued request stores:

- The key.
- The promise resolver function.

When the batch executes:

1. Extract all queued keys.
2. Call `queryMultiple(keys)`.
3. Resolve each stored promise using the corresponding returned value.

This design guarantees:

- No two calls to `queryMultiple` occur within `t` milliseconds.
- Requests are grouped efficiently.
- Each caller receives the correct value.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) queries | O(1) | Violates throttling requirement |
| Optimal | O(n) total | O(n) | Uses batching queue and scheduled flushes |

## Algorithm Walkthrough

1. Store the constructor parameters.

The class keeps:

- The provided `queryMultiple` function.
- The throttle duration `t`.
- A queue of pending requests.
- A flag or timer indicating whether a future batch execution is already scheduled.
2. Handle the first request immediately.

When `getValue(key)` is called:

- If no throttle cooldown is active, immediately execute:

```
queryMultiple([key])
```

- Resolve the caller's promise with the returned value.
- Start a timer for `t` milliseconds.

This matches the requirement that the first request executes immediately.
3. Queue requests during the throttle window.

If another request arrives before the timer expires:

- Store its key.
- Store the resolver function associated with its promise.
- Do not execute `queryMultiple` immediately.
4. Flush queued requests when the timer expires.

Once the cooldown timer finishes:

- Check whether queued requests exist.
- If the queue is non-empty:

- Extract all queued keys.
- Execute one batched call:

```
queryMultiple(allKeys)
```

- Resolve each stored promise with the corresponding result.
- Start another cooldown timer.
5. Continue recursively.

If more requests arrive during the new cooldown window, they are queued again and processed in the next batch.
6. Stop scheduling when the queue becomes empty.

If the timer fires and no queued requests exist:

- No new timer is created.
- The system returns to the idle state.

### Why it works

The core invariant is:

> At most one batch execution can begin during any throttle interval of length `t`.

Every immediate execution starts a cooldown timer. During this cooldown, requests are only queued, never executed. Once the cooldown ends, all accumulated requests are processed together.

Because each queued request stores its own resolver and results preserve positional correspondence, every caller receives the correct value exactly once.

## Python Solution

```python
from typing import List, Callable, Awaitable
import asyncio

class QueryBatcher:

    def __init__(
        self,
        queryMultiple: Callable[[List[str]], Awaitable[List[str]]],
        t: int
    ):
        self.queryMultiple = queryMultiple
        self.t = t / 1000  # convert ms to seconds

        self.pending = []
        self.cooldown_active = False

    async def getValue(self, key: str) -> str:
        loop = asyncio.get_running_loop()
        future = loop.create_future()

        if not self.cooldown_active:
            self.cooldown_active = True

            asyncio.create_task(
                self._execute_immediate(key, future)
            )
        else:
            self.pending.append((key, future))

        return await future

    async def _execute_immediate(self, key: str, future):
        result = await self.queryMultiple([key])
        future.set_result(result[0])

        asyncio.create_task(self._cooldown_loop())

    async def _cooldown_loop(self):
        while True:
            await asyncio.sleep(self.t)

            if not self.pending:
                self.cooldown_active = False
                return

            current_batch = self.pending
            self.pending = []

            keys = [key for key, _ in current_batch]

            results = await self.queryMultiple(keys)

            for (_, future), value in zip(current_batch, results):
                future.set_result(value)
```

After the code block, the implementation can be divided into three logical sections.

The constructor initializes shared state. The `pending` array stores queued requests that arrive during the throttle window. The `cooldown_active` flag determines whether new requests should execute immediately or be queued.

The `getValue` method creates a future for every incoming request. If no cooldown is active, the request executes immediately and starts the throttling cycle. Otherwise, the request is appended to the queue.

The `_cooldown_loop` method enforces the throttling behavior. It sleeps for exactly `t` milliseconds, then flushes every queued request in a single batched call. If no queued requests remain, the loop terminates and the system becomes idle again.

The implementation carefully preserves ordering between input keys and output values using positional matching with `zip`.

## Go Solution

```go
package main

import (
	"time"
)

type pendingRequest struct {
	key      string
	response chan string
}

type QueryBatcher struct {
	queryMultiple func([]string) <-chan []string
	t             time.Duration

	pending        []pendingRequest
	cooldownActive bool
	mutex          chan struct{}
}

func Constructor(
	queryMultiple func([]string) <-chan []string,
	t int,
) QueryBatcher {
	qb := QueryBatcher{
		queryMultiple: queryMultiple,
		t:             time.Duration(t) * time.Millisecond,
		mutex:         make(chan struct{}, 1),
	}

	qb.mutex <- struct{}{}

	return qb
}

func (qb *QueryBatcher) GetValue(key string) <-chan string {
	resultChan := make(chan string, 1)

	<-qb.mutex

	if !qb.cooldownActive {
		qb.cooldownActive = true
		qb.mutex <- struct{}{}

		go qb.executeImmediate(key, resultChan)
	} else {
		qb.pending = append(qb.pending, pendingRequest{
			key:      key,
			response: resultChan,
		})

		qb.mutex <- struct{}{}
	}

	return resultChan
}

func (qb *QueryBatcher) executeImmediate(
	key string,
	resultChan chan string,
) {
	results := <-qb.queryMultiple([]string{key})
	resultChan <- results[0]

	go qb.cooldownLoop()
}

func (qb *QueryBatcher) cooldownLoop() {
	for {
		time.Sleep(qb.t)

		<-qb.mutex

		if len(qb.pending) == 0 {
			qb.cooldownActive = false
			qb.mutex <- struct{}{}
			return
		}

		currentBatch := qb.pending
		qb.pending = nil

		qb.mutex <- struct{}{}

		keys := make([]string, len(currentBatch))

		for i, req := range currentBatch {
			keys[i] = req.key
		}

		results := <-qb.queryMultiple(keys)

		for i, req := range currentBatch {
			req.response <- results[i]
		}
	}
}
```

The Go implementation follows the same logical structure as the Python version, but concurrency primitives differ significantly.

Instead of Python futures and coroutines, Go uses channels and goroutines. Each request receives a dedicated response channel that eventually delivers the result.

A lightweight mutex mechanism is implemented using a buffered channel because multiple goroutines may access shared state concurrently.

Go slices are used for batching queued requests. Unlike Python lists, slices may be reset to `nil` after flushing the batch.

No integer overflow concerns exist here because the constraints are extremely small.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each request enters and leaves the queue once |
| Space | O(n) | Pending requests may accumulate in the queue |

The algorithm is linear in the number of requests because every request is processed exactly once. Queue operations are constant time, and batching only groups requests together without duplicating work.

The space complexity comes from storing unresolved queued requests and their associated promise resolvers.

import asyncio

async def run_tests():
    # Test 1: immediate batching behavior
    async def q1(keys):
        return [k + "!" for k in keys]

    batcher = QueryBatcher(q1, 100)

    r1 = await batcher.getValue("a")
    assert r1 == "a!"  # single immediate request

    # Test 2: multiple queued requests
    async def q2(keys):
        return [k.upper() for k in keys]

    batcher = QueryBatcher(q2, 50)

    task1 = asyncio.create_task(batcher.getValue("a"))
    task2 = asyncio.create_task(batcher.getValue("b"))
    task3 = asyncio.create_task(batcher.getValue("c"))

    results = await asyncio.gather(task1, task2, task3)

    assert results == ["A", "B", "C"]  # batched responses

    # Test 3: t = 0
    async def q3(keys):
        return keys

    batcher = QueryBatcher(q3, 0)

    result = await batcher.getValue("x")
    assert result == "x"  # no throttle delay

    # Test 4: delayed queryMultiple
    async def q4(keys):
        await asyncio.sleep(0.1)
        return [k + "_done" for k in keys]

    batcher = QueryBatcher(q4, 100)

    task1 = asyncio.create_task(batcher.getValue("a"))
    await asyncio.sleep(0.02)
    task2 = asyncio.create_task(batcher.getValue("b"))

    results = await asyncio.gather(task1, task2)

    assert results == ["a_done", "b_done"]  # delayed async resolution

    # Test 5: many sequential batches
    async def q5(keys):
        return [k * 2 for k in keys]

    batcher = QueryBatcher(q5, 50)

    r1 = await batcher.getValue("a")
    await asyncio.sleep(0.06)

    r2 = await batcher.getValue("b")

    assert r1 == "aa"  # first batch
    assert r2 == "bb"  # second independent batch

asyncio.run(run_tests())

## Edge Cases

### Zero throttle interval

When `t = 0`, the cooldown window effectively disappears. A buggy implementation may accidentally enter an infinite scheduling loop or incorrectly queue requests forever.

This implementation handles the case naturally because `asyncio.sleep(0)` immediately yields control back to the event loop, allowing batches to flush correctly without delay.

### Long-running `queryMultiple`

One subtle aspect of the problem is that throttling depends on query start times, not completion times.

A naive implementation might wait for the previous request to finish before allowing another batch, which would incorrectly serialize requests.

This implementation avoids that mistake because the cooldown timer is independent from query completion. New batches may begin even while previous asynchronous queries are still running.

### Requests arriving exactly at cooldown expiration

Timing boundaries often cause race conditions.

For example, if a request arrives at precisely `t` milliseconds after the previous batch started, it should be eligible for immediate batching rather than being delayed another cycle.

The implementation handles this safely because the cooldown loop checks the queue after the sleep completes. Any requests already enqueued before that moment are included in the next batch immediately.
