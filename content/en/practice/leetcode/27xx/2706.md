# LeetCode 2706 - Buy Two Chocolates

## Problem Understanding

This problem gives us an array `prices`, where each element represents the price of a chocolate, and an integer `money` representing the amount of money we currently have.

We must buy exactly two chocolates. Among all possible pairs of chocolates, we want to spend as little money as possible. After purchasing the two chocolates, we need to determine how much money remains.

There is one important restriction: the remaining money cannot be negative. In other words, the total cost of the two chocolates must be less than or equal to `money`.

If it is possible to buy two chocolates without going into debt, we return the leftover amount:

```
money - cost_of_two_chocolates
```

If every possible pair costs more than `money`, then we cannot make a valid purchase. In that situation, the problem asks us to return the original amount of money unchanged.

The constraints are very small:

- `2 <= prices.length <= 50`
- `1 <= prices[i] <= 100`
- `1 <= money <= 100`

The array always contains at least two chocolates, so we never need to handle an input with fewer than two elements.

Several edge cases are worth considering:

- The two cheapest chocolates may cost exactly `money`, leaving `0` leftover.
- The two cheapest chocolates may cost more than `money`, making any purchase impossible.
- Multiple chocolates may have the same price.
- The cheapest pair might appear anywhere in the array, not necessarily adjacent.
- Since we must minimize spending, we only care about the two smallest prices in the array.

## Approaches

### Brute Force

A straightforward approach is to examine every possible pair of chocolates.

For every pair of indices `(i, j)` where `i < j`, we compute the total cost:

```
prices[i] + prices[j]
```

Among all pairs, we find the minimum total cost. Once we know the cheapest possible pair, we check whether we can afford it.

If the minimum cost is less than or equal to `money`, we return the leftover amount. Otherwise, we return `money`.

This approach is correct because it explicitly evaluates every possible pair and therefore cannot miss the optimal one.

However, it performs unnecessary work. We are only interested in the two smallest prices, and checking every pair is more expensive than needed.

### Optimal Approach

The key observation is that the cheapest possible pair must consist of the two cheapest chocolates in the entire array.

Suppose the two smallest prices are `a` and `b`.

Any other chocolate has a price greater than or equal to one of these values, so replacing either `a` or `b` with another chocolate cannot decrease the total cost. Therefore, the minimum possible pair cost is simply:

```
a + b
```

We can find the two smallest prices with a single pass through the array.

After obtaining their sum:

- If the sum is greater than `money`, return `money`.
- Otherwise, return `money - sum`.

This avoids checking every pair and produces the answer in linear time.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Check every possible pair and find the minimum cost |
| Optimal | O(n) | O(1) | Track the two smallest prices during one scan |

## Algorithm Walkthrough

1. Initialize two variables, `smallest` and `second_smallest`, to very large values. These variables will store the two cheapest chocolate prices seen so far.
2. Iterate through every price in the array.
3. If the current price is smaller than `smallest`, then the previous `smallest` becomes the new `second_smallest`, and the current price becomes the new `smallest`.
4. Otherwise, if the current price is smaller than `second_smallest`, update only `second_smallest`.
5. After processing all prices, the two variables contain the cheapest and second-cheapest chocolate prices in the array.
6. Compute their total cost:

```
cost = smallest + second_smallest
```
7. If `cost > money`, purchasing the two chocolates would result in debt, so return `money`.
8. Otherwise, return:

```
money - cost
```

### Why it works

The algorithm maintains the invariant that after processing any prefix of the array, `smallest` and `second_smallest` contain the two lowest prices encountered so far.

By the end of the scan, they represent the two smallest prices in the entire array. Any valid pair of chocolates must have a total cost greater than or equal to the sum of these two values. Therefore, this sum is the minimum possible amount we can spend. If that minimum cost is affordable, it yields the maximum possible leftover money under the problem's requirement to buy the cheapest pair. If it is not affordable, no other pair can be affordable either because every other pair costs at least as much.

## Python Solution

```python
from typing import List

class Solution:
    def buyChoco(self, prices: List[int], money: int) -> int:
        smallest = float('inf')
        second_smallest = float('inf')

        for price in prices:
            if price < smallest:
                second_smallest = smallest
                smallest = price
            elif price < second_smallest:
                second_smallest = price

        cost = smallest + second_smallest

        if cost > money:
            return money

        return money - cost
```

The implementation follows the algorithm directly.

The variables `smallest` and `second_smallest` track the two lowest prices encountered so far. During the traversal, whenever a new minimum is found, the previous minimum is shifted into the second position. If the current value is not the smallest but is smaller than the second-smallest value, only the second position is updated.

After the loop finishes, the sum of these two values represents the cheapest possible purchase. A simple affordability check determines whether we return the leftover money or the original amount.

## Go Solution

```go
func buyChoco(prices []int, money int) int {
	smallest := 1 << 30
	secondSmallest := 1 << 30

	for _, price := range prices {
		if price < smallest {
			secondSmallest = smallest
			smallest = price
		} else if price < secondSmallest {
			secondSmallest = price
		}
	}

	cost := smallest + secondSmallest

	if cost > money {
		return money
	}

	return money - cost
}
```

The Go implementation mirrors the Python solution. Since Go does not provide a built-in infinity value for integers, a very large integer is used as the initial value for `smallest` and `secondSmallest`.

Integer overflow is not a concern because the maximum possible cost is only `100 + 100 = 200`, which is far below Go's integer limits.

## Edge Cases

### The cheapest pair costs exactly the available money

Consider:

```
prices = [1, 2]
money = 3
```

The cheapest pair costs exactly `3`, leaving `0` money. A buggy implementation might incorrectly require the leftover to be positive rather than non-negative. The problem explicitly allows zero leftover money, and the implementation correctly returns `0`.

### The cheapest pair is unaffordable

Consider:

```
prices = [5, 6]
money = 10
```

The cheapest possible pair costs `11`, which exceeds the budget. Since every other pair would cost at least as much, no valid purchase exists. The implementation detects this by checking whether the minimum pair cost exceeds `money` and returns the original amount unchanged.

### Multiple chocolates have the same minimum price

Consider:

```
prices = [2, 2, 2, 2]
money = 5
```

The correct pair uses two chocolates priced at `2`, for a total cost of `4`. Some implementations accidentally fail when duplicate minimum values appear. The two-variable tracking method correctly stores both occurrences and computes the proper minimum pair cost.

### The two cheapest chocolates appear late in the array

Consider:

```
prices = [100, 90, 1, 2]
money = 10
```

An implementation that assumes early values are likely to remain the minimums may produce incorrect results. The algorithm continuously updates the smallest and second-smallest values throughout the entire traversal, ensuring that late-arriving cheaper chocolates are handled correctly.
