## Solution

# Problem Understanding

The problem gives us an array of distinct strings called `words`. Every string has exactly two lowercase English letters. We need to determine how many valid pairs of strings can be formed.

A pair `(i, j)` is valid if:

- `i < j`
- `words[i]` is equal to the reverse of `words[j]`

Because each string can only belong to one pair, once two strings are matched together, they cannot be reused in another pair.

Since every string has length `2`, reversing a string is very simple. For example:

- `"ab"` reversed becomes `"ba"`
- `"dc"` reversed becomes `"cd"`

The task is to return the maximum number of such reverse pairs.

The input size is very small:

- `1 <= words.length <= 50`
- Each string length is exactly `2`

These constraints tell us that even an `O(n²)` brute-force solution is completely acceptable. However, there is also a cleaner and more efficient hash-set solution that runs in linear time.

An important detail is that all strings are distinct. This guarantees there are no duplicate entries like `["ab", "ab"]`. It also simplifies pair counting because each reverse match can only happen once.

Several edge cases are important:

- Strings like `"aa"` reverse to themselves. Since the array contains distinct strings, there can only be one `"aa"`, so it cannot form a pair.
- Some strings may have no reverse counterpart at all.
- Multiple independent pairs may exist in the same array.
- The array may contain only one element.

# Approaches

## Brute Force Approach

The most direct solution is to compare every pair of strings.

For every index `i`, we iterate through all indices `j > i`. For each pair, we reverse one string and check whether it matches the other.

If:

```
words[i] == reverse(words[j])
```

then we increment the answer.

Because the strings are distinct and each valid reverse pair is unique, we do not need additional bookkeeping to avoid reuse.

This approach is correct because it explicitly checks every possible valid pair. If a reverse relationship exists, we will eventually encounter it during the nested iteration.

The drawback is the quadratic time complexity. Even though the constraints are small enough for this to pass easily, we can still do better.

## Optimal Hash Set Approach

The key observation is that for any word, we only care whether its reversed version already exists.

For example:

- Current word: `"dc"`
- Reverse: `"cd"`

If `"cd"` has already been seen, then we have found a valid pair.

We can maintain a hash set of previously processed strings. For each word:

1. Compute its reversed version.
2. Check whether the reverse exists in the set.
3. If it exists, increment the answer.
4. Insert the current word into the set.

This works because:

- Every valid pair is detected exactly once.
- We only match against previously seen words, so we naturally avoid double counting.
- Hash set lookup is `O(1)` on average.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Compare every pair directly |
| Optimal | O(n) | O(n) | Uses a hash set to detect reverse matches efficiently |

# Algorithm Walkthrough

1. Initialize an empty hash set called `seen`.

This set stores all words we have already processed. We use a hash set because membership checks are extremely fast.
2. Initialize a variable `pairs = 0`.

This variable keeps track of the number of valid reverse pairs found so far.
3. Iterate through each word in `words`.

We process the array from left to right.
4. Reverse the current word.

Since every string has length `2`, reversing can be done with slicing:

```
reversed_word = word[::-1]
```
5. Check whether the reversed word already exists in `seen`.

If it does, then we have found a valid pair:

```
current word = reverse(previous word)
```

Increment `pairs`.
6. Add the current word to `seen`.

This allows future words to match against it.
7. After processing all words, return `pairs`.

## Why it works

The algorithm maintains the invariant that `seen` contains exactly the words processed before the current iteration.

When processing a word `w`, if `reverse(w)` exists in `seen`, then there is an earlier word whose reverse equals `w`. Therefore, a valid pair exists.

Because we only match against previously seen words, each pair is counted exactly once. Since every valid reverse relationship will eventually be encountered during iteration, the algorithm finds all possible pairs.

# Python Solution

```python
from typing import List

class Solution:
    def maximumNumberOfStringPairs(self, words: List[str]) -> int:
        seen = set()
        pairs = 0

        for word in words:
            reversed_word = word[::-1]

            if reversed_word in seen:
                pairs += 1

            seen.add(word)

        return pairs
```

The implementation starts by creating an empty hash set called `seen` and a counter called `pairs`.

For every word in the array, the code computes its reversed version using Python slicing. The expression `word[::-1]` creates a reversed copy of the string.

The algorithm then checks whether this reversed string already exists in `seen`. If it does, we increment the pair counter because a valid reverse pair has been found.

Finally, the current word is inserted into the set so later words can match against it.

At the end of the loop, the total number of valid pairs is returned.

# Go Solution

```go
func maximumNumberOfStringPairs(words []string) int {
	seen := make(map[string]bool)
	pairs := 0

	for _, word := range words {
		reversedWord := string([]byte{word[1], word[0]})

		if seen[reversedWord] {
			pairs++
		}

		seen[word] = true
	}

	return pairs
}
```

The Go solution follows the same logic as the Python version.

Instead of using a Python set, Go uses a `map[string]bool` to simulate hash set behavior.

Because each word always has length `2`, reversing the string is especially simple. We directly construct the reversed string using:

```go
string([]byte{word[1], word[0]})
```

No special handling for empty input is needed because the loop naturally works for empty or single-element slices.

Integer overflow is not a concern because the maximum answer is at most `25`.

# Edge Cases

## Self-reversing strings

Strings like `"aa"` or `"cc"` are equal to their own reverse. A naive implementation might incorrectly count these as pairs automatically.

However, the problem guarantees all strings are distinct. Since there can only be one `"aa"` in the array, it cannot pair with itself. The implementation handles this naturally because a pair is only counted when the reversed string already exists in `seen`.

## No valid reverse pairs

An input like:

```
["ab", "cd", "ef"]
```

contains no reverses at all. Some implementations may accidentally overcount if they use incorrect substring logic or compare characters improperly.

The hash-set approach avoids this issue by requiring an exact reverse match before incrementing the answer.

## Multiple independent pairs

Inputs such as:

```
["ab", "ba", "cd", "dc"]
```

contain several valid pairs. A buggy solution might accidentally reuse strings or double count pairs.

This implementation prevents double counting because each pair is discovered exactly once when the second matching string is processed.
