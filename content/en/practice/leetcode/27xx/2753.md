# LeetCode 2753 - Count Houses in a Circular Street II

## Problem Understanding

This problem gives us access to a hidden circular street through a very limited interface. We do not know the actual number of houses directly, and we cannot inspect the entire structure like a normal array. Instead, we can only interact with the street using three operations:

- Check whether the current house door is open
- Close the current house door
- Move one house to the right

The street is circular, meaning that after the last house, we return to the first house again.

The key challenge is that we are not given direct access to the house indices or the total size `n`. We are only told that the total number of houses is at most `k`, and at least one door is initially open.

Our task is to determine exactly how many houses exist in the circular street.

The important observation is that because the street is circular, if we keep moving right long enough, we eventually revisit the same house. However, we do not have a direct way to detect revisits. We therefore need to create our own marker or stopping condition using the door states.

The constraint `1 <= n <= k <= 10^5` tells us that a linear solution is expected. Any approach that repeatedly loops around the street many times could become too slow.

There are several important edge cases:

- The street may contain only one house.
- Only one door may initially be open.
- All doors may initially be open.
- The number of houses may equal `k`.
- The initial starting house could be any house in the cycle.

The guarantee that at least one door is initially open is extremely important because it gives us a reliable marker that we can use to identify when we have completed a full traversal.

## Approaches

### Brute Force Approach

A naive idea would be to repeatedly traverse the street and attempt to identify cycles by storing visited configurations or movement counts.

For example, we could try walking around while recording patterns of open and closed doors. Unfortunately, this does not work reliably because different houses may share the same door state. Since we have no unique identifiers for houses, we cannot distinguish one closed door from another closed door.

Another brute force style idea is to move exactly `k` steps and somehow infer whether we wrapped around earlier. However, without modifying the street in a structured way, there is no guaranteed method to determine the exact cycle length.

These approaches either fail to uniquely identify houses or require excessive repeated traversals.

### Optimal Approach

The key insight is that we can use door states as markers.

Since at least one door is initially open, we can first find an open door and close it. That modified house now becomes a unique marker because its state changed from open to closed due to our action.

After closing one open door, we continue moving right and counting houses until we encounter another open door. Every time we encounter an open door, we close it as well.

Eventually, after traversing the entire cycle, we return to the original house whose door we previously closed. At that point, every originally open door has been closed. Therefore, when we complete another full cycle, we will encounter no open doors at all.

A simpler and cleaner strategy is:

1. Walk up to `k` steps until we find an open door.
2. Close that door.
3. Continue moving right while counting steps.
4. The first time we encounter an open door again, we know we have completed a full cycle.
5. The number of steps taken equals the number of houses.

The reason this works is that the first open door encountered after closing one must correspond to revisiting another initially open house after one complete traversal of the circle.

A more elegant version repeatedly closes open doors and resets the counter until no open door appears within `k` steps. The final count then equals the number of houses.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(k²) | O(1) | Multiple full traversals with unreliable cycle detection |
| Optimal | O(k) | O(1) | Uses door state modifications as traversal markers |

## Algorithm Walkthrough

The optimal algorithm uses the fact that we can permanently modify door states.

1. First, traverse the street until we encounter an open door. Since the problem guarantees that at least one door is open, we are guaranteed to find one within at most `k` moves.
2. Once we find an open door, close it. This house now acts as a marker because its state has changed permanently.
3. Initialize a counter to track how many moves we make while traversing the street.
4. Move right one house at a time.
5. After each move:

- Increment the counter.
- Check whether the current door is open.
6. If the current door is open:

- Close it.
- Reset the counter to `0`.

Resetting is important because encountering an open door means we have not yet completed a full cycle since the most recently closed open door.
7. Continue this process until the counter reaches `k`.
8. Once the counter reaches `k`, it means we have moved `k` consecutive houses without finding any open doors. Since the street contains at most `k` houses, this implies we completed a full traversal of the cycle and returned to the most recently closed door.
9. Return the counter value, which equals the number of houses.

### Why it works

The invariant is that every time we encounter an open door, we close it and restart counting from that house.

Eventually, all originally open doors become closed. At that point, the only way to move `n` consecutive steps without encountering an open door is to complete exactly one full cycle of the circular street.

Because `n <= k`, the first time we successfully travel `k` consecutive steps without finding an open door corresponds to traversing the entire street exactly once. Therefore, the counter equals the total number of houses.

## Python Solution

```python
# Definition for a street.
# class Street:
#     def closeDoor(self):
#         pass
#     def isDoorOpen(self):
#         pass
#     def moveRight(self):
#         pass

from typing import Optional

class Solution:
    def houseCount(self, street: Optional['Street'], k: int) -> int:
        consecutive_closed = 0

        while consecutive_closed < k:
            if street.isDoorOpen():
                street.closeDoor()
                consecutive_closed = 0
            else:
                consecutive_closed += 1

            street.moveRight()

        return consecutive_closed
```

The implementation maintains a counter called `consecutive_closed`.

At each house, we inspect the door state.

If the door is open, we close it and reset the counter because we have discovered another previously unprocessed open house.

If the door is already closed, we increment the counter because we have moved one more step without encountering an open door.

After processing the current house, we move right to continue traversing the circular street.

Eventually, every initially open door becomes closed. Once that happens, we can traverse the entire cycle without encountering any open door. The number of consecutive closed houses encountered before seeing another open door therefore equals the total number of houses.

The loop stops once `consecutive_closed == k`. Since the actual number of houses is at most `k`, this guarantees we have completed a full cycle.

## Go Solution

```go
/**
 * Definition for a street.
 * type Street interface {
 *     CloseDoor()
 *     IsDoorOpen() bool
 *     MoveRight()
 * }
 */

func houseCount(street Street, k int) int {
    consecutiveClosed := 0

    for consecutiveClosed < k {
        if street.IsDoorOpen() {
            street.CloseDoor()
            consecutiveClosed = 0
        } else {
            consecutiveClosed++
        }

        street.MoveRight()
    }

    return consecutiveClosed
}
```

The Go implementation follows the same logic as the Python version.

Go uses explicit variable declarations and standard `for` loop syntax instead of Python's `while` loop. Since the problem constraints are well within integer limits, normal Go `int` values are sufficient and no overflow concerns arise.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(k) | Each house is processed a constant number of times |
| Space | O(1) | Only a few integer variables are used |

The algorithm runs in linear time relative to the upper bound `k`.

Each open door is closed exactly once. After all open doors are closed, we perform one final traversal around the cycle. Therefore, the total number of operations remains proportional to `k`.

No additional data structures are required, so the space usage is constant.

## Edge Cases

One important edge case is when the street contains exactly one house. In this scenario, moving right immediately returns to the same house. A naive implementation might accidentally enter an infinite loop if it does not properly handle the circular structure. Our solution works correctly because the single open door is closed immediately, and the next traversal observes one consecutive closed house.

Another tricky case occurs when only one door is initially open and every other door is closed. A flawed implementation might incorrectly assume that encountering many closed doors means the traversal is complete. Our algorithm avoids this issue because it only finalizes the answer after all open doors have been eliminated and a full uninterrupted cycle of closed doors is observed.

A third important edge case is when all doors are initially open. In this situation, the counter repeatedly resets to zero during the first traversal because every encountered house must be closed. The implementation correctly handles this by continuing until one entire cycle occurs with no remaining open doors.

Finally, the value of `k` may be much larger than the actual number of houses. A buggy solution might mistakenly return `k` instead of the true cycle length. Our implementation relies on actual traversal behavior rather than the upper bound itself, so it always returns the real number of houses.
