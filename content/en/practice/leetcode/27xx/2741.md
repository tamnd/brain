## Solution

## Problem Understanding

The problem gives us an array `nums` containing distinct positive integers. We must count how many permutations of the array satisfy a special divisibility condition.

A permutation is considered special if every pair of adjacent numbers satisfies at least one of these:

- `nums[i] % nums[i + 1] == 0`
- `nums[i + 1] % nums[i] == 0`

In other words, for every neighboring pair, one number must divide the other exactly.

We are not asked to construct the permutations themselves. We only need to return the total number of valid permutations modulo `10^9 + 7`.

The array length is at most `14`, which is the key constraint that drives the solution design. A naive approach that generates all permutations would require checking up to:

```
14! = 87,178,291,200
```

This is far too large for brute force enumeration.

However, `n <= 14` strongly suggests a bitmask dynamic programming solution because:

```
2^14 = 16384
```

which is manageable.

The numbers are distinct, which simplifies the problem because we never need to worry about duplicate permutations.

Some important edge cases include:

- Arrays where no two numbers are compatible, resulting in zero valid permutations.
- Arrays where every pair is compatible, resulting in `n!` valid permutations.
- Small arrays of size `2`, where the answer is either `0` or `2`.
- Large values up to `10^9`, which means we cannot rely on frequency arrays or small-value optimizations. We must compare numbers directly.

## Approaches

### Brute Force Approach

The brute force solution generates every permutation of the array and checks whether every adjacent pair satisfies the divisibility condition.

For each permutation:

1. Traverse adjacent elements.
2. Verify whether one divides the other.
3. Count the permutation if all pairs are valid.

This approach is correct because it explicitly examines every possible arrangement.

However, the time complexity is extremely large. There are `n!` permutations, and each requires `O(n)` validation work.

For `n = 14`, this becomes completely impractical.

### Optimal Approach, Bitmask Dynamic Programming

The key observation is that the validity of a permutation depends only on:

- Which numbers have already been used
- What the last chosen number is

This is a classic state compression dynamic programming problem.

We define a DP state:

```
dp[mask][last]
```

where:

- `mask` represents which indices have already been used
- `last` represents the index of the final number in the current partial permutation

The value stores the number of valid ways to build such a partial permutation.

From a current state, we try to append another unused number that satisfies the divisibility condition with the current last number.

Since there are only:

```
2^n * n
```

states, and each transition tries up to `n` next elements, the solution becomes feasible.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n! × n) | O(n) | Generates all permutations and validates each |
| Optimal | O(2^n × n^2) | O(2^n × n) | Uses bitmask DP to avoid recomputation |

## Algorithm Walkthrough

1. First, precompute which pairs of indices are compatible.

For every pair `(i, j)`, determine whether:

```
nums[i] % nums[j] == 0
```

or

```
nums[j] % nums[i] == 0
```

This avoids repeated divisibility checks during DP transitions.
2. Define the DP state.

Let:

```
dp(mask, last)
```

represent the number of valid special permutations that:

- use exactly the elements indicated by `mask`
- end at index `last`
3. Handle the base case.

If every element has been used:

```
mask == (1 << n) - 1
```

then we have formed one complete valid permutation, so return `1`.
4. Try extending the current permutation.

For every unused index `next_index`:

- check whether it is compatible with `last`
- if compatible, recursively compute:

```
dp(mask | (1 << next_index), next_index)
```
5. Memoize results.

Many states repeat during recursion. Memoization ensures each state is computed only once.
6. Start DFS from every possible first element.

Since any element can begin a permutation, initialize:

```
dp(1 << i, i)
```

for every index `i`.
7. Return the total modulo `10^9 + 7`.

### Why it works

The DP state completely captures all information needed to continue building a valid permutation:

- `mask` tells us which numbers are already used
- `last` tells us which number the next element must be compatible with

Because every recursive step only appends compatible unused numbers, every counted permutation is valid. Conversely, every valid permutation corresponds to exactly one path through the DP states. Therefore, the algorithm counts all valid permutations exactly once.

## Python Solution

```python
from typing import List
from functools import lru_cache

class Solution:
    def specialPerm(self, nums: List[int]) -> int:
        MOD = 10**9 + 7
        n = len(nums)

        compatible = [[False] * n for _ in range(n)]

        for i in range(n):
            for j in range(n):
                if i != j:
                    if nums[i] % nums[j] == 0 or nums[j] % nums[i] == 0:
                        compatible[i][j] = True

        full_mask = (1 << n) - 1

        @lru_cache(None)
        def dfs(mask: int, last: int) -> int:
            if mask == full_mask:
                return 1

            total = 0

            for next_index in range(n):
                if (mask >> next_index) & 1:
                    continue

                if compatible[last][next_index]:
                    next_mask = mask | (1 << next_index)
                    total += dfs(next_mask, next_index)
                    total %= MOD

            return total

        answer = 0

        for start in range(n):
            answer += dfs(1 << start, start)
            answer %= MOD

        return answer
```

The implementation begins by building a compatibility matrix. This matrix stores whether two indices can appear next to each other in a valid permutation. Precomputing this information reduces repeated divisibility checks during recursion.

The recursive function `dfs(mask, last)` represents the number of valid ways to complete a permutation using the current state.

The bitmask tracks which indices are already included. Each bit corresponds to one array position:

- `1` means used
- `0` means unused

Whenever all bits are set, a complete valid permutation has been formed, so the function returns `1`.

For each recursive call, the algorithm iterates through all unused indices. If the candidate index is compatible with the current last index, the algorithm extends the permutation recursively.

The `lru_cache` decorator memoizes states so that repeated subproblems are solved only once.

Finally, the algorithm starts a DFS from every possible starting element because any number may appear first in a valid permutation.

## Go Solution

```go
package main

func specialPerm(nums []int) int {
	const MOD int = 1e9 + 7

	n := len(nums)

	compatible := make([][]bool, n)
	for i := 0; i < n; i++ {
		compatible[i] = make([]bool, n)
	}

	for i := 0; i < n; i++ {
		for j := 0; j < n; j++ {
			if i != j {
				if nums[i]%nums[j] == 0 || nums[j]%nums[i] == 0 {
					compatible[i][j] = true
				}
			}
		}
	}

	fullMask := (1 << n) - 1

	dp := make([][]int, 1<<n)
	for i := range dp {
		dp[i] = make([]int, n)
		for j := range dp[i] {
			dp[i][j] = -1
		}
	}

	var dfs func(mask int, last int) int

	dfs = func(mask int, last int) int {
		if mask == fullMask {
			return 1
		}

		if dp[mask][last] != -1 {
			return dp[mask][last]
		}

		total := 0

		for nextIndex := 0; nextIndex < n; nextIndex++ {
			if (mask>>nextIndex)&1 == 1 {
				continue
			}

			if compatible[last][nextIndex] {
				nextMask := mask | (1 << nextIndex)
				total += dfs(nextMask, nextIndex)
				total %= MOD
			}
		}

		dp[mask][last] = total
		return total
	}

	answer := 0

	for start := 0; start < n; start++ {
		answer += dfs(1<<start, start)
		answer %= MOD
	}

	return answer
}
```

The Go implementation uses an explicit memoization table instead of Python's `lru_cache`.

Since Go does not support automatic memoization decorators, the DP table is initialized with `-1` to indicate uncomputed states.

The rest of the logic mirrors the Python solution closely:

- precompute compatibility
- use bitmask DP
- recursively extend permutations
- memoize computed states

Go integer arithmetic is safe here because all intermediate results are reduced modulo `10^9 + 7`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(2^n × n^2) | There are `2^n × n` states, each trying up to `n` transitions |
| Space | O(2^n × n) | DP memoization table stores one value per state |

The algorithm processes each DP state at most once because of memoization. Each state may iterate through all `n` possible next elements, giving total complexity:

```
O(2^n × n × n)
```

With `n <= 14`, this is efficient enough for the problem constraints.

## Edge Cases

One important edge case occurs when no two numbers are compatible. For example:

```
[7, 11, 13]
```

No adjacent pair can satisfy the divisibility rule, so no valid permutation exists. A naive recursive implementation might accidentally count incomplete paths. The DP solution avoids this because recursive states only extend through compatible transitions. If no transition exists, the recursion contributes `0`.

Another important case is when all numbers are mutually compatible. For example:

```
[2, 4, 8]
```

Every adjacent arrangement works, so the answer becomes `n!`. This stresses the branching factor of the recursion. Memoization is crucial here because many equivalent subproblems appear repeatedly.

A third edge case involves the number `1`. Since every integer is divisible by `1`, this value acts like a universal connector. For example:

```
[1, 7, 14]
```

The element `1` can connect otherwise incompatible numbers. Incorrect implementations sometimes overlook bidirectional divisibility and only check one direction. This implementation explicitly checks both:

```
a % b == 0 or b % a == 0
```

ensuring all valid connections are recognized correctly.
