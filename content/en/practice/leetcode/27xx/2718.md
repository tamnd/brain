## Solution

## Problem Understanding

The problem asks us to calculate the sum of all elements in an `n x n` matrix after performing a sequence of queries. Initially, the matrix contains all zeros. Each query either updates an entire row or an entire column to a specific value, overwriting any previous values in that row or column. The matrix is **0-indexed**, meaning the first row or column is index `0`.

The input `n` specifies the size of the square matrix. The `queries` list contains sublists of length 3: `[typei, indexi, vali]`. If `typei` is `0`, the row at `indexi` is set to `vali` in all columns; if `typei` is `1`, the column at `indexi` is set to `vali` in all rows.

The expected output is the **sum of all integers in the matrix after all queries are applied**.

Constraints indicate potentially very large matrices (`n` up to `10^4`) and many queries (`queries.length` up to `5*10^4`). This makes a naive approach of updating every element for each query impractical, as it could require `O(n * queries.length)` operations, which could reach `5*10^8` operations in the worst case.

Edge cases to consider include sequences of queries that overwrite the same row or column multiple times, very large values in `vali`, and scenarios where only rows or only columns are updated. The problem guarantees valid indices and values, so we do not need to handle out-of-bounds or negative numbers.

## Approaches

### Brute Force

The brute-force approach creates an `n x n` matrix and directly applies each query. For a row update, it iterates through all columns and sets the value; for a column update, it iterates through all rows. After all queries, the sum of the matrix is calculated by iterating through all elements.

While correct, this approach is too slow for large `n` because each query can touch up to `n` elements. If `n = 10^4` and `queries.length = 5*10^4`, the total operations can exceed `5*10^8`, which is not feasible within typical time limits.

### Optimal Approach

The key insight is that **the last update to a row or column determines the value of its cells**, and earlier updates are overwritten. Instead of updating the matrix explicitly, we can process queries in **reverse order** and only consider rows or columns that have not been counted yet. By tracking which rows and columns have been updated, we can efficiently compute the total contribution of each query to the sum.

This avoids updating individual cells and reduces the complexity to **O(n + queries.length)** in the worst case.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * queries.length) | O(n^2) | Directly update the matrix for each query, then sum all elements |
| Optimal | O(queries.length + n) | O(n) | Track updated rows and columns to compute contributions in reverse |

## Algorithm Walkthrough

1. Initialize a set for updated rows (`updated_rows`) and updated columns (`updated_cols`), and initialize `total_sum` to 0.
2. Iterate through the queries **from last to first**. This ensures we consider the most recent updates first.
3. For each query `[typei, indexi, vali]`:

- If it is a row update (`typei == 0`) and the row `indexi` has not been updated:

- Add `vali * (n - len(updated_cols))` to `total_sum`. This accounts for cells in this row that are **not already overwritten by column updates**.
- Mark this row as updated by adding `indexi` to `updated_rows`.
- If it is a column update (`typei == 1`) and the column `indexi` has not been updated:

- Add `vali * (n - len(updated_rows))` to `total_sum`. This accounts for cells in this column that are **not already overwritten by row updates**.
- Mark this column as updated by adding `indexi` to `updated_cols`.
4. After processing all queries, return `total_sum`.

**Why it works:** Each cell is effectively counted **exactly once**, using the last operation that affects it. Row updates contribute to all uncounted cells in that row, and column updates contribute to all uncounted cells in that column. By processing in reverse, we ensure that no cell is double-counted.

## Python Solution

```python
from typing import List

class Solution:
    def matrixSumQueries(self, n: int, queries: List[List[int]]) -> int:
        updated_rows = set()
        updated_cols = set()
        total_sum = 0
        
        # Process queries in reverse
        for t, idx, val in reversed(queries):
            if t == 0 and idx not in updated_rows:
                total_sum += val * (n - len(updated_cols))
                updated_rows.add(idx)
            elif t == 1 and idx not in updated_cols:
                total_sum += val * (n - len(updated_rows))
                updated_cols.add(idx)
        
        return total_sum
```

The code uses Python sets to track which rows and columns have already been processed. By iterating in reverse, the last updates are applied first. For row updates, we only add the value for columns that haven't been overwritten yet, and vice versa for column updates.

## Go Solution

```go
func matrixSumQueries(n int, queries [][]int) int64 {
    updatedRows := make(map[int]bool)
    updatedCols := make(map[int]bool)
    var totalSum int64 = 0
    
    for i := len(queries) - 1; i >= 0; i-- {
        t, idx, val := queries[i][0], queries[i][1], queries[i][2]
        if t == 0 {
            if !updatedRows[idx] {
                totalSum += int64(val) * int64(n - len(updatedCols))
                updatedRows[idx] = true
            }
        } else {
            if !updatedCols[idx] {
                totalSum += int64(val) * int64(n - len(updatedRows))
                updatedCols[idx] = true
            }
        }
    }
    
    return totalSum
}
```

In Go, maps are used to track updated rows and columns. We also explicitly cast integers to `int64` to prevent overflow, as the product of `n` and `val` can exceed the 32-bit integer limit.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(queries.length + n) | Each query is processed once, and set/map operations are O(1) on average |
| Space | O(n) | Sets/maps to track updated rows and columns can grow up to size n |

The key efficiency comes from avoiding explicit updates to all `n^2` cells and using sets to track which rows and columns are already counted.
