# LeetCode 2727 - Is Object Empty

## Problem Understanding

The problem asks us to determine whether a given JSON object or JSON array is empty.

An object is considered empty when it contains no key-value pairs. For example, `{}` is empty, while `{"a": 1}` is not.

An array is considered empty when it contains no elements. For example, `[]` is empty, while `[1, 2]` is not.

The input is guaranteed to come from `JSON.parse`, which means the value will always be either:

- A JavaScript object
- A JavaScript array

We do not need to handle functions, class instances, maps, sets, or other special JavaScript types.

The expected output is a boolean:

- `true` if the object or array is empty
- `false` otherwise

The constraints tell us that the serialized JSON length can be as large as `10^5`, so the structure may contain many keys or many array elements. Because of this, we should think carefully about efficiency.

The follow-up question asks whether we can solve the problem in `O(1)` time. This is the key insight of the problem. Instead of iterating through all keys or all array elements, we only need to determine whether at least one element exists.

Several edge cases are important:

- Empty object: `{}`
- Empty array: `[]`
- Arrays containing falsy values like `0`, `false`, or `null`
- Objects with keys whose values are falsy
- Very large arrays or objects

A naive implementation might incorrectly treat falsy values as empty, but the problem only cares about whether entries exist, not about the values themselves.

## Approaches

### Brute Force Approach

A straightforward solution is to iterate through the entire object or array and count how many entries it contains.

For an array, we could loop through every element and count them.

For an object, we could iterate through every key and count the number of key-value pairs.

If the count is zero, the structure is empty. Otherwise, it is not.

This approach is correct because it explicitly checks every possible entry. However, it is inefficient because it scans the entire structure even when the first entry already proves the structure is non-empty.

For very large inputs, this unnecessary traversal wastes time.

### Optimal Approach

The key observation is that we do not need to inspect all contents. We only need to know whether at least one element or key exists.

In JavaScript:

- Arrays expose their size through the `length` property
- Objects expose their keys through `Object.keys(obj)`

If the number of keys or elements is zero, the structure is empty.

This allows us to determine the answer immediately without manually traversing the contents ourselves.

For arrays, checking `obj.length === 0` is effectively `O(1)`.

For objects, `Object.keys(obj).length === 0` is commonly accepted for this problem and is concise and readable.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Iterates through every key or element |
| Optimal | O(1) for arrays, O(n) for objects | O(n) for objects | Uses built-in size information or key extraction |

## Algorithm Walkthrough

1. Receive the input object or array.
2. Determine whether the input is an array or an object.
3. If the input is an array, check its `length` property.

- If `length` is `0`, return `true`
- Otherwise, return `false`
4. If the input is an object, retrieve all keys using `Object.keys(obj)`.
5. Check the number of keys.

- If the number of keys is `0`, return `true`
- Otherwise, return `false`

### Why it works

An array is empty exactly when it contains zero elements, and the `length` property directly stores that information.

An object is empty exactly when it has zero keys, and `Object.keys(obj)` returns all enumerable keys in the object.

Therefore, checking whether the count is zero correctly determines emptiness in all valid cases.

## Python Solution

Although the original LeetCode problem is written in JavaScript, below is an equivalent Python implementation that follows the same logic.

```python
from typing import Any

class Solution:
    def isEmpty(self, obj: Any) -> bool:
        if isinstance(obj, list):
            return len(obj) == 0

        return len(obj.keys()) == 0
```

The implementation first checks whether the input is a list.

If it is a list, the solution simply checks whether its length is zero.

Otherwise, the input is treated as a dictionary. The solution retrieves the dictionary keys and checks whether the number of keys is zero.

This directly matches the algorithm described earlier and avoids unnecessary traversal logic.

## Go Solution

```go
package main

import "reflect"

func isEmpty(obj interface{}) bool {
	value := reflect.ValueOf(obj)

	switch value.Kind() {
	case reflect.Slice, reflect.Array:
		return value.Len() == 0

	case reflect.Map:
		return len(value.MapKeys()) == 0
	}

	return false
}
```

In Go, the input type is represented using `interface{}` because the value may be either an array-like structure or a map.

The solution uses the `reflect` package to inspect the runtime type.

For slices and arrays, `Len()` provides the number of elements.

For maps, `MapKeys()` retrieves all keys, and the solution checks whether the number of keys is zero.

Unlike Python or JavaScript, Go does not have built-in dynamic JSON object handling without reflection, so reflection is the cleanest general-purpose approach here.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(1) for arrays, O(n) for objects | Array length is stored directly, object key extraction scans keys |
| Space | O(n) for objects | `Object.keys()` or equivalent stores all keys |

For arrays, retrieving the length is constant time because the size is stored internally.

For objects, generating the key list requires traversing the keys, which takes linear time and additional memory proportional to the number of keys.

## Edge Cases

### Empty Array

An empty array, `[]`, is the most direct edge case. A naive implementation that checks truthiness instead of length might accidentally confuse empty arrays with other falsy values. The implementation correctly handles this by explicitly checking whether the array length equals zero.

### Arrays Containing Falsy Values

Arrays such as `[0, false, null]` can cause bugs in incorrect implementations that test element truthiness rather than element existence. Even though all values are falsy, the array still contains three elements. The solution correctly checks only the array length.

### Objects With Keys but Empty-Looking Values

An object like `{"a": null}` is not empty because it still contains a key-value pair. Some incorrect solutions may mistakenly treat `null` values as absent entries. The implementation avoids this mistake because it checks the number of keys rather than the stored values.

### Very Large Inputs

The constraints allow large JSON structures. A brute-force traversal wastes time by scanning every entry manually. The optimized solution relies on built-in size information and efficient key retrieval, making it concise and performant even for large inputs.
