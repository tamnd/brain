# LeetCode 2704 - To Be Or Not To Be

## Problem Understanding

This problem asks us to implement a small testing utility similar to what is commonly found in unit testing frameworks.

We need to write a function called `expect` that accepts any value `val`. Instead of returning a simple value, it returns an object containing two methods:

- `toBe(expected)`
- `notToBe(expected)`

The behavior of these methods is based on JavaScript's strict equality operators.

The `toBe(expected)` method should check whether the original value and `expected` are strictly equal using `===`. If they are equal, the method returns `true`. Otherwise, it throws the error `"Not Equal"`.

The `notToBe(expected)` method performs the opposite check. It verifies that the two values are not strictly equal using `!==`. If they are different, it returns `true`. If they are equal, it throws the error `"Equal"`.

The input examples represent functions being executed. The online judge captures either the returned value or the thrown error and compares it against the expected output.

Because the problem only requires two equality checks, there are no large input sizes, arrays, graphs, or other complex data structures involved. The task is primarily about correctly implementing the required API and handling equality conditions exactly as specified.

Several edge cases are important:

- Comparing identical primitive values such as `5` and `5`.
- Comparing different values such as `5` and `null`.
- Comparing boolean values.
- Comparing reference types, where strict equality checks whether both variables refer to the exact same object.
- Ensuring the correct error message is thrown, since even a small difference in the string would cause the solution to fail.

## Approaches

### Brute Force Approach

The most direct solution is to implement each method independently.

When `toBe` is called, we compare the stored value with the provided argument. If they are equal, we return `true`; otherwise, we throw `"Not Equal"`.

Similarly, when `notToBe` is called, we compare the values again. If they are different, we return `true`; otherwise, we throw `"Equal"`.

Since only a single comparison is required for each method call, there is no repeated work or inefficiency.

### Key Insight

The key observation is that the returned object only needs access to the original value passed into `expect`.

This makes closures a natural fit. The `expect` function stores the original value, and the returned methods can access it later whenever they are invoked.

Each method performs exactly one equality comparison and then either returns `true` or throws the required error.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(1) | O(1) | Direct comparison inside each method |
| Optimal | O(1) | O(1) | Uses closure to retain the original value and perform a single comparison |

In practice, the brute-force and optimal approaches are effectively the same because the problem itself is extremely small.

## Algorithm Walkthrough

1. Define the function `expect(val)` and store the provided value.
2. Return an object containing two methods, `toBe` and `notToBe`.
3. When `toBe(expected)` is called, compare `val` and `expected` using strict equality.
4. If the values are equal, return `true`.
5. Otherwise, throw the error `"Not Equal"`.
6. When `notToBe(expected)` is called, compare `val` and `expected` using strict inequality.
7. If the values are different, return `true`.
8. Otherwise, throw the error `"Equal"`.

### Why it works

The correctness follows directly from the specification. The closure preserves the original value supplied to `expect`. Each method performs exactly the comparison required by the problem statement and produces the required result for both the success and failure cases. Therefore, every valid invocation behaves exactly as specified.

## Python Solution

Although LeetCode expects a JavaScript implementation for this problem, the following Python version demonstrates the same logic.

```python
from typing import Any, Callable

class Expect:
    def __init__(self, value: Any):
        self.value = value

    def toBe(self, expected: Any) -> bool:
        if self.value == expected:
            return True
        raise Exception("Not Equal")

    def notToBe(self, expected: Any) -> bool:
        if self.value != expected:
            return True
        raise Exception("Equal")

def expect(val: Any) -> Expect:
    return Expect(val)
```

The implementation stores the original value inside an `Expect` object. The `toBe` method checks equality and raises an exception with the required message if the comparison fails. The `notToBe` method performs the opposite check and raises `"Equal"` when the values match.

Both methods execute exactly one comparison and therefore directly correspond to the algorithm described earlier.

## Go Solution

```go
package main

import "errors"

type Expect struct {
	value interface{}
}

func expect(val interface{}) *Expect {
	return &Expect{value: val}
}

func (e *Expect) ToBe(expected interface{}) (bool, error) {
	if e.value == expected {
		return true, nil
	}
	return false, errors.New("Not Equal")
}

func (e *Expect) NotToBe(expected interface{}) (bool, error) {
	if e.value != expected {
		return true, nil
	}
	return false, errors.New("Equal")
}
```

The Go implementation uses a struct to store the original value and methods attached to that struct to perform the comparisons. Unlike JavaScript, Go does not throw exceptions for ordinary control flow, so errors are returned explicitly as `error` values.

## Edge Cases

One important edge case occurs when the two values are exactly the same. In this situation, `toBe` must return `true`, while `notToBe` must throw `"Equal"`. It is easy to accidentally reverse the logic or return the wrong error message. The implementation explicitly handles this case through the equality comparison.

Another important edge case involves completely different types, such as comparing `5` and `null`. Since strict equality requires both value and type compatibility, these values are not equal. The implementation correctly throws `"Not Equal"` for `toBe` and returns `true` for `notToBe`.

A third edge case involves reference types. In JavaScript, strict equality for objects checks whether both references point to the exact same object. Two separate objects with identical contents are not considered equal. Because the solution relies directly on the language's equality operator, it naturally follows the correct reference comparison semantics.

A final edge case concerns error handling. The online judge expects the exact strings `"Not Equal"` and `"Equal"`. Any variation in capitalization, spacing, or wording would fail. The implementation uses these exact messages, ensuring compliance with the specification.
