# LeetCode 2784 - Check if Array is Good

## Problem Understanding

This problem asks us to determine whether a given array `nums` is a permutation of a very specific array structure called `base[n]`.

The definition of `base[n]` is:

```
[1, 2, 3, ..., n - 1, n, n]
```

This means:

- Every integer from `1` to `n - 1` appears exactly once.
- The integer `n` appears exactly twice.
- The total length of the array is `n + 1`.

For example:

```
base[1] = [1, 1]
base[2] = [1, 2, 2]
base[3] = [1, 2, 3, 3]
base[4] = [1, 2, 3, 4, 4]
```

The given array can be in any order because the problem only requires it to be a permutation of `base[n]`.

The key observation is that the largest value in a valid array must be `n`. Since `n` appears in the array, the maximum element uniquely determines which `base[n]` we should compare against.

For example, if the maximum value in `nums` is `5`, then the only possible valid target array is:

```
[1, 2, 3, 4, 5, 5]
```

No other value of `n` can work.

The constraints are very small:

- `1 <= nums.length <= 100`
- `1 <= nums[i] <= 200`

Because the input size is tiny, even sorting-based solutions are easily fast enough. We do not need sophisticated optimization techniques.

Several edge cases are important:

- The smallest valid array is `[1,1]`.
- Arrays with the wrong length for their maximum value must be rejected.
- Arrays where a value less than `n` appears multiple times must be rejected.
- Arrays missing a required value between `1` and `n-1` must be rejected.
- Arrays containing values greater than `n` cannot be valid because `n` is defined as the maximum element.

## Approaches

### Brute Force Approach

A straightforward approach is to explicitly construct the expected `base[n]` array and compare it with the input.

First, find the maximum value `n` in `nums`. Then construct:

```
[1, 2, ..., n - 1, n, n]
```

Sort both arrays and compare them element by element.

This works because two arrays are permutations of each other if and only if their sorted versions are identical.

The approach is correct because the constructed array represents the only possible valid target. If the sorted arrays match, then `nums` is exactly a permutation of `base[n]`. Otherwise it is not.

### Key Insight

The maximum element completely determines the expected structure.

If `n = max(nums)`, then a valid array must satisfy two conditions:

1. Its length must be exactly `n + 1`.
2. After sorting, it must equal:

```
[1, 2, ..., n - 1, n, n]
```

Since the constraints are small, sorting provides a clean and efficient solution.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) | O(n) | Construct expected array and compare sorted versions |
| Optimal | O(n log n) | O(1) or O(n) | Sort and verify expected pattern directly |

Although both have the same asymptotic complexity, the optimal version avoids constructing a separate target array and directly validates the sorted array.

## Algorithm Walkthrough

1. Sort the input array.
2. Let `n` be the largest element, which is the last element of the sorted array.
3. Check whether the array length is exactly `n + 1`. If not, return `False` immediately because a valid `base[n]` always has length `n + 1`.
4. Verify that the first `n - 1` positions contain exactly:

```
1, 2, 3, ..., n - 1
```

This ensures each value from `1` to `n - 1` appears exactly once.
5. Verify that the final two elements are both equal to `n`.
6. If every check succeeds, return `True`.

### Why it works

After sorting, every permutation of `base[n]` becomes:

```
[1, 2, ..., n - 1, n, n]
```

The algorithm checks exactly this structure. The length check guarantees the correct number of elements, the first section check guarantees every value from `1` through `n - 1` appears exactly once, and the final check guarantees exactly two occurrences of `n`. Therefore the algorithm accepts precisely the valid arrays and rejects all others.

## Python Solution

```python
from typing import List

class Solution:
    def isGood(self, nums: List[int]) -> bool:
        nums.sort()

        n = nums[-1]

        if len(nums) != n + 1:
            return False

        for i in range(n - 1):
            if nums[i] != i + 1:
                return False

        return nums[-2] == n and nums[-1] == n
```

The implementation begins by sorting the array. Once sorted, the largest element becomes the final element, which gives us the candidate value `n`.

The next step verifies the required length. A valid `base[n]` always contains `n + 1` elements. If the lengths differ, the answer is immediately `False`.

The loop checks that the first portion of the sorted array contains exactly the values `1` through `n - 1` in order. Any duplicate, missing value, or unexpected number causes the check to fail.

Finally, the last two elements must both equal `n`. If they do, the array matches the required pattern and the function returns `True`.

## Go Solution

```go
import "sort"

func isGood(nums []int) bool {
	sort.Ints(nums)

	n := nums[len(nums)-1]

	if len(nums) != n+1 {
		return false
	}

	for i := 0; i < n-1; i++ {
		if nums[i] != i+1 {
			return false
		}
	}

	return nums[len(nums)-2] == n && nums[len(nums)-1] == n
}
```

The Go implementation follows the same logic as the Python version. The array is sorted using `sort.Ints`, the maximum value is obtained from the last position, and the required pattern is verified directly.

No special handling for integer overflow is necessary because all values are very small. Go slices are modified in place during sorting, which keeps the implementation concise.

## Edge Cases

### The Smallest Valid Array

The array `[1,1]` corresponds to `base[1]`. This case is easy to mishandle because there are no values in the range `1` through `n - 1`. The implementation handles it naturally because the verification loop executes zero iterations when `n = 1`.

### Incorrect Length Despite Correct Values

Consider:

```
[2,1,3]
```

The values appear reasonable, but `base[3]` must contain four elements, not three. A solution that only checks frequencies might incorrectly accept such inputs. The explicit length check prevents this error.

### Missing Intermediate Values

Consider:

```
[1,4,4,2,2]
```

The maximum value suggests `n = 4`, but the value `3` is missing. A naive implementation that only checks the maximum value and duplicate count could incorrectly return `True`. The sorted pattern verification guarantees every number from `1` through `n - 1` appears exactly once.

### Too Many Copies of the Maximum Value

Consider:

```
[1,2,3,3,3]
```

The value `3` appears three times instead of two. After sorting, the length and ordering checks fail because a valid `base[3]` must contain exactly four elements. The implementation therefore correctly rejects the input.

### Duplicate Values Below n

Consider:

```
[1,1,2,3,3]
```

The value `1` appears twice while `2` appears only once. Sorting produces:

```
[1,1,2,3,3]
```

During verification, the second position is expected to contain `2` but contains `1`, causing the algorithm to return `False`. This ensures every value below `n` appears exactly once.
