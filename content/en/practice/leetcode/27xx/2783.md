## Solution

## Problem Understanding

This problem asks us to analyze flight bookings and determine two values for every flight:

1. How many passengers successfully received a seat on the flight.
2. How many passengers ended up on the waitlist because the flight exceeded capacity.

We are given two database tables.

The `Flights` table stores information about each flight:

| Column | Meaning |
| --- | --- |
| `flight_id` | Unique identifier of the flight |
| `capacity` | Maximum number of passengers that can be seated |

The `Passengers` table stores ticket bookings:

| Column | Meaning |
| --- | --- |
| `passenger_id` | Unique identifier of the passenger |
| `flight_id` | Flight booked by the passenger |

The important detail is that every passenger attempts to reserve a seat on a flight. If the number of passengers booked on a flight is less than or equal to the flight capacity, everyone gets a confirmed seat. If the number of bookings exceeds the capacity, only up to `capacity` passengers are considered booked successfully, and the remaining passengers are waitlisted.

For every flight, we must return:

| Column | Meaning |
| --- | --- |
| `flight_id` | Flight identifier |
| `booked_cnt` | Number of passengers who got seats |
| `waitlist_cnt` | Number of passengers placed on the waitlist |

The result must be sorted by `flight_id` in ascending order.

A key observation is that the problem does not care which passengers specifically got seats. It only asks for counts. That means we only need the total number of passengers per flight.

The constraints imply a relational aggregation problem. We are essentially comparing:

- total passengers booked for a flight
- maximum seats available for that flight

The core logic becomes:

```
booked_cnt = min(total_passengers, capacity)
waitlist_cnt = max(total_passengers - capacity, 0)
```

Several edge cases are important:

- A flight may have fewer passengers than capacity.
- A flight may have exactly as many passengers as capacity.
- A flight may have more passengers than capacity.
- A flight may have no passengers at all.
- A naive query could accidentally lose flights with zero passengers if it uses an inner join instead of a left join.

The problem guarantees that `flight_id` is unique in the `Flights` table and `passenger_id` is unique in the `Passengers` table, so there are no duplicate rows to handle.

## Approaches

### Brute Force Approach

A brute force solution would process each flight independently and repeatedly scan the entire `Passengers` table to count how many passengers booked that flight.

For every row in `Flights`, we would:

1. Traverse all rows in `Passengers`.
2. Count how many passengers belong to the current flight.
3. Compare the count with the flight capacity.
4. Compute booked and waitlisted counts.

This approach is correct because it explicitly counts all passengers for every flight. However, it is inefficient because the `Passengers` table is scanned repeatedly.

If there are `F` flights and `P` passengers, the total complexity becomes:

```
O(F × P)
```

This becomes unnecessarily expensive for large datasets.

### Optimal Approach

The key insight is that we only need the total passenger count per flight once.

Instead of rescanning the `Passengers` table for every flight, we can aggregate passenger counts using `GROUP BY`.

The optimal strategy is:

1. Group passengers by `flight_id`.
2. Count passengers for each flight.
3. Join the aggregated result with the `Flights` table.
4. Use simple arithmetic to compute booked and waitlisted counts.

This dramatically improves efficiency because the passenger table is processed only once.

In SQL, aggregation naturally solves this type of counting problem efficiently.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(F × P) | O(1) | Repeatedly scans all passengers for every flight |
| Optimal | O(F + P) | O(F) | Uses aggregation and joining efficiently |

## Algorithm Walkthrough

1. Start with the `Flights` table because every flight must appear in the final output, even if no passengers booked it.
2. Aggregate the `Passengers` table by `flight_id` using `GROUP BY`. This produces the total number of passengers for each flight.
3. Use a `LEFT JOIN` between `Flights` and the aggregated passenger counts. A left join is important because flights with zero passengers would disappear with an inner join.
4. For each joined row, compute the number of booked passengers using:

```
MIN(total_passengers, capacity)
```

This ensures the number of confirmed passengers never exceeds the available seats.

1. Compute the waitlist count using:

```
MAX(total_passengers - capacity, 0)
```

This ensures negative waitlist values never occur.

1. Use `COALESCE` to replace `NULL` passenger counts with `0` for flights that have no passengers.
2. Return the result ordered by `flight_id` in ascending order.

### Why it works

The algorithm works because every flight's result depends only on two values:

- the flight capacity
- the total number of passengers requesting seats

The aggregation step guarantees we count all passengers exactly once per flight. The formulas:

```
booked_cnt = min(passenger_count, capacity)
waitlist_cnt = max(passenger_count - capacity, 0)
```

correctly model the booking rules in all possible cases:

- under capacity
- exactly at capacity
- over capacity

Because every flight is preserved through the left join, the final result is complete and correct.

## Python Solution

```python
# Write your MySQL query statement below

SELECT
    f.flight_id,
    LEAST(COALESCE(p.passenger_count, 0), f.capacity) AS booked_cnt,
    GREATEST(COALESCE(p.passenger_count, 0) - f.capacity, 0) AS waitlist_cnt
FROM Flights f
LEFT JOIN (
    SELECT
        flight_id,
        COUNT(*) AS passenger_count
    FROM Passengers
    GROUP BY flight_id
) p
ON f.flight_id = p.flight_id
ORDER BY f.flight_id;
```

This solution begins by aggregating the `Passengers` table. The subquery groups rows by `flight_id` and counts the number of passengers booked on each flight.

The `LEFT JOIN` connects each flight with its passenger count. If a flight has no passengers, the joined count becomes `NULL`, so `COALESCE` converts it into `0`.

The `LEAST` function calculates the confirmed bookings. Even if more passengers booked than available seats, the booked count cannot exceed capacity.

The `GREATEST` function computes the waitlist count. If the passenger count is smaller than capacity, the expression would otherwise become negative, so `GREATEST(..., 0)` prevents invalid values.

Finally, the result is sorted by `flight_id` as required.

## Edge Cases

### Flights With No Passengers

A flight may exist in the `Flights` table without any corresponding rows in `Passengers`. This is a common source of bugs because an inner join would completely remove such flights from the result.

The implementation handles this correctly by using a `LEFT JOIN`. Missing passenger counts become `NULL`, and `COALESCE` converts them into `0`.

### Flights Exactly At Capacity

When the number of passengers equals the capacity, all passengers should be confirmed and nobody should be waitlisted.

This case is handled naturally because:

```
LEAST(capacity, capacity) = capacity
GREATEST(capacity - capacity, 0) = 0
```

The formulas work without requiring special conditional logic.

### Flights That Are Heavily Overbooked

A flight may have far more passengers than available seats. A naive implementation might accidentally allow booked counts to exceed capacity.

The implementation prevents this using:

```
LEAST(passenger_count, capacity)
```

No matter how many passengers book the flight, the booked count is capped at the flight capacity.
