# LeetCode 2757 - Generate Circular Array Values

## Problem Understanding

This problem asks us to implement a generator that traverses a circular array. Unlike a normal array where moving beyond either end would be invalid, this array wraps around. Moving forward past the last element returns to the first element, and moving backward past the first element returns to the last element.

The function receives two inputs:

- `arr`, the circular array.
- `startIndex`, the index where the generator should begin.

The generator behaves as follows:

- The first call to `next()` yields `arr[startIndex]`.
- Every subsequent call passes a value named `jump`.
- A positive `jump` moves forward through the circular array.
- A negative `jump` moves backward through the circular array.
- A jump of zero leaves the current position unchanged.
- After updating the position, the generator yields the value at the new index.

The key observation is that the movement is circular. For example, in an array of length 5:

- Moving forward 6 positions is equivalent to moving forward 1 position.
- Moving backward 7 positions is equivalent to moving backward 2 positions.

The constraints tell us:

- `arr.length` can be as large as `10^4`.
- Individual jumps can be as large as `10^4` in magnitude.
- The number of generator calls is relatively small.

Even though the constraints are not enormous, repeatedly simulating each individual step inside a jump would be inefficient and unnecessary. We can compute the destination index directly using modular arithmetic.

Important edge cases include:

- An array containing only one element.
- Very large positive jumps.
- Very large negative jumps.
- A jump value of zero.
- Jumps whose magnitude is much larger than the array length.

The problem guarantees that:

- The array contains at least one element.
- `startIndex` is always a valid index.

## Approaches

### Brute Force Approach

A straightforward solution is to simulate every movement one step at a time.

Whenever a jump value arrives:

- If the jump is positive, move forward one position repeatedly.
- If the jump is negative, move backward one position repeatedly.
- Wrap around whenever the beginning or end of the array is reached.

This approach is correct because it exactly follows the problem statement. However, it performs unnecessary work. For example, a jump of `10000` would require `10000` individual movements even though many complete cycles around the array contribute nothing to the final position.

### Key Insight

A circular array naturally corresponds to modular arithmetic.

Suppose:

- Current index = `i`
- Jump = `j`
- Array length = `n`

The new position is simply:

$$(i + j) \bmod n$$

Python's modulo operation already handles negative values correctly.

Instead of performing every intermediate movement, we directly compute the final index after the jump.

This reduces each generator update from potentially thousands of operations to constant time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O( | jump | ) per call |
| Optimal | O(1) per call | O(1) | Uses modular arithmetic to jump directly |

## Algorithm Walkthrough

1. Store the array length `n`.
2. Initialize the current position as `startIndex`.
3. Yield `arr[currentIndex]` immediately when the generator starts.
4. Whenever a new jump value is received through `yield`, update the position using:

$$currentIndex = (currentIndex + jump) \bmod n$$
5. Yield the value located at the updated index.
6. Continue indefinitely, repeating the update-and-yield process for every new jump.

### Why it works

The crucial invariant is that `currentIndex` always represents the correct position in the circular array immediately before a value is yielded.

Modular arithmetic exactly models circular movement. Adding a jump computes the desired displacement, while taking modulo `n` wraps the index into the valid range `[0, n-1]`. Therefore every generator output matches the position that would be reached by explicitly simulating the movement step by step.

## Python Solution

```python
from typing import List, Generator

class Solution:
    def cycleGenerator(
        self,
        arr: List[int],
        startIndex: int
    ) -> Generator[int, int, None]:
        n = len(arr)
        index = startIndex

        jump = yield arr[index]

        while True:
            index = (index + jump) % n
            jump = yield arr[index]
```

### Implementation Explanation

The generator begins by storing the array length and the starting position.

The statement:

```
jump = yield arr[index]
```

serves two purposes. First, it yields the initial value. Second, when the next call to `next(value)` occurs, the supplied value becomes the new value of `jump`.

Inside the infinite loop, the current position is updated using:

```
index = (index + jump) % n
```

This handles:

- Forward movement
- Backward movement
- Wrapping around the array
- Large jumps

After computing the new index, the generator yields the corresponding array value and waits for the next jump.

Because each update requires only a single arithmetic operation, every generator step runs in constant time.

## Go Solution

```go
func cycleGenerator(arr []int, startIndex int) func(int) int {
	n := len(arr)
	index := startIndex
	firstCall := true

	return func(jump int) int {
		if firstCall {
			firstCall = false
			return arr[index]
		}

		index = ((index + jump) % n + n) % n
		return arr[index]
	}
}
```

### Go-specific Notes

Go does not support JavaScript-style generators natively. The implementation above uses a closure that preserves state between calls.

One important difference is modulo behavior. In Python:

```
(-1) % 5 == 4
```

However in Go:

```
(-1) % 5 == -1
```

Therefore the expression:

```
((index + jump) % n + n) % n
```

is used to ensure the resulting index is always non-negative.

## Edge Cases

### Array of Length One

When the array contains only one element, every possible jump should lead back to the same position. A common bug is assuming movement always changes the index. Since `(0 + jump) % 1` is always `0`, the implementation naturally handles this case without special logic.

### Very Large Positive Jumps

A naive implementation might iterate through every intermediate position, causing unnecessary work. For example, jumping `10000` positions in an array of length `4` ultimately returns to the same index because `10000 % 4 = 0`. The modular arithmetic approach computes this instantly.

### Very Large Negative Jumps

Negative movement often introduces bugs because different languages handle modulo differently. Python's modulo operator always returns a valid non-negative index, so expressions such as:

```
(0 + (-1)) % 5
```

correctly evaluate to `4`. This guarantees proper backward wrapping even for extremely large negative jumps.

### Zero Jump

A jump value of zero should leave the current position unchanged. Some implementations accidentally advance the pointer regardless of the jump value. Since the formula becomes:

```
index = (index + 0) % n
```

the current index remains exactly where it was, producing the correct result.

### Starting at Either End of the Array

Starting at index `0` or at index `n - 1` can expose off-by-one errors. Because all movement is expressed through modular arithmetic, wrapping from the first element to the last element and vice versa is handled uniformly and correctly.
