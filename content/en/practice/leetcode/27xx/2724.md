# LeetCode 2724 - Sort By

## Problem Understanding

The problem gives us two inputs: an array `arr` and a function `fn`.

Our goal is to return a new ordering of the array such that the elements are sorted in ascending order according to the numeric value produced by applying `fn` to each element.

In other words, every element in the array has an associated key:

```
key = fn(element)
```

The array must then be sorted based on these keys from smallest to largest.

The elements themselves can be any valid JSON values. They might be numbers, objects, arrays, strings, or other structures. The important detail is that `fn` always returns a number, and that number determines the ordering.

For example:

```
arr = [5, 4, 1, 2, 3]
fn(x) = x
```

The keys are simply:

```
5 -> 5
4 -> 4
1 -> 1
2 -> 2
3 -> 3
```

Sorting by those keys produces:

```
[1, 2, 3, 4, 5]
```

The constraints allow up to:

```
arr.length <= 500,000
```

This is a very large array. Any solution that repeatedly scans the array or performs quadratic work will be far too slow. We need an efficient sorting strategy.

The problem also guarantees an important property:

```
fn will never duplicate numbers for a given array
```

This means every element has a unique sorting key. There are no ties, so the final ordering is completely determined.

Some important edge cases include arrays with a single element, arrays already sorted according to `fn`, arrays sorted in reverse order, arrays containing negative numbers, and arrays containing complex objects or nested arrays. The guarantee that all keys are unique removes the need to handle equal-key ordering.

## Approaches

### Brute Force Approach

A straightforward way to solve the problem is to repeatedly find the smallest remaining element according to `fn`.

For each position in the result:

1. Scan all remaining elements.
2. Compute `fn` for each one.
3. Find the smallest key.
4. Remove that element and append it to the answer.

This approach is essentially selection sort.

It works because at every step we choose the smallest remaining key, which is exactly how ascending order is defined.

However, each selection requires scanning many remaining elements. For an array of size `n`, we perform approximately:

```
n + (n - 1) + (n - 2) + ... + 1
```

comparisons, which is:

```
O(n²)
```

With up to 500,000 elements, this is completely impractical.

### Optimal Approach

The key observation is that the problem is simply asking us to sort the array according to a custom key function.

Modern programming languages already provide highly optimized sorting algorithms. Instead of manually searching for the next smallest element, we can directly tell the sorting routine to use `fn(element)` as the comparison key.

The sorting algorithm computes the ordering based on these keys and rearranges the elements accordingly.

Since comparison-based sorting requires:

```
O(n log n)
```

time in the general case, this is dramatically faster than the quadratic brute-force approach.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) or O(n) | Repeatedly find the smallest remaining key |
| Optimal | O(n log n) | O(log n) to O(n), depending on implementation | Use built-in sort with `fn` as the key |

## Algorithm Walkthrough

1. Receive the array `arr` and the function `fn`.
2. For every element, define its sorting key as `fn(element)`.
3. Invoke the language's built-in sorting routine.
4. Configure the sort so that comparisons are based on the numeric key returned by `fn`.
5. The sorting algorithm rearranges elements into ascending order of those keys.
6. Return the sorted array.

### Why it works

Every element is assigned a unique numeric key through `fn`. The sorting algorithm guarantees that elements are arranged in ascending order of these keys. Since the problem defines the desired order exactly as ascending `fn` output, the resulting array satisfies the specification. The uniqueness guarantee ensures there are no ambiguous ties.

## Python Solution

```python
from typing import List, Any, Callable

class Solution:
    def sortBy(self, arr: List[Any], fn: Callable[[Any], int]) -> List[Any]:
        return sorted(arr, key=fn)
```

The implementation is very concise because Python's built-in `sorted()` function already supports custom sorting keys.

The parameter `key=fn` tells Python to evaluate `fn(element)` for each element and use that value when determining order.

Python's sorting implementation is highly optimized and runs in `O(n log n)` time. The returned list contains the same elements as the input array, but arranged according to the numeric values produced by `fn`.

## Go Solution

```go
import "sort"

func sortBy(arr []interface{}, fn func(interface{}) int) []interface{} {
	sort.Slice(arr, func(i, j int) bool {
		return fn(arr[i]) < fn(arr[j])
	})
	return arr
}
```

In Go, the standard library provides `sort.Slice`, which accepts a custom comparison function.

The comparison function receives indices `i` and `j` and returns whether element `i` should appear before element `j`.

Unlike Python's `sorted()`, which returns a new list, `sort.Slice` sorts the slice in place. After sorting, the same slice is returned.

There are no integer overflow concerns because the comparison only evaluates the ordering of the values returned by `fn`.

## Edge Cases

### Single Element Array

An array containing only one element is already sorted regardless of the key function. A buggy implementation might assume at least two elements exist and attempt invalid comparisons. The built-in sorting routine naturally handles this case and returns the element unchanged.

### Negative Key Values

The function `fn` may return negative numbers. Some incorrect implementations accidentally assume keys are non-negative or use them as array indices. Since the solution relies solely on numerical comparison, negative values are handled correctly and appear before larger values.

### Complex JSON Objects

The array elements themselves may not be directly comparable. For example, objects and nested arrays cannot generally be ordered using normal comparison operators. The problem avoids this issue by providing a numeric key function. The implementation never compares the raw elements directly, it only compares the values returned by `fn`.

### Already Sorted or Reverse Sorted Input

Arrays may already be in the correct order or may be completely reversed. The algorithm does not rely on any existing ordering and correctly produces the ascending order defined by the key function in either situation.

### Large Input Size

With up to 500,000 elements, quadratic algorithms become infeasible. The use of the language's optimized sorting implementation ensures the solution scales efficiently and remains practical for the largest allowed inputs.
