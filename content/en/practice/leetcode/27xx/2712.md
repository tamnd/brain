## Solution

## Problem Understanding

We are given a binary string `s` consisting only of `'0'` and `'1'`. The goal is to make every character in the string equal, meaning the final string must contain either all `'0'` characters or all `'1'` characters.

We are allowed to perform two kinds of operations:

1. Choose an index `i` and flip every character from index `0` through `i`. This is a prefix flip, and it costs `i + 1`.
2. Choose an index `i` and flip every character from index `i` through `n - 1`. This is a suffix flip, and it costs `n - i`.

Flipping means changing `'0'` to `'1'` and `'1'` to `'0'`.

The important detail is that the cost depends on how many characters are included in the operation. Smaller ranges are cheaper, which means we want to avoid unnecessary large flips.

The input size can be as large as `10^5`, which immediately tells us that quadratic solutions will be too slow. We need an algorithm that runs in linear time.

The key observation is that we only care about positions where adjacent characters differ. If two neighboring characters are already equal, we do not need to separate them with an operation. Every transition between `'0'` and `'1'` creates a boundary that eventually must be resolved.

Some important edge cases include:

- A string of length `1`, which is already uniform.
- A string where all characters are already equal, which should return `0`.
- Alternating strings like `"010101"`, which maximize the number of transitions.
- Strings where the cheapest operation is not obvious because prefix and suffix costs differ.

## Approaches

### Brute Force Approach

A brute force solution would try to simulate all possible sequences of prefix and suffix flips.

One possible interpretation is to treat every state of the string as a node and attempt all valid operations recursively or with breadth first search. Since each operation changes multiple bits, the number of reachable states grows exponentially.

Another brute force style approach would attempt dynamic programming over all possible transformed strings. Since a binary string of length `n` has `2^n` possible states, this quickly becomes infeasible.

The brute force method is correct because it explores all possible ways to transform the string and eventually finds the minimum total cost. However, it is far too slow for `n = 10^5`.

### Key Insight

The important observation is that we only need to care about positions where adjacent characters differ.

Suppose `s[i] != s[i - 1]`. At some point, this boundary must disappear. To eliminate it, we must flip either:

- the prefix ending at `i - 1`, costing `i`, or
- the suffix starting at `i`, costing `n - i`

We always choose the cheaper option:

$$\min(i, n - i)$$

Each transition can be handled independently because removing one boundary does not create new unavoidable boundaries elsewhere.

Therefore, the total minimum cost is simply:

$$\sum_{i=1}^{n-1}
[s[i] \neq s[i-1]] \cdot \min(i, n-i)$$

This leads to a clean greedy solution in linear time.

## Algorithm Walkthrough

1. Initialize a variable `total_cost = 0`.
2. Iterate through the string starting from index `1`.
3. For every index `i`, compare `s[i]` with `s[i - 1]`.
4. If the characters are equal, nothing needs to be done because there is no boundary between different groups.
5. If the characters are different, then this boundary must eventually be removed.
6. There are exactly two efficient ways to remove the boundary:

- Flip the prefix ending at `i - 1`, which costs `i`
- Flip the suffix starting at `i`, which costs `n - i`
7. Add the smaller of these two costs to `total_cost`.
8. Continue until the end of the string.
9. Return `total_cost`.

### Why it works

Every place where adjacent characters differ represents a required transition that must be eliminated before the entire string becomes uniform. A flip operation can remove such a boundary from either the left side or the right side. Since the boundary can always be resolved independently with the cheaper option, summing the minimum possible cost for each transition produces the globally optimal answer.

## Python Solution

```python
class Solution:
    def minimumCost(self, s: str) -> int:
        n = len(s)
        total_cost = 0

        for i in range(1, n):
            if s[i] != s[i - 1]:
                total_cost += min(i, n - i)

        return total_cost
```

The implementation follows the greedy observation directly.

First, we store the length of the string in `n`. Then we initialize `total_cost` to accumulate the answer.

The loop starts from index `1` because every comparison needs both the current character and the previous character.

Whenever `s[i] != s[i - 1]`, we have found a transition boundary between groups of different bits. We compute the cheaper way to eliminate that boundary:

- `i` represents the cost of flipping the prefix `[0, i - 1]`
- `n - i` represents the cost of flipping the suffix `[i, n - 1]`

We add the minimum of these two values to the running total.

Finally, we return the accumulated cost.

## Go Solution

```go
func minimumCost(s string) int64 {
    n := len(s)
    var totalCost int64 = 0

    for i := 1; i < n; i++ {
        if s[i] != s[i-1] {
            leftCost := i
            rightCost := n - i

            if leftCost < rightCost {
                totalCost += int64(leftCost)
            } else {
                totalCost += int64(rightCost)
            }
        }
    }

    return totalCost
}
```

The Go implementation mirrors the Python solution closely.

One important difference is the use of `int64` for the return value and accumulated cost. Since the maximum possible total cost can exceed the range of a 32 bit integer for large inputs, using `int64` ensures correctness.

Strings in Go are indexed directly as bytes, which works here because the input only contains ASCII characters `'0'` and `'1'`.

## Edge Cases

### Single Character Strings

A string of length `1` is already uniform because there is only one character. A careless implementation might attempt to access adjacent elements and cause index errors. This implementation safely starts iterating from index `1`, so the loop never runs for length `1`.

### Already Uniform Strings

Strings like `"00000"` or `"11111"` contain no transitions. Since the algorithm only adds cost when adjacent characters differ, the total cost correctly remains `0`.

### Alternating Strings

Strings like `"010101"` contain the maximum possible number of transitions. These are important because they produce the largest costs and stress the correctness of the greedy logic. The implementation handles them naturally by processing every transition independently.

### Transitions Near the Ends

A boundary near the beginning or end of the string has a very cheap operation available. For example, in `"011111"`, flipping the first character costs only `1`. An incorrect solution might always choose prefix flips or suffix flips uniformly. This implementation correctly takes `min(i, n - i)` for every transition, ensuring the cheaper side is always selected.
