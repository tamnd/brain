## Solution

## Problem Understanding

We are given a string `s` consisting only of lowercase English letters. We must perform exactly one operation:

1. Choose any non-empty substring.
2. Replace every character in that substring with the previous character in the alphabet.

- `'b'` becomes `'a'`
- `'c'` becomes `'b'`
- `'a'` wraps around and becomes `'z'`

After performing this operation once, we must return the lexicographically smallest possible resulting string.

A lexicographically smaller string is one that becomes smaller at the first position where two strings differ. For example:

- `"abc"` is smaller than `"abd"`
- `"aac"` is smaller than `"abc"`

This means earlier characters are much more important than later characters. Improving a character near the front of the string usually matters more than improving many characters later.

The input size can be as large as `3 * 10^5`, so we cannot afford expensive algorithms that try all substrings explicitly. Any quadratic solution would be too slow.

An important detail is the transformation rule for `'a'`. Decreasing `'a'` produces `'z'`, which is much larger lexicographically. Therefore, changing `'a'` is usually harmful unless we are forced to do so.

Several edge cases are especially important:

- Strings made entirely of `'a'`
- Strings beginning with many `'a'`
- Strings with no `'a'`
- Single-character strings
- Choosing whether to stop extending the substring once an `'a'` is encountered

These cases determine the greedy strategy.

## Approaches

### Brute Force Approach

The most direct solution is to try every possible substring.

For every pair `(l, r)`:

1. Copy the string.
2. Apply the operation to substring `s[l:r+1]`.
3. Compare the resulting string with the current best answer.
4. Keep the lexicographically smallest result.

This approach is correct because it exhaustively checks all valid operations.

However, the number of substrings is `O(n^2)`. Constructing each modified string costs `O(n)`, so the total complexity becomes `O(n^3)`.

With `n` up to `3 * 10^5`, this is completely infeasible.

### Key Insight for the Optimal Approach

To minimize a string lexicographically, we should improve characters as early as possible.

Decreasing any character except `'a'` makes it smaller, which helps lexicographically. But decreasing `'a'` turns it into `'z'`, which hurts badly.

Therefore:

- We should skip leading `'a'` characters.
- Once we encounter the first non-`'a'` character, we should decrease it.
- We should continue decreasing consecutive non-`'a'` characters.
- We should stop immediately when we encounter an `'a'`.

Why stop at `'a'`?

Because turning `'a'` into `'z'` makes the string larger. Also, continuing beyond that point would force us to include the `'a'` inside the chosen substring.

If the entire string consists of `'a'`, we cannot avoid converting at least one `'a'` into `'z'` because the operation must be applied to a non-empty substring. In that special case, the optimal choice is to change only the last character.

Changing the last character minimizes the damage because later positions matter less lexicographically.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n³) | O(n) | Tries every substring and rebuilds strings |
| Optimal | O(n) | O(n) | Greedily modifies the first contiguous non-`'a'` segment |

## Algorithm Walkthrough

1. Convert the string into a mutable array of characters.

Strings are immutable in Python, so using a list allows efficient character updates.
2. Scan from left to right until finding the first character that is not `'a'`.

Leading `'a'` characters should not be modified because they would become `'z'`, making the string larger.
3. If every character is `'a'`, change the last character to `'z'` and return.

The operation must be performed at least once. Modifying the final character minimizes the lexicographic increase.
4. Starting from the first non-`'a'` character, continue moving right while characters are not `'a'`.

Every such character should be decreased by one because this makes the string lexicographically smaller.
5. Stop as soon as an `'a'` is encountered.

Including that `'a'` in the substring would turn it into `'z'`, which is harmful.
6. Convert the character array back into a string and return it.

### Why it works

The algorithm always improves the earliest possible position in the string. Since lexicographic order depends first on earlier positions, decreasing the first non-`'a'` character is always optimal.

After starting the operation, decreasing additional consecutive non-`'a'` characters can only help because they also become smaller. However, including an `'a'` would increase that position to `'z'`, making the result worse.

If no non-`'a'` characters exist, the operation is unavoidable, and modifying only the final character produces the smallest possible increase.

## Python Solution

```python
class Solution:
    def smallestString(self, s: str) -> str:
        chars = list(s)
        n = len(chars)

        index = 0

        # Skip leading 'a' characters
        while index < n and chars[index] == 'a':
            index += 1

        # Entire string is 'a'
        if index == n:
            chars[-1] = 'z'
            return ''.join(chars)

        # Decrease consecutive non-'a' characters
        while index < n and chars[index] != 'a':
            chars[index] = chr(ord(chars[index]) - 1)
            index += 1

        return ''.join(chars)
```

The implementation follows the greedy strategy directly.

First, the string is converted into a list so characters can be modified in place. The algorithm then skips all leading `'a'` characters because modifying them would produce `'z'`.

If the scan reaches the end of the string, then every character was `'a'`. Since the operation is mandatory, the last character is changed to `'z'`.

Otherwise, the algorithm begins decreasing consecutive non-`'a'` characters. Each character is shifted backward in the alphabet using ASCII arithmetic:

```python
chr(ord(chars[index]) - 1)
```

The loop stops immediately upon reaching an `'a'` because including it would worsen the result.

Finally, the modified character list is joined back into a string.

## Go Solution

```go
func smallestString(s string) string {
    chars := []byte(s)
    n := len(chars)

    index := 0

    // Skip leading 'a'
    for index < n && chars[index] == 'a' {
        index++
    }

    // Entire string is 'a'
    if index == n {
        chars[n-1] = 'z'
        return string(chars)
    }

    // Decrease consecutive non-'a' characters
    for index < n && chars[index] != 'a' {
        chars[index]--
        index++
    }

    return string(chars)
}
```

The Go implementation uses a byte slice because strings in Go are immutable.

Since lowercase English letters are ASCII characters, decrementing a character can be done directly with:

```go
chars[index]--
```

No special overflow handling is needed because the algorithm never decrements `'a'` inside the main modification loop.

The logic otherwise mirrors the Python solution exactly.

## Edge Cases

One important edge case is when the entire string consists only of `'a'` characters. A naive implementation might attempt to avoid modification entirely, but the problem requires exactly one non-empty substring operation. Since changing any `'a'` produces `'z'`, the best strategy is to modify only the last character. This minimizes the lexicographic damage because later positions matter less.

Another important case is a string with leading `'a'` characters followed by non-`'a'` characters, such as `"aaabc"`. Modifying the leading `'a'` characters would immediately worsen the string because `'a'` becomes `'z'`. The implementation correctly skips all leading `'a'` characters before starting the operation.

A third critical edge case is encountering an `'a'` inside the string after modification has already started. For example, in `"bcade"`, the optimal substring is only `"bc"`. If the algorithm continued through the `'a'`, that character would become `'z'`, producing a worse result. The implementation explicitly stops modification at the first `'a'`.

Single-character strings are also easy to mishandle. For `"a"`, the result must be `"z"` because the operation is mandatory. For `"b"`, the result becomes `"a"`. The implementation naturally handles both cases through the same logic without requiring special-case branches.
