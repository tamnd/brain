## Solution

## Problem Understanding

This problem is an interactive-style Union-Find problem where we are not given the categories directly. Instead, we are given access to a helper API:

```
haveSameCategory(a, b)
```

This function tells us whether two elements belong to the same category.

There are `n` elements labeled from `0` to `n - 1`. Every element belongs to exactly one category. Multiple elements may share the same category, or every element may have its own unique category.

Our task is to determine how many distinct categories exist among all `n` elements.

The important detail is that we do not know the category IDs themselves. We only know whether two elements are equivalent through pairwise comparisons. This makes the problem similar to discovering connected components in a graph, where elements in the same category are connected together.

For example, if:

```
0 and 1 are same
1 and 2 are same
```

then all three elements belong to one category, even if we never directly compare `0` and `2`.

The constraints are small:

```
1 <= n <= 100
```

This immediately tells us that an `O(n^2)` solution is perfectly acceptable. Since there are at most 10,000 pairwise comparisons, we can safely compare every pair of elements if needed.

An important observation is that categories form equivalence classes. The relation returned by `haveSameCategory` behaves like:

- Reflexive: an element is in the same category as itself
- Symmetric: if `a` matches `b`, then `b` matches `a`
- Transitive: if `a` matches `b` and `b` matches `c`, then `a` matches `c`

This structure is exactly what Union-Find, also called Disjoint Set Union or DSU, is designed to represent.

Several edge cases are important:

- `n = 1`, there is only one element, so the answer must be `1`
- Every element belongs to a unique category
- All elements belong to one category
- Categories may contain uneven numbers of elements
- Comparisons may only indirectly connect elements through transitivity

A naive implementation that only checks direct neighbors would fail on transitive relationships, so we need a structure that merges connected groups correctly.

## Approaches

### Brute Force Approach

A straightforward brute-force solution would attempt to explicitly build groups.

For every element, we could compare it with all previously seen elements. If it matches one of them, we place it into that category. Otherwise, we create a new category.

Conceptually, this works because every new element either belongs to an existing category or starts a new one.

However, maintaining categories manually becomes cumbersome because categories are discovered dynamically. If two previously separate groups later turn out to belong together, we would need to merge them carefully.

Although this approach can still work within constraints, the bookkeeping becomes error-prone and inefficient conceptually.

### Optimal Approach

The key insight is that this problem is fundamentally about grouping equivalent elements together.

Whenever:

```
haveSameCategory(i, j) == true
```

we know that `i` and `j` belong to the same connected component.

Union-Find is ideal here because it efficiently supports:

- Merging two groups
- Tracking which elements belong to the same group
- Counting the number of distinct connected components

We compare every pair `(i, j)` where `i < j`. If they belong to the same category, we union them together.

At the end, the number of unique parents in the DSU structure equals the number of unique categories.

Since `n <= 100`, the `O(n^2)` pairwise comparison cost is completely acceptable.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Manually maintains category groupings |
| Optimal | O(n² * α(n)) | O(n) | Uses Union-Find to merge equivalent elements efficiently |

Here, `α(n)` is the inverse Ackermann function, which is effectively constant in practice.

## Algorithm Walkthrough

1. Initialize a Union-Find data structure.

Each element initially belongs to its own category, so every node starts as its own parent.
2. Iterate through every pair of elements.

We use nested loops:

```
for i in range(n):
    for j in range(i + 1, n):
```

We only check pairs once because category equality is symmetric.
3. Query the category handler.

For every pair `(i, j)`, call:

```
categoryHandler.haveSameCategory(i, j)
```

If the result is `True`, the two elements belong to the same category.
4. Union matching elements.

When two elements share a category, merge their sets inside the DSU structure.

This ensures all connected elements eventually share the same root parent.
5. Count unique roots.

After processing all pairs, every category corresponds to one connected component.

We compute:

```
len(set(find(i) for i in range(n)))
```

The number of distinct roots equals the number of unique categories.

### Why it works

Union-Find maintains the invariant that all elements in the same connected component share the same representative root.

Whenever `haveSameCategory(i, j)` is true, we merge their sets. Because Union-Find preserves transitive connectivity, indirectly related elements also end up in the same group.

Therefore, after processing all valid pairwise relationships, each connected component corresponds exactly to one category. Counting unique roots gives the correct answer.

## Python Solution

```python
# Definition for a category handler.
# class CategoryHandler:
#     def haveSameCategory(self, a: int, b: int) -> bool:
#         pass

from typing import Optional

class Solution:
    def numberOfCategories(
        self,
        n: int,
        categoryHandler: Optional['CategoryHandler']
    ) -> int:

        parent = list(range(n))

        def find(x: int) -> int:
            if parent[x] != x:
                parent[x] = find(parent[x])
            return parent[x]

        def union(a: int, b: int) -> None:
            root_a = find(a)
            root_b = find(b)

            if root_a != root_b:
                parent[root_b] = root_a

        for i in range(n):
            for j in range(i + 1, n):
                if categoryHandler.haveSameCategory(i, j):
                    union(i, j)

        unique_roots = set()

        for i in range(n):
            unique_roots.add(find(i))

        return len(unique_roots)
```

The solution begins by initializing the `parent` array, where each element initially points to itself. This means every element starts as its own separate category.

The `find` function locates the representative root of a set. It also performs path compression, which flattens the tree structure and improves future lookup efficiency.

The `union` function merges two sets if their roots differ. We connect one root to the other so that both groups become part of the same connected component.

Next, the nested loops iterate through every unordered pair of elements. For each pair, we ask the handler whether the two elements belong to the same category.

Whenever the handler returns `True`, we union the two elements together.

Finally, we compute the set of distinct roots. Every unique root corresponds to one category, so the size of this set is the answer.

## Go Solution

```go
/**
 * Definition for a category handler.
 * type CategoryHandler interface {
 *     HaveSameCategory(int, int) bool
 * }
 */

func numberOfCategories(n int, categoryHandler CategoryHandler) int {
	parent := make([]int, n)

	for i := 0; i < n; i++ {
		parent[i] = i
	}

	var find func(int) int

	find = func(x int) int {
		if parent[x] != x {
			parent[x] = find(parent[x])
		}
		return parent[x]
	}

	union := func(a int, b int) {
		rootA := find(a)
		rootB := find(b)

		if rootA != rootB {
			parent[rootB] = rootA
		}
	}

	for i := 0; i < n; i++ {
		for j := i + 1; j < n; j++ {
			if categoryHandler.HaveSameCategory(i, j) {
				union(i, j)
			}
		}
	}

	uniqueRoots := make(map[int]bool)

	for i := 0; i < n; i++ {
		root := find(i)
		uniqueRoots[root] = true
	}

	return len(uniqueRoots)
}
```

The Go implementation follows the same algorithmic structure as the Python solution.

Instead of Python lists and sets, Go uses:

- A slice for the `parent` array
- A map to track unique roots

Go does not support nested named functions in exactly the same way as Python, so the `find` function is declared using a variable assignment with recursion through a closure.

No integer overflow concerns exist because `n <= 100`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n² * α(n)) | Every pair is checked once, DSU operations are nearly constant |
| Space | O(n) | Parent array and root tracking structures |

The nested loops examine all unordered pairs of elements, which gives `O(n²)` comparisons.

Each Union-Find operation uses path compression, making `find` and `union` effectively constant time in practice. Technically, the amortized complexity is `O(α(n))`.

The extra memory usage comes from the DSU parent array and the set or map storing unique roots.

## Edge Cases

One important edge case occurs when `n = 1`. In this scenario, there are no pairs to compare at all. A buggy implementation might incorrectly return `0` because no unions occur. Our implementation handles this correctly because the single element remains its own root, resulting in exactly one unique category.

Another important case is when all elements belong to distinct categories. Here, every call to `haveSameCategory` returns `False`. The DSU structure never performs any unions, so every node remains its own representative. Counting distinct roots correctly returns `n`.

A third important edge case occurs when all elements belong to the same category. In this case, many pair comparisons return `True`, and repeated unions occur between already connected elements. A fragile implementation might accidentally double-count groups or corrupt parent relationships. Union-Find naturally handles repeated merges safely because unions only occur when roots differ.

A subtle edge case involves non-contiguous category membership. For example:

```
[1,2,1,2]
```

Elements belonging to the same category are separated throughout the array. Any implementation relying only on adjacent comparisons would fail here. Our algorithm compares all pairs, ensuring all equivalent elements are discovered and merged correctly.
