# LeetCode 2730 - Find the Longest Semi-Repetitive Substring

## Problem Understanding

The problem gives us a string `s` consisting only of digits (`'0'` through `'9'`). We must find the length of the longest contiguous substring that is **semi-repetitive**.

A substring is considered semi-repetitive if it contains **at most one adjacent pair of equal digits**.

An adjacent pair occurs whenever two neighboring characters are the same. For example:

- `"1223"` contains one adjacent pair, `"22"`.
- `"111"` contains two adjacent pairs, because positions `(0,1)` form `"11"` and positions `(1,2)` form another `"11"`.
- `"5494"` contains no adjacent equal digits.

The goal is not to count pairs or return the substring itself. We only need to return the maximum possible length of a valid substring.

The constraints are quite small:

- `1 <= s.length <= 50`
- Digits only

Even though the input size is tiny, the problem is fundamentally a substring optimization problem, which naturally suggests a sliding window solution. It is usually best to develop the most efficient approach rather than relying on the small constraint.

Several edge cases are important:

- A string of length 1 is always semi-repetitive.
- A string with no adjacent equal digits is entirely valid.
- A string like `"111111"` contains many overlapping adjacent pairs.
- Multiple adjacent-pair locations may appear in the string, requiring us to shrink the window correctly.
- The longest valid substring may occur in the middle rather than at the beginning or end.

## Approaches

### Brute Force

A straightforward approach is to generate every possible substring and check whether it is semi-repetitive.

For each substring, we scan through its characters and count how many adjacent equal pairs exist. If the count is at most one, the substring is valid, and we update the answer.

This approach is correct because every possible substring is examined. Therefore, the longest valid one cannot be missed.

However, there are `O(n²)` substrings, and checking each substring takes `O(n)` time. This leads to a total complexity of `O(n³)`.

### Key Observation

A substring becomes invalid only when it contains **more than one adjacent equal pair**.

This property is ideal for a sliding window.

As we extend the right side of the window:

- We can immediately detect whether a new adjacent pair was introduced.
- We keep track of how many adjacent equal pairs currently exist inside the window.
- If the count exceeds one, we move the left boundary forward until the window becomes valid again.

Because each character enters and leaves the window at most once, the solution runs in linear time.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n³) | O(1) | Check every substring and count adjacent pairs |
| Optimal | O(n) | O(1) | Sliding window maintaining at most one adjacent pair |

## Algorithm Walkthrough

### Step 1: Initialize the sliding window

Create two pointers:

- `left` for the start of the window
- `right` for the end of the window

Also maintain:

- `pair_count`, the number of adjacent equal pairs currently inside the window
- `answer`, the maximum valid window length found so far

Initially:

- `left = 0`
- `pair_count = 0`
- `answer = 1`

### Step 2: Expand the window

Move `right` from left to right across the string.

Whenever `right > 0`, compare:

```
s[right] == s[right - 1]
```

If true, a new adjacent pair has appeared.

Increase `pair_count`.

### Step 3: Restore validity when necessary

The window is valid only when:

```
pair_count <= 1
```

If `pair_count` becomes 2, repeatedly move `left` forward.

Before advancing `left`, check whether removing the character at `left` eliminates an adjacent pair.

Specifically, if:

```
s[left] == s[left + 1]
```

then that adjacent pair is leaving the window, so decrement `pair_count`.

Continue shrinking until `pair_count <= 1`.

### Step 4: Update the answer

Once the current window is valid, its length is:

```
right - left + 1
```

Update the maximum answer.

### Step 5: Continue until the end

Repeat the expansion and shrinking process for every position of `right`.

The largest valid window encountered is the result.

### Why it works

The sliding window always maintains the invariant that the current window contains at most one adjacent equal pair. Whenever adding a new character violates this condition, the left boundary moves until the condition is restored. Since every valid substring ending at the current `right` pointer is represented by the smallest valid window after shrinking, the algorithm never misses a candidate for the maximum length. Therefore, the maximum window size encountered during the scan is exactly the length of the longest semi-repetitive substring.

## Python Solution

```python
class Solution:
    def longestSemiRepetitiveSubstring(self, s: str) -> int:
        n = len(s)

        left = 0
        pair_count = 0
        answer = 1

        for right in range(n):
            if right > 0 and s[right] == s[right - 1]:
                pair_count += 1

            while pair_count > 1:
                if left + 1 < n and s[left] == s[left + 1]:
                    pair_count -= 1
                left += 1

            answer = max(answer, right - left + 1)

        return answer
```

The implementation follows the sliding window strategy directly.

The variable `pair_count` stores how many adjacent equal pairs currently exist inside the window. Whenever the right pointer extends the window and creates a new adjacent pair, the count increases.

If the count exceeds one, the window is no longer semi-repetitive. The left pointer is then advanced until enough characters have been removed to reduce the count back to one or zero.

After each adjustment, the current window satisfies the problem requirement, so its length becomes a candidate answer.

Because both pointers move only forward, the algorithm runs efficiently in linear time.

## Go Solution

```go
func longestSemiRepetitiveSubstring(s string) int {
	n := len(s)

	left := 0
	pairCount := 0
	answer := 1

	for right := 0; right < n; right++ {
		if right > 0 && s[right] == s[right-1] {
			pairCount++
		}

		for pairCount > 1 {
			if left+1 < n && s[left] == s[left+1] {
				pairCount--
			}
			left++
		}

		length := right - left + 1
		if length > answer {
			answer = length
		}
	}

	return answer
}
```

The Go implementation is nearly identical to the Python version. Since strings in Go can be indexed directly by position and the input contains only ASCII digits, byte indexing is sufficient. No special handling for Unicode characters is required. All calculations fit comfortably within Go's `int` type because the maximum string length is only 50.

## Edge Cases

### Single Character String

A string of length one contains no adjacent pairs and is automatically semi-repetitive. Implementations that initialize the answer incorrectly or assume at least two characters may fail on this case. The solution initializes the maximum length to `1`, ensuring correct behavior.

### Long Run of Identical Digits

Strings such as `"1111111"` are tricky because adjacent pairs overlap. The substring `"111"` contains two adjacent pairs, not one. A common mistake is to count duplicate groups instead of actual adjacent pair occurrences. The sliding window counts every adjacent equality individually, which correctly limits the answer to `2`.

### Multiple Pair Locations

A string like `"112233"` contains adjacent pairs at both `"11"` and `"22"`, and later `"33"`. When the second pair enters the window, the left boundary must move until one entire pair is removed. The shrinking logic explicitly checks whether the boundary crosses an adjacent equal pair and decrements `pair_count` accordingly, ensuring the window always remains valid.

### Pair at the Boundary Being Removed

Consider a window where the leftmost two characters form a duplicate pair. When the left pointer moves past that boundary, the pair should no longer be counted. Forgetting to update the count can leave stale information and cause unnecessary shrinking. The implementation handles this by checking `s[left] == s[left + 1]` before incrementing `left`, removing the pair count exactly when it exits the window.

### Entire String Already Valid

Inputs such as `"5494"` or `"123456"` never violate the constraint. The algorithm naturally handles this situation because the shrinking loop is never entered, allowing the answer to grow to the full string length.
