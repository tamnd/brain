## Solution

## Problem Understanding

The problem requires creating a utility function that executes multiple asynchronous functions in parallel and collects their results or errors in a manner similar to `Promise.all` in JavaScript. Specifically, we are given an array `functions` where each element is a function returning a promise. We must execute all these functions concurrently, then produce a new promise that either resolves with an array of resolved values or rejects with the reason of the first rejected promise.

The input represents a small array of asynchronous functions (1 to 10), each function taking no arguments. The output is a single promise whose behavior depends on the resolution of all input promises: it resolves when all have resolved, with their results in order, or rejects immediately on the first rejection.

Important edge cases include a single function, all functions resolving at different times, or at least one function rejecting. The input guarantees valid promise-returning functions and a small number of elements, which means performance is not constrained by the size of the array but correctness and ordering are critical.

## Approaches

A brute-force approach would manually execute each function sequentially, collecting results or handling rejections. While this works, it does not satisfy the parallel execution requirement. The key insight for an optimal solution is that each function can be invoked immediately, and we can attach a `then` and `catch` handler to each resulting promise. By tracking how many promises have resolved and capturing the first rejection, we can create a new promise that resolves or rejects correctly, preserving order.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force Sequential Execution | O(n * t) | O(n) | Executes promises one after another, too slow and not parallel |
| Optimal Parallel Execution | O(n) | O(n) | Executes all promises immediately and resolves or rejects based on the first outcome |

## Algorithm Walkthrough

1. Create a new promise `resultPromise` which will be returned to the caller.
2. Initialize an array `results` of the same length as `functions` to store resolved values.
3. Maintain a counter `resolvedCount` to track how many promises have resolved.
4. Iterate through the array `functions` using their indices.
5. For each function, immediately call it to obtain a promise.
6. Attach a `then` handler that stores the resolved value in the `results` array at the correct index and increments `resolvedCount`.
7. If `resolvedCount` equals the length of `functions`, resolve `resultPromise` with the `results` array.
8. Attach a `catch` handler that rejects `resultPromise` immediately if any promise fails.
9. Return `resultPromise`.

**Why it works:** The invariant is that we only resolve `resultPromise` when all promises have successfully completed, and we reject immediately on the first failure. Storing results by index guarantees that the final array preserves the original order of functions.

## Python Solution

```python
from typing import List, Callable, Any
import asyncio

class Solution:
    def promiseAll(self, functions: List[Callable[[], asyncio.Future]]) -> asyncio.Future:
        loop = asyncio.get_event_loop()
        result_future = loop.create_future()
        n = len(functions)
        results = [None] * n
        resolved_count = 0

        def handle_resolve(i: int, value: Any):
            nonlocal resolved_count
            results[i] = value
            resolved_count += 1
            if resolved_count == n and not result_future.done():
                result_future.set_result(results)

        def handle_reject(reason: Any):
            if not result_future.done():
                result_future.set_exception(reason)

        for idx, func in enumerate(functions):
            try:
                promise = func()
                promise.add_done_callback(
                    lambda fut, i=idx: handle_resolve(i, fut.result()) if not fut.exception() else handle_reject(fut.exception())
                )
            except Exception as e:
                handle_reject(e)

        return result_future
```

In this Python implementation, we use `asyncio.Future` to mimic JavaScript-style promises. Each function is invoked immediately. Resolutions and rejections are tracked using a callback to ensure proper ordering and immediate rejection on failure.

## Go Solution

```go
package main

import (
    "errors"
    "sync"
)

func promiseAll(functions []func() (interface{}, error)) (chan []interface{}, chan error) {
    n := len(functions)
    resultsChan := make(chan []interface{}, 1)
    errorChan := make(chan error, 1)
    results := make([]interface{}, n)
    var wg sync.WaitGroup
    var once sync.Once

    wg.Add(n)
    for i, fn := range functions {
        go func(index int, f func() (interface{}, error)) {
            defer wg.Done()
            res, err := f()
            if err != nil {
                once.Do(func() {
                    errorChan <- err
                })
                return
            }
            results[index] = res
        }(i, fn)
    }

    go func() {
        wg.Wait()
        select {
        case <-errorChan:
            // already sent error
        default:
            resultsChan <- results
        }
    }()

    return resultsChan, errorChan
}
```

In Go, we use goroutines to execute functions in parallel, a `WaitGroup` to synchronize completion, and a `sync.Once` to ensure only the first error is reported. Channels are used to return results or errors.

## Edge Cases

The first edge case is a single function. This is trivial but ensures the algorithm handles minimal input and resolves correctly. The implementation correctly waits for the single promise and returns its result.

The second edge case involves a function that immediately throws an error. This tests whether the algorithm can catch synchronous exceptions and reject the returned promise correctly. Our implementation wraps function calls in try-except to handle this.

The third edge case is multiple functions resolving at vastly different times. This could trip up naive implementations that assume sequential completion. By storing results by index and tracking resolved counts, the implementation ensures correct order and waits for the last resolution.
