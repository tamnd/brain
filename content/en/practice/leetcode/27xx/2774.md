## Solution

## Problem Understanding

The problem asks us to extend arrays with a method called `upperBound()` that returns the last index where a target value appears in a sorted array. The array is sorted in ascending order and may contain duplicate values. If the target does not exist in the array, the method must return `-1`.

In simpler terms, given a sorted array such as `[3,4,6,6,6,6,7]` and a target value `6`, we want to locate the final occurrence of `6`. Since the target appears multiple times consecutively, the correct answer is the index of the rightmost occurrence, which is `5`.

The important detail is that the array is already sorted. This property allows us to use binary search efficiently instead of scanning the entire array one element at a time.

The input consists of:

- A sorted ascending array `nums`
- A target integer `target`

The expected output is:

- The last index where `target` appears
- `-1` if the target is absent

The constraints are relatively small, with array length up to `10^4`, so even a linear scan would technically pass. However, the follow up specifically asks for an `O(log n)` solution, which strongly suggests binary search.

Several edge cases are important:

- The target may not exist at all
- The array may contain only one element
- Every element may equal the target
- The target may appear multiple times consecutively
- The target may be smaller than the first element or larger than the last element

A naive binary search that stops immediately after finding the target would fail for duplicate values because it may return a middle occurrence instead of the final occurrence.

## Approaches

### Brute Force Approach

The simplest solution is to scan the array from left to right and keep updating the answer whenever the target is found.

For example, in `[3,4,6,6,6,6,7]`, we iterate through every index:

- Index `2` contains `6`
- Index `3` contains `6`
- Index `4` contains `6`
- Index `5` contains `6`

The final recorded index becomes `5`.

This approach is correct because we inspect every element, guaranteeing that we eventually discover the last occurrence. However, the runtime is `O(n)` because we may need to traverse the entire array.

Since the array is sorted, we can do better.

### Optimal Binary Search Approach

The key observation is that all duplicates appear together in a sorted array. Instead of stopping when we first find the target, we can continue searching toward the right side to locate the final occurrence.

Binary search works because:

- If `nums[mid] < target`, the target must be on the right
- If `nums[mid] > target`, the target must be on the left
- If `nums[mid] == target`, we record the index and continue searching right to see if another occurrence exists

This modification preserves the logarithmic runtime while guaranteeing we find the rightmost occurrence.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(1) | Scan entire array and track last matching index |
| Optimal | O(log n) | O(1) | Modified binary search for rightmost occurrence |

## Algorithm Walkthrough

### Optimal Binary Search Algorithm

1. Initialize two pointers, `left` and `right`, representing the current search range. Initially, `left = 0` and `right = len(nums) - 1`.
2. Create a variable `answer` initialized to `-1`. This variable stores the latest valid occurrence of the target found during the search.
3. While `left <= right`, calculate the middle index:

```
mid = (left + right) // 2
```

This splits the search space roughly in half.
4. Compare `nums[mid]` with the target.

- If `nums[mid] < target`, the target can only exist on the right side, so move:

```
left = mid + 1
```
- If `nums[mid] > target`, the target can only exist on the left side, so move:

```
right = mid - 1
```
- If `nums[mid] == target`, we found a valid occurrence. Store:

```
answer = mid
```

However, this may not be the final occurrence. Continue searching the right half:

```
left = mid + 1
```
5. Continue shrinking the search space until the pointers cross.
6. Return `answer`. If the target never appeared, `answer` remains `-1`.

### Why it works

The algorithm maintains the invariant that any occurrence of the target beyond the current `mid` must lie in the remaining search space. Whenever a target match is found, we record it but continue moving right because the array is sorted and later occurrences must appear at higher indices. Eventually, the search exhausts all possible positions, and the stored answer is guaranteed to be the rightmost occurrence.

## Python Solution

```python
from typing import List

class Solution:
    def upperBound(self, nums: List[int], target: int) -> int:
        left = 0
        right = len(nums) - 1
        answer = -1

        while left <= right:
            mid = (left + right) // 2

            if nums[mid] < target:
                left = mid + 1
            elif nums[mid] > target:
                right = mid - 1
            else:
                answer = mid
                left = mid + 1

        return answer
```

The implementation begins by initializing the binary search boundaries and the `answer` variable. The `answer` variable is important because binary search may discover multiple valid target positions during the search process.

Inside the loop, the midpoint is calculated using integer division. The comparison logic determines which half of the array can safely be discarded.

When the midpoint value equals the target, the current index is saved as a potential answer. However, the algorithm does not stop there. Instead, it shifts the left boundary rightward to continue searching for a later occurrence.

Eventually, the loop terminates when no search space remains. At that point, `answer` either contains the rightmost target index or remains `-1` if the target was never found.

## Go Solution

```go
package main

func upperBound(nums []int, target int) int {
	left := 0
	right := len(nums) - 1
	answer := -1

	for left <= right {
		mid := left + (right-left)/2

		if nums[mid] < target {
			left = mid + 1
		} else if nums[mid] > target {
			right = mid - 1
		} else {
			answer = mid
			left = mid + 1
		}
	}

	return answer
}
```

The Go implementation follows the same logic as the Python version. One small difference is the midpoint calculation:

```go
mid := left + (right-left)/2
```

This form avoids potential integer overflow in languages with fixed integer sizes. While overflow is not a practical issue for this problem's constraints, it is considered best practice in Go and many systems languages.

Go slices are dynamically sized views over arrays, so no additional memory allocation is needed for the binary search.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(log n) | Binary search halves the search space each iteration |
| Space | O(1) | Only constant extra variables are used |

The runtime is logarithmic because each iteration discards half of the remaining array. Even for large arrays, the number of operations grows very slowly. The algorithm uses only a few integer variables, so the auxiliary space complexity remains constant.

## Edge Cases

One important edge case occurs when the target does not exist in the array. A careless implementation might accidentally return the position where the target would be inserted rather than `-1`. This implementation avoids that issue by initializing `answer` to `-1` and only updating it when an exact match is found.

Another important case is when every element equals the target, such as `[2,2,2,2]`. A standard binary search might stop at the first match and incorrectly return index `1` or `2`. This implementation continues searching right after every match, guaranteeing the final index is returned.

A third edge case involves arrays with only one element. Binary search implementations sometimes fail because of incorrect loop conditions or pointer updates. Here, the condition `while left <= right` ensures that single-element arrays are processed correctly, whether the target exists or not.

A fourth subtle edge case occurs when the target is smaller than every array element or larger than every array element. In these scenarios, the search boundaries eventually cross without finding a match, and the algorithm correctly returns `-1` without accessing invalid indices.
