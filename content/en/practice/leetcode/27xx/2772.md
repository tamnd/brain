# LeetCode 2772 - Apply Operations to Make All Array Elements Equal to Zero

## Problem Understanding

We are given an array `nums` and an integer `k`.

The only operation allowed is selecting any contiguous subarray of length `k` and decreasing every element in that subarray by `1`.

The goal is to determine whether it is possible to perform this operation any number of times so that every element of the array becomes exactly `0`.

A useful way to think about the problem is that every operation affects exactly `k` consecutive positions. We cannot increase values, and we cannot operate on fewer or more than `k` elements at a time.

For example, if `nums[i] = 3`, then position `i` must be included in exactly three decrement operations overall. Every decrement operation contributes `1` reduction to every position it covers.

The constraints are important:

- `nums.length` can be as large as `100,000`
- `nums[i]` can be as large as `1,000,000`

These limits immediately rule out any simulation that performs operations one by one. A solution that literally executes every decrement operation could require billions of updates.

Some important edge cases include:

- `k = 1`, where each element can be reduced independently.
- `k = nums.length`, where every operation affects the entire array.
- Arrays containing zeros that should not trigger unnecessary operations.
- Situations where the last few elements require reductions but there is no remaining length-`k` window available.
- Large values where explicitly simulating each decrement would be prohibitively expensive.

The challenge is therefore to efficiently determine whether a valid sequence of operations exists without actually performing every decrement individually.

## Approaches

### Brute Force

A straightforward approach is to process the array from left to right.

When we reach index `i`, if its value is positive, we are forced to apply that many operations starting at `i`, because once we move past `i`, no future window can affect it.

For example, if the current value at position `i` is `5`, we perform the operation five times on the window `[i, i + k - 1]`.

This approach is correct because the leftmost non-zero element must be fixed immediately. There is no alternative choice.

However, each operation updates `k` elements. If values are large, we may perform millions of operations, and each operation touches up to `k` positions.

The worst-case complexity becomes extremely large:

- Time: `O(sum(nums) * k)`
- Space: `O(1)`

This is far too slow for the given constraints.

### Key Insight

When we stand at position `i`, the only thing that matters is how many decrement operations from previous windows are currently affecting this index.

Instead of actually decrementing entire windows, we can track the cumulative effect of active operations using a difference array.

Suppose previous operations have already reduced position `i` by `current_decrement`.

Then the effective value at index `i` is:

`nums[i] - current_decrement`

If this effective value is negative, we have over-decremented and the answer is immediately `false`.

If the effective value is positive, then we are forced to start exactly that many new operations at index `i`.

Rather than updating the next `k` elements individually, we record that these operations begin affecting indices now and stop affecting indices after `i + k - 1`.

This transforms expensive range updates into constant-time bookkeeping.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(sum(nums) × k) | O(1) | Explicitly performs every decrement operation |
| Optimal | O(n) | O(n) | Uses a difference array to track active decrements |

## Algorithm Walkthrough

### Optimal Greedy + Difference Array

1. Create a difference array `difference` of size `n + 1`, initialized with zeros.
2. Maintain a variable `active_decrements` representing how many previously started operations are currently affecting the current index.
3. Iterate through the array from left to right.
4. At each index `i`, first apply any scheduled expiration:

`active_decrements += difference[i]`

This updates the total number of active operations affecting position `i`.
5. Compute the effective value:

`remaining = nums[i] - active_decrements`

This is the amount still needing to be removed from index `i`.
6. If `remaining < 0`, previous operations have already reduced this element below zero. This is impossible, so return `false`.
7. If `remaining == 0`, this position is already satisfied. Continue to the next index.
8. If `remaining > 0`, we must start exactly `remaining` new operations at index `i`.
9. Before starting them, verify that a length-`k` window can begin here:

`i + k <= n`

If not, there is no way to affect this position anymore, so return `false`.
10. Add these new operations to the active count:

`active_decrements += remaining`
11. Schedule their expiration:

`difference[i + k] -= remaining`
12. Continue until all indices have been processed.
13. If no contradiction is encountered, return `true`.

### Why it works

The algorithm relies on a greedy invariant.

When processing index `i`, no future operation can start before `i`, and any operation starting after `i` cannot affect position `i`. Therefore, if the effective value at `i` is `remaining`, we are forced to start exactly `remaining` operations at index `i`.

There is no alternative choice. Any valid solution must make the same decision. The difference array efficiently tracks the cumulative effect of all such forced choices. Since every step is uniquely determined, the algorithm correctly decides whether a valid sequence of operations exists.

## Python Solution

```python
from typing import List

class Solution:
    def checkArray(self, nums: List[int], k: int) -> bool:
        n = len(nums)

        difference = [0] * (n + 1)
        active_decrements = 0

        for i in range(n):
            active_decrements += difference[i]

            remaining = nums[i] - active_decrements

            if remaining < 0:
                return False

            if remaining > 0:
                if i + k > n:
                    return False

                active_decrements += remaining
                difference[i + k] -= remaining

        return True
```

The implementation follows the greedy strategy exactly.

The `difference` array stores when previously started operations stop contributing. As we move through the array, `active_decrements` represents the total number of currently active decrement operations.

For each position, we compute its effective value after all previously applied operations. If that value is negative, we have already decremented too much. If it is positive, we are forced to start exactly that many new operations.

Instead of modifying an entire window, we increase `active_decrements` immediately and schedule its removal at `i + k` through the difference array. This allows every index to be processed in constant time.

## Go Solution

```go
func checkArray(nums []int, k int) bool {
	n := len(nums)

	difference := make([]int, n+1)
	activeDecrements := 0

	for i := 0; i < n; i++ {
		activeDecrements += difference[i]

		remaining := nums[i] - activeDecrements

		if remaining < 0 {
			return false
		}

		if remaining > 0 {
			if i+k > n {
				return false
			}

			activeDecrements += remaining
			difference[i+k] -= remaining
		}
	}

	return true
}
```

The Go implementation mirrors the Python solution closely.

Slices are used for the difference array. Since the constraints guarantee that values fit comfortably within standard integer ranges for the target platform, Go's `int` type is sufficient. No special handling for nil slices is required because the difference array is always explicitly allocated.

The key optimization is that range updates are represented implicitly through the difference array. Every operation start and expiration is recorded in constant time, eliminating the need to touch all `k` elements of a window. As a result, the algorithm scales linearly with the array length.

## Edge Cases

### Edge Case 1: `k = 1`

When `k = 1`, each operation affects only a single element. Every value can be reduced independently until it reaches zero. A naive implementation might still perform millions of operations. The difference-array approach handles this naturally by processing each position once and scheduling the operation's expiration immediately at the next index.

### Edge Case 2: Reaching the End of the Array

Suppose we arrive at index `i` and discover that the effective value is still positive, but `i + k > n`. In this situation there is no valid window starting at `i`, so no future operation can affect this position. The implementation explicitly checks this condition and returns `false` immediately.

### Edge Case 3: Over-Decrementing a Position

Because operations overlap, previously started windows may reduce a position more times than its original value allows. This manifests as:

```
remaining < 0
```

Once this occurs, the array has effectively gone below zero at that position, which can never be corrected because operations only decrease values. The implementation detects this immediately and returns `false`.

### Edge Case 4: Arrays Already Equal to Zero

An array consisting entirely of zeros requires no operations. Some implementations accidentally perform unnecessary work or incorrectly start windows. Here, every index has `remaining = 0`, so the algorithm simply walks through the array and returns `true`.

### Edge Case 5: Large Values

Values can be as large as `1,000,000`. Explicit simulation would be far too expensive because it might require millions of decrement operations. The difference-array method never performs operations individually. It stores only aggregate effects, allowing the algorithm to remain `O(n)` regardless of how large the numbers are.
