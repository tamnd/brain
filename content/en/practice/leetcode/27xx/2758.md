# LeetCode 2758 - Next Day

## Problem Understanding

This problem asks us to extend date handling functionality so that calling `nextDay()` on a date object returns the calendar date of the following day as a string in the format `YYYY-MM-DD`.

The input is a valid date. The output must always be the immediately succeeding calendar day. This means the implementation must correctly handle all normal date transitions, including month boundaries and year boundaries.

For example, if the current date is `2014-06-20`, then the next day is simply `2014-06-21`. However, if the current date is `2017-10-31`, the next day crosses into a new month, producing `2017-11-01`.

The problem guarantees that `new Date(date)` produces a valid JavaScript `Date` object. This simplifies the implementation because we do not need to validate malformed dates or handle invalid input formats.

Even though the problem is classified as easy, several edge cases can still introduce bugs in naive implementations. Month transitions are one important case because different months contain different numbers of days. Leap years are another source of complexity because February can contain either 28 or 29 days. Year transitions such as December 31st to January 1st must also work correctly. Finally, timezone behavior can create subtle formatting issues if the solution does not normalize the date properly.

Because JavaScript already provides built in date arithmetic, the safest and cleanest approach is to rely on the language runtime rather than manually computing month lengths and leap year rules.

## Approaches

### Brute Force Approach

A brute force approach would manually parse the year, month, and day from the input date string. The implementation would then determine how many days exist in the current month, increment the day, and handle rollover into the next month or next year when necessary.

This approach can produce the correct answer if implemented carefully. However, it requires explicitly handling leap year logic, month length lookup tables, and rollover rules. Manual date arithmetic is error prone because many special cases exist.

For example, February behaves differently during leap years, and months alternate between 30 and 31 days except for February. A manual implementation increases the risk of off by one mistakes and invalid date transitions.

### Optimal Approach

The optimal approach uses built in date arithmetic. Since JavaScript `Date` objects already understand leap years, month lengths, timezone normalization, and calendar rollover behavior, we can simply add one day to the existing date.

The key observation is that date libraries and built in date objects are specifically designed to handle these transitions safely and efficiently. By using the existing API, we avoid reimplementing complicated calendar logic ourselves.

After incrementing the date by one day, we format the resulting date back into the required `YYYY-MM-DD` string format.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(1) | O(1) | Manually handles leap years and month rollover |
| Optimal | O(1) | O(1) | Uses built in date arithmetic and formatting |

## Algorithm Walkthrough

1. Start with the current `Date` object.
2. Extract the current day value using the built in date API. In JavaScript, this is done with `getDate()`.
3. Add one to the current day value. Instead of manually adjusting the month or year, use `setDate(currentDay + 1)`. The JavaScript engine automatically handles overflow into the next month or year.
4. Convert the updated date back into a string in `YYYY-MM-DD` format. A convenient way is to use `toISOString()` and take only the first 10 characters.
5. Return the formatted string.

### Why it works

The correctness comes from the fact that JavaScript `Date` objects internally manage valid calendar arithmetic. When `setDate()` receives a value outside the valid day range for a month, the runtime automatically normalizes the result into the correct month and year. Therefore, incrementing the day by one always produces the correct next calendar date.

## Python Solution

Although the original LeetCode problem is JavaScript based, the following Python implementation demonstrates the same logic using Python's `datetime` library.

```python
from datetime import datetime, timedelta

class Solution:
    def nextDay(self, date: str) -> str:
        current_date = datetime.strptime(date, "%Y-%m-%d")
        next_date = current_date + timedelta(days=1)
        return next_date.strftime("%Y-%m-%d")
```

The implementation begins by parsing the input string into a `datetime` object using `strptime`. Once the date is represented as a structured object, we add one day using `timedelta(days=1)`.

Python's datetime library automatically handles leap years, month transitions, and year transitions. Finally, the result is formatted back into the required string format using `strftime`.

This closely mirrors the optimal JavaScript solution conceptually, even though the APIs differ slightly.

## Go Solution

```go
package main

import (
	"time"
)

func nextDay(date string) string {
	currentDate, _ := time.Parse("2006-01-02", date)

	nextDate := currentDate.AddDate(0, 0, 1)

	return nextDate.Format("2006-01-02")
}
```

The Go implementation uses the standard `time` package. The layout string `"2006-01-02"` is Go's required formatting reference for parsing and formatting dates.

The `AddDate(0, 0, 1)` function increments the date by one day while automatically handling month and year rollover. Like Python and JavaScript, Go's time library safely manages leap years and calendar normalization internally.

One Go specific detail is that `time.Parse` returns both a parsed time and an error value. Since the problem guarantees valid input, the error can safely be ignored here.

## Edge Cases

One important edge case is the transition from the end of a month into the next month. For example, `2021-01-31` should become `2021-02-01`. A manual implementation might incorrectly produce an invalid date such as `2021-01-32`. Using built in date arithmetic avoids this issue entirely because the runtime automatically normalizes overflow days.

Another critical edge case is leap year handling. Dates like `2020-02-28` should become `2020-02-29`, while `2019-02-28` should become `2019-03-01`. Leap year rules are surprisingly subtle because years divisible by 100 are not leap years unless they are also divisible by 400. Built in date libraries correctly implement these rules automatically.

A third important edge case is year rollover. For example, `2023-12-31` must become `2024-01-01`. This requires simultaneously resetting both the month and day while incrementing the year. The implementation handles this safely because the underlying date library manages all calendar transitions internally.
