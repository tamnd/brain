## Solution

## Problem Understanding

This problem involves simulating the relocation of marbles across integer positions. You are given an initial array `nums` representing the positions of the marbles. Each position can have multiple marbles. Additionally, you have two arrays, `moveFrom` and `moveTo`, which dictate a sequence of moves: on the `i`th move, all marbles at position `moveFrom[i]` are relocated to `moveTo[i]`. After performing all moves in order, you are asked to return a **sorted list of all positions that have at least one marble**, i.e., the occupied positions.

The inputs are constrained as follows: there can be up to `10^5` marbles and moves, and positions can range up to `10^9`. These constraints rule out a solution that manipulates arrays indexed by positions because that would be too large in memory. Instead, a set or a hash map must be used to track occupied positions efficiently.

Important edge cases include situations where multiple marbles occupy the same position, moves consolidate multiple positions into a single position, or where moves relocate marbles to positions that were previously unoccupied. The problem guarantees that `moveFrom[i]` always contains at least one marble at the time of the move, which prevents invalid moves.

## Approaches

A brute-force approach would attempt to iterate through `nums` for each move, checking for each marble if it is at `moveFrom[i]` and updating it to `moveTo[i]`. While correct, this approach is extremely inefficient because for `n` marbles and `m` moves, it has a worst-case time complexity of `O(n*m)`, which is too slow for the upper limit of `10^5`.

The key insight for an optimal solution is that we do not care about individual marbles themselves, only the **occupied positions**. Therefore, we can maintain a **set of occupied positions**. For each move, we check if `moveFrom[i]` exists in the set; if it does, we remove it and add `moveTo[i]`. Using a set ensures that each lookup, addition, and removal is `O(1)` on average, reducing the total complexity to `O(n + m)`.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n*m) | O(n) | Iterate through all marbles for each move; too slow for large inputs |
| Optimal | O(n + m + k log k) | O(n + m) | Maintain a set of occupied positions and update for each move; `k` is the number of unique occupied positions at the end |

## Algorithm Walkthrough

1. Initialize a set `occupied` containing all unique positions from the `nums` array. This ensures we track only the positions that have at least one marble.
2. Iterate over the moves using an index `i` from `0` to `len(moveFrom) - 1`.
3. For each move, check if `moveFrom[i]` exists in the `occupied` set. If it does, remove it and add `moveTo[i]` to the set. If it does not exist, skip it because no marbles occupy that position.
4. After all moves have been applied, convert the `occupied` set to a sorted list. Sorting ensures the final output is in ascending order as required by the problem statement.
5. Return the sorted list of occupied positions.

**Why it works:** The algorithm works because at each step, the set `occupied` correctly represents all positions containing at least one marble. The operations of removal and addition simulate the relocation of marbles efficiently. By converting the set to a sorted list at the end, we satisfy the problem's output requirement without ever needing to track individual marbles.

## Python Solution

```python
from typing import List

class Solution:
    def relocateMarbles(self, nums: List[int], moveFrom: List[int], moveTo: List[int]) -> List[int]:
        occupied = set(nums)
        for f, t in zip(moveFrom, moveTo):
            if f in occupied:
                occupied.remove(f)
                occupied.add(t)
        return sorted(occupied)
```

**Explanation:** We initialize a set `occupied` with the unique positions of marbles. For each move, we check if the source position `f` exists in the set. If it does, we remove it and add the destination `t`. Finally, we sort the set and return it as a list. This ensures correct simulation and correct ordering.

## Go Solution

```go
import "sort"

func relocateMarbles(nums []int, moveFrom []int, moveTo []int) []int {
    occupied := make(map[int]struct{})
    for _, num := range nums {
        occupied[num] = struct{}{}
    }

    for i := 0; i < len(moveFrom); i++ {
        f, t := moveFrom[i], moveTo[i]
        if _, exists := occupied[f]; exists {
            delete(occupied, f)
            occupied[t] = struct{}{}
        }
    }

    result := make([]int, 0, len(occupied))
    for pos := range occupied {
        result = append(result, pos)
    }
    sort.Ints(result)
    return result
}
```

**Explanation:** The Go version uses a `map[int]struct{}` to implement a set for occupied positions. Each move checks if the key exists in the map. If it does, the key is removed and the new position is added. Finally, all keys are collected into a slice and sorted to produce the final result.

## Edge Cases

One edge case is a single marble being moved. The implementation correctly removes it from its original position and adds it to the destination, resulting in a correct single-element output.

Another edge case occurs when multiple marbles occupy the same position. Our approach tracks positions rather than individual marbles, so regardless of the count at that position, moving it once correctly updates the occupied set.

Finally, moves can lead to merging multiple positions into a single destination. The set ensures that duplicates are automatically ignored, so the algorithm correctly handles consolidation without counting
