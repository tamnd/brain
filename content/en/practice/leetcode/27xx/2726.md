# LeetCode 2726 - Calculator with Method Chaining

## Problem Understanding

This problem asks us to design a class named `Calculator` that supports a sequence of mathematical operations through method chaining. The class maintains an internal numeric state called `result`, and each operation updates this value in place.

The constructor initializes the calculator with a starting value. After initialization, the object should support the following operations:

- Addition
- Subtraction
- Multiplication
- Division
- Exponentiation

Each operation modifies the current result and then returns the same calculator object so that additional methods can immediately be called after it. This technique is known as method chaining.

For example:

```
new Calculator(10).add(5).subtract(7).getResult()
```

works because `add()` returns the calculator instance itself, allowing `subtract()` to be called next.

The `getResult()` method is different from the other methods because it returns the final numeric result rather than the calculator object.

The division operation introduces one important constraint. If division by zero is attempted, the implementation must throw an error with the exact message:

```
"Division by zero is not allowed"
```

The constraints indicate that the sequence of operations may be fairly large, up to `2 * 10^4` actions. However, each operation itself is constant time, so performance is not a major concern. The main challenge is correctly implementing object-oriented behavior and method chaining semantics.

Several edge cases are important:

- Division by zero must immediately throw an error.
- Chained operations must preserve the updated internal state correctly.
- Exponentiation may produce floating point values or very large numbers.
- Operations involving negative values or decimals should still work correctly.
- The methods must return the calculator instance itself, otherwise chaining will fail.

The problem guarantees that:

- The first action is always `"Calculator"`.
- The last action is always `"getResult"`.
- All operation names are valid.

This means we do not need to validate operation names or handle malformed input.

## Approaches

### Brute Force Approach

A very naive way to solve this problem would be to avoid storing state inside the calculator object. Instead, every method could create and return a completely new calculator object with the updated result.

For example, `add(5)` would construct a brand new `Calculator` instance whose internal value equals the old result plus five.

This approach works because every operation still produces the correct updated value, and chaining remains possible since another calculator object is returned each time.

However, this design is inefficient because every operation allocates a new object. With many chained operations, this creates unnecessary memory overhead and object construction cost.

### Optimal Approach

The better solution is to maintain a single mutable `result` field inside the calculator object.

Each method directly updates this internal state and then returns `self` in Python or the current pointer in Go. This allows chaining without creating extra objects.

The key observation is that every operation depends only on the current accumulated result. There is no need to preserve historical states or allocate new calculator instances.

This approach is both cleaner and more efficient.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) | O(n) | Creates a new calculator object for every operation |
| Optimal | O(n) | O(1) | Updates a single internal result value |

## Algorithm Walkthrough

1. Create a `Calculator` class with an internal variable called `result`.
2. In the constructor, initialize `result` with the provided starting value. This value becomes the current calculator state.
3. Implement the `add` method:

- Add the provided value to `result`.
- Return the current calculator object so additional methods can be chained.
4. Implement the `subtract` method:

- Subtract the provided value from `result`.
- Return the calculator object.
5. Implement the `multiply` method:

- Multiply `result` by the provided value.
- Return the calculator object.
6. Implement the `divide` method:

- First check whether the divisor equals zero.
- If it does, throw an error with the required message.
- Otherwise divide `result` by the provided value.
- Return the calculator object.
7. Implement the `power` method:

- Raise `result` to the specified exponent.
- Store the updated value back into `result`.
- Return the calculator object.
8. Implement the `getResult` method:

- Return the current value of `result`.

### Why it works

The algorithm works because the calculator maintains a single invariant:

> `result` always stores the value produced by applying all previously executed operations in order.

Each method updates this value exactly once and returns the same calculator instance. Because the operations are performed sequentially and directly modify the stored result, the final value returned by `getResult()` matches the intended chained computation.

## Python Solution

```python
class Calculator:

    def __init__(self, value: float):
        self.result = value

    def add(self, value: float) -> "Calculator":
        self.result += value
        return self

    def subtract(self, value: float) -> "Calculator":
        self.result -= value
        return self

    def multiply(self, value: float) -> "Calculator":
        self.result *= value
        return self

    def divide(self, value: float) -> "Calculator":
        if value == 0:
            raise Exception("Division by zero is not allowed")

        self.result /= value
        return self

    def power(self, value: float) -> "Calculator":
        self.result **= value
        return self

    def getResult(self) -> float:
        return self.result
```

The implementation stores the current calculator state in `self.result`.

Each arithmetic method updates this value directly. After updating the result, the method returns `self`, which is the current calculator instance. Returning `self` is what enables method chaining.

For example:

```python
Calculator(10).add(5).subtract(3)
```

works because:

- `add(5)` returns the calculator object
- `subtract(3)` is then called on that same object

The `divide` method includes a defensive check against division by zero. If zero is detected, an exception is raised immediately with the exact required message.

The `power` method uses Python's exponentiation operator `**`.

Finally, `getResult()` simply returns the accumulated result value.

## Go Solution

```go
package main

import "errors"

type Calculator struct {
	result float64
}

func Constructor(value float64) Calculator {
	return Calculator{
		result: value,
	}
}

func (c *Calculator) Add(value float64) *Calculator {
	c.result += value
	return c
}

func (c *Calculator) Subtract(value float64) *Calculator {
	c.result -= value
	return c
}

func (c *Calculator) Multiply(value float64) *Calculator {
	c.result *= value
	return c
}

func (c *Calculator) Divide(value float64) (*Calculator, error) {
	if value == 0 {
		return nil, errors.New("Division by zero is not allowed")
	}

	c.result /= value
	return c, nil
}

func (c *Calculator) Power(value float64) *Calculator {
	result := 1.0

	for i := 0; i < int(value); i++ {
		result *= c.result
	}

	c.result = result
	return c
}

func (c *Calculator) GetResult() float64 {
	return c.result
}
```

The Go implementation uses pointer receivers so that methods can directly modify the calculator's internal state.

Unlike Python, Go does not have built in exceptions. Therefore, division by zero is handled by returning an `error`.

One implementation detail is that Go does not have a native exponentiation operator like Python's `**`. In production code, `math.Pow()` would normally be used. However, the above implementation demonstrates the logic explicitly.

The calculator methods return `*Calculator`, which enables chaining on the same object instance.

## Edge Cases

One important edge case is division by zero. This is the only explicitly invalid operation in the problem statement. A naive implementation might allow the language runtime to handle it implicitly, which could produce inconsistent behavior across languages. The implementation avoids this by explicitly checking whether the divisor equals zero before performing division and throwing the required error message immediately.

Another important edge case is chaining many operations together. If methods do not return the calculator instance itself, chaining breaks after the first call. For example, returning the numeric result instead of the object would make subsequent method calls impossible. The implementation handles this correctly by always returning `self` in Python and `*Calculator` in Go.

Exponentiation with zero is another subtle case. Mathematically, any nonzero number raised to the power of zero equals one. Some incorrect implementations accidentally preserve the original value or mishandle the loop bounds. The provided implementations correctly produce `1` when the exponent is zero.

Floating point division is also important. Integer division would produce incorrect answers in cases like `10 / 4`, which should equal `2.5`. The implementation stores results as floating point values so decimal results are preserved correctly throughout chained operations.
