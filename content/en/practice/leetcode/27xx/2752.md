## Solution

## Problem Understanding

The problem gives us a `Transactions` table where each row represents a transaction made by a customer on a specific date. The table guarantees that the pair `(customer_id, transaction_date)` is unique, which means a customer can have at most one transaction per day.

We need to determine which customers achieved the longest streak of consecutive transaction days. A consecutive streak means that for every adjacent pair of dates in the streak, the difference is exactly one day.

For example, if a customer made transactions on:

- `2023-05-01`
- `2023-05-02`
- `2023-05-03`

then their streak length is `3`.

However, if another customer made transactions on:

- `2023-05-01`
- `2023-05-03`
- `2023-05-04`

then the longest consecutive streak is only `2`, because `2023-05-03` and `2023-05-04` are consecutive, but `2023-05-01` and `2023-05-03` are not.

The goal is to compute the maximum consecutive streak achieved by any customer, then return every `customer_id` whose best streak equals that global maximum. The result must be sorted in ascending order by `customer_id`.

The key challenge is efficiently detecting consecutive date sequences within each customer's transaction history.

The constraints are not explicitly listed in the prompt, but since this is a Hard database problem, we should assume the dataset may be large. That means solutions involving repeated scans or quadratic comparisons between dates are likely too slow.

Several edge cases are important:

- A customer may have only one transaction. Their maximum streak is `1`.
- Multiple customers may tie for the longest streak.
- Transactions may contain gaps of many days.
- The longest streak may appear in the middle of a customer's transaction history.
- Since `(customer_id, transaction_date)` is unique, we do not need to handle duplicate same-day transactions.

## Approaches

### Brute Force Approach

A brute-force solution would process each customer independently and attempt to build every possible consecutive sequence from every starting date.

For each transaction date of a customer, we could repeatedly check whether the next day exists in the dataset. This would allow us to compute the streak length beginning at each date.

For example, suppose a customer has dates:

```
2023-05-01
2023-05-02
2023-05-03
2023-05-05
```

Starting from `2023-05-01`, we would check whether:

- `2023-05-02` exists
- `2023-05-03` exists
- `2023-05-04` exists

and stop once a date is missing.

We would repeat this process for every date.

This approach is correct because it explicitly verifies every possible consecutive chain. However, it becomes inefficient when customers have many transaction dates. Repeatedly searching for next-day existence can lead to quadratic behavior.

### Key Insight for the Optimal Solution

The important observation is that consecutive sequences can be identified efficiently after sorting dates.

If we sort all transaction dates for a customer, then consecutive dates appear adjacent in sorted order.

For example:

```
2023-05-01
2023-05-02
2023-05-03
2023-05-05
2023-05-06
```

We can scan linearly:

- `05-02` follows `05-01` by one day, streak becomes `2`
- `05-03` follows `05-02` by one day, streak becomes `3`
- `05-05` is not consecutive, reset streak to `1`
- `05-06` follows `05-05`, streak becomes `2`

This transforms the problem into a standard "longest consecutive run in sorted order" problem.

In SQL, the cleanest way to solve this uses window functions. Specifically:

- `ROW_NUMBER()` assigns increasing indices per customer.
- Dates in a consecutive streak maintain a constant difference between:

- the date value
- the row number index

This is a classic "gaps and islands" technique.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Repeatedly checks consecutive dates from every starting point |
| Optimal | O(n log n) | O(n) | Sorts dates and detects streak groups efficiently |

## Algorithm Walkthrough

1. Partition transactions by `customer_id`.

Each customer's transaction history must be analyzed independently because streaks cannot span multiple customers.
2. Sort transaction dates within each customer.

Consecutive streaks only make sense in chronological order. Sorting ensures adjacent dates are examined correctly.
3. Assign a row number to each transaction per customer.

For example:

| Date | Row Number |
| --- | --- |
| 2023-05-01 | 1 |
| 2023-05-02 | 2 |
| 2023-05-03 | 3 |
4. Compute a grouping key using:

```
transaction_date - INTERVAL row_number DAY
```

The key insight is that all dates belonging to the same consecutive streak produce the same adjusted value.

Example:

| Date | Row | Adjusted |
| --- | --- | --- |
| 2023-05-01 | 1 | 2023-04-30 |
| 2023-05-02 | 2 | 2023-04-30 |
| 2023-05-03 | 3 | 2023-04-30 |

Since the adjusted value is constant, these rows belong to the same streak.
5. Group by `(customer_id, adjusted_key)`.

Each group now represents one consecutive sequence.
6. Count the size of each group.

The count equals the streak length.
7. Compute the maximum streak per customer.

A customer may have multiple streaks, so we keep only their best one.
8. Compute the global maximum streak across all customers.
9. Return every customer whose best streak equals the global maximum.
10. Sort results by `customer_id`.

### Why it works

The algorithm relies on the invariant that consecutive dates increase by exactly one day while row numbers also increase by exactly one. Therefore, subtracting the row index from the date produces a constant value across an uninterrupted consecutive sequence. Any gap breaks this invariant and creates a new grouping key, correctly separating streaks.

## Python Solution

```python
# complete solution here

class Solution:
    def consecutiveAvailableSeats(self):
        return """
        WITH numbered AS (
            SELECT
                customer_id,
                transaction_date,
                ROW_NUMBER() OVER (
                    PARTITION BY customer_id
                    ORDER BY transaction_date
                ) AS rn
            FROM Transactions
        ),
        grouped AS (
            SELECT
                customer_id,
                transaction_date,
                DATE_SUB(transaction_date, INTERVAL rn DAY) AS grp
            FROM numbered
        ),
        streaks AS (
            SELECT
                customer_id,
                COUNT(*) AS streak_length
            FROM grouped
            GROUP BY customer_id, grp
        ),
        customer_best AS (
            SELECT
                customer_id,
                MAX(streak_length) AS max_streak
            FROM streaks
            GROUP BY customer_id
        ),
        global_best AS (
            SELECT MAX(max_streak) AS longest_streak
            FROM customer_best
        )
        SELECT customer_id
        FROM customer_best
        WHERE max_streak = (
            SELECT longest_streak
            FROM global_best
        )
        ORDER BY customer_id;
        """
```

The solution uses several Common Table Expressions to structure the logic cleanly.

The `numbered` CTE assigns a row number to each transaction ordered by date within each customer.

The `grouped` CTE computes the grouping key using `DATE_SUB(transaction_date, INTERVAL rn DAY)`. Consecutive dates share the same adjusted date value.

The `streaks` CTE groups rows by customer and grouping key, then counts the rows in each group. Each count represents one consecutive streak length.

The `customer_best` CTE computes the maximum streak for every customer.

The `global_best` CTE computes the overall maximum streak among all customers.

Finally, the outer query returns every customer whose best streak matches the global maximum.

## Edge Cases

A customer with only one transaction is an important edge case because their streak length should still count as `1`. Some incorrect implementations may accidentally ignore customers who never form a multi-day streak. This solution handles the case naturally because every transaction forms a group, even if the group contains only one row.

Another important case is when there are gaps between dates. For example:

```
2023-05-01
2023-05-03
2023-05-04
```

A naive implementation might incorrectly count all three dates together. The grouping-key technique correctly separates the first date from the latter consecutive pair because the adjusted date value changes when a gap appears.

A third edge case occurs when multiple customers tie for the global maximum streak. Some implementations mistakenly return only one customer. This solution explicitly compares every customer's maximum streak against the global maximum and returns all matching customers sorted in ascending order.
