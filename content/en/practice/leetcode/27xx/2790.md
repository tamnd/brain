# LeetCode 2790 - Maximum Number of Groups With Increasing Length

## Problem Understanding

We are given an array `usageLimits`, where `usageLimits[i]` tells us how many times the number `i` may be used across all groups.

The numbers available are exactly `0` through `n - 1`, where `n = len(usageLimits)`.

Our goal is to create as many groups as possible while satisfying two constraints:

1. Every group contains only distinct numbers.
2. Group lengths must be strictly increasing.

The actual values of the numbers do not matter. What matters is how many times each number can participate across different groups.

For example, if `usageLimits[2] = 5`, then the number `2` can appear in up to five different groups, but never more than once within the same group because groups require distinct elements.

The output is the maximum number of groups that can be formed.

A key observation is that if we create `k` groups, the smallest possible lengths are:

- 1
- 2
- 3
- ...
- k

Any other strictly increasing sequence would require even more elements. Therefore, if we can build `k` groups at all, we can assume their sizes are exactly:

$$1 + 2 + 3 + \cdots + k = \frac{k(k+1)}{2}$$

The constraints are large:

- `n ≤ 100000`
- `usageLimits[i] ≤ 10^9`

This immediately rules out any solution that explicitly constructs groups or simulates element placement.

Important edge cases include:

- A single number with a huge usage limit.
- Many numbers with usage limit `1`.
- Extremely large usage limits requiring 64-bit arithmetic.
- Cases where total available usages are large enough, but distinctness restrictions prevent forming additional groups.

## Approaches

### Brute Force

A brute force strategy would attempt to explicitly construct groups.

Suppose we want to create groups of sizes `1, 2, 3, ...`. We could repeatedly choose available numbers and place them into groups while tracking remaining capacities.

This eventually becomes a complicated combinatorial assignment problem. Even checking whether a given number of groups is feasible would require repeatedly selecting elements and updating counts.

Since:

- `n` can be `100000`
- usage limits can be extremely large

such simulation is far too expensive.

### Key Insight

The actual identities of the numbers are not important. What matters is how much total capacity is available.

Sort `usageLimits`.

After sorting, process capacities from smallest to largest.

Let `available` represent the total capacity accumulated so far.

Whenever we have enough accumulated capacity to form the next group, we should do so immediately.

Suppose we have already formed `groups` groups.

The next group must have size:

$$groups + 1$$

If accumulated capacity is at least this amount, we can create that group and deduct its required size from the accumulated capacity.

This greedy strategy works because smaller capacities are handled first. Any unused capacity can always be carried forward to help create larger groups later.

| Optimal Greedy + Sorting | O(n log n) | O(1) extra, excluding sorting | Uses accumulated capacity and greedily forms groups |

## Algorithm Walkthrough

### Step 1

Sort `usageLimits` in nondecreasing order.

This ensures small capacities are processed first and prevents wasting large capacities too early.

### Step 2

Maintain:

- `available`, total unused capacity accumulated so far
- `groups`, number of groups successfully formed

Initially:

```
available = 0
groups = 0
```

### Step 3

Iterate through every usage limit.

For each value:

```
available += limit
```

This adds all capacity contributed by the current number.

### Step 4

Check whether we can form the next group.

The next group must have size:

```
groups + 1
```

If:

```
available >= groups + 1
```

then create that group.

### Step 5

After creating the group:

```
available -= groups + 1
groups += 1
```

The used capacity is consumed, while remaining capacity stays available for future groups.

### Step 6

Continue until all usage limits have been processed.

The final value of `groups` is the answer.

### Why it works

After sorting, every capacity contributes to a common pool of available usages. Whenever enough capacity exists to form the smallest currently required group, delaying its creation can never improve the answer because future groups require even more capacity. Therefore forming groups greedily as soon as possible maximizes the total number of groups. The remaining unused capacity is preserved and can contribute to larger groups later. This invariant guarantees optimality.

## Python Solution

```python
from typing import List

class Solution:
    def maxIncreasingGroups(self, usageLimits: List[int]) -> int:
        usageLimits.sort()

        available = 0
        groups = 0

        for limit in usageLimits:
            available += limit

            if available >= groups + 1:
                groups += 1
                available -= groups

        return groups
```

### Implementation Explanation

The array is first sorted.

The variable `available` stores accumulated capacity that has not yet been assigned to any group. As we iterate through the sorted limits, each value is added into this pool.

The variable `groups` stores how many groups have already been formed.

Whenever the pool contains enough capacity to build the next required group of size `groups + 1`, we immediately create that group. The required capacity is removed from `available`, and the group count increases.

At the end of the traversal, `groups` contains the maximum achievable number of increasing-length groups.

## Go Solution

```go
import "sort"

func maxIncreasingGroups(usageLimits []int) int {
	sort.Ints(usageLimits)

	var available int64 = 0
	groups := 0

	for _, limit := range usageLimits {
		available += int64(limit)

		if available >= int64(groups+1) {
			groups++
			available -= int64(groups)
		}
	}

	return groups
}
```

### Go-Specific Notes

The main difference is the use of `int64` for `available`.

Since:

- `n` can be `100000`
- `usageLimits[i]` can be `10^9`

the accumulated capacity may reach:

$$10^5 \times 10^9 = 10^{14}$$

which exceeds 32-bit integer limits.

The rest of the implementation mirrors the Python version exactly.

## Edge Cases

### Only One Available Number

Consider:

```
[1000000000]
```

Although the number can be used a billion times, there is only one distinct value available. Since groups must contain distinct elements and lengths must increase, only one group of size `1` can be formed. The algorithm correctly returns `1`.

### Extremely Large Usage Limits

Consider:

```
[1000000000, 1000000000, 1000000000]
```

The accumulated capacity becomes very large, potentially exceeding 32-bit integer limits. The implementation uses Python's arbitrary precision integers and `int64` in Go, preventing overflow.

### Many Small Capacities

Consider:

```
[1,1,1,1,1]
```

The total capacity is only `5`. Forming three groups would require:

$$1 + 2 + 3 = 6$$

usages, which is impossible. The greedy accumulation logic naturally stops at two groups because the available capacity never reaches the requirement for the third group.

### Unsorted Inputs

Consider:

```
[8,4,6,2,1]
```

Processing capacities in arbitrary order could make reasoning difficult. Sorting ensures the greedy invariant holds, allowing the algorithm to safely accumulate capacity and form groups whenever possible. This guarantees correctness regardless of the original order of the input.
