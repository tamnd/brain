## Solution

## Problem Understanding

This problem asks us to implement behavior similar to JavaScript's built in `Promise.allSettled()` function, but without actually using that method.

We are given an array called `functions`. Each element in this array is itself a function. When one of these functions is executed, it returns a promise. That promise may either:

1. Resolve successfully with some value
2. Reject with an error or rejection reason

Our task is to create a function that executes all of these asynchronous operations in parallel and returns a single promise. That returned promise should eventually resolve with an array describing the outcome of every individual promise.

The important detail is that the returned array must preserve the original order of the input functions, regardless of which promises finish first.

If a promise resolves successfully, we store:

```
{
    "status": "fulfilled",
    "value": resolved_value
}
```

If a promise rejects, we store:

```
{
    "status": "rejected",
    "reason": rejection_reason
}
```

The returned promise should only resolve after every promise has either fulfilled or rejected.

For example, suppose the second promise finishes before the first one. Even then, the result array must still place the first promise's result at index `0` and the second promise's result at index `1`.

The constraints are small:

- `1 <= functions.length <= 10`

This means performance is not a major issue from an asymptotic perspective. However, the problem is fundamentally about correctly coordinating asynchronous operations, preserving ordering, and handling both success and failure states.

A naive implementation can easily fail in several ways:

- Losing the original ordering because promises resolve at different times
- Rejecting early when one promise fails
- Forgetting to wait for all promises
- Never resolving the final promise if one rejection is unhandled
- Incorrectly tracking completion count

The problem guarantees that every array element is a callable function returning a promise, so we do not need to validate input types.

## Approaches

### Brute Force Approach

The simplest idea is to process every promise sequentially.

We could execute the first function, wait for its promise to settle, store its result, then move to the next function and repeat until all functions are processed.

This approach works because every promise result is captured correctly and stored in order. Since we wait for each operation individually, handling fulfillment and rejection becomes straightforward.

However, this solution is inefficient because it destroys parallelism.

If one promise takes 100 milliseconds and another takes 200 milliseconds, sequential execution takes 300 milliseconds total. The problem expects all promises to execute concurrently, meaning the total runtime should instead be determined only by the slowest promise.

Sequential execution therefore wastes time unnecessarily.

### Optimal Approach

The key insight is that all promises can be started immediately and independently.

Instead of waiting for one promise before starting the next, we invoke every function right away. Each promise gets its own fulfillment handler and rejection handler.

We maintain:

- A results array of fixed size
- A counter tracking how many promises have settled

Whenever a promise resolves or rejects:

1. We store the appropriate result object at the correct index
2. We increment the settled counter
3. If all promises have settled, we resolve the final promise

This approach preserves ordering because every promise writes its result directly into its original index position.

It also ensures true parallel execution because all asynchronous tasks begin immediately.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n) asynchronous steps | O(n) | Executes promises sequentially, loses parallelism |
| Optimal | O(n) | O(n) | Executes all promises concurrently while preserving order |

## Algorithm Walkthrough

1. Create a new promise that will eventually resolve with the final results array.
2. Initialize a results array with the same length as the input array. This array will store the settlement information for each promise.
3. Initialize a counter called `completed` to track how many promises have finished.
4. Iterate through the input array using both index and function reference.
5. For each function:

- Invoke the function immediately to start its asynchronous operation.
- Attach a `.then()` handler for successful resolution.
- Attach a `.catch()` handler for rejection.
6. Inside the `.then()` handler:

- Store:

```
{
    "status": "fulfilled",
    "value": resolved_value
}
```

at the corresponding index in the results array.

- Increment the `completed` counter.
- If `completed == len(functions)`, resolve the outer promise with the results array.
7. Inside the `.catch()` handler:

- Store:

```
{
    "status": "rejected",
    "reason": error
}
```

at the corresponding index.

- Increment the `completed` counter.
- If all promises have settled, resolve the outer promise.
8. After all handlers are attached, execution continues asynchronously until every promise settles.

### Why it works

The algorithm works because every promise independently reports its result into a predetermined index in the results array. Since each promise only writes to its own index, asynchronous completion order does not matter.

The `completed` counter guarantees that the outer promise resolves exactly once, and only after every inner promise has either fulfilled or rejected.

Therefore:

- No promise result is lost
- Ordering is preserved
- Rejections do not terminate execution early
- The final promise resolves only after all work is complete

## Python Solution

```python
from typing import List, Callable, Any
import asyncio

class Solution:
    async def promiseAllSettled(
        self,
        functions: List[Callable[[], asyncio.Future]]
    ) -> List[dict]:

        async def handle(fn):
            try:
                value = await fn()
                return {
                    "status": "fulfilled",
                    "value": value
                }
            except Exception as error:
                return {
                    "status": "rejected",
                    "reason": str(error)
                }

        tasks = [handle(fn) for fn in functions]

        return await asyncio.gather(*tasks)
```

### Implementation Explanation

The implementation uses Python's `asyncio` framework to mirror JavaScript promise behavior.

The helper function `handle()` wraps the execution of a single asynchronous function. Inside this helper:

- We `await` the promise returned by the function
- If execution succeeds, we return a fulfillment object
- If execution raises an exception, we catch it and return a rejection object

This ensures that every asynchronous operation always produces a valid result object instead of propagating exceptions upward.

Next, we create a list of tasks:

```
tasks = [handle(fn) for fn in functions]
```

This immediately schedules all asynchronous operations concurrently.

Finally:

```
return await asyncio.gather(*tasks)
```

waits for all tasks to complete while preserving the original order of the input list.

Unlike JavaScript `Promise.all()`, our helper ensures exceptions are converted into result objects, so `gather()` never fails early.

## Go Solution

```go
package main

import (
	"fmt"
	"sync"
)

type Result struct {
	Status string
	Value  interface{}
	Reason interface{}
}

func promiseAllSettled(
	functions []func() (interface{}, error),
) []Result {

	n := len(functions)

	results := make([]Result, n)

	var wg sync.WaitGroup

	for i, fn := range functions {
		wg.Add(1)

		go func(index int, function func() (interface{}, error)) {
			defer wg.Done()

			value, err := function()

			if err != nil {
				results[index] = Result{
					Status: "rejected",
					Reason: err.Error(),
				}
			} else {
				results[index] = Result{
					Status: "fulfilled",
					Value:  value,
				}
			}
		}(i, fn)
	}

	wg.Wait()

	return results
}

func main() {
	functions := []func() (interface{}, error){
		func() (interface{}, error) {
			return 15, nil
		},
		func() (interface{}, error) {
			return nil, fmt.Errorf("Error")
		},
	}

	results := promiseAllSettled(functions)

	fmt.Println(results)
}
```

### Go Specific Notes

The Go implementation uses goroutines to achieve parallel execution.

A `sync.WaitGroup` is used to wait until all goroutines complete. Each goroutine writes directly into its assigned index inside the results slice, preserving order regardless of completion timing.

Unlike Python or JavaScript, Go does not have built in promises. Instead, asynchronous behavior is represented here using goroutines and error returning functions.

The `interface{}` type allows storing arbitrary fulfilled values or rejection reasons.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each promise is started and processed exactly once |
| Space | O(n) | Results array and task tracking require linear space |

The algorithm processes each asynchronous function one time and stores one result object per function.

Although the promises execute concurrently in real time, the algorithmic work itself is linear in the number of input functions.

The space complexity is also linear because we must maintain a result entry for every promise.

## Edge Cases

### Promises Completing Out of Order

One of the most important edge cases is when promises finish in a different order from the input array.

For example, the second promise may resolve before the first promise. A naive implementation that simply appends results as promises finish would produce the wrong ordering.

Our implementation avoids this problem by storing every result directly at its original index position.

### Rejected Promises

Another common bug occurs when a promise rejects. Many asynchronous aggregation utilities fail immediately when any task throws an exception.

This problem specifically requires collecting both successful and failed outcomes together.

Our implementation wraps every asynchronous call inside a `try/except` block so that rejections become structured result objects instead of uncaught exceptions.

### All Promises Rejecting

Some implementations accidentally assume at least one promise succeeds.

For example:

```
[
    reject("A"),
    reject("B")
]
```

The correct output must still contain both rejection objects.

Our solution handles this naturally because every promise is processed independently and every outcome contributes one entry to the results array.
