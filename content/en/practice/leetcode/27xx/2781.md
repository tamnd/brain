# LeetCode 2781, Length of the Longest Valid Substring

## Problem Understanding

The problem gives us a string `word` and a list of forbidden strings called `forbidden`.

We need to find the length of the longest substring of `word` such that none of its internal substrings appear in the forbidden list.

This detail is extremely important. We are not checking whether the substring itself is forbidden. Instead, every possible contiguous substring inside it must also avoid the forbidden set.

For example, suppose:

```
word = "cbaaaabc"
forbidden = ["aaa", "cb"]
```

The substring `"aabc"` is valid because neither `"aaa"` nor `"cb"` appears anywhere inside it. However, `"cbaa"` is invalid because it contains `"cb"`.

The constraints are large:

- `word.length` can be up to `100000`
- `forbidden.length` can also be up to `100000`
- each forbidden string has length at most `10`

The small maximum forbidden length is the key observation that enables an efficient solution.

A naive approach that checks every substring against every forbidden pattern would be far too slow because there are `O(n^2)` substrings in a string of length `n`.

Several edge cases are important:

- The forbidden list may contain single-character strings
- The entire word may already be valid
- Every substring longer than `1` may be invalid
- Forbidden strings may overlap with one another
- Multiple forbidden patterns may end at the same index
- The answer could be `0` only if every character itself is forbidden

The restriction that forbidden strings have length at most `10` strongly suggests that we only ever need to inspect a small window around each position.

## Approaches

### Brute Force Approach

The brute-force solution considers every possible substring of `word`.

For each substring:

1. Generate the substring
2. Check every internal substring
3. Determine whether any internal substring belongs to `forbidden`

If none are forbidden, update the answer.

This works because it directly follows the problem definition. However, it is computationally infeasible.

There are `O(n^2)` substrings, and validating each substring can itself require `O(n^2)` work in the worst case. Even with optimizations, this approach becomes far too slow for `n = 100000`.

### Key Insight

The crucial observation is that every forbidden string has length at most `10`.

That means when we extend a substring to the right, the only new forbidden substrings that could appear must end at the current position and have length at most `10`.

This allows us to use a sliding window approach.

We maintain a valid window `[left, right]`.

For every `right` position:

- Look backward at most `10` characters
- Check whether any substring ending at `right` is forbidden
- If a forbidden substring is found, move `left` just past the start of that forbidden substring

This ensures the current window always remains valid.

Using a hash set for forbidden strings gives constant-time lookup for candidate substrings.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n³) or worse | O(1) excluding substring storage | Checks all substrings explicitly |
| Optimal Sliding Window | O(n × 10) | O(k) | Uses bounded backward checking because forbidden length ≤ 10 |

Here, `k` is the number of forbidden strings.

## Algorithm Walkthrough

1. Store all forbidden strings in a hash set.

A hash set allows constant-time membership checks. Since we will repeatedly test whether short substrings are forbidden, fast lookup is essential.
2. Initialize two variables:

- `left = 0`, the left boundary of the current valid window
- `best = 0`, the maximum valid substring length found so far
3. Iterate through the string using `right` as the ending position of the current window.
4. For each `right`, examine substrings ending at `right`.

Since forbidden strings have maximum length `10`, we only need to check:

```
word[start:right+1]
```

where:

```sql
start ranges from right down to max(left, right - 9)
```
5. If a forbidden substring is found:

- The substring starts at `start`
- Any valid window must begin after this forbidden substring starts
- Update:

```
left = start + 1
```

This removes the forbidden substring from the window.
6. Continue checking shorter substrings ending at `right`.

We still continue because another forbidden substring could force `left` even farther right.
7. After processing all candidate substrings ending at `right`, the window `[left, right]` is guaranteed valid.
8. Update the answer:

```
best = max(best, right - left + 1)
```
9. Return `best`.

### Why it works

The sliding window invariant is:

> After processing each `right`, the window `[left, right]` contains no forbidden substring.

Whenever a forbidden substring appears, we move `left` past its starting index, guaranteeing that forbidden substring can no longer exist inside the window.

Because we check all possible forbidden substring lengths ending at each position, no invalid window can survive undetected.

Since `left` only moves forward, the algorithm remains efficient.

## Python Solution

```python
from typing import List

class Solution:
    def longestValidSubstring(self, word: str, forbidden: List[str]) -> int:
        forbidden_set = set(forbidden)

        left = 0
        best = 0
        n = len(word)

        for right in range(n):
            # Check substrings ending at 'right'
            # Only lengths up to 10 matter
            for start in range(right, max(left - 1, right - 10), -1):
                current = word[start:right + 1]

                if current in forbidden_set:
                    left = start + 1
                    break

            best = max(best, right - left + 1)

        return best
```

### Implementation Explanation

The solution begins by converting the forbidden list into a set for fast lookup.

The variable `left` tracks the beginning of the current valid window, while `best` stores the maximum valid length encountered so far.

The outer loop expands the window one character at a time using `right`.

For every `right`, the inner loop checks substrings ending at `right` and going backward up to `10` characters. This bound exists because forbidden strings cannot exceed length `10`.

If a forbidden substring is detected, the left boundary is moved just beyond the forbidden substring's starting position. This immediately restores validity.

The `break` statement is important. Since we scan from longer suffixes toward shorter ones, once we find a forbidden substring, shifting `left` removes it and all longer substrings beginning even farther left are already irrelevant.

Finally, after ensuring the current window is valid, we update the maximum length.

## Go Solution

```go
package main

func longestValidSubstring(word string, forbidden []string) int {
	forbiddenSet := make(map[string]bool)

	for _, s := range forbidden {
		forbiddenSet[s] = true
	}

	left := 0
	best := 0
	n := len(word)

	for right := 0; right < n; right++ {
		startLimit := right - 9
		if startLimit < left {
			startLimit = left
		}

		for start := right; start >= startLimit; start-- {
			current := word[start : right+1]

			if forbiddenSet[current] {
				left = start + 1
				break
			}
		}

		length := right - left + 1
		if length > best {
			best = length
		}
	}

	return best
}
```

### Go-Specific Notes

The Go version uses a `map[string]bool` instead of a Python set.

Substring slicing in Go uses:

```
word[start:right+1]
```

Since strings in Go are byte slices and the input only contains lowercase English letters, byte indexing is completely safe here.

The algorithm and complexity remain identical to the Python implementation.

## Edge Cases

### Single-Character Forbidden Strings

If the forbidden list contains individual characters such as `"a"` or `"e"`, then any substring containing those characters becomes invalid immediately.

This can cause the sliding window to collapse frequently, even to an empty window. The implementation handles this naturally because forbidden substrings of length `1` are checked exactly like longer forbidden strings.

### Overlapping Forbidden Substrings

Consider:

```
word = "ababab"
forbidden = ["aba"]
```

The forbidden substring appears multiple times with overlap. A naive implementation may incorrectly move the left boundary too little or too much.

The sliding window approach correctly adjusts `left` whenever a forbidden substring ending at the current position is found, ensuring no overlap survives inside the valid window.

### Entire Word Already Valid

If no forbidden substring ever appears inside `word`, the optimal answer is simply the full length of the string.

The implementation naturally handles this because `left` never moves from `0`, so the window continuously expands until it covers the entire string.
