## Solution

## Problem Understanding

The problem gives us a table named `Transactions`, where each row represents a single transaction made by a customer on a specific date. Every transaction has four fields:

- `transaction_id`, the unique identifier for the transaction
- `customer_id`, the customer who made the transaction
- `transaction_date`, the date of the transaction
- `amount`, the amount spent in that transaction

The statement guarantees that `(customer_id, transaction_date)` is unique. This means a customer can have at most one transaction per day. That guarantee is important because it simplifies the logic for detecting consecutive days.

We need to identify all periods where a customer made transactions for at least three consecutive days, and where the transaction amount strictly increased every day during that period.

A valid sequence must satisfy two conditions simultaneously:

1. The dates must be consecutive calendar days.
2. The amounts must be strictly increasing from one day to the next.

For every qualifying sequence, we must return:

- `customer_id`
- the first date in the sequence, called `consecutive_start`
- the last date in the sequence, called `consecutive_end`

The result must be sorted by:

1. `customer_id`
2. `consecutive_start`
3. `consecutive_end`

all in ascending order.

A very important detail is that longer valid streaks should be returned as a single interval. For example, if a customer has increasing consecutive transactions for four days in a row, we return one interval spanning all four days, not multiple overlapping intervals of length three.

Another important detail is that customers may have multiple valid streaks separated by gaps in dates or decreases in transaction amounts.

Some edge cases that can easily cause bugs are:

- Transactions with missing dates between them
- Consecutive dates where the amount decreases or stays equal
- Streaks longer than three days
- Customers with only one or two transactions
- Multiple separate valid streaks for the same customer

Because this is a database problem, the intended solution relies heavily on SQL window functions such as `LAG`, `ROW_NUMBER`, and cumulative grouping logic.

## Approaches

### Brute Force Approach

A brute force solution would process each customer's transactions in chronological order and try every possible starting point.

For each transaction:

1. Attempt to extend a sequence day by day.
2. Check whether the next transaction is exactly one day later.
3. Check whether the amount is strictly larger.
4. Continue until the sequence breaks.
5. If the sequence length is at least three, record the interval.

This works because it explicitly verifies every possible valid streak.

However, this approach becomes inefficient because many ranges are checked repeatedly. If a customer has many transactions, the same subarrays are revisited over and over again.

In SQL, this type of procedural logic is also awkward and inefficient compared to set-based operations.

### Optimal Approach

The key observation is that we only care about whether the current row continues the previous valid streak.

For every transaction, we compare it with the previous transaction for the same customer:

- Is the date exactly one day later?
- Is the amount strictly larger?

If both conditions hold, the current transaction belongs to the same streak. Otherwise, a new streak begins.

Once we identify where streaks start, we can assign a group identifier to every streak using a cumulative sum. Then we aggregate each group to compute:

- minimum date
- maximum date
- count of rows

Finally, we keep only groups with at least three transactions.

This transforms the problem from checking many overlapping ranges into a single linear pass over ordered transactions.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeatedly scans overlapping transaction ranges |
| Optimal | O(n log n) | O(n) | Uses sorting and SQL window functions to build streak groups efficiently |

The `O(n log n)` complexity comes primarily from sorting transactions by customer and date.

## Algorithm Walkthrough

### Step 1: Sort transactions by customer and date

We first process each customer's transactions in chronological order.

This ordering is essential because consecutive-day checks only make sense when transactions are examined in time order.

### Step 2: Compare each row with the previous transaction

We use the `LAG()` window function to retrieve:

- previous transaction date
- previous transaction amount

for the same customer.

This allows us to determine whether the current transaction continues a valid streak.

The continuation condition is:

```
transaction_date = previous_date + INTERVAL 1 DAY
AND amount > previous_amount
```

If either condition fails, the streak breaks.

### Step 3: Mark streak boundaries

We create a flag:

- `0` if the current row continues the previous streak
- `1` if a new streak starts

The first transaction for every customer always starts a new streak.

### Step 4: Assign a group number

We compute a cumulative sum of the boundary flags.

Every time a new streak begins, the cumulative sum increases.

As a result, all rows belonging to the same valid consecutive increasing streak receive the same group identifier.

### Step 5: Aggregate each group

For every `(customer_id, group_id)` pair, we compute:

- minimum transaction date
- maximum transaction date
- number of transactions

These values represent the streak interval.

### Step 6: Keep only groups of length at least three

The problem requires at least three consecutive days.

Therefore, we filter groups where:

```
COUNT(*) >= 3
```

### Step 7: Return results in sorted order

Finally, we sort by:

1. `customer_id`
2. `consecutive_start`
3. `consecutive_end`

as required.

### Why it works

The algorithm maintains the invariant that every group contains exactly one maximal streak of consecutive dates with strictly increasing amounts.

Whenever the consecutive-date or increasing-amount condition fails, we start a new group. Therefore:

- no invalid rows remain inside a group
- no valid streak is split incorrectly

Since every row belongs to exactly one group, aggregating by group produces all maximal valid streaks.

## Python Solution

LeetCode database problems are solved using SQL rather than procedural Python code. The following is the complete MySQL solution.

```python
# Write your MySQL query statement below

WITH transaction_history AS (
    SELECT
        customer_id,
        transaction_date,
        amount,
        LAG(transaction_date) OVER (
            PARTITION BY customer_id
            ORDER BY transaction_date
        ) AS prev_date,
        LAG(amount) OVER (
            PARTITION BY customer_id
            ORDER BY transaction_date
        ) AS prev_amount
    FROM Transactions
),

grouped_transactions AS (
    SELECT
        customer_id,
        transaction_date,
        amount,
        SUM(
            CASE
                WHEN prev_date IS NULL
                     OR transaction_date != DATE_ADD(prev_date, INTERVAL 1 DAY)
                     OR amount <= prev_amount
                THEN 1
                ELSE 0
            END
        ) OVER (
            PARTITION BY customer_id
            ORDER BY transaction_date
        ) AS group_id
    FROM transaction_history
)

SELECT
    customer_id,
    MIN(transaction_date) AS consecutive_start,
    MAX(transaction_date) AS consecutive_end
FROM grouped_transactions
GROUP BY customer_id, group_id
HAVING COUNT(*) >= 3
ORDER BY customer_id, consecutive_start, consecutive_end;
```

The first CTE, `transaction_history`, retrieves the previous transaction date and amount for each customer using `LAG()`.

The second CTE, `grouped_transactions`, determines whether the current row starts a new streak. If:

- there is no previous row,
- the dates are not consecutive,
- or the amount does not increase,

then a new group begins.

A cumulative `SUM()` assigns a unique `group_id` to each streak.

Finally, we group by `customer_id` and `group_id` to calculate the start and end dates for each streak. The `HAVING COUNT(*) >= 3` clause ensures that only qualifying streaks are returned.

## Go Solution

Database problems on LeetCode do not require Go code submissions because the platform expects a pure SQL query.

Still, the same logic can be implemented procedurally in Go if the data were loaded into memory.

```go
package main

import (
	"fmt"
	"sort"
	"time"
)

type Transaction struct {
	TransactionID int
	CustomerID    int
	Date          time.Time
	Amount        int
}

type Result struct {
	CustomerID      int
	ConsecutiveStart time.Time
	ConsecutiveEnd   time.Time
}

func findConsecutiveTransactions(transactions []Transaction) []Result {
	sort.Slice(transactions, func(i, j int) bool {
		if transactions[i].CustomerID == transactions[j].CustomerID {
			return transactions[i].Date.Before(transactions[j].Date)
		}
		return transactions[i].CustomerID < transactions[j].CustomerID
	})

	var results []Result

	n := len(transactions)
	i := 0

	for i < n {
		start := i
		j := i + 1

		for j < n &&
			transactions[j].CustomerID == transactions[j-1].CustomerID {

			expectedDate := transactions[j-1].Date.AddDate(0, 0, 1)

			if transactions[j].Date.Equal(expectedDate) &&
				transactions[j].Amount > transactions[j-1].Amount {
				j++
			} else {
				break
			}
		}

		if j-start >= 3 {
			results = append(results, Result{
				CustomerID:       transactions[start].CustomerID,
				ConsecutiveStart: transactions[start].Date,
				ConsecutiveEnd:   transactions[j-1].Date,
			})
		}

		if j == i+1 {
			i++
		} else {
			i = j
		}
	}

	return results
}

func main() {
	fmt.Println("Consecutive transaction detection")
}
```

The Go implementation uses sorting and linear scanning instead of SQL window functions.

One Go-specific detail is date handling. The code uses `time.Time` and `AddDate(0, 0, 1)` to check for consecutive calendar days safely.

Another detail is slice traversal. Since Go does not provide built-in window functions like SQL, we manually compare adjacent rows during iteration.

## Edge Cases

### Consecutive Dates but Equal Amounts

A common bug is treating non-decreasing amounts as valid. The problem explicitly requires strictly increasing amounts.

For example:

```
100 -> 100 -> 120
```

must fail immediately at the second transaction.

The implementation handles this correctly with:

```
amount <= prev_amount
```

which starts a new group whenever the increase is not strict.

### Missing Calendar Dates

Another subtle issue is gaps in dates.

For example:

```
2023-05-01
2023-05-03
2023-05-04
```

Even though amounts may increase, May 2 is missing, so the sequence is invalid.

The query correctly checks:

```
transaction_date = DATE_ADD(prev_date, INTERVAL 1 DAY)
```

which enforces exact day-to-day continuity.

### Long Valid Sequences

A naive implementation might output overlapping subranges for long streaks.

For example, a five-day streak could incorrectly produce:

```
Day1-Day3
Day2-Day4
Day3-Day5
```

The grouping strategy avoids this entirely. All rows in the same uninterrupted streak receive the same `group_id`, so only one maximal interval is returned.

### Multiple Independent Streaks for One Customer

Customers may have several separate valid periods.

The implementation correctly creates a new group whenever continuity breaks, allowing the same customer to appear multiple times in the final output.
