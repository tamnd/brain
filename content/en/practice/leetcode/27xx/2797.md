## Solution

## Problem Understanding

This problem asks us to implement a custom version of partial function application. We are given two inputs:

- A function `fn`
- An array `args`

We must return a new function called `partialFn`.

When `partialFn` is later called with additional arguments, called `restArgs`, we combine those arguments with the original `args` array according to a special placeholder rule.

The key detail is that the string `"_"` inside `args` acts as a placeholder. Each placeholder should be replaced by values from `restArgs`, in order from left to right.

After all placeholders are filled, if there are still unused values remaining in `restArgs`, those values should be appended to the end of the argument list.

Finally, the resulting argument list should be passed into `fn` as separate positional arguments, not as a single array.

For example, suppose:

```
args = [1, "_", 3]
restArgs = [2, 4]
```

The placeholder `"_"` gets replaced by `2`, producing:

```
[1, 2, 3]
```

There is still one unused value, `4`, so it gets appended:

```
[1, 2, 3, 4]
```

Then we call:

```
fn(1, 2, 3, 4)
```

The constraints are important:

- `args.length` and `restArgs.length` can each be as large as `5 * 10^4`
- The number of placeholders is guaranteed to be less than or equal to `restArgs.length`

These constraints tell us that we need a linear time solution. Any approach that repeatedly inserts into arrays or creates excessive copies could become inefficient.

The problem also guarantees that there are enough `restArgs` values to fill every placeholder, so we never need to worry about missing replacement values.

Several edge cases are important:

- `args` may contain no placeholders at all
- Every element in `args` could be a placeholder
- `restArgs` may contain extra unused values that must be appended
- The function may accept fewer or more arguments than originally expected, JavaScript allows this naturally
- The placeholder is specifically the string `"_"`, not any falsy value or empty string

## Approaches

### Brute Force Approach

A straightforward solution is to repeatedly modify the original `args` array whenever a placeholder is encountered.

One possible brute force implementation would:

1. Scan through `args`
2. Whenever a placeholder is found, remove it and insert a value from `restArgs`
3. After processing all placeholders, append any remaining values from `restArgs`
4. Call `fn`

This works because every placeholder is replaced in order, exactly matching the problem requirements.

However, repeatedly modifying arrays can become expensive. Array insertions or removals may require shifting many elements. With up to `5 * 10^4` elements, this can degrade toward quadratic behavior in some implementations.

The brute force idea is conceptually simple, but it performs unnecessary array mutations.

### Optimal Approach

The key observation is that we never actually need to modify the original array in place.

Instead, we can build a new result array in a single left-to-right pass.

While iterating through `args`:

- If the current element is not `"_"`, copy it directly
- If it is `"_"`, consume the next value from `restArgs`

After finishing the scan, append any remaining `restArgs` values.

This approach is efficient because:

- Every element is processed exactly once
- No costly insertions or removals occur
- The logic naturally preserves ordering

The resulting complexity is linear in the total number of elements.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeated array modifications can cause shifting |
| Optimal | O(n + m) | O(n + m) | Single pass construction of result array |

Here, `n` is the size of `args` and `m` is the size of `restArgs`.

## Algorithm Walkthrough

1. Create and return a new function `partialFn`.

The problem requires partial application behavior, so our outer function returns another function that will later receive `restArgs`.
2. Initialize an empty array called `mergedArgs`.

This array will store the final argument sequence passed into `fn`.
3. Maintain an index called `restIndex`.

This index tracks how many values from `restArgs` have already been used to replace placeholders.
4. Iterate through every element in `args`.

For each value:

- If the value is `"_"`, take `restArgs[restIndex]`, append it to `mergedArgs`, and increment `restIndex`
- Otherwise, append the original value directly

This preserves the original order while replacing placeholders correctly.
5. After processing all elements of `args`, append any unused values from `restArgs`.

These are the extra arguments that were not needed for placeholders.
6. Call `fn` using the spread operator.

The problem explicitly states that the arguments must be passed separately, not as an array.
7. Return the result of `fn`.

### Why it works

The algorithm maintains a simple invariant:

- Every placeholder consumes exactly one value from `restArgs`, in left-to-right order
- Every non-placeholder value remains in its original position
- Any unused `restArgs` values are appended afterward

Because we process `args` sequentially and consume `restArgs` sequentially, the final constructed array exactly matches the required behavior.

## Python Solution

```python
from typing import Any, Callable, List

class Solution:
    def partial(self, fn: Callable[..., Any], args: List[Any]) -> Callable[..., Any]:

        def partialFn(*restArgs: Any) -> Any:
            merged_args = []
            rest_index = 0

            for value in args:
                if value == "_":
                    merged_args.append(restArgs[rest_index])
                    rest_index += 1
                else:
                    merged_args.append(value)

            while rest_index < len(restArgs):
                merged_args.append(restArgs[rest_index])
                rest_index += 1

            return fn(*merged_args)

        return partialFn
```

The implementation directly follows the algorithm described earlier.

The outer `partial` method receives the original function and the initial argument list. It returns the nested function `partialFn`, which captures both `fn` and `args` through closure.

Inside `partialFn`, we create `merged_args` to hold the final argument sequence. We also track `rest_index`, which points to the next unused value in `restArgs`.

The loop through `args` handles placeholder replacement. Whenever `"_"` appears, we consume the next available value from `restArgs`. Otherwise, we preserve the original value.

After the loop, there may still be unused values in `restArgs`. These are appended to the end exactly as required by the problem statement.

Finally, we call `fn(*merged_args)` so that each value becomes an independent positional argument.

## Go Solution

```go
package main

type Fn func(...interface{}) interface{}

func partial(fn Fn, args []interface{}) func(...interface{}) interface{} {
	return func(restArgs ...interface{}) interface{} {
		mergedArgs := make([]interface{}, 0, len(args)+len(restArgs))
		restIndex := 0

		for _, value := range args {
			if str, ok := value.(string); ok && str == "_" {
				mergedArgs = append(mergedArgs, restArgs[restIndex])
				restIndex++
			} else {
				mergedArgs = append(mergedArgs, value)
			}
		}

		for restIndex < len(restArgs) {
			mergedArgs = append(mergedArgs, restArgs[restIndex])
			restIndex++
		}

		return fn(mergedArgs...)
	}
}
```

The Go implementation follows the same logic as the Python version, but there are several language-specific details.

Go does not support variadic arguments of arbitrary types directly without `interface{}`, so both `args` and `restArgs` use `[]interface{}`.

To detect placeholders, we perform a type assertion to check whether the value is a string equal to `"_"`.

The spread syntax in Go is written as:

```go
fn(mergedArgs...)
```

Unlike Python, Go requires explicit slice expansion using `...`.

The solution preallocates capacity for `mergedArgs`, which slightly improves efficiency by reducing reallocations.

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n + m) | Each element of `args` and `restArgs` is processed at most once |
| Space | O(n + m) | The merged argument array stores all resulting arguments |

The algorithm performs a single pass through `args` and may append remaining elements from `restArgs`. No nested loops or repeated array shifts occur, so the runtime scales linearly with the total number of inputs.

The additional memory usage comes from constructing the final merged argument list.

## Edge Cases

One important edge case occurs when there are no placeholders in `args`. In this situation, every value from `restArgs` should simply be appended to the end. A buggy implementation might incorrectly ignore `restArgs` entirely if no placeholders are found. Our implementation avoids this by always appending remaining `restArgs` after processing the original array.

Another important case is when every element in `args` is a placeholder. This stresses whether the replacement logic correctly consumes values sequentially without skipping or duplicating indices. Since our solution uses a dedicated `rest_index` that advances exactly once per placeholder, every replacement occurs correctly.

A subtle edge case involves distinguishing the placeholder string `"_"` from other similar values. For example, an empty string `""`, the string `"__"`, or the value `None` should not be treated as placeholders. Our implementation checks specifically for equality with `"_"`, ensuring only the exact required token triggers replacement behavior.

A final edge case involves extra unused `restArgs` values. Some implementations may accidentally discard them after placeholder replacement. The problem explicitly requires these values to be appended at the end, and our final loop guarantees that behavior.
