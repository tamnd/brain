# LeetCode 2742 - Painting the Walls

## Problem Understanding

The problem gives us two arrays, `cost` and `time`, where each index represents one wall. If we hire the paid painter for wall `i`, we must pay `cost[i]`, and that painter will spend `time[i]` units of time painting that wall.

At the same time, there is also a free painter. The free painter can paint any wall in exactly `1` unit of time and costs nothing. However, the free painter is only allowed to work while the paid painter is busy. This restriction is the most important part of the problem.

Suppose we hire the paid painter for a wall that takes `3` units of time. During those `3` units, the free painter can independently paint up to `3` additional walls for free. Therefore, choosing a paid wall with `time[i] = 3` effectively allows us to complete `4` walls total:

- The paid painter paints wall `i`
- The free painter can paint up to `3` more walls during that time

The goal is to paint all `n` walls while minimizing the total money spent.

The constraints are important:

- `n <= 500`
- `cost[i] <= 10^6`
- `time[i] <= 500`

A brute force search over all subsets of walls would be exponential and far too slow. Since `n` is moderate, this strongly suggests a dynamic programming solution.

An important observation is that every wall chosen for the paid painter contributes:

- One directly painted wall
- `time[i]` units of free painter capacity

So each paid wall contributes coverage of `time[i] + 1` total walls.

Several edge cases can easily cause bugs:

- A wall with very large `time[i]` may cover all remaining walls by itself
- Multiple cheap walls may be better than one expensive wall
- The free painter cannot work unless the paid painter is active
- We must avoid double counting wall coverage
- We only care about painting all walls, not the exact scheduling order

## Approaches

### Brute Force Approach

A brute force solution would try every possible subset of walls assigned to the paid painter.

For each subset:

- Compute the total cost
- Compute the total available painting capacity

If the paid walls together provide enough total coverage to paint all walls, then the subset is valid.

Specifically, if we choose paid walls with total contribution:

$$\sum (time[i] + 1)$$

and that sum is at least `n`, then all walls can be completed.

This approach is correct because it checks every possible combination. However, there are `2^n` subsets, which becomes impossible when `n = 500`.

### Optimal Dynamic Programming Approach

The key insight is that this is essentially a knapsack style optimization problem.

Each paid wall acts like an item:

- Cost = `cost[i]`
- Coverage gained = `time[i] + 1`

We want to achieve at least `n` total coverage while minimizing total cost.

This transforms the problem into:

- Minimum cost to reach coverage `n`

Dynamic programming works well because:

- The state space is only `O(n)`
- Each wall is processed once
- Transitions resemble 0/1 knapsack

We define:

$$dp[j]$$

as the minimum cost needed to cover exactly `j` walls.

Then each wall updates the DP array in reverse order.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(1) | Tries every subset of paid walls |
| Optimal DP | O(n^2) | O(n) | Knapsack style DP minimizing cost |

## Algorithm Walkthrough

1. Let `n` be the number of walls.
2. Create a DP array of size `n + 1`.

Each position `dp[j]` represents the minimum cost required to paint `j` walls.
3. Initialize all values to infinity except:

```
dp[0] = 0
```

This means painting zero walls costs nothing.

1. Process each wall one by one.

If we pay for wall `i`:

- We spend `cost[i]`
- We gain coverage of `time[i] + 1`
2. For each wall, iterate backward through the DP array.

Backward iteration is necessary because each wall can only be used once, exactly like 0/1 knapsack.
3. Suppose the current wall contributes:

```
gain = time[i] + 1
```

Then:

```
newCovered = min(n, currentCovered + gain)
```

We cap at `n` because covering more than `n` walls provides no extra benefit.

1. Update the DP state:

```
dp[newCovered] =
    min(dp[newCovered],
        dp[currentCovered] + cost[i])
```

1. After processing all walls, return:

```
dp[n]
```

This is the minimum cost needed to paint all walls.

### Why it works

The DP maintains the invariant that `dp[j]` always stores the minimum possible cost to achieve coverage of `j` walls using processed walls so far.

Each wall can either:

- Be skipped
- Be chosen as a paid wall

Choosing a wall increases total coverage by `time[i] + 1` and increases total cost by `cost[i]`.

Because every valid subset is considered exactly once through DP transitions, the final result is the minimum possible total cost.

## Python Solution

```python
from typing import List

class Solution:
    def paintWalls(self, cost: List[int], time: List[int]) -> int:
        n = len(cost)

        INF = float("inf")
        dp = [INF] * (n + 1)
        dp[0] = 0

        for c, t in zip(cost, time):
            gain = t + 1

            for covered in range(n, -1, -1):
                new_covered = min(n, covered + gain)

                dp[new_covered] = min(
                    dp[new_covered],
                    dp[covered] + c
                )

        return dp[n]
```

The implementation directly follows the knapsack style dynamic programming approach.

The DP array stores the minimum cost needed to cover a certain number of walls. Initially, only `dp[0]` is valid because painting zero walls costs zero money.

For every wall, we compute how much total coverage it contributes. Since hiring the paid painter for wall `i` also enables the free painter for `time[i]` units, the total contribution becomes:

```python
time[i] + 1
```

The backward iteration ensures that each wall is used at most once. This is the standard pattern for 0/1 knapsack problems.

The transition updates the best known cost for the new coverage amount.

Finally, `dp[n]` contains the minimum cost required to paint all walls.

## Go Solution

```go
package main

import "math"

func paintWalls(cost []int, time []int) int {
	n := len(cost)

	dp := make([]int, n+1)

	for i := 1; i <= n; i++ {
		dp[i] = math.MaxInt32
	}

	dp[0] = 0

	for i := 0; i < n; i++ {
		gain := time[i] + 1

		for covered := n; covered >= 0; covered-- {
			newCovered := covered + gain

			if newCovered > n {
				newCovered = n
			}

			if dp[covered]+cost[i] < dp[newCovered] {
				dp[newCovered] = dp[covered] + cost[i]
			}
		}
	}

	return dp[n]
}
```

The Go implementation mirrors the Python solution closely.

Since Go does not have a built in infinity value for integers, we initialize unreachable states using `math.MaxInt32`.

The DP array uses integer slices, and updates are performed in reverse order to preserve the 0/1 knapsack behavior.

Integer overflow is not an issue because the maximum possible answer is bounded by:

$$500 \times 10^6$$

which fits safely within Go's 64 bit integer range, and also within signed 32 bit limits for this problem.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n^2) | For each wall, we iterate through all DP states |
| Space | O(n) | DP array of size `n + 1` |

The algorithm processes each of the `n` walls once. For every wall, we potentially update all `n + 1` DP states. Therefore, the total runtime is quadratic.

The memory usage is linear because only a one dimensional DP array is maintained.

## Edge Cases

One important edge case occurs when a single paid wall can effectively cover every wall. For example, if `time[i] + 1 >= n`, then hiring that one paid painter is sufficient to finish the entire job. A buggy implementation might continue counting unnecessary walls or fail to cap coverage at `n`. The solution handles this correctly using:

```
min(n, covered + gain)
```

Another important case occurs when many small cheap walls outperform one expensive wall with large time coverage. A greedy strategy that always chooses the largest `time[i]` would fail here. The dynamic programming approach avoids this issue because it evaluates every valid subset combination.

A third important edge case involves the free painter restriction. The free painter only works while the paid painter is occupied. Some incorrect solutions mistakenly assume the free painter can always work independently. The key observation that each paid wall contributes exactly `time[i] + 1` total wall coverage correctly models the synchronization requirement.

A final subtle edge case occurs when coverage exceeds `n`. Without clamping the DP state to `n`, the algorithm could attempt out of bounds updates or maintain unnecessary states. The implementation safely caps all coverage values at `n`.
