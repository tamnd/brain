# LeetCode 2787 - Ways to Express an Integer as Sum of Powers

## Problem Understanding

The problem gives us two positive integers, `n` and `x`. We must determine how many different ways we can represent `n` as the sum of distinct positive integers raised to the power `x`.

More formally, we want to count the number of unique sets:

$$[n_1, n_2, ..., n_k]$$

such that:

$n = n_1^x + n_2^x + \cdots + n_k^x$

Every integer in the set must be unique. This means we cannot reuse the same base number multiple times.

For example, if `n = 10` and `x = 2`, we are looking for combinations of distinct perfect squares whose sum equals `10`. The only valid combination is:

$$1^2 + 3^2 = 1 + 9 = 10$$

so the answer is `1`.

The important detail is that order does not matter. The combinations `{1, 3}` and `{3, 1}` represent the same solution and must only be counted once.

The constraints are relatively small:

- `1 <= n <= 300`
- `1 <= x <= 5`

These constraints strongly suggest that dynamic programming is appropriate. Since `n` is at most `300`, we can afford algorithms with roughly `O(n^2)` complexity.

Another key observation is that the number of usable integers is limited. For a given `x`, only integers satisfying:

$i^x \le n$

can participate in the sum.

Important edge cases include:

- `x = 1`, where every integer contributes itself, making the problem equivalent to counting subsets with unique numbers.
- Very small `n`, such as `1`, where only one representation may exist.
- Cases where no valid combination exists.
- Ensuring uniqueness, since allowing repeated integers would overcount solutions.

## Approaches

### Brute Force Approach

A brute force solution would recursively try every possible subset of integers whose `x`th powers do not exceed `n`.

We first generate all valid powers:

$$1^x, 2^x, 3^x, ...$$

up to `n`.

Then, for every number, we make two choices:

1. Include it in the current sum.
2. Skip it.

This forms a classic subset exploration tree.

The brute force solution is correct because it explicitly checks every possible subset of valid powers. If a subset sums to `n`, it contributes one valid answer.

However, this approach is inefficient because the number of subsets grows exponentially. If there are `m` valid powers, the recursion explores up to:

$$2^m$$

states.

Although `m` is not huge for this problem, exponential growth is still undesirable.

### Optimal Dynamic Programming Approach

The key insight is that this problem is actually a variation of the classic subset sum counting problem.

Each usable power value can either:

- contribute once, or
- not contribute at all.

That is exactly the behavior of a 0/1 knapsack problem.

We define a dynamic programming array where:

- `dp[s]` represents the number of ways to form sum `s`.

Initially:

- `dp[0] = 1`

because there is exactly one way to form sum `0`, choose nothing.

For every power value, we update the DP array in reverse order. Reverse traversal is critical because it prevents reusing the same number multiple times.

This converts an exponential brute force search into a polynomial time solution.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^m) | O(m) | Explores every subset recursively |
| Optimal | O(m × n) | O(n) | Uses 0/1 knapsack dynamic programming |

Here, `m` is the number of integers satisfying `i^x <= n`.

## Algorithm Walkthrough

1. Generate all valid powers.

We iterate from `1` upward and compute:

$i^x$

as long as the value does not exceed `n`.

These values become the candidate numbers for the subset sum problem.
2. Initialize the DP array.

We create:

```
dp = [0] * (n + 1)
```

where `dp[s]` stores the number of ways to form sum `s`.

We set:

```
dp[0] = 1
```

because there is one way to make sum `0`.
3. Process each power value.

For every power value `p`, we iterate backward from `n` down to `p`.

Backward iteration is important because it ensures each power is used at most once.
4. Update DP states.

For each sum `s`, we add:

```
dp[s - p]
```

into `dp[s]`.

This means every existing way to form `s - p` can now extend to form `s` by including power `p`.
5. Apply modulo arithmetic.

Since the number of combinations may become large, every update is performed modulo:

```
10**9 + 7
```
6. Return the answer.

The final result is stored in:

```
dp[n]
```

### Why it works

The algorithm maintains the invariant that after processing the first `k` powers, `dp[s]` equals the number of ways to form sum `s` using only those processed powers.

Backward iteration guarantees that each power contributes at most once, which preserves the uniqueness requirement. Since every valid subset is considered exactly once, the final value `dp[n]` is precisely the number of valid representations.

## Python Solution

```python
class Solution:
    def numberOfWays(self, n: int, x: int) -> int:
        MOD = 10**9 + 7

        powers = []
        num = 1

        while (value := num ** x) <= n:
            powers.append(value)
            num += 1

        dp = [0] * (n + 1)
        dp[0] = 1

        for power in powers:
            for current_sum in range(n, power - 1, -1):
                dp[current_sum] = (
                    dp[current_sum] + dp[current_sum - power]
                ) % MOD

        return dp[n]
```

The implementation begins by generating all usable `x`th powers that do not exceed `n`. These values form the candidate set for the subset sum problem.

The DP array stores the number of ways to build every possible sum from `0` to `n`. Setting `dp[0] = 1` establishes the base case.

The nested loops implement the 0/1 knapsack transition. For each power value, the inner loop traverses sums in descending order so that the same power cannot be reused multiple times.

Each update:

```python
dp[current_sum] += dp[current_sum - power]
```

means that every way to form `current_sum - power` can be extended by adding the current power.

Finally, `dp[n]` contains the total number of valid combinations.

## Go Solution

```go
package main

const MOD int = 1_000_000_007

func power(base int, exp int) int {
	result := 1

	for exp > 0 {
		result *= base
		exp--
	}

	return result
}

func numberOfWays(n int, x int) int {
	powers := []int{}

	for num := 1; ; num++ {
		value := power(num, x)

		if value > n {
			break
		}

		powers = append(powers, value)
	}

	dp := make([]int, n+1)
	dp[0] = 1

	for _, p := range powers {
		for sum := n; sum >= p; sum-- {
			dp[sum] = (dp[sum] + dp[sum-p]) % MOD
		}
	}

	return dp[n]
}
```

The Go solution follows the same logic as the Python implementation.

One difference is that Go does not provide a built in exponentiation operator for integers, so a helper `power` function is implemented manually.

The DP array is represented using a slice. Since all values remain within modulo bounds, standard `int` arithmetic is sufficient for the given constraints.

## Edge Cases

One important edge case occurs when `n` itself is a perfect `x`th power. For example, `n = 9` and `x = 2`. A buggy implementation might accidentally miss the single element subset `{3}` if it incorrectly assumes combinations require multiple numbers. The DP solution handles this naturally because every power is processed individually and can independently contribute to a valid sum.

Another important case is when no valid representation exists. For example, `n = 2` and `x = 2`. The only usable square is `1`, and uniqueness prevents using it twice. Some recursive solutions accidentally allow repeated usage and incorrectly return `1`. The backward DP traversal prevents this by enforcing 0/1 usage for every power.

A third important edge case occurs when `x = 1`. In this scenario, every positive integer up to `n` becomes a candidate. This creates many possible subset combinations and can easily expose overcounting bugs. The implementation correctly avoids counting permutations separately because each number is processed exactly once in a fixed order.

Finally, very small values such as `n = 1` are important because they rely entirely on correct initialization of the base case. Setting `dp[0] = 1` ensures that the algorithm can build valid sums starting from zero, allowing `1^x` to correctly contribute one valid representation.
