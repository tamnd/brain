# LeetCode 2733 - Neither Minimum nor Maximum

## Problem Understanding

The problem gives us an array of distinct positive integers and asks us to return any value that is neither the smallest element nor the largest element in the array. If no such value exists, we must return `-1`.

In simpler terms, we need to find an element that lies somewhere in the middle when the numbers are ordered. The exact value does not matter, as long as it is not the minimum or maximum value.

The input array contains only distinct integers, which is important because it guarantees that the minimum and maximum are unique. This removes ambiguity when comparing elements.

The constraints are very small:

- The array length is between `1` and `100`
- Each value is between `1` and `100`
- All values are distinct

Because the input size is tiny, almost any reasonable solution will run efficiently. However, the goal is still to design a clean and optimal solution.

There are several important edge cases to consider:

- Arrays with fewer than three elements cannot contain a valid answer, because every element must be either the minimum or maximum.
- Arrays with exactly three elements contain exactly one valid answer, which is the middle value.
- Since all numbers are distinct, we never need to handle duplicate values affecting the minimum or maximum logic.

## Approaches

### Brute Force Approach

A straightforward approach is to sort the array and return the second element after sorting. Since the array elements are distinct, the first element in sorted order is the minimum and the last element is the maximum. Any element in between is valid.

For arrays with fewer than three elements, we return `-1`.

This approach is correct because sorting fully determines the ordering of the numbers. Once sorted, selecting any non-boundary element guarantees that it is neither the minimum nor maximum.

However, sorting takes `O(n log n)` time, which is more work than necessary because we do not actually need the full ordering of the array.

### Optimal Approach

The key observation is that we only need one element that is neither the smallest nor the largest. We do not need to sort the entire array.

If the array contains at least three elements, we can simply examine the first three numbers. Among any three distinct numbers, one will always be the minimum, one will always be the maximum, and one will be in the middle.

Therefore, we can:

1. Take the first three elements
2. Find the minimum and maximum among them
3. Return the remaining value

This works because the middle value among three distinct numbers is guaranteed to be neither the minimum nor the maximum.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) | O(1) or O(n) | Sort the array and return a middle element |
| Optimal | O(1) | O(1) | Only inspect the first three elements |

## Algorithm Walkthrough

1. Check the size of the array. If the array contains fewer than three elements, immediately return `-1`. With only one or two elements, every element must be either the minimum or maximum.
2. Take the first three elements from the array. Because all values are distinct, these three numbers must contain exactly one minimum, one maximum, and one middle value.
3. Compute the minimum and maximum values among the three numbers. This allows us to identify which value cannot be returned.
4. Iterate through the three elements again. The element that is neither equal to the minimum nor equal to the maximum is the valid answer.
5. Return that middle value.
6. The function technically never reaches the end without returning when the array has at least three distinct elements, but returning `-1` as a fallback keeps the implementation complete.

### Why it works

Among any three distinct integers, exactly one value lies strictly between the other two. That value cannot be the global minimum or global maximum of those three numbers. Since the problem only asks for any value that is neither minimum nor maximum, returning this middle value always satisfies the condition.

## Python Solution

```python
from typing import List

class Solution:
    def findNonMinOrMax(self, nums: List[int]) -> int:
        if len(nums) < 3:
            return -1

        first = nums[0]
        second = nums[1]
        third = nums[2]

        minimum = min(first, second, third)
        maximum = max(first, second, third)

        for value in (first, second, third):
            if value != minimum and value != maximum:
                return value

        return -1
```

The implementation begins by handling the edge case where the array length is less than three. In that situation, there is no valid answer.

Next, the code extracts the first three elements. We then compute the minimum and maximum values among those three numbers using Python's built in `min` and `max` functions.

Finally, we loop through the three values and return the one that is neither the minimum nor the maximum.

The algorithm uses only constant extra space and performs only a small fixed number of operations.

## Go Solution

```go
func findNonMinOrMax(nums []int) int {
    if len(nums) < 3 {
        return -1
    }

    first := nums[0]
    second := nums[1]
    third := nums[2]

    minimum := min(first, min(second, third))
    maximum := max(first, max(second, third))

    values := []int{first, second, third}

    for _, value := range values {
        if value != minimum && value != maximum {
            return value
        }
    }

    return -1
}

func min(a, b int) int {
    if a < b {
        return a
    }
    return b
}

func max(a, b int) int {
    if a > b {
        return a
    }
    return b
}
```

The Go version follows the same logic as the Python implementation. One difference is that Go does not provide built in `min` and `max` functions for integers, so helper functions are implemented manually.

The solution uses slices to store the three values temporarily for iteration. Since the constraints are tiny, there are no concerns about integer overflow or memory usage.

## Edge Cases

One important edge case is an array with only one element. In this scenario, the single value is simultaneously the minimum and maximum. A careless implementation might attempt to access additional elements or incorrectly return the only value. The solution avoids this by immediately returning `-1` when the array length is less than three.

Another important case is an array with exactly two elements. Both elements must be either the minimum or maximum, so no valid answer exists. The implementation handles this using the same early length check.

A third important edge case is an array with exactly three elements. In this situation, there is exactly one valid answer, the middle value. The algorithm is specifically designed around this property, which guarantees correctness while avoiding unnecessary processing.

Finally, unsorted arrays can easily cause mistakes if someone assumes the first or last element is automatically minimum or maximum. The implementation explicitly computes the minimum and maximum values among the chosen elements, ensuring correctness regardless of input order.
