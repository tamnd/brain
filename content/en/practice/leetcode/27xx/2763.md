## Solution

## Problem Understanding

The problem defines the imbalance number of an array based on the gaps between adjacent elements after sorting the array.

Suppose we have a subarray `arr`. If we sort it into `sarr`, we look at every adjacent pair:

```
sarr[i], sarr[i+1]
```

If the difference between them is greater than `1`, then that pair contributes `1` to the imbalance number.

For example:

```
arr = [3,1,4]
sorted(arr) = [1,3,4]
```

The adjacent differences are:

```
3 - 1 = 2  -> contributes 1
4 - 3 = 1  -> contributes 0
```

So the imbalance number is `1`.

The task is to compute the sum of imbalance numbers across every possible non-empty subarray of `nums`.

The input array length is at most `1000`, and every value is also bounded by `n`. This is important because it allows us to use compact frequency arrays or boolean presence arrays efficiently.

A naive approach would generate every subarray, sort it, and compute the imbalance. Since there are `O(n²)` subarrays and sorting each one costs `O(n log n)`, that quickly becomes too slow.

There are several important edge cases:

- Arrays where all elements are equal, because duplicates should not create imbalance.
- Strictly consecutive arrays like `[1,2,3,4]`, where every subarray has imbalance `0`.
- Arrays with large gaps like `[1,5,9]`, where many subarrays contribute imbalance.
- Duplicate values mixed with gaps, because adding a duplicate should not incorrectly increase imbalance.
- Single-element subarrays, which always have imbalance `0` because there are no adjacent pairs after sorting.

The problem guarantees:

- `1 <= nums.length <= 1000`
- `1 <= nums[i] <= nums.length`

This bounded value range is the key observation that enables an efficient solution.

## Approaches

### Brute Force

The most direct solution is to enumerate every subarray.

For each starting index `i`, we expand the subarray one element at a time toward the right. For every subarray, we sort its elements and count how many adjacent sorted values differ by more than `1`.

This works because it directly implements the definition of imbalance.

However, its performance is poor:

- There are `O(n²)` subarrays.
- Sorting each subarray costs up to `O(n log n)`.
- Computing imbalance after sorting costs another `O(n)`.

The overall complexity becomes roughly `O(n³ log n)`, which is too slow for `n = 1000`.

### Key Insight

Instead of recomputing the imbalance from scratch for every subarray, we can build each subarray incrementally.

Suppose we already know the imbalance of:

```
nums[i:j]
```

Now we extend it with a new value `x = nums[j+1]`.

The key observation is that inserting one value only affects nearby numbers:

- If neither `x-1` nor `x+1` exists in the subarray, then `x` creates a new gap, so imbalance increases by `1`.
- If both neighbors exist, then `x` bridges an existing gap, so imbalance decreases by `1`.
- If exactly one neighbor exists, imbalance stays unchanged.
- If `x` already exists, nothing changes.

Because values are bounded by `n`, we can track existence with a boolean array or hash set.

This allows us to update imbalance in `O(1)` time while expanding the subarray, producing an `O(n²)` solution.

## Algorithm Walkthrough

### Optimal Incremental Construction

1. Iterate through every possible starting index `left`.

We will build all subarrays beginning at `left` by extending the right boundary one step at a time.
2. Create a boolean array called `seen`.

Since `nums[i] <= n`, we can safely allocate an array of size `n + 2`.

This structure tells us whether a value already exists in the current subarray.
3. Initialize the current imbalance value to `0`.

This variable stores the imbalance of the current subarray as we extend it.
4. Expand the subarray using a second pointer `right`.

At each step, let:

```
value = nums[right]
```
5. If `value` has not appeared before, determine how it affects imbalance.

There are four possibilities:

- Neither `value-1` nor `value+1` exists:

The new number creates a new disconnected segment in sorted order, so imbalance increases by `1`.
- Both neighbors exist:

Previously there was a gap between `value-1` and `value+1`. Inserting `value` removes that gap, so imbalance decreases by `1`.
- Exactly one neighbor exists:

The new number simply extends an existing consecutive block, so imbalance stays unchanged.
- `value` already exists:

Duplicates do not change sorted adjacency relationships, so imbalance stays unchanged.
6. Mark `value` as seen.
7. Add the current imbalance into the answer.

One subtle detail is that the algorithm temporarily overcounts by `1` whenever a disconnected component is created. To compensate, we subtract `1` for every subarray before adding to the final answer.

An equivalent cleaner formulation is:

- Start imbalance at `-1`
- Then apply the insertion rules directly

This version naturally produces the correct imbalance values.
8. Continue until every subarray has been processed.

### Why it works

The imbalance number depends only on gaps between consecutive distinct values in sorted order. When inserting a new value into the subarray, only the relationships involving `value-1` and `value+1` can change. No other adjacency relationships are affected.

Therefore, we can maintain the imbalance incrementally in constant time per insertion, guaranteeing correctness while avoiding repeated sorting.

## Python Solution

```python
from typing import List

class Solution:
    def sumImbalanceNumbers(self, nums: List[int]) -> int:
        n = len(nums)
        answer = 0

        for left in range(n):
            seen = [False] * (n + 2)
            imbalance = -1

            for right in range(left, n):
                value = nums[right]

                if not seen[value]:
                    imbalance += 1

                    if seen[value - 1]:
                        imbalance -= 1

                    if seen[value + 1]:
                        imbalance -= 1

                    seen[value] = True

                answer += imbalance

        return answer
```

The outer loop fixes the starting index of the subarray.

For each starting position, we reset the `seen` array because we are beginning a completely new collection of subarrays.

The `imbalance` variable is initialized to `-1`. This is a useful trick because the first inserted number always creates one connected component, but a single-element subarray should have imbalance `0`.

As we extend the subarray:

- If the value is new, we first assume it creates a new disconnected segment.
- Then we check whether neighboring values already exist.
- Each neighboring value reduces the number of disconnected gaps.

The resulting `imbalance` is exactly the number of gaps larger than `1` in sorted order.

Finally, we accumulate the imbalance contribution from every subarray.

## Go Solution

```go
func sumImbalanceNumbers(nums []int) int {
	n := len(nums)
	answer := 0

	for left := 0; left < n; left++ {
		seen := make([]bool, n+2)
		imbalance := -1

		for right := left; right < n; right++ {
			value := nums[right]

			if !seen[value] {
				imbalance++

				if seen[value-1] {
					imbalance--
				}

				if seen[value+1] {
					imbalance--
				}

				seen[value] = true
			}

			answer += imbalance
		}
	}

	return answer
}
```

The Go implementation mirrors the Python logic closely.

The `seen` slice is recreated for every starting index. Since values are guaranteed to be at most `n`, a boolean slice of length `n + 2` safely supports checking `value-1` and `value+1` without boundary issues.

Go uses `int` for all arithmetic here, which is sufficient because the maximum possible answer fits comfortably within standard integer limits for the given constraints.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n²) | Every subarray is processed once, and each extension is O(1) |
| Space | O(n) | The `seen` array stores presence information for values `1..n` |

The outer loop runs `n` times, and the inner loop also runs at most `n` times. Every insertion and imbalance update uses constant-time neighbor checks, so the total runtime is quadratic.

The auxiliary space comes from the boolean presence array.

## Edge Cases

### All Elements Are Equal

Consider:

```
[3,3,3,3]
```

A buggy implementation might incorrectly count duplicates as additional gaps. However, duplicates do not create new adjacency differences in sorted order.

The implementation handles this correctly by checking:

```
if not seen[value]:
```

Only the first occurrence of a value affects imbalance.

### Consecutive Numbers in Any Order

Consider:

```
[4,1,3,2]
```

Even though the original order is unsorted, the sorted version becomes:

```
[1,2,3,4]
```

which has no gaps.

A naive incremental method could incorrectly depend on insertion order. This solution avoids that problem because it only tracks whether neighboring values exist, which accurately models the sorted adjacency structure regardless of insertion order.

### A Gap That Later Gets Filled

Consider:

```
[1,3,2]
```

After inserting `3`, there is a gap between `1` and `3`, so imbalance becomes `1`.

When `2` is inserted later, it bridges that gap:

```
[1,2,3]
```

Now the imbalance should return to `0`.

The implementation handles this correctly because when both neighbors exist:

```
if seen[value - 1]:
    imbalance -= 1

if seen[value + 1]:
    imbalance -= 1
```

The newly inserted value merges two consecutive regions into one, removing the imbalance contribution.
