## Solution

## Problem Understanding

The problem asks us to count how many contiguous subarrays satisfy a very specific condition. A subarray is considered continuous if every pair of elements inside it differs by at most `2`.

More formally, for a subarray `nums[i...j]`, we must ensure:

$$|nums[a] - nums[b]| \le 2$$

for every pair of indices `a` and `b` inside the subarray.

A very important observation is that this condition is equivalent to saying:

$$\max(nums[i...j]) - \min(nums[i...j]) \le 2$$

If the largest and smallest values differ by at most `2`, then every pair of numbers inside the subarray also differs by at most `2`.

The input is an integer array `nums`, and the output is the total number of valid subarrays.

The constraints are large:

- `nums.length` can be up to `100000`
- Values can be as large as `10^9`

These constraints immediately tell us that an `O(n^2)` solution will be too slow. A quadratic algorithm would require around `10^10` operations in the worst case, which is far beyond acceptable limits.

Some important edge cases include:

- Arrays with a single element, every subarray is automatically valid.
- Arrays where all values are identical, every possible subarray is valid.
- Arrays with large jumps, forcing the sliding window to shrink frequently.
- Alternating values near the limit, such as `[1,3,1,3]`, where many windows remain valid.
- Very large values, which means we should avoid assumptions about small ranges or counting sort techniques.

## Approaches

### Brute Force Approach

The most straightforward solution is to examine every possible subarray.

For each starting index `i`, we expand the subarray one element at a time toward the right. While expanding, we maintain the current minimum and maximum values. If at any point:

$$\max - \min > 2$$

then the subarray becomes invalid.

This approach is correct because it explicitly checks every subarray and verifies the condition directly.

However, there are `O(n^2)` possible subarrays. Even if minimum and maximum are maintained incrementally, the algorithm still takes quadratic time.

With `n = 100000`, this is far too slow.

### Optimal Approach, Sliding Window with Monotonic Queues

The key insight is that the validity condition depends only on the minimum and maximum values inside the current window.

This naturally suggests a sliding window approach.

We maintain a window `[left, right]` such that:

$$\max(window) - \min(window) \le 2$$

As we expand `right`, we efficiently track:

- The current maximum
- The current minimum

To do this in constant amortized time, we use two monotonic deques:

- An increasing deque for the minimum
- A decreasing deque for the maximum

When the window becomes invalid, we move `left` forward until validity is restored.

For every position `right`, all subarrays ending at `right` and starting anywhere from `left` to `right` are valid. That contributes:

$$right - left + 1$$

new valid subarrays.

This transforms the solution into linear time.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Checks every subarray explicitly |
| Optimal | O(n) | O(n) | Sliding window with monotonic deques |

## Algorithm Walkthrough

1. Initialize two deques:

- `min_deque` stores elements in increasing order.
- `max_deque` stores elements in decreasing order.
2. Initialize:

- `left = 0`
- `result = 0`
3. Iterate through the array using `right`.
4. Insert `nums[right]` into the monotonic structures.

- For `min_deque`, remove elements from the back while they are larger than the current value.
- For `max_deque`, remove elements from the back while they are smaller than the current value.
5. Append the current value to both deques.
6. Check whether the window is valid.

- The minimum is at the front of `min_deque`.
- The maximum is at the front of `max_deque`.
- If `max - min > 2`, the window is invalid.
7. While the window is invalid:

- If `nums[left]` equals the front of `min_deque`, remove it.
- If `nums[left]` equals the front of `max_deque`, remove it.
- Increment `left`.
8. Once the window becomes valid again, count all valid subarrays ending at `right`.

- There are exactly `right - left + 1` such subarrays.
9. Add this count to the result.
10. Continue until the entire array is processed.

### Why it works

The sliding window always maintains the invariant:

$$\max(window) - \min(window) \le 2$$

The monotonic deques ensure that we can retrieve the current minimum and maximum in constant time.

When the window becomes invalid, moving `left` forward eventually removes either the minimum or maximum causing the violation. Once validity is restored, every subarray ending at `right` and beginning within the valid window is guaranteed to satisfy the condition.

Because every element enters and leaves each deque at most once, the algorithm runs in linear time.

## Python Solution

```python
from collections import deque
from typing import List

class Solution:
    def continuousSubarrays(self, nums: List[int]) -> int:
        min_deque = deque()
        max_deque = deque()

        left = 0
        result = 0

        for right, value in enumerate(nums):
            while min_deque and min_deque[-1] > value:
                min_deque.pop()

            while max_deque and max_deque[-1] < value:
                max_deque.pop()

            min_deque.append(value)
            max_deque.append(value)

            while max_deque[0] - min_deque[0] > 2:
                if nums[left] == min_deque[0]:
                    min_deque.popleft()

                if nums[left] == max_deque[0]:
                    max_deque.popleft()

                left += 1

            result += right - left + 1

        return result
```

The implementation directly follows the sliding window strategy.

The two deques maintain the current minimum and maximum values efficiently.

When inserting a new value:

- The minimum deque removes larger values from the back because they can never become the minimum while the current smaller value exists.
- The maximum deque removes smaller values from the back because they can never become the maximum afterward.

The shrinking loop ensures the window always remains valid.

Finally, the expression:

```python
right - left + 1
```

counts all valid subarrays ending at the current position.

## Go Solution

```go
func continuousSubarrays(nums []int) int64 {
    minDeque := []int{}
    maxDeque := []int{}

    left := 0
    var result int64 = 0

    for right, value := range nums {
        for len(minDeque) > 0 && minDeque[len(minDeque)-1] > value {
            minDeque = minDeque[:len(minDeque)-1]
        }

        for len(maxDeque) > 0 && maxDeque[len(maxDeque)-1] < value {
            maxDeque = maxDeque[:len(maxDeque)-1]
        }

        minDeque = append(minDeque, value)
        maxDeque = append(maxDeque, value)

        for maxDeque[0]-minDeque[0] > 2 {
            if nums[left] == minDeque[0] {
                minDeque = minDeque[1:]
            }

            if nums[left] == maxDeque[0] {
                maxDeque = maxDeque[1:]
            }

            left++
        }

        result += int64(right - left + 1)
    }

    return result
}
```

The Go implementation mirrors the Python logic closely.

One important difference is that the return type is `int64`. The number of subarrays can be as large as:

$$\frac{n(n+1)}{2}$$

For `n = 100000`, this exceeds the range of a 32 bit integer.

Slices are used as deques. Removing from the front is handled with slicing:

```go
deque = deque[1:]
```

## Edge Cases

### Single Element Arrays

A single element always forms a valid subarray because the difference between the minimum and maximum is zero.

This case can expose off by one errors in window calculations. The implementation handles it naturally because:

```
right - left + 1
```

correctly evaluates to `1`.

### All Elements Equal

Consider:

```
[7,7,7,7]
```

Every possible subarray is valid because the minimum and maximum are identical.

This stresses whether the monotonic queues correctly preserve duplicates. The implementation stores duplicate values rather than collapsing them, ensuring removals remain synchronized with the left pointer.

### Large Violations Requiring Frequent Shrinking

Consider:

```
[1,10,1,10]
```

Every time a new element is added, the window becomes invalid immediately.

This case validates that the shrinking loop correctly removes outdated elements from both deques and restores the invariant before counting subarrays.

### Difference Exactly Equal to Two

The condition allows differences up to and including `2`.

For example:

```
[1,3]
```

is valid because:

$$3 - 1 = 2$$

A common bug is accidentally using:

```
>= 2
```

instead of:

```
> 2
```

The implementation correctly preserves windows where the difference is exactly `2`.
