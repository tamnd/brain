## Solution

## Problem Understanding

The problem asks us to generate every possible valid combination of `n` pairs of parentheses.

A valid parentheses string is one where every opening parenthesis `"("` has a matching closing parenthesis `")"` in the correct order. This is the same rule used when checking whether mathematical expressions are balanced.

For example:

- `"(()())"` is valid because every opening parenthesis is eventually closed correctly.
- `"())("` is invalid because the ordering breaks the matching structure.

The input `n` represents the number of pairs of parentheses we must use. Since each pair contributes exactly two characters, every generated string will have length `2 * n`.

The output should contain all possible well-formed combinations. The order of the output does not matter unless explicitly specified, which it is not in this problem.

The constraints are relatively small:

```
1 <= n <= 8
```

This is important because the number of valid combinations grows very quickly. In fact, the count follows the Catalan numbers:

| n | Number of Valid Combinations |
| --- | --- |
| 1 | 1 |
| 2 | 2 |
| 3 | 5 |
| 4 | 14 |
| 5 | 42 |
| 6 | 132 |
| 7 | 429 |
| 8 | 1430 |

Even though the number grows exponentially, `n <= 8` is small enough for recursive backtracking to work efficiently.

There are several important observations and edge cases:

- Every valid string must contain exactly `n` opening and `n` closing parentheses.
- At no point during construction can the number of closing parentheses exceed the number of opening parentheses. If this happens, the string immediately becomes invalid.
- When `n = 1`, there is only one possible answer: `"()"`.
- A naive implementation might generate all possible strings of length `2n` and then validate them afterward, but most of those strings are invalid and waste computation.

The key challenge is generating only valid combinations efficiently.

## Approaches

### Brute Force Approach

The brute-force approach generates every possible string of length `2 * n` using only `"("` and `")"`.

Since each position has two choices, there are:

$2^{2n}$

possible strings.

After generating each string, we validate whether it is well-formed. Validation can be done using a balance counter:

- Increment for `"("`
- Decrement for `")"`
- If the balance ever becomes negative, the string is invalid
- At the end, the balance must equal zero

This approach is correct because it checks every possible candidate, so it cannot miss any valid answer.

However, it is inefficient because the overwhelming majority of generated strings are invalid. For example, when `n = 8`, we generate:

$2^{16}=65536$

possible strings, but only 1430 are valid.

Most computation is wasted on impossible combinations.

### Optimal Approach, Backtracking

The key insight is that we do not need to generate invalid strings at all.

Instead of building every possible sequence and validating later, we can construct the string incrementally while maintaining validity at every step.

At any point:

- We may add `"("` if we still have opening parentheses remaining
- We may add `")"` only if it would not make the string invalid

More specifically:

- `openCount < n` means we can still place another opening parenthesis
- `closeCount < openCount` means we can safely place a closing parenthesis

This pruning strategy eliminates invalid branches immediately.

The algorithm uses recursive backtracking because:

- We build the solution character by character
- Each recursive call represents a partial decision state
- After exploring one choice, we backtrack and try another

This approach generates only valid combinations, making it significantly more efficient than brute force.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^(2n) * n) | O(2^(2n) * n) | Generates every possible string, then validates each one |
| Optimal | O(Cn * n) | O(n) excluding output | Backtracking generates only valid combinations, where Cn is the nth Catalan number |

## Algorithm Walkthrough

1. Initialize an empty result list that will store all valid parentheses combinations.
2. Define a recursive backtracking function that keeps track of:

- The current string being built
- The number of opening parentheses used
- The number of closing parentheses used
3. Check whether the current string length equals `2 * n`.

- If it does, the string is complete and guaranteed to be valid.
- Add it to the result list.
4. If the number of opening parentheses used is less than `n`, add `"("`.

- This is always safe because we are allowed to use up to `n` opening parentheses.
5. If the number of closing parentheses used is less than the number of opening parentheses used, add `")"`.

- This condition ensures we never create an invalid prefix.
- A valid parentheses string can never have more closing parentheses than opening ones at any point.
6. Recursively continue building the string after each choice.
7. Once all recursive paths are explored, return the result list.

The recursion naturally explores every valid configuration while avoiding invalid ones entirely.

## Python Solution

```
from typing import List

class Solution:
    def generateParenthesis(self, n: int) -> List[str]:
        result = []

        def backtrack(current: str, open_count: int, close_count: int) -> None:
            if len(current) == 2 * n:
                result.append(current)
                return

            if open_count < n:
                backtrack(current + "(", open_count + 1, close_count)

            if close_count < open_count:
                backtrack(current + ")", open_count, close_count + 1)

        backtrack("", 0, 0)

        return result
```

The solution begins by creating an empty list called `result`, which stores all valid parentheses combinations.

The nested `backtrack` function performs the recursive exploration. It takes three parameters:

- `current`, the string constructed so far
- `open_count`, the number of `"("` characters used
- `close_count`, the number of `")"` characters used

The base case occurs when the string length reaches `2 * n`. At that point, the string is complete, so it is added to the result list.

The first recursive branch attempts to add an opening parenthesis. This is allowed only if we have not yet used all `n` opening parentheses.

The second recursive branch attempts to add a closing parenthesis. This is allowed only when the number of closing parentheses is smaller than the number of opening parentheses already placed. This condition preserves validity throughout construction.

The recursion explores every valid path and automatically stops exploring invalid possibilities.

Finally, the algorithm returns the completed result list.

## Go Solution

```
func generateParenthesis(n int) []string {
    result := []string{}

    var backtrack func(current string, openCount int, closeCount int)

    backtrack = func(current string, openCount int, closeCount int) {
        if len(current) == 2*n {
            result = append(result, current)
            return
        }

        if openCount < n {
            backtrack(current+"(", openCount+1, closeCount)
        }

        if closeCount < openCount {
            backtrack(current+")", openCount, closeCount+1)
        }
    }

    backtrack("", 0, 0)

    return result
}
```

The Go implementation follows the same recursive backtracking logic as the Python version.

There are several Go-specific details worth noting:

- Go uses slices instead of Python lists, so `result` is declared as `[]string`.
- The recursive function is declared using a function variable because Go requires recursive anonymous functions to be declared before assignment.
- String concatenation uses the `+` operator similarly to Python.
- Go strings are immutable, so each recursive call creates a new string when appending parentheses.

There are no integer overflow concerns because `n <= 8`, making all string lengths very small.

## Worked Examples

### Example 1

Input:

```
n = 3
```

We must generate all valid strings containing 3 pairs of parentheses.

Initial state:

| Current | Open Count | Close Count |
| --- | --- | --- |
| "" | 0 | 0 |

The recursion begins.

#### Step 1

Add `"("` because `open_count < n`.

| Current | Open Count | Close Count |
| --- | --- | --- |
| "(" | 1 | 0 |

#### Step 2

Again add `"("`.

| Current | Open Count | Close Count |
| --- | --- | --- |
| "((" | 2 | 0 |

#### Step 3

Again add `"("`.

| Current | Open Count | Close Count |
| --- | --- | --- |
| "(((" | 3 | 0 |

Now we cannot add another `"("` because we already used all 3 opening parentheses.

#### Step 4

Add `")"`.

| Current | Open Count | Close Count |
| --- | --- | --- |
| "((()" | 3 | 1 |

#### Step 5

Add another `")"`.

| Current | Open Count | Close Count |
| --- | --- | --- |
| "((())" | 3 | 2 |

#### Step 6

Add final `")"`.

| Current | Open Count | Close Count |
| --- | --- | --- |
| "((()))" | 3 | 3 |

Length equals `2 * n = 6`, so this string is added to the result.

The recursion then backtracks and explores other valid branches:

```
(()())
(())()
()(())
()()()
```

Final output:

```
["((()))", "(()())", "(())()", "()(())", "()()()"]
```

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(Cn * n) | There are Cn valid combinations, and each string has length 2n |
| Space | O(n) excluding output | Recursive call stack depth can reach at most 2n |

The number of valid parentheses combinations is the nth Catalan number:

$C_n=\frac{1}{n+1}\binom{2n}{n}$

Each valid string requires `O(n)` time to construct because its length is `2n`.

The recursion depth is at most `2n`, so the auxiliary space used by the call stack is linear in `n`.

The output itself requires much more space because all valid combinations must be stored.

## Edge Cases

### Smallest Valid Input, `n = 1`

This is the smallest allowed input and produces exactly one valid result:

```
["()"]
```

A buggy implementation might incorrectly return an empty list or fail to terminate recursion properly. The current implementation handles this naturally because the recursion reaches the base case immediately after placing one opening and one closing parenthesis.

### Preventing Invalid Prefixes

A major source of bugs is allowing more closing parentheses than opening parentheses during construction.

For example:

```
")("
```

This can never become valid, regardless of future additions.

The condition:

```
if close_count < open_count:
```

prevents this entirely. The algorithm never explores invalid prefixes, which dramatically improves efficiency and correctness.

### Maximum Constraint, `n = 8`

At the upper limit, there are 1430 valid combinations.

A brute-force solution may still run slowly because it explores tens of thousands of invalid candidates. The backtracking solution remains efficient because it prunes invalid paths early and generates only valid strings.

The recursion depth also remains manageable because the maximum string length is only 16 characters.
