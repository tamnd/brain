# LeetCode 814 - Binary Tree Pruning

## Problem Understanding

This problem asks us to modify a binary tree by removing every subtree that does not contain at least one node with value `1`.

A subtree consists of a node and all of its descendants. If an entire subtree contains only `0` values, that subtree must be removed from the tree. Removing a subtree means returning `None` in Python or `nil` in Go for that subtree root.

The input is the root node of a binary tree where every node contains either `0` or `1`. The output should be the root of the same tree after pruning all invalid subtrees.

The important detail is that pruning can propagate upward. Consider a node with value `0` whose children are both removed because they contain no `1`. After the children are removed, that node itself also becomes a subtree without a `1`, so it must also be removed.

The constraints are relatively small, with at most 200 nodes. This means even less efficient solutions could pass, but the structure of the problem naturally suggests a recursive depth first traversal. Since pruning decisions depend on whether descendants contain a `1`, we need information from children before deciding whether to keep the current node.

Several edge cases are important:

- The entire tree may contain only `0`s, in which case the answer should be `None`.
- A leaf node with value `1` must always remain.
- A node with value `0` may still remain if one of its descendants contains a `1`.
- Pruning must happen bottom up, because parent decisions depend on child results.

## Approaches

### Brute Force Approach

A brute force solution would repeatedly scan subtrees to determine whether they contain a `1`.

For every node, we could run a helper function that traverses the entire subtree rooted at that node and checks whether any node contains `1`. If not, we remove that subtree. The issue is that many subtrees overlap, causing repeated traversal of the same nodes.

For example, when checking a parent subtree, we may traverse all descendants. Then when checking one of its children, we traverse many of those same descendants again.

This approach is correct because every subtree is independently validated, but it is inefficient due to repeated work.

### Optimal Approach

The key insight is that pruning decisions should be made in postorder traversal order.

A node can only determine whether it should remain after its left and right children have already been processed. Once the children are recursively pruned, the current node can decide:

- Keep itself if:

- its value is `1`, or
- at least one child still exists
- Remove itself otherwise

This naturally leads to a depth first recursive solution where each recursive call returns either:

- the pruned subtree root, or
- `None` if the subtree should be removed

Each node is visited exactly once, making the solution efficient.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(h) | Repeatedly scans overlapping subtrees |
| Optimal | O(n) | O(h) | Single postorder DFS traversal |

Here:

- `n` is the number of nodes
- `h` is the height of the tree

## Algorithm Walkthrough

1. Start a recursive depth first traversal from the root.
2. Recursively process the left subtree first. The recursive call returns either:

- the pruned left subtree root, or
- `None` if the entire left subtree should be removed.
3. Recursively process the right subtree using the same logic.
4. Update the current node's `left` and `right` pointers with the returned pruned subtrees. This step is important because children may already have been removed.
5. After both children are processed, determine whether the current node should remain:

- If the node value is `1`, keep it.
- If either child still exists, keep it because the subtree contains a `1`.
- Otherwise, remove the node by returning `None`.
6. Continue returning upward until the root is processed.
7. The final returned node is the fully pruned tree root.

### Why it works

The algorithm works because every pruning decision is made after fully processing the node's descendants. By the time we evaluate a node, its children already represent correctly pruned subtrees.

If both children are removed and the node value is `0`, then the entire subtree rooted at that node contains no `1`, so removing it is correct. Otherwise, the subtree contains at least one `1`, so it must remain.

This invariant holds recursively for every subtree in the tree.

## Python Solution

```
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

from typing import Optional

class Solution:
    def pruneTree(self, root: Optional[TreeNode]) -> Optional[TreeNode]:

        def dfs(node: Optional[TreeNode]) -> Optional[TreeNode]:
            if not node:
                return None

            node.left = dfs(node.left)
            node.right = dfs(node.right)

            if node.val == 0 and not node.left and not node.right:
                return None

            return node

        return dfs(root)
```

The implementation uses a nested helper function named `dfs` to perform the recursive traversal.

The base case handles `None` nodes immediately. If the current node is empty, there is nothing to prune.

Next, the algorithm recursively processes the left and right children. The returned values are assigned back to `node.left` and `node.right`. This is important because subtrees may already have been removed during recursion.

After processing both children, the algorithm checks whether the current node should remain. A node should be removed only if:

- its value is `0`
- its left child is `None`
- its right child is `None`

If all three conditions hold, the subtree rooted at this node contains no `1`, so the function returns `None`.

Otherwise, the node is preserved and returned upward.

The final call `dfs(root)` returns the fully pruned tree.

## Go Solution

```
/**
 * Definition for a binary tree node.
 * type TreeNode struct {
 *     Val int
 *     Left *TreeNode
 *     Right *TreeNode
 * }
 */
func pruneTree(root *TreeNode) *TreeNode {

    var dfs func(node *TreeNode) *TreeNode

    dfs = func(node *TreeNode) *TreeNode {
        if node == nil {
            return nil
        }

        node.Left = dfs(node.Left)
        node.Right = dfs(node.Right)

        if node.Val == 0 && node.Left == nil && node.Right == nil {
            return nil
        }

        return node
    }

    return dfs(root)
}
```

The Go implementation follows the same recursive structure as the Python version.

Go uses `nil` instead of `None` for empty pointers. Since binary tree nodes are represented using pointers, removing a subtree simply means returning `nil`.

The recursive helper function is declared using a function variable because recursive anonymous functions in Go require prior declaration.

No special handling for integer overflow or memory allocation is needed because node values are restricted to `0` and `1`, and the recursion depth is bounded by the tree height.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Every node is visited exactly once |
| Space | O(h) | Recursive call stack depth equals tree height |

The algorithm performs a single DFS traversal of the tree. Each node is processed once, and every operation inside the recursive call takes constant time.

The space complexity comes entirely from recursion stack usage. In the worst case of a completely skewed tree, the recursion depth becomes `n`. In a balanced tree, the recursion depth is `log n`.

## Edge Cases

### Entire Tree Gets Removed

A tree containing only `0` values is a critical edge case because the final answer should be `None`.

A naive implementation might only remove leaf nodes and accidentally leave the root behind. This solution avoids that issue because pruning decisions propagate upward. Once all children are removed, the root is reevaluated and removed as well if its value is `0`.

### Root Value Is 0 But Descendant Contains 1

Another tricky case occurs when the root itself is `0`, but somewhere deeper in the tree there is a `1`.

In this situation, the root must remain because its subtree contains a `1`. The implementation handles this correctly because the node is preserved whenever either child still exists after pruning.

### Deeply Skewed Trees

A skewed tree behaves like a linked list and creates the maximum recursion depth.

Some implementations may incorrectly assume both children exist or fail when repeatedly pruning one side. This solution safely handles missing children because every recursive call first checks whether the node is `None` before processing it.
