## Solution

## Problem Understanding

The problem gives us a singly linked list where every node contains a unique integer value. We are also given an array called `nums`, and every value in `nums` is guaranteed to exist somewhere in the linked list.

We need to determine how many connected components are formed by the values in `nums`. Two values are considered connected if they appear consecutively in the linked list.

Another way to think about the problem is this: imagine taking only the nodes whose values appear in `nums` and highlighting them inside the linked list. Consecutive highlighted nodes belong to the same component. Whenever a highlighted sequence ends and a new highlighted sequence begins later, that creates another component.

For example, consider:

```
Linked list: 0 -> 1 -> 2 -> 3
nums = [0,1,3]
```

The highlighted nodes are:

```
[0] -> [1] -> 2 -> [3]
```

Nodes `0` and `1` are consecutive, so they form one component. Node `3` is isolated, so it forms another component. The answer is `2`.

The constraints are important because they tell us what kind of solution is expected:

- The linked list length can be up to `10^4`
- Node values are unique
- All values in `nums` are unique
- Every value in `nums` exists in the linked list

Since the input size is moderate, an `O(n)` solution is ideal. A quadratic approach would still possibly pass in some languages, but it is unnecessary and inefficient.

The uniqueness guarantee simplifies the problem significantly. We never need to worry about duplicate values appearing in multiple places in the list. Each value maps to exactly one node.

Several edge cases are important to recognize upfront:

- `nums` may contain only one value
- Every node in the linked list may belong to `nums`
- Components may appear at the beginning or end of the list
- Multiple isolated nodes may exist
- A component may consist of a single node

A naive implementation can easily make mistakes when detecting where a component starts or ends, especially near the tail of the linked list.

## Approaches

### Brute Force Approach

A straightforward approach is to iterate through the linked list and, for every node, check whether its value exists in `nums` by scanning the array linearly.

If the current node belongs to `nums`, we then determine whether it starts a new component. One way to do that is:

- The node is in `nums`
- Either the previous node was not in `nums`, or this is the first node

Because membership checking requires scanning `nums` every time, each lookup costs `O(k)` where `k = len(nums)`.

Since we potentially perform this lookup for every linked list node, the overall complexity becomes `O(n * k)`.

This approach is correct because it directly follows the definition of connected components, but it is inefficient due to repeated linear membership checks.

### Optimal Approach

The key observation is that the expensive part of the brute-force solution is checking whether a value belongs to `nums`.

We can eliminate that overhead by storing all values from `nums` inside a hash set.

A hash set provides average `O(1)` membership checks. Then we traverse the linked list once and detect the end of each component.

A node represents the end of a component if:

- Its value exists in the set
- Its next node does not exist, or its next node's value is not in the set

Every time we encounter such a node, we increment the component count.

This works because each connected component has exactly one ending node.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n * k) | O(1) | Repeated linear scans of `nums` |
| Optimal | O(n) | O(k) | Uses a hash set for constant-time membership checks |

## Algorithm Walkthrough

1. Create a hash set from the `nums` array.

The hash set allows constant-time membership testing. Instead of repeatedly scanning the array, we can immediately determine whether a node belongs to a component.
2. Initialize a variable `components` to `0`.

This variable stores the total number of connected components found so far.
3. Start traversing the linked list from `head`.

We inspect every node exactly once.
4. For each node, first check whether its value exists in the hash set.

If the value is not in the set, this node is irrelevant to component formation and we continue to the next node.
5. If the current node belongs to the set, determine whether it is the end of a component.

A node ends a component when either:

- `current.next` is `None`, meaning we reached the end of the list
- `current.next.val` is not in the set
6. Whenever a component-ending node is found, increment `components`.

Each component contributes exactly one ending node, so counting endings correctly counts components.
7. Continue traversing until the linked list is exhausted.
8. Return `components`.

### Why it works

The algorithm relies on the fact that every connected component forms a contiguous segment in the linked list.

Instead of trying to detect the start and continuation of components, we count their endings. Every component has exactly one final node whose next node either does not exist or is not part of `nums`.

Because every component contributes exactly one such node, counting these nodes produces the correct number of connected components.

## Python Solution

```
# Definition for singly-linked list.
# class ListNode:
#     def __init__(self, val=0, next=None):
#         self.val = val
#         self.next = next

from typing import List, Optional

class Solution:
    def numComponents(self, head: Optional[ListNode], nums: List[int]) -> int:
        nums_set = set(nums)
        components = 0

        current = head

        while current:
            if current.val in nums_set:
                if current.next is None or current.next.val not in nums_set:
                    components += 1

            current = current.next

        return components
```

The implementation begins by converting `nums` into a hash set called `nums_set`. This transformation is the core optimization because it reduces membership checks from linear time to constant average time.

The variable `components` stores the running total of connected components.

We then traverse the linked list using the pointer `current`. For every node, we first check whether its value belongs to the set. If it does not, the node cannot contribute to any component.

If the node does belong to the set, we determine whether it marks the end of a component. This happens when:

- The node is the final node in the linked list
- The next node exists but its value is not in the set

Whenever one of these conditions is true, we increment the component count.

Finally, after processing all nodes, we return the total count.

## Go Solution

```
/**
 * Definition for singly-linked list.
 * type ListNode struct {
 *     Val int
 *     Next *ListNode
 * }
 */
func numComponents(head *ListNode, nums []int) int {
    numsSet := make(map[int]bool)

    for _, num := range nums {
        numsSet[num] = true
    }

    components := 0
    current := head

    for current != nil {
        if numsSet[current.Val] {
            if current.Next == nil || !numsSet[current.Next.Val] {
                components++
            }
        }

        current = current.Next
    }

    return components
}
```

The Go implementation follows the same logic as the Python version. Since Go does not have a built-in hash set type, we use a `map[int]bool` to simulate one.

The traversal logic remains identical:

- Check whether the current node belongs to the set
- Detect whether it ends a component
- Increment the counter when appropriate

Go uses `nil` instead of Python's `None`, and map lookups return the zero value `false` when a key does not exist.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each node is visited once, and set lookups are O(1) on average |
| Space | O(k) | The hash set stores all values from `nums` |

The linked list traversal processes each node exactly once. Every membership check against the hash set takes constant average time, so the overall runtime is linear in the size of the linked list.

The extra space comes entirely from storing the values from `nums` in the hash set.

## Edge Cases

One important edge case occurs when the entire linked list belongs to `nums`. In this scenario, every node is connected consecutively, so the answer should be exactly `1`. A buggy implementation might accidentally count every node separately. The current solution avoids this because it only increments the count at the final node of the component.

Another important case is when every matching node is isolated. For example:

```
Linked list: 0 -> 1 -> 2 -> 3 -> 4
nums = [0,2,4]
```

Here, no two selected nodes are adjacent. Each node should form its own component. The implementation handles this naturally because every matching node immediately satisfies the component-ending condition.

A third edge case appears when a component ends at the tail of the linked list. Accessing `current.next.val` without checking whether `current.next` exists would cause runtime errors. The implementation safely checks:

```
current.next is None
```

before accessing the next node's value.

Another subtle edge case is a single-node linked list. In this situation, the only node is simultaneously the start and end of a component. The implementation correctly counts it because the node belongs to the set and `current.next` is `None`.
