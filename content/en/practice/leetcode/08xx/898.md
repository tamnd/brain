## Solution

## Problem Understanding

The problem asks us to compute how many distinct values can be produced by taking the bitwise OR of every possible non-empty subarray in the input array.

A subarray is a contiguous segment of the array. For every such segment, we calculate the bitwise OR of all its elements. After computing these OR values for all possible subarrays, we count how many unique results exist.

For example, if the array is `[1,2,4]`, the subarrays are:

- `[1]`
- `[2]`
- `[4]`
- `[1,2]`
- `[2,4]`
- `[1,2,4]`

Their OR values are:

- `1`
- `2`
- `4`
- `1 | 2 = 3`
- `2 | 4 = 6`
- `1 | 2 | 4 = 7`

The distinct results are `{1,2,3,4,6,7}`, so the answer is `6`.

The input consists of:

- An integer array `arr`
- Array length can be as large as `5 * 10^4`
- Each number is between `0` and `10^9`

The large input size immediately tells us that an `O(n^2)` or `O(n^3)` algorithm is likely too slow. Since there are potentially about `1.25 billion` subarrays when `n = 5 * 10^4`, we cannot explicitly compute the OR for every subarray in the naive way.

The important observation is that bitwise OR behaves differently from operations like sum or multiplication. Once a bit becomes `1`, it never turns back to `0` when more numbers are OR-ed together. This monotonic property significantly limits how many distinct OR values can appear at each step.

Several edge cases are important:

- Arrays containing only zeros, because repeated OR operations still produce zero.
- Arrays with many duplicate values, where many subarrays generate the same OR result.
- Arrays with increasing powers of two, where OR values grow rapidly and produce many unique combinations.
- Single-element arrays, which should return `1` because there is exactly one subarray.

## Approaches

### Brute Force Approach

The most direct solution is to generate every possible subarray and compute its bitwise OR.

For every starting index `i`, we expand the subarray to every ending index `j >= i`. While expanding, we maintain the cumulative OR value:

```
current_or |= arr[j]
```

Each computed OR value is inserted into a set so duplicates are removed automatically.

This works because every non-empty subarray is examined exactly once, and the OR for that subarray is correctly computed.

However, this approach is too slow for the given constraints. There are `O(n^2)` subarrays, and although we can update the OR incrementally, we still perform quadratic work overall. With `n = 5 * 10^4`, this becomes infeasible.

### Key Insight for the Optimal Solution

The important observation is that the number of distinct OR values ending at any position is actually small.

Suppose we process the array from left to right. At index `i`, consider all subarrays that end at `i`.

If we already know all OR values for subarrays ending at `i - 1`, then every subarray ending at `i` is formed by:

- Taking a previous subarray ending at `i - 1`
- OR-ing its value with `arr[i]`

Additionally, the single-element subarray `[arr[i]]` must also be included.

The crucial property is that bitwise OR only turns bits on. Since integers have at most about 30 bits in this problem, the number of distinct OR values at each position remains bounded by roughly 30.

This allows us to maintain only the distinct OR results from the previous step instead of all subarrays.

## Algorithm Walkthrough

1. Initialize a global set called `all_results`.

This set stores every distinct OR value seen across all subarrays in the entire array.
2. Initialize another set called `previous_ors`.

This represents all distinct OR values of subarrays ending at the previous index.
3. Iterate through each number `num` in the array.

At each step, we compute all OR values for subarrays ending at the current position.
4. Create a new set called `current_ors`.

First, include the single-element subarray:

```
{num}
```
5. For every value `value` in `previous_ors`, compute:

```
value | num
```

This extends every previous subarray by appending the current element.
6. Insert all these new OR values into `current_ors`.

Duplicate OR values are automatically removed because we use a set.
7. Add all values from `current_ors` into `all_results`.

These are all distinct OR values for subarrays ending at the current index.
8. Update:

```
previous_ors = current_ors
```

This prepares the next iteration.
9. After processing the entire array, return:

```
len(all_results)
```

### Why it works

At every index, `previous_ors` contains exactly the distinct OR values for all subarrays ending at the previous position. Appending the current number to each of those subarrays produces every possible subarray ending at the current position. Including the single-element subarray ensures completeness. Since every subarray must end somewhere, collecting these OR values across all indices guarantees that every subarray OR is considered exactly once.

## Python Solution

```
from typing import List

class Solution:
    def subarrayBitwiseORs(self, arr: List[int]) -> int:
        all_results = set()
        previous_ors = set()

        for num in arr:
            current_ors = {num}

            for value in previous_ors:
                current_ors.add(value | num)

            all_results.update(current_ors)
            previous_ors = current_ors

        return len(all_results)
```

The solution maintains two sets throughout the iteration.

`previous_ors` stores all distinct OR values of subarrays ending at the previous index. When processing a new number, we create `current_ors` by extending all previous subarrays and OR-ing them with the current number.

The line:

```
current_ors = {num}
```

handles the single-element subarray starting at the current position.

Then:

```
current_ors.add(value | num)
```

extends all previous subarrays.

After computing all OR values ending at the current index, we merge them into the global result set:

```
all_results.update(current_ors)
```

Finally, we replace `previous_ors` with `current_ors` because the next iteration only cares about subarrays ending at the current position.

## Go Solution

```
func subarrayBitwiseORs(arr []int) int {
	allResults := make(map[int]bool)
	previousORs := make(map[int]bool)

	for _, num := range arr {
		currentORs := make(map[int]bool)
		currentORs[num] = true

		for value := range previousORs {
			currentORs[value|num] = true
		}

		for value := range currentORs {
			allResults[value] = true
		}

		previousORs = currentORs
	}

	return len(allResults)
}
```

The Go implementation follows the same logic as the Python version but uses `map[int]bool` to simulate sets.

Unlike Python, Go does not have a built-in set type, so maps are the standard approach. The expression:

```
currentORs[value|num] = true
```

adds a value into the set.

Go integers are safe here because all values are within standard integer limits. No special overflow handling is required.

## Edge Cases

One important edge case is an array containing only zeros. Since `0 | 0 = 0`, every subarray produces the same OR value. A buggy implementation might incorrectly count duplicates multiple times. Using sets ensures that only one distinct value is stored.

Another important case is arrays with many repeated numbers, such as `[7,7,7]`. Every OR operation still produces `7`, so the answer should be `1`. The algorithm handles this naturally because duplicate OR values collapse into a single set entry.

A third edge case involves arrays with rapidly expanding OR combinations, such as powers of two. For example, `[1,2,4,8]` generates many distinct OR values because each element activates a different bit. This stresses the correctness of the transition logic. The implementation correctly generates all cumulative OR states by extending every previous OR value with the current element.

A final edge case is the smallest possible array size, where the input contains only one element. In that situation, the only valid subarray is the element itself. The initialization of `current_ors = {num}` guarantees correct handling of this case.
