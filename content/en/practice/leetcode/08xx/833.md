## Solution

## Problem Understanding

The problem requires performing a series of conditional string replacements on a given string `s`. You are provided with three parallel arrays: `indices`, `sources`, and `targets`. Each index `i` in these arrays represents a replacement operation. Specifically, for the `i`-th operation, you must check if the substring of `s` starting at `indices[i]` matches `sources[i]`. If it does, you replace that substring with `targets[i]`; otherwise, you leave the string unchanged at that position.

The key constraints and requirements are that the replacements must happen **simultaneously**. This means the replacement at one index should not affect the indices of other replacements. The problem guarantees that the replacement operations will not overlap, so you do not need to handle conflicts where two replacement substrings intersect.

The input constraints indicate that `s` is relatively small (up to 1000 characters), and the number of replacements `k` is also small (up to 100). This suggests that a solution that iterates over the string a few times is feasible. Edge cases to consider include replacements that do not match the substring, replacements at the start or end of the string, and multiple replacements that are not contiguous.

## Approaches

A **brute-force approach** would iterate over each replacement operation and try to perform it directly on the string. One might attempt to replace substrings in the order they appear in the arrays, but this could cause index shifts if a replacement changes the string length. To avoid index shifts, one would have to rebuild the string carefully after each operation, which adds unnecessary complexity and potential for off-by-one errors.

The **optimal approach** leverages the observation that replacements do not overlap. You can first map the indices where replacements occur, verify matches, and then rebuild the string in one pass. Sorting the indices is unnecessary if replacements are non-overlapping, but iterating through `s` while checking for replacements ensures that the string is rebuilt correctly without affecting other replacement operations.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(k * n) | O(n) | Check and replace each substring sequentially; can involve repeated string reconstruction |
| Optimal | O(n + k) | O(n + k) | Use a map from index to replacement and rebuild the string in a single pass |

## Algorithm Walkthrough

1. Create a dictionary that maps each replacement index `i` to its corresponding source and target strings.
2. Initialize an empty list `result` that will accumulate characters and replacement strings as we iterate through `s`.
3. Iterate over each index `i` of the string `s`.
4. For each index, check if there is a replacement operation starting at `i`.
5. If a replacement exists, verify if the substring `s[i:i+len(source)]` matches the `source`.
6. If it matches, append the corresponding `target` to `result` and skip over the length of the source in the string.
7. If there is no replacement or the source does not match, append the original character `s[i]` to `result`.
8. Continue this process until the end of the string.
9. Join the `result` list into a single string and return it.

**Why it works:** By checking replacements directly at each index and skipping the appropriate number of characters when a replacement occurs, the algorithm ensures that each replacement is applied exactly where intended without affecting the position of other replacements. The non-overlapping guarantee ensures that no conflicts occur.

## Python Solution

```
from typing import List

class Solution:
    def findReplaceString(self, s: str, indices: List[int], sources: List[str], targets: List[str]) -> str:
        # Map each index to its source and target
        replacement_map = {idx: (src, tgt) for idx, src, tgt in zip(indices, sources, targets)}
        
        result = []
        i = 0
        while i < len(s):
            if i in replacement_map:
                src, tgt = replacement_map[i]
                if s[i:i+len(src)] == src:
                    result.append(tgt)
                    i += len(src)
                    continue
            result.append(s[i])
            i += 1
        return ''.join(result)
```

The Python solution builds a map from indices to their respective sources and targets. By iterating through the string once and checking this map, it ensures replacements happen in the correct positions while maintaining linear complexity relative to the length of the string and the number of replacements.

## Go Solution

```
func findReplaceString(s string, indices []int, sources []string, targets []string) string {
    replacementMap := make(map[int][2]string)
    for i := 0; i < len(indices); i++ {
        replacementMap[indices[i]] = [2]string{sources[i], targets[i]}
    }

    result := []byte{}
    i := 0
    for i < len(s) {
        if val, ok := replacementMap[i]; ok {
            src, tgt := val[0], val[1]
            if i+len(src) <= len(s) && s[i:i+len(src)] == src {
                result = append(result, []byte(tgt)...)
                i += len(src)
                continue
            }
        }
        result = append(result, s[i])
        i++
    }
    return string(result)
}
```

In Go, we use a map from `int` to a fixed-size array of two strings to store sources and targets. String concatenation is avoided in favor of appending to a byte slice, which is more efficient in Go. Edge cases like reaching the end of the string are explicitly checked with `i+len(src) <= len(s)`.

## Complexity Analysis

| Measure | Complexity | Explanation |

The algorithm achieves linear complexity relative to the string length because each character is visited at most once, and the map allows constant-time checks for replacements.

## Edge Cases

One important edge case is a replacement at the start of the string. If the algorithm incorrectly advances the pointer, it could miss characters, but our implementation correctly moves past the replaced substring only when a match occurs.

Another edge case is a replacement at the end of the string. The algorithm checks `i + len(src) <= len(s)` to avoid out-of-bounds slicing, ensuring safe replacements.

Finally, multiple replacements with consecutive indices could introduce off-by-one errors. By storing replacements in a map and only advancing the index when a replacement occurs, we guarantee that each replacement is applied exactly where intended without affecting subsequent replacements.
