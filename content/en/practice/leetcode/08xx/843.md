## Solution

## Problem Understanding

This is an interactive problem. Unlike standard LeetCode problems where you directly return a value, here you communicate with a hidden system through the `Master.guess()` API.

You are given a list of unique six-letter words. One of those words has been secretly selected. Your task is to determine the secret word by repeatedly calling:

```
master.guess(word)
```

The API returns:

- `6` if the guessed word exactly matches the secret word
- A number from `0` to `5` representing how many characters match in both value and position
- `-1` if the guessed word is not part of the original `words` array

The important detail is that a "match" means both the character and its position must be identical. For example:

```
"acckzz"
"abcczz"
```

These have 4 matches because positions `0`, `2`, `4`, and `5` match.

The challenge is that we are limited in how many guesses we can make. The test cases guarantee that a reasonable strategy can find the answer within the allowed limit, which is typically between 10 and 30 guesses.

The constraints are relatively small:

- At most 100 words
- Each word has fixed length 6
- All words are unique

Because the input size is small, expensive pairwise comparisons between words are acceptable. The real challenge is designing a strategy that eliminates as many impossible candidates as possible after each guess.

Several edge cases are important:

- The word list may contain only one word
- Many words may look very similar
- Some guesses may produce zero matches, which should eliminate a large portion of candidates
- The optimal strategy is not necessarily to guess randomly
- Since this is interactive, we must only guess words from the provided list

The problem guarantees that the secret word exists in the input list, so we never need to handle impossible situations.

## Approaches

### Brute Force Approach

The simplest approach is to repeatedly guess words arbitrarily from the remaining candidate list.

After each guess, we receive the number of matching positions. We can then remove all words that would not produce the same match count relative to the guessed word.

For example, suppose we guess:

```
acckzz
```

and receive a result of `3`.

Then every remaining candidate must also share exactly 3 matching positions with `"acckzz"`.

This brute-force filtering process is correct because the secret word must always remain consistent with all previous responses.

However, blindly choosing guesses is inefficient. Some guesses provide very little information and eliminate only a few candidates. In worst cases, random guessing may exceed the allowed number of attempts.

### Key Insight for the Optimal Approach

The main insight is that each guess partitions the remaining candidate words into groups based on match counts.

A good guess is one that minimizes the size of the largest possible remaining group.

This is a minimax strategy:

- For every possible guess, compute how it partitions the candidate set
- Count how many words would remain for each possible match score from `0` to `6`
- Choose the guess whose largest partition is as small as possible

This maximizes the worst-case elimination power.

The intuition is similar to binary search. We want every guess to reduce uncertainty as much as possible.

Since there are only 100 words and each word has length 6, we can afford pairwise comparisons between all words.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(T × n²) | O(n) | Random or sequential guessing with filtering |
| Optimal Minimax | O(T × n²) | O(n²) | Chooses guesses that minimize worst-case remaining candidates |

Here:

- `n` is the number of words
- `T` is the number of guesses, bounded by allowed guesses

Although both approaches have similar asymptotic complexity, the minimax strategy dramatically improves practical performance.

## Algorithm Walkthrough

### Optimal Minimax Strategy

1. Start with all words as possible candidates.

Initially, every word could be the secret word.
2. Define a helper function that counts matching positions between two words.

Since all words have fixed length 6, we compare each character index directly.
3. For every candidate guess, compute how it partitions the remaining words.

For a chosen guess word:

- Compare it against every remaining candidate
- Compute the number of matching positions
- Group candidates by match count

For example:

```
Guess: "acckzz"

Match count 0 -> 5 words
Match count 1 -> 12 words
Match count 2 -> 20 words
Match count 3 -> 8 words
```
4. Compute the size of the largest partition for each guess.

The largest partition represents the worst-case number of candidates that could remain after the guess.
5. Select the guess with the smallest worst-case partition.

This minimizes the maximum uncertainty after receiving feedback.
6. Call `master.guess()` with the selected word.

If the result is `6`, we found the secret word and terminate immediately.
7. Filter the candidate list.

Keep only words that would produce the same match count relative to the guessed word.

This preserves consistency with the API response.
8. Repeat until the secret word is found.

Each iteration shrinks the candidate set significantly.

### Why it works

The algorithm maintains an important invariant:

> Every remaining candidate is consistent with all previous feedback from the `Master`.

When we receive a match count for a guess, any word producing a different match count cannot possibly be the secret word and is safely removed.

The minimax strategy guarantees that each guess provides strong information gain in the worst case, which keeps the number of guesses small enough to satisfy the interactive constraints.

## Python Solution

```
# """
# This is Master's API interface.
# You should not implement it, or speculate about its implementation
# """
# class Master:
#     def guess(self, word: str) -> int:

from typing import List
from collections import defaultdict

class Solution:
    def findSecretWord(self, words: List[str], master: 'Master') -> None:
        
        def match_count(word1: str, word2: str) -> int:
            matches = 0
            
            for i in range(6):
                if word1[i] == word2[i]:
                    matches += 1
            
            return matches
        
        candidates = words[:]
        
        while candidates:
            
            best_guess = None
            min_worst_group = float('inf')
            
            for guess_word in candidates:
                
                groups = defaultdict(int)
                
                for other_word in candidates:
                    if guess_word != other_word:
                        score = match_count(guess_word, other_word)
                        groups[score] += 1
                
                worst_group_size = max(groups.values(), default=0)
                
                if worst_group_size < min_worst_group:
                    min_worst_group = worst_group_size
                    best_guess = guess_word
            
            result = master.guess(best_guess)
            
            if result == 6:
                return
            
            next_candidates = []
            
            for word in candidates:
                if match_count(best_guess, word) == result:
                    next_candidates.append(word)
            
            candidates = next_candidates
```

The implementation begins with a helper function `match_count()` that compares two words character by character and counts matching positions.

We maintain a `candidates` list representing all words still consistent with previous guesses.

For every iteration, we evaluate every candidate word as a potential guess. For each guess, we simulate how the remaining candidates would be partitioned by match score.

The dictionary `groups` stores how many words fall into each match-count category.

We then compute the largest partition size. The guess with the smallest largest partition becomes our chosen guess because it gives the strongest worst-case reduction.

After calling `master.guess()`, we immediately stop if the result is `6`.

Otherwise, we filter the candidate set. Only words producing the same match count against the guessed word can still be valid secrets.

This process continues until the secret word is discovered.

## Go Solution

```
/**
 * // This is the Master's API interface.
 * // You should not implement it, or speculate about its implementation
 * type Master struct {
 * }
 *
 * func (this *Master) Guess(word string) int {}
 */

func findSecretWord(words []string, master *Master) {

	matchCount := func(a string, b string) int {
		matches := 0

		for i := 0; i < 6; i++ {
			if a[i] == b[i] {
				matches++
			}
		}

		return matches
	}

	candidates := make([]string, len(words))
	copy(candidates, words)

	for len(candidates) > 0 {

		bestGuess := ""
		minWorstGroup := int(1e9)

		for _, guessWord := range candidates {

			groups := make(map[int]int)

			for _, otherWord := range candidates {
				if guessWord != otherWord {
					score := matchCount(guessWord, otherWord)
					groups[score]++
				}
			}

			worstGroupSize := 0

			for _, count := range groups {
				if count > worstGroupSize {
					worstGroupSize = count
				}
			}

			if worstGroupSize < minWorstGroup {
				minWorstGroup = worstGroupSize
				bestGuess = guessWord
			}
		}

		result := master.Guess(bestGuess)

		if result == 6 {
			return
		}

		nextCandidates := []string{}

		for _, word := range candidates {
			if matchCount(bestGuess, word) == result {
				nextCandidates = append(nextCandidates, word)
			}
		}

		candidates = nextCandidates
	}
}
```

The Go implementation follows the exact same minimax strategy as the Python version.

One notable difference is that Go does not provide Python-style dynamic dictionaries with default values, so we explicitly use:

```
groups := make(map[int]int)
```

Go strings are byte-indexable, which works safely here because all characters are lowercase English letters.

Slices are used for candidate filtering. We create a new slice each iteration instead of modifying the existing slice in place.

No integer overflow concerns exist because all values are very small.

## Edge Cases

### Single Candidate Remaining

If the candidate list contains only one word, the algorithm must immediately guess it. This is important because continuing minimax calculations would be unnecessary overhead. The implementation naturally handles this because the only remaining candidate becomes the best guess.

### Many Highly Similar Words

A difficult case occurs when most words differ by only one character. Random guessing performs poorly because each guess eliminates very few candidates. The minimax strategy specifically addresses this by selecting guesses that create the most balanced partitions possible.

### Zero Match Responses

Receiving a result of `0` can eliminate a large number of candidates. A buggy implementation might accidentally retain incompatible words. The filtering step avoids this by strictly enforcing:

```
match_count(best_guess, word) == result
```

Only perfectly consistent candidates survive.

### Guessing the Secret Immediately

Sometimes the first minimax-selected word is already the secret. The implementation immediately terminates when the API returns `6`, preventing unnecessary additional guesses.

### All Words Produce Similar Partitions

In some datasets, many guesses appear equally good. The algorithm still works because any minimax-valid guess preserves correctness. The problem guarantees that a reasonable strategy succeeds within the allowed guess limit.
