## Solution

## Problem Understanding

The problem gives us a lowercase English string `s` and an integer array `shifts` of the same length. Each value `shifts[i]` tells us how many times we must shift the first `i + 1` characters of the string forward in the alphabet.

A single shift operation moves a character to the next alphabet letter. For example:

- `'a'` becomes `'b'`
- `'c'` becomes `'d'`
- `'z'` wraps around and becomes `'a'`

The operations are cumulative. Every shift affects the string permanently before the next shift operation is applied.

For example, if:

```
s = "abc"
shifts = [3,5,9]
```

then:

- Shift the first 1 character by 3:

`"abc"` → `"dbc"`
- Shift the first 2 characters by 5:

`"dbc"` → `"igc"`
- Shift the first 3 characters by 9:

`"igc"` → `"rpl"`

The final answer is `"rpl"`.

The key observation is that every character is affected by multiple shift operations. Character `s[i]` is shifted by every `shifts[j]` where `j >= i`.

For example:

- `s[0]` is shifted by `shifts[0] + shifts[1] + shifts[2] + ...`
- `s[1]` is shifted by `shifts[1] + shifts[2] + ...`
- `s[2]` is shifted by `shifts[2] + ...`

The constraints are important:

- `s.length` can be as large as `10^5`
- Each `shifts[i]` can be as large as `10^9`

These limits immediately tell us that simulating every shift one by one is impossible. A brute force approach would be far too slow.

Another important detail is that the alphabet wraps around. Since there are only 26 lowercase letters, shifting by 26 has no effect. This means we can safely take every total shift modulo 26.

Important edge cases include:

- A string of length 1
- Very large shift values like `10^9`
- Multiple wraparounds past `'z'`
- All zero shifts
- Cases where cumulative sums become extremely large

The problem guarantees:

- `s` only contains lowercase English letters
- `shifts.length == s.length`
- Both inputs are non-empty

This means we do not need to handle invalid input or mismatched sizes.

## Approaches

### Brute Force Approach

The most direct solution is to literally simulate every operation exactly as described.

For each `shifts[i]`, we shift the first `i + 1` characters forward by `shifts[i]` positions.

One implementation could repeatedly update characters in the prefix:

- For each index `i`

- For each character from `0` to `i`

- Shift that character by `shifts[i]`

This approach is correct because it directly follows the problem statement.

However, it is extremely inefficient. In the worst case:

- There are `n` operations
- Each operation may update up to `n` characters

This leads to `O(n^2)` time complexity.

With `n = 10^5`, this would require roughly `10^10` operations, which is far too slow.

### Optimal Approach

The key insight is that each character receives the sum of all shifts to its right.

For character `s[i]`, the total shift is:

```
shifts[i] + shifts[i+1] + ... + shifts[n-1]
```

This is a suffix sum problem.

Instead of repeatedly modifying prefixes, we can process the string from right to left while maintaining a running cumulative shift.

At each position:

1. Add the current `shifts[i]` to a running total
2. Reduce the total modulo 26
3. Apply the resulting shift to `s[i]`

This works because:

- Every character only needs to be processed once
- The running suffix sum naturally captures all future shifts

The result is an `O(n)` solution.

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Simulates every prefix shift directly |
| Optimal | O(n) | O(n) | Uses suffix sums and processes from right to left |

## Algorithm Walkthrough

1. Convert the string into a mutable list of characters.

Strings are immutable in Python, so modifying characters directly is inconvenient. Converting to a list allows in-place updates.
2. Initialize a variable called `total_shift` to 0.

This variable stores the cumulative suffix sum of shifts as we move from right to left.
3. Traverse the string from the last index down to index 0.

We process backwards because each character depends on all shifts to its right.
4. Add `shifts[i]` to `total_shift`.

This updates the running suffix sum.
5. Take `total_shift % 26`.

Since the alphabet has only 26 letters, any larger shift repeats cyclically.
6. Convert the current character into its alphabet index.

For example:

- `'a'` → 0
- `'b'` → 1
- `'z'` → 25
7. Apply the cumulative shift.

Compute:

```
new_position = (old_position + total_shift) % 26
```
8. Convert the shifted position back into a character.
9. Store the updated character back into the list.
10. After processing all characters, join the list into a final string and return it.

### Why it works

For every index `i`, the character `s[i]` is affected by every shift operation whose range includes index `i`. Since operation `j` shifts the prefix `[0...j]`, character `i` receives all shifts where `j >= i`.

The running suffix sum exactly represents:

```
shifts[i] + shifts[i+1] + ... + shifts[n-1]
```

Therefore, each character receives precisely the total number of shifts required by the problem statement.

## Python Solution

```
from typing import List

class Solution:
    def shiftingLetters(self, s: str, shifts: List[int]) -> str:
        chars = list(s)
        total_shift = 0

        for i in range(len(s) - 1, -1, -1):
            total_shift = (total_shift + shifts[i]) % 26

            current_position = ord(chars[i]) - ord('a')
            shifted_position = (current_position + total_shift) % 26

            chars[i] = chr(shifted_position + ord('a'))

        return "".join(chars)
```

The implementation starts by converting the input string into a list of characters because Python strings cannot be modified in place.

The variable `total_shift` stores the cumulative suffix sum. As the loop moves from right to left, each new `shifts[i]` value is added into this running total.

The modulo operation is applied immediately to keep the value small and avoid unnecessary large integers.

For each character:

- `ord(chars[i]) - ord('a')` converts the character into a zero-based alphabet index
- The cumulative shift is added
- Another modulo 26 ensures wraparound behavior
- `chr(...)` converts the index back into a lowercase character

Finally, the modified list is joined back into a string.

## Go Solution

```
func shiftingLetters(s string, shifts []int) string {
    chars := []byte(s)
    totalShift := 0

    for i := len(s) - 1; i >= 0; i-- {
        totalShift = (totalShift + shifts[i]) % 26

        currentPosition := int(chars[i] - 'a')
        shiftedPosition := (currentPosition + totalShift) % 26

        chars[i] = byte(shiftedPosition) + 'a'
    }

    return string(chars)
}
```

The Go solution follows exactly the same logic as the Python version.

Some Go-specific details are worth noting:

- Strings in Go are immutable, so we convert the string into a `[]byte`
- Character arithmetic is performed using integer conversions
- The `% 26` operation keeps values within the alphabet range
- Since Go uses explicit typing, conversions between `byte` and `int` are necessary

Integer overflow is not a problem because the cumulative shift is reduced modulo 26 during every iteration.

## Edge Cases

One important edge case is a single-character string. Since there is only one character, the algorithm must correctly handle a loop with exactly one iteration. The implementation naturally supports this because the reverse traversal still works correctly when `n = 1`.

Another important case is very large shift values such as `10^9`. A naive implementation might repeatedly shift characters billions of times, which would be impossibly slow. The solution avoids this by taking every cumulative shift modulo 26, since shifting by 26 returns a character to its original position.

Wraparound behavior is another common source of bugs. Characters near `'z'` must correctly cycle back to `'a'`. The modulo arithmetic:

```
(current_position + total_shift) % 26
```

guarantees that values always remain within the valid alphabet range from 0 to 25.

A subtle edge case occurs when all shifts are zero. In this situation, the original string should remain unchanged. Since the cumulative shift never increases, every character maps back to itself correctly.

Finally, cumulative sums can become extremely large when many shift values are added together. By applying modulo 26 during every iteration, the implementation prevents integer growth while preserving correctness.
