## Solution

## Problem Understanding

The problem gives us a string `s` and an integer `k`. In one operation, we may choose any character among the first `k` characters of the current string, remove it from its position, and append it to the end of the string. We may repeat this operation as many times as we want.

Our goal is to determine the lexicographically smallest string that can be produced after performing any number of valid operations.

Lexicographical order is the same ordering used in dictionaries. A string is considered smaller if, at the first position where two strings differ, it contains a smaller character. For example, `"abc"` is smaller than `"acb"` because `'b' < 'c'` at the second position.

The constraints are relatively small, with the string length at most `1000`, but the state space of possible transformations can still become enormous if explored naively. This means a brute-force simulation of all reachable strings is not practical.

There are two very different behaviors depending on the value of `k`.

When `k = 1`, the only allowed move is taking the first character and moving it to the end. This operation is exactly a left rotation of the string. Therefore, the only strings we can ever produce are the rotations of the original string.

When `k > 1`, the problem changes dramatically. In this case, we gain enough freedom to rearrange characters arbitrarily. The smallest possible string is therefore simply the fully sorted string.

Several edge cases are important:

- A string with all identical characters, such as `"aaaa"`, should remain unchanged regardless of `k`.
- A single-character string should always return itself.
- For `k = 1`, we must examine every rotation carefully because we cannot arbitrarily reorder characters.
- For `k > 1`, duplicate letters must still be handled correctly when sorting.

## Approaches

### Brute Force Approach

A brute-force solution would treat the problem as a graph traversal problem. Each unique string configuration is a node, and every valid operation creates edges to new configurations.

Starting from the original string, we could perform a breadth-first search or depth-first search to generate all reachable strings. Among all generated states, we would track the lexicographically smallest one.

This approach is correct because it explores every valid transformation sequence. Eventually, it would discover the optimal answer.

However, the number of possible states grows extremely quickly. Even for moderately sized strings, the number of distinct permutations or rotations becomes enormous. In the worst case, this becomes factorial in the length of the string, which is completely impractical for `n = 1000`.

### Key Insight

The crucial observation is that the problem behaves differently depending on whether `k` equals `1` or is greater than `1`.

If `k = 1`, the operation always moves the first character to the end. This means every operation is simply a rotation. Therefore, the answer must be the smallest rotation of the string.

If `k > 1`, we can effectively simulate adjacent swaps and eventually generate any permutation of the characters. Once arbitrary permutations become possible, the lexicographically smallest arrangement is simply the sorted order of the characters.

This insight transforms the problem from an exponential search problem into a simple string manipulation problem.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n!) | O(n!) | Explores all reachable string states |
| Optimal | O(n²) for k=1, O(n log n) for k>1 | O(n) | Uses rotation observation and sorting insight |

## Algorithm Walkthrough

### Case 1: `k = 1`

1. Observe that the only allowed move is rotating the string left by one position.
2. Generate every possible rotation of the string. For a string of length `n`, there are exactly `n` distinct rotations.
3. For each rotation, compare it with the current best answer lexicographically.
4. Keep the smallest rotation found.
5. Return the smallest rotation after checking all possibilities.

### Case 2: `k > 1`

1. Observe that choosing among the first `k` characters gives enough flexibility to rearrange characters freely.
2. Since any permutation becomes achievable, the lexicographically smallest possible string is simply the sorted sequence of characters.
3. Sort the characters of the string in ascending order.
4. Return the sorted string.

### Why it works

For `k = 1`, every operation is exactly a cyclic rotation, so the set of reachable strings is precisely the set of all rotations. Checking all rotations guarantees we find the smallest reachable string.

For `k > 1`, the operation provides enough freedom to simulate swaps between neighboring characters. Since adjacent swaps can generate any permutation, every arrangement of characters becomes reachable. Therefore, the smallest possible arrangement is the sorted string.

## Python Solution

```
class Solution:
    def orderlyQueue(self, s: str, k: int) -> str:
        if k == 1:
            smallest = s
            n = len(s)

            for i in range(1, n):
                rotated = s[i:] + s[:i]
                if rotated < smallest:
                    smallest = rotated

            return smallest

        return "".join(sorted(s))
```

The implementation first checks whether `k` equals `1`.

If it does, the code iterates through every possible rotation. A rotation starting at index `i` is created by concatenating `s[i:]` with `s[:i]`. Each rotation is compared against the current smallest string, and the minimum is retained.

If `k` is greater than `1`, the implementation simply sorts the characters using Python's built-in sorting functionality and joins them back into a string.

The solution directly mirrors the mathematical observations from the algorithm discussion, which keeps the implementation concise and efficient.

## Go Solution

```
package main

import (
	"sort"
)

func orderlyQueue(s string, k int) string {
	if k == 1 {
		smallest := s
		n := len(s)

		for i := 1; i < n; i++ {
			rotated := s[i:] + s[:i]

			if rotated < smallest {
				smallest = rotated
			}
		}

		return smallest
	}

	chars := []byte(s)
	sort.Slice(chars, func(i, j int) bool {
		return chars[i] < chars[j]
	})

	return string(chars)
}
```

The Go implementation follows the same logic as the Python version.

For `k = 1`, rotations are constructed using string slicing and concatenation. Go allows direct lexicographical comparison of strings using the `<` operator, which simplifies comparisons.

For `k > 1`, the string is converted into a byte slice so it can be sorted in place using `sort.Slice`.

There are no integer overflow concerns because all indices remain within the string length bounds. Since the input only contains lowercase English letters, byte-based manipulation is completely safe.

| Measure | Complexity | Explanation |
| --- | --- | --- |

When `k = 1`, we generate `n` rotations, and each rotation construction costs `O(n)`, leading to `O(n²)` total time complexity.

When `k > 1`, the dominant operation is sorting the characters, which costs `O(n log n)`.

The extra space usage comes from constructing rotated strings or storing sorted character arrays.

## Edge Cases

A very important edge case is when `k = 1`. Many incorrect solutions assume that characters can always be rearranged freely. However, with `k = 1`, the only operation allowed is rotation. For example, `"cba"` cannot become `"abc"` because arbitrary swaps are impossible. The implementation correctly handles this by explicitly generating all rotations instead of sorting.

Another important edge case is strings containing duplicate characters, such as `"aaaa"` or `"abab"`. Duplicate letters can make lexicographical comparisons tricky because multiple rotations may appear equivalent for long prefixes. The implementation safely compares entire strings directly, ensuring the correct smallest result is chosen.

A third edge case is very small input sizes, especially a single-character string. A naive rotation loop might accidentally produce index errors or unnecessary operations. The implementation handles this naturally because the loop over rotations simply does not execute when the length is `1`, and the original string is returned immediately.

A final subtle edge case occurs when the original string is already the smallest possible answer. For example, `"abc"` with `k = 1` should remain unchanged. The implementation initializes `smallest` with the original string and only updates it when a strictly smaller rotation is found, guaranteeing correctness in this scenario.
