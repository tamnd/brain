## Solution

## Problem Understanding

The problem gives us an array called `hand`, where each value represents a card number, and an integer `groupSize`. We need to determine whether it is possible to rearrange all cards into groups such that:

- Every group contains exactly `groupSize` cards
- The cards inside each group are consecutive integers
- Every card must be used exactly once

For example, if `groupSize = 3`, then valid groups could look like `[1,2,3]`, `[5,6,7]`, or `[10,11,12]`. A group like `[1,2,4]` would not be valid because the numbers are not consecutive.

The important detail is that the cards can appear in any order in the input array. We are free to rearrange them however we want.

The constraints are important because they guide us toward the appropriate algorithmic complexity:

- `hand.length <= 10^4`
- `hand[i] <= 10^9`

The relatively large input size means an exponential or quadratic brute-force solution will likely be too slow. The large possible card values also suggest that we should not rely on fixed-size arrays indexed by card value. Instead, a hash map is more appropriate.

There are several important edge cases:

- If the total number of cards is not divisible by `groupSize`, the answer is immediately `false`
- Duplicate values must be handled carefully because the same card value may participate in multiple groups
- Missing intermediate values should immediately invalidate a potential sequence
- A `groupSize` of `1` should always return `true`, because every individual card forms a valid group
- Large gaps between numbers are allowed, but sequences themselves must still be consecutive

The core challenge is ensuring that every card is placed into exactly one valid consecutive group without conflicts.

## Approaches

### Brute Force Approach

A brute-force strategy would repeatedly attempt to build groups manually from the remaining cards.

One possible implementation is:

1. Sort the array
2. Start from the smallest unused card
3. Try to find the next `groupSize - 1` consecutive values
4. Mark those cards as used
5. Repeat until all cards are consumed

This works because starting from the smallest remaining card is logically necessary. If the smallest card cannot begin a valid sequence, then no solution exists.

However, a naive implementation becomes inefficient because repeatedly searching for required consecutive cards can take linear time for each lookup. Since we may do this for many groups, the total complexity can degrade toward `O(n^2)`.

With `n = 10^4`, quadratic behavior is undesirable.

### Optimal Greedy Approach

The key observation is that the smallest remaining card must always start a new sequence.

Suppose the smallest unused value is `x`. There is no smaller card available that could precede it, so the only possible valid placement for `x` is as the beginning of a group:

```
[x, x+1, x+2, ..., x+groupSize-1]
```

This naturally leads to a greedy strategy:

- Count how many times each card appears
- Process card values in sorted order
- Whenever a card still has remaining copies, attempt to build consecutive groups starting from it
- Reduce frequencies as groups are formed

A hash map allows constant-time frequency updates, and sorting ensures we always process the smallest available card first.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | Repeated searches for consecutive cards |
| Optimal | O(n log n) | O(n) | Uses sorting and frequency counting |

## Algorithm Walkthrough

1. First, check whether the total number of cards is divisible by `groupSize`.

If `len(hand) % groupSize != 0`, then it is impossible to divide all cards evenly into groups. We can immediately return `false`.
2. Count the frequency of every card.

We use a hash map where:

```
frequency[value] = number of occurrences
```

This allows us to efficiently track how many unused copies of each card remain.
3. Sort the cards.

Sorting ensures we always process cards from smallest to largest. This is critical because the smallest unused card must begin a new group.
4. Iterate through the sorted cards.

For each card:

- If its frequency is already zero, it has already been used in previous groups, so we skip it.
- Otherwise, this card must start a new consecutive sequence.
5. Attempt to form a consecutive group.

For a starting card `x`, we must consume:

```
x, x+1, x+2, ..., x+groupSize-1
```

For each required value:

- If its frequency is zero, then the sequence cannot be completed, so return `false`
- Otherwise, decrement its frequency
6. Continue until all cards are processed.

If every sequence can be formed successfully, return `true`.

### Why it works

The correctness comes from the greedy invariant:

> The smallest unused card must begin a sequence.

There is no smaller card available that could legally appear before it. Therefore, if we cannot build a valid consecutive group starting from the smallest remaining value, then no valid arrangement exists.

Because we always consume cards in the only valid way they can appear, the greedy strategy is both safe and complete.

## Python Solution

```
from collections import Counter
from typing import List

class Solution:
    def isNStraightHand(self, hand: List[int], groupSize: int) -> bool:
        if len(hand) % groupSize != 0:
            return False

        frequency = Counter(hand)

        for card in sorted(hand):
            if frequency[card] == 0:
                continue

            for next_card in range(card, card + groupSize):
                if frequency[next_card] == 0:
                    return False

                frequency[next_card] -= 1

        return True
```

The implementation begins with the divisibility check because an uneven number of cards can never form equal-sized groups.

Next, `Counter` is used to store the frequency of each card. This gives efficient constant-time lookups and updates.

The cards are processed in sorted order so that we always encounter the smallest remaining card first. This matches the greedy reasoning discussed earlier.

Inside the loop, if a card's frequency is already zero, that means it was consumed as part of a previous sequence. We skip it because it no longer represents an unused card.

When we encounter a card with positive frequency, we attempt to build a full consecutive sequence starting from that value. The inner loop checks every required next card. If any required card is unavailable, the function immediately returns `False`.

Otherwise, frequencies are decremented to mark those cards as used.

If all groups are formed successfully, the function returns `True`.

## Go Solution

```
package main

import (
	"sort"
)

func isNStraightHand(hand []int, groupSize int) bool {
	if len(hand)%groupSize != 0 {
		return false
	}

	frequency := make(map[int]int)

	for _, card := range hand {
		frequency[card]++
	}

	sort.Ints(hand)

	for _, card := range hand {
		if frequency[card] == 0 {
			continue
		}

		for nextCard := card; nextCard < card+groupSize; nextCard++ {
			if frequency[nextCard] == 0 {
				return false
			}

			frequency[nextCard]--
		}
	}

	return true
}
```

The Go implementation follows the same logic as the Python version.

Instead of `Counter`, Go uses a `map[int]int` to store frequencies.

The `sort.Ints()` function sorts the slice in ascending order so we can process cards greedily from smallest to largest.

Go maps return the zero value for missing keys, which works naturally here because a missing card should behave like frequency zero.

Since the constraints fit comfortably within Go's `int` type, integer overflow is not a concern.

## Edge Cases

### Total Cards Not Divisible by Group Size

If the total number of cards cannot be evenly partitioned into groups of size `groupSize`, then no valid arrangement exists.

For example:

```
hand = [1,2,3,4,5]
groupSize = 4
```

Even before considering card values, the math already fails because five cards cannot form groups of four without leftovers.

The implementation handles this immediately with:

```
if len(hand) % groupSize != 0:
    return False
```

This prevents unnecessary computation.

### Duplicate Values Participating in Multiple Groups

Duplicate cards are one of the trickiest aspects of the problem.

For example:

```
[1,1,2,2,3,3]
```

This should form:

```
[1,2,3]
[1,2,3]
```

A naive implementation might accidentally reuse cards incorrectly or fail to track counts precisely.

The frequency map solves this cleanly by explicitly counting how many unused copies remain. Every time a card is consumed, its count decreases by one.

This guarantees that no card is reused improperly.

### Missing Intermediate Values

Consider:

```
[1,2,4,5,6,7]
groupSize = 3
```

When trying to build a sequence starting at `1`, we require:

```
[1,2,3]
```

But `3` does not exist.

Some incorrect solutions only check whether enough total cards exist rather than verifying every intermediate consecutive value.

The implementation correctly checks every required value individually:

```
if frequency[next_card] == 0:
    return False
```

This immediately detects broken sequences.

### Group Size Equal to One

When `groupSize = 1`, every card individually forms a valid consecutive sequence.

For example:

```
[5,100,999]
```

is automatically valid because each single-card group satisfies the rules.

The implementation naturally handles this case without special logic because the inner loop only checks one card at a time.
