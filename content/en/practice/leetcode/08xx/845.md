## Solution

## Problem Understanding

The problem asks us to find the length of the longest contiguous subarray that forms a valid mountain.

A mountain is defined by two strict conditions:

1. The values must strictly increase up to some peak element.
2. After the peak, the values must strictly decrease.

This means a valid mountain must contain:

- At least one increasing step
- At least one decreasing step
- A peak that is not at either end of the subarray

For example, the subarray `[1,4,7,3,2]` is a mountain because:

- `1 < 4 < 7`
- `7 > 3 > 2`

The peak is `7`, and it has both an increasing side and a decreasing side.

The input is an integer array `arr`, and the output should be the length of the longest mountain subarray. If no valid mountain exists, we return `0`.

The constraints are important:

- `arr.length` can be up to `10^4`
- A quadratic solution might still pass, but it is not ideal
- The follow-up explicitly asks for a one-pass, `O(1)` space solution

This strongly suggests that we should avoid storing large auxiliary arrays if possible.

Several edge cases are important:

- Arrays with fewer than 3 elements can never form a mountain.
- Flat sections like `[1,2,2,1]` are invalid because the increase and decrease must both be strict.
- Purely increasing arrays like `[1,2,3,4]` are not mountains.
- Purely decreasing arrays like `[4,3,2,1]` are not mountains.
- Multiple mountains may exist, and we must return the longest one.

## Approaches

### Brute Force Approach

The brute-force approach is to examine every possible subarray and check whether it forms a valid mountain.

For every pair `(start, end)`:

1. Extract the subarray.
2. Try to find a peak.
3. Verify that:

- Elements strictly increase before the peak
- Elements strictly decrease after the peak

If valid, update the maximum length.

This works because every possible subarray is checked exhaustively. However, it is inefficient because there are `O(n^2)` subarrays, and validating each one can take `O(n)` time.

The worst-case complexity becomes `O(n^3)`.

With `n = 10^4`, this is far too slow.

### Optimal Approach

The key observation is that a mountain is completely determined by a peak.

If we know an index is a valid peak, meaning:

- `arr[i - 1] < arr[i]`
- `arr[i] > arr[i + 1]`

then we can expand outward:

- Move left while values are strictly increasing
- Move right while values are strictly decreasing

This gives the full mountain length centered at that peak.

An even better optimization is to scan the array once while tracking:

- Length of current uphill climb
- Length of current downhill descent

Whenever both are positive, we have a valid mountain.

This achieves:

- `O(n)` time
- `O(1)` extra space

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n³) | O(1) | Checks every possible subarray |
| Optimal | O(n) | O(1) | Single pass using uphill and downhill tracking |

## Algorithm Walkthrough

### Optimal One-Pass Algorithm

1. Initialize three variables:

- `up = 0`, length of current increasing sequence
- `down = 0`, length of current decreasing sequence
- `longest = 0`, best mountain length found so far
2. Traverse the array from index `1` to `n - 1`.
3. Before processing the current pair, check whether we need to reset:

- If we are descending and encounter an increase again
- Or if two adjacent values are equal

In either case, the current mountain structure is broken, so reset:

- `up = 0`
- `down = 0`
4. If `arr[i] > arr[i - 1]`, we are climbing uphill:

- Increment `up`
5. Else if `arr[i] < arr[i - 1]`, we are descending:

- Increment `down`, but only meaningful if we previously climbed
6. A valid mountain exists only if:

- `up > 0`
- `down > 0`

At that point:

- Mountain length = `up + down + 1`
- Update `longest`
7. Continue until the end of the array.
8. Return `longest`.

### Why it works

The algorithm maintains the lengths of the current increasing and decreasing slopes. A valid mountain requires both slopes to exist.

Whenever the sequence becomes invalid, such as equal adjacent elements or a new ascent after descending, the counters are reset because a new mountain must begin.

Every valid mountain is identified exactly once during the traversal, and its full length is computed using the accumulated uphill and downhill lengths.

## Python Solution

```
from typing import List

class Solution:
    def longestMountain(self, arr: List[int]) -> int:
        n = len(arr)

        if n < 3:
            return 0

        up = 0
        down = 0
        longest = 0

        for i in range(1, n):
            if (down > 0 and arr[i] > arr[i - 1]) or arr[i] == arr[i - 1]:
                up = 0
                down = 0

            if arr[i] > arr[i - 1]:
                up += 1
            elif arr[i] < arr[i - 1]:
                if up > 0:
                    down += 1

            if up > 0 and down > 0:
                longest = max(longest, up + down + 1)

        return longest
```

The implementation begins by handling the smallest invalid case. Arrays shorter than length `3` cannot form mountains.

The variables `up` and `down` track how many increasing and decreasing transitions have occurred in the current candidate mountain.

The reset condition is critical. Once we begin descending, encountering another increase means the previous mountain has ended. Similarly, equal adjacent values invalidate strict monotonicity.

During traversal:

- Increasing pairs extend the uphill section.
- Decreasing pairs extend the downhill section only if an uphill already exists.

Whenever both counters are positive, a valid mountain exists, and we compute its length.

The expression `up + down + 1` works because:

- `up` counts increasing edges
- `down` counts decreasing edges
- Adding `1` accounts for the peak element itself

## Go Solution

```
func longestMountain(arr []int) int {
	n := len(arr)

	if n < 3 {
		return 0
	}

	up := 0
	down := 0
	longest := 0

	for i := 1; i < n; i++ {
		if (down > 0 && arr[i] > arr[i-1]) || arr[i] == arr[i-1] {
			up = 0
			down = 0
		}

		if arr[i] > arr[i-1] {
			up++
		} else if arr[i] < arr[i-1] {
			if up > 0 {
				down++
			}
		}

		if up > 0 && down > 0 {
			length := up + down + 1
			if length > longest {
				longest = length
			}
		}
	}

	return longest
}
```

The Go implementation follows the same logic as the Python version.

Since Go does not provide a built-in `max` function for integers, we manually compare and update `longest`.

The array is represented as a slice, and integer overflow is not a concern because the maximum array size is only `10^4`.

## Edge Cases

### Arrays Shorter Than Length 3

A mountain requires at least three elements because there must be an ascent, a peak, and a descent.

Arrays like `[1]` or `[1,2]` cannot possibly satisfy the definition. Without an explicit early return, implementations may accidentally access invalid indices or incorrectly count partial sequences.

The solution handles this immediately with:

```
if n < 3:
    return 0
```

### Flat Sections

Equal adjacent values are one of the most common sources of bugs.

For example:

```
[1,2,2,1]
```

This is not a mountain because the climb is not strictly increasing.

The implementation resets both counters whenever equal values appear:

```
arr[i] == arr[i - 1]
```

This ensures plateaus never become part of a mountain.

### Multiple Mountains in One Array

Arrays may contain several mountains back-to-back:

```
[1,2,1,2,3,2,1]
```

A naive implementation may fail to properly reset state between mountains.

This solution resets when a new ascent begins after a descent:

```
down > 0 and arr[i] > arr[i - 1]
```

That allows the algorithm to correctly start tracking a new mountain without interference from the previous one.

### Descending Before Any Ascent

Consider:

```
[5,4,3,2,1]
```

The array is entirely decreasing. Some implementations mistakenly count this as a mountain.

The algorithm avoids this by only allowing `down` to increase if `up > 0`.

That guarantees every counted mountain has a valid increasing section before the descent begins.
