# LeetCode 842, Split Array into Fibonacci Sequence

## Problem Understanding

This problem asks us to determine whether a string of digits can be partitioned into a valid Fibonacci-like sequence.

A Fibonacci-like sequence follows the same additive rule as the classical Fibonacci sequence. Starting from the third number, every number must equal the sum of the previous two numbers. The difference is that the first two numbers are not fixed, we are free to choose them from the string.

For example, given the string `"123456579"`, we can split it into:

```
[123, 456, 579]
```

because:

```
123 + 456 = 579
```

The input is a string instead of an array of integers because we must decide where to split the digits. Every split must satisfy several constraints:

1. Every piece must be a valid non-negative integer.
2. No integer may exceed the 32-bit signed integer limit:

$0 \le x < 2^{31}$
3. Numbers cannot contain leading zeroes unless the number itself is exactly `"0"`.
4. The final sequence must contain at least three numbers.

The goal is to return any valid Fibonacci-like sequence. If no such sequence exists, we return an empty array.

The constraints are important:

- `num.length <= 200`
- digits only

A length of 200 is far too large for unrestricted brute force partition generation because the number of ways to split a string grows exponentially. However, the 32-bit integer restriction greatly limits how large each number can become. Since a 32-bit signed integer has at most 10 digits, the search space becomes manageable with pruning.

Several edge cases are especially important:

- Strings with leading zeroes such as `"0123"`
- Strings where numbers exceed 32-bit integer range
- Cases where multiple valid answers exist
- Strings that almost form a Fibonacci sequence but fail near the end
- Inputs containing many zeroes such as `"0000"`

A naive implementation can easily mishandle leading zeroes or overflow conditions.

## Approaches

### Brute Force Approach

The brute force idea is to try every possible way to split the string into substrings, convert each substring into an integer, and then verify whether the resulting sequence satisfies the Fibonacci property.

For a string of length `n`, every position can either contain a split or not contain a split. That creates roughly:

$2^{n-1}$

possible partitionings.

For each partitioning, we would need to:

1. Parse every substring
2. Reject invalid leading zeroes
3. Check integer bounds
4. Verify Fibonacci conditions

This guarantees correctness because every possible split is explored. However, the runtime becomes exponential and completely impractical for strings up to length 200.

### Optimal Backtracking Approach

The key observation is that once the first two numbers are fixed, the entire remaining sequence is determined.

Suppose we choose:

```
a = 11
b = 0
```

Then the next number must be:

```
11
```

After that:

```
0 + 11 = 11
```

Then:

```
11 + 11 = 22
```

and so on.

This dramatically reduces the search space. Instead of exploring arbitrary partitions everywhere, we only need to:

1. Try all possibilities for the first number
2. Try all possibilities for the second number
3. Recursively verify whether the remaining string matches the required sums

The 32-bit limit also keeps the number of candidate first and second numbers small.

Backtracking works naturally here because:

- We build the sequence incrementally
- We abandon invalid paths immediately
- We stop as soon as a valid sequence is found

## Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(2^n) | O(n) | Explores all possible partitions |
| Optimal Backtracking | O(n²) in practice | O(n) | First two numbers determine the rest |

## Algorithm Walkthrough

1. Initialize an empty sequence list that will store the current Fibonacci sequence being built.
2. Use backtracking starting from index `0` in the string. At every recursive call, we try to choose the next number.
3. For the current index, iterate through all possible substring endings to form the next number.
4. Reject substrings with leading zeroes. If the current substring starts with `'0'` and its length is greater than one, stop exploring further extensions.
5. Convert the substring into an integer and check whether it exceeds the 32-bit signed integer limit. If it does, stop processing longer substrings because they will only become larger.
6. If the sequence currently has at least two numbers, compute the required next value:

$f[i-2] + f[i-1] = f[i]$
7. Compare the candidate number with the required sum:

- If the candidate is smaller, continue extending the substring because adding more digits may increase it.
- If the candidate is larger, stop immediately because further digits will only make it larger.
- If it matches exactly, continue recursion.
8. Append the valid number to the sequence and recursively continue from the next index.
9. If recursion reaches the end of the string and the sequence length is at least three, return success.
10. If a recursive branch fails, remove the last number from the sequence and continue exploring other possibilities.
11. If all possibilities fail, return an empty list.

### Why it works

The algorithm is correct because it systematically explores every valid choice for the first two numbers. After those are fixed, the Fibonacci property uniquely determines every subsequent value. Backtracking guarantees that every feasible sequence is checked, while pruning removes impossible branches early. Therefore, if a valid Fibonacci-like sequence exists, the algorithm will find one.

## Python Solution

```
from typing import List

class Solution:
    def splitIntoFibonacci(self, num: str) -> List[int]:
        MAX_INT = 2**31 - 1
        sequence = []

        def backtrack(index: int) -> bool:
            if index == len(num):
                return len(sequence) >= 3

            current = 0

            for end in range(index, len(num)):
                # Leading zero check
                if end > index and num[index] == '0':
                    break

                current = current * 10 + int(num[end])

                # 32-bit integer limit
                if current > MAX_INT:
                    break

                size = len(sequence)

                if size >= 2:
                    expected = sequence[-1] + sequence[-2]

                    if current < expected:
                        continue

                    if current > expected:
                        break

                sequence.append(current)

                if backtrack(end + 1):
                    return True

                sequence.pop()

            return False

        if backtrack(0):
            return sequence

        return []
```

The implementation follows the exact backtracking strategy described earlier.

The `sequence` list stores the currently constructed Fibonacci sequence. The recursive `backtrack` function attempts to build a valid continuation starting from a given string index.

The variable `current` is constructed digit by digit instead of repeatedly slicing substrings and converting them. This is more efficient because it avoids repeated integer parsing.

The leading zero rule is enforced with:

```
if end > index and num[index] == '0':
    break
```

This allows `"0"` but rejects `"01"` or `"0001"`.

The Fibonacci validation only begins once at least two numbers already exist in the sequence. At that point, the expected next value is uniquely determined.

An important optimization is:

```
if current > expected:
    break
```

Since adding more digits only increases the number, there is no reason to continue exploring that branch.

The recursion terminates successfully only when:

- the entire string is consumed
- at least three numbers exist

Otherwise, the path is rejected.

## Go Solution

```
func splitIntoFibonacci(num string) []int {
	const MAX_INT = 1<<31 - 1

	sequence := []int{}

	var backtrack func(index int) bool

	backtrack = func(index int) bool {
		if index == len(num) {
			return len(sequence) >= 3
		}

		current := 0

		for end := index; end < len(num); end++ {
			// Leading zero check
			if end > index && num[index] == '0' {
				break
			}

			current = current*10 + int(num[end]-'0')

			// 32-bit integer limit
			if current > MAX_INT {
				break
			}

			size := len(sequence)

			if size >= 2 {
				expected := sequence[size-1] + sequence[size-2]

				if current < expected {
					continue
				}

				if current > expected {
					break
				}
			}

			sequence = append(sequence, current)

			if backtrack(end + 1) {
				return true
			}

			sequence = sequence[:len(sequence)-1]
		}

		return false
	}

	if backtrack(0) {
		return sequence
	}

	return []int{}
}
```

The Go solution is structurally identical to the Python version, but there are several language-specific considerations.

Go slices are used for the sequence. Backtracking removes the last element using:

```
sequence = sequence[:len(sequence)-1]
```

Digit parsing is handled manually using:

```
int(num[end] - '0')
```

instead of built-in conversion functions.

Go does not distinguish between `nil` and empty slices in LeetCode output formatting, but returning `[]int{}` makes the intent explicit.

Overflow handling is especially important in Go because integer overflow does not raise exceptions automatically.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n²) in practice | Limited first and second number choices, aggressive pruning |
| Space | O(n) | Recursion stack and sequence storage |

The theoretical worst-case behavior can appear exponential, but practical pruning makes the runtime much closer to quadratic.

The 32-bit integer restriction is extremely important because it limits candidate numbers to at most 10 digits. That bounds the branching factor significantly.

Space complexity is linear because the recursion depth and sequence length are both proportional to the number of digits in the string.

## Edge Cases

### Leading Zeroes

Inputs such as `"0123"` are dangerous because naive substring parsing may incorrectly accept `"01"` as the integer `1`.

The implementation explicitly checks:

```
if end > index and num[index] == '0':
    break
```

This allows `"0"` but immediately rejects longer numbers beginning with zero.

### Integer Overflow

The problem requires every number to fit within a signed 32-bit integer range.

Without explicit overflow checks, large substrings could produce invalid results or overflow behavior in some languages.

The implementation stops exploring as soon as:

```
current > 2**31 - 1
```

This guarantees all produced values satisfy the constraint.

### Multiple Valid Answers

Some inputs can generate more than one valid Fibonacci-like sequence.

For example:

```
"1101111"
```

can produce:

```
[11,0,11,11]
```

or:

```
[110,1,111]
```

The problem allows returning any valid sequence. The backtracking algorithm naturally returns the first successful sequence it encounters.

### All Zeroes

An input such as `"0000"` is easy to mishandle because many implementations reject all multi-digit zero-prefixed numbers.

However, the valid split is:

```
[0,0,0,0]
```

since every individual number is exactly `"0"`.

The implementation handles this correctly because single-character zero substrings are allowed.
