## Solution

## Problem Understanding

In this problem, we control a car moving on an infinite number line. The car starts at position `0` with speed `+1`. We are given two possible instructions:

- `'A'` for accelerate
- `'R'` for reverse

The accelerate instruction updates both the position and the speed:

- `position += speed`
- `speed *= 2`

This means every accelerate command doubles the current speed while moving the car in the current direction.

The reverse instruction changes only the direction of movement:

- If the speed is positive, it becomes `-1`
- If the speed is negative, it becomes `1`

Importantly, reversing does not move the car.

The goal is to reach a target position using the shortest possible instruction sequence.

The input is a single integer `target`, representing the destination position we want to reach. The output is the minimum number of instructions required to land exactly on that position.

The constraint `1 <= target <= 10^4` is extremely important. A naive search over all instruction sequences grows exponentially because each step branches into two choices. Since the target can be as large as 10,000, brute force without optimization becomes infeasible.

The most important observation is that repeated acceleration produces positions of the form:

$$1, 3, 7, 15, 31, \dots$$

After `k` accelerations, the position becomes:

$$2^k - 1$$

This structure strongly suggests a dynamic programming solution.

Several edge cases can trip up an implementation:

- The target may already be exactly reachable by repeated accelerations, such as `3`, `7`, or `15`
- Overshooting the target may still be optimal
- Sometimes it is better to reverse before overshooting
- The optimal path may temporarily move away from the target

The problem guarantees the target is positive and reasonably bounded, which makes memoized dynamic programming practical.

## Approaches

### Brute Force Approach

A straightforward solution is to perform a breadth first search over all possible states.

Each state consists of:

- Current position
- Current speed

From any state, we can generate two next states:

- Accelerate
- Reverse

Since every instruction has equal cost, BFS guarantees the first time we reach the target is the shortest instruction sequence.

This approach is correct because BFS explores states in increasing order of instruction count.

However, the state space grows very quickly. Speeds double repeatedly, positions can become very large in both positive and negative directions, and many redundant states appear. Even with pruning, the number of states becomes enormous.

This makes pure BFS impractical for the upper constraint.

### Optimal Dynamic Programming Approach

The key insight is that after `k` accelerations, the car reaches:

$$2^k - 1$$

This creates natural milestones.

For any target `t`, there are two possibilities:

1. Reach or overshoot the target with repeated accelerations
2. Reverse before overshooting and approach from another direction

Suppose:

$$2^{k-1} - 1 < t < 2^k - 1$$

Then we have two strategies:

- Overshoot to `2^k - 1`, reverse, and solve the remaining distance
- Stop before overshooting, reverse, move backward some distance, reverse again, then approach the target

These overlapping subproblems make dynamic programming ideal.

We define:

$$dp[t] = \text{minimum instructions to reach } t$$

Memoization ensures each target is solved once.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | Exponential | Explores all reachable states with BFS |
| Optimal | O(target log target) | O(target) | Dynamic programming with memoization |

## Algorithm Walkthrough

1. Define a recursive function `dfs(target)` that returns the minimum instructions needed to reach `target`.
2. Compute the minimum number of accelerations `k` such that:

$$2^k - 1 \ge target$$

This identifies the first acceleration sequence that reaches or passes the target.
3. If:

$$2^k - 1 = target$$

then the answer is simply `k`, because exactly `k` accelerations land on the target.
4. Otherwise, consider the overshoot strategy.

Accelerate `k` times to reach:

$$2^k - 1$$

Then reverse once and recursively solve the remaining distance:

$$(2^k - 1) - target$$

Total instructions:

$$k + 1 + dfs((2^k - 1) - target)$$
5. Next, consider reversing before overshooting.

Accelerate only `k - 1` times to reach:

$$2^{k-1} - 1$$

Reverse once, move backward with `m` accelerations, reverse again, then approach the remaining distance.
6. After moving backward `m` times, the backward distance traveled is:

$$2^m - 1$$

The current position becomes:

$$(2^{k-1} - 1) - (2^m - 1)$$
7. The remaining distance to the target is:

$$target - \left((2^{k-1} - 1) - (2^m - 1)\right)$$
8. Total instructions for this strategy are:

$$(k - 1) + 1 + m + 1 + dfs(remaining)$$

which simplifies to:

$$k + m + 1 + dfs(remaining)$$
9. Try all valid `m` values and take the minimum answer.
10. Store results in a memoization table so repeated subproblems are solved only once.

### Why it works

The algorithm works because every optimal path must eventually fit one of two patterns:

- Overshoot the target and come back
- Reverse before overshooting

The recursive decomposition always reduces the remaining distance, and memoization guarantees optimal substructure is reused efficiently. Since every possible optimal turning point is considered, the minimum computed value is globally optimal.

## Python Solution

```
from functools import lru_cache

class Solution:
    def racecar(self, target: int) -> int:

        @lru_cache(None)
        def dfs(t: int) -> int:
            k = t.bit_length()

            # Exact match: 2^k - 1 == t
            if (1 << k) - 1 == t:
                return k

            # Option 1: overshoot and reverse
            answer = k + 1 + dfs((1 << k) - 1 - t)

            # Option 2: reverse before overshooting
            for m in range(k - 1):
                forward_pos = (1 << (k - 1)) - 1
                backward_pos = (1 << m) - 1

                remaining = t - (forward_pos - backward_pos)

                instructions = (k - 1) + 1 + m + 1 + dfs(remaining)

                answer = min(answer, instructions)

            return answer

        return dfs(target)
```

The implementation uses a recursive memoized dynamic programming function named `dfs`.

The first step computes `k`, the smallest number of accelerations needed to reach or exceed the target. Python's `bit_length()` provides this efficiently.

If repeated accelerations land exactly on the target, the function immediately returns `k`.

Otherwise, the code evaluates the overshoot strategy first. This corresponds to accelerating `k` times, reversing once, and solving the smaller remaining distance recursively.

The second loop evaluates all possibilities where we reverse before overshooting. The variable `m` represents how many backward accelerations we perform after the first reverse.

For each possibility, the code computes the remaining distance and recursively solves it.

Memoization through `@lru_cache` ensures every target value is computed only once.

## Go Solution

```
package main

func racecar(target int) int {
	memo := make(map[int]int)

	var dfs func(int) int

	min := func(a, b int) int {
		if a < b {
			return a
		}
		return b
	}

	dfs = func(t int) int {
		if val, exists := memo[t]; exists {
			return val
		}

		k := 0
		for (1<<k)-1 < t {
			k++
		}

		// Exact match
		if (1<<k)-1 == t {
			memo[t] = k
			return k
		}

		// Overshoot strategy
		answer := k + 1 + dfs((1<<k)-1-t)

		// Reverse before overshooting
		for m := 0; m < k-1; m++ {
			forwardPos := (1 << (k - 1)) - 1
			backwardPos := (1 << m) - 1

			remaining := t - (forwardPos - backwardPos)

			instructions := (k - 1) + 1 + m + 1 + dfs(remaining)

			answer = min(answer, instructions)
		}

		memo[t] = answer
		return answer
	}

	return dfs(target)
}
```

The Go implementation mirrors the Python logic closely.

Since Go does not provide built in memoization decorators, a `map[int]int` is used manually for caching computed states.

The value of `k` is computed using a loop instead of `bit_length()`.

Go integers are sufficient for the constraint range, so integer overflow is not a concern here.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(target log target) | Each state tries at most log(target) transitions |
| Space | O(target) | Memoization stores one result per target |

The recursive function computes each target value once because of memoization.

For each target, the algorithm iterates over at most `log(target)` possible reversal points. Since there are at most `target` distinct states, the total complexity becomes:

$$O(target \log target)$$

The memoization cache requires linear space.

## Edge Cases

One important edge case occurs when the target is exactly:

$$2^k - 1$$

Examples include `1`, `3`, `7`, and `15`. In these cases, the optimal solution is simply repeated acceleration with no reversals. A buggy implementation might still attempt recursive decomposition and produce a longer answer. The implementation avoids this by immediately returning `k` when an exact match is detected.

Another tricky case occurs when overshooting is optimal. For example, target `6` is reached most efficiently by going to `7`, reversing, and stepping backward once. Some naive strategies try to avoid overshooting entirely, which can miss optimal solutions. The dynamic programming solution explicitly considers overshoot paths.

A third important edge case occurs when reversing before overshooting produces the best result. Some targets are cheaper to reach by partially advancing, reversing backward briefly, then correcting direction again. The loop over all `m` values ensures every such turning configuration is considered.

Finally, very large targets near the upper constraint could cause exponential recursion without memoization. The cache guarantees each subproblem is solved only once, keeping the runtime manageable even for `target = 10000`.
