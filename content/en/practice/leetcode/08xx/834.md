## Solution

## Problem Understanding

The problem gives us an undirected tree with `n` nodes labeled from `0` to `n - 1`. A tree is a connected graph with exactly `n - 1` edges and no cycles. Every pair of nodes is connected by exactly one simple path.

For every node `i`, we must compute the total distance from node `i` to all other nodes in the tree. The distance between two nodes is the number of edges on the unique path connecting them.

The input consists of:

- An integer `n`, the number of nodes
- A list `edges`, where each edge connects two nodes

The output is an array `answer` where:

- `answer[i]` equals the sum of distances from node `i` to every other node

For example, if node `0` is connected to five other nodes with distances `1, 1, 2, 2, 2`, then:

```
answer[0] = 1 + 1 + 2 + 2 + 2 = 8
```

The constraints are important:

- `n` can be as large as `3 * 10^4`
- The graph is guaranteed to be a valid tree

These constraints immediately rule out expensive algorithms. Since there can be up to thirty thousand nodes, any solution worse than roughly `O(n log n)` or `O(n)` is likely too slow.

A naive solution that computes shortest paths from every node independently would take `O(n^2)` time, which is too expensive for the upper bound.

There are several important edge cases:

- A tree with only one node, where the answer must be `[0]`
- A tree with two nodes, where both answers are `1`
- Highly unbalanced trees, such as a linked-list shaped tree
- Star-shaped trees where one node connects to all others

The problem guarantees the graph is connected and acyclic, so we do not need to handle invalid inputs or disconnected components.

## Approaches

### Brute Force Approach

The most direct solution is to compute the distance from every node to all other nodes independently.

For each node:

1. Run BFS or DFS
2. Compute the distance to every other node
3. Sum those distances

Since a tree has `n` nodes and `n - 1` edges, a BFS or DFS from one node takes `O(n)` time.

Doing this for all `n` nodes results in:

```
O(n * n) = O(n^2)
```

This approach is correct because BFS computes shortest path distances in an unweighted graph, and a tree is an unweighted graph.

However, when `n = 30000`, an `O(n^2)` solution performs roughly nine hundred million operations, which is too slow.

### Optimal Approach

The key observation is that many distance calculations overlap.

Instead of independently computing distances for every node, we can reuse information using a technique called rerooting dynamic programming on trees.

The solution works in two DFS passes:

1. First DFS:

- Compute the size of every subtree
- Compute the sum of distances from an arbitrary root, usually node `0`
2. Second DFS:

- Reuse the parent's answer to compute the child's answer in constant time

The crucial insight is this:

When moving the root from a parent node to a child node:

- Nodes inside the child's subtree become one step closer
- All other nodes become one step farther

If:

- `subtree_size[child]` nodes become closer
- `n - subtree_size[child]` nodes become farther

Then:

```
answer[child] =
answer[parent]
- subtree_size[child]
+ (n - subtree_size[child])
```

This allows every answer to be computed in linear time overall.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n²) | O(n) | BFS or DFS from every node |
| Optimal | O(n) | O(n) | Two DFS traversals with rerooting DP |

## Algorithm Walkthrough

### Step 1: Build the adjacency list

Since the graph is undirected, every edge connects nodes in both directions.

We create an adjacency list:

```
graph[u].append(v)
graph[v].append(u)
```

This representation is efficient for tree traversal.

### Step 2: Initialize arrays

We maintain two important arrays:

- `subtree_size[i]`

- Number of nodes in the subtree rooted at `i`
- `answer[i]`

- Sum of distances from node `i` to all other nodes

Initially:

```
subtree_size[i] = 1
```

because every node counts itself.

### Step 3: First DFS, compute subtree sizes and root distance sum

We root the tree at node `0`.

During DFS:

1. Visit each child
2. Recursively process the child's subtree
3. Add the child's subtree size into the current node's subtree size
4. Update the root distance sum

The recurrence is:

```
subtree_size[node] += subtree_size[child]

answer[0] += depth
```

At the end:

- `subtree_size[i]` contains subtree sizes
- `answer[0]` contains the total distance from node `0` to all nodes

### Step 4: Second DFS, reroot the tree

Now we propagate answers from parent to child.

Suppose we know:

```
answer[parent]
```

When moving the root from `parent` to `child`:

- Every node inside the child's subtree becomes one edge closer
- Every other node becomes one edge farther

So:

```
answer[child] =
answer[parent]
- subtree_size[child]
+ (n - subtree_size[child])
```

We recursively apply this formula throughout the tree.

### Step 5: Return the result

After the second DFS, every `answer[i]` has been computed.

### Why it works

The algorithm relies on a structural property of trees.

When rerooting from a parent to a child:

- Exactly the nodes inside the child's subtree decrease distance by `1`
- Exactly the nodes outside the child's subtree increase distance by `1`

No node is missed or double counted because trees contain unique paths and no cycles.

The first DFS gathers enough information about subtree sizes. The second DFS transforms one node's answer into another in constant time. Since each edge is traversed only a constant number of times, the algorithm is linear.

## Python Solution

```
from typing import List
from collections import defaultdict

class Solution:
    def sumOfDistancesInTree(self, n: int, edges: List[List[int]]) -> List[int]:
        graph = defaultdict(list)

        for u, v in edges:
            graph[u].append(v)
            graph[v].append(u)

        subtree_size = [1] * n
        answer = [0] * n

        def dfs1(node: int, parent: int, depth: int) -> None:
            answer[0] += depth

            for neighbor in graph[node]:
                if neighbor == parent:
                    continue

                dfs1(neighbor, node, depth + 1)

                subtree_size[node] += subtree_size[neighbor]

        def dfs2(node: int, parent: int) -> None:
            for neighbor in graph[node]:
                if neighbor == parent:
                    continue

                answer[neighbor] = (
                    answer[node]
                    - subtree_size[neighbor]
                    + (n - subtree_size[neighbor])
                )

                dfs2(neighbor, node)

        dfs1(0, -1, 0)
        dfs2(0, -1)

        return answer
```

The implementation starts by building an adjacency list using `defaultdict(list)`. Since the tree is undirected, each edge is inserted in both directions.

The `subtree_size` array tracks how many nodes belong to each subtree. Every node starts with size `1` because each subtree includes the node itself.

The first DFS performs two tasks simultaneously:

- It computes subtree sizes
- It computes the total distance sum for node `0`

The `depth` parameter represents the current node's distance from the root. Adding all depths together gives the total distance from node `0`.

After processing a child subtree, the child's subtree size is added into the current node's subtree size.

The second DFS performs rerooting DP. Once the answer for a parent is known, the answer for each child can be derived in constant time using the rerooting formula.

Finally, the filled `answer` array is returned.

## Go Solution

```
package main

func sumOfDistancesInTree(n int, edges [][]int) []int {
	graph := make([][]int, n)

	for _, edge := range edges {
		u := edge[0]
		v := edge[1]

		graph[u] = append(graph[u], v)
		graph[v] = append(graph[v], u)
	}

	subtreeSize := make([]int, n)
	answer := make([]int, n)

	for i := 0; i < n; i++ {
		subtreeSize[i] = 1
	}

	var dfs1 func(node, parent, depth int)

	dfs1 = func(node, parent, depth int) {
		answer[0] += depth

		for _, neighbor := range graph[node] {
			if neighbor == parent {
				continue
			}

			dfs1(neighbor, node, depth+1)

			subtreeSize[node] += subtreeSize[neighbor]
		}
	}

	var dfs2 func(node, parent int)

	dfs2 = func(node, parent int) {
		for _, neighbor := range graph[node] {
			if neighbor == parent {
				continue
			}

			answer[neighbor] =
				answer[node] -
					subtreeSize[neighbor] +
					(n - subtreeSize[neighbor])

			dfs2(neighbor, node)
		}
	}

	dfs1(0, -1, 0)
	dfs2(0, -1)

	return answer
}
```

The Go implementation follows the same logic as the Python version.

Instead of `defaultdict`, Go uses a slice of slices:

```
graph := make([][]int, n)
```

Recursive DFS functions are declared using function variables because Go requires recursive anonymous functions to be assigned before use.

Go slices are dynamically sized, so adjacency lists naturally grow as edges are added.

The constraints fit comfortably within Go's `int` type, so no special overflow handling is necessary.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each edge is traversed a constant number of times |
| Space | O(n) | Adjacency list, recursion stack, subtree array, and answer array |

The tree contains `n - 1` edges. Both DFS traversals process each edge once in each direction, resulting in linear time complexity.

The adjacency list stores all edges, which requires `O(n)` space. The recursion stack may also grow to `O(n)` in a highly unbalanced tree.

## Edge Cases

### Single Node Tree

A tree containing only one node is the smallest valid input.

In this case:

```
n = 1
edges = []
```

There are no other nodes to measure distance to, so the answer must be `[0]`.

This case can expose bugs in implementations that assume every node has neighbors or that DFS always traverses edges. The implementation handles this naturally because the adjacency list is empty and both DFS traversals terminate immediately.

### Linear Chain Tree

A tree shaped like a linked list is highly unbalanced:

```
0 - 1 - 2 - 3 - 4
```

This structure creates maximum recursion depth and highly asymmetric subtree sizes.

Naive implementations may accidentally recompute distances repeatedly, causing quadratic behavior. The rerooting DP solution still processes each edge only a constant number of times, preserving linear complexity.

### Star-Shaped Tree

A star-shaped tree has one central node connected to all others:

```
    1
    |
2 - 0 - 3
    |
    4
```

The center node has minimal total distance, while all leaf nodes have much larger totals.

This case tests whether subtree size calculations are correct. When rerooting from the center to a leaf, nearly all nodes become farther away. The formula:

```
answer[child] =
answer[parent]
- subtree_size[child]
+ (n - subtree_size[child])
```

correctly captures this shift.

### Deep Recursive Trees

A deeply skewed tree may produce recursion depth close to `n`.

Although the algorithm remains `O(n)`, recursive DFS implementations in some languages may hit recursion limits. Python solutions on LeetCode typically work within the platform's recursion allowance for this problem, but in production systems an iterative DFS might be preferred for very deep trees.
