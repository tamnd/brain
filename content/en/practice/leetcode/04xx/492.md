## Solution

## Problem Understanding

The problem gives us a single integer `area`, representing the total area of a rectangle. We need to find two integers, `L` and `W`, such that:

- `L * W == area`
- `L >= W`
- `L - W` is as small as possible

In other words, we are looking for two factors of `area` that are as close to each other as possible. The closer the two numbers are, the more square-like the rectangle becomes.

For example, if the area is `36`, the possible factor pairs are:

- `(36, 1)`
- `(18, 2)`
- `(12, 3)`
- `(9, 4)`
- `(6, 6)`

Among these, `(6, 6)` has the smallest difference, so it is the correct answer.

The input is guaranteed to satisfy `1 <= area <= 10^7`, which means the number can be fairly large, but not so large that efficient square root operations become infeasible. Since the area is always positive, we never need to handle invalid or negative inputs.

The key observation from the constraints is that a naive search through every possible pair could still work for `10^7`, but there is a significantly better mathematical approach. The problem is fundamentally about factorization and divisor pairing.

There are several important edge cases:

- When `area` is a perfect square, the optimal rectangle is a square itself. For example, `area = 49` gives `[7, 7]`.
- When `area` is prime, the only valid factor pair is `[area, 1]`.
- When `area = 1`, the only rectangle possible is `[1, 1]`.
- Large composite numbers require an efficient divisor search to avoid unnecessary iterations.

## Approaches

### Brute Force Approach

A straightforward approach is to test every possible width from `1` up to `area`. For every number `w` that divides `area`, we compute:

```
l = area / w
```

We then check whether this pair produces a smaller difference than previous candidates.

This method is correct because it examines every possible factor pair, guaranteeing that the optimal pair will eventually be found.

However, this approach is inefficient because it may require checking millions of values. With `area = 10^7`, iterating all the way up to the number itself is unnecessary and wasteful.

### Optimal Approach

The important mathematical insight is that factor pairs of a number appear symmetrically around its square root.

For example, for `36`:

- `1 × 36`
- `2 × 18`
- `3 × 12`
- `4 × 9`
- `6 × 6`

As the width increases toward the square root, the length decreases toward it. The pair with the minimum difference is always the pair closest to the square root.

This means we only need to search downward starting from:

$\sqrt{area}$

The first divisor we encounter will automatically produce the optimal rectangle dimensions.

This reduces the search space dramatically from `O(area)` to `O(sqrt(area))`.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(area) | O(1) | Checks every possible divisor from 1 to area |
| Optimal | O(sqrt(area)) | O(1) | Starts from square root and finds closest factor pair |

## Algorithm Walkthrough

1. Compute the integer square root of `area`.

We begin at the square root because the optimal factor pair must be closest to it. Starting there minimizes the number of checks needed.
2. Iterate downward from the square root to `1`.

For each candidate width `w`, check whether it divides `area` evenly.
3. Check divisibility using modulo.

If:

$area \bmod w = 0$

then `w` is a valid factor.
4. Compute the corresponding length.

Once we find a divisor:

$L = \frac{area}{W}$

Because we started near the square root, this pair automatically has the smallest possible difference.
5. Return `[L, W]`.

Since `w` is iterated downward, `L` will always be greater than or equal to `W`.

### Why it works

Every factor pair of a number lies on opposite sides of the square root. The pair closest to the square root has the minimum difference between the two values. By starting at the square root and moving downward, the first divisor found must produce the optimal rectangle dimensions. No later divisor can produce a smaller difference.

## Python Solution

```
from typing import List
import math

class Solution:
    def constructRectangle(self, area: int) -> List[int]:
        width = int(math.sqrt(area))

        while area % width != 0:
            width -= 1

        length = area // width

        return [length, width]
```

The implementation begins by computing the integer square root of `area`. This gives the largest possible candidate width that could lead to the smallest difference between dimensions.

The `while` loop searches downward until it finds a divisor. The modulo operation checks whether the current width divides the area exactly.

Once a valid divisor is found, the corresponding length is computed using integer division. Since the search starts near the square root, the first valid pair is guaranteed to be optimal.

Finally, the method returns `[length, width]` in the required order.

## Go Solution

```
package main

import "math"

func constructRectangle(area int) []int {
    width := int(math.Sqrt(float64(area)))

    for area%width != 0 {
        width--
    }

    length := area / width

    return []int{length, width}
}
```

The Go implementation follows the same logic as the Python version. The main difference is that `math.Sqrt` operates on `float64`, so explicit type conversion is required.

The result is returned as a slice using `[]int{length, width}`.

Integer overflow is not a concern because the maximum value of `area` is only `10^7`, which easily fits within Go's integer range.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(sqrt(area)) | In the worst case we check every number from sqrt(area) down to 1 |
| Space | O(1) | Only a few variables are used |

The algorithm is efficient because it avoids checking all possible divisors. Since divisors come in symmetric pairs around the square root, searching downward from the square root guarantees that we quickly find the optimal solution.

Even in the worst case, the algorithm performs at most about `sqrt(10^7) ≈ 3162` iterations, which is very fast.

## Edge Cases

### Perfect Square Inputs

When the area is a perfect square, the optimal rectangle is a square itself. This case can sometimes expose off-by-one errors if the square root calculation is handled incorrectly.

For example, with `area = 49`, the algorithm starts at `7`, immediately detects that `49 % 7 == 0`, and returns `[7, 7]`.

The implementation handles this naturally because it begins searching from the square root.

### Prime Numbers

Prime numbers only have two divisors: `1` and themselves. A naive implementation might incorrectly assume a better factor pair exists.

For `area = 37`, the algorithm checks every number down to `1` before finding a valid divisor. It then correctly returns `[37, 1]`.

This ensures correctness even when no balanced rectangle exists.

### Minimum Input Value

The smallest possible input is `area = 1`. Some implementations fail here if they accidentally decrement below `1` or mishandle square root conversion.

In this implementation:

```
sqrt(1) = 1
```

Since `1 % 1 == 0`, the loop terminates immediately and returns `[1, 1]`.

This guarantees proper handling of the lower boundary condition.
