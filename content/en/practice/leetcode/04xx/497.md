## Solution

## Problem Understanding

The problem gives us a collection of non-overlapping axis-aligned rectangles. Each rectangle is represented as `[ai, bi, xi, yi]`, where `(ai, bi)` is the bottom-left corner and `(xi, yi)` is the top-right corner.

The key detail is that we are working with integer coordinate points, and points on the boundary are included. That means the rectangle contains every integer point `(x, y)` such that:

- `ai <= x <= xi`
- `bi <= y <= yi`

For example, the rectangle `[1, 1, 2, 2]` contains these four integer points:

- `(1, 1)`
- `(1, 2)`
- `(2, 1)`
- `(2, 2)`

The task is to design a data structure that supports random sampling of points. Every valid integer point across all rectangles must have exactly the same probability of being returned.

This requirement is important. We are not selecting rectangles uniformly. Larger rectangles contain more integer points, so they must be chosen more often. If one rectangle contains 100 valid points and another contains 10, then the larger rectangle should be selected 10 times more frequently.

The constraints tell us several useful things:

- There are at most 100 rectangles, which is small.
- Each rectangle side length is at most 2000, meaning an individual rectangle may contain millions of integer points.
- There can be up to `10^4` calls to `pick()`.

These constraints strongly suggest that explicitly storing every integer point is not a good idea. A rectangle of size `2001 x 2001` already contains over 4 million points.

The problem also guarantees that rectangles do not overlap. This simplifies the probability model because every integer point belongs to exactly one rectangle.

Several edge cases are important:

- Rectangles may contain only a single point.
- Coordinates may be negative.
- One rectangle may be dramatically larger than the others.
- The rectangles are non-overlapping, so no point duplication needs to be handled.
- Random selection must remain uniform across all points, not across rectangles.

## Approaches

### Brute Force Approach

The most direct approach is to explicitly enumerate every integer point contained in every rectangle.

For each rectangle:

- Iterate through every integer `x` from `ai` to `xi`
- Iterate through every integer `y` from `bi` to `yi`
- Store `(x, y)` into a list

Then, during `pick()`, generate a random index into that list and return the corresponding point.

This approach is correct because every valid point appears exactly once in the list. Choosing a random index uniformly therefore produces a perfectly uniform random point.

However, this approach becomes extremely memory-intensive. A single rectangle may contain millions of points, and storing all points across rectangles could require enormous memory. The preprocessing time is also very large.

### Optimal Approach

The key observation is that we do not need to store every point individually.

Instead:

1. Compute how many integer points each rectangle contains.
2. Build a prefix sum array over those counts.
3. Randomly select a global point index.
4. Use binary search to determine which rectangle contains that index.
5. Randomly generate a point inside that rectangle.

Suppose:

- Rectangle A contains 4 points
- Rectangle B contains 10 points
- Rectangle C contains 6 points

Then the prefix sums become:

| Rectangle | Point Count | Prefix Sum |
| --- | --- | --- |
| A | 4 | 4 |
| B | 10 | 14 |
| C | 6 | 20 |

Now generate a random integer from `1` to `20`.

- `1..4` maps to A
- `5..14` maps to B
- `15..20` maps to C

This automatically gives each rectangle probability proportional to its number of points.

Once a rectangle is selected, generating a random integer coordinate inside it is straightforward.

## Comparison of Approaches

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(total_points) preprocessing, O(1) pick | O(total_points) | Explicitly stores every integer point |
| Optimal | O(n) preprocessing, O(log n) pick | O(n) | Uses prefix sums and binary search |

## Algorithm Walkthrough

1. During initialization, compute how many integer points each rectangle contains.

For a rectangle `[a, b, x, y]`, the number of integer points is:

$(x-a+1)(y-b+1)$

The `+1` is necessary because boundaries are inclusive.
2. Build a prefix sum array.

Each entry stores the total number of points covered by all rectangles up to that index.

For example:

| Rectangle | Points | Prefix Sum |
| --- | --- | --- |
| 0 | 4 | 4 |
| 1 | 10 | 14 |
| 2 | 6 | 20 |

This structure allows us to map a random point index to a rectangle.
3. In `pick()`, generate a random integer between `1` and `total_points`.

Every integer in this range corresponds to exactly one valid point across all rectangles.
4. Use binary search on the prefix sum array to locate the rectangle containing the chosen index.

If the random number is `12`:

- It is greater than `4`
- It is less than or equal to `14`

Therefore it belongs to rectangle `1`.
5. Once the rectangle is identified, randomly generate:

- `x` between `a` and `x`
- `y` between `b` and `y`

Since every point inside the rectangle is equally likely, and rectangles are selected proportional to their size, every valid point overall has equal probability.

### Why it works

The algorithm creates a one-to-one mapping between integers in the range `[1, total_points]` and valid integer points across all rectangles.

Each rectangle occupies a contiguous range in the prefix sum array whose size equals the number of points inside that rectangle. Selecting a random integer uniformly therefore selects rectangles proportionally to their area in integer-point space.

Then, because points inside the chosen rectangle are selected uniformly, every valid point across all rectangles has identical probability.

## Python Solution

```
from typing import List
import random
import bisect

class Solution:

    def __init__(self, rects: List[List[int]]):
        self.rects = rects
        self.prefix_sums = []

        total_points = 0

        for a, b, x, y in rects:
            point_count = (x - a + 1) * (y - b + 1)
            total_points += point_count
            self.prefix_sums.append(total_points)

    def pick(self) -> List[int]:
        target = random.randint(1, self.prefix_sums[-1])

        rect_index = bisect.bisect_left(self.prefix_sums, target)

        a, b, x, y = self.rects[rect_index]

        random_x = random.randint(a, x)
        random_y = random.randint(b, y)

        return [random_x, random_y]

# Your Solution object will be instantiated and called as such:
# obj = Solution(rects)
# param_1 = obj.pick()
```

The implementation begins by storing the rectangles and constructing a prefix sum array.

For each rectangle, the number of integer points is computed using inclusive dimensions. These counts accumulate into `total_points`, and each cumulative total is appended to `prefix_sums`.

During `pick()`, the algorithm generates a random integer between `1` and the total number of valid points.

The `bisect_left` function performs binary search on the prefix sum array. This efficiently identifies which rectangle owns the randomly selected point index.

Finally, a random integer coordinate is generated independently for the `x` and `y` dimensions inside that rectangle.

Because rectangle selection probability is proportional to point count, and point selection inside a rectangle is uniform, the final distribution is perfectly uniform across all valid points.

## Go Solution

```
package main

import (
	"math/rand"
	"sort"
)

type Solution struct {
	rects      [][]int
	prefixSums []int
}

func Constructor(rects [][]int) Solution {
	prefixSums := make([]int, 0, len(rects))

	totalPoints := 0

	for _, rect := range rects {
		a, b, x, y := rect[0], rect[1], rect[2], rect[3]

		pointCount := (x - a + 1) * (y - b + 1)

		totalPoints += pointCount
		prefixSums = append(prefixSums, totalPoints)
	}

	return Solution{
		rects:      rects,
		prefixSums: prefixSums,
	}
}

func (this *Solution) Pick() []int {
	totalPoints := this.prefixSums[len(this.prefixSums)-1]

	target := rand.Intn(totalPoints) + 1

	rectIndex := sort.Search(len(this.prefixSums), func(i int) bool {
		return this.prefixSums[i] >= target
	})

	rect := this.rects[rectIndex]

	a, b, x, y := rect[0], rect[1], rect[2], rect[3]

	randomX := rand.Intn(x-a+1) + a
	randomY := rand.Intn(y-b+1) + b

	return []int{randomX, randomY}
}

/**
 * Your Solution object will be instantiated and called as such:
 * obj := Constructor(rects);
 * param_1 := obj.Pick();
 */
```

The Go implementation follows the same algorithmic structure as the Python solution.

Instead of `bisect_left`, Go uses `sort.Search` for binary search. The random number generation uses `rand.Intn`, which returns values in `[0, n)`, so `+1` is required when generating the target prefix sum index.

Slices are used for both rectangles and prefix sums. Since Go integers are sufficiently large for the problem constraints, no special overflow handling is required.

- Random numbers `1..16` map to rectangle `0`

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) preprocessing, O(log n) per pick | Prefix sums require linear construction, binary search handles selection |
| Space | O(n) | Stores prefix sums |

The preprocessing phase iterates through all rectangles once to compute cumulative point counts.

Each `pick()` operation performs binary search over at most 100 rectangles, making the runtime logarithmic in the number of rectangles. Since `n <= 100`, this is extremely efficient.

The algorithm avoids storing individual points, which dramatically reduces memory usage.

## Edge Cases

One important edge case is a rectangle that contains only a single point, such as `[5, 5, 5, 5]`. A naive implementation might incorrectly compute width or height as zero and conclude the rectangle contains no points. The correct formula uses inclusive bounds, so the point count becomes `(5 - 5 + 1) * (5 - 5 + 1) = 1`. The implementation handles this naturally.

Another important edge case involves negative coordinates. Since coordinates may range down to `-10^9`, implementations that incorrectly assume non-negative indices can fail. This solution never uses coordinates as array indices. Instead, it directly generates random integers within coordinate ranges, so negative values work without any special handling.

A third edge case occurs when one rectangle is vastly larger than the others. For example, one rectangle may contain millions of points while another contains only one. A flawed solution that selects rectangles uniformly would produce incorrect probabilities. The prefix sum strategy avoids this issue because rectangles are selected proportionally to their point counts.

A final edge case involves very thin rectangles, such as horizontal or vertical lines. A rectangle like `[0, 5, 10, 5]` has height `1`, not `0`, because the endpoints are inclusive. The implementation correctly computes point counts and random coordinate ranges using inclusive arithmetic.
