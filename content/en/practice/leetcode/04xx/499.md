## Solution

## Problem Understanding

This problem asks us to navigate a rolling ball through a maze and determine the shortest sequence of moves that causes the ball to fall into a hole.

The maze is represented as a 2D grid where:

- `0` means an empty cell the ball can roll through
- `1` means a wall
- the ball starts at position `ball`
- the destination hole is at position `hole`

The important detail is that the ball does not move one step at a time. Instead, once it starts rolling in a direction, it continues moving until either:

- it hits a wall, or
- it falls into the hole

This changes the problem significantly from a normal grid BFS problem because each move can traverse many cells.

The goal is not merely to determine whether the hole is reachable. We must return the instruction string describing the path with:

1. the minimum total travel distance
2. if multiple paths have the same shortest distance, the lexicographically smallest instruction string

The instruction string uses:

- `'u'` for up
- `'d'` for down
- `'l'` for left
- `'r'` for right

The distance is measured as the number of empty spaces traveled, including the hole cell but excluding the starting cell.

The constraints allow grids up to `100 x 100`, which means up to 10,000 cells. A naive exhaustive search over all possible rolling sequences would become extremely expensive because each state can branch in four directions and cycles are possible.

Several edge cases are especially important:

- The hole may be encountered before hitting a wall. The rolling simulation must stop immediately when the hole is reached.
- Multiple shortest paths may exist. We must choose the lexicographically smallest instruction string.
- The ball may revisit the same stopping cell with a shorter distance or lexicographically smaller path.
- Some cells are not true stopping positions because the ball rolls through them.
- The hole may be unreachable.

This problem is fundamentally a shortest path problem on an implicit weighted graph.

## Approaches

### Brute Force Approach

A brute force solution would try all possible sequences of directions using DFS backtracking.

From each stopping position, we could recursively roll in four directions, simulate where the ball lands, and continue exploring until either:

- the hole is reached, or
- all possibilities are exhausted

To avoid infinite recursion, we would need some kind of visited tracking. However, this becomes difficult because revisiting a cell with a different path length or instruction sequence may still produce a better answer later.

The brute force approach is correct because it eventually explores every possible route to the hole. Unfortunately, the number of possible rolling sequences grows exponentially. Since cycles are possible, the search space becomes extremely large.

Even with pruning, this approach is too slow for a `100 x 100` grid.

### Key Insight

The important observation is that this is actually a shortest path problem on a weighted graph.

Each stopping position acts like a graph node. Rolling in one direction produces an edge whose weight equals the traveled distance.

Because edge weights are not uniform, normal BFS is insufficient. A move may travel many cells, so edges have varying costs.

This naturally suggests Dijkstra's algorithm.

Dijkstra's algorithm works perfectly because:

- all edge weights are non-negative
- we need the shortest total distance
- we may revisit nodes if we find a shorter path

The lexicographical tie-breaking can also be integrated naturally. When two paths reach the same position with equal distance, we keep the lexicographically smaller instruction string.

Thus, each state in the priority queue contains:

- total distance traveled
- instruction string
- current row
- current column

The priority queue automatically prioritizes:

1. smaller distance
2. lexicographically smaller path

This guarantees that the first time we pop the hole from the heap, we have the optimal answer.

## Algorithm Walkthrough

### Step 1: Define the movement directions

We define the four possible rolling directions:

- up
- down
- left
- right

Each direction also has its corresponding instruction character.

We intentionally process directions in lexicographical order:

- `d`
- `l`
- `r`
- `u`

This helps naturally preserve lexicographical ordering during heap processing.

### Step 2: Initialize Dijkstra's priority queue

We use a min-heap where each entry contains:

1. total distance traveled
2. path string
3. current row
4. current column

The heap automatically prioritizes:

- smaller distance first
- lexicographically smaller path if distances tie

Initially:

- distance = `0`
- path = `""`
- position = starting ball position

### Step 3: Store best known states

We maintain a dictionary:

```
best[(row, col)] = (distance, path)
```

This stores the best known way to reach each stopping position.

A new state is considered better if:

- it has smaller distance, or
- same distance but lexicographically smaller path

### Step 4: Pop the best state from the heap

We repeatedly pop the minimum state from the heap.

If the popped position is the hole, we immediately return the path because Dijkstra guarantees optimality.

### Step 5: Roll in all four directions

For each direction:

1. start from the current cell
2. keep rolling until:

- a wall is hit, or
- the hole is reached
3. count traveled distance
4. determine final stopping position

The rolling simulation is the core of the problem.

A critical detail is that we must stop immediately if the hole is encountered during rolling, even if no wall has been reached yet.

### Step 6: Relax edges

After rolling:

- compute new total distance
- compute new path string

If this state improves the best known state for the destination position, update it and push it into the heap.

### Step 7: Continue until heap is empty

If the heap becomes empty without reaching the hole, return `"impossible"`.

### Why it works

Dijkstra's algorithm guarantees that when a node is removed from the priority queue, we have already found the minimum-cost path to it.

Our priority queue orders states by:

1. shortest distance
2. lexicographically smallest instruction string

Therefore, the first time the hole is popped from the heap:

- its distance is globally minimal
- among all minimal-distance paths, its instruction string is lexicographically smallest

This exactly matches the problem requirements.

## Python Solution

```
from typing import List
import heapq

class Solution:
    def findShortestWay(
        self,
        maze: List[List[int]],
        ball: List[int],
        hole: List[int]
    ) -> str:

        rows = len(maze)
        cols = len(maze[0])

        directions = [
            (1, 0, 'd'),
            (0, -1, 'l'),
            (0, 1, 'r'),
            (-1, 0, 'u'),
        ]

        heap = [(0, "", ball[0], ball[1])]

        best = {
            (ball[0], ball[1]): (0, "")
        }

        while heap:
            dist, path, row, col = heapq.heappop(heap)

            if [row, col] == hole:
                return path

            if best[(row, col)] < (dist, path):
                continue

            for dr, dc, move in directions:

                nr, nc = row, col
                steps = 0

                while True:
                    next_r = nr + dr
                    next_c = nc + dc

                    if (
                        next_r < 0 or
                        next_r >= rows or
                        next_c < 0 or
                        next_c >= cols or
                        maze[next_r][next_c] == 1
                    ):
                        break

                    nr = next_r
                    nc = next_c
                    steps += 1

                    if [nr, nc] == hole:
                        break

                new_dist = dist + steps
                new_path = path + move

                state = (nr, nc)

                if (
                    state not in best or
                    (new_dist, new_path) < best[state]
                ):
                    best[state] = (new_dist, new_path)
                    heapq.heappush(
                        heap,
                        (new_dist, new_path, nr, nc)
                    )

        return "impossible"
```

The implementation follows the Dijkstra strategy directly.

The `directions` array stores movement vectors and instruction characters together. This keeps the rolling logic compact and easy to understand.

The priority queue stores tuples in the order:

```
(distance, path, row, col)
```

Python tuple comparison is extremely useful here because it automatically compares:

1. distance first
2. path second

This perfectly matches the problem requirements.

The rolling simulation repeatedly advances the ball until:

- a wall blocks further movement, or
- the hole is encountered

The moment the hole is reached, rolling stops immediately.

The `best` dictionary prevents unnecessary processing. A position is revisited only if we discover:

- a shorter distance, or
- an equal distance but lexicographically smaller path

The stale-state check:

```
if best[(row, col)] < (dist, path):
    continue
```

prevents outdated heap entries from being processed again.

Finally, if the heap empties without reaching the hole, the hole is unreachable.

## Go Solution

```
package main

import (
	"container/heap"
)

type State struct {
	dist int
	path string
	row  int
	col  int
}

type PriorityQueue []State

func (pq PriorityQueue) Len() int {
	return len(pq)
}

func (pq PriorityQueue) Less(i, j int) bool {
	if pq[i].dist == pq[j].dist {
		return pq[i].path < pq[j].path
	}
	return pq[i].dist < pq[j].dist
}

func (pq PriorityQueue) Swap(i, j int) {
	pq[i], pq[j] = pq[j], pq[i]
}

func (pq *PriorityQueue) Push(x interface{}) {
	*pq = append(*pq, x.(State))
}

func (pq *PriorityQueue) Pop() interface{} {
	old := *pq
	n := len(old)
	item := old[n-1]
	*pq = old[:n-1]
	return item
}

func findShortestWay(maze [][]int, ball []int, hole []int) string {

	rows := len(maze)
	cols := len(maze[0])

	directions := [][]int{
		{1, 0},
		{0, -1},
		{0, 1},
		{-1, 0},
	}

	moves := []string{"d", "l", "r", "u"}

	pq := &PriorityQueue{}
	heap.Init(pq)

	heap.Push(pq, State{
		dist: 0,
		path: "",
		row:  ball[0],
		col:  ball[1],
	})

	best := make(map[[2]int]State)

	startKey := [2]int{ball[0], ball[1]}
	best[startKey] = State{
		dist: 0,
		path: "",
	}

	for pq.Len() > 0 {

		curr := heap.Pop(pq).(State)

		if curr.row == hole[0] && curr.col == hole[1] {
			return curr.path
		}

		key := [2]int{curr.row, curr.col}

		bestState := best[key]

		if curr.dist > bestState.dist ||
			(curr.dist == bestState.dist &&
				curr.path > bestState.path) {
			continue
		}

		for i := 0; i < 4; i++ {

			dr := directions[i][0]
			dc := directions[i][1]

			nr := curr.row
			nc := curr.col
			steps := 0

			for {

				nextR := nr + dr
				nextC := nc + dc

				if nextR < 0 ||
					nextR >= rows ||
					nextC < 0 ||
					nextC >= cols ||
					maze[nextR][nextC] == 1 {
					break
				}

				nr = nextR
				nc = nextC
				steps++

				if nr == hole[0] && nc == hole[1] {
					break
				}
			}

			newDist := curr.dist + steps
			newPath := curr.path + moves[i]

			nextKey := [2]int{nr, nc}

			prev, exists := best[nextKey]

			if !exists ||
				newDist < prev.dist ||
				(newDist == prev.dist &&
					newPath < prev.path) {

				best[nextKey] = State{
					dist: newDist,
					path: newPath,
				}

				heap.Push(pq, State{
					dist: newDist,
					path: newPath,
					row:  nr,
					col:  nc,
				})
			}
		}
	}

	return "impossible"
}
```

The Go implementation mirrors the Python solution closely, but several language-specific details are worth noting.

Go does not provide a built-in heap with tuple ordering, so we implement the `heap.Interface` manually. The `Less` method handles both:

- distance comparison
- lexicographical path comparison

The `best` map uses a fixed-size array key:

```
[2]int{row, col}
```

This is idiomatic and hashable in Go.

Unlike Python tuples, Go requires explicit comparison logic for stale states and updates.

No integer overflow concerns exist because the maximum path length is bounded by the maze size.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(mn log(mn) × max(m,n)) | Each state may roll across an entire row or column |
| Space | O(mn) | Heap and best-state storage |

The maze contains at most `m × n` stopping positions.

For each position, we may explore four directions. Each rolling simulation can travel up to `max(m,n)` cells.

Dijkstra operations on the heap cost `log(mn)`.

Therefore the total complexity becomes:

```
O(mn × max(m,n) × log(mn))
```

The space usage comes primarily from:

- the priority queue
- the best-state dictionary

Both store at most `mn` states.

## Edge Cases

### The hole is encountered before hitting a wall

This is the most important edge case in the entire problem. A naive rolling simulation might continue rolling until a wall is reached, which would incorrectly skip over the hole.

For example, if the ball rolls horizontally and passes through the hole before reaching a wall, the ball must immediately fall into the hole.

The implementation handles this correctly with:

```
if [nr, nc] == hole:
    break
```

inside the rolling loop.

### Multiple shortest paths exist

Two different instruction sequences may produce the same minimum distance.

The problem requires returning the lexicographically smallest path among them.

This can easily produce incorrect results if we only optimize for distance.

The solution handles this by:

- storing `(distance, path)` pairs
- comparing tuples lexicographically
- ordering the heap by both distance and path

Thus:

```
(6, "lul") < (6, "ul")
```

so `"lul"` is correctly preferred.

### Revisiting the same stopping position

A stopping position may be reached multiple times.

Some revisits are beneficial because they may provide:

- shorter distance
- same distance but better lexicographical ordering

A naive visited set would incorrectly discard such states.

Instead, we store the best known `(distance, path)` pair for every stopping position and only prune truly worse states.

This is why Dijkstra works correctly even with repeated node visits.
