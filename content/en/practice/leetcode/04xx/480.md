## Solution

## Problem Understanding

The problem asks us to compute the median for every contiguous subarray, or sliding window, of size `k` in the array `nums`.

A sliding window starts at the beginning of the array and moves one step to the right at a time. For each position of the window, we must determine the median of the numbers currently inside it.

The median depends on whether the window size is odd or even:

- If `k` is odd, the median is the middle element after sorting the window.
- If `k` is even, the median is the average of the two middle elements after sorting the window.

For example, if the window contains `[1, 3, -1]`, sorting gives `[-1, 1, 3]`, so the median is `1`.

If the window contains `[1, 2, 3, 4]`, sorting gives `[1, 2, 3, 4]`, so the median is `(2 + 3) / 2 = 2.5`.

The input consists of:

- `nums`, an integer array
- `k`, the window size

The output must be an array of floating point values where each value represents the median of one window position.

The constraints are extremely important:

- `nums.length` can be as large as `100000`
- `k` can also be as large as `100000`

These limits immediately rule out inefficient solutions. If we sort every window independently, the runtime becomes too large. We need a data structure that supports:

- Efficient insertion
- Efficient removal
- Efficient access to the median

Another important detail is that values can be duplicated and can also be negative. This means we cannot rely on uniqueness when removing elements from a data structure.

Several edge cases are important:

- `k = 1`, every element is its own median
- Duplicate values, where removing one occurrence must not accidentally remove all occurrences
- Even-sized windows, where the median must be computed as a floating point average
- Negative numbers
- Very large input sizes, where any `O(k log k)` work per window becomes too slow

## Approaches

### Brute Force Approach

The most straightforward solution is to process every sliding window independently.

For each window:

1. Extract the `k` elements
2. Sort them
3. Compute the median from the sorted list

This approach is correct because sorting directly gives the ordered structure needed to determine the median.

However, the performance is poor.

There are `n - k + 1` windows. Sorting each window costs `O(k log k)`, so the total complexity becomes:

```
O((n - k + 1) * k log k)
```

In the worst case, this is far too slow for `n = 100000`.

### Optimal Approach

The key observation is that consecutive windows differ by only two operations:

- One element leaves the window
- One element enters the window

Instead of rebuilding and sorting the entire window every time, we should maintain a dynamic structure that supports incremental updates.

The optimal solution uses two heaps:

- A max heap for the smaller half of numbers
- A min heap for the larger half of numbers

This structure allows us to:

- Access the middle values in `O(1)`
- Insert elements in `O(log k)`
- Remove elements lazily in `O(log k)`

The challenge is removal. Python heaps do not support deleting arbitrary elements efficiently. To solve this, we use lazy deletion with a hash map.

Instead of immediately removing an element from a heap, we mark it as invalid in a dictionary. Whenever that value reaches the top of the heap, we physically remove it.

This keeps all operations efficient while preserving correctness.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O((n - k + 1) * k log k) | O(k) | Sort every window independently |
| Optimal | O(n log k) | O(k) | Two heaps with lazy deletion |

## Algorithm Walkthrough

### Optimal Two Heap Algorithm

1. Maintain two heaps to divide the current window into two halves.

The first heap, called `small`, is a max heap containing the smaller half of the elements. Since Python only provides a min heap, we store negative values to simulate a max heap.

The second heap, called `large`, is a normal min heap containing the larger half of the elements.
2. Keep the heaps balanced.

The balancing rules are:

- `small` may contain at most one more element than `large`
- Every element in `small` must be less than or equal to every element in `large`

This guarantees that the median is always accessible from the heap tops.
3. Insert a new number into one of the heaps.

If the number is smaller than or equal to the maximum element in `small`, it belongs in `small`.

Otherwise, it belongs in `large`.

After insertion, rebalance the heaps if necessary.
4. Remove outgoing elements lazily.

When the window moves, one element leaves the window.

Standard heaps cannot remove arbitrary elements efficiently, so we do not immediately delete it. Instead:

- Record the value in a dictionary called `delayed`
- Reduce the logical size of the appropriate heap

Later, if that value reaches the top of a heap, we physically remove it.
5. Prune invalid heap tops.

Before reading heap tops or rebalancing, repeatedly remove elements that are marked for deletion.

This ensures the heap tops always represent valid window elements.
6. Compute the median.

- If `k` is odd, the median is the top of `small`
- If `k` is even, the median is the average of the tops of both heaps
7. Continue sliding the window until all medians are collected.

### Why it works

The algorithm maintains two critical invariants:

- All elements in `small` are less than or equal to all elements in `large`
- The heaps differ in size by at most one element

Because of these properties:

- When `k` is odd, the extra element in `small` is exactly the median
- When `k` is even, the two middle elements are the tops of the two heaps

Lazy deletion does not affect correctness because invalid elements are removed before heap tops are used.

## Python Solution

```
from typing import List
import heapq
from collections import defaultdict

class DualHeap:
    def __init__(self, k: int):
        self.small = []  # max heap via negative values
        self.large = []  # min heap

        self.delayed = defaultdict(int)

        self.small_size = 0
        self.large_size = 0

        self.k = k

    def prune(self, heap):
        while heap:
            num = -heap[0] if heap == self.small else heap[0]

            if self.delayed[num]:
                heapq.heappop(heap)
                self.delayed[num] -= 1

                if self.delayed[num] == 0:
                    del self.delayed[num]
            else:
                break

    def balance(self):
        if self.small_size > self.large_size + 1:
            num = -heapq.heappop(self.small)
            heapq.heappush(self.large, num)

            self.small_size -= 1
            self.large_size += 1

            self.prune(self.small)

        elif self.small_size < self.large_size:
            num = heapq.heappop(self.large)
            heapq.heappush(self.small, -num)

            self.large_size -= 1
            self.small_size += 1

            self.prune(self.large)

    def insert(self, num: int):
        if not self.small or num <= -self.small[0]:
            heapq.heappush(self.small, -num)
            self.small_size += 1
        else:
            heapq.heappush(self.large, num)
            self.large_size += 1

        self.balance()

    def erase(self, num: int):
        self.delayed[num] += 1

        if num <= -self.small[0]:
            self.small_size -= 1

            if num == -self.small[0]:
                self.prune(self.small)
        else:
            self.large_size -= 1

            if self.large and num == self.large[0]:
                self.prune(self.large)

        self.balance()

    def get_median(self) -> float:
        if self.k % 2 == 1:
            return float(-self.small[0])

        return (-self.small[0] + self.large[0]) / 2.0

class Solution:
    def medianSlidingWindow(self, nums: List[int], k: int) -> List[float]:
        dual_heap = DualHeap(k)
        result = []

        for i in range(k):
            dual_heap.insert(nums[i])

        result.append(dual_heap.get_median())

        for i in range(k, len(nums)):
            dual_heap.insert(nums[i])
            dual_heap.erase(nums[i - k])

            result.append(dual_heap.get_median())

        return result
```

The implementation is organized around the `DualHeap` class, which encapsulates all heap logic.

The `small` heap stores the lower half of numbers as negative values to simulate a max heap. The `large` heap stores the upper half normally.

The `delayed` dictionary tracks elements that should eventually be removed. This avoids the expensive operation of searching inside heaps.

The `insert` method places a value into the correct heap and then restores balance.

The `erase` method marks an element for deletion and adjusts logical heap sizes. Physical removal only occurs when invalid elements reach the top.

The `prune` method repeatedly removes invalid top elements from a heap.

Finally, `get_median` computes the median directly from heap tops.

## Go Solution

```
package main

import (
	"container/heap"
)

type IntHeap struct {
	data []int
	less func(a, b int) bool
}

func (h IntHeap) Len() int {
	return len(h.data)
}

func (h IntHeap) Less(i, j int) bool {
	return h.less(h.data[i], h.data[j])
}

func (h IntHeap) Swap(i, j int) {
	h.data[i], h.data[j] = h.data[j], h.data[i]
}

func (h *IntHeap) Push(x interface{}) {
	h.data = append(h.data, x.(int))
}

func (h *IntHeap) Pop() interface{} {
	old := h.data
	n := len(old)

	x := old[n-1]
	h.data = old[:n-1]

	return x
}

func (h *IntHeap) Top() int {
	return h.data[0]
}

type DualHeap struct {
	small      *IntHeap
	large      *IntHeap
	delayed    map[int]int
	smallSize  int
	largeSize  int
	k          int
}

func NewDualHeap(k int) *DualHeap {
	small := &IntHeap{
		data: []int{},
		less: func(a, b int) bool {
			return a > b
		},
	}

	large := &IntHeap{
		data: []int{},
		less: func(a, b int) bool {
			return a < b
		},
	}

	heap.Init(small)
	heap.Init(large)

	return &DualHeap{
		small:   small,
		large:   large,
		delayed: make(map[int]int),
		k:       k,
	}
}

func (dh *DualHeap) prune(h *IntHeap) {
	for h.Len() > 0 {
		num := h.Top()

		if count, exists := dh.delayed[num]; exists {
			if count == 1 {
				delete(dh.delayed, num)
			} else {
				dh.delayed[num]--
			}

			heap.Pop(h)
		} else {
			break
		}
	}
}

func (dh *DualHeap) balance() {
	if dh.smallSize > dh.largeSize+1 {
		num := heap.Pop(dh.small).(int)
		heap.Push(dh.large, num)

		dh.smallSize--
		dh.largeSize++

		dh.prune(dh.small)

	} else if dh.smallSize < dh.largeSize {
		num := heap.Pop(dh.large).(int)
		heap.Push(dh.small, num)

		dh.largeSize--
		dh.smallSize++

		dh.prune(dh.large)
	}
}

func (dh *DualHeap) insert(num int) {
	if dh.small.Len() == 0 || num <= dh.small.Top() {
		heap.Push(dh.small, num)
		dh.smallSize++
	} else {
		heap.Push(dh.large, num)
		dh.largeSize++
	}

	dh.balance()
}

func (dh *DualHeap) erase(num int) {
	dh.delayed[num]++

	if num <= dh.small.Top() {
		dh.smallSize--

		if num == dh.small.Top() {
			dh.prune(dh.small)
		}
	} else {
		dh.largeSize--

		if dh.large.Len() > 0 && num == dh.large.Top() {
			dh.prune(dh.large)
		}
	}

	dh.balance()
}

func (dh *DualHeap) getMedian() float64 {
	if dh.k%2 == 1 {
		return float64(dh.small.Top())
	}

	return float64(dh.small.Top()+dh.large.Top()) / 2.0
}

func medianSlidingWindow(nums []int, k int) []float64 {
	dh := NewDualHeap(k)
	result := []float64{}

	for i := 0; i < k; i++ {
		dh.insert(nums[i])
	}

	result = append(result, dh.getMedian())

	for i := k; i < len(nums); i++ {
		dh.insert(nums[i])
		dh.erase(nums[i-k])

		result = append(result, dh.getMedian())
	}

	return result
}
```

The Go implementation follows the same logical structure as the Python version, but Go requires explicit heap interfaces through `container/heap`.

Instead of storing negative numbers to simulate a max heap, the implementation defines custom comparison functions through the `less` callback.

Go also requires explicit type assertions when pushing and popping heap elements because the heap interface uses `interface{}`.

The lazy deletion logic remains identical.

## Example 1

Input:

```
nums = [1,3,-1,-3,5,3,6,7]

```

Since `k = 3` is odd, the median is always the top of the max heap.

| Window | small (max heap) | large (min heap) | Median |
| --- | --- | --- | --- |
| [1,3,-1] | [1,-1] | [3] | 1 |
| [3,-1,-3] | [-1,-3] | [3] | -1 |
| [-1,-3,5] | [-1,-3] | [5] | -1 |
| [-3,5,3] | [3,-3] | [5] | 3 |
| [5,3,6] | [5,3] | [6] | 5 |
| [3,6,7] | [6,3] | [7] | 6 |

Final result:

```
[1,-1,-1,3,5,6]
```

### Step-by-step transition

#### Initial window `[1,3,-1]`

Insert `1`:

```
small = [1]
large = []
```

Insert `3`:

```
small = [1]
large = [3]
```

Insert `-1`:

```
small = [1,-1]
large = [3]
```

Median:

```
1
```

#### Slide to `[3,-1,-3]`

Outgoing element:

```
1
```

Incoming element:

```
-3
```

After balancing:

```
small = [-1,-3]
large = [3]
```

Median:

```
-1
```

The same process repeats for all remaining windows.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n log k) | Each insertion and removal costs O(log k) |
| Space | O(k) | Heaps and delayed deletion map store at most k elements |

Each element is inserted exactly once and removed exactly once.

Heap operations cost `O(log k)`, and lazy deletion ensures we never perform expensive arbitrary removals.

The delayed deletion map only stores elements currently waiting to be pruned, so its size also remains bounded by `O(k)`.

## Edge Cases

### Window Size Equals One

When `k = 1`, every window contains exactly one element. The median is simply that element itself.

This case can expose bugs in balancing logic because one heap will always remain empty. The implementation handles this naturally because the max heap is allowed to contain one more element than the min heap.

### Duplicate Values

Duplicate numbers are particularly tricky because heaps do not distinguish between identical values.

For example:

```
nums = [5,5,5,5]
```

If removal is implemented incorrectly, deleting one `5` could accidentally invalidate multiple occurrences.

The lazy deletion map solves this by storing counts of pending removals. Each duplicate is tracked independently.

### Even Window Sizes

When `k` is even, the median is the average of the two middle elements.

This creates two common sources of bugs:

- Integer division instead of floating point division
- Incorrect heap balancing

The implementation explicitly divides by `2.0`, guaranteeing a floating point result.

The balancing invariant ensures both middle elements are always available at the heap tops.

### Large Input Sizes

With `100000` elements, inefficient operations become catastrophic.

Any solution that repeatedly sorts windows or removes arbitrary heap elements directly will time out.

The implementation avoids this by:

- Using heaps for logarithmic insertion
- Using lazy deletion instead of direct heap removal
- Maintaining balance incrementally rather than rebuilding structures

This keeps the runtime efficient even at maximum constraints.
