## Solution

## Problem Understanding

This problem is an interactive grid traversal problem where we control a robot using a very limited API. Unlike normal grid traversal problems, we do not receive the grid as input during execution. Instead, the robot itself is our only way to explore the environment.

The room is conceptually an `m x n` grid. Each cell is either:

- `0`, representing a wall or blocked space
- `1`, representing an empty cell the robot can occupy and clean

The robot starts somewhere in the room on a valid empty cell, but we are not told its coordinates. We are also not allowed to inspect the grid directly. The only operations available are:

- `move()`, attempts to move forward one cell
- `turnLeft()`, rotates the robot 90 degrees counterclockwise
- `turnRight()`, rotates the robot 90 degrees clockwise
- `clean()`, cleans the current cell

The challenge is that movement is relative to the robot's current orientation, not absolute coordinates. Since we cannot see the map, we must build our own internal coordinate system while exploring.

The problem guarantees that all reachable empty cells are connected to the starting position. This is extremely important because it means a full traversal from the starting point can eventually reach every cleanable cell.

The grid dimensions can be as large as `100 x 200`, so there may be up to 20,000 cells. A naive strategy that repeatedly revisits cells or blindly searches without remembering visited positions could become extremely inefficient or even loop forever.

Several edge cases are important:

- A room with only one open cell
- Narrow corridors where turning and backtracking must be handled carefully
- Large open spaces with many traversal paths
- Dead ends where the robot must correctly return to its previous location
- Cyclic paths where revisiting already explored cells could cause infinite recursion if not tracked properly

The most difficult part of this problem is not traversal itself, but maintaining consistent orientation and position while backtracking through an unknown environment.

## Approaches

A brute-force approach would repeatedly attempt movement in all directions from every reachable location without remembering which cells were already visited. The robot could endlessly revisit the same cells through different paths, especially in rooms containing cycles. Since the robot has no global map, such an approach would repeatedly re-explore already cleaned regions and could easily devolve into exponential behavior.

Although this brute-force exploration would eventually clean all reachable cells in finite environments, it would perform enormous amounts of redundant work. More importantly, without proper visitation tracking, recursive exploration could become infinite.

The key insight is that even though the real grid coordinates are unknown, we can create a virtual coordinate system relative to the starting position. We may treat the starting cell as `(0, 0)` and track movement directions ourselves.

Once we assign coordinates relative to the start, the problem becomes a standard graph traversal problem. Each reachable cell is a node, and edges exist between adjacent reachable cells. A depth-first search traversal naturally fits because DFS allows us to:

- Explore as far as possible into a branch
- Clean cells immediately upon discovery
- Backtrack to the previous cell after exploration
- Maintain consistent robot orientation throughout traversal

The most important implementation detail is backtracking. After recursively exploring a neighboring cell, the robot must physically return to its previous position and restore its original orientation. Without this guarantee, subsequent exploration directions become incorrect.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential | O(N) | Repeatedly revisits cells and may loop infinitely |
| Optimal DFS Backtracking | O(N) | O(N) | Visits each reachable cell once while maintaining robot state |

Here, `N` is the number of reachable empty cells.

## Algorithm Walkthrough

1. Create a virtual coordinate system.

Since we do not know the robot's actual position in the grid, we pretend the starting location is `(0, 0)`. Every movement updates relative coordinates accordingly.

We define four directions:

- Up `(-1, 0)`
- Right `(0, 1)`
- Down `(1, 0)`
- Left `(0, -1)`

We also track the robot's current facing direction as an index from `0` to `3`.
2. Maintain a visited set.

We use a hash set to store coordinates that have already been cleaned and explored.

This prevents:

- Infinite loops
- Redundant exploration
- Excessive recursive calls

Since coordinate lookup in a hash set is average `O(1)`, this is highly efficient.
3. Perform DFS from the current cell.

When entering a cell:

- Mark it visited
- Call `clean()`

Then attempt exploration in all four directions.
4. Rotate through all directions systematically.

For each of the four directions:

- Compute the candidate neighbor coordinate
- If it has not been visited, attempt `move()`
- If movement succeeds:

- Recursively explore the neighbor
- Backtrack afterward

Whether movement succeeds or fails, rotate right once to test the next direction.
5. Backtrack after recursive exploration.

This is the critical step.

After DFS finishes exploring a neighboring cell, the robot must return to its previous location and orientation.

We accomplish this with:

- Turn 180 degrees
- Move forward once
- Turn 180 degrees again

This guarantees:

- The robot returns to the parent cell
- The original orientation is restored
6. Continue until all reachable cells are explored.

Because every reachable cell is eventually discovered exactly once, the entire room gets cleaned.

### Why it works

The algorithm performs a complete DFS traversal over the implicit graph formed by reachable cells. Every reachable empty cell is eventually visited because DFS recursively explores all accessible neighbors. The visited set prevents repeated exploration. The backtracking operation guarantees that after exploring a branch, the robot returns to the exact prior state, meaning future exploration decisions remain correct. Therefore, every reachable cell is cleaned exactly once, and the traversal terminates successfully.

## Python Solution

```
# """
# This is the robot's control interface.
# You should not implement it, or speculate about its implementation
# """
# class Robot:
#     def move(self):
#         pass
#
#     def turnLeft(self):
#         pass
#
#     def turnRight(self):
#         pass
#
#     def clean(self):
#         pass

from typing import Set, Tuple

class Solution:
    def cleanRoom(self, robot) -> None:
        """
        :type robot: Robot
        :rtype: None
        """

        directions = [
            (-1, 0),  # up
            (0, 1),   # right
            (1, 0),   # down
            (0, -1)   # left
        ]

        visited: Set[Tuple[int, int]] = set()

        def go_back() -> None:
            robot.turnRight()
            robot.turnRight()

            robot.move()

            robot.turnRight()
            robot.turnRight()

        def dfs(row: int, col: int, direction: int) -> None:
            visited.add((row, col))
            robot.clean()

            for i in range(4):
                new_direction = (direction + i) % 4

                dr, dc = directions[new_direction]
                next_row = row + dr
                next_col = col + dc

                if (next_row, next_col) not in visited:
                    if robot.move():
                        dfs(next_row, next_col, new_direction)
                        go_back()

                robot.turnRight()

        dfs(0, 0, 0)
```

The solution begins by defining a relative coordinate system and direction vectors. Since the robot initially faces upward, direction `0` corresponds to up.

The `visited` set tracks every explored coordinate. This prevents repeated exploration and guarantees termination.

The `go_back()` helper is essential. Recursive DFS changes the robot's physical position, so after recursion completes, we must physically move the robot back to its previous cell. The double rotation plus movement restores both position and orientation.

The `dfs()` function performs recursive exploration. Upon entering a cell, it marks the coordinate as visited and cleans the cell.

The loop iterates through all four directions relative to the current orientation. Before attempting movement, the algorithm computes the next coordinate in the virtual map.

If the target coordinate has not been visited and `move()` succeeds, DFS recursively explores that neighboring cell.

After recursive exploration completes, `go_back()` restores the robot to the original state before continuing with other directions.

Finally, `turnRight()` rotates the robot so that each loop iteration aligns the robot with the next direction.

## Go Solution

```
/**
 * // This is the robot's control interface.
 * // You should not implement it, or speculate about its implementation
 * type Robot struct {
 * }
 *
 * // Returns true if the cell in front is open and robot moves into the cell.
 * func (robot *Robot) Move() bool {}
 *
 * // Robot will stay on the same cell after calling TurnLeft/TurnRight.
 * // Each turn will be 90 degrees.
 * func (robot *Robot) TurnLeft() {}
 * func (robot *Robot) TurnRight() {}
 *
 * // Clean the current cell.
 * func (robot *Robot) Clean() {}
 */

func cleanRoom(robot *Robot) {
	directions := [][]int{
		{-1, 0}, // up
		{0, 1},  // right
		{1, 0},  // down
		{0, -1}, // left
	}

	visited := make(map[[2]int]bool)

	var goBack func()
	goBack = func() {
		robot.TurnRight()
		robot.TurnRight()

		robot.Move()

		robot.TurnRight()
		robot.TurnRight()
	}

	var dfs func(row, col, direction int)

	dfs = func(row, col, direction int) {
		visited[[2]int{row, col}] = true
		robot.Clean()

		for i := 0; i < 4; i++ {
			newDirection := (direction + i) % 4

			dr := directions[newDirection][0]
			dc := directions[newDirection][1]

			nextRow := row + dr
			nextCol := col + dc

			if !visited[[2]int{nextRow, nextCol}] {
				if robot.Move() {
					dfs(nextRow, nextCol, newDirection)
					goBack()
				}
			}

			robot.TurnRight()
		}
	}

	dfs(0, 0, 0)
}
```

The Go implementation closely mirrors the Python solution. Since Go does not have tuples or hashable lists, coordinates are stored as fixed-size arrays `[2]int`, which can be used as map keys.

Recursive helper functions are declared using function variables before assignment because Go requires explicit recursive closure definitions.

Unlike Python, Go does not require explicit type hints because the language is statically typed.

No integer overflow concerns exist because coordinates remain bounded by room dimensions.

| Step | Robot Position | Facing | Action | Visited |
| --- | --- | --- | --- | --- |
| 2 | (-1,0) | Up | Move and recurse | {(0,0),(-1,0)} |

| Step | Robot Position | Facing | Action |
| --- | --- | --- | --- |
| 3 | (-1,0) | Right | move() returns false |

Eventually, DFS reaches a dead end. At that point:

| Step | Robot Position | Action |
| --- | --- | --- |
| 4 | current cell | Finish recursive exploration |
| 5 | parent cell | go_back() restores previous state |

This process repeats until every reachable cell is visited and cleaned.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(N) | Each reachable cell is visited once |
| Space | O(N) | Visited set and recursion stack |

Here, `N` is the number of reachable empty cells.

Although the robot may physically traverse some paths multiple times during backtracking, each edge is traversed a constant number of times. Therefore, the total movement cost remains linear in the number of reachable cells.

The recursion stack may reach depth `O(N)` in worst-case corridor-shaped rooms.

## Edge Cases

A single-cell room is the smallest possible input. The robot cannot move in any direction, so all four movement attempts fail immediately. A buggy implementation might incorrectly recurse or fail to clean the starting cell. This implementation cleans first before exploring, guaranteeing correctness.

Long corridor-shaped rooms create extremely deep recursion paths. Without proper backtracking, the robot could lose orientation or fail to return to prior cells. The `go_back()` helper guarantees that after exploring deeper cells, the robot always returns to the previous location with the same orientation.

Rooms containing cycles are especially dangerous. For example, a square loop allows multiple paths to reach the same cell. Without a visited set, DFS would repeatedly revisit cells forever. The implementation stores every explored coordinate in `visited`, ensuring each cell is processed exactly once.

Dead-end branches are another common source of bugs. After exploring a dead end, the robot must return to the parent cell and continue exploring remaining directions. If orientation restoration is incorrect, future direction calculations become invalid. The backtracking logic restores both position and orientation, preventing this issue completely.
