## Solution

## Problem Understanding

The problem gives us a decimal integer `n` as a string, and asks us to find the smallest base `k >= 2` such that the representation of `n` in base `k` consists entirely of the digit `1`.

For example, if `n = 13` and `k = 3`, then:

```
13 in base 3 = 111
```

because:

```
1 × 3² + 1 × 3¹ + 1 × 3⁰ = 9 + 3 + 1 = 13
```

So base `3` is a good base for `13`.

More generally, if a number has `m + 1` digits and all digits are `1` in base `k`, then:

```
n = 1 + k + k² + ... + k^m
```

This is a geometric series. Using the geometric sum formula:

$n = 1 + k + k^2 + \cdots + k^m = \frac{k^{m+1}-1}{k-1}$

The task is therefore equivalent to finding integers `k >= 2` and `m >= 1` such that the equation above holds, and among all such valid bases, returning the smallest `k`.

The constraints are extremely important. The value of `n` can be as large as `10^18`, which immediately rules out brute force simulation over all possible bases. Any algorithm with linear or even square root complexity in `n` would be far too slow.

Another important observation is that every number `n` always has at least one valid good base:

```
n = 11 in base (n - 1)
```

because:

```
1 × (n - 1) + 1 = n
```

So the answer always exists.

The tricky cases come from very large values of `n`, potential integer overflow during exponentiation, and cases where the optimal base corresponds to many digits versus only two digits. Efficiently searching for both the digit count and the base is the core challenge.

## Approaches

### Brute Force Approach

A direct approach would try every possible base `k` from `2` up to `n - 1`. For each base, we could repeatedly divide `n` by `k` and verify whether every digit in that base representation is `1`.

This works because it explicitly checks the definition of a good base. If any base produces a representation containing only `1`s, we record it.

However, this approach is computationally infeasible. In the worst case, we would test almost `10^18` bases. Even if each conversion were logarithmic, the total runtime would still be astronomically large.

The brute force method is therefore correct but unusable for the given constraints.

### Optimal Approach

The key insight comes from the geometric series structure:

$n = 1 + k + k^2 + \cdots + k^m$

If the representation has `m + 1` digits, then:

```
n > k^m
```

which implies:

$k < n^{1/m}$

This gives us two crucial optimizations.

First, the maximum possible number of digits is limited. The smallest base is `2`, so the maximum number of digits satisfies:

$n = 1 + 2 + 2^2 + \cdots + 2^m = 2^{m+1}-1$

Therefore:

$m \leq \lfloor \log_2 n \rfloor$

For `n <= 10^18`, this is at most about `60`.

Second, for a fixed digit count `m + 1`, the base `k` is bounded and can be searched efficiently.

We iterate over possible lengths from largest to smallest. Larger digit counts correspond to smaller bases, and since we want the smallest base, we should search longer representations first.

For each `m`, we estimate:

```
k ≈ n^(1/m)
```

and test whether that base satisfies the geometric series equation.

This reduces the search space dramatically and makes the problem solvable in logarithmic time.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n log n) | O(1) | Tries every base and converts the number |
| Optimal | O((log n)²) | O(1) | Uses geometric series properties and bounded search |

## Algorithm Walkthrough

1. Convert the input string `n` into an integer.

We need numeric operations such as exponentiation and division, so parsing the string once at the beginning simplifies the implementation.
2. Compute the maximum possible exponent `m`.

Since the smallest base is `2`, the longest all-ones representation occurs in base `2`.

We therefore compute:

```
max_m = floor(log2(n))
```

This bounds the number of digits we need to consider.
3. Iterate over possible values of `m` from largest to smallest.

Here, `m` means the highest exponent in the geometric series:

```
n = 1 + k + k² + ... + k^m
```

The total number of digits is `m + 1`.

We iterate from large `m` downward because larger digit counts produce smaller bases. The first valid base we find is therefore the answer.
4. Estimate the candidate base.

For a fixed `m`, the dominant term is `k^m`, so:

```
k ≈ n^(1/m)
```

We compute the integer approximation of this root.
5. Compute the geometric series sum using the candidate base.

Instead of using the closed-form formula directly, we build the sum iteratively:

```
1 + k + k² + ... + k^m
```

This avoids floating point precision problems and keeps the logic straightforward.
6. Compare the computed sum with `n`.

If they are equal, then every digit in base `k` is `1`, so we return `k`.
7. If no longer representation works, return `n - 1`.

Every number satisfies:

```
n = 11 in base (n - 1)
```

so this fallback is always valid.

### Why it works

The algorithm checks every possible digit length that an all-ones representation can have. For each length, the geometric series uniquely determines the base. By searching lengths from largest to smallest, we guarantee that the first valid base found is the smallest possible base. Since all valid representations must satisfy the geometric series equation, no solution can be missed.

## Python Solution

```
class Solution:
    def smallestGoodBase(self, n: str) -> str:
        number = int(n)

        # Maximum possible m where representation has (m + 1) digits
        max_m = number.bit_length() - 1

        for m in range(max_m, 1, -1):
            # Candidate base
            k = int(number ** (1 / m))

            if k < 2:
                continue

            current = 1
            total = 1

            for _ in range(m):
                current *= k
                total += current

            if total == number:
                return str(k)

        return str(number - 1)
```

The implementation begins by converting the string input into an integer called `number`.

The expression `number.bit_length() - 1` computes `floor(log2(number))`, which gives the maximum possible exponent `m`.

The outer loop iterates downward through all candidate lengths. For each `m`, we estimate the base using the `m`-th root of `number`.

We then rebuild the geometric series iteratively. The variable `current` stores the current power of `k`, and `total` accumulates the sum.

If the computed series equals `number`, we have found a valid good base and immediately return it. Because we iterate from larger lengths to smaller lengths, this is guaranteed to be the smallest valid base.

If no larger representation exists, the final answer is `number - 1`.

## Go Solution

```
package main

import (
	"math"
	"strconv"
)

func smallestGoodBase(n string) string {
	number, _ := strconv.ParseInt(n, 10, 64)

	maxM := bitsLen(number) - 1

	for m := maxM; m >= 2; m-- {
		k := int64(math.Pow(float64(number), 1.0/float64(m)))

		if k < 2 {
			continue
		}

		var current int64 = 1
		var total int64 = 1

		for i := 0; i < m; i++ {
			current *= k
			total += current
		}

		if total == number {
			return strconv.FormatInt(k, 10)
		}
	}

	return strconv.FormatInt(number-1, 10)
}

func bitsLen(x int64) int {
	length := 0

	for x > 0 {
		length++
		x >>= 1
	}

	return length
}
```

The Go solution follows the same algorithmic structure as the Python version. Since Go does not provide a direct equivalent of Python's `bit_length()`, we implement a helper function called `bitsLen`.

Go also requires explicit integer types. We use `int64` throughout because the input can be as large as `10^18`, which exceeds the range of 32-bit integers.

The iterative geometric sum computation avoids overflow issues that could arise from using direct exponentiation formulas.

## Edge Cases

One important edge case occurs when the smallest good base is `2`. Numbers of the form:

```
2^(m+1) - 1
```

have binary representations consisting entirely of `1`s. For example, `31` becomes `11111` in base `2`. These cases test whether the algorithm correctly prioritizes longer representations before shorter ones. Since we iterate from larger `m` downward, the implementation correctly finds base `2` first.

Another important edge case is when no representation longer than two digits exists. In such situations, the answer must be `n - 1`, because every number can always be represented as `11` in base `n - 1`. Large prime-like numbers often fall into this category. The implementation handles this by returning `number - 1` if no earlier match is found.

A third edge case involves extremely large inputs close to `10^18`. Direct exponentiation formulas can overflow or suffer from floating point inaccuracies. The implementation avoids this by computing the geometric sum iteratively using integer arithmetic. This keeps calculations safe and precise even near the upper constraint boundary.
