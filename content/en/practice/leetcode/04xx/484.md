## Solution

## Problem Understanding

The problem gives us a string `s` consisting only of the characters `'I'` and `'D'`. The length of the string is `n - 1`, which means the final permutation must contain `n` integers.

Each character describes the relationship between two adjacent numbers in the permutation:

- `'I'` means increasing, so `perm[i] < perm[i + 1]`
- `'D'` means decreasing, so `perm[i] > perm[i + 1]`

Our goal is to construct a valid permutation using every integer from `1` to `n` exactly once, while also ensuring that the permutation is the lexicographically smallest among all valid answers.

Lexicographical order works similarly to dictionary order. When comparing two permutations, we compare elements from left to right. The first smaller value determines the smaller permutation. For example:

- `[2,1,3]` is lexicographically smaller than `[3,1,2]` because `2 < 3`
- `[1,3,2]` is lexicographically smaller than `[1,4,2]` because the first differing element is `3 < 4`

The input size can be as large as `10^5`, which immediately tells us that exponential or factorial approaches are impossible. Any algorithm that attempts to generate all permutations will time out. We need a solution close to linear time.

Several edge cases are important:

- A string containing only `'I'`, such as `"III"`, should produce the naturally increasing sequence `[1,2,3,4]`
- A string containing only `'D'`, such as `"DDD"`, should produce the smallest valid decreasing arrangement `[4,3,2,1]`
- Alternating patterns like `"DIDID"` require careful handling because local decisions affect future lexicographical ordering
- Long consecutive runs of `'D'` are especially important because they force reversed segments in the output

The problem guarantees that every character is either `'I'` or `'D'`, so we do not need to validate the input format.

## Approaches

### Brute Force Approach

A brute force solution would generate every permutation of the integers from `1` to `n`, then check whether the permutation satisfies the pattern described by `s`.

For each generated permutation:

- Compare adjacent elements
- Verify that every `'I'` position is increasing
- Verify that every `'D'` position is decreasing
- Keep the lexicographically smallest valid permutation

This approach is correct because it exhaustively examines every possible permutation and therefore cannot miss the optimal answer.

However, the number of permutations is `n!`, which grows extremely quickly. Even for relatively small values of `n`, this becomes computationally impossible. Since `n` can be as large as `100001`, brute force is completely infeasible.

### Optimal Greedy Observation

The key insight is that the lexicographically smallest permutation should use the smallest available numbers as early as possible.

If we start with the sequence:

```
1 2 3 4 5 ...
```

then every `'I'` condition is already satisfied naturally.

The only difficulty comes from consecutive `'D'` segments. For example:

```
s = "DDI"
```

requires:

```
a > b > c < d
```

To satisfy the decreasing relationships while remaining lexicographically smallest, we should reverse only the minimal necessary segment.

Suppose we begin with:

```
1 2 3 4
```

The first two positions must decrease, so we reverse the first three numbers:

```
3 2 1 4
```

This is the smallest possible valid arrangement.

The optimal strategy therefore becomes:

- Start with numbers `1` through `n`
- Whenever we encounter a consecutive block of `'D'`, reverse the corresponding subarray
- Leave increasing sections untouched

This greedy approach works because reversing the smallest possible segment preserves lexicographical minimality.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n! × n) | O(n) | Generates all permutations and validates each one |
| Optimal | O(n) | O(1) auxiliary, excluding output | Reverse contiguous decreasing segments |

## Algorithm Walkthrough

1. Create the initial permutation containing numbers from `1` to `n`.

If the string length is `m`, then the permutation size must be `m + 1`. We initialize:

```
[1,2,3,...,m+1]
```

This arrangement is already lexicographically smallest before applying constraints.
2. Scan the string from left to right.

We use an index `i` to examine each character in `s`.
3. When we encounter an `'I'`, continue scanning.

Increasing relationships already work naturally in the current arrangement, so no modification is needed.
4. When we encounter a `'D'`, identify the entire consecutive block of `'D'`.

For example:

```
s = "IIDDDI"
```

The consecutive `'D'` block spans indices `2` through `4`.

We continue moving forward until the block ends.
5. Reverse the corresponding segment in the permutation.

If the `'D'` block spans indices `[start, end]`, then we reverse:

```
perm[start : end + 2]
```

The extra `+1` is necessary because relationships between adjacent elements involve one more number than characters.

Example:

```
s = "DD"
perm = [1,2,3]
```

Reverse indices `[0:3]`:

```
[3,2,1]
```
6. Continue scanning after the reversed block.

Once the decreasing sequence has been fixed, we resume processing the remaining characters.
7. Return the final permutation.

### Why it works

The algorithm maintains the invariant that the prefix processed so far is the lexicographically smallest valid arrangement.

Consecutive `'D'` characters require a strictly decreasing sequence. The smallest way to create such a sequence is to reverse the smallest contiguous increasing range that covers the entire `'D'` block.

Any smaller modification would fail to satisfy the decreasing constraints, and any larger modification would unnecessarily increase earlier values, producing a lexicographically larger permutation.

Because each `'D'` block is handled independently and minimally, the final result is globally optimal.

## Python Solution

```
from typing import List

class Solution:
    def findPermutation(self, s: str) -> List[int]:
        n = len(s) + 1
        
        permutation = list(range(1, n + 1))
        
        index = 0
        
        while index < len(s):
            if s[index] == 'I':
                index += 1
                continue
            
            start = index
            
            while index < len(s) and s[index] == 'D':
                index += 1
            
            permutation[start:index + 1] = reversed(
                permutation[start:index + 1]
            )
        
        return permutation
```

The implementation begins by constructing the default increasing permutation from `1` to `n`. This gives the lexicographically smallest starting point.

The main loop scans through the pattern string. Whenever an `'I'` is encountered, no work is needed because the current ordering already satisfies increasing relationships.

When a `'D'` is found, the algorithm records the start position of the decreasing block. It then advances until the entire consecutive sequence of `'D'` characters has been identified.

Once the full block is known, the corresponding segment in the permutation is reversed. This transforms an increasing range into the minimal valid decreasing range.

The reversal operation is efficient because every index belongs to at most one reversed segment, giving overall linear time complexity.

## Go Solution

```
func findPermutation(s string) []int {
	n := len(s) + 1

	permutation := make([]int, n)

	for i := 0; i < n; i++ {
		permutation[i] = i + 1
	}

	index := 0

	for index < len(s) {
		if s[index] == 'I' {
			index++
			continue
		}

		start := index

		for index < len(s) && s[index] == 'D' {
			index++
		}

		left := start
		right := index

		for left < right {
			permutation[left], permutation[right] =
				permutation[right], permutation[left]

			left++
			right--
		}
	}

	return permutation
}
```

The Go implementation follows the same logic as the Python version, but slice reversal must be implemented manually because Go does not provide a built in reversed iterator.

Instead of slicing assignment, two pointers are used:

- `left` starts at the beginning of the segment
- `right` starts at the end
- Elements are swapped until the pointers meet

Go slices are reference backed, so the reversal modifies the permutation in place without allocating additional arrays.

There are no integer overflow concerns because the maximum value stored is at most `100001`.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n) | Each index is visited at most twice, once during scanning and once during reversal |
| Space | O(1) auxiliary, excluding output | Reversal is done in place |

The algorithm is linear because every character in the string is processed a constant number of times. Even though reversal operations occur, each element participates in at most one reversal, so the total work across all reversals remains linear.

The auxiliary space complexity is constant because no additional data structures proportional to input size are used. The output array itself is required by the problem and is not counted as extra space.

## Edge Cases

A string consisting entirely of `'I'` characters is an important edge case because the initial increasing permutation already satisfies all constraints. A buggy implementation might unnecessarily modify the array or perform incorrect reversals. This implementation handles the case naturally because it only performs reversals when encountering `'D'`.

A string consisting entirely of `'D'` characters is another critical case. For example:

```
s = "DDD"
```

requires the entire permutation to become decreasing. A common mistake is reversing too small a range or failing to include the extra element required by the relationship count. The implementation correctly reverses indices `0` through `n - 1`, producing the smallest valid decreasing arrangement.

Consecutive `'D'` blocks in the middle of the string can also cause subtle bugs. Consider:

```
s = "IIDDDII"
```

The algorithm must reverse exactly the segment corresponding to the decreasing block and no more. Reversing too large a segment would produce a lexicographically larger answer, while reversing too small a segment would violate the constraints. By detecting maximal contiguous `'D'` sequences and reversing only those ranges, the implementation handles these cases correctly.
