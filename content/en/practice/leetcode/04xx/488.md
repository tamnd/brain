## Solution

## Problem Understanding

The problem models a recursive elimination game. We are given a string `board` representing the current sequence of colored balls on the table, and another string `hand` representing the balls we are allowed to insert. Each character is one of five possible colors: `R`, `Y`, `B`, `G`, or `W`.

At every move, we may choose one ball from the hand and insert it at any position in the board, including both ends. After insertion, if there exists a consecutive group of at least three balls of the same color, that entire group disappears. This removal may create new groups that also satisfy the removal condition, causing a chain reaction. The board keeps collapsing until no removable groups remain.

The goal is to remove every ball from the board while using the minimum possible number of insertions. If it is impossible to clear the board using the available balls in the hand, we return `-1`.

The constraints are very important here:

- `board.length <= 16`
- `hand.length <= 5`

These limits are small enough that exponential search is possible, but only with aggressive pruning and memoization. A naive brute-force exploration of all insertion orders quickly explodes because every move can branch into many possible board states.

Another important observation is that the board never initially contains a removable group of size three or larger. This means every elimination sequence must be triggered by insertions.

Several edge cases can easily break naive implementations:

- Chain reactions after removals must continue until the board stabilizes.
- Different insertion orders can lead to the same board state, so recomputing them repeatedly is wasteful.
- Sometimes inserting into a group of two is optimal because it immediately clears the group.
- Sometimes inserting into a group of one is necessary to create future chain reactions.
- A board may become empty during recursive elimination, which must terminate correctly.

The challenge is not merely simulating the removals, but searching efficiently through the enormous decision space.

## Approaches

### Brute Force Approach

The most direct solution is to try every possible insertion at every possible position using every available ball in the hand.

For each recursive state:

1. Choose a ball from the hand.
2. Try inserting it at every index in the board.
3. Collapse the board by repeatedly removing groups of three or more.
4. Recurse on the resulting board and remaining hand.
5. Track the minimum number of insertions needed to reach an empty board.

This approach is correct because it explores every legal move sequence. If a valid solution exists, eventually one recursive branch will discover it.

However, the search space becomes enormous. If the board length is `n` and the hand size is `m`, then each step may consider approximately:

- `m` possible balls
- `n + 1` insertion positions

The total branching factor becomes very large, and many states repeat due to different insertion orders producing identical configurations.

Without memoization or pruning, this approach is far too slow.

### Key Insight

The crucial observation is that the game state is fully determined by:

- The current board configuration
- The remaining balls in hand

If we encounter the same `(board, hand)` state multiple times, the answer from that state onward will always be identical.

This naturally suggests memoized depth-first search.

A second key optimization comes from recognizing that most insertions are useless. Inserting a ball randomly into unrelated positions usually does not help. The most valuable insertions are those that help complete groups of three.

Instead of blindly trying all insertions, we can focus on removing existing groups. Suppose a consecutive group has size:

- `1`, then we need `2` more balls to eliminate it
- `2`, then we need `1` more ball to eliminate it

If we have enough matching balls in hand, we can remove that group immediately and recurse on the reduced board.

This dramatically reduces the search space.

### Approach Comparison

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | Exponential, roughly O((n * m)^m) | O(m) recursion depth | Explores every insertion position and ball choice |
| Optimal | Exponential but heavily pruned, roughly O(states) | O(states) memoization | DFS with memoization and targeted eliminations |

## Algorithm Walkthrough

### Step 1: Count Available Balls

We first count how many balls of each color exist in the hand.

A frequency map is ideal because we repeatedly need to check:

- Whether a color is available
- How many copies remain
- When to restore counts during backtracking

For example:

```
Counter(hand)
```

might produce:

```
{'R': 2, 'B': 1, 'W': 1}
```

### Step 2: Create a Board Reduction Function

Whenever balls are inserted or groups are removed, chain reactions may occur.

We therefore create a helper function that repeatedly removes consecutive groups of length at least three until the board stabilizes.

For example:

```
WWBBBWW
```

becomes:

```
WWWW
```

which then becomes:

```
empty
```

This reduction step is critical because every recursive state should operate on a fully stabilized board.

### Step 3: Define the Recursive DFS State

The recursive function will represent:

```
dfs(current_board)
```

The hand counts are maintained externally and mutated during recursion.

The DFS returns:

- Minimum insertions needed to clear the board
- Infinity if impossible

### Step 4: Scan Consecutive Groups

We iterate through the board and identify maximal consecutive groups.

For example:

```
RRBBBW
```

contains:

- `RR`
- `BBB`
- `W`

For each group:

- Let `group_size` be its length
- Let `needed = 3 - group_size`

If `needed <= 0`, the group already disappears during reduction, so stabilized boards never contain such groups.

Possible cases:

- Group size `1`, need `2`
- Group size `2`, need `1`

### Step 5: Try Eliminating the Group

If we have enough balls in hand for that color:

```
hand[color] >= needed
```

then:

1. Spend those balls
2. Remove the group from the board
3. Reduce the resulting board
4. Recurse
5. Restore the hand count after recursion

This is classic backtracking.

### Step 6: Track the Minimum Answer

For every valid recursive branch:

```
answer = min(answer, needed + recursive_result)
```

The final answer is the smallest insertion count among all possibilities.

### Step 7: Memoize States

Many recursive paths produce identical board states.

We memoize using:

```
(board_string, hand_counts)
```

This avoids recomputing expensive subproblems.

A common technique is converting the hand counter into a sorted tuple so it becomes hashable.

### Why it Works

The algorithm works because every recursive step considers all meaningful ways to eliminate a group from the board. Any successful solution must eventually remove every group, and removing a group requires inserting enough matching balls to reach three consecutive balls.

Memoization guarantees that each unique game state is solved only once. Since recursion explores every feasible elimination strategy while always taking the minimum insertion count, the returned answer is optimal.

## Python Solution

```
from collections import Counter
from functools import lru_cache
from math import inf

class Solution:
    def findMinStep(self, board: str, hand: str) -> int:
        hand_count = Counter(hand)

        def reduce_board(s: str) -> str:
            changed = True

            while changed:
                changed = False
                i = 0
                new_board = ""

                while i < len(s):
                    j = i

                    while j < len(s) and s[j] == s[i]:
                        j += 1

                    if j - i >= 3:
                        changed = True
                    else:
                        new_board += s[i:j]

                    i = j

                s = new_board

            return s

        @lru_cache(None)
        def dfs(current_board: str, hand_state: tuple) -> int:
            current_board = reduce_board(current_board)

            if not current_board:
                return 0

            counts = dict(hand_state)

            answer = inf
            i = 0

            while i < len(current_board):
                j = i

                while j < len(current_board) and current_board[j] == current_board[i]:
                    j += 1

                color = current_board[i]
                group_size = j - i
                needed = 3 - group_size

                if counts.get(color, 0) >= needed:
                    counts[color] -= needed

                    next_board = current_board[:i] + current_board[j:]

                    next_state = tuple(sorted(
                        (k, v) for k, v in counts.items() if v > 0
                    ))

                    result = dfs(next_board, next_state)

                    if result != inf:
                        answer = min(answer, needed + result)

                    counts[color] += needed

                i = j

            return answer

        initial_state = tuple(sorted(hand_count.items()))

        result = dfs(board, initial_state)

        return result if result != inf else -1
```

The implementation begins by converting the hand into a frequency map. This allows constant-time access to remaining ball counts during recursion.

The `reduce_board` helper repeatedly scans the board and removes consecutive groups of length at least three. The loop continues until no further removals occur, correctly handling chain reactions.

The DFS function is memoized using `lru_cache`. Each state is identified by:

- The current board string
- The remaining hand counts encoded as a tuple

Inside DFS, the board is first stabilized using `reduce_board`. If the board becomes empty, no additional insertions are needed, so the function returns `0`.

The algorithm then scans each consecutive group. For every group, it computes how many additional balls are required to eliminate it. If enough balls are available in the hand, it temporarily spends them, removes the group, and recursively solves the smaller problem.

After recursion completes, the hand count is restored. This backtracking step is essential because multiple recursive branches reuse the same state object.

Finally, if no solution is found, the algorithm returns `-1`.

## Go Solution

```
package main

import (
	"math"
	"sort"
	"strconv"
)

func findMinStep(board string, hand string) int {
	handCount := map[byte]int{}

	for i := 0; i < len(hand); i++ {
		handCount[hand[i]]++
	}

	var reduceBoard func(string) string

	reduceBoard = func(s string) string {
		changed := true

		for changed {
			changed = false
			newBoard := ""

			for i := 0; i < len(s); {
				j := i

				for j < len(s) && s[j] == s[i] {
					j++
				}

				if j-i >= 3 {
					changed = true
				} else {
					newBoard += s[i:j]
				}

				i = j
			}

			s = newBoard
		}

		return s
	}

	serialize := func(counts map[byte]int) string {
		keys := make([]byte, 0, len(counts))

		for k := range counts {
			keys = append(keys, k)
		}

		sort.Slice(keys, func(i, j int) bool {
			return keys[i] < keys[j]
		})

		result := ""

		for _, k := range keys {
			if counts[k] > 0 {
				result += string(k) + strconv.Itoa(counts[k])
			}
		}

		return result
	}

	memo := map[string]int{}

	var dfs func(string, map[byte]int) int

	dfs = func(currentBoard string, counts map[byte]int) int {
		currentBoard = reduceBoard(currentBoard)

		if currentBoard == "" {
			return 0
		}

		key := currentBoard + "|" + serialize(counts)

		if val, exists := memo[key]; exists {
			return val
		}

		answer := math.MaxInt32

		for i := 0; i < len(currentBoard); {
			j := i

			for j < len(currentBoard) && currentBoard[j] == currentBoard[i] {
				j++
			}

			color := currentBoard[i]
			groupSize := j - i
			needed := 3 - groupSize

			if counts[color] >= needed {
				counts[color] -= needed

				nextBoard := currentBoard[:i] + currentBoard[j:]

				result := dfs(nextBoard, counts)

				if result != math.MaxInt32 {
					answer = min(answer, needed+result)
				}

				counts[color] += needed
			}

			i = j
		}

		memo[key] = answer

		return answer
	}

	result := dfs(board, handCount)

	if result == math.MaxInt32 {
		return -1
	}

	return result
}

func min(a int, b int) int {
	if a < b {
		return a
	}

	return b
}
```

The Go implementation follows the same recursive strategy as the Python version, but several language-specific differences are worth noting.

Go does not provide a built-in memoization decorator like Python's `lru_cache`, so memoization is implemented manually using a `map[string]int`.

Because Go maps are mutable reference types, the hand counts are updated in place during recursion and restored afterward during backtracking.

The memoization key is constructed by serializing the board together with the remaining hand counts. Since maps are unordered, the keys must be sorted before serialization to ensure consistent state encoding.

Go also lacks Python's infinity representation, so `math.MaxInt32` is used as a sentinel value for impossible states.

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | Exponential in number of states | Memoized DFS explores unique board-hand states |
| Space | O(states) | Memoization cache plus recursion stack |

The theoretical worst-case complexity remains exponential because the problem itself has combinatorial branching. However, the constraints are small enough that memoization and aggressive pruning make the solution feasible.

The most important optimization is that we only attempt insertions that can immediately eliminate a group. This removes a massive number of useless branches from the search tree.

## Edge Cases

One important edge case occurs when the board becomes empty during recursive reduction. A naive implementation might continue searching unnecessarily or fail to terminate correctly. The solution handles this by immediately returning `0` once the reduced board is empty.

Another tricky case involves chain reactions. Removing one group may create another removable group elsewhere in the board. For example:

```
WWBBBWW
```

first becomes:

```
WWWW
```

which then disappears entirely. A single-pass reduction would fail here. The implementation repeatedly scans until no more removals occur.

A third subtle case is repeated states reached through different move orders. Two different insertion sequences may eventually produce the same board and remaining hand configuration. Without memoization, the algorithm would recompute the same expensive subtree many times. The memoized DFS guarantees that each unique state is solved only once.

Another edge case involves groups of size one. It is tempting to ignore them because they require two insertions, but sometimes removing such groups is necessary for optimal play. The implementation correctly evaluates both size-one and size-two groups by computing:

```
needed = 3 - group_size
```

Finally, impossible states must propagate correctly through recursion. The implementation uses infinity as a sentinel value for unsolvable branches. Only valid recursive results contribute to the minimum answer, preventing impossible paths from contaminating the final result.
