## Solution

## Problem Understanding

The problem asks us to find the "next greater element" for every number in `nums1`, based on the ordering inside `nums2`.

The next greater element of a value `x` is defined as the first value to the right of `x` in the array that is strictly larger than `x`. If no such value exists, the answer is `-1`.

The important detail is that `nums1` is a subset of `nums2`. Every value in `nums1` appears exactly once in `nums2`, because all elements are distinct. This uniqueness guarantee simplifies the problem significantly because we can safely map each number to exactly one next greater value.

For each element in `nums1`, we must:

1. Locate that element inside `nums2`
2. Scan to the right
3. Find the first larger number
4. Return that number, or `-1` if none exists

The constraints are relatively small, with arrays up to length `1000`, so even a quadratic solution would technically pass. However, the follow up specifically asks for an `O(nums1.length + nums2.length)` solution, which suggests that the intended approach uses preprocessing and efficient data structures.

The uniqueness constraint is especially important because it allows us to build a direct mapping:

- value -> next greater value

without worrying about duplicate keys.

Several edge cases are important to consider:

- Elements that have no greater value to their right
- Arrays that are strictly decreasing
- Arrays that are strictly increasing
- `nums1` containing only one element
- `nums1` being equal to `nums2`
- Very small arrays such as length `1`

A naive implementation can easily become inefficient if it repeatedly scans `nums2` for every element in `nums1`.

## Approaches

### Brute Force Approach

The brute force solution directly follows the problem definition.

For every element in `nums1`, we first locate its position in `nums2`. Once found, we continue scanning to the right until we encounter a larger value. If we find one, that value becomes the answer. Otherwise, the answer is `-1`.

This approach is correct because it explicitly checks every candidate to the right in order, guaranteeing that the first greater element is found.

However, the problem is inefficiency. For every element in `nums1`, we may scan most of `nums2`. In the worst case, this leads to quadratic time complexity.

### Optimal Approach, Monotonic Stack

The key insight is that while scanning `nums2`, we can compute the next greater element for every number in a single pass.

Suppose we process numbers from left to right. When we encounter a new number, that number may serve as the next greater element for several earlier numbers.

We use a stack to keep track of numbers whose next greater element has not yet been found.

The stack is maintained in decreasing order. This is why the technique is called a monotonic decreasing stack.

When a new number is greater than the top of the stack:

- we pop the smaller number
- we now know that the current number is its next greater element
- we store this relationship in a hash map

Each number is pushed once and popped once, giving linear time complexity.

After preprocessing `nums2`, we can answer every query from `nums1` in constant time using the hash map.

| Approach | Time Complexity | Space Complexity | Notes |
| --- | --- | --- | --- |
| Brute Force | O(n × m) | O(1) | Repeatedly scans `nums2` for each query |
| Optimal | O(n + m) | O(n) | Uses a monotonic stack and hash map |

Here:

- `n = len(nums1)`
- `m = len(nums2)`

## Algorithm Walkthrough

1. Create an empty stack and an empty hash map.

The stack will store numbers from `nums2` whose next greater element has not yet been determined.

The hash map will store:

`number -> next greater number`
2. Iterate through `nums2` from left to right.

Each new number may become the next greater element for previous smaller numbers.
3. While the stack is not empty and the current number is greater than the top of the stack:

- pop the top element from the stack
- record that the current number is the next greater element for the popped value

This works because the current number is the first larger value encountered to the right.
4. Push the current number onto the stack.

It has not yet found its own next greater element, so it must remain available for future comparisons.
5. After processing all numbers in `nums2`, any remaining elements in the stack do not have a next greater element.

Their answer is implicitly `-1`.
6. Build the final result for `nums1`.

For each number:

- look it up in the hash map
- if absent, return `-1`

### Why it works

The stack always maintains elements in decreasing order from bottom to top. Whenever we encounter a larger number, we immediately resolve the next greater element for all smaller numbers on top of the stack.

Each number waits in the stack until the first greater value to its right appears. Because we process from left to right, the first greater value encountered is guaranteed to be the correct next greater element.

Since every element is pushed and popped at most once, the algorithm is both correct and efficient.

## Python Solution

```
from typing import List

class Solution:
    def nextGreaterElement(self, nums1: List[int], nums2: List[int]) -> List[int]:
        next_greater = {}
        stack = []

        for num in nums2:
            while stack and num > stack[-1]:
                smaller = stack.pop()
                next_greater[smaller] = num

            stack.append(num)

        result = []

        for num in nums1:
            result.append(next_greater.get(num, -1))

        return result
```

The implementation begins by creating a dictionary called `next_greater`. This dictionary stores the next greater value for elements in `nums2`.

The stack stores elements whose next greater value has not yet been found.

As we iterate through `nums2`, we compare the current number with the top of the stack. If the current number is larger, we repeatedly pop smaller elements and record the current number as their next greater element.

After processing the popping logic, we push the current number onto the stack because its own next greater value may appear later.

Finally, we construct the answer for `nums1` using dictionary lookups. If a number does not exist in the dictionary, it means no greater element was found, so we return `-1`.

## Go Solution

```
func nextGreaterElement(nums1 []int, nums2 []int) []int {
	nextGreater := make(map[int]int)
	stack := []int{}

	for _, num := range nums2 {
		for len(stack) > 0 && num > stack[len(stack)-1] {
			smaller := stack[len(stack)-1]
			stack = stack[:len(stack)-1]

			nextGreater[smaller] = num
		}

		stack = append(stack, num)
	}

	result := make([]int, 0, len(nums1))

	for _, num := range nums1 {
		if value, exists := nextGreater[num]; exists {
			result = append(result, value)
		} else {
			result = append(result, -1)
		}
	}

	return result
}
```

The Go implementation follows the exact same algorithmic structure as the Python solution.

The stack is implemented using a slice of integers. Pushing uses `append`, and popping is done by slicing off the last element.

The hash map uses Go's built in `map[int]int` type.

One small difference is that Go requires explicit existence checking when reading from a map. We therefore use:

```
value, exists := nextGreater[num]
```

to determine whether a next greater element was recorded.

There are no integer overflow concerns because the input constraints are very small.

## Worked Examples

### Example 1

Input:

```
nums1 = [4,1,2]
nums2 = [1,3,4,2]
```

We process `nums2` step by step.

| Current Number | Stack Before | Action | Stack After | Map |
| --- | --- | --- | --- | --- |
| 1 | [] | Push 1 | [1] | {} |
| 3 | [1] | 3 > 1, pop 1 | [] | {1: 3} |
| 3 | [] | Push 3 | [3] | {1: 3} |
| 4 | [3] | 4 > 3, pop 3 | [] | {1: 3, 3: 4} |
| 4 | [] | Push 4 | [4] | {1: 3, 3: 4} |
| 2 | [4] | 2 < 4, push | [4, 2] | {1: 3, 3: 4} |

The remaining stack elements `[4, 2]` have no next greater element.

Now answer queries:

| nums1 Value | Next Greater |
| --- | --- |
| 4 | -1 |
| 1 | 3 |
| 2 | -1 |

Final result:

```
[-1, 3, -1]
```

### Example 2

Input:

nums1 = [2,4]

Processing `nums2`:

| --- | --- | --- | --- | --- |
| 1 | [] | Push | [1] | {} |
| 2 | [1] | Pop 1 | [] | {1: 2} |
| 2 | [] | Push | [2] | {1: 2} |
| 3 | [2] | Pop 2 | [] | {1: 2, 2: 3} |

| 4 | [3] | Pop 3 | [] | {1: 2, 2: 3, 3: 4} |

Now answer queries:

| nums1 Value | Next Greater |
| --- | --- |
| 2 | 3 |

Final result:

```
[3, -1]
```

## Complexity Analysis

| Measure | Complexity | Explanation |
| --- | --- | --- |
| Time | O(n + m) | Each element is pushed and popped at most once |
| Space | O(m) | Stack and hash map may store all elements |

Where:

- `n = len(nums1)`
- `m = len(nums2)`

The key observation is that no element is processed more than twice in the stack, once when pushed and once when popped. This guarantees linear time complexity.

The hash map stores next greater mappings for elements in `nums2`, and the stack may temporarily hold many elements in decreasing sequences, leading to linear auxiliary space usage.

## Edge Cases

One important edge case is a strictly decreasing array such as:

```
nums2 = [5,4,3,2,1]
```

In this situation, no element has a next greater value. A buggy implementation may incorrectly leave stale values or fail to return `-1`. Our implementation handles this naturally because elements remaining in the stack after traversal simply never receive entries in the hash map. Missing entries default to `-1`.

Another important edge case is a strictly increasing array such as:

```
nums2 = [1,2,3,4,5]
```

Here, every element except the last immediately finds a next greater value. This stresses repeated popping behavior inside the stack loop. Our implementation correctly pops smaller elements until the stack becomes empty, ensuring every value maps to the first larger element on its right.

A third important edge case is very small input sizes, especially arrays of length `1`.

```
nums1 = [1]
nums2 = [1]
```

There is no element to the right, so the answer must be `-1`. Some implementations accidentally assume a right neighbor exists or mishandle empty stacks. Our algorithm safely handles this because the stack operations are guarded with emptiness checks.

Another subtle case occurs when multiple elements eventually share the same next greater element:

```
nums2 = [3,2,1,4]
```

Both `2` and `1` map to `4`. The monotonic stack correctly resolves each pending smaller element when `4` appears, producing accurate mappings for all of them independently.
