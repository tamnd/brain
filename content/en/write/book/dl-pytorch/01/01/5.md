## 1.5 GPUs and Accelerators

Deep learning became practical at scale because neural network computation maps well to parallel hardware. Training a model requires many repeated tensor operations, especially matrix multiplication, convolution, normalization, and attention. These operations contain large numbers of independent arithmetic computations. GPUs and specialized accelerators execute these computations in parallel.

A CPU can run deep learning models, but it is usually optimized for general-purpose control flow, low-latency execution, and a smaller number of powerful cores. A GPU is optimized for throughput. It contains many simpler cores and high memory bandwidth, which makes it suitable for dense numerical workloads.

### 1.5.1 CPU Versus GPU Computation

A CPU is good at complex branching, operating system tasks, data orchestration, and sequential work. A GPU is good at running the same operation over many numbers at once.

| Hardware | Strength | Deep learning role |
|---|---|---|
| CPU | Control flow and data loading | Preprocessing, orchestration, small models |
| GPU | Parallel numerical throughput | Training and inference |
| TPU | Large tensor operations | Large-scale model training |
| NPU | Low-power neural inference | Phones, laptops, edge devices |

In PyTorch, tensors can be stored on different devices. A CPU tensor lives in host memory. A CUDA tensor lives in GPU memory.

```python
import torch

x_cpu = torch.randn(1024, 1024)
x_gpu = torch.randn(1024, 1024, device="cuda")
```

Operations on `x_cpu` run on the CPU. Operations on `x_gpu` run on the GPU.

### 1.5.2 Checking for GPU Availability

A training script should detect the available device.

```python
import torch

device = "cuda" if torch.cuda.is_available() else "cpu"
print(device)
```

A model must be moved to the same device as its input tensors.

```python
from torch import nn

model = nn.Linear(784, 10).to(device)

x = torch.randn(32, 784).to(device)
y = model(x)
```

If the model is on the GPU and the input is on the CPU, PyTorch raises a device mismatch error. Device placement must be explicit and consistent.

### 1.5.3 Moving Batches to the Device

In a training loop, each mini-batch must be moved to the chosen device.

```python
for x, y in loader:
    x = x.to(device)
    y = y.to(device)

    logits = model(x)
    loss = loss_fn(logits, y)

    optimizer.zero_grad()
    loss.backward()
    optimizer.step()
```

This pattern is simple but important. A common beginner error is moving the model to the GPU while leaving the data on the CPU.

For nested batches, it is useful to write a helper function.

```python
def to_device(batch, device):
    if torch.is_tensor(batch):
        return batch.to(device)
    if isinstance(batch, dict):
        return {k: to_device(v, device) for k, v in batch.items()}
    if isinstance(batch, (list, tuple)):
        return type(batch)(to_device(v, device) for v in batch)
    return batch
```

Then:

```python
for batch in loader:
    batch = to_device(batch, device)
```

### 1.5.4 Matrix Multiplication on GPUs

The main workhorse of deep learning is matrix multiplication.

A linear layer over a batch computes

$$
Y = XW^\top + b.
$$

If

$$
X \in \mathbb{R}^{B \times D}
$$

and

$$
W \in \mathbb{R}^{H \times D},
$$

then

$$
Y \in \mathbb{R}^{B \times H}.
$$

Large matrix multiplications are highly optimized on GPUs.

```python
B = 4096
D = 4096
H = 4096

x = torch.randn(B, D, device=device)
w = torch.randn(H, D, device=device)

y = x @ w.T
```

This operation performs billions of multiply-adds. On a CPU, it may be slow. On a modern GPU, it can be efficient because the work is regular and massively parallel.

### 1.5.5 CUDA

CUDA is NVIDIA’s platform for GPU programming. In PyTorch, CUDA support allows tensors and operations to run on NVIDIA GPUs.

A CUDA tensor is created with:

```python
x = torch.randn(1024, 1024, device="cuda")
```

or moved from CPU:

```python
x = x.to("cuda")
```

You can inspect device information:

```python
print(torch.cuda.device_count())
print(torch.cuda.get_device_name(0))
```

When multiple GPUs are present, you can select one:

```python
device = torch.device("cuda:0")
model = model.to(device)
```

Different accelerators may use different PyTorch device backends. For example, Apple Silicon uses `mps`:

```python
device = (
    "cuda" if torch.cuda.is_available()
    else "mps" if torch.backends.mps.is_available()
    else "cpu"
)
```

### 1.5.6 Memory Transfer Costs

Moving data between CPU and GPU is expensive compared with computation on the GPU. A slow training loop often moves tensors too frequently.

Poor pattern:

```python
for layer in model:
    x = x.to("cuda")
    x = layer(x)
    x = x.to("cpu")
```

Better pattern:

```python
model = model.to(device)

for x, y in loader:
    x = x.to(device)
    y = y.to(device)

    logits = model(x)
```

Move data once per batch. Keep computation on the same device.

When using a `DataLoader`, pinned memory can speed up CPU-to-GPU transfers.

```python
loader = DataLoader(
    dataset,
    batch_size=128,
    shuffle=True,
    num_workers=4,
    pin_memory=True,
)
```

Then use non-blocking transfers:

```python
x = x.to(device, non_blocking=True)
y = y.to(device, non_blocking=True)
```

This is useful when the input pipeline and GPU computation overlap.

### 1.5.7 GPU Memory

GPU memory is usually the limiting resource in training. A model needs memory for:

| Memory use | Description |
|---|---|
| Parameters | Learned weights and biases |
| Gradients | One gradient tensor per parameter |
| Optimizer states | Extra tensors used by optimizers |
| Activations | Intermediate tensors needed for backpropagation |
| Temporary buffers | Workspace used by kernels |

For Adam or AdamW, optimizer states can be large. Each parameter often has two additional state tensors: the first and second moment estimates. Thus optimizer memory can exceed parameter memory.

A rough training memory estimate for AdamW in full precision:

| Component | Approximate size |
|---|---:|
| Parameters | 1x |
| Gradients | 1x |
| Adam first moment | 1x |
| Adam second moment | 1x |
| Activations | model and batch dependent |

The activation term often dominates for large batches, long sequences, and deep models.

### 1.5.8 Batch Size and Memory

Batch size controls how many examples are processed at once. Larger batches can improve hardware utilization, but they require more memory.

```python
batch_size = 32
```

If training runs out of GPU memory, reduce the batch size first.

Other memory-saving techniques include:

| Technique | Effect |
|---|---|
| Mixed precision | Stores some tensors in lower precision |
| Gradient checkpointing | Recomputes activations during backward |
| Gradient accumulation | Simulates larger batches with smaller micro-batches |
| Model sharding | Splits model states across devices |
| Activation offloading | Moves activations outside GPU memory |

Gradient accumulation is especially common.

```python
accum_steps = 4

optimizer.zero_grad()

for step, (x, y) in enumerate(loader):
    x = x.to(device)
    y = y.to(device)

    logits = model(x)
    loss = loss_fn(logits, y)
    loss = loss / accum_steps
    loss.backward()

    if (step + 1) % accum_steps == 0:
        optimizer.step()
        optimizer.zero_grad()
```

This uses several smaller micro-batches before one optimizer update.

### 1.5.9 Mixed Precision Training

Modern accelerators can run lower-precision arithmetic faster than standard 32-bit floating point. Mixed precision training uses lower precision where safe and higher precision where needed.

Common formats:

| Format | Bits | Use |
|---|---:|---|
| `float32` | 32 | Standard training |
| `float16` | 16 | Fast GPU training |
| `bfloat16` | 16 | Stable large-model training |
| `float8` | 8 | Emerging large-scale training and inference |

In PyTorch, automatic mixed precision is used with `autocast`.

```python
scaler = torch.cuda.amp.GradScaler()

for x, y in loader:
    x = x.to(device)
    y = y.to(device)

    optimizer.zero_grad()

    with torch.cuda.amp.autocast():
        logits = model(x)
        loss = loss_fn(logits, y)

    scaler.scale(loss).backward()
    scaler.step(optimizer)
    scaler.update()
```

For newer PyTorch versions, the device-specific form is preferred:

```python
scaler = torch.amp.GradScaler("cuda")

with torch.amp.autocast("cuda"):
    logits = model(x)
    loss = loss_fn(logits, y)
```

Mixed precision reduces memory use and can increase throughput. It also introduces numerical considerations. Some operations are sensitive to lower precision and must remain in higher precision.

### 1.5.10 Tensor Cores

Tensor Cores are specialized units on modern NVIDIA GPUs for matrix operations. They accelerate mixed-precision matrix multiplication and convolution.

They are especially useful for:

- Large linear layers
- Transformer attention projections
- Feedforward blocks
- Convolutions
- Batched matrix multiplication

To use Tensor Cores efficiently, tensor shapes should often be multiples of hardware-friendly sizes, such as 8, 16, or 64 depending on dtype and device.

This is one reason model dimensions such as 768, 1024, 2048, and 4096 are common. They align well with efficient matrix multiplication kernels.

### 1.5.11 Keeping the GPU Busy

A GPU can be underutilized even when the model is correct. Common causes include slow data loading, small batches, excessive CPU work, frequent synchronization, and inefficient tensor shapes.

Symptoms include low GPU utilization and high CPU activity.

Useful practices:

| Practice | Reason |
|---|---|
| Increase `num_workers` | Parallelize data loading |
| Use `pin_memory=True` | Faster host-to-device transfer |
| Batch operations | Improve parallelism |
| Avoid Python loops over samples | Reduce overhead |
| Avoid frequent `.item()` calls | Prevent synchronization |
| Use profiling tools | Find bottlenecks |

The method `.item()` transfers a scalar from GPU to CPU and forces synchronization. This is fine for logging occasionally, but harmful inside tight loops.

Poor pattern:

```python
for x, y in loader:
    loss = train_step(x, y)
    losses.append(loss.item())
```

Better pattern: log less frequently.

```python
if step % 100 == 0:
    print(loss.item())
```

### 1.5.12 Profiling GPU Workloads

Performance should be measured before optimization. PyTorch provides a profiler.

```python
with torch.profiler.profile(
    activities=[
        torch.profiler.ProfilerActivity.CPU,
        torch.profiler.ProfilerActivity.CUDA,
    ],
    record_shapes=True,
) as prof:
    logits = model(x)
    loss = loss_fn(logits, y)
    loss.backward()

print(prof.key_averages().table(sort_by="cuda_time_total"))
```

Profiling helps answer practical questions:

| Question | What to inspect |
|---|---|
| Is the GPU busy? | CUDA time and utilization |
| Is data loading slow? | CPU time and dataloader stalls |
| Are tensor shapes inefficient? | Operator timings |
| Is memory high? | Activation and allocation patterns |
| Are transfers frequent? | CPU-GPU copy operations |

Without profiling, optimization is often guesswork.

### 1.5.13 Multi-GPU Training

When one GPU is insufficient, training can be distributed across several GPUs.

The most common method is data parallelism. Each GPU has a model replica and processes a different mini-batch. Gradients are synchronized before updating parameters.

PyTorch provides `DistributedDataParallel`, usually abbreviated DDP.

Conceptually:

```text
GPU 0: model replica + batch 0
GPU 1: model replica + batch 1
GPU 2: model replica + batch 2
GPU 3: model replica + batch 3

Backward pass computes local gradients.
Gradients are averaged across GPUs.
Each replica applies the same update.
```

For very large models, data parallelism may be insufficient because the model itself cannot fit on one GPU. Then model parallelism or sharding is needed.

Common strategies:

| Strategy | Purpose |
|---|---|
| Data parallelism | Split batches |
| Tensor parallelism | Split large matrix operations |
| Pipeline parallelism | Split layers across devices |
| Fully sharded data parallelism | Shard parameters, gradients, optimizer state |
| Expert parallelism | Split mixture-of-experts modules |

These techniques become central in foundation model training.

### 1.5.14 Accelerator-Aware Model Design

Model architecture affects hardware efficiency.

Two models may have similar parameter counts but different runtime behavior. A model with many small operations may run slower than a model with fewer large matrix multiplications. GPUs prefer large, regular operations.

Transformer models are hardware-friendly because they rely heavily on dense matrix multiplication. However, long-context attention can be expensive because attention scores scale with sequence length squared.

For sequence length \(T\), standard self-attention forms an attention matrix of shape

$$
T \times T.
$$

For a batch with heads:

$$
[B, H, T, T].
$$

This can dominate memory for long sequences.

Efficient attention methods reduce memory traffic, approximate attention, use sparsity, or change the architecture.

### 1.5.15 Inference on Accelerators

Training and inference have different constraints.

Training requires forward computation, activation storage, backward computation, gradients, and optimizer states. Inference usually requires only the forward pass.

Inference can therefore use:

| Technique | Benefit |
|---|---|
| `torch.no_grad()` | Avoid gradient graph |
| `model.eval()` | Correct inference behavior |
| Quantization | Lower memory and faster compute |
| KV caching | Faster autoregressive decoding |
| Compilation | Lower overhead |
| Batching | Higher throughput |

Example:

```python
model.eval()

with torch.no_grad():
    logits = model(x)
```

For language models, inference often stores key and value tensors from previous tokens. This is called KV caching. It avoids recomputing attention over the full prefix at every generated token.

### 1.5.16 Common Device Errors

A common PyTorch error is mixing devices.

```python
model = nn.Linear(10, 2).to("cuda")
x = torch.randn(4, 10)

logits = model(x)  # error
```

The fix is to move `x`:

```python
x = x.to("cuda")
logits = model(x)
```

Another common problem is creating a new tensor inside `forward()` without placing it on the same device.

Poor pattern:

```python
def forward(self, x):
    bias = torch.zeros(x.shape[-1])
    return x + bias
```

Better pattern:

```python
def forward(self, x):
    bias = torch.zeros(x.shape[-1], device=x.device, dtype=x.dtype)
    return x + bias
```

A general rule: new tensors created during computation should usually inherit device and dtype from existing tensors.

### 1.5.17 Practical Device Template

A robust single-GPU training script has this shape:

```python
import torch
from torch import nn

device = (
    "cuda" if torch.cuda.is_available()
    else "mps" if torch.backends.mps.is_available()
    else "cpu"
)

model = MyModel().to(device)
loss_fn = nn.CrossEntropyLoss()
optimizer = torch.optim.AdamW(model.parameters(), lr=3e-4)

for epoch in range(num_epochs):
    model.train()

    for x, y in train_loader:
        x = x.to(device, non_blocking=True)
        y = y.to(device, non_blocking=True)

        logits = model(x)
        loss = loss_fn(logits, y)

        optimizer.zero_grad(set_to_none=True)
        loss.backward()
        optimizer.step()

    model.eval()

    with torch.no_grad():
        for x, y in val_loader:
            x = x.to(device, non_blocking=True)
            y = y.to(device, non_blocking=True)

            logits = model(x)
            val_loss = loss_fn(logits, y)
```

The call `zero_grad(set_to_none=True)` can reduce memory operations compared with setting gradients to zero.

### 1.5.18 Summary

GPUs and accelerators make deep learning practical by executing tensor operations in parallel. PyTorch exposes these devices through tensor placement. A tensor has a device, and operations run where the tensor lives.

Effective accelerator use requires consistent device placement, large batched operations, limited CPU-GPU transfers, enough data-loading throughput, and careful memory management. As models grow, accelerator-aware design becomes part of model design itself.

Training performance is rarely determined by one factor. It depends on model architecture, tensor shapes, memory layout, precision, batch size, data pipeline, kernel efficiency, and communication between devices. A good PyTorch practitioner learns to treat hardware as part of the model’s execution environment.
