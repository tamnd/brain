## 1.6 PyTorch Versus Other Frameworks

PyTorch is one of several major frameworks for deep learning. A framework provides the basic machinery needed to define models, run tensor operations, compute gradients, optimize parameters, load data, distribute training, and deploy trained models.

The main frameworks in modern deep learning include PyTorch, TensorFlow, JAX, Keras, and specialized inference runtimes such as ONNX Runtime and TensorRT. Each framework has a different design center.

| Framework | Design center | Typical use |
|---|---|---|
| PyTorch | Python-first dynamic deep learning | Research, training, production systems |
| TensorFlow | Graph-based production ML platform | Large production ML systems, mobile deployment |
| Keras | High-level model building | Fast prototyping, teaching, applied workflows |
| JAX | NumPy-like functional computation with compilation | Research, high-performance numerical computing |
| ONNX Runtime | Framework-independent inference | Portable deployment |
| TensorRT | NVIDIA-optimized inference | Low-latency GPU serving |

PyTorch is not automatically the best framework for every problem. Its main advantage is that it gives direct control over tensor computation while keeping the programming model close to ordinary Python.

### 1.6.1 The PyTorch Design

PyTorch emphasizes imperative execution. When you write an operation, it runs immediately.

```python
x = torch.randn(4, 8)
y = x.relu()
print(y)
```

This makes PyTorch easy to debug. You can inspect values, shapes, gradients, and intermediate results using normal Python tools.

A PyTorch model is usually an `nn.Module`:

```python
from torch import nn

class Classifier(nn.Module):
    def __init__(self):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(784, 256),
            nn.ReLU(),
            nn.Linear(256, 10),
        )

    def forward(self, x):
        return self.net(x)
```

This design is explicit. The programmer controls the forward computation. PyTorch records the computation graph dynamically and uses it for automatic differentiation.

PyTorch tends to be preferred when the model contains custom logic, unusual training loops, dynamic control flow, or experimental architectures.

### 1.6.2 TensorFlow

TensorFlow was designed as a system for building and deploying machine learning computation graphs. Earlier TensorFlow versions emphasized static graphs. Users built a graph first, then executed it in a session.

Modern TensorFlow uses eager execution by default, but graph execution remains central through `tf.function`.

A TensorFlow model is often written using Keras layers:

```python
import tensorflow as tf

model = tf.keras.Sequential([
    tf.keras.layers.Dense(256, activation="relu"),
    tf.keras.layers.Dense(10),
])
```

TensorFlow has strong production infrastructure, especially around TensorFlow Serving, TensorFlow Lite, TensorFlow.js, and deployment pipelines.

TensorFlow is often useful when a team already depends on TensorFlow tooling, mobile deployment, browser deployment, or existing production ML infrastructure.

### 1.6.3 Keras

Keras is a high-level API for building neural networks. It focuses on simple model construction and readable training workflows.

A Keras model can be trained with:

```python
model.compile(
    optimizer="adam",
    loss="sparse_categorical_crossentropy",
    metrics=["accuracy"],
)

model.fit(train_data, epochs=10, validation_data=val_data)
```

Keras is useful for:

| Use case | Reason |
|---|---|
| Teaching | Minimal boilerplate |
| Prototyping | Fast model construction |
| Standard workflows | Built-in training loops |
| Applied projects | Concise APIs |

The tradeoff is control. PyTorch usually exposes lower-level details more directly. Keras can also support custom training loops, but its default style is more abstract.

For this book, PyTorch is the main framework because it makes the mechanics of deep learning visible: tensors, modules, gradients, optimizers, and training loops.

### 1.6.4 JAX

JAX is a framework for high-performance numerical computing. Its programming model resembles NumPy but adds automatic differentiation, vectorization, and compilation.

JAX code often uses functional transformations:

```python
import jax
import jax.numpy as jnp

def f(x):
    return jnp.sum(x * x)

grad_f = jax.grad(f)
```

Important JAX transformations include:

| Transformation | Purpose |
|---|---|
| `grad` | Automatic differentiation |
| `jit` | Just-in-time compilation |
| `vmap` | Automatic vectorization |
| `pmap` | Parallel mapping across devices |

JAX is popular in research areas where functional programming, compiler-level optimization, and mathematical transformation of programs are important. It is also widely used in some large-scale model training systems.

The main difference is style. PyTorch usually writes models as stateful modules. JAX usually favors pure functions and explicit parameter passing.

PyTorch style:

```python
logits = model(x)
loss = loss_fn(logits, y)
loss.backward()
optimizer.step()
```

JAX style:

```python
loss, grads = jax.value_and_grad(loss_fn)(params, batch)
params = update(params, grads)
```

The JAX style is mathematically clean and compiler-friendly. The PyTorch style is direct and object-oriented.

### 1.6.5 PyTorch and JAX Compared

PyTorch and JAX are often compared because both are popular in research.

| Dimension | PyTorch | JAX |
|---|---|---|
| Programming style | Imperative, object-oriented | Functional transformations |
| Model state | Stored in modules | Passed explicitly |
| Debugging | Direct Python debugging | More care under `jit` |
| Compilation | `torch.compile` | Core design through `jit` |
| Autograd | Dynamic graph autograd | Function transformations |
| Ecosystem | Broad applied deep learning | Strong numerical research ecosystem |
| Learning curve | Lower for Python programmers | Higher for stateful model builders |

Neither style is universally better. PyTorch is usually easier when building standard neural networks and custom training loops. JAX is strong when the program benefits from functional transformations, compilation, and precise control over array programs.

### 1.6.6 ONNX and Inference Runtimes

Training frameworks and inference runtimes solve different problems.

A training framework must support gradients, optimizers, data loading, distributed training, and experimentation. An inference runtime only needs to execute a trained model efficiently.

ONNX is a portable model representation. A model trained in PyTorch can sometimes be exported to ONNX and then run using ONNX Runtime.

```python
torch.onnx.export(
    model,
    example_input,
    "model.onnx",
    input_names=["input"],
    output_names=["logits"],
)
```

ONNX is useful when deployment needs framework independence. For example, a team may train in PyTorch but deploy in a C++ service, cloud inference platform, or edge runtime.

Specialized runtimes such as TensorRT optimize inference for NVIDIA GPUs. They may perform layer fusion, precision lowering, memory planning, and kernel selection.

### 1.6.7 Research Versus Production

A framework’s suitability depends on the phase of the project.

During research, priorities include:

| Research priority | Why it matters |
|---|---|
| Fast iteration | Try many ideas |
| Debuggability | Inspect failures |
| Flexibility | Express unusual models |
| Ecosystem access | Use existing models and libraries |

During production, priorities include:

| Production priority | Why it matters |
|---|---|
| Reliability | Stable service behavior |
| Throughput | Serve many requests |
| Latency | Respond quickly |
| Observability | Monitor failures |
| Portability | Run across environments |
| Cost efficiency | Reduce hardware spend |

PyTorch is strong in research and increasingly strong in production. TensorFlow has historically been strong in production deployment. JAX is strong for high-performance research and some large-scale training systems. ONNX Runtime and TensorRT are often used when inference efficiency matters more than training flexibility.

### 1.6.8 Why This Book Uses PyTorch

This book uses PyTorch because it exposes the core ideas of deep learning directly.

The learner can see:

| Concept | PyTorch object |
|---|---|
| Data arrays | `torch.Tensor` |
| Model structure | `nn.Module` |
| Parameters | `nn.Parameter` |
| Loss functions | `nn loss` modules or functions |
| Gradients | `.grad` tensors |
| Backpropagation | `loss.backward()` |
| Optimization | `torch.optim` |
| Batches | `DataLoader` |

A minimal PyTorch training loop is close to the mathematical definition of learning:

```python
for x, y in loader:
    pred = model(x)
    loss = loss_fn(pred, y)

    optimizer.zero_grad()
    loss.backward()
    optimizer.step()
```

This loop contains the key components: prediction, error measurement, differentiation, and parameter update.

### 1.6.9 When PyTorch Is a Good Choice

PyTorch is a good choice when you need direct control over model code and training behavior.

Typical cases include:

| Case | Why PyTorch fits |
|---|---|
| Research models | Flexible Python control flow |
| Custom loss functions | Easy tensor-level programming |
| Nonstandard training loops | Direct optimizer control |
| Transformers and LLMs | Strong ecosystem support |
| Computer vision | Mature `torchvision` ecosystem |
| Graph learning | PyTorch Geometric ecosystem |
| Distributed training | DDP, FSDP, ecosystem tools |
| Teaching deep learning | Clear execution model |

It is also a good default for many applied deep learning projects because pretrained models, libraries, and examples are widely available.

### 1.6.10 When Another Framework May Be Better

Another framework may be a better fit when the surrounding requirements dominate the model code.

TensorFlow may be preferable when the system already uses TensorFlow Serving, TensorFlow Lite, or TensorFlow.js.

Keras may be preferable when the goal is quick standard modeling with minimal code.

JAX may be preferable when the project needs functional transformations, compiler-first execution, or large-scale numerical research patterns.

ONNX Runtime may be preferable when the model is already trained and portable inference is the main requirement.

TensorRT may be preferable when NVIDIA GPU inference latency and throughput are the dominant constraints.

The choice of framework should follow the system requirements rather than ideology.

### 1.6.11 Interoperability

Modern deep learning work often uses several tools together.

A project may:

1. Train a model in PyTorch.
2. Use Hugging Face for pretrained weights and tokenizers.
3. Export the model to ONNX.
4. Serve it with ONNX Runtime or TensorRT.
5. Monitor it in a production service.

The training framework, model hub, tokenizer library, inference runtime, and serving system may be separate components.

For example:

```text
PyTorch training
    -> checkpoint
    -> ONNX export
    -> inference runtime
    -> API service
```

This separation is common in production systems.

### 1.6.12 Framework Choice and Model Understanding

The framework is a tool. It should not hide the mathematics.

A deep learning practitioner should understand what the framework is doing:

| Framework operation | Mathematical idea |
|---|---|
| `model(x)` | Function evaluation |
| `loss_fn(pred, y)` | Objective computation |
| `loss.backward()` | Gradient computation |
| `optimizer.step()` | Parameter update |
| `model.eval()` | Inference mode |
| `torch.no_grad()` | Disable gradient graph |

Once these ideas are clear, moving between frameworks becomes easier. PyTorch, TensorFlow, JAX, and other systems differ in syntax and execution model, but they solve many of the same underlying problems.

### 1.6.13 Summary

PyTorch is a Python-first deep learning framework built around tensors, dynamic computation graphs, automatic differentiation, and neural network modules. It is especially strong when flexibility, debuggability, and custom training logic matter.

TensorFlow emphasizes production ML infrastructure. Keras emphasizes high-level model construction. JAX emphasizes functional transformations and compilation. ONNX Runtime and TensorRT emphasize deployment and inference.

This book uses PyTorch because it keeps the core mechanics visible. You write the forward computation, compute a loss, call backpropagation, and update parameters. That direct workflow makes PyTorch a good framework for learning deep learning from first principles while still being practical for modern systems.
