## 1.4 Tensor-Based Computation

PyTorch programs are tensor programs. A tensor stores numbers in a structured array, and most model computation is expressed as operations over tensors. This includes matrix multiplication, convolution, attention, normalization, indexing, reshaping, reduction, and elementwise arithmetic.

A neural network can be understood as a composition of tensor transformations:

$$
X \longrightarrow H_1 \longrightarrow H_2 \longrightarrow \cdots \longrightarrow Y.
$$

The input tensor \(X\) contains data. The intermediate tensors \(H_1, H_2, \ldots\) contain hidden representations. The output tensor \(Y\) contains predictions, logits, generated tokens, embeddings, or other model outputs.

### 1.4.1 Why Tensors Are Central

Deep learning uses tensors because they match both the mathematics and the hardware.

Mathematically, many learning operations are vectorized. A batch of examples becomes a matrix or higher-dimensional tensor. A layer applies the same operation to many examples at once. This makes the code concise and the computation regular.

Hardware also favors tensor operations. GPUs and accelerators are designed to run many numerical operations in parallel. Large matrix multiplications and convolutions can be executed much faster than scalar Python loops.

For example, this loop-based computation is slow:

```python
outputs = []

for x in batch:
    y = model(x)
    outputs.append(y)

outputs = torch.stack(outputs)
```

A tensor-based version is preferred:

```python
outputs = model(batch)
```

The second version lets PyTorch execute optimized kernels over the whole batch.

### 1.4.2 Shapes as Types

In ordinary programming, types tell us what operations are valid. In tensor programming, shapes play a similar role.

A tensor with shape `[32, 784]` may represent a batch of 32 flattened images, each with 784 features. A tensor with shape `[32, 3, 224, 224]` may represent 32 RGB images. A tensor with shape `[16, 128, 768]` may represent 16 token sequences, each with 128 tokens and 768-dimensional embeddings.

The shape tells us what the tensor means.

| Shape | Common meaning |
|---|---|
| `[B, D]` | Batch of feature vectors |
| `[B, C, H, W]` | Batch of images |
| `[B, T]` | Batch of token IDs |
| `[B, T, D]` | Batch of token embeddings |
| `[N, D]` | Node features or sample features |
| `[B, K]` | Class logits for a batch |

Shape errors are among the most common PyTorch errors. The model may be conceptually correct, but if a tensor has the wrong axis order or incompatible dimensions, the program fails or trains incorrectly.

A useful habit is to annotate tensor shapes in comments:

```python
# x: [B, D]
# weight: [D, H]
# bias: [H]
h = x @ weight + bias  # [B, H]
```

This makes tensor code easier to inspect.

### 1.4.3 Elementwise Operations

Elementwise operations apply independently to each tensor entry.

If

$$
x =
\begin{bmatrix}
1 \\
2 \\
3
\end{bmatrix},
$$

then

$$
x + 1 =
\begin{bmatrix}
2 \\
3 \\
4
\end{bmatrix}.
$$

In PyTorch:

```python
import torch

x = torch.tensor([1.0, 2.0, 3.0])

print(x + 1)
print(x * 2)
print(x ** 2)
```

Many activation functions are elementwise:

```python
x = torch.randn(4)

relu = torch.relu(x)
sigmoid = torch.sigmoid(x)
tanh = torch.tanh(x)
```

Elementwise operations preserve shape:

```python
x = torch.randn(32, 128)
y = torch.relu(x)

print(y.shape)  # torch.Size([32, 128])
```

### 1.4.4 Matrix Multiplication

Matrix multiplication is the central operation in neural networks.

Suppose

$$
X \in \mathbb{R}^{B \times D}
$$

is a batch of \(B\) examples, each with \(D\) features. Suppose

$$
W \in \mathbb{R}^{D \times H}.
$$

Then

$$
Y = XW
$$

has shape

$$
Y \in \mathbb{R}^{B \times H}.
$$

In PyTorch:

```python
B = 32
D = 128
H = 64

X = torch.randn(B, D)
W = torch.randn(D, H)

Y = X @ W

print(Y.shape)  # torch.Size([32, 64])
```

A linear layer adds a bias:

$$
Y = XW + b.
$$

```python
b = torch.randn(H)

Y = X @ W + b
```

The bias is broadcast across the batch dimension.

PyTorch’s `nn.Linear` stores weights in shape `[out_features, in_features]`. Therefore it computes

$$
Y = XW^\top + b.
$$

```python
from torch import nn

linear = nn.Linear(128, 64)

X = torch.randn(32, 128)
Y = linear(X)

print(linear.weight.shape)  # torch.Size([64, 128])
print(Y.shape)              # torch.Size([32, 64])
```

### 1.4.5 Broadcasting

Broadcasting allows PyTorch to combine tensors with different but compatible shapes.

For example:

```python
X = torch.randn(32, 64)
b = torch.randn(64)

Y = X + b
```

Here `b` has shape `[64]`, but PyTorch treats it as if it had shape `[1, 64]`, then repeats it across the first dimension.

Broadcasting compares dimensions from the right. Two dimensions are compatible when they are equal or one of them is 1.

| Tensor A | Tensor B | Result |
|---|---|---|
| `[32, 64]` | `[64]` | `[32, 64]` |
| `[32, 64]` | `[1, 64]` | `[32, 64]` |
| `[16, 32, 64]` | `[64]` | `[16, 32, 64]` |
| `[16, 32, 64]` | `[32, 64]` | `[16, 32, 64]` |
| `[16, 32, 64]` | `[16, 1, 1]` | `[16, 32, 64]` |

Broadcasting is convenient, but it can hide shape mistakes. If a tensor broadcasts when you expected a shape error, the model may compute the wrong result silently.

### 1.4.6 Reductions

A reduction collapses one or more axes.

Common reductions include:

```python
x = torch.randn(32, 128)

x_sum = x.sum()
x_mean = x.mean()
x_max = x.max()
```

A full reduction returns a scalar tensor:

```python
loss = x.mean()
print(loss.shape)  # torch.Size([])
```

A reduction over one axis keeps the other axes:

```python
x = torch.randn(32, 128)

mean_per_example = x.mean(dim=1)
print(mean_per_example.shape)  # torch.Size([32])

mean_per_feature = x.mean(dim=0)
print(mean_per_feature.shape)  # torch.Size([128])
```

For classification logits:

```python
logits = torch.randn(32, 10)

pred = logits.argmax(dim=1)
print(pred.shape)  # torch.Size([32])
```

The operation `argmax(dim=1)` chooses the highest-scoring class for each example.

### 1.4.7 Reshaping

Reshaping changes tensor shape without changing the logical data values.

```python
x = torch.randn(32, 3, 224, 224)

flat = x.reshape(32, -1)

print(flat.shape)  # torch.Size([32, 150528])
```

The value `-1` means “infer this dimension.”

Common reshaping operations include:

| Operation | Purpose |
|---|---|
| `reshape` | Change shape |
| `view` | Change shape when memory layout allows |
| `flatten` | Collapse axes |
| `unsqueeze` | Add an axis of length 1 |
| `squeeze` | Remove axes of length 1 |
| `permute` | Reorder axes |
| `transpose` | Swap two axes |

Example:

```python
x = torch.randn(64)

x = x.unsqueeze(0)
print(x.shape)  # torch.Size([1, 64])

x = x.squeeze(0)
print(x.shape)  # torch.Size([64])
```

Axis reordering is common in image processing:

```python
x = torch.randn(32, 224, 224, 3)  # NHWC
x = x.permute(0, 3, 1, 2)         # NCHW

print(x.shape)  # torch.Size([32, 3, 224, 224])
```

PyTorch convolution layers usually expect `[B, C, H, W]`.

### 1.4.8 Views and Memory Layout

A tensor has a logical shape and a physical memory layout. Some operations return views into the same storage rather than copying data.

For example:

```python
x = torch.randn(3, 4)
y = x.view(12)
```

The tensor `y` may share memory with `x`. Changing one can affect the other.

After `permute`, the tensor may no longer be contiguous in memory:

```python
x = torch.randn(32, 224, 224, 3)
y = x.permute(0, 3, 1, 2)

print(y.is_contiguous())
```

Some operations require contiguous memory. Calling `.contiguous()` creates a contiguous copy:

```python
y = y.contiguous()
```

This distinction affects performance and correctness when using `view`.

A safe default is to use `reshape` unless you specifically need `view`.

### 1.4.9 Indexing and Masking

Indexing selects entries or subtensors.

```python
x = torch.randn(32, 3, 224, 224)

first_image = x[0]
first_channel = x[:, 0, :, :]
center_patch = x[:, :, 96:128, 96:128]
```

Boolean masks select entries satisfying a condition:

```python
x = torch.randn(10)

mask = x > 0
positive_values = x[mask]
```

Masks are also used in attention and sequence modeling. For example, padding masks indicate which tokens are real and which tokens are padding.

```python
tokens = torch.tensor([
    [4, 18, 92, 0, 0],
    [7, 11, 6, 31, 2],
])

padding_mask = tokens == 0
```

The mask has the same shape as `tokens` and contains Boolean values.

### 1.4.10 Devices

Every tensor lives on a device. Common devices include CPU and CUDA GPU.

```python
x = torch.randn(32, 128)

print(x.device)
```

Move a tensor to GPU:

```python
device = "cuda" if torch.cuda.is_available() else "cpu"

x = x.to(device)
```

A model also has parameters on a device:

```python
model = nn.Linear(128, 64).to(device)

x = torch.randn(32, 128).to(device)

y = model(x)
```

All tensors in an operation must usually be on the same device. This is a common source of errors.

```python
# RuntimeError if x is on CUDA and model is on CPU
```

A robust training script explicitly moves both model and batches to the selected device.

```python
for x, y in loader:
    x = x.to(device)
    y = y.to(device)

    logits = model(x)
```

### 1.4.11 Data Types and Precision

The tensor data type controls how numbers are stored.

| dtype | Use |
|---|---|
| `torch.float32` | Standard training |
| `torch.float16` | Mixed precision on GPUs |
| `torch.bfloat16` | Large model training on modern accelerators |
| `torch.float64` | High-precision numerical work |
| `torch.int64` | Class labels and token IDs |
| `torch.bool` | Masks |

Example:

```python
x = torch.randn(32, 128, dtype=torch.float32)
labels = torch.randint(0, 10, (32,), dtype=torch.int64)
mask = torch.zeros(32, 128, dtype=torch.bool)
```

Loss functions often expect specific dtypes. For example, `nn.CrossEntropyLoss` expects floating-point logits and integer class labels.

```python
loss_fn = nn.CrossEntropyLoss()

logits = torch.randn(32, 10)          # float
labels = torch.randint(0, 10, (32,))  # int64

loss = loss_fn(logits, labels)
```

Using the wrong dtype can cause errors or unintended behavior.

### 1.4.12 Vectorization

Vectorization means replacing Python loops with tensor operations.

Loop version:

```python
y = torch.empty_like(x)

for i in range(x.shape[0]):
    y[i] = x[i] * 2 + 1
```

Vectorized version:

```python
y = x * 2 + 1
```

The vectorized form is shorter and faster because it delegates work to optimized kernels.

Vectorization is especially important for training. Python loops over examples prevent the GPU from being used efficiently. A good PyTorch program expresses computation over full batches whenever possible.

### 1.4.13 Tensor Computation in a Linear Classifier

Consider a classifier for \(K\) classes. Each input has \(D\) features, and a batch contains \(B\) examples.

```python
B = 32
D = 784
K = 10

x = torch.randn(B, D)
model = nn.Linear(D, K)

logits = model(x)

print(logits.shape)  # torch.Size([32, 10])
```

The output `logits` contains one row per example and one column per class.

Targets have shape `[B]`:

```python
target = torch.randint(0, K, (B,))
```

The loss is scalar:

```python
loss_fn = nn.CrossEntropyLoss()
loss = loss_fn(logits, target)

print(loss.shape)  # torch.Size([])
```

This gives the basic tensor flow:

| Object | Shape | Meaning |
|---|---|---|
| `x` | `[B, D]` | Input batch |
| `weight` | `[K, D]` | Classifier weights |
| `bias` | `[K]` | Classifier bias |
| `logits` | `[B, K]` | Class scores |
| `target` | `[B]` | Class labels |
| `loss` | `[]` | Scalar objective |

Training then calls:

```python
optimizer.zero_grad()
loss.backward()
optimizer.step()
```

The gradients have the same shapes as the parameters.

### 1.4.14 Tensor Computation in a Transformer Block

A transformer block also consists of tensor operations, but with richer shapes.

For token embeddings:

$$
X \in \mathbb{R}^{B \times T \times D},
$$

where \(B\) is batch size, \(T\) is sequence length, and \(D\) is embedding dimension.

Self-attention projects \(X\) into queries, keys, and values:

$$
Q = XW_Q,\quad K = XW_K,\quad V = XW_V.
$$

Each has shape `[B, T, D]` before splitting into heads.

Attention scores have shape `[B, num_heads, T, T]`. The final output returns to shape `[B, T, D]`.

This illustrates a broader rule: deep learning models mostly preserve batch structure while transforming feature structure.

```python
B = 8
T = 128
D = 768

x = torch.randn(B, T, D)

layer = nn.TransformerEncoderLayer(
    d_model=D,
    nhead=12,
    batch_first=True,
)

y = layer(x)

print(y.shape)  # torch.Size([8, 128, 768])
```

Tensor shape tracking is essential for understanding transformers.

### 1.4.15 Tensor Computation and Gradients

When tensors participate in autograd, PyTorch tracks the operations used to create them.

```python
x = torch.randn(32, 128)
linear = nn.Linear(128, 64)

y = linear(x)
loss = y.pow(2).mean()

loss.backward()
```

The parameter gradients match parameter shapes:

```python
print(linear.weight.shape)
print(linear.weight.grad.shape)

print(linear.bias.shape)
print(linear.bias.grad.shape)
```

A useful invariant:

```text
parameter.grad.shape == parameter.shape
```

This holds because each parameter entry has one corresponding partial derivative.

### 1.4.16 Numerical Stability

Tensor computations can overflow, underflow, or lose precision.

For example, directly computing softmax as

$$
\frac{e^{x_i}}{\sum_j e^{x_j}}
$$

can overflow when some \(x_i\) is large.

A stable implementation subtracts the maximum logit before exponentiation:

```python
x = torch.randn(32, 10) * 100

z = x - x.max(dim=1, keepdim=True).values
probs = z.exp() / z.exp().sum(dim=1, keepdim=True)
```

PyTorch loss functions often include stable implementations internally. For classification, prefer `nn.CrossEntropyLoss` over manually applying `softmax` followed by log loss.

```python
loss_fn = nn.CrossEntropyLoss()
loss = loss_fn(logits, target)
```

This combines log-softmax and negative log-likelihood in a numerically stable way.

### 1.4.17 Tensor Computation and Performance

Efficient tensor computation depends on more than big-O complexity. Hardware utilization matters.

Important performance rules:

| Rule | Reason |
|---|---|
| Use batches | Increases parallelism |
| Avoid Python loops over examples | Prevents kernel launch overhead |
| Keep tensors on the same device | Avoids slow transfers |
| Use contiguous layouts when needed | Improves memory access |
| Prefer built-in operators | Uses optimized kernels |
| Use mixed precision carefully | Improves throughput and memory use |
| Profile before optimizing | Finds the actual bottleneck |

A slow training loop often comes from data loading or CPU-GPU transfer rather than the model itself.

### 1.4.18 Summary

Tensor-based computation is the operational core of PyTorch. Data, parameters, activations, losses, and gradients are tensors. Neural networks are programs that transform tensors through differentiable operations.

To read PyTorch code well, track tensor shape, dtype, device, and gradient status. Shape determines mathematical validity. Dtype determines numerical representation. Device determines where computation runs. Gradient status determines whether autograd records the operation.

Most deep learning bugs appear first as tensor bugs. A disciplined PyTorch programmer treats tensor shape as part of the program’s meaning.
