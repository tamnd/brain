A PyTorch project should separate concerns. Model code should define computation. Data code should prepare examples and batches. Training code should connect models, data, losses, optimizers, logging, checkpoints, and evaluation. This separation keeps experiments readable and reduces hidden coupling.

A small project can start as one file. That is acceptable for a short experiment. As soon as the project has multiple datasets, models, or training runs, a clearer structure becomes useful.

### Minimal Project Layout

A minimal project can use this layout:

```text
project/
  train.py
  model.py
  data.py
  eval.py
  requirements.txt
  README.md
```

Each file has one main responsibility.

| File | Responsibility |
|---|---|
| `model.py` | Model classes and layers |
| `data.py` | Dataset and DataLoader construction |
| `train.py` | Training loop |
| `eval.py` | Evaluation logic |
| `requirements.txt` | Dependencies |
| `README.md` | Setup and usage notes |

This layout is enough for many early experiments.

### Package Layout

A larger project should use a package layout:

```text
project/
  pyproject.toml
  src/
    dlproject/
      __init__.py
      data/
        __init__.py
        datasets.py
        transforms.py
      models/
        __init__.py
        mlp.py
        cnn.py
        transformer.py
      training/
        __init__.py
        loop.py
        checkpoint.py
        metrics.py
      evaluation/
        __init__.py
        evaluate.py
      utils/
        __init__.py
        seed.py
        device.py
  scripts/
    train_mnist.py
    eval_checkpoint.py
  configs/
    mnist.yaml
  tests/
    test_shapes.py
    test_training_step.py
  checkpoints/
  runs/
```

The `src/` layout prevents accidental imports from the working directory and makes packaging cleaner. The `scripts/` directory contains command-line entry points. The `configs/` directory stores experiment settings. The `tests/` directory checks shape contracts, model behavior, and training utilities.

Generated directories such as `checkpoints/` and `runs/` should usually be ignored by version control.

### Model Code

Model files should define modules, not training logic.

Example `models/mlp.py`:

```python
from torch import nn

class MLP(nn.Module):
    def __init__(self, in_features: int, hidden_features: int, out_features: int):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(in_features, hidden_features),
            nn.ReLU(),
            nn.Linear(hidden_features, out_features),
        )

    def forward(self, x):
        return self.net(x)
```

This module does not know about epochs, optimizers, metrics, files, or devices. It only defines the forward computation.

A useful rule: `forward()` should describe the model’s mathematical computation. Avoid putting training-loop side effects inside it.

### Data Code

Data code should construct datasets, transforms, and data loaders.

Example `data/datasets.py`:

```python
from torch.utils.data import DataLoader, TensorDataset
import torch

def make_toy_loaders(batch_size: int = 64):
    x = torch.randn(4096, 20)
    y = (x.sum(dim=1) > 0).long()

    train_ds = TensorDataset(x[:3072], y[:3072])
    val_ds = TensorDataset(x[3072:], y[3072:])

    train_loader = DataLoader(train_ds, batch_size=batch_size, shuffle=True)
    val_loader = DataLoader(val_ds, batch_size=batch_size)

    return train_loader, val_loader
```

For real datasets, this file may include custom `Dataset` classes, image transforms, tokenization, data filtering, and collate functions.

Data code should make the shape and dtype of each batch predictable.

### Training Code

Training code connects the model, data, loss, optimizer, and device.

Example `training/loop.py`:

```python
import torch

def train_one_epoch(model, loader, loss_fn, optimizer, device):
    model.train()

    total_loss = 0.0
    total_examples = 0

    for x, y in loader:
        x = x.to(device)
        y = y.to(device)

        logits = model(x)
        loss = loss_fn(logits, y)

        optimizer.zero_grad(set_to_none=True)
        loss.backward()
        optimizer.step()

        batch_size = x.size(0)
        total_loss += loss.item() * batch_size
        total_examples += batch_size

    return total_loss / total_examples
```

Evaluation should be separate:

```python
import torch

@torch.no_grad()
def evaluate(model, loader, loss_fn, device):
    model.eval()

    total_loss = 0.0
    total_correct = 0
    total_examples = 0

    for x, y in loader:
        x = x.to(device)
        y = y.to(device)

        logits = model(x)
        loss = loss_fn(logits, y)

        pred = logits.argmax(dim=-1)

        batch_size = x.size(0)
        total_loss += loss.item() * batch_size
        total_correct += (pred == y).sum().item()
        total_examples += batch_size

    return {
        "loss": total_loss / total_examples,
        "accuracy": total_correct / total_examples,
    }
```

The training loop mutates parameters. The evaluation loop does not.

### Script Entry Point

A script should assemble the components.

Example `scripts/train_toy.py`:

```python
import torch
from torch import nn

from dlproject.data.datasets import make_toy_loaders
from dlproject.models.mlp import MLP
from dlproject.training.loop import train_one_epoch, evaluate

def main():
    device = (
        "cuda" if torch.cuda.is_available()
        else "mps" if torch.backends.mps.is_available()
        else "cpu"
    )

    train_loader, val_loader = make_toy_loaders(batch_size=64)

    model = MLP(
        in_features=20,
        hidden_features=64,
        out_features=2,
    ).to(device)

    loss_fn = nn.CrossEntropyLoss()
    optimizer = torch.optim.AdamW(model.parameters(), lr=1e-3)

    for epoch in range(10):
        train_loss = train_one_epoch(
            model=model,
            loader=train_loader,
            loss_fn=loss_fn,
            optimizer=optimizer,
            device=device,
        )

        val_metrics = evaluate(
            model=model,
            loader=val_loader,
            loss_fn=loss_fn,
            device=device,
        )

        print(
            f"epoch={epoch} "
            f"train_loss={train_loss:.4f} "
            f"val_loss={val_metrics['loss']:.4f} "
            f"val_acc={val_metrics['accuracy']:.4f}"
        )

if __name__ == "__main__":
    main()
```

The script should be thin. It should configure and call library code rather than contain all logic inline.

### Configuration

Hard-coding every value in Python makes experiments hard to reproduce. Common settings should be stored in a configuration file or command-line arguments.

Example YAML configuration:

```yaml
seed: 1234

data:
  batch_size: 64

model:
  in_features: 20
  hidden_features: 64
  out_features: 2

training:
  epochs: 10
  learning_rate: 0.001
  weight_decay: 0.01
```

A simple configuration loader:

```python
import yaml

def load_config(path: str):
    with open(path, "r") as f:
        return yaml.safe_load(f)
```

A configuration should capture the settings needed to reproduce a run: dataset, model shape, optimizer, learning rate, batch size, epochs, random seed, and relevant preprocessing.

### Checkpoints

A checkpoint stores the training state so that training can resume or a model can be evaluated later.

A useful checkpoint includes:

| Field | Meaning |
|---|---|
| `model` | Model parameters |
| `optimizer` | Optimizer state |
| `epoch` | Current epoch |
| `config` | Experiment configuration |
| `metrics` | Last known metrics |

Example:

```python
def save_checkpoint(path, model, optimizer, epoch, config, metrics):
    torch.save(
        {
            "model": model.state_dict(),
            "optimizer": optimizer.state_dict(),
            "epoch": epoch,
            "config": config,
            "metrics": metrics,
        },
        path,
    )
```

Loading:

```python
def load_checkpoint(path, model, optimizer=None, map_location="cpu"):
    checkpoint = torch.load(path, map_location=map_location)
    model.load_state_dict(checkpoint["model"])

    if optimizer is not None and "optimizer" in checkpoint:
        optimizer.load_state_dict(checkpoint["optimizer"])

    return checkpoint
```

For inference, the model state is often enough. For resuming training, the optimizer state is necessary.

### Logging

A training run should record losses, metrics, configuration, and checkpoint paths. For small projects, printing logs may be enough.

```python
print(f"epoch={epoch} train_loss={train_loss:.4f} val_acc={val_acc:.4f}")
```

For larger projects, logs should be structured.

```python
import json

def log_json(record):
    print(json.dumps(record, sort_keys=True))
```

Example record:

```python
log_json({
    "epoch": epoch,
    "train_loss": train_loss,
    "val_loss": val_metrics["loss"],
    "val_accuracy": val_metrics["accuracy"],
})
```

Structured logs are easier to parse, compare, and store.

TensorBoard can also be used:

```python
from torch.utils.tensorboard import SummaryWriter

writer = SummaryWriter("runs/toy")

writer.add_scalar("loss/train", train_loss, epoch)
writer.add_scalar("loss/val", val_metrics["loss"], epoch)
writer.add_scalar("accuracy/val", val_metrics["accuracy"], epoch)
```

### Reproducibility Utilities

A project should centralize seed setup.

```python
import random
import numpy as np
import torch

def seed_everything(seed: int):
    random.seed(seed)
    np.random.seed(seed)
    torch.manual_seed(seed)

    if torch.cuda.is_available():
        torch.cuda.manual_seed_all(seed)
```

Use it at the start of the script:

```python
seed_everything(config["seed"])
```

Reproducibility also depends on library versions, hardware, data ordering, and nondeterministic kernels. A seed helps, but it does not guarantee identical results across every machine.

### Shape Tests

Deep learning projects benefit from small tests that check tensor shapes.

Example `tests/test_shapes.py`:

```python
import torch

from dlproject.models.mlp import MLP

def test_mlp_shape():
    model = MLP(20, 64, 2)
    x = torch.randn(8, 20)

    y = model(x)

    assert y.shape == (8, 2)
```

Shape tests catch many errors before long training runs begin.

A training step test is also useful:

```python
from torch import nn

def test_training_step_runs():
    model = MLP(20, 64, 2)
    loss_fn = nn.CrossEntropyLoss()
    optimizer = torch.optim.AdamW(model.parameters(), lr=1e-3)

    x = torch.randn(8, 20)
    y = torch.randint(0, 2, (8,))

    logits = model(x)
    loss = loss_fn(logits, y)

    optimizer.zero_grad(set_to_none=True)
    loss.backward()
    optimizer.step()

    assert torch.isfinite(loss)
```

These tests do not prove the model is good. They verify that the code path works.

### Device Utilities

Device selection should be shared rather than copied into every script.

```python
import torch

def get_default_device():
    if torch.cuda.is_available():
        return torch.device("cuda")
    if torch.backends.mps.is_available():
        return torch.device("mps")
    return torch.device("cpu")
```

A recursive batch mover is also useful:

```python
def move_to_device(x, device):
    if torch.is_tensor(x):
        return x.to(device)
    if isinstance(x, dict):
        return {k: move_to_device(v, device) for k, v in x.items()}
    if isinstance(x, list):
        return [move_to_device(v, device) for v in x]
    if isinstance(x, tuple):
        return tuple(move_to_device(v, device) for v in x)
    return x
```

This helps with batches that contain dictionaries, masks, token IDs, labels, and metadata.

### Configuration Versus Code

Configuration should describe choices. Code should define behavior.

Good configuration values:

| Good config field | Reason |
|---|---|
| `learning_rate` | Experiment choice |
| `batch_size` | Experiment choice |
| `hidden_features` | Model size choice |
| `epochs` | Training choice |
| `dataset_path` | Data choice |

Avoid putting complex logic into configuration. When configuration becomes a programming language, debugging becomes harder. Keep configuration simple and explicit.

### Common Project Mistakes

Several project structure mistakes appear often.

| Mistake | Consequence |
|---|---|
| Putting all logic in one script | Hard to reuse and test |
| Mixing training logic into model code | Hard to evaluate or deploy |
| Hard-coding paths | Breaks on other machines |
| Ignoring seeds and versions | Runs become hard to reproduce |
| Saving only model weights | Training cannot resume exactly |
| No validation loop | Overfitting goes unnoticed |
| No shape tests | Errors appear late |
| Silent device assumptions | CPU and GPU behavior diverge |

These are engineering problems, not mathematical problems. They still determine whether experiments are reliable.

### A Complete Small Skeleton

A compact but clean project can be organized around a few functions:

```python
import torch
from torch import nn
from torch.utils.data import DataLoader, TensorDataset

def make_loaders(batch_size):
    x = torch.randn(4096, 20)
    y = (x.sum(dim=1) > 0).long()

    train_ds = TensorDataset(x[:3072], y[:3072])
    val_ds = TensorDataset(x[3072:], y[3072:])

    return (
        DataLoader(train_ds, batch_size=batch_size, shuffle=True),
        DataLoader(val_ds, batch_size=batch_size),
    )

def make_model():
    return nn.Sequential(
        nn.Linear(20, 64),
        nn.ReLU(),
        nn.Linear(64, 2),
    )

def train_one_epoch(model, loader, loss_fn, optimizer, device):
    model.train()
    total = 0.0
    count = 0

    for x, y in loader:
        x = x.to(device)
        y = y.to(device)

        loss = loss_fn(model(x), y)

        optimizer.zero_grad(set_to_none=True)
        loss.backward()
        optimizer.step()

        total += loss.item() * x.size(0)
        count += x.size(0)

    return total / count

@torch.no_grad()
def evaluate(model, loader, loss_fn, device):
    model.eval()
    total = 0.0
    correct = 0
    count = 0

    for x, y in loader:
        x = x.to(device)
        y = y.to(device)

        logits = model(x)
        loss = loss_fn(logits, y)
        pred = logits.argmax(dim=-1)

        total += loss.item() * x.size(0)
        correct += (pred == y).sum().item()
        count += x.size(0)

    return total / count, correct / count

def main():
    device = (
        "cuda" if torch.cuda.is_available()
        else "mps" if torch.backends.mps.is_available()
        else "cpu"
    )

    train_loader, val_loader = make_loaders(batch_size=64)
    model = make_model().to(device)

    loss_fn = nn.CrossEntropyLoss()
    optimizer = torch.optim.AdamW(model.parameters(), lr=1e-3)

    for epoch in range(10):
        train_loss = train_one_epoch(model, train_loader, loss_fn, optimizer, device)
        val_loss, val_acc = evaluate(model, val_loader, loss_fn, device)

        print(
            f"epoch={epoch} "
            f"train_loss={train_loss:.4f} "
            f"val_loss={val_loss:.4f} "
            f"val_acc={val_acc:.4f}"
        )

if __name__ == "__main__":
    main()
```

This can later be split into `data.py`, `model.py`, `training.py`, and `train.py`.

### Summary

A PyTorch project should make the core workflow explicit: load data, build a model, compute loss, backpropagate gradients, update parameters, evaluate metrics, and save checkpoints.

Good structure separates model definitions, data loading, training loops, evaluation, configuration, logging, and tests. This separation reduces accidental complexity. It also makes experiments easier to reproduce, compare, debug, and deploy.
