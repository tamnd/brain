## Conditionals

Real programs contain branches.

A mathematical function may be written as a single expression:

$$
f(x)=x^2+3x+1.
$$

A program rarely looks like this internally. Instead, programs contain conditional execution:

```text
if x > 0:
    y = x * x
else:
    y = -x
```

Automatic differentiation must define how derivatives behave when execution depends on runtime conditions.

This creates several challenges:

| Challenge | Description |
|---|---|
| branch selection | derivative depends on executed path |
| discontinuity | derivative may jump at boundaries |
| inactive branches | unexecuted code contributes nothing locally |
| dynamic graphs | graph structure depends on data |
| control dependence | derivatives depend on branch predicates |
| non-smoothness | derivative may not exist |

Conditionals are where differentiation stops being purely algebraic and becomes semantic.

## Programs with Piecewise Definitions

A conditional program defines a piecewise function.

Example:

$$
f(x) =
\begin{cases}
x^2 & x > 0 \\
-x & x \le 0
\end{cases}
$$

The derivative is:

$$
f'(x) =
\begin{cases}
2x & x > 0 \\
-1 & x < 0
\end{cases}
$$

At

$$
x=0,
$$

the derivative is undefined because the one-sided derivatives disagree:

$$
\lim_{x\to0^+}f'(x)=0,
\quad
\lim_{x\to0^-}f'(x)=-1.
$$

Automatic differentiation does not repair non-differentiability. It propagates derivatives through the executed branch.

## Execution Trace Semantics

AD differentiates the executed computation trace.

Suppose:

```text
if x > 0:
    y = x * x
else:
    y = -x
```

If

$$
x=2,
$$

the executed trace is:

```text
y = x * x
```

Reverse mode differentiates this path and produces:

$$
\frac{dy}{dx}=2x=4.
$$

If

$$
x=-2,
$$

the executed trace is:

```text
y = -x
```

and the derivative becomes:

$$
\frac{dy}{dx}=-1.
$$

The inactive branch contributes nothing locally.

This is an important principle:

AD differentiates the realized execution, not all possible executions simultaneously.

## Computational Graphs with Branches

In static expression graphs, structure is fixed.

Conditionals create dynamic graph structure.

Example:

```text
if cond:
    y = f(x)
else:
    y = g(x)
```

At runtime, only one subgraph becomes active.

The effective graph is:

$$
x \to f \to y
$$

or:

$$
x \to g \to y.
$$

The other branch is absent from the local computation trace.

Dynamic graph systems construct the graph during execution, naturally supporting this behavior.

Static graph systems must represent branching explicitly in the IR.

## Branch Predicates

The branch condition itself may depend on differentiable values.

Example:

```text
if x > 0:
    y = x * x
else:
    y = x
```

The predicate:

$$
x>0
$$

depends on $x$.

But the comparison operation is discontinuous:

$$
\mathbf{1}_{x>0}.
$$

Its derivative is undefined at the threshold and zero almost everywhere else.

Most AD systems therefore treat predicates as non-differentiable control operations.

The derivative propagates only through the selected branch output.

## Local Differentiability

Conditionals often create functions that are differentiable almost everywhere but not globally smooth.

Example:

$$
f(x)=\max(x,0).
$$

This is the ReLU function used in neural networks.

Its derivative is:

$$
f'(x) =
\begin{cases}
1 & x>0 \\
0 & x<0
\end{cases}
$$

At:

$$
x=0,
$$

the derivative is undefined.

Yet AD systems successfully train neural networks using this function.

Why?

Because optimization almost always operates away from the exact non-smooth boundary.

AD computes derivatives on the realized branch and proceeds.

## Subgradients

Some non-smooth functions admit generalized derivatives.

For convex functions, the subgradient generalizes the derivative.

For:

$$
f(x)=|x|,
$$

the derivative is:

$$
f'(x)=
\begin{cases}
1 & x>0 \\
-1 & x<0
\end{cases}
$$

At:

$$
x=0,
$$

the subgradient set is:

$$
[-1,1].
$$

Many ML systems define an arbitrary representative value at non-differentiable points.

For ReLU:

| Point | Common AD convention |
|---|---|
| $x>0$ | derivative $1$ |
| $x<0$ | derivative $0$ |
| $x=0$ | usually $0$ |

This is a pragmatic convention rather than a mathematical theorem.

## Reverse Mode Through Conditionals

Reverse mode traverses the executed branch backward.

Suppose:

```text
if x > 0:
    y = x * x
else:
    y = -x
```

During the forward pass, the runtime records which branch executed.

The backward pass replays only that branch.

If:

$$
x>0,
$$

reverse mode propagates:

$$
\bar{x} += 2x\bar{y}.
$$

Otherwise:

$$
\bar{x} += -\bar{y}.
$$

The inactive branch contributes no adjoints.

The branch decision becomes part of the tape or execution trace.

## Differentiating Both Branches

Sometimes one wants derivatives from both branches.

This occurs in:

| Area | Motivation |
|---|---|
| probabilistic programming | expectation over branches |
| differentiable rendering | visibility discontinuities |
| program analysis | symbolic path sensitivity |
| control optimization | switching systems |

Ordinary AD does not do this automatically.

Standard AD computes derivatives along one realized execution path.

Analyzing all paths requires symbolic execution or specialized semantics.

## Smooth Approximations

A common engineering technique replaces hard conditionals with smooth approximations.

Example:

$$
\max(x,0)
$$

can be approximated by:

$$
\log(1+e^x).
$$

This is the softplus function.

Its derivative is smooth:

$$
\frac{d}{dx}\log(1+e^x) =
\frac{1}{1+e^{-x}}.
$$

Similarly:

| Hard operation | Smooth approximation |
|---|---|
| step function | sigmoid |
| max | softmax |
| argmax | softargmax |
| indicator | temperature-scaled sigmoid |

This avoids discontinuous derivatives and improves optimization behavior.

## Conditional Probability Paths

In probabilistic systems, conditionals may represent stochastic branching.

Example:

```text
if sample() < p:
    ...
```

The branch itself becomes random.

Differentiating expectations over such programs is more complicated because the branch probability depends on parameters.

Methods include:

| Method | Idea |
|---|---|
| score-function estimator | differentiate probability weights |
| reparameterization | move randomness outside |
| relaxed sampling | smooth discrete decisions |
| straight-through estimator | approximate backward pass |

Ordinary AD alone is insufficient here.

## Static Graph Representation

Static graph systems represent conditionals explicitly.

Example IR:

```text
cond(pred, true_branch, false_branch)
```

The AD transform differentiates both branch bodies symbolically.

At runtime:

1. evaluate predicate,
2. choose branch,
3. run corresponding derivative code.

This requires branch-compatible types and shapes.

Tensor systems often require both branches to produce outputs with matching structure.

## Dataflow vs Control Flow

Conditionals introduce control dependence.

Ordinary arithmetic edges represent dataflow:

$$
x \to y.
$$

Conditionals additionally create control structure:

$$
\text{predicate} \to \text{branch selection}.
$$

AD systems must decide:

| Question | Consequence |
|---|---|
| is predicate differentiable? | gradient semantics |
| is branch static? | compile-time specialization |
| can branches allocate state? | tape correctness |
| can branch shapes differ? | graph compatibility |

Modern differentiable languages increasingly model control flow explicitly rather than treating it as a special case.

## Nested Conditionals

Nested branches create piecewise partitions of the input space.

Example:

```text
if x > 0:
    if y > 0:
        ...
    else:
        ...
else:
    ...
```

Each region may have a different derivative rule.

The program becomes a piecewise-smooth manifold.

AD follows the local region determined by execution.

This creates derivative discontinuities at region boundaries.

## Higher-Order Derivatives

Conditionals complicate higher-order derivatives further.

Suppose:

$$
f(x)=|x|.
$$

First derivative:

$$
f'(x)=
\begin{cases}
1 & x>0 \\
-1 & x<0
\end{cases}
$$

Second derivative:

$$
f''(x)=0
$$

away from zero, but undefined at zero.

Higher-order AD systems must decide:

| Situation | Possible behavior |
|---|---|
| undefined derivative | error |
| generalized derivative | subgradient |
| arbitrary convention | fixed value |
| symbolic distribution | theoretical treatment |

Most practical systems use simple conventions.

## Branching and Optimization

Branch discontinuities can destabilize optimization.

Problems include:

| Issue | Cause |
|---|---|
| exploding gradients | abrupt branch change |
| unstable Hessians | derivative jumps |
| optimization stagnation | flat inactive regions |
| non-determinism | branch sensitivity |
| gradient noise | threshold switching |

Smooth relaxations are often introduced specifically to improve optimization stability.

## Differentiable Programming Perspective

Conditionals reveal an important conceptual point.

Programs are not merely algebraic expressions. They are dynamic computational processes.

Differentiating such processes requires semantics for:

| Semantic object | Needed for AD |
|---|---|
| control flow | branch differentiation |
| runtime state | tape correctness |
| execution trace | local derivative path |
| discontinuity | generalized gradients |
| dynamic graph structure | nested transforms |

This is why differentiable programming increasingly studies language semantics rather than only numerical calculus.

## Practical Rules

Most AD systems follow these operational rules:

| Rule | Meaning |
|---|---|
| differentiate executed path only | local execution semantics |
| inactive branches contribute nothing | trace-local differentiation |
| predicates are non-differentiable | control treated discretely |
| branch boundaries may be undefined | no automatic smoothing |
| user may define conventions | custom gradients/subgradients |

These conventions work surprisingly well in practice, especially in machine learning.

## Conceptual Perspective

Conditionals turn smooth computation into piecewise computation.

Automatic differentiation therefore becomes local:

$$
\text{differentiate the branch that actually executed}.
$$

The derivative no longer describes one globally smooth function. Instead it describes the local geometry of a specific execution region within a piecewise computational structure.
