## Source Transformation

Source transformation is an approach to automatic differentiation in which a program is transformed into a new program that computes derivatives. Instead of executing operations immediately and recording them dynamically, the AD system rewrites the original computation into differentiated code ahead of time or during compilation.

The transformed program is itself an ordinary program. It can be compiled, optimized, analyzed, and executed independently.

This approach treats differentiation as a program transformation problem.

## Core Idea

Suppose a program computes:

$$
y = f(x).
$$

Source transformation constructs another program:

$$
(x, \dot{x})
\mapsto
(y, \dot{y}),
$$

for forward mode, or:

$$
(x, \bar{y})
\mapsto
(y, \bar{x}),
$$

for reverse mode.

The transformed program explicitly propagates derivatives.

Unlike tracing systems, source transformation works at the level of syntax, intermediate representation, or compiler IR.

## Example: Scalar Program

Original program:

```python
def f(x):
    a = x * x
    b = sin(a)
    return b
```

Mathematically:

$$
a = x^2,
\qquad
b = \sin(a).
$$

Forward-mode transformed program:

```python
def f_forward(x, dx):
    a = x * x
    da = 2 * x * dx

    b = sin(a)
    db = cos(a) * da

    return b, db
```

The derivative propagation has become explicit program logic.

## Reverse-Mode Transformation

Reverse mode introduces adjoint variables.

Transformed program:

```python
def f_reverse(x):
    a = x * x
    b = sin(a)

    db = 1.0
    da = cos(a) * db
    dx = 2 * x * da

    return b, dx
```

The backward pass is synthesized directly into the transformed program.

In practice, reverse mode usually requires saved intermediates from the forward pass.

## Programs as Computational Graphs

A straight-line program corresponds to a computation graph.

Example:

```python
a = x * y
b = sin(a)
c = b + z
```

Graph:

```text
x ---\
      * --> a --> sin --> b ---\
y ---/                          +
                                 --> c
z -----------------------------/
```

Source transformation rewrites this graph structurally.

Each primitive operation is replaced with differentiated code templates.

## Static vs Dynamic Transformation

Source transformation may occur:

| Stage | Description |
|---|---|
| Ahead-of-time | Before execution |
| JIT-time | During compilation |
| IR-level | On compiler intermediate representation |
| AST-level | On syntax trees |
| Bytecode-level | On VM instructions |

Different AD systems choose different transformation layers.

## AST Transformation

Some systems transform abstract syntax trees directly.

Example:

Original AST:

```text
Mul(x, x)
```

Forward-transformed AST:

```text
Pair(
    Mul(x, x),
    Add(
        Mul(dx, x),
        Mul(x, dx)
    )
)
```

The transformed syntax tree explicitly represents primal and derivative computation.

AST transformation preserves high-level program structure.

## Intermediate Representation Transformation

Modern systems often transform compiler IR instead of source syntax.

Advantages:

| Benefit | Reason |
|---|---|
| Language independence | Common IR |
| Optimization reuse | Existing compiler passes |
| Type information | Explicit in IR |
| Control flow analysis | Easier |
| SSA integration | Natural |

Examples:

| System | IR |
|---|---|
| Enzyme | LLVM IR |
| JAX/XLA | HLO |
| Swift AD | SIL |
| MLIR systems | MLIR dialects |

IR-level AD increasingly dominates production systems.

## Forward-Mode Transformation

Forward mode augments each variable with a tangent.

Original:

```python
y = x * z
```

Transformed:

```python
y  = x * z
dy = dx * z + x * dz
```

Each primal variable receives a tangent companion.

Conceptually:

$$
x \to (x, \dot{x}).
$$

This is equivalent to dual-number propagation.

## Reverse-Mode Transformation

Reverse mode is more complex.

It introduces:

1. Forward computation.
2. Intermediate storage.
3. Backward accumulation.

Example:

Original:

```python
a = x * y
b = sin(a)
```

Transformed:

```python
a = x * y
b = sin(a)

db = 1
da = cos(a) * db

dx = y * da
dy = x * da
```

Reverse transformation requires dependency ordering.

The backward pass executes in reverse topological order.

## Wengert Lists

One classical representation is the Wengert list.

Original program:

```text
v1 = x
v2 = y
v3 = v1 * v2
v4 = sin(v3)
```

Reverse pass:

```text
adj[v4] += 1
adj[v3] += cos(v3) * adj[v4]
adj[v2] += v1 * adj[v3]
adj[v1] += v2 * adj[v3]
```

Source transformation can generate these adjoint updates statically.

## SSA Form

Modern compiler IRs often use Static Single Assignment (SSA) form.

Example:

```text
v1 = x * y
v2 = sin(v1)
v3 = v2 + z
```

Each variable is assigned exactly once.

SSA is useful because:

| Property | AD Benefit |
|---|---|
| Explicit dependencies | Easier reverse ordering |
| Immutable values | Simpler adjoint logic |
| Dominator structure | Control-flow analysis |
| Optimization compatibility | Compiler reuse |

Many modern AD systems are effectively SSA transformations.

## Control Flow

Control flow complicates source transformation.

Example:

```python
if x > 0:
    y = x * x
else:
    y = -x
```

The transformed derivative code must preserve branch structure.

Forward mode:

```python
if x > 0:
    y  = x * x
    dy = 2 * x * dx
else:
    y  = -x
    dy = -dx
```

Reverse mode must replay or reconstruct the executed branch.

## Loops

Loops create repeated derivative structure.

Example:

```python
y = x
for i in range(n):
    y = sin(y)
```

Forward mode propagates tangents iteratively.

Reverse mode must reverse the loop.

Generated reverse pass:

```python
for i in reversed(range(n)):
    ...
```

Intermediates from each iteration may need storage.

This creates major memory costs.

## Tape vs Recompute

Reverse-mode transformation often faces a choice:

| Strategy | Memory | Compute |
|---|---:|---:|
| Store intermediates | High | Low |
| Recompute intermediates | Low | High |

Static source transformation can optimize this tradeoff.

Checkpointing strategies may be inserted automatically.

## Closure Conversion

Functional languages complicate AD because functions may capture variables.

Example:

```python
def outer(x):
    def inner(y):
        return x * y
    return inner
```

The transformed derivative program must preserve closure environments.

Differentiable functional languages often require:

```text
closure conversion
lambda lifting
environment differentiation
```

These are compiler problems, not merely calculus problems.

## Mutation

Mutation complicates reverse mode.

Example:

```python
x += y
```

Now the old value of $x$ is overwritten.

Reverse mode may require the previous value.

Possible strategies:

| Strategy | Description |
|---|---|
| Functionalization | Convert mutation to immutable form |
| Versioning | Store old values |
| Tape logging | Record overwritten state |

Many AD systems internally eliminate mutation before differentiation.

## Aliasing

Aliasing means multiple names refer to the same storage.

Example:

```python
a = x
b = x
```

Updating gradients correctly requires understanding alias relationships.

View operations complicate this further:

```python
y = x.transpose()
```

The transformed program must preserve storage semantics.

## Higher-Order Differentiation

Source transformation composes naturally.

Transform once:

$$
f \to Df.
$$

Transform again:

$$
Df \to D^2f.
$$

This enables higher-order derivatives.

However, transformed code can grow rapidly.

Naive repeated transformation may create:

| Problem | Cause |
|---|---|
| Code explosion | Nested derivative logic |
| Redundant computation | Repeated expansion |
| Huge IR graphs | Higher-order composition |

Efficient higher-order AD requires simplification and optimization passes.

## Optimization Opportunities

Source transformation exposes derivative computation to compiler optimization.

Possible optimizations:

| Optimization | Effect |
|---|---|
| Dead-code elimination | Remove unused derivatives |
| Common subexpression elimination | Share repeated terms |
| Loop fusion | Reduce memory traffic |
| Constant folding | Simplify expressions |
| Algebraic simplification | Reduce arithmetic |
| Kernel fusion | GPU efficiency |

Tracing systems often lose some of this static structure.

## Symbolic Simplification

Source transformation resembles symbolic differentiation, but with program structure preserved.

Example:

Naive derivative:

$$
(x^2)' = x + x.
$$

Simplified:

$$
2x.
$$

Compiler optimization can automatically simplify generated derivative code.

This improves performance significantly.

## Activity Analysis

Not all variables influence outputs.

Example:

```python
a = x * y
b = 5
c = a + 1
```

Variable $b$ is inactive.

Activity analysis determines:

```text
which variables need derivatives
which intermediates require storage
which paths affect outputs
```

This is essential for efficient transformed programs.

## Reverse Accumulation

Reverse mode generates accumulation logic.

Example:

```python
adj[x] += local_grad
```

Accumulation is necessary because multiple paths may contribute to the same variable.

Source transformation must preserve this additive semantics exactly.

## Memory Planning

Static transformation enables memory scheduling.

The compiler can determine:

```text
tensor lifetimes
reuse opportunities
buffer aliasing
activation checkpointing
```

This is much harder in purely dynamic systems.

## Differentiating Entire Programs

Source transformation can differentiate:

| Program Feature | Possible? |
|---|---|
| Loops | Yes |
| Recursion | Yes |
| Functions | Yes |
| Closures | Yes |
| Data structures | Sometimes |
| Dynamic dispatch | Difficult |
| Mutation | Difficult but possible |

The more dynamic the language, the harder transformation becomes.

## Source Transformation vs Operator Overloading

| Source Transformation | Operator Overloading |
|---|---|
| Compile-time rewriting | Runtime tracing |
| Global program view | Local operation view |
| Better optimization potential | Simpler implementation |
| Handles control flow structurally | Control flow executed dynamically |
| Complex compiler integration | Easier library design |
| Often faster | Often more flexible |

Modern systems increasingly combine both approaches.

## Real Systems

Examples of source-transformation AD:

| System | Approach |
|---|---|
| Tapenade | Source-to-source Fortran/C |
| Enzyme | LLVM IR transformation |
| Swift AD | Compiler-integrated |
| JAX | Staged tracing + IR transforms |
| Zygote | Source-level Julia transformation |

The field has gradually moved from source-level rewriting toward IR-level compiler integration.

## Correctness

A source-transformed program should satisfy:

$$
\text{Transformed Program} =
\text{Derivative of Original Program}.
$$

Correctness proofs often proceed by induction on syntax or IR structure.

Primitive derivative rules provide local correctness.

Transformation composition provides global correctness.

## Challenges

Source transformation faces major difficulties:

| Challenge | Reason |
|---|---|
| Mutation | Overwritten state |
| Aliasing | Shared storage |
| Exceptions | Nonlocal control flow |
| Dynamic typing | Unknown structure |
| Reflection | Runtime program changes |
| Recursion | Reverse scheduling |
| Memory growth | Reverse-mode storage |

Compiler-quality infrastructure is often required.

## Compiler Perspective

From a compiler viewpoint, AD is:

$$
\text{Program} \to \text{Differentiated Program}.
$$

The transformed program is another IR graph subject to ordinary optimization.

Thus AD increasingly belongs to compiler engineering rather than purely numerical analysis.

## Summary

Source transformation performs automatic differentiation by rewriting programs into derivative programs. Forward mode augments variables with tangents. Reverse mode synthesizes adjoint computations and backward passes.

The transformation may occur at the source, AST, IR, or compiler level. Modern systems increasingly operate on SSA-style IRs because they expose dependencies, control flow, and optimization opportunities clearly.

Source transformation turns differentiation into a compiler problem: dependency analysis, activity analysis, memory planning, mutation handling, and optimization become central concerns.
