Modern automatic differentiation systems appear complex:
- tensor runtimes,
- graph compilers,
- tracing systems,
- GPU kernels,
- distributed execution,
- higher-order transformations.

However, the core mechanisms are surprisingly small.

A minimal AD engine needs only:
- a representation of values,
- local derivative rules,
- compositional propagation,
- graph or tape structure.

This chapter constructs AD systems from first principles. The focus is not framework APIs or large-scale optimization. The goal is understanding the minimal machinery required to build:
- forward-mode engines,
- reverse-mode engines,
- differentiable execution graphs,
- gradient propagation systems.

The central idea is simple:

A differentiable program is an ordinary computation augmented with derivative propagation rules.

## Minimal Forward Mode Engine

Forward mode is the simplest form of automatic differentiation.

A primal value:

$$
x
$$

is augmented with a tangent value:

$$
\dot x.
$$

The engine propagates both simultaneously.

For a function:

$$
f : \mathbb{R}^n \to \mathbb{R}^m,
$$

forward mode computes:

$$
(f(x),Df(x)v).
$$

The derivative information flows in the same direction as ordinary execution.

## The Core Representation

A forward-mode engine can represent each scalar as:

$$
x + \epsilon \dot x,
\quad
\epsilon^2=0.
$$

Operationally this becomes a pair:

```text
(value, tangent)
```

Example:

```text
x = (3.0, 1.0)
```

means:
- primal value $3$,
- perturbation $1$.

If $x$ is the differentiation variable, its tangent is seeded with $1$.

Other independent variables receive tangent $0$.

## Primitive Operations

Each primitive operation defines:
- primal evaluation,
- tangent propagation.

### Addition

For:

$$
z=x+y,
$$

the derivative rule is:

$$
\dot z=\dot x+\dot y.
$$

Implementation:

```go
type Dual struct {
    Value   float64
    Tangent float64
}

func Add(a, b Dual) Dual {
    return Dual{
        Value:   a.Value + b.Value,
        Tangent: a.Tangent + b.Tangent,
    }
}
```

## Elementary Functions

For:

$$
z=\sin(x),
$$

the derivative rule is:

$$
\dot z=\cos(x)\dot x.
$$

Implementation:

```go
func Sin(a Dual) Dual {
    return Dual{
        Value:   math.Sin(a.Value),
        Tangent: math.Cos(a.Value) * a.Tangent,
    }
}
```

Each primitive operator carries:
- primal semantics,
- differential semantics.

Complex programs inherit differentiation automatically through composition.

## Example Evaluation

Suppose:

$$
f(x)=x^2+\sin(x).
$$

We evaluate at:

$$
x=2.
$$

Seed:

```go
x := Dual{
    Value:   2,
    Tangent: 1,
}
```

Compute:

```go
y := Add(Mul(x, x), Sin(x))
```

The engine propagates:
- primal values,
- tangent values,

simultaneously.

Result:
- `y.Value = f(2)`
- `y.Tangent = f'(2)`

No symbolic algebra appears. No numerical approximation appears. The chain rule emerges from local propagation rules.

## Multi-Variable Functions

For:

$$
f(x,y),
$$

different derivatives are obtained by different seeds.

To compute:

$$
\frac{\partial f}{\partial x},
$$

seed:

```text
x.tangent = 1
y.tangent = 0
```

To compute:

$$
\frac{\partial f}{\partial y},
$$

seed:

```text
x.tangent = 0
y.tangent = 1
```

Forward mode therefore computes directional derivatives.

The full Jacobian requires multiple passes.

## Jacobian-Vector Products

More generally, seed an arbitrary vector:

$$
v=(v_1,\ldots,v_n).
$$

The propagated tangent computes:

$$
Df(x)v.
$$

Forward mode naturally computes Jacobian-vector products.

This is one reason forward mode is efficient when:
- input dimension is small,
- directional derivatives are sufficient,
- higher-order products are required.

## Nested Dual Numbers

Higher-order derivatives can be computed by nesting dual numbers.

Example:

```text
Dual[Dual[float64]]
```

This creates tangent-of-tangent structure.

Operationally:
- outer tangent tracks first derivative,
- inner tangent tracks second derivative.

However, naive nesting introduces:
- perturbation confusion,
- type complexity,
- performance overhead.

Practical systems often use specialized higher-order methods instead.

## Operator Overloading

Forward mode is especially compatible with operator overloading.

Replace ordinary numbers:

```text
float64
```

with:

```text
Dual
```

Then ordinary code becomes differentiable automatically.

Example:

```go
func F(x Dual) Dual {
    return Add(
        Mul(x, x),
        Sin(x),
    )
}
```

This is one reason forward mode is popular in:
- C++,
- Julia,
- operator-overloading systems,
- embedded differentiable DSLs.

## Computational Complexity

Each primitive operation performs:
- primal computation,
- tangent computation.

Cost remains a constant-factor multiple of ordinary execution.

For:

$$
f:\mathbb{R}^n\to\mathbb{R}^m,
$$

forward mode computes:

$$
Df(x)v
$$

in:

$$
O(C(f)).
$$

Full Jacobians require $n$ passes.

Forward mode therefore scales well for:
- small input dimension,
- large output dimension,
- directional sensitivity analysis.

## No Tape Required

Forward mode requires no graph storage and no tape.

Derivative propagation follows evaluation order directly.

Memory overhead is therefore low.

This makes forward mode attractive for:
- embedded systems,
- scientific computing,
- online sensitivity analysis,
- streaming computations.

Reverse mode trades higher memory for cheaper scalar-output gradients.

## Dynamic vs Static Structure

A minimal forward-mode engine can operate:
- dynamically,
- statically.

### Dynamic Systems
Use runtime objects:
- operator overloading,
- boxed values,
- runtime dispatch.

### Static Systems
Transform code ahead of time:
- compiler IR rewriting,
- staged differentiation,
- template specialization.

Both implement the same semantic rules.

## Arrays and Tensors

Scalar dual numbers generalize naturally to tensors.

Each tensor element carries:
- primal value,
- tangent value.

Or more efficiently:
- primal tensor,
- tangent tensor.

Tensor operations then propagate tangents structurally.

Example:

$$
Z=XY.
$$

Forward propagation computes:

$$
\dot Z=\dot X Y + X\dot Y.
$$

This extends ordinary matrix calculus directly.

## Sparse Tangents

For large systems, tangents are often sparse.

Suppose only a few input directions matter.

Then:
- sparse tangent vectors,
- compressed seeds,
- graph coloring,

can dramatically reduce cost.

A minimal engine can support sparse propagation simply by changing tangent representation.

## Functional Interpretation

Semantically, forward mode lifts computations into tangent space.

A function:

$$
f:X\to Y
$$

becomes:

$$
Tf:TX\to TY.
$$

The engine implements this lifting operationally.

Each primitive operation becomes a tangent-aware operator.

Composition preserves the chain rule automatically.

## Limitations of Minimal Forward Mode

A minimal forward engine lacks:
- reverse mode,
- graph optimization,
- checkpointing,
- GPU execution,
- higher-order optimization,
- kernel fusion,
- batching transforms.

However, the essential semantics are complete.

Every industrial AD system still implements this same underlying tangent propagation logic.

## The Essential Structure

A forward-mode AD engine requires only:
- augmented numeric values,
- local derivative rules,
- compositional evaluation.

The chain rule emerges mechanically from propagation.

No symbolic differentiation is required.

No finite differences are required.

Differentiation becomes ordinary program execution over an enriched algebra of values.
