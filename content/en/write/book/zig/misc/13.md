## Part 6. Pointers and Memory

### Memory Model Basics

Memory is where your program stores values while it runs.

Every variable, array, struct, string, and object-like value must live somewhere in memory. Zig makes this visible. It does not hide memory behind a large runtime system. When you write Zig, you often think directly about where data lives, who owns it, and how long it stays valid.

This can feel new if you come from Python, JavaScript, Java, or Go. Those languages manage most memory details for you. Zig gives you more control, so it also asks you to be more precise.

### Values Live in Memory

Consider this simple code:

```zig
const x: i32 = 42;
```

This creates an integer value.

Conceptually, the value `42` is stored somewhere in memory. The name `x` lets your code refer to that value.

Now consider:

```zig
var y: i32 = 100;
```

This creates a mutable integer value. The value can change:

```zig
y = 200;
```

The variable `y` refers to a location where an `i32` is stored. When you assign a new value, Zig writes a new `i32` into that location.

### Addresses

Every memory location has an address.

An address is like a number that identifies where something lives in memory.

You usually do not need to print or manipulate raw addresses directly, but the idea is important. A pointer is a value that stores an address.

```zig
var x: i32 = 42;
const p = &x;
```

The expression `&x` means:

```text
the address of x
```

So `p` points to `x`.

The type of `p` is:

```zig
*i32
```

Read this as:

```text
pointer to i32
```

### Dereferencing a Pointer

A pointer stores an address. To access the value at that address, use `.*`.

```zig
var x: i32 = 42;
const p = &x;

p.* = 99;
```

After this code, `x` is `99`.

The pointer `p` points to `x`, and `p.*` means:

```text
the value stored at the place p points to
```

So this:

```zig
p.* = 99;
```

means:

```text
write 99 into x through the pointer
```

### Stack Memory

Most local variables live on the stack.

Example:

```zig
fn add(a: i32, b: i32) i32 {
    const result = a + b;
    return result;
}
```

The parameters `a` and `b`, and the local variable `result`, are temporary values used while the function runs.

When the function returns, its local stack storage is no longer valid.

This is why you must not return a pointer to a local variable:

```zig
fn badPointer() *i32 {
    var x: i32 = 42;
    return &x;
}
```

This is wrong because `x` belongs to the function call. When the function returns, `x` is gone. The returned pointer would point to memory that is no longer valid.

That kind of pointer is called a dangling pointer.

### Heap Memory

The heap is memory used for values that need to live longer than one function call, or whose size is not known in a simple fixed way.

In Zig, heap allocation is explicit. You normally allocate through an allocator.

```zig
const buffer = try allocator.alloc(u8, 1024);
defer allocator.free(buffer);
```

This asks the allocator for 1024 bytes.

The returned value is a slice:

```zig
[]u8
```

The memory stays valid until you free it:

```zig
allocator.free(buffer);
```

Zig does not automatically free this memory for you. That is your responsibility.

### Stack vs Heap

Stack memory is automatic.

Heap memory is manual.

| Kind | Typical Use | Lifetime | Cleanup |
|---|---|---|---|
| Stack | Local variables, small fixed data | Until function returns | Automatic |
| Heap | Dynamic buffers, long-lived data | Until explicitly freed | Manual |

Use stack memory when the size is known and the value does not need to outlive the function.

Use heap memory when you need dynamic size or longer lifetime.

### Ownership

Ownership answers one question:

```text
who is responsible for cleaning this up?
```

For stack values, ownership is simple. The function owns its local variables, and they disappear automatically when the function returns.

For heap values, ownership must be clear.

Example:

```zig
fn makeBuffer(allocator: std.mem.Allocator) ![]u8 {
    return try allocator.alloc(u8, 1024);
}
```

This function returns allocated memory. The caller owns it.

The caller should free it:

```zig
const buffer = try makeBuffer(allocator);
defer allocator.free(buffer);
```

A good Zig API makes ownership clear.

### Lifetime

Lifetime means how long a value stays valid.

This pointer is valid:

```zig
var x: i32 = 42;
const p = &x;

p.* = 10;
```

Both `x` and `p` are used in the same scope. The pointer does not outlive the value.

This pointer is invalid:

```zig
fn makePointer() *i32 {
    var x: i32 = 42;
    return &x;
}
```

The pointer outlives `x`. That is not safe.

When working with pointers, always ask:

```text
does the pointed-to value still exist here?
```

### Slices Are Pointer Plus Length

A slice is one of the most common Zig types.

```zig
[]u8
```

A slice is not just a pointer. It contains:

```text
a pointer to the first element
a length
```

For example:

```zig
var data = [_]u8{ 10, 20, 30, 40 };
const slice = data[1..3];
```

The slice refers to part of the array:

```text
20, 30
```

Its length is `2`.

Slices are safer than raw pointers because they carry length information. Zig can use that length for bounds checking in safe modes.

### Memory Safety

Memory safety means avoiding mistakes such as:

```text
using memory after it is freed
reading outside an array
writing outside a buffer
returning pointers to dead stack values
freeing the same allocation twice
forgetting to free memory
```

Zig helps with some of these through types and runtime safety checks.

But Zig does not remove your responsibility. If you allocate memory, you must free it. If you store a pointer, you must ensure the pointed-to value is still alive. If you pass a slice, you must understand who owns the backing memory.

### No Garbage Collector

Zig does not use a garbage collector by default.

A garbage collector automatically finds unused heap memory and frees it later. That can make programming easier, but it also adds runtime behavior that may be hard to predict.

Zig chooses explicit allocation instead.

This gives you control over:

```text
when memory is allocated
when memory is freed
which allocator is used
how much memory is used
what happens when allocation fails
```

That control is important for systems programming.

### A Simple Mental Model

For now, keep this model:

A value is stored somewhere.

A pointer stores where a value is.

A slice stores where a sequence starts and how long it is.

Stack memory is temporary and automatic.

Heap memory is manual and must be freed.

Ownership says who must clean up.

Lifetime says how long a value remains valid.

Pointers and memory are central to Zig. You do not need to master everything at once. But you do need to become comfortable asking where data lives and how long it stays valid.
