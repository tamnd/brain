### Networking APIs

Networking means communicating with another program through a network.

The other program may be on the same machine, on another machine in your home network, or on a server across the internet. A browser talking to a website is networking. A database client talking to a database server is networking. A multiplayer game, chat app, proxy, crawler, and API client all use networking.

At the lowest level, network code sends and receives bytes.

Protocols give those bytes meaning.

#### Clients and Servers

Most beginner networking code has two roles.

A client connects to another program.

A server waits for other programs to connect.

For example, when your program calls a web API, it acts as a client.

When your program listens for HTTP requests, it acts as a server.

The usual shape is:

```text
client -> connects -> server
client <- receives <- server
```

The client usually starts the connection.

The server usually listens.

#### IP Addresses and Ports

An IP address identifies a machine or network endpoint.

Examples:

```text
127.0.0.1
192.168.1.10
8.8.8.8
```

The address:

```text
127.0.0.1
```

means the local machine. It is often called localhost.

A port identifies a service on that machine.

Examples:

```text
80    HTTP
443   HTTPS
5432  PostgreSQL
6379  Redis
```

A server usually listens on an address and port:

```text
127.0.0.1:9000
```

This means: listen on the local machine, port `9000`.

#### TCP

TCP is a reliable byte stream protocol.

Reliable means TCP tries to deliver bytes in order, without loss or duplication. If something goes wrong, the connection fails.

Byte stream means TCP does not preserve message boundaries.

If one side writes:

```text
hello
```

and then:

```text
world
```

the other side may read:

```text
helloworld
```

or:

```text
he
```

then:

```text
lloworld
```

The bytes are ordered, but your program must define message boundaries.

This is one of the most important networking lessons.

TCP gives you a stream, not messages.

#### UDP

UDP sends datagrams.

A datagram is a single packet-like message.

UDP does not guarantee delivery. It does not guarantee ordering. It does not retry automatically.

UDP is useful when low latency matters more than perfect delivery, or when the protocol handles reliability itself.

Examples include some games, DNS, voice/video traffic, and custom protocols.

Beginners should usually start with TCP.

#### A TCP Server Shape

A TCP server usually follows this structure:

```text
create listening socket
bind to address and port
listen for connections

loop:
    accept connection
    read bytes
    write bytes
    close connection
```

In Zig, the exact APIs may differ by version, but the resource pattern is stable.

Conceptually:

```zig
const address = try std.net.Address.parseIp("127.0.0.1", 9000);

var server = try address.listen(.{});
defer server.deinit();

while (true) {
    const connection = try server.accept();
    defer connection.stream.close();

    // read and write using connection.stream
}
```

This is the usual server loop.

Accept a connection.

Use the connection.

Close it.

Then accept another connection.

#### A Tiny Echo Server

An echo server sends back whatever the client sends.

Conceptually:

```zig
const std = @import("std");

pub fn main() !void {
    const address = try std.net.Address.parseIp("127.0.0.1", 9000);

    var server = try address.listen(.{});
    defer server.deinit();

    while (true) {
        const connection = try server.accept();
        defer connection.stream.close();

        var buffer: [1024]u8 = undefined;

        const n = try connection.stream.read(&buffer);
        if (n == 0) continue;

        try connection.stream.writeAll(buffer[0..n]);
    }
}
```

This code shows the idea.

It listens on:

```text
127.0.0.1:9000
```

It accepts one connection.

It reads bytes.

It writes the same bytes back.

It closes the connection.

Real servers usually handle multiple clients, repeated reads, timeouts, errors, and shutdown. This small version teaches the core shape.

#### Testing with Netcat

On many systems, you can test a TCP server with `nc`.

Run the Zig server in one terminal.

In another terminal:

```bash
nc 127.0.0.1 9000
```

Then type:

```text
hello
```

The server should echo it back.

This is a useful way to test raw TCP programs before adding a larger protocol.

#### A TCP Client Shape

A TCP client usually follows this structure:

```text
resolve or parse address
connect
write request bytes
read response bytes
close connection
```

Conceptually:

```zig
const address = try std.net.Address.parseIp("127.0.0.1", 9000);

const stream = try std.net.tcpConnectToAddress(address);
defer stream.close();

try stream.writeAll("hello\n");

var buffer: [1024]u8 = undefined;
const n = try stream.read(&buffer);

std.debug.print("{s}\n", .{buffer[0..n]});
```

This connects to the echo server, sends text, reads the response, and prints it.

#### DNS

Humans usually use names, not IP addresses.

Example:

```text
example.com
```

The network needs an address.

DNS translates names into addresses.

A client may connect to:

```text
example.com:80
```

but first it must resolve `example.com` into an IP address.

Networking code often has two steps:

resolve host name

connect to one of the returned addresses

This can fail if DNS is unavailable, the name does not exist, or the network is down.

#### Message Boundaries

TCP is a byte stream, so your protocol must define where one message ends.

Common approaches:

Newline-delimited messages:

```text
hello\n
status\n
quit\n
```

Length-prefixed messages:

```text
[length][bytes]
[length][bytes]
```

Fixed-size messages:

```text
exactly 64 bytes per message
```

Sentinel-terminated messages:

```text
bytes until 0
```

HTTP uses headers and body rules.

Redis uses RESP.

PostgreSQL has its own binary protocol.

Your program must know its protocol.

#### Newline Protocol Example

A very simple protocol can use one line per message.

Client sends:

```text
ping\n
```

Server replies:

```text
pong\n
```

This is easy to debug with terminal tools.

But line protocols need clear rules about encoding, maximum line length, and invalid input.

Never let a client send an unlimited line into memory.

#### Reading in a Loop

One `read` is not enough for most network code.

This is too simple for real protocols:

```zig
const n = try stream.read(&buffer);
```

A proper stream reader usually loops:

```zig
while (true) {
    const n = try stream.read(&buffer);

    if (n == 0) break;

    // process buffer[0..n]
}
```

A read of `0` means the other side closed the connection.

The same idea appeared in file reading. Streams repeat this pattern.

#### Partial Writes

Just as reads can be partial, writes can also be partial.

That is why `writeAll` is useful.

Prefer:

```zig
try stream.writeAll(data);
```

over:

```zig
const n = try stream.write(data);
```

unless you deliberately want to handle partial writes yourself.

For protocols, partial writes are usually not what you want.

#### Timeouts

Network code should use timeouts.

Without timeouts, a program may wait forever for a server that never responds or a client that connects and then stops sending data.

Important timeouts include:

connection timeout

read timeout

write timeout

idle timeout

total request timeout

The exact API depends on the networking layer. The design principle is stable:

A network program should decide how long it is willing to wait.

#### Handling Many Clients

The tiny server above handles one client at a time.

That means while one client is connected, other clients must wait.

Real servers usually need concurrency.

Common models:

one thread per connection

thread pool

event loop

async I/O

nonblocking sockets

The right model depends on the program.

For beginners, first write a single-connection server. Then handle one connection at a time. Then introduce concurrency.

Do not start with a complex server architecture before you understand the byte stream.

#### Localhost Development

Use localhost while learning.

```text
127.0.0.1
```

This avoids external network problems.

A server bound to `127.0.0.1` accepts connections only from the local machine.

A server bound to `0.0.0.0` may accept connections from other machines, depending on firewall and network settings.

Be careful with `0.0.0.0`. It can expose your server beyond your machine.

#### Binary Protocols

A binary protocol sends structured bytes instead of text.

For example, a message might be:

```text
1 byte   message type
4 bytes  payload length
N bytes  payload
```

Binary protocols can be efficient and precise, but they require more care.

You must decide:

endianness

integer sizes

maximum lengths

versioning

invalid message handling

alignment assumptions

Do not write raw structs directly to the network unless the layout is explicitly defined and portable.

#### Text Protocols

A text protocol sends readable text.

Example:

```text
GET / HTTP/1.1
Host: example.com
```

Text protocols are easier to debug. You can inspect them with terminal tools.

They may be larger and slower to parse than binary protocols, but they are often good enough.

For beginner projects, a simple text protocol is usually better.

#### Security

Network input is external input.

Treat it as untrusted.

A client can send invalid bytes.

A client can send too much data.

A client can connect and stop responding.

A client can disconnect halfway through a message.

A client can send messages in the wrong order.

Your server must handle these cases without crashing or leaking resources.

At minimum, enforce limits:

maximum message size

maximum line length

maximum request body size

maximum number of connections

timeouts

#### A Small Protocol Design

Here is a tiny line protocol:

```text
PING
ECHO text
QUIT
```

Responses:

```text
PONG
text
BYE
```

Rules:

Each command ends with `\n`.

A line may not exceed 1024 bytes.

Unknown commands return `ERR`.

This is simple enough to implement, test, and debug.

The important part is that the protocol has rules. Without rules, network programs become ambiguous.

#### Common Mistakes

Do not assume one read equals one message.

Do not ignore partial writes.

Do not accept unlimited input.

Do not expose a test server on all interfaces by accident.

Do not block forever without timeouts.

Do not trust client data.

Do not confuse TCP byte streams with UDP datagrams.

Do not write normal Zig structs directly as network formats without defining layout and byte order.

#### The Core Pattern

For a TCP server:

```zig
const address = try std.net.Address.parseIp("127.0.0.1", 9000);

var server = try address.listen(.{});
defer server.deinit();

while (true) {
    const connection = try server.accept();
    defer connection.stream.close();

    // read request
    // write response
}
```

For a TCP client:

```zig
const address = try std.net.Address.parseIp("127.0.0.1", 9000);

const stream = try std.net.tcpConnectToAddress(address);
defer stream.close();

try stream.writeAll("hello\n");
```

For reading:

```zig
while (true) {
    const n = try stream.read(&buffer);
    if (n == 0) break;

    use(buffer[0..n]);
}
```

#### What You Should Remember

Networking sends and receives bytes.

A client connects.

A server listens.

An IP address identifies a network endpoint.

A port identifies a service.

TCP is a reliable byte stream.

UDP sends datagrams without reliability guarantees.

TCP does not preserve message boundaries.

Your protocol must define where messages begin and end.

Use limits and timeouts.

Treat all network input as untrusted.

Networking in Zig is the same discipline you have already seen: explicit resources, explicit errors, careful buffers, and clear ownership.
