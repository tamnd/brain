### When Not to Use `comptime`

`comptime` is one of Zig’s strongest features, but it should not be used everywhere.

A simple rule works well:

Use `comptime` when the information is part of the program’s structure.

Use runtime code when the information comes from the outside world.

Program structure includes types, array sizes, field names, enum tags, build options, and fixed configuration.

Outside-world information includes user input, files, network data, database rows, environment variables, command-line arguments, and the current time.

#### `comptime` Cannot Know Runtime Data

This does not work:

```zig
fn makeBuffer(comptime size: usize) [size]u8 {
    return undefined;
}

pub fn main() void {
    var size: usize = 1024;
    const buffer = makeBuffer(size);
    _ = buffer;
}
```

`size` is a runtime variable.

Even if it currently holds `1024`, Zig treats it as runtime data because it is stored in a `var` and may change while the program runs.

This works:

```zig
const buffer = makeBuffer(1024);
```

The compiler knows `1024` during compilation.

#### Do Not Use `comptime` for User Input

User input arrives after the program starts.

So this idea is wrong:

```zig
const size = readSizeFromUser();
const Buffer = FixedBuffer(size);
```

A type like `FixedBuffer(size)` must be known while compiling.

User input is known only while running.

Use a runtime allocation instead:

```zig
const buffer = try allocator.alloc(u8, size);
defer allocator.free(buffer);
```

Here, `size` can come from the user because heap allocation happens at runtime.

#### Do Not Use `comptime` Just to Look Clever

This is a bad use of compile-time programming:

```zig
fn add(comptime a: i32, comptime b: i32) i32 {
    return a + b;
}
```

For ordinary addition, this is unnecessary.

Just write:

```zig
fn add(a: i32, b: i32) i32 {
    return a + b;
}
```

The optimizer can already handle simple constants.

Use `comptime` when the type, layout, generated code, or validity of the program depends on the value.

#### Do Not Force Runtime Choices into Compile-Time Choices

Suppose a user chooses a sort mode:

```zig
const mode = readModeFromUser();
sort(mode, items);
```

That should be runtime logic.

Do not design the function like this:

```zig
fn sort(comptime mode: SortMode, items: []i32) void {
    switch (mode) {
        .fast => quickSort(items),
        .stable => mergeSort(items),
    }
}
```

This only works when the caller knows the mode at compile time.

For user input, use a normal parameter:

```zig
fn sort(mode: SortMode, items: []i32) void {
    switch (mode) {
        .fast => quickSort(items),
        .stable => mergeSort(items),
    }
}
```

Runtime data should stay runtime data.

#### Do Not Over-Specialize

Every compile-time specialization can produce more generated code.

For example:

```zig
const A = Parser(.json);
const B = Parser(.toml);
const C = Parser(.yaml);
```

This may be reasonable.

But if you generate hundreds of specialized versions for small differences, your binary may grow. Compilation may also become slower.

Specialization is useful when it removes real overhead or improves correctness. It is wasteful when it only duplicates similar code.

#### Do Not Hide Simple Logic Behind Reflection

Reflection with `@typeInfo` is powerful.

But this is often worse than direct code:

```zig
fn printUser(user: User) void {
    printStruct(User, user);
}
```

If `User` has only two fields and a custom format, direct code may be clearer:

```zig
fn printUser(user: User) void {
    std.debug.print("id = {}, name = {s}\n", .{
        user.id,
        user.name,
    });
}
```

Use reflection when you need a reusable pattern across many types.

Do not use it to avoid writing three clear lines.

#### Do Not Use Compile-Time Code for Large Runtime Work

A compile-time loop runs inside the compiler.

That means it increases compilation cost.

This is fine:

```zig
const table = makeSmallLookupTable();
```

This may be a problem:

```zig
const huge_table = generateMillionsOfEntries();
```

If compile-time generation becomes large, builds become slower and memory-heavy.

Sometimes it is better to load data from a file, generate a separate artifact, or compute it at runtime.

#### Do Not Confuse `const` with `comptime`

`const` means the binding cannot be reassigned.

It does not always mean the value is known at compile time.

This can be compile-time known:

```zig
const x = 123;
```

This is runtime data:

```zig
const x = readNumberFromUser();
```

Both use `const`, but only the first one is known before the program runs.

`const` controls mutability.

`comptime` controls when the value must be known.

#### Do Not Make APIs Harder to Call

This API is too strict for ordinary use:

```zig
fn repeat(comptime count: usize, text: []const u8) void {
    for (0..count) |_| {
        std.debug.print("{s}\n", .{text});
    }
}
```

It only accepts compile-time counts.

That may be fine if `count` controls generated code, but not if it is just a loop count.

Prefer:

```zig
fn repeat(count: usize, text: []const u8) void {
    for (0..count) |_| {
        std.debug.print("{s}\n", .{text});
    }
}
```

Now callers can pass values from files, arguments, user input, or other runtime sources.

#### Good Uses of `comptime`

Use `comptime` when it gives you something concrete:

| Use | Why It Fits |
|---|---|
| Generic containers | Element type must be known |
| Fixed-size arrays | Length is part of the type |
| Type-based formatting | Type determines generated code |
| Build options | Configuration is known before runtime |
| Static lookup tables | Data can be prepared once |
| Reflection over fields | Field layout is known to the compiler |
| Compile-time validation | Invalid code can be rejected early |

#### Weak Uses of `comptime`

Avoid `comptime` when it only adds restriction:

| Use | Better Choice |
|---|---|
| User input | Runtime parameter |
| File contents | Runtime parsing |
| Network responses | Runtime logic |
| Ordinary arithmetic | Normal functions |
| Simple loops | Runtime loops |
| Small one-off formatting | Direct code |
| Too many specializations | Shared runtime code |

#### A Good Design Question

Before adding `comptime`, ask:

Does this value change the shape of the program?

If yes, `comptime` may be right.

Examples:

```zig
fn Stack(comptime T: type, comptime capacity: usize) type
```

Here, both `T` and `capacity` affect the type layout.

```zig
fn encode(comptime T: type, value: T) []u8
```

Here, `T` affects which encoding code is generated.

But this does not need `comptime`:

```zig
fn add(a: i32, b: i32) i32
```

The values do not change the program’s structure. They are just data.

#### Mental Model

`comptime` is for structure.

Runtime is for data.

Use `comptime` when the compiler needs the information to build the program.

Use runtime code when the running program needs the information to do work.

That boundary keeps Zig code clear.
